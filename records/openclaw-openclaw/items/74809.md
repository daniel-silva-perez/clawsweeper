## Summary

Keep open. Current main still activates the top-level managed proxy by forcing `global-agent` for Node `http`/`https`, while Discord gateway sockets only receive an explicit proxy agent when Discord's owner-specific proxy config is set. The report includes a precise minimal reproduction for the TLS hostname mismatch, and the current tests preserve the implicated global-agent contract rather than proving HTTPS/WSS TLS success through a local HTTP CONNECT proxy.

## What This Changes

The issue asks OpenClaw to make top-level `proxy.enabled=true` route Discord TLS traffic through a local HTTP forward proxy without changing TLS server-name verification to the proxy host.

## Best Possible Solution

Keep the top-level managed proxy contract, but replace or constrain the unsafe Node core hook so HTTPS and WSS traffic use HTTP CONNECT while preserving TLS verification for the original upstream host. Add regression coverage with a local CONNECT proxy plus real HTTPS/WSS targets, and update the network proxy docs if the supported surface changes.

## Reproduction Assessment

Yes. The issue provides a high-confidence reproduction path with exact config, observed Discord logs, and a minimal `global-agent` script; current main still follows that hook path.

## Solution Assessment

No current solution is sufficient. Discord and Telegram owner-specific proxy settings are useful workarounds, but the maintainable fix needs to address the top-level managed proxy path without losing process-wide egress coverage.
