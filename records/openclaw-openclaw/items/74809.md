## Summary

Keep open. Current main still uses the documented top-level managed proxy path with forced global-agent routing for Node http/https, while the working Discord-specific proxy path is separate and only applies when Discord proxy config is set. The report is well scoped and has enough source-backed reproduction evidence to remain actionable, but the fix should be maintainer-led because it touches a security-sensitive process-wide egress guardrail.

## What This Changes

The issue reports that enabling the top-level managed network proxy with a local HTTP forward proxy causes Discord gateway TLS hostname verification failures through OpenClaw's process-wide global-agent routing path.

## Best Possible Solution

Keep the top-level managed proxy contract, but replace or narrow the unsafe global-agent path so Node core HTTPS and WebSocket traffic use an HTTP CONNECT-capable proxy route while TLS hostname verification remains tied to the original destination. Add regression coverage with a local CONNECT proxy plus HTTPS/WSS targets, and update the network proxy docs if the supported surface changes.

## Reproduction Assessment

Yes. The issue includes a minimal global-agent reproduction using the same GLOBAL_AGENT_* variables current main still injects, and current source still calls bootstrapGlobalAgent during top-level proxy activation.

## Solution Assessment

No current fix is present. The per-Discord proxy configuration is a valid workaround and shipped adjacent feature, but it does not solve the documented top-level proxy.enabled failure path.
