## Summary

Keep open. Current main still implements the reported top-level proxy path with forced `global-agent` routing for Node `http`/`https`, while the Discord-specific proxy path only helps when `channels.discord.*.proxy` is configured. The prior Discord proxy work and the reporter's workaround are valid context, but they do not fix the documented `proxy.enabled=true` TLS/SNI failure path.

## What This Changes

The issue reports that enabling the top-level managed proxy with a local HTTP forward proxy causes Discord TLS hostname verification failures through the process-wide proxy path.

## Best Possible Solution

Keep the top-level proxy contract, but replace or constrain the unsafe `global-agent` path so Node core HTTPS and `ws` traffic use an HTTP CONNECT proxy without changing TLS verification to the proxy hostname. Add regression coverage with a local CONNECT proxy plus HTTPS/WSS targets that proves the proxy sees the upstream host while TLS validation remains bound to the original destination, then update the network proxy docs if the supported surface changes.

## Reproduction Assessment

Yes. The issue includes a minimal `global-agent` reproduction using the same env variables that current main still injects, and the source path still calls `bootstrapGlobalAgent()` for top-level proxy activation.

## Solution Assessment

No current fix is present. The per-Discord proxy configuration is a useful workaround and adjacent shipped feature, but the narrow maintainable fix needs to address the top-level managed proxy implementation without weakening its security coverage.
