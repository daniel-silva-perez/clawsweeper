## Summary

Keep open. Current main still activates the documented top-level proxy by forcing `global-agent` for Node `http`/`https`, and the existing Discord-specific proxy support is only a workaround when `channels.discord.*.proxy` is configured. The tests still preserve the current global-agent contract and do not cover TLS hostname/SNI safety for a real HTTPS target through a local forward proxy.

## What This Changes

The issue reports that enabling the top-level `proxy.enabled` HTTP forward-proxy path can break Discord gateway TLS hostname verification when OpenClaw routes Node core HTTPS through `global-agent`.

## Best Possible Solution

Keep the top-level proxy contract, but replace or constrain the process-wide Node core hook so HTTPS and WSS traffic use an HTTP CONNECT proxy while preserving TLS verification against the original destination. Add regression coverage with a local CONNECT proxy and a certificate-valid HTTPS/WSS target, then update the network proxy docs if supported coverage changes.

## Reproduction Assessment

Yes. The issue includes a minimal `global-agent` reproduction using the same env variables current main still writes, and the source path still bootstraps `global-agent` for top-level proxy activation.

## Solution Assessment

No. There is no current fix on main; the Discord per-account proxy configuration is a useful workaround, but the maintainable fix needs to address the managed top-level proxy implementation without weakening egress coverage.
