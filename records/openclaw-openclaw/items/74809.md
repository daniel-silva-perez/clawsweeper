## Summary

Keep open. Current main still activates the documented top-level managed proxy by setting GLOBAL_AGENT_* variables and bootstrapping global-agent for Node http/https, while the Discord-specific explicit proxy workaround remains a separate owner path. The report is actionable and security-sensitive because the fix has to preserve managed egress coverage while correcting TLS destination verification through an HTTP CONNECT proxy.

## What This Changes

The issue reports that top-level proxy.enabled with a local HTTP forward proxy breaks Discord TLS by causing certificate hostname verification to use the proxy host instead of the Discord destination.

## Best Possible Solution

Keep the top-level managed proxy contract, but replace or narrow the unsafe global-agent path so Node core HTTPS and WebSocket traffic use an HTTP CONNECT-capable route while TLS hostname verification remains tied to the original destination. Add regression coverage with a local CONNECT proxy plus HTTPS/WSS targets, and update the network proxy docs if the supported surface changes.

## Reproduction Assessment

Yes. The issue includes a minimal global-agent reproduction using the same GLOBAL_AGENT_* variables current main still injects, and source inspection confirms startProxy still bootstraps that Node http/https hook.

## Solution Assessment

No current fix is present. The per-Discord explicit proxy configuration is a valid workaround, but it does not repair the documented top-level proxy.enabled failure path.
