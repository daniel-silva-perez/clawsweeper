## Summary

Keep open. Current main still documents and implements the top-level managed proxy by forcing `global-agent` for Node `http`/`https`, while Discord gateway sockets use `ws` without an explicit proxy agent unless `channels.discord.proxy` is configured. The related Discord-specific proxy work is useful and matches the reporter's workaround, but it does not fix the reported top-level `proxy.enabled=true` TLS failure path.

## What This Changes

The issue requests that the documented top-level managed network proxy route Discord and Telegram traffic through a local HTTP forward proxy without breaking TLS server-name verification.

## Best Possible Solution

Keep the top-level proxy contract, but replace or constrain the unsafe `global-agent` path so Node `http`/`https` and `ws` traffic use an HTTP CONNECT proxy without changing the upstream TLS server name. Add regression coverage with a local CONNECT proxy and HTTPS/WSS targets that proves the proxy sees the upstream host while TLS verification remains for the original destination, then update the network proxy docs if the supported surface changes.
