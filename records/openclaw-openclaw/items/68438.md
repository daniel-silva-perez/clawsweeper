## Summary

Keep open. Current main still enforces stored Android gateway TLS pins, only prompts on first-time TLS trust, and has no source-level path to accept a legitimate reverse-proxy certificate rotation without clearing or changing stored app state.

## What This Changes

The issue asks Android node connections behind TLS-terminating reverse proxies to recover from renewed certificates without requiring users to clear app data and repeat onboarding.

## Best Possible Solution

Add an explicit Android certificate-rotation recovery flow that keeps TLS pinning enabled by default, detects a stored-pin mismatch, presents a clear certificate-changed confirmation or trusted-fingerprint reset, updates `gateway.tls.<stableId>` only after user approval, and documents/tests the reverse-proxy setup-code/manual path.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: store a gateway TLS fingerprint, then reconnect to the same stable endpoint with a different certificate; current main resolves the old pin, throws `gateway TLS fingerprint mismatch`, and never reaches the first-time trust prompt.

## Solution Assessment

Unclear as proposed because the issue lists several alternatives with different security properties. The maintainable direction is an explicit user-approved repin/reset flow; disabling pinning or treating missing setup-code fingerprints as no-pin would be a broader security downgrade.
