## Summary

Keep open. Current main still stores a per-endpoint Android TLS pin and rejects a changed certificate fingerprint, while the trust prompt only runs when no stored pin exists. The report is plausible and security-sensitive, so it should get maintainer product/security review rather than cleanup closure.

## What This Changes

Handle Android gateway TLS certificate rotation for reverse-proxy-terminated wss endpoints without forcing users to wipe app data and repeat onboarding.

## Best Possible Solution

Add an Android certificate-change recovery path that keeps pinning fail-closed by default but lets the user deliberately replace a stale pin after seeing the old/new fingerprint and an explicit trust warning. Cover manual wss reverse-proxy endpoints and discovered endpoints with focused unit tests, and avoid automatically trusting an advertised TXT fingerprint or silently disabling pins.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: save a trusted fingerprint for a manual non-loopback wss endpoint, then reconnect when the server presents a different certificate; current TLS validation throws the mismatch error and the prompt path is skipped because a stored expected fingerprint exists. I did not run a live Android/reverse-proxy test in this read-only review.

## Solution Assessment

Unclear. The requested direction is valid, but the safest solution is not to disable pinning wholesale when setup codes omit `tlsFingerprint`; a dedicated certificate-change prompt or maintainer-approved trust policy would preserve the security model while fixing renewal lockout.
