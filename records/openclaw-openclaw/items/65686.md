## Summary

Current main still lacks the requested isolated-cron outbound payload policy. The config contract has no `cron.deliveryPayloads` or equivalent `full`/`last`/`none` setting, the resolver preserves all deliverable payloads except for channel-owned final-text preference, and direct dispatch still falls back to synthesized text when payloads are empty. The linked implementation PR #63650 is closed unmerged, and no open PR references this issue.

## What This Changes

Add an operator-configurable isolated cron outbound payload policy so runner channel delivery can send all resolved payloads, only the last payload, or no channel message while preserving run records/UI output.

## Best Possible Solution

Keep this issue open and implement a narrow `cron.deliveryPayloads` config with default `full`, `last` selection after existing payload resolution, and `none` that suppresses runner outbound delivery without falling back to synthesized text. Thread the same policy through final and interim isolated-cron resolution, update schema/help/docs/changelog, and add targeted resolver/dispatch/config tests.

## Repair Work Prompt

Implement a scoped isolated-cron outbound payload policy for this issue. Add a typed `cron.deliveryPayloads` setting with allowed values `full`, `last`, and `none`; default must be `full` and preserve current behavior. Apply the mode after existing payload resolution: `last` keeps only the final outbound payload when multiple resolved payloads exist, and `none` suppresses runner outbound channel delivery while preserving summary/output/run records. Make sure `none` cannot fall back to a synthesized text send in `deliverViaDirect`, but do not change agent-owned `message` tool behavior or per-job `delivery.mode` semantics. Update config type/schema/help/labels/generated schema docs as required, public docs for cron/config, and add a user-facing changelog entry crediting @zhouhe-xydt. Keep the implementation in cron/config surfaces only; do not change channel adapters or unrelated delivery policy. Related refs: this issue, #63650, #57322, #63228.
