## Summary

Keep this open. Current main has a canonical provider-normalization contract, but the Control UI Usage path still returns, groups, filters, suggests, and displays raw provider strings, so documented aliases such as `z-ai`/`z.ai` and `aws-bedrock` can still appear or split from their canonical providers. The `bailian` example is not established by current source or docs, so that alias should not be added without maintainer approval.

## What This Changes

The issue asks for Control UI model-provider displays to use normalized provider values instead of original config/session values across affected views.

## Best Possible Solution

Land a focused Usage-surface fix that reuses the existing provider-id canonicalization contract for display labels, filters, suggestions, query matching, and aggregate grouping, while preserving raw persisted session/config fields and leaving any new alias contract to maintainer approval.
