## Summary

Keep open. Current main still has a provider-id normalizer, but the Usage/Control UI path aggregates, filters, suggests, exports, and renders raw provider strings, so supported aliases such as `z-ai`, `z.ai`, and `aws-bedrock` can still appear or group separately from canonical providers. The discussion usefully connects this to #47620 and #47621, but those closed picker issues do not resolve the remaining Usage surface.

## What This Changes

The issue requests normalized model provider display in Control UI surfaces instead of showing raw configured provider aliases.

## Best Possible Solution

Land a focused Usage-surface fix that reuses the existing provider-id canonicalization contract for aggregate grouping, display labels, provider filters, query suggestions/query matching, and CSV export while preserving raw stored session/config fields.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: a usage entry with provider `z-ai`, `z.ai`, or `aws-bedrock` flows into `sessions.usage` and the Usage UI with raw provider strings used for aggregate keys, filters, suggestions, badges, and labels.

## Solution Assessment

Yes for the narrow repair direction. The maintainable fix is to call the existing canonical provider-id helper at Usage boundaries, not to add a separate UI alias table or silently rewrite stored session/config values.
