## Summary

Keep open. Current main still has a canonical provider-id normalizer, but the Usage/Control UI path continues to aggregate, filter, suggest, export, query-match, and display raw provider strings, so supported aliases such as `z.ai`/`z-ai` and `aws-bedrock` can still appear or group separately from canonical providers. The linked picker issues #47620 and #47621 are closed, but they do not resolve this remaining Usage surface.

## What This Changes

Normalize model provider IDs before displaying, grouping, filtering, querying, and exporting provider values in the Control UI Usage surface so aliases render consistently with canonical provider IDs.

## Best Possible Solution

Land a focused Usage-surface fix that applies the existing provider-id canonicalization contract to aggregate keys, display labels, filter/suggestion values, query matching, and CSV export while preserving raw stored session/config fields and leaving new alias contracts to maintainer approval.
