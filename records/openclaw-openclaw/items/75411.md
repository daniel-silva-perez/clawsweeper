## Summary

Keep open. Current main still has the same routing gap the reporter described: the guarded OpenAI Responses fetch exists, but embedded runs only select the boundary-aware Responses transport when the base stream is `undefined` or exactly `streamSimple`; a non-`streamSimple` session default remains `session-custom` and can bypass the 429 fail-fast path.

## What This Changes

The issue asks for Copilot openai-responses HTTP 429s to fail fast even when embedded runs use the session-custom stream strategy.

## Best Possible Solution

Add a narrow runtime fix that ensures GitHub Copilot/openai-responses embedded runs use a provider-owned or boundary-aware Responses transport with `buildGuardedModelFetch` even when the session starts from the pi default stream closure, while preserving truly custom stream functions and Copilot's header/payload wrapper behavior.

## Reproduction Assessment

Yes. The live reproduction requires an exhausted Copilot quota, but the routing failure is reproducible with a focused unit test that passes a non-`streamSimple` base stream for `github-copilot/gpt-5.4` and verifies the selected strategy does not remain `session-custom`.

## Solution Assessment

Yes, the best fix is in the OpenClaw stream-selection layer or an equivalent provider-owned Responses stream path, not by broadening idle-timeout handling alone. The key invariant should be that terminal HTTP 429 handling is independent of whether the embedded session started from the pi default stream closure.
