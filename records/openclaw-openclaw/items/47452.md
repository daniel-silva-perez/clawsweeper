## Summary

Keep open. Current main still leaves the reported custom `qiniu` Anthropic Messages route without a replay policy that drops `thinking` blocks for `moonshotai/kimi-k2.5`, and the related fix PR was closed unmerged rather than superseded by a shipped fix.

## What This Changes

The issue requests that OpenClaw stop sending or replaying unsupported `thinking` content blocks when using `moonshotai/kimi-k2.5` through a custom `qiniu` Anthropic Messages provider route with reasoning disabled.

## Best Possible Solution

Add a documented replay-policy path for Anthropic-compatible routes that reject `thinking` blocks, so `qiniu/moonshotai/kimi-k2.5` drops or converts historical thinking before replay while Claude signed-thinking preservation remains intact.

## Reproduction Assessment

Yes. The source-level reproduction is to resolve transcript policy for provider `qiniu`, model API `anthropic-messages`, and model id `moonshotai/kimi-k2.5`: current main skips `dropThinkingBlocks`, and the reporter's log shows the corresponding provider 400 for `Invalid value: thinking`.

## Solution Assessment

Unclear. The closed PR's old provider-capability hint is not the best current fix because runtime policy now flows through `buildReplayPolicy` or the unowned fallback; a maintainer should choose between a provider-owned Qiniu/Kimi route, a public model compat flag, or a carefully bounded generic fallback.
