## Summary

Current main still lacks a fix for the reported custom Qiniu Anthropic-compatible Kimi path. The unowned Anthropic replay fallback only drops thinking blocks for Claude model IDs, replay cleanup only strips them when that policy is enabled, and the reported `supportsReasoningEffort` flag is not an Anthropic replay compatibility knob.

## What This Changes

The issue requests that Kimi K2.5 through a Qiniu `anthropic-messages` custom provider stop sending unsupported `thinking` blocks when reasoning support is disabled in model configuration.

## Best Possible Solution

Add a provider-owned or generic replay-policy mechanism that lets Anthropic-compatible Kimi routes declare unsupported thinking blocks, then cover `qiniu/moonshotai/kimi-k2.5` with regression tests proving historical `thinking` blocks are dropped or converted before replay.
