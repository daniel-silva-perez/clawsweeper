## Summary

Keep open. Current main still preserves `thinking` blocks for the reported custom `qiniu` Anthropic Messages route because unowned Anthropic replay only enables `dropThinkingBlocks` for Claude model IDs, and the related fix PR was closed unmerged. The proposed old capability-bag direction is also stale against the current provider replay hook contract.

## What This Changes

The issue reports that `moonshotai/kimi-k2.5` on a custom Qiniu Anthropic Messages provider rejects replayed `thinking` content blocks and asks OpenClaw to strip or convert them when the model is configured as non-reasoning.

## Best Possible Solution

Add a documented replay-policy path for Anthropic-compatible providers or model configs that reject `thinking` blocks, with regression coverage proving `qiniu/moonshotai/kimi-k2.5` drops or converts historical thinking before replay while Claude signed-thinking behavior remains intact.

## Reproduction Assessment

Yes. The source-level reproduction is to resolve transcript policy for provider `qiniu`, API `anthropic-messages`, and model id `moonshotai/kimi-k2.5`: current main does not set `dropThinkingBlocks`, and the reporter supplied the matching provider HTTP 400.

## Solution Assessment

Unclear. The old PR's capability-bag hint is not the best current fix because that contract is deprecated; maintainers should choose between a provider-owned hook, a new documented compat field, or a Qiniu/Moonshot-specific plugin path.
