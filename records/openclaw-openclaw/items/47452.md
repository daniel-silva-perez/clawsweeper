## Summary

Current main still does not implement the reported Kimi/Qiniu behavior for a custom `anthropic-messages` provider. The unowned Anthropic replay fallback only enables `dropThinkingBlocks` for Claude model IDs, so `moonshotai/kimi-k2.5` keeps thinking blocks unless a provider plugin supplies a replay policy. The related fix PR #47459 was closed unmerged, and no exact replacement PR appears to be open.

## Best Possible Solution

Keep this open for maintainer boundary review. The best fix is either a Qiniu/MoonshotAI-compatible provider plugin with a `buildReplayPolicy` that drops unsupported Kimi thinking blocks, or a documented generic custom-provider replay compatibility flag for `anthropic-messages` models. Either path should add regression coverage proving `moonshotai/kimi-k2.5` does not replay `thinking` blocks when configured as unsupported.
