## Summary

Keep open. Current main still preserves historical `thinking` blocks for the reported custom `qiniu` Anthropic Messages route because the unowned fallback only drops them for Claude model IDs, and the related fix PR was closed unmerged rather than shipped.

## What This Changes

The issue asks OpenClaw to stop sending Anthropic `thinking` content blocks to `moonshotai/kimi-k2.5` on a custom `qiniu` Anthropic Messages provider when reasoning support is disabled.

## Best Possible Solution

Add a documented replay-policy path for Anthropic-compatible routes that reject `thinking` blocks, with regression coverage proving `qiniu/moonshotai/kimi-k2.5` drops or converts historical thinking before replay while Claude signed-thinking behavior remains intact.

## Reproduction Assessment

Yes. The source-level reproduction is to resolve transcript policy for provider `qiniu`, API `anthropic-messages`, and model id `moonshotai/kimi-k2.5`: current main falls through to the unowned fallback, does not set `dropThinkingBlocks`, and the reporter supplied the matching provider 400 for `Invalid value: thinking`.

## Solution Assessment

Unclear. The closed PR’s old provider-capability hint is not the best current fix because that bag is deprecated and runtime policy now flows through `buildReplayPolicy` or the unowned fallback; a maintainer should choose the provider-owned or generic config contract before implementation.
