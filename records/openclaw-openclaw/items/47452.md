## Summary

Keep open. Current main still leaves the reported custom `qiniu` Anthropic Messages route without `dropThinkingBlocks`: unowned Anthropic-compatible replay only strips thinking for Claude model IDs, and the related fix PR was closed unmerged. The issue is valid, but the remaining fix needs maintainer choice on the public provider/model replay-policy boundary.

## What This Changes

The issue asks OpenClaw to stop sending `thinking` content blocks for a custom `qiniu/moonshotai/kimi-k2.5` Anthropic Messages configuration when the model is configured as non-reasoning or reasoning-effort unsupported.

## Best Possible Solution

Add a documented provider/model replay-policy path for Anthropic-compatible providers that reject `thinking` blocks, with regression coverage proving `qiniu/moonshotai/kimi-k2.5` drops or converts replayed and outbound thinking while Claude signed-thinking replay remains intact.

## Reproduction Assessment

Yes. The source-level reproduction is to resolve transcript policy for provider `qiniu`, API `anthropic-messages`, and model id `moonshotai/kimi-k2.5`: current main leaves `dropThinkingBlocks` false, and the reporter supplied the matching HTTP 400 from the provider.

## Solution Assessment

Unclear. The old PR’s `dropThinkingBlockModelHints` direction is stale because the legacy provider capabilities bag is deprecated; maintainers should choose between a provider-owned hook, a documented generic compat field, or a Qiniu/Moonshot-specific plugin path.
