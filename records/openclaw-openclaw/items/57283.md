## Summary

Keep this PR open. The underlying bug still exists on current main: both prompt-side failover and the extracted assistant failover helper await locked auth-profile cooldown marking before attempting profile rotation, while the PR’s intended ordering remains a valid narrow fix for #57281. The branch is stale/dirty against current main, so it needs a rebase or replacement that updates the new helper split and adds ordering-focused regression coverage.

## What This Changes

The PR defers auth-profile cooldown marking until after auth-profile rotation and makes the mark fire-and-forget in both prompt-side and assistant-side failover paths in `src/agents/pi-embedded-runner/run.ts`.

## Best Possible Solution

Land a focused current-main fix that rotates before the locked cooldown write in both prompt-side failover and `handleAssistantFailover`, preserves timeout no-cooldown behavior, keeps failed-profile-specific warnings guarded, and adds regression tests that make a slow marker unable to delay `advanceAuthProfile`.

## Reproduction Assessment

Yes. The code path is reproducible by triggering a rate-limit prompt or assistant failover with an active auth profile while `maybeMarkAuthProfileFailure` is slow or lock-blocked; current main awaits that mark before rotation, and #57281 supplies production log evidence of the stall.

## Solution Assessment

No, not as-is. The ordering change is the right direction, but the PR must be rebased onto the current helper split, should keep profile-id guards on profile-specific warnings, and needs targeted ordering tests before it is the best maintainable fix.
