## Summary

Keep this PR open for maintainer review. Current main still has the generic no-config/no-env gateway call wrapper fallback at 10 seconds, but the PR appears mis-scoped as the fix for the linked WeCom large-file issue because the CLI/plugin message-action path passes an explicit 10,000ms timeout before the generic fallback is consulted.

## Best Possible Solution

Keep this PR open and ask for maintainer direction. If the intended change is generic gateway-call alignment, simplify the ternary and add a regression test proving no env/config now times out at 15 seconds. If the intended fix is linked #74164, retarget the change to the CLI/message plugin-action timeout boundary with tests around gateway-executed message.action dispatch, preserving the plugin boundary and avoiding WeCom-specific core logic.
