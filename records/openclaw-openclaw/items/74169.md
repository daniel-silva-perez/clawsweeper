## Summary

Keep this PR open for maintainer review. Current main still has the generic no-config/no-env gateway call wrapper fallback at 10 seconds, so the PR is not obsolete. However, the PR is likely mis-scoped as the fix for linked #74164 because the CLI/plugin message-action path supplies an explicit 10,000ms timeout before the generic gateway fallback is consulted.

## Best Possible Solution

Keep this PR open and ask for maintainer direction. If the intended change is generic gateway-call alignment, simplify the ternary and add a regression test proving no-env/no-config calls use the 15 second shared default. If the intended fix is linked #74164, retarget the timeout change to the CLI/message plugin-action boundary with tests around gateway-executed message.action dispatch, preserving plugin boundaries and avoiding WeCom-specific core logic.
