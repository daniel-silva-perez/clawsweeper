## Summary

Keep this PR open for maintainer review: the proposed generic gateway fallback may be directionally useful, but it does not fix the linked WeCom media-send path because that path already passes an explicit 10000ms timeout before `src/gateway/call.ts` can apply its no-timeout fallback.

## What This Changes

The PR changes `src/gateway/call.ts` so calls without an explicit timeout resolve the shared preauth handshake timeout default of 15000ms instead of falling back to 10000ms.

## Best Possible Solution

Align the timeout at the gateway-executed message-action boundary, or centralize the default so `message.action` calls do not inject an unrelated 10s timeout when the user has not configured one. Keep the fix plugin-agnostic, simplify the generic fallback if retained, and add focused tests for both the generic gateway call default and the CLI message-action path.

## Reproduction Assessment

Yes. The linked WeCom report maps to `messageCommand` -> `runMessageAction` -> gateway plugin execution -> `callGatewayLeastPrivilege`, and current main visibly supplies `timeoutMs: 10_000` on that path. A focused fake gateway-plugin test can reproduce the bad timeout without a live WeCom account.

## Solution Assessment

No. This PR changes only the no-explicit-timeout fallback in `src/gateway/call.ts`, but the reported path passes an explicit 10000ms timeout from `src/infra/outbound/message-action-runner.ts`, so the patch is not the narrowest complete fix for the linked regression.
