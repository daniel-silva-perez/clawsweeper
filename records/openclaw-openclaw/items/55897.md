## Summary

Keep open. Current main still uses oldest-first round-robin auth-profile ordering and explicitly avoids prioritizing `lastGood`, so the requested MRU-sticky API-key behavior remains unresolved. 402 classification and billing failover are broader than the reported release, but current evidence does not prove final-only rendering for all intermediate 402 retries, and related #48623 overlaps without fully superseding this API-key MRU plus silent-402 request.

## What This Changes

The issue requests MRU-sticky selection for multiple `api_key` auth profiles after `/new` and silent intermediate 402 failover until all keys fail.

## Best Possible Solution

Keep the issue open and choose a supported auth-profile selection policy: either make API-key profiles MRU/lastGood sticky after successful use, or add an explicit MRU-versus-round-robin config option. Add focused coverage for `/new` selection, cooldown skipping, auth-profile rotation, and final-only 402 surfacing, then update the model-failover docs.
