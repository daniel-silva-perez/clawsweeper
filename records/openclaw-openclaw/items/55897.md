## Summary

Keep open. Current main still implements oldest-first round-robin auth-profile ordering and explicitly does not prioritize the last successful profile, so the MRU-sticky behavior requested here remains unresolved. The 402 classifier/failover path has improved, but that is only partial coverage and does not close the primary `/new` key-selection problem. Related #48623 overlaps the lastGood ordering concern and also remains open, but it does not fully supersede this issue's API-key MRU plus final-only 402 request.

## Best Possible Solution

Keep the issue open and narrow the fix to a supported auth-profile selection policy: either make API-key profiles MRU/lastGood sticky after successful use, or add an explicit config option for MRU versus round-robin in `src/agents/auth-profiles/order.ts`. Add focused tests for `/new` selection, cooldown skipping, and final-only 402 surfacing through `advanceAuthProfile()`/embedded runner failover, then update the model-failover docs to describe the chosen policy.
