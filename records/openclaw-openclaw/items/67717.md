## Summary

Keep this PR open. Current main still routes 403 HTML provider responses to the generic `auth_html_403` path and tells users to re-authenticate, while the linked #67670 scenario is specifically a Cloudflare browser challenge. The related merged #67642 work covers non-403 CDN/HTML pages, but it does not implement this PR’s targeted 403 challenge classification or raw-HTML suppression for a new failure kind.

## Best Possible Solution

Review and land this PR or an equivalent focused patch: add a `cloudflare_challenge` classification before generic 403 HTML handling, use cautious actionable copy, preserve generic 403 behavior, add regression tests for challenge markers and false positives, and suppress raw HTML in the relevant lifecycle/failover log messages. Keep #67670 open until the implementation merges.
