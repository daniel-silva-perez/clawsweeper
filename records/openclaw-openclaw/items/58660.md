## Summary

This should close as ClawHub/external-plugin work rather than OpenClaw core work: the report is about a WeChat plugin rename/migration (`openclaw-weixin` vs `openclawwechat`), current OpenClaw documents WeChat as an external plugin, and the existing manifest/install surfaces already provide generic replacement metadata and explicit stale-plugin cleanup paths without hardcoding a service-specific mapping in core.

## What This Changes

The issue requests that installing `openclawwechat` handle migration from the older `openclaw-weixin` plugin by avoiding false update/install state, cleaning stale directories, clarifying ID mapping, and documenting the upgrade path.

## Best Possible Solution

Keep OpenClaw core generic and channel-agnostic; the external WeChat plugin/ClawHub package should publish the canonical id, any legacy id metadata, replacement preference metadata, and migration docs or installer wrapper steps for cleaning old `openclaw-weixin`/`openclawwechat` state.

## Reproduction Assessment

No, not as an OpenClaw core issue: the reported behavior depends on third-party `openclawwechat` and old `openclaw-weixin` artifacts plus a 2026.3.28 local install state. Current main can verify the core boundary and generic metadata/cleanup paths, but not the external package’s published metadata.

## Solution Assessment

Yes, closing as ClawHub/external-plugin work is the narrowest maintainable path. Adding a hardcoded `openclaw-weixin -> openclawwechat` cleanup policy in core would violate the plugin boundary and risk deleting similarly named user plugins; the existing manifest, uninstall, and doctor surfaces are the safer generic solution.
