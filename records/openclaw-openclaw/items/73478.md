## Summary

Keep open. Current main still appears to miss the reported live Gateway WebSocket behavior: media-bearing image replies can be persisted for WebChat/history, but the agent-run live chat delta/final path still builds messages from text buffers only, so this is not a conservative cleanup close.

## What This Changes

The issue asks for Gateway WebSocket chat events from `chat.send` image-generation replies to include usable generated image/media information instead of only assistant text.

## Best Possible Solution

Define the Gateway WebSocket media contract explicitly. If live chat events should include generated images, add an additive managed-media projection for agent-run final chat events and cover an image-generation/mediaUrls `chat.send` flow where both the live final event and `chat.history` expose usable image information without raw base64; if the intended contract is history-only, document that client requirement in the protocol/WebChat docs.

## Reproduction Assessment

Yes, at the focused code-path level: trigger `chat.send` through an agent run that starts `onAgentRunStart` and later delivers a media-bearing final payload. Current main appends media to transcript/history but the live final event is emitted from `server-chat.ts` text buffers only; live MiniMax should still be verified before patching.

## Solution Assessment

Unclear pending maintainer contract choice. The safest implementation would reuse the existing managed outgoing image projection for live final events rather than passing through raw data URLs or provider-specific fields.
