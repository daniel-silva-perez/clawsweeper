## Summary

Keep open. Current main still drops assistant media when projecting agent-run Gateway events onto the WebSocket `chat` stream: assistant event data can carry `mediaUrls`, but `src/gateway/server-chat.ts` broadcasts only text content for delta, flush, and final chat payloads. The report is recent, unprotected, author is not a maintainer, and the remaining fix is narrow enough for a ClawSweeper repair PR.

## What This Changes

The issue reports that Gateway WebSocket chat output for an image-generation reply does not include generated image information when `chat.send` triggers the conversation.

## Best Possible Solution

Teach the Gateway agent-event chat bridge to project assistant media into WebSocket `chat` final output using the existing managed outgoing image attachment path, keeping the behavior generic to assistant media rather than MiniMax-specific and avoiding raw base64/data URLs in broadcasts.

## Reproduction Assessment

Yes. A focused source-level reproduction can register a chat-linked run, emit an assistant event with `data.text` plus `data.mediaUrls`, then emit lifecycle end; current `chat` delta/final payloads contain only text content from `src/gateway/server-chat.ts`.

## Solution Assessment

Yes. Reusing or factoring the existing managed media projection used by `chat.send` completion/history is the narrowest maintainable fix; provider-specific MiniMax code or a docs-only downgrade would not address the Gateway contract gap.
