## Summary

Keep open. Current main still separates the reported live Gateway chat WebSocket path from the media/history projection: live assistant `chat` delta/final events are built from text buffers, while generated-image reply payloads are converted into managed image blocks for transcript/history. The previous ClawSweeper review identified the same gap, and the current source at adc20fed still supports that conclusion.

## What This Changes

The issue reports that Gateway WebSocket `chat.send` replies for generated images include only assistant text and omit the generated image information in live `chat` events.

## Best Possible Solution

Add an additive media-bearing projection for Gateway `chat.send` live final events when delivered reply payloads include generated-image media, reusing the managed outgoing image/sanitization path and avoiding raw data URLs or base64 in broadcasts. If maintainers intentionally want history-only media delivery, document that clients must refresh `chat.history` after final events.

## Reproduction Assessment

Yes for a source-level reproduction path: the existing Gateway test can model the report by stubbing `dispatcher.sendFinalReply({ mediaUrls: [...] })` and observing the live final `chat` event before fetching `chat.history`. Current coverage proves history receives managed image blocks, but it does not prove the live event contains image information.

## Solution Assessment

Yes, a narrow additive Gateway live-event fix is the best default if WebSocket chat events are expected to carry generated reply attachments. A docs-only resolution is safer only after an explicit maintainer decision that live events should stay text-only and media should be delivered through `chat.history`.
