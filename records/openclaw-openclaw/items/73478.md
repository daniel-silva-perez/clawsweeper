## Summary

Current main still has a real Gateway live-chat media gap for agent-run WebSocket events. A newer main-only helper can broadcast media-bearing final messages for non-agent `chat.send` completion, but the agent-event bridge still turns assistant events into `chat` delta/final payloads from text buffers only, ignoring `mediaUrls` that the agent stream already carries.

## What This Changes

The issue requests that Gateway WebSocket `chat.send` events include generated image information for image-generation replies instead of only emitting assistant text.

## Best Possible Solution

Add an additive media-aware projection for Gateway agent-run live `chat` final events when assistant stream/final reply data includes generated-image media, reusing the existing managed outgoing image URL path and avoiding raw base64 in broadcasts. If maintainers intentionally want history-only media delivery, document that WebSocket clients must refresh `chat.history` after final events.

## Reproduction Assessment

Yes. A focused source-level reproduction can emit an assistant event with `data.text` plus `data.mediaUrls` through the existing `server-chat.agent-events` harness and then end the lifecycle; current `chat` delta/final payloads will contain only text content.

## Solution Assessment

Yes. A narrow Gateway live-event fix is the best maintainable default because the agent stream already preserves `mediaUrls` and the Gateway already has managed media projection helpers; a docs-only resolution is appropriate only if maintainers explicitly define live events as text/completion notifications.
