## Summary

Keep open. Current main still has a Gateway live WebSocket behavior gap: generated-image media is projected into transcript/history, but live `chat` delta/final events for agent-backed `chat.send` runs are assembled from text buffers and do not include reply media. The prior ClawSweeper comment reached the same conclusion, and the current main source, docs, tests, and latest release tag still support it.

## What This Changes

The issue reports that `openclaw gateway` WebSocket `chat.send` output does not include generated image information for a MiniMax image-generation reply.

## Best Possible Solution

Add an additive media-bearing projection for Gateway live `chat.send` final events when delivered reply payloads include generated-image media, reusing the existing managed outgoing image/sanitization path and avoiding raw base64 in broadcasts. If maintainers intentionally want live events to remain text-only completion notifications, document that Gateway clients must refresh `chat.history` after final events to obtain generated media.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists by stubbing `dispatcher.sendFinalReply({ mediaUrls: [...] })` in the existing Gateway chat test path with an agent-started run: current coverage proves `chat.history` receives image content, while `server-chat.ts` live final events are still text-buffer based.

## Solution Assessment

Yes, a narrow additive Gateway live-event fix is the best default if WebSocket chat events are expected to carry generated reply attachments. A docs-only outcome would be appropriate only after an explicit maintainer decision that media is intentionally history-only for Gateway clients.
