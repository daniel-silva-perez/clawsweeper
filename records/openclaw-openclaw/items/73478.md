## Summary

Keep open. Current main still leaves the reported agent-run Gateway WebSocket path text-only: live `chat` delta/final events are assembled from buffered assistant text in `server-chat.ts`, while media-bearing reply payloads are projected into transcript/history and into non-agent final broadcasts through a different `chat.send` path.

## What This Changes

The issue reports that Gateway WebSocket `chat` events from `chat.send` after image generation omit image/media information and only include assistant text.

## Best Possible Solution

Add an additive, provider-agnostic media projection for agent-run `chat.send` live final events when delivered reply payloads include generated media. Reuse the managed outgoing image path and existing display projection helpers, avoid raw base64/data URL broadcasts, preserve history behavior, and document history-only delivery only if maintainers explicitly choose that contract.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to model an `onAgentRunStart` `chat.send` flow whose dispatcher emits `sendFinalReply({ mediaUrls: [...] })`; current code can persist/project that media into history but the live lifecycle final event is still assembled from text buffers only.

## Solution Assessment

Yes. A narrow additive Gateway live-event fix is the best maintainable solution if WebSocket `chat` events are expected to carry generated reply attachments; a docs-only solution would need an explicit maintainer decision that clients must refresh `chat.history` for media.
