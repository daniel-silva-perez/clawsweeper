## Summary

Keep open. Current main still leaves the reported live Gateway chat WebSocket surface unresolved: agent-run delta/final chat events are constructed from text buffers only, while media-bearing reply payloads are handled on the WebChat/history projection side. The fresh issue and prior ClawSweeper comment both point at the live push path, so this is not a conservative close.

## Best Possible Solution

Keep this open and clarify the Gateway WebSocket media contract. If live chat events should carry generated images, add an additive media-bearing projection for agent-run final chat events in the Gateway chat path and cover image_generate/mediaUrls through chat.send, asserting both live chat final and chat.history expose usable image information without leaking raw base64. If the intended contract is history-only, document that clients must refresh chat.history after final events.
