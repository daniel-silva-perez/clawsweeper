## Summary

Keep open. Current main still separates the reported live Gateway chat WebSocket path from the media/history projection: live chat delta/final events are text-buffer based, while generated-image information is only projected into transcript/history. The prior ClawSweeper comment already identified this same live-event gap, and the current source still supports that conclusion.

## What This Changes

The issue reports that Gateway WebSocket chat events for an image-generation conversation contain only assistant text and do not expose generated image information.

## Best Possible Solution

Add an additive media-bearing projection for Gateway `chat.send` live final events when delivered reply payloads include generated-image media, reusing the managed outgoing image/sanitization path and avoiding raw base64 in broadcasts. If maintainers intentionally want history-only media delivery, document that clients must refresh `chat.history` after final events.

## Reproduction Assessment

Yes for a source-level reproduction path: the existing test can model the report by stubbing `dispatcher.sendFinalReply({ mediaUrls: [...] })`. Current coverage proves `chat.history` contains managed image blocks, while `src/gateway/server-chat.ts` still emits live final chat messages from text buffers only.

## Solution Assessment

Yes, a narrow additive Gateway live-event fix is the best default if WebSocket chat events are expected to carry generated reply attachments. A docs-only resolution would be safer only after an explicit maintainer decision that live events should remain completion/text notifications and media should be history-only.
