## Summary

Keep this PR open because current main still does not implement per-provider `retryOnStatus`, and the PR is a relevant implementation candidate. It should not merge as-is: the fetch-level retry shape can be retried again by upstream SDKs as a connection failure, the branch needs current-main retry-default updates, and the new public config surface needs help/docs alignment.

## What This Changes

The PR adds `models.providers.*.retry` with attempts, delays, backoff factor, and `retryOnStatus`, attaches that config to resolved model objects, wraps guarded provider fetches with status-code retry logic, extends shared retry utilities/tests, and updates Discord retry defaults plus generated baseline hashes.

## Best Possible Solution

Keep the PR open and revise it on current main: preserve the current guarded-fetch timeout, terminal-retry, and SSE behavior; implement status retries without causing SDK retry multiplication; update all affected retry defaults; and add config help/docs/tests/changelog if maintainers accept the new public config contract.

## Reproduction Assessment

Not applicable as a user bug reproduction; this is a feature PR. The review issues have high-confidence static reproduction paths from the PR patch, current-main source, and upstream SDK source showing how thrown custom-fetch errors enter SDK connection retry handling.

## Solution Assessment

No, not as currently proposed. The Symbol-attached model config follows an existing local pattern, but the fetch wrapper should avoid throwing the final HTTP response into SDK connection retry logic and the public config/schema/docs surface must be completed on current main.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Return the final HTTP response instead of throwing from fetch:** `src/agents/provider-transport-fetch.ts:96-104`
  - body: After the wrapper exhausts its own `retryAsync` attempts, the custom fetch throws an Error for the HTTP status. Both OpenAI and Anthropic SDKs treat thrown fetch errors as connection failures and run their own retry loop, so `retry.attempts` can be exceeded and the final error becomes a connection error instead of the provider HTTP error. Keep the final Response visible to the SDK, likely with SDK retry disabled for that response, after the configured status retries are exhausted.
  - confidence: 0.87
- **[P2] Update all Required retry defaults on current main:** `src/infra/retry.ts:10`
  - body: Adding `backoffFactor` to `RetryConfig` makes every `Required<RetryConfig>` default require that key. Current main now has `extensions/discord/src/delivery-retry.ts` with `DISCORD_DELIVERY_RETRY_DEFAULTS satisfies Required<RetryConfig>` but no `backoffFactor`, so the rebased PR will fail type-check unless that default or the `Required` contract is updated.
  - confidence: 0.86
- **[P3] Add help and docs for the new retry config:** `src/config/zod-schema.core.ts:337`
  - body: The new `models.providers.*.retry` block is a public config surface, but the PR only changes the schema and generated hashes. `FIELD_HELP`, labels, and user-facing docs have no entries for `models.providers.*.retry` or its subfields, so config docs and CLI schema help will not explain the accepted keys.
  - confidence: 0.78
