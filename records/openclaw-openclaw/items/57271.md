## Summary

Keep open. Current main still rejects `docker.binds` sources outside the runtime allowlist unless the dangerous override is enabled, while the custom bind docs still present external host-path examples without explaining the default roots or override in that section. The related #59718 is adjacent design work, not a superseding tracker for this narrower docs/runtime mismatch.

## Best Possible Solution

Keep this issue open. Maintainers should decide the intended Docker bind contract, then either document the current default allowlist plus `agents.defaults.sandbox.docker.dangerouslyAllowExternalBindSources` in the public sandboxing/config docs with examples that work by default, or deliberately change the runtime policy to accept configured external bind sources outside only the blocked paths and add security-focused regression coverage. Coordinate with #59718 for the broader least-privilege writable-bind model.

## Risks / Open Questions

- Changing the runtime default to allow arbitrary non-blocked external bind sources would broaden host filesystem exposure, especially for writable mounts.
- A docs-only fix is probably safer if the current allowlist is intentional, but it must replace or qualify examples that fail by default.
- The final direction should be coordinated with #59718 so writable custom bind semantics and `workspaceAccess` behavior do not diverge.
