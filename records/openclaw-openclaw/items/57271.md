## Summary

Keep open. Current main still rejects Docker bind sources outside the runtime allowlist unless the dangerous override is enabled, while the sandbox docs still show arbitrary host-path bind examples without documenting the required override in the custom bind section. The related #59718 is useful design context but does not supersede this narrower docs/runtime mismatch.

## Best Possible Solution

Keep this issue open. Maintainers should decide the intended Docker bind contract, then either document the current default allowlist plus `agents.defaults.sandbox.docker.dangerouslyAllowExternalBindSources` on the public sandboxing/config docs, or change the runtime policy to accept explicitly configured non-blocked external bind sources by default with security-focused regression coverage. Coordinate the design with #59718 for least-privilege writable custom binds.

## Risks / Open Questions

- Changing the default to accept arbitrary non-blocked external bind sources would broaden host filesystem exposure, especially for writable mounts.
- A docs-only fix is likely safer if the current allowlist is intentional, but it must be explicit enough to stop users from following examples that fail by default.
- The final direction should be coordinated with #59718 so writable custom bind semantics and workspaceAccess behavior do not diverge.
