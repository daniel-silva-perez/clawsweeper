## Summary

Keep this PR open. Current main and the latest stable tag still leave `storeLocks` entries in place after `locked()` completes, while the PR is a focused implementation with a regression test and no blocking correctness finding from this read-only review.

## What This Changes

The PR adds identity-checked cleanup for the per-store cron lock tail and a white-box unit test covering cleanup after the final operation and retention while a second operation is queued.

## Best Possible Solution

Review and ship the identity-checked cleanup with the regression test after updating the branch, preserving same-state and same-store serialization while pruning only the final current per-store lock entry.

## Reproduction Assessment

Yes. The source-level reproduction is direct: a `locked()` call with a unique `storePath` stores a `keepAlive` promise in `storeLocks` and current main has no cleanup path, so repeated unique paths accumulate entries.

## Solution Assessment

Yes. The proposed identity check is the narrowest maintainable fix because earlier queued operations cannot delete a newer tail, future calls already fall back to a resolved promise, and the same cleanup pattern exists in the keyed async queue helper.
