## Summary

Keep this PR open. Current main still leaves the module-level cron `storeLocks` entry in place after each operation, while the PR's identity-checked `finally` cleanup is focused, matches the reported memory-growth failure mode, and has no definite correctness issue in the provided diff. The remaining step is maintainer review, rebase/mergeability, and validation rather than cleanup closure or a separate repair lane.

## What This Changes

The PR adds identity-checked cleanup of completed per-store cron lock promises in `src/cron/service/locked.ts` and adds a unit test file covering cleanup after the last operation and retention while a later operation is queued.

## Best Possible Solution

Review and ship the identity-checked cleanup with the regression test after rebase and validation, preserving the existing same-state and same-store serialization semantics while pruning only the final resolved per-store lock entry.

## Reproduction Assessment

Yes. The source-level reproduction is direct on current main: calling `locked()` with a unique `storePath` sets `storeLocks` and the function returns without deleting that key; repeated unique paths therefore accumulate entries. I did not run the new test because this review is read-only.

## Solution Assessment

Yes. The proposed identity check is the narrowest maintainable fix because earlier operations cannot delete a newer queued lock, and future calls after the final operation already fall back to `Promise.resolve()`. The test-only export is a small white-box testing surface, not a blocking correctness problem.
