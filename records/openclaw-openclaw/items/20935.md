## Summary

Current main still does not provide an append-only audit log for every agent memory-file mutation. The existing memory host event journal covers recall, promotion, and dreaming completion only; ordinary writes, edits, apply_patch, memory flush appends, and some memory-core direct writes are not captured with the requested what/when/why/session/user/tamper-evidence context. The Pinchy comment is useful prior art, and the later maintainer Codex review correctly treats this as remaining core memory work.

## Best Possible Solution

Keep this issue open and design a first-class memory-change audit layer around the shared memory/core write surface. The likely direction is an append-only JSONL audit stream for every MEMORY.md and memory/*.md mutation with timestamp, operation, relative path, changed-content summary or content hash, trigger/why, session/run/user context when available, and an explicit integrity strategy such as hash chaining or signing. The implementation should cover promotion, memory flush, dreaming, ordinary write/edit, apply_patch, sandbox writes, and plugin-owned memory writes, then document the privacy and audit contract in the public memory docs.

## Risks / Open Questions

- A correct implementation needs a privacy decision for memory contents: full diffs are useful for auditability but may expose sensitive personal data, while hashes/summaries reduce leakage but weaken human traceability.
- The write surface is cross-cutting: memory-core promotion/dreaming, memory flush, host/sandbox write/edit tools, apply_patch, and plugin-owned writes may need a shared hook or contract.
- Tamper-evidence semantics need design before implementation: append-only JSONL alone is not equivalent to signed, hash-chained, or externally anchored integrity evidence.
