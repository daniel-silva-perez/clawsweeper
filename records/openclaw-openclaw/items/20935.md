## Summary

Keep this issue open. Current main has a memory host event journal, but it records recall, promotion, and dreaming lifecycle events rather than an append-only audit log for every MEMORY.md and memory/*.md mutation with what/when/why/session/user/integrity context. The Pinchy comment is useful prior art for a signed audit trail, and the latest maintainer comment already points in the same direction.

## Best Possible Solution

Keep this issue open for a maintainer-led design of a first-class memory mutation audit layer. The likely direction is an append-only JSONL audit stream around shared memory-file write surfaces, covering MEMORY.md and memory/*.md mutations with timestamp, operation, relative path, changed-content summary or hash, trigger/why, session/run/user context when available, and an explicit integrity strategy. Document the privacy and audit contract in the public memory docs once the design lands.

## Risks / Open Questions

- A correct design is security- and privacy-sensitive: full diffs improve traceability but can expose sensitive memory contents; hashes/summaries preserve more privacy but reduce forensic value.
- Partial instrumentation could create false assurance because the requested audit has to cover memory-core promotion/dreaming, memory flush, host and sandbox write/edit/apply_patch, and plugin-owned memory writes.
- Tamper evidence needs a clear trust model, storage location, rotation/retention policy, and decision on whether hash chaining or signing is required.
