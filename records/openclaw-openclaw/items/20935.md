## Summary

Keep open. Current main has a memory host event journal, but it records recall, promotion, and dreaming lifecycle events rather than an append-only audit log for every MEMORY.md and memory/*.md mutation with what/when/why/session/user/integrity context. The Pinchy comment is useful prior art, and the latest maintainer comment already frames this as first-class memory/security design work rather than a cleanup close.

## Best Possible Solution

Keep this open for a maintainer-led design of a first-class memory mutation audit layer. The likely direction is an append-only JSONL audit stream around shared memory-file write surfaces, covering MEMORY.md and memory/*.md mutations with timestamp, operation, relative path, changed-content summary or hash, trigger/why, session/run/user context when available, and an explicit integrity strategy. Once designed, document the privacy and audit contract in the public memory docs.
