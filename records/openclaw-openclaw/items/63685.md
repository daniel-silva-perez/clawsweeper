## Summary

Keep open. Current main supports the corrected Google Gemma 4 id (`gemma-4-26b-a4b-it`) in docs and tests, but it still treats the reported `gemma-4-26b` shape as a valid forward-compatible Gemma id and sends it to Google unchanged. Because the reporter said they selected the model from terminal setup, this remains a Google plugin/setup validation or aliasing issue rather than a closeable typo.

## Best Possible Solution

Keep this issue open. The best fix belongs in the Google plugin/setup surface: confirm whether terminal onboarding, model picker entries, or `openclaw models set` can lead users to `gemma-4-26b`; then replace that choice with `gemma-4-26b-a4b-it`, add a provider-owned alias if Google intends that shorthand, or fail before dispatch with a clear hint. Add regression coverage for the reported invalid id path.
