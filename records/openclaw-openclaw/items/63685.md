## Summary

Keep open. Current main documents and tests the corrected Google Gemma 4 id `gemma-4-26b-a4b-it`, but it still accepts the reported `gemma-4-26b` shape as a forward-compatible Gemma id and sends it to Google unchanged. The reporter also said they selected it from terminal setup, so the remaining work is provider-owned setup/model selection, aliasing, or validation rather than a closeable user typo.

## Best Possible Solution

Keep this issue open. The best fix belongs in the Google plugin/setup surface: confirm whether terminal onboarding, model picker entries, or `openclaw models set` can lead users to `gemma-4-26b`; then replace that choice with `gemma-4-26b-a4b-it`, add a provider-owned alias if Google intends that shorthand, or fail before dispatch with a clear hint. Add regression coverage for the reported invalid id path.
