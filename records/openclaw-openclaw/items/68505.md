Current main still reuses the inline-shaped REM preview body for standalone REM reports, while the latest PR shape addresses the earlier review concern by keeping inline headings nested and lowering them only for separate reports; I found no blocking correctness or security finding in the diff.

## What This Changes

The PR updates memory-core REM dreaming markdown generation and tests so inline daily-note REM blocks keep `###` section headings while standalone REM report files render those same sections as `##` headings.

## Best Possible Solution

Land a narrow writer-level fix that preserves the REM preview body for inline daily notes and normalizes only the standalone REM report body, with regression coverage for both storage modes; keep broader Dream Diary creation/UI issues on their own track.

## Reproduction Assessment

## Solution Assessment

Yes. The latest PR shape is the narrow maintainable fix because it leaves the generator's inline contract intact and adjusts heading depth at the writer seam where the standalone report context is known.
