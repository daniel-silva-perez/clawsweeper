## Summary

Keep open. Current main still improves some startup/import paths but does not implement the requested lazy channel connect lifecycle: there is no consumed `channels.lazyConnect` option, startup sidecars still await channel startup, the channel manager still starts every registered channel account, and the public channel adapter still exposes `startAccount`/`stopAccount` rather than a register/connect split.

## What This Changes

Add a default-on lazy channel connection lifecycle for the Gateway, with opt-out config, deferred channel transport startup, and status semantics that distinguish registered from connected channels.

## Best Possible Solution

Design an additive lazy channel lifecycle that keeps registration/status/routing cheap at startup, adds a documented config or successor to `channels.lazyConnect`, lets eligible channel plugins declare whether `startAccount` can be deferred, preserves eager startup for channels that require it, connects on first safe inbound/outbound use, and updates readiness/status/tests/docs around registered versus connected/running states.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: on current main, search shows no `channels.lazyConnect` implementation, `startGatewaySidecars()` still awaits `startChannels()`, and `startChannels()` still calls `startAccount()` for every registered channel plugin.

## Solution Assessment

No. Current main is not the requested solution; the narrowest maintainable fix needs an explicit lifecycle/config/capability design rather than treating the existing setup-entry and import deferral work as equivalent to lazy channel connection.
