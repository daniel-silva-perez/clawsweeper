## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks the requested custom-provider model verification command, while the PR branch has blocking transport, JSON/exit-code, supported-API coverage, redaction, and rebase concerns that need maintainer review.

## What This Changes

The PR adds an `openclaw models custom-test` subcommand with `--provider`, `--concurrency`, and `--json` flags, reusing exported custom-onboarding verification helpers to probe configured custom provider models.

## Best Possible Solution

If the feature is accepted, rebase it onto current main and implement custom-provider verification through the normal runtime probe path or a shared runtime probe helper. Enumerate configured custom models with `model.api ?? provider.api`, cover all supported custom OpenAI/Anthropic/Responses APIs, honor provider/model headers and request transport policy, redact sensitive endpoint fields, keep JSON stdout parseable, exit non-zero on failed probes, and add focused CLI tests plus `docs/cli/models.md` coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Route custom tests through runtime transport:** `src/commands/models/custom-test.ts:168-173`
  - body: The new command probes by calling onboarding helpers with only `baseUrl`, `apiKey`, and `modelId`. That bypasses the normal model runtime, so provider/model headers, request auth/proxy/TLS settings, and private-network policy are ignored and the command can report false health for configurations that work through the runtime path.
  - confidence: 0.92
- **[P2] Include every supported custom API:** `src/commands/models/custom-test.ts:123-124`
  - body: Target collection only accepts provider-level `api` values from a three-item allowlist. Current config supports `openai-responses`, `openai-codex-responses`, and per-model `api`, so supported custom providers or mixed-api provider entries are silently skipped.
  - confidence: 0.94
- **[P1] Keep JSON output parseable:** `src/commands/models/custom-test.ts:152-154`
  - body: This progress log runs even when `--json` is set, so stdout contains human text before the JSON payload. That breaks callers piping `openclaw models custom-test --json` into JSON parsers; the empty-target branch has the same pattern.
  - confidence: 0.93
- **[P1] Exit non-zero for JSON failures:** `src/commands/models/custom-test.ts:216-218`
  - body: The JSON branch writes results and returns before computing `failCount` or calling `runtime.exit(1)`. Failed probes therefore produce a success exit code in JSON mode, which makes CI or scripts treat broken models as passing unless they re-parse the payload.
  - confidence: 0.94
- **[P1] Redact provider base URLs in JSON results:** `src/commands/models/custom-test.ts:174-177`
  - body: Each result stores and emits the raw provider `baseUrl`. Provider URLs are treated as sensitive in this repo because they can contain embedded credentials or private topology, so JSON output can leak secrets into logs and artifacts.
  - confidence: 0.9
- **[P2] Validate concurrency as a positive integer:** `src/commands/models/custom-test.ts:100-102`
  - body: `--concurrency 0.5` passes validation, but `Array.from({ length: 0.5 })` creates no workers. The command then returns an empty successful result without testing any configured targets.
  - confidence: 0.86
- **[P2] Fail an explicit empty provider filter:** `src/commands/models/custom-test.ts:142-148`
  - body: When `--provider` is misspelled or matches no testable models, the command logs an empty-target message and exits successfully. In automation, that typo becomes a false pass even though nothing was verified.
  - confidence: 0.84
- **[P2] Normalize Azure Foundry URLs before probing:** `src/commands/models/custom-test.ts:168-169`
  - body: Onboarding stores Azure Foundry/OpenAI provider URLs as `.../openai/v1`, but the reused onboarding probe appends the Azure deployment path to Azure hostnames. Passing the stored URL through unchanged can produce `.../openai/v1/openai/deployments/...` and falsely fail a working provider.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The PR adds new provider network probes but bypasses runtime transport policy and emits sensitive provider URLs in JSON output.

Concerns:

- **[medium] Raw provider URLs in JSON output:** `src/commands/models/custom-test.ts:176`
  - body: `CustomModelTestResult` includes raw `baseUrl` values and writes them in `--json` results. Current config redaction treats provider base URLs as sensitive because they may include embedded credentials or private endpoint topology.
  - confidence: 0.9
- **[medium] Direct fetch bypasses guarded runtime policy:** `src/commands/models/custom-test.ts:168`
  - body: The command reuses onboarding `fetchWithTimeout` verification instead of the runtime/provider HTTP path. That skips configured request headers/auth/proxy/TLS and private-network guard semantics, creating false health results and a broader network behavior than the normal model execution path.
  - confidence: 0.86
