## Summary

Keep this PR open. Current main at 5828dcdb05aad587bff831467e7fa51f73312ff8 still does not implement `openclaw models custom-test`, and the existing `models status --probe` path is auth/profile oriented rather than a full verification pass over every configured custom provider model. The PR remains a plausible feature contribution, but the current PR head still needs maintainer review because it bypasses normal runtime transport semantics and has JSON/redaction/failure-exit gaps.

## Best Possible Solution

Keep this PR open for maintainer review. If the feature is accepted, implement custom-provider verification through the existing runtime probe path or a shared runtime probe helper, enumerate every configured custom provider model using `model.api ?? provider.api`, include Responses-family APIs, honor provider/model headers and request transport settings, keep JSON stdout parseable, return non-zero on failed probes, redact sensitive endpoint fields, and add focused tests plus `docs/cli/models.md` coverage.

## Risks / Open Questions

- Merging the PR as-is could report false custom-provider health for configurations that depend on runtime request headers, auth overrides, proxy/TLS settings, or private-network policy.
- The PR head can emit non-parseable JSON and success exit codes for failed JSON-mode probes, which weakens CI/script usage of the new command.
- The PR head includes raw provider `baseUrl` values in JSON results, which can expose embedded credentials, query tokens, or private endpoint topology in logs.
- The command can skip supported `openai-responses` or `openai-codex-responses` custom providers and models with model-level `api` overrides.
- No focused tests or docs are present in the PR diff to lock down target enumeration, redaction, stdout/stderr behavior, failure exits, or transport parity.
