## Summary

Keep this PR open. Current main still does not implement `openclaw models custom-test`, and the existing `models status --probe` path is an auth/profile health probe rather than a full verification pass over every configured custom provider model. The PR remains a plausible feature contribution, but the current diff still needs maintainer review for runtime transport parity, JSON/exit behavior, redaction, API coverage, tests, and docs.

## Best Possible Solution

Keep this PR open for maintainer review. If the feature is accepted, implement custom-provider verification through the existing runtime probe path or a shared runtime probe helper, enumerate every configured custom provider model using `model.api ?? provider.api`, include Responses-family APIs, honor provider/model headers and request transport settings, keep JSON stdout parseable, return non-zero on failed probes, redact sensitive endpoint fields, and add focused tests plus `docs/cli/models.md` coverage.

## Risks / Open Questions

- Merging the PR as-is could report false custom-provider health for configurations that depend on runtime request headers, auth overrides, proxy/TLS settings, or private-network policy.
- The PR adds a network-using command while its Security Impact says no new network calls, so maintainer review should verify privacy, transport, and logging behavior.
- `--json` can be mixed with human log output and can return success for failed probes, weakening CI/script usage.
- Raw provider `baseUrl` values in JSON output can expose embedded credentials, query tokens, or private endpoint topology.
- The current API allowlist can skip supported `openai-responses`, `openai-codex-responses`, and model-level `api` overrides.
- The diff lacks focused tests and docs for enumeration, transport parity, redaction, stdout/stderr behavior, and failure exits.
