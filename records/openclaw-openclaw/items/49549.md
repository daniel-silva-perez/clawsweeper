## Summary

Keep this PR open. Current main still lacks an outbound-only WhatsApp allowlist, but the PR is not merge-ready: the latest head is non-mergeable, its own WhatsApp resolveTarget tests fail because direct allowTo is ignored, and the implementation still misses directory-backed target resolution for allowTo-only contacts.

## What This Changes

The PR adds a WhatsApp allowTo config/schema/docs surface and threads it through outbound message, action, and implicit delivery target checks so outbound recipients can differ from inbound allowFrom.

## Best Possible Solution

Rebase the work onto the current WhatsApp plugin layout, decide whether OpenClaw wants the full allowTo contract or the narrower explicit-target change, then implement the chosen behavior consistently across resolver, channel outbound, action authorization, implicit delivery, directory lookup, config metadata, docs, and regression tests.

## Reproduction Assessment

Yes. Current main can be verified by static inspection: outbound WhatsApp resolution and config schema still have only allowFrom. The PR-specific blocker is also reproducible from GitHub's latest check annotations, which show the newly added allowTo resolveTarget tests failing on head ad85c9d497f1892721d8292188f6dc7bb940fa4e.

## Solution Assessment

No, not as-is. The direction may be valid if maintainers want an outbound-only allowlist, but the current patch ignores direct allowTo adapter inputs, misses directory-backed target lookup, lacks generated config metadata follow-through, and is stale/non-mergeable against current main.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Pass direct allowTo through resolveTarget:** `extensions/whatsapp/src/channel.ts:164-166`
  - body: The PR adds allowTo to the outbound adapter parameter type and the new tests pass allowTo directly to resolveTarget, but this implementation only reads account?.allowTo from cfg. When cfg is absent, allowTo is discarded, which matches the current failing CI annotations for the new resolveTarget tests. Destructure allowTo and pass allowTo ?? account?.allowTo into resolveWhatsAppOutboundTarget.
  - confidence: 0.96
- **[P2] Include allowTo contacts in directory resolution:** `docs/channels/whatsapp.md:169`
  - body: The new docs/config make allowTo a general outbound recipient list, but WhatsApp directory peers still come from account.allowFrom only. Non-ID flows that resolve through the directory, such as name/picker/autocomplete targets, will not find contacts that are only listed in allowTo, so this feature only works for exact E.164/JID inputs unless the directory surface is updated too.
  - confidence: 0.9
- **[P2] Regenerate config metadata for allowTo:** `src/config/zod-schema.providers-whatsapp.ts:48`
  - body: Adding allowTo to the WhatsApp Zod schema without updating the generated config metadata leaves schema-driven consumers such as config docs/control UI unaware of the new field and risks generated-drift checks. Regenerate and commit the relevant config metadata artifacts with the new property.
  - confidence: 0.82
