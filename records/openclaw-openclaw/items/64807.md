## Summary

Keep open. Current main still contains the reported divergence path: the gateway publishes Dashboard update availability from persisted update-check state and skips live registry work during the throttle interval, while the CLI status command performs a registry-backed check. No open PR or duplicate issue was found that already tracks or fixes this exact cache-divergence bug.

## What This Changes

The issue reports that the Dashboard/Overview update banner can miss package updates because the gateway trusts stale or empty persisted update-check cache state while the CLI reports a live registry-backed update.

## Best Possible Solution

Fix the gateway update-check owner path so Dashboard/Overview availability reflects a valid registry result, does not let stale or fresh-empty persisted state suppress the banner for the full interval, preserves the previous good cache when npm resolution is empty, and retries empty registry results on a short backoff with regression coverage for both stale-cache and fresh-empty-cache cases.

## Repair Work Prompt

Fix the reported Dashboard/Overview versus CLI update availability divergence for issue 64807. Keep the change scoped to gateway update-check cache behavior: stale or fresh-empty persisted `update-check.json` state must not suppress the Dashboard update banner for the normal full interval when the registry has a newer package version. In `src/infra/update-startup.ts`, avoid treating an unresolved `resolveNpmChannelTag` result as a successful full-interval check; preserve any previous good `lastAvailableVersion`/`lastAvailableTag` when npm resolution is empty, and schedule/record behavior so the next gateway check retries on a short backoff rather than waiting 24 hours. Add regression tests in `src/infra/update-startup.test.ts` for at least: a fresh cache with no `lastAvailableVersion` should not block a near-term retry, and a cache containing the current local version should not prevent the gateway from discovering a newer registry version when the persisted availability is stale. Keep CLI semantics in `src/cli/update-cli/status.ts` unchanged unless a small shared helper is needed. Do not change update.run install/restart behavior or unrelated UI rendering. Add a user-facing changelog Fixes entry crediting `Thanks @Fuma2013` if the repository’s active changelog section still expects one.
