## Summary

Keep open. Current main still contains the reported divergence path: the gateway derives Dashboard update availability from persisted update-check state and returns during the throttle window, while the CLI status command performs a registry-backed check. The report is concrete, recent, unprotected, and narrow enough for a focused fix PR.

## What This Changes

The issue requests hardening the gateway update-check cache so Dashboard/Overview update availability stays consistent with `openclaw update status` and is not suppressed by stale or fresh-empty persisted state.

## Best Possible Solution

Fix the gateway-owned update-check cache path so a valid registry result drives Dashboard availability, previous good availability is preserved when npm resolution is empty, and empty registry results retry on a short backoff with regression coverage for stale-cache and fresh-empty-cache cases.

## Reproduction Assessment

Yes. Seed `update-check.json` with a current `lastCheckedAt` and stale or missing `lastAvailableVersion`, then run `runGatewayUpdateCheck`; current code returns before a live registry lookup, while `openclaw update status` uses the registry-backed path.

## Solution Assessment

Yes. The narrow maintainable fix belongs in `src/infra/update-startup.ts` with focused tests; the CLI and UI should not need independent workaround logic if the gateway snapshot state is corrected.
