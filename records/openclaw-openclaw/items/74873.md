## Summary

Keep open. Current main does not have drain-manifest support, so the PR is not obsolete, but the current patch should not merge because it removes the recovery marker after writing it during successful shutdown and persists session-routing metadata without the repository's existing private atomic state-file pattern.

## What This Changes

The PR adds a ChatRunRegistry snapshot API, a new drain-manifest read/write/delete module with tests, shutdown write/delete wiring, registry injection into the close handler, and marks secrets-reloader system events untrusted.

## Best Possible Solution

Revise this PR so shutdown records whether active work was snapshotted, preserves that manifest until no active work was present or the boot-time recovery consumer clears it, stores a true runId or a true persisted session id with matching schema names, and uses the existing owner-only atomic state-file pattern with bounded safe reads/deletes.

## Reproduction Assessment

Yes. A focused diff review reproduces the primary failure: with any non-empty chatRunRegistry.entries() result, the PR writes a manifest and then deleteDrainManifest() runs after the same successful close, so the next boot has no marker to consume.

## Solution Assessment

No. The registry snapshot seam is a reasonable starting point, but the lifecycle cleanup and state-file persistence are not the narrowest reliable solution; the safer path is durable owner-only atomic state plus cleanup tied to no-active-work or confirmed recovery completion.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve manifests for active shutdowns:** `src/gateway/server-close.ts:477`
  - body: After writing the drain manifest, the close handler deletes it after every successful shutdown. A normal restart with active runs therefore removes the marker before the next boot can recover those runs, so gate deletion on an empty snapshot or let the recovery consumer clear it.
  - confidence: 0.93
- **[P2] Record the registry key as a run id:** `src/gateway/server-drain-manifest.ts:48-52`
  - body: The value written as sessionId comes from the chat-run registry key, but current gateway code registers and looks up that map by run id. A boot consumer that treats this as a persisted session id can resume or route the wrong work; rename it to runId or persist a real session id.
  - confidence: 0.89
- **[P2] Write the manifest atomically with private permissions:** `src/gateway/server-drain-manifest.ts:71`
  - body: draining-sessions.json stores session routing metadata, but the PR writes directly to the final path with fs.writeFileSync. Use the repository's temp-file, 0o600, fsync/close, rename pattern so readers cannot observe partial JSON or broader-than-intended permissions.
  - confidence: 0.87
- **[P2] Snapshot the environment before mutating it:** `src/gateway/server-drain-manifest.test.ts:11`
  - body: ORIGINAL_ENV stores the live process.env object, so the beforeEach mutations also affect the captured value and afterEach can leak OPENCLAW_STATE_DIR or OPENCLAW_TEST_FAST into later tests. Capture a shallow copy or use the existing env-stubbing helpers.
  - confidence: 0.84
- **[P3] Return write failures to the close handler:** `src/gateway/server-drain-manifest.ts:72-74`
  - body: writeDrainManifest catches write errors internally, so the caller's try/catch cannot record drain-manifest-write and shutdown can report cleanly after failing to snapshot recovery state. Return a status or throw so close results include the warning.
  - confidence: 0.78
