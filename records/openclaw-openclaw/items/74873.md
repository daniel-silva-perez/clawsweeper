## Summary

Keep this PR open. Current main still has no drain-manifest implementation, and the PR is the active implementation attempt for the interrupted-generation recovery path discussed in #40712, but the latest head still has blocking schema and local-persistence issues despite green checks.

## What This Changes

The PR adds a shutdown-time gateway drain manifest, a `ChatRunRegistry.entries()` snapshot API, close-handler wiring and tests, plus a small system-event trust flag change.

## Best Possible Solution

Revise this PR so the gateway snapshots active runs into a generic recovery manifest with clear `runId` versus persisted session identity semantics, no tracker-specific metadata, owner-only atomic persistence, and reader/delete guards appropriate for a boot-time recovery marker.

## Reproduction Assessment

Yes. A focused source review reproduces the blockers: current main uses the chat-run registry key as a run/client-run identifier, while the PR writes it as `sessionId`, and the PR writes the manifest with a direct `fs.writeFileSync` to the final path.

## Solution Assessment

No. The shutdown snapshot seam is a reasonable direction, but the proposed schema and persistence mechanics are not the narrowest maintainable solution; the safer path is a generic run metadata contract plus the existing private temp-file/rename pattern.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Record registry keys as run IDs:** `src/gateway/server-drain-manifest.ts:49`
  - body: `entry.sessionId` is the chat-run registry key, but current gateway callers add and remove these entries by run/client-run IDs. A boot recovery reader that treats this field as a persisted session ID can resume or route the wrong session; rename this manifest field to `runId` or persist the actual session ID separately.
  - confidence: 0.9
- **[P2] Write the manifest atomically with private permissions:** `src/gateway/server-drain-manifest.ts:72`
  - body: `draining-sessions.json` contains session routing metadata, but this writes directly to the final path with the default creation mode. If shutdown is interrupted readers can observe partial JSON, and permissive umasks may expose the file beyond the owner; use the existing temp-file `0o600` plus rename pattern or equivalent.
  - confidence: 0.88
- **[P2] Remove tracker-specific metadata from gateway state:** `src/gateway/server-drain-manifest.ts:13`
  - body: The gateway recovery manifest now includes `linearTicketId` parsed from arbitrary session keys. Gateway core recovery state should stay tracker-agnostic, and this regex can misclassify unrelated keys; keep the manifest generic and let external tooling infer work-item metadata outside the gateway contract.
  - confidence: 0.84
- **[P3] Snapshot process.env before mutating it:** `src/gateway/server-drain-manifest.test.ts:13`
  - body: `ORIGINAL_ENV` stores the live `process.env` object, so the `beforeEach` mutations also affect the captured value and `afterEach` can leak `OPENCLAW_STATE_DIR` or `OPENCLAW_TEST_FAST` into later tests. Capture `{ ...process.env }` up front or use the existing env helpers; the same pattern was also added in the close-handler test.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff introduces persistent local recovery state containing routing metadata, but the file write is not atomic and does not force owner-only permissions.

Concerns:

- **[low] Manifest file can expose or corrupt routing metadata:** `src/gateway/server-drain-manifest.ts:72`
  - body: `draining-sessions.json` stores session keys and run IDs, but `fs.writeFileSync` writes directly to the final path with default permissions. Use an owner-only atomic write so local readers cannot observe partial JSON or metadata created under a permissive umask.
  - confidence: 0.88
