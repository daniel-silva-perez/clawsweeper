## Summary

Keep open. Current main does not implement a drain manifest, and this active PR should not merge unchanged because it removes the recovery marker before a boot-time consumer can read it, serializes a run-id key as `sessionId`, and writes local recovery metadata non-atomically with default permissions. The existing ClawSweeper and recovery-side comments support revising this PR branch rather than closing or replacing it.

## What This Changes

The PR adds a `ChatRunRegistry` snapshot API, a drain-manifest read/write/delete module with tests, shutdown snapshot/delete wiring, close-handler registry plumbing, and a small `server.impl.ts` system-event trust flag change.

## Best Possible Solution

Revise the PR so shutdown writes a durable owner-only atomic manifest only when active work is snapshotted, preserves that manifest until no active work existed or recovery explicitly consumes it, stores the registry key as `runId` unless a true persisted session id is available, reports write failures through shutdown warnings, and adds close-handler coverage for empty and active-run restart paths.

## Reproduction Assessment

Yes. A focused code review is enough: with a non-empty registry, the PR writes a manifest, clears chat state, completes shutdown, and then deletes the manifest before the next boot can read it.

## Solution Assessment

No as submitted. The snapshot seam is a reasonable direction, but the lifecycle, schema naming, write durability, and warning propagation need tightening before this is the narrowest maintainable recovery foundation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve manifests with active runs across restart:** `src/gateway/server-close.ts:468-474`
  - body: After writing the drain manifest, the close handler unconditionally deletes it after any successful shutdown. A SIGTERM/restart with in-flight runs therefore removes the marker the boot-time recovery PR needs, so gate deletion on no active work being snapshotted or leave cleanup to the recovery consumer.
  - confidence: 0.93
- **[P2] Store the registry key as a run id:** `src/gateway/server-drain-manifest.ts:50-52`
  - body: The value written as `sessionId` comes from the chat-run registry key, but current gateway code registers chat and agent runs by idempotency/run id. A boot consumer that treats this as a persisted session id can route or resume the wrong thing; rename the field to `runId` or persist a real session id.
  - confidence: 0.88
- **[P2] Write the manifest atomically with private permissions:** `src/gateway/server-drain-manifest.ts:68`
  - body: `draining-sessions.json` stores session routing metadata, but the PR writes directly to the final path with default permissions. Use the existing owner-only temp-file plus rename pattern so local readers cannot observe partial JSON or broader-than-intended file mode.
  - confidence: 0.86
- **[P2] Snapshot process.env by value in tests:** `src/gateway/server-drain-manifest.test.ts:11-24`
  - body: `ORIGINAL_ENV` captures the live `process.env` object, so `beforeEach` mutates the captured value and `afterEach` can leak `OPENCLAW_STATE_DIR` or `OPENCLAW_TEST_FAST` into later tests. Capture a shallow copy up front or use the repo's env-stubbing helpers.
  - confidence: 0.84
- **[P3] Report manifest write failures to shutdown:** `src/gateway/server-drain-manifest.ts:70-72`
  - body: `writeDrainManifest` catches write errors internally, so the caller's try/catch will not record `drain-manifest-write` and shutdown can report cleanly even after failing to snapshot recovery state. Return a status or throw so the close result records the warning.
  - confidence: 0.78
