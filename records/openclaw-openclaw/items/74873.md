## Summary

Keep this PR open. Current main still lacks the drain-manifest feature, and the PR is a relevant implementation candidate for interrupted-generation recovery, but the latest head still introduces blocking schema, persistence, and core-boundary issues before it should merge.

## What This Changes

The PR adds `ChatRunRegistry.entries()`, a new gateway drain-manifest module with read/write/delete helpers and tests, shutdown wiring to snapshot active runs before clearing state, and a small `server.impl.ts` trusted-event change.

## Best Possible Solution

Revise this PR so the shutdown snapshot is generic and recovery-ready: store a real `runId` or an actual persisted session id with clear names, write owner-only atomic JSON using the existing state-file pattern, keep work-tracker metadata out of gateway core, and leave stale-manifest TTL or checkpoint semantics to the boot-time recovery consumer with focused tests for active-run and empty-shutdown cases.

## Reproduction Assessment

Yes. A focused source review reproduces the blockers: current main registers the chat-run registry with `clientRunId`, while the PR serializes that key as `sessionId`, and the PR writes the final recovery file directly with `fs.writeFileSync`.

## Solution Assessment

No. The shutdown snapshot seam is a reasonable direction, but the current PR is not the narrowest maintainable solution because the manifest schema is misleading, the write path is not atomic/private, and Linear-specific metadata leaks into a generic gateway contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Record registry keys as run IDs:** `src/gateway/server-drain-manifest.ts:49`
  - body: `entry.sessionId` is the chat-run registry key, but current `chat.send` registers that key with `clientRunId`. A recovery consumer that treats this as a persisted session id can resume or route the wrong work; rename this manifest field to `runId` or persist the actual stored session id separately.
  - confidence: 0.9
- **[P2] Write the manifest atomically with private permissions:** `src/gateway/server-drain-manifest.ts:72`
  - body: `draining-sessions.json` contains session routing metadata, but this writes directly to the final path with the default file mode. If shutdown is interrupted, readers can observe partial JSON, and a permissive umask can expose the file beyond the owner; use a `0o600` temp-file plus rename pattern like the restart intent writer.
  - confidence: 0.89
- **[P2] Remove Linear-specific fields from gateway state:** `src/gateway/server-drain-manifest.ts:52`
  - body: The gateway core manifest now derives `linearTicketId` from arbitrary session keys. Core recovery state should stay tracker-agnostic, and this regex can misclassify unrelated keys; keep the manifest generic and let external workflow tooling infer ticket metadata outside the gateway contract.
  - confidence: 0.84
- **[P3] Snapshot environment before mutating it:** `src/gateway/server-drain-manifest.test.ts:13`
  - body: `ORIGINAL_ENV` captures the live `process.env` object, so the test mutations in `beforeEach` also mutate the captured value and `afterEach` can leak `OPENCLAW_STATE_DIR`/`OPENCLAW_TEST_FAST` into later tests. Capture `{ ...process.env }` up front or use the existing env helpers; the same pattern was also added in the close-handler test.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff introduces persistent local recovery state with routing metadata, but the write path is not atomic or owner-only and the schema adds unnecessary work-tracker metadata.

Concerns:

- **[low] Manifest file may expose or corrupt routing metadata:** `src/gateway/server-drain-manifest.ts:72`
  - body: `draining-sessions.json` stores session keys and run ids, but `fs.writeFileSync` writes directly to the final path with default creation permissions. Use an owner-only atomic write so local readers cannot observe partial JSON or metadata created under a permissive umask.
  - confidence: 0.89
- **[low] Gateway state persists derived tracker metadata:** `src/gateway/server-drain-manifest.ts:52`
  - body: The manifest adds `linearTicketId` by parsing session keys in core gateway code, which unnecessarily persists external work-item metadata in a generic recovery file and broadens what the local state file exposes.
  - confidence: 0.78
