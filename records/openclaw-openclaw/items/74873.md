## Summary

Keep open. The PR is useful source work, but the current head still deletes the drain manifest before a boot-time recovery consumer could see it, and the new manifest writer persists private session routing metadata without the repository's owner-only atomic write pattern.

## What This Changes

The PR adds `ChatRunRegistry.entries()`, a new gateway drain-manifest module with tests, and shutdown wiring to snapshot active chat runs before clearing gateway chat state.

## Best Possible Solution

Revise the shutdown lifecycle so a manifest written for active runs survives until a recovery consumer reads and clears it, or until a real drain is proven. Persist the file through the existing private atomic-write pattern and add a close-handler regression that starts with an active registry entry and verifies the manifest remains after interrupted shutdown.

## Reproduction Assessment

Yes. A focused review path is to call the patched close handler with a non-empty `chatRunRegistry`: the diff writes the manifest, clears chat state, then deletes the manifest before returning, so a boot-time reader would observe nothing.

## Solution Assessment

No. Snapshotting the chat run registry is a reasonable seam, but unconditional end-of-close deletion is not the right lifecycle boundary; cleanup should be owned by confirmed drain completion or boot-time recovery consumption.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Keep the drain manifest after interrupting active runs:** `src/gateway/server-close.ts:474`
  - body: The close handler writes the manifest before clearing chat state, but then always deletes it after shutdown returns. A SIGTERM/restart with in-flight chat runs will therefore leave no manifest for boot-time recovery, because this close path does not prove those runs finished. Delete only when no active runs were snapshotted, after a proven drain, or after the recovery consumer handles the file.
  - confidence: 0.93
- **[P2] Write the manifest with private atomic permissions:** `src/gateway/server-drain-manifest.ts:67`
  - body: `draining-sessions.json` stores session keys and run identifiers, but this writes directly to the final path using the process umask. Use the existing owner-only atomic JSON pattern so local users cannot read routing metadata and recovery readers do not see partial JSON.
  - confidence: 0.82
- **[P2] Snapshot the test environment before mutating it:** `src/gateway/server-drain-manifest.test.ts:12`
  - body: `ORIGINAL_ENV` stores the live `process.env` object, so setting `OPENCLAW_STATE_DIR` and `OPENCLAW_TEST_FAST` mutates the captured value too. The `afterEach` restore can leak those variables into later tests; snapshot with `{ ...process.env }` or use `vi.stubEnv`.
  - confidence: 0.86
