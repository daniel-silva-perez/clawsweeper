## Summary

Keep open. The recovery feature is useful, and the recovery-side discussion reinforces the need for it, but this active PR still has blocking lifecycle, schema, and state-file safety issues before merge. Current main does not already implement the drain manifest, and the PR's latest diff still deletes the marker after a normal successful shutdown, writes it non-atomically with default permissions, and labels the active run key as a session id.

## What This Changes

The PR adds `ChatRunRegistry.entries()`, a drain-manifest read/write/delete module with tests, close-handler wiring to snapshot active chat runs before clearing gateway state, and a `server.impl.ts` change that passes the registry into shutdown while marking secrets reloader system events untrusted.

## Best Possible Solution

Revise this PR so shutdown writes a durable, owner-only atomic manifest for active runs and leaves it in place until no active work was snapshotted, a real drain is proven, or the boot-time recovery consumer clears it. The manifest schema should either store the active registry key as `runId` or include a real persisted session id, and close-handler tests should cover the active-run SIGTERM/restart path.

## Reproduction Assessment

Yes. A focused review path is enough: with a non-empty registry, the PR writes the manifest, clears chat state, then calls `deleteDrainManifest()` before returning, so a normal successful restart leaves no file for boot-time recovery. The run-id/session-id mismatch is also traceable from current `addChatRun(clientRunId, ...)` and `registry.peek(evt.runId)` call sites.

## Solution Assessment

No. Snapshotting the active registry is a reasonable seam, but this implementation uses the wrong lifecycle boundary, an unsafe persistence pattern, and a misleading schema field. The safer solution is to keep the manifest until recovery/drain ownership clears it, and to use the repository's existing private atomic write pattern.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve manifests for active shutdown recovery:** `src/gateway/server-close.ts:474`
  - body: After writing the manifest, the close handler clears chat state and then calls `deleteDrainManifest()` for every successful close. A normal SIGTERM/restart with in-flight runs therefore removes the only recovery marker before the boot-time consumer can read it; delete only when no active runs were snapshotted or after confirmed recovery/drain cleanup.
  - confidence: 0.93
- **[P2] Store the registry key as a run id, not a session id:** `src/gateway/server-drain-manifest.ts:50`
  - body: `entry.sessionId` is the registry key, but current gateway code registers runs with `addChatRun(clientRunId, ...)` and finalizes them with `registry.peek(evt.runId)`. This manifest will label a run id as `sessionId`, which can misroute or confuse the recovery consumer; rename the field to `runId` or persist an actual session id.
  - confidence: 0.88
- **[P2] Use owner-only atomic writes for the manifest:** `src/gateway/server-drain-manifest.ts:67`
  - body: `draining-sessions.json` contains `sessionKey` and run identifiers, but this writes directly to the final path with default permissions and no atomic replace. Use the existing owner-only atomic JSON/text pattern so local readers cannot observe private routing metadata or partial JSON.
  - confidence: 0.84
- **[P2] Snapshot process.env before mutating it:** `src/gateway/server-drain-manifest.test.ts:11`
  - body: `ORIGINAL_ENV` stores the live `process.env` object, so `beforeEach` mutates the captured value too. The `afterEach` restore can leak `OPENCLAW_STATE_DIR` and `OPENCLAW_TEST_FAST` into later tests; snapshot the object or use `vi.stubEnv`.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff introduces a persistent local manifest containing session routing metadata, but writes it non-atomically with default file permissions.

Concerns:
