## Summary

Keep open. The PR is pointed at a useful gateway recovery gap, but the current patch would remove the drain manifest at the end of any successful shutdown, including the exact SIGTERM/restart-with-active-runs case it is meant to preserve for boot-time recovery.

## What This Changes

The PR adds `ChatRunRegistry.entries()`, a new drain-manifest read/write/delete module, shutdown manifest wiring, and drain-manifest unit tests.

## Best Possible Solution

Revise the shutdown path so a manifest written for active runs survives until a boot-time recovery consumer reads and clears it, or delete it only after a real drain is proven. Use the existing private atomic JSON write pattern and add a close-handler test that starts with an active registry entry and verifies the manifest still exists after interrupted shutdown.

## Reproduction Assessment

Yes. A focused review path is to call the patched close handler with a non-empty `chatRunRegistry`: the diff writes the manifest, clears chat state, then deletes the manifest before returning, so a boot-time reader would observe nothing.

## Solution Assessment

No. Snapshotting the chat run registry is a reasonable seam, but unconditional end-of-close deletion is not the right lifecycle boundary; cleanup should be owned by confirmed drain completion or boot-time recovery consumption.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Keep the drain manifest after interrupting active runs:** `src/gateway/server-close.ts:474`
  - body: The close handler writes the manifest before `chatRunState.clear()`, but then always deletes it after shutdown returns. A normal SIGTERM/restart with in-flight chat runs will therefore leave no manifest for boot-time recovery, because this close path does not wait for those runs to finish. Only delete after the snapshotted runs have actually drained or after the recovery consumer handles the file.
  - confidence: 0.93
- **[P2] Write the manifest with private atomic permissions:** `src/gateway/server-drain-manifest.ts:58`
  - body: `draining-sessions.json` stores session keys and run identifiers, but this uses `fs.writeFileSync` with the process umask and a non-atomic write. Other durable state helpers write `0o600` files atomically; use that pattern so readers do not see partial JSON and shared hosts do not expose routing metadata.
  - confidence: 0.78

## Security Review
