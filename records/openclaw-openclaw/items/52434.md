## Summary

Keep open. Current main still does not implement typing feedback for inline tool-dispatched skills, and the proposed patch is not mergeable as-is because it can let a typing-indicator failure skip the tool entirely and can bypass existing typing suppression policy.

## What This Changes

The PR adds an awaited `typing.startTypingLoop()` before `tool.execute()` in the inline `dispatch.kind === "tool"` skill path.

## Best Possible Solution

Implement inline tool-dispatch typing through the existing policy-aware typing signaler semantics, keep typing-start failures non-fatal to skill execution, and add focused coverage for allowed, suppressed, and rejected typing-start paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep typing failures from aborting tools:** `src/auto-reply/reply/get-reply-inline-actions.ts:244-246`
  - body: `startTypingLoop()` can reject because the typing controller uses `rethrowOnError: true`. With this await inside the same `try` block as `tool.execute()`, a transient channel typing failure returns the error reply before the skill tool runs.
  - confidence: 0.93
- **[P2] Respect typing policy before starting inline typing:** `src/auto-reply/reply/get-reply-inline-actions.ts:246`
  - body: `handleInlineActions` runs before the normal run path resolves typing policy, so this unconditional call can emit typing for `typingMode: "never"`, `opts.suppressTyping`, heartbeat, system, or internal-webchat paths that should stay silent.
  - confidence: 0.9
