## Summary

Keep open: current main still lacks the desired inline tool-dispatch typing behavior, but the submitted patch should not merge as-is because it bypasses typing suppression and lets a failed typing callback prevent the skill tool from running.

## What This Changes

The PR adds an awaited `typing.startTypingLoop()` call before executing a tool-dispatched skill command in `src/auto-reply/reply/get-reply-inline-actions.ts`.

## Best Possible Solution

Implement inline tool-dispatch typing through the existing typing policy/signaler semantics, keep typing-start failures non-fatal to skill execution, and add narrow tests for allowed, suppressed, and rejected typing-start behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep typing failures from aborting tools:** `src/auto-reply/reply/get-reply-inline-actions.ts:244-246`
  - body: `startTypingLoop()` can reject because the typing controller uses `rethrowOnError: true`. With this await inside the same `try` block as `tool.execute()`, a transient channel typing failure returns the error reply before the skill tool runs.
  - confidence: 0.93
- **[P2] Respect typing policy before starting inline typing:** `src/auto-reply/reply/get-reply-inline-actions.ts:246`
  - body: `handleInlineActions` runs before the normal run path resolves typing policy, so this unconditional call can emit typing for `typingMode: "never"`, `opts.suppressTyping`, heartbeat, system, or internal-webchat paths that should stay silent.
  - confidence: 0.9

## Repair Work Prompt

Treat this PR as useful source work. Repair inline tool-dispatched skill typing without broad refactors. In `src/auto-reply/reply/get-reply-inline-actions.ts`, do not call `typing.startTypingLoop()` unconditionally inside the same `try/catch` as `tool.execute()`. Start typing only through existing typing policy/signaler semantics or an equivalent policy-aware helper so `typingMode: "never"`, `opts.suppressTyping`, heartbeat, system-event, and internal-webchat paths stay silent. Typing-start failure must be non-fatal and must never prevent `tool.execute()` from running. Add focused tests in `src/auto-reply/reply/get-reply-inline-actions.skip-when-config-empty.test.ts` for allowed typing, suppressed typing, and rejected `startTypingLoop()` still executing the tool. Update `src/auto-reply/reply/reply-utils.test.ts` only if shared typing signaler behavior changes. No dependency, workflow, release, or package metadata changes.
