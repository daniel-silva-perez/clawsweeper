## Summary

Current main still does not implement the PR's intended behavior: inline skill commands with `dispatch.kind === "tool"` execute the tool directly without starting the typing loop. The PR remains useful, but its submitted patch is not merge-ready because it starts typing inside the same try/catch as `tool.execute()` and does not respect the existing typing suppression/mode policy.

## Best Possible Solution

Keep this PR open and update it narrowly: start typing for inline tool-dispatched skill execution through the existing typing-mode/suppression policy, make typing-start failure non-fatal so the tool still executes, and add focused inline-action tests for allowed start, suppression, and typing-start rejection.
