## Summary

This PR should stay open. Current main still awaits provider auth resolution before the existing per-provider usage timeout, so the PR is not obsolete or already implemented. The linked #74085 regression is concrete, but the proposed timeout behavior still needs maintainer review because it uses the existing non-cancelling timeout helper and adds stderr warning behavior to a JSON-oriented status path.

## Best Possible Solution

Keep this PR open for focused maintainer review. Validate the #74085 non-TTY subprocess repro on current main, then either land a narrow provider-auth timeout with regression coverage and accepted diagnostic behavior, or adjust the fix to make auth resolution cancellable/per-provider-bounded if non-cancelled work can still keep the CLI process alive.

## Risks / Open Questions

- The proposed timeout does not cancel `resolveProviderAuths()` work; plugin hooks, OAuth refresh, SecretRef resolution, subprocesses, sockets, prompts, or timers may continue after the loader returns a fallback.
- The latest PR emits `console.warn(...)` on auth timeout. That may help diagnosis, but maintainers should confirm stderr output is acceptable for `status --usage --json` automation.
- Returning `{ providers: [] }` on auth timeout can hide the difference between no configured providers and degraded auth discovery unless the status surface exposes or documents that degraded state.
- The unit test in the PR proves the loader promise resolves when auth never settles, but a subprocess-level repro is still useful to prove the CLI process exits if underlying auth work remains active.
