## Summary

Keep this PR open. The linked regression is plausible and current main still has no timeout around provider auth discovery, but the latest patch introduces a definite false timeout warning for normal empty-auth results, so it should not merge as-is.

## What This Changes

The PR wraps provider usage auth discovery in a timeout, emits a warning on the empty-provider timeout path, and adds a unit test that mocks a never-settling auth resolver.

## Best Possible Solution

Land a narrow provider-auth timeout only after timeout detection distinguishes a real timeout from legitimate empty auth and the JSON status subprocess behavior is validated without noisy false warnings for normal no-provider cases.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Track the real auth-timeout branch before warning:** `src/infra/provider-usage.load.ts:105`
  - body: When `resolveProviderAuths()` legitimately resolves to `[]`, this race still reports a timeout: the `authPromise.then(() => false)` continuation is raced against an already resolved `Promise.resolve(true)`, so `didTimeout` becomes `true`. That makes normal no-provider `status --usage --json` runs emit a timeout warning on stderr. Return a timeout sentinel from the timeout wrapper or set a flag only from the timer branch before warning.
  - confidence: 0.92
