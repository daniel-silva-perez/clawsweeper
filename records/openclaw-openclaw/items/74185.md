## Summary

Keep this PR open. Current main still has the unbounded provider-auth discovery path that can affect `openclaw status --usage --json`, but the latest PR diff introduces a false timeout-warning path that should be fixed before merge.

## What This Changes

The PR wraps provider usage auth discovery in a timeout, warns when the timeout is inferred, and adds a unit test for a never-settling auth resolver.

## Best Possible Solution

Land a narrow provider-auth timeout for status usage after timeout detection is reliable. The final behavior should let `openclaw status --usage --json` terminate from automation while preserving a quiet normal empty-provider result and using a clear timed-out sentinel or timer-branch boolean for diagnostics.

## Reproduction Assessment

Yes. The linked report provides a concrete `spawnSync` reproduction for the original non-TTY status command problem, and current main confirms the auth phase is unbounded. The review finding is directly reproduced by the Node `Promise.race` probe against the PR's timeout-detection expression.

## Solution Assessment

No. Wrapping auth resolution is the right narrow direction, but this implementation infers timeout after the fact with a race that also flags normal empty auth resolution. A safer fix is to return a timeout sentinel from the helper or set an explicit boolean only when the timeout branch fires.
