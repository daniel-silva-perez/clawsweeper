## Summary

Keep this issue open. Current main still intentionally creates an `agent` follow-up turn for each approved async gateway exec completion, keyed by the individual approval id. The repo has partial mitigations now, including `/stop` clearing session queues and configurable queue modes, but I did not find source, docs, or tests showing task-level completion suppression, batching of multiple approved exec completions, stale-completion discard, or guaranteed user-message priority over pending completion follow-ups. The 2026-04-26 confirmation comment is useful current-context evidence and links the same root cause family to #69386 and the wrapper-policy request #72062.

## Best Possible Solution

Keep this issue open as the canonical async exec approval completion-cascade bug. The best product/code direction is a core follow-up scheduler or circuit breaker around exec approval completions: coalesce multiple completions for the same session/task when safe, let `/stop` and newer user input preempt or cancel pending completion follow-ups, suppress stale completions after a task is abandoned, and preserve the existing per-approval audit/idempotency guarantees. Any wrapper-denial policy work should remain separate under #72062 because it reduces one trigger path but does not solve the async completion cascade itself.
