## Summary

Keep this issue open. Current main still sends approved async exec completions back through a follow-up agent turn keyed by the individual approval id, and I did not find task-level batching, stale-completion suppression, or a way for /stop/new user input to cancel already-scheduled exec approval completion follow-ups. The 2026-04-26 confirmation comment also gives recent production evidence that this symptom family is still active and related to, but not replaced by, #69386 and #72062.

## Best Possible Solution

Keep this open as the canonical async exec approval completion-cascade bug. The best direction is a core follow-up scheduler or circuit breaker that can coalesce multiple completions for the same session/task when safe, let /stop and newer user input preempt or cancel pending completion follow-ups, suppress stale completions after a task is abandoned, and preserve per-approval audit/idempotency and direct-diagnostics behavior. Keep wrapper-denial policy work separate under #72062 and denial/stop retry behavior separate under #69386 unless maintainers choose to merge the designs.
