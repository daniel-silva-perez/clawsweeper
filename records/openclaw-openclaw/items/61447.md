## Summary

Keep open. Current main still omits generated compaction summary text from the typed `after_compaction` plugin hook payload even in code paths where the summary is already available, and the exact prior implementation PR was closed unmerged as a dirty branch rather than shipped.

## What This Changes

The issue requests adding generated compaction summary text, and optionally summary token metadata, to the `after_compaction` plugin hook payload for plugin consumers.

## Best Possible Solution

Add an optional summary-text field to the typed plugin hook contract and populate it from existing compaction result summaries in the direct, engine-owned, overflow/timeout, subscribe-time, and harness paths where available. Keep unsupported native paths honest by leaving it undefined, update focused tests and plugin docs, and avoid changing compaction algorithms or transcript persistence.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: current main’s type and hook emitters omit summary text, while the compaction result fixtures and runtime paths already contain `summary` values before the hook is fired.

## Solution Assessment

Yes for the central request. A backward-compatible optional hook payload field is the narrow maintainable fix; parsing inline `<summary>` tags or session JSONL from a plugin remains a workaround, not the best core contract.
