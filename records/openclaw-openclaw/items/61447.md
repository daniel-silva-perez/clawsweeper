## Summary

Current main does not implement the requested after_compaction summary payload. The compaction result already carries generated summary text, but the public plugin hook event type and current emitters only pass counts/session file metadata. This is an actionable core plugin API gap, not a ClawHub-only request.

## Best Possible Solution

Keep this open and implement an additive plugin hook contract change: add an optional summary field such as summaryText, optionally summaryTokens if there is a stable token estimate, thread it through every after_compaction emitter from the existing compaction result, update plugin/internal hook docs, and add targeted tests that prove plugins receive the generated summary on each supported compaction path.

## Risks / Open Questions

- Exposing summary text expands plugin visibility into conversation-derived content; the implementation should treat this as intentional plugin hook data and document the privacy/trust boundary.
- The implementation needs to cover all after_compaction emitters, including direct Pi compaction, context-engine-owned compaction, recovery paths, harness/Codex paths, and subscribe handlers, or plugins will see inconsistent payloads.
