## Summary

Keep this PR open. Current main still hardcodes the control-plane write limiter at 3 requests per 60 seconds, does not expose `gateway.controlPlane.rateLimit` in the runtime config surface, and does not wire any control-plane limiter config during gateway startup. The PR is still a plausible implementation candidate, but it needs a focused rebase/regeneration pass before merge because the provided PR context shows generated config baseline churn and prior review comments flagged schema/baseline drift.

## Best Possible Solution

Keep the PR open as a real implementation candidate. The best path is to rebase it onto current main, keep the diff focused on `gateway.controlPlane.rateLimit`, regenerate only the required config schema/docs artifacts from the current schema, retain validation for positive integer limits, and run targeted gateway/config-schema checks before maintainer review.
