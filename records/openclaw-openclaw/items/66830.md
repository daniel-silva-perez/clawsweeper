## Summary

Keep open. Current main still has the reported floor-only threshold behavior in both preflight compaction and memoryFlush, while the focused replacement PR #74010 is open and unmerged; the issue should close only after that PR or an equivalent narrow fix lands.

## What This Changes

The issue asks for preflight compaction and memoryFlush threshold calculations to honor the effective configured `agents.defaults.compaction.reserveTokens` budget instead of only `reserveTokensFloor`.

## Best Possible Solution

Land #74010 or an equivalent narrow core patch that resolves one effective compaction reserve from `reserveTokens`, `reserveTokensFloor`, and the small-context cap, then uses that value consistently in preflight compaction and memoryFlush gates with focused regression tests.

## Reproduction Assessment

Yes. This is reproducible by source inspection on current main: both gate paths subtract `reserveTokensFloor`, while `pi-settings.ts` separately treats `reserveTokens` as the effective target above the floor.

## Solution Assessment

Yes. The maintainable fix direction is to centralize effective reserve resolution and thread that value into both gates; the open PR appears to target exactly that boundary rather than changing broader compaction policy.
