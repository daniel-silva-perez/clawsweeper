## Summary

Keep open. Current main still computes both the preflight compaction and memory-flush thresholds from `reserveTokensFloor`, while the documented/user-facing reserve knob and the embedded Pi pre-prompt path use `reserveTokens`; an open replacement PR, #74010, is the active implementation path and should be reviewed rather than closing this issue.

## What This Changes

The issue requests that memory-flush and preflight compaction thresholds use the effective `reserveTokens` budget instead of treating `reserveTokensFloor` as the full reserve.

## Best Possible Solution

Land a focused core fix that resolves the effective compaction reserve as `max(reserveTokens, reserveTokensFloor)` at the appropriate config/Pi-settings boundary, threads that value into the preflight and memory-flush gates, keeps the public memory-flush plan contract stable, and adds regression tests plus the existing changelog entry. #74010 is the current candidate for that end state.

## Reproduction Assessment

Yes. This is reproducible by source inspection on current main: the two gate formulas never read `agents.defaults.compaction.reserveTokens`, so setting `reserveTokens` above `reserveTokensFloor` cannot move their thresholds.

## Solution Assessment

Yes, the active fix direction is the narrow maintainable one: compute an effective reserve budget and pass it to the existing preflight/memoryFlush gate helpers with focused tests. The issue should remain open until #74010, or an equivalent patch, merges.
