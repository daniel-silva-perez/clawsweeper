## Summary

Keep open: this is a valid security-adjacent redaction fix, but it remains incomplete against current main because several config-derived sinks still allow custom patterns to replace defaults. Current main has only partial overlapping fixes, so this PR should not be closed as implemented or merged as-is.

## What This Changes

The PR adds a default-plus-custom redaction helper, uses it in the no-options `redactSensitiveText()` and `redactToolDetail()` paths, and adds tests for additive and explicit caller-supplied pattern behavior.

## Best Possible Solution

Define one config-derived redaction contract in a shared resolver/helper used by console logs, file logs, log tailing, persisted transcript text, tool-detail display, and safety-boundary payloads. Keep explicit caller-supplied replacement semantics only where intentional, documented, and covered by targeted tests, then align the logging docs.

## Reproduction Assessment

Yes. A focused reproduction is to configure `logging.redactPatterns` with only an email regex, then send an API-key-shaped value through log tailing or persisted transcript text; current code compiles only the configured regex on those paths, so the default API-key/PEM patterns are not applied.

## Solution Assessment

No. The PR is directionally useful but not the best complete fix because it merges defaults in selected call sites instead of establishing a shared config-derived redaction contract for every sink that reads `logging.redactPatterns`.
