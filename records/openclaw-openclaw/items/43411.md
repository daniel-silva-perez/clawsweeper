## Summary

Keep this PR open. Current main still has replacement semantics for config-derived `logging.redactPatterns` in shared redaction resolution, and the PR remains relevant but incomplete against the current logging, log-tail, and transcript redaction surfaces.

## What This Changes

This PR changes logging redaction so config-derived custom patterns are merged with built-in defaults in `redactSensitiveText` and `redactToolDetail`, with tests for additive config behavior and explicit caller-supplied pattern behavior.

## Best Possible Solution

Define one config-derived redaction contract, put the default-plus-custom merge in a shared resolver used by log tailing, tool detail, and persisted transcript redaction, preserve explicit caller-supplied pattern semantics only where intentionally documented, then align docs/schema help and regression tests across those sinks.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Merge config patterns in the shared resolver:** `src/logging/redact.ts:151`
  - body: The patch makes the no-options `redactSensitiveText()` path additive, but `resolveRedactOptions()` still resolves config with `resolvePatterns(resolved.patterns)`. `readConfiguredLogTail()` uses that resolver directly, so configuring only an email regex can still drop default API-key/token/PEM masking when logs are tailed. Move the config/default merge into the shared config-derived resolver or another helper used by all config sinks.
  - confidence: 0.89
