## Summary

Keep open: current main only partially overlaps with this PR, and the PR still leaves config-derived redaction paths where custom `logging.redactPatterns` can replace built-in token/API-key/PEM defaults. The remaining issue is security-sensitive shared logging behavior, so it needs maintainer-guided revision rather than cleanup closure.

## What This Changes

The PR adds a default-plus-custom redaction helper, uses it in the no-options `redactSensitiveText()` and `redactToolDetail()` paths, and adds tests for additive and explicit caller-supplied pattern behavior.

## Best Possible Solution

Define one config-derived redaction contract in the shared resolver/helper used by console logs, file logs, log tailing, persisted transcript text, tool-detail display, and safety-boundary payloads. Keep explicit caller-supplied replacement semantics only where intentional, documented, and covered by targeted tests, then align the logging docs.
