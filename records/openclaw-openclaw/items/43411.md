## Summary

Keep open. The PR is aimed at a real redaction bug, but the patch is not sufficient for current main because it makes config-derived patterns additive only in selected call paths while other log/transcript sinks still allow custom patterns to replace the built-in secret patterns.

## What This Changes

The PR adds a default-plus-custom redaction pattern helper, updates the no-options `redactSensitiveText()` and `redactToolDetail()` paths to use it, and adds tests for additive and explicit caller-supplied pattern behavior.

## Best Possible Solution

Define one config-derived redaction contract in the shared resolver/helper used by file logs, console logs, log tailing, tool-detail display, persisted transcript text, and safety-boundary payloads. Keep explicit caller-supplied replacement semantics only where intentional, documented, and covered by targeted tests.
