## Summary

Keep this PR open. The branch addresses a real redaction weakness and the second commit resolved the earlier Greptile test-quality comment, but the proposed fix is incomplete against current main because config-derived patterns still replace defaults in shared resolver paths used by log tailing and persisted transcript redaction.

## What This Changes

The PR adds a default-plus-custom redaction pattern helper, changes `redactSensitiveText` and `redactToolDetail` config paths to use it, and adds tests for additive config behavior plus explicit caller-supplied pattern behavior.

## Best Possible Solution

Define one config-derived redaction contract, put the default-plus-custom merge in the shared config resolver used by log tailing, tool detail, and persisted transcript text, preserve explicit caller-supplied pattern semantics only where documented, and align docs plus regression tests across those sinks.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.85

Full review comments:

- **[P2] Merge config patterns in the shared resolver:** `src/logging/redact.ts:151`
  - body: The patch merges defaults only in the no-options `redactSensitiveText()` and `redactToolDetail()` paths. `resolveRedactOptions()` still resolves config patterns with replacement semantics, so `readConfiguredLogTail()` and persisted transcript redaction can still lose default API-key/token/PEM masking when a user adds a custom pattern. Move the config/default merge into the shared config-derived resolver or another helper used by all config sinks.
  - confidence: 0.89
