## Summary

Keep open. Current main still implements replacement semantics for several config-derived redaction paths, and this PR only makes selected call sites additive. Because this is a shared redaction contract with security impact and the branch is currently dirty, it needs maintainer-guided rework rather than cleanup closure.

## What This Changes

The branch adds a default-plus-custom redaction helper, uses it for no-options redaction and tool-detail redaction, and adds tests for additive/custom and explicit pattern behavior.

## Best Possible Solution

Define a single config-derived redaction resolver for console logs, file logs, log tailing, persisted transcript text, tool-detail display, and safety-boundary payloads. Keep direct caller-supplied replacement semantics only where intentional, documented, and covered by tests.

## Reproduction Assessment

Yes. Configure logging.redactPatterns with only an email regex, then send an API-key-shaped value through log tailing or persisted transcript text; current code compiles only the configured regex on those paths, so default API-key/PEM patterns are not applied.

## Solution Assessment

No. The PR is directionally useful, but the maintainable fix is to centralize config-derived default-plus-custom behavior in the shared resolver instead of merging defaults in only selected call sites.
