## Summary

Keep open. The PR targets a real redaction bug, but it is incomplete against current main because some config-derived sinks can still treat user `redactPatterns` as replacements and drop built-in token/API-key/PEM masking.

## What This Changes

The PR adds default-plus-custom redaction pattern handling for selected logging paths and tests additive config behavior plus explicit caller-supplied pattern behavior.

## Best Possible Solution

Define one shared config-derived redaction contract, put default-plus-custom merging in the resolver/helper used by log, log-tail, tool-detail, and persisted transcript sinks, and document explicit caller-supplied pattern replacement only where that API behavior is intentional.
