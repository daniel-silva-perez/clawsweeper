## Summary

Keep open. The PR is directionally useful, but current main still has config-derived redaction paths where custom `logging.redactPatterns` can replace built-in secret patterns, and the PR only merges defaults in selected call sites. Because this is a shared secret-redaction contract and the branch is dirty against current main, it needs maintainer-guided revision rather than cleanup closure.

## What This Changes

The PR adds a default-plus-custom redaction helper, uses it for no-options text redaction and tool-detail display, and adds tests for combined defaults/custom patterns plus explicit caller-supplied replacement semantics.

## Best Possible Solution

Centralize config-derived default-plus-custom redaction in the shared logging resolver, route console logs, file logs, log tailing, persisted transcript text, tool-detail display, and safety-boundary payloads through that contract, and preserve explicit caller-supplied replacement semantics only where intentional, documented, and covered by tests.

## Reproduction Assessment

Yes. Configure `logging.redactPatterns` with only an email regex, then trace the current-main log-tail or persisted-transcript paths: they compile only the configured pattern list, so default API-key/token/PEM patterns are absent. This is visible from source inspection without running tests in this read-only review.

## Solution Assessment

No. The PR fixes the no-options and tool-detail paths, but the narrower maintainable fix is to move additive config-derived behavior into the shared resolver and align all sinks and docs with that contract.
