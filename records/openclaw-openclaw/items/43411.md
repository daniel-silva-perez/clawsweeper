## Summary

Keep open. Current main still has replacement semantics for config-derived redaction in shared logging paths, and this PR only makes selected call sites additive, leaving log tailing and persisted transcript text able to drop the built-in token/API-key/PEM patterns when a user configures a custom regex. Because the affected surface is shared and security-sensitive, this needs maintainer-guided revision rather than cleanup closure.

## What This Changes

The PR adds a default-plus-custom redaction helper, uses it for no-options redaction and tool-detail redaction, and adds tests for additive/custom and explicit pattern behavior.

## Best Possible Solution

Define one config-derived redaction resolver for console logs, file logs, log tailing, persisted transcript text, tool-detail display, and safety-boundary payloads. Preserve direct caller-supplied replacement semantics only where intentional, documented, and covered by tests, then align `docs/logging.md` and `docs/gateway/logging.md` with that contract.

## Reproduction Assessment

Yes. Configure `logging.redactPatterns` with only an email regex, then send an API-key-shaped value through log tailing or persisted transcript text; current source compiles only the configured regex on those paths, so default API-key/PEM patterns are not applied.

## Solution Assessment

No. The PR is directionally useful, but the maintainable fix is to centralize config-derived default-plus-custom behavior in the shared resolver instead of merging defaults in only selected call sites.
