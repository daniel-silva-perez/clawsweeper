## Summary

Keep open: current main still leaves Pi SDK auto-compaction enabled for safeguard mode unless a context engine declares ownership, and open PR #73839 is the active unmerged implementation candidate that explicitly targets this report.

## What This Changes

The issue reports that safeguard compaction is blocked because Pi SDK auto-compaction remains enabled when no context engine owns compaction, causing repeated `already_compacted_recently` failures in long-running sessions.

## Best Possible Solution

Review and land #73839, or an equivalent narrow patch, so the Pi auto-compaction guard treats configured safeguard mode as OpenClaw-owned compaction across embedded and CLI compaction paths while preserving default/off and context-engine-owned behavior.

## Reproduction Assessment

Yes. A static reproduction path is clear: configure `agents.defaults.compaction.mode = "safeguard"` without an owning context engine, then current main enables the safeguard extension while `shouldDisablePiAutoCompaction` still returns false for the Pi guard.

## Solution Assessment

Yes. The narrow maintainable fix is to centralize the policy in `pi-settings` and thread the configured compaction mode through the embedded and CLI call sites, rather than adding channel-specific recovery logic.
