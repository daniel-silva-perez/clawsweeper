## Summary

Keep open: current main still leaves Pi SDK auto-compaction enabled for safeguard mode unless a context engine owns compaction, and open PR #73839 is already the active implementation candidate for the same reported bug.

## What This Changes

The issue requests that OpenClaw disable Pi SDK auto-compaction when safeguard compaction owns the lifecycle, preventing repeated `already_compacted_recently` failures and unchecked session growth.

## Best Possible Solution

Review and land #73839 or an equivalent narrow patch that makes the Pi auto-compaction guard treat safeguard mode as OpenClaw-owned compaction, threads that state through embedded and CLI compaction call sites, and covers safeguard, default/off, context-engine-owned, and unsupported-settings-manager cases.

## Reproduction Assessment

Yes. With `agents.defaults.compaction.mode = "safeguard"` and no owning context engine, current main enables the safeguard extension but `applyPiAutoCompactionGuard` still returns `disabled: false`, leaving Pi's own post-turn threshold compaction active.

## Solution Assessment

Yes. The narrow maintainable solution is to put the policy in `pi-settings` and pass the configured compaction mode through the embedded and CLI call sites, rather than adding channel-specific recovery or startup migration logic.
