## Summary

Keep open. Current main still has the stale sessions cache bug, the linked bug remains open, and this PR is the active implementation candidate, but the patch is not merge-ready because it also clears the last-known-good sessions snapshot on every transient `sessions.list` exception and is dirty against current main.

## What This Changes

The PR updates the Control UI sessions controller to assign `sessionsResult = res ?? null` for `sessions.list` responses and also moves `sessionsResult = null` into the catch path.

## Best Possible Solution

Revise this PR narrowly on current main: on a successful `sessions.list`, assign `state.sessionsResult = res ?? null`; run checkpoint invalidation/refetch only when `res` exists; preserve the current generic-error behavior that keeps last-known-good data; keep the missing-scope clear path; and add focused sessions controller tests for undefined-success clearing and transient-error preservation.

## Reproduction Assessment

Yes. A focused unit path can seed `state.sessionsResult` with a non-empty result, mock `client.request("sessions.list")` to resolve `undefined`, call `loadSessions`, and observe that current main leaves the stale result in place.

## Solution Assessment

No. The success-path `res ?? null` change is the right narrow fix, but clearing `sessionsResult` for every exception is broader than the reported bug and can regress transient reconnect or RPC failure UX.

## Review Findings
