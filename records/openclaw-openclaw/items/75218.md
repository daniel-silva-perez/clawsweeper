## Summary

Keep open. Current main does not yet implement Tavily as a web_fetch provider, and this active contributor PR is the right general feature direction, but the proposed patch is not merge-ready because it does not satisfy the existing provider fallback payload contract and it routes fetch-only overrides through shared Tavily search resolution.

## What This Changes

The PR adds Tavily as a bundled web_fetch provider backed by Tavily Extract, with new Tavily webFetch config, manifest contracts, docs, secretref metadata, and tests.

## Best Possible Solution

Keep the feature in this PR, but revise it so Tavily Extract output is adapted into the existing web_fetch provider payload shape and fetch-specific credential/base-url resolution is separated from the existing Tavily search resolver. Add coverage through createWebFetchTool so the final web_fetch result, not just the provider-local helper, is verified.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to force createWebFetchTool through provider fallback with a Tavily Extract-shaped payload lacking top-level text; current core reads only payload.text, so the final web_fetch text is empty. The resolver regression reproduces by configuring plugins.entries.tavily.config.webFetch.* and then calling Tavily search, which uses the same shared resolveTavilyApiKey/resolveTavilyBaseUrl functions.

## Solution Assessment

No. The product direction is reasonable, but the implementation should map Tavily Extract results into the established web_fetch fallback contract and add fetch-scoped Tavily key/base-url helpers instead of changing shared search resolution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Return text from the Tavily fetch provider:** `extensions/tavily/src/tavily-fetch-provider.ts:35-51`
  - body: Core normalizes provider fallback output from `payload.text` before wrapping and caching. This provider returns the raw Tavily Extract shape with `results[].rawContent`/`content`, so selecting Tavily can produce an empty `web_fetch` text response even when extraction succeeds. Map the selected extracted content into a top-level `text` payload and cover it through `createWebFetchTool`.
  - confidence: 0.91
- **[P2] Keep fetch overrides out of search resolution:** `extensions/tavily/src/config.ts:63-78`
  - body: `resolveTavilyApiKey` and `resolveTavilyBaseUrl` are also used by `runTavilySearch`, so preferring `webFetch.apiKey`/`webFetch.baseUrl` here makes a fetch-only override silently change Tavily `web_search` and `tavily_search` requests. Add fetch-specific resolvers for the new provider and leave the existing search resolver order intact.
  - confidence: 0.87
