## Summary

Keep this PR open. Current main routes child stdout/stderr through `createWindowsOutputDecoder()`, but the default/non-Windows UTF-8 path still decodes each Buffer independently with `buffer.toString("utf8")`, so main does not definitely implement the PR’s intended boundary-safe child output decoding. The PR needs to be rebased/ported onto the shared decoder while preserving the newer Windows legacy-codepage fallback and adding focused regression coverage.

## Best Possible Solution

Keep the PR open and port the fix into the current shared child-output decoder: make the default UTF-8 path streaming/boundary-safe, preserve Windows legacy-codepage fallback behavior, and add focused tests for split UTF-8 chunks plus stdout/stderr flush behavior before maintainer review or landing.

## Evidence
