## Summary

Keep this PR open. Current main moved child stdout/stderr through the shared `createWindowsOutputDecoder()` path, so the original one-file `StringDecoder` patch is stale, but the PR’s intended default UTF-8 boundary-safe streaming behavior is still not implemented for the non-Windows/default decoder path. The right next step is to port the fix into the shared decoder while preserving the Windows console-codepage behavior shipped for #50519.

## Best Possible Solution

Keep the PR open and ask for a port into `src/infra/windows-encoding.ts`: make the default UTF-8 path streaming/boundary-safe, keep the Windows legacy-codepage fallback behavior intact, and add focused tests for split UTF-8 chunks when `platform` is non-Windows or `windowsEncoding` is null/UTF-8, plus child stdout/stderr flush wiring if needed.
