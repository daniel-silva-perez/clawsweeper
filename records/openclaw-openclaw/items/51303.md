## Summary

Keep this PR open. The reported UTF-8 split-chunk corruption is still present on current main, but the submitted child-adapter patch is stale because stdout/stderr now route through the shared Windows output decoder; the fix needs to move into that shared decoder rather than land as-is.

## What This Changes

The PR replaces per-chunk child stdout/stderr `chunk.toString()` decoding with per-stream `StringDecoder("utf8")` instances and end-of-stream flushing.

## Best Possible Solution

Move the boundary-safe UTF-8 streaming behavior into `src/infra/windows-encoding.ts` so the default decoder path is safe, keep the Windows legacy-codepage fallback intact, and add focused split-CJK regression tests for non-Windows/null-or-UTF-8 encoding cases.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the underlying decode failure class: splitting a multibyte UTF-8 character across chunks reproduces the U+FFFD corruption, and current main routes child stdout/stderr through a default decoder branch with the same per-chunk `Buffer.toString("utf8")` behavior.

## Solution Assessment

Is this the best way to solve the issue? No, not as submitted against current main; the original idea is correct, but the maintainable fix now belongs in the shared output decoder rather than direct `StringDecoder` instances in `child.ts`.
