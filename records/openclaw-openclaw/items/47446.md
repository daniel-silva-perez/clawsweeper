## Summary

Keep open. Current main already has the gateway-level env proxy bootstrap, but it does not implement the remaining Discord env-proxy fallback or the claimed proxy tunnel ECONNRESET mitigation; the PR branch is useful source work but has blocking package-boundary, NO_PROXY, and undici option-surface defects before it can merge.

## What This Changes

The PR adds gateway and Discord env-proxy handling, changes Discord WebSocket/REST proxy agent construction to pass keep-alive options, updates focused proxy tests, and includes small gateway command-path refactors.

## Best Possible Solution

Revise the PR into a narrow Discord proxy fix that uses public plugin SDK exports, gates each Discord REST/WebSocket target through NO_PROXY-aware env proxy selection, configures undici proxy tunnel policy through supported ProxyAgent/EnvHttpProxyAgent options, and leaves unrelated gateway command refactors out.

## Reproduction Assessment

Yes for the merge-blocking patch defects: the package-boundary violation is visible in the PR diff, NO_PROXY bypass follows from the current helper contracts, and the undici option issue is supported by undici v8.1.0 source. The original live Discord proxy failure still has no completed manual reproduction in the PR discussion.

## Solution Assessment

No. The direction is useful, but the branch is not the best implementation because it bypasses the plugin boundary, ignores NO_PROXY for env-derived Discord proxying, and puts the claimed keep-alive policy on the wrong undici option surface.

  - body: This production import reaches from the Discord plugin into core src through a relative path. Bundled plugins are supposed to use public SDK subpaths or local barrels, and this bypasses the package boundary that third-party plugins also rely on. Import the helper from openclaw/plugin-sdk/fetch-runtime instead.
  - confidence: 0.94
- **[P1] Honor NO_PROXY before using the Discord env proxy fallback:** `extensions/discord/src/monitor/gateway-plugin.ts:271`
  - body: resolveEnvHttpProxyUrl("https") only selects a proxy URL; it does not check whether discord.com or gateway.discord.gg is excluded by NO_PROXY. With HTTPS_PROXY set and Discord bypassed, this forces gateway traffic through the proxy the operator asked to skip.
  - confidence: 0.9
- **[P2] Put EnvHttpProxyAgent tunnel policy on supported options:** `src/infra/net/undici-global-dispatcher.ts:96`
  - body: Passing keepAlive under connect does not configure the proxied HTTPS tunnel path for undici's ProxyAgent-backed EnvHttpProxyAgent. The ECONNRESET mitigation can remain ineffective for env-proxied fetch traffic unless the policy is passed through the supported proxy transport options and covered by tests.
  - confidence: 0.84
- **[P2] Configure the gateway metadata ProxyAgent correctly:** `extensions/discord/src/monitor/gateway-plugin.ts:288`
  - body: The Discord gateway metadata fetch agent also puts keepAlive under connect, but undici's ProxyAgent builds proxy tunnel connections from proxyTls/requestTls. Metadata fetches can still reuse stale proxy tunnels and continue surfacing ECONNRESET.
  - confidence: 0.82

  - confidence: 0.82
