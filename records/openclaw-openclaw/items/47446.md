## Summary

Keep open. The PR contains useful proxy work and current main only covers the gateway-wide EnvHttpProxyAgent bootstrap, not the Discord WebSocket/env fallback, but the branch is not mergeable as-is because it adds a plugin-boundary violation, bypasses NO_PROXY semantics, and puts the undici proxy keep-alive policy on unsupported option paths.

## What This Changes

The PR bootstraps gateway undici env-proxy handling, adds ambient proxy fallback for Discord gateway traffic, changes Discord and global proxy agent construction to disable keepAlive, updates focused proxy tests, and includes small gateway command-path refactors.

## Best Possible Solution

Revise or replace the PR narrowly: keep the useful Discord env-proxy fallback, implement it through plugin-safe SDK exports, gate Discord REST and gateway hosts with NO_PROXY-aware logic, configure undici proxy tunnel reuse through supported proxyTls-style options with focused tests, and drop unrelated gateway command-path refactors.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Stop importing core internals from the Discord plugin:** `extensions/discord/src/monitor/gateway-plugin.ts:7`
  - body: This production import reaches from the Discord plugin into core src through a relative path. Bundled plugins are supposed to use public SDK subpaths or local barrels, and this bypasses the package boundary that third-party plugins also rely on. Import the existing helper from openclaw/plugin-sdk/fetch-runtime instead.
  - confidence: 0.94
- **[P1] Honor NO_PROXY before using the Discord env proxy fallback:** `extensions/discord/src/monitor/gateway-plugin.ts:271`
  - body: resolveEnvHttpProxyUrl("https") only selects a proxy URL; it does not check whether discord.com or gateway.discord.gg is excluded by NO_PROXY. With HTTPS_PROXY set and Discord bypassed, this forces gateway traffic through the proxy the operator asked to skip.
  - confidence: 0.9
- **[P2] Put EnvHttpProxyAgent tunnel policy on supported options:** `src/infra/net/undici-global-dispatcher.ts:126`
  - body: Passing keepAlive under connect does not configure the proxied HTTPS tunnel path for undici's ProxyAgent-backed EnvHttpProxyAgent. The ECONNRESET mitigation can remain ineffective for env-proxied fetch traffic unless the policy is passed through the supported proxy transport options and covered by tests.
  - confidence: 0.84
- **[P2] Configure the gateway metadata ProxyAgent correctly:** `extensions/discord/src/monitor/gateway-plugin.ts:288`
  - body: The Discord gateway metadata fetch agent also puts keepAlive under connect, but undici's ProxyAgent builds proxy tunnel connections from proxyTls/requestTls. Metadata fetches can still reuse stale proxy tunnels and continue surfacing ECONNRESET.
  - confidence: 0.82
- **[P2] Configure the Discord REST ProxyAgent correctly:** `extensions/discord/src/monitor/rest-fetch.ts:18`
  - body: The explicit Discord REST proxy agent uses the same unsupported connect.keepAlive shape. REST calls through channels.discord.proxy can still reuse a closed proxy tunnel, so the claimed local-proxy ECONNRESET fix is not enforced on this path.
  - confidence: 0.82
