## Summary

Keep open. Current main already has the broad CLI/gateway env-proxy bootstrap, but it still does not add a Discord Gateway WebSocket env-proxy fallback, so this PR is not obsolete. The branch should not merge as-is because it crosses the plugin boundary, ignores target-specific NO_PROXY behavior, and places the proposed undici proxy tunnel policy on option fields that the proxied path does not use.

## What This Changes

The PR adds env-proxy handling for gateway startup and Discord Gateway traffic, changes Discord proxy agents to pass keep-alive options, updates focused proxy tests, and includes small gateway command refactors.

## Best Possible Solution

Revise the branch into a narrow Discord proxy fix that uses public plugin SDK exports, chooses env proxies with target-aware NO_PROXY checks for Discord API and Gateway hosts, configures undici proxy transport policy through supported ProxyAgent and EnvHttpProxyAgent option surfaces, and leaves unrelated gateway command refactors out.

## Reproduction Assessment

Yes for the merge-blocking patch defects: they reproduce by diff and source inspection against the plugin boundary rules, proxy-env helper contract, and undici v8.1.0 source. The original live Discord proxy failure is still unclear because the PR has not completed the manual local-proxy verification.

## Solution Assessment

No. The direction is useful, but this branch is not the best implementation because it bypasses the public plugin boundary, applies env proxy fallback without NO_PROXY target checks, and puts the tunnel policy on unsupported undici option paths.
