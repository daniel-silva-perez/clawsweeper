## Summary

Keep open. Current main covers the gateway-wide undici env-proxy bootstrap, but it still does not cover the PR's Discord WebSocket env-proxy fallback or the claimed ECONNRESET proxy-transport fix, and the submitted patch has blocking package-boundary and proxy-correctness issues.

## What This Changes

The PR bootstraps undici env-proxy handling in gateway startup, derives Discord gateway proxy agents from env proxy vars when explicit Discord config is absent, changes Discord/undici proxy agent construction to pass keepAlive false, and adds focused proxy tests plus small gateway refactors.

## Best Possible Solution

Keep the useful PR open as source material, but require a narrower revision or replacement patch: retain the gateway env-proxy bootstrap already on main, add Discord WebSocket env-proxy fallback through plugin-safe SDK/local helpers that honor NO_PROXY for Discord hosts, apply the undici proxy transport policy at the option layer undici actually reads with focused tests, and remove unrelated gateway command-path edits.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Stop importing core internals from the Discord plugin:** `extensions/discord/src/monitor/gateway-plugin.ts:7`
  - body: This production import reaches from the bundled Discord plugin into src/infra. Bundled plugins must load through their package files and SDK/local artifacts; in packaged installs where src is absent, the Discord plugin can fail before initialization. Use openclaw/plugin-sdk/fetch-runtime or a local plugin artifact instead.
  - confidence: 0.94
- **[P1] Honor NO_PROXY before deriving the Discord env proxy:** `extensions/discord/src/monitor/gateway-plugin.ts:271`
  - body: resolveEnvHttpProxyUrl("https") only picks a proxy URL; it does not evaluate NO_PROXY for discord.com or gateway.discord.gg. With HTTPS_PROXY set and Discord excluded, this forces gateway traffic through a proxy the operator explicitly bypassed. Gate the fallback with shouldUseEnvHttpProxyForUrl for the Discord REST and WebSocket targets.
  - confidence: 0.9
- **[P2] Set EnvHttpProxyAgent tunnel options where undici reads them:** `src/infra/net/undici-global-dispatcher.ts:126`
  - body: Putting keepAlive under connect on EnvHttpProxyAgent does not configure ProxyAgent's proxied HTTPS tunnel settings, so the claimed ECONNRESET mitigation can remain ineffective for env-proxied Discord REST/fetch traffic. Move the relevant policy to the proxy/target tunnel options undici uses and cover that construction in tests.
  - confidence: 0.84
- **[P2] Use supported ProxyAgent options for Discord proxy agents:** `extensions/discord/src/monitor/rest-fetch.ts:18`
  - body: The explicit Discord ProxyAgent construction passes keepAlive under connect, but ProxyAgent reads proxy transport options from proxyTls/requestTls and manages pooling separately. Explicit Discord proxy REST/gateway metadata calls can still reuse stale proxy tunnels, so the ECONNRESET fix is not actually enforced here.
  - confidence: 0.82
