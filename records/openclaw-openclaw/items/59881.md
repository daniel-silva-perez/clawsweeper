## Summary

Current main still appears to build agent hook contexts with `channelId` from `messageChannel ?? messageProvider`, and the inbound Discord run path supplies `messageProvider` as the provider name without a separate actual conversation ID. The reported per-channel isolation bug is still plausible and should remain open.

## Best Possible Solution

Keep this issue open and fix the agent hook context construction. Add a shared helper that derives provider and conversation ID from trusted run metadata, preferably `OriginatingTo`/`To` and `parseRawSessionConversationRef(parseThreadSessionSuffix(sessionKey).baseSessionKey ?? sessionKey)`, then use it consistently for `before_prompt_build`, `agent_end`, related conversation hooks, CLI/embedded/Codex run paths, and tests. `ctx.messageProvider` should stay the provider name, while `ctx.channelId` should be the actual channel/conversation ID when available.
