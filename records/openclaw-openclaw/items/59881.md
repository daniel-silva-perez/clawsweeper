## Summary

Current main still appears to expose provider/surface identity as `ctx.channelId` for agent lifecycle hooks rather than the actual Discord channel/conversation identifier reported here. The previous Codex review comment remains accurate against current main: the data needed to derive the real conversation ID exists in inbound metadata and session keys, but the hook context construction still uses `messageChannel ?? messageProvider`, and the queued run path now passes the originating provider as `messageChannel`.

## Best Possible Solution

Keep this issue open and fix hook context construction in a shared helper. Derive `ctx.messageProvider` from trusted provider/origin fields and derive `ctx.channelId` from the actual conversation target when available, using `OriginatingTo`/`To` and `parseThreadSessionSuffix` plus `parseRawSessionConversationRef` for session-key fallback. Apply it consistently to `before_prompt_build`, `agent_end`, related lifecycle hooks, CLI/embedded/Codex run paths, and add regression tests for Discord `agent:<agentId>:discord:channel:<channelId>` plus at least one non-Discord channel.
