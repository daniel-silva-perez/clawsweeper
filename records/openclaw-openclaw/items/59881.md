## Summary

Keep this open. Current main still builds plugin hook contexts with `ctx.channelId` from `messageChannel ?? messageProvider`, while the Discord queued run path passes the provider/surface as `messageChannel` and carries the actual conversation target separately in `OriginatingTo`/`messageTo` and the session key. That matches the prior ClawSweeper comment and leaves the reported per-channel plugin isolation bug plausible on current main.

## What This Changes

The issue reports that plugin agent hooks expose the provider name in `ctx.channelId` instead of the actual provider conversation or channel ID, causing plugins to merge per-channel state.

## Best Possible Solution

Add one shared hook-context derivation helper that preserves `ctx.messageProvider` as the provider/surface and derives `ctx.channelId` from trusted conversation metadata, preferably `messageTo`/`OriginatingTo` or the base parsed session key, then use it consistently in agent hook contexts and cover Discord channel-session cases in focused tests.

## Reproduction Assessment

Yes. A queued Discord run with `SessionKey: agent:main:discord:channel:c1`, `OriginatingChannel: discord`, and `OriginatingTo: channel:c1` flows into `runEmbeddedPiAgent` with `messageChannel: discord`, and the current hook context then exposes `ctx.channelId` as `discord` instead of `c1`.

## Solution Assessment

Yes, fixing the shared hook context construction is the best direction. A plugin-local workaround would leave the SDK contract misleading and would not help other `before_prompt_build`, `agent_end`, `llm_input`, or `llm_output` consumers.

Current agent hook contexts expose `ctx.channelId` as the provider/surface name for Discord channel runs because the runner uses `params.messageChannel ?? params.messageProvider`. Preserve `ctx.messageProvider` for provider checks. Add a changelog fix entry crediting the reporter `@bradfreels`; update plugin hook docs if the field semantics are now documented. Do not broaden plugin APIs beyond the hook context fix.
