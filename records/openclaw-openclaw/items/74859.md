## Summary

Keep open. Current main still has the reported abort-before-prompt race: the runner handles a pre-aborted signal, but later still evaluates `activeSession.prompt(...)` before `abortable()` can reject, and cleanup does not defensively abort or clear the underlying Pi Agent. The issue is also paired with open PR #74979 using closing syntax, so it should remain open until that PR is reviewed and merged, revised, or replaced.

## What This Changes

The issue reports that pre-aborted embedded agent run signals can reject the runner wrapper after `activeSession.prompt()` has already started, leaving a background Pi Agent loop that continues making LLM calls.

## Best Possible Solution

Review and land #74979, or an equivalent narrow replacement, so the embedded runner checks the run abort state before every `activeSession.prompt(...)` call, cancels any escaped active Agent during abort cleanup, clears queued work where appropriate, and adds focused regression coverage for pre-aborted and abort-during-preflight paths. The configurable max-LLM-call cap is useful defense-in-depth but should be a separate product-policy follow-up unless maintainers want that broader change now.

## Reproduction Assessment

Yes. The issue provides a deterministic pre-aborted `AbortSignal` test shape, and current main still has the same static call-order hazard; this sweep did not run the test because the review was read-only.

## Solution Assessment

Yes. The narrowest maintainable fix is to make prompt startup abort-aware and add cleanup/test coverage at the embedded Pi runner boundary, rather than changing provider behavior or adding a broad max-call policy in the first repair.
