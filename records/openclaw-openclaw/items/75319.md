## Summary

Keep open, but this PR should not merge as-is. Current main still documents and uses plaintext session files, so the security goal is a real product direction, but the PR head has static build/runtime blockers, misses the primary transcript write path, and introduces concrete key-management and plaintext-downgrade concerns.

## What This Changes

The branch adds session transcript and session-store encryption helpers, migration utilities, gateway/session reader changes, and a status document intended to transparently encrypt JSONL transcripts and sessions.json.

## Best Possible Solution

Design encryption at the central session persistence boundary with an explicit key source, recovery and rollback semantics, fail-closed migration, complete reader/writer coverage, docs, changelog, focused tests, and maintainer security approval before landing.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists from the PR head: the patch contains an `await` in a synchronous function, CommonJS `require()` calls in ESM-loaded code paths, and only wraps a secondary transcript append helper while current main's normal embedded run still uses `SessionManager.open(params.sessionFile)` directly.

## Solution Assessment

No. The proposed approach is not the best maintainable solution because it patches selected readers and one helper instead of the central persistence seam, and it stores local key material alongside the encrypted state it is meant to protect.

  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff introduces at-rest encryption and migration behavior with concrete key co-location and plaintext fallback concerns.

Concerns:

- **[high] Session keys are stored beside encrypted state:** `src/config/sessions/encryption.ts:38`
  - body: The new key wrapper stores both the passphrase file and wrapped master key under the OpenClaw state directory. A same-user filesystem reader who can copy session files can likely copy these key files too, so this does not satisfy the stated filesystem-compromise defense boundary without a different key source such as a user secret or platform keystore.
  - confidence: 0.88
- **[medium] Transcript migration can preserve plaintext lines:** `src/config/sessions/migrate-transcripts.ts:36`
  - body: `migrateTranscriptToEncrypted` catches per-line encryption failures, appends the original plaintext line, and proceeds to write the migrated file. That can leave sensitive transcript content unencrypted while the migration appears successful.
  - confidence: 0.9
