## Summary

Keep open. The PR is a security-sensitive implementation candidate, but the current head has blocking TypeScript/ESM/runtime defects and does not cover the primary session persistence path it claims to protect.

## What This Changes

The branch adds AES-GCM helpers, session transcript and sessions.json encryption/migration utilities, and gateway/session reader changes intended to make session storage encrypted transparently.

## Best Possible Solution

Design encryption at the central session persistence boundary, with explicit key-management and recovery semantics, fail-closed migration, complete transcript/session metadata coverage, tests, docs, changelog, and maintainer security approval before landing.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists from the PR head: the branch adds await inside a non-async function, CommonJS require calls in ESM code, and only wraps a secondary delivery-mirror write path while current main still uses the direct embedded SessionManager.open path.

## Solution Assessment

No. The proposed patch is not the best maintainable solution because it patches selected readers and one helper rather than the single persistence seam, and its key storage does not satisfy the stated filesystem-compromise threat model.

## Security Review

Status: needs_attention

Summary: The diff introduces at-rest encryption and migration behavior with concrete key-management and plaintext-downgrade concerns.

Concerns:

- **[high] Session keys are stored beside encrypted data:** `src/config/sessions/encryption.ts:45`
  - body: The new key wrapper stores both the passphrase file and wrapped master key under the OpenClaw state directory. A same-user filesystem reader who can copy sessions can likely copy these key files too, so this does not provide the claimed defense-in-depth boundary without a different key source such as a user secret or platform keystore.
  - confidence: 0.88
- **[medium] Migration can preserve plaintext transcript lines:** `src/config/sessions/migrate-transcripts.ts:35`
  - body: `migrateTranscriptToEncrypted` catches per-line encryption failures, appends the original plaintext line, and still returns success after writing the file. That can leave sensitive content unencrypted while callers believe migration completed.
  - confidence: 0.9
