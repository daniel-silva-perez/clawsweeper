## Summary

Keep this issue open. Current main still recovers eligible queued outbound deliveries by replaying the saved delivery payload, including Telegram entries with only retry/backoff state and `lastError`; it does not classify uncertain Telegram `sendMessage` failures or quarantine them before replay. The transcript side also still depends on an optional mirror idempotency key, while the routed reply path builds mirrors without one.

## Best Possible Solution

Keep this issue open and fix the core outbound recovery policy: persist or derive uncertain-delivery state for non-idempotent Telegram sends, quarantine or require confirmation instead of blind replay for uncertain `sendMessage` failures, and persist a stable delivery-mirror idempotency key for routed replies so recovery cannot append duplicate mirror transcript entries.
