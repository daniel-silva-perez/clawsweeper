## Summary

Keep this issue open. Current main still replays eligible queued outbound entries through the normal delivery path, including Telegram entries whose prior `sendMessage` failure only left `lastError` and `retryCount`. The useful KeepMyClaw discussion is consistent with the remaining root-cause risk, but it is verification tooling rather than a core fix.

## Best Possible Solution

Keep this issue open and fix the core outbound recovery policy: classify non-idempotent Telegram send failures with uncertain delivery state, quarantine or require confirmation instead of blind replay, and persist a stable mirror idempotency key for routed replies so recovery cannot append duplicate delivery-mirror transcript entries.

## Evidence

There is no Telegram-specific uncertain-delivery quarantine before replay.

- Closing now would leave the user-visible duplicate Telegram delivery path open for queued `sendMessage` entries whose previous network failure has uncertain server-side state.
- The transcript side can still diverge when replayed mirror entries lack a stable idempotency key and the matching assistant answer is no longer the latest assistant entry.
