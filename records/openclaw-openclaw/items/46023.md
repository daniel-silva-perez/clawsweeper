## Summary

Keep open. Current main still has the reported Telegram reply-media limit gap, and open PR #74490 is already paired to this issue with closing syntax but has not merged.

## What This Changes

The issue requests that Telegram final replies with `mediaUrl` or `mediaUrls` honor configured `mediaMaxMb` by threading `mediaMaxBytes` into outbound reply media loading.

## Best Possible Solution

Review and land PR #74490, or an equivalent narrow Telegram plugin fix, that carries the already-computed `mediaMaxBytes` through message processing, native-command delivery, `deliverReplies`, and `deliverMediaReply`, then asserts `loadWebMedia` receives the configured `maxBytes` for final media replies.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: configure Telegram `mediaMaxMb`, trigger a final reply with media, and the current code computes the cap but never passes it into `deliverReplies`, `deliverMediaReply`, or `loadWebMedia`.

## Solution Assessment

Yes. The narrow maintainable solution is to thread the existing Telegram-owned `mediaMaxBytes` setting through the reply delivery path and add regression tests; closing this issue before the paired PR merges would be premature.
