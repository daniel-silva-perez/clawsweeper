## Summary

Keep open. Current main still has the reported Telegram reply-media limit gap: Telegram computes a configured media cap and the direct send path uses it, but final reply and native-command reply delivery still do not pass that cap into media loading. Open PR #74490 is already paired to this issue with closing syntax and is the right implementation path to review before closing.

## What This Changes

The issue asks for Telegram final replies with outbound media to honor the configured mediaMaxMb/mediaMaxBytes limit during reply delivery.

## Best Possible Solution

Review and land PR #74490, or an equivalent narrow Telegram plugin fix, that threads the already-computed mediaMaxBytes through message processing, dispatch, native command delivery, deliverReplies, and deliverMediaReply, then passes it as maxBytes to buildOutboundMediaLoadOptions with regression tests for final replies and native command replies.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: configure Telegram mediaMaxMb, trigger a final or native-command reply with media, and the source shows the cap is computed but not passed into deliverReplies, deliverMediaReply, or loadWebMedia.

## Solution Assessment

Yes. The narrow maintainable solution is parameter threading within the Telegram plugin plus focused tests; closing this issue before the paired PR merges would be premature.
