## Summary

Current main still matches the reported Telegram outbound final-reply media limit bug: `mediaMaxBytes` is computed and direct sends honor `mediaMaxMb`, but normal final replies and native command replies do not pass that cap into `deliverReplies`, `deliverMediaReply`, or `loadWebMedia`. The related fix PR #46425 is useful implementation context, but it is closed unmerged and therefore does not supersede this issue.

## What This Changes

Telegram final replies with outbound media should propagate the configured `mediaMaxMb`/`mediaMaxBytes` limit into the final reply media-loading path.

## Best Possible Solution

Keep this issue open and implement the narrow Telegram plugin fix: thread the computed `mediaMaxBytes` from bot setup through normal message processing, dispatch delivery base options, native command delivery base options, `deliverReplies`, and `deliverMediaReply`, then pass it as `maxBytes` to `buildOutboundMediaLoadOptions` while preserving current behavior when no cap is supplied. Use #46425 as prior context for the replacement PR and add focused regression tests.

Current main computes `mediaMaxBytes` from `opts.mediaMaxMb ?? telegramCfg.mediaMaxMb ?? 100`, and direct `sendMessageTelegram` already passes that value to `loadWebMedia`, but final reply delivery and native command delivery do not thread it through to `deliverMediaReply`/`buildOutboundMediaLoadOptions`. - Stay inside the Telegram plugin and focused tests.

- Preserve existing behavior when `mediaMaxBytes` is absent. Do not change the already-correct direct send path defaults or media loading internals.
- Add focused regression tests: `deliverReplies` passes configured `mediaMaxBytes` to `loadWebMedia`; absent `mediaMaxBytes` omits `maxBytes`; dispatch/native command paths forward the cap where their mocks make that observable.
