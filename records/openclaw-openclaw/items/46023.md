## Summary

Keep open. Current main still matches the reported Telegram outbound final-reply media limit gap: the Telegram plugin computes a configured media cap and direct sends honor it, but normal final reply delivery and native command reply delivery do not expose or pass that cap into `deliverMediaReply`/`loadWebMedia`. The related fix PR #46425 is useful implementation context, but it is closed unmerged and does not supersede the remaining bug.

## What This Changes

This issue requests that Telegram final outbound media replies honor `channels.telegram.mediaMaxMb` by threading `mediaMaxBytes` into `deliverMediaReply` and `loadWebMedia`.

## Best Possible Solution

Land a narrow Telegram plugin fix that threads the computed media cap from bot setup through normal message processing and native command delivery into `deliverReplies` and `deliverMediaReply`, then passes it as `maxBytes` to `buildOutboundMediaLoadOptions`, while leaving the already-correct direct send path unchanged.

Current main computes `mediaMaxBytes` in `extensions/telegram/src/bot-core.ts`, but normal final replies and native command replies do not pass it through to `deliverReplies`/`deliverMediaReply`, so `loadWebMedia` receives no `maxBytes`. Thread the computed cap through the current code structure: bot setup -> `createTelegramMessageProcessor` -> `dispatchTelegramMessage` -> delivery base options -> `deliverReplies` -> `deliverMediaReply` -> `buildOutboundMediaLoadOptions({ maxBytes })`. Add focused regression tests that prove configured `mediaMaxMb` reaches `loadWebMedia` for outbound media replies and native command media replies, and that the option is omitted/preserved when unset if that is the existing behavior.
