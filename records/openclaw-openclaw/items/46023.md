## Summary

Keep open. Current main still matches the reported Telegram outbound final-reply media limit gap: the Telegram plugin computes `mediaMaxBytes` and the direct send path honors `mediaMaxMb`, but final reply delivery and native command reply delivery still do not expose or forward that cap into `deliverMediaReply`/`loadWebMedia`. The related fix PR #46425 is useful context, but it is closed unmerged and does not supersede this issue.

## What This Changes

Telegram final replies with outbound media should propagate the configured `mediaMaxMb`/`mediaMaxBytes` limit into the final reply media-loading path.

## Best Possible Solution

Keep this issue open and land a narrow Telegram plugin fix that threads the computed media cap from bot setup through normal and native reply delivery into `loadWebMedia`, with focused regression tests and no change to the already-correct direct send path.

## Repair Work Prompt

Fix the Telegram outbound final-reply media limit bug tracked by openclaw/openclaw#46023. Current main computes `mediaMaxBytes` from `opts.mediaMaxMb ?? telegramCfg.mediaMaxMb ?? 100`, and direct `sendMessageTelegram` already passes that value to `loadWebMedia`, but final reply delivery and native command reply delivery do not thread it through to `deliverMediaReply` / `buildOutboundMediaLoadOptions`. Use #46425 as prior implementation context, but implement against current main.

Expected boundary:
- Stay inside the Telegram plugin and focused tests.
- Thread `mediaMaxBytes` from `bot-core.ts` into `createTelegramMessageProcessor`, then into `dispatchTelegramMessage`, then into normal `deliveryBaseOptions` in `bot-message-dispatch.ts`.
- Thread `mediaMaxBytes` through `registerTelegramNativeCommands` and `buildCommandDeliveryBaseOptions` for native command reply delivery.
- Add optional `mediaMaxBytes?: number` to `deliverReplies` params and `deliverMediaReply` params.
- Pass `maxBytes: params.mediaMaxBytes` into `buildOutboundMediaLoadOptions({ mediaLocalRoots, maxBytes })` in `deliverMediaReply`.
- Preserve existing behavior when `mediaMaxBytes` is absent. Do not change the already-correct direct send path defaults or media loading internals.
- Add focused regression tests: `deliverReplies` passes configured `mediaMaxBytes` to `loadWebMedia`; absent `mediaMaxBytes` omits `maxBytes`; dispatch/native command paths forward the cap where their mocks make that observable.
- Add a user-facing changelog fix entry under the active version if changelog policy applies for this channel bug, with `Thanks @Designdocs` and `Thanks @artwalker` if prior PR work is reused; do not credit bots.

Validation: run the targeted Telegram tests first, then the changed gate per repo policy/Testbox before handoff. Do not edit unrelated Telegram direct-send behavior or media loading internals.
