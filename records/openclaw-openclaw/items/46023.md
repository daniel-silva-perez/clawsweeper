## Summary

Keep open. Current main still does not propagate the configured Telegram media cap into final reply media loading, and open PR #74490 is already paired to this bug with closing syntax but has not merged.

## What This Changes

Telegram final replies with `mediaUrl` or `mediaUrls` should honor `channels.telegram.mediaMaxMb` by threading the computed `mediaMaxBytes` through outbound reply delivery into `loadWebMedia`.

## Best Possible Solution

Review and land #74490, or an equivalent narrow Telegram plugin fix, that passes the existing computed `mediaMaxBytes` through message processing, native command delivery, `deliverReplies`, and `deliverMediaReply`, then asserts `loadWebMedia` receives `maxBytes` for configured and default cap behavior.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: configure Telegram `mediaMaxMb`, trigger a final reply with media, and current main shows the computed cap is not accepted by `deliverReplies` or passed into the reply-media `loadWebMedia` call.

## Solution Assessment

Yes. The paired PR direction is the narrow maintainable fix because it threads an existing Telegram-owned setting through the existing reply delivery path; closing this issue before that implementation merges would be premature.
