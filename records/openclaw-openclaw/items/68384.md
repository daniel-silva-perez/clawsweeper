## Summary

Keep open. Current main still accepts full trimmed non-JSON stdout for a single `jsonOnly: false` exec SecretRef, so this PR's intended hardening is not implemented or obsolete. The PR is small and the Greptile review in the discussion supports the approach, but it touches the `src/secrets/` trust boundary, is related to still-open security-labeled #64821, and should get maintainer/secops review before merge.

## What This Changes

This PR changes `src/secrets/resolve.ts` so single-id `jsonOnly: false` exec SecretRef stdout must be valid JSON or exactly one non-empty plain-text line, and adds a focused regression test in `src/secrets/resolve.test.ts`.

## Best Possible Solution

Review and, if secops agrees with the fail-closed contract, land a resolver-boundary fix that rejects ambiguous non-JSON exec SecretRef stdout with regression coverage. Before closing related #64821, verify whether same-line polluted output is also impossible or separately guarded, and document any tightened `jsonOnly: false` plain-output contract.
