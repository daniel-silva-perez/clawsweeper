## Summary

Keep open. Current main has useful adjacent work for non-image `chat.send` attachments and media-understanding, but it still does not implement the requested direct MP4/WebM/MOV chat upload workflow: Control UI filters videos out, direct agent prompt attachments are image-only, and non-image Gateway attachments are staged as agent-readable media paths rather than delivered as native video content to provider chat APIs.

## Best Possible Solution

Keep this open until OpenClaw has a first-class direct video chat attachment design: allow and preview MP4/WebM/MOV in Control UI, define a video-capable chat attachment/content contract, persist transcript metadata, route by provider/model video capability with size/offload policy, preserve media-understanding fallback where appropriate, and cover UI, Gateway parsing, agent dispatch, provider/native media behavior, and docs with focused tests.

## Risks / Open Questions

- A narrow change that only removes the UI video filter would likely create a misleading UX: videos could be staged as generic media files without a provider-native video prompt contract, capability routing, transcript model, or size/offload policy.
- Video attachments have larger storage, WebSocket payload, provider-cost, sandbox staging, and media-root implications than current image/non-video document uploads, so this needs explicit product and security review before implementation.
- Existing media-understanding and `infer video describe` paths are useful alternatives, but they are not the same as first-class direct video chat upload to the active LLM.
