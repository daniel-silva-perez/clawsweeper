## Summary

Current main still does not support direct video upload to LLM chat models. The Control UI explicitly rejects video MIME types and common video extensions, tests lock that rejection in, and Gateway chat attachments still model native prompt payloads as images plus generic offloaded media rather than provider-native video content. The related draft PR #74082 is useful partial groundwork, but it explicitly leaves native provider delivery and product/security design unresolved.

## Best Possible Solution

Keep this issue open as the product/design tracker for first-class direct video chat attachments to LLMs. Maintainers should review #74082 for the UI/Gateway fallback groundwork, then decide and implement the native-provider contract across Control UI, Gateway attachment parsing/staging, model capability routing, provider dispatch, docs, and tests.

## Risks / Open Questions

- A UI-only change that accepts MP4/WebM/MOV would still stage video as generic media and could mislead users into expecting provider-native video understanding from the active LLM.
- Direct video-to-provider support needs explicit format/size limits, storage/offload behavior, preview/transcript metadata, provider capability routing, and model-specific content-block policy.
- Open draft PR #74082 already covers adjacent upload/preview/fallback work, so an automated replacement PR would risk duplicating active work and missing the remaining product/security design.
