## Summary

Keep this PR open. Current main still exposes brew-only skill installers in Linux container onboarding when Homebrew is absent, but the proposed patch is not quite merge-ready because it adds a narrower duplicate container detector instead of reusing the existing cgroup-aware container detection already present in core.

## What This Changes

The PR adds Linux-container detection to skill onboarding, hides brew-only missing skill installers when brew is absent, shows a container-specific note, and adds onboarding/container-context tests.

## Best Possible Solution

Keep the PR open, but update it to use or centralize the existing cgroup-aware container detector, preserve the focused onboarding UX tests, add detector coverage for the supported container signals, and resolve the policy relationship with #42303 before landing.

## Reproduction Assessment

Yes. A high-confidence code path exists: on current main, a missing brew-only skill with `brew` absent remains in `installable`, so Linux container onboarding can show it and then emit the Homebrew prompt after selection; I did not run tests because this review was constrained to read-only inspection.

## Solution Assessment

No, not as-is. The UX direction is narrow and plausible, but the detector should reuse the existing `src/infra/container-environment.ts` behavior instead of adding a partial duplicate, and the PR needs maintainer policy review against the competing Linuxbrew approach.
