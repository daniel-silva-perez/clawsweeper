## Summary

Keep open. Current main still lacks the requested pricing-threshold split, but this PR should not merge as-is because the branch is stale against current Control UI auth/freshness behavior and introduces concrete functional, security, CSS, and generated-artifact regressions.

## What This Changes

The PR adds `gateway.controlUi.chat.showPricingThreshold` and rewires Control UI chat context notices, bootstrap config, styles, config schema, generated docs, and tests to separate higher-rate pricing hints from model context-limit warnings.

## Best Possible Solution

Rebase or replace the branch on current main, add the chat setting inside the already-authorized bootstrap response, implement the threshold split in the current `ui/src/ui/chat/context-notice.ts` view-model path, use a fresh context/request usage signal for threshold claims, and update only the current generated-doc hash artifact.

## Reproduction Assessment

Yes. The auth regression is reproducible by comparing PR-head `src/gateway/control-ui.ts:347-376` with current main `src/gateway/control-ui.ts:759-808`; the pricing false positive is reproducible with high cumulative `inputTokens` and low or missing fresh `totalTokens`. The CSS issue is reproducible from the PR-head import order where `components.css` loads before `chat/layout.css`.

## Solution Assessment

No. The proposed direction is useful, but this branch is not the best current solution because it applies the behavior to stale Gateway/UI paths, bases pricing claims on aggregate session input, and carries obsolete generated artifacts instead of integrating with current main’s auth and freshness model.
