## Summary

Keep open. The docs gap is real, but the proposed troubleshooting text does not match current `main`: first-time allowlisted node commands are not dropped while node pairing is pending, paired-node command upgrades stay pinned to the approved command set until the node reconnects after approval, and rejecting a pending request does not issue a fresh token.

## What This Changes

This PR adds a Troubleshooting section to `docs/gateway/pairing.md` for dropped commands, missing pending requests, and stale tokens, and normalizes a few typographic quote/apostrophe characters.

## Best Possible Solution

Revise the docs-only change so the troubleshooting section matches current behavior: distinguish first-time pending node pairing from paired-node command upgrades, explain that command-upgrade approvals require the node host to reconnect or restart before the newly approved commands appear, and describe stale-token recovery as creating a new request or removing and re-pairing before approval.

## Reproduction Assessment

Yes. The docs gap is visible on current `main`, and the patch defects are reproducible by tracing `reconcileNodePairingOnConnect`, live command registration, `node.pair.approve`/`reject`, and the existing gateway tests that cover pending and reconnect command behavior.

## Solution Assessment

No. Adding troubleshooting near the pairing docs is the right surface, but this PR is not the best solution until its recovery steps are aligned with current source and tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Document reconnect before retrying upgraded commands:** `docs/gateway/pairing.md:207-211`
  - body: The new text tells operators to re-trigger a command after approval, but reconnect command upgrades are still pinned to the previously approved command set for the existing session. Approval updates the pairing store; the node host must reconnect or restart before the newly approved commands can appear, so this guidance will leave operators retrying against the old live command surface.
  - confidence: 0.9
- **[P3] Fix the stale-token recovery flow:** `docs/gateway/pairing.md:222-223`
  - body: This says to reject and re-approve the request to issue a fresh token, but rejecting a node-pair request removes it and does not issue a token. The recovery path should tell operators to make the node create a new request, then approve that request, or use the remove/re-pair flow.
  - confidence: 0.82
