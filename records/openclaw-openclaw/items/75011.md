## Summary

Keep open. The documentation gap is real, but the proposed troubleshooting text is not accurate enough for current gateway behavior because node command visibility is fixed from the connect-time reconciliation result; approval updates durable pairing state, not an already-connected node session.

## What This Changes

This PR updates docs/gateway/pairing.md by adding a Troubleshooting section for node pairing recovery scenarios and normalizing a few smart quotes/apostrophes.

## Best Possible Solution

Merge a docs-only update after the troubleshooting section matches current gateway behavior: keep the node-pairing/device-pairing distinction, explain that dropped commands are not replayed, and tell operators to reconnect or restart the node host after approving command upgrades before retrying commands.

## Reproduction Assessment

Yes. The docs gap is visible on current main in docs/gateway/pairing.md, and the patch defect is reproducible by tracing connect-time command reconciliation, node-pair approval, and node-list command selection in source plus existing tests.

## Solution Assessment

No, not as currently proposed. Adding troubleshooting near the node command gating docs is the right direction, but the recovery wording must include the reconnect/restart step and avoid implying approval alone refreshes a live node's command surface.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Document reconnect before retrying approved commands:** `docs/gateway/pairing.md:209-211`
  - body: The new troubleshooting text tells operators to re-trigger a command after approval, but current gateway code snapshots a node's effective commands when the node connects. Approval updates the pairing store only; an already-connected node still reports the old live command list until it reconnects, so the recovery step should say to reconnect or restart the node host before retrying.
  - confidence: 0.88
- **[P3] Fix stale-token recovery wording:** `docs/gateway/pairing.md:221-222`
  - body: This says to reject and re-approve the request to issue a fresh token, but rejecting a node-pair request removes it and does not issue a token. If the token is lost or stale after approval, the guidance should say to make the node create a new request, then approve that new request, or otherwise use the documented remove/re-pair flow.
  - confidence: 0.76
