## Summary

Keep open. The documentation gap is real, but the proposed troubleshooting wording does not match current gateway behavior closely enough to merge: it overstates pending-state command dropping, misses the reconnect/restart requirement for approved command upgrades, and gives an invalid stale-token recovery path.

## What This Changes

This PR adds a Troubleshooting section to `docs/gateway/pairing.md` for node pairing command visibility, missing pending requests, and stale tokens, plus normalizes a few smart quotes.

## Best Possible Solution

Rebase the docs update onto current main and rewrite the troubleshooting section to distinguish first-time node pairing, reconnect command upgrades, and token recovery: first-time allowlisted commands can be visible before node pairing exists, approved command upgrades require the node host to reconnect or restart, and stale/lost token recovery should use remove-and-re-pair or a fresh request approval flow.

## Reproduction Assessment

Yes. The docs gap is visible in `docs/gateway/pairing.md`, and the proposed wording defects are reproducible by comparing the PR diff with `node-connect-reconcile`, `node-pairing`, `node-catalog`, and the existing gateway tests on current main.

## Solution Assessment

No. Adding troubleshooting near the pairing docs is the right shape, but this wording is not the narrowest maintainable solution because it mixes stale pending-command assumptions with command-upgrade and token-rotation recovery paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Scope pending-command recovery to current behavior:** `docs/gateway/pairing.md:206-208`
  - body: Current main no longer supports this blanket premise. When no node-pairing record exists, `reconcileNodePairingOnConnect` returns the node's declared allowlisted commands, and the gateway test asserts those commands are available before node pairing exists. Please scope this recovery text to the actual command-upgrade or queued-invoke behavior instead of telling operators the commands were likely dropped during the pending state.
  - confidence: 0.86
- **[P2] Tell operators to reconnect after approving upgrades:** `docs/gateway/pairing.md:209-211`
  - body: The text tells operators to re-trigger after approval, but command upgrades are snapshotted when the node connects. Approval updates the pairing store only, and `node.list` prefers the connected session's live command list, so newly approved commands remain hidden until the node host reconnects or restarts.
  - confidence: 0.9
- **[P3] Fix stale-token recovery wording:** `docs/gateway/pairing.md:221-222`
  - body: Rejecting a node-pair request removes the pending entry and does not issue a token. If a token is stale or lost after approval, the guidance should tell operators to remove/re-pair or make the node create a fresh request and approve that request.
  - confidence: 0.82
