## Summary

Keep open. Current main still has the session-id isolation groundwork, but it does not expose `openclaw agent --new-session`, does not plumb a `newSession` option through the CLI command, and does not return a generated top-level `sessionId` in JSON output. The related #69108 proposed this exact feature, but it is closed and unmerged in the provided GitHub context.

## Best Possible Solution

Implement this as a narrow core CLI feature: add `--new-session` to `openclaw agent`, plumb `newSession` through `AgentCliOpts`, reject it with `--session-id`, generate one `crypto.randomUUID()` in `agentCliCommand`, pass the effective `sessionId` through gateway/local/fallback paths, include top-level `sessionId` in JSON output, and update parser, command, session-resolution, docs, and JSON-output tests. The closed #69108 is a useful reference, but the remaining work needs a fresh or reopened PR against current main.

## Risks / Open Questions

- An implementation should generate the UUID exactly once before gateway/local/fallback selection so one invocation cannot split across sessions.
- The new flag should be mutually exclusive with `--session-id` and should preserve existing default agent-main-session reuse when the flag is absent.
- The JSON shape should be covered for gateway success, `--local`, and embedded fallback so automation can reliably reconnect with the generated id.
