## Summary

Keep this issue open. Current main has the lower-level `--session-id` isolation behavior, but it still does not expose `openclaw agent --new-session`, generate a fresh session id in the CLI, or return a generated top-level `sessionId` contract in JSON output. The related #69108 described the right direction, but it is closed and unmerged in the provided context.

## Best Possible Solution

Implement this as a narrow core CLI feature: add `--new-session` to `openclaw agent`, add a `newSession` option to the command types, reject it with `--session-id`, generate one `crypto.randomUUID()` in `agentCliCommand`, pass that effective `sessionId` through Gateway/local/fallback paths, expose top-level `sessionId` in JSON output, and update parser, command, session-resolution, docs, and JSON-output tests. #69108 is useful reference material, but the remaining work needs a fresh or reopened PR against current main.

## Risks / Open Questions

- An implementation should generate the UUID exactly once before Gateway, local, or embedded fallback selection so one invocation cannot split across sessions.
- The new flag should be mutually exclusive with `--session-id` and should preserve existing default agent-main-session reuse when the flag is absent.
- The JSON shape needs coverage for Gateway success, `--local`, and embedded fallback so automation can reliably reconnect with the generated id.
