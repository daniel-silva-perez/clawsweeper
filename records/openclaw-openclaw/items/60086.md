## Summary

Keep open. Current main still lacks the requested operator-facing queue acknowledgement and stale approval recovery behavior, but this PR is not merge-ready because its stale-active drain check can strand existing queued follow-up work when the new enqueue attempt is rejected.

## What This Changes

The PR adds detailed follow-up enqueue results, active-run queued/backlog-full reply acknowledgements, stale approval-id recovery text, and focused auto-reply queue/approval tests.

## Best Possible Solution

Keep the PR open and preserve its user-facing direction, but repair the active-run liveness logic so stale snapshots restart the follow-up drain regardless of whether the current enqueue was accepted, then rebase the approval error handling onto the current shared helper.

## Reproduction Assessment

Yes. The branch regression is reproducible by combining `isRunActive() === false` with `enqueueFollowupRunDetailed` returning `duplicate` or `queue_full`; current main already has an e2e assertion that stale active snapshots restart the drain.

## Solution Assessment

No. The requested behavior is worth landing, but gating stale-run recovery on `enqueueResult.accepted` is not the safest implementation; queue acknowledgement should depend on the detailed result while drain recovery remains independent of that result.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Restart stale drains even when enqueue is rejected:** `src/auto-reply/reply/agent-runner.ts:274`
  - body: The stale-active fallback used to call `finalizeWithFollowup` whenever `isRunActive()` was false after the enqueue path. Gating that restart on `enqueueResult.accepted` means a stale active snapshot plus a `duplicate` or `queue_full` result leaves already queued work idle until a later accepted enqueue happens. Keep the liveness drain restart independent of the current enqueue result.
  - confidence: 0.91
