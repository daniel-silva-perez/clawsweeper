## Summary

Keep this PR open: current main still lacks the requested node WebSocket upgrade-header path, so the branch remains a useful implementation candidate for the Cloudflare Zero Trust workflow. The latest PR head is not ready to land because it still has credential-handling and precedence issues around env-derived headers, plus stale-branch/changelog cleanup work.

## What This Changes

The PR adds repeatable `--header` support for `openclaw node run` and `openclaw node install`, threads those headers through node config/service env and `GatewayClient`, redacts node service env in status output, and updates docs, changelog, and targeted tests.

## Best Possible Solution

Land this PR or an equivalent core change after rebasing it onto current main and making header source semantics explicit: repeatable CLI headers should win over env/config, env-derived secret headers should stay runtime-only unless the user explicitly asks to persist them, service status/logs/args must not expose secrets, docs must match the storage model, and the changelog needs proper attribution.

## Reproduction Assessment

Yes. The underlying gap has a clear mock-proxy reproduction in the PR body, and current main statically shows no way for node run/install to pass custom upgrade headers into `GatewayClient`; the PR-head regressions are reproducible from the inspected config/env precedence paths.

## Solution Assessment

No, not as currently written. The `--header`/`GatewayClient` surface is the right narrow direction, but this PR needs the env-secret persistence and CF env precedence issues fixed before it is the maintainable solution.
