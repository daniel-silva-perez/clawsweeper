## Summary

Current main still does not implement this PR’s central behavior: `openclaw node run` and `openclaw node install` have no repeatable `--header` option, node-host config/runtime has no header field or resolver, `GatewayClient` has no custom upgrade-header option, and the public node CLI docs still only document `OPENCLAW_GATEWAY_*` auth for node hosts. This PR remains the active implementation candidate for the Cloudflare Zero Trust WebSocket-upgrade workflow described in #42792, but it needs normal maintainer review before merge because the proposed path carries credential-bearing headers through CLI/config/service environment/status surfaces.

## Best Possible Solution

Keep this PR open and review it as the active implementation candidate for #42792. The best path is to land this PR or an equivalent core change that adds repeatable `--header` support to `openclaw node run` and `openclaw node install`, validates header syntax, defines config/env/flag precedence, forwards headers through `GatewayClient`, avoids leaking secret-bearing headers in program args/status/logs, updates `docs/cli/node.md`, fixes changelog attribution, and runs targeted node CLI, node-host, GatewayClient, daemon install/service-env, status redaction, and docs checks.
