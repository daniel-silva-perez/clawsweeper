## Summary

Keep open. This contributor PR is paired with the open linked bug report, and current main still has the metadata-write timestamp behavior the PR targets. The approach matches the existing preserve-activity precedent, but there is one low-severity test portability issue to address before normal maintainer review and validation continue.

## What This Changes

The PR adds a preserve-activity option to session-store entry updates, applies it to compaction and memory-flush metadata writes, stops compaction bookkeeping from writing `updatedAt`, and adds regression tests for legacy daily/idle reset behavior.

## Best Possible Solution

Land a narrow version of this fix after switching the new fixture root to `os.tmpdir()` and validating the touched session-store, compaction, and memory-flush paths. The core reset policy should stay unchanged; metadata-only writers should opt into preserve-activity behavior explicitly.

## Reproduction Assessment

Yes. The linked bug gives concrete daily and idle legacy-row scenarios, and current main source confirms the required path: compaction/flush metadata can advance `updatedAt` while reset freshness falls back to `updatedAt` when explicit lifecycle timestamps are missing.

## Solution Assessment

Mostly yes. Opting only metadata-only compaction and memory-flush writes into the existing preserve-activity merge policy is the narrowest maintainable fix; the safer adjustment is to make the new regression test use the platform temp directory before merge.

## Review Findings

Overall correctness: patch is correct

- **[P3] Use os.tmpdir() for the fixture root:** `src/auto-reply/reply/session.compaction-daily-reset-race.test.ts:25`

Import `node:os` and build the prefix from `os.tmpdir()`.
