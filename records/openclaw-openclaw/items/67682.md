## Summary

Keep this issue open. Current main still supports MCP credentials only on shared top-level `mcp.servers` entries, with no per-agent schema/type contract, runtime merge path, or cache fingerprint component for agent-specific env/header overlays. Related MCP tool-policy work does not solve same-server credential attribution.

## Best Possible Solution

Keep this open as a core MCP runtime/config feature. Add a schema-backed per-agent MCP credential/identity overlay, merge it before embedded Pi and CLI backend MCP config creation, include the effective overlay in runtime/cache fingerprints, redact secrets consistently, and add tests proving two agents can use the same MCP server id with different effective env/header credentials.

## Risks / Open Questions

- A future implementation must include the effective per-agent credential overlay in runtime/cache fingerprints and lifecycle decisions so one agent's credentials cannot bleed into another agent's MCP process or HTTP client.
- A duplicate-server-id workaround can preserve attribution operationally, but it changes tool names and forces per-agent tool-policy maintenance instead of solving same-server per-agent credentials.
