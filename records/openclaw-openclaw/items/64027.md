## Summary

Keep this PR open because current main still lacks Feishu in-place card update support, but the proposed patch is not merge-ready: it only patches when a new top-level `message_id` is present, while current Feishu card-action handling already supports `open_message_id`/`context.open_message_id` callback shapes, and the branch also mixes in unrelated Feishu reply-dispatcher and monitor comment test churn.

## What This Changes

The PR adds an optional Feishu card-action update-card payload, parses a callback message id, calls `updateCardFeishu` for cancel/confirm/quick actions, and also changes Feishu reply-dispatcher footer/session-store behavior while deleting monitor comment handler tests.

## Best Possible Solution

Request a focused revision that keeps the Feishu card-update wiring, normalizes the target card id from all callback shapes, isolates update failures from command dispatch, adds targeted unit/lifecycle tests for cancel/confirm/quick/malformed/missing-id cases, and removes unrelated reply footer and monitor comment test changes before live Feishu validation.

## Reproduction Assessment

Yes. The current-main gap reproduces by inspection of the card-action path: structured cancel/confirm/quick actions have no `uc` payload and never call `updateCardFeishu`; the PR-specific issue is reproducible with an accepted callback carrying `open_message_id` but no `message_id`, which the new helper would skip.

## Solution Assessment

No. The implementation direction is reasonable, but the narrow maintainable fix should normalize the existing `open_message_id` callback target as a fallback and keep unrelated Feishu reply-dispatcher/test churn out of this feature PR.
