## Summary

Keep this PR open. Current main still forwards Nextcloud Talk rich-object `object.content` as plain text, does not populate inbound media URLs or MIME hints, and still drops empty media-only inbound messages. The PR is a relevant implementation candidate for the linked Nextcloud Talk attachment bug, but it needs a focused revision before merge because the current diff promotes rich-object links into media fetch inputs without enough trusted-origin/MIME handling.

## Best Possible Solution

Keep the PR open and request a focused update: parse authenticated Nextcloud Talk rich-object `object.content`, resolve placeholders to display text, extract only trusted Nextcloud attachment URLs, forward matching `MediaUrls` and `MediaTypes`, allow media-only inbound delivery, preserve downstream SSRF protections, and add regression tests for file shares, malformed JSON fallback, private/loopback or off-origin rejection, extensionless image links, no-placeholder rich objects, and mention-placeholder overlap tracked by #66700.

## Risks / Open Questions

- Merging the PR as-is could turn user-controlled rich-object JSON text into outbound media fetches unless extracted links are constrained to the authenticated/trusted Nextcloud origin or an equivalent SSRF-safe policy.
- Without forwarding `MediaTypes` from rich-object `mimetype`, extensionless Nextcloud share links such as `/f/<id>` may still be treated as unknown and skipped by media understanding.
- The current PR parser behavior described in review comments can still surface raw JSON for valid rich objects whose `message` has no matching placeholders.
