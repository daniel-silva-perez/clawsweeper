## Summary

Keep this PR open. Current main still does not implement the intended Nextcloud Talk rich-object attachment parsing, and the PR remains a relevant implementation candidate, but it should be revised before merge for trusted-origin URL handling, MIME propagation, media-only delivery, and overlap with mention placeholder parsing.

## Best Possible Solution

Keep this PR open and request a focused revision: parse Nextcloud Talk rich-object `object.content` in the plugin, extract trusted Nextcloud attachment URLs, pass matching `MediaUrls` and `MediaTypes`, allow media-only inbound messages through, preserve downstream SSRF guard behavior, and add regression tests for file shares, malformed JSON fallback, private/loopback rejection, extensionless image links, no-placeholder rich objects, and mention placeholders covered by #66700.

## Risks / Open Questions

- Closing now would drop a still-relevant implementation candidate while current main continues to surface Nextcloud Talk file shares as raw JSON text or drop empty media-only messages.
- Merging the PR as-is could promote user-controlled rich-object links into downstream remote media fetch inputs without constraining them to the trusted Nextcloud origin.
- Without MIME propagation, extensionless Nextcloud share links such as `/f/<id>` may still be classified as unknown and skipped by image/audio/video handling.
- Open #66700 overlaps rich-object placeholder handling for Nextcloud Talk mentions, so the final parser should coordinate attachment display text with mention-gating behavior.
