## Summary

Keep this PR open. Current main still does not implement the intended Nextcloud Talk rich-object attachment parsing, and the PR remains a plausible implementation candidate, but it needs revision before merge for URL-origin safety, MIME propagation, media-only handling, and overlap with mention placeholder parsing.

## Best Possible Solution

Keep this PR open and request a focused revision: implement rich-object parsing in the Nextcloud Talk plugin, pass `MediaUrls` and matching `MediaTypes`, allow media-only inbound messages through the handler, constrain extracted links to trusted Nextcloud attachment origins with SSRF-safe behavior, and add regression tests for file shares, malformed JSON fallback, private/loopback rejection, extensionless image links, no-placeholder rich objects, and mention placeholders covered by #66700.

## Risks / Open Questions

- Closing this PR now would drop a still-relevant implementation candidate for a real bundled Nextcloud Talk plugin bug that current main has not fixed.
- Merging the current PR as-is could promote user-controlled rich-object JSON links into media fetch inputs; the revision should validate trusted Nextcloud attachment origins and keep downstream SSRF guard behavior intact.
- The PR's URL-only extraction may leave extensionless Nextcloud file links unselected by media-understanding unless MIME data is carried through as `MediaTypes`.
- Open #66700 overlaps rich-object placeholder handling for Nextcloud Talk mentions, so the final parser should coordinate attachment display text and mention-gating behavior.
