## Summary

Keep open. Current main still does not implement the intended Nextcloud Talk rich-object file-share parsing, and this PR remains a relevant implementation candidate, but the patch needs revision before it is safe and complete.

## Best Possible Solution

Keep this PR open and request a focused revision: parse Nextcloud Talk object.content rich objects in the plugin, extract only trusted Nextcloud attachment URLs, pass matching MediaUrls and MediaTypes, allow media-only inbound delivery, preserve downstream SSRF guard behavior, and add regression tests for file shares, malformed JSON fallback, private/loopback or off-origin rejection, extensionless image links, no-placeholder rich objects, and mention placeholders tracked by #66700.

## Risks / Open Questions

- Closing this PR now would drop a still-relevant implementation candidate while current main continues to expose Nextcloud Talk file shares as raw JSON text or drop empty media-only messages.
- Merging the PR as-is could promote user-controlled rich-object links into downstream media fetch inputs without constraining them to the trusted Nextcloud origin.
- Without MIME propagation, extensionless Nextcloud share URLs such as /f/<id> may still be classified as unknown and skipped by image/audio/video handling.
- The parser overlaps open #66700 mention-placeholder handling, so the final implementation should coordinate attachment display text with group mention detection.
