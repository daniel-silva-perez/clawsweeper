## Summary

Current main still does not implement the Nextcloud Talk rich-object file-share parsing this PR targets, so the PR is not obsolete and should remain open. The patch remains a plausible implementation candidate, but the review discussion identifies concrete issues to resolve before merge: trusted-origin/SSRF handling for promoted links, MIME propagation for extensionless Nextcloud share URLs, and valid rich-object display text when no placeholders match.

## Best Possible Solution

Keep this PR open and request a focused revision: parse Nextcloud Talk object.content rich objects in the plugin, resolve placeholders to display text, extract only trusted Nextcloud attachment URLs, forward matching MediaUrls and MediaTypes, allow media-only inbound delivery, preserve downstream SSRF protections, and add regression tests for file shares, malformed JSON fallback, private/loopback or off-origin rejection, extensionless image links, no-placeholder rich objects, and mention-placeholder overlap tracked by #66700.

## Risks / Open Questions

- Closing this PR now would drop a still-relevant implementation candidate while current main continues to expose Nextcloud Talk file shares as raw JSON text or drop empty media-only messages.
- Merging the PR as-is could promote user-controlled rich-object links into downstream media fetch inputs without constraining them to the trusted Nextcloud origin or preserving the intended SSRF guard posture.
- Without forwarding MIME hints alongside URLs, extensionless Nextcloud share links such as /f/<id> may still be classified as unknown and skipped by media understanding.
