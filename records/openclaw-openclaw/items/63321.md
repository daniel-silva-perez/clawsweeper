## Summary

Keep this PR open. Current main has not implemented the proposed malformed-output handling, but the PR should not be closed as implemented or merged as-is: the shared helper change would convert malformed `tailscale whois --json` output into an empty successful payload, changing the intended short error-cache path, and it would map malformed `tailscale funnel status --json` output to the disabled-Funnel setup path instead of an unknown/parse-failure path.

## Best Possible Solution

Keep the PR open and request a narrower fix: preserve parse failures as errors for whois/auth and unknown Funnel status, only returning an empty object at call sites where empty output is semantically valid. Add focused regression tests for malformed `tailscale status --json`, `tailscale whois --json` cache TTL behavior, and `tailscale funnel status --json` diagnostics before considering merge.

## Risks / Open Questions

- The PR changes a shared parser used by Tailscale hostname, status, Funnel, and whois paths; returning `{}` erases the difference between malformed output and a valid empty object.
- Malformed `tailscale whois --json` output would be cached through the normal success TTL path instead of the shorter error TTL path, which can prolong Tailscale Serve auth failures.
- Malformed `tailscale funnel status --json` output would be presented as disabled Funnel and can steer users into setup/remediation steps even when the real state is unknown.
- Security review: the diff is limited to `src/infra/tailscale.ts` and does not add dependencies, CI, scripts, or package-resolution changes; the main concern is auth-adjacent availability/diagnostics, not an observed auth bypass.
