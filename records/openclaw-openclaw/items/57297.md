## Summary

Keep this issue open. Current main has partial WebChat audio/TTS handling: media-bearing TTS tool results can be promoted into WebChat assistant audio content, and the UI can render audio controls. The specific requested observability is still missing: there are no structured `tts.generate.*`, `tts.route.selected`, or `tts.delivery.*` events, no correlation id tying the tool call to WebChat delivery, and no TTS-specific user-visible final delivery/playback status. The related playback bug #57296 is also still open, so this same-author companion should not be closed independently.

## Best Possible Solution

Keep this open and implement the remaining core WebChat/Gateway observability path: emit structured `tts.generate.started/succeeded`, `tts.route.selected`, and `tts.delivery.succeeded/failed` events with a run/tool correlation id and actionable reason codes; persist or broadcast enough delivery state for WebChat to show final TTS delivery/playback status; add focused coverage near `src/gateway/server-methods/chat.directive-tags.test.ts` and `ui/src/ui/chat/grouped-render.test.ts`; coordinate the fix with #57296 and the broader audio work tracked in #65859.
