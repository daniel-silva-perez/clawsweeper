## Summary

Current main still does not implement the requested security/API contract. Skills cannot declare schema-validated heartbeat-callable functions, heartbeat config has no remote structured invocation surface, and heartbeat execution still converts tasks/instructions into model prompt text before calling the agent runtime.

## What This Changes

The issue requests sandboxed skill functions so remote heartbeat endpoints can return validated structured calls instead of free-form natural-language instructions.

## Best Possible Solution

Keep this open for maintainer product/security design. A complete solution should add a versioned declared-callable contract, document and parse it, add a structured remote heartbeat fetch/parse path, validate call names and params against declared schemas, execute only through existing tool/plugin capability boundaries, and add regression tests proving injected free-form text never reaches the model as heartbeat instructions.

## Reproduction Assessment

Not applicable as a feature/security design request, but the absence is high-confidence: current-main source and docs show heartbeat remains prompt-driven and no sandboxed remote invocation contract exists.

## Solution Assessment

No, current main is not the best way to solve the issue because it still relies on model obedience around heartbeat prompt text. The safer path is a core-enforced structured contract before any ClawHub trust badge or service-specific plugin pattern depends on it.
