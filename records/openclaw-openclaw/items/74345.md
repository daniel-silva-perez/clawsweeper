## Summary

Keep open. This is a fresh, detailed gateway reliability report, and current main still contains the ACP task-maintenance close paths, session-store scanning, and session-write-lock behavior that match the reported fingerprint. Later commits add adjacent Telegram, Discord, and gateway client-start mitigations, but I found no source, test, or release evidence that the idle ACP cleanup loop and multi-minute event-loop saturation are fixed.

## What This Changes

The issue requests investigation and repair of an idle gateway regression where task-registry ACP cleanup repeatedly fails, holds session-store locks far past their budget, saturates the event loop, and stalls Telegram/Discord transports.

## Best Possible Solution

Fix the ACP task-maintenance lifecycle so stale/orphaned ACP sessions are cleaned up with bounded non-blocking work, failed runtime closes cannot loop every sweep without backoff or quarantine, stale metadata can be cleared safely, active persistent bindings are preserved, and regression tests cover failed-close loops plus session-lock/liveness behavior.
