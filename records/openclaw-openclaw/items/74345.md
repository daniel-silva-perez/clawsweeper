## Summary

Keep open: current main still contains the reported ACP task-maintenance close/list/session-lock surfaces, and the visible post-report commits do not prove the idle ACP cleanup loop or long event-loop stalls are fixed.

## What This Changes

The issue reports a severe gateway reliability regression in v2026.4.27 where idle task-registry ACP cleanup saturates the event loop, leaks ACP sessions, holds session-write locks, and stalls Telegram/Discord transports.

## Best Possible Solution

Make ACP task maintenance bounded and non-blocking: timebox runtime close/list operations, stop retrying the same failed cleanup every sweep without backoff or quarantine, safely clear stale parent-owned metadata when the backend is unavailable, preserve active persistent bindings, and add regression coverage for stale ACP metadata plus lock/event-loop health.

## Reproduction Assessment

Yes for the reported build: the issue gives exact checkout/build/systemd steps plus five correlated log signals on v2026.4.27/9d8de70. Current-main reproduction is still unconfirmed because this review was read-only and did not run a live gateway with the reporter's ACP/channel setup.

## Solution Assessment

Unclear because there is no proposed patch. The right direction is a focused current-main reproduce/bisect followed by a narrow ACP maintenance/runtime fix, not closing this based on adjacent Telegram, subagent, or plugin-registry changes.
