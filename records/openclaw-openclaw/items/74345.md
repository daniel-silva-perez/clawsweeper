## Summary

Keep open. Current main still has the same ACP task-maintenance close loop, ACP session-store scan, close-session runtime/meta-write path, and session-write-lock watchdog fingerprint described in the report; the later visible main change bounds subagent orphan recovery, not this ACP cleanup path. The open cross-reference #74860 is related gateway event-loop work, but its own report says the fingerprint is distinct rather than a superseding fix.

## What This Changes

The issue reports that OpenClaw 2026.4.27 at commit 9d8de70 pegs the gateway event loop at idle, leaks ACP sessions through repeated task-registry maintenance close failures, holds session-write locks for hundreds of seconds, and stalls Telegram/Discord until rolling back to a8b64b7.

## Best Possible Solution

Current main should gain bounded ACP maintenance cleanup: nonblocking or batched ACP session-store scanning, per-session close timeout/backoff or quarantine after repeated failures, preservation of active persistent bindings, and regression coverage proving stale ACP metadata cannot repeatedly peg the gateway event loop or leak sessions.

## Reproduction Assessment

Unclear for current main. The issue has a strong bad-commit reproduction and symptom fingerprint, but this review did not run the live systemd gateway with ACP wrappers and stale session metadata against current main.

## Solution Assessment

Unclear. The visible post-report subagent recovery cap is not the right complete fix for this ACP-specific failure; the safer route is a targeted ACP maintenance fix after confirming the current-main fingerprint, rather than treating #74860 or the Telegram WSL fix as superseding this report.
