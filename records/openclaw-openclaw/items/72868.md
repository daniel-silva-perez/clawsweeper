## Summary

Keep open. Current main still has the reported shared Google thinking sanitizer gap, so the PR addresses a real latency/cost issue, but the current diff should not merge until the forced `thinkingBudget=0` path is positively gated to models that support that budget control; the already-posted Gemma-prefix and PR-description review comments were addressed, and #73416 is adjacent rather than superseding.

## What This Changes

The PR changes shared Google thinking payload sanitization to create or force `thinkingConfig.thinkingBudget = 0` when `thinkingLevel` is `off`, imports the shared Gemma 4 model-id matcher, and adds regression tests for `config`, `generationConfig`, and no-container payloads.

## Best Possible Solution

Revise this PR in place: keep the Gemini 2.5 Flash/Flash-Lite thinking-off injection, factor a positive helper for thinkingBudget-capable Google model IDs, use it in both the container and no-container branches, and add negative coverage for an older or custom Google model ID.

## Reproduction Assessment

Yes. A deterministic unit-level reproduction is `sanitizeGoogleThinkingPayload({ payload: { config: {} }, modelId: "gemini-2.5-flash", thinkingLevel: "off" })`; current main leaves the config empty because the sanitizer only rewrites an existing `thinkingConfig`.

## Solution Assessment

No, not as currently written. The shared sanitizer is the right layer, but the branch should positively target models documented to accept `thinkingBudget=0` instead of every non-excluded Google model ID.
