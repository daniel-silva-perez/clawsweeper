## Summary

Keep open. Current main still only mutates existing Google thinking containers, so this PR addresses a real Gemini 2.5 Flash off-mode gap, but the diff should be revised before merge because the forced-zero path is broader than the documented thinkingBudget model family. The related #73416 is separate Vertex/Gemini CLI guard work, not a superseding implementation.

## What This Changes

The PR changes shared Google thinking payload sanitization to create or force `thinkingConfig.thinkingBudget = 0` when `thinkingLevel` is `off`, adds a prefixed Gemma 4 guard, and adds regression tests for `config`, `generationConfig`, and no-container payloads.

## Best Possible Solution

Keep this PR open and ask for a small revision that positively gates forced `thinkingBudget=0` injection to models documented or deliberately allowlisted to accept that budget control, while preserving the existing exclusions for 2.5 Pro, Gemini 3, and Gemma 4. Leave the Vertex/Gemini CLI wrapper widening to #73416.

## Reproduction Assessment

Yes. A deterministic unit-level reproduction is `sanitizeGoogleThinkingPayload({ payload: { config: {} }, modelId: "gemini-2.5-flash", thinkingLevel: "off" })`; current main leaves `config` empty, while the PR's new tests assert `thinkingConfig: { thinkingBudget: 0 }`.

## Solution Assessment

No, not as currently written. The shared sanitizer is the right layer for the fix, but the guard should positively match thinkingBudget-capable models instead of applying to every non-excluded Google model ID.
