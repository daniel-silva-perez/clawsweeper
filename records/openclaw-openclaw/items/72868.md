## Summary

Keep this PR open. It fixes a real current-main gap for Gemini 2.5 Flash thinking-off payloads, and the author already addressed the earlier prefixed-Gemma review note, but the current branch still injects `thinkingBudget: 0` for every non-excluded Google model ID instead of positively limiting that path to Gemini 2.5 budget-capable models; it also lacks the required user-facing changelog entry.

## What This Changes

The PR changes shared Google thinking payload sanitization to create or force `thinkingConfig.thinkingBudget = 0` when `thinkingLevel` is `off`, updates Gemma 4 model-id matching, and adds regression tests for config, generationConfig, and no-container payloads.

## Best Possible Solution

Revise this PR rather than close it: keep the shared sanitizer fix for Gemini 2.5 Flash/Flash-Lite, but make the forced-zero and no-container fallback paths positively require the Gemini 2.5 thinking-budget family while preserving existing Gemini 2.5 Pro, Gemini 3, and Gemma 4 behavior. Add a changelog entry under Unreleased Fixes and leave the Vertex/Gemini CLI wrapper widening to #73416.

## Reproduction Assessment

Yes. A deterministic unit reproduction for the original bug is `sanitizeGoogleThinkingPayload({ payload: { config: {} }, modelId: "gemini-2.5-flash", thinkingLevel: "off" })`, which current main leaves unchanged while the PR fixes it. The introduced blocker is also reproducible at unit level by using `modelId: "gemini-2.0-flash"`, where the PR would inject `thinkingBudget: 0` outside the documented Gemini 2.5 budget family.

## Solution Assessment

No, not as currently written. The shared sanitizer is the right layer, but the maintainable fix is to gate forced `thinkingBudget=0` on budget-capable Gemini 2.5 models instead of applying it to all non-excluded Google model IDs.
