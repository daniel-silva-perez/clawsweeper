## Summary

Keep this PR open. Current main still lacks the thumbnail-strip/add-more affordance, but the latest PR head still has two `.onChange(of: pickerItems)` observers on the same state, which can duplicate attachment loading before the async loader clears the selection.

## What This Changes

The PR modifies the shared SwiftUI chat composer to replace filename attachment capsules with shrinking image thumbnail tiles, remove overlays, and a trailing add-more picker tile.

## Best Possible Solution

Keep this PR as the canonical UI implementation candidate, but update the branch so `pickerItems` is observed exactly once from a stable parent view with an empty-selection guard, then validate the focused iPhone add-more, multi-select, remove, and send flows.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on the latest PR head: once the attachment strip and toolbar picker are both mounted, both observers can receive the same `pickerItems` change before the async loader resets state.

## Solution Assessment

No. The feature direction is appropriate, but the current PR head is not the best implementation because duplicating shared picker-state observers is less maintainable and can duplicate attachments; a single stable observer is the safer fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Move picker observation to one stable owner:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/ChatComposer.swift:254-255`
  - body: The add-more picker adds a second `.onChange` on the same `pickerItems` state while the toolbar picker still observes it too. When attachments exist and both pickers are mounted, a new selection can start two `loadPhotosPickerItems` tasks before the loader clears `pickerItems`, appending duplicate attachments. Observe `pickerItems` once from an always-present parent and ignore the empty reset.
