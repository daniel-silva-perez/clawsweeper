## Summary

Keep open. Current main handles the adjacent Linux no-display crash by falling back to headless mode for implicit starts and by returning a clearer error for explicit headed starts, but it still does not make visible WSL2 Chrome launches work by passing `DISPLAY=:0`. The maintainer comment on April 25 explicitly called out that distinction, and the related upstream PRs for this exact fallback are closed and unmerged.

## What This Changes

Request to make browser control launch visible Chrome in WSL2 by setting `DISPLAY=:0` for non-headless local Chrome spawns when no display variable is inherited.

## Best Possible Solution

Add a narrow browser plugin launch-path fix for WSL2 local managed headed starts: detect WSL2 through the existing plugin SDK runtime helper, provide `DISPLAY=:0` only when no display variable is inherited, avoid changing headless or remote CDP behavior, and document the WSL2 visible-browser case beside the existing Linux troubleshooting guidance.

## Reproduction Assessment

Yes. The issue has a high-confidence code-level reproduction: on Linux with a local managed profile, `headless=false`, and no `DISPLAY` or `WAYLAND_DISPLAY`, current main returns the explicit headed no-display error before spawn; if that guard were bypassed, the Chrome spawn env still would not add `DISPLAY=:0`.

## Solution Assessment

No current-main solution fully solves this report. The shipped headless fallback is a good adjacent mitigation for implicit starts, but the narrow maintainable solution for this issue is WSL2-specific spawn environment handling that preserves existing display configuration.
