## Summary

Keep open. Current main still fires the `message:received` and `message:sent` internal hooks without consuming hook-produced `event.messages`, and open PR #41375 is the maintainer-labeled closing fix path for this issue.

## What This Changes

The issue reports that internal hook handlers for `message:received` and `message:sent` can append content to `event.messages`, but that content is not delivered back to the user, observed on Feishu in OpenClaw 2026.2.26.

## Best Possible Solution

Keep this issue open until PR #41375 or an equivalent maintainer-approved patch lands. The end state should drain hook-produced replies only where OpenClaw has a safe reply target, add regression coverage for that delivery contract, and update the hooks docs so `event.messages` no longer promises delivery on lifecycle-only surfaces.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to register a `message:received` hook that pushes to `event.messages` and send a Feishu or other routed inbound message; current main constructs and fires the event without retaining or draining the messages array.

## Solution Assessment

Yes. The best path is to review, update, and land PR #41375 or an equivalent maintainer fix rather than close this issue or create another parallel repair PR, because that PR already uses closing syntax and carries maintainer labels.
