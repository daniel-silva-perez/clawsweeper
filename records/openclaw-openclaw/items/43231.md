## Summary

Keep this issue open. Current main has partial Telegram forum-topic support: it persists and surfaces the current topic name beside the numeric topic id. It still does not provide the requested agent-visible name-to-topic_id resolver, topic catalog/list action, forum_topics metadata map, or name-based routing to another topic by title.

## Best Possible Solution

Keep this open and implement the missing capability in the bundled Telegram plugin. Reuse the persisted topic-name cache, add an agent-visible topic catalog or resolver action with topic_id/title/source/freshness/closed state, optionally support explicit operator aliases, route only exact unique matches, and fall back to numeric topic ids or clarification on ambiguity. Add focused tests for listing/resolution, duplicate names, stale or closed topics, persistence, and send routing, then update Telegram channel and message docs.

## Risks / Open Questions

- A resolver built only from observed Telegram service messages can be incomplete after cold starts, bot downtime, or topics that never emitted metadata to the bot.
- Duplicate, renamed, closed, or reopened topics need explicit ambiguity and freshness handling before name-based sends can be safe.
- Prompt-only changes would not solve delivery: the message action surface, target parsing, docs, and tests need to agree on exact and ambiguous matches.
