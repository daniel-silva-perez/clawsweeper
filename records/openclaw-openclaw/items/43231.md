## Summary

Keep this issue open. Current main has useful partial Telegram forum support from #65973/#66107: it learns, persists, and exposes the current topic name next to the numeric topic id. It still does not expose a forum topic catalog/resolver or support name-based outbound routing to another topic, which is the central request here.

## Best Possible Solution

Keep this issue open and implement the remaining capability in the bundled Telegram plugin. Reuse the persisted topic-name cache, expose an agent-visible topic catalog or `topic-list`/`topic-resolve` action with `topic_id`, title, source/freshness, and closed state, support only exact unique name matches for outbound sends, ask for clarification on ambiguity, and add focused tests for duplicate names, stale/closed topics, persistence, metadata, and send routing.

## Risks / Open Questions

- A resolver backed only by observed Telegram service messages can be incomplete after cold starts, bot downtime, cache eviction, or topics whose metadata was never seen by the bot.
- Duplicate, renamed, closed, reopened, or stale topic names need explicit ambiguity and freshness handling before name-based sends are safe.
- Fuzzy or partial name matching could cause exactly the misrouting this issue is trying to prevent; exact unique matches plus clarification fallback are the safer path.
