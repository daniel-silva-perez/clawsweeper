## Summary

Keep this issue open. Current main has partial Telegram forum-topic support: it learns, persists, and exposes the current topic name beside the numeric topic id, but it still does not provide a topic-name-to-topic_id resolver, a forum_topics map/catalog, a list/resolve action, or name-based outbound routing to another Telegram topic.

## Best Possible Solution

Keep this issue open and implement the missing capability in the bundled Telegram plugin. Reuse the persisted topic-name cache, add an agent-visible topic catalog or resolver action with topic_id/title/source/freshness/closed state, support only exact unique matches for sends, fall back to clarification on ambiguity, and update target parsing/action docs plus focused tests for duplicate names, stale/closed topics, persistence, and send routing.

## Risks / Open Questions

- A resolver backed only by observed Telegram service messages can be incomplete after cold starts, bot downtime, or topics that never emitted metadata to the bot.
- Duplicate, renamed, closed, or reopened topics need explicit ambiguity and freshness handling before name-based sends are safe.
- The current persisted cache reduces restart loss, so the remaining gap is not persistence; it is agent-visible catalog/resolution and safe outbound routing by title.
