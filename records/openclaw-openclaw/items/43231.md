## Summary

Keep this issue open. Current main has partial Telegram forum support: it persists and exposes the current topic name next to the current numeric topic id, but it still does not provide a topic catalog, name-to-id resolver, or name-based outbound routing for another forum topic.

## Best Possible Solution

Implement the remaining capability in the bundled Telegram plugin: expose an agent-visible topic catalog or topic-list/topic-resolve action backed by the persisted topic-name cache, include topic id, title, closed state, and freshness/source metadata, and allow outbound sends by exact unique topic name only with clarification on ambiguity. Add focused tests for duplicate names, stale/closed topics, persistence, metadata exposure, and send routing.

## Risks / Open Questions

- A resolver backed only by observed Telegram service messages may be incomplete after bot downtime, cache eviction, or topics the bot never observed.
- Duplicate, renamed, closed, reopened, or stale topic names need explicit ambiguity and freshness handling before name-based sends are safe.
- Fuzzy or partial name matching could cause the misrouting this issue is trying to prevent; exact unique matches with clarification fallback are safer.
