## Summary

Keep open. Current main still has only partial downstream cleanup for failed child runs; it does not specifically handle transient Docker sandbox container creation failure with explicit gateway logging, sandbox registry failure state, immediate subagent/session error metadata, or a retry. Related #39248 overlaps the broader non-main sandbox spawn symptoms, but this report adds distinct Docker create/start observability and recovery requirements.

## Best Possible Solution

Keep this open and fix the sandboxed subagent path directly: add a regression that forces Docker container create/start to fail after an accepted `sessions_spawn` using `agents.defaults.sandbox.mode: "non-main"`; then log a sanitized Docker error, persist a failed child run/session status visible to `subagents list` or session metadata, avoid indefinite running state, and optionally retry create/start once. Keep #39248 linked for broader non-main sandbox startup coverage.
