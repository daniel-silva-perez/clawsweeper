## Summary

Keep open. Current main has partial downstream cleanup for failed or lost subagent runs, but it does not fully implement the reported Docker sandbox creation failure behavior: `sessions_spawn` can be accepted before the child run resolves the sandbox/container, Docker create/start has no retry or explicit failure logging/registry entry, and there is no targeted regression proof for transient Docker container creation failure under `agents.defaults.sandbox.mode: "non-main"`. Related #39248 overlaps the non-main sandbox spawn area, but this report adds distinct Docker logging, metadata, error-state, and retry requirements.

## Best Possible Solution

Keep this open and fix the sandboxed subagent path directly: add a regression that forces Docker container creation/start to fail after an accepted `sessions_spawn` with `agents.defaults.sandbox.mode: "non-main"`, then make the async child run log a sanitized Docker error, persist a failed session/subagent/task status visible to `subagents list`, avoid indefinite running state, and optionally retry create/start once. Keep #39248 linked for broader non-main sandbox startup coverage, but track this issue for the Docker failure surfacing requirement.

## Evidence

## Risks / Open Questions

- The exact failure is transient and Docker-environment dependent, so the best next proof is a forced-failure regression around the sandbox backend rather than manual reproduction.
- Existing sweeper and wait handling may reduce how long a run appears zombie-like, but they do not satisfy the requested Docker-specific log, metadata, and retry behavior.
