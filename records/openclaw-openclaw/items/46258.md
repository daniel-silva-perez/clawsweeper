## Summary

Keep open: current main still does not implement Telegram `ackReactionTiming`, and this PR is a live implementation candidate. It is not merge-ready because the public config generated outputs are stale and the deferred Telegram ack path bypasses the existing supported-emoji validation.

## What This Changes

The PR adds shared and Telegram-scoped `ackReactionTiming` config, defers Telegram ack/status reactions until agent run start, updates docs/help/types/schemas, and adds Telegram-focused tests.

## Best Possible Solution

Update the PR on current main, keep the Telegram implementation inside the plugin boundary, preserve supported-emoji validation for immediate and deferred reactions, regenerate the config schema/channel metadata/docs baselines, and land the shared timing contract consistently with the Slack and Discord companion work.

## Reproduction Assessment

Not applicable as a feature PR rather than a bug report. The review findings are source-reproducible: PR head adds public config without generated outputs, and the deferred path sends raw `ackReactionValue` while current main filters Telegram ack emoji first.

## Solution Assessment

No, not yet. The runtime direction is narrow and the Greptile `setThinking()` gap appears addressed, but the PR needs generated artifact updates and a validation-preserving deferred ack path before it is the best merge path.
