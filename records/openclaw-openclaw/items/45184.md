## Summary

Keep open. Current main still only filters Matrix traffic by sender account and only resolves mentions for the current bot account, so a human message that explicitly mentions another configured Matrix bot can still dispatch for an account with `requireMention: false`. The stale label was removed after ClawSweeper review, and the related broad Matrix participation PR does not specifically implement this deterministic pre-dispatch skip.

## What This Changes

Request an opt-in Matrix pre-dispatch policy that skips non-mentioned bot accounts when a room message explicitly mentions another configured Matrix account.

## Best Possible Solution

Add a backward-compatible Matrix room/account opt-in that runs before typing, media/history work, session recording, and LLM dispatch. It should skip only when a visible explicit mention targets another configured Matrix bot account, while preserving self mentions, DMs, commands, `@room`, and existing `allowBots` behavior.

## Reproduction Assessment

Yes. A high-confidence static reproduction is a Matrix room event from a human sender whose `m.mentions.user_ids` or visible Matrix mention targets another configured bot, while the current account has `requireMention: false` and no self mention; current main has no gate that drops that event before dispatch.

## Solution Assessment

Yes. A narrow deterministic Matrix plugin policy is the best way to solve the user-visible typing/token problem; prompt-only suppression or a late AI participation gate would still allow unnecessary dispatch work.
