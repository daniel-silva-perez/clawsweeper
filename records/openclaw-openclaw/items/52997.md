## Summary

Keep open. Current main still intentionally isolates Telegram forum topics and DM topics into separate session keys, while only partially satisfying the request by preserving thread/topic metadata for prompts, delivery, and session-list rows. The requested opt-in shared session mode for all topics/threads is not present in the Telegram config schema, docs, or routing tests, and the related reports in the provided context cover narrower topic targeting or UI labels rather than this shared-context behavior.

## Best Possible Solution

Keep this open for a Telegram-plugin-owned design. If maintainers accept the product behavior, add an explicit opt-in setting under the Telegram group/topic and DM-thread config surface, preserve `MessageThreadId`/topic metadata and reply delivery targets, decide how shared group sessions interact with non-main sandboxing and session-tool visibility, and cover forum topics plus DM topics with regression tests and docs.

## Risks / Open Questions

- A shared session mode for forum topics would deliberately cross an isolation boundary that currently affects context, sandbox expectations for non-main group sessions, session-tool visibility, and delivery last-route behavior. It needs explicit opt-in semantics and tests for access control, reply routing, and metadata preservation.
- Existing metadata plumbing (`MessageThreadId`, `TopicName`, `deliveryContext.threadId`) is useful but does not solve the core request by itself; maintainers should avoid closing this as a topic-label or command-targeting duplicate.
