## Summary

Keep open. Current main implements Discord native reply references via `replyToMode`, but it does not implement the requested opt-in visible `<@userId>` injection for guild replies. The Discord config schema is strict and has no `autoMentionReply`-style option, outbound delivery only carries `replyToId`/`replyToMode` into Discord sends, and the send path posts `message_reference` without adding a mention or `allowed_mentions.replied_user` policy. The existing discussion also points to the Discord plugin owning this formatting behavior, which matches the current code boundary rather than a cleanup close.

## Best Possible Solution

Keep this issue open and implement the behavior in the bundled Discord plugin or a clearly documented Discord plugin extension point. A good fix would add an opt-in `channels.discord.autoMentionReply` or equivalent schema/docs field, resolve the original author from inbound reply context or by fetching the referenced Discord message, prepend `<@authorId>` only to the first visible outbound guild reply when absent, and add focused tests for `replyToMode: first/all/batched`, DMs, existing mentions, chunking, media/components, and streaming preview/final fallback.

## Risks / Open Questions

- Mention injection would change Discord notification behavior, so it should remain opt-in and be guarded for guild-only replies, duplicate existing mentions, DMs, media/component sends, streaming previews, and `replyToMode` variants.
- An external message_sending hook currently receives `replyToId`/`threadId` but not the original reply author ID, so closing this as ClawHub/plugin work would be premature unless a plugin fetches the referenced Discord message itself or the hook/API is expanded.
