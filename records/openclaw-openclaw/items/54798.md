## Summary

Keep open. Current main has a per-cron-job running marker that prevents the same cron job from being manually/timer-started twice, but it does not provide the requested shared atomic counter, `openclaw sessions kv incr` surface, named `sessions_spawn` mutex, or agent-facing file-lock primitive for concurrent cron sub-agents sharing daily counters. The closed related skip-if-running request helps one overlap class, but it does not supersede this cross-agent shared-state coordination request.

## Best Possible Solution

Keep this open for a focused coordination primitive decision. The strongest implementation path is a narrow, generic atomic state API, such as a workspace/session-scoped `kv incr` tool/RPC/CLI backed by existing lock-safe store helpers, with concurrency tests covering two cron-spawned sub-agents incrementing/resetting the same daily counter. If maintainers prefer keeping it outside core, document a ClawHub plugin path using `api.registerTool` plus `openclaw/plugin-sdk/file-lock`, but that should be an explicit product call because the current repo does not yet provide the requested built-in surface.
