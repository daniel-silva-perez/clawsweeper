## Summary

Keep open for maintainer judgment. Current main intentionally keeps the shared Control UI gateway token in sessionStorage, scrubs legacy localStorage tokens, and documents that choice; it also has a safer persistent reconnect path through paired device tokens. The requested one-line localStorage change would alter admin-auth persistence policy, so this should not be auto-closed or queued as a routine fix PR.

## What This Changes

The issue requests that the Control UI persist the gateway token across browser restarts by storing it in localStorage instead of sessionStorage.

## Best Possible Solution

Keep the shared gateway token session-scoped and, if the browser-restart UX is still painful, improve the safer persistent path: paired device-token reconnect, `openclaw dashboard` bootstrap guidance, or a maintainer-approved explicit “remember this device” design that does not silently store the shared admin secret in localStorage.

## Reproduction Assessment

Yes for the storage observation: current source and tests show the shared token is stored in sessionStorage and omitted from localStorage, so the visible token field can be empty after the browser session ends. The stronger claim that users must always re-enter it is only partial on current main because paired device tokens can reconnect without the shared token after a successful approved connection.

## Solution Assessment

No: the suggested one-line localStorage change is not the best default fix because it changes the security posture for an admin surface and conflicts with current docs/tests. A safer solution needs maintainer product/security review around persistent device auth or explicit opt-in UX.
