## Summary

Keep this issue open. Current main intentionally keeps the shared Control UI gateway token in sessionStorage, scrubs legacy localStorage tokens, and documents that choice as a security boundary for the admin dashboard. A cached paired-device token can reduce re-entry after approval, but that is not the same as persisting the shared gateway token across browser restarts, so this remains a product/security decision rather than an implemented-on-main cleanup close.

## Best Possible Solution

Keep this open for maintainer product/security judgment. A good fix would either document that browser-close shared-token re-entry is intentional and rely on paired-device auth, or add an explicit opt-in 'remember this browser' design with logout/rotation behavior, tests, and docs; it should not silently swap the shared-token storage helper to localStorage.

## Risks / Open Questions

- Changing the suggested helper from sessionStorage to localStorage would reintroduce long-lived storage for an admin shared secret and contradict current docs, changelog, and regression tests.
- The adjacent cached device-token path has its own rotation/logout semantics, including the separately tracked #65408 concern, so any durable remember-token UX should be reviewed as an auth product/security change.
