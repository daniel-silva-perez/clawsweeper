## Summary

Keep open. Current main has partial TUI state separation, but it still exposes raw `streaming` and `running` activity strings and converts yielded or silent finals back to idle, so the operator-facing distinction requested here is not implemented yet.

## What This Changes

The issue requests clearer TUI session-state labels that distinguish idle, token generation, non-streaming work, yielded subagent waits, reconnecting, and disconnected states.

## Best Possible Solution

Add a narrow TUI status-state contract that derives Idle, Generating, Working, Waiting, Reconnecting, and Disconnected from connection state, chat deltas, lifecycle/tool events, and yielded/liveness metadata, then update focused TUI tests and the TUI docs.

## Reproduction Assessment

Yes. The code path is high-confidence: chat deltas set `streaming`, empty finals set `idle`, and the yielded metadata produced by the runner is not consumed by the TUI status renderer.

## Solution Assessment

Yes, a focused TUI/gateway status repair is the best path. The core `sessions_yield` behavior already exists, so the maintainable fix is to surface that state additively rather than redesign subagent orchestration.
