## Summary

Keep this open. Current main has some status separation, but it still does not implement the requested TUI state model: token deltas are still labeled as streaming, lifecycle/tool work is labeled running, yielded sessions are finalized back to idle rather than shown as waiting for subagents, and disconnect handling does not expose a distinct reconnecting state.

## What This Changes

Request to replace the TUI's ambiguous connected/streaming status with distinct idle, generating, working, waiting, reconnecting, and disconnected labels.

## Best Possible Solution

Introduce a small typed TUI activity-state model and feed it from gateway chat/agent metadata: idle when connected and no active run, generating on model token deltas, working on lifecycle/tool execution, waiting when a run yields or is paused for subagent completion, reconnecting while the gateway client is recovering, and disconnected when the gateway is unreachable. Cover the state transitions with focused TUI event-handler tests and update the TUI docs/changelog for the user-facing labels.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is available by simulating a chat run that sends a delta and then a yielded final/wait snapshot: current TUI code marks the delta as streaming and then finalizes the run to idle because yielded/liveness metadata is not represented in Tui ChatEvent handling.

## Solution Assessment

No, current main is not the best solution yet. The narrow maintainable fix is to propagate existing yielded/paused metadata through the chat event or session status path and map it through an explicit TUI status state, rather than only renaming the existing free-form strings.
