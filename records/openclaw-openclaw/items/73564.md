## Summary

Current main still has the reported Signal attachment bug: Signal RPC responses are capped by a hardcoded 1 MiB client limit before `getAttachment` can return its base64 payload, while `channels.signal.mediaMaxMb` is only applied after the RPC result is already available. The related items in the provided context are adjacent Signal timeout/attachment work, but they do not supersede or implement this HTTP response-cap fix.

## Best Possible Solution

Keep this issue open and fix it in the Signal plugin. Thread a response-size limit through `signalRpcRequest`/`requestSignalHttpText` for `getAttachment`, derive that limit from effective per-account `mediaMaxMb` with base64 and JSON headroom or expose a documented Signal-specific cap, and add regression coverage proving allowed multi-megabyte attachments pass while responses above the effective cap still fail.

## Risks / Open Questions

- Fixing this by removing the HTTP cap entirely would weaken the client memory guard; the safer path is a bounded per-call cap derived from the configured media limit plus base64/JSON headroom, or an explicit Signal config key with conservative defaults.
