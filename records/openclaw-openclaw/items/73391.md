## Summary

Keep open. Current main still has the lazy CLI parent-option loss and eager-registration race described by this PR, so the PR is not obsolete and should not be closed as implemented. The PR is a focused implementation candidate with relevant regression tests, but the fallback-argv review concern and async public helper contract should be resolved or explicitly accepted before merge.

## Best Possible Solution

Keep this PR open as the active implementation candidate. Review and resolve the fallback-argv concern, confirm the async registration contract is acceptable for the plugin SDK surface, then land after the targeted CLI/browser tests plus the appropriate changed gate are green. Current main does not yet implement the intended fix.

## Evidence

- **Current main lazy reparse still starts from the immediate parent:** `reparseProgramFromActionArgs` sets `root` to `actionCommand?.parent ?? program`, reads `rawArgs` from that immediate parent, builds fallback argv from only the action command name, and then reparses `program`. For nested lazy commands such as `openclaw browser --browser-profile nuan status`, that is the path the PR is trying to replace with root-program reparse.
  - file: [src/cli/program/action-reparse.ts:10](https://github.com/openclaw/openclaw/blob/dc541662f81503aecc08c193bf9ddf3208a95abb/src/cli/program/action-reparse.ts#L10)

- **Commander dependency behavior matches the reported root cause:** OpenClaw pins Commander 14.0.3. Upstream Commander 14.0.3 initializes `rawArgs` in `_prepareUserArgs` for `parse`/`parseAsync`, clears `rawArgs` in `restoreStateBeforeParse`, and `_dispatchSubcommand` calls the subcommand's `_parseCommand` directly. That supports the PR's claim that the immediate subcommand parent can lack root `rawArgs` during lazy reparse.

## Risks / Open Questions

- The Greptile review comment identifies a plausible fallback regression when root `rawArgs` is empty for nested programmatic usage; maintainers should either fix the fallback path to include intermediate command segments or explicitly accept that edge case.
- `registerCommandGroups` is exported through `src/plugin-sdk/cli-runtime.ts`, so changing it from sync to async should get an explicit compatibility check for external plugin callers, especially eager-mode callers that may not await the returned promise.
