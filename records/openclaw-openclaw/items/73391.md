## Summary

Keep open. Current `main` still has both CLI lazy-runtime behaviors this PR is meant to fix: lazy nested subcommand reparse can lose parent options, and eager registration can return before async command-group imports finish. The PR is not obsolete, and the latest provided PR context shows the Greptile fallback-path concern was addressed by a follow-up commit.

## Best Possible Solution

Keep this PR open as the active implementation candidate. Review the async `plugin-sdk/cli-runtime` compatibility surface, verify the latest fallback-path commit is acceptable, then land after the targeted CLI/browser tests and the appropriate changed gate are green. Current `main` does not yet implement the intended fix.

## Evidence

- **Current main lazy reparse still starts from the immediate parent:** `reparseProgramFromActionArgs` sets `root` to `actionCommand?.parent ?? program`, reads `rawArgs` from that command, builds fallback argv from only the leaf action command, and calls `program.parseAsync(...)`. It does not walk to the root command or reparse from the root, so it still matches the reported parent-option loss path for nested lazy commands.
  - file: [src/cli/program/action-reparse.ts:10](https://github.com/openclaw/openclaw/blob/2a0af6754e2bac4b48feaaf33932681bc99a29f8/src/cli/program/action-reparse.ts#L10)
