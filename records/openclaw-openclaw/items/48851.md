## Summary

Keep this PR open. Current main still lacks API/model call-count plumbing for `/status`, the response usage footer, session persistence, and embedded runner metadata, so the requested feature is not implemented. The PR remains a useful implementation candidate and did respond to earlier bot feedback by adding followup persistence and tool-loop counting, but it is stale against the current status/usage split and still has correctness issues around usage-less calls and stale persisted counts.

## What This Changes

The PR adds per-run API call counting through embedded runner/subscription metadata, session entry persistence, `/status`, the response usage footer, and focused unit tests.

## Best Possible Solution

Keep the PR open as a contribution candidate, but port it onto current main and make the call-count contract explicit. The maintainable end state is a latest-run model/API call counter that is captured at the model request/response boundary, propagated through main and followup runs, persisted or cleared without stale values, rendered in `/status` and the response footer, and covered by tests for tool loops, followups, zero/no-usage responses, and current-main helper paths.

## Reproduction Assessment

Yes for the current-main gap: static inspection of current main shows no call-count field in status rendering, response footer formatting, session persistence, or embedded runner metadata. The PR-introduced stale-count path is also reproducible by persisting a run with a positive callCount, then a later run/update with zero or missing callCount and observing that the merged session entry can retain the old value.

## Solution Assessment

No, not as submitted. The implementation direction is reasonable, but the patch should be rebased onto the current split modules and should count calls independently of usage metadata while updating or clearing the latest-run session field on every relevant run.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Count calls before filtering usage metadata:** `src/agents/pi-embedded-subscribe.ts:275`
  - body: `callCount` is incremented only after the usage path accepts nonzero usage metadata. Providers or error paths that make a model request but omit usage will report no call, which breaks the PR's API-call accounting goal and the per-call billing use case.
  - confidence: 0.82
- **[P2] Overwrite callCount when the latest run has no count:** `src/auto-reply/reply/session-usage.ts:101-103`
  - body: The new session field is only patched when `params.callCount > 0`. Because session entries are merged, a later run that passes `0` or omits the field can keep the previous run's count and make `/status` show stale current-turn calls.
  - confidence: 0.86
