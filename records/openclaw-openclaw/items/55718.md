## Summary

Keep open. Current main still supports manual per-session labels and channel account `name` display fields, but it does not implement the requested account-config-driven default label for newly created inbound channel sessions. I found no `displayLabel` channel-account config field, and the inbound session metadata path records origin metadata from the message context without looking up the configured account name/display label.

## Best Possible Solution

Keep this open as a valid core/channel-session enhancement. The likely implementation is to reuse account `name` as the default session label, or add `displayLabel` as a documented alias if maintainers prefer that public spelling, then update inbound session metadata creation to resolve the channel/account from the message context and seed `session.label` or `displayName` only for new/unlabeled sessions. Add regression coverage around `recordSessionMetaFromInbound`, gateway session rows, and Control UI session display, plus docs for channel account configuration.

## Risks / Open Questions

- Adding a separate `displayLabel` while `name` already exists could create two overlapping account display-name concepts unless maintainers choose an alias/migration policy.
- The implementation should only seed labels for new or unlabeled sessions so user-edited `session.label` values are not overwritten.
- This touches the shared channel/session boundary, so config schema, docs, gateway session rows, and channel/plugin account surfaces need to stay aligned.
