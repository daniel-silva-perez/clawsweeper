## Summary

Keep this PR open. Current main still lacks the Chat Completions tool protocol support targeted here: request parsing has no `tools`, `tool_choice`, or `tool_call_id`, the agent command path does not forward `clientTools`, and response paths cannot emit Chat Completions `tool_calls`. The PR is a focused implementation candidate for the still-open #54405 gap and related #54174 streaming behavior, but the stream finalization concern should be resolved before landing.

## Best Possible Solution

Keep this PR open and review or rework it on current main. The best path is to fix the streaming finalization logic so Chat Completions tool-call streams wait for the command result or another proven-safe completion condition before finalizing, keep the focused request parsing/`clientTools` forwarding/response mapping, preserve the regression coverage in `src/gateway/openai-http.test.ts`, and then land it as the implementation for #54405 and the related #54174 behavior.

## Risks / Open Questions

- The PR patch's one-second finalize-without-result timer can still close a streaming response with `finish_reason: "stop"` before slow `pendingToolCalls` metadata arrives, which would break the client tool follow-up loop intermittently.
- Client-supplied function schemas are being accepted on a full operator-access Gateway endpoint. That aligns with the existing Responses client-tool model and does not add server-side third-party execution, but validation, name-conflict behavior, and the delegated-client boundary should remain explicit in review.
