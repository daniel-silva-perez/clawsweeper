## Summary

Keep open. Current main still has the one-shot cron re-trigger bug, and the PR’s cron hunk is the right direction, but this branch should not merge as-is because it bundles unrelated WhatsApp, transcript, secrets, and exec-env changes and introduces a blocking transcript repair regression.

## What This Changes

The PR adds `skipAtIfAlreadyRan: true` to the cron timer and due-job paths, and also changes WhatsApp append handling, session transcript pairing repair, gateway secrets startup retries, and provider-key filtering for exec child environments.

## Best Possible Solution

Land a narrow cron-only fix that enables `skipAtIfAlreadyRan` for the normal timer and due-job paths, with regression coverage for completed one-shot jobs and transient retry eligibility; split the unrelated WhatsApp, transcript, secrets, and exec-env changes into separate reviews.

## Reproduction Assessment

Yes. On current main, an enabled completed `schedule.kind === "at"` job with a past `nextRunAtMs` and terminal `lastStatus` can still pass the ordinary `nowMs >= nextRunAtMs` check through `onTimer` or `runDueJobs` because those paths do not enable the existing guard.

## Solution Assessment

No. The cron call-site change is the right narrow fix, but this PR is not the best merge vehicle because it bundles unrelated behavior and introduces an invalid transcript pairing path.

Full review comments:

- **[P1] Rewrite sanitized tool-result IDs before pairing:** `src/agents/session-transcript-repair.ts:446-449`
  - body: When the fallback maps a sanitized `rawId` to the original tool-call ID, the code stores the unchanged `toolResult`. The repaired transcript can then contain an assistant tool call with the original ID next to a paired tool result that still carries the sanitized ID, which strict replay validation can reject. Rewrite the result ID to the matched original before adding it to `spanResultsById`.
  - confidence: 0.91
- **[P3] Report total startup attempts consistently:** `src/gateway/server.impl.ts:470-471`
  - body: The retry loop performs the initial attempt plus three retries, but the warning logs `Attempt 3/3 failed, retrying` before the fourth total attempt. That makes startup diagnostics misleading; log total attempts as 4 or label the field as retry count.
  - confidence: 0.78
