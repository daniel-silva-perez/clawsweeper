## Summary

Keep open. The cron bug is still present on current main and the PR's cron direction is valid, but the branch should not merge as-is because it bundles unrelated fixes and includes a blocking transcript-repair regression.

## What This Changes

The PR branch adds the completed-one-shot guard to cron timer/due-job collection and also changes WhatsApp append handling, session transcript pairing repair, gateway secrets startup retries, and exec environment provider-key filtering.

## Best Possible Solution

Land a narrow cron-only fix that enables `skipAtIfAlreadyRan` for the normal timer and due-job paths, with regression coverage for completed one-shot jobs and transient retry eligibility; split the unrelated WhatsApp, transcript, secrets, and exec-env changes into separate reviews.

## Reproduction Assessment

Yes. Current main's `onTimer` and `runDueJobs` paths collect runnable jobs without `skipAtIfAlreadyRan`, so an enabled completed `at` job with a past `nextRunAtMs` and terminal `lastStatus` can pass the ordinary `nowMs >= nextRunAtMs` check again.

## Solution Assessment

No. Passing `skipAtIfAlreadyRan: true` at the two cron call sites is the right narrow direction, but this PR is not the best vehicle because it mixes unrelated changes and introduces an invalid transcript pairing path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Rewrite sanitized tool-result IDs before pairing:** `src/agents/session-transcript-repair.ts:439-443`
  - body: When the fallback maps a sanitized `rawId` to the original tool-call ID, the code stores the unchanged `toolResult`. The repaired transcript can then contain an assistant tool call with the original ID next to a paired tool result that still carries the sanitized ID, which strict replay validation can reject. Rewrite the result ID to the matched original before adding it to `spanResultsById`.
  - confidence: 0.91
- **[P3] Report total startup attempts consistently:** `src/gateway/server.impl.ts:470-471`
  - body: The retry loop performs the initial attempt plus three retries, but the warning logs `Attempt 3/3 failed, retrying` before the fourth total attempt. That makes startup diagnostics misleading; log total attempts as 4 or label the field as retry count.
  - confidence: 0.78
