## Summary

Keep open, but do not merge this branch as-is. Current main still has the normal cron tick gap the PR targets, but the PR bundles unrelated WhatsApp, transcript, gateway-secrets, and exec-env changes, and the transcript repair hunk introduces a blocking mismatched tool-result ID regression.

## What This Changes

The PR branch changes cron one-shot scheduling, WhatsApp append-message access control, session transcript tool-result repair, gateway secrets startup retries, and exec environment API-key stripping.

## Best Possible Solution

Land a narrow cron-only fix with regression tests, then split or separately review the unrelated WhatsApp, transcript, secrets retry, and exec-env changes under their own issues or PRs.

## Reproduction Assessment

Yes. A completed enabled one-shot `at` job with a past `nextRunAtMs` and `lastStatus: "ok"` is still runnable through current main's `onTimer` and `runDueJobs` paths because those call sites do not enable `skipAtIfAlreadyRan`.

## Solution Assessment

No. Passing `skipAtIfAlreadyRan: true` in the two cron call sites is the right narrow direction, but this branch is not the best vehicle because it mixes unrelated fixes and includes a blocking transcript repair regression.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Rewrite sanitized tool-result IDs before pairing:** `src/agents/session-transcript-repair.ts:439-443`
  - body: When the fallback matches a sanitized `rawId` to an original tool-call ID, the code stores the unchanged `toolResult`. The repaired transcript can then contain an assistant tool call with the original ID next to a tool result that still carries the sanitized ID, which strict replay validation can reject. Rewrite the result ID to the matched original before adding it to `spanResultsById`.
  - confidence: 0.91
- **[P3] Report total startup attempts consistently:** `src/gateway/server.impl.ts:470-471`
  - body: The retry loop makes the initial attempt plus three retries, but the warning logs `Attempt 3/3 failed, retrying` before a fourth total attempt. That makes startup diagnostics misleading when operators are watching secret-provider failures; log either total attempts as `4` or label the field as retry count.
  - confidence: 0.78
