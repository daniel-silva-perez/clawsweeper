## Summary

Keep open. Current main contains useful mitigations for the JSON5 manifest parsing, repeated mirror-decision source reads, and one FileHandle leak class, but I do not see high-confidence proof that the steady-state bundled runtime mirror/runtime-deps hot loop is fixed. The named loader path still re-enters mirror preparation under the runtime-deps filesystem lock, the root dist materialization helper can still remove and relink/copy materialized chunks on repeated calls, and the retained runtime-deps marker still records only specs rather than a completed/freshness marker.

## Best Possible Solution

Keep this issue open as the focused plugin-loader/runtime-deps hot-loop tracker. The best next step is a narrow loader/runtime-deps fix that makes steady-state bundled runtime dist mirror preparation idempotent: avoid repeated materialized root chunk rewrites when source and target signatures match, add a process-level or disk-level freshness marker for prepareBundledPluginRuntimeDistMirror where safe, keep cache invalidation tied to source fingerprints and clearPluginLoaderCache/test resets, and add regression coverage proving repeated bundled plugin loads do not re-mirror/rewrite unchanged root chunks or repeatedly parse manifests. After that, retest a packaged Gateway with Telegram or WhatsApp enabled and confirm event-loop responsiveness/CPU settles.

## Repair Work Prompt

Fix the remaining steady-state bundled plugin runtime mirror hot-loop risk tracked by this issue. Current main already caches manifest loads and mirror materialization decisions, but `prepareBundledPluginRuntimeDistMirror` still re-enters under the runtime-deps filesystem lock and `materializeBundledRuntimeMirrorDistFile` still removes/relinks or copies materialized root dist chunks on repeated unchanged loads. Implement a conservative idempotence/freshness guard so unchanged materialized root dist files are not rewritten and the dist mirror prepare path can skip avoidable work when the source root/file signatures are unchanged. Cover both the active loader path in `src/plugins/loader.ts` and the extracted helper path in `src/plugins/bundled-runtime-root.ts`, or refactor carefully so they share the same behavior without broad plugin API changes. Keep runtime-deps manifest compatibility; do not change package versions, plugin public contracts, or install command security flags. Add focused tests proving repeated bundled runtime root preparation/load does not rewrite unchanged materialized root chunks, still refreshes when source files change, and keeps the existing manifest/FileHandle cache tests passing. Add a single-line changelog fix entry under Unreleased with reporter credit; use `Refs` unless the PR includes packaged Gateway/profiler proof sufficient to say `Fixes`.
