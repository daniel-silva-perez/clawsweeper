## Summary

Keep open. Current main has a recent manifest-load cache that references this issue, but it only addresses repeated JSON/JSON5 manifest parsing. I did not find a direct fix or regression proof for the reported steady-state bundled-runtime mirror/stat loop, per-turn staged runtime-deps work, or missing completion-marker behavior described by multiple reporters and related #73647.

## Best Possible Solution

Keep this issue as the focused tracker unless maintainers explicitly consolidate it with #73647 or another canonical regression. The best fix path is targeted instrumentation around resolvePluginLoadCacheContext, getCachedPluginRegistry/setCachedPluginRegistry, prepareBundledPluginRuntimeDistMirror, ensureBundledPluginRuntimeDeps, loadPluginManifestRegistry, staged runtime-deps install roots, and gateway/channel/provider call sites, followed by regression proof that steady-state startup, RPCs, channel starts, and agent turns do not repeatedly re-mirror bundled runtime roots, reparse manifests, rewalk staged deps, or rerun npm install for already-staged dependencies. If confirmed, normalize compatible cache keys, reuse the Gateway metadata snapshot consistently, add a completion marker/hash for staged deps, and add a per-process debounce or memo for runtime mirror preparation.
