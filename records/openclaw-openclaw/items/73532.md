## Summary

Keep open. The report is recent, detailed, and distinct from the broader gateway CPU-spin issue. Current main has startup and metadata-cache improvements, but the implicated loader paths still perform bundled runtime mirror scans and manifest parsing on cache misses, and I did not find a direct fix, regression test, or shipped release note proving the post-startup hot loop is gone.

## Best Possible Solution

Keep this issue open as the focused tracker for the plugin-loader hot loop. The best next step is targeted instrumentation around resolvePluginLoadCacheContext, getCachedPluginRegistry/setCachedPluginRegistry, prepareBundledPluginRuntimeDistMirror, discovery/loadPluginManifestRegistry, and Gateway steady-state RPC/provider/tool call sites, followed by a regression test or profiler proof that repeated post-startup calls do not re-mirror bundled runtime roots or reparse manifests unnecessarily. If confirmed, normalize compatible cache keys and add a per-process debounce or memo for steady-state runtime mirror preparation.
