## Summary

Keep open. Current main has useful mitigations for repeated manifest parsing and metadata reuse, but I did not find source, test, changelog, release, or history evidence proving the reported steady-state bundled-runtime mirror/runtime-deps hot loop is fixed. The issue remains recent, detailed, and still more specific than the adjacent closed reports.

## Best Possible Solution

Keep this issue open as the focused plugin-loader/runtime-deps hot-loop tracker unless maintainers explicitly consolidate it with a newer canonical regression. The best fix path is targeted instrumentation around resolvePluginLoadCacheContext, getCachedPluginRegistry/setCachedPluginRegistry, prepareBundledPluginRuntimeDistMirror, ensureBundledPluginRuntimeDeps, staged install roots, and gateway/channel call sites, followed by regression or profiler proof that steady-state startup, RPCs, channel starts, and agent turns do not repeatedly re-mirror dist roots, reparse manifests, rewalk staged deps, or rerun npm install. If confirmed, normalize compatible cache keys, reuse Gateway metadata snapshots consistently, add a durable completion/freshness marker or per-process memo/debounce for runtime-deps/mirror preparation, and wire the gateway CPU scenario harness into release proof.
