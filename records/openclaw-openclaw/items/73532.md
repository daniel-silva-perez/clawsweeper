## Summary

Keep open. Current main contains several real runtime-deps, mirror, and manifest-cache mitigations, but the issue thread still has specific post-mitigation reports and current code retains reachable request-time plugin loading, runtime-deps, and mirror-lock paths that need affected-host proof before this can be closed.

## What This Changes

The issue asks OpenClaw to stop a gateway/plugin-loader hot loop where bundled runtime-deps mirror preparation and JSON5 manifest parsing repeatedly saturate CPU, starve the event loop, and break Telegram, WS RPC, and OpenAI-compatible request paths.

## Best Possible Solution

Keep this as the canonical runtime-deps/plugin-loader starvation tracker until maintainers retest an affected packaged host on current main or the next package. If it still reproduces, add focused timing/cache-hit instrumentation around plugin registry loading, request-time plugin tools, runtime-deps materialization, dist-mirror locks, and channel startup, then land the smallest proven fix while leaving narrower follow-ups such as #74860 and #75032 to track split-out behavior.

## Reproduction Assessment

Yes for the released affected range: the thread gives concrete Linux, Docker, and Raspberry Pi setups with plugin counts, profiles, logs, and v2026.4.24-v2026.4.27 behavior. It is unclear for exact current main at aa9db998 because this review did not run the affected host and current main has later mitigations than some reports.

## Solution Assessment

Unclear. Current main has sensible mitigations, but they are not yet proven as the complete fix for request-time stalls and repeated staging; maintainer-led retest plus narrower root-cause selection is safer than closing or queueing a one-shot repair branch.
