## Summary

Keep open. The report is recent, detailed, and not superseded by the broader open gateway CPU-spin issue. Current main has useful startup/cache improvements, but the loader still performs the reported synchronous mirror and manifest work on cache misses, and I did not find a direct fix or regression proof for the indefinite post-startup hot loop described here.

## Best Possible Solution

Keep this issue open and use it as the focused tracker for the plugin-loader hot loop. The best next step is targeted instrumentation around resolvePluginLoadCacheContext, getCachedPluginRegistry/setCachedPluginRegistry, prepareBundledPluginRuntimeDistMirror, and loadPluginManifestRegistry during steady-state Gateway RPC/status/provider calls, followed by a regression benchmark or test proving repeated post-startup calls do not re-mirror bundled runtime dist roots or reparse manifests unnecessarily. If confirmed, normalize or share compatible cache keys and add a per-process debounce/memo for steady-state runtime mirror preparation.
