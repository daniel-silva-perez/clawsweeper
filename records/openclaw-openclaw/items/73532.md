## Summary

Keep open. Current main contains several runtime-deps and plugin-loader mitigations, but the latest discussion reports residual post-v2026.4.27 CPU, event-loop delay, RSS growth, and repeated staging behavior, and I did not find affected-host proof that the broader gateway starvation regression is fully fixed.

## What This Changes

The issue requests a fix for a bundled plugin loader/runtime-deps hot loop that repeatedly mirrors/parses plugin runtime state, pegs CPU, grows RSS, and starves Gateway WebSocket and channel polling work.

## Best Possible Solution

Keep this as the canonical runtime-deps/plugin-loader starvation tracker until maintainers can retest an affected packaged install on current main or the next package. If it still reproduces, add focused timing/cache instrumentation around runtime-deps plan materialization, mirror-lock work, plugin registry cache hit/miss behavior, channel startup/login, and agent-turn tool creation, then land the smallest fix that eliminates repeated staging and long event-loop stalls with packaged Linux/Docker proof.

## Reproduction Assessment

Yes for released packaged installs: the issue and comments provide concrete Linux/Raspberry Pi/systemd setups, plugin counts, profiles, logs, and v2026.4.24-v2026.4.27 behavior. ## Solution Assessment

Unclear. The current mitigations are directionally correct, but they are not yet the best proven end-to-end fix because recent evidence still reports event-loop stalls and repeated staging after earlier fixes; maintainer retest and narrower root-cause selection should precede closure or an automated repair branch.
