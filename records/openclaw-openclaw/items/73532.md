## Summary

Keep open. Current main contains useful partial mitigations for repeated manifest parsing, runtime-mirror dist-file reads, and one FileHandle leak path, but the issue remains recent, concrete, and not proven fixed. A maintainer explicitly landed 75df09b9ec as a narrow slice and said not to close because it was not proof that the full steady-state runtime-deps mirror/stat loop is gone.

## Best Possible Solution

Keep this open as the focused plugin-loader/runtime-deps hot-loop tracker. The best next step is maintainer instrumentation and release-path repro proof around resolvePluginLoadCacheContext, getCachedPluginRegistry/setCachedPluginRegistry, prepareBundledPluginRuntimeDistMirror, ensureBundledPluginRuntimeDeps, staged-deps scans, and channel/gateway call sites. If the loop is confirmed, fix it in the plugin loader/runtime-deps paths with stable compatible cache keys, a stronger completion/freshness marker or per-process debounce, bounded/yielding mirror verification, and regression/profiler coverage showing steady-state packaged installs do not repeatedly re-mirror, re-stat, reparse manifests, or rerun npm installs.
