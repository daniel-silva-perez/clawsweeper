## Summary

Keep open. This remains a recent, detailed, multi-reporter plugin-loader/staged-runtime-deps regression report. Current main has adjacent plugin metadata/startup mitigations and lifecycle tracing, but I did not find a direct fix, release note, regression test, or source change proving the reported steady-state hot loop around bundled runtime mirror preparation and manifest registry rebuilds is gone.

## Best Possible Solution

Keep this issue as the focused tracker unless maintainers explicitly consolidate it with #73647 or a newer canonical regression. The best fix path is targeted instrumentation around resolvePluginLoadCacheContext, getCachedPluginRegistry/setCachedPluginRegistry, prepareBundledPluginRuntimeDistMirror, loadPluginManifestRegistry, staged runtime-deps install roots, and gateway/channel/provider call sites, followed by a regression test or profiler proof that steady-state RPCs, channel starts, and agent turns do not repeatedly re-mirror bundled runtime roots, reparse manifests, or rerun npm install for already-staged deps. If confirmed, normalize compatible cache keys, reuse the Gateway metadata snapshot consistently, and add a per-process debounce or memo for runtime mirror preparation.
