## Summary

Keep open. Current main has a useful manifest-load cache mitigation, but I did not find a source, test, release note, or history trail proving the reported steady-state plugin-loader/runtime-deps mirror hot loop is fixed. The issue remains recent, detailed, and corroborated by #73647 and multiple same-day reporter comments.

## Best Possible Solution

Keep this issue open as the focused plugin-loader/runtime-deps hot-loop tracker unless maintainers explicitly consolidate it with #73647 or a newer canonical regression. The best fix path is targeted instrumentation around resolvePluginLoadCacheContext, getCachedPluginRegistry/setCachedPluginRegistry, prepareBundledPluginRuntimeDistMirror, loadPluginManifestRegistry, staged runtime-deps install roots, and gateway/channel/provider call sites, followed by a regression test or profiler proof that steady-state RPCs, channel starts, and agent turns do not repeatedly re-mirror bundled runtime roots, reparse manifests, or rerun npm install for already-staged deps. If confirmed, normalize compatible cache keys, consistently reuse Gateway plugin metadata snapshots, and add a durable completion/freshness marker or per-process memo/debounce for runtime mirror preparation.
