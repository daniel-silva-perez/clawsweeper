## Summary

Keep this PR open. Current main still lacks any `:thread:` handling in silent-reply classification, so the direct-thread rewrite bug is not implemented on main; however, the proposed branch still needs maintainer follow-up because its parser-based check can classify caller-shaped direct keys as internal and suppress the direct-message rewrite path.

## What This Changes

The PR adds a thread-session check to silent-reply conversation classification, adds regression tests for direct and cron thread session keys, and adds a changelog fix entry.

## Best Possible Solution

Land a refined fix that classifies trusted internal thread sessions as non-rewriting without letting arbitrary direct session keys opt out of direct silent-reply rewrites, and cover direct-thread, cron-thread, and parser/trust-boundary edge cases in tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Validate thread keys before suppressing direct rewrites:** `src/shared/silent-reply-policy.ts:81-83`
  - body: The new branch treats any key for which `parseThreadSessionSuffix()` returns a `threadId` as `internal`, but that helper only finds the last `:thread:` marker and accepts the remaining tail. Since explicit session creation preserves caller-provided key structure, a direct key shaped like `agent:main:telegram:direct:123:thread:x` can bypass the direct silent-reply rewrite path. Please gate this on a trusted generated-thread shape, or add stricter validation for this classifier use.
  - confidence: 0.84
