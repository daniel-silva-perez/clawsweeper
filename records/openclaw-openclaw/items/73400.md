## Summary

Keep open. Current main still has the reported silent-reply classification gap, and this PR is a useful focused fix attempt, but the branch still needs maintainer review because it treats a generic parsed thread suffix as internal provenance and leaves group/channel-thread ordering inconsistent with the stated behavior.

## What This Changes

The PR imports `parseThreadSessionSuffix`, classifies parsed thread-suffixed silent-reply session keys as internal, adds direct/cron thread regression tests, and adds a changelog entry.

## Best Possible Solution

Land a refined narrow fix that treats trusted generated internal thread sessions as non-rewriting/internal without letting arbitrary caller-supplied or user-facing direct thread session keys opt out of direct silent-reply rewrites, with coverage for direct-thread, cron-thread, group/channel-thread, and malformed suffix cases.

## Reproduction Assessment

Yes. Current-main behavior is reproducible by checking `classifySilentReplyConversationType({ sessionKey: "agent:main:telegram:direct:123:thread:99" })`, which follows the direct branch, and by following `createOutboundPayloadPlan` where direct `NO_REPLY` payloads are rewritten; this review did not run tests because the task is explicitly read-only.

## Solution Assessment

No. The PR is close in scope, but using the generic thread parser as proof of internal provenance is not the narrowest maintainable fix; a safer solution should gate on trusted generated/internal thread shape or metadata and explicitly decide how user-facing channel threads are classified.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Validate thread provenance before returning internal:** `src/shared/silent-reply-policy.ts:81-83`
  - body: The new branch returns `internal` whenever `parseThreadSessionSuffix()` yields a `threadId`, but that parser accepts any non-empty final `:thread:` tail and `sessions.create` can preserve caller-shaped keys. A direct session key ending in `:thread:x` can therefore bypass the direct silent-reply rewrite path; gate this on trusted generated/internal session shape or explicit metadata before returning `internal`.
  - confidence: 0.86
- **[P3] Move trusted thread detection before group policy:** `src/shared/silent-reply-policy.ts:72-73`
  - body: The PR says threaded sessions should be internal regardless of parent surface, but the group/channel branch still runs before the thread check. With configured group policies that differ from internal policies, a `...:group:...:thread:...` key can still take group behavior, so either move the trusted-thread predicate above group/channel or narrow and test the intended behavior.
  - confidence: 0.72
