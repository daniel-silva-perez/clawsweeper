## Summary

Keep this PR open. Current main still classifies `:direct:`/`:dm:` session keys before considering `:thread:`, so the direct-thread silent-reply bug described by the PR is not implemented on main. The PR is a plausible targeted fix, but it still needs normal review, especially around the structured thread parsing/security concern raised in review.

## Best Possible Solution

Review and refine this PR rather than closing it. The likely path is to add a safe `:thread:` classification before direct/DM rewrites, preserve existing channel/group behavior intentionally, and add regression tests for direct-thread silent replies plus any parser/security edge cases maintainers want to lock down.

## Risks / Open Questions

- The PR's final diff imports `parseThreadSessionSuffix`, but that helper currently accepts any last `:thread:` marker and the full trailing tail as `threadId`; reviewers should decide whether silent-reply classification needs stricter trusted-shape validation before merge.
- The proposed check still runs after the `:group:`/`:channel:` branch in the PR diff, so group/channel thread keys would remain classified as `group`; today that has the same default silent/rewrite behavior as `internal`, but it does not fully match the PR's stated invariant.
