## Summary

Keep open. The underlying Gemini CLI OAuth proxy/hung-exit bug still appears valid on current main, but this PR is not mergeable as-is: current maintainer triage already called the branch too broad/red, and the diff mixes the useful Gemini/dispatcher work with unrelated Discord, LINE, Nostr, and plugin-sdk test changes that introduce concrete regressions.

## What This Changes

The PR attempts to route Gemini CLI OAuth guarded fetches through environment proxies, prefer immediate dispatcher teardown for proxy dispatchers, add proxy keep-alive changes, and also changes Discord proxy handling, LINE setup, Nostr bus/key helpers, and plugin-sdk tests.

## Best Possible Solution

Keep the bug open and replace this with a narrow Gemini CLI OAuth fix that uses the current plugin-sdk/core proxy seam, preserves target-aware NO_PROXY and ALL_PROXY semantics, and only changes central dispatcher cleanup if verified against undici behavior. Discord, LINE, Nostr, and plugin-sdk test cleanups should be dropped or split into separate reviewed PRs.

## Reproduction Assessment

Yes. #46184 provides a concrete macOS HTTP(S) proxy reproduction for google-gemini-cli OAuth, and current main inspection shows the OAuth helper still does not opt into trusted env-proxy mode.

## Solution Assessment

No. The intended Gemini/proxy fix direction is plausible, but this PR is not the best solution because it mixes unrelated channel rewrites with network security-sensitive changes and has review-flagged regressions; a narrow current-main patch is safer.

Full review comments:

- **[P1] Honor NO_PROXY before applying Discord env proxy fallback:** `extensions/discord/src/monitor/provider.ts:438`
  - body: The env fallback turns HTTPS_PROXY/HTTP_PROXY into an explicit Discord proxy without checking the target host against NO_PROXY. In environments that set a global proxy but exempt discord.com, the REST/gateway paths will now be forced through the proxy and can fail or violate the operator's routing policy; use a target-aware helper instead of a raw env URL.
  - confidence: 0.9
- **[P2] Use a target-aware proxy gate for Gemini OAuth:** `extensions/google/oauth.http.ts:11`
  - body: hasProxyEnvConfigured() also fires for proxy settings that the trusted env-proxy dispatcher path may not actually use for this target, and it does not check NO_PROXY. This can still send the OAuth exchange direct or ignore bypass rules in proxy-only setups; gate on the current target-aware HTTP proxy helper and add coverage for HTTP(S), ALL_PROXY, and NO_PROXY cases.
  - confidence: 0.86
- **[P1] Reinstate strict Nostr key validation:** `extensions/nostr/src/nostr-bus.ts:24-35`
  - body: normalizePubkey now returns the lowercased input for any malformed non-npub value. Callers rely on this helper to reject invalid sender/recipient IDs early; accepting arbitrary strings pushes bad keys into encryption, profile, and allowlist paths where they fail later and less predictably.
  - confidence: 0.92
- **[P2] Write LINE allowlist updates to the selected account:** `extensions/line/src/setup-surface.ts:164`
  - body: The new onboarding allowlist handler hardcodes DEFAULT_ACCOUNT_ID, so configuring a non-default LINE account updates the wrong allowlist. The existing setup surface routes allowFrom through the active accountId, which should be preserved for multi-account setups.
  - confidence: 0.84
