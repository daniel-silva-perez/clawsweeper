## Summary

Keep open. Current main still ties Codex app-server startup to the per-turn timeout and has no `appServer.startupTimeoutMs` config/docs/manifest surface, while this PR is a narrow additive fix with focused regression coverage. The related runtime-deps work addresses missing mirrored packages, but not the 30-minute startup-timeout amplification this PR targets.

## What This Changes

The PR adds optional `plugins.entries.codex.config.appServer.startupTimeoutMs` support, uses it for Codex app-server startup instead of per-turn `timeoutMs`, updates docs/manifest/changelog, and adds resolver plus startup-timeout regression tests.

## Best Possible Solution

Land this PR or an equivalent narrow change after rebasing onto current main and running the targeted Codex extension validation, preserving separate semantics for startup timeout, per-turn timeout, and app-server request timeout.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: on current main, a never-resolving app-server client factory under `params.timeoutMs = 30 * 60_000` would use that full turn budget for startup because `withCodexStartupTimeout` receives `params.timeoutMs`.

## Solution Assessment

Yes. The proposed direction is the narrow maintainable fix: add a documented resolver-backed `startupTimeoutMs` and use it only at the startup wrapper. The safer finishing step is rebasing the docs/changelog hunks and validating the current head.
