## Summary

Keep open. Current main still binds Codex app-server startup to the per-turn timeout and has no documented or manifest-backed startup timeout knob; the PR is a narrow additive fix for that remaining behavior, and the related runtime-deps fixes do not replace this startup ceiling.

## What This Changes

The PR adds optional `appServer.startupTimeoutMs` config/runtime support for the Codex app-server, uses it for startup timeout handling instead of the per-turn budget, and updates docs, manifest, changelog, and focused tests.

## Best Possible Solution

Land this PR or an equivalent narrow change so Codex app-server startup has its own documented, configurable timeout while leaving post-startup turn `timeoutMs` and control-plane `requestTimeoutMs` semantics unchanged.

## Reproduction Assessment

Yes. The current-main source path is deterministic: make the Codex app-server client factory never resolve with a long `params.timeoutMs`, and startup waits on `withCodexStartupTimeout` using that long per-turn value; the PR adds a regression test for the 30-minute-budget case.

## Solution Assessment

Yes. An additive `appServer.startupTimeoutMs` resolved through the existing Codex plugin config path is the narrowest maintainable fix, because it separates startup failure recovery without changing turn execution or request deadlines.
