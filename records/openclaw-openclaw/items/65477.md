## Summary

Keep open. Current main still formats the channel `/usage` footer from token/cache/cost/session data only; `runReplyAgent` knows the actual provider/model, but that label is not passed into the footer formatter. The open related PRs remain implementation candidates rather than proof that the request has shipped.

## What This Changes

Request to include the actual active provider/model name in the per-response `/usage` footer for messaging-channel replies.

## Best Possible Solution

Add actual provider/model attribution to the existing per-response usage footer path, using the resolved `providerUsed` and `modelUsed` from `runReplyAgent` and an existing provider/model label helper such as `formatProviderModelRef`; preserve current token/cache/cost/session behavior and cover both formatter and runner paths with focused tests.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a feature request, but the source-level check is high confidence: enabling `responseUsage` reaches a formatter that has no provider/model input and therefore cannot emit the requested label.

## Solution Assessment

Is this the best way to solve the issue? Yes, the requested footer attribution is a narrow maintainable change because the runner already knows the actual provider/model that answered; `/status` and `messages.responsePrefix` are useful adjacent surfaces but do not replace the usage-footer behavior.
