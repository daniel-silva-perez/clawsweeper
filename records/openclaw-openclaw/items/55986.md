## Summary

Keep open. Current main still does not provide OpenRouter audio transcription: the OpenRouter media-understanding provider is image-only, the audio docs and default audio provider order omit OpenRouter, and the audio runner requires a provider-level transcribeAudio hook. The PR is directionally valid because OpenRouter documents chat/completions audio input with input_audio, but the branch is stale against current main and has blocking integration/security issues that should be fixed in a rebase or replacement PR.

## What This Changes

This draft PR adds an OpenRouter media-understanding transcription provider that posts base64 audio as input_audio to chat/completions, registers it, adds registration/request-shape tests, and documents tools.media.audio configuration.

## Best Possible Solution

Implement the feature on top of current main by extending the existing OpenRouter media-understanding provider to support both image and audio, use the current provider HTTP policy helpers for the new chat/completions audio request, update plugin manifest metadata and user-facing docs, and keep auto-detection behavior unchanged unless maintainers explicitly choose an OpenRouter audio autoPriority.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Import HTTP helpers from the provider HTTP SDK:** `extensions/openrouter/media-understanding-provider.ts:2-9`
  - body: On current main, openclaw/plugin-sdk/media-understanding exports media types and media helpers, but it does not export assertOkOrThrowHttpError, normalizeBaseUrl, or postJsonRequest. A rebased version of this patch will fail typechecking until those imports come from openclaw/plugin-sdk/provider-http, leaving only the media request/result types on the media-understanding subpath.
  - confidence: 0.9
- **[P2] Preserve the existing OpenRouter image media provider:** `extensions/openrouter/media-understanding-provider.ts:157-160`
  - body: Current main already uses this same provider file for OpenRouter image understanding with image capabilities, an image default model, and describeImage/describeImages hooks. This patch defines the provider as audio-only, so resolving the stale add/add conflict by taking this file would remove OpenRouter image media understanding. Extend the existing provider instead of replacing it.
  - confidence: 0.88
- **[P1] Keep provider HTTP policy on custom base URLs:** `extensions/openrouter/media-understanding-provider.ts:108-143`
  - body: The new request path sets allowPrivateNetwork from any non-empty baseUrl and never passes params.request through resolveProviderHttpRequestConfig, so custom OpenRouter endpoints bypass the proxy/TLS/private-network policy path used by the other media providers. Resolve the base URL, headers, request overrides, and dispatcher policy through the provider HTTP helper, then pass the returned policy into postJsonRequest.
  - confidence: 0.86
