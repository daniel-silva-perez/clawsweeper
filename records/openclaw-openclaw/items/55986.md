## Summary

Keep open. The feature gap is real on current main, but this draft PR is stale and currently incorrect: current main already has an OpenRouter image media-understanding provider, the patch imports HTTP helpers from the wrong SDK subpath, and the new audio egress path needs the current provider HTTP policy and auto-detection behavior handled before merge.

## What This Changes

The PR adds an OpenRouter media-understanding audio provider that posts base64 `input_audio` to OpenRouter chat completions, registers it, adds request-shape and registration tests, and documents `tools.media.audio` configuration.

## Best Possible Solution

Extend the current OpenRouter media-understanding provider so it preserves image support and adds audio deliberately, routes the new chat-completions audio call through the provider HTTP policy helpers, updates manifest metadata/docs/tests together, and either guards active-model auto transcription or documents and tests that behavior as an intentional product choice.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Import HTTP helpers from provider HTTP SDK:** `extensions/openrouter/media-understanding-provider.ts:2-9`
  - body: On current main, `openclaw/plugin-sdk/media-understanding` does not export `assertOkOrThrowHttpError`, `normalizeBaseUrl`, or `postJsonRequest`; those live under `openclaw/plugin-sdk/provider-http`. A rebased patch will fail typechecking until the HTTP helpers come from the provider HTTP SDK subpath.
  - confidence: 0.92
- **[P1] Keep OpenRouter audio explicit or document auto egress:** `extensions/openrouter/media-understanding-provider.ts:168-169`
  - body: Adding `transcribeAudio` makes the current audio auto-resolution path eligible to send voice notes through the active OpenRouter model before local CLI fallback when an OpenRouter key is available. That contradicts the PR's explicit-only security scope, so guard this path or update behavior, docs, and tests intentionally.
  - confidence: 0.86
- **[P2] Preserve existing OpenRouter image understanding:** `extensions/openrouter/media-understanding-provider.ts:168-173`
  - body: Current main already uses this provider for OpenRouter image understanding with image capabilities, an image default model, and image description hooks. A rebased implementation needs to extend that provider and manifest metadata to include both image and audio instead of replacing it with an audio-only provider.
  - confidence: 0.9
- **[P1] Keep provider HTTP policy on custom base URLs:** `extensions/openrouter/media-understanding-provider.ts:108-143`
  - body: The new request path enables private-network access for any custom `baseUrl` and never passes `params.request` through `resolveProviderHttpRequestConfig`, so proxy/TLS/private-network policy is bypassed for this audio egress path. Resolve the base URL, headers, request overrides, and dispatcher policy through the provider HTTP helper before calling `postJsonRequest`.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff adds an outbound audio path and currently bypasses current provider HTTP policy controls while also making OpenRouter eligible for non-explicit active-model audio egress.

Concerns:

- **[medium] Custom base URLs bypass provider HTTP policy:** `extensions/openrouter/media-understanding-provider.ts:108`
  - body: The PR sets `allowPrivateNetwork` from any non-empty `baseUrl` and does not forward request overrides through `resolveProviderHttpRequestConfig`, so configured proxy, TLS, and private-network controls can be skipped for OpenRouter audio transcription.
  - confidence: 0.88
- **[medium] Audio may egress through active OpenRouter model automatically:** `extensions/openrouter/media-understanding-provider.ts:168`
  - body: Because current audio auto-resolution uses providers that expose `transcribeAudio`, adding this hook can send audio to OpenRouter when it is the active model and an API key is available, even though the PR describes the behavior as explicit-only.
  - confidence: 0.84
