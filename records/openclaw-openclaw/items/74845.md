## Summary

Keep open. Current main still exposes node-host commands as a request/result relay, with plugin node-host handlers returning one final payload and `system.run` surfacing only terminal `exec.finished`/`exec.denied` events. The related progress/status items and the open ACP node-runtime PR are adjacent, but none is a merged general-purpose streaming/progress contract for `node.invoke` or plugin node-host commands.

## What This Changes

Add an opt-in long-running or streaming node-host command capability so callers can receive progress or intermediate updates before the final node command result.

## Best Possible Solution

Design an additive, versioned node command progress contract that preserves existing one-shot invokes while allowing opt-in progress events for node-host and plugin-owned commands, with clear authorization, lifecycle, timeout, cancellation, and documentation coverage across gateway protocol, node host, SDK runtime, CLI, and tests.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is static contract inspection: `node.invoke`, plugin node-host command handlers, and docs all show one-shot request/result behavior rather than a progress-capable command path.

## Solution Assessment

Unclear pending maintainer design. A shared protocol/SDK primitive is the right direction for the stated platform problem, but simply increasing invoke timeouts or using integration-specific events would not provide the reusable progress contract requested here.
