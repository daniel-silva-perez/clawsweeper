## Summary

Keep open. Current main still exposes node-host commands as a one-request/one-final-result contract, while existing progress-like node events are separate system events and not a caller-visible stream for a pending invoke. The existing ClawSweeper discussion already reached the same keep-open conclusion, and the adjacent progress/timeout/session items do not supersede this general node-host command API request.

## What This Changes

Add an opt-in long-running or streaming node-host command path that can emit progress or intermediate updates before a final result.

## Best Possible Solution

Design an additive, versioned node command progress contract that preserves existing one-shot invokes while allowing opt-in progress events for node-host and plugin-owned commands, with documented authorization, lifecycle, timeout, cancellation, cleanup, and caller delivery semantics across gateway protocol, node host, plugin SDK, CLI/runtime helpers, and tests.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is static contract inspection: node.invoke, plugin node-host handlers, node-host dispatch, tests, and docs all show final-result behavior rather than a progress-capable command path.

## Solution Assessment

Unclear pending maintainer design. A shared protocol/SDK primitive is the maintainable direction for the stated platform problem, but increasing timeouts or adding integration-specific events would not satisfy the requested reusable caller-visible progress contract.
