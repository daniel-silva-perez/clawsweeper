## Summary

Keep open. Current main still implements node-host invocation as a one-request/one-final-response contract, and the existing node-originated events are separate gateway events rather than progress messages delivered to the caller waiting on a pending invoke. Related work such as configurable invoke timeout in open PR #68857 and ACP/session progress tracking in #54895 is adjacent, but neither replaces the requested node-host command progress primitive.

## What This Changes

The issue requests an additive long-running node-host command path that can stream progress or intermediate updates before returning a final result.

## Best Possible Solution

Add a versioned, opt-in node-invoke progress contract that preserves the existing one-shot invoke behavior while letting node-host and plugin-owned commands emit caller-visible progress before the final result, with protocol schemas, SDK/docs, lifecycle/security rules, and focused tests updated together.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high-confidence static contract inspection: node.invoke, protocol schemas, plugin node-host handlers, dispatch, tests, and docs all show final-result behavior rather than a progress-capable command path.

## Solution Assessment

Unclear pending maintainer design. A shared additive protocol/SDK primitive is the maintainable direction for the stated platform need; increasing timeouts or reusing general node.event notifications would not satisfy caller-visible progress for a pending invoke.
