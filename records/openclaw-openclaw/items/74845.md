## Summary

Keep open. Current main still treats node-host invocation as a one-request, one-final-result flow; existing progress-like events are channel/tool previews or system-event notifications, not caller-visible updates for a pending node command. The request appears valid but needs maintainer design across protocol, SDK, lifecycle, and security boundaries.

## What This Changes

Request an opt-in long-running or streaming node-host command primitive so callers can receive progress or intermediate updates before the final result.

## Best Possible Solution

Add an additive, versioned node-command progress contract that preserves existing one-shot invokes while allowing opted-in progress events, with documented authorization, lifecycle, timeout, cancellation, cleanup, caller delivery, and contract tests across gateway protocol, node host, plugin SDK/runtime, and caller helpers.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high-confidence static contract inspection: node.invoke, protocol schemas, node-host dispatch, plugin runtime types, tests, and docs all show final-result behavior rather than a progress-capable command path.

## Solution Assessment

Unclear pending maintainer design. A shared additive protocol/SDK primitive is the maintainable direction for the stated platform need; increasing timeouts or reusing system-level node.event notifications would not satisfy caller-visible progress for a pending invoke.
