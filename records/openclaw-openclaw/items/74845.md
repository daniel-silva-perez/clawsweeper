## Summary

Keep open. Current main still exposes node-host invocation as a one-request/one-final-result contract, while existing node-originated progress-like events are system-event notifications rather than a caller-visible progress stream for a pending node command. The related session/ACP progress work is adjacent, not a replacement for this node-host command primitive.

## What This Changes

Request an opt-in node-host command path that can emit progress or intermediate updates before a final result while preserving existing one-shot behavior, security, and lifecycle expectations.

## Best Possible Solution

Design an additive, versioned node command progress contract that keeps existing one-shot invokes compatible while allowing opt-in progress events for node-host and plugin-owned commands, with documented authorization, lifecycle, timeout, cancellation, cleanup, caller delivery, and tests across the gateway protocol, node host, plugin SDK/runtime, and CLI/runtime helpers.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high-confidence static contract inspection: node.invoke, protocol schemas, plugin node-host handlers, dispatch, tests, and docs all show final-result behavior rather than a progress-capable command path.

## Solution Assessment

Unclear pending maintainer design. A shared additive protocol/SDK primitive is the maintainable direction for the stated platform need; increasing timeouts or reusing system-level node.event notifications would not satisfy caller-visible progress for a pending invoke.
