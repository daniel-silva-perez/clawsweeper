## Summary

Keep open. Current main still implements node-host command invocation as a one-request/one-final-result contract, while progress-like node events are separate terminal system events rather than a caller-visible progress stream for a pending invoke. The related ACP runtime, task-status, sessions progress, and timeout items are adjacent, but none is a merged general-purpose streaming/progress contract for plugin node-host commands.

## What This Changes

Request an opt-in long-running node-host command path that can send progress or intermediate updates to the caller before returning a final result.

## Best Possible Solution

Design an additive, versioned node command progress contract that preserves existing one-shot invokes while allowing opt-in progress events for node-host and plugin-owned commands, with documented authorization, lifecycle, timeout, cancellation, cleanup, and caller delivery semantics across gateway protocol, node host, plugin SDK, CLI/runtime helpers, and tests.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high-confidence static contract inspection: node.invoke, plugin node-host handlers, node-host dispatch, tests, and docs all show final-result behavior rather than a progress-capable command path.

## Solution Assessment

Unclear pending maintainer design. A shared protocol/SDK primitive is the maintainable direction for the stated platform problem, but increasing timeouts or adding integration-specific events would not satisfy the reusable caller-visible progress contract requested here.
