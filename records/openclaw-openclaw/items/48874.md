## Summary

Current main has useful adjacent pieces, but it does not implement the RFC’s central multi-session architecture. Direct messages still collapse to the main session by default, the session entry/API do not have active/archived/pinned/tags/channel-binding semantics, Control UI “new session” still maps to `/new` reset rather than the proposed create-session workflow, and archive behavior is delete/reset/compaction artifact handling rather than a restorable active/archive session lifecycle. The discussion is useful prior art, including the linked `global-architecture` fork components, but this is still broad core routing/storage/UI/memory work rather than a closeable cleanup item.

## Best Possible Solution

Keep this issue open as an umbrella/design thread unless maintainers want to split it. The concrete path is to turn it into smaller core issues/PRs: canonical session identity/routing policy, first-class session lifecycle state and archive/restore APIs, Control UI create/archive/restore/pin workflows, and a clearly scoped memory/RAG integration contract. The linked fork work can be treated as prior art during design review, but any merge should land incrementally with tests across routing, session store migration, Gateway methods, and UI behavior.
