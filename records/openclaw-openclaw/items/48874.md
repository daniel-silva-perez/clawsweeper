## Summary

Keep open. Current main has useful adjacent work in DM scoping, multi-agent routing, dashboard session creation, session cleanup, and memory search, but it still does not implement the RFC's central architecture or the requested first-class archived/pinned/tagged session lifecycle and public knowledge-base contract.

## What This Changes

The issue requests a multi-session architecture with shared model inference, isolated channel/task sessions, restorable session lifecycle management, WebUI session controls, and a shared public knowledge/RAG layer.

## Best Possible Solution

Keep this as an umbrella/design thread or split it into maintainer-owned slices: canonical session identity and dmScope/mainKey policy; first-class active/archive/pin/tag lifecycle plus archive/restore APIs; Control UI create/archive/restore/pin workflows; and a scoped shared memory/RAG contract. Treat Artyomkun's linked fork and the existing narrower WebChat/session work as prior art rather than a completed core implementation.

## Reproduction Assessment

Not applicable for the full RFC because it is an architecture proposal, not a single bug. There is a high-confidence source-backed verification path for the main gaps: current docs, routing tests, protocol schemas, and UI handlers still show default main-session collapse and missing archive/pin/public-KB lifecycle surfaces.

## Solution Assessment

No, a single broad re-architecture branch is not the best way to solve this safely. The maintainable path is additive protocol and storage slices with migration tests, then channel/UI/memory follow-through once maintainers settle the contract.
