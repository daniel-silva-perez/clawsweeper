## Summary

Keep open: current main still has the misleading Terminal Hatch idle state during post-connect initialization, and PR #75120 is open, unmerged, and explicitly targets the same issue. The issue should remain open until that PR lands or is replaced by an equivalent narrow TUI fix.

## What This Changes

The issue reports that `openclaw onboard` Terminal Hatch appears stuck or ultra slow because the local embedded TUI shows `local ready | idle` instead of a loader before the first hatch message starts.

## Best Possible Solution

Land PR #75120 if review confirms it preserves existing streaming/sending states; otherwise implement the same narrow TUI change that marks post-connect refresh/history work as busy and covers it with focused TUI regression tests. Any measured performance problem in `refreshAgents()` or `loadHistory()` should be tracked separately.

## Reproduction Assessment

Yes for the UI-state bug: `openclaw onboard` -> Terminal Hatch launches local embedded TUI, and current `client.onConnected` shows `local ready` while `refreshAgents()` and `loadHistory()` run before `sendMessage()`. The exact reported one-minute duration remains unclear without machine-specific logs or profiling.

## Solution Assessment

Yes, the linked direction is the narrowest maintainable fix: reuse the existing TUI busy-loader path for post-connect initialization without changing provider/runtime behavior. The issue should not close until that fix lands or an equivalent replacement is merged.
