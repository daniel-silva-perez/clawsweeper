## Summary

Keep open: current main still only supports standard file chooser arming and direct file-input setting for browser uploads. The ByteUploader/custom uploader fallback is not implemented, and the adjacent open file-chooser issue tracks a different stale-ref/timeout problem rather than this custom drop/paste/native-drag capability.

## What This Changes

Add a browser upload fallback for custom or non-standard upload components such as ByteUploader when standard file input APIs report success but leave the page with no selected files.

## Best Possible Solution

Add a generic browser-plugin upload fallback only after agreeing on the contract: require an explicit target/ref or selector, preserve the existing uploads-root validation, prove the behavior with a local custom drop-zone fixture, and document when the fallback is used. Keep standard input/filechooser behavior as the default path.

## Reproduction Assessment

No. The issue gives a plausible ByteUploader symptom and source inspection confirms current upload paths cannot simulate drop, paste, or native drag, but there is no minimal fixture, public URL, or live ByteDance reproduction in the thread.

## Solution Assessment

Unclear. A browser-plugin fallback may be the right place to solve this, but the safest shape is an explicit, tested fallback mode rather than silent clipboard or native-drag behavior applied globally.
