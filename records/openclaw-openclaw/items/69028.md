## Summary

Keep open. Current main still has the Feishu silent-loss path: media save/download failures are caught and logged inside the Feishu media resolver, then normal agent dispatch can continue with a media placeholder and no chat-visible warning. The newer empty-message guard only skips blank content with no media, so it does not cover file/image/video messages that parse to nonempty placeholders.

## What This Changes

The issue asks the bundled Feishu plugin to notify the chat and avoid normal agent processing when inbound media exceeds `mediaMaxMb` and is dropped.

## Best Possible Solution

Fix this inside the bundled Feishu plugin by making media resolution return structured success/failure state, sending a Feishu-visible warning that includes the configured `mediaMaxMb` limit, and preventing an ordinary agent run when the triggering input was only dropped media. The regression coverage should exercise oversized single-file media and embedded post image/media failures.

## Reproduction Assessment

Yes. The issue has concrete steps, and the current code path is deterministic: configure Feishu with `mediaMaxMb`, send media above that limit, `saveMediaBuffer` throws, `resolveFeishuMediaList` logs and returns no failure marker, and `handleFeishuMessage` can continue into dispatch.

## Solution Assessment

Yes. A plugin-local structured media failure plus chat notification is the narrowest maintainable solution; moving this into generic core would miss Feishu-specific reply targeting and embedded post handling.
