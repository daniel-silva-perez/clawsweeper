## Summary

Keep open. Current main already has the ID-prefix lookup portion, and the narrowed PR still adds useful reply-to approval UX, but the latest patch extracts the first `ID:` token from replied text and can misroute approvals when plugin messages contain newline-bearing freeform fields before the canonical ID line.

## What This Changes

The PR branch adds reply-only `/approve <decision>` parsing that extracts an approval ID from `ReplyToBody`, exports parser/extractor helpers for tests, and updates exec/plugin approval prompt text to advertise reply and prefix shortcuts.

## Best Possible Solution

Merge the reply-to approval UX after the extractor reads the canonical rendered approval ID as a full trimmed line, ignores earlier freeform `ID:` lines, and adds regression coverage for embedded `ID:` lines and custom IDs that contain whitespace. Keep relying on main’s existing `lookupPendingId` prefix behavior.

## Reproduction Assessment

Yes. A source-level reproduction is a plugin approval whose title or description contains a newline followed by `ID: plugin:other` before the canonical rendered request ID, then replying with `/approve allow-once`; the PR regex captures the earlier line. A second focused check is an exec approval with a custom ID containing whitespace, which the PR extractor truncates at the first token.

## Solution Assessment

No. Prefix matching is already solved on current main, and the reply-to part should parse the known approval-message structure and preserve the canonical full ID line rather than taking the first `ID:` token from arbitrary replied text.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Capture the canonical full ID line:** `src/auto-reply/reply/commands-approve.ts:40`
  - body: `APPROVAL_ID_RE` matches the first line named `ID:` and captures only `\S+`. Plugin approval messages render freeform `Title:` and `Description:` before the canonical request ID, and those schema fields allow embedded newlines, so a reply-only approval can route to a spoofed or truncated ID. Parse the canonical rendered request ID as a full trimmed line and cover embedded `ID:` lines.
  - confidence: 0.9
