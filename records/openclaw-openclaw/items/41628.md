## Summary

Keep open. Current main already implements the ID-prefix approval lookup portion safely, but it still does not implement reply-to-message `/approve <decision>` or advertise that shortcut. The current PR head is scoped to that remaining behavior and deserves maintainer review, with one parser edge case to fix before merge.

## Best Possible Solution

Keep this PR open as the focused implementation candidate for reply-to-message approvals. The best path is to keep current main's shipped `lookupPendingId` behavior, add reply-body ID extraction inside `handleApproveCommand`, update exec and plugin approval hint text, preserve existing authorization checks, change the extractor to parse the full trimmed `ID:` line, and retain focused tests for exec/plugin reply-only approvals plus whitespace/custom-ID edge cases.

## Risks / Open Questions

- Approval resolution is authorization-sensitive. The PR appears to preserve the existing sender authorization, plugin-vs-exec routing, and gateway-scope checks after extracting the ID, but that should remain the main review focus.
- The PR extractor currently captures only the first non-whitespace token after `ID:`. It should capture and trim the full line so valid generic approval IDs are not truncated or misrouted.
