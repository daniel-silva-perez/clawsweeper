## Summary

Keep open. Current main already covers unique approval ID prefix lookup and the resolved-entry guard, but main still rejects reply-only `/approve <decision>`; the latest PR is useful but has a concrete approval-routing bug in the new reply-body extractor that should block merge.

## What This Changes

The PR adds reply-only `/approve <decision>` parsing from `ReplyToBody`, exports parser/extractor tests, and updates exec/plugin approval prompt text to mention reply and ID-prefix shortcuts.

## Best Possible Solution

Merge the reply-to approval UX after the extractor reads the canonical rendered approval ID as a full trimmed line, avoids earlier freeform `ID:` lines, and adds regression coverage for plugin title/description text containing embedded `ID:` lines plus whitespace-bearing custom IDs. Keep relying on main's existing prefix lookup instead of adding a separate resolver.

## Reproduction Assessment

Yes. A source-level reproduction is a plugin approval whose title or description contains a newline such as `ID: plugin:other` before the canonical rendered request ID, then replying with `/approve allow-once`; the PR regex captures the injected line first. A second focused check is an exec approval with a custom ID containing whitespace, which the PR extractor truncates at the first token.

## Solution Assessment

No. The product direction is good, but this implementation is not the safest fix for an approval authorization path; it should parse the known approval-message structure and capture the full canonical `ID:` line before resolving.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Capture the canonical full ID line:** `src/auto-reply/reply/commands-approve.ts:40`
  - body: `APPROVAL_ID_RE` captures only `\S+` from the first line named `ID:`. Plugin approval messages render freeform title/description before the canonical request ID, and those schema fields permit newlines, so a reply-only approval can route to a spoofed or truncated ID. Parse the canonical rendered request ID as a full trimmed line and cover embedded `ID:` lines.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff has no supply-chain changes, but it changes approval authorization handling and can misroute reply-extracted approval IDs.

Concerns:

- **[medium] Reply extraction can misroute approvals:** `src/auto-reply/reply/commands-approve.ts:40`
  - body: The new extractor uses the first non-space token from the first `ID:` line in replied text. Since plugin approval text can contain earlier freeform `ID:` lines before the canonical ID, a reply-only approval can be applied to the wrong pending request.
