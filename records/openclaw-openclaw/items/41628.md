## Summary

Keep open. Current main already has prefix-based approval lookup with the resolved-exact-ID guard, and the author has rebased this PR down to the still-missing reply-to approval UX. The remaining branch is useful, but it still has a concrete approval-routing bug: reply extraction captures the first non-space token from the first `ID:` line, which can truncate generic approval IDs or pick a freeform plugin title/description line before the canonical approval ID.

## What This Changes

The PR branch adds single-token reply-to `/approve <decision>` parsing with `ReplyToBody` ID extraction, updates exec and plugin approval prompt hints, and adds parser/forwarder tests.

## Best Possible Solution

Merge the reply-to approval UX after the extractor reads the canonical rendered approval ID line as a full trimmed value and adds regression coverage for whitespace-containing approval IDs plus plugin approval messages whose title or description contains an earlier `ID:` line. Keep relying on the existing main prefix lookup rather than adding another resolver.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Capture the canonical full ID line:** `src/auto-reply/reply/commands-approve.ts:40`
  - body: `APPROVAL_ID_RE` captures only `\S+` from the first line named `ID:`. Approval IDs are generic non-empty strings, and plugin approval text renders freeform title/description before the canonical request ID, so reply-only approvals can submit a truncated or spoofed ID and resolve the wrong pending request. Capture the canonical full ID line and trim it before resolving.
  - confidence: 0.9
