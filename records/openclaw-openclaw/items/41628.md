## Summary

Keep open. Current main already has the safe ID-prefix approval lookup, but it still lacks reply-to-message approval handling; this PR is scoped to that remaining behavior, though it needs one correctness/security fix before merge.

## What This Changes

The PR adds reply-to approval ID extraction for single-decision `/approve`, updates exec and plugin approval hint text, and adds focused parser/handler/message tests.

## Best Possible Solution

Merge the reply-to-message approval UX after changing the extractor to capture and trim the full `ID:` line, preserving current gateway prefix lookup and existing authorization checks, with regression coverage for custom/whitespace IDs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Capture the full replied ID line:** `src/auto-reply/reply/commands-approve.ts:40`
  - body: `APPROVAL_ID_RE` captures only `\S+`, so an approval message with `ID: task 1` is extracted as `task`. Gateway approval IDs are schema-level non-empty strings, and the resolver supports prefix lookup, so this can fail or submit the decision for a different pending approval. Capture the rest of the `ID:` line and trim it instead.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff does not introduce supply-chain changes, but the new reply extractor can misresolve authorization-sensitive approval IDs.

Concerns:

- **[medium] Reply extraction can misresolve custom approval IDs:** `src/auto-reply/reply/commands-approve.ts:40`
  - body: Because the new regex truncates at whitespace, a custom exec approval ID can be shortened before it reaches the gateway prefix resolver, potentially approving the wrong pending command or failing a valid approval.
  - confidence: 0.78
