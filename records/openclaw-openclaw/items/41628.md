## Summary

Keep open. Current main already has the unique-prefix approval lookup portion, but it still does not implement the reply-to-message `/approve <decision>` flow that this PR now targets after the rebase. The rebased PR is focused and does not add dependencies, workflow changes, or package execution changes, but it still needs review because the proposed reply-body extractor captures only the first non-whitespace token after `ID:` while current approval ID schemas allow generic non-empty strings.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to land the reply-to-message approval behavior against the current `handleApproveCommand` routing, preserve the existing prefix lookup implementation on main, parse the full trimmed `ID:` line from `ReplyToBody`, keep exec/plugin authorization and fallback semantics unchanged, update both exec and plugin approval hint text, and keep focused tests for single-token reply approvals, plugin-prefixed IDs, no-reply errors, and whitespace/custom-ID edge cases.
