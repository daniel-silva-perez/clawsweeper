## Summary

Keep open. Current main already implements unique-prefix approval lookup, but not reply-only `/approve <decision>` parsing; the PR's remaining reply-to change is useful but still has a source-backed correctness/security bug because it extracts the first non-space token from the first `ID:` line in replied text.

## What This Changes

The PR adds reply-only `/approve <decision>` parsing via `ReplyToBody`, exports parser/extractor helpers with tests, and updates exec/plugin approval prompt hints to mention reply and prefix shortcuts.

## Best Possible Solution

Merge the reply-to approval UX after the extractor reads the canonical rendered approval ID as a full trimmed value and adds regression coverage for plugin approval title/description text that contains earlier `ID:` lines. Keep using main's existing prefix lookup instead of reintroducing a separate resolver.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a plugin approval whose title or description contains a newline like `ID: plugin:other` before the canonical rendered request ID, then replying with `/approve allow-once`; the PR regex matches the injected line first. A second reproduction is an exec approval with a valid custom ID containing spaces, which the PR truncates at the first token.

## Solution Assessment

No. The PR is on the right product path, but the current extractor is not the safest implementation for an approval authorization path; parse the known approval-message structure and full canonical `ID:` line before resolving.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Capture the canonical full ID line:** `src/auto-reply/reply/commands-approve.ts:40`
  - body: `APPROVAL_ID_RE` captures only `\S+` from the first line named `ID:`. Plugin approval messages render freeform title/description before the canonical request ID and the schema permits newlines there, so reply-only approvals can route to a spoofed or truncated ID. Parse the canonical rendered request ID as a full trimmed line and cover embedded `ID:` lines.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff has no supply-chain changes, but it changes approval authorization handling and can misroute reply-extracted approval IDs.

Concerns:

- **[medium] Reply extraction can misroute approvals:** `src/auto-reply/reply/commands-approve.ts:40`
  - body: The new extractor uses the first non-space token from the first `ID:` line in replied text. Since plugin approval text can contain earlier freeform `ID:` lines before the canonical ID, a reply-only approval can be applied to the wrong pending request.
  - confidence: 0.86
