## Summary

Keep open. Current main already has the ID-prefix approval lookup, but it does not implement reply-only `/approve <decision>` handling, so the PR is still useful. The proposed reply extractor still has a concrete approval-routing bug because it reads the first single-token `ID:` line from the replied message instead of the canonical rendered approval id line.

## What This Changes

The PR adds reply-to `/approve <decision>` parsing with approval-id extraction from `ReplyToBody`, updates exec and plugin approval prompt hints to mention reply and prefix shortcuts, and adds parser/forwarder tests.

## Best Possible Solution

Merge the reply-to approval UX after the extractor reads the canonical rendered approval id as a full trimmed line and adds regression coverage for plugin approval messages whose title or description contains an earlier `ID:` line. Keep relying on the current main prefix lookup rather than adding another resolver.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Capture the canonical full ID line:** `src/auto-reply/reply/commands-approve.ts:40`
  - body: `APPROVAL_ID_RE` captures only `\S+` from the first line named `ID:`. Plugin approval messages render freeform title/description before the canonical request id, and the schema allows newlines there, so reply-only approvals can submit a truncated or spoofed id and resolve the wrong pending request. Parse the canonical rendered approval id line as a full trimmed value before resolving.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff has no supply-chain changes, but it changes approval authorization handling and can misroute reply-extracted approval ids.

Concerns:

- **[medium] Reply extraction can misroute approvals:** `src/auto-reply/reply/commands-approve.ts:40`
  - body: The new extractor uses the first non-space token from the first `ID:` line in the replied message. Because plugin approval text can contain earlier freeform `ID:` lines before the canonical id, a reply-only approval can be routed to the wrong pending request.
