## Summary

Keep open. Current main already covers the prefix-matching half safely, but it still does not support reply-only `/approve <decision>` commands, so the PR is not obsolete. The remaining branch is useful but should not merge yet because its reply extractor can submit a truncated or spoofed approval ID into an authorization-sensitive resolver.

## What This Changes

The PR adds single-token reply-to-message `/approve <decision>` handling, tests for approval command parsing and reply extraction, and exec/plugin approval hint text for reply-to and unique-prefix shortcuts.

## Best Possible Solution

Merge the reply-to approval UX only after the extractor identifies the canonical request ID line, captures and trims the full value, and adds regression coverage for whitespace/custom IDs plus plugin messages containing other `ID:` text before the real request ID.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Extract the canonical full ID line:** `src/auto-reply/reply/commands-approve.ts:40`
  - body: `APPROVAL_ID_RE` takes the first non-space token from any `ID:` line. Approval IDs are protocol-level non-empty strings, and plugin approval messages put freeform title/description text before the real `ID:` line, so reply-only approvals can submit a truncated or spoofed prefix and resolve the wrong pending approval. Match the actual request ID line and trim the full value.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff has no supply-chain changes, but it touches the approval authorization path and can misresolve reply-extracted approval IDs.

  - body: The new extractor captures the first non-space token from the first `ID:` line before sending it to gateway prefix lookup. For generic approval IDs or plugin messages with earlier `ID:` content, that can approve or deny a different pending request than the one the user replied to.
