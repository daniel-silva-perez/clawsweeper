## Summary

Keep open. Current main already has safe approval ID prefix lookup, but it does not implement reply-only `/approve <decision>` handling; this PR is still useful, but the proposed reply-body extractor has a blocking correctness/security issue on the approval path.

## What This Changes

The branch adds reply-to `/approve <decision>` parsing with ID extraction from `ReplyToBody`, updates exec and plugin approval prompt hints, and adds parser/forwarder tests.

## Best Possible Solution

Merge the reply-to approval UX after the extractor reads the canonical full request ID, trims the full line value, and adds regression coverage for whitespace IDs plus plugin approval messages with misleading earlier `ID:` text. Keep the existing main prefix lookup rather than reintroducing a separate resolver.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Extract the canonical full ID line:** `src/auto-reply/reply/commands-approve.ts:40`
  - body: `APPROVAL_ID_RE` captures only `\S+` from the first line named `ID:`. Approval IDs are protocol-level non-empty strings, and plugin approval text can include freeform lines before the real request ID, so reply-only approvals can submit a truncated or spoofed prefix and resolve the wrong pending request. Capture the canonical full ID line and trim it before resolving.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff has no supply-chain changes, but it changes approval authorization handling and can misresolve reply-extracted approval IDs.

Concerns:

- **[medium] Reply extraction can misresolve approval IDs:** `src/auto-reply/reply/commands-approve.ts:40`
  - body: The new extractor captures the first non-space token from the first `ID:` line in the replied message. Because approval IDs are generic strings and plugin request text can contain earlier `ID:` lines, a reply-only approval can be routed to the wrong pending request.
  - confidence: 0.86
