## Summary

Keep open. Current main still has tool-name allow/deny plus a boolean `tools.fs.workspaceOnly` guard, while host exec approvals remain command-allowlist based; the requested path-keyed RWX policy across file tools and exec is not implemented. The linked implementation attempts are closed unmerged, and the newer overlapping issue was closed as duplicate back to this canonical thread.

## What This Changes

Add backward-compatible path-keyed RWX permissions for file tools and exec so operations are authorized by target path instead of only by command or tool name.

## Best Possible Solution

Keep this as the canonical maintainer thread and design a versioned, backward-compatible filesystem permission contract that covers read, write, edit, apply_patch, and exec target paths. The implementation should preserve existing exec approval behavior, define exact precedence and symlink/canonicalization semantics, document platform limits, and add focused unit/integration coverage before shipping.

## Reproduction Assessment

Yes for the current-state gap: source and test inspection show file tools are only gated by `workspaceOnly`, with outside-workspace writes allowed when it is false or unset, and host exec approvals evaluate command allowlists rather than target paths. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

Unclear as an implementation plan, but keeping the issue open is the right triage outcome. The requested path-RWX direction matches the observed gap, while the concrete schema and OS enforcement details still need maintainer/secops review before a replacement PR is safe.
