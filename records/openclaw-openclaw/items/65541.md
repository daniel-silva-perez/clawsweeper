## Summary

Keep this PR open. Current main still does not implement the proposed behavior: disabled block streaming buffers text-only and captioned media block replies for final delivery, tests assert that behavior, and docs describe `blockStreamingDefault: "off"` as final-reply-only. The PR is not obsolete, but it needs maintainer product/behavior judgment because it would change documented semantics and interacts with direct-send dedupe and same-channel queued delivery.

## Best Possible Solution

Keep this PR open for maintainer judgment. If OpenClaw should support progressive, naturally bounded text delivery while block streaming is disabled, update `reply-delivery.ts`, disabled-streaming tests, streaming docs, final-payload dedupe semantics, and the same-channel dispatcher await/fallback contract together. If `blockStreamingDefault: "off"` should remain final-only, close this PR with that product decision and route the same-channel timing problem through #32868.

## Risks / Open Questions

- Broadening `directlySentBlockKeys` to disabled-streaming text increases the lost-text blast radius if same-channel dispatcher delivery is queued successfully but later transport delivery fails.
- The PR's all-payload direct-send approach conflicts with current captioned media buffering tests that keep text+media together for final delivery when block streaming is off.
