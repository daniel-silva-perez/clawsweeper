## Summary

Keep open. Current main still uses the request-side OpenAI `model` value as the response `model` in both JSON and SSE chat-completions output, and there is still no documented or implemented response attribution header for the selected backend provider/model.

## What This Changes

Expose the actual underlying provider/model used by `/v1/chat/completions` responses, either in the response body or a response header, so clients can display model attribution without probing the agent.

## Best Possible Solution

Add backend model attribution to the Gateway OpenAI-compatible chat-completions surface while preserving agent-target request routing. The safest end state is an additive response attribution mechanism, documented and tested for both non-streaming and streaming responses, that reports fallback-selected provider/model accurately.

## Repair Work Prompt

Fix the Gateway OpenAI-compatible `/v1/chat/completions` response attribution gap from https://github.com/openclaw/openclaw/issues/55450. Current behavior returns the request-side agent target (`openclaw`, `openclaw/default`, or `openclaw/<agentId>`) as `model` in JSON and SSE chunks, while clients need the actual backend provider/model used after overrides and fallback. Keep request routing semantics intact: do not repurpose the request `model` field away from agent targeting unless maintainers explicitly choose that product change. Prefer an additive response attribution surface such as `X-OpenClaw-Model` and/or a documented stream-safe field/chunk strategy that can accurately report fallback-selected provider/model. Thread actual selected model data from the agent runtime boundary, likely via an `onModelSelected` callback or result metadata, into `src/gateway/openai-http.ts`. Cover non-streaming responses, streaming chunks or metadata, request `x-openclaw-model` overrides, default agent model selection, and fallback-selected model attribution. Update `docs/gateway/openai-http-api.md` and add a user-facing changelog entry with reporter credit (`Thanks @k9spy`). Do not change provider-specific routing, model allowlist enforcement, or the `/v1/models` agent-target contract.
