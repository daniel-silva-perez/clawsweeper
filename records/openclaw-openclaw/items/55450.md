## Summary

Keep open. Current main has the Gateway OpenAI-compatible request-side model override, and related UI/model-switching work exists, but the `/v1/chat/completions` response still uses the request agent-target `model` value and does not expose the actual backend provider/model as a response header.

## What This Changes

Expose the actual backend model used by `/v1/chat/completions` so third-party clients can display it without probing the agent.

## Best Possible Solution

Add an additive actual-model attribution path for Gateway chat completions, preferably `X-OpenClaw-Model: <provider>/<model>` on both non-streaming and streaming responses, with docs and tests that keep request `model` routing as an agent target.

## Repair Work Prompt

Fix the Gateway OpenAI-compatible `/v1/chat/completions` response so clients can read the actual backend provider/model used for a turn. Keep the documented request contract intact: the JSON request `model` remains an OpenClaw agent target (`openclaw`, `openclaw/default`, `openclaw/<agentId>`), and `x-openclaw-model` remains a request-side backend override. Implement an additive response attribution path, preferably `X-OpenClaw-Model: <provider>/<model>`, for both non-streaming and streaming responses.
