## Summary

Current main still returns the OpenAI-compatible chat-completions `model` field as the requested OpenClaw agent target, not the underlying provider/model. The backend override exists only as an inbound `x-openclaw-model` request header, and the docs explicitly describe the agent-first contract, so the requested response/header exposure remains real work.

## Best Possible Solution

Keep this open and implement an additive API contract: expose the actual resolved provider/model from agent run metadata as a response header such as `x-openclaw-model` and in streaming/non-streaming tests, while preserving the existing agent-target `model` field unless maintainers explicitly approve a versioned breaking change. Update `docs/gateway/openai-http-api.md` to document the response-side field/header and its fallback/failover semantics.

## Risks / Open Questions

- Changing the standard `model` field to a provider/model id would conflict with the documented agent-first contract and could break clients that use `openclaw/default` or `openclaw/<agentId>` routing.
- A correct implementation should define what to report when the requested override differs from the final model used after defaults, aliases, or failover.
