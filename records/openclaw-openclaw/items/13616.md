## Summary

Keep open. Current main has a first-party local backup archive creator and verifier, but it still does not provide the requested native restore/import/export/list/cloud/encryption/scheduled-backup lifecycle or component-aware cron/session restore semantics. The latest maintainer-side discussion also kept the native umbrella open despite useful community workarounds.

## What This Changes

The issue requests built-in backup, export, import, restore, listing, scheduling, cloud storage, encryption, and selective component handling for OpenClaw config, cron jobs, sessions, credentials, skills, and workspace state.

## Best Possible Solution

Keep this as the native umbrella for a reviewed first-party backup lifecycle built on the existing create/verify archive and migration seams, with safe extraction, dry-run validation, explicit component modeling for cron and sessions, secret/credential policy, overwrite and rollback rules, cloud/encryption choices, and docs that point to community tools as optional workarounds.

## Risks / Open Questions

- Native restore/import can overwrite config, credentials, cron jobs, sessions, and workspaces, so path traversal, secret handling, dry-run, conflict, rollback, and permissions rules need explicit maintainer design before implementation.
- Several overlapping open follow-ups and PRs cover different slices of the request; maintainers should decide whether the canonical path is backup restore, migrate export/import, profile portability, plugin guidance, or a staged combination.
- Community plugins cover many disaster-recovery and cloud-backup use cases today, but they do not settle the first-party session-history and component-aware restore contract requested here.
