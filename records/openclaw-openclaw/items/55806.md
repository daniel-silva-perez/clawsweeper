## Summary

Keep open. Current main has partial overlapping observability: diagnostic tool events include duration, low-cardinality parameter shape, and error category/code, and Control UI agent events redact tool args/results. It still does not implement the issue’s full requested surface: no structured `toolResult` fields such as `errorType`/`errorRecoverable`, no INFO/file-log sanitized argument summary, and no generic context/file/memory writeback audit trail beyond the narrow apply_patch summary event. The related PR #55846 was closed unmerged, so there is no merged implementation to close against.

## Best Possible Solution

Keep this issue open and implement it by extending the existing diagnostic event, redaction, agent-event, OTEL/Prometheus, and transcript/toolResult surfaces rather than adding a parallel logging path. The likely path is to define bounded sanitized argument summaries, decide whether structured error metadata belongs in persisted `toolResult` records or only diagnostics, and add a generic writeback/audit event contract that mutating tools can emit consistently.

## Risks / Open Questions

- The issue intentionally bundles three related but distinct observability surfaces; closing it for the existing diagnostics/OTEL subset would drop the remaining transcript and writeback audit requirements.
- A future implementation needs a privacy-preserving schema so sanitized summaries and writeback records do not reintroduce sensitive tool input, output, file content, or message content leakage.
