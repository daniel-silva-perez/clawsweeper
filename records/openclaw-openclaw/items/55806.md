## Summary

Keep open. Current main has useful partial observability through diagnostic tool execution events, redacted Control UI agent events, OTel documentation, and apply_patch summaries, but it still does not implement the full requested surface: INFO/file-log sanitized argument summaries, persisted toolResult errorType/errorRecoverable metadata, or generic context/file/memory writeback audit tracking. The related implementation PR #55846 is closed and unmerged, so there is no merged canonical fix to close this against.

## What This Changes

Request sanitized tool-call argument logging, structured tool-result error metadata, and audit tracking for tool/context writebacks.

## Best Possible Solution

Extend the existing diagnostic event, agent-event, OTel, transcript/toolResult, and mutation-summary surfaces under one privacy-preserving contract. Maintainers should decide where structured error metadata is durable, then add a generic writeback/audit event for mutating tools with tests proving secrets, message bodies, and file contents are not exposed.

## Reproduction Assessment

Not applicable as a feature/audit gap rather than a runtime crash. The high-confidence verification path is static and behavioral: inspect current tool execution code and run a tool call while checking diagnostics, file logs, agent events, and transcript/toolResult records for the missing fields.

## Solution Assessment

Unclear; the issue’s direction is valid, but a standalone INFO log line plus new toolResult fields is not necessarily the best maintainable shape. The safer path is to build on the existing redacted diagnostics/agent-event/plugin middleware surfaces and make any persisted transcript changes explicit and backwards-compatible.
