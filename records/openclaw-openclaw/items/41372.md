## Summary

Keep this issue open. Current main still has actionable cron follow-up from the field report: the scheduled-jobs guide still says plain `openclaw cron list` lists all jobs, while the CLI/service default is enabled-only; the `jobs`/`entries` JSON wrapper shapes remain underdocumented; and `cron add` is still create-only with no idempotent setup/upsert path. Related closed PRs attempted part of the docs cleanup but were not merged, and #68015 overlaps the operator-field-report theme without superseding these cron gaps.

## Best Possible Solution

Keep this issue open for maintainer triage. First fix the cron documentation so plain `openclaw cron list` is documented as enabled-only with `openclaw cron list --all` for disabled jobs, document the `jobs` and `entries` JSON wrapper shapes, and decide whether cron needs an idempotent setup/upsert command. Then split or link focused follow-ups for any still-current Discord receive, isolated web-search, onboarding token, update-process, and small-VPS resource findings.
