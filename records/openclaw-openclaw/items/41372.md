## Summary

Keep this issue open. Current main has partial cron cleanup, including a `cron create` alias and better current examples, but meaningful parts of the field report are still current: the scheduled-jobs guide still says plain `openclaw cron list` lists all jobs even though CLI/service default is enabled-only, the user-facing cron docs still do not document the `jobs`/`entries` JSON wrapper shapes, and `cron add` remains create-only rather than an idempotent setup/upsert path.

## Best Possible Solution

Keep this issue open for maintainer triage. Fix the cron docs so plain `openclaw cron list` is documented as enabled-only with `openclaw cron list --all` for disabled jobs, document the `jobs` and `entries` JSON wrapper shapes, and decide whether cron needs an idempotent setup/upsert command. Then split or link focused follow-ups for any still-current Discord receive, isolated web-search, onboarding token, update-process, and small-VPS resource findings.
