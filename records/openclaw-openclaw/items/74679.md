## Summary

Keep open. Current main still has the reported routing hazard: the OpenAI-compatible handler resolves the requested agent from `model`, but an explicit `x-openclaw-session-key` is returned unchanged, and downstream agent execution falls back to the default agent when that key is not already `agent:<id>:...` shaped. The public docs and existing tests currently treat that header as a full session-routing override, while related open PR #69060 proposes a separate scoped session header, so this needs a maintainer API-contract decision rather than cleanup closure.

## What This Changes

The issue requests that `/v1/chat/completions` keep `model: "openclaw/<agentId>"` agent routing when callers also send `x-openclaw-session-key` as a session identifier.

## Best Possible Solution

Decide the public HTTP session contract, then implement it consistently: either canonicalize non-agent `x-openclaw-session-key` values under the selected agent while preserving canonical `agent:<id>:...` overrides, or land/document a separate scoped session header such as #69060 and make the existing header's full-override semantics explicit. Add focused regression tests for raw header plus `model`, canonical header plus `model`, `x-openclaw-agent-id`, and `/v1/responses` parity; add a user-facing changelog entry if runtime behavior changes.
