## Summary

Keep open. Current main still preserves a raw `x-openclaw-session-key` before agent-scoped session generation, forwards only that key into execution, and downstream routing falls back to the default agent for non-`agent:<id>:` keys. The remaining decision is the public HTTP precedence contract, especially because docs currently say both that `model` selects the agent and that this header fully controls session routing.

## What This Changes

The issue asks OpenAI-compatible chat completions to respect `model: "openclaw/<agentId>"` routing when `x-openclaw-session-key` is also supplied.

## Best Possible Solution

Define the HTTP API precedence contract for raw session keys versus explicit agent selection, then update `/v1/chat/completions`, `/v1/responses`, docs, and regression tests so raw-header requests either route predictably to the selected agent or are clearly treated as full session-routing overrides without misleading model echoes.
