## Summary

Keep open. Current main still has the reported routing hazard: the OpenAI-compatible handler resolves the model-selected agent, but a raw `x-openclaw-session-key` is preserved as the execution session key and no explicit agent override is forwarded. The public docs also describe that header as full session-routing control, and the related open PR preserves that precedence, so this needs maintainer API-contract judgment rather than cleanup closure.

## What This Changes

The issue requests that `/v1/chat/completions` honor `model: "openclaw/<agentId>"` agent routing even when `x-openclaw-session-key` is supplied as the session key header.

## Best Possible Solution

Define and document the public precedence contract for full-routing session keys versus agent-scoped session hints, then make Chat Completions and OpenResponses consistent with regression tests for raw headers, canonical `agent:<id>:` headers, `x-openclaw-agent-id`, model-selected agents, and previous-response/session reuse.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to send `model: "openclaw/beta"` with raw `x-openclaw-session-key: my-session-key`; current main forwards that raw key without an explicit `agentId`, and downstream routing falls back to the default agent for non-`agent:<id>:` keys.

## Solution Assessment

Unclear. The requested behavior may be the best operator experience, but it conflicts with the documented full-override meaning of `x-openclaw-session-key`; the safer path starts with a maintainer decision on whether to change that header or steer clients to an agent-scoped hint header like #69060.
