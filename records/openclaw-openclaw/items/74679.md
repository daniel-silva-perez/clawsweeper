## Summary

Keep open. Current main still preserves the reported routing hazard, and there is an explicit public-contract conflict because the docs and related open PR treat `x-openclaw-session-key` as a full session-routing override.

## What This Changes

Requests that `/v1/chat/completions` respect `model: "openclaw/<agentId>"` agent routing when a raw `x-openclaw-session-key` header is also supplied.

## Best Possible Solution

Define the public precedence contract for full-routing session keys versus agent-scoped session hints. Then either pass the selected `agentId` explicitly for raw header keys or keep the full-override contract and document/use an agent-scoped header such as the related `x-openclaw-session` proposal, with regression tests for both OpenAI-compatible endpoints.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is `model: "openclaw/beta"` plus raw `x-openclaw-session-key: my-session-key`: current main forwards the raw key without an explicit `agentId`, and downstream routing falls back to the default agent for non-`agent:<id>:` keys.

## Solution Assessment

Unclear. The requested behavior is plausible, but it conflicts with the documented full-override meaning of `x-openclaw-session-key`; the safest fix starts with a maintainer contract decision and then a narrow implementation plus docs/tests.
