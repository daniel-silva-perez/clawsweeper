## Summary

Keep open. Current main still has the reported routing hazard: a raw `x-openclaw-session-key` is preserved as the execution session key, while the agent selected from `model: "openclaw/<agentId>"` is not passed as an explicit execution override. Because the public docs currently describe that header as full session routing control and open PR #69060 intentionally preserves that precedence, this needs maintainer API-contract judgment rather than cleanup closure.

## What This Changes

The issue asks for `/v1/chat/completions` to honor `model: "openclaw/<agentId>"` agent routing even when callers provide `x-openclaw-session-key` as the session identifier header.

## Best Possible Solution

Define the public precedence contract for raw full-routing session keys versus agent-scoped session hints, then implement the chosen behavior consistently for OpenAI Chat Completions and OpenResponses with regression tests for raw headers, canonical `agent:<id>:...` headers, `x-openclaw-agent-id`, model-selected agents, and previous-response/session reuse.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: send `model: "openclaw/cathy"` with raw `x-openclaw-session-key: my-session-key`; current main forwards that raw key without an explicit agent override, and downstream routing resolves raw keys to the default agent.

## Solution Assessment

Unclear. The requested behavior is plausible, but it conflicts with the documented full-override meaning of `x-openclaw-session-key`; the safer maintainable fix starts with a maintainer decision on whether to change that header or use a separate agent-scoped header like the one proposed in #69060.
