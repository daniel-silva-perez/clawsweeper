## Summary

Keep this PR open. Current main still lacks MCP serve scope selection, so the contribution is not obsolete, but the proposed least-privilege behavior is incomplete because explicit scopes are only passed into `GatewayClient` and can be superseded by cached device-token scopes.

## What This Changes

The PR adds a repeatable `openclaw mcp serve --scope` allowlist for `operator.read`, `operator.write`, and `operator.approvals`, threads it through the MCP server and bridge, and updates docs, changelog, and focused tests.

## Best Possible Solution

Update this PR so an explicit MCP `--scope` selection controls the effective Gateway connect scopes even when a cached device token is reused, preserve the current default read/write/approvals behavior when `--scope` is omitted, add a regression for the cached-token path, then land this as the narrow replacement for #74030 if maintainers agree with the product contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Honor explicit MCP scopes with cached tokens:** `src/mcp/channel-bridge.ts:126`
  - body: This only passes requested scopes into `GatewayClient`, but `GatewayClient` reuses stored device-token scopes before `opts.scopes`. With a cached broader operator token, `openclaw mcp serve --scope operator.read` can still connect with the stored broader scopes, which defeats the least-privilege bridge profile this flag is meant to create.
