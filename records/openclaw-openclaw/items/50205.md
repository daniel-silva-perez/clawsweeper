## Summary

Keep open. Current main still does not implement the requested `models.providers.google-vertex.requestLabels` / Gemini request-label configuration. Provider config validation is strict and has no `requestLabels` field, the Google generateContent request type/body builder has no `labels` field, and the Google plugin only maps `google-vertex` into the Google provider policy path without adding billing-label support. The discussion's Vertex-vs-non-Vertex distinction remains the right implementation boundary rather than a reason to close.

## Best Possible Solution

Keep the issue open and implement this in the Google provider/config surface: add a validated provider-level `requestLabels` field or equivalent Google-owned config, document the Vertex/non-Vertex support boundary, inject labels into the correct Gemini/Vertex request payload path, and add schema, transport, and docs coverage. If maintainers prefer the existing OpenAI-compatible `extra_body` workaround for Vertex only, document that explicitly instead of closing this as already implemented.
