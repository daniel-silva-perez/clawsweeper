## Summary

Keep this PR open for maintainer handling. The `<notts>` feature goal is still valid on current main, but the branch is not merge-ready: it targets the pre-plugin core TTS runtime, misses `<notts>` behavior in its own shell helper and tagged-mode trigger, and mixes in an unrelated gateway authorization expansion that needs security review.

## What This Changes

This PR adds `<notts>`/`<tts>` TTS preprocessing, tests, docs, and prompt hints, while also changing tool-result audio propagation, macOS speech recognition hints, media/MIME handling, a shell TTS filter, and gateway node-role method authorization.

## Best Possible Solution

Implement `<notts>` as a narrow current-main speech-core change with focused runtime, contract, config/docs, and tagged-mode coverage. Keep gateway authorization, macOS dictation, media conversion, MIME, and tool-result audio work in separate PRs with their own owners and validation.

## Reproduction Assessment

Yes. The PR diff and current-main source provide a high-confidence reproduction path: current speech-core has no `<notts>` behavior, the PR shell filter omits `<notts>`, and the PR tagged-mode check does not consistently gate on all enabled TTS tag semantics.

## Solution Assessment

No. The requested feature is reasonable, but this PR is not the best way to solve it because it is stale against the active plugin-owned TTS runtime and bundles unrelated runtime and security-sensitive gateway changes.
