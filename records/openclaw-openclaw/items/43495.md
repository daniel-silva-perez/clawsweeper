## Summary

Keep open. Current main still does not implement `<notts>` visual-only TTS support, and this PR is not merge-ready because it targets the pre-migration TTS runtime path, still has concrete `<notts>` filtering/gating defects, and mixes the feature with a gateway role authorization change that needs separate security review.

## What This Changes

This PR adds `<notts>`/`<tts>` TTS preprocessing, docs, tests, and a standalone shell filter, while also changing gateway node authorization, `audioAsVoice` propagation, macOS speech recognition, media transcription conversion, and MIME handling.

## Best Possible Solution

Split and rebase the work: add `<notts>` support in `extensions/speech-core/src/tts.ts` with focused speech-core tests and `docs/tools/tts.md` updates, drop side changes already present on main, and handle any node-role UI access change in a separate security-reviewed gateway design.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Rebase preprocessing onto speech-core:** `src/tts/tts.ts:42`
  - body: Current main routes TTS through the `speech-core` runtime via the plugin-sdk facade. This branch wires preprocessing into the old `src/tts` runtime surface, so after rebase the active auto-TTS path would not process `<notts>` at all.
  - confidence: 0.9
- **[P1] Keep node role from bypassing operator scopes:** `src/gateway/role-policy.ts:21-23`
  - body: Adding UI/read/write methods to the node-allowed set makes them callable by node-role clients, and gateway authorization returns before operator-scope checks for role `node`. That grants paired node clients scope-free access to methods such as `chat.send` and `config.get`.
  - confidence: 0.88
- **[P2] Handle notts in the shell filter:** `scripts/tts-filter.sh:22-44`
  - body: The added shell filter only strips or unwraps `<tts>` tags. A `<notts>...</notts>` block would show raw tags in text output and be spoken in voice output, contradicting the visual-only contract.
  - confidence: 0.92
- **[P2] Respect disabled tag processing in tagged mode:** `src/tts/tts.ts:869-874`
  - body: In `auto: "tagged"`, the new tag check can trigger TTS from literal `<tts>` markup even when tag processing is disabled. Gate tag-based auto-TTS on the resolved `processTtsTags` setting so disabling tag processing also disables the tag trigger.
  - confidence: 0.84

## Security Review
