## Summary

Keep this PR open, but do not merge it as-is. The requested `<notts>` feature is still not present on current `main`, yet the branch mixes that feature with unrelated gateway, macOS, media, and agent changes and introduces concrete correctness/security issues in the proposed diff.

## What This Changes

The PR adds `<notts>` visual-only TTS preprocessing, TTS prompt/docs/tests, a shell TTS filter, and several unrelated gateway role, macOS speech, media conversion, MIME, and tool-result audio delivery changes.

## Best Possible Solution

Implement the `<notts>` feature as a narrow current-main change in the active speech-core TTS runtime, with focused tests for visible/spoken splitting, tagged auto-mode, disabled tag processing, and docs in the existing TTS docs. Drop or split the unrelated gateway, macOS, media, and tool-result changes into separately reviewed PRs.

## Reproduction Assessment

Yes. The PR diff itself reproduces the two TTS defects: `scripts/tts-filter.sh` has no `<notts>` regex, and tagged auto-mode only checks `<tts>` while current `main` has no `<notts>` support at all.

## Solution Assessment

No. The feature direction is reasonable, but this PR is not the best way to solve it because it is stale against the active speech-core runtime, incomplete in companion paths, and includes an unrelated node-role authorization expansion.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Remove the node-role operator bypass:** `src/gateway/role-policy.ts:20-22`
  - body: This change lets node-role clients call UI read/write methods such as `config.get`, `chat.send`, `chat.history`, and `sessions.preview`. Current role policy keeps node methods separate from operator-scoped RPCs, so this broad authorization expansion is security-sensitive and unrelated to the TTS tag feature.
  - confidence: 0.9
- **[P2] Handle `<notts>` in the shell filter:** `scripts/tts-filter.sh:24-40`
  - body: The added shell filter mirrors the TTS preprocessing path but only handles `<tts>`. A message containing `<notts>visual</notts>` would print raw tags in the text output and speak the visual-only content in the voice output, contradicting the new tag semantics.
  - confidence: 0.94
- **[P2] Gate tagged mode on all enabled TTS tags:** `src/tts/tts.ts:869-870`
  - body: Tagged auto-TTS now fires only for `<tts>`, so a reply that uses only `<notts>` plus normal spoken summary text still skips audio. The same check also ignores `processTtsTags: false`, where literal tags should not force tagged-mode audio.
  - confidence: 0.88
