## Summary

Keep this PR open, but it is not merge-ready. Current main still lacks `<notts>` visual-only TTS support, while the branch targets the older `src/tts` implementation instead of the current `extensions/speech-core` runtime and also mixes in a gateway role-policy change that needs separate security review.

## What This Changes

This PR adds `<notts>`/`<tts>` TTS preprocessing, tests, docs, and system-prompt hints, plus unrelated gateway role, macOS speech recognition, media transcription, MIME, and `audioAsVoice` propagation changes.

## Best Possible Solution

Split the work: implement `<notts>` support in `extensions/speech-core/src/tts.ts` with focused runtime tests and docs in `docs/tools/tts.md`, drop side changes already present on main, and handle any node-role UI authorization expansion as a separate security-reviewed gateway design.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Rebase preprocessing onto speech-core:** `src/tts/tts.ts:42`
  - body: Current main routes TTS through `src/plugin-sdk/tts-runtime.ts` into `extensions/speech-core/src/tts.ts`; this branch wires the old `src/tts` implementation instead. After rebase, the new preprocessing either conflicts or misses the runtime path where TTS actually runs, so `<notts>` would not be applied.
  - confidence: 0.9
- **[P1] Keep node role from bypassing operator scopes:** `src/gateway/role-policy.ts:21-23`
  - body: Adding UI/read/write methods to the node-allowed set makes them callable by node-role clients, and current gateway authorization skips operator-scope checks once a node role is authorized. That would give paired node clients scope-free access to methods such as `chat.send` and `config.get`; keep those methods operator-scoped or add a narrower device permission model with tests.
  - confidence: 0.88
- **[P2] Handle notts in the shell filter:** `scripts/tts-filter.sh:22-44`
  - body: The added `scripts/tts-filter.sh` still only strips or unwraps `<tts>` tags. A `<notts>...</notts>` block would show raw tags in text output and be spoken in voice output, which contradicts the feature's visual-only contract.
  - confidence: 0.92
- **[P2] Respect disabled tag processing in tagged mode:** `src/tts/tts.ts:869-874`
  - body: In `auto: "tagged"`, the new tag check can trigger TTS from literal `<tts>` markup even when `messages.tts.processTtsTags` is false. Gate tag-based auto-TTS on the resolved `processTtsTags` setting so disabling tag processing also disables tag-triggered audio.
  - confidence: 0.84
