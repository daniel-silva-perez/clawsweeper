## Summary

Keep this PR open, but do not merge it as-is. The feature goal is still valid, while the branch is broad and stale against the active speech-core TTS runtime, leaves `<notts>` gaps in companion paths, and adds an unrelated gateway authorization expansion that needs security review.

## What This Changes

This PR adds visual-only `<notts>` and spoken-only `<tts>` TTS preprocessing with docs/tests/config, plus a shell filter, audioAsVoice propagation, a macOS dictation hint, whisper-cli conversion, MIME mapping, and a gateway node-role method allowlist.

## Best Possible Solution

Implement `<notts>` as a narrow current-main change in `extensions/speech-core`, with focused runtime, contract, docs, and config coverage for visible/spoken splitting, tagged auto-mode behavior, and disabled tag processing. Split gateway authorization, macOS speech recognition, media conversion, MIME, and tool-result audio work into separate PRs.

## Reproduction Assessment

Yes. The PR diff and current-main source give a high-confidence proof path: current speech-core has no `<notts>` behavior, the PR shell filter omits `<notts>`, and the PR tagged-mode check only looks for `<tts>` tags/directives.

## Solution Assessment

No. The requested feature is reasonable, but this branch is not the best solution because it targets the old core TTS path, mixes unrelated runtime and gateway changes, and introduces a security-sensitive authorization change.
