## Summary

Keep open. Current main still does not implement the requested cloud relay path: mobile connectivity remains direct Gateway WebSocket over LAN, tailnet, or manual host/port; remote QR requires an already reachable gateway URL or Tailscale Serve/Funnel; gateway config/schema lack the proposed cloud OAuth/API/heartbeat/relay block; and the root CLI registry has no cloud login/status/invite command. The prior ClawSweeper keep-open comment remains accurate on the current main SHA, and related #55914 is still open.

## Best Possible Solution

Keep this issue open for maintainer design review. The best path is to decide whether cloud mobile reachability is first-party Gateway/mobile behavior or should be enabled by a new explicit plugin/API seam, then implement the chosen design with config/schema/docs, OAuth credential handling, gateway heartbeat and relay lifecycle, mobile UX, and security tests that preserve device approval. Coordinate with #55914 so the shared cloud config and discovery story stay coherent.
