## Summary

Keep open. Current main still does not expose Gemini safety settings in config, schema, docs, or the runtime payload path, so the intended feature remains useful; however, the current PR branch is stale against main and still has blocking wrapper-lifecycle defects before it should merge.

## What This Changes

The PR adds `models.providers.*.safetySettings` config/schema/docs/generated baselines/tests and injects configured Gemini safety thresholds into `google-generative-ai` payloads, while also touching Z.AI `tool_stream` handling and scoped Vitest config setup.

## Best Possible Solution

Keep this PR, or a narrowed successor, as the implementation candidate for Gemini safety settings. The landing shape should rebase onto current main, keep the config contract strict and documented, inject settings only for the active `google-generative-ai` provider while preserving explicit payload values, avoid `google-gemini-cli`, repair or remove stale wrapper behavior, split unrelated Z.AI/Vitest changes unless maintainers want them here, and add regression coverage for provider/config switching.

## Reproduction Assessment

Yes. The current-main gap is reproducible by adding `models.providers.<provider>.safetySettings` to a Gemini provider config: the strict provider schema has no such field, and source search shows no runtime payload injector.

## Solution Assessment

No, not as currently shaped. Provider-scoped config with explicit-payload precedence is the right direction, but the PR should rebuild or scope wrappers from a stable base and avoid leaking stale Google safety or Z.AI `tool_stream` mutations across later provider runs.
