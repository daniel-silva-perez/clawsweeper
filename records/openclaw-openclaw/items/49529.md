## Summary

Keep open. Current main still lacks the requested Gemini safety-settings config, schema, docs, tests, and payload injection path, so this PR is not obsolete. The PR remains a useful implementation candidate for the linked feature request, but the current diff still needs maintainer review because it is stale against the current provider-wrapper layout and has wrapper-lifetime issues that can leak stale payload mutation across later runs.

## What This Changes

The PR adds `models.providers.*.safetySettings` config/schema/docs/generated baselines/tests and injects configured Gemini safety thresholds into `google-generative-ai` payloads, while also touching Z.AI `tool_stream` handling and a scoped Vitest config test.

## Best Possible Solution

Keep this PR, or a narrowed successor, as the implementation candidate for the open Gemini safety-settings feature. The landing shape should be rebased onto current main, keep the contract strict and documented, inject safety settings only for the active `google-generative-ai` provider while preserving explicit request payload values, avoid `google-gemini-cli`, add regression coverage for provider switching/config removal, and split unrelated Z.AI or Vitest changes unless maintainers explicitly want them here.

## Reproduction Assessment

Yes. The current-main gap is reproducible by adding `models.providers.<provider>.safetySettings` to a Gemini provider config: the strict provider schema has no such field, and source search shows no runtime payload injector; I did not run tests because this was a read-only review.

## Solution Assessment

No, not as currently shaped. Provider-scoped config plus explicit-payload precedence is the right direction, but the current patch should be rebased and should avoid stale nested stream wrappers, preferably by using the existing provider runtime wrapper seam or rebuilding from a stable base each run.
