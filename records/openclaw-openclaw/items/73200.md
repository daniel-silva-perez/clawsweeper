## Summary

Keep this PR open. Current main still has the behavior this branch changes, so the PR is not obsolete; however, the warning-only `lost` severity change conflicts with current task-audit docs and needs maintainer policy confirmation plus docs cleanup before it is ready to land.

## What This Changes

The PR changes task audit so all `lost` findings report `warn`, adds a 1-second tolerance for `startedAt < createdAt`, and updates the colocated audit tests for the new severity/order and timestamp-race cases.

## Best Possible Solution

Keep the PR in normal maintainer review. The desired end state is either a maintainer-approved warning-only lost policy with docs and tests updated consistently, or a narrower timestamp-tolerance-only fix while retaining the documented expired-lost error behavior.

## Reproduction Assessment

Yes. A direct `listTaskAuditFindings` input with `status: "lost"` and expired `cleanupAfter` still reproduces the old `error` severity on current main, and a direct task with `startedAt < createdAt` still follows the no-tolerance audit branch.

## Solution Assessment

No, not yet. The timestamp tolerance is narrow and plausible, but the all-warn `lost` change is not the best complete solution until the documented audit policy is either updated or explicitly preserved.
