## Summary

Keep this PR open. Current main still has the exact task-audit behavior the PR proposes to change: expired or unstamped `lost` task findings are errors, and direct audit inputs still flag any `startedAt < createdAt` ordering as `inconsistent_timestamps` with no tolerance. The PR is a small, plausible behavior change rather than obsolete work already implemented on main.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to decide whether terminal `lost` audit findings should always be warnings, update `docs/automation/tasks.md` if that policy changes, keep or refine the audit-level skew regression coverage if tolerance is desired beyond registry normalization, and remove the redundant assertion before merge if maintainers agree.
