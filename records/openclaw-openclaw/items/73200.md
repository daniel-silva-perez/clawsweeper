## Summary

Keep this PR open. Current main still has the task-audit behavior the PR proposes to change: expired or unstamped `lost` task findings are errors, and direct audit inputs still flag any `startedAt < createdAt` ordering as `inconsistent_timestamps` with no tolerance. The PR is a narrow, plausible behavior change that needs maintainer review rather than cleanup closure.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to decide whether terminal `lost` audit findings should always be warnings, update `docs/automation/tasks.md` if that policy changes, keep or refine audit-level timestamp-skew regression coverage if tolerance is desired beyond registry normalization, and remove the redundant assertion before merge.
