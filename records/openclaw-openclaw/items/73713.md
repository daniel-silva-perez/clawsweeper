## Summary

Keep open. Current main still drops nested fetch causes on the embedding CLI path: the shared CLI wrapper prints `String(err)`, and the remote embedding HTTP helper only enriches non-OK HTTP responses, so pre-response transport exceptions can still collapse to `TypeError: fetch failed`. The Voyage provider uses the shared guarded remote helper without a local custom fetch override, making this a narrow diagnostic/error-preservation bug rather than a close.

## What This Changes

The issue requests auditing `openclaw infer embedding create` for Node 24/Voyage fetch failures and preserving underlying fetch cause/code details.

## Best Possible Solution

Make the embedding CLI and shared remote embedding HTTP path cause-aware while preserving the guarded fetch and SSRF architecture. The end state should report provider context plus nested fetch cause messages and stable codes, with Undici/runtime pin changes considered only if a live Node 24 smoke still points there after the underlying cause is visible.

## Reproduction Assessment

Yes for the diagnostic gap: a provider rejection shaped like `TypeError('fetch failed')` with a nested cause currently reaches `runtime.error(String(err))`, which loses the cause/code. Unclear for the live Voyage transport failure because this read-only review had no Voyage credential and did not run the reporter's Node 24 setup.

## Solution Assessment

Yes, the narrow maintainable repair is to use cause-aware formatting in the default CLI wrapper and/or wrap pre-response remote embedding transport errors with provider context while preserving the original cause. Changing provider auth, default selection, Undici package pins, or SSRF policy would be premature without the surfaced cause proving that layer is still broken.
