## Summary

Keep open. Current main still leaves the reported diagnostic gap in place: the embedding CLI bubbles provider errors into a shared wrapper that prints `String(err)`, and the remote embedding JSON helper only prefixes non-OK HTTP responses, not fetch/network exceptions with nested Undici causes. The Voyage path does use the shared guarded remote embedding helper and does not pass a custom `fetchImpl`, so the transport-bypass suspicion was partly checked, but the cause-preserving failure output remains missing.

## What This Changes

The issue asks OpenClaw to audit the Node 24 Voyage embedding path for fetch/dispatcher mixing and to preserve underlying fetch or Undici cause details when `openclaw infer embedding create` fails.

## Best Possible Solution

Make the `infer embedding create` failure path cause-aware while keeping the existing guarded fetch and SSRF policy architecture. The narrow maintainable fix is to use the existing cause-preserving formatter in the CLI wrapper and/or wrap remote embedding fetch exceptions with the provider `errorPrefix` while preserving the original `cause` and Undici `code`, then add regression tests for nested fetch causes and run a live Node 24 Voyage smoke when credentials are available.

## Reproduction Assessment

Partially. The live Voyage failure cannot be reproduced here without a credential, but the diagnostic bug has a high-confidence unit reproduction: make the embedding provider or remote helper reject with `TypeError('fetch failed')` carrying an `UND_ERR_INVALID_ARG` cause and the current wrapper will print only `String(err)`.

## Solution Assessment

Yes. The best fix is a narrow error-preservation change in the CLI/remote embedding helper path; changing provider auth, model selection, SSRF policy, or dispatcher construction would be premature unless a live smoke still proves a transport bug after cause details are surfaced.
