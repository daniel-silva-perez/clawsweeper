## Summary

Keep open. Current main still leaves the reported diagnostic gap: `infer embedding create` lets provider fetch failures reach the generic CLI wrapper, which still prints `String(err)`, while the remote embedding JSON helper only adds provider context for HTTP non-OK responses. The Voyage path has been checked and already uses the shared guarded embedding helper without a custom `fetchImpl`, so the remaining actionable work is cause-preserving error reporting, not a broad Voyage auth or dispatcher rewrite.

## What This Changes

The issue asks OpenClaw to audit the Node 24 Voyage embedding path for fetch/dispatcher mixing and preserve underlying fetch or Undici cause details when `openclaw infer embedding create` fails.

## Best Possible Solution

Keep the existing guarded fetch and plugin-owned Voyage provider path, then make the embedding failure path cause-aware by using the existing formatter in the CLI wrapper and/or wrapping remote embedding fetch exceptions with provider context while preserving the original cause and code.

## Reproduction Assessment

Yes for the diagnostic gap: a focused unit test can make the embedding provider or remote helper reject with `TypeError('fetch failed')` carrying an `UND_ERR_INVALID_ARG` cause and observe that current output loses the nested cause. No live Voyage reproduction was available here because the credential was absent.

## Solution Assessment

Yes. A narrow error-preservation fix in the CLI wrapper and remote embedding helper is the maintainable solution; changing auth resolution, model defaults, SSRF policy, dispatcher construction, or dependencies would be premature without a new failing transport test.
