## Summary

Keep open. Current main still does not close the reported diagnostic gap: the embedding CLI awaits the provider directly, the embedding command uses the generic CLI wrapper without an error handler, and that wrapper still prints String(err), which drops nested fetch causes such as UND_ERR_INVALID_ARG. The Voyage provider path does use the shared guarded remote embedding helper and does not pass a custom fetchImpl, so the transport suspicion is partly audited, but network/fetch exceptions still are not wrapped with the provider prefix or cause/code detail.

## Best Possible Solution

Keep this issue open. The best fix is to make the infer embedding failure path cause-aware, either by using formatErrorMessage in the shared CLI wrapper or by adding an embedding-specific onError handler, and to wrap remote embedding fetch/network exceptions with the provider errorPrefix while preserving the original cause and Undici code. Add regression tests that inject TypeError('fetch failed') with an UND_ERR_INVALID_ARG-style cause, then run a live Node 24 Voyage CLI smoke with valid credentials.
