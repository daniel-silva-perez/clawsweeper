## Summary

Keep open. Current main still preserves the reported diagnostic failure: the embedding command sends provider exceptions through a shared CLI wrapper that prints `String(err)`, so nested fetch causes and Undici codes are still lost. The Voyage path uses the shared guarded remote embedding helper without a Voyage-local custom fetch override, so this is a narrow CLI/remote-HTTP error-preservation bug rather than a core scope close.

## What This Changes

The issue asks OpenClaw to audit the Voyage `infer embedding create` path on Node 24 and preserve nested fetch/Undici causes when embedding provider requests fail.

## Best Possible Solution

Make the embedding CLI and shared remote embedding HTTP helper cause-aware while keeping the guarded fetch and SSRF architecture intact. The end state should report provider context plus nested fetch cause messages and stable codes, with focused tests for `TypeError('fetch failed')` carrying an Undici-style cause.

## Reproduction Assessment

Unclear for the live Voyage transport failure because this read-only review did not run a credentialed Node 24 smoke. Yes for the diagnostic gap: a provider rejection shaped like `TypeError('fetch failed')` with a nested cause currently reaches `runtime.error(String(err))`, which loses the cause/code.

## Solution Assessment

Yes, the narrow maintainable fix is to preserve error cause/code text in `runCommandWithRuntime` and/or the remote embedding HTTP helper. Changing provider auth, default selection, Undici package pins, or SSRF policy would be premature unless the live smoke still fails after the underlying cause is visible.
