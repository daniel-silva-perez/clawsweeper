## Summary

Keep open. Current main still preserves the diagnostic failure reported here: `openclaw infer embedding create` lets provider embedding failures bubble into the shared CLI wrapper, and that wrapper still prints `String(err)`, which drops nested fetch causes and Undici error codes. The Voyage provider itself uses the shared guarded remote embedding helper without a Voyage-local custom fetch override, so the remaining work is a narrow CLI/memory-host error-preservation fix rather than a scope cleanup.

## What This Changes

The issue asks for the Voyage embedding CLI path to succeed under Node 24 and to preserve nested fetch causes such as Undici error codes when provider fetches fail.

## Best Possible Solution

Make the embedding CLI and shared remote embedding HTTP helper cause-aware while preserving the guarded fetch and SSRF architecture. The end state should include provider context plus nested fetch cause messages and stable error codes for transport failures, with focused regression coverage for `TypeError('fetch failed')` carrying an Undici-style cause.

## Reproduction Assessment

Yes for the diagnostic gap: a provider rejection shaped like `TypeError('fetch failed')` with a nested cause currently reaches `runtime.error(String(err))`, which loses the cause and code. Unclear for the live Voyage transport failure because this read-only review did not run a credentialed Node 24 smoke.

## Solution Assessment

Yes, the narrow maintainable fix is to preserve error cause/code text in `runCommandWithRuntime` and/or the remote embedding HTTP helper. Changing provider auth, default provider selection, Undici pins, or SSRF policy would be premature unless the surfaced underlying cause proves those are involved.
