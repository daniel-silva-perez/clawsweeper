## Summary

Keep open. Current main shipped the server-side ACP stdout protection for the linked bug, but this PR is broader: it still proposes a CLI startup guard in `runCli()` before command/subcommand/plugin registration, and current main does not have an equivalent ACP-specific stderr route there.

## Best Possible Solution

Keep this PR open for maintainer review, or supersede it with a focused current-main patch that routes ACP bridge logs to stderr immediately after console capture and before CLI command/subcommand/plugin registration, explicitly excludes the real `openclaw acp client` mode, and decides whether the standalone ACP server entrypoint also needs console capture. Then cover bridge options, client options, and option-value false positives in tests.
