## Summary

Keep open. Current main still has the reported session-lock cleanup gap: a same-PID lock without a valid starttime is treated as reclaimable when it is absent from the process-local held-lock map, and the contended acquisition path can remove that lock before retrying. The issue discussion and linked PR #71903 identify the same failure mode, but that PR remains an open implementation candidate rather than merged/shipped behavior.

## What This Changes

The issue reports lost Telegram delivery after main-agent tool execution when session-lock cleanup removes an active transcript lock as dead-pid.

## Best Possible Solution

Review, land, or supersede PR #71903 with a conservative fix in src/agents/session-write-lock.ts that protects active same-PID/no-starttime owners on platforms without starttime identity and adds regression coverage for both the lock state and the reported delivery path.
