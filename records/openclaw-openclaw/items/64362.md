## Summary

Keep open. Current main still contains the reported lock-cleanup gap: a same-PID lock with no valid starttime is treated as reclaimable when it is not in the process-local held-lock map, and contended acquisition removes it before retrying. On macOS, process starttime identity is unavailable, so the reporter's environment cannot use the Linux PID-recycle guard. Existing tests still assert reclaiming a current-PID lock without starttime, while active protection only covers locks present in HELD_LOCKS. The provided GitHub context also shows PR #71903 is still an open implementation candidate, not shipped main behavior.

## Best Possible Solution

Keep this issue open. Review, land, or supersede PR #71903 with a conservative change in src/agents/session-write-lock.ts that protects same-PID/no-starttime live owners on platforms without starttime identity, preserves safe stale-lock cleanup, and adds regression coverage for the lock state and ideally the reported delivery path.
