## Summary

Current main still contains the reported session-lock cleanup gap. A same-PID lock without verifiable starttime is treated as reclaimable when it is not present in the process-local held-lock map; on macOS, starttime identity is unavailable. The current tests still assert that behavior, and the related PR #71903 remains an open implementation candidate rather than shipped code.

## Best Possible Solution

Keep this issue open. Review, land, or supersede PR #71903 with a conservative fix in src/agents/session-write-lock.ts that protects same-PID/no-starttime live owners on platforms without starttime identity, preserves safe stale-lock cleanup, and adds regression coverage for the lock state and ideally the reported delivery path.
