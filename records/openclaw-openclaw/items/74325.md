## Summary

Keep open. Current main has partial startup work beyond v2026.4.26, including non-blocking primary-model prewarm, scoped provider discovery options, per-channel startup trace labels, and a fast skip for missing `gateway_start` hooks. That is not enough to close this report: the issue is fresh, the post-ready 28-30s stall remains outside startup tracing, and current main still disables the trace at `ready` with no `OPENCLAW_GATEWAY_STARTUP_TRACE_AFTER_READY` path or live proof that the reported WebSocket handshake starvation is gone.

## What This Changes

Investigate and fix deterministic gateway restart event-loop stalls around `sidecars.channels` and the immediate post-ready phase that cause local WebSocket handshakes to time out.

## Best Possible Solution

Keep this open and ask the reporter to rerun on current main or the next 2026.4.27 prerelease with `OPENCLAW_GATEWAY_STARTUP_TRACE=1`. The next maintainer step should add after-ready tracing around post-ready hook/plugin work and then use the existing per-channel trace labels to decide whether the remaining fix belongs in channel startup, provider/model discovery, plugin services, or a specific third-party plugin path.
