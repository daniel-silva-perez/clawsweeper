## Summary

Keep open. Current main has more mitigations than the reported 2026.4.26 build, including detached model prewarm and runtime-mirror hardening, but the combined report is not resolved: channel startup still gates readiness, startup tracing still stops at `ready`, and the related lazy-channel/runtime-mirror work remains open rather than a merged terminal fix.

## What This Changes

The issue reports deterministic gateway restart event-loop blocking during `sidecars.channels` plus a second freeze immediately after `gateway ready`, causing local plugin WebSocket handshakes to time out.

## Best Possible Solution

Keep this report open until maintainers capture a current-main restart trace with after-ready attribution, then either fix the measured blocker directly or explicitly link each resolved phase to the lazy-channel and runtime-mirror trackers after their fixes land and verify the CoClaw WebSocket recovery case.

## Reproduction Assessment

Yes for the reported 2026.4.26 release and host: the issue includes concrete restart steps, startup trace timings, and repeated plugin-side event-loop-lag observations. Current-main reproduction is still unclear because this read-only review did not run the reporter's WSL2 packaged restart scenario.

## Solution Assessment

Unclear. Current main's prewarm and runtime-mirror mitigations are useful but do not prove the combined restart/post-ready starvation issue is fixed; after-ready instrumentation plus a current-main rerun is the safer way to assign the final fix boundary.
