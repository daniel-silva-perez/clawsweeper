## Summary

Keep open. Current main contains real partial mitigations for the 2026.4.26 startup path, but it does not prove the reported user problem is gone: channel startup is still readiness-gating, the after-ready stall remains outside startup tracing, and a newer same-author report points to remaining synchronous bundled-runtime mirror work that is still open.

## What This Changes

The issue reports deterministic gateway restart event-loop starvation where `sidecars.channels` and immediate post-ready work delay readiness and cause local WebSocket handshakes to time out.

## Best Possible Solution

Keep the issue open until maintainers add after-ready event-loop/startup tracing or otherwise capture the post-ready stall, then route the fix to the measured owner path: lazy/non-blocking channel startup, bundled-runtime mirror cooperation, or a narrower plugin/provider startup defect.

## Reproduction Assessment

Yes. The issue includes deterministic WSL2/npm 2026.4.26 steps, `OPENCLAW_GATEWAY_STARTUP_TRACE=1` timings, plugin-side event-loop lag data, bonjour watchdog logs, and WebSocket handshake timeout symptoms; I did not run a live reproduction during this read-only review.

## Solution Assessment

No for closing as implemented. Current main is a partial mitigation, not a complete solution, because it still awaits eager channel startup and disables startup event-loop tracing at `ready`; the safer path is targeted instrumentation plus consolidation with the newer runtime-mirror and lazy-channel trackers as appropriate.
