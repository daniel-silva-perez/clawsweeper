## Summary

Keep open. Current main has useful post-release mitigations and diagnostics, but they do not prove the reported restart stall is fixed: channel startup is still awaited before readiness, the startup trace still stops at `ready`, and the immediate post-ready freeze remains outside the existing trace surface.

## What This Changes

The issue asks OpenClaw to diagnose and fix deterministic gateway restart event-loop stalls: about 45 seconds during channel startup plus about 28 seconds immediately after readiness, causing local WebSocket handshake timeouts.

## Best Possible Solution

Keep the issue open, have the reporter rerun on current main or the next prerelease with `OPENCLAW_GATEWAY_STARTUP_TRACE=1`, and add temporary after-ready tracing around post-ready hook/plugin/client startup work if the second freeze still reproduces. Once the trace identifies the slow owner, fix it in the channel, provider/model-discovery, plugin-service, or hook path that actually blocks the event loop.
