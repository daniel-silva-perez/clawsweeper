## Summary

Keep this open. Current main still has no shared per-server startup cooldown/backoff for rate-limited bundle-MCP streamable HTTP or SSE initialization failures; startup errors are logged and disposed locally, while runtime caching remains session-scoped. The overlapping broad reconnect PR was closed unmerged, and the focused open reconnect PR is adjacent lost-session work rather than a fix for rate-limited initialization.

## What This Changes

The issue requests per-server exponential backoff, jitter, and attempt limits for bundle-MCP server initialization when remote MCP servers return rate-limit errors.

## Best Possible Solution

Add a small shared bundle-MCP startup cooldown layer in src/agents/pi-bundle-mcp-runtime.ts that classifies transient rate-limit initialization failures for streamable HTTP/SSE servers, tracks attempts per server across session runtimes with capped jittered backoff and an hourly attempt cap, skips only cooling servers while preserving other MCP tools, and resets state after successful startup.

## Reproduction Assessment

Yes. A high-confidence reproduction can use a fake streamable-HTTP MCP server that returns 429 or temporarily_unavailable during initialize/listTools, then start two separate bundle-MCP sessions; current source shows each session can reach getCatalog() and attempt startup without shared cooldown.

## Solution Assessment

Yes. A narrow runtime-level cooldown is the best direction because the state needs to be shared across session runtimes; placing it only in materialization would remain too session-local, and reviving the broader #66542 reconnect patch would bring unrelated behavior and prior review concerns.

## Repair Work Prompt

Fix bundle-MCP startup retry storms for rate-limited remote MCP servers. Implement a narrow per-server startup cooldown/backoff layer in src/agents/pi-bundle-mcp-runtime.ts, not a broad reconnect rewrite. The observable bug is that streamable HTTP/SSE MCP initialization failures such as HTTP 429 or JSON bodies containing temporarily_unavailable / rate-limit text are retried by each new session with no shared cooldown. Expected boundary: classify transient rate-limit startup failures, record attempts per server across session runtimes, apply capped jittered exponential backoff with an hourly attempt cap, skip only servers currently cooling down while keeping other MCP servers available, reset state on successful initialization, and leave permanent auth failures such as invalid_token visible rather than cooling them indefinitely.
