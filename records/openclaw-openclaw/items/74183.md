## Summary

Keep this issue open. Current main still has no per-server bundle-MCP startup cooldown or attempt cap for rate-limited streamable HTTP/SSE initialization; failures are logged and disposed, then the next session can try again. The related LLM cooldown work is a different provider path, and broader MCP retry PR #66542 was closed unmerged with a maintainer request for a narrower resubmission.

## What This Changes

Add per-server exponential backoff with jitter and attempt caps for bundle-MCP server initialization failures caused by rate-limited HTTP MCP servers.

## Best Possible Solution

Add a small shared startup-cooldown layer owned by `src/agents/pi-bundle-mcp-runtime.ts`: record per-server transient startup failures across session runtimes, apply capped jittered backoff plus an hourly attempt limit, skip only cooling servers while keeping other MCP servers usable, and reset state after a successful connection.

## Repair Work Prompt

Fix the bundle-MCP startup retry storm reported in openclaw/openclaw#74183 with a narrow runtime change. In `src/agents/pi-bundle-mcp-runtime.ts`, add shared per-server startup cooldown state for streamable HTTP/SSE MCP initialization failures that look transient/rate-limited (`429`, `too many requests`, `rate limit`, `temporarily_unavailable`). The cooldown must be shared across session runtimes, use capped jittered exponential backoff, enforce an hourly attempt cap per server, skip only the cooling server while allowing other MCP servers to remain available, and reset the server state after a successful connection. Do not resurrect the broad PR #66542 design: no unbounded parallel startup, no broad reconnect-on-generic-error classifier, no raw transport error cause propagation, and no user-visible leaking of sensitive launch details. Keep permanent auth failures such as `invalid_token` visible instead of hiding them behind transient cooldown. Add targeted Vitest coverage in `src/agents/pi-bundle-mcp-runtime.test.ts` for cooldown classification, cross-session sharing, attempt cap, success reset, and non-rate-limit auth failure behavior. Update docs/changelog only if maintainers treat the behavior as user-facing; do not credit Codex/OpenClaw/steipete in changelog, and ask for maintainer attribution if no acceptable non-bot reporter credit exists.
