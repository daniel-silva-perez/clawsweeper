## Summary

Keep this issue open. Current main still has no shared per-server bundle-MCP startup cooldown/backoff for rate-limited streamable HTTP or SSE initialization; the closest prior LLM cooldown work is provider/auth-profile scoped, and the broader MCP retry PR was closed unmerged after review concerns.

## What This Changes

Add per-server exponential backoff with jitter to bundle-MCP server initialization when rate-limited HTTP/SSE MCP servers fail startup.

## Best Possible Solution

Add a small shared startup cooldown layer in `src/agents/pi-bundle-mcp-runtime.ts`: classify transient rate-limit initialization failures for streamable HTTP/SSE servers, track per-server attempts across session runtimes with capped jittered backoff and an hourly attempt limit, skip only cooling servers while leaving other MCP servers available, and reset state after successful startup.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure a streamable HTTP MCP server that returns 429 or `temporarily_unavailable` during initialization/listing, then materialize bundle MCP tools for two different session IDs; current main will attempt startup again per session because there is no shared cooldown state.

## Solution Assessment

Yes. The narrow maintainable solution is a shared bundle-MCP startup cooldown in the runtime layer, not reviving the closed broad reconnect PR; that keeps startup throttling separate from mid-session reconnect behavior and avoids the prior PR's security/resource concerns.

## Repair Work Prompt

Fix bundle-MCP startup retry amplification for rate-limited HTTP MCP servers. In `src/agents/pi-bundle-mcp-runtime.ts`, add a shared per-server startup cooldown layer used by session MCP runtimes before attempting server initialization. Classify transient rate-limit startup failures for `streamable-http` and `sse` transports, including HTTP 429 and error text such as `temporarily_unavailable` / `Rate limited, try again later`; do not classify permanent auth/config failures such as `invalid_token` as cooldown-only. Track attempts per server across session runtimes, apply capped jittered exponential backoff with an hourly attempt cap, skip only servers currently cooling down while leaving other configured MCP servers available, and reset the server state after successful connect/listTools. Keep the change generic; do not hard-code Cloudflare or any bundled plugin id. Add focused tests in `src/agents/pi-bundle-mcp-runtime.test.ts` for per-server cooldown across sessions, non-rate failures not entering cooldown, other servers still materializing, and reset-on-success. Update docs/changelog only if the behavior becomes user-facing; if adding a changelog entry, keep it one line and credit the reporter appropriately. Related refs: https://github.com/openclaw/openclaw/issues/74183, https://github.com/openclaw/openclaw/issues/68527, https://github.com/openclaw/openclaw/pull/66542.
