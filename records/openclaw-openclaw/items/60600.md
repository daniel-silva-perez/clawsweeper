## Summary

Keep open. Current main still sends the Gemini native-PDF API key in the generateContent URL query string, while the PR moves the same key into the documented and locally established x-goog-api-key header without changing the request body or response handling. The patch is narrow and appears correct, but it still needs normal maintainer review/landing rather than cleanup closure.

## What This Changes

The PR changes src/agents/tools/pdf-native-providers.ts so Gemini PDF generateContent requests omit the key query parameter and send the API key in the x-goog-api-key header.

## Best Possible Solution

Land this narrow auth hardening, preferably with a focused assertion in src/agents/tools/pdf-native-providers.test.ts that Gemini requests do not include a key query parameter and do include x-goog-api-key, then run the targeted native PDF provider test or changed gate before merge.

## Reproduction Assessment

Yes. On current main, inspecting geminiAnalyzePdf shows the outbound generateContent URL includes ?key= and the existing fetch headers lack x-goog-api-key; a mocked fetch test for that function would reproduce the leak without live Gemini credentials.

## Solution Assessment

Yes. Moving the already-normalized API key from the URL to x-goog-api-key is the narrowest maintainable fix because it matches Google's documented generateContent authentication and OpenClaw's other Google request helpers without changing transport, payload, model selection, or parsing.
