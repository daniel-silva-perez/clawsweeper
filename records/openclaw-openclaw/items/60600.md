## Summary

Keep this PR open. Current main still sends the native Gemini PDF API key in the `generateContent?key=` URL, and the existing PDF native-provider tests do not lock in the safer header-auth behavior. The PR is a narrow one-file security hardening change; the related broader #46377 remains open/stale and is not a reason to close this focused candidate.

## Best Possible Solution

Keep this PR open. The best path is to review and land the narrow Gemini PDF auth change, preferably with a small regression test in `src/agents/tools/pdf-native-providers.test.ts`; handle broader native-PDF SSRF hardening separately through #46377 or a refreshed maintainer patch.
