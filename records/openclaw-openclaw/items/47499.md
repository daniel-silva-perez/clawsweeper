## Summary

Keep open. Current main has separate sandbox-browser noVNC observer support and Codex Computer Use setup, but it does not provide the requested dashboard-embedded VNC viewer or an agent-initiated full-desktop takeover permission flow. The linked draft PR was closed unmerged and the remaining request needs product and security review rather than conservative cleanup closure.

## What This Changes

The issue requests an embedded Control UI VNC viewer plus a future agent-initiated full desktop takeover flow with user permission and tokenized VNC control.

## Best Possible Solution

Define a first-class desktop observation/control design before implementation: semantic observation through Codex Computer Use, cua-driver, AX/UIA, or similar should be the primary agent input; VNC/noVNC can be a visual preview when useful. Any takeover path should require explicit operator approval, scoped short-lived tokens, gateway auth/origin checks, denial fallback, auditability, and docs/tests for Control UI and sandbox-browser behavior.

## Reproduction Assessment

Not applicable as a feature request. The current-main check is high-confidence: targeted searches and docs/source inspection show no embedded VNC viewer in ui/src, while the only noVNC support is the separate sandbox observer-token path.

## Solution Assessment

Unclear. The requested VNC-first takeover direction is not clearly the best solution because the discussion notes coordinate-click brittleness; a safer maintainable design would combine semantic desktop-control backends with an optional visual preview and explicit permission boundaries.

## Security Review

Status: needs_attention

Summary: The issue proposes a security-sensitive full-desktop control surface, so any implementation needs explicit auth, permission, token, origin, and credential-handling design before coding.

Concerns:

- **[high] Constrain desktop takeover authority:**
  - body: A dashboard VNC viewer plus agent takeover can expose full host input and screen state. The design should require operator approval, short-lived scoped grants, denial fallback, and audit logging rather than storing reusable desktop-control credentials.
  - confidence: 0.86
- **[high] Avoid unsafe VNC proxying and password storage:**
  - body: The prior linked PR review context flagged plaintext VNC password persistence and arbitrary-target proxy risk. A replacement design should keep VNC endpoints loopback/scoped, avoid persistent browser-stored passwords, and enforce gateway auth and SSRF/origin boundaries.
  - confidence: 0.83
