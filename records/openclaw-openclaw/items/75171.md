## Summary

Keep open. Current main still matches the reported public SDK compatibility regression: the focused channel reply SDK subpath exports `createReplyPrefixContext`, but the deprecated compat/root SDK surface used by legacy external plugins during the migration window does not expose it, so a legacy Feishu plugin import can still resolve to `undefined` at dispatch time.

## What This Changes

External Feishu/Lark plugins using legacy `openclaw/plugin-sdk` imports should continue dispatching messages after update instead of failing on a missing `createReplyPrefixContext` export.

## Best Possible Solution

Restore the missing reply-prefix and reply-pipeline helpers on the deprecated compat/root SDK surface for the documented migration window, keep `openclaw/plugin-sdk/channel-reply-pipeline` as the canonical focused import, and add contract tests that lock the legacy root behavior until the removal date.

## Reproduction Assessment

Yes. The issue provides a deterministic Feishu dispatch path, and current main reproduces the underlying failure condition by source inspection: legacy root imports are routed through compat, while compat lacks the helper exported by the focused subpath.

## Solution Assessment

Yes. A narrow public SDK compatibility repair is the maintainable fix; changing only the Feishu plugin would leave other external plugins using the still-supported deprecated root/compat surface broken.
