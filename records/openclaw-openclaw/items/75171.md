## Summary

Keep open. Current main still matches the reported SDK compatibility regression: the helper exists on the focused channel reply SDK subpath, but the deprecated root/compat surface used by external plugins during the migration window does not expose it, producing the reported runtime `TypeError`.

## What This Changes

External Feishu/Lark plugin dispatch should keep working after updating to OpenClaw 2026.4.27 when it still imports channel reply helpers through the deprecated root/compat plugin SDK surface.

## Best Possible Solution

Restore the missing channel reply helper exports on the deprecated compat surface used by the plugin-loader root alias for the documented migration window, keep `openclaw/plugin-sdk/channel-reply-pipeline` as the canonical focused import, and add contract coverage so future SDK migration edits cannot drop these legacy exports silently.

## Reproduction Assessment

Yes. The issue gives deterministic Feishu dispatch steps, and current main reproduces the underlying failure condition by source inspection: `createReplyPrefixContext` is exported by `channel-reply-pipeline` but not by the compat-backed root SDK path used by legacy external plugin imports.

## Solution Assessment

Yes. A narrow SDK compatibility repair is the best fix; changing only bundled Feishu would not help external plugins still allowed to use deprecated root/compat imports during the migration window.
