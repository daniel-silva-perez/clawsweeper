## Summary

Keep open. Current main still has the alias-resolution gap described in the existing ClawSweeper comment: bare aliases can resolve through the shared model resolver, but a provider-qualified display alias such as `lmstudio-moe/MoE` is parsed as a direct provider/model ref and can be persisted as `model: "MoE"` for that provider instead of the configured canonical model ID.

## What This Changes

The issue asks for Control UI/TUI mid-session model switching to resolve configured display aliases to canonical provider/model IDs before storing the session override.

## Best Possible Solution

Resolve configured model aliases in the shared model-selection boundary before session persistence, including provider-qualified display aliases where the provider matches the configured alias target, and keep the Control UI picker values canonical when catalog metadata is available.

## Reproduction Assessment

Yes. A current-main code-path reproduction is a session row with `modelProvider: "lmstudio-moe"`, `model: "MoE"`, and a configured catalog entry whose canonical ID has alias `MoE`; the UI can surface `lmstudio-moe/MoE`, and Gateway parses that as a direct model ref instead of resolving the alias.

## Solution Assessment

Yes for a shared resolver fix, unclear for a UI-only fix. The narrow maintainable path is to canonicalize aliases in the shared Gateway/model-selection layer and add UI tests only to prevent the picker from reintroducing provider-qualified display aliases.
