## Summary

Keep open. Current main still has the reported alias-resolution gap, and an open PR explicitly closes this issue, so the issue should remain open until that PR is reviewed and either merged or rejected.

## What This Changes

The issue reports that Control UI/TUI model switching can persist a display alias such as `MoE` instead of the canonical provider model ID, causing local providers to reject the next request.

## Best Possible Solution

Review and land a narrow shared resolver fix that maps `provider/Alias` to the alias target only when the provider matches, with regression coverage in model selection and `sessions.patch`; avoid solving this only in one UI caller.

## Reproduction Assessment

Yes. A session/catalog setup with provider `lmstudio-moe`, server model `MoE`, and alias `MoE` for canonical `qwen3.6-35b-a3b` follows the current source path to `lmstudio-moe/MoE`, which the Gateway treats as a direct provider/model ref.

## Solution Assessment

Yes, if implemented at the shared Gateway/model-selection boundary. The open linked PR’s stated direction matches that narrower approach better than a UI-only fix because it protects all `sessions.patch` callers.
