## Summary

Keep open. Current main still misses the provider-qualified alias case: the shared resolver checks the whole raw string against configured aliases before parsing, so `lmstudio-moe/MoE` does not match alias `MoE` and can be treated as provider `lmstudio-moe` plus raw model `MoE`. The issue is already paired with open closing PR #75198, so it should stay open until that PR or an equivalent fix lands.

## What This Changes

Resolve model aliases selected mid-session in the TUI/Control UI to canonical provider/model IDs before persisting session overrides.

## Best Possible Solution

Land or adapt PR #75198's shared resolver approach: canonicalize provider-qualified aliases when the provider prefix matches the alias target, with focused regression coverage in `src/agents/model-selection.test.ts` and `src/gateway/sessions-patch.test.ts` so all `sessions.patch` callers persist canonical provider/model IDs.

## Reproduction Assessment

Yes. A high-confidence code reproduction is to configure `agents.defaults.models["lmstudio-moe/qwen3.6-35b-a3b"].alias = "MoE"`, send `sessions.patch` with `model: "lmstudio-moe/MoE"`, and observe from the current resolver that it parses `MoE` as the raw model instead of resolving the alias.

## Solution Assessment

Yes. The shared resolver/session-patch fix direction is the narrow maintainable solution; a UI-only workaround would leave `/model`, TUI, SDK, and other session-patch callers with divergent alias behavior.
