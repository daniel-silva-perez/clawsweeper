## Summary

Keep open. Current main still does not resolve provider-qualified display aliases like `lmstudio-moe/MoE` to the configured canonical model ID, and the issue is already paired with open closing PR #75198, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue reports that Control UI/TUI mid-session model switching can persist a configured display alias instead of the canonical provider model ID, causing LM Studio to reject the next request.

## Best Possible Solution

Resolve configured aliases at the shared model-selection/Gateway boundary, including `provider/Alias` when the provider matches the alias target, and keep focused resolver plus `sessions.patch` regression tests proving the persisted session override is the canonical provider/model ID. The existing open PR #75198 is the current implementation candidate.

## Reproduction Assessment

Yes. A high-confidence code reproduction is to configure `agents.defaults.models["lmstudio-moe/qwen3.6-35b-a3b"].alias = "MoE"`, send `sessions.patch` with `model: "lmstudio-moe/MoE"`, and inspect whether the session stores `qwen3.6-35b-a3b` or treats `MoE` as the raw model; current main follows the latter path from the source inspection.

## Solution Assessment

Yes for the shared resolver direction, not for a UI-only workaround. The narrow maintainable fix is to canonicalize provider-qualified aliases before session persistence so Control UI, TUI, `/model`, and other session-patch callers share the same behavior.
