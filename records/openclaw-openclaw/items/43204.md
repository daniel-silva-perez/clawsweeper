## Summary

Current main still does not implement this PR's intended TTL-expiry feedback path: the typing TTL only logs and cleans up, with no hook or default user-visible notice. The PR is still potentially useful, but it should not land as-is because its head is stale and mixed-scope, including unrelated macOS/Tlon/package-resolution changes and an unresolved synchronous-throw path in the custom TTL handler wrapper.

## Best Possible Solution

Keep this PR open as reference, but do not merge it as-is. The maintainable path is a fresh or heavily rebased, auto-reply-only change against current main that adds the chosen TTL-expiry feedback contract, contains both synchronous throws and async rejections from custom handlers, preserves routed-reply ordering, adds focused tests for TTL expiry, suppressed typing, and routed replies, and drops the unrelated macOS/Tlon/package-resolution changes. Coordinate product direction with #40760 before choosing TTL-expiry notices versus longer timeout-aware typing budgets or both.

## Risks / Open Questions

- Landing this PR as-is could leave custom `onTypingTtlExpired` handlers able to throw synchronously out of the timer callback because the PR wrapper uses `Promise.resolve(ttlHandler()).catch(...)`, which evaluates the handler before the catch chain exists.
- The PR mixes the auto-reply UX fix with unrelated macOS onboarding/remote probe changes and Swift formatting, increasing review and regression surface outside the stated bug.
- The PR changes package resolution for the Tlon plugin via a GitHub tarball/lockfile update, which is supply-chain-sensitive and unrelated to typing TTL feedback; current main has already moved past that dependency shape.
