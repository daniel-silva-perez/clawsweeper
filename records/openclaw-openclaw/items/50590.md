## Summary

Keep open. Current main still has the reported runtime gap: non-rw sandbox setup copies eligible skills into the sandbox workspace, but embedded runs can keep using a host-built skill snapshot whose prompt wins over sandbox-loaded entries, so prompt-visible skill locations can still point at global `~/.openclaw/skills/...` paths. Same-day issue discussion also reports an active workaround, and the adjacent open PRs are not merged fixes for the runtime remap contract.

## What This Changes

The issue asks OpenClaw to make skills usable from non-main sandboxed sessions by resolving `SKILL.md` paths to sandbox-readable copies, or otherwise allowlisting the managed skills directory.

## Best Possible Solution

Sandboxed runs with `workspaceAccess: "none"` or `"ro"` should render `<available_skills>` from sandbox-readable copied skill entries, or remap snapshot prompt locations to those copies, while preserving canonical host-resolved skill metadata for config, env, and session state. Add regression coverage for a managed global skill used from a non-main sandboxed session.
