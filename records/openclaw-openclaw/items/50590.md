## Summary

Keep open. Current main still has the reported skills/sandbox mismatch: non-rw sandbox setup mirrors skills into the sandbox workspace, but embedded and cron-backed runs can still reuse a prebuilt host-side skills snapshot whose prompt takes precedence over sandbox-loaded entries. Same-day issue discussion also reports an active workaround, and the two adjacent PRs are open/unmerged rather than a shipped runtime remap.

## What This Changes

Sandboxed non-main runs should advertise skill `SKILL.md` locations that are readable inside the sandbox instead of host-global `~/.openclaw/skills/...` paths.

## Best Possible Solution

Sandboxed runs with `workspaceAccess: "none"` or `"ro"` should render `<available_skills>` from sandbox-readable copied skill entries, or remap prebuilt snapshot prompt locations to those copies, while preserving canonical host-resolved skill metadata for config, env overrides, and session state. Add focused regression coverage for a managed skill used from a non-main sandboxed embedded or cron session.
