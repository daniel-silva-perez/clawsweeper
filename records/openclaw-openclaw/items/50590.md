## Summary

Keep open. Current main has the sandbox skill-copy step, but a sandboxed run can still consume a prebuilt skills snapshot whose prompt advertises the original managed/global skill path, so the reported sandbox read failure is not fully fixed.

## What This Changes

Ensure sandboxed non-main runs resolve skill `SKILL.md` locations from sandbox-local copies rather than the global managed skills path.

## Best Possible Solution

Keep this issue open and fix the runtime contract: after sandbox skill sync, sandboxed runs with `workspaceAccess` `none` or `ro` should advertise sandbox-readable `skills/<skill>/SKILL.md` locations, or rebuild/remap the skills prompt from sandbox-local copies when consuming a skills snapshot. Preserve canonical `resolvedSkills` for host-side env/config handling and add regression coverage for a managed `~/.openclaw/skills/<name>/SKILL.md` skill under a non-main sandboxed session.
