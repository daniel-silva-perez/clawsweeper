## Summary

Keep open. Current main still has the sandbox/skills split described in the report: non-readwrite sandbox setup copies eligible skills into the sandbox workspace, but embedded and cron runs can continue to render a prebuilt skills snapshot whose prompt wins over sandbox-loaded entries. The April 29 issue comment also says the symptom is still current, and the adjacent open PRs are prompt/sync-adjacent rather than proof that the runtime remap contract is fixed.

## What This Changes

The issue asks OpenClaw to make sandboxed non-main and cron sessions advertise or resolve skill `SKILL.md` locations to sandbox-readable copied skill files instead of host/global managed skill paths.

## Best Possible Solution

Sandboxed runs with `workspaceAccess: "none"` or `"ro"` should render `<available_skills>` from sandbox-readable copied skill entries, or remap snapshot prompt locations to those copies, while retaining canonical host-resolved skill metadata for runtime config and environment handling. Add focused regression coverage for managed skills in non-main embedded and cron sessions.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction path: build a skills snapshot from the agent workspace, run an embedded or cron session with non-readwrite sandbox access, and observe that the snapshot prompt is preferred over entries from the sandbox workspace. A live sandbox run was not executed because this review was read-only.

## Solution Assessment

Unclear for the final implementation, but a prompt-only instruction is not the best fix because the advertised location can still be inaccessible. The maintainable direction is a runtime remap or post-sandbox prompt rebuild that keeps the sandbox boundary intact instead of broadly exposing host skill directories.
