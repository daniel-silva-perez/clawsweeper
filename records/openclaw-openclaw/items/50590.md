## Summary

Keep open. Current main still has the reported sandbox/skills mismatch: non-rw sandbox setup copies eligible skills into the sandbox workspace, but embedded runs can still keep and render a prebuilt host-workspace skills snapshot before rebuilding from the sandbox-readable copy. Same-day issue discussion also reports an active workaround, and the adjacent PRs are open/unmerged rather than shipped runtime fixes.

## What This Changes

Sandboxed non-main runs should advertise or resolve managed skill `SKILL.md` locations through sandbox-readable copies instead of host-global `~/.openclaw/skills/...` paths.

## Best Possible Solution

Sandboxed runs with `workspaceAccess: "none"` or `"ro"` should render `<available_skills>` from sandbox-readable copied skill entries, or remap prebuilt snapshot prompt locations to those copies, while preserving canonical host-resolved skill metadata for config, env overrides, and session state. Add focused regression coverage for a managed skill used from a non-main sandboxed embedded or cron session.
