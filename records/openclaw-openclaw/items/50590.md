## Summary

Current main has partial sandbox skill support, but the reported failure is not safely resolved. The sandbox setup copies eligible skills into the sandbox workspace, yet sandboxed runs can still reuse a prebuilt skills snapshot whose prompt locations came from the original workspace or shared managed skill root. That means the model may still be told to read `~/.openclaw/skills/<name>/SKILL.md` even after a sandbox-local copy exists. A related open PR, #74161, is a prompt-level mitigation, but it does not prove current main remaps skill locations to sandbox-readable paths.

## Best Possible Solution

Keep this issue open and fix the runtime contract: after sandbox skill sync, sandboxed runs with `workspaceAccess` `none` or `ro` should advertise sandbox-readable `skills/<skill>/SKILL.md` locations, or rebuild/remap the skills prompt from the sandbox-local copies when consuming a skills snapshot. Preserve canonical `resolvedSkills` for host-side env/config handling, and add a regression test covering a managed `~/.openclaw/skills/<name>/SKILL.md` skill under a non-main sandboxed session.
