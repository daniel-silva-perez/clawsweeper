## Summary

Keep open. Current main still has the reported mismatch: non-rw sandbox setup copies eligible skills into the sandbox workspace, but embedded and cron runs can keep using a prebuilt skills snapshot whose prompt wins over sandbox-loaded entries, so the advertised `SKILL.md` path can remain host/global instead of sandbox-readable. The April 29 discussion also confirms this is still happening in practice.

## What This Changes

The issue reports that non-main sandboxed agent runs resolve managed skill `SKILL.md` paths to the global `~/.openclaw/skills/...` location instead of the sandbox-local synced copy, causing sandbox read failures.

## Best Possible Solution

Sandboxed runs with `workspaceAccess: "none"` or `"ro"` should render `<available_skills>` from sandbox-readable copied skill entries, or remap snapshot prompt locations to those copies, while preserving canonical host-resolved skill metadata for runtime config and env handling. Add focused regression coverage for managed skills in non-main embedded and cron sessions.
