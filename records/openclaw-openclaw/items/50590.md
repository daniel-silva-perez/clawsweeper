## Summary

Keep open. Current main still has the risky split: non-rw sandbox setup syncs eligible skills into the sandbox workspace, but embedded runs can reuse a prebuilt skills snapshot whose prompt takes precedence over sandbox-loaded entries. That means a managed skill can still be advertised with a host/global `~/.openclaw/skills/.../SKILL.md` location even when the sandbox file tools need the sandbox-local copy. The same-day issue discussion also confirms the symptom is still current, and the adjacent PRs in the thread do not yet prove the runtime remap contract is fixed.

## What This Changes

The issue asks for sandboxed non-main agent runs to resolve skill `SKILL.md` locations to sandbox-readable copies, or otherwise allow access to managed skills, instead of advertising inaccessible global `~/.openclaw/skills` paths.

## Best Possible Solution

Sandboxed runs with `workspaceAccess: "none"` or `"ro"` should render `<available_skills>` from sandbox-readable skill copies, or remap snapshot prompt locations to those copies, while keeping canonical host-resolved skill metadata available for config, env, and session state. Add focused regression coverage for a managed `~/.openclaw/skills/<name>/SKILL.md` skill used from a non-main sandboxed session.
