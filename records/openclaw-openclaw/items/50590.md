## Summary

Keep open. Current main still has the reported sandbox/skills mismatch: non-readwrite sandbox setup mirrors skills into the sandbox workspace, but embedded and cron-style runs can keep using a prebuilt skills snapshot whose prompt is returned before sandbox-local entries are loaded. The April 29 issue follow-up also says the problem is still current, and the adjacent open PRs are not merged proof of a runtime remap fix.

## What This Changes

The issue asks for sandboxed non-main skill runs to resolve SKILL.md locations to sandbox-readable copied skills instead of host-global managed skill paths.

## Best Possible Solution

Sandboxed runs with workspaceAccess "none" or "ro" should render <available_skills> from sandbox-readable copied skill entries, or remap snapshot prompt locations to those copies, while retaining host-resolved skill metadata only for config, env, and session state. Add focused regression coverage for managed ~/.openclaw/skills entries used from embedded and isolated cron non-main sandbox sessions.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to build a skills snapshot from the agent workspace, enter a non-readwrite sandbox run, and observe that the snapshot prompt is preferred over entries loaded from the sandbox workspace; live sandbox execution was not run because this review was read-only.

## Solution Assessment

No for a host allowlist or prompt-only workaround as the primary fix, because the advertised location can still be inaccessible or the sandbox boundary can be weakened. The maintainable solution is a runtime remap or post-sandbox prompt rebuild that keeps file reads sandbox-local.
