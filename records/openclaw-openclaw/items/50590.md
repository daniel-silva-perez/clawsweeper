## Summary

Keep open. Current main copies eligible skills into the sandbox workspace for non-rw sandbox access, but embedded runs can still reuse a host-built skills snapshot whose prompt advertises the original managed/global skill path. The issue discussion also has a current confirmation and workaround, and the adjacent prompt-only PR does not fix the runtime remap contract.

## What This Changes

The issue asks OpenClaw to make sandboxed non-main sessions read sandbox-local SKILL.md copies, or otherwise allow access to managed skills, instead of advertising inaccessible global managed-skill paths.

## Best Possible Solution

Sandboxed runs with workspaceAccess none or ro should build the available-skills prompt from sandbox-readable skill copies, or remap snapshot prompt locations to those copies, while keeping canonical resolvedSkills available for host-side config and env handling. Add regression coverage for a managed ~/.openclaw/skills/<name>/SKILL.md skill under a non-main sandboxed session.
