## Summary

Keep open. Current main still has the reported skills/sandbox split: non-rw sandbox setup copies eligible skills into the sandbox workspace, but embedded and cron-backed runs can keep using a prebuilt host-side skills snapshot whose prompt takes precedence over sandbox-loaded entries. Same-day issue discussion also reports an active workaround, and the adjacent open PRs do not yet prove a merged runtime remap fix.

## What This Changes

The issue requests that sandboxed non-main runs resolve skill `SKILL.md` locations to sandbox-readable skill copies instead of host-managed skill paths.

## Best Possible Solution

Sandboxed runs with non-rw workspace access should render `<available_skills>` from sandbox-readable copied skill entries, or remap snapshot prompt locations to those copies, while preserving canonical host-resolved skill metadata for config, environment, and session state. Add regression coverage for a managed shared skill used from a non-main sandboxed embedded or cron session.
