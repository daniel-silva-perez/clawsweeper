## Summary

Keep open. Current main has a partial sessions.list optimization, but the reported sessionId resolution and MCP conversation_get paths still reuse list-row construction, and selected rows can still synchronously read transcript files for usage, derived titles, and last-message previews.

## What This Changes

The issue requests avoiding O(total sessions) session list, sessionId lookup, and MCP conversation lookup work by applying filters/limits before transcript enrichment, adding direct sessionId lookup, and moving transcript-derived summary fields out of query-time synchronous reads.

## Best Possible Solution

Keep the current limit-before-enrichment behavior, add a metadata-only session key/sessionId lookup and MCP conversation descriptor path that avoids list-row hydration, and treat persistent write-time summary indexing as a separate storage design if maintainers choose that direction.

## Reproduction Assessment

Yes. Static reproduction against current main is high-confidence: sessionId resolution still calls listSessionsFromStore without a limit, and MCP getConversation still makes a 500-row sessions.list request with transcript-derived fields enabled by default.

## Solution Assessment

Unclear. The proposed SessionSummaryIndex may be a good long-term storage direction, but the narrowest maintainable next step is likely direct metadata lookup plus an MCP descriptor path that avoids transcript-backed row hydration.
