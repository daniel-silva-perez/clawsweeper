## Summary

Keep open. Current main has implemented the cheap part of the report by applying search, active-window filtering, and limit before gateway row enrichment, with regression coverage and a changelog entry. The full reported problem is not resolved: returned rows can still synchronously hydrate transcript-derived usage/title fields, sessionId resolution still materializes a sessions.list result and filters it, and MCP getConversation still scans listConversations({ limit: 500 }). Related #57715/#59436 overlap with the sessions.list hot path, but they do not fully cover the direct sessionId and MCP lookup pieces from this report.

## Best Possible Solution

Keep this issue open and either consolidate it into a canonical performance tracking issue that explicitly covers sessions.list, direct sessionId lookup, and MCP getConversation, or fix the remaining pieces directly. A practical path is to preserve the shipped early search/limit behavior, add a lightweight direct store lookup for sessionId/key resolution that keeps current visibility/deleted-agent semantics, teach MCP getConversation to avoid listConversations(limit: 500), and separately decide whether write-time summary persistence is needed for transcript-derived usage/title fields.
