## Summary

Keep open. Current main has a partial bounded-list optimization, but the remaining reported behavior is still present: sessionId resolution and MCP conversation_get still reuse session-list row materialization, and returned rows can still synchronously hydrate transcript-derived usage/title fields.

## What This Changes

The issue requests avoiding full session-row materialization and synchronous transcript reads for session list, direct sessionId lookup, and MCP conversation_get paths, ideally via precomputed summary metadata or an index.

## Best Possible Solution

Keep the current early limit/search behavior, then add a metadata-only lookup surface for sessionId/key resolution and MCP conversation_get so those paths do not construct list rows or touch transcripts. Treat persistent write-time summary indexing as a separate design decision unless maintainers choose it as the canonical storage direction.

## Reproduction Assessment

Yes. Static reproduction against current main is high-confidence: sessionId resolution still routes through listSessionsFromStore without a limit, and MCP getConversation still uses a 500-row sessions.list request whose rows may synchronously hydrate transcript fields.

## Solution Assessment

Unclear. The current main implementation is only partial; the safest next step is likely a narrow direct lookup/descriptor path, while the proposed SessionSummaryIndex is broader and should be explicitly scoped by maintainers.
