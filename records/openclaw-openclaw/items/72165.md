## Summary

Keep open. Current main fixes the bounded session-list part by applying search, activity filtering, and limit before row enrichment, but the broader report remains valid: returned rows can still synchronously hydrate transcript-derived fields, sessionId resolution still goes through full sessions.list row materialization, and MCP getConversation still scans a 500-row conversation list.

## What This Changes

The issue requests faster session listing and lookup by avoiding full session-row materialization, synchronous transcript reads, list-based sessionId lookup, and MCP getConversation list reuse.

## Best Possible Solution

Keep the shipped/current early-filtering behavior, then add a metadata-only or indexed lookup path for sessionId/key resolution and MCP getConversation so they avoid sessions.list row enrichment. Treat persistent write-time summary indexing as a separate design decision unless maintainers explicitly want this issue to own it.

## Reproduction Assessment

Yes. Static reproduction against current main is high confidence: sessionId resolution still calls listSessionsFromStore with no limit, and MCP getConversation still calls listConversations({ limit: 500 }) before finding one session. No live latency benchmark was run because this was a read-only review.

## Solution Assessment

No, current main is only a partial solution. The narrow maintainable next step is to remove list-row materialization from direct lookup paths while preserving existing visibility and deleted-agent semantics; a full persisted SessionSummaryIndex should be decided separately.
