## Summary

Keep open. Current main has a partial sessions.list optimization that limits entries before row enrichment, but the reported direct sessionId resolution and MCP conversation_get paths still reuse list-row construction, and returned rows can still synchronously read transcript files for usage, derived titles, and last-message previews.

## What This Changes

The issue asks for session list and lookup queries to avoid full session row materialization and synchronous transcript reads before filtering, search, direct lookup, and pagination are applied.

## Best Possible Solution

Keep the current limit-before-enrichment behavior, add metadata-only direct key/sessionId lookup and an MCP conversation descriptor path that hydrates only the requested target, and leave any persistent write-time summary index as a separate storage-design follow-up.

## Reproduction Assessment

Yes. Static reproduction against current main is high-confidence: sessionId resolution still calls listSessionsFromStore without a limit, and MCP getConversation still makes a 500-row sessions.list request with transcript-derived fields enabled by default.

## Solution Assessment

Unclear as proposed. The SessionSummaryIndex may be a good long-term storage direction, but the narrowest maintainable first fix is direct metadata lookup plus a targeted MCP descriptor path that avoids transcript-backed row hydration for single-session lookups.

## Repair Work Prompt

Fix the remaining performance problem from https://github.com/openclaw/openclaw/issues/72165. Current main already limits sessions.list before row enrichment; do not replace that or build a broad persistent SessionSummaryIndex. Add a narrow metadata-only direct lookup path for key/sessionId resolution that preserves includeGlobal/includeUnknown, agentId, spawnedBy visibility, deleted-agent validation, duplicate-sessionId errors, and legacy key migration semantics. Update MCP conversation_get so it no longer calls listConversations({ limit: 500, includeLastMessage: true }) to find one session; use a direct descriptor/metadata path that hydrates at most the requested session and preserves toConversation fields. Keep core generic, no bundled plugin IDs. Add or update focused tests for sessionId lookup avoiding list-row hydration and for MCP getConversation avoiding a 500-row sessions.list scan. Because this is a user-facing perf fix, add a single-line CHANGELOG.md Fixes/Changes entry crediting Thanks @6tizer. Validate with targeted tests and then the changed gate per repo policy.
