## Summary

Keep open. Current main only partially addresses the report: session listing now applies limit before row enrichment, but sessionId resolution and MCP conversation lookup still route through session-list materialization, and returned rows can still synchronously hydrate transcript-derived fields.

## What This Changes

The issue requests a session summary/direct lookup path so list/search/limit and single-session/MCP lookups avoid full row materialization and synchronous transcript reads across all sessions.

## Best Possible Solution

Keep the existing early filter/limit behavior, add a metadata-only direct lookup surface for key/sessionId and MCP conversation descriptors that avoids transcript reads, and treat persisted write-time summary indexing as a separately scoped storage decision.

## Reproduction Assessment

Yes. Static reproduction against current main is high confidence: sessions.resolve(sessionId) calls listSessionsFromStore without a limit, and MCP getConversation calls listConversations(limit: 500) with last-message/derived-title behavior that can hit synchronous transcript reads.

## Solution Assessment

Unclear. The reported problem is real, but the proposed SessionSummaryIndex is broader than the narrowest maintainable fix; a direct lookup/descriptor path should be reviewed before committing to persisted summary indexing.
