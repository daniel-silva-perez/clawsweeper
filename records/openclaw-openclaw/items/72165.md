## Summary

Keep open. Current main has a partial list-side optimization, but the reported direct lookup and MCP paths still reuse session list row construction, and selected rows can still synchronously read transcript files for usage, titles, and previews.

## What This Changes

The issue requests avoiding full session row materialization and synchronous transcript reads for session list pagination, sessionId lookup, and MCP conversation_get, possibly through write-time session summaries and direct lookup.

## Best Possible Solution

Keep the existing early filter/search/limit behavior, then add a metadata-only lookup/descriptor path for sessionId/key resolution and MCP conversation_get so those flows do not build list rows or scan transcript-derived fields. Treat persistent write-time summary indexing as a separate storage design unless maintainers choose it as the canonical direction.

## Reproduction Assessment

Yes. Static reproduction against current main is high-confidence: sessionId lookup still calls listSessionsFromStore without a limit, and MCP getConversation still makes a 500-row sessions.list request that can request transcript-derived fields.

## Solution Assessment

Unclear. The reported SessionSummaryIndex may be a good long-term storage direction, but the narrowest maintainable next fix is likely direct metadata lookup plus a conversation descriptor path that avoids list-row hydration.
