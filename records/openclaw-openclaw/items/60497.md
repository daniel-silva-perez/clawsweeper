## Summary

Keep open. Current main still suppresses default-verbosity, non-timeout exec/bash failures before the mutating-tool warning path runs, and current tests lock in that suppression, so a command-not-found exec can still leave success-looking assistant text as the only user-visible payload.

## What This Changes

The issue asks OpenClaw to surface failed exec tool calls, especially command-not-found failures, instead of allowing fabricated successful assistant output to stand alone.

## Best Possible Solution

Update the embedded-run payload policy so failed mutating exec/bash actions, including command-not-found, emit a concise default-visible failure payload and prevent success-looking assistant text from standing alone after the matching action failed, while preserving explicit warning suppression, verbose raw details, sessions_send suppression, and timed-out cron behavior.

## Reproduction Assessment

Yes for the OpenClaw-side behavior: current source and tests provide a deterministic reproduction path through `buildEmbeddedRunPayloads` with `lastToolError.toolName = "exec"`, `error = "Command not found"`, default verbosity, and assistant success text. The exact NemoClaw/NVIDIA model transcript was not re-run live.

## Solution Assessment

Yes, a narrow payload-policy fix is the best path. The exec runtime already classifies command-not-found and the subscription handler retains failed tool state, so changing the destructive-command catalog, NemoClaw sandbox setup, or low-level exec runtime would not address the remaining OpenClaw user-visible failure.
