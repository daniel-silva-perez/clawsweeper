## Summary

Keep open. Current main has partial operational hardening for service restart limits, Telegram doctor SecretRef resolution, Chrome SingletonLock cleanup, and multiple state-dir warnings, but the central reported `channels.telegram.retry.jitter: true` startup validation failure is still present: the shared retry schema remains number-only and Telegram still uses that schema directly. The related boolean-jitter fix PR #52137 is closed and unmerged in the provided context, so there is no landed canonical fix to close against.

## Best Possible Solution

Keep this issue open or split it. The concrete remaining maintainer path is to either accept boolean retry jitter values by preprocessing `true`/`false` in the shared retry schema with focused config regression tests, or explicitly keep jitter numeric-only and add doctor/config repair guidance that flags or converts the boolean before gateway startup. Once that lands, the existing service restart, Telegram doctor SecretRef, Chrome lock, and multi-state-dir hardening can be cited in an implemented-on-main close review.
