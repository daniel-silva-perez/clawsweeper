## Summary

Keep open. Current main still has the Slack capabilities failure path: Slack scope discovery only tries `auth.scopes` and `apps.permissions.info`, does not read `auth.test` metadata or scope headers, and the CLI still formats failures as `Bot scopes: ...`. The reproduced follow-up comment and closed unmerged PR #44692 make this a valid narrow Slack plugin bug, not cleanup closure.

## What This Changes

The issue reports that `openclaw channels capabilities` fails Slack bot scope discovery with `unknown_method` for modern Slack bot tokens.

## Best Possible Solution

Implement the fix inside the Slack plugin: discover scopes for modern granular bot tokens through `auth.test` response metadata or accessible Web API scope headers, keep legacy scope methods compatible, and add focused tests for extraction plus `channels capabilities` diagnostics.

## Reproduction Assessment

Yes. The issue has concrete CLI steps, a follow-up commenter reproduced it on current upstream/main, and current-main source inspection still shows the two-method failure path. I did not run a live workspace call because no Slack token was present in the environment or profile.

## Solution Assessment

Yes. A Slack-plugin fix is the narrow maintainable solution because capability diagnostics and Slack probing are already plugin-owned; no core channel API change is needed. The smaller implementation is to add `auth.test` after the legacy methods, with maintainers free to make it the primary path if they prefer modern bot tokens as the fast path.
