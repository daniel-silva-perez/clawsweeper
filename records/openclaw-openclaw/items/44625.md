## Summary

Keep open. Current main still has the Slack capabilities failure path: scope discovery only tries the two legacy Slack permission methods, does not read `auth.test` scope metadata, and the CLI diagnostics still surface failed scope discovery as `Bot scopes: ...`. The follow-up reproduction comment and closed unmerged PR #44692 are useful context, but no merged or open replacement fix is present.

## What This Changes

The issue asks for Slack channel capability checks to report scopes normally for modern Slack bot tokens instead of failing with `unknown_method` from legacy scope APIs.

## Best Possible Solution

Implement the fix inside the Slack plugin: discover scopes for modern granular bot tokens through `auth.test` response metadata or exposed scope headers, preserve compatibility with the legacy methods, and add focused tests for helper extraction plus `channels capabilities` diagnostics.

## Reproduction Assessment

Yes. The issue has concrete CLI steps, a follow-up commenter reproduced it on current upstream/main, and current-main source inspection shows the two-method failure path still exists. A live workspace check was not run because no Slack token was available.

## Solution Assessment

Yes. The narrow maintainable fix belongs in `extensions/slack`, because the plugin already owns capability diagnostics and already uses `auth.test` for probing; no core channel API change is needed.

Add focused tests for successful `auth.test` scope extraction after legacy `unknown_method` failures, deduped/sorted scopes, all-method failure messaging, and capabilities diagnostics output.
