## Summary

Keep this open. Current main still has the reported Slack capabilities failure path: scope discovery only tries `auth.scopes` and `apps.permissions.info`, and the command renders those failures through Slack capability diagnostics. The closed fallback PR is useful prior art, but it was not merged and no current-main replacement exists.

## What This Changes

The issue requests that Slack channel capability diagnostics report bot scopes normally for modern Slack bot tokens instead of failing with `unknown_method` from legacy Slack API methods.

## Best Possible Solution

The Slack plugin should use one tested scope-discovery helper that works for modern granular bot tokens and legacy workspace-app tokens, preferably by adding `auth.test` response-metadata scope extraction without broadening the core channel API or moving Slack-owned behavior out of the plugin.

## Repair Work Prompt

Fix the Slack capabilities scope-discovery bug reported in openclaw/openclaw#44625. Current `extensions/slack/src/scopes.ts` only tries `auth.scopes` and `apps.permissions.info`; modern granular Slack bot tokens can return `unknown_method` for both, causing `openclaw channels capabilities` to render a bot-scope error. Implement the fix in the Slack plugin, not core. Add `auth.test` support to the scope-discovery helper, parsing granted scopes from the Slack SDK `WebAPICallResult.response_metadata.scopes` / `x-oauth-scopes` mapping. Preserve the existing legacy methods as fallbacks or pick the smallest ordering change that maintainers can review; do not conflate `acceptedScopes` with granted token scopes unless you expose it separately or document the diagnostic semantics. Add colocated tests for the fallback chain in `extensions/slack/src/scopes.test.ts`, including successful `auth.scopes`, fallback to `apps.permissions.info`, fallback to `auth.test` after two `unknown_method` responses, joined errors, and thrown-error handling. Keep the lazy diagnostics seam in `extensions/slack/src/channel.ts` intact and update `extensions/slack/src/channel.lazy-seams.test.ts` or `src/commands/channels/capabilities.test.ts` only if behavior changes require it. Treat PR #44692 as useful prior art, but adapt paths to the current `extensions/slack` layout. Add a single-line `CHANGELOG.md` Unreleased `### Fixes` entry for the user-facing Slack capabilities fix, with appropriate thanks such as `Thanks @Qquanwei` and credit for prior context if reused. Do not change Slack onboarding defaults, token storage, core channel APIs, package versions, or unrelated Slack send/error behavior.
