## What This Changes

The issue asks OpenClaw to make Slack channel capability scope discovery work for modern Slack bot tokens instead of reporting `auth.scopes` and `apps.permissions.info` `unknown_method` errors.

## Best Possible Solution

Keep the behavior inside the Slack plugin and update the scope-discovery helper so modern granular bot tokens and legacy workspace-app tokens both produce granted bot scopes. The likely path is to use `auth.test` response metadata from `x-oauth-scopes` while preserving the existing capability diagnostics contract and lazy loading boundary.

## Repair Work Prompt

Fix the Slack capabilities scope lookup for modern granular Slack bot tokens. Current `extensions/slack/src/scopes.ts` only tries `auth.scopes` and `apps.permissions.info`, so `openclaw channels capabilities` reports `unknown_method | unknown_method` for standard `xoxb-...` manifest-created apps. Keep the fix owned by the Slack plugin; do not change core channel APIs, Slack auth setup, plugin loader contracts, dependency versions, or generated/baseline files. Prefer a small helper update that obtains granted scopes from `auth.test` / Slack SDK `response_metadata.scopes` (`x-oauth-scopes`) while preserving compatibility with the existing legacy methods. Do not treat `response_metadata.acceptedScopes` as granted bot scopes unless tests clearly document why. Add or update focused tests around the scope helper, Slack capability diagnostics lazy seam, and CLI rendering. Add a user-facing changelog fix entry in the active version, crediting `Thanks @Qquanwei` and, if using the reproduction analysis, `@martingarramon`; never credit bots. Related prior art: PR #44692 was closed unmerged and should be used only as context.
