## Summary

Keep this open. Current main still discovers Slack scopes only through the two legacy methods that the report says return `unknown_method` for modern granular bot tokens, while the Slack SDK in use can expose OAuth scopes from `auth.test` response metadata. The prior fallback PR is closed unmerged, so there is no shipped or current-main fix to close against.

## What This Changes

Slack capabilities should report normal bot/user scopes for modern Slack bot tokens instead of failing with joined `unknown_method` errors from legacy scope APIs.

## Best Possible Solution

The Slack plugin should discover granted scopes for modern granular bot tokens and legacy workspace-app tokens through one tested helper, returning normal capabilities diagnostics instead of `unknown_method` for standard Slack bot installs.

## Repair Work Prompt

Fix the Slack capabilities scope-discovery bug from https://github.com/openclaw/openclaw/issues/44625. Current `extensions/slack/src/scopes.ts` only tries `auth.scopes` and `apps.permissions.info`, which can both return `unknown_method` for modern granular Slack bot tokens. Add a modern-token path using `client.auth.test()` and the pinned `@slack/web-api` response shape, especially `response_metadata.scopes` and `response_metadata.acceptedScopes` populated from OAuth scope headers. Prefer the conservative behavior of preserving the existing two legacy attempts and adding `auth.test` as a fallback unless the code/tests show a stronger reason to make `auth.test` primary. Keep the change inside the Slack plugin; do not add core owner IDs, startup migrations, dependency bumps, or unrelated Slack behavior changes. Add focused coverage in a new colocated `extensions/slack/src/scopes.test.ts` for each success source, fallback after `unknown_method`, response metadata extraction, dedup/sort behavior, and all-method failure diagnostics. Update any capabilities seam tests only if the public diagnostic source text changes. Add a single-line changelog fix entry if the repo’s active changelog section expects user-facing bug fixes, crediting @Qquanwei and @martingarramon, and credit @dsantoreis if reusing material from PR #44692.
