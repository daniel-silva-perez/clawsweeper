## Summary

Keep open. Current main still only tries `auth.scopes` and `apps.permissions.info` for Slack scope discovery, so the reported `unknown_method | unknown_method` path remains possible for modern `xoxb-*` Slack bot tokens; the related source PR was closed without merge.

## What This Changes

The issue reports that `openclaw channels capabilities` fails to query Slack bot scopes for a configured Slack channel, returning `unknown_method` errors from the current scope discovery methods.

## Best Possible Solution

Fix this inside the Slack plugin by adding a modern bot-token scope discovery path, ideally through `auth.test` metadata or accessible Web API scope headers, while preserving legacy workspace-app fallbacks and adding focused tests for extraction and capability diagnostics.

## Reproduction Assessment

Yes. The issue has concrete CLI steps, a contributor reproduced it on upstream main, and current source inspection still shows the exact two-method failure path that produces the joined `unknown_method` diagnostic.

## Solution Assessment

Yes, a Slack-plugin fix is the right boundary. No core channel API change is needed; maintainers only need to choose whether `auth.test` is a third fallback or the primary path for modern bot tokens.
