## Summary

Keep this issue open. Current main has ACP thread binding, output streaming, `/acp steer`, and native sub-agent steering controls, but ACP inbound chat still enters the same serialized `runTurn` path, and explicit native sub-agent steering remains a restart flow rather than a documented live same-turn input path.

## What This Changes

The issue requests live inbound chat steering so Discord and other chat-surface messages can reach running ACP and sub-agent sessions before the active turn finishes.

## Best Possible Solution

Define a documented active-turn input capability in the ACP runtime and manager contract, route bound chat replies to it while a turn is running only when the backend advertises support, preserve queued-turn fallback otherwise, and make native sub-agent live-input versus restart behavior explicit in docs and tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to start one ACP `runTurn` and submit a second bound reply or `/acp steer` for the same session; current manager coverage shows the second turn is serialized until the first finishes, and native sub-agent steering is implemented as a restart.

## Solution Assessment

No. The existing `/acp steer` and bound follow-up routes are useful controls, but they reuse serialized turn dispatch; the safer solution is an explicit backend-aware live-input capability rather than bypassing the actor queue ad hoc.
