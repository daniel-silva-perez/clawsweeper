## Summary

Keep this issue open. Current main still has thread-bound ACP routing and visible output streaming, but follow-up chat messages and `/acp steer` are still modeled as serialized ACP turns rather than out-of-band input to an active turn; native sub-agent steering is restart-based, so the requested live input loop remains unimplemented.

## What This Changes

The issue requests live in-thread input/steering for running ACP and sub-agent sessions from Discord and other chat surfaces.

## Best Possible Solution

Add a real active-turn input capability to the ACP runtime/manager contract outside the per-session actor queue, route bound chat messages to it while an ACP turn is running, and document/test backend capability fallback. For native sub-agents, settle the intended live-vs-restart behavior and make bound thread replies follow that contract consistently.
