## Summary

Keep this issue open. Current main still has ACP thread/session routing and visible output streaming, but inbound chat messages and `/acp steer` are still dispatched as serialized ACP turns through `runTurn`, not as live input to an already-running turn. Native sub-agent steering is also restart-based, so the requested interactive input loop remains unimplemented. The related merged ACP thread-bound work provides the infrastructure, but it does not close this same-turn input gap.

## What This Changes

The issue requests that messages sent in a bound Discord or other chat thread be delivered as live input to an already-running ACP or sub-agent session instead of waiting for the current turn to finish.

## Best Possible Solution

Add an explicit active-turn input capability to the ACP runtime and manager contract outside the serialized per-session actor queue, route bound chat messages to it while a turn is running when the backend advertises support, and document/test fallback behavior for backends that cannot accept live input. Separately, define the native sub-agent contract so bound thread replies and explicit steer commands behave consistently instead of relying on an implicit restart path.
