## Summary

Keep this issue open. Current main has thread-bound ACP routing, output streaming, `/acp steer`, and native sub-agent steering, but the inbound path still goes through serialized turn dispatch or restart semantics rather than delivering user messages into an already-running turn. The related merged ACP thread-bound work and closed ACP control issue provide the session/control infrastructure, but they do not implement the same-turn input loop requested here.

## What This Changes

Request live inbound chat steering so Discord and other bound chat-surface messages can reach running ACP and sub-agent sessions before the active turn finishes.

## Best Possible Solution

Add a documented active-turn input capability to the ACP runtime and manager contract, route bound chat replies to that capability while a turn is running when the backend advertises support, and keep queued-turn fallback for unsupported backends. Separately define the native sub-agent contract so bound-thread replies and explicit steer commands have intentional live-input or restart behavior with focused tests and docs.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to start one ACP runTurn and send a second bound reply or `/acp steer` to the same session; current manager coverage shows the second runTurn is serialized until the first completes, and sub-agent steering is implemented as a restart.

## Solution Assessment

No current implementation is the best solution yet. The existing `/acp steer` and bound follow-up routes are useful controls, but they reuse the serialized turn path; the maintainable fix needs an explicit runtime capability and backend-aware fallback rather than bypassing the actor queue ad hoc.
