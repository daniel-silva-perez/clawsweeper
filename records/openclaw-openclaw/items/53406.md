## Summary

Keep this issue open. Current main still has ACP thread-bound routing and output streaming, but ordinary bound chat messages become serialized ACP `runTurn` prompt turns, not out-of-band input to an active ACP turn. The ACP runtime contract has no live input method, `/acp steer` uses the same serialized turn path, and native sub-agent steering is still restart-based through the control surface.

## Best Possible Solution

Add a real ACP active-turn input capability outside the per-session actor queue, expose it through the ACP runtime/manager contract with backend capability and fallback semantics, and route bound chat/thread messages to that path while a turn is active. Add regression coverage proving a second bound Discord/chat message reaches the active ACP runtime before the first `runTurn` completes. For native sub-agents, decide whether bound replies should live-steer, restart-steer, or remain queued/restart behavior, then document and test that contract.

## Risks / Open Questions

- ACP backends may not all support live same-turn input, so the eventual runtime contract needs explicit capability and fallback semantics.
- Native OpenClaw embedded sessions have queue-mode steering machinery, but that does not close the ACP gap and does not settle the desired behavior for ordinary bound sub-agent thread replies.
