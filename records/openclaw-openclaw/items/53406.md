## Summary

Keep this issue open. Current main has ACP thread-bound routing, output streaming, and explicit ACP controls, but it still lacks an out-of-band active-turn input path: bound ACP chat messages become serialized `runTurn` prompt turns, and `/acp steer` uses the same serialized manager path. Native sub-agent control also does not settle the requested chat-surface behavior because explicit `/subagents steer` is implemented as a restart, while queue-mode steering applies to embedded OpenClaw runs rather than ACP runtimes.

## Best Possible Solution

Add a real ACP active-turn input capability outside the per-session actor queue, expose it through the ACP runtime/manager contract with backend capability and fallback semantics, and route bound chat/thread messages to that path while a turn is active. Add regression coverage proving a second bound Discord/chat message reaches the active ACP runtime before the first `runTurn` completes. For native sub-agents, define and document whether bound replies should live-steer, restart-steer, or remain queued, then cover that contract in focused tests.

## Risks / Open Questions

- ACP backends may not all support live same-turn input, so the eventual ACP runtime API needs explicit capability discovery and fallback semantics.
- Native sub-agent bound-reply behavior needs a maintainer product decision: live steer at the next model boundary, restart-steer, queued follow-up, or a documented split between ordinary replies and explicit control commands.
