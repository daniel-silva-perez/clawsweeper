## Summary

Keep this open. Current main has partial lower-level Google Vertex transport support for an option-level `location`, but the per-model config path still does not expose or forward a model-specific Vertex location, so configured google-vertex models still cannot reliably mix global preview endpoints and regional GA endpoints under one provider setup.

## What This Changes

The issue requests per-model Google Vertex AI location configuration so preview models can use global endpoints while GA models use regional endpoints under the same credentials.

## Best Possible Solution

Implement this as Google-provider-owned Vertex endpoint selection: document a per-model google-vertex `params.location` override, preserve environment fallback for existing setups, feed the resolved value into the custom Vertex transport options, and add tests proving preview/global and GA/regional models can coexist under the same project and ADC credentials.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to configure two `google-vertex/<model>` entries with different desired locations: a top-level `location` is rejected by the generated schema, while `params.location` is currently accepted as config but never forwarded into `resolveGoogleVertexLocation`, so the request still falls back to `GOOGLE_CLOUD_LOCATION`.

## Solution Assessment

No current solution is in place. The issue's goal is valid, but the narrowest maintainable fix is provider-owned mapping from per-model Google Vertex params into transport options rather than adding a provider-specific `location` field to the generic agent model-entry schema.

Prefer a documented per-model `agents.defaults.models["google-vertex/<model>"].params.location` contract over adding a generic top-level `location` field to `AgentModelEntryConfig` unless a maintainer explicitly asks for the broader schema change.
