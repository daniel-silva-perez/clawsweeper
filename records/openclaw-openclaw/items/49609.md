## Summary

Current main still uses the reported x11vnc + Xvfb noVNC stack, so the reported non-Latin-1 clipboard path has not been fixed. The linked TigerVNC PR was closed unmerged, and no newer canonical issue or merged PR appears to supersede this report.

## Best Possible Solution

Keep this issue open. The likely fix belongs in the sandbox-browser image and entrypoint: replace or augment the current x11vnc path with a VNC server that supports Extended Clipboard/UTF-8, preserve loopback-only and authenticated observer behavior, add any required TigerVNC tooling packages, bump the sandbox browser security hash if existing containers need recreation, and verify with a Docker/noVNC manual test that CJK/Cyrillic clipboard text round-trips correctly.
