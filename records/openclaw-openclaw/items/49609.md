## Summary

Current main still uses the reported sandbox-browser x11vnc + Xvfb + noVNC path, with no TigerVNC/Xtigervnc replacement or other UTF-8 clipboard handling in the repository. The related TigerVNC PR is closed unmerged, so this remains an actionable sandbox-browser infrastructure bug rather than a cleanup close.

## Best Possible Solution

Keep this issue open. The best fix is a sandbox-browser image and entrypoint change that replaces or augments x11vnc with a VNC server that negotiates Extended Clipboard/UTF-8, preserves OpenClaw's existing noVNC password/token and loopback security properties, bumps SANDBOX_BROWSER_SECURITY_HASH_EPOCH so existing containers are recreated, updates docs/tests as needed, and verifies CJK/Cyrillic clipboard round-trip through noVNC in a Docker smoke test.
