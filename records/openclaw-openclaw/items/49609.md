## Summary

Current main still uses the reported x11vnc + Xvfb + noVNC stack, and the related TigerVNC PR is closed unmerged. The issue remains actionable in the sandbox-browser image/entrypoint path.

## Best Possible Solution

Keep this issue open. The best fix is a sandbox-browser image and entrypoint change that replaces or augments x11vnc with a VNC server that supports Extended Clipboard/UTF-8, preserves loopback-only publishing and noVNC password/token behavior, updates the sandbox-browser security hash if existing containers must be recreated, and verifies CJK/Cyrillic clipboard round-trip through noVNC in a Docker smoke test.
