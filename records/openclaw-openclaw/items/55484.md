## Summary

Keep open. Current main still lacks the requested non-thread ACP persistent-affinity contract: ACP spawns expose only `run` and thread-bound `session`, non-thread ACP spawns still get fresh random session keys, and existing persistent ACP bindings are conversation-config bindings rather than cron/orchestrator affinity reuse.

## What This Changes

Add a non-thread persistent ACP affinity mode for `sessions_spawn` so cron, hook, and orchestrator callers can reuse deterministic ACP sessions via an `affinityKey`.

## Best Possible Solution

Keep this as an upstream design tracker and implement the contract in the ACP spawn/tool owner path after maintainer approval: add explicit `mode: "persistent"` plus `affinityKey` for trusted non-thread callers, derive deterministic keys from requester owner identity, backend, cwd, and affinity key, preserve existing affinity sessions on later dispatch failure, and document/test the authorization and cleanup boundaries.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence reproduction path: `sessions_spawn({ runtime: "acp", mode: "persistent", affinityKey: "..." })` has no schema/parser support, `mode: "session"` without `thread=true` returns `thread_required`, and non-thread ACP spawns use random session keys.

## Solution Assessment

Unclear. The downstream implementation shape is plausible, but upstream should first approve requester identity, authorization, key scoping, cleanup, and failed-dispatch behavior before changing core ACP session reuse.
