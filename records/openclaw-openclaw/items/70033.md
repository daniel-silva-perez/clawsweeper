## Summary

Keep open. Current main still leaves the reported Bedrock Converse path exposed: the pinned Bedrock stream parser can turn incomplete or malformed tool-use input into `{}`, while OpenClaw only installs its malformed tool-call argument repair wrapper for Kimi Anthropic-compatible streams and `openai-completions`, not `bedrock-converse-stream`.

## What This Changes

The issue asks OpenClaw to handle large Bedrock-generated `write`/`exec` tool arguments without repeatedly executing empty `{}` arguments and failing validation.

## Best Possible Solution

Add a focused Bedrock Converse regression fixture and provider-safe repair path: enable or adapt the existing malformed tool-call argument repair for `amazon-bedrock`/`bedrock-converse-stream` when it can reconstruct large `write`/`exec` arguments, and emit a clear non-looping diagnostic when a final required-argument tool call remains unrecoverably empty.

## Reproduction Assessment

Partially. The issue gives concrete Bedrock environment details and repeated validation failures, and current source shows a high-confidence empty-arguments failure path, but there is no checked-in captured Bedrock stream fixture for the exact large-content threshold.

## Solution Assessment

Unclear for the full retry/chunking request, but a narrow fixture-backed Bedrock stream repair plus unrecoverable-empty diagnostics is the best maintainable first fix. Automatic chunking of tool arguments would be a broader product/runtime policy change.
