## Summary

Keep this issue open. Current main still always builds and injects inbound channel metadata into the agent run, including a trusted system prompt block and user-role untrusted metadata blocks, and the shared, Telegram, and Discord config surfaces do not expose a disable or minimize toggle for trusted DM conversations. Related PR #41299 is still an open separator/stripping proposal, not the requested metadata-injection config API.

## Best Possible Solution

Keep this issue open for maintainer product/security judgment. The best implementation is likely a documented shared reply-runtime option that defaults metadata injection on, supports narrow per-channel/account or trusted-DM scoping, covers Telegram and Discord, and adds tests for both existing metadata-on behavior and the opt-in minimized/disabled path.

## Risks / Open Questions

- A broad metadata-off switch could weaken prompt provenance, sender verification, copied-metadata stripping, and routing clarity if it is not opt-in and narrowly scoped.
- The current prompt-level workaround does not remove metadata from model context, so trusted personal DM users still pay the metadata-noise cost until product behavior changes.
