## Summary

Keep this open. Current main still always injects inbound channel metadata into the reply run, both as a trusted system prompt block and as user-role untrusted metadata blocks, and the shared/Telegram/Discord config surfaces still do not expose a disable or minimize toggle for trusted DM cases. Related PR #41299 is still a separator/stripper proposal, not a metadata injection toggle or config API.

## Best Possible Solution

Keep this open for maintainer product/security judgment. The best implementation is likely a documented shared reply-runtime option that defaults metadata injection on, supports narrow per-channel/account or trusted-DM scoping, covers Telegram and Discord, and adds tests for both the existing metadata-on behavior and the opt-in minimized/disabled path.

## Risks / Open Questions

- A broad metadata-off switch could weaken prompt provenance, sender verification, copied-metadata stripping, and routing clarity if it is not opt-in and narrowly scoped.
- The current workaround remains prompt-level instruction rather than product behavior, so trusted personal DM users still pay the metadata-noise cost in model context.
