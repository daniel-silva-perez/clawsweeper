## Summary

Keep open. Current `main` still has no `session.resetPrompt` or equivalent config surface, and the model-backed reset startup prompt is still built from the hardcoded `BARE_SESSION_RESET_PROMPT_BASE` in the gateway and soft-reset paths. Plain channel text commands now acknowledge bare `/new` and `/reset` without invoking the model, but that does not cover the requested configurable prompt for model-backed reset/startup turns or the Control UI-style gateway path.

## What This Changes

Add a configurable session reset startup prompt, with global and per-agent override support, while falling back to the existing hardcoded reset prompt when unset.

## Best Possible Solution

Keep this as the canonical narrow feature request for a configurable model-facing reset startup prompt. Add a full replacement string with fallback to the current prompt, support root and per-agent precedence, apply it only to the normal reset prompt base, and update config schema/help/docs/tests/changelog while leaving reset archival and reset-greeting model/suppression policy to separate issues.

## Reproduction Assessment

Yes. Current-main source and regression tests provide a high-confidence reproduction path: model-backed reset runs still use the hardcoded prompt, while config validation has no accepted `resetPrompt` field; the discussion also includes a concrete Ollama/gemma4 tool-call-loop report tied to the prompt wording.

## Solution Assessment

Yes, if maintainers accept the requested config shape. A full replacement string with fallback is the narrowest maintainable fix for the reported prompt wording problem; expanding this into pre-reset backups or the broader reset-greeting model/suppression work would mix separate concerns.
