## Summary

Keep open. Current main partially threads gateway runtime snapshots into health refreshes, so a fresh probe can inherit `running` and `lastStartAt`. The reported mismatch is still actionable because the gateway primes and serves a cached health snapshot before channel startup can populate runtime state, and `openclaw health --json` can return that cached non-probe payload while `status --deep` forces a live probe. The token-source half of the reporter's follow-up also remains: health builds Telegram and Slack summaries from a runtime-only snapshot, while `channels.status` uses the fuller per-account snapshot path that carries credential-source metadata.

## Best Possible Solution

Keep this issue open. The best fix is to make gateway health snapshots use the same runtime-plus-resolved-account snapshot contract as `channels.status`, or refresh/replace the primed health cache after channel startup completes. Add regression coverage that compares `health --json`, `health` with `probe:true`, and `channels.status --probe` for a running configured Telegram account, including `running`, `lastStartAt`, and `tokenSource`; Slack should get a similar passive-token-source check because the issue discussion reports the same class there.
