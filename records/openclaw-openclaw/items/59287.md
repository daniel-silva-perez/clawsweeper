## Summary

Keep open. Current main still has the monitoring-facing mismatch class described here: plain `openclaw health --json` is a non-probe, cache-eligible path, and the health summary still does not use the same resolved per-account snapshot path as `channels.status`. Runtime-backed `running` can be correct after channel startup, but the startup health cache can still be primed before channels populate runtime state, and Telegram/Slack credential-source fields can still default to `none` in health while channel status has the resolved account metadata.

## Best Possible Solution

Keep this issue open. The best fix is to make gateway health use the same runtime-plus-resolved-account snapshot contract as `channels.status`, or to replace/refresh the primed health cache after channel startup completes. Add regression coverage comparing `health --json`, live/probe health, and `channels.status --probe --json` for configured running Telegram, including `running`, `lastStartAt`, and `tokenSource`; add the analogous Slack credential-source coverage because the discussion reports the same class there.
