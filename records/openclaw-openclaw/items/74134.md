## Summary

Keep open. Current main does not contain this bundled file-transfer plugin, the PR has a protected `maintainer` label, and the latest PR head still has blocking security/control-plane issues around raw `node.invoke` access and the undeclared policy config surface.

## What This Changes

The PR adds a bundled `file-transfer` plugin with four file/directory agent tools, matching node-host commands, gateway node-command policy/media redirect changes, audit wording, dependencies, and handler/policy tests.

## Best Possible Solution

Keep the PR open for maintainer/security review; enforce or reject file-transfer commands at the gateway `node.invoke` boundary so every caller follows the same path policy, add the policy config to the typed/schema/docs surface or move it under plugin config, remove inaccurate watch claims, then validate with targeted plugin/gateway tests and the changed gate.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Enforce file-transfer policy at gateway dispatch:** `src/gateway/node-command-policy.ts:55-63`
  - body: Adding these commands to the dangerous allowlist makes them available to direct `node.invoke` callers once `gateway.nodes.allowCommands` includes them. The dedicated plugin tools run `gatekeep()`, but `openclaw nodes invoke` and raw gateway `node.invoke` still reach `file.fetch`, `dir.fetch`, `dir.list`, and `file.write` with only the generic command allowlist, bypassing `gateway.nodes.fileTransfer` path approval and audit. Enforce the policy in the gateway dispatch path or reject raw file-transfer commands there.
  - confidence: 0.9
- **[P2] Declare the file-transfer policy config:** `extensions/file-transfer/src/shared/policy.ts:81-95`
  - body: The plugin reads `gateway.nodes.fileTransfer`, but the PR does not add that key to the strict `gateway.nodes` config schema/types/help. Normal config validation will reject the documented policy block, and allow-always persistence writes an undeclared key, so operators cannot enable the deny-by-default policy through the supported config path. Add the typed schema/docs surface or store this under the plugin's own config.
  - confidence: 0.87
- **[P3] Remove the advertised watch capability:** `extensions/file-transfer/index.ts:53-56`
  - body: The plugin description says it can watch files, but this PR only registers fetch/list/fetch-directory/write tools and no `file_watch` command. Until that command exists, the plugin metadata should not advertise watch support.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff introduces arbitrary node filesystem read/write/archive transfer and still leaves direct gateway/CLI `node.invoke` able to bypass the plugin-level path policy.

Concerns:

- **[high] Raw node.invoke bypasses file-transfer policy:** `src/gateway/node-command-policy.ts:55`
  - body: The PR routes the agent `nodes` tool away from raw file commands, but the gateway `node.invoke` method itself still forwards any allowed command after checking only the generic command allowlist. Once an operator enables the new file commands, direct CLI/API callers can skip the plugin `gatekeep()` path allowlist, approval flow, and audit.
  - confidence: 0.9
- **[medium] Policy config is outside the validated config contract:** `extensions/file-transfer/src/shared/policy.ts:91`
  - body: The security model depends on `gateway.nodes.fileTransfer`, but the strict config schema at PR head does not accept that key. This makes the intended path policy hard to configure through supported config flows and can push users toward the weaker `allowCommands` control alone.
  - confidence: 0.84
