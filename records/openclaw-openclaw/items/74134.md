## Summary

Keep this PR open. Current main does not include the bundled file-transfer plugin or the generic plugin node-invoke policy seam, and this PR has the protected `maintainer` label plus a security-sensitive filesystem transfer surface. The latest PR head still appears to accept `ask: "always"` canonical-path policy results as if they were allowlist matches, leaving a symlink target escape for `followSymlinks=true`.

## What This Changes

The PR adds a bundled file-transfer plugin with file and directory fetch/list/write tools, dangerous node-host commands, a gateway plugin node-invoke policy seam, media-store hardening, SDK exports, docs, generated baselines, and tests.

## Best Possible Solution

Keep the PR open, change canonical-path validation so it requires a real `matched-allow` result or performs a separate approval/deny decision for the canonical target, add regression coverage for `ask: "always"` plus `followSymlinks=true` on both read and write paths, then rerun the targeted file-transfer, gateway policy, SDK contract, and type gates before maintainer sign-off.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Require canonical allowlist matches:** `extensions/file-transfer/src/shared/node-invoke-policy.ts:318-319`
  - body: When `ask: "always"` is configured, `evaluateFilePolicy()` returns `ok` before checking `allowReadPaths` or `allowWritePaths`. This canonical-path check treats that prompt result as an allowlist match, so with `followSymlinks=true` a prompted request for an allowed-looking symlink path can resolve to a disallowed target and still proceed. Require `reason === "matched-allow"` for canonical paths, or prompt/deny using the canonical path before returning or writing bytes; the same pattern below in postflight needs the same treatment.
  - confidence: 0.86

## Security Review
