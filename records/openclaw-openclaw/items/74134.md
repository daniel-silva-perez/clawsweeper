## Summary

Keep open. This PR carries the protected `maintainer` label and introduces a security-sensitive node filesystem surface that current `main` does not already implement. The author addressed several bot findings in later commits, but source inspection still shows blocking policy/config issues and smaller correctness problems before this should merge.

## What This Changes

The PR adds a bundled `file-transfer` plugin with four agent tools and matching node-host commands, plus node-command allowlist updates, agent `nodes` redirect logic, audit wording, package metadata, lockfile entries, and unit tests.

## Best Possible Solution

Keep the PR open for maintainer/security review. The durable shape should enforce or reject file-transfer commands at the gateway `node.invoke` boundary, expose the path policy through the typed/schema/help/docs config contract or a plugin-owned config surface, fix the data-loss and empty-file edge cases, then validate targeted plugin/gateway/config tests plus the changed gate in Testbox.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Enforce file policy at gateway dispatch:** `src/gateway/node-command-policy.ts:59-64`
  - body: Adding these commands to the dangerous allowlist lets any caller that has `node.invoke` access reach `file.fetch`, `dir.list`, `dir.fetch`, and `file.write` once `gateway.nodes.allowCommands` includes them. The new redirect blocks only the agent `nodes` tool path; raw gateway, CLI, and plugin-runtime `node.invoke` still forward after the generic allowlist check, bypassing `gatekeep()`, path approval, and file-transfer audit.
  - confidence: 0.91
- **[P2] Declare the file-transfer config key:** `extensions/file-transfer/src/shared/policy.ts:91-95`
  - body: The plugin reads `gateway.nodes.fileTransfer`, but the strict gateway config schema/types/help still do not declare that key. Operators following the documented policy block can have config validation reject it, and `allow-always` writes an undeclared key, so the deny-by-default policy is not available through the supported config contract.
  - confidence: 0.88
- **[P2] Check the expected hash before writing:** `extensions/file-transfer/src/node-host/file-write.ts:208-214`
  - body: `expectedSha256` is compared only after the temp file has been renamed into place. If it mismatches with `overwrite=true`, the old file has already been replaced and the handler then unlinks the target; compare the expected hash with the decoded buffer before any write instead of doing destructive cleanup after rename.
  - confidence: 0.8
- **[P3] Allow zero-byte file transfers:** `extensions/file-transfer/src/tools/file-write-tool.ts:90-92`
  - body: An empty base64 string is the correct encoding for a zero-byte file, but `file_write` rejects it as missing. The fetch side has the same shape when it treats an empty `base64` response as invalid, so empty files cannot round-trip through the new binary file tools.
  - confidence: 0.79

## Security Review

Status: needs_attention

Summary: Security review needs attention because the patch adds arbitrary node filesystem read/write/archive transfer while policy enforcement remains limited to the dedicated agent tools.

Concerns:

- **[high] Raw node.invoke bypasses file-transfer policy:** `src/gateway/node-command-policy.ts:59`
  - body: The gateway dispatch path still checks only the generic node command allowlist before forwarding the new file commands. Once an operator enables those commands, direct CLI/API/plugin callers can skip the path allowlist, operator approval, and file-transfer audit path.
  - confidence: 0.91
- **[medium] Path policy is outside strict config schema:** `extensions/file-transfer/src/shared/policy.ts:91`
  - body: The security model depends on `gateway.nodes.fileTransfer`, but the PR does not add that key to the strict config schema/types/help. That makes the intended policy hard to configure through supported flows and can leave only the weaker `allowCommands` control in place.
  - confidence: 0.87
- **[medium] Hash mismatch cleanup can delete node data:** `extensions/file-transfer/src/node-host/file-write.ts:208`
  - body: The node write handler performs the expected-hash check after rename and unlinks the target on mismatch. A wrong hash supplied through direct `node.invoke` can therefore destroy an overwritten path instead of failing before I/O.
  - confidence: 0.76
