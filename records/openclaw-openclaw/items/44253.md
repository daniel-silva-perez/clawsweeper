## Summary

Keep open. Current main still does not provide a config-enforced delegate-only/self-deny tool policy for an orchestrator agent. The schema accepts `allow`/`alsoAllow`/`deny` but no `selfDeny` or `delegateOnly`, and subagent filtering still runs after the same agent-level policy, so an agent-level deny continues to remove the tool before a subagent policy can use it. The related #50038 umbrella is closed and #59225 is about schema parameter filtering, not this capability.

## Best Possible Solution

Add a core tool-policy concept such as `selfDeny` or `delegateOnly` with schema/types/docs support. The implementation should remove those tools only from the current agent's model-visible tool list after normal allow/deny resolution, preserve them for eligible subagent policy resolution, keep normal `deny` absolute, and add focused tests proving that an orchestrator cannot directly use `exec`/`write`/`edit` while its spawned worker subagents can.
