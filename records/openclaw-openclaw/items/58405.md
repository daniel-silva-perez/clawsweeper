## Summary

Keep this PR open, but it should not merge as-is. Current main still lacks the requested per-request OpenResponses skill override, while the PR’s implementation appears to turn the request field into a replacement for the selected agent’s configured skill allowlist, which can re-enable skills that an agent config intentionally excluded.

## What This Changes

The branch adds an optional `skills` array to the `/v1/responses` request schema, forwards it through the OpenResponses HTTP handler into agent command options, builds an ephemeral skill snapshot from that override, and documents the new field.

## Best Possible Solution

Land a rebased version only if the request-level `skills` field is a per-request narrowing control that intersects with the selected agent’s effective allowlist and preserves empty-array and omitted-field semantics. If maintainers want true allowlist replacement, it should be treated as an explicit admin-only API contract with docs, scope checks, and tests covering locked-down agents.

## Reproduction Assessment

Yes. Current main can be checked statically because the strict schema omits `skills`; the PR concern can be reproduced with an agent configured as `skills: []` or a narrow allowlist, then sending `/v1/responses` with `skills` naming another eligible workspace skill and observing the override becomes the whole snapshot filter.

## Solution Assessment

No. The additive API direction is reasonable, but replacing the agent filter is not the safest maintainable solution; the safer implementation is to intersect the request override with `resolveAgentSkillsFilter(...)` unless a separate admin-only broadening contract is deliberately designed.
