## Summary

Keep open. Current main does not implement this PR's per-request skills override for `/v1/responses`: the request schema is strict and has no `skills` field, the HTTP handler never reads or forwards `payload.skills`, and agent execution still builds skill snapshots from the configured agent allowlist/session state. The PR is plausible core API work, but it needs maintainer review for the API and security semantics before it should land.

## Best Possible Solution

Keep this PR open for maintainer review. If OpenResponses should support per-request skill control, land a focused implementation with schema, handler, agent-command, docs, and tests for accepted/rejected schema values, streaming and non-streaming forwarding, empty-array disabling, omitted-field default behavior, and non-persistence across session turns. Before merge, define whether request `skills` may widen beyond `agents.list[].skills` or must be intersected with the configured agent allowlist.
