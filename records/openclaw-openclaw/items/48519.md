## Summary

Keep open. Current main has ingress/message provenance, external-content boundary markers, transcript-scoped hook interception, plugin approvals, and a memory-wiki provenance sidecar, but it still does not carry structured provenance through tool results into durable state writes or expose read-time provenance for stored state. Related tracks #48503, #48509, #54508, and #28913 are still open, so this is not implemented or clearly superseded.

## What This Changes

The issue requests end-to-end provenance metadata that survives from input/tool results through durable state writes and future reads of persistent state.

## Best Possible Solution

Keep this open as a core security/plugin-API design item, or explicitly canonicalize it under #54508 while preserving the persistent-state details. The end state should be a generic provenance/taint contract for tool results and durable writes, coordinated with #48503 and #48509, with read-time metadata, policy hooks, docs, and regression tests across memory, file, cron/config, and auto-capture surfaces.

## Reproduction Assessment

Not applicable as a feature/security architecture request. The high-confidence verification path is current-main source and docs inspection of hook types, input provenance, external-content wrapping, and memory persistence surfaces, which shows the requested durable provenance chain is absent.

## Solution Assessment

Yes: keeping this open for maintainer/security design is the best current outcome. A narrow plugin-only workaround or single memory filter would not solve the requested cross-surface provenance contract because the core SDK/runtime still lacks durable write and read-time provenance seams.
