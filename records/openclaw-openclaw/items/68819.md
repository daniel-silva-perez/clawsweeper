## Summary

Keep open. Current main still has the Windows `claude-cli` supervisor spawn gap, and the linked issue was explicitly closed as superseded by this PR. The PR is a useful implementation candidate, but the proposed resolver still diverges from Node's Windows command lookup in command-selection edge cases, so it should not merge as-is.

## What This Changes

This PR adds a Windows command-shim argv resolver for the supervisor child adapter and unit tests covering `.cmd` to `.exe`/`.js` rewrites, PATH lookup, and non-Windows fallthrough.

## Best Possible Solution

Land a corrected core process fix that resolves npm-style Windows `.cmd` shims for supervisor child spawns while matching Node's effective Windows PATH selection and `.com`/`.exe` precedence, then cover the `claude-cli` path with focused unit tests and current-main validation.

## Reproduction Assessment

Yes. The reporter provided Windows 11 steps and logs, and current main still shows `claude-cli` uses bare `claude` while the supervisor child adapter has no current `.cmd` shim-to-target resolver for that command path.

## Solution Assessment

No, not as written. Resolving the shim to an underlying executable or JS entrypoint is the right direction, but the implementation should first align PATH-key selection and extensionless command precedence with Node/libuv before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Select PATH the same way Node will:** `src/process/windows-command.ts:26-28`
  - body: `resolveWindowsPathEnv` walks `Object.entries` in insertion order, but Node sorts env keys on Windows and keeps the first case-insensitive duplicate. With an env like `{ Path: shimDir, PATH: exeDir }`, the resolver can parse a shim from one PATH while the child receives another, leading to wrong-binary execution. Sort and dedupe PATH keys the same way Node does before resolving shims.
  - confidence: 0.88
- **[P2] Honor .com before cmd shim resolution:** `src/process/windows-command.ts:86-87`
  - body: For extensionless Windows spawns, libuv checks `.com` and then `.exe`, and does not resolve `.cmd` without a shell. This branch only checks for `<name>.exe` before selecting `<name>.cmd`, so `tool.com` on PATH can be bypassed in favor of a later shim target. Treat `.com` as a higher-precedence direct executable too.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff avoids shelling through `cmd.exe`, but the new command resolver can diverge from Node's Windows lookup and execute an unintended PATH target.

Concerns:

- **[medium] Duplicate PATH variants can select the wrong binary:** `src/process/windows-command.ts:26`
  - body: Because the resolver uses insertion order while Node sorts duplicate Windows env keys, a crafted or merged env object with both `PATH` and `Path` can make OpenClaw resolve a shim from a different PATH than the child process actually receives.
  - confidence: 0.84
- **[low] .com precedence is not preserved:** `src/process/windows-command.ts:86`
  - body: The extensionless-command branch only checks `.exe` before rewriting to `.cmd`, but Node/libuv tries `.com` before `.exe`; this can bypass a direct `.com` executable and run a shim target instead.
  - confidence: 0.8
