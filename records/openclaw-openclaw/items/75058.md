## Summary

Keep open. The report is fresh and credible, current main still does not show deterministic exact path-literal preservation for compaction output, and open PR #75093 is already the direct implementation candidate with GitHub closing syntax for this issue.

## What This Changes

Context compaction should preserve slash-separated file paths exactly so later tool calls do not receive corrupted paths with inserted spaces.

## Best Possible Solution

Review and land PR #75093, or an equivalent narrow change in the compaction/safeguard path, so path-like literals needed for future tool calls are extracted and appended deterministically with regression coverage proving slash-separated paths survive unchanged.

## Reproduction Assessment

No high-confidence end-to-end reproduction is available from the checkout alone. The issue provides credible raw_params logs and current source review shows a plausible gap, but reproducing the corruption requires the original long transcript or a deterministic MiniMax compaction replay.

## Solution Assessment

Yes, the best direction is a narrow deterministic preservation layer in the compaction/safeguard pipeline rather than additional prompt wording alone. Because PR #75093 is already the direct open fix candidate, the issue should remain open until that PR is reviewed and merged or replaced.
