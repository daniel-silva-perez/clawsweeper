## Summary

Keep open. The report is fresh and includes credible `raw_params.path` evidence, while current main still relies on LLM-generated identifier preservation plus bounded audits and bounded file-operation suffixes rather than deterministically carrying every path literal needed after compaction.

## What This Changes

The issue reports that long-session context compaction rewrites slash-separated file paths into spaced tokens, causing later file-read tool calls to use invalid paths.

## Best Possible Solution

Harden compaction so path-like literals needed for future tool calls are preserved outside normal prose summarization: extract relevant file paths from the full summarization input and file-operation metadata, append them in a deterministic exact-path section or suffix, and add regression coverage proving slash-separated absolute paths survive safeguard compaction unchanged.

## Reproduction Assessment

No, not with high confidence from the checkout alone. The issue has credible `raw_params` logs, but it lacks the transcript/checkpoint or a deterministic MiniMax long-context scenario needed to replay the corruption locally.

## Solution Assessment

Yes for the product direction: exact path literals should survive compaction. The maintainable implementation should be deterministic path preservation in the safeguard, not only stricter prompt wording or another LLM quality retry.
