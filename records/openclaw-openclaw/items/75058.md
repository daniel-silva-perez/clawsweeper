## Summary

Keep open. The reporter's raw tool-call logs are credible, and current main still relies on LLM-facing preservation instructions, a bounded recent-message identifier audit, and bounded file-operation suffixes rather than deterministic exact preservation of every slash-separated path needed after compaction. The discussion also has a focused volunteer pass from samrusani, so this should stay tracked until that hardening lands or is superseded.

## What This Changes

The issue requests that context compaction preserve slash-separated file path literals exactly so later tool calls do not receive paths with inserted spaces.

## Best Possible Solution

Harden the compaction and safeguard pipeline so path-like literals needed for future file operations are extracted from summarization inputs and file-operation metadata, emitted in a deterministic exact-path section or protected suffix, and covered by regression tests proving slash-separated paths remain byte-exact after compaction.

## Reproduction Assessment

No, not with high confidence from the checkout alone. The raw_params logs are credible, but the issue lacks the original transcript/checkpoint or a deterministic MiniMax long-context replay needed to reproduce the corruption end to end.

## Solution Assessment

Yes, the narrow maintainable direction is deterministic path preservation in the compaction pipeline. Prompt wording, identifier-policy defaults, and quality retries are useful but insufficient as the only guard for tool-call path literals.
