## Summary

Keep open. The report is fresh and credible, current main still relies on prompt guidance, quality auditing, and bounded file-operation suffixes rather than deterministic exact path-literal carry-forward, and open PR #75093 already references this issue with closing syntax as the direct fix candidate.

## What This Changes

The issue reports that context compaction can alter slash-separated file paths, causing later tool calls to use paths with inserted spaces and fail to find files.

## Best Possible Solution

Review and land PR #75093, or an equivalent narrow compaction safeguard change, so path-like literals needed for later tool calls are extracted from model-facing compaction input and appended deterministically with regression coverage proving slash-separated paths survive unchanged.

## Reproduction Assessment

No, there is not a high-confidence end-to-end reproduction path from the checkout alone. The raw tool logs are credible and current source review shows a plausible gap, but reproducing the corruption requires the original long transcript or a deterministic MiniMax compaction replay.

## Solution Assessment

Yes, the best direction is a narrow deterministic preservation layer in the compaction/safeguard pipeline, not more prompt wording alone. Because PR #75093 already targets that path with closing syntax, this issue should wait for that PR review rather than be closed or duplicated.
