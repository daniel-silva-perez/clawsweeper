## Summary

Keep open. The report is fresh and credible, current main still relies on LLM-facing identifier instructions plus bounded audits/file-operation suffixes rather than a deterministic exact path-literal carry-forward, and open PR #75093 already references this issue with closing syntax as the direct fix candidate.

## What This Changes

The issue reports that after long-context compaction, slash-separated file paths are rewritten with inserted spaces, causing later file-read tool calls to fail.

## Best Possible Solution

Review and land PR #75093, or an equivalent narrow compaction/safeguard hardening change, so path-like literals needed for future tool calls are extracted and appended deterministically with regression coverage proving slash-separated paths survive unchanged.

## Reproduction Assessment

No high-confidence end-to-end reproduction is available from the checkout alone. The raw_params logs are credible and current source review shows a plausible gap, but reproducing the corruption requires the original long transcript or a deterministic MiniMax compaction replay.

## Solution Assessment

Yes, the best direction is a narrow deterministic preservation layer in the compaction/safeguard pipeline, not more prompt wording alone. Because PR #75093 already targets that path with closing syntax, this issue should wait for that PR review rather than be closed or duplicated.
