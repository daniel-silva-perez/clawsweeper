## Summary

Keep this issue open. Current main still has the reported `/agent` read-guard gap: the workspace-root guard maps only `containerWorkdir`, the sandboxed read-tool wiring passes only that option, and existing coverage still expects `/agent/secret.txt` to fail with `Path escapes sandbox root` when `tools.fs.workspaceOnly` is enabled.

## Best Possible Solution

Keep the issue open. The best fix is to make the sandboxed read workspace-root guard validate through the sandbox mount table, or pass an explicit `/agent` mapping only when the sandbox context has `workspaceAccess: "ro"` and that mount exists. Add regression coverage for both `wrapToolWorkspaceRootGuardWithOptions` and `createOpenClawCodingTools` proving `/agent/...` reads succeed while write/edit/apply_patch remain blocked; treat `/tmp` reads as a separate policy decision.
