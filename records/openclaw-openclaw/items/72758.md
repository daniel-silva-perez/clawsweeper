## Summary

Keep open for normal maintainer review. Current main still routes notify-mode TwiML through stored webhook `Url` handling, while this PR is a narrow implementation that appears to fix the stated Twilio notify path without changing conversation calls; it is not already implemented on main and should be reviewed, validated, and merged or rejected by a maintainer.

## What This Changes

The PR changes the Twilio voice-call provider to send notify-mode inline TwiML through the `Twiml` create-call parameter while keeping conversation calls on webhook `Url`, and adds provider tests for both parameter paths.

## Best Possible Solution

Review and land this PR or an equivalent narrow change that sends direct `Twiml` only for one-shot notify calls while preserving webhook `Url` for conversation/dynamic calls; after acceptance, remove the obsolete notify TwiML storage policy in a separate cleanup if maintainers want the code simplified.

## Reproduction Assessment

Yes. A high-confidence unit reproduction is to call `TwilioProvider.initiateCall` on current main with `inlineTwiml` and a mocked `apiRequest`; current main sends a webhook `Url` and stores TwiML instead of sending a direct `Twiml` parameter.

## Solution Assessment

Yes. The PR is the narrowest maintainable fix for the observable problem because it changes only Twilio outbound create-call parameters for the already-prepared notify TwiML path and keeps dynamic conversation calls on the existing webhook path.
