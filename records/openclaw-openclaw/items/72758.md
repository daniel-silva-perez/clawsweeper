## Summary

Keep open. Current main still stores notify-mode TwiML and sends Twilio a webhook `Url`, while this PR makes the narrow intended change to send prepared notify TwiML through the Twilio `Twiml` create-call parameter and keeps dynamic conversation calls on `Url`; I did not find a blocking correctness or security issue in the proposed diff.

## What This Changes

The PR changes Twilio outbound call creation so notify-mode inline TwiML is sent via the `Twiml` call parameter while conversation calls keep using `Url`, and adds provider tests for both paths.

## Best Possible Solution

Review and land this PR or an equivalent narrow change that sends direct `Twiml` only for one-shot notify calls while preserving webhook `Url` for conversation/dynamic calls; a small follow-up cleanup can remove the now-unused notify TwiML storage policy once the behavior change is accepted.

## Reproduction Assessment

Yes. A high-confidence unit reproduction is to call `TwilioProvider.initiateCall` on current main with `inlineTwiml` and a mocked `apiRequest`; current main sends `Url` and stores TwiML instead of sending a direct `Twiml` parameter.

## Solution Assessment

Yes. The PR is the narrowest maintainable fix for the observable problem because it changes only Twilio outbound create-call params for the already-prepared notify TwiML path and keeps dynamic conversation calls on the existing webhook path.
