## Summary

Keep this issue open. Current main has partial typed-hook timeout infrastructure, but the reported OpenClaw core guardrail is still missing for legacy `before_agent_start`: handlers without an explicit registration `timeoutMs` can still be awaited indefinitely on both the model-resolution and prompt-build paths. The related upstream OpenViking plugin issue appears handled externally, and OpenClaw PR #48741 attempted the right area, but that PR is closed unmerged and current main still lacks a global/default timeout that covers this legacy hook.

## Best Possible Solution

Add a bounded fail-open default timeout for legacy `before_agent_start`, or a documented global plugin hook timeout policy that covers it, while preserving per-registration `timeoutMs` overrides. The fix should log plugin id, hook name, elapsed budget, and run id when available, then continue the run. Add regression tests for hung `before_agent_start` handlers on both model-resolution and prompt-build paths, and align config schema/docs if a global setting is introduced.

## Risks / Open Questions

- External or legacy plugins that register `before_agent_start` without `opts.timeoutMs` can still block agent runs before a provider call is made.
- Operators still have no documented global `plugins.hookTimeoutMs`-style setting to enforce a safe default across third-party plugins.
- Docs now describe generic default hook timeouts, but the actual default map does not cover `before_agent_start`, which can mislead plugin authors and operators.
- Current tests prove timeout behavior for `before_prompt_build`, not the reported legacy hook hang on model-resolution and prompt-build paths.
