## Summary

Current main does not implement the proposed CLI command, so this PR should stay open rather than be closed as implemented or superseded. The main maintainer question is product/API consistency: existing gateway docs define `config.patch` as partial merge-patch and `config.apply` as full-config replacement, while this PR proposes `openclaw config apply` with partial patch semantics.

## What This Changes

The PR adds `openclaw config apply` for JSON5 config patches from files or stdin, including recursive object patching, null deletes, `--replace-path`, tests, and related CLI/channel/install docs.

## Best Possible Solution

If maintainers want this capability, land a reviewed version that keeps the existing validation, dry-run, SecretRef, and atomic write safeguards, but align the command name and docs with the existing `config.patch` versus `config.apply` contract or explicitly document the intentional CLI/RPC distinction.
