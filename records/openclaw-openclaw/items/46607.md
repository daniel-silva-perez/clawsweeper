## Summary

Keep open. Current main still leaves Google/Gemini headless audio `@file` references as text, and this PR is a useful direction, but the patch needs rework before merge because it hooks a stale/fallback wrapper and uses the process cwd instead of the active workspace or sandbox boundary for prompt-controlled file reads.

## What This Changes

The PR adds a Google/Gemini payload rewrite in `src/agents/pi-embedded-runner/extra-params.ts` to convert resolvable local audio `@file` refs into `inlineData` parts, plus regression tests in `src/agents/pi-embedded-runner-extraparams.test.ts`.

## Best Possible Solution

Implement the audio-ref conversion in the active Google provider wrapper or shared transport seam, thread the active workspace/sandbox root into the resolver with realpath boundary checks, reuse shared media MIME detection where possible, and cover valid audio, cwd-versus-workspace, sandbox escape, oversized, unresolved, and common M4A cases.

## Reproduction Assessment

Yes. On current main, a Google/Gemini string user prompt containing `@sample.ogg` is converted by `convertGoogleMessages` into a text part, not an audio `inlineData` part; the PR diff provides focused tests for the intended rewrite but does not cover the active plugin-wrapper path.

## Solution Assessment

No. The requested behavior is valid, but this PR is not the best solution yet because it patches the fallback wrapper, resolves prompt-controlled paths against `process.cwd()`, and duplicates narrow audio sniffing rather than using the active provider and media seams.
