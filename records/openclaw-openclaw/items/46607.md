## Summary

Keep open. The requested Google/Gemini audio @file behavior is still missing on current main and the PR is a useful implementation attempt, but the branch is not merge-ready because it resolves prompt-controlled file reads against process.cwd(), targets a stale Google wrapper location relative to current main, and still has narrow M4A detection.

## What This Changes

The PR adds Google/Gemini payload rewriting and regression tests to convert resolvable local audio @file references in user text parts into Gemini inlineData while leaving missing, absolute, and oversized refs as text.

## Best Possible Solution

Implement the rewrite at the active Google provider wrapper or transport seam, thread the session workspaceDir or sandbox media-access boundary into the resolver, enforce realpath/stat/size checks inside that boundary, reuse shared media MIME detection for supported audio, and cover cwd-versus-workspace, sandbox escape, oversize, unresolved refs, and valid audio refs with focused tests.

## Reproduction Assessment

Yes. A google-gemini-cli user text payload containing @sample.ogg reproduces the gap: current main leaves it as text, while the PR tests demonstrate the intended inlineData conversion.

## Solution Assessment

No. The requested behavior is valid, but this PR is not yet the best fix because it resolves from process.cwd(), patches a stale wrapper location relative to current main, and duplicates narrow MIME sniffing instead of using the shared media path.
