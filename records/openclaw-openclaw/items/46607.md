## Summary

Keep open. Current main still lacks inline Google/Gemini audio @file handling, so the PR remains useful, but the current head is dirty against main and has blocking correctness/security issues around filesystem boundaries and M4A detection that need author or maintainer rework before merge.

## What This Changes

The PR branch adds Google/Gemini headless payload rewriting that converts local audio @file references in user text parts into Gemini inlineData parts and adds regression coverage.

## Best Possible Solution

Rework the conversion on the current Google payload wrapper or transport seam, pass the active workspaceDir/effectiveWorkspace into the resolver, keep rewriting limited to user turns, enforce realpath/stat/size checks, use shared MIME detection for audio including common M4A containers, and add focused coverage for cwd-versus-workspace, sandbox escape, oversize, unresolved refs, and valid audio refs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Use the active workspace for audio refs:** `src/agents/pi-embedded-runner/extra-params.ts:251-256`
  - body: resolveGoogleInlineAudioPath bounds refs to process.cwd(). Current embedded runs pass an effectiveWorkspace into applyExtraParamsToAgent and provider hooks, so daemon/headless runs whose cwd differs from the session workspace can read from the wrong root or miss valid refs. Thread the active workspace through this resolver before inlining.
  - confidence: 0.88
- **[P2] Accept common M4A MP4 brands:** `src/agents/pi-embedded-runner/extra-params.ts:242-244`
  - body: The .m4a sniff only accepts a small audio-specific major-brand set. Common M4A files can use ISO MP4 brands such as isom or mp42, so valid audio remains plain text instead of becoming inlineData. Use shared MIME detection or accept compatible M4A brands before rejecting.
  - confidence: 0.8
