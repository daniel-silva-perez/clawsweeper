## Summary

Keep open. Current main still does not convert Google/Gemini headless text @audio refs into inlineData, so the PR remains useful, but the patch is not merge-ready because the new file-read path uses the process cwd rather than the active run workspace and the M4A sniffing is too narrow for common valid files.

## What This Changes

The PR adds Google/Gemini headless payload rewriting that converts local audio @file references in user text parts into Gemini inlineData parts with role, realpath, size, and MIME checks plus regression tests.

## Best Possible Solution

Adapt the audio-ref conversion to the current Google provider stream-wrapper path using the active workspaceDir as the trust boundary, keep user-turn-only rewriting, retain size and realpath safeguards, use robust/shared MIME detection for audio including common M4A containers, and land focused regression coverage plus user-facing docs/changelog when the behavior ships.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Use the active workspace for audio refs:** `src/agents/pi-embedded-runner/extra-params.ts:251-256`
  - body: `resolveGoogleInlineAudioPath` hard-codes `process.cwd()` as the root. Current embedded runs pass an `effectiveWorkspace` into `applyExtraParamsToAgent`, so daemon/headless runs whose process cwd differs from the session workspace can either miss valid `@clip.ogg` refs or read from the wrong root. Thread the run workspace into this resolver before inlining.
  - confidence: 0.88
- **[P2] Accept common M4A MP4 brands:** `src/agents/pi-embedded-runner/extra-params.ts:242-244`
  - body: The `.m4a` sniff only accepts the listed audio-specific major brands. Common `.m4a` files can use ISO MP4 brands such as `isom` or `mp42`, so valid audio remains plain text instead of becoming `inlineData`. Accept compatible brands or use shared MIME detection before rejecting.
  - confidence: 0.8
