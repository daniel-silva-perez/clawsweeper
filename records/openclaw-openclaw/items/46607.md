## Summary

Keep open. The PR targets a real missing behavior on current main: Google string prompts containing local audio refs still become plain text, not audio `inlineData`. The branch has addressed some earlier review items, including user-role filtering, size guard, and temp cleanup, but it still needs maintainer/author rework because the new file reads are bounded to `process.cwd()` rather than the active workspace/sandbox, and the hook is placed on a fallback wrapper that current Google plugin wrapping can bypass.

## What This Changes

The PR rewrites Google/Gemini headless payload text parts so resolvable local audio `@file` references become Gemini `inlineData` parts, with regression tests for inlined, missing, absolute, and oversized refs.

## Best Possible Solution

Implement the audio-ref conversion in the active Google provider wrapper or transport seam, thread the session workspace/sandbox media boundary into the resolver, reuse shared MIME detection or equivalent supported-audio sniffing, and cover valid audio, cwd-versus-workspace, sandbox escape, oversize, unresolved refs, and M4A brand cases.

## Reproduction Assessment

Yes. A focused static reproduction is available: on current main, a Google/Gemini string user prompt containing `@sample.ogg` is converted by `convertGoogleMessages` into `{ text: "...@sample.ogg..." }`, while the PR's tests show the intended `inlineData` conversion path.

## Solution Assessment

No. The requested behavior is valid, but this PR is not yet the best fix because it uses process cwd as the file boundary, patches a stale/fallback wrapper relative to current plugin wrapping, and duplicates narrow audio sniffing instead of relying on shared media detection.
