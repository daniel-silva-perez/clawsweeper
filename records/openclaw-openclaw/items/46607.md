## Summary

Keep open. Current main still does not implement inline Google/Gemini audio @file inlining for headless payloads, and this PR addresses a real gap from #46601, but the proposed patch should not merge until it uses the active workspace/sandbox boundary and broadens M4A detection.

## What This Changes

The PR adds Google/Gemini payload rewriting that converts resolvable local audio @file references in user text parts into inlineData parts, with regression tests for present, missing, absolute, and oversized refs.

## Best Possible Solution

Implement the rewrite in the Google/Gemini provider-owned wrapper or transport seam, pass the active workspaceDir/effectiveWorkspace into the resolver, enforce realpath/stat/size checks against that boundary, reuse shared MIME detection for supported audio including common M4A containers, and add focused coverage for cwd-versus-workspace, sandbox escape, oversize, unresolved refs, and valid audio refs.

## Reproduction Assessment

Yes. The code path is reproducible by sending a google-gemini-cli user text part containing @sample.ogg: current main keeps it as text, while the PR tests demonstrate the intended inlineData conversion.

## Solution Assessment

No. The requested behavior is valid, but the current PR is not the best implementation because it resolves files from process.cwd() and uses local MIME sniffing instead of the active workspace/sandbox and shared media utilities.

  - body: resolveGoogleInlineAudioPath bounds refs to process.cwd(), but embedded/headless runs already pass effectiveWorkspace into applyExtraParamsToAgent.   - body: The .m4a sniff only accepts a small audio-specific major-brand set. Common M4A files can use ISO MP4 brands such as isom or mp42, so valid audio refs can remain plain text instead of becoming inlineData. Reuse shared MIME detection or accept compatible M4A brands.
  - confidence: 0.8
