## Summary

Keep this PR open. Current main still does not implement `agent_error` or any equivalent pre-broadcast lifecycle-error replacement hook. The PR is focused and plausible core plugin API work, but it needs maintainer review for the hook contract, conversation-access guard, lifecycle timing, docs, and changelog hygiene before merge.

## Best Possible Solution

Keep the PR open for maintainer review as a core plugin API proposal. If accepted, add the hook with an explicit raw/processed/redacted contract, gate it consistently with conversation-access policy, add loader/contract tests for the guard plus lifecycle tests for replacement/failure/timeout behavior, update public plugin docs, fix changelog attribution, and then run the targeted lifecycle and plugin-hook test lanes before merge.

## Risks / Open Questions

- The PR adds `agent_error` to the hook name list but not to `CONVERSATION_HOOK_NAMES`, so non-bundled plugins may be able to observe provider error text without the explicit conversation-access opt-in used for nearby lifecycle/conversation hooks.
- The hook is invoked before `buildTextObservationFields` produces the redacted lifecycle error text, so maintainers should decide whether plugin handlers receive raw, formatted, or redacted error material.
- The PR currently updates the changelog without the required `Thanks @author` attribution and does not update public plugin docs for the new hook contract.
- Even with the new 2s timeout, the hook sits on the lifecycle terminal path; maintainer review should confirm the lifecycle timing and compaction retry behavior are acceptable.
