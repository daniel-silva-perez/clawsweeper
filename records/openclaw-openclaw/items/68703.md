## Summary

Keep open. Current main still has the Discord guild-admin requester-authorization gap, and #68716 is an open, unmerged PR that directly targets this issue; policy says the issue should stay open until that fixing PR lands or is replaced.

## What This Changes

Discord guild-admin mutation actions are reported to execute with bot credentials without requiring a trusted requester identity or checking the sender's Discord guild permissions.

## Best Possible Solution

Keep this issue open and land #68716, or an equivalent Discord-plugin patch, that marks privileged guild-admin actions as trusted-requester-only, threads requesterSenderId into the guild runtime, enforces the corresponding Discord permission bits before bot-credentialed mutations, and adds focused regression tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: dispatch a Discord guild-admin action such as channel-delete on current main, observe the core trusted-requester hook is not declared, then follow the handler into runtime.guild where the bot mutation runs after only the feature gate.

## Solution Assessment

Yes. The narrow maintainable direction is to keep core generic and fix the Discord plugin: declare the privileged guild-admin actions as requiring trusted requester identity, forward that identity, and enforce Discord permission checks in the Discord runtime before mutation.
