## Summary

Current main still lacks a modifying plugin hook between assistant reply text generation and TTS directive parsing. Existing hooks either observe output, claim dispatch, or run during delivery after TTS is already applied, so the requested per-message voice-routing seam remains valid.

## Best Possible Solution

Keep this issue open and implement an additive, documented modifying plugin hook that runs after reply text is available and before TTS directive parsing on default reply paths. The change should update SDK hook types, runner wiring, dispatch integration, docs, and targeted tests, while preserving existing static TTS config precedence, directive policy, and downstream `message_sending` behavior.

## Clownfish Work Prompt

Implement an additive modifying plugin hook for per-message TTS routing from this issue. Add a public hook, likely `before_tts`, that runs after assistant reply text is available and before `maybeApplyTtsToReplyPayload` calls `parseTtsDirectives` on default reply paths.

Expected boundary:
- Update hook types/registry/runner in `src/plugins/hook-types.ts` and `src/plugins/hooks.ts` with an event that exposes the reply text plus useful routing context such as session key, channel, account id, recipient/conversation id when available, and agent id if already available in the dispatch context.
- Support a modifying result with `text?: string` and a TTS skip decision, preserving no-op behavior when no hook is registered or no result is returned.
- Wire the hook in `src/auto-reply/reply/dispatch-from-config.ts` before TTS application for final replies and any other default reply paths that currently call `maybeApplyTtsToReplyPayload` for tool/block/synthetic final audio, without changing `message_sending` downstream semantics.
- If `skip` is returned, bypass TTS for that message but still normalize and deliver the original or transformed visible payload as appropriate.
- Preserve existing static TTS config precedence, `modelOverrides` policy, directive stripping, provider fallback behavior, and custom OpenAI-compatible voice handling.
- Treat final assistant text as conversation-sensitive if the existing plugin policy requires it; document any required `allowConversationAccess` behavior for non-bundled plugins.
- Do not add extension-specific voice policy, bundled plugin ids, provider-specific routing defaults, or a startup/load-time migration.
- Update public docs in `docs/plugins/hooks.md` and `docs/tools/tts.md`; add a changelog entry if this repo treats new plugin hooks as user-facing, crediting `Thanks @IGI30` and not crediting automation.

Add focused tests for hook runner merge/skip semantics and dispatch behavior: injected `[[tts:voice=...]]` reaches directive parsing before audio generation, `skip` bypasses TTS, no-hook behavior remains unchanged, and downstream `message_sending` remains after TTS.
