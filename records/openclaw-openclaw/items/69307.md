## Summary

Keep open. Current main still lacks a modifying hook between assistant reply generation and TTS directive parsing. The existing plugin hooks either observe output without modification, run before/around dispatch, or run at delivery after TTS has already been applied; the requested per-message voice-routing seam is a real missing plugin API rather than a ClawHub-only plugin request.

## Best Possible Solution

Keep this issue open and implement an additive, documented modifying plugin hook that runs after reply text is available and before TTS directive parsing for the default reply paths. The hook should preserve existing TTS config behavior, support dynamic text/directive injection and a clear skip-TTS result, and include SDK types, hook runner wiring, docs, and targeted tests.

## Clownfish Work Prompt

Implement an additive `before_tts` modifying plugin hook for OpenClaw. The observable bug/feature is that plugins cannot dynamically inject `[[tts:voice=...]]` or skip TTS after the assistant reply is generated but before `parseTtsDirectives()`/`maybeApplyTtsToPayload()` runs. Scope the fix to public plugin hook plumbing and the auto-reply TTS dispatch paths; do not change provider-specific TTS behavior, directive syntax, or static TTS config precedence.

Expected implementation boundary:
- Add typed hook name/result/event support in `src/plugins/hook-types.ts` and a runner method in `src/plugins/hooks.ts` using the existing sequential modifying hook pattern.
- Treat the hook as raw conversation/reply content access for non-bundled plugins; update `CONVERSATION_HOOK_NAMES`, config help/docs, and loader tests as needed so trusted external plugins must opt in explicitly.
- Invoke the hook before `maybeApplyTtsToReplyPayload()` for assistant reply payloads in `src/auto-reply/reply/dispatch-from-config.ts`, including final replies, block-streaming TTS, and the synthetic final TTS-from-blocks path. Preserve visible-text cleanup and normal delivery when `{ skip: true }` suppresses TTS.
- Event fields should include text, sessionKey when available, channel/ttsChannel, destination/conversation id when available, accountId, agentId, and kind (`final`/`block`/`tool` only if the implementation intentionally covers that path). Result should allow `text?: string` and `skip?: boolean`; text changes should feed TTS directive parsing and visible-text cleanup exactly once.
- Add targeted unit coverage for hook wiring/merge behavior and dispatch integration showing a hook-injected `[[tts:voice=...]]` reaches `maybeApplyTtsToPayload`, and `{ skip: true }` bypasses TTS without cancelling normal text delivery.
- Update public docs in `docs/plugins/hooks.md` and TTS docs if they mention model-driven directives as the only dynamic per-message override path. Changelog is expected because this is a user-facing plugin API addition; credit the issue author, not Codex/OpenClaw.

## Risks / Open Questions

- Adding this hook creates a new public plugin contract; naming, event fields, result merge semantics, and docs/tests need to be locked down rather than implemented as an internal-only shortcut.
- The hook exposes and can modify assistant reply text before TTS, so non-bundled plugin access should be treated like other raw conversation hooks and gated through allowConversationAccess or an equally explicit policy.
- The TTS path has final, block-streaming, tool-progress, and synthetic TTS-only reply cases; a partial implementation could still miss the user's per-message routing path or leak directives into visible text.
