## Summary

Keep open. Current main still prunes `--fix-missing` entries from a single `resolveSessionFilePath()` result, which prefers the persisted `sessionFile`; if that stale path is missing, cleanup deletes the store entry without checking the existing `<sessionId>.jsonl` fallback that normal transcript readers already use. The related PR #72167 was closed unmerged, so there is not yet a landed implementation candidate to close against.

## Best Possible Solution

Keep this issue open until PR #72167 or an equivalent fix lands. The fix should make `pruneMissingTranscriptEntries()` use the same transcript-candidate semantics as normal readers, preserve explicit custom transcript paths, and add focused cleanup tests for stale `sessionFile` with a valid `<sessionId>.jsonl` fallback plus the both-missing case.
