## Summary

Keep this issue open. Current main still makes `sessions cleanup --fix-missing` prune from a single `resolveSessionFilePath()` result, which can prefer a stale persisted `sessionFile` and miss the existing `<sessionId>.jsonl` fallback that normal transcript readers already use. The linked implementation candidate #72167 is closed and unmerged, so there is no landed fix to close against.

## Best Possible Solution

Keep this issue open until #72167 or an equivalent patch lands. The fix should make `pruneMissingTranscriptEntries()` use the same transcript-candidate semantics as normal readers, preserve explicit custom transcript paths, and add focused cleanup tests for stale generated `sessionFile` plus valid `<sessionId>.jsonl`, valid custom `sessionFile`, and both-missing cases.
