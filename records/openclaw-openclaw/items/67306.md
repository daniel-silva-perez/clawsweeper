## Summary

Keep this issue open. Current main has since improved the related raw-inference behavior for `infer model run`, but the requested JSON Schema input contract, output validation, repair/retry semantics, and structured schema-failure reporting are still absent from the CLI and Gateway agent surfaces.

## What This Changes

Add first-class JSON Schema support to at least one core inference CLI surface, ideally `openclaw infer model run` and later `openclaw agent`, with validation, explicit repair behavior, and machine-readable failures.

## Best Possible Solution

Implement a narrow provider-neutral JSON Schema contract on `openclaw infer model run` first: accept a schema from a file or inline JSON, validate the final JSON output with shared structured errors, define bounded repair or no-repair semantics clearly, preserve local/gateway consistency, update docs/tests, and leave broader `openclaw agent` exposure as an explicit follow-up decision.

## Repair Work Prompt

Implement the first provider-neutral JSON Schema support slice for this issue. Scope it narrowly to `openclaw infer model run` unless a small additive Gateway field is required for local/gateway parity. Add `--json-schema <path-or-inline>` to the capability metadata and command parser. Parse the value as either a readable file path or inline JSON object, validate that it is an object schema, run the model as today, parse the final text output as JSON, validate it with the existing shared AJV/schema helper or a small adjacent adapter, and return machine-readable validation/parse failure data in `--json` mode. Define repair behavior explicitly: either implement a bounded retry/repair loop with a documented attempt count, or document and test no automatic repair for the first slice while preserving a clear repair-exhausted error shape if retries are added. Keep provider abstraction neutral; do not add provider-specific native schema modes in this PR. Preserve existing raw `model.run` behavior from #69030. Update `docs/cli/infer.md` and targeted tests in `src/cli/capability-cli.test.ts`; update `src/gateway/protocol/schema/agent.ts` and server-method tests only if the schema contract must cross the Gateway request boundary. Add a user-facing changelog entry with proper attribution if this becomes a shipped feature.
