## Summary

Keep open. Current main still lacks spoken Discord voice model escalation; it only supports a static Discord voice model override and the normal Discord model picker. The PR contains useful source work, but it is stale against the current bundled Discord plugin layout, bundles a gateway registry fix already shipped via #47902, and still has concrete voice-command correctness and privacy/policy issues.

## What This Changes

The PR adds hardcoded spoken Discord voice model escalation/de-escalation triggers to the legacy Discord voice manager and adds a gateway plugin HTTP registry bridge with regression tests.

## Best Possible Solution

Keep this as a feature candidate only if maintainers want spoken model switching. The clean implementation should be a fresh, narrow Discord plugin change under extensions/discord/src/voice that uses explicit low-false-positive commands, consumes command-only utterances, persists first-turn overrides correctly, honors configured or allowed model references instead of hardcoded policy-sensitive defaults where possible, and adds focused voice tests plus docs. The gateway registry changes should be dropped because #47902 already solved that path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Require explicit voice mode commands:** `src/discord/voice/manager.ts:84`
  - body: The medium-tier pattern matches a common phrase inside normal speech, so a user saying something like "think about this approach" can silently downgrade the session model. Spoken model switches should require an explicit command phrase such as a mode name or direct model command.
  - confidence: 0.91
- **[P2] Consume matched voice commands before agent dispatch:** `src/discord/voice/manager.ts:692-693`
  - body: After applying an escalation trigger, the code falls through and sends the literal command text to agentCommandFromIngress. Pure commands such as "go deep" or "quick mode" will both change the model and trigger an unnecessary, confusing agent reply.
  - confidence: 0.9
- **[P2] Persist overrides for missing session entries:** `src/discord/voice/manager.ts:783-785`
  - body: The session-store update returns when the current session entry is absent, so the first spoken escalation after joining a fresh voice session does not persist. The command should create or otherwise initialize the session entry before applying the override.
  - confidence: 0.84
- **[P3] Allow clearing the gateway registry without casts:** `src/plugins/runtime.ts:58-59`
  - body: The new getter returns null, and the tests clear the global with null as never, but the setter only accepts PluginRegistry. Widening the setter to accept null would avoid unsafe casts and make teardown match the exported contract.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff has no CI, dependency, lockfile, or external code-execution changes, but it does introduce persistent hardcoded cloud model switching from transcribed speech.

Concerns:

- **[medium] Avoid hardcoded cloud routing from incidental speech:** `src/discord/voice/manager.ts:79`
  - body: A matched voice transcript can persistently switch future turns to hardcoded Anthropic cloud models, and one trigger is broad enough to fire during ordinary conversation. That can change privacy, cost, and provider policy behavior without an explicit configured allowed-model decision.
  - confidence: 0.88
