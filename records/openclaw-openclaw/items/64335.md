## Summary

Keep this PR open. Current main does not implement the PR's central behavior for gateway runs: Z.AI still resolves a single env auth candidate through `ZAI_API_KEY`/`Z_AI_API_KEY`, while the runtime auth-profile rotation path only rotates over auth profiles already present in the store/order. The PR appears to address a real gap between the current docs' generic multi-key rotation language and the Z.AI gateway implementation, so it is not obsolete on main.

## Best Possible Solution

Review this PR as an active implementation candidate. The best path is to keep the Z.AI-owned env discovery in `extensions/zai`, keep the core changes generic where they are required for runtime-only API-key overlays and auth-profile ordering, and require targeted tests proving multi-key Z.AI gateway failover, single live override behavior, SecretRef preservation, and non-persistence of runtime-only API-key overlays.

## Risks / Open Questions

- Security review still matters before merge because the PR changes credential flow, secret env-var metadata, auth-profile persistence filtering, and non-interactive setup behavior. The main risk to verify is that runtime-only env API keys and SecretRef-backed keys cannot be written back to `auth-profiles.json` as plaintext or deleted from persisted profiles during cooldown/usage saves.
- The PR is large for the stated bug and touches docs, generated SDK baselines, core auth overlay contracts, provider env-var discovery, onboarding, and Z.AI plugin code. Maintainers should confirm each non-Z.AI core seam is necessary and covered by focused tests before landing.
- Current main has generic docs for multi-key env rotation, but gateway LLM execution does not appear to use the generic `collectProviderApiKeys` helper for auth-profile failover. That mismatch should be resolved by either this PR or a narrower follow-up, not by closing the PR as already implemented.
