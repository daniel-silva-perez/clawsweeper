## Summary

Keep this PR open. Current main still gives command-palette hover and keyboard-active rows the same background, so the PR addresses a real UI ambiguity, but the proposed split needs one CSS specificity fix before merge because `:hover` can still override the active styling when both states apply to the same item.

## What This Changes

This PR changes `ui/src/styles/components.css` to split command palette hover and active backgrounds into separate theme-aware `color-mix` rules.

## Best Possible Solution

Update this PR or land an equivalent one-file CSS change so the keyboard-active style wins over hover when both apply, while keeping distinct hover and active backgrounds across default and light theme modes.

## Reproduction Assessment

Yes. Current main can be verified from source: the CSS groups hover and active under the same background, and the renderer applies active state from keyboard navigation while pointer hover can still be present. The proposed patch's remaining issue is reproducible whenever a row is both hovered and `.cmd-palette__item--active`.

## Solution Assessment

No, not quite. The PR is the right narrow surface, but the selector specificity should be adjusted with an active-hover rule or matched specificity so the active styling remains visible when it overlaps hover.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Make the active rule win over hover:** `ui/src/styles/components.css:3536-3541`
  - body: The new hover selector is more specific than `.cmd-palette__item--active`, and the light-mode hover selector is also more specific than the light active rule. When the keyboard-active row is under the pointer, it will render with the hover background instead of the selected background, so the selected state can still disappear in the mixed mouse/keyboard case this PR is meant to fix.
