## Summary

Keep this PR open: current main still has the original command-palette hover/active ambiguity, but the proposed CSS is not merge-ready because the new hover selectors outrank the active selectors when both states apply. The diff is CSS-only and did not raise a security or supply-chain concern.

## What This Changes

The PR modifies `ui/src/styles/components.css` to split command palette item hover and active backgrounds into separate theme-aware `color-mix` rules for dark and light modes.

## Best Possible Solution

Land a one-file CSS update that keeps command palette hover and keyboard-active rows visually distinct in both themes and defines an intentional active-plus-hover state, without changing palette logic, layout, icons, or other UI surfaces.

## Reproduction Assessment

Yes. Current main reproduces the original issue from source because hover and active share `var(--bg-hover)`, and the PR’s remaining defect reproduces by applying both `:hover` and `.cmd-palette__item--active` to the same row and following CSS specificity.

## Solution Assessment

No, not as written. The CSS-only boundary is the right maintainable approach, but the patch needs an active-hover rule or matched specificity so keyboard selection is not hidden by pointer hover.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Make active styling win over hover:** `ui/src/styles/components.css:3536-3549`
  - body: The added hover selectors are more specific than the active selectors, so an item that is both `:hover` and `.cmd-palette__item--active` renders the hover color instead of the selected color. During mixed pointer/keyboard use this can hide the keyboard-selected state under the pointer; add an active-hover rule or match the active selector specificity.
  - confidence: 0.87
