## Summary

Keep this PR open. Current main and the latest release still use one shared command palette hover/active background, so the underlying UI ambiguity remains, but the proposed CSS has a selector-precedence bug that should be fixed before merge.

## What This Changes

The branch modifies `ui/src/styles/components.css` to split command palette hover and active backgrounds into separate theme-aware `color-mix` rules.

## Best Possible Solution

Land a one-file CSS update that keeps command palette hover and keyboard-active rows visually distinct in dark and light themes, and makes the selected state intentional when the selected row is also hovered, without changing palette logic or layout.

## Reproduction Assessment

Yes. Current main reproduces the original issue from source because hover and active share the same background rule; the PR’s remaining defect reproduces by applying both `:hover` and `.cmd-palette__item--active` to the same row and following CSS specificity.

## Solution Assessment

No, not as written. The CSS-only boundary is the right maintainable approach, but it needs an active-hover rule or matched specificity so hover does not accidentally hide the selected-state styling.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Make active styling win over hover:** `ui/src/styles/components.css:3536-3549`
  - body: The added hover selectors are more specific than the active selectors, so a row that is both hovered and `.cmd-palette__item--active` renders the hover color instead of the selected color. During mixed pointer/keyboard use this can hide the keyboard-selected state; add an active-hover rule or match specificity.
  - confidence: 0.87
