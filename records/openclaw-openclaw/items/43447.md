## Summary

Keep open. Current main still has the reported classifier/UI precedence problem: HTTP 429 becomes `rate_limit` before billing evidence can win, mixed rate-limit/balance text checks rate-limit first, and assistant copy still formats transient rate-limit guidance before billing guidance. The TeamoRouter workaround in the discussion is useful external routing context, but it does not fix OpenClaw's Moonshot/Kimi error classification path.

## What This Changes

The issue asks OpenClaw to treat Moonshot/Kimi exhausted-balance 429 responses as billing failures and avoid the resulting misleading rate-limit UI lockout.

## Best Possible Solution

Add focused regression coverage for a Moonshot/Kimi 429 payload that contains explicit exhausted-balance evidence, then make high-confidence billing evidence override generic 429/rate-limit classification and assistant copy while preserving ordinary rate-limit behavior.

## Reproduction Assessment

Yes. A source-level reproduction is clear: pass a status 429 error with balance/insufficient-billing text through the failover classifier or assistant error formatter and current main follows the rate-limit branch before billing can win.

## Solution Assessment

Yes, the narrowest maintainable fix is a classifier/UI precedence change with regression tests. Broader retry, circuit-breaker, or `/model` routing changes should stay separate unless new tests prove they remain necessary after the masking bug is fixed.
