## Summary

Current main still preserves the central bug: Moonshot/Kimi HTTP 429 errors are classified as rate_limit before explicit insufficient-balance or credit text can take over, and the bundled Moonshot plugin has no provider-owned classifier to override that behavior.

## Best Possible Solution

Keep this issue open. Add a focused regression for a Moonshot/Kimi HTTP 429 payload whose code or message clearly indicates insufficient balance or credits, then fix classification in the Moonshot provider via `classifyFailoverReason` or adjust precedence so explicit billing evidence overrides generic 429 for that provider. Verify both runtime state and UI copy: billing-disabled/auth-profile behavior should be used, and the user should see a top-up or switch-key message instead of a generic rate-limit warning.
