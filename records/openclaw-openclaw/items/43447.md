## Summary

Keep open. Current main still leaves the central Moonshot/Kimi zero-balance case vulnerable to rate-limit misclassification: HTTP 429 is classified as `rate_limit` before billing evidence can win, message classification checks rate-limit wording before billing wording, and assistant error formatting tries rate-limit copy before billing copy. The discussion shows real Moonshot credit/recharge impact; the linked TeamoRouter workaround helps users route around failures, but it does not fix OpenClaw’s provider error classification.

## What This Changes

The issue requests clearer Moonshot/Kimi exhausted-balance handling, bounded background retry behavior, and a provider-independent model-switch escape path when the active provider is failing.

## Best Possible Solution

Add regression coverage for a Moonshot/Kimi HTTP 429 payload containing exhausted-balance evidence, then make billing evidence override generic 429/rate-limit classification and assistant UI copy while preserving ordinary rate-limit behavior. Use the existing generic classifier precedence or the provider-owned classifier seam; avoid broader retry or circuit-breaker redesign in the same PR unless a test proves it is required.
