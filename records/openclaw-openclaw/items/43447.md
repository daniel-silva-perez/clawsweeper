## Summary

Keep open. Current main still preserves the masking path: explicit HTTP 429 is classified as `rate_limit` before billing evidence can win, mixed rate-limit/billing messages check rate-limit first, and assistant error copy formats transient rate-limit guidance before billing guidance. The TeamoRouter workaround in the discussion is useful external routing context, but it does not fix OpenClaw's Moonshot/Kimi classifier behavior.

## What This Changes

The issue asks OpenClaw to classify Moonshot/Kimi exhausted-balance 429 responses as billing or insufficient funds and avoid the resulting retry/UI lockout behavior.

## Best Possible Solution

Add focused regression coverage for a Moonshot/Kimi HTTP 429 payload that contains explicit exhausted-balance evidence, then make billing evidence override generic 429/rate-limit classification and assistant error copy while preserving ordinary rate-limit handling. Prefer a generic precedence fix if it is safe; otherwise use a Moonshot-owned provider hook through the existing provider plugin contract.
