## Summary

Keep open. Current main still has the reported masking path: explicit HTTP 429 status is classified as `rate_limit` before billing evidence can win, raw mixed 429/balance text is classified as rate-limit before billing, and user-facing assistant copy formats transient rate-limit text before checking billing. The TeamoRouter comment is useful external failover context, but it is a workaround outside OpenClaw and does not fix the shared classifier behavior.

## What This Changes

The issue asks OpenClaw to surface Kimi/Moonshot zero-balance 429 responses as billing/insufficient-funds failures and avoid locking users into a misleading rate-limit loop.

## Best Possible Solution

Add focused regression coverage for a Moonshot/Kimi-style 429 response containing explicit exhausted-balance evidence, then make high-confidence billing signals override generic 429/rate-limit classification and assistant copy while preserving ordinary 429 rate-limit behavior. Keep provider-specific special cases, retry circuit breakers, and command-routing changes separate unless a real payload or regression test proves they are still necessary.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: pass `status: 429` with `insufficient_balance`, `credit balance`, or equivalent exhausted-balance text through `classifyFailoverReasonFromHttpStatus`, `classifyFailoverReason`, or `formatAssistantErrorText`; current main takes the rate-limit branch before billing can win.

## Solution Assessment

Yes. The best fix is a narrow shared classifier and formatter precedence change with regression tests; a Moonshot-only special case would be less maintainable unless Moonshot returns a unique payload that the shared billing matcher cannot recognize.
