## Summary

Keep open. Current main still preserves the core masking path: explicit HTTP 429 is classified as `rate_limit` before billing evidence can win, mixed rate-limit/billing messages check rate-limit first, and assistant UI copy formats transient rate-limit text before billing guidance. The TeamoRouter workaround mentioned in the discussion can route around provider outages, but it does not fix OpenClaw's Moonshot/Kimi classification behavior.

## What This Changes

The issue asks OpenClaw to classify Moonshot/Kimi exhausted-balance 429 responses as billing or insufficient funds and avoid the resulting retry/UI lockout behavior.

## Best Possible Solution

Add regression coverage for a Moonshot/Kimi HTTP 429 payload that contains explicit exhausted-balance evidence, then make billing evidence override generic 429/rate-limit classification and assistant error copy while preserving ordinary rate-limit handling. Use the existing generic classifier precedence if safe, or a Moonshot-owned `classifyFailoverReason` hook if provider-specific handling is needed; keep retry/circuit-breaker and command-routing redesign out of this focused fix unless newly reproduced.
