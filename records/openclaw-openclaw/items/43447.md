## Summary

Keep open. Current main still has the central Moonshot/Kimi billing-as-rate-limit bug: HTTP 429 is classified as `rate_limit` before billing evidence can win, UI copy checks rate-limit copy before billing copy, and the Moonshot plugin does not provide an overriding classifier. Adjacent cascade issues have been fixed separately, and a prior exact PR was stale-closed unmerged, so this remains actionable.

## What This Changes

The issue requests clearer Moonshot/Kimi insufficient-funds handling, bounded background retry behavior, and an emergency model-switch path when the active provider is failing.

## Best Possible Solution

Add a focused regression for a Moonshot/Kimi HTTP 429 payload that clearly indicates insufficient balance or suspended billing, then make billing evidence override generic 429/rate-limit classification and UI copy for that provider or explicit billing payload shape. The end state should mark billing failures through auth-profile disabled billing behavior and show top-up/switch-key guidance while preserving ordinary 429 rate-limit handling.
