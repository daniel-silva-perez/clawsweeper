## Summary

Keep open. Current main still leaves the central Moonshot/Kimi zero-balance case misclassified: explicit HTTP 429 is forced to `rate_limit`, message classification checks rate-limit wording before billing evidence, and user-facing error formatting also prefers transient rate-limit copy before billing copy. Adjacent generic failover and billing matchers have improved, but they do not yet cover the reported 429-with-insufficient-balance shape.

## What This Changes

The issue asks OpenClaw to treat Moonshot/Kimi exhausted-credit 429 responses as billing failures, avoid misleading rate-limit UI copy, and prevent provider-error cascades that trap users on a broken model.

## Best Possible Solution

Add a focused regression for a Moonshot/Kimi HTTP 429 payload with explicit exhausted-balance evidence, then make that billing evidence override generic 429/rate-limit classification and UI copy. Keep ordinary `rate_limit_error` 429s and retryable 402 usage-window behavior unchanged; use either a narrow generic precedence rule or the existing Moonshot provider classifier seam.
