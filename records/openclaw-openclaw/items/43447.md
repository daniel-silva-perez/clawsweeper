## Summary

Current main still appears to misclassify the reported Moonshot zero-balance 429 as `rate_limit`: generic status and text precedence beat the existing billing matcher, and the Moonshot plugin has no provider-owned classifier. The exact prior fix PR was closed unmerged and no open replacement surfaced, so this remains actionable.

## What This Changes

The issue asks OpenClaw to treat Moonshot/Kimi 429 responses that indicate exhausted credits as billing failures, stop misleading rate-limit UI copy, and avoid provider-error cascades that trap users on a broken model.

## Best Possible Solution

Add a focused regression for a Moonshot/Kimi HTTP 429 payload that clearly indicates exhausted balance, then make billing evidence override generic 429/rate-limit classification and UI copy for that provider or explicit billing payload shape. The final behavior should classify the failure as billing, guide the user to top up or switch keys/models, and leave normal transient 429 handling intact.
