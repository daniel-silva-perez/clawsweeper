## Summary

Keep open. Current main still has the reported masking path: HTTP 429 returns `rate_limit` before billing evidence can win, mixed raw text checks rate-limit patterns before billing patterns, and user-facing assistant copy formats rate-limit text before the billing branch. The TeamoRouter comment is useful external failover context, but it is a workaround outside OpenClaw and does not fix the shared classifier path.

## What This Changes

The issue requests that Moonshot/Kimi exhausted-balance responses that arrive as HTTP 429 surface as billing or insufficient-funds errors instead of rate-limit errors, without locking users into the failing provider path.

## Best Possible Solution

Add focused regression coverage for a Moonshot/Kimi-style 429 response containing explicit exhausted-balance evidence, then make high-confidence billing signals override generic 429/rate-limit classification and assistant copy while preserving ordinary 429 rate-limit behavior. Keep provider-specific special cases, retry circuit breakers, and command-routing changes separate unless a real payload or regression test proves they are still necessary.

## Reproduction Assessment

Yes. ## Solution Assessment

Yes. The narrowest maintainable fix is shared classifier and formatter precedence with regression tests, not a Moonshot-only core special case; provider-local handling should be added only if Moonshot’s real payload needs a matcher the shared billing patterns do not cover.
