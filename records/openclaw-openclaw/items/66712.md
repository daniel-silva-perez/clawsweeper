## Summary

Keep open. Current main still only has execution-scoped direct cron delivery dedupe keyed by jobId plus runStartedAt, and startup catch-up still drops deliveryAttempted from its returned outcome. This PR remains a focused implementation candidate, with one maintainer review question around whether the proposed slot key should include threadId.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to land a focused process-local scheduled-slot dedupe guard alongside the existing execution-level cache, include regression coverage for the same scheduled slot with different runStartedAt values, preserve the startup catch-up deliveryAttempted propagation, and decide whether threadId belongs in the slot key before merge.

## Risks / Open Questions

- The proposed slot key omits delivery.threadId, while current direct-delivery idempotency includes threadId and delivery targets explicitly carry it. That could collapse distinct thread/topic deliveries for the same job, slot, channel, account, and recipient unless reviewed or fixed before merge.
- No local tests were run because this was a read-only cleanup review and test commands can create checkout artifacts; GitHub check-runs for the PR head reported 56 success and 10 skipped.
- The proposed guard is process-local only. That is documented in the PR, but it does not address cross-process duplicate delivery races.
