## Summary

Keep open. Current main still has the scheduled-slot replay gap this PR is trying to close, and the PR is narrow with green public checks, but the proposed slot key omits `threadId` even though threaded/topic delivery is part of the cron delivery contract and the existing execution-level key includes it.

## What This Changes

The PR adds an in-memory scheduled-slot cron delivery dedup cache, resets it in the double-announce tests, adds a same-slot/different-runStartedAt regression test, and propagates `deliveryAttempted` from startup catch-up outcomes.

## Best Possible Solution

Keep the slot-level guard, but make its destination identity match the existing direct-delivery identity by including normalized `threadId`, then add a regression test for two deliveries to the same chat/account/slot with different thread IDs before merging.

## Reproduction Assessment

Yes. The main bug is reproducible with two `dispatchCronDelivery` calls for the same job and `job.state.nextRunAtMs` but different `runStartedAt`; the PR's new test covers that path. The review finding is reproducible by varying only `resolvedDelivery.threadId` between those two calls after the first slot delivery is recorded.

## Solution Assessment

No, not as-is. The scheduled-slot guard is the right narrow direction, but the key should include `stringifyRouteThreadId(delivery.threadId)` so topic/thread destinations are not collapsed.

  - body: The new slot cache keys on job, scheduledAtMs, channel, accountId, and normalized `to`, but cron delivery supports topic/thread destinations and the existing execution-level key includes `threadId`.
