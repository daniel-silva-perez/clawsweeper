## Summary

Keep this PR open. Current main still only deduplicates direct cron delivery by execution id, which is derived from job id plus runStartedAt, so it does not implement the PR's scheduled-slot guard for same planned slot/different runStartedAt replays. The PR also still contains useful work for startup catch-up outcome propagation: runStartupCatchupCandidate currently forwards delivered but drops deliveryAttempted.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to land a focused process-local scheduled-slot dedup guard alongside the existing execution-level cache, include regression coverage for same scheduled slot with different runStartedAt, keep the startup catch-up deliveryAttempted propagation, and verify whether the slot key should include threadId before merge.
