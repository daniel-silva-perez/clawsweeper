## Summary

Keep open. Current main still has the silent config-recovery catches and the misleading success log when backup restore copy fails. The PR overlaps with newer open #70515 for the backup-copy failure path, but it is not fully superseded because it also targets health-state and clobbered-snapshot diagnostics that are still silent on main.

## Best Possible Solution

Keep this PR open for maintainer review, but reconcile it with #70515 before merging. The best path is to land one tested restore-failure implementation, preferably with #70515's audit/error fields, then either fold in this PR's remaining health-state and clobbered-snapshot diagnostics with focused tests or split those into a narrower follow-up.

## Risks / Open Questions

- This PR overlaps with open #70515 on the backup-restore failure path; maintainers should avoid landing two independent log/audit semantics for the same branch.
- This PR does not add regression tests for the new failure logging, so the intended behavior could regress without coverage.
- The new warnings include raw filesystem error messages, which can expose local paths in logs; that is probably acceptable for config recovery diagnostics but should be reviewed deliberately.
