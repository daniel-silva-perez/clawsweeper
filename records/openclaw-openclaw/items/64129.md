## Summary

Keep open. Current main still lets Google Gemini API-key setup select Gemini as the primary default, while heartbeat runs continue to use the normal default model unless a heartbeat-specific model is configured. The prior Codex review in the issue discussion was directionally correct, and the current checkout still shows the same paid-background-use gap for an existing non-empty HEARTBEAT.md.

## Best Possible Solution

Keep this issue open and add an explicit paid-background-use boundary. A conservative fix should either preserve the existing primary model during Google/provider setup unless the user explicitly confirms switching defaults, or require explicit confirmation/disablement before heartbeat can inherit a newly configured paid external provider. Add regression coverage for setup with an existing primary model plus an actionable HEARTBEAT.md so heartbeat cannot silently begin paid Gemini usage.

## Risks / Open Questions

- Users with an existing actionable HEARTBEAT.md can still generate paid Gemini background turns after setup changes the primary default model.
- The current mitigations are manual or partial: empty/comment-only HEARTBEAT.md skips, heartbeat.model, lightContext, and isolatedSession reduce cost only when users already know to configure them.
- No regression test currently covers the reported combination of Google setup, an existing primary model, and a non-empty heartbeat file requiring explicit paid-background-use consent.
