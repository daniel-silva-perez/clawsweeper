## Summary

Keep this PR open. Current main still does not implement the intended TUI `/new` hook-visible reset behavior, and the PR is an active implementation candidate for open regression #49918 rather than obsolete cleanup.

## Best Possible Solution

Keep the PR open and review it as the implementation candidate for #49918. The best fix should make TUI `/new` emit `command:new` through the normal reset hook path while preserving per-client `tui-*` isolation, pass the leaving session as the hook context, avoid blank session-list clutter, and either avoid or fully document/constrain any additive Gateway protocol field. Before merge, update the changelog attribution and run the targeted TUI command-handler and gateway session reset tests.

## Risks / Open Questions

- The PR adds a public `sessions.reset.hookSourceKey` field without current main protocol docs or an obvious authorization boundary for choosing a different hook context session.
- The proposed PR behavior still needs maintainer review for bundled hook attribution: review comments note `event.sessionKey` may point at the new `tui-*` key while hooks are archiving the old transcript.
- Calling `sessions.reset` on a fresh `tui-*` key may create blank persisted session rows before the user sends a message, which can clutter session listing/history.
- The PR changelog entry shown in the provided patch lacks the required `Thanks @author` attribution and would need cleanup before merge if it remains.
