## Summary

Keep this PR open. Current main still has the TUI `/new` path creating an isolated `tui-*` session via `setSession(...)` without calling the gateway `sessions.reset(reason="new")` path, while the linked regression remains open and the PR is the active implementation candidate. The PR also changes the admin-only Gateway reset protocol and hook attribution behavior, so it needs maintainer review rather than conservative cleanup closure.

## Best Possible Solution

Keep the PR open and review it as the implementation candidate for #49918. The best fix should make TUI `/new` emit hook-visible `command:new` behavior while preserving per-client `tui-*` isolation, keep `event.sessionKey` and `previousSessionEntry` semantics correct for bundled and custom hooks, avoid cluttering session history with empty sessions, and either document/approve the Gateway protocol addition or solve the attribution problem without a new protocol field. Before merge, add/adjust regression tests for those semantics and fix the changelog attribution.
