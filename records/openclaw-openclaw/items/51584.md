## Summary

Keep this PR open. Current main still does not implement the PR's intended static bundled-channel fallback in `sandbox explain`, and the linked Telegram bug remains represented by the open issue and the related open PR #51286. The PR is narrow and not a supply-chain concern, but landing it alone would be incomplete because documented bare `telegram:slash:<id>` input and stored channel metadata still need coordinated handling.

## Best Possible Solution

Keep this PR open and review it together with #51286 and the linked Telegram bug #51245. The best fix is a narrow `src/commands/sandbox-explain.ts` update that combines static bundled-channel normalization with non-agent session-key normalization, applies the fallback to stored channel metadata as well as session-key inference, and adds regression tests for agent-scoped and documented bare Telegram slash sessions.

## Risks / Open Questions

- Landing this PR alone likely would not fix the documented bare `openclaw sandbox explain --session telegram:slash:<id>` form, because current main returns colon-containing raw input unchanged and provider inference only works after `agent:` session parsing succeeds.
- `resolveActiveChannel` still normalizes stored `lastChannel`/`channel` metadata only through `normalizeAnyChannelId`; if the CLI plugin registry is empty, stored `telegram` metadata may still fall through without the same static fallback.
- The change affects elevated host exec availability, so maintainers should require focused regression tests for both `agent:main:telegram:slash:<id>` and the documented bare form before merge.
