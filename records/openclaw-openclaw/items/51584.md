## Summary

Current main still does not implement the PR's intended static bundled-channel fallback in `sandbox explain`. Runtime Telegram metadata fixes are present, and #51286 is a related open PR for bare `telegram:slash:<id>` input, but neither makes this fallback redundant on main. The PR remains a focused implementation candidate, with one important gap: as written it helps agent-scoped keys but does not normalize the documented bare Telegram slash key.

## Best Possible Solution

Keep this PR open and review it together with #51286 and the linked Telegram bug. The best fix is a narrow `src/commands/sandbox-explain.ts` update that uses bundled/static channel normalization for built-in channels in both session-key inference and stored metadata resolution, decides whether bare `telegram:slash:<id>` remains supported, and adds regression tests for `agent:main:telegram:slash:<id>` plus the documented bare form if it remains part of the CLI contract.

## Risks / Open Questions

- The PR as written appears to help canonical agent-scoped keys such as `agent:main:telegram:slash:<id>`, but raw documented input like `telegram:slash:<id>` still fails before provider inference because `parseAgentSessionKey` does not parse it.
- `resolveActiveChannel` still normalizes stored `lastChannel`/`channel` metadata only through `normalizeAnyChannelId`; if stored metadata exists while the CLI plugin registry is empty, the fallback likely needs to apply there too.
- Related open PR #51286 addresses bare session-key normalization, so maintainers should coordinate the two PRs rather than landing overlapping partial fixes.
