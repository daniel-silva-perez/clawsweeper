## Summary

Keep this PR open. The proposed fallback helps only the agent-scoped session-key inference path, but it does not fix the PR body's documented bare `telegram:slash:<id>` case because current `normalizeExplainSessionKey` preserves non-agent colon keys and `inferProviderFromSessionKey` exits before the new fallback can run. The related Telegram bug and broader normalization PR remain open, so this needs coordinated maintainer review rather than closure or merge as-is.

## What This Changes

This PR imports `normalizeChatChannelId` in `src/commands/sandbox-explain.ts` and uses it as a fallback after `normalizeAnyChannelId` when inferring a channel from an agent-scoped session key.

## Best Possible Solution

Land one consolidated sandbox-explain fix that normalizes documented bare Telegram slash keys into the agent-scoped store/inference path, applies bundled-channel fallback consistently to both session-key inference and stored channel metadata, and adds regression tests for bare and agent-scoped Telegram slash sessions with an empty runtime registry.

## Reproduction Assessment

Yes. The failure path is reproducible by code inspection on current main: `--session telegram:slash:<id>` is preserved as a raw colon key, then rejected by `parseAgentSessionKey`, so the new fallback would not run and the channel remains unknown without usable stored metadata.

## Solution Assessment

No. The PR's fallback is a useful ingredient, but it is not the best complete fix because it only handles agent-scoped inference and leaves the documented bare session form and stored metadata path unresolved.
