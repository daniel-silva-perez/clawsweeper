## Summary

Keep this PR open. Current main still does not implement the intended static bundled-channel fallback in `sandbox explain`; the command only uses the runtime channel plugin registry, while the bundled-channel normalizer that resolves `telegram` exists elsewhere. The PR is narrow and security-safe, but it needs maintainer review and regression coverage, especially around raw legacy `telegram:slash:<id>` input.

## Best Possible Solution

Review and likely land an adjusted version of this PR for the linked Telegram elevated/sandbox explain bug: add a static bundled-channel fallback in `sandbox explain` active-channel resolution, cover built-in channels when the runtime plugin registry is empty, and add regression tests for `agent:main:telegram:slash:<id>` plus raw `telegram:slash:<id>` if that CLI input remains supported.

## Risks / Open Questions

- The PR as written likely covers `agent:main:telegram:slash:<id>` but not raw legacy `telegram:slash:<id>` CLI input, because `inferProviderFromSessionKey` returns `undefined` when `parseAgentSessionKey` fails.
- `resolveActiveChannel` still normalizes stored `lastChannel`/`channel` candidates only through `normalizeAnyChannelId`; maintainers should decide whether the static fallback should be applied there too, not only in inferred session-key parsing.
- No regression test currently proves built-in channel resolution when the runtime plugin registry is empty.
