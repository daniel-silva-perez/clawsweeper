## Summary

Keep this PR open. Current main still does not implement the requested static bundled-channel fallback in `sandbox explain`: it imports and calls only `normalizeAnyChannelId`, which resolves through the active runtime plugin channel registry. Telegram native slash sessions still use the agent-scoped `agent:<agentId>:telegram:slash:<userId>` shape, so with an empty/unloaded CLI registry the diagnostic can still report `channel: (unknown)` even though the bundled channel normalizer can resolve `telegram`. The PR is narrow and does not introduce a visible security or supply-chain risk, but it should get maintainer review and regression coverage before merge.

## Best Possible Solution

Review and likely land an adjusted version of this PR for the linked Telegram elevated/sandbox-explain bug: add a bundled-channel fallback in `sandbox explain` active-channel resolution, cover runtime-registry-empty behavior for built-in channels such as Telegram, and add regression tests for canonical `agent:main:telegram:slash:<id>` plus raw `telegram:slash:<id>` if that CLI input is still intended to work.

## Risks / Open Questions

- The PR as written likely helps canonical agent-scoped keys such as `agent:main:telegram:slash:<id>`, but raw legacy CLI input like `telegram:slash:<id>` still does not parse as an agent session key, so maintainers should decide whether that input remains supported.
- `resolveActiveChannel` still normalizes stored `lastChannel`/`channel` candidates only through `normalizeAnyChannelId`; if stored metadata can be present while the runtime registry is empty, the static fallback may need to be applied there too.
- No regression test currently proves built-in channel resolution in `sandbox explain` with an empty runtime plugin registry.
