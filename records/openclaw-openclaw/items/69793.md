## Summary

Keep this issue open. Current main still builds Telegram inbound photo text from Telegram message shape rather than the saved media content type: `msg.photo` maps directly to `<media:image>`, and the empty-text media fallback still hard-codes `<media:image>` for every non-audio attachment. The linked PR #69893 explicitly included a proposed fix for this issue, but that PR was closed unmerged, so the patch did not land on main.

## Best Possible Solution

Keep the issue open and land a narrow Telegram fix that derives the media placeholder from the saved `allMedia[].contentType` when media exists, while preserving type-based placeholders for failed downloads with no saved media. Add focused tests for non-image, image, video, mixed, and audio-preflight cases; #69893 can be used as prior art but should not be revived wholesale because it bundled unrelated fixes and was closed unmerged.
