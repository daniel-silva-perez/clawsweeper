## Summary

Keep this issue open. Current main still builds Telegram no-text media body text from the Telegram message shape first, so `msg.photo` can produce `<media:image>` even when the saved attachment content type is non-image. The related broad PR #69893 is closed unmerged, and a read-only search found no open PR currently tracking this exact issue.

## Best Possible Solution

Keep this issue open and land a focused Telegram plugin fix. Derive no-user-text media placeholders from saved `allMedia[].contentType` when media exists, preserve message-shape placeholders for failed downloads with no saved media, preserve audio transcript behavior, avoid hard-coding a Telegram photo MIME fallback, and add regression tests for non-image, image, video, mixed media, and audio cases.
