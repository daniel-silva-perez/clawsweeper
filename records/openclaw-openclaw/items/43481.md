## Summary

Keep open: current main still lacks a provider-level media trust policy for model fallback, and the issue is paired with open PR #57959 using closing syntax. Because this is security-sensitive fallback-boundary work, cleanup should not close it or queue a competing repair while the existing PR/replacement decision is pending.

## What This Changes

Add provider-level fallback trust controls so users can decide whether images and attachments may cross provider boundaries during model fallback.

## Best Possible Solution

Review and either land PR #57959 if maintainers accept the fail-closed cross-provider media policy, or replace it with a centralized schema/docs-backed fallback trust policy that covers explicit images, prompt-detected local images, attachments, same-provider retries, and cross-provider retries consistently.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path for the remaining gap: configure cross-provider model fallbacks and submit a prompt that references a local image; current main has no provider trust config and no fallback trust guard around prompt image detection, even though explicit opts.images are dropped on fallback retries.

## Solution Assessment

No current-main solution exists. The maintainable direction is a single fallback-boundary policy wired through model fallback, attempt execution, embedded prompt image detection, config schema/help, docs, and focused same-provider/cross-provider tests.
