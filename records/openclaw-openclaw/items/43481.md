## Summary

Keep open: current main still lacks provider-level media trust configuration for model fallback, and the issue remains paired with open PR #57959 using closing syntax. This is security-sensitive fallback-boundary work, so cleanup should not close it or queue a competing automated repair while the existing PR is awaiting maintainer review.

## What This Changes

The issue requests provider-level trust controls so model fallback can distinguish text-retry trust from permission to send user images or attachments across provider boundaries.

## Best Possible Solution

Review and either land PR #57959 with any needed revisions, or replace it with a centralized fallback-boundary implementation that carries primary and target provider context, covers explicit and prompt-detected media consistently, adds schema/docs for the trust control, and tests same-provider plus cross-provider behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: configure model fallbacks across providers, submit explicit images or prompt-referenced local images, and current main has no schema/runtime trust policy to decide whether that media may cross provider boundaries.

## Solution Assessment

No current-main solution exists. The maintainable direction is a single fallback-boundary policy wired through model fallback, attempt execution, prompt image detection, config schema, docs, and tests rather than ad hoc image stripping in only one path.
