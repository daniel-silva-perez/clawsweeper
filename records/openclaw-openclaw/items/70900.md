## Summary

Keep this PR open. Current main still has the unguarded assistant `surface_error` throw path, and the PR is a focused source/test fix for a plausible success-to-error regression caused by raw text classification diverging from the `stopReason === "error"` failover signal.

## What This Changes

The PR adds `params.failoverFailure` to the assistant `surface_error` throw guard and adds a unit test for `failoverFailure=false` with a classified stale `failoverReason`.

## Best Possible Solution

Land this PR, or an equivalent minimal guard plus the focused regression test, then run the normal changed gate. Keep any broader rethink of text-only assistant rotation separate so the #70848/#70124 provider-error surfacing behavior stays intact.

## Reproduction Assessment

Yes. A high-confidence pure-helper reproduction is `handleAssistantFailover` with `initialDecision: { action: "surface_error", reason: "billing" }`, `failoverFailure=false`, `failoverReason="billing"`, and `billingFailure=false`; current main throws, while the PR expects `continue_normal`.

## Solution Assessment

Yes. For the reported no-fallback success-to-error conversion, gating the terminal `surface_error` throw on `failoverFailure` is the narrowest maintainable fix because that signal is already stop-reason gated, while genuine provider failures still throw.
