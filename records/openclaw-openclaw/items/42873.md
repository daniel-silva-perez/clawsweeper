## Summary

Keep this PR open. Current main still strips a trailing own-line `NO_REPLY` from `text\n\nNO_REPLY` and returns the preamble as deliverable text, while the current regression test still asserts that behavior. The PR remains a plausible implementation candidate and should be coordinated with the open follow-up work in #46626 and #69394 rather than closed as obsolete.

## Best Possible Solution

Keep this PR open for maintainer review or author follow-up. The best path is to consolidate it with #46626 and #69394, add a shared cached own-line or trailing silent-token detector, apply it first to `src/auto-reply/tokens.ts` and `src/auto-reply/reply/normalize-reply.ts`, and update tests to flip the current own-line trailing `NO_REPLY` expectation while preserving inline stripping, JSON envelope suppression, glued-leading-token handling, case-insensitivity, media/channelData behavior, and the cron-specific suppression already on main.

## Risks / Open Questions

- The PR changes product behavior currently covered by #30916 tests: substantive text ending with an own-line trailing `NO_REPLY` would become fully silent instead of delivering the stripped preamble.
- The PR patch shown in the provided context uses a case-sensitive `text.includes(silentToken)` guard, while current main uses a case-insensitive guard; a rebase should preserve mixed-case `No_Reply` behavior.
- The same-author follow-up #46626 reports additional delivery paths beyond `normalizeReplyPayload`, so this PR may be necessary but not sufficient for the full leak class.
