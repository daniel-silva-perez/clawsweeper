## Summary

Keep this PR open. Current main still normalizes `text\n\nNO_REPLY` by stripping the trailing token and returning the preceding text, and the current regression test still asserts that behavior. The PR remains a relevant implementation candidate, with related open work in #46626 and #69394 that maintainers should coordinate before landing or superseding it.

## Best Possible Solution

Keep this PR open for maintainer review or author follow-up. The best path is a rebased shared silent-token detector in `src/auto-reply/tokens.ts` and `src/auto-reply/reply/normalize-reply.ts`, with tests that flip the current own-line trailing `NO_REPLY` expectation while preserving inline token stripping, JSON envelope suppression, glued-leading-token handling, case-insensitivity, media/channelData behavior, and the cron-specific suppression already landed for #64976. Coordinate with #46626 and #69394 before landing or closing duplicates.

## Risks / Open Questions

- The PR changes an intentional product behavior currently covered by #30916 tests: a substantive reply ending with an own-line trailing `NO_REPLY` would become fully silent instead of delivering the stripped preamble.
- The patch shown in the provided PR diff uses `text.includes(silentToken)`, while current main uses a case-insensitive guard; a rebase should preserve mixed-case `No_Reply` handling.
- #46626 reports additional delivery paths beyond `normalizeReplyPayload`, so this PR may be necessary but not sufficient for the full leak class.
