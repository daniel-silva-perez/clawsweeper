## Summary

Keep this PR open. Current main still strips a trailing own-line `NO_REPLY` from `normalizeReplyPayload()` and returns the preceding text, and the shipped `v2026.4.27` test suite still asserts that old behavior. The PR targets a real leak class, but it needs maintainer follow-up because the patch is stale against newer case-insensitive and leading-token handling, and related open PRs cover overlapping or broader delivery paths.

## What This Changes

The PR adds an own-line `NO_REPLY` detector, uses it in auto-reply normalization to suppress the whole response when the token appears on its own trailing line, and updates token and normalizer tests.

## Best Possible Solution

Adopt a shared, cached, case-insensitive silent-token detector that treats an own-line trailing `NO_REPLY` as turn-level silence, preserve inline token stripping and glued-leading-token behavior, add mixed-case/media/channelData tests, and coordinate this normalizer change with the broader delivery-path work in #46626 and the alternate normalizer proposal in #69394.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence reproduction: `normalizeReplyPayload({ text: "File's there. Not urgent.\n\nNO_REPLY" })` strips only the trailing token, keeps non-empty text, and the current test suite asserts that non-null result.

## Solution Assessment

No, not as-is. The PR's direction is useful, but the proposed patch needs a current-main rebase and a case-insensitive helper/guard before it is the safest maintainable fix; maintainers should also decide whether to land it alone or supersede it with coordinated work from #46626 or #69394.
