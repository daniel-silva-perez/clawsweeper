## Summary

Current main still does not implement group pairing. Group policy is still limited to open/disabled/allowlist, pairing remains DM-only in schema, docs, resolver behavior, and CLI, and Telegram/WhatsApp group access still blocks or allows groups without a pending approval/owner-notification state. The related open #8495 and #42400 items overlap group-join notification and read-only/no-reply behavior, but they do not supersede the group approval flow requested here.

## Best Possible Solution

Keep this open for maintainer/product design. The best path is an additive group-pairing contract: extend config/schema/docs with groupPolicy="pairing", persist pending group approvals keyed by channel/account/group, add CLI approve/reject operations, emit owner-visible group-add or first-seen-unconfigured-group notifications, and wire Telegram/WhatsApp through the shared state. Coordinate the optional read-only reader-agent mode with #42400 or split it into a narrower follow-up.
