## Summary

Keep open. Current main still implements DM-only pairing while group access is limited to open, disabled, or allowlist behavior; there is no group pending approval state, owner notification on group add, group approve/reject CLI, or pending read-only group routing. The existing issue comments and related open work support the same conclusion: #42400 covers silent group observation/suppression, and #8495 covers WhatsApp auto-leave/join events, but neither is a canonical cross-channel group pairing contract.

## What This Changes

Requested feature: add a group pairing flow that notifies the owner when a bot is added to an unconfigured group, supports approve/reject, and optionally routes pending groups through read-only observation.

## Best Possible Solution

Define an additive cross-channel group-pairing contract: extend config/schema/docs, persist pending approvals by channel/account/group, deliver owner-visible group-add notifications, add authorized approve/reject operations, and wire Telegram plus WhatsApp through shared policy state. Treat reader-agent observation as a separately scoped mode unless maintainers explicitly include it in the approval contract.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Static verification against current main is high confidence: config/schema/docs exclude `groupPolicy: "pairing"`, group runtime paths block or allow groups directly, and pairing state/CLI remain DM-only.

## Solution Assessment

Unclear as written. Group owner notification and approve/reject are a maintainable direction, but bundling pending reader-agent observation into the same first change may not be the narrowest safe solution until maintainers settle the authorization and isolation contract.

## Security Review

Status: needs_attention

Summary: This issue is security-sensitive because it changes group authorization, owner notification, and pending read-only processing semantics.

Concerns:

- **[medium] Define group approval authorization first:**
  - body: Group join notifications and approve/reject operations decide who can expose or process untrusted group content. Maintainers should define owner identity, notification routing, approval state storage, and authorization checks before implementation.
  - confidence: 0.9
- **[medium] Bound pending read-only routing explicitly:**
  - body: A pending-group reader agent could still ingest untrusted group content. Its tools, memory visibility, outbound capabilities, and retention behavior need a clear contract before implementation.
  - confidence: 0.84
