## Summary

Keep open. Current main still has DM-only pairing while group access remains limited to open, disabled, and allowlist behavior; the requested pending group approval state, owner notification on group add, group approve/reject commands, and pending read-only reader-agent routing are not implemented. The related #8495 and #42400 discussions are adjacent, not canonical implementations of this cross-channel group pairing request.

## What This Changes

Request a cross-channel group pairing flow that notifies the owner when the bot joins a group, holds unconfigured groups pending approval, and optionally routes pending groups through a read-only reader agent.

## Best Possible Solution

Define an additive cross-channel group-pairing contract: extend config/schema/docs, persist pending approvals by channel/account/group, deliver owner-visible group-add notifications, add authorized approve/reject operations, and wire Telegram plus WhatsApp through shared policy state. Treat reader-agent observation as a separately scoped mode unless maintainers explicitly include it in the approval contract.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Static verification is high-confidence because current main rejects groupPolicy:"pairing", keeps pairing state/CLI DM-only, and group runtime paths directly allow or block messages without pending approval.

## Solution Assessment

Unclear as written. Owner notification plus approve/reject is a maintainable direction, but bundling restricted reader-agent ingestion into the same first change may not be the narrowest safe solution until the authorization and isolation contract is settled.
