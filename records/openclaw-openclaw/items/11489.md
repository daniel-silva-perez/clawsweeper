## Summary

Current main still implements DM pairing plus group open/disabled/allowlist handling. It does not have group pairing approval state, group approve/reject CLI semantics, owner notification for new group additions, or pending read-only group-agent routing, so this issue should remain open for maintainer/product design.

## Best Possible Solution

Keep this open for maintainer/product design. The likely path is an additive group-pairing contract: add groupPolicy="pairing" to config/schema/docs, persist pending group approvals keyed by channel/account/group, expose group approve/reject CLI operations, emit owner-visible group-join or first-seen-unconfigured-group notifications, and make Telegram/WhatsApp honor that state. The pending read-only reader-agent mode should be explicitly scoped into that design or split into a coordinated follow-up with #42400.
