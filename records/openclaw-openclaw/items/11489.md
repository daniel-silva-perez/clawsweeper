## Summary

Keep this open. Current main still implements DM/device pairing only; group handling remains `open | disabled | allowlist`, with explicit group allowlists and no group approval state, reject/approve CLI shape, owner notification, or pending read-only group-agent routing. The existing discussion also treats #8495 and #42400 as adjacent rather than superseding this request.

## Best Possible Solution

Keep this open for maintainer/product design. The likely path is an additive group-pairing contract: add `groupPolicy: "pairing"` to config/schema/docs, persist pending group approvals keyed by channel/account/group, expose CLI approve/reject operations, emit owner-visible group-join or first-seen-unconfigured-group notifications, and make Telegram/WhatsApp honor that state. The pending read-only reader-agent mode should either be scoped into that design explicitly or split into a coordinated follow-up with #42400.
