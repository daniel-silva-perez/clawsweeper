## Summary

Keep open. Current main still implements pairing only for DMs while group access remains open, disabled, or allowlist-based; there is no pending group approval state, owner notification on group add, group approve/reject CLI, or pending reader-agent routing. The existing maintainer and ClawSweeper comments match the current source, and the related open work is adjacent rather than a canonical fix for this cross-channel approval flow.

## What This Changes

The issue requests a cross-channel group pairing flow that notifies the owner when a bot is added to a group, blocks normal processing until approval, supports approve/reject operations, and optionally routes pending groups through a read-only reader agent.

## Best Possible Solution

Define an additive group-pairing contract across core and channel plugins: config/schema support for a pending group policy, persisted approval state keyed by channel/account/group, owner-visible group-add notifications, authorized approve/reject operations, and Telegram/WhatsApp implementations. Treat reader-agent observation as a separately designed mode unless maintainers explicitly include it in the first approval contract.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Static verification is high-confidence because current main rejects groupPolicy:"pairing", keeps pairing state and CLI sender-code based, and group runtime paths directly allow or block messages without pending approval.

## Solution Assessment

Unclear as written. Owner notification plus approve/reject group state is a maintainable direction, but combining pending approval with read-only reader-agent ingestion is not clearly the narrowest safe first change until authorization and isolation semantics are designed.
