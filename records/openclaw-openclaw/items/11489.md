## Summary

Keep open. Current main still implements DM-only pairing while group access remains open, disabled, or allowlist-based; there is no pending group approval state, owner notification on group add, group approve/reject CLI surface, or pending read-only group routing. The linked open items are adjacent rather than canonical fixes for this cross-channel group-pairing contract.

## What This Changes

Add a group pairing flow that notifies the owner when a bot joins an unconfigured group, lets the owner approve or reject that group, and optionally routes pending groups through read-only observation.

## Best Possible Solution

Define an additive cross-channel group-pairing contract: extend config/schema/docs, persist pending approvals by channel/account/group, deliver owner-visible group-add notifications, add authorized approve/reject operations, and wire Telegram plus WhatsApp through shared policy state. Treat reader-agent observation as a separately scoped mode unless maintainers explicitly include it in the approval contract.

## Security Review

Status: needs_attention

Summary: The issue is security-sensitive because it changes group authorization, owner notification, and pending read-only processing semantics.

Concerns:

- **[medium] Define group approval authorization first:**
  - body: Group join notifications and approve/reject operations decide who can expose or process untrusted group content. Maintainers should define owner identity, notification routing, approval state storage, and authorization checks before code lands.
  - confidence: 0.9
- **[medium] Bound pending read-only routing explicitly:**
  - body: A pending-group reader agent could still ingest untrusted group content. Its tools, memory visibility, outbound capabilities, and content-retention behavior need a clear contract before implementation.
  - confidence: 0.84
