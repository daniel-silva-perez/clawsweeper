## Summary

Keep this open. Current main still has DM-only pairing and group access remains limited to open, disabled, or allowlist behavior; the owner notification, pending group approval state, group approve/reject CLI shape, and optional read-only pending-group routing are not implemented. The existing maintainer-side Codex review in the issue discussion reached the same conclusion, and the related open work only covers adjacent group suppression/read-only observation, not group pairing approval.

## What This Changes

Add a group pairing flow for Telegram/WhatsApp-style group additions, including owner notification, approve/reject controls, and optional read-only handling for unconfigured groups.

## Best Possible Solution

Define an additive group-pairing contract that keeps group authorization explicit: extend config/schema/docs, persist pending group approvals keyed by channel/account/group, add owner-visible notifications and CLI approve/reject operations, and wire Telegram/WhatsApp through shared policy state. Treat the read-only reader-agent behavior as either a clearly scoped extension of that contract or a separate follow-up aligned with #42400.
