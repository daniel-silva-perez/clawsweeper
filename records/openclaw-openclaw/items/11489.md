## Summary

Keep open. Current main still lacks the requested group pairing flow: group policies are limited to open, disabled, or allowlist, while pairing remains DM-only. Telegram and WhatsApp currently block unauthorized group traffic instead of notifying an owner, storing pending group approval, adding group approve/reject commands, or routing through a pending read-only agent. Recent issue discussion reached the same conclusion, and related #42400 and #8495 are adjacent rather than canonical replacements.

## What This Changes

Request to add a cross-channel group pairing mode with owner notification, group approve/reject controls, and optional read-only pending-group routing for Telegram, WhatsApp, and similar group-capable channels.

## Best Possible Solution

Define an additive cross-channel group-pairing contract: extend config/schema/docs, persist pending group approvals keyed by channel/account/group, send owner-visible group-add notifications, add authorized CLI or inline approve/reject operations, and wire Telegram/WhatsApp through shared policy state. Treat reader-agent or silent observation as a separately scoped mode unless maintainers explicitly choose to combine it with approval.
