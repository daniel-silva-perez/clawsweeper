## Summary

Keep this issue open. Current main still does not carry Google Chat thread identity as a first-class outbound, streamed-block, or queued-followup routing field. Google Chat delivery can thread correctly only when `payload.replyToId` is already a valid `spaces/.../threads/...` value, but the current inbound/core handoff can still fall back to the message resource and queued followups still depend on `MessageThreadId`, which Google Chat does not populate.

## Best Possible Solution

Keep this issue open as the canonical streamed-block and queued-followup Google Chat thread-routing bug. The best fix is a consolidated Google Chat-owned patch with a narrow generic seam only where needed: preserve the inbound `message.thread.name` (and any thread key/space data needed) through outbound reply payloads, block pipeline keys/conflict checks, and follow-up queue routing; ensure every Google Chat text chunk, media send, edit/fallback, and queued drain uses a valid `spaces/.../threads/...` value rather than `spaces/.../messages/...`; add `googlechat` to `messages.queue.byChannel`; and add focused regression tests for typing-placeholder update, chunk 2+, coalesced chunks, and concurrent queued messages in the same thread.
