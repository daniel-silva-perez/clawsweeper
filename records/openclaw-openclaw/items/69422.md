## Summary

Keep this issue open. Current main still lacks a first-class Google Chat thread route through outbound reply payloads, block reply coalescing, and queued follow-up routing. The Google Chat sender can thread correctly only when `payload.replyToId` is already a valid `spaces/.../threads/...` value; upstream paths can still pass a message resource or omit thread metadata entirely.

## Best Possible Solution

Keep this issue open as the canonical Google Chat streamed-block and queued-follow-up thread-routing bug. The best fix is a consolidated Google Chat-owned patch with the smallest generic seam needed: preserve inbound `message.thread.name` and route metadata through outbound reply payloads, block reply keys/conflict checks, Google Chat delivery, and follow-up queue routing; ensure chunks, media, edits/fallback sends, and queued drains use valid `spaces/.../threads/...` values; add `googlechat` to `messages.queue.byChannel`; and cover the typing-placeholder, chunk 2+, coalesced, and concurrent queued-message cases with focused regression tests.
