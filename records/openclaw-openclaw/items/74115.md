## Summary

Keep open. Current main still intentionally blocks normal Gateway startup when `gateway.mode` is missing, while the supported configless path is Crestodian or the explicit `--allow-unconfigured` bootstrap flag; the reported no-provider WebUI crash and Raw mode regression still lack enough concrete reproduction detail to choose a focused code fix.

## What This Changes

The issue asks for Gateway and WebUI startup to work without complete settings or a configured model/provider, and adds that Raw mode is not working.

## Best Possible Solution

Codify the desired startup contract: either keep the current guarded Gateway startup and improve the setup/recovery messaging, or explicitly design a safe no-config/no-provider bootstrap state for the Gateway and Control UI. Any verified WebUI crash or Raw mode regression should then get its own focused regression test and fix in the relevant UI or Gateway RPC path.

## Reproduction Assessment

Unclear. The Gateway policy portion is source- and test-reproducible because normal startup blocks missing `gateway.mode` while `--allow-unconfigured` starts the bootstrap path, but the no-provider WebUI crash and Raw mode failure do not have enough concrete data to reproduce with high confidence.

## Solution Assessment

Unclear. This is not ready for a single automated fix because the main startup behavior is intentional and documented, while the other failures need exact reproduction before changing Control UI or model-list behavior.
