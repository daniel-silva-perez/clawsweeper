## Summary

Keep open. Current main still intentionally blocks normal Gateway startup when `gateway.mode` is missing or non-local, while the reported WebUI no-provider crash and Raw mode failure do not include enough concrete reproduction data to identify a narrow code fix.

## What This Changes

The issue asks for Gateway and WebUI startup to tolerate missing configuration and missing model/provider setup instead of crashing or blocking users before setup is complete.

## Best Possible Solution

Define an explicit maintainer-approved startup contract: either normal loopback Gateway startup supports a no-config/no-provider bootstrap state, or the existing guard remains while setup and repair paths become clearer. Any verified WebUI no-provider crash or Raw mode regression should then be split into focused Control UI/Gateway RPC fixes with regression tests.

## Reproduction Assessment

Unclear. The Gateway policy portion is reproducible from source and tests because normal `gateway run` blocks missing `gateway.mode` while `--allow-unconfigured` starts the bootstrap path, but the WebUI no-provider crash and Raw mode failure lack concrete steps or logs.

## Solution Assessment

Unclear. This is not ready for a single automated code fix because the main startup behavior is intentional and documented, while the UI/model complaints need exact reproduction before changing Control UI or Gateway model-list behavior.
