## Summary

Keep open. Current main has configless recovery helpers and safer model-list fallbacks, but normal Gateway startup still intentionally blocks missing or clobbered `gateway.mode`, and the no-provider WebUI crash plus Raw mode complaint do not have enough concrete reproduction detail to choose a focused code fix.

## What This Changes

Request that Gateway and the browser Control UI start without full config/model/provider setup and avoid crashing when no provider is configured, with a follow-up complaint that Raw mode is not working.

## Best Possible Solution

The durable end state should be an explicit maintainer-approved startup contract: either normal loopback Gateway startup safely supports a no-config/no-provider bootstrap state, or the existing guard remains and setup recovery paths are made clearer. Any verified WebUI no-provider crash or Raw-mode regression should then get its own focused regression test and fix in the Control UI/Gateway RPC surface.

## Reproduction Assessment

Unclear. The Gateway policy portion is source- and test-reproducible because normal `gateway run` blocks missing `gateway.mode` while `--allow-unconfigured` starts the bootstrap path, but the WebUI no-provider crash and Raw mode failure lack concrete reproduction steps or logs.

## Solution Assessment

Unclear. This is not ready for a single narrow automated fix because the main startup behavior is intentional and documented, while the other reported failures need exact reproduction before changing Control UI or model-list behavior.
