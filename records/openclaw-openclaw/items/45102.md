Current main has useful adjacent Gateway liveness diagnostics, bounded media/audio execution, and conservative stuck-session recovery, but I found no code, docs, tests, release note, or discussion evidence proving that the reported Windows + local FunASR sustained-load slowdown and restart loop is fixed. ## What This Changes

The issue reports that Gateway on Windows with local FunASR voice recognition becomes slow and unresponsive after several long-running audio tasks, causing `/v1/responses` timeouts until the Gateway is restarted.

## Best Possible Solution

Keep the issue open until maintainers can collect current-main diagnostics from the Windows + local FunASR setup, including Gateway stability events, event-loop metrics, memory/RSS, audio config, session/transcript sizes, and whether the active work is external CLI transcription, agent processing, or transcript replay.
