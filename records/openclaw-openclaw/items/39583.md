## Summary

Keep open. Current main still does not implement this PR's intended vLLM fix: `extractTextFromChatContent` returns string inputs before content-block handling, so JSON-stringified chat content-block arrays remain raw JSON instead of being normalized like real arrays. The PR is still useful, but its latest heuristic needs maintainer tightening because a `type`-only JSON-array guard can drop unrelated JSON string content.

## Best Possible Solution

Keep this PR open for maintainer follow-up. The best path is a conservative shared normalization helper that parses strings only when they have a recognizable chat content-block shape, mirrors real-array extraction for text-only and mixed media/text blocks, preserves arbitrary JSON strings, routes relevant string-body callers through the same helper where appropriate, and adds focused regression tests for vLLM text-only blocks, mixed blocks, invalid JSON fallback, and non-chat typed JSON arrays.

## Risks / Open Questions

- Closing this PR would discard a focused implementation candidate for the linked vLLM regression while current main still preserves JSON-stringified content-block arrays as raw strings.
- Merging the PR as-is could silently drop unrelated JSON string content such as `[{"type":"event","payload":"x"}]`, because the proposed `type`-only guard can route non-chat arrays into the text-block extractor and return `null`.
- The PR test plan still lacks live vLLM verification, and current main lacks a regression test for the stringified vLLM/OpenAI-compatible content-block shape.
