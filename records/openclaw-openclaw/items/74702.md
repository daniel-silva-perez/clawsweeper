## Summary

Keep open: the report matches a real SDK compatibility regression. `openclaw@2026.4.22` shipped `./plugin-sdk/zalouser`, `openclaw@2026.4.27` does not, and the published `@larksuite/openclaw-lark` versions named in the report still import that subpath.

## What This Changes

The issue asks OpenClaw to restore or provide a migration path for the missing `openclaw/plugin-sdk/zalouser` subpath so `@larksuite/openclaw-lark` can load after OpenClaw 2026.4.27.

## Best Possible Solution

Restore the shipped compatibility surface or add an explicit migration/compatibility gate so OpenClaw 2026.4.27+ does not fail at plugin load time for a package combination that worked against 2026.4.22. The cleanest end state is a deprecated, narrow `zalouser` facade that forwards only the already-generic helpers needed by existing packages, plus docs/tests/changelog that steer new plugins to generic subpaths.

## Repair Work Prompt

Fix the `openclaw/plugin-sdk/zalouser` compatibility regression reported for `@larksuite/openclaw-lark` after OpenClaw 2026.4.27.

Observable bug: `@larksuite/openclaw-lark@2026.4.7` and `2026.4.8` import `openclaw/plugin-sdk/zalouser`; `openclaw@2026.4.22` exported that subpath, but `openclaw@2026.4.27` and current main do not, so plugin loading fails with `Cannot find module .../plugin-sdk/root-alias.cjs/zalouser`.

Expected fix boundary: restore a deprecated, compatibility-only `openclaw/plugin-sdk/zalouser` subpath or an equivalent host-side compatibility path. Prefer forwarding to existing generic SDK helpers such as `command-auth`, `allow-from`, `channel-reply-pipeline`, etc.; do not create new broad policy or cross-plugin runtime coupling. Mark the subpath deprecated in docs so new plugins use generic SDK subpaths.

Likely files: add/restore `src/plugin-sdk/zalouser.ts`; update `scripts/lib/plugin-sdk-entrypoints.json`, `src/plugin-sdk/entrypoints.ts` classification, generated `package.json` exports, API baseline, SDK docs/subpath docs, targeted contract tests, and `CHANGELOG.md` with `Thanks @Goron01`. Do not change unrelated plugin loader behavior or external package metadata.

Validation: run targeted SDK/export tests first, then the plugin SDK export/API checks, and run `pnpm build` because this touches public SDK package exports and runtime aliasing. Use Testbox for broad changed-gate validation if the changed lane expands.
