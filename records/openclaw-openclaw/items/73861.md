## Summary

Keep this issue open. The report is new, concrete, and still matches current main’s observable `sessions_send` path: it starts a one-shot Gateway `agent` RPC, waits through `agent.wait`, and can surface `status: "timeout"` or `gateway closed (1000 normal closure)` without proving the multi-agent Windows failure is fixed. The related closed reports cover adjacent Gateway handshake/RPC failures, not this `sessions_send` routing/wait failure mode.

## Best Possible Solution

Keep this issue open and investigate the `sessions_send` path end to end on Windows with multiple local agents: session key/label resolution, `agent` RPC acceptance, active run registration, lifecycle/dedupe terminal snapshot delivery, Gateway close handling, and reply history readback. The best fix should add targeted diagnostics or recovery at the missed-signal point plus a focused regression test for repeated `sessions_send` waits or recoverable Gateway close handling.
