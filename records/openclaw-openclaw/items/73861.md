## Summary

Keep this issue open. Current main still routes `sessions_send` through a one-shot Gateway `agent` RPC followed by `agent.wait`, and the current code can still surface both `status: "timeout"` and `gateway closed (1000 normal closure)` without evidence that the reported Windows multi-agent failure is fixed. The related closed reports cover adjacent Gateway handshake, cron, and browser command failures, not this cross-session send/wait routing mode.

## Best Possible Solution

Keep this issue open and reproduce the `sessions_send` path end to end on Windows with multiple local agents. The best fix should identify the missed-signal point across session key/label resolution, `agent` RPC acceptance, active run registration, lifecycle/dedupe terminal snapshots, close-code handling, and reply history readback, then add either targeted recovery or diagnostics plus focused regression coverage for repeated `sessions_send` waits and recoverable Gateway closes.
