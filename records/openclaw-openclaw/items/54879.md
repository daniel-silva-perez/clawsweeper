## Summary

Keep open. Current main still has no browser `act` kind or CLI/API/docs surface for CDP-backed IME/native text insertion, and both existing text paths still go through Playwright `fill`/`type` or Chrome MCP `fill`. The report is recent, concrete, and the prior ClawSweeper comment already identified the same missing capability; I rechecked it against current main.

## What This Changes

Request an additive browser `act` action that inserts text through CDP/IME-native input for contenteditable rich editors where regular type/fill/paste paths are unreliable.

## Best Possible Solution

Add a documented, additive browser action such as `insertTextIme` that focuses the requested target and sends CDP `Input.insertText` for managed/raw CDP profiles, with schema, route normalization, CLI, docs, and regression coverage. Existing-session Chrome MCP profiles should either implement equivalent native insertion if the driver supports it or return a specific unsupported error that preserves the action contract.
