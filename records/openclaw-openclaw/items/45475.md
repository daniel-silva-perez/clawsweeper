## Summary

Keep open. Current main still routes Slack `message_changed` and `message_deleted` through subtype system-event handling that can enqueue a session, while the requested opt-in `channels.slack.ignoreSubtypes` config key is absent from the public type, strict schema, tests, and Slack docs. The related `thread_broadcast` issue is fixed, and the closed unmerged PRs for hard-coded edit/delete suppression do not satisfy this configurable request.

## What This Changes

Add a Slack `channels.slack.ignoreSubtypes` configuration key that drops configured edit/delete subtypes before system-event authorization, enqueue, and session wake.

## Best Possible Solution

Implement a Slack-owned opt-in subtype ignore list in the public config type and strict Zod schema, carry the resolved account config into the monitor context, and short-circuit configured edit/delete subtypes in the Slack event layer before system-event authorization or enqueue. Defaults should remain unchanged, Slack docs/config help should describe the key, and focused tests should prove both schema acceptance and ignored-event behavior.

## Reproduction Assessment

Yes. A high-confidence source/test reproduction exists: with `dmPolicy: "open"`, current tests expect a `message_changed` event to enqueue one system event, while the requested `ignoreSubtypes` key is absent from config validation and docs.

## Solution Assessment

Yes. The requested opt-in filter before `authorizeAndResolveSlackSystemEventContext(...)` and `enqueueSystemEvent(...)` is the narrowest maintainable path; prompt instructions are fragile, and unconditional edit/delete suppression is broader than the reported configuration need.
