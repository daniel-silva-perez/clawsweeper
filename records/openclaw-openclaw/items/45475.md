## Summary

Keep open. Current main still lacks a Slack `ignoreSubtypes` config surface, and Slack edit/delete subtypes still resolve to handlers that can enqueue system events with session keys. The adjacent `thread_broadcast` problem is fixed on main, but that does not satisfy this requested opt-in edit/delete suppression knob.

## What This Changes

Add an opt-in Slack channel/account config key that drops configured `message_changed` and `message_deleted` subtype events before they wake agent sessions.

## Best Possible Solution

Implement a Slack-owned config surface with account inheritance, strict schema/type/docs coverage, and event-layer filtering for configured edit/delete subtypes before system-event session routing. Keep defaults unchanged and preserve current Assistant `message_changed` and `thread_broadcast` behavior unless explicitly configured otherwise.

## Reproduction Assessment

Yes. Current source and tests provide a high-confidence path: a `message_changed` event with open DM policy resolves a subtype handler and calls `enqueueSystemEvent`, while the proposed `ignoreSubtypes` key is absent from the public type and strict schema surfaces.

## Solution Assessment

Yes. The narrowest maintainable fix is an opt-in Slack account/channel config key applied before `authorizeAndResolveSlackSystemEventContext(...)` and `enqueueSystemEvent(...)`; prompt instructions or a core-wide migration would be broader and less reliable.
