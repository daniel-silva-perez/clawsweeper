## Summary

Keep open. Current main still implements Slack `message_changed` and `message_deleted` as system events that can enqueue against a session, while the requested `channels.slack.ignoreSubtypes` config surface is absent from the public type, strict schema, tests, and docs. The adjacent `thread_broadcast` problem is already fixed, and a closed unmerged hard-coded suppression PR exists, but neither satisfies this issue's opt-in config request.

## What This Changes

Add a Slack `channels.slack.ignoreSubtypes` configuration key that drops configured edit/delete subtypes before system-event enqueue and session wake.

## Best Possible Solution

Implement a Slack-owned opt-in subtype ignore list in the public config type and strict schema, resolve it through the account config path, and short-circuit configured edit/delete subtypes in the Slack event layer before system-event authorization or enqueue. Keep defaults unchanged, document the key, and cover both accepted config and ignored-event behavior with focused Slack tests.

## Reproduction Assessment

Yes. A focused source/test reproduction exists: `message_changed` with `dmPolicy: "open"` currently resolves a subtype handler and enqueues a system event, while `ignoreSubtypes` is absent from config validation and docs.

## Solution Assessment

Yes. The requested opt-in config filter before `authorizeAndResolveSlackSystemEventContext(...)` and `enqueueSystemEvent(...)` is the narrowest maintainable fix; prompt-level instructions or unconditional edit/delete dropping are broader and less reliable.
