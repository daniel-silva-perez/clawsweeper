## Summary

Keep open. Current main still lacks a `channels.slack.ignoreSubtypes` config surface, and Slack `message_changed` / `message_deleted` events still resolve to subtype handlers that can enqueue system events with a session key. The related `thread_broadcast` behavior has been fixed on main, but that does not supersede this edit/delete suppression request.

## What This Changes

Add an opt-in Slack channel/account config key such as `channels.slack.ignoreSubtypes` so configured `message_changed` and `message_deleted` events are dropped before they wake an agent session.

## Best Possible Solution

Implement a narrow Slack-owned config surface with account inheritance, schema/type/docs coverage, and event-layer filtering for configured metadata-only edit/delete subtypes before system-event session routing. Keep defaults unchanged and document how this interacts with real inbound Slack Assistant messages.

## Reproduction Assessment

Yes. Current source and tests provide a high-confidence path: a `message_changed` event with open DM policy resolves a subtype handler and calls `enqueueSystemEvent`, while the proposed `ignoreSubtypes` key is absent from both type and strict schema surfaces.

## Solution Assessment

Yes. The best fix is a Slack plugin-local, opt-in account config key applied before `authorizeAndResolveSlackSystemEventContext(...)` / `enqueueSystemEvent(...)`; a core-wide migration or prompt workaround would be broader and less reliable.
