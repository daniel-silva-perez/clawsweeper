## Summary

Keep open. Current main still does not cover the reported bare tool-name-plus-JSON leak, so `exec{"command": ...}` can pass through the shared assistant visible-text path while XML, bracketed, downgraded, reasoning, memory, and provider-token scaffolding are stripped.

## What This Changes

The issue asks OpenClaw to stop exposing bare pseudo tool calls such as `exec{"command":"openclaw status","description":"Check OpenClaw version and status"}` in shell and Telegram replies.

## Best Possible Solution

Add a conservative shared visible-text sanitizer or repair stage for standalone bare tool-name-plus-balanced-JSON pseudo calls before assistant text reaches delivery/history, with regression coverage for final answer extraction and Telegram rendering while preserving literal examples in code/prose.

## Reproduction Assessment

Yes. The user provided concrete steps and current-main source inspection gives a deterministic reproduction path: `exec{"command": ...}` does not match the XML, bracketed block, downgraded tool-call, or diagnostic sanitizer paths, so it remains visible text.

## Solution Assessment

Yes for the recommended direction. The narrow maintainable fix belongs in the shared assistant visible-text path rather than Telegram-only delivery, because the same extracted assistant text feeds multiple user-visible surfaces.
