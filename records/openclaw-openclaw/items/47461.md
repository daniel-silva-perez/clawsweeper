## Summary

Keep open. Current main does not implement Telegram bot-avatar synchronization from OpenClaw identity/avatar changes. The Control UI assistant avatar override is explicitly browser-local, while the Telegram plugin has no current call path for Telegram profile-photo APIs. The request is actionable in the bundled Telegram plugin rather than closeable as already implemented or out of scope.

## Best Possible Solution

Keep this issue open and implement it, if maintainers want the feature, inside the bundled Telegram plugin. The clean path is an explicit opt-in sync from persisted OpenClaw agent identity or ui.assistant.avatar, not the browser-local Control UI override, using Telegram setMyProfilePhoto/removeMyProfilePhoto with image validation, multi-account handling, reload/startup hooks, tests, and Telegram docs updates.

## Risks / Open Questions

- Implementing this naively could upload browser-local data URLs or private local avatar files to Telegram without an explicit operator-controlled persistence/sync boundary.
- Multi-account Telegram setups need per-account behavior and clear opt-in semantics, otherwise one OpenClaw avatar change could update the wrong bot profile.
- The issue body suggests setChatPhoto, but that changes a chat photo; bot profile sync should use Telegram's bot profile-photo API when the dependency/runtime path supports it.
