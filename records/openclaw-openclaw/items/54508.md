## Summary

Keep this issue open. Current main has useful external-content markers, immediate `web_fetch` untrusted metadata, narrow user-message provenance, plugin approval hooks, and host exec approvals, but it still does not implement the requested end-to-end model: persistent taint through derived content, default content-derived execution denial, exec risk classes, or provenance-aware secret/outbound enforcement. The related issues are useful narrower slices, not proof that this umbrella is shipped or superseded.

## Best Possible Solution

Keep this issue open as the core security architecture umbrella unless maintainers explicitly split it into canonical implementation issues. The best path is structured provenance/taint metadata for external inputs and tool outputs, preservation through summaries and persistence, stable action/risk/provenance fields in `before_tool_call` and approval payloads, and core enforcement for content-derived exec, secret access, and outbound actions, coordinated with narrower follow-ups such as #54373, #48519, #48503, #41140, #47492, and #53185.

- A future fix likely needs careful public-contract design because `before_tool_call`, message hooks, approval payloads, transcript persistence, and plugin hook events are extension-facing surfaces.
