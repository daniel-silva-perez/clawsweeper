## Summary

Keep open. Current main still has the Discord presence cache and listener, but it does not implement this PR's intended behavior: `MsgContext` has no `SenderClientStatus`, the inbound conversation metadata has no `sender_client_status`, and Discord inbound processing does not read `getPresence()` or pass client status into the finalized context. The PR remains a focused implementation candidate rather than obsolete cleanup.

## Best Possible Solution

Review and update the PR rather than close it: keep the behavior in the Discord plugin owner path, read `getPresence(accountId, sender.id)` during inbound processing, add a typed optional `SenderClientStatus` to the context, render a non-empty `sender_client_status` only in the user-role conversation metadata, and add targeted tests for presence present/absent plus system-prompt cache stability. If maintainers accept the privacy tradeoff, this is a narrow Discord-channel feature rather than a core refactor.
