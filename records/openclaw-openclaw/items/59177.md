## Summary

Keep open. Current main still has the Discord presence cache and listener, but it does not expose sender client status in inbound conversation metadata; the PR is directionally useful but is now dirty against current main after the Discord inbound context refactor.

## What This Changes

The PR adds an optional Discord `SenderClientStatus` context field, reads cached `client_status` during inbound processing, and renders `sender_client_status` in the user-role conversation metadata block.

## Best Possible Solution

Adapt the feature into the current Discord inbound context builder, keep the dynamic status in the user-role conversation metadata rather than the system prompt, omit the field when no platform status is present, and add focused tests for rendering and cache-stability behavior.

## Reproduction Assessment

Yes for source-level verification: with cached Discord presence available, current main never reads `getPresence` while constructing the inbound `MsgContext`, and `buildInboundUserContextPrefix` has no `sender_client_status` key to render. A live Discord reproduction would require enabling the Presence Intent and sending messages from different clients.

## Solution Assessment

Partially. The PR's use of the existing presence cache and user-role metadata is the right direction, but the current branch is no longer the best merge shape because main moved inbound context construction into `extensions/discord/src/monitor/message-handler.context.ts` and should also guard the empty-object case.
