## Summary

Keep open. Current main still uses WhatsApp receive-time inbound debouncing, while `messageTimestamp` is preserved as metadata and append-age input rather than as a coalescing key, late-arrival window, config option, or tested batching rule.

## What This Changes

The issue requests batching WhatsApp inbound messages by WhatsApp `messageTimestamp` so forward/comment or rapid same-timestamp Baileys deliveries can arrive as one agent turn even when received several seconds apart.

## Best Possible Solution

Add a WhatsApp-owned, backwards-compatible timestamp coalescing path for text-only inbound messages from the same account, conversation, and sender when WhatsApp server timestamps are identical or within a very small documented tolerance. Keep media, location, reply-context, access-control, self-message, and control-command behavior on their existing immediate or guarded paths.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is two WhatsApp `notify` text messages from the same account/chat/sender with the same `messageTimestamp`, where the second is delivered after `debounceMs`; current main flushes the first by receive-time `setTimeout` before the second can join.

## Solution Assessment

Yes. The requested direction is the narrow maintainable fix if implemented inside the WhatsApp plugin as opt-in or otherwise backwards-compatible timestamp coalescing; changing shared inbound debounce semantics for every channel would be broader than this Baileys-specific delivery artifact.
