## Summary

Keep open. Current main still preserves WhatsApp/Baileys `messageTimestamp`, but inbound batching is still driven by the shared receive-time debounce timer keyed by account, conversation, and sender. Config, docs, and tests still describe only `debounceMs` receive-window batching, so delayed same-server-timestamp deliveries can still split into separate agent turns.

## Best Possible Solution

Keep this issue open. The best path is a WhatsApp-owned timestamp coalescing implementation, or a small additive shared inbound debouncer seam if maintainers prefer reuse, that groups delayed messages from the same account/conversation/sender when their WhatsApp server timestamps are identical or within a documented small tolerance. It should preserve current defaults, avoid increasing latency for unrelated messages, add fake-timer regression coverage for delayed same-timestamp deliveries, and document any new config surface.
