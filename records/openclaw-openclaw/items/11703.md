## Summary

Keep open. Current main still batches WhatsApp inbound text through the shared receive-time debounce timer; WhatsApp `messageTimestamp` is preserved as message metadata, but it is not used as a coalescing key, grouping window, or config-controlled timestamp bucket. The existing maintainer and ClawSweeper comments remain accurate, and this is narrow enough for a focused WhatsApp repair PR.

## What This Changes

The issue requests WhatsApp inbound batching based on Baileys `messageTimestamp` so simultaneous forward-and-comment or rapid messages arrive as one agent turn even when Baileys receive delivery is delayed.

## Best Possible Solution

Add a backwards-compatible WhatsApp-owned timestamp coalescing path for text-only inbound messages from the same account, conversation, and sender when their WhatsApp server timestamps are identical or within a very small documented tolerance, with config/schema/docs and fake-timer coverage for delayed Baileys delivery.
