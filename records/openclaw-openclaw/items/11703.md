## Summary

Keep open. Current main still routes WhatsApp inbound batching through the shared receive-time debouncer keyed by account, conversation, and sender; `messageTimestamp` is preserved as message metadata and used for append-age filtering, but not as a coalescing key, late-arrival timer, config option, doc surface, or regression-tested batching rule.

## What This Changes

The issue requests WhatsApp inbound batching that can group same-chat, same-sender messages by Baileys/WhatsApp `messageTimestamp` instead of only by receive-time debounce.

## Best Possible Solution

Add a WhatsApp-owned, backwards-compatible timestamp coalescing path for text-only inbound messages from the same account, conversation, and sender when WhatsApp server timestamps are identical or within a small documented tolerance. Keep the shared core debouncer generic unless a small provider-agnostic hook is necessary, and preserve immediate flushing for media, location, reply-context, access-control, self-echo, and control-command paths.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is two text-only WhatsApp `notify` messages from the same account/chat/sender with the same `messageTimestamp`, where the second arrives after `debounceMs`; current main flushes the first via receive-time `setTimeout` before the second can join.

## Solution Assessment

Yes. The requested direction is the narrow maintainable shape if implemented inside the WhatsApp plugin as opt-in or otherwise backwards-compatible timestamp coalescing; changing shared debounce semantics globally would be broader than the Baileys/WhatsApp delivery artifact described here.
