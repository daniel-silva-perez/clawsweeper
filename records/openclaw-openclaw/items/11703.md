## Summary

Keep open. Current main still routes WhatsApp inbound batching through the receive-time debouncer keyed by account, conversation, and sender; WhatsApp `messageTimestamp` is preserved as metadata but is not used as a coalescing key, late-arrival window, config option, or documented behavior. The existing maintainer and bot review comments remain accurate against the current checkout.

## What This Changes

The issue requests batching WhatsApp inbound messages by WhatsApp server `messageTimestamp` rather than only by receive-time debounce.

## Best Possible Solution

Add a WhatsApp-owned, backwards-compatible timestamp coalescing path for text-only inbound messages from the same account, conversation, and sender when WhatsApp server timestamps are identical or within a very small documented tolerance. Preserve existing media, location, reply-context, access-control, self-message, and control-command behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is two WhatsApp `notify` text messages from the same account/chat/sender with the same `messageTimestamp`, where the second is delivered after `debounceMs`; current main flushes the first by receive-time `setTimeout` before the second can join.

## Solution Assessment

Yes. The narrow maintainable direction is WhatsApp-owned timestamp coalescing, possibly using the existing debouncer extension points, rather than changing shared inbound debounce semantics for every channel.
