## Summary

Keep open. Current main still batches WhatsApp inbound messages through the shared receive-time debounce path; Baileys `messageTimestamp` is preserved as message metadata but is not used as a coalescing key, timer, or config-controlled grouping rule. The prior maintainer review comment remains accurate against the current checkout, and this is narrow enough to route as a focused WhatsApp repair candidate.

## What This Changes

The issue requests WhatsApp inbound batching based on Baileys `messageTimestamp` rather than only message receive time.

## Best Possible Solution

Add a WhatsApp-owned, backwards-compatible timestamp coalescing path for text-only inbound messages from the same account, conversation, and sender when their WhatsApp server timestamps are identical or within a very small documented tolerance. Preserve current media, location, reply-context, access-control, self-message, and control-command flushing behavior, and document any new config surface.
