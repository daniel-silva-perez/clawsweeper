## Summary

Keep open. Current main still preserves WhatsApp `messageTimestamp` as metadata but batches WhatsApp inbound text through the shared receive-time debounce timer, so messages delivered several seconds apart with the same WhatsApp server timestamp still become separate agent turns.

## What This Changes

The issue requests batching WhatsApp inbound messages by WhatsApp server `messageTimestamp` rather than only by receive-time debounce so forwarded/comment bursts arrive as one agent turn.

## Best Possible Solution

Add a WhatsApp-owned, backwards-compatible timestamp coalescing path for text-only inbound messages from the same account, conversation, and sender when their WhatsApp server timestamps are identical or within a very small documented tolerance, while preserving immediate flush behavior for media, locations, reply-context-sensitive messages, access-control failures, self-message handling, and control commands.
