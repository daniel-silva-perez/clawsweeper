## Summary

Keep open. Current main still preserves WhatsApp `messageTimestamp` as inbound metadata, but batching is still driven by the shared receive-time debounce key and timer, with no timestamp bucket or config-controlled timestamp coalescing path. The existing maintainer review remains accurate against the current checkout, and the remaining work is narrow enough for a focused WhatsApp fix PR.

## What This Changes

The issue requests batching WhatsApp inbound messages with identical or very close Baileys `messageTimestamp` values into one agent turn instead of relying only on receive-time debounce.

## Best Possible Solution

Add a WhatsApp-owned, backwards-compatible timestamp coalescing path for text-only inbound messages from the same account, conversation, and sender when their WhatsApp server timestamps are identical or within a very small documented tolerance. Preserve current media, location, reply-context, access-control, self-message, offline append, and control-command flushing behavior, and document any new config surface.
