## Summary

Keep open. Current main still preserves WhatsApp `messageTimestamp` as metadata but batches inbound WhatsApp messages through the shared receive-time debounce path, so the reported forward-plus-comment delivery gap is still a real WhatsApp-owned batching gap.

## What This Changes

The issue requests batching WhatsApp messages by WhatsApp server `messageTimestamp`, instead of only by the time Baileys delivers each message to OpenClaw.

## Best Possible Solution

Add a WhatsApp-owned, backwards-compatible timestamp coalescing path for text-only inbound messages from the same account, conversation, and sender when their WhatsApp server timestamps are identical or within a very small documented tolerance, while preserving current media, location, reply-context, access-control, self-message, and control-command flushing behavior.
