## Summary

Keep open. Current main still uses WhatsApp `messageTimestamp` only as message metadata and for access/offline handling, while inbound batching remains keyed by account, conversation, and sender with a receive-time debounce timer. The prior maintainer/bot reviews remain accurate on the current checkout, and this is a narrow WhatsApp-owned fix candidate rather than a stale or already-implemented request.

## What This Changes

The issue requests WhatsApp inbound batching based on Baileys `messageTimestamp` instead of only receive-time debounce so simultaneous forwarded/comment messages arrive in one agent turn.

## Best Possible Solution

Add a WhatsApp-owned, backwards-compatible timestamp coalescing path for text-only inbound messages from the same account, conversation, and sender when their WhatsApp server timestamps are identical or within a very small documented tolerance, with config/docs/tests covering the new behavior.
