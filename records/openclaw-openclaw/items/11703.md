## Summary

Keep open. Current main still preserves WhatsApp `messageTimestamp` as inbound metadata, but batching remains the shared receive-time debounce keyed by account, conversation, and sender; there is no timestamp coalescing key, timer, config surface, doc, or regression test for late Baileys delivery with identical server timestamps. The existing maintainer and bot review comments remain accurate against the current checkout, and this is narrow enough for a focused WhatsApp repair PR.

## What This Changes

The issue requests WhatsApp inbound batching that can group same-chat, same-sender messages by WhatsApp server `messageTimestamp` instead of only by receive-time debounce.

## Best Possible Solution

Add a WhatsApp-owned, backwards-compatible timestamp coalescing path for text-only inbound messages from the same account, conversation, and sender when WhatsApp server timestamps are identical or within a small documented tolerance. Keep the shared core debouncer generic unless a small provider-agnostic hook is necessary, preserve immediate flushing for media/location/reply-context/control-command paths, and document any new WhatsApp config surface.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is two text-only WhatsApp `notify` messages from the same account/chat/sender with the same `messageTimestamp`, where the second arrives after `debounceMs`; current main flushes the first via receive-time `setTimeout` before the second can join.

## Solution Assessment

Yes, the requested direction is the right narrow shape if implemented inside the WhatsApp plugin as opt-in or otherwise backwards-compatible timestamp coalescing. A global change to shared debounce semantics would be broader than the Baileys/WhatsApp delivery artifact described here.
