## Summary

Keep open. Current main still preserves WhatsApp/Baileys `messageTimestamp`, but inbound batching is still driven by the shared receive-time debounce timer keyed by account, conversation, and sender. No config, docs, or located regression test implements timestamp-aware WhatsApp coalescing, so delayed same-server-timestamp deliveries can still split into separate agent turns.

## What This Changes

Request timestamp-aware WhatsApp inbound batching so delayed messages with the same or nearby WhatsApp server `messageTimestamp` are grouped into one agent turn instead of relying only on receive-time debounce.

## Best Possible Solution

Keep this issue open and implement a narrow WhatsApp-owned timestamp coalescing path, or a minimal additive shared debouncer seam if maintainers prefer reuse. It should group text-only messages from the same account/conversation/sender when their WhatsApp server timestamps are identical or within a documented small tolerance, preserve current defaults and media/control-command flush behavior, add fake-timer regression coverage for delayed same-timestamp delivery, and document any new config surface.

## Repair Work Prompt

Implement a narrow fix for https://github.com/openclaw/openclaw/issues/11703. Current WhatsApp inbound batching preserves Baileys `messageTimestamp` as the envelope timestamp but still debounces only by receive-time using the shared key account/conversation/sender. Add timestamp-aware coalescing for WhatsApp text messages from the same account/conversation/sender when Baileys delivers multiple messages with identical or very close WhatsApp server timestamps after the normal debounce window. Prefer a WhatsApp-owned implementation in `extensions/whatsapp/src/inbound/monitor.ts` unless a small additive shared debouncer seam is clearly cleaner. Preserve current defaults and avoid increasing latency for unrelated messages; if a new config option is added, make it opt-in or otherwise backward-compatible, update `src/config/zod-schema.providers-whatsapp.ts`, `src/config/types.whatsapp.ts`, config tests/metadata as required, and document it in `docs/channels/whatsapp.md` and/or `docs/concepts/messages.md`. Keep media, location, quoted-reply, and control-command flush semantics intact. Add fake-timer regression coverage for two same-chat/same-sender WhatsApp messages with the same `messageTimestamp` arriving after the normal receive debounce would have flushed, proving they are delivered as one batched agent turn. Add a user-facing changelog entry with `Thanks @VsevolodUstinov`. Do not move WhatsApp-specific policy into core unless the shared seam stays generic and fully backward-compatible.
