## Summary

Keep open. Current main still routes WhatsApp inbound batching through the shared receive-time debounce window, while `messageTimestamp` is only preserved as message metadata; there is no timestamp-aware WhatsApp coalescing config, docs, or regression coverage. The prior review comment still matches the current checkout.

## What This Changes

Request timestamp-aware WhatsApp inbound batching so delayed Baileys deliveries with identical or very close `messageTimestamp` values arrive as one agent turn instead of being split by receive-time debounce expiry.

## Best Possible Solution

Add a WhatsApp-owned, backwards-compatible timestamp coalescing path for text-only messages from the same account, conversation, and sender when their WhatsApp server timestamps are identical or within a documented tiny tolerance. Preserve current media, location, reply-context, and control-command flush behavior, and document any new config surface.
