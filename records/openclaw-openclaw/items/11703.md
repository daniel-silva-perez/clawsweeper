## Summary

Current main still does not implement timestamp-aware WhatsApp batching. WhatsApp inbound messages are debounced by a receive-time timer keyed only by account, conversation, and sender; `messageTimestamp` is preserved as the inbound `timestamp` and used for access-control/history checks, but it is not part of the batching key or a delayed coalescing window. The config/schema/docs expose only the existing debounce knobs, and the related #19716 timestamp-context PR was closed unmerged and did not address batching.

## Best Possible Solution

Keep this issue open. The best fix is a WhatsApp-owned timestamp coalescing path, or an additive shared inbound debouncer seam if maintainers want reuse, that groups by account/conversation/sender plus same-or-near `messageTimestamp`, tolerates delayed Baileys delivery beyond the receive-time debounce window, adds fake-timer regression coverage for delayed same-timestamp deliveries, and documents any new config surface.
