## Summary

Keep open. Current main still batches WhatsApp inbound messages only by the receive-time debounce buffer keyed by account, conversation, and sender. WhatsApp `messageTimestamp` is preserved as the inbound message timestamp and used for access/offline checks, but it is not used to hold delayed same-timestamp Baileys deliveries together, and there is no config/docs surface for timestamp coalescing.

## Best Possible Solution

Keep this issue open. The best path is a WhatsApp-owned timestamp coalescing design, or an additive shared inbound debouncer seam if maintainers want reuse, that groups by account/conversation/sender plus same-or-near WhatsApp `messageTimestamp`, tolerates delayed Baileys delivery beyond the receive-time debounce window, adds fake-timer regression coverage for delayed same-timestamp deliveries, and documents any new config surface.
