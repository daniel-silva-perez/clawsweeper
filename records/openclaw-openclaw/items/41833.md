## Summary

Keep this issue open. Current main has strong adjacent pieces: cron scheduling, cron run history, task records, Task Flow, standing orders, compaction preservation of promised follow-ups, and a reminder guard that warns when a reply says it will remind/follow up but did not create a cron job. It still does not provide the requested native commitment primitive with durable commitment records, `openclaw commit add/list/done`, completion reconciliation, overdue escalation, or startup/session surfacing for uncompleted commitments. The related PENDING.md workspace-pattern reports are not canonical duplicates for this core runtime/API request.

## Best Possible Solution

Design a native commitment layer above cron/tasks/Task Flow: durable commitment records with due time, owner/channel target, optional linked cron/task/flow run, `add/list/done/cancel/status` operations, reconciliation from cron/task outcomes, overdue or failed-run escalation, and startup/session surfacing for uncompleted commitments. Reuse the existing scheduler and task ledger rather than building a parallel runner, and make natural-language capture explicit enough to avoid accidental durable promises.

## Risks / Open Questions

- Natural-language commitment capture needs product policy for false positives, user consent, and which conversational statements become durable state.
- Overdue escalation needs clear routing, privacy, authority, and failure semantics across channels, sessions, cron, tasks, and Task Flow.
- A phrase-based reminder guard is useful but could create a false sense that broader commitments are handled; it only warns on narrow reminder-like phrasing and does not track completion.
