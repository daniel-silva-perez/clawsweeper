## Summary

Current main still has adjacent automation pieces but not the requested native commitment primitive. Cron can schedule and record runs, tasks track detached work, Task Flow tracks multi-step flows, standing orders document discipline, and the reminder guard warns for narrow reminder-like promises without a cron add. None of those provide durable commitment records, `openclaw commit add/list/done`, completion reconciliation, overdue escalation, or startup/session surfacing for unresolved conversational commitments.

## Best Possible Solution

Keep this issue open and design a native commitment layer above the existing scheduler and task ledger: durable commitment records with due time, owner/channel target, optional linked cron/task/flow run, `add/list/done/cancel/status` operations, reconciliation from cron/task outcomes, overdue or failed-run escalation, and startup/session surfacing for unresolved commitments. Reuse cron/tasks/Task Flow where possible instead of building a parallel runner, and make natural-language capture explicit enough to avoid accidental durable state.

## Risks / Open Questions

- Natural-language commitment capture needs a clear product policy to avoid false positives and accidental durable promises.
- Overdue escalation crosses channels, sessions, cron, tasks, Task Flow, privacy, and authority boundaries, so routing and failure semantics need explicit design.
- The current reminder guard is useful but narrow; treating it as the full solution would leave broader deferred-task commitments uncovered.
