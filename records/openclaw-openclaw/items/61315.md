## Summary

Keep this PR open. Current main still does not implement the requested usage aggregation behavior, and the PR remains directionally useful, but the branch still needs maintainer rework because the no-filter overview can mix full-range aggregates with limited-session diagnostics and the limit-warning fallback regresses when overview metadata is absent.

## What This Changes

The PR updates gateway sessions.usage aggregation/types and Control UI usage rendering/helpers/tests to add matched-session overview metadata, token-aware channel/agent ranking, token-primary Top Agents/Top Channels cards, and regression coverage.

## Best Possible Solution

Keep the useful direction but rework the branch into a consistent usage contract: limit should trim only returned rows, overview totals/aggregates/diagnostics should all use the same documented matched-session scope, old responses without overview should preserve the existing cap warning behavior, zero-cost channel/agent rankings should remain visible by token signal, and full-range summary reads should be bounded or cached. Coordinate channel attribution with #52521 rather than mixing that separate behavior into this PR.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to seed more matched sessions than the requested limit with zero-cost token-heavy usage, then compare sessions.usage totals/aggregates and the unfiltered Control UI overview against the returned session subset; the current main code inspection shows the limited-scope behavior, and the PR tests cover part of that path.

## Solution Assessment

No. The PR's core direction is maintainable, but the current patch is not the best final form because it still mixes full overview aggregates with limited-session peak/mosaic diagnostics and regresses the no-overview limit-warning fallback; the safer fix is an explicit overview-scope contract with compatible UI fallbacks and bounded aggregation I/O.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep overview diagnostics on the same scope:** `ui/src/ui/views/usage.ts:306-310`
  - body: When `useGatewayOverview` is true, this switches the no-filter cards to gateway-wide aggregates, but the same render path still passes `aggregateSessions` into peak-hour and mosaic diagnostics. With `matchedSessions > returnedSessions`, the overview combines full-range totals/error days with peak-hour and mosaic data from only the limited returned rows, so the diagnostics can point at the wrong hours. Feed those diagnostics from an overview-scoped source or avoid mixing them with full-range aggregates.
  - confidence: 0.84
- **[P2] Preserve the limit warning without overview data:** `ui/src/ui/app-render-usage-tab.ts:26-28`
  - body: When `overview` is absent, this expression falls back to `sessions.length > sessions.length`, which is always false. Mixed-version or cached responses without overview metadata will no longer show the previous `>= 1000` cap warning even when the returned list is likely truncated, making capped usage data appear complete.
  - confidence: 0.9
