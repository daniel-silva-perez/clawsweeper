## Summary

Keep this PR open. Current main still falls back to config-only `channels status` output for any `callGateway(...)` failure, including `--probe`, so the PR’s intended behavior is not implemented. The provided related-item context points to different `channels status --probe` work, not a canonical replacement for this failure-handling change. Security review of the provided diff found no supply-chain or permissions concern because it only touches the command path and a focused unit test.

## Best Possible Solution

Review and land a current-main version of this PR, ideally keeping the focused regression test and addressing the minor bot-review cleanup about unnecessary mock setup. If maintainers want a narrower distinction between unreachable gateway and post-gateway probe failure, that policy should be encoded in `src/commands/channels/status.ts`; otherwise the proposed `opts.probe` rethrow matches the existing documented expectation that probe/deep status requires a reachable gateway.
