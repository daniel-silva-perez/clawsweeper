## Summary

Keep open. Current main still has the permissive bearer fallback defaults described by the report: HTTPS enforcement is opt-in, the attachment predicate is optional, and an auth retry can add `Authorization: Bearer ...` after a 401/403. The linked #43005 is related network-header hardening, but it covers cross-origin redirect header retention, not this fetch-auth fallback behavior.

## Best Possible Solution

Keep this issue open as a focused SDK/security hardening item. The best path is to make bearer fallback fail closed before any public exposure: require HTTPS by default with an explicit local-development opt-out, require an allowlist or `shouldAttachAuth` decision before attaching tokens, add regression tests for HTTP and missing predicates, and document the intended `fetch-auth` export status if it becomes a supported SDK helper.

## Risks / Open Questions

- Changing the helper to fail closed may be breaking for any internal or unpublished consumers that rely on permissive defaults.
- Because the helper is not currently documented as a public SDK subpath and no production callers were found, maintainers should decide whether to harden it, remove it, or publish it intentionally with safe defaults.
