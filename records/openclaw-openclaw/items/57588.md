## Summary

Keep open. Current main still matches the reported bearer-token fallback behavior: HTTPS enforcement is opt-in, the URL attachment predicate is optional, and the retry path can attach `Authorization: Bearer ...` after a 401/403. The linked #43005 remains related network-header hardening, but it covers cross-origin redirect header retention rather than this fallback retry helper.

## Best Possible Solution

Keep this as a focused SDK/security hardening issue. The best path is to make `fetchWithBearerAuthScopeFallback` fail closed by default: require HTTPS unless explicitly opted out, require an allowlist or `shouldAttachAuth` decision before attaching tokens, add regression tests for HTTP and missing predicates, and decide whether `fetch-auth` should remain internal or become a documented SDK helper with safe defaults.

## Risks / Open Questions

- The helper can still attach bearer tokens to arbitrary URLs for any internal or unpublished caller that omits `requireHttps` and `shouldAttachAuth`.
- Changing the defaults to fail closed may break callers that intentionally relied on permissive fallback behavior, so the fix should be explicit and covered by regression tests.
