## Summary

Keep this issue open. Current main still supports reply threading through assistant output tags, reply payload metadata, channel replyToMode policy, and channel-specific settings, but it does not implement the requested global inbound user directive that strips trailing phrases such as "reply in thread" or "rit" and forces the response to target the current inbound message. The related implementation PR was closed unmerged, and its review feedback points to fixable implementation details rather than the feature being obsolete.

## What This Changes

Add a built-in inbound directive that detects trailing reply-in-thread phrases, strips them before the agent sees the message, and forces the outbound reply into the current message thread.

## Best Possible Solution

Implement a narrow shared auto-reply inbound directive in core: detect deliberate trailing user phrases after inbound context finalization, strip the phrase from all agent and command body surfaces, carry a per-turn flag on MsgContext, and translate that flag into replyToCurrent/replyToId through the existing payload threading pipeline. Keep channel-specific thread settings like Feishu replyInThread separate, and document the new user-facing shortcut if it ships.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: send or unit-test a finalized inbound context with a trailing phrase like "rit"; current main has no parser or MsgContext flag, so the phrase remains normal prompt/command text and cannot force replyToCurrent unless the model emits an outbound tag.

## Solution Assessment

Yes, with one constraint: the maintainable solution is a small core directive feeding the existing reply payload pipeline, not per-channel special casing. The safer default is to omit the bare "thread" trigger or gate it behind explicit configuration because silent stripping can misinterpret ordinary chat text.
