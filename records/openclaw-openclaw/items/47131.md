## Summary

Keep this issue open. Current main still has outbound reply tags, implicit reply-threading policy, and channel-specific thread settings, but it does not implement the requested inbound trailing-phrase directive that strips text such as `reply in thread` or `rit` and forces the response to target the current inbound message. The only linked implementation PR is closed and unmerged, with review feedback pointing to fixable implementation details rather than obsolete work.

## What This Changes

Add a built-in inbound directive that detects trailing reply-in-thread phrases, strips them before the agent and command parser see the message, and forces the outbound reply into the current message thread.

## Best Possible Solution

Implement a narrow shared auto-reply directive in core: detect deliberate trailing user phrases, strip them consistently from all inbound text surfaces, carry a typed per-turn flag on `MsgContext`, and translate that flag through the existing reply payload threading path. Keep Feishu and other channel-specific thread settings separate, and update user-facing docs/changelog when the shortcut ships.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: send or unit-test a finalized inbound context with text like `hello rit`; current main has no parser or context flag, so the phrase remains normal prompt/command text and cannot force reply-to-current unless outbound policy or the model does it independently.

## Solution Assessment

Yes, with constraints. The narrow maintainable fix is a shared inbound directive feeding existing reply payload threading, while avoiding the broad bare `thread` default and preserving channel reply policies.
