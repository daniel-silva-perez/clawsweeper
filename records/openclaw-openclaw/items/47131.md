## Summary

Keep this issue open. Current main has outbound reply tags and config-driven reply threading, but it does not implement the requested inbound suffix directive that strips phrases like `reply in thread` or `rit` before the model sees them and forces the outbound reply to target the inbound message. The related implementation PR #47297 was closed unmerged, so it is not a superseding fix.

## Best Possible Solution

Add a small core auto-reply inbound directive path: parse configured trailing phrases from the clean inbound command/agent body, strip them before prompt construction, carry a per-turn `replyToCurrent` override into final and streaming reply payload construction, and cover `replyToMode: off`, `batched`, and normal threaded channel routing with focused tests. Reuse the existing outbound `replyToCurrent`/`replyToId` plumbing rather than adding channel-specific behavior.

## Risks / Open Questions

- Trigger phrases like `thread` are broad, so an implementation should require end-of-message matching and include false-positive tests.
- The feature needs an explicit policy decision about whether a user directive should bypass `replyToMode: off`, matching the existing explicit outbound tag behavior only when intended.
