## Summary

Current main still does not implement this PR’s core change: chat-bash forwards some `tools.exec` defaults into `createExecTool`, but not `host`, `security`, or `ask`. The PR addresses a real security-policy consistency gap, but it should not be closed as implemented, and its current routing change still needs maintainer/contributor attention because it narrows valid `!` shell aliases.

## Best Possible Solution

Keep this PR open and ask for a focused update: retain the `host`/`security`/`ask` inheritance in `handleBashChatCommand`, add regression coverage proving chat-bash receives configured `tools.exec` policy defaults, and rework the markdown-image exclusion so it does not block valid punctuation-prefixed `!` shell commands.
