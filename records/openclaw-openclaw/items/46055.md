## Summary

Keep open. Current main still lacks a Telegram waiting reaction and per-message queued callback, and this PR is useful, but it is not merge-ready because the new runtime `waiting` emoji key is not added to the public config/schema/generated contract and the branch needs a current-main rebase.

## What This Changes

The PR adds a Telegram queued-waiting status reaction by threading a per-message `onQueued` callback through queue, embedded-runner, auto-reply, and Telegram dispatch code, with status-reaction tests and Telegram reaction 429 backoff.

## Best Possible Solution

Land a rebased version that keeps the per-message callback design, fires the queued notification from both session and global lanes with a once guard, preserves account+chat 429 backoff, and updates the config type, strict Zod schema, generated schema/help, docs, and focused tests for the new `waiting` status key.

## Reproduction Assessment

Yes. Send a second Telegram message in the same session while a first run occupies the session lane; current main sets `thinking` before execution and has no `onQueued` waiting-state path. For the review finding, a `messages.statusReactions.emojis.waiting` override would be rejected by the current strict config schema because that key is absent.

## Solution Assessment

No, not yet. The callback-based design is the narrow maintainable direction and the PR head addresses earlier session-lane and 429-backoff comments, but it still misses the public config/generated surfaces and needs a current-main rebase before merge.
