## Summary

Keep open. Current main does not implement the Telegram waiting reaction or queue callback, and the contributor branch contains useful targeted work, but it is not merge-ready because the new runtime `waiting` emoji is not added to the public status-reaction config contract or generated schema/help surfaces.

## What This Changes

The branch adds a Telegram waiting reaction for queued runs by threading a per-message `onQueued` signal through queue and reply execution, extending status-reaction controller/variant tests, and adding Telegram reaction 429 backoff.

## Best Possible Solution

Land a rebased version that keeps the per-message callback design, fires the queued notification from both session and global lanes with a once guard, preserves the account+chat 429 backoff, and updates the current config type, strict Zod schema, generated schema/help, docs, and focused tests for the new `waiting` status key.

## Reproduction Assessment

Yes. Send a second Telegram message in the same session while a first run occupies the session lane; current main sets thinking before execution and has no `onQueued` waiting state. For the review finding, a `messages.statusReactions.emojis.waiting` override would be a natural companion to the new runtime key, but the current strict schema would still reject it unless the contract is updated.

## Solution Assessment

No, not yet. The callback-based direction is the narrow maintainable fix and the PR head addresses the earlier session-lane and 429-backoff comments, but it still misses the public config/generated surfaces and needs a current-main rebase before merge.
