## Summary

Keep open: current main still lacks the Telegram waiting-reaction/onQueued behavior, and this contributor PR is useful but not merge-ready because the new runtime waiting emoji key is not added to the strict config contract or generated config help. The earlier session-lane, emoji-collision, typing-lifecycle, and Telegram 429 backoff concerns have been addressed in later PR commits, so the remaining path is a maintainer/author rebase and contract cleanup rather than closure.

## What This Changes

The PR adds a Telegram waiting status reaction, threads a per-message onQueued callback through queue and reply execution paths, adjusts Telegram reaction variants, adds status-reaction 429 backoff, and extends status-reaction tests.

## Best Possible Solution

Land a rebased version that keeps the per-message callback design, fires queued notification from both session and global lanes with a once guard, preserves current lane-timeout behavior, keeps the Telegram waiting-to-thinking transition and account+chat 429 backoff, and updates the config type, strict Zod schema, generated schema/help, and relevant docs for the new waiting status key.

## Reproduction Assessment

Yes. The source-level reproduction path is clear: send a second Telegram message in the same session while a first run occupies the session lane; current main has no onQueued callback or waiting state, so the reaction can move from ack to thinking without a queued-wait indicator.

## Solution Assessment

No, not as currently proposed. The callback-based direction is the narrowest maintainable approach, and the PR has addressed earlier review defects, but it still needs config-contract coverage and a current-main rebase before it is the best mergeable solution.
