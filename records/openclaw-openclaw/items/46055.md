## Summary

Keep this contributor PR open: the requested Telegram queue-waiting UX is still absent from current main, and the PR contains useful targeted work, but it is not merge-ready because the new runtime `waiting` status is not added to the public config contract or generated config surfaces.

## What This Changes

The PR adds a Telegram waiting status reaction for queued messages by threading a per-message `onQueued` callback through queue/reply execution, extending status-reaction controller and Telegram variants, adding Telegram dispatch wiring, adding status-reaction 429 backoff, and extending tests.

## Best Possible Solution

Land a rebased version that keeps the per-message callback design, fires queued notification from both session and global lanes with a once guard, preserves current lane-timeout behavior, keeps the Telegram waiting-to-thinking transition and account+chat 429 backoff, and updates the config type, strict Zod schema, generated schema/help, and relevant docs/tests for the new waiting status key.

## Reproduction Assessment

Yes. For the user-visible bug, send a second Telegram message in the same session while a first run occupies the session lane; current main can show thinking while the message is still queued because there is no onQueued/waiting state. For the review finding, configure `messages.statusReactions.emojis.waiting` after the PR change and the strict current config contract would still reject that key.

## Solution Assessment

No, not yet. The callback-based direction is the narrow maintainable approach and the PR has addressed earlier session-lane and 429-backoff defects, but the current proposal still misses the config contract/generated surfaces and needs a current-main rebase before merge.
