## Summary

Keep open: current main still lacks the Telegram waiting reaction and per-message queued callback, so the PR is not obsolete. The branch is useful but not merge-ready because the new `waiting` status is added at runtime without updating the strict config contract, generated schema/help, and docs surfaces; it also needs a current-main rebase before validation.

## What This Changes

The PR adds a Telegram waiting status reaction for queued messages by threading an `onQueued` callback through queue/reply execution, extending the status-reaction controller and Telegram variants, adding Telegram dispatch wiring, adding status-reaction 429 backoff, and extending tests.

## Best Possible Solution

Land a rebased version that keeps the per-message callback design, fires queued notification from both session and global lanes with a once guard, preserves current lane-timeout behavior, keeps Telegram waiting-to-thinking transitions and account+chat 429 backoff, and updates the config type, strict Zod schema, generated schema/help, docs, and focused tests for the new `waiting` status key.

## Reproduction Assessment

Yes. For the user-visible behavior, send a second Telegram message in the same session while a first run occupies the session lane; current main sets thinking before execution and has no `onQueued` waiting state. For the review finding, configure `messages.statusReactions.emojis.waiting` after the PR runtime change and the current strict schema would still reject that key.

## Solution Assessment

No, not yet. The callback-based direction is the narrow maintainable approach and the PR has addressed earlier session-lane and 429-backoff defects, but the current proposal still misses the config contract/generated surfaces and needs a current-main rebase before merge.
