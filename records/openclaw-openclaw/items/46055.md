## Summary

Keep open. Current main still lacks a Telegram waiting reaction and per-message queue callback, so the PR is not obsolete, but the patch is not merge-ready because it adds a new status-reaction emoji key without updating the strict config contract, generated schema/help, or docs.

## What This Changes

The PR adds a Telegram waiting status reaction, threads a per-message onQueued callback through queue and reply execution paths, and adds Telegram status-reaction 429 backoff.

## Best Possible Solution

Land a rebased version that keeps the per-message callback design, fires queued notification from both session and global lanes with a once guard, preserves Telegram waiting-to-thinking transitions and account+chat 429 backoff, and updates the config type, zod schema, generated schema/help, and user docs for the new waiting status key.

## Reproduction Assessment

Yes. Current main has source-level reproduction evidence: queued Telegram turns can still move from queued ack to thinking without any waiting state because the status controller has no waiting API and the command queue has no onQueued signal.

## Solution Assessment

No, not as currently proposed. The callback-based approach is the right direction, but the patch needs the config-contract update and a rebase over current Telegram dispatch before it is the narrowest maintainable merge.
