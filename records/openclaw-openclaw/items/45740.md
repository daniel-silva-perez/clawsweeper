## Summary

Keep this open. Current main still has the reported gh-issues hardening gap: raw issue bodies and review-comment JSON are embedded into spawned child task prompts, while the runtime currently offers sandbox selection and config-level subagent tool policy rather than a per-spawn untrusted-content or tool-scope boundary. The linked prior PR was closed unmerged, and the external prompt-warning PR is useful context but only a partial mitigation.

## What This Changes

The issue requests hardening of the gh-issues automation so untrusted GitHub issue bodies and review comments are not interpolated directly into tool-enabled sub-agent prompts.

## Best Possible Solution

Define a maintainer-approved first mitigation boundary: escape and wrap gh-issues issue/comment content as untrusted data before child task construction or route it through a structured summary/data channel, then decide whether unattended gh-issues spawns should require sandboxed target agents or first-class per-spawn tool profiles. Preserve notification behavior and add regression coverage for hostile issue/comment content plus any sessions_spawn policy changes.

## Reproduction Assessment

Yes. A static current-main reproduction is high confidence: the gh-issues skill template instructs the orchestrator to substitute `{body}` and `{json_array_of_actionable_comments}` directly into child task strings. There is not yet a current-main PoC showing the model crosses a security boundary after receiving that content.

## Solution Assessment

Unclear as a final implementation because there is no merged patch. The linked prompt-warning/fence approaches are useful first layers, but the stronger maintainable direction is structured untrusted-data handling plus an explicit maintainer decision on sandbox/tool-scope defaults for unattended subagents.
