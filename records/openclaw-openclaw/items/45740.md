## Summary

Keep this open. Current main still passes raw GitHub issue bodies and review-comment JSON into gh-issues child-agent task prompts, while sessions_spawn currently offers opt-in sandboxing and config-level subagent tool policy rather than a first-class untrusted-data or per-spawn tool-scope boundary. The prior target-repo PR was closed unmerged, and the fork PR discussed on the issue is only a partial prompt-warning layer.

## What This Changes

The issue requests hardening the gh-issues skill so raw GitHub issue bodies and review comment bodies are not injected into tool-enabled sub-agent prompts without isolation or runtime controls.

## Best Possible Solution

Define a maintainer-approved mitigation boundary: escape and wrap gh-issues issue/comment content as untrusted data before child task construction or route it through a structured summary/data channel, then decide whether unattended gh-issues spawns should require sandboxed target agents or first-class per-spawn tool profiles. Preserve notification behavior and add regression coverage for hostile issue/comment content plus any sessions_spawn policy changes.

## Reproduction Assessment

Yes. A static current-main reproduction is high confidence: `skills/gh-issues/SKILL.md` directly substitutes `{body}` and `{json_array_of_actionable_comments}` into child task strings. There is not yet a current-main PoC showing the model crosses a security boundary after receiving that content.

## Solution Assessment

Unclear because there is no merged patch. The prompt-warning/fence approaches discussed on the issue are useful first layers, but the stronger maintainable direction is structured untrusted-data handling plus an explicit maintainer decision on sandbox and tool-scope defaults for unattended subagents.
