## Summary

Current main still has the reported gh-issues hardening gap: the bundled skill tells the orchestrator to pass raw GitHub issue bodies and review-comment JSON into child-agent prompts, while `sessions_spawn` exposes sandbox selection but no per-call tool allow/deny policy. The discussion and linked PRs are useful mitigation context, but the open fork PR is unmerged and the closed #45748 was not merged, so this should remain open for maintainer/security-hardening judgment.

## Best Possible Solution

Keep this issue open for maintainer/security-hardening judgment. The best path is to choose the intended first mitigation boundary explicitly: escape and wrap gh-issues issue/comment bodies as untrusted data before child prompt construction, and separately decide whether `sessions_spawn` needs per-call tool profiles or safer unattended-mode defaults such as sandbox-required children. Treat the linked prompt-warning PRs as useful partial mitigation context, not as resolving the runtime-level concern unless maintainers explicitly accept that narrower scope.
