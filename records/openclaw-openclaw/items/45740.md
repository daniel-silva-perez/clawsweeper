## Summary

Keep this open. Current main still routes raw GitHub issue bodies and review-comment JSON into `gh-issues` child-agent task prompts, and the available runtime controls are opt-in sandboxing plus config-level subagent tool policy rather than a first-class untrusted-data or per-spawn tool-scope boundary. The prior target-repo PR was closed unmerged, while the linked fork PR is only a partial prompt-warning layer.

## What This Changes

Harden the `gh-issues` skill/runtime so GitHub issue bodies and review comments are treated as untrusted data before child-agent prompts, with appropriate unattended-spawn sandbox or tool-scope controls.

## Best Possible Solution

Define a maintainer-approved runtime boundary for automation skills that ingest external GitHub content: escape and wrap or summarize issue/comment text as untrusted data before child task construction, then decide whether unattended `gh-issues` spawns should require sandboxed target agents, named tool profiles, or per-spawn capability constraints. Preserve notification behavior and add regression coverage for hostile issue/comment content plus any sessions_spawn policy changes.

## Reproduction Assessment

Yes. A static current-main reproduction is high confidence: `skills/gh-issues/SKILL.md` still directly substitutes `{body}` and `{json_array_of_actionable_comments}` into child task strings. There is not yet a high-confidence runtime PoC showing the model crosses a security boundary after receiving that content.

## Solution Assessment

Unclear because there is no merged patch. The prompt-warning/fence approach discussed in PR #45748 and the fork PR is a useful first layer, but the stronger maintainable direction is structured untrusted-data handling plus an explicit maintainer decision on sandbox and tool-scope defaults for unattended subagents.
