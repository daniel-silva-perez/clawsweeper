## Summary

Keep this PR open. Current main still uses user-token-first Slack reads regardless of `userTokenReadOnly: false`, but the submitted patch edits the old `src/agents/tools/slack-actions.*` paths that are no longer tracked. The intended fix needs to be ported to the active Slack plugin runtime and reviewed for the bot-token read security and fallback concerns already raised in the PR discussion.

## Best Possible Solution

Keep this PR open for maintainer product/security review. If maintainers want `userTokenReadOnly: false` to make Slack reads prefer the bot token, port the change to `extensions/slack/src/action-runtime.ts`, add regression tests for false/read behavior plus bot-denied fallback semantics, and update `docs/channels/slack.md`. If maintainers prefer the current least-privilege user-token-first read model, close manually with that rationale and document the #42752 workaround.

## Risks / Open Questions

- Porting the PR as written could make Slack reads use a broader bot token for tool-supplied `channelId`, `threadId`, or `fileId` values without a new authorization boundary or explicit opt-in.
- Bot-token-first reads without retry can regress workspaces where both tokens exist but only the user token can read a private channel, file, or specific read scope.
- The current patch cannot affect current main until it is rebased or rewritten against the active Slack plugin runtime.
