## Summary

Keep this PR open. Current main still does not implement the intended Slack token-selection change, and the PR edits obsolete pre-plugin files rather than the active Slack plugin runtime. The proposed behavior also still needs maintainer product/security review because bot-token-first reads can broaden access for tool-supplied Slack channel, thread, and file targets.

## Best Possible Solution

Keep this PR open for maintainer review. If maintainers want userTokenReadOnly=false to make Slack reads prefer the bot token, port the selector change to extensions/slack/src/action-runtime.ts, add regression tests for false/read behavior plus bot-denied fallback semantics, and update https://docs.openclaw.ai/channels/slack. If maintainers prefer the current user-token-first read model, close manually with the product/security rationale and document the workaround from #42752.

## Risks / Open Questions

- Porting the PR as-is could make reads use a broader bot token for tool-supplied Slack channel, thread, or file IDs without an added authorization boundary or explicit opt-in.
- Bot-token-first reads without retry/fallback may regress workspaces where both tokens exist but only the user token can read a private channel or file.
- The current patch cannot affect main because it edits deleted pre-plugin Slack action files instead of extensions/slack/src/action-runtime.ts and active tests/docs.
