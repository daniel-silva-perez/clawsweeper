## Summary

Keep this PR open. Current main still does not implement the PR's intended Slack token-selection behavior: active Slack reads remain user-token-first, and the PR patch targets obsolete pre-plugin files rather than the current Slack plugin runtime. The proposed behavior also still needs maintainer product/security review because bot-token-first reads can broaden read access for tool-supplied Slack targets.

## Best Possible Solution

Keep this PR open for maintainer review rather than auto-closing. If maintainers want `userTokenReadOnly: false` to make Slack reads prefer the bot token, port the selector change to `extensions/slack/src/action-runtime.ts`, add active regression tests for false/read behavior and bot-denied fallback semantics, and update `docs/channels/slack.md`. If maintainers prefer the current user-token-first read model, close manually with the product/security rationale and document the workaround from #42752.

## Risks / Open Questions

- Porting the PR as-is could make reads use a broader bot token for tool-supplied Slack channel, thread, or file IDs without an added authorization boundary or explicit opt-in.
- A bot-token-first selector without retry/fallback can regress workspaces where both tokens exist but only the user token can read a private channel or file.
- The current patch cannot affect current main because it edits deleted pre-plugin Slack action files instead of `extensions/slack/src/action-runtime.ts` and active tests/docs.
