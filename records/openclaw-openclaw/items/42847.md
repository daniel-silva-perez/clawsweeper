## Summary

Keep this PR open. Current main still does not implement the intended Slack token-selection change, and the submitted patch targets deleted pre-plugin files rather than the active Slack plugin runtime. The proposed bot-token-first read behavior also needs maintainer product/security review before it is ported.

## Best Possible Solution

Keep this PR open for maintainer product/security review. If maintainers want userTokenReadOnly=false to make Slack reads prefer the bot token, port the change to extensions/slack/src/action-runtime.ts, add regression tests for false/read behavior and bot-denied fallback semantics, and update https://docs.openclaw.ai/channels/slack. If maintainers prefer the current user-token-first read model, close manually with that rationale and document the #42752 workaround.

## Risks / Open Questions

- Porting the PR as-is could make Slack reads use a broader bot token for tool-supplied channelId, threadId, or fileId values without a new authorization boundary or explicit opt-in.
- Bot-token-first reads without retry/fallback can regress workspaces where both tokens exist but only the user token can read a private channel or file.
- The submitted patch cannot affect current main until it is rebased or rewritten against extensions/slack/src/action-runtime.ts and active tests/docs.
