## Summary

Keep this issue open. Current main still drops provider/model context at the final user-facing transient error formatter, and tests still assert the generic overloaded/rate-limit text. There is also an open PR (#55851) whose body uses closing syntax for this issue, so this should remain open until that PR or a refreshed equivalent is reviewed and merged or closed.

## Best Possible Solution

Review and either land or close PR #55851, or implement a smaller equivalent on current main: thread the active failed provider/model into `formatRateLimitOrOverloadedErrorCopy`, keep context sanitized, and update formatter, payload, lifecycle, and Telegram-facing error tests so overloaded and rate-limit messages identify the upstream provider.
