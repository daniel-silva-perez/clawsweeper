## Summary

Keep this issue open. Current main still has the active provider/model available before formatting assistant errors, but the transient overload/rate-limit formatter only receives the raw error text and still returns generic user-facing copy for Anthropic overload and generic 429 cases. The provided timeline also shows open PR #55851 with closing syntax for this issue, so this should remain open until that PR, or a refreshed equivalent, is reviewed and merged or closed.

## Best Possible Solution

Review and either land or close PR #55851, or implement a smaller equivalent on current main: pass sanitized active provider/model context into the transient overload/rate-limit formatter, preserve the existing provider-specific raw rate-limit detail handling, and update formatter, payload, lifecycle, and Telegram-facing tests so generic 529/429 messages identify the upstream provider that actually failed.
