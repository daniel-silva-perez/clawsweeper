## Summary

Keep open. Current main still uses the same Baileys 7.0.0-rc.9 WhatsApp Web backend called out in the report, and the current OpenClaw Baileys patch only covers media upload behavior, not the QR/auth handshake paths. The QR login flow still creates a Baileys socket, logs `WhatsApp QR received.`, attaches the login waiter, and can keep returning `Still waiting for the QR scan` when the socket never reaches `connection === "open"` or `close`. The reporter's side-by-side whatsapp-web.js pairing proof keeps this actionable, and the related maintainer PR context explicitly says that reconnect work does not replace this separate QR-pairing report.

## Best Possible Solution

Keep this issue open and route it to WhatsApp plugin maintainers for an explicit Baileys/backend decision. The best fix should stay owned by `extensions/whatsapp`, either proving and patching the Baileys QR/auth handshake or replacing the QR pairing backend, with regression coverage for QR login failure/retry/error surfacing and live macOS ARM64 Docker QR-pairing proof before closure. Adjacent work such as #47710, #63939, and #73732 should remain context, not closing evidence, unless it lands with QR-pairing proof.
