## Summary

Keep open. Current main still uses the same Baileys 7.0.0-rc.9 WhatsApp backend implicated by the report, and OpenClaw’s local Baileys patch does not touch the QR/login handshake paths called out in the issue. The later WhatsApp reliability work visible in the provided context explicitly says it does not replace this QR-pairing report.

## Best Possible Solution

Keep the issue open and fix it in the WhatsApp plugin/runtime path. The likely maintainer path is to either update or patch the Baileys auth handshake used by extensions/whatsapp, then add focused regression coverage around QR login wait/retry/error surfacing and validate with a real Docker/macOS ARM64 QR pairing run. If maintainers decide the Chromium/wwebjs sidecar is the strategic backend, that should be a separate product decision rather than a cleanup close.
