## Summary

Keep open. Current main and the latest release still use the reported Baileys RC9 WhatsApp login surface without the proposed sendUnifiedSession/auth-handshake patch, and an open unmerged closing PR specifically targets this silent post-scan QR failure.

## What This Changes

The issue reports WhatsApp QR pairing stalling after QR scan on Docker/macOS ARM64 with Baileys 7.0.0-rc.9, despite the same phone pairing through WhatsApp Web or whatsapp-web.js.

## Best Possible Solution

Ship a WhatsApp-plugin-owned fix that makes QR scan advance to connected state or surface a clear failure, while preserving the existing media-upload patch and keeping WhatsApp-specific recovery out of core.

## Reproduction Assessment

Yes. The issue gives concrete Docker/macOS ARM64 steps and same-phone side-by-side evidence where WhatsApp Web/whatsapp-web.js pairs while OpenClaw's Baileys flow stalls after QR receipt; source inspection shows current main still uses that same dependency and login path.

## Solution Assessment

Unclear. The open #74349 sendUnifiedSession patch is a plausible narrow fix in the owning plugin patch surface, but it still needs maintainer review and live QR-pairing validation before it can be treated as the best fix.
