## Summary

Keep open. Current main and the latest release still use Baileys 7.0.0-rc.9 with only the existing media-upload patch, while the QR login flow still waits on Baileys after logging `WhatsApp QR received.` The issue also has an open, unmerged closing PR specifically targeting the reported QR auth handshake failure, so the issue should remain open until that PR or an equivalent fix lands and is verified.

## What This Changes

QR-based WhatsApp linking fails silently after scanning in OpenClaw 2026.3.x Docker on macOS ARM64 while browser/whatsapp-web.js pairing works with the same phone and account.

## Best Possible Solution

Review, validate, and either land #74349 or replace it with an equivalent WhatsApp-plugin-owned fix. Closure should wait for shipped code that patches or replaces the Baileys QR/auth handshake, preserves existing media-upload behavior, and includes targeted tests plus live QR-pairing proof from QR receipt through connected state or a surfaced failure.

Yes. The reporter provides concrete Docker/macOS ARM64 steps and same-phone side-by-side proof that OpenClaw's Baileys QR flow stalls after QR receipt while whatsapp-web.js pairs; I did not live-run WhatsApp in this read-only review, but current main still matches the reported dependency and flow.
