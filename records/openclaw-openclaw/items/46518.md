## Summary

Keep open. Current main and the latest release still ship the WhatsApp plugin on Baileys 7.0.0-rc.9 with the existing media-upload-only patch and no `sendUnifiedSession` handshake change, while an open unmerged closing PR specifically targets this QR pairing failure.

## What This Changes

The issue reports that WhatsApp QR pairing on macOS ARM64 Docker reaches `WhatsApp QR received.` and then stalls silently until the phone times out.

## Best Possible Solution

Review, validate, and land #74349 or an equivalent WhatsApp-plugin-owned Baileys handshake fix. The end state should preserve the existing media-upload patch, avoid core WhatsApp-specific policy changes, include targeted regression coverage for the OpenClaw login state machine, and have live QR-pairing proof that the scan advances to connected state or surfaces a clear failure.

## Reproduction Assessment

Yes. The issue provides concrete Docker/macOS ARM64 steps plus same-phone side-by-side evidence where whatsapp-web.js pairs and OpenClaw's Baileys flow stalls after QR receipt; current main still matches the reported dependency and login flow.

## Solution Assessment

Unclear. The open #74349 `sendUnifiedSession` patch is a plausible narrow fix, but it has not merged and the PR body says live QR pairing was not verified, so it still needs maintainer review and live proof before this can close.
