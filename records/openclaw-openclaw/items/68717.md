## Summary

Keep open. Current main does not implement the default-enabled failure-alert behavior, so this PR is not obsolete; however it flips a documented public config default and needs maintainer review plus docs/changelog alignment before merge. The prior bot review context is useful: the latest PR diff addresses the missing-test concern and preserves the per-job override path, leaving public contract/product alignment as the main blocker.

## What This Changes

This PR changes cron failure alerts without explicit per-job or global alert config from opt-in to opt-out and adds a regression test for the default alert path.

## Best Possible Solution

Land a revised version only after maintainers confirm opt-out failure alerts are the intended product default, with the runtime guard, default-path regression test, config reference/docs, changelog, and changed-gate validation all aligned.

## Reproduction Assessment

Yes. On current main, a recurring job with no job-level `failureAlert` and no global `cron.failureAlert.enabled: true` follows the line 392 guard and returns no alert config, so repeated errors cannot emit an alert by default; the PR's new test is the focused reproduction for that path.

## Solution Assessment

No, not as submitted. The code and test are a narrow fit for the runtime bug, but changing the default without updating the documented config default and changelog is not the best complete fix.
