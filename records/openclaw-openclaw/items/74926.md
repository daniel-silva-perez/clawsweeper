## Summary

Keep open. This is a maintainer-labeled MEMBER PR touching a public Gateway data-access surface, and the diff still has a concrete taskId query correctness issue before it should land.

## What This Changes

The PR adds Gateway `artifacts.list/get/download` RPCs over transcript media blocks, SDK artifact helper types/methods, generated Swift protocol models, docs/changelog updates, and handler/SDK tests.

## Best Possible Solution

Land a revised artifact RPC hardening PR that preserves the read-scoped transcript-provenance design, resolves taskId queries through the task registry/session ownership data, and updates the public protocol docs before closing the linked artifact API issue.

## Reproduction Assessment

Yes. The taskId bug is reproducible by creating a task record whose `taskId` differs from its `runId`, adding a transcript artifact tagged with that taskId, and calling `artifacts.list({ taskId })`; the PR routes that taskId through the run-id resolver and returns no session.

## Solution Assessment

No, not yet. The overall Gateway/SDK artifact RPC direction is appropriate, but the taskId lookup path is not the narrowest correct implementation until it uses the task registry to find the owning session before filtering transcript artifacts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Resolve taskId queries through task records:** `src/gateway/server-methods/artifacts.ts:272`
  - body: `taskId` is advertised as a supported artifact query, but this path sends it to `resolveSessionKeyForRun`, which only resolves run IDs/session IDs. Existing task records store `taskId` separately from `runId`, so a taskId-only query will usually return `artifact_scope_not_found` even when that task produced transcript artifacts. Resolve the task record by taskId and use its session key before filtering by `taskId`.
  - confidence: 0.9
- **[P3] Document the new artifact RPCs in the protocol reference:** `src/gateway/server-methods-list.ts:74-76`
  - body: The PR advertises `artifacts.list/get/download` as Gateway methods, but `docs/gateway/protocol.md` still omits them from the operator method reference. Since these are public protocol additions, consumers reading the Gateway protocol docs will miss the new contract.
  - confidence: 0.78
