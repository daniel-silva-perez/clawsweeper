## Summary

Keep open. Current main still has passive reaction notification plumbing for the main requested channels, not a documented opt-in mode that wakes the agent on reactions. Feishu has a channel-specific synthetic-turn path, and the plugin runtime exposes heartbeat wake helpers, but the cross-channel config/API requested here is still missing. The existing issue remains the canonical tracker; #56708 and #11747 are closed overlapping reports, while #73681 is a narrower Google Chat inbound-reaction gap.

## Best Possible Solution

Keep this issue open as the canonical cross-channel product request. Implement a documented opt-in reaction wake mode, for example reactionTriggersTurn: "off" | "own" | "all" | "allowlist", that preserves passive default behavior, enqueues the reaction event, and requests a scoped heartbeat or agent turn only after existing reaction notification filters and authorization checks pass. Update config schemas/types/generated metadata, channel docs, and focused tests for the reaction-capable bundled channels; leave #73681 as the separate Google Chat inbound-reaction delivery gap.

## Risks / Open Questions

- Default behavior should stay passive/off so ordinary group-chat reactions do not wake agents unexpectedly.
- Wake calls must happen only after channel-specific reaction filters, access checks, allowlists, and sender authorization pass; the Slack reaction path in particular should be checked carefully so the new mode does not broaden ingress.
- Session routing needs channel-specific care: Telegram reactions lack message_thread_id, Discord/Slack thread routing can involve parent channels, and Signal/Matrix/BlueBubbles use different peer identifiers.
- The config shape is a product/API decision across bundled and third-party channel plugins, so a maintainer should confirm the final name and modes before implementation.
