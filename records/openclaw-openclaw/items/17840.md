## Summary

Keep open. Current main still has reaction notification plumbing for several channels, but reactions are only queued as system events; there is no documented opt-in config or per-message flag that turns a reaction into an agent wake/turn. The earlier cleanup comment remains accurate on current main SHA 9bb1e59447a5fdf2c0164672df6c748188b26442, and the related closed overlaps (#56708 and #11747) do not supersede this broader cross-channel request.

## Best Possible Solution

Keep this issue open as the canonical cross-channel product/API request. A maintainer should choose the opt-in contract, such as `reactionTriggersTurn: "off" | "own" | "all" | "allowlist"` or a narrower per-message `awaitReaction` flag, then implement wake behavior only after existing channel auth and reaction notification filters pass. The implementation should update config schemas/generated metadata, channel docs, and focused tests for the reaction-capable bundled plugins.
