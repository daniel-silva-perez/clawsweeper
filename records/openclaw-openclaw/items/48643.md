## Summary

Keep this PR open. Current main has the backend `sessions.create` seam that a reworked implementation should use, but it still does not ship the dedicated Control UI flow for creating a parallel chat thread without resetting the active session. The PR also has unresolved functional review findings around rollback/error preservation and dialog keyboard handling.

## Best Possible Solution

Keep this PR open as an implementation candidate. Rebase it onto current main, replace the bespoke thread creation path with the gateway `sessions.create` API using the current session as `parentSessionKey` and optional initial task/message, preserve the active transcript and actionable error on failure, fix modal focus/Escape handling, and add focused UI tests for successful creation, rollback, and in-flight dismissal guards.

## Risks / Open Questions

- Closing this as implemented would drop useful UI work for a feature still absent from current main.

- Because the feature creates and switches authenticated Gateway sessions from the browser, the final implementation should use the existing `sessions.create` scope and idempotency/error patterns rather than inventing a parallel client contract.
