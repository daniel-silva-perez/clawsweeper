## Summary

Keep this PR open. Current main still does not ship the intended dedicated chat UI flow for creating a parallel thread without resetting the active session. Main now has a generic `sessions.create` gateway API that looks like the better backend seam for this work, but the Control UI still exposes only the reset-style `New session` path and the PR has unresolved functional review findings.

## Best Possible Solution

Keep this PR open as an implementation candidate, but rework it against current main. The best path is to rebase, use the gateway `sessions.create` API for session/thread creation where possible, preserve the current transcript and actionable error on failed creation, fix modal focus/Escape handling, and add focused tests for successful create-thread, failure rollback, and dismissal guard behavior before maintainer review.

## Risks / Open Questions

- Closing this PR as implemented would drop useful UI work for a feature that is still missing from current main.
- Landing the PR unchanged risks the reviewed rollback problems: failed create-thread attempts can lose the specific error and may blank the current transcript if history reload cannot restore it.
- The PR appears stale relative to current main’s `sessions.create` API; a rebase should avoid bespoke client-side session creation when the gateway now owns that contract.
- The modal Escape/focus behavior called out in review still needs a UI accessibility fix before merge consideration.
