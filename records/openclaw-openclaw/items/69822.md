## Summary

Keep open. Current main still lacks the requested `socket.drain` lifecycle routing for message-only subscribers, while this PR is a focused implementation of that gateway behavior. I did not find a concrete blocking correctness or security issue in the latest diff; the remaining work is maintainer review of the public event contract/docs and final validation.

## What This Changes

The PR adds a read-scoped `socket.drain` gateway event for session reset/delete/new teardowns, exposes message-subscriber lookup on `GatewayRequestContext`, routes drain notifications to event and message subscribers, adjusts teardown slow-consumer handling, and adds gateway coverage.

## Best Possible Solution

Keep this PR as the active implementation surface, then land a focused version after maintainer review confirms the `socket.drain` event contract, protocol documentation expectations, and changed-gate results.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: on current main, `sessions.messages.subscribe` clients are tracked separately, while lifecycle `emitSessionsChanged` only targets session event subscribers and `socket.drain` is not registered in the event scope guard.

## Solution Assessment

Yes, with maintainer validation pending. Reusing the existing subscriber registries and session mutation helper is the narrowest maintainable fix; the safer follow-through is to document the event if it is intended as a public client contract.
