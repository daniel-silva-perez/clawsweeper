Keep this PR open, but do not merge it as-is. Current main does not implement the proposed heartbeat stop button, `sessions.activity` RPC, or progress-prompt guidance, but the branch combines broad Gateway session-stop semantics with UI and prompt changes and still has concrete blocking review findings.

## What This Changes

This PR adds WebChat heartbeat/session stop controls, a new `sessions.activity` RPC, broader `sessions.abort` cleanup, stream/progress handling changes in Control UI/OpenClawKit, and system-prompt guidance for milestone/progress updates.

## Best Possible Solution

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep normal chat abort on chat.abort:** `ui/src/ui/controllers/chat.ts:337`
  - body: The regular WebChat stop path now sends `sessions.abort` for active generation aborts. The new server handler disables session-bound cron jobs and stops queues/subagents/ACP, so clicking Stop on one answer can pause unrelated reminders or progress jobs. Keep ordinary generation stop on `chat.abort`, or require an explicit autonomy-stop action for broader cleanup.
  - confidence: 0.9
- **[P2] Use snapshots as replacements when no delta is present:** `ui/src/ui/controllers/chat.ts:208`
  - body: When a non-prefix assistant snapshot arrives without `delta`, this fallback appends it to the existing stream. Gateway snapshots are authoritative in that case, so rewritten or replayed text can be duplicated/corrupted in the UI and in locally committed abort partials.
  - confidence: 0.86
- **[P2] Exclude dropped followups from pending size:** `src/auto-reply/reply/queue/state.ts:43`
  - body: `droppedCount` tracks discarded queue entries, not pending work. Including it makes `followupQueued > 0` and `canStop: true` after the queue is actually empty, causing the stop control to appear for work that cannot be stopped.
  - confidence: 0.91
- **[P2] Return cleanup state after timeout errors:** `src/gateway/server-methods/sessions.ts:1030-1040`
  - body: By this point queues, subagents, ACP, and cron jobs may already have been cleaned up. If the embedded-run wait times out, the handler returns only `UNAVAILABLE`, so clients cannot reconcile the work that was already stopped and may keep stale UI state or retry unnecessarily.
  - confidence: 0.75
