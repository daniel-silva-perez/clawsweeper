## Summary

Current main still lacks the heartbeat stop/session-activity and progress-prompt pieces, while the PR branch introduces blocking regressions around normal WebChat stop semantics, stream merging, queue activity counts, and broadened session-abort authority.

## What This Changes

The PR adds WebChat heartbeat/session stop controls, a new `sessions.activity` RPC, broader `sessions.abort` cleanup, stream/progress handling changes in Control UI and OpenClawKit, and system-prompt guidance for milestone/progress updates.

## Best Possible Solution

Split and rebase the work: keep ordinary chat generation Stop on `chat.abort`, design any session activity/autonomy-stop API against #39127 with protocol docs and authorization coverage, and land progress-prompt guidance separately with focused prompt or behavior tests.

Live UI testing is still needed before accepting any revised heartbeat control UX.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep normal chat Stop on chat.abort:** `ui/src/ui/controllers/chat.ts:337`
  - body: The regular WebChat stop path now calls `sessions.abort`, whose handler also clears queues, stops subagents, cancels ACP sessions, and disables session-bound cron jobs. Stopping one answer can pause unrelated session automation, so ordinary generation Stop should stay on `chat.abort` or the broader cleanup needs a distinct explicit action.
  - confidence: 0.9
- **[P2] Use snapshots as replacements without delta:** `ui/src/ui/controllers/chat.ts:208`
  - body: When a non-prefix assistant snapshot arrives without `delta`, this fallback appends it to the current stream. Gateway snapshots are authoritative in that case, so rewritten or replayed text can be duplicated or corrupted in the UI and in locally committed abort partials.
  - confidence: 0.86
- **[P2] Exclude dropped followups from pending size:** `src/auto-reply/reply/queue/state.ts:43`
  - body: `droppedCount` tracks discarded queue entries, not pending work. Including it makes `followupQueued` positive and `canStop: true` after the queue is actually empty, causing the stop control to appear for work that cannot be stopped.
  - confidence: 0.91
- **[P2] Return cleanup state after abort timeout errors:** `src/gateway/server-methods/sessions.ts:1030-1040`
  - body: By this point queues, subagents, ACP sessions, and cron jobs may already have been cleaned up. If the embedded-run wait times out, returning only `UNAVAILABLE` leaves clients unable to reconcile stopped work and can keep stale UI state or trigger redundant retries.
  - confidence: 0.75
- **[P3] Refresh heartbeat state when another client disables it:** `ui/src/ui/app-gateway.ts:413-419`
  - body: Incoming `heartbeat` events only set `heartbeatsEnabled` to true, while no connected event path clears it when another client disables heartbeats. The heart-pulse button can therefore remain visible until reconnect even though heartbeats are off.
  - confidence: 0.78
