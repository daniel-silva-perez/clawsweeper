## Summary

Keep this PR open for maintainer handling, but it should not merge as-is. Current main still lacks the heartbeat/session stop UI and the proposed progress-prompt section, while the PR introduces blocking behavior regressions in normal WebChat Stop, stream merging, queue activity counts, and timeout reporting, plus a security-sensitive broadening of session abort authority.

## What This Changes

The PR branch adds WebChat heartbeat/session stop UI, a `sessions.activity` RPC, broader `sessions.abort` cleanup, assistant stream/progress handling changes including OpenClawKit updates, and system-prompt guidance for milestone/progress updates.

## Best Possible Solution

Split and rebase the work: keep ordinary chat generation Stop on `chat.abort`, design any session/autonomy stop and activity state against #39127 with protocol docs and authorization coverage, and land progress-prompt guidance separately with focused prompt or behavior tests.

## Reproduction Assessment

Yes for the review findings: the PR head shows the changed lines directly, and current main provides comparison points for `chat.abort` behavior, protocol surface, docs, and tests. No live UI run was needed for the logic bugs because they are visible in the diff and current-main contracts.

## Solution Assessment

No. Combining heartbeat controls, a new activity RPC, destructive abort cleanup, stream merging, OpenClawKit changes, and prompt policy in one stale/unmergeable PR is not the narrowest maintainable path; the safer path is split/rebase plus explicit activity-design review.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep normal Stop on chat.abort:** `ui/src/ui/controllers/chat.ts:337`
  - body: The regular WebChat Stop path now sends `sessions.abort`, whose handler also clears queues, stops subagents, cancels ACP, and disables session-bound cron jobs. Stopping one answer can pause unrelated session automation, so ordinary generation Stop should stay on `chat.abort` or the broader cleanup needs a distinct explicit action.
  - confidence: 0.9
- **[P2] Replace snapshots instead of appending them:** `ui/src/ui/controllers/chat.ts:208`
  - body: When a non-prefix assistant snapshot arrives without `delta`, this fallback appends it to the current stream. Gateway snapshots are authoritative in that case, so replayed or rewritten text can be duplicated or corrupted in the UI and in locally committed abort partials.
  - confidence: 0.86
- **[P2] Exclude dropped followups from pending size:** `src/auto-reply/reply/queue/state.ts:43`
  - body: `droppedCount` tracks discarded queue entries, not pending work. Including it makes `followupQueued > 0` and `canStop: true` after the queue is actually empty, causing the stop control to appear for work that cannot be stopped.
  - confidence: 0.91
- **[P2] Return cleanup state after timeout errors:** `src/gateway/server-methods/sessions.ts:1030-1039`
  - body: By this point queues, subagents, ACP sessions, and cron jobs may already have been cleaned up. If the embedded-run wait times out, the handler returns only `UNAVAILABLE`, so clients cannot reconcile the work already stopped and may keep stale UI state or retry unnecessarily.
  - confidence: 0.75
- **[P3] Reset heartbeat visibility from incoming state:** `ui/src/ui/app-gateway.ts:416-418`
  - body: The heartbeat event handler only sets `heartbeatsEnabled` to true. If another client disables heartbeats through `set-heartbeats`, this connected UI has no incoming path to set the flag back to false, so the heart button can stay visible until reconnect.
  - confidence: 0.72
