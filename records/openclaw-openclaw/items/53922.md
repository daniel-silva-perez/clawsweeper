## Summary

Keep this PR open, but the current patch should not merge as-is. Current main does not implement the heartbeat/autonomy stop or progress-prompt changes, while the PR broadens normal WebChat Stop into a more destructive session-abort path and introduces concrete stream, queue-count, and timeout reconciliation bugs. The activity API portion also overlaps the still-open canonical activity-state design in #39127.

## What This Changes

This PR changes Control UI/WebChat and OpenClawKit chat state to show heartbeat/autonomy stop controls, adds a `sessions.activity` RPC, expands `sessions.abort` cleanup, forwards assistant deltas, and adds system-prompt guidance for progress and milestone updates.

## Best Possible Solution

Split and rebase the work: keep ordinary chat generation Stop on `chat.abort`, design session/autonomy stop and activity state against #39127 with protocol docs and authorization coverage, and land any progress-prompt guidance separately with focused prompt or behavior tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep normal chat abort on chat.abort:** `ui/src/ui/controllers/chat.ts:337`
  - body: The normal WebChat stop path now sends `sessions.abort`, whose handler also clears queues, stops subagents, cancels ACP sessions, and disables session-bound cron jobs. Stopping one in-flight answer can pause unrelated session automation, so ordinary generation Stop should stay on `chat.abort` or the broader cleanup needs a distinct explicit action.
  - confidence: 0.9
- **[P2] Use snapshots as replacements when no delta is present:** `ui/src/ui/controllers/chat.ts:208`
  - body: When a non-prefix assistant snapshot arrives without `delta`, this fallback appends it to the current stream. Gateway snapshots are authoritative in that case, so rewritten or replayed text can be duplicated or corrupted in the UI and locally committed abort partials.
  - confidence: 0.86
- **[P2] Exclude dropped followups from pending size:** `src/auto-reply/reply/queue/state.ts:43`
  - body: `droppedCount` tracks discarded queue entries, not pending work. Including it makes `followupQueued > 0` and `canStop: true` after the queue is actually empty, causing the stop control to appear for work that cannot be stopped.
  - confidence: 0.91
- **[P2] Return cleanup state after timeout errors:** `src/gateway/server-methods/sessions.ts:1030-1040`
  - body: By this point queues, subagents, ACP sessions, and cron jobs may already have been cleaned up. If the embedded-run wait times out, the handler returns only `UNAVAILABLE`, so clients cannot reconcile the work already stopped and may keep stale UI state or retry unnecessarily.
  - confidence: 0.75
- **[P3] Reset heartbeat visibility from incoming state changes:** `ui/src/ui/app-gateway.ts:416`
  - body: The new event handler only sets `heartbeatsEnabled` to true when a heartbeat event arrives. If another client disables heartbeats through `set-heartbeats`, this connected UI has no incoming path to set the flag back to false, so the heart button can stay visible until reconnect.
  - confidence: 0.72
