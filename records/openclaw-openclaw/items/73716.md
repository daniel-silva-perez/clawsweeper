## Summary

Keep this issue open. Current main still has the reported raw/canonical Control UI path: the web UI sends and tracks a raw `state.sessionKey`, the gateway resolves that request to the canonical session key for chat broadcasts, and the web `handleChatEvent` drops the event before checking the matching `runId`. The provided related items are adjacent Control UI live-update/stuck reports, but none is a canonical duplicate or merged implementation for this raw `main` versus `agent:main:main` case.

## Best Possible Solution

Keep the issue open and fix the web Control UI. The narrowest likely fix is in `ui/src/ui/controllers/chat.ts`/`ui/src/ui/app-gateway.ts`: accept `chat` events for the active `chatRunId` even when the event session key is a canonical alias of the current raw key, or normalize the active UI session to the gateway canonical key before sending. Add regression coverage for `state.sessionKey: "main"`, `payload.sessionKey: "agent:main:main"`, and matching `runId`, plus any deferred `session.message` reload behavior that should work across the same alias pair.
