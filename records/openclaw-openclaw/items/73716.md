## Summary

Keep open. Current main still has the reported raw-versus-canonical session-key mismatch: Control UI sends its active `state.sessionKey` unchanged, the gateway resolves that raw key to a canonical session key for the run and broadcasts `chat` events with the canonical key, and the UI drops those events before considering the matching active `runId`. The linked open PR #73721 is the implementation candidate and has not merged, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue requests that Control UI show live chat responses when it sends a raw session alias such as `main` but gateway `chat` events for the same run use a canonical key such as `agent:main:main`.

## Best Possible Solution

Review and land PR #73721, or an equivalent small Control UI change, so same-`runId` live `chat` events are accepted across raw/canonical session-key aliases while unrelated run and session events remain filtered; add regression coverage for `main` versus `agent:main:main`.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is `handleChatEvent` with `state.sessionKey = "main"`, `state.chatRunId = "run-1"`, and a `chat` payload using `sessionKey = "agent:main:main"` plus `runId = "run-1"`; current main returns `null` at the session-key check.

## Solution Assessment

Yes. The narrowest maintainable fix belongs in Control UI event acceptance or session-key normalization, not provider/model handling; the active-run fallback proposed by PR #73721 is plausible if it continues to reject events where neither session key nor active run id matches.
