## Summary

Keep this issue open. Current main still has the reported raw-versus-canonical session-key path: the Control UI sends the active raw session key, the gateway resolves and emits the canonical key, and the UI drops the live chat event before considering the matching runId. Open PR #73721 directly targets this issue and is still unmerged, so the issue should remain open until that PR is reviewed and landed or closed.

## What This Changes

Fix Control UI live chat handling so an in-flight chat response is displayed live when `chat.send` used a raw session key but the gateway emits the same run under the canonical session key.

## Best Possible Solution

The desired end state is a small Control UI fix in `handleChatEvent` or session-key normalization that accepts the active in-flight run when `runId` matches across raw `main` and canonical `agent:main:main`, while preserving the guard that prevents unrelated sub-agent or fleet events from leaking into the active chat. Add focused regression coverage for the raw/canonical match and for a non-matching run.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to create a `ChatState` with `sessionKey: "main"` and `chatRunId: "run-1"`, then pass a `chat` payload with `sessionKey: "agent:main:main"` and `runId: "run-1"`; current `handleChatEvent` returns `null` at the session-key check before runId matching.

## Solution Assessment

Yes. The narrow maintainable solution is either to normalize the UI to the gateway canonical session key or to accept events for the active client run when `runId` matches; PR #73721's active-run fallback is a plausible small fix if it keeps the unrelated-run guard intact.
