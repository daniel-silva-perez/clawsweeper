## Summary

Keep this issue open. Current main still sends the Control UI's active session key unchanged, canonicalizes that key in the gateway, and then rejects returning live `chat` events before the active `runId` can be considered; open PR #73721 is the linked implementation candidate and is not merged.

## What This Changes

Control UI should render live chat deltas/finals for an active in-flight run when the UI uses raw `main` but the gateway emits canonical `agent:main:main`.

## Best Possible Solution

Land a small Control UI fix that either normalizes the active UI session to the gateway canonical key or accepts same-`runId` live `chat` events for the active client run, with regression tests covering raw `main` versus canonical `agent:main:main` and mismatched unrelated runs.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is `handleChatEvent` with `state.sessionKey = "main"`, `state.chatRunId = "run-1"`, and a `chat` payload using `sessionKey = "agent:main:main"` plus `runId = "run-1"`; current main returns `null` at the session-key check.

## Solution Assessment

Yes. The narrowest maintainable fix belongs in Control UI chat event acceptance or session-key normalization; PR #73721's active-run fallback is a plausible direction if it continues to reject events where neither session key nor active run id matches.
