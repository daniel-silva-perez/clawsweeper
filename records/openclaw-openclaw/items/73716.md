## Summary

Keep this issue open. Current main still sends the Control UI request with the raw active session key, canonicalizes that key in the gateway, and drops the returning live event in the web chat handler before the matching runId can be considered. The open linked PR is the implementation candidate, not a reason to close the issue before merge.

## What This Changes

The issue requests that Control UI live chat continue displaying in-flight responses when `chat.send` used a raw session key but gateway chat events arrive with the canonical session key.

## Best Possible Solution

The end state should be a small Control UI fix that either normalizes the active session to the gateway canonical key or accepts chat events for the active client run when `runId` matches, with regression coverage for raw `main` versus canonical `agent:main:main` and a non-matching run guard.

## Reproduction Assessment

Yes. A source-level reproduction is clear: create a chat state with `sessionKey: "main"` and active `chatRunId`, then pass a `chat` payload with `sessionKey: "agent:main:main"` and the same `runId`; current `handleChatEvent` returns `null` at the session-key check before runId matching.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the web Control UI event handling or session-key normalization layer; the linked PR's active-run fallback is a plausible small fix as long as it preserves the unrelated-run/sub-agent guard.
