## Summary

Keep this issue open. Current main still sends the Control UI request with the raw active session key, canonicalizes that key in the gateway, and drops the returning live chat event before the matching runId can be considered. The open linked PR is the implementation candidate, so the issue should wait for review and merge rather than being closed now.

## What This Changes

The issue reports that Control UI live chat updates can be dropped when the UI sends a raw session key such as `main` while the gateway streams events for the same run under the canonical key `agent:main:main`.

## Best Possible Solution

The end state should be a small Control UI fix that either normalizes the active session to the gateway canonical key or accepts chat events for the active client run when runId matches, with regression coverage for raw `main` versus canonical `agent:main:main` and a guard for unrelated runs.

## Reproduction Assessment

Yes. A source-level reproduction is clear: create chat state with `sessionKey: "main"` and `chatRunId: "run-1"`, then pass a `chat` payload with `sessionKey: "agent:main:main"` and the same runId; current `handleChatEvent` returns `null` at the session-key check before runId matching.

## Solution Assessment

Yes for the proposed direction, but it is not landed on main yet. The narrow maintainable fix belongs in the Control UI event/session normalization path, and the linked PR's active-run fallback is plausible if it preserves the unrelated-run guard.
