## Summary

Keep this issue open. Current main at 0f3a9d812b40cbc8e545abcb80a1f12a93946cf6 still sends chat from the UI with the raw active session key, the gateway canonicalizes raw main aliases for emitted chat events, and the UI drops those events before checking the matching runId. The provided GitHub context also shows #73721 is open and unmerged with closing syntax for this issue, so the maintainer path is to review/land or close that PR before closing this issue.

## Best Possible Solution

Keep the issue open and review #73721 as the implementation candidate. The narrow product fix belongs in the Control UI chat event path: either normalize the active UI session to the gateway canonical key before sending, or accept chat events for the active chatRunId when the sessionKey differs only because the gateway emitted a canonical alias. Add a regression test for state.sessionKey = "main", payload.sessionKey = "agent:main:main", and matching runId, plus a negative case where neither session nor runId matches.
