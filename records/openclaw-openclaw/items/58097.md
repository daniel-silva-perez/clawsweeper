## Summary

Keep this PR open. Current main still has the Nextcloud Talk signature verifier's early length-mismatch return and manual XOR comparison, so the hardening is still relevant; the proposed approach looks functionally sound from static review but needs maintainer/security review and a rebase before it can land.

## What This Changes

The PR updates Nextcloud Talk webhook signature verification to use padded `crypto.timingSafeEqual` and adds a regression test that checks the compare path still runs for truncated signatures.

## Best Possible Solution

Land a narrow, rebased Nextcloud Talk signature hardening that removes the length-mismatch early return, uses a repository-approved constant-time comparison path, and keeps a regression test proving truncated signatures do not bypass the compare path.

## Reproduction Assessment

Yes. Current main visibly reproduces the issue by inspection: `verifyNextcloudTalkSignature` returns at the length check before the compare path, and the PR's proposed test gives a concrete truncated-signature check for that invariant.

## Solution Assessment

Unclear as the exact best implementation. The padded `timingSafeEqual` approach is sound and matches LINE, but the existing plugin SDK `safeEqualSecret` helper is the shared length-safe string comparison path already used by Feishu, so maintainers should choose one consistently when rebasing.
