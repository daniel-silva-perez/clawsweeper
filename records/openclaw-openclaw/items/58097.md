## Summary

Keep this PR open. Current main still verifies Nextcloud Talk webhook signatures with an immediate length-mismatch return and a manual XOR loop, so the security hardening this PR targets remains useful; the proposed branch replaces that path with padded `timingSafeEqual` and adds the requested regression coverage. The main follow-up is maintainer review/rebase, especially because current main now also exposes a shared `safeEqualSecret` helper that may be the preferred final shape.

## What This Changes

The PR replaces Nextcloud Talk webhook signature's length early-return and manual XOR comparison with padded `crypto.timingSafeEqual`, and adds a regression test ensuring mismatch-length signatures still invoke the compare path.

## Best Possible Solution

Land a rebased Nextcloud Talk hardening that removes the length early-return/manual XOR comparison, keeps invalid signature behavior unchanged, and preserves a focused mismatch-length regression test; using `safeEqualSecret` from `plugin-sdk/security-runtime` would align the final code with the shared helper now present on main.

## Reproduction Assessment

Yes. The current-main code path is reproducible by inspection: any non-empty signature whose length differs from the SHA-256 hex digest length exits at `extensions/nextcloud-talk/src/signature.ts:28` before the compare loop, which is the reported timing shortcut.

## Solution Assessment

Yes for the functional security fix: the PR pads buffers before `timingSafeEqual`, avoids the early length branch, and keeps the final equality result gated on equal byte length. The narrow safer alternative on current main is to call the existing shared `safeEqualSecret` helper instead of duplicating its logic.
