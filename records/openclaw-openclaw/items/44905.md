## Summary

Keep open. Current main fixes some related subsets, including commentary-phase suppression and stripping exact silent text from media payload captions, but the shared outbound/Discord path still lacks a guard for raw tool-routing text, envelope field names, generic tool-call JSON, and REPLY_SKIP/ANNOUNCE_SKIP-style control tokens before Discord delivery.

## What This Changes

The issue asks OpenClaw to prevent Discord from exposing internal LLM/tool-control artifacts such as NO_REPLY, commentary routing, tool-call envelopes, and control sentinels as visible channel messages.

## Best Possible Solution

Add a conservative delivery-boundary sanitizer for internal trace/control artifacts, backed by a Discord send-level backstop for bypass paths, with regression tests that preserve legitimate prose, code fences, Discord formatting, media-only sends, and the existing silent-reply policy.

## Reproduction Assessment

Yes. A focused static reproduction is a Discord outbound payload containing a raw tool-routing line such as `to=functions.exec` or a small tool JSON envelope; current main shows that text passes through shared delivery and Discord send without a matching sanitizer or final guard.

## Solution Assessment

No, current main is not the complete fix. The narrow maintainable direction is a shared delivery-boundary filter plus a Discord adapter/send backstop, not only the existing commentary-phase and exact NO_REPLY handling.
