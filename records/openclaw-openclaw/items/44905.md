## Summary

Keep open. Current main covers some related subsets, including commentary-phase suppression and exact NO_REPLY media-caption normalization, but the Discord/shared outbound boundary still does not suppress raw tool-routing text, envelope field names, generic tool-call JSON, or REPLY_SKIP/ANNOUNCE_SKIP-style control tokens before user-visible Discord delivery.

## What This Changes

Prevent Discord from delivering internal LLM tool-routing, control-token, commentary, and raw tool-call artifacts as visible channel messages.

## Best Possible Solution

Add a conservative delivery-boundary sanitizer for internal trace/control artifacts, backed by a Discord send-level backstop for bypass paths, while preserving legitimate prose, code fences, Discord-native formatting, media-only sends, and the existing silent-reply policy.

## Security Review

Status: needs_attention

Summary: This issue is security/privacy-sensitive because current outbound paths can still treat internal tool routing and control text as user-visible Discord content.

Concerns:

- **[medium] Internal traces can reach outbound channel text:** `src/infra/outbound/deliver.ts:484`
  - body: The shared delivery path only strips runtime scaffolding and applies channel-specific text hooks; it lacks a universal guard for tool-routing, envelope-field, raw tool JSON, and control-token artifacts before channel delivery.
  - confidence: 0.87
- **[medium] Discord send lacks a final control-token guard:** `extensions/discord/src/send.outbound.ts:307`
  - body: The Discord send path forwards normalized text to media or text send helpers without a final exact-token or tool-envelope suppression check, so bypasses above this layer can still become channel messages.
  - confidence: 0.82

  - body: The diagnostics helper warns when assistant output looks like a tool call, but it explicitly treats that content as text and does not prevent delivery leakage.
