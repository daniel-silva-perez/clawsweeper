## Summary

Keep open. Current main contains partial fixes for commentary-phase output and exact NO_REPLY media captions, but it still lacks a shared delivery-boundary or Discord send backstop for raw tool-routing lines, envelope field names, generic tool-call JSON arguments, and REPLY_SKIP/ANNOUNCE_SKIP-style control tokens.

## What This Changes

The issue reports that Discord can expose internal tool-call/control artifacts such as NO_REPLY, commentary routing text, to=functions lines, envelope fields, raw tool JSON, and REPLY_SKIP-style sentinels as visible channel messages.

## Best Possible Solution

Add a conservative shared delivery-boundary sanitizer for internal trace/control artifacts, keep legitimate prose/code/media delivery intact, and add a Discord send-level backstop for bypass paths. Cover the remaining raw routing/envelope/control-token cases with focused outbound and Discord regression tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction is a Discord outbound payload whose text is `to=functions.memory_search`, `recipient_name`, a small tool-call argument JSON object, or `REPLY_SKIP`; current main shows those values can pass through shared normalization and Discord send without a matching sanitizer or final guard.

## Solution Assessment

No. Current main is only a partial solution; the narrow maintainable path is a maintainer-reviewed shared outbound filter plus a Discord backstop, not adding Discord to the plain-text HTML sanitizer list or relying only on extraction helpers.
