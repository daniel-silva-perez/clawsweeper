## Summary

Keep open. Current main has partial fixes for commentary-phase delivery and the exact media-caption NO_REPLY case, but it still lacks a shared outbound/Discord delivery-boundary guard for raw `to=functions.*` routing lines, envelope field names, generic tool-call JSON arguments, and `REPLY_SKIP`/`ANNOUNCE_SKIP` control tokens.

## What This Changes

The issue requests suppressing Discord-visible internal model/tool traces and control tokens before any user-facing channel delivery.

## Best Possible Solution

Add a conservative shared delivery-boundary sanitizer for internal trace/control artifacts, preserve legitimate prose/code/media delivery, and add a Discord send-level backstop for bypass paths. Keep the existing commentary-phase and exact NO_REPLY media fixes, but cover the remaining raw routing/envelope/control-token cases with targeted regression tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction is a Discord outbound payload whose text is `to=functions.memory_search`, `recipient_name`, a small tool-call argument JSON object, or `REPLY_SKIP`; current main shows those values can pass through shared normalization and Discord send without a matching sanitizer or final guard.

## Solution Assessment

No. Current main is only a partial solution; the narrow maintainable path is a maintainer-reviewed shared outbound filter plus a Discord backstop, rather than relying on source-specific extraction helpers or adding Discord to the plain-text HTML sanitizer list.
