## Summary

Keep this PR open. Current main still does not add the requested gateway auth entries to the startup dangerous-config flag collector: `gateway.auth.mode="none"` and trusted-proxy allow-all states are covered by security audit checks, but not by the collector that feeds gateway startup warnings. The PR is a focused security hardening candidate, though it needs a rebase into the current refactored collector shape plus targeted tests.

## Best Possible Solution

Keep this PR open as a focused gateway hardening candidate. Rebase it onto the current dangerous-flag collector split, add collector and startup-log tests for `gateway.auth.mode="none"` plus trusted-proxy allow-all states where absent and empty `allowUsers` behave the same, and decide whether to add schema-level whitespace-only `userHeader` rejection with any generated schema follow-through.

## Risks / Open Questions

- The PR patch targets the older single-file collector shape; current main has split dangerous flag collection across `src/security/dangerous-config-flags.ts`, `src/security/dangerous-config-flags-core.ts`, and `src/security/core-dangerous-config-flags.ts`, so it needs a rebase before review can be completed.
- Trusted-proxy startup warning logic should cover both `allowUsers: undefined` and `allowUsers: []`, matching runtime and audit semantics.
- Because this is gateway authentication hardening, maintainers should require targeted regression tests for collector output and startup warning behavior before merge.
- Schema-level whitespace rejection for `trustedProxy.userHeader` may require generated schema/docs alignment if maintainers accept it in addition to the existing runtime startup guard.
