## Summary

Keep open. Current main and the latest tagged release still use a single non-paginated Feishu `documentBlock.list` call when clearing content before `feishu_doc` write, while the bundled skill promises replace-all behavior and the locked SDK exposes paginated `page_token` results. The EronFan comment and external fork PR are useful prior art, but that PR is still open outside `openclaw/openclaw` and the fix has not landed here.

## What This Changes

The issue reports that `feishu_doc` write only clears the first page of Feishu Docx blocks before inserting replacement content, leaving old blocks in large documents.

## Best Possible Solution

Land a focused Feishu plugin fix in `extensions/feishu/src/docx.ts`: paginate `client.docx.documentBlock.list` inside `clearDocumentContent` until no next page token remains, delete the full top-level range, and add regression coverage where `feishu_doc` write sees multiple block-list pages before inserting replacement content.

## Reproduction Assessment

Yes. A source-level or mocked reproduction is high confidence: make `documentBlock.list` return a first page with `data.page_token` and more root blocks on a second page, then current `writeDoc` will call the list method once and delete only the first page count. A live Feishu document was not used in this read-only pass.

## Solution Assessment

Yes, paginating `clearDocumentContent` is the narrowest maintainable fix for the reported write behavior. The external fork patch is directionally right, but the in-repo solution should also include a regression test for multi-page clearing before it lands.
