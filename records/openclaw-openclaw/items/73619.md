## Summary

Keep this issue open. Current main still selects an omitted message-tool `accountId` from the inherited `agentAccountId`/`defaultAccountId` before checking the calling agent's channel binding. The related subagent and cron binding paths improve some caller setup, but they do not fix the shared `message` tool conflict when both an inherited/default account and a different bound account are present.

## Best Possible Solution

Add a regression test where an omitted Slack or generic channel `message` call has both an inherited/default account and a different calling-agent binding. Then change the shared message-action precedence to: explicit tool `accountId` first, calling agent channel binding second for omitted args, and inherited/default account only as the fallback when no binding exists.

## Risks / Open Questions

- The fix should preserve explicit `accountId` overrides and only fall back to inherited `deliveryContext.accountId` or channel default account when the calling agent has no usable binding for the resolved channel.
- Because `runMessageAction` is shared across channel plugins, the regression test should be generic enough to protect Slack while avoiding an unintended precedence change for other multi-account channels.
