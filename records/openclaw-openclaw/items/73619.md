## Summary

Keep this issue open. Current main still turns an omitted message-tool account into the inherited agentAccountId/defaultAccountId before the outbound runner checks the calling agent's channel binding, so the reported Slack attribution problem remains when a parent/default account and a different bound child-agent account are both present. Related cron and subagent binding work improves missing-account cases, but it does not fix this shared message-tool precedence conflict.

## Best Possible Solution

Add a regression test where an omitted Slack or generic channel message call has both an inherited/default account and a different calling-agent channel binding. Then change the shared message-action precedence to explicit accountId first, calling agent binding second for omitted args, and inherited/default account only as the fallback when no binding exists. Keep the behavior generic in runMessageAction/createMessageTool so Slack is protected without hard-coding Slack-specific policy.

## Risks / Open Questions

- Changing precedence in the shared message runner affects all channel plugins, so the fix should preserve explicit accountId overrides and only prefer bindings for omitted args.
- Because createMessageTool currently writes inherited agentAccountId into params.accountId, the fix may need to preserve an explicit-vs-omitted signal instead of treating all params.accountId values the same.
- Slack uses account-specific bot/app/user tokens, so the current bug can cause real cross-workspace or wrong-identity sends in multi-account deployments.
