## Summary

Keep this issue open. Current main still resolves an omitted `message` tool `accountId` to the tool's ambient `agentAccountId` and passes it as `defaultAccountId` before `runMessageAction` can fall back to the calling agent's channel binding, so the reported Slack attribution problem remains actionable. The existing tests cover `defaultAccountId` and binding fallback separately, but not the conflict where an inherited/default account differs from the calling agent's Slack binding.

## Best Possible Solution

Add a regression test where an omitted Slack `message` tool `accountId` has an inherited/default account that differs from the calling agent's Slack binding, then update `createMessageTool`/`runMessageAction` so explicit `accountId` wins, the calling agent's channel binding wins for omitted args, and inherited `deliveryContext.accountId` is only the no-binding fallback.

## Evidence

That means an omitted tool argument still becomes an account selection before binding fallback runs.

## Risks / Open Questions

- The fix needs tight precedence: explicit tool `accountId` must still win, the calling agent's channel binding should win only for omitted tool args, and inherited/default delivery context should remain a fallback when no binding exists.
- Secret scoping in `createMessageTool` currently uses the ambient account before dispatch; any fix should keep command-time secret resolution aligned with the final selected account.
