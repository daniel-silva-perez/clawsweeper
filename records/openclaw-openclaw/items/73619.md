## Summary

Keep this issue open. Current main still has the requested precedence bug in the message tool itself: omitted `accountId` is filled from the inherited/current delivery account before `runMessageAction` can use the calling agent's channel binding. There are useful subagent and cron binding improvements on current main, but they do not fix the core `message` tool conflict when both an inherited/default account and an agent binding exist.

## Best Possible Solution

Add a regression test where an omitted Slack `message` call has both an inherited/default account and a different calling-agent channel binding. Then change the precedence so explicit tool args win, the calling agent's binding wins for omitted args, and inherited `deliveryContext.accountId` or `defaultAccountId` is only the fallback when no binding exists.

## Evidence

- **Message tool still injects inherited account:** `createMessageTool` resolves `accountId` as the explicit tool arg or `agentAccountId`, writes that value into `params.accountId`, and passes it to the runner as `defaultAccountId`. When `agentAccountId` came from inherited delivery context, the omitted tool arg is no longer distinguishable from an explicit/default account.
