## Summary

Keep this issue open. Current main still resolves an omitted message-tool account through the inherited/default account before the calling agent's channel binding, and PR #74052 is the active unmerged fix path, so this should stay open until that PR lands or is replaced.

## What This Changes

The issue requests changing omitted `message` tool `accountId` resolution so Slack sends default to the calling agent's configured channel account instead of an inherited delivery-context account.

## Best Possible Solution

Land PR #74052 or an equivalent shared message-routing fix that preserves explicit `accountId`, then prefers the calling agent's channel account binding for omitted values, and falls back to inherited/default account only when no binding exists.

## Reproduction Assessment

Yes. A focused `runMessageAction` regression can configure both `defaultAccountId` and a different binding for `agentId`, omit `params.accountId`, and observe current main choose the default before the binding.

## Solution Assessment

Yes. The linked PR direction is the narrowest maintainable solution because the bug is in shared message account precedence and wrapper behavior, not Slack delivery itself.
