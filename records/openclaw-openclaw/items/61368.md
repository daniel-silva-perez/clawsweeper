## Summary

Keep open. Current main still exposes only the narrow allowlisted `agents.files.*` bootstrap-file surface, while the requested full workspace CRUD API and Control UI file manager are not implemented. An open, unmerged implementation PR already says it closes this issue, so the right cleanup action is to leave the issue open until that PR is reviewed, landed, replaced, or closed.

## What This Changes

Add a full `agents.workspace.*` gateway API and Control UI workspace file manager for browsing, reading, writing, deleting, moving, uploading, downloading, and previewing files inside agent workspaces for remote or container deployments.

## Best Possible Solution

Keep this issue open while maintainers decide the product boundary and review #62417 or a narrower replacement. If accepted as core, the end state should be typed Gateway protocol methods, explicit read/admin scopes, fs-safe boundary enforcement, resource limits, docs, focused tests, and a Control UI file manager that does not weaken the existing bootstrap-file API.

## Reproduction Assessment

Not applicable as a feature request, but the current-main verification path is high confidence: exact search plus method registry, protocol schema, server handler, UI tabs, tests, and docs all show only the existing allowlisted `agents.files.*` surface.

## Solution Assessment

Unclear. A core Gateway/UI API may be appropriate for remote operator workflows, but VISION.md and the plugin SDK also support plugin-owned Gateway methods and UI descriptors, so maintainers should decide whether this belongs in core, a trusted plugin, or a smaller API seam before landing the open PR.
