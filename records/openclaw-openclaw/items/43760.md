## Summary

Keep open. Current main still cannot express an independent memorySearch fallback endpoint/model: the public schema and types keep `fallback` as a string, the resolver produces one shared `remote` block, and both fallback creation paths reuse that shared provider configuration. The closed related PR was superseded by this narrower issue, so the requested capability still needs repo work.

## What This Changes

The issue requests separate `memorySearch` fallback provider configuration so the primary embedding provider and fallback can use independent model and remote endpoint settings.

## Best Possible Solution

Add a backward-compatible fallback object form for memorySearch that can carry fallback provider, model, remote, headers, input-type, dimensionality, and local options while preserving the current string shorthand. Thread that resolved fallback config through validation, public types, provider creation, runtime fallback activation, docs/help/generated schema, and focused regression tests.

## Reproduction Assessment

Yes. A static reproduction is high-confidence: the reported fallback object is rejected by the current strict schema, and the string fallback path reuses the single resolved `remote` object that Gemini reads before provider defaults.

## Solution Assessment

Yes. A fallback object form is the most maintainable fix because it keeps existing configs compatible while avoiding more one-off top-level keys such as `fallbackRemote` and `fallbackModel`.
