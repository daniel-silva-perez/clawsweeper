## Summary

Keep this issue open. Current main still models `memorySearch.fallback` as a provider string only, resolves one shared top-level `remote` block, and both provider-creation fallback paths reuse that shared request. The linked implementation candidate, PR #44878, is closed unmerged, so there is no shipped or canonical replacement fix to close against.

## Best Possible Solution

Keep this issue open. The best implementation is a backward-compatible config extension that preserves string fallback syntax while adding an object fallback form such as `{ provider, model, remote }`, then threads fallback-specific model, remote baseUrl/apiKey/headers, input-type, and dimensionality through `MemorySearchSchema`, `MemorySearchConfig`, `ResolvedMemorySearchConfig`, `resolveMemorySearchConfig`, immediate provider creation, runtime fallback activation, docs/help/labels/generated schema, and regression tests for Gemini primary plus OpenAI-compatible fallback remote/model behavior. Closed PR #44878 is useful prior art but would need review or replacement before this can close.

## Risks / Open Questions

- Closing this now would drop a still-present config limitation for mixed embedding providers such as Gemini primary plus OpenAI-compatible fallback endpoint/model.
- A custom `models.providers.<id>` setup may help some endpoint-routing cases, but current `memorySearch` fallback still lacks an explicit fallback model/remote contract and the fallback request path still shares `settings.remote`.
- A correct fix changes public config shape, schema help/labels/generated schema, and runtime behavior, so it should preserve `fallback: "providerId"` compatibility and include regression coverage.
