## Summary

Keep open. Current main still exposes `memorySearch.fallback` as a provider-id string with a single shared `remote` block, and both startup/runtime fallback paths reuse the primary memory-search options. The related broad issue is closed in favor of this narrower request, and I found no open PR with closing syntax for this issue.

## What This Changes

Allow `memorySearch` fallback embeddings to use an independent provider, model, and remote endpoint instead of sharing the primary `memorySearch.remote` configuration.

## Best Possible Solution

Add a backward-compatible fallback object form for memorySearch that preserves the existing string shorthand while allowing fallback provider, model, remote, headers, input-type, dimensionality, and local settings to resolve independently through schema, public types, runtime provider creation, docs, generated schema/help, and focused regression tests.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence reproduction path: the requested fallback object is rejected by the strict schema, and the current string fallback path reuses the single resolved `remote` block that Gemini reads before its provider defaults.

## Solution Assessment

Yes. A fallback object is the most maintainable fix because it keeps existing configs compatible and avoids a growing set of one-off top-level keys such as `fallbackRemote` and `fallbackModel`.
