## Summary

Keep open. Current main still has the reported memorySearch limitation: fallback is a single provider string, and both startup and runtime fallback paths reuse the same resolved remote/model/input/local settings. The related unmerged PR #44878 and closed broader issue #55590 leave this narrower fallback-provider/model/remote config gap as active work.

## What This Changes

The issue requests separate memorySearch fallback provider, model, and remote settings so primary and fallback embedding providers can use independent endpoints.

## Best Possible Solution

Add a backward-compatible fallback object form for memorySearch that preserves the existing string shorthand while allowing fallback provider, model, remote, headers, input type, dimensions, and local settings to resolve independently through schema, public types, runtime provider creation, docs/generated config help, and regression tests.

## Reproduction Assessment

Yes. Static current-main inspection gives a high-confidence reproduction path: the requested fallback object is rejected by the schema/type contract, and the existing string fallback path reuses the single resolved remote block that Gemini reads before provider defaults.

## Solution Assessment

Yes. A fallback object is the narrowest maintainable direction because it keeps existing configs compatible while avoiding a growing set of separate top-level keys such as fallbackRemote and fallbackModel.
