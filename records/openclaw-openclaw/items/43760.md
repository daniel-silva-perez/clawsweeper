## Summary

Current main still does not implement separate `memorySearch` fallback remote/model configuration. The public schema/type accept `fallback` only as a provider string, the resolver emits one shared `remote` block, and both immediate and runtime fallback paths reuse that shared request. Related PR #44878 is closed unmerged and broader related issue #55590 is closed, so there is no canonical merged fix to close against.

## Best Possible Solution

Keep this issue open. The best implementation is a backward-compatible config extension that preserves string fallback syntax while adding an object fallback form such as `{ provider, model, remote }`. It should thread fallback-specific model, remote baseUrl/apiKey/headers, input-type, and dimensionality decisions through `MemorySearchSchema`, `MemorySearchConfig`, `ResolvedMemorySearchConfig`, `resolveMemorySearchConfig`, immediate provider creation, runtime fallback activation, docs/help/labels/generated schema, and regression tests for Gemini primary plus OpenAI-compatible fallback remote/model behavior. Closed PR #44878 is useful prior art but would need review or replacement before this can close.

## Risks / Open Questions

- Closing this would drop a still-present config limitation for mixed embedding providers, especially Gemini primary with an OpenAI-compatible fallback endpoint/model.
- A provider-id or `models.providers` workaround may help some endpoint-routing cases, but it does not provide the requested fallback-specific `memorySearch` model/remote contract and current fallback paths still pass the shared `memorySearch.remote`.
- A correct fix changes a public config surface and should preserve `fallback: "providerId"` compatibility while adding regression coverage.
