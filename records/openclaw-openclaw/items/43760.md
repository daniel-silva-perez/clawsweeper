## Summary

Current main still has the reported memorySearch limitation: fallback is still a provider string, the resolver produces one shared remote block, and both immediate and runtime fallback paths reuse that shared configuration. The related implementation PR is closed unmerged, so this remains valid follow-up work rather than cleanup-close material.

## What This Changes

The issue requests a separate memorySearch fallback provider configuration so a primary embedding provider and fallback embedding provider can use independent model and remote endpoint settings.

## Best Possible Solution

Add a backward-compatible fallback object form to the memory search config contract and thread fallback-specific provider, model, remote, headers, input-type, dimensionality, and local options through validation, resolved config, provider creation, docs/help/generated schema, and focused regression tests while keeping the existing string form intact.
