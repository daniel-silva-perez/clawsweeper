## Summary

Keep open. Current main still lacks a `promptMode` parameter on the `sessions_spawn` tool schema and native spawn forwarding path, even though the embedded prompt builder already supports `"none"`; this is a real API gap, but the raw-prompt behavior needs maintainer design before an automated fix.

## What This Changes

Expose an optional `promptMode: "full" | "minimal" | "none"` parameter on `sessions_spawn` so native sub-agents can be spawned with deterministic blank prompt context.

## Best Possible Solution

Add a typed native `sessions_spawn.promptMode` surface only after maintainers approve the policy: preserve the existing default, document the parameter, keep sandbox/tool policy enforcement unchanged, and handle `"none"` so the assigned task and attachment receipts still reach the child without injecting workspace prompt context.

## Reproduction Assessment

Yes. A static current-main check is sufficient: the tool schema and spawn forwarding path do not expose `promptMode`, while the prompt builder and embedded runner already support internal prompt-mode selection.

## Solution Assessment

Unclear. The requested direction is plausible and narrow, but simply forwarding the field is not the best fix because `promptMode: "none"` currently bypasses `extraSystemPrompt`; the implementation needs an explicit task/attachment delivery design.
