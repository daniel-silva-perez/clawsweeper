## Summary

Current main already keeps OpenAI-Codex off the OpenAI WebSocket transport, but it does not implement the proposed soft/hard `before_tool_call` contract, and the current PR diff still has two concrete P2 correctness defects plus a stale branch surface against the current hook type layout.

## What This Changes

The PR restricts OpenAI WebSocket transport selection, adds soft/hard `before_tool_call` block semantics with docs and tests, and changes live test `.profile` environment loading.

## Best Possible Solution

Refresh the branch against current main, keep the current endpoint-aware WebSocket guard, implement `blockMode` on the current hook type and runner/consumer paths, clear approval state whenever a soft block is cleared, make profile loading continue until profile-derived variables or static parsing succeed, and update the public plugin docs plus focused regression tests.

Yes. Static inspection gives a high-confidence reproduction path: merge a soft block with `requireApproval`, then a lower-priority `{ block: false }`, and the PR still preserves approval state; similarly, a first shell that emits an unrelated new env var can stop `.profile` fallback early.

## Solution Assessment

No, not as-is. The direction is plausible, but current main already has the better Codex transport guard, and the remaining hook/env changes need a rebase plus the two P2 defects fixed before this is the narrow maintainable solution.

  - body: When a lower-priority hook returns `{ block: false }`, this branch clears the soft block fields but the merged result can still retain `acc.requireApproval`. The consumer then prompts for approval even though the soft block was cleared, so the clear operation is incomplete.
