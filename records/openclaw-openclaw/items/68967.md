## Summary

Keep this PR open. Current main already implements the Google Chat threaded-reply portion, but it does not implement the PR’s central `sessionThread` option or per-thread session isolation. The current Google Chat monitor still resolves routing with `peer.id = spaceId`, stores `SessionKey: route.sessionKey`, and the config type/schema/docs expose no `sessionThread` or `threadSessionMode` setting.

## Best Possible Solution

Keep the PR open and review/adapt it as the implementation candidate for `channels.googlechat.sessionThread`: add the config/schema/generated docs surface, derive a thread-scoped session key without changing space-based routing, pass that key through envelope/session metadata, preserve explicit outbound reply targets, and keep the current threaded send behavior covered by tests.
