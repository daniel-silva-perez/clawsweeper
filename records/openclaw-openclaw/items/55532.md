## Summary

Keep open. Current main has a shipped Feishu WebSocket startup retry loop and heartbeat defaults, but the reported post-start reconnect/backoff behavior is still not fully owned by OpenClaw, and open unmerged PRs explicitly claim this issue with closing syntax.

## What This Changes

The issue reports Feishu WebSocket system-busy/PingInterval failures causing fixed-interval reconnect loops and asks for exponential backoff, closed-socket race handling, and optional health checking.

## Best Possible Solution

Resolve one canonical Feishu reconnect PR with a Feishu-owned wrapper or supervisor that avoids global `WSClient` prototype mutation, cancels pending waits on shutdown, resets backoff after successful recovery, avoids stacking delays on the SDK loop, and adds focused tests for SDK retry exhaustion and PingInterval handling. Split the token-cache poisoning concern into an upstream or dependency-specific follow-up if it is not fixed locally.

## Reproduction Assessment

Unclear. The report includes concrete logs and the pinned SDK source supports the failure theory, but current main only has deterministic coverage for startup rejection, not the post-start reconnect-exhaustion path.

## Solution Assessment

No for closing now. The current implementation is a partial repair for startup failures and heartbeat defaults; the narrow maintainable path is to review or land a Feishu-owned reconnect supervisor PR before closing this issue.
