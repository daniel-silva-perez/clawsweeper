## Summary

Keep open. Current main still restores the same-gateway persisted Control UI session selection from browser storage and treats non-main routed keys as explicit user selections during session-default normalization; only token-only launch links reset to main. That makes the reported stale Feishu/direct-session restore plausible and not yet fixed.

## What This Changes

This issue asks Control UI startup restore to fall back to the real main session instead of restoring persisted historical routed session keys such as `agent:*:*:direct:*` or `agent:*:*:group:*` unless a session was explicitly requested.

## Best Possible Solution

Control UI should distinguish explicit session navigation from browser-storage startup restore. If maintainers agree with the UX direction, add a startup-only guard that maps persisted routed direct/group channel keys back to the gateway main session while preserving explicit `?session=` links and deliberate in-app session switches, with focused storage/settings/gateway UI regression coverage.

## Reproduction Assessment

Yes. Seed `openclaw.control.settings.v1` or the scoped `sessionsByGateway` entry with `sessionKey` and `lastActiveSessionKey` set to `agent:main:feishu:direct:ou_x`, then load Control UI without `?session=` or a token-only launch; the current code path restores that key into `host.sessionKey` and preserves it during defaults normalization.

## Solution Assessment

Unclear pending maintainer product judgment. The proposed fallback is technically narrow, but it conflicts with the existing documented behavior that reopening the same gateway resumes the last session, so maintainers should first decide whether routed direct/group sessions are excluded from persisted startup restore.
