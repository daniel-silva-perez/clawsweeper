## Summary

Current main still restores persisted Control UI session selections from local storage without filtering historical routed direct/group keys. The only implemented reset path I found is for tokenized URLs without an explicit session, and explicit `?session=` deep links are intentionally preserved. This remains a real, narrow restore-policy question rather than an implemented fix or a duplicate of the adjacent channel-delivery reports.

## Best Possible Solution

Keep this issue open and implement a small Control UI restore-policy change: detect persisted routed session keys matching `agent:<agent>:<channel>:(direct|group):<id>` during passive startup/local-storage restore and fall back to the gateway-resolved main session, while keeping explicit `?session=` URLs and deliberate in-app session switches intact. Add focused regression coverage in the Control UI storage/startup/default-normalization tests for persisted direct and group keys.
