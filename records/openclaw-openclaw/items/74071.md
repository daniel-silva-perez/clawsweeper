## Summary

Keep open. Current main still treats `ANNOUNCE_SKIP` as a whole-message exact token, so the reported multi-line final-line case remains valid; PR #74136 is already the open closing implementation candidate and should be reviewed or replaced rather than spawning a second repair lane.

## What This Changes

Allow sub-agent announce suppression when a multi-line assistant output ends with `ANNOUNCE_SKIP` as a standalone final line.

## Best Possible Solution

Land PR #74136 or an equivalent narrow patch that preserves exact-token behavior, accepts a standalone final-line `ANNOUNCE_SKIP`, keeps `REPLY_SKIP` and `NO_REPLY` semantics unchanged, and updates focused helper/announce-flow tests plus docs or prompt wording if this becomes the intended contract.

## Reproduction Assessment

Yes. A source-level reproduction is `isAnnounceSkip("summary\nANNOUNCE_SKIP")`, which still returns false from the current helper; the announce call sites then treat that text as visible reply content.

## Solution Assessment

Yes. The narrow maintainable fix is to extend only announce-skip recognition to a standalone final line while preserving existing exact-token and silent-reply behavior; the open PR is the active implementation candidate for that path.
