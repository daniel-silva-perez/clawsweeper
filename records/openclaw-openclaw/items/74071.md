## Summary

Keep open. Current main still implements `isAnnounceSkip` as an exact trimmed-token comparison, so a multi-line sub-agent response ending with a standalone `ANNOUNCE_SKIP` still falls through as visible announce content. The issue is also paired with open PR #74136, which uses closing syntax for this behavior, so it should stay open until that PR lands or is replaced.

## What This Changes

The issue requests that sub-agent announce suppression accept multi-line output whose final standalone line is `ANNOUNCE_SKIP`, instead of only accepting replies that are exactly the token.

## Best Possible Solution

Review and land PR #74136, or an equivalent narrow patch, so announce suppression accepts the existing exact token plus a standalone final-line `ANNOUNCE_SKIP` while preserving `REPLY_SKIP` and `NO_REPLY` semantics. Add focused helper coverage and one announce-flow regression test, then align the prompt and docs wording.

## Reproduction Assessment

Yes. A source-level reproduction is `isAnnounceSkip("summary\nANNOUNCE_SKIP")`, which returns false on current main because the helper compares the whole trimmed string to the token; the announce paths then treat that text as visible content.

## Solution Assessment

Yes, the requested direction is the narrowest maintainable fix if it only accepts a standalone final-line token. The active PR #74136 is already the implementation candidate for that path, so the maintainer action is normal PR review rather than a separate repair job.
