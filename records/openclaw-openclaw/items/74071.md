## Summary

Keep open. Current main still treats `ANNOUNCE_SKIP` as an exact trimmed reply, so a multi-line sub-agent output ending with the token remains visible announce content. The issue is also paired with open PR #74136, which explicitly uses closing syntax for this behavior, so the issue should stay open until that PR lands or is replaced.

## What This Changes

The issue requests that `isAnnounceSkip` also suppress multi-line sub-agent replies whose final standalone line is `ANNOUNCE_SKIP`.

## Best Possible Solution

Review and land PR #74136, or an equivalent narrow patch, so announce suppression accepts the canonical exact token plus a standalone final-line `ANNOUNCE_SKIP` while preserving `REPLY_SKIP` and `NO_REPLY` semantics. Add focused helper coverage and one announce-flow regression test, then align the docs and prompt wording.

## Reproduction Assessment

Yes. A focused source-level reproduction is `isAnnounceSkip("summary\nANNOUNCE_SKIP")`, which returns false on current main because the helper compares the whole trimmed string to the token; the announce output and delivery paths then treat the text as visible content.

## Solution Assessment

Yes. The narrowest maintainable fix is to change only `isAnnounceSkip` to recognize a standalone final-line token, preserve stricter `REPLY_SKIP` behavior, and cover both helper and announce-flow paths; PR #74136 is already the active candidate for that approach.
