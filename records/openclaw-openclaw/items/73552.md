## Summary

Keep open. Current main still supports SearXNG category searches but does not retry `general` after an empty categorized response, and the generic web-search runtime only falls back on unavailable/erroring providers, not successful zero-result payloads.

## What This Changes

Request automatic fallback to a general SearXNG/web-search category when a category-specific search returns no results.

## Best Possible Solution

Implement the fallback inside the SearXNG plugin client: after an empty result set from a non-general category request, retry once with `general` while preserving the existing endpoint trust checks, timeout, count cap, language, cache behavior, and web-content wrapping. Avoid changing generic core provider fallback semantics because zero results can be a valid successful response for other providers.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is `runSearxngSearch` with `categories: "weather"` and a mocked SearXNG JSON response of `{"results":[]}`; current code builds only the categorized URL and returns `count: 0`.

## Solution Assessment

Yes, if done provider-locally. A SearXNG-only empty-category retry is narrower and safer than treating all zero-result provider responses as fallback failures in core.
