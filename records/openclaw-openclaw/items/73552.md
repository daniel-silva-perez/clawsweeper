## Summary

Keep open. Current `main` still performs a single SearXNG category request and returns a successful zero-result payload without retrying `general`; the generic web-search fallback path only handles provider unavailability/errors, so the requested reliability behavior remains missing.

## What This Changes

The issue requests that category-specific SearXNG/web_search requests retry once with the general category when the category response is empty.

## Best Possible Solution

Add the fallback in the SearXNG plugin client rather than in the core provider loop: after an empty categorized result, retry once with `categories=general` when the original effective category set is non-empty and does not already include `general`, while preserving cache, timeout, endpoint validation, count, language, and result wrapping behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is `runSearxngSearch` with `categories: "weather"` and a mocked SearXNG JSON body of `{"results":[]}`; current code would build only the categorized URL and return `count: 0`.

## Solution Assessment

Yes, provider-local fallback is the best direction. Changing the generic core fallback to treat zero results as failure would be broader and risk incorrect retries for providers or queries where zero results is a valid successful answer.
