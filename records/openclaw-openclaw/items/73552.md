## Summary

Keep open. Current main still supports SearXNG category parameters and generic provider fallback, but it does not retry a successful categorized SearXNG response that parses to zero results as a `general` search, so the requested reliability behavior remains missing and narrowly fixable.

## What This Changes

Request to retry category-specific web searches as a general search when the category-specific request returns no results.

## Best Possible Solution

Keep category-specific routing, but add a SearXNG-owned one-shot retry: when a non-general category search returns zero parsed results, retry the same query/count/language/base URL with `categories=general`, return those results if present, and document/test the behavior inside the SearXNG plugin surface.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to mock a SearXNG JSON endpoint so the first request with `categories: "weather"` returns `{ "results": [] }` and a second `general` request would return results; current `runSearxngSearch` only performs the first request and returns `count: 0`.

## Solution Assessment

Yes, if scoped to the SearXNG provider/client. Core already handles provider-level fallback, but category semantics are provider-owned, so a provider-local retry is the narrowest maintainable fix without adding SearXNG-specific policy to core.
