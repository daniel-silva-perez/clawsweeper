## Summary

Keep open. Current main still returns an empty SearXNG result payload after a categorized request succeeds with zero parsed results; the generic web-search runtime only falls back across providers for unavailable providers, thrown execution errors, or structured missing-key errors, not for successful empty result sets. The existing ClawSweeper comment is consistent with current main and this remains a narrow provider-owned fix candidate.

## What This Changes

The issue requests retrying a categorized web search as a general SearXNG search when the categorized provider request returns no results.

## Best Possible Solution

Keep category-specific routing, but add a SearXNG-owned one-shot retry: when a non-general category request returns zero parsed results, retry the same query with categories=general, return those results if present, and document the behavior without moving SearXNG-specific category policy into core.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to mock a SearXNG JSON endpoint so the first request with categories="weather" returns {"results":[]} and a second general request would return rows; current runSearxngSearch only performs the first request and returns count: 0.

## Solution Assessment

Yes, a provider-local fallback is the narrowest maintainable direction. Core already handles provider availability/error fallback, while SearXNG category semantics belong in the SearXNG plugin/client.
