## Summary

Keep open. Current main supports SearXNG category searches, but it sends one categorized request and returns an empty successful payload without retrying `general`; the generic web-search runtime fallback only handles unavailable/erroring providers, not zero-result category responses. This is a narrow SearXNG plugin fix candidate, not implemented behavior or a duplicate close.

## What This Changes

The issue requests a fallback that retries web search with the general category when a category-specific search returns no results.

## Best Possible Solution

Implement the fallback inside the SearXNG plugin client: when the effective category set is non-empty and does not already include `general`, retry once with the general category after an empty parsed result set, while preserving the existing trusted endpoint, timeout, cache, count, language, and result-wrapping behavior. Add focused SearXNG unit tests and update docs/changelog if the new fallback is user-visible.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is `runSearxngSearch` with `categories: "weather"` and a mocked SearXNG JSON response of `{"results":[]}`: current code builds only the categorized URL and returns `count: 0` without a second `categories=general` request.

## Solution Assessment

Yes, if implemented provider-locally. A SearXNG-specific empty-category fallback is narrower and safer than changing generic core provider fallback semantics, because zero results can be a valid successful response for other providers and queries.
