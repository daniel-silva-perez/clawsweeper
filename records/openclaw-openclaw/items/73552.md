## Summary

Keep open. Current main supports SearXNG category parameters and provider-level fallback, but it does not retry the same SearXNG query as `general` when a categorized request succeeds with an empty result set, so the requested reliability behavior remains missing and is narrow enough to fix.

## What This Changes

Request to retry categorized web-search requests as a general search when the category-specific search returns no results.

## Best Possible Solution

Keep category-specific routing, but implement a SearXNG-owned one-shot fallback: when a non-general category search returns zero parsed results, retry the same query with `categories=general`, return those results if present, and cover the behavior in provider tests and docs without adding SearXNG-specific policy to core.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to mock a SearXNG JSON endpoint so the first request with `categories: "weather"` returns `{ "results": [] }` and a second `general` request would return results; current `runSearxngSearch` only performs the first request and returns `count: 0`.

## Solution Assessment

Yes, if scoped to the SearXNG provider/client. The core web-search runtime already handles provider fallback, but category semantics are provider-owned, so a provider-local retry is narrower and avoids embedding SearXNG category policy in core.
