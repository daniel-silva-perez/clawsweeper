## Summary

Keep open. Current main still accepts SearXNG categories and performs a single categorized request, then returns a successful zero-result payload without retrying `general`; the shared web-search runtime only falls through on provider unavailability or thrown errors, so the requested reliability behavior remains missing.

## What This Changes

Add a SearXNG web-search fallback that retries the same query with `general` when a categorized search returns no results.

## Best Possible Solution

Implement the retry inside the SearXNG plugin client: after an empty result from a non-general effective category set, retry once with `categories=general` while preserving endpoint validation, timeout, cache TTL, count, language, and result wrapping behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is `runSearxngSearch` with `categories: "weather"` and a mocked SearXNG JSON body of `{"results":[]}`; current main builds only the categorized URL and returns `count: 0`.

## Solution Assessment

Yes. A provider-local fallback is the narrowest maintainable fix because treating zero results as a generic runtime failure would change semantics for other providers and for queries where no results is a valid answer.
