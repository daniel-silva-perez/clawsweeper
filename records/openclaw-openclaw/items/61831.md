## Summary

Keep open. Current main has named browser profiles, Chrome MCP sessions cached by profile, per-session tab tracking/cleanup, and an opt-in sandboxed browser path, but the normal host browser tool still routes omitted `profile` calls from every agent to the same configured default browser profile. `agentSessionKey` is used for tab bookkeeping and lifecycle cleanup, not for selecting or creating an isolated per-agent Chrome process/profile.

## Best Possible Solution

Keep this open for a focused browser isolation decision. The strongest implementation path is either first-class agent-aware host browser profile resolution with deterministic CDP port/user-data-dir allocation and Chrome MCP cache separation, or clear documentation that sandboxed browser `scope: "agent"`/`"session"` is the supported isolation model while host browser profiles remain intentionally shared unless configured otherwise.

## Risks / Open Questions

- Optional sandboxed browser containers may be an acceptable supported isolation model for some deployments, but the host browser/profile behavior still needs explicit maintainer product decision or documentation.
- Closing this as implemented would conflate per-session tab cleanup with browser state isolation; cookies, localStorage, selected-page state, and Chrome MCP process state can still be shared by callers using the same profile.
