## Summary

Keep open. Current main has manual browser profiles, opt-in sandbox browser isolation, and session-tab cleanup, but ordinary non-sandbox browser tool calls with no `profile` still route through the shared default `openclaw` profile, and Chrome MCP sessions are cached by profile rather than by agent.

## What This Changes

The issue asks OpenClaw to give each agent an isolated Chrome process, CDP port, user profile, and browser tool lifecycle instead of sharing one browser instance across agents.

## Best Possible Solution

Define and document one supported browser isolation model for multi-agent deployments, then implement it consistently across host managed profiles, sandbox browser bridges, node proxy routing, and Chrome MCP existing-session profiles while preserving explicit profile overrides.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction path is two non-sandbox agents with the browser tool enabled, both omitting `profile`, then opening and listing tabs; both resolve to the shared default `openclaw` profile unless the operator explicitly configures separate profiles or sandbox browser isolation.

## Solution Assessment

Unclear. The requested per-agent Chrome instance is a plausible solution, but the best maintainable fix needs product/design review because OpenClaw already has manual profiles, sandbox-scoped browsers, node proxies, and existing-session Chrome MCP behavior that must remain compatible.
