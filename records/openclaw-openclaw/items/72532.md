## Summary

Keep this PR open, but do not merge it as-is. The underlying Feishu multi-bot group report is real against current main, and preserving the receiving bot's own group mention is a plausible fix direction, but the patch makes bot mention preservation global and regresses mention-prefixed slash commands in direct Feishu chats where the command pipeline still expects the bot mention to be stripped.

## What This Changes

The PR makes Feishu mention normalization preserve every mention placeholder as an `<at>` tag, including the receiving bot's own mention, and updates Feishu mention tests and comments to match that behavior.

## Best Possible Solution

Amend the Feishu plugin fix so group messages can preserve the receiving bot's own `<at>` context for multi-bot prompts while direct Feishu chats still keep mention-prefixed slash commands working. The narrowest path is to preserve bot tags only where the downstream command body is normalized, or to add an equivalent direct-chat command-body normalization, with regression coverage for both multi-bot group content and `@Bot /model` in p2p.

## Reproduction Assessment

Yes. Current main reproduces the reported source behavior with `parseFeishuMessageEvent` plus two Feishu mentions: the receiving bot's own placeholder is replaced with an empty string, and the PR diff would preserve it; the introduced regression is reproducible by a p2p `@_bot_1 /model` event because direct command parsing receives the preserved `<at>` tag before the slash.

## Solution Assessment

No. The proposed direction is close for group NO_REPLY, but making `botStripId` a global no-op is broader than the maintainable fix because direct Feishu command handling still depends on p2p bot mention stripping.
