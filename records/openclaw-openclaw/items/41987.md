## Summary

Keep open. The reported bug is still valid on current main, but this PR patches the legacy core browser-tool path while the active browser tool is registered from the bundled browser plugin, so the branch would not make `browser:screenshot` honor `agents.defaults.imageMaxDimensionPx`.

## What This Changes

The PR threads config into the legacy core browser tool and adds legacy browser-tool tests asserting `browser:screenshot` passes `imageSanitization` from `agents.defaults.imageMaxDimensionPx`.

## Best Possible Solution

Port the behavior into the active browser plugin: carry the plugin tool context config or live runtime-config getter into `createLazyBrowserTool` and runtime `createBrowserTool`, derive image sanitization limits from `agents.defaults.imageMaxDimensionPx` inside the plugin boundary, apply those limits to browser-generated image results, and cover configured plus default behavior in plugin-path tests.

## Reproduction Assessment

Yes. On current main, the `action: "screenshot"` path reaches `extensions/browser/src/browser-tool.ts:759` and calls `imageResultFromFile` without `imageSanitization`, so a config such as `agents.defaults.imageMaxDimensionPx=2000` cannot override the default 1200px limit for that result.

## Solution Assessment

No. The PR has the right fix idea, but it targets stale legacy core files; the maintainable fix is the same config plumbing in `extensions/browser` using the existing plugin tool-context config/runtimeConfig/getRuntimeConfig seam.
