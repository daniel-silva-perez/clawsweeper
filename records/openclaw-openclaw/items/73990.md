## Summary

Keep open. Current main still has the no-telemetry gap described here: transcript usage fallback is driven by provider usage/model snapshots, while a contentful transcript with no assistant usage telemetry still returns no usable snapshot and can leave status/session displays at zero-ish context. The related open #75357 is narrower transport telemetry work, not a superseding fallback fix.

## What This Changes

Request a conservative transcript-derived context estimate for local Llama/OpenAI-compatible sessions when provider usage telemetry is missing or zero.

## Best Possible Solution

Add a generic transcript fallback in the Gateway/status usage path that keeps positive provider usage first, then derives a conservative prompt/context token estimate from visible transcript content when no usable usage-derived context total exists, with regression coverage for no-telemetry local/OpenAI-compatible transcripts.

## Reproduction Assessment

Yes. The current `readLatestSessionUsageFromTranscript` test has a deterministic no-telemetry user/assistant transcript and expects `null`, and the status/session paths drop that missing snapshot before formatting context usage.

## Solution Assessment

Yes. A narrow extractor/status fallback is the maintainable direction because it preserves provider telemetry when present and only estimates from transcript text for the missing-telemetry case.
