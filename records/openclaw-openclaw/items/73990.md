## Summary

Keep open. Current main still recovers context usage only from provider usage fields or transcript entries with usable usage/model metadata; it does not estimate prompt/context tokens from non-empty transcript text when local/OpenAI-compatible telemetry is absent or zero.

## What This Changes

The issue requests a conservative transcript-derived context usage estimate for local/OpenAI-compatible sessions when provider usage telemetry is missing or zero.

## Best Possible Solution

Implement a generic Gateway session-usage fallback that preserves valid provider usage first, then uses a conservative visible-transcript text estimate only when no positive usage-derived context total is available, with regression coverage for local/OpenAI-compatible no-telemetry transcripts.

## Reproduction Assessment

Yes. A deterministic reproduction exists in the current test fixture: a transcript with user and assistant content but no usage telemetry returns `null`, and the status path drops that snapshot before formatting context usage.

## Solution Assessment

Yes. A narrow change in transcript usage extraction/status fallback is the maintainable path; provider transports and OpenAI-compatible usage normalization should remain authoritative when they report usable tokens.

Implement a generic transcript-derived fallback in the Gateway session usage path, most likely in `src/gateway/session-utils.fs.ts`, so non-empty visible transcript text can produce a conservative prompt/context token estimate when provider usage is missing or all zero. Preserve provider-reported usage, promptTokens, cache fields, costs, carried-forward fresh totals, and model identity whenever they are valid.
