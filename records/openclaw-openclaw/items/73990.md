## Summary

Keep open. Current main and latest `v2026.4.29` still recover context usage from provider usage fields or assistant entries with usage/model identity; a contentful transcript with no usage telemetry is still expected to return `null`, so local/OpenAI-compatible sessions can continue showing zero-ish context instead of a transcript-derived estimate.

## What This Changes

Request a generic fallback that estimates context usage from non-empty transcript content when local Llama/OpenAI-compatible providers omit or zero usage telemetry.

## Best Possible Solution

Implement a generic Gateway/status transcript fallback that preserves valid provider usage first, then estimates a conservative prompt/context token count from visible transcript text only when no positive usage-derived context total exists, with focused regression coverage for no-telemetry local/OpenAI-compatible transcripts.

## Reproduction Assessment

Yes. The current fixture in `src/gateway/session-utils.fs.test.ts` creates a user/assistant transcript with no usage telemetry and asserts `null`; the status code then drops that missing snapshot before formatting context usage.

## Solution Assessment

Yes. A narrow extractor/session-status fallback is the maintainable fix because provider transports and OpenAI-compatible usage normalization should remain authoritative whenever they report usable token telemetry.
