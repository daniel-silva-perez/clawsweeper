## Summary

Keep open. Current main still only recognizes direct Slack `<@botId>` mentions and configured regex mention patterns, with no `<!subteam^...>` parsing, user-group membership lookup, config field, scope, or docs support; the closed related PR was not merged and maintainer feedback points to an API-backed Slack user-group membership path instead.

## What This Changes

The issue requests that Slack user-group mentions wake eligible OpenClaw agents in channels where mention gating is enabled.

## Best Possible Solution

Implement the fix inside the Slack plugin: extract `<!subteam^S...>` tokens, resolve each subteam through Slack membership using a bounded cache, treat the message as explicit only when the current bot user ID is returned, and update setup/docs/scope diagnostics for `usergroups:read`. The config-only allowlist from the closed PR should not be the primary behavior unless maintainers explicitly ask for a fallback.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: with text `<!subteam^S...> hello`, current main leaves both direct Slack mention signals false, so a channel with `requireMention=true` reaches the `no-mention` skip path instead of dispatching.

## Solution Assessment

Yes for keeping this open and queueing a focused repair. The best remaining path is the maintainer-requested API-backed membership check, not closing this as implemented or reviving the config-only PR unchanged.
