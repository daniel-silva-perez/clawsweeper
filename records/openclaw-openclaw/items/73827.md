## Summary

Keep open. Current main still only treats direct Slack `<@botUserId>` tokens or configured regexes as explicit mentions, while the linked config-only PR was closed unmerged after maintainer feedback favoring an API-backed `usergroups.users.list` membership check.

## What This Changes

Support Slack user-group mentions so agents in a Slack user group can wake when addressed with `<!subteam^...>` in a channel.

## Best Possible Solution

Implement this inside the Slack plugin as an API-backed membership fallback: extract `<!subteam^S...>` tokens, call `ctx.app.client.usergroups.users.list` with bounded caching, treat the message as explicit only when `ctx.botUserId` is returned, fail closed on missing scope/errors, and update Slack docs/setup manifest/tests. Do not make a global config-only allowlist the primary behavior.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: with text `<!subteam^S...> hello`, current main sets `hasAnyMention=false` and `explicitlyMentioned=false`, so a Slack room with `requireMention=true` reaches the `no-mention` skip path instead of dispatching.

## Solution Assessment

Yes for the remaining maintainer-directed repair path. The closed config-only PR is not the best fix because it could wake a bot that is not an actual Slack user-group member; the narrower durable solution is a Slack-owned `usergroups.users.list` membership check plus docs/scope handling.
