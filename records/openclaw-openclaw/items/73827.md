## Summary

Keep open. Current main still treats Slack explicit mentions as direct `<@botUserId>` tokens or configured regex matches, with no `<!subteam^...>` or Slack user-group membership path; the earlier config-only PR was closed unmerged after maintainer feedback favoring an API-backed `usergroups.users.list` fallback.

## What This Changes

Request Slack channel mention gating to wake eligible OpenClaw agents when a Slack user-group token like `<!subteam^S...>` mentions a group containing their bot users.

## Best Possible Solution

Implement this in the Slack plugin as user-group membership resolution rather than a core config-only allowlist: extract `<!subteam^S...>` tokens, query Slack `usergroups.users.list` with a bounded cache, treat the message as explicit only when `ctx.botUserId` is a listed member, and update Slack setup/docs/scope diagnostics for `usergroups:read`. Keep any manual `subteamMentions` config as a separate follow-up only if maintainers still want an explicit fallback.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: with text `<!subteam^S...> hello`, current main sets `hasAnyMention=false` and `explicitlyMentioned=false`, so a Slack channel with `requireMention=true` reaches the `no-mention` skip path instead of dispatching.

## Solution Assessment

No, the closed config-only PR is not the best remaining solution because maintainer feedback asked for API-backed membership via `usergroups.users.list`. The narrower maintainable path is Slack-owned token extraction plus membership lookup, tests, and docs/scope updates, without adding a global per-agent config field as the primary behavior.
