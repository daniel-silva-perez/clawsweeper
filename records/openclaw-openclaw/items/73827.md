## Summary

Keep open. Current main still lacks Slack user-group/subteam mention handling, and the linked config-only implementation PR was closed unmerged after maintainer discussion preferred an API-backed `usergroups.users.list` membership check instead.

## What This Changes

Slack user-group mentions like `<!subteam^S...>` should wake agents that are actual members of the mentioned Slack user group when channel mention gating is enabled.

## Best Possible Solution

Implement the Slack-owned API path: parse `<!subteam^S...>` tokens, resolve mentioned user-group membership through Slack `usergroups.users.list`, cache conservatively, treat only routed bot users that are actual members as explicitly mentioned, and update Slack setup/docs/tests for the new scope and fallback behavior.

## Reproduction Assessment

Yes. The report gives concrete Slack setup steps, and static current-main inspection shows a high-confidence failure path: subteam-only text does not affect `explicitlyMentioned`, so `requireMention` rooms can still take the `no-mention` drop path.

## Solution Assessment

Yes, the maintainer-preferred API membership lookup is the best known direction because it uses Slack as the source of truth and avoids agents responding to user groups they are not actually in. The earlier config allowlist can remain reference material, but should not be revived without explicit maintainer approval.
