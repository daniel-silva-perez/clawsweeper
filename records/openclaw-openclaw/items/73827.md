## Summary

Keep open. Current main still drops Slack user-group-only mentions at the mention gate, and the linked config-only PR was closed unmerged after maintainer feedback favoring an API-backed membership check.

## What This Changes

Support Slack `<!subteam^S...>` user-group mentions as explicit mentions for agents that are actual members, so requireMention-gated channel messages can wake eligible agents.

## Best Possible Solution

Implement this inside the Slack plugin as an API-backed membership path: extract subteam mention IDs, check Slack user-group membership for the routed bot user with bounded caching, treat membership as an explicit mention, fail closed on missing scope or API errors, and update Slack setup docs, manifest scopes, tests, and changelog.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: a channel message containing only `<!subteam^S...>` leaves both `hasAnyMention` and `explicitlyMentioned` false in current main, so a `requireMention=true` room reaches the no-mention skip path.

## Solution Assessment

Yes for the remaining maintainer-directed API-backed path; it avoids duplicated OpenClaw config and only wakes bots that Slack reports as actual group members. The closed config-only allowlist PR is not the best primary fix because it can drift from Slack membership.
