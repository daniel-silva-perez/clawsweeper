## Summary

Keep open. Current `main` still only treats direct Slack `<@botUserId>` mentions or configured mention regexes as explicit mentions, while the paired config-only PR was closed unmerged after maintainer feedback favoring an API-backed `usergroups.users.list` membership check.

## What This Changes

The issue requests that Slack `<!subteam^...>` user-group mentions wake eligible OpenClaw agents instead of being dropped by the Slack mention gate.

## Best Possible Solution

Implement this inside the Slack plugin as an API-backed membership path: extract `<!subteam^S...>` tokens, call `ctx.app.client.usergroups.users.list` with bounded caching, treat the message as explicit only when `ctx.botUserId` is returned, fail closed on missing scope or API errors, and update Slack setup docs/manifest/tests/changelog. Do not make the global config-only allowlist the primary behavior.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: with message text `<!subteam^S...> hello`, current `main` sets `hasAnyMention=false` and `explicitlyMentioned=false`, so a Slack room with `requireMention=true` reaches the no-mention skip path unless a separate regex pattern happens to match.

## Solution Assessment

Yes, the maintainer-directed API-backed path is the best remaining direction. The closed config-only PR is not the best fix because it can wake a bot that is not an actual Slack user-group member.
