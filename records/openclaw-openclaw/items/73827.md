## Summary

Keep open. Current main still treats Slack explicit mentions as direct `<@botUserId>` tokens or configured regex matches, so `<!subteam^S...>` user-group mentions do not clear the Slack room `requireMention` gate. The linked config-only PR was closed unmerged after maintainer feedback favoring an API-backed membership check.

## What This Changes

The issue requests Slack user-group mentions to wake OpenClaw agents that are members of the mentioned Slack subteam.

## Best Possible Solution

Implement the fix inside the Slack plugin: parse `<!subteam^S...>` tokens, check Slack user-group membership through `ctx.app.client.usergroups.users.list`, treat the message as explicit only when `ctx.botUserId` is returned, and update Slack setup/docs/tests for the new read-only scope. Keep the closed config-only allowlist as rejected source context, not the primary behavior.

## Reproduction Assessment

Yes. The source-level path is high confidence: a room message containing only `<!subteam^S...>` leaves `hasAnyMention=false` and `explicitlyMentioned=false`, so with `requireMention=true` the Slack handler reaches the `no-mention` skip path.

## Solution Assessment

Yes. The maintainer-directed API membership check is the narrowest durable fix because it follows Slack’s actual user-group membership and avoids waking bots that are not members; a global config allowlist would be easier but less correct.
