## Summary

Keep this issue open. Current main still processes TTS directives in the normal auto-reply/ACP reply path, but cron isolated `delivery.mode: "announce"` still sends final text payloads through outbound delivery after only silent-token normalization, so `[[tts]]` / `[[tts:text]]` can remain visible instead of being converted into a voice/media payload.

## Best Possible Solution

Implement a shared pre-delivery TTS transform for isolated cron `delivery.mode: "announce"` before `deliverOutboundPayloads`, using the same effective config context as normal replies: agent id, channel, account id, final-kind delivery, and any preserved session TTS mode. Add a regression test where a cron announce final reply containing `[[tts]]` or `[[tts:text]]` produces a synthesized media/voice payload and strips the raw directive from visible text. Keep #33082 treated as related but separate because it covered tool-produced media, not model-emitted TTS tags.
