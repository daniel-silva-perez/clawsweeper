## Summary

Keep open. Current main does not show a blanket WebChat/Control UI exec block, but it still shows a real contract mismatch: Control UI identifies as the `webchat` surface while `tools.byProvider` is resolved from the selected model provider/model, and the generated schema help still describes the key as channel/surface-scoped. The remaining decision is whether to clarify docs/schema or add a deliberate surface-policy seam without weakening host exec approvals.

## What This Changes

The issue asks whether WebChat/Control UI exec should be allowed when elevated sender rules and `tools.byProvider.webchat` allow `exec`, or whether another policy layer is blocking it.

## Best Possible Solution

Align the public contract with the implementation. The maintainable end state is for config docs, schema help, and generated schema metadata to say `tools.byProvider` keys model providers/models, while WebChat/Control UI surface policy remains separate; if maintainers want surface-specific tool grants, add an explicit message-surface policy seam that still leaves `tools.exec.security`, allowlists, and approvals in force.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: configure `tools.byProvider.webchat` while using any selected model provider other than `webchat`, and `resolveEffectiveToolPolicy()` will not apply that policy; with `tools.exec.security=allowlist` and no matching allowlist or durable approval, the gateway path can still throw `exec denied: allowlist miss`.

## Solution Assessment

Unclear. Treating `tools.byProvider.webchat` as a way to bypass host exec allowlists would not be the best fix; the safer path is either schema/docs clarification or a separately designed surface-policy contract.
