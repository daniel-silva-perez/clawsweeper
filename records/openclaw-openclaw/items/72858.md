## Summary

Keep open. Current main still shows a real contract mismatch: WebChat/Control UI identifies as the WebChat surface, but `tools.byProvider` resolution is driven by the selected model provider/model, while generated schema help still describes the key as channel/surface-scoped. The reporter's follow-up narrows this away from a blanket Control UI exec block and toward allowlist-mode plus policy-contract semantics.

## What This Changes

This issue asks whether WebChat/Control UI exec should honor configured elevated sender and tool policy settings instead of still failing with an exec allowlist miss.

## Best Possible Solution

Align the public contract with implementation first: schema help, generated metadata, and docs should consistently say `tools.byProvider` keys the selected model provider/model. If maintainers want surface-specific WebChat/Control UI grants, add a separate explicit surface-policy contract with tests that preserve the existing host exec allowlist and approval layers.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: configure `tools.byProvider.webchat` while the session's selected model provider is something else, then run in `tools.exec.security=allowlist` without a matching allowlist or durable approval; current main will not apply the `webchat` provider policy and the host exec path can still throw an allowlist miss.

## Solution Assessment

Unclear. Treating `tools.byProvider.webchat` as permission to open host exec from the browser is not the safest narrow fix; the safer maintainable path is docs/schema clarification unless maintainers explicitly approve and design a separate surface-policy seam.
