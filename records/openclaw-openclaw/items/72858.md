## Summary

Keep open. Current code does not show a blanket Control UI/WebChat exec block, but it does show a real contract mismatch: `tools.byProvider` is resolved from the selected model provider/model, while config help still describes it as channel/provider or surface-scoped, which makes the reported `webchat` policy configuration understandable. Exec approval allowlist mode remains a separate layer, and the remaining argument-level allowlist capability is already tracked separately by #60427.

## What This Changes

The issue asks whether Control UI/WebChat exec should still fail with `exec denied: allowlist miss` after allowing the Control UI sender and adding `exec`/`process` to `tools.byProvider.webchat`.

## Best Possible Solution

Align the public contract with the implementation. The narrowest maintainable path is to clarify schema help/docs that `tools.byProvider` keys model providers/models, explain that WebChat/Control UI surface policy is separate, and keep exec approval allowlist behavior under `tools.exec.security`/`ask` and host approvals. If maintainers want WebChat-specific tool policy, add an explicit surface/channel policy seam rather than making tool policy bypass exec approvals.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: with a Control UI/WebChat message surface and a selected model provider other than `webchat`, `tools.byProvider.webchat` is not consulted by `resolveEffectiveToolPolicy()`, and `security=allowlist` can still deny an unsatisfied command as an allowlist miss.

## Solution Assessment

Unclear. Letting `tools.byProvider.webchat` bypass host exec allowlists would not be the best fix; the safer alternatives are a docs/schema-help correction or a separately designed message-surface tool policy contract.
