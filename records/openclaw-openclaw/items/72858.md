## Summary

Keep open. Current main still shows a real contract mismatch rather than a blanket WebChat/Control UI exec block: Control UI connects as the WebChat surface, while `tools.byProvider` is resolved from the selected model provider/model, and generated schema help still describes that key as channel/surface-scoped.

## What This Changes

The issue asks whether WebChat/Control UI exec should run when elevated sender rules and `tools.byProvider.webchat` allow `exec`, or whether the remaining `allowlist miss` is an additional policy bug.

## Best Possible Solution

Align the public contract with the implementation. Docs, schema help, and generated schema metadata should consistently say `tools.byProvider` keys model providers/models; if maintainers want WebChat/Control UI-specific tool grants, add a separate surface-policy contract that still leaves `tools.exec.security`, allowlists, and approvals in force.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: configure `tools.byProvider.webchat` while using a selected model provider other than `webchat`; `resolveEffectiveToolPolicy()` will not apply that policy, and with `tools.exec.security=allowlist` plus no matching allowlist or durable approval, gateway exec can still throw `exec denied: allowlist miss`.

## Solution Assessment

Unclear. Treating `tools.byProvider.webchat` as permission to open host exec from the browser is not the safest fix; the narrower maintainable path is docs/schema clarification unless maintainers explicitly design a separate surface-policy seam.
