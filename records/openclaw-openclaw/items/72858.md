## Summary

Keep open. Current main still shows a real WebChat/Control UI tool-policy contract mismatch rather than a blanket Control UI exec block: Control UI connects as the WebChat surface, while `tools.byProvider` is resolved from the selected model provider/model, and schema help still describes that setting in surface/channel terms. The reporter's retest and the prior ClawSweeper comment both narrow this to allowlist/provider-policy semantics, not an already-fixed WebChat denial.

## What This Changes

The issue asks whether WebChat/Control UI exec should be allowed after adding the Control UI sender to elevated allow rules and configuring `tools.byProvider.webchat` to allow `exec` and `process`, or whether the remaining `exec denied: allowlist miss` is an intentional hidden policy layer.

## Best Possible Solution

Align the public contract with the implementation. If `tools.byProvider` is intentionally model-provider/model scoped, update schema help and generated metadata so WebChat users do not expect `tools.byProvider.webchat` to grant Control UI exec; if maintainers want surface-scoped WebChat grants, add a separate policy seam that keeps host exec allowlists and approvals in force.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: configure `tools.byProvider.webchat` while the selected model provider is not `webchat`; current `resolveEffectiveToolPolicy()` will not apply that entry, and `tools.exec.security=allowlist` can still reach `exec denied: allowlist miss` without a matching allowlist or durable approval.

## Solution Assessment

Unclear. Treating `tools.byProvider.webchat` as direct permission for browser-originated host exec is not clearly the best fix; the safer narrow path is docs/schema clarification unless maintainers deliberately design a separate surface-policy contract.
