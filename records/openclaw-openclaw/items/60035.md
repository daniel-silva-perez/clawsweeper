## Summary

Keep open. Current main has meaningful process-wide HTTP forward-proxy support, but the requested SOCKS support, noProxy semantics, and per-agent/per-tool/browser override model are still absent or intentionally constrained, so the central feature request remains unresolved.

## What This Changes

Add first-class system/custom HTTP and SOCKS proxy configuration across CLI/runtime networking, with standard env fallback plus per-agent and per-tool overrides for web and browser traffic.

## Best Possible Solution

Keep the current process-wide HTTP forward-proxy support as the baseline, then scope the remaining work into maintainable follow-ups: top-level SOCKS/noProxy semantics, per-tool web overrides, per-agent/runtime inheritance, and a browser proxy UX that preserves SSRF guarantees.

## Reproduction Assessment

Not applicable as a feature request rather than a runtime bug. The current-main verification path is source/docs inspection: schema and tests reject SOCKS, web tool config has no proxy override, and browser docs/code intentionally keep managed Chrome direct by default.

## Solution Assessment

No; the current HTTP-only implementation is a useful partial solution but not the best complete answer to this issue. The safer path is maintainer-scoped design that extends proxy support without weakening existing SSRF and browser safety boundaries.
