## Summary

Keep open. Current main has not implemented the PR’s intended origin-based attribution, but the proposed two-line origin-first change is not safe to merge as-is because it buckets whole-session totals by a mutable session origin snapshot and also changes row metadata consumed by the usage UI.

## What This Changes

The PR changes `sessions.usage` to prefer `storeEntry.origin.provider` and `origin.chatType` over delivery `channel` and `chatType`, with a regression test for swapped webchat/Telegram usage attribution.

## Best Possible Solution

Fix the attribution bug with an explicit origin-versus-delivery model: compute channel aggregates from reliable origin granularity, preserve or explicitly expose delivery metadata for row consumers, constrain aggregate keys, and cover swapped webchat/Telegram plus mixed-origin long-lived sessions in tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Avoid attributing whole sessions to the latest origin:** `src/gateway/server-methods/usage.ts:658`
  - body: `loadSessionCostSummary()` returns totals for the whole session/date range, but this change buckets all of those totals by `storeEntry.origin.provider`. Since `mergeOrigin()` overwrites the stored provider on later inbound metadata, a mixed-origin bridged session can have all historical usage relabeled as whichever origin was seen last.
  - confidence: 0.86
- **[P2] Preserve delivery metadata on session rows:** `src/gateway/server-methods/usage.ts:658-659`
  - body: The origin-first `channel`/`chatType` values are also emitted as `sessions[].channel` and `sessions[].chatType`. The usage UI uses those row fields for badges, filter options, `channel:`/`chat:` queries, and CSV export, so this removes delivery-side metadata even though the aggregate bucket is the part that needs origin attribution.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: No workflow, dependency, lockfile, lifecycle, or package-publishing surface changes, but the data-plane aggregation key change needs attention.

Concerns:

- **[low] Constrain origin provider aggregation keys:** `src/gateway/server-methods/usage.ts:658`
  - body: The patch makes `origin.provider` the preferred `byChannelMap` key, while `normalizeMessageChannel()` preserves unknown normalized strings. If inbound origin metadata can be influenced, this can spoof usage buckets or create high-cardinality response payloads.
  - confidence: 0.72
