## Summary

Keep this PR open. Current main still attributes `sessions.usage` channel buckets from `storeEntry.channel` before `origin.provider`, so the intended fix is not implemented. The PR is a plausible fix for the linked swapped webchat/Telegram report, but it needs maintainer review because it also changes visible session row semantics and uses mutable origin metadata as a whole-session aggregation key.

## What This Changes

This PR changes `sessions.usage` to prefer `storeEntry.origin.provider` and `origin.chatType` over delivery `storeEntry.channel` and `chatType`, and adds a regression test for swapped webchat/Telegram attribution.

## Best Possible Solution

Keep this PR open for maintainer review or rework. The best fix is a bounded usage attribution change that resolves the swapped webchat/Telegram aggregate bug while making the API contract explicit: keep or add delivery metadata for rows, validate or cap origin-derived aggregate keys, and add regression coverage for swapped cross-channel sessions plus mixed-origin sessions. Coordinate with #61315 if maintainers want one consolidated usage aggregation fix.

## Risks / Open Questions

- The PR changes public `sessions[].channel` and `sessions[].chatType` row fields, not only `aggregates.byChannel`, so UI filters, badges, queries, and CSV export would show origin values instead of delivery session metadata.
- Origin provider is normalized but not allowlisted; using it as a returned field and aggregation key could create spoofed or high-cardinality channel buckets if untrusted origin metadata reaches the session store.
- Session cost summaries are whole-session/date-range totals, while origin metadata is mutable at the session level; mixed-origin sessions can still be misattributed unless the attribution contract is narrowed or split per turn.
- Related open #61315 also changes usage aggregation behavior, so maintainers should coordinate before landing overlapping dashboard fixes.
