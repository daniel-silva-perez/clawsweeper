## Summary

Keep this PR open for maintainer review. Current main still attributes `sessions.usage` channel buckets from delivery `storeEntry.channel` before `origin.provider`, so the underlying #52436 problem is not implemented on main; however, this PR’s two-line fix appears to change both aggregate attribution and per-session row metadata, and review comments raised a real correctness concern because `origin` is mutable session-level metadata rather than per-turn attribution.

## What This Changes

The PR changes `sessions.usage` to prefer `storeEntry.origin.provider` and `origin.chatType` over delivery `channel` and `chatType`, and adds a regression test for swapped webchat/Telegram attribution.

## Best Possible Solution

Keep the bug open until the usage API separates attribution from delivery metadata cleanly: preserve or explicitly expose delivery channel fields for row consumers, compute channel usage from the correct origin granularity, and constrain aggregate keys enough to avoid spoofed or unbounded channel buckets. The regression should cover swapped webchat/Telegram traffic plus mixed-origin sessions where the latest origin is not representative of all usage.

## Risks / Open Questions

- The PR changes `sessions[].channel` and `sessions[].chatType` to origin values, while existing UI badges, filters, and CSV export consume those row fields as the displayed session metadata.
- Because `origin.provider` is merged and overwritten at session level on inbound metadata updates, using it to bucket an entire session/date-range summary can retroactively move all historical session usage to the latest origin.
- `origin.provider` is normalized but unknown strings are preserved, so using it directly as a `byChannelMap` key may allow spoofed or high-cardinality buckets if any inbound context field is user-influenced.
