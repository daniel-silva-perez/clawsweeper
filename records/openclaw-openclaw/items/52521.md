## Summary

Keep open: current main still has the reported channel-first usage attribution, but this PR is not ready as-is because it buckets whole-session/date-range totals by mutable session-level origin metadata and also changes row metadata semantics consumed by the usage UI.

## What This Changes

The PR changes `sessions.usage` to prefer `storeEntry.origin.provider` and `origin.chatType` over delivery `channel` and `chatType`, with a regression test for swapped webchat/Telegram attribution.

## Best Possible Solution

Fix the attribution bug with an explicit model for origin versus delivery: compute channel aggregates from the correct origin granularity, preserve or explicitly expose delivery metadata for row consumers, validate aggregate keys, and cover swapped webchat/Telegram plus mixed-origin long-lived sessions in tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Avoid attributing whole sessions to the latest origin:** `src/gateway/server-methods/usage.ts:658`
  - body: `loadSessionCostSummary()` returns totals for the whole session/date range, but this line buckets all of those totals by `storeEntry.origin.provider`. Since `mergeOrigin()` overwrites the stored provider on later inbound metadata, a mixed-origin bridged session can have all historical usage relabeled as whichever origin was seen last.
  - confidence: 0.86
- **[P2] Preserve delivery metadata on session rows:** `src/gateway/server-methods/usage.ts:658-659`
  - body: The origin-first `channel`/`chatType` values are also emitted as `sessions[].channel` and `sessions[].chatType`. The usage UI uses those row fields for badges, filter options, `channel:`/`chat:` queries, and CSV export, so this removes the delivery-side metadata even though the aggregate bucket is the part that needs origin attribution.
  - confidence: 0.78
