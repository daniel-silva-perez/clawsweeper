## Summary

Keep this PR open, but do not merge it as-is. The underlying swapped webchat/Telegram usage bug looks valid and is not fixed on current main, but the patch changes shared row metadata and buckets whole-session totals by a mutable latest-origin snapshot, so it needs maintainer/API judgment before landing.

## What This Changes

The PR changes `sessions.usage` to prefer `storeEntry.origin.provider` and `origin.chatType` over delivery `channel` and `chatType`, and adds a backend regression test for swapped webchat/Telegram usage attribution.

## Best Possible Solution

Fix the attribution bug with explicit origin-vs-delivery semantics: preserve or intentionally rename row metadata, compute channel aggregates from the correct origin granularity, constrain aggregate keys, and cover both simple swapped-channel sessions and mixed-origin long-lived sessions in tests.

## Reproduction Assessment

Yes for the reported swapped-channel scenario: current main visibly prefers delivery metadata and the PR adds a focused fixture that would fail before the priority inversion. The mixed-origin regression risk is supported by code-path inspection rather than an existing executable test.

## Solution Assessment

No: the proposed two-line priority inversion is narrower than a refactor, but it is not the best fix because it relabels row metadata and cannot accurately attribute whole-session totals when origin changes within one session.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Avoid attributing whole sessions to the latest origin:** `src/gateway/server-methods/usage.ts:621`
  - body: `loadSessionCostSummary()` returns totals for the whole session/date range, but `entry.origin.provider` is a mutable session-level snapshot. Since later inbound metadata can overwrite the stored origin, this line can retroactively move all historical usage in a mixed-origin bridged session into whichever origin was seen last.
  - confidence: 0.86
- **[P2] Preserve delivery metadata on session rows:** `src/gateway/server-methods/usage.ts:621-622`
  - body: The origin-first `channel`/`chatType` values are also emitted as `sessions[].channel` and `sessions[].chatType`. Current usage UI filters, badges, filter options, and CSV export consume those row fields directly, so this removes delivery-side metadata even though only aggregate attribution needs origin semantics.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff has no CI, dependency, lockfile, lifecycle, or publishing changes, but it expands origin metadata into aggregate keys without validation or cardinality bounds.

Concerns:
