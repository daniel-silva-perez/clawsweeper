## Summary

Keep this PR open because current main still has the reported delivery-channel-first attribution behavior, but the proposed patch should not merge as-is. The diff fixes the simple swapped-session case, yet it also buckets whole-session totals by a mutable latest-origin snapshot, changes existing row/filter/CSV channel semantics, and needs an aggregate-key constraint before origin metadata becomes the preferred bucket key.

## What This Changes

The PR changes `sessions.usage` to prefer `storeEntry.origin.provider` and `origin.chatType` over delivery `channel` and `chatType`, and adds a regression test for swapped webchat/Telegram usage attribution.

## Best Possible Solution

Fix the swapped usage attribution with an explicit origin-versus-delivery model: aggregate by trustworthy origin granularity, preserve or intentionally rename row metadata fields, constrain channel bucket keys, and cover both swapped webchat/Telegram sessions and mixed-origin long-lived sessions in tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Avoid attributing whole sessions to the latest origin:** `src/gateway/server-methods/usage.ts:658`
  - body: `loadSessionCostSummary()` returns totals for the whole session/date range, but this line would bucket all of those totals by `storeEntry.origin.provider`. Since `mergeOrigin()` overwrites the stored provider when later inbound metadata arrives, a mixed-origin bridged session can have historical usage relabeled as whichever origin was seen last.
  - confidence: 0.86
- **[P2] Preserve delivery metadata on session rows:** `src/gateway/server-methods/usage.ts:658-659`
  - body: The origin-first `channel`/`chatType` values are also emitted as `sessions[].channel` and `sessions[].chatType`. The usage UI uses those row fields for badges, filter options, `channel:`/`chat:` queries, and CSV export, so this removes delivery-side metadata even though the aggregate bucket is the part that needs origin attribution.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: No CI, dependency, lockfile, lifecycle, or publishing surface changed, but the data-plane aggregate key change needs a small security hardening decision.

Concerns:

- **[low] Constrain origin provider aggregation keys:** `src/gateway/server-methods/usage.ts:658`
  - body: The patch makes `origin.provider` the preferred `byChannelMap` key, while `normalizeMessageChannel()` preserves unknown normalized strings. If inbound origin metadata can be influenced, this can spoof usage buckets or create high-cardinality response payloads.
  - confidence: 0.72
