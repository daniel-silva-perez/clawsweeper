## Summary

Keep open: current main still has the reported delivery-first channel attribution, but this PR is not safe to merge as written because it re-buckets whole-session totals by a mutable session-level origin snapshot and changes the meaning of session row metadata.

## What This Changes

The PR changes `sessions.usage` to prefer `storeEntry.origin.provider` and `origin.chatType` over delivery `channel` and `chatType`, and adds a regression test for swapped webchat/Telegram attribution.

## Best Possible Solution

Model origin and delivery explicitly: preserve row metadata or expose separate fields deliberately, compute channel aggregates from the correct origin granularity, constrain aggregate keys, and cover swapped webchat/Telegram plus mixed-origin long-lived sessions in focused tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Avoid attributing whole sessions to the latest origin:** `src/gateway/server-methods/usage.ts:621`
  - body: `loadSessionCostSummary()` returns totals for the entire session/date range, but `entry.origin.provider` is a mutable session-level snapshot. Since later inbound metadata can overwrite the stored origin, this line can retroactively move all historical usage in a mixed-origin bridged session into whichever origin was seen last.
  - confidence: 0.86
- **[P2] Preserve delivery metadata on session rows:** `src/gateway/server-methods/usage.ts:621-622`
  - body: The origin-first `channel`/`chatType` values are also emitted as `sessions[].channel` and `sessions[].chatType`. Current usage UI filters, badges, and CSV export consume those row fields directly, so this removes delivery-side metadata even though only aggregate attribution needs origin semantics.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff has no CI, dependency, lockfile, lifecycle, or publishing changes, but it expands origin metadata into aggregate keys without validation or cardinality bounds.

Concerns:

- **[low] Constrain origin provider aggregation keys:** `src/gateway/server-methods/usage.ts:621`
  - body: The patch makes `origin.provider` the preferred `byChannelMap` key, while channel normalization preserves unknown strings. If inbound origin metadata can be influenced by a caller or plugin, usage buckets can be spoofed or made high-cardinality.
  - confidence: 0.72
