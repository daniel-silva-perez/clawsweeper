## Summary

Keep open. Current main still uses delivery-channel-first attribution, so the PR is not obsolete, but the proposed origin-first session-level change is not safe to merge as-is because it buckets whole-session totals by a mutable origin snapshot, changes row metadata consumed by the UI/API, and needs aggregate-key hardening.

## What This Changes

The PR changes `sessions.usage` to prefer `storeEntry.origin.provider` and `origin.chatType` over delivery `channel` and `chatType`, and adds a regression test for swapped webchat/Telegram attribution.

## Best Possible Solution

Fix the swapped attribution with an explicit origin-versus-delivery model: aggregate by a trustworthy origin dimension at the right granularity, preserve or rename row metadata intentionally, constrain channel bucket keys, and cover swapped webchat/Telegram plus mixed-origin long-lived sessions in tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Avoid attributing whole sessions to the latest origin:** `src/gateway/server-methods/usage.ts:658`
  - body: `loadSessionCostSummary()` returns totals for the whole session/date range, but this changed line would bucket all of those totals by `storeEntry.origin.provider`. Since `mergeOrigin()` overwrites the stored provider when later inbound metadata arrives, mixed-origin bridged sessions can have historical usage relabeled as whichever origin was seen last.
  - confidence: 0.86
- **[P2] Preserve delivery metadata on session rows:** `src/gateway/server-methods/usage.ts:658-659`
  - body: The origin-first `channel`/`chatType` values are also emitted as `sessions[].channel` and `sessions[].chatType`. The usage UI uses those row fields for badges, filter options, `channel:`/`chat:` queries, and CSV export, so this removes delivery-side metadata even though only the aggregate bucket needs origin attribution.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: No CI, dependency, lockfile, lifecycle, or publishing surface changed, but the aggregate-key change needs small data-plane hardening before merge.

Concerns:

- **[low] Constrain origin provider aggregation keys:** `src/gateway/server-methods/usage.ts:658`
  - body: The patch makes `origin.provider` the preferred `byChannelMap` key, while `normalizeMessageChannel()` preserves unknown normalized strings. If inbound origin metadata can be influenced, this can spoof usage buckets or create high-cardinality response payloads.
  - confidence: 0.72
