## Summary

Keep this PR open: the reported multi-provider Ollama routing bug is valid on current main, but the proposed wrapper still replaces the selected runtime model URL with the first registered provider URL, so the patch does not fix the central failure path.

## What This Changes

The PR updates `extensions/ollama/src/stream.ts` to resolve Ollama chat URL and SSRF policy from `model.baseUrl` per request and wraps configured Ollama stream functions with a captured resolved base URL.

## Best Possible Solution

Keep the fix in the Ollama plugin: the registered stream should prefer the selected runtime model's non-empty `baseUrl` and provider headers per request, use captured defaults only as fallback, and add focused regression coverage for two providers sharing `api: "ollama"`. Alias or manifest work for `remote-ollama` should remain a separate follow-up unless maintainers explicitly combine it.

## Reproduction Assessment

Yes. The source path is reproducible without live network: register two Ollama-compatible providers sharing `api: "ollama"`, then invoke the first registered configured stream with a second runtime model that carries a different `baseUrl` and assert the guarded fetch URL.

## Solution Assessment

No. Per-request URL resolution is the right layer, but this implementation overwrites the runtime `model.baseUrl` with the first wrapper's captured `resolvedBaseUrl`; the safer repair is to preserve runtime URL and headers, with closure defaults only as fallback.
