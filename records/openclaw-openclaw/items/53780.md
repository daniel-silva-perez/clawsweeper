## Summary

Keep open: current main still lacks the media-specific outbound sanitizer, so this PR is not obsolete. The branch is dirty against main and the proposed patch has blocking regressions around non-media error formatting plus an unsafe recursive cause-chain helper.

## What This Changes

The PR adds a safe MediaFetchError message/helper, redacts media fetch failures at outbound attachment, broadcast, and message_sent boundaries, and adds focused sanitizer tests.

## Best Possible Solution

Land a rebased media-specific sanitizer that maps MediaFetchError failures to one stable channel-safe message at the affected outbound boundaries, keeps formatErrorMessage as the fallback for every non-media failure, and uses cycle-safe cause traversal with media and non-media regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: fetchRemoteMedia creates detailed MediaFetchError messages, attachment hydration forwards loadWebMedia failures, and current broadcast/message_sent paths do not replace MediaFetchError with a stable safe message.

## Solution Assessment

No, not as proposed. The direction is right, but the branch should preserve current non-media error formatting/redaction and make the cause-chain detector cycle-safe before merge.
