## Summary

Keep this PR open. Current main has a generic normal-reply media fallback, but it still does not implement the PR's media-fetch sanitizer at the message-action, broadcast-result, or message_sent hook boundaries. The branch addresses a real privacy-sensitive path, but the proposed patch has concrete non-media diagnostic regressions and needs maintainer review/rebase before merge.

## What This Changes

The branch adds a safe MediaFetchError message/helper, redacts media fetch failures at attachment hydration, broadcast result, and message_sent failure boundaries, and adds focused sanitizer tests.

## Best Possible Solution

Land a rebased, reviewed media-specific sanitizer that maps MediaFetchError failures to one stable channel-safe message at outbound action and hook boundaries, preserves existing diagnostics for non-media failures, uses cycle-safe cause traversal, and covers attachment hydration, broadcast results, and message_sent failures in tests.

  - body: The new fallback converts every non-Error delivery failure into "unknown error" before emitting message_sent.   - confidence: 0.9

  - body: The broadcast failure path has the same regression: non-media thrown strings or objects now become "unknown error" in the per-target result payload. That loses actionable channel/action diagnostics unrelated to MediaFetchError; only replace the text when the caught error is a media fetch failure.
  - confidence: 0.88
- **[P2] Guard cause-chain traversal against cycles:** `src/media/fetch.ts:40-41`
  - body: isMediaFetchError is called on arbitrary caught outbound errors, but it recursively follows Error.cause without tracking seen values. A cyclic cause chain from a plugin or adapter error would overflow before the original failure can be handled; use an iterative traversal with a seen set like formatErrorMessage.
  - confidence: 0.79
