## Summary

Keep this PR open. Current main has broader default protections that hide raw tool-error details unless verbose is enabled, but it does not implement the PR's media-fetch-specific safe error boundary. The same media fetch errors can still flow through `loadWebMedia()`, `formatErrorMessage(err)`, broadcast action results, and `message_sent` hook error fields. The PR is a plausible privacy fix, but it needs maintainer review and at least the already-raised non-Error diagnostic regression addressed before merge.

## Best Possible Solution

Keep the PR open for maintainer review or implement an equivalent targeted fix on main: add a media-fetch error classifier and safe message, apply it in media action parameter hydration, broadcast result construction, and delivery `message_sent` failure emission, preserve useful diagnostics for non-media/non-Error failures, and add focused tests for exact sanitized output with raw details retained only internally.

## Risks / Open Questions

- Because this touches channel-visible privacy behavior, regression tests should assert exact safe-message equality and absence of PII at the action-result and delivery-hook boundaries.
