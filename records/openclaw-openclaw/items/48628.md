## Summary

Keep this issue open. Current main still treats inbound audio as media-understanding/STT input, injects the resulting transcript into the reply context, and suppresses transcribed audio from agent prompt attachment notes. The only visible native forwarding track is the related open PR #48900, which is not merged or shipped in current main.

## Best Possible Solution

Keep this issue open as the product tracker for native inbound audio forwarding. The best path is to review, harden, and either land #48900 or a successor that adds a typed prompt-media contract for audio, config for transcript-only versus audio/audio+transcript behavior, capability checks, payload limits, docs, and regression coverage across Telegram/Signal-style voice notes.
