## Summary

Keep this issue open. Current main still does not pass raw Telegram/Signal audio into the model prompt: audio attachments are routed through media-understanding/STT, successful transcription replaces the agent body with text, and transcribed audio is deliberately removed from prompt attachment notes. The related native forwarding PR #48900 remains the plausible implementation track, but it is not merged or shipped on current main.

## Best Possible Solution

Keep this issue open and use it as the product/behavior tracker for native audio forwarding from messaging channels. The likely path is to review and harden #48900 or a successor PR so audio-capable reply models can receive raw audio, optionally alongside the STT transcript, with explicit config, capability checks, size/token limits, and provider-specific transport support.
