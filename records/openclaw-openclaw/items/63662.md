Ship this, if desired, as a narrow memory-core/config change: add the schema and docs for the new memory settings, filter `MEMORY.md` only for `recall-only`, register the auto-recall hook in a hot-reload-safe way using the supported live config API, and remove unrelated scripts, QA, protocol, and delivery changes from the branch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Register auto-recall unconditionally:** `extensions/memory-core/src/auto-recall.ts:80-82`
  - body: The hook is skipped when startup config has no `autoRecall`, even though the handler reads live config per turn. Enabling `agents.defaults.memorySearch.autoRecall` after startup will not install a `before_prompt_build` handler until restart, so the new setting appears ignored.
  - confidence: 0.88
- **[P2] Use the supported live config API:** `extensions/memory-core/src/auto-recall.ts:90`
  - body: This bundled plugin calls `api.runtime.config.loadConfig()` from runtime code. That API is deprecated and the repo's config-boundary guard flags it; use `api.runtime.config.current()` or pass the loaded config so the architecture check stays green.
  - confidence: 0.86
- **[P2] Keep plugin delivery resolution first:** `src/utils/delivery-context.ts:116-126`
  - body: The new thread-child fallback runs before `messaging.resolveDeliveryTarget` for Telegram. Topic conversations require the Telegram resolver to normalize the target and numeric `threadId`; bypassing it can route replies with raw conversation ids.
  - confidence: 0.87
