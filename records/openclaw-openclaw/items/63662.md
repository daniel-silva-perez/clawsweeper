## Summary

Keep open. Current main still lacks the requested first-class `memoryInjection` and `memorySearch.autoRecall` config, so the PR is not obsolete; however, the proposed branch is not merge-ready because it introduces blocking runtime/config and delivery-routing regressions and mixes unrelated guardrail/channel changes into the memory feature.

## What This Changes

The PR adds configurable `agents.defaults.memoryInjection`, `agents.defaults.memorySearch.autoRecall`, memory-core prompt-injection code, config schema/docs/tests, and several unrelated script, QA, and delivery-context changes.

## Best Possible Solution

Ship this as a narrow memory configuration change: keep `recall-only` as a core bootstrap filter, implement automatic recall through supported live runtime config with hot-reload behavior, preserve channel-plugin delivery normalization, and split unrelated guardrail/QA/channel changes out of the feature branch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Register auto-recall even when startup config is off:** `extensions/memory-core/src/auto-recall.ts:80-82`
  - body: The hook is skipped entirely when startup config has no `autoRecall`, even though the handler is designed to read live config per turn. If a user enables `agents.defaults.memorySearch.autoRecall` via hot config reload, no `before_prompt_build` hook exists to observe that change until the plugin or gateway restarts.
  - confidence: 0.88
- **[P2] Use the supported live config API:** `extensions/memory-core/src/auto-recall.ts:90`
  - body: This bundled plugin calls `api.runtime.config.loadConfig()` from the prompt hook. That API is deprecated for runtime plugin code and is covered by the repo's deprecated-internal-config-api guard; use `api.runtime.config.current()` or the existing live plugin config pattern instead so the branch does not add a banned config reload path.
  - confidence: 0.84
- **[P2] Keep plugin delivery normalization ahead of fallbacks:** `src/utils/delivery-context.ts:114-125`
  - body: The new early fallback returns before `messaging.resolveDeliveryTarget` for Telegram thread children. Telegram's plugin parser converts canonical topic conversations into `{ to: chatId, threadId: topicId }`; bypassing it can leave raw conversation ids in `threadId` and change routing for topic replies.
  - confidence: 0.87
