## Summary

Keep open. Current main still has a single native chat window controller and the visible new-session/reset paths use `sessions.reset`, which intentionally tears down the target session’s active work; `sessions.create` exists as a backend primitive but is not wired into an independent multi-window desktop flow. The remaining work needs maintainer product direction before it is a safe automated repair lane.

## What This Changes

The issue requests multiple independent desktop conversation windows so opening a new conversation does not pause, hide, or interrupt an existing active session.

## Best Possible Solution

Add an explicit desktop/control UI flow for independent conversation windows that creates or selects distinct session keys, keeps one chat controller per visible session/window, and uses existing non-blocking `chat.send`/`sessions.create` primitives instead of reset semantics for ordinary parallel work.

## Reproduction Assessment

Unclear: I did not run the macOS app live, but the static reproduction path is strong: open or switch a different native chat session and `WebChatManager.show` replaces the single normal window, while `/new`/reset paths call `sessions.reset`, which tests prove aborts active work.

## Solution Assessment

No: reset/new-session behavior is not the best way to solve parallel conversations because reset is intentionally destructive for the target session. The safer direction is a per-window/per-session controller model backed by `sessions.create` and existing non-blocking chat sends.
