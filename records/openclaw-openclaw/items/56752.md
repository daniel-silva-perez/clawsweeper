## Summary

Keep this open. Current main has useful session primitives, but the visible desktop chat path still has a single normal chat window controller and reset/new-session behavior intentionally clears or aborts the target session, so the requested multi-window parallel UX is not implemented as a product surface.

## What This Changes

The issue requests desktop multi-window parallel sessions so starting a new conversation window does not pause or interrupt active sessions in existing windows.

## Best Possible Solution

Add an explicit desktop/control UI flow for independent conversation windows that creates or selects distinct session keys, keeps one chat controller per visible session/window, and uses existing non-interrupting session send APIs instead of resetting the active session.

## Reproduction Assessment

Unclear: I did not run the macOS app live, but the static path is high-confidence: `WebChatManager.show` replaces the existing normal chat window for a different session, and `sessions.reset` is tested to abort/clear active work.

## Solution Assessment

No: using reset/new-session semantics is not the best way to solve parallel conversations because reset intentionally interrupts the target session. The safer direction is to build on `sessions.create` plus per-session window controllers and reserve reset for explicit destructive resets.
