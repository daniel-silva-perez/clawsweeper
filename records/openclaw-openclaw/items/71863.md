## Summary

Keep open. Current main still has the synchronous Signal daemon shutdown race, and this PR is a focused implementation for the linked restart bug with green CI; the remaining concrete blocker I found is the required changelog entry, plus maintainer/live Signal restart validation before merge.

## What This Changes

This PR makes the Signal plugin daemon shutdown awaitable with SIGTERM-to-SIGKILL escalation, awaits monitor cleanup, and adds focused daemon and autostart monitor regression tests.

## Best Possible Solution

Land the Signal-owned lifecycle fix after adding the active-version changelog entry, keeping the awaitable/idempotent daemon stop behavior and regression tests, then validate the autoStart restart path with a live Signal setup before merge.

## Reproduction Assessment

Yes. Current main has a high-confidence code-level reproduction path: with `channels.signal.autoStart=true`, a Signal config reload stops then starts the channel, but the monitor currently resolves shutdown after a fire-and-forget SIGTERM rather than after the child daemon exits.

## Solution Assessment

Yes, with one process follow-up. Making the Signal daemon handle awaitable and awaiting monitor cleanup is the narrowest maintainable fix in the owning plugin; the PR still needs the required changelog entry and live restart validation before merge.
