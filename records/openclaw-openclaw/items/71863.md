## Summary

Keep open. Current main still has the Signal daemon shutdown race: the daemon stop contract is synchronous and monitor cleanup does not await child-process exit. The PR is an active, focused Signal-plugin fix for the linked issue, and I found no blocking correctness or security issue in the supplied diff.

## What This Changes

The PR makes Signal daemon shutdown awaitable, adds SIGTERM-to-SIGKILL escalation, awaits monitor cleanup, and adds focused daemon/monitor regression tests.

## Best Possible Solution

Land a Signal-owned lifecycle fix that makes daemon stop idempotent and awaitable, waits for signal-cli exit after SIGTERM, escalates after a bounded timeout, and keeps regression coverage for daemon stop semantics and aborted monitor shutdown without adding a core API change.

## Reproduction Assessment

Yes. Current main has a high-confidence code-level reproduction path: restart the Signal channel in autostart mode, where `stopChannel()` waits for the monitor task but the monitor currently resolves after a fire-and-forget SIGTERM instead of waiting for the child daemon to exit.

## Solution Assessment

Yes. The proposed direction is the narrowest maintainable solution because Signal owns the signal-cli daemon lifecycle, and the gateway already waits on plugin channel tasks during restart.
