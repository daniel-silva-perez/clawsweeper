## Summary

Keep this PR open. Current main does not implement the proposed `messages.outbound.text.emDash` config surface or shared outbound em-dash rewrite pipeline, so the PR is not obsolete. It remains a plausible feature implementation, but the existing review comments identify real user-visible text mutation issues and hook-ordering semantics that should be resolved before merge.

## Best Possible Solution

Keep the PR open as the active implementation candidate. The next maintainer path is to review the config shape and pipeline placement, fix the rewrite helper so it only changes punctuation caused by em-dash replacement, add edge-case tests for leading em dashes and existing trailing commas, and explicitly document or adjust the `message_sending` hook ordering before merge.

## Risks / Open Questions

- The PR's current rewrite helper can mutate punctuation outside the intended em-dash replacement case, including stripping an existing trailing comma from text with no em dash.
- The PR's current rewrite helper can produce a leading comma for messages that start with an em dash unless that case is handled and covered by tests.
- The proposed pipeline changes what `message_sending` hooks observe by feeding them already-normalized content; that ordering needs maintainer agreement and documentation or code comments if accepted.
