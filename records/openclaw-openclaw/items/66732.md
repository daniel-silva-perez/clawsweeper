## Summary

Current main still does not resolve the reported Telegram reminder-note leak. The exact unscheduled-reminder guard note still exists, the reply runner can still append it whenever `successfulCronAdds` is zero and no same-session cron job is found, successful scheduling telemetry is still tied to native `cron.add`, and Telegram outbound direct text/media/payload paths still forward the supplied text without a targeted suppressor.

## Best Possible Solution

Keep this issue open. The best maintainer path is either a narrow Telegram plugin suppressor for the exact false unscheduled-reminder note variants across direct text, media captions, and payload/interactive fallback sends, or a broader scheduling telemetry fix that reliably marks every successful reminder creation path before the guard runs. Add focused regression coverage for Telegram direct text, media, payload fallback, native `cron.add`, and any supported helper-created reminder path.
