## Summary

Keep open. Current main and the latest release still keep ordinary Gateway startup separate from broad doctor repair: startup does targeted degraded loading and a few explicit startup writes, while `doctor --fix` remains gated behind explicit repair/yes modes. The issue discussion also includes credible safety counterexamples from running `doctor --fix` at startup, so this needs maintainer policy before implementation.

## What This Changes

Request auto-applying safe, non-destructive doctor fixes during Gateway startup, or adding an opt-in `doctor.autoFix` setting with logs for applied changes.

## Best Possible Solution

Define a narrow startup repair contract before adding code: allow only explicitly classified idempotent, low-risk repairs to run unattended; keep sandbox, auth, allowlist/open-policy, exec policy, channel availability, and plugin disablement changes manual or opt-in; log every changed config path with backup/recovery details; and document/test protected intentional config behavior.

## Reproduction Assessment

Yes. A high-confidence static and test-backed path exists: `startGatewayServer` calls `loadGatewayStartupConfigSnapshot`, and the plugin-local startup invalidity regression test expects `wroteConfig: false` with a manual `openclaw doctor --fix` warning.

## Solution Assessment

No, not as broad startup execution of the existing doctor repair sequence. The maintainable solution is a startup-safe allowlist or explicit opt-in policy, not unattended application of all current `doctor --fix` mutations.

## Security Review

Status: needs_attention

Summary: Unattended startup repair is security-sensitive because the existing doctor repair path can mutate access, plugin, channel, and command-execution related configuration.

  - confidence: 0.9

  - body: Invalid-plugin repair deletes the invalid config payload and sets the entry disabled. That can be appropriate for explicit `doctor --fix`, but it is risky as an unattended startup mutation without protected-config semantics and visible operator logs.
