## Summary

Keep open. Current origin/main still does not auto-apply general safe doctor fixes during ordinary gateway startup: startup uses targeted recovery, degraded loading, bundled-plugin/runtime exceptions, and manual doctor hints, while the broad doctor repair sequence remains gated behind explicit repair/yes modes. The issue discussion includes credible safety counterexamples from a startup doctor --fix workaround, and related open PR #64684 is an adjacent diagnostics slice rather than a resolution for unattended startup repair.

## What This Changes

Request to auto-apply safe, non-destructive doctor fixes on gateway start or add a doctor.autoFix option while logging what was applied.

## Best Possible Solution

Define a narrow startup repair contract before implementation: allow only explicitly classified, idempotent startup-safe mutations; exclude sandbox, auth, allowlist, channel availability, and plugin disablement changes unless clearly opted in; log every changed config path with backup/recovery details; keep broad repair behind manual openclaw doctor --fix or a clearly named opt-in setting.

## Security Review

Status: needs_attention

Summary: Unattended gateway-start repair is security-sensitive because broad doctor fixes can silently alter sandbox, allowlist, auth, exec, or plugin/channel availability settings.

Concerns:

- **[medium] Define unattended repair safety policy first:** `src/commands/doctor/repair-sequencing.ts:52`
  - body: runDoctorRepairSequence covers broad repair categories, including channel mutations, allowlist/open-policy repair, stale/invalid plugin cleanup, and exec safe-bin profile repair. Gateway startup should not call this path until maintainers define an explicit safe-mutation allowlist, audit logging, backup behavior, and opt-in rules for security or availability changes.
  - confidence: 0.87
- **[medium] Avoid silent plugin or channel disablement:** `src/commands/doctor/shared/invalid-plugin-config.ts:59`
  - body: Invalid-plugin repair deletes the plugin config payload and disables the plugin entry. That is reasonable for explicit doctor --fix, but it would be risky as an unattended startup mutation without protected/intentional config policy and clear operator logs.
  - confidence: 0.84
