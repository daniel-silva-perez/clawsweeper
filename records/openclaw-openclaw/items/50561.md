## Summary

Keep open. Current main has targeted startup recovery and read-time legacy compatibility, but it does not implement a general gateway-start `doctor --fix` auto-apply path or a `doctor.autoFix` policy. The issue discussion also added concrete safety concerns around silent config mutation, so the remaining work needs an explicit, audited repair policy rather than closure as already implemented.

## Best Possible Solution

Keep this open for a scoped maintainer design: classify doctor mutations into audited startup-safe repairs versus prompt/CLI-only repairs, add field-level logging and backups for any automatic writes, respect protected or intentional config values, and cover the gateway startup path with tests. The related open doctor safety work in #39118 may be useful context, but it does not by itself implement this issue’s requested gateway-start auto-fix policy.

## Risks / Open Questions

- A naive gateway-start `doctor --fix` hook could repeat the silent-mutation failures described in the discussion, especially for plugin enablement, channel config, sandbox policy, or supervisor metadata.
- Existing docs and tests separate read-time/startup compatibility from explicit repair writes; changing that boundary needs targeted classification, logging, backups, and regression coverage.
