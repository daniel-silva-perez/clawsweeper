## Summary

Keep open. Current main still does not auto-apply general safe doctor fixes during ordinary gateway startup: startup performs targeted recovery/degraded loading and emits manual doctor hints, while the broad doctor repair sequence remains gated behind explicit repair/yes modes. The issue discussion also contains credible safety counterexamples from a startup `doctor --fix` workaround, and the related open PR is scoped to failure-path diagnostics rather than this broader unattended startup repair request.

## What This Changes

The issue requests auto-applying safe or non-destructive doctor fixes on gateway startup, or adding a `doctor.autoFix` option, while logging any applied fixes.

## Best Possible Solution

Define a narrow startup repair contract before implementation: classify which doctor mutations may run unattended, exclude sandbox/security/channel-availability/plugin-disablement changes unless explicitly opted in, log every changed config path with backup and recovery details, and keep broader repair behind manual `openclaw doctor --fix` or a clearly named opt-in setting.
