## Summary

Keep open. Current main still does not auto-apply general safe doctor fixes during ordinary gateway startup: startup uses targeted recovery/degraded loading and manual doctor hints, while the broader repair sequence remains gated behind explicit repair/yes modes. The issue discussion includes credible safety counterexamples from a startup `doctor --fix` workaround, and related open PR #64684 is a narrower failure-diagnostics slice rather than a fix for unattended startup repair.

## What This Changes

The issue requests unattended safe/non-destructive doctor fixes during gateway startup, or an opt-in `doctor.autoFix` setting, with logs for any auto-applied changes.

## Best Possible Solution

Define a narrow startup repair contract first: allow only explicitly classified, idempotent startup-safe mutations; exclude sandbox/security/auth/allowlist/channel-availability/plugin-disablement changes unless clearly opted in; log every changed config path with backup/recovery details; keep broad repairs behind manual `openclaw doctor --fix` or a clearly named opt-in setting.

## Security Review

Status: needs_attention

Summary: Unattended gateway-start repair is security-sensitive because broad doctor fixes can silently alter sandbox, allowlist, auth, or plugin/channel availability settings.

Concerns:

- **[medium] Define unattended repair safety policy first:** `src/commands/doctor/repair-sequencing.ts:52`
  - body: `runDoctorRepairSequence` currently covers broad repair categories, including channel mutations, open-policy/allowlist repair, stale/invalid plugin cleanup, and exec safe-bin profile repair. Startup auto-fix should not call this path until maintainers define an explicit safe-mutation allowlist, audit logging, backups, and opt-in rules for security or availability changes.
  - confidence: 0.87
- **[medium] Avoid silent plugin or channel disablement:** `src/commands/doctor/shared/invalid-plugin-config.ts:59`
  - body: Invalid-plugin repair removes the plugin config payload and disables the plugin entry. That is reasonable for explicit `doctor --fix`, but it would be risky as an unattended startup mutation without protected/intentional config policy and clear operator logs.
  - confidence: 0.84
