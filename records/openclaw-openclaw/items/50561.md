## Summary

Keep open. Current main still does not implement unattended safe doctor repair on ordinary gateway startup: startup performs targeted recovery/degraded loading and emits manual `openclaw doctor --fix` hints, while the broader doctor repair sequence only runs from explicit repair/yes modes. The discussion also adds credible safety counterexamples from a startup `doctor --fix` workaround, so this needs maintainer policy design before an automated fix PR is safe.

## What This Changes

The issue requests auto-applying safe/non-destructive doctor fixes during gateway startup, or an explicit `doctor.autoFix` option, with logs showing what was applied.

## Best Possible Solution

Define a narrow startup repair contract first: classify which doctor mutations are safe for unattended gateway startup, exclude security/sandbox/channel-availability/plugin-disablement changes unless explicitly opted in, log every changed config path with backup/recovery details, and keep broader repair behind manual `openclaw doctor --fix` or a clearly named opt-in setting.
