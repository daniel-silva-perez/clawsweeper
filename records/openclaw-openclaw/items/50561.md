## Summary

Keep open. Current main still does not auto-apply the general doctor repair sequence during ordinary Gateway startup; startup uses narrow compatibility/recovery paths and a few explicit startup writes, while broad doctor repair remains gated behind explicit repair/yes modes. The issue discussion also contains credible safety counterexamples from a startup doctor --fix workaround, and the related open PR is scoped to diagnostics on restart/update failure paths rather than unattended auto-repair.

## What This Changes

Request for Gateway startup to auto-apply safe, non-destructive doctor repairs or expose a doctor.autoFix option, while logging any applied changes so repeated manual doctor --fix warnings do not persist.

## Best Possible Solution

Define a narrow startup repair contract before implementation: only explicitly classified idempotent, low-risk mutations should run unattended; sandbox, auth, allowlist/open-policy, exec policy, channel availability, and plugin disablement changes should remain manual or require a clear opt-in. Startup repair should log every changed config path, preserve backups, document the contract, and add tests for protected/intentional config behavior.

Summary: Unattended startup repair is security-sensitive because existing doctor repair can change sandbox-adjacent, allowlist/open-policy, exec safe-bin, plugin, and channel configuration without an approved startup-safe allowlist.
