## Summary

Keep open. Current main still has targeted startup recovery plus explicit doctor repair modes, not unattended safe doctor fixes on every gateway start. The discussion also includes credible safety counterexamples from a startup `doctor --fix` workaround, so this needs maintainer product/safety design rather than cleanup closure.

## What This Changes

The issue requests automatic safe/non-destructive doctor repairs during gateway startup, or an opt-in `doctor.autoFix` setting that logs any applied changes.

## Best Possible Solution

Define a narrow startup repair contract before implementation: classify which doctor mutations are safe for unattended startup, exclude security and channel-availability changes unless explicitly opted in, log every changed config path with backup/recovery details, and keep broader repair behind manual `openclaw doctor --fix` or a clearly named opt-in setting.

## Risks / Open Questions

- Unattended startup repair could silently alter intentional security, sandbox, channel availability, allowlist, or plugin settings unless the safe mutation set is explicitly classified and logged.
- The current doctor repair sequence is broader than startup recovery, so invoking `doctor --fix` at gateway start would mix channel, plugin, allowlist, runtime-dependency, and config cleanup behavior into startup.
- The related open PR #64684 may improve diagnostics for restart/update failures, but it intentionally does not resolve this startup auto-repair request.
