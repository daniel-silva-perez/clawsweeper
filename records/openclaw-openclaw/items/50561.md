## Summary

Keep open. Current main still does not auto-apply the full doctor repair sequence on gateway start; startup has targeted recovery/degraded-startup handling and plugin auto-enable persistence, while broad config repair remains explicitly gated behind `openclaw doctor --fix`/`--repair`. The discussion also contains credible safety counterexamples where startup `doctor --fix` reportedly changed intentional sandbox and Telegram settings silently, so this needs maintainer product/safety design rather than conservative cleanup closure.

## Best Possible Solution

Keep this open for maintainer product/safety design. The best path is a narrow startup repair contract that classifies which doctor mutations are safe for unattended startup, excludes security/channel availability changes unless protected or explicitly opted in, logs every changed path with backup/recovery details, and leaves broader repair behind manual `openclaw doctor --fix` or a clearly named opt-in setting.
