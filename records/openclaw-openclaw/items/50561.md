## Summary

Keep open. Current main still does not implement unattended safe doctor repair during ordinary Gateway startup: startup uses targeted recovery/degraded loading plus a plugin auto-enable write, while the broad doctor repair sequence remains gated behind explicit repair/yes modes. The discussion also contains credible safety counterexamples from a startup `doctor --fix` workaround, so this needs maintainer policy design before implementation.

## What This Changes

The issue requests automatically applying safe, non-destructive doctor fixes on Gateway startup, or adding a `doctor.autoFix` option with logs for applied changes.

## Best Possible Solution

Define a narrow startup repair contract before adding code: only explicitly classified idempotent, low-risk mutations should run unattended, while sandbox, auth, allowlist/open-policy, exec policy, channel availability, and plugin disablement changes stay manual or opt-in. The implementation should log every changed config path, preserve backup/recovery details, document the policy, and add tests for protected/intentional config behavior.

## Reproduction Assessment

Yes. A high-confidence current-main check is available by following `startGatewayServer` to `loadGatewayStartupConfigSnapshot` and confirming the test expectation that plugin-local invalidity starts degraded with `wroteConfig: false` and a doctor-fix hint.

## Solution Assessment

No, not as the issue is phrased. Auto-applying the existing broad doctor repair sequence on startup would be too risky; the safer path is a maintainer-approved startup-safe allowlist plus clear opt-in/protected-config semantics.

## Security Review

Status: needs_attention

Summary: Unattended startup repair is security-sensitive because existing doctor repair can mutate sandbox-adjacent, channel access, plugin availability, allowlist/open-policy, and exec safe-bin configuration.

Concerns:

- **[medium] Define unattended repair safety policy first:** `src/commands/doctor/repair-sequencing.ts:52`
  - body: `runDoctorRepairSequence` covers broad repair categories, including channel mutations, allowlist/open-policy repair, stale/invalid plugin cleanup, and exec safe-bin profile repair. Gateway startup should not call this path until maintainers define a startup-safe mutation allowlist, backup/logging behavior, and opt-in rules.
  - confidence: 0.9

  - body: Invalid-plugin repair deletes the plugin config payload and disables the entry. That can be appropriate for explicit `doctor --fix`, but it is risky as an unattended startup mutation without protected config semantics and visible operator logs.
