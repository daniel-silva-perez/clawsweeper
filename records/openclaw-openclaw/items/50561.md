## Summary

Keep open. Current main still has only targeted startup recovery and explicit doctor repair modes, not unattended safe doctor fix application on every gateway start. The issue discussion also includes credible counterexamples where running doctor --fix at startup reportedly changed intentional sandbox and Telegram settings, so this needs maintainer product/safety design rather than conservative cleanup closure.

## What This Changes

Request automatic safe/non-destructive doctor repairs on gateway start, or an opt-in doctor.autoFix setting that logs what was changed.

## Best Possible Solution

Keep this open for maintainer product/safety design. The best path is a narrow startup repair contract that classifies which doctor mutations are safe for unattended startup, excludes security/channel availability changes unless explicitly opted in, logs every changed path with backup/recovery details, and leaves broader repair behind manual openclaw doctor --fix or a clearly named opt-in setting.
