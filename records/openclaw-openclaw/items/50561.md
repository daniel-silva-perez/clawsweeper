## Summary

Keep open. Current main still does not auto-apply general safe doctor fixes on gateway start: gateway startup has targeted recovery paths plus explicit `openclaw doctor --fix` hints, while the broader doctor repair sequence is only entered from explicit repair/yes modes. The discussion also includes credible safety counterexamples from a startup `doctor --fix` workaround, so this needs maintainer product/safety design rather than cleanup closure.

## What This Changes

The issue requests unattended safe doctor repairs or an opt-in `doctor.autoFix` setting during gateway startup, with logs for any applied changes.

## Best Possible Solution

Define a narrow startup repair contract before implementation: classify which doctor mutations are safe for unattended startup, exclude security/channel-availability/sandbox/plugin-disablement changes unless explicitly opted in, log every changed config path with backup/recovery details, and keep broader repair behind manual `openclaw doctor --fix` or a clearly named opt-in setting.

- Unattended startup repair could silently alter intentional security, sandbox, channel availability, allowlist, or plugin settings unless the safe mutation set is explicitly classified and logged.

- The related open PR #64684 may improve diagnostics on failure paths, but it intentionally does not settle this startup auto-repair policy.
