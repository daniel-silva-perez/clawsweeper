## Summary

Keep open. Current main has non-sandbox support for `media://inbound/*`, but the sandboxed image-tool path can still hand `media://inbound/<id>` to the sandbox bridge as a relative path, producing the reported `/workspace/media:/inbound/<id>` shape before `loadWebMedia` can hydrate the managed media URI. This is distinct from the closed sharp dependency report #74111.

## Best Possible Solution

Keep this issue open and fix the shared sandbox media resolver so canonical `media://inbound/<id>` references are mapped to the sandbox-visible `media/inbound/<id>` claim-check file before direct bridge resolution can misinterpret the URI as `media:/inbound`. Add focused sandbox resolver coverage plus a sandboxed image-tool regression test; preserve the existing non-sandbox `loadWebMedia` behavior and reject unsupported or unsafe media URI forms.

## Clownfish Work Prompt

Fix the sandboxed `media://inbound/<id>` handling bug reported in openclaw/openclaw#74123. Current behavior can resolve `media://inbound/foo.png` as a workspace-relative path like `/workspace/media:/inbound/foo.png` before `loadWebMedia` can hydrate the managed media URI. Implement the fix in the shared sandbox media path layer, preferably `src/agents/sandbox-media-paths.ts`, so canonical `media://inbound/<id>` refs with an `inboundFallbackDir` are parsed and validated, then mapped to `media/inbound/<id>` before direct bridge resolution. Do not broaden sandbox filesystem access, do not accept unsupported `media://` locations, and do not use a loose `path.basename` fallback that would accept nested or encoded path separators. Add focused tests in `src/agents/sandbox-media-paths.test.ts` for `media://inbound/<file>` resolving through the inbound fallback and unsafe/unsupported URI forms being rejected. Add or update a sandboxed image-tool regression test in `src/agents/tools/image-tool.test.ts` that uses a sandbox bridge and confirms the image tool reads the inbound media file instead of `media:/inbound`. Consider whether the same shared helper fix covers PDF/image_generate/video/music callers without per-tool changes. No changelog entry is required unless maintainers want one for this user-facing bug fix.
