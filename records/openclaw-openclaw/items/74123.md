## Summary

Keep open. Current main still sends sandboxed `media://inbound/<id>` image references through direct sandbox bridge path resolution before the inbound fallback can run, and the linked fix PRs are still open rather than merged.

## What This Changes

The issue asks for `media://inbound/<image>` image references to resolve to OpenClaw-managed inbound media instead of a malformed workspace `media:/inbound` path.

## Best Possible Solution

Keep this issue open until a reviewed fix lands. The preferred end state is a narrow sandbox media resolver path that detects canonical `media://inbound/<id>` before bridge path resolution, validates the inbound ID with the existing media rules, maps it to the sandbox-staged `media/inbound/<id>` file, and adds focused resolver plus sandboxed image-tool regression coverage; a loader-level stranded-URI guard can be considered as defense-in-depth.

## Reproduction Assessment

Yes. The source path gives a high-confidence reproduction: sandboxed image input preserves `media://inbound/<id>`, direct bridge resolution runs before inbound fallback, and Node path resolution converts that string under `/workspace` into `/workspace/media:/inbound/<id>`.

## Solution Assessment

Yes for the narrow sandbox resolver direction, with coordination. PR #74186 appears to target the root sandbox bridge corruption; PR #74278 may improve error handling for stranded URIs but does not by itself prove the valid sandbox claim-check file is mapped before bridge resolution.
