## Summary

Keep open. Current main still has the resolver ordering that can turn a canonical `media://inbound/<id>` reference into a workspace-relative `media:/inbound/...` path before managed-media hydration, and two open implementation PRs already target the reported bug with closing syntax.

## What This Changes

The issue asks OpenClaw to resolve `media://inbound/...` image references to managed inbound media instead of treating the URI as a malformed workspace path.

## Best Possible Solution

Land or replace the existing implementation PRs with one narrow fix that recognizes canonical `media://inbound/<id>` before sandbox bridge path resolution, preserves inbound ID validation and workspace boundaries, and adds focused sandbox resolver plus image-tool regression coverage. A generic `loadWebMedia` stranded-URI guard is useful as defense in depth, but should not substitute for fixing the sandbox resolver path that can corrupt the URI before `loadWebMedia` sees it.

## Reproduction Assessment

Yes. The source path is high-confidence: sandboxed image inputs reach `resolveSandboxedBridgeMediaPath`, that resolver delegates to the bridge before fallback, and standard path resolution turns `media://inbound/example.png` into `/workspace/media:/inbound/example.png`.

## Solution Assessment

Yes. Keeping the issue open while maintainers review #74186 and #74278 is the best current path; the narrow maintainable fix belongs at the shared sandbox media resolver, with optional web-media guardrails for stranded scheme inputs.
