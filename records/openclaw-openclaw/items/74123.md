## Summary

Keep open. Current main still has the sandbox resolver ordering that can corrupt a canonical `media://inbound/<id>` reference before managed-media hydration, and open implementation PRs #74186 and #74278 already reference this issue with closing syntax, so the issue should stay open until one fix lands or those PRs are resolved.

## What This Changes

The issue reports that image-tool calls using `media://inbound/<image>.png` resolve into a malformed workspace `media:/inbound/...` local path instead of the OpenClaw-managed inbound media file.

## Best Possible Solution

Land or replace the existing implementation PRs with one narrow fix that handles canonical `media://inbound/<id>` before sandbox bridge path resolution, preserves inbound media ID validation and sandbox path boundaries, and adds focused regression coverage for both the sandbox resolver and stranded scheme-bearing media inputs.

## Reproduction Assessment

Yes. The high-confidence source path is sandboxed image input to `resolveSandboxedBridgeMediaPath`, then direct bridge path resolution, where normal path resolution turns `media://inbound/example.png` into `/workspace/media:/inbound/example.png`; no live channel test was run because this was a read-only review.

## Solution Assessment

Yes. Keeping the issue open while maintainers review, combine, or close #74186 and #74278 is the best current path; the primary fix should be in the shared sandbox media resolver, with a `loadWebMedia` stranded-URI guard as defense in depth rather than a substitute.
