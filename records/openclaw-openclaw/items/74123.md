## Summary

Keep open. Current main still has the sandbox resolver ordering that can corrupt a canonical `media://inbound/<id>` image reference into a workspace-relative `media:/inbound/...` path, and two open implementation PRs already target this issue.

## What This Changes

Fix image-tool handling so `media://inbound/<image>` resolves to the managed inbound media file instead of an invalid workspace path.

## Best Possible Solution

Land or replace the existing implementation PRs with one narrow fix that handles canonical `media://inbound/<id>` before sandbox bridge path resolution, preserves inbound media ID validation and sandbox boundaries, and adds focused sandbox resolver plus image-tool regression coverage. A generic `loadWebMedia` guard is useful as defense in depth, but should not be the only fix if the sandbox resolver can still corrupt the URI first.

## Reproduction Assessment

Yes. The current-main source path plus a direct Node `path.resolve('/workspace', 'media://inbound/example.png')` check reproduces the malformed `/workspace/media:/inbound/example.png` shape without needing a live provider call.

## Solution Assessment

Yes for triage. Keeping the issue open while maintainers review or consolidate #74186 and #74278 is the best current path; queueing a third automated repair would duplicate active implementation work.
