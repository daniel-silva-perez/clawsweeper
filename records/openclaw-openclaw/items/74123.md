## Summary

Keep open. Current main still routes sandboxed `media://inbound/...` image references through the sandbox bridge resolver before any managed-media fallback, which can collapse the URI into a workspace-relative `media:/inbound/...` path. The two open closing PRs, #74186 and #74278, are implementation candidates rather than a reason to close this issue before merge.

## What This Changes

The issue requests that `media://inbound/<image>` image references resolve to managed inbound media instead of being treated as malformed workspace-local `media:/inbound/...` paths.

## Best Possible Solution

Land or replace the existing implementation PRs with one narrow fix that recognizes canonical `media://inbound/<id>` before sandbox bridge path resolution, preserves inbound ID validation and sandbox workspace boundaries, and adds focused regression coverage in both the sandbox resolver path and the shared media-loader guard if needed.

## Reproduction Assessment

Yes. The code path is deterministic: sandboxed image input reaches `resolveSandboxedBridgeMediaPath`, direct bridge resolution runs before fallback, and standard path resolution maps `media://inbound/example.png` into a `media:/inbound/...` workspace path.

## Solution Assessment

Yes. Keeping this issue open while reviewing #74186 and #74278 is the best current path; the maintainable fix belongs at the shared sandbox media resolver, with a generic `loadWebMedia` stranded-scheme guard as defense in depth rather than the only repair.
