## Summary

Keep this PR open. Current main still emits reset command hooks without a keyed prior-session dedupe; it only has a per-command `resetHookTriggered` marker that prevents one fallback path. The PR is not obsolete, but it needs rework for the unresolved review concerns before it is merge-ready.

## Best Possible Solution

Keep the PR open and rework it against `src/auto-reply/reply/commands-reset-hooks.ts`. The best path is a narrowly scoped dedupe that records the duplicate claim before the first await, is bounded to the command/request or another safe short-lived scope so valid later resets still fire, and adds regression tests for duplicate re-entry plus valid subsequent resets. No dependency, workflow, CI, release, or publishing changes are needed.

## Risks / Open Questions

- The PR's current implementation can still allow duplicate emissions if duplicate callers interleave before the dedupe key is recorded.
- A process-wide dedupe keyed only by action, session key, and previous session id can skip legitimate later `/new` or `/reset` hooks when the same session id remains stable.
- Closing this PR now would likely drop useful regression coverage and a plausible remaining bug because current main does not implement equivalent prior-session suppression.
