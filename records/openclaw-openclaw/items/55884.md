## Summary

Keep this PR open. Current main still emits reset command hooks without the PR's keyed prior-session dedupe, so the intended duplicate-suppression behavior is not implemented on main. The PR is focused and potentially useful, but it needs rework for the unresolved pre-await claim and dedupe-scope review concerns before it is merge-ready.

## Best Possible Solution

Keep this PR open and rework it against the current `src/auto-reply/reply/commands-reset-hooks.ts` implementation. The best fix is a narrow reset-hook duplicate claim recorded before the first await, scoped to a command/request or another short-lived reset context so valid later resets still fire, with regression tests for duplicate re-entry and valid subsequent resets. No dependency, workflow, release, or publishing changes are needed.

## Risks / Open Questions

- The PR's current dedupe key is recorded after an awaited hook call, so interleaved duplicate callers can still both emit the hook.
- A process-wide dedupe keyed only by action, normalized session key, and previous session id can skip legitimate later `/new` or `/reset` hooks if the prior session id remains stable.
- Closing this PR now would discard useful regression tests and a plausible remaining bug because current main does not implement equivalent prior-session duplicate suppression.
