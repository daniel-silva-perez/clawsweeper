## Summary

Keep this PR open. Current main still lacks adaptive daily-plus-idle AND semantics, so the contribution remains relevant, but the patch is not merge-ready because the adaptive hook reason can disagree with the lifecycle timestamps used for the actual reset decision and the help text says an optional defaulted field is required.

## What This Changes

The PR adds `session.reset.mode: "adaptive"` across session reset policy, config types/schema/help/docs, hook reason typing, and focused tests so daily reset and idle expiry use AND semantics.

## Best Possible Solution

Land one coherent public contract for daily-plus-idle AND semantics, either by finishing this focused adaptive-mode PR or by explicitly superseding it with the broader #59411 design, while keeping runtime logic, exported types, Zod schema, generated schema/help, docs, tests, and hook reason reporting aligned.

## Reproduction Assessment

Yes. Current main can be verified by source inspection: adaptive is absent from the reset type/schema and freshness still uses daily-or-idle OR semantics; the PR behavior can be checked with daily-only, idle-only, and both-stale policy cases.

## Solution Assessment

No, not as-is. The focused adaptive mode is a maintainable direction, but the patch should first reuse the lifecycle-based stale state for adaptive hook reasons and correct the `idleMinutes` wording/defaulting contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Use lifecycle timestamps for adaptive hook reasons:** `src/auto-reply/reply/session.ts:141-143`
  - body: The adaptive branch recomputes daily staleness from `entry.updatedAt`, while the reset decision on current main uses `sessionStartedAt` and `lastInteractionAt`. If bookkeeping updates `updatedAt` after the daily boundary while lifecycle timestamps remain stale, the session rolls over but this path returns `undefined` instead of `"adaptive"`; reuse the same lifecycle stale state or expose it from `SessionFreshness`.
  - confidence: 0.84
- **[P2] Describe adaptive idleMinutes as defaulted:** `src/config/schema.help.ts:1330`
  - body: The new help text says `idleMinutes` is required for adaptive mode, but the resolver defaults it to `DEFAULT_IDLE_MINUTES` when omitted. Users will not get a validation error, so update this text and the generated schema description to say adaptive uses the default idle window unless configured.
  - confidence: 0.91
