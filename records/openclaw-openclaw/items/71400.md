## Summary

Keep this PR open, but it is not merge-ready. Current main still lacks adaptive daily-plus-idle reset semantics, while the PR is dirty against main and has concrete review findings around lifecycle timestamp alignment and user-facing help text.

## What This Changes

The PR adds `session.reset.mode: "adaptive"` across config types, Zod/generated schema/help, session freshness logic, hook/harness reason typing, docs, and focused session tests.

## Best Possible Solution

Land one coherent public contract for daily-plus-idle AND semantics that preserves the current lifecycle timestamp model and system-event behavior, either by finishing this focused adaptive-mode PR or by explicitly superseding it with #59411.

## Reproduction Assessment

Yes. Current-main source inspection reproduces the gap: `adaptive` is absent from the reset type/schema and reset freshness still uses OR semantics; the intended behavior can be covered with daily-only, idle-only, and both-stale policy cases.

## Solution Assessment

No, not as-is. The focused mode is a plausible maintainable direction, but the branch must be rebased, adaptive hook reasons must use the same lifecycle stale state as the reset decision, and `idleMinutes` help/generated schema text must match the defaulted optional behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Use lifecycle timestamps for adaptive hook reasons:** `src/auto-reply/reply/session.ts:132-134`
  - body: The adaptive branch recomputes daily staleness from `entry.updatedAt`, while current reset freshness is based on `sessionStartedAt` and `lastInteractionAt`. A system-event or bookkeeping write after the daily boundary can refresh `updatedAt`, so the session rolls over but this path returns `undefined` instead of `"adaptive"`; reuse the same lifecycle stale state used for the reset decision.
  - confidence: 0.84
- **[P2] Describe adaptive idleMinutes as defaulted:** `src/config/schema.help.ts:1330`
  - body: The new help text says `idleMinutes` is required for adaptive mode, but the resolver defaults it to `DEFAULT_IDLE_MINUTES` when omitted. Users will not get a validation error, so the help text and generated schema description should say adaptive uses the default idle window unless configured.
  - confidence: 0.91
