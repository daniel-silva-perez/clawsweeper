## Summary

Keep open. Current main still lacks adaptive session reset semantics, so the PR remains relevant, but it is not merge-ready: the patch has a lifecycle timestamp bug in adaptive hook reason reporting, misleading `idleMinutes` help/generated-schema wording, a missing required changelog entry, and an unresolved product/API overlap with open #59411.

## What This Changes

The PR adds `session.reset.mode: "adaptive"` across session reset config types, schema/help/docs, freshness evaluation, session-end reason typing, and focused tests so daily and idle rollover use AND semantics for that mode.

## Best Possible Solution

Land one coherent daily-plus-idle AND contract after maintainers choose the public API shape. If this PR is selected, update it to use lifecycle-based stale state for adaptive hook reasons, describe `idleMinutes` as defaulted in help/generated schema, add the required changelog entry, preserve existing `daily` and `idle` semantics, and validate the focused session/config/hook lanes.

## Reproduction Assessment

Yes. Current main can be verified by source inspection: `adaptive` is absent from the reset type/schema, freshness still uses OR semantics, and docs still describe whichever condition expires first. The PR-specific hook bug is reproducible with an adaptive rollover where `sessionStartedAt` and `lastInteractionAt` are stale but `updatedAt` was refreshed after the daily boundary.

## Solution Assessment

No, not as-is. The focused `adaptive` mode is a plausible narrow implementation, but the patch should first align hook reason logic with lifecycle timestamps, fix the optional/defaulted `idleMinutes` contract text, add changelog coverage, and be reconciled with #59411's broader API proposal.
