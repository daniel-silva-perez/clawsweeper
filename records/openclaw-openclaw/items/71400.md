## Summary

Keep this PR open. Current main still lacks adaptive session reset support, so the branch is not obsolete, but the patch is not merge-ready because adaptive hook reasons use the wrong freshness timestamp, `idleMinutes` help text contradicts the implementation, the user-facing feature lacks a changelog entry, and open #59411 overlaps on the public config contract.

## What This Changes

The PR adds `session.reset.mode: "adaptive"` across reset policy evaluation, config types/schema/help, docs, session-end reason typing, and focused tests so daily and idle expiry use AND semantics for that mode.

## Best Possible Solution

Land one coherent daily-plus-idle reset contract after maintainers choose the public API shape. If this PR is selected, align adaptive hook reasons with lifecycle freshness, describe defaulted `idleMinutes` accurately in both help and generated schema text, add the required changelog entry, and preserve existing `daily` and `idle` behavior.

## Reproduction Assessment

Yes. Current main can be verified by source inspection: adaptive is absent from the reset type/schema, freshness still uses OR semantics, and docs still describe whichever condition expires first. The PR-specific hook bug is reproducible with an adaptive rollover where `sessionStartedAt` and `lastInteractionAt` are stale but `entry.updatedAt` was refreshed after the daily boundary.

## Solution Assessment

No, not as-is. A narrow `adaptive` mode is a plausible implementation, but this patch should first reuse the lifecycle freshness state for hook reasons, fix the optional/defaulted `idleMinutes` wording, add changelog coverage, and be reconciled with #59411's broader API proposal.
