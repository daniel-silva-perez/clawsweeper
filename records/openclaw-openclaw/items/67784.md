## Summary

Keep open. Current main still routes Mattermost message-tool sends through the plugin `handleAction` path, and that handler still ignores explicit `threadId` and does not forward `mediaLocalRoots` or `mediaReadFile` to the Mattermost sender. The `filePath`/`path` portion is partly mitigated by shared runner normalization, but the Mattermost-owned handler remains incomplete for local file upload into a thread, and the linked open PRs are partial implementation candidates rather than grounds to close this issue.

## What This Changes

The issue requests that Mattermost message-tool sends honor `threadId`, `filePath`/`path`, and media-local file access roots.

## Best Possible Solution

Keep this issue open until a coordinated Mattermost-owned fix lands. The end state should update `extensions/mattermost/src/channel.ts` so `handleAction` preserves existing `replyToId` precedence, accepts explicit `threadId` as the Mattermost root post fallback, resolves media aliases consistently, forwards `mediaLocalRoots` and `mediaReadFile` to `sendMessageMattermost`, and adds focused regression tests for local file upload into a thread.
