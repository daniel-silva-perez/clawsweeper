## Summary

Keep open. Current main still binds and accepts the Gateway WebSocket before startup sidecars are ready, and the WebSocket upgrade/connect path does not return a retryable not-ready response while `/readyz` reports `startup-sidecars` pending. The existing fixes cover handler-attachment ordering and a small RPC startup denylist, but not the startup-window handshake timeout reported here.

## Best Possible Solution

Keep this issue open and fix the Gateway startup/WebSocket boundary. The best implementation is either to delay accepting WebSocket upgrades until the readiness contract is satisfied, or to return a prompt retryable not-ready response during `startup-sidecars` and update Control UI, TUI, and CLI clients to retry it cleanly. Add a regression test that holds sidecars pending, connects a client during that window, and asserts no long handshake timeout.
