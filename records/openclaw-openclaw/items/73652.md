## Summary

Keep open. Current main still exposes the reported startup window: Gateway WebSocket handlers are attached and the TCP listener starts before post-attach sidecars finish, `/readyz` remains red for `startup-sidecars`, and the WebSocket upgrade/connect path does not return a retryable not-ready response for that readiness state. The earlier related startup fixes cover handler attachment ordering and selected startup-gated RPC methods, but not this handshake/upgrade boundary.

## Best Possible Solution

Keep this issue open and fix the Gateway WebSocket startup boundary. The likely product/code path is to either delay accepting normal Gateway WS upgrades until readiness is satisfied, or return a prompt retryable not-ready response during `startup-sidecars`, then make Control UI, TUI, and CLI clients retry that response cleanly. Add a regression test that holds sidecars pending, connects during the window, and proves the client does not sit until the preauth handshake timeout.
