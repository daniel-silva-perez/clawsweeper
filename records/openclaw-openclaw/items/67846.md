## Summary

Keep this PR open. Current main still lacks upfront validation for Memory Dreaming cron frequency/timezone, and the PR addresses a real failure mode with targeted SDK and memory-core tests, but the latest diff still has a merge-blocking duplicate-schedule edge: invalid-config reconciliation preserves all managed duplicates while recording only the oldest job's timezone for runtime gating.

## What This Changes

The PR adds a memory dreaming cron/timezone validator to the memory host SDK, wires it into memory-core cron reconciliation and runtime execution, updates focused tests, and refreshes plugin SDK API metadata/baseline hashes.

## Best Possible Solution

Land the validator after fixing the preserved-schedule timezone model so invalid parse/inline-frequency config can safely keep any retained last-known-good managed schedule running without accepting a mismatched day-boundary timezone. Then rebase on current main and rerun the targeted memory-core tests plus the plugin SDK API check.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Track every preserved schedule timezone:** `extensions/memory-core/src/dreaming.ts:470`
  - body: This invalid-config path intentionally leaves duplicate managed jobs in place, but it records the preserved timezone from only the oldest `primary`. The runtime guard later decides whether to run a preserved schedule by comparing the current config timezone with that single value, so if a retained duplicate with a different timezone is the job that fires, dreaming can be skipped even though the fallback schedule is valid, or run under the wrong day boundary. Track the retained managed schedule timezones as a set/consensus, or bind the runtime decision to the firing job's schedule, before using this gate.
  - confidence: 0.88
