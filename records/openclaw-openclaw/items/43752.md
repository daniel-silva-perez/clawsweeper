## Summary

Keep open: current main and the latest release still omit the gateway port from lock-file payloads, so the PR is addressing a real gap. The branch is not merge-ready as proposed because current main now parses lock payloads through `LockPayloadSchema`, and the PR does not update that schema or add schema-backed read coverage.

## What This Changes

The PR adds an optional `port` field to the gateway lock payload, writes bounded provided ports during lock acquisition, reads the field on the older parser path, and adds two persistence tests.

## Best Possible Solution

Repair the branch or open a narrow replacement that adds a bounded optional `port` to both `LockPayload` and `LockPayloadSchema`, writes valid `opts.port` values to the lock file, and covers writing, omitting, and schema-backed reading without changing lock acquisition, stale-lock recovery, port probing, or config persistence semantics.

Yes. A source-level reproduction is high confidence: on current main, `runGatewayLoop` forwards a resolved port and `acquireGatewayLock({ port: 18790 })` captures it, but the writer builds and serializes a payload with no `port` field.

## Solution Assessment

No, not as-is. The requested direction is the narrow maintainable fix, but the PR must be updated for current main's `LockPayloadSchema` parser so `readLockPayload` preserves the new field.
