## Summary

Keep this PR open, but it should not merge as-is. Current main still has the reported gap: `runGatewayLoop` forwards the effective port into `acquireGatewayLock`, but the lock payload schema and writer do not persist it. The PR direction is useful, and Greptile's range-validation note was addressed, but the branch predates the current schema-backed lock parser and needs a small rebase repair.

## What This Changes

The PR adds an optional `port` field to the gateway lock payload, writes valid provided ports into the lock file, reads the field back on the older parser path, and adds two lock-file persistence tests.

## Best Possible Solution

Repair the branch or open a narrow replacement that adds a bounded optional `port` to `LockPayloadSchema`, keeps `LockPayload` aligned with that schema, persists valid `opts.port` values in the written lock payload, and adds focused tests for writing, omitting, and schema-backed reading without changing lock acquisition, stale-lock recovery, port probing, or config persistence semantics.

Yes. A source-level reproduction is high confidence: start or call `acquireGatewayLock({ port: 18790 })` on current main, and the writer builds a payload without `port` even though the run loop already forwards `lockPort`.

## Solution Assessment

No, not as-is. The proposed direction is the narrow maintainable fix, but the branch must be updated for current main's `LockPayloadSchema` parser instead of only the older manual parser path.
