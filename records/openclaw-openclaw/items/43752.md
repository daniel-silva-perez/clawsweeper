## Summary

Current main still does not implement this PR’s intended behavior. The gateway run path forwards the effective CLI port into lock acquisition, but the lock payload type/schema and writer still omit `port`, and current gateway-lock tests do not cover lock-file port persistence. The PR remains a focused bug-fix candidate, but it needs a rebase/update for the current Zod-backed lock payload schema.

## Best Possible Solution

Keep this PR open. Rebase it onto current main, add an optional bounded `port` field to `LockPayload` and `LockPayloadSchema`, persist valid ports when `acquireGatewayLock` receives `opts.port`, and land focused `src/infra/gateway-lock.test.ts` coverage for provided-port persistence and absent-port backward compatibility.
