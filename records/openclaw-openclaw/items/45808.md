## Summary

Keep this PR open. Current main still does not implement Discord account-level `agentId` routing: Discord account config/schema do not accept `agentId`, the routing entry point only returns explicit configured bindings, there is no Discord binding synthesizer, and current docs still direct users toward explicit account-scoped bindings rather than per-account agent ownership.

## Best Possible Solution

Keep this PR open as an implementation candidate. The best path is to rebase it onto current main, add Discord account-level `agentId` to the accepted config contract, synthesize effective route bindings only in routing/runtime code without writing them back to config, preserve explicit binding precedence, define disabled/tokenless account behavior, expose effective routing in the real status surfaces, and update Discord/multi-agent docs plus focused routing/status tests.

## Risks / Open Questions

- The PR changes routing and account ownership semantics for Discord, which is permission-sensitive in multi-agent setups; maintainers should review fallback behavior before merge.
- The PR discussion identifies a concrete tokenless-account risk: synthesized bindings could make outbound auto-selection choose an unconfigured Discord account unless disabled/tokenless accounts are filtered or defined intentionally.
- The visibility part of the PR may still be incomplete unless effective account bindings propagate through the primary channel status/Mission Control path, not only helper flows.
- The PR is stale against current SDK layout: current main exposes `openclaw/plugin-sdk/routing` and has no `src/plugin-sdk/discord.ts` subpath, while the PR diff adds a Discord-specific SDK export.
