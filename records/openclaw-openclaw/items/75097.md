## Summary

Keep open. Current `main` still exhibits both reported audit-warning behaviors, and this PR is a narrow implementation for the linked report; the only concrete review gap I found is that user-facing security audit docs still describe the old trigger/severity model.

## What This Changes

The PR changes security audit collectors, tests, and changelog text so prefix-constrained hook session-key overrides report at INFO and the multi-user heuristic fires only when open or wildcard broadening signals are present.

## Best Possible Solution

Merge a version of this PR after updating the security audit docs/check matrix to match the new severities and trigger criteria, then let the linked report close through the PR merge path once CI has covered the touched audit tests.

## Reproduction Assessment

Yes. A high-confidence reproduction path exists by source inspection on current `main`: the hook audit always warns on `allowRequestSessionKey=true`, and the trust-model heuristic fires for configured group targets alone; the existing tests also encode the old allowlist-only warning path.

## Solution Assessment

Yes, with a docs follow-up. The code change is a narrow maintainable fix because it preserves WARN/CRITICAL coverage for missing hook prefixes and open/wildcard channel policies while removing the reported scoped single-operator false positives.
