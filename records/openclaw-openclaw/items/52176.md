## Summary

Keep open. Current main still does not implement an optional Docker uv build arg, and this PR remains a plausible focused contribution, but its latest diff is not merge-ready because it adds unpinned, unverified remote script execution in the Docker build path and does not wire the new arg through the maintained setup/docs surfaces.

## Best Possible Solution

Keep the PR open for review or replacement with a hardened Docker implementation: pin a uv version, verify the downloaded artifact or checksum, install both `uv` and `uvx`, set a stable node-user tool binary path for `uv tool install`, wire the arg through Docker/Podman setup scripts and docs, and add a targeted Dockerfile/setup test or smoke that proves uv-backed skill install works in the non-root runtime image.

## Risks / Open Questions

- Do not merge the current diff as-is: the Docker build would execute a remote latest installer as root without a pinned version or integrity verification.
- Even after installing uv/uvx, uv-installed skill executables can land outside the non-root `node` user's PATH unless the image sets a stable tool executable directory or PATH policy.
- The proposed build arg is not integrated into `scripts/docker/setup.sh`, `scripts/podman/setup.sh`, or Docker docs, so maintained setup flows would not expose it consistently.
