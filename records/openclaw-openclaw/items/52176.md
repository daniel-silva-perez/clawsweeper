## Summary

Keep open. The request is plausible, but this PR is still an open Docker infrastructure change with concrete pre-merge issues: the branch adds mutable remote installer execution during image build and does not fully handle the runtime path needed by uv-installed tools. Current main does not already implement the new build arg, and the discussion confirms an existing custom-build workaround, so this should stay in maintainer review rather than be auto-closed or replaced automatically.

## What This Changes

The PR adds an optional `OPENCLAW_INSTALL_UV` Docker build arg that downloads Astral's uv installer during the runtime image build and moves `uv` and `uvx` into `/usr/local/bin`.

## Best Possible Solution

If maintainers want first-class uv support in the official Docker surface, install uv from a pinned and verifiable artifact, expose both launchers and uv-installed tool shims to the `node` runtime user via `UV_TOOL_BIN_DIR` or `PATH`, and add Docker docs plus a focused Dockerfile/unit check and container smoke. If maintainers do not want that image surface, document the existing custom-image workaround instead.

## Reproduction Assessment

Yes for the review findings. The source-level reproduction is clear from the PR diff plus current main: the new build arg runs an unpinned remote installer, while OpenClaw installs uv-backed skills with `uv tool install` and then checks `requires.bins` against PATH-visible executables.

## Solution Assessment

No. The proposed patch is not the best way to solve the request because it makes builds nondeterministic and incomplete for uv-installed tool commands; the safer path is a pinned/verifiable install plus explicit runtime tool-bin handling, or documenting the existing custom apt build-arg workaround.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Pin and verify the uv installer before executing it:** `Dockerfile:203`
  - body: This build step runs whatever `https://astral.sh/uv/install.sh` serves at build time as root whenever the arg is enabled. Rebuilding the same OpenClaw commit can execute different installer bytes, and a compromised upstream endpoint would be baked into the image; use a pinned and verifiable uv artifact instead.
  - confidence: 0.94
- **[P2] Expose uv tool installs to the node user's PATH:** `Dockerfile:198-203`
  - body: Moving `uv` and `uvx` to `/usr/local/bin` does not make commands created by `uv tool install` visible. OpenClaw installs uv-backed skills with `uv tool install` and then checks `requires.bins` on PATH, so a skill like `nano-pdf` can install successfully but still be filtered out unless the image sets `UV_TOOL_BIN_DIR` or adds the node tool-bin directory to PATH.
  - confidence: 0.87
