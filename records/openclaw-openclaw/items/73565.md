## Summary

Keep this issue open. Current main still has transcript persistence paths that call `SessionManager.open(...).appendMessage(...)` without `guardSessionManager`, while the existing guard is the path that applies transcript redaction before persistence. The issue is also paired with open, unmerged PR #73563, so policy keeps this open until that PR or an equivalent fix lands.

## What This Changes

The issue requests applying the existing session transcript redaction guard to bare `appendMessage` disk-persistence paths in core, CLI attempt execution, gateway injection, and the bundled Codex transcript mirror.

## Best Possible Solution

Land PR #73563 or an equivalent targeted fix after security review, ensuring every bare transcript append entry point uses `guardSessionManager` and threads configuration into redaction where applicable, then close this issue after merge.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path exists: pass token-bearing text through the config transcript append, CLI attempt transcript, or gateway inject helper with redaction enabled; current main writes through bare `SessionManager.appendMessage` while the guarded run/compact paths show the expected pre-write redaction path.

## Solution Assessment

Yes. Reusing `guardSessionManager` at the identified append sites is the narrowest maintainable fix because it applies the existing hook, tool-result guard, and transcript redaction infrastructure instead of adding a parallel scrubber.
