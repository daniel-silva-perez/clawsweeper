## Summary

Keep this issue open. Current main still has unguarded session transcript append paths, and the linked open PR #73563 is the active implementation candidate for the same security-sensitive sink family.

## What This Changes

Ensure all session transcript `appendMessage` persistence paths pass through the existing transcript redaction guard before writing message payloads to disk.

## Best Possible Solution

Review and land PR #73563 or an equivalent maintainer-approved fix that routes every bare transcript append sink through `guardSessionManager` with config where applicable, then close this issue after the guarded behavior has source and targeted test coverage.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path exists: send token-bearing text through the config transcript append, CLI text-turn transcript, gateway inject helper, or Codex mirror while redaction is enabled; current main shows direct `SessionManager.appendMessage` calls on those paths while the guarded attempt/compact paths show the expected hook.

## Solution Assessment

Yes. Reusing `guardSessionManager` is the narrowest maintainable direction because it applies the existing hook, redaction, and tool-result guard infrastructure instead of adding a parallel scrubber.
