## Summary

Keep this issue open. Current main still has transcript append paths that bypass `guardSessionManager`, and PR #73563 is an open, unmerged implementation candidate explicitly aimed at this security-sensitive gap.

## What This Changes

The issue reports that session transcript `appendMessage` write paths persist messages without consistently passing through the existing redaction guard infrastructure.

## Best Possible Solution

Review PR #73563 against the current transcript persistence contract, ensure every bare append sink uses the existing guard/redaction path with config where applicable, and close this issue only after that PR or an equivalent fix merges.

## Reproduction Assessment

Yes. A focused source-level reproduction path exists: send a token-bearing message through the config transcript append, CLI attempt transcript, or gateway inject helper with redaction enabled; current main appends through bare `SessionManager` instances while the guarded attempt/compact paths show the expected redaction hook.

## Solution Assessment

Yes. Reusing `guardSessionManager` at the identified append entry points is the narrowest maintainable direction because it applies the existing hook and redaction infrastructure instead of introducing a parallel scrubber.
