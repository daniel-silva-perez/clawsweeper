## Summary

Keep open. Current main still has transcript persistence sinks that append messages without the existing guarded redaction path, and the linked open PR #73563 is the focused implementation candidate that should be reviewed before this tracker is closed.

## What This Changes

Request applying existing transcript redaction guard infrastructure to bare `appendMessage` persistence paths in core session management, CLI attempt execution, gateway inject, and bundled extension mirror surfaces.

## Best Possible Solution

Land a reviewed fix that wraps each remaining bare transcript write entry point with the existing guard, threads `OpenClawConfig` where user-configured redaction applies, preserves Pi parentId/idempotency/update semantics, and adds regression tests for each sink.

## Reproduction Assessment

Yes. Static current-main proof is high-confidence: the named sinks call `appendMessage` without `guardSessionManager`, while the guard tests show redaction is applied through the guarded path; a focused runtime repro would append a message containing a configured redaction pattern through each API and inspect the JSONL output.

## Solution Assessment

Yes. Reusing the existing guarded session-manager path is the narrowest maintainable direction; PR #73563 should be reviewed for config propagation, update events, and plugin SDK boundary safety before merge.
