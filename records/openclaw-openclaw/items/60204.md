## Summary

Keep this issue open. Current main still has no implementation that detects, retires, or hardens a host-local `~/.openclaw/bin/ensure-whatsapp.sh` crontab health check. The existing legacy cron doctor path only repairs OpenClaw cron store JSON, and the gateway service diagnostics only scan managed service definitions/systemd paths, not user crontabs or the reported `whatsapp-health.log` false-positive path.

## Best Possible Solution

Add a targeted Linux doctor or health diagnostic that reads the user's crontab in read-only mode, detects entries invoking `~/.openclaw/bin/ensure-whatsapp.sh`, and warns that the cron-based WhatsApp health check is legacy or unsupported because it can misreport gateway health when the systemd user bus is unavailable. If any maintained installer still creates the script, retire that install path or harden the script so `systemctl --user` bus failures are logged as environment problems rather than `Gateway inactive`. Keep this issue open until that behavior is implemented or explicitly descoped by a maintainer.
