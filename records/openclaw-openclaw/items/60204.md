## Summary

Keep this issue open. Current main handles OpenClaw-managed cron store migrations and systemd user-bus failures in the daemon/service paths, but I found no implementation that detects or retires the reported host-local `~/.openclaw/bin/ensure-whatsapp.sh` crontab entry or its misleading `whatsapp-health.log` messages. The related open PR explicitly calls this a separate legacy WhatsApp health-check issue rather than fixing it.

## Best Possible Solution

Add a targeted doctor/health diagnostic for Linux user crontabs that invoke `~/.openclaw/bin/ensure-whatsapp.sh`, warn that the cron-based WhatsApp health check is legacy/unsupported and can misreport when the systemd user bus is unavailable, and provide explicit removal or migration guidance. If the script is still distributed by an installer, either retire that install path or harden the script so `systemctl --user` bus failures are logged as environment problems rather than `Gateway inactive`.

## Risks / Open Questions

- Closing now would leave the false-positive legacy WhatsApp health log path untracked for affected Linux users.
- A fix should avoid mutating user crontabs automatically unless the user opts in; a read-only doctor warning plus explicit repair/removal guidance is the conservative first step.
- Before retiring the script path completely, maintainers should confirm whether an installer or older distribution outside this source tree still creates `~/.openclaw/bin/ensure-whatsapp.sh`.
