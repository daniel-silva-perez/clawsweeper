## Summary

Keep this issue open. Current main still has no implementation that detects, retires, or hardens a host-local `~/.openclaw/bin/ensure-whatsapp.sh` crontab health check; existing doctor coverage is scoped to OpenClaw cron-store JSON, while systemd user-bus fallback code only helps managed gateway service operations. The related open PR explicitly treats this legacy WhatsApp health-check path as separate work.

## What This Changes

Detect, retire, or harden legacy user-crontab `ensure-whatsapp.sh` checks that can falsely log gateway inactivity when `systemctl --user` cannot reach the user bus.

## Best Possible Solution

Add a targeted Linux doctor/health diagnostic for user crontabs that invoke `~/.openclaw/bin/ensure-whatsapp.sh`, preferably through a plugin-owned WhatsApp doctor surface or another additive seam that keeps core mostly generic. The diagnostic should warn that the cron-based WhatsApp health check is legacy or unsupported and can misreport when the systemd user bus is unavailable; if any maintained installer still creates the script, retire that path or harden the script to log bus-environment failures instead of `Gateway inactive`.

## Reproduction Assessment

Yes for the underlying failure mode: the issue gives an env-isolated `systemctl --user is-active openclaw-gateway.service` command that reproduces `Failed to connect to bus: No medium found`. Full end-to-end reproduction depends on an affected host having the legacy crontab and script, which are not present in current main.

## Solution Assessment

Yes, a read-only doctor warning with explicit guidance is the narrowest maintainable first fix. Retiring or hardening the script is appropriate only after confirming whether a supported installer still distributes it.
