## Summary

Keep open. Current main partially addresses runaway tool loops with configurable loop detection, but the narrow edit no-op gap from the report is still present: OpenClaw validates edit shape without rejecting identical `oldText` and `newText`, then delegates to the base edit tool before any OpenClaw-owned no-op rejection.

## What This Changes

The issue requests an edit-tool pre-validation guard for identical replacement text plus broader repeated tool-error circuit-breaker behavior and agent guidance.

## Best Possible Solution

Add a shared edit pre-validation guard that rejects any replacement whose `oldText` equals `newText` with a clear non-retryable no-op error before the base edit tool executes, covering both host and sandbox edit paths with focused tests. Leave global loop-detection default behavior unchanged unless maintainers separately decide to change policy.

## Reproduction Assessment

Yes. A focused unit check can wrap a dummy edit tool with `REQUIRED_PARAM_GROUPS.edit` or exercise the host/sandbox edit factory using `edits: [{ oldText: "same", newText: "same" }]`; current source shows that payload passes OpenClaw validation and reaches the base edit tool.

## Solution Assessment

No, current main is not the completed solution for the edit-safety part because it relies on downstream edit behavior instead of rejecting the no-op parameters in OpenClaw. The best repair is narrow edit-specific validation, not a default global circuit-breaker change.
