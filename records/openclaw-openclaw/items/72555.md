## Summary

Keep open. Current main still only partially covers the report: edit replacement validation accepts identical `oldText` and `newText`, the edit recovery wrapper reaches the upstream edit tool without a no-op guard, and the broader tool-loop circuit breaker remains opt-in by design.

## What This Changes

The issue requests edit-tool pre-validation for identical replacement text, clearer non-retryable edit errors, and a circuit breaker for repeated tool failures.

## Best Possible Solution

Add an OpenClaw-owned edit validation guard that rejects identical `oldText` and `newText` before the base edit tool runs, with a clear no-op/non-retryable error and focused host/sandbox coverage; leave global circuit-breaker defaults unchanged unless maintainers separately decide to change policy.

## Reproduction Assessment

Yes. A focused unit path can wrap a dummy edit tool with `REQUIRED_PARAM_GROUPS.edit` and pass `edits: [{ oldText: "same", newText: "same" }]`; current source shows that payload passes validation and reaches the base tool. I did not execute a mutating edit during this read-only review.

## Solution Assessment

No, current main is not the best completed solution because it lacks the edit-specific guard. The best narrow repair is pre-validating identical edit replacements; enabling default global loop blocking should stay a separate maintainer decision.
