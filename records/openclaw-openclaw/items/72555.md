## Summary

Keep open. Current main still has only partial safeguards: edit parameter validation accepts identical replacements, the edit recovery wrapper delegates to the upstream edit tool before any no-op rejection, and repetitive tool-loop blocking remains disabled by default. The narrow edit pre-validation gap is actionable; the broader default circuit-breaker policy needs maintainer judgment.

## What This Changes

Request to add edit-tool no-op pre-validation, repeated tool-failure circuit breaking, retryability classification, and agent guidance for repeated tool errors.

## Best Possible Solution

Add an OpenClaw-owned edit validation guard that rejects replacements where `oldText` and `newText` are identical before the upstream edit tool runs, with a clear non-retryable error and focused tests for host/sandbox edit creation. Leave default loop-detection policy unchanged unless maintainers separately decide to make broader circuit-breaker behavior default-on.

## Reproduction Assessment

Yes. A focused test can call the OpenClaw edit validation path with `edits: [{ oldText: "same", newText: "same" }]`; current source shows that payload satisfies validation and reaches the wrapped base edit tool. I kept the review read-only and did not execute a mutating edit.

## Solution Assessment

No, current main is only a partial solution. The best narrow fix is edit-specific pre-validation; enabling the global circuit breaker or changing retry policy by default is a broader product decision because the existing docs and tests intentionally keep loop detection opt-in.
