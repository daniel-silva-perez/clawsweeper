## Summary

Keep open. Current main still lacks the edit-specific no-op guard for identical `oldText` and `newText`, while the broader repeated-tool circuit breaker remains opt-in by documented policy, so the narrow edit validation gap is the actionable part.

## What This Changes

Request edit-tool pre-validation for identical replacement text, clearer non-retryable edit errors, repeated tool-failure circuit breaking, and agent retry guidance.

## Best Possible Solution

Add an OpenClaw-owned edit validation guard that rejects identical `oldText` and `newText` before the base edit tool runs, with a clear no-op/non-retryable error and focused host/sandbox coverage; leave global circuit-breaker defaults unchanged unless maintainers separately decide to change policy.

## Reproduction Assessment

Yes. A focused source-level reproduction can wrap a dummy edit tool with `REQUIRED_PARAM_GROUPS.edit` and call it with `edits: [{ oldText: "same", newText: "same" }]`; current validation would pass that payload through to the base tool. I did not run a mutating edit during this read-only review.

## Solution Assessment

No. Current main is not the best completed solution because the edit-specific guard is missing; the narrow maintainable fix is pre-validating identical edit replacements while keeping default loop-detection policy unchanged.
