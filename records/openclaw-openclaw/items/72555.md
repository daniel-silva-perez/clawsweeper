## Summary

Keep open. Current main still has only partial safeguards: edit replacement validation accepts identical strings and delegates to the upstream edit tool, while the broader repeated-tool loop breaker remains intentionally opt-in by source, docs, and tests.

## What This Changes

The issue requests edit-tool pre-validation for identical `oldText`/`newText` replacements plus stronger circuit-breaker and error-classification behavior for repeated tool failures.

## Best Possible Solution

Add an OpenClaw-owned edit validation guard that rejects any replacement where `oldText` and `newText` are identical before the upstream edit tool runs, with a clear non-retryable error and focused host/sandbox coverage. Keep the broader default circuit-breaker policy separate unless maintainers explicitly decide to make loop detection stricter by default.

## Reproduction Assessment

Yes. A source-level reproduction is to invoke the OpenClaw edit validation path with `edits: [{ oldText: "same", newText: "same" }]`; current validation accepts that shape and the wrapper delegates to `base.execute`. I did not run a mutating edit because this review is read-only.

## Solution Assessment

No. Current main is only a partial solution: it has configurable loop detection and edit recovery, but not the requested no-op edit pre-validation. The narrow validation guard is the maintainable repair; default-on global circuit breaking should remain a separate maintainer decision.
