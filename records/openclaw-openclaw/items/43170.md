## Summary

Keep this PR open. Current main has superseded part of the branch with a generic configured-binding registry and a working Feishu ACP route/readiness path, but it still does not implement the PR's remaining central behavior: `acp.defaultChannels`, catch-all ACP bindings without `match.peer.id`, QQ Bot configured ACP routing, or acpx auto-enable from ACP bindings alone. The PR discussion remains useful, including the resolved reverse-lookup concern and the companion QQ Bot plugin pointer, so this is not a conservative cleanup close.

## Best Possible Solution

Keep this PR open for maintainer follow-up, ideally as a rebase or split. Preserve current main's generic configured-binding registry and shipped Feishu path, then port the QQ Bot portion by adding a QQ Bot configured-binding provider plus inbound `resolveConfiguredBindingRoute` / `ensureConfiguredBindingRouteReady` handling through `openclaw/plugin-sdk/conversation-runtime`, with route/provider tests and QQ Bot docs. Treat `acp.defaultChannels`, catch-all ACP bindings, and acpx auto-enable from bindings as a separate product/security decision.

## Risks / Open Questions

- `acp.defaultChannels` and catch-all bindings can route every conversation on selected channels into an external ACP harness, so they need explicit product/security review before landing.
- Auto-enabling the acpx runtime from ACP bindings/defaultChannels broadens external CLI execution activation beyond current main's ACP config gates.
- QQ Bot integration must preserve existing access control, group policy, mention gating, queueing, and failure behavior while adding configured binding route readiness.
- The PR branch predates the current generic configured-binding registry and should be rebased or split to avoid reintroducing older ACP-specific route helper seams.
