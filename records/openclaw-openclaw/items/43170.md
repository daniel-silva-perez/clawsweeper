## Summary

Keep this PR open. Current main has already superseded part of the PR with a generic configured-binding registry and Feishu ACP route readiness, but it still does not implement the PR's central remaining behavior: `acp.defaultChannels`, catch-all ACP bindings without `match.peer.id`, QQ Bot configured ACP routing, or acpx auto-enable from ACP bindings alone. The PR discussion is still useful, including the resolved reverse-lookup concern and the companion QQ Bot plugin pointer, so this is not cleanup-closeable.

## Best Possible Solution

Keep this PR open for maintainer follow-up, ideally as a rebase or split. Preserve current main's generic configured-binding registry and shipped Feishu path, then port the QQ Bot portion by adding a QQ Bot configured-binding provider plus inbound `resolveConfiguredBindingRoute` / `ensureConfiguredBindingRouteReady` handling through `openclaw/plugin-sdk/conversation-runtime`, with route/provider tests and QQ Bot docs. Treat `acp.defaultChannels`, catch-all ACP bindings, and acpx auto-enable from bindings as a separate product/security decision.

## Risks / Open Questions

- The PR's `acp.defaultChannels` and catch-all bindings can route every conversation on selected channels into an external ACP harness, which needs explicit product/security review before landing.
- Auto-enabling the acpx runtime from ACP bindings/defaultChannels broadens external CLI execution activation from config that current main does not treat as ACP runtime configuration.
- The latest PR discussion includes an unresolved concern that synthesized default-channel bindings may ignore `acp.defaultAgent`, making the documented quick-start route to the wrong harness agent.
- QQ Bot is bundled on current main, but its engine path has no configured-binding provider or route readiness handling; a rebase must preserve existing access control, group policy, mention gating, and failure behavior.
