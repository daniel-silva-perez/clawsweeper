## Summary

Keep open. Current main has already implemented the Feishu side through the newer registry-backed configured-binding architecture, but this PR is not fully obsolete: `acp.defaultChannels` is still absent, catch-all ACP bindings without `match.peer.id` are still rejected/skipped, and QQ Bot still lacks configured ACP binding provider/inbound route readiness handling. The PR discussion also contains useful resolved review notes and a linked QQ Bot companion implementation that should inform a rebased split.

## Best Possible Solution

Keep this PR open as stale-but-useful work to rebase or split. Preserve main's generic configured-binding registry, then port the remaining QQ Bot support by adding a QQ Bot configured-binding provider plus inbound `resolveConfiguredBindingRoute` / `ensureConfiguredBindingRouteReady` handling through `openclaw/plugin-sdk/conversation-runtime`, with route/provider tests and QQ Bot docs. Treat `acp.defaultChannels` and catch-all ACP bindings as a separate maintainer product/security decision.

## Risks / Open Questions

- Landing this PR as-is would likely conflict with the current registry-backed configured-binding architecture and the current narrow `openclaw/plugin-sdk/conversation-runtime` SDK path.
- `acp.defaultChannels` and catch-all bindings are broad routing controls: a small config change can send every conversation on a channel/account through an external ACP CLI runtime, so they need explicit product and security review.
- QQ Bot integration needs to fit its existing bridge/engine split without bypassing access control, mention gating, route policy, or configured-binding readiness failure handling.
- Security pass found no new workflow, dependency, lockfile, or package-publishing changes in the provided PR file list, but the runtime-routing and acpx auto-enable behavior broadens external code execution reach.
