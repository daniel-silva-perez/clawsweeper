## Summary

Keep open. Current main has superseded the narrow Feishu/type-widening part with a generic configured-binding registry and shipped Feishu ACP route readiness, but the PR is not fully implemented on main: `acp.defaultChannels` is absent, catch-all ACP bindings without `match.peer.id` are still rejected/skipped, and QQ Bot has no configured ACP binding provider or inbound configured-binding route readiness handling. The PR discussion also has useful context, including the resolved reverse-lookup review and the companion QQ Bot implementation pointer `sliverp/qqbot#156`, so the work is stale but salvageable rather than cleanup-closeable.

## Best Possible Solution

Keep this PR open for maintainer follow-up, ideally as a rebase or split. Preserve current main's generic configured-binding registry and shipped Feishu path, then port the QQ Bot portion by adding a QQ Bot configured-binding provider plus inbound `resolveConfiguredBindingRoute` / `ensureConfiguredBindingRouteReady` handling through `openclaw/plugin-sdk/conversation-runtime`, with route/provider tests and QQ Bot docs. Treat `acp.defaultChannels`, catch-all ACP bindings, and ACP auto-enable from bindings as a separate product/security decision.
