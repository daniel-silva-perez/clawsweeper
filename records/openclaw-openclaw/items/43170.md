## Summary

Keep this PR open. Current main has already superseded the narrow Feishu/type-widening part with a generic configured-binding registry and Feishu ACP route readiness, but it still does not implement the PR's central remaining behavior: `acp.defaultChannels`, catch-all ACP bindings without `match.peer.id`, QQ Bot configured ACP binding/provider support, or acpx auto-enable from ACP bindings alone. The discussion is still useful, including the resolved reverse-lookup review and the companion QQ Bot pointer `sliverp/qqbot#156`, so this is stale but salvageable rather than cleanup-closeable.

## Best Possible Solution

Keep this PR open for maintainer follow-up, ideally as a rebase or split. Preserve current main's generic configured-binding registry and shipped Feishu path, then port the QQ Bot portion by adding a QQ Bot configured-binding provider plus inbound `resolveConfiguredBindingRoute` / `ensureConfiguredBindingRouteReady` handling through `openclaw/plugin-sdk/conversation-runtime`, with route/provider tests and QQ Bot docs. Treat `acp.defaultChannels`, catch-all ACP bindings, and acpx auto-enable from bindings as a separate product/security decision.

## Risks / Open Questions

- The PR's `acp.defaultChannels` and catch-all binding behavior would route every conversation on selected channels into an external ACP harness; that is a broad product/security decision and should be reviewed explicitly before landing.
- The PR auto-enables the `acpx` runtime from ACP bindings/defaultChannels, which can start external harness execution paths from config that current main does not treat as ACP runtime configuration.
- QQ Bot is now bundled in this repo, but its current engine path has no configured-binding provider or route readiness step; a rebase must fit the existing bridge/engine split and preserve access control, mention gating, group policy, and readiness-failure handling.
- The PR diff is not a supply-chain or CI change from the provided file list, but it does touch runtime routing, SDK exports, schema, and plugin auto-enable behavior, so the security review concern is behavioral rather than dependency-origin related.
