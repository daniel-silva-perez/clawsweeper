## Summary

Keep this issue open. Current main still does not expose a TUI vi/vim input mode, and the same-author implementation PR referenced from the timeline was closed unmerged. The existing TUI composer remains a pi-tui Editor subclass with OpenClaw shortcut handling around Escape/Ctrl/Alt keys, while docs and CLI/config surfaces still only document the normal editor behavior. The related #22770 is adjacent external-editor compose work, not a replacement for in-place modal keybindings.

## Best Possible Solution

Keep the issue open and use the closed #55975 as prior art. The best fix is a focused TUI-owned implementation: add an opt-in config/CLI/env surface, implement mode-aware input handling in `src/tui/components/custom-editor.ts`, preserve current default behavior, reconcile Escape semantics with abort/overlays/autocomplete, and update TUI docs plus targeted CustomEditor tests. Treat #22770 as a separate external-editor compose feature, not a substitute for inline vi bindings.
