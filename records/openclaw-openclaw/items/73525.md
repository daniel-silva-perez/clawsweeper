## Summary

Keep open. Current main still preserves the reported split: default `openclaw status` avoids live channel status and setup fallback, then renders the Channels table from read-only discovery only, while `openclaw channels status` renders gateway `channelAccounts`. The issue is also paired with open, unmerged PR #73662 using closing syntax, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Plain `openclaw status` should populate its Channels table consistently with `openclaw channels status` for configured, running channels.

## Best Possible Solution

Keep the fast default status contract, but guarantee visible rows for configured channels: either merge reachable gateway account snapshots into missing Channels table rows or make read-only/setup discovery reliably produce generic configured-channel rows. Review, adjust, or land PR #73662 rather than starting a parallel fix.

## Reproduction Assessment

Yes. The issue has concrete command steps plus an independent Discord/Telegram confirmation, and current source/tests show the same split between default status table rendering and `channels status` gateway account rendering.

## Solution Assessment

Unclear until PR #73662 is reviewed. Its gateway-snapshot augmentation is a plausible narrow fix for reachable-gateway cases, but maintainers should confirm it preserves the default status latency contract and handles unreachable-gateway/read-only fallback behavior intentionally.
