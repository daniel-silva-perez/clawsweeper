## Summary

Keep open. Current main still has the reported split: top-level `openclaw status` fetches `channels.status` for warning overlays but renders the visible Channels table from the separate read-only discovery path, while `openclaw channels status` renders `channelAccounts` directly. There is also an open linked fix PR with closing syntax, so this issue should remain open until that PR is reviewed, merged, or superseded.

## What This Changes

Make `openclaw status` populate its Channels table consistently with live channel account data shown by `openclaw channels status`.

## Best Possible Solution

Review and land #73662 or an equivalent narrow fix that feeds reachable-gateway `channels.status.channelAccounts` into the top-level status Channels rows while preserving the read-only discovery table as the unreachable-gateway fallback, with regression coverage for non-empty live channel rows.
