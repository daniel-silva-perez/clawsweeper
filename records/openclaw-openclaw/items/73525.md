## Summary

Keep open. Current main still builds the visible `openclaw status` Channels table from read-only discovery and does not merge the gateway `channelAccounts` payload that `openclaw channels status` renders, and the linked fix PR #73662 is still an open implementation candidate rather than a merged fix.

## What This Changes

The issue requests that `openclaw status` populate its Channels table consistently with `openclaw channels status` when channels are configured and running.

## Best Possible Solution

Keep the issue open and resolve it by reviewing and landing #73662 or an equivalent narrow change that fills missing visible Channels rows from the existing gateway status snapshot while preserving the fast read-only fallback when the gateway is unavailable.

## Reproduction Assessment

Yes. The issue gives concrete CLI steps, another user confirmed the same behavior across Discord and Telegram, and current source/test inspection shows the default status path can return read-only table rows while the dedicated channels command renders live `channelAccounts`.

## Solution Assessment

Yes at the issue level: using the existing `channels.status` snapshot to fill missing visible rows is the narrow maintainable fix. The exact implementation in #73662 still needs normal PR review for row synthesis semantics and fallback behavior.
