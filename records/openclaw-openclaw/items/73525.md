## Summary

Keep open. Current main still leaves default `openclaw status` on the fast read-only channel-table path without live `channels.status` rows or setup-runtime fallback, while `openclaw channels status` renders from gateway `channelAccounts`. The issue is also paired with open closing PR #73662, so it should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue asks for `openclaw status` to show configured running channels in its Channels table instead of rendering only headers while `openclaw channels status` shows the channels correctly.

## Best Possible Solution

Keep the issue open and resolve it by landing #73662 or an equivalent narrow fix that makes default `openclaw status` show configured/running gateway channel accounts while preserving the intended fast-status contract and unreachable-gateway fallback behavior.

## Reproduction Assessment

Yes. The issue gives concrete CLI steps, a second commenter confirmed the same Discord/Telegram behavior, and current tests/source show default status still disables the live and setup-fallback paths that would provide visible rows.

## Solution Assessment

Unclear for the active fix until #73662 is reviewed or adjusted. Synthesizing missing rows from the gateway snapshot is a plausible direction, but the final solution must cover the plain `openclaw status` command path, not only `--deep` or direct overview callers.
