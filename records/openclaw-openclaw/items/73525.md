## Summary

Keep open. Current main still keeps default text status off live channel checks and setup fallback, and the visible Channels table is still returned from read-only discovery without merging the gateway `channelAccounts` rows that `openclaw channels status` renders. The issue also has an open closing PR, #73662, so it should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue asks for `openclaw status` to populate its Channels table with configured/running channels when `openclaw channels status` already reports those channels correctly.

## Best Possible Solution

Preserve the fast status contract while guaranteeing that configured/running channels are visible in `openclaw status`; reviewing and landing #73662 or an equivalent narrow fix is the clearest path, with coverage for plain status, `--deep`, and unreachable-gateway behavior.

## Reproduction Assessment

Yes. The issue has concrete command steps plus an independent Discord/Telegram confirmation, and current source/tests show the same split: plain `openclaw status` skips live `channels.status` and setup fallback while `openclaw channels status` renders gateway `channelAccounts`.

## Solution Assessment

Unclear until #73662 is reviewed. Gateway-snapshot synthesis is a plausible narrow fix for reachable-gateway cases, but maintainers should confirm it preserves the intended fast read-only behavior and handles missing/unreachable gateway cases correctly.
