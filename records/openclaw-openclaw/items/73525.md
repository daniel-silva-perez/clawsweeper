## Summary

Keep open. Current main still has the split described in the report: `openclaw channels status` renders the live `channels.status` gateway payload, while default `openclaw status` queries that payload only for issue overlays and builds the visible Channels table from a separate config/read-only plugin-discovery path. The adjacent changelog entry for #72993 covers configured Telegram setup metadata, but it does not make the top-level status table use the live channel account rows that the reporter shows working in `channels status`.

## Best Possible Solution

Keep this open and fix the CLI status owner path: when the gateway is reachable, build the visible `openclaw status` Channels rows from the same `channels.status` account snapshot used by `openclaw channels status`, while keeping the current read-only config/plugin-discovery table as the fallback for unreachable gateways. Add a regression test where `channels.status` returns Telegram account rows and default `openclaw status` renders a non-empty Channels table.
