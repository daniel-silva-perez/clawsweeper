## Summary

Keep open. Current main still has the reported split: plain text `openclaw status` keeps live channel status and setup-runtime fallback disabled, then renders the visible Channels table from read-only plugin discovery without merging the gateway `channelAccounts` rows that `openclaw channels status` displays. The issue is also paired with open closing PR #73662, so the right cleanup path is review or adjust that PR rather than closing this issue now.

## What This Changes

The issue asks for `openclaw status` to populate its Channels table for configured/running channels instead of showing only table headers while `openclaw channels status` shows those channels correctly.

## Best Possible Solution

Keep the issue open and resolve it by landing PR #73662 or an equivalent narrow fix that preserves fast default status while guaranteeing visible Channels rows for configured/running gateway accounts, with regression coverage for default status, `--deep`, and unreachable-gateway fallback.

## Reproduction Assessment

Yes. The issue provides concrete CLI steps, another commenter confirmed the same behavior on Discord and Telegram, and current code/tests show the same reproducible split between default `status` and `channels status`.

## Solution Assessment

Unclear until #73662 is reviewed. Synthesizing missing rows from the existing gateway snapshot is a narrow maintainable direction, but maintainers should confirm it is preferable to changing the read-only/setup fallback contract directly.
