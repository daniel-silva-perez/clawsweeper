## Summary

Keep open. Current main still matches the reported Synology Chat incompatibility: the active webhook handler rejects HEAD through the non-POST 405 path and still acknowledges accepted POST deliveries with 204/no body. The issue is also paired with an open same-author implementation PR using closing syntax, so it should remain open until that PR or a narrow replacement lands.

## What This Changes

Request Synology Chat webhook handling to accept HEAD probes and return HTTP 200 with a {"success":true} body for successful POST acknowledgements instead of 204 No Content.

## Best Possible Solution

Land #53441, or explicitly supersede it with a focused Synology Chat plugin change that accepts HEAD probes, returns 200 with {"success":true} for successful POST ACKs, preserves existing POST token/allowlist/rate-limit behavior, and updates the webhook plus integration tests that currently expect 204.
