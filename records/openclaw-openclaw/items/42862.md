## Summary

Keep open. Current main still does not render a user-typed bare image URL inline in the Control UI chat. Existing image rendering is attachment-, structured-block-, MEDIA-, or managed-outgoing-media-driven, while plain user text remains text/links. The Control UI CSP also intentionally blocks remote image hotlinking, so the best fix needs a Gateway-managed ingest/proxy design rather than a client-only regex-to-img change.

## Best Possible Solution

Keep this issue open for a deliberate Control UI/Gateway feature: detect eligible bare image URLs in user chat text, ingest or proxy them through same-origin managed media with SSRF/MIME/size protections, render them inline with normal text/link fallback, and add focused coverage for composer/send, chat.send history projection, grouped rendering, markdown sanitization, managed media, and CSP behavior.

## Risks / Open Questions

- A client-only regex-to-img implementation would fight the current Control UI CSP and privacy model because arbitrary remote image fetches are intentionally blocked.
- A Gateway-managed implementation needs SSRF/DNS-rebinding protections, MIME sniffing, size limits, caching/error states, and preserved text/link fallback.
- Existing channel-specific URL handling and assistant media handling should not be treated as solving the Control UI user-typed URL request.
