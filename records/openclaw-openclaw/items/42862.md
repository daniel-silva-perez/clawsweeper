## Summary

Keep open. Current main still leaves a user-typed bare image URL in Control UI chat as text/link content; image rendering is wired through explicit attachments, structured image blocks, assistant MEDIA entries, transcript media, or managed media. This overlaps the broader remote-media policy work, but it is not fully covered by the markdown-image-only tracker.

## What This Changes

Request automatic inline rendering of image URLs typed into the Control UI chat window instead of leaving them as plain text or links.

## Best Possible Solution

Implement a deliberate Control UI/Gateway media path: detect eligible bare image URLs in user chat text, ingest or proxy them through same-origin managed media under an explicit remote-media policy, render the resulting managed image inline, and keep normal text/link fallback when validation or policy blocks the URL.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: submit a message like `https://example.com/image.png` with no attachments; the UI creates a text content block, Gateway skips attachment parsing, and rendering never creates an image block.

## Solution Assessment

No for the issue’s suggested client-only regex-to-img approach. The maintainable path should be Gateway-managed or explicitly policy-controlled, because current CSP/docs intentionally prevent arbitrary remote image fetches from the operator browser.

## Security Review

Status: needs_attention

Summary: The requested behavior is security-sensitive because untrusted chat text could cause remote image fetches unless routed through a reviewed media policy.

Concerns:

- **[medium] Avoid arbitrary browser image fetches:** `docs/web/control-ui.md:348`
  - body: Rendering typed remote URLs directly as img tags would let chat content trigger requests from the operator browser, conflicting with the documented no-remote-fetch policy and enabling tracking/read-timing leakage.
  - confidence: 0.94
- **[medium] Align any renderer change with CSP:** `src/gateway/control-ui-csp.ts:47`
  - body: The current CSP omits remote http(s) image sources; a client-only renderer change would either produce broken images or require broadening img-src without the required privacy and network controls.
  - confidence: 0.93
- **[medium] Reuse SSRF-safe media ingestion:** `src/media/store.ts:239`
  - body: Gateway-side fetching must use the existing pinned-hostname, redirect, MIME, and byte-limit style protections rather than adding an ad hoc URL downloader for chat text.
  - confidence: 0.88
