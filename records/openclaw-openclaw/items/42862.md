## Summary

Keep open. Current main still does not render a user-typed bare image URL inline in the Control UI chat. Images render when they arrive as explicit attachments, structured image blocks, assistant MEDIA entries, transcript media, or managed outgoing media, but plain user text remains text/links. The existing CSP also intentionally blocks remote image hotlinking, so this needs a deliberate Gateway-managed media design rather than a client-only regex-to-img change.

## Best Possible Solution

Keep this issue open for a deliberate Control UI/Gateway feature: detect eligible bare image URLs in user chat text, ingest or proxy them through same-origin managed media with SSRF/MIME/size protections, preserve normal text/link fallback, and add focused coverage for composer/send, chat.send history projection, grouped rendering, markdown sanitization, managed media, and CSP behavior.

## Risks / Open Questions

- A client-only regex-to-img implementation would conflict with the current Control UI CSP and privacy model because arbitrary remote image fetches are intentionally blocked.
- A Gateway-managed implementation needs SSRF/DNS-rebinding guardrails, MIME and size validation, caching/error states, and preserved text/link fallback before it should ship.
- Existing channel-specific handling, such as QQBot outbound URL extraction, should not be treated as solving the Control UI chat rendering request.
