## Summary

Keep open. The latest PR diff is still an implementation candidate, but it has a concrete correctness bug: it changes any observed 250 ms Node network-family attempt timeout to 300 ms, and Node reports the same 250 ms value for both its default and an explicit operator override. Current main does not already contain the proposed process-wide timeout mutation, and the linked Telegram IPv4 fallback issue is adjacent rather than a superseding generic fix.

## What This Changes

The PR adds a guarded `net.getDefaultAutoSelectFamilyAttemptTimeout()` / `net.setDefaultAutoSelectFamilyAttemptTimeout(300)` step inside `ensureGlobalUndiciStreamTimeouts()` and extends that module's tests for default-vs-custom timeout behavior.

## Best Possible Solution

Keep the PR open and revise it so OpenClaw either preserves explicit Node network-family timeout policy, including the 250 ms case, or makes the process-wide timeout change opt-in. The PR body and tests should match the latest implementation, and validation should cover the intended built-in-fetch/provider-fetch path without regressing the existing per-dispatcher behavior.

## Review Findings

- **[P2] Preserve explicit 250 ms network-family timeouts:** `src/infra/net/undici-global-dispatcher.ts:62`
  - body: This guard treats any observed 250 ms timeout as Node's default and changes it to 300 ms. Please detect explicit flags/`NODE_OPTIONS` or make this mutation opt-in.
