## Summary

Keep this open. Current main has useful mitigations for pricing opt-out, Telegram HTTP/1 transport hardening, bounded dispatcher pools, and liveness diagnostics, but the broad Gateway degradation is not proven fixed and source still preserves two reported failure surfaces: initial pricing refresh can start before post-attach channel readiness when enabled, and stopChannel timeout still leaves the account marked running with the stale task retained.

## What This Changes

The issue reports chronic Gateway runtime degradation across pricing fetches, Telegram polling/send paths, and Gateway RPC responsiveness on long-running Windows Gateway installs, with direct same-host network probes remaining fast.

## Best Possible Solution

Keep this as the canonical broad Gateway runtime-degradation tracker until maintainers validate current main or the next release on an affected long-lived Windows Gateway, land or replace the pricing-startup subset in #73486, repair or consciously reject the stopChannel timeout recovery gap tracked by #71412/#73648, and then split any remaining RPC or Telegram transport root cause into narrower follow-ups once isolated.

## Reproduction Assessment

Yes. The issue has high-confidence reproduction evidence from detailed setup steps, repeated logs, same-host direct-network counterexamples, multiple independent corroborating comments, and a pricing-skip A/B test; this review did not perform a live Windows run.

## Solution Assessment

No complete solution is currently in place. Current main has useful mitigations, but the narrowest maintainable path is to validate the remaining behavior live, land targeted fixes for the pricing ordering and channel timeout recovery pieces, then isolate any still-broad Gateway RPC/event-loop degradation.
