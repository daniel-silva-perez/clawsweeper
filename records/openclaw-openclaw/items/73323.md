## Summary

Keep this issue open. Current main contains useful adjacent mitigations for pricing startup, Telegram polling recovery, model prewarm ordering, and readiness diagnostics, but it does not source-prove a fix for the broader Windows gateway runtime degradation described here: recurring pricing timeouts, Telegram getUpdates/sendMessage failures, and slow gateway RPCs in the same long-running process. The discussion also has multiple same-day Windows reproductions, and #73486 is an open, narrower pricing-startup candidate rather than a merged resolution for the full report.

## Best Possible Solution

Keep this issue open as the canonical tracker for the broader Windows gateway runtime degradation. Review #73486 as a narrower pricing-startup candidate, then ask for or run a focused long-running Windows gateway reproduction with Telegram enabled and models.pricing.enabled toggled, collecting event-loop delay, handle/socket counts, undici dispatcher state, RPC timings, and Telegram/pricing timing logs. If pricing deferral fixes only startup readiness, leave this issue open for the remaining process-internal runtime stalls; if a merged fix fully resolves the symptoms, close later with shipped source and release evidence.
