## Summary

Keep this issue open. Current main has useful mitigations and diagnostics, but the broad Gateway degradation is not source-proven fixed: pricing refresh still starts from the initial runtime-services path when enabled, the remote pricing timeout remains 60s, and the channel stop-timeout path still preserves the stale running task state that matches one reported Telegram recovery failure mode.

## What This Changes

The issue reports chronic Gateway runtime degradation affecting model-pricing fetches, Telegram polling/sending, and Gateway RPC responsiveness across recent releases on Windows and related hosts.

## Best Possible Solution

Keep this as the canonical broad Gateway runtime-degradation tracker until maintainers validate current main or the next release on an affected long-lived Windows Gateway, land or replace the pricing-startup subset in #73486, fix or consciously reject the stopChannel timeout recovery gap tracked by #71412, and split any still-reproducing RPC/event-loop or Telegram transport root cause into narrower follow-ups.

## Reproduction Assessment

Yes. The issue has a high-confidence reproduction path from detailed setup steps, repeated logs, multiple independent corroborating comments, same-host direct-network counterexamples, and a pricing-skip A/B test; this review did not perform a live Windows run.

## Solution Assessment

No complete solution is currently in place. Current main adds useful opt-out, transport, timeout, and diagnostics mitigations, but the narrowest maintainable path is live validation plus targeted fixes for the still-source-backed pricing-ordering and channel timeout recovery gaps before closing the broader tracker.
