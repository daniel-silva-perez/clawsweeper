## Summary

Keep this issue open. Current main has useful mitigations for pricing opt-out, Telegram HTTP/1/bounded-dispatcher transport, request timeouts, and diagnostics, but the broad Gateway degradation is not proven fixed and current source/tests still preserve important reported failure surfaces: pricing refresh can still start from initial runtime setup when enabled, the remote pricing timeout is still 60s, and the stopChannel timeout path still leaves the stale task/running state pinned by regression coverage.

## What This Changes

The issue reports chronic multi-subsystem Gateway runtime degradation across pricing fetches, Telegram polling/send operations, slow Gateway RPCs, and health checks on long-running Windows Node gateways, with corroborating cross-platform comments and pricing-skip isolation data.

## Best Possible Solution

Keep this as the canonical broad Gateway runtime-degradation tracker until maintainers validate current main or the next release on an affected long-lived Windows Gateway, review/land or replace #73486 for the pricing-startup subset, fix or consciously reject the stopChannel timeout recovery gap tracked by #71412/#73648, and then split any remaining RPC/event-loop or Telegram transport root cause into narrower follow-ups once isolated.

## Reproduction Assessment

Yes. The issue has high-confidence reproduction evidence from detailed setup steps, repeated logs, same-host direct-network counterexamples, multiple corroborating comments, and a pricing-skip A/B test; this review did not perform a live Windows run.

## Solution Assessment

No complete solution is currently in place. Current main contains useful mitigations, but the narrow maintainable path is to validate the remaining behavior live, land targeted fixes for pricing ordering and channel timeout recovery, then isolate any remaining Gateway RPC/event-loop degradation.
