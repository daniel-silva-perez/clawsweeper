## Summary

Keep this open. Current main contains useful adjacent mitigations for pricing opt-out, Telegram HTTP/1 transport, polling recovery, diagnostics, and gateway timeout fallback sessions, but the reported cross-subsystem runtime degradation is still not proven fixed. Pricing refresh still starts from the initial runtime-services path when enabled, pricing fetches still use 60s timeouts, Telegram stall handling remains mostly reactive, and the channel stop-timeout zombie-task path is still present on main with a separate open fix PR.

## What This Changes

The issue requests investigation and repair for chronic Gateway runtime degradation spanning model-pricing fetches, Telegram polling/send, and normal Gateway RPCs on Windows Node 24/25 while standalone network probes remain fast.

## Best Possible Solution

Keep this as the broad Gateway runtime degradation tracker until maintainers can run a long-lived Windows Gateway on current main or the next release with Telegram enabled, compare pricing enabled versus disabled, capture stability/event-loop/handle/socket/RPC timing data during stalls, and then either land the narrow subset PRs or split a smaller root-cause issue once the failure boundary is proven.
