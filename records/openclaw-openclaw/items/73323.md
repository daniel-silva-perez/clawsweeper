## Summary

Keep this issue open. Current main has useful adjacent mitigations for pricing opt-out, Telegram HTTP transport cleanup, diagnostics, and reactive polling restart, but the latest issue discussion and current source do not show that the broad Gateway runtime degradation is fixed; the pricing-startup subset and Telegram/channel stop-timeout subset remain separate active paths.

## What This Changes

The issue reports chronic Gateway runtime degradation where pricing fetches time out, Telegram polling and outbound API calls stall or fail, and Gateway RPCs become slow on Windows/Node even though standalone network probes to the same endpoints are fast.

## Best Possible Solution

Keep this as the canonical broad Gateway runtime degradation tracker. Review or replace the narrow pricing-startup and channel zombie-task PRs, then run a long-lived Windows Gateway with Telegram enabled on current main or the next release, compare pricing enabled versus disabled, capture stability/event-loop/handle/socket/RPC timing data during a stall, and split only once the remaining failure boundary is proven.
