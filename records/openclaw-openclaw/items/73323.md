## Summary

Keep this issue open. Current main has adjacent mitigations for pricing opt-out, Telegram transport cleanup, request timeouts, and stability diagnostics, but I did not find source-backed proof that the chronic cross-subsystem Gateway degradation is fixed. Pricing can still start from the initial runtime-services path, the remote catalog fetch timeout remains 60s, Telegram recovery is still reactive to stalls, and the normal RPC slowdown symptoms remain broader than the open pricing-startup subset in #73486.

## What This Changes

The issue reports chronic Gateway runtime degradation on Windows/Node where model-pricing fetches time out, Telegram polling and sends stall, and normal Gateway RPCs take tens of seconds despite fast direct network probes.

## Best Possible Solution

Keep this as the canonical broad Gateway runtime degradation tracker. Review, land, or replace #73486 for the pricing-startup subset, then run a long-lived Windows Gateway with Telegram enabled on current main or the next release, comparing `models.pricing.enabled` true and false while collecting stability diagnostics, event-loop metrics, handle/socket counts, Telegram transport logs, and timings for the slow RPC methods. Split only after the pricing subset is fixed and the chronic runtime/RPC behavior is either proven fixed or isolated to a narrower canonical report.
