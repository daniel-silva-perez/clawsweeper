## Summary

Keep this issue open. Current main has useful mitigations for pricing opt-out, Telegram request timeouts, HTTP/1 dispatchers, polling restart diagnostics, and stability reporting, but the broad cross-subsystem Gateway degradation is not proven fixed. Pricing refresh still starts from the initial runtime-services path when enabled, remote pricing fetches still use a 60s timeout, and the channel stop-timeout path still leaves the account marked running with a test pinning that a second start is blocked.

## What This Changes

The issue requests investigation and repair for chronic Gateway runtime degradation where pricing fetches time out, Telegram polling/send operations stall or fail, and Gateway RPCs slow dramatically on Windows 11 with Node 24/25.

## Best Possible Solution

Keep this as the canonical broad Gateway runtime-degradation tracker until maintainers validate current main or the next release on a long-lived Windows Gateway with Telegram enabled. Land or replace the narrow pricing-startup and stopChannel timeout fixes, then split any remaining process-wide RPC/event-loop or Telegram transport root cause into smaller follow-up issues once isolated.

## Reproduction Assessment

Yes. The issue gives a concrete Windows 11 + Node 24/25 + global npm + Telegram reproduction path, and multiple commenters corroborate similar Gateway-internal stalls while standalone fetch/curl to the same endpoints stays fast.

## Solution Assessment

No complete solution is in place. Current main has mitigations and a narrow Telegram timeout fix, but the remaining pricing startup path, 60s remote pricing timeout, and pinned stopChannel timeout behavior mean closing this as fixed would be premature.
