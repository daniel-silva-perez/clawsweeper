## Summary

Keep this issue open. Current main has useful mitigations for pricing opt-out and Telegram transport recovery, but I did not find source-backed or live-validation proof that the broad cross-subsystem Gateway degradation is fixed; pricing can still start from the initial runtime-services path when enabled, remote pricing fetches still use 60s timeouts, and the stopChannel timeout branch still preserves the blocked-task state matching the reported restart failures.

## What This Changes

The issue reports chronic Gateway-internal network and timer degradation on Windows/Node, including pricing fetch timeouts, Telegram polling/send failures, and slow Gateway RPCs despite fast standalone network probes.

## Best Possible Solution

Keep this as the canonical broad Gateway runtime-degradation tracker until maintainers validate current main or the next release on an affected long-lived Windows Gateway, land or replace the pricing-startup subset in #73486, repair the stopChannel timeout recovery path tracked by #71412/#73648 or an equivalent fix, and then split any remaining Gateway RPC or Telegram transport root cause into narrower follow-up issues only after it is isolated.

Yes. The issue has high-confidence reproduction evidence from detailed setup steps, repeated logs, direct-network counterexamples, multiple independent user comments, and a pricing-skip A/B test; this review did not perform a live Windows run.
