## Summary

Keep this issue open. Current main has adjacent mitigations for pricing startup and model prewarm, plus Telegram stall watchdog/transport recovery, but it still does not prove a full fix for the reported cross-subsystem Windows runtime degradation: pricing fetch 60s timeouts, Telegram polling/send stalls, slow gateway RPCs, and high handle counts after the gateway has been running.

## Best Possible Solution

Keep this open for gateway and Telegram maintainer triage. The best path is to reproduce or instrument a long-running Windows gateway with Telegram enabled, compare models.pricing.enabled true vs false, collect event-loop delay, handle/socket counts, undici dispatcher state, and RPC timing during stalls, then fix the shared runtime/network degradation or split a narrower canonical follow-up once the remaining failing subsystem is proven.
