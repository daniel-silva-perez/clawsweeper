## Summary

Keep this open. Current main has useful mitigations for pricing opt-out and Telegram transport recovery, but it still has source-backed paths matching the reported degradation: initial pricing refresh can still start during runtime setup, remote pricing fetches still use 60s timeouts, and the channel stop-timeout path still preserves a blocked task. The discussion is fresh and corroborated, and related narrow work remains open rather than merged and proven on the reported Windows/Node scenario.

## What This Changes

The issue reports chronic Gateway runtime degradation where model-pricing fetches time out, Telegram polling/sendMessage stalls or fails, and Gateway RPC calls become very slow on long-running Windows/Node Gateway processes.

## Best Possible Solution

Keep this as an active broad runtime-degradation tracker until maintainers validate current main or the next release on a long-lived Windows Gateway with Telegram enabled, settle the pricing-startup work, fix or consciously defer the stop-timeout zombie path, and then split any remaining root cause into narrower canonical issues only after the subsets are isolated.

## Reproduction Assessment

Yes. The issue provides detailed setup steps, timing/log signatures, direct-network counterexamples, and an A/B pricing-skip test, and multiple commenters report matching symptoms on separate hosts.

## Solution Assessment

No complete solution is currently in place. The existing pricing opt-out and Telegram transport hardening are useful mitigations, but current main still preserves source-level behaviors that match the reported pricing delay and channel restart failure modes.
