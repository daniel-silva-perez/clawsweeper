## Summary

Keep this open. Current main has useful mitigations around pricing opt-out, Telegram HTTP/1 transport lifecycle, restart diagnostics, and readiness/event-loop reporting, but the broad cross-subsystem Gateway degradation is not proven fixed and at least one directly related channel stop-timeout path is still pinned as broken by current tests.

## What This Changes

The issue asks OpenClaw to fix chronic Gateway-internal network/timer degradation where pricing fetches, Telegram Bot API calls, and Gateway RPCs stall or time out on long-running Windows Node processes despite fast standalone network probes.

## Best Possible Solution

Keep this as the canonical broad Gateway runtime degradation tracker until maintainers can validate current main or the next release on a long-lived Windows Gateway with Telegram enabled, land or replace the narrow pricing-startup and channel stop-timeout fixes, and then split any remaining root cause into smaller tracked issues only after the pricing and Telegram lifecycle subsets are isolated.

## Reproduction Assessment

Yes. The issue provides a high-confidence reproduction path on Windows 11 with Node 24/25, global npm OpenClaw 2026.4.23 through 2026.4.26, Telegram enabled, and a 2-5 minute wait; multiple commenters corroborate similar symptoms, and the reporter's pricing-skip A/B test narrows but does not eliminate the chronic Telegram failure.

## Solution Assessment

No complete solution is currently in place. The existing mitigations are useful, and #73486 is a plausible narrow pricing-startup fix, but current main still has 60s pricing fetches, initial pricing startup, reactive Telegram recovery, and a pinned stopChannel timeout zombie path, so the best fix requires maintainer-led triage rather than closing this as implemented.
