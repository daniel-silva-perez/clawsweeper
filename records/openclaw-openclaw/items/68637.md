## Summary

Keep this PR open. Current main still rejects empty or whitespace-only `write.content`, while the PR is a narrow use of the existing `allowEmpty` mechanism with targeted regression coverage and no blocking functional or security findings found.

## What This Changes

The PR adds `allowEmpty: true` to the `write` tool's required `content` parameter and adds tests for empty, newline-only, whitespace-only, missing, non-string, and missing-path cases.

## Best Possible Solution

Land this narrow validation change with its focused tests, keeping path, missing-content, and non-string-content rejection strict. No new core API or docs surface appears necessary for the requested behavior.

## Reproduction Assessment

Yes. On current main, `REQUIRED_PARAM_GROUPS.write.content` lacks `allowEmpty` and `assertRequiredParams` requires `value.trim().length > 0`, so `content: ""` or whitespace-only content follows the reported missing-content error path; #68636 also contains live retry logs.

## Solution Assessment

Yes. Adding `allowEmpty: true` only to the write `content` group is the narrowest maintainable fix because the shared validator already preserves strict string typing and missing-key rejection; an optional follow-up integration test could prove actual zero-byte file creation through `createOpenClawCodingTools`.
