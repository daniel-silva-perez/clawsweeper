## Summary

Keep open. Current `main` and the latest release still use a 120000ms QMD embed timeout fallback, pass that value directly to `qmd embed` for vector modes, and log the raw timeout/backoff without naming the config knob that fixes it. The QMD docs still describe first-search slowness, BM25-only mode, and search timeouts, but not the reported `qmd embed timed out after 120000ms` recovery path.

## What This Changes

The issue requests a longer or mode-aware QMD embedding timeout default, an actionable timeout warning naming `memory.qmd.update.embedTimeoutMs`, and QMD troubleshooting documentation for local GGUF embedding timeouts.

## Best Possible Solution

Implement a focused QMD-owned fix: make the unset embed timeout local-friendly for `vsearch`/`query` GGUF embedding, add a timeout warning that names `memory.qmd.update.embedTimeoutMs` and the effective value, and document the symptom and recommended values in the QMD docs/config reference.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure QMD with `searchMode: "query"` or `"vsearch"`, leave `memory.qmd.update.embedTimeoutMs` unset, and current `main` resolves 120000ms and passes it to `qmd embed`; a hung embed then logs only the raw timeout/backoff warning.

## Solution Assessment

Yes. The narrow maintainable fix is in QMD config resolution, QMD embed failure diagnostics, and QMD docs; the scheduling/disable semantics in #63626 and #73432 should remain separate.
