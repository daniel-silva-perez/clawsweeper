## Summary

Keep open. Current main still defaults QMD embedding to 120000ms, passes that timeout directly into `qmd embed` for vector modes, and logs the generic timeout/backoff without surfacing `memory.qmd.update.embedTimeoutMs`. The docs expose the knob only as a generic config row and lack the requested timeout troubleshooting entry, while the builtin local embedding path already uses longer local/self-hosted batch defaults.

## What This Changes

The issue requests a higher or mode-aware QMD embedding timeout default, an actionable timeout message naming `memory.qmd.update.embedTimeoutMs`, and QMD troubleshooting documentation.

## Best Possible Solution

Keep open and make a focused QMD vector-embed fix: raise `memory.qmd.update.embedTimeoutMs` to a local-friendly default such as 600000ms for vector modes or implement an equivalent mode-aware/default-aware policy, add an actionable timeout warning that names `memory.qmd.update.embedTimeoutMs`, and add QMD troubleshooting/config docs. Keep #63626 and #73432 separate because they track embed scheduling/disable semantics rather than the timeout budget and diagnostic hint.
