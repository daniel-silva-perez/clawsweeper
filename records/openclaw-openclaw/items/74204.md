## Summary

Keep open. Current main still falls back to a 120000 ms QMD embed timeout, passes that value directly to `qmd embed` for vector modes, and logs the raw timeout/backoff without naming the config knob. The related reports in the supplied context cover adjacent scheduling/disable behavior, not this timeout default and diagnostic gap.

## What This Changes

The issue requests a longer or mode-aware QMD embedding timeout default, an actionable timeout warning naming `memory.qmd.update.embedTimeoutMs`, and QMD troubleshooting docs for local GGUF embedding timeouts.

## Best Possible Solution

Make a focused QMD vector-embed fix: raise or make mode-aware the default embed timeout for `vsearch` and `query`, preserve explicit `memory.qmd.update.embedTimeoutMs` overrides, add an actionable timeout warning that names the config key and current/default value, and document the recovery path in the public QMD docs. Keep QMD scheduling/disable semantics tracked separately in the adjacent reports.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: with QMD `searchMode` set to `query` or `vsearch` and no `memory.qmd.update.embedTimeoutMs` override, current main resolves `120000` and passes it to `qmd embed`; timeout failures then flow into the generic backoff warning.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix, with one caveat: the maintainer should choose whether the new default is uniformly 600000 ms or only applies to vector/local-GGUF modes. The warning and docs improvements are clearly required either way.
