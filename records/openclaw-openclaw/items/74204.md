## Summary

Keep open. Current main still resolves QMD embed timeout to 120000ms for vector search modes, passes that value directly to `qmd embed`, and logs the raw timeout/backoff without naming the config fix; the docs expose the knob but still lack the requested timeout-specific troubleshooting guidance.

## What This Changes

The issue asks OpenClaw to make QMD vector embedding more reliable on local GGUF workloads by raising or adapting `memory.qmd.update.embedTimeoutMs`, surfacing that knob in timeout errors, and documenting the workaround.

## Best Possible Solution

Implement a focused QMD vector-embed fix: use a local-friendly default or equivalent mode-aware/default-aware timeout policy for `vsearch` and `query`, make timeout warnings name `memory.qmd.update.embedTimeoutMs`, and add QMD troubleshooting/config docs plus tests and a changelog entry.

## Reproduction Assessment

Yes. A high-confidence code reproduction is to resolve `memory.backend: "qmd"` with `memory.qmd.searchMode: "query"` and no `embedTimeoutMs`: current main resolves 120000ms, passes it to `qmd embed`, and logs the generic timeout/backoff message. The exact wall-clock duration of the reporter's GGUF workload was not live-reproduced in this read-only review.

## Solution Assessment

Yes. The narrowest maintainable fix is QMD-owned: adjust the QMD embed timeout default or policy, improve the timeout diagnostic, and document the knob; changing builtin memory embedding behavior or folding this into the adjacent embed scheduling issues would not directly solve the reported failure mode.
