## Summary

Keep open. Current main still has the reported QMD vector embedding timeout and diagnostic gap: the default remains 120000ms, vector modes pass that value directly to `qmd embed`, timeout failures log the raw subprocess error without naming the config fix, and the QMD docs still lack a dedicated embed-timeout troubleshooting entry.

## What This Changes

The issue requests a higher or mode-aware default for `memory.qmd.update.embedTimeoutMs`, an actionable timeout warning that names that config key, and QMD troubleshooting documentation for local GGUF embedding timeouts.

## Best Possible Solution

Implement a focused QMD vector-embed fix: use a local-friendly default such as 600000ms for vector embedding work or an equivalent mode-aware/default-aware policy, add a timeout warning that names `memory.qmd.update.embedTimeoutMs` and the current value, and add QMD troubleshooting plus config-reference guidance while leaving the separate scheduling/disable issues to their own trackers.
