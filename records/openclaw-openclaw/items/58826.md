## Summary

Keep open. Current main has usage/cost visibility, provider quota snapshots, context-size knobs, model fallback after provider errors, opt-in tool-loop detection, and internal retry caps, but I did not find built-in policy that enforces per-request, per-session, daily token, cost, or request budgets before model dispatch.

## What This Changes

Add configurable built-in token budget, request cap, rate-limit, warning/fallback, and loop guard controls across request, session, daily, global, and per-agent scopes.

## Best Possible Solution

Define an additive core budget-policy contract at the gateway or agent-runtime model-dispatch boundary, with global and per-agent config, clear warn/block/pause/fallback actions, status/doctor visibility, and tests for pre-dispatch enforcement. Keep loop detection related but separately configurable so cost/request budgets and repeated-tool-call safety do not become one ambiguous knob.

## Reproduction Assessment

Not applicable as a feature request; there is no failing reproduction to run. The source/docs/config inspection gives a high-confidence verification path that current main lacks the requested built-in budget enforcement controls.

## Solution Assessment

Unclear until maintainers choose the product/API shape. Built-in core enforcement appears to be the right end state for the central spend-control problem, but the current proposal is broad enough that a narrower accepted design should come before implementation.

## Risks / Open Questions

- Maintainers still need to choose canonical scope and action semantics: hard block, warn-only, pause, fallback, and automatic interruption have different UX and operational tradeoffs.
- Provider quota windows and usage metadata are provider-specific and sometimes approximate, so enforcement should not treat observability data as exact billing authority without clear contracts.
- The work overlaps open #42475 and #9912; a future implementation should coordinate config shape and boundaries rather than creating competing budget and loop-limit policies.
- Plugin hooks can cover some host-trusted policy cases, but the requested built-in model-dispatch guardrail likely needs explicit core/runtime semantics for pre-call enforcement and status reporting.
