## Summary

Current `main` still preserves the reported post-compaction timeout messaging gap. Overflow compaction can succeed and retry, later model idle timeouts can still enter model fallback as timeout failures, and mixed fallback exhaustion with verbose external failure details off still drops to the generic `/new` runner failure text instead of a compaction/timeout-aware recovery message.

## Best Possible Solution

Keep this issue open. The best fix is to carry structured compaction outcome state from the embedded runner through model fallback into the external reply formatter, then add a regression for successful auto-compaction followed by a retried model idle timeout and unavailable or exhausted fallback candidates with verbose failure details off. The user-facing recovery should state that compaction succeeded, the retried model call timed out, and fallback models were unavailable or exhausted.
