## Summary

Keep open. Current main still has the reported failure shape: successful compaction can be observed before retry, but a later mixed timeout plus unavailable-fallback exhaustion with verbose details off still formats as the generic external runner failure instead of a compaction/timeout-aware recovery message.

## What This Changes

Improve embedded-agent failure reporting so successful auto-compaction followed by a retried model timeout and exhausted or unavailable fallbacks preserves the real cause chain instead of collapsing to the generic `/new` fallback.

## Best Possible Solution

Carry structured successful-compaction state into the fallback-summary failure formatter and add a narrow external-chat recovery message for completed compaction followed by retried-model timeout and unavailable or exhausted fallbacks, without broadening this into model-switch UX or subagent output-race work.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is available by having the embedded run emit a completed compaction event, then throw a `FallbackSummaryError` with timeout plus billing attempts while `resolvedVerboseLevel` is off; the adjacent mixed-cause test already proves the current final payload is generic.

## Solution Assessment

Yes. The narrowest maintainable fix is to use existing structured compaction and fallback-attempt state in `runAgentTurnWithFallback` formatting; resetting sessions or changing global fallback policy would be broader and riskier than the reported messaging problem requires.
