## Summary

Keep open. Current main still does not expose the original queued `chat.send` run IDs on the eventual `agent` websocket event: collect mode can collapse multiple queued messages into one follow-up run, that follow-up run gets a fresh runtime ID, and the public agent-event schema currently carries only one `runId`. The earlier duplicate report #50296 is closed, so this issue remains the live tracker.

## What This Changes

Request to include the original queued `chat.send` run ID set on the websocket agent event when queued messages are merged or otherwise executed under a different run ID.

## Best Possible Solution

Add a backward-compatible agent-event lineage field, such as `originalRunIds`, and propagate `chat.send` idempotency keys through queued, collected, and steered message paths while preserving the existing `runId` identity used for abort, dedupe, sequencing, and session state.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: send multiple `chat.send` requests into an active session with collect/steer behavior enabled, then inspect agent websocket events; the queued follow-up path creates a fresh execution run ID and the event schema has no original ID set.

## Solution Assessment

Yes, the requested additive metadata is the right direction; replacing the actual execution `runId` with the first or last queued ID would be riskier because current abort, dedupe, sequence, and session lifecycle code treats `runId` as the runtime execution identity.
