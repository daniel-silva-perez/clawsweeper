## Summary

Current main still creates a fresh execution runId for queued follow-up and collect-mode processing, and websocket agent events do not carry the original chat.send runId set. The protocol schema has no provenance field, collect batching does not preserve grouped source run IDs, and gateway event remapping only works when a registry entry already exists for the execution run.

## Best Possible Solution

Keep this issue open and implement an additive provenance path: preserve queued chat.send client run IDs through FollowupRun and collect/steer batching, add a public agent-event field such as originalRunIds or sourceRunIds, attach it to emitted websocket events, and cover collect/steer-backlog behavior with focused gateway and queue tests. Using the first or last queued runId as the execution runId is a possible compatibility option, but additive provenance is the safer contract for third-party clients.

## Risks / Open Questions

- This touches the public Gateway websocket protocol, so any fix should be additive and compatible with existing clients that key on execution runId.
- Collect mode may intentionally produce one answer for multiple input messages; source run IDs can identify provenance, but they do not by themselves guarantee one response per original question.
- Queue metadata must stay scoped to the same route/session authorization group so source IDs from different destinations cannot be mixed into one event.
