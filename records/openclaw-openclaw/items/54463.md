## Summary

Keep this open. Current main still documents the QMD symlink-loop case as a live runtime limitation and the implementation still hands plain path/pattern collections to QMD before running `qmd update`; I found no current source, schema, or test evidence of pre-traversal symlink pruning, realpath cycle detection, or a QMD exclusion/no-follow contract that would satisfy the reported `ENAMETOOLONG` failure mode.

## What This Changes

The issue asks OpenClaw to make QMD-backed memory indexing robust against workspace-visible temporary monorepos whose package-manager symlink loops can expand paths until indexing fails with `ENAMETOOLONG`, while keeping docs and runtime behavior aligned.

## Best Possible Solution

Keep this as an active QMD traversal robustness item. Add the controlled symlink-loop repro as regression coverage, then make traversal cycle-safe/no-follow upstream or add an OpenClaw-side pre-update pruning/exclusion layer before `qmd update` can recurse through workspace-visible temporary monorepos; leave the current docs workaround in place until the runtime fix ships.
