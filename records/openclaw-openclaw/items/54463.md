## Summary

Keep this issue open. Current main still treats the QMD symlink-loop case as an unresolved runtime limitation: docs document a workaround, config only exposes plain path/name/pattern collection inputs, and the QMD manager still delegates collection scanning to qmd update without an OpenClaw-side pre-traversal symlink/cycle guard. Maintainer discussion in the provided issue context also explicitly kept the runtime issue open after the docs correction.

## Best Possible Solution

Keep this issue open as a focused QMD traversal robustness bug. Add the controlled symlink-loop repro as regression coverage, then make traversal cycle-safe/no-follow upstream or add an OpenClaw-side pre-update pruning/exclusion layer before qmd update can recurse through workspace-visible temp monorepos. Keep the current docs workaround until a runtime fix ships.

## Risks / Open Questions

- Users who enable QMD with broad indexed workspace roots can still break indexing when workspace-visible temporary monorepos include recursive package-manager symlink structures.
- A directory-exclusion feature would only satisfy this issue if pruning happens before QMD/Bun traversal follows symlinks; post-yield filtering would not prevent path expansion.
- The exact fix point remains a maintainer decision between upstream QMD traversal behavior, OpenClaw-side pre-update pruning/exclusion, or both.
