## Summary

Keep open. Current main still only closes ACP runtime state for the session being reset; it does not enumerate ACP child sessions owned by that parent. The newly linked open PRs address adjacent completion, cache-pressure, and acpx process-reaper paths, but the parent-reset child ACP cleanup gap is still not present on main.

## What This Changes

The issue requests reliable cleanup for ACP mode=run child runtimes and orphaned adapter processes after completion, parent session reset, and gateway startup.

## Best Possible Solution

Keep this as the canonical tracker until parent session reset closes owned ACP child sessions using existing session-store ownership metadata, with regression coverage for parent-plus-child reset. Completion-trigger cleanup and OS process-tree reaping should land through the existing adjacent PRs or an acpx-boundary follow-up after live validation.

## Reproduction Assessment

Yes for the core reset gap: seed a parent session plus an ACP child whose spawnedBy or parentSessionKey points at the parent, then run sessions.reset on the parent and observe that current main only calls ACP cleanup for the target key. The full OS orphan-process symptom still needs live ACP/acpx validation.

## Solution Assessment

Yes, the narrowest maintainable core fix is to enumerate matching ACP child session entries in src/gateway/session-reset-service.ts and reuse closeAcpRuntimeForSession for each one. PID/process-group cleanup should stay in the acpx/runtime boundary unless live proof shows the generic close/cancel path still leaks descendants.
