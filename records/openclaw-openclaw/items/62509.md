## Summary

Keep open. Current main still runs both reported installer finalization probes without an internal timeout, and read-only stub checks show the helper paths hang until an outer timeout kills them. The cross-referenced #73814 is useful related installer-hang evidence, but it appears to cover a separate piped-stdin/truncated-function failure rather than a canonical fix for this finalization-probe timeout issue.

## What This Changes

The issue asks for installer finalization probes to use short timeouts so `npm prefix -g` or `openclaw daemon status --json` cannot hang installation at `[3/3] Finalizing setup`.

## Best Possible Solution

Add a small portable installer timeout guard for nonessential finalization probes, apply it to npm prefix/config lookups and daemon-status detection, warn and continue on timeout, and cover the behavior with focused installer helper tests plus a changelog fix entry.

## Reproduction Assessment

Yes. A read-only harness that sources `scripts/install.sh` and stubs either `npm prefix` or the daemon-status command to sleep hangs until killed by an outer timeout, matching the reported failure class.

## Solution Assessment

Yes. The narrow maintainable fix is installer-level bounding of best-effort probes, optionally passing `--no-probe --timeout 5000` to daemon status, while keeping install success independent of those checks.

In `scripts/install.sh`, make finalization best-effort probes bounded so a slow or stuck `npm prefix -g`, `npm config get prefix`, or `openclaw daemon status --json` cannot hang `[3/3] Finalizing setup` indefinitely. Prefer a small portable shell helper that works on macOS/Linux even when GNU `timeout` is unavailable, is safe under `set -euo pipefail`, does not consume stdin, and returns an empty/failed result after a short timeout while warning and continuing.
