## Summary

Keep open. Current main still has the reported gateway shutdown watchdog, and recent shutdown hardening covers websocket/HTTP close and restart drain visibility, but I did not find a gateway shutdown contract that enumerates, cancels, or bounded-drains active ProcessSupervisor exec/process children. The issue discussion also explicitly distinguishes merged PR #61565 as an adjacent websocket-close fix, not a fix for the referenced child-process and pipe handles.

## Best Possible Solution

Keep this issue as the canonical tracker for gateway shutdown blocked by active child-process or pipe handles. The best fix is a narrow core shutdown contract: expose active ProcessSupervisor run enumeration plus bounded cancel/drain APIs, wire that into gateway shutdown before the force-exit watchdog, add privacy-safe diagnostics identifying active run ids/backends/pids and enough command context to debug, and add regression coverage proving active supervisor children cannot trip the shutdown watchdog. Treat #61565 and the current websocket/HTTP/restart-drain work as adjacent mitigations, not closure evidence.
