## Summary

Keep open. The issue asks for a distributed/multi-gateway deployment where one gateway controls many agents while another stores/processes shared memory. Current main has useful pieces for the performance problem, but it does not implement connected gateway nodes or a shared gateway memory/control-plane topology; the docs explicitly frame multiple gateways as isolated profiles/ports/state.

## Best Possible Solution

Keep this open as a product/design issue. The near-term documented path should be one Gateway controlling multiple agents with embeddings offloaded to a dedicated provider endpoint or memory plugin backend. If maintainers want the broader feature, define a memory-service/plugin contract or gateway clustering design that covers shared memory storage, concurrency, auth, and failure behavior; if plugin APIs are already sufficient, the implementation could live as a ClawHub memory plugin with core docs pointing to it.

## Risks / Open Questions

- The request overlaps with existing workarounds: one Gateway with many agents, paired nodes, remote embedding endpoints, LanceDB remote storage, and the Honcho plugin. Maintainers should clarify whether the desired outcome is embedding offload, shared memory service, or true gateway clustering.
- A true distributed gateway/shared-memory feature would touch auth, state ownership, locks, memory consistency, plugin boundaries, and operational docs, so it needs explicit product/API design rather than conservative cleanup closure.
