## Summary

Current main already fixes the reported Slack SecretRef failure: the CLI resolves scoped channel secrets before dispatch, the message action runner and Slack action adapter preserve the resolved runtime config, and Slack send now requires that config instead of re-reading raw config. The fix was landed in commit 95331e5cc52d35e9c46997a30e11ba94328b569b and shipped in v2026.4.22, which supersedes the reported 2026.4.21 install.

## What This Changes

Make `openclaw message send --channel slack` preserve resolved runtime config so SecretRef-backed Slack bot tokens work instead of being treated as missing.

## Best Possible Solution

Keep the shipped runtime-config threading behavior and close this issue as fixed; users on 2026.4.21 should upgrade to v2026.4.22 or newer, including the current v2026.4.27 release.

## Reproduction Assessment

Yes. The report gives a concrete reproduction on 2026.4.21 using Slack account SecretRefs plus `openclaw message send --channel slack`, and current source/tests exercise the exact fixed config-forwarding path.

## Solution Assessment

Yes. Resolving secrets at the CLI/runtime boundary and requiring the resolved config through channel action/send helpers is the narrow maintainable fix because it prevents ambient raw config reloads across Slack and other channel sends.
