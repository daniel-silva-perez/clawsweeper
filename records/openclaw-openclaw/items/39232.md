## Summary

Keep this issue open. Current main still does not expose a universal pre-filter raw inbound plugin hook: the typed hook catalog has no `on_raw_inbound`, `onRawInbound`, or `before_channel_filter`, public plugin registration is constrained to known hook names, and unknown typed hooks are ignored. Existing `message_received` and `before_dispatch` hooks run later than the requested pre-filter observation point, and WhatsApp still applies inbound access control before delivering messages into the downstream auto-reply/hook path. The related PR #39240 is useful prior art, but it was closed unmerged in the provided GitHub context.

## Best Possible Solution

Keep this open for maintainer API design. A good implementation would add an observe-only, privacy-aware pre-filter inbound contract to the plugin/channel SDK, wire participating channels before access control and mention gating only behind explicit operator opt-in, document the privacy model in public plugin and channel docs, and add contract plus channel-ordering tests. If maintainers decide ambient raw observation is not supported, document that policy and point users to narrower current alternatives such as opt-in `message_received` or channel-specific internal ingest behavior.
