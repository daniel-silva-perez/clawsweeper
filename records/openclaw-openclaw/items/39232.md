## Summary

Keep this issue open. Current main still lacks a universal pre-filter raw inbound plugin hook: the typed hook catalog has no `on_raw_inbound`, `onRawInbound`, or `before_channel_filter`, external plugin registration is constrained to known hook names, and unknown typed hooks are explicitly ignored. WhatsApp still runs access control before creating an inbound message for the auto-reply path, and its opt-in `message_received` hook fires later after group mention gating. Telegram and Signal have channel-specific internal mention-skip ingest paths, but those are not a cross-channel typed plugin seam. The related PR #39240 was closed unmerged in the provided GitHub context.

## Best Possible Solution

Keep this open for maintainer/API design. A good implementation would add an observe-only, privacy-aware pre-filter inbound contract to the plugin/channel SDK, wire participating channels before access control and mention gating only behind explicit opt-in, document the privacy model in public plugin and channel docs, and add contract plus channel-ordering tests. If maintainers decide raw ambient observation is not supported, document that policy and point users to current narrower alternatives such as opt-in `message_received` or channel-specific internal ingest behavior.

## Risks / Open Questions

- A pre-filter raw inbound observer would expose message content, sender identifiers, and group metadata before access control and mention gating; any implementation needs explicit opt-in, privacy documentation, and security review.
