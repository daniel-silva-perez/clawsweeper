## Summary

Keep open: current main still has the slow completion-cache timeout behavior this PR targets, and the patch has no clear correctness or security blocker. The remaining merge caveat is normal maintainer validation, especially because the public head still has some broad failing check-runs that do not appear tied to the touched files from the accessible annotations.

## What This Changes

The PR raises completion-cache write timeouts from 30s to 120s in update and doctor repair, adds non-JSON update progress and duration logs, adds focused tests, and records the fix in the changelog.

## Best Possible Solution

Land this PR or an equivalent maintainer patch once CI is current: keep the bounded 120-second cap in both update and doctor completion-cache generation, preserve JSON-mode silence and the manual-refresh hint, keep the progress output if maintainers accept the UX, and retain regression coverage for both paths.

## Reproduction Assessment

Yes. The PR body gives a concrete slow-install reproduction where direct cache refresh completes in about 36s but the same command under a 30s `spawnSync` timeout returns ETIMEDOUT, and current main still contains that 30s timeout in both affected paths.

## Solution Assessment

Yes. The patch is the narrowest maintainable fix for the reported slow-but-successful cache rebuilds; a background/asynchronous refresh could reduce waiting further, but it would be a larger behavior change than needed for this follow-up.
