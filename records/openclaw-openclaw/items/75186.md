## Summary

Keep open. Current main does not already implement Gateway plugin-management RPCs, and the PR is a plausible owner-aligned feature, but it should not merge as-is because the new public protocol surface registers request parameter schemas without typed result schemas for the structured responses it returns.

## What This Changes

The PR adds admin-scoped Gateway `plugins.*` RPC handlers and shared plugin-management services for listing, inspecting, doctoring, refreshing, installing, updating, uninstalling, enabling, and disabling plugins, plus docs, scopes, rate limits, tests, and Swift protocol parameter models.

## Best Possible Solution

Land this as a complete Gateway protocol feature only after the response contracts are schema-typed and regenerated alongside the params, with maintainer review of the remote plugin-management security boundary and the targeted handler/service/protocol tests passing.

## Reproduction Assessment

Not applicable as a feature PR against current main; the absence of the RPC family is source-verifiable in `src/gateway/server-methods-list.ts`. The review finding is reproducible by inspecting the PR branch's `ProtocolSchemas`, which registers new plugin params without matching result schemas.

## Solution Assessment

No, not yet. The high-level direction of sharing plugin-management services and exposing admin-scoped Gateway RPCs is maintainable, but the public protocol contract is incomplete without typed result schemas and generated client models for the returned payloads.
