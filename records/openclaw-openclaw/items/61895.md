## Summary

Keep open. Current main has adjacent ACP cleanup and stale-session recovery work, but the specific persistent binding repair leak is not proven fixed: OpenClaw still delegates replacement to acpx, bound persistent sessions are intentionally preserved while bindings remain active, and the pinned acpx 0.6.1 process lifecycle still appears to terminate only the immediate child rather than a whole backend process tree.

## What This Changes

Persistent ACP sessions should clean up old ACP backend process trees during ensureSession repair and gateway shutdown so orphaned claude-agent-acp and plugin-tools-serve processes do not accumulate.

## Best Possible Solution

Fix ACP process lifecycle at the acpx runtime boundary: track the spawned backend PID/process group, terminate the previous recorded tree before fresh persistent repair, close the tree on session/gateway shutdown, and then consume the released acpx version in the bundled plugin with focused OpenClaw regression coverage.

## Reproduction Assessment

Yes. The report gives a concrete reproduction path: use a persistent Telegram/DM ACP binding, restart or lose the backend, send another message, and inspect for ppid=1 claude-agent-acp/plugin-tools-serve processes after each repair cycle; I did not run the live loop in this read-only sweep.

## Solution Assessment

No. Current main has useful metadata and stale-session recovery pieces, but it does not appear to be the narrowest complete fix because it does not prove PID/process-group cleanup of the old backend tree before replacement; the safer fix is process-tree ownership in acpx plus OpenClaw integration tests.
