## Summary

Keep this issue open. Current main still uses the static `session.parentForkMaxTokens` guard and skips oversized parent-session inheritance with only a log warning; the pluggable compaction provider and checkpoint infrastructure exists, but it is not wired into the parent-fork overflow path.

## Best Possible Solution

Keep the issue open. The fix belongs in the session owner path, likely around `src/auto-reply/reply/session.ts` and `src/auto-reply/reply/session-fork.ts`: compact/checkpoint the oversized parent before forking, emit a one-time visible/operator warning when inheritance is skipped, or resolve a model-aware cap from existing context-window metadata. Add focused session tests and update the public session/config docs for the chosen behavior, reusing the shipped compaction provider and checkpoint infrastructure where appropriate.
