## Summary

Keep this issue open. Current main has real mitigation for session-store growth, including entry caps, load-time maintenance, disk-budget cleanup, and cron run-session pruning, but it still persists full `skillsSnapshot` and `systemPromptReport` payloads directly in session entries. The latest issue discussion reports the same retained-row bloat on v2026.4.26, so the central static-payload duplication is not implemented away yet.

## What This Changes

The issue asks OpenClaw to stop `sessions.json` from growing and causing gateway OOMs or lock latency by pruning ephemeral sessions, cleaning orphan artifacts, lazy-loading or reducing session metadata, and deduplicating per-session `skillsSnapshot` and `systemPromptReport` payloads.

## Best Possible Solution

Keep this as the broad session-store bloat tracker and implement a backward-compatible static-payload storage design: retain current pruning/capping mitigations, move skill snapshots and system-prompt reports out of per-session rows by hash/reference or recomputation, compact/migrate legacy rows, and add tests that many cron/subagent/Telegram sessions no longer duplicate large static payloads while existing sessions still load.
