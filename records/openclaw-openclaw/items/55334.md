## Summary

Keep this issue open. Current main has meaningful mitigation for session-store growth, including enforced session maintenance, load-time capping, cron run-session retention, and subagent auto-archive, but it still persists full `skillsSnapshot` and `systemPromptReport` payloads directly on session entries. The central duplication/OOM concern is therefore not fully fixed on main.

## Best Possible Solution

Keep this issue open as the broad session-store bloat tracker. The best fix is to keep the current retention/capping/reaper mitigations, then change the session schema/writers in `src/config/sessions/*`, `src/auto-reply/reply/session-updates.ts`, `src/auto-reply/reply/session-usage.ts`, and related agent/cron paths so static skill and system-prompt data is stored by lightweight reference/cache instead of duplicated in each row. Add compatibility migration/compaction for old rows and targeted tests proving many sessions no longer duplicate static payloads.

## Risks / Open Questions

- Closing as implemented would hide the remaining per-session duplication of static skill and system-prompt metadata.
- Closing as duplicate of narrower static-schema reports would lose the broader operational evidence here: cron retention, subagent rows, orphan artifacts, startup/list memory pressure, and multiple production data points.
- The existing retention and capping mitigations reduce worst-case growth, but they still leave large retained rows and do not prove the OOM pattern is eliminated for high-skill/high-cron installations.
