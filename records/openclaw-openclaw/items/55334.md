## Summary

Keep this issue open. Current main has useful retention and capping mitigations for session-store growth, but it still stores the full skills snapshot inline on each session entry and still writes that snapshot from normal reply, CLI agent, and isolated cron paths. That leaves the central per-session duplication problem unresolved.

## Best Possible Solution

Keep this issue open as the broad session-store bloat tracker. The best fix is to keep the current retention, capping, cron reaper, and subagent cleanup mitigations, then change the session schema and writers in `src/config/sessions/*`, `src/auto-reply/reply/session-updates.ts`, `src/agents/agent-command.ts`, and `src/cron/isolated-agent/*` so static skill snapshot data is stored by lightweight hash/reference/cache rather than duplicated per row. Add compatibility compaction for existing rows and tests proving many sessions no longer duplicate the same skill payload.
