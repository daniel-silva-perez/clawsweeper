## Summary

Keep open. Current main has shipped the owner/non-owner bearer mitigation, but it still does not implement this PR's central per-backend token-bound scope model for session key, account id, and message channel; the proposed branch is useful but not merge-ready because of token lifecycle defects and an unrelated plugin loader change.

## What This Changes

The PR replaces header-derived MCP loopback scope with registered per-backend bearer-token scope, removes scope headers/env vars from CLI MCP config, adds scoped cache invalidation and tests, and also changes the plugin public-surface loader fallback path.

## Best Possible Solution

Land a rebased, narrow security patch that keeps the shipped owner/non-owner bearer hardening, binds session key, account id, message channel, and owner status to per-backend loopback tokens, removes all mutable scope headers/env vars, fixes token revocation and server-close races, and adds negative spoofing tests for every scoped field. Split the plugin loader fallback into a separate PR if it is still needed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Guard runtime clearing by server instance:** `src/gateway/mcp-http.loopback-runtime.ts:29-30`
  - body: `clearActiveMcpLoopbackRuntime()` now unconditionally clears the global runtime and token table. Because `closeMcpLoopbackServer()` drops the active server before awaiting `server.close()`, a concurrent `ensureMcpLoopbackServer()` can start a new loopback server and register tokens, then the old close callback wipes those fresh tokens and causes valid backends to get 401s. Pass a server identity/generation into the clear path and only clear the matching runtime.
  - confidence: 0.9
- **[P2] Revoke the token even when backend cleanup fails:** `src/agents/cli-runner.ts:124-126`
  - body: The new revocation runs after `await context.preparedBackend.cleanup?.()`. If cleanup throws, `unregisterMcpLoopbackToken` is skipped, leaving a bearer token registered after the backend has torn down. Wrap cleanup and revocation so revocation runs in its own `finally`.
  - confidence: 0.88
- **[P2] Release tokens on every prepare failure:** `src/agents/cli-runner/prepare.ts:145-152`
  - body: `prepareCliRunContext` registers a token before the rest of preparation completes, but only catches `prepareCliBundleMcpConfig` failures. Later failures before the context is returned leave the token live with no backend lifecycle to revoke it. Cover all post-registration work with cleanup on error.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The patch improves the scope-binding security model, but token lifecycle bugs and an unrelated plugin loader execution-path change need security review before merge.

Concerns:

- **[medium] Registered tokens can survive failed cleanup:** `src/agents/cli-runner.ts:124`
  - body: If backend cleanup throws, the newly added unregister call is skipped, leaving a bearer token active after run teardown. That keeps stale authorization alive longer than intended and grows the token scan set.
  - confidence: 0.88
- **[medium] Old server close can wipe fresh token registry:** `src/gateway/mcp-http.loopback-runtime.ts:29`
  - body: The unconditional global clear can erase tokens registered by a concurrently started newer loopback server, destabilizing the auth boundary and causing valid backends to fail authorization.
  - confidence: 0.9
- **[low] Unrelated loader fallback changes plugin execution path:** `src/plugins/public-surface-loader.ts:116`
  - body: The PR catches module-resolution failures in the plugin public-surface loader and falls back to jiti execution. That may be reasonable, but it is unrelated to the MCP auth fix and should be split or reviewed separately as a plugin code-loading change.
  - confidence: 0.62
