## Summary

Keep open. Current main still has the plausible Windows host-exec failure path described in the report: Windows exec selects a single PowerShell executable, passes that argv to the process supervisor, and the shared spawn fallback only retries EBADF by default, not EPERM. The open Microsoft tracker only inventories this issue and does not supersede it.

## What This Changes

The issue requests a fix for Windows 11 host exec commands failing immediately with `spawn EPERM` when invoked through Feishu or the web UI.

## Best Possible Solution

Do a focused Windows exec investigation: capture the selected shell path, cwd, service/user context, original spawn error fields, and command route, then add either a narrow validated PowerShell fallback or a clearer diagnostic with mocked Windows coverage and one native Windows smoke proof.

## Reproduction Assessment

No. The issue has clear reported steps and current-main source still matches an unhandled `spawn EPERM` path, but this read-only review did not reproduce it on Windows and the report lacks logs showing the selected shell or process context.

## Solution Assessment

Unclear; there is no proposed patch. A blanket EPERM retry is not clearly the best fix for host exec, so the safer direction is a reproduced Windows-specific fallback or diagnostic tied to the failing shell-selection path.
