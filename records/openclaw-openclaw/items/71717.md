## Summary

Keep open. Current main still matches the plausible Windows host-exec failure path: Windows exec selects one PowerShell wrapper, non-PTY host exec passes that argv to the supervisor, and the shared spawn fallback still retries only EBADF by default, not EPERM. The related Microsoft tracker PR is open inventory, not an implementation that resolves the report.

## What This Changes

The issue requests fixing Windows host exec so commands sent through Feishu or the Web UI no longer fail immediately with `spawn EPERM`.

## Best Possible Solution

Keep this open and do a focused Windows exec investigation: capture selected shell path, cwd, command text, original spawn error fields, and service/user context, then add either a narrow validated fallback or a clearer diagnostic with mocked Windows coverage and native Windows smoke proof.

## Solution Assessment

Unclear; there is no proposed patch. A blanket EPERM retry is not the best default for host exec, so the safer path is to reproduce the failing Windows shell selection or add targeted diagnostics before choosing a constrained fallback.
