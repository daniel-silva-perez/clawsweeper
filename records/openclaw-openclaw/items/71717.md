## Summary

Keep open. Current main still has the reported Windows host-exec failure path available: gateway exec selects one PowerShell wrapper and hands it to the child-process supervisor, while the only spawn retry helper still retries `EBADF` rather than `EPERM`; the open Microsoft tracker only inventories this issue and does not supersede it.

## What This Changes

The issue reports that on Windows 11, every agent exec-tool command fails immediately with `spawn EPERM` even though the OpenClaw CLI itself runs.

## Best Possible Solution

Keep this open and do a focused Windows exec investigation: capture the selected shell path, cwd, original spawn error code, and user-visible command context; then add either a narrow validated PowerShell fallback or a clearer diagnostic, covered by mocked Windows exec/supervisor tests and, ideally, one native Windows smoke proof.

## Reproduction Assessment

No. The issue has plausible steps and current-main source still matches an unhandled `spawn EPERM` path, but this Linux review did not reproduce it on Windows and the report has no logs or selected-shell evidence.

## Solution Assessment

Unclear; there is no proposed patch. A blanket `EPERM` retry would be risky for host exec, so the best path is first to add targeted diagnostics or reproduce the failing Windows shell selection, then decide whether a constrained fallback is warranted.
