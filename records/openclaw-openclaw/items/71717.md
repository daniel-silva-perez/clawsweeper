## Summary

Keep open. Current main still has the plausible Windows host-exec failure path described in the report: Windows exec selects a single PowerShell wrapper, non-PTY exec hands that argv to the supervisor, and the shared spawn retry helper still retries only EBADF by default, not EPERM. The related Microsoft tracker PR is an open inventory/tracking item, not an implementation or duplicate that resolves this issue.

## What This Changes

The issue reports that on Windows 11, agent exec commands from Feishu or the web UI fail immediately with `spawn EPERM` even though the OpenClaw CLI itself runs.

## Best Possible Solution

Keep this open and do a focused Windows exec investigation: capture selected shell path, cwd, original spawn error code, and user-visible command context, then add either a narrow validated PowerShell fallback or a clearer diagnostic with mocked Windows coverage plus one native Windows smoke proof if feasible.

## Reproduction Assessment

No. The issue gives plausible Windows 11 steps and current main still matches an unhandled `spawn EPERM` path, but this review did not reproduce it on Windows and the report has no logs or selected-shell evidence.

## Solution Assessment

Unclear; there is no proposed patch. A blanket EPERM retry is not the best default for host exec, so the safer solution is to reproduce the failing Windows shell selection or add targeted diagnostics before deciding on a constrained fallback.
