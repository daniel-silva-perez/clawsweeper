## Summary

Keep open. Current main still has the plausible Windows host-exec failure path: non-sandbox exec selects one PowerShell wrapper, hands that argv to the supervisor, and the shared spawn retry helper still retries only EBADF by default, not EPERM. The open Microsoft tracker is only an inventory item and does not supersede or fix this report.

## What This Changes

Fix Windows 11 host exec so commands launched through Feishu or the web UI do not immediately fail with `spawn EPERM` on npm-installed OpenClaw.

## Best Possible Solution

Keep this open and do a focused Windows exec investigation: capture selected shell path, cwd, original spawn error code, and user-visible command context; then add either a narrow validated PowerShell fallback or a clearer diagnostic with mocked Windows exec/supervisor coverage and one native Windows smoke proof if feasible.

## Reproduction Assessment

No. The issue has plausible steps and current-main source still matches an unhandled `spawn EPERM` path, but this review did not reproduce it on Windows and the report has no logs or selected-shell evidence.

## Solution Assessment

Unclear; there is no proposed patch. A blanket EPERM retry would be risky for host exec, so the safer path is to reproduce the failing Windows shell selection or add targeted diagnostics before deciding on a constrained fallback.
