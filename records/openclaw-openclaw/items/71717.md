## Summary

Keep open. Current main still matches the plausible Windows host-exec failure path: Windows exec selects a PowerShell wrapper, non-PTY host exec sends that argv through the supervisor child adapter, and the shared spawn fallback still retries only `EBADF` by default, not `EPERM`. The related Microsoft tracker is open inventory, not a merged implementation.

## What This Changes

The issue reports that on Windows 11 with OpenClaw 2026.4.23 installed from npm, every agent exec command launched from chat or the web UI fails immediately with `spawn EPERM`.

## Best Possible Solution

Add a focused native Windows exec investigation path: capture the selected shell path, cwd, command text, service/user context, and original spawn error fields, then implement either a constrained validated fallback or a clearer diagnostic with mocked Windows coverage and a native Windows smoke check.

## Reproduction Assessment

No. The issue gives plausible Windows 11 steps and current main still has a code path where spawn-time EPERM would surface immediately, but there is no native Windows reproduction log or selected-shell evidence in the report or checkout.

## Solution Assessment

Unclear; there is no proposed patch. A blanket EPERM retry is not the best first move for host exec, so the safer solution is targeted Windows reproduction followed by a constrained fallback or diagnostic once the failing permission case is known.
