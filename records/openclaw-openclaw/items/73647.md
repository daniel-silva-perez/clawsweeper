## Summary

Keep open. This is a fresh, concrete regression with multiple same-class reproductions, and current main still lacks proof that the v2026.4.26 channel/runtime-deps CPU spin is fixed. Adjacent startup/runtime-deps mitigations exist, but the linked draft PR #73678 is still only a partial implementation candidate and not present on current main.

## Best Possible Solution

Keep this issue open and review #73678 as a focused manifest-load caching candidate. Maintainers should reproduce on a packaged npm/global or Docker gateway, collect startup trace and CPU/event-loop profile data around repeated versioned runtime-deps reads, then fix the responsible hot path in manifest loading, plugin registry snapshot reuse, bundled runtime mirror/deps staging, channel startup, or affected channel plugins. Add a regression that distinguishes the already-covered openclaw-unknown case from repeated openclaw-<version>-* stage-root inspection after gateway readiness.
