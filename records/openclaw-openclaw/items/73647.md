## Summary

Keep this issue open. Current main now carries the useful manifest-load cache slice from the discussion, but the report is a fresh packaged-gateway CPU regression with Telegram, Mattermost, Docker, macOS, and acpx datapoints. I do not see enough source, test, release, or live-runtime proof that the broader staged runtime-deps/channel-runtime spin is fixed.

## Best Possible Solution

Keep this issue open until maintainers either land and validate the full packaged-gateway fix, or explicitly consolidate the remaining runtime-deps/channel-runtime CPU work under #73532. The next best engineering step is a packaged npm/global or Docker repro on current main with CPU/event-loop profile data, then a regression that distinguishes the already-covered manifest-cache and openclaw-unknown cases from repeated openclaw-<version>-* stage-root inspection after gateway readiness.
