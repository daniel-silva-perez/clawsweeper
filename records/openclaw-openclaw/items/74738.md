## Summary

Keep open. The linked provider-payload bug still exists on current main, and the PR contains a useful focused transport guard, but it also introduces an unrelated persistent Pi retry-settings change that should not merge as part of this fix.

## What This Changes

The PR wraps OpenAI Completions `tools` and `tool_choice` payload emission with `supportsModelTools(model)`, adds disabled-tool regression tests, and disables Pi retry in prepared embedded Pi settings.

## Best Possible Solution

Keep the OpenAI Completions guard for `compat.supportsTools === false`, fix the regression test to use real assistant `toolCall` and `toolResult` history, and split or redesign the Pi retry-loop work so any retry override is runtime-only and reviewed against issue #73781 separately.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Avoid persisting Pi retry settings during run preparation:** `src/agents/pi-project-settings.ts:57`
  - body: `setRetryEnabled(false)` is not a transient override in the pinned Pi settings manager: it mutates global retry settings and calls `save()`. When `createEmbeddedPiSettingsManager()` returns the file-backed manager for trusted project settings with no bundle settings, preparing a run can permanently write `retry.enabled: false` into the agent settings. Please remove this from the transport fix or make it a non-persistent runtime override.
  - confidence: 0.9
- **[P3] Use real tool-history shapes in the fallback test:** `src/agents/openai-transport-stream.test.ts:2411-2420`
  - body: This fixture uses `type: "tool-call"` and `role: "tool"`, but `hasToolHistory()` only detects assistant `toolCall` blocks or `toolResult` messages. As written, the test would still pass if the fallback branch were not exercised; use the production `toolCall`/`toolResult` message shape so it proves `tools: []` is suppressed.
  - confidence: 0.92
