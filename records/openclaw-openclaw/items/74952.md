## Summary

Keep open. Current main does not implement the requested maintenance-window scheduler feature, and this open replacement PR carries useful work from the closed unmerged predecessor, but the branch is not merge-ready because it regresses existing heartbeat wake contracts and includes an unrelated CI supply-chain-sensitive change.

## What This Changes

This PR adds `cron.maintenance` configuration, a shared maintenance phase resolver, cron and heartbeat role gating with deferred replay/status diagnostics, config hot-reload coverage, UI/protocol/docs/tests, and a CI changed-file secret scan.

## Best Possible Solution

Revise the PR so the maintenance-window feature preserves current cron and heartbeat contracts, schedules deferred cron replay independently of ordinary future wake times, and splits the CI secret-scanning guard into a separately reviewed security/CI change before running the focused scheduler/config tests and changed gate.

## Reproduction Assessment

Yes. A high-confidence reproduction path exists by code inspection: current-main tests assert cron main-session wakes pass `heartbeat: { target: "last" }`, while the PR head drops that override; a deferred maintenance run plus any unrelated future cron wake also shows the `armTimer` replay starvation path.

## Solution Assessment

No. The feature direction is plausible, but this is not the best implementation until it preserves existing heartbeat override and retryable busy-skip contracts, fixes deferred replay arming, and separates the unrelated CI workflow change.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve per-wake heartbeat overrides:** `src/infra/heartbeat-runner.ts:1034`
  - body: Current main passes per-wake heartbeat overrides such as `{ target: "last" }` from cron into targeted heartbeat runs so main-session cron output routes back to the last chat. The new helper always passes `params.agent.heartbeat` to `runOnce`, and the wake handler no longer forwards `params.heartbeat`, so targeted or replayed wakes can lose their delivery target.
  - confidence: 0.9
- **[P2] Keep all retryable busy skips retryable:** `src/infra/heartbeat-runner.ts:1240-1242`
  - body: Current main treats `requests-in-flight`, `cron-in-progress`, and `lanes-busy` as retryable busy skips. The new runner paths only special-case `requests-in-flight`, so `cron-in-progress` or `lanes-busy` results can advance heartbeat schedules or return as non-retryable outcomes instead of letting the wake layer retry.
  - confidence: 0.86
- **[P2] Arm deferred cron replay alongside normal wake times:** `src/cron/service/timer.ts:625-638`
  - body: Deferred maintenance runs are only considered when `nextWakeAtMs(state)` is null. If any unrelated job has a future `nextRunAtMs`, a deferred run that is already allowed waits for that ordinary wake instead of replaying promptly when the maintenance phase allows it.
  - confidence: 0.88
