## Summary

Keep open. The requested maintenance-window feature is not present on current main, but the PR branch is not merge-ready because it regresses current heartbeat wake contracts, can delay deferred cron replay behind unrelated future jobs, and mixes an unrelated CI supply-chain change into the scheduler feature.

## What This Changes

This PR adds `cron.maintenance` config, a shared maintenance resolver, cron and heartbeat role gating with deferred replay/status diagnostics, hot-reload coverage, docs/tests, UI/protocol types, and a CI secret-scanning guard.

## Best Possible Solution

Rebase the scheduler work onto current main, preserve the heartbeat wake override and retryable busy-skip contracts, schedule deferred cron replay independently of ordinary future wake times, and split the CI secret guard into a separate reviewed change before running focused cron/heartbeat/config tests plus the changed gate.

## Reproduction Assessment

Yes. A high-confidence reproduction path exists by code inspection: main-session cron `wakeMode: "now"` depends on the `{ heartbeat: { target: "last" } }` wake override, the PR helper drops it, and a deferred maintenance run with another future cron wake shows `armTimer` bypassing prompt replay when `nextWakeAtMs` is non-null.

## Solution Assessment

No. The feature direction is plausible, but this PR is not the best implementation until it preserves current-main scheduler contracts and separates the unrelated CI workflow change from the cron/heartbeat behavior change.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve per-wake heartbeat overrides:** `src/infra/heartbeat-runner.ts:1034`
  - body: Current main passes per-wake heartbeat overrides such as `{ target: "last" }` from cron into targeted heartbeat runs so main-session cron output routes back to the last chat. This helper always passes `params.agent.heartbeat` to `runOnce`, so targeted or replayed wakes drop those overrides and can lose their delivery target.
  - confidence: 0.9
- **[P2] Keep all retryable busy skips retryable:** `src/infra/heartbeat-runner.ts:1240-1242`
  - body: Current main treats `requests-in-flight`, `cron-in-progress`, and `lanes-busy` as retryable busy skips. The new runner paths only special-case `requests-in-flight`, so `cron-in-progress` or `lanes-busy` can advance schedules or return as non-retryable outcomes instead of letting the wake layer retry.
  - confidence: 0.86
- **[P2] Arm deferred cron replay alongside normal wake times:** `src/cron/service/timer.ts:625-638`
  - body: Deferred maintenance runs are only considered when `nextWakeAtMs(state)` is null. If any unrelated job has a future `nextRunAtMs`, a deferred run that is already allowed waits for that ordinary wake instead of replaying promptly when the maintenance phase allows it.
  - confidence: 0.88
