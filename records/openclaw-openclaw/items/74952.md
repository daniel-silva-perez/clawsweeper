## Summary

Keep open. Current main does not implement `cron.maintenance`, and this replacement PR still needs review work before it is mergeable: the branch regresses current heartbeat wake contracts and cron deferred-replay scheduling, and it mixes an unrelated CI secret-scanning change into the feature branch.

## What This Changes

The PR adds `cron.maintenance` config, a shared maintenance phase resolver, cron and heartbeat role gating with deferred replay/status diagnostics, docs/tests, hot-reload coverage, UI/protocol types, and a CI secret-scanning guard.

## Best Possible Solution

Rebase this work onto current main, preserve the existing heartbeat wake override and retryable busy-skip contracts, schedule deferred cron replay independently of unrelated future wake times, and split or separately review the CI secret-scanning change before running the focused cron/heartbeat/config tests and the changed gate.

## Reproduction Assessment

Yes. A high-confidence reproduction path exists by code inspection: a main-session cron job with `wakeMode: "now"` relies on the current `{ heartbeat: { target: "last" } }` wake override, while the PR head's new helper ignores that override; a deferred maintenance run plus another future cron job also shows `armTimer` bypassing deferred replay because `nextWakeAtMs` is non-null.

## Solution Assessment

No. The feature direction is plausible, but this is not the best implementation until it preserves current-main scheduler contracts and separates the unrelated CI workflow change from the cron/heartbeat behavior change.

Full review comments:

- **[P2] Preserve per-wake heartbeat overrides:** `src/infra/heartbeat-runner.ts:1034`
  - body: Current main passes per-wake heartbeat overrides such as `{ target: "last" }` from cron into targeted heartbeat runs so main-session cron output routes back to the last chat. The new helper always passes `params.agent.heartbeat` to `runOnce`, so deferred or targeted wakes drop those overrides and can lose their delivery target.
  - confidence: 0.9
- **[P2] Keep all retryable busy skips retryable:** `src/infra/heartbeat-runner.ts:1240-1242`
  - body: Current main treats `requests-in-flight`, `cron-in-progress`, and `lanes-busy` as retryable busy skips. The new runner paths only special-case `requests-in-flight`, so `cron-in-progress` or `lanes-busy` results can advance heartbeat schedules or return as non-retryable outcomes instead of letting the wake layer retry.
  - confidence: 0.86
- **[P2] Arm deferred cron replay alongside normal wake times:** `src/cron/service/timer.ts:625-638`
  - body: Deferred maintenance runs are only considered when `nextWakeAtMs(state)` is null. If any unrelated job has a future `nextRunAtMs`, a deferred run that is already allowed will wait for that ordinary wake instead of replaying promptly when the maintenance phase allows it.
  - confidence: 0.88
