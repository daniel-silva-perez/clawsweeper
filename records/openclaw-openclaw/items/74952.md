## Summary

Keep open. Current main does not already implement this maintenance-window feature, but the PR head is not merge-ready because it regresses current cron/heartbeat delivery and retry contracts, can delay deferred cron replay behind unrelated future wakes, and mixes in a CI supply-chain change that needs separate review.

## What This Changes

The PR adds `cron.maintenance` configuration, maintenance phase resolution, cron and heartbeat role gating with deferred replay, status diagnostics, config reload handling, docs, tests, and a CI changed-file secret scan.

## Best Possible Solution

Revise the PR so the maintenance-window scheduler preserves the current heartbeat wake API and retryable busy-skip semantics, schedules deferred cron replay independently of unrelated future wake times, rebases cleanly onto current main, and moves the CI secret-scanning guard to a separately reviewed security/CI change.

## Reproduction Assessment

Yes. A high-confidence reproduction path exists by code inspection and current-main tests: cron main-session jobs require `heartbeat: { target: "last" }`, while the PR runner path drops per-wake overrides; a deferred maintenance run plus any unrelated future cron wake also exercises the replay arming delay.

## Solution Assessment

No. The feature direction is plausible, but this implementation is not the best fix until it preserves existing cron/heartbeat contracts and separates the unrelated CI supply-chain change.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve per-wake heartbeat overrides:** `src/infra/heartbeat-runner.ts:1034`
  - body: The new shared runner path always passes `params.agent.heartbeat` to `runOnce`, and the wake handler no longer forwards per-wake heartbeat overrides. Cron-triggered main-session wakes rely on `heartbeat: { target: "last" }`, so queued or replayed wakes can lose their delivery target and silently drop replies.
  - confidence: 0.9
- **[P2] Keep all retryable busy skips retryable:** `src/infra/heartbeat-runner.ts:1240-1242`
  - body: The new paths only treat `requests-in-flight` as retryable. Current main also retries `cron-in-progress` and `lanes-busy`, so those busy skips can now advance heartbeat schedules or return as non-retryable outcomes instead of letting the wake layer retry.
  - confidence: 0.86
- **[P2] Arm deferred replay alongside normal wake times:** `src/cron/service/timer.ts:625-638`
  - body: Deferred maintenance replay is considered only when `nextWakeAtMs(state)` is null. If any unrelated job has a future wake, an already-allowed deferred run waits for that ordinary wake instead of replaying promptly when the maintenance phase allows it.
  - confidence: 0.88
