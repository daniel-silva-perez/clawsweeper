## Summary

Keep open. Current main still has the Telegram message-action runtime resolve omitted `accountId` tokens through `resolveTelegramToken`, which normalizes the omission to the implicit `default` account and can miss a configured `channels.telegram.defaultAccount`; the lower-level send path and account resolver already have the intended fallback, and the related fix PRs were closed unmerged.

## What This Changes

The issue requests that outgoing Telegram message actions honor the configured default Telegram account without requiring `--account`, with clearer multi-bot warnings, validation, and documentation.

## Best Possible Solution

Keep the issue open and land a narrow Telegram plugin fix in `extensions/telegram/src/action-runtime.ts`: resolve omitted-account token-gated message actions through `resolveTelegramAccount`, pass the resolved token and resolved account ID into the Telegram send/action helpers, and add focused regression coverage for named `defaultAccount` plus explicit-account behavior.

## Reproduction Assessment

Yes. A high-confidence reproduction is a config with `channels.telegram.defaultAccount = "work"`, `channels.telegram.accounts.work.botToken`, no top-level/default token, then `handleTelegramAction` or `openclaw message send --channel telegram` without `accountId`; the current code resolves the token as implicit `default` instead of `work`.

## Solution Assessment

Yes. The narrow maintainable fix is to align action-runtime token resolution with the existing `resolveTelegramAccount` semantics already used by the direct send helper, while preserving explicit-account and unknown-account guard behavior.

Fix the Telegram default-account routing bug for this issue. Current main has `extensions/telegram/src/action-runtime.ts` read `accountId` from action params and call `resolveTelegramToken(cfg, { accountId })`; when `accountId` is omitted, that resolves the implicit `default` token instead of `channels.telegram.defaultAccount`. Update the Telegram action runtime so all token-gated message actions resolve through `resolveTelegramAccount({ cfg, accountId })` when selecting the token, and pass the resolved token plus resolved `accountId` to helper calls including send, poll, react, delete, edit, sticker, createForumTopic, editForumTopic, and the optional pin-after-send path.
