## Summary

Keep open. Current main still has the Telegram message-action path read an omitted `accountId` directly from the action params and call `resolveTelegramToken(cfg, { accountId })`, which normalizes the omitted account to `default` instead of using `channels.telegram.defaultAccount`. The lower-level Telegram send helper already uses the account resolver with default fallback, matching the maintainer-context comment that the remaining bug is specifically in `openclaw message send` / sibling message actions. The related PRs #61140 and #61144 were closed unmerged in the provided GitHub context, so there is no implemented-on-main close yet.

## Best Possible Solution

Fix `extensions/telegram/src/action-runtime.ts` so all token-gated Telegram message actions resolve a `ResolvedTelegramAccount` through `resolveTelegramAccount({ cfg, accountId })` when `accountId` is omitted, then pass the resolved token and resolved account ID into send, poll, react, edit, delete, sticker, and forum-topic actions. Add focused regression tests in `extensions/telegram/src/action-runtime.test.ts` for a named `channels.telegram.defaultAccount` with no top-level/default token, plus explicit-account behavior to preserve multi-bot routing.

## Risks / Open Questions

- Closing now would leave the `openclaw message send --channel telegram` omitted-account path unresolved for named default Telegram accounts without an implicit `default` or top-level token.
