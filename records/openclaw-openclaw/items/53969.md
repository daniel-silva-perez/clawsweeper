## Summary

Keep open. Current main has shipped several important hardening pieces from the request, including atomic JSON writes, Codex CLI bootstrap-only behavior, shared provider/profile refresh serialization, and refresh_token_reused recovery. The remaining central work is still real: OAuth-producing setup paths still persist via the unlocked generic upsert, and duplicate-enrollment protection is still keyed to provider/profile/store coordination rather than a stable OAuth enrollment or copied-store single-writer boundary.

## Best Possible Solution

Keep the issue open and narrow it to the remaining hardening: route OAuth-producing mutations such as provider auth-result persistence and writeOAuthCredentials through updateAuthProfileStoreWithLock or a lock-safe OAuth-specific upsert; add feasible duplicate-enrollment detection or warnings using stable identity metadata such as accountId/email where available; and document the single-writer boundary for copied auth-profiles.json files, multiple hosts/stores, sibling sync, and Codex CLI coexistence. Preserve the shipped atomic writes, Codex bootstrap-only behavior, same-profile refresh serialization, stable Codex profile IDs, and refresh_token_reused recovery.

## Risks / Open Questions

- Closing now would drop tracking for OAuth-producing setup paths that can still write auth-profiles.json without the repo's lock-safe updater.
- Duplicate OpenAI Codex refresh-token chains under different profile IDs, store roots, state directories, or hosts can still race because the current refresh lock key is provider plus profileId, not a stable OAuth enrollment identity.
- The docs now explain token-sink and Codex CLI bootstrap semantics, but copied-store and multi-host single-writer guidance remains incomplete for the deployment pattern described by the reporter.
