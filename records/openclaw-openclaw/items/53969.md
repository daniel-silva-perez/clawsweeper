## Summary

Keep open. Current main implements several important parts of the request: atomic auth-store writes, Codex CLI bootstrap-only behavior, same-provider/profile refresh serialization, refresh_token_reused recovery, and stable OpenAI Codex profile IDs. The central hardening is still incomplete because OAuth-producing writes still use the unlocked upsert path, and duplicate-enrollment protection is still keyed to provider/profile/store coordination rather than a stable OAuth enrollment or copied-store single-writer boundary.

## Best Possible Solution

Keep the issue open and narrow it to the remaining hardening: route OAuth-producing writes such as provider auth-result persistence and writeOAuthCredentials through updateAuthProfileStoreWithLock or a lock-safe OAuth-specific upsert; add feasible duplicate-enrollment detection or warnings within one store using accountId/email/stable identity metadata; and document the single-writer boundary for copied auth-profiles.json files, multiple hosts/stores, sibling sync, and Codex CLI coexistence. Preserve the shipped atomic writes, Codex bootstrap-only behavior, same-profile refresh serialization, stable Codex profile IDs, and refresh_token_reused recovery.

## Risks / Open Questions

- Closing now would drop tracking for OAuth-producing mutation paths that can still write auth-profiles.json without the repo's lock-safe updater.
- Duplicate OpenAI Codex refresh-token chains under different profile IDs, store paths, or hosts can still race because the current refresh lock key is provider plus profileId, not a stable OAuth enrollment identity.
- Docs now explain Codex CLI bootstrap behavior, but copied-store and multi-host single-writer guidance is still not explicit enough for the deployment pattern described by the reporter.
