## Summary

Keep this PR open. Current main still lacks the PR's central reconnect-grace behavior: once startup connect grace has elapsed, a running channel with connected: false is still evaluated as unhealthy immediately and the health monitor can restart it. Related Discord/gateway hardening on main adds cooldown/hourly caps and lifecycle cleanup, but it does not supersede the delayed-intervention behavior proposed here.

## Best Possible Solution

Keep this PR open as a still-useful implementation candidate. The best path is to rebase it onto current main, add a generic reconnectGraceMs window to ChannelHealthPolicy and ChannelHealthTimingPolicy, consume the already-typed lastDisconnect.at data with a clear fallback policy, update both the monitor and readiness call sites, preserve current stale-socket and restart-cap behavior, and add focused policy, monitor, and readiness tests for recent versus expired disconnects.
