## Summary

Keep this PR open. Current main still does not implement the PR’s central reconnect-grace behavior: after startup connect grace, a running channel with `connected: false` is still evaluated as unhealthy immediately and the health monitor can restart it. Main has related Discord/gateway hardening, typed `lastDisconnect` data, stale-socket improvements, restart cooldowns, and hourly caps, but those do not replace the delayed-intervention behavior proposed here.

## Best Possible Solution

Keep this PR open as a still-useful implementation candidate. The best path is to rebase it onto current main, add a generic reconnect-grace window to `ChannelHealthPolicy` and `ChannelHealthTimingPolicy`, consume the already-typed `lastDisconnect.at` data with a clear fallback policy, update both the monitor and readiness call sites, preserve current stale-socket and restart-cap behavior, and add focused policy, monitor, and readiness tests for recent versus expired disconnects. The open drain-window PR is related but solves a different restart-abort problem, so it should not supersede this reconnect-grace work by itself.

## Risks / Open Questions

- Closing this as implemented would drop the specific mitigation for health-monitor teardown during an in-progress Discord reconnect.
- A rebase needs to preserve current stale-socket transport-activity behavior, restart cooldown/hour caps, and readiness semantics while adding reconnect grace.
- The grace window must not suppress recovery for genuinely stopped, gave-up, startup-not-ready, or long-disconnected channels.
