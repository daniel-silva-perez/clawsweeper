## Summary

Keep open. Current main still has no manifest-level install-time `configPatch`: the manifest type/loader do not expose it, install results cannot carry it, install persistence only mutates plugin install state, CLI install paths pass only identity/source metadata, and docs still direct users to configure plugins manually after install. The request is valid but needs maintainer design because manifest-driven config writes are a trust-sensitive core plugin API with unresolved merge and array semantics.

## Best Possible Solution

Keep this issue open for maintainer design of a typed core plugin-install/API contract. The best path is to add a manifest-level config intent or equivalent install-time plugin config API, sanitize and validate patches before writing, define merge/conflict/array semantics, show an auditable diff and skip path, document uninstall/manual-revert guidance, and add regression coverage across local path, linked path, archive, npm, ClawHub, marketplace, bundled, and chat-native install flows while preserving unrelated config.
