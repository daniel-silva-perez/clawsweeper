## Summary

Keep open. Current main still lacks a manifest-level install-time config patch contract: plugin manifests do not expose `configPatch`, install results cannot carry it, install persistence only updates plugin enablement/install records/slot state, and public docs still describe manual plugin configuration after install. The prior provider-auth `configPatch` path is real prior art, but it remains scoped to provider setup/auth flows, while the linked implementation PR is closed unmerged and the third-party config-mutation contract still needs maintainer design.

## What This Changes

The issue requests a `configPatch` field in `openclaw.plugin.json` that `openclaw plugins install` would validate, deep-merge into user config, and report during plugin installation.

## Best Possible Solution

Define a documented, auditable install-time plugin config intent contract that works across service, tool, provider, hook, and channel setup without accepting arbitrary silent config writes. It should validate before commit, show the exact diff with a skip path, define conflict and array behavior, record provenance for uninstall guidance, and cover every install route with regression tests and docs.
