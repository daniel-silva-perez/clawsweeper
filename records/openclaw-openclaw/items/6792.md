## Summary

Keep open. Current main still lacks a manifest-level install-time config patch contract: manifests do not expose `configPatch`, install results cannot carry it, install persistence only updates plugin enablement/install records/slot state, and docs still tell users to configure plugins manually after install. The issue discussion shows real service/tool use cases and valid provider-auth prior art, but the linked implementation PR is closed unmerged and the remaining work needs product/security design for third-party config mutation.

## What This Changes

Request to let `openclaw.plugin.json` declare an install-time `configPatch` that `openclaw plugins install` applies and validates so plugins can set required config without manual edits.

## Best Possible Solution

Define a documented, auditable install-time plugin config intent contract that works across service, tool, provider, hook, and channel setup. It should validate before commit, show an exact diff with a skip path, define conflict and array behavior, record provenance for uninstall guidance, and cover CLI/chat/ClawHub/npm/local install routes with tests and docs.

## Reproduction Assessment

Not applicable as a feature request, not a bug reproduction. There is a high-confidence current-main verification path: inspecting the manifest type/loader, install result, persistence path, callers, and docs shows that manifest-level `configPatch` is still absent.

## Solution Assessment

No: the raw automatic deep-merge proposal is not yet the best maintainable solution without guardrails. A constrained, documented config-intent contract with validation, diff display, skip behavior, allowed-path/trust rules, and provenance is safer.

## Risks / Open Questions

- Manifest-level patches would let third-party install metadata persistently mutate tool, provider, hook, channel, or model routing, so trust rules, confirmation behavior, and allowed-path policy need maintainer design first.
- Merge semantics remain product-sensitive, especially arrays like `tools.alsoAllow` where replace versus union changes existing plugin permissions.
- Uninstall provenance and rollback guidance are not defined, so users could be left without an auditable way to understand or undo install-time config changes.
