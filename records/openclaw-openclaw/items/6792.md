## Summary

Current `main` still does not implement manifest-level install-time `configPatch`. The existing `configPatch` surface remains provider-auth scoped, while the plugin manifest type, manifest loader, install result, CLI install command, and install persistence path do not carry or apply a manifest patch. The discussion includes concrete plugin ergonomics from `claude-search-proxy` and a custom-tool allowlist use case, but PR #43421 is closed and unmerged, and the latest maintainer/Codex comment correctly keeps this open as a trust-sensitive plugin API design issue.

## Best Possible Solution

Keep this issue open for maintainer design of a typed, auditable install-time plugin config intent API. A good solution should define the manifest contract, sanitize allowed patch surfaces, validate before write, specify conflict and array semantics, show an auditable diff and skip path, document uninstall/manual-revert behavior, and add regression coverage across npm, local/link, archive, bundled, ClawHub, and chat-native install flows without clobbering unrelated config.
