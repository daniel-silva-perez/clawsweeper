## Summary

Current main still does not implement manifest-level install-time `configPatch`. The only current `configPatch` surface is provider-auth result plumbing, while the plugin manifest type, manifest loader, install result, and install persistence path do not accept or apply a manifest patch. The linked implementation PR #43421 is closed and unmerged, and the latest maintainer/Codex comment explicitly keeps this open because manifest-driven config writes are a trust-sensitive public plugin API design question.

## Best Possible Solution

Keep this issue open for maintainer design of a typed, auditable install-time plugin config intent API. A good solution should define the manifest contract, sanitize allowed patch surfaces, validate before write, specify conflict and array semantics, show an auditable diff and skip path, document uninstall/manual-revert behavior, and add regression coverage across npm, local/link, archive, bundled, and chat-native install flows without clobbering unrelated config.

## Risks / Open Questions

- Manifest-driven config writes are trust-sensitive: a third-party plugin could alter model, provider, tool, hook, channel, service, or secret-adjacent config during install.
- Merge semantics remain unresolved, especially arrays such as `tools.alsoAllow` where replacement could clobber entries from other plugins but generic union behavior may be field-specific.

- The existing attempted implementation PR #43421 is closed and unmerged, so there is no canonical accepted implementation to close this against.
