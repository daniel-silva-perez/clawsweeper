## Summary

Keep this open. Current main has several adjacent mitigations, but none proves the reported gateway-wide freeze is fixed: Telegram polling recovery, health monitoring, diagnostics heartbeat, and nested session lanes all depend on timers or per-session state, while the report says all Telegram pollers and timer-driven logs go silent together. The issue is recent, has another affected-user comment, and the related stuck-session work is narrower than this all-account polling freeze.

## What This Changes

The issue reports that nested `sessions_send` subagent activity can leave the gateway process alive while all configured Telegram polling and health-monitor logging stop until the gateway is restarted.

## Best Possible Solution

Reproduce the scenario on current main with multiple Telegram bot accounts and a coordinator-to-subagent `sessions_send`, capture a diagnostics/stability export plus logs from the affected run, then land a focused liveness or recovery change that proves Telegram polling and diagnostic timers continue after nested A2A completion or escalates to a bounded restart when they do not.
