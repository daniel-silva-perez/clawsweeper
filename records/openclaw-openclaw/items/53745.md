## Summary

Keep this open. Current main has adjacent Telegram polling watchdogs, per-session nested lanes, health diagnostics, and stuck-session recovery, but I did not find proof for the reported combined failure mode: nested `sessions_send` activity followed by gateway-wide silence across all Telegram pollers and timer-driven health output. The recent affected-user comment and the distinct related closed issue make conservative cleanup unsafe.

## What This Changes

The issue reports that coordinator-to-subagent `sessions_send` activity can leave the gateway process alive but stop all Telegram polling, health-monitor output, and outbound sends until a manual restart.

## Best Possible Solution

Reproduce the scenario on current main with multiple Telegram bot accounts and coordinator-to-subagent `sessions_send`, collect a `openclaw gateway diagnostics export` bundle from an affected run, then land a targeted fix with a regression proving Telegram polling and health-monitor/diagnostic timers continue after nested A2A completion or that the gateway escalates to a bounded restart when they do not.
