## Summary

Keep open. Current main intentionally implements the Feishu broadcast-mention behavior the reporter disputes, and the docs/tests encode it, so this is a live channel policy question rather than obsolete cleanup. A direct revert could reintroduce the closed Feishu/Lark @all false-positive from #37706, so maintainer judgment is needed before changing behavior.

## What This Changes

The issue asks OpenClaw to preserve Feishu @all/@_all events for agent context and move any suppression decision from hard-coded channel mention filtering to prompt or configuration policy.

## Best Possible Solution

Keep this open until Feishu channel maintainers decide the intended product policy. If the policy changes, represent broadcast mentions as a distinct Feishu channel fact that can be passed into agent context, while keeping an explicit and tested rule for whether broadcast-only messages satisfy mention gating or require a config/prompt opt-in.

## Reproduction Assessment

Yes. A focused fixture with only @_all text, @all mention metadata, or post content user_id all produces mentionedBot=false in current tests; with group requireMention=true, the handler returns before agent dispatch.

## Solution Assessment

No. The requested direct revert is not clearly the safest fix because it can reintroduce #37706; a narrower solution would separate broadcast context from the boolean bot-mention gate and add explicit policy/config coverage.
