## Summary

Keep open. Current main intentionally filters Feishu broadcast-only `@all`/`@_all` out of bot-mention detection and forwarded mention targets, and the broader open #74817 review now treats this issue as the canonical place for the remaining `@all` semantics decision.

## What This Changes

The issue requests changing Feishu broadcast mention handling so `@all`/`@_all` can reach agent context and be governed by prompt or config rather than being hard-filtered as non-bot mentions.

## Best Possible Solution

Represent Feishu broadcast mentions as an explicit channel fact that can be included in inbound context without automatically satisfying the bot-mention gate, then let maintainers choose prompt/config semantics and update Feishu tests and docs in the same change.

## Reproduction Assessment

Yes. A focused fixture with a group message containing only `@_all` text or mention metadata `{ key: "@_all", id.open_id: "all" }` produces `mentionedBot=false`; with `requireMention=true`, the group handler returns before dispatch.

## Solution Assessment

Unclear. The requested revert is not clearly the best fix because it can restore the broadcast false-positive that PR #72658 fixed; preserving broadcast context separately from the boolean bot-mention gate is safer.
