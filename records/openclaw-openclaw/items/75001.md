## Summary

Keep open. Current main intentionally implements the behavior the reporter disputes, but this is a live Feishu channel policy question rather than obsolete cleanup: whether broadcast mentions should be preserved for agent context without satisfying the bot-mention gate. Related #74817 overlaps on the @everyone delivery symptom, but it also combines a separate multi-bot echo-loop request and different suspected API behavior, so this focused post-#72658 policy question is not a high-confidence duplicate close.

## What This Changes

The issue asks OpenClaw to stop hard-filtering Feishu broadcast `@all`/`@_all` mentions at the channel layer and instead handle suppression through prompt or configuration policy.

## Best Possible Solution

Represent Feishu broadcast mentions as a distinct channel fact that can be included in agent context when a message is dispatched, while keeping a separate, explicit policy for whether broadcast-only messages satisfy mention gating or are suppressed by prompt/configuration.

## Reproduction Assessment

Yes. A focused fixture with a group message containing only `@_all` text, mention metadata `{ key: "@_all", id.open_id: "all" }`, or post `user_id: "all"` produces `mentionedBot=false`; with `requireMention=true`, the group handler returns before dispatch.

## Solution Assessment

Unclear. The requested revert is not obviously the safest fix because it can restore the #37706 false-positive; a narrower maintainable solution would separate broadcast context visibility from the boolean bot-mention gate.
