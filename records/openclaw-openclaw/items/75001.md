## Summary

Keep open. Current main intentionally implements the Feishu broadcast-mention behavior the reporter disputes, and the behavior is also documented and covered by tests, so this is an active channel policy question rather than stale cleanup or an already-fixed report. A direct revert is risky because the related closed work fixed the opposite noisy-reply behavior where Feishu/Lark `@all` alone counted as a bot mention.

## What This Changes

The issue requests that Feishu broadcast `@all`/`@_all` mentions be preserved for agent context and handled by prompt/config policy instead of being filtered from the Feishu channel mention gate.

## Best Possible Solution

Keep the issue open until Feishu channel maintainers choose the intended product policy. If the policy changes, represent broadcast mentions as a distinct Feishu channel fact that can be passed into agent context, while keeping an explicit, tested rule for whether broadcast-only messages satisfy mention gating or require config/prompt opt-in.

## Reproduction Assessment

Yes. Current test fixtures with broadcast-only `@_all` text, `@all` mention metadata, and post-content `user_id: all` all produce `mentionedBot=false`; with group `requireMention=true`, the handler returns before agent dispatch.

## Solution Assessment

No, the requested direct revert is not clearly the safest fix because it can reintroduce #37706. A narrower maintainable solution would separate broadcast mention context from the boolean bot-mention gate and add explicit policy/config coverage if maintainers want broadcast-only messages to reach agents.
