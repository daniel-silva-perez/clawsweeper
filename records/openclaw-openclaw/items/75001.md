## Summary

Keep open. Current main still implements the Feishu broadcast-only `@all`/`@_all` suppression that the report disputes, and the behavior is covered by source, tests, docs, changelog, and the shipped `v2026.4.27` release; this is a maintainer policy question rather than a closeable obsolete report.

## What This Changes

The issue asks OpenClaw to stop dropping Feishu broadcast-only `@all`/`@_all` mentions at the channel mention-gate layer and instead expose them to prompt/config-level response policy.

## Best Possible Solution

Represent Feishu broadcast mentions as a distinct inbound fact that can be included in agent context without automatically satisfying the bot-mention gate, then document and test the chosen prompt/config semantics for whether agents should respond.

## Reproduction Assessment

Yes. A focused Feishu fixture containing only `@_all` text or mention metadata with `id.open_id: "all"` produces `mentionedBot=false`; with `requireMention` enabled, the group handler returns before dispatch.

## Solution Assessment

Unclear. The requested revert is not obviously the narrowest maintainable fix because it may reopen the false-positive fixed by #72658; a safer design is to preserve broadcast context separately from the boolean bot-mention gate or add explicit config semantics.
