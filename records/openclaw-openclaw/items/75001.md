## Summary

Keep open. Current main still deliberately treats Feishu broadcast-only `@all`/`@_all` as not mentioning the bot, filters those mentions out of forwarded mention targets, and returns before dispatch when group `requireMention` is true. The report asks maintainers to reconsider that recent channel policy rather than reporting behavior that is already implemented or cleanly superseded by the broader open Feishu issue.

## What This Changes

The issue asks OpenClaw to stop silently filtering Feishu broadcast `@all`/`@_all` mentions at the channel API layer and instead expose them to the agent or gate them via prompt/config.

## Best Possible Solution

Represent Feishu broadcast mentions as a distinct channel fact that can be included in agent context without automatically satisfying the bot-mention gate, then let maintainers choose prompt/config semantics and update Feishu tests and docs in the same change.

## Reproduction Assessment

Yes. A focused fixture with a group message containing only `@_all` text or mention metadata `{ key: "@_all", id.open_id: "all" }` produces `mentionedBot=false`; with `requireMention=true`, the group handler returns before dispatch.

## Solution Assessment

Unclear. The requested revert is not obviously the best fix because it can restore the broadcast false-positive that PR #72658 addressed; a safer alternative is to preserve broadcast context separately from the boolean bot-mention gate.
