## Summary

Keep open. Current main still enforces the Feishu broadcast mention behavior being challenged, and this is a recent channel-policy dispute rather than an implemented, stale, or cleanly superseded item. The related open Feishu issue is broader and overlaps, but it does not replace this narrower post-PR semantics question.

## What This Changes

The issue asks OpenClaw to stop dropping Feishu broadcast-only `@all`/`@_all` mentions at the channel gate and instead expose or control that behavior through prompt/config policy.

## Best Possible Solution

Represent Feishu broadcast mentions as a distinct channel fact that can be included in agent context without automatically satisfying the direct bot-mention gate, then let maintainers choose prompt/config semantics and update Feishu tests and docs in the same change.

## Reproduction Assessment

Yes. A focused fixture with a group message containing only `@_all` text or mention metadata `{ key: "@_all", id.open_id: "all" }` produces `mentionedBot=false`; with `requireMention=true`, the group handler returns before dispatch.

## Solution Assessment

Unclear. The requested revert is not obviously the best fix because it can restore the earlier broadcast false-positive; a safer path is to preserve broadcast context separately from the boolean bot-mention gate.
