## Summary

Keep open. Current main still has the reported disabled-thinking gap: per-model `params.thinking: false`/`disabled`/`none` is not normalized to `off`, the DeepSeek V4 wrapper can therefore build an enabled-thinking payload, and `extra_body.thinking.type=disabled` is applied afterward without a final DeepSeek cleanup. The issue is also paired with open implementation PRs #74403 and #74418, so it should remain open until maintainer review lands or rejects those fixes.

## What This Changes

The issue asks OpenClaw to prevent DeepSeek v4-pro disabled-thinking sessions from sending inconsistent `reasoning_content` history after `extra_body.thinking.type=disabled` is configured.

## Best Possible Solution

Land one narrow fix, likely by reviewing #74403 and #74418 together, that computes disabled DeepSeek thinking consistently before provider wrapping and also prevents a final `extra_body.thinking.type=disabled` payload from containing conflicting enabled-thinking fields or replayed `reasoning_content`. Add a regression test for the exact reporter config.

## Reproduction Assessment

Yes. The high-confidence source path is `params.thinking: false` falling through to the DeepSeek V4 default `high`, followed by the DeepSeek wrapper inserting enabled-thinking replay fields and the later `extra_body` merge changing only `thinking` to disabled.

## Solution Assessment

Unclear whether either open PR is the best final solution by itself. The maintainable fix is a single normalization path for `false`/`disabled`/`none` plus a final payload guard if `extra_body` can still override provider-owned thinking fields.
