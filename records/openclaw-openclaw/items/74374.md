## Summary

Keep open. Current main still has the reported disabled-thinking composition gap: per-model `params.thinking: false` or `"disabled"` is not normalized to `off`, the DeepSeek V4 wrapper can still build an enabled-thinking payload, and the later `extra_body.thinking.type=disabled` merge does not perform a final DeepSeek cleanup. The issue is also paired with open implementation PRs #74403 and #74418, so it should remain open until maintainers land or reject those fixes.

## What This Changes

The issue asks OpenClaw to avoid sending an inconsistent DeepSeek V4 payload when thinking is configured disabled but prior assistant history contains `reasoning_content`.

## Best Possible Solution

Land one narrow, test-backed fix after reviewing #74403 and #74418 together: normalize boolean and alias disabled thinking to `off` before provider wrapping, and add a final payload guard for `extra_body.thinking.type=disabled` so conflicting DeepSeek replay fields cannot survive the post-wrapper merge. The regression test should use the reporter’s exact DeepSeek V4 config shape and preserve separate tracking for the broader #75040 collision report.

## Reproduction Assessment

Yes. The high-confidence reproduction path is the current source pipeline for the reporter config: `params.thinking:false` falls through to DeepSeek V4 default `high`, the provider wrapper adds enabled thinking and replay `reasoning_content`, and the post-wrapper `extra_body` merge changes only `thinking` to disabled.

## Solution Assessment

No, closing this now is not the best solution because current main still has the payload collision path. The maintainable solution is a single reviewed fix combining thinking-level normalization with a final DeepSeek payload consistency guard, rather than treating either open PR as landed before review.
