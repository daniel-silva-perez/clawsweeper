## Summary

Keep open. Current main contains an adjacent DeepSeek V4 replay fix, but the reporter's disabled-thinking configuration is still not resolved consistently before the final OpenAI-compatible payload is sent. Two open PRs reference this issue with closing syntax, so the issue should remain open for maintainer review of those fixes.

## What This Changes

The issue reports a DeepSeek v4-pro regression where `thinking` disabled through `extra_body` can still leave `reasoning_content` in replayed assistant history and trigger a 400 on the next turn.

## Best Possible Solution

Land one narrow fix that computes an effective disabled-thinking decision before the final DeepSeek V4 payload is sent, covering `off`, accepted disabled aliases, boolean `false`, and `extra_body.thinking.type=disabled`; add a regression test proving the final payload cannot contain conflicting enabled-thinking fields or replayed `reasoning_content` in the disabled path.

## Reproduction Assessment

Yes. The source-level path is high confidence: per-model `params.thinking: false` is ignored, the DeepSeek wrapper can build an enabled-thinking payload, and `extra_body` can later overwrite `thinking` to disabled without replay cleanup.

## Solution Assessment

Unclear which open PR is the best final shape alone. The safest maintainable solution is a single consistent normalization path, plus a final payload guard if `extra_body` can still override provider-owned thinking fields.
