## Summary

Keep open. Current main does not implement the requested gateway event, and the PR is useful, but the branch should not merge as-is because the emitted diff is built from post-write source-vs-runtime snapshots rather than previous-vs-next config state, nested sensitive values can still enter the broadcast payload, and the new protocol schema is not wired into the exported schema surface.

## What This Changes

The PR adds a `config.changed` gateway event, a config diff helper and payload schema, admin-scoped broadcast wiring, subscription cleanup, and feature-list registration for config hot-reload notifications.

## Best Possible Solution

Keep the PR open and revise it so the event payload is built from a true previous-to-next config comparison, arbitrary nested values are deep-redacted before broadcast, the new event contract is exported through the protocol schema/docs surface, and focused tests cover normal changes, no-op writes, redaction, and client-visible schema registration.

## Reproduction Assessment

Yes. Source inspection provides a high-confidence reproduction path: a normal config write whose source and runtime snapshots match after refresh will make `diffConfigs(evt.sourceConfig, evt.runtimeConfig)` return an empty array, so the advertised `config.changed` event is not emitted for the actual model-name change.

## Solution Assessment

No. The existing listener/broadcast seam is a reasonable place to emit the event, but the implementation needs a previous snapshot or an extended write notification contract, plus deep redaction and protocol wiring, before it is the narrow maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Build the diff from the previous config:** `src/gateway/server-runtime-subscriptions.ts:101`
  - body: `sourceConfig` and `runtimeConfig` on the write notification are both post-write snapshots. Comparing them means a normal edit where source and runtime match emits no `config.changed`, while source/runtime materialization differences can produce unrelated entries. Capture the previous compare snapshot or extend the notification to carry before/after state.
  - confidence: 0.93
- **[P1] Deep-redact values before broadcasting them:** `src/gateway/config-diff.ts:68`
  - body: `redactIfSensitive` only checks the immediate key before recursion. If a nonsensitive parent changes type, or the depth cap emits an object as a single value, nested fields like `apiKey` or `token` are pushed into `before`/`after` unredacted and broadcast to connected admin clients.
  - confidence: 0.9
- **[P2] Export the new protocol schema:** `src/gateway/protocol/schema/config-diff.ts:1`
  - body: The new `ConfigChangedEventPayloadSchema` module is never exported from `schema.ts` or registered in `ProtocolSchemas`, so generated clients and schema consumers will not see the new event payload contract even though the gateway advertises the event name.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete sensitive-data exposure risk in the new admin broadcast payload.

Concerns:

- **[medium] Nested secrets can bypass redaction:** `src/gateway/config-diff.ts:68`
  - body: When a nonsensitive parent object containing secret-like nested keys changes type, or when the depth cap emits a whole object, `before`/`after` can contain unredacted nested credentials. That payload is sent over a new gateway event, so the redaction contract needs to be deep and value-wide before merge.
  - confidence: 0.9
