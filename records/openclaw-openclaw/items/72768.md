## Summary

Keep open. Current main does not implement `config.changed`, and the PR still has blocking correctness, security-redaction, and protocol-contract issues before it can merge.

## What This Changes

The PR adds a `config.changed` gateway event, structured config diff helper, admin-scoped broadcast wiring, subscription cleanup, feature-list registration, and a new event payload schema for config hot-reload notifications.

## Best Possible Solution

Revise the PR so the event is built from a true previous-to-next config comparison, the payload redacts arbitrary nested values before broadcast, and the protocol schema/docs/test surfaces expose and validate the new event contract.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: a config write whose refreshed source and runtime snapshots are equal will make `diffConfigs(evt.sourceConfig, evt.runtimeConfig)` return no changes, so the advertised event is not emitted for the actual edit.

## Solution Assessment

No. The listener/broadcast seam is reasonable, but this implementation needs previous-state data, deep value-wide redaction, and protocol export wiring before it is the narrow maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Build the diff from the previous config:** `src/gateway/server-runtime-subscriptions.ts:101`
  - body: `sourceConfig` and `runtimeConfig` are both post-write snapshots, so a normal edit where runtime matches source produces an empty diff and no `config.changed` event. Capture the previous compare snapshot or extend the notification contract to carry before/after state.
  - confidence: 0.93
- **[P1] Deep-redact values before broadcasting them:** `src/gateway/config-diff.ts:68-71`
  - body: `redactIfSensitive` only checks the immediate key before recursion. If a nonsensitive parent changes type, or the depth cap emits an object as one value, nested fields like `apiKey` or `token` can be sent unredacted in `before`/`after`.
  - confidence: 0.9
- **[P2] Export the new protocol schema:** `src/gateway/protocol/schema/config-diff.ts:1`
  - body: The new payload schema is added as a standalone module but is not exported from the schema barrel or registered in `ProtocolSchemas`, leaving schema consumers and generated clients unable to discover the new event contract.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete sensitive-data exposure risk in the new admin-scoped config-change broadcast payload.

Concerns:

- **[medium] Nested secrets can bypass redaction:** `src/gateway/config-diff.ts:68`
  - body: When a nonsensitive parent object containing secret-like nested keys changes type, or when the depth cap emits a whole object, `before`/`after` can contain unredacted nested credentials and broadcast them to admin clients.
  - confidence: 0.9
