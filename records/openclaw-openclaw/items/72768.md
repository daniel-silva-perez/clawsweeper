## Summary

Keep open. Current main does not implement `config.changed`, and the PR still has blocking correctness and security issues: it diffs two post-write snapshots instead of previous versus next config, can leak nested sensitive values in the new broadcast payload, and leaves the new protocol schema outside the exported schema registry.

## What This Changes

The PR adds a config diff helper, a `config.changed` gateway event payload schema, admin-scoped broadcast registration, a runtime config write listener, and shutdown cleanup for that listener.

## Best Possible Solution

Revise the PR so the event is emitted from a true previous-to-next config comparison in the reload/write pipeline, every emitted value is deep-redacted before broadcast, the event contract is exported through the protocol schema registry, and focused tests cover changed, unchanged, redacted, and client-visible schema behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a config write that changes a model name but leaves source and runtime snapshots equivalent after refresh makes `diffConfigs(evt.sourceConfig, evt.runtimeConfig)` return no user-visible model-name diff, so the advertised event is missing or wrong.

## Solution Assessment

No. The existing listener and broadcast seams are close, but the PR needs a previous snapshot or extended notification contract plus deep redaction and protocol export wiring before it is the narrow maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Build the diff from the previous config:** `src/gateway/server-runtime-subscriptions.ts:101`
  - body: `sourceConfig` and `runtimeConfig` on the write notification are both post-write snapshots. Comparing them means normal edits where source and runtime match emit no real `config.changed` diff, while source/runtime projection differences can produce unrelated entries; build from previous-to-next state instead.
  - confidence: 0.93
- **[P1] Deep-redact mixed-type diff values:** `src/gateway/config-diff.ts:71`
  - body: When a nonsensitive parent changes between an object and another type, the object reaches this branch unchanged and is pushed into `before`/`after` with nested keys like `apiKey` or `token` intact. Redact arbitrary nested values before adding them to the broadcast payload.
  - confidence: 0.9
- **[P2] Export the new protocol schema:** `src/gateway/protocol/schema/config-diff.ts:1`
  - body: The new schema module is not re-exported from `schema.ts` or registered in `ProtocolSchemas`, so protocol generators and schema consumers will not discover the `config.changed` payload contract even though the gateway advertises the event.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete sensitive-data exposure risk in the new admin-scoped config diff broadcast.

Concerns:

- **[medium] Nested secrets can bypass redaction:** `src/gateway/config-diff.ts:71`
  - body: The redaction helper only checks the immediate key before recursion; mixed-type changes or depth-capped entries can serialize a whole object containing nested credentials into `before` or `after` and send it over `config.changed`.
  - confidence: 0.9
