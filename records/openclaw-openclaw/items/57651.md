## Summary

Keep this PR open for maintainer follow-up. Current main still uses an in-memory ACP bridge session store, while this PR is not merge-ready because its primary persistence path is broken and it also mixes unrelated compaction, shared-context, and gateway run-abort behavior with security-sensitive review findings.

## What This Changes

This PR adds a file-backed ACP bridge session store backed by `~/.openclaw/acp-sessions.json` and also adds enhanced/parallel compaction, shared context, config, and gateway active-run auto-abort changes.

## Best Possible Solution

Split out a focused ACP bridge persistence change, probably in `src/acp/session.ts` and `src/acp/translator.ts`, with a typed persisted-session schema, corrupt-file handling, restart/loadSession tests, and docs if behavior changes. Keep enhanced compaction, shared context, and gateway active-run replacement behavior in separate reviewed PRs.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. Static diff inspection is enough for the PR blockers: `_getAllSessions` uses a type-only `Record` expression and the current main bridge store can be verified from source as in-memory only.

## Solution Assessment

Is this the best way to solve the issue? No. The requested ACP bridge persistence is plausible, but this PR combines multiple unrelated behavior changes and introduces correctness and security-sensitive blockers; a narrow persistence PR with validation and restart tests is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Return real session data from the store:** `src/acp/session.ts:95`
  - body: `Record<string, AcpSession>` is a TypeScript type, not a runtime value, and `_getAllSessions` is not declared on `AcpSessionStore`. This means the patch either fails typechecking or cannot retrieve sessions for `saveToDisk`, so the new default file-backed store does not actually persist sessions.
  - confidence: 0.96
- **[P1] Keep enhanced compaction config local to the run:** `src/agents/pi-extensions/compaction-safeguard.ts:288-290`
  - body: Calling `configureEnhancedCompaction()` inside the per-compaction callback rewrites module-global config shared by every session in the process. Concurrent sessions can race and run with another session's settings; pass the config into `enhancedCompaction()` instead.
  - confidence: 0.88
- **[P2] Validate persisted ACP sessions before loading:** `src/acp/session.ts:119-130`
  - body: The file store reads arbitrary JSON and then uses `sessionData.sessionKey` and `sessionData.cwd` without checking the record shape. Corrupt or unexpected state should be skipped explicitly instead of creating sessions with invalid routing data.
  - confidence: 0.82
- **[P2] Use explicit relationships for shared-context access:** `src/agents/shared-context.ts:257-260`
  - body: The `parent-child` access check grants access when one agent ID is a string prefix of another. Human-readable or attacker-influenced IDs such as `main` and `main-attacker` could unintentionally share context; track explicit parent-child links instead.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff adds local session deserialization and cross-agent shared-context access, and both need hardening before merge.

Concerns:

- **[medium] Validate persisted session JSON:** `src/acp/session.ts:119`
  - body: The file-backed store trusts parsed JSON fields before recreating ACP bridge sessions. Invalid or manipulated local state could rebind a bridge session to unexpected session keys or cwd values instead of being rejected cleanly.
  - confidence: 0.82
- **[medium] Avoid prefix-based context sharing:** `src/agents/shared-context.ts:257`
  - body: The new shared-context manager treats string prefixes as a parent-child relationship, which can over-share context between unrelated agents when IDs are human-readable or externally influenced.
  - confidence: 0.84
