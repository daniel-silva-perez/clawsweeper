## Summary

Keep this PR open for maintainer triage. Current main has durable ACP harness/session metadata in the ACP control plane, but it does not implement the file-backed stdio ACP bridge session store proposed here. The PR also mixes unrelated enhanced-compaction, shared-context, and gateway auto-abort changes, and the existing review comments identify functional and security-sensitive blockers, so it should not be treated as obsolete or merge-ready.

## Best Possible Solution

Keep this PR open only as a candidate for maintainer follow-up. The best path is to split a focused ACP bridge persistence change in `src/acp/session.ts`/`src/acp/translator.ts`, add restart/loadSession tests for the bridge mapping, validate persisted JSON before use, and leave enhanced compaction, shared context, and gateway active-run behavior for separate reviewed PRs. Do not merge the current combined PR until the P1 review findings and access-boundary concerns are fixed.

## Risks / Open Questions

- The PR is not merge-ready as submitted: the provided review context says the primary file-backed persistence path is non-functional because `_getAllSessions` returns a TypeScript type expression instead of actual session data.
- The PR mixes several unrelated behavior changes: ACP bridge persistence, enhanced/parallel compaction, cross-agent shared context, and gateway active-run auto-abort. That makes review and rollback risk much higher than a focused ACP persistence change.
- Security-sensitive concerns remain in the PR diff: deserialized session JSON lacks shape guards, and `parent-child` shared-context access is inferred from string prefixes, which could over-share context if agent ids are human-readable or attacker-influenced.
- No supply-chain, workflow, lockfile, or package-publishing changes are visible in the provided file list, but the PR does add new file I/O and cross-session context-sharing behavior that needs explicit maintainer/security review.
