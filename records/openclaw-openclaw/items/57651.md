## Summary

Keep this PR open for maintainer follow-up. Current main still uses an in-memory ACP stdio bridge session store and does not implement the file-backed bridge persistence proposed here. The current ACP control-plane has durable ACP harness metadata, but that is a different product surface from the `openclaw acp` stdio bridge. The PR also remains unsafe to treat as merge-ready because the provided review context flags a nonfunctional persistence implementation plus security-sensitive shared-context and deserialization issues.

## Best Possible Solution

Keep this PR open only as a salvage candidate. The best path is a focused ACP bridge persistence change in `src/acp/session.ts` and `src/acp/translator.ts`, with validated persisted JSON, safe/atomic state writes, restart/loadSession tests for the bridge mapping, and no unrelated compaction, shared-context, or gateway auto-abort changes. The compaction/shared-context work should be split into separately reviewed PRs with explicit security boundaries.

## Risks / Open Questions

- The PR is not merge-ready as submitted: the stated file-backed ACP persistence path is reported as nonfunctional because `_getAllSessions` returns a TypeScript type expression instead of real session data.
- The diff mixes unrelated work across ACP bridge persistence, enhanced/parallel compaction, shared context, gateway active-run behavior, and config schema, which makes review and rollback risk much higher than a focused ACP persistence PR.
- The proposed persisted-session JSON path needs explicit shape validation before use; otherwise corrupt or unexpected state can produce confusing behavior at the bridge boundary.
- The proposed shared-context parent-child access check uses string-prefix inference, which can over-share context if agent IDs are human-readable or attacker-influenced.
- The proposed enhanced-compaction integration mutates module-global config per invocation, which can race across concurrent sessions.
