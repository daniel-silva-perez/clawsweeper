## Summary

Keep this issue open. Current main has abort, queue, steering, and process-cancel primitives, including a legacy `interrupt` queue mode, but it still lacks the requested unified per-session run generation, broad stale-output fences, pre-tool currentness gate, and default new-message takeover. The same-author draft PR #70363 remains open and explicitly partial, so this should stay as the canonical tracker for maintainer review.

## What This Changes

Request stronger run interruptibility so `/stop` or a new message invalidates the active run, prevents stale outputs and further tool/process effects, and lets the newest user instruction take over.

## Best Possible Solution

Keep this as the canonical core interruptibility tracker. Review or split #70363, then land focused core changes for a session/run generation primitive, stale-output fences across reply, typing, followup, tool-progress, and event emission paths, a pre-tool currentness gate in the Pi tool path, process cancellation tied to the invalidated run scope, and regression tests proving stale runs cannot emit visible effects or launch further tools.
