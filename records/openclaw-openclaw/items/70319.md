## Summary

Keep this issue open. Current main still has useful abort, queue cleanup, backend cancel, and process-cancel primitives, but it does not provide the requested unified per-session run invalidation, broad stale-output fencing, pre-tool currentness gate, or default new-message takeover. The same-author draft PR #70363 is open and explicitly partial, so it is an implementation candidate rather than a reason to close this issue.

## What This Changes

The issue requests stronger run-scoped interruptibility so `/stop` or a new message invalidates the active run, cancels owned subprocess/tool work, skips pending tool calls, fences stale output, and lets the new message take over immediately.

## Best Possible Solution

Keep this issue as the canonical core interruptibility tracker. Review or split #70363, then land focused core changes for a session/run generation primitive, stale-output fences across reply, typing, followup, tool-progress, and event emission paths, a pre-tool abort/currentness gate in the Pi tool path, new-message takeover that invalidates and cancels the active run, and regression tests proving stale runs cannot emit visible effects or launch further tools.
