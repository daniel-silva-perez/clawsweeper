## Summary

Keep open. Current main still emits the Model Aliases prompt section with wording that prefers aliases, and the alias-line builder lists every configured `agents.defaults.models` alias without distinguishing primary, fallback, or cross-provider overlap. The runtime preserves exact `provider/model` strings once received, but that does not address the reported parent-model substitution before `sessions_spawn` is called.

## What This Changes

The issue asks OpenClaw to prevent model aliases in the system prompt from encouraging agents to replace an explicit `provider/model` sub-agent model with a different provider alias.

## Best Possible Solution

Keep the issue open and update the model-alias prompt contract so aliases are presented only as shortcuts for bare alias requests, while exact `provider/model` strings are framed as routing identifiers that must be passed through verbatim. If maintainers want stronger protection, add a narrow guard or prompt/tool schema wording for `sessions_spawn.model` without changing normal fallback semantics.

## Reproduction Assessment

Unclear for the full end-to-end failure because it depends on LLM tool-argument choice. The risky prompt state is reproducible on current main by configuring model aliases: OpenClaw emits all alias lines and still tells the agent to prefer aliases.

## Solution Assessment

Yes for keeping this open; current main has not removed or corrected the risky prompt guidance. The narrowest maintainable fix is prompt/schema wording plus focused tests, with optional alias filtering only if it can be done without changing configured fallback behavior.
