## Summary

Keep open. Current main has narrower retry-storm mitigations: in-run compaction/retry caps, model fallback/provider cooldowns, one transient HTTP retry, and targeted auto-reset paths for context overflow, compaction failure, role ordering, and known corruption. I did not find the requested gateway/session-level circuit breaker that persists consecutive session failure state, marks a session unhealthy, blocks or rotates future same-session retries after a threshold, and exposes operator/UI actions. The linked #62614 symptom was closed for narrower retry caps; related #42933 remains an open implementation candidate and #59161 is a narrower consecutive-abort tracker, not a complete superseding issue.

## Best Possible Solution

Keep this issue open as a core gateway/runtime reliability enhancement, or explicitly consolidate it into a canonical session-breaker tracker. The likely implementation should persist per-session consecutive failure counters and signatures, reset on success or explicit reset, trip configurable thresholds, mark breaker/unhealthy state with cooldown metadata, gate heartbeat/cron/auto-reply/session-send before launching another turn, optionally call the existing reset path while preserving continuity metadata, emit structured logs/events, and expose operator/UI actions such as resume anyway and start fresh session.
