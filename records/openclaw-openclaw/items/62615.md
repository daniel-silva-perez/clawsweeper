## Summary

Keep open. Current main has provider/auth-profile cooldown and model-fallback behavior, plus one-shot aborted-run tracking, but it does not have a gateway/session-level circuit breaker that persists consecutive session failures, marks a session unhealthy, suppresses automatic retries/resumes after a threshold, or exposes operator/UI recovery actions. The related open items cover narrower retry/resume variants, so this broader session-health request still needs maintainer product scoping.

## What This Changes

Request a gateway/runtime session-level circuit breaker that bounds retries on unhealthy persistent sessions after consecutive failures, emits visible unhealthy-session state, and optionally rotates or resumes safely.

## Best Possible Solution

Define a durable session-health contract in the gateway/session lifecycle: track consecutive failure signatures and retry exposure per session, reset on successful model output or explicit operator action, suppress automatic same-session resume/retry when tripped, expose the unhealthy state through sessions RPC/status/UI/logs, and integrate with existing provider cooldown/model fallback without bypassing user-selected strict model behavior.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No; this is primarily a feature request and the linked observed retry storm lacks a runnable trace here. The code review does provide high-confidence evidence that current main lacks the requested persistent session breaker state.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose scope; a session-level breaker is directionally sound, but the safest implementation should reuse session lifecycle and fallback telemetry rather than adding an isolated retry wrapper. A narrower first slice could suppress auto-resume after consecutive model errors, aligned with #52384, while leaving UI/token-budget expansion for follow-up.
