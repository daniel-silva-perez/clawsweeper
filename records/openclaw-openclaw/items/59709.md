## Summary

Keep this issue open. Current main has a much stronger generic cron delivery path, but the QQ Bot path is not proven fixed: the QQ Bot adapter can still turn an engine send error into a returned outbound result with an empty message id, while cron marks direct delivery as delivered when the outbound result list is non-empty.

## What This Changes

Cron scheduled agent-turn jobs configured for QQ Bot announce delivery should actually notify the configured QQ session instead of recording `delivered: true` with no received message.

## Best Possible Solution

Keep the issue open and add a QQ Bot cron regression that proves cron run history only reports delivered after a real successful QQ outbound send. If QQ accepts proactive messages but does not surface them to the user, document that limitation and expose a clearer delivery diagnostic instead of a green `delivered` status.

## Reproduction Assessment

Unclear for the full end-to-end QQ notification because this review did not run against live QQ credentials. There is a high-confidence unit-level reproduction path for the false-positive status: mock QQ Bot outbound to return `{ error, messageId: "" }` and verify cron currently can still count the non-empty outbound result as delivered.

## Solution Assessment

No, current main is not enough to close this. The narrowest maintainable fix is to start in the QQ Bot adapter by converting engine `error` results into thrown delivery failures, and only add shared outbound validation if tests show other channel adapters need the same guard.
