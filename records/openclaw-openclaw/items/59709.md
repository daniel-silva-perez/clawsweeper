## Summary

Keep open. Current main has moved isolated cron announce delivery onto the shared outbound adapter path, and QQ Bot now has cron-reminder tooling, but the QQ Bot delivery false-positive class is not proven fixed: QQ Bot send failures can still be represented as outbound results with an empty message id and `meta.error`, while cron marks direct delivery as delivered when the outbound result array is non-empty.

## What This Changes

The issue asks for cron scheduled tasks targeting QQ Bot to actually deliver notifications instead of reporting `delivered: true` / `deliveryStatus: delivered` when no QQ message arrives.

## Best Possible Solution

Keep the current shared outbound-adapter architecture, but make QQ Bot cron delivery accounting depend on verified send success: adapter errors should throw or be filtered as non-delivery, run history should show `not-delivered` with the QQ error, and a QQ-specific isolated cron regression should cover both success and failure paths before closing this issue as fixed.

## Reproduction Assessment

Unclear for a live QQ Bot account because this review was read-only and did not use service credentials. There is, however, a high-confidence unit-level reproduction path for the reported symptom: a QQ Bot send failure can return an error result with no message id, and cron can still count the non-empty result array as delivered.

## Solution Assessment

No, closing this as already implemented is not the best path. The safer fix is a narrow delivery-accounting change plus QQ Bot cron regression coverage, while preserving the newer direct outbound delivery path.

## Repair Work Prompt

Fix the QQ Bot cron delivery false-positive reported in openclaw/openclaw#59709. Current main sends isolated cron announce output through `deliverOutboundPayloads`, but cron marks delivery as successful when the result array is non-empty. QQ Bot outbound can return `{ messageId: '', meta: { error } }` when the engine returns an error, so cron can still persist `delivered: true` without a QQ push. Implement a narrow fix so QQ Bot adapter errors are treated as failed delivery, preferably by making the QQ Bot outbound adapter throw/return no delivery identity on `result.error` or by teaching the shared outbound text path to reject adapter error results consistently. Preserve the existing direct outbound delivery architecture and do not change QQ credentials, scheduling semantics, or unrelated channel behavior. Add focused regression coverage for isolated cron announce to QQ Bot/error-shaped adapter results and a successful QQ Bot path. Add a user-facing changelog fix entry crediting @bxy3045134656 if the bug fix is user-visible.
