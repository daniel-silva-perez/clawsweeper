## Summary

Keep open. Current main has improved generic isolated-cron direct delivery and QQ Bot reminder scheduling, but the QQ Bot outbound adapter still wraps engine send failures into a non-empty outbound result, while cron marks any non-empty result list as delivered. That leaves the reported `delivered=true` / no QQ notification false-positive path plausible and actionable for QQ Bot.

## Best Possible Solution

Keep this issue open. The best fix is to make QQ Bot outbound delivery propagate engine send failures as delivery failures, or make the shared outbound/cron layer treat adapter error metadata as not delivered, then add focused tests for QQ Bot cron success and QQ Bot engine-error -> `deliveryStatus: not-delivered`. After that lands and ships, close as implemented with the fixing commit and release.
