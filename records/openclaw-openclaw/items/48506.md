## Summary

Keep this issue open. Current main still exposes only the raw outbound `to`, content, routing ids, and generic metadata on `message_sending`; it does not surface the resolved target/cross-context policy context requested here. The related CaMeL/security-context issues are adjacent background, not a merged implementation or canonical replacement for this specific hook enrichment.

## Best Possible Solution

Keep the issue open as a focused plugin hook/API feature request. The best implementation path is to add an optional, backward-compatible resolved target context to `PluginHookMessageSendingEvent`, populate it consistently from the existing target resolver and cross-context policy data at every `message_sending` call site, and document/test the new fields for third-party plugin authors.
