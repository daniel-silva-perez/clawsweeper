## Summary

Keep this open. Current main still exposes `message_sending` with raw routing fields only, while resolved target and cross-context information remains internal to outbound resolution/policy code; because this is a public plugin SDK and security-policy API shape with related open hook proposals, it needs maintainer API/security review rather than cleanup.

## What This Changes

The issue requests adding an optional resolved `targetContext` surface to `PluginHookMessageSendingEvent` so plugins can distinguish external, cross-context, target-type, channel, and account-relationship details for outbound messages.

## Best Possible Solution

Define and document a backwards-compatible, optional target-context field on the public plugin hook contract, populated from the existing generic target-resolution and cross-context policy helpers with explicit privacy/security boundaries and contract tests.

## Reproduction Assessment

Yes. The API gap is reproducible by inspecting current main: the public event type and inspected delivery call sites lack resolved target context, while resolver and cross-context policy helpers already compute related data internally.

## Solution Assessment

Unclear. The proposed nested optional `targetContext` direction is plausible, but the final field names, stability guarantees, and information-disclosure boundaries should be settled by maintainers before implementation.
