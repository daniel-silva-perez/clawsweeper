## Summary

Keep this open. Current main still exposes `message_sending` as a rewrite/cancel hook with raw routing fields only, while the resolved target and cross-context data the request needs exists only in internal outbound code; because this is a public plugin SDK and security-policy API shape, it needs maintainer API/security review rather than stale cleanup.

## What This Changes

Request to add optional resolved destination context to `PluginHookMessageSendingEvent` so plugins can make DLP, allowlisting, auditing, and cross-context security decisions without reimplementing target resolution.

## Best Possible Solution

Add a backwards-compatible, optional target-context surface to the public plugin hook contract, populated from existing generic target-resolution and cross-context policy helpers without leaking provider internals beyond an intentionally documented SDK shape.

## Reproduction Assessment

Yes. The API gap is reproducible by reading current `main`: the `message_sending` event type and both inspected delivery call sites lack resolved target context while internal resolver/policy code already computes related data.

## Solution Assessment

Unclear. The proposed optional nested `targetContext` direction is plausible, but the final fields and privacy/security boundaries should be settled by maintainers before implementation.
