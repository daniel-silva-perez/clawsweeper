## Summary

Keep open. Current main still does not expose actual agent/tool execution traces on `QaBusMessage`; the QA bus contract, creation path, clone/read/search paths, and qa-channel delivery path still carry message text, metadata, attachments, and reactions only. The existing `/debug/requests` `plannedToolName` support is a mock-provider workaround used by scenarios, not the requested bus-level trace surface populated from real tool execution.

## What This Changes

The issue requests an optional tool-call trace surface on QA bus messages so qa-lab scenarios can assert actual tool invocations instead of only response text.

## Best Possible Solution

Keep this open and design a backwards-compatible QA trace surface sourced from actual agent tool-start events, with sanitized tool names and arguments, then carry it through QA bus create, clone, snapshot, read, search, poll, and wait paths. The implementation should include focused qa-lab/qa-channel tests plus at least one scenario assertion that fails when an agent fabricates success without calling the expected tool.
