## Summary

Keep open. Current main still lacks a QA bus tool-call trace surface: the public `QaBusMessage` contract, qa-lab message creation, snapshot/read/search paths, and qa-channel delivery all carry text/metadata/attachments/reactions only. Internal agent events do have sanitized tool-start args, and the mock provider exposes planned tool names, but neither is plumbed into QA bus messages as requested.

## What This Changes

Add an optional tool-call trace surface to QA bus messages so qa-lab scenarios can assert actual tool use instead of relying only on response text or mock-provider planned requests.

## Best Possible Solution

Keep this open and design a backwards-compatible QA trace surface sourced from actual agent tool-start execution events, with sanitized tool names and arguments. Carry it through QA bus create, clone, snapshot, read, search, poll, and wait paths, and add focused qa-lab/qa-channel tests plus one scenario that fails when an agent fabricates success without calling the expected tool.

## Reproduction Assessment

Yes. Source inspection on current main is enough to reproduce the missing surface: `QaBusMessage` and the QA bus create/read/search/poll/delivery paths have no field or API that can return actual tool-call traces.

## Solution Assessment

Unclear. The requested optional `QaBusMessage.toolCalls` field is a plausible solution, but maintainers should first decide whether this belongs in the public plugin SDK contract or in a narrower qa-lab scenario trace/assertion API with argument sanitization.
