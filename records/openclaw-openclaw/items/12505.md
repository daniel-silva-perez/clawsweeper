## Summary

Keep open. Current main has useful optional tool-sandbox backends and a public sandbox-backend SDK seam, but it still lacks the central secure-by-default plugin/process isolation architecture requested here; recent maintainer and ClawSweeper comments also treat this as the canonical sandbox/isolation tracker rather than resolved work.

## What This Changes

The issue requests a unified, secure-by-default, multi-platform sandbox architecture with tiered presets, plugin/process isolation, IPC-proxied SDK calls, and per-plugin manifest policies for env, filesystem, network, tools, and resources.

## Best Possible Solution

Keep this as the canonical owner-reviewed sandbox/isolation tracker unless maintainers split it into narrower milestones. The desired end state is a staged design for secure-by-default presets, native plugin or skill execution isolation, IPC/proxy SDK enforcement, manifest-declared env/fs/net/resource policy, credential scoping, resource controls, and migration guidance while preserving current Docker/SSH/OpenShell backend groundwork.

## Reproduction Assessment

Not applicable as a normal bug reproduction. There is a high-confidence verification path for the feature gap: inspect current config/schema/docs/plugin architecture and confirm sandboxing remains optional tool isolation while native plugins still run in-process.

## Solution Assessment

No, this should not be solved as one automated repair. The maintainable path is maintainer-owned design review followed by narrow milestones for defaults, plugin process boundaries, manifest policy, credential scoping, and backend enforcement.
