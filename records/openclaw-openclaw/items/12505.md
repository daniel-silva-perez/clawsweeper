## Summary

Keep open. Current main has useful optional tool sandbox backends and a public sandbox-backend SDK seam, but the central request remains unimplemented: secure-by-default tiered presets, native plugin/process isolation, IPC-proxied plugin SDK calls, per-plugin env/fs/net/resource manifest policy, and default credential scoping. Recent maintainer and ClawSweeper comments also already treat this as the canonical sandbox/isolation architecture tracker rather than resolved work.

## What This Changes

The issue requests a unified, secure-by-default sandbox architecture with multi-platform isolation backends, tiered presets, plugin capability manifests, credential scoping, and resource controls.

## Best Possible Solution

Keep this as the canonical owner-reviewed sandbox/isolation tracker unless maintainers split it into narrower milestones. The desired end state is a staged design and implementation for secure-by-default presets, native plugin or skill execution isolation, IPC/proxy SDK enforcement, manifest-declared env/fs/net/resource policies, credential scoping, resource controls, and migration guidance while preserving current Docker/SSH/OpenShell backend groundwork.

## Reproduction Assessment

Not applicable as a normal bug reproduction. There is a high-confidence verification path for the feature gap: inspect current config/schema/docs/plugin architecture and confirm sandboxing remains optional tool isolation while native plugins still run in-process.

## Solution Assessment

No, this is not best handled as one automated repair. The maintainable path is maintainer-owned design review followed by narrow milestones for defaults, plugin process boundaries, manifest policy, credential scoping, and backend enforcement.
