## Summary

Keep open. Current main has useful optional tool sandbox backends and a public SDK seam for sandbox backends, but it still does not implement the central request: secure-by-default tiered presets, process-isolated native plugin execution, IPC-proxied plugin SDK calls, per-plugin env/fs/net/resource manifest policy, or default credential scoping. The existing maintainer and ClawSweeper comments already treat this as the canonical sandbox/isolation architecture tracker, not resolved work.

## What This Changes

The issue requests a secure-by-default, unified sandbox architecture with multi-platform isolation, tiered presets, plugin capability manifests, IPC-proxied plugin execution, credential scoping, and resource/network/filesystem controls.

## Best Possible Solution

Keep this as the canonical owner-reviewed sandbox/isolation tracker unless maintainers split it into narrower milestones. The desired end state is a staged design and implementation for secure-by-default presets, plugin or skill execution isolation, IPC/proxy SDK enforcement, manifest-declared env/fs/net/resource policies, credential scoping, resource controls, and migration guidance while preserving the current Docker/SSH/OpenShell backend groundwork.

## Reproduction Assessment

Not applicable as a normal bug reproduction. There is a high-confidence verification path for the feature gap: inspect current config/schema/docs/plugin architecture and confirm sandboxing remains optional tool isolation while native plugins still run in-process.

## Solution Assessment

No, this is not best handled as a single automated repair. The maintainable path is maintainer-owned design review followed by narrow milestones for defaults, plugin process boundaries, manifest policy, credential scoping, and backend enforcement.
