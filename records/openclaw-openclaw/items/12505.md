## Summary

Keep open. Current main has useful optional tool sandbox backends, Docker setup support, and a public sandbox-backend SDK seam, but it still does not implement the central request: secure-by-default tiered presets, process-isolated native plugin execution, IPC-proxied plugin SDK calls, per-plugin env/fs/net/resource manifest policy, or default credential scoping. The maintainer and ClawSweeper comments already treat this as the canonical sandbox/isolation architecture tracker rather than resolved work.

## What This Changes

This issue requests a unified, multi-platform sandbox architecture with secure-by-default tiered presets, plugin and skill process isolation, IPC-proxied SDK access, and manifest-declared env/fs/net/resource policies.

## Best Possible Solution

Keep this as the canonical owner-reviewed sandbox/isolation tracker unless maintainers split it into narrower milestones. The desired end state is a staged design and implementation for secure-by-default presets, plugin or skill execution isolation, IPC/proxy SDK enforcement, manifest-declared env/fs/net/resource policies, credential scoping, resource controls, and migration guidance while preserving current Docker/SSH/OpenShell backend groundwork.

## Reproduction Assessment

Not applicable as a normal bug reproduction. There is a high-confidence verification path for the feature gap: inspect current config/schema/docs/plugin architecture and confirm sandboxing remains optional tool isolation while native plugins still run in-process.

## Solution Assessment

No, this is not best handled as one automated repair. The maintainable path is maintainer-owned design review followed by narrow milestones for defaults, plugin process boundaries, manifest policy, credential scoping, and backend enforcement.
