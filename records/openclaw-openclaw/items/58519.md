## Summary

Keep this issue open. Current main has useful Slack Socket Mode mitigations, including configurable ping/pong timeouts and bounded Slack media downloads, but it still does not implement the issue’s core request: isolating Socket Mode keepalive from gateway/agent-turn event-loop starvation or otherwise guaranteeing the Slack transport can service pings during multi-minute heavy turns.

## Best Possible Solution

Keep the issue open and treat it as a bundled Slack plugin/gateway reliability bug. The best fix is to decouple Slack Socket Mode liveness from heavy agent-turn work: either move the Socket Mode transport/ack boundary into an isolated worker or restructure Slack inbound handling so Slack acknowledgments and keepalive stay prompt while prepare/agent dispatch runs asynchronously. A complete fix should include regression coverage with a synthetic blocked/slow turn and should preserve the existing #14248 timeout tuning and #61850 bounded-media safeguards.

## Risks / Open Questions

- Closing this now would conflate the narrower #14248 timeout/config fix and the #61850 media-download fix with the broader keepalive isolation problem.
- A 15s or configurable Slack SDK pong timeout can reduce false positives, but it does not protect a Socket Mode client whose timers and WebSocket callbacks are still scheduled on a starved gateway event loop.
- Gateway event-loop health diagnostics improve observability, but they do not by themselves prevent Slack Socket Mode ping/pong deadlines or lost inbound processing during heavy turns.
