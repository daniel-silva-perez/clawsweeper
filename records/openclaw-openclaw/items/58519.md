## Summary

Keep this open. Current main and v2026.4.27 include useful Slack Socket Mode mitigations, especially a 15s default pong timeout plus configurable ping/pong options, but the central report is broader: Slack keepalive still runs through Bolt/socket-mode in the Gateway process with no worker-thread or process isolation, so heavy main-thread work can still starve the SDK timers.

## What This Changes

The issue reports Slack Socket Mode pong timeouts and possible silent Slack reply loss when heavy Gateway or agent work starves the Node.js event loop.

## Best Possible Solution

Keep the shipped timeout/media hardening, then scope a dedicated Slack transport reliability design that either isolates Socket Mode keepalive from the Gateway event loop or identifies and yields/removes the specific synchronous Gateway hot paths that can block Slack heartbeat timers, with regression coverage for reconnect-window delivery behavior.

## Reproduction Assessment

Unclear for the full user-visible message-loss claim: the discussion has credible production logs, and current source confirms no worker/process isolation for Slack keepalive, but this review did not reproduce a dropped Slack reply against live Slack credentials. The missing isolation path is reproducible by static inspection.

## Solution Assessment

No, current main is not the complete solution for this issue. The 15s timeout and config pass-through are the right partial mitigation for #14248, but the best fix for this report needs a scoped transport-isolation or event-loop starvation repair rather than closing on timeout tuning alone.
