## Summary

Keep open. Current main still has both reported behaviors: unclassified assistant `errorMessage` text can still reach lifecycle/delivery payloads after redaction instead of a generic fallback, and assistant visible-text extraction still joins multiple matching text blocks for delivery-facing paths. The issue discussion correctly notes these are related but not identical blast-radius changes: raw-error hardening is narrow, while multi-text selection touches shared text extraction policy.

## Best Possible Solution

Keep this issue open. Review and land #41803 or an equivalent small change for the unclassified assistant-error fallback, with a regression for an OpenAI 500/help.openai.com/request-ID shape. Then split or explicitly scope the multi-text behavior as a delivery-boundary policy, avoiding an accidental shared extractor contract change unless maintainers want that across TUI, gateway display, and chat-history surfaces.
