## Summary

Keep open. Current main still uses assistant error text as the fallback for terminal error delivery, and visible-text extraction still joins multiple matching text blocks instead of applying a delivery-only pick-one policy. The discussion's recommendation to split or explicitly scope the two fixes remains accurate.

## Best Possible Solution

Keep this issue open. The best path is to land a small delivery/lifecycle hardening change so terminal assistant errors without a known friendly classification emit a generic user-facing fallback or no chat reply, with a regression for the plain OpenAI 500/help-link/request-id string. The multi-text behavior should be split or explicitly scoped as a delivery-only policy after owner review, because the shared extractors currently join text blocks for multiple surfaces.
