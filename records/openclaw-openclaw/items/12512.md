## Summary

Keep open. Current main still exposes SKILL.md as a model-followed instruction surface and provides eligibility, allowlist, visibility, scan, sandbox, and tool-policy controls, but not the requested trusted/untrusted instruction boundary or per-skill role, memory, and tool capability enforcement. Recent maintainer-side review also kept this open, and there has been no relevant change in the central prompt/skills paths since that review.

## What This Changes

Request to isolate SKILL.md instructions from trusted agent context with instruction precedence, role locking, behavioral monitoring, and per-skill capability boundaries for role, memory, and tool influence.

## Best Possible Solution

Define a maintainer-approved trusted/untrusted skill precedence model, extend skill metadata and runtime policy with role, memory, and tool-influence controls, enforce that boundary in prompt assembly and runtime execution, and add malicious SKILL.md regression tests plus public docs for the new contract.

## Reproduction Assessment

Yes for the missing-control behavior: current main can be inspected with any prompt-visible skill path, and the generated system prompt tells the model to read and follow its SKILL.md. A deterministic exploit transcript would depend on model behavior, but the absence of isolation and per-skill capability metadata is directly reproducible from source and docs.

## Solution Assessment

Unclear as a direct implementation plan. The requested direction is valid, but delimiters or pattern filtering alone would be weaker than a designed capability contract with explicit prompt precedence, runtime enforcement, compatibility tests, and docs.
