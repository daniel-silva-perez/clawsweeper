## Summary

Keep open. Current main still exposes skills through a model-facing catalog and instructs the model to read and follow selected SKILL.md files; the implemented controls are eligibility, prompt visibility, allowlists, install/generation scanning, sandbox/tool policy, and config gates, not the requested trusted/untrusted instruction isolation, role locking, or per-skill role/memory/tool capability enforcement. The April 26 and April 29 issue comments also kept this open for maintainer security-design follow-up, and the checked code at current main still matches that conclusion.

## What This Changes

The issue requests isolating untrusted SKILL.md instructions from trusted agent/system context, with role locking, injection filtering/monitoring, and per-skill role, memory, and tool capability enforcement.

## Best Possible Solution

Maintainers should define and document a trusted/untrusted skill precedence model, extend skill metadata and runtime policy with role, memory, and tool-influence controls, enforce the boundary through prompt assembly plus runtime policy, and add malicious SKILL.md regression coverage around the prompt and skill paths.
