## Summary

Keep open. Current main still exposes skills through a model-facing catalog and directs the model to read and follow a selected SKILL.md, while the implemented controls cover load eligibility, prompt visibility, agent allowlists, sandbox/tool policy at agent scope, and install/security scanning rather than the requested trusted/untrusted instruction isolation, role locking, per-skill memory/tool capability enforcement, or runtime monitoring. The recent maintainer-side issue discussion also explicitly kept this open for core security hardening.

## What This Changes

The issue requests instruction isolation for SKILL.md content, including role locking, output filtering or monitoring, and per-skill capability boundaries for role, memory, and tool influence.

## Best Possible Solution

Define and document a maintainer-approved trusted/untrusted skill precedence model, extend skill metadata and runtime policy with per-skill role, memory, and tool-influence controls, enforce that boundary through prompt assembly plus runtime policy, and add malicious SKILL.md regression coverage and docs.
