## Summary

Keep open. Current main still exposes skills through a model-facing catalog and instructs the model to read and follow selected SKILL.md files; the implemented controls are visibility, allowlists, install/generation scanning, sandbox/tool policy, and config gates, not the requested trusted/untrusted instruction isolation or per-skill role/memory/tool capability enforcement. The April 26 and April 29 issue comments also kept this open, and the relevant source has not materially changed since that review.

## What This Changes

The issue requests a trusted/untrusted instruction-isolation boundary for SKILL.md content so skill instructions cannot override agent role, memory, or tool policy.

## Best Possible Solution

Define and document a trusted/untrusted skill precedence model, extend skill metadata and runtime policy with role, memory, and tool-influence controls, enforce that boundary through prompt assembly plus runtime policy, and add malicious SKILL.md regression coverage around the affected prompt and skill paths.
