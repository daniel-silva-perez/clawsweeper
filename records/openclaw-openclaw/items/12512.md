## Summary

Keep open. Current main still exposes SKILL.md through a model-facing skills catalog and instructs the model to read and follow selected skill files; the implemented controls cover eligibility, visibility, install scanning, sandbox/tool policy at agent/session scope, and generated-skill scanning, but not the requested instruction isolation, role locking, per-skill memory/tool capability enforcement, or runtime behavioral monitoring. The recent maintainer-side review in the issue discussion also explicitly kept this open, and the relevant files have not changed since that review.

## What This Changes

Add instruction isolation and enforceable security boundaries so SKILL.md content cannot override agent role, memory, or tool policy.

## Best Possible Solution

Define a maintainer-approved trusted/untrusted precedence model for skills, add typed per-skill capability metadata for role, memory, and tool influence, enforce it through prompt assembly and runtime policy, and cover malicious SKILL.md cases with focused regression tests and docs.
