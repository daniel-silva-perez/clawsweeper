## Summary

Keep open. Current main still exposes skills through a model-facing catalog and instructs the model to read and follow selected SKILL.md files; existing controls cover visibility, allowlists, install scanning, path hardening, and agent/tool sandbox policy, not the requested trusted/untrusted instruction isolation, role locking, or per-skill memory/tool capability enforcement.

## What This Changes

The issue requests an instruction-isolation architecture for SKILL.md so untrusted skill instructions cannot override agent role, memory behavior, or tool policy.

## Best Possible Solution

Define and document a maintainer-approved trusted/untrusted skill precedence model, extend skill metadata and runtime policy with per-skill role, memory, and tool-influence controls, enforce that boundary through prompt assembly plus runtime policy, and add malicious SKILL.md regression coverage.
