## Summary

Keep open. Current main still exposes prompt-visible skills through an available-skills catalog and tells the model to read and follow selected SKILL.md files; existing controls cover eligibility, visibility, allowlists, install scanning, and sandbox/config gates, not the requested trusted/untrusted instruction boundary, role locking, or per-skill role/memory/tool capability enforcement.

## What This Changes

The issue requests core security hardening that isolates untrusted SKILL.md instructions from the agent context, with role locking, filtering/monitoring, and per-skill capability metadata.

## Best Possible Solution

Define a maintainer-approved trusted/untrusted skill precedence model, extend skill metadata and runtime policy with role, memory, and tool-influence controls, enforce that boundary in prompt assembly and runtime policy, and add malicious SKILL.md regression tests plus docs for the new contract.

## Reproduction Assessment

Yes for the missing-control behavior: current main can be inspected by building a prompt-visible skill and seeing that the system prompt tells the model to read and follow its SKILL.md. A live prompt-injection exploit is model-dependent, but the absence of the requested enforcement boundary is directly reproducible from source and tests.

## Solution Assessment

Unclear as a direct implementation plan. The requested direction is valid, but pattern filtering or delimiters alone would be weaker than a designed capability contract with prompt precedence rules and runtime policy enforcement.
