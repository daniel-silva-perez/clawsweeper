## Summary

Keep open. Current main still exposes prompt-visible skills through an available-skills catalog and tells the model to read and follow selected SKILL.md files; the existing controls cover eligibility, visibility, allowlists, install scanning, and plugin prompt-hook policy, not the requested trusted/untrusted skill instruction boundary or per-skill role, memory, and tool capability enforcement.

## What This Changes

The issue requests instruction isolation and capability enforcement so SKILL.md content cannot override the agent role, memory, or tool policy through prompt injection.

## Best Possible Solution

Define a maintainer-approved trusted/untrusted skill precedence model, extend skill metadata and runtime policy with role, memory, and tool-influence controls, enforce that boundary in prompt assembly and runtime policy, and add malicious SKILL.md regression tests plus docs for the new contract.

## Reproduction Assessment

Yes for the missing-control behavior: current main can be inspected by adding or loading a prompt-visible skill and observing that the system prompt tells the model to read and follow its SKILL.md. A live prompt-injection exploit is model-dependent, but the missing isolation and capability contract are directly reproducible from source and existing prompt tests.

## Solution Assessment

Unclear as a direct implementation plan. The requested direction is valid, but delimiters or pattern filtering alone would be weaker than a designed capability contract with prompt precedence rules and runtime policy enforcement.
