## Summary

Keep open. Current main still exposes prompt-visible skills through an available-skills catalog and instructs the model to read and follow selected SKILL.md files; existing controls cover visibility, eligibility, allowlists, sandbox/runtime scope, and generated-skill scanning, but not the requested trusted/untrusted instruction boundary or per-skill role, memory, and tool capability enforcement.

## What This Changes

The issue requests a core instruction-isolation and capability-enforcement boundary for SKILL.md content so malicious skills cannot override role, memory, or tool permissions.

## Best Possible Solution

Define a maintainer-approved trusted/untrusted skill precedence model, extend skill metadata and runtime policy with role, memory, and tool-influence controls, enforce that boundary in prompt assembly and runtime policy, and add malicious SKILL.md regression tests plus docs for the new contract.

## Reproduction Assessment

Yes for the missing-control behavior: current main can be inspected or exercised with a prompt-visible skill, and the generated system prompt tells the model to read and follow its SKILL.md. A concrete live prompt-injection exploit remains model-dependent, but the absent isolation contract is directly reproducible from source and prompt tests.

## Solution Assessment

Unclear as a direct implementation plan. The requested direction is valid, but delimiters or pattern filtering alone would be weaker than a designed capability contract with explicit prompt precedence and runtime enforcement.
