## Summary

Keep open. Current main still exposes skills through a model-facing catalog and tells the model to read and follow selected SKILL.md files; the current controls cover eligibility, prompt visibility, agent allowlists, path hardening, install scanning, and general sandbox/tool policy rather than the requested trusted/untrusted instruction isolation, role locking, or per-skill memory/tool capability enforcement. The issue discussion also includes recent maintainer-side reviews that explicitly kept this open for core security hardening.

## What This Changes

The issue requests a SKILL.md instruction-isolation and capability-enforcement model so skill content cannot override trusted agent instructions, memory policy, or tool boundaries.

## Best Possible Solution

Define and document a maintainer-approved trusted/untrusted skill precedence model, extend skill metadata and runtime policy with per-skill role, memory, and tool-influence controls, enforce that boundary through prompt assembly plus runtime policy, and cover malicious SKILL.md cases with focused regression tests and docs.
