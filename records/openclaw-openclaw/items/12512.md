## Summary

Keep open. Current main still makes SKILL.md a model-followed instruction surface and only provides eligibility, allowlist, visibility, scanning, and sandbox/tool controls, not the requested trusted/untrusted instruction boundary or per-skill role, memory, and tool capability enforcement. The recent maintainer-side discussion also kept this open, and the relevant prompt/skills paths have not materially changed since that review.

## What This Changes

Request to isolate SKILL.md instructions from trusted agent context by adding instruction precedence, role locking, behavioral monitoring, and per-skill capability boundaries for role, memory, and tool influence.

## Best Possible Solution

Define a maintainer-approved trusted/untrusted skill precedence model, extend skill metadata and runtime policy with role, memory, and tool-influence controls, enforce that boundary in prompt assembly and runtime execution, and add malicious SKILL.md regression tests plus public docs for the new contract.

## Reproduction Assessment

Yes for the missing-control behavior: current main can be inspected or exercised with any prompt-visible skill, and the generated system prompt tells the model to read and follow its SKILL.md. A concrete malicious-skill exploit remains model-dependent, but the absence of isolation and per-skill capability metadata is directly reproducible from source and docs.

## Solution Assessment

Unclear as a direct implementation plan. The requested direction is valid, but delimiters or pattern filtering alone would be weaker than a designed capability contract with explicit prompt precedence, runtime enforcement, and compatibility tests.
