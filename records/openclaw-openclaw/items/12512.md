## Summary

Keep open. Current main at 39f810911c382f92ea80b2fb406fdf7dbd405721 still exposes skills through a model-facing available-skills catalog and instructs the model to read and follow selected SKILL.md files. Existing controls are visibility, install scanning, environment/config eligibility, and allowlists, not the requested instruction isolation, role locking, per-skill memory/tool capability enforcement, or behavioral monitoring.

## What This Changes

The issue requests a core SKILL.md instruction-isolation architecture with trusted/untrusted precedence, role locking, behavioral monitoring, and per-skill role/memory/tool capability metadata.

## Best Possible Solution

Keep this open as core security hardening. Define a trusted/untrusted precedence model for SKILL.md, add enforceable per-skill capability metadata for role, memory, and tool influence, apply it through prompt assembly/read handling/tool policy/memory/sandbox behavior, and add malicious-skill regression tests plus docs after maintainer design approval.
