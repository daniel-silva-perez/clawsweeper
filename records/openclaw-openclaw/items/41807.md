## Summary

Keep this PR open. Current main still lacks a global `agents.defaults.systemPromptFile` schema/type/runtime path, and the PR is a useful implementation candidate for the closed stale request in #36190, but it is not mergeable as-is: it leaves public config schema/docs surfaces stale, needs an explicit prompt-file trust boundary, and misses CLI session invalidation for file changes. The broader additive prompt/rules design in #50103 is also still open context for maintainer API direction.

## What This Changes

The PR adds an optional `agents.defaults.systemPromptFile` config key, reads the configured local file, prepends its contents to CLI and embedded runner system prompts, and adds helper tests for file-reading behavior.

## Best Possible Solution

Land a tightened, maintainer-reviewed prompt-default design: either update this PR or fold it into the broader #50103 global prompt/rules work, with explicit path/trust semantics, CLI session invalidation based on prompt-file identity or content, generated config schema/help/docs baselines, focused tests for both runner paths, and a user-facing changelog entry.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: current main has no global `agents.defaults.systemPromptFile` path, and the PR's CLI reuse bug follows from hashing `params.extraSystemPrompt` before the later file read while resumed CLI runs do not resend first-session system prompts.

## Solution Assessment

No. The hook point is plausible, but this is not the best merge shape until the public config surface, prompt-file trust boundary, and CLI session invalidation are handled; #50103 may also be the better place to settle the final API.
