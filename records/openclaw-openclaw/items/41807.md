## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks `agents.defaults.systemPromptFile`, so the feature is not obsolete; the proposed patch leaves the generated/public config schema surface stale, needs a clearer immutable prompt-file trust boundary, and misses CLI session invalidation when the configured file changes.

## What This Changes

The PR adds an optional `agents.defaults.systemPromptFile` config key, reads the configured local file, prepends its contents to CLI and embedded runner system prompts, and adds helper tests for file-reading behavior.

## Best Possible Solution

Land a tightened, maintainer-reviewed prompt-default design: either fold this into the broader #50103 global prompt/rules work or update this PR with explicit file trust semantics, CLI session invalidation based on prompt-file content or identity, generated config schema/help/docs baselines, and focused tests for both runner paths.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: current main has no global `agents.defaults.systemPromptFile` schema/type/runtime path, and the CLI reuse bug follows from hashing `params.extraSystemPrompt` before the PR's later file read while resumed CLI runs do not resend first-session system prompts.

## Solution Assessment

No. The hook point is plausible, but this is not the best merge shape until the public config surface, trust boundary, and CLI session invalidation are handled; #50103 may also be the better place to decide the final API.

  - body: The helper rereads the configured path on each session while the feature is positioned as operator-unmodifiable guidance.   - confidence: 0.86

Reusable CLI sessions are therefore not invalidated when this config is added or the file changes, and resumed CLI runs skip resending the system prompt, so existing sessions can keep running without the operator prompt.
  - confidence: 0.84
