## Summary

Keep this PR open. Current main still does not provide a default-level file-based system prompt include, so the contribution is not obsolete, but the branch should not merge as-is because it leaves the public config schema/docs surface incomplete, needs a clearer immutable-file trust boundary, and misses CLI session reuse invalidation.

## What This Changes

The PR adds an optional `agents.defaults.systemPromptFile` config key, reads a configured local file, prepends its contents to CLI and embedded runner system prompts, and adds helper tests for file reading behavior.

## Best Possible Solution

Land a tightened version only after maintainer review: decide whether this belongs in the broader #50103 global prompt design, validate or clearly document absolute outside-workspace file semantics, include file content or file identity in CLI session invalidation, update generated config schema/help/docs baselines, and keep focused helper plus runner coverage.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: current main has no `agents.defaults.systemPromptFile` schema/type/runtime path, and the CLI reuse bug follows from hashing `params.extraSystemPrompt` before the PR's later file read while resumed CLI runs do not resend the system prompt.

## Solution Assessment

No. The hook point is plausible, but this patch is not the best merge shape until the public config surface, trust boundary, and CLI session invalidation are handled; #50103 may also be the better place to settle the API shape.

  - confidence: 0.86

Reusable CLI sessions are therefore not invalidated when this config is added or the file changes, and resumed CLI runs skip resending the system prompt, so existing sessions can keep running without the operator prompt.
