## Summary

Keep this PR open. Current main still does not implement `agents.defaults.systemPromptFile`, and the PR remains a plausible implementation candidate, but it is not ready to merge because it misses the generated config/docs surface, leaves the immutable prompt-file boundary undefined, and does not include prompt-file content in CLI session reuse invalidation. The broader #50103 issue may become the canonical product direction if maintainers prefer an inline/rules-based prompt config surface.

## What This Changes

The PR adds an optional `agents.defaults.systemPromptFile` config key, a helper that reads a configured local file, injects that content into CLI and embedded runner system prompts, and adds focused helper tests.

## Best Possible Solution

Land a tightened version only after maintainer review: define the global prompt config API, enforce or clearly validate/document the outside-workspace immutable file boundary, include prompt-file content in CLI session reuse invalidation, regenerate schema/help/docs/baselines, and keep focused CLI plus embedded prompt coverage. If #50103 becomes canonical, fold this file include use case into that design instead of merging a narrower incompatible surface.

## Reproduction Assessment

Yes. Current main can be verified statically: there is no `agents.defaults.systemPromptFile` schema/type/runtime path, and the PR's CLI reuse bug follows from hashing only `params.extraSystemPrompt` before the file read while resume skips resending the system prompt.

## Solution Assessment

No, not in its current form. The hook point is plausible, but the patch needs public config generation/docs, a defined immutable-file trust boundary, and CLI session reuse invalidation before it is the narrow maintainable fix.

Full review comments:

- **[P2] Regenerate and document the config schema surface:** `src/config/zod-schema.agent-defaults.ts:41`
  - body: Adding this public key only to Zod/types leaves the served config schema, field labels/help, docs, and config-doc baselines stale. `buildBaseConfigSchema()` serves the committed generated schema, so `openclaw config schema` will not expose the new key until those generated/docs surfaces are updated.
  - confidence: 0.91
- **[P2] Define the prompt-file trust boundary:** `src/agents/system-prompt-file.ts:14-16`
  - body: The helper rereads the configured path on each session while the feature is positioned as operator-unmodifiable guidance. If the configured path is workspace-relative or agent-writable, an agent can affect future prompts, so the patch should enforce, snapshot, or document and validate the required outside-workspace deployment model.
  - confidence: 0.84
- **[P2] Include prompt-file content in CLI session hashing:** `src/agents/cli-runner/helpers.ts:78-82`
  - body: The file content is appended inside `buildSystemPrompt()`, after `prepare.ts` has already computed `extraSystemPromptHash`. Reusable CLI sessions therefore are not invalidated when this config is added or the file changes, and `execute.ts` skips resending the system prompt on resume, so some CLI sessions can keep running without the new operator prompt.
  - confidence: 0.83
