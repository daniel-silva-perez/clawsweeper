## Summary

Keep this issue open. Current main still does not handle the reported inline `[TOOL_CALL]{tool => ..., args => ...}[/TOOL_CALL]` pseudo tool-call syntax: the shared sanitizer strips XML tool tags and newline-delimited `[tool_name]` JSON blocks, the diagnostic detector only warns for supported tool-shaped text formats, and WhatsApp delivery still sends normalized text chunks through `msg.reply`. The reporter's follow-up is important because it identifies the failure as plain assistant text, not a structured tool invocation.

## Best Possible Solution

Keep this issue open. Add shared handling for inline `[TOOL_CALL]{tool => ..., args => ...}[/TOOL_CALL]` assistant text before any channel delivery. At minimum, suppress it from user-visible WhatsApp replies and history; only promote it into real tool calls if a deterministic parser can validate the syntax against registered tools. Add focused shared sanitizer/parser tests plus a WhatsApp auto-reply regression proving the raw pseudo call is not passed to `msg.reply`.
