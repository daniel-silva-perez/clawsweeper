## Summary

Keep this issue open. Current main still does not handle the reported inline `[TOOL_CALL]{tool => ..., args => ...}[/TOOL_CALL]` pseudo tool-call text. The shared sanitizer removes XML tool-call tags, standalone `[tool_name]` JSON blocks, and downgraded `[Tool Call: ...]` scaffolding, but not this uppercase inline `tool =>` / `args =>` form; WhatsApp delivery still sends the resulting normalized text chunks directly.

## Best Possible Solution

Add shared handling for inline `[TOOL_CALL]{tool => ..., args => ...}[/TOOL_CALL]` assistant text before any channel delivery. At minimum, suppress it from user-visible replies and sanitized history; only promote it into real tool calls if a deterministic parser can validate the syntax against the registered tools for that turn. Add focused shared sanitizer/parser coverage and a WhatsApp auto-reply regression proving the raw pseudo call is not passed to `msg.reply`.

## Evidence

That keeps the remaining work focused on text handling before channel delivery, not Tavily execution itself.
- **Exact syntax has no current source/docs/test coverage:** A targeted search found no handling for `[TOOL_CALL]`, `[/TOOL_CALL]`, `tool =>`, or `args =>` in the inspected source, docs, tests, and WhatsApp plugin files.
