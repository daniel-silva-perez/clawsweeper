## Summary

Keep this open. Current main suppresses several tool-text leaks, including XML tool blocks and standalone bracketed JSON tool requests, but it still has no handling for the reported inline uppercase `[TOOL_CALL]{tool => ..., args => ...}[/TOOL_CALL]` pseudo-call form. WhatsApp normalizes reply text and then sends the resulting chunks through `msg.reply`, so unsupported pseudo-call text can still reach the user-visible reply path.

## What This Changes

The issue reports WhatsApp replies exposing inline `[TOOL_CALL]{tool => ..., args => ...}[/TOOL_CALL]` pseudo tool-call text instead of executing or hiding the tool interaction.

## Best Possible Solution

Add shared handling for inline `[TOOL_CALL]{tool => ..., args => ...}[/TOOL_CALL]` assistant text before channel delivery. The safest end state is that the raw pseudo-call never appears in WhatsApp replies or visible history; execution should only be added if a deterministic parser validates the syntax and registered tool name without broadening untrusted text execution.
