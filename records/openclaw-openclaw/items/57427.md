## Summary

Keep open. Current main still loads global/config env into the gateway process but sandbox exec builds a separate environment from defaults, sandbox backend env, and per-call env only; the OpenShell backend attaches configured providers but does not pass resolved provider credential values into the remote shell. The related implementation PRs are still open, and the remaining fix needs an explicit admin-approved sandbox env contract rather than blanket `.env` forwarding.

## What This Changes

The issue asks OpenClaw to make admin-provided environment variables from `~/.openclaw/.env` or config available to sandbox exec/OpenShell provider commands so provider placeholders do not reach outbound requests unresolved.

## Best Possible Solution

Define and document an explicit sandbox env forwarding contract for admin-approved keys, including OpenShell behavior, then add regression coverage showing the configured key reaches sandbox exec while unrelated sensitive host env remains blocked.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: put a key only in `~/.openclaw/.env` or config runtime env, run sandbox/OpenShell exec, and the current env builder has no path that includes that key unless it is also in sandbox backend env or per-call env.

## Solution Assessment

Yes, keeping this open for a narrow security-reviewed fix is the right direction. The safer solution is not blanket `.env` inheritance, but explicit admin-approved forwarding or an OpenShell provider credential handoff with tests and docs.
