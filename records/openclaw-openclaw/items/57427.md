## Summary

Keep open. Current main still has no path that forwards `~/.openclaw/.env`, `env.vars`, or OpenShell provider credential values into sandbox `exec` environments. OpenShell attaches provider names during sandbox creation, but command execution receives only the computed sandbox env (`sandbox.docker.env`/backend env plus per-call `env`), so the reported unresolved placeholder behavior remains plausible. Related PRs #40290 and #44569 are still only implementation candidates in the provided context, not landed fixes.

## Best Possible Solution

Keep this issue open. The best maintainer path is to define an explicit sandbox env forwarding contract for admin-approved keys, covering OpenShell and other sandbox backends, then add tests proving approved keys reach sandbox exec while unrelated sensitive host env remains blocked. Review PRs #40290 and #44569 as candidates, but close only after a focused fix lands and the docs describe the safe configuration path.

## Risks / Open Questions

- A fix that blindly forwards all gateway process env or global dotenv values into sandbox exec would leak secrets into less-trusted runtimes; it needs an explicit admin-approved allow/pass contract with regression coverage.
- The OpenShell provider placeholder behavior may also need coordination with OpenShell's proxy/provider credential semantics, but current OpenClaw main still lacks the core-side resolved-value forwarding path.
