## Summary

Keep open. Current main still does not forward trusted global `.env` or config env vars into sandbox exec/OpenShell remote commands unless they are explicitly present in sandbox backend env or the tool-call env. The OpenShell plugin attaches configured providers, but it does not resolve or pass provider credential values into the remote shell, and docs still describe sandbox env as separate from host `process.env`. The public issue text still matches that unresolved placeholder behavior. citeturn1search0

## Best Possible Solution

Keep this issue open. The best maintainer path is to define an explicit sandbox env forwarding contract for admin-approved keys, including OpenShell, and add regression coverage showing a configured key reaches sandbox exec while unrelated sensitive host env stays blocked. Review related PRs such as #40290/#44569 as implementation candidates, but close this only after a focused fix lands and docs explain the safe configuration path.

## Risks / Open Questions

- A fix that blindly passes all host or global `.env` values into sandbox exec would leak secrets into less-trusted runtimes and contradict current docs; it needs an explicit admin-controlled allow/pass mechanism and tests.
- The final behavior may need coordination with OpenShell's L7 proxy/provider placeholder semantics, but OpenClaw current main still lacks a core-side path to provide the resolved value.
