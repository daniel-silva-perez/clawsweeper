## Summary

Keep this PR open. Current main still lacks the proposed reply-run quiet-timer watchdog and still has user-triggered empty-final paths that can finalize as undefined, while the PR remains focused on the relevant auto-reply surface and has addressed the prior bot review comments.

## What This Changes

The PR adds a deterministic auto-reply progress watchdog, visible-activity reset wiring, an empty-final fallback helper, and focused tests for long or silent reply runs.

## Best Possible Solution

Review and land this PR, or an equivalent narrow core auto-reply implementation, after maintainer acceptance of the user-visible notice cadence and normal CI. The end state should keep the watchdog generic to reply-run lifecycle state, avoid channel-specific policy in core, and preserve existing preview/tool-progress behavior without duplicate visible delivery.

## Reproduction Assessment

Yes. Current main has a clear source-level reproduction path: a non-heartbeat reply run whose `runResult.payloads` is empty or whose payloads are filtered to zero reaches `finalizeWithFollowup(undefined, ...)`, and a long run has no generic quiet-timer notice unless an existing channel preview/tool-progress path is active.

## Solution Assessment

Yes, with normal maintainer review. The PR uses the existing reply operation phase model and block-reply callback instead of adding channel-specific logic, and the empty-final fallback is narrowly gated for user-triggered runs with no other visible delivery path.
