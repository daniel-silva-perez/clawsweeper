## Summary

Keep open. Current main still lacks the Mattermost-specific tool-threading context needed for implicit same-thread `message` sends, and the cron/heartbeat delivery-context concern also remains visible in the Mattermost monitor path. The provided timeline shows open unmerged closing PRs #52120 and #52236 already targeting these two paths, so this issue should stay open until those PRs land or are replaced.

## What This Changes

The issue asks Mattermost proactive `message` tool sends and cron/heartbeat wakes to inherit the active thread context when no explicit reply target is provided.

## Best Possible Solution

Land or replace the Mattermost-owned fixes so the plugin supplies guarded tool-threading context for same-target sends and persists route thread IDs for thread-scoped channel sessions, with focused regressions for explicit `replyTo`, implicit same-thread sends, cross-channel suppression, `replyToMode`, and cron/heartbeat wake delivery.

## Reproduction Assessment

Yes. The issue gives concrete Mattermost reproduction steps, and current main provides a high-confidence static reproduction: the Mattermost send action cannot inherit thread context because it ignores `toolContext` and the plugin does not build Mattermost tool context.

## Solution Assessment

Yes. The narrow maintainable path is to fix Mattermost's plugin/threading and route metadata seams; adding a broad core special case would bypass the existing generic plugin threading contract.
