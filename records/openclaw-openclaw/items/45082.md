## Summary

Keep open. Current main still lacks the Mattermost-owned tool context and send-action fallback needed for proactive message-tool sends to inherit an active Mattermost thread, and the cron/heartbeat route metadata concern is still visible in current code. Open closing PRs already target both reported paths, so this should stay open for maintainer PR review rather than be closed or queued as a separate repair.

## What This Changes

The issue requests that Mattermost proactive `message` sends and cron/heartbeat wake replies inherit the active thread context when a thread-scoped session has no explicit `replyTo`.

## Best Possible Solution

Review, rebase, land, or replace the linked Mattermost PRs with a Mattermost-owned `buildToolContext`, guarded send-action inheritance for same-target thread sends, non-direct delivery-context persistence where needed, and focused regression tests for explicit `replyTo`, implicit same-thread sends, cross-channel suppression, `replyToMode` behavior, and cron/heartbeat wake delivery.
