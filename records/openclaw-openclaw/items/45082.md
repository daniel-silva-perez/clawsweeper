## Summary

Keep open. Current main still leaves the Mattermost message-tool send path using only explicit `replyTo`/`replyToId`, and the plugin still lacks the Mattermost tool-context builder needed for implicit same-thread sends. The cron/heartbeat wake path also still has direct-only last-route persistence, while open PRs #52120 and #52236 already target the two reported gaps.

## What This Changes

Default Mattermost proactive message-tool sends and cron/heartbeat wake replies to the active Mattermost thread when the session is thread-scoped and no explicit reply target is provided.

## Best Possible Solution

Review, rebase, land, or replace #52120 and #52236 with a Mattermost-owned implementation that supplies tool thread context, guardedly inherits same-target thread roots for implicit sends, persists non-direct thread delivery context where needed, and adds focused regressions for explicit `replyTo`, implicit same-thread sends, cross-channel suppression, `replyToMode`, and cron/heartbeat wake delivery.

## Reproduction Assessment

Yes. The issue has concrete Mattermost reproduction steps, and current source gives a high-confidence static reproduction path: the message-tool send path cannot inherit Mattermost thread context, and non-direct cron/heartbeat route metadata is still not persisted.

## Solution Assessment

Yes for the linked implementation direction. The narrow maintainable fix belongs in the Mattermost plugin and inbound session-route metadata path rather than a broad core special case; maintainers should decide whether to land the existing PRs or replace them with an equivalent current-main patch.
