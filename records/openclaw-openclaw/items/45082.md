## Summary

Keep open. Current main still lacks the Mattermost-owned threading context needed for implicit same-thread `message` tool sends, and the cron/heartbeat route persistence path still only updates last-route metadata for DMs. The issue is also paired with open same-author closing PRs that directly target the remaining paths, so it should stay open until those PRs land or are replaced.

## What This Changes

The issue asks for proactive Mattermost `message` sends and cron/heartbeat wake replies to inherit the active thread context instead of posting at the channel root or in a DM.

## Best Possible Solution

Land or replace the Mattermost-owned fixes so the plugin supplies guarded tool-threading context for same-target sends and persists route thread IDs for thread-scoped channel sessions, with regressions for explicit `replyTo`, implicit same-thread sends, cross-channel suppression, `replyToMode`, and cron/heartbeat wake delivery.

## Reproduction Assessment

Yes. The issue gives concrete Mattermost reproduction steps, and current-main static checks show the same failure path: Mattermost does not build tool context and its send action can only use explicit reply targets.

## Solution Assessment

Yes, the repair direction is the narrow maintainable one. The fix belongs in the Mattermost plugin and route metadata seams rather than a core special case for one bundled plugin.
