## Summary

Keep open. Current main still leaves the Mattermost message-tool send path without thread-context inheritance, and the cron/heartbeat case still depends on delivery context that Mattermost only persists for direct sessions. Open PRs #52120 and #52236 explicitly target those two reported cases with closing syntax, so this issue should remain open until those PRs merge or are superseded.

## What This Changes

Mattermost proactive message sends and cron/heartbeat wake replies should default to the active Mattermost thread when the session has thread context and `replyToMode: "all"`.

## Best Possible Solution

Mattermost should add thread-aware message-tool context for send actions and persist routable delivery context, including `threadId`, for thread-scoped channel/group sessions, with regression coverage for both proactive message sends and cron/heartbeat wake delivery.
