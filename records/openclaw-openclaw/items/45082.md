## Summary

Keep open. Current main still does not provide Mattermost-specific tool threading context for implicit same-thread `message` sends, and two open closing PRs already target the reported message-tool and cron/heartbeat paths, so closing this before those PRs land would be premature.

## What This Changes

Mattermost proactive `message` sends and cron/heartbeat wake replies should inherit the active thread context for thread-scoped sessions when no explicit `replyTo` is provided.

## Best Possible Solution

Land or replace the linked Mattermost-owned fixes with guarded tool thread inheritance, a Mattermost `buildToolContext`, route-context persistence where needed, and focused regressions for explicit `replyTo`, implicit same-thread sends, cross-channel suppression, `replyToMode`, and cron/heartbeat wake delivery.

## Reproduction Assessment

Yes. The issue gives concrete Mattermost reproduction steps, and the current source provides a high-confidence static reproduction for the message-tool case: Mattermost does not build tool context, so implicit same-thread sends lack `replyToMode` and do not receive an injected reply target.

## Solution Assessment

Yes, the linked implementation direction appears to be the narrow maintainable path. This should stay in the Mattermost plugin/threading and inbound route metadata seams rather than adding a broad core special case.
