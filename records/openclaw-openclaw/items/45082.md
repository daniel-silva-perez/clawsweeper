## Summary

Keep open. Current main still leaves the Mattermost message-tool send path using only explicit `replyTo`/`replyToId`, and the Mattermost plugin still lacks the tool-context builder needed for implicit same-thread sends. The cron/heartbeat case also remains direct-only in last-route persistence. Open PRs #52120 and #52236 already target the two reported paths, so this should stay open for normal maintainer PR review instead of being closed or queued as a duplicate repair.

## What This Changes

Default Mattermost proactive message-tool sends and cron/heartbeat wake replies to the active Mattermost thread when the session is thread-scoped.

## Best Possible Solution

Review, rebase, land, or replace #52120 and #52236 with a Mattermost-owned implementation that supplies tool thread context, guardedly inherits same-target thread roots for implicit sends, persists non-direct thread delivery context where needed, and adds focused regressions for explicit `replyTo`, implicit same-thread sends, cross-channel suppression, `replyToMode`, and cron/heartbeat wake delivery.

## Reproduction Assessment

Yes. The issue provides concrete Mattermost steps, and current source gives a high-confidence static reproduction path: message-tool sends without explicit `replyTo` cannot receive Mattermost thread context, while cron/heartbeat wake routing still lacks non-direct `deliveryContext.threadId` persistence.

## Solution Assessment

Yes, the open PR direction is the narrowest maintainable fix: keep the behavior in the Mattermost plugin and the inbound session-route metadata path rather than adding a broad core special case. Maintainer review should decide whether to rebase those PRs directly or replace them with an equivalent current-main patch.
