## Summary

Keep open. Current main still leaves the core reported Mattermost message-tool path without thread-root inheritance: the Mattermost send action only forwards explicit `replyTo`/`replyToId`, and the plugin does not provide the thread-aware tool-context hook that would supply the active Mattermost thread root. The issue is also paired with open implementation PRs #52120 and #52236, so it should remain open until those PRs land or are explicitly superseded.

## What This Changes

Mattermost proactive message-tool sends and cron/heartbeat wake replies should inherit the active thread-scoped session context instead of posting at the channel root or DM fallback.

## Best Possible Solution

Land or supersede the paired Mattermost fixes so the plugin supplies thread-aware tool context for proactive sends and stores routable `deliveryContext.threadId` for thread-scoped channel/group sessions, with regression tests for both message-tool sends and cron/heartbeat wake delivery.
