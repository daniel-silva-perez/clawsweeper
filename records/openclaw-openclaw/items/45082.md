## Summary

Keep open: current main still does not provide Mattermost thread context to proactive `message` tool sends without an explicit `replyTo`, and open closing PRs already target the remaining work. The cron/heartbeat subcase has newer routing code that should be rechecked during PR review, but the central message-tool behavior is not implemented on main.

## What This Changes

Mattermost proactive message sends and wake replies should inherit the active thread-scoped session context when no explicit reply target is supplied.

## Best Possible Solution

Add a Mattermost-owned threading tool context or equivalent hook so same-target proactive sends inherit the active thread root safely, with tests covering explicit `replyTo`, same-channel implicit thread inheritance, cross-channel suppression, and any cron/heartbeat wake behavior that remains after current session-key routing.
