## Summary

Keep open. Current main has fixes for several neighboring symptoms, but I did not find a source-backed fix for this issue's remaining central failure: native Windows Scheduled Task restart can reach `ready` and then lose HTTP responsiveness or the port. The merged follow-up PR was explicitly marked `Related to #63491` while fixing #48771, and the changelog likewise says the restart policy-close work fixes #48771 only.

## Best Possible Solution

Keep this issue open and narrow it to the remaining Windows post-ready liveness failure. Treat #48771/#72660 as covering the restart probe false negative and the current cron fixes as covering stale `runningAtMs`; the next fix should add or validate Windows Scheduled Task restart coverage showing that after `ready`, `/health` or `/healthz`, `/`, WebSocket probe/status, and port ownership remain stable. If the reporter can provide the offered logs/status snapshots, attach them here or split a focused follow-up only for the post-ready HTTP/port loss.
