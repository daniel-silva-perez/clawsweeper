## Summary

Keep this PR open because current main still does not expose full conversation context on `before_tool_call`, but the proposed branch should not merge as-is: it widens conversation data access for every existing tool-call hook and is stale against newer hook-contract/fail-closed changes on main.

## What This Changes

The PR adds optional `systemPrompt`, `messages`, and `tools` fields to the `before_tool_call` hook payload and threads lazy context getters through embedded Pi tool creation, split tool definitions, and client tool wrapping, with focused tests plus incidental formatting churn.

## Best Possible Solution

Land a rebased, narrowly scoped additive hook-context change only after the new payload obeys the plugin conversation-access model, preserves fail-closed behavior when snapshot creation fails, updates the current `hook-types` contract/docs, and covers built-in, MCP/LSP, and client-tool paths with focused tests.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the PR's motivating gap: current main source and docs show `before_tool_call` does not carry the requested context, and the PR body provides an external-plugin capture path that would demonstrate the new fields on the branch.

## Solution Assessment

Is this the best way to solve the issue? No as-is; extending the existing hook is a reasonable narrow direction, but the patch should first add an explicit conversation-access gate or equivalent permission model and make snapshot failures degrade without bypassing policy enforcement.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Gate conversation snapshots behind opt-in:** `src/agents/pi-tools.before-tool-call.ts:227-229`
  - body: This unconditionally adds the effective system prompt, transcript, and tool schemas to every `before_tool_call` event. Non-bundled plugins can already register this hook without `hooks.allowConversationAccess=true`, while current conversation-bearing hooks are blocked unless that opt-in is set. As written, installing any existing tool-call hook silently grants it full conversation access; gate these fields behind the conversation-access policy or add an explicit permission before exposing them.
  - confidence: 0.89
- **[P2] Preserve hook enforcement when snapshot cloning fails:** `src/agents/pi-tools.before-tool-call.ts:224-229`
  - body: `cloneHookSnapshot` uses `structuredClone` while constructing the hook event. If a tool param, message payload, or tool schema contains a non-cloneable value, this throws before handlers run, and this branch's catch path falls through to `{ blocked: false, params }`. That skips `before_tool_call` guardrails for the very call being inspected; drop only the problematic optional snapshot field or otherwise preserve fail-closed hook enforcement.
  - confidence: 0.78
- **[P3] Omit undefined system prompts consistently:** `src/agents/pi-tools.before-tool-call.ts:227`
  - body: This spread checks whether `getSystemPrompt` exists, not whether it returns a value. If it returns `undefined`, in-process hooks see a present `systemPrompt` key while JSON-delivered hooks omit it after serialization. Read the getter once and only include the field when the result is defined.
  - confidence: 0.77

## Security Review

Status: needs_attention

Summary: The PR intentionally expands plugin data access, but the diff does not add the permission gate needed for full transcript and system-prompt exposure and can skip policy hooks if snapshot cloning throws.

Concerns:

- **[medium] Ungated conversation and system prompt exposure:** `src/agents/pi-tools.before-tool-call.ts:227`
  - body: Every existing non-bundled `before_tool_call` handler would receive full model-facing context without the `allowConversationAccess` opt-in used for other conversation hooks. That changes the plugin trust boundary and can expose sensitive transcript/system prompt data to plugins that only asked to inspect tool params.
  - confidence: 0.9
- **[medium] Snapshot clone failure can bypass guardrails:** `src/agents/pi-tools.before-tool-call.ts:224`
  - body: The added snapshot cloning happens before dispatching the hook. On this branch, a clone error is caught by the hook failure path and the tool call is allowed, which can bypass guardrail plugins relying on `before_tool_call` to block unsafe actions.
  - confidence: 0.78
