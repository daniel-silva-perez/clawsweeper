## Summary

Keep this issue open. Current main still exposes `before_model_resolve` as an override-only plugin hook and does not provide typed or runtime support for returning `block`/`blockReason` to abort model dispatch. The issueâ€™s budget-guard example is a real external-plugin use case that needs a core plugin hook contract, not ClawHub-only plugin work.

## Best Possible Solution

Add an additive core plugin API change: extend `PluginHookBeforeModelResolveResult` with `block?: boolean` and `blockReason?: string`, merge it with terminal sticky-block semantics matching `before_tool_call`, propagate a blocked model-resolution outcome through `resolveHookModelSelection` and the embedded runner before `resolveModelAsync`, surface the reason to the user, and update hook merger, runner, SDK/public docs, and focused tests.

## Risks / Open Questions

- The implementation needs a clear user-facing blocked-model result path before provider dispatch, plus additive SDK/docs/tests coverage so existing override-only plugins remain compatible.
- Throwing from hooks is not an adequate workaround because the current hook runner catches fail-open failures in this path.
