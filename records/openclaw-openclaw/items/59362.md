## Summary

Keep this issue open. Current main still routes normal gateway exec-tool commands through the shared shell config, and that config still selects PowerShell on Windows with `-NoProfile -NonInteractive -Command`. The existing `windowsHide: true` setting is present, but that is exactly the combination the report says still flashes. There is a separate direct-argv helper path for internal command execution, but the user-facing gateway exec-tool path does not use it for ordinary shell command strings.

## Best Possible Solution

Keep the issue open and implement a conservative Windows gateway exec improvement: detect simple executable argv-safe commands and spawn them directly with `windowsHide: true`, while retaining the PowerShell wrapper for commands that need shell syntax such as pipes, redirects, variable expansion, globbing, or PowerShell-only constructs. Add targeted Windows unit coverage showing simple exec calls bypass PowerShell, shell-feature commands still use the wrapper, and approval/display text remains stable.
