## Summary

Keep open. Current main still matches the core report: non-sandbox host exec command strings are built as Windows PowerShell invocations, and the prior merged windowsHide work covered lower-level exec helpers rather than adding a direct-spawn path for the exec tool.

## What This Changes

The issue asks OpenClaw to prevent Windows console flashes during exec tool runs by avoiding the PowerShell wrapper for simple commands that do not need shell features.

## Best Possible Solution

Add a Windows-aware host exec spawn planner that only bypasses PowerShell for a parsed single native executable invocation with no shell-only syntax, while keeping the existing PowerShell route for PTY runs, shell features, builtins, and ambiguous commands. Cover the planner with Windows unit tests and keep approval enforcement ahead of any spawn decision.

## Reproduction Assessment

Yes. The issue provides concrete Windows steps, and source inspection confirms the non-sandbox host exec path still produces a PowerShell -Command child argv on Windows; the only missing piece is fresh visual confirmation on a Windows desktop session.

## Solution Assessment

Unclear as written. The direct-spawn direction is likely right for simple native executables, but replacing all PowerShell use with cmd.exe or direct argv would risk breaking documented Windows shell-command behavior and the existing reason PowerShell was chosen for utility output capture.
