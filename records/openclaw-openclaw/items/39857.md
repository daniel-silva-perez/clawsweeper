## Summary

Keep open. Current main still has the override-first resolver and raw configured/persisted override paths that can report a larger effective context window than a known smaller model limit, and the same-author implementation candidate #39859 remains open and unmerged. Related #72608 may cover adjacent status behavior, but it does not replace the shared resolver and persistence fix requested here.

## What This Changes

The issue requests that effective session context-window resolution/reporting cap configured or persisted context-token overrides by a smaller known model window.

## Best Possible Solution

Keep this issue open until #39859 or an equivalent narrow fix lands. The desired end state is that positive configured or persisted contextTokens act as ceilings whenever the active model window is known, while preserving existing override behavior for unknown model windows and adding focused coverage for the resolver, reply setup, session persistence, and /status display paths.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: configure agents.defaults.contextTokens above a configured or discovered model window, then follow the resolver/session/status paths shown above; the current code returns or stores the override before applying the smaller known model limit.

## Solution Assessment

Yes. The maintainable fix belongs in the shared model-aware context resolution and the raw persistence/reporting callers; a docs-only or status-only change would leave downstream stored metadata and reply setup behavior incorrect.
