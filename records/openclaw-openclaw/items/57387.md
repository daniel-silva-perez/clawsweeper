## Summary

Keep open. Current main has session-tool visibility, an agent-to-agent allowlist, gateway authentication, provenance metadata, and plugin hooks, but it does not implement HMAC/token-based mutual authentication or cryptographic verification for `sessions_send` source identity. The related hook-system work is useful defense in depth, not a substitute for signed inter-session provenance.

## Best Possible Solution

Keep this issue open for a core design. The likely fix belongs in the `sessions_send`/Gateway agent path: define a non-forgeable session or agent credential managed by the Gateway, sign the source/target/message/idempotency/timestamp payload, verify it before accepting inter-session provenance, and add per-agent policy so orchestrators can require authenticated A2A while leaf sessions remain optional. Preserve existing visibility and `tools.agentToAgent` allowlist checks as authorization gates around the new provenance proof.

## Risks / Open Questions

- This is security-sensitive core behavior, so the design should align with the documented single-Gateway trust model instead of treating session keys as secrets.
- Plugin hooks from the related hook-system work can provide extra policy checks, but relying on prompt-visible provenance or hook convention alone would not meet the requested mutual-auth property.
