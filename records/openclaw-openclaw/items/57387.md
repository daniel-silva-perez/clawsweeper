## Summary

Keep open. Current main still has policy gates around `sessions_send` but not HMAC/token-based mutual authentication or cryptographic verification of inter-session sender identity. The latest issue discussion describes an external HMAC/gateway-mapping mitigation in production, which supports that this remains an unimplemented core hardening request rather than an obsolete report.

## Best Possible Solution

Keep this issue open for a core Gateway/session-send design. The best path is a Gateway-managed, non-forgeable inter-session credential or signature over sender, target, message, idempotency key, timestamp/nonce, and routing context; verify it before accepting `inter_session` provenance, while keeping `tools.agentToAgent`, session visibility, and plugin hooks as separate authorization and defense-in-depth layers.

## Risks / Open Questions

- Security-sensitive core design remains unresolved: session keys and prompt-visible provenance are routing/context metadata, not non-forgeable sender proof.
- A fix would likely touch Gateway protocol/session-send behavior and needs additive compatibility handling, regression tests, and docs updates.
- Existing mitigations such as loopback Gateway binding, `tools.agentToAgent`, session visibility, and plugin hooks reduce exposure but do not satisfy HMAC/token-based mutual authentication.
