## Summary

Keep open. Current main has gateway authentication, HTTP deny-listing for `sessions_send`, session-tool visibility, `tools.agentToAgent` allowlist checks, plain inter-session provenance metadata, and plugin hooks that can block tool calls. It still does not implement HMAC/token-based mutual authentication or cryptographic verification that a `sessions_send` message originated from the claimed sender session. The fresh production-context comment describes an external HMAC/gateway-mapping mitigation, which reinforces that the requested structural core behavior remains unimplemented rather than obsolete.

## Best Possible Solution

Keep this issue open for a core Gateway/session-send design. The best implementation would add a Gateway-managed, non-forgeable inter-session credential or signature over sender, target, message, idempotency key, timestamp/nonce, and relevant routing context; verify it before accepting `inter_session` provenance; and keep `tools.agentToAgent` and session visibility as authorization gates around the new provenance proof. Plugin hooks can remain defense in depth, but they should not be the only trust mechanism.
