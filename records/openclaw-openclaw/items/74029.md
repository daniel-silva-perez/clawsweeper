## Summary

Keep this PR open. Current main still sends top-level metadata for `openai-codex-responses`, and the contributor PR targets the right bug, but the proposed one-line builder change appears incomplete because the live Responses send path re-merges `turnState.metadata` immediately before `client.responses.create(...)`.

## Best Possible Solution

Keep the PR open and ask for an amendment: ensure the final payload sent to `client.responses.create(...)` cannot re-add top-level metadata for `openai-codex-responses`, while preserving metadata for regular `openai-responses` and `azure-openai-responses`. The fix likely belongs either at the final `mergeTransportMetadata` call site or in provider turn-state/payload policy, plus a regression test that exercises the post-merge send payload boundary.
