## Summary

Keep open. Current main still has a credible pre-listen startup hang surface: Gateway startup awaits early runtime and discovery before the HTTP listener is bound, and discovery still awaits each registered discovery service without a core timeout. The existing EADDRINUSE listener retry is bounded, but it runs only after that pre-listen work completes, so it does not conclusively address the reported no-/healthz, no-sidecars symptom. A second commenter linked a similar slow-startup report, which increases confidence this is worth maintainer follow-up.

## Best Possible Solution

Keep this open and make pre-listen startup robust. The likely focused fix is to prevent local discovery/Bonjour startup from blocking the HTTP listener indefinitely: either move discovery advertisement off the blocking runtime.early path or wrap each gatewayDiscoveryServices[].advertise() call in a short bounded timeout that logs and continues. Add a regression test with a never-resolving discovery service and verify the Gateway still reaches /healthz or fails with a bounded, actionable error. Also ask the reporter to retry current main/latest image with OPENCLAW_DISABLE_BONJOUR=1 to isolate the suspected path.
