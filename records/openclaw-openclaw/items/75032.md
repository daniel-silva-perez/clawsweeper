## Summary

Keep open: current main still lacks the requested public plugin config opt-out, and package-level bundled runtime dependency repair still defaults to running in the install root unless a separate execution root is explicitly supplied. The linked reports provide concrete operator impact, and the remaining work is narrow enough for a focused repair PR.

## What This Changes

The issue requests a public `plugins.installBundledRuntimeDeps` opt-out, an isolated bundled runtime dependency install default, and documentation for the resulting operator matrix.

## Best Possible Solution

Add the config field to the public plugin config contract, thread it through gateway prestage and plugin loading so `false` skips bundled runtime-dependency installs, default package-level repair to an isolated execution root with atomic replacement, and document/test the supported operator matrix.

## Reproduction Assessment

Yes for the code-level gap: source inspection shows the public config key is absent, the loader opt-out is internal, gateway prestage ignores any user opt-out, and package-level repair does not pass an isolated execution root. The production stalls and disk growth are credible linked field evidence, but they were not live-reproduced here.

## Solution Assessment

Yes. The requested direction is the narrowest maintainable fix because it exposes existing internal seams and changes the runtime-dependency owner path instead of adding another environment-only workaround.
