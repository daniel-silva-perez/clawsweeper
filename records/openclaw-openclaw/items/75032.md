## Summary

Keep open: current main still lacks the public plugin config opt-out and still runs package-level bundled runtime dependency repair without an isolated execution root by default. The issue is concrete, supported by linked field reports, and narrow enough for a focused repair PR.

## What This Changes

Expose a `plugins.installBundledRuntimeDeps: false` config escape hatch, default bundled runtime dependency repair to an isolated execution root, and document/test the resulting operator matrix.

## Best Possible Solution

Add the config field to the public plugin config contract, thread it through gateway prestage and plugin loading so `false` skips bundled runtime dependency installs, default package-level repair to an isolated execution root with atomic replacement, and update docs plus regression tests for opt-out, isolated default, and legacy/shared-root compatibility if retained.

## Reproduction Assessment

Yes for the code-level gap: current source shows the config key is absent, gateway prestage has no user opt-out, and package-level repair defaults execution to the install root. The real deployment stalls and disk bloat are credible linked field evidence but were not live-reproduced here.

## Solution Assessment

Yes. Exposing the existing loader opt-out and using the existing isolated installer seam is the narrowest maintainable direction, provided the PR preserves dependency-contract behavior and documents any explicit legacy/shared-root mode.
