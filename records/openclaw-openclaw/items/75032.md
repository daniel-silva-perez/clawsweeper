## Summary

Keep open: current main still lacks the requested public plugin config opt-out and still defaults package-level bundled runtime-dependency repair to the shared install root unless a caller explicitly passes an isolated execution root. The linked field reports make the operator problem concrete enough for a focused repair lane, and no open paired fix PR was found.

## What This Changes

Expose a `plugins.installBundledRuntimeDeps: false` config escape hatch, make bundled plugin runtime dependency staging use an isolated execution root by default, and document the resulting operator matrix.

## Best Possible Solution

Add the config field to the public plugin config contract, thread it through gateway prestage and plugin loading so `false` skips bundled runtime-dependency installs, default package-level repair to an isolated execution root with atomic replacement, and publish tests plus docs for the supported operator matrix.

## Reproduction Assessment

Yes for the code-level gap: source inspection shows the config key is absent, the loader opt-out is internal, gateway prestage ignores user config, and package-level repair does not pass an isolated execution root. The production stalls and disk bloat remain field-reported rather than live-reproduced here.

## Solution Assessment

Yes. Exposing the existing opt-out seam and moving package-level repair to the existing isolated installer path is narrower than adding another environment-only workaround, provided the PR keeps dependency-install safety controls intact.
