## Summary

Keep open: current main still does not expose the requested public config opt-out, still pre-stages missing bundled runtime deps at gateway startup without a user-configured skip path, and still uses the shared package-level install root unless the install is already plugin-root scoped. The linked production reports and the newer Windows/channel comment make this concrete enough for a focused repair lane rather than cleanup closure.

## What This Changes

Expose an operator config opt-out for bundled runtime dependency installs, default bundled runtime-dep repair to an isolated execution root, cover the channel startup surface, and document the supported behavior matrix.

## Best Possible Solution

Add the public config contract and docs for disabling bundled runtime-dep installs, thread that value through gateway pre-stage and plugin/channel loading, default package-level runtime-dep repair to an isolated execution root with atomic replacement, and add regression coverage for plugin and channel startup selection so disabled or unmanaged deps do not trigger lazy installs.

## Reproduction Assessment

Yes for the code-level gap: current source inspection shows the config field is absent, the loader opt-out is internal, gateway pre-stage has no skip path, and package-level repair does not default to an isolated execution root. The operational stalls and disk growth are supported by linked field reports but were not live-reproduced here.

## Solution Assessment

Yes, with one channel-scope clarification. Exposing the existing internal install opt-out and changing the runtime-deps owner path to isolated execution is narrower and more maintainable than adding another environment-only workaround; the Windows channel report should be handled in the same contract or a clearly documented sibling channel flag.
