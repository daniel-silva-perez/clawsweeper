## Summary

Keep open: current main still lacks the public plugin config opt-out and does not default package-level bundled runtime dependency repair to an isolated execution root, while the linked field reports make this concrete enough for a focused repair lane.

## What This Changes

Expose a public `plugins.installBundledRuntimeDeps: false` opt-out, default bundled plugin runtime dependency installs to isolated execution roots, and document the operator behavior matrix.

## Best Possible Solution

Add the config field to the public plugin config contract, thread it through gateway prestage and plugin loading so `false` skips bundled runtime dependency installs, make package-level repair use an isolated execution root with atomic replacement by default, and publish docs plus regression tests for the supported operator matrix.

## Reproduction Assessment

Yes for the code-level gap: source inspection shows the config key is absent, the loader opt-out is internal, gateway prestage ignores any user opt-out, and package-level repair does not pass an isolated execution root. The production stalls and disk growth are credible linked field evidence, but they were not live-reproduced here.

## Solution Assessment

Yes. The requested direction is the narrowest maintainable fix because it exposes existing internal seams and changes the installer default in the runtime-dependency owner path instead of adding another environment-only workaround.
