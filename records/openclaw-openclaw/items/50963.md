## Summary

Keep open. Current main still creates a fresh native subagent session key for each `sessions_spawn` native run, has no `reuse`/TTL/session-target parameter, and still gates native `mode="session"` behind thread binding hooks. Teams advertises thread-capable chat routing but does not register `subagent_spawning`/delivery/end hooks. The KeepMyClaw comment is useful operational backup context, and #67000 plus open #39372 overlap, but neither fully supersedes this Teams/expert-agent reuse request on current main.

## What This Changes

Request session reuse for expert subagents, especially Teams-spawned agents, instead of creating a fresh `sessions_spawn(mode="run")` child session on every spawn.

## Best Possible Solution

Keep the request visible until maintainers choose the boundary. A maintainable end state would either add Teams-owned subagent binding hooks with delivery/end cleanup and docs, or introduce a generic native `sessions_spawn` reuse target with bounded TTL, conflict handling, and isolation guarantees that works without channel-specific thread hooks.

## Reproduction Assessment

Yes. A code-level reproduction is high confidence: two native `sessions_spawn` calls with the same `agentId` will allocate different `agent:<id>:subagent:<uuid>` keys, and Teams has no registered `subagent_spawning` hook for the thread-bound session path.

## Solution Assessment

Unclear. The issue’s preferred core registry reuse may be the broader fix, but a Teams hook is the smaller channel-specific fix; open #39372 is a third partial approach that may unlock headless persistent sessions without automatic reuse.
