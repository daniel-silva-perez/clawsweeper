## Summary

Keep open. Current main has useful generic Active Memory timeout mitigations, including a hard deadline and repeated-timeout circuit breaker, but it still does not prove the reported Cerebras `gpt-oss-120b` reasoning-model path is fixed or that provider work stops cleanly after timeout. The prior ClawSweeper comment reached the same conclusion, and the current checkout still recommends the affected model in docs while only covering mocked timeout behavior.

## What This Changes

Active Memory should handle `cerebras/gpt-oss-120b` timeouts without pinning gateway CPU or degrading health/admin probes.

## Best Possible Solution

Keep this as a focused Active Memory/Cerebras regression. The best fix is to reproduce or accurately simulate `gpt-oss-120b` reasoning/tool-call output in the embedded recall path, prove timeout abort cancels underlying provider/session work, preserve gateway health/admin responsiveness after timeout, and revise the Active Memory docs if this model is not currently safe as a recommended fast recall option.

## Reproduction Assessment

Yes, with Cerebras credentials and the reported Discord/gateway setup: configure Active Memory to `cerebras/gpt-oss-120b`, send an eligible channel message, then watch Active Memory logs, gateway CPU, `/health`, and websocket/admin probes. I did not run it live here because no Cerebras key was present.

## Solution Assessment

No, the current main implementation is only a generic mitigation. The hard timeout and circuit breaker reduce repeated stalls, but they do not prove the model-specific provider stream is aborted or that gateway health recovers after the Cerebras reasoning-model timeout.
