## Summary

Keep open. Current main still has the documented Active Memory plus Cerebras gpt-oss-120b setup that the report targets, and the source only proves generic timeout mitigation, not that the Cerebras reasoning stream aborts cleanly or that gateway health remains responsive after the timeout.

## What This Changes

The issue reports that Active Memory using `cerebras/gpt-oss-120b` times out well beyond the configured budget and can leave the local gateway CPU-bound, despite direct Cerebras chat/completions calls succeeding.

## Best Possible Solution

Keep this as a focused Active Memory/Cerebras regression until maintainers can reproduce or simulate the gpt-oss-120b reasoning/tool stream, prove abort cleanup and gateway responsiveness during and after timeout, and either fix the provider/runtime path or qualify the Cerebras Active Memory recommendation.

## Reproduction Assessment

Yes. The reporter supplied exact config, logs, direct Cerebras API controls, and recovery steps; this read-only review did not live-run Cerebras, so the next proof should be a live run or provider-harness simulation of the same stream and abort behavior.

## Solution Assessment

Unclear. The current watchdog and circuit breaker are useful mitigations, but the best fix must prove that the embedded Cerebras gpt-oss path cancels provider work and preserves gateway health, not only that later recalls are skipped.
