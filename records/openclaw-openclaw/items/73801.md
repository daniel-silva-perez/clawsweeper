## Summary

Keep open. Current main has useful Active Memory timeout mitigations, including an aborting watchdog and a per-agent/model circuit breaker, but it still does not prove the reported Cerebras gpt-oss-120b reasoning-response path aborts cleanly or preserves gateway health. The public docs still recommend the affected model for Active Memory, so this remains an actionable reliability regression rather than cleanup.

## What This Changes

The issue reports that Active Memory using Cerebras gpt-oss-120b can exceed the configured timeout and leave the local gateway CPU-bound or slow to answer health/admin probes.

## Best Possible Solution

Keep this as a focused Active Memory/Cerebras regression until a maintainer can reproduce or simulate gpt-oss-120b reasoning/tool-call streaming in the embedded recall path, prove provider aborts stop the underlying work, assert gateway health remains responsive during and after timeout, and update the Active Memory recommendation if the model is not safe for recall.

## Reproduction Assessment

Yes for the reporter path: the issue includes concrete config, steps, logs, direct API controls, and post-disable recovery observations. This review did not live-run Cerebras, so the high-confidence maintainer path is to rerun that exact setup or add a provider-harness simulation of a reasoning-only/tool-call stream that respects or ignores abort.

## Solution Assessment

Unclear. The current circuit breaker is a useful mitigation for repeated timeouts, but the best fix still needs to prove clean abort and gateway responsiveness for the Cerebras reasoning-model path rather than only skipping later recalls.
