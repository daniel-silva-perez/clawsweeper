## Summary

Keep open. Current main has useful Active Memory timeout mitigations, including an aborting watchdog and a per-agent/model circuit breaker, but it still does not prove the reported Cerebras gpt-oss-120b reasoning-response path aborts cleanly or preserves gateway health. The docs still recommend the affected model for Active Memory, and the open setup-grace PR would only address one adjacent timeout-boundary mismatch, so this remains an actionable reliability regression.

## What This Changes

The issue reports that Active Memory using Cerebras gpt-oss-120b can exceed the configured timeout and leave the local gateway CPU-bound or slow to answer health and admin probes.

## Best Possible Solution

Keep this as a focused Active Memory/Cerebras regression until a maintainer can reproduce or simulate gpt-oss-120b reasoning/tool-call streaming in the embedded recall path, prove provider aborts stop the underlying work, assert gateway health remains responsive during and after timeout, and update the Active Memory recommendation if the model remains unsafe for recall.

## Reproduction Assessment

Yes for the reporter path: the issue gives exact config, steps, logs, direct API controls, and recovery observations. This review did not live-run Cerebras, so the maintainer reproduction path is to rerun that setup or add a high-fidelity provider harness for a reasoning-only/tool-call stream under abort.

## Solution Assessment

Unclear. The current circuit breaker and the open setup-grace PR are useful mitigations for timeout families, but the best fix still needs to prove clean abort and gateway responsiveness for the Cerebras reasoning-model path rather than only skipping later recalls or extending one timeout budget.
