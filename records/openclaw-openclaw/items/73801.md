## Summary

Keep open. Current main has useful generic Active Memory timeout mitigations, but it still recommends `cerebras/gpt-oss-120b` for Active Memory and does not prove that this specific Cerebras reasoning-model path aborts cleanly or preserves gateway responsiveness. The report is recent, concrete, model-specific, and still has no source/docs/test evidence that would support a conservative cleanup close.

## What This Changes

The issue reports that Active Memory configured with `cerebras/gpt-oss-120b` can exceed its configured timeout, return no summary, and leave the local gateway CPU-bound even though direct Cerebras chat/completions calls succeed.

## Best Possible Solution

Keep this as a focused Active Memory/Cerebras regression until maintainers can reproduce it live or simulate the gpt-oss reasoning/tool-call stream in a provider harness, then either fix embedded recall abort cleanup and gateway responsiveness or qualify/remove the Cerebras Active Memory recommendation.

## Reproduction Assessment

Yes. The reporter supplied a high-confidence reproduction path with exact config, logs, direct Cerebras controls, and recovery steps; this review did not live-run Cerebras, so the next proof should rerun that setup or add a faithful provider-harness simulation.

## Solution Assessment

Unclear. The current watchdog, partial-timeout recovery, and circuit breaker are good mitigations, but the best fix needs provider-specific proof that the underlying Cerebras work stops on timeout and leaves `/health` plus websocket/admin probes responsive.
