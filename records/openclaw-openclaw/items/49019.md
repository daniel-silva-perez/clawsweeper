## Summary

Keep open. Current main still exposes POST /hooks/agent as async-only ingress: the payload, normalizer, HTTP response, dispatcher, docs, and tests do not include an opt-in wait/result-return mode. Open PR #67433 is a direct implementation candidate and remains unmerged in the provided context, so this issue should stay open for maintainer review.

## Best Possible Solution

Keep this issue open and review PR #67433, or an equivalent maintainer-owned implementation. The accepted change should add opt-in blocking/result-return behavior to POST /hooks/agent, keep current async runId-only behavior by default, return a narrow deterministic status/result shape, document the new fields, and add focused gateway hook tests for default async behavior, blocking success/error/timeout, idempotency, delivery suppression, and hook security boundaries.

## Risks / Open Questions

- The requested change is security-sensitive because POST /hooks/agent is an external authenticated ingress; any implementation needs to preserve hook token checks, hooks.allowedAgentIds, hooks.allowRequestSessionKey, hooks.allowedSessionKeyPrefixes, idempotency behavior, and the current HTTP deny posture for sessions_send.
- A blocking/result mode needs explicit timeout, error/status, duplicate retry, and delivery-suppression semantics so callers do not accidentally create long-lived HTTP work or duplicate main-session summaries.
