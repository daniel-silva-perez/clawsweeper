## Summary

Keep open. Current main still implements POST /hooks/agent as async-only ingress: the payload and docs do not expose waitForResult/resultMode/blocking/announceToMain fields, the HTTP handler returns only { ok: true, runId }, and the gateway dispatcher launches the isolated turn in a fire-and-forget async closure. Open PR #67433 is a direct, unmerged implementation candidate, so this issue should stay open until that PR or an equivalent maintainer-reviewed core change lands.

## Best Possible Solution

Keep this issue open and review PR #67433, or an equivalent maintainer-reviewed implementation, as the candidate path. The core change should add opt-in blocking/result-return behavior to POST /hooks/agent, return a narrow deterministic result/status, keep current async runId-only behavior by default, document the request/response fields, and add focused gateway hook tests for default async mode, blocking success/error/timeout, idempotency, delivery suppression, and security policy boundaries.

## Risks / Open Questions

- The remaining change is security-sensitive because /hooks/agent is an external authenticated ingress; any implementation needs to preserve hook token checks, agent/session allowlists, idempotency semantics, default async behavior, and the current HTTP deny posture for sessions_send.
- A blocking/result mode also needs explicit timeout, error, and delivery-suppression semantics so callers do not accidentally create long-lived HTTP work or duplicate main-session summaries.
