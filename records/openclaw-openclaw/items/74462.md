## Summary

Keep open. Current main still rejects the requested durable config keys, and the reported runtime support is not present in current source, docs, or command registration, so this needs a maintainer product/architecture decision rather than closure or a schema-only sweep.

## What This Changes

The issue requests a durable context-safety config surface, preferably `session.contextSafety`, with matching schema, exported types, generated config metadata, docs, and tests.

## Best Possible Solution

Decide the intended product contract first. If context safety is intended, implement it end to end across runtime consumption, schema, exported config types, generated config help/labels, docs, and focused validation/runtime tests; if not, document the supported `agents.defaults.compaction` controls and explain or remove any package artifact drift that suggests `contextSafety` is supported.

## Reproduction Assessment

Yes for the validation failure: current main's strict session and root schemas reject both `session.contextSafety` and top-level `contextSafety`. The runtime-support half is unclear because the reported symbols and slash commands are absent from current source and docs.

## Solution Assessment

Unclear. A schema-only addition is not the best fix unless maintainers confirm the runtime contract; the safer path is an end-to-end config/runtime/docs decision.
