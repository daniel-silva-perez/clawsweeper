## Summary

Keep open. Current main still has no durable `contextSafety` config contract: `session` is strict without that field, the root schema is strict with no top-level alias, normal validation goes through `OpenClawSchema.safeParse`, and `SessionConfig` omits the type. I also found no current source/docs/tests symbol for the reported `contextSafety` runtime or named rollover/handoff commands, so this needs maintainer product confirmation rather than a schema-only cleanup.

## What This Changes

The issue requests a durable config path for context-safety settings because `session.contextSafety` and top-level `contextSafety` are rejected even though the reporter observed a runtime artifact reading those keys.

## Best Possible Solution

Maintainers should decide the intended contract first. If context safety is supported, implement it end to end with schema, exported types, generated config metadata/docs, validation coverage, and runtime tests proving the durable config is consumed; otherwise document the existing compaction/session controls and remove or avoid any generated alias that suggests unsupported config.

## Reproduction Assessment

Yes for the schema rejection path: the current strict `SessionSchema`, strict root `OpenClawSchema`, and validator call make `session.contextSafety` and top-level `contextSafety` reproducibly invalid. The runtime-support half is unclear because current source/docs/tests and `v2026.4.27` lack the referenced symbols.

## Solution Assessment

Unclear. A schema-only fix is not the best path unless maintainers confirm or add the runtime behavior; the safer solution is to align schema, types, docs, generated metadata, and runtime support in one product decision.
