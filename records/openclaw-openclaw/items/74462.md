## Summary

Keep open. Current main still rejects the requested durable config paths, and the referenced context-safety runtime/command surface is not present in the current source, docs, tests, or release-tag source search. The existing ClawSweeper comment reached the same conclusion, and this pass against current main still needs maintainer product/runtime confirmation before accepting new config keys.

## What This Changes

The issue requests a durable `contextSafety` config contract, preferably under `session.contextSafety`, with matching schema, TypeScript types, config metadata, docs, and an explicit decision on a top-level legacy alias.

## Best Possible Solution

Decide the intended product contract first. If context safety is supported, add an end-to-end implementation: a `ContextSafetyConfigSchema`, `session.contextSafety?: ContextSafetyConfig`, public config types, schema help/labels/generated metadata, docs, validation tests, and runtime tests proving the durable config is consumed. If it is not intended, document the supported compaction/session reset controls and clean up any generated-package artifact that suggests otherwise.

## Reproduction Assessment

Yes. The schema rejection path is reproducible from source: `session.contextSafety` is outside `SessionSchema.strict()`, top-level `contextSafety` is outside the strict root schema, and normal raw config validation calls `OpenClawSchema.safeParse(normalizedRaw)`.

## Solution Assessment

Unclear. The reporter's suggested schema/type additions are likely part of a fix only if maintainers confirm the runtime contract; accepting the keys without runtime consumption would not be the narrowest maintainable solution.
