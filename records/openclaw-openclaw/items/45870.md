## Summary

Keep open. The test cleanup goal is useful and current main already has the Windows `taskkill.exe` cleanup behavior, but this PR still changes home-prefix expansion and executable-resolution semantics in ways that introduce concrete regressions. It should not be closed as implemented or merged as-is.

## What This Changes

The PR updates Windows path and scheduled-task unit test expectations while changing executable lookup, home-prefix expansion, and stable Homebrew Node path resolution helpers.

## Best Possible Solution

Keep or reapply the scheduled-task assertion cleanup, but revise or split the runtime helper changes so home expansion preserves unmatched tilde tokens, PATHEXT only drives suffix search, and stable Node path fallback casing stays internally consistent with focused regression tests.

## Reproduction Assessment

Yes. There are direct code-level reproduction paths: under the PR, `expandHomePrefix("~alice/bin/tool", { home: "/home/me" })` can become cwd-anchored, and a Windows PATH lookup for an existing `node.exe` can fail when the provided env narrows `PATHEXT` to `.CMD`.

## Solution Assessment

No. The test-normalization direction is reasonable, but the proposed branch is broader than the narrowest maintainable fix because it changes executable and approval-path semantics while aligning Windows assertions.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve unmatched tilde tokens:** `src/infra/home-dir.ts:73`
  - body: Wrapping the replacement in `path.resolve(...)` also resolves inputs the regex intentionally leaves untouched. A value like `~alice/bin/tool` becomes a cwd-anchored path, and callers use this helper for executable and allowlist paths, so an unresolved token can silently target a different filesystem path.
  - confidence: 0.92
- **[P2] Bypass PATHEXT checks for explicit executable files:** `src/infra/executable-path.ts:41`
  - body: Passing the caller env into `isExecutableFile` makes Windows reject an already-found file when its extension is not in PATHEXT. With `PATHEXT=.CMD`, a concrete `node.exe` path can return `undefined` even though the executable exists; PATHEXT should drive extension search, not invalidate qualified paths.
  - confidence: 0.88
- **[P3] Normalize formula casing before fallback:** `src/infra/stable-node-path.ts:32`
  - body: The Cellar matcher is case-insensitive, but the default formula fallback still checks `formula === "node"`. A path that only matched because of the `i` flag, such as `Cellar/Node/.../node.exe`, skips the prefix bin fallback.
  - confidence: 0.62

## Security Review

- **[low] Home-prefix expansion can retarget approval paths:** `src/infra/home-dir.ts:73`
  - body: The proposed `path.resolve(...)` wrapper can turn an unmatched `~user/...` token into a cwd-relative absolute path. Because this helper feeds exec allowlist and command-resolution paths, that can change what an approved pattern or command target refers to.
