## Summary

Keep open. The scheduled-task test direction matches current runtime behavior, but this contributor PR also changes runtime path helpers and still has concrete P2 regressions in home-prefix expansion and Windows PATHEXT handling. It is also linked to still-open watchdog work, so it needs maintainer review or a refreshed branch rather than cleanup closure.

## What This Changes

The PR updates Windows scheduled-task and path-related unit tests while changing executable lookup, home-prefix expansion, and stable Homebrew Node path resolution helpers.

## Best Possible Solution

Keep the Windows scheduled-task assertion cleanup, but revise or split the runtime helper changes so home expansion preserves unmatched tilde tokens, PATHEXT only drives suffix search, and stable Node path fallback casing is internally consistent with focused regression tests.

## Reproduction Assessment

Yes. The blocking issues have direct code-level reproduction paths: under the PR, expandHomePrefix("~alice/bin/tool", { home: "/home/me" }) can become cwd-anchored, and a Windows PATH lookup for an existing node.exe can fail when env PATHEXT is .CMD.

## Solution Assessment

No. The test-normalization goal is reasonable, but the proposed branch is broader than the narrowest maintainable fix because it changes executable and approval-path semantics while aligning Windows assertions.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve unmatched tilde tokens:** `src/infra/home-dir.ts:73`
  - body: Wrapping the replacement in path.resolve(...) also resolves inputs the regex intentionally leaves untouched. A value like ~alice/bin/tool becomes a cwd-anchored path, and callers use this helper for executable and allowlist paths, so an unresolved token can silently target a different filesystem path.
  - confidence: 0.92
- **[P2] Bypass PATHEXT checks for explicit executable files:** `src/infra/executable-path.ts:41`
  - body: Passing the caller env into isExecutableFile makes Windows reject an already-found file when its extension is not in PATHEXT. With PATHEXT=.CMD, a concrete node.exe path can return undefined even though the executable exists; PATHEXT should drive extension search, not invalidate qualified paths.
  - confidence: 0.88
- **[P3] Normalize formula casing before fallback:** `src/infra/stable-node-path.ts:32`
  - body: The Cellar matcher is case-insensitive, but the default formula fallback still checks formula === "node". A path that only matched because of the i flag, such as a Cellar/Node/.../node.exe path, skips the prefix bin fallback.
  - confidence: 0.62

## Security Review

Status: needs_attention

Summary: No workflow, dependency, lockfile, or release-script changes were found, but the diff changes executable-resolution and approval-path semantics.

Concerns:

- **[low] Home-prefix expansion can retarget approval paths:** `src/infra/home-dir.ts:73`
  - body: The proposed path.resolve(...) wrapper can turn an unmatched ~user/... token into a cwd-relative absolute path. Because this helper feeds exec allowlist and command-resolution paths, that can change what an approved pattern or command target refers to.
  - confidence: 0.82
- **[low] PATHEXT gating can deny explicit executables:** `src/infra/executable-path.ts:41`
  - body: Forwarding caller PATHEXT into isExecutableFile makes explicit files depend on extension-search policy. That can break host exec resolution under restricted environments and create surprising allowlist misses for concrete binaries.
  - confidence: 0.78
