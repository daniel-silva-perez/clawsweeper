## Summary

Keep open. The scheduled-task test cleanup matches current runtime behavior, but the PR still makes runtime helper changes that introduce executable-resolution and home-prefix regressions; the existing review comments from Greptile and ClawSweeper point at the same risky areas, and current main does not make the whole branch redundant.

## What This Changes

The PR updates Windows-specific scheduled-task and path-related unit tests while modifying executable lookup, home-prefix expansion, and stable Homebrew Node path resolution helpers.

## Best Possible Solution

Refresh or split the branch: keep the Windows scheduled-task assertion cleanup, but revise the runtime helper changes so home-prefix expansion only resolves matched home syntax, PATHEXT drives suffix search rather than explicit-file validation, and stable Node path fallback casing is internally consistent with focused regression tests.

## Reproduction Assessment

Yes. The blocking findings have code-level reproduction paths: under the PR, expandHomePrefix("~alice/bin/tool", { home: "/home/me" }) can become a cwd-anchored path, and a Windows PATH lookup for an existing node.exe can fail when the provided env has PATHEXT=.CMD.

## Solution Assessment

No. The test-normalization direction is reasonable, but the proposed runtime changes are broader than needed and regress current approval/executable semantics; a narrower patch should preserve current helper contracts while updating only the platform-specific expectations that are truly wrong.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve unmatched tilde tokens:** `src/infra/home-dir.ts:73`
  - body: Wrapping the replacement in path.resolve(...) also resolves inputs the regex intentionally leaves untouched. A value like ~alice/bin/tool becomes a cwd-anchored path, and callers use this helper for executable and allowlist paths, so an unresolved token can silently target a different filesystem path.
  - confidence: 0.92
- **[P2] Do not gate explicit files on PATHEXT:** `src/infra/executable-path.ts:41`
  - body: Passing the caller env into isExecutableFile makes Windows reject an already-found file when its extension is not in PATHEXT. With PATHEXT=.CMD, a concrete node.exe path can return undefined even though the executable exists; PATHEXT should drive extension search, not invalidate qualified paths.
  - confidence: 0.88
- **[P3] Normalize formula casing before the fallback:** `src/infra/stable-node-path.ts:32`
  - body: The Cellar matcher is case-insensitive, but the default formula fallback still checks formula === "node". A path that only matched because of the i flag, such as a Cellar/Node/.../node.exe path, skips the <prefix>/bin/node.exe fallback.
  - confidence: 0.62

## Security Review

Status: needs_attention

Summary: No workflow, dependency, lockfile, or release-script changes were found, but the runtime exec path changes affect approval and executable-resolution semantics.

Concerns:

- **[low] Home-prefix expansion can retarget approval paths:** `src/infra/home-dir.ts:73`
  - body: The proposed path.resolve(...) wrapper can turn an unmatched ~user/... token into a cwd-relative absolute path. Because this helper feeds exec allowlist and command-resolution paths, that can change what an approved pattern or command target refers to.
  - confidence: 0.82
- **[low] PATHEXT gating can deny explicit executables:** `src/infra/executable-path.ts:41`
  - body: Forwarding caller PATHEXT into isExecutableFile makes explicit files depend on extension-search policy. That can break host exec resolution under restricted environments and create surprising allowlist misses for concrete binaries.
  - confidence: 0.78
