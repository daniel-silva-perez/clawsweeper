## Summary

Keep open. The Windows test cleanup is useful, but the branch should not merge as-is because it changes executable and path helper semantics in ways that can retarget unresolved paths and reject already-qualified Windows executables.

## What This Changes

The branch updates scheduled-task and path-related Windows assertions while changing home-prefix expansion, Windows executable/PATHEXT checks, and Homebrew stable Node path matching.

## Best Possible Solution

Keep the scheduled-task assertion cleanup and Windows-friendly test normalization, but split out or revise the helper changes so tilde expansion stays replacement-only, explicit executable paths are validated independently from caller PATHEXT search policy, and stable-node fallback casing is covered by focused regression tests.

## Reproduction Assessment

Yes. The PR diff gives helper-level reproduction paths: `expandHomePrefix("~alice/bin/tool", { home: "/home/me" })` would become cwd-absolute, and a concrete Windows `.exe` can be rejected when caller PATHEXT is narrowed to a different extension.

## Solution Assessment

No. The test-normalization goal is reasonable, but this branch is broader than the narrowest maintainable fix because it changes security-adjacent executable and approval-path semantics while updating assertions.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve unmatched tilde tokens:** `src/infra/home-dir.ts:73`
  - body: Wrapping the replacement in `path.resolve(...)` also resolves inputs the regex intentionally leaves untouched. A value like `~alice/bin/tool` becomes cwd-anchored, and callers use this helper for executable and allowlist paths, so an unresolved token can silently target a different filesystem path.
  - confidence: 0.92
- **[P2] Bypass PATHEXT checks for already-qualified executables:** `src/infra/executable-path.ts:41`
  - body: Passing the caller env into `isExecutableFile` makes Windows reject an already-found file when its extension is not in PATHEXT. With `PATHEXT=.CMD`, a concrete `node.exe` path can return `undefined` even though the executable exists; PATHEXT should drive suffix search, not invalidate qualified paths.
  - confidence: 0.88
- **[P3] Normalize formula casing before fallback:** `src/infra/stable-node-path.ts:32`
  - body: The Cellar matcher is case-insensitive, but the default formula fallback still checks `formula === "node"`. A path that only matched because of the `i` flag, such as `Cellar/Node/.../node.exe`, skips the prefix bin fallback.
  - confidence: 0.62

## Security Review

Status: needs_attention

Summary: No workflow, dependency, lockfile, or release-script change was found, but the diff changes executable-resolution and approval-path semantics.

Concerns:

- **[low] Home-prefix expansion can retarget approval paths:** `src/infra/home-dir.ts:73`
  - body: The proposed `path.resolve(...)` wrapper can turn an unmatched `~user/...` token into a cwd-relative absolute path. Because this helper feeds exec allowlist and command-resolution paths, that can change what an approved pattern or command target refers to.
  - confidence: 0.82
- **[low] PATHEXT gating can deny explicit executables:** `src/infra/executable-path.ts:41`
  - body: Forwarding caller PATHEXT into `isExecutableFile` makes explicit files depend on extension-search policy. That can break host exec resolution under restricted environments and create surprising allowlist misses for concrete binaries.
  - confidence: 0.78
