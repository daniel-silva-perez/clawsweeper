## Summary

Keep open. The scheduled-task test direction matches current main's Windows `taskkill.exe` cleanup behavior, but the PR still introduces blocking regressions in home-prefix expansion and executable resolution, so it is not safe to merge as-is.

## What This Changes

The PR updates Windows scheduled-task and path-related tests while changing home-prefix expansion, Windows executable/PATHEXT handling, and stable Homebrew Node path resolution helpers.

## Best Possible Solution

Keep or reapply the scheduled-task assertion cleanup and Windows-friendly test normalization, but leave `expandHomePrefix` replacement-only, keep explicit executable file validation independent of caller PATHEXT, and add focused regression tests for those helper contracts before merging.

## Reproduction Assessment

Yes. The PR diff gives direct helper-level reproduction paths: `expandHomePrefix("~alice/bin/tool", { home: "/home/me" })` is retargeted through `path.resolve`, and an explicit `.exe` can be rejected when the caller env narrows PATHEXT to another extension.

## Solution Assessment

No. The test-normalization goal is reasonable, but this branch is broader than the narrowest maintainable fix because it changes security-adjacent helper semantics while updating Windows assertions.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve unmatched tilde tokens:** `src/infra/home-dir.ts:73`
  - body: Wrapping the replacement in `path.resolve(...)` also resolves inputs the regex intentionally leaves untouched. A value like `~alice/bin/tool` becomes cwd-anchored, and callers use this helper for executable and allowlist paths, so an unresolved token can silently target a different filesystem path.
  - confidence: 0.92
- **[P2] Bypass PATHEXT checks for explicit executable files:** `src/infra/executable-path.ts:41`
  - body: Passing the caller env into `isExecutableFile` makes Windows reject an already-found file when its extension is not in PATHEXT. With `PATHEXT=.CMD`, a concrete `node.exe` path can return `undefined` even though the executable exists; PATHEXT should drive suffix search, not invalidate qualified paths.
  - confidence: 0.88
- **[P3] Normalize formula casing before fallback:** `src/infra/stable-node-path.ts:32`
  - body: The Cellar matcher is case-insensitive, but the default formula fallback still checks `formula === "node"`. A path that only matched because of the `i` flag, such as `Cellar/Node/.../node.exe`, skips the prefix bin fallback.
  - confidence: 0.62

## Security Review

Concerns:

- **[low] Home-prefix expansion can retarget approval paths:** `src/infra/home-dir.ts:73`
  - body: The proposed `path.resolve(...)` wrapper can turn an unmatched `~user/...` token into a cwd-relative absolute path. Because this helper feeds exec allowlist and command-resolution paths, that can change what an approved pattern or command target refers to.
