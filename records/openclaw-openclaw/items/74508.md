## Summary

Keep open: current main still has the Telegram preview overflow stop behavior, and this draft PR addresses a valid reasoning-streaming gap, but the latest head has blocking cleanup and 429 retry regressions. The related #71817 config PR is complementary and does not supersede this runtime streaming fix.

## What This Changes

The PR changes Telegram draft streaming to chain overlong reasoning previews, label continuation messages, throttle shared answer/reasoning draft lanes with 429 backoff, suppress partial thinking-tag answer leakage, and add Telegram tests.

## Best Possible Solution

Revise this PR so intentional reasoning overflow chunks are retained separately from stale/superseded preview cleanup, and so 429 retry recovery cannot replace newer pending preview text with an older failed snapshot. Add focused Telegram regressions for both behaviors before running the changed gate.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Retain intentional overflow chunks:** `extensions/telegram/src/draft-stream.ts:379-382`
  - body: The overflow path now reports a completed reasoning chunk through `onSupersededPreview`, but dispatch treats reasoning previews from that callback as archived cleanup targets and deletes them at the end of the turn. A long reasoning stream would briefly chain and then lose the completed chunks except the active last one, so the overflow path needs a retention signal or separate callback.
  - confidence: 0.91
- **[P2] Preserve newer pending text on 429 retries:** `extensions/telegram/src/draft-stream.ts:479`
  - body: On a 429 this callback returns `false`, and the shared draft loop responds by restoring the failed text snapshot into `pendingText`. If another stream update arrived while the Telegram request was in flight, that newer pending text is overwritten and the retry can resend stale preview content. Preserve the newer pending value or use a retry path that does not clobber it.
  - confidence: 0.84
