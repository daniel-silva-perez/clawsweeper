## Summary

Keep open. The PR addresses a real current-main gap, but the latest branch still has a blocking correctness regression: the new overflow offset survives public fresh-message rotations, so later independent Telegram previews can be sliced from a stale offset. The existing ClawSweeper comment already called out this defect on the current head, and the latest diff still shows no reset path.

## What This Changes

The PR changes Telegram draft streaming to split overflowing previews across chained messages, adds reasoning-continuation rendering, scales draft throttles when answer and reasoning lanes run concurrently, and suppresses partial reasoning closing-tag answer flicker with tests.

## Best Possible Solution

Revise the branch so overflow-internal rotations preserve the active split offset, while public fresh-message rotations reset the offset for a new logical preview; add a regression that overflows once, calls forceNewMessage(), then sends a new short update.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Reset overflow offsets for fresh preview rotations:** `extensions/telegram/src/draft-stream.ts:323`
  - body: After an overflow chain sets `textBaseOffset`, the exported `forceNewMessage()` path still leaves that offset in place. Existing callers use `forceNewMessage()` for unrelated preview rotations, so the next independent `update()` is sliced with the stale offset and can drop or suppress its beginning. Split internal overflow rotation from public reset, or reset the offset for public fresh-message starts and cover overflow followed by `forceNewMessage()`.
  - confidence: 0.9
