## Summary

Keep this draft PR open. Current main still has the reported Telegram reasoning-overflow problem, but the proposed branch is not ready because completed overflow chunks are routed through the existing superseded-preview cleanup path and would be deleted at final cleanup; the new 429 backoff path also drops forced lane rotations instead of deferring them.

## What This Changes

The PR changes the Telegram plugin draft-streaming path to split long reasoning previews across chained messages, add reasoning-continuation rendering, add shared throttling and 429 backoff for concurrent draft lanes, suppress partial reasoning-tag leakage, and extend Telegram streaming tests.

## Best Possible Solution

Revise this PR in-place so overflow-completed reasoning chunks have an explicit retained lifecycle separate from stale superseded previews, and so 429 backoff preserves requested lane rotations by applying them before the retry or returning a signal callers can honor. Add regressions that run a long reasoning preview through final cleanup and a 429-during-rotation case.

## Reproduction Assessment

Yes. The cleanup bug has a high-confidence source reproduction: make the PR overflow path call onSupersededPreview for a reasoning chunk, then follow the existing dispatch cleanup path that deletes every archived reasoning preview ID during final cleanup.

## Solution Assessment

No. Overflow chaining is the right product direction, but reusing the superseded-preview cleanup callback and swallowing forceNewMessage during 429 backoff are not the narrowest maintainable implementation because they conflate retained chunks with stale previews and can lose lane boundaries.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Preserve chained reasoning chunks during cleanup:** `extensions/telegram/src/draft-stream.ts:379-382`
  - body: The overflow path reports intentionally completed reasoning chunks through onSupersededPreview, but Telegram dispatch treats reasoning previews from that callback as archived cleanup targets and deletes them during final cleanup. A long reasoning stream would briefly chain and then lose completed chunks except the active last one, so the overflow-retention path needs a distinct signal or cleanup exclusion plus a dispatch regression.
  - confidence: 0.9
- **[P2] Preserve forced rotations during 429 backoff:** `extensions/telegram/src/draft-stream.ts:543-548`
  - body: When a 429 backoff is active, forceNewMessage now returns after logging that the lane rotation is deferred, but nothing records or applies that deferred reset. Callers continue updating lane bookkeeping and enqueue the next segment, so after backoff the next reasoning or answer boundary can edit the old preview instead of starting a new Telegram message.
  - confidence: 0.84
