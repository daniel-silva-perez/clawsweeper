## Summary

Keep open. Current main still has TTS backend RPCs, provider voice metadata, docs, and CLI/chat command controls, but the Control UI Settings path still lacks a discoverable TTS enable toggle, provider picker, default voice picker, and sample playback UI.

## What This Changes

Add a first-class Control UI Settings surface for enabling TTS, selecting the default provider and voice, and optionally playing a sample.

## Best Possible Solution

Add a focused Text-to-Speech panel in Control UI Settings backed by existing TTS RPCs, provider inventory, and speech-provider voice metadata, with only a minimal generic voice persistence/listing extension if the current RPCs are not sufficient.

Yes. A high-confidence source reproduction is available by inspecting the Quick Settings render path and searching ui/src for TTS RPC consumers; current main exposes backend TTS methods but no Settings UI consumer.

## Solution Assessment

Yes. A first-class Settings panel over the existing Gateway TTS methods is the narrowest maintainable direction; the safer alternative to UI hardcoding is a small generic backend seam for default voice persistence if needed.
