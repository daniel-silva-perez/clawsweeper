## Summary

Keep open. Current main still exposes TTS through gateway RPCs, CLI/config, docs, and provider voice metadata, but the requested first-class Control UI Settings toggle, provider selector, default voice picker, and sample playback are still absent; the existing keep-open review comments remain consistent with the current source.

## What This Changes

Add a first-class Text-to-Speech section to Control UI Settings with enable/disable, provider and default voice selectors, and optional sample playback backed by existing TTS preferences and provider capabilities.

## Best Possible Solution

The end state should be a discoverable Settings panel that reads TTS status/provider inventory, writes the same prefs/config paths used by existing CLI and RPC flows, distinguishes unavailable providers, persists provider-specific default voices, and uses tts.convert for a short sample playback.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/discoverability request. The high-confidence verification path is current-main static inspection of Quick Settings, Settings wiring, advanced config sections, and targeted TTS RPC usage search, which shows the requested controls are absent.

## Solution Assessment

Yes, the requested Control UI settings surface is the narrowest maintainable direction. It should reuse existing TTS RPCs and provider voice metadata first, adding only a generic voice-list/set-default seam if the current contracts cannot safely persist default voice choices.
