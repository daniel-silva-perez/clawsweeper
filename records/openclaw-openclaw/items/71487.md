## Summary

Keep open. Current main still exposes TTS through docs, local preferences, slash/CLI-style controls, and Gateway RPCs, but the requested first-class Control UI Settings toggle, provider picker, provider-filtered voice picker, and sample playback surface are still absent.

## What This Changes

The issue requests a first-class Control UI Settings section for enabling or disabling TTS, choosing a default provider and voice, and optionally playing a sample through the existing TTS preference/runtime path.

## Best Possible Solution

Add a first-class Text-to-Speech panel in Control UI Settings that reads current TTS status/providers, toggles auto-TTS through the existing Gateway methods, marks configured versus unavailable providers, filters voice options by selected provider, persists a default voice through a generic TTS preference contract, and plays a short provider-backed sample through tts.convert.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/discoverability request. The high-confidence verification path is static inspection of the current Control UI Settings wiring plus targeted searches for TTS RPC usage, which show the requested controls are absent.

## Solution Assessment

Yes. The requested Control UI settings surface is the narrowest maintainable solution, provided it reuses existing TTS RPCs and adds only a generic voice preference/listing seam if the current contracts are insufficient.
