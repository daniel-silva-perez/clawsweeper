Keep open. Current main still has backend TTS capability and CLI/chat-command affordances, but the requested first-class Control UI Settings surface is absent; the two existing review comments that kept this open remain consistent with current source inspection.

## What This Changes

The issue requests a visible Control UI Settings section for TTS enablement, default provider selection, provider-filtered default voice selection, and optional sample playback.

## Best Possible Solution

Add a first-class Text-to-Speech panel to Control UI Settings that reads current TTS status/providers, toggles auto-TTS through existing RPCs, shows configured versus unavailable providers, filters voices by selected provider, persists the selected default voice through a generic TTS preference/config path, and plays a short sample through tts.convert.

Not applicable as a bug reproduction; this is a feature/discoverability request. The high-confidence verification path is static current-main inspection of Control UI Settings wiring plus targeted searches for TTS RPC usage, which show the requested controls are absent.

## Solution Assessment

Yes, the requested Control UI settings surface is the narrowest maintainable direction. It should reuse existing TTS RPCs/provider metadata first and add only a generic voice listing/persistence seam if the current contracts cannot safely support the default voice picker.
