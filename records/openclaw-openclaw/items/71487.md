## Summary

Keep open. Current main still exposes TTS through docs, config, chat commands, and gateway RPCs, but the requested first-class Control UI Settings toggle, provider selector, provider-filtered default voice picker, and sample playback are still absent.

## What This Changes

Add a discoverable Text-to-Speech section in Control UI Settings for enabling/disabling TTS, choosing the default provider and voice, and optionally playing a sample.

## Best Possible Solution

Add a first-class Text-to-Speech panel in Control UI Settings that reads current TTS status/providers, toggles auto-TTS through existing RPCs, marks unavailable providers clearly, filters voice options by selected provider, persists the selected voice through the agreed TTS preference/config contract, and plays a short sample through tts.convert.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/discoverability request. The high-confidence verification path is static current-main inspection of Control UI Settings wiring plus targeted searches for TTS RPC usage, which show the requested controls are absent.

## Solution Assessment

Yes, the requested Control UI settings surface is the narrowest maintainable solution. It should reuse existing TTS RPCs and provider/runtime voice-listing seams first, adding only a generic voice preference/listing seam if current contracts are insufficient.
