## Summary

Keep open. Current main still has backend TTS RPCs, docs, provider metadata, and CLI/config-driven controls, but the Control UI Settings path still lacks a discoverable TTS enable toggle, provider selector, provider-filtered default voice picker, or sample playback control.

## What This Changes

The issue requests a first-class Control UI Settings surface for enabling TTS, choosing a default provider, choosing a provider-filtered default voice, and optionally playing a sample.

## Best Possible Solution

Add a focused Text-to-Speech panel in Control UI Settings backed by the existing TTS RPCs, using provider inventory/configured state for the provider picker and speech-provider voice metadata/listVoices for the voice picker. Keep provider-specific voice persistence generic through the existing messages.tts provider config contract or a minimal additive RPC rather than hardcoding bundled provider behavior in core UI.

Yes. A high-confidence source reproduction is available: inspect the Quick Settings render path and run the targeted ui/src search for TTS RPC consumers; current main has no Settings UI consumer despite the backend handlers being present.

## Solution Assessment

Yes. A first-class Settings panel over the existing Gateway TTS methods is the narrowest maintainable direction; the only likely extra work is an additive generic voice-persistence seam if config.patch is not ergonomic enough for provider-specific voice fields.
