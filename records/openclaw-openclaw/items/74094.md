## Summary

Keep open. Current main still exposes raw task status/progress fields plus several independent task renderers, but it does not provide the requested shared compressed operational-summary contract with current state, last good checkpoint, blocker, and next action. The earlier ClawSweeper comment reached the same result, and the task/status files remain unchanged on the relevant surface.

## What This Changes

Add a shared compact operational summary layer for tasks and reuse it across CLI task output, task summary views, and auto-reply status rendering.

## Best Possible Solution

Add an additive TaskOperationalSummary contract in the task subsystem, derive it deterministically from structured task state, expose it through task-domain views, and make CLI/chat/status/session_status/notification renderers consume it while retaining existing raw fields for compatibility.

## Reproduction Assessment

Not applicable as a runtime reproduction because this is an enhancement request. There is a high-confidence verification path by source/docs inspection: current main exposes raw summaries and independent formatting, not the requested shared operational summary contract.

## Solution Assessment

Yes, a shared additive builder is the narrowest maintainable direction if it stays deterministic and compatibility-preserving. Per-surface wording tweaks would leave the inconsistency the issue is trying to remove.
