## Summary

Keep open. Current main has cleanup-aware task snapshots, raw progress/terminal summaries, task-count summaries, and several task renderers, but it still lacks the requested shared compressed operational-summary contract with current state, last good checkpoint, current blocker, and recommended next action. CLI, chat, status, session_status, plugin/runtime views, and notifications still compose related wording independently.

## What This Changes

Add a shared compact operational-summary layer for task status across task CLI output, task summary views, and auto-reply/status surfaces.

## Best Possible Solution

Introduce a shared deterministic task operational-summary builder in the task subsystem, expose an additive structured/rendered summary where useful, and make CLI, /tasks, /status, session_status, task-domain views, and notification rendering consume it while preserving existing raw task fields.

## Repair Work Prompt

Implement compressed operational summaries for task status as a narrow additive OpenClaw task-system feature. Add a shared deterministic builder in `src/tasks/task-status.ts` or a sibling task helper that derives a structured summary plus compact rendered text from `TaskRecord` and available structured task state/events: current state, last good checkpoint when known, current blocker when present, and recommended next action when known. Reuse the builder in `src/commands/tasks.ts` list/show output, `src/auto-reply/reply/commands-tasks.ts`, `src/status/status-text.ts`, `src/agents/tools/session-status-tool.ts`, `src/tasks/task-executor-policy.ts`, and task-domain/plugin views if exposing the summary to plugins is useful. Keep raw `TaskRecord` fields and existing JSON shapes backwards-compatible; any public/plugin type change must be additive. Do not reintroduce queued/running-only summaries, do not parse arbitrary freeform progress text into overconfident actions, and do not replace Task Flow blocked-summary behavior. Update `docs/automation/tasks.md`, `docs/cli/tasks.md`, and the changelog for the visible output change, with user-facing credit to `@neofdezf-cloud` if a changelog entry is added.
