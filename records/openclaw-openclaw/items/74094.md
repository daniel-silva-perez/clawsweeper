## Summary

Keep open. Current main still exposes raw task status/progress/terminal fields and several independent task renderers, but it does not provide the requested shared compressed operational-summary contract with current state, last good checkpoint, current blocker, and next action. The prior ClawSweeper comment reached the same conclusion, and the relevant task/status files have not changed since that review.

## What This Changes

Add a shared compressed operational summary for tasks across CLI, task/status views, and auto-reply surfaces, including current state, last successful checkpoint, blocker, and next recommended action.

## Best Possible Solution

Add an additive TaskOperationalSummary contract in the task subsystem, derive it deterministically from structured task state, expose it through task-domain views, and make CLI, chat /tasks, /status, session_status, and notification renderers consume it while preserving existing raw fields for compatibility.

Not applicable as a runtime reproduction because this is an enhancement request. There is a high-confidence source verification path: current main exposes raw fields and separate renderers, not the requested shared operational summary contract.

## Solution Assessment

Yes, a shared additive builder is the narrowest maintainable direction if it stays deterministic and compatibility-preserving. Per-surface wording tweaks would leave the inconsistency this issue is trying to remove.
