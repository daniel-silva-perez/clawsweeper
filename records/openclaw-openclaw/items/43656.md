## Summary

Keep open. Current main still lacks cross-Gateway session-tool routing and the linked feature issue remains open, but this PR is not mergeable as-is because it widens raw Gateway URL routing around existing agent-tool guardrails, sends invalid remote spawn payloads, and diverges from current session-send semantics.

## What This Changes

The PR adds gateway.peers config/schema/help plus gateway, gatewayUrl, and gatewayToken handling for sessions_send and sessions_spawn through a new peer resolver and tests.

## Best Possible Solution

Land a maintainer-approved Gateway federation design that keeps raw URL overrides behind existing guardrails, defines trusted config-backed peer credentials and TLS/caller authorization, sends valid Gateway protocol payloads, preserves current local session semantics, updates docs, and adds end-to-end remote send/spawn coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep explicit gatewayUrl on the guarded path:** `src/agents/tools/gateway-peer.ts:66-67`
  - body: The helper returns arbitrary explicit gatewayUrl values and the session tools pass them directly to callGateway, bypassing the current allowlist and canonical validation for agent-tool URL overrides. Keep raw URLs behind resolveGatewayOptions; reserve any bypass for trusted config-backed peers with a documented policy.
  - confidence: 0.9
- **[P1] Include an idempotencyKey in remote spawn calls:** `src/agents/tools/sessions-spawn-tool.ts:167-179`
  - body: The remote sessions_spawn branch calls the Gateway agent method without idempotencyKey, but AgentParamsSchema requires a non-empty idempotencyKey. Cross-Gateway spawn requests can be rejected before the remote run starts.
  - confidence: 0.91
- **[P2] Require credentials for configured peers before routing:** `src/agents/tools/gateway-peer.ts:59-62`
  - body: Named peers can return gatewayToken: undefined, and callGateway requires explicit credentials for explicit URL overrides. Either require peer credentials or add a separate trusted config-backed auth path before routing.
  - confidence: 0.86
- **[P2] Reject sessionKey and label together on remote sends:** `src/agents/tools/sessions-send-tool.ts:71-80`
  - body: The remote sessions_send path only checks that at least one target is present, then silently prefers sessionKey when both sessionKey and label are supplied. The local path rejects this combination, and the remote branch should preserve that exclusivity.
  - confidence: 0.84
- **[P2] Use the per-session nested lane for remote sends:** `src/agents/tools/sessions-send-tool.ts:133`
  - body: Current main scopes nested agent lanes with resolveNestedAgentLaneForSession(resolvedKey), but the remote send branch hard-codes the global nested lane. That can reintroduce cross-session nested-lane contention for peer sends.
  - confidence: 0.78
