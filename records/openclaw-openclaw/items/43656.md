## Summary

Keep open. Current main still lacks the requested cross-gateway session targeting, so this PR is useful source work, but it is stale against main and still has blocking correctness and security issues in the proposed remote routing path.

## What This Changes

The PR adds a `gateway.peers` config surface, a peer resolver, `gateway`/`gatewayUrl`/`gatewayToken` parameters for `sessions_send` and `sessions_spawn`, remote `callGateway` forwarding paths, and peer resolver tests.

## Best Possible Solution

Land this only after a maintainer-approved gateway federation design keeps raw URL overrides behind existing guardrails, defines the config-backed peer trust and credential policy, sends valid Gateway protocol payloads, preserves local session semantics, adds setup docs, and includes end-to-end coverage for both remote send and spawn.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep raw gatewayUrl on the guarded path:** `src/agents/tools/gateway-peer.ts:66-67`
  - body: The helper accepts any explicit `gatewayUrl` and the new session tool paths pass it directly to `callGateway`, bypassing the current agent-tool allowlist that restricts overrides to loopback or `gateway.remote.url`. Keep raw URL overrides behind the existing guard, or limit the bypass to a reviewed config-backed peer path with a defined trust policy.
  - confidence: 0.88
- **[P1] Include idempotencyKey in remote spawn requests:** `src/agents/tools/sessions-spawn-tool.ts:167-179`
  - body: The remote `sessions_spawn` branch calls the Gateway `agent` method without `idempotencyKey`, but `AgentParamsSchema` requires it. Cross-gateway spawn requests will be rejected as invalid params before the remote run starts.
  - confidence: 0.93
- **[P2] Require peer credentials before returning URL overrides:** `src/agents/tools/gateway-peer.ts:59-62`
  - body: Named peers can return `gatewayToken: undefined`, and the tests document tokenless peers as supported. `callGateway` requires explicit credentials for any explicit URL override, so those peers fail before the remote request is attempted unless the implementation adds a different trusted config-backed path or makes credentials required.
  - confidence: 0.87
- **[P2] Preserve sessionKey and label exclusivity remotely:** `src/agents/tools/sessions-send-tool.ts:71-80`
  - body: The local `sessions_send` path rejects callers that provide both `sessionKey` and `label`, but the remote branch only checks that at least one exists and then silently prefers `sessionKey`. That can skip the requested remote label resolution and route to the wrong session.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff introduces security-sensitive remote Gateway routing from agent tools and currently bypasses the existing raw URL guardrail for explicit targets.

Concerns:

- **[high] Raw gatewayUrl bypasses the agent-tool allowlist:** `src/agents/tools/gateway-peer.ts:66`
  - body: Current main validates agent-tool URL overrides through `resolveGatewayOptions`; this PR accepts arbitrary explicit `gatewayUrl` values and passes them directly to `callGateway`, broadening outbound WebSocket and credential behavior from agent tool calls.
  - confidence: 0.88
- **[medium] Peer routing lacks a complete trust policy:** `src/config/types.gateway.ts:1`
  - body: The PR adds persisted peer URLs and optional SecretInput tokens, but it does not define credential requirements, tokenless-peer semantics, TLS pinning, or caller authorization for remote `agent` calls that can trigger model usage and tool execution on another Gateway.
  - confidence: 0.78
