## Summary

Keep open. Current main still does not implement cross-Gateway routing for `sessions_send` or `sessions_spawn`, and the linked feature issue remains open, but this PR is not safe to merge as-is because the branch bypasses existing agent-tool Gateway URL guardrails, sends an invalid remote spawn payload, and diverges from current session-send behavior.

## What This Changes

The PR adds `gateway.peers` config/schema/help plus `gateway`, `gatewayUrl`, and `gatewayToken` handling for `sessions_send` and `sessions_spawn` through a new peer resolver and tests.

## Best Possible Solution

Land a maintainer-approved Gateway federation design that keeps raw URL overrides behind existing guardrails, defines trusted config-backed peer credentials and TLS/caller authorization, sends valid Gateway protocol payloads, preserves current local session semantics, updates docs, and adds end-to-end remote send/spawn coverage.

## Reproduction Assessment

Yes. Current-main inspection shows the requested cross-Gateway session-tool fields are absent, and the PR defects are reproducible by static protocol checks: remote `sessions_spawn` omits required `idempotencyKey`, tokenless peers flow into `callGateway` URL override auth checks, and remote sends diverge from current local validation/lane behavior.

## Solution Assessment

No. The PR addresses a real feature gap, but the current implementation is not the best way to solve it because it widens raw URL routing and remote execution semantics without a complete trusted-peer policy or valid Gateway protocol payloads.
