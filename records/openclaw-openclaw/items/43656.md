## Summary

Keep open. Current main still lacks the requested cross-gateway session routing, and this PR is the active implementation attempt for a real feature gap, but the proposed patch remains unsafe to merge because it changes security-sensitive Gateway routing semantics and still has concrete protocol/behavior bugs.

## What This Changes

The PR adds a `gateway.peers` config map, a peer resolver, `gateway`/`gatewayUrl`/`gatewayToken` inputs for `sessions_send` and `sessions_spawn`, remote Gateway forwarding paths, and peer resolver tests.

## Best Possible Solution

Land a maintainer-approved Gateway federation design that keeps raw URL overrides behind existing guardrails, clearly defines trusted config-backed peer credentials and authorization, sends valid Gateway protocol payloads, preserves local session input semantics, documents setup, and adds end-to-end coverage for remote send and spawn.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep raw gatewayUrl on the guarded path:** `src/agents/tools/gateway-peer.ts:66-67`
  - body: The helper accepts explicit `gatewayUrl` and the new session tool paths pass it directly to `callGateway`, bypassing the current agent-tool allowlist that restricts overrides to loopback or `gateway.remote.url`. Keep raw URL overrides behind the existing guard, and limit any bypass to a reviewed config-backed peer path with a defined trust policy.
  - confidence: 0.88
- **[P1] Include idempotencyKey in remote spawn requests:** `src/agents/tools/sessions-spawn-tool.ts:167-179`
  - body: The remote `sessions_spawn` branch calls the Gateway `agent` method without `idempotencyKey`, but `AgentParamsSchema` requires it. Cross-gateway spawn requests will be rejected as invalid params before the remote run starts.
  - confidence: 0.93
- **[P2] Require usable credentials for configured peers:** `src/agents/tools/gateway-peer.ts:59-62`
  - body: Named peers can return `gatewayToken: undefined`, and the tests document tokenless peers as supported. `callGateway` requires explicit credentials for any explicit URL override, so those peers fail before the remote request is attempted unless the implementation adds a different trusted config-backed path or makes credentials required.
  - confidence: 0.87
- **[P2] Preserve sessionKey and label exclusivity remotely:** `src/agents/tools/sessions-send-tool.ts:71-80`
  - body: The local `sessions_send` path rejects callers that provide both `sessionKey` and `label`, but the remote branch only checks that at least one exists and then silently prefers `sessionKey`. That can skip the requested remote label resolution and route to the wrong session.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff introduces security-sensitive remote Gateway routing from agent tools and currently weakens the existing raw URL override guardrail.

Concerns:

- **[high] Raw gatewayUrl bypasses the agent-tool allowlist:** `src/agents/tools/gateway-peer.ts:66`
  - body: Current main validates agent-tool URL overrides through `resolveGatewayOptions`; this PR accepts arbitrary explicit `gatewayUrl` values and passes them directly to `callGateway`, broadening outbound WebSocket and credential behavior from agent tool calls.
  - confidence: 0.88

  - confidence: 0.78
