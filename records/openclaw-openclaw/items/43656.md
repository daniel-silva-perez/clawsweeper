## Summary

Keep open. Current main still lacks cross-Gateway targeting for sessions_send/sessions_spawn, so this PR is not obsolete, but the proposed branch is not mergeable as-is: it bypasses the existing agent-tool URL guard, sends an invalid remote spawn payload, and diverges from current main session semantics. The linked feature request #43605 remains open, and this PR is the active implementation candidate rather than something to close before merge.

## What This Changes

The PR adds gateway.peers config/schema/help plus gateway, gatewayUrl, and gatewayToken handling for sessions_send and sessions_spawn through a new peer resolver and tests.

## Best Possible Solution

Land a maintainer-approved Gateway federation design that keeps raw URL overrides behind existing guardrails, defines trusted config-backed peer credentials and TLS/caller authorization, sends valid Gateway protocol payloads, preserves current local session semantics, updates docs, and adds end-to-end remote send/spawn coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep explicit gatewayUrl on the guarded path:** `src/agents/tools/gateway-peer.ts:66-67`
  - body: The helper accepts explicit gatewayUrl and the new session tool paths pass it directly to callGateway, bypassing the current allowlist that restricts agent-tool overrides to loopback or gateway.remote.url. Keep raw URL overrides behind resolveGatewayOptions, and limit any peer bypass to reviewed config-backed peers with a defined trust policy.
  - confidence: 0.9
- **[P1] Include an idempotencyKey in remote spawn calls:** `src/agents/tools/sessions-spawn-tool.ts:167-179`
  - body: The remote sessions_spawn branch calls the Gateway agent method without idempotencyKey, but AgentParamsSchema requires a non-empty idempotencyKey. Cross-Gateway spawn requests can be rejected before the remote run starts.
  - confidence: 0.91
- **[P2] Require credentials for configured peers before routing:** `src/agents/tools/gateway-peer.ts:59-62`
  - body: Named peers can return gatewayToken: undefined, and the tests explicitly document a tokenless peer. callGateway requires explicit credentials for any explicit URL override, so those peers fail locally unless the implementation either requires peer credentials or adds a separate trusted config-backed auth path.
  - confidence: 0.86
- **[P2] Reject sessionKey and label together on remote sends:** `src/agents/tools/sessions-send-tool.ts:71-80`
  - body: The remote sessions_send path only checks that at least one target is present, then silently prefers sessionKey when both sessionKey and label are supplied. The local path rejects this combination, and the remote branch should preserve that exclusivity to avoid routing to the wrong session.
  - confidence: 0.84
- **[P2] Use the per-session nested lane for remote sends:** `src/agents/tools/sessions-send-tool.ts:127-133`
  - body: Current main scopes nested agent lanes with resolveNestedAgentLaneForSession(resolvedKey), but the remote send branch hard-codes the global nested lane. That reintroduces cross-session nested-lane contention for peer sends and diverges from the current local sessions_send behavior.
  - confidence: 0.78
