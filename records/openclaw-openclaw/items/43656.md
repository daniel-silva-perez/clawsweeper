## Summary

Keep open. Current main does not already implement cross-Gateway session targeting, and this open implementation PR is still useful, but the proposed patch is not merge-ready because it bypasses the existing agent-tool Gateway URL guardrail, sends invalid remote spawn payloads, and leaves Gateway federation trust semantics unresolved.

## What This Changes

The PR adds `gateway.peers` config/schema/help, a peer resolver, `gateway`/`gatewayUrl`/`gatewayToken` fast paths for `sessions_send` and `sessions_spawn`, and peer-resolution tests.

## Best Possible Solution

Land a maintainer-approved Gateway federation design that keeps raw URL overrides behind existing guardrails, defines trusted config-backed peer credentials and TLS/caller authorization, sends valid Gateway protocol payloads, preserves local session semantics, updates docs, and adds end-to-end remote send/spawn coverage.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence reproduction path: a cross-Gateway `sessions_spawn` call reaches the remote `agent` schema without `idempotencyKey`, and explicit `gatewayUrl` inputs are visibly routed around the current `resolveGatewayOptions` guard.

## Solution Assessment

No. The PR addresses a real feature gap, but this is not the best solution yet because it widens remote Gateway routing and execution semantics without a complete trusted-peer/auth policy or valid protocol payloads.

  - body: The helper returns arbitrary explicit `gatewayUrl` values, and the new session paths pass them directly to `callGateway`. That bypasses the existing `resolveGatewayOptions` allowlist for agent-tool URL overrides, so keep raw URL overrides on the guarded path and reserve any bypass for trusted config peers with a documented policy.
  - confidence: 0.9

  - body: The remote `sessions_spawn` branch calls the Gateway `agent` method without `idempotencyKey`, but `AgentParamsSchema` requires a non-empty `idempotencyKey`. Cross-Gateway spawn requests can be rejected as invalid params before the remote run starts.
  - confidence: 0.91
- **[P2] Require credentials for configured peers before routing:** `src/agents/tools/gateway-peer.ts:59-62`
  - body: Named peers can return `gatewayToken: undefined`, but `callGateway` treats the peer URL as an explicit URL override and rejects URL overrides without explicit credentials. Either require peer credentials or add a separate trusted config-backed auth path before routing.
  - confidence: 0.86
- **[P2] Reject sessionKey and label together on remote sends:** `src/agents/tools/sessions-send-tool.ts:71-74`
  - body: The remote `sessions_send` path only checks that at least one target is present, then silently prefers `sessionKey` when both `sessionKey` and `label` are supplied. The local path rejects this combination, and the remote branch should preserve that exclusivity.
  - confidence: 0.84
- **[P2] Use the per-session nested lane for remote sends:** `src/agents/tools/sessions-send-tool.ts:127-140`
  - body: Current main scopes nested agent lanes with `resolveNestedAgentLaneForSession(resolvedKey)`, but the remote send branch hard-codes the global nested lane. That can reintroduce cross-session nested-lane contention for peer sends.
  - confidence: 0.78
