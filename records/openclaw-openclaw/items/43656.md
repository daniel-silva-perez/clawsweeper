## Summary

Keep open. Current main does not implement cross-Gateway session send/spawn, and this PR is still the active implementation candidate for the same-author feature issue, but the latest patch has blocking protocol, auth, and trust-boundary problems that need maintainer review before merge.

## What This Changes

The PR adds a `gateway.peers` config map plus `gateway`/`gatewayUrl`/`gatewayToken` parameters and remote Gateway forwarding paths for `sessions_send` and `sessions_spawn`.

## Best Possible Solution

Land a maintainer-approved Gateway federation design that keeps raw URL overrides behind existing guardrails, defines trusted config-backed peer credentials and TLS/caller authorization semantics, sends valid Gateway protocol payloads, preserves local session/A2A behavior where applicable, updates user docs, and adds end-to-end remote send/spawn coverage.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence reproduction path: on the PR branch, `sessions_spawn(gateway=...)` reaches the remote `agent` method without the required `idempotencyKey`, and `gatewayUrl` inputs visibly bypass the current `resolveGatewayOptions` allowlist.

## Solution Assessment

No. The feature direction addresses a real multi-Gateway gap, but this PR is not the best mergeable fix until the protocol payload, URL/auth guardrails, tokenless peer semantics, and federation trust policy are resolved.

  - body: Named peers can return `gatewayToken: undefined`, but both remote branches pass `peerUrl` to `callGateway` as an explicit URL override. `callGateway` requires explicit credentials for URL overrides, so tokenless peers fail before reaching the remote Gateway unless a separate trusted-peer auth path is added.
  - confidence: 0.86
