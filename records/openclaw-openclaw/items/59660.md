## Summary

Keep open. Current main has several update-safety and repair mechanisms, but the requested user problem is broader than what is proven implemented: non-breaking business-safe updates and no unexpected agent/permission changes. The report is recent, has no duplicate/canonical linked item in the provided context, and is salvageable as a product/update-stability enhancement rather than closeable cleanup.

## Best Possible Solution

Keep this open and narrow it into an actionable update-stability track: define the stable-channel compatibility promise, make the updater/doctor output clearly distinguish automatic compatibility repairs from opt-in security hardening, add or verify pre-update backup/rollback guidance for business automation users, and ask the reporter for the exact update path, before/after versions, config diff, logs, and the specific agent/channel/tool permission that changed.
