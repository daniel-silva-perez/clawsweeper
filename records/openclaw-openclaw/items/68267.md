## Summary

Keep open. Current main does not expose a run-scoped `chat.history` parameter, and this PR is the focused implementation for that feature, but the latest patch still has blocking correctness and protocol-follow-through issues: the matcher can include another run's messages for valid punctuation/symbol-containing IDs, and the public protocol generated artifacts are not updated. The PR is also currently reported as dirty against main, so it needs author or maintainer follow-up before merge.

## What This Changes

The PR adds an optional `runId` parameter to gateway `chat.history`, filters raw transcript messages by that identifier before chat-history size-budget processing, and adds matcher-focused tests.

## Best Possible Solution

Keep the additive `chat.history({ runId })` direction, but land it on current main with exact run identity semantics across the full `NonEmptyString` contract and regenerated protocol artifacts. A maintainable final patch should match explicit marker forms or structured fields without prefix leakage, apply the filter before display/budget projection, and update generated protocol clients in the same change.

## Reproduction Assessment

Yes. The remaining matcher bug is reproducible by evaluating the proposed regex against messages containing `run_id=run:2` with query `run` and `run_id=🚀🎯` with query `🚀`; both return matches even though they are different run IDs. Current main cannot reproduce the feature behavior because it does not have `runId` filtering yet.

## Solution Assessment

No. The PR picks the right surface and sequence, and earlier review issues were addressed, but this is not the best final implementation until matcher semantics match the schema contract, generated protocol outputs are committed, and the branch is brought current with main.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Treat every runId character as part of the token:** `src/gateway/server-methods/chat.history-run-id-filter.ts:25`
  - body: `runId` is accepted as any non-empty string, but this boundary only treats letters, numbers, marks, `_`, and `-` as token characters. A valid ID like `run:2` or `🚀🎯` can therefore be returned when the caller asks for the prefix `run` or `🚀`, which breaks the run-scoping guarantee and can expose another run's messages in the response. Match exact marker forms or use a boundary rule that is valid for the whole schema contract.
  - confidence: 0.9
- **[P2] Regenerate the protocol artifacts with the schema change:** `src/gateway/protocol/schema/logs-chat.ts:36`
  - body: Adding `runId` to `ChatHistoryParamsSchema` changes the public Gateway protocol, but this PR does not update the generated JSON schema or Swift protocol models that `protocol:check` diffs. Typed macOS/iOS clients will keep a stale `ChatHistoryParams` shape and the protocol check will fail until the generated artifacts are committed.
  - confidence: 0.84
