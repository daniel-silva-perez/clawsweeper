## Summary

Keep this issue open. Current main still has Telegram Business update-name coverage only: grammY/OpenClaw can request `business_connection` and `business_message`, but the Telegram plugin still has no runtime handlers for those update types, no business fields in update dedupe/sequential keys, and no `business_connection_id` mapping for outbound replies. Open PR #53629 explicitly closes this issue and remains open/unmerged, so the best cleanup path is to review/land that PR or an equivalent fix before closing.

## Best Possible Solution

Keep this issue open as the canonical Telegram Business support tracker. Review and land PR #53629, or an equivalent Telegram plugin fix, that registers `business_connection` and `business_message` handlers before the normal message handler, routes `business_message` through the existing inbound pipeline, extends dedupe/sequential key handling, tracks and persists `business_connection_id` for outbound replies, and adds focused Telegram tests plus docs.
