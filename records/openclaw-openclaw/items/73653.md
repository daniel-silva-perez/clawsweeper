## Summary

Keep this issue open. Current main still has the reported Gateway protocol gap: `agent` request params are validated against a closed `AgentParamsSchema`, there is no `adapterMeta`/`adapterContext` namespace or `paperclip` escape hatch, and unknown top-level fields are rejected before runtime handling. The linked Paperclip-side workaround and the open OpenClaw PR are useful context, but neither has landed on current main nor fully resolves the generic adapter metadata contract.

## What This Changes

Allow third-party Gateway adapters to attach metadata to `agent` request params without failing closed-schema validation.

## Best Possible Solution

Add an additive, documented Gateway protocol field such as `adapterMeta` or `adapterContext` to `AgentParamsSchema`, keep the rest of the top-level schema strict, add protocol/server regression coverage for third-party metadata, and coordinate Paperclip to use the namespaced field once released.

## Reproduction Assessment

Yes. A request with valid `message`, `sessionKey`, and `idempotencyKey` plus an unknown top-level `paperclip` field follows the current source path into `validateAgentParams` and fails closed-schema validation before runtime handling.

## Solution Assessment

Yes for the issue's preferred direction: a typed namespaced metadata record is the narrowest maintainable core fix. A blanket passthrough or hard-coded `paperclip` field would solve some symptoms but leaves weaker validation or a vendor-specific public contract.
