## Summary

Keep open. Current main still reproduces the models-list provider drift for bare configured entries and fallbacks, and the PR addresses the right surface but leaves alias handling inconsistent when a bare configured key is inferred away from the default provider.

## What This Changes

The PR updates `resolveConfiguredEntries` to infer a unique configured provider for bare model refs and adds focused tests for bare keys, bare fallbacks, ambiguous IDs, and explicit provider prefixes.

## Best Possible Solution

Land a narrow models-list fix that uses the same bare-ref provider inference contract for configured rows and fallback rows while keeping alias index keys aligned with the resolved provider. The regression coverage should include bare configured keys with aliases and alias fallback values, not only the non-alias cases.

## Reproduction Assessment

Yes. A config with `models.providers.google.models: [{ id: "gemini-2.0-flash-001" }]` and a bare `agents.defaults.models` key or fallback reproduces the current-main display drift by inspection of `resolveConfiguredEntries`; the PR finding reproduces when that bare key also has an `alias`.

## Solution Assessment

No, not as submitted. Inferring the provider is the right direction, but the alias index and `aliasesByKey` need to resolve bare keys under the same effective provider or a shared helper so aliases and alias fallbacks stay consistent.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep aliases keyed to the inferred provider:** `src/commands/models/list.configured.ts:76-80`
  - body: When a bare `agents.defaults.models` key has an `alias`, the PR parses the configured row with the inferred provider, but `aliasesByKey` is still populated from an alias index built with `DEFAULT_PROVIDER`. The row can therefore show as `google/<model>` while its alias remains stored under `openai/<model>`, dropping the alias tag and letting alias fallbacks resolve to the wrong provider.
  - confidence: 0.86
