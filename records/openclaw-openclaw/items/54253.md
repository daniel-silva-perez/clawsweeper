## Summary

Keep this issue open. Current main contains proxy/service cleanup that explains several later workaround comments, but it does not implement or disprove the original RISC-V64 OpenRouter/OpenAI-compatible transport failure. The reported OpenRouter setup still routes through the shared guarded Undici fetch path, and there is no RISC-V64-specific fallback, diagnostic, or regression coverage in the relevant provider/network code.

## Best Possible Solution

Keep this issue open as the canonical RISC-V64 provider-transport tracker. Preserve the shipped proxy/service cleanup as a partial mitigation, then collect board/distro, Node binary source and version, process.arch, direct globalThis.fetch versus node:https results against the same provider, redacted service proxy environment, and raw gateway logs. If RISC-V64 Undici/fetch is confirmed, add either a guarded node:https fallback or an explicit compatibility diagnostic in src/infra/net/runtime-fetch.ts, src/infra/net/fetch-guard.ts, or src/agents/provider-transport-fetch.ts with focused regression coverage for dispatcher selection and error reporting.
