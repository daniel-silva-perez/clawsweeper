## Summary

Keep this issue open. Current main has proxy/service cleanup that explains several later workaround comments, but it still does not implement or disprove the original RISC-V64 OpenRouter/OpenAI-compatible transport failure. The relevant provider path still goes through guarded Undici fetch, and there is no RISC-V64-specific fallback or diagnostic in the current network/provider code.

## Best Possible Solution

Keep this issue open as the canonical RISC-V64 provider-transport tracker. Maintain the shipped proxy/service cleanup as a partial mitigation, then collect board/distro, Node binary source and version, process.arch, direct globalThis.fetch versus node:https results against the same provider, redacted service proxy environment, and raw gateway logs. If RISC-V64 Undici/fetch is confirmed, add either a guarded fallback or an explicit compatibility diagnostic in the shared provider transport path around src/infra/net/runtime-fetch.ts, src/infra/net/fetch-guard.ts, or src/agents/provider-transport-fetch.ts, with focused regression coverage for dispatcher selection/error reporting.
