## Summary

Keep this issue open. Current main still has the reported post-`connect.challenge` blind spot: `sendConnect()` can synchronously fail while assembling/signing the device connect frame before any `connect` request is sent, and that call is still inside a broad `handleMessage()` catch that only logs a debug-level "gateway client parse error". The linked Google Chat PR is acknowledged in the thread as unrelated, and the older 3000ms handshake-timeout fixes do not cover this newer post-challenge/no-reply symptom.

## Best Possible Solution

Keep the issue open and patch `src/gateway/client.ts`: wrap the `sendConnect()` assembly/signing/public-key/send path in its own error handling that calls `onConnectError`, logs at an actionable level, closes/resets the socket, and does not mark the challenge as successfully sent before the frame is actually queued. Narrow `handleMessage()` so only JSON parse/validation failures are reported as parse errors. Add a focused `src/gateway/client.test.ts` regression where a post-challenge connect-frame helper throws, asserting that the CLI surfaces a connect error, closes cleanly, and does not sit open with no `connect` frame.

## Risks / Open Questions

- The exact throw/stall point is not proven by current evidence because the reporter’s standalone Ed25519 signing test succeeded, so a fix should add targeted error propagation/instrumentation around the whole connect-frame assembly/send path rather than assume only crypto signing fails.
- Do not treat this as the older 3000ms pre-auth timeout issue: those fixes are in the changelog, but this report is newer and specifically says the gateway challenge is received while the client never sends the signed connect reply.
