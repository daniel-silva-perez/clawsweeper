## Summary

Keep this issue open. The discussion already corrected the accidental #68614 cross-reference as unrelated, and current `main` still has the same post-challenge silent-failure shape: `handleMessage()` calls `sendConnect()` for `connect.challenge` inside a broad catch, while `sendConnect()` performs synchronous auth selection, device payload construction, signing, and public-key normalization before the `connect` request is sent.

## What This Changes

The issue requests a fix for gateway-backed CLI commands hanging after the WebSocket gateway sends `connect.challenge` but before the CLI emits the signed connect frame.

## Best Possible Solution

Implement a narrow Gateway client fix: isolate JSON parsing from event handling, wrap post-challenge connect-frame assembly/signing/send in explicit error handling, report failures through the connection error path, close or reset the socket cleanly, and add a regression test proving a thrown device-signing or public-key normalization failure after `connect.challenge` does not leave the CLI waiting silently.

## Reproduction Assessment

Yes. The issue includes a concrete handshake trace, and current `main` has a deterministic unit-level reproduction path by making device connect-frame assembly throw after `connect.challenge` and observing that no `connect` frame is sent under the broad handler catch.

## Solution Assessment

Yes, the narrow client-side error propagation fix is the best direction. Gateway-side idle timeout tuning and HMAC fallback are not sufficient because the observed failure is before the CLI emits the signed connect request.

## Repair Work Prompt

Fix openclaw/openclaw#68944. The observable bug is that gateway-backed CLI commands can hang after receiving `connect.challenge` when the post-challenge client path throws before sending the signed `connect` request. Keep the fix in the generic Gateway client/CLI transport boundary. In `src/gateway/client.ts`, do not let synchronous failures from auth selection, device payload construction, `signDevicePayload()`, public-key normalization, or connect request-frame construction get swallowed by the generic `handleMessage()` parse-error catch. Narrow `handleMessage()` so parse/validation failures remain parse errors, and route connect-frame failures through `onConnectError`, actionable logging, and a clean socket close/reset. Consider wiring `src/gateway/call.ts` to settle promptly on `onConnectError` if needed, but do not change the protocol, add HMAC fallback, or log tokens/private keys/signature payloads. Add focused regression coverage in `src/gateway/client.test.ts` and, if the CLI wrapper needs wiring, `src/gateway/call.test.ts`: simulate a valid `connect.challenge` followed by a thrown device signing or public-key normalization step and assert the CLI/client surfaces a connect error, closes/resets the socket, and does not leave an open socket with no connect frame. Add a user-facing `CHANGELOG.md` fix entry crediting @WaMaSeDu and @mj-deving if the final behavior change is user-facing.
