## Summary

Keep open. Current main still stages backup metadata under `os.tmpdir()`, macOS service startup still sets `TMPDIR` inside the OpenClaw state directory, and the archive creation call still includes both the explicit manifest path and the state source tree. The linked open PR #75031 uses closing syntax for this issue, so the issue should remain open until that PR or an equivalent focused fix lands.

## What This Changes

The issue asks for backup creation to stage manifest metadata outside the archived OpenClaw state tree so `TMPDIR=~/.openclaw/tmp` cannot produce duplicate `manifest.json` entries.

## Best Possible Solution

Keep the issue open until PR #75031 or an equivalent focused fix lands. The final state should guarantee backup manifest staging is outside every archived source and include a `TMPDIR`-inside-state regression proving the archive has exactly one root manifest.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: Darwin service startup places `TMPDIR` under state, backup creation stages `manifest.json` under `os.tmpdir()`, and tar archives both that explicit manifest and the state tree. A live archive was not produced because this review was required to keep the checkout read-only.

## Solution Assessment

Yes in direction. The narrow maintainable solution is to use `resolvePreferredOpenClawTmpDir()` or a backup-specific staging resolver outside all included sources, plus a regression that asserts exactly one root manifest when ambient `TMPDIR` points inside state.
