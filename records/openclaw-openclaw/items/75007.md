## Summary

Keep open. Current main and the latest release still stage backup metadata under `os.tmpdir()`, while macOS service startup sets `TMPDIR` inside the OpenClaw state directory; the backup tar call then includes both the explicit manifest path and the state source tree. The linked open PR #75031 uses closing syntax for this issue, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue reports that `openclaw backup create` can archive `manifest.json` twice when macOS LaunchAgent `TMPDIR` points at `~/.openclaw/tmp`.

## Best Possible Solution

Keep the issue open until the linked PR or an equivalent focused backup fix lands. The final code should stage the manifest outside every included backup source, add a TMPDIR-inside-state regression, and prove the produced archive has exactly one root manifest.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: Darwin service setup puts TMPDIR under the state directory, backup creation uses `os.tmpdir()` for the manifest staging directory, and tar archives both the explicit manifest and the state tree.

## Solution Assessment

Yes in direction. The narrow maintainable solution is to use the existing OpenClaw temp-dir resolver or a backup-specific staging resolver, with a containment guard ensuring the chosen staging root is outside all included sources.
