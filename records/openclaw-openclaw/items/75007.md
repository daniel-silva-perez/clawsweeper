## Summary

Keep open. Current main still stages backup metadata under `os.tmpdir()` while macOS service startup sets `TMPDIR` inside the OpenClaw state directory, and the tar input includes both that explicit manifest path and the backed-up state tree. Open PR #75031 is linked as the implementation candidate and is still open/unmerged, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Ensure `openclaw backup create` stages its manifest outside the backed-up state payload when `TMPDIR` points to `~/.openclaw/tmp`, preventing duplicate `manifest.json` entries.

## Best Possible Solution

Review and land PR #75031 or an equivalent focused backup fix. The final implementation should choose a staging directory outside every included backup source, keep the manifest as a single root archive entry, and add a regression that sets `TMPDIR` under the state directory and asserts exactly one `manifest.json` in the created archive.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: Darwin service startup puts `TMPDIR` under the state directory, backup creation uses `os.tmpdir()` for manifest staging, and tar archives both the explicit manifest and the state source tree.

## Solution Assessment

Yes in direction. Using the existing OpenClaw temp-dir resolver, ideally with a containment guard against all included sources, is the narrow maintainable fix for this backup-specific staging bug.
