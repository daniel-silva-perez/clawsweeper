## Summary

Keep open. Current main still stages backup metadata under `os.tmpdir()` while macOS service startup sets `TMPDIR` under the OpenClaw state directory, and the backup tar call still archives both the explicit manifest and the state source tree. PR #75031 is open and references this issue as its closing target, so this issue should remain open until that PR or an equivalent focused fix lands.

## What This Changes

The report asks for `openclaw backup create` to stage its temporary manifest outside the archived state tree when macOS service `TMPDIR` points at `~/.openclaw/tmp`, preventing duplicate `manifest.json` entries.

## Best Possible Solution

Review and land PR #75031 or an equivalent focused change that uses a backup staging directory outside every included source tree, removes the ambient `os.tmpdir()` dependency for manifest staging, and adds a regression proving a state-dir `TMPDIR` archive has exactly one root manifest.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists from current source: macOS service `TMPDIR` resolves under state, backup creation writes `manifest.json` under ambient `os.tmpdir()`, and tar archives both that explicit manifest and the state tree. I did not create a live archive because this review was required to keep the checkout read-only.

## Solution Assessment

Yes in direction. The narrow maintainable fix is to use `resolvePreferredOpenClawTmpDir()` or a backup-specific equivalent outside all archived sources, plus a regression test for `TMPDIR` inside state.
