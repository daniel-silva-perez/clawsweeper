## Summary

Keep this issue open. The report is concrete, and current main still does not normalize Baidu/Qianfan-specific error payloads into a user-visible error message; it only has generic empty-error retry/fallback handling.

## Best Possible Solution

Keep this issue open. The best fix is a narrow Qianfan provider adapter or shared OpenAI-compatible error-normalization helper that promotes Baidu/Qianfan non-2xx and streamed error payloads into errorMessage, including error.code/message/type, top-level code/message/requestId, and legacy error_code/error_msg/error_message shapes. Add focused regression coverage in src/agents/provider-http-errors.test.ts and src/agents/openai-transport-stream.test.ts while preserving the existing silent-error retry as a fallback.
