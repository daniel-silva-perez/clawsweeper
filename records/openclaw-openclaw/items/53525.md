## Summary

Keep open. Current main still leaves a plausible Qianfan silent-error path unresolved: Qianfan is configured as an OpenAI-compatible completions provider, but the shared error formatter and stream parser still do not normalize all Baidu/Qianfan-style error payloads or convert top-level streamed error chunks into a visible `errorMessage`.

## What This Changes

The issue asks OpenClaw to surface Baidu Qianfan API error details instead of recording an error turn with no content and no user-visible error text.

## Best Possible Solution

Add a narrow shared OpenAI-compatible provider-error normalization path that recognizes Qianfan/Baidu nested and top-level error payloads, converts streamed error-only chunks into a sanitized `errorMessage`, preserves valid usage-only chunks, and covers the behavior with focused regression tests.

## Reproduction Assessment

Yes. The report includes a concrete transcript, and current-main source inspection gives a high-confidence focused reproduction of the parser gap without needing live Qianfan credentials; a captured live fixture would make the regression test stronger.

## Solution Assessment

Yes. The best fix is a generic shared transport/error-normalization change plus Qianfan regression coverage, because Qianfan is manifest-routed through `openai-completions`; adding bundled-provider IDs or ad hoc Qianfan policy in core would be less maintainable.
