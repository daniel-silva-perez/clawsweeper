## Summary

Keep open. Current main still has the reported web_fetch sanitizer gap: inline hidden-style handling is exact-pattern based and does not normalize opaque CSS colors, compare explicit foreground/background colors, or reject tiny nonzero font sizes. Because this is prompt-injection hardening with false-positive tradeoffs, it needs maintainer/security policy review rather than cleanup closure.

## What This Changes

The issue requests hardening web_fetch HTML visibility sanitization against near-invisible CSS patterns such as near-white text, matching foreground/background colors, and tiny nonzero font sizes.

## Best Possible Solution

Add conservative CSS declaration handling in the shared sanitizer: normalize common explicit opaque colors, remove elements whose explicit foreground and background colors normalize to the same value, and apply a maintainer-approved tiny-font threshold. Cover the basic and Readability extraction paths with targeted tests, and add changelog or security notes if maintainers classify this as user-facing hardening.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass sanitizeHtml HTML containing `style="color:white"`, `style="font-size:1px"`, or matching `color`/`background-color`; none of those declarations match the existing removal predicates, so the payload would remain in extracted content.

## Solution Assessment

No, not as a simple named-color regex patch. The maintainable direction is a small CSS declaration normalizer and explicit foreground/background comparison, plus a policy decision for tiny-font thresholds to avoid over-removing legitimate content.
