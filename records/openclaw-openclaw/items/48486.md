## Summary

Keep open. Current main still has the reported web_fetch sanitizer gap: hidden-style handling is exact-pattern based and does not normalize opaque CSS colors, compare explicit foreground/background colors, or reject tiny nonzero font sizes. Because this is prompt-injection hardening with false-positive tradeoffs, it should stay with maintainer/security review rather than stale cleanup.

## What This Changes

The issue asks OpenClaw to harden web_fetch HTML sanitization against near-invisible CSS patterns such as near-white text, tiny nonzero font sizes, and matching foreground/background colors.

## Best Possible Solution

Add conservative CSS declaration handling in the shared sanitizer: normalize common explicit opaque color forms, remove elements whose explicit foreground and background colors normalize to the same value, and apply a maintainer-approved tiny-font policy. Cover sanitizeHtml, the basic extraction fallback, and the bundled Readability plugin path with targeted tests; add changelog or security notes if maintainers classify this as user-facing hardening.

## Reproduction Assessment

Yes. The current source gives a high-confidence reproduction path: pass sanitizeHtml HTML containing `style="color:white"`, `style="font-size:1px"`, or matching `color`/`background-color`; none of those declarations match the existing removal predicates, so the payload would remain in extracted content.

## Solution Assessment

No, not as a simple named-color regex patch. The maintainable direction is a small CSS declaration normalizer and explicit foreground/background comparison, plus a policy decision for tiny-font thresholds to avoid over-removing legitimate content.
