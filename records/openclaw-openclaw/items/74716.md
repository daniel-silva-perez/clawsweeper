## Summary

Keep open. This is a protected maintainer-labeled XL PR that changes privileged macOS native process hosting, Gateway node protocol, Codex MCP injection, and package transfer behavior; current main does not already contain this native node-hosted MCP path. Static review found one blocking proxy session-scoping issue that should be fixed before maintainer/platform review decides whether to merge.

## What This Changes

The PR adds node-hosted MCP session protocol and loopback proxy support, macOS Computer Use package hosting/transfer, Codex thread MCP injection, generated Swift protocol models, docs, SDK/runtime types, and regression tests.

## Best Possible Solution

Keep the native node-hosted MCP architecture only if the proxy gives each Codex MCP client/thread an isolated native MCP session or another proven-safe session boundary, then complete maintainer/platform review of the protocol, macOS host, package validation, generated models, docs, and CI before merge.

## Reproduction Assessment

Not applicable as a feature PR, but the review finding has a high-confidence static reproduction path: two independent clients using the same `/mcp/node/<node>/<server>` URL are routed through the same `NodeMcpProxySession` key and can collide on JSON-RPC ids or initialization state.

## Solution Assessment

No. Routing permission-sensitive Computer Use through OpenClaw.app matches the stated product direction, but the current proxy-session key is too broad for independent Codex MCP clients; the safer fix is to key sessions per client/thread or mint unique loopback URLs for each injected thread config.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Scope node MCP proxy sessions per client:** `src/gateway/node-mcp-http-proxy.ts:377-383`
  - body: `NodeMcpHttpProxyManager` caches one `NodeMcpProxySession` per node/server, so every Codex thread configured with the same `/mcp/node/<node>/<server>` URL shares a single stdio MCP backend. Independent MCP clients commonly reuse request ids during `initialize`, and this proxy's pending-response map is keyed only by id, so concurrent or separate threads can collide or send a second initialization into an already-initialized backend. Please key the proxy by a per-thread/client session or issue unique loopback URLs so each Codex thread gets its own native MCP session.
  - confidence: 0.82
