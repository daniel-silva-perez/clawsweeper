## Summary

Keep open. Current main does not implement node-hosted MCP sessions, and the PR is a protected maintainer-labeled protocol/security-sensitive implementation slice. The latest force-pushed head fixes the earlier pair-request forwarding and `onclose` findings, but it still has a concrete pending-open close race that can leave a node-hosted MCP session active after the client closes.

## What This Changes

The PR adds additive gateway node-hosted MCP protocol fields/events, pairing approval/storage, node registry session routing, a node MCP client transport, generated Swift models, tests, and docs for Mac-hosted MCP stdio streams.

## Best Possible Solution

Keep the protocol slice on this branch, make session cancellation robust while an MCP open is still pending, clear or settle the matching registry state when close wins the race, then validate the targeted gateway/protocol tests and generated protocol artifacts before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Close sessions that are still opening:** `src/gateway/node-mcp-client-transport.ts:76-82`
  - body: The registry sends `node.mcp.session.open` and records an active session before `start()` resolves, but `close()` only sends `node.mcp.session.close` after `started` is true. If the MCP client closes while the node open is still pending, the node can finish launching and keep the session/process alive without a live client consuming output. Track the opening state and close/settle the registry session when `close()` wins this race.
  - confidence: 0.87

## Security Review

Status: needs_attention

Summary: The diff introduces privileged node-hosted MCP session plumbing, and the pending-open close race can leave a permission-sensitive node MCP process/session alive after caller cancellation.

Concerns:

- **[medium] Pending MCP opens are not cancelled on close:** `src/gateway/node-mcp-client-transport.ts:76`
  - body: Because `close()` only sends a node close frame after `started` is true, a caller that closes during the open handshake can abandon a node-hosted MCP session after the node has already received the open request. That matters for computer-use style MCP servers because process ownership and macOS permissions live on the node side.
  - confidence: 0.84
