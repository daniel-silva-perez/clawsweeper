## Summary

Keep this PR open. It has the protected `maintainer` label and the latest head still has a blocking lifecycle issue: an MCP open that times out is removed from gateway state without sending a node-side close, so a slow Mac host can still launch the permission-sensitive backend after the gateway has abandoned it.

## What This Changes

The PR adds node-hosted MCP session plumbing, Mac app Computer Use MCP hosting, Codex thread MCP config injection, pairing approval updates, generated Swift protocol models, tests, and docs for routing Codex Computer Use through OpenClaw.app.

## Best Possible Solution

Keep the branch open for maintainer/platform review, make every sent MCP open closeable on timeout as well as caller cancellation, add a regression test for the timeout path, then rerun targeted gateway/protocol checks plus exact-head CI before merge.

## Reproduction Assessment

Yes. Advertise a node MCP server, call `openMcpSession()` with a short timeout, never deliver `node.mcp.session.open.result`, and observe that the timeout path removes active state without emitting `node.mcp.session.close`.

## Solution Assessment

No. The architecture direction may be reasonable, but the current patch is not the best mergeable form until the sent-open timeout path either sends a close frame or remains closeable until the node acknowledges shutdown.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Close timed-out MCP opens on the node:** `src/gateway/node-registry.ts:348-350`
  - body: `openMcpSession()` sends `node.mcp.session.open`, but the timeout handler deletes pending and active state without sending `node.mcp.session.close`. A slow Mac node can still finish launching the permission-sensitive MCP backend, and later cleanup cannot reach it because `closeMcpSession()` no longer has an active session to close.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff introduces privileged node-hosted MCP process plumbing, and the open-timeout path can abandon a Mac-hosted MCP process after requesting launch.

Concerns:

- **[medium] Timed-out MCP opens are not closed on the node:** `src/gateway/node-registry.ts:348`
  - body: When a node-hosted MCP open times out, the gateway drops local session state without sending `node.mcp.session.close`; because the Mac node owns the permission-sensitive child process, a slow launch can continue without a live gateway client consuming or closing it.
  - confidence: 0.88
