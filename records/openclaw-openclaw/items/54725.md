## Summary

Keep this PR open. Current main still does not expose session identity to provider extra-param hooks, so the feature is not implemented, but the proposed patch has a contract bug because it advertises the fields through derived provider contexts that it does not populate at runtime.

## What This Changes

The PR adds optional `sessionKey` and `sessionId` fields to `ProviderPrepareExtraParamsContext` and threads them from embedded attempts into provider extra-param preparation.

## Best Possible Solution

Expose session identity through one coherent provider hook contract: either keep the type scoped to hooks that are actually populated, or pass the fields through the supported runtime-plan, embedded attempt, transport, wrapper, and related provider hook paths with focused contract tests and provider-doc updates if the public API expands.

## Reproduction Assessment

Yes for the review finding: code inspection shows the PR adds fields to a base context used by derived provider hooks, while the runtime context construction shown on current main only supplies the existing fields. A provider plugin reading the new fields from transport or stream-wrapper hooks would see `undefined`.

## Solution Assessment

No, not as written. The narrow maintainable fix is to either scope the new fields to the hook that is actually populated or thread them consistently through every advertised provider context, including the current runtime-plan path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Pass session fields through advertised contexts:** `src/plugins/types.ts:477-482`
  - body: Adding `sessionKey` and `sessionId` to `ProviderPrepareExtraParamsContext` also exposes them on `ProviderExtraParamsForTransportContext` and `ProviderWrapStreamFnContext`, but the patch only populates `prepareExtraParams`. Plugins reading the newly typed fields from transport or stream-wrapper hooks would receive `undefined` at runtime, so either thread the fields through those contexts too or keep the new fields scoped to the prepare hook.
