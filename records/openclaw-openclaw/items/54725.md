## Summary

Keep open. Current main still does not expose sessionKey/sessionId on ProviderPrepareExtraParamsContext, and the related request-metadata work is still open rather than shipped. The PR remains a plausible implementation candidate, but it needs a current-main rebase plus tests/docs and a privacy/product review for exposing stable session identifiers to provider plugins.

## Best Possible Solution

Keep this PR open and review it as part of the provider request-metadata work. The best implementation is an additive provider-plugin API update on current main: extend the relevant provider contexts, thread sessionKey/sessionId through the embedded-runner runtime-plan and direct apply paths, add focused tests proving provider hooks receive the fields, and document the privacy-safe attribution pattern for proxies such as LiteLLM/OpenRouter.

## Risks / Open Questions

- The PR as submitted predates current main's runtimePlan.resolveExtraParams path; a rebase likely needs to thread sessionKey/sessionId through src/agents/runtime-plan/build.ts and compact/runtime-plan call sites, not only the older applyExtraParamsToAgent call shape.
- Exposing stable sessionKey values to provider plugins can enable useful attribution, but also makes it easier for provider code to forward channel/session identifiers to third-party services. Maintainers should decide whether docs recommend raw sessionKey, sessionId, hashing, or opt-in config.
- I did not find a current test that asserts provider prepareExtraParams or wrapStreamFn receives sessionKey/sessionId, so regression coverage should be added before merge.
