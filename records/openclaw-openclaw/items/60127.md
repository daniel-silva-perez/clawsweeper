## Summary

Keep open. Current main still documents a single trusted-operator Gateway model and has adjacent multi-agent, per-agent tool policy, sender/session, and operator-scope controls, but it does not implement the requested shared-server tenant isolation model with tenant identity propagation, tenant RBAC, per-tenant quotas, tenant-scoped plugin enablement, tenant-attributed audit, or hard memory/workspace boundaries. Related open issues cover useful slices, but none fully supersedes this tenant-isolation request.

## What This Changes

Request secure shared-server multi-tenancy in one OpenClaw instance with tenant isolation, RBAC, scoped resources, and strict data boundaries instead of one deployment per user.

## Best Possible Solution

If shared-server tenancy is in scope, start with an RFC/design that threads tenant identity through Gateway auth/request context, protocol, sessions_send/chat paths, agent/workspace/memory/plugin config resolution, quotas, audit logging, and cross-tenant denial tests/docs. If it is out of scope, document that hostile shared-Gateway tenancy is unsupported and split narrower API-seam issues for external managed-service implementations.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is a source/docs check: the protocol has no tenant identity, docs disclaim hostile shared tenancy, and related controls are session/agent/operator scoped rather than tenant scoped.

## Solution Assessment

Unclear because there is no proposed patch here. A small proxy/plugin or per-agent setting is not the best complete solution for strict tenant isolation unless core first exposes tenant identity and tenant-scoped state/config/memory/plugin contracts; the safer path is maintainer RFC/security review before implementation.
