## Summary

Keep open. Current `main` still does not emit the requested Gateway v2 promotion metadata surface; the relevant Gateway protocol, agent RPC, OpenAI-compatible Chat Completions, OpenResponses, and shared usage paths remain narrower than the requested config/routing/timing/reasoning-token envelope. The related diagnostics and trajectory work is useful context for redaction, but it is an export/support workflow rather than the live benchmark-facing response metadata requested here.

## What This Changes

The issue requests a Gateway v2 response metadata envelope with config lineage, route and policy identifiers, timing metrics, and token usage including reasoning tokens for benchmark smoke tests.

## Best Possible Solution

Define a redacted, additive Gateway v2 metadata envelope in the typed protocol boundary, then wire it consistently through the benchmark-facing live response surfaces. Reuse diagnostics/trajectory internals only where their privacy model is explicit, and document missing-provider behavior, token semantics, and backward compatibility.

## Reproduction Assessment

Not applicable as a runtime bug reproduction; this is an instrumentation feature request. Source inspection gives a high-confidence verification path because the current schemas and response builders do not contain the requested fields.

## Solution Assessment

Unclear because there is no patch yet. The suggested routing-layer envelope is directionally right, but the maintainable fix needs maintainer-approved redaction, versioning, target surfaces, and provider-specific token semantics before implementation.
