## Summary

Keep open. Current main still does not emit the requested promotion-grade Gateway v2 routing/config/timing/token metadata surface on the WS chat schema, agent RPC result path, OpenAI compatibility endpoint, or OpenResponses endpoint. Related diagnostics and trajectory work is useful context, but it is not the requested response-envelope contract.

## Best Possible Solution

Keep this open for a deliberate additive Gateway v2 metadata design. Define a redacted response metadata envelope for config lineage, route class/fallback chain, policy id/version, gateway version, bounded decision trace or trace id, route/first-token/total timing, and token usage including reasoning tokens when available, then wire it through the WS chat path, agent RPC result/wait path, and the HTTP compatibility endpoint(s) the benchmark targets with docs and focused contract tests.

## Risks / Open Questions

- requested_config, resolved_config, actual_config, and decision_trace fields could expose secrets, auth profile data, policy internals, prompts, local paths, or runtime details unless the contract defines redaction and access scope up front.
- This is a public Gateway protocol and compatibility endpoint change, so an ad hoc metadata blob would risk schema/client/docs drift.
- reasoning_tokens is provider-specific and is not currently part of the shared normalized usage shape, so the implementation needs explicit fallback behavior when providers omit it.
