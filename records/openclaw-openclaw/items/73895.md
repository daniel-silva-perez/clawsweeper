## Summary

Keep open. Current main still lacks the requested promotion-grade Gateway v2 metadata response surface; the existing WebSocket chat, agent RPC, OpenAI-compatible HTTP, and OpenResponses paths expose standard result/usage shapes but not routing/config lineage, fallback chain, policy identity, decision trace, or timing fields.

## What This Changes

The issue requests an additive Gateway v2 metadata response surface for routing/config lineage, policy, timings, fallback chain, gateway version, and token usage including reasoning tokens.

## Best Possible Solution

Design an additive Gateway v2 metadata envelope owned by the Gateway routing layer, with redacted config lineage, route class, fallback chain, policy id/version, gateway version, bounded trace id/details, timing fields, and extended usage. Wire it through the benchmark-targeted WS/RPC/HTTP surfaces with docs and contract tests while preserving current response compatibility.

## Reproduction Assessment

Yes. This is a feature gap rather than a flaky runtime bug, and the absence is reproducible by inspecting the current response schemas/builders and exact field searches on current main.

## Solution Assessment

Unclear until maintainers choose the public contract. The requested direction is plausible, but the best maintainable solution should be a typed, redacted, additive metadata envelope rather than ad hoc fields copied from trajectory internals.

## Risks / Open Questions

- Config lineage and decision traces could expose secrets, auth profile details, local paths, prompts, policy internals, or provider payloads unless the contract defines redaction and scope gating first.
- This is a public Gateway protocol and HTTP compatibility surface, so the change likely needs additive schema/docs/client updates rather than an opaque metadata blob.
- Reasoning-token availability is provider-specific, and the current shared usage shape does not carry reasoning token counts.
- The issue references a benchmark-side harness outside this repository, so maintainers still need to confirm which Gateway endpoint(s) that harness treats as canonical.
