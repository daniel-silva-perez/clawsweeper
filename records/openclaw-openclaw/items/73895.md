## Summary

Keep open. Current main still lacks the requested promotion-grade Gateway v2 response metadata: exact field search plus the WS chat schema, agent RPC result path, and OpenAI/OpenResponses HTTP handlers show no emitted config lineage, route class, fallback chain, policy, timing, decision trace, or reasoning-token fields. The #72936 diagnostics and trajectory work is useful adjacent context, but it is support/export infrastructure rather than the live response envelope this issue asks for.

## What This Changes

Request an additive Gateway v2 metadata surface that emits requested/resolved/actual config lineage, routing and fallback decisions, policy/version identifiers, timing fields, and token usage including reasoning tokens so the benchmark smoke harness can run without stubs.

## Best Possible Solution

Define a redacted, additive Gateway v2 metadata envelope in the typed protocol layer, then wire it consistently through agent RPC, WS chat/final events if needed, and the HTTP compatibility endpoints the benchmark targets. Reuse diagnostics or trajectory internals only where their privacy model is explicit, and add focused contract tests and docs for missing-provider fields and backward compatibility.

## Reproduction Assessment

Not applicable as a runtime bug reproduction; this is an instrumentation feature request. Current-main source inspection gives a high-confidence verification path because the relevant schemas and response builders do not contain the requested fields.

## Solution Assessment

Unclear for implementation because there is no patch yet. The suggested routing-layer envelope is directionally right, but the best solution needs maintainer-approved redaction, versioning, and client/API scope decisions before code is written.
