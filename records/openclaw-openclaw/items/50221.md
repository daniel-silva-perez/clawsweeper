## Summary

Keep open. The underlying scope-hardening concern is valid, but this branch is not mergeable as-is: it over-constrains write-scoped `chat.send` callers, conflicts with current-main directive hardening shape, and needs a fresh plugin config-write design against current runtime APIs. The PR discussion already identifies the command-capability design gap and asks for maintainer direction.

## What This Changes

The PR changes internal gateway scope handling by deriving `chat.send` command authorization from `operator.admin`, blocking non-admin `/exec` default persistence, propagating gateway scopes into plugin command execution, and adding regression tests for those paths.

## Best Possible Solution

Preserve least-privilege hardening without making every `chat.send` slash command admin-only: add a command-level capability or required-scope model, keep read-only commands available to `operator.write`, preserve current main’s centralized directive persistence gate, and guard the current plugin config mutation APIs at the shared write boundary.

## Reproduction Assessment

Yes. The static path is high confidence: the PR sets `CommandAuthorized=false` for `operator.write` `chat.send` callers, `resolveCommandAuthorization` then makes `command.isAuthorizedSender=false`, and read-only commands such as `/status`, `/help`, `/commands`, and `/config show` are rejected before their narrower gates run. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No. The branch’s admin-default `chat.send` check is too broad for the current command model, and the plugin config-write hardening must be updated for current main’s `mutateConfigFile` / `replaceConfigFile` APIs. A scope-aware command capability layer is the safer maintainable direction.
