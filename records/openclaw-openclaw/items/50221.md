## Summary

Keep this PR open. Current main has absorbed part of the internal-scope cleanup, especially shared `/exec` persistence checks and command-specific admin gates, but it has not implemented the PR's central `chat.send` authorization change or the proposed plugin runtime config sink guard. The PR also still carries a real least-privilege regression risk: setting `CommandAuthorized=false` for all `operator.write` `chat.send` callers would block normal slash-command authorization before command-specific policy can run.

## Best Possible Solution

Keep this PR open for maintainer direction. The best path is to preserve main's command-specific least-privilege model, use command capability or `requiredScopes` metadata where commands need stricter scopes, decide explicitly whether plugin runtime config writes need a generic request-scoped sink guard, and revise the PR plus regression tests around that chosen policy.

- Because this PR changes auth-sensitive gateway and plugin runtime behavior, it needs maintainer product/security direction rather than conservative cleanup closure.
