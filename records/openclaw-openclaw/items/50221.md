## Summary

Keep open. Current main has absorbed part of the internal-scope cleanup, especially `/exec` persistence and several command-specific admin gates, but it has not definitely implemented this PR’s intended change. The proposed `chat.send` authorization behavior also has unresolved review feedback because it can over-block least-privilege `operator.write` clients before command-specific policy runs.

## Best Possible Solution

Keep the PR open for maintainer direction rather than closing it as implemented. The best path is to preserve main’s command-specific least-privilege model, add explicit command capability or required-scope metadata where needed, decide whether plugin runtime config writes need a generic sink-level guard or plugin-declared scopes, and update the PR/tests accordingly before merge.

## Risks / Open Questions

- Merging the PR as-is could break least-privilege internal clients by denying all session-scoped slash commands from `operator.write` callers before read-only command paths can run.
- The PR touches authorization-sensitive gateway/plugin paths. The provided file list does not show CI, dependency, lockfile, release-script, or downloaded-artifact changes, so the main security concern is authorization behavior rather than supply-chain drift.
- Current main’s plugin-command `requiredScopes` support reduces some risk, but it does not provide the PR’s sink-level guard for arbitrary plugin runtime config writes.
