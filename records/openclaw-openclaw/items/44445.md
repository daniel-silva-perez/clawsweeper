## Summary

Keep open. Current main still stores raw user transcript text in session-memory output, so the central metadata-pollution bug is valid. The PR is not ready as-is because it mixes that fix with scheduled reset hook dispatch now covered by open PR #61675, and the sanitizer approach has unresolved fidelity regressions around message_id-looking user text, slash-prefixed real content, and timestamp-prefixed transcripts.

## Best Possible Solution

Keep this PR open, but narrow the implementation to session-memory transcript sanitization. The fix should live around `src/hooks/bundled/session-memory/transcript.ts`, reuse existing inbound metadata, envelope, and message-id cleanup helpers, preserve user-authored content, and add focused regression tests. Leave daily/idle reset hook dispatch to #61675 or another dedicated lifecycle-hook PR.

## Risks / Open Questions

- Current main still persists channel-injected metadata into session-memory files for Feishu-style user turns.
- This PR mixes the metadata fix with scheduled reset hook dispatch; that lifecycle behavior is separately tracked by #61675 and has broader plugin-hook semantics.
- The PR's generic message_id stripping can remove a real user-authored leading `[message_id: ...]` line after metadata blocks unless the cleanup is scoped carefully.
- Applying slash-command filtering after sanitization can omit legitimate slash-prefixed user content from metadata-wrapped channels.
- Timestamp-prefixed `BodyForAgent` transcripts need coverage because a leading timestamp can prevent leading-sentinel metadata stripping from running.
