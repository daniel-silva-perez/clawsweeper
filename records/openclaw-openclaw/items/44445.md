## Summary

Keep open. Current main still writes raw user transcript text into session-memory output, so the metadata pollution bug is real, but this PR is not ready as-is because it mixes the sanitizer with scheduled reset hook dispatch and has concrete content-loss regressions in the proposed cleanup path.

## What This Changes

The PR strips leading inbound metadata from user turns before session-memory stores them and adds scheduled reset hook dispatch plus regression tests.

## Best Possible Solution

Keep the metadata fix focused in the session-memory transcript extraction layer, using the existing inbound metadata, envelope, and message-id cleanup primitives with tests for Feishu blocks, timestamp/envelope variants, slash-prefixed real content, and user-authored message-id-looking lines. Leave daily/idle reset hook dispatch to #61675 or a dedicated lifecycle-hook change.

## Reproduction Assessment

Yes. A source-level reproduction is clear: a JSONL user message containing Feishu-style inbound metadata will be read by `getRecentSessionContent()` and appended raw to the session-memory markdown on current main.

## Solution Assessment

No. Sanitizing session-memory user text is the right direction, but this PR is not the best form because it changes scheduled reset command-hook dispatch and its cleanup can drop valid user content; the safer path is a narrow transcript-layer sanitizer plus focused regression tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Separate automatic rollovers from /new hook dispatch:** `src/auto-reply/reply/get-reply.ts:356-368`
  - body: When `scheduledResetTriggered` is true, this emits `command:new` hooks for the first ordinary message after a stale daily reset even though no `/new` or `/reset` command was sent, and it bypasses the authorized-sender guard used by the manual path. That can run reset automations and plugin hooks on normal traffic; keep the lazy reset hook work in the dedicated #61675 path instead of bundling it here.
  - confidence: 0.88
- **[P2] Preserve slash-prefixed user text after cleanup:** `src/hooks/bundled/session-memory/handler.ts:83-84`
  - body: The slash-command filter now runs after metadata stripping, so a legitimate channel message whose real content starts with `/` is dropped from session-memory. For example, a Feishu user asking about `/var/log` would be kept before this change because the stored text started with metadata, but would now disappear from the memory artifact.
  - confidence: 0.86
- **[P2] Preserve user-authored message_id-looking lines:** `src/auto-reply/reply/strip-inbound-meta.ts:240-245`
  - body: This loop removes any standalone `[message_id: ...]` line immediately after the metadata blocks, even when that line is the user's actual first line. The cleanup should only remove a known transport hint, such as one matched to injected metadata, rather than treating all message_id-looking user text as metadata.
  - confidence: 0.8
