## Summary

Keep open. Current main still derives Docker sandbox and sandbox-browser container names from the sandbox scope key only, so multiple co-hosted instances with the same default agent id can still choose the same Docker name. `workspaceDir` is included in the config hash, but not in the name, and a hot hash mismatch logs a recreate hint while continuing to reuse the existing container.

## Best Possible Solution

Keep this issue open and implement the core sandbox fix in `src/agents/sandbox/docker.ts` and `src/agents/sandbox/browser.ts`: include stable instance identity such as the resolved OpenClaw state/workspace root or HOME-derived identity in the slug/hash input, add regression tests proving two different HOME/workspace roots with the same `agent:main` scope produce different container names, and document the one-time container-name migration/recreate behavior.

## Risks / Open Questions

- Closing now would leave a plausible cross-instance workspace contamination bug untracked for Docker sandbox users running multiple OpenClaw HOME directories on one host.
- A fix should account for migration/recreate behavior because adding instance identity to the name will orphan or supersede existing stable container names.
