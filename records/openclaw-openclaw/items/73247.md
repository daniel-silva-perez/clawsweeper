## Summary

Keep this draft PR open. Current main still leaves the Slack `deliverReplies()` path outside `message_sending`, so the hook-coverage half addresses a real gap, but the proposed bundled guard is not merge-ready because it ignores Slack account scoping for credentials/cache and needs maintainer product/security review before becoming a stock plugin.

## What This Changes

The PR wires `message_sending` into Slack `deliverReplies()` sends and adds a bundled `slack-addressee-guard` plugin, manifest, and tests.

## Best Possible Solution

Land or adapt a narrow Slack hook-coverage fix with tests for normal replies, block replies, media replies, and Slack streaming/fallback behavior. Treat the addressee guard separately: either move it to a non-core plugin path or make it account-scoped, documented, labeler-covered, and owned as an official Slack safety plugin.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: register a `message_sending` hook, route a Slack inbound reply through current main's locally built Slack dispatcher, and observe that `deliverReplies()` reaches `sendMessageSlack()` without calling the hook runner.

## Solution Assessment

No. The `deliverReplies()` hook wiring is a plausible part of the fix, but bundling the addressee guard in the same PR with global-token and account-agnostic state is not the narrowest maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Use account-scoped Slack credentials for guard lookups:** `extensions/slack-addressee-guard/index.ts:294`
  - body: The guard resolves one `SLACK_BOT_TOKEN` during registration and then handles every Slack `message_sending` event, even though the hook context carries `accountId` and Slack supports per-account bot tokens. In a multi-account gateway, a reply for a non-default account will query `conversations.replies` with the default/env token, so the guard can fail open or make decisions from the wrong workspace. Resolve the token for the event account before bundling this plugin.
  - confidence: 0.86
- **[P2] Scope the last-human cache by Slack account:** `extensions/slack-addressee-guard/index.ts:211-212`
  - body: The cache key is only `channelId:threadTs`, but this hook can run for multiple Slack accounts. A lookup from one account can be reused for another account with the same channel/thread identifiers, causing rewrite or cancel decisions against an unrelated thread. Include the account id in the parsed target/cache key.
  - confidence: 0.81
- **[P3] Remove or implement the advertised logPath setting:** `extensions/slack-addressee-guard/openclaw.plugin.json:20`
  - body: The manifest exposes `logPath` and the config type accepts it, but the implementation never reads or writes that setting. Users can configure a JSONL decision log that silently does nothing; either implement it through an approved runtime helper or drop the schema/UI field.
  - confidence: 0.82

## Security Review
