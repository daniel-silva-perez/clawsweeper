## Summary

Keep this draft PR open. Current main still does not run `message_sending` for Slack reply delivery, so the core hook-coverage problem remains real, but the proposed branch is not merge-ready because it mixes that focused fix with a new bundled guard plugin that has concrete correctness and credential-scope issues needing maintainer review.

## What This Changes

This draft wires `message_sending` hooks into Slack `deliverReplies()` send paths and adds a bundled `slack-addressee-guard` plugin with tests.

## Best Possible Solution

Adapt or land the Slack hook coverage as a focused change with tests for normal text, blocks, media, streaming fallback, and preview/final edit behavior. Split the addressee guard into a separate maintainer decision: either move it to ClawHub or harden it as an official Slack plugin with account-scoped credentials, full Slack conversation id support, leading-mention-only matching, docs, labeler/config parity, and focused tests.

## Reproduction Assessment

Yes. A high-confidence reproduction exists by source inspection on current main: Slack reply delivery calls `sendMessageSlack()` directly through `deliverReplies()`, while the shared `message_sending` hook insertion only exists in wrapper dispatcher helpers Slack does not use for this prebuilt dispatcher path.

## Solution Assessment

No, not as currently proposed. The Slack `deliverReplies()` hook wiring is the right narrow direction, but the branch is not the best complete solution while it mixes in a bundled addressee guard with parser, config, and credential-scope problems and leaves streaming/preview coverage open.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Only rewrite leading target mentions:** `extensions/slack-addressee-guard/index.ts:351`
  - body: The guard enters whenever the target user is mentioned anywhere, but `repairContent()` prepends the last human when the target mention is not leading. A normal reply like `I checked with <@target>` would be rewritten to start with the last human even though it is not the accidental leading-address case described by the plugin. Gate this path on a leading target mention before fetching or rewriting.
  - confidence: 0.9
- **[P2] Accept private Slack conversation ids:** `extensions/slack-addressee-guard/index.ts:130`
  - body: This rejects every Slack conversation id that does not start with `C`, so private channels, MPIMs, and DMs using `G...` or `D...` ids silently skip the guard even though `conversations.replies` and the Slack channel support those surfaces. Accept the Slack conversation id prefixes that the channel can deliver replies to.
  - confidence: 0.86
- **[P3] Remove the unused logPath setting:** `extensions/slack-addressee-guard/openclaw.plugin.json:20`
  - body: The manifest exposes `logPath` and the UI help promises optional JSONL decision logging, but the plugin implementation never reads `cfg.logPath` or writes those records. Either implement the logging behavior or remove the config field before shipping the bundled plugin.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The PR adds no new dependencies, but the new bundled guard has a concrete account-scoped Slack credential handling concern.

Concerns:

- **[medium] Use account-scoped Slack credentials:** `extensions/slack-addressee-guard/index.ts:87`
  - body: The guard resolves only `process.env.SLACK_BOT_TOKEN` and does not use the Slack account token associated with `ctx.accountId` or the delivery path. In multi-account or SecretRef-only installs, a bundled plugin could query `conversations.replies` with the wrong workspace credential or fail open despite the Slack account being configured.
  - confidence: 0.84
