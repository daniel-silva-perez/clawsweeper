## Summary

Keep this PR open. Current main still does not implement the core Slack behavior the PR targets: Slack agent reply delivery builds its own dispatcher and ultimately sends through `deliverReplies()`, where the block, text, and media paths call `sendMessageSlack()` directly without `message_sending`. The PR is a plausible implementation candidate for a real Slack hook coverage gap, but it is draft and mixes that core fix with a new bundled addressee-guard plugin that needs maintainer product/security review or a split.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to land or adapt the Slack `message_sending` coverage fix with focused tests for text, blocks, media, and relevant streaming/fallback paths. Treat the addressee guard as a separate product/security decision: either move it to a ClawHub/plugin follow-up, or make it a generic official Slack safety plugin with account-scoped credentials, accurate schema/docs, and clear bundled-plugin ownership.

## Risks / Open Questions

- The PR bundles two separable changes: a broadly useful Slack hook-coverage fix and a deployment-specific safety plugin. Maintainers may want these split before merge.
- The proposed `slack-addressee-guard` uses global `SLACK_BOT_TOKEN` rather than the active Slack account credential, which is risky for multi-account Slack installs.
- The plugin schema exposes `logPath`, but the implementation in the PR diff does not write decision logs, creating config/UI drift if shipped as-is.
- The plugin checks whether content mentions the target anywhere, while the stated behavior is about leading accidental mentions; in cancel mode this could drop messages with a non-leading legitimate mention.
- The plugin's channel parser accepts only `C...` Slack channel IDs, while existing Slack/thread code recognizes broader Slack conversation IDs such as `D`, `G`, `U`, and `W`; coverage may be incomplete for DMs/private conversations.
- Slack streaming/final-preview edit paths should be checked before merge so `message_sending` coverage is neither partial nor double-fired.
