## Summary

Keep this PR open. Current main still lacks a blocked-key check in `resolveConfigPath`, and the submitted patch is a narrow security hardening change that matches the repository's existing `isBlockedObjectKey` pattern; it should go through maintainer review and normal validation rather than be closed.

## What This Changes

This PR imports `isBlockedObjectKey` into `src/shared/config-eval.ts` and makes `resolveConfigPath` return `undefined` when a dot-path segment is `__proto__`, `prototype`, or `constructor`.

## Best Possible Solution

Land a tested version of the guard in `src/shared/config-eval.ts`, using `isBlockedObjectKey` before dynamic property access and adding focused `src/shared/config-eval.test.ts` cases for blocked segments; no broader config-path API change appears necessary.

## Reproduction Assessment

Yes. By source inspection on current main, `resolveConfigPath` indexes raw dot-path segments on ordinary objects, so paths such as `constructor` or `__proto__.toString` can resolve inherited prototype values instead of returning `undefined`.

## Solution Assessment

Yes. The proposed inline `isBlockedObjectKey` check before property access is the narrowest maintainable fix and matches `parseConfigPath`; the safer landing version should include the missing regression test.
