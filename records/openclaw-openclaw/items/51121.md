## Summary

Keep this PR open. Current main still has the un-cleared 500 ms timeout in `IMessageRpcClient.stop()`, so the intended leak cleanup is not implemented on main. The PR is narrow and useful, but it needs revision because the current lifecycle keeps `closed` as an instance-level promise that stays resolved across restart cycles, matching the review concern that clearing the new timer can remove the fallback kill for a reused client.

## Best Possible Solution

Revise and keep this PR as the implementation candidate: fix `IMessageRpcClient.stop()` so the stop timeout is cleared when the child closes first, while making the close waiter per spawned child or otherwise preserving fallback SIGTERM across repeated start/stop cycles. Add focused fake-timer tests for child-close-before-timeout, timeout-kills-hung-child, and restart-then-stop behavior, then run the targeted iMessage plugin test lane.
