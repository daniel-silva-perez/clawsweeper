## Summary

Keep this PR open. Current main still has the un-cleared iMessage stop timeout, so the intended leak fix is not implemented on main. The PR is a plausible narrow fix candidate, but the existing review concern remains: because `IMessageRpcClient.closed` is instance-scoped, clearing the new stop timer in `finally` can cancel the fallback `SIGTERM` on repeated start/stop cycles unless the close waiter is reset per child.

## Best Possible Solution

Revise this PR rather than close it: make the close waiter/reset state per spawned child, then clear the stop timeout in `finally` without losing fallback `SIGTERM` on repeated start/stop cycles. Add focused fake-timer tests for child-close-before-timeout, timeout-kills-hung-child, and restart-then-stop, then run the targeted iMessage plugin test lane.
