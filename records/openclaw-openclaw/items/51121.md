## Summary

Keep this PR open. Current main still has the un-cleared iMessage stop timeout, and the PR is a useful narrow fix candidate, but the current patch needs revision because `IMessageRpcClient.closed` is still an instance-level promise that remains resolved across client reuse and can cancel the fallback SIGTERM on a later stop.

## Best Possible Solution

Revise this PR rather than close it: make the close waiter per spawned child or otherwise reset/scope it so `stop()` can clear the timeout when the child closes first while preserving fallback SIGTERM across repeated start/stop cycles. Add focused fake-timer tests for child-close-before-timeout, timeout-kills-hung-child, and restart-then-stop, then run the targeted iMessage plugin test lane.
