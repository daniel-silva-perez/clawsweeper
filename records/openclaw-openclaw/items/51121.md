## Summary

Keep this PR open. Current main still has the un-cleared iMessage stop timeout, and the PR remains a plausible fix candidate, but it should be revised because `IMessageRpcClient.closed` is instance-scoped and can already be resolved on a reused client, which makes the PR's `finally` cleanup cancel the fallback SIGTERM for a later hung child.

## Best Possible Solution

Revise this PR rather than close it: make the close waiter/reset state per spawned child, then clear the stop timeout in a `finally` without losing fallback SIGTERM on repeated start/stop cycles. Add focused fake-timer tests for child-close-before-timeout, timeout-kills-hung-child, and restart-then-stop, then run the targeted iMessage plugin test lane.
