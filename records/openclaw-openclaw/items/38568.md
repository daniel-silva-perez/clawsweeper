## Summary

Keep this issue open. Current main still builds the system prompt Runtime line from static runtime metadata only; context usage is available through status/session_status, which is the workaround the issue asks to avoid.

## Best Possible Solution

Implement this in core prompt assembly. Add an explicit context usage object to RuntimeInfoInput or adjacent system prompt params, populate it from the effective contextTokenBudget and current prompt/context token estimate before buildEmbeddedSystemPrompt/buildAgentSystemPrompt runs, render a compact Runtime field such as context=49% or context=98k/200k (49%), and add focused tests for buildRuntimeLine plus embedded and CLI wiring while leaving status/session_status behavior intact.

## Risks / Open Questions

- The implementation should use a fresh pre-request prompt/context snapshot, not after-turn provider usage that can lag by one turn.
- The denominator should be the effective runtime context token budget, including contextTokens/config caps, not only a model-native contextWindow.
- Because this field is intentionally volatile, prompt-cache behavior and byte stability around the existing cache boundary need focused tests.
- CLI-backed runs may not have the same live token snapshot as embedded runs; the implementation needs a defined omit/fallback behavior there.
