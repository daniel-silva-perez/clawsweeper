## Repair Work Prompt

Implement context-window usage injection into the system prompt Runtime line for this issue. Current main exposes context usage through /status and session_status, but RuntimeInfoInput/buildRuntimeLine plus embedded and CLI prompt wiring do not pass or render it.

Expected fix boundary:
- Add a small typed optional context usage value to RuntimeInfoInput or adjacent system prompt params, such as used tokens plus budget and/or a computed percent.
- Render a compact Runtime field such as `context=49%` or `context=98k/200k (49%)` in buildRuntimeLine only when reliable finite data is available; omit it instead of guessing.
- Populate embedded runs from the effective runtime context budget and a fresh pre-request prompt/context estimate. Do not derive the display from stale last-turn provider totals when those can include output, cached totals, or multiple tool-loop calls.
- Define CLI behavior explicitly: populate only from a real available snapshot, otherwise omit the field.
- Keep the volatile field below the prompt cache boundary and add tests proving the stable prefix is not churned.
- Leave status/session_status behavior intact except for sharing a formatter if that keeps output consistent.
- Coordinate with Runtime-line identifier proposals #40044 and #66312 so fields do not conflict.
- Update docs/concepts/system-prompt.md and docs/reference/token-use.md if behavior changes, and add a single-line CHANGELOG.md entry under the active Changes section with appropriate Thanks @blortski attribution.

Do not add plugin-specific behavior, startup migrations, broad prompt rewrites, dependency changes, or unrelated status output churn.

## Risks / Open Questions

- The implementation should use a fresh pre-request prompt/context estimate; last-turn provider usage can lag and may include output or tool-loop totals.
- The denominator should be the effective runtime context token budget, including contextTokens/config caps, rather than only the model-native contextWindow.
- Because the new field is intentionally volatile, tests should prove it stays below the prompt cache boundary and does not churn the stable prefix.
- CLI-backed runs may not have a reliable live token snapshot, so the fix needs a defined omit or fallback behavior there.
