## Summary

Keep open. Current main still lacks the requested Plugin SDK workflow seams, and the PR is the active implementation surface, but the inspected head still has a concrete scope-compatibility regression in `plugins.sessionAction` that should block merge until fixed.

## What This Changes

This PR adds generic Plugin SDK workflow seams for session actions, host-mediated attachments, scheduled session turns, finalize retry support, priority injections, Gateway/Swift protocol updates, advanced workflow fixtures, and QA hardening.

## Best Possible Solution

Fix the session-action required-scope check so it uses the same scope implication rules as Gateway authorization, especially `operator.write` satisfying `operator.read`, then complete maintainer review and exact-head validation for the public SDK/protocol surface before landing.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to register a plugin session action with `requiredScopes: [READ_SCOPE]` and invoke it with a client whose scopes are only `[WRITE_SCOPE]`; Gateway pre-auth accepts the request, but the handler rejects it at the exact-match required-scope check.

## Solution Assessment

No. The PR is the right implementation direction for the missing workflow seams, but the current head is not merge-ready until the action-level scope check matches the existing Gateway authorization contract and has regression coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Treat write scope as satisfying read-scoped actions:** `src/gateway/server-methods/plugin-host-hooks.ts:102`
  - body: The action-specific authorization check requires an exact scope match, so a client with only `operator.write` is rejected for a plugin action declaring `requiredScopes: [operator.read]`. Gateway pre-auth now lets any operator scope reach this handler, and the generic Gateway auth model treats write as read-compatible, so this creates an inconsistent authorization regression for read-only plugin actions. Use the same read/write implication here and add handler coverage for write-scoped clients invoking read-scoped actions.
  - confidence: 0.91
