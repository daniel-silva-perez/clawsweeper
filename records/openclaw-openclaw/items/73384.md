## Summary

Keep open. Current main still lacks the workflow-seam surface this PR is adding, and the PR is active/unmerged rather than obsolete. The latest PR head fixes the previously reported read/write scope regression, but it still has a blocking attachment-delivery config propagation bug that should be patched before merge.

## What This Changes

This PR adds Plugin SDK workflow seams for session actions, host-mediated attachments, scheduled session turns, finalize retry, priority injections, Gateway/Swift protocol updates, advanced workflow fixtures, and QA hardening.

## Best Possible Solution

Keep this PR open, patch the attachment delivery config propagation, then complete maintainer API/security review and exact-head validation before landing the workflow seams.

## Reproduction Assessment

Yes. On PR head, load a bundled plugin with a registry-scoped config/session store and call sendSessionAttachment; route lookup can use that config, but the actual send path resolves channel/account settings through global runtime config because cfg is not passed to sendMessage.

## Solution Assessment

No. The implementation direction is the right generic SDK shape, but the current patch is not merge-ready until the attachment send path uses the same registry-scoped config as the route lookup and has regression coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Pass registry config to attachment delivery:** `src/plugins/host-hook-workflow.ts:161-170`
  - body: `sendPluginSessionAttachment` now threads `params.config` into `extractDeliveryInfo`, but the actual `sendMessage` call omits `cfg: params.config`. In non-default session-store, channel, or multi-workspace contexts, route lookup can succeed from the plugin registry config while outbound delivery resolves against the process-global config, causing attachments to fail or use the wrong channel/account. Pass the same config through and assert the mocked send receives it.
  - confidence: 0.87

## Security Review

Status: needs_attention

Summary: The diff has one concrete privacy-sensitive delivery concern: host-mediated attachments may be sent using the wrong runtime config.

Concerns:

- **[medium] Attachment delivery may use global config:** `src/plugins/host-hook-workflow.ts:161`
  - body: Because the attachment helper drops the registry-scoped config before calling sendMessage, a multi-workspace or isolated plugin context can route an attachment using one config but deliver it through another config's channel/account resolution. That can fail delivery or send sensitive artifacts to the wrong configured target.
