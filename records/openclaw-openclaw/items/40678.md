## Summary

Keep open. Current main still preserves TUI session isolation and does not provide an opt-in observe or mirror mode; the remaining work is a real feature gap that needs maintainer design because it expands transcript visibility across channel and session boundaries.

## What This Changes

The issue requests an opt-in TUI mode that can display iMessage, Telegram, cron, and other channel activity for the same agent without reverting isolated `/new` TUI sessions.

## Best Possible Solution

Add an explicit authorized TUI observe surface that keeps default `/new` isolation, separates read-only observation from interactive sending, labels channel/session/run origin, supports recent aggregate previews plus live updates, and documents the privacy model.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: events or history for a non-current session are ignored because live `chat` handling requires `state.currentSessionKey` and history loads only that same key. I did not run a live multi-channel setup because this review was read-only.

## Solution Assessment

Unclear. The requested UX is reasonable, but the best implementation is not a bare flag that widens event filters; it should be an explicit observe/read authorization path coordinated with #41548 and #73402.
