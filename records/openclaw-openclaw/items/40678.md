## Summary

Keep open. Current main still lacks an opt-in TUI observe or mirror mode: the TUI command surface, runtime options, history loading, and live event handling remain scoped to one active session. Gateway session-message primitives now exist, but the TUI does not consume them, and the feature needs maintainer design because it would intentionally expand cross-channel transcript visibility.

## What This Changes

Request an opt-in TUI mode, flag, or config option to observe or mirror iMessage, Telegram, cron, and subagent activity routed to the same agent without reverting `/new` session isolation.

## Best Possible Solution

Add an explicit authorized TUI/Gateway observe path that keeps default `/new` isolation, labels channel/session/run origin, separates read-only observation from interactive sending, supports live aggregate events plus recent-history previews, and updates focused TUI, Gateway, and docs coverage.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: a TUI on `agent:main:main` will ignore a live chat or BTW event for another session key because `handleChatEvent` and `handleBtwEvent` return on session mismatch, and history loading reads only `state.currentSessionKey`. I did not run a live multi-channel setup because this review was read-only, but the source and tests prove the current behavior.

## Solution Assessment

Unclear. The requested direction is plausible, but the best implementation is not a bare flag that widens event filters; it should be an explicit observe/read authorization path with origin labels and a clear relationship to the broader unified sync work in #41548.
