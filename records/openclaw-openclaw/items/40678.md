## Summary

Keep this open. Current main still has isolated TUI sessions and active-session filtering, with no opt-in TUI observe or mirror mode; the requested cross-channel visibility remains a real feature gap that needs maintainer design because it widens transcript read access across channels and sessions.

## What This Changes

The issue requests an opt-in TUI observe or mirror mode that shows iMessage, Telegram, cron, and other channel activity in the TUI without reverting `/new` session isolation.

## Best Possible Solution

Add an explicit authorized TUI observe surface that keeps default `/new` isolation, separates read-only observation from interactive sending, labels channel/session/run origin, supports recent aggregate previews plus live updates, and documents the privacy model.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: the current TUI loads history only for `state.currentSessionKey` and ignores live chat/BTW events for non-current session keys, so other channel/session activity will not render in the TUI.

## Solution Assessment

Unclear. The requested UX is reasonable, but the safest implementation is not a bare flag that widens event filters; it should be a distinct observe/read authorization path coordinated with #41548 and #73402.
