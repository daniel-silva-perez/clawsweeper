## Summary

Keep open. Current main still lacks an opt-in TUI observe or mirror mode: the CLI/options surface has no flag or config state for it, history loading is scoped to one active session, live chat/BTW events are filtered to that session, and the TUI does not consume the existing Gateway session-message subscription primitives. The request is valid but needs maintainer design because it intentionally expands cross-channel transcript visibility.

## What This Changes

Request an opt-in TUI flag, config option, or observe mode to mirror iMessage, Telegram, cron, and subagent activity for the same agent without reverting isolated `/new` sessions.

## Best Possible Solution

Add an explicit authorized TUI/Gateway observe path that keeps default `/new` isolation, labels channel/session/run origin, separates read-only observation from interactive sending, supports live aggregate events plus recent-history previews, and updates focused TUI, Gateway, and docs coverage.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: a TUI on one session will ignore live chat or BTW events for another session key, and history loading reads only `state.currentSessionKey`. I did not run a live multi-channel setup because this review was read-only, but source and tests establish the current behavior.

## Solution Assessment

Unclear. The requested direction is plausible, but the best implementation is not a bare flag that widens event filters; it should be an explicit observe/read authorization path with origin labels and a clear relationship to #41548 and #73402.
