## Summary

Keep this issue open. Current main still lacks an explicit opt-in TUI observe or mirror mode: TUI exposes a single `--session`, runtime options have no observe state, `/new` still deliberately creates an isolated `tui-<uuid>` session, live chat/BTW events are filtered to the current session, and history loading requests one session key at a time. Related #41548 remains open and broader, but it does not clearly supersede this focused TUI monitoring workflow.

## Best Possible Solution

Keep this open as a core TUI/Gateway UX feature. Implement an explicit `openclaw tui --observe` or `tui.mirrorChannels` mode that preserves default `/new` isolation, aggregates recent same-agent activity through a deliberate Gateway/session event or history surface, labels channel and session origin, and defines read-only versus interactive behavior. Add focused tests for same-agent different-session events, channel-originated final/side-result events, aggregate history loading, and read-only behavior; coordinate with #41548 only if maintainers choose that broader issue as the canonical umbrella.
