## Summary

Keep this open. Current main still hardcodes the bundled `session-memory` writer to `<workspace>/memory/YYYY-MM-DD-slug.md`; the implemented startup-context cleanup for related memory naming work makes slugged files readable, but it does not add configurable output paths, filenames, or placeholder expansion.

## What This Changes

Add configurable output path and filename templates for the bundled `session-memory` hook so saved session files can follow workspace conventions instead of always using the flat `memory/YYYY-MM-DD-slug.md` layout.

## Best Possible Solution

Implement a narrow, backward-compatible `session-memory` template contract for session artifacts: keep the current default path exactly, add validated relative output-path and filename templates with documented date/time/slug placeholders, update bundled hook docs and public docs, and leave startup-context loading semantics unchanged.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: on current main the handler always builds `memoryDir` from `workspaceDir + "memory"`, always builds `filename` as `YYYY-MM-DD-slug.md`, and has tests expecting that fixed filename shape.

## Solution Assessment

Yes, for a narrowed implementation. Configurable relative templates are the maintainable fix for this request; adding a separate daily writer or empty-directory cleanup should be treated as separate scope unless maintainers explicitly want that bundled behavior.
