## Summary

Keep this issue open. The discussion already narrowed the useful scope correctly: current main has partial offline-like behavior, but only for narrow, volatile queues. Android Voice uses an in-memory FIFO that flushes to `chat.send` on reconnect, and iOS has an in-memory watch quick-reply queue, but Android/iOS chat and iOS Talk still require live gateway health/connectivity. There is no general, persistent mobile command outbox with captured timestamps, retry state, queue management, or cross-platform replay semantics.

## Best Possible Solution

Keep open and narrow it into a staged mobile outbox feature: define a shared queued-intent payload (`clientMessageId`, `capturedAt`, timezone, input type, text/audio reference, retry state), add encrypted persistent storage per platform, wire Android Mic/Chat and iOS Chat/Talk into it, reuse gateway `chat.send` idempotency, and add tests for offline capture, app restart, reconnect replay, and queue management UI.

## Risks / Open Questions

- Closing now would drop a valid product gap: the existing queues are narrow, volatile, and not crash/reboot persistent.
- A correct implementation needs mobile storage/encryption choices, captured-time semantics, idempotent replay, OS background-sync expectations, and user controls for review/delete/retry.
