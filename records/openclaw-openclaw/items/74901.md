## Summary

Keep open. The PR addresses a plausible TTS delivery bug, but the current patch broadens a local-file media trust boundary by allowing a result-controlled flag to bypass the existing external-provenance and exact-name checks.

## What This Changes

The PR changes `filterToolResultMediaUrls` so `details.media.trustedLocalMedia === true` preserves local media paths even when `builtinToolNames` does not include the emitting tool name.

## Best Possible Solution

Keep the TTS delivery fix, but bind the `trustedLocalMedia` bypass to an internally trusted exact `tts` result with no external provenance, and add positive TTS coverage plus negative unregistered, case-variant, and MCP/external-result coverage.

## Reproduction Assessment

Yes. The original bug can be reproduced at the code level by passing a TTS local OGG path with `builtinToolNames` defined but missing `tts`; the PR regression can be reproduced by an unregistered or MCP-provenance result that sets `details.media.trustedLocalMedia: true` for a local path.

## Solution Assessment

No. Respecting TTS-owned trusted local media is the right direction, but this implementation is too broad because the result field itself bypasses provenance and exact-name checks instead of being tied to the built-in TTS source.
