## Summary

Keep open. The PR targets a real TTS-to-Telegram delivery regression, but the patch broadens local-file media trust from the current registered/internal tool provenance model to a result-controlled `trustedLocalMedia` flag, which can bypass the protections current main applies to MCP and raw-name collision cases.

## What This Changes

This PR changes `filterToolResultMediaUrls` so `details.media.trustedLocalMedia === true` bypasses the exact `builtinToolNames` local-media gate.

## Best Possible Solution

Keep the intended TTS delivery fix, but constrain local-path bypass to an internally trusted exact `tts` source with non-external result provenance, and add positive coverage for `builtinToolNames` omitting `tts` plus negative coverage for MCP, unregistered, and case-variant tools.

## Reproduction Assessment

Yes. The original bug has concrete user steps in the linked TTS report, and the PR regression can be reproduced with a focused unit case where an MCP or unregistered result sets `details.media.trustedLocalMedia: true` for a local path.

## Solution Assessment

No. Respecting TTS-owned `trustedLocalMedia` is a reasonable direction, but this implementation is broader than the existing trust model because it lets the result object itself bypass external-result and exact-name checks.
