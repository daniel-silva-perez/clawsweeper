## Summary

Keep open. The PR targets a valid TTS voice-delivery failure, but the proposed fix broadens a delivery trust boundary by letting `details.media.trustedLocalMedia` bypass the existing external-result and exact registered-name checks for local filesystem media.

## What This Changes

The PR changes `filterToolResultMediaUrls` so `details.media.trustedLocalMedia === true` preserves local media paths even when `builtinToolNames` does not include the emitting tool name.

## Best Possible Solution

Preserve the intended TTS delivery fix by trusting `trustedLocalMedia` only for an internally trusted, exact built-in `tts` result with no external provenance, while keeping the raw-name gate for every other tool and adding positive and negative media-filter tests.

## Reproduction Assessment

Yes. The user bug can be reproduced at the code level when `filterToolResultMediaUrls` receives a TTS local OGG path with `builtinToolNames` defined but omitting `tts`; the PR regression can be reproduced with an unregistered or external tool result that sets `details.media.trustedLocalMedia: true` for a local path.

## Solution Assessment

No. Respecting TTS-owned `trustedLocalMedia` is the right direction, but this implementation is too broad because the result field bypasses provenance and exact-name checks instead of being tied to the built-in TTS source.
