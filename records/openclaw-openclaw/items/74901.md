## Summary

Keep open. The linked TTS/Telegram bug is actionable, but this PR is not merge-ready because the proposed bypass makes `details.media.trustedLocalMedia` a result-controlled trust source before the existing external-tool and exact-name gates run.

## What This Changes

This PR changes `filterToolResultMediaUrls` so `details.media.trustedLocalMedia === true` bypasses the exact `builtinToolNames` registration gate for local tool-result media.

## Best Possible Solution

Keep the intended TTS fix, but constrain it to an internally trusted source such as exact raw built-in `tts` and non-external result provenance, with positive coverage for `builtinToolNames` omitting `tts` and negative coverage for MCP, unregistered, and case-variant tools.

## Reproduction Assessment

Yes. A focused unit case can exercise the proposed patch with an MCP or unregistered tool result containing `details.media.mediaUrl: "/etc/passwd"` and `trustedLocalMedia: true`; current main filters that local path, while the PR would allow it. The original Telegram issue also has concrete user steps, but I did not run a live channel reproduction.

## Solution Assessment

No. Respecting `trustedLocalMedia` for built-in TTS is likely the right direction, but this implementation is broader than the trust model and should preserve `isExternalToolResult` plus exact raw-name collision protections.
