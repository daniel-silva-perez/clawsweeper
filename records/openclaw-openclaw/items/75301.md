## Summary

Keep open. Current main has only partial cleanup surfaces for bundled plugin runtime deps and orphan transcripts; it does not provide the requested cross-cache inventory/prune command, and the related open reports show the runtime-deps and session-cleanup lifecycle problems are still live.

## What This Changes

Add a CLI or doctor maintenance surface that inventories cache-class directories under `~/.openclaw` by size and safely prunes stale runtime-deps roots, orphan transcripts, and other regenerable caches without touching user-owned state.

## Best Possible Solution

Keep this issue open and design a core-owned cache inspection/prune surface with dry-run output, explicit cache-vs-data classes, a narrow deletion allowlist, current-version protection for runtime deps, and opt-in handling for expensive or user-adjacent caches such as memory and browser state.

## Reproduction Assessment

Not applicable as a feature request. Current-main verification is source/docs inspection: no `openclaw caches` command exists, `plugins deps --prune` only removes `openclaw-unknown-*` runtime-deps roots, and doctor only interactively archives orphan transcripts.

## Solution Assessment

Unclear until maintainers choose the top-level command versus doctor integration. The safest direction is the requested dry-run-first cache inventory with narrow class-based pruning; broad automatic deletion should wait for explicit guardrails.
