## Summary

Keep open. Current main still lacks the requested cross-cache inventory/prune surface; the existing cleanup paths are partial and narrowly scoped, and related runtime-deps/session cleanup issues remain open enough that deletion policy needs maintainer design before implementation.

## What This Changes

Request an `openclaw caches` or doctor-integrated command that reports `~/.openclaw` cache/data sizes and safely prunes stale runtime-deps roots, orphan transcripts, and regenerable browser/tools caches.

## Best Possible Solution

Keep this issue open and design a core-owned cache inventory/prune surface with dry-run output, explicit cache-vs-data classes, a narrow deletion allowlist, current-version and lock protection for runtime deps, freshness protection for transcripts, and opt-in handling for browser/tools/memory caches.

## Reproduction Assessment

Not applicable as a feature request. Current-main source/docs inspection is enough to verify the gap: there is no `openclaw caches` command, and existing `plugins deps`, `sessions cleanup`, and doctor cleanup paths only cover slices of the requested behavior.

## Solution Assessment

Unclear until maintainers choose the product surface. The requested dry-run-first inventory plus class-based pruning is directionally safer than automatic update-time deletion, but the command boundary and cache/data policy need maintainer approval.
