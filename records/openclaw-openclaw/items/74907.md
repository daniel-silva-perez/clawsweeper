## Summary

Keep open. The report is a current, well-supported 2026.4.x regression: the direct Anthropic Messages converter already groups consecutive internal tool results, but the affected Claude CLI/OAuth route uses a separate warm stdio live-session path with `--replay-user-messages`, and current main has no targeted fix or regression coverage for replay after parallel tool calls.

## What This Changes

The issue requests fixing Claude CLI/OAuth session replay so assistant turns with multiple tool calls do not produce orphan Anthropic `tool_use` blocks on later user messages.

## Best Possible Solution

Add a focused regression for the `claude-stdio` live-session replay path with an assistant turn containing 2+ tool calls followed by split stored tool results, then adjust that boundary so follow-up turns either keep all matching results adjacent/grouped for Claude replay or prune tool calls and results symmetrically. The direct Anthropic HTTP converter should remain unchanged unless the new test proves a shared helper is needed.

## Reproduction Assessment

Yes. The reporter gives a high-confidence fresh-session reproduction against `claude-cli`/OAuth, and the current source confirms that route is separate from the already-grouping direct Anthropic converter.

## Solution Assessment

No current solution is present on main. The reporter's original direct-converter suggestion is not the narrowest target anymore; the safer fix starts in the Claude CLI live stdio replay path and adds coverage there first.
