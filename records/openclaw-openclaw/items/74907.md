## Summary

Keep open. The reporter and follow-up comments narrowed this to a current 2026.4.x Claude CLI live-session replay regression, and current main still routes `claude-cli` OAuth runs through warm stdio with `--replay-user-messages` while writing only the current user prompt. The direct Anthropic Messages converter already groups consecutive `toolResult` records, so the remaining actionable surface is the separate CLI live-session replay path, not a closeable implemented-on-main case.

## What This Changes

The issue reports that 2026.4.x session replay for multi-tool Claude turns can submit assistant `tool_use` blocks without all matching `tool_result` blocks on a later user message.

## Best Possible Solution

Add a focused regression for the Claude CLI warm stdio replay path, then fix that boundary so a session with one assistant turn containing multiple parallel tool calls can accept a later user message without replaying orphaned Anthropic `tool_use` blocks. The direct Anthropic HTTP converter's existing grouping behavior should be preserved, and any broader Copilot/provider symptom should be split only if the regression proves a distinct path.

## Reproduction Assessment

Yes. The reporter gives a high-confidence fresh-session reproduction: start a new `claude-cli`/Anthropic OAuth session, trigger one multi-tool assistant turn, then send any follow-up user message and observe Anthropic rejecting orphan `tool_use` IDs.

## Solution Assessment

Unclear for the exact implementation until the Claude CLI replay regression is written. The reporter's initial direct-converter suggestion is not the narrowest fix because current main already groups direct Anthropic tool results; the safer path starts in `claude-stdio` live-session replay handling.
