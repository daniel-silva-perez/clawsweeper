## Summary

Keep open. The reporter’s follow-up comments make this a current, reproducible 2026.4.x Claude CLI live-session replay bug rather than only a compaction problem, and current main does not contain a targeted fix. The direct Anthropic Messages converter already coalesces consecutive internal tool results, but the affected `claude-cli` stdio path is separate and still launches a warm Claude process with replay-oriented flags while sending one user message per turn.

## What This Changes

The issue reports that OpenClaw 2026.4.x can replay Claude/Anthropic multi-tool turns so assistant `tool_use` blocks are submitted without immediately matching `tool_result` blocks on follow-up messages.

## Best Possible Solution

Add a focused regression around the Claude CLI stdio live-session replay path and fix that boundary so a session containing parallel tool calls can accept a later user message without submitting orphaned Anthropic `tool_use` blocks. The fix should preserve the direct Anthropic HTTP converter’s existing grouping behavior and avoid broad transcript rewrites unless the regression proves they are necessary.

## Reproduction Assessment

Yes. The reporter provides a high-confidence fresh-session reproduction: start a new `claude-cli`/Anthropic OAuth session, trigger a multi-tool assistant turn, then send a simple follow-up message and observe Anthropic rejecting orphan `tool_use` IDs.

## Solution Assessment

Unclear until the CLI stdio replay regression is written. The reporter’s original HTTP-converter suggestion is not the narrowest target anymore because current main already groups direct Anthropic tool results; the safer fix starts in `claude-stdio` live-session/replay handling.
