## Summary

Keep open. Current main still lacks timeout-driven run-reply session reset, so the PR is not obsolete, but the patch is not merge-ready because it can reset on mixed fallback failures, misses payload-only timeout failures, and is stale against current direct callers/tests.

## What This Changes

The PR branch adds a timeout reset callback to run-reply execution, wires it through `runReplyAgent`, returns timeout-specific retry copy, and adds run-reply tests for timeout reset paths.

## Best Possible Solution

Land a rebased implementation that rotates the run-reply session only after definitive all-model timeout exhaustion on a user turn, handles thrown, `meta.error`, and payload-only timeout surfaces, and preserves existing compaction, role-ordering, transient HTTP retry, billing, rate-limit, auth, heartbeat, and external-chat behavior.

## Reproduction Assessment

Yes. Current main has no timeout reset callback and the remaining patch bugs are reproducible with focused unit cases: mixed fallback attempts should not reset, while an embedded timeout returned only as an `isError` payload should reset.

## Solution Assessment

No, not as written. The direction is maintainable, but the safer fix should use `FallbackSummaryError.attempts` for all-attempt timeout proof, classify timeout payloads before accepting a run, and update current direct callers/tests after rebase.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Require all fallback attempts to time out:** `src/auto-reply/reply/agent-runner-execution.ts:588-595`
  - body: `isTimeoutError(err)` is applied to the aggregate fallback error. A mixed fallback chain can include one timed-out attempt plus auth, billing, rate-limit, or another failure, so this can rotate and discard the transcript even though all models did not time out. Use the structured fallback attempts and require every relevant attempt to carry timeout evidence before resetting.
  - confidence: 0.91
- **[P2] Handle timeout payloads before accepting the run:** `src/auto-reply/reply/agent-runner-execution.ts:475-480`
  - body: The embedded runner can return a timeout as an `isError` payload with no `meta.error`. This branch only checks `runResult.meta?.error`, then the loop breaks as a successful run, so that timeout surface keeps the same bloated session and leaves the retry loop unresolved.
  - confidence: 0.88

  - body: Adding `resetSessionAfterTimeout` as a required parameter means current direct `runAgentTurnWithFallback` test callers no longer satisfy the parameter object after rebase. Update the shared helper and explicit call sites with a stub callback or the test typecheck lane will fail.
  - confidence: 0.84
