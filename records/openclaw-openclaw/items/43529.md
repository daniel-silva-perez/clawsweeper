## Summary

Keep open. Current main still does not implement timeout-driven session rotation, so the PR is not obsolete, but the latest patch is not merge-ready because its timeout classification can reset sessions for mixed fallback failures and it misses the embedded timeout error-payload surface discussed in review.

## What This Changes

The PR adds a timeout reset callback through `runReplyAgent` and `runAgentTurnWithFallback`, returns timeout-specific retry text, and adds focused tests for timeout reset, reset-unavailable fallback text, heartbeat, and transient HTTP behavior.

## Best Possible Solution

Implement timeout session rotation on current main using structured all-model timeout evidence, handle both `meta.error` and timeout `isError` payloads before the run loop exits, and preserve existing heartbeat, transient HTTP retry, billing, rate-limit, auth, compaction, and role-ordering behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Gate resets on all-attempt timeout evidence:** `src/auto-reply/reply/agent-runner-execution.ts:591-595`
  - body: This applies timeout detection to the aggregate fallback error, whose message can contain one timed-out attempt plus auth, billing, rate-limit, or other failures. That can rotate and discard the active transcript for a non-timeout exhaustion case; require structured evidence that every relevant fallback attempt timed out before calling the timeout reset.
  - confidence: 0.91
- **[P2] Classify timeout error payloads before breaking:** `src/auto-reply/reply/agent-runner-execution.ts:479-483`
  - body: The embedded runner can return a timeout as an `isError` payload with no `meta.error`. This branch only checks `runResult.meta?.error`, then exits the loop as a successful run, so that timeout surface keeps the same bloated session and leaves the reported retry loop unresolved.
  - confidence: 0.88
