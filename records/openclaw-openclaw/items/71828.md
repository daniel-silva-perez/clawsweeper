## Summary

Keep open. Current main still has the reported liveness gap: the active keepalive tick sends typing but does not renew the run-side typing TTL, so this narrow PR remains relevant and is not superseded by the broader open active-run presence work.

## What This Changes

The PR refreshes the shared typing controller TTL on every active keepalive tick and adds a regression test for long runs that continue beyond the original TTL window.

## Best Possible Solution

Land this narrow TTL-renewal fix with its regression test after maintainer review and exact-head CI, or deliberately fold the same behavior and test into the broader #65919 implementation while keeping that umbrella open for non-Discord/non-typing presence leasing.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to create a typing controller with a 6s keepalive interval and 10s TTL, start the loop, advance fake timers to the first keepalive tick, then advance past 10s total; current main cleans up because the keepalive tick does not refresh the TTL.

## Solution Assessment

Yes. The proposed change is the narrowest maintainable fix for this specific bug: refresh the run-side TTL only after the existing guarded keepalive tick successfully sends typing, preserving run-complete blocking, dispatch-idle cleanup, and existing channel callback behavior.
