## Summary

Keep this open because an open linked PR uses closing syntax for the report, and the source review found a plausible remaining mismatch between the accepted spawn result and the model ultimately used by the child run. Current main covers the happy path for explicit model persistence, but a stored override can still be cleared later by allowlist validation while the spawn result has already reported modelApplied.

## What This Changes

The issue asks for sessions_spawn(model=...) to make the spawned sub-agent actually run on the requested model, and not report modelApplied true when the request falls back to the orchestrator/default model.

## Best Possible Solution

Keep the issue paired with #74892 and verify the fix against the exact contract: an allowed sessions_spawn.model should run the child on that provider/model, while an invalid or disallowed model should fail visibly or return modelApplied false with a warning before the child run can silently use the default. Add regression coverage for the allowed path and the disallowed-model fallback path.

## Reproduction Assessment

Yes. The reporter supplied concrete spawn/trajectory evidence, and source tracing gives a high-confidence focused reproduction path: request a model that resolves during spawn but is outside the later agent allowlist, then inspect the child trajectory and tool result.

## Solution Assessment

Unclear. Narrowing modelApplied may be necessary, but the best fix should align spawn-time validation with agent-command validation so the tool result cannot claim success before the child run clears the override to the default model.
