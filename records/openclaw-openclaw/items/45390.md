## Summary

Keep this open. Current main still has daily/idle session freshness, heartbeat-aware lifecycle timestamps, isolated heartbeat sessions, and reset rollover cleanup, but it does not provide the requested configurable normal-session maximum age or maximum turn rotation policy. The discussion correctly calls out a data-loss-sensitive persistence concern, and the related open memory-rollover PRs should be consolidated before choosing the public contract.

## What This Changes

The issue requests configurable maximum session age and maximum session turns that automatically rotate long-running agent sessions after saving memory and logging the rollover.

## Best Possible Solution

Define a single generic session reset contract that supports explicit maximum age and maximum conversation turns, persists the needed accounting, excludes or deliberately handles maintenance traffic, emits clear rollover reasons/logs, and coordinates pre-rotation memory persistence through the existing hook/lifecycle system.

## Reproduction Assessment

Not applicable as a feature request. We do have high-confidence static verification that the requested knobs are absent by inspecting the current config types, strict schema, runtime freshness gate, docs, and targeted source search.

## Solution Assessment

Unclear until maintainer product/API review. The maintainable direction is a unified session reset policy, but the public config shape, turn-count semantics, and automatic memory behavior should be consolidated with #42867, #42677, and #50933 before implementation.
