## Summary

Keep this open. Current main has daily/idle reset freshness that no longer relies on generic metadata writes, plus an opt-in isolated heartbeat workaround, but normal sessions still lack the requested configurable max-age duration and max-turn rotation policy; bundled session-memory also remains explicit `/new`/`/reset` scoped rather than a shipped pre-rotation persistence path for automatic rollover.

## What This Changes

Request a configurable automatic normal-session rotation policy with max age and max turn limits that saves memory, starts a fresh session, and logs the rollover.

## Best Possible Solution

Define one generic normal-session reset contract under the existing session reset surface that supports max-age and max-turn limits, persists any needed counters, emits typed rollover reasons, deliberately excludes maintenance traffic, and coordinates pre-rotation memory persistence through the hook/lifecycle system.

## Reproduction Assessment

Not applicable as a feature request. Static verification is high-confidence: schema, types, runtime freshness checks, tests/docs, and bundled hook metadata show daily/idle/manual reset behavior but no configurable normal-session max-age or max-turn policy.

## Solution Assessment

Unclear until maintainer/API review. The requested direction is valid, but the maintainable fix should extend the existing session.reset lifecycle contract and hook semantics rather than adding standalone keys that bypass current resetByType/resetByChannel behavior.
