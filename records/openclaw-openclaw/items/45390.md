## Summary

Keep this open. Current main has daily and idle reset freshness that no longer lets heartbeat/system-event metadata writes keep sessions alive, but it still lacks the requested configurable normal-session maximum age and maximum turn-count rotation policy; the pre-rotation memory persistence piece also remains tied to open rollover-hook PRs rather than shipped behavior.

## What This Changes

Request a configurable session maximum age and maximum turn-count policy that automatically persists memory, starts a fresh session, and logs rotation.

## Best Possible Solution

Define one generic session reset contract for normal sessions that extends the existing session.reset model with max-age and max-turn limits, persists any needed counters, emits typed rollover reasons, excludes maintenance traffic deliberately, and coordinates pre-rotation memory persistence through the hook/lifecycle system already used for session boundaries.

## Reproduction Assessment

Not applicable as a feature request. Static verification is high-confidence: current schema, types, runtime freshness checks, tests, and docs show daily/idle reset behavior but no configurable max-age or max-turn normal-session policy.

## Solution Assessment

Unclear until maintainer/API review. The requested direction is valid, but the maintainable path is a unified reset policy and rollover-memory contract rather than adding isolated keys that bypass the existing session.reset surface.
