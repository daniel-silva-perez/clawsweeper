## Summary

Keep open. Current main still has the reported recovery gap: the runner auto-resets for role-ordering and Gemini-style function-call corruption, but the Anthropic empty-message schema failure still falls through to the generic external failure path, and adjacent empty-payload work does not archive/drop the poisoned assistant-only session history.

## What This Changes

Add recovery for assistant-only or no-user transcript corruption that causes repeated Anthropic `messages: at least one message is required` failures, plus clearer reset acknowledgement after `/new` or auto-reset.

## Best Possible Solution

Add a focused transcript-corruption recovery path for Anthropic-compatible runs that detects zero replayable user messages or the provider's empty-messages schema error, preserves the bad transcript for diagnosis, rotates/drops the active poisoned session, and returns the existing friendly corruption-reset copy. Keep any `/new` ready ping aligned with the existing reset command and hook conventions instead of introducing a separate channel-specific behavior.

## Reproduction Assessment

Yes. The issue provides an exact transcript shape, and current source inspection gives a high-confidence source-level reproduction path: assistant-only sentinel error entries are preserved, failed assistant turns are removed before Anthropic conversion, and the resulting empty-message schema error is not classified for reset. I did not run tests because this review was explicitly read-only.

## Solution Assessment

Yes for a narrow auto-reset or read-time repair guard that treats assistant-only/no-user replay history as terminally corrupt. A generic Anthropic transport fallback alone is not the best solution because it can hide the 400 while leaving the active session poisoned and the reset UX unclear.
