## Summary

Keep open. Current main still has the reported recovery gap: assistant-only error transcripts can remain unrepaired, Anthropic replay can still reduce them to an empty provider message list, and the auto-reset detector does not match the Anthropic `messages: at least one message is required` schema error; the reset acknowledgement UX also remains ack-only.

## What This Changes

The issue requests automatic recovery for leading-assistant or no-user session transcripts that trigger Anthropic empty-message 400s, plus a clearer post-reset ready signal after `/new` or auto-reset.

## Best Possible Solution

Add a focused corruption-recovery path that treats assistant-only or zero-replayable-user transcripts as terminally corrupt for Anthropic-compatible runs, archives or rotates the bad transcript, drops the poisoned active session, avoids replaying assistant-only history into the replacement session, and returns the existing friendly corruption-reset text. The ready-signal improvement should clearly distinguish system/session-control acknowledgements from real model output and stay aligned with the WebChat follow-up in #75314.

## Reproduction Assessment

Yes. A source-level reproduction is clear: seed a JSONL session with only assistant `stopReason: "error"` sentinel turns and no user turn, then send through Anthropic-compatible replay; current repair treats the sentinel as already repaired, transport drops failed assistant turns, and the outgoing Anthropic `messages` can be empty.

## Solution Assessment

Yes, a narrow read-time or failure-time corruption reset is the maintainable fix for this issue. A generic Anthropic transport fallback alone is not sufficient because it would avoid one 400 while preserving the corrupt transcript and the ambiguous reset UX.
