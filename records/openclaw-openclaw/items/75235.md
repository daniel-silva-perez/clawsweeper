## Summary

Keep open. Current main still has the reported failure class: assistant-only error turns are considered repaired, then filtered before Anthropic-compatible transport, and the auto-reset path does not recognize the Anthropic empty-messages schema error, so the user can still fall through to the generic external failure instead of a session reset.

## What This Changes

Requests auto-recovery for assistant-only/no-user session transcripts that cause Anthropic empty-message 400s, plus clearer confirmation after manual or automatic resets.

## Best Possible Solution

Add a focused corruption-recovery path for Anthropic-compatible empty-message failures caused by zero replayable user turns or assistant-only error transcripts. The end state should archive the bad transcript, rotate/drop the active session, avoid replaying the poisoned history, return the existing friendly corruption-reset message, and keep any reset-ready ping aligned with the existing reset command flow.

## Reproduction Assessment

Yes. A source-level reproduction is clear: create a session JSONL with only assistant `stopReason: "error"` sentinel entries and no user entries, then follow the current replay path through session repair, Anthropic validation, failed-assistant filtering, and Anthropic payload construction.

## Solution Assessment

Yes for a narrow auto-reset or read-time corruption guard. A transport-only fallback that injects a minimal user message would be incomplete because it can hide the provider 400 while leaving the poisoned session and unclear reset UX in place.
