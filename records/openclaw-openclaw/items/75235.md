## Summary

Keep open. The supplied transcript/logs describe a real current-main failure class: assistant-only error turns can be treated as already repaired, then filtered out before Anthropic transport so the provider receives no replayable user message, while the auto-reset detector only covers Gemini function-call ordering and generic role-ordering errors. Related empty-payload work in #73926 and #74731 is useful context but does not supersede the requested leading-assistant/no-user transcript recovery or the reset acknowledgement UX gap.

## What This Changes

The issue requests automatic recovery for Anthropic sessions whose persisted transcript contains only leading assistant error turns or no user-role messages, plus clearer confirmation after manual or automatic resets.

## Best Possible Solution

Add a focused corruption-recovery path that treats Anthropic-compatible zero-user or assistant-only failed transcripts as terminal session corruption, archives the bad transcript using the existing reset snapshot convention, removes the active session entry, and returns the existing friendly corruption-reset message. Any reset-ready acknowledgement should be implemented through the existing reset command or hook flow so Telegram and other auto-reply channels stay consistent.

## Reproduction Assessment

Yes. The issue provides a concrete bad transcript shape, and current main has a clear source-level reproduction path: sentinel assistant error turns are left on disk, dropped before Anthropic transport, and the resulting empty `messages` payload is not caught by the reset detector.

## Solution Assessment

Yes for the narrow auto-reset/read-time repair direction. A transport-only fallback that injects a minimal user message would prevent the provider 400 but would leave the poisoned session and unclear reset UX in place.
