## Summary

Keep open. The report is source-backed: cron jobs can carry a calling agent id on add, but the current cron list/update/remove/run/runs paths do not pass or enforce a caller agent scope, so an owner-authorized agent cron tool can enumerate or operate on jobs outside its agent. The sessions_list surface appears to have separate visibility guards, so the clear remaining work is cron-specific server-side scoping plus review of global admin semantics.

## What This Changes

Request server-side per-agent isolation for cron tool responses and cron job mutations, with sessions_list checked for similar cross-agent leakage.

## Best Possible Solution

Add a hard caller-agent scope to the cron tool and gateway cron requests, filter job lists and run logs by that scope, and reject update/remove/run/runs against jobs outside it while preserving an explicit admin/operator path for global cron management.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to create cron jobs for two agents, then invoke the cron tool list from one owner-authorized agent; current source calls cron.list without agentId and listPage returns the shared job store without an agent filter.

## Solution Assessment

Yes, server-side filtering is the right class of fix. The safest implementation is to enforce scope in the gateway/service contract rather than relying on prompt instructions or client-side filtering, with tests for cross-agent list, update, remove, run, and runs history denial.
