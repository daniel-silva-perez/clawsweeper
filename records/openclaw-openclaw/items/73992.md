## Summary

Keep open. This is a fresh, concrete report against OpenClaw 2026.4.25 showing `openclaw nodes list` timeouts plus a follow-up `gateway probe` timeout window. Current main still has the older non-network `node.list` implementation, and it has adjacent handshake/probe improvements, but I did not find high-confidence proof that the reported post-2026.4.25 intermittent loopback RPC wedge is fixed or superseded.

## Best Possible Solution

Keep this issue open as the current concrete `openclaw nodes list` loopback timeout report. The best maintainer path is to reproduce or instrument repeated `nodes list` on current main/v2026.4.27 with gateway logs, `/readyz` event-loop diagnostics, RPC timing around `node.pair.list` and `node.list`, and an immediate `gateway probe --json` after a failure. If the gateway accepts the WebSocket but read RPCs time out, route the fix toward gateway responsiveness/session-lock/read-diagnostic isolation; if only `node.list` stalls, add a focused node-list timing guard or cache around pairing-file reads.
