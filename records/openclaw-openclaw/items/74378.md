Keep open. The report is current, has a concrete Windows/npm reproduction, current main still has no merged short-lived CLI process-lifetime fix, and PR #74425 remains the linked implementation candidate rather than a reason to close this issue before merge.

## What This Changes

The issue reports that supported short-lived Windows npm CLI commands such as `status` and `--version` leave `node.exe ... openclaw.mjs` processes alive after command completion.

## Best Possible Solution

Review PR #74425 against the current CLI lifecycle and either land a focused fix or replace it with targeted teardown for the resource that keeps short-lived Windows npm CLI commands alive, while preserving long-running gateway/provider behavior and adding Windows smoke coverage for `status` and `--version`.

Yes. The reporter provides a high-confidence Windows npm reproduction for `status` and `--version`, including process observations that separate the stuck CLI process from the healthy gateway; I did not live-run Windows in this read-only Linux review.

## Solution Assessment

Unclear. Keeping this issue open is the correct tracking state while PR #74425 is open, but the best implementation still needs maintainer review because a forced process exit may be less maintainable than closing the specific dispatcher/proxy/resource handle that keeps Node alive.
