## Summary

Keep open. Current main still has a real Codex app-server steering durability gap: the caller can treat a message as steered before the async `turn/steer` delivery result is known, and a later rejection or terminal turn has no followup fallback. The PR is a narrow fix direction, but it is not implemented on main and the branch needs maintainer review/rebase against current main before merge.

## What This Changes

The PR makes Codex app-server queued steering return a delivery-success boolean, adds a durable embedded-message helper, awaits that helper in `runReplyAgent` so failed steering can fall through to followup queueing, updates related queue-handle types, and adds regression tests.

## Best Possible Solution

Land a rebased narrow fix that preserves active-run steering while guaranteeing each queued user message is either accepted by live Codex `turn/steer` or queued through the existing followup path when steering is terminal or rejected, with the current-main completion-flush test reconciled to that contract.

## Reproduction Assessment

Yes. The high-confidence source path is a default `steer` message queued during an active Codex turn: current main returns from `runReplyAgent()` after `queueEmbeddedPiMessage()` reports true, but the actual debounced `turn/steer` can fail after completion with no result returned to schedule a followup.

## Solution Assessment

Yes, the proposed direction is the narrow maintainable fix: make steering delivery observable and fall back through the existing followup queue. The branch still needs rebase and validation before it is merge-ready.
