## Summary

Keep open. Current main still does not provide the durable delivery guarantee described by the PR: it can accept queued Codex steering synchronously, return from the reply runner, then only later attempt or log a failed `turn/steer` without scheduling the documented followup fallback. The PR direction addresses that gap, but the branch is dirty against current main and needs maintainer rebase/reconciliation plus changed-gate proof before merge.

## What This Changes

The PR makes Codex app-server queued steering report delivery success or failure, adds an async durable embedded-message helper used by `runReplyAgent`, adds fallback regression coverage, and adjusts one changed-test routing expectation.

## Best Possible Solution

Land a rebased narrow fix that keeps the active-run steering model but guarantees each queued user message is either accepted by live Codex `turn/steer` or falls through to the existing followup queue when steering is terminal, unavailable, or rejected.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to queue default Codex steering during an active app-server turn with a debounce window, complete the turn before the timer flushes, and observe that current main has already returned success to `runReplyAgent()` while any later terminal `turn/steer` failure is only logged.

## Solution Assessment

Yes, the PR's delivery-result approach is the right narrow shape because it makes the existing caller wait for an accepted or rejected steering result and then reuse the existing followup path. It still needs rebase against current main's partial flush patch and validation before it is mergeable.
