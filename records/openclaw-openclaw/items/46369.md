## Summary

Keep open. Current main still removes only the cron job definition, while the cron-owned session-store entry and run-log artifact remain outside the delete path; the linked fix PR #46506 is open and unmerged, so this issue should stay open until that PR or an equivalent fix lands.

## What This Changes

Deleting a cron job via `cron.remove` should also clean up its `sessions.json` cron session records and `cron/runs/<jobId>.jsonl` run history so deleted jobs stop appearing in the Control UI session selector.

## Best Possible Solution

Land a narrow cron-owned cleanup after a successful job deletion: remove the base cron session entry and matching run entries for the deleted job, unlink the safe run-log path, tolerate missing artifacts, preserve unrelated sessions, and cover the behavior with focused regression tests.

## Reproduction Assessment

Yes. Seed or create an isolated cron job with a base `agent:<id>:cron:<jobId>` session entry and a run log, call `cron.remove`, then list sessions; current main removes the job but leaves the visible base session and does not unlink the run log.

## Solution Assessment

Yes. The narrow maintainable direction is cleanup in cron service ownership after successful deletion, using the existing session-store and run-log helpers; PR #46506 appears aligned but still needs maintainer review, rebase, landing, or replacement.
