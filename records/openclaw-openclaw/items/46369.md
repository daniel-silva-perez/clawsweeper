## Summary

Keep open. Current main still removes only the cron job definition, while cron-owned session-store entries and run-log artifacts remain outside the delete path; the issue is also paired with open, unmerged PR #46506, so it should stay open until that PR or an equivalent fix lands.

## What This Changes

Deleting a cron job via `cron.remove` should also remove the job-owned session records and run-log artifact so the Control UI does not list ghost cron sessions.

## Best Possible Solution

Land a narrow cron-owned cleanup after successful job deletion: remove the base cron session entry and matching run entries for the deleted job, unlink the safe run-log path, tolerate missing artifacts, preserve unrelated sessions, and cover the behavior with focused regression tests.

## Reproduction Assessment

Yes. Seed or create an isolated cron job with a base `agent:<id>:cron:<jobId>` session entry and a run log, call `cron.remove`, then list sessions; current main removes the job but leaves the base session visible and never unlinks the run log.

## Solution Assessment

Yes. A narrow cleanup in the cron service after a successful delete is the best maintainable direction because cron owns the job id, session-key convention, and run-log path; PR #46506 appears aligned but still needs maintainer review, rebase, landing, or replacement before this issue can close.
