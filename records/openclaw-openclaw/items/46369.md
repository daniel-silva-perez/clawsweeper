## Summary

Keep open. Current main still removes only the cron job definition, while cron-owned session records and run logs are handled elsewhere by retention/pruning paths rather than immediate job deletion. The issue is also paired with open, unmerged PR #46506, so it should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue requests that deleting a cron job through `cron.remove` also remove the job-owned session-store records and `cron/runs/<jobId>.jsonl` run history.

## Best Possible Solution

Keep this issue open until the cron service performs job-owned artifact cleanup after a successful delete: remove the base cron session entry and matching run entries from the relevant agent session store, unlink the safe run-log path, tolerate missing artifacts, preserve unrelated sessions, and cover the behavior with focused regression tests.

## Reproduction Assessment

Yes. The issue has concrete reproduction steps, and current-main static inspection gives a high-confidence reproduction path: create an isolated cron job that persists a base cron session entry/run log, call `cron.remove`, and the current remove implementation never touches those artifacts.

## Solution Assessment

Yes. A narrow cron-owned cleanup after successful removal is the best maintainable direction; PR #46506 is aligned with that direction but needs maintainer review or replacement before this issue can close.
