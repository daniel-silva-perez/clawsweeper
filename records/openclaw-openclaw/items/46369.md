## Summary

Keep open. Current main still routes `cron.remove` through a removal path that only filters the job definition, persists the cron store, rearms the timer, and emits a removal event; it does not update the agent session store or unlink the per-job run log. The issue is also already paired with open, unmerged PR #46506, so it should remain open until that PR or an equivalent fix lands.

## What This Changes

Deleting a cron job should also remove matching cron session-store records and per-job run-log files so removed jobs stop appearing as ghost sessions in the Control UI.

## Best Possible Solution

Keep this issue open until the cron service removes job-owned artifacts after a successful delete: delete the base cron session entry and matching run entries for the relevant agent store, unlink the safe `runs/<jobId>.jsonl` path, tolerate missing artifacts, preserve unrelated sessions, and cover the behavior with focused regression tests.

## Reproduction Assessment

Yes. The issue has concrete reproduction steps, and the current code path gives a high-confidence static reproduction: create an isolated cron job that writes a session entry/run log, call `cron.remove`, and the current remove implementation never touches those artifacts.

## Solution Assessment

Yes. A narrow cron-owned cleanup after successful removal is the best maintainable fix; the open PR direction is aligned, but it still needs maintainer review before this issue can close.
