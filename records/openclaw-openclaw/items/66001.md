## Summary

Keep this PR open. Current main still has the deliberate no-op `reconcileOrphans()` contract and does not reconcile before process-tool requests, so the PR’s intended behavior is not implemented on main. The PR is also paired with the still-open #65983 and has an unresolved ownership bug in review that should be fixed before merge.

## Best Possible Solution

Keep the PR open and request changes: make orphan reconciliation ownership-aware across both process session slug and `sessionKey`, narrow the reconciliation cancellation reason type, add regression tests for sessionKey-owned runs, then review whether the partial in-process fix should land while #65983 continues tracking durable restart/cold-start orphan reaping.

## Risks / Open Questions

- As proposed, reconciliation can cancel legitimate background exec runs whose supervisor `sessionId` is an agent `sessionKey` rather than the process registry’s session slug.
- The PR explicitly addresses only the in-process/session-loss half of #65983, not durable cold-start orphan reaping; maintainers should decide whether that partial scope is acceptable or should be split.
- The `reason` type widening noted in review could let future callers label forced orphan cancellation as a natural exit unless narrowed.
