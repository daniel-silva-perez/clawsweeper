## Summary

Keep this issue open. Current main has shipped generic Feishu presentation/button support for outbound payloads, but the specific command-options UX requested here is still not wired up: `/reasoning` with no argument still produces a plain text `Options: on, off, stream.` reply, and the Feishu reply dispatcher sends command replies as text/markdown-card text rather than semantic button presentations. The related Feishu card callback issue/PR in the provided context concerns extracting callback fields, not rendering command option menus.

## Best Possible Solution

Keep this issue open and implement it by having directive/command option menus produce a semantic `presentation` or explicit interactive button payload for simple choice sets, then route Feishu command replies through the existing Feishu presentation/card delivery path. Add focused coverage for `/reasoning` in Feishu so clicking `on`, `off`, or `stream` dispatches the corresponding command while retaining the current plain-text fallback elsewhere.

## Risks / Open Questions

- Avoid parsing arbitrary `Options:` text into buttons globally; complex option lines such as exec defaults are intentionally left as plain text today.
- Reuse the existing semantic `presentation` and Feishu card-action envelope paths so Feishu buttons submit authorized command text consistently with current card interactions.
- Preserve text fallback for Feishu document-comment replies and channels/surfaces that cannot render interactive buttons.
