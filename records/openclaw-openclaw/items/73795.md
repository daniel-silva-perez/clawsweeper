## Summary

Keep open. Current main still does not support or honor `streaming.preview.toolProgress: false` for WhatsApp or LINE, and the related Slack PR is a narrower channel-specific fix rather than a resolution for the cross-channel config/runtime gap.

## What This Changes

The issue requests preserving and honoring `streaming.preview.toolProgress: false` on channels without an existing `streaming.mode`, so tool/progress details are not delivered to end users.

## Best Possible Solution

Add a documented, schema-backed suppression contract for non-preview-streaming channels, preserve the config in channel metadata, and wire affected channel dispatch paths to suppress generic tool/progress delivery when the effective config disables tool progress. Coordinate Slack separately so the open Slack PR does not drift from the shared semantics.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: WhatsApp and LINE strict schemas do not accept the reported nested key, and their runtime dispatch paths do not pass the suppression flag consumed by the shared progress delivery code.

## Solution Assessment

No current fix is present on main. Preserving `streaming.preview` alone would be incomplete; the maintainable fix needs both config/schema support and runtime suppression wiring for the affected channel delivery paths.
