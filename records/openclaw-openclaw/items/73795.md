## Summary

Keep open. Current main still lacks a supported `streaming.preview.toolProgress: false` path for WhatsApp and LINE: the strict config surfaces do not expose the nested streaming key, and the runtime dispatch paths for those channels do not pass the suppression flag that the shared reply dispatcher requires. The related open Slack PR is narrower and does not resolve the cross-channel config/runtime gap described here.

## What This Changes

The issue requests preserving and honoring `streaming.preview.toolProgress: false` for channels that do not already declare `streaming.mode`, especially WhatsApp, LINE, Sendblue, and Slack-like channels.

## Best Possible Solution

Add a documented, schema-backed suppression contract for channels without preview streaming, preserve the config through normalization/metadata, and wire affected channel runtimes such as WhatsApp and LINE to suppress generic tool/progress delivery when the effective setting disables it. Include regression coverage for config preservation and outbound progress suppression, then reconcile Slack docs/defaults with the same contract.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: WhatsApp and LINE schemas/metadata do not accept the nested `streaming.preview.toolProgress` key, and their dispatch paths do not pass the suppression flag consumed by the shared reply dispatcher. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No current fix is present on main. Preserving the config key alone would not be sufficient; the runtime also needs to map the effective setting to `suppressDefaultToolProgressMessages` or an equivalent channel delivery policy.
