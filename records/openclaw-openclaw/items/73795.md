## Summary

Keep open. Current main still lacks a supported cross-channel path for `streaming.preview.toolProgress: false` on WhatsApp and LINE: their strict config surfaces do not expose the nested streaming key, and their runtime dispatch paths do not pass the shared suppression option. The related Slack and slash-command items are adjacent, not canonical fixes for this config/runtime gap.

## What This Changes

The issue requests that `streaming.preview.toolProgress: false` be preserved and respected for channels that do not already declare `streaming.mode`, especially WhatsApp, LINE, Sendblue, and Slack-like delivery paths.

## Best Possible Solution

Define a documented, schema-backed delivery contract for suppressing tool/progress messages independently of preview streaming, expose it through the affected channel config surfaces and metadata, and wire WhatsApp, LINE, and any Slack-compatible path to the shared dispatcher suppression option with regression tests and docs.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: WhatsApp and LINE strict schemas do not accept the nested `streaming.preview.toolProgress` key, and their reply paths do not pass the shared suppression flag that actually hides generic progress messages. I did not execute live reload/tests because this review was read-only.

## Solution Assessment

No current fix is present on main. Preserving the config key alone would be incomplete; the maintainable fix also has to map the effective setting into channel runtime delivery policy, probably via the existing `suppressDefaultToolProgressMessages` seam or a documented successor.
