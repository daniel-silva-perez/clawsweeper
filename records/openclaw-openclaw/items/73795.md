## Summary

Keep open. Current main still lacks a supported `streaming.preview.toolProgress: false` preservation and suppression path for WhatsApp and LINE, and the related open Slack PR is narrower than this cross-channel config/runtime gap.

## What This Changes

The issue requests that `streaming.preview.toolProgress` be preserved and honored for channels that do not already declare a preview-streaming `mode`, so end users do not receive internal tool/progress narration.

## Best Possible Solution

Add a documented, schema-backed suppression contract for non-preview-streaming channels, preserve it in WhatsApp and LINE config metadata, and wire the effective value into the shared auto-reply suppression option while reconciling the narrower Slack PR separately.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: WhatsApp and LINE strict schemas do not accept the nested key, and their dispatch paths do not pass the suppression flag consumed by shared auto-reply delivery.

## Solution Assessment

No current fix is present on main. Preserving the config key alone would not fully solve the leak; the runtime must also map the effective setting to `suppressDefaultToolProgressMessages` or an equivalent delivery policy.
