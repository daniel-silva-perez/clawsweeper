## Summary

Keep open. Current main still has the same source-level gap for the affected non-preview channels: WhatsApp and LINE do not expose `streaming.preview.toolProgress` in their supported config surfaces, and their reply dispatch paths do not pass the shared suppression option that prevents default tool/progress messages from being delivered. The related slash-command and Slack reports are adjacent but do not cover this cross-channel config/runtime contract.

## What This Changes

The issue asks for `streaming.preview.toolProgress: false` to survive config normalization and suppress tool/progress messages across affected non-Telegram channels even when `streaming.mode` is absent.

## Best Possible Solution

Define a documented, schema-backed delivery contract for suppressing tool/progress messages independently of preview streaming, expose it in affected channel config metadata, and wire WhatsApp, LINE, and any Slack-compatible path to the shared dispatcher suppression option with focused regression tests and docs.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction remains: WhatsApp and LINE schemas do not include `streaming.preview.toolProgress`, and their dispatch paths never pass `suppressDefaultToolProgressMessages`; I did not execute live reload or channel tests because this review was read-only.

## Solution Assessment

No current fix is present on main. Preserving the config key alone would be incomplete unless the effective setting is also mapped into runtime delivery policy, preferably through the existing suppression seam or a documented successor.
