## Summary

Keep open. Current main still does not provide a supported `streaming.preview.toolProgress: false` suppression path for WhatsApp or LINE, and the related open Slack PR is narrower than the cross-channel config/runtime gap described here.

## What This Changes

The issue requests that `streaming.preview.toolProgress: false` be preserved and honored on channels without `streaming.mode` so tool/progress narration does not reach end users.

## Best Possible Solution

Add a documented, schema-backed suppression contract for non-preview-streaming channels, then wire WhatsApp and LINE runtime dispatch to suppress generic tool/progress delivery when the effective config disables tool progress. Keep Slack reconciliation explicit, regenerate channel metadata/docs, and add regression coverage for config preservation plus outbound progress suppression.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: `channels.whatsapp.streaming.preview.toolProgress` and the corresponding LINE key are not supported by the strict channel schemas, and the WhatsApp/LINE dispatch paths do not pass the suppression flag consumed by the shared progress delivery code.

## Solution Assessment

No current fix is present on main. Preserving the config key alone would not be sufficient; the runtime also needs to map the effective setting to `suppressDefaultToolProgressMessages` or an equivalent channel delivery policy.
