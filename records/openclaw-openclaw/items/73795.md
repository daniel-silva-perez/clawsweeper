## Summary

Keep open. Current main still has the same cross-channel gap: the shared `streaming.preview.toolProgress` key exists and is honored by preview-streaming channels, but WhatsApp and LINE still have strict schemas without a nested `streaming.preview` path and their inbound reply paths do not pass a suppression flag into generic tool/progress delivery. The related open Slack PR is narrower and does not cover the WhatsApp/LINE config preservation and delivery behavior reported here.

## What This Changes

The issue requests that `streaming.preview.toolProgress: false` be preserved and honored for channels such as WhatsApp, LINE, Sendblue, and Slack even when the channel config does not define `streaming.mode`.

## Best Possible Solution

Add a supported, documented suppression path for non-preview-streaming channels: either extend the shared channel config contract so WhatsApp and LINE can preserve and honor `streaming.preview.toolProgress: false`, or add a global/default tool-progress suppression setting with clear per-channel precedence. The fix should update strict schemas, generated metadata, channel docs, and regression tests for config validation/round-trip plus outbound progress delivery.

## Reproduction Assessment

Yes. A high-confidence code reproduction is to put `channels.whatsapp.streaming.preview.toolProgress: false` or the equivalent LINE config into the config path that validates/repairs channel config: the strict channel schemas do not define `streaming`, unknown-key repair removes unsupported keys, and the current runtimes do not pass `suppressDefaultToolProgressMessages` for those channels.

## Solution Assessment

Unclear. The reporter's suggested options are plausible, but the best maintainable fix needs maintainer judgment on whether suppression should be a per-channel streaming config, a broader global default, or both with explicit precedence and no change to approval/error/media delivery.
