## Summary

Keep this PR open. Current main still does not implement the PR's intended malformed `required` cleanup: shared tool-parameter normalization can pass `required: null` through for non-Gemini/OpenAI-compatible providers, and the Gemini cleaner still preserves nested non-array `required` values. The PR remains a focused implementation candidate for the linked Bailian GLM-5 rejection, but it needs rebasing/adaptation to the current schema-normalization layout and a maintainer call on the Gemini key-copy hardening note.

## Best Possible Solution

Keep this PR open and ask for a rebase onto current main. The best fix is to strip own non-array `required` fields in the shared tool-parameter schema normalization path before provider-specific cleaning, preserve valid `required` arrays, add regression coverage for top-level Bailian/OpenAI-compatible schemas and nested Gemini schemas, and explicitly resolve whether Gemini dynamic-key hardening belongs in this PR or a tracked follow-up.
