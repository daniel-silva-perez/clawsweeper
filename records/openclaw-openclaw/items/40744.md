## Summary

Keep this PR open. Current main at 68561a8c948b49e85ae012646de550a49d6c63a still does not strip own non-array `required` values from the shared tool-schema normalization path, and the Gemini cleaner still only drops empty `required` arrays while copying malformed non-array values through. The PR remains a focused implementation candidate for the Bailian GLM-5 failure discussed in #40568, but it needs a rebase onto the current schema-normalization layout and a maintainer decision on the existing Gemini key-copy hardening note.

## Best Possible Solution

Keep this PR open and request a rebase onto current main. The best implementation is to strip own non-array `required` fields in the shared tool-parameter schema normalization path before provider-specific cleaning, preserve valid `required` arrays, add regression tests for top-level strict OpenAI-compatible/Bailian-style schemas and nested Gemini schemas, and explicitly resolve whether the Gemini dynamic-key hardening is part of this PR or a tracked follow-up.
