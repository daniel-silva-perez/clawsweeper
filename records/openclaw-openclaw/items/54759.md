## Summary

Keep this PR open. Current main still has the OpenAI-compatible HTTP path hardcoded to generate `openai` session-key prefixes, and the existing e2e test still asserts `openai-user:alice`; the latest release tag shows the same behavior. The PR is a focused implementation candidate for the linked bug rather than something current main has already shipped.

## Best Possible Solution

Keep this PR open as a focused bug-fix candidate. The best path is to review or replace it with a main-branch change that derives the OpenAI-compatible generated session-key prefix from the resolved agent/provider, adds hermetic coverage for Anthropic/OpenAI/Google-style config switching, and explicitly documents or tests the compatibility behavior for existing implicit sessions.

## Risks / Open Questions

- Changing generated session keys is behaviorally non-backward-compatible for clients that rely on implicit `/v1/chat/completions` session continuity; existing explicit `x-openclaw-session-key` callers are less affected.
- The proposed approach should be reviewed for model override semantics: if `x-openclaw-model` can route a single request to a provider different from the agent default, maintainers should decide whether the session prefix should follow the default provider or the per-request override.
