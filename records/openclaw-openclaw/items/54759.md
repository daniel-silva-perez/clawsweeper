## Summary

Keep open. Current main still hard-codes the OpenAI-compatible chat-completions session prefix, so the PR is not obsolete, but the proposed patch has a correctness gap: it derives the prefix from the agent default before honoring `x-openclaw-model`, so override requests can still get a misleading provider prefix.

## What This Changes

The PR adds provider-prefix resolution for OpenAI-compatible HTTP session keys, threads a pre-resolved agent id through gateway request context, and adds gateway tests for provider-prefixed generated session keys.

## Best Possible Solution

Maintainers should first decide the intended namespace for generated OpenAI-compatible HTTP sessions. If provider-prefixed keys are desired, the implementation should derive the prefix from the accepted backend model selection, including `x-openclaw-model`, preserve explicit `x-openclaw-session-key`, rebase onto current gateway config access, and document or test the compatibility impact.

Overall correctness: patch is incorrect

Overall confidence: 0.82

- **[P2] Derive the prefix after resolving model overrides:** `src/gateway/openai-http.ts:449`
  - body: The OpenAI-compatible endpoint supports `x-openclaw-model` as the backend provider/model override, but this computes `providerPrefix` from the agent default before `resolveOpenAiCompatModelOverride`. A default Anthropic agent handling `x-openclaw-model: openai/gpt-5.4` would dispatch an OpenAI run under an `anthropic-*` session key, leaving the misleading-provider behavior in place for override requests.
