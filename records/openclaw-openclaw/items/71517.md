## Summary

Keep this PR open. Current main still lacks the shared filename decoder and still leaves Feishu inbound payload filenames largely raw, while the latest provided head remains relevant to the linked filename-decoding reports and addresses the earlier path-sanitization review concern.

## What This Changes

The PR adds a shared media filename decoder, routes remote media Content-Disposition parsing through it, exposes filename helpers through plugin SDK surfaces, and normalizes Feishu inbound filenames before saving media.

## Best Possible Solution

Review and rebase this PR against current main, keep the shared decoder in the generic media layer, expose only the SDK surface that current plugin-boundary rules support, and leave #48788 open for the broader channel migration after this scoped fix lands.

## Reproduction Assessment

Yes. Code inspection on current main gives a high-confidence reproduction path: RFC 5987 language-tagged `filename*` values still go through the narrow local parser in `src/media/fetch.ts`, and Feishu payload `file_name` values still reach inbound save paths without the PR's mojibake recovery and basename normalization.

## Solution Assessment

Yes for the scoped first step. A shared media decoder plus Feishu inbound normalization is the narrow maintainable direction, but the final merge should align the SDK export with current main rather than reviving the stale Feishu-specific SDK subpath.
