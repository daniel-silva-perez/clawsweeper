## Summary

Keep open. The underlying mixed exec/non-exec heartbeat queue bug is still real on current main, but this PR should not merge as-is because its restore helper appends deferred inspected events after any newer tail events, breaking FIFO ordering.

## What This Changes

The PR adds a system-event restore helper, changes exec-heartbeat cleanup to restore non-exec inspected events, and adds regression coverage for mixed exec/non-exec queues plus duplicate deferred reminders.

## Best Possible Solution

Keep the narrow deferred-event preservation fix, but restore deferred inspected entries ahead of any uninspected tail only after the inspected prefix is actually consumed, while preserving duplicate reminders, `MAX_EVENTS` trimming, and queue metadata semantics with focused tests.

## Reproduction Assessment

Yes. A deterministic unit path is to snapshot `[Reminder A, exec E]`, enqueue `Reminder B` before cleanup, then consume the inspected prefix and restore deferred entries; the PR's append-based helper yields `[Reminder B, Reminder A]` instead of preserving FIFO order.

## Solution Assessment

No. Restoring deferred entries is the right direction, but appending them after the current queue is not the safest implementation; the helper should rebuild or prepend the deferred inspected entries ahead of the uninspected tail and guard on successful prefix consumption.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve deferred-event FIFO order:** `src/infra/system-events.ts:191`
  - body: `restoreSystemEventEntries` appends restored entries after whatever remains in the queue. Because heartbeat cleanup runs after the model call, another system event can be queued after preflight but before cleanup; the later event then stays ahead of older deferred entries. Restore the deferred prefix ahead of the tail, and only restore after the inspected prefix was actually consumed.
