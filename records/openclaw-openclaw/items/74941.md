## Summary

Keep open. The targeted bug is real on current main, but this PR can reorder older deferred events behind later queued events and can restore stale deferred entries even when the inspected prefix was not consumed.

## What This Changes

The PR adds a system-event restore helper, changes exec heartbeat cleanup to restore non-exec inspected events, and adds regression coverage for mixed exec/non-exec queues plus duplicate deferred reminders.

## Best Possible Solution

Keep the narrow deferred-event preservation direction, but restore deferred inspected entries ahead of any uninspected tail only after the inspected prefix is actually removed, while preserving duplicate reminders, `MAX_EVENTS` trimming, and queue metadata semantics.

## Reproduction Assessment

Yes. Snapshot `[Reminder A, exec E]`, enqueue `Reminder B` before cleanup, then consume the inspected prefix and restore deferred entries; the PR's append helper yields `[Reminder B, Reminder A]` instead of preserving FIFO order.

## Solution Assessment

No. Restoring deferred entries is the right direction, but appending them after the current queue and ignoring the consume result is not the safest implementation; the helper or caller should rebuild/prepend the deferred prefix ahead of the tail and skip restore when prefix consumption fails.
