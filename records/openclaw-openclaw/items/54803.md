## Summary

Keep open, but this PR should not merge as-is. The user-facing `/status` improvement is still valid, yet the branch applies it to a stale inline renderer while bundling unrelated Fly.io, Docker, workflow, runtime-config, docs-template, and SDK changes with blocking correctness and security concerns.

## What This Changes

The PR adds usage-fetch error messages to chat `/status` and also changes Fly.io deployment workflow/runtime scripts, Docker runtime/tooling, runtime config sync, gateway `/health` handling, docs templates, and channel plugin SDK outbound-send imports.

## Best Possible Solution

Land the status UX change as a narrow current-main patch in `src/status/status-text.ts`, with regression coverage for structured `usageEntry.error` and thrown or timed-out usage-summary failures. Split or drop the Fly.io, Docker, workflow, runtime-config, docs-template, and SDK import changes into separate owner-reviewed PRs.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to mock `loadProviderUsageSummary` for `buildStatusText` so it returns a provider entry with `error` or rejects; current main skips the structured error and suppresses the thrown error instead of rendering a usage line.

## Solution Assessment

No. The PR's stated UX direction is good, but this branch is not the best solution because it changes a stale renderer and mixes broad deployment/security-sensitive changes into a small status rendering fix.

  - body: The diff adds error rendering to the old inline status command, but current main routes chat `/status` through `buildStatusText`. After rebasing, this needs to live in `src/status/status-text.ts` or provider errors and timeouts will still be hidden.
  - confidence: 0.86

  - body: This startup sync rewrites `agents.defaults.workspace` to `${stateDir}/workspace` whenever `OPENCLAW_WORKSPACE_DIR` is unset. Existing deployments with a custom workspace in `openclaw.json` would silently move agent context/session paths after restart.
  - confidence: 0.89
- **[P1] Let the gateway start when Tailscale bootstrap fails:** `scripts/fly-entrypoint.sh:90`
  - body: Because the entrypoint runs with `set -e`, a non-zero `tailscale up` exits before `exec "$@"`. A transient Tailnet or auth-key failure would stop the whole gateway instead of only disabling Tailscale exposure.
  - confidence: 0.88
- **[P1] Drop privileges before starting OpenClaw:** `Dockerfile:262-263`
  - body: The branch removes the previous `USER node` hardening and the new entrypoint ultimately execs the gateway as root. If root is required for bootstrapping, the runtime app should drop to the node user before starting OpenClaw.
  - confidence: 0.87
- **[P3] Use the shared probe handler for /health:** `src/gateway/server-http.ts:771-774`
  - body: `/health` is already registered as a live gateway probe and current main handles it before config loading. This exact-URL shortcut returns a different shape and leaves the shared probe map with a dead duplicate path.
  - confidence: 0.8

## Security Review

Status: needs_attention

Summary: Security review needs attention because the diff introduces container privilege and supply-chain risks unrelated to the `/status` UX fix.

Concerns:

- **[high] Gateway would run as root in the Docker image:** `Dockerfile:262`
  - body: The PR removes the existing non-root runtime hardening and adds an entrypoint that ultimately execs the gateway as root, broadening the impact of any runtime compromise.
  - confidence: 0.87
- **[medium] Root-executed installers are not pinned:** `Dockerfile:131`
  - body: The Dockerfile executes remote Bun and Tailscale install scripts without version pins, so each build can execute different upstream code as root and produce a less auditable image.
  - confidence: 0.84
- **[medium] Persistent root operator config increases blast radius:** `scripts/fly-entrypoint.sh:38`
  - body: The entrypoint symlinks root-scoped Codex, GitHub CLI, Git, and SSH configuration into the Fly volume; combined with running the gateway as root, a runtime compromise could persist and reuse operator credentials across restarts.
  - confidence: 0.78
