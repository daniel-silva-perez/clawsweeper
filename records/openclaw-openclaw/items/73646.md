## Summary

Keep open. Current main still logs the Crestodian no-TTY error and returns normally on the explicit subcommand path, so the CLI wrapper does not convert it into a non-zero exit. Two open unmerged PRs already target the bug, so this issue should stay open until one is reviewed and landed.

## What This Changes

The issue asks for `openclaw crestodian` to exit non-zero when invoked without an interactive TTY and without a `--message` one-shot request.

## Best Possible Solution

Review and land one focused fix PR that makes the explicit `runCrestodian()` no-TTY branch fail with exit code 1, adds regression coverage for stdin-not-TTY and stdout-not-TTY, and preserves `--message` and `--json` headless behavior.

## Reproduction Assessment

Yes. The current-main source gives a high-confidence reproduction path: call `runCrestodian()` without `message` or `json` using non-TTY input/output, and it logs the error then returns normally. The reporter also provided deterministic shell reproductions against v2026.4.27.

## Solution Assessment

Yes. The narrowest maintainable fix is to make the existing Crestodian no-TTY guard fail through the CLI's normal non-zero path, with unit coverage around that branch; a broader shared prompt/TTY refactor can stay separate from this issue and #73562.
