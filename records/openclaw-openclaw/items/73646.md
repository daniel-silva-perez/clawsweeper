## Summary

Keep open. Current main still has the explicit Crestodian no-TTY path log the error and return normally, while the surrounding CLI wrapper only exits non-zero when an error is thrown. The issue is also already paired with two open, unmerged closing PRs, so it should remain open until maintainers review and land one fix.

## What This Changes

Requests that `openclaw crestodian` fail with a non-zero exit code when run without an interactive TTY and without `--message`, matching sibling CLI commands.

## Best Possible Solution

Land one focused fix that makes the `runCrestodian()` no-TTY guard fail with exit code 1, adds regression coverage for stdin-not-TTY and stdout-not-TTY, and preserves `--message` plus `--json` headless behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: invoke `runCrestodian()` without `message` or `json` while stdin or stdout is not a TTY, and it logs the error then returns normally. The reporter also provided deterministic shell reproductions against v2026.4.27.

## Solution Assessment

Yes. The narrowest maintainable fix is to make the existing Crestodian no-TTY guard fail through the CLI's normal non-zero path with unit coverage; a broader shared TTY helper can be handled separately from this issue and #73562.
