## Summary

Keep open. Current main still lets the explicit Crestodian no-TTY path log an error and return successfully, while two open, unmerged closing PRs already target the same narrow bug, so the issue should wait for maintainer review and one merged fix.

## What This Changes

Make `openclaw crestodian` exit non-zero when invoked without an interactive TTY and without `--message`, matching sibling interactive CLI commands.

## Best Possible Solution

Land one focused fix that makes the `runCrestodian()` no-TTY guard exit non-zero, preferably with stdin-not-TTY and stdout-not-TTY regression coverage, while preserving `--message` and `--json` headless behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: call `runCrestodian()` without `message` or `json` while stdin or stdout is not a TTY, and it logs the error then returns normally; the reporter also provided deterministic shell reproductions against v2026.4.27.

## Solution Assessment

Yes. The narrowest maintainable solution is to change the existing Crestodian no-TTY guard to fail through the CLI's normal non-zero path and add targeted regression tests; a broader shared TTY helper can be handled separately with #73562 if maintainers want that cleanup.
