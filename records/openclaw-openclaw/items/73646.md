## Summary

Keep this issue open. Current main still has the explicit `openclaw crestodian` no-TTY path log the error and return normally, so the shared command wrapper has no thrown error to convert into exit 1. The related implementation PR #73654 is closed and unmerged, so it no longer protects or resolves the issue.

## Best Possible Solution

Keep the issue open and land an equivalent of the closed PR #73654: make the explicit `runCrestodian()` no-TTY branch throw or otherwise set exit 1 through the runtime, add a regression test for `openclaw crestodian < /dev/null` semantics, and preserve `openclaw crestodian --message ...` automation.
