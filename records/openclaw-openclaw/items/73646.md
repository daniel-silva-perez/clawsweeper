## Summary

Keep this issue open. Current main still has the reported Crestodian explicit-subcommand no-TTY behavior: `runCrestodian()` logs the TTY error and returns normally, while the CLI wrapper only exits non-zero through its catch path. The linked fix PRs are closed unmerged, so there is no merged or shipped implementation to close against.

## Best Possible Solution

Keep the issue open and land a focused fix: make the explicit `runCrestodian()` no-TTY branch fail through the existing CLI error path, preferably by throwing the same message or otherwise setting exit 1, then add regression coverage for non-TTY input while preserving `--message` and `--json`.
