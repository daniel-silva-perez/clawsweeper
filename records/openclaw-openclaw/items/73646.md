## Summary

Keep open. Current main still logs the Crestodian no-TTY error and returns normally on the explicit subcommand path, so the CLI wrapper does not force a non-zero exit. Open PRs #73928 and #74059 already target this bug with closing syntax, so the issue should remain open until maintainers review and land one fix.

## What This Changes

The issue asks for `openclaw crestodian` to exit non-zero when launched without an interactive TTY and without a `--message` or `--json` headless path.

## Best Possible Solution

Review and land one focused fix PR that makes the explicit `runCrestodian()` no-TTY guard fail with exit code 1, adds regression coverage for stdin-not-TTY and stdout-not-TTY, and preserves `--message` and `--json` headless behavior.

## Reproduction Assessment

Yes. The current-main source gives a high-confidence reproduction path: call `runCrestodian()` without `message` or `json` using non-TTY input/output, and it logs the error then returns normally. The reporter also supplied deterministic shell reproductions against v2026.4.27.

## Solution Assessment

Yes. The narrowest maintainable fix is to make the existing Crestodian no-TTY guard fail through the CLI's normal non-zero path with unit coverage; a broader shared TTY helper can be handled separately with #73562.
