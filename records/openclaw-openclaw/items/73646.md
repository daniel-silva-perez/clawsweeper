## Summary

Keep open. Current main still has the Crestodian no-TTY path log an error and return normally, so the explicit `openclaw crestodian` command can still complete successfully. The issue is already paired with two open, unmerged PRs that target the same bug, so it should remain open until maintainers review and land one fix.

## What This Changes

Request to make `openclaw crestodian` exit non-zero when invoked without an interactive TTY and without `--message`, matching sibling interactive CLI commands.

## Best Possible Solution

Land one focused fix that makes the existing `runCrestodian()` no-TTY guard fail with exit code 1, adds regression coverage for stdin-not-TTY and stdout-not-TTY, and preserves `--message`, `--json`, and interactive TUI behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: call `runCrestodian()` without `message` or `json` while stdin or stdout is not a TTY, then observe that the explicit subcommand wrapper sees a normal return instead of a failure. The reporter also provided deterministic shell reproductions against v2026.4.27.

## Solution Assessment

Yes. The narrowest maintainable solution is to change the existing Crestodian no-TTY guard to use the CLI's normal non-zero failure path and add unit coverage; a broader shared TTY helper can be handled separately with the related plugins-uninstall prompt issue.
