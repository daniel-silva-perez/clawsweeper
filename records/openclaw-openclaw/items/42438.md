## Summary

Keep this issue open. Current main still supports the silent-drop half through `dmPolicy: "allowlist"`, but it does not provide an owner notification/webhook for blocked DM attempts or a configurable way to suppress live pairing replies while still recording/notifying. The related #64081 tracks background task notification defaults, not channel DM access control.

## Best Possible Solution

Keep this open and implement it as an additive channel access-control feature: preserve current `allowlist` silent blocking, add an explicit blocked-DM owner-notification event/config path with privacy-minimal payloads, and wire it through WhatsApp and Telegram first. If pairing-mode reply suppression is desired, add a separate `pairingResponse`/`replyOnPairingRequest` option rather than changing the meaning of `allowlist` or the default `pairing` policy.

## Risks / Open Questions

- A blocked-DM notification feature should be opt-in and metadata-minimal so sender IDs, names, and message content are not leaked to plugins or webhooks unexpectedly.
- Adding new `dmPolicy` values would affect the public config/schema/SDK contract and every DM-capable channel; a separate additive `notifyOnBlocked` or blocked-DM event may be safer than overloading the enum.
- Pairing-reply suppression and owner notification are related but distinct behaviors and should likely be separate config knobs so existing pairing security defaults do not silently change.
