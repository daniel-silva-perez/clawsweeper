## Summary

Keep this issue open. Current main covers the silent-drop half through `dmPolicy: "allowlist"`, but it does not implement an owner notification/webhook policy for blocked DM attempts, nor a configurable way for pairing mode to suppress live pairing replies while still recording or notifying the owner. The cross-referenced #64081 is about background task notification defaults, not channel DM access control.

## Best Possible Solution

Keep this open and design it as an additive channel access-control feature: preserve existing `allowlist` silent blocking, then add an explicit owner-notification path for blocked DM attempts, likely via a shared blocked-DM event/config knob with per-channel wiring for WhatsApp and Telegram. If pairing reply suppression is desired, make that a separate explicit pairing-response option rather than overloading `allowlist`.

## Risks / Open Questions

- A notify-on-blocked feature should be opt-in and metadata-minimal so it does not leak message content or sensitive sender details to plugins/webhooks unexpectedly.
- Adding new shared `dmPolicy` values would affect all DM-capable channels, config schemas, generated metadata, docs, setup UI, and plugin SDK contracts.
