## Summary

Keep open. Current main has added mitigations and trace/warn timing for embedded-run startup, but it still routes eligible Active Memory turns through a blocking lightweight embedded run and has not proven or implemented the requested sub-second bootstrap/low-latency path. The fresh discussion and related open performance reports point to remaining runEmbeddedPiAgent/setup latency rather than an obsolete report.

## What This Changes

Request to profile and optimize `runEmbeddedPiAgent` lightweight bootstrap latency for Active Memory blocking pre-reply runs, and document realistic timeout and latency guidance.

## Best Possible Solution

Keep this issue as the focused Active Memory lightweight embedded-runner latency/profiling task until maintainers have measured current main on a representative Docker/provider setup, identified the slow pre-prompt stages, and either cached/bypassed safe per-run invariants or added a warm-reuse API while preserving isolation, timeout, fail-open, and memory-tool allowlist behavior. After that, update the Active Memory docs with measured timeout and latency guidance.
