## Summary

Keep open. Current main still routes Active Memory through a blocking before-prompt embedded Pi run with `bootstrapContextMode: "lightweight"`, but the source shows timeout/fail-open mitigation and tuning docs rather than a shipped profiling/optimization fix proving lightweight embedded bootstrap is fast enough for blocking plugins. The fresh issue discussion reports persistence on v2026.4.24, after the original v2026.4.12 report.

## Best Possible Solution

Keep this issue open as the focused Active Memory lightweight embedded-runner latency task. Add stage timing around queue wait, runtime plugin loading, hook/model/harness/auth resolution, tool creation and allowlist filtering, bootstrap context, bundle/LSP tooling, session manager open/repair, context-engine bootstrap/maintenance, resource loader reload, embedded session creation, and first prompt start. Then cache or bypass safe per-run invariants for lightweight blocking runs while preserving hard timeout/fail-open behavior. After the root cause is fixed, update https://docs.openclaw.ai/concepts/active-memory with measured timeout and latency guidance.
