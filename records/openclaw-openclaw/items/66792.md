## Summary

Keep open. Current main has useful timeout and documentation mitigations, but it still routes Active Memory through a blocking `before_prompt_build` embedded run with `bootstrapContextMode: "lightweight"`, and the embedded runner still performs substantial queue/setup/tool/session/context work before the first prompt-start log. The provided discussion also includes fresh v2026.4.24 evidence that the symptom persisted after the original v2026.4.12 report.

## Best Possible Solution

Keep this issue open as the focused Active Memory lightweight embedded-runner latency task. Add stage timing around queue wait, runtime plugin loading, hook/model/harness/auth resolution, tool creation and allowlist filtering, bootstrap context, bundled LSP/MCP decisions, session manager open/repair/prewarm, context-engine bootstrap/maintenance, resource loader reload, embedded session creation, and first prompt start. Then cache or bypass safe per-run invariants for lightweight blocking runs while preserving the hard timeout/fail-open behavior. After the root cause is fixed, update https://docs.openclaw.ai/concepts/active-memory with measured timeout and latency guidance.
