## Summary

Keep open: current main still short-circuits the fresh-token preflight path before token-threshold compaction is evaluated, and the issue is already paired with open unmerged implementation PRs for the same bug.

## What This Changes

The issue reports that proactive compaction stops re-firing after fresh persisted token totals cause the preflight compaction path to return early.

## Best Possible Solution

Land or replace a focused preflight fix so fresh persisted totals at or above `contextWindow - reserveTokensFloor - softThresholdTokens` reach the existing threshold helper, while below-threshold fresh totals still avoid expensive transcript estimation and the transcript-byte compaction trigger remains independent. Add a regression test for `totalTokensFresh: true`, cached tokens above threshold, and no transcript-byte trigger.

## Reproduction Assessment

Yes. A deterministic source-level reproduction is a `SessionEntry` with `sessionId`, fresh persisted `totalTokens` above the threshold, and no active transcript-byte trigger; current main returns at `agent-runner-memory.ts:454` before calling `shouldRunPreflightCompaction`.

## Solution Assessment

Yes, the best fix direction is to gate the fresh-token fast path on the computed threshold or otherwise evaluate `shouldRunPreflightCompaction` before returning. This is narrower than changing `compactionCount` semantics, which current code already increments monotonically.
