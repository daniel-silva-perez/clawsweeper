## Summary

Keep this issue open. Current main has fixed the old filename coupling for newly created cron job ids by using UUIDs, but the reported user-facing path is still broken: cron still rejects explicit `session:<sessionKey>` targets and `current` session binding when the channel-native session key contains `/` or `\`. The related implementation PR #64057 was closed unmerged, and current tests still assert the rejection behavior.

## Best Possible Solution

Implement a cron-boundary fix that treats `session:<id>` and resolved `current` session keys as opaque runtime identifiers while keeping run-log filenames UUID-backed or encoded and preserving NUL/path traversal protections only at filesystem boundaries. Add regression coverage for slash-bearing DingTalk-style session keys through normalize, create/update, current-session binding, persisted validation/manual-run, and run-log safety.

## Risks / Open Questions

- A fix should not simply remove filesystem-boundary protections; NUL and run-log job id path traversal checks should remain at `resolveCronRunLogPath` or equivalent filename boundaries.
- Persisted jobs with currently invalid `session:` targets load with a warning and then fail closed on validation/manual-run paths, so repair or compatibility behavior should be intentional rather than silently rewriting opaque session keys.
