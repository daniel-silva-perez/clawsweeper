## Summary

Keep open. Current main still rejects slash- and backslash-bearing cron `session:<id>` targets even though cron job ids are now UUID-backed, so the original filename-coupling concern is only partly addressed. The linked fix PR #64057 was closed unmerged, and current tests still assert the rejection behavior instead of covering DingTalk-style native conversation ids.

## What This Changes

The issue asks cron to accept channel-native session keys containing path separators by encoding or otherwise decoupling session keys from filesystem-backed job/run-log identifiers.

## Best Possible Solution

Implement a cron-boundary fix that treats `session:<id>` payloads and resolved `current` session keys as opaque runtime session keys, while keeping run-log filenames keyed by safe UUID job ids and preserving strict validation at actual filesystem path boundaries.

## Repair Work Prompt

Fix the cron session-target bug reported in openclaw/openclaw#64030. Current main already creates UUID-backed cron job ids, but `assertSafeCronSessionTargetId` still rejects `/` and `\`, so explicit `session:<sessionKey>` targets and resolved `current` targets fail for channel-native ids such as DingTalk group conversation ids. Treat cron session target ids as opaque runtime session keys instead of filesystem names: preserve non-empty validation and NUL rejection, but allow path separators in `session:<id>` and `current`-resolved session keys. Do not change `buildAgentSessionKey`, do not normalize channel peer ids, and do not weaken `assertSafeCronRunLogJobId` or `resolveCronRunLogPath`; filesystem job/run-log ids must remain safe. Prefer preserving the raw session key in persisted `sessionTarget` rather than encoding it unless every session resolution path decodes consistently. Update regression tests for explicit DingTalk-style `session:agent:main:dingtalk:group:cid3tmd4xb19xjfk/wogxwy2a==`, `current` session binding with the same shape, create/update/manual validation, persisted load/run fail-closed behavior, and unchanged run-log path traversal protection. Add a single-line user-facing changelog fix entry with `Thanks @BrilliantWang`. Use #64057 as prior context, but do not copy its run-log guard weakening if it is no longer needed.
