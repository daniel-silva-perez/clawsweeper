## Summary

Keep open. Current main covers the reporter's concrete `multimedia.nt.qq.com.cn` example through the default `*.qq.com.cn` QQ Bot media allowlist, but it still does not expose or wire a supported `channels.qqbot.mediaAllowHosts` option. The remaining work is valid and narrow, but it changes an SSRF allowlist surface, so it needs maintainer/security approval before an automated fix lane should implement it.

## What This Changes

Add `channels.qqbot.mediaAllowHosts` so QQ Bot media downloads can merge user-configured hostnames with the default SSRF allowlist.

## Best Possible Solution

Implement the option inside the bundled QQ Bot plugin, not core: add the config type/schema/manifest/docs metadata, normalize configured host patterns through existing SSRF policy semantics, merge them with the conservative QQ/Tencent defaults, and pass the resulting policy through both inbound attachment downloads and outbound URL fallback downloads while preserving HTTPS-only and private-network protections.

## Reproduction Assessment

Yes for the remaining config gap: static inspection of current main shows no QQ Bot `mediaAllowHosts` schema/type/docs/manifest surface and both media download paths still call the fixed `downloadFile` helper. The original `multimedia.nt.qq.com.cn` host failure is no longer the best reproduction because `*.qq.com.cn` is now in the default allowlist.

## Solution Assessment

Unclear until maintainers approve the security semantics. The narrow maintainable direction is a QQ Bot-owned config and policy merge, not a core migration or a broad media fetch policy change.
