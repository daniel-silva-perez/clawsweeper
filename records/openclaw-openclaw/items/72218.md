## Summary

Keep open. The PR addresses a real current-main heartbeat payload leak, but the branch is not merge-ready because it strips heartbeat final payloads in message mode before the heartbeat runner can apply heartbeat-mode `ackMaxChars` suppression, which would make short token-prefixed acks visible as chat text.

## What This Changes

The PR changes `buildReplyPayloads` so heartbeat final payloads also pass through `HEARTBEAT_OK` sanitation, keeps Bun socket error formatting scoped to non-heartbeat runs, and adds focused payload-builder regression tests.

## Best Possible Solution

Keep the narrow payload-layer fix direction, but preserve heartbeat ack semantics: exact OK-only heartbeat final payloads should be suppressed before channel delivery, while token-prefixed short heartbeat acks must still be handled by heartbeat-mode `ackMaxChars` logic or an equivalent local check with focused regression coverage.

## Reproduction Assessment

Yes. Current main can be reproduced through `buildReplyPayloads({ isHeartbeat: true, payloads: [{ text: "HEARTBEAT_OK" }] })`, and the PR regression can be reasoned from `HEARTBEAT_OK all good` being converted to `all good` before heartbeat-mode suppression runs.

## Solution Assessment

No, not as written. Removing the raw heartbeat bypass is the right direction, but applying message-mode stripping to all heartbeat final payloads is not the narrowest safe fix because it changes the documented heartbeat ack contract.
