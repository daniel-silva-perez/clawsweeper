## Summary

Keep this PR open, but not merge-ready as written. Current main still has the heartbeat final-payload bypass the PR targets, yet the branch now strips heartbeat tokens in message mode before the heartbeat runner can apply heartbeat-mode `ackMaxChars` suppression, which can turn a short ack-style heartbeat reply into visible text.

## What This Changes

This PR changes `buildReplyPayloads` so heartbeat final payloads also pass through `HEARTBEAT_OK` token sanitation, keeps Bun socket error expansion scoped to non-heartbeat runs, and adds focused payload-builder tests.

## Best Possible Solution

Keep the narrow payload-layer fix, but preserve heartbeat ack semantics: exact OK-only heartbeat final payloads should be suppressed before channel delivery, while heartbeat replies that include `HEARTBEAT_OK` plus short ack text must still reach the heartbeat runner's `mode: "heartbeat"` / `ackMaxChars` logic or be handled equivalently with focused regression coverage.

## Reproduction Assessment

Yes. The original issue has a direct code-path reproduction through `buildReplyPayloads({ isHeartbeat: true, payloads: [{ text: "HEARTBEAT_OK" }] })` on current main, and the PR regression can be reasoned from `HEARTBEAT_OK all good` being stripped in message mode before heartbeat-mode suppression runs.

## Solution Assessment

No, not as written. Removing the bypass is the right direction, but applying message-mode stripping to all heartbeat final payloads is not the safest fix because it can erase the token before heartbeat-specific ack suppression sees it.
