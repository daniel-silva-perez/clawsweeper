## Summary

Keep open. Current main still lacks a config-level exec path denylist, and the open unmerged PR #75101 explicitly targets this issue, so the issue should remain open until that PR is reviewed and landed or maintainers choose a different security design.

## What This Changes

Request an opt-in `tools.exec.denyPathPatterns` config surface that blocks exec commands whose literal path-like arguments match configured globs before allowlist, approval, or execution.

## Best Possible Solution

Review PR #75101 as the canonical implementation candidate. The desired end state is a default-off, security-reviewed hard-deny path policy enforced in the shared exec policy path before allowlist/approval execution, with config types/schema/help/docs and focused tests for literal argv matching, tilde/home handling, agent/global precedence, and documented v1 bypasses.

## Reproduction Assessment

Yes for the missing feature: current-main source, schema, docs, and exec policy review show no accepted `denyPathPatterns` key and no runtime path-pattern gate. The reported production token leak itself is not publicly reproducible from the repository.

## Solution Assessment

Unclear until maintainer security review. A narrow default-off hard-deny gate in shared exec policy looks like the maintainable direction, while action modes such as warn or require-approval should not be assumed without product/security sign-off.
