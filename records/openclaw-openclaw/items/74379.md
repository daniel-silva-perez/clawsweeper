## Summary

Keep open. Current main still lacks the requested exec path-deny configuration and runtime gate, and open PR #75101 explicitly targets this issue with closing syntax, so the issue should remain open until that PR is reviewed, merged, or replaced.

## What This Changes

The issue requests an opt-in `tools.exec.denyPathPatterns` configuration gate that blocks exec commands whose literal path-like arguments match sensitive path globs before execution.

## Best Possible Solution

Review and land PR #75101, or a narrow replacement, that adds a pre-approval hard-deny `tools.exec.denyPathPatterns` gate, updates config schema/types/help/docs, covers global plus per-agent semantics, and documents v1 bypass limits such as indirection and symlinks.

## Reproduction Assessment

Yes for the missing control: current main has no accepted `tools.exec.denyPathPatterns` config key and no runtime deny-path phase in the exec policy paths. The private production leak itself is not publicly reproducible from the available issue data.

## Solution Assessment

Unclear until PR #75101 is reviewed. A shared pre-approval hard-deny gate appears to be the right direction for this secret-exfiltration class, but the exact config semantics and documented limits need maintainer review.
