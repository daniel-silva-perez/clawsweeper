## Summary

Keep open. Current main still lacks the requested exec path-deny config and runtime gate, and an open PR explicitly targets this issue with closing syntax, so the issue should remain open until that PR is reviewed, merged, or replaced.

## What This Changes

Request a config-level `tools.exec.denyPathPatterns` policy that blocks exec commands whose literal path-like arguments match sensitive glob patterns before execution.

## Best Possible Solution

Review PR #75101 or a narrow replacement that adds a hard-deny `tools.exec.denyPathPatterns` gate before allowlist/approval execution, updates config schema/types/help/docs, and covers literal argv, tilde/home, cwd-relative paths, shell-wrapper payloads, safeBins interaction, and documented bypass limits.

## Reproduction Assessment

Yes for the missing control: current main has no accepted `tools.exec.denyPathPatterns` config key and no runtime deny-path phase in the exec policy paths. The private production leak itself is not publicly reproducible from the available issue data.

## Solution Assessment

Unclear until PR #75101 is reviewed. A shared pre-approval hard-deny gate appears directionally right for the reported secret-exfiltration class, but the exact config semantics and documented bypass scope need maintainer security/product review.
