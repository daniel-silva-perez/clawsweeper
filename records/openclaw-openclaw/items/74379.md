Keep open. Current main still lacks the requested config-backed exec path denylist, and open PR #75101 already targets this issue with closing syntax, so the issue should stay open until that implementation is reviewed, merged, or replaced.

## What This Changes

Request a `tools.exec.denyPathPatterns` config option that blocks exec commands whose literal path-like arguments match sensitive file globs before execution.

## Best Possible Solution

Review and land PR #75101, or a narrow replacement, if it adds a shared pre-exec hard-deny path gate with schema/types/help/docs/tests and clear v1 bypass limits.

## Reproduction Assessment

Yes for the missing control: source inspection on current main shows no accepted `tools.exec.denyPathPatterns` config key and no runtime deny-path phase before allowlist or approval handling. The private production leak itself is not publicly reproducible from the issue data.

## Solution Assessment

Unclear until the linked implementation is reviewed. A shared hard-deny gate before allowlist/approval is the right direction for this class of secret exfiltration, but the exact config shape and documented bypass scope need maintainer security/product review.
