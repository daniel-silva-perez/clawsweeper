## Summary

Keep this issue open. Current main still has a path where Anthropic/stream JSON.parse failures become assistant errorMessage text, and the user-facing sanitizers do not recognize JSON parse SyntaxError messages like the reported `Expected ':' after property name in JSON at position ...`. Related #59118 is still open and #60593 tracks a broader recurring version, but neither is merged or canonical enough to close this issue.

## Best Possible Solution

Keep the issue open and land a narrow core fix in the agent error-formatting/sanitization path: detect V8/JSON.parse SyntaxError shapes in `formatAssistantErrorText` and `sanitizeUserFacingText` only under error context, return friendly user copy, and add regression coverage for the delivery path. Separately decide whether these malformed Anthropic/tool-call stream failures should trigger retry or failover, likely coordinated with #60593 and any PR such as #59118.

If that throws, the catch path calls `failTransportStream`, which stores `error.message` directly in `output.errorMessage`. That matches the raw V8 JSON.parse wording reported by the user.

## Risks / Open Questions

- A sanitizer-only fix would stop raw error leakage but may leave retry/failover behavior unresolved for the recurring Anthropic stream failures tracked in #60593.
- JSON parse pattern matching should be gated to explicit error contexts so ordinary user text or code examples are not rewritten.
- A good regression test should mock the post-compaction/Anthropic error payload through the reply delivery path instead of depending on a flaky live compaction reproduction.
