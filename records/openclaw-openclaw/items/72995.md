## Summary

Keep open. Current main still has the reported failure class: the managed web_search path does not propagate the agent tool AbortSignal into the web-search provider contract or Gemini runtime, and provider-origin AbortError failures still bypass the normal tool failure logging path. The exact intermittent live Gemini trigger is not proven here, but the source-backed cancellation/logging gap is still present.

## What This Changes

Fix intermittent Gemini web_search aborts so provider aborts are distinguishable from real run cancellation and produce useful gateway diagnostics.

## Best Possible Solution

Add a backwards-compatible web-search execution context that carries AbortSignal and abort diagnostics through createWebSearchTool, RunWebSearchParams, WebSearchProviderToolDefinition, provider SDK helpers, and the Gemini implementation; then classify and log unexpected provider aborts while preserving real user/session cancellation as cancellation.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Unclear for the live intermittent Gemini race, but yes for the supporting failure class: a provider that throws an AbortError while the tool signal is not aborted will bypass normal tool failure logging on current main.

## Solution Assessment

Is this the best way to solve the issue? Unclear as a final root-cause fix until the live race is pinned down; the narrow maintainable direction is an additive provider execution-context seam plus cancellation/logging tests, not a Gemini-only timeout tweak.
