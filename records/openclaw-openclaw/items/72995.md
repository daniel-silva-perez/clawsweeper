## Summary

Keep open. Current main still matches the reported failure class: managed `web_search` does not carry the agent tool AbortSignal into the web-search runtime/provider contract, Gemini search only receives the configured timeout, and AbortError-shaped tool failures are rethrown before the normal tool error log path.

## What This Changes

The issue reports intermittent near-instant Gemini-backed `web_search` aborts that return `This operation was aborted` without a matching `gateway.err.log` entry.

## Best Possible Solution

Add a backwards-compatible web-search execution context that carries AbortSignal and abort diagnostics through the managed tool wrapper, runtime, provider SDK type, provider helpers, and Gemini implementation; log unexpected provider AbortError failures safely while still treating true user/session cancellation as cancellation.

## Reproduction Assessment

Unclear for the intermittent live Gemini abort itself: the reporter provided concrete steps, but it is timing-dependent and no Gemini key was available for live proof during this read-only review. The supporting failure mode is source-reproducible: the managed path drops the signal and the adapter rethrows AbortError before normal tool logging.

## Solution Assessment

Unclear as a final fix because the suspected `runAbortController` race is plausible but not yet proven as the only trigger. The maintainable direction is a narrow, additive execution-context seam plus AbortError classification/logging tests rather than a Gemini-only workaround.
