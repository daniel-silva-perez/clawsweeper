## Summary

Keep open. Current main still matches the reported failure class: the managed `web_search` path does not pass the agent tool `AbortSignal` into the web-search runtime or provider contract, Gemini receives only the configured timeout, and `AbortError`-shaped failures are rethrown before the normal tool error logging path.

## What This Changes

The issue reports intermittent near-instant Gemini `web_search` aborts with no corresponding `gateway.err.log` entry and asks for the tool/provider abort path to be diagnosed and fixed.

## Best Possible Solution

Add a backwards-compatible web-search execution context that carries `AbortSignal` and abort diagnostics through the managed tool wrapper, runtime, provider SDK type, provider helpers, and Gemini implementation; log unexpected provider `AbortError` failures safely while preserving true user/session cancellation as cancellation.

## Reproduction Assessment

Unclear for the intermittent live Gemini abort itself because it is timing-dependent and no Gemini key was available for live proof. The supporting failure mode is source-reproducible: the managed path drops the signal, the provider contract has no signal parameter, and the adapter rethrows `AbortError` before normal tool logging.

## Solution Assessment

Unclear as a final fix because the suspected session-level `runAbortController` race is plausible but not proven as the only trigger. The maintainable direction is a narrow additive execution-context seam plus AbortError classification/logging tests rather than a Gemini-only workaround.
