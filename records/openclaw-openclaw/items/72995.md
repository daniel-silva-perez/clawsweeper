## Summary

Keep open. Current main still has the reported failure class: the managed `web_search` path does not propagate the agent tool `AbortSignal` into the web-search runtime or Gemini provider, and `AbortError`-shaped provider failures still bypass the normal `[tools] ... failed` logging path.

## What This Changes

The issue requests fixing intermittent instant Gemini `web_search` aborts and ensuring abort-shaped failures are logged instead of disappearing from `gateway.err.log`.

## Best Possible Solution

Add a backwards-compatible web-search execution context carrying `AbortSignal` and abort diagnostics through the managed tool wrapper, runtime types, provider SDK helpers, and Gemini implementation; then classify/log unexpected provider aborts while preserving true user/session cancellation as cancellation.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Unclear for the live intermittent Gemini abort because it is timing-dependent and no Gemini key was available, but the supporting failure path is source-reproducible by having a provider raise an `AbortError` through the current adapter.

## Solution Assessment

Is this the best way to solve the issue? Unclear as a final root-cause fix because the suspected session-level abort race is plausible but not proven; the narrow maintainable direction is an additive execution-context seam plus cancellation/logging tests, not a Gemini-only timeout tweak.
