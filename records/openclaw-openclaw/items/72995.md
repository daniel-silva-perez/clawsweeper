## Summary

Keep open. Current main still matches the reported failure class: the managed `web_search` wrapper does not forward the agent tool signal into the web-search runtime/provider contract, the Gemini provider only receives configured timeout data, and `AbortError` is still rethrown before the normal tool error log path.

## What This Changes

The issue asks OpenClaw to fix intermittent near-instant Gemini-backed `web_search` aborts that return `This operation was aborted` without a corresponding gateway error log entry.

## Best Possible Solution

Add a backward-compatible web-search execution context that carries `AbortSignal` and safe abort diagnostics through the managed tool wrapper, runtime, provider SDK type, provider helpers, and Gemini implementation, while preserving true user/session cancellation behavior.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Unclear for the intermittent live Gemini abort itself because the report is timing-dependent and this read-only pass did not run a live Gemini request. The supporting source path is high confidence: current main drops the managed tool signal and rethrows `AbortError` before normal tool logging.

## Solution Assessment

Is this the best way to solve the issue? Not applicable as a proposed patch because this is an issue, not a PR. The maintainable direction is an additive execution-context seam with focused abort/logging tests, not a Gemini-only workaround.
