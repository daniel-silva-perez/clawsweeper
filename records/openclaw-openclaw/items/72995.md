## Summary

Keep open. Current main still matches the reported failure class: managed `web_search` does not carry the agent tool cancellation signal into the web-search runtime or provider contract, Gemini only receives the configured timeout, and `AbortError`-shaped failures still bypass the normal tool error logging path.

## What This Changes

The issue reports intermittent near-instant Gemini `web_search` aborts with no `gateway.err.log` entry and asks for the Gemini/tool abort path to be diagnosed and fixed.

## Best Possible Solution

Add a backwards-compatible web-search execution context that carries `AbortSignal` and abort diagnostics through the managed tool wrapper, runtime, provider SDK type, provider helpers, and Gemini implementation; log unexpected provider `AbortError` failures safely while preserving true user/session cancellation as cancellation.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Unclear for the intermittent live Gemini abort itself because it is timing-dependent and no Gemini key was available, but the supporting failure mode is source-reproducible in the current managed web-search path.

## Solution Assessment

Is this the best way to solve the issue? Unclear as a final fix because the suspected session-level `runAbortController` race is plausible but not proven as the only trigger; the maintainable direction is a narrow additive execution-context seam plus AbortError classification/logging tests.
