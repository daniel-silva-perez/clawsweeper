## Summary

Current main still lacks a normal allowlist satisfaction path for POSIX shell builtins such as cd/export/pwd in shell-chain segments. The related closed #40478 fixed durable exact-command allow-always reuse after a prompt, but it does not solve preconfigured allowlist mode for fresh builtin-containing chains. A narrow docs clarification remains actionable; runtime builtin auto-allow needs separate maintainer design.

## Best Possible Solution

Keep this issue open until a docs-only clarification lands in the exec approvals allowlist docs explaining that path and bare-name allowlist entries match filesystem/PATH-resolved executables, not POSIX shell builtins, with workarounds such as git -C <path> status. If maintainers want runtime behavior changes, design an opt-in builtin policy separately with explicit cwd/env mutation semantics and tests.

## Repair Work Prompt

Implement a docs-only clarification for the shell-builtin exec-allowlist issue. Update the exec approvals docs to state that normal path and bare command-name allowlist entries match filesystem/PATH-resolved executables, not POSIX shell builtins such as cd, export, unset, pwd, or shell-builtin echo. Explain that each top-level shell-chain segment must satisfy allowlist evaluation, so cd <path> && git status can still prompt even when git and bash are allowlisted. Include practical workarounds such as git -C <path> status or commands that avoid cwd/env-mutating builtins. Mention related #40478 only as the separate exact-command allow-always reuse case if useful. Do not change runtime policy, do not add safeBuiltins, do not auto-allow builtins, and do not change tests except docs validation. No changelog entry is needed for a docs-only clarification.
