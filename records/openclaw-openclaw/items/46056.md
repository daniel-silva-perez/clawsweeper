## Summary

The reported allowlist gap still appears current on main. The allowlist evaluator still requires each parsed shell segment to match an executable path, safe bin, skill allowance, or special skill prelude; unresolved shell builtins like `cd` do not get a builtin-specific allowance. Current docs explain that every top-level segment must satisfy allowlist rules, but they do not document the shell-builtin edge case or provide the requested workaround clearly enough to close this issue.

## Best Possible Solution

Keep this issue open. The conservative path is to add explicit docs under exec approvals explaining that shell builtins are not path-resolvable and to add regression tests for `cd` alone and `cd /path && git status` with `/usr/bin/git` and `/usr/bin/bash` allowlisted. If maintainers want behavior changes, implement a deliberately narrow, documented builtin policy in `src/infra/exec-approvals-allowlist.ts` rather than treating the shell binary allowlist as trust for all builtins.
