## Summary

Keep open. Current main still has the reported allowlist gap for pathless POSIX shell builtins such as `cd`, and the user-facing exec approvals docs still do not call out that builtin segments cannot satisfy path or bare-name allowlist entries.

## What This Changes

The issue asks OpenClaw to document or handle shell builtins such as `cd` in exec allowlist mode so users do not get unexpected approval prompts when surrounding binaries are allowlisted.

## Best Possible Solution

Add a narrow documentation clarification to the exec approvals allowlist section explaining that pathless POSIX shell builtins cannot satisfy ordinary path or bare-name allowlist entries, with a safer workaround such as using tool-native directory flags. A runtime change should be a separate opt-in policy design with focused tests for cwd and environment mutation semantics.

## Reproduction Assessment

Yes. The high-confidence reproduction is source-level on current main: `cd <dir> && git status` is split into shell-chain parts, the `cd` segment has no resolved executable path, `matchAllowlist` returns `null`, and the chain remains an allowlist miss.

## Solution Assessment

Yes for a docs-only first fix. It is the narrowest maintainable response to the documented current behavior; auto-allowing builtins by default is not the best runtime fix without a separate security design.
