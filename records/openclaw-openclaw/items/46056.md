## Summary

Keep open. Current main still evaluates shell-chain segments through path-backed allowlist matches, safeBins, skill auto-allow, or durable exact-command reuse; POSIX shell builtins still have no normal allowlist satisfaction path, and the docs do not explicitly call out that pitfall. The related closed #40478 work is useful context, but it only covers exact-command allow-always reuse after approval, not fresh preconfigured allowlist mode.

## What This Changes

The issue asks OpenClaw to document and ideally handle POSIX shell builtins such as `cd`, `export`, and `pwd` in exec approval allowlist mode.

## Best Possible Solution

Add a narrow docs clarification to the exec approvals allowlist section now. If maintainers later want runtime behavior, design an opt-in builtin policy with explicit cwd/env mutation semantics and regression tests for single builtins and chained commands.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: `cd /path && git status` is split into shell-chain parts, the `cd` segment has no resolved executable path, `matchAllowlist` returns `null`, and the chain returns `allowlistSatisfied: false`.

## Solution Assessment

Yes for a docs-only first step; it is the narrowest maintainable fix that matches the current implementation and avoids changing security semantics. Auto-allowing builtins is not yet the best default runtime fix without a separate opt-in policy design.

## Repair Work Prompt

Treat https://github.com/openclaw/openclaw/issues/46056 as a narrow docs-only fix. Update `docs/tools/exec-approvals.md` in the allowlist section to explain that normal path and bare command-name allowlist entries match filesystem/PATH executables, not POSIX shell builtins such as `cd`, `export`, `unset`, `pwd`, or shell-provided `echo`. Make clear that shell chains still require every top-level segment to satisfy allowlist rules, so `cd /path && git status` can prompt even when `/usr/bin/git` and the shell binary are allowlisted. Recommend command forms that avoid builtin chain segments where practical, such as `git -C <path> status`. Do not change runtime allowlist behavior, do not introduce `safeBuiltins`, and do not broaden exec approval policy. No changelog entry is needed for this docs-only clarification unless maintainers ask. Run `pnpm docs:list`, `pnpm check:docs`, and `git diff --check`.
