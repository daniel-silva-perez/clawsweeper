## Summary

Keep open. Current main still evaluates pathless POSIX shell builtins as ordinary command segments with no allowlist/safe-bin/skill satisfaction path, and the public exec approvals docs still do not explain that configured path or bare-name allowlist entries cannot satisfy those builtin segments.

## What This Changes

Clarify or change exec approval allowlist behavior for POSIX shell builtins such as cd, export, pwd, and echo when used inside shell command chains.

## Best Possible Solution

Add a focused documentation clarification to the exec approvals allowlist section explaining that pathless POSIX shell builtins cannot satisfy ordinary allowlist entries, with safer workarounds such as tool-native directory flags. Treat runtime builtin auto-allow as a separate opt-in policy design with security review and regression tests.

## Reproduction Assessment

Yes. The high-confidence reproduction is source-level on current main: cd is parsed as a shell-chain segment, its command resolution has no executable path, matchAllowlist returns null, and the chain evaluation remains an allowlist miss.

## Solution Assessment

Yes for a docs-only first fix. It is the narrow maintainable response to current behavior; auto-allowing builtins by default is not the best runtime change without explicit opt-in policy and tests for cwd and environment mutation semantics.

## Repair Work Prompt

Fix the documentation gap for openclaw/openclaw#46056 without changing runtime exec approval behavior. Update docs/tools/exec-approvals.md in the allowlist section to explain that POSIX shell builtins such as cd, export, pwd, true/false, and echo when executed as shell builtins do not resolve to filesystem executable paths, so ordinary path or bare-name allowlist entries will not satisfy those shell-chain segments. Mention that each top-level chain segment must satisfy allowlist rules and give a safer workaround such as using tool-native directory flags like git -C <path> instead of cd <path> && git status. Keep the wording security-conscious and avoid implying that adding bash/sh/zsh to the allowlist grants trust to builtins. Do not add safeBuiltins, change safeBins, alter src/infra runtime behavior, or add approval bypasses. Changelog is not expected for this docs-only clarification. Validate with pnpm docs:list, pnpm check:docs, and git diff --check.
