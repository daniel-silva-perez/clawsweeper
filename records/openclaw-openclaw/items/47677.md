## Summary

Keep open. Current main still implements Telegram reactions as notification/context events only: standard emoji reactions are enqueued into the session, but they do not wake an agent turn, do not normalize Telegram custom emoji, and do not expose per-emoji action semantics. The issue is also paired with open PR #47687, which uses closing syntax for this work, and open #64752 tracks the narrower wake-trigger bug, so this should remain open until that implementation path is resolved.

## What This Changes

Request first-class Telegram reaction inputs that can wake the agent, normalize standard and custom emoji, support configurable operator semantics, and provide observability/debugging without downstream patching.

## Best Possible Solution

Review #47687 against #64752 and either land it or replace it with a narrower maintained Telegram-plugin implementation that adds a typed reaction semantics/wake contract, standard and custom emoji normalization, targeted observability, tests, and docs while preserving existing Telegram access controls.
