## Summary

Current main still does not implement this PR's intended behavior, and this PR is not safe to close as obsolete. The exact Anthropic `specified API usage limits` / `regain access` wording is still not covered by the billing matcher and still hits the broad `usage limit` rate-limit path. The duplicate-reply half also remains same-target gated on main, while the PR's unconditional text dedupe has a credible cross-target false-positive risk already called out in review.

## Best Possible Solution

Keep this PR open, but rework or split it. The failover half should add a narrow Anthropic billing matcher/test for the exact `specified API usage limits` plus `regain access` wording without shadowing periodic usage-window limits. The reply-dedupe half should solve multi-handle same-person delivery through identity-aware target matching or a channel-owned alias/suppression hook for BlueBubbles/iMessage, with positive tests for same-person email/phone aliases and negative tests for same-text cross-target sends.
