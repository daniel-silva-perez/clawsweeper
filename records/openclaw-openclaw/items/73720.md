## Summary

Keep this PR open. Current main still has the shared resolver and gateway persistence gaps the PR targets: caller-provided group metadata can reach `resolveGroupToolPolicy()` before server-derived session/spawn context, and the gateway session entry still writes raw or stale group metadata without the PR's trust validation. The overlapping #72921 is still open/unmerged, so it is not a landed superseding fix.

## Best Possible Solution

Keep this PR open and review it as a candidate canonical fix, or reconcile it with #72921 before landing one implementation. The landed fix should validate caller `groupId` inside `resolveGroupToolPolicy()` for every caller, prefer session/spawn-derived group IDs over caller candidates, clear untrusted group metadata before gateway persistence/runtime forwarding, preserve legitimate first-turn group selectors, and keep the resolver, coding-tools, allowlist, and gateway regression tests.
