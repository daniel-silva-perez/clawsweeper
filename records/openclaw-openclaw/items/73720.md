## Summary

Keep this PR open. Current main still has the shared resolver gap the PR targets: `resolveGroupToolPolicy()` trusts caller-provided `groupId` before session/spawn-derived group IDs, while the existing trust check is only local to the final bundled-tool filtering pass. The gateway session entry write also still preserves an existing or resolved `groupId` without validating it against the canonical session key. The PR is security-sensitive but focused, and the provided diff does not show unrelated supply-chain or CI changes.

## Best Possible Solution

Review this PR against the overlapping #72921 and land one canonical security fix that validates caller `groupId` inside `resolveGroupToolPolicy()` for all callers, preserves legitimate session/spawn-derived group behavior, and clears untrusted group fields before gateway session-entry persistence. Keep the focused resolver, coding-tools, allowlist, and gateway regression tests from this PR or equivalent coverage.
