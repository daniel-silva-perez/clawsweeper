## Summary

Keep this PR open. Current main still trusts caller-provided `groupId` in the shared group tool policy resolver and still persists `groupId` session-entry fields without validating them against server-derived session/spawn context. The PR is an active, focused security hardening change; it overlaps #72921, but this PR also covers the gateway persistence path, so it should be reconciled rather than closed.

## Best Possible Solution

Review this PR together with #72921 and land one canonical security fix. The merged path should validate caller `groupId` inside `resolveGroupToolPolicy()` for every caller, prefer server-derived session/spawn group context, clear untrusted `groupChannel`/`groupSpace`, validate or clear group fields before gateway session-entry persistence, and keep focused regression coverage for the resolver, coding-tools/allowlist paths, and session-entry write behavior.
