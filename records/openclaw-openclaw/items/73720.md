## Summary

Keep this PR open. Current main still has the authorization gap the PR targets: the shared group tool-policy resolver accepts caller-provided groupId before session/spawn-derived group IDs, the core tool and explicit-allowlist paths pass that raw value into the resolver, and the gateway session entry still persists group metadata without the trust validation described in the PR. The PR is security-sensitive and overlaps #72921, so maintainers should choose or reconcile one canonical fix rather than close this implementation as obsolete.

## Best Possible Solution

Review this PR against #72921 and land one canonical security fix that validates caller groupId inside resolveGroupToolPolicy for all callers, preserves legitimate session/spawn-derived group behavior and first-turn group selectors, clears untrusted session-entry group metadata, and keeps the focused resolver, coding-tools, allowlist, and gateway regression coverage.
