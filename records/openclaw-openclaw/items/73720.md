## Summary

Keep this PR open. Current main still has the shared resolver gap the PR targets: `resolveGroupToolPolicy()` prepends caller-provided `groupId` before server-derived session/spawn context, while the existing trust check remains scoped to the final bundled-tool policy pass. The gateway session-entry path also still preserves raw or stale group metadata without validating it against the canonical session context. #72921 overlaps, but it is still open and unmerged, so there is no landed canonical fix to close this PR against.

## Best Possible Solution

Review and land one canonical security fix, either this PR or a reconciled version with #72921. The fix should validate caller `groupId` inside `resolveGroupToolPolicy()` for all callers, keep session/spawn-derived group IDs ahead of caller candidates, clear untrusted group selectors before gateway session persistence and runtime forwarding, and retain the resolver, coding-tools, allowlist, and gateway regression tests described in this PR.
