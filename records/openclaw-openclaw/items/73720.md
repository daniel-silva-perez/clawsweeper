## Summary

Keep open. Current main at 955b4df0939f90b4b6de5e7e6f3149dd125e1e50 still has the shared resolver gap this PR targets: `resolveGroupToolPolicy()` trusts caller-provided `groupId` before session/spawn-derived context, the existing trust check is private to the bundled-tool final policy pass, and the gateway session-entry write still preserves raw or stale group metadata. This PR is open, unmerged, and still useful; #72921 overlaps but is also open and unmerged, so maintainers should choose or reconcile one canonical security fix rather than close this PR.

## Best Possible Solution

Keep this PR open and review it as a candidate canonical fix, or reconcile it with #72921. The landed solution should validate caller `groupId` inside `resolveGroupToolPolicy()` for every caller, keep session/spawn-derived group IDs ahead of caller candidates, prefer session/spawn-derived channel context, clear untrusted group metadata before gateway session persistence and runtime forwarding, preserve first-turn trusted group selectors, and keep the resolver, core-tool, allowlist, and gateway regression tests.
