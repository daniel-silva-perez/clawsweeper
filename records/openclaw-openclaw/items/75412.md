## Summary

Keep open. Current main still matches the reported source-level root cause: active-memory invokes the embedded runner with a memory-only allowlist, but the embedded attempt constructs the full OpenClaw tool registry before filtering and still builds the system prompt, resource loader, and agent session. The report is concrete and actionable, but the fix crosses tool policy, plugin boundaries, prompt composition, and possible session/provider caching, so it needs maintainer design before an automated repair lane attempts it.

## What This Changes

Request a new opt-in embedded-run fast path so active-memory and similar memory-only runs avoid full tool-registry construction and other fixed setup costs before streaming.

## Best Possible Solution

Add a generic embedded-run lightweight construction path that treats explicit `toolsAllow` as a construction-time constraint where safe, including plugin-owned memory tools, while preserving the existing policy/sandbox/auth checks. Keep prompt slimming measurable and conservative, and treat resource-loader or provider-session reuse as a separate benchmarked design after the first narrow fast path lands.

## Reproduction Assessment

Yes for the source-level reproduction: active-memory invokes the embedded runner with a memory-only allowlist, and current main still constructs the full tool registry before filtering it. The exact latency numbers need a live benchmark on a representative workspace with `[trace:embedded-run] prep stages` enabled.

## Solution Assessment

Unclear. A fast-path concept is directionally right, but a broad `fastPath: true` switch that also changes prompts and caches provider/session state is riskier than first adding a narrower construction-time tool allowlist or lightweight embedded profile with benchmark coverage.
