## Summary

Keep open. Current main still performs a single Feishu message resource download with the supplied `image|file` type, and inbound non-image media still routes through `type: "file"`; there is no 5xx retry, `type=media` fallback, or video-file heuristic on main. The linked Feishu bug #49855 and alternative PR #50164 are still open, so this PR remains an implementation candidate rather than something already implemented or clearly superseded. Security review found the PR diff limited to Feishu code/tests, with no CI, dependency, release, publishing, or secret-handling changes.

## Best Possible Solution

Keep this PR open for maintainer review or consolidate it with #50164 and #49855. The eventual Feishu fix should prove the correct download strategy for iOS file-type videos, cover real 5xx/Axios error shapes, wire video metadata through every production media path that needs it or avoid optional gating, and add targeted regression tests around the chosen retry/fallback behavior.
