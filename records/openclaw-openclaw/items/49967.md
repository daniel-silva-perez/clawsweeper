## Summary

Keep this PR open. Current main still performs a single Feishu message-resource download with the supplied `image|file` type, and inbound Feishu video/media paths still route through `type: "file"`; there is no merged retry, `type=media` fallback, or video-file heuristic. The linked bug #49855 and alternative PR #50164 are still open, so this remains a useful implementation candidate rather than an implemented or clearly superseded change.

## Best Possible Solution

Keep this PR open for maintainer review or consolidate it with #50164 and #49855. The eventual Feishu fix should settle whether the correct recovery is a supported `type=file` retry/backoff, a documented `type=media` fallback, or another Feishu API path; it should cover real 502/Axios error shapes, wire any video metadata through all affected production paths, and add targeted regression tests for the chosen behavior.
