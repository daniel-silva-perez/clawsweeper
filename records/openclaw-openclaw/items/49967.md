## Summary

Keep this PR open. Current main still performs a single Feishu message-resource download using the supplied `image|file` type, and inbound Feishu video/media paths still resolve to `type: "file"`; there is no merged media-endpoint fallback, retry/backoff, or video-file heuristic. The linked bug and alternative PR remain open, so this PR is still an implementation candidate rather than implemented or clearly superseded.

## Best Possible Solution

Keep this PR open for maintainer review or consolidate it with #50164 and #49855. The eventual Feishu fix should settle whether the correct recovery is a supported same-`type=file` retry/backoff, a documented `type=media` fallback, or another Feishu API path; it should handle real 502/Axios error shapes, cover post and non-post video metadata paths, and add targeted regression tests for the chosen behavior.
