## Summary

Keep this PR open for maintainer review. Current `main` still rejects pending Control UI RPCs with a generic close error after a structured connect-handshake failure, while the PR is a narrow two-file fix with a focused regression test and no blocking review findings found.

## What This Changes

The PR changes `GatewayBrowserClient` to reject pending Control UI RPCs with a structured `GatewayRequestError` when a failed connect handshake causes the socket close, and adds a unit test for preserving `details.code`.

## Best Possible Solution

Land this narrow UI gateway fix, or recreate the same behavior on current `main`: use the structured connect failure to reject already-pending Control UI RPCs when that failure caused the close, preserve generic close errors for ordinary disconnects, and keep the regression test.

## Reproduction Assessment

Yes. The path is reproducible with the mocked WebSocket unit setup: start `GatewayBrowserClient`, leave an RPC pending, return a failed connect response with `details.code`, emit the close, and observe current `main` flushing the pending request with a generic close error.

## Solution Assessment

Yes. The PR is the narrowest maintainable fix because `pendingConnectError` is already produced by the failed connect path and consumed by close-state handling; reusing it for `flushPending` preserves ordinary close behavior while exposing structured auth/pairing details to pending callers.
