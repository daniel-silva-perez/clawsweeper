## Summary

Keep this issue open. Current main still exposes only the source-channel timestamp through the shared inbound context, prompt metadata, channel turn builder, direct-DM helper, and hook contracts; Telegram has an internal `receivedAtMs`, but it is used for buffering/debug timing and is not exposed as shared agent-visible inbound metadata.

## What This Changes

Request to add a Gateway receipt timestamp such as `receivedAt` or `gatewayReceivedAt` to inbound metadata so agents can separate channel delivery latency from processing latency.

## Best Possible Solution

Add a backward-compatible receipt timestamp stamped once at channel/Gateway ingress, preserve `Timestamp`/`timestamp` as the source-channel message time, expose the new value through agent-visible inbound metadata plus typed hook/SDK payloads, and document buffered-path semantics.

## Repair Work Prompt

Implement the Gateway receipt timestamp requested in https://github.com/openclaw/openclaw/issues/53678. Keep `Timestamp` / agent-visible `timestamp` as the source-channel message time. Add a backward-compatible receipt-time field through the shared inbound context, channel turn builder, direct-DM helper, canonical hook mapping, plugin hook event types, and agent-visible inbound metadata. Use an internal numeric millisecond value such as `ReceivedAt?: number`; expose an agent metadata field with a stable documented name such as `received_at`/`receivedAt` consistent with local metadata style, and expose SDK/hook payloads as an additive optional `receivedAt?: number`. Stamp the value once at channel/Gateway ingress when not supplied. For buffered, debounced, or coalesced turns, use the first received message time for the assembled inbound turn unless an existing channel contract clearly requires a different rule; document and test that choice.
