## Summary

Keep this issue open. Current main still has only the source-channel `Timestamp` in shared inbound context, agent-visible conversation metadata, SDK inbound helpers, and plugin hook payloads; the existing `receivedAt`/`receivedAtMs` matches are local plugin timing/debug fields and do not satisfy a cross-channel Gateway receipt timestamp contract.

## Best Possible Solution

Keep this issue open. Add an additive shared receipt timestamp, such as `ReceivedAtMs` or `GatewayReceivedAtMs`, to `MsgContext`/`FinalizedMsgContext`, SDK inbound envelope/direct-DM helpers, and inbound hook payloads; stamp it once at channel ingress or finalization when absent while allowing buffered plugins to preserve the first receipt time; render a stable ISO `receivedAt` or `gatewayReceivedAt` in agent-visible inbound metadata; and cover the metadata builder, hook mapper, SDK helper, and one buffered or long-poll-style channel path with focused tests.

## Risks / Open Questions

- The fix would touch public plugin SDK and hook contracts, so it should be additive and preserve existing `Timestamp` semantics as the source-channel timestamp.
- Buffered, debounced, and long-poll channel plugins need a way to preserve the first Gateway receipt time rather than having core stamp a later dispatch/finalization time.
- A per-message receipt timestamp is volatile; it should be rendered in dynamic user-role metadata or hook payloads, not stable system prompt metadata that affects prompt caching.
