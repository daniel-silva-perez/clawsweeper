## Summary

Keep this issue open. Current main still carries only the source-channel timestamp through shared inbound context, agent-visible conversation metadata, SDK direct-DM ingress helpers, and message hook payloads; the `receivedAtMs` matches I found are Telegram-local buffering/debug timings and are not exposed as the requested cross-channel Gateway receipt timestamp contract.

## What This Changes

The issue requests adding a Gateway receipt timestamp to inbound message metadata so agents can distinguish channel delivery delay from OpenClaw processing latency.

## Best Possible Solution

Add a backward-compatible shared receipt timestamp field stamped at inbound ingress, preserve source-channel `Timestamp`, expose the new value to agent-visible metadata plus plugin/internal hook payloads and SDK helpers, and cover at least one buffered channel path so debounced messages retain the first receipt time.

## Risks / Open Questions

- Existing `Timestamp` consumers expect source-channel time, so a fix must add a separate field rather than repurpose `Timestamp`.
- Buffered, debounced, and long-poll paths need to preserve the earliest Gateway receipt time; stamping only at final dispatch would make the new metric less accurate.
- Receipt time is per-turn volatile metadata and should stay out of stable system prompt metadata to avoid prompt-cache churn.
