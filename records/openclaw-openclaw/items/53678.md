## Summary

Keep this issue open. Current main still exposes only the source-channel `Timestamp` through shared inbound context, agent-visible metadata, SDK inbound helpers, and plugin/internal hook payloads; existing `receivedAt` matches are local plugin timing fields, not the requested cross-channel Gateway receipt timestamp contract.

## What This Changes

The issue requests an additive Gateway receipt timestamp in inbound metadata so agents can distinguish channel delivery delay from OpenClaw processing latency.

## Best Possible Solution

Add a backward-compatible shared receipt timestamp field, stamp it once when an inbound message enters OpenClaw unless a channel provides an earlier preserved ingress time, surface it to agents as an ISO `receivedAt` or `gatewayReceivedAt`, and expose it through SDK and hook payloads with focused coverage across metadata, SDK, hook, and one buffered channel path.

## Repair Work Prompt

Implement the Gateway receipt timestamp requested by this issue. Add an additive shared field such as `ReceivedAtMs` or `GatewayReceivedAtMs` to inbound context types without changing existing `Timestamp` semantics, which must remain the source-channel timestamp. Stamp a receipt time once when absent, preferably at channel ingress/finalization, while allowing buffered channels to pass through the first preserved ingress time. Surface a stable ISO `receivedAt` or `gatewayReceivedAt` in agent-visible inbound user metadata, not in stable system prompt metadata. Propagate the numeric receipt timestamp through SDK inbound helpers/direct-DM dispatch and plugin/internal message hook payloads. Include focused tests for `buildInboundUserContextPrefix`, `finalizeInboundContext` or the chosen stamping seam, `dispatchInboundDirectDmWithRuntime` / inbound envelope helpers, `message-hook-mappers`, and one buffered or long-poll-style path such as Telegram's existing `receivedAtMs` flow. Update docs/changelog only if the new field is user/plugin-visible; changelog entries need normal OpenClaw attribution. Do not add channel-specific WeChat code to core and do not rename or repurpose `Timestamp`.
