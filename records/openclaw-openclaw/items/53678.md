## Summary

Keep this issue open. Current main still does not expose a shared Gateway receipt timestamp for inbound turns; the agent-visible metadata, shared context, direct-DM helper, generic turn builder, and hook contracts still carry only the source-channel timestamp, with Telegram's local `receivedAtMs` remaining internal to buffering/debug paths.

## What This Changes

The issue requests an additive inbound metadata field such as `receivedAt` or `gatewayReceivedAt` so agents can distinguish Gateway receipt time from source-channel message time for latency tracking.

## Best Possible Solution

Add a backward-compatible shared receipt timestamp stamped once at inbound Gateway/channel ingress, preserve `Timestamp` as the source-channel message time, expose the new value through agent-visible inbound metadata and typed hook/SDK payloads, and document the semantics for buffered paths.
