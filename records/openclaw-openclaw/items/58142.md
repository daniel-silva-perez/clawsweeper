Keep this PR open. Current main still does not implement per-skill SKILL.md model routing, and the linked feature issue #43260 remains open. The PR is an active implementation candidate, but it still needs maintainer design/security review around field names, precedence, fallback, and provider/cost/privacy behavior.

Keep this PR open for maintainer review, or supersede it with a narrower implementation that defines the SKILL.md model-routing contract, preserves parsed metadata through cached snapshots, applies routing at a clear skill invocation/read boundary, documents provider/privacy/cost and user override precedence, and adds parser, cached-snapshot, alias/path, fallback, and plugin-hook interaction tests.

## Risks / Open Questions

- Security review: the proposed feature lets third-party or workspace SKILL.md metadata influence the provider/model used for later prompt content. Even if it only uses configured providers, maintainers should define precedence, visibility, consent, cost controls, and privacy expectations before merge.
- Functional review: provided review comments still mention edge cases around read-tool aliases such as file_path and home-relative paths; those should be verified or fixed before landing.
- Scope review: VISION.md prefers optional behavior as plugins when existing APIs suffice, but the current before_model_resolve hook is not selected-skill-aware, so either this PR or a narrower skill-metadata/plugin hook design still needs maintainer product judgment.
