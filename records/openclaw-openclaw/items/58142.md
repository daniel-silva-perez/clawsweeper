## Summary

Keep this PR open. Current main still does not implement SKILL.md `model`/`modelProfile` metadata or a skill-aware runtime model switch, and the related feature issue #43260 remains open. This PR is an active implementation candidate, not conservative cleanup.

## Best Possible Solution

Keep this PR open for maintainer review, or supersede it with a narrower implementation that defines the SKILL.md model-routing contract, preserves parsed skill metadata through cached snapshots, applies routing at a well-defined skill invocation/read boundary, documents provider/privacy/cost and precedence behavior, and adds focused parser, cached-snapshot, alias/path, fallback, and plugin-hook interaction tests. If maintainers prefer plugin-driven routing, add a skill-selection or skill-metadata hook first; the current run-level `before_model_resolve` hook alone does not implement this frontmatter feature.

## Risks / Open Questions

- This PR defines a public skill metadata/runtime contract, so maintainers still need to settle field names, precedence against explicit user/session model choices, fallback behavior, docs, and migration expectations before merge.
- Security and cost review remains important: third-party SKILL.md metadata could silently route subsequent prompt content to a different configured provider or a more expensive model using existing credentials.
- Provided review context includes later unresolved implementation concerns around `file_path` read aliases and `~`-relative skill paths; those should be verified or fixed before merge.
- The proposed implementation appears to use ModelRegistry methods through casts in tests/implementation context, so dependency API stability should be checked before landing.
