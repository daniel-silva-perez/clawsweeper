## Summary

Keep open: current main still has explicit rejection paths for oversized image attachments, while only some adjacent surfaces already compress before upload. The remaining work is real, but it spans upload clients and the Gateway hard-cap boundary, so it needs maintainer direction before an automated repair lane should start.

## What This Changes

The issue requests automatic resize or compression for image attachments that exceed the current size limit before they are sent for model processing.

## Best Possible Solution

Keep the issue open and implement a bounded, consistent image-compression path for user-facing upload clients, with any Gateway fallback limited by an explicit larger pre-compression ceiling and preserving the existing hard post-compression image cap.

## Reproduction Assessment

Yes. A source-level reproduction is clear: send an image attachment whose decoded base64 size exceeds MAX_IMAGE_BYTES through parseMessageWithAttachments, or add more than 5,000,000 bytes through the Swift chat view model; current main throws the size-limit errors rather than compressing.

## Solution Assessment

Unclear. The requested behavior is a good UX target, but the safest implementation boundary needs maintainer choice: client-side compression is lower risk for selected files, while gateway-side compression needs careful memory/security caps.
