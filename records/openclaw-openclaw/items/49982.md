## Summary

Keep this issue open. Current main has image optimization in later media-loading paths, but the Control UI attachment upload path still sends the original base64 image to `chat.send`, and the Gateway still rejects oversized image attachments before offload/compression. The related closed image-attachment reports cover forwarding, persistence, and Discord fallback behavior, not automatic compression of oversized direct chat uploads.

## Best Possible Solution

Implement automatic image compression for direct chat attachments before the current Gateway size rejection, likely by reusing the existing media/image optimization helpers either in the Gateway attachment parser or in a Control UI preflight path. The implementation should preserve hard caps, avoid silent drops, update Control UI/Gateway tests for an oversized image that compresses below the limit, and keep explicit errors for images that cannot be reduced safely.
