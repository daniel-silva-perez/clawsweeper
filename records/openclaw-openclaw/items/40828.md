## Summary

Keep this PR open. Current main still does not implement the requested `/total_usage` chat/native command, Telegram quick-select UI, flexible hour/day/all range parsing, or cross-session model breakdown in the chat command path. The PR is an active refresh of the older closed/unmerged #11901, but it needs maintainer review and likely changes before merge because the provided review context flags performance, provider/model aggregation, and authorization-surface concerns.

## Best Possible Solution

Keep this PR open and ask for an update against current main if maintainers still want `/total_usage` as a first-class chat/native command. The best implementation path is to reuse or align with existing `sessions.usage` provider-plus-model aggregates, preserve transcript prefiltering for bounded ranges, keep authorization checks on both text and Telegram callback paths, and add focused command/infra tests plus public slash-command and token-use docs.

## Risks / Open Questions

- The PR should not be merged as-is without addressing reviewer concerns that its cost-summary file scan can regress narrow-range performance by dropping the current `mtimeMs < sinceTime` prefilter.
- The PR's proposed aggregate model map, as shown in the provided patch, keys by model name only and can mix same-named models from different providers; current `sessions.usage` uses provider-plus-model keys and is the safer shape to align with.
- The feature exposes broader cross-session usage and cost data through chat/Telegram controls, so maintainers should explicitly confirm the authorized-sender and callback handling policy before merge.
- No supply-chain or CI-execution regression was evident from the provided PR file list, but the runtime data-exposure surface still needs review.
