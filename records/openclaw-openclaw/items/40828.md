## Summary

Keep open. Current main still lacks the requested `/total_usage` command, and the refreshed PR is useful source work, but the branch has concrete merge-blocking issues in the usage aggregation contracts and typed fixtures.

## What This Changes

The PR adds a `/total_usage` chat command with range parsing, Telegram quick-select buttons, per-model aggregate output, usage summary type changes, focused tests, and slash-command documentation.

## Best Possible Solution

Land a corrected `/total_usage` implementation that reuses the existing usage range and provider/model aggregation contracts, keeps docs aligned with the generated command surface, and adds focused tests for all-time, bounded ranges, provider-colliding model IDs, and typed summary fixtures.

## Reproduction Assessment

Yes. The PR diff itself reproduces the blockers: the model aggregate key is model-only, both bounded discovery/scan filters are removed, and the required summary type change leaves an existing typed fixture incomplete.

## Solution Assessment

No. The feature direction fits the existing usage surfaces, but this patch is not the narrowest maintainable solution until it preserves bounded filtering and provider-aware aggregation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Partition model totals by provider and model:** `src/infra/session-cost-usage.ts:423-431`
  - body: The new aggregate key is only `entry.model`, while each bucket stores one provider from the first matching entry. Same-named models from different providers will merge tokens and costs and be labeled with whichever provider was seen first; use the provider-plus-model key shape already used by session usage.
  - confidence: 0.9
- **[P2] Restore bounded transcript prefiltering:** `src/infra/session-cost-usage.ts:399-402`
  - body: Removing the `mtimeMs < sinceTime` prefilter makes bounded queries such as `/total_usage 5h` enumerate and scan every historical transcript before record-level filtering drops old entries. Long-lived installs can turn a small range command into a full-history I/O scan.
  - confidence: 0.84
- **[P2] Keep discovery filtering bounded by requested start:** `src/infra/session-cost-usage.ts:497-500`
  - body: `discoverAllSessions()` no longer honors `params.startMs`, but `sessions.usage` still sorts and slices discovered sessions before loading per-session summaries. Out-of-range transcript files can consume the limit and push in-range sessions out of the response or totals.
  - confidence: 0.78
- **[P2] Update CostUsageSummary fixtures:** `src/infra/session-cost-usage.types.ts:69`
  - body: Adding `models` as a required `CostUsageSummary` field changes existing typed fixtures. `commands-session-usage.test.ts` still returns summary objects without `models`, so the test-type lane needs those fixtures updated or the field made backward-compatible.
  - confidence: 0.78
