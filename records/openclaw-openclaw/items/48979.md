## Summary

Current main still does not bridge a successful built-in `read` image result into Telegram media delivery. The pinned Pi read tool returns text plus inline image blocks without `details.path` or structured media metadata, OpenClaw's read wrapper only normalizes/sanitizes those blocks, and the shared extractor still returns no media for pathless image content, so Telegram never receives a `mediaUrl`/`mediaUrls` attachment for this path.

## What This Changes

The issue requests that images read by the built-in `read` tool be delivered through Telegram as media attachments instead of only producing image metadata/text.

## Best Possible Solution

Add a read-tool media bridge before the shared pending-media queue, most likely in the OpenClaw read wrapper or trusted read-tool result handling. It should attach structured media metadata for the validated read path, preserve the existing trusted local media gate, and add regression coverage for Pi-style pathless read image results plus untrusted-tool/path-boundary cases.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: a Pi-style read image result with only text and image blocks is covered by the current extractor test and produces no media, so the handler has nothing to queue for Telegram.

## Solution Assessment

No. The current workaround of calling Telegram directly is not the maintainable fix; the narrow solution is to emit safe structured media metadata from trusted read results and let the existing pending-media and Telegram delivery paths handle it.
