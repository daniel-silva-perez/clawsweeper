## Summary

Keep open. Current main can deliver Telegram media when a reply already contains `mediaUrl`/`mediaUrls` or `MEDIA:` directives, but the specific read-tool image handoff is still missing: the pinned Pi read tool returns image content without a media path, and OpenClaw’s extraction code explicitly returns no media for image tool results that lack `MEDIA:` or `details.path`.

## Best Possible Solution

Keep this issue open and fix the read-tool bridge. The likely product/code path is to have the OpenClaw read wrapper, or the trusted read-tool event handler using the recorded start args plus workspace/sandbox mapping, attach a safe structured media reference for successful image reads. Then the existing pending-tool-media queue and Telegram `sendPhoto` delivery path can handle it. Add regression coverage for a Pi-style read image result with no `details.path` flowing into `mediaUrls`, plus a Telegram/local-roots assertion.
