## Summary

Current main still does not bridge a successful Pi-style `read` image result into Telegram media delivery. Telegram can send images once a `mediaUrl`/`mediaUrls` attachment exists, but the read wrapper returns sanitized inline image content without adding structured media metadata, and the shared extractor explicitly returns no media for image content that lacks `MEDIA:` text or `details.path`.

## Best Possible Solution

Keep this issue open and implement the read-tool media bridge. The likely fix belongs in the OpenClaw read wrapper or trusted read-tool event handling: attach safe structured media metadata such as `details.media.mediaUrl` with `trustedLocalMedia`, derived from the actual read path and validated against the workspace/sandbox trust boundary. Add regression coverage for a Pi-style read image result with no `MEDIA:` text and no `details.path` flowing into queued `mediaUrls`, plus trust-boundary coverage so untrusted tools cannot smuggle local attachments.

## Risks / Open Questions

- A fix that treats arbitrary image content blocks as sendable local media could let plugin, MCP, or model-emitted content trigger unintended local file uploads. The bridge should be limited to trusted built-in `read` results for the actual successfully read path after safe workspace/root resolution.
- Closing this as a duplicate of #65426 would miss the remaining gap: #65426 covered delivery after a `MEDIA:` path exists, while this issue is about producing that path/metadata from `read` image output.
