## Summary

Keep open. Current main still lacks the specific read-tool image handoff: the pinned Pi read tool returns inline image content without a path, OpenClaw's read wrapper does not add structured media metadata, and the tool-media extractor intentionally returns no media when an image tool result has no `MEDIA:` directive or `details.path`. Telegram can send images once `mediaUrl`/`mediaUrls` exists, so the remaining work is the read-result bridge rather than Telegram Bot API delivery.

## Best Possible Solution

Keep this issue open and fix the read-tool bridge. The likely implementation is to have the OpenClaw read wrapper, or the trusted read-tool event handler using recorded start args plus workspace/sandbox path resolution, attach safe structured media metadata such as `details.media.mediaUrl` and `trustedLocalMedia` for successful image reads. Add regression coverage for a Pi-style read image result with no `MEDIA:` and no `details.path` flowing into `mediaUrls`, plus local-root/trust-boundary coverage for Telegram delivery.
