## Summary

Keep open. The PR moves in the right direction for the reported local memory runtime-deps failure, but the updated selector still checks raw per-agent `memorySearch` overrides instead of the effective defaults-plus-override config that runtime memory search uses, leaving a real path where local embeddings run without installing `node-llama-cpp`.

## What This Changes

The PR adds a config-gated `node-llama-cpp@3.18.1` managed runtime dependency for bundled `memory-core` local memory search, updates the local embedding setup hint to `openclaw doctor --fix`, and adds bundled runtime-deps coverage.

## Best Possible Solution

Keep the managed runtime-deps approach and keep `node-llama-cpp` out of root package dependencies, but derive local-memory use from the same effective per-agent memory config that runtime code uses. Add focused tests for inherited defaults, per-agent re-enable, configured provider ids, auto-local model files, and local fallback.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is `agents.defaults.memorySearch = { enabled: false, provider: "local" }` with an agent override `{ enabled: true }`; runtime merging enables local embeddings for that agent, while this PR's detector sees only the raw override and omits `node-llama-cpp`.

## Solution Assessment

No, not yet. Installing the native package through the managed runtime-deps plan is the right direction, but the selector should reuse or mirror the resolved memory config merge contract before deciding whether local embeddings are configured.
