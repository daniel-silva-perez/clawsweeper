## Summary

Keep this PR open: current main does not include the requested managed `node-llama-cpp` runtime dependency, and the PR head still has a P2 config-inheritance gap where local memory search can run without selecting that dependency for installation.

## What This Changes

The PR adds a config-gated `node-llama-cpp@3.18.1` managed runtime dependency for bundled `memory-core` local memory search, updates the missing-runtime setup hint, and adds bundled-runtime-deps tests.

## Best Possible Solution

Keep `node-llama-cpp` out of root package dependencies and install it through the managed runtime-deps plan, but derive local-memory use from the same effective per-agent memorySearch config that runtime uses. Add focused tests for inherited defaults, per-agent re-enable, custom provider ids, auto local model files, and fallback.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is `agents.defaults.memorySearch = { enabled: false, provider: "local" }` with an agent override `{ enabled: true }`; runtime resolves that agent to local embeddings, while this PR scans the raw override and omits `node-llama-cpp`.

## Solution Assessment

No, not yet. The managed runtime-deps approach is the narrow maintainable fix, but the selector must mirror or reuse the runtime memory config merge contract before deciding whether local embeddings are configured.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Merge memory defaults before selecting local runtime deps:** `src/plugins/bundled-runtime-deps-selection.ts:546-548`
  - body: The detector evaluates defaults and each agent override as separate objects. Runtime merges `enabled`, `provider`, `fallback`, and `local` settings first, so defaults like `{ enabled: false, provider: "local" }` plus an agent override `{ enabled: true }` still run local embeddings but install only the declared plugin deps. Build the same effective per-agent memory config before checking for local usage.
