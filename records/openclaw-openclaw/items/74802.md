## Summary

Keep this PR open. The managed runtime-deps direction is the right boundary for the linked local memory-search bug, and the latest branch covers several useful local-selection cases, but the selector still evaluates defaults and per-agent overrides separately while runtime memory search uses the merged effective config. That leaves a real local-embeddings path without `node-llama-cpp` installed.

## What This Changes

The PR adds config-gated `node-llama-cpp@3.18.1` to bundled `memory-core` managed runtime dependencies, updates the local setup hint to `openclaw doctor --fix`, and adds runtime-deps tests.

## Best Possible Solution

Keep `node-llama-cpp` out of root package dependencies and install it through the managed runtime-deps plan, but derive local-memory use from the same effective memory config contract as runtime search. Add focused tests for inherited defaults, per-agent re-enable, inherited local fallback, inherited configured provider ids, and inherited `local.modelPath` auto-selection.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is `agents.defaults.memorySearch = { enabled: false, provider: "local" }` plus an agent override `{ enabled: true }`: runtime resolves that agent to local embeddings, while the PR detector sees only the raw override and omits `node-llama-cpp`.

## Solution Assessment

No, not yet. The managed runtime-deps approach is the narrow maintainable fix, but the selector must reuse or mirror the runtime defaults-plus-override merge before deciding whether local embeddings are configured.
