## Summary

Keep this PR open. The updated patch now covers the earlier literal-provider gap by handling configured provider ids, auto-local model paths, and local fallback, but it still does not follow the runtime config merge contract for default memory settings inherited by per-agent overrides. That leaves a supported local-memory configuration able to select the local adapter while the managed runtime-deps plan omits `node-llama-cpp`.

## What This Changes

The PR adds a config-gated `node-llama-cpp@3.18.1` runtime dependency for bundled `memory-core`, updates the local embedding setup hint to use `openclaw doctor --fix`, and adds bundled runtime-deps tests.

## Best Possible Solution

Keep the managed runtime-deps approach and keep `node-llama-cpp` out of root package dependencies, but make the selector derive local-memory use from the same effective memory config that runtime code uses for each agent. Add focused tests for inherited defaults, per-agent re-enable, configured provider ids, auto-local model files, and local fallback.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a config with `agents.defaults.memorySearch` set to `{ enabled: false, provider: "local" }` and an agent override set to `{ enabled: true }`; current runtime merging selects the inherited local provider for that agent, while this PR’s detector sees only the raw override and does not add `node-llama-cpp`.

## Solution Assessment

No, not yet. Installing the native package through the managed bundled runtime-deps plan is the right fix direction, but the detector should reuse or mirror the resolved memory config merge contract rather than inspecting unmerged config fragments.
