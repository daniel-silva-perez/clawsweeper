## Summary

Keep this PR open. Current main still has the managed-runtime dependency gap for local memory embeddings, so the patch addresses a real bug, but the proposed selector is too narrow because it only recognizes the literal `local` provider string and misses supported local-selection paths that current runtime and docs allow.

## What This Changes

The PR adds config-gated `node-llama-cpp@3.18.1` to bundled `memory-core` runtime dependency selection, updates the local setup error hint, and adds bundled runtime-deps tests.

## Best Possible Solution

Keep the managed runtime-deps approach and keep `node-llama-cpp` out of root package dependencies, but make the selector follow the same memory provider resolution contract as runtime: literal `local`, `auto` when an existing local model path makes local eligible, and configured provider ids whose `api` resolves to `local`, with focused tests for those cases.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists by configuring local memory search and scanning or ensuring bundled runtime deps for `memory-core`; current main imports `node-llama-cpp` by bare specifier but does not include it in the managed dependency plan.

## Solution Assessment

No. The managed runtime-deps boundary is the right direction, but this patch is not yet the best fix because its detection logic does not match the documented and implemented memory provider resolver.
