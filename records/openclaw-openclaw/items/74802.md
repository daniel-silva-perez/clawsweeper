## Summary

Keep this PR open. The current PR head still has the correctness gap from the earlier review: it detects local memory runtime dependencies from raw default/per-agent config blocks instead of the effective merged memory config that runtime code actually uses, so some valid inherited local-memory configurations would still omit `node-llama-cpp` from the managed runtime plan.

## What This Changes

The PR adds config-gated `node-llama-cpp@3.18.1` managed runtime dependency selection for bundled `memory-core` local memory search, updates the local setup hint to `openclaw doctor --fix`, and adds bundled runtime-deps tests.

## Best Possible Solution

Keep the managed runtime-deps approach and keep `node-llama-cpp` out of root package dependencies, but derive local-memory dependency selection from the same effective per-agent memory config used at runtime. Add focused coverage for inherited defaults, per-agent re-enable, configured provider ids backed by `local`, auto-local model files, and local fallback.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is `agents.defaults.memorySearch = { enabled: false, provider: "local" }` plus an agent override `{ enabled: true }`; runtime merging enables local embeddings for that agent, while this PR's detector sees only the raw override and omits `node-llama-cpp`.

## Solution Assessment

No, not yet. Installing the pinned native package through the managed runtime-deps plan is the right direction, but the selector must reuse or mirror the runtime memory config merge contract before deciding whether local embeddings are configured.
