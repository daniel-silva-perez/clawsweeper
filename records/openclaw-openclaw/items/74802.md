## Summary

Keep this PR open. The managed runtime-deps direction is the right boundary, but the selector still misses effective per-agent memory configs assembled from defaults plus overrides, so local embeddings can run without installing `node-llama-cpp`.

## What This Changes

The PR adds a config-gated `node-llama-cpp@3.18.1` managed runtime dependency for bundled `memory-core` local memory search, updates the local embedding setup hint to `openclaw doctor --fix`, and adds runtime-deps coverage.

## Best Possible Solution

Keep `node-llama-cpp` out of root dependencies and install it through the managed runtime-deps plan, but derive local-memory use from the same effective per-agent config that runtime memory search resolves. Add focused tests for inherited defaults, per-agent re-enable, custom provider ids, auto local model files, and local fallback.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is `agents.defaults.memorySearch = { enabled: false, provider: "local" }` with an agent override `{ enabled: true }`; runtime resolves that agent to local embeddings, while this PR's detector sees only the raw override and omits `node-llama-cpp`.

## Solution Assessment

No, not yet. The managed runtime-deps approach is the narrow maintainable fix, but the selector must reuse or mirror the runtime memory config merge contract before deciding whether local embeddings are configured.
