## Summary

Keep open. The linked bug is valid, but this PR is not mergeable as-is because it changes only fallback text selection while the all-error-payload path still trips fatal structured-error precedence and overwrites summary/output/delivery with the raw last error. The PR body closes #74807, so the right cleanup path is to repair this implementation rather than close either side early.

## What This Changes

This PR changes isolated cron payload outcome fallback logic so normalized `finalAssistantVisibleText` can be used for fallback summary/output before falling back to the first raw payload text.

## Best Possible Solution

Repair the cron/embedded-runner boundary so the final assistant-visible reply wins only for a proven recovered or self-corrected mutating-tool failure, while unresolved failures remain fatal; cover that boundary with focused helper and payload regression tests.

## Reproduction Assessment

Yes. A high-confidence unit reproduction is `resolveCronPayloadOutcome` with only `isError: true` payloads, `finalAssistantVisibleText`, and `preferFinalAssistantVisibleText: true`; current tests already assert that this returns the raw last error, and the PR’s fallback-only change does not reach the fatal override.

## Solution Assessment

No. The PR is not the best way to solve the issue because it changes fallback text after payload summary extraction, but the reported path is decided by fatal structured-error precedence; the safer fix is to classify recovered versus unresolved mutating failures before producing or honoring a fatal cron payload.
