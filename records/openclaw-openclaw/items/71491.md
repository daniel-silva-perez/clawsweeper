## Summary

Keep open. Current main still lacks the Moonshot/Kimi-specific outgoing payload repair for replayed assistant tool-call messages that are missing top-level `reasoning_content`, and the focused fix PR #74846 is still open and unmerged. The report is concrete, current, and distinct from the already-merged native Kimi tool-call ID fix in #70030; the LCM-side cross-link is useful context rather than a reason to close this OpenClaw issue.

## What This Changes

The issue asks OpenClaw to preserve or synthesize Moonshot/Kimi `reasoning_content` on replayed assistant tool-call messages after LCM compaction or cross-model fallback so Kimi thinking sessions do not fail with provider 400s.

## Best Possible Solution

Review and land a narrow Moonshot-only outgoing payload repair, preferably via #74846, that preserves existing `reasoning_content`, adds a placeholder only for replayed assistant tool-call messages when Moonshot thinking is enabled, and leaves generic OpenAI-compatible replay behavior unchanged.

## Reproduction Assessment

Yes. The issue body and comments give concrete long-session and fallback reproductions, and source inspection shows current main can still send Moonshot thinking requests without repairing replayed assistant tool-call messages that lack top-level `reasoning_content`.

## Solution Assessment

Yes, with maintainer validation. The provider-wrapper boundary targeted by #74846 is the narrowest maintainable fix because it runs after context-engine assembly, mirrors the existing DeepSeek compatibility pattern, and avoids changing generic replay policy for other OpenAI-compatible providers.
