## Summary

Keep open. Current main has substantial Telegram polling hardening that likely covers the Telegram half of the report, but the issue also reports an indefinite Discord disconnect/health-monitor restart loop. Current Discord code still marks gateway close/reconnect events as disconnected and the gateway health monitor still restarts disconnected channels; I did not find a current-main implementation that proves the Discord loop is resolved for this report.

## Best Possible Solution

Keep this issue open, but narrow maintainer follow-up to the unresolved Discord stability path and a current-version retest. Ask for logs from current main or a post-v2026.4.26 build showing whether Discord still cycles after the new gateway timeout/liveness changes; if it does, fix it in the Discord plugin/runtime by distinguishing transient Carbon reconnects from monitor-worthy disconnected states, adding a regression test for repeated close/reconnect events, and keeping the existing Telegram polling fixes as already-covered context.
