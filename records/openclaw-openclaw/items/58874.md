## Summary

Keep this PR open. Current main preserves requester delivery targets through `requesterOrigin`, but it still hardcodes the subagent completion `inputProvenance.sourceChannel` as `webchat`, so the PR’s stated requester-channel provenance change is not implemented on main. The linked bug report being closed is not enough to close this PR because the remaining PR diff is a distinct, still-unlanded provenance change.

## Best Possible Solution

Keep the PR open and ask for a small follow-up: derive completion `sourceChannel` from the normalized/resolved requester origin, add regression coverage for non-webchat and normalized channel values in `inputProvenance.sourceChannel`, and update the PR body to match the final scope. If maintainers decide only delivery routing mattered and provenance should stay internal, close with that explicit product judgment rather than an automated implemented-on-main cleanup.
