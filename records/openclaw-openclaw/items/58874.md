## Summary

Keep this PR open: the reported channel-provenance bug is still present on current main, and the PR is useful but incomplete. The final one-line patch fixes only the simplest completion path and still derives `sourceChannel` from the raw requester origin instead of the normalized/resolved delivery origin used by the announce flow.

## What This Changes

The PR changes `src/agents/subagent-announce.ts` so completion announcements pass `params.requesterOrigin?.channel ?? INTERNAL_MESSAGE_CHANNEL` as `sourceChannel` instead of always using `INTERNAL_MESSAGE_CHANNEL`.

## Best Possible Solution

Update the completion announce flow to derive `sourceChannel` from the normalized/resolved origin used for delivery, not the raw params object, then add regression coverage for a non-webchat requester and the fallback/reparenting path. Keep the broader timeout and concurrency concerns from the discussion out of this narrow fix unless a maintainer opens separate work for them.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: pass a non-webchat requester origin through `runSubagentAnnounceFlow` and the current main path still supplies `INTERNAL_MESSAGE_CHANNEL` to the delivery layer, which becomes `inputProvenance.sourceChannel`.

## Solution Assessment

No. The PR points in the right direction, but using `params.requesterOrigin?.channel` is not the best fix because the flow already computes normalized and sometimes replaced requester origins; the safer narrow fix is to use the resolved delivery origin for provenance.

## Review Findings

Overall correctness: patch is incorrect

- **[P2] Use the resolved requester channel for provenance:** `src/agents/subagent-announce.ts:607`

Derive `sourceChannel` from the resolved delivery origin instead.
