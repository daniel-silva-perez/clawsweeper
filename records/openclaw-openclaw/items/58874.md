## Summary

Keep this PR open: current main still hardcodes completion announce provenance to the internal webchat channel, and the PR is useful but incomplete because it reads the original requester params instead of the normalized/resolved requester origin that the delivery path actually uses.

## What This Changes

The PR changes `src/agents/subagent-announce.ts` so completion announcements pass `params.requesterOrigin?.channel ?? INTERNAL_MESSAGE_CHANNEL` as `sourceChannel` instead of always passing `INTERNAL_MESSAGE_CHANNEL`.

## Best Possible Solution

Derive completion announce `sourceChannel` from the same normalized/resolved delivery origin used for the requester handoff, then add focused regression coverage for non-webchat provenance and the reparented requester fallback path. Keep the broader timeout, fire-and-forget, and concurrency suggestions as separate work.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to run the announce flow with a missing subagent requester that resolves to a WhatsApp/Telegram parent; delivery uses the recovered origin, while the PR line would still use the original params channel or fall back to webchat.

## Solution Assessment

No. The PR points at the right field, but the best fix is to use the resolved delivery origin (`completionDirectOrigin`, `directOrigin`, or `targetRequesterOrigin`) rather than raw `params.requesterOrigin`.
