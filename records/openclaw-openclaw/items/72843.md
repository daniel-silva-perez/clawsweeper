## Summary

Keep this PR open. The underlying bug is valid on current main and the PR moves in the right direction, but the second commit overcorrects the `index.js` false-positive fix: documented Docker/Fly daemon launches still run as `node dist/index.js gateway` from `/app` or a relative path, which will not have an exact `openclaw` ancestor and will still skip eager context-window warmup.

## What This Changes

The PR updates `src/agents/context.ts` to recognize `dist/index.js`/`index.mjs` gateway entries and allow eager context-window cache warmup for `gateway`, with focused tests in `src/agents/context.test.ts`.

## Best Possible Solution

Keep the PR open and adjust the entrypoint detection so real OpenClaw gateway daemons are recognized across npm/global installs, dev checkouts, Docker, Podman, and Fly-style `dist/index.js` launches, while preserving the generic `index.js` plugin-sdk false-positive guard and adding regression tests for both boundaries.

## Reproduction Assessment

Yes. A focused check of the PR head logic with `shouldEagerWarmContextWindowCache(["node", "dist/index.js", "gateway"])` or `shouldEagerWarmContextWindowCache(["node", "/app/dist/index.js", "gateway"])` would return false because neither path has an exact `openclaw` ancestor, matching the documented Docker/Fly daemon launch shape.

## Solution Assessment

No, not as-is. Removing `gateway` from the skip list is appropriate and the second commit fixes the broad bare-`index.js` regression, but the ancestor-only entry gate is too narrow for supported daemon deployments; a safer fix should use a real OpenClaw packaged-entry signal that also covers container/cloud `dist/index.js` launches.
