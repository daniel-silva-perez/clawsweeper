## Summary

Keep open. Current main still skips eager context-window cache warmup for gateway daemon argv shapes, and the PR heads in the right direction, but it should not merge as-is because the tightened `index.js` recognition still misses documented container/cloud `node dist/index.js gateway` launches from `/app`, leaving the same 200k fallback/persistence bug in those supported daemon deployments.

## What This Changes

The PR updates `src/agents/context.ts` to recognize `dist/index.js`/`index.mjs` OpenClaw entry paths and allow eager context-window cache warmup for `gateway`, with focused tests in `src/agents/context.test.ts`.

## Best Possible Solution

Keep the PR open and adjust the entrypoint detection so real OpenClaw gateway daemons are recognized across npm/global installs, dev checkouts, and documented container/cloud `dist/index.js` launches, while preserving the plugin-sdk false-positive guard and adding regression tests for both sides of that boundary.

## Reproduction Assessment

Yes. A focused unit check against the PR logic with `shouldEagerWarmContextWindowCache(["node", "/app/dist/index.js", "gateway"])` would return false because `/app` has no exact `openclaw` ancestor, matching documented Docker/Fly gateway launch shapes.

## Solution Assessment

No, not as-is. Removing `gateway` from the skip list is the right direction and the second commit fixes the generic `index.js` false positive, but the ancestor-only `index.js` gate is too narrow for supported daemon deployments; a safer fix should use a real OpenClaw entry signal that also covers `/app/dist/index.js` and relative `dist/index.js` gateway launches.
