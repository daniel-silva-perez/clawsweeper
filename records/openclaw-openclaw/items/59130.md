## Summary

Keep open. Current main still performs LanceDB auto-recall as semantic vector search and injects the returned memories without recency-aware ranking or source/timestamp provenance, and the linked open PR #59141 is the active implementation candidate for this same issue.

## What This Changes

Request recency-aware and provenance-aware LanceDB memory recall so freshness-sensitive prompts do not answer from stale injected memory context.

## Best Possible Solution

Keep this issue open until #59141 or a replacement lands. The durable fix belongs in memory-lancedb: detect freshness intent, retrieve enough candidates to rerank, use createdAt and source timestamps when available, surface memory provenance or staleness hints to the assistant, and add a regression test for an older semantic match losing to a newer relevant memory.

## Reproduction Assessment

Yes. A mocked before_prompt_build test can return LanceDB rows where an older memory has the better vector score and a newer matching memory has a later createdAt; current main will preserve the vector-search order and inject no timestamp/provenance clue.

## Solution Assessment

Unclear. The linked PR’s direction is the narrowest current implementation lane because it stays in memory-lancedb, but maintainers should decide whether createdAt reranking alone is enough or whether source timestamp/provenance fields need to be included in the same fix.
