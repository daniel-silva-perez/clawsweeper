## Summary

Keep this PR open for revision. The current diff duplicates the existing false-only Private API cache guard, but the PR's own reported failure path is the cache returning null, so the proposed guards do not prevent the optional BlueBubbles read/typing endpoints from being called in that case.

## What This Changes

The PR adds early cached Private API disabled checks to `markBlueBubblesChatRead` and `sendBlueBubblesTyping` in `extensions/bluebubbles/src/chat.ts`.

## Best Possible Solution

Revise the BlueBubbles optional read/typing behavior so Private-API-only calls cannot produce user-visible fallback errors when capability is disabled or still unknown after probing. The fix should cover the null cache path explicitly and add regression coverage for known-false and cold/unknown cache states.

## Reproduction Assessment

Yes. The PR provides a concrete live reproduction with a BlueBubbles server reporting `private_api: false`, and current main's source shows the null cached-status path still falls through to the read/typing endpoint calls.

## Solution Assessment

No. The proposed false-only outer guards are not the narrowest reliable fix because the inner helper already has the same false-only guard; a safer solution must handle the null/unknown status path or gate these optional calls before dispatching them.
