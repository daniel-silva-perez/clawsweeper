## Summary

Keep this PR open. Current main still does not implement the intended session sync status observability in the memory status payload, JSON output, or `openclaw memory status` rendering. The PR is stale against the current `extensions/memory-core` plugin layout, but the author did address the review comments about timestamp consistency and Map aggregation, so it remains a plausible implementation candidate after a port.

## Best Possible Solution

Keep this PR open and ask for a rebase/port into `extensions/memory-core`. The best implementation is an additive typed session-sync status helper on the memory-core manager, surfaced in the status payload and `openclaw memory status` JSON/plain/deep output, with focused manager and CLI tests. Carry forward the author's fixes for timestamp consistency and Map aggregation, and add error sanitization before display or logging.

## Risks / Open Questions

- The PR cannot merge as-is because it targets pre-plugin memory/CLI paths rather than the current `extensions/memory-core` implementation.
- A port should sanitize or truncate any displayed/logged session sync error before exposing it in status output, because raw errors can include local paths or content-derived text.
- The status shape should remain additive and backwards-compatible, likely via `MemoryProviderStatus.custom.sessionSync` or an equivalent typed extension field.
- No current-main tests cover the proposed session sync status payload or CLI rendering yet.
