## Summary

Keep this issue open. Current main has gateway-backed session storage, a Sessions management view, and a visible chat toolbar New Session control, but the Control UI still wires that control to the `/new` reset path on the current session instead of creating a separate persistent dashboard chat via `sessions.create`. The related closed #69801 only accounts for the visible button surface; it does not supersede the broader multi-session/new-chat workflow requested here.

## Best Possible Solution

Keep this issue open and implement the focused Control UI integration: wire the New Session/New Chat affordance to `sessions.create`, navigate immediately to `/chat?session=<returned-key>`, refresh `sessions.list`, and keep typed `/new` or `/reset` as the existing reset path. The existing Sessions view can remain the management surface for rename/delete; export/import or retention controls should be split into narrower follow-ups if maintainers still want them.
