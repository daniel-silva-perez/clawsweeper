## Summary

Keep this issue open. Current main has important pieces of the requested surface, including gateway-backed sessions, a Sessions view with label/delete controls, and a visible toolbar New Session button, but the Control UI button still dispatches `/new` as a reset of the current chat rather than creating a separate persistent dashboard chat/session. The backend already exposes `sessions.create`, so the remaining work is a focused Control UI integration rather than a core storage/API gap.

## Best Possible Solution

Keep the issue open and implement the remaining Control UI flow by wiring the New Session/New Chat affordance to `sessions.create`, switching immediately to the returned key via `/chat?session=<key>`, refreshing `sessions.list`, and preserving slash-command `/new` as the existing reset path. The current Sessions view can remain the management surface for rename/delete; export/import should be handled as a narrower follow-up if maintainers still want it.
