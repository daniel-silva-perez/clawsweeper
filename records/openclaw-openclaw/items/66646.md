## Summary

Keep open. Current main contains the #68700 classifier fix and stale-lock reclaim work, but a direct session write-lock timeout inside an agent attempt is still unclassified, and the model fallback runner still advances to the next candidate for unclassified errors while candidates remain.

## What This Changes

The issue asks for session file lock timeouts to be treated as infrastructure/session-busy failures instead of model failures that cascade through the fallback chain.

## Best Possible Solution

Keep this as the canonical core agents bug and add a narrow infrastructure-error escape for SessionWriteLockTimeoutError or OPENCLAW_SESSION_WRITE_LOCK_TIMEOUT at the model fallback wrapper or agent-attempt boundary. Generic unknown-error fallback can remain unchanged, while queue/retry/backoff can be handled separately at the session lane or caller boundary.

## Reproduction Assessment

Yes. A high-confidence unit reproduction is available by configuring a multi-candidate fallback chain and making the first run callback throw SessionWriteLockTimeoutError; current main should call the callback again for the next candidate because the error is unclassified.

## Solution Assessment

No. The current classifier-only fix is not the best complete solution because returning null from classification still flows into the generic unknown-error fallback path; the safer fix is an explicit non-model infrastructure/session-busy branch before the fallback loop advances.
