## Summary

Keep this issue open. Current main has the classifier portion of #68700 and stale-lock reclaim logic, but the central requested behavior is still incomplete: a direct SessionWriteLockTimeoutError thrown during an agent attempt is unclassified, and runWithModelFallback still advances unrecognized errors across remaining model candidates. The latest issue discussion also reports the failure on v2026.4.26, with #73684 closed as a duplicate into this thread and #69030 tracking only one `infer model run` trigger path.

## Best Possible Solution

Keep this as the canonical core agents bug. Add an explicit non-model-fallback path for SessionWriteLockTimeoutError and nested OPENCLAW_SESSION_WRITE_LOCK_TIMEOUT at the model fallback wrapper or agent-attempt boundary, then add a regression where a multi-candidate fallback chain invokes the run callback exactly once and surfaces a clear session-busy/infrastructure error. Queue/retry/backoff can be handled at the session lane or caller boundary, while #69030 remains a separate raw-inference follow-up for one trigger path.
