## Summary

Keep open. The merged classifier work and stale-lock cleanup reduce parts of the problem, but current main still lets a direct session write-lock timeout become an unclassified candidate failure and advance through remaining model fallback candidates.

## What This Changes

The issue asks OpenClaw to treat session write-lock timeouts as local infrastructure/session-busy errors rather than model failures that advance through the fallback chain.

## Best Possible Solution

Keep this as the canonical core agents bug and add a narrow session-busy infrastructure escape before the model fallback loop advances. The fix should preserve generic unknown-error fallback behavior, but explicitly rethrow or surface SessionWriteLockTimeoutError / OPENCLAW_SESSION_WRITE_LOCK_TIMEOUT without recording it as a model candidate failure.

## Reproduction Assessment

Yes. A high-confidence unit reproduction is to configure two fallback candidates and make the first run callback throw SessionWriteLockTimeoutError; current main will classify it as non-timeout but still call the second candidate because unclassified errors continue while candidates remain.

## Solution Assessment

No, the current classifier-only solution is incomplete. The safer fix is an explicit non-model infrastructure branch at the fallback wrapper or agent-attempt boundary, with queue/backoff handled separately if maintainers want that policy.
