## Summary

Keep open. Current main includes the #68700 classifier fix and session-lock cleanup improvements, but the central fail-fast behavior is still missing: a direct `SessionWriteLockTimeoutError` thrown while acquiring the session lock inside an agent attempt can still be caught by `runWithModelFallback()` as an unrecognized error and retried against remaining model candidates.

## Best Possible Solution

Keep this open and add a targeted non-model-fallback path for `SessionWriteLockTimeoutError` and nested `OPENCLAW_SESSION_WRITE_LOCK_TIMEOUT` at the fallback wrapper or agent-attempt boundary. Add a focused regression where a configured fallback chain has multiple candidates and the run callback throws a session write-lock timeout; assert the callback is invoked once and the surfaced error is a clear session-busy/infrastructure error. Queue/retry/backoff can be layered at the caller or session lane after the no-cascade behavior is locked in.
