## Summary

Keep this issue open. Current main contains the #68700 classifier fix that stops session write-lock timeouts from being labeled as provider timeouts, and it has stale-lock cleanup/reclaim logic, but the core requested behavior is still not implemented: a direct SessionWriteLockTimeoutError thrown inside an agent attempt remains an unrecognized error, and runWithModelFallback() still advances across remaining model candidates for unrecognized errors. The recent issue comments also report the failure on v2026.4.26, so this is still active rather than stale cleanup.

## Best Possible Solution

Keep this issue as the canonical core agents bug. Add a targeted non-model-fallback path for SessionWriteLockTimeoutError and nested OPENCLAW_SESSION_WRITE_LOCK_TIMEOUT at the model fallback wrapper or agent attempt boundary, plus a regression where a multi-candidate fallback chain invokes the run callback exactly once and surfaces a clear session-busy/infrastructure error. Queue/retry/backoff can then be handled at the session lane or caller boundary, while #69030 remains a separate raw-inference follow-up for one trigger path.
