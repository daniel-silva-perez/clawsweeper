## Summary

Keep open. Current main includes the #68700 classifier fix that stops session write-lock errors from being labeled as provider timeouts, but it still lacks the central fail-fast/session-busy behavior. A direct SessionWriteLockTimeoutError is unclassified, and runWithModelFallback still advances unrecognized errors while fallback candidates remain, so the same session lock can still be retried through the configured model chain.

## Best Possible Solution

Keep this issue open. Add a targeted non-model-failover path at the model fallback wrapper or agent-attempt boundary for SessionWriteLockTimeoutError and nested OPENCLAW_SESSION_WRITE_LOCK_TIMEOUT errors, surfacing a clear session-busy/infrastructure error. Add a regression where multiple fallback candidates are configured and the run callback throws a session write-lock timeout; assert the callback runs exactly once and no fallback candidate is attempted. Queue/retry/backoff can then be handled at the session lane or caller boundary, while #69030 remains a separate raw-inference follow-up for one trigger path.
