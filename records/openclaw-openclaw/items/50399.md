## Summary

Keep this PR open. Current main still does not solve the Feishu voice/video duration problem on the normal message and reply send path: it only forwards a duration if some caller already supplies one. The PR also is not a conservative close candidate because its own discussion and diff show security-sensitive process probing and regression-prone rewrites around Feishu media routing/download metadata. The affected surface is Feishu media upload/send via `sendMediaFeishu` and `client.im.file.create`, with the remaining user-facing bug tracked by #53798.

## Best Possible Solution

Keep this PR open for maintainer follow-up or replacement. The best fix is a smaller Feishu-only patch that preserves `extractFeishuDownloadMetadata` and `resolveFeishuOutboundMediaKind`, computes duration after media normalization with the shared `runFfprobe`/`execFile`-based helper, passes duration only for proven Feishu Opus/MP4 uploads, and adds focused tests for duration propagation plus the existing routing and metadata behavior. Coordinate the final fix with #53798.
