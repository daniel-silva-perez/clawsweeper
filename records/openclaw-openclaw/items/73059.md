## Summary

Keep open. Current main still contains the raw direct-execution path comparison that explains the Windows junction no-op, the gateway auto-build path can still report the missing Control UI index after a successful child exit, and the focused test coverage still only covers Windows shell launcher behavior. The related Windows UI build PR is adjacent and closed unmerged, not a canonical fix for this direct `node .../scripts/ui.js build` path.

## What This Changes

The issue reports that Windows linked source installs can skip the Control UI auto-build when `scripts/ui.js` is launched through the global npm junction path.

## Best Possible Solution

Keep the issue open until `scripts/ui.js` recognizes direct execution by canonicalizing both the argv entry path and module path through realpath, with a resolved-path fallback, and add focused regression coverage without changing Control UI app code or package dependencies.

## Reproduction Assessment

Yes. The issue gives concrete Windows `pnpm link --global` junction steps, and current main still has the exact raw path comparison that can make direct `node ...\scripts\ui.js build` skip `main()` when argv and module paths name the same file differently.

## Solution Assessment

Yes, the narrow maintainable solution is to fix the direct-execution guard in `scripts/ui.js` and cover it in `test/scripts/ui.test.ts`; changing the gateway auto-build caller or Control UI build output would be broader than necessary.
