## Summary

Keep open. Current main and the latest release still have the raw `scripts/ui.js` direct-execution guard described in the report, the gateway auto-build path can surface the reported missing-index symptom, and the existing focused tests do not cover junction or symlink-launched execution. Related PR #52291 is adjacent Windows `pnpm.cmd` quoting work, not a fix for the direct `node .../scripts/ui.js build` path.

## What This Changes

Fix Windows linked-source installs so junction-launched `scripts/ui.js build` executes the same Control UI build path as repo-local `node scripts/ui.js build`.

## Best Possible Solution

Make `scripts/ui.js` reliably recognize direct execution when the launch path and module path point to the same file through a junction or symlink, and lock that behavior with a focused regression test while leaving Control UI app code and package metadata unchanged.

## Reproduction Assessment

Yes. The issue gives concrete Windows `pnpm link --global` junction steps, and current main still has the raw path comparison that explains why direct `node ...\scripts\ui.js build` can no-op under a junction launch.

## Solution Assessment

Yes. The narrow maintainable fix is to normalize both the launch path and `import.meta.url` file path through realpath, with a resolved-path fallback if realpath fails, then add focused coverage in `test/scripts/ui.test.ts`.

In `scripts/ui.js`, replace the raw direct-execution comparison with a small helper that treats `process.argv[1]` and `fileURLToPath(import.meta.url)` as the same file after `fs.realpathSync.native` or equivalent realpath normalization, falling back to `path.resolve` when realpath fails.
