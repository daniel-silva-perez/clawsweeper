## Summary

Keep open. Current main still uses the raw `process.argv[1]` versus `import.meta.url` path comparison that explains the junction-launched no-op, and the gateway auto-build path can still surface the reported missing `dist/control-ui/index.html` symptom. The related Windows UI build PR is adjacent quoting/dependency-install work, not a canonical fix for this direct `node .../scripts/ui.js build` entrypoint guard.

## What This Changes

The issue asks for Windows source installs launched through the global npm junction to run the same `scripts/ui.js build` path as repo-local launches so Control UI assets are rebuilt on first start.

## Best Possible Solution

Make `scripts/ui.js` recognize direct execution when the launch path and module path resolve to the same file through a junction or symlink, preferably by realpath-normalizing both sides with a resolved-path or file-URL fallback, then lock it with a focused regression test and a changelog fix entry.

## Reproduction Assessment

Yes. The issue gives concrete Windows `pnpm link --global` junction steps, and current main still has the exact raw path comparison that can make direct `node ...\scripts\ui.js build` skip `main()` when `argv[1]` and `import.meta.url` name the same file through different paths.

## Solution Assessment

Yes. The narrow maintainable fix is in the script entrypoint guard and its tests; changing Control UI app code, package metadata, or the gateway build policy would be broader than needed.
