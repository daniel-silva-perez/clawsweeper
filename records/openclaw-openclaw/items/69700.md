## Summary

Keep this issue open. Current main still has the reported memory-wiki contract mismatch: `sources/*.md` files are collected as wiki source pages and lint still requires `id`/`pageType`/freshness metadata, while the docs still describe `sources/` as imported raw material. The current importers wrap raw content in structured source pages, so the safest fix is likely documentation clarification unless maintainers want to design a separate raw-file mode.

## Best Possible Solution

Keep this open and align the memory-wiki source-page contract. The narrowest likely fix is to clarify the Memory Wiki docs, CLI docs, and README that `sources/*.md` are structured wiki source pages with frontmatter, while raw originals are imported through `wiki ingest`, bridge, or unsafe-local and preserved inside source-page content/provenance. If maintainers want direct raw files inside the vault, design that as an explicit raw mode or ignored subdirectory rather than silently disabling all source-page lint.

## Clownfish Work Prompt

Fix the memory-wiki `sources/` frontmatter ambiguity reported in https://github.com/openclaw/openclaw/issues/69700. Current main intentionally treats `sources/*.md` as structured wiki source pages: `compile.ts` collects them, `lint.ts` requires `id`/`pageType`, and ingest/bridge/unsafe-local importers wrap raw content in frontmatter plus a fenced content block. Update docs only unless you find a maintainer-approved raw-mode design. Clarify in `docs/plugins/memory-wiki.md`, `docs/cli/wiki.md`, and `extensions/memory-wiki/README.md` that `sources/` contains imported source pages, not untouched raw drop-in files; raw originals should be imported through `openclaw wiki ingest`, bridge, or unsafe-local and are preserved through source provenance/content. Mention that direct `.md` files placed under `sources/` are part of the wiki contract and need frontmatter. Do not globally skip `sources/` in lint, because related #64696 depends on malformed source pages without valid frontmatter being detectable/repairable. No changelog entry is expected for docs-only clarification.
