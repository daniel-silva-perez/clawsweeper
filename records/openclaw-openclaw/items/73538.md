## Summary

Keep this PR open. Current main still does not implement the PR's intended Signal group-id allowlist behavior: Signal group access still delegates `groupAllowFrom` matching to the sender identity comparator, and the event handler extracts the inbound `groupId` but does not pass it into `resolveSignalAccessState`. The PR is a plausible implementation candidate for the linked open Signal bug, but it still needs maintainer review for the documented semantics and the narrow callback-scope issue raised in review.

## Best Possible Solution

Keep this PR open for Signal maintainer review. The best path is to decide and document the intended Signal `groupAllowFrom` semantics, adjust the PR so group-id matching is scoped to group allowlist evaluation without leaking into the DM access calculation, add/keep the regression tests, and avoid closing the remaining `requireMention` portion of #53308 unless it is also fixed.

## Risks / Open Questions

- The PR body uses closing syntax for #53308, but that linked issue also reports `requireMention` default behavior; this PR appears to address only the `groupAllowFrom` group-id matching part.
- The Greptile review comment identifies a narrow semantic issue: the proposed `isSenderOrGroupAllowed` closure captures `groupId` and is also used while computing the DM access decision for a group-message event. It is not described as a practical security bypass, but it is a real cleanup point before merge.
- The proposed behavior broadens Signal `groupAllowFrom` from sender identities to sender-or-group-id while current docs call it a group sender allowlist. Code, docs, and tests should be aligned intentionally.
