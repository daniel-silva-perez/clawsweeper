## Summary

Keep this PR open. Current main still does not implement the PR's intended Signal group-id allowlist behavior, and the PR is a plausible fix candidate. It should not be closed as cleanup because it still targets a real open Signal group access bug, but it needs maintainer review for access-control semantics, the Greptile-noted callback scope issue, and the fact that #53308 also reports a separate requireMention default concern.

## Best Possible Solution

Keep this PR open for Signal maintainer review. The best path is a Signal-local fix that either admits configured Signal groups through the existing channels.signal.groups group-id surface before sender-level groupAllowFrom filtering, or deliberately changes groupAllowFrom to also match group ids with docs updated to say so. In either case, scope group-id matching only to the group allowlist path, add regression coverage for group-id admission and DM non-matching, and avoid closing #53308 until the requireMention part is addressed or split.

## Risks / Open Questions

- The PR changes Signal group access-control semantics: if groupAllowFrom is treated as a sender allowlist, direct group-id matching can admit any sender in an allowed group, subject to mention gating.
- Greptile's review points out that the PR's isSenderOrGroupAllowed callback captures groupId and is also used while computing dmAccess, so group-id matching can leak into the DM allowlist calculation for group-message events. The current handler appears to avoid a practical bypass, but the callback should be scoped more tightly.
- The linked issue #53308 also reports requireMention default behavior; this PR only addresses the groupAllowFrom matching portion, so the closing reference should be adjusted or the remaining behavior verified separately.
- Docs currently describe Signal groupAllowFrom as a sender allowlist while groups entries are keyed by group id, so the accepted fix should align source, tests, and docs instead of leaving contradictory configuration semantics.
