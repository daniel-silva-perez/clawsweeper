## Summary

Keep this PR open. Current main has conservative stale session-lane recovery, but it still does not implement the PR's central behavior: a configurable stuck-session abort threshold, active-run abort from the diagnostic heartbeat, typed session.aborted telemetry, and diagnostic state transition back to idle after recovery. The PR remains a plausible implementation candidate for #71127 and the related #73510 report, with review work remaining.

## Best Possible Solution

Keep the PR open and rework it on top of current main's conservative recovery runtime. The best landing path is a deliberate diagnostics.stuckSessionAbortMs design with maintainer agreement on active-run abort policy, a race-safe session.aborted event emitted only after the state guard passes, focused heartbeat/recovery tests, docs and changelog coverage, generated schema/help updates, and coordination with #71127, #73510, and #73560.

## Risks / Open Questions

- Automatic active-run aborts can cancel legitimate long-running model/tool turns or clear queued work if the default threshold, disable semantics, or queue-clearing scope are wrong.
- Greptile's latest review identifies a remaining observability race in the PR: session.aborted can be emitted before the natural-recovery guard, producing a misleading event even when state is not changed.
- Because the PR adds a public config key and diagnostic event, docs, changelog, generated schema/help, reload semantics, and telemetry/event contracts need to stay aligned before landing.
- The PR overlaps the open linked bug #71127, the broader open report #73510, and the overlapping open #73560 attempt, so maintainers should choose one coordinated implementation path rather than close this work as obsolete.
