## Summary

Keep open. Current main still has warning-only stuck-session handling for active work, and this PR is a relevant implementation for the linked stuck-session recovery bug, but the PR head still has a concrete telemetry race that should be fixed before merge and needs maintainer review for the active-abort behavior.

## What This Changes

The PR adds `diagnostics.stuckSessionAbortMs`, heartbeat-triggered stuck-session abort/recovery, `session.aborted` diagnostics and OTel recording, config schema/reload updates, and focused diagnostics/config tests.

## Best Possible Solution

Land a maintainer-approved version of stuck-session recovery that aborts only after a clear threshold, emits abort telemetry only after confirming the diagnostic state is still the stale processing state, updates the public config/docs/changelog surfaces, and keeps the linked issue as the canonical bug until merge.

## Reproduction Assessment

Yes. Current main can be checked by starting the diagnostics heartbeat with a `processing` session past `diagnostics.stuckSessionWarnMs`: it emits `session.stuck` and calls only the conservative recovery path; the PR adds fake-timer tests for the abort threshold, retry, and late-recovery race.

## Solution Assessment

Unclear as-is. The implementation direction is narrow and relevant, but the PR should move `session.aborted` emission behind the latest-state guard and get maintainer approval for default active-run abort semantics before merge.
