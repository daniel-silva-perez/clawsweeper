## Summary

Keep this PR open. Current main now has conservative stale-session lane recovery, but it deliberately remains warning-only for active embedded/reply/lane work and still lacks the PR's `diagnostics.stuckSessionAbortMs` config, `session.aborted` event, and diagnostic-driven active abort behavior. The PR is still an active implementation candidate for the linked stuck-session reports, though it needs maintainer review against the newer recovery code and the remaining observability race noted in review.

## Best Possible Solution

Keep the PR open, but rebase it over `c500e870` and make it a deliberate opt-in or thresholded extension of the new conservative recovery runtime. The best landing path is to add `diagnostics.stuckSessionAbortMs` only with clear config/docs/changelog coverage, typed `session.aborted` telemetry, race-safe event ordering, focused diagnostics tests, and maintainer agreement that active-run aborts are acceptable for the linked stuck-session bug.

## Risks / Open Questions

- Automatic active-run aborts can cancel legitimate long-running model/tool turns or clear queued work if the default threshold, disable semantics, or queue-clearing scope are wrong.
- The PR head still emits `session.aborted` before the natural-recovery guard in the patch, matching the Greptile observability concern; that should be fixed or explicitly accepted before merge.
- Because current main added `diagnostic-stuck-session-recovery.runtime.ts` after the PR branch point, the PR should be rebased/reworked to reuse or extend that runtime instead of introducing a parallel abort path.
- The PR adds a user-visible config key and diagnostic event, so docs, changelog, generated schema/help, and event contract surfaces need to stay aligned before landing.
