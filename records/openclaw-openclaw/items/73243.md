## Summary

Keep open. Current main still has warning-only stuck-session diagnostics plus conservative stale-lane recovery, while this PR addresses the real unresolved auto-recovery gap; it should not be closed as obsolete. The patch still needs maintainer review because it changes default active-abort behavior and has a concrete telemetry race where `session.aborted` can be emitted after the session already recovered and started new work.

## What This Changes

The PR adds `diagnostics.stuckSessionAbortMs`, `session.aborted` diagnostics and OpenTelemetry handling, heartbeat-driven stuck-session abort recovery, config schema/reload updates, and focused diagnostics/config tests.

## Best Possible Solution

Keep this implementation path alive for the stuck-session recovery bug, but land it only after abort telemetry is moved behind the latest-state guard, maintainers approve the default active-run abort semantics, user-facing docs/changelog/config surfaces are aligned, and the targeted diagnostics/recovery validation plus changed gate pass.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Move abort telemetry behind the state guard:** `src/logging/diagnostic.ts:763`
  - body: `logSessionAbortRequested` runs before the latest-state guard, so a session that already recovered and started new work can still emit `session.aborted` logs/OTel even though this handler then returns without applying the recovery state transition. Move the log/emission after the guard so abort telemetry only records a recovery this heartbeat still owns.
  - confidence: 0.88
