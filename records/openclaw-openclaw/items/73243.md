## Summary

Keep this PR open. Current main still does not implement the PR's central behavior: a configurable `diagnostics.stuckSessionAbortMs` threshold that aborts active stuck sessions, emits typed `session.aborted` diagnostics, and transitions diagnostic state back to idle. Main has adjacent conservative lane recovery, but docs and tests show active embedded/reply/lane work remains warning-only by default. The PR is still a plausible implementation candidate for linked bug #71127, with review work remaining.

## Best Possible Solution

Keep the PR open and rework it on top of current main's conservative recovery runtime. The best landing path is a deliberate `diagnostics.stuckSessionAbortMs` design with maintainer agreement on active-run abort policy, race-safe event ordering, typed `session.aborted` telemetry, focused diagnostics/recovery tests, config docs/changelog coverage, and explicit handling of linked bug #71127 and related #73510.

## Risks / Open Questions

- Automatic active-run aborts could cancel legitimate long-running model/tool turns or clear queued work if the default threshold, disable semantics, or queue-clearing scope are wrong.
- Greptile's latest review notes a remaining observability race: the PR can emit `session.aborted` before the natural-recovery guard, creating a misleading event even when state is not changed.
- The PR should be reconciled with main's newer `diagnostic-stuck-session-recovery.runtime.ts` instead of maintaining a parallel abort path.
- Because it adds a user-visible config key and diagnostic event, docs, changelog, generated schema/help, and telemetry event contracts need to stay aligned before landing.
