## Summary

Keep open. Current main still contains the reported clipping conditions: `.usage-left-card` remains in the shared `overflow: clip` group, `.daily-chart-bars` still uses `overflow-x: auto`, and the tooltip is still absolutely positioned above each `.daily-bar-wrapper`. The linked fix PR was closed unmerged, so there is no landed implementation to close this against.

## What This Changes

The issue requests that the Control UI Usage tab's Daily Token Usage tooltip stay fully visible when tall bars place the absolutely positioned tooltip above the chart area.

## Best Possible Solution

Keep this open and fix the Usage tab in `ui/src/styles/usage.css`: remove `.usage-left-card` from the clipped shared rule, then either allow the Daily Usage tooltip to escape `.daily-chart-bars` or move the tooltip to fixed/portal-style positioning if horizontal scrolling must be preserved. Add a focused UI/CSS regression for tall daily bars so the tooltip remains readable.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to load the Usage tab with a daily data set that produces a near-maximum-height bar, hover that bar, and observe that the absolute tooltip is constrained by the current clipped/overflowing ancestors shown in source.

## Solution Assessment

Yes, a narrow Control UI fix is the right direction, but the exact CSS should account for long date ranges. If horizontal scrolling is required, fixed-position or portal-style tooltip coordinates are safer than only changing `.daily-chart-bars` to `overflow: visible`.
