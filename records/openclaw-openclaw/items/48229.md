## Summary

Keep this issue open. Current main still clears requested WebSocket operator scopes for device-less shared token/password connections, and the live GitHub PR that references this issue is still open rather than merged. The HTTP scope default mentioned in a comment is a separate path and does not fix the reported WebSocket behavior.

## Best Possible Solution

Keep this issue open. Review PR #57889 or replace it with a focused gateway fix that explicitly decides whether token-authenticated device-less WebSocket operator clients on LAN/Docker may retain `operator.write`, adds a targeted regression for that contract, and preserves the existing device-identity hardening for unrelated shared-secret clients.
