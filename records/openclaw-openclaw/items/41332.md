## Summary

Keep open. Current main does not implement the gateway-specific shutdown change, and this PR still targets real gateway, fallback, and cron surfaces, but the current head has blocking correctness issues: `process.reallyExit` does not avoid Node/C atexit, cooldown probing regresses the current transient fallback contract, and the new cron config key is not propagated to the public generated schema.

## What This Changes

The PR adds a gateway-specific runtime exit path using `process.reallyExit`, changes model-fallback cooldown probing, adds `cron.startupStaggerMs` to zod/types with timer behavior, and updates related tests plus fake-secret allowlist comments.

## Best Possible Solution

Refresh or split the work before landing: solve the GGML shutdown issue with a verified no-atexit/native cleanup strategy and macOS proof, preserve the current transient cooldown sibling-probe contract unless maintainers intentionally redesign fallback policy, and route cron restart controls through the public config/schema/docs work already tracked by #42039.

## Reproduction Assessment

Yes for the patch findings: source inspection against Node v22 proves `process.reallyExit` still reaches C `exit`, current-main tests prove the same-provider cooldown probe contract, and generated-schema lookup proves the config drift. The original GGML Metal crash still lacks a live macOS reproduction in the PR evidence.

## Solution Assessment

No. The current PR is not the best way to solve the issue because its gateway exit path does not bypass the reported C/C++ atexit crash path, its cooldown change conflicts with current behavior tests, and its cron config addition needs generated schema/docs alignment or consolidation with #42039.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Replace reallyExit with a true no-atexit path:** `src/runtime.ts:61`
  - body: Node's `process.reallyExit` is not equivalent to POSIX `_exit`: in Node v22 it calls `RunAtExit(env)` and then `Environment::Exit`, whose default handler calls C `exit()`. Since the reported GGML failure is in C/C++ atexit cleanup, this still runs the crash path while skipping JS exit listeners, so the gateway shutdown fix will not work as intended.
  - confidence: 0.91
- **[P2] Preserve transient cooldown fallback probes:** `src/agents/model-fallback.ts:452`
  - body: This makes active-cooldown attempts primary-only, so same-provider fallback siblings for transient `rate_limit`, `overloaded`, and `timeout` cooldowns are skipped until expiry. Current main has tests for those sibling probes with `allowTransientCooldownProbe: true`, so this regresses the fallback recovery contract.
  - confidence: 0.88
- **[P2] Regenerate the public config schema:** `src/config/zod-schema.ts:512`
  - body: `openclaw config schema` and schema lookup use the generated base schema, not only the zod source. Adding `startupStaggerMs` to zod/types without regenerating the public schema leaves CLI, Control UI, and config lookup unaware of the accepted key.
  - confidence: 0.84
