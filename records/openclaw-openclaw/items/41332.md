## Summary

This PR should stay open. Current main does not contain the proposed gateway runtime, cooldown-policy, or cron public-config changes, and the PR still has blocking correctness issues: the shutdown change does not prove a true no-atexit path, the cooldown change removes current same-provider transient fallback behavior, and the cron config addition is incomplete without generated schema/docs alignment or consolidation with #42039.

## What This Changes

The PR adds a gateway-specific runtime exit path using `process.reallyExit`, changes model-fallback cooldown probing, adds `cron.startupStaggerMs` to startup catch-up behavior and zod/types, and updates related tests plus fake-secret allowlist comments.

## Best Possible Solution

Refresh or split the PR into focused changes: prove a real no-atexit gateway shutdown strategy on the target macOS/GGML stack, keep transient same-provider cooldown probes unless maintainers intentionally redesign fallback policy, and expose cron catch-up controls through the generated public schema/docs path in the same direction as #42039.

## Reproduction Assessment

Yes for the PR findings: source inspection against current main and the provided PR diff reproduces the cooldown regression and generated-schema gap, and local Node plus Node source inspection reproduces the `_exit`/`reallyExit` API concern. The original GGML Metal crash itself still lacks a high-confidence live macOS reproduction in the PR record.

## Solution Assessment

No. This is not currently the best way to solve the issue because it bundles three behavior changes, the runtime exit fix is not dependency-proven, the cooldown policy conflicts with current tested recovery behavior, and the cron config surface needs generated schema/docs alignment or consolidation with #42039.
