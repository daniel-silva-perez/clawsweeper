## Summary

Keep open. Current main still does not implement the PR's three central changes: the gateway still exits through defaultRuntime/process.exit, model fallback still intentionally probes cooldowned providers/same-provider fallback models in transient cases, and cron does not expose or apply a startupStaggerMs setting for immediate missed-job catch-up. The PR is stale and should not land mechanically, but it remains a plausible implementation candidate for maintainer review after rebase and scope review.

## Best Possible Solution

Keep this PR open, but require a rebase and maintainer review before landing. The best path is to split or update the work so the gateway shutdown change has focused Node/Metal proof, the cooldown changes are reconciled with the current transient/model-scoped recovery contract, cron startup staggering includes config/docs/generated-schema/test coverage, and unrelated secret allowlist churn is removed or separately justified.

- Applying the PR mechanically could regress current model-fallback behavior: timeout/unknown-like transient cooldown probing and model-scoped sibling-model fallback are now documented and covered by tests.

- Adding cron.startupStaggerMs would create a new public config surface and should include docs/generated config/help alignment plus restart catch-up regression coverage.
- The PR contains incidental secret allowlist churn unrelated to the stated fix; current main now uses named fake constants in that test, so the rebase should remove or justify that change.
