## Summary

Keep open. Current `main` still wires both the Agents Skills panel and the top-level Skills tab to the same `skillsFilter` field, so this PR is not obsolete; the proposed patch is narrow and I found no blocking correctness or security issue in the diff.

## What This Changes

The PR adds an `agentSkillsFilter` state field and rewires the Agents Skills panel to read and write that field while leaving the top-level Skills tab on `skillsFilter`.

## Best Possible Solution

Land a rebased equivalent of the state split if independent filter memory is the intended UI behavior, ideally with a small regression test covering Agents Skills and top-level Skills filter isolation. If the team only wanted the closed click/autofill mitigation, close this PR as product-superseded rather than implemented.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: enter a filter through Agents Skills, switch to the top-level Skills tab, and current `main` still shows both surfaces reading and writing `state.skillsFilter`.

## Solution Assessment

Yes for the requested independent filter behavior: splitting the state at the app render boundary is the narrowest maintainable fix. It is separate from the already-landed input-name/autocomplete mitigation for the linked click/autofill symptom.
