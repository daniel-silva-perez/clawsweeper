## Summary

Keep open for maintainer review: current main still has the Agents Skills panel and the top-level Skills tab coupled through the same filter state, so the PR is not implemented on main. The linked click/autofill symptom was handled by a separate mitigation, which leaves a product/UX decision about whether independent filters should still land as standalone behavior.

## What This Changes

The PR adds an `agentSkillsFilter` Control UI state field and rewires the Agents Skills panel to read and write that field instead of the top-level Skills tab's `skillsFilter`.

## Best Possible Solution

If independent filters are desired, land a rebased equivalent of this small state split with focused regression coverage for the two Control UI filter surfaces. If the autofill mitigation is considered sufficient, close this PR as superseded by the closed #41605/#42293 path rather than as implemented on main.

## Reproduction Assessment

Yes. Current source is enough to reproduce the coupling path: enter a filter through the Agents Skills panel, then open the top-level Skills tab and it will read the same `state.skillsFilter`; a browser/UI regression test should cover that exact switch.

## Solution Assessment

Unclear. The proposed state split is narrow and technically sound, but the user-visible click-anywhere symptom was separately addressed with autofill attributes, so maintainer product judgment is needed before treating independent filters as the best final behavior.
