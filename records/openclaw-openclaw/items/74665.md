## Summary

Keep open. Current main still has the reported same-model OpenAI Responses replay gap: the OpenAI replay policy disables generic tool-call-id sanitization, the runner skips that sanitizer for Responses APIs, and both OpenClaw's native Responses transport and the pinned pi-ai converter can split same-model `call_id|item_id` values directly into `function_call.call_id`, `function_call.id`, and `function_call_output.call_id`.

## What This Changes

Normalize OpenAI Responses replay tool identifiers before provider replay so overlong or malformed `call_id` and `function_call.id` values are valid and consistently matched with tool results.

## Best Possible Solution

Add a Responses-specific deterministic id-normalization step at the replay/transport boundary that also applies to same-model replay, keeps assistant tool calls and matching tool results in sync, preserves valid reasoning-paired `fc_*` ids when safe, and updates transcript hygiene docs plus focused tests for overlong call ids and non-`fc_` item ids.
