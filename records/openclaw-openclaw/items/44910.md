## Summary

Keep open. Current main still does not recognize top-level FastAPI/Codex-style `detail` JSON payloads as provider errors, so the reported raw `{"detail":...}` shape can still bypass the existing sanitizer path. The issue is also paired with open, unmerged PR #45216, which uses closing syntax for this issue, so cleanup should wait for that PR to land, be replaced, or be explicitly closed.

## What This Changes

Prevent unconfigured OpenAI/Codex provider errors, especially FastAPI-style `detail` JSON payloads, from being forwarded verbatim to Telegram or other user chat surfaces.

## Best Possible Solution

Land or replace the narrow parser and test fix so top-level `detail` payloads become friendly provider errors, then verify the separate unconfigured-provider trigger path and add credential/probe suppression only if that trigger still exists.
