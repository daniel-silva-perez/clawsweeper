## Summary

Keep open. Current `main` still does not treat a top-level FastAPI/Codex-style `detail` string as an API error payload, so the reported raw JSON shape can still bypass the friendly formatting path. The issue is also paired with open, unmerged closing PR #45216, so cleanup should wait for that PR to land, be replaced, or be explicitly closed.

## What This Changes

Suppress or sanitize raw provider error payloads, especially top-level FastAPI/Codex `detail` JSON, so unconfigured OpenAI/Codex errors are not sent to Telegram or other user chat surfaces.

## Best Possible Solution

Land or replace the narrow parser and regression-test fix so top-level `detail` payloads are formatted as provider errors, then verify whether unconfigured Codex/provider probes can still fire and add credential-gated suppression only if that trigger remains.
