## Summary

Keep open. Current main still does not recognize top-level FastAPI/Codex-style `detail` error payloads, so the reported raw JSON can still bypass the shared formatter/sanitizer path; the issue is also linked to open PR #45216 with closing syntax, so it should stay open until that PR or a replacement lands.

## What This Changes

The issue requests that Codex/OpenAI provider errors from unconfigured providers stop leaking as raw JSON into Telegram/user chat, especially payloads shaped like `{ "detail": "..." }`.

## Best Possible Solution

Land or replace the focused parser and regression-test fix so shared assistant error formatting recognizes top-level `detail` payloads, then separately verify whether unconfigured Codex/OpenAI provider probes can still happen and need credential-gating.

## Reproduction Assessment

Yes for the reported chat-visible parser leak: current code statically shows `{ "detail": "..." }` is not classified as a raw API error and can fall through as short raw text. The original no-credentials provider-probe trigger remains unclear without a live trace.

## Solution Assessment

Yes as a first fix: the narrow shared parser/sanitizer regression path is the maintainable place to stop this raw payload shape from leaking across channels. Provider credential-gating should be handled as a separate follow-up if reproduced after the parser fix.
