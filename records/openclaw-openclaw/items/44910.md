## Summary

Keep open because current main still has the reported parser gap: a top-level FastAPI/Codex-style `detail` payload is not recognized as an API error payload and is not parsed into formatted assistant error text. The issue is also paired with open, unmerged PR #45216 using closing syntax, so it should stay open until that PR or a replacement lands.

## What This Changes

The issue requests that OpenAI/Codex provider errors, especially top-level `{"detail":"..."}` payloads from unconfigured Codex/OpenAI paths, stop leaking verbatim into Telegram user chat.

## Best Possible Solution

Land or replace the focused parser and regression-test fix from #45216, then verify the runtime provider-auth boundary so unconfigured Codex/OpenAI probes do not run and any escaped provider error is formatted or suppressed before Telegram delivery.
