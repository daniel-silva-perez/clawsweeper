## Summary

Keep open. Current main still misses top-level string `detail` error payloads, and the linked closing PR remains unmerged, so the reported chat leak is still a valid maintainer follow-up item.

## What This Changes

The issue asks OpenClaw to stop forwarding unconfigured OpenAI/Codex provider error JSON, especially FastAPI/Codex-style top-level `detail` payloads, into Telegram or other user chat surfaces.

## Best Possible Solution

Land or replace the focused parser/sanitizer fix from PR #45216 so top-level string `detail` payloads are classified, formatted, and covered by regression tests; then separately verify whether unconfigured Codex/OpenAI probes still need credential gating after the raw JSON leak is fixed.

## Reproduction Assessment

Yes for the parser leak: current main statically shows `{"detail":"..."}` is neither classified as a raw API error nor extracted as a message, so it can fall through as short raw text. The original unconfigured-provider trigger remains unclear without a live runtime trace.

## Solution Assessment

Yes as a first fix: extending the shared API error parser/formatter is the narrow maintainable place to stop this payload shape across chat surfaces. Credential gating for unconfigured providers should be a follow-up only if it still reproduces after that parser fix.
