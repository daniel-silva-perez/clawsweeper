## Summary

Keep open. Current main still has the reported parser gap for top-level FastAPI/Codex-style `detail` JSON, so the formatter and Telegram-facing sanitizer can still treat that payload as ordinary short text. The issue is also paired with open, unmerged PR #45216 using closing syntax, so the right next step is maintainer review of that PR or a replacement, not closing this issue early.

## What This Changes

Report asks OpenClaw to prevent unconfigured OpenAI/Codex provider errors, especially FastAPI-style `detail` JSON, from being forwarded into Telegram or other user chat surfaces.

## Best Possible Solution

Land or replace the focused parser and regression-test fix so shared assistant error formatting recognizes top-level `detail` payloads, then verify the provider-auth boundary so unconfigured Codex/OpenAI probes do not create chat-visible errors.
