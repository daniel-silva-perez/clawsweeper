## Summary

Keep open. Current main still does not classify top-level string `detail` API error payloads, so the reported raw JSON chat leak remains valid; the open closing PR is the active implementation path and should be reviewed, updated, merged, or replaced.

## What This Changes

The issue asks OpenClaw to stop forwarding unconfigured OpenAI Codex/FastAPI-style error payloads, especially top-level `detail` JSON, into Telegram/user chat.

## Best Possible Solution

Land or replace the focused parser/sanitizer fix from PR #45216 with regression coverage for top-level string `detail` payloads, then separately verify whether unconfigured Codex/OpenAI probing still needs credential gating after the raw JSON leak is fixed.

## Reproduction Assessment

Yes for the parser leak: current main statically shows `{"detail":"..."}` is not classified as a raw API error and short unclassified assistant errors can be returned raw. The deeper unconfigured-provider trigger remains unclear without a live Raspberry Pi/Telegram trace.

## Solution Assessment

Yes as a first fix: extending the shared API error parser and formatter is the narrow maintainable place to stop this payload shape across chat surfaces. Credential gating for unconfigured providers should be handled as a follow-up only if it still reproduces after the parser fix.
