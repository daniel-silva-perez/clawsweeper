## Summary

Keep open. Current main still does not recognize top-level FastAPI/Codex-style `detail` error payloads, so the reported raw JSON shape can bypass the shared formatter/sanitizer path; open PR #45216 remains the active closing implementation path and should be reviewed, landed, or replaced before this issue is closed.

## What This Changes

Request to stop Codex/OpenAI `detail`-style provider errors from appearing verbatim in Telegram/user chat, especially when the user has not configured Codex/OpenAI.

## Best Possible Solution

Land or replace the focused shared parser/sanitizer fix so top-level string `detail` payloads are classified, formatted, and covered by regression tests, then separately verify whether unconfigured Codex/OpenAI provider probes still need credential gating.

## Reproduction Assessment

Yes for the parser leak: current main statically shows `{"detail":"..."}` is not classified as a raw API error and can fall through as short raw text. The original no-credentials provider-probe trigger remains unclear without a live runtime trace.

## Solution Assessment

Yes as a first fix: the shared assistant error parser/formatter is the narrow maintainable place to stop this raw payload shape across chat surfaces. Provider credential-gating should be handled as a follow-up only if it still reproduces after the parser fix.
