## Summary

Keep open. Current main still does not recognize a top-level FastAPI/Codex-style `detail` string as a raw API error payload, so the reported short JSON can still bypass friendly formatting; the issue is also paired with open, unmerged PR #45216.

## What This Changes

The issue asks OpenClaw to prevent OpenAI/Codex provider errors from unconfigured providers, especially top-level `detail` JSON payloads, from being forwarded verbatim to Telegram/user chat.

## Best Possible Solution

Land or replace the narrow parser and regression-test fix so top-level `detail` payloads are formatted as provider errors, then separately verify whether unconfigured Codex/provider probes can still fire and add credential-gated suppression if that trigger remains.
