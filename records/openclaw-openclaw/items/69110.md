## Summary

Keep this issue open. Current main still exposes resolved provider/model identity to usage, diagnostics, model-call hooks, and response-prefix plumbing, but not to the awaited outbound `message_sending` delivery hook that can rewrite or cancel messages. The Telegram path named in the report has the same gap, and related PRs are still open implementation candidates rather than merged fixes.

## Best Possible Solution

Keep this issue open and coordinate one maintainer-approved provenance design. The best path is to thread resolved provider/model identity from the agent run/fallback result into final reply assembly and the awaited `message_sending` delivery hook, expose additive typed fields for plugins, cover central outbound plus Telegram/direct paths with tests, update plugin-hook docs, and reconcile overlap with #39761, #65149, and #44011 before merging competing APIs.
