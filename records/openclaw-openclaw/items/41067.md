## Summary

Keep this PR open. Current main still has the dashboard reconnect busy-state gap, but the PR branch is unmerged, based on an older main, and still has two concrete P2 correctness issues in recovered-run selection and restart-sentinel fallback handling.

## What This Changes

The PR extends the gateway hello snapshot with active chat run metadata, updates Control UI chat reconnect/navigation/busy-state behavior, adds shutdown/restart interruption transcript notices, and syncs generated protocol and secrets artifacts.

## Best Possible Solution

Land a rebased version that keeps the additive activeChatRuns protocol/schema/generated-client contract, deterministically recovers the correct active run for a session, preserves busy controls while abort is available, and keeps restart/shutdown notices durable without bypassing fallback delivery semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Choose the recovered run deterministically:** `ui/src/ui/app-gateway.ts:126-136`
  - body: `resolveActiveChatRunForSession` returns the first matching entry, but the server can expose more than one active run for the same session. Recovering an older run makes later events from the newer run fail the run-id check, leaving the UI with stale busy state or stream text. Prefer the existing host run when it is still active, or choose the newest `startedAtMs`.
  - confidence: 0.9
- **[P2] Return false when transcript injection fails:** `src/gateway/server-restart-sentinel.ts:32-39`
  - body: `injectRestartNoticeIntoSession` can throw from transcript path creation or file access, but callers only enqueue the fallback system event when the helper returns `false`. If it throws after the sentinel is consumed, the restart notice is lost. Catch helper errors and return `false` so the existing fallback runs.
  - confidence: 0.92
