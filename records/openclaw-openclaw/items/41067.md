## Summary

Keep this PR open for maintainer review. Current main still lacks the active-run snapshot/reconnect recovery needed for the reported dashboard busy-state loss, but the proposed patch has concrete correctness issues in active-run selection and restart-notice fallback handling before it should merge.

## What This Changes

The PR extends gateway hello snapshots with active chat run metadata, updates Control UI reconnect/navigation/busy-state handling, adds shutdown/restart interruption transcript notices, and syncs generated protocol and secrets artifacts.

## Best Possible Solution

Land a rebased implementation that keeps the additive activeChatRuns protocol/schema/generated-client contract, recovers the correct active run deterministically, preserves busy/queue controls while abort is available, and writes durable restart/shutdown notices without bypassing existing fallback delivery semantics.

## Reproduction Assessment

Yes. The linked dashboard issue #41064 gives focused steps, and current main supports the failure path: hello snapshots do not expose active runs and the Control UI hello handler clears the busy latch after reconnect.

## Solution Assessment

No, not as currently proposed. The direction is maintainable, but the patch needs deterministic active-run selection and exception-safe restart-notice fallback before it is the best fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Choose the recovered run deterministically:** `ui/src/ui/app-gateway.ts:126-136`
  - body: `resolveActiveChatRunForSession` returns the first matching entry, but the server can expose multiple active runs for the same session. Recovering an older run makes later events from the newer run fail the run-id check, leaving the UI with stale busy state or stream text. Prefer the existing host run when it is still active, or choose the newest `startedAtMs`.
  - confidence: 0.9
- **[P2] Return false when transcript injection fails:** `src/gateway/server-restart-sentinel.ts:32-39`
  - body: `injectRestartNoticeIntoSession` can throw from transcript path creation or file access, but callers only enqueue the fallback system event when the helper returns `false`. If it throws after the sentinel is consumed, the restart notice is lost. Catch helper errors and return `false` so the existing fallback runs.
  - confidence: 0.92
