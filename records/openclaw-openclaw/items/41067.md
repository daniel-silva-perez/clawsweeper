## Summary

Keep this PR open. Current main still lacks the active chat-run snapshot/reconnect recovery behavior, and this PR remains the concrete implementation candidate, but it should not merge as-is because the latest patch has two P2 correctness defects and needs a conflict-aware rebase over newer gateway startup/restart-sentinel code.

## What This Changes

The PR adds gateway and Control UI changes to expose active chat runs in the hello snapshot, recover dashboard chat busy state after reconnect/tab return, keep abortable runs busy, preserve last active chat navigation, and inject restart-interruption notices into transcripts.

## Best Possible Solution

Keep this PR as the implementation candidate, rebase it onto current main, and land a version that keeps the additive activeChatRuns protocol/schema/generated-client contract, deterministically recovers the right active run, preserves busy controls while abort is available, and keeps restart/shutdown notices durable without dropping fallback delivery semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Choose the recovered run deterministically:** `ui/src/ui/app-gateway.ts:126-136`
  - body: `resolveActiveChatRunForSession` returns the first matching entry, but the server can expose more than one active run for the same session. Recovering an older run makes later events from the newer run fail the run-id check, leaving the UI with stale busy state or stream text. Prefer the existing host run when it is still active, or choose the newest `startedAtMs`.
  - confidence: 0.9
- **[P2] Return false when transcript injection throws:** `src/gateway/server-restart-sentinel.ts:32-39`
  - body: `injectRestartNoticeIntoSession` can throw from transcript path creation or file access, but callers only enqueue the fallback system event when the helper returns `false`. If it throws after the sentinel is consumed, the restart notice is lost. Catch helper errors and return `false` so the existing fallback runs.
  - confidence: 0.92
