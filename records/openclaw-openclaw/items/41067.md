## Summary

Keep this PR open: current main still has the Dashboard chat reconnect/restart reliability gaps the PR targets, but the branch remains an open contributor implementation with two concrete P2 correctness issues and a stale base that needs maintainer-guided patch review before merge.

## What This Changes

The PR adds Gateway and Control UI changes to expose active chat runs in the hello snapshot, recover chat busy state across reconnects, keep abortable runs busy, preserve last active chat navigation, and add restart-interruption transcript notices.

## Best Possible Solution

Land a rebased version that keeps the additive activeChatRuns protocol/schema/generated-client contract, deterministically selects the recovered active run, preserves busy controls while abort is available, and keeps restart/shutdown notices durable without bypassing fallback delivery semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Choose the recovered run deterministically:** `ui/src/ui/app-gateway.ts:126-136`
  - body: `resolveActiveChatRunForSession` returns the first matching entry, but the server can expose more than one active run for the same session. Recovering an older run makes later events from the newer run fail the run-id check, leaving the UI with stale busy state or stream text. Prefer the existing host run when it is still active, or choose the newest `startedAtMs`.
  - confidence: 0.9
- **[P2] Return false when transcript injection fails:** `src/gateway/server-restart-sentinel.ts:32-39`
  - body: `injectRestartNoticeIntoSession` can throw from transcript path creation or file access, but callers only enqueue the fallback system event when the helper returns `false`. If it throws after the sentinel is consumed, the restart notice is lost. Catch helper errors and return `false` so the existing fallback runs.
  - confidence: 0.92
