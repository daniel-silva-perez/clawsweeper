## Summary

Keep this PR open. Current main still lacks the central dashboard reconnect recovery and restart/shutdown transcript-notice behavior, and the same-author dashboard issue #41064 remains open with this PR as its implementation candidate. The PR is useful, but it should not merge as-is because the latest branch is dirty against main and still has concrete correctness findings around active-run selection and restart-sentinel fallback delivery.

## What This Changes

The PR adds gateway and Control UI changes to expose active chat runs in the hello snapshot, recover dashboard chat busy state after reconnect/tab return, keep abortable runs in busy mode, preserve last active chat navigation, and inject restart-interruption notices into transcripts.

## Best Possible Solution

Keep this PR open as the implementation candidate, but rebase it onto current main and fix the active-run selection and restart-sentinel fallback bugs before maintainer review. The desired end state is an additive activeChatRuns snapshot contract, deterministic Control UI recovery for the right active run, busy controls while a run is abortable, last-active chat navigation, and restart/shutdown notices that use the existing transcript injection contract with safe fallback behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Choose the recovered run deterministically:** `ui/src/ui/app-gateway.ts:126-136`
  - body: `resolveActiveChatRunForSession` returns the first matching entry, but the server tracks active runs by runId and can expose more than one run for the same session. Recovering an older run id makes later events from the newer run fail the run-id check, leaving the UI on stale busy state or stale stream text. Prefer the existing host run when still active, or choose the newest `startedAtMs`.
  - confidence: 0.9
- **[P2] Return false when transcript injection throws:** `src/gateway/server-restart-sentinel.ts:32-39`
  - body: `injectRestartNoticeIntoSession` can throw from transcript path creation or file access, but callers only enqueue the fallback system event when the helper returns `false`. If it throws after the sentinel is consumed, the restart notice is lost. Catch helper errors and return `false` so the existing fallback runs.
  - confidence: 0.92
