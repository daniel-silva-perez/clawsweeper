## Summary

Keep this PR open. Current main still lacks the PR's central dashboard reconnect recovery: the hello snapshot does not expose active chat runs, the Control UI clears chat run state after hello, and shutdown still clears run state without aborting active chat runs or adding a durable transcript notice. The PR is also paired with still-open same-author issue #41064, which it is intended to close, so it should be reviewed or updated rather than treated as obsolete.

## Best Possible Solution

Keep this PR open as the implementation candidate for #41064. The best path is to rebase it onto current main, preserve the additive activeChatRuns snapshot/reconciliation design, fix the remaining active-run selection and transcript-injection fallback issues, keep transcript writes aligned with the SessionManager parentId contract, regenerate protocol and Swift artifacts, and run the targeted gateway/UI/protocol checks before maintainer review.

## Risks / Open Questions

- The PR is stale against current main and touches protocol/generated client surfaces, so it needs a rebase plus protocol artifact validation before merge.
- Review comments in the provided context still identify real correctness issues: choose the newest or already-tracked active run for a session, and fall back to a system event when restart-sentinel transcript injection throws.
- The shutdown transcript path is security-sensitive filesystem code; maintainers should verify restrictive permissions, idempotency, path resolution, and SessionManager-backed transcript appends.
- The PR may help #38836 but does not appear to fully implement broad all-channel restart notification for every interrupted external session, so that broader issue should stay separately scoped unless maintainers decide otherwise.
