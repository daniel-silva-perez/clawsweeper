## Summary

Keep open. Current main still lacks dashboard active-run recovery, but this PR should not merge as-is because the branch has concrete correctness issues in recovered run selection and restart-notice fallback handling, and it predates substantial current shutdown/restart-sentinel changes.

## What This Changes

The PR extends gateway hello snapshots with active chat run metadata, updates Control UI reconnect/navigation/busy-state handling, adds shutdown/restart interruption transcript notices, and syncs generated protocol plus secrets artifacts.

## Best Possible Solution

Land a rebased implementation that keeps the additive activeChatRuns protocol/schema/generated-client contract, recovers the correct active run deterministically, preserves busy/queue controls while abort is available, and integrates restart/shutdown notices with the current restart-sentinel and transcript append paths.

## Reproduction Assessment

Yes. The linked dashboard issue #41064 gives focused steps, and current main source confirms hello snapshots do not expose active runs while the Control UI hello handler clears the busy latch after reconnect.

## Solution Assessment

No, not as currently proposed. The direction is maintainable, but the patch needs deterministic active-run selection, exception-safe restart-notice fallback, and a careful rebase onto the current shutdown/restart-sentinel code before it is the best fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Choose the recovered run deterministically:** `ui/src/ui/app-gateway.ts:126-136`
  - body: The server can expose multiple active runs for the same session because active runs are tracked by runId. Returning the first matching snapshot entry can recover an older run; later events for the newer run then fail the run-id guard and leave stale busy or stream state. Prefer the existing host run if still active, otherwise choose the newest startedAtMs.
  - confidence: 0.9
- **[P2] Fallback when transcript injection fails:** `src/gateway/server-restart-sentinel.ts:22-39`
  - body: injectRestartNoticeIntoSession can throw from session lookup, transcript path resolution, or file creation, but callers only enqueue the system-event fallback when this helper returns false. Once the sentinel is consumed, that throw drops the restart notice. Catch helper errors and return false so the existing fallback still runs.
  - confidence: 0.92
