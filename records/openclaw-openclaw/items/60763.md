## Summary

Keep this PR open. Current main at c0ec58f4b6906ad06fd4c6f265d43be053b2d193 still does not expose `expiresAtMs` or `bindSession` on the Discord component spec, parser, message-tool schema, or build path, so the PR is not obsolete. It needs maintainer/security review before merge because the proposed unbound routing and caller-controlled expiry affect Discord interaction authorization and registry lifetime.

## Best Possible Solution

Keep the PR open as the active implementation candidate, but require a maintainer review pass that either adds safe authorization semantics for unbound components, bounds `expiresAtMs`, updates Discord docs/tool schema/tests, and then merges, or replaces it with a narrower maintainer-authored Discord implementation covering the same behavior safely.

## Risks / Open Questions

- The proposed `bindSession=false` changes Discord interaction routing from a stored session/agent/account context to the channel's resolved route; without a required allowlist or originating-user check, shared buttons/forms can change the authorization model for callbacks.
- The proposed `expiresAtMs` accepts caller-supplied finite numbers and stores them as registry expiry timestamps; merging should enforce a maximum TTL/window or another explicit retention policy.
- This is bundled Discord channel behavior, not a ClawHub/core-scope cleanup item; closing it now would drop a focused implementation candidate while current main still lacks the requested fields.
