## Summary

Keep this open. Current main still filters web_fetch provider candidates to bundled plugin origins before explicit/runtime provider matching, and the provided GitHub context shows an open focused PR (#74962) with closing syntax, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

External plugins that declare and register webFetchProviders should be resolved by web_fetch when they are explicitly configured or runtime-selected.

## Best Possible Solution

Review and land #74962 or an equivalent narrow fix that lets non-sandboxed, explicitly configured or runtime-selected web_fetch providers resolve across active non-bundled plugin origins while preserving the sandboxed bundled-provider restriction and existing SSRF/redirect protections.

## Reproduction Assessment

Yes. The issue has a high-confidence source-level reproduction on current main: external provider candidates are filtered out by the bundled-origin resolver, then the Readability-disabled branch throws the reported no-provider error when no bundled fallback is available.

## Solution Assessment

Yes. The narrow maintainable solution is to adjust provider resolution and add regression coverage for explicit/runtime non-bundled web_fetch providers, rather than documenting the already-exposed plugin contract as bundled-only.
