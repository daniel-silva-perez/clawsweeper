## Summary

Keep open. Current main still filters `web_fetch` provider resolution through bundled candidates before matching an explicit or runtime-selected provider, and an open focused PR (#74962) already uses closing syntax for this issue, so the issue should stay open until that PR or an equivalent fix lands.

## What This Changes

External plugins that declare and register `webFetchProviders` should be usable by `web_fetch` when explicitly configured, including when Readability is disabled.

## Best Possible Solution

Land #74962 or an equivalent narrow fix that lets non-sandboxed, explicitly configured or runtime-selected `web_fetch` providers resolve across active non-bundled plugin origins while keeping sandboxed fetch restricted to bundled providers.

## Reproduction Assessment

Yes. The issue body's reproduction has a source-level reproduction on current main: external provider candidates are filtered out by bundled-origin resolution, and the Readability-disabled branch then throws the reported no-provider error.

## Solution Assessment

Yes. The maintainable path is a focused resolver change plus regression tests for non-bundled explicit/runtime provider selection, rather than documenting the contract as bundled-only after exposing `webFetchProviders` and `registerWebFetchProvider`.
