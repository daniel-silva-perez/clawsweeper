## Summary

Keep open. Current main still resolves `web_fetch` provider definitions from bundled-origin candidates before explicit or runtime-selected provider matching, and the linked implementation PR #74962 is still open with closing syntax for this issue.

## What This Changes

External plugins that declare and register `webFetchProviders` should be usable by `web_fetch` when explicitly configured, including when Readability is disabled.

## Best Possible Solution

Implement the narrow resolver change so non-sandboxed `web_fetch` can resolve explicitly configured or runtime-selected providers from active non-bundled plugin origins, with regression tests for external provider selection and sandboxed bundled-only behavior.

## Reproduction Assessment

Yes. The issue body's steps map directly to current source: external provider candidates are filtered out by bundled-origin resolution, and the Readability-disabled path then throws the reported no-provider error.

## Solution Assessment

Yes. The best path is the focused resolver change already represented by PR #74962 or an equivalent patch, rather than documenting `webFetchProviders` as bundled-only after exposing the plugin contract.
