## Summary

Keep this issue open. Current main still has no typed, schema-backed, or documented config-level default task notification policy; ACP and Task Flow still use `done_only` when no explicit `notifyPolicy` is supplied, and the now-closed #73905 only covered the narrower cron-spawned subagent notification path rather than this broader config contract.

## What This Changes

Add a config-level default task notification policy so users can suppress or tune raw background task completion messages while preserving explicit per-task overrides.

## Best Possible Solution

Add a backwards-compatible core config surface under a maintainer-approved `agents.defaults.tasks` or equivalent shape, resolve the effective policy before ACP, subagent, and Task Flow records are created, preserve current behavior when unset, keep explicit task/flow `notifyPolicy` authoritative, and document the behavior on the public tasks, subagents, and config pages.

## Reproduction Assessment

Yes. This is a feature/config gap rather than a live runtime bug, and the high-confidence check is static: the strict agent-defaults schema/types/docs expose no task notification default while current ACP tests still prove default terminal notices are sent when no `notifyPolicy` is supplied.

## Solution Assessment

Unclear for implementation because there is no patch and the public API shape is the main unresolved question. The narrow maintainable path is to add one coherent config contract across task registry, ACP, subagent, and Task Flow creation rather than adding runtime-specific partial knobs.
