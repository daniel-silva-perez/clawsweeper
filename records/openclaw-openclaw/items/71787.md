## Summary

Keep this PR open. Current main still only accepts the narrower Feishu Schema 2.0 `context.open_chat_id` callback shape; it still does not normalize nested Schema 2.0 operator identity from `operator.user_id.{open_id,user_id,union_id}`. The PR remains a focused fix for that remaining Feishu card-action gap, and maintainer discussion already called it the better base than the closed narrower PR #71692.

## Best Possible Solution

Keep this PR open and review or land the nested Schema 2.0 identity normalization on top of current main. The best implementation path is to extract `operator.user_id.{open_id,user_id,union_id}` and equivalent event-level nested identity fields, preserve existing `context.open_chat_id` handling and malformed-payload guards, keep the lifecycle regression test for the nested callback shape, and keep the changelog attribution without changing outbound card schema, routing, permissions, or command semantics.
