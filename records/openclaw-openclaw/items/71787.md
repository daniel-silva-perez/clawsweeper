## Summary

Keep this PR open. Current main still accepts only flat Feishu card-action identity strings, so Schema 2.0 callbacks that provide the operator identity under `operator.user_id.*` can still be dropped before dispatch; the PR targets that missing parser path, adds focused lifecycle coverage, and addresses the prior changelog and dead-guard feedback.

## What This Changes

The PR updates the Feishu card-action parser to read nested Schema 2.0 identity fields, adds lifecycle coverage for that callback shape, and expands the existing changelog entry with the new attribution.

## Best Possible Solution

Land this narrow Feishu parser, lifecycle-test, and changelog update after normal maintainer validation, keeping instant card-update behavior in the separate #71677 enhancement thread.

## Reproduction Assessment

Yes. A `card.action.trigger` payload whose only operator open ID is `operator.user_id.open_id` leaves `openId` undefined on current main, causing the parser to return null and the monitor handler to drop the callback before dispatch.

## Solution Assessment

Yes. Normalizing Schema 2.0 identity fields at the Feishu monitor parser boundary is the narrowest maintainable fix because downstream card-action handling already consumes the normalized `FeishuCardActionEvent` and the PR keeps the existing token/openId/tag/action-value validation.
