## Summary

Keep this issue open. Current main and the latest release still validate only the first Docker apt key fingerprint before dearmoring the whole downloaded key file, and open PR #74254 is the direct implementation candidate that adds the requested single-primary-key guard and focused Dockerfile test coverage.

## What This Changes

The issue requests hardening Dockerfile Docker apt key verification so the downloaded key file must contain exactly one public key instead of validating only the first fingerprint before dearmoring.

## Best Possible Solution

Land PR #74254 or an equivalent narrow Dockerfile change that rejects anything except exactly one primary pub key before fingerprint comparison and before gpg --dearmor, with focused test coverage for the ordering and rejection behavior.

## Reproduction Assessment

Yes. The current-main reproduction is a static Dockerfile path: a key file with a matching first fpr record and an extra primary key would pass the awk ... exit fingerprint check, then gpg --dearmor would trust the whole file.

## Solution Assessment

Yes. Requiring exactly one primary pub record before using the first fpr and before dearmoring is the narrowest maintainable fix for the reported trust-root gap; PR #74254 implements that shape and should be reviewed rather than closing the issue early.
