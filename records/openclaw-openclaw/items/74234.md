## Summary

Keep this open. Current main still validates only the first fingerprint record from the Docker apt signing-key file and then dearmors the entire file, so the reported multi-primary-key gap remains present; the linked open PR #74254 is the direct implementation candidate and should be reviewed rather than closing this issue early.

## What This Changes

The issue asks OpenClaw to reject Docker apt signing-key files that contain more than one primary public key before fingerprint validation and dearmoring.

## Best Possible Solution

Land PR #74254 or an equivalent narrow Dockerfile hardening change that rejects anything other than exactly one primary `pub` key before comparing fingerprints and before dearmoring, with focused coverage for both the official Docker key and a multi-primary-key rejection case.

## Reproduction Assessment

Yes. The reproduction is source-level and live-checkable: duplicate the Docker Debian armored key stream and current main's first-`fpr` awk logic still returns the expected fingerprint while the stream contains two primary `pub` keys.

## Solution Assessment

Yes. The narrow maintainable fix is to validate the key-file shape before trusting the dearmored keyring; counting exactly one primary `pub` key before the existing fingerprint comparison preserves the current official Docker path while closing the reported gap.
