## Summary

Keep this issue open. Current main still checks only the first Docker apt key fingerprint and then dearmors the whole downloaded key file, while PR #74254 remains the open implementation candidate that directly addresses the reported gap.

## What This Changes

The issue asks OpenClaw to reject Docker apt signing-key files unless they contain exactly one public key before trusting the file in the Dockerfile build path.

## Best Possible Solution

Land PR #74254 or an equivalent narrow Dockerfile hardening change that rejects anything other than one primary `pub` key before fingerprint comparison and dearmor, with the focused Dockerfile ordering test preserved.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main reads the first `fpr` record and then dearmors the entire key file, so a multi-primary-key file can bypass the intended single-key trust assumption.

## Solution Assessment

Yes, the linked PR’s direction appears to be the narrowest maintainable fix: count primary `pub` records before comparing the fingerprint and before `gpg --dearmor`, while keeping subkey fingerprints allowed.
