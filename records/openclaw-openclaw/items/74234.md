## Summary

Keep this issue open. Current main still verifies only the first Docker apt key fingerprint and then dearmors the whole downloaded key file, while open PR #74254 directly addresses the remaining supply-chain hardening gap.

## What This Changes

The issue asks the Docker image build to reject Docker apt signing-key files with more than one primary public key instead of validating only the first fingerprint.

## Best Possible Solution

Land PR #74254 or an equivalent narrow Dockerfile hardening change that rejects any Docker apt key file with other than exactly one primary `pub` key before fingerprint comparison and dearmor, while preserving subkey fingerprints and the focused ordering test.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: a key file whose first `fpr` matches the pinned fingerprint but also contains an extra primary key would pass the current `awk ... exit` comparison, then `gpg --dearmor` would import the full file into the apt keyring.

## Solution Assessment

Yes. Counting primary `pub` records before comparing the first fingerprint and before dearmor is the narrowest maintainable fix because GnuPG distinguishes primary public keys from subkeys in `--with-colons` output.
