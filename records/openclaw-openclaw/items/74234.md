## Summary

Keep this issue open. Current main and the latest release still compare only the first `fpr` record from Docker's apt key file, then dearmor the whole file into the apt keyring; open PR #74254 is the direct implementation candidate for the remaining supply-chain hardening gap.

## What This Changes

Require Dockerfile Docker apt key validation to reject signing-key files that contain anything other than exactly one primary public key before fingerprint comparison and dearmor.

## Best Possible Solution

Land PR #74254 or an equivalent narrow Dockerfile hardening change that counts primary `pub` records before fingerprint comparison and dearmor, keeps subkeys valid, and preserves the focused Dockerfile ordering test.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: an armored key file with the expected key first and an extra primary key afterward would pass the current first-`fpr` comparison while `gpg --dearmor` still trusts the full file.

## Solution Assessment

Yes. The linked PR's single-primary-`pub` guard before fingerprint comparison and dearmor is the narrowest maintainable fix, and GnuPG's own colon-output semantics support distinguishing primary keys from subkeys.
