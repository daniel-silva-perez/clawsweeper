## Summary

Keep this open. Current main still validates only the first `fpr` record from Docker's downloaded apt key file and then dearmors the entire file into the apt keyring, while open PR #74254 remains the direct implementation candidate for this supply-chain hardening gap.

## What This Changes

The issue asks Docker apt GPG key validation to reject downloaded signing-key files that contain more than one public key before trusting the file for apt package verification.

## Best Possible Solution

Land PR #74254 or an equivalent narrow Dockerfile hardening change that rejects anything other than one primary `pub` key before fingerprint comparison and `gpg --dearmor`, while preserving subkey fingerprints and focused Dockerfile ordering coverage.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main reads the first `fpr` record and then dearmors the entire downloaded key file, so a multi-primary-key file can violate the intended single-key trust assumption.

## Solution Assessment

Yes. The requested direction is the narrowest maintainable fix: enforce exactly one primary `pub` record before comparing the expected fingerprint and before writing the dearmored keyring; current main has not applied that fix yet.
