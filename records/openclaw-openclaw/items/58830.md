## Summary

Keep open. Current main has useful pieces of the requested transparency and recovery story: `/status` and `/context`, opt-in compaction notices, compaction event hooks, session checkpoint branch/restore APIs, and plugin/context-engine seams. It still does not implement the central feature request: previewing the summary/removals before applying compaction, user approve/reject/edit confirmation modes, or the proposed global/per-agent `contextCompaction` controls. The related reports in the provided context are adjacent, not canonical superseders.

## Best Possible Solution

Keep this issue open and design a core compaction UX/state machine: per-agent/global modes such as auto, notify-only, and require-confirmation; a redacted pre-apply preview of summary plus affected messages; channel/UI approval with timeout/fallback behavior; and reuse of existing checkpoints, restore/branch APIs, and plugin hooks. Custom compaction strategies can remain plugin-owned, but the user-visible confirmation and recovery contract belongs in core.

## Risks / Open Questions

- The feature is broad and crosses runtime, session storage, channel delivery, Control UI/TUI, and provider-specific compaction behavior; it needs product/design narrowing before implementation.
- A confirmation flow must handle context-overflow recovery carefully, because waiting for approval can itself block a reply when the current prompt is already too large.
- Compaction previews would need redaction and size limits so they do not expose secrets or reintroduce the token pressure they are trying to solve.
