## Summary

Keep this PR open. Current main only replaces the consent card with a FileInfoCard on successful accept when a stored consent-card activity id exists; it still does not delete or replace the stale card on decline, expired/not-found, conversation mismatch, or upload-failure paths. The PR also targets older inlined handler code, while current main routes file-consent invokes through `file-consent-invoke.ts`, so the useful work needs a rebase rather than closure.

## Best Possible Solution

Keep this PR open and ask for a rebase onto current main. The fix should live in `extensions/msteams/src/file-consent-invoke.ts`, use the stored consent-card activity id when available, preserve or deliberately replace the current successful-accept `updateActivity` behavior, add best-effort deletion cleanup for decline, expired/not-found, conversation mismatch, and upload failure, and add focused tests for those paths plus delete failure resilience.
