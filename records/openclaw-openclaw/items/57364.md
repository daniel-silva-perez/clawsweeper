## Summary

Do not close this PR. Current main partially overlaps by replacing the consent card with a FileInfoCard after a successful accept, but it does not implement the PR’s broader cleanup request for decline, expired/not-found, conversation mismatch, or upload-failure paths. The PR is also stale against current main because the file-consent invoke logic now lives in `file-consent-invoke.ts`, not the older inlined handler shown in the PR diff.

## Best Possible Solution

Keep this PR open or ask the contributor to rebase it onto current main. The best implementation belongs in `extensions/msteams/src/file-consent-invoke.ts`, using the existing stored consent-card activity id where possible, deciding whether success should keep the current in-place FileInfoCard replacement or delete first, and adding focused tests for decline, expired/not-found, conversation mismatch, upload failure, and best-effort deletion failure.
