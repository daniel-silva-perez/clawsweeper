## Summary

Keep this PR open. Current main only replaces the consent card with a FileInfoCard on successful accept when a stored consent-card activity id exists; decline, expired/not-found, conversation mismatch, and upload-failure paths still do not delete the stale card. The PR’s idea remains useful, and the discussion shows the author addressed the upload-failure gap on the branch, but the work is stale against current main because file-consent invokes now route through `file-consent-invoke.ts` rather than the older inlined handler shape in `monitor-handler.ts`.

## Best Possible Solution

Keep this PR open and ask for a rebase onto current main. The implementation should move into `extensions/msteams/src/file-consent-invoke.ts`, prefer the stored `consentCardActivityId` from the pending upload/FS record after conversation validation, preserve or intentionally replace the current successful-accept `updateActivity` behavior, add best-effort cleanup for decline, expired/not-found, conversation mismatch, and upload failure, and add focused tests with `deleteActivity` assertions plus delete-failure resilience.
