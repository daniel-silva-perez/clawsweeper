## Summary

Keep this PR open. Current main still uses the generic 4,000-character web_fetch error-detail cap for all non-OK direct fetch failures after provider fallback, with no status-specific 401/403/407/451 soft-block cap or matching regression tests. The PR remains a focused implementation candidate, and its follow-up commit addresses the wrapper-truncation review concern by capping only the extracted detail while keeping the standard wrapper limit.

## Best Possible Solution

Keep this PR open for maintainer review. If maintainers want the token-saving behavior, land this PR or an equivalent change in `src/agents/tools/web-fetch.ts` that caps only extracted detail for 401/403/407/451 while preserving the normal untrusted-content wrapper, with regression coverage in `src/agents/tools/web-tools.fetch.test.ts`.
