## Summary

Current main still has the behavior this PR targets: direct non-OK web_fetch responses use the generic 4,000-character error-detail cap for all statuses after provider fallback. The PR remains a narrow, useful implementation candidate with tests, and current main does not yet provide an equivalent 401/403/407/451 cap.

## Best Possible Solution

Keep this PR open for maintainer review. If maintainers want the token-saving behavior, land this implementation or an equivalent change in src/agents/tools/web-fetch.ts with regression coverage in src/agents/tools/web-tools.fetch.test.ts; close only after that implementation merges to main.
