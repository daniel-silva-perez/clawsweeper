Current main still lets the reported Anthropic `tool_use`/`tool_result` mismatch wording fall through to the generic `LLM request rejected: ...` formatter branch, even though the transcript repair side is stronger than when the issue was filed. The issue is also paired with open, unmerged PR #45393 using closing syntax, so it should stay open until that PR is reviewed, landed, or replaced.

## What This Changes

The issue asks OpenClaw to stop surfacing raw Anthropic `tool_use`/`tool_result` mismatch errors to chat users and to add defensive repair or retry handling for the transcript pairing failure.

## Best Possible Solution

Keep the issue open and resolve it through review, rebase, landing, or replacement of PR #45393. The narrow end state is a mismatch-specific guard in `formatAssistantErrorText()` before the generic `invalid_request_error` fallback, plus focused regression tests for the reported Anthropic wording; any retry/repair expansion should stay scoped to the agent transcript-repair path.

## Reproduction Assessment

Yes. ## Solution Assessment

Yes for the narrow formatter fix tracked by the linked PR direction. Auto-retry with transcript repair may be useful later, but the safest immediate fix is the targeted user-facing classifier and regression coverage without broad provider retry behavior changes.
