## Summary

Keep open. Current main still does not classify the reported Anthropic tool_use/tool_result mismatch before the generic invalid_request_error formatter branch, so the raw provider wording can still become the user-facing `LLM request rejected: ...` text. The issue is also paired with open, unmerged PR #45393 using closing syntax, so cleanup should wait for that PR to land, be replaced, or be closed.

## What This Changes

The issue asks OpenClaw to avoid leaking raw Anthropic tool_use/tool_result mismatch errors to chat users and to tighten transcript repair for dropped or orphaned tool results.

## Best Possible Solution

Resolve this through review, rebase, landing, or replacement of PR #45393. The end state should add a mismatch-specific formatter guard before the generic invalid_request_error fallback, keep the transcript repair behavior provider-scoped, and include focused regression tests for the exact Anthropic wording and last-message/orphaned tool-call cases.

## Reproduction Assessment

Yes. A high-confidence unit-level reproduction is to pass a JSON invalid_request_error containing `tool_use` ids without `tool_result` blocks into `formatAssistantErrorText()`; current source and a direct regex probe show it misses the replay-invalid classifier and falls through to the raw `LLM request rejected: ...` output.

## Solution Assessment

Yes for the narrow linked-PR direction: a targeted formatter classifier plus regression coverage is the safest immediate fix for the user-facing leak. Broader auto-retry or transcript mutation changes can be considered separately, but they are not required to stop the raw error exposure.
