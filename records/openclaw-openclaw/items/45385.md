## Summary

Keep open. Current main still lets the reported Anthropic `tool_use`/`tool_result` mismatch wording miss the replay-invalid formatter path and fall through to the generic `invalid_request_error` copy, and this issue is already paired with open PR #45393 using closing syntax.

## What This Changes

The issue requests friendlier user-facing handling and stronger repair/retry behavior for Anthropic tool-call/result pairing mismatches that currently leak raw provider errors to chat users.

## Best Possible Solution

Land or replace the existing targeted fix so the formatter recognizes Anthropic tool-call/result pairing errors before the generic invalid-request fallback, keeps the copy recovery-oriented, and adds focused regression coverage for both the exact provider payload and orphaned last-message tool-call repair.

## Reproduction Assessment

Yes. A high-confidence unit-level reproduction is to feed the reported Anthropic `invalid_request_error` payload into the current formatter path; the replay-invalid regex misses it and the generic fallback emits the raw provider text.

## Solution Assessment

Yes for the immediate direction. A narrow formatter classifier plus regression tests is the smallest maintainable fix for the user-facing leak; broader retry or transcript mutation should be evaluated separately after the runtime race path is proven.
