## Summary

Keep open. Current main still leaves QQ Bot token acquisition as an unbounded plugin-owned fetch to `bots.qq.com`, and the channel connection waits on that token before it can open the QQ gateway WebSocket; the docs/status surface still lacks the requested token-endpoint timeout diagnostics.

## What This Changes

The issue requests diagnostics or a workaround for QQ Bot accounts staying disconnected when `https://bots.qq.com/app/getAppAccessToken` times out while core Gateway health remains good.

## Best Possible Solution

Keep the fix inside the bundled QQ Bot plugin: add a bounded timeout and structured network/timeout metadata around token acquisition, thread any existing plugin SDK fetch/proxy helper if appropriate, expose a focused probe/status signal for the token endpoint, and document `bots.qq.com` diagnostics in the QQ Bot troubleshooting docs.

## Reproduction Assessment

Yes. A high-confidence path is to make `https://bots.qq.com/app/getAppAccessToken` hang or time out and start a configured QQ Bot account; current `connect()` awaits the token fetch before opening the QQ WebSocket while core Gateway health can remain unrelated and healthy.

## Solution Assessment

Yes. A narrow QQ Bot plugin reliability/docs fix is the best route; a core Gateway change should only be introduced if implementing this proves a missing generic plugin network helper is needed.
