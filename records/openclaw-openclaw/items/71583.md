## Summary

Keep open. The live timeout appears egress-specific rather than a global QQ outage, but current main still has an OpenClaw-side gap: QQBot token acquisition has no bounded timeout, no QQ-specific status probe, and weak pre-WebSocket error surfacing when the token request stalls.

## What This Changes

The issue reports that QQBot token fetches to `bots.qq.com` time out across two uplinks while the OpenClaw gateway and dashboard remain healthy.

## Best Possible Solution

Keep the fix inside the QQBot plugin: bound token acquisition with an abortable timeout, surface token-stage failures in channel status/probe output, and document the specific `bots.qq.com` egress diagnostic path without adding QQ-specific policy to core gateway code.

## Reproduction Assessment

Yes for the OpenClaw failure mode: a mocked `fetch()` that never resolves would leave `TokenManager.getAccessToken()` pending because the current token request has no abort signal. The reporter's live network timeout is plausible but environment-specific, and this review did not reproduce that external timeout from the runner network.

## Solution Assessment

Yes. A QQBot-owned timeout, status/probe, and troubleshooting update is the narrowest maintainable solution; core should not special-case `bots.qq.com`, and the external egress workaround belongs in docs rather than generic gateway startup policy.
