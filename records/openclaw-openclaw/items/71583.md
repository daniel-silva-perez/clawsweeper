## Summary

Keep open. The issue is recent and specific, and current main still leaves QQ Bot token acquisition as an unbounded plugin-owned fetch before the channel can connect; docs and status output also do not yet give the requested `bots.qq.com` timeout diagnostics.

## What This Changes

The issue asks for OpenClaw-specific diagnostics or workarounds when QQ Bot token fetches to `bots.qq.com` time out while the core gateway and dashboard remain healthy.

## Best Possible Solution

Keep the fix inside the bundled QQ Bot plugin unless a missing generic SDK seam is proven. Add a bounded timeout and structured network/timeout metadata around token acquisition, consider the existing plugin SDK fetch/proxy helpers for proxy-aware behavior, add focused regression tests, and document `bots.qq.com` token endpoint diagnostics in the QQ Bot and channel troubleshooting docs.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to make `https://bots.qq.com/app/getAppAccessToken` hang or time out and start a configured QQ Bot account; current `connect()` awaits the unbounded token fetch before opening the WebSocket, while core gateway health can remain unrelated and healthy.

## Solution Assessment

Yes. The narrow maintainable direction is a QQ Bot plugin reliability/docs fix, not a core migration; core should stay generic unless implementing this exposes a missing plugin SDK network helper.
