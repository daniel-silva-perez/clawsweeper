## Summary

Keep open. Current main still has a QQBot-owned token-stage failure mode: the Tencent token POST is not bounded by a repo-owned abort signal or timeout, gateway connection waits on that token before opening WebSocket, and QQBot does not provide a live status probe or docs that would distinguish `bots.qq.com:443` egress failure from a disconnected channel account.

## What This Changes

The issue asks OpenClaw to diagnose or work around QQBot token fetch timeouts to `bots.qq.com` while the core gateway remains healthy.

## Best Possible Solution

Keep the fix inside the QQBot plugin: add a bounded abortable token fetch, propagate token-stage failures into runtime status, add a QQBot `probeAccount` path for token endpoint diagnostics, and document `bots.qq.com:443` egress checks without adding Tencent-specific policy to core gateway code.

## Reproduction Assessment

Yes for the repo-owned failure mode: stub `fetch()` in `TokenManager.getAccessToken()` so it never resolves, and current main has no token-fetch signal or timer to settle it while `GatewayConnection` is awaiting the token. The reporter's live A/B uplink timeout remains environment-specific and needs their logs or equivalent network access to reproduce exactly.

## Solution Assessment

Yes. A QQBot-owned timeout, probe, and troubleshooting update is the narrowest maintainable solution; core should remain plugin-agnostic and should not special-case `bots.qq.com`.
