## Summary

Keep open. Current main still has an OpenClaw-side QQBot diagnostics gap: token acquisition posts to the reported Tencent endpoint without a bounded timeout, startup waits on that token before the WebSocket connects, and `channels status --probe` has no QQBot token-stage probe to distinguish an egress problem from channel configuration or gateway health.

## What This Changes

The issue reports that QQBot token fetches to `bots.qq.com` time out across two uplinks while the OpenClaw gateway and dashboard remain healthy.

## Best Possible Solution

Keep the repair QQBot-owned: add an abortable token-fetch timeout, propagate token-stage failures into status/probe output, and document `bots.qq.com:443` egress diagnostics without adding QQ-specific endpoint policy to core gateway code.

## Reproduction Assessment

Yes for the OpenClaw failure mode: a mocked `fetch()` that never resolves would leave `TokenManager.getAccessToken()` pending because the token request has no abort signal. The reporter's live network timeout is environment-specific; this runner reached the endpoint successfully in about 1.16 seconds.

## Solution Assessment

Yes. A QQBot-owned timeout, status probe, and troubleshooting update is the narrowest maintainable fix; core should not special-case `bots.qq.com`, and external network workarounds belong in docs/status guidance.
