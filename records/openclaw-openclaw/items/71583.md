## Summary

Keep open. Current main still routes QQ Bot access-token acquisition through a plain plugin-owned fetch to `bots.qq.com` with no explicit timeout or status probe, and the connection path waits for that token before the QQ gateway WebSocket can open. The report is recent, concrete, and points to a narrow QQ Bot reliability/docs gap rather than an obsolete or external-only cleanup item.

## What This Changes

The issue asks for OpenClaw-specific diagnostics or workarounds when QQ Bot token fetches to `bots.qq.com` time out while the core gateway remains healthy.

## Best Possible Solution

Keep the repair inside the bundled QQ Bot plugin: bound access-token requests, preserve secret redaction, expose token-endpoint reachability in the channel status/probe surface, and document the expected diagnostics for `bots.qq.com` failures.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to make `https://bots.qq.com/app/getAppAccessToken` hang or reject and start a configured QQ Bot account; current main awaits that token before opening the QQ gateway WebSocket while the core gateway can remain healthy.

## Solution Assessment

Yes. The narrowest maintainable fix is a QQ Bot plugin reliability/docs change; a core gateway or SDK change should only be added if the implementation proves a reusable plugin network helper is missing.
