## Summary

Keep open. Current main still has the QQBot-owned failure mode described here: the token POST to Tencent has no abort signal or bounded timeout, gateway startup waits on that token before WebSocket connect, and `channels status --probe` has no QQBot token-stage probe or egress guidance to distinguish external network reachability from OpenClaw channel state.

## What This Changes

The issue asks for OpenClaw-specific diagnostics or workarounds for QQBot token-fetch timeouts to `bots.qq.com` that leave QQBot disconnected while the core gateway remains healthy.

## Best Possible Solution

Keep the fix inside the QQBot plugin: add a bounded abortable token fetch, surface token-stage failures into runtime status and `channels status --probe`, and document `bots.qq.com:443` egress checks without adding Tencent-specific policy to core gateway code.

## Reproduction Assessment

Yes for the OpenClaw failure mode: mock `fetch()` in `TokenManager.getAccessToken()` so it never resolves and the current implementation has no signal or timer to settle it. The reporter's live A/B uplink timeout is environment-specific and would need their logs or a live QQBot network to reproduce exactly.

## Solution Assessment

Yes. A QQBot-owned timeout, probe, and troubleshooting update is the narrowest maintainable path; core should stay extension-agnostic and should not special-case `bots.qq.com`.
