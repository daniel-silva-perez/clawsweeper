## Summary

Keep open. Current main still has the QQBot-owned token-stage failure mode: the Tencent token POST is unbounded, gateway connection waits on that token before WebSocket setup, and QQBot still lacks a live token-stage probe or egress troubleshooting docs.

## What This Changes

This issue asks OpenClaw to diagnose or work around QQBot token-fetch timeouts to `bots.qq.com` while the core gateway remains healthy.

## Best Possible Solution

Keep the fix inside the QQBot plugin: add a bounded abortable token fetch, propagate token-stage failures into runtime status, add a QQBot `probeAccount` path for token endpoint diagnostics, and document `bots.qq.com:443` egress checks without adding Tencent-specific policy to core gateway code.

## Reproduction Assessment

Yes for the repo-owned failure mode: stub `fetch()` in `TokenManager.getAccessToken()` so it never resolves, and current main has no token-fetch signal or timer to settle it while `GatewayConnection` awaits the token. The reporter's live A/B uplink timeout is environment-specific and needs their logs or equivalent network access to reproduce exactly.

## Solution Assessment

Yes. A QQBot-owned timeout, probe, and troubleshooting update is the narrowest maintainable solution; core should remain plugin-agnostic and should not special-case `bots.qq.com`.
