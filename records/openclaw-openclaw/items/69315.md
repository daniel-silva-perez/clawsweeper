## Summary

Keep open. Current main still awaits automatic TTS synthesis before delivering final, tool, block, and streamed-final audio payloads, and Telegram still serializes ordinary updates by chat/topic. The related #42539 is narrower, covering separate Telegram text-plus-voice delivery, while this request asks for a non-blocking/background TTS pipeline that avoids holding the inbound lane.

## Best Possible Solution

Keep this open for maintainer design. The strongest path is a shared messages.tts delivery policy plus a background delivery path that sends visible text immediately, generates audio with bounded concurrency, and posts or attaches the resulting voice/audio through existing channel delivery abstractions without blocking the active inbound turn. Coordinate with #42539 for Telegram separate text-plus-voice UX and #48179 for broader voice pipeline performance so the policies do not diverge.
