## Summary

Keep open. Current main still awaits TTS synthesis before final, tool, block, and streamed-final audio delivery, has no async/background TTS delivery policy in config or docs, and Telegram still serializes normal updates by chat/topic. Related #42539 and #48179 are adjacent but do not supersede this broader non-blocking TTS request.

## What This Changes

Request to add non-blocking/asynchronous TTS delivery so text replies and later inbound messages are not held on slow audio synthesis, especially for Telegram with local TTS endpoints.

## Best Possible Solution

Keep open for maintainer design. The strongest path is a shared messages.tts delivery policy plus a background delivery path that sends visible text immediately, generates audio with bounded concurrency, and posts or attaches voice/audio through existing channel delivery abstractions without blocking the active inbound lane; align the UX with #42539 and the broader voice pipeline goals in #48179.
