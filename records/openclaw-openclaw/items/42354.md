## Summary

Keep open. Current main has the useful partial Feishu WebSocket repair from #72411: heartbeat `wsConfig` is passed and rejected `start()` calls get monitor-owned backoff. That still does not prove the reported Feishu/Lark long-connection failure is fixed, because OpenClaw still treats `WSClient.start()` completion as readiness while the pinned Lark SDK reports real first-connect success/failure through callbacks after `start()` returns. The issue discussion also includes a separate Lark international/domain workaround, and setup/docs still steer Lark-domain installs into WebSocket mode by default.

## Best Possible Solution

Keep this issue open and resolve it in the Feishu plugin transport/setup surface: wire `WSClient` readiness through `onReady`/`onError` or the SDK `LarkChannel` readiness wrapper, add a bounded initial handshake timeout and recovery path for retry exhaustion, preserve the heartbeat `wsConfig`, add regression tests for async callback-reported failure after `start()` returns, and clarify the Lark international WebSocket/webhook path in setup and docs.
