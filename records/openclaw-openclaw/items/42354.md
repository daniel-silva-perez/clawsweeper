## Summary

Keep open. Current main has a partial Feishu WebSocket repair, but it still does not prove the reported Feishu/Lark long-connection failure is fixed: OpenClaw uses raw `WSClient.start()` as the readiness signal, while the pinned Lark SDK reports actual initial readiness/failure through callbacks after `start()` returns. The issue discussion also includes a concrete Lark international/domain workaround that current docs and setup still do not handle clearly.

## Best Possible Solution

Keep this issue open. The best fix belongs in the Feishu plugin transport/setup surface: construct the WebSocket client with SDK-supported `onReady`/`onError` callbacks or use the SDK `LarkChannel` readiness wrapper, wait for initial readiness or a bounded handshake timeout before logging started, retry callback-reported initial failures with monitor-owned backoff, and add a regression test where `start()` returns before async initial failure. Maintainers should also decide and document the Lark international path, either by guiding/forcing webhook mode where WebSocket is unsupported or by failing early with a clear configuration error.
