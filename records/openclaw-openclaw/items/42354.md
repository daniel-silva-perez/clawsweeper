## Summary

Keep open. Current main includes the heartbeat option shape and monitor-owned retry when `WSClient.start()` rejects, but the pinned Lark SDK's `start()` returns immediately after kicking off `reConnect(true)`, while successful or failed initial handshakes are reported later through `onReady`/`onError`. OpenClaw still does not wire those callbacks, so it can log the client as started before the first WebSocket connection is actually established. The Lark international/domain workaround from the discussion also remains actionable because docs and config still default Feishu/Lark to WebSocket mode.

## Best Possible Solution

Keep this issue open. The best fix is to align the Feishu WebSocket transport with the SDK lifecycle by waiting for `onReady`, rejecting/retrying on `onError` or handshake timeout, and adding a regression test where `start()` returns before asynchronous initial connection failure. Maintainers should also decide the Lark international behavior: validate/force webhook mode where WebSocket is unsupported, or produce a clear early configuration error with docs updates.
