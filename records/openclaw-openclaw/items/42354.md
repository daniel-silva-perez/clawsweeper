## Summary

Keep open. Current main includes a Feishu WebSocket heartbeat option and a retry path when `WSClient.start()` rejects, but the pinned Lark SDK still reports connection readiness asynchronously through callbacks that OpenClaw does not wire. The SDK constructor also does not type or consume the `wsConfig` field currently passed by OpenClaw, so the original `PingInterval`/`system busy` failure is not strongly fixed on main.

## Best Possible Solution

Keep this issue open. The best fix is in the Feishu plugin transport: construct `WSClient` with `onReady`/`onError` callbacks, make `monitorWebSocket()` wait for initial readiness or a handshake timeout before logging started, retry on callback failure with monitor-owned backoff, and add a regression test where `start()` returns before asynchronous initial connection failure. Maintainers should also decide and document the Lark international behavior: either force/guide webhook mode when WebSocket is unsupported, or fail early with a clear configuration error.
