## Summary

Keep open. Current main has a partial Feishu WebSocket repair, but it still does not prove the reported Feishu/Lark long-connection failure is fixed. OpenClaw still treats `WSClient.start()` resolution as readiness even though the pinned Lark SDK reports first connection success/failure through callbacks after `start()` returns, and the Lark international/domain path from the issue discussion is still unclear in setup/docs.

## Best Possible Solution

Keep this issue open. The best fix belongs in the Feishu plugin transport/setup surface: wait for SDK-supported `onReady`/`onError` or use the SDK `LarkChannel` readiness wrapper, add a bounded initial handshake timeout and retry path for callback-reported failures, and add a regression test where `start()` returns before async initial failure. Maintainers should also decide and document the Lark international path, either by guiding users to webhook mode or failing early with a clear configuration error.
