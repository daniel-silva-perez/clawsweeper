## Summary

Keep open. Current main contains the heartbeat `wsConfig` repair and a retry loop for rejected `start()` calls, but it still does not observe the Lark SDK's callback-based initial readiness/failure path, and the Lark international setup/docs path still defaults users toward WebSocket without a clear early failure or documented webhook requirement.

## What This Changes

Request to fix Feishu/Lark WebSocket long-connection setup so `connectionMode: "websocket"` establishes a persistent connection without `1000040351` or `PingInterval` failures and the Lark console detects the app.

## Best Possible Solution

Finish the Feishu plugin transport fix by wiring SDK-supported `onReady`/`onError` or the SDK `LarkChannel` readiness wrapper into startup, add a bounded initial handshake timeout and recovery loop for terminal callback failures, and document or enforce the supported Lark international transport path.

Yes for the remaining code-level failure mode: a mocked SDK client whose `start()` returns before later `onError` would still be treated as started by current main. Live reproduction against Feishu/Lark is unclear because no usable channel credentials were present.

## Solution Assessment

No, current main is only a partial solution: it fixes the heartbeat config symptom and rejected-start retries, but not callback-reported initial failure or retry exhaustion. The safer direction is the Feishu-owned supervisor/readiness path being explored in the open related PRs, plus a clear Lark international setup decision.
