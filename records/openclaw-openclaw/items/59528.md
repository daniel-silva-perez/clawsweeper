## Summary

Keep open. Current main still leaves a credible source path for the reported OpenShell stopped-status behavior: the standalone sandbox CLI does not preload plugins, while the OpenShell sandbox manager is registered only during full plugin registration. In that state `openclaw sandbox list` can see the OpenClaw registry entry but miss the OpenShell manager, so it reports the runtime as stopped even when OpenShell has or later creates the sandbox. The reporter also supplied concrete WSL/OpenShell steps and a follow-up narrowing the symptom to status still being stopped.

## Best Possible Solution

Keep this issue open and fix the sandbox CLI/backend lifecycle path. The likely maintainer direction is to make `openclaw sandbox list` and `openclaw sandbox recreate` load enabled sandbox-backend plugins before describing/removing runtimes, or add a narrow sandbox-backend manager registration path that preserves plugin-loader laziness. Add regression coverage proving an `openshell` registry entry invokes the OpenShell manager instead of falling through to `running: false`, and clarify docs if lazy creation after non-tool chat turns is intentional.
