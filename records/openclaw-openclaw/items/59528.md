## Summary

Keep open. Current main still has a plausible path for the reported OpenShell stopped-status behavior: the standalone `openclaw sandbox list` command does not have a command-catalog policy that preloads plugins, while the OpenShell backend manager is registered only during full plugin registration. When the manager is absent, sandbox listing falls back to `running: false`, matching the issue’s `Backend: openshell` plus stopped status symptom. The issue also has concrete reproduction details and is not stale.

## Best Possible Solution

Keep this issue open and fix the sandbox CLI/backend lifecycle path. The likely maintainer direction is to make `openclaw sandbox list` and `openclaw sandbox recreate` load enabled sandbox-backend plugins before listing/removing runtimes, or add a targeted lightweight backend-manager registration path for sandbox backends. Add a regression test with an `openshell` registry entry proving `sandbox list` invokes the OpenShell manager instead of falling through to `running: false`; separately clarify or adjust lazy creation semantics if maintainers expect an OpenShell sandbox to exist immediately after a chat turn without sandboxed tool execution.
