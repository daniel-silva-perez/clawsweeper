## Summary

Keep open. Current main still matches the reported failure path: the standalone sandbox CLI does not preload enabled plugins, while the OpenShell sandbox manager is only registered during full plugin registration, so an existing OpenShell registry entry can be rendered as stopped without asking OpenShell for its real status. The reporter’s follow-up that OpenShell eventually created a sandbox but OpenClaw still showed stopped is consistent with this lifecycle/status split.

## What This Changes

The issue reports that OpenShell-backed sandbox runtimes show as stopped and are not visible consistently after enabling the bundled OpenShell plugin on OpenClaw 2026.4.1.

## Best Possible Solution

Implement a generic sandbox backend plugin activation path for `openclaw sandbox list` and `openclaw sandbox recreate`, either by loading enabled sandbox-backend plugins before registry inspection/removal or by adding a targeted lightweight backend-manager registration seam. Keep core extension-agnostic, avoid hardcoding OpenShell in core, and add regression coverage with an OpenShell-style backend registry entry proving the CLI reaches the registered manager instead of falling through to `running: false`.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is available: an OpenShell registry entry plus the current sandbox CLI startup path leaves the OpenShell manager unregistered, and `listSandboxContainers()` then reports `running: false`; no live WSL/OpenShell run was performed.

## Solution Assessment

Yes, the narrowest maintainable fix is to repair the sandbox CLI/backend lifecycle so enabled sandbox backend managers are loaded before status/removal operations. A registry-global singleton change alone is not sufficient if the plugin is never loaded in the CLI process.

## Repair Work Prompt

Fix the OpenShell sandbox stopped-status bug reported in https://github.com/openclaw/openclaw/issues/59528. Current main lets `openclaw sandbox list` and `openclaw sandbox recreate` run without loading enabled plugins; OpenShell registers its sandbox backend manager only during full plugin registration, so `listSandboxContainers()` can fall back to `running: false` for `backendId: "openshell"`. Implement the narrowest generic fix: ensure sandbox CLI status/removal paths load enabled sandbox-backend plugins, or add a targeted plugin/backend-manager activation seam that works for third-party sandbox backend plugins too. Do not hardcode `openshell` in core and do not broaden unrelated CLI plugin preload behavior. Add regression coverage with an OpenShell-style or fake plugin-provided backend entry proving `sandbox list` reaches the registered manager and reports the manager result instead of stopped. Check open PR #67623 before coding because it touches the sandbox backend registry singleton, but do not treat it as a complete fix for this issue unless it also resolves sandbox CLI plugin activation. Changelog: add a user-facing fix entry if behavior changes for released users, with proper reporter credit. Validate with targeted CLI/sandbox tests and then the changed gate via the repo’s normal Testbox policy.
