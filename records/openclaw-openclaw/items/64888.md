## Summary

Keep open. Current main and the latest stable tag still lack the requested `cancelled` embedded-run liveness state, and the embedded runner still has rethrow paths that can leave terminal replay/liveness metadata unset. The related merged harness-seam work is useful context, but it explicitly left this cancellation/liveness cleanup as focused follow-up work.

## What This Changes

The issue requests an embedded-run error boundary that always emits terminal lifecycle metadata and a distinct `cancelled` liveness state for user-initiated aborts.

## Best Possible Solution

Add `cancelled` to the embedded-run liveness contract, return it for user-initiated aborts that are not timeouts, preserve `blocked` for hard failures and timeouts, and ensure every embedded-run rethrow path emits explicit replay/liveness terminal metadata with focused regression coverage.

## Repair Work Prompt

Fix the embedded-run lifecycle gaps tracked by this issue. Add `cancelled` to `EmbeddedRunLivenessState`, make `resolveRunLivenessState` return `cancelled` for user-initiated aborts where `aborted === true`, `timedOut === false`, and no stronger incomplete/error state applies, and update lifecycle/status consumers and tests that currently assume only `working | paused | blocked | abandoned`. Add an error boundary around the embedded attempt path so uncaught `runEmbeddedAttemptWithBackend`/prompt-error rethrows still emit terminal replay/liveness metadata, preserving `blocked` for hard errors/timeouts and rethrowing the original error. Do not redesign the harness SPI, do not rename the runner, and do not broaden this into UI/product status work beyond the required metadata consumers. Add focused regression coverage for a forced backend throw and for simulated user abort producing `livenessState: "cancelled"`. Add a single changelog Fixes entry with `Thanks @100yenadmin` if the project treats this user-visible lifecycle metadata change as changelog-worthy.
