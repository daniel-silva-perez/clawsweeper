## Summary

Keep this PR open. Current main still does not implement the intended bundled MCP tool-call timeout change: the embedded Pi bundle MCP runtime calls `session.client.callTool` without request options, so the SDK default 60-second timeout still applies. The PR remains a plausible implementation candidate for the reported MCP timeout bug, though it should ideally add focused regression coverage before merge.

## Best Possible Solution

Review this PR as an active implementation candidate: either land a focused version that passes `timeout: 300_000` and `resetTimeoutOnProgress: true` with a regression test for the bundled MCP runtime, or replace it with a maintainer-preferred configurable MCP tool-call timeout if OpenClaw wants operator control rather than a hard-coded default.

## Risks / Open Questions

- The PR does not include a regression test that asserts `callTool` receives longer request options or resets on progress notifications.
- A fixed 5-minute default addresses the linked premature-timeout bug, but related timeout reports may still want a configurable per-server or per-tool timeout surface.
- Increasing the wait from 60 seconds to 5 minutes can keep a turn occupied longer when an MCP server hangs or never reports progress.
