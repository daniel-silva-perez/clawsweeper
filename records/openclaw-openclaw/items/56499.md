## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks startup retry/backoff for secret-provider resolution, while this branch targets the old inline startup activator in `server.impl.ts`, includes unrelated WhatsApp append-upsert changes, and would need a narrower replacement in the current `server-startup-config.ts` path.

## What This Changes

The PR adds startup-only exponential retry/backoff around secret snapshot resolution in `src/gateway/server.impl.ts` and also changes WhatsApp append-upsert access-control handling in the WhatsApp plugin.

## Best Possible Solution

Land a focused fix for #56217 in the current `createRuntimeSecretsActivator` startup path, with deterministic tests for transient recovery and retry exhaustion, clear operator logging, preserved reload/restart-check last-known-good behavior, and no WhatsApp changes in the same branch.

## Reproduction Assessment

Yes. A high-confidence focused reproduction is to inject a `prepareRuntimeSecretsSnapshot` mock that rejects during `createRuntimeSecretsActivator(..., { reason: "startup" })`; current main rejects after the first failure, matching the startup crash that launchd then amplifies externally.

## Solution Assessment

No. The retry idea is directionally useful, but this patch targets a stale implementation location, retries the whole provider snapshot on hard failures, and mixes unrelated WhatsApp work; a narrow current-main replacement is safer.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Move startup retry to the active activator:** `src/gateway/server.impl.ts:454-477`
  - body: On current main, startup secret resolution runs through `createRuntimeSecretsActivator` in `src/gateway/server-startup-config.ts`; `server.impl.ts` only constructs that activator. This retry block lands in the old inline path, so the reported startup crash-loop would remain after rebasing unless the retry moves to the active activator.
  - confidence: 0.86
- **[P2] Avoid multiplying provider calls on persistent failures:** `src/gateway/server.impl.ts:463-477`
  - body: The retry wraps the whole `prepareSecretsRuntimeSnapshot` call. Since current resolution executes provider batches per snapshot, a persistent 1Password rate-limit/auth failure can turn each restart into up to four full provider-resolution batches before exiting, increasing the exact command volume this fix is meant to reduce.
  - confidence: 0.78
- **[P3] Show total attempts in retry logs:** `src/gateway/server.impl.ts:470-471`
  - body: With `STARTUP_MAX_RETRIES = 3`, the logs say `Attempt 3/3 failed, retrying...` before the fourth attempt runs. Use total attempts, or otherwise make the retry count unambiguous for operators reading incident logs.
  - confidence: 0.9
