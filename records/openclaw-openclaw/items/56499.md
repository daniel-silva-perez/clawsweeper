## Summary

Keep this PR open, but do not merge it as-is. Current main still has the immediate startup secret-resolution failure behavior, while this branch targets the old inline startup path, retries whole secret snapshots in a way that can increase provider calls on persistent failures, and mixes in unrelated WhatsApp append-upsert behavior.

## What This Changes

The PR adds startup-only exponential retry/backoff around secret snapshot resolution in `src/gateway/server.impl.ts` and also modifies WhatsApp append-upsert pairing/access-control handling.

## Best Possible Solution

Land a focused current-main fix for #56217 in `createRuntimeSecretsActivator`, with bounded startup-only retry/cooldown behavior, clear logs, deterministic tests for transient recovery and exhausted retries, preserved reload/restart-check last-known-good semantics, and no WhatsApp changes in the same branch.

## Reproduction Assessment

Yes. A focused reproduction is to inject a `prepareRuntimeSecretsSnapshot` mock that rejects during `createRuntimeSecretsActivator(..., { reason: "startup" })`; current main rejects after the first failure, matching the crash-loop trigger described by the linked issue.

## Solution Assessment

No. The retry idea is useful, but this patch is not the best implementation because it targets a stale location, retries whole snapshots on hard failures, and mixes unrelated WhatsApp work; a narrow replacement on the active startup activator is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Move startup retry to the active activator:** `src/gateway/server.impl.ts:454-477`
  - body: On current main, startup secret resolution runs through `createRuntimeSecretsActivator` in `src/gateway/server-startup-config.ts`; `server.impl.ts` only constructs that activator. This retry block lands in the old inline path, so the reported startup crash-loop would remain after rebasing unless the retry moves to the active activator.
  - confidence: 0.86
- **[P2] Avoid multiplying provider calls on persistent failures:** `src/gateway/server.impl.ts:463-477`
  - body: The retry wraps the whole `prepareSecretsRuntimeSnapshot` call. Current resolution executes provider batches per snapshot, so a persistent 1Password rate-limit/auth failure can turn each restart into up to four full provider-resolution batches before exiting, increasing the command volume this fix is meant to reduce.
  - confidence: 0.8
- **[P3] Show total attempts in retry logs:** `src/gateway/server.impl.ts:470-471`
  - body: With `STARTUP_MAX_RETRIES = 3`, the logs say `Attempt 3/3 failed, retrying...` before the fourth attempt runs. Use total attempts, or otherwise make the retry count unambiguous for operators reading incident logs.
  - confidence: 0.9
