## Summary

Keep this PR open. Current main still lacks startup retry/backoff for secret resolution, but the PR is stale against the active startup path: runtime secret activation now lives in `src/gateway/server-startup-config.ts`, while this patch adds retries to the old inline activator in `src/gateway/server.impl.ts`; it also carries an unrelated WhatsApp append/pairing change and a misleading retry-log denominator.

## What This Changes

The PR adds startup-only exponential retry/backoff around secret snapshot resolution and also changes WhatsApp append-upsert handling to suppress pairing replies before processing replayed history messages.

## Best Possible Solution

Land a focused fix for the startup secret-provider crash-loop in the current `createRuntimeSecretsActivator` path, preserving reload/restart-check last-known-good behavior, adding deterministic unit coverage for startup retry exhaustion and recovery, and handling any WhatsApp append/pairing work separately if it is still needed.

## Reproduction Assessment

Yes. A high-confidence focused reproduction is to inject a `prepareRuntimeSecretsSnapshot` mock that rejects during `createRuntimeSecretsActivator(..., { reason: "startup" })`; current main rejects after the first failure, and the full 1Password launchd loop is the external amplification of that same startup behavior.

## Solution Assessment

No, not as currently proposed. The retry concept is directionally useful for #56217, but the implementation must move into the current `src/gateway/server-startup-config.ts` activator, keep non-startup reload semantics unchanged, add tests, and avoid bundling unrelated WhatsApp changes.
