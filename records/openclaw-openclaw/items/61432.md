## Summary

Keep open. Current `main` still appears to drop `event.messages` pushed by `message:received` internal hooks: `dispatchReplyFromConfig` creates the hook event inline, passes it to `triggerInternalHook(...)` through `fireAndForgetHook(...)`, and never retains or routes the event's `messages` array. The docs and type contract still say hooks can push messages back to the user, and the command reset hook path shows the expected pattern of awaiting the hook and routing `hookEvent.messages`. The reporter's linked PR #61657 was a plausible attempted fix but is closed unmerged, so it is not a reason to close this issue.

## Best Possible Solution

Keep this issue open. The fix should update `dispatchReplyFromConfig` to retain and await the `message:received` internal hook event, route any pushed `hookEvent.messages` to the originating channel using the existing additive notice/reply path, and add regression coverage showing `event.messages.push(...)` from a `message:received` hook reaches the user. PR #61657 is useful prior art and could be rebased if it still matches the current dispatch structure.
