## Summary

Keep open. The PR is not ready to merge: the branch is dirty against current main, and the Gmail watcher backoff reset happens on process spawn, so the immediate-crash loop case the PR targets still repeats at the initial delay. The useful config and subagent pieces should be rebased into a narrower, tested fix rather than closed as obsolete.

## What This Changes

The PR wraps subagent output polling read errors, adds warnings for best-effort source-config fallback, adds Gmail watcher restart backoff, attempts to clear a media cleanup interval, and deletes the UI i18n registry test.

## Best Possible Solution

Rebase or replace the PR with a narrow runtime stability fix: implement Gmail watcher backoff that resets only after a healthy run window, keep the active i18n registry test, drop the obsolete media-server edit, and carry forward the subagent/config fixes with targeted tests and a changelog entry.

## Reproduction Assessment

Yes for the blocking PR regression: a local Node child_process check shows the spawn event fires before exit for an immediately exiting child, which proves the PR's reset point is too early. The other claimed gaps were source-inspected on current main rather than fully runtime-reproduced in this read-only pass.

## Solution Assessment

No. The PR direction is partly useful, but resetting Gmail watcher backoff on spawn is not the right solution; reset after a non-trivial healthy uptime window and preserve unrelated test coverage.
