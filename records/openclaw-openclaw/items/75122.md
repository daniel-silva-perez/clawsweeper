## Summary

Keep open. Current main partially addresses the report with aggregate liveness warnings, WhatsApp Baileys timing knobs, generic outbound delivery queue recovery, and unknown-root runtime-deps pruning, but the central reliability asks remain: no liveness source attribution for the blocking call path, no durable reconnect-safe queue for automatic WhatsApp replies, and no pruning of stale versioned runtime-deps roots.

## What This Changes

The issue asks OpenClaw to diagnose event-loop stalls behind WhatsApp 408 disconnects, log blocking-source attribution, replay dropped outbound replies after reconnect, expose Baileys timing knobs, and garbage-collect stale plugin runtime-deps trees.

## Best Possible Solution

Keep this as an active WhatsApp/Gateway reliability investigation. The maintainable end state is low-noise liveness stall attribution, removal or isolation of the synchronous blocker, idempotent reconnect-safe delivery for automatic WhatsApp replies, and safe cleanup of stale versioned runtime-deps roots while preserving the socket timing and generic delivery-queue improvements already on main.

## Reproduction Assessment

Unclear. The reporter supplied credible production logs and deterministic 408 timing, but current repository tests do not reproduce the long event-loop stalls or a queued outbound reply being dropped across reconnect.

## Solution Assessment

No automated close or narrow fix is the best path yet. The remaining work crosses live performance tracing, WhatsApp delivery semantics, and runtime-deps lifecycle policy, so maintainer scoping should precede implementation.
