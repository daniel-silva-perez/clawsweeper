## Summary

Keep open. Current main contains useful mitigations for the report, including liveness warnings, WhatsApp socket timing configuration, transient auto-reply retry, and stale unknown runtime-deps pruning, but it still lacks the central fixes: source attribution for long event-loop blocks, identification/removal of the synchronous blocker, durable reconnect-safe outbound replay, and cleanup for versioned runtime-deps trees.

## What This Changes

The issue asks OpenClaw to investigate WhatsApp 408 disconnects attributed to long Node event-loop stalls, add liveness attribution, replay queued outbound replies after reconnect, expose Baileys timing options, and garbage-collect stale plugin runtime-deps trees.

## Best Possible Solution

Keep this as an active WhatsApp/Gateway reliability investigation. The desired end state is low-noise attribution for liveness warnings, removal or offloading of the blocking path, an idempotent reconnect-safe outbound delivery design, and runtime-deps pruning that handles stale versioned stage roots without deleting active or locked installs.

## Reproduction Assessment

Unclear. The reporter supplied credible production logs and deterministic 408 timing, but current repository tests do not reproduce the long event-loop stalls or a queued outbound reply being dropped across reconnect.

## Solution Assessment

No. Current main is a partial mitigation, not the complete fix; widening Baileys timing knobs helps operators, but the maintainable solution needs source attribution, blocker removal, and carefully designed replay semantics.
