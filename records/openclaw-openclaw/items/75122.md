## Summary

Keep open. Current main has partial mitigations for the report, including WhatsApp socket timing config, reconnect draining for generic outbound queue entries, and aggregate liveness warnings, but it still does not identify the blocking async resource/call path, durably queue automatic WhatsApp source replies, or prune stale versioned runtime-deps trees.

## What This Changes

The issue asks OpenClaw to diagnose event-loop stalls behind WhatsApp 408 disconnects, add liveness source attribution, replay dropped outbound replies after reconnect, expose Baileys timing knobs, and garbage-collect stale plugin runtime-deps trees.

## Best Possible Solution

Keep this as an active WhatsApp/Gateway reliability investigation. The desired end state is low-noise liveness stall attribution, removal or isolation of the synchronous blocker, idempotent reconnect-safe delivery for automatic WhatsApp replies, and safe cleanup of stale versioned runtime-deps roots while preserving the socket timing and generic delivery-queue improvements already on main.

## Reproduction Assessment

Unclear. The reporter provided credible production liveness logs and deterministic 408 timing, but current repository tests do not provide a self-contained reproduction for the 33-100s event-loop blocks or the exact dropped automatic reply after a 408 close.

## Solution Assessment

No automated close or narrow repair is the best path yet. The remaining work crosses live performance tracing, delivery semantics, and runtime-deps lifecycle policy, so maintainer scoping should precede a focused implementation PR.
