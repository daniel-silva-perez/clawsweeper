## Summary

Keep open. Current main has useful partial mitigations for the report, including Baileys timing config, reconnect-aware sends, a generic delivery queue, and limited runtime-deps pruning, but it still does not attribute long liveness stalls to a blocking resource or fully cover the reported versioned runtime-deps accumulation and WhatsApp auto-reply replay path.

## What This Changes

The issue requests fixes for WhatsApp 408 disconnects tied to long Gateway event-loop stalls, including source attribution, reconnect-safe reply delivery, Baileys timing controls, and stale runtime-deps garbage collection.

## Best Possible Solution

Keep this as an active WhatsApp/Gateway reliability investigation. The maintainable end state is to add low-noise liveness blocker attribution, remove or offload the measured synchronous path, make WhatsApp auto-reply delivery reconnect-safe with clear idempotency rules, and define safe retention for stale versioned runtime-deps roots while coordinating with the open runtime-deps tracker.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Unclear. The reporter provided credible production logs and timing data, but current main still lacks a self-contained reproduction for the 33-100s event-loop stalls or the exact dropped auto-reply-after-disconnect path.

## Solution Assessment

Is this the best way to solve the issue? No. Current main has useful partial mitigations, but closing now would leave the root event-loop blocker, attribution gap, auto-reply durability question, and versioned runtime-deps retention policy unresolved.
