## Summary

Keep open. Current main still has the group/channel tool-only fallback and does not include this warning, so the PR is not obsolete. The patch is narrow and plausible, with one low-severity wording issue: it tells users to restart even though the current messages config path is documented and planned as hot-reload/no-restart. The linked issue also contains broader changelog, config.patch, and migration questions, so maintainers should decide whether this PR closes it or only partially addresses it.

## What This Changes

This PR adds a module-level one-shot auto-reply warning when group/channel visibleReplies config is absent and the resolver falls back to message_tool_only delivery.

## Best Possible Solution

Land the resolver-level runtime warning after adjusting the wording to match the supported config reload path, then make an explicit maintainer call on whether #74876 needs separate follow-up for changelog wording, config.patch usability, or migration behavior. A small logger assertion in source-reply-delivery-mode.test.ts would make the warning durable without broadening the patch.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: call the resolver or dispatch path with empty config and ChatType group/channel; current tests show the mode becomes message_tool_only and no normal final reply is sent until messages.groupChat.visibleReplies is automatic.

## Solution Assessment

Mostly yes. A resolver-level one-shot warning is a narrow maintainable mitigation because it fires only when the affected runtime path is reached, but the text should avoid making restart mandatory and the linked issue should not be closed automatically unless maintainers accept the narrower scope.
