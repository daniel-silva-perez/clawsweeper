## Summary

Keep open. Current main does not include the startup warning this PR proposes, and the linked group/channel visible-reply behavior is still present, so the PR is not obsolete. The diff is narrow and mostly correct, with one low-priority wording issue around restart guidance plus pending CI state for maintainers to resolve.

## What This Changes

This PR adds a guarded startup `log.warn` in `startGatewayServer()` when neither `messages.groupChat.visibleReplies` nor `messages.visibleReplies` is explicitly configured.

## Best Possible Solution

Land a narrow startup warning if maintainers accept warning-based mitigation, but make the user-facing text match current reload behavior and decide whether the broader #74876 requests need a separate follow-up instead of being closed by this PR.

## Reproduction Assessment

Yes. A high-confidence code-level path exists: with empty config and `ChatType` of `group` or `channel`, source reply delivery resolves to `message_tool_only`, and dispatch tests show normal final replies are not sent until `messages.groupChat.visibleReplies: "automatic"` is configured.

## Solution Assessment

Mostly yes. A startup warning is the narrowest fix that avoids reverting the safety default, but the proposed warning should not tell users a restart is required when current `messages` config hot-reloads under the documented gateway reload path.
