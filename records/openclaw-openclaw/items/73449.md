## Summary

Keep this PR open. The Codex tool-stream bridge addresses a real gap on current main, but the latest PR head still emits raw tool args and full-output result/update payloads into live agent-event surfaces without the existing Pi tool-stream redaction layer, so it should not merge as-is.

## What This Changes

The PR adds normalized Codex app-server `stream: "tool"` start/update/result projections and regression tests in the Codex event projector.

## Best Possible Solution

Keep the Codex tool-stream bridge, but route start args plus partial and final result payloads through the same redaction policy as the Pi tool stream, preferably via an SDK-safe shared sanitizer seam rather than a deep core import; add regression coverage for redacted command, MCP, dynamic, image, and full-output payloads while preserving the full-output gate and result deduplication.

## Reproduction Assessment

Yes. The missing current-main behavior is visible in the Codex projector handlers, and the PR regression is reproducible with projector notifications containing secret-bearing command args or output while inspecting emitted `stream: "tool"` events.

## Solution Assessment

No. The direction is the right narrow fix, but the implementation is not the best safe version until Codex tool events use the same producer-side redaction guarantees as the existing Pi tool stream.
