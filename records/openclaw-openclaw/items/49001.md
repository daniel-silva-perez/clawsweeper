## Summary

Keep this PR open, but not merge-ready as-is. Current main still lacks the durable post-reset startup marker and recovery steer behavior, while the PR has a concrete lifecycle bug: it clears the recovery window after a started run even when the startup recovery turn fails. The linked broader Discord channel/thread PR is useful context, but it is still open and not shipped proof.

## What This Changes

The PR adds a Discord thread post-reset startup window, wraps queued owner steering prompts with startup-recovery instructions, clears the window after a run, and adds focused regression tests for the queued-message race.

## Best Possible Solution

Land or replace this with an updated combined fix that marks Discord post-reset startup as pending, wraps queued owner steers with recovery instructions, preserves the marker across failed recovery attempts, clears it only after a successful recovery lifecycle, preserves current steering queue semantics, and includes regression coverage for thread and channel reset surfaces.

## Reproduction Assessment

Yes. The code path is high-confidence: on current main, an explicit reset can create a fresh session without any durable startup marker, and a queued owner message during a streaming run is steered as raw followupRun.prompt, so the startup-recovery instruction has no way to survive the race.

## Solution Assessment

No, not as-is. The PR’s direction is narrow and plausible, but clearing the marker after any started run means a provider or network failure during the recovery turn can drop the guard before startup recovery succeeds; an updated implementation should clear only after successful completion and should consider the open channel/thread follow-up.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.85

Full review comments:

- **[P2] Keep startup window until a run succeeds:** `src/auto-reply/reply/agent-runner.ts:412`
  - body: `startedRun` is set before awaiting `runAgentTurnWithFallback`, and the `finally` block clears `postRotationStartupUntilMs` whenever that flag is true. If the post-reset startup turn throws, for example from a transient provider or network failure, the recovery marker is removed before startup preload recovery succeeds, so the next queued owner message no longer gets the guard this PR depends on. Clear the window only after successful turn completion, or preserve it on failure.
  - confidence: 0.88
