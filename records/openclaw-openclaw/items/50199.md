## Summary

Keep open. Current main has deterministic skill source precedence, eligibility filters, per-agent allowlists, and alphabetical prompt ordering, but it does not implement user-configurable skill priority, priority metadata, language/cost-aware preference between different eligible skills, or `openclaw skills priority` CLI management. The request is a core skills policy/config gap rather than a ClawHub-only skill/plugin request.

## Best Possible Solution

Keep this issue open and design a core skills policy extension. A conservative path would add a documented `skills.entries.<skill>.priority` or `skills.priority` list in `openclaw.json`, validate it in the config schema, apply it consistently before prompt truncation and command catalog generation, preserve existing source precedence for same-name overrides, and add focused tests plus docs/CLI support if maintainers accept a management command.

## Risks / Open Questions

- The design still needs maintainer judgment on whether priority should affect prompt ordering, prompt truncation, slash-command ordering, same-name replacement, or all of those surfaces.
- OpenClaw currently exposes skills to the model and relies on description matching unless a skill is invoked explicitly, so a priority feature should be framed carefully as catalog ordering/visibility policy unless deterministic dispatch is added.
