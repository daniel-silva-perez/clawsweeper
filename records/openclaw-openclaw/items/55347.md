## Summary

Keep open. Current main has useful recovery building blocks, but it does not implement the requested native gateway self-healing workflow: no `gateway.selfHeal` config, no gateway-process health monitor with retry budget, no autonomous restart loop, and no configurable recovery brief target. The related open PR is only a narrow doctor-on-failure slice and explicitly does not cover retry loops or broad auto-repair.

## Best Possible Solution

Keep this issue open as the umbrella for native gateway self-healing. The best path is to review and land narrow slices such as the doctor-on-failure PR where correct, then design the remaining core behavior around a conservative gateway self-heal contract: process health probing/retry budget in the gateway/service layer, doctor-backed diagnostics, recovery notifications through the existing restart sentinel/session delivery path, and plugin-owned config repair rules rather than bundled-plugin policy in core.

A targeted `rg` for those names returned no implementation hits.

## Risks / Open Questions

- Native process self-healing crosses restart policy, config repair, messaging delivery, and security boundaries; it needs explicit maintainer design to avoid silent restart loops or unsafe auto-repair.
- Some shipped recovery pieces overlap with the request, so the implementation should reuse doctor, config reload recovery, restart sentinels, diagnostics bundles, and Crestodian rescue rather than adding a separate parallel recovery system.
