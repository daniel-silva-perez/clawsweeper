## Summary

Keep this issue open. Current main still normalizes `kimi-coding` to `kimi`, tests still encode that behavior, and the issue is paired with open candidate PR #74315 using closing syntax, so it should not be closed before that PR or a replacement fix lands.

## What This Changes

The issue asks OpenClaw to stop losing exact `kimi-coding` provider namespaces during model selection so `kimi-coding/k2p5` resolves without a misleading fallback.

## Best Possible Solution

Make exact configured provider IDs win before alias normalization wherever model selection consumes `models.providers`, while preserving intended legacy Kimi aliases for onboarding/auth; add regression coverage for a configured `kimi-coding/k2p5` default showing no selected-model fallback.

## Reproduction Assessment

Yes. Source inspection is enough for a high-confidence reproduction path: `normalizeProviderId("kimi-coding")` returns `kimi`, and configured defaults plus stored overrides feed through `normalizeModelRef`; tests currently assert that collapse.

## Solution Assessment

Unclear for the issue’s minimal alias-removal option. A config-aware exact-provider-first fix is likely safer than globally removing the alias, but it must cover defaults, catalog/status lookup, and persisted overrides before this can close.
