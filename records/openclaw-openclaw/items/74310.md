## Summary

Keep this open. Current main still normalizes `kimi-coding` to `kimi`, the latest tagged release has the same alias, and the issue is already linked to open candidate PR #74315 with closing syntax, so the issue should stay open until that PR or a replacement fix is reviewed and merged.

## What This Changes

The issue requests that provider-namespaced model IDs such as `kimi-coding/k2p5` preserve the configured provider namespace instead of being normalized to `kimi/k2p5`.

## Best Possible Solution

Review and land PR #74315 if maintainers want `kimi-coding` to be a distinct provider ID everywhere, or replace it with a narrow config-aware exact-provider-first fix that preserves exact configured namespaces while retaining intended `kimi` and `kimi-code` compatibility.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence reproduction path: `normalizeProviderId("kimi-coding")` returns `kimi`, and runtime default/override paths feed configured providers through `normalizeModelRef`.

## Solution Assessment

Unclear. The linked PR’s one-line alias removal is a plausible minimal fix, but the safer product contract may be exact configured provider lookup before fallback normalization so legacy built-in Kimi aliases keep working.
