## Summary

Keep this issue open. Current main and the latest release still collapse `kimi-coding` through the generic provider normalizer, and the issue is already paired with open candidate PR #74315, so it should stay open until that PR or a replacement fix is reviewed and merged.

## What This Changes

The issue asks for provider/model resolution to preserve configured `kimi-coding/...` model refs instead of rewriting them to `kimi/...`.

## Best Possible Solution

Review PR #74315 against the intended Kimi compatibility contract, then land it or replace it with an exact-configured-provider-first resolution path that preserves custom namespaces while keeping intended `kimi` and `kimi-code` compatibility.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: `normalizeProviderId("kimi-coding")` returns `kimi`, and the configured default/runtime paths call that normalizer on provider refs.

## Solution Assessment

Unclear. Removing the one alias is a plausible narrow fix, but the bundled Kimi plugin and docs currently treat `kimi-coding` as compatibility metadata around canonical `kimi`, so exact configured-provider lookup before alias fallback may be safer.
