## Summary

Keep this PR open. Current main still does not implement the PR's intended cron lifecycle hook system: there is no `cron.hooks` config, no per-job hook field, no Gateway protocol support for hooks, and scheduled, startup catch-up, and manual cron paths still execute jobs directly. Main has `cron_changed` plugin observation from #72773, but that is after-the-fact notification and does not provide `beforeRun` aborts or configured script lifecycle dispatch.

## Best Possible Solution

Keep the PR open as an implementation candidate. Maintainers should decide whether scheduler-level cron lifecycle hooks are a core cron API or a plugin/API extension; if the feature proceeds, harden per-job validation and import/reload behavior, then update config schema, Gateway protocol, docs, and tests for scheduled, catch-up, and manual cron paths together.

## Risks / Open Questions

- The PR adds an operator-configured dynamic script execution surface inside cron/Gateway paths, so it needs explicit maintainer and security review before merge.
- The review thread still contains hardening concerns around less-trusted persisted job hook data, including path trimming before safety checks and type validation before path handling.
- The design overlaps existing internal hooks and plugin hooks; maintainers should decide whether scheduler-level lifecycle control belongs in core cron or behind a plugin/API extension.
