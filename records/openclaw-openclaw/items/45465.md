## Summary

Keep this PR open. Current main still does not implement cron lifecycle hooks: there is no `cron.hooks` config, no per-job hook field, no Gateway protocol hook surface, and scheduled, startup catch-up, and manual cron execution paths still call the core job runner directly. The existing `cron_changed` plugin hook is useful observation infrastructure from #72773, but it is after-the-fact notification and does not provide configured `beforeRun` aborts or script dispatch at `afterComplete`, `onFailure`, and `afterRun`.

## Best Possible Solution

Keep the PR open as an implementation candidate. Maintainers should decide the product boundary first: either land a hardened core cron lifecycle API, or design a plugin/API extension that supports pre-run abort and outcome-specific lifecycle dispatch. If the feature proceeds, update config schema, Gateway protocol, docs, and tests for scheduled, startup catch-up, and manual cron paths together, and address the remaining path/type validation concerns before merge.

## Risks / Open Questions

- The PR adds a new dynamic code execution surface inside cron/Gateway paths, so it needs explicit maintainer and security review before merge.
- The latest provided review thread still has unresolved hardening concerns for less-trusted persisted job hook values, especially trimming paths before validation and validating entry types before path handling.
- The design overlaps existing internal hooks and the newer `cron_changed` plugin hook; maintainers should decide whether scheduler-level lifecycle control belongs in core cron, a plugin API extension, or a narrower plugin hook contract.
