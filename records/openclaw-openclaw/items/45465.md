## Summary

Keep this PR open. Current main still does not implement cron lifecycle hooks, and the existing `cron_changed` plugin hook is observation-only rather than a pre-run/abortable script lifecycle. The PR remains an implementation candidate, but it needs maintainer product and security review because it adds dynamic script execution from cron config and persisted job data.

## Best Possible Solution

Keep the PR open as an implementation candidate. Maintainers should decide the product boundary first: either land a hardened core cron lifecycle API, or design a plugin/API extension that supports pre-run abort and outcome-specific lifecycle dispatch. If the feature proceeds, update config schema, Gateway protocol, docs, and tests for scheduled, startup catch-up, and manual cron paths together, and resolve the remaining path/type validation concerns before merge.

## Risks / Open Questions

- The PR introduces cron-driven dynamic script imports from global config and persisted job hook values, which needs explicit maintainer/security review before merge.
- The provided review thread still includes unresolved hardening concerns for less-trusted per-job hook values, especially path trimming before validation and type checks before path handling.
- The design overlaps the existing `cron_changed` plugin hook; maintainers should decide whether pre-run abort and outcome-specific lifecycle control belongs in core cron, a plugin hook extension, or a narrower plugin API.
