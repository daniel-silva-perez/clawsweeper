## Summary

Keep open. The intended heredoc bug appears plausible, but current main has not implemented this PR and the proposed patch still has a concrete security-sensitive correctness issue: it would use raw command text for heredoc approval follow-ups even when approval is granted by `askFallback: "allowlist"` after a timeout rather than by an explicit human decision. Existing ClawSweeper and review-comment context already identified the same P1 concern, and the branch has not added focused regression coverage for the explicit-approval and timeout-fallback split.

## What This Changes

The PR changes gateway exec approval follow-up so heredoc-required approvals execute the original `params.command` instead of the rebuilt allowlist `enforcedCommand`, while leaving non-heredoc approvals on the enforced command path.

## Best Possible Solution

Land a narrow gateway approval fix that preserves raw heredoc command text only after explicit `allow-once` or `allow-always`, keeps `enforcedCommand` for allowlist timeout fallback approvals, and adds focused tests for both heredoc branches plus the unchanged non-heredoc path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Keep enforced command for heredoc timeout fallback:** `src/agents/bash-tools.exec-host-gateway.ts:285`
  - body: This switches every heredoc approval follow-up to `params.command`, but the same callback also runs when a request times out and `askFallback === "allowlist"` marks an allowlist match as approved without a human decision. That path should keep the rebuilt `enforcedCommand`; otherwise timeout fallback re-runs raw shell parsing instead of the pinned/quoted allowlist plan. Use raw command text only for explicit `allow-once` or `allow-always`.
  - confidence: 0.89
