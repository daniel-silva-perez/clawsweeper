## Summary

Keep open. The PR targets a real gateway exec heredoc approval bug that current main still has, but the proposed one-line switch changes a security-sensitive approval path too broadly: timeout fallback approvals with `askFallback: "allowlist"` would also run the raw heredoc command instead of the rebuilt enforced command.

## What This Changes

The PR changes `src/agents/bash-tools.exec-host-gateway.ts` so heredoc-required gateway approval follow-ups execute the original `params.command` instead of always passing the rebuilt allowlist `enforcedCommand`.

## Best Possible Solution

Land a narrow gateway approval fix that preserves raw heredoc command text only after explicit `allow-once` or `allow-always`, keeps `enforcedCommand` for allowlist timeout fallback approvals, and adds focused tests for explicit heredoc approval, timeout fallback, and unchanged non-heredoc behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Keep enforced command for heredoc timeout fallback:** `src/agents/bash-tools.exec-host-gateway.ts:285`
  - body: This switches every heredoc approval follow-up to `params.command`, but the same callback also runs when a request times out and `askFallback === "allowlist"` marks an allowlist match as approved without a human decision. That path should keep the rebuilt `enforcedCommand`; otherwise timeout fallback re-runs raw shell parsing instead of the pinned/quoted allowlist plan. Use raw command text only for explicit `allow-once` or `allow-always`.
  - confidence: 0.89
