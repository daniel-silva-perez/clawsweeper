## Summary

Keep open. The underlying heredoc approval bug appears real and current main still routes approved gateway follow-up execution through `enforcedCommand`, but this PR changes a security-sensitive host exec path too broadly: timeout fallback approvals can also reach the same callback and would execute raw `params.command` without an explicit human allow decision.

## What This Changes

The PR changes gateway exec approval follow-up so heredoc-required approvals execute the original `params.command` instead of the rebuilt allowlist `enforcedCommand`, while leaving non-heredoc approvals on the enforced command path.

## Best Possible Solution

Land a narrow gateway approval fix that preserves raw heredoc command text only after explicit `allow-once` or `allow-always`, keeps the rebuilt enforced command for allowlist timeout fallback approvals, and adds focused regression tests for both heredoc branches plus the unchanged non-heredoc path.

## Reproduction Assessment

Yes. For the review finding, a focused test can drive `processGatewayAllowlist` with `requiresHeredocApproval`, a timed-out approval decision, and `askFallback: "allowlist"`; the PR head then selects `params.command` at the changed line instead of keeping `enforcedCommand`.

## Solution Assessment

No. The PR fixes the likely heredoc mangling at the wrong decision granularity; the safer solution is to branch on an explicit allow decision, not on `requiresHeredocApproval` alone.
