## Summary

Keep open. Current main has not absorbed the PR’s intended heredoc execution fix, but the proposed branch changes a security-sensitive host exec approval path so timeout fallback approvals can execute raw heredoc shell text without an explicit human allow decision.

## What This Changes

The PR changes gateway exec approval follow-up so heredoc-required approvals execute the original `params.command` instead of the rebuilt allowlist `enforcedCommand`, with no tests or docs changes.

## Best Possible Solution

Land a narrow gateway approval fix that preserves raw heredoc command text only after explicit `allow-once` or `allow-always`, keeps `enforcedCommand` for allowlist timeout fallback approvals, and adds focused regression tests for both heredoc branches plus the unchanged non-heredoc path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep enforced command for heredoc timeout fallback:** `src/agents/bash-tools.exec-host-gateway.ts:285`
  - body: This switches every heredoc approval follow-up to `params.command`, but the same callback also runs when a request times out and `askFallback === "allowlist"` marks an allowlist match as approved without a human decision. That path should keep the rebuilt `enforcedCommand`; otherwise timeout fallback re-runs raw shell parsing instead of the pinned and quoted allowlist plan.
  - confidence: 0.89

## Security Review

Status: needs_attention

Concerns:

That drops the enforced command's pinned and quoted argv boundary for a security-sensitive host exec path.
