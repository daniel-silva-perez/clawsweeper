## Summary

Keep open. Current main still lacks `OPENCLAW_JITI_CACHE_DIR`, and the PR targets a real plugin-loader operational gap, but the branch is not merge-ready because its unset fallback forces `fsCache: true` and can override Jiti-level `JITI_FS_CACHE` / `JITI_CACHE` behavior that existing deployments may rely on.

## What This Changes

The PR adds an `OPENCLAW_JITI_CACHE_DIR` environment override to Jiti loader options in the shared plugin loader, plugin SDK root alias, and Matrix runtime wrapper, while disabling `fsCache` in one regression test.

## Best Possible Solution

Land a narrow implementation that only supplies a trimmed `OPENCLAW_JITI_CACHE_DIR` when it is non-empty, otherwise leaves `fsCache` undefined so Jiti continues honoring its own env/defaults; wire the same helper through the shared plugin loader, SDK root alias, and Matrix wrapper; add cache-key isolation if the effective option can vary; and document the accepted operator-facing env surface with targeted regression coverage.

## Risks / Open Questions

- Merging the PR unchanged would force `fsCache: true` when the OpenClaw override is unset or blank, potentially breaking deployments that already use `JITI_FS_CACHE=false` or `JITI_CACHE=false` to avoid filesystem cache writes.
- If the effective cache path can vary in-process during tests or embedded runs, the current Jiti loader cache key may reuse a loader created with a different `fsCache` value.
- The new user-facing env var lacks focused tests and a changelog/docs decision, so behavior and operator discovery are not locked down yet.
- The security-sensitive part is narrow but real: this changes filesystem writes in a runtime code-loading path, so the implementation should preserve Jiti's existing disable/failure behavior and avoid unrelated loader or dependency changes.
