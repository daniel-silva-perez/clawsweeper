## Summary

Keep open. Current main still lacks the requested OpenClaw-specific Jiti cache-dir override, but this PR is not merge-ready because the unset fallback forces `fsCache: true` and can override existing Jiti-level cache disabling via `JITI_FS_CACHE` or `JITI_CACHE`.

## What This Changes

The PR adds `OPENCLAW_JITI_CACHE_DIR` handling to the shared plugin loader, plugin SDK root alias, and Matrix runtime wrapper, and disables Jiti fsCache in one git-path regression test fixture.

## Best Possible Solution

Land a narrow implementation that only supplies a trimmed `OPENCLAW_JITI_CACHE_DIR` value when it is non-empty and otherwise omits `fsCache` so Jiti keeps its own env/default behavior. Wire that consistently through the shared plugin loader, SDK root alias, and Matrix wrapper, add cache-key isolation if the effective option varies, and add focused tests plus release-note coverage.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to apply the PR, leave `OPENCLAW_JITI_CACHE_DIR` unset, set `JITI_FS_CACHE=false`, and observe that OpenClaw passes explicit `fsCache: true` to `createJiti()`, overriding Jiti 2.6.1's env-derived default.

## Solution Assessment

No. The requested env override is a reasonable direction, but the current implementation is not the best fix because it changes existing Jiti env semantics; the safer solution is to return a path only for a non-empty OpenClaw override and otherwise leave `fsCache` undefined.
