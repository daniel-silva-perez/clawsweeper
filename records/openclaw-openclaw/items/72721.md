## Summary

Keep open. Current main still maps the reported Feishu failure to an SDK-backed probe/status path, and the related Feishu timeout cluster has open follow-up rather than a merged complete fix.

## What This Changes

The issue reports Feishu websocket startup failing because @larksuiteoapi/node-sdk/axios calls to open.feishu.cn time out from the gateway process on Alibaba Cloud ECS with Node.js 24.

## Best Possible Solution

Keep the fix in the Feishu plugin: define one consistent token/probe transport policy across startup, setup/status, and channel status, honor configured and caller timeouts plus proxy behavior, preserve requireMention safety while bot identity is unknown, and add focused unit coverage plus a redacted live verification path for the SDK-vs-direct-request failure mode.

## Reproduction Assessment

Unclear. The issue has concrete user steps and strong field diagnostics, but a high-confidence repo-local reproduction still needs Feishu credentials and the reported Node 24/ECS runtime behavior or a deterministic SDK transport fixture.

## Solution Assessment

Unclear. There is no proposed patch on this item; the open startup-latency PR is relevant but may not be sufficient for the reported axios/socket reset, so Feishu-owner review should decide the narrower transport fix.
