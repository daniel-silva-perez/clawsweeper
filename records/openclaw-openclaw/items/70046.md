## Summary

Keep this PR open. Current main and the latest release still reject bare `HH:MM` / `HH:MM:SS` one-shot cron times, while this PR is a focused, mergeable parser/test change for that remaining behavior. I found no blocking correctness or security issue in the diff, though one low-priority test flake and docs/help follow-up are worth maintainer attention.

## What This Changes

The PR adds a `parseAt` branch for bare `HH:MM` / `HH:MM:SS` values, resolves them against today's date in the requested IANA timezone or UTC, rejects out-of-range time-only input, and adds focused parser unit tests.

## Best Possible Solution

Keep this PR as the active implementation candidate for the time-only `--at` bug, with the parser fix in the CLI layer and the generic absolute-time parser left unchanged. Before landing, tighten the date-boundary-sensitive tests if desired, run the targeted cron CLI tests, and decide whether the same PR should update help/docs wording for the newly supported time-only form.

## Reproduction Assessment

Yes. The current-main reproduction path is source-backed: `cron add --at 09:00 --tz Asia/Shanghai` reaches `parseAt`, which has no time-only branch, and `Date.parse('09:00')` returns `NaN` on this Node runtime, causing the CLI to throw `Invalid --at`.

## Solution Assessment

Yes, with a small test-hardening caveat. Handling bare time-only strings in CLI `parseAt` is the narrow maintainable place because it has the `--tz` context and can reuse `parseOffsetlessIsoDateTimeInTimeZone`; changing `parseAbsoluteTimeMs` would make a lower-level parser infer an ambiguous date without caller context.
