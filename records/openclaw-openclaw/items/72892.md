## Summary

Keep open. Current main still has the reported bypass shape if Control UI/WebChat emits two runtime-distinct sends: inbound dedupe is keyed by route/session/message id, while `chat.send` uses the caller idempotency key as `MessageSid`. The UI has same-submit and same-run-id guards, but those do not prove a different-sender/different-run-id path is fixed, and the upstream second dispatch path still needs metadata from an affected run.

## What This Changes

The issue requests that Control UI/WebChat user messages enter the agent session once instead of being processed twice under different sender/run identities.

## Best Possible Solution

Keep this issue open, collect metadata for one duplicate pair, then remove the upstream double-dispatch path or add a narrowly scoped backend guard that catches this fan-out without changing normal repeated-message behavior.

## Reproduction Assessment

Unclear. The reporter gives plausible steps and the source explains why two different run ids would bypass dedupe, but the available evidence does not include a current-main duplicate pair or logs that identify the second dispatch path.

## Solution Assessment

Unclear. A secondary content fingerprint could be useful defense in depth, but the safest maintainable fix is to first trace why Control UI/WebChat emits two runtime-distinct dispatches and guard that specific path.
