## Summary

Keep open. Current main still makes the reported bypass plausible: inbound dedupe is keyed on route plus message id, `chat.send` uses the client idempotency key as `MessageSid`, and the Control UI guard only coalesces same-submit or reused-id paths. The related #72314 PR is useful adjacent work, but its missing-id fallback would not directly cover two runtime-distinct Control UI/WebChat dispatches with different ids.

## What This Changes

The issue requests fixing Control UI/WebChat so one browser message is not injected twice under different sender/run-id identities and does not trigger duplicate agent replies.

## Best Possible Solution

Keep the issue open, collect metadata for one duplicate pair, then either remove the upstream double dispatch or add a narrowly scoped backend guard for the Control UI/WebChat fan-out without changing normal repeated-message behavior.

## Reproduction Assessment

Unclear. The reporter gives plausible v2026.4.24 steps and current source explains how two different run ids would bypass dedupe, but the issue does not include a current-main duplicate pair or logs that identify the second dispatch path.

## Solution Assessment

Unclear. A secondary content fingerprint may be useful defense in depth, but the narrowest maintainable fix is to first trace why two runtime-distinct dispatches happen and guard or remove that path.
