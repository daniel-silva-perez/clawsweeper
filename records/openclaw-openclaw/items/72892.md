## Summary

Keep open. Current main still has the reported bypass shape: inbound dedupe is keyed on route/session/message id, while `chat.send` uses the caller idempotency key as `MessageSid`, so two Control UI/WebChat dispatches with different run ids would not collapse. The UI has a same-send in-flight guard, and #72314 is adjacent but only covers missing message IDs, so neither proves this different-sender/different-id path is fixed.

## What This Changes

The issue requests that Control UI WebChat send each user message into the agent session exactly once, instead of allowing a second `openclaw-control-ui` dispatch to trigger a duplicate run.

## Best Possible Solution

Keep this issue open, collect metadata for one duplicate pair, then either remove the upstream double-dispatch path or add a narrow backend guard that catches this fan-out without changing normal repeated-message behavior.

## Reproduction Assessment

Unclear. The source and discussion give a plausible static reproduction model for why two different run ids bypass dedupe, but the report does not include the duplicate pair metadata needed to identify and reproduce the second dispatch path on current main.

## Solution Assessment

Unclear. A secondary content fingerprint may be useful defense in depth, but the safest maintainable solution is to first trace why Control UI/WebChat emits two runtime-distinct dispatches and fix that specific path if possible.
