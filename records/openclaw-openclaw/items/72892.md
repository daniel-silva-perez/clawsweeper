## Summary

Keep open. Current main still makes the reported sender/run-id mismatch plausible: inbound dedupe remains keyed on route plus `MessageSid`, `chat.send` still maps each client idempotency key to `MessageSid`, and the Control UI guard only coalesces same in-flight submits before a second run id is generated. The related #72314 PR is useful adjacent work for idless retries, but it is open and does not directly cover two distinct Control UI/WebChat dispatches that both have ids.

## What This Changes

The issue requests fixing Control UI/WebChat so one browser message is not injected twice under different sender/run-id identities and does not trigger duplicate agent replies.

## Best Possible Solution

Keep the issue open, collect metadata for one duplicate pair, then either remove the upstream double dispatch or add a narrowly scoped backend guard for the Control UI/WebChat fan-out without changing normal repeated-message behavior.
