## Summary

Keep open. Current main still keys inbound dedupe on provider/route/session/message id and `chat.send` on the caller's idempotency key, so a duplicate Control UI/WebChat dispatch with a different run id would not be collapsed. The current UI has an in-flight duplicate-submit guard, but the discussion correctly notes that the linked #72314 is only adjacent and does not directly cover this different-id sender mismatch.

## What This Changes

Ensure Control UI/WebChat sends each user message into the session once so sender/run-id mismatches cannot create duplicate agent turns.

## Best Possible Solution

Keep this issue open, collect one duplicate pair's gateway debug logs and `sessions_history` metadata, then fix the upstream double-dispatch path or add a narrowly scoped backend dedupe guard that preserves intentional repeated messages.

## Reproduction Assessment

Unclear. The reporter gives plausible user steps and the discussion validates the key mechanics, but the available evidence does not include the duplicate pair metadata needed to identify the second dispatch path on current main.

## Solution Assessment

Unclear. A secondary content fingerprint may be useful defense-in-depth, but the safest maintainable fix is to first identify why Control UI/WebChat emits two distinct runs and then guard that specific path.
