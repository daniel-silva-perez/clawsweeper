Current main does not support a conservative cleanup close: the reported bypass remains plausible in source, the UI guards only cover duplicate same-submit or reused-idempotency paths, and the linked #72314 is adjacent but does not close a two-dispatch path with different message ids.

## What This Changes

Ensure Control UI WebChat sends each user message into an agent session only once, even when sender identities or run ids differ.

## Best Possible Solution

Keep this open, collect metadata for one duplicate pair, then either remove the upstream double dispatch or add a narrow backend guard that catches the same Control UI/WebChat fan-out without changing normal repeated-message behavior.

## Reproduction Assessment

Unclear. The reporter gives plausible v2026.4.24 steps and current source explains why two different run ids would bypass dedupe, but the available evidence does not include a current-main duplicate pair or logs that name the second dispatch path.

## Solution Assessment
