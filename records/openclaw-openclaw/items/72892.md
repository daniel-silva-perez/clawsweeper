## Summary

Keep open. Current main has same-submit and same-idempotency guards, but the backend inbound dedupe key still depends on route plus `MessageSid`, and `chat.send` still maps each fresh UI run id into `MessageSid`; I did not find source or history evidence that the reported two-dispatch Control UI/WebChat path has been eliminated. The related #72314 work is useful adjacent coverage for missing message ids, but it does not directly cover two runtime-distinct sends with different ids.

## What This Changes

The issue asks OpenClaw to stop Control UI/WebChat user messages from being processed twice when distinct sender/run identities bypass inbound dedupe.

## Best Possible Solution

Keep the issue open, collect metadata for one duplicate pair, then either remove the upstream double dispatch or add a narrowly scoped backend guard for Control UI/WebChat fan-out that preserves normal repeated-message behavior.

## Reproduction Assessment

Unclear. The reported v2026.4.24 steps and maintainer discussion give a plausible bypass, and current main shows how different run ids would evade dedupe, but there is no current-main log or `sessions_history` duplicate pair proving the second dispatch path.

## Solution Assessment

Unclear. The current same-submit and idempotency guards are good narrow fixes for repeated UI submits, but they are not sufficient evidence that this two-dispatch issue is solved; a broader content dedupe should wait until the upstream path is identified.
