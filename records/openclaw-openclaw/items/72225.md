## Summary

Current main still has the exact Control UI chat-session refresh behavior this PR targets: the shared active-window constant is 120 minutes, but three chat refresh paths still pass `activeMinutes: 0`, and `loadSessions` omits non-positive `activeMinutes` from the `sessions.list` request. The PR remains a narrow, relevant UI fix with test coverage and no visible security-sensitive diff surface.

## Best Possible Solution

Keep this PR open for maintainer review and merge it, or land an equivalent scoped fix: replace the three remaining Control UI `activeMinutes: 0` chat-session refresh callsites with `CHAT_SESSIONS_ACTIVE_MINUTES`, keep the constant value and sessions API unchanged, and retain regression coverage for initial chat refresh and session-switch refresh.
