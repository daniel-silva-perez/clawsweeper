## Summary

Keep this PR open. Current main still does not implement the PR’s central cron/profile context-window fix: isolated cron finalization uses the nonblocking lookup path with async loading disabled, and status still treats a stored positive contextTokens value as authoritative rather than repairing a stale 200k fallback.

## Best Possible Solution

Keep this PR open for maintainer review/rebase, or port the narrow fix onto current main: add a blocking provider/model context-window resolver suitable for cron persistence, use it when finalizing isolated cron sessions, add a guarded read-only status display repair only for known stale DEFAULT_CONTEXT_TOKENS fallbacks, preserve explicit context caps and invalid-value guards, and keep regression coverage for cron persistence plus status display repair.
