## Summary

Keep this PR open. Current main still does not implement the PR’s intended fix: isolated cron finalization persists context tokens through the synchronous nonblocking lookup path, and status still treats stored positive `contextTokens` values as authoritative without a stale-200k display repair. The PR remains a focused bug-fix candidate rather than obsolete or superseded.

## Best Possible Solution

Keep this PR open for maintainer review/rebase, or port the same fix onto current main: isolated cron should persist context tokens from a blocking provider-aware resolver, and status should repair only known stale `DEFAULT_CONTEXT_TOKENS` fallbacks while preserving explicit caps, ignoring invalid stored values, and keeping the status path read-only. Keep or add regression tests for cron persistence and status display repair.
