## Summary

Keep this PR open. Current main still does not implement the PR’s central fix: isolated cron finalization persists context tokens through a nonblocking lookup with async loading disabled, and status still treats a stored positive contextTokens value such as 200000 as authoritative instead of doing a guarded read-side repair. The provided PR discussion remains useful: Greptile and the earlier ClawSweeper review both identify the same root cause and nonblocking cron/status surfaces, and the current tree still matches that unresolved shape.

## Best Possible Solution

Keep this PR open for maintainer review/rebase, or port the narrow fix onto current main: add a blocking provider/model context-window resolver suitable for cron persistence, use it in isolated cron finalization, add guarded read-only status repair only for known stale DEFAULT_CONTEXT_TOKENS fallbacks, preserve explicit caps and invalid-value guards, and keep regression coverage for both cron persistence and status display repair.
