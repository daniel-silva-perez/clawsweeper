## Summary

Keep this PR open. Current main still has the central bug shape described in the PR: isolated cron finalization persists context tokens through a nonblocking lookup with async discovery disabled, and status still treats a stored positive value such as 200000 as authoritative instead of doing the guarded read-side repair proposed here.

## Best Possible Solution

Keep this PR open for maintainer review/rebase, or port the narrow fix onto current main: add a blocking provider/model context-window resolver for cron persistence, use it in isolated cron finalization, add guarded read-only status repair only for stale DEFAULT_CONTEXT_TOKENS fallbacks, preserve explicit caps and invalid-value guards, and keep regression coverage for cron persistence plus status display repair.
