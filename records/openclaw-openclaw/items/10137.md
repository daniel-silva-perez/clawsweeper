## Summary

Keep this issue open. Current main still only partially satisfies the requested `openai-responses` proxy compatibility layer: HTTP Responses payloads already use array `input` and `stream: true`, but generic `api: "openai-responses"` providers still lack stable opt-in controls for top-level fallback `instructions` and for omitting `max_output_tokens`. The provided discussion also treats #10140 as the closed duplicate and this issue as the canonical tracker.

## Best Possible Solution

Keep this issue open as the canonical tracker and leave #10140 closed as the duplicate. Add focused, opt-in Responses proxy compatibility controls for generic `api: "openai-responses"` providers/models: top-level fallback `instructions` derived from the sanitized system prompt when requested, and optional omission of `max_output_tokens`; preserve current defaults for array `input`, `stream: true`, and native provider behavior, then document and test the knobs.
