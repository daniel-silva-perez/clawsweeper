## Summary

Keep open. Current main partially covers the report by always sending Responses requests with array input and streaming enabled, and it has a newer Codex-only guard for system-prompt-only turns, but generic `openai-responses` proxy models still have no stable opt-in for fallback top-level `instructions` or for omitting `max_output_tokens`. The provided discussion also identifies #10140 as the closed duplicate and keeps this issue as the canonical tracker.

## What This Changes

Request configurable normalization for `api=openai-responses` proxy requests, including non-empty instructions, array input, streaming, and omission of unsupported fields such as `max_output_tokens`.

## Best Possible Solution

Add opt-in Responses proxy compatibility fields through model config/types/schema, catalog normalization, payload policy, transport construction, tests, generated config docs, public provider/local-model docs, and changelog, preserving existing native OpenAI, Codex, and Azure defaults.
