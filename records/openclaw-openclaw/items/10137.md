## Summary

Keep open. Current main partially addresses the report by always building Responses payloads with array `input` and `stream: true`, but the remaining requested stable compatibility controls are still missing: generic `openai-responses` proxy models cannot opt into top-level fallback `instructions`, and there is no compat/payload-policy option to omit `max_output_tokens`. The discussion identifies #10140 as the closed duplicate, so this issue remains the canonical tracker.

## What This Changes

Request stable Responses API compatibility settings for `api: "openai-responses"` proxy endpoints that need request normalization such as non-empty `instructions`, array input, forced streaming, and omission of unsupported fields like `max_output_tokens`.

## Best Possible Solution

Add opt-in Responses proxy compatibility fields through model config/types/schema, catalog normalization, payload policy, transport payload construction, focused tests, generated config docs, public provider/local-model docs, and changelog while preserving current native OpenAI, Codex, and Azure defaults.

## Reproduction Assessment

No live high-confidence reproduction was available in this read-only pass because the report depends on stricter third-party Responses-compatible relays. Static current-main inspection does reproduce the missing product surface: there is no config path to request top-level fallback `instructions` for generic `openai-responses` or to omit `max_output_tokens`.

## Solution Assessment

Yes, the requested opt-in compatibility layer is the narrowest maintainable direction. Automatically changing all `openai-responses` payloads would risk native OpenAI/Codex/Azure behavior, so the fix should be explicit compat config and policy-driven payload normalization.
