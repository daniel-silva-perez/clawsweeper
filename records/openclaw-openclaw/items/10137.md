## Summary

Keep open. Current main still only partially addresses the report: Responses payloads are array-shaped and streamed, and prompt-cache stripping is configurable, but there is still no stable opt-in path for generic `openai-responses` proxy fallback `instructions` or for omitting `max_output_tokens` on stricter relays.

## What This Changes

The issue requests stable per-model normalization and compatibility options for `openai-responses` proxy endpoints that can reject missing `instructions`, non-array input, missing streaming mode, or unsupported fields such as `max_output_tokens`.

## Best Possible Solution

Add explicit per-model Responses proxy compatibility fields through config types/schema, model-catalog normalization, payload policy, transport construction, focused tests, generated config docs, provider/local-model docs, and changelog while preserving all current native defaults.

## Reproduction Assessment

No, not for the relay-specific HTTP 400 because it depends on a stricter third-party Responses-compatible endpoint. Static inspection does give high confidence that current main cannot configure the remaining fallback `instructions` or omit-`max_output_tokens` behavior.

## Solution Assessment

Yes, the requested opt-in compatibility layer is the narrowest maintainable direction. Applying fallback instructions or omitting token caps globally would risk regressions for native OpenAI, Codex, and Azure Responses routes.
