## Summary

Keep open. Current main still covers only part of the report: Responses requests already use array `input` and `stream: true`, and some proxy-only field stripping exists, but there is still no stable opt-in for generic `openai-responses` fallback `instructions` or for omitting `max_output_tokens` on stricter Responses-compatible relays.

## What This Changes

Add configurable `openai-responses` request-normalization options for stricter Codex/Responses-compatible proxy endpoints that reject or require specific top-level payload fields.

## Best Possible Solution

Add explicit per-model Responses proxy compatibility fields across config types/schema, catalog normalization, payload policy, transport payload construction, focused tests, generated config docs, provider/local-model docs, and changelog, preserving current native OpenAI/Codex/Azure defaults.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No live high-confidence reproduction path was available because the 400 depends on stricter third-party Responses-compatible relays. Static inspection does reproduce the remaining product gap: current main has no stable config path for generic fallback `instructions` or omitting `max_output_tokens`.

## Solution Assessment

Is this the best way to solve the issue? Yes, an opt-in compatibility layer is the narrowest maintainable direction. Automatically changing every `openai-responses` payload would risk native OpenAI/Codex/Azure behavior, so this should stay policy-driven and per-model configurable.
