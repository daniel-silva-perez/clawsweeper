## Summary

Keep open. Current main still passes raw ACP initialize traffic into the ACP SDK, so the VS Code/Cursor date-string protocolVersion incompatibility is not implemented on main; the PR is useful source work but needs a current-main rebase and one correctness fix before merge.

## What This Changes

The PR wraps the ACP server stdio stream to rewrite non-number `initialize.params.protocolVersion` values to the ACP SDK `PROTOCOL_VERSION` before constructing `AgentSideConnection`.

## Best Possible Solution

Land a rebased current-main ACP server compatibility fix that normalizes only incoming `initialize.params.protocolVersion` values before SDK validation, preserves valid ACP numeric versions, uses the SDK-equivalent uint16 predicate or clearly scopes the shim to string MCP versions, adds a focused regression test, and credits both the reporter and source PR author in the changelog.

## Reproduction Assessment

Yes. The linked bug logs provide concrete VS Code/Cursor initialize messages with a string date protocolVersion, and source/dependency inspection shows current main feeds that value into SDK validation that requires an integer 0..65535.

## Solution Assessment

No, not as submitted. A stream-boundary transform is the narrow maintainable approach, but this patch is based on an older server bootstrap and its guard is narrower than the stated compatibility contract.

## Review Findings
