## Summary

Keep open. The linked ACP interop bug is still present on current main and this PR is aimed at the right SDK-boundary surface, but the branch is dirty/unrebaseable against current main and the submitted guard does not match the ACP SDK uint16 validation contract it is trying to bypass.

## What This Changes

The PR wraps the ACP server `ndJsonStream` with a transform that rewrites non-number `initialize.params.protocolVersion` values to the ACP SDK `PROTOCOL_VERSION` before constructing `AgentSideConnection`.

## Best Possible Solution

Land a rebased current-main ACP server compatibility fix that normalizes only incoming `initialize.protocolVersion` values before SDK validation, uses the same uint16 validity predicate as the SDK, preserves valid ACP numeric versions, adds a focused regression test, and then closes #56102 after merge.

## Reproduction Assessment

Yes. The linked bug report includes concrete VS Code/Cursor initialize logs with `protocolVersion: '2025-11-25'`, and current main still feeds raw `ndJsonStream` messages into the SDK path that validates `protocolVersion` as a uint16 number before OpenClaw handler logic runs.

## Solution Assessment

No, not as submitted. The transform boundary is the narrow maintainable place to solve this incompatibility, but the patch needs to be rebased onto the current server bootstrap and its guard should match the SDK integer/range contract before it is the best fix.
