## Summary

Keep open. Current main still passes the raw ACP `ndJsonStream` into the SDK, so the VS Code/Cursor date-string `protocolVersion` bug remains; the PR is directionally useful, but it is stale against the current ACP server bootstrap and its guard does not match the SDK uint16 validation contract.

## What This Changes

The PR adds an ACP server stream transform that rewrites non-number `initialize.params.protocolVersion` values to the ACP SDK `PROTOCOL_VERSION` before constructing `AgentSideConnection`.

## Best Possible Solution

Land a rebased current-main ACP server compatibility fix that normalizes only incoming `initialize.params.protocolVersion` values before SDK validation, uses the SDK-equivalent uint16 predicate, preserves valid ACP numeric versions, adds a focused regression test, and closes #56102 after merge.

## Reproduction Assessment

Yes. The linked bug report includes concrete VS Code/Cursor initialize logs with `protocolVersion: '2025-11-25'`, and current main still feeds raw initialize messages into the SDK path that validates `protocolVersion` as a uint16 number before OpenClaw handler logic runs.

## Solution Assessment

No, not as submitted. A transform at the ACP stream boundary is the narrow maintainable fix, but this patch needs to be rebased onto the current server bootstrap and should coerce any value that fails the SDK uint16 predicate, not only non-number values.

## Review Findings
