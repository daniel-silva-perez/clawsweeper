## Summary

Current main still has the reported sanitizer gap. `sanitizeUserFacingText()` strips `<final>` tags textually, then only collapses exact adjacent duplicate blocks separated by blank lines; it does not handle same-line repeats, single-newline repeats, partial-prefix repeats, or `</final>` boundary-aware cleanup. Two related PRs attempted the same-line part but were closed unmerged, and an adjacent open PR touches duplicate-block formatting rather than this issue’s missing behavior.

## Best Possible Solution

Keep this issue open and implement a narrow sanitizer fix in `src/agents/pi-embedded-helpers/sanitize-user-facing-text.ts`. The best path is to handle the concrete `</final>` same-line repeat and single-newline adjacent-repeat shapes conservatively, with regression tests in `src/agents/pi-embedded-helpers.sanitizeuserfacingtext.test.ts` that also prove normal repeated wording, ordinary single-newline formatting, and code-block indentation are preserved.

## Clownfish Work Prompt

Fix the output sanitizer gap described in https://github.com/openclaw/openclaw/issues/14629. Current main strips `<final>` tags textually and only collapses exact duplicate blocks separated by blank lines. Implement a conservative fix in `src/agents/pi-embedded-helpers/sanitize-user-facing-text.ts` for the concrete same-line/single-newline duplicate pattern that can result from `</final>` stripping, including the reported shape where the same prefix appears before and after the final close tag. Do not add a broad near-duplicate/substring remover that can delete intentional repetition. Preserve ordinary single-newline formatting, intentional short repeated words, and code-block indentation. Add focused regression tests in `src/agents/pi-embedded-helpers.sanitizeuserfacingtext.test.ts` for: same-line duplicate after `</final>` stripping; single-newline adjacent duplicate if supported by the chosen heuristic; the real-world `Atlas here... </final> Atlas here...`-style prefix repeat; no change for normal repeated wording; no indentation loss in duplicate block collapse. Use #16813 and #19113 as context only; they were closed unmerged and used a simple regex that may be too broad. Be aware of adjacent open PR #41851 about preserving indentation in duplicate-block output.
