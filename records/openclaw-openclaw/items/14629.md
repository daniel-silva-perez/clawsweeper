## Summary

Keep open: current main still has the reported sanitizer gap, and the issue has already been reviewed by a maintainer as valid. The relevant code still strips `<final>` tags by replacement, only collapses exact duplicate blocks separated by blank lines, and outbound replies still pass through that sanitizer.

## What This Changes

Improve `sanitizeUserFacingText()` duplicate cleanup so glued same-line, single-newline, or tag-boundary duplicate assistant output is removed conservatively without damaging normal formatting.

## Best Possible Solution

Add a conservative sanitizer pass for the concrete glued `</final>` repeat and explicitly supported adjacent same-line or single-newline duplicate forms, with regression tests proving ordinary repeated prose, normal single-newline formatting, trailing whitespace, and code-block indentation are preserved.

## Reproduction Assessment

Yes. From current source, the reported example becomes glued duplicate prose after `</final>` is replaced, and `collapseConsecutiveDuplicateBlocks()` returns it unchanged because there is no blank-line-separated duplicate block.

## Solution Assessment

Yes. A narrow sanitizer repair is the best maintainable direction; broad near-duplicate or whole-response coherence heuristics should be avoided unless they are tightly scoped to concrete leak patterns and covered by tests.
