## Summary

Keep open because current main still has the reported sanitizer gap: final tags are stripped with a plain regex replacement, duplicate collapsing only handles exact adjacent blocks separated by blank lines, and outbound reply delivery still calls this sanitizer. The maintainer discussion and related PR context also point the same way: the prior same-line duplicate PRs are closed unmerged, and the open adjacent PRs do not specifically close this issue.

## What This Changes

The issue requests stronger output sanitization for same-line, single-newline, tag-boundary, and partial duplicate assistant text that current paragraph-level duplicate collapse misses.

## Best Possible Solution

Add a conservative sanitizer pass for the concrete glued `</final>` repeat and explicitly supported adjacent same-line or single-newline duplicate forms, with regression tests proving ordinary repeated prose, normal single-newline formatting, trailing whitespace, and code-block indentation are preserved.

## Reproduction Assessment

Yes. From the current source, the reported example becomes glued duplicate prose after `</final>` is replaced, and `collapseConsecutiveDuplicateBlocks()` returns it unchanged because there is no blank-line-separated duplicate block.

## Solution Assessment

Yes, a narrow sanitizer repair is the best maintainable direction; a broad near-duplicate or whole-response coherence heuristic should be avoided unless it is tightly scoped by tests and concrete leak patterns.
