## Summary

Keep this PR open. Current main still drops short explicit `[[tts:text]]...[[/tts:text]]` speech text through the generic under-10-character TTS guards, while this PR is a narrow runtime/test fix and its second commit addresses the empty-markdown edge case raised in review.

## What This Changes

The PR changes speech-core TTS payload handling so explicit tagged speech text can bypass the untagged short-text guard, adds an empty post-markdown guard, and adds regression tests for tagged short, untagged short, and markdown-only tagged text.

## Best Possible Solution

Land this PR or an equivalent narrow patch in `extensions/speech-core/src/tts.ts` with focused regression coverage, then let linked #73758 close after merge. Keep #74955 as a separate review for message-tool send-path TTS parity.

## Reproduction Assessment

Yes. Current main can be reasoned through with `[[tts:text]]hello[[/tts:text]]`: the parser captures `hello`, but `maybeApplyTtsToPayload` returns at the under-10-character guard before `textToSpeech`. The PR adds direct regression coverage for that path.

## Solution Assessment

Yes. The PR changes only the two short-text gates to distinguish explicit tagged speech from untagged auto-TTS, while preserving a post-markdown empty guard; that is the narrowest maintainable fix for the reported runtime bug.
