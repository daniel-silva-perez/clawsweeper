## Summary

Keep this issue open. Current main has related Telegram transport lifecycle hardening and bounded Undici dispatcher pools, but the report is fresh against v2026.4.27 and describes normal-polling TIME_WAIT churn that is not proven fixed by the existing stall/rebuild cleanup work. The release-to-main diff does not show a Telegram change that addresses socket reuse after the reporter's version, and the current tests cover dispatcher options and cleanup rather than live connection reuse or stable TIME_WAIT behavior.

## What This Changes

The issue asks OpenClaw to reuse persistent HTTPS connections for Telegram long polling so normal polling does not create fresh TIME_WAIT sockets every roughly 10 seconds.

## Best Possible Solution

Keep the issue open and validate it with a live Telegram transport probe on current main and v2026.4.27. If client-side churn is confirmed, the fix should stay in the Telegram plugin transport or polling runner configuration, preserve the existing dispatcher cleanup and watchdog behavior, and add focused tests or instrumentation for dispatcher reuse without moving Telegram-specific policy into core.

## Reproduction Assessment

Yes. The issue includes a concrete live reproduction path: start the gateway with Telegram bots and monitor ESTABLISHED/TIME_WAIT sockets to Telegram every 10 seconds. I did not run it because it requires configured live Telegram credentials and network observation, but the steps are specific enough for maintainer validation.

## Solution Assessment

Unclear. The existing dispatcher lifecycle fixes are necessary but not sufficient proof for this report, and blindly adding TCP keepalive is not clearly the best fix because Undici already has keepalive behavior and the older TCP-keepalive PR targeted a different stall mode. The right fix depends on whether the churn comes from getUpdates, sendChatAction/control calls, server-side closes, proxy behavior, or client dispatcher reuse.
