## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Return the current targeted wake result:** `src/infra/heartbeat-runner.ts:1250`
  - body: After deferred replay, `ran` can already be true before the newly requested targeted wake runs. If that current `executeForAgent` call fails or skips, this return still reports `ran`, so callers lose the actual result for the wake they requested.
  - confidence: 0.88
- **[P1] Recheck deferred cron replay before future schedule slots:** `src/cron/service/timer.ts:625-638`
  - body: Deferred recurring jobs can keep a future `nextRunAtMs`; because replay checks are only armed when `nextAt` is absent, replay can wait until the next normal cron slot instead of running when the phase first becomes allowed.
  - confidence: 0.86
- **[P2] Preserve global FIFO for deferred cron runs:** `src/cron/service/timer.ts:600-610`
  - body: The replay builder sorts by each job's first deferred timestamp and then appends all runs for that job. That can replay newer runs from one job before older runs from another, breaking the advertised FIFO behavior across jobs.
  - confidence: 0.84
- **[P2] Bound or coalesce blocked heartbeat deferrals:** `src/infra/heartbeat-runner.ts:1263-1264`
  - body: Every blocked non-interval wake appends to `deferredRuns`. Repeated targeted, hook, or action wakes during a long blocked phase can grow the in-memory queue without limit and replay a burst later.
  - confidence: 0.83
- **[P2] Reject invalid maintenance agent IDs:** `src/infra/maintenance-phase.ts:143-146`
  - body: `normalizeAgentId` can fall back to `main` for malformed values while the schema only rejects empty strings. A typo can silently make the default agent maintenance-only, so invalid entries should be rejected before normalization.
  - confidence: 0.87
- **[P2] Restart schedulers when user timezone changes:** `src/gateway/config-reload-plan.ts:78-80`
  - body: Maintenance windows using timezone `user` depend on `agents.defaults.userTimezone`, but the reload rule only watches `cron.maintenance`. Changing the user timezone can leave cron and heartbeat evaluating stale phase inputs.
  - confidence: 0.82
- **[P3] Use the tick snapshot for replay gating:** `src/infra/heartbeat-runner.ts:1060`
  - body: `replayDeferredRuns` accepts a `now` snapshot but calls `Date.now()` for the maintenance check. Long replay loops can evaluate a different phase from the rest of the tick; use the provided snapshot consistently.
  - confidence: 0.73
