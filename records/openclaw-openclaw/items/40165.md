## Summary

Keep this issue open. Current main has adjacent NO_REPLY protections, but it still does not implement the requested shared gateway behavior: terminal suffix tokens remain visible in live chat/TUI projection and mixed outbound payloads, while separated malformed prefixes are still intentionally preserved rather than discarded.

## Best Possible Solution

Implement a shared silent-token normalization contract for gateway-visible and delivery-bound assistant text, then wire it into live chat projection, embedded TUI projection, chat history/display normalization if intended, and outbound delivery planning. Add focused regression coverage for exact suppression, separated-prefix discard or explicitly documented preservation, glued-prefix stripping, standalone terminal suffix stripping, newline suffix stripping, mixed case, media/non-text payloads, and preservation of legitimate embedded NO_REPLY mentions.

## Risks / Open Questions

- A broad suffix rule could remove legitimate explanatory mentions of NO_REPLY; the fix should strip only standalone terminal tokens and preserve embedded/non-terminal mentions in tests.
- OpenClaw currently has separate normalization paths for live Gateway display, embedded TUI, chat history, outbound payload planning, cron delivery, and channel-specific delivery, so the intended shared contract boundary needs deliberate maintainer choice.
