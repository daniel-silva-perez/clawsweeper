## Summary

Keep this issue open. Current main has partial NO_REPLY handling, including exact-token suppression, glued-prefix stripping for gateway live chat, suffix stripping in auto-reply normalization, and cron-specific trailing-token suppression, but the requested shared gateway/display/delivery contract is still not implemented for terminal suffix tokens or separated prefix forms across the affected surfaces.

## Best Possible Solution

Implement a shared silent-token normalization contract for assistant text before gateway display and before delivery-bound channel payloads. It should cover exact suppression, separated-prefix discard or a clearly documented preservation rule, glued-prefix cleanup, standalone terminal suffix stripping with cleaned content delivery, mixed case, media payloads, and legitimate embedded NO_REPLY mentions. Add focused regression coverage for gateway live/final chat, embedded TUI, outbound delivery planning, and the channel route used by Telegram/WhatsApp/WebChat.

## Risks / Open Questions

- A broad suffix rule could remove legitimate explanatory mentions of NO_REPLY; the implementation should strip only standalone terminal sentinel tokens and preserve embedded/non-terminal mentions.
- There are several existing normalization paths: gateway live projection, embedded TUI, auto-reply normalization, outbound payload planning, routeReply, cron delivery, and channel adapters. The best fix needs a deliberate shared boundary to avoid inconsistent behavior.
