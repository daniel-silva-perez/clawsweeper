## Summary

Keep this issue open. Current main still lacks the requested shared prefix/suffix NO_REPLY normalization across gateway display and outbound delivery. There is partial prompt guidance and a cron-specific trailing-token guard, but gateway live/final chat, embedded TUI, routeReply, and outbound payload planning still preserve important mixed NO_REPLY cases.

## Best Possible Solution

Implement a shared silent-token normalization contract used before gateway display and before delivery-bound channel payloads. It should define exact suppression, separated-prefix handling, glued-prefix cleanup, terminal suffix stripping with cleaned content delivery, case handling, media payload behavior, and legitimate embedded NO_REPLY mentions, then add focused regression tests for gateway live/final chat, embedded TUI, outbound payload planning, routeReply, and Telegram/WhatsApp/WebChat-class delivery paths.

## Risks / Open Questions

- A broad suffix rule could remove legitimate user-visible discussion of the literal token; the implementation should target standalone terminal sentinels and preserve embedded/non-terminal mentions.
- The normalization logic is split across gateway chat projection, embedded TUI, routeReply, outbound payload planning, cron delivery, and channel adapters, so a narrow fix can leave leaks on other surfaces.
- Direct-chat silent-reply rewrite policy needs an explicit compatibility decision so exact-token suppression, fallback rewrites, and external channel delivery stay consistent.
