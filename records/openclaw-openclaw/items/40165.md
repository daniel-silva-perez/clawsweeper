## Summary

Keep this issue open. Current main still has only partial NO_REPLY handling: exact-token and lead-fragment suppression exist, suffix stripping exists in some auto-reply/cron paths, but the shared gateway/display/delivery behavior requested here is not implemented across the affected surfaces.

## Best Possible Solution

Keep this open and implement a shared silent-token normalization contract before both gateway display and delivery-bound channel payloads. It should cover exact suppression, separated-prefix discard or a documented preservation rule, glued-prefix cleanup, standalone terminal suffix stripping with cleaned content delivery, mixed case, media payloads, and legitimate embedded NO_REPLY mentions. Add focused regression tests for gateway live/final chat, embedded TUI, outbound payload planning, routeReply/channel delivery used by Telegram/WhatsApp/WebChat, and docs/prompt wording that clarifies NO_REPLY is not a completion marker.

## Risks / Open Questions

- A broad suffix rule could remove legitimate explanatory uses of NO_REPLY; the fix should target standalone terminal sentinel tokens and preserve embedded/non-terminal mentions.
- The normalization paths are split across gateway live/final projection, embedded TUI, auto-reply normalization, routeReply, outbound payload planning, cron delivery, and channel adapters. Fixing only one path would leave observable leaks elsewhere.
- Direct-chat silent-reply policy can rewrite or preserve exact silent payloads depending on settings, so the implementation needs to align robust token cleanup with existing silentReply and silentReplyRewrite semantics.
