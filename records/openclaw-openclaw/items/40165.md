## Summary

Keep this issue open. Current main has partial NO_REPLY hardening, but it does not implement the requested shared prefix/suffix contract: gateway live/final chat still returns mixed terminal-suffix text, outbound payload normalization still preserves mixed prefix and suffix literals, and routeReply still has a regression test that delivers separated leading NO_REPLY text.

## Best Possible Solution

Keep this issue open and implement a shared silent-token normalization contract used by gateway live/final chat projection and outbound delivery planning. It should define exact suppression, separated-prefix suppression, glued-prefix cleanup, terminal-suffix stripping with cleaned-content delivery, case handling, media payload behavior, policy/rewrite interaction, and legitimate embedded NO_REPLY mentions, then update focused tests for gateway chat, embedded TUI, routeReply, outbound payload planning, Telegram-class delivery, and docs.

## Risks / Open Questions

- A broad terminal-suffix rule can remove legitimate user-visible discussion of the literal NO_REPLY token; prior work around #19537 deliberately tightened exact-token matching to avoid suppressing substantive replies.
- The behavior is currently split across gateway projection, auto-reply normalization, outbound payload planning, routeReply, and plugin delivery paths, so a local fix in only one surface would leave leaks or inconsistent behavior elsewhere.
- Media payloads and direct-chat silent-reply rewrite policy need explicit contract coverage so stripping text does not accidentally drop intended attachments or bypass visible fallback behavior where silence is disallowed.
