## Summary

Keep this issue open. Current main still has only partial NO_REPLY hardening: exact-token and some glued-leading-token cases are handled, but the shared gateway/outbound path still preserves separated-prefix and terminal-suffix mixed content such as `NO_REPLY thanks for the update` and `thanks NO_REPLY`. The previous ClawSweeper review and the 2026-04-27 Telegram confirmation remain consistent with an unresolved cross-surface delivery gap.

## Best Possible Solution

Keep the issue open and implement a shared silent-token normalization contract used by both display projection and delivery-bound payload planning. It should define exact suppression, separated-prefix handling, glued-prefix cleanup, terminal suffix stripping with cleaned-content delivery, case handling, media payload behavior, and legitimate embedded `NO_REPLY` mentions. Then update focused regression tests for gateway live/final chat, embedded TUI, outbound payload planning, routeReply, Telegram-class delivery, and the existing partial `normalizeReplyPayload` path; review related PR #62845 as a webchat-fragment candidate, not as the full fix.
