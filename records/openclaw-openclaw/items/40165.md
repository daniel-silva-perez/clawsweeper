## Summary

Keep this issue open. Current main has several adjacent NO_REPLY protections, including exact-token Telegram delivery normalization, but it still does not implement the requested shared gateway/display/delivery behavior for terminal suffix tokens or separated malformed prefixes.

## Best Possible Solution

Implement a shared silent-token normalization contract for gateway-visible and delivery-bound assistant text. Wire it into live chat projection, embedded TUI projection, outbound payload planning, and any intended history/display path. Add focused regression coverage for exact suppression, separated-prefix discard or documented preservation, glued-prefix stripping, standalone terminal suffix stripping, newline suffix stripping, mixed case, media/non-text payloads, and legitimate embedded NO_REPLY mentions.

## Risks / Open Questions

- A broad suffix rule could remove legitimate explanatory mentions of NO_REPLY; the fix should strip only standalone terminal tokens and preserve embedded/non-terminal mentions in tests.
- OpenClaw currently has several normalization paths, including gateway live display, embedded TUI, auto-reply normalization, outbound payload planning, channel delivery, and cron delivery. The shared contract boundary needs deliberate maintainer choice.
