## Summary

Keep open. Current main still compares skill OS requirements as raw platform strings, so `metadata.openclaw.os: ["macos"]` will not match macOS where Node reports `process.platform === "darwin"`; docs and tests cover canonical `darwin`, but not the reported alias path.

## What This Changes

The issue requests that `openclaw skills check` treat a skill declaring `metadata.openclaw.os: ["macos"]` as eligible on macOS/Darwin.

## Best Possible Solution

Add one shared platform canonicalization helper for skill requirement OS lists, remote platform lists, runtime eligibility, remote bin collection, and install-spec filters. Cover at least `macos`/`osx` to `darwin` and `windows` to `win32`, add focused regression tests for the reported `metadata.openclaw.os: ["macos"]` case, and update the skills docs to state the accepted spellings.

## Reproduction Assessment

Yes. A high-confidence static reproduction is `resolveMissingOs({ required: ["macos"], localPlatform: "darwin" })`, which returns `["macos"]`; `openclaw skills check` reaches that path via `buildWorkspaceSkillStatus` and then renders `missing.os`.

## Solution Assessment

Yes, a narrow shared normalizer is the best maintainable fix because the same raw OS comparison appears in requirement evaluation, runtime eligibility, remote skill probing, and install option filtering. Changing only the CLI output or only the docs would leave inconsistent behavior.
