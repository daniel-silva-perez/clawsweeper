## Summary

Keep this PR open. Current main still lacks the requested missing-python fallback for `writeFileWithinRoot` and `copyFileWithinRoot`, so the PR is not obsolete; however, the proposed copy fallback reopens the source path without tying it back to the originally verified source handle, and the user-facing fix still needs the required changelog entry before merge.

## What This Changes

The PR changes fs-safe write/copy paths to fall back to legacy write/copy behavior when the pinned Python helper cannot spawn, and adds focused fs-safe tests for fallback and non-spawn errors.

## Best Possible Solution

Land the missing-python fallback for write/copy, but preserve the verified source identity and size decision when copy fallback has to reopen the source, keep non-spawn helper errors non-fallbacking, add the required changelog entry, and validate with targeted fs-safe tests plus a python3-less helper or Docker smoke.

## Reproduction Assessment

Yes. The linked issue gives a concrete `node:*bookworm-slim` reproduction, and current main source confirms the Docker image lacks python3 by default while write/copy pinned helper failures still normalize to the misleading fs-safe error.

## Solution Assessment

No, not as-is. The high-level fallback direction is the narrow maintainable fix, but the copy fallback should revalidate the reopened source against the original verified file and the PR should carry the required release note before merge.

## Security Review

Concerns:
