## Summary

Keep open. Current main still immediately injects WhatsApp connected/disconnected lifecycle notices into the resolved session's system-event queue, and the shared queue only suppresses exact consecutive duplicate text; there is still no threshold, verbosity setting, reconnect coalescing, or surface-aware lifecycle filter. The related items in the supplied context cover different warning and health-monitor surfaces, so they do not supersede this request.

## What This Changes

The issue requests configurable suppression, throttling, or surface-aware filtering for transient channel connection lifecycle system events, especially short WhatsApp reconnects.

## Best Possible Solution

Implement a tested WhatsApp-owned channel lifecycle notification policy first, with an account/channel setting such as statusEvents and/or notifyAfterMs plus reconnect coalescing for outages that recover before the threshold. Promote a generic plugin SDK helper only if multiple channel plugins need the same policy.
