## Summary

Keep open. Current main still enqueues WhatsApp connected and disconnected lifecycle notices directly into the resolved session system-event queue, with only exact duplicate suppression in the shared queue and no threshold, verbosity setting, reconnect coalescing, or surface-aware filter. The related #39406 and #41940 context is adjacent but does not supersede this channel lifecycle notification policy request.

## What This Changes

The issue requests suppression, throttling, or surface-aware filtering for transient channel connection status system events so short reconnects do not clutter unrelated sessions.

## Best Possible Solution

Add a tested WhatsApp-owned lifecycle notification policy that coalesces short disconnect/reconnect cycles before they become session system events, with explicit config and docs for any user-facing verbosity or notify-after setting. Generalize the helper only after another channel needs the same behavior.
