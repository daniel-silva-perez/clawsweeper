## Summary

Keep open. Current main still immediately enqueues WhatsApp connected and disconnected lifecycle notices into the resolved session system-event queue; the shared queue only suppresses exact consecutive duplicate text, and there is no threshold, verbosity setting, reconnect coalescing, or surface-aware filter. The supplied related items cover adjacent tool-warning delivery and Feishu health-monitor reporting, so they do not supersede this channel lifecycle notification policy request.

## What This Changes

The issue requests suppression, throttling, or filtering of transient channel connection status system events so brief reconnect churn does not clutter active sessions.

## Best Possible Solution

Add a tested WhatsApp-owned lifecycle notification policy that coalesces short disconnect/reconnect cycles before they become session system events, with explicit config/docs for any user-facing verbosity or notify-after setting. Keep the shared system-event queue generic unless multiple channel plugins need the same policy.

## Reproduction Assessment

Yes. The code-level reproduction is high-confidence: any WhatsApp monitor connect followed by a close decision calls enqueueSystemEvent for connected and disconnected notices, and those queued events are formatted into the next prompt; a live Baileys flapping run was not needed to verify that enqueue path.

## Solution Assessment

Yes, a narrow WhatsApp-owned policy is the best current direction. Mutating the shared system-event queue would risk hiding unrelated durable system notices, while a channel-local gate can preserve current defaults and target the reported reconnect churn.
