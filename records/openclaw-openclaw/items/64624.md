## Summary

Keep open. Current main still immediately queues WhatsApp connected/disconnected lifecycle notices into the resolved session, while the shared system-event queue only suppresses exact consecutive duplicate text and has no threshold, verbosity, reconnect coalescing, or surface-aware filter. The related #39406 and #41940 context is adjacent but does not supersede this channel lifecycle notification policy request.

## What This Changes

Request to suppress, throttle, or filter transient channel connection status system events so short WhatsApp reconnects do not clutter unrelated active sessions.

## Best Possible Solution

Implement a tested WhatsApp-owned lifecycle notification policy first: coalesce short disconnect/reconnect cycles before they become session system events, expose a default-compatible config/docs surface for verbosity or notify-after behavior, and only promote a generic plugin SDK helper after another channel needs the same policy.
