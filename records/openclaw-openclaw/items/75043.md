## Summary

Keep open. Current main does not implement `messages.tts.autoEmotion`, and this PR still has a concrete ElevenLabs behavior bug: `similarityBoost`-only settings suppress auto-emotion even though the PR only writes `style`/`stability` and its docs say only those fields opt out. The author also requested maintainer product-boundary review, so the right next step is branch revision plus maintainer review.

## What This Changes

The PR adds opt-in `messages.tts.autoEmotion` inference in speech-core, maps inferred emotions to provider-native TTS controls, and updates provider adapters, config schema/help/labels, generated schema, docs, and focused tests.

## Best Possible Solution

Revise the PR so ElevenLabs auto-emotion is suppressed only by explicit `voiceSettings.style` or `voiceSettings.stability`, add a regression test for similarityBoost-only config, and then have the requested maintainers decide whether the shared speech-core mapping is an acceptable product boundary.

## Reproduction Assessment

Yes. On PR head, configure ElevenLabs with `messages.tts.autoEmotion.enabled: true` and only `voiceSettings.similarityBoost` in provider, persona, or request overrides; happy text then hits `hasElevenLabsVoiceSettings` and returns the existing overrides without adding `style`/`stability`.

## Solution Assessment

No, not as-is. The opt-in mapping approach may be maintainable if maintainers accept the boundary, but the current ElevenLabs guard is broader than both the fields this PR writes and the docs contract.
