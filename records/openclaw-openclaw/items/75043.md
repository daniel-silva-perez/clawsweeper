## Summary

Keep open. Current main does not implement this capability, and the PR still has a concrete ElevenLabs correctness bug: a similarity-only voice setting suppresses auto-emotion even though the PR only writes style/stability and documents only those fields as opt-outs.

## What This Changes

The PR adds opt-in `messages.tts.autoEmotion` inference, provider-specific TTS emotion mappings, provider adapter changes, config schema/help/labels, generated schema, docs, and focused tests.

## Best Possible Solution

Revise the ElevenLabs explicit-setting detection so only `voiceSettings.style` or `voiceSettings.stability` suppress auto-emotion, add a regression test for similarityBoost-only config, then have the requested maintainers decide whether shared speech-core is the right home for provider-aware emotion mapping.

## Reproduction Assessment

Yes. Configure ElevenLabs with `messages.tts.autoEmotion.enabled: true` and only `voiceSettings.similarityBoost`, then synthesize happy text; the PR predicate returns explicit=true before building the auto `style`/`stability` override.

## Solution Assessment

No, not as-is. The opt-in mapping approach may be maintainable if maintainers approve the boundary, but the current ElevenLabs guard is broader than the provider fields the PR actually owns and broader than the docs describe.
