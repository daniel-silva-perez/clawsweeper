## Summary

Keep open. Current main has ordered reactive model fallback and provider usage reporting, but it still lacks a proactive token/quota/cost threshold policy that feeds model selection, reverse failback semantics, or user-facing threshold-switch notifications; related #45771 remains open and overlaps on usage-aware pacing without subsuming cross-provider threshold-based failover.

## What This Changes

The issue requests automatic model failover between configured providers based on token or cost thresholds, with optional reverse failback and user notifications.

## Best Possible Solution

Add a documented core usage-aware model routing policy that evaluates provider usage snapshots and local token/cost state before candidate selection, preserves existing reactive error/cooldown fallback, defines reset and reverse-failback semantics, and exposes clear user notifications/status for threshold-triggered switches.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The gap is high-confidence by source and docs inspection: current main has configured fallbacks and usage snapshots, but no token/quota/cost threshold policy feeding model routing.

## Solution Assessment

Unclear until maintainers define the product policy. The capability likely belongs in core fallback/model-selection policy rather than a standalone plugin because it changes normal provider routing, but the exact thresholds, stale-data behavior, failback rules, and notification surfaces need design first.
