## Summary

Keep open. Current main has ordered reactive model fallback and provider usage reporting, but it still does not implement proactive token, quota, or cost threshold routing, optional reverse failback, or user-facing threshold-switch notifications. The related pace-aware rate-limiting issue is adjacent throttling work, not a canonical replacement for cross-provider threshold failover.

## What This Changes

Request automatic switching between configured LLM providers based on token or cost consumption thresholds, with ordered fallback models, optional reverse failback, and notifications when switching occurs.

## Best Possible Solution

Define a core usage-aware model routing policy that evaluates provider usage snapshots and local token/cost state before candidate selection, preserves the existing reactive failure and cooldown fallback path, documents reset and reverse-failback semantics, and emits explicit status or notification events for threshold-triggered switches.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The gap is high-confidence from source and docs inspection: current main has static configured fallback chains and usage reporting, but no threshold policy feeding model routing.

## Solution Assessment

Unclear until maintainers define the product policy. The capability likely belongs in core model selection because OpenClaw keeps the generic inference loop while provider plugins supply usage reporting, but a narrow automated repair is not safe without threshold and notification semantics.
