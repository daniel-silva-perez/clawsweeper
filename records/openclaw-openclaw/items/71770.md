## Summary

Keep open. Current main still has reactive model failover based on auth/profile cooldowns and failover-worthy errors, plus provider usage reporting surfaces, but it does not implement proactive token/quota/cost-threshold routing before a model call, reverse failover after quota reset, or threshold-switch notifications. The existing ClawSweeper comment reached the same conclusion, and current main at 7ddd815e still matches that behavior.

## Best Possible Solution

Keep this issue open for maintainer product/design work in the model selection and fallback path. A robust design would add a documented threshold policy surface, evaluate provider usage snapshots and local session cost/token state before candidate selection, integrate with runWithModelFallback without weakening existing cooldown/error fallback, emit lifecycle/user-visible notifications for threshold switches, and define reset/reverse-failover semantics. If maintainers want this policy to be plugin-owned, core should first add a deliberate usage-aware routing API.

## Risks / Open Questions

- Provider usage data is uneven across providers, so threshold semantics need clear behavior for providers with no quota API, multiple overlapping windows, subscription-style quotas, or only post-response token usage.
- Automatic switching can increase costs or surprise users unless threshold units, fallback order, reverse-failover behavior, and notifications are explicit and auditable.
- A plugin can make simple prompt-based model overrides today, but current plugin hooks do not expose enough usage-aware routing state to treat this as already satisfied outside core.
