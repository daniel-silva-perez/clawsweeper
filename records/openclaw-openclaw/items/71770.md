## Summary

Keep open. Current main has reactive model failover based on configured fallback chains, auth/profile cooldown state, and failover-worthy provider errors, plus separate provider usage reporting surfaces. It does not implement proactive token/quota/cost threshold routing, reverse failover after reset, or user-visible threshold-switch notifications. Related #45771 overlaps on pace/usage awareness but is not a complete duplicate of this ordered model failover request.

## Best Possible Solution

Keep this issue open for maintainer product/design work in the model selection and fallback path. A good implementation would define a documented usage-aware routing policy, evaluate provider usage snapshots and local session cost/token state before selecting a candidate, preserve existing runWithModelFallback cooldown/error behavior, emit user-visible lifecycle notifications for threshold switches, define reverse-failover/reset semantics, and coordinate with #45771 so pace-aware throttling and threshold-based model switching share one policy model.
