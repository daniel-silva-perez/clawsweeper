## Summary

Keep open. Current main has ordered reactive model fallback and separate usage/quota reporting, but source and docs still show no proactive token/quota/cost threshold policy feeding model selection, no reverse failback rule, and no user-visible threshold-switch notification path. Related #45771 overlaps on pace-aware usage behavior but remains open and does not subsume cross-provider threshold-based model failover.

## What This Changes

Request automatic switching between configured LLM providers based on token or cost thresholds, with ordered fallbacks, optional reverse failback, and notifications.

## Best Possible Solution

Add a documented core usage-aware model routing policy that evaluates provider usage snapshots and local token/cost state before candidate selection, preserves existing error/cooldown fallback behavior, defines reset and reverse-failover semantics, and coordinates with #45771 instead of treating this as a standalone plugin request.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The gap is reproducible by source and docs inspection: configured fallbacks and usage snapshots exist, but no token/cost threshold policy feeds model candidate selection.

## Solution Assessment

Unclear until maintainer design is done. The capability likely belongs in core model selection/fallback because it changes normal provider routing, but threshold, stale-data, reverse-failover, and notification rules need product design first.
