## Summary

Keep open. Current main has ordered reactive model fallback and provider usage reporting, but it still does not implement token, quota, or cost threshold-driven model routing, reverse failback, or user-facing threshold-switch notifications. Related #45771 is adjacent pace-aware throttling work, not a canonical replacement for cross-provider threshold failover.

## What This Changes

Request automatic switching among configured LLM providers based on token consumption or cost thresholds, with ordered fallbacks, optional reverse failback, and notifications.

## Best Possible Solution

Define a core usage-aware model routing policy that reads provider usage snapshots and local cost/token state before candidate selection, preserves the existing reactive error/cooldown fallback path, documents reset and reverse-failback behavior, and exposes explicit status/notification events for threshold-triggered switches.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The gap is high-confidence from source and docs inspection: current main has configured fallback chains plus usage snapshots, but no threshold policy feeding model routing.

## Solution Assessment

Unclear until maintainers define the product policy. The capability likely belongs in core model-selection/fallback policy rather than a standalone plugin because it changes normal provider routing, but threshold semantics and notifications need design first.
