## Summary

Keep open. Current main and the latest release still use the daily implicit reset default, so the PR is not obsolete, but the submitted patch is not mergeable as-is because its new guard suppresses the group/thread idle default whenever any unrelated resetByType entry exists. It also changes default context retention in shared channels, which needs maintainer product/privacy approval before a refreshed fix lands.

## What This Changes

The PR changes implicit session reset defaults so group and thread sessions use idle reset with a 10080-minute window while direct/DM sessions remain daily, and updates reset-policy tests.

## Best Possible Solution

If maintainers accept the behavior change, put the type-aware implicit default in src/config/sessions/reset-policy.ts, keep explicit base/type/channel overrides authoritative, add exact 10080 coverage for no-config and unrelated-override group/thread paths, and update the user-facing session docs plus changelog.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Scope the default check to this reset type:** `src/config/sessions/reset.ts:114-115`
  - body: The new impliedDefaultMode uses !hasExplicitReset, but hasExplicitReset becomes true when any resetByType entry exists. A config with only dm or direct configured therefore still resolves group/thread sessions to daily, so the primary fix does not apply for a common override shape.
  - confidence: 0.93
- **[P3] Assert the exact group idle window:** `src/config/sessions/sessions.test.ts:151`
  - body: The new test only checks that group idleMinutes is not 45, which would also pass for undefined or another wrong value. Pin the expected 10080-minute default so the behavior this PR adds is actually covered.
  - confidence: 0.82
