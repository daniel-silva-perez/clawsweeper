## Summary

Keep open. Current main still uses the old implicit daily reset behavior, and the submitted patch has a blocking guard bug that prevents the new group/thread default when an unrelated per-type override exists. The branch is also stale against the current reset-policy split, and the default shared-channel retention change needs maintainer privacy/product acceptance before merge.

## What This Changes

The PR changes implicit group and thread session reset defaults to idle with a 10080-minute window while leaving direct/DM sessions on daily reset, and updates reset-policy tests.

## Best Possible Solution

If maintainers accept the behavior change, implement the type-aware implicit default in `src/config/sessions/reset-policy.ts`, keep explicit base/type/channel overrides authoritative, add exact `10080` coverage for no-config and unrelated-override group/thread paths, and update the session docs plus changelog.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Scope the default check to this reset type:** `src/config/sessions/reset.ts:114-115`
  - body: The new `impliedDefaultMode` uses `!hasExplicitReset`, but `hasExplicitReset` becomes true when any `resetByType` entry exists. A config with only `dm` or `direct` configured therefore still resolves group/thread sessions to `daily`, so the primary fix does not apply for a common override shape.
  - confidence: 0.93

  - body: The updated test only checks that group `idleMinutes` is not `45`, which would also pass for `undefined` or another wrong value. Pin the expected `10080`-minute default so the behavior this PR adds is actually covered.
  - confidence: 0.82
