## Summary

Keep open. The requested behavior is not implemented on current main or the latest release, and the PR targets a real session-default gap, but the submitted patch has a blocking defaulting bug and needs maintainer product/privacy review before merge.

## What This Changes

The PR changes implicit session reset defaults so group and thread sessions use idle reset with a 10080-minute window while direct/DM sessions remain daily, and updates reset-policy tests.

## Best Possible Solution

If maintainers accept the behavior change, implement the type-aware implicit default in src/config/sessions/reset-policy.ts, keep explicit base/type/channel overrides authoritative, add exact 10080 coverage for no-config and unrelated-override group/thread paths, and update the user-facing session docs plus changelog.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Scope the default check to this reset type:** `src/config/sessions/reset.ts:114-115`
  - body: The new impliedDefaultMode uses !hasExplicitReset, but hasExplicitReset becomes true when any resetByType entry exists. A config with only dm or direct configured therefore still resolves group/thread sessions to daily, so the primary fix does not apply for a common override shape.
  - confidence: 0.93
- **[P3] Assert the exact group idle window:** `src/config/sessions/sessions.test.ts:151`
  - body: The new test only checks that group idleMinutes is not 45, which would also pass for undefined or another wrong value. Pinning the expected 10080-minute default would catch regressions in the behavior this PR is adding.
  - confidence: 0.82
