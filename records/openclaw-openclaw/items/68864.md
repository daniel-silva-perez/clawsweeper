## Summary

Keep this issue open. Current main still builds the Web UI checkpoint list from session-store `compactionCheckpoints` metadata, the gateway has no disk-discovery fallback for historical `.checkpoint.*.jsonl` files, and the directly related implementation candidate was closed unmerged.

## What This Changes

The issue requests that the Web UI Sessions list display existing on-disk compaction checkpoint files for historical sessions instead of showing an empty checkpoint modal.

## Best Possible Solution

Add a narrow gateway-side reconciliation path that discovers valid checkpoint transcript files for the requested session when stored checkpoint metadata is empty or stale, synthesizes safe metadata for list/get, and keeps branch/restore behavior consistent with the existing checkpoint and session-store contracts.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to create or preserve valid `.checkpoint.*.jsonl` files for a session whose store entry lacks `compactionCheckpoints`, then call `sessions.compaction.list` or open the Sessions UI checkpoint modal and observe an empty list.

## Solution Assessment

Yes. The narrowest maintainable fix is server-side discovery/reconciliation in the gateway because the UI already calls the right RPC and the existing parser already defines the safe checkpoint filename shape.
