## Summary

Keep open. Current main has meaningful partial restart protection, including the PR #71465 active-run drain path, lock-based main-session recovery, subagent orphan recovery, and cron startup interruption bookkeeping, but it still does not implement the full requested graceful-restart product surface: readiness gating/force semantics, durable drain-window message replay, configurable restart manifest behavior, and cron retry/requeue semantics all remain incomplete or deliberately narrower.

## What This Changes

The issue requests graceful gateway restart behavior with session interruption recovery, active-run protection, readiness gating, drain-time message replay, and cron/subagent interruption handling.

## Best Possible Solution

Keep this open as the umbrella until maintainers either split the remaining scope into narrower issues or define a single restart-recovery design that builds on the shipped drain, main-session, subagent, cron-state, and restart-sentinel mechanisms without adding a competing lifecycle path.

## Reproduction Assessment

Yes, for the remaining current behavior we have a high-confidence source/test reproduction path: `markGatewayDraining()` rejects new enqueues, gateway restart lacks a force/readiness gate, and cron restart-catchup tests assert interrupted jobs are marked failed or disabled rather than requeued. No live multi-agent restart was run because this review was read-only.

## Solution Assessment

No, closing this now is not the best way to solve the issue. The current implementation is a useful partial solution, but the remaining request needs maintainer scoping across gateway lifecycle, sessions, cron, subagents, and channel delivery rather than a blind follow-up patch.
