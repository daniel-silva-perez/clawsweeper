## Summary

Keep open. Current main has shipped partial restart-drain and recovery work, but the maintainer discussion explicitly left this broader session-recovery scope open and the source still lacks the proposed manifest/readiness-gate/drain-replay/cron-retry behavior as a complete product contract.

## What This Changes

Request graceful gateway restart behavior with session recovery, interruption propagation, restart readiness checks, drain-aware message replay, and retry handling for interrupted cron work.

## Best Possible Solution

Keep this open as the umbrella until maintainers decide the product shape: either define a core restart-recovery contract covering durable interruption manifests, drain-message replay, readiness/force semantics, and cron retry policy, or split those into narrower canonical issues tied to the existing partial drain and recovery hooks.

## Reproduction Assessment

Yes. Source-level repro paths are clear: set the command queue to gateway-draining and new work rejects instead of being durably replayed, and start cron with persisted runningAtMs to see jobs marked interrupted/failed rather than retried.

## Solution Assessment

No. The current partial implementation is a good foundation for active-run drain and limited recovery, but it is not the complete requested solution; the safer next step is maintainer-led scope splitting or a design pass before another broad PR.
