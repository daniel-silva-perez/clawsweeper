## Summary

Keep open. The reported heartbeat runaway is valid, but this patch would also throttle documented immediate wake paths and can consume retryable busy wakes before a heartbeat run actually happens, so it is not safe to merge as-is.

## What This Changes

The PR adds a centralized heartbeat cooldown helper with tests, routes targeted and broadcast heartbeat wakes through it, and tracks per-agent run timestamps plus flood state to limit repeated non-interval wakes.

## Best Possible Solution

Land a centralized heartbeat cooldown that stops repeated `exec-event` feedback while preserving explicit immediate wake contracts, task-completion delivery, and retryable busy behavior; add targeted regressions for those contracts plus the original repeated exec-event case before merging.

## Reproduction Assessment

Yes. The original bug has a high-confidence unit reproduction from the PR body and current main source: repeated scoped `exec-event` wakes bypass `nextDueMs`; the new review findings also have direct source-level reproductions through `wake`/`background-task` reasons after a recent heartbeat and targeted event wakes that return `requests-in-flight`.

## Solution Assessment

No. A shared cooldown helper is the right seam, but this implementation applies the interval gate too broadly and records cooldown state before knowing whether a heartbeat run actually happened; the safer fix is to distinguish immediate wake reasons and avoid seeding cooldown/flood state for retryable busy skips.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Preserve immediate wake-now paths:** `src/infra/heartbeat-cooldown.ts:130-131`
  - body: `shouldDeferWake` now returns `not-due` for every event-driven reason once an agent has run and `nextDueMs` is in the future. That includes `wake` from `openclaw system event --mode now` and `background-task` completion wakes, both of which are documented as immediate. After any recent heartbeat those requests are silently delayed until the next scheduled tick, so the cooldown needs to exempt explicit immediate notification paths or carry an explicit immediacy flag.
  - confidence: 0.93
- **[P2] Do not count busy skips as completed runs:** `src/infra/heartbeat-runner.ts:1825`
  - body: The runner records `lastRunStartedAtMs` before calling `runOnce`. If a targeted `exec-event` or task wake hits `requests-in-flight`, the wake layer retries the same reason, but the retry is now converted into `not-due` or `min-spacing` instead of remaining retryable, so the queued event can wait until the next scheduled heartbeat. Record cooldown bookkeeping only after a non-retryable attempt, or roll it back for retryable busy skips.
  - confidence: 0.87
