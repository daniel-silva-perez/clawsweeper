## Summary

Keep open: current main does not implement the new batch tool, so this PR is not obsolete, but the proposed patch is not merge-ready. The nested-lane performance problem it tries to solve was already addressed by merged #67785 using per-session nested lanes, while this branch forks current `sessions_send` behavior and misses several maintained correctness and session-boundary contracts.

## What This Changes

The PR adds a `sessions_send_concurrent` agent tool with docs, tests, tool-catalog registration, concurrent batch sends, A2A follow-up handling, and gateway nested-lane concurrency changes.

## Best Possible Solution

Build the batch surface as a thin orchestrator over the maintained `sessions_send` contracts or a shared helper extracted from them, preserving inter-session annotation, target validation, fixed enqueue timeout, reply correlation, per-session nested lanes, and the owned ACP child skip before adding focused docs and tests.

## Reproduction Assessment

Yes for the review findings: the latest PR diff can be checked directly against current main’s `sessions_send`, A2A, lane, dependency, docs, and regression-test contracts. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No. The branch adds a useful batch-send surface, but the implementation forks a sensitive maintained path and widens a shared lane that current main intentionally keeps serialized; the safer path is to reuse or extract the existing `sessions_send` behavior.

- **[P2] Use the repo's typebox package:** `src/agents/tools/sessions-send-concurrent-tool.ts:3`
  - body: Current main imports `Type` from the direct `typebox` dependency. This new file imports `@sinclair/typebox` without adding that package directly, so a clean install can fail to resolve the module.
  - confidence: 0.9
- **[P2] Preserve inter-session prompt annotation:** `src/agents/tools/sessions-send-concurrent-tool.ts:360`
  - body: The primary send passes `target.message` directly into the target session. Current `sessions_send` wraps injected text with the inter-session marker so the target sees it as tool-routed data instead of direct user-authored instruction.
  - confidence: 0.86
- **[P2] Reject targets that provide both sessionKey and label:** `src/agents/tools/sessions-send-concurrent-tool.ts:132-133`
  - body: A target can provide both `sessionKey` and `label`; this code keeps both and later ignores the label when `sessionKey` is present. `sessions_send` rejects that ambiguous combination to avoid silently routing generated payloads to the wrong session.
  - confidence: 0.9
- **[P2] Keep the shared nested lane serialized:** `src/gateway/server-lanes.ts:10`
  - body: Current main scopes nested work with per-session lanes and has a regression test that the shared `CommandLane.Nested` lane stays at default concurrency. Raising the shared lane here weakens that scheduler invariant.
  - confidence: 0.88
- **[P2] Route target sends through session lanes:** `src/agents/tools/sessions-send-concurrent-tool.ts:365`
  - body: The primary target run is enqueued on the shared nested lane. Current main uses `resolveNestedAgentLaneForSession(...)` so different target sessions run independently while repeated sends to the same session remain ordered.
  - confidence: 0.87
- **[P2] Route A2A steps through session lanes:** `src/agents/tools/sessions-send-concurrent-a2a.ts:177`
  - body: The concurrent A2A helper forces ping-pong and announce work onto the shared nested lane. Those follow-up turns should use the current session-scoped lane helper for the active session.
  - confidence: 0.84
- **[P2] Read the reply produced by this run:** `src/agents/tools/sessions-send-concurrent-tool.ts:521-529`
  - body: After `agent.wait`, the tool reads the latest assistant message from session history without a baseline or run correlation. If the session already had a reply or another send completes nearby, the result and A2A flow can use stale or unrelated text.
  - confidence: 0.84
- **[P2] Preserve the owned ACP child A2A guard:** `src/agents/tools/sessions-send-concurrent-tool.ts:405-417`
  - body: Current `sessions_send` skips ping-pong/announce when the requester owns its background ACP child because that child already reports completion through the internal completion path. The concurrent tool starts A2A follow-up unconditionally, which can re-enter that parent/child path.
  - confidence: 0.8

## Security Review

Status: needs_attention

Summary: The diff introduces session-boundary risks by bypassing current inter-session annotation and owned ACP child A2A guards.

Concerns:

- **[medium] Preserve inter-session instruction provenance:** `src/agents/tools/sessions-send-concurrent-tool.ts:360`
  - body: The batch send injects raw target messages instead of the current `[Inter-session message ... isUser=false]` annotation, weakening the boundary that tells the receiving model this content came from a tool-routed session message rather than the end user.
  - confidence: 0.82
- **[medium] Preserve the owned ACP child A2A guard:** `src/agents/tools/sessions-send-concurrent-tool.ts:405`
  - body: The PR starts A2A follow-up for accepted targets without mirroring the current `sessions_send` guard for parent-owned background ACP child sessions. That path can wake the parent and feed a parent response back into the child, crossing the session-control boundary current main intentionally avoids.
  - confidence: 0.78
