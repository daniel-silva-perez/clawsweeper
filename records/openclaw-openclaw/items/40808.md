## Summary

Keep this PR open. Current main still does not implement or document a `sessions_send_concurrent` tool or equivalent 1-20 target batch/progress result surface. Main has improved the underlying nested-lane direction with per-session nested lanes for existing `sessions_send`, but that only partially overlaps the PR motivation and does not replace the proposed tool API.

## Best Possible Solution

Keep this PR open for maintainer review or ask the author to rebase and narrow it onto current main. The best implementation path is to build any batch API on top of the current `sessions_send` validation, visibility guard, A2A flow, fixed enqueue timeout, and per-session nested-lane helpers; avoid widening the shared nested lane unless a maintainer explicitly wants that policy; and add tests for true multi-target concurrency, fire-and-forget A2A, per-target timeout/error isolation, ambiguous target rejection, session visibility, and sandboxed access.
