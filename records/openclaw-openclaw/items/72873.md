## Summary

Keep this issue open. Current main still accepts Telegram `edited_message` updates but does not register a Telegram inbound handler for them, and the linked closing PR is still the active implementation path rather than a reason to close the tracker early.

## What This Changes

The issue requests Telegram inbound handling for `edited_message` updates so streamed bot edits are debounced and dispatched instead of leaving OpenClaw with only the initial partial message.

## Best Possible Solution

Keep the fix extension-owned in the Telegram plugin: add an `edited_message` handler that reuses the normal inbound pipeline, coalesces rapid edits by chat/message id, avoids bot-authored echoes, and adds focused tests for registration, debounce, dedupe, and dispatch behavior.

## Reproduction Assessment

Yes. A synthetic grammY `edited_message` update reproduces the drop because current main requests/accepts that update type but has no registered handler to pass it into the inbound message pipeline.

## Solution Assessment

Yes for the implementation direction: a narrow Telegram plugin handler with debounce coverage is the maintainable fix boundary. The open PR should be reviewed, landed, or replaced before this tracker closes.
