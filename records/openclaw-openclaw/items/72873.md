## Summary

Keep this issue open. The Telegram plugin still requests/accepts `edited_message` updates but does not register an inbound `edited_message` handler, and the linked closing PR #73104 remains open and unmerged, so the right cleanup path is normal PR review rather than closing this tracker.

## What This Changes

The issue requests Telegram inbound handling for `edited_message` updates so streaming edits are debounced/coalesced and processed instead of leaving the agent with only the initial partial message.

## Best Possible Solution

Keep the fix extension-owned in the Telegram plugin: register an `edited_message` handler that reuses the normal inbound message pipeline, coalesces rapid edits by chat/message id, skips bot-authored echoes, and adds focused Telegram tests for registration, debounce, and dispatch behavior.

## Reproduction Assessment

Yes. A focused unit test or synthetic grammY context for an `edited_message` update reproduces the drop because allowed updates include edited messages while current main has no handler to dispatch them.

## Solution Assessment

Unclear for the exact open PR until it is reviewed, but the maintainable direction is clear: a narrow Telegram plugin handler with debounce coverage is the right boundary, not a core migration or cross-channel rewrite.
