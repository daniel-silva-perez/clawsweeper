## Summary

Keep this issue open. Current main still accepts Telegram edited-message updates at the allowed-update layer but does not register an inbound `edited_message` handler, and the linked open PR #73104 uses closing syntax for this issue, so the right cleanup path is to review and land or replace that PR before closing.

## What This Changes

Handle Telegram `edited_message` updates with debounce/deduplication so streaming bot edits are processed as final inbound content instead of only the initial partial message.

## Best Possible Solution

Keep the fix extension-owned in the Telegram plugin: add an `edited_message` handler that reuses the normal inbound message pipeline, coalesces rapid edits by chat/message id, skips bot-authored echoes, and adds focused Telegram tests for handler registration, debounce, and dispatch behavior.

## Reproduction Assessment

Yes. A focused unit test or synthetic grammY update for `edited_message` against current main would reproduce the drop because allowed updates include edited messages while no handler dispatches them.

## Solution Assessment

Unclear for the exact open PR until it is reviewed, but the maintainable direction is clear: a narrow Telegram plugin handler with debounce coverage is the right boundary, not a core migration or cross-channel rewrite.
