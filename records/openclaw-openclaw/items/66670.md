## Summary

Current main still does not populate `task_runs.agent_id` for `sessions_spawn` task records. The native and ACP spawn paths both resolve the target agent and encode it in `childSessionKey`, and the task registry/SQLite layer already supports `agentId`, but the task-run creation calls omit it.

## Best Possible Solution

Keep this issue open and fix the core attribution plumbing. Add `agentId` to `RegisterSubagentRunParams`, pass native `targetAgentId` into `registerSubagentRun`, pass ACP `targetAgentId` into both `createRunningTaskRun` calls, and add regression coverage proving cross-agent `sessions_spawn` writes both `TaskRecord.agentId` and SQLite `task_runs.agent_id`. Treat historical backfill or doctor repair as a separate product/maintenance decision.

## Risks / Open Questions

- Plugin and dashboard attribution remains fragile because readers must parse `childSessionKey` instead of using `task_runs.agent_id` directly.
- Native subagent and ACP spawn paths should be fixed together; changing only one would leave inconsistent attribution.
- A separate maintainer decision may be needed on whether to backfill historical rows with null `agent_id`; the core forward fix is still missing.
