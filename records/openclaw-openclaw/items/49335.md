## Summary

Keep this PR open. Current main still synchronously reads and parses the full session transcript before chat.history applies response limits, so the PR's central freeze fix is not implemented. The PR remains a plausible Gateway hot-path fix, but it needs maintainer review/rebase because the current branch has transcript semantics the PR must preserve: active-branch reads, compaction markers, stable __openclaw.seq metadata, pagination/SSE cursors, and media-heavy history sanitization.

## Best Possible Solution

Keep the PR open and ask for a targeted rebase onto current main. The best fix is a bounded transcript reader inside readSessionMessages that limits read/parse cost while preserving active-branch semantics, compaction dividers, sanitized media-heavy history, absolute monotonic __openclaw.seq values, and cursor/SSE behavior, with focused regression tests for large JSONL files and oversized UTF-8 records.
