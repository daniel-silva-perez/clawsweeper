## Summary

Keep this PR open. Current main still synchronously reads and parses the full transcript in readSessionMessages(), so the user-visible chat.history freeze problem is not implemented on main. The PR is a plausible Gateway fix, but the provided review discussion still raises correctness issues around stable sequence numbers, pagination cursors, compaction/non-message records, and tail truncation semantics.

## Best Possible Solution

Keep the PR open and require a targeted Gateway implementation that bounds transcript reading/parsing without breaking active-branch handling, compaction markers, sanitized media-heavy history, or stable absolute session-history sequence numbers. The likely path is a bounded or streaming transcript reader in the Gateway transcript utility layer with focused tests for large JSONL files, oversized lines, non-message records, active branches, and pagination cursors.

## Risks / Open Questions

- Merging the PR as-is could regress session-history pagination or SSE cursor behavior if tail truncation renumbers __openclaw.seq or placeholders omit seq metadata.
- A raw line-size guard before record-type detection could turn oversized compaction or other non-message transcript records into synthetic chat messages.
- The PR needs regression coverage for large transcripts, oversized UTF-8 JSONL records, active-branch transcripts, compaction records, sanitized media-heavy history, and cursor stability.
