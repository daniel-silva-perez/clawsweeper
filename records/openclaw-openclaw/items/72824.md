## Summary

Keep open: current main still has the reported GitHub Copilot capability mismatch, and the same contributor already has an open paired fix PR. Closing this issue before that PR is reviewed and merged would be premature, and a separate automated repair job would duplicate the existing implementation candidate.

## What This Changes

Make the GitHub Copilot plugin recognize Copilot's Claude Opus 4.6/4.7 1m variants as 1,000,000-token-context, 64,000-output-token models and allow `xhigh` for `claude-opus-4.7-1m-internal`.

## Best Possible Solution

Land a reviewed fix that aligns GitHub Copilot model metadata for the two Claude Opus 1m IDs across dynamic resolution, seeded defaults, status/context accounting, and thinking-level validation, with regression tests. After the narrow fix, consider a separate provider-discovery improvement that imports capabilities from Copilot's `/models` response instead of maintaining more static per-id entries.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: resolve `github-copilot/claude-opus-4.7-1m-internal` as an unknown Copilot model and current main returns 128K/8192 metadata, while the Copilot thinking profile omits `xhigh` because the model id is not in `COPILOT_XHIGH_MODEL_IDS`.

## Solution Assessment

Yes for the immediate bug: a narrow plugin-owned capability override plus tests is the least disruptive fix while the upstream model list is small. A live `/models` capability import would be cleaner long-term, but it is a broader async discovery/caching change and should not block the focused repair.
