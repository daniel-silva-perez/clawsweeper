## Summary

Keep open. Current main still does not detect the reported parent/child workspace split: the detector is a no-op, the doctor/status warning only emits from that detector output, and the docs describe generic extra workspaces without explaining Docker-to-host `workspace-local` versus `workspace-local/local` expectations. The reporter’s latest comment also narrowed the request to read-only detection, status reporting, and docs rather than automatic secret migration.

## What This Changes

Add doctor/status and documentation support for Docker-to-host migrations that leave operational files under `workspace-local` while the active workspace is `workspace-local/local`.

## Best Possible Solution

Implement a read-only doctor/status check that recognizes an active `workspace-local/local` workspace with relevant parent-held operational markers, prints both paths clearly, and updates the workspace and Docker docs with canonical post-migration guidance while preserving intentional per-agent child workspaces.

## Repair Work Prompt

Fix this issue by adding a narrow, read-only doctor/docs repair for Docker-to-host split workspaces. Detect when the active default-agent workspace resolves to a child such as `~/.openclaw/workspace-local/local` and the parent `~/.openclaw/workspace-local` contains relevant operational markers that users may reasonably confuse with active workspace state, such as bootstrap/memory files, `skills/`, `memory/`, or operator-held `secrets/` files. Report the active workspace and parent-held markers clearly in doctor/status output, but do not move, copy, delete, or canonicalize parent files. Preserve intentional per-agent child workspaces created by `resolveAgentWorkspaceDir()` for non-default agents; the warning should be about parent-held operational files near the active workspace, not a blanket ban on child workspaces. Update public docs to explain canonical post-migration workspace expectations and the difference between documented SecretRef/config state and operator/skill-specific files like `workspace-local/secrets/homeassistant.md`. Add focused Vitest coverage for the detector/status behavior and docs coverage if required. Changelog likely not required unless maintainers treat the doctor warning as user-facing release-note material. Do not change SecretRef semantics, do not introduce startup/load-time migrations, and do not add automatic repair without maintainer approval.
