## Summary

Keep this issue open. Current main still exposes `session_status` as a `/status`-style text response with only `ok`, `sessionKey`, `changedModel`, and `statusText` in structured details; it does not provide the requested remaining time or token budget fields. The related open #57705 is adjacent but tracks sub-agent concurrency capacity, not this issue's remaining session time/context budget request, so it is not a duplicate close target.

## What This Changes

The issue requests structured remaining time and token budget data from `session_status`, or a lightweight `session_budget` tool, so skills can gate expensive sub-agent workflows before spawning them.

## Best Possible Solution

Add a typed budget object to `session_status.details` using existing status/context accounting for token headroom and live run timeout data when available, returning explicit `null`/omitted values when a metric is unavailable. Document the field semantics and add focused tests around current-session status, unavailable live-run timing, and token headroom calculation.

## Reproduction Assessment

Yes. A source-level reproduction is enough here: the current `session_status` tool returns a text status card and a small details object without the requested fields, and the focused tests only assert `ok`/`statusText`.

## Solution Assessment

Yes, adding structured fields to `session_status.details` is the narrowest maintainable direction because agents already use that tool for status decisions. A separate `session_budget` tool should only be chosen if maintainers want to avoid the existing status-card construction cost or keep budget metrics as a smaller contract.
