## Summary

Keep open. Current main does not implement the proposed A2A ingress echo or dual-channel relay surface, and this draft PR still carries concrete blocking correctness/security issues around relay idempotency, guard ordering, and strict delivery semantics.

## What This Changes

The PR adds config-gated `sessions_send` ingress echo, nested relay guarding, dual-channel A2A relay mirroring, docs/plans, and targeted tests across session tools, config schema/help, gateway sessions_send, and outbound message send guarding.

## Best Possible Solution

Land a narrower, maintainer-reviewed A2A relay change that preserves the existing provenance contract, performs all deny/loop checks before any channel side effect, uses stable per-run relay ids, either enforces `requireDelivery` or removes the knob until implemented, and updates generated config/docs together.

## Reproduction Assessment

Yes. A focused source-level reproduction is clear: enable relay with the default `timeoutSeconds > 0` path and observe `runContextId` becomes `unknown`; enable ingress echo from an inter-session caller and observe echo happens before the nested guard; enable relay `requireDelivery` and observe send failures are swallowed.

## Solution Assessment

No. The product direction is plausible, but this PR is not the best mergeable form yet because the common wait path uses unstable relay identity, the guard runs after a channel side effect, and the strict relay config is documented but ineffective.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Run the nested-send guard before ingress echo:** `src/agents/tools/sessions-send-tool.ts:280-285`
  - body: The ingress echo is sent before the new nested `sessions_send` guard checks the caller history. A nested relay that should be blocked can still post the original payload to the target channel and only then return `forbidden`, which defeats the guard's purpose and creates an unintended channel side effect. Move the guard before any ingress echo or relay send.
  - confidence: 0.87
- **[P1] Pass the run id into wait-path relay sends:** `src/agents/tools/sessions-send-tool.ts:542`
  - body: In the normal `timeoutSeconds > 0` path, `startA2AFlow` is called without `runId`, so `runSessionsSendA2AFlow` uses `unknown` for the handoff id and relay idempotency keys. Repeated A-to-B relays will reuse keys like `unknown:relay:<from>:<to>:<index>` and can be deduped or mis-labeled. Pass the actual `runId`, as the fire-and-forget path already does.
  - confidence: 0.9
- **[P2] Honor strict relay delivery failures:** `src/agents/tools/sessions-send-tool.a2a.ts:88-95`
  - body: The PR adds `session.agentToAgent.relay.requireDelivery` and documents strict delivery, but `relayTurn` catches every `send` failure and only logs it. With `requireDelivery=true`, failed or missing relay delivery still completes as best-effort, so the strict config is ineffective.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff introduces channel-visible A2A side effects, and one guard-ordering bug can post to a target channel before a nested relay is rejected.

Concerns:

- **[medium] Nested relay guard runs after channel echo:** `src/agents/tools/sessions-send-tool.ts:280`
  - body: When ingress echo is enabled, the PR sends the original payload to the target channel before checking whether the caller is already handling an inter-session `sessions_send`. A blocked nested relay can therefore leak or duplicate content in a channel despite returning `forbidden`.
  - confidence: 0.87
