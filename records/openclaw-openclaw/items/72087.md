## Summary

Keep open. The report is fresh, has detailed lower-layer controls plus an independent macOS reproduction on the latest stable release, and current main still contains the same entry/bootstrap and Codex image transport surfaces without a linked or discoverable fix for this issue.

## What This Changes

Investigate a bug where packaged CLI entry invocations of `openclaw infer image generate --model openai/gpt-image-2` fail with Codex OAuth `fetch failed` while direct `runCli()` and provider calls succeed.

## Best Possible Solution

Keep the issue open and fix the CLI entry/runtime parity directly: isolate the process-wide entry or `runCli()` startup side effect that changes Codex image transport behavior, preserve SSRF/proxy protections, and add a deterministic regression that exercises the packaged entry-style invocation against a mocked Codex Responses image backend.

## Reproduction Assessment

Yes. The issue provides a concrete CLI command, failure output, lower-layer success controls, and a second macOS reproduction on v2026.4.27; this review verified the source path but did not perform a live OAuth run.

## Solution Assessment

Yes, the best direction is a narrow entry/bootstrap parity fix with regression coverage, not a provider rewrite. The provider already routes Codex OAuth image requests through the shared guarded `postJsonRequest()` path that the reporter says succeeds when called directly.
