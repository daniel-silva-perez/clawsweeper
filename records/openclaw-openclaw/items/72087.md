## Summary

Keep this open. The issue has a strong differential report across Linux and macOS, and current main still routes the packaged `openclaw.mjs` launcher through `dist/entry.js` before `runCli()`, while the OpenAI Codex image provider and its direct helper path remain separately healthy and tested. I did not find a merged change that specifically removes the entry/main bootstrap failure boundary or adds packaged-entry Codex OAuth image-generation coverage.

## What This Changes

Make packaged `openclaw infer image generate --model openai/gpt-image-2` behave the same as direct `runCli()` and direct OpenAI image-provider calls when using OpenAI Codex OAuth.

## Best Possible Solution

Add a focused regression fix at the launcher/entry/bootstrap boundary so packaged `openclaw.mjs` and `dist/entry.js` preserve the same network/auth behavior as direct `runCli()` for OpenAI Codex OAuth image generation. The proof should cover the entrypoint path, not only provider-unit behavior, and should keep existing proxy, SSRF, and provider transport policies intact.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes: the issue includes an exact `openclaw infer image generate --model openai/gpt-image-2` command, same-host lower-layer success controls, and an independent macOS reproduction; current main still has the same observable entrypoint boundary.

## Solution Assessment

Is this the best way to solve the issue? Yes: the narrow maintainable fix is to align the packaged entry/main bootstrap path with direct `runCli()` for this provider flow, while preserving the existing OpenAI image provider and guarded HTTP helper behavior that the report says already succeeds.
