## Summary

Keep this open. The report has a strong differential reproduction, a second macOS confirmation on the latest shipped release, and current main still has a distinct entry/main bootstrap path before `runCli()` with no source, test, or release evidence that the entrypath-only Codex OAuth image-generation failure has been fixed. The merged #75111 work is adjacent Codex OAuth transport work, but its stated root cause and touched surfaces are embedded-runner/auth-injection paths, not the packaged `dist/entry.js` image CLI boundary described here.

## What This Changes

The issue asks OpenClaw to make packaged `openclaw`/`openclaw.mjs`/`dist/entry.js` invocation behave like direct `runCli()` and direct provider calls for Codex OAuth OpenAI image generation instead of failing with `fetch failed`.

## Best Possible Solution

Make the packaged entry/main invocation preserve the same HTTP dispatcher, proxy, TLS, auth-profile, and provider behavior as direct `runCli()` for OpenAI Codex image generation, then add an entrypoint-level regression so future provider-only tests cannot miss this boundary.

## Solution Assessment
