## Summary

Keep open: current main and the latest release still keep the Android app at minSdk 31, so the PR's requested API 26 support has not been implemented or shipped. The PR contains useful compatibility work and contributor device evidence, but lowering the platform floor is still a broad Android support decision and the latest branch still exposes API 26-28 compatibility gaps in Settings assistant-role handling and motion command availability.

## What This Changes

The PR lowers the Android app minSdk from 31 to 26 and adds API-level fallbacks or gates for dynamic colors, foreground service startup, DNS discovery, location, notification listener access, thermal status, and motion-related permission/UI reporting.

## Best Possible Solution

Decide the Android support floor explicitly. If API 26 support is accepted, land a comprehensive compatibility pass that guards every API 29+ surface, keeps runtime command advertisement aligned with actual invocability, updates Android docs to the new floor, and records focused emulator or device proof for connection, Settings, permissions, discovery, notification, and node-command paths.

## Review Findings

Full review comments:

- **[P2] Guard assistant role before lowering minSdk:** `apps/android/app/build.gradle.kts:66`
  - body: Lowering `minSdk` makes API 26-28 devices supported, but `SettingsSheet` still evaluates `RoleManager` through `isAssistantRoleAvailable` and `isAssistantRoleHeld` without an API guard. `RoleManager` is API 29+, so opening Settings on Oreo/Pie can fail before the screen renders; hide or guard the assistant-role block below Q.
  - confidence: 0.86
- **[P2] Gate motion command availability below API 29:** `apps/android/app/src/main/java/ai/openclaw/app/tools/Features.kt:9`
  - body: The new `Features.Motion` gate is wired into UI and permission reporting, but `NodeRuntime` still advertises motion commands from sensor availability while `MotionHandler` requires `ACTIVITY_RECOGNITION`, which is API 29+. API 26-28 nodes with sensors can expose commands that always fail permission checks.
  - confidence: 0.88
