## Summary

Keep open. Current main and the latest release still require Android minSdk 31, so the requested API 26 support is not already implemented; the PR is useful source work but remains dirty against main and still has API-level correctness gaps before it can be merged.

## What This Changes

The PR lowers the Android app minSdk from 31 to 26 and adds API-level fallbacks or guards for dynamic colors, foreground service startup, DNS discovery, location, notification access, thermal status, and motion-related UI/permission reporting.

## Best Possible Solution

Decide the Android support floor explicitly. If API 26 support is accepted, rebase this PR and land a comprehensive compatibility pass that guards every API 29+ runtime/UI surface, keeps advertised node capabilities aligned with invocable commands, updates Android docs to the new floor, and includes focused API 26, 29, 30, and current-device validation.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to apply the PR support floor, then open Settings on API 26-28 for the unguarded `RoleManager` path, or run an API 26-28 node with motion sensors and observe motion commands advertised while invocations fail the API-29 `ACTIVITY_RECOGNITION` permission check.

## Solution Assessment

No. The direction may be valid, and earlier foreground-service and DNS issues were addressed, but this branch is not the best mergeable solution until it is rebased and the remaining API-level gates and runtime capability alignment are fixed.
