## Summary

Keep open. Current main and the latest shipped release still keep the Android app at `minSdk = 31`, so the requested API 26 support is not implemented; however, this PR is also not ready to merge because it is dirty against current main and still leaves API 26-28 compatibility gaps in Settings and runtime motion command advertisement.

## What This Changes

The PR lowers the Android app `minSdk` to 26 and adds API-level fallbacks or gates for dynamic colors, foreground service startup, DNS discovery, location, notification access, thermal status, and motion-related permission/UI reporting.

## Best Possible Solution

Decide the Android support floor explicitly. If API 26 support is accepted, rebase this work and land a comprehensive compatibility pass that guards every API 29+ UI/runtime surface, keeps advertised node capabilities aligned with invocable commands, updates Android docs to the new floor, and includes focused API 26, 29, 30, and current-device proof.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to lower the app to API 26 support, then open Settings on API 26-28 for the unguarded `RoleManager` path, or run an API 26-28 node with motion sensors and observe motion commands advertised while invocations fail the API-29 `ACTIVITY_RECOGNITION` permission check.

## Solution Assessment

No. The direction may be valid, and earlier foreground-service and DNS issues were addressed, but this branch is not the best mergeable solution until it is rebased and covers the remaining API-level gates and device/emulator validation.
