## Summary

Keep open. Current main still does not support Android API 26, the contributor PR contains useful compatibility work and field verification, but it remains a broad Android product/support decision with a concrete runtime availability bug and branch reconciliation still needed before merge.

## What This Changes

The PR lowers the Android app minSdk to 26 and adds compatibility guards or fallbacks for theme colors, foreground service startup, DNS discovery, location, notification listener access, thermal status, and motion-related UI/permission flows.

## Best Possible Solution

Keep the PR open until maintainers decide whether OpenClaw should support API 26. If they proceed, rebase the branch, fix runtime motion gating or permission semantics, align Android docs and any benchmark/test surfaces, and collect emulator or device proof across the newly supported API range; if the product decision is to stay API 31+, close it with that explicit rationale.

Full review comments:

- **[P2] Gate motion commands below API 29:** `apps/android/app/src/main/java/ai/openclaw/app/tools/Features.kt:8-9`
  - body: The new Features.Motion gate is only wired into UI and permission reporting. NodeRuntime still advertises motion commands from MotionHandler.isActivityAvailable()/isPedometerAvailable(), and MotionHandler still requires ACTIVITY_RECOGNITION. On API 26-28 that permission cannot be granted, so devices with motion sensors can expose commands that always fail; wire the same gate into runtime availability/permission handling or treat pre-Q as not requiring that permission.
