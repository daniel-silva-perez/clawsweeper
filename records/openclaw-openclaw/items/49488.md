## Summary

Keep open. Current main still does not implement API 26 Android support, and this active contributor PR is not safe to merge as-is because it is dirty against main, has an explicit maintainer scope concern, and still leaves a concrete runtime motion-command compatibility gap on API 26-28.

## What This Changes

The PR lowers the Android app minSdk from 31 to 26 and adds API-level compatibility guards or fallbacks for theme colors, foreground service startup, DNS discovery, location, notification listener access, thermal status, and motion-related UI/permission surfaces.

## Best Possible Solution

Decide the Android support floor explicitly. If API 26 support is accepted, rebase this PR, apply the API-level feature gate consistently through runtime command advertisement and invocation, update Android docs and related test surfaces, then collect focused emulator or device proof for install, connect, discovery, foreground service, permissions, and node command behavior on representative API 26-30 and current Android versions.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Gate motion commands below API 29:** `apps/android/app/src/main/java/ai/openclaw/app/tools/Features.kt:8-9`
  - body: The new Features.Motion gate is only applied to UI and permission reporting. NodeRuntime still advertises motion commands from MotionHandler.isActivityAvailable()/isPedometerAvailable(), and MotionHandler still requires ACTIVITY_RECOGNITION. Because that permission is API 29+, API 26-28 devices with sensors can expose motion commands that always fail; wire this gate into runtime availability/permission handling or define pre-Q motion as unsupported there too.
