## Summary

Keep open. Current main fixes the shared outbound delivery path and the originally confirmed Telegram reply paths, but the broader hook-coverage inconsistency still exists for WebChat/Gateway finals, embedded local TUI finals, and plugin-handled message actions that bypass the same sent-message hook surfaces.

## What This Changes

The issue asks for consistent internal `message:sent` and plugin message lifecycle hook coverage across outbound dispatch paths, with Telegram DM replies called out as the confirmed missing case.

## Best Possible Solution

Define and document one outbound lifecycle observer seam, or split the remaining work into canonical narrower fixes, so external channel replies, WebChat, Gateway-backed TUI, embedded TUI, and plugin-handled sends invoke the applicable pre-send and post-send hooks exactly once with stable session/run context and focused regression tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to enable a sent-message hook, send a WebChat or embedded TUI assistant reply or a plugin-handled message action, and observe that final delivery goes through direct broadcast/emit/plugin-dispatch paths with no hook-runner call; I did not run live tests because this review is read-only.

## Solution Assessment

No for the original minimal Telegram-only fix: that slice is already covered on current main, but it does not solve the remaining umbrella. The safer direction is a maintainer-approved shared observer seam or narrower fixes per surface with explicit exactly-once and rewrite/cancel semantics.
