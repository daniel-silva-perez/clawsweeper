## Summary

Keep open. Current main has hook coverage for the shared outbound delivery path and the originally confirmed Telegram reply path, but the broader inconsistency still exists: WebChat/Gateway chat finals, embedded TUI finals, and plugin-handled message actions still have outbound dispatch paths that do not go through the documented sent-message hook surfaces.

## What This Changes

The issue asks OpenClaw to make outbound `message:sent` and plugin `message_sent` hook emission consistent across multiple outbound dispatch paths so downstream hooks such as conversation-stream do not miss sent replies.

## Best Possible Solution

Keep this as the umbrella until maintainers either define one documented outbound lifecycle observer seam or split the remaining WebChat, Gateway-backed TUI, embedded TUI, and plugin-handled action gaps into canonical narrower fixes with explicit exactly-once, cancellation, and transcript-write semantics.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to enable a sent-message hook, send a WebChat or embedded TUI reply, and observe that final delivery goes through direct `broadcast`/`nodeSendToSession` or `this.emit("chat")` paths with no hook-runner call; I did not run live tests because this review is read-only.

## Solution Assessment

No, the originally suggested minimal Telegram-only fix is no longer sufficient for the remaining umbrella. The maintainable direction is either a shared outbound lifecycle observer or maintainer-approved narrower fixes per surface, because ad hoc hook calls risk inconsistent cancel, rewrite, and transcript semantics.
