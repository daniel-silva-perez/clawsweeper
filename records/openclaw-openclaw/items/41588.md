## Summary

Keep this issue open. Current main has mitigations for the original pain, including a 48-hour default runtime, model-stream idle detection, and an explicit timeout error payload, but it still does not implement the requested activity-aware run-level timeout or separate idle/max timeout contract for active multi-tool runs.

## Best Possible Solution

Add a run-level idle timeout refreshed by assistant stream activity and tool lifecycle completion, keep a separate absolute max runtime cap, expose both fields in schema/help/docs, and add focused tests proving active multi-tool runs are not aborted solely because the original wall-clock window elapsed.
