## Summary

Keep open. Current main has mitigations for the original 600s failure mode, including a 48-hour default runtime, model-stream idle watchdogs, compaction timeout handling, and explicit timeout error text, but the embedded run still has a run-start wall-clock abort timer and no public run-level idle/max timeout contract refreshed by tool or assistant activity.

## What This Changes

The issue requests an activity-aware agent run timeout so active multi-tool runs are not killed solely by elapsed wall-clock time.

## Best Possible Solution

Add a run-level idle timeout that is refreshed by assistant stream activity and tool lifecycle completion, keep a separate absolute maximum runtime cap, expose the contract through config schema/help/docs, and add focused tests proving active multi-tool runs can exceed the idle window's wall-clock predecessor while idle or over-cap runs still abort clearly.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure a short positive agents.defaults.timeoutSeconds and run an active multi-tool turn that lasts beyond it; runEmbeddedAttempt schedules the abort once from params.timeoutMs, so activity does not refresh the run deadline.

## Solution Assessment

Not applicable to a PR, but the best direction is option 2 from the discussion: separate idle and absolute run limits. The current 48-hour default and model idle watchdog reduce common failures, but they do not solve the configurable activity-aware runtime contract requested here.
