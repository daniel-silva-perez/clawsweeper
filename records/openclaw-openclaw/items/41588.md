## Summary

Keep this issue open. Current main has mitigations for the original 600s pain point, including a 48-hour default, model-stream idle watchdogs, and explicit timeout error replies, but the embedded run timeout is still a run-start wall-clock timer and there is no run-level idle timeout plus absolute cap config.

## Best Possible Solution

Add a run-level idle timeout that is refreshed by assistant stream activity and tool lifecycle completion, keep a separate absolute max runtime cap, expose both through config schema/help/docs, and add focused tests proving active multi-tool runs survive past the original wall-clock window while truly idle or over-cap runs still abort.

## Risks / Open Questions

- Operators who configure shorter agents.defaults.timeoutSeconds values can still abort active multi-tool runs by elapsed wall-clock time rather than inactivity.
- A run-level idle timeout should keep a separate absolute maximum runtime cap so continuous slow activity cannot keep a run alive indefinitely.
- The config schema still rejects agents.defaults.timeoutSeconds: 0, so the issue's config-level no-timeout option is not implemented even though some runtime overrides treat 0 as a timer-safe sentinel.
