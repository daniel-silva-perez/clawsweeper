## Summary

Keep this issue open. Current main has mitigations for the original 600s pain point, but it still does not implement the requested activity-aware run timeout or separate idle-vs-absolute run cap.

## Best Possible Solution

Add a run-level idle timeout refreshed by assistant stream activity and tool lifecycle completion, keep a separate absolute maximum runtime cap, expose both fields in config schema/help/docs, and add focused tests proving active multi-tool runs are not aborted solely because the original wall-clock window elapsed.

## Risks / Open Questions

- Operators who configure shorter `agents.defaults.timeoutSeconds` values can still abort active multi-tool runs based on elapsed wall-clock time rather than inactivity.
- A future idle-timeout implementation should keep a separate absolute maximum runtime cap so continuous slow activity cannot keep a run alive indefinitely.
- Schema/help/docs and focused tests need to change with the implementation so users can distinguish idle timeout from maximum runtime.
