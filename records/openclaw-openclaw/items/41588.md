## Summary

Keep this issue open. Current main has useful mitigations, but it still does not implement the requested activity-aware run timeout or separate run-level idle/max timeout contract. The embedded runner keeps a single wall-clock abort timer, while the existing idle watchdog only covers model stream silence.

## Best Possible Solution

Add a run-level idle timeout refreshed by assistant stream activity and tool lifecycle completion, retain a separate absolute max runtime cap, expose the new config contract in schema/help/docs, and add focused tests proving active multi-tool runs are not aborted solely because the original wall-clock window elapsed.

## Risks / Open Questions

- A future fix should retain an absolute max runtime cap so steady but unbounded activity cannot keep a run alive indefinitely.
