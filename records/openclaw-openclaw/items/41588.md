## Summary

Keep this issue open. Current main has important mitigations, including a 48-hour default runtime, model-stream idle detection, and explicit timeout error text, but the run-level abort is still a single wall-clock timer rather than an activity-aware idle/max timeout contract for active multi-tool runs.

## Best Possible Solution

Add a run-level idle timeout refreshed by assistant stream activity and tool lifecycle completion, keep a separate absolute max runtime cap, expose both fields in schema/help/docs, and add focused tests proving active multi-tool runs are not aborted solely because the original wall-clock window elapsed.

## Risks / Open Questions

- Operators who configure shorter `agents.defaults.timeoutSeconds` can still have active multi-tool runs aborted by elapsed wall-clock time rather than inactivity.
- The eventual fix should keep an absolute max runtime cap so steady but unbounded activity cannot keep a run alive forever.
