## Summary

Keep this issue open. Current main has mitigations for the original pain point, including a 48-hour default runtime, explicit timeout error payloads, support for zero timeout overrides in some command/tool paths, and a model-stream idle watchdog, but the embedded run abort remains a single run-level timer that is not refreshed by assistant/tool activity. The issue’s central request for separate activity-aware idle timeout plus absolute max runtime is still not implemented.

## Best Possible Solution

Add a run-level idle timeout refreshed by assistant stream activity and tool lifecycle completion, keep a separate absolute maximum runtime cap, expose both fields in schema/help/docs, and add focused tests proving active multi-tool runs are not aborted solely because the original wall-clock window elapsed.

## Risks / Open Questions

- Operators who configure shorter `agents.defaults.timeoutSeconds` can still have active multi-tool runs aborted by elapsed wall-clock time rather than inactivity.
- Changing this should preserve an absolute max runtime cap so slow but continuous activity cannot keep a run alive forever.
- The schema/docs/help surface needs to change with the implementation so users can distinguish idle timeout from maximum run duration.
