## Summary

Keep this open. Current main has useful mitigations for the original 600s pain point, including a 48-hour default agent timeout, model-stream idle watchdogs, compaction timeout recovery, and explicit timeout guidance, but it still does not implement the requested run-level activity-aware timeout or separate idle/max runtime configuration.

## Best Possible Solution

Add a run-level idle timeout refreshed by assistant stream activity and tool lifecycle completion, keep a separate absolute maximum runtime cap, expose both through config schema/help/docs, and add focused tests proving active multi-tool runs survive past the old wall-clock window while truly idle or over-cap runs still abort.
