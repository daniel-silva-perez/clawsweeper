## Summary

Keep open. Current main still resolves Discord thread starter text for thread messages and passes `ThreadStarterBody` into the inbound context without a first-turn guard. The reply prompt path still has an explicit non-new-session fallback that prepends `[Thread starter - for context]` when that field is present, and same-author PR #50522 remains the open implementation candidate with closing syntax for this issue.

## Best Possible Solution

Keep this issue open as the tracker. Review and land PR #50522 or an equivalent Discord adapter change that only supplies `ThreadStarterBody` for the first turn of the actual thread session, and add a focused Discord regression test matching the Slack existing-session coverage. If maintainers want defense in depth, also gate or remove the non-new-session fallback in the shared reply prompt path.
