## Summary

Keep open. Current main still passes Discord `ThreadStarterBody` into `finalizeInboundContext` without a first-turn guard, the shared reply path can still prepend the thread-starter marker on follow-up turns when that field is present, and the same-author implementation PR #50522 remains open and unmerged.

## Best Possible Solution

Keep this issue open as the tracker. Review and land PR #50522 or an equivalent Discord adapter change that only supplies `ThreadStarterBody` on the first turn of the routed thread session, with focused Discord regression coverage. If maintainers want defense in depth, also gate or remove the shared non-new-session `threadContextNote` fallback when a channel accidentally provides `ThreadStarterBody` after session creation.
