## Summary

Keep open. Current main still passes Discord `ThreadStarterBody` through unguarded on threaded messages, the shared reply path can still prepend the `[Thread starter - for context]` fallback on existing sessions, and open PR #50522 is an unmerged implementation candidate that names this issue as its closing target.

## Best Possible Solution

Keep this issue open as the canonical tracker. Review and land PR #50522 or an equivalent Discord adapter fix that computes the effective session key, only supplies `ThreadStarterBody` for a new effective thread session, and includes focused Discord regression coverage. Consider a defensive shared reply-path adjustment only if maintainers want to prevent any channel from reintroducing `ThreadStarterBody` on existing sessions.
