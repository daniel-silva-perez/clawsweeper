## Summary

Current main still passes Discord thread-starter context into inbound context without a first-turn guard, and the same-author implementation PR #50522 is still open and unmerged. This issue should remain open as the canonical tracker until that PR or an equivalent fix lands.

## Best Possible Solution

Keep this issue open as the canonical tracker. Review and land PR #50522 or an equivalent Discord adapter fix that only supplies `ThreadStarterBody` on the first turn of the effective thread session, with focused Discord regression coverage. If maintainers want defense in depth, also tighten the shared follow-up `threadContextNote` fallback so accidental channel-supplied `ThreadStarterBody` cannot re-enter existing-session prompts.
