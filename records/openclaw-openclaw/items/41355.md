## Summary

Keep open. Current main still resolves Discord thread starter text and passes `ThreadStarterBody` into the inbound context without gating it on the existing session timestamp. The reply prompt path can still turn a non-new-session `ThreadStarterBody` into `[Thread starter - for context]`, and same-author PR #50522 remains the open implementation candidate.

## Best Possible Solution

Keep this issue open as the tracker. Review and land #50522 or an equivalent Discord adapter change that only supplies `ThreadStarterBody` on the first thread-session turn, then add focused Discord regression coverage mirroring the existing Slack assertion for existing thread sessions.
