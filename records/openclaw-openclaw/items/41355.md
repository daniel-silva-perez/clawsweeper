## Summary

Current main still has the Discord thread-starter re-injection path described here, and same-author PR #50522 remains an open implementation candidate with closing syntax for this issue. This should stay open until that PR, or an equivalent Discord adapter fix plus regression coverage, is merged or explicitly rejected.

## Best Possible Solution

Keep this issue open as the tracker. Review and land PR #50522 or an equivalent Discord adapter change that only supplies `ThreadStarterBody` on the first turn of the actual thread session, with a focused Discord regression test matching Slack's existing-session coverage. If maintainers want defense in depth, also remove or gate the shared follow-up `threadContextNote` fallback for non-new sessions.
