## Summary

Keep this issue open. Current main still has the reported hot path: non-test `getSubagentRunsSnapshotForRead()` calls `loadSubagentRegistryFromDisk()` for each read snapshot, and that loader still uses the shared synchronous `fs.readFileSync` + `JSON.parse` helper. Main has retention/cleanup mitigations for completed subagent/task records, but not the requested cache or write-invalidation fix. The related cache PR #57863 was closed unmerged in the supplied GitHub context, so there is no merged canonical implementation to close against.

## Best Possible Solution

Keep this issue open. Implement an mtime-based or write-invalidated process-local cache in `src/agents/subagent-registry.store.ts` or a higher-level snapshot cache in `src/agents/subagent-registry-state.ts`, preserving cross-process visibility. Add focused regression coverage around repeated subagent read snapshots or `sessions.list` calls so unchanged `runs.json` is not reparsed on every RPC. The closed unmerged PR #57863 is a useful prior design reference, but it is not shipped.

## Risks / Open Questions

- The existing cleanup/TTL behavior likely reduces long-term file growth, so the original 68-run reproduction may be less common on current main, but the synchronous per-snapshot parse remains on the hot path.
- A cache fix must preserve the cross-process freshness behavior noted in the current code comment, likely with mtime checking plus invalidation after `saveSubagentRegistryToDisk()`.
- Regression coverage should prove repeated `sessions.list` or subagent read-helper calls do not repeatedly parse an unchanged `runs.json`.
