## Summary

Keep open: current main still has a plausible compaction-retry delivery race. The runner drains block replies before waiting for retry completion, but it does not drain again after the retry wait unblocks, and lifecycle resolves the compaction wait before the async channel flush path completes.

## What This Changes

The report asks OpenClaw to ensure retry-generated replies are delivered to messaging channels when overflow compaction rolls a session into a retried agent loop.

## Best Possible Solution

Establish a clear embedded-runner ordering contract so a compaction retry cannot be considered complete until retry-generated block replies have drained, then lock it with a regression test for pending compaction retry plus an async `onBlockReplyFlush()`.

## Repair Work Prompt

Fix openclaw/openclaw#47335. When overflow compaction sets `willRetry=true`, a retried agent loop can generate block replies whose async channel flush is still in flight when `waitForCompactionRetryWithAggregateTimeout()` unblocks. Implement a narrow embedded Pi runner/lifecycle fix so the retry wait cannot complete before retry-generated block replies have drained. Prefer an ordering contract in `handleAgentEnd()` that awaits the relevant block/channel flush before `resolveCompactionRetry()`, or add an idempotent post-wait drain in `runEmbeddedAttempt()` if that is the least invasive local contract. Do not change channel-specific delivery, compaction semantics, provider behavior, or unrelated final-payload suppression work from #62157.
