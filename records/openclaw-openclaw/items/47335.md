## Summary

Current main still leaves the reported compaction retry delivery race plausible and actionable. The runner flushes before waiting for compaction retry, but it does not drain the block reply pipeline after the retry wait resolves; lifecycle still resolves the retry promise before awaiting the async channel flush; and current tests explicitly preserve that ordering.

## What This Changes

The issue requests fixing a compaction retry race where a retry-generated reply can be recorded in transcript but not delivered because the async block reply flush is not awaited before the retry wait resolves.

## Best Possible Solution

Land a narrow embedded Pi runner/lifecycle fix that guarantees a compaction retry wait cannot unblock before retry-generated block replies have drained, with a regression test covering pending compaction retry plus an async `onBlockReplyFlush()`.

## Repair Work Prompt

Fix the compaction retry reply-delivery race reported in openclaw/openclaw#47335. Use #47338 as source context, but implement against current main. The observable bug is that a retry-generated assistant reply can be recorded in transcript but not delivered because `handleAgentEnd` resolves the compaction retry wait before the async block reply/channel flush finishes, while `runEmbeddedAttempt` only flushes before `waitForCompactionRetryWithAggregateTimeout()` and final text payload fallback is suppressed after successful block streaming. Keep the fix narrow to the embedded Pi runner/lifecycle ordering contract; do not change channel-specific delivery behavior, provider behavior, or broad final-payload semantics unless required by the regression. Prefer a minimal guarantee that a compaction retry wait cannot unblock before retry-generated block replies have drained, either by moving retry resolution after the relevant awaited flush in lifecycle or by adding an idempotent post-wait flush in the runner if that better matches existing contracts. Add/update regression coverage for pending compaction retry plus async `onBlockReplyFlush()`. Update changelog only if the repo expects user-facing bugfix entries for this lane; if added, credit the reporter/context with `Thanks @frankhli843` and `Thanks @Ryce` and keep the bullet single-line.
