## Summary

Keep open. Current main still has the reported compaction-retry delivery race: the runner drains block replies before the compaction retry wait, but retry completion can still unblock before the retry-generated async block-reply/channel flush has finished, and current tests assert that ordering.

## What This Changes

The issue reports that compaction retry can generate a reply that is recorded in the transcript but not delivered because retry block-reply flushing may still be in flight when the attempt returns.

## Best Possible Solution

Add a retry-specific delivery guarantee so compaction retry completion cannot let the attempt finish while retry-generated block replies are still draining, and cover it with a focused regression test that preserves the non-retry slow-channel ordering introduced by #59308.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes: a focused test can hold `onBlockReplyFlush()` pending while `pendingCompactionRetry` resolves, then observe the runner continuing while text final payloads are suppressed by successful block streaming.

## Solution Assessment

Is this the best way to solve the issue? Yes: the narrowest maintainable direction is a retry-specific post-wait flush or equivalent delivery barrier, rather than a broad channel-delivery ordering change or relying on final-payload fallback behavior.
