## Summary

Keep open. Current main still has the reported compaction-retry delivery race: the runner drains block replies before the retry wait but not after it, while lifecycle handling resolves the retry wait before awaiting the async block/channel flush. The source PR that directly proposed the post-wait barrier is closed unmerged, and the adjacent final-payload PR does not remove the need for a retry-specific flush guarantee.

## What This Changes

Ensure replies generated during compaction retry session rollover are fully delivered to messaging channels before the attempt finishes.

## Best Possible Solution

Add a retry-specific ordering guarantee so compaction retry completion cannot let the attempt finish while retry-generated block replies are still draining, while preserving the broader #59308 behavior that ordinary slow channel flushes do not block compaction wait completion.

Current main still lets compaction retry completion unblock the runner before retry-generated block replies are fully flushed: `runEmbeddedAttempt` flushes before `waitForCompactionRetryWithAggregateTimeout()` but not after, and `handleAgentEnd()` resolves pending compaction retry before awaiting the async block/channel flush.
