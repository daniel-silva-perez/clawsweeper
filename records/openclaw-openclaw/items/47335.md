## Summary

Keep open. Current main still has the reported compaction-retry delivery race: the runner drains the block reply pipeline before waiting for retry completion but does not drain again after the retry wait unblocks, while lifecycle handling resolves the compaction retry before awaiting the async block/channel flush. Existing tests still lock in that ordering, and text-only final payloads remain suppressed after successful block streaming, so the final payload path does not reliably rescue a retry reply whose block flush is still in flight.

## What This Changes

The issue reports that compaction retry/session rollover can generate a reply that is recorded in the transcript but never delivered to the messaging channel.

## Best Possible Solution

Add a retry-specific delivery barrier so compaction retry completion cannot let the attempt finish while retry-generated block replies are still draining, while preserving the general #59308 behavior that ordinary slow channel flushes do not block compaction wait completion.

## Repair Work Prompt

Fix the compaction retry delivery race tracked by https://github.com/openclaw/openclaw/issues/47335. Current main flushes the block reply pipeline before `waitForCompactionRetryWithAggregateTimeout()` but not after that wait resolves, while `handleAgentEnd()` resolves pending compaction retry before async `onBlockReplyFlush()` drains. Add a retry-specific ordering guarantee so retry-generated block replies are delivered before the attempt can finish and text-only final payload suppression can drop fallback payloads. Preserve #59308's intent: ordinary slow channel/block flushes must not make compaction retry wait hit the aggregate timeout again. Do not make broad payload-dedupe changes, do not change unrelated channel/plugin behavior, and coordinate with but do not rely on #62157. Add focused regression coverage with pending compaction retry plus an async `onBlockReplyFlush` that would have been unresolved when the runner continued. Because this is user-visible reply delivery, add a single-line changelog fix entry in the active version if runtime behavior changes, crediting @frankhli843 and @Ryce where appropriate.
