## Summary

Current main still leaves the reported silent-delivery race plausible and actionable: the runner flushes block replies only before the compaction retry wait, lifecycle resolves the retry promise before awaiting the async channel flush, and final text payload fallback is still suppressed after successful block streaming. The related narrow fix PR #47338 is closed and unmerged, while open #62157 is adjacent final-payload work and does not specifically close this compaction retry flush-ordering bug.

## What This Changes

Ensure generated replies are delivered when auto-compaction triggers a retry/session rollover by draining the retry-generated block reply pipeline before the run continues.

## Best Possible Solution

Keep this issue open and land a narrow embedded Pi runner/lifecycle fix that guarantees a compaction retry wait cannot unblock before the retry-generated block reply pipeline has drained. The fix should avoid channel-specific delivery changes and include a regression test with pending compaction retry plus an async `onBlockReplyFlush()`.

## Repair Work Prompt

Fix the compaction retry block-reply delivery race described in openclaw/openclaw#47335. Current main flushes `onBlockReplyFlush()` before `waitForCompactionRetryWithAggregateTimeout()`, but `handleAgentEnd` can resolve `pendingCompactionRetry` before async channel/block flush completion, so the retry-generated reply can be recorded but not delivered while final text fallback is suppressed. Implement a narrow generic embedded Pi runner/lifecycle fix so the compaction retry wait cannot unblock before the retry-generated block reply pipeline has drained. Acceptable directions include awaiting the relevant flush before resolving compaction retry in lifecycle, adding an idempotent post-wait flush in `attempt.ts`, or another small generic ordering fix that the tests prove. Do not add channel-specific delivery behavior, do not change provider/compaction policy, and do not broaden final payload semantics except as necessary for this bug. Add or update regression coverage for pending compaction retry plus async `onBlockReplyFlush()`, and update the active changelog with a one-line user-facing fix entry crediting the reporter/commenter as appropriate under repo policy.
