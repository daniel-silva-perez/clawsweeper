## Summary

Keep open. Current main still lacks a delivery barrier after compaction retry completion: the runner flushes block replies before the retry wait, but retry-generated block replies can still be draining when lifecycle resolves the retry wait and the attempt continues.

## What This Changes

The issue reports that compaction retry session rollover can generate a reply in the transcript but fail to deliver it to messaging channels because retry block-reply flushing races with attempt completion.

## Best Possible Solution

Add a retry-specific delivery guarantee so compaction retry completion cannot let the attempt finish while retry-generated block replies are still draining, with a focused regression test that preserves #59308's non-retry slow-channel ordering intent.

## Reproduction Assessment

Yes. A focused test can hold `onBlockReplyFlush()` pending while `pendingCompactionRetry` resolves, then verify the runner can currently continue before the retry-generated block delivery barrier drains.

## Solution Assessment

Yes. The narrowest maintainable path is a post-wait flush or equivalent retry-specific barrier, rather than broad channel-delivery reordering or relying on final-payload fallback behavior.
