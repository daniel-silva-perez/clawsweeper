## Summary

Keep open. Current `origin/main` still has the reported compaction-retry delivery race: the runner flushes block replies before waiting for compaction retry completion but does not flush again after the retry wait unblocks, while lifecycle handling still resolves the compaction retry before the async block/channel flush has drained. The closed source PR #47338 was not merged, and the adjacent final-payload work does not remove the text-only final-payload suppression that makes this race user-visible.

## What This Changes

The issue reports a compaction retry race where retry-generated block replies can be recorded in the session transcript but not delivered to messaging channels because attempt completion can outrun the async block reply flush.

## Best Possible Solution

Add a retry-specific delivery barrier so compaction retry completion cannot let the attempt finish while retry-generated block replies are still draining, with regression coverage for pending retry plus an async `onBlockReplyFlush`.

## Repair Work Prompt

Fix the compaction-retry delivery race reported in openclaw/openclaw#47335. Current `origin/main` flushes block replies before waiting for compaction retry completion, but retry-generated replies can be produced after that flush and lifecycle resolution can unblock the runner before the async block reply flush drains. Add a narrowly scoped ordering guarantee so retry completion cannot let `runEmbeddedAttempt` finish while retry-generated block replies are still in flight. Prefer a post-`waitForCompactionRetryWithAggregateTimeout()` idempotent `onBlockReplyFlush()` or an equivalently narrow retry-specific barrier; do not broadly revert #59308's intent that ordinary slow channel flushes should not block compaction wait resolution. Preserve channel-agnostic behavior; do not add Discord/Telegram-specific logic. Add or update regression coverage for pending compaction retry with async `onBlockReplyFlush`, and keep final-payload/block-streaming behavior intact except as required for this race. Add a single-line changelog fix entry in the active version with `Thanks @frankhli843` and include `Thanks @Ryce` if the implementation uses the comment's proposed lifecycle-ordering approach. Run the targeted tests and formatter listed in `workValidation`, then run `pnpm check:changed` in Testbox before handoff for runtime changes.
