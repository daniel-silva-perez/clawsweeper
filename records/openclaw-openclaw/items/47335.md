## Summary

Keep open: current main still has the compaction-retry delivery race. The runner drains block replies before the compaction retry wait, but retry-generated block replies can still be flushing when lifecycle resolves the retry wait and the attempt continues.

## What This Changes

The issue asks for compaction retry completion to wait for retry-generated block replies to drain before the attempt returns to the messaging channel flow.

## Best Possible Solution

Add a retry-specific post-wait `onBlockReplyFlush` or equivalent delivery barrier, with regression coverage proving the runner cannot finish a compaction retry while retry-generated block replies are still draining.

## Reproduction Assessment

Yes. A focused unit test can hold `onBlockReplyFlush` pending while `pendingCompactionRetry` resolves, then assert the runner currently continues after `waitForCompactionRetryWithAggregateTimeout` without awaiting the retry-generated delivery barrier.

## Solution Assessment

Yes. The narrowest maintainable solution is a post-wait flush or equivalent retry-specific barrier in the runner path; changing general lifecycle ordering would be riskier because current tests intentionally protect slow-channel compaction wait behavior.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/47335. Current main flushes `params.onBlockReplyFlush()` before `waitForCompactionRetryWithAggregateTimeout()` in `src/agents/pi-embedded-runner/run/attempt.ts`, but retry-generated block replies can still be flushing after the wait resolves because `handleAgentEnd()` resolves pending compaction retry before awaiting `onBlockReplyFlush`. Add a retry-specific delivery barrier, most likely an idempotent post-wait `onBlockReplyFlush()` in the runner path or an equivalent mechanism, so the attempt cannot finish while retry-generated block replies are still draining. Add focused regression coverage for a pending async block reply flush during compaction retry.
