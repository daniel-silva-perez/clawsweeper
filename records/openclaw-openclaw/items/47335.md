## Summary

Current main still appears to have the reported compaction-retry delivery race. The runner drains the block reply pipeline before waiting for compaction retry completion, but there is no second drain after the retry wait unblocks; lifecycle code still resolves the compaction wait before awaiting the channel/block flush, and current tests assert that ordering.

## What This Changes

The issue asks for replies generated during auto-compaction retry/session rollover to be delivered to messaging channels instead of remaining only in the transcript.

## Best Possible Solution

Add a retry-specific ordering guarantee so compaction retry completion cannot let the attempt finish while retry-generated block replies are still draining, while preserving the general #59308 behavior for ordinary slow channel flushes.

## Repair Work Prompt

Fix openclaw/openclaw#47335. Current main drains `onBlockReplyFlush` before `waitForCompactionRetryWithAggregateTimeout()` in `src/agents/pi-embedded-runner/run/attempt.ts`, but not after the wait resolves. `handleAgentEnd()` resolves pending compaction retry before awaiting the async block/channel flush path, so retry-generated block replies can still be in flight while final payload handling suppresses text because block streaming already succeeded. Implement a narrow guarantee that compaction-retry-generated block replies drain before `runEmbeddedAttempt` proceeds past the retry wait. Prefer an idempotent post-wait `params.onBlockReplyFlush()` in `attempt.ts` or an equivalent retry-specific contract. Do not broadly revert #59308's goal of preventing ordinary slow channel flushes from delaying compaction completion unless tests prove the replacement preserves that behavior. Add regression coverage for pending compaction retry plus async `onBlockReplyFlush()` showing the attempt cannot finish before the retry flush drains. Keep channel-specific delivery code and plugin APIs unchanged. Add a single-line changelog fix entry crediting `Thanks @frankhli843`.
