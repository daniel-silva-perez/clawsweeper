## Summary

Keep open. Current main still preserves the ordering described by this issue: the lifecycle handler resolves the compaction wait before the async block-reply/channel flush has completed, and the runner only has the earlier pre-wait flush from #35489. There is an outer pipeline flush before final payload assembly, but the block reply delivery path can still be awaiting normalization before enqueue, so that outer flush is not a complete guarantee. Final text payload fallback also still drops text-only payloads once any block streaming succeeded. The same-author fix PR #47338 was closed unmerged, while related PR #62157 remains an open broader final-payload filtering proposal rather than a merged fix for this compaction retry race.

## Best Possible Solution

Keep this issue open until the compaction retry path has a regression-tested ordering guarantee: either await the pending block-reply buffer/channel flush before resolveCompactionRetry(), or add an equivalent post-wait drain that also waits for pending block reply tasks before final-payload suppression. Review #62157 separately as a broader final-payload dedupe change, but it should not replace the compaction retry flush guarantee unless it explicitly covers this race.
