## Summary

Current main still does not provide a completed async block-reply flush after a compaction retry resolves. The runner flushes before the wait, lifecycle resolves the retry wait before channel flush completion, and text-only final payload fallback is still suppressed after successful block streaming, so the reported silent-delivery race remains plausible and actionable.

## Best Possible Solution

Keep this issue open and make a narrow embedded Pi runner/lifecycle fix that guarantees a compaction retry wait cannot unblock before the retry-generated block reply pipeline has drained. The fix should stay generic, avoid channel-specific delivery changes, and add a regression test for pending compaction retry plus async `onBlockReplyFlush()`.

## Clownfish Work Prompt

Fix this issue in OpenClaw. The observable bug is that when auto-compaction retries with `willRetry=true`, the retry-generated reply can be streamed into the block reply pipeline but the runner can observe compaction retry completion before async `onBlockReplyFlush()` finishes, leaving text-only final payload fallback suppressed and the user seeing no reply. Keep the fix narrow and generic to the embedded Pi runner/lifecycle path. Do not add channel-specific logic, do not change unrelated final-payload filtering, and do not touch dependency/vendor/lockfile state.

Expected boundary: either await/drain `onBlockReplyFlush()` before resolving pending compaction retry in `handleAgentEnd`, or add an idempotent post-wait flush in `runEmbeddedAttempt` after `waitForCompactionRetryWithAggregateTimeout()` resolves, choosing the approach that preserves existing lifecycle/error semantics best. Add regression coverage that models `pendingCompactionRetry > 0` with an async `onBlockReplyFlush()` and proves the retry wait/attempt cannot proceed past the critical point until the flush resolves. Update or replace tests that currently lock the risky ordering. Add a single-line user-facing changelog entry in the active version section crediting the reporter/commenter, e.g. `Thanks @frankhli843 and @Ryce`, if the repo’s current changelog section still expects entries.
