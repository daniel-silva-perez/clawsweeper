## Summary

Keep open: current main still has the reported compaction-retry delivery race. The runner drains block replies before waiting for compaction retry completion, but retry-generated block replies can still be flushing when lifecycle resolves the retry wait and the attempt continues.

## What This Changes

The issue requests a reliable delivery barrier so replies generated during a compaction retry are flushed to the messaging channel before the embedded attempt finishes.

## Best Possible Solution

Add a retry-specific ordering guarantee, most likely a post-wait onBlockReplyFlush or equivalent barrier, so compaction retry completion cannot let the attempt finish while retry-generated block replies are still draining.

## Reproduction Assessment

Yes. A focused test can hold onBlockReplyFlush pending while pendingCompactionRetry resolves, then assert the runner currently continues after waitForCompactionRetryWithAggregateTimeout without awaiting the retry-generated delivery barrier.

## Solution Assessment

Yes. The narrowest maintainable fix is a retry-specific post-wait flush or equivalent barrier; changing general lifecycle ordering would risk undoing #59308's slow-channel timeout protection.
