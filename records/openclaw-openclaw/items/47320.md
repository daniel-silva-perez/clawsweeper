## Summary

Keep open. Current main still lists only directly controlled subagents in both the model tool and slash-command paths; nested workers are available only as descendant/child-session summary metadata, not flattened rows with depth or parent information. The request is recent, has corroborating user context, is not superseded by the broader chain-tracking issue, and PR #14447 added nested orchestration without adding recursive listing.

## What This Changes

Request adding `depth` or `recursive` support to `subagents list` and the model `subagents` list action so a parent agent can inspect nested subagents with hierarchy metadata.

## Best Possible Solution

Add a bounded traversal mode that preserves today’s default direct-child list, supports finite `depth` plus all-descendants `recursive`, emits flat rows with `depth` and `parentSessionKey`, and documents the orchestrator workflow without expanding control permissions.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists by seeding a main-owned orchestrator run plus an orchestrator-owned worker run: the current list query returns only the orchestrator, while the worker is visible only through descendant summary metadata.

## Solution Assessment

Yes. The requested depth/recursive list option is the narrowest maintainable solution if it reuses the existing descendant traversal helpers and keeps non-list actions scoped to directly controlled children.
