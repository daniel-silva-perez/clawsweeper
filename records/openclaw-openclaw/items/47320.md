## Summary

This issue should stay open. Current main still has direct-child-only `subagents` listing: the tool schema has no `depth` or `recursive` parameter, the list path is fed by runs owned by the current controller session, and the slash command does not parse list flags. Main has useful adjacent metadata such as `pendingDescendants`, `childSessions`, and `sessions_list` `parentSessionKey`, but that does not provide the requested recursive `/subagents list --depth 2` or `{ action: "list", depth: 2 }` view with nested subagent task/status details. The provided +1 comment also supports a flattened output with `parentSessionKey`, which remains unimplemented. Related #67943 is about stale subagent lineage and spawn-depth limits, not this visibility feature.

## Best Possible Solution

Add backward-compatible `depth` and/or `recursive` parameters to the `subagents` tool and slash command, defaulting to the current direct-child behavior. Build the recursive rows from existing subagent registry descendant helpers and liveness rules, include `parentSessionKey`/depth plus label, status, runtime, model, task, and session key, and add focused tests for depth-0 seeing depth-2 workers without broadening kill/steer ownership semantics.

## Risks / Open Questions

- A future implementation should reuse existing liveness and child-link rules so recursive output does not resurrect stale store-only children or bypass subagent control-scope boundaries.
- `sessions_list` already exposes some hierarchy metadata, so the product decision should define whether recursive subagent listing is a wrapper around existing session rows, a subagent-run-specific view, or both.
