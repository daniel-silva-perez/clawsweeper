## Summary

Keep open. Current main still exposes only direct-child subagent listing: the tool schema has no `depth` or `recursive` parameter, both tool and slash-command paths feed `buildSubagentList` with direct controlled runs, and nested workers are only summarized as descendant counts or child session keys.

## What This Changes

The issue requests recursive or depth-bounded `subagents list` support so a parent agent can inspect depth-2 workers with hierarchy metadata.

## Best Possible Solution

Add a bounded, backward-compatible recursive list mode for both the tool and slash command: default to direct children, accept finite `depth` and `recursive`, return flat rows with hierarchy metadata, and document the behavior for orchestrator patterns.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: seed a main-owned orchestrator run plus an orchestrator-owned worker run, then the current direct-controller filter will only pass the orchestrator row to the main caller while the worker appears only as descendant summary metadata.

## Solution Assessment

Yes. The requested `depth`/`recursive` direction is the narrow maintainable fix if it preserves depth 1 as the default, clamps traversal, returns flat rows with `depth` and `parentSessionKey`, and keeps control actions scoped to their existing direct-owner rules.
