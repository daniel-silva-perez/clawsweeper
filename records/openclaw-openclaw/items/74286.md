## Summary

Keep open. Current main still builds announce handoffs from the child completion event and rewrite instruction, then queues or sends that same trigger into the requester path without an explicit bounded requester-conversation summary; the docs and tests still describe/assert only child result, stats, and follow-up instruction. The later Telegram direct-channel comment adds adjacent routing evidence rather than a cleanup reason.

## What This Changes

The issue requests that subagent and cron announce completion replies include enough requester/parent conversation context to avoid out-of-context user-facing responses.

## Best Possible Solution

Add a bounded requester-context handoff to the task-completion announce path, using a typed event field or prompt side-channel that is sanitized and size-limited, and cover both subagent and isolated-cron announce replies for active queued sessions and dormant direct delivery. Keep any stale-session canonicalization change scoped to the same delivery surface only if the same fix path proves it; otherwise split that into a follow-up.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction path: trigger a completion-mode subagent or isolated cron announce and inspect the generated `task_completion` trigger that is queued/sent to the requester. I did not run a live channel reproduction in this read-only review.

## Solution Assessment

Yes directionally, but the safest fix is not raw `last N messages` injection. The maintainable version should pass a bounded requester-context summary or reuse existing sanitized session-history/context machinery while preserving the current child-result event contract.
