## Summary

Keep open. Current main still constructs subagent and cron announce handoffs from the child completion event plus a rewrite instruction, then passes that same trigger into the requester session; I did not find source, tests, or docs showing a bounded parent/requester conversation summary. The newer Telegram direct-channel comment adds a related routing symptom, while the related delivery/account reports cover different failure modes rather than superseding this prompt-context bug.

## What This Changes

The issue asks OpenClaw to include enough parent/requester conversation context when subagent or cron announce completion wakes the parent session so the generated completion reply stays on topic.

## Best Possible Solution

Add a bounded, sanitized requester-context handoff for subagent and isolated-cron announce replies, using existing requester session state where possible and covering both active steered sessions and dormant direct agent calls with regression tests. Keep raw child result handling untrusted and avoid copying unrelated full transcripts into durable child completion events.

## Reproduction Assessment

Yes for the root prompt shape: current main deterministically builds the announce trigger from the child completion event plus a rewrite instruction, with no requester-context field. I did not run a live Feishu or Telegram cron reproduction in this read-only sweep, so the exact model wording is not live-reproduced here.

## Solution Assessment

No for the issue's literal "last N messages" suggestion if it means raw transcript copying; the safer maintainable fix is a bounded sanitized summary or existing requester-session context reference. The current implementation is not sufficient because active and dormant announce paths still receive only the child completion trigger.
