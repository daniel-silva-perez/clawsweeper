## Summary

Keep open. Current main still builds subagent announce handoffs from the child completion result plus a rewrite instruction, with no explicit requester-conversation context in the internal event, trigger message, delivery params, docs, or regression expectations. The related completed work covers delivery/routing resilience, not this prompt-context failure, and the newer Telegram stale-session addendum is useful context but does not make this cleanup-closeable.

## What This Changes

The issue asks OpenClaw to make subagent and cron `announce` completion replies aware of the parent/requester session context so the generated user-facing update is not based only on the child task result.

## Best Possible Solution

Add a bounded requester-context handoff for subagent and cron announce replies, using the parent/requester session state where possible and covering active steered sessions, dormant direct-agent delivery, and relevant cron/subagent announce paths with regression tests. Keep raw child output treated as untrusted data and avoid copying unrelated transcript content into durable child completion events.

## Reproduction Assessment

Yes for the core prompt-context bug: current source and tests show the announce trigger is built from the child task event plus rewrite instruction, with no requester-context field. The Telegram stale-session-key addendum has useful logs but needs a focused session-store fixture or live reproduction before treating it as the same root cause.

## Solution Assessment

Yes, the right direction is a narrow announce-handoff change that lets the requester turn see bounded requester context. The safer implementation is not the issue’s literal “last N messages” copy, but a sanitized summary or use of existing requester session history at delivery time.

## Repair Work Prompt

Fix the subagent/cron announce prompt-context bug for this issue on current main.

Observable bug: when a subagent or cron-backed child completion uses `delivery.mode: "announce"`, the requester session can be asked to generate a user-facing update from only the child result plus the announce rewrite instruction, which can produce an out-of-context reply.

Expected boundary:
- Add a bounded requester-context handoff for subagent and cron announce replies. Prefer using existing requester session state/history at the requester turn or a sanitized short summary; do not copy raw/unbounded transcript text into durable child completion events.
- Preserve current child-result sanitization and untrusted-content framing.
- Cover both active requester sessions (`queueEmbeddedPiMessage` / steered path) and dormant direct-agent announce calls.
- Include cron/subagent paths that can produce announce delivery, especially cron jobs that spawn subagents.
- Do not implement the broader native `announceTarget` feature from #27445 unless a tiny shared helper is needed for this fix.
- Treat the Telegram stale-session-key comment as related context: add a focused regression only if the same change can prove canonical requester session delivery without broad routing refactors; otherwise leave a clear follow-up note.
- Update `docs/tools/subagents.md` if the announce context contract changes, and add a user-facing changelog entry with reporter credit if behavior changes.

Related refs: #74286, #51818, #74080, #74179, #27445.

Likely files: `src/agents/subagent-announce.ts`, `src/agents/internal-events.ts`, `src/agents/subagent-announce-delivery.ts`, `src/cron/isolated-agent/delivery-dispatch.ts`, focused tests under `src/agents/*subagent-announce*` and `src/cron/isolated-agent/*`, `docs/tools/subagents.md`, `CHANGELOG.md`.
