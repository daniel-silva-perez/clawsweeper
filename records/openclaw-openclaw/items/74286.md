## Summary

Keep open. Current main still constructs subagent and cron announce handoffs from the child completion event plus a rewrite instruction, and I did not find a requester-context field or regression expectation in the event schema, trigger construction, delivery paths, docs, or tests. The related completed routing work helps delivery reliability, but it does not address this prompt-context failure; the Telegram stale-session addendum is useful related evidence and may need a focused fixture before being treated as the same root cause.

## What This Changes

The issue requests that subagent and cron completion announcements preserve enough parent/requester conversation context for the generated user-facing reply to stay relevant.

## Best Possible Solution

Add a bounded requester-context handoff for announce replies so the requester turn can ground the completion in the active parent conversation, while keeping child output untrusted and preserving existing delivery-routing behavior. The docs and regression tests should define the context boundary clearly for subagent, cron, active steered, and dormant direct-agent delivery paths.

## Reproduction Assessment

Yes. The current source provides a high-confidence static reproduction for the core bug: the announce trigger is built only from the child task completion event plus rewrite instruction, and the delivery paths forward that trigger unchanged. The Telegram stale-session addendum is credible but needs a focused session-store or live-channel reproduction before confirming the same root cause.

## Solution Assessment

Yes, a narrow announce-handoff fix is the best direction. The safer implementation is not raw last-N transcript injection, but a bounded sanitized summary or existing requester-session context at delivery time, with tests proving the reply turn has requester context without broad announceTarget or routing rewrites.
