## Summary

Keep this PR open. Current main still has the direct `loadProviderUsageSummary()` await in the `channels list --json` path, so the intended JSON fallback behavior is not implemented or obsolete. The PR is also not a duplicate of the related warning-UX work; it targets machine-readable JSON robustness.

## Best Possible Solution

Keep the PR open and revise it into the canonical fix: wrap the JSON-mode provider-usage load so stdout remains valid JSON, decide whether to emit a stderr warning for diagnostics, reset the usage mock in the regression test, and run the targeted channels command test plus the normal changed gate before landing.
