## Summary

Keep open. Current main still has the reported failure mode in `channels list --json`, but this PR is stale against main and the proposed catch silently drops usage-load failures, leaving a diagnosability regression to resolve before merge.

## What This Changes

The PR wraps provider usage loading for `channels list --json` so JSON output still emits without a `usage` field when usage loading throws, and adds a focused regression test.

## Best Possible Solution

Keep the narrow JSON fallback, rebase it onto current main, preserve the current provider-usage options, surface usage-load failures on stderr while keeping stdout valid JSON, reset the new usage mock between tests, and then rerun the focused channels test plus the changed gate.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to make `loadProviderUsageSummary` throw and call `channelsListCommand({ json: true })`; current main awaits that call before `writeRuntimeJson`, while the text path already catches the same class of failure.

## Solution Assessment

No, not as-is. Wrapping the JSON usage load is the right narrow location, but the merge-ready fix should keep current-main call options and preserve an operator-visible failure signal instead of silently omitting `usage`.
