## Summary

Keep open. Current main does not include this bundled guardrails plugin, and the proposed change is an XL, security-sensitive new plugin with outbound HTTP providers, API-key handling, dynamic local-code loading, new dependencies, and an official-plugin policy decision. The existing hook surface appears capable of supporting the idea, but the PR still needs maintainer/secops product review and at least one concrete normalization bug should be fixed before merge.

## What This Changes

This PR adds a bundled `extensions/guardrails` plugin that registers `before_dispatch` allow/block checks with blacklist, HTTP-provider, and dynamic-import connectors, plus package metadata, static asset packaging, lockfile updates, and tests.

## Best Possible Solution

Maintainers should first decide the ownership model: accept this as an official bundled security plugin only after secops/product review, docs alignment, dependency review, and resolved patch findings, or ask the contributor to publish it as an external ClawHub/native plugin using the existing hook surface.

## Risks / Open Questions

- The PR adds an official bundled security plugin with outbound HTTP calls, API-key-bearing configuration, endpoint overrides, dynamic local-code execution through `jiti`, and new supply-chain dependencies; this needs maintainer/secops review rather than an automated repair lane.
- The public `before_dispatch` docs and current implementation describe different lifecycle points, so landing a pre-agent guardrails plugin could cement an ambiguous plugin contract unless docs/contracts are aligned.
- The blacklist connector's normalization still leaves common invisible/bidi bypass characters unstripped, so the proposed default safety path can be evaded with simple Unicode interleaving.
- The module-level custom HTTP provider registry is shared across plugin instances; that may be acceptable for trusted single-process use, but it is a multi-tenant/test-isolation question for a bundled security plugin.
