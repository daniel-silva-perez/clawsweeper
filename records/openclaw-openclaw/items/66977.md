## Summary

Keep open. Current main still routes builtin memory through node:sqlite and sqlite-vec loadable-extension APIs, then degrades to fallback search when loading fails; the recent runtime-deps work keeps sqlite-vec packaged, but it does not add a macOS-capable SQLite runtime, alternate binding, or documented supported workaround for the reported native-load failure.

## What This Changes

The issue requests a supported macOS path for sqlite-vec vector search when node:sqlite cannot load SQLite extensions.

## Best Possible Solution

Keep this as the canonical macOS sqlite-vec/node:sqlite tracker until maintainers either validate a supported macOS Node/runtime combination where builtin sqlite-vec loads, switch builtin memory to a SQLite loading strategy that supports sqlite-vec on macOS, or explicitly document the macOS limitation and supported workaround while preserving the degraded fallback path.

## Reproduction Assessment

Yes for a macOS maintainer machine: the issue gives concrete node:sqlite and openclaw memory status commands, and a commenter corroborated a macOS vector-load failure with PATH-sensitive SQLite behavior. This Linux checkout only verifies the shared node:sqlite extension-loading boundary, not the macOS compile option itself.

## Solution Assessment

No current best fix is present on main. The existing fallback is useful degradation, but the durable solution needs a maintainer decision between supported macOS runtime selection, an alternate SQLite binding/loading strategy, or explicit docs/workaround support.
