## Summary

Keep open. Current main partially superseded the broad CJK failover gap through merged PR #56242, but it still does not cover the exact MiniMax 2062 markers from this PR, and this branch has blocking false-positive and reason-preservation problems that need maintainer-directed revision before merge.

## What This Changes

The PR adds Chinese MiniMax/CN-provider rate-limit markers to shared failover matching and changes assistant failover detection to scan short non-error assistant text for rate-limit or overload phrases.

## Best Possible Solution

Add precise MiniMax 2062 handling without standalone generic retry text. Prefer a MiniMax/provider-owned classifier for (2062), 速率限制, and 请求量较高 if the plugin/runtime has the right seam; otherwise keep only those specific markers in the shared matcher, and add positive MiniMax sample coverage plus negative auth/billing/maintenance retry-only cases.

## Reproduction Assessment

Yes. Static source inspection gives a high-confidence reproduction path: current main has no exact 速率限制, 请求量较高, or (2062) matcher, while the PR branch would match generic 请稍后重试 and short successful assistant text containing common rate-limit/overload phrases.

## Solution Assessment

No. The PR addresses a real remaining gap, but the current implementation is not the best fix because it combines precise MiniMax markers with an overly broad retry phrase and scans arbitrary non-error assistant output without preserving the matched failover reason.

  - body: The new non-error branch scans any short assistant content with broad rate-limit and overload matchers.   - confidence: 0.91
- **[P2] Remove generic retry text from rate-limit matching:** `src/agents/pi-embedded-helpers/failover-matches.ts:21`
  - body: `请稍后重试` means "please retry later" and is commonly appended to non-rate-limit failures. Since rate-limit classification runs before timeout, billing, and auth paths, this can force unrelated Chinese provider errors into `rate_limit`.
  - confidence: 0.9
- **[P2] Preserve matched failover reason for text-only errors:** `src/agents/pi-embedded-helpers/errors.ts:1048-1049`
  - body: The text-content branch returns only `true`, while the runner still derives `assistantFailoverReason` from `errorMessage`, which is empty for the targeted `end_turn` case. Those retries can be emitted as `unknown` and skip rate-limit or overload-specific handling.
  - confidence: 0.84
