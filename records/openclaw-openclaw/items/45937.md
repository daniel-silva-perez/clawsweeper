## Summary

Keep open. The PR addresses a real remaining MiniMax/CN-provider failover gap, but it is not merge-ready: current main has broader CJK failover coverage from merged #56242 while still lacking this PR’s exact MiniMax 2062 markers, and this PR adds over-broad matching that can misclassify normal assistant text or non-rate-limit provider failures.

## What This Changes

The PR adds MiniMax/CN rate-limit markers to shared failover patterns and changes assistant failover detection to scan short non-error assistant text for rate-limit or overload phrases.

## Best Possible Solution

Keep the useful MiniMax signal but narrow it: add precise handling for 速率限制, 请求量较高, and MiniMax code 2062 without standalone generic retry text, preferably in the MiniMax provider's classifyFailoverReason hook if that hook sees the relevant payloads. Only inspect non-error assistant content when the response is known to be a provider error payload, preserve the matched failover reason, and add positive MiniMax sample coverage plus negative retry-only billing/auth/maintenance cases.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence reproduction path: the PR would match generic 请稍后重试 and arbitrary short assistant text with broad failover patterns, while current main still lacks the exact MiniMax 速率限制/请求量较高/2062 markers. No dynamic tests were run because this review was required to keep the checkout read-only.

## Solution Assessment

No. The requested direction is valid, but the PR is not the best implementation because it mixes precise MiniMax markers with a generic retry phrase and scans normal assistant output without preserving a reason; a provider-owned or narrowly scoped matcher with focused tests is safer.

  - body: The new non-error branch scans any short assistant content with broad rate-limit and overload matchers.   - body: `请稍后重试` means "please retry later" and is commonly appended to non-rate-limit failures. Since rate-limit classification runs before timeout, billing, and auth paths, this can force unrelated Chinese provider errors into `rate_limit`.
  - confidence: 0.9
- **[P2] Preserve the matched reason for text-only failovers:** `src/agents/pi-embedded-helpers/errors.ts:1048-1049`
  - body: The text-content branch returns only `true`, while the runner still derives `assistantFailoverReason` from `errorMessage`, which is empty for the targeted `end_turn` case. These retries can be emitted as `unknown` and skip rate-limit or overload-specific handling.
  - confidence: 0.84
