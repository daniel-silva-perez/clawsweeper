## Summary

Keep open. The narrow MiniMax/CN rate-limit intent is still valid on current main, but this PR should not merge as-is because the proposed text matcher can classify normal successful assistant text as provider failure, the generic Chinese retry phrase is too broad for rate limits, and the text-only path loses the matched failover reason.

## What This Changes

The PR adds Chinese rate-limit patterns to the shared failover matcher and changes assistant failover detection to scan short non-error assistant text for rate-limit or overload patterns.

## Best Possible Solution

Add precise MiniMax/CN failover classification without standalone generic retry text. Prefer MiniMax's provider-owned `classifyFailoverReason` hook for the 2062/provider payload unless maintainers intentionally want the specific `速率限制` and `请求量较高` terms in the shared matcher; preserve the matched reason for any text-response path and add positive 2062 plus negative retry-only regression tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Restrict text matching to provider error payloads:** `src/agents/pi-embedded-helpers/errors.ts:1047`
  - body: The new non-error branch scans any short assistant content with broad rate-limit and overload matchers. A successful answer that mentions `rate limit`, `usage limit`, or `high demand` can be treated as a provider failure, causing profile rotation or fallback instead of returning the response.
  - confidence: 0.91
- **[P2] Remove generic retry text from rate-limit matching:** `src/agents/pi-embedded-helpers/failover-matches.ts:21`
  - body: `请稍后重试` means “please retry later” and is commonly appended to non-rate-limit failures. Since rate-limit classification runs before timeout, billing, and auth paths, this can force unrelated Chinese provider errors into `rate_limit`.
  - confidence: 0.9
- **[P2] Preserve the matched reason for text-only errors:** `src/agents/pi-embedded-helpers/errors.ts:1048`
  - body: The text-content branch returns only `true`, while the runner still derives `assistantFailoverReason` from `errorMessage`, which is empty for the targeted `end_turn` case. Those retries can be emitted as `unknown` and skip rate-limit or overload-specific handling.
  - confidence: 0.84
