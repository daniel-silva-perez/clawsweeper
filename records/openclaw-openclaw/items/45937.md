## Summary

Keep open for https://github.com/openclaw/openclaw/pull/45937. Current main partially covers CJK failover through #56242, but the MiniMax 2062/end-turn content case is not fully implemented, and this PR branch still has concrete blocking correctness and security findings.

## What This Changes

The PR adds MiniMax/CN rate-limit markers to shared failover patterns and expands assistant failover detection to scan short non-error assistant text for rate-limit or overload phrases.

## Best Possible Solution

Keep the useful MiniMax signal but narrow it: add precise handling for 速率限制, 请求量较高, and MiniMax code 2062 with focused positive and negative tests, and handle text responses only when they are known provider error payloads or through a provider-owned classification hook. Do not add standalone 请稍后重试 as a rate-limit marker.

## Reproduction Assessment

Yes. Static inspection provides a high-confidence path: current main lacks the exact MiniMax markers, while the PR diff would match generic retry text and normal short assistant text through shared failover patterns.

## Solution Assessment

No. The requested direction is valid, but this PR is not the best implementation because it combines precise MiniMax markers with a generic retry phrase and scans non-error assistant output without preserving the matched reason; a narrower matcher or provider-owned hook with regression tests is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Restrict text matching to provider error payloads:** `src/agents/pi-embedded-helpers/errors.ts:1047-1049`
  - body: The new non-error branch scans any short assistant content with broad rate-limit and overload matchers. A successful answer that mentions `rate limit`, `usage limit`, or `high demand` can be treated as a provider failure, causing profile rotation or fallback instead of returning the response.
  - confidence: 0.91
- **[P2] Remove generic retry text from rate-limit matching:** `src/agents/pi-embedded-helpers/failover-matches.ts:21`
  - body: `请稍后重试` means "please retry later" and is commonly appended to non-rate-limit failures. Since rate-limit classification runs before timeout, billing, and auth paths, this can force unrelated Chinese provider errors into `rate_limit`.
  - confidence: 0.9
- **[P2] Preserve the matched reason for text-only failovers:** `src/agents/pi-embedded-helpers/errors.ts:1048-1049`
  - body: The text-content branch returns only `true`, while the runner still derives `assistantFailoverReason` from `errorMessage`, which is empty for the targeted `end_turn` case. These retries can be emitted as `unknown` and skip rate-limit or overload-specific handling.
  - confidence: 0.84
