## Summary

Keep open. The underlying MiniMax/CN-provider rate-limit case is still not fully covered on current main, but the PR should not merge as-is because it adds broad non-error assistant text matching and a generic retry phrase that can misroute successful turns or unrelated provider failures.

## What This Changes

The PR adds Chinese MiniMax/CN-provider rate-limit strings to shared failover matching and broadens assistant failover detection to scan short non-error assistant text for rate-limit or overload patterns.

## Best Possible Solution

Add precise MiniMax 2062 classification without standalone generic retry text. Prefer a MiniMax provider-owned `classifyFailoverReason` hook for `(2062)`, `速率限制`, and `请求量较高`, or keep only those precise terms in the shared matcher if maintainers intentionally want generic CN-provider coverage; add positive MiniMax tests and negative retry-only/auth/billing tests.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence path: current main will not match the exact MiniMax sample terms, while the PR branch would also match generic `请稍后重试` and any short non-error assistant content containing shared rate-limit/overload phrases.

## Solution Assessment

No. The branch addresses a real gap, but the current approach is not the narrowest maintainable fix because it mixes precise MiniMax markers with a broad retry phrase and scans successful assistant text without preserving a failover reason.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Restrict text matching to provider error payloads:** `src/agents/pi-embedded-helpers/errors.ts:1047`
  - body: The new non-error branch scans any short assistant content with broad rate-limit and overload matchers. A successful answer that mentions `rate limit`, `usage limit`, or `high demand` can be treated as a provider failure, causing profile rotation or fallback instead of returning the response.
  - confidence: 0.91
- **[P2] Remove generic retry text from rate-limit matching:** `src/agents/pi-embedded-helpers/failover-matches.ts:21`
  - body: `请稍后重试` means “please retry later” and is commonly appended to non-rate-limit failures. Since rate-limit classification runs before timeout, billing, and auth paths, this can force unrelated Chinese provider errors into `rate_limit`.
  - confidence: 0.9
- **[P2] Preserve matched failover reason for text-only errors:** `src/agents/pi-embedded-helpers/errors.ts:1048`
  - body: The text-content branch returns only `true`, while the runner still derives `assistantFailoverReason` from `errorMessage`, which is empty for the targeted `end_turn` case. Those retries can be emitted as `unknown` and skip rate-limit or overload-specific handling.
  - confidence: 0.84
