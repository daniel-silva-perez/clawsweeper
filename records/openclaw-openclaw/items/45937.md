## Summary

Keep open. The PR targets a real remaining MiniMax/CN-provider gap: current main has broad CJK failover coverage from merged #56242, but it still does not match the exact MiniMax 2062 sample terms. The branch is not mergeable as-is because it adds an overly broad retry phrase and scans arbitrary short non-error assistant text, which can turn successful responses or non-rate-limit failures into fallback events.

## What This Changes

The PR adds Chinese MiniMax/CN-provider rate-limit markers to shared failover matching and changes assistant failover detection to scan short non-error assistant text for rate-limit or overload phrases.

## Best Possible Solution

Add precise MiniMax 2062 handling without standalone generic retry text. Prefer a MiniMax/provider-owned classifier for 2062, 速率限制, and 请求量较高 if the provider hook path can see the relevant errors; otherwise add only those narrow markers to the shared matcher, and add positive MiniMax sample coverage plus negative retry-only billing/auth/maintenance/overload cases. Non-error assistant text should only trigger fallback when the provider response is known to be an error payload and the matched reason is preserved.

## Reproduction Assessment

Yes. Static source inspection gives a high-confidence reproduction path: current main does not match the exact MiniMax sample markers, while the PR diff would match generic 请稍后重试 and short successful assistant text containing broad rate-limit/overload phrases. No dynamic tests were run because this review is required to keep the checkout read-only.

## Solution Assessment

No. The PR addresses a valid gap, but the current approach is not the narrowest maintainable fix because it combines precise MiniMax markers with a generic retry phrase, scans arbitrary non-error assistant output, and does not preserve the matched failover reason for that text-only path.
