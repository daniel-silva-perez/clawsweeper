## Summary

Keep this PR open. Current main still does not recognize the MiniMax 2062 sample as a rate-limit failover, but the proposed patch should not merge as-is because it can classify generic retry text and ordinary short assistant answers as provider failures.

## What This Changes

The PR adds Chinese rate-limit patterns to `src/agents/pi-embedded-helpers/failover-matches.ts` and extends `isFailoverAssistantError` in `src/agents/pi-embedded-helpers/errors.ts` to classify short non-error assistant text as failover-worthy.

## Best Possible Solution

Implement a precise MiniMax/CN failover classifier, preferably in MiniMax's provider-owned `classifyFailoverReason` hook or a conservative shared matcher. It should match the 2062/rate-limit payload, omit generic retry-only text, preserve the failover reason, and add positive MiniMax 2062 plus negative Chinese retry-phrase regression tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Restrict non-error text matching to real provider errors:** `src/agents/pi-embedded-helpers/errors.ts:1047`
  - body: This branch runs rate-limit and overload matchers against any short `end_turn` assistant text. A normal answer that mentions `rate limit`, `usage limit`, or `high demand` can be treated as a failed provider turn, causing rotation or fallback instead of returning the successful response.
  - confidence: 0.91
- **[P2] Remove generic retry text from rate-limit patterns:** `src/agents/pi-embedded-helpers/failover-matches.ts:21`
  - body: `请稍后重试` means “please retry later” and is commonly appended to non-rate-limit failures. Because rate limits are classified before timeout, billing, and auth paths, this token can force unrelated Chinese provider errors into `rate_limit`.
  - confidence: 0.9
- **[P2] Preserve the matched failover reason for text-only errors:** `src/agents/pi-embedded-helpers/errors.ts:1048`
  - body: The new text-content branch returns only `true`, while the runner still computes `assistantFailoverReason` from `errorMessage`, which is empty for the `end_turn` case. Those retries can be emitted as unknown and skip reason-specific backoff or cooldown behavior.
  - confidence: 0.84
