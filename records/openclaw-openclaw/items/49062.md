## Summary

Keep this issue open. Current main still has no explicit Telegram channel_post policy for sender-scoped groups[chatId].allowFrom, and the related same-author PR remains open while explicitly leaving that policy out of scope.

## What This Changes

Define and document how Telegram channel_post native commands should handle sender-scoped groups[chatId].allowFrom when Telegram does not expose a human sender identity.

## Best Possible Solution

Maintain an explicit Telegram channel_post auth rule for sender-scoped allowFrom: either treat channel posts as explicit-chat-only and ignore sender overrides, or fail closed with a clear unsupported-configuration message. The chosen policy should be covered by targeted native command/group-access tests and documented on the Telegram channel docs page.

## Reproduction Assessment

Yes. The static reproduction path is high-confidence: send a Telegram /status channel_post with sender-scoped groups[chatId].allowFrom configured, then trace native command handling through the ctx.message-only guard and sender-id-based auth helpers; live Telegram setup would only confirm the same dependency contract.

## Solution Assessment

Unclear until a maintainer chooses the product/security policy. The related PR is useful for explicit chat allow behavior, but it is not the best complete solution for this issue because it intentionally does not settle sender-scoped channel_post semantics.
