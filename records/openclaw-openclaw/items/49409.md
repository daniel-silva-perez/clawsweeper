## Summary

Keep this PR open. Current main still writes one slugged session-memory file per /new or /reset, still defaults to the LLM slug path outside tests, and still documents/tests the `YYYY-MM-DD-slug.md` behavior. The PR remains a plausible implementation candidate for canonical daily appends and nonblocking slugging, but the submitted diff has unresolved correctness concerns from review: duplicate daily headers under concurrent first writes, unclear zero-timeout semantics, and removed workspace-routing coverage.

## Best Possible Solution

Keep this PR open as a useful implementation candidate, but require a refreshed patch against current main. The best path is to implement canonical `memory/YYYY-MM-DD.md` writer appends, make LLM slugging opt-in or otherwise nonblocking for /new and /reset, create the first daily header without a TOCTOU race, preserve workspaceDir/display-session-key coverage, update hook docs/metadata, and validate with targeted session-memory and fs-safe tests plus the appropriate changed gate.

## Risks / Open Questions

- The submitted daily-file implementation can duplicate the top-level daily header when concurrent /new or /reset hooks both probe before either writes.
- The submitted timeout config needs an explicit policy for zero and invalid values before adding `OPENCLAW_SESSION_MEMORY_SLUG_TIMEOUT_MS` or `slugTimeoutMs`.
- Changing `appendFileWithinRoot` return semantics touches a shared safe-write helper, so symlink, hardlink, alias, and root-boundary protections need to remain covered.
