## Summary

Keep this PR open. Current main still implements the old session-memory writer behavior: the hook uses LLM slug generation by default outside tests, writes one `YYYY-MM-DD-slug.md` file per reset/new event, and the docs still describe that output. Main has only added reader-side compatibility for date-prefixed slugged files, so the PR’s central requested behavior is not implemented. The PR is still useful as a candidate, but it should not be treated as obsolete or safe to close because review comments identify real correctness issues in the submitted patch.

## Best Possible Solution

Keep the PR open as a useful implementation candidate, but require a refreshed patch against current main. The best path is to preserve main’s reader compatibility for slugged date-prefixed artifacts, decide explicitly whether session-memory should write canonical `memory/YYYY-MM-DD.md`, make LLM slug generation opt-in or otherwise nonblocking for reset/new hooks, implement daily appends without the header TOCTOU race, retain workspace-context regression coverage, update docs/config help, and run targeted session-memory/fs-safe tests plus the appropriate changed gate.

## Risks / Open Questions

- The PR as submitted can duplicate the top-level daily header when two reset/new hooks race on the first append for a day.
- The PR’s timeout fallback needs an explicit policy for zero and invalid values; the provided review comments show the current patch treats `0` as absent through `||` chaining.
- The submitted patch removes existing workspaceDir/display-session-key regression coverage that still protects current main behavior.
- Changing `appendFileWithinRoot` to return open metadata is security-sensitive and should be kept minimal, with symlink/race/hardlink protections preserved.
