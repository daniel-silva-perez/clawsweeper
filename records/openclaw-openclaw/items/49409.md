## Summary

Keep this PR open. Current main still writes per-session `YYYY-MM-DD-slug.md` memory files and still allows the session-memory hook to call the LLM slug generator by default on `/new` and `/reset`, so the PR’s central behavior is not implemented. Main has partially reduced the reader-side mismatch by loading both canonical daily files and date-prefixed slugged files, but the PR remains only a candidate and is not safe to merge as submitted because review comments identify a real header TOCTOU race, a timeout parsing issue, and a removed workspace-context regression test.

## Best Possible Solution

Keep this PR open as useful but require a refreshed patch against current main: preserve main’s reader compatibility for date-prefixed slugged files, decide explicitly whether the writer should switch to canonical `memory/YYYY-MM-DD.md`, make `/new` and `/reset` avoid LLM work by default with matching docs/config coverage, implement daily appends without the header TOCTOU race, and retain the workspaceDir/display-session-key regression test. Then run targeted session-memory and fs-safe tests plus the changed gate appropriate for hook and infra changes.

## Risks / Open Questions

- The PR as submitted can produce duplicate top-level daily headers when two `/new` or `/reset` hooks append to a new daily file concurrently.
- The PR’s timeout fallback, per review comments, treats `0` as absent through falsy `||` chaining; maintainers need an explicit policy for zero or positive finite timeout values.
- The PR removes existing workspaceDir/display-session-key regression coverage that still protects a current main code path.
- Changing `appendFileWithinRoot` to return `SafeWritableOpenResult` expands a security-sensitive filesystem helper contract; any refreshed patch should avoid exposing unnecessary handle state and preserve symlink/race protections.
