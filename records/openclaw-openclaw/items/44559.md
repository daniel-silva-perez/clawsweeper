## Summary

Keep this issue open. Current main still creates the native Windows gateway Scheduled Task as a logon-oriented task with `/SC ONLOGON`, `/RL LIMITED`, and `/RU <user> /NP /IT` when a task user is available, and the documented fallback remains a per-user Startup-folder login item. Related PR #68149 is open and relevant, but it is not merged and changes the Windows command-execution and privilege model, so it needs maintainer review rather than making this issue obsolete.

## Best Possible Solution

Keep this open for maintainer product/design handling. The best path is to decide the supported native Windows persistence model, then either add a tested service/non-interactive Scheduled Task or PowerShell ScheduledTasks mode that survives closing PowerShell/session with documented privilege and security tradeoffs, or explicitly document native Windows gateway install as login-session scoped and point headless users to WSL2/systemd. Review #68149 as the current implementation candidate before using it to close this issue.
