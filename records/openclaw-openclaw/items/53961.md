## Summary

Keep open. The delivery observability gap is still real on current main and related issue #49055 remains open, but this draft PR is not mergeable as-is: it can drop the parseable `--json --deliver` envelope on strict delivery failures, and it still treats intentional `message_sending` hook cancellations as failed delivery until #57843 or equivalent outcome metadata lands.

## What This Changes

This PR modifies `src/agents/command/delivery.ts` and `src/agents/command/delivery.test.ts` to add delivery status tracking, diagnostic delivery-failure logging, JSON `deliveryStatus` output, and delivery-path tests for agent command results.

## Best Possible Solution

Keep the delivery observability direction, but land it only after the branch is rebased onto current main, preserves the single parseable `--json` stdout contract on strict delivery errors, and consumes cancellation metadata from #57843 or an equivalent outbound delivery API. The JSON status surface should be coordinated with #57755 rather than merged as competing contract changes.

## Reproduction Assessment

Yes. Code inspection gives two high-confidence reproduction paths: strict `--json --deliver` can throw before the PR's delayed JSON block runs, and an all-cancelled `message_sending` hook returns `[]`, which the PR maps to `succeeded: false`.

## Solution Assessment

No. The proposed direction is useful, but the current patch is not the best maintainable solution until it preserves JSON output on throw paths and distinguishes expected hook cancellation from actual delivery failure.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Emit JSON before rethrowing delivery failures:** `src/agents/command/delivery.ts:330`
  - body: The PR suppresses the existing early JSON envelope whenever `--deliver` is set. In strict delivery mode, `deliverOutboundPayloads` can rethrow before the later JSON block runs, so `openclaw agent --json --deliver` can produce no parseable stdout despite the documented JSON contract.
  - confidence: 0.9
- **[P2] Treat hook cancellations as non-failures:** `src/agents/command/delivery.ts:426`
  - body: `deliverySucceeded = results.length > 0` treats every empty result set as a failed send, but `deliverOutboundPayloads` intentionally returns `[]` when `message_sending` hooks cancel all payloads. That creates false failure telemetry/status for expected cancellations, so this should consume cancellation metadata or wait for the DeliveryOutcome work.
  - confidence: 0.86
