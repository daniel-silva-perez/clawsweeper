## Summary

Keep open. Current main still lacks structured deliveryStatus tracking for command delivery, so the PR is not obsolete, but the current draft should not merge as-is because it can suppress parseable --json output on strict delivery failures and it still treats intentional hook cancellations as failed delivery. The useful path is maintainer sequencing with #57843 and #57755 rather than closing or replacing this PR automatically.

## What This Changes

This PR adds deliveryStatus tracking and diagnostic delivery-failure logging to deliverAgentCommandResult, includes that status in --json --deliver output, and expands command delivery tests.

## Best Possible Solution

Keep the delivery observability work, but land it only after the outbound delivery API can distinguish cancellation from failure and the command JSON path guarantees one parseable stdout envelope on both success and strict delivery-error paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Emit JSON before rethrowing delivery failures:** `src/agents/command/delivery.ts:330`
  - body: The changed `if (opts.json && !deliver)` suppresses the existing early JSON envelope for `--json --deliver`. In strict delivery mode, `deliverOutboundPayloads` can rethrow before the later status envelope runs, so scripts can receive no parseable JSON even though the CLI docs reserve stdout for JSON output.
  - confidence: 0.9
- **[P2] Do not mark hook cancellations as failed delivery:** `src/agents/command/delivery.ts:426`
  - body: `deliverySucceeded = results.length > 0` treats an all-cancelled `message_sending` hook as a failed send. Current outbound delivery intentionally continues without adding a result when hooks return cancel, so expected cancellations such as thread ownership would produce false failure status/logs.
  - confidence: 0.86
