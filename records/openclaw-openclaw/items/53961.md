## Summary

Keep open. This is a useful draft implementation, but it is not ready to merge: the PR can regress the `--json --deliver` stdout contract on strict delivery failures, and it still treats intentional hook cancellations as failed delivery while the needed cancellation metadata remains in separate open work.

## What This Changes

This PR adds `deliveryStatus` tracking, delivery failure diagnostics, JSON output surfacing for `openclaw agent --json --deliver`, and expanded tests for command delivery outcomes.

## Best Possible Solution

Keep the delivery observability direction, but land it only after outbound delivery exposes cancellation metadata or an equivalent signal, and ensure `openclaw agent --json --deliver` always emits one parseable JSON stdout envelope on both success and strict delivery-error paths. Consolidate the JSON status surface with #57755 after #57843 or equivalent API work is settled.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Emit JSON before rethrowing delivery failures:** `src/agents/command/delivery.ts:330`
  - body: The PR suppresses the existing early JSON envelope whenever `--deliver` is set. In strict delivery mode, `deliverOutboundPayloads` can rethrow before the later JSON block runs, so `openclaw agent --json --deliver` can produce no parseable stdout despite the documented JSON contract.
  - confidence: 0.9
- **[P2] Do not mark hook cancellations as failed delivery:** `src/agents/command/delivery.ts:426`
  - body: `deliverySucceeded = results.length > 0` treats an all-cancelled `message_sending` hook as a failed send. Current outbound delivery intentionally returns no result for cancelled payloads, so expected cancellations would produce false failure status/logging and could drive incorrect retry behavior.
  - confidence: 0.86
