## Summary

Keep open. Current main has not implemented the requested cooldown exemption, so the PR is not obsolete, but the branch is not merge-ready because the raw HTTP 400 short-circuit is broader than the existing failover classifier contract and can suppress real auth, billing, or rate-limit profile penalties.

## What This Changes

The PR adds status-aware guards to auth-profile failure marking, plumbs prompt error text into the embedded runner for HTTP status extraction, adds auth-profile cooldown unit tests, and adds a changelog entry.

## Best Possible Solution

Keep the narrow fix direction, but base the skip on the final failure classification instead of raw status alone: skip profile mutation for confirmed `format` payload-shape failures, preserve cooldown/disable behavior for `billing`, `auth`, `rate_limit`, and other specific reasons even when the HTTP status is 400, and cover prompt plus assistant paths with focused regression tests.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: a call like `markAuthProfileFailure({ reason: "billing", statusCode: 400 })` would return early in the PR even though current classifier and auth-profile policy preserve and record non-timeout `billing` failures.

## Solution Assessment

No. The PR solves part of the reported false-cooldown problem, but the best fix is to reuse the existing status-plus-message classification semantics and only skip when the resolved reason is `format`, rather than treating every raw HTTP 400 as a malformed-payload case.

Use a structured status source or a helper with deterministic last/leading-status semantics covered by tests.
