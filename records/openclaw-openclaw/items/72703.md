## Summary

Keep open. Current main still drains queued runtime system events as `System:` text and prepends them to user prompt bodies, and the linked closing PR #72761 is open and unmerged in the provided GitHub context. This should wait for maintainer review and merge of #72761 or an equivalent narrow fix.

## What This Changes

The issue asks to route background task updates, deferred maintenance notices, and heartbeat/system prompts through system or dedicated runtime context instead of polluting user-role prompt text.

## Best Possible Solution

Land #72761 or an equivalent focused change that routes drained queued runtime system events into `extraSystemPrompt` or another system-context channel, keeps them out of `commandBody`, `followupRun.prompt`, and transcript prompt text, and preserves the existing untrusted-event ownership downgrade.

## Reproduction Assessment

Yes. Current tests provide a high-confidence read-only reproduction path: mocked `drainFormattedSystemEvents` output is asserted to appear in `commandBody` and `followupRun.prompt` rather than runtime system context.

## Solution Assessment

Yes for the queued runtime system-event bug. Moving drained events into runtime system context is the narrow maintainable solution; heartbeat prompt semantics should stay scoped separately unless maintainers decide to change them too.
