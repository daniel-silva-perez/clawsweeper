## Summary

Keep open. Current main still routes drained queued system events into user prompt text, and the linked fix PR #72761 is open and unmerged, so this should wait for maintainer review/merge rather than be closed by cleanup.

## What This Changes

The issue asks for background task updates, deferred maintenance notices, and related runtime wake prompts to be delivered as system-role or dedicated runtime context instead of user-role prompt text.

## Best Possible Solution

Land #72761 or an equivalent narrow change that routes drained runtime-generated system events into existing runtime system context, keeps them out of `commandBody`, queued prompts, and transcript prompts, and preserves the current untrusted-event ownership downgrade and documented heartbeat behavior unless maintainers explicitly decide to change heartbeat semantics.

## Reproduction Assessment

Yes. Current main has a direct focused reproduction in `get-reply-run.media-only.test.ts`: mocked `drainFormattedSystemEvents` output is asserted to appear in `commandBody` and `followupRun.prompt` instead of runtime system context.

## Solution Assessment

Yes for the core queued-system-event bug. Moving drained events into `extraSystemPrompt` while removing `systemEventBlocks` from user prompt assembly is the narrowest maintainable path; heartbeat prompt semantics need maintainer scope confirmation because current docs and the linked PR keep them as user-message turns.
