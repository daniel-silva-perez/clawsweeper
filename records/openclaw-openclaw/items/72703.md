## Summary

Keep open. Current main still prepends drained runtime system events into user prompt text, and the linked fix PR #72761 is open and unmerged, so this should wait for maintainer review/merge or an equivalent fix rather than cleanup closure.

## What This Changes

The issue requests that background task updates, deferred maintenance notifications, and heartbeat-like runtime notices be delivered through system context or another non-user mechanism instead of as user-role prompt text.

## Best Possible Solution

Land #72761 or an equivalent narrow change that routes drained runtime-generated system events through runtime system context, keeps them out of commandBody, queued prompts, and transcript prompts, and preserves the existing untrusted-event ownership downgrade unless maintainers intentionally revise heartbeat semantics.

## Reproduction Assessment

Yes. Current main has a focused source/test reproduction path: mocked drainFormattedSystemEvents output is asserted to appear in commandBody and followupRun.prompt rather than runtime system context in src/auto-reply/reply/get-reply-run.media-only.test.ts.

## Solution Assessment

Yes for the queued runtime-system-event bug; moving drained events out of buildReplyPromptBodies and into extraSystemPrompt is the narrowest maintainable direction. It is unclear whether heartbeat prompt semantics should change too, because docs currently describe heartbeat/system-event surfacing as prompt text.
