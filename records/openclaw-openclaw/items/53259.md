## Summary

Keep open. Current main still appears to have the Discord elevated fallback bug, and this PR is a useful targeted fix, but the latest diff is not merge-ready because it hard-codes Discord fallback logic in core and regresses the human `sandbox explain` diagnostics that docs rely on.

## What This Changes

The PR adds a Discord elevated `allowFrom` fallback resolver, uses it from runtime elevated authorization and `sandbox explain` with account context, and adds focused regression tests.

## Best Possible Solution

Land a narrow rebased fix that implements Discord fallback through the Discord channel plugin or another extension-owned adapter, shares a generic fallback-resolution path between runtime auth and `sandbox explain` with account context, and preserves the existing human and JSON diagnostics contract.

## Reproduction Assessment

Yes. The issue is reproducible at the config-resolution level from current main: docs promise the Discord fallback, runtime only calls the plugin fallback hook, the Discord plugin does not provide that adapter, and `sandbox explain` only reads `tools.elevated.allowFrom[channel]`.

## Solution Assessment

No. The intended behavior is correct, but this PR is not the best final shape because it puts Discord-specific config fallback in core and removes existing `sandbox explain` human guidance; the safer fix is an extension-owned fallback adapter plus preserved diagnostics output.
