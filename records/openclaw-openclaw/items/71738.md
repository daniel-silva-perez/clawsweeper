## Summary

Keep this PR open for rework. Current main still has the Matrix thread-history and reply-placement problems this PR targets, but the proposed patch introduces lossy main-room pagination and stale thread-watermark behavior that should be fixed before merge.

## What This Changes

This PR scopes Matrix room history and message reads by thread, adjusts Matrix thread send/edit relation metadata, forwards `threadId` through Matrix message actions, and adds focused Matrix regression tests.

## Best Possible Solution

Land a reworked Matrix-only implementation that keeps thread scope through room-history, read, send, and edit paths; keeps pagination cursors aligned with returned events; rejects stale thread snapshot consumption after sub-queue eviction; updates Matrix history docs; and preserves focused Matrix regression coverage.

## Reproduction Assessment

Yes. The current-main behavior is reproducible by source inspection: thread relation fallback metadata is unconditional, threaded edits write reply context on the outer replace relation, room-history APIs have no thread scope, and Matrix reads have no thread endpoint path.

## Solution Assessment

No, not yet. The requested direction is narrow and maintainable, but this implementation should fix the main-room pagination cursor and thread sub-queue watermark liveness issues before it is the best merge path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve unread events when over-fetching main timeline:** `extensions/matrix/src/matrix/actions/messages.ts:183`
  - body: When main-room reads filter thread events, this path can fetch extra pages, collect more non-thread events than `limit`, then drop the surplus with `slice(0, limit)` while returning `nextBatch` from the last fetched page. Those dropped events are now behind the returned cursor, so callers paginating with it skip readable main-room history.
  - confidence: 0.88
- **[P2] Reject stale consumes for evicted thread queues:** `extensions/matrix/src/matrix/monitor/room-history.ts:411`
  - body: `consumeHistory` only validates the room generation before writing a thread watermark. If a thread sub-queue is evicted and later recreated while an older trigger is still in flight, the late consume can advance the new thread watermark past fresh messages and hide them from the agent.
  - confidence: 0.86
- **[P3] Clear legacy room watermarks during room eviction:** `extensions/matrix/src/matrix/monitor/room-history.ts:133-139`
  - body: `clearRoomWatermarks` now only recognizes JSON watermark keys, but `computePendingHistory` still reads legacy `agentId:roomId` keys. After upgrading, evicted rooms can leave legacy entries in the capped watermark map, reducing available capacity for active rooms.
  - confidence: 0.8
