## Summary

Keep open. The PR targets a real stale heartbeat/cron timestamp bug still present on current main, but the proposed matcher is too broad and can rewrite or drop user-authored reminder content that begins with `Current time:`.

## What This Changes

The PR changes `appendCronStyleCurrentTimeLine` to replace or collapse existing `Current time:` lines and adds colocated Vitest coverage for empty input, append, refresh, and duplicate-collapse cases.

## Best Possible Solution

Keep the shared-helper refresh approach, but constrain replacement to OpenClaw-generated timestamp lines or a backward-compatible internal marker, preserve user-authored `Current time:` content, and add a regression test for that preservation case.

## Reproduction Assessment

Yes. The original bug is reproducible by passing a helper output containing an old generated `Current time:` line back into `appendCronStyleCurrentTimeLine` with a newer `nowMs`; the PR regression is reproducible by passing reminder text with a user-authored line starting `Current time:` through the new matcher.

## Solution Assessment

No. The helper-level fix is the right narrow boundary, but the current regex is not the best implementation because it cannot distinguish generated timestamp lines from user-authored cron or workspace content.
