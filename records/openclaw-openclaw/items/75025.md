## Summary

Keep open. The PR targets a real stale-heartbeat timestamp bug, but it should not merge as written because the line-wide matcher would also rewrite user-authored cron reminder content that starts with `Current time:`; current `main` passes reminder text through verbatim before timestamp injection, and nearby code already treats user-authored `Current time:` text as a valid case to preserve.

## What This Changes

The PR modifies `src/agents/current-time.ts` to refresh or collapse existing `Current time:` lines instead of returning stale text, and adds `src/agents/current-time.test.ts` coverage for empty input, appending, refreshing, and duplicate-collapse behavior.

## Best Possible Solution

Keep the shared-helper fix, but make it distinguish OpenClaw-generated timestamp lines from user-authored content: refresh or collapse only generated timestamp lines, append a fresh generated line when none exists, and add regression coverage that preserves cron reminder text beginning with `Current time:`.

## Reproduction Assessment

Yes. The original bug is reproducible by calling `appendCronStyleCurrentTimeLine` with an 08:00 `nowMs`, feeding that output back with a 10:00 `nowMs`, and observing current `main` keep the 08:00 line. The PR regression is reproducible from the patch by passing cron reminder text with a line like `Current time: check the server clock`, which the broad matcher would replace or drop.

## Solution Assessment

No. Refreshing `appendCronStyleCurrentTimeLine` is the narrowest maintainable boundary, but the proposed `^Current time:.*$` matcher is too broad; the safer implementation should match only the helper-generated timestamp format or use a compatible internal marker.
