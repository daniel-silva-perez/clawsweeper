## Summary

Current main still matches the reported bug: `cron.runs` with `scope === "all"` builds `jobNameById` from an unfiltered `context.cron.list({ includeDisabled: true })`, then passes those names into the all-runs reader, which can attach `jobName` to returned entries. The broader per-session cron issue remains open, and the two related fix PRs are closed unmerged, so this narrower run-history name leak should stay open.

## Best Possible Solution

Keep this issue open and fix the cron owner path directly. Thread authenticated caller identity into the gateway cron handler/service contract, filter the all-scope `jobNameById` source by caller-owned `agentId`/`sessionKey` with an explicit admin override, and add a regression test with two session-owned jobs proving all-scope run history does not attach or query another session's job name. Coordinate with #35447, but keep this narrower run-history leak tracked until a fix merges.
