## Summary

Current main still has the reported `cron.runs` all-scope name leak. The handler still builds `jobNameById` from the unfiltered `context.cron.list({ includeDisabled: true })`, the cron service list APIs do not accept caller/session filters, and run-log pagination still injects those names into returned entries. Related PR #49197 was closed unmerged, and the broader open #35447 does not explicitly preserve this narrower run-history `jobName` path, so this issue should stay open.

## Best Possible Solution

Keep this issue open and fix the cron owner path directly: thread authenticated caller context into the `cron.runs` handler, scope the all-scope `jobNameById` source by `agentId`/`sessionKey` with an admin override, and add a regression test with two session-owned jobs proving all-scope run history does not attach or query another session's job name. Coordinate the implementation with the broader #35447 cron isolation work.
