## Summary

Current main still has the reported `cron.runs` all-scope job-name leak path. The handler builds `jobNameById` from an unfiltered `context.cron.list({ includeDisabled: true })`, the cron service/list-page contracts still have no caller ownership filters, and the run-log reader injects those names into the response. Related implementation attempts are closed unmerged, and the broader #35447 tracker remains open but does not replace this narrower run-history enrichment bug.

## Best Possible Solution

Keep this issue open and fix the cron owner path directly. Thread authenticated caller identity into the gateway cron handler/service contract, scope the all-scope `jobNameById` source by caller-owned `agentId`/`sessionKey` with an explicit admin override, and add a regression test with two session-owned jobs proving all-scope run history does not attach or query another session's job name. Coordinate the implementation with #35447, but keep this narrower run-history leak tracked until merged.
