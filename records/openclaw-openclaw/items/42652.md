## Summary

Current main still has the Discord announce-routing gap: Discord metadata does not opt into session hydration, so `resolveAnnounceTarget` returns the parsed fallback without an accountId. The issue should stay open because PR #73975 is an open, unmerged closing implementation path for the same bug.

## What This Changes

The issue asks for Discord A2A announce delivery to use the target session's stored accountId instead of falling back to the default Discord account.

## Best Possible Solution

Land PR #73975 or an equivalent narrow Discord-owned change that opts Discord announce targets into the existing session hydration contract and adds regression coverage proving `deliveryContext.accountId` or `lastAccountId` reaches the final gateway `send`.

## Reproduction Assessment

Yes. The current unit path reproduces the resolver behavior with a Discord session key: it returns a fallback target without `accountId` and does not call `sessions.list`; the send path then forwards that missing accountId.

## Solution Assessment

Yes. The narrowest maintainable fix is to opt Discord into the existing `preferSessionLookupForAnnounceTarget` hydration path and cover the Discord accountId regression, rather than changing the default for every channel.
