## Summary

Keep open because current main still has the Discord announce account-routing gap, and an open unmerged PR already targets the issue with closing syntax. The source still returns a Discord key-derived fallback before session hydration, so the target session account identity is not recovered on that path.

## What This Changes

The issue requests restoring Discord agent-to-agent announce delivery so the announce send uses the target session account identity instead of falling back to the default Discord account.

## Best Possible Solution

Land PR #73975 or an equivalent narrow Discord-owned change that opts Discord announce targets into session hydration and adds regression coverage proving deliveryContext.accountId or lastAccountId reaches the final gateway send.

## Reproduction Assessment

Yes. A source-level reproduction is clear: a Discord session key resolves to a fallback target without accountId, Discord metadata does not request session lookup, and the A2A send forwards that missing accountId.

## Solution Assessment

Yes. The narrowest maintainable direction is to review and land the existing Discord-specific opt-in plus regression tests, rather than changing announce lookup defaults for every channel.
