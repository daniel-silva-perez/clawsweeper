## Summary

Current main still has the reported Discord announce-routing gap, and the issue should remain open because PR #73975 is an open, unmerged implementation path that explicitly targets this bug. The already-closed duplicate #51626 supports the same multi-account failure mode but does not make this issue implemented.

## What This Changes

Discord agent-to-agent announce delivery should preserve the target session's stored accountId instead of falling back to the default Discord account.

## Best Possible Solution

Land PR #73975 or an equivalent narrow Discord-owned change that opts Discord announce targets into the existing session hydration contract and adds regression coverage proving the stored session account reaches the final gateway send.

## Reproduction Assessment

Yes. Current main has a source-level and unit-level reproduction path: a Discord session key resolves to a fallback target without `sessions.list`, while the final send forwards that missing accountId.

## Solution Assessment

Yes. The narrowest maintainable fix is to opt Discord into the existing `preferSessionLookupForAnnounceTarget` hydration path and cover the accountId regression, rather than changing announce lookup defaults for every channel.
