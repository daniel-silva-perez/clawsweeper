## Summary

Current main already implements the intended null-argument behavior in the correct pre-validation seam and shipped it in v2026.4.27 via PR #72673. This PR should not merge: its normalization runs inside the wrapped execute path, after pi-agent-core has already validated tool arguments, and the existing review comments also identify schema false negatives in the branch.

## What This Changes

The PR adds a before-tool-call helper that converts `null` params to `{}` for tools whose schema appears to be an empty object, then uses the normalized params for execution, diagnostics, and loop outcome tracking.

## Best Possible Solution

Keep the shipped prepareArguments-based implementation from PR #72673 and close this PR as obsolete rather than merging a later execute-wrapper normalization path.

## Reproduction Assessment

Yes. The current regression test drives a `wiki_lint`-style parameterless object tool through `runAgentLoop` with `arguments: null`, which is the reported failure shape; dependency source confirms validation happens before execute unless `prepareArguments` handles it.

## Solution Assessment

No for this PR. The maintainable fix is the current main implementation in schema normalization because it feeds pi-agent-core before `validateToolArguments`; the PR branch normalizes too late and has additional empty-schema false negatives.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Move null normalization before Pi validation:** `src/agents/pi-tools.before-tool-call.ts:480`
  - body: This normalization runs inside the wrapped `execute`, but pi-agent-core calls `prepareToolCallArguments` and `validateToolArguments` before invoking `tool.execute`. For the reported `arguments: null` case, validation returns an immediate error and this new code is never reached, so the PR does not fix the loop.
  - confidence: 0.96
- **[P2] Handle empty required arrays as optional:** `src/agents/pi-tools.before-tool-call.ts:109`
  - body: `Array.isArray(schema.required)` is true for `required: []`, so schemas that explicitly declare no required fields still skip normalization. That leaves the reported parameterless-object case failing for a common JSON Schema shape.
  - confidence: 0.91
- **[P2] Treat empty properties objects as parameterless:** `src/agents/pi-tools.before-tool-call.ts:110`
  - body: Checking only `properties !== undefined` treats `properties: {}` as having parameters. Empty properties are semantically parameterless, so this branch would still reject `null` for tools like the reported `wiki_lint` schema.
  - confidence: 0.9
- **[P2] Record calls and outcomes with matching params:** `src/agents/pi-tools.before-tool-call.ts:511-516`
  - body: `runBeforeToolCallHook` records the call with the original params before this normalization, while `recordLoopOutcome` records the outcome with normalized params. When those differ, the outcome cannot attach to the original history entry and loop detection loses the paired call/result signal.
  - confidence: 0.78
