## Summary

Keep open. Current main still drops OpenRouter tool-support metadata before runtime compatibility, so metadata-known toolless models can still enter normal agent execution with tools enabled; the raw model-run probe path suppresses tools but does not cover the reported agent path.

## What This Changes

The issue asks OpenClaw to handle OpenRouter models that do not advertise tool support by suppressing tool-capable runtime surfaces or failing early with a clear compatibility error.

## Best Possible Solution

Add OpenRouter tool-capability plumbing at the provider/runtime boundary: preserve `supported_parameters.includes("tools")` in dynamic OpenRouter capabilities, propagate it to `compat.supportsTools`, and keep the existing default-true behavior only when tool capability is genuinely unknown.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: live OpenRouter metadata lacks `tools` for the reported model, current OpenClaw drops that metadata, and normal `runEmbeddedAttempt()` treats absent compat as tool-capable.

## Solution Assessment

Yes. The narrowest maintainable fix is provider-local OpenRouter capability plumbing that feeds the existing generic tool gate; a new non-tool agent execution mode would be broader than needed for this bug.
