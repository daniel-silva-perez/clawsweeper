## Summary

Keep open. Current main still drops OpenRouter tool-support metadata before runtime compatibility, so a metadata-known toolless model like `openrouter/perplexity/sonar-deep-research` can still reach normal agent execution with tools enabled. The raw model-run/probe path suppresses tools, but that does not resolve the reported `openclaw agent` path.

## What This Changes

The issue asks OpenClaw to handle OpenRouter models that do not advertise tool support by suppressing tool-capable runtime surfaces or failing early with a clear compatibility error.

## Best Possible Solution

Add OpenRouter tool-capability plumbing at the provider/runtime boundary: parse `supported_parameters.includes("tools")`, preserve it in dynamic OpenRouter capabilities, refresh or invalidate old cache entries that lack the field, and set `compat.supportsTools` on dynamic OpenRouter models while preserving default-true behavior when capability metadata is genuinely unknown.
