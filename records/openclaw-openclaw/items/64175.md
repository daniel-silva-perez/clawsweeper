## Summary

Current main still does not map OpenRouter `supported_parameters` tool metadata into runtime model compatibility, so the reported primary agent/runtime path can still treat `openrouter/perplexity/sonar-deep-research` as tool-capable. The prior ClawSweeper assessment remains valid on the current tree; the safer one-shot model-run path does not fix normal agent execution.

## What This Changes

The issue asks OpenClaw to handle OpenRouter models without tool support by avoiding tool-capable routing or failing early with a clear compatibility error.

## Best Possible Solution

Add OpenRouter tool-capability plumbing at the provider/runtime boundary: parse `supported_parameters.includes("tools")`, preserve it in dynamic OpenRouter capabilities, refresh or invalidate old cache entries that lack the field, and set `compat.supportsTools` on dynamic OpenRouter models so the existing embedded-runner tool gate suppresses tools for known toolless models while preserving the current default for truly unknown metadata.
