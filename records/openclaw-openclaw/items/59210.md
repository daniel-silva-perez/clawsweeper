## Summary

Keep open. Current main still does not document the remote/headless WhatsApp QR handoff problem on the WhatsApp channel page, and the PR is docs-only and relevant, but it should be updated before merge to reflect the current Control UI/gateway/agent QR image data-url flow rather than implying terminal capture, saved PNG, or a standalone HTTP QR endpoint are the main/current paths.

## Best Possible Solution

Keep this PR open and ask for an update against current main: document that CLI WhatsApp login is still QR-based and terminal-oriented, recommend the current Control UI/gateway QR image flow for remote/headless pairing, avoid claiming phone-number or pairing-code login is available, avoid implying an unsupported standalone QR PNG/HTTP endpoint, and include any required docs i18n glossary entry for new short labels.
