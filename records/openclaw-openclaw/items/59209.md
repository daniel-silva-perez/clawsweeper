## Summary

Keep open. Current main has generic WSL2 remote CDP troubleshooting and generic browser doctor CDP HTTP/WebSocket checks, but it does not recognize or explain the reported Windows portproxy self-loop where svchost/iphlpsvc owns port 9222 and Chrome is not behind the endpoint. The cross-reference from #74163 is a maintainer tracker, not an implementation fix.

## What This Changes

Request OpenClaw to diagnose misleading WSL2 remote Chrome CDP empty responses caused by stale Windows netsh portproxy self-loop rules and guide users to verify the owning PID and CDP endpoint.

## Best Possible Solution

Add a browser diagnostics path and matching WSL2 remote CDP docs for this failure mode: when a WSL2/Windows remote CDP endpoint gives empty/unreachable /json/version behavior, surface a targeted hint to check netsh portproxy rules, netstat PID ownership, tasklist /svc for iphlpsvc, and Chrome process/CDP JSON health. Do not auto-delete Windows portproxy rules from OpenClaw.

## Reproduction Assessment

Yes for manual/field reproduction: the report gives exact Windows and WSL2 commands plus before/after evidence. I did not reproduce it live in this Linux runner, but a maintainer with Windows + WSL2 can recreate it by adding the 127.0.0.1:9222 -> 127.0.0.1:9222 portproxy self-loop and probing /json/version.

## Solution Assessment

Unclear as a full implementation because direct process-owner detection from WSL2 needs care around privileges and host command availability. The narrow maintainable solution is to add a targeted doctor/docs diagnostic and leave Windows portproxy cleanup as an explicit user-run recovery step.
