## Summary

Keep this PR open. Current main still has only the narrow cmd.exe wrapper guard and does not implement the proposed exported validator, null/control-character checks, or PowerShell-pattern guard. The PR is also not safe to merge as written because it rejects normal Windows paths, slash switches, equals-style flags, and echoes full argument values in errors.

## Best Possible Solution

Keep this PR open for maintainer review or replace it with a narrower Windows command-hardening PR. The fix should stay scoped to real cmd.exe wrapper hazards, keep path separators, scoped package specs, slash switches, and equals flags working, avoid applying PowerShell policy to non-PowerShell execution, redact argument values in validation errors, and add focused Windows regression tests for `.cmd`/`.bat` paths, npm fallback, equals flags, scoped package specs, and null/control-character handling.

## Risks / Open Questions

- Merging the PR as written would likely break ordinary Windows `.cmd`/`.bat` execution because the resolved command path is validated and normal Windows absolute paths contain backslashes.
- The expanded blacklist rejects legitimate slash switches, scoped package specs, and equals-style flags such as `--loglevel=error` and `--omit=optional`.
- The PR applies broad PowerShell-pattern policy on a cmd.exe wrapper path, which risks false positives for non-PowerShell execution.
- The PR's validation errors include full argument strings, which can leak credentials, tokens, or other sensitive argument values in exception messages and logs.
- Current main does not yet implement the PR's intended null-byte/control-character validation or exported validator, so useful hardening may still remain after a safer design is chosen.
