## Summary

Keep this PR open. Current main still has only the narrow cmd.exe wrapper guard and does not implement the proposed exported validator, null/control-character checks, or PowerShell-pattern guard. The PR also is not safe to merge as written because it rejects normal Windows paths, slash switches, equals-style flags, and echoes full argument values in errors.

## Best Possible Solution

Keep this PR open for maintainer/security review or replace it with a narrower Windows command-hardening PR. The fix should stay scoped to real cmd.exe wrapper hazards, keep path separators, scoped package specs, slash switches, and equals flags working, avoid applying PowerShell policy outside PowerShell execution, redact argument values in validation errors, and add focused Windows regression tests.

## Risks / Open Questions

- The PR as written would reject backslashes, so ordinary Windows absolute `.cmd`/`.bat` paths can throw before execution.
- Rejecting `/` and `=` would break legitimate Windows switches, scoped npm package specs, and OpenClaw npm update flags such as `--loglevel=error` and `--omit=optional`.
- Throwing errors with `JSON.stringify(arg)` can expose secrets or tokens embedded in argv values to logs and stack traces.
- Applying broad PowerShell cmdlet-name policy inside the cmd.exe wrapper path can create false positives without proving PowerShell execution is involved.
- The PR changes a command execution security surface but does not add focused regression tests for Windows path, flag, scoped package, and redaction behavior.
