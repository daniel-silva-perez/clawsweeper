## Summary

Current main still lacks the requested missed-job catch-up opt-out or freshness policy. Startup still plans missed jobs for replay or deferred replay, tests assert that behavior, and the public cron config/protocol/CLI/docs surfaces do not expose a global, per-job, or max-age/window control; related #42039 overlaps but does not fully decide the policy semantics.

## What This Changes

The issue requests a global, per-cron, or time-window-aware option to skip stale missed cron jobs when the Gateway restarts.

## Best Possible Solution

Keep the default replay behavior for backward compatibility, then add an explicit cron-owned missed-run policy that can skip stale catch-up safely: at minimum a global opt-out or max-age window, with a per-job override if maintainers want time-sensitive jobs to opt out individually. Any exposed maxMissedJobsPerRestart=0 behavior should be documented unambiguously, and the implementation should be coordinated with #42039 rather than treating that issue as a full substitute.

## Reproduction Assessment

Yes. A high-confidence reproduction exists from current tests and source: persist a cron job with a due past nextRunAtMs or a missed previous cron slot, start CronService, and current main enqueues/replays it or reschedules deferred overflow rather than skipping stale runs.

## Solution Assessment

Unclear. The issue identifies the right user problem, but the best public API is not settled; the safer maintainable path is a backward-compatible explicit policy rather than changing startup replay defaults or relying on maxMissedJobsPerRestart=0, which currently means defer rather than skip.
