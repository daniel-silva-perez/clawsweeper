## Summary

Keep open. Current main still replays or defers missed cron jobs on Gateway startup, and the public config, protocol, CLI, tool, and docs surfaces do not expose the requested opt-out or freshness-window policy; related #42039 overlaps but does not settle the skip semantics requested here.

## What This Changes

Request to add a global or per-cron policy that can skip or time-bound missed cron job catch-up after a Gateway restart.

## Best Possible Solution

Keep the existing catch-up behavior as the backward-compatible default, then add an explicit cron-owned missed-run policy that can skip stale catch-up safely. The policy should be reflected consistently in config/schema/help, protocol/job validation, CLI/tool surfaces, tests, and docs, and should be coordinated with #42039 so exposed maxMissedJobsPerRestart behavior is not confused with a skip policy.

## Reproduction Assessment

Yes. A high-confidence reproduction exists from current source and tests: persist a cron job with a due past nextRunAtMs or a missed previous cron slot with lastRunAtMs, start CronService, and current main enqueues the missed run or reschedules deferred overflow rather than skipping it.

## Solution Assessment

Unclear. The issue identifies the right user problem, but the best public API is not settled; the safer maintainable path is a backward-compatible explicit missed-run policy rather than changing startup replay defaults or relying on maxMissedJobsPerRestart=0, which currently means defer rather than skip.
