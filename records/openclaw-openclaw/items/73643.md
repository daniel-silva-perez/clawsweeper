## Summary

Keep open. Current main still has startup missed-job replay/defer behavior with no global opt-out, per-job skip flag, or freshness window, and the overlapping #42039 does not by itself settle the policy semantics needed for stale time-sensitive jobs.

## What This Changes

Request to add a global or per-cron missed-run policy so gateway restarts can skip stale cron catch-up jobs instead of replaying them after downtime.

## Best Possible Solution

Keep default replay behavior for backward compatibility, then add an explicit cron-owned missed-run policy that can skip stale catch-up safely, ideally with a global default plus a per-job or max-age override if maintainers want time-sensitive jobs to opt out individually. The config, protocol, CLI, docs, and restart catch-up tests should all describe the same semantics and be coordinated with #42039.

## Reproduction Assessment

Yes. A high-confidence reproduction exists from current source and tests: persist an overdue recurring cron job or a missed cron slot, start CronService, and current main enqueues/replays it or reschedules deferred overflow instead of skipping stale runs.

## Solution Assessment

Unclear. The request identifies the right user problem, but the best public API is not settled; the safer path is a backward-compatible explicit skip/window policy rather than changing startup defaults or assuming maxMissedJobsPerRestart=0 means skip.
