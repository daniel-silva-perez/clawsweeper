## Summary

Keep open. Current main still replays or defers missed cron work after Gateway restart and has no public global, per-job, or freshness-window policy to skip stale catch-up runs; related #42039 overlaps but does not fully settle the requested skip/window semantics.

## What This Changes

The issue requests a global or per-cron option, possibly with a freshness window, to prevent stale missed cron jobs from running after Gateway restart.

## Best Possible Solution

Keep the default replay behavior for compatibility, then add an explicit cron-owned missed-run policy that can skip stale catch-up safely. A complete implementation should update config, protocol, CLI, docs, generated schema/help surfaces, and restart catch-up tests, while coordinating with #42039 and the partial PRs already touching this area.

## Reproduction Assessment

Yes. A high-confidence reproduction exists from current source and tests: persist a recurring cron job with a past nextRunAtMs or a missed previous cron slot, start CronService, and current main enqueues/replays it or reschedules it instead of skipping stale runs.

## Solution Assessment

Unclear. The issue identifies the right user problem, but the best API is not settled; a backward-compatible explicit skip/freshness policy is safer than changing startup defaults or treating exposed stagger/limit knobs alone as a complete fix.
