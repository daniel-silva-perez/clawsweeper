## Summary

Keep open. Current main still replays or defers missed cron work after Gateway restart and has no public global, per-job, or freshness-window policy to skip stale catch-up runs; related #42039 overlaps but is narrower, and #41332/#42069 do not supersede this request.

## What This Changes

The issue requests a global, per-cron, or time-window policy to disable stale missed cron job catch-up after Gateway restart.

## Best Possible Solution

Add an explicit, backward-compatible cron missed-run policy that defaults to today’s replay/defer behavior but can skip stale catch-up safely. A complete fix should update config types, strict schema, generated schema/help metadata, gateway wiring, protocol/CLI if per-job policy is chosen, docs, changelog, and restart catch-up regression tests while coordinating with #42039.

## Reproduction Assessment

Yes. Source and existing tests provide a high-confidence reproduction: persist a recurring cron job with a past nextRunAtMs or a missed previous cron slot, start CronService, and current main enqueues/replays it or reschedules it rather than skipping stale runs.

## Solution Assessment

Unclear. The issue identifies a valid user problem, but the best API is not settled; an explicit skip/freshness policy is safer than treating exposed stagger/limit knobs alone as a complete solution.
