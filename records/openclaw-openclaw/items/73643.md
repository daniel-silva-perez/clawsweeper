## Summary

Current main still runs missed cron jobs during Gateway startup and still lacks a global, per-job, or max-age policy to skip stale missed runs. There is a partial hardcoded stale-delivery guard for isolated cron fallback delivery, but the job itself still executes, so it does not satisfy the requested catch-up disable option. This issue is recent, unprotected, actionable in the core cron scheduler/config/protocol surfaces, and the same-author #73644 is an open duplicate rather than a merged canonical fix.

## Best Possible Solution

Keep this issue open. Implement a backward-compatible missed-run policy in the cron-owned surface: a global default under cron, ideally with optional per-job override and/or max-age window, wired through CronConfig, strict schema/help, cron protocol/job types, CLI add/edit or config docs as appropriate, and runMissedJobs planning. Add focused tests for default replay, global skip, per-job override, and stale-window behavior; dedupe #73644 manually after choosing the canonical tracker.

## Risks / Open Questions

- Changing the default catch-up behavior could cause users to miss important recovery/reminder jobs after short outages, so any implementation should preserve current replay behavior unless explicitly configured otherwise.
- The hardcoded stale-delivery suppression may reduce some notification noise, but it is not a substitute for a scheduler-level missed-run policy because job execution and agent/message side effects still happen.
