## Summary

Keep open: current main still replays or reschedules missed cron work after Gateway restart and has no public global, per-job, or freshness-window policy to skip stale catch-up runs. The existing related work reduces startup burst pressure but does not solve the reported after-hours notification problem.

## What This Changes

Add a global, per-cron, or freshness-window option to disable or limit missed cron-job catch-up after Gateway restart.

## Best Possible Solution

Keep replay as the backward-compatible default, then add an explicit cron-owned missed-run policy that can skip stale catch-up safely. A complete solution should update CronConfig, strict/generated config schema, Gateway cron protocol, CLI/help/docs, and restart catch-up tests, while coordinating with #42039 so the exposed knobs do not imply that deferral is the same as skipping.

## Reproduction Assessment

Yes. Current source and tests give a high-confidence reproduction: persist a recurring cron job whose nextRunAtMs is in the past, start CronService, and current main executes it immediately or reschedules it for later rather than skipping it as stale.

## Solution Assessment

Unclear. The issue identifies a valid product problem, but the best API is not settled; a backward-compatible explicit skip/freshness policy is safer than changing defaults or exposing only stagger/limit knobs, because the current maxMissedJobsPerRestart path still defers missed jobs instead of disabling catch-up.
