## Summary

Keep this draft PR open. Current main still does not implement the intended TUI `awaiting follow-up` busy state after `sessions_yield`: the renderer only treats existing run states as busy, final events still settle non-error runs to `idle`, lifecycle handling ignores the typed `yielded`/`livenessState: "paused"` metadata, and history restore does not preserve or clear a follow-up wait state. The PR remains a focused implementation candidate, with a security review showing no apparent supply-chain or secrets-handling regression in the proposed file set.

## Best Possible Solution

Keep this draft PR open for maintainer review or revision. The best implementation is a focused TUI change that adds `awaiting follow-up` as a busy activity state, derives it from stable typed/shared `sessions_yield` metadata where available, restores and clears it during history/session changes, and includes targeted live-event plus history-restore tests.
