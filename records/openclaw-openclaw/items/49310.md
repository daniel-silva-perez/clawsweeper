## Summary

Keep this draft PR open. Current main still does not implement the PR's intended `awaiting follow-up` TUI busy state for `sessions_yield` waits: the renderer only treats `sending`, `waiting`, `streaming`, and `running` as busy; final chat events still settle non-error runs to `idle`; lifecycle events ignore the existing typed `yielded` / `livenessState: "paused"` metadata; and history restore does not restore or clear a follow-up wait state. The PR remains a focused implementation candidate, with a design concern around matching a duplicated literal marker instead of the existing typed/shared yield signal.

## Best Possible Solution

Keep this draft PR open for maintainer review or revision. The best path is a focused TUI change that adds an `awaiting follow-up` busy activity, derives it from a stable typed/shared `sessions_yield` signal where possible, restores and clears it during history/session changes, and includes targeted live-event plus history-restore tests.
