## Summary

Keep this PR open. Current main still does not implement the intended TUI busy state for sessions_yield follow-up waits: final chat events settle to idle/error, lifecycle end events ignore the existing yielded metadata, the status renderer has no awaiting-follow-up busy state, and history reload does not restore or clear that state from persisted sessions_yield context. The supplied PR remains a plausible focused TUI fix, though the marker-string approach should be reviewed against the existing typed yielded/liveness signal.

## Best Possible Solution

Keep this PR open for maintainer review or revision. The best fix is a focused TUI change that renders a sessions_yield follow-up wait as a busy state, derives it from a stable typed signal or shared marker, restores and clears it correctly during history reload/session changes, and covers both live event handling and history restore with targeted TUI tests.

## Risks / Open Questions

- The proposed string-marker detection is brittle if the sessions_yield context text changes; a shared constant or the existing typed yielded/liveness metadata would be more maintainable.
- The final implementation should explicitly clear the follow-up busy state on continuation, abort/error, session switch, and history without a valid pending yield so the spinner cannot become stale.
