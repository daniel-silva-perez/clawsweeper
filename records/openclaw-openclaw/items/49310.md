## Summary

Keep this draft PR open. Current main at 6d323ee73640afdf02ed9fa3aded13da23359b98 still does not implement an `awaiting follow-up` TUI busy state for `sessions_yield` waits, and the PR remains a focused implementation candidate rather than obsolete work. The patch is limited to TUI source/tests and does not appear to introduce a security or supply-chain-sensitive change, though the literal marker detection should likely be replaced with a shared helper or typed lifecycle signal.

## Best Possible Solution

Keep this PR open for maintainer review or revision. The best path is a focused TUI change that renders `sessions_yield` follow-up waits as a busy activity, derives that state from a stable typed/shared signal where possible, restores and clears it during history/session changes, and includes targeted live-event plus history-restore tests.
