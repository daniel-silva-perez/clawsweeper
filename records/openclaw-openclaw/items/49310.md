## Summary

Keep this draft PR open. Current main still does not implement the intended TUI busy state for sessions_yield follow-up waits: the status renderer has no awaiting-follow-up busy activity, final chat events still settle to idle/error only, lifecycle yielded metadata is ignored by the TUI, and history reload does not restore or clear a follow-up wait state. The PR is a focused TUI/test change and does not appear to introduce a security or supply-chain-sensitive surface, but the implementation should avoid duplicating the runner’s literal context marker if a shared typed signal is available.

## Best Possible Solution

Keep this PR open for maintainer review or revision. The best path is a focused TUI change that renders a sessions_yield follow-up wait as a busy activity, derives it from a stable typed/shared signal where possible, restores and clears it during history/session changes, and adds targeted live-event plus history-restore tests.
