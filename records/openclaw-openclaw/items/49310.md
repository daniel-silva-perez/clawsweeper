## Summary

Keep this PR open. Current main still does not implement the intended TUI busy state for `sessions_yield` follow-up waits: the status renderer has no `awaiting follow-up` busy state, chat finals still settle to `idle` or `error`, lifecycle `yielded` metadata is ignored, and history reload does not restore or clear a follow-up wait state.

## Best Possible Solution

Keep this PR open for maintainer review or revision. The best path is a focused TUI change that renders a `sessions_yield` follow-up wait as a busy state, derives it from a stable typed signal or shared marker, restores and clears that state during history reload/session changes, and adds targeted tests for live event handling plus history restore.

## Risks / Open Questions

- The proposed marker-string detection is brittle if the runner text changes; a shared constant or the existing typed `yielded`/`livenessState` metadata would be safer.
- The final fix should explicitly clear the follow-up busy state on continuation, abort/error, session switch, and history loads without a pending yield so the spinner cannot become stale.
