## Summary

Keep this draft PR open. Current main at 14f140d6f0d31fed802869622c38c7e8f8e19587 still does not implement an `awaiting follow-up` TUI busy state for `sessions_yield` waits, so the PR is not obsolete. The previous ClawSweeper comment’s conclusion still holds on the newer main SHA: this remains a focused TUI implementation candidate, with a small design concern around using a duplicated literal marker instead of the existing typed yield/liveness signal.

## Best Possible Solution

Keep this draft PR open for maintainer review or revision. The best path is a focused TUI change that adds an `awaiting follow-up` busy activity, derives it from a stable typed/shared `sessions_yield` signal where possible, restores and clears it during history/session changes, and includes targeted live-event plus history-restore tests.

## Risks / Open Questions

- The submitted approach duplicates and matches a literal `sessions_yield` context marker; the final version should prefer a shared helper/constant or the existing typed `yielded` plus `livenessState: "paused"` lifecycle signal to avoid drift.
- The final implementation should explicitly clear `awaiting follow-up` on continuation, abort/error, session switch, and history loads without a pending yield so the spinner cannot become stale.
