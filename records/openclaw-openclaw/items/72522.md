## Summary

Keep open. Current main still renders the credentials gate on every disconnected Control UI state, so the PR is addressing a real linked bug, but the branch is not merge-ready because explicit auth/session revocation closes in current server code can arrive as code/reason-only frames while the new sticky flag clears only when a structured error detail code is present.

## What This Changes

The PR adds a sticky Control UI `hasEverConnected` state, changes login-gate rendering, updates gateway close handling and tests, and adds a changelog entry to avoid credentials-gate flashes during transient WebSocket reconnects.

## Best Possible Solution

Keep the sticky-state direction, but make auth/session revocation closes clear the flag immediately by either mapping known auth close codes/reasons such as `4001 gateway auth changed` and `4001 device removed` in the UI or propagating structured close details from the gateway client, with targeted tests for those no-error close frames.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes: the linked issue gives UI steps, and current main source shows every WebSocket close sets `connected=false` while `renderApp` gates solely on `!connected`. The PR finding is also reproducible with a unit-level close event carrying `code: 4001`, `reason: "gateway auth changed"`, and no `error` payload.

## Solution Assessment

Is this the best way to solve the issue? No, not as written: a sticky prior-hello flag is a maintainable fix for transient disconnect flashes, but it must also handle code/reason-only auth revocation closes or the promised re-auth fallback is incomplete.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Clear sticky state on auth revocation closes:** `ui/src/ui/app-gateway.ts:408-409`
  - body: The server already closes authenticated Control UI sockets with `4001, "gateway auth changed"` and `4001, "device removed"` without a structured `error` payload, while `GatewayBrowserClient` only passes `pendingConnectError` as `error`. Because this branch clears `hasEverConnected` only when `host.lastErrorCode` is set, those explicit auth/session revocation closes keep the cached chat UI mounted until a later reconnect failure. Please clear the sticky flag for these close codes/reasons or propagate structured close details, and add a no-error close test.
  - confidence: 0.86
