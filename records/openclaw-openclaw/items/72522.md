## Summary

Keep open. The PR targets a real Control UI reconnect bug and addressed the Greptile prefix-review concern, but it is not merge-ready because explicit auth/session revocation closes can arrive as WebSocket code/reason-only frames and leave the new sticky mounted-state flag set.

## What This Changes

The PR adds a sticky Control UI `hasEverConnected` state across gateway, render, view-state, Lit state, tests, and changelog so transient WebSocket closes keep webchat mounted while structured auth/pairing failures restore the credentials gate.

## Best Possible Solution

Keep the sticky prior-session direction, but make explicit auth/session revocation closes clear the sticky state immediately by mapping known `4001` auth/device close reasons or by propagating structured close details from the gateway client, with focused no-error close tests.

## Reproduction Assessment

Yes. The linked bug has concrete tab-refocus steps, and current main shows every close flips `connected=false` while rendering the gate for any disconnected state; the PR-specific defect is reproducible with a successful hello followed by `onClose({ code: 4001, reason: "gateway auth changed" })` and no `error`.

## Solution Assessment

No, not as written. A sticky prior-hello flag is a maintainable way to avoid transient reconnect flashes, but the implementation must also handle code/reason-only auth revocation closes before it is the narrow safe fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Clear sticky state on auth revocation closes:** `ui/src/ui/app-gateway.ts:408-409`
  - body: After a successful hello, `GatewayBrowserClient` calls `onClose` with only `{code, reason, error: undefined}` for normal server close frames. The server uses `4001, "gateway auth changed"` and `4001, "device removed"` for explicit shared-auth/device revocation, so this new branch never clears `hasEverConnected` and `renderApp` keeps cached Control UI content mounted. Please treat those close code/reason pairs as auth failures or propagate structured detail into `onClose`, and cover the no-error close case.
