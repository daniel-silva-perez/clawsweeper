## Summary

Keep this PR open. Current main still has the original Control UI polling intervals, so the intended change has not landed. The PR remains potentially useful, but it should be revised before merge because the logs polling increase can lose live-tail data under the current cursor/maxBytes behavior, and the broader dashboard sluggishness work is still covered by related open threads.

## Best Possible Solution

Revise rather than close: keep the useful interval extraction and consider a safer nodes/debug reduction or tab-scoped nodes polling, but leave Logs at 2s unless the PR also increases/configures the tail window or moves logs to a lossless push/streaming path. Add focused polling interval tests plus a regression test for logs.tail cursor/maxBytes truncation, and coordinate the final approach with #45698 and #69454.
