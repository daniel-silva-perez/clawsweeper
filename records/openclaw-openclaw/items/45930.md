## Summary

Keep open. Current main has not implemented this PR, and the proposed logs polling slowdown is a likely functional regression for live log tailing because the UI still fetches only a bounded 500 lines / 250 KB per request while the server resets to the newest chunk when the cursor gap exceeds that byte budget.

## What This Changes

The PR extracts Control UI polling intervals in `ui/src/ui/app-polling.ts` and changes nodes/logs/debug polling from 5s/2s/3s to 30s/5s/10s.

## Best Possible Solution

Reduce unnecessary Control UI polling without weakening Logs tab live-tail semantics: preserve the short logs cadence or pair any slower cadence with a larger/safer tail budget, and handle nodes/debug churn through tab-scoped polling or immediate refresh behavior in the broader Control UI fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Restore the shorter logs polling cadence:** `ui/src/ui/app-polling.ts:19`
  - body: Changing `LOGS_POLL_MS` to 5 seconds can make the Logs tab skip data on busy gateways. The UI still requests only 500 lines / 250 KB per `logs.tail` call, and the server resets to the newest `maxBytes` chunk when the cursor gap exceeds that budget, so a gateway emitting more than 250 KB between polls loses the intervening log lines instead of merely updating less often.
  - confidence: 0.88
