## Summary

Keep this PR open, but it needs changes before merge. Current main still uses the existing 5s/2s/3s polling cadences, and the proposed 5s logs cadence is a functional regression because the live log tail still fetches a bounded 500-line/250 KB window and resets to the newest chunk when the cursor gap exceeds that byte budget.

## What This Changes

The PR extracts Control UI polling intervals in `ui/src/ui/app-polling.ts` and changes nodes/logs/debug polling from 5s/2s/3s to 30s/5s/10s.

## Best Possible Solution

Reduce unnecessary Control UI churn without weakening live-tail semantics: keep or otherwise protect the logs fetch cadence/budget, and prefer tab-scoped or immediate-refresh behavior for slow-moving nodes/debug data in coordination with #69454.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Restore the shorter logs polling cadence:** `ui/src/ui/app-polling.ts:19`
  - body: Changing `LOGS_POLL_MS` to 5 seconds can make the Logs tab skip data on busy gateways. The UI still requests only 500 lines / 250 KB per `logs.tail` call, and the server resets to the newest `maxBytes` chunk when the cursor gap exceeds that budget, so a gateway emitting more than 250 KB between polls loses the intervening log lines instead of merely updating less often.
  - confidence: 0.88
