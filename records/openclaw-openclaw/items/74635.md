## Summary

Keep open. Current main still rejects root-level `paperclip` metadata on Gateway `agent` requests, and an open PR specifically references this issue with closing syntax, so the issue should stay open until a canonical fix merges or that PR is closed.

## What This Changes

Paperclip's `openclaw_gateway` adapter needs Gateway agent invocation params to accept root-level `paperclip` metadata or move that context into a documented supported metadata field.

## Best Possible Solution

Land one additive Gateway protocol compatibility change in a canonical PR, preserving strict rejection of unrelated unknown fields, adding focused regression coverage, regenerating protocol artifacts, and documenting/changelogging the wire-contract change if maintainers accept the new field.

## Reproduction Assessment

Yes. A minimal Gateway `agent` request with `message`, `idempotencyKey`, and a root-level `paperclip` object follows the report and will fail current `validateAgentParams` because the schema is closed and omits that field.

## Solution Assessment

Unclear. The requested `paperclip: Type.Optional(Type.Unknown())` shim is the narrowest compatibility fix, but maintainers should decide whether a generic adapter metadata namespace is the better long-term protocol surface before merging one of the duplicate PRs.
