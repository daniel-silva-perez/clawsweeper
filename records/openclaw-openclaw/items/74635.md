## Summary

Keep this issue open. Current main still rejects the reported root-level `paperclip` field, and an open target PR explicitly closes this issue, so the issue should remain open until maintainers land or reject the implementation path.

## What This Changes

Paperclip's `openclaw_gateway` adapter sends a root-level `paperclip` field in Gateway `agent` params, and the issue asks OpenClaw to accept that metadata or route it through a supported field.

## Best Possible Solution

Land one additive Gateway protocol compatibility change for third-party adapter metadata in `agent` params, preferably a documented generic namespace unless maintainers explicitly approve the Paperclip-specific shim, then add regression coverage and close the linked Paperclip reports after merge.

## Reproduction Assessment

Yes. A minimal Gateway `agent` request with valid `message`, `sessionKey`, `idempotencyKey`, and a root-level `paperclip` object follows the current source path into `validateAgentParams` and fails closed-schema validation before runtime dispatch.

## Solution Assessment

Unclear. Accepting `paperclip: Type.Optional(Type.Unknown())` would unblock the reported integration, but a generic `adapterMeta`/`pluginParams`-style field is likely the more maintainable public protocol shape unless a maintainer approves the vendor-named compatibility field.
