## Summary

Keep open. Current `main` still rejects root-level `paperclip` metadata on Gateway `agent` requests, and open PR #74974 is already the linked implementation candidate, so this issue should remain open until that PR, or a maintainer-approved alternative, lands.

## What This Changes

The issue reports that Paperclip `openclaw_gateway` agent invocations fail because the Gateway agent params validator rejects a root-level `paperclip` metadata field.

## Best Possible Solution

Land one maintainer-approved, additive Gateway protocol compatibility change through the canonical PR path, preserving strict rejection of unrelated unknown fields while adding focused regression coverage and generated protocol artifact updates.

## Reproduction Assessment

Yes. A minimal Gateway `agent` request with valid `message` and `idempotencyKey` plus a root-level `paperclip` property follows the report and will fail the current closed `AgentParamsSchema` before dispatch.

## Solution Assessment

Unclear. Adding `paperclip: Type.Optional(Type.Unknown())` is the narrowest compatibility shim and matches the reporter workaround, but maintainers should decide whether a generic opaque adapter metadata field is the better long-term protocol surface.
