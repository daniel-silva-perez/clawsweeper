## Summary

Keep open. Current main still rejects root-level `paperclip` metadata on Gateway `agent` requests, and open PR #74974 explicitly targets this issue with closing syntax, so the issue should remain open until that PR is reviewed, merged, or replaced.

## What This Changes

The issue reports that Paperclip’s `openclaw_gateway` adapter sends a root-level `paperclip` field in agent invocation payloads, which current Gateway validation rejects as an unexpected property.

## Best Possible Solution

Review the linked implementation PR and land one additive Gateway protocol compatibility fix with regression coverage. The most maintainable end state is a documented opaque metadata contract for `agent` params that keeps strict rejection for unrelated unknown fields, with generated client artifacts and docs updated if the public protocol changes.

## Reproduction Assessment

Yes. A minimal Gateway `agent` request containing valid `message` and `idempotencyKey` plus a root-level `paperclip` property follows the current validation path and fails because the schema is closed and the handler rejects before dispatch.

## Solution Assessment

Yes for keeping this issue open. The current linked PR is a plausible narrow compatibility repair, but the final implementation should be reviewed for protocol shape because a generic metadata namespace may be safer than a vendor-named root field.
