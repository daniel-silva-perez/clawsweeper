## Summary

Keep this PR open. It addresses a real current-main prompt-noise problem for Telegram/direct inbound metadata, but the proposed compact path appears to drop the existing sender-label backfill used by display/history projection, and the branch also bundles an unrelated plugin CLI memory-slot change that should get separate maintainer attention before merge.

## What This Changes

The PR adds optional `messages.inboundMetadataMode: "compact-direct"` prompt rendering with config/docs/schema/test updates, compact metadata stripping, and a plugin CLI memory-slot loading adjustment for `wiki` and `memory` primary commands.

## Best Possible Solution

Revise the PR so compact direct prompts keep the model-visible metadata terse while preserving the existing display/history sender-label contract, ideally with a `stripEnvelopeFromMessage` regression test for compact direct content. Split or clearly justify the plugin CLI memory-slot change, then rerun focused auto-reply, gateway sanitizer, config schema/docs, and plugin CLI validation before normal maintainer review.

## Reproduction Assessment

Yes. Current main can be inspected through `buildInboundUserContextPrefix` for a direct Telegram-style context, and the PR diff shows the compact path suppressing the `Sender` block while the display sanitizer still depends on extracting sender labels from that metadata when `senderLabel` is absent.

## Solution Assessment

No, not as-is. The compact prompt direction is a reasonable narrow solution for the linked issues, but the patch should preserve sender attribution outside the model-visible prompt or teach the sanitizer a compact sender-label path, and the unrelated CLI loader change should be split or reviewed separately.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve sender labels in compact history:** `src/auto-reply/reply/inbound-meta.ts:288`
  - body: Compact mode skips the `Sender` block and only writes channel/chat/message/sender IDs. `stripEnvelopeFromMessage` currently backfills `senderLabel` by parsing the existing sender/conversation metadata before stripping it, so compact direct messages without an explicit `senderLabel` lose the display attribution that dashboard grouping relies on. Please preserve that sender label through a non-model-visible field or add compact extraction plus a gateway sanitizer test.
