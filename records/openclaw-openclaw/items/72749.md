## Summary

Keep this PR open. Current main still has the reported full inline metadata behavior, and the PR is a relevant implementation candidate for the linked Telegram metadata-bloat reports, but the proposed compact path drops the sender-name metadata that current display/history sanitization uses to preserve sender attribution. The branch also mixes in a separate plugin CLI memory-slot loader change that needs splitting or explicit maintainer review.

## What This Changes

The PR adds optional `messages.inboundMetadataMode: "compact-direct"` prompt rendering with config/schema/docs/tests, compact metadata stripping, and a plugin CLI loader adjustment that includes the configured memory slot plugin for `wiki` and `memory` primary commands.

## Best Possible Solution

Revise the PR so compact direct prompts stay terse for the model while preserving the existing user-visible sender-label contract, preferably with a `stripEnvelopeFromMessage` regression test for compact direct content. The memory-slot CLI loader change should be split into its own PR or explicitly justified and reviewed as a separate behavior change.

## Reproduction Assessment

Yes. A focused reproduction is to build compact direct metadata for a Telegram-style direct message, wrap it in a user message without an explicit `senderLabel`, then pass it through `stripEnvelopeFromMessage`; the current sanitizer only recovers labels from full Sender/Conversation JSON blocks.

## Solution Assessment

No, not as-is. The compact metadata direction is a reasonable narrow fix for #72704 and #62077, but the implementation should preserve sender attribution outside the model-visible prompt or teach the sanitizer to extract the compact sender label path, and it should avoid bundling unrelated plugin CLI behavior.
