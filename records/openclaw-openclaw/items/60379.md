## Summary

Keep open. Current main still has sticky session-level model switching for chat: `/model` persists a selected provider/model into session state and future turns/queued work inherit it. There are adjacent one-shot surfaces (`openclaw agent --model`, cron `--model`, `/btw` side questions), but none provides the requested normal chat/message syntax that runs exactly one response on a different text model and then restores the session default.

## Best Possible Solution

Add a first-class per-turn model override for normal chat/message runs, likely by reusing the existing model resolver and allowlist/auth validation but carrying the chosen provider/model on the current followup run instead of writing `providerOverride`/`modelOverride` to `SessionEntry`. Define syntax such as `/model <ref> --once <message>` or `/ask <ref> <message>`, document it in the model/slash-command docs, and add tests proving the next user message, queued followups, heartbeats, and cron runs keep the prior session selection.

## Risks / Open Questions

- Implementing this as a parser-only alias would be risky: current unrecognized flags after `/model <ref>` would become ordinary prompt text unless the parser and cleaned-message flow gain explicit one-shot handling.
- Queue, heartbeat, cron, and live-switch semantics need a clear rule so the one-shot model applies only to the intended user response and never retargets queued or background work.
