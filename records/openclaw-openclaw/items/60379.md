## Summary

Keep open. Current main still implements normal chat model changes as persistent session state: `/model` is the only normal chat model directive, it writes `providerOverride`/`modelOverride` into the session entry, and queued/future work is explicitly retargeted. Adjacent one-shot surfaces such as cron `--model` and `/btw` do not satisfy the requested single normal chat response on another model while preserving the session default.

## What This Changes

Request a one-shot provider/model override for a single normal chat response without changing the session default or retargeting queued/background work.

## Best Possible Solution

Add a first-class per-turn model override for normal chat runs that reuses existing model resolution, allowlist, and auth-profile validation, then carries the selected provider/model only on the current reply run. It should leave `SessionEntry` model override fields untouched, avoid refreshing queued followups, keep heartbeat and cron resolution unchanged, and document the chosen syntax in the model and slash-command docs.

## Reproduction Assessment

Yes. A source-level reproduction path is clear: parse a normal `/model <provider/model> ...` message and follow directive handling through `applyModelOverrideToSessionEntry`; current tests already assert the resulting session override and queued followup retargeting.

## Solution Assessment

Yes for the requested direction, but no current implementation exists. The maintainable fix is not a parser-only alias; it needs an explicit per-turn override path that is validated like `/model` but bypasses session persistence and queue retargeting.

## Repair Work Prompt

Implement per-message model override for normal chat for https://github.com/openclaw/openclaw/issues/60379. Add a first-class one-turn syntax, preferably `/model <provider/model-or-alias> --once <message>` if it fits the existing directive parser; `/ask <provider/model-or-alias> <message>` is acceptable only if command registration and docs are updated consistently. Reuse the existing `/model` resolver, allowlist, alias, auth-profile, and error behavior. The override must apply only to the current normal reply run, must not write `providerOverride`, `modelOverride`, `modelOverrideSource`, or auth-profile override fields to `SessionEntry`, must not call `refreshQueuedFollowupSession`, and must not alter heartbeat or cron model resolution. Preserve existing persistent `/model` behavior. Add focused tests proving: the one-shot response resolves the requested model; the next user message uses the prior/default session model; queued followups are not retargeted; heartbeat/cron paths keep their existing precedence; unrecognized or misplaced `--once` does not silently become prompt text in a dangerous way. Update `docs/concepts/models.md`, `docs/tools/slash-commands.md`, and any chat/TUI docs that list slash commands. Add a user-facing changelog entry in the active version with `Thanks @Nomlastew`.
