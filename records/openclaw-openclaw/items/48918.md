## Summary

Keep this issue open. Current main still lacks a supported skill-scoped user preferences/conventions mechanism: skill loading, discovery, prompt exposure, docs, and watcher behavior are all centered on `SKILL.md`, while `skills.entries.*.config` is only a structured config bag and is not rendered as natural-language skill context. The issue discussion remains actionable and specifically validates the markdown `preferences.md` approach.

## Best Possible Solution

Keep this issue open and design a small core skill-loading contract for skill-scoped user preferences. The likely path is a markdown companion file such as `preferences.md` or `conventions.md` under existing skill roots, loaded or surfaced only when the matching skill is used, with tests and docs for precedence, token limits, watcher refresh behavior, sandbox/Claude plugin materialization, and interaction with existing `skills.entries.*` config. `SKILL.md` should remain authoritative for skill instructions and safety requirements, with user preferences appended as lower-precedence local context.

## Risks / Open Questions

- A future implementation needs a clear precedence rule so user preferences can add local context without silently overriding safety-critical `SKILL.md` instructions.
- The watcher, prompt snapshot, and Claude plugin materialization paths all need to understand the same companion-file contract; otherwise preferences could be stale or missing in some runners.
- If maintainers choose structured `skills.entries.*.config` instead of markdown files, they still need a rendering/token-budget contract because the current config bag alone does not satisfy the reported natural-language preference use case.
