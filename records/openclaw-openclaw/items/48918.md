## Summary

Keep this issue open. Current main still lacks a supported skill-scoped user preferences/conventions mechanism: skill discovery and prompt construction are centered on `SKILL.md`, existing `skills.entries.*.config` is not rendered as skill context, and the watcher only treats `SKILL.md` changes as skill-refresh events. The issue is recent, concrete, non-maintainer-authored, unprotected by labels, and the comments add useful validation for the markdown `preferences.md` approach rather than making it obsolete.

## Best Possible Solution

Keep this issue open and design a small core skill-loading contract for skill-scoped user preferences. The likely path is a markdown companion file such as `preferences.md` or `conventions.md` under existing skill roots, loaded or surfaced only when the matching skill is used, with tests and docs for precedence, token limits, refresh watching, and Claude plugin materialization behavior. `SKILL.md` safety/instruction content should remain authoritative, with user preferences appended as lower-precedence local context.

## Risks / Open Questions

- A future implementation needs a clear precedence rule so user-authored preferences can add local context without overriding safety-critical `SKILL.md` instructions silently.
- If the design uses structured `skills.entries.*.config` instead of markdown files, it still needs a prompt/rendering contract; the current config bag alone does not solve the reported token-efficient natural-language preference use case.
