## Summary

Keep this PR open. Current main has only a generic all-media-dropped warning and still lacks the PR's intended dropped-media pipeline: partial drops are silent, catastrophic normalization clears media without user-visible metadata, outbound JSON/delivery/transcript paths have no dropped-media carrier, and routing/followup filters still treat dropped-media-only payloads as empty. The related #70109 candidate is closed and unmerged, so it does not supersede this active PR.

## Best Possible Solution

Keep this PR open for maintainer review. The best outcome is to land this PR after fixing any current head issues and validating the touched lanes, or replace it with an equivalent maintainer-approved implementation that preserves sanitized per-item dropped-media details through normalization, catastrophic fallback, block/routing/followup filters, outbound JSON/delivery, hooks/audit behavior, and transcript mirroring.

## Risks / Open Questions

- The PR touches hot outbound delivery, routing, hook, transcript, and block-streaming paths; maintainers should require focused typecheck/test proof before landing.
- Security review pass: no workflow/dependency/supply-chain files are changed, but the PR introduces user-visible diagnostic text and warning logs derived from media sources. The sanitizer and code-span handling need source/test verification against path disclosure, presigned URL leakage, data URI leakage, Markdown breakout, and mention injection.
- The PR intentionally emits dropped-media notices outside the `message_sending` hook path after sanitization. That may be a valid system-diagnostic choice, but maintainers should explicitly approve the moderation/compliance tradeoff.
- The fetched PR diff still shows some test payloads using `reason` where the new type uses `code`, and `deliveredPayloadIndices` appears written but not read in `deliver.ts`; these are merge-readiness concerns, not reasons to close the PR as obsolete.
