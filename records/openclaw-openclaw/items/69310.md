## Summary

Keep this PR open. Current main only has a generic warning when all reply media is dropped, but it still lacks the PR's intended sanitized dropped-media pipeline: partial media drops remain invisible, catastrophic normalization still clears media without user-facing metadata, outbound JSON/delivery/transcript paths have no dropped-media carrier, and routing/followup/block filters still treat dropped-media-only payloads as empty. The related #70109 candidate is closed and unmerged, so it does not supersede this PR.

## Best Possible Solution

Keep this PR open for maintainer review. The best outcome is to land this PR after resolving current head issues and validating the touched lanes, or replace it with an equivalent maintainer-approved core implementation that preserves sanitized per-item dropped-media details through normalization, catastrophic fallback, block/routing/followup filters, outbound JSON/delivery, hooks/audit behavior, and transcript mirroring.

## Risks / Open Questions

- The PR touches hot outbound delivery, routing, block streaming, hooks, transcript mirroring, and followup paths; it needs focused typecheck/test proof before landing.
- Security review pass: no workflow/dependency/supply-chain files are changed, but the PR emits user-visible notices and warning logs derived from media sources. The sanitizer/code-span handling should be verified against path disclosure, presigned URL leakage, data URI leakage, Markdown breakout, and mention injection.
- The PR intentionally emits dropped-media notices outside the `message_sending` hook path after sanitization. That may be the right system-diagnostic behavior, but maintainers should explicitly approve the moderation/compliance tradeoff.
- The fetched PR diff still shows some tests constructing `droppedMedia` entries with `reason` while the new type uses `code`, and `deliveredPayloadIndices` appears written but not used in the final mirror logic. These are merge-readiness issues, not reasons to close the PR as obsolete.
