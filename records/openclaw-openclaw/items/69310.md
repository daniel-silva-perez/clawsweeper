## Summary

Keep open. Current main still has low-visibility media-loss paths, and this PR targets a valid user-facing gap, but the latest branch still has a concrete user-visible regression: all-failed media payloads keep the legacy generic warning while also emitting the new detailed dropped-media notice.

## What This Changes

The PR adds dropped-media diagnostics for failed MEDIA normalization, carries them through reply routing/outbound delivery/JSON projection/transcript mirroring, upgrades warning logs, and adds focused tests.

## Best Possible Solution

Emit exactly one canonical attachment-failure notice for both all-failed and mixed-failed media cases by removing the legacy generic warning when droppedMedia is present, then rerun the focused reply/outbound tests and changed gate before maintainer approval.

## Reproduction Assessment

Yes. On the latest PR head, a payload where every MEDIA source fails enters the normalizedMedia.length === 0 branch, appends the legacy generic warning, preserves droppedMedia, and later delivery appends formatDroppedMediaNotice from that same droppedMedia.

## Solution Assessment

No. The droppedMedia pipeline is the right general direction, but retaining the old all-failed warning is not the best user-facing implementation because it produces duplicate diagnostics instead of one clear notice.
