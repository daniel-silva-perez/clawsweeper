## Summary

Keep open. The PR addresses a real current-main gap, but the latest head still has one user-visible regression: all-failed media payloads can now emit both the legacy generic warning and the new specific dropped-media notice.

## What This Changes

The PR adds a `droppedMedia` payload pipeline for reply media normalization failures, formats user/transcript notices, preserves dropped-media-only payloads through routing/outbound projections, upgrades warning logs, and adds focused tests.

## Best Possible Solution

Keep the contributor PR active, remove the legacy generic all-failed media warning when `droppedMedia` is emitted, and add a focused test proving an all-failed `MEDIA:` payload produces one clear attachment-failure notice.

## Reproduction Assessment

Yes. The static reproduction path is an all-failed media payload: `reply-media-paths.ts` adds the legacy text warning and `droppedMedia`, then `deliver.ts` inlines `formatDroppedMediaNotice` because no media remains.

## Solution Assessment

No, not quite yet. The dropped-media pipeline is the right direction, but the all-failed path should converge on the new specific notice instead of stacking it on top of the older generic warning.
