## Summary

Keep open. Current main still builds plugin tools as a point-in-time tool array for the run/session, while plugin install persists records and refreshes the cold registry only; I found no active-session tool delta, turn-boundary re-resolution, or Pi session allowlist refresh path. The related GatewayRestart fan-out PR is adjacent restart recovery work, not a replacement for policy-filtered active-session plugin-tool refresh.

## What This Changes

Request that successful `openclaw plugins install` operations make newly registered plugin tools available to already-active agent sessions without requiring a Gateway restart.

## Best Possible Solution

Add a generic runtime lifecycle path that re-resolves plugin-provided tools after a successful plugin install or registry refresh, applies the same policy filters used at normal run setup, updates active sessions only at a safe turn boundary, and invalidates affected prompt/tool-cache state without duplicating plugin runtime loads.

## Reproduction Assessment

Yes. The issue provides a concrete install -> immediate use -> Gateway restart comparison, and current-main source inspection confirms plugin installs refresh the registry while active runs build and apply a fixed tool snapshot during session setup.

## Solution Assessment

Unclear as a patch because this issue has no implementation. The requested direction is valid, but the safest maintainable solution is likely turn-boundary re-resolution through existing policy filters rather than arbitrary mid-stream mutation of active session state.
