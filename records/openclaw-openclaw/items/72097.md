## Summary

Keep open. Current main still refreshes plugin registry metadata after install, but active agent tool construction remains a one-time per-session/per-run array passed into the embedded session, and docs still describe Gateway restart as the live runtime boundary for plugin changes.

## What This Changes

Request automatic refresh of active agent-session tool availability after `openclaw plugins install` adds plugin-provided tools.

## Best Possible Solution

Add a generic core refresh path that re-resolves plugin-provided tools after successful plugin install or registry refresh and applies the updated tool definitions to active sessions before their next model call, without duplicating plugin runtimes or weakening tool policy.

## Reproduction Assessment

Yes. The issue provides a concrete install -> immediate use -> restart flow, and current-main source inspection shows the install path refreshes registry metadata while active session tool setup remains fixed at session creation/run setup.

## Solution Assessment

Unclear as a patch because no implementation is proposed here. The requested direction is sound, but the safest shape is a turn-boundary refresh/re-resolution path rather than arbitrary mid-stream mutation.

## Repair Work Prompt

Fix issue https://github.com/openclaw/openclaw/issues/72097. Current behavior: after `openclaw plugins install <pkg>` registers new plugin tools, already-active agent sessions do not see those tools until Gateway/session restart. Implement a narrow generic refresh path, preferably at a safe turn boundary, that re-resolves plugin tools after successful plugin install or plugin registry refresh and updates active session tool definitions, prompt/tool inventory, and active tool allowlists together. Do not add plugin-specific ids or owner-specific defaults. Do not duplicate plugin runtimes, HTTP routes, hooks, channel bindings, or registry entries. Preserve explicit `tools.allow`/`tools.deny`, profile, provider, owner-only, group, sandbox, optional-tool, and `pluginToolAllowlist` policy. Inspect the `@mariozechner/pi-coding-agent` session API before mutating an existing session; if the dependency cannot replace custom tools in-place, implement a turn-boundary re-create/re-query approach instead. Add focused regression coverage for install/registry refresh -> next turn sees a newly registered plugin tool in the same active session, plus a guard that policies still block unauthorized plugin tools. Update user-facing docs/changelog only if behavior changes are visible. Treat PR #44401 as adjacent GatewayRestart work, not a substitute for this toolset refresh.
