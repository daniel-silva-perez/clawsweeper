## Summary

Keep open: current main supports only static config-time Docker binds, and existing agent hooks run after sandbox context/container resolution, so the requested per-session bind policy remains missing. Because the request changes host-side execution and sandbox file exposure, it needs maintainer design/security review rather than an automated cleanup close.

## What This Changes

Requests a host-side pre-container-creation hook to resolve Docker sandbox bind mounts dynamically per session context.

## Best Possible Solution

If maintainers accept the capability, add an explicitly trusted core/plugin-SDK sandbox pre-create policy seam that receives bounded session context, returns validated bind specs, can fail container creation clearly, and is covered by config schema, docs, sandbox security tests, and Docker create argument tests. It should not be treated as a ClawHub-only plugin until core exposes a safe pre-container extension point.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Source inspection gives a high-confidence verification path: current main has only static `sandbox.docker.binds`, and the existing hook phases run after sandbox context resolution.

## Solution Assessment

Unclear: the proposed `resolveBinds` executable is plausible, but the best implementation needs maintainer security/product design because it executes host-side logic to decide filesystem exposure. A safer direction may be a typed trusted pre-create policy API with the same validation and timeout guarantees rather than an arbitrary script-only field.
