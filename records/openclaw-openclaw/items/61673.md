## Summary

Keep open. Current main supports static Docker bind mounts and validates them, but it does not implement the requested pre-container dynamic bind resolver or an equivalent hook. The existing plugin hooks are model/prompt/session lifecycle hooks, and the exposed sandbox-backend plugin seam does not currently provide the channel/user/metadata context needed to implement this feature externally without replacing the backend. Related bind-mount fixes and hook requests do not supersede this specific per-session pre-container mount policy gap.

## Best Possible Solution

Keep this open for maintainer product/API design. The best path is a generic pre-create sandbox policy seam, likely near `resolveSandboxContext()` or the Docker backend creation path, that receives structured session/requester/channel metadata, returns validated bind/container changes or a clear denial, has timeout/error semantics, participates in sandbox scope and config-hash behavior, and is documented with focused security tests. If maintainers prefer plugins, first extend the SDK with that narrow pre-create sandbox hook instead of forcing operators to replace the whole Docker backend.

## Risks / Open Questions

- The requested feature is security-sensitive: any implementation must validate dynamic output with the same bind checks, avoid cross-session mount leakage under `agent` or `shared` sandbox scopes, and define config-hash/recreate behavior for per-session bind decisions.
- A plugin-only workaround would likely require expanding the sandbox/plugin API first, because current backend registration does not expose the full channel/user/metadata context described in the issue.
