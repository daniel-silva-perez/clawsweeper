## Summary

Current main still does not implement the requested warn-to-info downgrade for bounded isolated cron retries. It has the adjacent run-scoped isolated cron fix, but loop detection still emits warn-level logs/events for warning results and the hook context still lacks session target, run lifetime, or schedule interval metadata needed to make this decision safely.

## Best Possible Solution

Keep this issue open and land a focused core/cron fix: pass explicit bounded isolated-cron context into loop-warning emission, demote only warning-level loop logs/diagnostics proven to come from isolated cron runs whose elapsed lifetime is within their schedule interval, and preserve current critical blocking, thresholds, run-id scoping, and auditability.

## Clownfish Work Prompt

Fix the open issue about `[agents/loop-detection]` warning noise for bounded isolated cron sessions. Current main already scopes isolated cron runs by run id/session key, but same-run repeated `exec` or tool calls still emit warn-level loop messages. Implement a focused additive change that passes explicit bounded isolated-cron context into the before_tool_call/loop-warning path and demotes only warning-level loop logs/diagnostics to info when the run is known to be an isolated cron run and its elapsed lifetime is within the job schedule interval. Preserve all critical/global circuit breaker blocking, thresholds, non-cron warnings, non-isolated cron warnings, run-id history scoping, and audit behavior. Do not infer policy from brittle session-key strings if typed context can be passed. Update focused tests for the demotion case and for preserved warning/critical cases. Update `docs/tools/loop-detection.md` if user-visible behavior is documented. Add a changelog fix entry with `Thanks @cjplanted` if the change is user-facing log behavior.
