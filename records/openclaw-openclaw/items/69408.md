## Clownfish Work Prompt

Fix the loop-detection log-noise issue for bounded isolated cron runs. Current main scopes tool-call history by run ID, but warning-level loop results still log with `log.warn` for same-run retries. Implement a narrow fix that carries explicit cron run context from isolated cron execution into `createOpenClawCodingTools` / `HookContext` / loop-warning emission. Prefer typed metadata such as trigger, jobId, sessionTarget, runStartedAtMs, and cronIntervalMs, or a small derived `boundedIsolatedCronRun` flag if that is cleaner. Demote only non-critical loop warnings to info when the context proves this is an isolated cron run whose elapsed lifetime is within the cron interval; if exact interval is unavailable, use the existing documented fallback idea conservatively and cover it with tests. Preserve critical loop blocking, error-level critical logs, loop thresholds, run-id scoping, and non-isolated warning behavior. Do not silence warnings entirely, change global defaults, alter plugin/channel-specific behavior, or broaden owner-only cron/tool policy. Add focused tests for: non-isolated warning remains warn, bounded isolated cron warning logs as info/audit only, critical remains blocked/error, and run-id scoping remains intact. Update `docs/tools/loop-detection.md` if the observable behavior changes, and add a single-line changelog Fixes entry credited with `Thanks @cjplanted`.
