Keep open. Current main has meaningful Telegram polling and transport hardening, including shipped timeout-clamp work in v2026.4.29, but the discussion and code still do not provide affected-host current-main proof that the macOS/India sticky IPv4 fallback loop is resolved.

## What This Changes

The issue reports Telegram long-polling never consuming `getUpdates` on a reachable Bot API host, with repeated sticky IPv4 fallback warnings and pending Telegram updates accumulating.

## Best Possible Solution

Keep the issue open until the affected host retests current main or v2026.4.29 with Telegram diagnostics. The desired end state is either confirmed healthy `getUpdates` completion through OpenClaw's gateway fetch path or a focused Telegram plugin fix in the transport or polling-session lifecycle based on that evidence.
