## Summary

Keep open. Current main has meaningful Telegram polling and transport hardening, but the issue discussion explicitly treats local success and the merged #71502 lifecycle fix as retest baselines, not closure evidence, and there is still no affected-host current-main proof that the India/macOS sticky IPv4 fallback loop is resolved.

## What This Changes

The issue reports Telegram long-polling failing to consume `getUpdates` while repeatedly entering the sticky IPv4 fallback path despite a valid bot token and reachable Telegram Bot API from the same host.

## Best Possible Solution

Keep Telegram ownership in the Telegram plugin and isolate the failing layer with current-main affected-host diagnostics. The desired end state is either confirmed healthy `getUpdates` completion through OpenClaw's in-gateway fetch wrapper or a focused fix in `extensions/telegram/src/fetch.ts` or `extensions/telegram/src/polling-session.ts` based on diagnostic evidence.

## Reproduction Assessment

No high-confidence current-main reproduction is available. The original 2026.4.22 steps and logs are concrete, but the missing path is an affected-host retest on current main with `OPENCLAW_DIAG=telegram` plus an OpenClaw `resolveTelegramFetch` `getMe`/`getUpdates` probe.

## Solution Assessment

Unclear. #71502 and de1ac12f are reasonable baselines, but the best next step is diagnostics before code because the remaining symptom could be transport fallback, polling ownership, or launch-environment specific.
