## Summary

Keep this open. Current main has Telegram polling watchdog, transport rebuild, and status-warning hardening, but the specific sticky IPv4 fallback loop reported from the affected macOS/India host is still a plausible unresolved transport problem and maintainers already noted local healthy probes do not disprove it.

## What This Changes

The issue reports that Telegram long-polling stops consuming Bot API updates on a reachable macOS host while repeatedly entering the sticky IPv4 fallback path.

## Best Possible Solution

Keep the issue open for a current-main affected-host retest with `OPENCLAW_DIAG=telegram` through one full stall/rebuild cycle. If an in-gateway probe using `resolveTelegramFetch` fails, focus the fix in `extensions/telegram/src/fetch.ts`; if that probe succeeds while gateway polling still wedges, continue in `extensions/telegram/src/polling-session.ts` using #71502 as the baseline.

## Reproduction Assessment

No high-confidence current-main reproduction is available. The original report has concrete steps and logs, but the missing proof is a current-main retest on the affected macOS/India host rather than local simulated tests or a healthy maintainer probe.

## Solution Assessment

Unclear. The current watchdog and visibility changes are useful but not sufficient evidence that the sticky IPv4 fallback failure is solved; the narrowest maintainable fix depends on whether the in-gateway fetch wrapper or polling-session lifecycle fails on the affected host.
