## Summary

Keep open. Current main has several Telegram polling hardening changes, including watchdog restarts, dirty transport rebuilds, status warnings, sticky IPv4 fallback, and a new timeout floor for polling clients, but there is still no high-confidence proof that the macOS/India sticky IPv4 fallback loop reported here is fixed. The maintainer discussion explicitly kept the report open for affected-host retest data, and the latest timeout fix may cover a neighboring symptom only if a low `channels.telegram.timeoutSeconds` was involved, which the issue body does not establish.

## What This Changes

The issue reports that Telegram getUpdates long-polling never consumes updates on a reachable macOS/India host and repeatedly logs the sticky IPv4 fallback warning.

## Best Possible Solution

Keep the issue open for an affected-host current-main retest with `OPENCLAW_DIAG=telegram` through a full stall/rebuild cycle. If an in-gateway probe using OpenClaw's `resolveTelegramFetch` fails for getMe/getUpdates, focus the fix in `extensions/telegram/src/fetch.ts`; if that probe succeeds while gateway polling still wedges, focus on polling session lifecycle and runner ownership in `extensions/telegram/src/polling-session.ts`, with #71502 and de1ac12f as the baseline.

## Reproduction Assessment

No high-confidence current-main reproduction is available. The original steps and logs are concrete, but the missing proof is a current-main retest on the affected macOS/India host with gateway diagnostics and an in-gateway `resolveTelegramFetch` probe.

## Solution Assessment

Unclear. Current main has useful recovery and visibility changes, and de1ac12f may solve low-timeout churn, but the narrowest maintainable fix depends on whether the affected host still fails inside the Telegram fetch wrapper or only in the polling runner lifecycle.
