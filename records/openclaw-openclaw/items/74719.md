## Summary

Keep open. Current main already rejects undefined `chat.send.message` at the RPC schema boundary and partially hardens undefined reply text, but it does not fully implement the PR’s intended runtime text narrowing because `normalizeReplyPayload` still reuses raw `payload.text` before later string-only silent-token paths. The PR is useful source work, but it is dirty against current main and needs to be reapplied to the refactored normalizer with regression coverage.

## What This Changes

The PR adds runtime guards for non-string chat send messages and non-string auto-reply payload text before string methods are called.

## Best Possible Solution

Reapply the useful part of the PR to current main by narrowing reply payload text once before all string operations in `normalizeReplyPayload`, preserve media/channel-data-only payload behavior, and add focused tests for undefined and non-string runtime text. Keep the chat.send boundary validation intact; only widen the sanitizer signature if tests prove direct unknown-safe use is intended.
