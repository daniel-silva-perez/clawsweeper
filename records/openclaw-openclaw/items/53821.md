## Summary

Keep open. The PR is still useful and addresses a real runtime gap, but it is not merge-ready against current main because it introduces a required SDK parameter without updating an existing bundled plugin caller, and one sessions_list path still discards the override-aware requester key for spawned-session filtering.

## What This Changes

The PR adds runtime wiring and tests for per-agent `agents.list[].sandbox.sessionToolsVisibility` overrides across session list, history, send, and status tools.

## Best Possible Solution

Rebase the PR on current main, keep the plugin SDK guard contract backward-compatible or update every current consumer, then make `sessions_list` use the override-aware `effectiveRequesterKey` returned by the shared helper and add a regression covering sandboxed spawned list visibility with `requesterAgentIdOverride`.

## Reproduction Assessment

Yes. The SDK issue reproduces by typechecking the PR against current main, where `extensions/memory-core/src/session-search-visibility.ts` calls the now-required `createSessionVisibilityGuard` parameter set without `mainKey`; the list bug reproduces with a sandboxed `agentSessionKey="global"`, `requesterAgentIdOverride="tony"`, clamp `spawned`, and a spawned `agent:tony:*` session.

## Solution Assessment

No. The requested direction is reasonable, but the current implementation is not the best mergeable solution until it preserves the exported SDK contract/current memory-core caller and threads the helper's override-aware requester key through list filtering.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P1] Keep the visibility guard SDK call compatible:** `src/plugin-sdk/session-visibility.ts:282-283`
  - body: `createSessionVisibilityGuard` is exported through `openclaw/plugin-sdk/session-visibility`, and current main already has `extensions/memory-core/src/session-search-visibility.ts` calling it without `mainKey`. Making `mainKey` required here means the PR will fail typechecking after rebase/merge unless every current SDK caller is updated or the parameter is optional/defaulted.
  - confidence: 0.93
- **[P2] Use the override-aware requester key for list filters:** `src/agents/tools/sessions-list-tool.ts:78-85`
  - body: `resolveSandboxedSessionToolContext` now synthesizes an override-aware `effectiveRequesterKey`, but `sessions_list` discards it and rebuilds `requesterInternalKey ?? alias`. In sandboxed `spawned` override flows this sends `spawnedBy: global` instead of `agent:<override>:<mainKey>`, so the override agent's own spawned sessions are hidden from the list.
  - confidence: 0.88
