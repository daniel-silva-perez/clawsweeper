## Summary

Keep open: current main still accepts `compat.supportsTools: false`, but the OpenAI Completions payload builder does not enforce it before emitting `tools`, `tool_choice`, or the tool-history `tools: []` fallback. A linked fix PR is open and unmerged, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue reports that OpenAI-compatible chat-completions models with `compat.supportsTools: false` still receive tool-related payload fields when tools or tool history are present.

## Best Possible Solution

Review and land PR #74738 or an equivalent narrow transport-layer fix that applies `supportsModelTools(model)` inside `buildOpenAICompletionsParams()` and adds regression coverage for both non-empty tools and the tool-history fallback.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call `buildOpenAICompletionsParams()` with `model.compat.supportsTools: false` and either non-empty `context.tools` or prior tool-call history; the current builder still writes a tool payload field in both paths.

## Solution Assessment

Yes. Enforcing the existing `supportsModelTools()` contract in the OpenAI Completions payload builder is the narrowest maintainable fix because it protects both normal runner calls and direct transport callers.
