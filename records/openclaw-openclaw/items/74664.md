## Summary

Keep open: current main still accepts `compat.supportsTools: false` but the active OpenAI-compatible completions payload builder does not check it before emitting `tools`, `tool_choice`, or the tool-history `tools: []` fallback. The issue is also paired with open PR #74738, so it should remain open until that PR is reviewed, merged, or closed.

## What This Changes

The issue requests that OpenAI-compatible chat-completions payload construction honor `compat.supportsTools: false` by omitting tool payload fields, including the tool-history fallback.

## Best Possible Solution

Review and land a narrow fix like #74738 if its diff is correct: gate `tools`, `tool_choice`, and the tool-history `tools: []` fallback inside `buildOpenAICompletionsParams()` using the existing `supportsModelTools()` contract, with regression tests for non-empty tools and replayed tool history.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call `buildOpenAICompletionsParams()` with `model.compat.supportsTools === false` and a non-empty `context.tools`; current main reaches the `context.tools` branch and assigns `params.tools` anyway.

## Solution Assessment

Yes, the linked fix direction is the narrow maintainable solution because the transport builder is the last payload boundary and should not rely only on upstream runner filtering. Current main is not fixed yet, so the best path is to resolve the paired PR rather than close this issue now.
