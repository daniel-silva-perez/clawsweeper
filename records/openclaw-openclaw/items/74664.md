## Summary

Keep open: current main still exposes `compat.supportsTools: false`, but the OpenAI Completions payload builder does not consult that flag and can still emit `tools`, `tool_choice`, or the tool-history `tools: []` fallback. The linked fix PR is open and unmerged, so this issue should stay open until that work lands or is replaced.

## What This Changes

The issue requests that OpenAI-compatible chat-completions payload construction omit `tools`, `tool_choice`, and the tool-history `tools: []` fallback when `compat.supportsTools` is false.

## Best Possible Solution

Land PR #74738 or an equivalent narrow fix in `src/agents/openai-transport-stream.ts` that gates OpenAI Completions tool emission with `supportsModelTools(model)` and covers both normal tools and replayed tool history in regression tests.

## Reproduction Assessment

Yes. The source-level reproduction is deterministic on current main: call `buildOpenAICompletionsParams()` with `model.compat.supportsTools: false` and non-empty `context.tools`, or with tool-call history and no current tools, and the builder still writes a `tools` field.

## Solution Assessment

Yes. The narrow maintainable fix is to enforce the existing `supportsModelTools()` contract inside the OpenAI Completions payload builder itself, because that protects direct transport callers and any caller that passes an unfiltered context.
