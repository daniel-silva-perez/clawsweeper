## Summary

Keep open: current main documents and accepts `compat.supportsTools: false`, and the embedded runner suppresses normal OpenClaw tool registration from that flag, but the current `@mariozechner/pi-ai` OpenAI Completions adapter still ignores that compat value and injects `tools` whenever `context.tools` or tool-history fallback is present. Existing OpenClaw payload wrappers cover similar `store` compatibility but not this tool payload path, so the report remains valid and narrow.

## What This Changes

OpenAI-compatible chat-completions models configured with `compat.supportsTools: false` should omit tool payload fields, including the tool-history `tools: []` fallback, when tools are otherwise available.

## Best Possible Solution

Add a narrow OpenAI Completions compatibility guard in the existing embedded stream wrapper layer, or consume an upstream `pi-ai` fix if one lands first, so `compat.supportsTools === false` removes tool-related payload fields even when `context.tools` or replayed tool history reaches the transport. Keep the existing OpenClaw-level tool gating and add payload-capture regression coverage for both non-empty `tools` and the tool-history `tools: []` fallback.
