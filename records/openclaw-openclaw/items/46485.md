## Summary

Keep open for maintainer review. Current main still does not expose a plugin runtime Gateway dispatcher, but this PR adds a broad public plugin API across a security-sensitive Gateway boundary without a settled method/capability contract, plugin identity propagation, docs, or test-helper coverage; the related broader API request in #39000 is still open.

## What This Changes

The PR exports the internal Gateway dispatcher, adds a required `PluginRuntime.gateway.dispatchMethod<T>(method, params)` namespace, wires it into Gateway plugin runtime options, and adds a non-Gateway fallback error.

## Best Possible Solution

Define a narrow, documented plugin runtime contract for the sanitized-history and inter-plugin request/response use cases, with explicit allowed methods or capability checks, caller identity propagation, fallback behavior, public SDK docs, and contract tests; coordinate that design with #39000 instead of exposing the internal dispatcher wholesale.

## Reproduction Assessment

Not applicable as a feature/API PR rather than a bug report. The high-confidence review path is static: current main lacks `api.runtime.gateway`, and the PR diff adds a broad required runtime method without the surrounding authorization, identity, docs, and test-helper updates.

## Solution Assessment

No. Exposing the internal dispatcher directly is not the narrowest maintainable solution; a scoped/capability-checked helper or purpose-built sanitized history/runtime service API would preserve the existing Gateway trust model better.
