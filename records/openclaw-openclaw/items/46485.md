## Summary

Keep open. Current main still does not expose a plugin `runtime.gateway.dispatchMethod` surface, so this PR is not obsolete; however, the proposed patch broadens the plugin runtime from narrow helpers to arbitrary Gateway RPC dispatch and still needs maintainer API/security review plus rebase/test updates before it is mergeable.

## What This Changes

The PR exports the internal gateway dispatcher, wires it into plugin runtime options, and adds `PluginRuntime.gateway.dispatchMethod<T>(method, params)` with an unavailable fallback outside Gateway contexts.

## Best Possible Solution

Keep the useful use case alive, but land it as an explicitly reviewed plugin runtime contract: either a narrow sanitized session-history helper that preserves `chat.history` projection semantics, or a constrained gateway dispatch helper with method authorization, plugin identity attribution, docs, API baseline/contract tests, and regression coverage for non-Gateway fallback behavior.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Update runtime mocks for gateway:** `src/plugins/runtime/types.ts:62-65`
  - body: Adding a required `PluginRuntime.gateway` member makes the existing `createPluginRuntimeMock` object incomplete: it is typed as `PluginRuntime` and currently only provides `subagent` and `nodes` at the end of the base object. Please add a default `gateway.dispatchMethod` mock or make the new runtime surface optional before changing the public type.
  - confidence: 0.9
- **[P2] Preserve plugin identity for gateway dispatches:** `src/plugins/runtime/types.ts:62-65`
  - body: The registry proxy only wraps `runtime.subagent` calls in `withPluginRuntimePluginIdScope`; other runtime properties pass through unchanged. Exposing `runtime.gateway.dispatchMethod` this way means service/event callers using the fallback path will dispatch without `pluginRuntimeOwnerId`, so plugin-owned authorization/audit paths cannot attribute the caller. Mirror the subagent wrapping or pass owner identity through the new gateway helper.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The patch exposes a broad in-process Gateway RPC dispatcher to plugin code and needs an explicit authorization and plugin-identity design before merge.

Concerns:

- **[medium] Arbitrary Gateway RPC dispatch from plugins:** `src/plugins/runtime/types.ts:62`
  - body: The new method accepts any gateway method name and params, while the existing dispatcher uses the current request client or a synthetic operator client. That widens plugin authority from curated helpers to the Gateway method bus, including read and write-scoped methods, without an allowlist or capability contract.
  - confidence: 0.87
- **[medium] Missing plugin caller attribution:** `src/plugins/registry.ts:1978`
  - body: Current runtime proxy code attaches plugin identity only around `runtime.subagent`; a pass-through `runtime.gateway` helper would not automatically set `pluginRuntimeOwnerId`, weakening ownership checks and auditability for fallback dispatches.
  - confidence: 0.78
