## Summary

Keep this PR open, but do not merge it as-is. It addresses a real missing Plugin SDK call-side Gateway API that is also tracked in #39000, while the proposed generic dispatcher needs maintainer design on authorization, caller identity, docs, and test contracts before it becomes a public runtime surface.

## What This Changes

This PR exports the internal Gateway dispatcher, adds a required `runtime.gateway.dispatchMethod<T>(method, params)` PluginRuntime namespace, and wires it into Gateway plugin runtime options with a non-Gateway fallback.

## Best Possible Solution

Define a narrow, documented Plugin SDK request/response seam for sanitized history and inter-plugin calls, with explicit allowed methods or capabilities, plugin caller identity propagation, fallback behavior, SDK docs, typed mocks, and contract tests coordinated with #39000.

## Reproduction Assessment

Not applicable as a feature/API PR rather than a bug report. The high-confidence check is static: current main lacks a plugin-side Gateway call API, while the PR diff adds a broad required runtime member without the needed authorization, identity, docs, and test-helper updates.

## Solution Assessment

No. Exposing the internal dispatcher directly is not the narrowest maintainable solution; a scoped helper, capability-checked dispatcher, or purpose-built sanitized-history/inter-plugin service API would preserve the Gateway trust model better.
