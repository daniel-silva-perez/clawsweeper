## Summary

Keep open. Current main still lacks the requested plugin runtime Gateway-dispatch surface, but this PR’s current shape is not merge-ready because it exposes the broad in-process Gateway RPC bus without a reviewed capability/allowlist contract, plugin identity propagation, docs, or complete runtime test-helper coverage.

## What This Changes

The PR exports the internal Gateway dispatcher and adds a required `PluginRuntime.gateway.dispatchMethod<T>(method, params)` namespace wired through plugin runtime options.

## Best Possible Solution

Design a narrow, documented plugin runtime contract that covers the sanitized-history and inter-plugin request/response use cases while explicitly defining allowed methods or capabilities, caller identity propagation, fallback behavior, docs, and contract tests; coordinate that with the broader open #39000 API discussion.

## Reproduction Assessment

Not applicable as a feature/API PR rather than a bug report. The high-confidence review path is static: current main lacks `api.runtime.gateway`, and the PR diff adds a broad required runtime method without the surrounding identity, authorization, docs, and test-helper updates.

## Solution Assessment

No. Exposing the internal dispatcher wholesale is not the narrowest maintainable solution; a safer path is a scoped/capability-checked runtime API or purpose-built helper that preserves plugin identity and documents exactly which Gateway methods plugin code may call.
