## Summary

Keep open. Current main does not implement the requested plugin-side Gateway caller, but the proposed direct dispatcher broadens plugin access to Gateway RPCs without a settled capability, authorization, and caller-identity contract; the related API request #39000 is still open.

## What This Changes

The PR exports the internal Gateway dispatcher and adds a required `api.runtime.gateway.dispatchMethod<T>(method, params)` runtime surface with a non-Gateway fallback.

## Best Possible Solution

Define a narrow, documented Plugin SDK runtime contract for sanitized history and inter-plugin request/response calls, with explicit allowed methods or capabilities, plugin caller identity propagation, fallback behavior, SDK docs, typed mocks, and contract tests coordinated with #39000.

## Reproduction Assessment

Not applicable as a feature/API PR rather than a bug report. The high-confidence check is static: current main lacks `api.runtime.gateway`, and the PR diff adds a broad required runtime member without the surrounding authorization, identity, docs, and test-helper updates.

## Solution Assessment

No. Exposing the internal dispatcher directly is not the narrowest maintainable solution; a scoped or capability-checked helper, or a purpose-built sanitized-history/inter-plugin API, would preserve the existing Gateway trust model better.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Gate Gateway RPC access before exposing it:** `src/plugins/runtime/types.ts:62-64`
  - body: Adding `runtime.gateway.dispatchMethod` as a public runtime member lets plugin code pass any Gateway method name and params into the internal dispatcher. This moves plugins from curated helpers to the Gateway method bus without an allowlist or capability contract, so method/scope gating needs to be part of the API before merge.
  - confidence: 0.9
- **[P2] Preserve plugin identity during gateway dispatch:** `src/gateway/server-plugins.ts:192-193`
  - body: The PR wires the raw dispatcher into runtime options, but current main's runtime proxy attaches plugin identity only around `runtime.subagent`. Calls through this new helper would not automatically set `pluginRuntimeOwnerId`, weakening owner-scoped checks and audit attribution for fallback dispatches.
  - confidence: 0.86
- **[P2] Add the required gateway namespace to runtime mocks:** `src/plugins/runtime/types.ts:62-65`
  - body: Making `gateway` a required `PluginRuntime` member leaves shared typed fixtures incomplete: current `createPluginRuntimeMock` returns `subagent` and `nodes` but no `gateway`. Add a default mock or make the new surface optional until fixtures and contract tests are updated.
  - confidence: 0.84
