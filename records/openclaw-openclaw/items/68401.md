## Summary

Keep open. Current main has addressed the narrower Discord listener-timeout symptom from #73204, but it still does not implement the broader request to process normal same-session channel messages while compaction is running: same-session turns and queued compaction serialize through the session lane, built-in compaction holds the session write lock through the provider compaction call, steering into a compacting run is rejected, and followups are queued for later rather than processed independently.

## What This Changes

Allow inbound channel messages to be processed, offloaded, or acknowledged with minimal disruption while session compaction is running.

## Best Possible Solution

Treat this as a core session-runtime design change: make compaction nonblocking through copy-on-write/background compaction or add a bounded busy-session acknowledgment/offload path, then cover Telegram/Discord-style inbound messages arriving during compaction without corrupting transcript order or followup delivery.
