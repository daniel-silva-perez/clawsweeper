## Summary

Keep open. The related #73204 Discord listener-timeout path appears addressed, but current main still does not implement the broader request to process normal same-session channel messages during compaction: same-session runs and compaction serialize through session lanes, steering into a compacting embedded run is rejected, and built-in compaction still holds the transcript write lock while the provider compaction call runs.

## What This Changes

Request that OpenClaw process, offload, or promptly acknowledge incoming channel messages while an active session is compacting.

## Best Possible Solution

Treat this as a core session-runtime design: provide nonblocking compaction through copy-on-write/background compaction or a bounded busy-session acknowledgment path, then add regression coverage proving an inbound Telegram/Discord-style message during compaction is accepted promptly without corrupting the compacted transcript or followup ordering.

## Risks / Open Questions

- A correct implementation has to isolate transcript appends from compaction summary/rotation so concurrent foreground messages cannot be lost, duplicated, or summarized against the wrong leaf.
- Channel behavior differs: Telegram can acknowledge webhook updates quickly, while Discord interaction handling has separate listener deadlines, so a core fix still needs channel-specific delivery and retry checks.
- The recently shipped #73204 fix narrows the message-loss symptom for Discord listener timeouts, but it may mask the broader remaining UX problem as delayed queued followups rather than dropped events.
