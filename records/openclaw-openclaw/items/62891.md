## Summary

Keep this issue open. The discussion narrows the report to a cron/channel readiness race during startup and SIGUSR1/config.patch hot reload. Current main still starts cron as soon as post-attach sidecars hand off, and hot cron reload still restarts cron without waiting for channel transport readiness, so due startup catch-up or re-armed cron jobs can execute before Discord reports connected.

## Best Possible Solution

Keep this issue open and implement a bounded cron readiness gate at the gateway/cron boundary. The likely path is to delay startup catch-up and automatic timer execution for channel-delivered cron jobs until the relevant configured channels are connected/healthy, a configurable timeout expires, or readiness fails with clear diagnostics. Cover full startup, config.patch/SIGUSR1 hot reload, Discord-style asynchronous readiness, missed-job catch-up, and no-channel or webhook-only cron jobs.
