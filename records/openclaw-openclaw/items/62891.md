## Summary

Keep open. Current main still starts cron without a channel-transport readiness gate, and hot config reload restarts cron before restarting channels; the existing mitigation only defers overdue isolated agent-turn catch-up jobs, not normal due ticks or channel-delivered cron work.

## What This Changes

The issue asks for startup and SIGUSR1/config.patch cron scheduling to wait for channel readiness so Discord-delivered cron/LLM work does not fire during the reconnect window.

## Best Possible Solution

Add a bounded, delivery-aware readiness gate at the gateway/cron boundary for initial activation and hot cron reload. It should use generic channel readiness, avoid Discord-specific core logic, skip jobs that do not need channel delivery, time out with clear diagnostics, and cover both full startup and SIGUSR1/config.patch reload behavior.
