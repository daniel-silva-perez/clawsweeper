## Summary

Keep open. Current main still starts cron without waiting for channel transport readiness, and the newer startup mitigation only defers overdue isolated agent-turn catch-up; it does not cover normal due-job ticks or hot cron reload before Discord reconnects.

## What This Changes

The issue requests startup and SIGUSR1/config-reload reconciliation so cron jobs do not run Discord-delivered or LLM work before the configured channel transport is ready.

## Best Possible Solution

Add a bounded delivery-aware readiness gate at the gateway/cron boundary for initial activation and hot reload. It should wait only for jobs that can deliver to configured channel transports, skip non-channel jobs, time out with clear diagnostics, and keep the core logic generic rather than Discord-specific.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is an enabled Discord channel plus a due or overdue isolated cron job with channel delivery, then startup or SIGUSR1/config.patch reload before Discord READY; current main starts cron and can run due jobs before channel transport readiness is checked.

## Solution Assessment

No. The current main mitigation is useful but incomplete because it delays only missed startup agent-turn catch-up, while normal due ticks and hot cron restart still lack a channel-readiness gate.
