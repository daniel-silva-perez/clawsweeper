## Summary

Keep open. Current main still starts and restarts cron without waiting for channel transport readiness, while the existing startup mitigation only defers overdue isolated agent-turn catch-up. Hot cron reload and normal due-job ticks can still run Discord-delivered work before the Discord WebSocket reports connected.

## What This Changes

The issue requests startup and SIGUSR1/config.patch cron scheduling to wait for session/channel readiness so Discord-delivered cron or LLM work does not time out before the channel transport is connected.

## Best Possible Solution

Add a bounded, delivery-aware readiness gate at the gateway/cron boundary for initial activation and hot cron restart. It should wait only when a runnable job depends on a configured channel transport, skip non-channel jobs, time out with clear diagnostics, and cover both full startup and SIGUSR1/config.patch reload behavior.

## Repair Work Prompt

Fix the current cron/channel readiness issue. Current main starts cron immediately at gateway scheduled-service activation and on config.patch/SIGUSR1 hot reload, so channel-delivered cron jobs can run before Discord-style transports report connected. Add a bounded, delivery-aware readiness gate at the gateway/cron boundary for initial startup activation and hot cron restart. The fix should wait only for runnable jobs that require a configured channel delivery target, skip or bypass the wait for non-channel jobs and webhook/no-delivery jobs, time out with clear diagnostics instead of hanging startup, and preserve existing startup-deferred missed agent job behavior. Do not move extension-owned Discord behavior into core beyond generic channel-readiness seams. Update tests around scheduled-service activation, reload ordering, readiness, cron startup catch-up, and timer execution. Update docs/changelog only if user-facing behavior or config surface changes.
