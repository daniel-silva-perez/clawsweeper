## Summary

Keep open. Current main has meaningful session and subagent primitives, including tree-scoped session history/send, subagent list/steer/kill controls, and bounded restart-abort recovery, but it does not fully satisfy the reported QClaw WebChat-WeChat continuity request or expose an explicit user-facing subagent retry/resume/status API.

## What This Changes

The issue requests better cross-session and cross-agent continuity for QClaw/OpenClaw, including parent control of spawned subagents, recovery after aborts, and WebChat-WeChat session sharing.

## Best Possible Solution

Keep this as an owner-reviewed session/subagent UX task: reproduce the QClaw WebChat-WeChat flow on current main with the external WeChat plugin, decide whether first-class retry/resume/status APIs are needed beyond steer and bounded orphan recovery, and document the supported privacy-safe continuity path for dmScope, identityLinks, bindings, memory, and plugin-owned channel behavior.

## Reproduction Assessment

Unclear. The issue gives a concrete user scenario, but it lacks the current-main config, QClaw and plugin versions, logs, and transcript snippets needed to reproduce the WebChat-WeChat and abort cases with high confidence from the repository alone.

## Solution Assessment

No, closing this as implemented or sending one automatic repair PR is not the best way to solve it. Current main covers important primitives, but the remaining behavior spans product semantics, privacy defaults, external WeChat plugin behavior, and client UX, so maintainers should choose the split before implementation.
