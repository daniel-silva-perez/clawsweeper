## Summary

Keep open. Current main covers several reported primitives now, including tree-scoped session history/send, subagent inspect/send/log slash controls, and bounded restart-abort recovery, but it does not fully settle the requested QClaw WebChat-WeChat continuity or expose an explicit agent tool API for subagent retry/resume/status beyond the existing controls.

## What This Changes

Request cross-session and cross-agent continuity so WebChat/WeChat and parent/child agents can share context, communicate, inspect status, recover from aborts, and avoid task-chain loss.

## Best Possible Solution

Keep this as an owner-reviewed session and subagent UX task: define the privacy-safe supported flow for QClaw/WebChat-WeChat continuity, decide whether the agent tool surface needs first-class subagent status/retry/resume actions in addition to existing slash controls and sessions_send/history, and document the expected dmScope, identityLinks, binding, and external WeChat plugin behavior.

## Reproduction Assessment

Unclear. The issue gives a concrete user scenario and an aborted child-agent example, but the repository evidence only verifies current primitives; it does not include enough QClaw, WeChat plugin, config, logs, or transcript data to reproduce the exact cross-channel and abort failures with high confidence.

## Solution Assessment

No. Closing as implemented or sending one automatic repair PR is not the best solution because the remaining behavior spans core session tools, subagent recovery semantics, privacy defaults, external WeChat plugin routing, and client UX; maintainers should choose the product/API split first.
