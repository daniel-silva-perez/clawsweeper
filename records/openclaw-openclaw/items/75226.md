## Summary

Keep this PR open, but do not merge it as-is. Current main still has the active non-streaming Pi steering gap this PR targets, but the patch defaults the future Pi tool-boundary steering behavior to `interrupt` for callers that omit the new option; existing subagent wake/completion steering callers omit it, so normal steering would become interruption once the paired Pi API lands. The PR also needs the required user-facing changelog entry.

## What This Changes

The PR lets embedded Pi active-run steering enqueue while a run is active but not streaming, adds a per-message `toolCallSteeringBehavior` option, classifies auto-reply steering as inject versus interrupt, and expands Pi run-registry and auto-reply tests.

## Best Possible Solution

Land a coordinated active-run steering fix that defaults normal omitted steering to inject, uses interrupt only for stop-like messages, covers auto-reply and subagent steering callers, and validates the published Pi API before enabling tool-boundary interruption semantics.

## Reproduction Assessment

Yes. The main-branch gap is source-reproducible by tracing a `shouldSteer`, `isActive: true`, `isStreaming: false` reply through `runReplyAgent` and `queueEmbeddedPiMessage`; the PR defect is source-reproducible by following existing subagent announce callers that do not pass `toolCallSteeringBehavior`.

## Solution Assessment

No, not yet. Steering active non-streaming Pi runs is the right direction, but a session-wide interrupt default is not the narrowest maintainable solution; omitted queue behavior should resolve to inject unless a caller explicitly classifies the message as stop-like.
