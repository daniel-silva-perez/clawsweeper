## Summary

Keep this PR open, but do not merge it as-is. Current main still has the active non-streaming steering gap the PR is trying to address, while the PR diff introduces a blocking defaulting bug: omitted Pi steering behavior becomes `interrupt`, and existing non-auto-reply steering callers do not pass the new option.

## What This Changes

The PR changes embedded Pi reply steering to queue messages into active non-streaming runs, adds per-message inject/interrupt steering behavior, and expands Pi run-registry and auto-reply steering tests.

## Best Possible Solution

Land this only after omitted/default active-run steering is treated as normal inject behavior, interrupt is reserved for stop-like messages, non-auto-reply steering callers are covered, and the OpenClaw seam is validated against the published Pi API.

## Reproduction Assessment

Yes. The current-main gap is source-reproducible through `runReplyAgent` and `queueEmbeddedPiMessage` gating on streaming, and the PR defect is source-reproducible by following subagent announce steering calls that omit `toolCallSteeringBehavior` into the PR's session-wide `interrupt` default.

## Solution Assessment

No, not as-is. Steering active non-streaming Pi runs is the right direction, but defaulting the session-wide Pi behavior to `interrupt` is broader than the intended stop-like path and makes existing normal steering callers unsafe.
