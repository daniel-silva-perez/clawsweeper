## Summary

Keep open. Current main still closes the WhatsApp socket without draining inbound debounce work, but this PR’s proposed `flushAll()` is not a complete fix because it snapshots buffered keys once and can return before late enqueues or active keyed work finish.

## What This Changes

This PR adds a shared inbound debouncer `flushAll()` API, calls it from the WhatsApp Web inbox close path before socket teardown, and updates channel/test debouncer mocks for the new method.

## Best Possible Solution

Revise the debouncer API into a real quiescence drain: stop or account for late enqueue work, repeatedly flush current buffers, wait for active keyed chains, and only then let the WhatsApp close path call `closeInboundMonitorSocket(sock)`, with focused regression tests for send-before-close behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Wait for the debouncer to fully drain before close:** `src/auto-reply/inbound-debounce.ts:235-237`
  - body: `flushAll()` snapshots `buffers.keys()` once and awaits only those `flushKey()` calls. During shutdown, an already-running `handleMessagesUpsert` can enqueue after that snapshot, and timer-backed flushes can leave active `keyChains` after their buffer was deleted. In both cases `close()` can still close the socket while a debounced reply or media send is pending.
  - confidence: 0.9
