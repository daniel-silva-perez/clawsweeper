## Summary

Keep open. The PR targets a real WhatsApp close-path gap that current main still has, but the proposed `flushAll()` is not a full drain: it snapshots only current buffers and can return while late enqueue work or active keyed flush chains are still running, after which the WhatsApp socket is closed.

## What This Changes

This PR adds a shared inbound debouncer `flushAll()` API, calls it from the WhatsApp Web inbox close path before socket teardown, and updates channel/test debouncer mocks for the new method.

## Best Possible Solution

Revise the debouncer into a real quiescence drain: account for late enqueue work, repeatedly flush current buffers, wait for active keyed chains, and only then let the WhatsApp close path call `closeInboundMonitorSocket(sock)`, with focused regression tests for send-before-close behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Wait for the debouncer to quiesce before closing:** `src/auto-reply/inbound-debounce.ts:235-237`
  - body: `flushAll()` snapshots `buffers.keys()` once and waits only for those `flushKey()` calls. An already-running `handleMessagesUpsert` can enqueue after that snapshot, and timer-backed flushes can have deleted their buffer while `keyChains` work is still active. Since the WhatsApp close path closes the socket immediately after `flushAll()`, debounced replies or media sends can still run against a closed socket.
  - confidence: 0.9
