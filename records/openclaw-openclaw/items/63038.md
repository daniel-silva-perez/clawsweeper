## Summary

Keep this PR open. Current main still lacks a debouncer-wide shutdown drain before the WhatsApp socket is closed, so the user-visible problem is not implemented on main. The PR is also not merge-ready as submitted because its new flushAll snapshots buffered keys once and does not prove late enqueues or active keyed flush work finish before sock.ws.close. Security review found no unexplained supply-chain, CI, dependency, lockfile, script, secret, permission, downloaded-artifact, or generated/vendor changes in the PR file list.

## Best Possible Solution

Keep this PR open for revision. The best fix is a debouncer drain/quiescence API in src/auto-reply/inbound-debounce.ts that repeatedly flushes current buffers and waits for active keyed chains until stable, then use it from extensions/whatsapp/src/inbound/monitor.ts after detaching inbound listeners and before closing the socket. Add focused WhatsApp regression tests proving reply and sendMedia complete before sock.ws.close, including late-enqueue and in-flight keyed-task cases, and keep plugin SDK return shapes plus bundled mocks aligned.
