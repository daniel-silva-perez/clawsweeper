## Summary

Keep this issue open. Current main still matches the reported gap: `memoryFlush` is invoked before the main turn, while completed runtime/Pi auto-compactions are only observed and counted after the embedded run path reports compaction events. The docs also promise memory flush before compaction, so this is a real behavior/docs mismatch rather than stale or out of scope.

## Best Possible Solution

Keep the issue open and fix the runtime/Pi compaction path. The strongest product fix is to wire a Gateway-side pre-compaction memory flush into runtime-triggered compaction, likely via the Pi `session_before_compact` hook or an explicit embedded-run pre-compact callback, then add an integration regression test where a normal user turn emits a completed compaction event and `memoryFlushCompactionCount` is advanced for that same cycle. If a true pre-compact flush is not feasible, add a documented post-compaction audit/flush fallback and update `docs/concepts/memory.md`, `docs/concepts/compaction.md`, and `docs/reference/session-management-compaction.md` to match the actual guarantee.
