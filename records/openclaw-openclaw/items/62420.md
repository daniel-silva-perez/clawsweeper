## Summary

Keep open. Current main still matches the report: the memory flush helper runs before the main embedded turn, while actual embedded auto-compaction can be observed and persisted after the attempt without a second flush/audit. The docs still describe memory flush as pre-compaction, per-cycle protection, so this is a real behavior mismatch rather than a stale or docs-only report.

## What This Changes

The issue asks OpenClaw to ensure enabled compaction memory flushes are not bypassed by runtime-triggered in-turn auto-compaction.

## Best Possible Solution

Add a narrow runtime guard so an enabled memory flush cannot leave a session with memoryFlushCompactionCount behind a completed embedded auto-compaction. Prefer an awaitable pre-compaction hook if the Pi API supports it; otherwise add a best-effort post-compaction flush/audit after the run and document the exact guarantee.

## Reproduction Assessment

Yes. A focused unit test can mock an embedded run that reports agentMeta.compactionCount while the pre-turn flush did not run, then assert current main increments compactionCount without updating memoryFlushCompactionCount or invoking the flush helper. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

Yes for keeping this open and queueing a focused fix. A docs-only answer would be weaker because public docs and config already describe memoryFlush as compaction protection; the maintainable fix is to connect the existing memory flush helper to actual completed auto-compaction accounting without changing CLI/read-only skip semantics.
