## Summary

Keep open. Current main still has only adjacent queueing, plugin-hook, and explicit `/btw` behavior; it does not implement the proposed pre-turn classifier, and the current hook API does not expose the prior assistant question text plus outbound timestamp needed for a clean external plugin implementation.

## What This Changes

Requests an optional pre-turn context-staleness middleware or documentation path that uses timestamp and intent heuristics to avoid treating delayed user messages as answers to the last assistant question.

## Best Possible Solution

Define a config-gated turn-admission design, either in core or through a versioned plugin hook, that receives last assistant question text, outbound timestamp, inbound text/timestamp, route/session metadata, and can allow, clarify, ignore, or treat the message as a side note. It should cover web chat plus at least one external channel and document interaction with queue modes and `/btw`.

## Reproduction Assessment

Not applicable: this is a feature/API proposal rather than a concrete shipped bug report. Source inspection gives a high-confidence current-main check that no automatic staleness classifier exists today.

## Solution Assessment

Unclear. The standalone module is useful prototype evidence, but directly wiring its heuristics into middleware is not clearly the best maintainable fix until maintainers decide whether this belongs in core turn admission or behind a stable plugin API seam.
