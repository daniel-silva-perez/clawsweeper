## Summary

Keep open. Current main still has adjacent mechanisms for active-run queueing, explicit `/btw` side questions, and plugin interception, but no turn-admission check that can compare the prior assistant question and outbound timestamp against the new inbound message before treating it as an answer. The linked standalone module is useful prototype context, but this needs maintainer product/API direction before a conservative cleanup close or autonomous fix PR.

## What This Changes

Add an optional pre-turn context-staleness or answer-intent check before treating a delayed inbound user message as the reply to the last assistant question.

## Best Possible Solution

Define a config-gated turn-admission design that can see the last assistant question text, outbound timestamp, inbound text/timestamp, session, and channel metadata, then return allow, clarify, ignore, or side-note handling. If maintainers prefer plugin-first, add a versioned hook/API seam with that context and document how it interacts with `/btw`, active-run queue modes, and external channels.

## Reproduction Assessment

Not applicable as a feature/API proposal rather than a concrete shipped regression. The high-confidence verification path is source and docs inspection, which shows current main has no automatic prior-question/inbound-message staleness classifier; a future fix should add a deterministic delayed-message scenario.

## Solution Assessment

Unclear. The standalone module is useful prototype evidence, but directly wiring those exact heuristics as middleware is not clearly the narrowest maintainable solution until maintainers choose core turn admission versus a public plugin seam.
