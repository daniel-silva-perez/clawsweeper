## Summary

Keep open. Current main has adjacent active-run queueing, inbound/plugin hooks, and explicit /btw side-question behavior, but it still does not implement an automatic pre-turn check that compares a delayed inbound message with the last assistant question before treating it as an answer. The remaining work needs maintainer product/API direction, especially because the existing plugin hook surface does not expose prior outbound-question context.

## What This Changes

The issue proposes an optional turn-admission middleware that uses timestamp and intent heuristics to avoid treating delayed side comments as answers to the assistant's last question.

## Best Possible Solution

Define a config-gated turn-admission design, either in core or as a public plugin hook, that receives the last assistant question text, outbound timestamp, inbound text/timestamp, route/session metadata, and can allow, clarify, ignore, or treat the message as a side note. Cover web chat and at least one external channel, and document how it interacts with /btw and active-run queue modes.

## Reproduction Assessment

Not applicable: this is a feature/API proposal rather than a concrete shipped bug report. The focused verification path is source inspection of current queueing, plugin hooks, and /btw behavior, which shows no automatic context-staleness classifier on main.

## Solution Assessment

Unclear. The standalone module is useful prototype evidence, but directly wiring those heuristics as middleware is not clearly the best maintainable solution until maintainers decide whether the behavior belongs in core turn admission or behind a versioned plugin API seam.
