## Summary

Keep open. Current main has adjacent queueing and explicit `/btw` side-question behavior, but it still lacks an automatic pre-turn check that compares a delayed inbound message against the last assistant question before treating it as an answer; the remaining work needs product/API direction rather than cleanup closure.

## What This Changes

The issue proposes an optional pre-turn context-staleness classifier that checks whether a delayed inbound message should be treated as an answer to the assistant’s last outbound question.

## Best Possible Solution

Define a small config-gated turn-admission design: either a core pre-turn classifier or a versioned plugin hook that receives last assistant question text, outbound timestamp, inbound text/timestamp, route/session metadata, and can allow, clarify, or treat the message as a side note. Cover web chat plus at least one external channel path, and document how it interacts with `/btw` and active-run queue modes.

## Reproduction Assessment

Not applicable: this is a feature/API proposal, not a concrete shipped bug report. The focused verification path is source inspection of turn queueing, `/btw`, and plugin hook contracts, which shows no current automatic context-staleness classifier.

## Solution Assessment

Unclear. The standalone module is useful prototype evidence, but directly wiring that exact heuristic as middleware is not clearly the best maintainable solution until maintainers decide whether this belongs in core turn admission or a public plugin API seam.
