## Summary

Keep open. Current main has adjacent protections for explicit side questions and active-run queueing, but I found no automatic pre-turn context-staleness check that compares the latest inbound message with the last assistant question before treating it as the next normal user turn. The existing plugin hooks also do not clearly expose enough last-outbound-question context to close this as external ClawHub-only work.

## What This Changes

The issue proposes optional pre-turn middleware that uses timestamp and intent heuristics to detect when a late inbound user message should not be treated as the answer to the last assistant question.

## Best Possible Solution

Define a small, config-gated turn-admission design: either a core pre-turn classifier or a new plugin hook that receives the last assistant question, outbound timestamp, inbound timestamp, inbound text, and route/session metadata, then can allow, clarify, or treat the message as a side note. Cover web chat and at least one channel path with focused tests, and document how it interacts with `/btw` and queue modes.

## Reproduction Assessment

Not applicable as a normal bug reproduction; this is a feature/design proposal. The focused verification path is source inspection of busy-run queueing, BTW, and plugin hook contracts, which shows no current automatic context-staleness classifier.

## Solution Assessment

Unclear. The proposed standalone Node module may be a good prototype, but wiring it directly as middleware is not clearly the best maintainable solution until maintainers decide whether this belongs in core turn admission or a versioned plugin API seam.
