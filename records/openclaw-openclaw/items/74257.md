## Summary

Keep open. The combined Telegram topic, exec approval, and context-overflow compaction leak is not proven fixed on current main: approved exec completions still resume through the normal Gateway agent path, overflow recovery still retries the prompt after compaction, and the existing guardrails cover narrower HEARTBEAT_OK stripping or silent final cases rather than the reported composed lifecycle.

## What This Changes

The issue asks OpenClaw to prevent Telegram topic exec approval followups and context-overflow compaction retries from rendering HEARTBEAT_OK or internal async-completion text to users.

## Best Possible Solution

Add a first-class internal async/no-op delivery state that survives exec approval followups, Gateway agent resume, heartbeat/system-event handling, and context-overflow compaction retries, then pin it with a Telegram topic regression that proves no HEARTBEAT_OK or internal orchestration prompt is delivered to users.

## Reproduction Assessment

Unclear: the issue has concrete logs but no high-confidence minimal reproduction steps. The best reproducible path would be a focused regression test that simulates a Telegram topic exec approval followup entering overflow compaction and asserts that internal/no-op payloads are consumed silently.

## Solution Assessment

Unclear as an implementation because there is no patch here. The narrow maintainable direction is a runtime delivery-state contract; output token stripping alone is not enough because the current code still routes approved exec followups through normal agent delivery and retries prompts after compaction.
