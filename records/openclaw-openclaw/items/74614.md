## Summary

Keep open. Current main still contains the reported early return before the terminal chat final event, the registry contract allows `shift()` to return `undefined`, and the related fix PR was closed without merging, so no cleanup close reason applies.

## What This Changes

The issue asks for gateway chat lifecycle finalization to fall back to the already-resolved session/run context and still emit a final chat event when `registry.peek()` succeeds but `registry.shift()` misses.

## Best Possible Solution

Update the gateway lifecycle finalizer so a `shift()` miss falls back to the resolved `sessionKey` and client/event run id, then add a focused regression test for that branch and keep broader timeout/no-terminal-event behavior tracked separately unless the same test proves it fixed.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to enter `finalizeLifecycleEvent` with `chatLink` truthy while forcing `chatRunState.registry.shift(evt.runId)` to return `undefined`; current main then returns before emitting the terminal `chat` event.

## Solution Assessment

Yes. The narrow producer-side fallback is the best current fix because it restores the terminal chat event at the gateway boundary instead of adding UI/TUI cleanup after the final event has already been suppressed.
