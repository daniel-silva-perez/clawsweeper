## Summary

Keep this issue open. Current main still contains the reported gateway lifecycle branch where `peek()` can resolve a chat link, `shift()` can miss, and the handler returns before emitting the terminal chat event; the linked fix PR was closed unmerged and no open PR currently references this issue.

## What This Changes

The issue reports that gateway chat finalization can leave Control UI/TUI clients stuck in a streaming or running state when a terminal lifecycle event sees a chat registry entry during `peek()` but loses it before `shift()`.

## Best Possible Solution

Land a narrow gateway fix that uses the already-resolved `sessionKey` and `eventRunId` fallback when `registry.shift()` misses after a successful `peek()`, with a regression test proving a terminal lifecycle event still emits exactly one final chat event and still runs the normal cleanup path.

## Reproduction Assessment

Yes. Current source contains the exact early-return path described, and a focused Vitest can reproduce it by adding a chat registry entry, making `shift()` return `undefined`, emitting assistant text followed by lifecycle `end`, and observing that no final chat event is emitted on current main.

## Solution Assessment

Yes. The narrowest maintainable fix is to reuse the existing no-chatLink fallback finalization path for shift misses while preserving `skipChatErrorFinal`, abort handling, final cleanup, and duplicate-final protection.
