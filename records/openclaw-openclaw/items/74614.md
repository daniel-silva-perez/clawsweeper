## Summary

Keep open. Current main still has the terminal lifecycle early return that would suppress `emitChatFinal` if `registry.shift(evt.runId)` misses after a visible `chatLink`, and the only exact fix PR found, #74678, is closed and unmerged. The source-level gap is clear, but the natural live trigger still needs care because the current in-memory registry is synchronous.

## What This Changes

The issue asks the gateway terminal lifecycle handler to emit a final chat event through the resolved session/run fallback when a chat registry link was visible but `registry.shift` returns no entry after streamed output.

## Best Possible Solution

Patch `finalizeLifecycleEvent` so a `chatLink` plus missing `finished` entry falls back to the already resolved `sessionKey` and `eventRunId`, then continues the normal cleanup, persistence, and sequence clearing path. Add a regression in `server-chat.agent-events.test.ts` that mocks or injects the registry shift miss after a successful peek, and add a user-facing changelog fix entry with proper reporter/source credit.

## Reproduction Assessment

Yes for a focused unit-level reproduction: inject or spy a chat registry where `peek(evt.runId)` returns a `ChatRunEntry` and `shift(evt.runId)` returns undefined, then send lifecycle `phase: "end"` and observe no final chat broadcast on current main. Natural live reproduction remains intermittent and is not fully established by the current synchronous registry implementation.

## Solution Assessment

Yes, the gateway fallback is the narrowest maintainable fix if maintainers accept the defensive branch: it finalizes at the source of the terminal event instead of adding another client-side watchdog. The safer implementation should avoid duplicate finals and should not broaden hidden-run visibility or aborted-run behavior.
