## Summary

Keep open. Current main still lacks first-class SSH transport for the node host: the CLI, persisted node-host gateway config, foreground runner, and installed-service argument builder all model only direct host/port/TLS connections. The reported reverse-proxy challenge-auth path has current source and unit coverage for remote token resolution, but the nginx/podman scenario was not live-reproduced, so the remaining transport/auth work needs maintainer design and security review rather than cleanup closure.

## What This Changes

The issue requests first-class SSH transport support for `openclaw node run` and a fix for node-host challenge auth when connecting through a TLS reverse proxy.

## Best Possible Solution

Implement a reviewed node-host SSH transport path that reuses the existing SSH port-forward helper for foreground and installed-service flows, connects `GatewayClient` only to the local loopback tunnel endpoint, preserves remote token/password and device-pairing semantics, updates node docs, and coordinates endpoint-security behavior with #26250.

## Reproduction Assessment

Yes for the remaining SSH-transport gap: current source and docs show node-host foreground and service flows use only direct host/port/TLS settings and document manual external SSH tunnels. Unclear for the reverse-proxy challenge-auth failure on current main because the review verified source/tests but did not run an nginx/podman live reproduction.

## Solution Assessment

No, the current direct-only node-host path is not the best complete solution for the reported workflow. The maintainable fix is node-host-owned SSH tunnel lifecycle with explicit loopback endpoint validation and credential/device-pairing coverage, reviewed alongside the related security issue.
