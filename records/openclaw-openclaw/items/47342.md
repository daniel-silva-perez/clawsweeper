## Summary

Keep open. Current main appears to have improved the remote credential and device-pairing path, but the central request is still unresolved: `openclaw node run` and installed node services still connect by direct host/port WebSocket arguments, while first-class SSH tunnel lifecycle exists only in gateway status/probe paths and the public node docs still tell users to manage an SSH tunnel externally.

## What This Changes

The issue requests first-class SSH transport support for the node host and a fix for node-host challenge authentication when connecting through a TLS reverse proxy.

## Best Possible Solution

Implement a reviewed node-host SSH transport that reuses the existing SSH port-forward helper for both foreground and installed-service flows, validates the local tunnel endpoint rather than the remote SSH target, preserves shared-token and device-pairing semantics, updates node docs, and coordinates with #26250.

## Reproduction Assessment

Yes for the remaining SSH-transport gap: current source shows node-host foreground and service paths accept only direct host/port/TLS settings and construct a direct WebSocket URL. The reverse-proxy challenge-auth symptom is only source/test assessed here, not live-reproduced against nginx/podman.

## Solution Assessment

No, the current state is not the best complete solution because it still relies on external tunnel management for node hosts. The maintainable fix is a node-host-owned transport path with explicit tunnel lifecycle and security checks, not closing this as already implemented.
