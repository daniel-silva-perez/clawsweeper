## Summary

Keep open. Current main has improved the remote credential and device-pairing side, but the central node-host SSH transport request is still unresolved: foreground node hosts and installed services still run through direct WebSocket host/port arguments, while docs continue to prescribe an external SSH tunnel.

## What This Changes

The issue requests first-class SSH transport support for node-host runs and installed node services, plus correct challenge authentication when a node connects through a TLS reverse proxy.

## Best Possible Solution

Implement a reviewed node-host SSH transport that reuses the existing SSH port-forward helper for both foreground and installed-service flows, validates the local tunnel endpoint rather than the remote SSH target, preserves shared-token/device-pairing semantics, updates node docs, and coordinates with #26250.

## Reproduction Assessment

Yes for the remaining SSH-transport gap: current source shows `openclaw node run` and service install paths accept only direct host/port/TLS settings and construct a direct WebSocket URL. The reverse-proxy challenge-auth half is only source/test verified here, not live-reproduced against nginx/podman.

## Solution Assessment

No, the current state is not the best complete solution because it still relies on external tunnel management or an app-managed tunnel. The maintainable fix is a node-host-owned transport path with explicit tunnel lifecycle and security checks, not closing this as already implemented.
