## Summary

Keep this issue open. Current main appears to have improved the reverse-proxy/device-pairing side by resolving remote-mode node-host credentials and creating visible device pairing requests, but the main requested capability remains missing: `openclaw node run` and installed node-host services still use a direct host/port WebSocket path and do not manage `gateway.remote.transport: "ssh"`, `sshTarget`, or `sshIdentity`.

## Best Possible Solution

Keep this issue open. Preserve the current remote-token and device-pairing improvements, then design a reviewed node-host SSH transport that reuses the existing SSH tunnel helper for both foreground `node run` and installed services, validates the local tunnel endpoint rather than the remote SSH target, keeps shared-secret/device-pairing semantics intact, updates node docs, and coordinates with #26250.
