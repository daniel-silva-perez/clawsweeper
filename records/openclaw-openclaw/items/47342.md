## Summary

Keep open. Current main appears to have improved the remote token/signature-auth path, but the central request is still not implemented: `openclaw node run` and `openclaw node install` still connect directly by host/port and do not honor `gateway.remote.transport: "ssh"`, `gateway.remote.sshTarget`, or `gateway.remote.sshIdentity`. The docs still prescribe a manually managed SSH tunnel for remote node hosts.

## Best Possible Solution

Keep this issue open and narrow the remaining work to first-class node-host SSH transport. Implement node run/install support that honors `gateway.remote.transport: "ssh"`, `gateway.remote.sshTarget`, and `gateway.remote.sshIdentity`, reuses `startSshPortForward`, connects the node host to the local forwarded endpoint, and preserves the existing remote token/password plus device-pairing flow. Add focused tests for both foreground and supervised node-host planning.
