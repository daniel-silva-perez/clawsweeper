## Summary

Keep open. Current main still does not provide first-class SSH transport for the node host: foreground and installed-service flows accept only direct host/port/TLS settings and build a direct GatewayClient URL. The remote-token challenge-auth path has current source/test coverage, but the nginx reverse-proxy scenario was not live-reproduced here, so the remaining SSH transport work needs maintainer design/security review rather than closure.

## What This Changes

The issue requests `openclaw node run` and installed node-host services to support `gateway.remote.transport: "ssh"` while preserving token-backed challenge auth for non-loopback `wss://` reverse-proxy connections.

## Best Possible Solution

Implement a reviewed node-host SSH transport path that reuses the existing SSH port-forward helper for both foreground and installed-service flows, connects `GatewayClient` only to the local loopback tunnel endpoint, preserves remote token/password and device-pairing semantics, updates node docs, and coordinates the endpoint-security model with #26250.

## Reproduction Assessment

Yes for the remaining SSH-transport gap: current source and docs show node-host foreground and service flows use direct host/port/TLS settings and document external SSH tunnels. The reverse-proxy challenge-auth failure is unclear for current main because it was source/test assessed, not live-reproduced against nginx/podman.

## Solution Assessment

No, the current state is not the best complete solution because users still need external tunnel management for headless node hosts. The narrow maintainable solution is node-host-owned SSH tunnel lifecycle with explicit loopback endpoint validation and credential/device-pairing coverage.
