## Summary

Keep open. Current main has improved the reverse-proxy/device-pairing side by resolving remote-mode node-host credentials and creating/reporting device pairing requests, but the central request remains unresolved: `openclaw node run` and `openclaw node install` still have no first-class SSH transport and still run through a direct WebSocket host/port path.

## Best Possible Solution

Keep this issue open. Preserve the current remote-token and device-pairing improvements, then design a reviewed node-host SSH transport that reuses the existing SSH tunnel helper for both foreground `node run` and installed services, validates the local tunnel endpoint correctly, keeps shared-secret/device-pairing semantics intact, updates docs, and coordinates with #26250.
