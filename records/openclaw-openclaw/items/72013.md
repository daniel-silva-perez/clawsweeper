## Summary

Keep open. Current main still lets persisted ACP one-shot records with pending identities and stable runtime ids enter startup reconciliation, so the reported warning can still occur for historical terminal metadata. The linked focused PR #72023 targeted this gap but is closed unmerged, and the shipped/current partial fixes only cover backend readiness and post-turn one-shot identity cleanup.

## What This Changes

Startup identity reconciliation should skip or quietly handle terminal one-shot or closed ACP session records instead of warning on stale pending identity metadata.

## Best Possible Solution

Add a narrow startup-reconcile guard in `AcpSessionManager.reconcilePendingSessionIdentities` that skips persisted one-shot ACP records before incrementing `checked` or touching the runtime, while preserving normal persistent pending-identity reconciliation. Add focused regression coverage for pending one-shot records with stable ACPX ids, and handle any historical raw closed-state metadata only through an explicit compatibility predicate or schema follow-through.

## Reproduction Assessment

Yes. A persisted ACP session with `acp.mode: "oneshot"`, `identity.state: "pending"`, and `identity.acpxSessionId` currently passes the startup reconcile predicate and increments `checked`, which then drives the gateway warning when reconciliation cannot resolve it.

## Solution Assessment

Yes. The issue’s suggested skip is the narrowest maintainable fix for the noisy startup path; post-turn identity reconciliation helps future successful runs but does not protect existing stale or failed one-shot records already in the session store.
