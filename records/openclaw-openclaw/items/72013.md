## Summary

Keep open. Current main has a partial ACP oneshot cleanup for newly completed runs, but startup reconciliation still scans persisted ACP records based on pending identity plus a stable session id, without excluding `mode: "oneshot"`. Historical pending one-shot records like the report describes can still reach the warning path. The related PR #72023 targeted this exact gap but is closed unmerged in the provided GitHub context.

## Best Possible Solution

Keep this issue open and land an equivalent to the closed PR #72023: skip persisted `mode: "oneshot"` ACP records in `AcpSessionManager.reconcilePendingSessionIdentities` before incrementing `checked` or touching the runtime, preserve normal persistent pending identity reconciliation, and add focused regression tests for skipped oneshot records. If historical raw `state: "closed"` metadata is still real, support it with a narrow legacy/raw predicate or type/schema follow-through.
