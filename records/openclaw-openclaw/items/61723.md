## Summary

Keep this issue open. Current main has adjacent hardening for Gateway reconnects, startup-unavailable `chat.history` retries, and graceful TUI terminal cleanup, but I did not find source, test, docs, release, or live evidence proving the reported first-render hang after Gateway restart or the alternate `kill -9` terminal-state path is fixed. The report is recent and has a second Ubuntu confirmation, so closing would overclaim.

## Best Possible Solution

Keep this issue open and narrow the next maintainer step to a TUI lifecycle regression proof: run `openclaw tui` across a managed Gateway restart and after forcibly killing a previous TUI in a pseudo-terminal or WSL2/Linux session. If it still reproduces, fix the TUI startup path so it resets/recovers terminal modes before first render and renders or fails visibly even when Gateway/plugin initialization logs occur; if it no longer reproduces, close later with explicit current-main reproduction evidence and the exact fixing commit or release.
