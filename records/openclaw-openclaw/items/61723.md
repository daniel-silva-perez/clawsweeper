## Summary

Keep open. Current main has Gateway client reconnect handling for an already-running TUI, but it does not yet show a startup recovery path for a fresh `openclaw tui` after stale terminal state from `kill -9` or a gateway restart; the report has concrete WSL2 steps plus an Ubuntu confirmation.

## What This Changes

The issue reports that `openclaw tui` starts but never renders after a gateway restart or after an ungraceful TUI kill, leaving plugin logs and high-memory live processes.

## Best Possible Solution

Make `openclaw tui` robust against stale terminal state by normalizing the terminal before TUI config/backend initialization and ensuring users see a visible connecting/recovering UI even if the Gateway is restarting or plugin validation emits logs. Keep the existing Gateway reconnect behavior and add focused regression tests around TUI launch order and terminal restore.

## Reproduction Assessment

Yes. The issue gives high-confidence manual repro steps for WSL2/systemd and `kill -9`, with a second Ubuntu confirmation; this review did not execute the live repro because the task is read-only.

## Solution Assessment

Unclear for current main. The existing Gateway reconnect timer is useful for an already-running client, but the narrow maintainable fix for this report appears to be TUI startup terminal recovery and first-render ordering rather than broad Gateway or plugin-loader changes.

Fix the TUI no-render recovery reported in https://github.com/openclaw/openclaw/issues/61723.
