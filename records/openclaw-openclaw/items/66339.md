## Summary

Keep this issue open. Current main still has useful QMD mitigations, including read-only SQLite busy timeouts, boot-update retry, embed locking, and a recent QMD open-failure cooldown, but those do not solve the reported gateway-process versus CLI-process contention. The gateway can keep a full QMD manager doing background maintenance while `openclaw memory search` creates a separate one-shot CLI QMD manager for the same agent; update serialization and pending-update waits are process-local or instance-local.

## Best Possible Solution

Fix this in the memory-core QMD owner path. Add bounded per-agent cross-process coordination around QMD update/embed maintenance and foreground CLI search/index for the same agent index, with stale-lock handling and a wait/skip/read-only outcome for `openclaw memory search`. Add a regression test that models gateway background QMD work contending with a separate CLI search process or process-equivalent manager for the same agent.
