## Summary

Keep open. Current main still routes Teams HTML `<img>` attachments through the normal inline-image placeholder path, so a Teams emoji CDN image can still become `<media:image>` context. The PR targets the right Microsoft Teams attachment helper, and the latest force-push addresses the earlier review concerns around mixed non-HTML attachments, host parsing, and src/alt regex divergence; it needs maintainer review and current-main validation rather than cleanup closure.

## What This Changes

The PR adds Microsoft Teams attachment parsing and tests so pure Teams emoji CDN HTML image tags return their Unicode `alt` text instead of an image placeholder.

## Best Possible Solution

Land a rebased version of the Teams-plugin-local helper that recognizes only pure Teams emoji CDN HTML image attachments, preserves mixed-content fallback and current inline-media limits, and keeps the regression tests near the attachment placeholder helpers.

## Reproduction Assessment

Yes. A code-level reproduction is clear on current main: a `text/html` attachment containing a Teams CDN `<img src=... alt=...>` is counted by `extractInlineImageCandidates()` and causes `buildMSTeamsAttachmentPlaceholder()` to return `<media:image>` when no text is present.

## Solution Assessment

Yes, with a current-main rebase. Handling this inside the Microsoft Teams attachment placeholder path is the narrowest maintainable fix, and using parsed URL hosts plus fallback for mixed/non-HTML content avoids the review regressions already discussed.
