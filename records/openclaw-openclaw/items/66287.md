## Summary

Current main still has the reported memory-core behavior: default dreaming promotion requires repeated signal and query/day diversity before writing to MEMORY.md, and the test suite explicitly preserves one-off high-scoring recalls being excluded. Manual CLI threshold overrides and promote-explain help inspect or force a one-off run, but they are not a durable priority lane for important single-event feedback.

## Best Possible Solution

Keep this open for memory-core maintainer/product design. The best implementation is a tested durable lane for high-priority single-event candidates, such as explicit remember-directive metadata, a manual priority flag, or a critical-feedback classifier, with narrow bypass semantics for the signal and diversity gates in extensions/memory-core/src/short-term-promotion.ts. Update memory-core tests plus https://docs.openclaw.ai/concepts/dreaming and the memory CLI docs once the behavior is defined.
