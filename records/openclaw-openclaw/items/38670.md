## Summary

Keep open. Current main still has global-only internal hook registration and dispatch, and it does not expose the proposed `hooks.internal.events`, `agents.defaults.hooks`, or `agents.list[].hooks` policy surfaces. The PR remains a coherent implementation candidate, but it changes core hook execution policy and has unresolved review concerns that need maintainer judgment before merge.

## Best Possible Solution

Keep this PR open for maintainer review. If the capability is accepted, update the PR to define strict schema/types for `hooks.internal.events`, `agents.defaults.hooks`, and `agents.list[].hooks`; pass config and resolved agent identity only for genuinely agent-scoped events; preserve global behavior for non-agent lifecycle hooks; document exact event matching or add wildcard semantics; and add focused tests for precedence, named hook gating, non-agent hook behavior, hot-path policy resolution, and backward compatibility. If maintainers reject the product direction, close it explicitly with that rationale.

## Risks / Open Questions

- The PR changes core internal hook execution policy and config schema, so maintainer product/security review is appropriate before merge.
- The proposed exact-match `events` allowlist can silently skip catch-all registrations such as `command` when only `command:new` is listed, unless that contract is documented or wildcard/prefix matching is added.
- Review comments indicate agent default hook policy may apply to non-agent gateway lifecycle events, potentially suppressing `gateway:startup` or shutdown hooks unexpectedly.
- The proposed resolver runs on dispatch paths including message hooks; caching or a clear performance decision is needed before accepting it on hot paths.
