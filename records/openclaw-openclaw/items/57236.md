## Summary

Current main does not implement this PR's intended behavior. The runner still forces a rebuild when `hasDirtySourceTree()` returns true, and existing tests/scripts still encode `dirty_watched_tree` as a build-triggering condition. This PR is also paired with the still-open same-author issue #57204, so the right cleanup path is maintainer review rather than auto-closing one side.

## Best Possible Solution

Keep this PR open as the implementation candidate for #57204. A maintainer should decide whether dirty watched source trees should fall through to mtime freshness in `resolveBuildRequirement()`; if yes, update the runner, the existing `dirty_watched_tree` expectations, and the gateway-watch regression helper together, then run targeted run-node coverage and the appropriate changed gate.
