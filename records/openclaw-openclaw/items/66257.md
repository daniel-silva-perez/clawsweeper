## Summary

Keep open. Current main still uses Gateway-first approval, and the proposed local-first path is not a safe drop-in because it bypasses Gateway-side caller-device authorization and skips the `device.pair.resolved` broadcast used by connected clients.

## What This Changes

The PR changes the devices CLI to approve local loopback pairing requests before calling the Gateway, suppress local fallback for explicit token/password auth, and update CLI tests for local-first and unknown-request fallback behavior.

## Best Possible Solution

Route this through maintainer/security review and revise the design so any direct local approval either remains behind the Gateway decision or faithfully mirrors Gateway authorization, event broadcast, and logging semantics, with tests for implicit loopback, explicit auth, unknown request IDs, and UI notification behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep Gateway authorization before local approval:** `src/cli/devices-cli.ts:168-172`
  - body: The new local-first branch approves from local pairing files before `device.pair.approve` can enforce the caller-device check. Because `approveLocalPairingRequest` supplies `operator.admin`, a paired non-admin local device-token session could approve another device's pending request instead of receiving the Gateway denial.
  - confidence: 0.88
- **[P2] Preserve pairing resolution broadcasts:** `src/cli/devices-cli.ts:168-172`
  - body: Returning after local approval skips the Gateway approval path that logs the approval and broadcasts `device.pair.resolved`. The Control UI reloads devices from that event, so other subscribers can stay stale after a CLI local-first approval unless the same side effect is emitted.
  - confidence: 0.85
