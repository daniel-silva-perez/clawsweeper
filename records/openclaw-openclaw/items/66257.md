## Summary

Keep open because current main has not implemented the PR's local-first approval behavior, and the proposed patch introduces concrete device-approval authorization and notification regressions that should block merge until maintainer review.

## What This Changes

The PR changes `openclaw devices approve` to try local loopback pairing approval before Gateway approval, suppresses local fallback for explicit token/password auth, and updates devices CLI tests for local-first and unknown-request behavior.

## Best Possible Solution

Keep the PR open and revise it so local approval either remains behind the Gateway decision or faithfully mirrors Gateway authorization and resolution side effects, with tests and docs covering implicit loopback, explicit auth, device-token ownership, unknown request IDs, and UI notification behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Preserve Gateway authorization before local approval:** `src/cli/devices-cli.ts:168-172`
  - body: The new local-first branch approves from pairing files before `device.pair.approve` can run. That path calls `approveLocalPairingRequest`, which grants `operator.admin` scopes, bypassing the Gateway's current non-admin device-token ownership check that denies approving another device's pending request. Keep Gateway authz in the approval path or mirror the same caller-device check before local approval.
  - confidence: 0.86
- **[P2] Preserve pairing resolution broadcasts:** `src/cli/devices-cli.ts:168-172`
  - body: Returning after local approval skips the Gateway approval method, which logs the approval and broadcasts `device.pair.resolved`. The Control UI reloads devices from that broadcast, so subscribers can stay stale after CLI local-first approvals. Emit an equivalent notification path or keep Gateway approval in the successful flow.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The PR changes a device approval path and can bypass Gateway-side device-token ownership authorization for implicit local loopback approvals.

Concerns:

- **[medium] Local-first approval bypasses Gateway authz:** `src/cli/devices-cli.ts:168`
  - body: The added local-first branch approves with `operator.admin` caller scopes before the Gateway can enforce its non-admin device-token restriction to the caller's own pending device, potentially allowing a paired local non-admin device session to approve a different requester.
