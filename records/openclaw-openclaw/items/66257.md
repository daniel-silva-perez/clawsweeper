## Summary

Current main still has gateway-first `devices approve` behavior and does not implement the PR's explicit `--token`/`--password` no-local-fallback guard. The PR remains a focused CLI/test implementation candidate, but it needs rebase/adaptation and maintainer judgment on the local-first approval semantics.

## Best Possible Solution

Keep this PR open as a focused implementation candidate. The maintainer path is to rebase it onto current main, preserve `readConnectPairingRequiredMessage`, decide whether local-first approval should emit gateway-equivalent log/broadcast behavior, keep the explicit token/password no-fallback guard if accepted, add regression coverage for direct local approve, unknown local request fallback to gateway approve, and explicit auth flags suppressing local fallback, then update the devices docs if behavior changes.

- The PR body claims explicit auth flags suppress local fallback, but the patch does not add a regression test for explicit `--token` or `--password`; docs would also need an update if that behavior lands.
