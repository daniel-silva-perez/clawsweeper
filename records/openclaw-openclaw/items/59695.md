## Summary

Keep open. Current main still uses SHA-1 in both changed call sites, and the PR is a narrow, correct cleanup; it is not obsolete or superseded. The PR body overstates repo-wide SHA-1 removal, but that does not make the actual two-line diff incorrect.

## What This Changes

The PR changes the MCP session catalog fingerprint helper and a media-understanding test fixture cache helper from `crypto.createHash("sha1")` to `crypto.createHash("sha256")`.

## Best Possible Solution

Land this small hash-algorithm cleanup after normal maintainer validation, or apply the same two replacements directly on main if the contributor branch cannot be used. No broader hashing migration is required for this PR.

## Reproduction Assessment

Not applicable as a cleanup PR rather than a runtime bug. The observable current-main state is high-confidence: `rg` and direct file inspection still show the two SHA-1 call sites this PR changes.

## Solution Assessment

Yes. For these two non-security cache/fingerprint values, replacing only the `createHash` algorithm string is the narrowest maintainable solution; callers consume opaque hex strings and do not depend on SHA-1 length.
