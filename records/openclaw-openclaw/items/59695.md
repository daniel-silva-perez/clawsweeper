## Summary

Keep this PR open. Current main still has the two SHA-1 call sites the PR targets, so the PR is not obsolete or implemented on main. The diff is narrow and appears functionally plausible, with the main follow-up being maintainer review of whether this cleanup should land as-is and whether the broad “remaining SHA-1” wording should be tightened.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to land the narrow SHA-256 substitution if maintainers want this cleanup, while tightening the description to say it covers the MCP runtime fingerprint and this test helper rather than all SHA-1 use in the repository. A focused regression check around MCP runtime recreation or existing changed-lane validation would be enough for this XS change.

## Risks / Open Questions

- The PR body’s claim that these are the two remaining SHA-1 call sites is too broad for current main; `rg` still finds SHA-1 in plugin code such as memory-wiki, QQ media upload, active-memory, and voice-call paths, some of which may be protocol- or compatibility-driven.
