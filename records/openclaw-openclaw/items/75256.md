## Summary

Keep this PR open. Current main still has the Mattermost draft-preview overwrite behavior that the linked issue reports, but the proposed patch is not merge-ready because its boundary split is not synchronized with the async draft lifecycle and its new public config shape is not aligned with the docs/generated config metadata.

## What This Changes

The PR adds Mattermost `streaming.mode`, resolves it per account, wires draft-preview boundary splitting at assistant/reasoning/tool transitions, and adds schema/account/draft-stream test coverage.

## Best Possible Solution

Keep the Mattermost-local feature direction, but make boundary splitting lifecycle-aware by flushing or awaiting in-flight draft work, or by adding a generation guard so late creates cannot rebind the next block to the old post. Then align the Mattermost docs/generated config metadata/changelog and add a delayed-client test for fast thinking-to-tool or partial-to-tool transitions.

## Reproduction Assessment

Yes. The user-facing bug can be reproduced with a Mattermost conversation that triggers tool calls and observing one preview post being rewritten; the PR-specific race can be reproduced with a delayed Mattermost client that receives `update("Thinking...")`, a boundary signal, and a tool update before the first create resolves.

## Solution Assessment

No, not as currently implemented. A Mattermost `block` preview mode is a narrow maintainable direction, but the current split operation should be async/lifecycle-aware and the public config/docs surfaces need to match before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Flush the draft before forcing a new post:** `extensions/mattermost/src/mattermost/draft-stream.ts:89`
  - body: `signalBoundary()` calls `forceNewMessage()` after content has only been enqueued. If a boundary arrives while the previous Mattermost create/edit is still pending, that in-flight request can restore the old post id after the reset, so the next update edits the prior block and `block` mode still overwrites phase content. Make the split wait for the prior update to flush or guard late creates by generation, and cover it with a delayed-client test.
  - confidence: 0.89
- **[P2] Align the public Mattermost streaming config:** `extensions/mattermost/src/config-schema-core.ts:116`
  - body: This adds a public `streaming` object to the Mattermost schema, but the checked-in docs/generated config metadata still expose scalar `channels.mattermost.streaming` or omit the field. Users and config UI can be led to a rejected or invisible config unless the docs/metadata/compat path are updated with the schema change.
  - confidence: 0.83
