## Summary

Keep open. The PR addresses a real gateway shutdown reliability problem, and current main has overlapping shutdown hardening, but the PR is not obsolete and is not safe to merge as-is: current main still does not provide the exact generic close-handler timeout this branch proposes, while the branch regresses HTTP close/restart behavior and leaves several timed-out promises able to reject later.

## What This Changes

The branch adds a 5s shutdown timeout helper and tests, then wraps gateway close-handler subsystem stops, WebSocket close, and HTTP server close in that timeout with optional warning logging.

## Best Possible Solution

Rebase or replace the PR with a small patch that preserves current main's shutdown result/warning model and HTTP/WebSocket close semantics, then adds a reusable bounded shutdown-step helper only where hung teardown can still block later cleanup. The helper should observe late rejections from losing promises and include focused tests for hung subsystem continuation, late rejection handling, and HTTP restart safety.

## Reproduction Assessment

Yes. The PR-introduced regressions can be reproduced with unit fakes: an HTTP server whose `close` callback arrives after the timeout shows early shutdown completion, and a promise that rejects after the timeout shows the late unhandled-rejection risk.

## Solution Assessment

No. The direction is useful, but this diff is not the best way to solve it because it treats an HTTP listener timeout as successful close and does not consistently observe timed-out promises; the safer path is to adapt current main's two-phase timeout/warning patterns.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep HTTP close pending through forced connection close:** `src/gateway/server-close.ts:185-189`
  - body: This races `httpServer.close()` against the 5s timeout and lets shutdown complete when the timeout wins. In the in-process restart path, `run-loop` starts the next gateway as soon as `close()` returns, so an HTTP listener that is still draining can keep the port bound and make restart fail with `EADDRINUSE`. Preserve the current two-phase behavior: grace period, force-close connections, then wait for the listener or fail the shutdown.
  - confidence: 0.9
- **[P2] Observe promises after timeout:** `src/gateway/server-close.ts:50-103`
  - body: `raceTimeout` leaves the losing promise dangling, but several call sites pass raw promises (`bonjourStop`, `canvasHost.close`, `canvasHostServer.close`, and `stopGmailWatcher`). If one times out and later rejects, that rejection occurs outside the surrounding `try/catch` and can trigger Node's unhandled-rejection path during shutdown. Attach a catch before racing or make the helper observe the losing promise, and add a late-rejection test.
  - confidence: 0.86
