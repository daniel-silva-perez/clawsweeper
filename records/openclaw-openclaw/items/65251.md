## Summary

Keep open. Current main supports bundle-declared LSP defaults, but not the requested zero-configuration project scan, PATH discovery, optional install/fix flow, user override semantics, update notices, or LSP health dashboard.

## What This Changes

Request automatic LSP server detection and configuration from project files and installed servers, with optional install/fix flows, overrides, update notices, and diagnostics.

## Best Possible Solution

Keep this open for product/API design. The maintainable end state should add safe project detection and PATH discovery, keep installs and repairs explicitly opt-in, support user overrides, and expose LSP health through CLI/UI diagnostics; if language-specific behavior should live outside core, first add and document a generic plugin or bundle API for dynamic workspace LSP configuration.

## Reproduction Assessment

Not applicable as a bug reproduction. The current-main gap is verifiable by reading the LSP bundle loader and runtime: they require enabled bundle config and do not scan workspace files, package manifests, editor settings, or PATH.

## Solution Assessment

No, the current bundle-backed LSP implementation is not the complete solution for this issue. The best path is a scoped API/product design that separates safe detection, explicit install authority, override behavior, and diagnostics rather than adding ad hoc core heuristics.
