## Summary

Keep this PR open. Current main still lacks the first-class Azure Claude onboarding path, but the proposed branch has concrete functional bugs in endpoint validation and generic token routing, and the provider ownership boundary still needs maintainer judgment.

## What This Changes

The PR adds a core `anthropic-azure-api-key` onboarding path with CLI flags, env vars, URL/resource normalization, auth-profile/config wiring, Anthropic docs, and focused onboarding/unit tests.

## Best Possible Solution

Land a rebased, provider-owned implementation in the appropriate Microsoft Foundry, Anthropic Azure, or new provider plugin surface. It should expose manifest auth-choice/CLI/env metadata, implement interactive and non-interactive setup through provider auth hooks, strictly validate `.services.ai.azure.com` resource labels and `/anthropic` paths, store auth-profile metadata, update provider docs, and cover URL, resource, env-var, generic `--token`, and SecretRef cases.

## Reproduction Assessment

Yes. The PR diff gives high-confidence focused reproductions: the new URL normalizer accepts invalid full-URL hosts and appends `/anthropic` to unsupported paths, and the `apiKey` token-provider route ignores the existing generic `--token` option.

## Solution Assessment

No. The current PR is useful source work, but a one-off core auth branch is not the narrowest maintainable solution now that provider plugins support manifest auth choices and setup hooks; the implementation also needs the validation and generic-token fixes before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Reject unsupported Azure endpoint paths:** `src/commands/anthropic-azure-utils.ts:59-60`
  - body: `normalizeAnthropicAzureBaseUrl` appends `/anthropic` to any full URL path that does not already end in that suffix. Inputs like `https://fabric-hub.services.ai.azure.com/anthropic/v1` are stored as `.../anthropic/v1/anthropic`, so onboarding succeeds with a malformed runtime base URL instead of failing fast.
  - confidence: 0.92
- **[P2] Validate Azure resource labels in URL input:** `src/commands/anthropic-azure-utils.ts:50-53`
  - body: The full-URL branch only checks that the hostname ends with `.services.ai.azure.com`. A value such as `https://fabric_hub.services.ai.azure.com/anthropic` bypasses the bare-resource validation and persists an invalid Azure resource host, pushing the failure to runtime.
  - confidence: 0.9
- **[P2] Honor generic token input for apiKey routing:** `src/commands/auth-choice.apply.api-providers.ts:177`
  - body: `--auth-choice apiKey --token-provider anthropic-azure --token ...` normalizes into this Azure branch, but the branch passes only `anthropicAzureApiKey` to the API-key helper. Existing scripted API-key flows that use the generic `--token` will be ignored and fall back to env or prompts.
  - confidence: 0.86
