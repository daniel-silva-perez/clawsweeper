## Summary

Keep this PR open. Current main still does not provide the requested Azure Claude AI Foundry onboarding path, and the proposed patch still has blocking functional review findings plus an unresolved provider-ownership decision before it should merge.

## What This Changes

The PR adds a core Azure Claude AI Foundry onboarding path with a new `anthropic-azure-api-key` auth choice, dedicated CLI flags and env vars, auth-profile/config wiring, Anthropic docs, and onboarding tests.

## Best Possible Solution

Land a rebased, provider-owned implementation in the appropriate Microsoft Foundry or Anthropic Azure provider surface. It should expose manifest auth choice and CLI/env metadata, implement interactive and non-interactive setup through provider auth hooks, strictly validate `.services.ai.azure.com` resource labels and `/anthropic` paths, store auth-profile metadata, update provider docs, and cover URL, resource, env-var, generic `--token`, and SecretRef cases in focused tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Reject unsupported Azure endpoint paths:** `src/commands/anthropic-azure-utils.ts:59-62`
  - body: Full URL inputs whose path does not already end exactly in `/anthropic` are accepted by appending the suffix. For example, `/anthropic/v1` is stored as `/anthropic/v1/anthropic`, so onboarding succeeds but runtime Anthropic requests use a malformed base URL. Reject unexpected path segments or canonicalize only known valid variants.
  - confidence: 0.92
- **[P2] Validate Azure resource labels in URL input:** `src/commands/anthropic-azure-utils.ts:53`
  - body: The URL branch only checks that the hostname ends with `.services.ai.azure.com`. A full URL such as `https://fabric_hub.services.ai.azure.com/anthropic` bypasses bare-resource character validation and is persisted even though Azure resource labels cannot contain underscores, pushing the failure to runtime instead of onboarding.
  - confidence: 0.9
- **[P2] Honor generic token input for apiKey routing:** `src/commands/auth-choice.apply.api-providers.ts:177`
  - body: `--auth-choice apiKey --token-provider anthropic-azure --token ...` normalizes into this branch, but the branch reads only `anthropicAzureApiKey`. That ignores the generic token supplied by existing scripted API-key flows and falls back to env or prompt, unlike the shared provider API-key helper behavior.
  - confidence: 0.86
