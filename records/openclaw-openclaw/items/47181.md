## Summary

Keep this PR open. Current main still lacks the exact Azure Claude AI Foundry onboarding path, but the branch is not ready to merge: it adds provider-specific core onboarding code despite the current plugin-owned auth-choice architecture, and the PR diff still has concrete URL-validation and generic-token routing bugs.

## What This Changes

The PR adds a core `anthropic-azure-api-key` onboarding choice with CLI flags, env-var handling, Azure Claude base URL normalization, auth-profile/config wiring, docs, and focused onboarding tests.

## Best Possible Solution

Land a rebased provider-owned implementation through the current plugin auth-choice and setup hooks, either in the Microsoft Foundry provider if it should own Claude deployments or in a separate Azure Claude provider plugin. It should expose manifest CLI/env metadata, implement interactive and non-interactive setup, strictly validate `.services.ai.azure.com` resource labels and `/anthropic` paths, support generic `apiKey` token routing, store auth-profile metadata through existing SecretInput semantics, update provider docs, and cover URL/resource/env/SecretRef cases.

## Reproduction Assessment

Yes. The PR diff gives high-confidence static reproductions: a URL such as `https://fabric-hub.services.ai.azure.com/anthropic/v1` is normalized to a malformed path, `https://fabric_hub.services.ai.azure.com/anthropic` bypasses bare-resource validation, and `--auth-choice apiKey --token-provider anthropic-azure --token ...` is routed to the Azure branch but ignored.

## Solution Assessment

No. The requested user problem is valid, but a new hard-coded core auth choice is not the narrowest maintainable fix now that provider plugins support manifest auth choices, setup hooks, and non-interactive auth delegation; the PR also needs the validation and generic-token fixes before it can be considered mergeable.
