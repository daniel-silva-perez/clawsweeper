## Summary

Keep open. Current main already implements the main cron `delivery.threadId` plumbing, schema, tests, and docs, but it still suppresses a same-chat failure destination when the primary announce delivery is topic-routed via `threadId`; the PR head contains useful follow-up source work for that remaining gap, but should be ported narrowly to current main rather than merged wholesale.

## What This Changes

The PR branch adds cron `delivery.threadId` typing, normalization, Gateway schema round-trip, isolated Telegram announce forwarding, forum-topic regression coverage, and failure-destination dedupe handling.

## Best Possible Solution

Port only the remaining failure-destination comparison behavior and focused regression coverage into current `src/cron/delivery-plan.ts` and `src/cron/delivery.test.ts`, preserving the already-shipped threadId plan/schema/docs support and leaving separate Telegram target-format work to its own thread.

## Reproduction Assessment

Yes. A source-level reproduction is an announce cron job with `delivery.channel='telegram'`, matching `to`/`accountId` in `failureDestination`, and primary `delivery.threadId='15'`; current `isSameDeliveryTarget` ignores the thread id and suppresses the failure destination as identical.

## Solution Assessment

Yes for the remaining behavior, but not by merging the stale branch wholesale. The maintainable path is a narrow current-main repair that ports the PR head's dedupe guard and regression test into the renamed delivery-plan files.
