## Summary

Keep open. Current main has absorbed the base cron `delivery.threadId` contract, normalization, gateway schema, and target-resolution plumbing, but it still lacks the PR head's failure-destination deduplication fix. `resolveFailureDestination()` can still treat a primary Telegram topic target and a same-chat no-thread failure destination as identical because `isSameDeliveryTarget()` ignores `delivery.threadId`.

## Best Possible Solution

Keep this PR open for maintainer review or a focused rebase. Preserve the threadId contract, normalization, gateway schema, and delivery-target plumbing already on main, then port the remaining `resolveFailureDestination()`/`isSameDeliveryTarget()` fix with a regression test proving that a primary same-chat Telegram topic target does not suppress a distinct no-thread failure destination.
