## Summary

Keep open. Current main already implements most of the original Telegram cron delivery.threadId path, but it still drops a same-chat failure destination for topic-routed announce jobs because the duplicate-target comparison ignores the primary delivery threadId. The PR remains useful as source material for that narrow current-main repair rather than as a broad stale merge.

## What This Changes

The PR branch adds cron delivery.threadId typing, normalization, Gateway schema round-trip, isolated Telegram announce forwarding, forum-topic regression coverage, and failure-destination dedupe handling.

## Best Possible Solution

Port the remaining failure-destination comparison fix and a focused regression test into current src/cron/delivery-plan.ts and src/cron/delivery.test.ts, preserving the already-shipped threadId plan/schema/docs support and leaving broader Telegram target-format work to separate threads.

## Reproduction Assessment

Yes. A source-level reproduction is an announce cron job with delivery.channel='telegram', delivery.to and failureDestination.to set to the same chat, delivery.accountId matching failureDestination.accountId, and primary delivery.threadId='15'; current isSameDeliveryTarget treats that failure destination as identical and returns null.

## Solution Assessment

Yes for the remaining behavior, but not by merging the stale branch wholesale. The narrow maintainable solution is to port the PR head's dedupe guard and regression coverage into the current delivery-plan files.
