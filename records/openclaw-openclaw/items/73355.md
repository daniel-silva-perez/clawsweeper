## Summary

Keep this PR open. Current main still has only the existing chained `setTimeout` cron scheduler and no independent stalled-chain watchdog, while this PR is explicitly paired with the open same-author cron stall report by closing syntax. The proposed direction is plausible, but it still needs normal maintainer review because the new catch re-arm path is not directly covered by the added test and the branch predates several recent cron changes on main. https://github.com/openclaw/openclaw/pull/73355

## What This Changes

The PR adds catch-handler re-arming and a five-minute cron watchdog interval, wires watchdog startup/shutdown state, and adds new watchdog/re-arm tests.

## Best Possible Solution

Keep this PR and the linked issue open until maintainers either land a rebased, directly tested watchdog/recovery implementation or choose a narrower scheduler design that gives cron an independently recoverable tick source without weakening existing timer-chain safeguards.

## Reproduction Assessment

Unclear. The linked report has concrete field evidence, including stale `nextWakeAtMs` after multi-day macOS gateway uptime and recovery after restart, but there is no deterministic short reproduction for the OS/timer stall; source inspection does confirm current main lacks the proposed recovery path.

## Solution Assessment

Unclear. The independent watchdog is a reasonable narrow safety net for a stalled `setTimeout` chain, but the `.catch` re-arm needs direct regression coverage and maintainer review against recent cron changes before this is clearly the best implementation.
