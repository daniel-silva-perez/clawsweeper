## Summary

Keep this issue open. Current main has retention-based cron run pruning and some `deleteAfterRun` cleanup paths, but it does not provide the requested completion-based ephemeral mode for recurring cron jobs or hook mappings. The reporter's comment correctly points at the relevant existing plumbing, but the missing public flag/schema/runtime wiring is still absent.

## Best Possible Solution

Add an explicit ephemeral/completion-cleanup option for isolated cron runs and hook agent mappings, wire it through the CronJob and HookMapping schemas/normalizers/docs, and after each run completes on success or error delete only the intended run session entry using the existing `sessions.delete`/transcript archival path. Cover recurring cron jobs, hook mappings, disabled delivery, and failure paths with focused tests; keep existing retention knobs as the fallback for non-ephemeral sessions.

## Risks / Open Questions

- This touches session deletion and transcript archival, so an implementation should avoid deleting shared/custom sessions that users expect to continue, especially for hook mappings with static session keys.
- The hook side needs an explicit transcript policy and tests for success, error, delivery disabled, and templated sessionKey cases.
