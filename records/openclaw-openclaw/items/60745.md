## Summary

Keep open. Current main still has retention-based cron run pruning and one-shot `deleteAfterRun` cleanup paths, but no first-class completion-based ephemeral mode for recurring isolated cron jobs or hook mappings. The previous ClawSweeper comment remains accurate after checking current main: the public schema/CLI/runtime wiring for `ephemeral` is still absent.

## What This Changes

Add an explicit ephemeral session mode for isolated cron jobs and hook agent mappings so owned run sessions are disposed immediately after completion instead of waiting for retention cleanup.

## Best Possible Solution

Add a public `ephemeral` option for owned isolated cron run sessions and hook agent mappings, delete only the exact run-owned session entry after success, skipped, or error completion, preserve cron run logs, and document the transcript policy while leaving retention settings as the non-ephemeral fallback.

## Reproduction Assessment

Yes. A high-confidence verification path is source-level: adding `ephemeral: true` to a hook mapping would hit the strict hook schema, cron CLI/schema have no equivalent flag, and completed isolated cron sessions still flow through the age-based reaper.

## Solution Assessment

Yes, the requested completion-cleanup mode is the right primitive for this workload if it is constrained to isolated/run-owned sessions. The safer alternative is not to extend existing `deleteAfterRun` blindly to arbitrary `session:*` targets; that would overlap the live-session deletion hazard in #69467.
