## Summary

Keep open. Current main still has source and test-covered paths where a REM phase report can be written while no `DREAMS.md` diary entry is created, and the docs describe Dream Diary generation as best-effort rather than a guaranteed artifact.

## What This Changes

The issue requests that successful memory-core REM runs create or update `DREAMS.md` so the Control UI Dreams diary is not empty after `memory/dreaming/rem/YYYY-MM-DD.md` is written.

## Best Possible Solution

Make the Dream Diary contract explicit. If a visible diary is guaranteed whenever REM produces usable report material, memory-core should append a deterministic fallback entry for unavailable, timed-out, or empty narrative runs; otherwise the docs and Dreams UI should clearly surface that phase reports may exist without diary entries.

## Reproduction Assessment

Yes. The source-level path is high-confidence: current code can write a REM phase report before diary generation, and tests cover cases where narrative generation returns without creating `DREAMS.md`.

## Solution Assessment

Unclear. A narrow fallback diary write is likely maintainable if `DREAMS.md` is intended as a guarantee, but current docs and tests encode best-effort behavior, so maintainer contract choice should come first.
