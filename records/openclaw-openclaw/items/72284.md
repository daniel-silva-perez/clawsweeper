## Summary

Keep open. Current main improves one fallback path by writing a local diary entry for request-scoped subagent runtime errors, but REM still writes phase reports/signals before optional best-effort diary generation, and several current paths can leave `DREAMS.md` absent while the Control UI reads only that diary file.

## What This Changes

The issue requests that a successful REM dreaming run create or update root `DREAMS.md` so the Control UI Dreams diary is not empty.

## Best Possible Solution

Make the Dream Diary contract explicit. If visible diary output is guaranteed whenever REM writes usable report material, memory-core should append a deterministic fallback entry for timeout, empty narrative, and unavailable narrative paths; otherwise the docs and Dreams UI should make phase-report-only runs clear.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: run REM with a report-writing storage mode and a narrative path that times out or returns no assistant text, and current code can leave `DREAMS.md` absent while `memory/dreaming/rem/YYYY-MM-DD.md` exists; current tests already cover no-file outcomes for no snippets and timeout.

## Solution Assessment

Unclear. A narrow fallback diary write would be maintainable if `DREAMS.md` is intended as a guarantee, but current docs and tests preserve best-effort no-write behavior, so maintainer contract choice should come first.
