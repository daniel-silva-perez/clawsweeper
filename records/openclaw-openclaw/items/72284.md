## Summary

Keep open. Current main still allows REM phase reports/signals to be written while Dream Diary generation remains optional, detached, and best-effort, so `DREAMS.md` can still be absent and the Control UI will render no diary content.

## What This Changes

The issue requests that successful REM dreaming runs create or update the root Dream Diary file so the Control UI Dreams tab has content.

## Best Possible Solution

Define the Dream Diary contract explicitly. If visible diary output is guaranteed after REM has usable material, memory-core should append a deterministic fallback entry for timeout, empty narrative, and unavailable narrative paths; otherwise docs and the Dreams UI should clearly surface phase-report-only runs instead of presenting an empty diary as a bug.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to run REM with separate report output and a narrative path that times out or returns no assistant text; current tests already cover `DREAMS.md` remaining absent for no snippets and timeout.

## Solution Assessment

Unclear. A narrow fallback diary write is maintainable if `DREAMS.md` is intended as a guarantee, but current docs and tests preserve best-effort no-write behavior, so maintainers should choose the contract before a repair PR changes it.
