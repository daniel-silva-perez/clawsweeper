## Summary

Keep open. Current main still permits REM phase reports and phase signals to be written while Dream Diary creation remains conditional and best-effort, and the Control UI still reads only `DREAMS.md`/`dreams.md` for diary content.

## What This Changes

The issue requests that memory-core create or update `DREAMS.md` after a successful REM phase so the Control UI Dreams tab shows diary content instead of an empty diary.

## Best Possible Solution

Define the Dream Diary contract explicitly, then either make memory-core append a deterministic fallback diary entry for REM runs with usable material or update the docs/UI to surface phase-report-only runs clearly instead of leaving the diary area empty.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to run REM so `writeDailyDreamingPhaseBlock` succeeds while the narrative path has no subagent, no usable snippets, a timeout/error, or no assistant text; existing tests already cover `DREAMS.md` staying absent for no snippets and timeout.

## Solution Assessment

Unclear. A narrow fallback diary write is maintainable only if `DREAMS.md` is intended to be guaranteed; current docs and tests still preserve best-effort no-write behavior, so the contract should be decided before changing code.
