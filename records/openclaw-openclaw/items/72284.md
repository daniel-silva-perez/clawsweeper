## Summary

Keep this issue open. Current main partially improves the Dream Diary path with a request-scoped fallback, but the REM phase report is still written independently from `DREAMS.md`, the diary append remains conditional and best-effort, and the Control UI still renders an empty diary when no `DREAMS.md` or `dreams.md` exists.

## What This Changes

The issue asks for memory-core REM dreaming to create or update `DREAMS.md` after a successful REM phase so the Control UI Dreams tab can display diary content.

## Best Possible Solution

Maintainers should decide the product contract explicitly. If a successful REM report with usable material must always surface in the Control UI, memory-core should write a deterministic fallback diary entry when the narrative subagent is absent, times out, or produces no text, with regression coverage for separate REM report plus diary creation; otherwise the docs and Dreams empty state should explain that phase reports can exist without a diary entry.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction path: current code can write the REM phase report before any diary attempt, and tests show `DREAMS.md` stays absent on timeout or no-snippet narrative paths. The reporter's exact installed environment was not live-reproduced.

## Solution Assessment

Unclear. The current request-scoped fallback is a good narrow fix for one failure mode, but it is not the full solution if the intended contract is that every successful REM report creates diary content; that broader contract needs maintainer confirmation before an automated repair PR.
