## Summary

Keep open. Current main partially improves the Dream Diary fallback path, but it still has source- and test-covered paths where a REM phase report can be written while no `DREAMS.md` diary entry is created, leaving the Control UI diary reader with no content.

## What This Changes

The issue asks memory-core to create or update `DREAMS.md` after successful REM runs so the Control UI Dreams tab shows Dream Diary content instead of staying empty.

## Best Possible Solution

Make the Dream Diary contract explicit in code and docs. If the UI-visible diary is intended to be guaranteed after REM writes usable report material, memory-core should append a deterministic fallback diary entry for no-entry, timeout, unavailable, and empty-narrative runs; otherwise the Dreams UI and docs should surface phase reports separately and explain that diary entries are best-effort.

## Reproduction Assessment

Yes. The high-confidence reproduction is source-level: current REM code can write a phase report before diary generation, and current tests assert no `DREAMS.md` for no-snippet and timeout narrative paths.

## Solution Assessment

Unclear. A narrow fallback diary write is likely maintainable if `DREAMS.md` is a product guarantee, but current docs and tests encode best-effort behavior, so maintainer contract review should come before an automated repair.
