## Summary

Keep open. Current main partially mitigates stale context display by treating `totalTokensFresh: false` as unknown, but the shared status formatter still does not use the latest compaction checkpoint `tokensAfter`, so it does not yet satisfy the requested post-compaction context display; the related open PR #72285 is the right implementation path to review.

## What This Changes

The issue asks for `/status` and post-message status cards to show the current post-compaction context size instead of stale pre-compaction totals.

## Best Possible Solution

Keep this open until #72285 is reviewed and either landed or replaced. The desired end state is a shared status formatter that prefers fresh totals, falls back to the latest valid compaction checkpoint `tokensAfter` when totals are stale, and has a focused regression test for stale post-compaction status output.

## Reproduction Assessment

Yes. A source-level reproduction is clear: pass `buildStatusMessage` a session entry with stale `totalTokens`, `totalTokensFresh: false`, and a latest compaction checkpoint with `tokensAfter`; current main will render unknown context instead of the checkpoint value because the formatter never reads the checkpoint.

## Solution Assessment

Yes, the open PR direction appears to be the narrow maintainable fix: use the existing freshness marker and compaction checkpoint data in the shared formatter, with a regression test. A broader migration or startup repair would be riskier than fixing the display fallback at the status boundary.
