## Summary

Keep open. Current main suppresses the stale pre-compaction total when `totalTokensFresh` is false, but it still renders unknown context instead of the requested post-compaction count from the latest checkpoint; the targeted fix PR #72285 remains open and unmerged.

## What This Changes

The issue requests that shared `/status` and post-message status output show the current post-compaction context size, with the pre-compaction peak retained only as compaction delta/checkpoint information.

## Best Possible Solution

Land PR #72285 or an equivalent narrow shared formatter change that preserves fresh totals, falls back to the latest valid checkpoint `tokensAfter` only when totals are stale or missing, and keeps regression coverage in the status formatter tests.

## Reproduction Assessment

Yes for the remaining gap. A high-confidence source-level reproduction is to call `buildStatusMessage` with `totalTokensFresh: false`, stale `totalTokens`, and a latest compaction checkpoint with `tokensAfter`; current main renders `Context: ?/limit` instead of that post-compaction value.

## Solution Assessment

Yes. The narrow maintainable solution is a shared status formatter fallback using the existing `SessionCompactionCheckpoint.tokensAfter` data, not a channel-specific workaround or broader session migration.
