## Summary

Keep open. Current main suppresses the stale pre-compaction total, but it still renders unknown context instead of the requested post-compaction token count because the shared status formatter does not fall back to the latest compaction checkpoint `tokensAfter`; the related open PR #72285 remains the targeted implementation path.

## What This Changes

The issue asks for `/status` and shared status cards to show the current post-compaction context size instead of a stale pre-compaction total.

## Best Possible Solution

Review and land PR #72285 or an equivalent narrow shared formatter change that prefers fresh totals, falls back to the latest valid checkpoint `tokensAfter` only when totals are stale or missing, and keeps regression coverage for the post-compaction status display.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call `buildStatusMessage` with stale `totalTokens`, `totalTokensFresh: false`, and a latest compaction checkpoint containing `tokensAfter`; current main renders `Context: ?/limit` instead of the checkpoint value.

## Solution Assessment

Yes. The narrow maintainable solution is a shared status formatter fallback using the existing freshness marker and checkpoint data, not a WhatsApp-specific workaround or a broader session migration.
