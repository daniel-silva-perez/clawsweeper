## Summary

Keep open. Current main has session-aware heartbeat routing, targeted wake handling, isolated heartbeat sessions, and newly added inferred commitments that can deliver through heartbeat in the same session scope, but it still does not provide native durable per-session heartbeat entries with their own schedules, enablement state, concurrency semantics, run history, or operator CRUD surface.

## What This Changes

The issue requests first-class native heartbeat entries scoped to explicit sessions so one agent can schedule and observe multiple proactive session-bound heartbeat runs.

## Best Possible Solution

Keep this as a core automation design issue. A complete solution should define whether heartbeat entries reuse cron’s durable scheduler/run-history substrate or get a separate versioned store, then add schema/API/CLI or UI surfaces, explicit skip/queue/parallel semantics, restart-safe entry state, session-bound execution guarantees, and docs that separate scheduling from gating, commitments, and cron.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is source and docs inspection: schema, scheduler state, RPC methods, and docs all show one heartbeat config/run path rather than durable per-session heartbeat entries.

## Solution Assessment

Unclear until maintainers choose the product boundary. The requested native entry model is a plausible core solution, but the safer design path should first decide how it composes with existing cron persistence, targeted wakes, and inferred commitments.
