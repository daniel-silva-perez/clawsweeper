## Summary

Keep this PR open. Current main and the latest release still have the reported custom-list behavior, and the proposed core utility change is narrow and appears functionally correct; the remaining maintainer decision is whether to fold in the copied browser helper and the small non-string filtering cleanup before landing.

## What This Changes

The PR updates `parseBooleanValue` to ignore malformed custom truthy/falsy options, compare custom entries case-insensitively, and add unit tests for those runtime input shapes.

## Best Possible Solution

Land a focused hardening patch for the shared helper, add the proposed regression tests, filter custom-list entries to strings before lowercasing for clarity, and either update the browser plugin copy in the same PR or record a clear follow-up decision.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence reproduction: `parseBooleanValue("yes", { truthy: ["YES"] })` misses because the input is lowercased but the custom set is not, and `parseBooleanValue("no", { falsy: { not: "an array" } })` reaches `new Set` with a non-iterable object.

## Solution Assessment

Mostly yes. The PR is the narrowest core fix for the reported utility behavior, but the best landing version should resolve the non-string filtering note and decide whether the copied browser helper should stay in sync.
