## Summary

Keep open. Current main still has a concrete Gmail hook setup/runtime mismatch that can prevent `hook:gmail:*` dispatch: the setup flow enables the Gmail preset, while the gateway requires explicit session-key policy for that preset's templated session key. The linked PR #60574 is useful context, but it was closed unmerged and its own ClawSweeper review says the token-removal premise did not match the two-hop Gmail/gog/OpenClaw auth model.

## What This Changes

Restore Gmail inbound hook-session creation and downstream Signal notification delivery after the reporter's update from OpenClaw 2026.3.28 to 2026.3.31.

## Best Possible Solution

Align the Gmail setup path, runtime policy, and docs so a setup-created Gmail configuration can safely create per-message `hook:gmail:*` sessions and deliver to configured channels. The likely fix is to have Gmail setup write the required constrained session-key policy when it enables the Gmail preset, or deliberately switch setup to a static-session mapping if maintainers do not want setup to enable templated hook session keys by default.

## Reproduction Assessment

Unclear for the full Gmail-to-Signal live regression because this review did not have the reporter's Gmail/PubSub/Signal environment. There is a high-confidence source-level reproduction for a current-main failure mode: setup enables the Gmail preset with a templated session key, while the gateway resolver requires missing session-key policy before that preset can dispatch.

## Solution Assessment

Unclear until maintainers choose the intended security posture for setup-created Gmail hooks. The narrowest maintainable direction is to make `openclaw webhooks gmail setup` produce a gateway-valid config that matches the documented Gmail preset policy, without changing the separate Pub/Sub push token and OpenClaw hook token auth hops.

## Repair Work Prompt
