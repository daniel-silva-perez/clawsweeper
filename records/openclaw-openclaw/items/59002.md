## Summary

Keep open. Current main still has deterministic Gmail hook setup blockers: setup-generated Tailscale endpoints can include `?token=` even though the Gateway rejects query-token auth, and the Gmail wizard enables a templated Gmail preset without writing the session-key policy that the current hook resolver requires. Related PR #60574 identified one root cause but was closed unmerged, so current main does not yet solve the reported loss of Gmail hook sessions and Signal delivery.

## What This Changes

Restore Gmail Pub/Sub hook ingestion so inbound mail creates `hook:gmail:<id>` sessions and configured Signal delivery after updating from 2026.3.28 to 2026.3.31.

## Best Possible Solution

Land a narrow Gmail hook setup fix that aligns all three contracts: generated Pub/Sub push endpoints never carry hook tokens in query strings, setup-generated Gmail preset config includes the minimum constrained session-key policy needed for `hook:gmail:<id>` dispatch, and mapped Gmail hooks can still deliver to configured channels such as Signal without duplicate fallback notifications.

## Reproduction Assessment

Yes for current-main source behavior: a setup-generated Gmail config enables the templated Gmail preset without the policy required by `resolveHooksConfig`, and Tailscale setup can still generate a push URL the Gateway rejects. The exact reporter environment still needs a live Gmail/Pub/Sub/Signal confirmation after the code fix.

## Solution Assessment

No current-main behavior is not the best solution because the documented setup path can still produce internally inconsistent hook configuration. The safer fix is a narrow setup/runtime contract repair with regression tests, not a broad change to Gateway hook auth or session policy.

Current main still has two deterministic setup/contract problems: `src/hooks/gmail-ops.ts` passes `pushToken` into `ensureTailscaleEndpoint`, and `src/hooks/gmail-setup-utils.ts` appends it as `?token=`, while the Gateway rejects query-token hook auth; also `openclaw webhooks gmail setup` enables the Gmail preset, whose session key is `hook:gmail:{{messages[0].id}}`, but setup does not write the constrained `allowRequestSessionKey` / `allowedSessionKeyPrefixes` policy required by current `src/gateway/hooks.ts`.
