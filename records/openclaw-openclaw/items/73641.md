## Summary

Keep this PR open. Current main still resolves the Telegram model-list current selection from `runtimeCfg.agents?.defaults?.model` when no stored session override exists, so a Telegram account bound to an agent with its own model can still show the global default with the check mark. The PR is a narrow implementation candidate: it reuses `resolveDefaultModelForAgent` for the bound agent and adds a regression test for the reported global-default-vs-agent-default case.

## Best Possible Solution

Review and land this focused PR, assuming the targeted Telegram regression test and normal changed gate pass. The fix should stay in the Telegram plugin runtime path: compute the bound agent default once with `resolveDefaultModelForAgent({ cfg: runtimeCfg, agentId: route.agentId })`, reuse it for stored override resolution, and use it for the no-session-override model-list current selection.
