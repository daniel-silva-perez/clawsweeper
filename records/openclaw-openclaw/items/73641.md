Keep this PR open for maintainer review and merge. Current main still has the Telegram check-mark bug, while the PR is focused, uses the existing agent-effective model resolver, includes a matching regression test, and has no blocking review findings from this pass.

## What This Changes

This PR changes Telegram session-state fallback model resolution to use the bound agent's effective default model and adds a regression test for the model-list check mark.

## Best Possible Solution

Land a narrow Telegram plugin fix that computes the bound agent's effective default model once, preserves stored override precedence, and uses that same resolved provider/model for the model-list current-selection fallback.

## Reproduction Assessment

Yes. Configure a Telegram binding to an agent with its own `agents.list[].model`, set a different `agents.defaults.model`, leave the session without a model override, and trigger `mdl_list_<provider>_1`; current main's source shows the global default drives the check mark.

## Solution Assessment

Yes. The PR uses the existing `resolveDefaultModelForAgent` contract inside the Telegram owner boundary, avoids a new core seam, and changes only the no-override fallback while keeping stored overrides first.
