## Summary

Keep open. Current main still has the Telegram model-picker bug, and the contributor patch appears to address the actual callback path without introducing a security-sensitive change; this sweep found no blocking review findings.

## What This Changes

The PR changes Telegram callback session-state resolution to use the bound agent's effective default model and adds a regression test for the model-list check mark.

## Best Possible Solution

Land a narrow Telegram plugin fix that computes the effective default for the bound agent once, reuses it for stored override handling, and uses that same resolved provider/model for the no-override model-list current selection.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to bind Telegram to an agent with its own `agents.list[].model`, set a different `agents.defaults.model`, leave the session without an override, and trigger `mdl_list_<provider>_1`; current main's source shows the global default is what gets passed to the keyboard.

## Solution Assessment

Yes. The proposed approach uses the existing `resolveDefaultModelForAgent` contract at the Telegram owner boundary, avoids a new core seam, and preserves stored override precedence while fixing only the no-override fallback.
