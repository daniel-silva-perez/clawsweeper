## Summary

Keep this PR open, but it is not merge-ready. Current main still lacks Android bearer/basic transport auth for reverse-proxy-protected gateways, so the contribution addresses a real gap, but the patch introduces blocking persistence and credential-scoping problems plus a security-sensitive TLS bypass that needs maintainer review.

## What This Changes

The PR adds Android manual/onboarding fields for bearer token, basic-auth credentials, and disabling TLS verification, then threads those values through secure preferences, gateway WebSocket setup, canvas WebView loading, and resolver tests.

## Best Possible Solution

Revise the PR into a narrow Android proxy-auth implementation that persists bearer/basic credentials, sends them only to the intended gateway transport origin, strips them from cross-origin redirects/subresources, adds focused Android tests and a changelog entry, and either removes the TLS-verification bypass or gates it behind an explicit maintainer-approved security design. Track iOS and setup-code/QR transport-auth expansion separately if maintainers want to split the broader #51872 scope.

## Reproduction Assessment

Yes. The persistence failure is reproducible by entering bearer/basic credentials, connecting, restarting the app, and observing that auto-reconnect resolves proxy auth from SecurePrefs load methods even though neither ConnectTab nor Onboarding saved the new credential fields; the header leak is reproducible with a configured proxy credential and any canvas page that loads an external HTTP(S) resource.

## Solution Assessment

No. The feature direction is valid, but this implementation is not the best fix because it stores and forwards proxy credentials inconsistently, applies Authorization headers to unbounded WebView HTTP(S) requests, and mixes in a persistent TLS-verification bypass; the safer path is origin-scoped transport auth with explicit persistence and security review.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Persist manual proxy credentials before connecting:** `apps/android/app/src/main/java/ai/openclaw/app/ui/ConnectTabScreen.kt:312-314`
  - body: The new manual bearer/basic fields are only passed into the immediate `connect()` call. On the next launch, reconnect calls `resolveGatewayConnectAuth()` and reads these values from `SecurePrefs`, but this screen never calls the new setters, so proxy auth is silently dropped after restart. Save the bearer token/basic user/basic password here before connecting.
  - confidence: 0.93
- **[P1] Persist onboarding proxy credentials before connecting:** `apps/android/app/src/main/java/ai/openclaw/app/ui/OnboardingFlow.kt:930-935`
  - body: Onboarding has the same transient-only path: it trims the new proxy credentials and sends them to `connect()`, but only the normal gateway password is persisted. A user who completes onboarding behind a bearer/basic proxy will lose those credentials on restart and auto-reconnect will fail.
  - confidence: 0.9
- **[P1] Scope canvas auth headers to the gateway origin:** `apps/android/app/src/main/java/ai/openclaw/app/ui/CanvasScreen.kt:116-118`
  - body: `shouldInterceptRequest` now applies `viewModel.canvas.getAuthHeaders()` to every `http`/`https` request made by the WebView. If the canvas page loads an external image/script/fetch target, the reverse-proxy bearer or basic credential is sent to that host. Limit these headers to the configured gateway origin and strip them on cross-origin redirects.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff introduces security-sensitive transport credential handling and a TLS bypass; canvas request header scoping is a concrete credential-leak risk.

Concerns:

- **[high] Canvas requests can leak proxy credentials:** `apps/android/app/src/main/java/ai/openclaw/app/ui/CanvasScreen.kt:116`
  - body: The WebView interceptor adds Authorization headers to every HTTP(S) request rather than only the gateway origin. External subresources or fetches can receive the user's reverse-proxy bearer/basic credential.
  - confidence: 0.88
- **[medium] TLS verification bypass persists as a manual setting:** `apps/android/app/src/main/java/ai/openclaw/app/gateway/GatewayTls.kt:63`
  - body: When disableTlsVerification is true, the new TLS path skips certificate validation and accepts hostname mismatches, while the UI persists the toggle. That is a broad transport-security downgrade outside the linked bearer/basic request and needs explicit maintainer-approved scope and warnings.
  - confidence: 0.76
