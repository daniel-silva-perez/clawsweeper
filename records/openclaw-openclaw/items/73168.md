## Summary

Keep this PR open. Current main still lacks Android transport-level bearer/basic auth for reverse-proxy WebSocket upgrades, so the contribution targets a real missing feature, but the latest head is not merge-ready because it still drops proxy credentials after restart, forwards Authorization headers to unbounded WebView HTTP(S) requests, and introduces a persistent TLS-verification bypass that needs security review.

## What This Changes

The PR adds Android manual/onboarding fields for bearer/basic reverse-proxy credentials and disabling TLS verification, stores new preference fields, threads proxy auth through gateway WebSocket and canvas loading paths, and adds resolver coverage.

## Best Possible Solution

Revise this into a narrow Android proxy-auth implementation that persists bearer/basic credentials, sends Authorization only to the intended gateway transport origin, strips credentials from cross-origin WebView requests and redirects, adds focused Android tests and a changelog entry, and removes or separately gates the TLS-verification bypass behind an approved security design.

## Reproduction Assessment

Yes. The persistence failure is reproducible by entering bearer/basic credentials, connecting, restarting, and following reconnect through resolveGatewayConnectAuth() to SecurePrefs loads that were never populated; the WebView leak is reproducible from source by any canvas page loading an external HTTP(S) resource while proxy auth is configured.

## Solution Assessment

No. The feature direction is valid, but this PR is not the best current fix because it persists and forwards proxy credentials inconsistently, scopes WebView Authorization too broadly, and mixes in a broad TLS-verification bypass; origin-scoped transport auth with explicit persistence is the safer path.

  - body: The manual connect path saves the normal gateway auth fields but only passes bearer/basic values to the immediate connect call.   - body: Onboarding has the same transient-only behavior: bearer/basic values are trimmed and sent to connect(), but only the normal gateway password/token fields are persisted. Users who complete onboarding behind a bearer/basic proxy will fail auto-reconnect after restart.
  - confidence: 0.92
- **[P1] Scope canvas auth headers to the gateway origin:** `apps/android/app/src/main/java/ai/openclaw/app/ui/CanvasScreen.kt:124-135`
  - body: shouldInterceptRequest now adds the proxy Authorization header to every HTTP(S) WebView request. External images, scripts, redirects, or fetches loaded by the canvas can receive the user's reverse-proxy credential. Limit these headers to the configured gateway origin and strip them on cross-origin requests.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff introduces security-sensitive transport credential handling and a persistent TLS bypass; canvas request header scoping is a concrete credential-leak issue.

Concerns:

- **[high] Canvas requests can leak proxy credentials:** `apps/android/app/src/main/java/ai/openclaw/app/ui/CanvasScreen.kt:124`
  - body: The WebView interceptor applies Authorization headers to every HTTP(S) URL rather than only the gateway origin, so external subresources or redirects can receive bearer/basic proxy credentials.
