## Summary

Keep this PR open. The linked feature request is valid and current main still lacks Android transport-level bearer/basic auth for reverse-proxy-protected gateways, but the latest PR head still has security-sensitive credential scoping issues in the Android canvas path and adds a persistent TLS verification bypass that needs maintainer security review before merge.

## What This Changes

The PR adds Android manual/onboarding fields for bearer/basic reverse-proxy auth and disabling TLS verification, persists the new auth fields, threads proxy Authorization headers through gateway WebSocket and canvas loading paths, and adds focused Android tests.

## Best Possible Solution

Revise the Android implementation so reverse-proxy Authorization is persisted but only applied to the gateway WebSocket upgrade and strictly same-origin canvas HTTP(S) requests, with scheme/host/port checked by a shared production helper. Split or remove the TLS verification bypass unless maintainers approve a documented, warning-heavy development-only flow with tests and changelog coverage.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure bearer/basic proxy auth, invoke canvas.navigate with an external HTTPS URL, and observe that the PR's wv.loadUrl(currentUrl, headers) path would send the proxy Authorization header to that URL; a second source-level path is an HTTPS gateway canvas page loading http://same-host:same-port mixed content, which passes the PR's authority-only interceptor check.

## Solution Assessment

No. The feature direction is the right place for the linked reverse-proxy request, and the latest head fixed the earlier persistence/duplicate-header problems, but this implementation is not safe until proxy credentials are scoped to the exact gateway/canvas origin and the TLS bypass is either removed or separately approved.
