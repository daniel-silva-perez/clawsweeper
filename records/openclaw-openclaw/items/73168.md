## Summary

Keep this PR open. The linked Android reverse-proxy transport-auth problem is real on current main, but the latest head is not merge-ready because bearer/basic credentials are not persisted for reconnect, WebView Authorization headers are applied to every HTTP(S) request, and the TLS-verification bypass needs explicit security review.

## What This Changes

The PR adds Android manual/onboarding fields for bearer/basic reverse-proxy credentials and TLS-verification bypass, stores new preference fields, threads proxy auth through gateway WebSocket and canvas loading paths, and adds resolver coverage.

## Best Possible Solution

Revise this into a narrow Android proxy-auth implementation that persists bearer/basic credentials, sends Authorization only to the intended gateway transport origin, strips credentials from cross-origin WebView requests and redirects, adds focused Android tests and a changelog entry, and removes or separately gates the TLS-verification bypass behind an approved security design.

## Reproduction Assessment

Yes. Enter bearer/basic credentials, connect, restart, and follow reconnect through resolveGatewayAuth() to SecurePrefs loads that were never populated; the WebView leak is source-reproducible by any canvas page loading an external HTTP(S) resource while proxy auth is configured.

## Solution Assessment

No. The feature direction is valid, but this PR is not the best current fix because it persists and forwards proxy credentials inconsistently, scopes WebView Authorization too broadly, and mixes in a broad TLS-verification bypass; origin-scoped transport auth with explicit persistence is the safer path.
