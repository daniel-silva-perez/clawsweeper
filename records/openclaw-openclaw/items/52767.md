## Summary

Keep open. Current main still dispatches the `gateway:startup` internal hook on gateway startup, the bundled `boot-md` handler still calls `runBootOnce` for each configured agent, and `runBootOnce` still runs whenever `BOOT.md` exists and is non-empty. Recent main has partial session-isolation mitigation, but not the durable idempotency/reset semantics requested in the issue and reinforced by the April 28 production-impact comment.

## Best Possible Solution

Keep the issue open for maintainer handling. The best path is to decide whether `BOOT.md` is meant to be a repeatable startup checklist or a first-run/at-most-once hook. If it should be at-most-once, implement a durable per-agent workspace guard with explicit reset/re-run semantics and regression tests proving repeated `gateway:startup` does not re-run the same completed `BOOT.md`; if it should remain repeatable, update docs and UX to make that contract clear and steer one-time initialization to `BOOTSTRAP.md`. Preserve the current boot-session isolation and add coverage that boot work cannot block channel readiness.

## Risks / Open Questions

- Closing now would leave a recent production stall report unresolved while current main still re-runs non-empty `BOOT.md` on every configured gateway startup.
- A naive sentinel could break the currently documented startup-check workflow, so the fix needs explicit reset/re-run semantics and likely a per-agent workspace or content/config-generation key.
- The fresh boot-session and mapping-restore mitigation reduces old main-session-history replay risk, but repeated boot work, model/tool cost, and startup pressure remain possible.
