## Summary

Keep open. Current main still intentionally dispatches `boot-md` on each configured `gateway:startup` and `runBootOnce` still runs whenever `BOOT.md` exists and is non-empty. Main has a partial mitigation for the later production comment: boot runs now use a generated boot session id and restore the main-session mapping afterward, but there is no durable per-workspace/per-agent idempotency guard, reset mechanism, or test proving a second gateway restart skips the same `BOOT.md`. The docs also currently describe `BOOT.md` as a startup checklist, while first-run bootstrapping is documented separately as `BOOTSTRAP.md`, so this needs maintainer product/behavior clarification rather than cleanup closure.

## Best Possible Solution

Keep the issue open for maintainer handling. The best path is to decide the intended `BOOT.md` contract, then either implement a durable per-agent boot-md idempotency guard with explicit reset/re-run semantics and regression tests for repeated `gateway:startup`, or update docs/UX to make `BOOT.md` clearly a repeatable startup checklist and steer first-run initialization to `BOOTSTRAP.md`. Preserve the current boot-session isolation/mapping-restore behavior and add coverage that repeated boot work cannot block channel readiness.

## Risks / Open Questions

- Closing now would leave a concrete, recently updated production stall report unresolved while current main still re-runs `BOOT.md` on every gateway restart.
- A naive sentinel could break legitimate startup-check workflows currently documented for `BOOT.md`; the fix should define reset/re-run semantics, likely per agent workspace and config/content generation.
- The current fresh boot session and mapping restore reduce part of the session-history risk, but they do not address repeated boot work, repeated model/tool cost, or repeated startup pressure.
