## Summary

Keep open. Current main still drops `accountId` before `/hooks/agent` reaches cron delivery: the direct hook payload and normalizer omit it, the replay scope and mapped-hook action surface do not include it, and `dispatchAgentHook()` hand-builds delivery with only `mode`, `channel`, and `to`. Cron downstream already supports `delivery.accountId`; the only meaningful change since the prior review is that current main's cron announce path now keeps the message tool enabled, so that part of the report is no longer the active gap. Open PR #43951 remains the focused closing implementation candidate.

## What This Changes

The issue requests preserving `/hooks/agent` `accountId` through hook normalization, idempotency scoping, mapped hooks, and cron delivery dispatch so multi-account channel delivery can select the intended account.

## Best Possible Solution

Review, rebase, and land or replace PR #43951 with a gateway-owned fix that trims and forwards `accountId` for direct and mapped hooks, includes it in replay scope, constructs `CronDelivery.accountId`, preserves sanitized hook metadata, keeps current main's announce message-tool behavior, documents the public field and trust model, and adds direct hook, mapped hook, and cron delivery regression tests.
