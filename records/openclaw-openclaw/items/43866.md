## Summary

Keep open. Current main still omits `accountId` from the direct `/hooks/agent` payload, normalizer, replay scope, mapped-hook dispatch, and synthetic cron delivery object, while the downstream cron delivery path already supports `delivery.accountId`. The issue is also paired with open closing PR #43951, so the right cleanup path is to review, land, or replace that PR rather than close the issue or queue a duplicate repair.

## What This Changes

The issue requests preserving `accountId` from `POST /hooks/agent` through hook dispatch into cron delivery so multi-account channel delivery can target the correct account.

## Best Possible Solution

Land or replace PR #43951 with a narrow gateway-owned fix that trims and forwards `accountId` for direct and mapped hook agent dispatch, includes it in replay/idempotency scope, constructs `CronDelivery.accountId`, updates the public hook docs, and adds regression coverage without adding WhatsApp-specific routing logic.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: a `/hooks/agent` request with `accountId` enters `normalizeAgentPayload()`, which never reads that field, and `dispatchAgentHook()` then creates cron delivery without it.

## Solution Assessment

Yes. The best fix boundary is hook-to-cron account propagation, because the downstream cron delivery contract already supports `accountId`; a channel-specific workaround would be narrower in symptom but less maintainable.
