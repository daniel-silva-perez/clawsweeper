## Summary

Keep open. Current main still loses `accountId` before `/hooks/agent` reaches cron delivery: the direct hook payload and normalizer omit it, the dispatch path builds delivery with only `mode`, `channel`, and `to`, and the mapped-hook/replay surfaces also have no account selector. Cron downstream already supports `delivery.accountId`, and the remaining work is paired with open closing PR #43951.

## What This Changes

The issue requests preserving `accountId` from `POST /hooks/agent` through hook normalization and cron delivery so multi-account channel delivery can target the correct account.

## Best Possible Solution

Review, rebase, and land or replace PR #43951 with a gateway-owned fix that trims and forwards `accountId` for direct and mapped hooks, includes it in replay scope, constructs `CronDelivery.accountId`, documents the public hook field, and adds direct hook, mapped hook, and cron delivery regression tests while preserving current cron message-tool behavior.

## Reproduction Assessment

Yes. A high-confidence source trace reproduces the bug: sending `accountId` to `/hooks/agent` enters `normalizeAgentPayload()`, which never reads the field, then `dispatchAgentHook()` creates cron delivery without it.

## Solution Assessment

Yes, the best fix is a narrow hook-to-cron accountId propagation change rather than a channel-specific workaround. PR #43951 is the focused candidate, with the caveat that current main's message-tool policy should not be regressed.
