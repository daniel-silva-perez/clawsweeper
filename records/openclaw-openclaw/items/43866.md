## Summary

Keep open. Current main still drops `accountId` before `/hooks/agent` reaches cron delivery: the direct hook payload and normalizer omit it, dispatch constructs delivery with only `mode`, `channel`, and `to`, and mapped-hook replay/dispatch also has no account selector. The remaining work is already paired with open closing PR #43951, so maintainers should review, land, or replace that PR rather than close this issue or queue a duplicate repair.

## What This Changes

The issue requests preserving `accountId` from `/hooks/agent` and mapped hook payloads into `CronDelivery` so multi-account channel delivery resolves the intended account.

## Best Possible Solution

Land or replace PR #43951 with a gateway-owned fix that trims and forwards `accountId` for direct and mapped hooks, includes it in replay scope, constructs `CronDelivery.accountId`, documents the public hook field, and adds regression coverage while preserving current non-webhook message-tool behavior.

## Reproduction Assessment

Yes. A source-level reproduction is enough: a `/hooks/agent` payload with `accountId` enters `normalizeAgentPayload()`, which never reads the field, then `dispatchAgentHook()` creates cron delivery without it.

## Solution Assessment

Yes, the right fix boundary is narrow hook-to-cron accountId propagation rather than a WhatsApp-specific workaround. Current main already has the downstream cron account contract, so the maintainable solution is to bridge the gateway hook payload into that contract.
