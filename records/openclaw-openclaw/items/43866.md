## Summary

Keep open. Current main still drops hook-supplied `accountId` before cron delivery, while the downstream cron delivery contract already supports `delivery.accountId`; the issue is also paired with open closing PR #43951, so maintainers should review, land, or replace that PR instead of closing the issue or queueing a duplicate repair. One detail in the original report has changed: current main now keeps the message tool enabled for announce delivery, so the remaining core bug is the hook-to-cron account propagation gap.

## What This Changes

The issue requests preserving `accountId` from `POST /hooks/agent` through direct and mapped hook dispatch into cron delivery so multi-account channels can select the correct outbound account.

## Best Possible Solution

Land or replace PR #43951 with a narrow gateway-owned fix that trims and preserves `accountId` for direct and mapped hook agent dispatch, includes it in replay/idempotency scope, constructs `CronDelivery.accountId`, documents the public `/hooks/agent` contract, and adds regression tests without WhatsApp-specific routing logic. Preserve sanitized hook names and current announce message-tool behavior unless maintainers intentionally split that into a separate behavior change.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: a `/hooks/agent` body with `accountId` enters `normalizeAgentPayload()`, the field is never read, and `dispatchAgentHook()` creates cron delivery without it even though cron delivery can carry `accountId`.

## Solution Assessment

Yes. The maintainable fix boundary is generic hook-to-cron account propagation, because cron delivery already owns account-aware target resolution; a channel-specific WhatsApp workaround would leave the shared hook contract broken.
