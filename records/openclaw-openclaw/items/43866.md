## Summary

Keep open. Current main still drops `accountId` before `/hooks/agent` reaches cron delivery: the direct hook payload and normalizer omit it, the replay scope and mapped-hook action surface do not include it, and `dispatchAgentHook()` hand-builds delivery with only `mode`, `channel`, and `to`. The downstream cron delivery pipeline already supports `delivery.accountId`, and the provided context shows open PR #43951 is still the focused closing implementation candidate.

## What This Changes

The issue requests preserving `/hooks/agent` `accountId` through hook normalization, replay scoping, and cron delivery dispatch so multi-account channel delivery can resolve the intended account.

## Best Possible Solution

Land or replace the focused hook bridge fix so direct and mapped hooks accept a trimmed `accountId`, include it in idempotency scope, construct `CronDelivery.accountId`, preserve sanitized hook metadata, keep current main's cron message-tool policy, document the public hook field and trust model, and add direct hook, mapped hook, and isolated cron delivery regression tests.
