## Summary

Keep this issue open. Current main at 68561a8c948b49e85ae012646de550a49d6c63a3 still strips `accountId` from direct `/hooks/agent` payloads and never adds it to the hand-built cron delivery object, while the downstream cron delivery path already supports `delivery.accountId`. The provided GitHub context also shows an open same-author implementation candidate, #43951, with closing syntax for this issue, so the maintainer path is to review/land or replace that PR rather than close the report early.

## Best Possible Solution

Keep this issue open until #43951 is reviewed and merged, or replace it with a focused current-main patch. The fix should add `accountId` to the direct and mapped hook agent contracts, normalize/trim it, include it in idempotency replay scope, propagate it into `CronDelivery.accountId` during hook dispatch, document it on the public webhooks page, and add regression coverage proving `/hooks/agent` reaches isolated cron delivery with the selected account.
