## Summary

Keep open. Current main still drops hook-supplied `accountId` before constructing cron delivery, while the downstream cron delivery path already supports `delivery.accountId`. The issue is also paired with open closing PR #43951, so maintainers should review, land, or replace that PR rather than close this issue or queue a duplicate automated repair.

## What This Changes

The issue requests that `/hooks/agent` preserve an incoming `accountId` through hook normalization and dispatch into cron delivery for multi-account channel routing.

## Best Possible Solution

Land or replace PR #43951 with a narrow gateway-owned fix that trims and preserves `accountId` for direct and mapped `/hooks/agent` dispatch, includes it in idempotency/replay scope, constructs `CronDelivery.accountId`, updates the public hook contract docs, and adds regression tests without adding WhatsApp-specific routing logic.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: a `/hooks/agent` body with `accountId` enters `normalizeAgentPayload()`, the field is never read, and `dispatchAgentHook()` creates cron delivery without it even though cron delivery can carry `accountId`.

## Solution Assessment

Yes. The maintainable fix is generic hook-to-cron account propagation because cron delivery already owns account-aware target resolution; a channel-specific WhatsApp workaround would leave the shared hook contract broken.
