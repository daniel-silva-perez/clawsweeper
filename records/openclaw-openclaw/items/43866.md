## Summary

Keep open. Current main still drops `accountId` at the `/hooks/agent` and mapped-hook boundaries before building the isolated cron job, while cron itself already supports `delivery.accountId`. The issue is also paired with open implementation PR #43951, so it should remain open until that PR is reviewed and merged or replaced.

## What This Changes

The issue requests preserving `accountId` through `/hooks/agent` dispatch so multi-account channel delivery can select the intended account.

## Best Possible Solution

The desired end state is a focused mainline fix that normalizes `accountId` for direct and mapped hooks, includes it in idempotency/replay scope, propagates it into `CronDelivery.accountId`, updates public hook docs, and adds regression coverage for isolated cron delivery with the selected account.
