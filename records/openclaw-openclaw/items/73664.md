## Summary

Keep this issue open. Current main still has no llama.cpp `/props.n_ctx` discovery path for self-hosted OpenAI-compatible models, and the provided discussion shows open PR #74057 explicitly targets this issue with closing syntax, so the issue should wait for normal PR review rather than being closed early.

## What This Changes

The issue asks OpenClaw to show llama.cpp’s actual runtime context window in session usage instead of the model’s training context window.

## Best Possible Solution

Review, revise if needed, and land PR #74057 or an equivalent guarded self-hosted provider discovery change that reads positive llama.cpp `/props.n_ctx` as the runtime `contextTokens`, handles trailing `/v1` base URLs, preserves explicit configured caps as authoritative, and adds regression coverage for `/v1/models` reporting `n_ctx_train: 262144` while `/props` reports `n_ctx: 65536`.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: a llama.cpp OpenAI-compatible server whose `/v1/models` metadata exposes only training context still goes through OpenClaw’s `/models`-only discovery path, while session display reads stored/configured context tokens without probing `/props.n_ctx`.

## Solution Assessment

Yes. Keeping this issue open while PR #74057 is reviewed is the right path; the narrow maintainable fix belongs in self-hosted provider discovery, with session display continuing to consume the resolved `contextTokens` value.
