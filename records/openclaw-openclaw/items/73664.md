## Summary

Keep this issue open. Current main still does not implement the llama.cpp-specific runtime context fix: there is no `/props`/`n_ctx` reader, existing generated `models.json` model limits are not preserved as authoritative during merge-mode regeneration, and session displays still use persisted/resolved `contextTokens` as the denominator.

## Best Possible Solution

Keep the issue open and fix the provider/context-resolution path rather than the sessions table. Best implementation: for llama.cpp/OpenAI-compatible local providers, store an effective runtime `contextTokens` by querying llama.cpp `/props` for `n_ctx` with the existing provider HTTP safety policy, or preserve an explicit same-provider/model runtime cap from configuration or existing `models.json` before live `/v1/models` metadata can replace it. Add focused coverage for the 262144 trained-context versus 65536 runtime-context case and verify sessions display `56k/65k`-style usage.
