## Summary

Keep this issue open. Current main still does not discover llama.cpp's loaded runtime context from `/props.n_ctx`, and the provided GitHub context shows #74057 is an open, unmerged PR with closing syntax that directly targets this report.

## What This Changes

The issue asks the self-hosted OpenAI-compatible llama.cpp provider path to use the actual loaded runtime context window for session usage instead of trained-context metadata or an inflated catalog value.

## Best Possible Solution

Land or revise #74057, or an equivalent narrow self-hosted provider discovery change, so explicit configured caps stay authoritative while llama.cpp-compatible servers can supply a smaller loaded runtime window via `/props.n_ctx` for session budgeting.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: a self-hosted OpenAI-compatible llama.cpp server whose `/v1/models` lacks runtime `n_ctx` but whose `/props` reports `n_ctx: 65536` cannot be corrected by current main because discovery only fetches `/models` and session listing does not perform async live probes.

## Solution Assessment

Yes. The best path is to keep this issue open while maintainers review the existing guarded `/props.n_ctx` PR, rather than creating a second repair branch or closing before the implementation merges.
