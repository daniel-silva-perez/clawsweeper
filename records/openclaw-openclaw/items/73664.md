## Summary

Keep this issue open. Current main still does not implement llama.cpp `/props.n_ctx` runtime-context discovery for self-hosted OpenAI-compatible providers, and #74057 is an open, unmerged PR with closing syntax that directly targets the reported behavior.

## What This Changes

Use the actual llama.cpp runtime context window for session-token budgeting instead of showing the trained `n_ctx_train` limit from `/v1/models`.

## Best Possible Solution

Review and land or revise #74057, or an equivalent narrow provider-discovery change, so explicit configured caps remain authoritative while llama.cpp-compatible servers can supply the smaller loaded runtime context via `/props.n_ctx`.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: a self-hosted llama.cpp OpenAI-compatible server whose `/v1/models` lacks runtime `n_ctx` cannot be corrected by current main because the relevant provider/session paths contain no `/props.n_ctx` handling.

## Solution Assessment
