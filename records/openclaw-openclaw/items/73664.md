## Summary

Keep this issue open. Current main has newer `contextTokens` support that can represent an effective runtime cap, but it still does not implement the reported llama.cpp fix: there is no `/props` or `n_ctx` reader, existing agent `models.json` model limits are not treated as authoritative during merge-mode regeneration, and session tables still display whatever context cap was stored or resolved upstream.

## Best Possible Solution

Keep the issue open and fix the provider/context-resolution path. For llama.cpp-style OpenAI-compatible local providers, read `/props.n_ctx` under the existing provider HTTP safety policy and store it as `contextTokens` while preserving `contextWindow` as native/training metadata. Also preserve explicit user runtime caps where appropriate and add focused regression coverage for a 262144 trained context versus 65536 runtime context so sessions display about `56k/65k`, not `56k/262k`.
