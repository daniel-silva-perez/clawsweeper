## Summary

Keep open. The linked bug is still present on current main, but this draft PR is not ready to land because the macOS unsafe-path fallback does not fail closed and the work overlaps the separate open fix candidate #72650.

## What This Changes

The PR broadens exec approvals path validation to allow trusted first-level `~/.openclaw` symlinks, adds Node and macOS path-hardening tests, and documents the supported symlink shape.

## Best Possible Solution

Land one reviewed, rebased fix that permits only the trusted first-level `.openclaw` symlink shape on supported platforms, fails closed consistently in Node and macOS runtime policy paths, preserves current-main concurrency hardening, and keeps deeper symlinks plus symlinked `exec-approvals.json` destinations rejected.

## Reproduction Assessment

Yes. The linked bug is reproducible from current-main source and tests: `~/.openclaw` is a path component below the trusted home root, so `assertNoSymlinkPathComponents()` rejects it before `saveExecApprovals()` can write the approvals file.

## Solution Assessment

No in current form. The direction is appropriate, but the macOS fallback for unsafe approvals paths should deny rather than return empty defaults, and maintainers should compare or consolidate this draft with #72650 before merge.
