## Summary

Keep this PR open for maintainer handling. Current main still rejects the reported `~/.openclaw` symlink shape, and this branch is useful source work, but it is a draft security-sensitive path-hardening change with an overlapping implementation candidate and a stale macOS approvals-store regression against current main.

## What This Changes

The PR broadens exec approvals path validation to allow trusted first-level `~/.openclaw` symlinks, adds Node and macOS path-hardening tests, and documents the supported symlink shape.

## Best Possible Solution

Land one reviewed, rebased implementation that permits only the trusted first-level `.openclaw` symlink shape on supported platforms, keeps deeper symlinks and symlinked `exec-approvals.json` destinations rejected, preserves current macOS file-locking and fail-closed runtime policy behavior, updates docs/changelog, and closes #72572 after merge.

## Reproduction Assessment

Yes. Current main has a source-level reproduction path: `openclaw exec-policy preset yolo` writes through `saveExecApprovals()`, which reaches `ensureDir()` and rejects `~/.openclaw` when that path component is a symlink; the existing test suite still asserts that rejection.

## Solution Assessment

No, not as-is. The implementation direction is plausible, but this draft should be rebased and consolidated with #72650, and the macOS approvals-store lock from current main must be preserved before the patch is a safe solution.
