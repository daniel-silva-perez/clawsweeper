## Summary

Keep this PR open. The proposed patch addresses a real normalization mismatch for some channels, but its global stripping in plugin message hook contexts would regress resolver-backed channels such as Discord, where `user:` and `channel:` are semantic parts of the canonical conversation id and are preserved by `inbound_claim`.

## What This Changes

The PR exports a hook conversation-id normalization helper, applies it to `toPluginMessageContext`, and updates mapper tests to expect stripped Telegram, Slack, and Discord-style prefixes.

## Best Possible Solution

Revise the mapper so plugin message hook `conversationId` values are normalized through the same channel-owned conversation contract as `inbound_claim` where available, while preserving semantic resolver-owned prefixes such as Discord `user:` and `channel:` ids. Add regression coverage for both fallback-prefix channels and resolver-backed typed ids.

## Reproduction Assessment

Yes. A Discord direct-message or the existing `claim-chat` fixture that resolves to `user:<id>` reproduces the mismatch: `toPluginInboundClaimContext` preserves `user:<id>`, while this PR would make `toPluginMessageContext` expose only `<id>`.

## Solution Assessment

No. The proposed global strip inside `toPluginMessageContext` is not the best fix because it erases channel-owned identity semantics; a safer fix should align message contexts with the channel resolver/fallback path used for inbound claims.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve resolver-owned conversation ids:** `src/hooks/message-hook-mappers.ts:176`
  - body: Applying `normalizeMessagingConversationId` here strips `user:` and `channel:` from every plugin message hook context. Discord intentionally returns those typed ids from its inbound resolver, and `toPluginInboundClaimContext` preserves resolver output, so the same Discord DM can become `inbound_claim` `user:<id>` but `message_received` `<id>`, breaking hook correlation and DM-vs-channel semantics. Use a channel-aware resolution path or avoid stripping semantic resolver prefixes.
