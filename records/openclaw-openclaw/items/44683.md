## Summary

Keep open. Current main and the latest release still do not include a Helm chart, and the Kubernetes docs still position Kustomize/raw manifests as the shipped path with Helm as a possible layer. The contribution is plausible, but it adds a new deployment packaging surface with unresolved correctness and security concerns, so it needs maintainer review rather than conservative cleanup closure.

## What This Changes

The PR adds a new `charts/openclaw/` Helm chart with values, templates, and chart README for deploying OpenClaw on Kubernetes alongside the existing raw manifests.

## Best Possible Solution

Keep this PR open for maintainer review. The chart should either be revised into an owner-approved, tested deployment surface that preserves the raw-manifest object/data contract and handles secrets safely, or maintainers should explicitly decide that Helm support belongs outside the core repository despite the current docs leaving room for a layered chart.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a current-main bug because main has no chart, but the PR defects have clear static reproduction paths: render with changed `service.port`, `secret.create=false` without `existingSecret`, or changed config/secret values and inspect the generated templates.

## Solution Assessment

Is this the best way to solve the issue? No, not as submitted. A Helm chart may be a reasonable direction, but this specific patch needs maintainer product approval plus chart fixes for rollout triggers, naming compatibility, port/config coupling, secret handling, and validation before it is the maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Trigger rollouts for config and secret changes:** `charts/openclaw/templates/deployment.tpl:15-17`
  - body: The pod template has no checksum annotations for the rendered ConfigMap or Secret. Because config is copied by the init container and secrets are injected as env vars, `helm upgrade` can succeed while the running pod keeps the old config/token until someone restarts the Deployment manually.
  - confidence: 0.9
- **[P1] Avoid storing credentials as Helm values:** `charts/openclaw/README.md:10-11`
  - body: The install docs pass `secret.gatewayToken` and provider keys through `--set`, and the chart models those credentials under values. That can retain gateway tokens and provider keys in Helm release state and shell history instead of keeping them only in a Kubernetes Secret workflow.
  - confidence: 0.82
- **[P2] Preserve existing Kubernetes resource names:** `charts/openclaw/templates/_helpers.tpl:8-9`
  - body: The fullname helper always renders `<release>-<chart>`, so the documented release name `openclaw` creates resources like `openclaw-openclaw` and a new PVC. That breaks the stated equivalence with the existing manifests and prevents an in-place migration from the current `openclaw` objects.
  - confidence: 0.91
- **[P2] Keep the gateway port tied to service.port:** `charts/openclaw/values.yaml:83-84`
  - body: `config.openclawJson` hardcodes gateway port `18789` while the chart exposes `service.port` for the container and Service. Overriding `service.port` renders Kubernetes resources for one port while the gateway still listens on another, causing failed probes or unreachable Service traffic.
  - confidence: 0.88
- **[P2] Require an existing secret when creation is disabled:** `charts/openclaw/templates/secret.tpl:1`
  - body: When `secret.create=false` and `secret.existingSecret` is empty, the helper still generates a Secret name but this template creates no Secret. The rendered Deployment then references a required token Secret that does not exist, leaving pods in `CreateContainerConfigError` instead of failing at render time.
  - confidence: 0.87

## Security Review

Status: needs_attention

Summary: The diff does not add third-party CI or dependency execution, but its Helm secret model can retain operator credentials outside the intended Secret-only workflow.

Concerns:

- **[medium] Credentials flow through Helm values:** `charts/openclaw/README.md:10`
  - body: The README tells users to pass `secret.gatewayToken` and provider API keys with `--set`, and `values.yaml` defines those fields as chart values. That can expose long-lived OpenClaw gateway tokens and model-provider keys through shell history and Helm release state; prefer existing Secret references or a documented external Secret-only path.
  - confidence: 0.82
