## Summary

Current main still has the reported boundary mismatch for dreaming/backfill: the bundled session-memory hook writes slugged `memory/YYYY-MM-DD-slug.md` artifacts, while memory-core live daily ingestion, CLI `rem-backfill`/`rem-harness`, and Gateway Dream Diary backfill still discover only exact `YYYY-MM-DD.md` files. Startup-context and memory-search paths cover some slugged-file visibility, but they do not solve the dreaming promotion/backfill path described here.

## Best Possible Solution

Keep this issue open and fix memory-core discovery so existing session-memory hook artifacts can participate in dreaming/backfill. The least disruptive path is a shared dated-memory-file matcher that accepts both `YYYY-MM-DD.md` and `YYYY-MM-DD-*.md`, then use it in live daily ingestion, CLI `rem-harness`/`rem-backfill`, Gateway Dream Diary backfill, and tests/docs without changing unrelated memory-search or hook behavior.

## Risks / Open Questions

- Accepting slugged files in dreaming/backfill could double-count content when both `YYYY-MM-DD.md` and several `YYYY-MM-DD-slug.md` files exist for the same day, so the fix needs deterministic ordering and dedupe behavior.
- `seedHistoricalDailyMemorySignals` currently normalizes staged result paths to `memory/<day>.md`; if it starts accepting slugged files, source attribution and rollback/debug output should preserve enough original path context.
- Memory search already indexes `memory/*.md`, so the bug is specifically the dreaming/backfill promotion path; a fix should avoid broad memory-indexing churn.
