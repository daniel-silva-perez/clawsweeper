## Summary

Keep this PR open, but do not merge it as-is. The Slack retry/unwrapping direction is well scoped, but the delivery-accounting change still drops successful `tool` dispatch counts before calling the shared visible-delivery helper, and the user-facing Slack fix is missing the required changelog entry.

## What This Changes

The PR adds bounded Slack `chat.postMessage` transient retries with wrapped request-error unwrapping, adjusts failed-delivery accounting for Slack replies, and adds focused Slack send/dispatch tests.

## Best Possible Solution

Keep the bounded Slack retry wrapper and wrapped-error unwrapping, but make failed-count correction generic over `ReplyDispatchKind`: subtract failed counts for `tool`, `block`, and `final`, pass the full corrected count record to the shared helper, add a tool-only Slack regression test, and add the required Unreleased changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is a Slack turn with `counts.tool > 0`, no block/final counts, no queued final, and no fallback signal; current main's helper treats it as visible, while the PR's corrected snapshot would pass only block/final counts and mark it undelivered.

## Solution Assessment

No, not as-is. The retry and Slack Web API wrapped-error handling are narrow and supported by dependency source, but the accounting fix should preserve the full shared dispatch-count contract instead of adapting only block/final counts.
