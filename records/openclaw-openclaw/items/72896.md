## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks the proposed Slack `chat.postMessage` transient retry path and still derives visible Slack delivery from queued counts without failed-count input, but the proposed retry classifier misses the real `@slack/web-api` wrapped network-error shape for the Node error codes it claims to handle.

## What This Changes

The PR adds bounded Slack `chat.postMessage` retries for selected transient socket/network failures, adjusts Slack delivery accounting to ignore failed queued replies, and adds focused Slack send and preview-fallback tests.

## Best Possible Solution

Land a rebased Slack plugin fix that keeps write retries bounded, recognizes `@slack/web-api` wrapped request errors such as `err.original.code`, and adapts failed-visible-reply accounting to current main's shared `hasVisibleInboundReplyDispatch` path with focused tests for wrapped request errors and failed queued final replies.

## Reproduction Assessment

Yes. The PR supplies focused unit reproduction paths for mocked transient retry and swallowed-delivery accounting, and current-main source inspection confirms those behaviors are not implemented; the remaining gap is that the network-code test shape does not match real wrapped WebClient errors.

## Solution Assessment

No, not as-is. The direction is narrow and useful, but the retry classifier should inspect wrapped Slack WebClient request errors and the dispatch-accounting change should be rebased onto current main's shared turn-dispatch helper.
