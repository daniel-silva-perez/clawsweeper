## Summary

Keep this PR open. The retry/unwrapping direction is supported by current Slack write-client settings and the Slack SDK error shape, but the latest PR head is still not merge-ready because its failed-count correction drops delivered `tool` counts even though shared turn-delivery semantics treat tool dispatches as visible replies.

## What This Changes

The PR adds bounded Slack `chat.postMessage` retries for transient socket/Web API request errors, adjusts Slack reply-delivery accounting with dispatcher failed counts, and adds focused Slack tests.

## Best Possible Solution

Update the PR so failed-count correction is generic over `ReplyDispatchKind`: subtract failed counts for `tool`, `block`, and `final`, pass the full corrected count record into `hasVisibleInboundReplyDispatch`, and add a Slack regression test for a delivered tool-only reply path. The bounded retry wrapper and Slack wrapped-error unwrapping can remain if maintainers accept the retry policy.

## Reproduction Assessment

Yes. A source-level reproduction is a Slack turn with `counts.tool > 0`, no block/final counts, and no observed/fallback signal: current main's helper marks it visible, while the PR head passes only block/final counts and treats it as undelivered.

## Solution Assessment

No, not as-is. The retry and wrapped-error fixes are narrow, but the accounting fix should preserve the full shared dispatch-count contract instead of adapting only block/final counts.
