## Summary

Keep open. Current main has relevant Feishu WebSocket hardening and generic vLLM/provider timeout plumbing, but the issue reports a Feishu-specific embedded run timeout with vLLM while WebUI works. The available discussion and current source do not prove that the reporter's failing path is fixed, and the issue is too recent for a stale-insufficient-info close.

## Best Possible Solution

Keep this issue open for maintainer triage. Ask the reporter to retest on current main or the latest release, confirm webhook versus WebSocket mode, provide gateway logs around the failed run, share vLLM first-token/stream latency, and include `models.providers.vllm.timeoutSeconds`, `agents.defaults.timeoutSeconds`, Docker networking, and proxy environment details. If logs show the WebSocket startup failure from #45674, close after confirmed retest; if the message reaches `dispatchReplyFromConfig` and still times out, trace the Feishu reply dispatcher plus vLLM guarded-fetch/idle-timeout path before changing core.
