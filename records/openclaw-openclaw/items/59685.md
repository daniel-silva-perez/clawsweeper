## Summary

Keep this PR open. Current main does not fully implement the branch’s live-gateway memory fallback or runtime/embedding split; it mainly avoids the original human status false negative by skipping memory inspection outside `--all`, while some status paths still rely on local memory runtime materialization. The discussion also includes useful independent reproduction context, so the work remains potentially valuable but needs a current-main rebase and maintainer review.

## What This Changes

The PR adds a live gateway memory-status fallback to `openclaw status`, splits `doctor.memory.status` into runtime and embedding health, and restores cached memory plugin state when reusing the active plugin registry.

## Best Possible Solution

Rebase or replace this with a current-main change that preserves the newer non-probing doctor semantics, adds a narrow live-gateway runtime/provider signal when local memory status is inconclusive, and validates any loader state-restoration change separately.

## Reproduction Assessment

Yes. The linked report and follow-up comment describe a concrete setup where a live memory plugin works through the gateway while status reports it unavailable, and the current source still shows local memory materialization paths that can diverge from live gateway state.

## Solution Assessment

Unclear. Trusting the live gateway runtime is the right direction, but this branch combines protocol, status UI, and loader-cache changes against older main; a smaller current-main rebase or split PR would be safer to review.
