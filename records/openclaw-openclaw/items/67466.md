## Summary

Keep open. Current main still lacks the useful Feishu status/startup timeout caps, but this PR is not safe to merge as-is because it detaches the gateway-managed Feishu account monitor from the long-lived startAccount promise and weakens probe diagnostics. Related open Feishu work should be coordinated rather than closing this PR as obsolete.

## What This Changes

The PR caps Feishu startup/status probes at 3000ms, parallelizes multi-account startup preflight, and starts Feishu account monitors as background tasks instead of awaiting them.

## Best Possible Solution

Preserve the long-lived monitorFeishuProvider({ accountId }) contract, extract or rework the 3000ms setup/status probe cap with focused coverage, and coordinate startup probe behavior with the open Feishu multi-account/rate-limit work so responsiveness improves without losing gateway lifecycle ownership.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Keep the account monitor promise long-lived:** `extensions/feishu/src/monitor.ts:87-94`
  - body: The Feishu gateway startAccount hook returns monitorFeishuProvider({ accountId }), and server-channels treats that returned promise as the lifetime of the running account. This branch now starts monitorSingleAccount in the background and returns immediately, so each account is marked exited/running:false and enters the restart loop while the real monitor is no longer tracked by the gateway task/abort state.
  - confidence: 0.96
- **[P2] Preserve account IDs on rejected startup probes:** `extensions/feishu/src/monitor.ts:122-131`
  - body: When fetchBotIdentityForMonitor rejects, Promise.allSettled only gives this branch the error, not the account that was being probed. The warning therefore logs feishu[unknown], which loses the key diagnostic needed to identify the failing account; catch inside the per-account map and return a degraded result with the account attached.
  - confidence: 0.88
