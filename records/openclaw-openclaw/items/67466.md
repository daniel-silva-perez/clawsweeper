## Summary

Keep this PR open. Current main still does not implement the PR's focused Feishu setup/status 3000ms probe cap or startup timeout reduction, while the PR as submitted is unsafe because it makes the gateway-managed single-account Feishu monitor return immediately instead of preserving the long-lived startAccount lifecycle promise.

## Best Possible Solution

Keep this PR open for maintainer rework. Preserve the long-lived `monitorFeishuProvider({ accountId })` return contract, extract the setup/status 3000ms timeout cap with focused regression coverage, and coordinate any startup probe concurrency or timeout change with the related Feishu multi-account work in #63481/#63475 so the gateway lifecycle remains tracked.

## Risks / Open Questions

- Merging the PR as-is would detach gateway-managed Feishu account monitors from the long-lived `startAccount` promise, causing restart/backoff behavior while the real monitor may continue in the background.

- Closing this PR now would drop still-unimplemented useful work, especially the setup/status 3000ms probe cap, unless that smaller change is extracted or tracked elsewhere.
