## Summary

Keep this PR open. Current main still lacks the PR's focused Feishu setup/status probe timeout cap and still leaves the multi-account startup preflight on the 30s default path, while the PR itself is not safe to merge as-is because it detaches the gateway-managed single-account monitor from the long-lived lifecycle promise.

## Best Possible Solution

Keep this PR open for maintainer rework rather than conservative cleanup closure. The best path is to preserve the long-lived `monitorFeishuProvider({ accountId })` return contract, coordinate the multi-account bot-info probe behavior with the related open Feishu startup work (#63481/#63475) or an equivalent lifecycle-safe queue, and extract the setup/status 3000ms timeout cap with focused regression coverage.

## Risks / Open Questions

- Merging the PR as-is would make gateway-managed Feishu accounts appear to exit immediately, triggering restart/backoff behavior while the real monitor may continue running outside tracked task ownership.

- Closing this PR now would drop a still-unimplemented, useful setup/status 3000ms probe cap unless that smaller change is extracted or tracked elsewhere.
