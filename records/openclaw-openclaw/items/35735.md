## Risks / Open Questions

- The proposed command is destructive: it mutates session stores and removes transcript files, so unresolved correctness comments should be addressed before merge.
- The PR implementation routes deletion through `updateSessionStore`; current `saveSessionStoreUnlocked` can run maintenance during writes, which can make real execution remove unrelated sessions while dry-run/output only reports predicate matches.
- The PR removes transcript files directly and reports paths as removed even when `fs.promises.rm` fails, which can mislead users about cleanup success.
- No supply-chain or CI execution regression is visible in the diff, but local filesystem deletion is safety-sensitive behavior that needs careful tests and maintainer review.
