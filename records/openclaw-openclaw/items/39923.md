## Summary

Keep this issue open. Current main still uses the numbered `.bak` / `.bak.1` through `.bak.4` config backup ring, cleanup still prunes non-numbered `.bak.*` files, and open PRs #39982 and #40392 explicitly target this issue as implementation candidates. PR #40392 is also by the same author as this issue, so closing the issue before that paired work is resolved would be premature.

## Best Possible Solution

Keep this issue open until a datetime-suffix backup implementation lands on main or maintainers explicitly decide to retain the numbered backup ring. The best path is to review and either land or close the open implementation candidates, especially the same-author PR #40392, with coverage for bounded timestamp pruning, permission discovery, collision handling, and legacy `.bak.N` migration. The broader #20460 can remain the place to discuss moving backups into a separate directory.
