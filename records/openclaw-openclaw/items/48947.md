## Summary

Keep this issue open. Current main has important partial fixes for adjacent failures: Control UI now receives live tool events, enabled loop detection can catch repeated exec no-progress results, and parameterless tools normalize null arguments. But the reported default web UI/OpenRouter runaway/no-feedback problem is not fully resolved: loop detection is still opt-in, `max_tool_calls` is accepted but ignored on the OpenResponses surface, and there is no general default tool-call budget or forced progress-summary circuit breaker that satisfies the core complaint.

## Best Possible Solution

Keep the issue open and narrow it into an actionable runtime/UI reliability fix: add a default-safe runaway guard or documented per-agent tool-call budget, force a progress or failure summary after repeated tool cycles without assistant-visible progress, and verify the Control UI refresh path with an OpenRouter/web UI reproduction. The already-shipped #34574 and #72587 fixes should be noted as partial mitigation, not as closure for this broader report.
