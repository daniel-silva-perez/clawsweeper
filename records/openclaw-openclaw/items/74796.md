## Summary

Keep open. Current main still routes shared thinking-policy lookup through startup-active provider hooks, while the OpenAI plugin is enabled by default but not startup-active; `openai-codex/gpt-5.5` therefore still depends on provider runtime visibility that main does not provide. The issue also has an open closing PR, #75295, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Make gateway/session thinking validation and metadata resolve provider-owned OpenAI Codex thinking profiles for startup-inactive enabled provider plugins so `openai-codex/gpt-5.5` accepts and advertises `xhigh`.

## Best Possible Solution

Land #75295 or an equivalent generic provider-runtime thinking-profile fallback that keeps the OpenAI Codex model rules inside the provider plugin, then cover gateway defaults, session rows, `sessions.patch`, and agent command validation for startup-inactive provider plugins.

## Reproduction Assessment

Yes. The report includes concrete config and commands from OpenClaw 2026.4.27, and current-main source still shows the same missing-provider-profile path for gateway/session validation without needing a live gateway run.

## Solution Assessment

Yes. Resolving provider thinking hooks from the provider runtime when the owning plugin is enabled but not startup-active is the narrow maintainable fix; hardcoding Codex `xhigh` in core or forcing broad OpenAI startup activation would weaken the plugin boundary.
