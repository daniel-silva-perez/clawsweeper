## Summary

Keep open. Current main still has the provider-profile visibility gap described in the issue: OpenAI Codex declares `xhigh` for `gpt-5.5`, but the shared thinking resolver used by gateway/session validation only sees active-registry provider hooks or catalog compat metadata, and the OpenAI plugin is enabled but not startup-active. The issue discussion also includes a second live 2026.4.27 reproduction showing the same hard rejection before Codex is reached.

## What This Changes

The issue reports that gateway/session thinking validation rejects or downgrades `xhigh` for `openai-codex/gpt-5.5` because the OpenAI provider thinking profile is not visible before plugin startup activation.

## Best Possible Solution

Add a generic provider-owned thinking-profile lookup that works for configured/default model validation before startup activation, then use it consistently across gateway session rows, `sessions.patch`, agent command validation, directive validation, and plugin runtime policy helpers without hardcoding OpenAI Codex model rules in core.

## Reproduction Assessment

Yes. The issue and follow-up give concrete 2026.4.27 commands and config that reject `--thinking xhigh`, and current-main source shows the same failure path when the OpenAI provider hook is absent from the active registry and no catalog compat row supplies `xhigh`.

## Solution Assessment

Yes, the requested fix direction is the maintainable one: expose provider thinking policy through a lightweight/runtime lookup or equivalent generic plugin contract. Startup-activating the whole OpenAI plugin or adding Codex-specific `xhigh` checks in shared core would solve this symptom with a weaker boundary.
