## Summary

Keep open. Current main still has the reported gateway gap: OpenAI Codex declares `xhigh` for `gpt-5.5`, but the shared thinking resolver only sees provider hooks from the active startup registry or catalog compat metadata, and the gateway session/agent paths use that resolver before a non-startup provider is necessarily active. Existing coverage proves active-registry and configured-catalog cases, but not the enabled bundled OpenAI plugin with `onStartup: false` case described here.

## What This Changes

Ensure gateway thinking validation and session metadata for `openai-codex/gpt-5.5` accept and advertise `xhigh` from the provider-owned thinking profile.

## Best Possible Solution

Add a generic provider-profile resolution path that can answer thinking-policy questions for configured or provider-owned models even when the owning provider plugin is enabled but not startup-active, then share that path across gateway session rows/defaults, `sessions.patch`, and agent-run validation. Cover `openai-codex/gpt-5.5` specifically without hardcoding OpenAI policy into core.
