## Summary

Keep open. Current main still has the reported gateway/provider-profile gap: the OpenAI Codex provider declares `xhigh` for `gpt-5.5`, but the shared thinking resolver only sees provider hooks from the active runtime registry or catalog compat metadata. The OpenAI plugin is enabled by default but not startup-active, so gateway session metadata, session patch validation, and agent validation can still omit or reject `xhigh` for the enabled bundled provider path described in the report.

## What This Changes

The issue asks OpenClaw to accept and advertise `xhigh` for `openai-codex/gpt-5.5` in gateway-backed sessions when the OpenAI plugin is enabled but not startup-active.

## Best Possible Solution

Add a generic provider-profile resolution path that can answer thinking-policy questions for configured or provider-owned models even when the owning enabled provider plugin is not startup-active. Share that path across gateway session rows/defaults, `sessions.patch`, and agent-run validation, and keep OpenAI-specific model policy inside the OpenAI plugin or a plugin-owned lightweight artifact rather than hardcoding it in core.

## Reproduction Assessment

Yes. The issue includes concrete config and command-line reproduction on v2026.4.27, a second live reproduction confirms the same hard rejection in local, gateway, and subagent paths, and current main source still has the active-registry-only resolver path that explains the failure.

## Solution Assessment

Yes, the suggested direction is the right boundary: expose provider-owned thinking profiles through a generic lightweight/provider-runtime lookup that works before startup activation. Eagerly startup-activating the whole OpenAI plugin or hardcoding Codex xhigh rules in core would be a less maintainable fix.
