## Summary

Keep open. Current main still has the reported gateway gap: OpenAI Codex declares `xhigh` for `gpt-5.5`, but the shared thinking resolver only sees provider hooks from the active startup registry or catalog compat metadata, while the bundled OpenAI plugin is enabled but not startup-active. Gateway session metadata and agent validation consume that shared resolver, so the reported downgrade/rejection remains source-backed.

## What This Changes

The issue requests that gateway session metadata and validation accept and advertise `xhigh` for `openai-codex/gpt-5.5` when the OpenAI provider plugin is enabled but not startup-active.

## Best Possible Solution

Add a generic provider-profile resolution path that can answer thinking-policy questions for configured or provider-owned models even when the owning provider plugin is enabled but not startup-active, then reuse it across gateway session defaults/rows, `sessions.patch`, and agent-run validation. Cover `openai-codex/gpt-5.5` specifically without hardcoding OpenAI policy into core.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: with no active OpenAI provider hook and no catalog compat row, `listThinkingLevelOptions("openai-codex", "gpt-5.5")` falls back to base levels, and the gateway rows/agent validation use that result.

## Solution Assessment

No patch is under review. The best direction is a generic lightweight provider-profile resolver or targeted activation seam; a core OpenAI whitelist or broad eager provider activation would be a weaker fix.
