## Summary

Keep open. Current main still matches the reported gateway/provider-profile gap: OpenAI Codex declares `xhigh` for `gpt-5.5`, but the shared thinking resolver only sees active-registry provider hooks or catalog compat metadata, and the OpenAI plugin remains startup-lazy without manifest compat metadata for this model.

## What This Changes

Make gateway session metadata, session patch validation, and agent thinking validation include and accept `xhigh` for `openai-codex/gpt-5.5` when the OpenAI plugin is enabled but not startup-active.

## Best Possible Solution

Add a generic provider-profile resolution path that can answer thinking-policy questions for configured provider-owned models before full startup activation. Reuse it across session defaults, session patch validation, and agent-run validation, keeping OpenAI Codex model policy in the OpenAI plugin or a lightweight plugin-owned artifact.

## Reproduction Assessment

Yes. The issue and follow-up provide concrete v2026.4.27 reproduction commands/config, and current main still has the active-registry-only resolver path that explains the rejection when the OpenAI plugin is not startup-active.

## Solution Assessment

Yes. The proposed direction is the narrow maintainable boundary: expose provider-owned thinking profiles through a generic lightweight/runtime lookup, rather than eager-loading the full OpenAI plugin or duplicating Codex `xhigh` rules in core.
