## Summary

Keep open. Current main still has the reported gateway/provider-profile visibility gap: the OpenAI Codex provider declares `xhigh` for `gpt-5.5`, but shared thinking resolution only consults the active startup plugin registry plus catalog compat fallback, and the bundled OpenAI plugin is enabled by default while explicitly not startup-active. Gateway session metadata and patch validation continue to use that shared resolver, so the issue remains a valid, narrow source-backed bug.

## What This Changes

Make gateway session metadata and validation accept and advertise `xhigh` for `openai-codex/gpt-5.5` even when the OpenAI provider plugin has not been startup-activated.

## Best Possible Solution

Add a generic provider-owned thinking-profile lookup that works for configured/default model validation before startup activation, then reuse it across gateway session rows, `sessions.patch`, agent command validation, and plugin runtime thinking policy helpers without embedding OpenAI Codex-specific rules in core.

## Reproduction Assessment

Yes. The issue body and follow-up comment provide concrete 2026.4.27 configs and commands that reject `--thinking xhigh`; current-main source still shows the fallback path that omits `xhigh` when the OpenAI provider hook is not active.

## Solution Assessment

Yes, the suggested shared/provider-profile plumbing is the maintainable direction. Startup-activating the whole OpenAI plugin or adding a core `openai-codex/gpt-5.5` special case would solve the symptom more narrowly but weaken the plugin contract.
