## Summary

Keep open. Current main still matches the reported failure mode: the OpenAI Codex provider declares xhigh for gpt-5.5, but the shared thinking resolver used by gateway/session validation only sees provider hooks from the active startup registry, and the bundled OpenAI plugin is explicitly not startup-active. Later main changes touched Codex OAuth transport, not this thinking-profile lookup path.

## What This Changes

The issue asks gateway/session thinking validation and metadata to accept and advertise xhigh for openai-codex/gpt-5.5 instead of rejecting or remapping it to high.

## Best Possible Solution

Add a generic provider-owned thinking-profile lookup that works for configured/default model validation before startup activation, then share it across gateway session rows, sessions.patch, agent command validation, subagent/session metadata, and plugin runtime policy helpers without hardcoding OpenAI Codex model rules in core.

## Reproduction Assessment

Yes. The issue and follow-up provide concrete gateway/local/subagent commands that reject xhigh on 2026.4.27, and current-main source still shows the same inactive-provider fallback path that would omit xhigh.

## Solution Assessment

Yes, the issue's suggested generic provider-profile/runtime lookup is the right direction. Startup-activating the full OpenAI plugin or adding Codex-specific xhigh checks in core would fix this symptom but would be a weaker plugin contract.
