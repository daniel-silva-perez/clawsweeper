## Summary

Keep open. Current main has partial heartbeat exec-completion cleanup, but it still treats structured `code 0` and `signal SIGTERM` completions with captured output as relayable, so the PR’s core user-visible fix is not implemented on main.

## What This Changes

The PR adds local-exec provenance to queued exec completion events, filters trusted structured `code 0` and `signal SIGTERM` completions out of heartbeat user relay, suppresses visible OK/reasoning delivery for those internal-only completions, JSON-encodes relayable exec output in prompts, and adds focused bash/heartbeat regression tests.

## Best Possible Solution

Land a rebased narrow heartbeat fix that marks only trusted local structured `code 0` and benign cleanup `signal SIGTERM` exec completions internal, preserves relay for actionable failures, and leaves the broader system-event audience model to #69492 unless maintainers choose to fold this slice into that design.

## Reproduction Assessment

Yes. The code-path reproduction is to enqueue the three structured exec completion shapes from the PR body and inspect heartbeat prompt/delivery behavior; current main still routes outputful `code 0` and `SIGTERM` completions through the relayable exec path.

## Solution Assessment

Yes, with rebase caveats. The PR's trust-gated local-exec provenance plus per-event filtering is a narrow maintainable tactical fix, but it should be reconciled with current main's newer relayability helper and maintainer policy for SIGTERM suppression.
