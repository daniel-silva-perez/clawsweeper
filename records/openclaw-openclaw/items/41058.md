## Summary

Keep open. Current main still emits the hardcoded OpenClaw-branded pairing reply and CLI approval hint, strict channel schemas and exported config types do not expose `pairingMessage`, and open PR #42617 uses closing syntax for this issue, so the issue should remain open until that implementation candidate is reviewed and merged or closed.

## What This Changes

Request to add optional per-channel `channels.*.pairingMessage` fields so operators can customize pairing challenge text and suppress the raw `openclaw pairing approve ...` CLI hint.

## Best Possible Solution

Review PR #42617 as the active implementation candidate. The durable end state should be a typed, strict-schema-validated, documented `pairingMessage` config surface that preserves current default text, supports CLI-hint suppression, and is either applied consistently across pairing-capable channels or intentionally staged with clear follow-up tracking.

## Reproduction Assessment

Yes. Static reproduction on current main is high-confidence: any unapproved sender hitting `dmPolicy: "pairing"` flows through `issuePairingChallenge` to `buildPairingReply`, which emits the OpenClaw branding and CLI hint, while strict schemas/types still omit `pairingMessage`.

## Solution Assessment

Unclear until maintainers review #42617. A typed optional config with backward-compatible defaults is the narrow maintainable direction, but maintainers still need to decide whether the PR’s iMessage/Telegram-first scope is acceptable for a request that also mentions other pairing-capable channels.
