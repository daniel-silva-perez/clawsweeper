## Summary

Keep open. Current main still implements the sandbox browser CDP path with the socat relay, CDP port offset, and cdpSourceRange ingress model that this issue asks maintainers to evaluate; the related readiness hardening improved startup behavior but did not remove the relay architecture.

## What This Changes

Evaluate whether the sandbox browser CDP path can remove the socat proxy and route CDP directly to Chromium while preserving readiness, exposure control, and documentation.

## Best Possible Solution

Keep this as the core sandbox browser CDP architecture tracker. If maintainers choose direct CDP, remove socat and the CDP port-offset logic from the image and entrypoint, update host orchestration, config hashing, tests, and docs, and preserve equivalent access-control and readiness guarantees. If the relay remains necessary, document that rationale and close the issue after that decision.

## Security Review

Status: needs_attention

Summary: No patch is under review, but the proposed direct-CDP direction touches CDP exposure controls and needs an explicit replacement for the current relay source-range guard.

Concerns:

- **[medium] Preserve CDP ingress restriction before removing socat:** `scripts/sandbox-browser-entrypoint.sh:181`
  - body: The current entrypoint applies `range=${CDP_SOURCE_RANGE}` on the socat listener while Chromium binds CDP to loopback on a derived port. A direct-CDP implementation needs an equivalent container-edge restriction or a documented security decision, otherwise browser CDP could become reachable from a broader Docker network surface.
  - confidence: 0.86
