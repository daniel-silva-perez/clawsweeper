## Summary

Keep open. Current main still uses the sandbox browser socat relay, derived Chromium CDP port, and cdpSourceRange ingress model that this issue asks maintainers to evaluate; the related hardening PR shipped readiness/lifecycle fixes but deliberately left the relay architecture in place.

## What This Changes

Evaluate whether the sandbox browser CDP path can remove the socat relay and route CDP directly to Chromium while preserving readiness and access-control guarantees.

## Best Possible Solution

Keep this as the sandbox browser CDP architecture tracker. If maintainers accept direct CDP, remove socat and the CDP port-offset logic from the image, entrypoint, orchestration, config hashing, docs, and tests while preserving readiness checks, loopback host publishing, bridge/auth controls, and equivalent CDP ingress restrictions; if the relay remains necessary, document that decision and close the tracker then.

## Reproduction Assessment

Not applicable as a feature/evaluation request rather than a runtime bug. Current-main static verification gives high confidence that the design under evaluation still exists: socat is installed, the entrypoint starts the relay, and tests/docs cover cdpSourceRange-based relay behavior.

## Solution Assessment

Unclear until maintainer architecture/security review decides whether direct CDP is the desired end state. The narrow maintainable path is either a deliberate direct-CDP migration with equivalent security/readiness guarantees, or explicit documentation that the relay remains intentional.
