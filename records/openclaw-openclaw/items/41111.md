## Summary

Keep open. Current main and the latest release still expose only `openclaw backup create` and `openclaw backup verify`; the requested native archive restore command has not shipped. The keepmyclaw ClawHub plugin is useful prior art and a workaround, and open PR #59280 is the direct implementation candidate, but neither resolves the first-party CLI gap yet.

## What This Changes

The issue requests a native `openclaw backup restore <archive>` command to restore from an OpenClaw backup archive.

## Best Possible Solution

Keep the issue open until OpenClaw ships a safe native `openclaw backup restore <archive>` workflow or maintainers explicitly consolidate it into a canonical backup/restore tracker. The implementation should live in the backup CLI surface with restore infra, registration tests, command behavior tests, and docs at `docs/cli/backup.md`.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to inspect current main's backup command registration, command tests, docs, and CLI index; all show create/verify only and no native archive restore command.

## Solution Assessment

Yes, the best direction is a first-party restore command if it includes conservative safety behavior. The plugin workaround is useful, but it does not complete the built-in backup lifecycle that core already started with create and verify.
