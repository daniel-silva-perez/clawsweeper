## Summary

Keep open. Current main still has the per-agent override path this PR is trying to remove: provider auth returns `agentModelOverride` when `setDefaultModel=false`, and `openclaw agents add` persists that value into the new agent entry. The author’s note that this was merged elsewhere is useful context, but the current source and tests still contradict the intended behavior.

## What This Changes

The PR changes provider/auth-choice model setup so provider `defaultModel` is not returned as an agent model override when an inherited `agents.defaults.model` primary exists, and updates related auth-choice tests.

## Best Possible Solution

Keep the work open and land a rebased, focused fix that preserves provider config/auth-profile/allowlist updates, restores `agents.defaults.model`, and suppresses `agentModelOverride` plus the agent-scoped default-model note when an inherited primary model exists. Only return a provider default override when there is no inherited primary model, and keep regression coverage for both branches.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is available: start with `agents.defaults.model.primary`, run provider auth through the `openclaw agents add` path with `setDefaultModel=false`, and current main returns an override that `agents add` writes as the new agent's `model`.

## Solution Assessment

Yes, the PR direction is the narrow maintainable fix: change the deferred default-model paths to distinguish inherited defaults from missing defaults, instead of changing runtime model resolution. It still needs normal maintainer review/rebase and cleanup of any stale test-only changes before merge.
