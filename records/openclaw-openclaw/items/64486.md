## Summary

Current main still implements an append-only transparency guard for unsupported reminder/follow-up commitments. The detector and scheduling signal exist, but the auto-reply path preserves the commitment text and appends the note when no cron.add succeeded; there is no pre-send forced revision or concrete scheduling gate.

## Best Possible Solution

Keep this issue open. The best implementation is a bounded pre-send gate in the auto-reply finalization path that reuses the existing commitment detector plus successfulCronAdds and session-cron coverage, then allows delivery only after the agent removes unsupported commitment language or creates a concrete cron/task/heartbeat-backed action. If maintainers prefer plugin-first, first extend the documented reply/finalization hook API to expose the required scheduling evidence.
