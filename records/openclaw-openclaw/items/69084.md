## Summary

Keep this PR open for revision. The reported webchat heartbeat noise is plausible, but the proposed one-line gate collapses `showOk` and `showAlerts`, which would regress the documented default behavior and the intentional OK-only configuration path.

## What This Changes

The PR changes gateway webchat heartbeat suppression from `showOk` to `showAlerts` and updates the gateway agent-event tests to expect that behavior.

## Best Possible Solution

Keep the two visibility contracts separate in the gateway chat projection: `showOk` should control OK-only heartbeat acknowledgments, `showAlerts` should control non-OK alert content, and the webchat path should classify normalized heartbeat final text before deciding what to suppress or deliver.

## Reproduction Assessment

Yes. The current gateway test harness already models a heartbeat run with `showOk=false` and `showAlerts=true`; applying the PR's gate would stop suppression in that default case and make ACK text visible.

## Solution Assessment

No. The proposed `showAlerts` swap is not the narrowest maintainable fix because it loses the documented `showOk` behavior; a safer fix should classify ACK-only versus alert heartbeat output and add tests for default, alerts-off, and OK-only configurations.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep heartbeat ACK and alert visibility separate:** `src/gateway/server-chat.ts:61`
  - body: With the default visibility (`showOk=false`, `showAlerts=true`), this gate now returns false, so `HEARTBEAT_OK` acknowledgments and prefixes bypass the existing normalization path and can appear in webchat. The opposite mixed config (`showOk=true`, `showAlerts=false`) also suppresses OK-only acknowledgments even though `showOk` explicitly enables them. Please preserve `showOk` for ACK/noise handling and apply `showAlerts` only to non-OK alert content.
