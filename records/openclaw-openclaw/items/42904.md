## Summary

Keep this PR open for maintainer rework. Current main fixes the generic `openclaw logs` / `logs.tail` cross-date path and writer-side daily rollover, but `openclaw channels logs` still reads only the exact configured log file and returns no lines when that file is missing. The PR also bundles unrelated cron warning work and its fallback approach needs tightening for custom `logging.file` paths.

## Best Possible Solution

Keep this PR open for maintainer rework. Split the cron warning commit back to the cron issue/PR path, then implement channel-log fallback by reusing `readConfiguredLogTail` or extracting a shared rolling-log resolver before applying channel filtering. Add focused `channelsLogsCommand` tests for missing current-day rolling files, current-day precedence, custom/non-rolling `logging.file`, JSON `file` output, and filtering against the resolved fallback file.

## Risks / Open Questions

- Merging the PR as-is could make `openclaw channels logs` read the newest `openclaw-*.log` sibling even when `logging.file` is custom or non-rolling, which can show stale or unrelated log lines from the same directory.
- The PR mixes the channel-log fix with unrelated cron warning behavior tracked separately, making review and landing less clean.
- The provided PR diff does not touch workflows, lockfiles, dependencies, lifecycle scripts, package publishing metadata, or release automation, so the main security concern is the log-source confusion/data exposure risk from the broad fallback scan rather than supply-chain execution.
