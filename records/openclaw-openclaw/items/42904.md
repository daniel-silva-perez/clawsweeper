## Summary

Keep this PR open. Current main fixes the generic `openclaw logs` / `logs.tail` path and writer-side daily rollover, but it still does not implement the PR’s channel-specific active-log fallback: `openclaw channels logs` reads the configured file directly and returns no lines when that file is missing. The PR also still bundles unrelated cron warning work that is already tracked by the open cron issue/PR pair, and its fallback needs to be narrowed before merge.

## Best Possible Solution

Keep this PR open for maintainer rework. Split the cron warning back to the existing cron PR, then make `openclaw channels logs` reuse `readConfiguredLogTail` or a shared exact-pattern rolling-log resolver before channel filtering. Add focused channel-log tests for missing current-day rolling files, current-day precedence, custom/non-rolling `logging.file`, and filtering against the resolved fallback file.

## Risks / Open Questions

- Merging the PR as-is would mix the channel-log fix with unrelated cron CLI warning behavior already tracked by #42196 / #42245.
- The PR fallback described in the provided diff scans for the newest `openclaw-*.log` in the configured directory whenever the configured file is missing; for custom or non-rolling `logging.file` setups, that can read stale or unrelated local log lines instead of the configured source.
- Security pass: the PR does not add dependencies, workflows, lockfiles, install scripts, release metadata, or downloaded code execution paths, but the broad fallback log selection has a local information-disclosure risk because it can display the wrong log file.
