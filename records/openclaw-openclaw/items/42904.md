## Summary

Keep this PR open for maintainer follow-up. Current main has improved the underlying rolling logger so long-lived gateway loggers move to the current-day file, and generic `logs.tail` already has missing-current-file fallback. But current main still does not implement this PR's specific channel-log change: `channelsLogsCommand` reads `getResolvedLoggerSettings().file` directly and does not reuse the rolling resolver, while the latest PR context shows a narrow branch that exports that resolver, uses it for `openclaw channels logs`, and adds focused regression tests.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to either land the narrow current branch direction, reusing the existing rolling-log resolver in `channelsLogsCommand` with the focused regression tests and changelog credit, or have a maintainer explicitly close it if the newer writer-side rollover fix is considered sufficient and channel-log fallback parity is not wanted.
