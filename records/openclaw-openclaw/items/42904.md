## Summary

Keep this PR open for maintainer follow-up. Current main fixes the generic `openclaw logs` / `logs.tail` cross-date path and writer-side daily rollover, but `openclaw channels logs` still reads only the exact configured log file and returns no lines when that file is missing. The PR also bundles unrelated cron warning work and its fallback approach still has a custom `logging.file` log-source confusion risk noted in review.

## Best Possible Solution

Keep this PR open for maintainer rework. Split the cron warning commit out, then implement `openclaw channels logs` on top of the shared rolling-log resolver, ideally by reusing `readConfiguredLogTail` or extracting a shared safe resolver before channel filtering. Add focused channel-log tests for missing current-day rolling files, current-day precedence, custom/non-rolling `logging.file`, JSON `file` output, and filtering against the resolved fallback file.
