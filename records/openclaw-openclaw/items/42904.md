## Summary

Keep this PR open. Current main still has the `openclaw channels logs` gap: it reads the exact configured logger file and returns no lines when that file is missing, while the generic `logs.tail` path already has a rolling-log fallback. The latest PR context is a narrow implementation candidate with focused tests and no security-sensitive file changes.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to land the narrow current branch if CI stays green: export the shared rolling-log resolver, call it from `channelsLogsCommand`, keep the focused regression tests for rolling and custom non-rolling log paths, and preserve the contributor credits. If maintainers decide command-level fallback parity is not wanted, that should be an explicit product decision rather than automated cleanup.
