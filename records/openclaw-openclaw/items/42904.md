## Summary

Keep this PR open. Current main still has the `openclaw channels logs` path reading the exact configured log file directly, so it misses the existing rolling-log fallback that `openclaw logs` and `logs.tail` already use. The PR remains a narrow implementation candidate for that remaining channel-log gap.

## Best Possible Solution

Keep this PR open for maintainer review and landing. The best path is to merge the narrow current branch direction: export and reuse the existing rolling-log resolver in `channelsLogsCommand`, keep fallback limited to default dated rolling log files, preserve custom non-rolling `logging.file` behavior, and land the focused regression tests plus changelog credit.
