## Summary

Keep this PR open. Current main covers the generic `openclaw logs` / `logs.tail` rolling-log fallback and writer-side rollover, but `openclaw channels logs` still reads the exact configured file and returns no lines when that file is missing. The PR also bundles unrelated cron warning work that is already tracked separately, and its channel-log fallback is broader than the safe rolling-log resolver on main.

## Best Possible Solution

Keep this PR open for maintainer rework rather than closing it as implemented. Split the cron warning commit back to the existing cron issue/PR path, then implement channel-log fallback by reusing `readConfiguredLogTail` or extracting a shared exact-pattern rolling-log resolver before channel filtering. Add focused `channelsLogsCommand` tests for missing current-day rolling files, current-day precedence, custom/non-rolling `logging.file`, and filtering against the resolved fallback file.
