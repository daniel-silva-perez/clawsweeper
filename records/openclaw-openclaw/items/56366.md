## Summary

Keep this issue open. Current main still loads and watches the personal `~/.agents/skills` source unconditionally, and the public skills config surface only offers extra directories, watcher toggles, prompt limits, allowlists, and per-skill disables after discovery. The related #64639 discussion supports the same short-term source-disable toggle and was treated as duplicate of this older canonical issue, so this is a good narrow repair candidate rather than cleanup to close.

## What This Changes

Request to add a configuration option that disables scanning built-in skill roots, especially `~/.agents/skills`, to avoid Claude Code skill conflicts and prompt bloat.

## Best Possible Solution

Add a backwards-compatible skills load setting that skips disabled built-in source roots before filesystem discovery and watcher registration, while preserving current defaults and precedence when unset. Keep native on-demand skill selection as a separate design discussion rather than bundling it into this short-term coexistence fix.

## Reproduction Assessment

Yes. The current source and existing fixture tests provide a high-confidence reproduction path: a skill under a mocked `~/.agents/skills` is loaded into the prompt, and the watcher registers that home path with no config guard.

## Solution Assessment

Yes for the short-term issue. A source-level opt-out is narrower and more maintainable than requiring dozens of `skills.entries.*.enabled: false` entries; per-agent allowlists and prompt limits are useful workarounds but do not stop discovery or watching.
