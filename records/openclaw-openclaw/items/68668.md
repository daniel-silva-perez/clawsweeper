## Summary

This should stay open because the issue is valid on current main and is already paired with an open, unmerged closing PR. The current lifecycle controller still calls the browser cleanup wrapper from both embedded completion paths without a per-run browser-cleanup dispatch flag, while the paired PR is the right place to review and land that narrow fix.

## What This Changes

The issue requests deduping `cleanupBrowserSessionsForLifecycleEnd` so embedded subagent completion cannot dispatch the browser cleanup wrapper twice for the same `runId`.

## Best Possible Solution

Review and either land or reject PR #68669. The desired end state is a small lifecycle-controller guard that dispatches browser cleanup once per run, resets correctly on rearm/restart, and preserves the existing idempotent browser-tab drain semantics.

## Reproduction Assessment

Yes. The issue's `Promise.all([completeSubagentRun(p), completeSubagentRun(p)])` path maps to the two current production callers that both pass `triggerCleanup: true`; I verified the converging code paths on current main, but did not run tests because this review was read-only.

## Solution Assessment

Yes. A per-record dispatch flag beside `endedHookEmittedAt`, reset on rearm, is the narrowest maintainable fix for the wrapper-level duplicate while leaving browser plugin IPC behavior unchanged.
