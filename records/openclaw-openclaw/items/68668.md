## Summary

Keep open. Current main still has both completion paths reaching `completeSubagentRun(..., triggerCleanup: true)` and the browser cleanup wrapper is invoked without a per-run dispatch guard; the open linked PR #68669 uses closing syntax, so this issue should remain open until that implementation is reviewed and either merged or closed.

## What This Changes

The issue requests a per-run guard so embedded subagent completion invokes `cleanupBrowserSessionsForLifecycleEnd` only once per `runId` when both lifecycle and gateway completion paths fire.

## Best Possible Solution

Review #68669 as the implementation candidate, verify it adds a synchronous per-run cleanup dispatch guard with a rearm reset and regression coverage, then merge it or close it if maintainers prefer a different dedupe point. After that PR merges, this issue can close as implemented.

## Reproduction Assessment

Yes. Current main has a high-confidence source-level reproduction path: invoke `completeSubagentRun` twice in parallel for the same run with `triggerCleanup: true`, as both production completion paths can do, and the unguarded wrapper call is reachable both times.

## Solution Assessment

Yes. A per-record synchronous dispatch guard colocated with the existing `endedHookEmittedAt` pattern is the narrowest maintainable fix for wrapper-level duplication; changing the browser extension would not address duplicated normalization, best-effort cleanup, facade dispatch, or warning overhead.
