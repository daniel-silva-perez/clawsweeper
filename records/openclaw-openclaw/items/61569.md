## Summary

Keep this issue open. Current main still sends the CLI `--node-id` value as `connect.client.instanceId`, but the gateway still derives the effective node ID from the authenticated device ID or client ID in both pairing reconciliation and NodeRegistry registration. That means a node host with a device identity can still be listed/described under the generated device UUID instead of the requested custom node ID. The two related PRs in the provided context identified the same root cause, but both are closed unmerged, so there is no canonical landed fix to close against.

## Best Possible Solution

Land a focused gateway fix that uses a shared resolved node ID helper, preferring a trimmed non-empty `connect.client.instanceId` over `connect.device?.id` and `connect.client.id` in both `reconcileNodePairingOnConnect` and `NodeRegistry.register`. Add tests covering `openclaw node run --node-id` with an authenticated device identity, pairing/list/describe behavior, and fallback behavior when `instanceId` is empty or absent.
