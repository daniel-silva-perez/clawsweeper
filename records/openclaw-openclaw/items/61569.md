## Summary

Keep this issue open. Current main still sends `--node-id` into the Gateway handshake as `client.instanceId`, but the gateway node registry, pairing reconciliation, and several node-owned request handlers still derive the effective node ID from `device.id ?? client.id`; the two linked fix PRs identified the same root cause but were closed unmerged.

## What This Changes

The issue requests that `openclaw node run --node-id <id>` register, list, describe, and target the node by the caller-provided custom node ID for exec node binding.

## Best Possible Solution

Introduce a shared gateway helper for the effective node ID for role=node connections that prefers a trimmed `connect.client.instanceId`, then falls back to `connect.device?.id`, then `connect.client.id`; use it consistently in registration, pairing lookup/request creation, node-caller handlers, and tests covering custom ID, generated-device fallback, and empty/whitespace instance IDs.

## Reproduction Assessment

Yes. A direct repro is to connect a headless node with `openclaw node run --node-id my-mac-node`, approve pairing, and then run `openclaw nodes describe --node my-mac-node`; the current source trace shows the custom ID is sent but not used as the registered/cataloged node key.

## Solution Assessment

Yes, a narrow code fix is appropriate, but it should be broader than only changing `NodeRegistry.register()`. The maintainable path is to centralize effective-node-ID resolution and update all gateway node call sites that currently recompute `device.id ?? client.id`.
