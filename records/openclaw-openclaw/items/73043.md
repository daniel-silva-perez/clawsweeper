## Summary

Keep open. Current main still routes simple-completion calls through pi-ai's provider API path and has no dispatch from `provider: "claude-cli"` or `baseUrl: "cli://claude"` to OpenClaw's registered Claude CLI backend, so the reported generic connection failure remains plausible and actionable.

## What This Changes

The issue asks OpenClaw to make Lossless/pi-ai simple completion work with the synthetic Claude CLI Sonnet model path or fail early with a clear unsupported-transport error.

## Best Possible Solution

Define a supported simple-completion contract for CLI runtime models: either add a first-class OpenClaw helper that runs non-mutating Claude CLI one-shot completions for plugin/context-engine callers, or detect CLI runtime/baseUrl inputs and return a clear unsupported-runtime error with docs guidance.

## Reproduction Assessment

Yes. The issue includes a concrete standalone harness, and static verification against current main plus the locked pi-ai tarball shows the call path stays in pi-ai's Anthropic provider rather than OpenClaw's Claude CLI backend.

## Solution Assessment

Unclear. Passing `cli://claude` through raw pi-ai is not a maintainable solution; the safer path is a deliberate OpenClaw simple-completion runtime contract or an explicit unsupported-runtime error.
