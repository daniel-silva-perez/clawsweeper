## Summary

Keep open. Current main still has no simple-completion path that maps a `claude-cli`/`cli://claude` model onto OpenClaw's Claude CLI backend, and it also does not return an early, specific unsupported-runtime error for that combination.

## What This Changes

The issue requests that Lossless/pi-ai simple completion either support OpenClaw's synthetic `claude-cli` Sonnet transport or fail with a clearer unsupported-runtime error.

## Best Possible Solution

Define the supported simple-completion contract for CLI runtime models: either add a first-class OpenClaw helper that can run a non-mutating Claude CLI one-shot completion for context engines, or detect CLI runtime/baseUrl inputs and return a clear unsupported-runtime error with docs guidance.

## Reproduction Assessment

Yes. The issue includes a concrete standalone harness, and static verification against current main plus the locked pi-ai source confirms the call path goes through pi-ai's Anthropic provider with `model.baseUrl`, not through OpenClaw's Claude CLI backend.

## Solution Assessment

Unclear. Passing `cli://claude` into raw pi-ai is not the best long-term solution; the maintainable fix is an explicit OpenClaw simple-completion runtime contract or an early unsupported error.
