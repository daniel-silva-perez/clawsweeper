## Summary

Keep open. Current main still has no path that maps a Lossless/pi-ai simple completion with `provider: "claude-cli"` or `baseUrl: "cli://claude"` onto OpenClaw's Claude CLI backend, and it also does not fail early with a clearer unsupported-runtime error.

## What This Changes

Investigate or fix Lossless/pi-ai simple-completion failures when a synthetic `claude-cli/claude-sonnet-4-6` model is used with the Anthropic Messages API and `cli://claude` transport.

## Best Possible Solution

Define a supported simple-completion contract for CLI runtime models: either add a first-class OpenClaw helper that can run a non-mutating Claude CLI one-shot completion for context engines, or detect CLI runtime/baseUrl inputs and return a clear unsupported-runtime error with docs guidance.

## Reproduction Assessment

Yes. The issue includes a concrete standalone harness, and static verification against current main plus the locked pi-ai tarball shows the call path goes through pi-ai's Anthropic provider with `model.baseUrl`, not through OpenClaw's Claude CLI backend.

## Solution Assessment

Unclear. Passing `cli://claude` to raw pi-ai is not a maintainable solution; the safer path is an explicit OpenClaw simple-completion runtime contract or an early unsupported error.
