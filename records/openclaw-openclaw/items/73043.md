## Summary

Keep open. Current main still treats `claude-cli` as an agent runtime/CLI backend, while the simple-completion path prepares a model and delegates to `pi-ai` completion without dispatching through the Claude CLI backend. The provided Lossless/pi-ai harness is consistent with the source-level behavior, and no merged implementation or clearer early unsupported-runtime error exists yet.

## What This Changes

The issue asks OpenClaw to make Lossless/pi-ai simple completions using `claude-cli/claude-sonnet-4-6` either route through the Claude CLI transport successfully or fail with a specific unsupported-transport/provider error.

## Best Possible Solution

Define the supported simple-completion contract for CLI runtime models: either provide a first-class OpenClaw helper that can run a non-mutating Claude CLI one-shot completion for context engines, or fail early through the plugin SDK/simple-completion helper with a clear unsupported-runtime message and docs guidance for Lossless/LCM.

## Reproduction Assessment

Yes. The issue includes a concrete standalone harness, and static verification shows `completeSimple` goes through pi-ai's Anthropic HTTP/SSE provider with `model.baseUrl`, not through OpenClaw's `claude-cli` runtime backend.

## Solution Assessment

Unclear. Routing summaries through Claude CLI may be useful, but it crosses runtime/session ownership boundaries; the safer maintainable direction is a deliberate core simple-completion contract, not passing `cli://claude` to raw pi-ai and hoping it dispatches to OpenClaw.
