## Summary

Keep open for maintainer review. Current main does not implement profile archive export/import, and the PR still has a blocking privacy/security issue in the new export sanitizer: credential-like keys already treated as sensitive elsewhere in OpenClaw can be written into the supposedly privacy-safe archive.

## What This Changes

The PR adds `openclaw profile export/import`, profile archive creation/import infrastructure, docs, CLI/command/infra tests, and adjacent test fixture adjustments.

## Best Possible Solution

If core profile portability is accepted, land it as a documented privacy-safe archive format that reuses OpenClaw's shared sensitive-key semantics, preserves non-destructive import behavior, imports only portable plugin install metadata, and includes regression coverage for credential-like keys, dictionary-name collisions, traversal rejection, local path omission, plugin record projection, and dry-run output.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Reuse shared sensitive-key matching for profile export:** `src/infra/profile-portability.ts:360-372`
  - body: The profile sanitizer only drops the hard-coded suffixes listed here, but current main already treats `privateKey`, `encryptKey`, and `serviceAccount`/`serviceAccountRef` as sensitive. Because profile archives include arbitrary plugin, skill, tool, and MCP config, those credentials can survive into a privacy-safe export; reuse `isSensitiveConfigPath` or mirror that contract with regression tests.
