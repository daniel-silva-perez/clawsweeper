## Summary

Keep open. Current main does not implement profile archive export/import, so the PR is not obsolete, but the latest head still has a blocking privacy/security gap: profile config export does not reuse the shared sensitive-path matcher and can include privateKey, encryptKey, or serviceAccountRef fields. Earlier review comments appear to have driven fixes for plugin install-record sanitization and dictionary-key collision handling, but this remaining sanitizer issue needs author and maintainer review before merge.

## What This Changes

The branch adds `openclaw profile export/import`, profile archive creation/import infrastructure, CLI registration, docs, tests, and adjacent bundled-plugin test setup updates.

## Best Possible Solution

If core profile portability is accepted, land it as a documented privacy-safe archive format that reuses the shared sensitive-key semantics, keeps plugin install metadata as a portable projection, preserves non-destructive import behavior, and has regression tests for credential-like names, traversal rejection, and local path omission.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Omit private-key patterns during profile export:** `src/infra/profile-portability.ts:360-372`
  - body: The profile export sanitizer only drops a hard-coded subset of secret-looking names, so fields like `plugins.entries.<id>.config.privateKey`, `encryptKey`, or `serviceAccountRef` survive into the archive even though the shared config-sensitive matcher treats those names as secrets. Because this command exports arbitrary plugin/profile config as a privacy-safe archive, reuse the shared sensitive-path logic or add the missing patterns with regression coverage.
  - confidence: 0.89
