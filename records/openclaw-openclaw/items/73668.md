## Summary

Keep open. Current main does not implement this PR's intended privacy-safe `openclaw profile export/import` workflow: the core command catalog and lazy registry still have `backup` and `migrate` but no `profile` command, docs still describe full-state backup or manual full-state machine moves, and the existing `--profile` code only selects an isolated state directory. The PR remains a plausible implementation candidate, but it needs maintainer product/security review because the latest head still contains a secret-stripping bypass pattern and imports plugin install records from an archive without visible schema/provenance validation.

## Best Possible Solution

Keep this PR open for maintainer product and security review. If privacy-safe profile portability belongs in core, fix the sanitizer with explicit dictionary-depth state, add a regression test for colliding entry ids, validate plugin install records before import, clarify dry-run JSON semantics, split unrelated changes out of the branch, then run the appropriate changed gates before merge.

## Risks / Open Questions

- The latest PR head can still leak secret-like config fields when a dictionary entry id collides with an internal dictionary-container key.
- Archive import writes plugin install records if the local index is missing; that can affect later plugin reinstall/update behavior and should be schema/provenance-checked before merge.
- Dry-run import still reports would-write entries under `filesWritten`, which may confuse JSON consumers unless renamed or separated from actual writes.
- The branch mixes the profile feature with unrelated browser trash, Signal release fetch, bundled plugin trust, and test setup changes; those should be split or explicitly justified before maintainer review.
