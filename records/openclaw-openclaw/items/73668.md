## Summary

Keep open. Current main does not implement the PR's intended privacy-safe `openclaw profile export/import` workflow, and the open PR is still the only implementation candidate. The latest head is also security- and product-sensitive, with a sanitizer pattern that can still bypass secret stripping when a dictionary entry id collides with internal container-key names.

## Best Possible Solution

Keep this PR open for maintainer product and security review. If profile portability belongs in core, fix the sanitizer with explicit dictionary-depth state, add a regression test for colliding entry ids, validate plugin install records before import, clarify dry-run JSON semantics, update the CLI docs/index if accepted, and split unrelated browser/Signal/plugin-loader changes out of the profile portability branch.

## Risks / Open Questions

- The privacy-safe export path may still leak secret-like fields from config entries whose id collides with internal dictionary-container keys.
- Importing plugin install records from an archive without a visible schema/provenance check is sensitive because those records can affect later plugin reinstall behavior.
- Dry-run import still appends would-write entries to `filesWritten`, which can mislead JSON consumers unless renamed or separated from actual writes.
- The PR mixes the profile feature with unrelated browser trash, Signal installer, bundled plugin trust, plugin discovery, and test setup changes; those deserve splitting or explicit maintainer justification before review.
