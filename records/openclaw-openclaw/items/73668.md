## Summary

Current main does not implement this PR’s intended privacy-safe `openclaw profile export/import` workflow. The existing `backup` surface creates/verifies full local-state archives and the existing `migrate` surface imports from other agent systems through providers; neither is the same non-destructive, secrets-stripping profile archive flow. Because the PR is recent, focused, and security-sensitive, it should stay open for maintainer product/security review rather than cleanup closure.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to decide whether privacy-safe profile portability belongs as a new core `openclaw profile` surface or under the existing `backup`/migration model, then harden the import design around trusted-source UX, archive size limits, schema validation for plugin install records, rollback/backups, and focused tests/docs.

## Risks / Open Questions

- The PR imports untrusted `.tar.gz` archives into config/workspace state and writes `plugins/installs.json`; plugin install metadata is documented as machine-managed state used by update, uninstall, diagnostics, and the cold plugin registry, so schema/provenance handling needs maintainer review before merge.
- The product boundary is unresolved: current core already has full-state `backup create/verify` and provider-based `migrate`, while this PR introduces a new core `profile` namespace. It should be reconciled with the backup/migration discussion before acceptance.
