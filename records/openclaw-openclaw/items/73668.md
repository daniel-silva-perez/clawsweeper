## Summary

Keep open. Current main does not implement this profile archive surface, and the latest PR head still has a credential-redaction gap in the privacy-safe export path: `serviceAccount` and `serviceAccountRef` survive even though main's shared sensitive-path matcher treats them as secrets.

## What This Changes

The PR adds `openclaw profile export/import`, a `.openclaw-profile.tar.gz` manifest/archive implementation, privacy-filtered config/workspace/plugin install payload handling, CLI/docs/tests, and adjacent test fixture updates.

## Best Possible Solution

If core profile portability is accepted, land it as a documented archive format that reuses the shared sensitive-key and sensitive-URL redaction semantics, preserves traversal rejection and non-destructive import behavior, exports only portable plugin install metadata, updates docs navigation and changelog, and has regression tests for each credential class the archive promises to exclude.

## Reproduction Assessment

Yes. A unit-level `buildSafeProfileConfig` input such as `skills.entries.gcp.config.serviceAccount` or `plugins.entries.gcp.config.serviceAccountRef` follows the PR sanitizer path and is cloned into the exported config, while current main's shared matcher classifies those paths as sensitive.

## Solution Assessment

No as currently proposed. The feature direction may be maintainable, but a privacy-safe export path should reuse `isSensitiveConfigPath` and sensitive URL redaction or mirror that full contract with focused regression coverage before merge.
