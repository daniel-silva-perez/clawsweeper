## Summary

Keep open. Current main still does not implement a top-level profile export/import command, so the PR is not obsolete, but the proposed privacy-safe archive surface is not safe to merge while the sanitizer can still export credential-bearing keys and URLs.

## What This Changes

The PR adds `openclaw profile export/import`, profile archive creation/import infrastructure, docs, command/CLI/infra tests, and adjacent gateway/plugin test fixture updates.

## Best Possible Solution

If core profile portability is accepted, land it as a documented archive format that reuses the shared sensitive-key and sensitive-URL redaction contracts, exports only portable plugin install metadata, preserves non-destructive import behavior, and keeps unrelated test fixture churn out unless it is needed for the feature.

## Reproduction Assessment

Yes. A unit-level `buildSafeProfileConfig` input with `plugins.entries.<id>.config.privateKey`, `encryptKey`, `serviceAccount`, or `mcp.servers.<id>.url` containing URL credentials will survive the PR sanitizer and can be written into the archive.

## Solution Assessment

No as currently proposed. The feature direction may be acceptable, but the sanitizer should reuse `isSensitiveConfigPath` plus the sensitive URL redaction helpers, with regression tests, before this privacy-safe export/import surface can merge.

## Review Findings

  - body: The profile sanitizer only drops token/password/API-key-style names. Fields such as `privateKey`, `encryptKey`, `serviceAccount`, and `serviceAccountRef` are already treated as sensitive by current main, but they survive under plugin, skill, MCP, or agent config and can be written into a privacy-safe archive. Reuse the shared sensitive matcher or mirror its full contract with regression coverage.
  - confidence: 0.9

Run matching paths through the shared sensitive URL redactor before including them in the archive.
