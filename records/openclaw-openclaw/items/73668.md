## Summary

Keep this PR open. Current main still does not implement the proposed privacy-safe `openclaw profile export/import` workflow, and the PR is not obsolete: it adds a new CLI surface, docs, archive creation/import code, and tests. The latest PR diff remains product- and security-sensitive because the sanitizer can still skip secret stripping for colliding dictionary entry names, dry-run import reports planned writes as `filesWritten`, and raw plugin install records are copied from an untrusted archive into local state without schema/provenance validation.

## Best Possible Solution

Keep this PR open for maintainer review. If privacy-safe profile portability belongs in core, fix the sanitizer by tracking dictionary depth explicitly, add a regression test for colliding dictionary entry ids, validate plugin install records before import, separate actual writes from dry-run planned writes in JSON output, and keep unrelated test harness trust changes out of the feature patch where possible.

## Risks / Open Questions

- Merging as-is can leak secret-like config fields into a supposedly privacy-safe archive when a plugin/server/backend/model dictionary entry id collides with sanitizer container names such as `entries`, `servers`, `models`, `cliBackends`, or `providerOptions`.
- The import path copies archive-provided plugin install records into trusted local state without validating the ledger schema or provenance, which can affect later plugin reinstall/update behavior.
- Dry-run import reports planned file writes in `filesWritten`, which can mislead JSON consumers and automation into treating a dry run as mutation proof.
- The PR is XL and introduces an untrusted archive import/export surface; it needs maintainer product and security review rather than conservative cleanup closure.
