## Summary

Keep open. Current main does not implement profile export/import, and the latest branch still has a blocking privacy bug: the exporter can include config keys that current main already classifies as sensitive. Earlier discussion about dictionary-key collisions and raw plugin install record copying appears to have been partially addressed, but the shared sensitive-key gap remains.

## What This Changes

The PR adds `openclaw profile export/import`, `.openclaw-profile.tar.gz` manifest/archive creation and import infrastructure, docs, CLI/command/infra tests, and adjacent test fixture updates.

## Best Possible Solution

If core profile portability is accepted, land it as a documented archive format that reuses the shared sensitive-key and sensitive-URL redaction semantics, exports only portable plugin install metadata, preserves non-destructive import behavior, keeps traversal protections, and avoids unrelated branch churn.

## Reproduction Assessment

Yes. A unit-level `buildSafeProfileConfig` input containing `plugins.entries.demo.config.privateKey`, `encryptKey`, or `serviceAccount` will survive the PR sanitizer because `shouldOmitPortableConfigKey` does not match those names, while current main's shared matcher and tests classify those paths as sensitive.

## Solution Assessment

No as currently proposed. The feature direction may be acceptable, and the latest head improves earlier plugin-install-record handling, but the privacy-safe exporter should reuse `isSensitiveConfigPath` and existing sensitive URL redaction semantics, or mirror that full contract with regression tests, before merge.
