## Summary

Keep open. Current main still does not provide the proposed profile archive command surface, and the latest PR head still has a blocking privacy/security bug: the branch-local config sanitizer misses sensitive key patterns that current main already redacts elsewhere.

## What This Changes

The PR adds `openclaw profile export/import`, a `.openclaw-profile.tar.gz` manifest/archive implementation for sanitized config, workspace markdown, and plugin install records, plus CLI docs, command/infra tests, and adjacent test-fixture updates.

## Best Possible Solution

If core profile portability is accepted, land it as a documented archive format that reuses shared sensitive-key and sensitive-URL semantics, includes regression coverage for sanitizer misses, exports only portable plugin install metadata, preserves non-destructive import behavior, and clearly distinguishes profile portability from full backup/restore or cloud sync.

## Reproduction Assessment

Yes. A unit-level `buildSafeProfileConfig` input with `plugins.entries.<id>.config.privateKey`, `encryptKey`, or `serviceAccount` will survive the current PR sanitizer because `shouldOmitPortableConfigKey` does not match those names, while current main's shared matcher marks them sensitive.

## Solution Assessment

No as currently proposed. The core feature direction may be acceptable, but the sanitizer should reuse `isSensitiveConfigPath` and the existing sensitive URL redaction contract, or mirror that full behavior with tests, before this privacy-safe archive surface can merge.
