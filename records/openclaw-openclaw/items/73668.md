## Summary

Keep open for author/maintainer handling. Current main does not implement this profile archive surface, and the latest PR head still has a privacy bug where path-like config fields can be exported despite the feature being described as privacy-safe; earlier review context usefully shows the branch already addressed prior secret and plugin-index findings, but this remaining sanitizer gap should block merge.

## What This Changes

The PR adds a new `openclaw profile export/import` CLI, `.openclaw-profile.tar.gz` manifest/archive infrastructure, profile portability docs, infra/CLI/command tests, and adjacent test fixture updates.

## Best Possible Solution

If core profile portability is accepted, land it as a documented archive format whose sanitizer consistently removes secrets and local path material from all exported config projections, keeps plugin install records as a portable projection, preserves non-destructive import behavior, and has regression tests for path-like config keys, key material, traversal rejection, and plugin-index rebuild behavior.

## Reproduction Assessment

Yes. A unit-level `buildSafeProfileConfig` input containing `plugins.entries.demo.config.sourcePath: "/home/alice/private"` or `mcp.servers.demo.localPath` will survive the PR sanitizer because `shouldOmitPortableConfigKey` does not match path-like suffixes, even though the profile archive is documented as privacy-safe.

## Solution Assessment

No as currently proposed. The latest head improves the earlier private-key and plugin-index findings, but the exporter should also strip path-like key suffixes or path-like values across arbitrary portable config before this can be treated as the narrow, maintainable privacy-safe solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Strip path-like config keys before exporting profiles:** `src/infra/profile-portability.ts:363-378`
  - body: The exporter promises to keep local paths out of profile archives, but `shouldOmitPortableConfigKey` only drops exact normalized keys such as `path`, `cwd`, and `dirs`. Since the archive includes arbitrary plugin, skill, MCP, and tool config, fields like `plugins.entries.demo.config.sourcePath`, `localPath`, `workspacePath`, or `configPath` can still write machine-specific paths into the privacy-safe archive. Treat path-like suffixes with the same redaction policy and add regression coverage.
  - confidence: 0.89

## Security Review

Status: needs_attention

Summary: Profile export can still leak local filesystem paths from arbitrary portable config fields into a privacy-safe archive.

Concerns:

- **[medium] Profile archive can include local paths:** `src/infra/profile-portability.ts:363`
  - body: The sanitizer omits exact path keys but misses common path-like suffixes such as `sourcePath`, `localPath`, `workspacePath`, and `configPath`. Because exported portable config includes arbitrary plugin/skill/MCP/tool settings, user names, private project names, or host-specific paths can be copied into the archive.
