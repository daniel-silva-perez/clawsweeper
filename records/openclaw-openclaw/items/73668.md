## Summary

Keep open. Current main does not implement the proposed privacy-safe `openclaw profile export/import` workflow, and this PR is not obsolete: it adds a new CLI command, docs, archive export/import infrastructure, and tests. The proposed diff is also security-sensitive and still needs maintainer review for sanitizer, dry-run reporting, and plugin install ledger import behavior.

## Best Possible Solution

Keep this PR open for maintainer product and security review. If privacy-safe profile portability belongs in core, fix the sanitizer by tracking dictionary depth explicitly, add collision regression tests, validate or rebuild imported plugin install records through existing plugin index helpers, separate actual writes from dry-run projections in the JSON result, and keep unrelated test harness trust changes out of the feature patch where possible.

## Risks / Open Questions

- Merging as-is can leak secret-like config fields into a supposedly privacy-safe archive when a plugin, MCP server, backend, or model entry ID collides with sanitizer dictionary-container names such as `servers` or `entries`.
- The import path copies untrusted archive-provided plugin install ledger bytes into machine-managed plugin state without first rebuilding or validating the ledger through the existing plugin index helpers.
- Dry-run import reports planned writes as `filesWritten`, which can mislead JSON consumers and human output.
- The PR is XL and introduces an untrusted archive import/export surface; it needs focused product and security review before landing.
- The diff also changes test harness trust behavior via `OPENCLAW_TEST_TRUST_BUNDLED_PLUGINS_DIR`; that may be test-only, but it is not central to the profile feature and should be reviewed separately.
