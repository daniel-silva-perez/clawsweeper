## Summary

Keep this issue open. Current main has Telegram group sender allowlisting, owner-derived Telegram approvers, Telegram-native approval delivery, and generic `/elevated full` auto-approval, but it still lacks the requested opt-in Telegram trusted-owner/group/tool-class approval-bypass policy with audit labeling and destructive-operation fallback behavior.

## Best Possible Solution

Keep this issue open. A complete fix should live primarily in the bundled Telegram plugin/config surface, with a generic approval-policy seam only if the host approval pipeline needs one. It should require resolved numeric owner IDs, allowed group/topic scope, configured tool classes, an audit reason such as trusted-owner-bypass, destructive-operation fallback behavior, schema/docs updates, and focused Telegram approval-path tests.
