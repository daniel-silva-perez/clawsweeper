## Summary

Keep this PR open. Current main still lacks the requested chunk-table text fallback when memory FTS is unavailable, but the PR is not ready to merge because its provider/hybrid fallback behavior and source filtering need correction, and the branch also carries stale-base changes outside the memory-core fix.

## What This Changes

The PR adds a memory-core chunk-scan keyword fallback for unavailable FTS5, re-enables related memory tests, skips trigram FTS tests on unsupported SQLite builds, and includes unrelated doctor/plugin-registry/gateway test fixture adjustments.

## Best Possible Solution

Land a narrow memory-core change that scans `chunks` only for the unavailable-FTS lexical path, threads the existing source filter through that fallback, preserves provider-backed vector-only semantics, and keeps hybrid relaxed fallback ordered by the merged ranking pipeline.

## Reproduction Assessment

Yes. The current-main code path is high-confidence: with indexed `chunks`, no provider, and unavailable/disabled FTS, `search()` returns `[]`; a focused regression can instantiate that state and assert keyword recall from `chunks`.

## Solution Assessment

No. The PR direction is right, but the current patch is not the best implementation because it widens lexical fallback into vector-only/provider paths, bypasses merged hybrid ranking, and omits source overrides in the new fallback helper.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve vector-only searches when fallback is lexical:** `extensions/memory-core/src/memory/manager.ts:400-401`
  - body: This fallback runs whenever hybrid is disabled or FTS is unavailable and `vectorResults` is empty. That changes vector-only/provider-backed configurations from returning vector results only into returning substring chunk hits, so a zero-vector query or vector failure can surface lexical matches even when hybrid text search was not enabled.
  - confidence: 0.9
- **[P2] Keep relaxed keyword fallback on merged ranking:** `extensions/memory-core/src/memory/manager.ts:420`
  - body: Returning raw `keywordResults` here skips `mergeHybridResults`, including temporal decay and MMR ordering. Current main keeps keyword-backed entries from the merged set with a relaxed threshold, so this can reorder stale or duplicate chunks ahead of the ranking policy the user configured.
  - confidence: 0.87
- **[P2] Pass source filters through fallback scans:** `extensions/memory-core/src/memory/manager.ts:526-535`
  - body: The new fallback helper builds a filter from all enabled sources instead of the `opts.sources` list already computed by `search()`. When a caller restricts search to memory or sessions, the fallback can return chunks from other enabled sources; pass the source override into the helper and use it in `buildSourceFilter`.
  - confidence: 0.86
