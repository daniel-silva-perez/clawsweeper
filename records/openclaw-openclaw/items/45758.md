## Summary

Current main still does not implement YAML root config support. OpenClaw discovers `openclaw.json` plus legacy `clawdbot.json`, parses config with JSON5, writes JSON, and documents JSON5 `~/.openclaw/openclaw.json`. The later discussion correctly points toward safer mutation paths such as `openclaw config set --batch-file` and `config.patch`, but those do not satisfy the requested `.yaml`/`.yml` config-file format.

## Best Possible Solution

Keep this open for an explicit maintainer product decision. If YAML support is accepted, implement it in core config discovery, parser selection, validation, include semantics, write behavior, tests, and docs with clear `openclaw.json` versus `openclaw.yaml`/`openclaw.yml` precedence. If OpenClaw intends to stay JSON5-only, close manually as a product decision and strengthen docs around `openclaw config set`, Control UI, and `config.patch` for comment-safe scripted edits.

## Risks / Open Questions

- Adding YAML would touch core startup/config I/O, precedence rules, include parsing, validation, write serialization, docs, and tests; it is not plugin-only because plugins load after root config resolution.
- YAML parser semantics could introduce implicit type coercion and different security expectations, so an implementation would need explicit tests for scalar handling, multiple-file precedence, includes, and write behavior.
- The existing safer mutation paths reduce the comment-preservation pain but do not answer the root `.yaml`/`.yml` config-file request.
