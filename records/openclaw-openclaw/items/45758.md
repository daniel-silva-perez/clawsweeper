## Summary

Keep this issue open. Current main still does not support `openclaw.yaml` or `openclaw.yml` as an alternative root config format; the shipped loader resolves `openclaw.json`, parses config through JSON5, and writes JSON. The later discussion usefully reframed part of the pain as comment-safe mutation, and current main has `openclaw config set --batch-file` plus gateway `config.patch`, but those do not implement the requested YAML config-file support.

## Best Possible Solution

Maintainers should make an explicit product decision. If YAML support is accepted, implement it in core config path resolution and config I/O with clear precedence for `openclaw.json` vs `openclaw.yaml`/`openclaw.yml`, validation and write behavior, tests, and docs. If the intended direction is JSON5-only, close manually as a product decision and keep strengthening docs around `openclaw config set`, batch files, Control UI, and `config.patch` for comment-safe scripted edits.

## Risks / Open Questions

- Adding YAML would touch core startup/config I/O, write serialization, precedence rules when multiple config files exist, docs, tests, and possibly include semantics; it should not be treated as a small plugin-only feature.
- The safer mutation paths reduce the comment-preservation pain, so maintainers may prefer a documented JSON5-only stance or a narrower docs follow-up instead of implementing a second root config format.
