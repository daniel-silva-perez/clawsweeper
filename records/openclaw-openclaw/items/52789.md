## Summary

Keep open. Current main still does not implement a cron `delivery.mode: "session"`; cron delivery remains `none`/`announce`/`webhook`, while `sessions_send` is still only available through the agent tool path with its own visibility and agent-to-agent policy checks.

## Best Possible Solution

Keep this issue open and decide/implement it as a core cron delivery feature if maintainers want the product behavior: extend cron delivery types, Gateway protocol schema, normalizer, CLI/UI/docs, and tests; add a framework-owned dispatch path that sends the final cron summary to a target session using `sessions_send`-equivalent provenance, visibility, and agent-to-agent policy checks; record success/failure diagnostics without posting to a visible chat channel.

## Risks / Open Questions

- A cron-owned session delivery implementation must preserve the same session visibility and `tools.agentToAgent` allowlist checks enforced by `sessions_send`; bypassing them would broaden cross-agent authority for scheduled jobs.
- Run-log and delivery-state semantics need explicit tests so session-delivery failure is recorded separately from job execution failure and does not create hidden user-visible chat delivery.
