## Summary

Keep this PR open for maintainer review. Current main still does not read structured `stdout` in the Swift chat tool-result formatter, while this PR is narrow and appears functionally correct; the remaining issues are normal review/validation and fixing the unrelated closing reference.

## What This Changes

The PR updates the Swift Chat UI tool-result formatter to prefer structured `stdout` for successful exec-style payloads before generic message fields, and adds two formatter tests for stdout precedence.

## Best Possible Solution

Land this formatter change after maintainer review with the closing reference corrected, focused OpenClawKit formatter tests passing, and the behavior kept scoped to successful stdout display rather than adding stderr/error policy in the same PR.

## Reproduction Assessment

Yes. Static inspection of current main gives a high-confidence reproduction path: a structured JSON tool result with `stdout` and no `message` falls through to an empty display string, while one with both `stdout` and `message` returns the generic message because `stdout` is not read.

## Solution Assessment

Yes. The proposed implementation is the narrow maintainable fix: reuse the existing string extraction helper, preserve error and nodes-summary precedence, and return stdout before generic message fields; the redundant empty-string check is nonblocking style cleanup.
