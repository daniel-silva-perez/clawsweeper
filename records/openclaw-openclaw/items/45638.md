## Summary

Keep open. Current main still sends the opaque Telegram callback query id into the generic callback fallback's `messageIdOverride`, so the agent-visible `MessageSid` remains unrelated to the originating Telegram message id. The adjacent callback metadata work present on main helps plugin interactive handlers, but it does not correct the generic `processMessage` path reported here.

## What This Changes

The issue requests fixing Telegram callback query handling so generic inline-button callbacks use the originating Telegram `message_id` as `messageIdOverride`, not the callback query id.

## Best Possible Solution

Land a focused Telegram plugin fix that preserves `callback.id` for callback identity, acknowledgement, and plugin-interactive dedupe, but passes `String(callbackMessage.message_id)` as `messageIdOverride` for the generic `processMessage` fallback. Add a regression assertion for `MessageSid` on the generic callback route and preserve the separate callback metadata behavior already used by plugin interactive handlers.

## Reproduction Assessment

Yes. A high-confidence unit reproduction is to invoke the registered `callback_query` handler with `callback.id` set to a nonnumeric value and `callback.message.message_id` set to a numeric id, then inspect the captured inbound payload; current main will source `MessageSid` from the callback id.

## Solution Assessment

Yes. The narrow owner-module change plus a targeted regression test is the best maintainable fix; changing core `MessageSid` semantics or the plugin interactive metadata path would be broader than needed.
