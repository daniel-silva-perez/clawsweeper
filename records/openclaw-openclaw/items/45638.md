## Summary

Keep open. Current main still passes the opaque Telegram callback query id into the generic callback fallback's messageIdOverride, so MessageSid can remain unrelated to the originating Telegram message id. The adjacent callback metadata path already carries callbackMessage.messageId for plugin interactive handlers, but that does not fix the processMessage fallback reported here.

## What This Changes

Use the Telegram callback message's message_id, not callback.id, as messageIdOverride in the generic Telegram callback_query processMessage fallback.

## Best Possible Solution

Land a focused Telegram plugin fix that preserves callback.id for answerCallbackQuery, callback identity, and plugin interactive metadata, but passes String(callbackMessage.message_id) as messageIdOverride for the generic processMessage fallback. Add a regression assertion for MessageSid on that callback route.

## Reproduction Assessment

Yes. A high-confidence unit reproduction is to invoke the registered callback_query handler with a nonnumeric callback.id and a numeric callback.message.message_id, then inspect the captured inbound payload; current main routes MessageSid from the callback id.

## Solution Assessment

Yes. The narrow owner-module change plus a targeted regression test is the maintainable fix; changing core MessageSid semantics or the plugin interactive metadata path would be broader than needed.
