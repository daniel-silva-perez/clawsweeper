## Summary

Keep open. Current main still passes the Telegram callback query id as the generic callback path's message id override, so the reported `MessageSid` continuity bug remains. The newer related open PR tracks exposing callback message metadata for agents, but it does not supersede this narrower `messageIdOverride` fix.

## What This Changes

The issue requests changing Telegram inline-button callback handling so `messageIdOverride` uses the originating callback message's Telegram `message_id` instead of the opaque callback query id.

## Best Possible Solution

Land a focused Telegram plugin fix that keeps `callback.id` only for callback identity and `answerCallbackQuery`, passes `String(callbackMessage.message_id)` as `messageIdOverride`, and adds a regression assertion that the generic callback route's payload has `MessageSid` equal to the originating Telegram message id.

## Reproduction Assessment

Yes. The current-main source path gives a high-confidence unit reproduction: invoke the generic callback-query route with a callback message id and inspect the captured inbound payload; today `MessageSid` comes from `callback.id` instead of `callback.message.message_id`.

## Solution Assessment

Yes. The requested owner-module change plus a targeted regression test is the narrowest maintainable solution, and it matches the existing debounce/text-fragment synthetic-message pattern without changing callback identity handling.
