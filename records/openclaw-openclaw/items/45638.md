## Summary

Keep open. Current main still passes the Telegram callback query id into the generic callback path's `messageIdOverride`, so the reported `MessageSid` continuity bug remains; the open related PR for callback message metadata is adjacent but does not supersede this narrower fix.

## What This Changes

The issue requests changing Telegram inline-button callback handling so `messageIdOverride` uses the originating callback message's Telegram `message_id` instead of the opaque callback query id.

## Best Possible Solution

Land a focused Telegram plugin fix that keeps `callback.id` only for callback identity and `answerCallbackQuery`, passes `String(callbackMessage.message_id)` as `messageIdOverride`, and adds a regression assertion that the generic callback route's payload has `MessageSid` equal to the originating Telegram message id.

## Reproduction Assessment

Yes. A high-confidence unit reproduction is to invoke the generic `callback_query` route with `callback.id` different from `callback.message.message_id` and inspect the captured inbound payload; current main will set `MessageSid` from `callback.id`.

## Solution Assessment

Yes. The narrow owner-module change plus a targeted regression test is the best fix for this issue, and it matches existing synthetic-message paths without changing callback identity handling or the separate #55462 metadata feature.
