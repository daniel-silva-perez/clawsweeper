## Summary

Keep open. Current main still routes fallback Telegram callback_query payloads into processMessage with messageIdOverride set to callback.id, not the originating Telegram callback message_id. The related fix PRs in the provided context were closed unmerged, and the latest shipped release contains the same current behavior.

## Best Possible Solution

Keep this issue open and apply the focused Telegram fix in extensions/telegram/src/bot-handlers.runtime.ts: pass String(callbackMessage.message_id) as messageIdOverride for fallback callback_query synthetic messages, while keeping callback.id for answerCallbackQuery and plugin callback metadata. Add a regression test that invokes the callback_query handler and asserts the emitted inbound payload MessageSid is the numeric callback message_id, not the callback query id.
