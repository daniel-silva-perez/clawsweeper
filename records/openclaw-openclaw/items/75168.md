## Summary

Keep open. Current main still has the unguarded async channel lifecycle logger dereferences, and the related open PR #75172 references this issue with closing syntax, so this issue should wait for that PR, or an equivalent fix, to merge. The later PR #75331 was closed unmerged and does not change current main.

## What This Changes

The issue reports a gateway crash when async channel lifecycle handlers dereference an undefined `channelLogs[channelId]` logger and asks for safe optional access at those call sites.

## Best Possible Solution

Land a narrow gateway fix that treats the captured channel logger as optional in `server-channels.ts` and adds regression coverage for a channel account start/exit path with no `channelLogs` entry. After that merges, close this through the normal linked-PR resolution.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a channel plugin present in the active registry while `channelLogs[channelId]` is missing, followed by a `startAccount` rejection or exit that reaches the async `log.error?.(...)` / `log.info?.(...)` handlers.

## Solution Assessment

Yes. A defensive `log?.error?.(...)` / `log?.info?.(...)` guard plus a focused regression test is the narrowest maintainable crash fix; a broader logger-map synchronization cleanup can be considered separately if reviewers want it.
