## Summary

Keep open. Current main still only enumerates direct Markdown children of each memory-wiki page-group directory, so nested `sources/.../*.md` pages remain absent from compile output, local wiki search/get fallback, and status counts.

## What This Changes

The issue requests recursive memory-wiki discovery of Markdown pages inside nested vault page-group subdirectories while preserving the user's existing directory organization.

## Best Possible Solution

Add a memory-wiki-owned recursive Markdown page collector for the existing page-group directories, reuse it across compile, local query/get fallback, and status, then document and test nested page-group support without adding a core/plugin SDK seam.

## Reproduction Assessment

Yes. The high-confidence reproduction path is to place a Markdown file under `sources/<subdir>/` and run wiki compile/search/status; current main only reads direct page-group entries with `fs.readdir` plus `entry.isFile()`, so that nested file is never enumerated.

## Solution Assessment

Yes. A shared extension-owned recursive collector is the narrowest maintainable fix; a new core API or a broader raw-source/lint policy change is not required for this issue.
