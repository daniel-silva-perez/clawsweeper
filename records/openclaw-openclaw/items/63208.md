## Summary

Current main still only discovers direct `.md` children under each memory-wiki page group, so nested pages under `sources/` are not compiled into indexes, digests, query results, or status counts. This is a valid, narrow plugin bug and is a good fix-PR candidate.

## Best Possible Solution

Keep this issue open and implement recursive markdown page discovery inside the memory-wiki plugin. The best fix is a shared deterministic collector used by compile, local query/get, and status counting, with focused tests for nested pages under `sources/` and generated indexes/digests preserving the relative path structure.

## Clownfish Work Prompt

Fix memory-wiki recursive vault page discovery for manually organized page-group subdirectories. Current main only reads direct `.md` files under `entities/`, `concepts/`, `sources/`, `syntheses/`, and `reports/`, so nested pages like `sources/OpenClaw配置文件指南/AGENTS.md` do not participate in compile indexes, backlinks, digests, local query/get, or status counts. Implement a deterministic recursive markdown collector in the memory-wiki plugin, likely as a shared helper, and use it from `compileMemoryWikiVault`, `readQueryableWikiPages`, and status counting. Preserve existing frontmatter ids; do not add core changes, migrations, or owner-specific policy. Keep generated group indexes excluded, avoid following symlinks/outside-vault paths, and keep output sorting stable. Add focused tests for a nested source page appearing in root and `sources/index.md`, `.openclaw-wiki/cache/agent-digest.json`, local query/get, and status counts. Update docs if needed to mention nested page organization, and add a changelog Fixes entry crediting `@arlen8411`. Related ref: https://github.com/openclaw/openclaw/issues/63208.
