## Summary

Keep open. Current main still uses flat-only Markdown discovery for memory-wiki compile, local search/get, and status counting, so nested source pages remain outside the compiled artifacts and local query path.

## What This Changes

Request recursive memory-wiki Markdown discovery so pages under subdirectories such as `sources/OpenClaw配置文件指南/AGENTS.md` are compiled, searchable, retrievable, and counted without flattening the vault.

## Best Possible Solution

Add one shared deterministic recursive Markdown page collector in the memory-wiki plugin and use it consistently for compile, local query/get fallback, and status counts, with regression tests and docs that describe nested page-group support.

## Repair Work Prompt

Fix the memory-wiki nested page discovery bug reported in https://github.com/openclaw/openclaw/issues/63208. Implement a shared deterministic recursive Markdown page collector inside the memory-wiki plugin, then use it for compile page discovery, local wiki search/get fallback discovery, and status page/source counts. Keep the fix scoped to the memory-wiki plugin and docs; do not add core plugin APIs or broad config migrations unless the existing schema contract truly requires it. The collector should scan only recognized wiki page-group directories (`entities`, `concepts`, `sources`, `syntheses`, `reports`), return normalized slash-separated relative paths, sort deterministically, preserve current flat-file behavior, exclude generated `index.md` consistently, and avoid following symlinks or walking outside the vault page groups. Add regression tests for nested source pages under `sources/<subdir>/*.md` that prove compile includes the page in counts/digests/indexes, local search/get can find and read it by relative path/title, and status counts it with source provenance. Update `docs/plugins/memory-wiki.md` and `docs/cli/wiki.md` to mention recursive page-group support. Add a user-facing changelog fix entry crediting `Thanks @arlen8411`. Validate with the targeted memory-wiki tests and formatter check listed in `workValidation`; run the changed gate in Testbox before handoff if the selected lanes become broad.
