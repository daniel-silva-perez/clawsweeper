## Summary

Current main still has the reported memory-wiki limitation. `openclaw wiki compile`, local wiki query/get, and wiki status enumerate only direct Markdown children under each top-level page group, so nested pages such as `sources/OpenClaw配置文件指南/AGENTS.md` are not compiled, searched, read, or counted. The existing ClawSweeper comment is still directionally correct: this is a valid, narrow memory-wiki plugin bug and remains a good focused fix-PR candidate.

## Best Possible Solution

Keep this issue open and implement recursive Markdown page discovery inside the memory-wiki plugin. The best focused fix is a shared deterministic page-file collector used by compile, local query/get, and status counting; focused nested-source regression tests; user-facing memory-wiki/wiki docs updates; and a changelog fix entry crediting @arlen8411.

## Clownfish Work Prompt

Fix the memory-wiki plugin so nested Markdown files under wiki page-group directories are discovered consistently. Current `main` only scans direct children in `extensions/memory-wiki/src/compile.ts`, `extensions/memory-wiki/src/query.ts`, and `extensions/memory-wiki/src/status.ts`, so a page like `sources/OpenClaw配置文件指南/AGENTS.md` is skipped by `openclaw wiki compile`, local wiki query/get, and status counts. Add a shared deterministic recursive Markdown collector inside the memory-wiki plugin, use it for compile page summaries, query/get file listing, and status counting, and preserve stable relative-path page IDs/links. Do not follow symlinks or walk outside the expected wiki page-group roots. Decide and test compatibility behavior for nested `index.md`, since current code excludes basename `index.md`. Add focused regression tests for nested `sources/<topic>/*.md` in compile, query, and status tests. Update `docs/plugins/memory-wiki.md` and `docs/cli/wiki.md` if the behavior changes, and add a changelog fix entry crediting @arlen8411. Keep the fix extension-owned; do not add bundled plugin ids or memory-wiki policy to core.
