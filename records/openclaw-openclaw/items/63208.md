## Summary

Keep open. Current main still discovers memory-wiki Markdown pages only from the direct children of each page-group directory, so nested source files remain missing from compile output, local wiki search/get fallback, and status counts. The prior ClawSweeper comment remains accurate after rechecking against current main SHA 2d53b49b20e19790afbba4f4cbb2b78042bb3530.

## What This Changes

Support recursive subdirectory scanning for memory-wiki vault page groups so nested Markdown source files are compiled, searchable, and counted while preserving users' existing organization.

## Best Possible Solution

Add an extension-owned recursive Markdown page collector for memory-wiki page groups, use it consistently in compile, local query/get fallback, and status, document nested page-group support, and cover nested source pages with focused regression tests.

## Repair Work Prompt

Implement recursive Markdown discovery for the bundled memory-wiki plugin. Add a shared extension-owned helper that walks only recognized wiki page-group roots (`entities`, `concepts`, `sources`, `syntheses`, `reports`), returns deterministic POSIX-normalized relative paths, ignores `index.md` consistently with current behavior, does not follow symlinks, and cannot escape the vault root. Use it in compile page discovery, local query/get fallback discovery, and status counts so nested source pages are compiled, searchable/gettable by path/title/id fallback where current lookup rules allow, and counted. Add regression tests for nested `sources/<topic>/<page>.md` in compile, query, and status. Update public docs for nested page groups. Do not change core plugin APIs, do not broaden raw-source lint/frontmatter policy beyond what the recursive tests require, and keep #69700 separate unless a maintainer explicitly chooses to combine the product decision.
