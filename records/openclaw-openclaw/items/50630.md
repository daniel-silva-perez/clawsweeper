## Summary

Keep open. Current main still accepts the reported Tailscale Serve plus auth mode none startup shape, the auth path still returns success for mode none before Tailscale identity-header auth can run, and linked PR #50631 remains an open unmerged implementation candidate for this same issue.

## What This Changes

The issue asks OpenClaw to prevent Tailscale Serve from exposing the Gateway to a Tailnet when `gateway.auth.mode` is `none`.

## Best Possible Solution

Harden Gateway startup so explicit no-auth mode cannot be combined with Tailscale Serve remote exposure unless maintainers intentionally define a separate verified identity-bearing HTTP contract, while preserving trusted loopback-only no-auth use and updating runtime-config tests plus public Gateway security docs.

## Reproduction Assessment

Yes. The deterministic source-level reproduction is the config shape `gateway.auth.mode="none"` with `gateway.tailscale.mode="serve"`; current tests still assert that this shape starts successfully with loopback binding.

## Solution Assessment

Yes. A startup-time guard in `resolveGatewayRuntimeConfig` is the narrowest maintainable fix for the reported boundary gap, with tests proving Serve rejects mode none while loopback-only no-auth remains allowed.
