## Summary

Keep open. Current main still accepts the reported `gateway.auth.mode="none"` plus `gateway.tailscale.mode="serve"` startup shape, and the auth path returns success for mode none before Tailscale identity-header verification can run. An open, unmerged PR already references this issue with closing syntax, so this needs maintainer security review rather than auto-close or a separate repair lane.

## What This Changes

The issue requests Gateway startup hardening so Tailscale Serve cannot expose a `gateway.auth.mode="none"` Gateway to Tailnet peers without credentials.

## Best Possible Solution

Harden Gateway startup so explicit auth mode none cannot be combined with Tailscale Serve remote exposure unless maintainers intentionally define a separate verified identity-bearing contract, while preserving trusted loopback-only no-auth use and updating runtime-config tests plus public gateway/security docs for the final behavior.
