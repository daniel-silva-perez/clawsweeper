## Summary

Keep open. Current main still has the reported startup-validation gap for `gateway.auth.mode="none"` with `gateway.tailscale.mode="serve"`, and an open same-author PR (#50631) explicitly references this issue with closing syntax, so this issue should remain the tracker until that PR or an equivalent fix lands.

## Best Possible Solution

Keep this issue open and review #50631 or land an equivalent focused fix that rejects explicit `gateway.auth.mode: "none"` whenever `gateway.tailscale.mode: "serve"` exposes the loopback gateway through Tailscale. The fix should include a runtime-config regression test for Serve+none rejection while preserving the existing Tailscale Serve identity-header path for Control UI/WebSocket auth when using supported auth modes.
