## Summary

Keep open. Current main still matches the reported gateway auth-boundary gap: startup validation rejects Tailscale Funnel without password auth but does not reject Tailscale Serve with auth mode none, and the auth path accepts mode none before Tailscale identity-header verification can run. An open, unmerged PR already references this issue with closing syntax, so this should stay open for maintainer review rather than be auto-closed.

## What This Changes

The issue asks OpenClaw to reject or otherwise authenticate Gateway configurations that combine Tailscale Serve exposure with `gateway.auth.mode: "none"`.

## Best Possible Solution

Harden Gateway startup so explicit auth mode none cannot be combined with Tailscale Serve remote exposure unless maintainers intentionally define a verified identity-bearing alternative, then update runtime-config tests and public gateway/security docs to match the final contract.
