## Summary

Keep this PR open: the reported Discord token-routing problem is real on current main, but the current PR branch is not merge-ready because it still treats internal account scans as explicit account targeting and is dirty against main.

## What This Changes

The PR changes Discord token resolution so explicit non-default account IDs without per-account tokens return `source: "none"`, threads an explicitness flag through account resolution, and adds Discord token/account regression tests.

## Best Possible Solution

Land a rebased Discord-owned fix that distinguishes user-targeted account IDs from internal/default/list-discovery account resolution, keeps explicit sends strict, preserves channel-token fallback for implicit routes, and covers token, account, and action-discovery tests.

## Reproduction Assessment

Yes. A static reproduction path is clear in PR head: with `channels.discord.token`, `defaultAccount: "work"`, and `accounts.work` lacking a token, unscoped message action discovery scans `work`, marks it explicit, gets `tokenSource: "none"`, and drops Discord actions.

## Solution Assessment

No. The strict token guard is the right direction for explicitly targeted sends, but using mere presence of `accountId` inside `resolveDiscordAccount` is not the best boundary because internal scans also pass account IDs; the resolver needs an explicit caller intent option or separate implicit scan path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Treat account scans as implicit:** `extensions/discord/src/accounts.ts:67`
  - body: `callerProvidedAccountId` is true for any non-empty `accountId`, but unscoped discovery calls `listEnabledDiscordAccounts`, which maps every configured account into `resolveDiscordAccount`. For a top-level-token config with `defaultAccount: "work"` and no per-account token, that scan now returns `tokenSource: "none"`, so `listTokenSourcedAccounts` hides all Discord message actions even though the channel token is valid for the implicit route.
