## Summary

Current main still has the reported schema/runtime mismatch: `message.send` advertises inline attachment fields, but the shared send path and Telegram plugin only route URL/path-style media and do not reject `buffer`/`filename`/MIME fields. The issue also has open implementation candidates, so it should stay open until one canonical fix lands or maintainers intentionally choose the explicit-rejection path.

## What This Changes

The issue asks OpenClaw to stop Telegram `message.send` from returning success when inline attachment fields such as `buffer`, `filename`, and MIME type are accepted but not delivered.

## Best Possible Solution

Keep this issue open and land or replace #42674/#63578 with a focused fix that either hydrates inline attachment fields through `message.send` into Telegram media delivery or rejects those fields explicitly for `send`, with regression coverage across the shared outbound runner and Telegram plugin path.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: current main accepts `buffer`/`filename` in the `message.send` schema, skips hydration for `send`, and reaches Telegram with only URL/path media fields.

## Solution Assessment

Unclear. Both proposed directions from the report are maintainable, but the best current path is one canonical maintainer-reviewed PR rather than another duplicate implementation.
