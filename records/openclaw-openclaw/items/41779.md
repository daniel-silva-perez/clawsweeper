## Summary

Current main still has the reported mismatch: `message.send` advertises inline attachment fields, but the shared send path and Telegram plugin only consume URL/path-style media. The prior ClawSweeper review remains accurate against current main `bdfb408ce6eeb724a638c106104dbbcae50b612c`; open PRs #42674 and #63578 are implementation candidates, not merged fixes.

## Best Possible Solution

Keep this issue open until main either supports inline `buffer`/`filename`/`contentType`/`mimeType` attachments for `message.send` through the shared outbound path and Telegram plugin, or rejects those fields explicitly for `send` with a clear error. The practical maintainer path is to review, land, or replace #42674/#63578 with focused shared outbound and Telegram regression coverage.
