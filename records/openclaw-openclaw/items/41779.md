## Summary

Keep open. Current main still exposes inline attachment fields on `message.send`, but the shared outbound send path and Telegram runtime still only consume URL/path-style media. `buffer`, `filename`, `contentType`, and `mimeType` are neither hydrated into a Telegram upload nor explicitly rejected, so the false-success bug described here still looks current. Related open PRs #42674 and #63578 remain implementation candidates, not proof that this issue is fixed on main.

## Best Possible Solution

Keep this issue open until main either supports inline `buffer`/`filename`/`mimeType` attachments for `message.send` through the shared outbound path and Telegram plugin, or rejects those fields explicitly for `send` with a clear error. The practical maintainer path is to review, land, or replace #42674/#63578 with focused shared outbound and Telegram regression coverage.
