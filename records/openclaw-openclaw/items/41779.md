## Summary

Keep open. Current main still exposes inline attachment fields on `message.send`, but the shared hydration path still skips `send`, the shared send runner only builds URL/path-style media, and Telegram still forwards only `mediaUrl` into `sendMessageTelegram`. The related open PRs #42674 and #63578 remain implementation candidates, not proof that this issue is fixed on main.

## Best Possible Solution

Keep this issue open until main either supports inline `buffer`/`filename`/`mimeType` attachments for `message.send` through the shared outbound path and Telegram plugin, or rejects those fields explicitly for `send` with a clear error. The concrete maintainer path is to review, land, or replace #42674/#63578 with focused shared outbound and Telegram regression coverage.
