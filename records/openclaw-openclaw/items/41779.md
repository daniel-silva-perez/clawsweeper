## Summary

Current main still has the reported mismatch: `message.send` advertises inline attachment fields, but the shared send runner and Telegram plugin path do not hydrate or forward `buffer`/`filename`/`contentType`/`mimeType` into Telegram media delivery, and they do not reject those fields. The issue also has active implementation candidates in #42674 and #63578, so it should stay open until a focused fix lands or maintainers choose an explicit rejection path.

## Best Possible Solution

Keep this issue open. The best maintainer path is to review and land or replace #42674/#63578 with a focused fix that either hydrates inline `buffer`/`filename`/`contentType`/`mimeType` through `message.send` into Telegram media delivery, or rejects those fields explicitly for `send` with regression coverage in the shared outbound runner and Telegram plugin tests.
