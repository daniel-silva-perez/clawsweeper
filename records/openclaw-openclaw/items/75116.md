## Summary

Keep open. Current main still lacks a first-class per-server `injectCallerContext` contract and does not inject caller headers into user or plugin remote bundle MCP entries; the open paired implementation PR #75118 references this issue with closing syntax and remains unmerged, so the issue should stay open until that PR is reviewed, landed, or closed.

## What This Changes

Add an opt-in per-server `mcp.servers.<name>.injectCallerContext` flag so remote bundled MCP servers can receive OpenClaw caller identity headers without changing existing configs by default.

## Best Possible Solution

Review PR #75118 against the caller-identity trust boundary, then land a shared bundle-MCP/config change if accepted: default-off, owner-managed per-server opt-in, URL-only injection, case-insensitive non-overwrite header merge, stripped downstream flag, placeholder env population when needed, and matching schema/docs/tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure a remote `mcp.servers.<name>.url` entry with the proposed flag and inspect the generated bundle MCP config on main; the flag is not explicitly modeled, no caller headers are merged, and placeholder env is only emitted for the loopback runtime.

## Solution Assessment

Yes, pending maintainer approval. A per-server default-off owner opt-in is the narrow maintainable shape; a global flag or plugin self-opt-in would increase accidental caller-identity disclosure risk.
