## Summary

Keep open: the PR targets a real current-main responsiveness problem and has already added the missing changelog entry, but the latest supplied diff still masks `loadGatewayModelCatalog()` failures for default/configured `models.list` views. That P2 regression should block automerge until the timeout path preserves pre-timeout rejection behavior.

## What This Changes

The PR adds a 750 ms fallback for non-`all` `models.list` catalog discovery, focused gateway handler tests, CLI docs, and an Unreleased changelog entry.

## Best Possible Solution

Keep the timeout fallback for slow discovery, but race the raw catalog promise so pre-timeout rejections keep the existing error path; only observe or suppress late rejections after the timeout fallback has already won, and lock that behavior with a focused regression test.

## Reproduction Assessment

Yes. Current main reproduces the slow-path root cause by deferring `loadGatewayModelCatalog()` for a configured/default `models.list` request, and the proposed regression is reproducible by rejecting that promise before 750 ms.

## Solution Assessment

No as proposed. The timeout approach is the narrow maintainable fix for responsiveness, but the implementation should preserve existing catalog failure semantics rather than converting immediate failures into empty successful results.
