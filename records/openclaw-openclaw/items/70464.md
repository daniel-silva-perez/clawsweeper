## Summary

Keep this PR open. Current main has since added a generic plugin-owner cleanup path for plugin-created sessions, but it has not implemented this PR's stable per-workspace/per-phase dreaming session key, separate timestamped idempotency key, or pre-run cleanup. The PR is still substantive, but it likely needs a rebase and design adjustment because its gateway changes special-case `memory-core` in core and may narrow the newer generic plugin-owned cleanup seam.

## Best Possible Solution

Keep this PR open for a rebase onto current main. The best path is to preserve the generic `pluginOwnerId` cleanup seam, move memory-core-specific stable key and pre-cleanup behavior into `extensions/memory-core`, keep `idempotencyKey` unique per run, and add focused tests for request-scoped clients, fallback behavior, wrong-plugin/wrong-session denial, and stale-context cleanup.

## Risks / Open Questions

- The stable key design should be checked for concurrent same-workspace/phase dream runs and for stale-context cleanup failure behavior before landing.
