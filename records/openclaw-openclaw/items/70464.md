## Summary

Keep open. Current main still does not implement the stable memory-core dreaming session key, separate timestamped idempotency key, or pre-run stale-session cleanup, so this PR remains substantive. It needs maintainer review because its gateway changes narrow the current generic plugin-owned cleanup contract and touch a security-sensitive session-delete privilege path.

## What This Changes

The PR changes memory-core dreaming narrative sessions to stable workspace/phase keys with timestamped idempotency keys and pre-run cleanup, and updates gateway plugin delete-session scope handling plus related tests.

## Best Possible Solution

Land the memory-core-owned stable key, timestamped idempotency key, and stale-session pre-cleanup behavior with focused tests, while preserving main's generic `pluginOwnerId` cleanup seam and keeping any gateway changes extension-agnostic and ownership-checked.

## Risks / Open Questions

- Merging the gateway portion as-is may regress the documented generic plugin-owned `deleteSession` contract for third-party plugins by limiting synthetic admin cleanup to `memory-core` dreaming keys.
- The PR adds bundled plugin identity and key-prefix policy to gateway core, which conflicts with the repository boundary preference for extension-agnostic core seams when an owner contract already exists.
- Stable session reuse makes stale-context and concurrency behavior more important; failed pre-cleanup currently logs and proceeds, and the PR changes request-scoped fallback expectations from no cleanup attempt to one cleanup attempt.
- The diff touches privileged session deletion. Even without dependency, workflow, lockfile, or package-script changes, this needs maintainer security review before merge.
