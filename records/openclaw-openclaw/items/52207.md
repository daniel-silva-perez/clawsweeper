## Summary

Keep this PR open for maintainer judgment. Current main already shipped the SearXNG provider work, but Tavily freshness for the generic `web_search` provider is still missing, and this PR mixes that useful remaining feature with superseded SearXNG code plus unrelated model-runtime/config changes that introduce blocking regressions.

## What This Changes

The PR branch adds a bundled SearXNG web-search plugin with docs/config/onboarding wiring, adds Tavily generic `web_search` freshness-to-days support, updates generated metadata and audit/config surfaces, and changes Pi embedded model-provider normalization code.

## Best Possible Solution

Keep the shipped SearXNG implementation from #57317. If Tavily freshness is still desired, land a narrow Tavily-plugin change against current main that accepts OpenClaw's current freshness values, maps them to Tavily request/cache behavior deliberately, updates focused Tavily/web-search tests and docs, and avoids legacy config paths or unrelated provider-normalization changes.

## Reproduction Assessment

Yes. The review issues are reproducible by static inspection: current main documents `freshness` as `day`/`week`/`month`/`year`, the PR mapper handles only `pd`/`pw`/`pm`/`py`, the PR accepts a Tavily config path current Tavily resolution does not read, and the PR changes OpenRouter model normalization outside the web-search scope.

## Solution Assessment

No. This is not the best way to solve the remaining issue because it combines a superseded SearXNG provider, a useful Tavily freshness feature, and unrelated model-runtime changes; the maintainable path is a Tavily-only patch plus focused docs/tests.

  - body: The new OpenRouter branch rewrites every `openrouter` model using `openai-completions` to `openai-responses`, including custom or proxied `baseUrl` deployments.   - body: The Tavily mapper handles only `pd`/`pw`/`pm`/`py`, but current `web_search` docs and validation examples use `day`/`week`/`month`/`year`. A caller passing `freshness: "week"` would be accepted by the documented tool surface but silently send no Tavily recency filter.
  - confidence: 0.82
- **[P2] Do not accept Tavily keys on an unread config path:** `src/config/zod-schema.agent-runtime.ts:319-325`
  - body: Adding `tools.web.search.tavily.apiKey` to the strict runtime schema makes that config validate, but Tavily credential resolution still reads only `plugins.entries.tavily.config.webSearch.apiKey` or `TAVILY_API_KEY`. Users can follow the accepted config and still get missing-key errors at runtime.
  - confidence: 0.8
- **[P2] Count keyless SearXNG search in exposure audits:** `src/security/audit-extra.sync.ts:335-339`
  - body: SearXNG is intentionally usable with only a base URL, but this audit heuristic treats web search as configured only when an API key or key env var exists. Active keyless SearXNG search can therefore be underreported in small-model exposure auditing.
  - confidence: 0.78
- **[P2] Avoid defaulting SearXNG to blocked localhost:** `extensions/searxng/src/config.ts:5`
  - body: The provider defaults to `http://localhost:8080` while private-network access defaults to false and is enforced by the network guard. A default or onboarding-style SearXNG selection can therefore point at localhost and reject its own request until a separate flag is set.
  - confidence: 0.74
