## Summary

Keep open: current main already shipped the SearXNG portion through a separate merged PR, but it still lacks Tavily freshness support in the generic `web_search` provider. This PR is mixed-scope and not mergeable as-is because it also introduces unrelated provider-normalization changes and a security-audit blind spot.

## What This Changes

The PR branch adds SearXNG provider/docs/onboarding/config wiring, Tavily generic `web_search` freshness-to-days support, generated config/audit updates, and Pi embedded runner provider-normalization changes.

## Best Possible Solution

Keep the shipped SearXNG implementation from #57317, then land Tavily freshness as a narrow Tavily-plugin change against current main: add the generic freshness field, normalize supported values to Tavily `days`, include that value in request/cache behavior, and update focused Tavily/web-search docs and tests without changing provider transport normalization.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Keep custom OpenRouter endpoints on their configured API:** `src/agents/pi-embedded-runner/model.provider-normalization.ts:25-38`
  - body: The new `isOpenRouter` branch rewrites every OpenRouter model using `openai-completions` to `openai-responses`, even when the user configured a custom or proxied `baseUrl`. Existing proxies may only implement Chat Completions, so this unrelated change can break working OpenRouter deployments.
  - confidence: 0.86
- **[P2] Do not accept Tavily keys on an unread config path:** `src/config/zod-schema.agent-runtime.ts:319-325`
  - body: Adding `tools.web.search.tavily.apiKey` to the strict runtime schema makes that config validate, but the Tavily resolver still reads only `plugins.entries.tavily.config.webSearch.apiKey` or `TAVILY_API_KEY`. Users can follow the accepted config and still get missing-key errors at runtime.
  - confidence: 0.8
- **[P2] Count keyless SearXNG configs in web-search audits:** `src/security/audit-extra.sync.ts:335-339`
  - body: SearXNG is intentionally keyless when configured with a base URL, but the updated audit heuristic only treats SearXNG as enabled when an API key or env key exists. With `enabled` unset, active keyless SearXNG search can be reported as off in exposure auditing.
  - confidence: 0.78
