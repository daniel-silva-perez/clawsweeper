## Summary

Keep open. Current main has already shipped the SearXNG half through a different merged implementation, but the Tavily generic `web_search` freshness support is still missing and this PR is too mixed and stale to merge as-is.

## What This Changes

The PR branch adds a bundled SearXNG web-search plugin with docs/config/onboarding wiring, adds Tavily freshness-to-days support for generic `web_search`, updates generated config/audit surfaces, and changes Pi embedded runner provider normalization.

## Best Possible Solution

Keep the shipped SearXNG implementation from #57317. If Tavily freshness is still desired, land it as a narrow Tavily-plugin change against current main: add the generic freshness field using OpenClaw's current `day/week/month/year` shape, map it to the appropriate Tavily request parameter after checking the current API contract, include it in cache behavior, and update focused Tavily docs/tests without changing provider transport normalization or adding ignored config paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Preserve custom OpenRouter transport endpoints:** `src/agents/pi-embedded-runner/model.provider-normalization.ts:25-32`
  - body: The new `isOpenRouter` branch makes every OpenRouter `openai-completions` model use responses transport, even when a user configured a custom or proxied `baseUrl`. Existing OpenRouter-compatible proxies may only implement chat completions, so this unrelated change can break working deployments.
  - confidence: 0.86
- **[P2] Do not validate unread Tavily config:** `src/config/zod-schema.agent-runtime.ts:319-325`
  - body: Adding `tools.web.search.tavily.apiKey` to the strict runtime schema makes that config look supported, but Tavily credential resolution still reads only `plugins.entries.tavily.config.webSearch.apiKey` or `TAVILY_API_KEY`. Users can follow the accepted config and still hit missing-key errors at runtime.
  - confidence: 0.8
- **[P2] Count keyless SearXNG configs in audits:** `src/security/audit-extra.sync.ts:335-339`
  - body: SearXNG is intentionally usable with only a base URL, but this audit heuristic only treats it as configured when an API key or env key exists. Active keyless SearXNG search can therefore be reported as disabled in exposure auditing.
  - confidence: 0.78
- **[P2] Avoid defaulting SearXNG to blocked localhost:** `extensions/searxng/src/config.ts:5`
  - body: The provider defaults to `http://localhost:8080`, while private-network access defaults to false and is enforced by the network guard. A default or onboarding-style SearXNG selection can point at localhost and then reject its own request until a separate flag is set.
  - confidence: 0.74

## Security Review

Status: needs_attention

Summary: No dependency or lockfile supply-chain issue was found, but the PR introduces a concrete exposure-audit underreporting risk for keyless SearXNG web search.

Concerns:

  - body: The PR adds keyless SearXNG search but extends the web-search audit heuristic only with key-based checks. A config using a SearXNG base URL with `tools.web.search.enabled` unset can run web search while the audit reports it as disabled.
