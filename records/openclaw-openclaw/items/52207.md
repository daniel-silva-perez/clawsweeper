## Summary

Keep open for maintainer handling. Current main already ships the SearXNG provider, but Tavily freshness for the generic `web_search` path is still missing, so the PR is not fully obsolete; the submitted branch is mixed-scope and still has blocking correctness/security issues that should be split or replaced rather than merged as-is.

## What This Changes

The PR branch adds a bundled SearXNG web-search provider with docs/config/onboarding wiring, adds Tavily generic `web_search` freshness-to-days support, updates generated config/audit surfaces, and changes Pi embedded runner/provider normalization code.

## Best Possible Solution

Keep the shipped SearXNG implementation from #57317, then land Tavily freshness as a narrow Tavily-plugin change against current main: add the generic freshness field, map supported recency values to Tavily `days`, include the value in request/cache behavior, and update focused Tavily/web-search docs and tests without changing provider transport normalization.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Keep custom OpenRouter endpoints on their configured API:** `src/agents/pi-embedded-runner/model.provider-normalization.ts:25-38`
  - body: The new `isOpenRouter` branch rewrites every OpenRouter model using `openai-completions` to `openai-responses`, even when the user configured a custom or proxied `baseUrl`. Existing proxies may only implement Chat Completions, so this unrelated change can break working OpenRouter deployments.
  - confidence: 0.86
- **[P2] Do not accept Tavily keys on an unread config path:** `src/config/zod-schema.agent-runtime.ts:319-325`
  - body: Adding `tools.web.search.tavily.apiKey` to the strict runtime schema makes that config validate, but the Tavily resolver still reads only `plugins.entries.tavily.config.webSearch.apiKey` or `TAVILY_API_KEY`. Users can follow the accepted config and still get missing-key errors at runtime.
  - confidence: 0.8
- **[P2] Count keyless SearXNG configs in web-search audits:** `src/security/audit-extra.sync.ts:335-339`
  - body: SearXNG is intentionally usable with only a base URL, but the updated audit heuristic only treats SearXNG as configured when an API key or env key exists. Active keyless SearXNG search can therefore be reported as off in exposure auditing.
  - confidence: 0.78
- **[P2] Do not default SearXNG to a blocked local endpoint:** `extensions/searxng/src/config.ts:5`
  - body: The provider defaults to `http://localhost:8080`, but the network guard only allows private/loopback targets when `allowPrivateNetwork` resolves true. A default or onboarding-style SearXNG selection can point at localhost and then be rejected by its own guard.
  - confidence: 0.74
