## Summary

Keep open. Current main still routes TUI `/model list` and `/model status` through `sessions.patch` as model refs, so the PR is not obsolete, but the proposed branch should not merge as-is because it bypasses the shared model command semantics for session-aware listing and detailed status diagnostics.

## What This Changes

The PR changes the TUI `/model` handler and colocated tests so `/model list` opens the model selector and `/model status` prints session model information instead of treating those words as model IDs.

## Best Possible Solution

TUI should stop treating `list` and `status` as model IDs while preserving shared semantics: `/model list` should browse models with the active agent/session context, `/model status` should retain the detailed selected/runtime/default/auth/endpoint output, and regression tests should assert these subcommands do not call `sessions.patch`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Route /model list through the shared command path:** `src/tui/tui-command-handlers.ts:296`
  - body: The new `args === "list"` branch calls `openModelSelector()`, whose backend `models.list` request carries no active agent/session context and resolves visibility from the default workspace. In multi-agent TUI sessions this can show the wrong catalog; route this subcommand through the shared model command path or make the TUI list API session-aware.
  - confidence: 0.84
- **[P2] Preserve full /model status diagnostics:** `src/tui/tui-command-handlers.ts:298-304`
  - body: The new `status` branch replaces the shared `/model status` output with one `state.sessionInfo` line or `unknown`. That drops selected vs runtime model, default model, agent, auth file, provider auth state, and endpoint details, and can report unknown while the shared command can still answer.
  - confidence: 0.88
