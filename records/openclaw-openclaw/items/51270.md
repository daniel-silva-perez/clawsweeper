## Summary

Keep this PR open, but do not treat it as ready as-is. Current main still mishandles the TUI `/model list` and `/model status` inputs, so the branch is not obsolete; however, the proposed TUI-only handling drops the agent-aware model browsing path and the full shared `/model status` diagnostics that the docs and shared command implementation provide.

## What This Changes

This PR changes the TUI `/model` command handler so `/model list` opens the model selector, `/model status` prints the current session model, and the TUI command handler tests cover those branches.

## Best Possible Solution

Keep the TUI from treating `list` and `status` as model IDs while preserving the shared command semantics: `/model list` should use the active agent/session-aware model browsing path, `/model status` should retain the detailed auth/runtime/provider diagnostics, and regression tests should assert that these subcommands do not call `sessions.patch`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Route /model list through the agent-aware command path:** `src/tui/tui-command-handlers.ts:296`
  - body: The new `args === "list"` branch calls `openModelSelector()`, whose backend `listModels()` call has no active agent or session parameter and ultimately uses `models.list` with default-agent visibility. The shared `/model list` path passes the active agent/workspace/session context, so multi-agent TUI sessions can show the wrong catalog. Forward this subcommand through the shared command path or make the TUI model-list API session-aware.
  - confidence: 0.84
- **[P2] Preserve the full /model status diagnostics:** `src/tui/tui-command-handlers.ts:298-304`
  - body: The new `status` branch replaces the shared `/model status` behavior with a single `state.sessionInfo` line or `unknown`. The documented shared output includes selected vs runtime model, default model, agent, auth file, provider auth state, and endpoint details, and the gateway can still answer even when `sessionInfo` is sparse. Route this subcommand to the shared diagnostics instead of duplicating a local summary.
  - confidence: 0.88
