## Summary

Keep open. Current main still appears to let the reported Discord/PluralKit original-plus-proxy shape bypass the existing guards when the two events have different Discord message IDs and missing or nonmatching bound-webhook metadata; the paired implementation PR is closed unmerged.

## What This Changes

Discord bound-thread PluralKit messages should be canonicalized so an original/proxy pair produces one inbound turn, one run, one reply, and no lingering duplicate typing state.

## Best Possible Solution

Add Discord-owned provider/preflight canonicalization for bound-thread PluralKit original/proxy pairs before dispatch. Prefer an explicit Discord or PluralKit signal when available; otherwise use a short-lived fallback scoped to Discord bound-thread agent-session traffic, and ensure suppressed duplicates do not start or extend Discord typing indicators.

## Reproduction Assessment

Yes. The issue gives a live repro shape, and a focused regression can feed a bound-thread original/proxy pair with different Discord IDs and missing or nonmatching bound-webhook metadata through preflight/dispatch to verify whether two turns or typing callbacks are produced.

## Solution Assessment

Yes. A narrow Discord provider/preflight fix is the best direction; broadening the shared inbound dedupe by content would risk cross-channel behavior and should be avoided unless a generic contract need appears.
