## Summary

Keep open. Current main still appears to handle Discord bound-thread PluralKit original/proxy pairs only through webhook-id suppression and message-id dedupe, so a pair with different Discord IDs and missing or nonmatching webhook metadata can still reach dispatch as separate inbound pressure. The paired implementation PR #52007 is closed unmerged, and the issue discussion includes a live repro report plus useful clarification that the existing webhook suppression is only the first line of defense.

## What This Changes

The issue requests Discord bound-thread handling that collapses PluralKit original/proxy message pairs into one inbound turn and prevents duplicate-run or lingering-typing side effects.

## Best Possible Solution

Add Discord provider/preflight canonicalization for bound-thread PluralKit original/proxy pairs before they reach shared dispatch. Prefer an explicit Discord or PluralKit signal; if unavailable, use a short-lived fallback scoped to Discord bound-thread agent-session traffic and ensure suppressed duplicates do not start or extend typing indicators.
