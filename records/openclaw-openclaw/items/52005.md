## Summary

Keep open. Current main still has same-message Discord replay guarding, identity-based bound-thread webhook suppression, and shared MessageSid-based inbound dedupe, but I did not find a current path that canonicalizes a PluralKit original/proxy pair with different Discord message ids and missing or nonmatching webhook metadata before dispatch. The paired implementation PR #52007 is closed unmerged, and the discussion includes a live repro plus useful clarification that existing webhook suppression is only the first line of defense.

## What This Changes

Make PluralKit-originated messages in Discord bound threads behave as a single inbound turn without duplicate context, extra run pressure, or lingering typing side effects.

## Best Possible Solution

Add Discord-owned provider/preflight canonicalization for bound-thread PluralKit original/proxy pairs before they reach shared dispatch. Prefer an explicit Discord or PluralKit signal; if unavailable, use a short-lived fallback scoped to Discord bound-thread agent-session traffic and ensure suppressed duplicates do not start or extend typing indicators.
