## Summary

Keep open. Current main still relies on Discord bound-thread webhook-id suppression plus shared MessageSid-based inbound dedupe, and I did not find a path that canonicalizes a PluralKit original/proxy pair with different Discord message ids and missing or nonmatching webhook metadata before dispatch. The paired implementation PR #52007 is closed unmerged, while the discussion includes a live repro and a useful correction that existing webhook suppression is only the first line of defense.

## What This Changes

Discord bound-thread PluralKit original/proxy message pairs should be handled as one inbound turn without duplicate run pressure or lingering typing indicators.

## Best Possible Solution

Add Discord provider/preflight canonicalization for bound-thread PluralKit original/proxy pairs before they reach shared dispatch. Prefer an explicit Discord or PluralKit signal; if unavailable, use a short-lived fallback scoped to Discord bound-thread agent-session traffic and ensure suppressed duplicates do not start or extend typing indicators.
