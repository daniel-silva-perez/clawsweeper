## Summary

Keep open. Current main still handles same-message Discord replays and matching bound-thread webhook echoes, but it does not clearly canonicalize the reported PluralKit original/proxy pair when the two Discord messages have different IDs and missing or nonmatching webhook metadata; the paired implementation PR is closed unmerged.

## What This Changes

The issue requests that Discord bound threads treat a PluralKit original/proxied message pair as one inbound turn, without duplicate context pressure or lingering typing indicators.

## Best Possible Solution

Add Discord-owned provider/preflight canonicalization for bound-thread PluralKit original/proxy pairs before dispatch. Prefer an explicit Discord or PluralKit signal when available; if not, use a short-lived fallback scoped to Discord bound-thread agent-session traffic and ensure suppressed duplicates do not start or extend typing indicators.
