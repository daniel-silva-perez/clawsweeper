## Summary

Keep open. Current main still has Discord bound-thread webhook suppression keyed to webhook identity and shared inbound dedupe keyed to Discord message ID, but I did not find a path that canonicalizes a PluralKit original/proxy pair with different message IDs and missing or nonmatching webhook metadata before dispatch. The paired implementation PR #52007 is closed unmerged, while the discussion includes a live Discord/PluralKit repro and useful narrowing from other commenters.

## What This Changes

Ensure PluralKit-originated messages in Discord-bound threads are treated as one inbound turn with no duplicate context pressure, second run wobble, or lingering typing indicator.

## Best Possible Solution

Add Discord-owned provider/preflight canonicalization for bound-thread PluralKit original/proxy pairs before dispatch. Prefer an explicit Discord or PluralKit signal when available; if none is reliable, use a short-lived fallback scoped to Discord bound-thread agent-session traffic and make sure suppressed duplicates do not start or extend typing indicators.
