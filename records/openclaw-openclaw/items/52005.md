## Summary

Keep open. Current main still has identity-based Discord bound-thread webhook suppression and shared message-id dedupe, but I did not find a current-main path that collapses a PluralKit original/proxy pair with different Discord message ids and missing or nonmatching webhook metadata before dispatch. The linked implementation PR #52007 is closed unmerged, while the issue discussion includes a live repro report and useful clarification that the existing webhook suppression is only the first line of defense.

## What This Changes

The issue requests Discord bound-thread handling that treats a PluralKit original/proxied message pair as one inbound turn and prevents duplicate-run or typing-indicator side effects.

## Best Possible Solution

Add Discord provider/preflight canonicalization for bound-thread PluralKit original/proxy pairs before dispatch. Prefer an explicit Discord or PluralKit signal; if unavailable, use a short-lived fallback scoped to Discord bound-thread agent-session traffic and ensure suppressed duplicates do not start or extend typing indicators.
