## Summary

Keep open. Current main still has only identity-based Discord bound-thread webhook suppression plus shared message-id inbound dedupe, so the reported PluralKit original/proxy pair with different IDs or missing/nonmatching webhook metadata is not clearly collapsed before dispatch; the paired implementation PR #52007 is closed unmerged.

## What This Changes

The issue asks for PluralKit-originated messages in bound Discord threads to be handled as one inbound turn, without duplicate context pressure or lingering typing indicators.

## Best Possible Solution

Add Discord provider/preflight canonicalization for bound-thread PluralKit original/proxy pairs before they reach dispatch. Prefer an explicit Discord or PluralKit signal; if that is not available, use a short-lived fallback scoped to Discord bound-thread agent-session traffic and ensure suppressed duplicates do not start or extend typing indicators.
