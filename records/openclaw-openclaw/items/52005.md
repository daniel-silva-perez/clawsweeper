## Summary

Keep open. Current main still has the same observable gap described in the discussion: Discord bound-thread webhook suppression is keyed to a matching bound webhook id, PluralKit lookup is skipped for webhook-authored messages, and shared inbound dedupe still keys on MessageSid rather than collapsing an original/proxy pair with different ids. The paired implementation PR #52007 is closed unmerged, so it is not an implementation candidate currently landing this issue.

## Best Possible Solution

Keep this issue open. The best fix belongs in the Discord plugin preflight/process path: add a regression for a bound thread receiving the original Discord message plus a PluralKit proxy copy with different message ids and missing or nonmatching webhook metadata, collapse the pair before dispatch, and ensure suppressed duplicates do not start or extend Discord typing. Prefer an explicit PluralKit/Discord canonicalization signal; if unavailable, use a tightly scoped short-TTL bound-thread/account/session/content fallback.

## Risks / Open Questions

- A fallback based on message content can suppress legitimate repeated messages if it is broader than Discord bound-thread agent-session traffic, so any fix should stay narrow and short-lived.
- The ideal fix may be an explicit Discord/PluralKit canonical signal earlier in preflight; a content-clone fallback should be treated as a safety net if that signal is unavailable.
