## Summary

Keep open. Current main still appears to lack the provider-auth env scrub that motivated the security part of this PR, but the PR is not merge-ready as-is: the final branch mixes that useful fix with stale live-test HOME/profile changes that have unresolved correctness and secret-handling issues, and the approval/follow-up code has since evolved on main.

## What This Changes

The PR scrubs known provider auth env vars from exec child environments, changes two-phase exec approval pending behavior, routes webchat approval follow-ups internally with Control UI guidance, and adds live-test HOME isolation/docs updates.

## Best Possible Solution

Rebase and split the useful work: land a narrow provider-auth scrub in `src/infra/host-env-security.ts` using `listKnownProviderAuthEnvVarNames` with focused sanitizer coverage, then separately re-evaluate the two-phase no-route approval behavior against current `approval-shared.ts` and keep or update the newer follow-up routing tests already present on main. Drop or rework the stale live-test HOME changes so profile loading is live-only and profile-sourced config/real-home flags are honored.

## Reproduction Assessment

Yes. Source inspection on current main is enough to reproduce the provider-key leak path: inherited base env entries are copied unless they match the dangerous inherited policy, and provider auth env vars are not in that policy or consumed by the sanitizer. The PR review findings are also reproducible from the final patch ordering in `installTestEnv`.

## Solution Assessment

No. The provider-env scrub direction is the right minimal security fix, but this combined stale PR is not the best merge vehicle because it adds unrelated live-test profile handling with unresolved ordering bugs and targets older approval/follow-up code.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Gate profile sourcing to live runs:** `test/test-env.ts:430`
  - body: `installTestEnv()` now calls `loadProfileEnv(realHome)` for every test process, before HOME isolation. That makes normal unit/integration tests inherit local `~/.profile` variables such as provider keys and machine-specific config, so profile loading should stay behind live-test or explicit real-home opt-in.
  - confidence: 0.93
- **[P2] Snapshot live env after loading the profile:** `test/test-env.ts:426`
  - body: `liveEnvSnapshot` is captured before `loadProfileEnv(realHome)`, so profile-defined `OPENCLAW_CONFIG_PATH` or `OPENCLAW_STATE_DIR` are missing when `stageLiveTestState()` chooses which config/auth state to copy. Live tests can therefore stage the wrong default config even though the profile was sourced.
  - confidence: 0.9
- **[P2] Honor real-home opt-in loaded from profile:** `test/test-env.ts:424`
  - body: `allowRealHome` is computed before the profile import, but the new docs tell users they can set `OPENCLAW_LIVE_USE_REAL_HOME=1` for live runs. If that flag lives in `~/.profile`, this code still forces temp-HOME isolation unless it was already exported by the parent shell.
  - confidence: 0.9
