## Summary

Keep open. Current main still has the cumulative stream path, and this branch only fixes commit-time segment storage; the live preview still renders the full cumulative snapshot after committed segments, so merging it would not fully fix the WebChat duplication. The overlapping open Web UI streaming PRs mean this needs maintainer coordination around one canonical branch.

## What This Changes

The PR changes Control UI tool-stream segment commits to store only the newly streamed suffix after the total length of prior committed stream segments.

## Best Possible Solution

Land one focused WebChat streaming fix that treats cumulative assistant snapshots consistently across committed pre-tool segments and the live preview, preferably with a shared helper or explicit committed-offset state plus regression tests for multi text/tool cycles.

## Reproduction Assessment

Yes. The source path is enough to reproduce: gateway chat deltas are cumulative, Control UI stores the cumulative value in `chatStream`, and the renderer appends prior segments plus the live stream unchanged, so a pre-tool segment like `AAA` followed by live `AAABBB` repeats `AAA`.

## Solution Assessment

No. Segment slicing is a necessary part of the fix, but this PR is not the narrowest complete solution because the live preview needs the same committed-offset dedupe and focused regression coverage.
