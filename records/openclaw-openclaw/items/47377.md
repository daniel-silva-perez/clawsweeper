## Summary

Keep open. Current main still stores full cumulative WebChat stream snapshots as tool-interrupted segments and renders the live stream snapshot unchanged after those segments. This PR is a narrow useful attempt, but it does not cover the live-preview duplication path and overlaps with open PRs #46985 and #47399.

## What This Changes

The PR changes Control UI tool-stream segment commits to store only newly streamed text after the total length of prior stream segments instead of storing the full cumulative `chatStream`.

## Best Possible Solution

Land one focused WebChat streaming fix that treats cumulative assistant snapshots consistently across both committed pre-tool segments and the live preview, preferably with a small shared helper or explicit committed-offset state plus regression tests for multi-tool cumulative snapshots.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Deduplicate the live stream preview too:** `ui/src/ui/app-tool-stream.ts:444-451`
  - body: This only slices text when committing `chatStreamSegments`. After a tool starts, the next chat delta still assigns the gateway's full cumulative snapshot to `chatStream`, and `buildChatItems` renders that live stream unchanged after the committed segment. That means the pre-tool text can still appear again in the live bubble until another tool event or final reload. Preserve the committed offset for live rendering as well, or route both paths through the same dedupe helper.
  - confidence: 0.86
