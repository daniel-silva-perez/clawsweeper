## Summary

Keep open. Current main has Feishu direct messages in documented scope and still maps `user:ou_...` targets to `open_id`, but the Feishu send and chat/member-info paths still do not unwrap Lark SDK/Axios response bodies from HTTP 400 rejections. The issue discussion adds a native Feishu `code: 9499` response with `log_id`, plus a v2026.4.27 failure report, so this remains an active Feishu plugin diagnostic/behavior bug.

## What This Changes

The issue asks for Feishu message-tool direct sends to `user:ou_...` targets to stop failing with opaque HTTP 400 errors while group sends continue to work.

## Best Possible Solution

Keep the issue open and first add a plugin-local Feishu error-body extraction path that preserves vendor `code`/`msg`/`log_id` for message sends and `feishu_chat` lookups. After those details are visible, maintainers can decide whether the DM failure needs a payload fix, a Feishu permission/setup docs update, or a separate package/runtime fix.
