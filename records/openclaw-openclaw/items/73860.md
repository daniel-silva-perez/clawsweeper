## Summary

Keep open. Current main recognizes Feishu `user:ou_...` targets as `open_id`, so the obvious target-parsing bug is not present, but the Feishu send and chat tool paths still appear to let Axios-shaped Lark SDK HTTP 400 failures propagate as generic messages without preserving the vendor `code`, `msg`, `log_id`, or troubleshooter body captured in the issue comments. The reported live DM failure may still be Feishu permission or tenant policy, so this should not be closed as implemented.

## What This Changes

The issue reports HTTP 400 failures when the shared message tool sends Feishu direct messages to user targets while group sends continue to work.

## Best Possible Solution

Keep the Feishu bundled plugin’s `user:ou_...` direct-send contract, add a Feishu-specific API error normalizer that preserves vendor status, `code`, `msg`, `log_id`, and troubleshooter details across message send and `feishu_chat` operations, then use those richer errors to determine whether the remaining 9499 DM failure is a code defect or an operator-side Feishu permission/configuration problem.

## Reproduction Assessment

Unclear for the live DM delivery failure because this review did not have Feishu credentials to reproduce `im.message.create` against a real tenant. There is a high-confidence local reproduction path for the diagnostic part: mock `client.im.message.create` or `contact.user.get` to throw an Axios-shaped 400 with `response.data` and verify current code reports only the generic error instead of Feishu `code`/`log_id`.

## Solution Assessment

No, current main is not the best solution because it preserves the open_id target type but does not preserve Feishu’s actionable error body when the SDK throws. The narrow maintainable fix is plugin-local error normalization plus tests, without changing core message-tool routing or the Lark SDK dependency first.
