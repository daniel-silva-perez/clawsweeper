## Summary

Keep open. Current main already maps `user:ou_...` Feishu message targets to `receive_id_type: open_id`, so the old receive-id-type regression is not the remaining issue. The current Feishu send and `feishu_chat` paths still do not preserve Axios/Lark HTTP error response bodies, while the reporter supplied Feishu `im.message.create` vendor errors with `code: 9499`, `log_id`, and troubleshooter details that OpenClaw currently hides behind `Request failed with status code 400`.

## Best Possible Solution

Keep this issue open. First land a narrow Feishu-plugin diagnostics fix that preserves redacted Feishu vendor error bodies for message sends and `feishu_chat` lookups, including `code`, `msg`, `log_id`, and troubleshooter when present. Then use those details to decide whether the DM `9499` path needs a send/content fix, a setup/permission docs update, or a vendor workaround.
