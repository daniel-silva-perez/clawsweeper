## Summary

Keep open. Current main preserves Feishu `user:ou_...` targets as `open_id`, so the obvious target-prefix regression is not present, but the issue discussion includes live Feishu 9499 responses and current send/chat code still drops the vendor response body behind generic Axios error text.

## What This Changes

The issue reports that Feishu direct-message sends through the shared message tool return HTTP 400 for `user:ou_...` targets while group sends work.

## Best Possible Solution

Keep the current `user:ou_...` direct-message target contract, add Feishu-local API error normalization for message send and `feishu_chat`, and then use the preserved Feishu code/log/troubleshooter details to decide whether the remaining 9499 is app configuration guidance or a second code fix.

## Reproduction Assessment

Unclear for the live direct-message delivery failure because this review did not run against a real Feishu tenant. Yes for the confirmed diagnostic gap: mock `client.im.message.create` or `client.contact.user.get` to reject with an Axios-style HTTP 400 body and verify current main reports only the generic error message.

## Solution Assessment

No, current main is not sufficient because it preserves the target type but still loses the Feishu response body needed to debug the actual 9499 failure. A narrow plugin-local error normalizer is the best next repair before changing routing or delivery semantics.
