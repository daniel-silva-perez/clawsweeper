## Summary

Keep open. Current main still has the same Feishu direct-user send path: `message` send resolves `target=user:ou_...` through the Feishu plugin, normalizes it to `ou_...`, infers `receive_id_type: "open_id"`, and calls Feishu `im.message.create`. That means the older missing-`receive_id_type` and unknown-channel bugs do not explain this current v2026.4.26 400; the next step needs the Feishu/Lark response body or log_id to distinguish code behavior from scope, tenant visibility, or bot-DM policy.

## Best Possible Solution

Keep open and investigate in the Feishu plugin. Capture the full Feishu/Lark response body for both `im.message.create` and `contact.user.get`, add a focused regression test for `message` send with `target=user:ou_...`, then either fix a concrete target/ID-type mismatch, document required Feishu scopes and tenant/direct-DM constraints, or improve Feishu SDK/Axios error unwrapping so users see the actionable vendor error.
