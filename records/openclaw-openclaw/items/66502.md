## Summary

Keep this PR open because current main still lacks the stable pnpm package-root fallback, but do not merge the patch as-is. The proposed candidate ordering still chooses a version-pinned `.pnpm/.../dist/index.js` path in the common case where both the versioned package path and stable `node_modules/openclaw` symlink are accessible.

## What This Changes

The PR adds pnpm package-root candidate discovery and a regression test so gateway service ProgramArguments can resolve from a versioned `.pnpm/.../openclaw.mjs` entrypoint to a stable `node_modules/openclaw/dist` entrypoint.

## Best Possible Solution

Keep the PR open as the focused implementation candidate, but revise the resolver so stable pnpm package-root candidates are tried before version-pinned package dist candidates for `.pnpm/<pkg>@<version>/node_modules/<pkg>/openclaw.mjs` inputs. Add a regression where both stable and versioned dist entries are accessible, then rebase onto the current wrapper-aware daemon install code.

## Reproduction Assessment

Yes. A deterministic unit fixture can set `process.argv[1]` to a versioned pnpm `openclaw.mjs` path and make both the versioned `dist/index.js` and stable `node_modules/openclaw/dist/index.js` accessible; the PR’s current candidate order returns the versioned path first.

## Solution Assessment

No. The direction is right, but appending stable pnpm candidates after normal package-root dist candidates does not solve the real pnpm global case where both paths exist; the stable candidate needs priority and direct regression coverage.
