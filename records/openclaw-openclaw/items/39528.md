## Summary

Keep this issue open. Current main has meaningful context and gateway-write hardening, but the remaining sensitive behavior is still real: owner-authorized model-callable gateway actions can restart, patch/apply allowed config, or run updates without a separate recent user-consent artifact.

## What This Changes

The issue asks OpenClaw to preserve soul/user/memory and red-line instructions across long multi-turn sessions and to prevent model-initiated config edits or gateway restarts unless the user explicitly consents.

## Best Possible Solution

Keep this open as a product/security-behavior bug and add a hard, recent explicit-consent mechanism for model-initiated gateway restart, config.apply, config.patch, and update.run. The end state should preserve the existing owner-only exposure, admin scopes, allowlist, dangerous-flag blocking, base-hash checks, rate limits, and post-compaction red-line refresh, while adding server-enforced proof that the user approved the specific sensitive action recently enough.

## Reproduction Assessment

Unclear for the reporter's exact long-session model behavior because no transcript or logs were provided. Yes for the remaining consent gap: current source shows the owner gateway tool and server protocol can perform restart, allowed config writes, and update.run without any approval id or consent token field.

## Solution Assessment

No, the current implementation is not the best complete solution for the reported behavior because prompt reminders and owner/admin checks are not the same as explicit per-action consent. The safer direction is a hard approval/consent contract enforced at the agent gateway tool and preferably at the RPC/server layer too.
