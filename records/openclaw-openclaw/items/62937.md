## Summary

Keep open. Current main has not implemented the PR's intended macOS Seatbelt support, and the paired feature request remains open, but the draft patch should not merge as-is because it changes the host exec security boundary, is dirty against main, and leaves the advertised default profile allow-by-default.

## What This Changes

The PR adds a macOS-only `sandboxProfile` exec parameter, wraps host `runExecProcess` calls with `/usr/bin/sandbox-exec`, and adds a Seatbelt helper module plus profile-generation tests.

## Best Possible Solution

If native macOS Seatbelt support is accepted, implement it as a maintainer-configured sandbox backend or backend capability under `agents.defaults.sandbox`, keep defaults fail-closed and non-model-selectable, update schema/docs, and add unit plus macOS live validation for network and filesystem boundaries. If maintainers reject Seatbelt in core, close this PR and the paired feature request after that explicit decision.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Deny file operations before allowlisting paths:** `src/agents/bash-tools.exec-runtime.ts:130-135`
  - body: The default profile starts with `(allow default)` and never denies file operations before adding path-specific allow rules, so `sandboxProfile: "default"` would not enforce the advertised read/write restriction. A sandbox profile used as a security boundary needs to deny by default and then explicitly allow the required paths.
  - confidence: 0.83
- **[P2] Export the profile helper used by the new test:** `src/agents/bash-tools.sandbox-exec.test.ts:3`
  - body: The new test imports `getSandboxProfileString` from `bash-tools.exec-runtime.js`, but the helper added in the runtime file is local. Test collection or typecheck will fail until the helper is exported or the test exercises a public API.
  - confidence: 0.96
