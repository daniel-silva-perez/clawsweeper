## Summary

Keep open. Current main still lacks the proposed macOS Seatbelt exec backend or `sandboxProfile` tool parameter, and this PR is paired with the still-open feature request #58730. The proposed patch is not mergeable as-is because it has blocking correctness and security issues in the advertised sandbox policy and its new test import, so it needs maintainer/security review rather than cleanup closure.

## What This Changes

The PR adds a macOS `sandboxProfile` exec parameter, wraps host `runExecProcess` calls with `/usr/bin/sandbox-exec` on Darwin, and adds a Seatbelt helper module plus profile-generation tests.

## Best Possible Solution

If native macOS Seatbelt support is accepted, implement it as a maintainer-configured sandbox backend or backend capability under `agents.defaults.sandbox`, keep defaults fail-closed and non-model-selectable, update schema/docs, and add regression plus macOS live validation for network and filesystem boundaries. If maintainers reject Seatbelt in core, close this PR and the paired feature request together after that explicit decision.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Deny file operations before allowlisting paths:** `src/agents/bash-tools.exec-runtime.ts:130-135`
  - body: The default profile starts with `(allow default)` and never denies file operations before adding path-specific allow rules, so `sandboxProfile: "default"` would not enforce the advertised read/write restriction. A sandbox profile used as a security boundary needs to deny by default and then explicitly allow the required paths.
  - confidence: 0.83
- **[P2] Export the profile helper used by the new test:** `src/agents/bash-tools.sandbox-exec.test.ts:3`
  - body: The new test imports `getSandboxProfileString` from `bash-tools.exec-runtime.js`, but the helper added in the runtime file is local. Test collection/typecheck will fail until the helper is exported or the test exercises a public API.
  - confidence: 0.96
