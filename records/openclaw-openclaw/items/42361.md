## Summary

Current main still does not implement the PR's headline `web_fetch` network-error diagnostic. The direct fetch catch path tries provider fallback and then rethrows the original transport error unchanged, so a no-fallback DNS/connection/timeout failure can still surface as bare `fetch failed`. The PR contains a useful fix direction for #27081, but it is bundled with unrelated Feishu, Telegram, model/config, tool-result, and package override changes that need separate maintainer review.

## Best Possible Solution

Keep this PR open for maintainer rework or replacement. The best path is a narrow `web_fetch`-only patch that wraps direct network-level failures after provider fallback misses, extracts `error.cause.message` or `error.cause.code`, preserves the original error as `cause`, includes safe requested-URL context, and adds a regression test for the no-fallback network-error case. Split or drop the unrelated Feishu, Telegram, config/schema, model/table, image-helper, tool-result, and package override changes.

## Risks / Open Questions

- Landing the branch as-is would mix the focused `web_fetch` diagnostic fix with unrelated channel, provider/config, model-scan, tool-result, and dependency-resolution changes.
- The package override deletion is supply-chain/install-sensitive and, in the provided PR file list, is not paired with a lockfile update.
- The Feishu external-key refactor in the PR broadens accepted path separators for values that are used as Feishu API path parameters.
- The Telegram negative-ID change appears to conflict with current Telegram allowlist guidance that sender allowlists expect numeric sender user IDs and negative group chat IDs belong under `channels.telegram.groups`.
- The final web-fetch patch should decide whether chat-facing diagnostics may include full query strings or should redact sensitive URL components while still preserving useful requested-URL context.
