## Summary

Keep this PR open. Current main still lacks the run-generation registry, ReplyOperation generation stamp, abort invalidation, and block-reply stale-run fence proposed here, while the paired umbrella issue remains open; this is an active draft implementation candidate rather than obsolete cleanup.

## What This Changes

The PR adds a per-session run-generation registry, stamps ReplyOperation with a captured generation/isCurrent helper, increments generation on abort paths, adds an optional block-reply stale-run fence, wires that fence into agent-runner execution, and adds focused tests.

## Best Possible Solution

Keep this draft PR open and review it against #70319. The best end state is either a focused generation-fence foundation with a corrected delivery-fence test and explicit follow-up tracking for the deferred surfaces, or a documented maintainer decision to use a different unified cancellation architecture.

## Reproduction Assessment

Yes, partially: source inspection on current main gives a high-confidence static reproduction of the missing generation registry, ReplyOperation stamp, abort invalidation, and block-delivery fence. A live timing race was not run because this sweep is read-only, but the absent code path is concrete.

## Solution Assessment

Unclear/partial: the PR is a plausible foundation and is not implemented on main, but it only wires one emission site and leaves several #70319 guarantees to follow-up work. The narrow maintainable path is to review the architecture, fix the weak delivery-fence test, and either land this as an explicit foundation or ask for a broader single PR.
