## Summary

Keep open: this is a fresh Telegram/Gateway reliability report against the current release window, and current main does not prove the persistent unresponsive-after-error state has been fixed. The inspected code has partial recovery and diagnostics for generic runner failures, Telegram dispatch fallback replies, session-corruption resets, channel health restarts, and polling stalls, but those paths do not establish recovery from normal Telegram messages becoming permanently wedged until a full gateway restart.

## What This Changes

The issue asks OpenClaw to diagnose intermittent Telegram gateway failures that surface a generic error, add automatic recovery so manual gateway restarts are not required, and improve logging for future diagnosis.

## Best Possible Solution

Keep this issue open while maintainers collect the gateway log window and diagnostics export around the reported failures. The desired end state is a narrow fix in the owning layer: reset a corrupt session if a transcript/turn state is wedged, clean up Telegram dispatch or lane state if a failed turn leaves future messages blocked, improve polling/channel-health recovery if transport liveness is the cause, and add targeted diagnostics so the next generic fallback records the actionable failure class.

## Reproduction Assessment

Unclear: the issue gives real occurrence dates and the exact Telegram fallback text, but it has no reliable trigger or logs. Current-main source inspection confirms matching fallback and partial recovery paths, not a high-confidence reproduction of the permanent unresponsive state.

## Solution Assessment

Unclear: automatic recovery is a reasonable goal, but the best implementation depends on the missing root cause. The safer path is targeted recovery plus logging in the layer proven by gateway logs rather than an unconditional gateway restart after any generic Telegram error.
