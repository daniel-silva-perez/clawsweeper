## Summary

Keep open: this is a fresh Telegram/Gateway reliability report, and current main still does not prove that the recurring generic Telegram failure recovers without a manual gateway restart. The code has several partial protections, but they cover classified runner failures, stale lanes without active work, polling transport stalls, and best-effort fallback replies rather than the reported permanent unresponsive state after an intermittent normal message failure.

## What This Changes

The issue requests root-cause diagnosis, automatic recovery, and better logging for intermittent Telegram Gateway failures that leave the assistant unresponsive until `openclaw gateway restart`.

## Best Possible Solution

Keep the issue open while maintainers collect the gateway log window and diagnostic state around the reported failures. The durable fix should land in the layer proven by those logs: Telegram dispatch cleanup if the lane is left blocked, auto-releasing stale session bookkeeping if no active work exists, bounded abort if an embedded run is genuinely hung, polling recovery if transport liveness is the cause, and targeted logging that records the failure class before the generic fallback is sent.

## Reproduction Assessment

Unclear: the issue has credible occurrence dates and the exact user-visible Telegram fallback, but no reliable trigger or logs. Source inspection confirms matching fallback and partial recovery paths, not a high-confidence reproduction of the permanent unresponsive state.

## Solution Assessment

Unclear: automatic recovery is the right outcome, but an unconditional gateway restart after a generic Telegram error would be too broad without root-cause evidence. The best path is targeted recovery and diagnostics in the owning layer identified from logs.
