## Summary

Keep this PR open. Current main still drops mixed-message `/model <provider/model> <message>` before model selection can apply it, while the PR is a relevant implementation candidate but introduces blocking non-persistence/default-selection regressions around stored session model and auth-profile overrides.

## What This Changes

The PR preserves mixed-message `/model` selections as one-shot directive state, applies the selected provider/model/auth profile to the current reply run, updates slash-command docs, and adds regression coverage for non-persistent follow-up behavior.

## Best Possible Solution

Preserve mixed-message `/model` as separate one-shot state, apply provider/model/profile only to the current run, and recompute context/thinking/reasoning defaults through a non-mutating model-selection path that explicitly suppresses stored session model/auth-profile overrides. Add regressions where a session already has a persistent model override and a persistent auth profile before the one-shot switch, then verify the follow-up turn still uses the stored session state.

## Reproduction Assessment

Yes. Current main can be checked with an authorized mixed message such as `/model openai/gpt-4.1-mini please sync now`: parsing strips and records the directive, but `resolveReplyDirectives()` clears `hasModelDirective` once normal body text remains, so the run continues on the session/default model. The PR regressions are also reproducible by seeding a stored session model/auth-profile override before sending the one-shot message.

## Solution Assessment

No. The PR takes the right product direction, but the current patch is not the best implementation because it reuses the persistent session model-selection path for one-shot recomputation, which can reapply stored model overrides and mutate stored auth-profile overrides. A dedicated non-persistent recompute path or an explicit suppress-stored-overrides parameter would be safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve session auth profile during one-shot recompute:** `src/auto-reply/reply/get-reply-directives.ts:466-480`
  - body: After a mixed-message `/model` changes `provider`/`model`, this rebuild passes the live `sessionEntry` and `sessionStore` back into `createModelSelectionState`. That helper clears `sessionEntry.authProfileOverride` when the stored profile belongs to the old provider, so a one-shot cross-provider message can permanently delete the user's persistent auth-profile override even though this syntax is meant to be non-persistent.
  - confidence: 0.88
- **[P2] Compute defaults from the one-shot model:** `src/auto-reply/reply/get-reply-directives.ts:466-480`
  - body: This recompute uses `hasModelDirective: false` while still passing session state, so `createModelSelectionState` can reapply a stored session model override before `resolveDefaultThinkingLevel()` and `resolveDefaultReasoningLevel()` run. The agent run uses the one-shot provider/model, but thinking or reasoning defaults can come from the old session model; suppress stored overrides for this one-shot refresh or avoid the persistent session path.
  - confidence: 0.9
