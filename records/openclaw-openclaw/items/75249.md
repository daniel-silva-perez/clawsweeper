## Summary

Keep open. Current main still clones `event.context` before plugin hook handlers, which conflicts with the documented mutable `agent:bootstrap` context. The PR is a narrow fix with relevant regression coverage and I did not find a blocking correctness or security issue, but it still needs normal maintainer review, CI, and likely a changelog entry before landing.

## What This Changes

The PR changes the plugin registry hook wrapper to pass the original hook context with temporary `pluginConfig` injection and adds bootstrap-hook regression coverage for mutation persistence and `pluginConfig` cleanup.

## Best Possible Solution

Land this PR or an equivalent narrow change that preserves the original hook context for mutable hook contracts while restoring/removing temporary `pluginConfig` after each handler, with CI proof and the required changelog entry.

## Reproduction Assessment

Yes. A deterministic reproduction is to register a plugin `agent:bootstrap` hook that assigns `event.context.bootstrapFiles`, then call `applyBootstrapHookOverrides`; current main’s wrapper sends that assignment to a cloned context while core reads the original event context afterward.

## Solution Assessment

Yes. Passing the original context and restoring `pluginConfig` in `finally` is the smallest compatible fix for the existing mutable hook contract; replacing the API with immutable return values would be a larger follow-up design, not the right immediate regression fix.
