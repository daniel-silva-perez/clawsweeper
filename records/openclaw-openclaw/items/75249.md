## Summary

Keep this PR open for normal maintainer review. Current main still clones plugin hook context in `api.registerHook`, which drops `agent:bootstrap` context replacements even though the bootstrap helper and docs treat `context.bootstrapFiles` as mutable; the PR is a narrow fix and I found no blocking correctness or security issue in the patch.

## What This Changes

The PR changes `src/plugins/registry.ts` so plugin hook handlers receive the original hook context with temporary `pluginConfig` injection, and adds `src/agents/bootstrap-hooks.test.ts` regression coverage for bootstrap mutation preservation and cleanup behavior.

## Best Possible Solution

Land this PR or an equivalent narrow change after changelog and CI proof, preserving mutable hook-context behavior while ensuring per-handler `pluginConfig` is restored or removed after each plugin hook call.

## Reproduction Assessment

Yes. Register a plugin `agent:bootstrap` hook through `api.registerHook` that assigns `event.context.bootstrapFiles`, then call `applyBootstrapHookOverrides`; current main sends that assignment to a cloned context while core reads the original event context afterward.

## Solution Assessment

Yes. Passing the original context and restoring `pluginConfig` in `finally` is the smallest compatible fix for the existing mutable hook contract; an immutable return-value hook API would be a larger design change, not the right immediate regression fix.
