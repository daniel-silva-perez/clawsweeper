## Summary

Keep open. Current main still has the voice-call CLI operational commands creating a process-local runtime that starts the webhook listener, while the Gateway already registers voicecall RPC methods that could own the live runtime. The newer setup-helper comments add useful lifecycle evidence, but the original port-collision bug remains source-level reproducible enough for a focused plugin repair.

## What This Changes

The issue asks for `openclaw voicecall call` and related voice-call CLI paths to avoid binding the webhook port when a running Gateway already owns it, with follow-up evidence of a `voicecall setup --json` lifecycle hang.

## Best Possible Solution

Keep the runtime owned by the Gateway for normal operational voice-call commands, delegate CLI call/control actions through the existing `voicecall.*` Gateway RPC methods when a Gateway is available or explicitly selected, preserve a deliberate standalone fallback, and make read-only setup/status commands exit without initializing long-lived voice runtime resources.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path for the original bug: Gateway startup starts the voice-call runtime and listener, and a separate `voicecall call` process still reaches `ensureRuntime()` -> `createVoiceCallRuntime()` -> `webhookServer.start()` -> `listen(3334, 127.0.0.1)`.

## Solution Assessment

Yes. A plugin-local Gateway-aware CLI path using the existing `openclaw/plugin-sdk/gateway-runtime` helper and registered `voicecall.*` methods is the narrowest maintainable direction; adding a broad core runtime registry would be larger than needed.
