## Summary

Keep open. Current main still routes `openclaw voicecall call` and the other operational voice-call CLI commands through a process-local runtime that starts the webhook listener, while the Gateway already has matching `voicecall.*` RPC methods available. The report remains valid and narrow enough for a focused plugin-local repair.

## What This Changes

The issue requests Gateway-aware behavior for `openclaw voicecall call` so the CLI does not bind the voice-call webhook port when the Gateway already owns it.

## Best Possible Solution

Add a Gateway-aware path in the voice-call plugin CLI that delegates operational commands through the existing `voicecall.*` Gateway RPC methods when selected, keeps local runtime startup for explicit standalone use, and emits an actionable diagnostic when the local fallback hits the Gateway-owned webhook port.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: one Gateway process starts the voice-call service and binds the webhook, while a separate CLI process reaches `ensureRuntime()` -> `createVoiceCallRuntime()` -> `webhookServer.start()` -> `listen(3334, 127.0.0.1)`.

## Solution Assessment

Yes. Using the existing plugin-facing Gateway RPC helper and the already registered `voicecall.*` methods is the narrowest maintainable fix; moving ownership into core or adding a cross-process runtime registry would be broader than needed.
