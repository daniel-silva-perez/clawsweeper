## Summary

Keep open. Current main still wires the operational `voicecall` CLI commands to a process-local voice-call runtime, and runtime creation still starts the webhook listener on the configured bind/port. The existing #58115 fix covers shared runtime reuse inside one process, but it does not solve a separate CLI process delegating to a Gateway-owned runtime.

## What This Changes

The issue asks for `openclaw voicecall call` to avoid binding the same voice-call webhook port already owned by a running Gateway, preferably by delegating to the Gateway or by emitting a clearer diagnostic.

## Best Possible Solution

Add a plugin-local Gateway-aware path for operational `voicecall` CLI commands using the supported `openclaw/plugin-sdk/gateway-runtime` surface and the existing `voicecall.*` Gateway methods, while preserving standalone local runtime behavior for explicit local use and improving the diagnostic when a local bind collides with a Gateway-owned webhook port.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: a Gateway service start creates the voice-call runtime and binds the webhook, while a separate `openclaw voicecall call` process runs `ensureRuntime()` again and reaches another `listen(3334, 127.0.0.1)`. A live Twilio call was not needed to confirm the binding path.

## Solution Assessment

Yes. Delegating the CLI through the existing plugin-owned Gateway RPC methods is the narrowest maintainable fix; a new core runtime registry or cross-process singleton would be broader than this plugin-local collision requires.

Fix openclaw/openclaw issue 72345. In the voice-call plugin, make operational `openclaw voicecall` CLI commands Gateway-aware so they do not start a second local `VoiceCallWebhookServer` when the Gateway already owns the voice-call webhook port. Keep the implementation within `extensions/voice-call` and supported SDK imports such as `openclaw/plugin-sdk/gateway-runtime`; do not import core `src/**` internals or add bundled plugin IDs to core.
