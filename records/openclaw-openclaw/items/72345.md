## Summary

Keep open. Current main still routes `openclaw voicecall call` and related operational CLI commands through a process-local voice-call runtime, and runtime creation still starts the webhook server on the configured bind/port. The existing Gateway RPC methods are already present, and the related #58115 fix only proves in-process runtime sharing; it does not provide cross-process CLI delegation when the Gateway already owns the webhook listener.

## What This Changes

Make `openclaw voicecall call` and related voice-call CLI operations work when the Gateway is already running the voice-call webhook listener.

## Best Possible Solution

The desired end state is for voice-call CLI call/control commands to share the Gateway-owned runtime through existing Gateway RPC when available, keep setup and dry-run diagnostics runtime-free, and retain a deliberate documented standalone path without expanding core ownership boundaries.

## Reproduction Assessment

Yes. The source-level reproduction path is high confidence: Gateway startup owns the voice-call runtime/listener, while a separate `voicecall call` process still reaches `ensureRuntime()` -> `createVoiceCallRuntime()` -> `webhookServer.start()` -> `listen(3334, 127.0.0.1)`. The issue comments add live macOS/Twilio evidence for both the original port collision and a remaining CLI lifecycle symptom.

## Solution Assessment

Yes. A plugin-local Gateway-aware CLI path using the existing `openclaw/plugin-sdk/gateway-runtime` helper and registered `voicecall.*` methods is the narrowest maintainable fix; a broad core runtime registry would be larger than needed for this bug.
