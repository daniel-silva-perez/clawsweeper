## Summary

Current main still has the reported voice-call CLI/gateway port-conflict path. The plugin CLI commands still call the process-local `ensureRuntime()`, that runtime still starts `VoiceCallWebhookServer`, and the webhook server still listens on the configured default `127.0.0.1:3334`. The existing `voicecall.*` gateway RPC methods and public `plugin-sdk/gateway-runtime` helper make a focused fix feasible, but the CLI has not been wired to use them yet.

## Best Possible Solution

Keep this issue open and implement a gateway-aware `voicecall` CLI path. The best fix is to delegate `call`, `start`, `continue`, `speak`, `dtmf`, `end`, and `status` through the existing `voicecall.*` Gateway RPC methods when the user selects a running Gateway target or has remote Gateway mode configured, while preserving the current local-runtime behavior for standalone use. Add a regression test proving delegated CLI commands do not call `createVoiceCallRuntime()` and do not start a webhook listener; also add a targeted `EADDRINUSE` error if local runtime startup collides with the Gateway-owned webhook port.
