## Summary

Keep open. Current main still routes the voice-call CLI through the process-local runtime, and that runtime still starts a webhook listener on the configured voice-call bind/port before placing calls, so a separate CLI process can still collide with a Gateway-owned listener. Existing gateway RPC methods and the public gateway runtime helper make this a focused repair candidate rather than a product-scope decision.

## What This Changes

The issue requests that `openclaw voicecall call` avoid starting a second voice-call webhook runtime when a running Gateway already owns the plugin webhook port.

## Best Possible Solution

Implement a gateway-aware `voicecall` CLI path that delegates operational commands to the existing `voicecall.*` Gateway RPC methods when the user selects a running Gateway target or configured remote Gateway mode, while keeping local runtime startup for explicit standalone use. Add a clearer local-runtime `EADDRINUSE` diagnostic for the fallback path.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: a Gateway process starts the voice-call webhook, while a separate CLI process has no shared `globalThis` runtime and reaches `createVoiceCallRuntime()` -> `webhookServer.start()` -> `listen(3334, 127.0.0.1)`.

## Solution Assessment

Yes. Delegating through the already-registered `voicecall.*` Gateway RPC methods is the narrowest maintainable fix; adding a separate cross-process runtime registry or moving voice-call ownership into core would be broader than needed.

## Repair Work Prompt

Fix the voice-call CLI/Gateway port conflict reported in https://github.com/openclaw/openclaw/issues/72345. Current `openclaw voicecall call` and related operational commands create a local voice-call runtime, which starts `VoiceCallWebhookServer` on the same configured port already owned by a running Gateway. Implement a gateway-aware path in the voice-call plugin CLI that delegates `call`, `start`, `continue`, `speak`, `dtmf`, `end`, and `status` through the existing `voicecall.*` Gateway RPC methods when the user selects/configures a Gateway target; preserve the existing standalone local-runtime behavior for explicit local use. Use the public `openclaw/plugin-sdk/gateway-runtime` surface or a local plugin runtime barrel, and do not import core `src/**` internals from extension production code. Add regression coverage proving delegated CLI commands call the Gateway RPC helper and do not call `createVoiceCallRuntime()` or start a webhook listener. Add a clearer `EADDRINUSE` diagnostic for the local-runtime fallback path that tells the user the Gateway may already be owning the voice-call webhook port. Update user-facing voicecall docs and add a changelog fix entry if CLI flags or error text change, crediting @serrurco. Keep the fix limited to voice-call CLI/runtime delegation and tests.
