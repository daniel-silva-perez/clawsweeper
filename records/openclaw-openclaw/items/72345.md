## Summary

Keep open. Current main still has the same source-level failure mode: operational `voicecall` CLI commands create a process-local voice-call runtime, and runtime creation starts the webhook listener that the Gateway voice-call service already owns. The newer comments usefully narrow the follow-up: the in-process runtime-sharing symbols help one class of `EADDRINUSE`, but they do not provide cross-process CLI delegation to the running Gateway.

## What This Changes

The issue asks for `openclaw voicecall call` and related voice-call CLI operations to avoid starting a second webhook runtime when the Gateway already owns the voice-call listener.

## Best Possible Solution

Keep the issue open and fix the voice-call plugin CLI locally: operational commands should use the existing `voicecall.*` Gateway RPC methods when a Gateway is reachable or explicitly selected, preserve a deliberate local-runtime fallback for standalone use, and emit a clear diagnostic if local fallback collides with the Gateway-owned webhook port.

## Reproduction Assessment

Yes. The high-confidence source path is Gateway service startup binding the voice-call webhook, followed by a separate `openclaw voicecall call` process reaching `ensureRuntime()` -> `createVoiceCallRuntime()` -> `webhookServer.start()` -> `listen(...)` on the same configured bind/port.

## Solution Assessment

Yes. Delegating the CLI's operational commands through the existing plugin-facing Gateway RPC helper is the narrowest maintainable fix; adding a core runtime registry or a plugin-specific core special case would be broader than needed.

## Repair Work Prompt

Fix the voice-call CLI/Gateway runtime ownership bug tracked by https://github.com/openclaw/openclaw/issues/72345. In the voice-call plugin, make operational CLI commands (`call`, `start`, `continue`, `speak`, `dtmf`, `end`, `status`, and live `smoke --yes`) delegate to the existing `voicecall.*` Gateway RPC methods through the public `openclaw/plugin-sdk/gateway-runtime` helpers when a Gateway is reachable or explicitly selected, while preserving an intentional local-runtime fallback for standalone use.
