## Summary

Keep open. Current main has improved the Telegram/ACP ordering surface since the reported 2026.3.8 build, including final-only ACP projection, Telegram preview/final separation, and terminal ACP completion checks. Those mitigations still do not implement the central safety property requested here: preventing or flagging a final Telegram/ACP answer that claims a file/message action completed when no corresponding mutating tool or action actually ran.

## Best Possible Solution

Keep this issue open for maintainer design. The best path is a runtime/harness-boundary verified-action shape: record completed mutating tool/message/file events, expose that ledger to a final-answer gate for ACP/Telegram, and start with an opt-in or narrowly scoped guard. Regression coverage should simulate a Telegram/ACP final reply claiming a file/message send with no matching completed action and assert that the user sees a correction, block, or explicit warning.
