## Summary

Keep open. Current main has improved the Telegram/ACP ordering surface with final-only ACP projection, Telegram preview/final separation, and a terminal ACP done-event guard, but it still does not implement the central safety property requested here: blocking, revising, or warning on a final Telegram/ACP answer that claims a file/message action completed when no corresponding mutating action actually ran.

## Best Possible Solution

Keep this open for maintainer design. The best path is a runtime/harness-boundary verified-action shape: record completed mutating tool/message/file events where OpenClaw can observe them, expose that evidence to a final-answer gate for ACP/Telegram, and start with an opt-in or narrowly scoped warning/revision/block behavior. Regression coverage should simulate a Telegram/ACP final reply claiming a file/message send with no matching completed action and assert that the user sees a correction, block, or explicit warning.

## Risks / Open Questions

- A final-answer verification guard needs careful design to avoid false positives from benign summaries or no-op replies.
- ACP external harnesses may execute native tools outside OpenClaw-owned Pi tool events, so requiring OpenClaw dynamic tool evidence for every claimed action could incorrectly block valid external-harness work unless native/ACP tool receipts are bridged first.
- A completed-action ledger may record sensitive file paths or message targets; any implementation should follow existing transcript, hook, and telemetry redaction boundaries.
