## Summary

Keep open. Current main has useful mitigations for Telegram/ACP streaming, tool-progress visibility, and mutating-tool error warnings, but it still does not implement the central safety property reported here: detecting or preventing a final Telegram/ACP answer that claims a file/message action completed when no matching action actually ran.

## Best Possible Solution

Keep this issue open and implement a generic verified-action path at the agent runtime/harness boundary, with Telegram/ACP coverage. The likely shape is a configurable mode that records mutating file/message/tool attempts and completions, exposes that evidence to before_agent_finalize or an equivalent core finalization gate, and revises, blocks, or surfaces a warning when final text claims completion without matching evidence. Add a focused regression where an ACP/Telegram turn claims a file/message send but no corresponding tool/action completion exists.

## Risks / Open Questions

- A warning-only fix for lastToolError still misses the worst failure mode: no tool/action ran, so there is no tool error to warn about.
- Regex-based acknowledgement detection is helpful for known failures but is not robust enough to serve as an anti-confabulation gate for completed-action claims.
- A Telegram-only preview delay would reduce early leakage but would not address false success in the final assistant answer.
