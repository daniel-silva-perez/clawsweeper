## Summary

Current main still lacks durable normal WebChat send handling across a WebSocket reconnect window. Disconnected submits return before queueing, in-flight RPCs are rejected on close, and reconnect only resumes the chat queue for seq-gap recovery, while the gateway already has an ACK/idempotency contract that a focused UI retry path can use.

## What This Changes

The issue asks for WebChat messages to be queued client-side, ACKed by the gateway, retried after reconnect, and surfaced as failed if delivery cannot complete.

## Best Possible Solution

Add a focused Control UI outbound-send queue for normal `chat.send` that captures the message before transport, preserves a stable idempotency key through ACK, retries transient closed/not-connected failures after reconnect hello, and shows bounded failed-delivery feedback without changing the gateway protocol unless the existing ACK contract proves insufficient.
