## Summary

Keep open. Current main still has the reported WebChat data-loss path: normal sends are not queued before transport, reconnect-window RPCs are rejected, and reconnect only resumes the existing queue for seq-gap recovery, while the gateway already has an ACK/idempotency contract that can support a focused retry fix.

## What This Changes

The issue requests durable WebChat/Control UI message delivery across WebSocket reconnects using a client-side outbound queue, server ACKs, retry, and failed-delivery feedback.

## Best Possible Solution

Add a focused Control UI outbound queue for normal `chat.send` that captures the message before transport, keeps one stable idempotency key until ACK, retries transient closed/not-connected failures after reconnect hello, and surfaces bounded failed-delivery feedback using the existing gateway ACK/dedupe contract.

## Repair Work Prompt

Fix the WebChat/Control UI reconnect-window data-loss bug tracked by openclaw/openclaw#45952. On current main, normal `chat.send` submissions can disappear because `handleSendChat` returns while disconnected and `GatewayBrowserClient` rejects pending/new RPCs during socket close/reconnect. Implement a focused Control UI outbound-send queue for normal WebChat sends: capture the message and attachments before transport, assign and preserve one stable idempotency key through ACK, retry transient `gateway not connected` / socket-closed failures after reconnect hello, and remove queued state only once the gateway ACK/dedupe contract proves the send was accepted (`started`, `in_flight`, or completed `ok` for the same key as appropriate). Add bounded failed-delivery UI feedback after retry exhaustion. Keep the fix scoped to Control UI/WebChat normal sends; do not change channel plugins, broad gateway protocol, or full run-resume/history semantics unless the existing ACK contract is demonstrably insufficient. Preserve attachment payloads carefully so retries do not lose side-store data or leak object URLs. Add/update focused tests for disconnected submit queueing, pending RPC rejection followed by reconnect replay with the same idempotency key, ACK-lost duplicate replay returning `in_flight`/`ok`, retry exhaustion feedback, and attachment retry behavior. Add a user-facing changelog fix entry with `Thanks @Urberliner` and include related credit only if the fix explicitly covers adjacent reports.
