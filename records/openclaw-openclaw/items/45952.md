## Summary

Keep this open. Current main still has the reconnect-window message-loss path: normal sends are not captured before the transport check, pending WebSocket RPCs are rejected on close, and normal reconnect does not replay an unACKed `chat.send`, while the Gateway already has ACK/idempotency behavior that a focused UI retry queue can use.

## What This Changes

The issue requests durable WebChat/Control UI message delivery across WebSocket reconnects through client-side queueing, ACK-based retry, and failed-delivery feedback.

## Best Possible Solution

Add a Control UI outbound-send queue for normal `chat.send` that records the message before transport, keeps a stable idempotency key until ACK/terminal reconciliation, flushes transient not-connected or socket-close failures after reconnect hello, and shows bounded failed-delivery state without changing the Gateway protocol unless the existing ACK contract proves insufficient.

## Reproduction Assessment

Yes. The high-confidence reproduction path is to submit while `host.connected` is false, or let the socket close while `chat.send` is awaiting ACK; current main either returns before queueing or rejects the pending RPC and does not replay it after reconnect.

## Solution Assessment

Yes. The narrow maintainable fix is in the Control UI send lifecycle, preserving and replaying the original outbound request against the existing Gateway ACK/dedupe contract rather than redesigning the protocol first.
