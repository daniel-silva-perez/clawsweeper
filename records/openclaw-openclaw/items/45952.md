## Summary

Current main still has the reported WebChat reconnect data-loss path: normal sends return before queueing when disconnected, browser RPCs are rejected and flushed on socket close, and reconnect only resumes the existing chat queue for seq-gap recovery. The server already has ACK and idempotency semantics that make this a focused Control UI reliability fix rather than a stale or already-implemented issue.

## What This Changes

The issue requests reliable WebChat message delivery across WebSocket reconnect windows using client-side queueing, ACK tracking, retry, and failed-delivery feedback.

## Best Possible Solution

Add a focused Control UI outbound-send queue for normal `chat.send` that captures the message before transport, preserves a stable idempotency key until ACK, retries transient closed/not-connected failures after reconnect hello, and shows bounded failed-delivery feedback while relying on the existing gateway ACK/dedupe contract unless it proves insufficient.
