## Summary

Keep open: current main still lacks a durable WebChat outbound queue/ACK replay path for normal `chat.send` during a WebSocket reconnect window. The session-switch queue fix and open queue-persistence PR are adjacent, but they do not prove this disconnected or in-flight send case is fixed.

## What This Changes

The issue requests client-side queuing, server ACK handling, reconnect retry, and failed-delivery UI feedback for WebChat messages sent during WebSocket reconnects.

## Best Possible Solution

Implement a focused Control UI outbound-send queue for normal `chat.send`: capture the outbound item before transport, keep its idempotency key stable until ACK, retry transient `gateway closed` / `gateway not connected` failures after reconnect hello, and show a bounded failed-delivery state if delivery cannot be confirmed. Use the existing `chat.send` ACK/idempotency contract before adding any new protocol surface.
