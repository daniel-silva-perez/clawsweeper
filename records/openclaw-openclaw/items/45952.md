## Summary

Keep open. Current main still has the reported WebChat reconnect data-loss path: normal sends are not durably queued before transport, pending browser RPCs are rejected when the socket closes, and the reconnect path only resumes the chat queue for seq-gap recovery even though the gateway already supports chat.send ACK/idempotency semantics.

## What This Changes

Request client-side queuing and ACK-based retry for WebChat messages submitted during WebSocket reconnect windows so normal chat sends are not silently lost.

## Best Possible Solution

Add a focused Control UI outbound-send queue for normal `chat.send` that captures the message before transport, preserves one stable idempotency key until ACK, retries transient closed/not-connected failures after reconnect hello, and surfaces bounded failed-delivery feedback using the existing gateway ACK/dedupe contract.
