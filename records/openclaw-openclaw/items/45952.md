## Summary

Keep open. Current main still lacks a durable Control UI/WebChat outbound queue for normal `chat.send` during a WebSocket reconnect window: disconnected submits return before queueing, in-flight RPCs are rejected on socket close, and failed sends are converted to UI errors instead of being replayed with a stable idempotency key. The recently merged session-switch queue preservation is adjacent but does not cover this reconnect loss case.

## What This Changes

The issue requests client-side queueing, ACK-based retention, reconnect retry, and failed-delivery UI feedback so WebChat messages are not silently lost during WebSocket reconnects.

## Best Possible Solution

Add a focused Control UI outbound-send queue for normal `chat.send` that captures the message before transport, preserves the idempotency key until ACK, retries transient closed/not-connected failures after reconnect hello, and surfaces bounded failed-delivery feedback without changing the gateway protocol unless the existing ACK contract proves insufficient.
