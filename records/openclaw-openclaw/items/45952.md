## Summary

Current main still has the reported WebChat reconnect data-loss path: disconnected normal sends return before queueing, in-flight browser RPCs are rejected on socket close, and reconnect only resumes the busy-run chat queue for seq-gap recovery. The gateway already exposes the ACK/idempotency behavior needed for a focused UI retry queue, so this should remain open as a valid narrow reliability bug.

## What This Changes

Request client-side queueing and ACK-based retry for WebChat messages submitted during WebSocket reconnect windows so normal chat sends are not silently lost.

## Best Possible Solution

Add a focused Control UI outbound-send queue for normal `chat.send` that captures the message before transport, preserves one stable idempotency key until ACK, retries transient closed/not-connected failures after reconnect hello, and surfaces bounded failed-delivery feedback using the existing gateway ACK/dedupe contract.
