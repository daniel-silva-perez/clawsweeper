## Summary

Keep this open. Current main still has the reconnect-window delivery gap: normal WebChat sends are not retained before transport failure, pending browser RPCs are rejected on socket close, and ordinary reconnect does not replay an unACKed `chat.send`, even though the Gateway already supports same-key ACK/idempotency behavior.

## What This Changes

Add client-side queuing, ACK tracking, reconnect retry, and failed-delivery feedback for WebChat messages sent across WebSocket reconnects.

## Best Possible Solution

Add a focused Control UI outbound-send queue for normal `chat.send` that records the request and stable idempotency key before transport, retries transient closed/not-connected failures after reconnect hello, reconciles against existing Gateway ACK/dedupe responses, and shows bounded failed-delivery state without redesigning the Gateway protocol unless that contract proves insufficient.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to submit while the WebSocket closes before the `chat.send` ACK, or call the send path while `host.connected` is false; current main either rejects the pending RPC without replay or returns before queueing.

## Solution Assessment

Yes. The best fix is a narrow Control UI send/reconnect lifecycle change that uses the existing Gateway ACK/idempotency contract; a protocol redesign is not necessary for the first repair unless implementation proves the current ACK semantics insufficient.
