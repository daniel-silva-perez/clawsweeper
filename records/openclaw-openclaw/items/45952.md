## Summary

Current main still lacks a durable WebChat outbound queue/ACK replay path for normal `chat.send` messages submitted during a WebSocket reconnect window. The related session-switch queue fix and open #73154 queue-persistence PR are adjacent, but neither proves this exact in-flight/disconnected send case is fixed.

## What This Changes

Requests a WebChat/Control UI client-side outbound queue with ACK-based replay so messages sent during WebSocket reconnects are not lost.

## Best Possible Solution

Keep this issue open and implement a focused Control UI outbound-send queue for normal `chat.send`: capture the outbound item before transmission, keep a stable idempotency key until the `chat.send` ACK, retry transient `gateway closed` / `gateway not connected` failures after reconnect hello, and surface a bounded failed-delivery state if delivery cannot be confirmed. Coordinate with #51549, #73154, and the merged #73621 queue work; avoid a new protocol surface unless the existing `chat.send` ACK/idempotency contract proves insufficient.

## Repair Work Prompt

Fix WebChat/Control UI message loss during WebSocket reconnect for this issue. On current main, normal `chat.send` messages can be dropped when the socket is closed or reconnecting because `handleSendChat` returns before queueing while disconnected, `sendChatMessage` clears failed sends instead of retaining them, and `GatewayBrowserClient` rejects/flushes pending RPCs on close. Implement a focused Control UI outbound send queue for normal `chat.send` submissions: enqueue before attempting the RPC, keep one stable idempotency key until the server ACKs, treat the existing `chat.send` response (`started`, `in_flight`, `ok`) as the ACK surface, replay transient `gateway closed` / `gateway not connected` failures after reconnect hello, and show a bounded failed-delivery state if retries cannot confirm delivery. Preserve attachment payload handling and do not create duplicate agent runs. Coordinate with #51549, #73154, and the merged #73621 session-switch queue work; do not add a new gateway protocol unless the existing ACK/idempotency contract is proven insufficient. Likely files are `ui/src/ui/app-chat.ts`, `ui/src/ui/controllers/chat.ts`, `ui/src/ui/app-gateway.ts`, `ui/src/ui/gateway.ts`, and their tests. Add a single-line user-facing changelog fix entry crediting @Urberliner, and credit any source PR author if code from #73154 is used. Do not touch unrelated gateway transport policy or broad plugin surfaces.
