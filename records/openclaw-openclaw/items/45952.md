## Summary

Current main still does not provide a durable WebChat outbound queue/ACK replay path for a normal `chat.send` submitted during a WebSocket reconnect. The merged #73621 work preserves the existing busy-run queue across session switches, and open PR #73154 targets #51549 queue persistence, but neither proves this reconnect-window unacked-send case is fixed.

## Best Possible Solution

Keep this issue open and coordinate it with #51549, #73621, and PR #73154. The best fix is a focused Control UI outbound send queue for normal `chat.send` submissions: treat the existing `chat.send` response as the ACK, keep the same idempotency key until ACK, replay transient `gateway closed` / `gateway not connected` sends after reconnect hello, and surface a bounded failed-delivery state if retries cannot confirm delivery. Avoid a new protocol surface unless the current RPC ACK proves insufficient.
