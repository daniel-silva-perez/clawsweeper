## Summary

Keep this open. Current main still has the WebChat reconnect-window data-loss path: normal sends are not captured before transport when disconnected, in-flight WebSocket RPCs are rejected on close, and normal reconnect does not replay an unACKed `chat.send`, even though the gateway already supports ACK/idempotency semantics suitable for a focused UI retry queue.

## What This Changes

The issue requests reliable WebChat message delivery across WebSocket reconnects by adding client-side queueing, ACK tracking, retry on reconnect, and failed-delivery feedback.

## Best Possible Solution

Add a focused Control UI outbound-send queue for normal `chat.send` that captures the message before transport, preserves a stable idempotency key through ACK, retries transient closed/not-connected failures after reconnect hello, and shows bounded failed-delivery feedback while relying on the existing gateway ACK/dedupe contract unless it proves insufficient.

## Reproduction Assessment

Yes: the high-confidence code path is to submit while `host.connected` is false, or let the socket close while `chat.send` is awaiting ACK. Current main either returns before queueing or rejects the pending RPC and converts it to a visible error without any reconnect replay.

## Solution Assessment

Yes: the best fix is a narrow Control UI reliability change, not a gateway protocol redesign. The server already has ACK and same-key duplicate behavior, so the UI should preserve and replay the original outbound request state.
