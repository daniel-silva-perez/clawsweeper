## Summary

Keep open: current main still lacks a normal WebChat outbound send queue that survives a WebSocket reconnect window, even though the gateway already exposes ACK/idempotency behavior that the UI can use for safe retry.

## What This Changes

The issue requests client-side WebChat queuing, ACK handling, reconnect retry, and visible failed-delivery feedback so messages sent during WebSocket reconnects are not lost.

## Best Possible Solution

Add a focused Control UI outbound-send path for normal `chat.send` that records the message before transport, preserves a stable idempotency key through ACK, retries transient closed/not-connected failures after reconnect hello, and surfaces bounded failed-delivery feedback while using the existing gateway ACK/dedupe contract.
