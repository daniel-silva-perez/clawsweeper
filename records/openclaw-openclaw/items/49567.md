## Summary

Keep open. Current main still does not implement per-binding `tools.allow`/`tools.deny` for route or ACP bindings: the schemas and exported binding types have no `tools` field, route resolution returns only routing/session metadata, ACP configured-binding records do not preserve tool policy, and the tool-policy pipeline has no binding-scoped step. Existing Telegram group `tools` config can cover some group use cases, but it is not the requested route/ACP binding surface and does not make the example config valid.

## Best Possible Solution

Add an additive core binding policy surface: extend route and ACP binding schemas/types with `tools.allow`/`tools.deny` using existing policy semantics, carry the matched binding policy through route resolution and configured ACP binding materialization, apply it in the shared tool-policy pipeline after agent/group policy and before sandbox/subagent restrictions, then document and test validation plus runtime enforcement for a Telegram group binding that denies `sessions_spawn` while DMs remain unaffected.

## Risks / Open Questions

- Closing now would leave the documented example config invalid because strict route/ACP binding schemas still reject `tools` as an additional property.
- Telegram group-level `channels.telegram.groups.<id>.tools.deny` may solve part of the reporter's immediate group-chat use case, but it does not provide the requested binding-scoped route/ACP contract or ACP runtime propagation.
- A schema-only implementation like the closed #49593 would make config validation pass without enforcing tool filtering unless route/ACP resolution and the tool-policy pipeline are wired at the same time.
