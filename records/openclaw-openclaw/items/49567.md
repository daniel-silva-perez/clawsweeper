## Summary

Keep open. Current main still does not implement per-binding `tools.allow`/`tools.deny` for route or ACP bindings. The route and ACP binding schemas remain strict without a `tools` field, the exported binding types do not carry a binding-scoped tool policy, route resolution returns only agent/session routing data, ACP configured-binding materialization carries only ACP runtime fields, and the tool-policy pipeline still applies profile/global/provider/agent/group/sandbox/subagent policy with no binding-level step. The prior Codex review comment remains consistent with current main; the closed PR #49593 was schema-only and unmerged, and the open PR #58434 is a related per-request OpenResponses feature rather than this config/runtime surface.

## Best Possible Solution

Add an additive core binding policy surface: extend route and ACP binding schemas/types with `tools.allow`/`tools.deny` using the existing policy semantics, carry the matched binding policy through route and ACP binding resolution, apply it in the tool-policy pipeline after agent/group policy and before sandbox/subagent restrictions, and document/test validation plus runtime enforcement for a Telegram group binding that denies `sessions_spawn` while DMs remain unaffected.
