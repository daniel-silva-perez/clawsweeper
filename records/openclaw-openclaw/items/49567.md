## Summary

Keep open. Current main still lacks binding-level tool policy support for route or ACP bindings: validation rejects `bindings[].tools`, the public binding and resolved-route types cannot carry it, ACP configured-binding records do not preserve it, and the tool-policy pipeline has no binding-scoped step. The related closed PR #49593 supplied useful schema-only context but did not merge, and open PR #58434 targets per-request OpenResponses `tool_deny`, not route/ACP binding policy.

## What This Changes

The issue requests `tools.allow`/`tools.deny` support on route and ACP bindings so tool availability can be restricted per matched channel/group binding.

## Best Possible Solution

Add an additive binding policy surface end to end: schema/types for route and ACP bindings, route resolution and ACP configured-binding storage that preserve the matched policy, enforcement in the shared tool-policy pipeline after agent/group policy and before sandbox/subagent restriction, plus docs and regression coverage for a Telegram group binding that denies `sessions_spawn` while DMs remain unaffected.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: `bindings[].tools` is absent from strict route/ACP schemas, and downstream types and route/ACP materialization cannot carry a policy even if validation accepted it.

## Solution Assessment

No schema-only change is enough. The maintainable fix needs validation, typed config, route/ACP propagation, policy-pipeline enforcement, docs, and tests so accepted config actually changes effective tools.
