## Summary

Keep this open. Current main has meaningful session persistence and restart-recovery hardening, but this report describes a Feishu/local-gateway disconnect where a specific Feishu message id was absent from all transcripts; I did not find a Feishu reconnect catch-up/backfill path or user-facing session-switch notification that would cover messages missed while the gateway/network was down. Related #54459 was closed as implemented for restart transcript loss, and #38836 remains open for Feishu restart notifications, but this issue is not safe to auto-close as either fully implemented or a clean duplicate.

## What This Changes

The issue asks OpenClaw to preserve or recover Feishu DM session history across a backend session disconnect/recreation, or notify the user when context has switched.

## Best Possible Solution

Define the supported Feishu behavior for gateway/network downtime. If OpenClaw should recover missed messages, add a durable Feishu reconnect/backfill or generic inbound WAL/recovery path; if it cannot recover provider-undelivered events, document that limit and add same-channel notification for interrupted or switched sessions without weakening the existing session store, transcript lock, and restart recovery paths.

## Reproduction Assessment

Unclear. The issue has a concrete live timeline and missing message id, but I did not reproduce a laptop-sleep Feishu WebSocket outage against current main; current tests cover delivered events and restart recovery, not provider-side redelivery after gateway downtime.

## Solution Assessment

Unclear. The current restart/session hardening is necessary but not sufficient evidence for this Feishu offline gap; the maintainable fix likely needs maintainer choice between Feishu backfill, generic durable inbound recovery, same-channel notification, or documented unsupported behavior.
