## Summary

Keep this open. Current main has real session persistence and restart-recovery hardening, but this report is specifically about a Feishu/local-gateway disconnect where a concrete Feishu message id was absent from every transcript; I did not find a Feishu reconnect backfill path or user-visible session-switch notice that would cover messages missed while the gateway or network was down. Related #54459 covers restart transcript loss, while #38836 remains open for Feishu-visible interruption notification.

## What This Changes

The issue asks OpenClaw to preserve or recover Feishu DM session history across backend session disconnect/recreation, or notify the user when context has switched.

## Best Possible Solution

Define the supported behavior for Feishu downtime. If OpenClaw should recover missed messages, add a durable Feishu reconnect/backfill or generic inbound recovery path; if provider-undelivered events cannot be recovered, document that boundary and add a same-channel interruption/session-switch notice for recoverable restarts.

## Reproduction Assessment

Unclear. The issue has a concrete live timeline and message id, but I did not reproduce a laptop-sleep Feishu WebSocket outage against current main; current tests cover delivered events and restart recovery, not provider-side redelivery after gateway downtime.

## Solution Assessment

Unclear. The current restart/session hardening is necessary but not sufficient for this Feishu offline gap; the best fix needs maintainer scoping between Feishu backfill, generic inbound durability, same-channel notification, or documenting unsupported offline delivery semantics.
