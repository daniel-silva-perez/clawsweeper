## Summary

Keep open. Current main persists session store/transcripts once the agent dispatch path runs, and Feishu does pass message IDs into the agent context, but I did not find a current-main fix for the reported failure mode where a Feishu message is accepted around a disconnect and never reaches any session JSONL. The Feishu handler still records the message ID in persistent dedupe before the later agent dispatch/transcript path, which can plausibly match the reported “message ID absent from all session files” symptom after a crash, sleep, or interrupted local gateway run.

## Best Possible Solution

Keep this issue open. The best fix is likely in the Feishu inbound pipeline plus shared channel dispatch semantics: do not permanently record a Feishu message ID as processed until dispatch/session transcript creation has succeeded, or first durably queue the inbound event so it can be replayed after gateway sleep/restart. Add a regression test for dispatch failure or process interruption between Feishu receipt and transcript persistence, and document any unavoidable local-gateway offline delivery limits.
