## Summary

Keep this issue open. Current main still has a boundary-unsafe default UTF-8 decode path in the shared child-output decoder, and open PR #51303 explicitly tracks this issue with closing syntax, so the right next step is review or porting that fix rather than closing early.

## What This Changes

Fix streaming Chinese/CJK character corruption where split UTF-8 chunks can decode into U+FFFD replacement characters in WebChat/group replies.

## Best Possible Solution

Keep this issue open and use PR #51303 as the implementation lane. The best fix is to make createWindowsOutputDecoder boundary-safe for default UTF-8 on non-Windows and UTF-8 Windows configurations, preserve the Windows legacy-codepage fallback, add focused split-CJK tests for the default path and child adapter wiring, and then close this issue after the fix is merged or intentionally superseded.
