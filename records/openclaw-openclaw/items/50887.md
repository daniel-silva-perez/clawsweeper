## Summary

Keep open. Current main still has a default/non-Windows UTF-8 stream decode path that decodes each chunk independently, and the issue is already paired with open PR #51303, so it should remain open until that PR lands, is revised, or is superseded.

## What This Changes

The issue asks OpenClaw to preserve Chinese/CJK characters during streamed WebChat/group-chat responses instead of replacing split multi-byte UTF-8 characters with U+FFFD.

## Best Possible Solution

Resolve this through PR #51303 or a narrow replacement that makes default UTF-8 output decoding boundary-safe on non-Windows, preserves the Windows legacy-codepage fallback, adds default-path split-CJK coverage plus child-adapter wiring coverage, and separately checks JSONL/socket or truncation paths needed for the reported WebChat route.
