## Summary

Keep open: current main still contains a chunk-boundary-unsafe UTF-8 decode path relevant to streamed child/process output, and the issue is paired with open PR #51303, so it should not be closed before that PR lands or is superseded.

## What This Changes

The issue requests boundary-safe streaming assembly so longer Chinese responses in WebChat/group chat do not replace random multibyte characters with U+FFFD.

## Best Possible Solution

Resolve through PR #51303 or a narrow replacement that makes the default UTF-8 output decoder boundary-safe, preserves the Windows legacy-codepage fallback, adds split-CJK coverage for the default decoder and child adapter wiring, and separately verifies any JSONL/socket or truncation surfaces needed for the reported WebChat route.
