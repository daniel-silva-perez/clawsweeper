## Summary

Keep open. Current main and the latest release still contain a default UTF-8 stream decode path that decodes each chunk independently, and open PR #51303 is already paired with this issue using closing syntax, so this should stay open until that PR lands or is superseded.

## What This Changes

The issue reports U+FFFD replacement characters appearing in longer Chinese WebChat group-chat responses, suspected from UTF-8 byte-boundary unsafe streaming assembly.

## Best Possible Solution

Resolve through PR #51303 or a narrow superseding fix that makes default UTF-8 output decoding boundary-safe, preserves the Windows legacy-codepage fallback, adds split-CJK coverage for both the decoder and child-adapter wiring, and verifies any JSONL/socket or truncation surfaces needed for the reported WebChat path.
