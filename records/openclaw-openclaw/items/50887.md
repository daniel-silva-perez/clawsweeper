## Summary

Keep this issue open. Current main still has a boundary-unsafe UTF-8 decode path in the shared child-output decoder used by streamed child stdout/stderr, the latest release still contains that behavior, and open PR #51303 explicitly tracks this issue with closing syntax, so the right maintainer action is to review, land, or supersede that implementation rather than close early.

## What This Changes

The issue reports Chinese/CJK characters being replaced by U+FFFD during streamed WebChat/group-chat replies and asks OpenClaw to preserve multibyte UTF-8 characters across chunk boundaries.

## Best Possible Solution

Keep the issue open and use PR #51303, or a narrow replacement, to make createWindowsOutputDecoder boundary-safe for default UTF-8 on non-Windows and UTF-8 Windows configurations, preserve the Windows legacy-codepage fallback, add focused split-CJK tests for the default decoder path and child adapter wiring, then close after the fix lands or is intentionally superseded.
