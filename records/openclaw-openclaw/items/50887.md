## Summary

Keep this issue open. Current main still routes child stdout/stderr through `createWindowsOutputDecoder`, and that decoder still falls back to per-chunk `buffer.toString("utf8")` on the default non-Windows/UTF-8 path. Open PR #51303 explicitly uses closing syntax for this issue, so the right path is to review or port that fix into the current shared decoder layer rather than close the issue first.

## Best Possible Solution

Keep this issue open and use PR #51303 as the implementation lane. The best fix is to make `createWindowsOutputDecoder` boundary-safe for the default UTF-8 path on non-Windows and UTF-8 Windows configurations, preserve the Windows legacy-codepage fallback, and add focused tests for split UTF-8 chunks plus child adapter wiring.
