## Summary

Keep this open. Current main still has a boundary-unsafe UTF-8 decode path relevant to the reported CJK corruption, and the issue is explicitly paired with open PR #51303, so repository policy points to reviewing, landing, or superseding that implementation before closure.

## What This Changes

Fix streaming assembly so Chinese/CJK UTF-8 characters are not replaced with U+FFFD in WebChat/group chat responses when multibyte sequences cross chunk boundaries.

## Best Possible Solution

Keep the issue open and resolve it through PR #51303 or a narrow replacement that makes createWindowsOutputDecoder boundary-safe for default UTF-8/non-Windows output, preserves the Windows legacy-codepage fallback, adds focused split-CJK tests for the default decoder path and child adapter wiring, then closes after the fix lands or is intentionally superseded.
