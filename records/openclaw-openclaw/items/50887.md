## Summary

Keep this issue open. The linked fix PR #51303 is still open and uses GitHub closing syntax for this issue, and current main still has the default UTF-8 split-chunk failure mode in the shared decoder path used by the child adapter on Linux/non-legacy decoding.

## Best Possible Solution

Keep this issue open and use #51303 as the active implementation thread, but ask for a current-main port: make createWindowsOutputDecoder() boundary-safe for the default non-Windows/UTF-8 path, preserve the Windows legacy-codepage fallback, and add focused regression tests for split Chinese UTF-8 chunks. After that PR or an equivalent fix merges, close this issue as implemented by the merge.
