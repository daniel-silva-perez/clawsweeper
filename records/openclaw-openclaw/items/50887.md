## Summary

Keep open. Current main still has a boundary-unsafe default UTF-8 decode path for streamed child/process output, the latest release keeps the same fallback, and open PR #51303 is already paired with this issue using closing syntax, so the issue should remain open until that PR lands or is superseded.

## What This Changes

The issue reports that streamed Chinese text in WebChat/group chat can contain random U+FFFD replacement characters, likely from decoding split UTF-8 chunks independently.

## Best Possible Solution

Resolve through PR #51303 or a narrow replacement that makes the default UTF-8 output decoder boundary-safe, preserves the Windows legacy-codepage fallback, adds split-CJK coverage for both the default decoder and child adapter wiring, and audits the remaining JSONL/socket or truncation surfaces needed for the reported WebChat path.
