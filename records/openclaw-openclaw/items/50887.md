## Summary

Keep open. Current main still has boundary-unsafe UTF-8 decoding on the default/non-Windows streamed child-output path, the latest shipped tag still contains the same fallback, and open PR #51303 is already paired with this issue via closing syntax.

## What This Changes

The issue asks OpenClaw to preserve Chinese/CJK characters during streamed WebChat group-chat responses instead of replacing split multi-byte UTF-8 characters with U+FFFD.

## Best Possible Solution

Resolve this through PR #51303 or a narrow replacement that makes the shared default UTF-8 output decoder boundary-safe on non-Windows, preserves the Windows legacy-codepage fallback, adds split-CJK coverage for the default decoder and child-adapter wiring, and verifies any JSONL/socket or truncation surface needed by the reported WebChat route.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partially yes: the byte-boundary corruption class reproduces with a read-only Node split-CJK probe and matches the current default decoder code, but the full cursor2api/WebChat group-chat path was not live-reproduced in this review.

## Solution Assessment

Is this the best way to solve the issue? Not yet: early closure is not appropriate; the best path is to review, revise, land, or supersede PR #51303 with a shared decoder fix and focused regression coverage rather than queueing a duplicate repair job.
