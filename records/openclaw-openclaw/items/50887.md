## Summary

Keep open. Current main still has a boundary-unsafe default UTF-8 decode path for streamed child/process output, the latest release tag retains the same fallback, and open PR #51303 is already paired with this issue using closing syntax, so the issue should remain open until that PR lands or is superseded.

## What This Changes

The issue reports random U+FFFD replacement characters in longer Chinese WebChat group-chat replies, likely from splitting multi-byte UTF-8 characters across streaming chunks.

## Best Possible Solution

Resolve through PR #51303 or a narrow replacement that makes default UTF-8 chunk decoding boundary-safe, preserves the legacy Windows codepage fallback, adds split-CJK regression coverage for the default decoder and child adapter wiring, and checks any JSONL/socket or truncation surface needed for the reported WebChat path.
