## Summary

Keep this PR open, but do not merge it as-is. Current main still has the hard `sessions_send` wait-timeout behavior, so the contribution addresses a real gap, but the patch conflates soft wait-window expiry with terminal aborted timeouts and can route stale assistant history through delayed announce delivery.

## What This Changes

The PR changes `sessions_send` so an initial wait timeout returns accepted/pending delivery and starts A2A announce delivery, adds a bounded-follow-up comment in the A2A flow, and adds delayed-timeout regression tests.

## Best Possible Solution

Revise the fix so `sessions_send` preserves hard timeout/failure semantics for terminal aborted runs, treats only proven in-flight wait-window expiry as accepted pending delivery, passes baseline protection into delayed A2A reply reads, and covers delayed reply, no-new-reply, second-wait-timeout, and terminal-timeout cases in focused tests.

## Reproduction Assessment

Yes. The current-main path is reproducible with `sessions_send` using `timeoutSeconds > 0` where the target `agent` call is accepted and the first `agent.wait` returns `status: "timeout"`; the tool returns a hard timeout at `src/agents/tools/sessions-send-tool.ts:379`. The PR regressions can be reproduced with mocks for a terminal aborted timeout and for a timed-out run that finishes without a new post-baseline assistant reply.

## Solution Assessment

No. The direction is reasonable for soft wait-window expiry, but the current patch is not the safest solution because it converts every timeout to accepted and drops the baseline used to avoid stale replies. The safer fix is to classify timeout source explicitly and thread the baseline into the delayed announce path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Do not mark completed timeouts as accepted:** `src/agents/tools/sessions-send-tool.ts:381-382`
  - body: `agent.wait` returns `status: "timeout"` both when the caller's wait window expires and when the run has already ended as an aborted timeout. This branch converts every timeout into accepted pending delivery, so a terminal timed-out run is reported as pending success and can leave callers waiting for an announce that will never arrive.
  - confidence: 0.9
- **[P1] Pass the baseline into delayed announce reads:** `src/agents/tools/sessions-send-tool.ts:378`
  - body: The normal wait path captures `baselineReply` so an old assistant message is not reused as the new run's reply, but the new timeout branch calls `startA2AFlow(undefined, runId)`. The delayed A2A flow then reads the latest assistant reply without that baseline, so a timed-out run that finishes without new assistant text can announce stale history to the requester.
  - confidence: 0.89
