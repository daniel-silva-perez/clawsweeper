## Summary

Keep open. Current main has useful mitigations for some tool-loop cases, but it does not yet solve this issue's central report: deterministic tool argument failures can still repeat, generic same-tool retries are opt-in/warn-only, and block-reply delivery intentionally allows identical text again at a new assistant message boundary.

## Best Possible Solution

Keep this issue open and implement a default safety path for deterministic tool-call validation failures: classify missing/invalid required parameters as non-retryable, record schema-validation failures even when execution does not reach the tool body, add a low per-turn cap for repeated identical failures, and suppress or coalesce user-visible commentary while the assistant is about to retry a failed tool call. Add regression coverage for `exec({})` or missing `command` under a chat-channel block-streaming path.
