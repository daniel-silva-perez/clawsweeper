## Summary

Keep open. Current main still has Discord voice channels on the text-mediated path, while the realtime Gemini Live/OpenAI provider bridge exists only in adjacent Voice Call and Google Meet surfaces; the linked #73699 architecture question should shape this before implementation.

## What This Changes

Add a Discord voice configuration and runtime path that can select Gemini Live or another realtime audio provider instead of only using STT, a text agent turn, and TTS playback.

## Best Possible Solution

Define an additive Discord plugin realtime mode that reuses the shared realtime voice provider bridge, keeps `channels.discord.voice.model` as the text-pipeline LLM override, and documents session, transcript, permission, and tool-consult behavior alongside #73699 before code lands.

## Reproduction Assessment

Not applicable as a feature request. Current-main verification is high confidence: schema, runtime, tests/docs searches, and docs all show Discord voice still follows STT -> text agent -> TTS rather than a realtime provider session.

## Solution Assessment

Unclear. A separate `channels.discord.voice.realtime` branch is plausible, but the maintainable solution depends on the #73699 voice-as-I/O decision and on how native-audio transcripts/tools map into Discord sessions.
