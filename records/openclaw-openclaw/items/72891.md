## Summary

Keep open. Current main still routes Discord voice channels through the text-mediated pipeline the issue describes, while Gemini Live/realtime voice is available only through adjacent surfaces such as Voice Call, Google Meet, and Talk. The linked #73699 discussion is not a duplicate; it raises an architecture decision that should shape this feature before implementation.

## What This Changes

Request a Discord voice-channel configuration path that can use Gemini Live or another realtime audio provider instead of only selecting the text LLM used after STT.

## Best Possible Solution

Define an additive Discord plugin realtime mode that reuses the shared realtime voice provider bridge, keeps `channels.discord.voice.model` as the text-pipeline LLM override, and explicitly documents session, transcript, permission, and tool-consult behavior alongside #73699 before implementation.

## Reproduction Assessment

Not applicable as a feature request. Current-main verification is high confidence: the schema, runtime, tests, and docs all show Discord voice still uses STT, a text agent turn, and TTS rather than a realtime provider session.

## Solution Assessment

Unclear. A separate `channels.discord.voice.realtime` configuration is plausible, but the best implementation depends on the maintainer decision for #73699 and how native-audio transcripts/tools should map into Discord text sessions.
