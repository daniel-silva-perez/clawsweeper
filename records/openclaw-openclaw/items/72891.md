## Summary

Keep open. Current main still implements Discord voice channels as a WAV capture, STT, text agent turn, and TTS playback pipeline; realtime audio provider support exists in adjacent Voice Call, Google Meet, and shared realtime-voice surfaces, but no Discord voice realtime schema, config branch, runtime bridge, or docs path has landed. The linked #73699 discussion is an open architecture dependency for Discord session/transcript behavior, not a duplicate or shipped fix.

## What This Changes

Request to add a Discord voice realtime mode or config path that can select Gemini Live or another realtime audio provider instead of only using `channels.discord.voice.model` for text responses.

## Best Possible Solution

Define an additive Discord voice realtime mode that reuses the shared realtime voice provider bridge, keeps `channels.discord.voice.model` as the text-pipeline LLM override, and specifies how Discord sessions, transcripts, delivery, permissions, and realtime tool/consult calls interact with #73699 before updating schema, runtime, docs, and focused tests.

## Reproduction Assessment

Not applicable as a feature request. Static current-main verification is high confidence: the strict Discord voice schema rejects a realtime branch, and the runtime/docs show WAV capture -> STT -> text agent turn -> TTS rather than a native realtime provider session.

## Solution Assessment

Unclear. A separate Discord voice realtime config path is plausible and matches the report, but the best maintainable fix needs maintainer direction for voice-as-I/O, transcript/delivery behavior, routing, permissions, and tool policy before implementation.
