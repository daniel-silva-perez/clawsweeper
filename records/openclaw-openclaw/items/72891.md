## Summary

Keep open. Current main still implements Discord voice as the text-mediated pipeline described in the request, while the native realtime provider bridge exists only in adjacent Voice Call, Google Meet, and Talk surfaces; the related #73699 discussion is an architecture dependency rather than a duplicate or shipped fix.

## What This Changes

Request adding a Discord voice configuration and runtime path that can select a native realtime audio provider such as Gemini Live instead of only using the current STT -> text agent -> TTS pipeline.

## Best Possible Solution

Define an additive Discord plugin realtime mode that reuses the shared realtime voice provider bridge, keeps `channels.discord.voice.model` as the existing text-pipeline LLM override, and documents/tests how Discord voice sessions, transcripts, permissions, delivery, and consult tools behave alongside the #73699 voice-as-I/O decision.

## Reproduction Assessment

Not applicable as a feature request. Current-main verification is high confidence: schema, runtime, docs, and targeted search all show Discord voice still uses STT, a text agent turn, and TTS rather than a native realtime provider session.

## Solution Assessment

Unclear. A separate `channels.discord.voice.realtime` configuration is plausible and avoids overloading `voice.model`, but the best implementation depends on maintainer direction for #73699 and the Discord-specific transcript/tool/permission policy.
