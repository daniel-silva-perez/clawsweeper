## Summary

Keep open. Current main still implements Discord voice channels as WAV capture, STT, a text agent turn, and TTS playback; Gemini Live/realtime provider support exists in adjacent Voice Call, Google Meet, Talk, and provider-plugin surfaces, but there is no Discord voice realtime config, schema branch, runtime bridge, or docs path. The related #73699 discussion is an open architecture dependency for Discord voice session/transcript behavior, not a duplicate or shipped fix.

## What This Changes

Add a Discord voice configuration and runtime path that can use Gemini Live or another realtime audio provider instead of only the current STT-to-text-agent-to-TTS pipeline.

## Best Possible Solution

Define an additive Discord voice realtime mode that reuses the shared realtime voice provider bridge without overloading channels.discord.voice.model, then specify how Discord voice sessions, transcripts, delivery, permissions, and consult/tool calls interact with #73699 before updating schema, types, runtime, docs, and focused tests.

## Reproduction Assessment

Not applicable as a feature request. Static current-main verification is high confidence: the strict schema rejects a Discord realtime branch, and the runtime/docs show Discord voice still uses STT, a text agent turn, and TTS rather than a native realtime provider session.

## Solution Assessment

Unclear. A separate channels.discord.voice.realtime or equivalent config path is plausible, but the best solution needs maintainer direction for voice-as-I/O, transcript behavior, routing, permissions, and tool policy before implementation.
