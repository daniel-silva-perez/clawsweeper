## Summary

Keep open. Current main still routes Discord voice through discrete capture, batch STT, a text agent turn, and TTS playback, while Gemini Live/realtime provider support is available only through adjacent realtime voice surfaces. The linked #73699 discussion is relevant architecture context, but it is not a duplicate or shipped fix for selecting an end-to-end realtime provider in Discord voice channels.

## What This Changes

Add a Discord voice configuration path for end-to-end realtime audio providers such as Gemini Live instead of only the current STT to text LLM to TTS pipeline.

## Best Possible Solution

Define an additive Discord voice realtime mode, separate from channels.discord.voice.model, that reuses the shared realtime voice provider bridge while explicitly specifying Discord session routing, transcript delivery, text-channel interaction, permissions, and realtime consult/tool policy in coordination with #73699.

## Reproduction Assessment

Not applicable as a feature request. Static current-main verification is high confidence: schema, config types, docs, runtime, tests, and targeted search all show Discord voice still uses STT, a text agent turn, and TTS rather than a native realtime provider session.

## Solution Assessment

Unclear. A separate channels.discord.voice.realtime-style config path is plausible, but the best implementation depends on maintainer decisions for #73699, Discord transcript behavior, session routing, and safe realtime tool access.
