## Summary

Keep open. Current main still implements Discord voice as the reported STT -> text agent turn -> TTS playback pipeline, while the Gemini Live/realtime bridge is present only on adjacent Voice Call, Google Meet, Talk, and VoiceClaw surfaces. The related #73699 discussion is an open architecture dependency for Discord voice session/transcript behavior, not a shipped fix or duplicate that closes this request.

## What This Changes

Add a Discord voice configuration and runtime path that can select a realtime audio provider such as Google Gemini Live instead of only choosing the text-response model used after STT.

## Best Possible Solution

Define an additive Discord voice realtime mode that reuses the shared realtime voice provider bridge without overloading `channels.discord.voice.model`, and specify how Discord voice sessions, transcripts, delivery, permissions, and consult/tool calls behave relative to #73699 before implementing schema, runtime, docs, and tests.

## Reproduction Assessment

Not applicable as a feature request. Current-main verification is high confidence: schema, runtime, docs, tests, and targeted search all show Discord voice still uses STT, a text agent turn, and TTS rather than a native realtime provider session.

## Solution Assessment

Unclear. A separate `channels.discord.voice.realtime` or equivalent config path is plausible and maintainable, but the best shape depends on maintainer direction for #73699 and Discord-specific transcript, routing, permission, and tool policy.
