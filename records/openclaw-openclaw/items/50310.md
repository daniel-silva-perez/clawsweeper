## Summary

Keep open. Current main has adjacent session primitives, but it still only marks archived Discord thread sessions stale, leaves the `sessions.json` key visible through `sessions.list`, has no Discord liveness sweep for channel/thread 403/404, and does not expose an agent-facing `sessions_delete` or `sessions_gc` tool. The related #49259 remains broader Dashboard prune UX work rather than a replacement for this Discord-specific report.

## What This Changes

Request auto-cleanup or agent-safe deletion/GC for stale Discord thread session entries in `sessions.json` when backing Discord threads are archived, deleted, or inaccessible.

## Best Possible Solution

Add a Discord-owned reconciliation path, probably through a preview/apply API or carefully scoped maintenance command, that identifies confirmed orphaned Discord channel/thread sessions, revalidates them before removal, and either removes or hides them from `sessions_list` with tests for archived, deleted/404, inaccessible/403, multi-account, and missing-transcript cases.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to create a `sessions.json` entry like `agent:<agentId>:discord:channel:<threadId>`, trigger the existing archive path, and observe that `updatedAt` becomes `0` while `listSessionsFromStore` still enumerates the key. I did not run a live Discord bot, but the provided report includes live verification on OpenClaw 2026.3.13.

## Solution Assessment

No. Current main is only a partial stale-marking and manual-cleanup solution; the maintainable fix should be an explicit Discord/plugin liveness cleanup contract rather than blind startup deletion or an unrestricted agent deletion tool.
