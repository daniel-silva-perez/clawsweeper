## Summary

Current main still has the reported failure mode: when the reserve plus soft threshold consumes the whole context window, the token-based memory flush and preflight compaction gates disable themselves instead of preserving a positive trigger. The issue should stay open because PR #50694 is still the open, unmerged implementation candidate that explicitly targets this bug.

## What This Changes

The issue requests that memory compaction still trigger when `reserveTokensFloor` is configured equal to the model `contextWindow`, instead of letting a non-positive flush threshold disable compaction.

## Best Possible Solution

Keep this issue open and review or land PR #50694, or replace it with a maintainer-preferred fix that guarantees a positive token-based flush and preflight-compaction threshold when reserve settings are equal to or greater than the context window, with focused regression tests for both equal and greater-than-context values.

## Reproduction Assessment

Yes. A focused static reproduction is to call `shouldRunMemoryFlush` or `shouldRunPreflightCompaction` with positive token usage, `contextWindowTokens` equal to `reserveTokensFloor`, and any nonnegative `softThresholdTokens`; current main computes a non-positive threshold and returns false.

## Solution Assessment

Unclear. A clamp is the right general direction, but maintainers still need to choose the narrow policy location and warning behavior so both the memory-flush and preflight-compaction paths stay consistent.
