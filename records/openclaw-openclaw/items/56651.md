## Summary

Keep open. Current main has adjacent mitigations and knobs for WebSocket handshakes, channel health monitoring, and subagent announce delivery, but it still does not expose the requested gateway-level announce, channel connection, or plugin initialization timeout policy; the remaining work spans config/API design plus gateway/channel/plugin runtime behavior rather than a closed duplicate.

## What This Changes

The issue requests configurable gateway announce, channel connection, and plugin initialization timeouts so startup or delivery stalls fail fast with structured logging instead of hanging indefinitely.

## Best Possible Solution

Define a small, documented gateway timeout policy that separates pre-auth handshake, channel account startup/connection, plugin registration/import/preflight, and announce delivery. Then add config types, Zod/schema/help/docs, focused tests for each phase, and preserve the existing health-monitor restart limits as the recovery mechanism rather than overloading them for startup semantics.
