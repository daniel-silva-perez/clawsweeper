## Summary

Keep this PR open for maintainer review, but do not treat it as already implemented or safe to merge as-is. Current main has shipped the narrower password fallback and explicit loopback-proxy opt-in, while this PR still proposes token fallback and implicit same-host loopback trusted-proxy acceptance that conflict with current startup validation, docs, and the active gateway-auth RFC direction.

## What This Changes

The PR changes gateway auth to add trusted-proxy local shared-secret fallback, including token/password fallback for loopback requests, altered local-direct detection, implicit same-host loopback proxy header handling, and related auth tests.

## Best Possible Solution

Keep the current explicit security model unless maintainers intentionally change it: password fallback for direct internal clients, `allowLoopback` for same-host reverse proxies, and no trusted-proxy token fallback without a coordinated config/docs/RFC decision.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Require explicit opt-in for loopback proxy headers:** `src/gateway/auth.ts:301-302`
  - body: This exception accepts trusted-proxy identity from a loopback source whenever the Host is non-local and any forwarding context is present. A local process can set those headers itself, so this makes same-host loopback trusted-proxy auth implicit instead of gated by `trustedProxy.allowLoopback`. Keep the loopback guard fail-closed unless the explicit config opt-in is set.
  - confidence: 0.9
- **[P2] Preserve raw loopback local-direct detection:** `src/gateway/auth.ts:150`
  - body: `resolveRequestClientIp` intentionally returns `undefined` when the socket address is in `trustedProxies` and no forwarded client header is present. With loopback in `gateway.trustedProxies`, a direct `127.0.0.1` request without proxy headers stops being local-direct, which affects shared call sites such as readiness details, local session kill, WS locality, and realtime owner checks.
  - confidence: 0.86
- **[P2] Align token fallback with startup validation:** `src/gateway/auth.ts:387`
  - body: The new fallback accepts `auth.token` in trusted-proxy mode, but startup validation still rejects that configuration as mutually exclusive. As a result, the advertised local token fallback cannot be used through normal gateway config unless the validation and docs contract change with it.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff changes gateway authentication trust boundaries and introduces concrete loopback trusted-proxy and token-mode concerns.

Concerns:

- **[high] Implicit loopback trusted-proxy trust:** `src/gateway/auth.ts:301`
  - body: The PR would allow trusted-proxy identity headers from `127.0.0.1` based on Host and forwarding-header shape instead of requiring `trustedProxy.allowLoopback`; local processes can forge those inputs, so this weakens the explicit same-host proxy trust boundary.

  - body: The PR adds token fallback inside trusted-proxy mode while current main treats token and trusted-proxy as mutually exclusive. That changes credential eligibility for a security-sensitive auth mode without the matching startup/docs contract.
