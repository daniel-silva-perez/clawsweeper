## Summary

Keep this PR open for maintainer/security review, but do not merge it as-is. The current branch changes gateway authentication trust boundaries in ways that conflict with current main's explicit loopback and credential contract: loopback trusted-proxy headers become implicitly accepted, token fallback is added where startup/docs reject it, and local-direct detection is changed away from the raw socket rule main currently relies on.

## What This Changes

This PR modifies gateway trusted-proxy authentication and tests to add local loopback shared-secret fallback, token-first fallback, and implicit same-host loopback proxy handling.

## Best Possible Solution

Keep the explicit current model unless maintainers deliberately redesign it: same-host reverse proxies should use `trustedProxy.allowLoopback`, local direct internal callers should use password fallback, and token or service-identity fallback should be handled through the RFC with matching config, docs, startup validation, and tests.

## Reproduction Assessment

Yes. The PR branch can be inspected or exercised with `authorizeGatewayConnect` using trusted-proxy mode, a loopback socket, a non-local Host plus forwarding headers to show implicit loopback trusted-proxy acceptance, and `auth.token` in trusted-proxy mode to show the mismatch with current startup validation.

## Solution Assessment

No. This is not the best way to solve the reported deployment pain because it broadens trusted-proxy eligibility and credential modes without the explicit config and docs contract that current main uses; the safer path is the current password-only/allowLoopback split or the RFC service-identity design.

- **[P1] Require explicit opt-in for loopback proxy headers:** `src/gateway/auth.ts:301-302`
  - body: This exception accepts trusted-proxy identity from a loopback source based on Host and forwarding-header shape instead of `trustedProxy.allowLoopback`. A local process can forge those headers, so same-host loopback trusted-proxy auth becomes implicit rather than explicitly configured.
  - confidence: 0.9

  - body: `isLocalDirectRequest` now depends on proxy-aware client IP resolution. When loopback is listed in `gateway.trustedProxies` and no forwarded client header is present, that resolver can return no client IP, so direct loopback requests may stop being classified as local-direct.
  - confidence: 0.86
- **[P2] Align token fallback with startup validation:** `src/gateway/auth.ts:390-396`
  - body: The fallback accepts `auth.token` in trusted-proxy mode, but current startup validation rejects trusted-proxy configs that also set a shared token. The advertised token fallback is therefore not usable through normal config unless validation and docs change with it.
  - confidence: 0.88
- **[P2] Validate trusted-proxy config before fallback:** `src/gateway/auth.ts:528-537`
  - body: The local shared-secret fallback runs before checking that `trustedProxy` and `trustedProxies` are configured. A local request with a configured password can therefore succeed where current main fails closed with `trusted_proxy_config_missing` or `trusted_proxy_no_proxies_configured`.
  - confidence: 0.82
