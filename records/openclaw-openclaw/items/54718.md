## Summary

Keep this PR open, but it should not merge as-is. The branch changes a security-sensitive gateway auth boundary, and the current discussion plus source/docs show that main intentionally keeps token auth mutually exclusive with trusted-proxy mode and requires explicit `trustedProxy.allowLoopback` before accepting loopback proxy identity headers.

## What This Changes

The PR adds trusted-proxy local fallback changes in `src/gateway/auth.ts` and tests, including token/password fallback for loopback requests and same-host loopback proxy handling.

## Best Possible Solution

Keep the current explicit model unless maintainers deliberately change it: password-only local-direct fallback for trusted-proxy internal callers, explicit `trustedProxy.allowLoopback` for same-host reverse proxies, and any token/service identity fallback handled through the RFC with coordinated config, docs, and tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Require explicit opt-in for loopback proxy headers:** `src/gateway/auth.ts:301-302`
  - body: This exception accepts trusted-proxy identity from a loopback source based on Host and forwarding-header shape instead of `trustedProxy.allowLoopback`. A local process can forge those headers, so same-host loopback trusted-proxy auth becomes implicit rather than explicitly configured.
  - confidence: 0.9
- **[P2] Preserve raw loopback local-direct detection:** `src/gateway/auth.ts:151-153`
  - body: `isLocalDirectRequest` is changed to depend on proxy-aware client IP resolution. When loopback is listed in `gateway.trustedProxies` and no forwarded client header is present, that resolver can return no client IP, so direct loopback requests may stop being classified as local-direct.
  - confidence: 0.86
- **[P2] Align token fallback with startup validation:** `src/gateway/auth.ts:390-396`
  - body: The fallback accepts `auth.token` in trusted-proxy mode, but current startup validation rejects trusted-proxy configs that also set a shared token. The advertised token fallback is therefore not usable through normal config unless validation and docs change with it.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff changes gateway authentication trust boundaries and introduces concrete loopback trusted-proxy and token-mode concerns.

Concerns:

  - body: The PR would allow trusted-proxy identity headers from `127.0.0.1` based on Host and forwarding-header shape instead of requiring `trustedProxy.allowLoopback`; local processes can forge those inputs, weakening the explicit same-host proxy trust boundary.
