## Summary

Keep open. Current main still installs the non-context-engine tool-result guard from `params.model.contextWindow`/`maxTokens`, while the attempt already receives the resolved `contextTokenBudget` and adjacent runtime paths use it. The PR is a narrow alignment of that divergent call site, and I found no blocking functional or security issue in the diff.

## What This Changes

The PR changes `runEmbeddedAttempt` so the default tool-result context guard prefers `params.contextTokenBudget` before falling back to model `contextWindow`, `maxTokens`, or `DEFAULT_CONTEXT_TOKENS`.

## Best Possible Solution

Land the targeted fallback change, preferably with a small wiring regression for the attempt call site, then let the linked bug close through the PR merge.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists by resolving a model where `contextTokenBudget` exceeds `model.contextWindow` and observing that current main installs the default guard with the smaller model window.

## Solution Assessment

Yes. The PR is the narrowest maintainable fix if OpenClaw continues treating `contextTokens` as the effective runtime budget, because it reuses the already-resolved budget instead of adding new policy or normalization logic.
