## Summary

Keep this open. Current main has the raw QQ Bot WebSocket dispatch payload on debug, but normal inbound QQ Bot handling still logs the full user-visible message body at info, so the central privacy/noise request remains unresolved.

## What This Changes

Request to reduce QQ Bot plugin default info-level log noise and avoid logging full inbound message bodies or WebSocket payloads at info by default.

## Best Possible Solution

Fix this inside the QQ Bot plugin by making default info logs metadata-only, such as account id, message id, sender/peer id, message type, and content length, while keeping full message bodies and raw dispatch payloads behind debug or an explicit verbose diagnostic path. Add focused tests that fail if normal info logging includes inbound message text.

## Reproduction Assessment

Yes. A static current-main reproduction is high confidence: any queued QQ Bot message enters `handleMessage`, and the first log statement includes `event.content` at info before the inbound pipeline runs.

## Solution Assessment

Yes. A narrow QQ Bot plugin-local logging change is the best fit; changing core logging/redaction globally or adding a new verbosity knob is unnecessary unless maintainers want a separate opt-in diagnostic mode.
