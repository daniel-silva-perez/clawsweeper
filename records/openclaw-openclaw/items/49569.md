## Summary

Keep this PR open. Current main still does not retry Telegram command-menu `setMyCommands` on 429 `retry_after`; it only handles `BOT_COMMANDS_TOO_MUCH`. The PR is a focused candidate and the earlier unrelated lockfile/registry concerns are gone, but the current PR head still parses only top-level `error_code` plus description text, missing the structured grammY/Telegram `parameters.retry_after` shapes already supported elsewhere. It also has a failing `checks-node-test` check, so it should not be closed or landed as-is.

## Best Possible Solution

Keep the PR open. Ask for a rebase/rework that implements command-menu 429 handling against current main, reuses or mirrors the existing structured `retry_after` extraction behavior, adds regression tests for top-level and wrapped structured retry-after shapes, keeps the process-local command hash/cache contract explicit, and clears the failing check before maintainer review/landing.
