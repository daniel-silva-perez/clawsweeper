## Summary

Keep this PR open. Current main still does not implement retry_after-aware backoff for Telegram command-menu setMyCommands, and the PR remains a plausible focused fix, but its current head still needs maintainer review/rework before landing.

## Best Possible Solution

Keep the PR open, ask for a rebase onto current main, and narrow the implementation to Telegram command-menu retry_after handling that reuses or matches the existing Telegram rate-limit helper shapes. The fix should prefer structured parameters.retry_after, support nested response/error.parameters shapes, keep the current cache contract explicit, add regression tests for structured and fallback retry_after cases, and clear the failing check before landing.
