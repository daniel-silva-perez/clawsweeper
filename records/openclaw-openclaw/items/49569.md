## Summary

Keep this PR open. Current main still does not implement retry_after-aware backoff for Telegram command-menu setMyCommands, and the PR is a focused implementation candidate, but its current head still misses structured Telegram/grammY retry_after shapes and has a failing check, so it is not ready to close or land as-is.

## Best Possible Solution

Keep the PR open and ask for a rebase/rework that implements command-menu retry_after handling against the current process-local cache design. The fix should reuse or match the existing Telegram/channel retry helper behavior, prefer structured parameters.retry_after including wrapped response/error.parameters shapes, keep the exhaustion/cache behavior explicit, add structured and fallback retry_after regression tests, and clear the failing current-head check before landing.
