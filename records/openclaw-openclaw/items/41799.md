## Summary

Keep this PR open. Current main still does not tolerate runtime legacy `{ ok, to }` results from `messaging.normalizeTarget`; those object results are rejected by string coercion and the resolver falls back to the raw target. The PR addresses a real missing compatibility path, but its current diff needs cleanup before maintainer review, especially unrelated secret-baseline churn and the broad OAuth test mock.

## Best Possible Solution

Keep the PR open and rework it against current main. Add a small runtime-only helper in `src/infra/outbound/target-normalization.ts` that treats the normalizer result as `unknown`, extracts a trimmed nonblank string `to` from legacy `{ ok, to }` values, and leaves malformed objects on the existing fallback/validation path. Cover it in `src/infra/outbound/target-normalization.test.ts` plus one resolver or send-runner regression, then remove the unrelated `.secrets.baseline` churn and broad OAuth mock before maintainer review.
