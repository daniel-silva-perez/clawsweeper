## Summary

Keep this issue open. Current main has improved neighboring fallback paths, including thrown unrecognized errors and OpenRouter no-endpoints errors, but the reported normal embedded run result with only an unclassified business-error payload can still reach `candidate_succeeded`. The exact `CE-011` / `禁止访问该模型` shape is absent from current classifiers and tests.

## Best Possible Solution

Keep this issue open and add a focused embedded fallback regression: simulate an Opus timeout, then a Sonnet normal embedded result containing only an `isError` business rejection such as `{"success":false,"code":"CE-011","message":"当前ak因违规请求被禁止访问该模型"}`, and assert fallback continues to the next configured model. The conservative fix should classify structured provider business-error payloads or the `CE-011` / `禁止访问该模型` wording before `candidate_succeeded` is recorded, preferably by returning or throwing a `FailoverError` with an auth or unknown failover reason from the assistant/result classifier path.
