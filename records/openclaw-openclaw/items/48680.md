## Summary

Keep this issue open. Current main still has the reported gap for an HTTP-200 provider business rejection that surfaces as an embedded `isError` payload but is not classified by the failover classifiers. Neighboring paths are better now, including thrown unrecognized errors and OpenRouter no-endpoints errors, but `CE-011` / `当前ak` / `禁止访问` is still absent from source and tests, so the Sonnet attempt can still be accepted as `candidate_succeeded` before the third fallback is tried.

## Best Possible Solution

Keep this issue open and add a focused embedded fallback regression for the exact shape: primary timeout, then a normal Sonnet embedded result with only an `isError` payload such as `{"success":false,"code":"CE-011","message":"当前ak因违规请求被禁止访问该模型"}`, then assert the chain continues to the next fallback model. The fix should classify the structured business rejection or `禁止访问`/CE-011 text before `candidate_succeeded` is recorded, preferably by converting it into a `FailoverError` with an auth or unknown failover reason while preserving side-effect guards.
