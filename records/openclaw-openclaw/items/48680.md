## Summary

Keep open. Current main still allows the reported HTTP-200 business rejection shape to return as a normal embedded error-only result; the fallback loop only treats it as failed if the result or assistant classifier recognizes it, and the current classifiers still do not cover `success:false`, `CE-011`, or the reported Chinese denial wording.

## What This Changes

The issue requests that model fallback treat provider business rejections surfaced as embedded `isError` payloads as failed candidates so later fallback models are attempted.

## Best Possible Solution

Land a focused classifier fix plus regression coverage for an error-only embedded result containing structured `success:false` / `CE-011` / `禁止访问` denial text. The fix should convert that semantic provider rejection into a failover classification before `candidate_succeeded` is recorded, while preserving the current safeguards for visible replies, aborts, and side effects.

## Reproduction Assessment

Yes. The issue provides concrete gateway logs, and current source gives a high-confidence code reproduction path: first candidate fails, second returns a nonthrowing error-only embedded result with the CE-011 payload, and the existing classifier returns null for that non-GPT result so the fallback loop can mark success.

## Solution Assessment

Yes. The narrowest maintainable fix is in the embedded result/failover classifier path with focused tests; changing the fallback loop to retry every error payload without classifier and side-effect guards would be riskier.
