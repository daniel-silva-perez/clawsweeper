## Summary

Keep this issue open. Current main has fixed several neighboring fallback bugs, including OpenRouter 404/no-endpoints and structured 402/529-style failures, but the specific reported HTTP-200 business rejection shape is still not covered generically. A normal embedded run result with only an unclassified `isError` payload can still reach `candidate_succeeded` unless the assistant error text is recognized by the failover classifier or the result classifier turns it into a `FailoverError`. The reported `CE-011` / `禁止访问该模型` wording is absent from the current classifiers and tests.

## Best Possible Solution

Add a focused fallback classification path for provider business-error run results before `candidate_succeeded` is recorded. The conservative fix likely belongs in the assistant/result failover classifier: recognize structured business error payloads or the CE-011/banned-model wording as auth/failover-worthy, throw or return a `FailoverError`, and add regression coverage that a Sonnet failure continues to the next configured model instead of stopping the chain. Coordinate with the related open unclassified-error fallback work rather than closing this as already fixed.
