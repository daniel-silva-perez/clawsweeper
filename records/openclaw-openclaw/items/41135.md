## Summary

Keep this issue open. Current main still implements auth-profile ordering, lastUsed-based rotation, session auto-pinning, cooldown/failover, and status/probe reporting, but it does not implement the requested opt-in provider-profile pool routing layer with named routing strategies, task/none stickiness, weights, per-profile in-flight/concurrency limits, pool-state JSON, or a `models routing` CLI surface. The existing ClawSweeper review comment reached the same conclusion, and current main remains consistent with that assessment.

## Best Possible Solution

Keep this as an open design/feature issue. The best path is an additive core auth-routing contract: strict config/schema for pool mode, strategy, stickiness, profile list, weights, and concurrency limits; runtime in-flight/cooldown-aware profile selection in the embedded runner/provider auth path; observable route state through `models status --json`; and CLI controls after the config and runtime semantics are stable. Provider hooks can participate without owning the generic scheduler.
