## Summary

Keep this issue open. Current main still has auth-profile ordering, lastUsed-based round-robin, session-sticky auto-pinning, cooldown/failover, and auth status/probe reporting, but it does not implement the requested opt-in provider-profile pool routing layer with named strategies, task/none stickiness, weights, per-profile in-flight/concurrency limits, pool state in status JSON, or a `models routing` CLI surface. The prior ClawSweeper comment reached the same conclusion, and the only touched-surface commit since that review is unrelated to auth routing.

## Best Possible Solution

Keep this as an open design/feature issue. The best path is an additive core auth-routing contract: strict config/schema for pool mode, strategy, stickiness, profile list, weights, and concurrency limits; runtime in-flight/cooldown-aware profile selection in the embedded runner/provider auth path; observable route state through `models status --json`; and CLI controls after the config and runtime semantics are stable. Provider hooks can participate without owning the generic scheduler.

## Risks / Open Questions

- A correct implementation spans config/schema, auth-state persistence, embedded runner selection, model fallback, status JSON, CLI/docs, and provider hook participation; a provider-only tweak would likely leave batch routing semantics inconsistent.
- The Gemini probe-timeout example may deserve a narrower bug investigation, but that would not replace the broader missing provider-pool routing contract.
