## Summary

Keep this issue open. Current main still implements auth profile order, cooldown/disabled failover, provider-owned single-profile preference hooks, and session pinning, but it does not implement the requested opt-in provider-profile pool routing layer with pool config, named strategies, task/none stickiness, per-profile concurrency/in-flight state, weights, quota-aware state, pool-state JSON, or `models routing` CLI controls. The existing issue discussion also contains a recent ClawSweeper review that reached the same keep-open conclusion.

## Best Possible Solution

Keep this as an open core feature/design issue. The best path is an additive auth-routing contract with strict config/schema for pool mode, strategies, stickiness, profile lists, weights, and concurrency limits; runtime in-flight/cooldown-aware selection in the embedded runner and fallback path; pool state in `models status --json`; and CLI controls after the config/runtime semantics are stable. Provider hooks can provide hints, but the generic scheduler and state model should be core-owned.
