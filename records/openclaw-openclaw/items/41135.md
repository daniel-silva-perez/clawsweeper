## Summary

Keep this issue open. Current main still has auth profile ordering, cooldown/disabled failover, session pinning, and a provider hook that can prefer one profile from the resolved order, but it does not implement the requested opt-in provider-profile pool routing layer with pool config, named strategies, task/none stickiness, per-profile concurrency/in-flight state, weights, quota-aware state, pool-state JSON, or `models routing` CLI controls. The latest ClawSweeper comment in the issue also kept it open, and the current main review matches that assessment.

## Best Possible Solution

Keep this as an open core feature/design issue. The best path is an additive auth-routing contract with strict config/schema for pool mode, strategies, stickiness, profile lists, weights, and concurrency limits; runtime in-flight/cooldown-aware selection in the embedded runner and fallback path; observable route state through `models status --json`; and CLI controls after config/runtime semantics are stable. Provider hooks can participate, but the generic scheduler and state model should be core-owned.
