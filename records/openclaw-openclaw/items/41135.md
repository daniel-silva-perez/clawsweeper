## Summary

Keep this issue open. Current main still has auth-profile ordering, lastUsed/cooldown-based rotation, auto/user session pinning, provider hooks that can prefer a single profile, and status/probe reporting, but it does not implement the requested opt-in provider-profile pool routing layer with named strategies, task/none stickiness, weights, in-flight/concurrency limits, pool-state JSON, or a `models routing` CLI surface. The existing ClawSweeper comment reached the same conclusion, and current main remains consistent with that assessment.

## Best Possible Solution

Keep this as an open feature/design issue. The best path is an additive core auth-routing contract: strict config/schema for pool mode, strategy, stickiness, profile list, weights, and concurrency limits; runtime in-flight/cooldown-aware profile selection in the embedded runner and fallback path; observable route state through `models status --json`; and CLI controls after the config and runtime semantics are stable. Provider hooks can participate, but the generic scheduler and state model should be core-owned.

## Risks / Open Questions

- The Gemini CLI probe-timeout example may deserve a narrower bug investigation, but fixing that path would not satisfy the broader missing provider-pool routing contract.
- A complete implementation likely spans config/schema, auth-state persistence, embedded runner selection, fallback behavior, status JSON, CLI/docs, and provider hook participation; a provider-only tweak would be insufficient for the generic scheduler semantics requested.
