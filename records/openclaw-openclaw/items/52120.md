## Summary

Keep this PR open. Current main still lacks the Mattermost tool-context hook needed for the linked proactive message-tool threading bug, and the paired bug report remains open, but the PR branch should not be landed as-is because current main now has shared outbound threading helpers that this older patch does not account for.

## What This Changes

The PR modifies the Mattermost message action and tests to derive active thread context for message-tool sends and use it as an implicit reply target when no explicit reply target is supplied.

## Best Possible Solution

Adapt the Mattermost-owned fix to current main: add a Mattermost `buildToolContext` that supplies normalized conversation id, active thread root, account-scoped `replyToMode`, and `hasRepliedRef`; add a current-main-compatible `resolveAutoThreadId` or equivalent shared-helper integration so Mattermost sends use the thread root before dispatch and mirror routing; preserve explicit reply precedence and same-channel guards; cover all/first/off, explicit replies, and cross-channel cases with focused tests.

## Reproduction Assessment

Yes. The PR body and linked bug give a concrete reproduction path for `replyToMode: "all"` thread-scoped Mattermost sessions, and current main source inspection shows the missing `buildToolContext`/`replyToMode` path that makes the message tool fall back out of the thread.

## Solution Assessment

No. The requested direction is correct, but this PR is not the best current-main solution because it was written before newer shared outbound threading helpers and it preserves `first` mode inside an existing thread where Mattermost follow-ups should continue using the active thread root.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Route Mattermost auto-threading before generic reply injection:** `extensions/mattermost/src/channel.ts:501-504`
  - body: On current main, returning `currentChannelId` in the same `channel:<id>` shape as message targets lets the shared runner set `params.replyTo` from `currentMessageId` before Mattermost `handleAction` runs. That explicit reply masks this PR's `currentThreadTs` fallback, so a threaded Mattermost send can use the latest message id as `rootId` instead of the thread root. Wire Mattermost through `resolveAutoThreadId`/transport or otherwise prevent generic current-message reply injection for this channel.
  - confidence: 0.78
- **[P2] Promote active first-mode threads to all:** `extensions/mattermost/src/channel.ts:495-496`
  - body: When the active Mattermost session is already inside a thread, preserving `replyToMode="first"` means later tool sends stop inheriting `currentThreadTs` after `hasRepliedRef` is true and fall back to the channel root. Mattermost's existing thread handling keeps `first` follow-ups on the active root, so the effective tool mode should be `all` for an existing thread, not only when the configured mode is `off`.
  - confidence: 0.82
