## Summary

Keep open. Current main does not implement the PR's central change: `models.providers.<name>.retry` is not accepted by the model provider config schema/types, there is no provider retry registry/runner, and the OpenAI-compatible transports still create provider streams directly. The PR is focused and has follow-up fixes for several review comments, but it needs maintainer review, a rebase onto current transport timeout handling, and validation. It also overlaps open PR #38640, so maintainers should choose one canonical provider-retry path rather than auto-close this useful work.

## Best Possible Solution

Keep this PR open for maintainer review. If OpenClaw wants configurable provider-call retries, land one canonical implementation that adds `models.providers.<name>.retry` to the public config contract, registers retry policy during provider normalization without stale runtime state, wraps current OpenAI-compatible transport creation while preserving SDK timeout/failover behavior, documents the scope, and adds focused tests for retry classification, config reload/removal, abort handling, and non-retryable client errors. Also resolve the overlap with #38640 so only one provider-retry design remains active.

## Risks / Open Questions

- The PR head is currently not mergeable and is based behind current main; current transport code now uses `buildOpenAISdkRequestOptions` for SDK timeouts, so a rebase must preserve timeout behavior while adding retry wrapping.
- The implementation only wires OpenAI-compatible transports in the provided diff, while the linked problem statement for #31104 cited Anthropic cron failures. Maintainers should confirm the intended scope or document that non-OpenAI transports remain out of scope.
- Opt-in provider retries can increase token/API spend and may duplicate provider work if a backend accepted a request before surfacing a transient failure. Error classification and idempotency assumptions need careful review.
- The PR changes shared `retryAsync` abort behavior and runner signatures, which affects channel retry callers as well as provider retry. That needs regression coverage for existing channel retry semantics.
- Visible check-runs for the current PR head are only automation/label jobs, not build/test proof. The PR body claims tests, but maintainers still need current targeted tests and the normal changed gate after rebase.
