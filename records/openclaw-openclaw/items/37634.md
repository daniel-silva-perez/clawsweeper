## Summary

Current main still leaves `workspaceAccess: "none"` in the read-only/non-`rw` bucket even though it uses the isolated sandbox workspace. The linked open PR #46300 remains the implementation candidate; this issue should stay open until that PR, or a superseding fix, lands with sandbox permission review.

## Best Possible Solution

Keep this issue open as the focused tracker. The best path is to review, land, or supersede #46300 with a complete fix that makes `workspaceAccess: "none"` writable only for the isolated sandbox workspace across Docker/browser mounts, local and remote filesystem bridges, memory flush gating, docs, and regression tests, with `@openclaw/secops` review of the permission semantics.
