## Summary

Current main still leaves this issue unresolved. `workspaceAccess: "none"` uses the isolated sandbox workspace, but Docker/browser mounts, local and remote filesystem bridges, memory-flush gating, docs, and regression tests still treat that workspace as read-only or non-writable. The related open PR #46300 remains the implementation candidate rather than proof this is fixed.

## Best Possible Solution

Keep this issue open as the focused tracker. The best path is to review, land, or supersede #46300 with a complete fix that makes `workspaceAccess: "none"` writable only for the isolated sandbox workspace across Docker/browser mounts, local and remote filesystem bridges, memory flush gating, config-hash recreation, docs, and regression tests, with `@openclaw/secops` review of the permission semantics.
