## Summary

Keep this issue open. Current main still gives `workspaceAccess: "none"` an isolated sandbox workspace, but the Docker/browser mount layer and both filesystem bridges still treat that workspace as read-only/write-blocked. The earlier fix PRs #37276 and #38154 are closed unmerged, and #46300 is only an open implementation candidate, not shipped behavior.

## Best Possible Solution

Keep this issue open as the focused tracker. The best path is to review, land, or supersede #46300 with a complete fix across Docker/browser mounts, host and remote filesystem bridges, mutation gates, and regression tests, then update sandbox docs to match. If maintainers intentionally prefer the current no-write semantics for `workspaceAccess: "none"`, close only after an explicit product/security decision and docs confirmation.
