## Summary

Keep this issue open. Current main still gives `workspaceAccess: "none"` an isolated sandbox workspace, but the Docker/browser mount layer, local filesystem bridge, remote filesystem bridge, memory-flush gating, tests, and sandbox docs still treat that mode as read-only or non-writable. The open PR #46300 is a plausible implementation candidate, but it is not merged and appears to need reconciliation with current main’s remote/OpenShell bridge and docs.

## Best Possible Solution

Keep this issue open as the focused tracker. The best path is to review, land, or supersede #46300 with a complete fix that makes `workspaceAccess: "none"` writable only for the isolated sandbox workspace across Docker/browser mounts, host and remote filesystem bridges, memory flush gating, config-hash recreation, docs, and regression tests, with `@openclaw/secops` review of the permission semantics.
