## Summary

Keep this issue open. Current main still models memory as one active exclusive memory slot, with newer non-exclusive supplement APIs for companion memory-adjacent plugins. Those supplement APIs partially address cases like prompt/corpus additions, but they do not implement the requested purpose-specific slots (`memory.recall`, `memory.compaction`, `memory.capture`, `memory.userModel`) or per-agent memory slot selection. The discussion also contains recent concrete production/use-case evidence, so this is not stale or superseded.

## Best Possible Solution

Keep this issue open as the canonical multi-slot memory architecture RFC. A good implementation path would extend the core plugin/config contract with typed purpose-specific memory roles or an equivalent composable memory capability model, preserve backward compatibility for `plugins.slots.memory`, clarify per-agent scoping, and update the SDK/docs/tests so third-party memory plugins can compose without relying on workaround slots or sidecars.

## Risks / Open Questions

- The current supplement APIs and active-memory plugin solve some companion-memory workflows, so any future multi-slot design should build on those seams instead of duplicating them.
- This is a broad architecture/product decision touching config schema, plugin activation, runtime loading, docs, and third-party plugin compatibility; it needs maintainer design review before implementation.
