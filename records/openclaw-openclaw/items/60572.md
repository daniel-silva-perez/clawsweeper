## Summary

Keep open. Current main still treats memory as one active slot, with supplemental memory APIs and a narrow dreaming sidecar path, but it does not implement purpose-specific memory slots or per-agent memory slot selection. The issue discussion includes recent concrete production/plugin-composition use cases, so this remains an active architecture RFC rather than cleanup.

## Best Possible Solution

Keep this issue open as the canonical multi-slot memory architecture RFC. A good implementation would add a backward-compatible composable memory-role contract or purpose-specific memory slots, preserve current `plugins.slots.memory` behavior, decide whether per-agent slot scoping belongs in the config model, and update SDK/docs/tests so third-party memory plugins can compose without workaround slots or sidecars.
