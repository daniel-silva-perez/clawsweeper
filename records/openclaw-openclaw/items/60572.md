## Summary

Keep this issue open. Current main still models memory as one active memory slot, with additive supplement APIs and a narrow dreaming coexistence path, but it does not implement the requested purpose-specific memory slots or per-agent memory slot selection. The recent discussion includes concrete production and plugin-composition use cases, so this remains an active architecture RFC rather than stale cleanup.

## Best Possible Solution

Keep this issue open as the canonical multi-slot memory architecture RFC. A good path would add a backward-compatible composable memory-role contract or purpose-specific memory slots, preserve `plugins.slots.memory` behavior, define per-agent scoping if maintainers want it, and update SDK/docs/tests so third-party memory plugins can compose without workaround slots or sidecars.
