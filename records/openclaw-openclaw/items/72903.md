## Summary

Close as implemented: current main no longer lets an omitted `gateway.reload.deferralTimeoutMs` wait forever. The restart deferral path now resolves missing/invalid timeout config to a 300000 ms default, forces the gateway restart when that bound is reached, and has a regression test for the omitted-config case; the fix is present in `v2026.4.29-beta.1`.

## What This Changes

The issue reports that inbound webchat messages can start new task runs while a restart-required config change is deferred, keeping the gateway restart pending indefinitely and leaving the main session in processing.

## Best Possible Solution

Keep the bounded default restart deferral as the shipped fix for the indefinite wedge. If maintainers later want nicer UX during the deferral window, queueing or rejecting inbound messages should be tracked as a separate enhancement rather than blocking cleanup of this deadlock report.

## Reproduction Assessment

Yes. The issue provides concrete runtime steps and a log timeline, and current main has a focused fake-timer regression test proving that an omitted timeout no longer waits indefinitely while active task work remains present.

## Solution Assessment

Yes. A hard default timeout is one of the issue’s listed acceptable fixes and is the narrowest maintainable change for the indefinite restart-deferred deadlock; it does not add broader inbound queue/reject semantics, but it removes the unbounded stuck-processing failure mode.
