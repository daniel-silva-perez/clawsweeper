## Summary

Keep open. Current main still has the CLI selection gap: `openclaw message send --media` can select a channel account, but the CLI always runs outbound delivery under the default agent, while the media layer only adds the workspace for the agent id it receives. The related media-root PRs in the supplied context are still open and do not provide a merged current-main CLI fix.

## What This Changes

Request `openclaw message send --media` to accept local media from the relevant configured agent workspace, or expose an equivalent safe media-root allowlist for per-agent workspace paths.

## Best Possible Solution

Add a documented `openclaw message ... --agent <id>` selector, validate it against configured agents, thread it through the message CLI helper and `messageCommand` into `runMessageAction`, and use that selected agent workspace for media roots. Keep `--account` scoped to channel account selection; only add account-to-agent inference if it is unique, deterministic, documented, and tested.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to configure a non-default agent workspace and run CLI media send with `--account` but no agent selector: current `messageCommand` still passes the default agent id, so workspace-only media roots remain scoped to the default agent rather than the account's intended agent.

## Solution Assessment

No current-main solution is present. The narrowest maintainable fix is an explicit validated agent selector rather than automatically adding every configured workspace to the local media allowlist.
