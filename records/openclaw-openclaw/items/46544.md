## Summary

Keep open. Current main still has the reported CLI gap: `openclaw message send --media` accepts `--account`, but no CLI `--agent` selector is exposed or threaded, so media-root resolution still uses the configured default agent rather than the workspace for another bound account. The related media-root PRs are useful adjacent context, but they are open and not merged current-main fixes for this CLI path.

## What This Changes

The issue requests that CLI `message send --media` allow media files from the selected or configured agent workspace instead of only default media roots/default-agent roots.

## Best Possible Solution

Add a documented `openclaw message ... --agent <id>` selector, validate it against configured agents, thread it through the message CLI helper and `messageCommand` into `runMessageAction`, and keep media roots scoped to that selected agent. Account-to-agent inference should only be added if it is unique, deterministic, documented, and tested; current main should not include every configured workspace by default.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to configure a non-default agent workspace bound to a Telegram account and run CLI `message send --media` with `--account`; current `messageCommand` still passes the default agent id, so media roots are not scoped to the bound agent workspace.

## Solution Assessment

No current-main solution is present. The narrowest maintainable fix is an explicit validated agent selector, with account inference kept secondary and least-privilege if maintainers want it later.
