## Summary

Current main still leaves the reported CLI gap open: `openclaw message` accepts `--account` but no agent selector, `messageCommand` always passes the default agent into `runMessageAction`, and media roots are derived from that supplied agent id. The adjacent media-root PRs in the provided context are open/unmerged and do not currently implement a merged CLI agent-selection fix.

## What This Changes

`openclaw message send --media` should use the selected agent workspace for local media access instead of only default roots or the default agent workspace.

## Best Possible Solution

Keep this open and implement a least-privilege CLI fix: add an explicit `--agent` selector to `openclaw message`, thread it through `createMessageCliHelpers` and `messageCommand` into `runMessageAction`, and use that selected agent id for media-root resolution. If maintainers also want `--account` inference, only infer an agent when the account binding is unique and unambiguous; do not allowlist all agent workspaces globally.
