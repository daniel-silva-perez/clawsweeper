## Summary

Current main still leaves the reported CLI gap open for restrictive or workspace-only media policies: the message CLI accepts `--account` but has no agent selector, `messageCommand` always supplies the default agent to `runMessageAction`, and media roots are scoped from that supplied agent id. Adjacent media-root PRs in the provided context remain open/unmerged and do not add a merged CLI agent-selection fix.

## What This Changes

`openclaw message send --media` should use the selected agent workspace for local media access instead of only default roots or the default agent workspace.

## Best Possible Solution

Keep the media allowlist least-privilege by selecting one concrete agent workspace for CLI sends: add an explicit message CLI `--agent` option, thread it through the CLI helpers and `messageCommand` into `runMessageAction`, and only add account-to-agent inference if the account binding is unique and unambiguous.
