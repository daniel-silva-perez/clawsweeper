## Summary

Keep open. Current main has strong agent-scoped media-root plumbing once an agent id is supplied, but the CLI `message send` path still targets the configured default agent rather than resolving a non-default agent from `--account` or exposing `--agent`. That leaves the reported multi-account/non-default workspace case unsolved under strict workspace/media policies.

## Best Possible Solution

Keep this issue open and implement a narrow CLI targeting fix: either add `openclaw message send --agent <id>` or infer the bound agent from `--channel` + `--account`, pass that agent id into `runMessageAction`, and add regression coverage for a non-default agent with `workspace-<id>` sending local media under strict media roots.
