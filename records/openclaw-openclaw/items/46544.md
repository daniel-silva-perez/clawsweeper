## Summary

Keep open. Current main still has the reported CLI media-root gap: the message CLI can select an account, but it does not expose or thread an agent selector, so outbound media roots remain scoped to the configured default agent rather than the workspace for a non-default bound agent. The adjacent media-root PRs are relevant context but are open, unmerged, and do not prove a current-main fix for this CLI path.

## What This Changes

The issue requests that CLI `openclaw message send --media` allow media from the workspace associated with the selected agent/account instead of only the default media roots.

## Best Possible Solution

Add a documented `openclaw message ... --agent <id>` selector, validate it against configured agents, thread it through the message CLI helper and `messageCommand` into `runMessageAction`, and keep media roots scoped to that selected agent workspace. Account-to-agent inference should remain secondary and only be added if it is unique, deterministic, documented, and tested; current main should not include every configured workspace by default.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to configure a non-default agent workspace bound to a Telegram account and run `openclaw message send --media` with `--account`; current `messageCommand` still supplies the default agent id, while the outbound runner scopes roots from that supplied id.

## Solution Assessment

No current-main solution is present. The narrowest maintainable direction is an explicit validated agent selector; broadening to all workspaces or open-ended configured roots is less safe for a filesystem media allowlist.
