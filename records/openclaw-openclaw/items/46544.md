## Summary

Keep open. Current main still has the reported CLI selection gap: `openclaw message send --media` exposes channel account selection but does not expose or thread an agent selector, so outbound media roots are still scoped from the default agent passed by the CLI.

## What This Changes

The issue requests that CLI `message send --media` use an agent-scoped workspace root, or an equivalent configured allowlist, so files from non-default agent workspaces can be sent.

## Best Possible Solution

Add a documented `openclaw message ... --agent <id>` selector, validate it against configured agents, thread it through the message CLI helper and `messageCommand` into `runMessageAction`, and keep media roots scoped to that selected agent. Account-to-agent inference should only be added if it is unique, deterministic, documented, and tested.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to configure a non-default agent workspace bound to a Telegram account and run CLI `message send --media` with `--account`; current `messageCommand` still passes the default agent id, so media roots are not scoped to the bound agent workspace.

## Solution Assessment

No current-main solution is present. The narrowest maintainable direction is an explicit validated agent selector, with any account inference kept secondary and least-privilege.
