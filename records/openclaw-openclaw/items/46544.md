## Summary

Current main still leaves the reported CLI gap open: `openclaw message send --media` exposes `--account` but no `--agent`, `messageCommand` still passes the configured default agent to `runMessageAction`, and the media root helpers only add the workspace for the agent id they are given. The issue should stay open for a scoped fix or maintainer decision around account-to-agent inference and filesystem-read boundaries.

## Best Possible Solution

Keep this issue open. The best path is a least-privilege CLI fix: add an explicit `--agent` selector to `openclaw message`, thread it through `messageCommand` into `runMessageAction`, and only consider `--account` to `agentId` inference when the binding is unique and unambiguous. Add regression coverage proving a non-default agent workspace is included in `mediaLocalRoots` without adding unrelated workspaces, update the message CLI docs, and avoid broad all-agent workspace allowlisting.
