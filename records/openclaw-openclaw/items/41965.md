## Summary

Keep open. Current main still does not implement the requested WhatsApp document-forcing behavior: there is no supported `MEDIA:/path#doc` parsing, no WhatsApp `forceDocument`/`asDocument` propagation, and image media still routes through WhatsApp image payloads in both normal outbound send and auto-reply delivery. The prior implementation candidate #42044 was closed unmerged, so it remains useful prior art rather than a shipped fix.

## Best Possible Solution

Keep this issue open. The best fix is to add a shared outbound media delivery hint, either by parsing a reserved `#doc` suffix before media resolution or by carrying structured `asDocument` metadata, then wire it through WhatsApp normal sends and auto-reply delivery so image MIME can be preserved while Baileys receives a document payload. Add parser/loader tests, WhatsApp send-path tests, auto-reply tests, and WhatsApp docs. #42044 remains useful prior art but should not close this issue because it did not merge.

## Risks / Open Questions

- A future `#doc` suffix needs explicit semantics for real URL fragments, local filenames containing `#`, filename derivation, and safe local-path resolution.
- WhatsApp has both normal outbound send and auto-reply delivery paths; implementing only one would leave inconsistent user-visible behavior.
- The existing `forceDocument`/`asDocument` names are generic in plumbing but documented as Telegram-only, so broadening them should update the contract, docs, and tests together.
