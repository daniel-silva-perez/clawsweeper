## Summary

Keep open. Current main still does not implement `MEDIA:/path#doc` parsing or WhatsApp document-forcing for image media. A generic `forceDocument`/`asDocument` flag exists in outbound plumbing, but current code and docs still make it Telegram-only, and WhatsApp direct sends plus auto-replies continue to classify image MIME as image and send Baileys image payloads. Related PR #42044 is closed unmerged prior art, not shipped behavior.

## Best Possible Solution

Keep this issue open. The best fix is to add a shared outbound media delivery hint, either by parsing a reserved `#doc` suffix before media resolution or by carrying structured `asDocument`/`forceDocument` metadata, then wire it through WhatsApp direct sends and auto-reply delivery so image MIME can be preserved while Baileys receives a document payload. Add parser/loader tests, WhatsApp direct-send tests, auto-reply tests, and WhatsApp docs; use #42044 as prior art, not as a close reason.

## Risks / Open Questions

- Implementing `#doc` needs explicit semantics for real URL fragments, local filenames containing `#`, filename derivation, and safe local path resolution.
- WhatsApp has at least two relevant outbound paths, direct sends and auto-reply delivery; updating only one would leave inconsistent behavior.
- The current generic `forceDocument` flag is documented and honored as Telegram-only, so broadening it should update tool schema wording, WhatsApp docs, and tests together.
