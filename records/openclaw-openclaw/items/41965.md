## Summary

Current main still does not implement a supported `MEDIA:/path#doc`, `forceDocument`, or equivalent delivery hint for WhatsApp images. Media tokens remain plain strings, the shared loader strips only the `MEDIA:` prefix and classifies image MIME as `image`, and the WhatsApp outbound path sends `image` kind through the image payload while only Telegram consumes `forceDocument`/`asDocument`.

## Best Possible Solution

Keep this issue open. The best fix is a landed shared outbound media delivery hint, probably parsed as a reserved `#doc` suffix or structured `asDocument` metadata before media resolution, with only that reserved hint stripped from local/remote paths. Then wire the hint through normal WhatsApp send and auto-reply delivery so image MIME can still be preserved while downstream delivery uses the document payload. Add focused parser/loader tests, WhatsApp image-as-document tests, and docs. The closed unmerged #42044 is useful prior art but not a shipped fix.

## Risks / Open Questions

- Closing as implemented would be premature because the only known direct implementation candidate, #42044, is closed unmerged and current main still lacks the behavior.
- The existing generic-looking `forceDocument`/`asDocument` names are easy to mistake for a WhatsApp solution, but the current WhatsApp adapter does not consume them.
- A future implementation needs a precise reserved-suffix or structured-hint contract so legitimate URL fragments, local filenames containing `#`, filename derivation, MIME preservation, and local-read safety checks stay well-defined.
