## Summary

Keep open. Current main still lacks the requested `MEDIA:/path#doc` parsing/stripping and WhatsApp document-forcing behavior; the existing `forceDocument`/`asDocument` path is documented and wired for Telegram only, while the related PR #42044 is closed unmerged prior art rather than shipped behavior.

## Best Possible Solution

Keep this issue open. The best fix is to add a shared outbound media delivery hint, either by parsing a reserved `#doc` suffix before media resolution or by carrying structured `asDocument` metadata, then wire it through WhatsApp direct sends and auto-reply delivery so image MIME can be preserved while Baileys receives a document payload. Add parser/loader tests, WhatsApp send-path tests, auto-reply tests, and WhatsApp docs; use #42044 as prior art, not as a close reason.

## Risks / Open Questions

- A future implementation needs explicit semantics for real URL fragments, local filenames containing `#`, document filename derivation, and safe local path resolution.
- WhatsApp has at least two user-visible outbound paths here: direct sends and auto-reply delivery. Implementing only one would leave inconsistent behavior.
- The existing `forceDocument`/`asDocument` names are generic in code but Telegram-only in current docs and channel behavior, so broadening them should update the contract, docs, and tests together.
