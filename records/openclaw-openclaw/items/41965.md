## Summary

Keep open. Current main still lacks the requested `MEDIA:/path#doc` behavior and no equivalent WhatsApp document-forcing option is wired through the outbound or auto-reply paths. The prior implementation PR #42044 is closed unmerged, so it remains useful prior art rather than shipped behavior.

## Best Possible Solution

Keep this issue open. The best fix is to add a shared outbound media delivery hint, either by parsing a reserved `#doc` suffix before media resolution or by carrying structured `asDocument` metadata, then wire that hint through WhatsApp normal sends and auto-reply delivery so image MIME can be preserved while Baileys receives a document payload. Add parser/loader tests, WhatsApp send-path tests, auto-reply tests, and WhatsApp docs. #42044 remains useful prior art but should not close this issue because it did not merge.

## Risks / Open Questions

- A future `#doc` implementation needs precise semantics for real URL fragments, local filenames containing `#`, filename derivation, and safe local path resolution.
- WhatsApp has at least two user-visible outbound paths here: explicit sends and auto-reply delivery. Implementing only one would leave inconsistent behavior.
- The existing `forceDocument` / `asDocument` names are generic in code but Telegram-only in current docs and channel behavior, so broadening them should update the contract, docs, and tests together.
