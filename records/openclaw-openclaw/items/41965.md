## Summary

Keep open. Current main still does not implement a MEDIA `#doc` delivery hint or a WhatsApp force-document path for image media; the parser and loader carry only URLs, image MIME still classifies as image, and WhatsApp direct-send plus auto-reply code still route image media through image payloads. Related PR #42044 is useful prior art, but it was closed unmerged, and related #55363 covered a neighboring CLI/gateway flag gap rather than shipping this feature.

## What This Changes

Request to force selected media, especially WhatsApp images, to be delivered as document attachments via a `#doc`-style MEDIA hint.

## Best Possible Solution

Add a shared outbound media delivery hint that is parsed before media resolution, preserves the original MIME/file name, and lets WhatsApp direct-send and auto-reply choose a document payload for otherwise image media. Cover parser/loader behavior, gateway/direct delivery, WhatsApp direct send, WhatsApp auto-reply, and user-facing docs.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: current `MEDIA:/tmp/screenshot.png#doc` would be carried as a plain URL/path with `#doc`, while an image that loads successfully still classifies as `image` and follows WhatsApp image-send branches.

## Solution Assessment

Unclear for the exact `#doc` syntax, because raw URL fragments can collide with real URL/path semantics. The maintainable fix is to parse a reserved delivery hint before media resolution, preferably carrying structured `forceDocument` metadata through outbound media rather than relying only on string suffixes at send time.

Current main still lacks a way for `MEDIA:/path/to/image.png#doc` or equivalent structured metadata to force WhatsApp image media to be sent as a document.
