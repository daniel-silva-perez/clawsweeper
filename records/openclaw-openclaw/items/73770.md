## Summary

Keep this PR open. Current main still has the single-agent memory_search implementation and lacks the proposed agent scope resolver and provenance fields, but the PR is not merge-ready because its cross-agent path can abort healthy agents on one scoped error and it introduces implicit cross-agent memory access for an agent named `chief`.

## What This Changes

The PR branch adds memory_search scope resolution across configured agents, provenance fields on memory search results, related test/mocking changes, and stacked memory CLI embedding-provider registration changes.

## Best Possible Solution

Keep this as an active implementation candidate, but require explicit opt-in for any cross-agent memory scope, tolerate partial scoped-agent failures, preserve the public type contract cleanly, and add focused tests/docs for the accepted visibility model before merge.

## Reproduction Assessment

Yes. A high-confidence logic reproduction is a chief-scoped search with multiple allowed agents where one getMemoryManagerContext call returns an error; the PR's pre-loop guard returns unavailable before querying the remaining healthy managers.

## Solution Assessment

No. The direction is plausible, but this implementation is not the narrowest safe solution because it grants a default cross-agent reader and aborts partial multi-agent searches; a safer fix should fail closed and return healthy scoped results when possible.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Skip failed scoped managers instead of aborting all results:** `extensions/memory-core/src/tools.ts:237-245`
  - body: When any scoped agent's memory manager returns an error, this guard returns an unavailable payload before the loop runs. A chief search across several agents will lose valid results from healthy agents if one configured agent is temporarily broken, even though the loop below already has a per-agent `continue` path.
  - confidence: 0.91
- **[P1] Require explicit opt-in for cross-agent memory scope:** `extensions/memory-core/src/memory-permissions.ts:56-58`
  - body: Defaulting missing `chiefAgentIds` to `["chief"]` gives any agent named `chief` cross-agent memory search without configuration. Current memory and multi-agent docs describe per-agent isolation, so broad memory visibility should require an explicit `searchScope` setting or fail closed.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The PR changes a memory privacy boundary and currently grants implicit cross-agent read access to a hard-coded coordinator id.

Concerns:

- **[high] Implicit chief can read other agents' memory:** `extensions/memory-core/src/memory-permissions.ts:57`
  - body: With no `searchScope.chiefAgentIds` configured, the resolver treats `chief` as a coordinator and returns all configured agents as searchable. Memory files may contain private per-agent context, so this should require explicit operator opt-in and documentation.
  - confidence: 0.86
