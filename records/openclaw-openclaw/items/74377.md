## Summary

Keep open. Current main still has the same unresolved Telegram to embedded Pi to direct Anthropic tool propagation gap: tools are constructed in the embedded runner, but the Anthropic transport can only serialize the `context.tools` array it receives, and there is still no merged regression proving a Telegram-origin direct Anthropic run preserves a non-empty provider-facing tools array.

## What This Changes

Fix a regression where Telegram-origin Anthropic runs send `tools: []` to Anthropic even after the embedded runner computes OpenClaw tools, causing plain-text `tool_use` output instead of native tool calls.

## Best Possible Solution

Add a focused regression around the embedded runner/session stream boundary that captures the Anthropic provider context for a Telegram-origin run and asserts the expected OpenClaw tools are present, then fix the session/tool handoff so active tools cannot be cleared after `effectiveTools` is non-empty.

## Reproduction Assessment

Yes. The report includes version, channel, provider, config, and debug probes, and current source gives a clear reproduction target: Telegram inbound reply through embedded Pi into a direct Anthropic stream capture that inspects `context.tools`.

## Solution Assessment

No merged solution is proven yet. The narrow maintainable fix is to preserve the tool array through Pi session state into the Anthropic stream context, not to paper over the symptom by stripping empty arrays or adding an Anthropic-only fallback that bypasses runner policy.
