## Repair Work Prompt

Fix the Telegram/direct-Anthropic tool propagation regression reported in https://github.com/openclaw/openclaw/issues/74377. The observable bug is that `runEmbeddedAttempt` computes non-empty OpenClaw tools, but the Anthropic provider receives `context.tools` as an empty array, causing Claude to describe tool_use as plain text instead of emitting real tool_use blocks. Add a focused regression that simulates the Telegram-origin embedded runner path into the direct Anthropic stream boundary and asserts OpenClaw tool definitions, including `exec`, reach provider-facing `context.tools`. Do not solve this by only stripping `tools: []`; that is #70790's strict-provider payload concern and does not restore tool availability. Keep the fix in the generic runner/provider boundary unless a Telegram-specific adapter bug is proven.
