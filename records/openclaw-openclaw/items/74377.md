## Summary

Keep open. Current main still shows the embedded runner computing and registering an allowlist from effective tools, but there is no source or regression proof that a Telegram direct-Anthropic run reaches the provider with non-empty `context.tools`; the related open #70790 strips empty payload fields and does not restore missing tools.

## What This Changes

The issue reports that Telegram direct-Anthropic runs compute 17 tools in the embedded runner but pass an empty tools context to Anthropic, causing Claude to emit plain-text tool_use instead of real tool calls.

## Best Possible Solution

Add a focused regression for a Telegram/channel embedded direct-Anthropic run with coding tools enabled, assert the provider-facing stream context receives non-empty tool definitions, then fix the Pi session or stream handoff so effective tools cannot be lost before Anthropic sees them.
