## Summary

Current main implements the reported Telegram/direct-Anthropic tool propagation path: OpenClaw-managed tools are registered as Pi custom tools, the exact registered names are passed as the Pi session allowlist, and the session explicitly activates those names before Anthropic serializes `context.tools`. The fix is present in current main and is included in the v2026.4.29 release lineage.

## What This Changes

The issue reports that Telegram-origin embedded Pi runs with direct Anthropic compute a non-empty OpenClaw tool set but send an empty provider `context.tools` array, causing text-only tool use instead of native tool blocks.

## Best Possible Solution

Keep the current session-allowlist implementation and ask for a fresh report only if v2026.4.29 or newer still shows a non-empty `effectiveTools` set but an empty Anthropic `context.tools` array, with the provider payload and session/tool policy diagnostics attached.

## Reproduction Assessment

Yes. The original report has high-confidence live reproduction data and debug probes; current-main verification traces that same path from Telegram through embedded Pi into Anthropic and shows the former empty-context handoff is now covered by the session tool activation contract.

## Solution Assessment

Yes. Preserving the exact registered custom-tool names through Pi session creation and active-tool selection is the narrow maintainable fix; adding an Anthropic transport fallback to runner-local state would hide a broken session registry instead of fixing the shared provider boundary.
