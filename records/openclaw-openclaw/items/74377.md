## Summary

Keep open. Current main has the earlier Pi session tool-allowlist repair, and that repair is present in the reported v2026.4.22 tag, so the fresh report cannot be closed as already fixed. The remaining failure is plausible in the embedded runner handoff between OpenClaw's computed `effectiveTools`, Pi's active session tools, and the session-managed Anthropic stream path, and there is no current regression proof that Telegram plus direct Anthropic receives a non-empty provider `context.tools` array.

## What This Changes

The issue reports that Telegram sessions build a non-empty tool list but the direct Anthropic provider receives an empty `context.tools` array, causing the model to print fake tool-use text instead of emitting real tool calls.

## Best Possible Solution

Add a focused regression that simulates a Telegram/channel embedded run on a direct Anthropic model with coding tools enabled and asserts the provider-facing context receives the expected non-empty tool definitions, then fix the embedded session or stream handoff so Pi and Anthropic cannot observe an empty tool surface after `effectiveTools` is non-empty.
