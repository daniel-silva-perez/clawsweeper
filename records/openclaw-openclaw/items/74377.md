## Summary

Keep open. Current main still has the same unresolved boundary: OpenClaw computes and registers tools for the embedded Pi session, but there is no source or test proof that a Telegram/direct-Anthropic run reaches the provider with a non-empty `context.tools` array. The reported release already contained the earlier Pi session allowlist repair, so this cannot be closed as already implemented.

## What This Changes

The issue reports that Telegram-channel runs compute coding tools in the embedded runner but pass an empty `tools` array into the Anthropic provider, causing the model to emit textual tool calls instead of real tool-use blocks.

## Best Possible Solution

Add a focused regression that simulates a Telegram/channel embedded run with direct Anthropic and coding tools enabled, asserts the provider-facing stream context receives the expected non-empty tool definitions, then fix the embedded session or stream handoff so Pi and Anthropic cannot observe an empty tool surface after `effectiveTools` is non-empty.
