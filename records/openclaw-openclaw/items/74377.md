## Summary

Keep open. Current main still has the same runner/provider boundary shape described in the report: the embedded attempt computes and registers tools, while the Anthropic transport can only serialize whatever reaches it as `context.tools`; I found no merged source or test proof that Telegram/direct-Anthropic turns preserve that non-empty array through the stream handoff. The related open #70790 handles strict providers rejecting empty payloads, but it would not restore missing Anthropic tool availability.

## What This Changes

The issue asks to fix Telegram direct-Anthropic runs where the embedded runner computes tools but the provider receives `tools: []`, causing tool requests to be generated as plain text instead of real tool_use blocks.

## Best Possible Solution

Add a focused regression for the embedded Telegram/direct-Anthropic path that intercepts the stream context and asserts OpenClaw tool definitions reach `context.tools`, then fix the runner/session handoff so active Pi tools cannot be cleared after `effectiveTools` is non-empty.

## Solution Assessment

No, the related empty-payload stripping path is not the best solution for this issue. The maintainable fix is to preserve the non-empty tool surface through Pi session state and provider stream context rather than removing `tools: []` after the tools have already been lost.
