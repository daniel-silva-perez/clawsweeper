## Summary

Keep open. Current main still computes and registers OpenClaw tools for the embedded Pi session, but I did not find source, test, or release proof that a Telegram/direct-Anthropic turn reaches the provider with a non-empty `context.tools` array. The related open #70790 strips empty tool payloads for providers that reject them, but that would not restore Anthropic tool availability for this report.

## What This Changes

The issue reports that Telegram runs with direct Anthropic compute a non-empty OpenClaw tool set, but the Anthropic provider receives an empty `context.tools` array and therefore produces plain-text tool requests instead of API tool_use blocks.

## Best Possible Solution

Add a focused regression around the embedded Telegram/direct-Anthropic path that captures the stream context handed to the provider and asserts `context.tools` contains the expected OpenClaw tool definitions, then fix the runner or stream handoff so Pi active tools cannot be cleared after `effectiveTools` is non-empty.

## Reproduction Assessment

Yes. The issue provides exact version, channel, provider, config, and debug probes showing the non-empty runner tool set followed by an empty provider context; the missing repo-side piece is an automated regression that simulates that handoff on current main.

## Solution Assessment

No, the adjacent empty-payload stripping approach is not the best solution for this issue. The maintainable fix is to preserve the non-empty tool surface through Pi session state and the provider stream boundary, not to strip `tools: []` after tools have already been lost.
