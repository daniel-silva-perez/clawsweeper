## Summary

Keep open. Current main still returns ACP `sessions_spawn` acceptance after child session creation and gateway dispatch, while the child transcript user row is appended only after `acpManager.runTurn` completes. That leaves the reported accepted-but-empty `sessions_history` failure mode unresolved for stalled or never-materialized ACP/Codex turns.

## Best Possible Solution

Keep this issue open and change the ACP spawn/turn contract so an accepted child session has durable diagnostic state before `runTurn` can stall: pre-append the initial task as a child transcript user row or expose an equivalent first task/session event through `sessions_history` or status. Add a regression test where ACP dispatch is accepted but `runTurn` stalls or fails before output, proving the child session is not empty and successful post-turn persistence does not duplicate the prompt.
