## Summary

Keep open. Current main has improved ACP spawn/background-task plumbing and newer bundled acpx/codex-acp versions, but I did not find source or test evidence that an accepted ACP child session durably persists the initial user task before the ACP turn completes. The exact empty-transcript failure mode can still occur if the ACP turn stalls or never reaches the post-run transcript persistence path.

## Best Possible Solution

Keep this issue open and fix the ACP spawn contract so an accepted child session has durable diagnostic state before returning: either persist a first user/task transcript row before `runTurn`, or record an equivalent visible task/session event that `sessions_history`/status can surface when the delegated ACP turn never starts. Add a regression test where ACP dispatch is accepted but `runTurn` stalls or fails before output, proving the child session is no longer empty and the task reaches a debuggable terminal or running state.

This path records background task metadata, but it does not write the initial task text to the transcript itself.
