## Summary

Keep open. Current main has session-delta throttling and embedding-cache documentation, but once a session transcript is selected for sync the builtin memory path still rebuilds the full session entry, chunks/embeds the full content, and clears existing rows before writing. There is no durable last-synced offset or partial-session indexing path on main, and open PR #59577 remains an implementation candidate rather than shipped behavior.

## Best Possible Solution

Keep this issue open and land a reviewed incremental session-sync implementation in the memory-core plugin and memory host SDK. The fix should add durable append/offset or chunk metadata, reuse the existing embedding cache where appropriate, preserve full-reindex fallback for truncation, compaction, parser/chunking/model/config changes, and add regression coverage for append growth, unchanged transcripts, compaction, and failure recovery. PR #59577 is the current candidate/design reference, but the issue should close only after an implementation merges.

## Risks / Open Questions

- The original report mentions ChromaDB and a shorter sync cadence, while current main uses the builtin SQLite/sqlite-vec path and a 5 second dirty debounce; those details changed, but the central full rebuild/delete/reinsert behavior remains in the current code path.
- A correct fix must handle overlapping chunks, transcript truncation or compaction, parser/chunking/model/config changes, FTS/vector cleanup, and partial embedding failures without leaving stale search results.
- Open PR #59577 may need review or porting to the current `extensions/memory-core` and `src/memory-host-sdk` layout before it can land.
