## Summary

Keep open. Current main still rebuilds changed session transcripts as whole entries and writes them through the generic file-level clear-and-reinsert path; the current debounce, dirty-file tracking, delta thresholds, whole-hash check, and optional embedding cache reduce cost but do not provide durable incremental session indexing. The open related implementation candidate #59577 is the right place for maintainer review or replacement before this issue can close.

## What This Changes

The issue requests incremental indexing for growing session JSONL memory files so unchanged transcript chunks are not repeatedly deleted, re-embedded, and reinserted.

## Best Possible Solution

Land a current-main session-specific incremental indexing path that preserves valid existing chunks, stores durable append or chunk metadata, updates chunks/vector/FTS rows consistently, and falls back to a full reindex when the transcript is truncated, compacted, rechunked, or unsafe to update incrementally.

## Reproduction Assessment

Yes. Enable session memory, append enough data to a session JSONL to cross the configured session delta threshold, and the dirty-session path deterministically rebuilds the full session entry and writes through the clear-and-reinsert index path. I did not run a live sync because this review is read-only, but the source path gives a high-confidence reproduction.

## Solution Assessment

Unclear for the issue's exact `lastSyncOffset` proposal because the discussion correctly calls out chunk-overlap boundaries. The narrow maintainable path is to review, land, or replace #59577 with a current-main implementation that proves append growth, overlap, unchanged transcripts, compaction/truncation fallback, FTS-only indexing, and recovery behavior.
