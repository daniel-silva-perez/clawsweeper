## Summary

Keep open. Current main still builds and hashes a full rendered session entry for changed transcripts, then writes through the generic file-level clear-and-reinsert path. The existing debounce, dirty-file, whole-hash, delta-threshold, and embedding-cache mitigations reduce churn, but they do not implement the durable incremental session indexing requested here; #59577 remains the open implementation candidate.

## What This Changes

The issue requests incremental indexing for growing session JSONL memory files instead of repeatedly clearing and reinserting all chunks for each changed transcript.

## Best Possible Solution

Implement session-specific incremental indexing that preserves valid existing chunks, stores durable append or chunk metadata, falls back to full reindexing when a transcript is compacted or otherwise unsafe, and covers append growth, unchanged transcripts, overlap boundaries, truncation/compaction, FTS-only indexing, FTS metadata refresh, and failed-write recovery in tests.
