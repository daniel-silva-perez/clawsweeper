## Summary

Keep open. Current main still routes changed session transcripts through whole-file session rendering and the generic file-level clear-and-reinsert writer; the debounce, dirty-file, delta-threshold, whole-hash, and embedding-cache mitigations reduce frequency/cost but do not implement durable incremental session indexing. The related open implementation candidate #59577 is the right item for maintainer review rather than closing this issue now.

## What This Changes

The issue requests incremental indexing for growing session JSONL memory files so unchanged transcript chunks are not repeatedly deleted, re-embedded, and reinserted.

## Best Possible Solution

Implement and land a session-specific incremental indexing path that preserves valid existing chunks, stores durable append/chunk metadata, and falls back to a full reindex when the transcript is compacted, truncated, rechunked, or otherwise unsafe to update incrementally.

## Reproduction Assessment

Yes. The code path itself gives a high-confidence reproduction: enable session memory, append enough data to a session JSONL to pass the configured delta threshold, and the dirty session path will rebuild the full entry and write through the clear-and-reinsert index path.

## Solution Assessment

Unclear for the exact proposed `lastSyncOffset` approach because the issue discussion correctly flags chunk-overlap boundaries. Keeping this open and reviewing #59577 is the narrow maintainable path; any fix should prove append growth, overlap, unchanged transcripts, compaction/truncation fallback, FTS-only indexing, and recovery behavior.
