## Summary

Keep open. Current main still only dirty-gates which session transcripts are selected; once a changed session is selected, it is rebuilt as a full rendered entry and written through the shared clear-and-reinsert indexing path. The related #59577 PR is still open and unmerged, while #50908 was closed unmerged, so the remaining work is not implemented on main.

## What This Changes

The issue asks for session memory sync to avoid full delete-and-reinsert indexing of growing session JSONL transcripts and use an incremental, chunk-preserving sync path.

## Best Possible Solution

Add a session-specific incremental indexing path that preserves valid existing chunks, stores durable append or chunk metadata, updates vector, FTS, chunk, and file state consistently, and falls back to full reindex when a transcript is truncated, compacted, rechunked, or otherwise unsafe to update incrementally.

## Reproduction Assessment

Yes. With session memory enabled and `sources` including `sessions`, appending enough JSONL data to cross the configured session delta threshold deterministically reaches `syncSessionFiles`, which rebuilds the whole selected session entry and writes it through the clear-and-reinsert path.

## Solution Assessment

Unclear for the exact `lastSyncOffset` proposal because the discussion correctly flags chunk-overlap boundaries. The safer solution is a reviewed session-specific incremental writer backed by hash or chunk metadata with full-reindex fallbacks for unsafe changes.
