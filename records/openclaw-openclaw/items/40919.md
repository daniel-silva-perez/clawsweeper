## Summary

Keep open. Current main still dirty-gates which session transcripts are selected, but a changed selected transcript is rebuilt as a whole entry and written through the shared clear-and-reinsert index path. The related open implementation candidate #59577 is the right maintainer review or replacement path before this issue can close.

## What This Changes

The issue requests incremental session JSONL indexing so growing transcripts only process new or changed content instead of rebuilding all chunks on each sync.

## Best Possible Solution

Add a session-specific incremental indexing path that preserves valid existing chunks, stores durable append or chunk metadata, updates vector/FTS/chunk/file state consistently, and falls back to full reindex when a transcript is truncated, compacted, rechunked, or otherwise unsafe to update incrementally.

## Reproduction Assessment

Yes. With session memory enabled and `sources` including `sessions`, appending enough JSONL data to cross the configured session delta threshold deterministically reaches `syncSessionFiles`, which rebuilds the whole session entry and writes it through the clear-and-reinsert path.

## Solution Assessment

Unclear for the exact `lastSyncOffset` proposal because the discussion correctly flags chunk-overlap boundaries. The maintainable solution is a reviewed session-specific incremental writer, likely backed by hash/chunk metadata with full-reindex fallbacks for unsafe changes.
