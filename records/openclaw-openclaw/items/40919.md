## Summary

Keep open. Current main still limits which session files are selected for sync, but once a changed transcript is selected it rebuilds the full session entry and writes it through the shared clear-and-reinsert path. The open related implementation candidate #59577 is the right maintainer review or replacement path before this issue can close.

## What This Changes

Request incremental session-memory indexing for growing JSONL transcripts instead of repeatedly deleting and reinserting all chunks for a changed session file.

## Best Possible Solution

Add a session-specific incremental indexing path that preserves valid existing chunks, stores durable append or chunk metadata, updates vector/FTS/chunk/file state consistently, and falls back to full reindex when a transcript is truncated, compacted, rechunked, or otherwise unsafe to update incrementally.

## Reproduction Assessment

Yes. With session memory enabled and `sources` including `sessions`, append enough JSONL data to pass the configured session delta threshold; the current source path deterministically rebuilds the full session entry and writes it through the clear-and-reinsert index writer.

## Solution Assessment

Unclear for the exact `lastSyncOffset` proposal because the discussion correctly flags chunk-overlap boundaries. The maintainable solution is a reviewed session-specific incremental writer, likely hash/chunk metadata backed, with full-reindex fallbacks for unsafe changes.
