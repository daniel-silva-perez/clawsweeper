## Summary

Keep this PR open. Current main still validates omitted plugin config as `{}` in both non-activating manifest snapshot and CLI metadata paths, so the linked false-positive validation bug is not implemented on main; the branch is a focused candidate, but the intentional skipped-validation behavior still needs maintainer security/product review before merge.

## What This Changes

The PR changes the plugin loader to skip config schema validation when config is `undefined` during non-activating snapshot/CLI metadata loads, adds regression tests for that behavior, and includes two SwiftFormat-only macOS brace edits.

## Best Possible Solution

Keep this PR open for maintainer review and either land it or an equivalent loader change that distinguishes omitted config in non-activating metadata/snapshot loads from real activation, keeps activation and explicit-config validation strict, preserves normalized/defaulted `api.pluginConfig`, and explicitly resolves the validation-bypass concern before merge.

## Reproduction Assessment

Yes. A focused reproduction is a native plugin with a `configSchema.required` field and no `plugins.entries.<id>.config` in a non-activating `loadOpenClawPlugins({ activate: false, loadModules: false })` or `loadOpenClawPluginCliRegistry` call; current main validates `undefined` as `{}` and reports invalid config.

## Solution Assessment

Unclear pending maintainer security review. Functionally, the PR is the narrowest source-level fix for the false-positive bug, but maintainers should confirm that discovery/CLI metadata registration is safe when required runtime config is omitted and represented as `{}` plus schema defaults.
