## Summary

Keep this issue open. Current main still wires the Control UI/WebChat New Session button through `/new`, and the gateway reset path rotates the session id under the same session key while archiving the previous transcript. Because the session switcher is populated from current `sessions.list` rows, the old reset archive is not exposed as a separate selectable chat. The discussion also correctly moved away from the unsupported workspace explanation toward session-store/history behavior; #49287 was closed in favor of this issue, while #70330 is adjacent rather than a shipped fix for this specific New Session/switcher bug.

## Best Possible Solution

Keep the issue open. The best fix is to change the Control UI/WebChat New Session flow so it creates and selects a distinct visible session, likely through `sessions.create`, or to add a first-class reset archive/history recovery surface if `/new` continues to mean reset-current-key. Add regression coverage that after New Session the prior chat remains reachable from the session switcher and `chat.history` loads for both sessions.
