## Summary

Keep this issue open. Current main still routes the Control UI/WebChat New Session action through the `/new` reset path, which replaces the active session entry and archives the prior transcript instead of leaving it as a selectable chat. The latest issue comment confirms this still reproduces on a fresh install, while the active fix candidate is open and unmerged.

## What This Changes

Report that Control UI/WebChat New Session makes the previous chat inaccessible from the UI instead of creating a separate switchable session.

## Best Possible Solution

Review and land #73725 or an equivalent focused fix that makes WebChat New Session create and select a distinct visible session through `sessions.create`, while keeping explicit reset behavior separate. If `/new` remains a reset-current-key command, add a first-class archive/history recovery surface so prior transcripts remain reachable.
