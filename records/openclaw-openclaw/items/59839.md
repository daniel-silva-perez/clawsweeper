## Summary

Keep this issue open because the checked-out main still routes the visible Control UI/WebChat New session button through the reset-backed `/new` chat path, which replaces the active session entry and archives the previous transcript instead of creating a separate selectable chat. The issue is already paired with focused maintainer PR #73725, so this should not be swept closed or duplicated by another repair job.

## What This Changes

The issue requests that Control UI/WebChat New session start a fresh chat while keeping the previous chat visible, switchable, and recoverable from the UI.

## Best Possible Solution

Land #73725 or an equivalent focused change that makes the visible WebChat New session/New Chat affordance call `sessions.create`, switch to the returned dashboard session key, refresh the session selector, and leave `/new` or `/reset` as explicit reset behavior with clear recovery semantics for reset archives.
