## Summary

Keep this issue open because current main still wires the visible Control UI/WebChat New Session action to the `/new` reset path, which replaces the active session-store entry and archives the previous transcript rather than creating a separate selectable session. The issue is also already covered by open maintainer PR #73725, so a duplicate repair job would be the wrong next step.

## What This Changes

Control UI/WebChat New Session should create and select a new visible chat session while keeping the previous chat accessible from session switching/history UI.

## Best Possible Solution

Land #73725 or an equivalent focused change that makes the WebChat New Session/New Chat affordance call `sessions.create`, switch to the returned dashboard session, refresh session options, and keep `/new` or `/reset` as explicit reset behavior. If resets continue archiving transcripts under the same key, add a first-class UI recovery path for reset archives.
