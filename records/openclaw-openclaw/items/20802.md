## Summary

Keep open. Current main still exposes `llm_input`, `llm_output`, and `after_tool_call` as void/observe-only plugin hooks, and the embedded runner does not await or apply returned prompt, systemPrompt, block, blockReason, assistantTexts, or tool-result replacement values. The open #61485 overlaps the `llm_input`/`llm_output` portion, but it does not cover this PR's broader adapter-side `after_tool_call` result-mutation proposal. Current main does have bundled-only runtime-neutral tool-result middleware for pre-model tool-output rewriting, but docs explicitly restrict that high-trust seam from external plugins, so it does not make this PR fully implemented or obsolete.

## Best Possible Solution

Keep this PR open unless maintainers explicitly reject the broader direction or land a canonical replacement. The best path is to review or land a focused mutable `llm_input`/`llm_output` implementation such as #61485, then split or redesign adapter-side `after_tool_call` result mutation as a separate plugin API decision that preserves exactly-once hook dispatch, defines the trust boundary, and validates replacement tool-result shape before model exposure.

## Risks / Open Questions

- The PR overlaps #61485 for `llm_input`/`llm_output`; maintainers should avoid landing two independent hook-contract implementations.
- Adapter-side `after_tool_call` mutation conflicts with current exactly-once dispatch tests unless the API defines how adapter mutation composes with subscription-handler observation and persisted tool results.
- The PR's proposed `after_tool_call` replacement result is typed as `unknown` in the provided diff and cast to `AgentToolResult<unknown>` in the adapter path, unlike current tool-result middleware, which validates shape and size.
- Prompt and `systemPrompt` mutation is security-sensitive and should be reviewed against `allowPromptInjection=false`, audit logging, image re-resolution, failover/block semantics, payload cloning, and original-content leakage.
