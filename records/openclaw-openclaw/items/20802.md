## Summary

Keep this PR open. Current main still keeps `llm_input`, `llm_output`, and `after_tool_call` as observation-only void hooks, and the PR's broader `after_tool_call` result-mutation scope conflicts with current exactly-once tool-hook ownership tests. The narrower mutable LLM-hook work overlaps open #61485, but that does not cover this PR's full tool-result mutation proposal.

## Best Possible Solution

Keep this PR open for maintainer follow-up. The best path is to review or land one focused, hardened mutable `llm_input`/`llm_output` implementation, likely through #61485 or an equivalent patch, then split `after_tool_call` result mutation into a separate plugin API design with exactly-once dispatch ownership, replacement-result validation, prompt-policy constraints, docs, and targeted regression tests.
