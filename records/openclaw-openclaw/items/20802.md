## Summary

Keep this PR open. Current main still exposes `llm_input`, `llm_output`, and `after_tool_call` as observation-only hooks, and the embedded runner fires `llm_input`/`llm_output` without awaiting or consuming hook results. The overlapping open PR #61485 covers only the `llm_input`/`llm_output` subset, while this PR also proposes `after_tool_call` tool-result mutation, which remains an unresolved plugin API/security design surface.

## Best Possible Solution

Keep this PR open for maintainer follow-up. The best path is to review or land one focused, hardened mutable `llm_input`/`llm_output` implementation, likely via #61485 or an equivalent patch, then split `after_tool_call` result mutation into a separate plugin API design with dispatch ownership, result validation, prompt-policy constraints, docs, and targeted regression tests.
