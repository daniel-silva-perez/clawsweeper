## Summary

Keep this PR open. Current main still keeps llm_input, llm_output, and after_tool_call as observation-only void hooks; the PR is not implemented on main. The related open PR #61485 overlaps the mutable llm_input/llm_output part, but it does not cover this PR's after_tool_call result-mutation scope, which remains a separate security-sensitive hook ownership/design question.

## Best Possible Solution

Keep this PR open for maintainer follow-up. The best path is to review or land one focused, hardened mutable llm_input/llm_output implementation, likely via #61485 or an equivalent patch, then split after_tool_call result mutation into a separate plugin API design with exactly-once dispatch ownership, replacement-result validation, prompt-policy constraints, docs, and targeted regression tests.
