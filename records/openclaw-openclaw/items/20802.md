## Summary

Keep open for maintainer review. Current main still has observation-only `llm_input`, `llm_output`, and `after_tool_call`, so the requested capability is not implemented, but this PR is not merge-ready because the proposed branch has blocking correctness and security-policy issues and overlaps a narrower open LLM-hook PR.

## What This Changes

This PR changes plugin hook contracts so `llm_input`, `llm_output`, and adapter-side `after_tool_call` can return modifications, applies those results in embedded agent/tool paths, and updates hook tests.

## Best Possible Solution

Land or supersede this with one focused, hardened mutable `llm_input`/`llm_output` implementation, then handle `after_tool_call` result mutation as a separate plugin API design with exact-once dispatch, result validation, policy constraints, docs, and regression tests.

## Reproduction Assessment

Yes. Static inspection of current main reproduces the remaining gap, and static inspection of the PR diff plus the existing exact-once regression test gives high-confidence reproduction paths for the proposed patch bugs; I did not run tests because this was a read-only review.

## Solution Assessment

No. The direction is useful, but this branch is broader than the safest fix and introduces prompt-policy, prompt-submission, output-redaction, and after-tool-call dispatch problems; the safer path is a focused hardened LLM-hook patch plus separate after-tool-call API review.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Apply prompt overrides before building the submission:** `src/agents/pi-embedded-runner/run/attempt.ts:2840`
  - body: `llmInputResult.prompt` only updates `effectivePrompt`, but this happens after `promptSubmission` and prompt images have already been computed. The later LLM call still submits `promptSubmission.prompt`, so a masking plugin can return a redacted prompt while the original prompt is sent to the provider. Move the hook before prompt submission/image resolution or recompute both after applying the result.
  - confidence: 0.93
- **[P1] Enforce prompt-injection policy for llm_input:** `src/plugins/types.ts:1561-1566`
  - body: The PR adds `prompt` and `systemPrompt` mutations to `llm_input`, but the existing `allowPromptInjection=false` policy only constrains the current prompt-injection hook list. A plugin with conversation access but prompt injection disabled could still rewrite the provider prompt/system prompt through this new result type. Add policy coverage or strip those fields before applying the hook result.
  - confidence: 0.9
- **[P2] Keep after_tool_call from firing twice:** `src/agents/pi-tool-definition-adapter.ts:174-184`
  - body: The adapter now calls `runAfterToolCall`, while the subscription handler still invokes the same hook through the new void runner. In embedded runs both paths are active, and current regression coverage explicitly requires one `after_tool_call` per tool call to avoid duplicated plugin side effects. Redesign dispatch ownership before adding adapter-side result mutation.
  - confidence: 0.88
- **[P2] Remove original assistant content when overriding output:** `src/agents/pi-embedded-runner/run/attempt.ts:3175-3177`
  - body: When `llm_output` returns replacement `assistantTexts`, the original `lastAssistant` and message snapshots are still returned downstream. That leaves the unredacted assistant content/reasoning/error fields available to later payload, trajectory, or retry paths even though the hook requested replacement output. Clear or rebuild the assistant artifacts consistently with the replacement.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff introduces prompt and response mutation surfaces without fully carrying existing prompt-injection and redaction safeguards forward.

Concerns:

- **[high] Mutable llm_input bypasses prompt-injection policy:** `src/plugins/types.ts:1561`
  - body: `llm_input` gains `prompt` and `systemPrompt` result fields, but current policy only treats `agent_turn_prepare`, `before_prompt_build`, `before_agent_start`, and `heartbeat_prompt_contribution` as prompt-injection hooks. Without a matching registry constraint, a plugin configured with `allowPromptInjection=false` can still rewrite model instructions through the new hook surface.
  - confidence: 0.9
- **[medium] Output override can leave original assistant text reachable:** `src/agents/pi-embedded-runner/run/attempt.ts:3176`
  - body: The proposed `llm_output` override changes `assistantTexts` but leaves the original `lastAssistant` object and snapshots intact, so sensitive text that a plugin intended to redact can remain available to downstream handling.
  - confidence: 0.78
