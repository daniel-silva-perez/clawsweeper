## Summary

Keep this PR open. Current main still treats `llm_input`, `llm_output`, and `after_tool_call` as observation-only hooks, and the only open overlapping PR in the provided context covers the `llm_input`/`llm_output` portion, not this PR's adapter-side `after_tool_call` result-mutation proposal. The PR also changes prompt, system prompt, and tool-result trust boundaries, so it needs maintainer API/security review rather than conservative cleanup closure.

## Best Possible Solution

Keep this PR open unless maintainers explicitly reject the broader direction. The likely path is to review or land a focused mutable `llm_input`/`llm_output` implementation such as #61485, then split adapter-side `after_tool_call` result mutation into a separate plugin API design with exactly-once dispatch, validated replacement result shape, docs, and targeted regression tests.
