## Summary

Keep this PR open for maintainer review rather than cleanup-closing it. Current main still lacks the mutable LLM hook behavior, but this branch has concrete correctness and security issues, overlaps the narrower open #61485 path for `llm_input`/`llm_output`, and mixes `after_tool_call` result mutation into a surface that still needs explicit API ownership design.

## What This Changes

This PR changes plugin hook contracts and embedded agent/tool call sites so `llm_input`, `llm_output`, and adapter-side `after_tool_call` can return modifications, with runner and hook tests updated around those result values.

## Best Possible Solution

Ship a focused, hardened mutable `llm_input`/`llm_output` contract first, likely by reviewing #61485 or an equivalent patch, then handle pre-model tool-result mutation through a separately designed plugin API or trusted middleware surface with result validation, exact-once dispatch rules, docs, and regression tests.

## Reproduction Assessment

Yes. Static inspection of current main reproduces the missing mutable-hook capability, and static inspection of the PR head gives high-confidence reproduction paths for the proposed patch bugs; the existing exact-once after-tool-call tests are the relevant regression guard.

## Solution Assessment

No. The direction is useful, but this branch is broader than the narrowest maintainable fix and currently changes prompt/output and tool-result mutation before prompt-policy, derived prompt state, redaction cleanup, and dispatch ownership are resolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Recompute prompt images after hook rewrites prompt:** `src/agents/pi-embedded-runner/run/attempt.ts:2840`
  - body: `llmInputResult.prompt` is applied after `detectAndLoadPromptImages` has already built `imageResult`. If a redaction plugin removes or changes image references, the old images are still sent with the rewritten prompt, so move the hook before detection or recompute and roll back the image payload after applying the hook result.
  - confidence: 0.9
- **[P1] Apply prompt-injection policy to llm_input results:** `src/plugins/types.ts:1563-1566`
  - body: Adding `prompt` and `systemPrompt` result fields makes `llm_input` prompt-mutating, but `allowPromptInjection=false` only constrains the existing prompt-injection hook names. A plugin with conversation access but prompt injection disabled can still rewrite model instructions through this result type.
  - confidence: 0.9
- **[P2] Keep after_tool_call from dispatching twice:** `src/agents/pi-tool-definition-adapter.ts:177-184`
  - body: The adapter now dispatches `runAfterToolCall`, while the subscribe handler still dispatches the same plugin hook through `runAfterToolCallVoid`. In embedded runs both paths are active, so plugin side effects fire twice and the existing exact-once regression coverage is violated.
  - confidence: 0.88
- **[P2] Strip original assistant data when replacing output:** `src/agents/pi-embedded-runner/run/attempt.ts:3177-3179`
  - body: When `llm_output` returns replacement `assistantTexts`, the original `lastAssistant` and message snapshots are still returned downstream. That can leave unredacted assistant content, reasoning, or error metadata reachable after a hook requested replacement output.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff introduces prompt and response mutation surfaces without fully carrying existing prompt-injection and redaction safeguards forward.

Concerns:

- **[high] Mutable llm_input bypasses prompt-injection policy:** `src/plugins/types.ts:1563`
  - body: `llm_input` gains `prompt` and `systemPrompt` result fields, but current policy only treats existing prompt-building hooks as prompt-injection surfaces. Without matching registry constraints, a plugin configured with `allowPromptInjection=false` can still rewrite model instructions.
  - confidence: 0.9
- **[high] Prompt override can retain stale prompt artifacts:** `src/agents/pi-embedded-runner/run/attempt.ts:2840`
  - body: The PR applies prompt overrides after prompt-local image detection. For a redaction plugin, stale image artifacts can still be sent unless the implementation moves the hook earlier or recomputes all derived prompt payload state.
  - confidence: 0.9
- **[medium] Output override can leave original assistant text reachable:** `src/agents/pi-embedded-runner/run/attempt.ts:3177`
  - body: The proposed `llm_output` override changes `assistantTexts` but leaves the original `lastAssistant` object and message snapshots intact, so sensitive text that a plugin intended to redact can remain available to downstream handling.
  - confidence: 0.78
