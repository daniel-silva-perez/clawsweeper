## Summary

Keep this PR open. Current main still treats `llm_input`, `llm_output`, and `after_tool_call` as observation-only hooks: their handler contracts return `void`, the runner delegates all three to `runVoidHook`, and the embedded run path invokes LLM hooks without awaiting or applying returned prompt/output changes. The overlapping #61485 is still open and narrower because it targets `llm_input`/`llm_output`, while this PR also proposes adapter-side `after_tool_call` result mutation. That tool-result mutation conflicts with the current exactly-once ownership tests and needs maintainer API/security review or a split.

## Best Possible Solution

Keep this PR open for maintainer follow-up. The best path is to review or land one focused mutable `llm_input`/`llm_output` implementation, likely through #61485 or an equivalent hardened patch, then split adapter-side `after_tool_call` result mutation into a separate plugin API design with exactly-once dispatch ownership, replacement-result validation, prompt-policy constraints, docs, and targeted regression tests.

## Risks / Open Questions

- Landing this PR as-is would expand plugin authority over system prompts, LLM payloads, assistant text, and model-visible tool results; that needs explicit maintainer API/security review and public docs.
- Adapter-side `after_tool_call` result mutation conflicts with current subscription-handler ownership and the exactly-once dispatch regression contract unless the ownership model is redesigned.
- The `llm_input`/`llm_output` portion overlaps with open #61485, so maintainers should avoid two divergent mutable-hook implementations.
- The proposed tool-result replacement path accepts `unknown` and casts it to an agent tool result, creating malformed-result risk without validation.
