## Summary

Keep open for maintainer review. Current main still leaves the requested mutable LLM/tool-result hook capability unimplemented, but the PR branch is not merge-ready: prior discussion already identified overlap with #20426/#37128 and the narrower open #61485 path, and the current diff still has blocking prompt submission, prompt-policy, output-redaction, and after-tool-call dispatch issues.

## What This Changes

This PR changes plugin hook contracts so `llm_input`, `llm_output`, and adapter-side `after_tool_call` can return modifications, applies those results in embedded agent/tool paths, and updates hook tests.

## Best Possible Solution

Ship a focused, hardened mutable `llm_input`/`llm_output` contract first, with prompt-policy enforcement, image/prompt recomputation, output artifact stripping, docs, and contract tests; handle pre-model tool-result mutation separately through a clearly owned plugin API or trusted middleware design.

## Reproduction Assessment

Yes. Static inspection of current main reproduces the missing mutable-hook capability, and static inspection of this PR diff plus existing exact-once tests gives high-confidence reproduction paths for the proposed patch bugs; tests were not run because this review was read-only.

## Solution Assessment

No. The direction is useful, but this branch is broader than the narrowest maintainable fix and currently mixes LLM prompt/output mutation with `after_tool_call` result mutation before the prompt-policy and dispatch ownership questions are resolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Apply prompt overrides before building the submission:** `src/agents/pi-embedded-runner/run/attempt.ts:2840`
  - body: `llmInputResult.prompt` updates `effectivePrompt` only after `promptSubmission` and prompt images have already been computed. The later model call still submits `promptSubmission.prompt`, so a masking plugin can return a redacted prompt while the original prompt is sent to the provider. Move the hook before prompt submission/image resolution or recompute both after applying the result.
  - confidence: 0.93
- **[P1] Enforce prompt-injection policy for llm_input:** `src/plugins/types.ts:1561-1566`
  - body: The PR adds `prompt` and `systemPrompt` mutations to `llm_input`, but the existing `allowPromptInjection=false` policy only constrains the current prompt-injection hook list. A plugin with conversation access but prompt injection disabled could still rewrite model instructions through this result type. Add policy coverage or strip those fields before applying the hook result.
  - confidence: 0.9
- **[P2] Keep after_tool_call from firing twice:** `src/agents/pi-tool-definition-adapter.ts:174-184`
  - body: The adapter now calls `runAfterToolCall`, while the subscription handler still invokes the same hook through the new void runner. In embedded runs both paths are active, and current regression coverage explicitly requires one `after_tool_call` per tool call to avoid duplicated plugin side effects. Redesign dispatch ownership before adding adapter-side result mutation.
  - confidence: 0.88
- **[P2] Remove original assistant content when overriding output:** `src/agents/pi-embedded-runner/run/attempt.ts:3175-3177`
  - body: When `llm_output` returns replacement `assistantTexts`, the original `lastAssistant` and message snapshots are still returned downstream. That leaves the unredacted assistant content, reasoning, or error fields available to later payload, trajectory, or retry paths even though the hook requested replacement output. Clear or rebuild the assistant artifacts consistently with the replacement.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff introduces prompt and response mutation surfaces without fully carrying existing prompt-injection and redaction safeguards forward.

Concerns:

- **[high] Mutable llm_input bypasses prompt-injection policy:** `src/plugins/types.ts:1561`
  - body: `llm_input` gains `prompt` and `systemPrompt` result fields, but current policy only treats `agent_turn_prepare`, `before_prompt_build`, `before_agent_start`, and `heartbeat_prompt_contribution` as prompt-injection hooks. Without matching registry constraints, a plugin configured with `allowPromptInjection=false` can still rewrite model instructions through the new hook surface.
  - confidence: 0.9
- **[high] Prompt override can still leak the original prompt:** `src/agents/pi-embedded-runner/run/attempt.ts:2840`
  - body: The PR applies prompt overrides after prompt submission and image resolution have already been built. For a PII redaction plugin, that means the unredacted prompt can still be the provider payload even though the hook returned a masked replacement.
  - confidence: 0.93
- **[medium] Output override can leave original assistant text reachable:** `src/agents/pi-embedded-runner/run/attempt.ts:3176`
  - body: The proposed `llm_output` override changes `assistantTexts` but leaves the original `lastAssistant` object and snapshots intact, so sensitive text that a plugin intended to redact can remain available to downstream handling.
  - confidence: 0.78
