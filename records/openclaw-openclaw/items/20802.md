## Summary

Keep this PR open for maintainer review. Current main still lacks mutable `llm_input`/`llm_output` behavior, but this branch remains too broad and has blocking prompt-security, derived-payload, output-redaction, and `after_tool_call` dispatch issues; the open #61485 branch appears to be the narrower overlap for the LLM hook portion.

## What This Changes

The PR changes plugin hook result types, hook runners, and embedded agent/tool call paths so `llm_input`, `llm_output`, and adapter-side `after_tool_call` can return blocking or mutation results, with hook runner and adapter tests added.

## Best Possible Solution

Ship a focused mutable `llm_input`/`llm_output` contract with prompt-policy enforcement, payload cloning, prompt image re-resolution or rollback, and output redaction cleanup; then design pre-model tool-result mutation separately with exact-once dispatch ownership, result validation, docs, and regression tests.

## Reproduction Assessment

Yes. Static inspection of current main reproduces the missing mutable-hook capability, and static inspection of the PR diff gives high-confidence reproduction paths for the proposed patch bugs against the existing prompt-image, prompt-policy, output-return, and exact-once `after_tool_call` tests.

## Solution Assessment

No. The direction is useful, but this PR is not the narrowest maintainable solution because it combines LLM prompt/output mutation with tool-result mutation before the security policy, derived prompt state, redaction cleanup, and dispatch ownership are resolved.

## Review Findings

Overall confidence: 0.9

Full review comments:

- **[P1] Recompute prompt images after llm_input rewrites prompt:** `src/agents/pi-embedded-runner/run/attempt.ts:2840-2841`
  - body: `llmInputResult.prompt` is applied after `detectAndLoadPromptImages` has already built `imageResult`. If a redaction plugin removes or changes image references, the old images are still sent with the rewritten prompt, so the hook needs to run before image detection or recompute and roll back the image payload after applying the hook result.
  - confidence: 0.9
- **[P1] Enforce prompt policy on llm_input rewrites:** `src/plugins/types.ts:1561-1563`
  - body: Adding `prompt` and `systemPrompt` result fields makes `llm_input` prompt-mutating, but the existing `allowPromptInjection=false` policy only constrains the current prompt-injection hook names. A plugin with conversation access but prompt injection disabled could still rewrite model instructions through this result type.
  - confidence: 0.9
- **[P2] Keep after_tool_call from firing twice:** `src/agents/pi-tool-definition-adapter.ts:177-184`
  - body: The adapter now dispatches `runAfterToolCall`, while the subscribe handler still dispatches the same plugin hook through `runAfterToolCallVoid`. In embedded runs both paths are active, so plugin side effects can fire twice and the current exact-once regression coverage is violated.
  - confidence: 0.88
- **[P2] Strip original assistant data on output replacement:** `src/agents/pi-embedded-runner/run/attempt.ts:3177-3179`
  - body: When `llm_output` returns replacement `assistantTexts`, the original `lastAssistant` and message snapshots are still returned downstream. That can leave unredacted assistant content, reasoning, or error metadata reachable after a hook requested replacement output.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff introduces prompt and response mutation surfaces without fully carrying existing prompt-injection and redaction safeguards forward.

Concerns:

- **[high] Mutable llm_input bypasses prompt-injection policy:** `src/plugins/types.ts:1561`
  - body: `llm_input` gains `prompt` and `systemPrompt` result fields, but current policy only treats existing prompt-building hooks as prompt-injection surfaces. Without matching registry constraints, a plugin configured with `allowPromptInjection=false` can still rewrite model instructions.
  - confidence: 0.9
- **[high] Prompt override can retain stale prompt artifacts:** `src/agents/pi-embedded-runner/run/attempt.ts:2840`
  - body: The PR applies prompt overrides after prompt-local image detection. For a redaction plugin, stale image artifacts can still be sent unless the implementation moves the hook earlier or recomputes all derived prompt payload state.
  - confidence: 0.9
- **[medium] Output override can leave original assistant text reachable:** `src/agents/pi-embedded-runner/run/attempt.ts:3177`
  - body: The proposed `llm_output` override changes `assistantTexts` but leaves the original `lastAssistant` object and message snapshots intact, so sensitive text that a plugin intended to redact can remain available to downstream handling.
  - confidence: 0.78
