## Summary

Keep this PR open. Current main still treats `llm_input`, `llm_output`, and `after_tool_call` as observation-only hooks: the public hook handler map returns `void`, the runner delegates all three to `runVoidHook`, and the embedded runner fires LLM hooks without awaiting or applying returned data. The PR is not implemented on main and still contains useful work, but the broader `after_tool_call` result-mutation portion conflicts with current exactly-once ownership tests and needs maintainer API/security review or a split from the narrower mutable LLM hook work tracked by #61485.

## Best Possible Solution

Keep this PR open for maintainer follow-up. The best path is to review or land one focused, hardened mutable `llm_input`/`llm_output` implementation, likely through #61485 or an equivalent patch, then split adapter-side `after_tool_call` result mutation into a separate plugin API design with exactly-once dispatch ownership, replacement-result validation, prompt-policy constraints, docs, and targeted regression tests.
