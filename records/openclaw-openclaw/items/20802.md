## Summary

Keep this PR open. Current main still treats llm_input, llm_output, and after_tool_call as observation-only void hooks, and the existing after_tool_call tests intentionally keep dispatch owned by the subscription handler rather than the adapter. The related open PR #61485 overlaps the llm_input/llm_output portion, but this PR still includes a broader after_tool_call result-mutation design that is not implemented on main and needs maintainer review.

## Best Possible Solution

Keep this PR open for maintainer follow-up. The best path is to review or land one focused, hardened mutable llm_input/llm_output implementation, likely via #61485 or an equivalent patch, then split after_tool_call result mutation into a separate plugin API design with exactly-once dispatch ownership, replacement-result validation, prompt-policy constraints, docs, and targeted regression tests.
