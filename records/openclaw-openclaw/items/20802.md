## Summary

Keep this PR open. Current main at bdfb408ce6eeb724a638c106104dbbcae50b612c still treats `llm_input`, `llm_output`, and `after_tool_call` as observation-only hooks, while this PR proposes real hook contract and runtime behavior changes that are not implemented on main. The broader `after_tool_call` result-mutation portion also conflicts with current exactly-once ownership tests and still needs maintainer API/security review or a split from the narrower mutable LLM hook work.

## Best Possible Solution

Keep this PR open for maintainer follow-up. The best path is to review or land one focused, hardened mutable `llm_input`/`llm_output` implementation, likely through #61485 or an equivalent patch, then split adapter-side `after_tool_call` result mutation into a separate plugin API design with exactly-once dispatch ownership, replacement-result validation, prompt-policy constraints, docs, and targeted regression tests.

## Risks / Open Questions

- Landing this PR as-is would expand plugin authority over system prompts, prompts, assistant text, and model-visible tool results; that needs explicit maintainer API/security review, permission/policy constraints, and public docs.
- Adapter-side `after_tool_call` result mutation conflicts with the current subscription-handler ownership model and exactly-once dispatch regression contract unless the ownership model is redesigned.
- The proposed PR diff accepts a replacement tool result as `unknown` and casts it to `AgentToolResult<unknown>`, creating malformed-result risk without validation.
- The `llm_input`/`llm_output` part overlaps with open #61485, so maintainers should avoid two divergent mutable-hook implementations.
