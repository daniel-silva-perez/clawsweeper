## Summary

Keep this PR open. Current main still treats `llm_input`, `llm_output`, and `after_tool_call` as observation-only hooks, and the remaining adapter-side `after_tool_call` result-mutation work is not implemented by main or fully covered by the narrower open #61485 work. The PR also changes model-visible prompt/output and tool-result trust boundaries, so it needs maintainer API/security review or a split rather than conservative cleanup closure.

## Best Possible Solution

Keep this PR open for maintainer follow-up. The likely path is to review or land the focused mutable `llm_input`/`llm_output` implementation in #61485, then split adapter-side `after_tool_call` result mutation into a separate plugin API design with exactly-once dispatch, validated replacement result shape, documentation, and targeted regression tests.

- The `llm_input`/`llm_output` portion overlaps with open #61485, so maintainers should avoid two divergent hook-contract implementations.
