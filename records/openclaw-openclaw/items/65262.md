## Summary

Keep open. Current main still has the reported CLI diagnostic path for `openclaw commands` when `plugins.allow` is configured, and `/commands` is still registered and documented as a chat slash command rather than a CLI/plugin command. The PR targets the right bug, but the branch should be refreshed against current main and the remaining display-text control-flow concern should be addressed before merge.

## What This Changes

The PR changes CLI missing-command diagnostics and tests so `commands` can be described as a built-in runtime slash command, preserves plugin allow-list diagnostics, and includes historical Matrix ACP test expectation updates.

## Best Possible Solution

Land a focused CLI policy fix in the current `run-main-policy`/`run-main` flow: classify built-in chat slash commands from a structured registry or typed diagnostic result, preserve plugin allow/disable messages for real plugin command aliases, keep Commander unknown-command UX intact, and cover `commands` in both policy and exit-path tests.

## Reproduction Assessment

Yes. The source path is high-confidence: with a non-empty `plugins.allow` that does not include `commands`, current main reaches the generic `plugins.allow` exclusion branch before Commander parsing, while the chat registry and docs identify `/commands` as a built-in slash command.

## Solution Assessment

No. The PR addresses the right user-visible bug, but branching on diagnostic text is brittle and the branch is stale against current main's refactored policy module; a typed diagnostic/code path is the safer maintainable solution.
