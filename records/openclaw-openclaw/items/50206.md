## Summary

Keep this PR open. Current main still lacks opt-in per-agent LLM request headers, and the submitted patch is a plausible narrow implementation, but it has a blocking provider-alias lookup bug and needs privacy wording before it is merge-ready.

## What This Changes

The PR adds an opt-in `injectAgentHeader` model-provider config field, schema/help/label entries, runner stream wrapping, test support, and tests to send `X-OpenClaw-Agent` on outgoing LLM requests.

## Best Possible Solution

Land a narrow version that keeps the feature explicitly opt-in, resolves provider config through the existing normalized-provider helper, adds alias/case regression coverage plus enabled/disabled/missing-agent cases, and documents the trusted-provider privacy implication. If maintainers prefer this outside core, first expose `agentId` through the provider hook context so a plugin can implement it cleanly.

## Reproduction Assessment

Yes. The current-main absence is reproducible by checking the strict provider schema and runner wrapper path, and the PR bug is reproducible with a config keyed by an alias such as `z-ai` while runtime uses the normalized-equivalent provider id `z.ai`.

## Solution Assessment

No. The proposed core shape is close, but the direct provider-map lookup is not the narrowest reliable implementation because it bypasses the normalized provider resolution already used by model config loading.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Resolve provider config through normalized lookup:** `src/agents/pi-embedded-runner/extra-params.ts:328`
  - body: The new check reads `cfg.models.providers[provider]` directly. Existing model resolution accepts normalized aliases like `z.ai`/`z-ai`, so a config that successfully selects a model through an alias will never enable `injectAgentHeader`. Please use the normalized provider helper and add alias coverage.
  - confidence: 0.92
- **[P3] Document the agent-id disclosure:** `src/config/schema.help.ts:708-709`
  - body: The new help text says this is useful for local proxies but does not warn that enabling it sends agent identifiers to whichever provider or proxy is configured. Please call out trusted-proxy/operator use so users do not enable it on third-party providers unknowingly.
  - confidence: 0.8
