## Summary

Keep open. Current main still lacks the requested opt-in per-agent provider header, and this PR is a useful implementation attempt, but it should not merge as-is because the new provider lookup bypasses normalized provider matching and the user-facing help omits the privacy/trusted-proxy caveat.

## What This Changes

The PR adds an opt-in `injectAgentHeader` model-provider config flag, schema/help/label entries, a runner stream wrapper, test support, and tests to send `X-OpenClaw-Agent` with the active agent ID on outgoing model requests.

## Best Possible Solution

Land a narrow version that stays explicitly opt-in, resolves provider config through the normalized provider path, adds alias/case regression coverage, and documents that this is intended for trusted local or operator-controlled proxies. If maintainers prefer plugin ownership, first expose `agentId` through the provider hook context so a plugin can implement it cleanly.

## Reproduction Assessment

Yes. The current-main absence is reproducible by checking the strict provider schema and runner wrapper path, and the PR bug is reproducible by configuring a provider under an alias such as `z-ai` while runtime resolution uses the normalized-equivalent provider id `z.ai`.

## Solution Assessment

No. The proposed shape is close, but the direct provider-map lookup is not the narrowest reliable implementation because model resolution already uses normalized provider matching; the docs also need a trusted-provider privacy note.

## Review Findings

- **[P2] Resolve provider config through normalized lookup:** `src/agents/pi-embedded-runner/extra-params.ts:328`
  - body: The new check reads `cfg.models.providers[provider]` directly. Existing model resolution accepts normalized aliases like `z.ai`/`z-ai`, so a config that successfully selects a model through an alias will never enable `injectAgentHeader`. Please use the normalized provider helper and add alias coverage.
  - confidence: 0.92

  - body: The new help text says this is useful for local proxies but does not warn that enabling it sends agent identifiers to whichever provider or proxy is configured. Please call out trusted-proxy/operator use so users do not enable it on third-party providers unknowingly.
  - confidence: 0.8
