## Summary

Keep this PR open. Current main still does not implement opt-in per-agent LLM request headers, the PR is a plausible implementation candidate for the still-missing behavior, and the remaining concerns are review/update items rather than reasons to close it.

## Best Possible Solution

Keep the PR open and ask for a focused update: use normalized provider config resolution, keep the option explicitly opt-in, add regression coverage for alias/case-normalized provider IDs plus enabled/disabled/missing-agent cases, and document the trusted-proxy privacy implication. If maintainers prefer this outside core, first extend the provider/plugin hook context to include agentId so a plugin can implement dynamic per-agent request headers cleanly.

## Risks / Open Questions

- The submitted implementation should resolve provider config through the existing normalized-provider lookup pattern; direct provider-key lookup can make the feature silently fail for aliases or case-normalized provider IDs.
- Even as an opt-in feature, X-OpenClaw-Agent exposes local agent identifiers to the configured provider or proxy, so docs/help should describe it as a trusted-proxy/local-operator feature.
- Closing this PR now would drop the only in-repo implementation candidate found for the per-agent LLM request header behavior; the related feature reports in the provided context are closed while main still lacks the behavior.
