## Summary

Keep open. Current main still has the reported cleanup gap: the runtime-deps pruning path only considers `openclaw-unknown-*` roots, Gateway startup and `openclaw plugins deps --prune` use that same helper, and the focused test still asserts that a stale versioned root is preserved. Recent update/restart work reduces adjacent failure modes, but it does not implement cleanup or ignore handling for stale `openclaw-{version}-*` roots.

## What This Changes

The issue asks OpenClaw to clean up stale versioned `plugin-runtime-deps/openclaw-{old-version}-*` roots after `openclaw update` and restart the Gateway through the updated install so old runtime mirrors cannot crash-loop startup.

## Best Possible Solution

Keep this open until runtime-deps owners implement a shared policy that either prunes stale OpenClaw-owned versioned roots or ignores incompatible old roots during startup/update resolution, with regression coverage for an old versioned root beside the current install root and preserved handling for locks, `openclaw-unknown-*` roots, and path-list staging.

## Reproduction Assessment

Yes for the current-main cleanup gap: source inspection and the existing pruning test show stale versioned roots survive. The full launchd crash-loop was not replayed during this read-only review.

## Solution Assessment

Unclear whether the issue body's update-only deletion sketch is the best final fix. A safer solution is an owner-reviewed cleanup or ignore policy shared by Gateway startup, package-update recovery, and any future cache-prune command.
