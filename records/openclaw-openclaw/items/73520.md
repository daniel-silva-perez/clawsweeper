## Summary

Keep open. Current main has adjacent hardening for package-update restarts and unknown-root pruning, but it still does not clean or ignore stale versioned `plugin-runtime-deps/openclaw-{version}-*` roots in the reported class; the current regression test still asserts that an old versioned root is retained.

## What This Changes

The issue requests a fix for macOS npm-global updates where stale versioned `plugin-runtime-deps` directories can leave the Gateway crash-looping until the old runtime-deps root is manually removed.

## Best Possible Solution

Implement an ownership-aware runtime-deps root policy in the plugin/runtime-deps area: remove old unlocked OpenClaw-owned `openclaw-{version}-{hash}` roots or explicitly exclude incompatible stale roots from startup/update resolution, while preserving path-list layering, lock safety, and package-update hard restart behavior.

## Reproduction Assessment

Yes for the current-main cleanup gap: the helper only prunes `openclaw-unknown-*`, and the regression test asserts a versioned root survives. The complete macOS launchd crash-loop still needs a live npm-global update smoke with an old `openclaw-2026.4.24-*` root beside the current root.

## Solution Assessment

Unclear whether the issue's update-only deletion sketch is the best final fix. A safer maintainable solution is a locked, ownership-aware cleanup or ignore path that works during both package update and Gateway startup without breaking `OPENCLAW_PLUGIN_STAGE_DIR` layering.
