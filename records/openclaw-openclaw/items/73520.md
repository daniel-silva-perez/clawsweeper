## Summary

Keep open. Current main still preserves stale versioned `openclaw-{version}-*` runtime-deps roots, while only pruning `openclaw-unknown-*` roots; the related restart, mirror, timeout, and lock fixes reduce adjacent failure modes but do not implement the reported cleanup or ignore policy.

## What This Changes

The issue requests automatic cleanup or safe ignoring of stale versioned `plugin-runtime-deps` roots after an `openclaw update` so Gateway does not crash-loop on old staged runtime code.

## Best Possible Solution

Implement a runtime-deps owner policy that either removes stale OpenClaw-owned versioned roots or makes startup/update resolution ignore incompatible old roots, while preserving locks, staged path-list layering, and the updated-install restart flow. Add a regression with an old versioned root beside the current install root.

## Reproduction Assessment

Yes for the current-main cleanup gap: source inspection and the focused regression test show stale versioned roots survive pruning. The full launchd crash-loop itself was not live-reproduced during this read-only review.

## Solution Assessment

Unclear whether the issue body's update-only deletion sketch is the best final fix. A safer solution is an ownership-aware cleanup or ignore path shared by Gateway startup and package-update recovery, with explicit tests for versioned roots, locks, and layered stage dirs.
