## Summary

Keep open. Current main still only prunes stale `openclaw-unknown-*` runtime-deps roots, and the focused regression test still asserts that stale versioned `openclaw-{version}-*` roots survive pruning, so the reported update crash-loop class is not implemented away yet.

## What This Changes

The issue requests automatic cleanup or safe ignoring of stale versioned `plugin-runtime-deps/openclaw-{version}-*` roots after package updates, plus a fresh restart path so the updated Gateway does not load old runtime-deps artifacts.

## Best Possible Solution

Define a runtime-deps owner policy that removes or ignores stale OpenClaw-owned versioned roots during update/startup without deleting active locked roots or breaking layered stage dirs, then add a regression with an old versioned root beside the current install root.

## Reproduction Assessment

Yes for the current-main cleanup gap: source inspection and the existing focused test show stale versioned runtime-deps roots survive pruning. The full launchd crash-loop was not live-reproduced here.

## Solution Assessment

Unclear whether the issue body's update-only deletion sketch is the safest final fix. A maintainable fix should share ownership-aware cleanup or ignore logic between Gateway startup and package-update recovery, with explicit lock and layered-root tests.
