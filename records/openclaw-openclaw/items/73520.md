## Summary

Keep open. Current main has partial mitigation for the update failure class, but it still does not clean or ignore stale versioned `plugin-runtime-deps/openclaw-{version}-*` roots: startup pruning is limited to `openclaw-unknown-*`, and the regression test still asserts that an old versioned root is retained. The linked PRs/issues are related runtime-deps hardening, not a canonical fix for the versioned stale-root crash-loop.

## What This Changes

The issue asks OpenClaw to prevent Gateway crash-loops after `openclaw update` by cleaning or ignoring stale versioned plugin-runtime-deps roots from older package versions and restarting through the updated install.

## Best Possible Solution

Keep this open and implement a guarded runtime-deps root policy in the plugin/runtime-deps owner area: either remove stale `openclaw-{version}-{hash}` roots that are provably OpenClaw-owned and unlocked, or make startup/update resolution ignore incompatible old versioned roots. The fix should preserve path-list layering, lock safety, package-update restart behavior, and add a regression for an old `2026.4.24` root beside the current install.

## Reproduction Assessment

Yes for the current-main cleanup gap: `src/plugins/bundled-runtime-deps-roots.ts` only prunes `openclaw-unknown-*`, and `src/plugins/bundled-runtime-deps.test.ts` asserts a versioned root survives. The full five-hour macOS launchd loop was not live-reproduced during this read-only review.

## Solution Assessment

Unclear whether the issue's update-only deletion sketch is the best final fix. The safer maintainable direction is a locked, ownership-aware cleanup or ignore path that works both during package update and Gateway startup without breaking `OPENCLAW_PLUGIN_STAGE_DIR` layering.
