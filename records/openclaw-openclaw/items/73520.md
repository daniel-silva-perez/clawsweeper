## Summary

Keep open. Current main still only prunes stale `openclaw-unknown-*` runtime-deps roots and the regression test explicitly preserves an old versioned `openclaw-2026.4.25-*` root, so the reported stale versioned-root update failure is not fixed. The package-update restart path has been hardened, and related runtime-deps lock/mirror fixes exist, but none is a canonical cleanup or ignore policy for stale `plugin-runtime-deps/openclaw-{version}-*` roots.

## What This Changes

Clean up or ignore stale versioned `plugin-runtime-deps` roots after `openclaw update` so a Gateway started from the updated package cannot crash-loop on old bundled runtime dependency trees.

## Best Possible Solution

Implement a guarded runtime-deps root policy in the plugin/runtime-deps owner area that either removes stale OpenClaw-owned `openclaw-{version}-{hash}` roots or makes scan/startup resolution ignore incompatible old versioned roots. Preserve locks, path-list layering, package-update restart behavior, and add a regression with an old 2026.4.24 root beside the current install root.

## Reproduction Assessment

Yes for the current-main cleanup gap: source inspection shows startup pruning only targets `openclaw-unknown-*`, and the test suite explicitly asserts a stale versioned root survives. The exact macOS launchd crash-loop was not live-run during this read-only review.

## Solution Assessment

Unclear whether the issue's update-only deletion sketch is the best final fix. A safer maintainable fix is a generic, locked, ownership-aware cleanup or ignore path that works during both package update and Gateway startup without breaking staged-root layering.
