## Summary

Keep open. Current main still lacks the PR's Twitch refresh writeback and the OpenProse fast-loop example still has the fall-through shape this PR is trying to fix. The work looks useful, but the existing review comments raise real persistence and secret-handling issues that should be addressed before merge.

## Best Possible Solution

Keep this PR open as a useful implementation candidate. The best path is to rebase it onto current main, fix the token writeback edge cases, preserve or skip env-managed token references safely, add coverage for config/env/none token sources plus missing expiresIn/obtainmentTimestamp cases, and keep the OpenProse else-branch fix or split it into a separate focused PR if requested.

## Risks / Open Questions

- The PR touches OAuth access and refresh token persistence. Before merge, it needs an explicit design/test for env-template and env-backed credentials so refreshed secrets are not unexpectedly written as plaintext config values.
- The proposed updateTokenFields behavior described in review comments can rewrite config unnecessarily when obtainmentTimestamp is absent and can clear expiresIn when a refresh payload omits it.
- The PR mixes a Twitch runtime/config persistence change with an OpenProse example control-flow fix. That is not a close reason, but it increases review surface and argues for focused tests or a split if maintainers prefer.
