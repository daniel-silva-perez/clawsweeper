## Summary

Keep this PR open. Current main still has the silent TLS chmod, fs-safe close, allow-from malformed JSON, and internal-hook non-null assertion paths that the PR set out to clean up. The PR is also not merge-ready as-is: its allow-from hunk is stale against the current helper module, it logs TLS permission failures at info level, and two `logWarn(..., { error })` additions would pass a non-runtime object where the logger expects `RuntimeEnv`. Security pass: the submitted diff does not add dependencies, workflows, package metadata, downloaded artifacts, or other supply-chain expansion, but it does touch security-sensitive FS/TLS/pairing paths and can make formerly non-throwing fallback paths throw.

## Best Possible Solution

Keep this PR open for author or maintainer revision. A safe version should rebase onto current main, move malformed allow-from JSON logging into `src/pairing/allow-from-store-file.ts`, log TLS chmod failures at warn severity, report close errors without passing metadata objects as `RuntimeEnv`, preserve non-throwing cleanup/fallback behavior, and add focused regression tests for those error paths.

## Risks / Open Questions

- The submitted patch can turn file-handle cleanup and malformed allow-from JSON fallback into throwing paths because `{ error }` is not a `RuntimeEnv`.
- TLS key/cert permission restriction failures would be logged at `info`, below the warning severity expected for a possible private-key or certificate permission degradation.
- The allow-from logging change targets `src/pairing/pairing-store.ts`, but current main's sync parsing and fallback behavior lives in `src/pairing/allow-from-store-file.ts`.
- No focused regression tests currently prove TLS chmod failure logging, close-failure logging, or malformed allow-from JSON logging behavior.
