## Summary

Keep this PR open. Current main at e476523082a6fff3a0c3048077d2f0e52413fbf2 still does not implement the PR's intended cleanup: TLS chmod failures are swallowed, fs-safe close failures are suppressed, malformed allow-from JSON fallback is still silent in the current helper module, and the internal-hook non-null assertion remains. The submitted patch also has unresolved correctness/security-review issues, so it needs revision rather than auto-close as obsolete.

## Best Possible Solution

Keep this PR open for author or maintainer revision. A mergeable version should rebase onto current main, log TLS chmod failures at warn severity, add a close helper that reports errors without throwing, move malformed allow-from JSON logging into `src/pairing/allow-from-store-file.ts` while preserving the `exists: true` empty fallback, keep the internal-hook type cleanup, and add focused regression tests for the error paths.

## Risks / Open Questions

- The submitted fs-safe and pairing-store changes pass `{ error }` where `logWarn` expects a `RuntimeEnv`, so formerly non-throwing cleanup and malformed JSON fallback paths can throw `TypeError`.
- The TLS chmod failure logging in the submitted PR uses info level, which can hide a private key/cert permission restriction failure from warning/error monitoring.
- The pairing-store hunk is stale against current main because sync allow-from parsing now lives in `src/pairing/allow-from-store-file.ts`.
- The PR adds no focused regression tests for TLS chmod failure logging, close-failure logging, or malformed allow-from JSON warning behavior.
