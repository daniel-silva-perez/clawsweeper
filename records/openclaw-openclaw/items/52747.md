## Summary

Keep open. The PR addresses a real ACP control-plane gap, but its current timeout design can release the same-session lane while the old operation is still running, and can undercount live runtimes when idle close hangs; those are blocking concurrency/resource-accounting concerns for ACP runtime control.

## What This Changes

The PR adds configurable `acp.sessionLane.taskTimeoutMs`, wires it through ACP session actor execution and config schema/help/generated metadata, and adds manager coverage for a hung initialize task unblocking a queued initialize.

## Best Possible Solution

Keep the session-lane deadline feature, but revise it so deadlines cannot break per-session serialization: add cooperative cancellation where runtime contracts support it, fence or retire stale timed-out operations before later same-session work can observe side effects, keep runtime handles counted until close is confirmed or represented as closing, clamp timer values to Node's supported range, and cover those paths with focused manager/config regression tests.

## Reproduction Assessment

Yes. A mocked ACP runtime whose `ensureSession()` never settles will keep the current `SessionActorQueue` tail pending and block later same-key actor work; the PR's own test shape demonstrates that gap, and source inspection shows why the proposed early rejection changes serialization semantics.

## Solution Assessment

No. Releasing the queue-visible promise on timeout is not the best fix because `KeyedAsyncQueue` uses that promise as the serialization boundary; the safer solution needs cancellation/fencing and explicit runtime accounting before the lane is released.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve serialization after lane timeouts:** `src/acp/control-plane/session-actor-queue.ts:32-40`
  - body: `KeyedAsyncQueue` advances same-key work when the promise returned from this wrapper settles, but the timeout rejects that promise while `params.op()` continues in the detached chain. Later initialize/status/cancel/close work for the same ACP session can then run concurrently with stale side effects, so the timeout needs cancellation/fencing or must keep the queue tail locked until the old operation is safe to retire.
  - confidence: 0.94
- **[P1] Keep evicted runtimes counted until close finishes:** `src/acp/control-plane/manager.core.ts:1892-1899`
  - body: Idle eviction clears `runtimeCache` before `runtime.close()` and this PR runs that path under the lane timeout. If close hangs and the timeout releases the lane, the live runtime is no longer counted by `maxConcurrentSessions`; clear only after confirmed close or track a closing state that still counts.
  - confidence: 0.9
- **[P2] Clamp session-lane timeouts to Node's timer range:** `src/config/zod-schema.ts:519`
  - body: The schema accepts any nonnegative integer, but Node sets delays above `2147483647` ms to `1` ms. A large configured timeout would therefore fail ACP lane tasks immediately, so reject or cap oversized values before scheduling the timer.
  - confidence: 0.91
