## Summary

Keep open. Current main still lacks ACP session-lane timeout support, so this PR addresses a real stuck-lane gap, but the branch's timeout design releases the same-session queue while the underlying actor operation can continue and can drop idle runtime accounting before close is confirmed. That makes the patch unsafe to merge without maintainer design review.

## What This Changes

The PR adds configurable `acp.sessionLane.taskTimeoutMs`, passes it through ACP session actor runs and idle eviction, updates config schema/help/generated metadata, and adds manager tests for a hung initialize task unblocking the next queued initialize.

## Best Possible Solution

Keep the PR open and revise the ACP deadline design so initialize/status/cancel/close/idle-eviction deadlines cannot violate per-session serialization, stale timed-out operations are cancelled or fenced before later same-session work can observe side effects, runtime handles remain counted until close is confirmed or tracked as closing, timer values are clamped to Node-safe ranges, and the config/runtime behavior is covered by focused regression tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve serialization after lane timeouts:** `src/acp/control-plane/session-actor-queue.ts:31-39`
  - body: `KeyedAsyncQueue` advances same-key work when the promise returned from this wrapper settles, but the timeout rejects that promise while `params.op()` continues in the detached chain. That can let later initialize/status/cancel/close work for the same ACP session run concurrently with stale side effects, so the timeout needs cancellation/fencing or must keep the queue tail locked until the old operation is safe to retire.
  - confidence: 0.94
- **[P1] Keep evicted runtimes counted until close finishes:** `src/acp/control-plane/manager.core.ts:1892-1899`
  - body: Idle eviction clears `runtimeCache` before `runtime.close()` and now runs under the lane timeout. If close hangs and the timeout releases the lane, the live runtime is no longer counted by `maxConcurrentSessions`; clear only after confirmed close or track a closing state that still counts.
  - confidence: 0.9
- **[P2] Clamp session-lane timeouts to Node's timer range:** `src/config/zod-schema.ts:519`
  - body: The schema accepts any nonnegative integer, but Node clamps delays above `2147483647` ms to `1` ms. A large configured timeout would therefore fail ACP lane tasks immediately, so reject or cap oversized values before scheduling the timer.
  - confidence: 0.91

## Security Review

Status: needs_attention

Summary: The diff changes ACP runtime-control concurrency and can weaken session serialization and runtime resource accounting if merged as-is.

Concerns:

- **[high] Timeout can break per-session serialization:** `src/acp/control-plane/session-actor-queue.ts:31`
  - body: Rejecting the queue-visible promise while the underlying actor operation continues can let later same-session work run concurrently with stale side effects from the timed-out task.
  - confidence: 0.94
- **[high] Idle eviction can orphan runtime accounting:** `src/acp/control-plane/manager.core.ts:1892`
  - body: A timed-out idle close can leave the runtime process/session alive after its cache entry was removed, weakening `maxConcurrentSessions` enforcement and cleanup tracking.
  - confidence: 0.88
- **[medium] Large timeout values can force immediate failures:** `src/config/zod-schema.ts:519`
  - body: Because Node clamps oversized timer delays to `1` ms, the permissive config schema can turn an intended long timeout into near-immediate ACP task failure.
  - confidence: 0.9
