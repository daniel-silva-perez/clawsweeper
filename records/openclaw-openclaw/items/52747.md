## Summary

Keep open. Current main still has the ACP stuck-session-lane gap, but this PR's timeout implementation needs maintainer review before merge because the queue-visible timeout can release per-session serialization while the timed-out operation is still running, and the idle-eviction path can undercount live runtimes after a timed-out close.

## What This Changes

The PR adds configurable `acp.sessionLane.taskTimeoutMs`, wraps ACP session actor queue tasks with a timer, wires the setting through config schema/help/generated metadata, and adds ACP manager regression coverage for a hung initialize task.

## Best Possible Solution

Keep the session-lane deadline work, but revise it so deadlines do not break per-session serialization, add cooperative cancellation where runtime contracts support it, fence or retire stale timed-out operations, keep runtime cache/concurrency accounting accurate until close is confirmed or explicitly tracked as closing, clamp timer values to Node-safe ranges, and keep focused regression coverage for initialize, close/eviction, and config validation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve serialization after lane timeouts:** `src/acp/control-plane/session-actor-queue.ts:31-39`
  - body: `KeyedAsyncQueue` advances same-key work when this wrapper's promise settles, but the timeout rejects that promise while `params.op()` keeps running. A timed-out `ensureSession`, status, cancel, or close can still mutate the same ACP session while the next queued operation starts, so the timeout needs cancellation/fencing or must keep the queue tail locked until the old operation is safe to retire.
  - confidence: 0.94
- **[P1] Keep evicted runtimes counted until close finishes:** `src/acp/control-plane/manager.core.ts:1892-1899`
  - body: Idle eviction clears `runtimeCache` before `runtime.close()` and now runs under the lane timeout. If close hangs and the timeout releases the lane, the live runtime is no longer counted by `maxConcurrentSessions`; clear the entry only after confirmed close or track a closing state that still counts.
  - confidence: 0.89
- **[P2] Clamp task timeouts to Node's timer range:** `src/config/zod-schema.ts:519`
  - body: The schema accepts any nonnegative integer, but Node sets `setTimeout` delays above `2147483647` ms to `1` ms. A large configured timeout would therefore fail ACP lane tasks immediately, so reject or cap oversized values before scheduling the timer.
  - confidence: 0.91

## Security Review

Status: needs_attention

Summary: Security review needs attention because the diff touches ACP runtime-control concurrency and can weaken session serialization and resource accounting.

Concerns:

- **[high] Timeout can break per-session serialization:** `src/acp/control-plane/session-actor-queue.ts:31`
  - body: Rejecting the queue-visible promise while the underlying actor operation continues can let later same-session work run concurrently with stale side effects from the timed-out task.
  - confidence: 0.94
- **[high] Idle eviction can orphan runtime accounting:** `src/acp/control-plane/manager.core.ts:1892`
  - body: A timed-out idle close can leave the runtime process/session alive after its cache entry was removed, weakening `maxConcurrentSessions` enforcement and cleanup tracking.
  - confidence: 0.88
- **[medium] Large timeout values can force immediate failures:** `src/config/zod-schema.ts:519`
  - body: Because Node clamps oversized timer delays to `1` ms, the permissive config schema can turn an intended long timeout into near-immediate ACP task failure.
  - confidence: 0.9
