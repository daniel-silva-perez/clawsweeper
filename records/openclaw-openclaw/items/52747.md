## Summary

Keep this PR open. Current main still lacks an ACP session-actor lane timeout for hung initialize or ensure-runtime work, so the PR addresses a real control-plane gap; the proposed branch still needs careful concurrency and resource-accounting review before merge.

## What This Changes

The PR adds a configurable `acp.sessionLane.taskTimeoutMs`, wires it through ACP session actor execution and config schema/help files, and adds a regression test for a hung initialize task unblocking the next queued initialize.

## Best Possible Solution

Keep a session-lane deadline design in the ACP control plane, but revise it so deadlines cover initialize/status/cancel/close/idle-eviction tasks without breaking per-session serialization, add cooperative cancellation where runtime contracts support it, fence or retire stale timed-out operations, keep runtime cache/concurrency accounting until close is confirmed, clamp and unref timers, and document the config with focused tests.

## Risks / Open Questions

- The PR timeout can reject the promise used for per-session queue serialization while the underlying ACP operation continues, allowing stale same-session side effects to race with newer work.
- Applying the timeout to idle eviction is risky while `runtimeCache.clear()` happens before `runtime.close()` completes; a timed-out close can orphan runtime state and undercount active sessions.
- The proposed `taskTimeoutMs` schema accepts very large values, so timer clamping and disable/default semantics need explicit maintainer review.
- The branch is an XL control-plane refactor around initialize/status/cancel/close/idle-eviction behavior, so focused regression coverage is needed before merge.
