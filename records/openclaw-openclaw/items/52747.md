## Summary

Keep this PR open. Current main still lacks an ACP session-actor lane timeout for hung initialize/ensure-runtime work, so the PR addresses a real control-plane gap. The implementation candidate remains concurrency- and resource-sensitive because its timeout releases the queue-visible promise while the original operation can keep running.

## Best Possible Solution

Keep this PR open as an implementation candidate for a real ACP control-plane bug. The best path is a session-lane deadline design that covers initialize/status/cancel/close/idle-eviction actor tasks, preserves per-session serialization or fences stale post-timeout effects, propagates cooperative cancellation where runtime APIs support it, keeps runtime cache/concurrency accounting until close is confirmed, clamps and unrefs timers safely, and wires acp.sessionLane.taskTimeoutMs through schema/help/docs plus focused regression tests.

## Risks / Open Questions

- The PR's timeout can reject the promise that KeyedAsyncQueue uses for serialization while the underlying ACP operation keeps running, allowing newer same-session work to race with stale side effects.
- Applying the same timeout to idle eviction is risky because current eviction clears runtimeCache before runtime.close() completes; releasing the lane after a timed-out close can orphan runtimes and undercount maxConcurrentSessions.
- The proposed schema accepts very large taskTimeoutMs values, which need timer clamping/validation because Node timers overflow/clamp outside the practical 32-bit delay range.
- The PR is an XL refactor around the ACP control plane, so it needs careful maintainer review and regression coverage for initialize, status, cancel, close, idle eviction, and stale post-timeout side effects before merge.
