## Summary

Keep open. Current main still lacks an ACP session-lane task deadline, so the PR is not obsolete, but the latest diff introduces queue-visible timeout semantics that can release same-session serialization while the underlying ACP operation keeps running. The prior Aisle and ClawSweeper review comments already identified the same concurrency, resource-accounting, and timer-range concerns, and local source inspection confirms those concerns are tied to the current queue/runtime contracts.

## What This Changes

The PR adds configurable `acp.sessionLane.taskTimeoutMs`, wraps ACP actor-queue tasks in a timeout, threads that timeout through ACP manager session and idle-eviction paths, updates config schema/help/generated metadata, and adds manager coverage for a hung initialize task unblocking a queued initialize.

## Best Possible Solution

Keep the ACP session-lane deadline feature, but revise it so deadlines cannot break per-session serialization: add cooperative cancellation where runtime contracts support it, fence or retire stale timed-out operations before later same-session work can observe side effects, keep runtime handles counted until close is confirmed or represented as closing, clamp timer values to Node's supported range, and cover those cases with focused manager/config regression tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve serialization after lane timeouts:** `src/acp/control-plane/session-actor-queue.ts:31-39`
  - body: `KeyedAsyncQueue` advances same-key work when the promise returned from this wrapper settles, but the timeout rejects that promise while `params.op()` continues in the detached chain. Later initialize/status/cancel/close work for the same ACP session can then run concurrently with stale side effects, so the timeout needs cancellation/fencing or must keep the queue tail locked until the old operation is safe to retire.
  - confidence: 0.94
- **[P1] Keep evicted runtimes counted until close finishes:** `src/acp/control-plane/manager.core.ts:1892-1899`
  - body: Idle eviction clears `runtimeCache` before `runtime.close()` and now runs under the lane timeout. If close hangs and the timeout releases the lane, the live runtime is no longer counted by `maxConcurrentSessions`; clear only after confirmed close or track a closing state that still counts.
  - confidence: 0.9
- **[P2] Clamp session-lane timeouts to Node's timer range:** `src/config/zod-schema.ts:519`
  - body: The schema accepts any nonnegative integer, but Node sets delays above `2147483647` ms to `1` ms. A large configured timeout would therefore fail ACP lane tasks immediately, so reject or cap oversized values before scheduling the timer.
  - confidence: 0.91

## Security Review

Status: needs_attention

Summary: The diff changes ACP runtime-control concurrency in ways that can weaken per-session serialization and runtime resource accounting if merged as-is.

Concerns:

- **[high] Timeout can break per-session serialization:** `src/acp/control-plane/session-actor-queue.ts:31`
  - body: Rejecting the queue-visible promise while the underlying actor operation continues can let later same-session work run concurrently with stale side effects from the timed-out task.
  - confidence: 0.94
- **[high] Idle eviction can orphan runtime accounting:** `src/acp/control-plane/manager.core.ts:1892`
  - body: A timed-out idle close can leave the runtime process/session alive after its cache entry was removed, weakening `maxConcurrentSessions` enforcement and cleanup tracking.
  - confidence: 0.88
- **[medium] Large timeout values can force immediate failures:** `src/config/zod-schema.ts:519`
  - body: Because Node sets oversized timer delays to `1` ms, the permissive config schema can turn an intended long timeout into near-immediate ACP task failure.
  - confidence: 0.9
