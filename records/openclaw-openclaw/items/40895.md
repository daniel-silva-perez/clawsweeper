## Summary

Keep open. Current main still lowers Discord mention requirements for bot-owned autoThread threads and active bound thread sessions before the no-mention drop path runs, so the reported follow-up replies can still bypass `requireMention`. The related #52568 is open and narrower, while #41072 was closed unmerged and explicitly scoped away from Discord mention-gating changes.

## What This Changes

The issue requests Discord thread follow-up messages to honor `requireMention` on each message, or to add an explicit thread-specific mention requirement setting.

## Best Possible Solution

Define one explicit Discord thread-reply mention policy, preferably aligned with Slack's `thread.requireExplicitMention` behavior, then update Discord preflight, autoThread ownership handling, bound-thread session behavior, config types, strict schema, generated/help metadata, docs, and regression tests for both bot-owned autoThread replies and bound thread sessions.

## Reproduction Assessment

Yes. Static source inspection and existing test expectations give a high-confidence reproduction path: construct a bot-owned autoThread thread or bound Discord thread session with `requireMention: true`, send a follow-up without a mention, and preflight currently resolves `shouldRequireMention` to false instead of taking the no-mention drop branch.

## Solution Assessment

Unclear. The issue's strict per-message rule would address the reported bypass, but current tests encode implicit thread continuity for some workflows, so the best maintainable fix is an explicit, documented Discord thread-reply mention policy rather than a silent behavior flip.
