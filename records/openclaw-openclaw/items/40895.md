## Summary

Keep open. Current main still has two Discord thread paths that can bypass the configured mention requirement for follow-up thread messages: bot-owned autoThread threads force mention gating off, and bound thread sessions lower the preflight mention requirement before the no-mention drop branch runs. The related #52568 is useful but narrower and still open; #41072 was closed unmerged and explicitly avoided Discord mention-gating changes.

## What This Changes

The issue requests Discord thread messages to apply mention requirements per message, or to expose an explicit thread-specific mention policy, so follow-up thread replies do not keep triggering after an initial mention.

## Best Possible Solution

Define one explicit Discord thread-reply mention policy, preferably aligned with Slack's `thread.requireExplicitMention` semantics, then update Discord preflight and allow-list behavior, config types, strict schema, generated/help metadata, docs, and regression tests for both bot-owned autoThread replies and bound thread sessions.

## Reproduction Assessment

Yes. Static source inspection and existing tests give a high-confidence reproduction path: construct a bot-owned autoThread thread or bound Discord thread session with `requireMention: true`, send a follow-up message without a mention, and preflight currently does not take the no-mention drop path.

## Solution Assessment

Unclear. The issue's strict per-message rule would fix the report, but the safest maintainable direction is an explicit Discord thread-reply mention policy because current autoThread and bound-session behavior appears intentional for some workflows.
