## Summary

Keep open. Current main still normalizes hosted custom-scheme browser origins through `URL.origin`, which becomes the opaque string `"null"`, so explicit `gateway.controlUi.allowedOrigins` entries like `tauri://localhost` still cannot match. The PR is a narrow active fix candidate for the linked open bug and I found no blocking correctness issue in the provided diff.

## What This Changes

This PR updates Gateway origin normalization to preserve `scheme://host` for hosted custom URL schemes, adds custom-scheme origin-check regression cases, and adds a changelog entry.

## Best Possible Solution

Merge this narrow Gateway parser/test fix after maintainer review and CI, preserving explicit allowlist semantics rather than broadening wildcard behavior or Host-header fallback.

## Reproduction Assessment

Yes. Current main can be checked by reading `parseOrigin` and by a Node URL probe showing custom hosted schemes serialize to `origin:"null"`, which makes `allowlist.has(parsedOrigin.origin)` miss `tauri://localhost`.

## Solution Assessment

Yes. The proposed fix is the narrowest maintainable path: preserve `scheme://host` only when WHATWG URL gives an opaque origin but still provides protocol and host, then continue through the existing lower-case exact allowlist check.
