## Summary

Keep this issue open. The report is against v2026.4.26, and current main does not show a clear fix for the observable Discord exec-approval failure. Main still gives the model an approval-pending tool result that contains both the full shell command and a `/approve <slug> ...` line, while existing tests assert that shape. Discord native approval plumbing exists, but I did not find source or test evidence that current main prevents the final assistant reply from conflating the command text with the approval slug for weaker Ollama-style models, or that the missing Discord DM card path was fixed after the reported release.

## Best Possible Solution

Keep this issue open and fix the approval-pending flow so Discord native approval sessions do not depend on a weak model copying mixed free-text correctly. A good fix would either suppress the model-visible manual `/approve` line when native Discord approval delivery is active, or make the tool result and final-response guidance unambiguous enough that `/approve` can only use the slug/decision fields. Add targeted tests around Discord native approval configuration, final visible reply behavior, and the existing manual fallback path.
