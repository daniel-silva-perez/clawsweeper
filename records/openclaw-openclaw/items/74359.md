## What This Changes

The PR adds a digest-pinned `OPENCLAW_BUN_IMAGE` source stage to the root Dockerfile, copies Bun from that stage into the build stage, removes the `bun.sh` installer execution, and extends Docker digest tests to cover ARG-backed `FROM` stages.

## Best Possible Solution

Land a reviewed Docker hardening change that removes build-time execution of the remote Bun installer, keeps the Bun image digest-pinned and aligned with CI, preserves multi-arch Docker behavior, keeps digest coverage for ARG-backed stages, and closes #74356 through the merge after CI and Docker build proof pass.

## Reproduction Assessment

Yes for the reported supply-chain gap: current main can be verified by source inspection at `Dockerfile:47`, where the Docker build runs the remote Bun installer through `bash`. The PR's runtime correctness still needs Docker build or install-smoke validation rather than a source-only reproduction.

## Solution Assessment

Yes, the proposed digest-pinned multi-stage `COPY` is the narrowest maintainable fix for this Dockerfile issue. The test helper should ideally model Docker ARG scope more precisely, but that is a low-severity test hardening issue rather than a blocker to the main approach.

## Review Findings

Overall correctness: patch is correct

Overall confidence: 0.8

Full review comments:

- **[P3] Limit ARG defaults to global Dockerfile scope:** `src/docker-image-digests.test.ts:36-45`
  - body: `resolveArgDefaults` now reads every `ARG`, including stage-local declarations after a `FROM`. Docker only lets ARG defaults declared before the first `FROM` feed later `FROM` image substitutions, so a future stage-local ARG could make this test pass even though Docker would not resolve that `FROM` as written. Stop scanning at the first `FROM`, or model ARG scope explicitly.
  - confidence: 0.74
