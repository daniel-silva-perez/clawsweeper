## Summary

Keep open. Current main still has a plausible onboarding UX bug: the keyless local Whisper dependency appears in the install list, while the later credential loop can still ask for keys for separate missing API-backed skills, and two open PRs explicitly claim closing coverage for the same report.

## What This Changes

The issue reports that `openclaw onboard` lists local `openai-whisper` as not requiring an API key, then later asks for OpenAI Whisper or ElevenLabs credentials.

## Best Possible Solution

Onboarding should keep local `openai-whisper` visibly keyless and only prompt for `OPENAI_API_KEY` or `ELEVENLABS_API_KEY` when the corresponding API-backed skill is being configured, with explicit prompt text and regression coverage for the mixed local/API skills path.

## Reproduction Assessment

Yes. A high-confidence reproduction can be modeled by a `setupSkills` unit scenario where `openai-whisper`, `openai-whisper-api`, and `sag` are all missing: current main shows the keyless local install option, then the all-missing-skill credential loop can prompt for the API-backed skills.

## Solution Assessment

No current-main solution is in place yet. The narrow maintainable fix belongs in `src/commands/onboard-skills.ts` with focused tests in `src/commands/onboard-skills.test.ts`; the active linked PRs should be reviewed or superseded rather than closing the issue now.
