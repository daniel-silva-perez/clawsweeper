## Summary

Keep open. Current main still has the plausible onboarding UX bug: the install picker can show the keyless local `openai-whisper` skill, then the same skills setup flow separately prompts for API keys for env-gated skills such as `openai-whisper-api` and `sag`. This issue is also already paired with open closing PRs #74407 and #74891, so it should not be closed before one of those lands or is replaced.

## What This Changes

The issue requests that `openclaw onboard` avoid contradictory API-key prompts after presenting local OpenAI Whisper setup as not needing a key.

## Best Possible Solution

Keep this issue open until a linked PR lands. The desired end state is a narrow `setupSkills` repair with regression coverage: local `openai-whisper` remains keyless, credential prompts are clearly scoped to the actual env-gated skills being configured, and env-only skills still prompt when appropriate.

## Reproduction Assessment

Yes. The static reproduction path is clear: make the skills status contain missing keyless `openai-whisper` plus missing env-gated `openai-whisper-api`/`sag`; current `setupSkills` will show the install list first and then prompt for every missing `primaryEnv` skill.

## Solution Assessment

Unclear until PR review. The narrowest maintainable solution appears to be in `src/commands/onboard-skills.ts` with tests for mixed keyless/keyed skills; a pure note may improve framing, but #74891's selected-skill gating better matches the skipped-install variant described in the discussion.
