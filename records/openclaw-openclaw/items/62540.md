## Summary

Keep open. Current main has substantial rate-limit handling, but it is fallback/cooldown/error reporting rather than the requested paused auto-retry of the same pending model call. Provider SDKs handle short retries; long Retry-After responses are surfaced for model/auth fallback; exhausted rate limits still end the reply operation with a user-facing failure instead of persisting retry_at and resuming automatically.

## Best Possible Solution

Keep this issue open and implement it as core reply-run/provider orchestration work: parse Retry-After/retry-after-ms/body retry hints, persist rate-limit retry metadata on the active reply/session state, expose a paused-for-rate-limit status with retry_at, schedule a bounded jittered retry through existing wake/scheduler primitives, and resume the pending model call without duplicate final replies. The likely touch points are src/agents/provider-transport-fetch.ts, src/agents/model-fallback.ts, src/auto-reply/reply/agent-runner-execution.ts, reply-run/session status contracts, and focused tests for successful retry, repeated 429 backoff, strict-model behavior, and duplicate suppression.

## Risks / Open Questions

- Closing this as implemented would conflate model fallback with same-step auto-resume. Users with no viable fallback, strict user-selected models, or long Retry-After hints can still end at a terminal rate-limit message and need manual prompting.
- A future implementation needs careful duplicate-reply protection because retrying a pending model call after partial reasoning/tool activity can replay side effects unless the retry boundary is explicit.
