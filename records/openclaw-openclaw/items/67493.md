Keep this PR open. The maintainable path is to add a dedicated chat-reply APNs payload kind and iOS route, wire both agent-event and direct chat.send completion paths with requester device/connection tracking, use current canonical main-session resolution, cover direct and relay sends plus skip/cleanup cases, and make an explicit privacy decision for notification body contents.

## Risks / Open Questions

- Merging as-is would add an unused helper because the PR body defers all call-site wiring to a follow-up.
- The proposed helper reuses sendApnsAlert, which emits openclaw.kind "push.test" instead of a dedicated chat-reply kind.
- The PR's main-session comparison appears incompatible with current canonical main-session keys such as "global" and "agent:<id>:<mainKey>".
- The iOS notification routing path for chat replies is still missing on current main.
- Including assistant reply text in APNs alert bodies is privacy-sensitive and should be an explicit product/security decision.
- The PR tests do not cover the connected-client early exit, relay delivery path, or stale-registration cleanup paths called out in review comments.
