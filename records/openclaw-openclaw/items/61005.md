## Summary

Keep open. Current main still blocks the reported Android onboarding state: `Connected (operator offline)` is modeled when the node session is connected and the operator session is offline, but the onboarding finish gate still requires both sessions to be connected. The linked open PR #61269 uses closing syntax for this issue, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue requests that Android onboarding complete when the Android node is connected to the gateway even if the operator session is offline.

## Best Possible Solution

Keep this issue open and review or land #61269, or land an equivalent Android app fix that makes node-connected/operator-offline a non-blocking final onboarding state while keeping genuinely disconnected states blocked and covered by `OnboardingFlowLogicTest`.

## Reproduction Assessment

Yes. The focused reproduction path is the pure helper state `isConnected = false` and `isNodeConnected = true`: current main returns false and the existing unit test asserts that blocked behavior, while `NodeRuntime` can display exactly `Connected (operator offline)` for that state.

## Solution Assessment

Yes, the linked fix direction appears to be the narrow maintainable solution: change the Android onboarding completion gate to use node session readiness, update the final status copy as needed, and keep the operator-offline state informational rather than blocking.
