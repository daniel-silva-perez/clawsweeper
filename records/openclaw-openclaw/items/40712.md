## Summary

Keep open. Current main has meaningful restart mitigations: managed gateway restarts drain active work, startup cleans stale transcript locks, and main-session restart recovery can resume some interrupted turns. But the central request is still not implemented: durable partial assistant-generation persistence/resume after crash, forced restart, or drain timeout. Current recovery only resumes when the transcript tail is user/tool/toolResult, while assistant-tail partial output is explicitly failed, and tests assert that behavior.

## Best Possible Solution

Keep this issue open and narrow it to the remaining product/code gap: durable generation-state recovery for interrupted assistant output after crash, forced restart, or drain timeout. The likely implementation belongs around SessionManager-backed checkpoint/status writes, startup recovery that distinguishes generating/completed-not-delivered/sent/interrupted states, and channel delivery idempotency. Add focused regression coverage for assistant partial tails, completed-before-channel-delivery, and provider replay safety.

## Risks / Open Questions

- A durable partial-generation implementation must preserve the SessionManager parentId/DAG transcript contract; raw JSONL status rewrites could sever the active branch or confuse compaction/history.
- Provider replay safety is non-trivial: resuming after a partial assistant tail can duplicate output, replay invalid tool/reasoning state, or violate provider constraints unless persisted state is carefully modeled.
- Channel delivery recovery needs coordination with previews, final sends, queued deliveries, and acknowledgement/idempotency state so Telegram/Discord/Slack users do not receive duplicate or out-of-order text.
- Existing managed-restart drain behavior already protects normal restart paths and should be preserved while adding crash/forced-restart recovery.
