## Summary

Keep open. Current main has adjacent temp-path hardening, feature-specific durable exports, and per-plugin artifact TTL metadata, but it still lacks the requested general scratch/reproducible/durable provenance contract, janitor semantics, and broad runtime-critical volatile-path validation.

## What This Changes

Request a built-in temp artifact provenance model and durable-path guardrails so OpenClaw-created system-temp files can be classified, cleaned up safely, or rejected when runtime-critical.

## Best Possible Solution

Define a small core artifact-provenance contract first: a registry or sidecar schema under OpenClaw state, an SDK/API path for plugins and first-class tools to declare scratch/reproducible/durable artifacts, doctor or cleanup reporting that understands those classes, and targeted volatile-path checks for service wrappers, module targets, and active install/build paths. Align that design with the artifact RPC work in #74706 rather than building an isolated janitor feature.

## Reproduction Assessment

Not applicable as a conventional bug reproduction. The current-main gap is high-confidence by source inspection: adjacent helpers and guards exist, but no general temp artifact provenance/classification contract is present.

## Solution Assessment

Unclear until maintainers choose the product and API boundary. The requested direction is plausible, but the maintainable path should start with a narrow shared contract and reporting surface instead of a broad one-shot cleanup implementation.
