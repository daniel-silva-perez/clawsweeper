## Summary

This should stay open. Current main has adjacent temp-path hardening, durable trajectory export handling, and per-plugin diff artifact TTL metadata, but it still does not provide the requested general scratch/reproducible/durable provenance contract, cleanup/janitor semantics, or broad volatile-path validation for runtime-critical state.

## What This Changes

The issue requests a built-in temp artifact provenance model with durable-path guardrails so OpenClaw-created files outside the workspace can be classified, tracked, cleaned up, and blocked from becoming runtime-critical temp state.

## Best Possible Solution

Define a small core artifact-provenance contract first: a registry or sidecar schema under OpenClaw state, a plugin/agent-tool API for declaring scratch/reproducible/durable artifacts, doctor/cleanup reporting that understands those classes, and targeted volatile-path checks for service wrappers, module targets, and active install/build paths.

## Reproduction Assessment

Not applicable as a conventional bug reproduction. The current-main gap is reproducible by source inspection: temp helpers, trajectory export, diffs artifacts, service wrapper validation, and guardrail scripts exist, but no general temp artifact provenance/classification contract is present.

## Solution Assessment

Unclear until maintainers choose the product and API boundary. The requested direction is plausible, but the maintainable path should start with a narrow shared contract and reporting surface rather than a broad one-shot janitor or SOP implementation.
