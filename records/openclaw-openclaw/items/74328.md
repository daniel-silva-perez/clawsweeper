## What This Changes

The issue reports a gateway regression after v2026.4.22 where v2026.4.26/current main can saturate or stall the gateway event loop, delay probes, and break inbound channel processing, especially with accumulated runtime state.

## Best Possible Solution

Keep this issue open as a concrete regression tracker within the #74209/#74630 cluster, validate current main after the runtime-mirror hardening, then land a narrow fix in the confirmed root path: runtime-deps/mirror churn if fs.stat still dominates, sidecar/readiness wiring if channel READY stalls, or bounded task-maintenance recovery if stale runtime state wedges the event loop.

## Reproduction Assessment

Yes for the narrowed regression: the issue comments give a high-confidence version-switch path using the same state directory, `OPENCLAW_GATEWAY_STARTUP_TRACE=1`, CPU sampling, gateway probes, and real channel round trips. No for the original clean-state fs.stat-storm framing, which the reporter explicitly walked back after reruns.

## Solution Assessment

Unclear because there is no proposed patch and current main only shows adjacent partial fixes. The safer solution is to reproduce against current 29a35f04 first and then patch the smallest confirmed startup/runtime-deps/readiness path instead of closing or broadly disabling configured plugins.
