## Summary

Keep this PR open. Current main still does not implement the PR's intended config-schema behavior: generic `models.providers.<id>` entries still require `models`, so a provider entry with only `baseUrl`/credentials is rejected before runtime code can use its custom endpoint. The related open bug #39589 remains a valid target, and the newer diagnostic issue #73577 reinforces that this config path is still user-visible.

## Best Possible Solution

Keep the PR or canonical bug #39589 open until main accepts generic `models.providers.<id>` entries without a `models` array, preserves a safe parsed/runtime array contract for downstream iteration, regenerates the generated config schema, and adds focused regression coverage for config parsing plus OpenAI embedding `baseUrl` fallback behavior. Treat #73577 as related diagnostic follow-up rather than a replacement for this schema fix.
