## Summary

Keep open. Current main still lacks `memory_add` and daily `----` block-aware indexing, so this draft PR is not obsolete, but the proposed patch needs changes before merge because it adds a persistent memory write path and currently has concrete block-format and tool-policy regressions.

## What This Changes

The PR adds a memory-core `memory_add` tool that appends normalized Markdown blocks to daily memory files, updates daily-file indexing to split on `----`, and exposes the tool in catalog, gateway policy, display metadata, docs, and tests.

## Best Possible Solution

Land a focused memory-core write slice only after preserving existing read-only memory policy semantics or getting explicit maintainer approval for the new default, making appends robust against existing separator-free daily files, keeping the current generic gateway plugin-tool fallback, and updating all duplicated config/tool docs and targeted tests.

## Reproduction Assessment

Yes. Static inspection reproduces both the current missing feature and the proposed regressions: main has no `memory_add`, while the PR append code merges into existing separator-free daily files and `CORE_TOOL_GROUPS` automatically expands `group:memory` from the new `sectionId`.

## Solution Assessment

No. The implementation direction is plausible, but this patch is not yet the safest maintainable form because it silently changes existing memory-group authority and does not isolate new blocks when appending to current separator-free daily files.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Separate new blocks after existing raw memory:** `extensions/memory-core/src/memory/add-memory-block.ts:50`
  - body: When today's memory file already exists without a trailing `----` separator, this appends the formatted block after whitespace only. `parseMemoryBlocks` then treats the prior raw content and the newly added text as one block until the new trailing separator, so the new memory is not indexed as its own semantic unit. Close or separate any existing raw tail before appending the new block.
  - confidence: 0.91
- **[P2] Keep memory writes out of existing memory groups:** `src/agents/tool-catalog.ts:149-150`
  - body: Adding `memory_add` to the `memory` section and `coding` profile makes existing `group:memory` policies grant a persistent write tool; before this change that group was read-only (`memory_search`, `memory_get`). Gate the write primitive behind an explicit tool name or a separately reviewed write group/profile so existing memory-only configs do not gain write authority on upgrade.
  - confidence: 0.88
- **[P3] Update the other gateway tools reference:** `docs/tools/index.md:190`
  - body: This docs table now says `group:memory` includes `memory_add`, but `docs/gateway/config-tools.md` still documents the group as only `memory_search` and `memory_get`. Update the duplicate public config-tools page too so users do not get conflicting policy guidance.
  - confidence: 0.82
