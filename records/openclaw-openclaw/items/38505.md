## Summary

Keep this issue open. Current main still only wires Discord create/reaction/thread/presence listeners, while Carbon 0.16.0 exposes MessageUpdateListener and MessageDeleteListener that the bundled Discord plugin is not using. The message handler and replay guard remain keyed to create-message delivery by account/channel/message id, so edited messages would need an explicit edit-aware dispatch path rather than naive same-id reprocessing. Related #53654 overlaps and is open, but it is broader and not clearly established as the canonical superseder.

## Best Possible Solution

Keep the issue open and implement this in the bundled Discord plugin. Add a Carbon MessageUpdateListener, ignore non-content updates, route qualifying edited messages through the same Discord authorization/preflight/session path with edited content as authoritative, make inbound replay/dedupe edit-aware, and add focused tests for mention-added edits, DMs or active threads, unchanged-content updates, bot/self edits, and duplicate update delivery. Coordinate with #53654 if delete-to-cancel is done in the same pass, but do not close this narrower issue until implementation lands.

## Risks / Open Questions

- A naive implementation that reuses the existing create-message handler without changing replay keys would likely drop edited messages that share the original message id.
- Edit handling should suppress non-content updates and bot/self edits so Discord embed/reaction/system updates do not unexpectedly trigger agent turns.
- If implemented together with #53654, delete-to-cancel needs separate session mapping and abort behavior from edit-to-reprocess; coupling them too tightly could make the narrower edit feature harder to validate.
