## Summary

Keep this PR open. Current main still has the original `job == null` launch guard, so the PR's intended Android reconnect fix is not implemented on main. The PR also has a substantive review concern: its narrow `!isActive` guard does not address `disconnect()` lazily cancelling whatever `job` points to later, so the best path is to revise rather than close as obsolete.

## Best Possible Solution

Revise this PR or replace it with a focused Android GatewaySession lifecycle fix: make `job` visibility/synchronization explicit, snapshot and clear the job synchronously in `disconnect()` before async cancellation, and add a robust immediate `disconnect()` to `connect()` regression test with contextual timeout failures. After that, review and land the Android app change.
