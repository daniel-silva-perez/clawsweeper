## Summary

Keep this PR open, but do not merge it as-is. Current main still has the Android `GatewaySession` stale-job reconnect gap the PR targets, while the proposed patch can start a replacement `runLoop()` that a pending `disconnect()` coroutine may then cancel; it also needs the required user-facing changelog entry.

## What This Changes

The PR changes Android `GatewaySession.connect()` to relaunch `runLoop()` when the cached job is missing or inactive and adds a unit test for immediate `disconnect()` to `connect()` reconnect behavior.

## Best Possible Solution

Revise the Android session lifecycle so `disconnect()` snapshots and clears the old job before asynchronous cancellation, then keep `connect()` from treating a stale inactive job as authoritative. Add deterministic reconnect coverage and a single-line changelog entry before running the Android unit test lane.

## Reproduction Assessment

Yes. The relevant path is a rapid `connect()` -> `disconnect()` -> `connect()` sequence where `disconnect()` has made the old job inactive but has not yet cleared `job`; the PR’s added test covers the user-facing reconnect shape, and the remaining PR race follows from the lazy `job` read in the pending disconnect coroutine.

## Solution Assessment

No. Relaunching from `connect()` when `!existingJob.isActive` addresses the original stale reference, but without changing `disconnect()` it creates a symmetric race where the pending disconnect task can cancel the freshly launched run loop; snapshot-and-clear in `disconnect()` is the narrower maintainable fix.
