## Summary

Keep open: current main still lacks the intended Android reconnect fix, but this PR should not merge as-is because the added inactive-job relaunch path can race with the existing asynchronous disconnect cleanup and cancel the newly launched runLoop.

## What This Changes

The PR changes Android GatewaySession.connect() to relaunch runLoop() when the cached job is null or inactive and adds a Robolectric regression test for an immediate disconnect/connect sequence.

## Best Possible Solution

Fix job ownership in GatewaySession by snapshotting and clearing the old job synchronously in disconnect() before launching async cancellation, then keep a focused reconnect regression test with an explicit second-connection signal and useful timeout diagnostics.

## Reproduction Assessment

Yes. The relevant path is a rapid GatewaySession.disconnect() followed by connect() while the old job is inactive but still cached; the current code and PR review comments give a concrete interleaving that leaves desired set without an active runLoop.

## Solution Assessment

No. The PR addresses the stale inactive job check but leaves disconnect() owning the mutable job field asynchronously, which can cancel the replacement job; the narrower maintainable fix is to transfer ownership of the old job synchronously in disconnect().

- **[P1] Snapshot the old job before relaunching:** `apps/android/app/src/main/java/ai/openclaw/app/gateway/GatewaySession.kt:157`
  - body: The new inactive-job branch can assign a fresh `job` while the earlier `disconnect()` cleanup coroutine has not run yet. Because that coroutine still evaluates `job?.cancelAndJoin()` against the mutable field, it can cancel the fresh loop and then set `job = null`, leaving `desired` populated with no active `runLoop()`. Snapshot and clear the old job synchronously in `disconnect()` before async cancellation, or otherwise serialize job ownership.
  - confidence: 0.9

  - body: If the reconnect never happens, this polling loop reports a generic timeout instead of the connection state that the existing helper includes. Complete a second deferred from `onConnected` or reuse a helper so failures include `lastDisconnect` and request count context.
