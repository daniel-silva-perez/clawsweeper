## Summary

Keep this PR open. Current main implements the central plugin-command continuation behavior under the public `continueAgent` field, but it does not fully supersede the PR: the current command handler still assumes a non-null plugin result, and public API/docs text still describes `registerCommand` as bypassing the LLM. The PR should not merge as-is because its proposed `shouldContinue` name is stale, but the remaining guard and documentation cleanup are still useful to port or explicitly reject.

## Best Possible Solution

Keep this PR open for maintainer follow-up, but do not merge it as-is. The best path is a small rebase or follow-up patch that preserves the current `continueAgent` API, ports the nullish-result guard, and updates the `registerCommand` JSDoc plus the stale SDK overview wording. Maintainers should also decide whether a continuing command should keep or suppress a reply payload, because current main keeps the reply object while the PR discussion suggested making suppression explicit.

## Risks / Open Questions

- Merging this PR as-is would add or prefer the stale `shouldContinue` API name after current main has standardized on `continueAgent`.
- A JavaScript or malformed plugin command handler that returns `undefined` or `null` can still throw in the current handler path because the result is dereferenced and destructured without a guard.
- Plugin authors can still be misled by exported JSDoc and SDK overview text that describes plugin commands as bypassing the LLM without mentioning the opt-in continuation path.
