## Summary

Keep open. Current main still loses pending approval state on gateway restart and the user-facing `/approve` and Telegram inline callback paths still surface or rethrow the raw not-found failure instead of explaining expiry/restart recovery. The related narrow fix PRs are closed unmerged, and the shipped restart-session hardening only covers stale transcript replay, not stale chat buttons.

## What This Changes

The issue asks for pending approvals to survive gateway restarts and/or for stale Telegram approval callbacks and `/approve` attempts to show actionable expiry/restart guidance instead of a generic not-found failure.

## Best Possible Solution

Implement a focused UX fix that maps approval-not-found failures to actionable re-request guidance in both the generic `/approve` reply path and Telegram inline callback path, with regression tests. Leave disk-backed pending approval persistence for separate maintainer storage/lifecycle design.

## Reproduction Assessment

Yes. The static path is high-confidence: current tests already mock `unknown or expired approval id` for `/approve` and Telegram callbacks, and the restart-loss mechanism follows from process-local approval `Map`s recreated on gateway startup.

## Solution Assessment

Yes for the immediate repair: translating the existing approval-not-found classifier at the user-facing surfaces is the narrowest maintainable fix. Persisting approvals could be useful later, but it is broader than needed to resolve the confusing stale-button error.
