## Summary

Keep open. Current main still drops in-flight approval records across a gateway process restart and still lets the stale approval miss reach user-facing approval surfaces: `/approve` formats the raw gateway sentinel, while Telegram inline callbacks rethrow the not-found failure as retryable. The supplied discussion also shows the two narrow related fix PRs are closed unmerged, so no canonical merged fix has superseded this issue.

The issue requests either durable pending approval restoration across gateway restarts, clearer stale Telegram approval-button guidance, or both.

## Best Possible Solution

Implement a focused UX fix that maps approval-not-found failures to actionable re-request guidance in both the generic `/approve` reply path and Telegram inline callback path, with regression tests. Keep durable disk persistence of pending approvals as a separate product/storage decision.

Fix OpenClaw issue #64664 with a narrow UX-only approval-not-found repair. Do not add disk persistence, do not change approval authorization, do not change approval lifecycle semantics, and do not broaden Telegram callback permissions.

- In `src/auto-reply/reply/commands-approve.ts`, map `isApprovalNotFoundError(...)` to actionable copy that says the approval request is no longer available, may have expired or been cleared by a gateway restart, asks the user to re-request the action, and tells manual `/approve` users to double-check the ID.
