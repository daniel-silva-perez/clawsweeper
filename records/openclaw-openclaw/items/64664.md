## Summary

Keep open. Current main still keeps pending approval IDs in process-local managers that are recreated on gateway restart, and the user-facing `/approve` and Telegram inline callback paths still surface or rethrow the raw `unknown or expired approval id` failure. The related implementation PRs #67982 and #64406 are both closed unmerged, while the open #66685 work only suppresses follow-up warning logs and does not address stale button UX.

## What This Changes

The issue asks OpenClaw to preserve or recover pending approvals across gateway restarts, or at minimum show actionable stale-approval guidance instead of raw unknown/expired errors for Telegram approval buttons and `/approve` submissions.

## Best Possible Solution

Implement a focused UX fix that maps approval-not-found failures to actionable re-request guidance in both the generic `/approve` reply path and the Telegram inline callback path, with regression tests. Keep disk-backed pending approval persistence as a separate maintainer product decision.

## Repair Work Prompt

Fix the stale approval UX reported in the current issue. Current main still returns or rethrows `unknown or expired approval id` when a pending approval ID is gone after expiry or gateway restart. Implement a narrow repair that translates approval-not-found failures into actionable guidance in both `src/auto-reply/reply/commands-approve.ts` and `extensions/telegram/src/bot-handlers.runtime.ts`, with focused regression tests in `src/auto-reply/reply/commands-approve.test.ts` and `extensions/telegram/src/bot.test.ts`. Use `isApprovalNotFoundError(...)` or the existing structured sentinel rather than broad string matching when possible. The message should tell users the approval request is no longer available, mention expiry/restart as possible causes, tell them to re-request the action, and stay accurate for manually typed bad IDs by suggesting they double-check the ID. Do not add disk persistence, change approval authorization, alter approval lifecycle semantics, or broaden exec/plugin fallback behavior. Closed PRs #67982 and #64406 are useful references, but rebase the idea onto current main instead of copying stale code. #66685 is only warning-log suppression and should not be treated as the UX fix. Add a single-line user-facing changelog entry under the active Fixes section crediting the issue reporter and any source PR contributors used; do not credit bots.
