## Summary

Keep this issue open. Current main has Feishu sender-scoped group sessions, DM/comment dynamic agent creation, and explicit Feishu subagent thread-binding hooks, but it still does not automatically create or route each Feishu group sender/role to an isolated `__mc_*` subagent with its own workspace, skills/config, memory, lifecycle, and admin controls. The KinthAI comments are useful prior-art and design context, but they are not a merged OpenClaw implementation or a canonical replacement.

## Best Possible Solution

Keep the issue open for Feishu plugin product/API design. The likely path is an opt-in Feishu group mode that maps sender or role identity to bounded isolated agent/subagent profiles, provisions workspace/agentDir/skills/model/config/memory, routes replies back to the originating group/thread, and adds admin controls plus isolation, lifecycle, and security coverage. If maintainers prefer this outside the bundled Feishu plugin, first define the plugin/API seam needed for external Feishu routing and provisioning.

## Risks / Open Questions

- This feature crosses group-chat trust boundaries: per-user memory, skills, workspace access, model/config overrides, admin commands, lifecycle limits, and cost attribution need explicit opt-in policy and security tests.
- A ClawHub or external plugin path may be viable only after maintainers define a stable Feishu routing/provisioning seam; current main does not show a way to alter the bundled Feishu inbound group behavior externally.
- Keeping this open may overlap with narrower Feishu subagent/thread-binding work, but the current request is broader and no canonical replacement currently tracks the full per-sender/role provisioning model.
