## Summary

Keep this issue open. Current main has adjacent pieces: Feishu can scope group sessions by sender/topic, Feishu can dynamically create full agents for direct-message/comment senders, and Feishu supports explicit thread-bound subagent delivery. It still does not automatically create or route each Feishu group sender/role to an isolated `__mc_*` subagent with its own workspace, skills/config, memory, lifecycle, and admin controls. The new KinthAI comment is useful prior-art/workaround context, but it is not an OpenClaw implementation or a canonical replacement issue/PR.

## Best Possible Solution

Keep the issue open for Feishu plugin product/design work. The likely path is an explicit opt-in Feishu group mode that maps sender or role identity to bounded isolated agent/subagent profiles, provisions workspace/agentDir/skills/model/config/memory, routes replies back to the originating group/thread, adds admin controls and lifecycle limits, and covers isolation/security with tests and docs. If maintainers want this outside the bundled Feishu plugin, first define the plugin/API seam needed for an external Feishu routing implementation.

## Risks / Open Questions

- This feature crosses group-chat trust boundaries. Per-user memory, skills, workspaces, model/config overrides, and admin commands need explicit opt-in authorization, lifecycle limits, and security documentation.
- A ClawHub or external plugin implementation might be viable if it owns the Feishu channel path, but current main does not show a stable external seam for altering the bundled Feishu inbound routing/provisioning behavior without maintainer product/API design.
