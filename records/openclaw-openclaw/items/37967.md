## Summary

Keep this issue open. Current main has Feishu sender-scoped group sessions, DM/comment dynamic agent creation, and explicit thread-bound subagent hooks, but it still does not automatically create or route each Feishu group sender/role to an isolated `__mc_*` subagent with its own workspace, skills/config, memory, lifecycle, and admin controls. The KinthAI comment is useful prior-art/workaround context, but it is not an OpenClaw implementation or a canonical replacement.

## Best Possible Solution

Keep the issue open for Feishu plugin product/API design. The likely implementation is an opt-in Feishu group mode that maps sender or role identity to bounded isolated agent/subagent profiles, provisions workspace/agentDir/skills/model/config/memory, routes replies back to the originating group/thread, and adds admin controls plus isolation/security tests and docs. If maintainers prefer this outside the bundled Feishu plugin, first define the plugin/API seam needed for external Feishu routing and provisioning.

## Risks / Open Questions

- This feature crosses group-chat trust boundaries: per-user memory, skills, workspaces, model/config overrides, and admin commands need explicit opt-in authorization, lifecycle limits, and security documentation.
- A ClawHub or external plugin path might be viable only if maintainers expose a stable Feishu routing/provisioning seam; current main does not show a way to alter the bundled Feishu inbound behavior externally.
