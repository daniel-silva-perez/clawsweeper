## Summary

Keep open. Current main has adjacent Feishu sender-scoped group session routing, DM/comment dynamic agent creation, and explicit Feishu subagent thread-binding hooks, but it still does not automatically create or route each Feishu group sender/role into isolated `__mc_*` subagents with their own workspace, skills/config, memory, lifecycle, and admin controls.

## Best Possible Solution

Keep this open for Feishu plugin product/API design. The best path is an opt-in Feishu group mode that maps sender or role identity to bounded isolated agent/subagent profiles, provisions workspace/agentDir/skills/model/config/memory, routes replies back to the originating group or topic, and includes admin controls plus isolation, lifecycle, quota, and security coverage. If maintainers prefer this outside the bundled Feishu plugin, first define the plugin/API seam needed for external Feishu group routing and provisioning.

## Risks / Open Questions

- This crosses trust and security boundaries: per-user memory, workspace access, skills, model/config overrides, admin commands, lifecycle limits, and cost attribution need explicit opt-in policy and tests.
- There may be overlap with narrower Feishu sender-scoped session and thread-binding work, but no current canonical item in the provided context tracks the full automatic per-sender/role provisioning request.
- A ClawHub or external plugin implementation may be viable only after maintainers define a stable Feishu inbound routing/provisioning seam; current main does not show a way for an external plugin to replace the bundled Feishu group-message routing behavior.
