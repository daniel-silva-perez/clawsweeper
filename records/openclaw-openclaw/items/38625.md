## Summary

Keep open. Current main still only partially overlaps the Crab fleet-monitoring request: WebSocket hello exposes a gateway version and admin-scoped `update.run` exists, but `/health` and `/healthz` still return only liveness status, there is no `POST /admin/update` HTTP route, and the typed update RPC still has no `target_version` or `targetVersion` equivalent.

## Best Possible Solution

Keep this issue open until maintainers either implement the requested HTTP API for Crab-style fleet tooling or explicitly document a WS-only management direction. A conservative implementation would add `version` to `/health` and `/healthz` without forcing config load, update probe tests/docs, then decide whether `POST /admin/update` should exist as an authenticated/rate-limited wrapper over `update.run`. If version pinning is accepted, extend the typed update schema and runner target contract rather than adding a one-off HTTP-only parameter.

## Risks / Open Questions

- Adding `POST /admin/update` is security-sensitive because it exposes host package update and restart orchestration over HTTP; it should reuse existing admin scopes, token auth, rate limiting, audit logging, restart coalescing, and update-runner semantics.
- Adding `version` to unauthenticated `/health` and `/healthz` may intentionally trade off fleet observability against version disclosure; maintainers should make that product/security call explicitly.
- The CLI/update runner has internal tag/dev-target concepts, but the public `update.run` schema does not expose version pinning. Adding `target_version` only to HTTP would split the control-plane contract.
