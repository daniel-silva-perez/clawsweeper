## Summary

Current main still does not implement the requested Crab-facing HTTP API. `/health` and `/healthz` still return shallow liveness JSON without `version`, there is no `POST /admin/update` HTTP route, and the typed WS `update.run` params still reject target-version fields. Existing WS surfaces partially overlap through `hello-ok.server.version`, admin-scoped `update.run`, and `update.status`, but they do not satisfy the issue's HTTP health/update request.

## Best Possible Solution

Keep this issue open until maintainers decide the public management direction for Crab-style fleet tooling. A conservative first step is to add a config-independent `version` field to `/health` and `/healthz` with probe/docs/tests updated. For updates, maintainers should either document WS-only `update.run` / `update.status` as the intended integration path or add an authenticated, admin-scoped, rate-limited HTTP wrapper that reuses the typed update runner contract; if target version pinning is accepted, extend the shared protocol/update-runner schema rather than making it HTTP-only.

## Risks / Open Questions

- Adding `POST /admin/update` would expose host package update and gateway restart orchestration over HTTP; auth, admin scopes, rate limiting, audit logging, browser-origin behavior, and target-version validation need maintainer review.
- The current WS update flow only partially overlaps the Crab request. Closing now would leave HTTP-only fleet tools without the requested `/health` version display or REST update trigger.
- Version pinning is not part of the typed `update.run` contract today; adding it only to HTTP would risk diverging HTTP and WS behavior.
- Adding `version` to shallow liveness probes should not force config loading, because current tests require `/healthz` to work before config load.
