## Summary

Keep open. Current main still only partially overlaps the Crab fleet-monitoring request: WebSocket clients can see a gateway version and call `update.run`, but `/health` and `/healthz` still return only `{ ok, status }`, there is no `POST /admin/update` HTTP route, and the typed update RPC still has no `target_version` equivalent.

## Best Possible Solution

Keep this issue open until maintainers either implement the requested HTTP API for Crab-style fleet tooling or explicitly decide and document a WS-only management direction. A conservative implementation would add `version` to `/health` and `/healthz` without forcing config load, update the probe tests/docs, then decide whether `POST /admin/update` should exist as an authenticated, rate-limited wrapper over the existing update runner. If version pinning is accepted, extend the typed update schema and runner tag/target contract rather than adding a one-off HTTP-only parameter.

## Risks / Open Questions

- Adding an authenticated HTTP update endpoint is security-sensitive because it exposes host update/restart orchestration over HTTP; it should reuse existing admin scope, rate limiting, audit logging, update runner semantics, restart handling, and target validation rather than adding ad hoc package-manager execution.
- Exposing `version` on an unauthenticated health endpoint may reveal exact deployment versions to any client that can reach the probe endpoint; that may be acceptable for monitoring but needs an explicit maintainer stance.
- The current product/docs direction already exposes update orchestration through WS RPC, CLI, and Control UI; adding a parallel HTTP admin endpoint needs API/product judgment rather than automated cleanup.
