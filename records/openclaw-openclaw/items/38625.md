## Summary

Current main still does not implement the requested Crab-facing HTTP API. `/health` and `/healthz` remain shallow liveness responses without `version`, there is no `POST /admin/update` HTTP route, and the existing WS `update.run` schema rejects target-version fields. There is useful partial overlap through WS `hello-ok.server.version`, admin-scoped `update.run`, and `update.status`, but that does not satisfy the issue's HTTP health/update request.

## Best Possible Solution

Keep this issue open until maintainers decide the public management direction for Crab-style fleet tooling. A conservative implementation would first add a config-independent `version` field to `/health` and `/healthz` with probe/docs/tests updated. For updates, maintainers should either document the WS-only `update.run` / `update.status` direction or add an authenticated, admin-scoped, rate-limited HTTP wrapper that reuses the typed update runner contract; if target version pinning is accepted, extend the shared protocol/update-runner schema rather than making it HTTP-only.
