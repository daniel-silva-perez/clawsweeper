## Summary

Keep this PR open for maintainer follow-up. Current main still does not implement per-account SOUL files, and the proposed branch is not merge-ready because the new config write surface is not backed by validated channel schemas, runtime bootstrap resolution, account-aware prompt/cache behavior, or guarded path handling.

## What This Changes

The PR adds a `--soul` option for `openclaw channels add`, writes `soulFile` into channel account config from CLI/onboarding, adjusts channel setup registry snapshots, and adds command/setup tests.

## Best Possible Solution

Rework the feature as an end-to-end account-scoped bootstrap contract: add a documented `soulFile` channel/account schema field where supported, validate workspace-relative paths, preserve default `SOUL.md` fallback, pass channel/account context into bootstrap resolution and prompt assembly, make cache identity account-aware, update docs/UI metadata if the config becomes public, and cover default plus named-account behavior with focused regression tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: current main has no `soulFile`/`--soul` implementation, bootstrap loads fixed `SOUL.md`, and write-time validation uses strict schemas that do not declare the PR's new key. No runtime test was run because this was a read-only review.

## Solution Assessment

No. The proposed shape is not yet the best fix because it writes a new config field before defining the schema/runtime contract that makes the field valid and effective. The safer path is to complete the config contract, guarded file resolution, account-aware cache behavior, and tests together.
