## Summary

Keep this PR open. Current main still does not implement per-account SOUL files end to end: the `channels add` CLI has no `--soul` flag, the add/onboarding paths do not persist account-scoped `soulFile`, channel account schemas do not expose that key, and runtime bootstrap still loads fixed workspace `SOUL.md` files rather than a channel/account-specific path. The earlier #42861 attempt is closed unmerged, and the review discussion on this PR already identified the same missing schema/runtime wiring, so the work is neither obsolete nor implemented on main.

## Best Possible Solution

Keep this PR open as a candidate for revised feature work. The best path is to rebase onto current main, add a documented `soulFile` account config contract for supported channels/plugins, update strict schemas and generated config/docs/UI surfaces, wire `channels add` and onboarding to write validated workspace-relative paths, pass channel/account context into bootstrap resolution and prompt assembly, make bootstrap caching account-aware, preserve guarded workspace-bound file reads, and add regression tests for default fallback plus named-account SOUL behavior.

## Risks / Open Questions

- Functional risk in the current PR shape: writing `soulFile` from CLI/onboarding without adding validated channel account schema support and runtime bootstrap resolution would not deliver the requested persona behavior end to end.
- Security review risk: `--soul` is user-controlled path input. A complete implementation needs workspace-bound normalization/guarded reads, clear relative-path semantics, and account-aware cache keys so one channel/account cannot read arbitrary files or reuse another account's SOUL snapshot.
- No supply-chain, CI workflow, lockfile, dependency-source, publishing metadata, or secret-permission changes were visible in the provided PR file list; the security-sensitive surface is the new file path configuration and read behavior.
