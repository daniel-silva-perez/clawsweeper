## Summary

Keep this PR open for maintainer follow-up. Current main still does not implement per-account SOUL files, and the proposed branch is not merge-ready because the saved config field is not backed by channel schemas, runtime bootstrap resolution, account-aware prompt/cache behavior, or guarded path handling.

## What This Changes

The PR adds a `--soul` option to `openclaw channels add`, stores `soulFile` under channel account config from CLI/onboarding, adjusts channel setup registry snapshot lookup, and adds channel add/setup tests.

## Best Possible Solution

Rework the feature as an end-to-end account-scoped bootstrap contract: add a documented `soulFile` channel/account schema field where supported, validate workspace-relative paths, preserve default `SOUL.md` fallback, pass channel/account context into bootstrap resolution and prompt assembly, make cache identity account-aware, update docs/UI metadata if the config becomes public, and cover default plus named-account behavior with focused regression tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: current main has no `soulFile`/`--soul` implementation, bootstrap loads fixed `SOUL.md`, and write-time validation uses strict schemas that do not declare the PR's new key. No runtime test was run because this was a read-only review.

## Solution Assessment

No. The proposed shape is not yet the best fix because it writes a new config field before defining the schema/runtime contract that makes the field valid and effective. The safer path is to complete the config contract, guarded file resolution, account-aware cache behavior, and tests together.

  - body: The new `--soul` branch stores `soulFile` under `channels.<channel>.accounts.<id>`, but config commits revalidate through strict channel account schemas that do not define this key.   - body: This persists a per-account SOUL path, but runtime bootstrap still loads the fixed workspace `SOUL.md` and prompt assembly only reacts to already-loaded `soul.md` basenames. Even after the schema issue is fixed, the configured file will not affect replies unless bootstrap resolution receives channel/account context and reads the guarded file.
  - confidence: 0.87

  - body: `opts.soul.trim()` is accepted as an arbitrary string even though the value is meant to drive prompt file injection. Reject absolute paths, traversal, blank normalized paths, and non-workspace targets before persisting this field so a later reader cannot accidentally load files outside the agent workspace.
  - confidence: 0.78
