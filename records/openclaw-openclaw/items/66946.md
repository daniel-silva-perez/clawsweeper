## Summary

Keep open: current main still has the duplicated Control UI bootstrap config path behavior, so the reported 404 remains a valid narrow gateway/UI bug rather than a cleanup-close candidate.

## What This Changes

The issue reports that opening Control UI under `/__openclaw__/` causes the frontend to request `/__openclaw__/__openclaw/control-ui-config.json`, returning 404 and preventing chat from working.

## Best Possible Solution

Make the bootstrap config endpoint consistently base-path-relative across the shared contract, frontend loader, gateway route, Vite dev stub, tests, and public docs, with explicit coverage for `gateway.controlUi.basePath: "/__openclaw__"`.

## Repair Work Prompt

Fix the duplicated Control UI bootstrap config URL reported in this issue. On current main, `CONTROL_UI_BOOTSTRAP_CONFIG_PATH` is `"/__openclaw/control-ui-config.json"`, while the frontend and gateway both prepend `basePath`; when Control UI is opened at `/__openclaw__/`, the browser can request `/__openclaw__/__openclaw/control-ui-config.json` and get a 404. Make the bootstrap config endpoint consistently base-path-relative across `src/gateway/control-ui-contract.ts`, `ui/src/ui/controllers/control-ui-bootstrap.ts`, `src/gateway/control-ui.ts`, `ui/vite.config.ts`, tests, and `docs/web/control-ui.md`. Add explicit coverage for `gateway.controlUi.basePath: "/__openclaw__"` and consider a compatibility alias for the old endpoint shape so cached bundles or partially upgraded deployments do not break during rollout. Do not change unrelated Control UI routes, tab/base-path inference beyond what the endpoint fix requires, provider/channel behavior, or generated locale bundles. Add a single-line active-version changelog fix entry with proper `Thanks @...` attribution for the reporter and any contributor whose source PR work is reused.
