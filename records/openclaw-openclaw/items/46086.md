## Summary

Keep open. Current main and the latest shipped tree still reject active multimodal memory when `memorySearch.provider` resolves to `local`, while docs and tests encode Gemini Embedding 2 as the supported multimodal path. The remaining decision is product/provider scope: either make local multimodal real or make the unsupported-local guidance clearer.

## What This Changes

The issue asks for `memorySearch.multimodal` to work again with `provider: "local"` after the reporter observed a regression when upgrading from 2026.2.12 to 2026.2.13.

## Best Possible Solution

Choose one explicit contract for multimodal memory. If local multimodal is unsupported, improve validation, doctor/setup text, and docs so local users get a clear migration path to Gemini or a supported provider; if it is supported, add a local adapter capability plus real `embedBatchInputs` handling with dependency-backed tests before relaxing the guard.

## Reproduction Assessment

Yes. A config with `memorySearch.provider: "local"`, active `memorySearch.multimodal.enabled`, and an image or audio modality reaches `resolveMemorySearchConfig` and throws because the local adapter has no multimodal capability; structured indexing would also reject the local provider because it lacks `embedBatchInputs`.

## Solution Assessment

Unclear. The current Gemini-only guard is the narrowest safe implementation for the provider code that exists today, but it does not settle whether the reporter’s local-provider workflow should be restored or explicitly documented as unsupported.
