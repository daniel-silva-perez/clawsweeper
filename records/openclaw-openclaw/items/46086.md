## Summary

Keep open. Current main still rejects active multimodal memory for the local provider because multimodal validation requires an adapter capability declaration, and the bundled local embedding provider only exposes text/token embedding calls. The current Gemini-only path is documented and tested, so this needs maintainer product/provider judgment rather than cleanup closure.

## What This Changes

The issue requests restoring or clarifying `memorySearch.multimodal` behavior for the `local` memory provider after an upgrade made active multimodal settings require Gemini-capable embeddings.

## Best Possible Solution

Choose one explicit memory embedding contract. Either keep local embeddings text-only and improve validation/docs/doctor guidance for `memorySearch.multimodal`, or add real local structured-input embedding support through an adapter that declares `supportsMultimodalEmbeddings` and implements `embedBatchInputs` with dependency-backed tests and docs.

## Reproduction Assessment

Yes. A config with `memorySearch.provider: "local"`, active `memorySearch.multimodal.enabled`, and image/audio modalities follows `resolveMemorySearchConfig` and throws because the local adapter has no multimodal capability declaration; if structured inputs reached indexing, the manager would also reject providers without `embedBatchInputs`.

## Solution Assessment

Unclear. The current Gemini-only guard is the narrowest safe implementation for the code that exists today, but it does not answer the reporter's local-provider regression expectation; maintainers need to decide between unsupported-local guidance and real local multimodal embedding support.
