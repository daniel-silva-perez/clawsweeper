## Summary

Keep open. Current main still rejects `memorySearch.multimodal` for the bundled `local` adapter, and the shipped multimodal path is intentionally scoped to Gemini Embedding 2. The remaining decision is whether maintainers want to document local multimodal memory as unsupported or add a real local structured-input embedding adapter.

## What This Changes

The issue asks to restore or clarify local-provider support for `memorySearch.multimodal`, which the reporter says stopped working after upgrading from 2026.2.12.

## Best Possible Solution

Maintain one clear contract: either explicitly document and validate that bundled local memory embeddings are text-only, or add local multimodal memory only after a local embedding backend can accept structured non-text inputs, with provider capability tests, indexing tests, and docs updated together.

## Reproduction Assessment

Yes. A config with `memorySearch.provider: "local"` and active `memorySearch.multimodal.modalities` follows the same validation path as the existing unsupported-provider test and throws before indexing because the local adapter has no multimodal capability declaration.

## Solution Assessment

Unclear pending maintainer product judgment. The current Gemini-only fail-closed implementation is defensible, but it does not satisfy the reporter’s local-provider expectation; the safer path depends on whether maintainers want unsupported-local documentation or real local structured-input support.
