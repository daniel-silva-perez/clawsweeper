## Summary

Keep open. Current main and the latest release still enforce a Gemini-only multimodal memory contract: active `memorySearch.multimodal` is rejected unless the configured adapter declares multimodal support, the bundled `local` adapter does not declare that support, and the local node-llama embedding path is text-only. That means the reporter’s local-provider regression is not implemented on main, not stale yet, and still needs maintainer product/provider judgment rather than cleanup closure.

## What This Changes

The issue requests restoring or clarifying `memorySearch.multimodal` behavior with `memorySearch.provider: "local"` after upgrades from 2026.2.12 began requiring Gemini.

## Best Possible Solution

Keep one clear memory embedding contract: either explicitly treat bundled local embeddings as text-only everywhere users configure multimodal memory, or add local multimodal support through an adapter that declares `supportsMultimodalEmbeddings` and implements structured `embedBatchInputs`, with provider capability tests, indexing tests, config help, and docs updated together.

## Reproduction Assessment

Yes. A config using `memorySearch.provider: "local"` with active `memorySearch.multimodal.modalities` follows the current validation path and throws because the local adapter has no multimodal capability declaration; the unsupported-provider test covers the same fail-closed path.

## Solution Assessment

Unclear. The current Gemini-only fail-closed implementation is internally consistent and documented, but it does not satisfy the reporter’s local-provider expectation; the best fix depends on whether maintainers want unsupported-local documentation or actual local structured-input support.
