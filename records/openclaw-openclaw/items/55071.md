## Summary

Current main still does not implement skill-driven plugin context visibility. There is no `metadata.openclaw.activates.plugins` type or parser contract, skill snapshots do not carry activation dependencies, and plugin tools are assembled from the active plugin registry plus static allow/deny policy rather than matched skill metadata.

## Best Possible Solution

Keep this issue open for maintainer design and implementation. The likely path is a typed `metadata.openclaw.activates.plugins` contract, parser and snapshot propagation, a concrete pre-model skill activation/matching stage, request-time plugin tool filtering that preserves existing deny/allow/optional precedence, plugin-bundled skill semantics, docs, and focused contract tests.

## Risks / Open Questions

- A real implementation would change request-time tool visibility and must preserve explicit `tools.deny`, plugin disable/deny policy, optional-tool semantics, plugin trust boundaries, provider/model tool compatibility, and prompt-cache stability.
- The proposal needs a precise pre-model activation trigger. Current OpenClaw exposes an available-skills catalog and asks the model to read skills on demand; it does not already have a deterministic matched-skill phase that can safely drive tool schema selection.
