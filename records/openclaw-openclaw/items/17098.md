## Summary

Keep this PR open. Current main still has no xCloud hosting guide or navigation, but the PR head is not merge-ready because it still has misleading provider wording and does not document the hosted Control UI authentication, origin, and device-pairing path needed for an official third-party deployment guide.

## What This Changes

This PR adds an xCloud managed-VPS hosting guide, docs navigation and redirects for `/install/xcloud`, and an xCloud card on the VPS provider page.

## Best Possible Solution

Publish an xCloud guide only after a maintainer or xCloud/provider owner verifies the managed deployment against current OpenClaw Gateway and Control UI contracts, then revise the page to document the exact auth mode, allowed origins, device-pairing or trusted-proxy/Tailscale path, remote-access posture, Telegram allowlist behavior, and credential handling.

## Reproduction Assessment

Not applicable as a docs feature PR rather than a runtime bug. The high-confidence verification path is source and docs inspection: current main lacks xCloud coverage, while the PR head adds the page and exposes the provider wording and hosted-Control-UI gaps.

## Solution Assessment

No, this is not the best way to solve the docs gap as-is. The docs surface is reasonable, but official docs should not endorse the hosted xCloud flow until the provider-specific auth, origin, device-pairing, and credential behavior is verified and documented.
