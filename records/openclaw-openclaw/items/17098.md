## Summary

Keep this PR open. Current main still has no xCloud hosting docs, but the PR is not merge-ready because the proposed guide still has a provider credential wording bug and does not verify or document the hosted Control UI auth, origin, and device-pairing path required by current OpenClaw.

## What This Changes

The PR adds `docs/install/xcloud.mdx`, xCloud redirects and navigation in `docs/docs.json`, and an xCloud card on the VPS provider page.

## Best Possible Solution

Publish an xCloud guide only after a maintainer or xCloud/provider owner verifies the managed deployment against current OpenClaw Gateway and Control UI contracts, then revise the page to document the exact auth mode, origin allowlist, device-pairing or trusted-proxy/Tailscale path, remote-access posture, Telegram allowlist behavior, and credential handling.

## Reproduction Assessment

Not applicable as a docs feature PR rather than a runtime bug. The high-confidence verification path is source/docs inspection: current main lacks xCloud docs, while the PR head adds the page and exposes the provider wording and hosted-Control-UI gaps.

## Solution Assessment

No, the PR is not the best way to solve the docs gap as-is. The docs surface is reasonable, but official docs should not endorse the hosted xCloud flow until the provider-specific auth, origin, device-pairing, and credential behavior is verified and documented.
