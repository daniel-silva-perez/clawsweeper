## Summary

Keep this PR open. Current main still lacks xCloud hosting docs, so the contribution is not obsolete, but the PR head still has provider wording errors and it documents a hosted admin Control UI flow without proving how xCloud satisfies current OpenClaw auth, allowed-origin, and device-pairing requirements.

## What This Changes

The PR adds an xCloud hosting guide under `docs/install`, registers xCloud navigation and redirects in `docs/docs.json`, and adds an xCloud card to the VPS provider picker.

## Best Possible Solution

Publish an xCloud guide only after a maintainer or xCloud/provider owner verifies the managed deployment against current OpenClaw Gateway and Control UI security contracts, then revise the page to document the exact auth, origin, device-pairing, remote-access, Telegram allowlist, and provider-credential behavior.

## Reproduction Assessment

Not applicable as a docs feature PR rather than a runtime bug. The high-confidence verification path is docs/source inspection: current main lacks xCloud coverage, while the PR head adds the page and exposes reviewable wording and hosted-Control-UI gaps.

## Solution Assessment

No, the PR is not the best way to solve this as-is. The docs surface is appropriate, but the content needs provider-wording fixes and explicit, verified hosted Control UI auth/origin/device-pairing details before it becomes official documentation.
