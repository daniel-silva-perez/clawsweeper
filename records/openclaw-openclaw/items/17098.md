## Summary

Keep this PR open. Current main still has no xCloud hosting docs, so the contribution is not obsolete, but the PR is not ready to merge because it still has misleading provider wording and does not document how the hosted Control UI flow satisfies current Gateway auth, allowed-origin, device-pairing, and remote-access requirements.

## What This Changes

This PR adds a new xCloud managed-VPS hosting guide, xCloud redirects and navigation in docs/docs.json, and an xCloud card on the VPS provider page.

## Best Possible Solution

Publish an xCloud guide only after a maintainer or xCloud/provider owner verifies the managed deployment against current OpenClaw Gateway and Control UI contracts, then revise the page to document the exact auth mode, origin allowlist, device-pairing or trusted-proxy/Tailscale path, remote-access posture, Telegram allowlist behavior, and credential handling.

## Reproduction Assessment

Not applicable as a docs feature PR rather than a runtime bug. The high-confidence verification path is source and docs inspection: current main lacks xCloud coverage, while the PR head adds the page and exposes the provider wording and hosted-Control-UI gaps.

## Solution Assessment

No, this is not the best way to solve the docs gap as-is. The docs surface is reasonable, but official docs should not endorse the hosted xCloud flow until the provider-specific auth, origin, device-pairing, and credential behavior is verified and documented.
