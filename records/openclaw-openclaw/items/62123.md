## Summary

Keep open. Current main still does not provide an ordinary plugin hook that awaits async tool-result mutation before transcript/model visibility: `after_tool_call` remains observation-only, `tool_result_persist` remains synchronous, and the async tool-result middleware that does exist is deliberately limited to bundled plugins. The related open PR #64907 and issues #29132/#52849 cover adjacent pre-model result mutation work, but they do not make this async external-plugin request implemented or safe to close.

## What This Changes

The issue requests an awaited plugin hook after tool execution and before transcript/model visibility that can replace or suppress tool results for async validation of external tool output.

## Best Possible Solution

Resolve the overlapping hook requests with a maintainer-owned contract decision: either document that async pre-model tool-result rewriting remains bundled-only, or add a narrow trusted plugin API with explicit ordering, timeout, failure, block/suppress, shape validation, prompt-injection policy, docs, SDK baseline, and runtime coverage across Pi/Codex paths.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: an async `tool_result_persist` handler is ignored by the current hook runner, and an external plugin attempting `registerAgentToolResultMiddleware` is rejected by registry diagnostics and tests.

## Solution Assessment

Unclear. The requested async seam addresses a real gap for secondary scans, but exposing it as a normal external plugin hook is not obviously the safest solution without a trust policy and failure-mode design; the existing bundled middleware shows the safer implementation shape but is intentionally not available to external plugins.
