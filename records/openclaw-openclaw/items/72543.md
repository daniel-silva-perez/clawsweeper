## Summary

Keep this issue open. Current main normalizes `main` aliases for selected settings, but the shared Control UI session option builder still only renders `sessions.list` rows plus the active key, so it can omit `agent:main:main` whenever the user is viewing another session and the store-backed session list does not include the canonical main row.

## What This Changes

The issue requests a stable pinned Control UI chat dropdown option for the canonical main assistant session and consistent normalization of `main` aliases to `agent:main:main`.

## Best Possible Solution

Add a shared Control UI normalization and option-building path that derives the canonical main key from `hello.snapshot.sessionDefaults.mainSessionKey`, prepends a stable main-session option before discovered sessions, keeps desktop and mobile selects aligned, and covers the non-main active-session case with focused tests.

## Reproduction Assessment

Yes. A focused unit reproduction can call `resolveSessionOptionGroups` with active session `agent:main:subagent:x` and a `sessionsResult` that lacks `agent:main:main`; current code returns only the supplied rows plus the active key, so the canonical main option is absent.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix if it is driven from Gateway `sessionDefaults` rather than a literal persona string. The safest implementation is to inject through the shared option-group builder and reuse the same normalization for selected and persisted session keys.
