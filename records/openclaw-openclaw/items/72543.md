## Summary

Keep this issue open. Current main has useful alias normalization for `main`, but the shared chat session option builder still only renders rows returned by `sessions.list` plus the active session key, so it does not always synthesize or pin the canonical main session when the user is viewing another session.

## What This Changes

The issue requests that the Control UI chat session dropdown always include and normalize a stable canonical main session entry for `agent:main:main`.

## Best Possible Solution

Add a shared Control UI helper that derives the canonical main key from `hello.snapshot.sessionDefaults.mainSessionKey`, prepends a stable main-session option before discovered sessions, and reuses the same normalization for selected and persisted session keys. Cover the shared option builder so desktop and mobile dropdowns stay aligned.

## Reproduction Assessment

Yes. A focused unit reproduction can call `resolveSessionOptionGroups` with an active non-main `sessionKey` and a `sessionsResult` that lacks `agent:main:main`; current code returns only the supplied rows plus the active session, so the canonical main option is absent.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix if it is driven from Gateway `sessionDefaults` rather than a literal persona string. The safer implementation is to inject through the shared option-group builder, not duplicate desktop and mobile dropdown logic.
