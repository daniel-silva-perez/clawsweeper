## Summary

Keep open. Current main and the latest tagged release still call bundled MCP tools without passing MCP request timeout options, while the SDK version pinned by the repo defaults requests to 60,000 ms. Open PR #73536 is an unmerged implementation candidate with closing syntax, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Request configurable bundle MCP tool-call timeouts so long-running MCP tools are not capped by the SDK's 60-second default.

## Best Possible Solution

Keep the issue open and review PR #73536 as the canonical implementation candidate. The desired end state is for bundle MCP sessions to resolve a per-server request timeout, pass it to `Client.callTool(..., undefined, { timeout })`, and add focused test/docs coverage for the chosen config semantics.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path exists: configure a bundle MCP server with a timeout above 60 seconds and call a tool that runs longer than 60 seconds; current main forwards no MCP `RequestOptions.timeout`, so the SDK default applies.

## Solution Assessment

No for current main. The narrow maintainable fix is to pass a resolved per-server timeout into the MCP SDK call path, with maintainer review on whether that timeout should reuse `connectionTimeoutMs` or be a separate public config field.
