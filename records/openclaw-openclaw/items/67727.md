## Summary

Keep this PR open. Current main still does not implement `messages.inbound.botDebounceMs` or bot-aware inbound debounce resolution, and the PR remains a plausible focused implementation path. The main blocker is product completeness: the proposed branch adds config/resolver support, but the PR discussion explicitly defers production channel call-site wiring, so users would not get bot-specific debounce behavior on normal traffic until provider-native bot-origin state is threaded through.

## Best Possible Solution

Keep this PR open for maintainer review. Either explicitly land it as preparatory config/SDK work with a clear follow-up item, docs/changelog, generated schema parity, and no claim of completed production behavior, or complete the product path before merge by threading per-message `senderIsBot`/bot-origin state through shared and direct inbound debounce paths for the relevant channel plugins, using provider-native bot flags and focused production-path tests.

## Risks / Open Questions

- Merging this as a complete user-facing feature would expose `messages.inbound.botDebounceMs` while ordinary production channel traffic still uses global or per-channel debounce windows unless call sites pass reliable bot-origin state.
- Bot-origin detection should come from provider-native flags such as Slack/Discord/Telegram event metadata, not from spoofable text, display names, or message content conventions.
- Because `resolveInboundDebounceMs` is exported through plugin SDK channel debounce surfaces, the final API shape should be treated as a backwards-compatible plugin-facing contract and covered with targeted tests/docs before merge.
