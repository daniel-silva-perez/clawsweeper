## Summary

Keep this issue open. Current main still stores the original provider message id on the initial queued FollowupRun, but collect and overflow drains do not carry messageId forward, followup runtime does not pass queued.messageId as currentMessageId, and followup delivery does not apply reply threading with a queued currentMessageId. The related PR #36235 is closed and unmerged, so no canonical merged fix exists.

## Best Possible Solution

Implement a narrow core queue/followup continuity fix: normalize queued FollowupRun.messageId values, include trusted per-item message_id metadata in collect prompt blocks, propagate messageId through collect and overflow summary drains, pass queued.messageId into runEmbeddedPiAgent as currentMessageId, and use that currentMessageId in followup delivery reply threading. Keep this generic in core with no Feishu-specific policy, and cover collect-mode, summary drain, followup-runner, and followup-delivery regressions.
