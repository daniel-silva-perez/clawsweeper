## Summary

Keep this issue open. Current main still captures Feishu/Lark's original platform message_id and stores it on the initial queued FollowupRun, but collect/summary drains and followup execution do not preserve that id into queued prompt metadata, runtime currentMessageId, or reply threading. The two directly related fix PRs, #36222 and #36235, are closed and unmerged.

## Best Possible Solution

Implement a narrow queue/followup continuity fix in core: normalize FollowupRun.messageId, include trusted per-item message_id metadata in collect prompt blocks, propagate a suitable queued messageId through summary/single-message drains, pass queued.messageId into runEmbeddedPiAgent as currentMessageId, and preserve replyToId through followup delivery so Feishu and other threaded channel adapters can use platform-native reply APIs.

## Clownfish Work Prompt

Fix the queued message_id continuity bug for this issue. Current main captures Feishu/Lark's provider message_id at ingress and stores it as FollowupRun.messageId, but collect/summary drains and followup execution lose it before tools/native replies can target the original message.

Expected boundary:
- In `src/auto-reply/reply/queue/drain.ts`, normalize queued `messageId` values and preserve them through collect and summary drains.
- Add trusted per-item metadata blocks to collect prompts, e.g. a small JSON block containing `{ "message_id": "<original-provider-id>" }`, sourced from `FollowupRun.messageId`, not from untrusted prompt text.
- For single-message or otherwise unambiguous followups, propagate the normalized queued `messageId` onto the synthetic `FollowupRun`.
- In `src/auto-reply/reply/followup-runner.ts`, pass normalized `queued.messageId` into `runEmbeddedPiAgent` as `currentMessageId`.
- Ensure followup delivery/threading can use that current message id so Feishu outbound receives `replyToId` and can call native `im.message.reply` with the original platform id.
- Do not change queue dedupe policy, routing policy, channel adapter APIs, provider defaults, or Feishu auth/config behavior.
- Be careful with collect batches containing multiple provider ids: preserve per-item ids in prompt metadata rather than pretending the whole batch has one reply target unless the target is unambiguous.

Related refs: this issue, #36222, #36235, and adjacent #54456.
