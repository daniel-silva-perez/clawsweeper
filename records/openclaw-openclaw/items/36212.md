## Summary

Keep this issue open. Current main still records the original provider message id on the initial FollowupRun, but collect drains, summary drains, followup runtime setup, and followup reply-threading do not propagate that id where the agent or Feishu reply delivery can use it. The related fix PRs are closed and unmerged in the provided GitHub context.

## Best Possible Solution

Implement a narrow core queue/followup continuity fix: normalize FollowupRun.messageId, add trusted per-item message_id metadata to collect prompt blocks, propagate a suitable messageId through summary/single-message drains, pass queued.messageId into runEmbeddedPiAgent as currentMessageId, and feed currentMessageId into followup delivery reply-threading. Cover collect-mode, summary drain, and followup-runner behavior with focused regression tests; avoid Feishu-specific core logic.
