## Summary

Keep open. Current main still implements fallback as provider/model candidate selection and reuses the prepared run context, while the issue discussion already has recent maintainer/Codex comments saying this needs design rather than cleanup closure.

## What This Changes

Requests a fallback path that can switch to a different agent, workspace, or context profile instead of sending the primary agent context to smaller fallback models.

## Best Possible Solution

Design a first-class fallback-context contract, then update config types, strict schemas, schema help/docs, fallback candidate metadata, reply-runner/session plumbing, and regression tests so fallback retries can use alternate context without corrupting the original agent state.

## Reproduction Assessment

Yes: static reproduction is enough. The proposed fallbackAgent, fallbackWorkspace, contextProfiles, bootstrapFiles, and per-model bootstrap budget fields are absent from strict config surfaces, and fallback candidates only carry provider/model while retry execution reuses the original run context.

## Solution Assessment

No current-main implementation solves the issue. The requested agent-switching shape may be valid, but maintainers should first choose between fallbackAgent, fallbackWorkspace, and model-specific context-profile semantics before an implementation PR.
