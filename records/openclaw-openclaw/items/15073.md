## Summary

Keep this issue open. Current main still implements model fallback as provider/model candidate selection only. It supports per-agent model fallback string lists, but fallback attempts continue to use the original agent id, workspace, agentDir, session file, runtime config, and skills snapshot, so the requested fallback-agent/workspace/context-profile behavior remains missing.

## Best Possible Solution

Keep this open for maintainer design. The best path is to choose a first-class core contract, either fallbackAgent/fallbackWorkspace semantics or a narrower model-specific context profile/budget mechanism. A complete implementation likely needs config types, strict schemas, schema help/docs, fallback candidate metadata carrying context identity, reply-runner/session plumbing that can safely swap agentDir/workspaceDir/sessionFile/skills/context per fallback attempt, and regression tests proving fallback retries use alternate context without corrupting the original session.

## Risks / Open Questions

- A fallback-agent implementation has session-safety risk: swapping agentDir, workspaceDir, or sessionFile mid-run could corrupt or misattribute the original session unless ownership semantics are explicit.
- Changing fallback context can alter tool and skill visibility; maintainers need to decide whether fallback attempts inherit requester permissions or use the target fallback agent's policy.
- The issue proposes several viable designs, so implementing one without product/API direction could lock in the wrong public config contract.
