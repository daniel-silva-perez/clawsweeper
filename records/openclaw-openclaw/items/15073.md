## Summary

Keep this issue open. Current main still supports per-agent model fallback as provider/model string selection only; fallback attempts reuse the same agent id, workspace, agentDir, session file, runtime config, and skills snapshot, so the requested fallback-agent/workspace/context-profile behavior is still missing.

## Best Possible Solution

Keep this issue open for maintainer design. The best path is to choose a first-class core contract, either fallbackAgent/fallbackWorkspace semantics or a narrower model-specific context profile/budget mechanism. A complete implementation likely needs config types, strict schemas, schema help/docs, fallback candidate metadata carrying context identity, reply-runner/session plumbing that can safely swap agentDir/workspaceDir/sessionFile/skills/context per fallback attempt, and regression tests proving fallback retries use alternate context without corrupting the original session.

## Risks / Open Questions

- A fallback-agent implementation has session-safety risk: swapping agentDir/workspaceDir/sessionFile mid-run could corrupt or misattribute the original session unless the session ownership model is explicit.
- Changing fallback context can alter tool and skill visibility; maintainers need to decide whether fallback should inherit requester permissions or use the target fallback agent's policy.
- The issue offers several viable designs (fallbackAgent, fallbackWorkspace, per-model bootstrap budgets, contextProfiles), so an implementation PR needs product/API direction before code starts.
