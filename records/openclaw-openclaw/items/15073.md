## Summary

Keep open. Current main still implements fallback as provider/model candidate selection, not as a fallback-time switch to another agent, workspace, session, skills snapshot, or context profile; the recent maintainer discussion also treats this as an API/session-design item rather than cleanup closure.

## What This Changes

The issue requests a per-agent fallback mode that can use a different agent, workspace, or model-specific context profile when the primary model fails.

## Best Possible Solution

Design a first-class core fallback-context contract, then update config types, strict schemas, schema help/docs, fallback candidate metadata, reply-runner/session plumbing, and regression tests so fallback retries can use alternate context without corrupting the original agent state.
