## Summary

Keep open. Current main still implements fallback as provider/model candidate selection while reusing the current run's agent identity, workspace, session file, agentDir, and skills snapshot; the recent maintainer discussion also explicitly kept this as design work rather than cleanup closure.

## What This Changes

The issue requests a per-agent or per-context fallback mechanism so a failed primary model can switch to a smaller/minimal agent workspace instead of reusing the original rich context.

## Best Possible Solution

Design a first-class fallback-context contract, then update config types, strict schemas, schema help/docs, fallback candidate metadata, reply-runner/session plumbing, and regression tests so fallback retries can use alternate context without corrupting the original agent state.

## Reproduction Assessment

Yes for a focused static reproduction: the proposed fallbackAgent/fallbackWorkspace/context-profile fields are rejected or ignored by current strict config surfaces, and fallback candidates only carry provider/model. A live rate-limit run is not needed to prove the missing configuration and plumbing.

## Solution Assessment

No current implementation is not the best solution because the requested behavior is still missing. The best next step is maintainer API/session design before any automated patch chooses between agent switching, workspace override, or model-specific context profile semantics.
