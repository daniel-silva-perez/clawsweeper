## Summary

Keep open. Current main still only falls back across provider/model candidates and reuses the already-prepared run context, so it does not satisfy the request to switch agent identity, workspace, agentDir, session file, skills snapshot, or a model-specific context profile during fallback.

## What This Changes

The issue requests a fallback mode where a failed primary model can switch to another agent or smaller context/workspace instead of reusing the original rich agent context.

## Best Possible Solution

Design a first-class fallback-context contract, then update config types, strict schemas, schema help/docs, fallback candidate metadata, reply-runner/session plumbing, and regression tests so fallback retries can use alternate context without corrupting the original agent state.

## Reproduction Assessment

Yes for a focused static reproduction: the proposed fallbackAgent, fallbackWorkspace, contextProfiles, bootstrapFiles, and per-model bootstrap budget fields are rejected or ignored by current strict config surfaces, and fallback candidates only carry provider/model. A live rate-limit run is not needed to prove the missing configuration and plumbing.

## Solution Assessment

No current implementation is not the best solution because the requested behavior is still missing. The best next step is maintainer API/session design before any automated patch chooses between agent switching, workspace override, or model-specific context profile semantics.
