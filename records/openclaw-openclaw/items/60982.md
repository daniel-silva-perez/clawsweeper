## Summary

Keep open. Current main does not implement the PR's intended Discord final-reply sanitization: final Discord delivery delegates the original reply payloads to generic outbound delivery, the Discord outbound adapter does not register a sanitizer, and draft preview finalization still only strips inline directive tags. The PR is a plausible fix direction, but it still needs maintainer review and resolution of the sanitizer/preview/media edge cases raised in review.

## Best Possible Solution

Keep this PR open and finish it as a Discord-owned fix: add a code-region-aware outbound sanitizer through the Discord delivery/adapter path, apply the same behavior to draft preview final edits without dropping existing previews or media finals, and land focused regression tests for final text, preview edits, `<thinking>`, `<relevant-memories>`, code fences, and sanitized-empty media payloads.

## Risks / Open Questions

- Landing the PR as-is may still corrupt legitimate Discord output if role-marker or separator stripping is broader than intended, especially around code blocks or prose endings.
- The review discussion reports a draft-stream media edge case where sanitized-empty text can mark a final as handled before attachments are sent; that needs confirmation or a fix before merge.
- Leaving the PR unlanded means Discord final replies on current main can still expose assistant-internal scaffolding when it arrives in a non-reasoning final payload.
