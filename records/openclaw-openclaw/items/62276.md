## Summary

Keep open. Current main has better gateway port diagnostics and restart-loop mitigations, but there is not enough evidence to close this as implemented, duplicate, or stale. This issue includes a native Windows foreground-vs-service port collision, plus a later Linux reproduction where a user systemd unit installed by onboarding conflicts with an existing system-level service. Current main still treats the foreground collision as a GatewayLockError path with stop hints; it does not clearly prevent or resolve the competing-service topology described in the discussion.

## Best Possible Solution

Keep this issue open as a gateway lifecycle and daemon-collision report. The best maintainer path is to reproduce or request focused status output for native Windows after enabling dreaming and for Linux with a custom system-level service plus `openclaw onboard --install-daemon`; then either add installer/doctor preflight that detects an existing gateway service/listener on the configured port and warns or skips the user-unit install, or narrow the issue to support/docs if the only remaining behavior is an intentional foreground start while a healthy managed gateway is already running.
