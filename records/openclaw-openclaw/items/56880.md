## Summary

Keep open. Current main still documents and implements one active run per session, with active followups either steered into the current run or queued for later rather than spawned as independent concurrent turns. The issue is also already paired with an open, unmerged closing PR, so it should remain open until that PR is reviewed and merged, replaced, or closed.

## What This Changes

The issue requests concurrent or fast-lane handling for new messages in an active chat session so short follow-ups are not blocked behind a long-running agent turn.

## Best Possible Solution

Keep the issue open while maintainers review the linked fast-lane PR or replace it with a narrower design that preserves the one-writer-per-session safety model while allowing clearly bounded quick replies, interrupt, or steering behavior.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: while a session run is active, the current queue policy and runner enqueue followups instead of starting a second independent turn, and the Discord docs state queued runs preserve per-session ordering.

## Solution Assessment

Unclear. A constrained fast lane may be a maintainable answer, but full concurrent turns within one session raise context coherence and tool/file collision risks; the safest path is review of the open implementation PR against those constraints.
