## Summary

Current main has several adjacent robustness mechanisms, but it still does not implement the central request: a first-class Gateway process watchdog with configurable restart-window/auto-stop/notification behavior, a user-configurable global error recovery strategy map, richer process/error counters in health output, or a deliberate degraded-startup mode beyond existing scoped config/plugin recovery.

## Best Possible Solution

Keep this open as a maintainer-owned reliability umbrella, or split it into focused tasks: define a gateway.watchdog config/schema/docs surface for process restart-loop detection and auto-stop/notification behavior; decide whether user-configurable global error recovery strategies are wanted; enrich Gateway health/readiness with safe process/error counters; and design degraded startup semantics that build on existing last-known-good config recovery, plugin-local degraded startup, diagnostics bundles, and channel health monitoring.

## Risks / Open Questions

- The issue bundles several separable product decisions; implementing it as one PR would likely be too broad for review.
- A process-level watchdog must compose carefully with systemd/launchd/Docker supervisors, existing restart deferral, shutdown diagnostics, last-known-good config recovery, and channel health monitoring.
- Richer health counters need a privacy-safe public contract so monitoring data is useful without exposing sensitive runtime, channel, or session details.
- Configurable global error recovery strategies could mask real crashes or create retry loops unless the fatal/non-fatal boundary is explicitly designed and tested.
