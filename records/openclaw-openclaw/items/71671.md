## Summary

Keep this PR open. Current main still counts non-contiguous no-progress records because `getNoProgressStreak` skips over intervening tools and missing-result records, while the PR is a focused tail-semantics fix with matching regression coverage and no blocking review findings.

## What This Changes

The PR changes no-progress streak counting to stop at the current repeated tool-call tail, adds two loop-detection regression tests, and adds a changelog entry.

## Best Possible Solution

Land this scoped tail-semantics fix, or an equivalent maintainer-authored patch, with regression coverage for both known polling no-progress detection and the global no-progress circuit breaker while preserving existing generic-repeat warning policy.

## Reproduction Assessment

Yes. A pure unit reproduction is to record repeated identical `process` poll outcomes to the critical threshold, record another tool outcome, then ask `detectToolCallLoop` about the same `process` retry; current main still counts the stale earlier poll records.

## Solution Assessment

Yes. Changing `getNoProgressStreak` to stop at the first non-matching or missing-result tail record is the narrowest maintainable fix for this false positive and matches nearby contiguous-tail helpers without changing broader repeat-warning policy.
