## Summary

Keep open. The PR targets a real Control UI/Gateway thinking-level limitation, but the patch does not implement its stated all-level behavior for profiled providers because non-empty provider profiles return before the expanded fallback list is used.

## What This Changes

The PR expands shared and UI fallback thinking-level lists with `xhigh`, `adaptive`, and `max`, adjusts ranks and labels, updates thinking tests, and changes the thinking docs to describe all-level dropdown behavior for non-binary providers.

## Best Possible Solution

Maintainers should decide whether pickers should stay model-accurate or intentionally expose remappable levels. If the all-level behavior is desired, implement it in a central post-processing seam after provider-profile normalization, or update provider-owned profiles consistently, with coverage for at least one real profiled provider plus the Gateway row and Control UI selector path.

## Reproduction Assessment

Yes. A high-confidence static reproduction is available: with any active non-empty provider profile such as OpenAI, `resolveThinkingProfile` returns before the expanded base profile, and Gateway rows pass that exact list to the Control UI.

## Solution Assessment

No. Expanding only fallback lists and docs is not the best solution because it misses the main provider-profile path; a documented provider-profile post-processing policy or provider-owned profile update is safer after maintainer contract review.
