## Summary

Keep open. The PR is not ready because the expanded base list does not affect the main provider-profile path: `resolveThinkingProfile` returns non-empty provider profiles before reaching the base fallback, so profiled providers such as OpenAI and GitHub Copilot would still not gain the advertised `adaptive`/`max` dropdown options. Greptile's dead-code note is directionally right, but the blocking issue is that the branch does not implement its stated behavior for the active Gateway/UI path.

## What This Changes

The branch expands shared and UI fallback thinking-level lists with `xhigh`, `adaptive`, and `max`, adjusts ranks and labels, and updates thinking tests and docs for all-level dropdown behavior.

## Best Possible Solution

Maintainers should first decide whether the Control UI should remain model-accurate or deliberately expose remappable levels. If the all-level behavior is desired, implement it in the provider-profile path or in a documented central post-processing seam, then cover at least one real profiled provider plus the Gateway row/UI selector path.

## Reproduction Assessment

Yes. A high-confidence static reproduction is available: with any non-empty provider profile, `resolveThinkingProfile` returns at `src/auto-reply/thinking.ts:168`, and Gateway/UI rows render those returned `thinkingLevels`; OpenAI's current profile still lacks `adaptive` and `max`.

## Solution Assessment

No. Expanding the fallback base list is not the narrowest maintainable solution because it misses explicit provider profiles and changes docs toward a contract the runtime does not actually enforce for main providers.
