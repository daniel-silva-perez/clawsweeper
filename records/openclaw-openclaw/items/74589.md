## Summary

Keep open. Current main still has the Anthropic-compatible payload boundary that can forward an empty converted message array, and the issue is paired with open closing PR #74731, so it should remain open until that PR is reviewed and merged or rejected.

## What This Changes

MiniMax M2.7 provider requests should stop sending an empty Anthropic Messages `messages` array after tool-call replay or message filtering.

## Best Possible Solution

Review #74731 as the narrow transport-boundary fix, require regression coverage for empty or fully filtered converted Anthropic messages, and close this issue only after that PR or an equivalent current-main fix lands. Keep the broader `/new` empty-prompt family tracked separately under #73926.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No for a deterministic end-to-end MiniMax session, because the report is intermittent and does not include a minimal transcript. There is a high-confidence code-level path: empty or fully filtered replay history can produce `messages: []` in the Anthropic-compatible payload builder on current main.

## Solution Assessment

Is this the best way to solve the issue? Yes, the best path is a final guard in the Anthropic-compatible transport payload builder, because it is narrower and more reliable than mutating session history or injecting broad provider-specific recovery policy elsewhere.
