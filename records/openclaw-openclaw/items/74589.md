## Summary

Keep open. Current main still builds Anthropic-compatible request payloads by directly forwarding the converted message array, and the conversion path can return an empty array when the replay context is empty or fully filtered. The issue is also paired with open closing PR #74731, so it should remain open until that PR or an equivalent fix is reviewed and merged or rejected.

## What This Changes

The issue requests a MiniMax/Anthropic-compatible provider-boundary guard so tool-call replay cannot send `messages: []` and trigger MiniMax 400/2013 empty-chat-content failures.

## Best Possible Solution

Review and land #74731, or an equivalent final guard in `src/agents/anthropic-transport-stream.ts`, with regression coverage for empty and fully filtered converted Anthropic messages. Keep the broader `/new` empty-prompt family tracked separately under #73926.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No for a deterministic live MiniMax session, because the report is intermittent and lacks a minimal transcript. Yes for the code-level failure path: an empty or fully filtered Anthropic replay context can still produce `messages: []` on current main.

## Solution Assessment

Is this the best way to solve the issue? Yes, a final Anthropic-compatible transport-boundary guard is the narrowest maintainable fix because MiniMax M2.7 uses that route and earlier prompt guards do not prove the converted provider payload is non-empty.
