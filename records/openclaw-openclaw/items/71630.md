## Summary

Keep this open. Current main still does not render an always-visible In progress/Interrupted/Done runtime chip in the Control UI composer, and the broader gateway-owned activity API remains a separate open dependency rather than an implementation of this narrower UI request.

## What This Changes

Add a compact session-scoped runtime status indicator beside the Control UI composer voice/file actions showing at least in-progress, interrupted, and done states.

## Best Possible Solution

Implement a focused Control UI composer chip driven by explicit local chat lifecycle state now, fail closed to Interrupted when reconnect continuity is uncertain, and leave the cross-client authoritative activity model to the separate gateway API work.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Static verification against current main is high-confidence because the composer render path, run controls, reconnect cleanup, and exported gateway methods do not provide the requested status chip.

## Solution Assessment

Yes. A narrow UI implementation is the best first step as long as it uses existing lifecycle signals, avoids transcript messages, and does not try to solve the broader sessions.activity contract in the same change.
