## Summary

Current main already covers the reported gateway MCP process leak with shipped lifecycle cleanup: stdio MCP transports terminate process trees, session reset/delete retires the session MCP runtime, idle/daily rollovers retire previous runtimes, and an idle TTL sweep catches orphaned session-scoped runtimes. The core safety-net commit shipped before the current latest release, so the stale v2026.4.9 behavior no longer matches supported current behavior.

## What This Changes

The issue reports that gateway sessions can leave stdio MCP child processes such as `vibe-trading-mcp` running after session end.

## Best Possible Solution

Keep the shipped generic MCP lifecycle handling rather than adding `vibe-trading`-specific cleanup: session-owned runtimes should retire through the shared manager, transports should kill their process trees, and the idle TTL sweep should remain the backstop for missed explicit cleanup paths.

## Reproduction Assessment

Yes. The report gives a concrete affected version, channel path, stdio MCP process symptom, and code path; current main has focused tests and source checks for the cleanup paths, though no live external Feishu/vibe-trading process repro was run here.

## Solution Assessment

Yes. The current implementation is the maintainable fix because it cleans the generic session-scoped MCP runtime and stdio process tree across reset/delete, rollover, shutdown, and idle orphan cases without baking a service-specific MCP server name into core.
