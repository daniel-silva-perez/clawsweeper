## Summary

Keep open. Current main still has the ASCII-only Discord outbound mention matcher, so the Unicode/CJK rewrite remains useful, but this PR is not merge-ready because the new CJK URL guard still misses Markdown-style link tokens. The broader open #59818 also overlaps and needs maintainer choice before landing one path.

## What This Changes

The branch expands Discord outbound mention rewriting to Unicode/CJK handles, adds regression coverage for email, URL, code-span, reserved, and already-formatted mention cases, and still shows Yuanbao catalog pinning/contract updates in the PR file list.

## Best Possible Solution

Land a Discord-only parser and regression-test update after the URL-context guard handles Markdown/link URL tokens, then either close or explicitly defer the broader multi-account cache behavior to #59818.

## Reproduction Assessment

Yes. Current main can be checked by seeding the Discord directory cache with `张三` and calling `rewriteDiscordKnownMentions('你好@张三，收到请回复。', { accountId: 'default' })`; the source regex also directly shows why it cannot match. A focused PR-regex probe reproduces the remaining Markdown-link URL corruption risk.

## Solution Assessment

No, not yet. The narrow parser direction is maintainable, but the current PR should skip CJK matches inside Markdown/link URL tokens and be coordinated against #59818 before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Skip CJK matches inside Markdown link URLs:** `extensions/discord/src/mentions.ts:104-108`
  - body: The URL guard only strips simple leading wrappers from the whitespace token. For `[资料](https://x.com/你好@张三)`, the token becomes `资料](https://...)`, so `URL_SCHEME_TOKEN_PATTERN` does not match and the CJK pass rewrites `@张三`, corrupting the link. Detect URL schemes after Markdown/link prefixes or skip link tokens before rewriting.
  - confidence: 0.87
