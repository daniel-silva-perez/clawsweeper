## Summary

Keep this PR open, but do not merge it as-is. The underlying cleanup is still partly useful on current main, yet the proposed orphan sweep can delete state for an active long-running tool call and break observable hook, messaging, and cron bookkeeping.

## What This Changes

The PR removes local `clamp` and `isRecord` helpers in favor of shared utilities and adds per-run idle timestamp sweeping for `toolStartData` in embedded tool handlers.

## Best Possible Solution

Revise the PR into a focused handler fix that keeps active tool calls protected, preferably by tracking per-run in-flight tool counts or an equivalent lifecycle guard before sweeping orphaned entries. Keep the harmless `clamp` dedupe if it still applies, drop or rebase the obsolete doctor hunk, and add regression tests for normal start/end cleanup, abandoned entries, active tools running beyond the sweep threshold, hook duration/params, messaging-send detection, and cron.add tracking.

## Reproduction Assessment

Yes. The PR bug has a deterministic event sequence: start one tool, let more than five minutes pass without an end event, start any other tool so the sweep runs, then finish the first tool and observe that `startData` is gone.

## Solution Assessment

No. The utility imports are a narrow cleanup, but a run-idle timestamp is not enough to distinguish an abandoned run from an active long-running tool call; the safer solution is to guard sweeping with explicit in-flight state or a lifecycle signal that proves no tool call is active.

## Review Findings

Full review comments:

- **[P2] Track in-flight tools before sweeping runs:** `src/agents/pi-embedded-subscribe.handlers.tools.ts:67-76`
  - body: `lastRunActivity` is only refreshed on tool start/end, so a single tool call running longer than five minutes looks idle and can be purged when any later tool start triggers the sweep. When that call ends, `startData` is gone, so `after_tool_call` loses duration/original params and cron.add or messaging-send bookkeeping can be skipped. Track active tool counts, or otherwise prove a run has no in-flight calls, before deleting its entries.
