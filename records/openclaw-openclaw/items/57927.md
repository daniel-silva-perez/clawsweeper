## Summary

Keep this PR open. Current main still does not implement the PR's central orphan-cleanup behavior for `toolStartData`: entries are only inserted on tool start and removed on matching tool end. The doctor `isRecord` cleanup has effectively moved into the newer shared normalizer path, but `pi-tools.read.ts` still has the local duplicate `clamp`. The proposed PR sweep also has an unresolved correctness finding: its run-idle heuristic can evict state for an active tool call that runs longer than five minutes.

## Best Possible Solution

Keep this PR open and request a focused revision: keep or split the harmless utility dedupe, drop or rebase the stale doctor hunk against the shared normalizer structure, and replace the orphan cleanup with a design that tracks active in-flight tool calls separately from abandoned runs. Add targeted tests for normal start/end cleanup, abandoned entries, active long-running tools past the cleanup threshold, `after_tool_call.durationMs`, messaging-send detection, and `cron.add` success tracking.
