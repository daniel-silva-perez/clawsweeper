## Summary

Keep open. Current main still leaves meaningful parts of the reported Telegram delivery failure cluster unresolved: permanent-error classification is incomplete, queue enqueue has no stable logical-send identity, and Telegram message-tool sends can still bypass the chunk-aware outbound path.

## What This Changes

Stabilize Telegram delivery retries by avoiding duplicate queue entries for one logical send, permanently failing unrecoverable Telegram 400/403 errors, and pre-splitting long Telegram send-message payloads.

## Best Possible Solution

Land one focused bug-fix PR that adds a stable logical-send identity at the queue boundary, expands permanent delivery classification for unrecoverable Telegram 400/403 responses, and routes Telegram message-tool sends through the same chunk-aware path as normal outbound Telegram replies.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the permanent-error and long-message paths: current-main tests can assert that `400: Bad Request: message is too long` remains pending, and a Telegram message action above 4096 characters reaches a single direct `sendMessageTelegram` default-path send. The duplicate UUID behavior has strong incident evidence, but should be pinned with a new regression around the higher-level retry trigger.

## Solution Assessment

Is this the best way to solve the issue? Yes. The narrow maintainable fix is to reuse existing outbound chunking and recovery seams while adding queue-level idempotency, not to add Telegram-only retry loops or silent truncation.
