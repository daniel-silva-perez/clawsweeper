## Summary

Keep open. Current main still leaves meaningful parts of this Telegram delivery bug cluster unresolved: the permanent-error matcher does not include Telegram `message is too long` or the reported bot/user 403 variants, queue enqueue still creates a new UUID per enqueue with no stable logical-send key, and Telegram message actions can still bypass the chunk-aware outbound adapter.

## What This Changes

Fix Telegram delivery queue retry idempotency, permanent-failure handling for Telegram 400/403 errors, and long Telegram send-message splitting before the Bot API call.

## Best Possible Solution

Land one focused fix that adds a stable logical-send identity at the queue/retry boundary, extends permanent Telegram error classification for doomed 400/403 responses, and routes Telegram message-action sends through the same chunk-aware delivery behavior as normal outbound Telegram replies.

## Reproduction Assessment

Yes. The permanent-error and long-message paths are reproducible from current code inspection with focused tests: `400: Bad Request: message is too long` currently misses the permanent matcher, and a Telegram action/default send above 4096 characters is not chunk-planned before `sendMessageTelegram`.

## Solution Assessment

Yes. The requested direction is the narrow maintainable fix if implemented carefully: keep retry accounting at the queue boundary, classify provider-permanent Telegram errors before retrying, and reuse Telegram's existing chunking seam instead of adding another truncation path.
