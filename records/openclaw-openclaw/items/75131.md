## Summary

Keep open. Current main still has only partial safeguards: recovery has a permanent-error path, and normal Telegram outbound delivery can chunk text, but the reported Telegram oversized/403 failures are not classified, the Telegram message action still reaches a non-chunked default send path, and queue enqueue still creates a new UUID for each enqueue without a stable logical-send key.

## What This Changes

The issue requests idempotent delivery queue retries, permanent classification for Telegram oversized/forbidden delivery failures, and Telegram message pre-splitting at the Telegram text limit.

## Best Possible Solution

Land one focused fix that adds a stable logical-send identity or dedupe key at the delivery queue boundary, extends permanent Telegram failure classification for deterministic Bot API 400/403 cases, and makes Telegram message-tool sends reuse the existing chunk-aware outbound send behavior instead of a separate direct-send path.

## Reproduction Assessment

Yes for the permanent-failure and long-message action paths: current main can be checked with a recovery test using `400: Bad Request: message is too long` and a Telegram `sendMessage` action over the text limit. The duplicate-UUID incident is strongly evidenced by the report, but still needs a focused regression test for the upstream logical retry trigger.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix if implemented at the existing seams: queue identity in storage/delivery, permanent classification in recovery, and Telegram chunking through the established outbound adapter/send helper. A broad core migration or startup-time queue repair would be a less targeted solution.
