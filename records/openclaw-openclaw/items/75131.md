## Summary

Keep open. Current main has partial delivery-queue and Telegram chunking safeguards, but the exact report is not fully fixed: the permanent-error classifier still misses Telegram `message is too long` and several 403 cases, the message-tool Telegram send action bypasses the normal chunked outbound path, and queue storage still has no stable queue-level idempotency key for repeated logical send attempts.

## What This Changes

The issue reports three Telegram delivery defects: retry enqueues duplicate queue entries, permanent Telegram 400/403 failures are retried during recovery, and long send-message payloads are not split before hitting Telegram limits.

## Best Possible Solution

Land one focused fix that gives queue retries a stable logical-send identity, classifies permanent Telegram Bot API 400/403 failures before retrying, and routes Telegram message-tool sends through the same chunk-aware delivery path used by normal outbound Telegram replies.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the permanent-failure and long-message paths: a recovery test with `400: Bad Request: message is too long` should currently stay pending, and a Telegram `message` tool send above 4096 characters follows the plugin action path into a single direct send. The duplicate-UUID incident is well evidenced by the report, but the exact higher-level retry trigger still needs a targeted regression test.

## Solution Assessment

Is this the best way to solve the issue? Yes, the requested direction is the maintainable shape if kept narrow: fix queue idempotency at the queue/retry boundary, extend the existing permanent-error classifier, and reuse Telegram's existing chunking instead of adding another ad hoc truncation path.
