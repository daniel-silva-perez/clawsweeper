## Summary

Keep open. Current main still has the locked, cache-bypassing session-store rereads that match the reported multi-minute `session-write-lock` holds, and the direct fix remains an open PR rather than merged. The latest tagged release still contains the same locked reread path.

## What This Changes

The issue reports slow startup, slow replies, and multi-minute `sessions.json.lock` holds during session-store writes, first on Windows and then corroborated on macOS and Windows by commenters.

## Best Possible Solution

Keep this issue open and land #68554 or an equivalent narrow fix that gives `updateSessionStore` and `updateSessionStoreEntry` a no-disk-read, no-full-clone path inside the serialized lock while preserving write-through cache invariants, stale-cache detection, maintenance behavior, and concurrent update correctness.

## Reproduction Assessment

Yes. The issue discussion has concrete timing logs, and current main has a source-level reproduction path: grow or prime `sessions.json`, trigger repeated session-store writes, and observe synchronous cache-bypassing reads/parses running inside the session-store lock.

## Solution Assessment

Yes for the narrow direction, but the implementation should be stronger than a simple `skipCache` removal. The maintainable fix is to avoid redundant disk rereads and full cached-store parsing while keeping the existing lock and cache coherency guarantees intact.
