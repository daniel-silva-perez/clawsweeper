## Summary

Keep open. Current main and the latest tagged release still preserve the reported session-store lock pattern: store writes acquire the session-store lock, then force a cache-bypassing synchronous disk read/JSON parse while the lock is held. The issue has multiple recent Windows/macOS corroborations, and open PR #68554 remains the direct implementation candidate rather than a reason to close before merge.

## What This Changes

The issue reports very slow gateway startup, CLI/RPC replies, and first-turn responses with `[session-write-lock]` held for minutes on `sessions.json.lock`, especially on Windows npm installs.

## Best Possible Solution

Keep this issue open and review or land #68554, or an equivalent maintainer-owned fix that stops forcing cache-bypassing disk rereads inside `updateSessionStore` and `updateSessionStoreEntry` while preserving cache coherency and external-writer safety. Update the stale-cache regression coverage to match the intended contract and verify startup/RPC behavior with a large session store.

## Reproduction Assessment

Yes. The comments include concrete Windows and macOS lock-hold/RPC timing logs, and current main has a source-level reproduction path: grow or prime `sessions.json`, trigger repeated session store writes, and observe the locked task performing synchronous cache-bypassing disk reads/parses.

## Solution Assessment

Yes, the #68554-style fix appears to be the narrowest maintainable direction for this specific store-lock latency: use the write-through session store cache inside the serialized lock instead of rereading disk every time. #71903 should remain adjacent hardening, not a substitute for the store-lock performance fix.
