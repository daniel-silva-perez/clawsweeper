## Summary

Keep open. Current main still contains the locked, cache-bypassing session-store rereads that match the multi-minute `session-write-lock` holds, and the direct fix remains open rather than merged. The latest tagged release also still has the same locked reread path, so this is not a cleanup-close candidate.

## What This Changes

The issue reports slow OpenClaw startup and reply latency with multi-minute `sessions.json.lock` holds, first on Windows and then corroborated by additional Windows and macOS reports.

## Best Possible Solution

Keep this issue open and review #68554 or an equivalent narrow fix that gives `updateSessionStore` and `updateSessionStoreEntry` a no-disk-read, no-full-clone path inside the serialized lock while preserving write-through cache invariants, stale-cache detection, maintenance behavior, and concurrent update correctness.

## Reproduction Assessment

Yes. The issue discussion has concrete timing logs, and current main has a source-level reproduction path: grow or prime `sessions.json`, trigger repeated session-store writes, and observe synchronous cache-bypassing reads/parses running inside the session-store lock.

## Solution Assessment

Yes for the narrow direction, but the implementation should be stronger than a simple `skipCache` removal. The best fix is to avoid redundant disk rereads and full cached-store cloning/parsing while keeping the session-store lock and cache coherency guarantees intact.
