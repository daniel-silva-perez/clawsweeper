## Summary

Keep open. Current main still matches the reported session-store lock contention path: both main session-store update helpers acquire the session write lock and then force cache-bypassing synchronous `sessions.json` reads/parses while the lock is held, and the narrow fix remains open in #68554 rather than merged or shipped.

## What This Changes

The issue reports very slow startup and reply latency, especially on Windows, with `session-write-lock` warnings showing `sessions.json.lock` held for minutes.

## Best Possible Solution

Keep this issue open and land #68554 or an equivalent narrow fix that removes redundant disk rereads from the locked session-store update paths while preserving cache coherency, write ordering, stale-cache detection, and concurrent update correctness.

## Reproduction Assessment

Yes. The discussion provides concrete timing logs, and current main has a source-level reproduction path: use a large or frequently updated `sessions.json`, trigger queued `updateSessionStore` calls, and observe synchronous cache-bypassing reads/parses running inside the session-store lock.

## Solution Assessment

Yes for the linked direction, but it is not on main yet. The narrow maintainable fix is to avoid redundant `skipCache` disk rereads inside the serialized write lock without weakening write-through cache invariants.
