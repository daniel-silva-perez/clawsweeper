## Summary

Keep this issue open. Current main at 5828dcdb05aad587bff831467e7fa51f73312ff8 still has the reported session-store write pattern: `updateSessionStore` and `updateSessionStoreEntry` acquire the session store lock, then force `loadSessionStore(..., { skipCache: true })`, which bypasses the in-process cache and performs synchronous disk read/JSON parse while the lock is held. The issue is fresh, has multiple Windows/macOS corroborating reports, and open PR #68554 remains the direct implementation candidate rather than a reason to close before merge.

## Best Possible Solution

Keep this issue open and review or land #68554, or an equivalent maintainer-owned fix that removes redundant locked `skipCache` disk rereads from `updateSessionStore` and `updateSessionStoreEntry` while preserving cache coherency and external-writer safety. Add or update focused coverage for the intended cache/lock contract and verify startup, `node.list`, `sessions.list`, cron activity, restart recovery, and first-turn reply latency with a large session store.
