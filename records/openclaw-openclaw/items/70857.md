## Summary

Keep open. Current main still matches the reported failure mode: session-store updates acquire the session write lock and then force cache-bypassing synchronous reads/parses before releasing it, and the latest tagged release still contains the same path. The issue now has multiple Windows and macOS reports plus an open targeted PR, so this is not a cleanup-close candidate.

## What This Changes

The issue reports multi-minute OpenClaw startup and reply latency with `sessions.json.lock` held far beyond the expected window during session-store writes.

## Best Possible Solution

Keep the issue open and review or land #68554, or an equivalent narrow fix, after it removes redundant cache-bypassing disk work from the locked update path while preserving write-through cache invariants, stale-cache detection, concurrent update correctness, and the relevant session-store tests.

## Reproduction Assessment

Yes. The issue and comments provide concrete lock-hold logs across Windows and macOS, and current main has a source-level reproduction path: grow or prime `sessions.json`, trigger repeated session-store writes, and observe synchronous cache-bypassing reads/parses inside the session-store lock.

## Solution Assessment

Yes, the best direction is the narrow session-store update path targeted by #68554. The implementation should account for the existing stale-cache test contract rather than relying on a simple option removal without replacement coverage.
