## Summary

Keep this PR open. Current main still does not implement the PR's cron-specific `server_error` behavior: `FailoverReason` lacks `server_error`, structured Codex/OpenAI `server_error` payloads are still classified as `timeout`, cron `retryOn: ["server_error"]` only matches raw 5xx text, and the cron protocol cannot expose `lastErrorReason: "server_error"`. The PR remains a focused implementation candidate, with review work needed around 4xx precedence before merge.

## Best Possible Solution

Keep this PR open. Rebase it onto current main, add `server_error` as a first-class failover and cron protocol reason, map it to fallback status 500, make cron retry detection consult the failover classifier before raw regex fallback, keep transport/server errors out of auth-profile health poisoning, and add focused regression coverage for structured Codex/OpenAI payload retries plus explicit and string-parsed 4xx precedence.
