## Summary

Keep this PR open. Current main still does not implement the PR's cron-specific `server_error` behavior: `FailoverReason` has no `server_error`, structured Codex/OpenAI `server_error` payloads still classify as `timeout`, cron retry detection only matches raw 5xx text for `server_error`, and the protocol schema cannot expose `lastErrorReason: "server_error"`. The PR remains a focused implementation candidate, with review needed around 4xx precedence before merge.

## Best Possible Solution

Keep this PR open. Rebase it onto current main, add `server_error` as a first-class failover and cron protocol reason, map it to fallback status 500, make cron retry detection consult classifier output before raw regex fallback, keep transport/server errors out of auth-profile health poisoning, and add focused regression coverage for structured Codex/OpenAI payload retries plus explicit and string-parsed 4xx precedence.
