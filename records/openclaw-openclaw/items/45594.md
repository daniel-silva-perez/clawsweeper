## Summary

Keep open. Current main still advertises `cron.retry.retryOn: ["server_error"]`, but it does not implement this PR's first-class cron `server_error` behavior. Structured Codex/OpenAI `server_error` payloads are still classified as `timeout`, the cron retry gate only matches `server_error` through raw 5xx text, and the cron protocol state schema cannot expose `lastErrorReason: "server_error"`.

## Best Possible Solution

Keep this PR open. Rebase it onto current main, add `server_error` as a first-class failover and cron protocol reason, classify structured OpenAI/Codex `server_error` payloads as `server_error` without overriding 4xx auth/billing/format precedence, map it to fallback status 500, make cron retry detection consult classifier output before regex fallback, keep transport/server errors out of auth-profile health poisoning, and land the focused regression tests.
