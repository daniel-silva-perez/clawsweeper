## Summary

Keep this PR open. Current main still advertises `cron.retry.retryOn: ["server_error"]`, but it does not implement the PR's first-class cron `server_error` behavior: the failover reason union and cron protocol schema omit it, structured Codex/OpenAI `server_error` payloads still classify as `timeout`, and cron retry matching only checks raw 5xx text.

## Best Possible Solution

Keep this PR open. Rebase it onto current main, add `server_error` as a first-class failover and cron protocol reason, classify structured OpenAI/Codex `server_error` payloads as `server_error` only without overriding 4xx auth/billing/format precedence, map it to fallback status 500, make cron retry detection consult classifier output before regex fallback, keep transport/server errors out of auth-profile health poisoning, and land the focused regression coverage.
