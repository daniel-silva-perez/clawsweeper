## Summary

Keep open. Current main has useful mitigations for diagnostics, session-store maintenance, chat.history response caps, and active-memory cache bounds, but it does not prove the broad long-uptime gateway OOM is fixed. The issue discussion now includes multiple recent reports across Docker, macOS, webchat/Discord, Telegram-only heavy compaction/tool use, and a 31-agent memory-plugin deployment, while current source still has plausible whole-store, whole-transcript, polling, watcher, and runtime-retention paths that need profiling before closure.

## Best Possible Solution

Keep this issue open as the broad gateway heap-exhaustion tracker. The best maintainer path is to collect a current diagnostics export plus heap snapshots from an affected build, keep #55334 as the sessions.json/session-store subtracker, and split or fix proven retainers by owner: bounded or push-based node refresh, lazy/paged session-store listing, tail/page chat.history reads, compaction snapshot deduplication, and memory-plugin watcher/cache/session cleanup in the owning plugin/runtime modules.
