## Summary

Keep open. Current main still lacks configurable per-job fixed-delay retries for recurring cron jobs, so this PR remains a relevant implementation candidate for open feature request #49740; however, the PR head still has blocking correctness questions around exhausted retry cadence, retry edits, incomplete policies, and one-shot flag behavior.

## What This Changes

The PR adds configurable retry settings for recurring cron jobs, wiring retry policy and retry-cycle state through cron runtime types, gateway protocol models, generated Swift clients, CLI add/edit options, and regression tests.

## Best Possible Solution

Keep this PR open as an implementation candidate for #49740, or supersede it with a focused maintainer patch that adds per-job recurring retry policy end to end while preserving existing backoff protections, validating retry policy shape, defining one-shot behavior, clearing/recomputing retry state on edits, updating docs/generated clients, and covering the edge cases with regression tests.

## Reproduction Assessment

Yes. Current main can be inspected directly: cron docs, CLI option lists, types, and gateway schema lack per-job retry controls, and the scheduler only applies exponential backoff; the PR-specific defects are reproducible by code-path inspection on the PR head without needing a live gateway.

## Solution Assessment

No, not as-is. The direction is the right product boundary for the open feature request, but the current patch needs tighter validation and scheduler semantics before it is the narrowest maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve cadence after exhausted retries:** `src/cron/service/timer.ts:498`
  - body: When a configured recurring retry is no longer scheduled, this branch picks `Math.max(effectiveNaturalNextRunAtMs, fallbackNextRunAtMs)`. For short schedules, the global error backoff can win after the last retry fails, so a job such as every 5s with one 1s retry skips the preserved natural slot and drifts away from the advertised cadence-preserving behavior.
  - confidence: 0.9
- **[P2] Recompute failed jobs when retry settings change:** `src/cron/service/jobs.ts:596-600`
  - body: `applyJobPatch()` stores the new retry policy, but `update()` only recomputes `nextRunAtMs` for schedule/enabled changes or missing next-run state. Editing a failed recurring job to add `--retry-count` and `--retry-delay` can therefore leave the existing later natural run in place, making the new retry settings a no-op for the incident-response case.
  - confidence: 0.86
- **[P2] Reject incomplete retry policies:** `src/gateway/protocol/schema/cron.ts:293-294`
  - body: The gateway schema accepts `retryCount` and `retryDelayMs` independently, but `applyJobResult()` only retries when both are positive. API callers, and CLI paths that pass only one flag, can create a job that succeeds validation while never retrying, so this should be an all-or-nothing policy or have a documented default.
  - confidence: 0.88
- **[P2] Reject retry flags for one-shot schedules:** `src/cli/cron-cli/register.cron-add.ts:274-275`
  - body: These fields are forwarded for any `cron add` schedule, but the runtime consults the per-job retry policy only in the recurring error branch. `openclaw cron add --at ... --retry-count ... --retry-delay ...` therefore appears to configure retries while the one-shot path keeps using the existing global transient retry behavior; reject the flags for one-shot jobs or implement the same policy there.
  - confidence: 0.82
