## Summary

Keep open. The linked Telegram data-loss path is real, but this PR is not mergeable as-is: the strict-mode recovery still leaks unclosed reasoning tails after whitespace-only captured content, and the branch includes broad unrelated gateway, hooks, UI, config, generated backup, and command/runtime changes that introduce additional correctness and security review blockers.

## What This Changes

This PR changes shared reasoning-tag stripping and tests to recover answer text after malformed unclosed thinking tags, while also carrying broad unrelated changes across gateway/session handling, hooks, UI, config/schema, command/runtime code, and generated `dist.bak/*` files.

## Best Possible Solution

Do not land this branch as-is. The maintainable end state is a narrow shared-text/Telegram fix for #49104 that preserves strict mode as a safety boundary after any captured prefix, uses a raw captured-length or equivalent zero-capture guard instead of trim-based recovery, adds regression tests for whitespace-prefix and zero-capture recovery cases, and leaves unrelated hook/gateway/UI/config/generated changes to separate reviewed PRs.

## Reproduction Assessment

Yes. The code path is deterministic: with strict mode and default `trim: "both"`, a whitespace prefix before an unclosed `<think>` produces an empty `trimmedResult`, so the current/PR guard recovers and exposes the tail; Telegram calls that strict path for answer extraction.

## Solution Assessment

No. The proposed direction addresses part of the data-loss bug, but `!result.trim()` is the wrong boundary for strict mode and the branch mixes the fix with unrelated, review-blocking changes; a narrower fix should use a raw zero-capture check plus focused Telegram/shared-text tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep strict recovery from treating whitespace as zero capture:** `src/shared/text/reasoning-tags.ts:88`
  - body: The new strict-mode guard still uses `!result.trim()`, so an input like `"\n<think>secret"` recovers `secret` even though strict mode already captured a prefix. Telegram uses this strict stripper for answer extraction, so whitespace-formatted model output can expose an unclosed reasoning tail instead of failing closed.
  - confidence: 0.92
- **[P2] Restore the sessions.changed broadcast after reset:** `src/gateway/server-methods/sessions.ts:996`
  - body: This removes the `emitSessionsChanged` call after a successful `sessions.reset`. Other connected clients rely on `sessions.changed` to reload the updated session ID and metadata, so resets from one tab can leave other tabs stale until a manual refresh or another mutation happens.
  - confidence: 0.87
- **[P2] Put a timeout around awaited transcribed hooks:** `src/auto-reply/reply/message-preprocess-hooks.ts:28-34`
  - body: `message:transcribed` is now awaited before the agent reply starts, but the await has no timeout. A slow or stuck hook handler can block the inbound reply path indefinitely for that message, unlike the previous fire-and-forget behavior.
  - confidence: 0.84
- **[P2] Use execFile for workspace memory extraction:** `src/hooks/bundled/session-memory/handler.ts:185-187`
  - body: The new session-memory path builds a shell command with interpolated `extractScript` and `sessionId`. If either value contains shell metacharacters, the shell can execute more than the intended Python script; use `execFile("python3", [extractScript, "--session", sessionId], { cwd, timeout })` instead.
  - confidence: 0.86
