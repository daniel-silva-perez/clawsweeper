## Summary

Keep open. The PR is based on useful bug context, but it should not merge as-is because the branch still has concrete correctness and security blockers, and the current main implementation does not prove the stricter raw zero-capture safety invariant discussed in review.

## What This Changes

This PR changes shared reasoning-tag stripping and tests to recover answer text after malformed unclosed thinking tags, while also carrying unrelated gateway, session, hook, UI, config, and generated backup-tree changes.

## Best Possible Solution

Do not land this branch as-is. The maintainable end state is a narrow shared-text and Telegram fix for the malformed reasoning-tag delivery path that preserves strict mode after any raw captured prefix, adds whitespace-prefix and zero-capture regression tests, and leaves hook, gateway, UI, config, and generated-file work to separate reviewed PRs.

## Reproduction Assessment

Yes. The review path is deterministic by code inspection: strict mode with default `trim: "both"` treats a whitespace-only prefix as empty before recovering an unclosed `<think>` tail, and Telegram uses that strict path for answer extraction.

## Solution Assessment

No. The branch addresses part of the data-loss bug, but the trim-based zero-capture guard is not the safest boundary and the unrelated runtime/security churn makes this a poor merge vehicle; a narrow raw-length guard with focused tests is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep strict recovery from treating whitespace as zero capture:** `src/shared/text/reasoning-tags.ts:88`
  - body: The strict-mode guard still uses trimmed output, so an input like `"\n<think>secret"` can recover `secret` even though strict mode already captured a prefix. Telegram uses this strict stripper for answer extraction, so whitespace-formatted model output can expose an unclosed reasoning tail instead of failing closed.
  - confidence: 0.92
- **[P2] Restore the sessions.changed broadcast after reset:** `src/gateway/server-methods/sessions.ts:996`
  - body: This removes the `emitSessionsChanged` call after a successful `sessions.reset`. Other connected clients rely on `sessions.changed` to reload the updated session ID and metadata, so resets from one tab can leave other tabs stale until a manual refresh or another mutation happens.
  - confidence: 0.87
- **[P2] Add a timeout around awaited transcribed hooks:** `src/auto-reply/reply/message-preprocess-hooks.ts:28-34`
  - body: `message:transcribed` is now awaited before the agent reply starts, but the await has no timeout. A slow or stuck hook handler can block the inbound reply path indefinitely for that message, unlike the previous fire-and-forget behavior.
  - confidence: 0.84
- **[P2] Use execFile for workspace memory extraction:** `src/hooks/bundled/session-memory/handler.ts:185-187`
  - body: The new session-memory path builds a shell command with interpolated `extractScript` and `sessionId`. If either value contains shell metacharacters, the shell can execute more than the intended Python script; use `execFile("python3", [extractScript, "--session", sessionId], ...)` instead.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: Security review needs attention because the PR introduces unrelated shell execution and a large generated backup tree.

Concerns:

- **[medium] Shell interpolation in bundled memory hook:** `src/hooks/bundled/session-memory/handler.ts:185`
  - body: `execAsync` is called with a shell command assembled from `extractScript` and `sessionId`; this can execute unintended shell syntax if path or session values contain metacharacters. Use `execFile` with an argv array if this feature remains.
  - confidence: 0.86
- **[low] Generated backup tree obscures supply-chain review:** `dist.bak/acp-cli-BpCQk37E.js:1`
  - body: The PR adds a large `dist.bak/*` generated JavaScript tree unrelated to the Telegram fix. Even if not imported, committing generated backup code makes the review surface too large and can hide executable changes from normal source review.
  - confidence: 0.78
