## Summary

Keep this issue open. Current main has the browser backend, CLI, docs, and Google Meet manual-action reporting, but the Control UI still lacks the requested visible browser-login handoff panel with managed browser status, active tab context/actions, and an operator resume path. The provided discussion also includes a maintainer comment explicitly saying this should remain the canonical Control UI browser-login handoff tracker, separate from the Google Meet plugin work.

## Best Possible Solution

Keep this issue open as the canonical Control UI product tracker. The best implementation should add a visible browser handoff surface backed by existing browser.request/browser node context: show managed browser/profile status, list active tabs with stable tab ids/labels, support open/focus/close or exact-session access, surface structured LOGIN_REQUIRED/manual-action details from tools such as Google Meet, and let the agent resume against the same browser state after the operator completes login or permission prompts.
