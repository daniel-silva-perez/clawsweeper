## Summary

Keep open. Current main has a browser tool, CLI, and admin Gateway method for status/profiles/tabs/open/focus/close, and Google Meet now returns structured manual-action details, but the Control UI still has no visible browser route or handoff panel. A maintainer comment also explicitly identifies this as the canonical browser-login handoff surface to keep open, separate from the Google Meet plugin work.

## What This Changes

Request a visible Control UI browser panel for browser status, active tabs, tab actions, direct managed-session access, and user/agent login handoff with resume against the same browser state.

## Best Possible Solution

Add a first-class Browser area in the Control UI, backed by the existing browser control surface or an additive typed Gateway method, that shows profile/status/readiness, active tabs with open/focus/close actions, exact node/profile/tab context for manual blockers, and a resume path after the operator completes login or permission prompts. Keep Google Meet-specific detection in the plugin, but make the handoff surface generic for browser-backed workflows.

## Reproduction Assessment

Not applicable as a feature request rather than a runtime bug. Code inspection gives high-confidence evidence of the gap: the Control UI has no browser route/view while browser CLI/tool operations and Google Meet manual-action details exist elsewhere.

## Solution Assessment

Not applicable because there is no proposed patch to assess. The requested direction is the right product-level fix, but the maintainable implementation should be generic Control UI browser handoff infrastructure rather than a Google Meet-only panel or another CLI/documentation workaround.
