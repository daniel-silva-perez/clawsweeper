## Summary

Current main still matches the reported 0/0 skip condition: `hasAlreadyFlushedForCurrentCompaction` treats missing `compactionCount` as `0` and returns true when `memoryFlushCompactionCount` is also `0`, and the current regression test asserts that behavior. The related PR attempts were closed unmerged, so there is no shipped/current fix for this specific bug.

## Best Possible Solution

Keep this issue open and fix the session-state contract: distinguish never-flushed from already-flushed-at-compaction-0, likely by using `undefined`/`null` as the never-flushed sentinel plus a doctor or migration path for legacy `memoryFlushCompactionCount: 0` entries when no `memoryFlushAt` exists. Add focused tests for `shouldRunMemoryFlush` and `runMemoryFlushIfNeeded` covering `compactionCount: 0`, `memoryFlushCompactionCount: 0`, and the post-flush dedupe case so the first flush is allowed without repeated flush loops.

## Risks / Open Questions

- Changing `0/0` to always mean never-flushed could cause repeated silent flushes within the initial compaction cycle, because current code records `memoryFlushCompactionCount: 0` after a successful flush when no compaction completed.
- The April 4 comment identifies broader preflight/context-budget bypasses; fixing this counter sentinel should be scoped as one memory-flush gate fix and not assumed to solve all proactive compaction failures.
