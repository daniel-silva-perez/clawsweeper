## Summary

Current main still treats the reported persisted 0/0 memory-flush counter state as already flushed, so the issue remains valid. A correct fix needs to distinguish legacy or never-flushed rows from real cycle-0 flush records while preserving the documented once-per-compaction-cycle guard.

## What This Changes

The issue asks OpenClaw to allow `memoryFlush` to run for never-flushed sessions whose `compactionCount` and `memoryFlushCompactionCount` are both zero.

## Best Possible Solution

Define a backward-compatible session-state contract for memory flush metadata, likely using `memoryFlushAt` or an explicit future sentinel, then update the helper, runner persistence, and focused tests while keeping broader pre-request context-budget work separate.
