## Summary

Keep open. Current main still treats the reported persisted 0/0 memory-flush counter state as already flushed, and the latest release tag carries the same helper behavior; the open related PR changes adjacent counter persistence but does not close this issue.

## What This Changes

Memory flush should trigger for eligible sessions whose compaction and memory-flush counters are both zero instead of treating that state as already flushed.

## Best Possible Solution

Define a backward-compatible session-state contract that distinguishes never-flushed or legacy 0/0 rows from recorded cycle-0 flushes, likely using `memoryFlushAt` or an explicit future sentinel, then update the helper, runner persistence, and focused tests while keeping broader pre-request context-budget work separate.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: with `totalTokens` above threshold and both counters at 0, `shouldRunMemoryFlush` reaches `hasAlreadyFlushedForCurrentCompaction`, which returns true and skips the flush; the focused test currently encodes that zero state as already flushed.

## Solution Assessment

No current fix is present on main. The safest solution is not the closed direct PRs' naive 0/0 special case, but a coordinated metadata-contract fix that preserves the documented once-per-cycle guard and remains compatible with #51421.
