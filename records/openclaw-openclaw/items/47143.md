## Summary

Keep open. Current main still treats the reported persisted 0/0 memory-flush counter state as already flushed, the latest release snapshot has the same helper behavior, and the open adjacent PR changes related counter persistence without resolving this specific ambiguity.

## What This Changes

The issue asks OpenClaw to trigger memoryFlush for high-token sessions that have never been compacted when both compactionCount and memoryFlushCompactionCount are 0.

## Best Possible Solution

Define a backward-compatible memoryFlush metadata contract that distinguishes never-flushed or legacy rows from a recorded flush at compaction count 0, then update the helper, runner persistence, focused tests, docs or changelog as needed, while keeping broader pre-request context-budget work separate.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: with totalTokens above threshold and both counters at 0, shouldRunMemoryFlush reaches hasAlreadyFlushedForCurrentCompaction, which returns true and skips memoryFlush.

## Solution Assessment

No fix is present on main. A naive 0/0 special case is not clearly safest because a real pre-compaction flush can persist 0 with memoryFlushAt; the better fix should disambiguate the state contract and coordinate with #51421.
