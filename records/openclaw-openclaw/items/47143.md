## Summary

Keep open: current main still treats the reported persisted 0/0 memory-flush counter state as already flushed, so an over-threshold session row can skip memoryFlush. The ordinary new-session path clears the marker now, but legacy or explicitly persisted rows remain ambiguous and need a state-contract fix that preserves once-per-cycle deduplication.

## What This Changes

The issue reports that memoryFlush is skipped for sessions whose persisted compactionCount and memoryFlushCompactionCount are both 0.

## Best Possible Solution

Define a backward-compatible memory-flush metadata contract, likely using memoryFlushAt or an explicit future sentinel to distinguish never-flushed or legacy 0/0 rows from recorded cycle-0 flushes, then update the helper, runner persistence, and focused tests while keeping broader pre-request context-budget work separate.
