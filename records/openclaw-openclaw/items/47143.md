## Summary

Current main still contains the reported memoryFlush skip path, so this should stay open. The 0/0 counter state is still treated as already flushed, while the runner can also legitimately persist a cycle-0 flush as 0 with `memoryFlushAt`, so the fix needs a backward-compatible state-contract update rather than a naive `compactionCount === 0` special case.

## What This Changes

The issue asks for memoryFlush to run for over-threshold sessions whose session-store counters are `compactionCount: 0` and `memoryFlushCompactionCount: 0` instead of treating that state as already flushed.

## Best Possible Solution

Define a backward-compatible memoryFlush metadata contract that distinguishes never-flushed or legacy 0/0 rows from recorded cycle-0 flushes, update the helper and runner persistence accordingly, and add focused regression coverage while leaving broader pre-request context-budget work to the related issues.
