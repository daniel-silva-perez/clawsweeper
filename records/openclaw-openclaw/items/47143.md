## Summary

Keep open. Current main still treats the reported 0/0 memory-flush counter state as already flushed, and the latest release contains the same helper behavior, so the user-visible bug is not implemented or obsolete.

## What This Changes

The issue requests that memoryFlush run for high-token sessions whose persisted state has compactionCount and memoryFlushCompactionCount both set to 0 instead of treating that state as already flushed.

## Best Possible Solution

Define a backward-compatible memoryFlush state contract that distinguishes legacy or never-flushed 0/0 rows from recorded cycle-0 flushes, update the helper and runner persistence consistently, and add focused regression coverage while leaving broader pre-request context-budget work to the related memoryFlush issues.
