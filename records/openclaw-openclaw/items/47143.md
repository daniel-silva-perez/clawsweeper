## Summary

Current main still has the reported 0/0 memory-flush skip path, so this should stay open. The ordinary new-session path now clears the flush marker, but persisted or explicitly written rows with both counters at 0 remain ambiguous and need a backward-compatible state-contract fix that preserves once-per-compaction-cycle deduplication.

## What This Changes

The issue asks for memoryFlush to trigger for over-threshold sessions that have never compacted, even when both compaction counters are stored as 0.

## Best Possible Solution

Define a backward-compatible memory-flush metadata contract, likely using `memoryFlushAt` or an explicit future sentinel to distinguish never-flushed or legacy 0/0 rows from recorded cycle-0 flushes, then update the helper, runner persistence, and focused tests while keeping broader pre-request context-budget work separate.
