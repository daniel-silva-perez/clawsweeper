## Summary

Current main still treats the reported persisted 0/0 counter state as already flushed, so the issue should remain open. The fix is not a blanket `0/0 => not flushed` change because current runner behavior can legitimately persist `memoryFlushCompactionCount: 0` after a cycle-0 flush; the remaining work is to distinguish legacy/no-flush rows from real recorded cycle-0 flushes while preserving the once-per-cycle guard.

## What This Changes

The issue asks OpenClaw to let `memoryFlush` run for never-flushed sessions whose `compactionCount` and `memoryFlushCompactionCount` are both zero.

## Best Possible Solution

Define a backward-compatible session-state contract that distinguishes never flushed from recorded cycle-0 flushes, likely using `memoryFlushAt` or an explicit future sentinel, then update the helper, runner persistence, and focused tests while keeping broader pre-request context-budget work separate.
