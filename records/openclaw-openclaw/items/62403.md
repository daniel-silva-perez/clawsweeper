## Summary

Keep this PR open. Current main still renders the Sessions table rows with an unkeyed `paginated.flatMap(...)`, while label edits call `sessions.patch`, update `updatedAt`, and emit `sessions.changed`; the controller then replaces the matching row and the view re-sorts by updated time. The PR’s keyed Lit `repeat(row.key)` approach and regression test still target a real unimplemented UI row-identity bug. Security review of the supplied diff found no new permissions, dependency sources, lifecycle hooks, network calls, or secret handling changes; it only uses the existing `lit` dependency and adds a UI regression test.

## Best Possible Solution

Refresh and review this PR rather than closing it. The best implementation path is to key the Sessions table rows by `row.key` with Lit `repeat()`, preferably wrapping the main and optional expanded row in one TemplateResult per key, keep the browser regression test for label edit plus updated-time reorder, then run the targeted UI test and changed gate before merge.
