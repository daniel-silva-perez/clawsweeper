## Summary

Keep open. Current main has useful mitigations for Control UI/WebChat history rendering and payload size, but it does not prove this freeze is fixed: the WebSocket Control UI still requests one non-paginated `chat.history` window, `chat.history` still reads/parses the full transcript before response caps apply, and the related tail-read fix in #49335 is still open in the provided context. The issue also still lacks the browser console/network trace needed to distinguish transcript-size, Chrome/Lit first-render, model selector, or WebSocket message handling causes.

## Best Possible Solution

Keep this issue open. Ask the reporter for a current-main reproduction with DevTools console errors, failed Network/WS frames, browser/OpenClaw versions, session file sizes, and whether resetting sessions still restores the UI. If session-size recovery reproduces, review and land #49335 or an equivalent bounded transcript reader. If the browser locks before any `chat.history` request, route it through the Chrome/Lit Control UI path tracked by #47979. If accumulated history is the root cause, wire the existing paginated session-history API or an additive WebSocket cursor contract into Control UI with load-older/virtualized rendering.
