## Summary

Keep open. Current main still has multiple plausible unfixed paths for this Control UI/WebChat freeze report: the issue lacks the browser console/network trace needed to isolate the root cause, `chat.history` still reads/parses full transcript files before response caps apply, and the WebSocket chat history UI still has no pagination/load-older path. An open related PR (#49335) targets the transcript-read freeze vector, but it is not merged.

## Best Possible Solution

Keep this issue open. Ask the reporter for a current-main reproduction with DevTools console errors, failed requests/WS frames, browser/OpenClaw versions, session file sizes, and whether resetting sessions still restores the UI. If the session-size recovery path reproduces, review and land a current-main-safe version of #49335 or equivalent bounded transcript reader. If it is a Chromium first-render issue, coordinate with #47979 and add a targeted fallback or browser smoke. If it is accumulated history, extend the WebSocket history contract or wire the existing paginated history API into Control UI with load-older/virtualized rendering.
