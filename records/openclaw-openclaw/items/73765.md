## Summary

Keep open. Current main honors `skills.entries.<key>.enabled: false` when a snapshot is rebuilt, and live config reload now bumps the in-memory skills snapshot version, but persisted long-lived sessions can still reuse a stale `version: 0` snapshot after a gateway restart or offline config edit because there is no durable snapshot input fingerprint. A related open PR overlaps with the right direction but does not yet cover all current-main snapshot builders, so this should remain tracked for maintainer follow-up.

## What This Changes

Existing session skills snapshots should refresh after `skills.entries.<skill>.enabled` changes across gateway restart or offline config edits so disabled skills disappear from `<available_skills>` without requiring `/new`.

## Best Possible Solution

Persist and compare a deterministic skills snapshot input fingerprint across all session snapshot builders. The fingerprint should cover effective `config.skills`, the normalized agent skill filter, resolved skill roots/entries or their stable metadata, and remote eligibility inputs needed for visibility; existing sessions should rebuild when it differs after restart while keeping the current live watcher/version bump as a fast invalidation path.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to persist a session with `skillsSnapshot.version: 0`, edit `skills.entries.<name>.enabled` while the gateway is stopped, restart, then send a turn on the same session; current main compares 0 to 0 and reuses the stale snapshot.

## Solution Assessment

Yes, a durable input fingerprint is the best direction. A startup-only bump would mask some restart cases, but it would not precisely track config/root/eligibility inputs and would not address the broader stale-path family as cleanly.
