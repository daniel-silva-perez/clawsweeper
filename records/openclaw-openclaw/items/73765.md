## Summary

Keep open. Current main still has the stale persisted skills snapshot gap for restart/offline config edits: the disabled-skill filter works when a snapshot is rebuilt, but an existing version-0 session snapshot is reused when the process-local skills version also starts at 0.

## What This Changes

The issue asks for `skills.entries.<name>.enabled=false` and related skills input changes to invalidate stale skills snapshots in existing long-lived sessions.

## Best Possible Solution

Refresh persisted skills snapshots based on restart-stable snapshot inputs. A deterministic fingerprint over effective skills config, agent skill filter, and resolved skill roots/entries would be the most precise fix; the open startup-version seeding PR may be an acceptable narrower repair if maintainers prefer refresh-on-restart semantics.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to persist a session with `skillsSnapshot.version: 0`, add `skills.entries.<name>.enabled=false` while the gateway is stopped, restart, and send another turn on the same session; current main computes version 0 again and reuses the old snapshot.

## Solution Assessment

Yes for the issue’s durable fingerprint direction; it directly models the inputs that make a persisted snapshot stale. The open draft startup-seed fix is simpler and likely covers restart/offline edits, but it is less precise and remains unmerged.
