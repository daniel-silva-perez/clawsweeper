## Summary

Keep this open. Current main still relies on an in-memory skills snapshot version plus agent skill-filter comparison, so a persisted snapshot with version 0 can survive a restart and continue advertising a skill after `skills.entries.<name>.enabled` is changed to `false`.

## What This Changes

The issue requests persistent invalidation of existing session skill snapshots when per-skill config, especially `skills.entries.<name>.enabled: false`, changes while a long-lived session already has a cached `skillsSnapshot`.

## Best Possible Solution

Persist a deterministic skill snapshot input signature and refresh existing sessions whenever the current signature differs. The signature should cover `config.skills`, effective skill roots/enumeration, workspace identity/path inputs that affect resolved skill paths, and the agent skill filter, while preserving backwards compatibility for snapshots that do not yet have the new field.

## Reproduction Assessment

Yes. A high-confidence reproduction follows directly from the current code path: create a persisted session snapshot with `version: 0`, set `skills.entries.<name>.enabled: false`, restart so refresh-state returns 0 again, then reuse the same session; the version and filter checks do not force a rebuild.

## Solution Assessment

Yes, the requested persistent invalidation is the right direction. A startup-only bump like the earlier closed PRs would help restart cases but would still miss edits not observed by the watcher; a content/input signature is the narrower durable fix.

## Repair Work Prompt

Fix the stale skill snapshot invalidation gap described in https://github.com/openclaw/openclaw/issues/73765. Existing long-lived sessions must stop advertising a skill after `skills.entries.<skillKey>.enabled` changes to `false`, even if the edit happened while the gateway was stopped and the process-local skills snapshot version resets to 0. Implement a backwards-compatible persistent snapshot input signature, such as `skillsSnapshot.inputsHash`, rather than relying only on `globalVersion`. The signature should deterministically cover inputs that affect eligible skills: `config.skills`, effective agent skill filter, relevant skill roots/enumeration or resolved skill identities/paths, and workspace/config inputs needed to avoid stale moved paths. Update both agent command and auto-reply session update paths. Keep core generic; do not add bundled skill IDs or plugin-specific policy. Add focused tests that prove a cached version-0 snapshot rebuilds after a `skills.entries.<name>.enabled:false` config change across a simulated restart, and consider coverage for stale workspace/path inputs related to #39719 if the same signature covers it. Update docs only if the refresh semantics text changes, and add a user-facing changelog fix entry with `Thanks @100yenadmin`.
