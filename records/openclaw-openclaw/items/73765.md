## Summary

Keep open. Current main still has the stale skills snapshot invalidation gap: a persisted version-0 snapshot is reused after restart or offline config edits, so a now-disabled skill can remain in the prompt until a new session or live watcher bump forces a rebuild.

## What This Changes

The issue asks OpenClaw to make `skills.entries.<name>.enabled: false` take effect for existing long-lived sessions even when the skills config changes while the gateway is stopped or after restart.

## Best Possible Solution

Add a deterministic, restart-stable skills snapshot input fingerprint and refresh persisted session snapshots when it changes. The fingerprint should cover the effective skills config, agent skill filter, and resolved skill roots/entries closely enough to catch disabled skills and stale paths without depending on watcher timing.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to persist a session with `skillsSnapshot.version: 0`, edit `skills.entries.<name>.enabled=false` while the gateway is stopped, restart, and send another turn on the same session; current main computes version 0 again and reuses the old snapshot.

## Solution Assessment

Yes. A durable fingerprint is the narrowest maintainable fix for this cache-invalidation family; a startup-only bump would refresh more than necessary and still would not model the actual snapshot inputs.
