## Summary

Keep open. Current main still reuses persisted version-0 skills snapshots after restart/offline config edits, so a skill disabled in config can remain in an existing session prompt until the snapshot is rebuilt by some other path. The existing open draft PR #71497 targets the same restart/version root cause but is not merged and does not close this issue.

## What This Changes

The issue asks for `skills.entries.<name>.enabled=false` and related skills config changes to invalidate stale skills snapshots in existing long-lived sessions.

## Best Possible Solution

Add restart-stable skills snapshot invalidation based on the effective inputs that make a snapshot valid, ideally a deterministic fingerprint over effective skills config, agent skill filter, and resolved skill roots/entries. If maintainers choose the narrower startup-version approach in #71497, it should still include coverage for disabled-skill snapshots after restart.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to persist an existing session with `skillsSnapshot.version: 0`, add `skills.entries.<name>.enabled=false` while the gateway is stopped, restart, and send another turn on the same session; current main computes version 0 again and reuses the old snapshot.

## Solution Assessment

Yes for the issue's durable inputs-hash direction; it directly models when a persisted snapshot is stale. The open draft startup-seeding fix is narrower and likely covers restart/offline edits, but it is not merged and may be less precise for other stale input changes.
