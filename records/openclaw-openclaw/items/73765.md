## Summary

Keep open. Current main has the right disabled-skill filter and now invalidates snapshots when the running config reloader observes `skills.*` changes, but the reported restart/offline-edit path still has no durable snapshot input hash or persisted version marker, so a persisted version-0 snapshot can be reused after restart.

## What This Changes

The issue asks OpenClaw to invalidate existing session skill snapshots when `skills.entries.<skill>.enabled` changes, especially when the edit happens while the gateway is stopped or before a restart.

## Best Possible Solution

Store and compare a durable, deterministic snapshot input fingerprint for the effective skills surface, covering `config.skills`, the effective agent skill filter, and resolved skill roots/entries. Existing sessions should rebuild when that fingerprint differs after restart, without relying on the in-process watcher version alone.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is: persist a session with `skillsSnapshot.version: 0`, edit `skills.entries.<name>.enabled` while the gateway is stopped, restart, and send a turn on the same session; current `shouldRefreshSnapshotForVersion(0, 0)` leaves the stale snapshot in place.

## Solution Assessment

Yes, the issue's durable `inputsHash` direction is the narrowest maintainable fix. A startup-only bump would hide this specific symptom by refreshing every session after every restart, but a persisted input fingerprint better covers config changes and the related stale-path family without depending on watcher timing.

## Repair Work Prompt

Fix the stale skills snapshot invalidation gap reported in https://github.com/openclaw/openclaw/issues/73765. Current main honors `skills.entries.<skill>.enabled: false` only when `buildWorkspaceSkillSnapshot` runs, but existing persisted sessions can keep `skillsSnapshot.version: 0` after a restart because `getSkillsSnapshotVersion()` resets to 0 and `shouldRefreshSnapshotForVersion(0, 0)` returns false. Implement a durable, deterministic snapshot input fingerprint stored on `SkillSnapshot` and refreshed when it differs. The fingerprint should cover the effective `config.skills` inputs, effective agent skill filter, and resolved skill roots/entries sufficiently to also avoid stale skill paths where practical. Keep the fix in the generic skills/session snapshot path; do not add extension-specific IDs or startup-only blanket refresh policy unless justified by tests. Add focused tests that simulate a persisted snapshot built before a `skills.entries.<name>.enabled=false` config change, then a fresh process/version-0 state, and verify the next turn rebuilds and excludes the disabled skill. Preserve the existing live config-reload bump and watcher behavior. Update docs only if the snapshot refresh contract changes; add a changelog fix entry with reporter credit if user-visible changelog policy applies.
