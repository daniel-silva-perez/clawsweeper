## Summary

Keep open. Current main still relies on an in-memory skills snapshot version plus the live config watcher, so a persisted version-0 snapshot can survive a gateway restart or offline config edit and continue advertising a now-disabled skill.

## What This Changes

The issue requests durable invalidation of existing session skill snapshots when `skills.entries.<name>.enabled` changes, especially across gateway restarts or edits made while the watcher is not running.

## Best Possible Solution

Add a durable, deterministic, secret-safe snapshot input fingerprint to skill snapshots and compare it before reusing persisted snapshots. The fingerprint should cover effective skills config, agent skill filtering, and resolved skill roots/entries closely enough to catch disabled skills and stale paths after restart without depending on watcher timing.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to persist a session with `skillsSnapshot.version: 0`, change `skills.entries.<name>.enabled` while the gateway is stopped, restart, and send another turn on the same session; current main computes version 0 again and reuses the old snapshot.

## Solution Assessment

Yes. The durable fingerprint direction is the narrowest maintainable fix; a startup-only bump would mask restart cases by refreshing all sessions after every restart, but it would still be less precise and would not address the broader stale-input family cleanly.

## Repair Work Prompt

Fix the stale skills snapshot invalidation gap described by this issue. Existing long-lived sessions must stop reusing a persisted skills snapshot when `skills.entries.<name>.enabled` or other effective skill snapshot inputs change while the gateway is stopped or after a restart. Implement a durable, deterministic, secret-safe snapshot input fingerprint on `SkillSnapshot` or an equivalent persisted field, compare it in `src/agents/agent-command.ts` before reusing `sessionEntry.skillsSnapshot`, and rebuild when it differs. Cover config-driven disabled skills, effective agent skill filters, and resolved skill roots/entries sufficiently to also avoid stale path reuse where practical. Do not persist plaintext secrets or raw secret-bearing config values; represent eligibility-affecting secret state without exposing secret material. Keep the existing watcher/version path working. Add focused tests for the offline edit/restart case where a cached version-0 snapshot with a disabled skill is refreshed on the next turn. Add a user-facing changelog fix entry with appropriate non-bot attribution. Do not broaden plugin APIs or change unrelated session reset behavior.
