## Summary

Keep open. Current main still has the reported durable cache-invalidation gap: disabled-skill filtering works when a snapshot is rebuilt, and live config reloads bump the in-memory snapshot version, but persisted version-0 snapshots can still be reused after restart or offline config edits because no durable skills input fingerprint is stored or compared.

## What This Changes

The issue reports that `skills.entries.<name>.enabled: false` is ignored for existing long-lived sessions when their persisted `skillsSnapshot` remains stale across a gateway restart or config edit that the watcher did not observe.

## Best Possible Solution

Store and compare a deterministic snapshot input fingerprint for the effective skills surface, covering `config.skills`, the effective agent skill filter, and resolved skill roots or entries. Existing sessions should rebuild when that fingerprint differs after restart, without depending on the in-process watcher version alone.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to persist an existing session with `skillsSnapshot.version: 0`, edit `skills.entries.<name>.enabled` while the gateway is stopped, restart, and send another turn on the same session; current main treats cached version 0 versus next version 0 as fresh.

## Solution Assessment

Yes, the durable `inputsHash` direction proposed in the issue is the narrowest maintainable fix. A startup-only bump would reduce one symptom, but a persisted input fingerprint better covers offline config edits and related stale skill-root/session-cache cases without refreshing every session unnecessarily.
