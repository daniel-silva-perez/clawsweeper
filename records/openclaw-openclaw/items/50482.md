## Summary

Current main still transcribes Telegram voice/audio messages, but it does not expose the requested native voice-note/message-type flag to the agent. Telegram `msg.audio` and `msg.voice` are both normalized to generic audio, shared `MsgContext` and the rendered inbound metadata lack an `is_voice`/`message_type` equivalent, and the existing tests only cover transcript delivery plus media transcription indexes. Related #40838 remains a broader open request for richer Telegram-native context, but this issue is a narrower actionable slice rather than obsolete or duplicate.

## Best Possible Solution

Keep this issue open and implement an additive, typed inbound message-origin field. Populate Telegram from native `msg.voice` before audio/voice normalization, expose it through `MsgContext` and the dynamic inbound conversation-info JSON, document it, and add focused shared inbound metadata plus Telegram voice/audio regression tests. Coordinate naming and scope with #40838, but do not wait for that broader issue to solve this narrower voice-note use case.
