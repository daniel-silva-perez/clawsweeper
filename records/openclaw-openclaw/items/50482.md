## Summary

Keep open. Current main still normalizes Telegram voice notes into generic audio/transcript context and does not expose a voice-note or native message-origin field to the agent-visible inbound metadata. The broader open Telegram metadata request in #40838 is related naming context, but it does not supersede this narrower voice-origin request.

## What This Changes

Expose whether an inbound message originated as a Telegram voice note, or another message-origin type, in the agent-visible inbound context metadata.

## Best Possible Solution

Add a backwards-compatible generic inbound message-origin field to shared context, populate it from Telegram before voice is normalized to generic audio, render it in the agent-visible per-turn conversation metadata, and document the public field with focused regression coverage for voice notes versus ordinary audio and text.
