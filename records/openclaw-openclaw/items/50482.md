## Summary

Keep open. Current main still turns Telegram voice notes into generic audio/transcript context, and neither the shared inbound context type nor the rendered agent metadata exposes a voice-note or native message-type marker. Related #40838 is broader Telegram metadata work and can inform naming, but it does not supersede this narrower request.

## What This Changes

Expose whether an inbound agent turn originated from a voice note, preferably through a generic message-origin or message-type field populated by Telegram and rendered in agent context.

## Best Possible Solution

Add a backwards-compatible generic inbound message-origin field to the shared context, populate it from Telegram before voice is normalized to generic audio, render it in the dynamic conversation-info metadata for agents, and document the new field with focused regression tests for voice notes versus ordinary audio/text.
