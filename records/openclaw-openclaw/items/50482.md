## Summary

Current main still does not expose a voice-note or native message-type flag to agents. Telegram voice notes are transcribed and routed, but the native `msg.voice` versus `msg.audio` distinction is collapsed into generic audio before the shared inbound context and rendered metadata are built; related #40838 is broader and does not supersede this narrower actionable request.

## What This Changes

The issue requests an additive inbound metadata field so agents can distinguish messages that originated as voice notes from typed text or generic audio attachments.

## Best Possible Solution

Keep this open and implement an additive typed inbound message-origin field. Capture Telegram `msg.voice` before audio/voice normalization, carry it through `MsgContext`, render it in the dynamic inbound conversation-info JSON, document it, and add focused shared inbound metadata plus Telegram voice/audio regression tests. Coordinate naming with #40838, but do not block this narrower voice-note use case on that broader context request.
