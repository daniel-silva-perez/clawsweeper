## Summary

Keep open. Current main still loses the Telegram voice-note distinction before agent-visible context rendering: voice notes become generic audio/transcript context, and neither the shared context type nor inbound metadata exposes a native message-origin marker. The broader open Telegram metadata request in #40838 is related, but it does not supersede this narrower voice-origin request.

## What This Changes

Expose an agent-visible inbound metadata field that lets agents distinguish voice notes from typed text or regular audio messages.

## Best Possible Solution

Add a backwards-compatible generic inbound message-origin or modality field to shared context, populate it from Telegram before `msg.voice` is normalized to generic audio, render it in the per-turn agent-visible conversation metadata, and document it with regression coverage for voice notes versus ordinary audio and text.
