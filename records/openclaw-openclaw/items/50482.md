## Summary

Keep open: current main still collapses Telegram voice notes and audio files into generic audio context and never renders a voice/native-message-type field for agents. The related #40838 is broader and still open, so it does not supersede this narrower, actionable request.

## What This Changes

The issue requests exposing whether an inbound message, especially Telegram, originated as a voice note in agent-visible inbound context metadata.

## Best Possible Solution

Add a backwards-compatible, generic inbound message-origin field to the shared context, populate it from Telegram before `msg.voice` is normalized to generic audio, render it in the dynamic conversation-info metadata, and cover the contract with focused shared and Telegram voice/audio regression tests plus docs.
