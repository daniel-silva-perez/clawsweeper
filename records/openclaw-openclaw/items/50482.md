## Summary

Keep open. Current main already transcribes Telegram voice notes and frames the transcript for the agent, but it does not expose the requested structured voice/message-type flag. The Telegram path collapses native `voice` and `audio` into the same generic audio placeholder, and the inbound metadata/context surfaces do not carry `is_voice` or `message_type`.

## Best Possible Solution

Add a typed, generic inbound message modality/source field to the channel-to-agent context contract, populate Telegram from `msg.voice` before it is collapsed into generic audio handling, render it in a cache-safe dynamic inbound context block, and cover it with Telegram plus shared inbound-context tests and docs. Consider equivalent population for WhatsApp, Feishu, Signal, Discord voice-message attachments, and other channels that can identify native voice notes.

## Risks / Open Questions

- Putting a volatile `is_voice` field into the stable system metadata would conflict with the current prompt-cache stability design; this likely needs a cache-safe dynamic context placement.
- The field should be designed generically across channels so it distinguishes voice notes, generic audio attachments, and realtime voice/call surfaces without hard-coding Telegram-only semantics in core.
