## Summary

Keep open. Current main still normalizes Telegram voice notes into generic audio context and only exposes media MIME/transcript information to the agent; there is no agent-visible voice-note or native message-type field in the shared context or rendered inbound metadata. Related #40838 is broader Telegram metadata work and should inform naming, but it does not supersede this narrower request.

## What This Changes

The issue asks OpenClaw to expose whether an inbound message originated as a Telegram voice note, or more generally its native message type, in the agent-visible inbound context.

## Best Possible Solution

Add a backwards-compatible, generic inbound message-origin field to the shared context, populate it from Telegram before voice is normalized to generic audio, render it in the dynamic conversation-info metadata for agents, and cover voice-vs-audio behavior with focused shared and Telegram regression tests plus docs.

Add a backwards-compatible generic inbound message-origin field to the shared MsgContext contract, preferably a native/message type style field rather than a Telegram-only boolean. Populate it from Telegram before msg.voice is collapsed into generic audio, preserving the distinction between Telegram voice notes and regular audio files. Render the new field in the dynamic user-role conversation-info metadata, not the stable trusted inbound_meta system block unless an existing maintainer pattern clearly requires otherwise. Add focused tests that prove Telegram voice notes expose the field, regular Telegram audio does not claim voice-note origin, and shared inbound metadata renders the field safely.
