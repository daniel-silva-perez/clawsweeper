## Summary

Keep open. Current main still normalizes Telegram voice notes into generic audio/transcript context, and neither the shared context type nor the rendered inbound metadata exposes a voice-note or native message-type marker. Related #40838 is broader Telegram metadata work and does not supersede this narrower request.

## What This Changes

Request to expose whether an inbound agent turn originated from a Telegram voice note, preferably through a generic message type or origin field in agent-visible metadata.

## Best Possible Solution

Add a backwards-compatible generic inbound message-origin or modality field to the shared context contract, populate it from Telegram before voice is normalized to generic audio, render it for agents in the inbound conversation metadata, and document the field with regression coverage for voice notes versus ordinary audio and text.
