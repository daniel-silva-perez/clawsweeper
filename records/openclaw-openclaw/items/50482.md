## Summary

Keep open. Current main still normalizes Telegram voice notes into generic audio/transcript context, and neither the shared context type nor the rendered agent metadata exposes a voice-note or native message-type marker. Related #40838 is broader Telegram metadata work and does not supersede this narrower request.

## What This Changes

Expose whether an inbound Telegram or other channel message originated as a voice note in the agent-visible context metadata.

## Best Possible Solution

Add a backwards-compatible generic inbound origin or modality field to the shared context contract, populate it from Telegram before voice is normalized to generic audio, render it for agents in the volatile conversation metadata, and document the field with regression coverage for voice notes versus ordinary audio and text.

## Reproduction Assessment

Yes. A code-level reproduction is high-confidence: a Telegram message with msg.voice reaches STT and MediaTranscribedIndexes, but current source and tests show no agent-visible voice-origin or native message-type field in MsgContext or rendered inbound metadata.

## Solution Assessment

Yes. The narrow maintainable direction is additive context plumbing with a generic field shape; a Telegram-only prompt string or core special case would be less reusable and harder for other channel plugins to follow.
