## Summary

Keep open. Current main still collapses Telegram voice notes into generic audio/transcript context, and the shared context plus rendered inbound metadata do not expose a voice-note or native message-type marker. Related #40838 is broader Telegram metadata work and does not supersede this narrower request.

## What This Changes

Expose whether an inbound message originated as a voice note in the agent-visible inbound context metadata, especially for Telegram messages already routed through STT.

## Best Possible Solution

Add a backwards-compatible generic inbound message-origin or modality field to the shared context contract, populate it from Telegram before msg.voice is normalized to generic audio, render it for agents in the volatile conversation metadata, and document the field with regression coverage for voice notes versus ordinary audio and text.

## Reproduction Assessment

Yes. A code-level reproduction is high-confidence: a Telegram message with msg.voice reaches STT and MediaTranscribedIndexes, but current source and tests show no agent-visible voice-origin field in MsgContext or inbound metadata.

## Solution Assessment

Yes, with a generic shape. The narrow maintainable fix is additive context plumbing rather than a Telegram-only core concept; the field should be reusable by other channels and rendered where per-turn metadata already lives.
