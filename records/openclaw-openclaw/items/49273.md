## Summary

Keep this open. Current main still defaults `commands.nativeSkills` to `"auto"`, still resolves that default on for Discord and Telegram, and still documents that behavior; the existing Telegram/Discord overflow mitigations reduce the startup-failure blast radius but do not implement the requested opt-in default.

## What This Changes

Change the default native skill command registration setting from `auto` to `none`/false so Telegram and Discord do not register every skill as a platform command unless users opt in.

## Best Possible Solution

Keep this issue open for maintainer judgment. If maintainers choose the default flip, update the config schema default, generated schema/help, public docs, resolver tests, Discord/Telegram expectations, and changelog migration note together; if they keep `auto`, land or close the narrower overflow-guidance PR and make the warnings point clearly at `commands.nativeSkills` and per-provider overrides.

## Reproduction Assessment

Yes for the remaining default behavior: source and resolver tests deterministically show `commands.nativeSkills: "auto"` enables native skill commands for Discord and Telegram on current main. The original hard startup failure is partly mitigated now by Telegram capping/retry and Discord dropping per-skill commands after overflow, so the exact 2026.2.21 failure mode was not assumed as current behavior.

## Solution Assessment

Unclear. The requested default change is a coherent fix for quota exhaustion, but it is a product/migration decision; the safer narrow alternative is to keep `auto` and make overflow diagnostics consistently point to `commands.nativeSkills` and `channels.<provider>.commands.nativeSkills`.
