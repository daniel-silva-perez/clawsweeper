## Summary

Keep open. Current main has partial exec-approval hardening, including durable exact-command fallback and Windows-only argPattern support, but it does not implement the issue’s central cross-platform binary+argv allowlist model, deny-over-allow full-command rules, dry-run matcher, approval-log clustering, or progressive rule proposal workflow.

## Best Possible Solution

Keep this open and turn it into a maintainer-owned exec-approval policy design: cross-platform binary+argv allow rules, deny-over-allow full-command matching, a dry-run matcher, an approval/audit log, and an opt-in rule-generalization workflow that proposes scoped patterns only after explicit user confirmation. Cross-link or split implementation with #41140, #6615, #39979, and #46956 as maintainers choose.

## Risks / Open Questions

- Approving a resolved POSIX executable path can still over-trust future argv variants when allowlist mode uses path matches; strictInlineEval only covers a subset of interpreter inline-eval risk and does not provide the requested general binary+args model.
- Related work is split across #41140, #6615, #39979, #46956, and closed PRs #48586/#48590, so maintainers should avoid landing a partial fix that conflicts with the broader policy design.
- The current argPattern field is exposed in schema/UI types but is Windows-only in matcher behavior, which can mislead users expecting cross-platform argument scoping.
