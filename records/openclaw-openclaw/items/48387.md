## Summary

Current main still does not implement this PR’s intended behavior. Explicit WhatsApp sends still reach the plugin resolver with mode "explicit", but the WhatsApp resolver does not branch on that mode and still rejects direct E.164 targets outside a non-empty allowFrom list. The related allowTo proposal in #49549 is open and not a merged/canonical replacement, and the PR patch is stale against the current plugin layout. A security pass also shows the PR would intentionally widen outbound authorization and appears to reverse a shipped security fix, so it needs maintainer product/security judgment rather than cleanup closure.

## Best Possible Solution

Keep this PR open for maintainer judgment, but do not merge it as-is. The best path is to decide the WhatsApp outbound authorization model alongside #49549, #25513, and #25039: either port a deliberately approved explicit-mode bypass into extensions/whatsapp/src with regression tests and docs, or land an opt-in allowTo/per-account outbound policy and then close this PR as superseded after that replacement merges.

## Risks / Open Questions

- The PR intentionally broadens the WhatsApp outbound authorization boundary: callers using explicit mode could send to arbitrary valid direct WhatsApp numbers despite a restrictive allowFrom list.
- That boundary change conflicts with a documented shipped security fix from 2026.2.22, so merging this exact behavior should require explicit maintainer product/security approval.
- The patch targets the old src/whatsapp layout and would need porting to extensions/whatsapp/src before it could be validated on current main.
- #49549 may be the better opt-in model, but it is still open and not a merged canonical replacement.
