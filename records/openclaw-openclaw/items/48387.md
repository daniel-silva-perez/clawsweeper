## Summary

Keep this PR open. Current main still rejects explicit direct WhatsApp outbound targets when a non-empty `allowFrom` list does not include the target, and the PR’s intended behavior is not implemented or shipped. The PR is still a focused implementation candidate, but it needs porting to the current plugin path and maintainer judgment against the broader open `allowTo` design in #49549.

## Best Possible Solution

Keep this PR open as a focused implementation candidate. The best path is to port the resolver change and regression test to `extensions/whatsapp/src`, then have maintainers decide whether explicit user-chosen WhatsApp sends should bypass `allowFrom` or whether the outbound-only `allowTo` design in #49549 should be the canonical authorization model.

## Risks / Open Questions

- The PR intentionally broadens the WhatsApp outbound authorization boundary: explicit user-chosen sends could reach any valid direct number even when `allowFrom` is restrictive.
- The patch targets the older `src/whatsapp` layout, so it must be rebased or ported to `extensions/whatsapp/src` before it can land cleanly.
- Maintainers should choose one coherent model between this explicit-mode bypass and the broader outbound-only `allowTo` policy proposed in #49549.
