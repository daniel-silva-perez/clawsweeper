## Summary

Keep this PR open. Current main still does not implement the proposed explicit WhatsApp outbound bypass, and the proposed behavior conflicts with the documented security boundary that direct-message outbound targets are checked against `allowFrom` in all send modes. The related `allowTo` PR remains the better opt-in product direction but is still open/unmerged, so this PR is not obsolete enough for cleanup closure.

## Best Possible Solution

Keep this PR open for maintainer product/security judgment, but do not merge it as-is. The best path is to decide the WhatsApp outbound authorization model alongside #49549, #25039, and #25513: either land an explicitly approved opt-in `allowTo`/outbound policy with schema, docs, and regression tests in `extensions/whatsapp/src`, or deliberately approve and document an explicit-mode bypass with matching tests and security rationale.

## Risks / Open Questions

- Merging the PR as described would intentionally widen WhatsApp outbound authorization by letting explicit direct sends bypass a restrictive `allowFrom` list.
- That behavior conflicts with the documented 2026.2-era security fix that explicitly included `mode: "explicit"` in allowFrom enforcement.
- The provided PR diff only changes a resolver and test, so there is no observed CI, dependency, workflow, package, or supply-chain mutation; the meaningful security concern is the authorization boundary change.
- The patch targets the old `src/whatsapp` layout, while current main uses `extensions/whatsapp/src`, so it would need porting before any maintainer-approved version could land.
