## Summary

Keep this open. The report is fresh, concrete, and still matches current main: the plugin loader cache key does not include packaged OpenClaw version or package-root identity, while an open PR already references this issue as its closing implementation candidate.

## What This Changes

The issue asks OpenClaw to stop reusing stale bundled browser plugin runtime-deps paths after a 2026.4.26 to 2026.4.27 package upgrade, even after doctor repair and gateway restart.

## Best Possible Solution

Review and land PR #75071, or an equivalent generic loader/runtime-deps fix, so bundled plugin loader cache identity changes when the packaged OpenClaw version or package-root identity changes. Keep the fix at the shared plugin loader seam rather than adding browser-specific recovery behavior.

## Reproduction Assessment

Yes. The issue gives a concrete reproduction path: run 2026.4.26 with the browser plugin enabled, upgrade to 2026.4.27, run doctor repair, restart the systemd gateway, then run `openclaw browser status` and observe an import from the old runtime-deps root.

## Solution Assessment

Likely yes. The linked PR's direction, adding package-version/package-root identity to the loader cache with a regression test, is narrower and more maintainable than a browser-only workaround or a doctor-only cleanup.
