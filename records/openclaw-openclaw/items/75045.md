## Summary

Keep this open. The report is a fresh, concrete upgrade regression with an exact stale `plugin-runtime-deps` import path after moving from 2026.4.26 to 2026.4.27, and current main has nearby runtime-deps, persisted plugin registry, and loader-cache logic but no clearly proven repair for the browser case described after `doctor --repair`.

## What This Changes

The issue requests that bundled browser plugin loading recover after a 2026.4.26 to 2026.4.27 upgrade instead of importing from the removed old `plugin-runtime-deps` mirror.

## Best Possible Solution

Keep the issue open and make bundled plugin runtime-deps upgrade recovery version-aware: startup and doctor repair should rebuild from the current package root, invalidate stale persisted/cached bundled plugin records that point into prior external mirrors, and include a browser-focused upgrade regression test without adding browser-specific policy to core.

## Reproduction Assessment

Yes. The issue provides a high-confidence reproduction path using a 2026.4.26 install with browser enabled, upgrade to 2026.4.27, `openclaw doctor --repair`, gateway restart, and `openclaw browser status`; the exact missing old mirror path is also included.

## Solution Assessment

Unclear. Current main has generic runtime-deps staging and stale persisted-registry checks, but the evidence does not prove they solve this browser old-mirror import after `doctor --repair`; the safer fix is a narrow stale-runtime-root invalidation/repair with targeted regression coverage.
