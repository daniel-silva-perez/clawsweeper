Keep this open. The report is fresh, concrete, and paired with an open closing PR, while current `main` still shows the adjacent loader cache and doctor/runtime-deps repair paths without a proven merged fix for the stale 2026.4.26 browser runtime mirror after a 2026.4.27 upgrade.

## What This Changes

The issue reports a 2026.4.26 to 2026.4.27 upgrade regression where the browser plugin still imports bundled runtime files from the old `plugin-runtime-deps` mirror after `doctor --repair` and gateway restarts.

## Best Possible Solution

Review and land a narrow generic fix that makes bundled plugin runtime-deps loading version-aware across package upgrades, with a regression test for same-path package upgrades and no browser-specific policy in core. After the linked PR merges and verifies the upgrade path, this issue can close as implemented by that merge.

## Reproduction Assessment

Yes. The issue gives a high-confidence reproduction path: run 2026.4.26 with browser enabled, upgrade to 2026.4.27, run `openclaw doctor --repair`, restart the gateway, then run `openclaw browser status` and observe an `ERR_MODULE_NOT_FOUND` import from the old 2026.4.26 runtime-deps root.

## Solution Assessment

Unclear but the linked direction is plausible. A generic package-identity cache invalidation in the plugin loader is the narrowest maintainable direction shown so far, but it should be reviewed against doctor repair, persisted registry, and restart behavior before this issue is closed.
