## Summary

Keep this open. The report is fresh and concrete, current main still does not include package-version or package-root identity in the plugin loader cache key, and open PR #75071 already references this issue as the implementation candidate.

## What This Changes

The issue reports that after upgrading OpenClaw 2026.4.26 to 2026.4.27, the browser plugin still imports from the old versioned bundled plugin runtime-deps root after doctor repair and gateway restarts.

## Best Possible Solution

Review and land PR #75071, or an equivalent generic fix, so bundled plugin loader/runtime-deps cache identity changes when the packaged OpenClaw version or package-root identity changes; keep the repair at the generic loader/runtime-deps seam rather than adding browser-specific recovery policy.

## Reproduction Assessment

Yes. The issue gives a concrete path: run 2026.4.26 with the browser plugin enabled, upgrade to 2026.4.27, run doctor repair, restart the systemd gateway, then run `openclaw browser status` and observe an import from the old runtime-deps root.

## Solution Assessment

Likely yes for the linked direction. Adding package-version/package-root identity to the loader/runtime-deps cache is narrower and more maintainable than a browser-specific workaround, and current main does not yet show that change.
