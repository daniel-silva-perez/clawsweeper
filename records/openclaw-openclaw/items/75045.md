## Summary

Keep this open. The report is fresh and concrete, current main still lacks package-version or package-root identity in the plugin loader cache key, and open PR #75071 is already the implementation candidate for the same runtime-deps upgrade regression.

## What This Changes

The requested fix is to stop the browser plugin from loading stale 2026.4.26 bundled runtime-deps after upgrading OpenClaw to 2026.4.27 and running doctor repair.

## Best Possible Solution

Review and land PR #75071, or an equivalent generic loader/runtime-deps fix, so the bundled plugin registry cache identity changes when the packaged OpenClaw version or package root identity changes; keep the repair at the shared plugin loader seam rather than adding browser-specific recovery policy.

## Reproduction Assessment

Yes. The issue provides a high-confidence reproduction path: enable browser on 2026.4.26, upgrade to 2026.4.27, run doctor repair, restart the systemd gateway, then run `openclaw browser status` and observe an import from the old runtime-deps root.

## Solution Assessment

Likely yes for the linked direction, pending PR review. Adding package-version/package-root identity to the generic plugin loader cache is narrower and more maintainable than a browser-only workaround or a doctor repair that tries to reset process-local state.
