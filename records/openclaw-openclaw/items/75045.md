## Summary

Keep open. Current main still computes the plugin loader cache key without bundled package version or package-root identity, while an open linked PR is already the implementation candidate for the reported upgrade-cache regression.

## What This Changes

The issue reports stale browser bundled runtime-deps imports after upgrading from OpenClaw 2026.4.26 to 2026.4.27, even after doctor repair and gateway restarts.

## Best Possible Solution

The desired end state is a generic plugin loader/runtime-deps cache identity that changes when the packaged OpenClaw package version or package-root identity changes, with regression coverage at the loader/runtime-deps seam and no browser-specific recovery policy.

## Reproduction Assessment

Yes. The issue provides a high-confidence reproduction path: run 2026.4.26 with browser enabled, upgrade to 2026.4.27, run doctor repair, restart the systemd gateway, then run `openclaw browser status` and observe an import from the old runtime-deps root.

## Solution Assessment

Likely yes. The linked direction of invalidating the generic plugin loader cache on packaged OpenClaw identity changes is narrower and more maintainable than adding browser-specific cleanup, but it should close only after PR #75071 or an equivalent fix merges.
