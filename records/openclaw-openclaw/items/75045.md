## Summary

Keep this open. The report is fresh, concrete, and still matches current main: the plugin registry cache key does not include packaged OpenClaw version/package-root identity, while bundled runtime-deps roots are versioned by package version. Open PR #75071 is already linked with closing syntax and is the implementation candidate, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue reports that after upgrading from 2026.4.26 to 2026.4.27, the browser plugin still imports from the old versioned bundled plugin-runtime-deps path even after `openclaw doctor --repair` and gateway restarts.

## Best Possible Solution

Land PR #75071, or an equivalent generic loader/runtime-deps fix, so the plugin registry cache identity changes when the packaged OpenClaw version or package-root identity changes. Keep the fix at the shared plugin loader/runtime-deps seam rather than adding browser-specific repair behavior.

## Reproduction Assessment

Yes. The issue provides a high-confidence reproduction path: enable the browser plugin on 2026.4.26, upgrade to 2026.4.27, run `openclaw doctor --repair`, restart the systemd gateway, then run `openclaw browser status` and observe an import from the old versioned runtime-deps root.

## Solution Assessment

Likely yes for the linked fix direction, subject to PR review. Including packaged OpenClaw version/package-root identity in the loader cache key is narrower and more maintainable than a browser-specific workaround because the stale path comes from shared bundled plugin runtime-deps loading.
