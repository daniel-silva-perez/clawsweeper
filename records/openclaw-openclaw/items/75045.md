## Summary

Keep this open. The report is fresh and concrete, current main still does not visibly include the package-version/package-identity invalidation needed for the plugin loader cache, and open PR #75071 already references this issue with closing syntax as the implementation candidate.

## What This Changes

The issue reports that after upgrading from OpenClaw 2026.4.26 to 2026.4.27, the bundled browser plugin keeps importing from the old plugin-runtime-deps root even after doctor repair and gateway restarts.

## Best Possible Solution

Review and land PR #75071, or an equivalent generic fix, so bundled plugin loader/runtime-deps cache identity changes when the packaged OpenClaw version or package root identity changes; keep the fix at the generic loader/runtime-deps seam rather than adding browser-specific recovery policy.

## Reproduction Assessment

Yes. The issue gives a high-confidence path: browser enabled on 2026.4.26, upgrade to 2026.4.27, run `openclaw doctor --repair`, restart the systemd gateway, then run `openclaw browser status` and observe an import from the old runtime-deps root.

## Solution Assessment

Likely yes for the linked direction, subject to PR review. Adding package-version/package-identity cache invalidation at the plugin loader/runtime-deps seam is narrower and more maintainable than a browser-specific workaround, and current main does not yet show that change.
