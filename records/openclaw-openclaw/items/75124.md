## Summary

Keep open. Current main still matches the reported policy gap: `command-dispatch: tool` skill commands build a raw OpenClaw tool set and only apply owner-only filtering, while the documented and normal agent paths run profile, provider/agent, group, sandbox, and subagent policy filters.

## What This Changes

The issue requests that user-invocable skills using `command-dispatch: tool` dispatch only through the current session's effective tool set instead of raw OpenClaw tools.

## Best Possible Solution

Keep this open and route it through security/tool-policy review. The durable fix should make skill tool dispatch resolve the same session-effective tool surface used by normal agent turns and `/tools`, including profile, provider/agent, group/channel, sandbox, subagent, and owner-only policy, with focused regression tests for a restricted profile and a leaf subagent session.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: define a user-invocable skill with `command-dispatch: tool` and `command-tool: sessions_spawn` in a restricted or leaf subagent session, then invoke it and compare the inline handler's raw `createOpenClawTools(...)` lookup with the filtered `createOpenClawCodingTools(...)` surface.

## Solution Assessment

No. The current implementation is not the best way to solve the issue because it constructs a lower-level tool set in a user-invocable path; the safer direction is to share or extract the effective tool resolver used by the normal runtime instead of duplicating partial filtering.
