## Summary

Keep open. Current main still matches the reported authorization gap: inline skill commands with `command-dispatch: tool` build raw OpenClaw tools and only apply owner-only filtering, while normal agent and `/tools` paths use the layered session-effective policy pipeline. The linked fix PR #75125 is open and unmerged, so this issue should remain open until that fix or an equivalent reviewed patch lands.

## What This Changes

The issue reports that user-invocable skills using `command-dispatch: tool` can resolve a named tool outside the session-effective tool policy surface.

## Best Possible Solution

Make the inline `command-dispatch: tool` path resolve tools through the same session-effective surface used by normal agent turns and `/tools`, preserving target-session group/subagent/sandbox/provider context and adding focused regression coverage for restrictive profiles and leaf subagent sessions.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: define a user-invocable skill with `command-dispatch: tool` and `command-tool: sessions_spawn`, invoke it from a restrictive or leaf-subagent session, and compare the inline branch's raw `createOpenClawTools(...)` lookup with the filtered `createOpenClawCodingTools(...)` surface.

## Solution Assessment

No. The current implementation is not the best way to solve the issue because it constructs a lower-level tool set in a user-invocable path; the safer maintainable direction is a shared session-effective tool resolver for deterministic skill command dispatch.
