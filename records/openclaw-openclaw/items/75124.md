## Summary

Keep open. Current main still matches the reported authorization-policy gap: inline skill slash commands using `command-dispatch: tool` construct raw OpenClaw tools and only apply owner-only filtering, while normal session tools use the layered effective policy pipeline. The linked fix PR #75125 is open and unmerged, so this issue should remain open until that PR or an equivalent reviewed fix lands.

## What This Changes

The issue requests that tool-dispatched skill slash commands use the same effective session tool policy as normal agent turns and `/tools`.

## Best Possible Solution

Review and land #75125 or an equivalent narrowly scoped patch that resolves inline tool-dispatched skill commands through the session-effective tool surface, with regression coverage for restrictive profiles and leaf subagent sessions.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: create a user-invocable skill with `command-dispatch: tool` and a restricted tool such as `sessions_spawn`, invoke it from a restrictive or leaf-subagent session, and observe that current main resolves the tool from raw `createOpenClawTools(...)` after only owner-only filtering.

## Solution Assessment

No for current main. The current raw-tool dispatch path is not the right solution for a user-invocable command surface; the maintainable fix is to reuse the same effective policy resolver used by normal agent turns and `/tools`.
