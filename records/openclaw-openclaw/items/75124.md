## Summary

Keep open. Current main still matches the reported authorization gap: inline `command-dispatch: tool` skill commands build the raw OpenClaw tool set and then apply only owner-only filtering, while the normal command/agent path resolves the session-effective tool set through `createOpenClawCodingTools(...)` and the policy pipeline. The issue also has an open closing PR (#75125), so the right maintainer action is to review and land or reject that PR rather than close this issue early.

## What This Changes

The issue requests that `command-dispatch: tool` skill slash commands honor the same profile, group/channel, sandbox, and subagent effective tool policy as the current session.

## Best Possible Solution

Route inline skill tool dispatch through the same session-effective tool resolver used by normal agent turns, then add focused regression coverage proving restrictive profiles, group/channel policy, sandbox policy, subagent leaf policy, and owner-only filtering all apply before a tool-dispatched skill can execute.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: create a user-invocable skill with `command-dispatch: tool` and a restricted tool such as `sessions_spawn`, invoke it from a restrictive or leaf-subagent session, and compare the inline branch's raw `createOpenClawTools(...)` lookup with the filtered `createOpenClawCodingTools(...)` surface.

## Solution Assessment

No. The current implementation is not the best way to solve the issue because it bypasses the central effective tool-policy pipeline; the safer direction is the open PR's general shape, provided maintainer review confirms the forwarded runtime-policy context is complete.
