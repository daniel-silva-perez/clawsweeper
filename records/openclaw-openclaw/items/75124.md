## Summary

Keep open. Current main still has the reported authorization-policy gap, and the linked open PR is the implementation candidate rather than a reason to close this issue before merge.

## What This Changes

The issue reports that user-invocable skill slash commands using `command-dispatch: tool` can resolve raw tools instead of the session-effective tool policy surface.

## Best Possible Solution

Make inline skill tool dispatch resolve the same session-effective tool surface used by normal agent turns, then add targeted regression coverage for a restrictive tool profile and a leaf subagent session. Review and land the linked implementation PR if it preserves the current session, group, sandbox, sender, and model policy inputs.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists by defining a user-invocable `command-dispatch: tool` skill targeting `sessions_spawn` and invoking it from a restrictive or leaf-subagent session; current main’s inline path resolves via raw `createOpenClawTools(...)` instead of the filtered `createOpenClawCodingTools(...)` surface.

## Solution Assessment

No. Current main is not the best way to solve the issue because it uses a lower-level tool factory on a user-invocable command path; the safer direction is a shared session-effective resolver with regression tests, which is what #75125 attempts to implement.
