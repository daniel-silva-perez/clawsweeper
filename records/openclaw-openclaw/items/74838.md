## Summary

Keep open for maintainer review. Current main still has the reported undefined-result crash path, and this PR's one-line executor fallback is a correct narrow fix, but an overlapping open PR (#74856) covers the same bug with text and Telegram regression tests, so maintainers should choose the canonical branch rather than auto-close or auto-repair this one.

## What This Changes

The branch changes `executePluginCommand` to return `{}` when a plugin command handler returns `null` or `undefined` instead of passing the nullish value through.

## Best Possible Solution

Land one canonical fix that normalizes nullish plugin command handler returns at the executor boundary, keeps the public SDK return type strict, and includes regression coverage for text and Telegram/native command consumers; either adopt #74856 or add equivalent tests to this PR before reconciling the overlap.

## Reproduction Assessment

Yes. A plugin command handler with no explicit return reproduces the source-level path: current main returns `undefined` from `executePluginCommand`, then the text command handler reads `result.continueAgent`; the reported Telegram issue exercises the same shared executor path.

## Solution Assessment

Mostly yes. The central `result ?? {}` guard is the right maintainable boundary, but the best merge artifact should include regression tests and should be coordinated with #74856, which already adds those tests and Telegram call-site defense.
