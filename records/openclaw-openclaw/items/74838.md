## Summary

Keep open. Current main and the reported release still have the unguarded plugin-command result path, and the PR's one-line nullish fallback is a narrow fix for the crash described in #74800. I found no blocking correctness or security issue in the diff, but it still needs normal maintainer review/CI and preferably a focused regression test.

## What This Changes

This PR changes plugin command execution to return an empty command result when a handler returns null or undefined instead of passing the nullish value through to callers.

## Best Possible Solution

Land the shared execution-boundary guard, ideally with a focused regression test that exercises an undefined handler result through `handlePluginCommand`; keep the public SDK type strict while defensively normalizing nullish runtime output from third-party plugins.

## Reproduction Assessment

Yes. A plugin command handler that falls through returns undefined; current main returns that value from `executePluginCommand`, and the text command handler immediately dereferences `result.continueAgent`. This is a high-confidence source-level reproduction of the reported TypeError, even without a live Telegram run.

## Solution Assessment

Yes. Normalizing at `executePluginCommand` is the narrowest maintainable boundary because text, Telegram, Discord, and diagnostics command paths all consume that helper; an empty reply payload already maps to no visible reply or a safe native fallback.
