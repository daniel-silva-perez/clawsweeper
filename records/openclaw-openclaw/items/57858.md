## Summary

Keep open. Current main still has a bundled `coding-agent` skill, but its Claude Code launch guidance runs `claude --permission-mode bypassPermissions --print` directly and does not unset `CLAUDECODE`, `CLAUDE_CODE_SESSION`, or `CLAUDE_CODE_ENTRYPOINT`. The related Claude CLI backend hardening applies to OpenClaw-managed `claude-cli` backend runs, not the raw `bash background:true` path that this skill tells agents to use.

## Best Possible Solution

Keep this issue open and patch the bundled `skills/coding-agent/SKILL.md` Claude Code guidance so every Claude launch uses `env -u CLAUDECODE -u CLAUDE_CODE_SESSION -u CLAUDE_CODE_ENTRYPOINT claude --permission-mode bypassPermissions --print ...`. Add a small bundled-skill guidance test, likely near `src/agents/skills.env-path-guidance.test.ts`, so the regression is caught. Consider separately whether the generic host exec sanitizer should drop those Claude nesting variables for broader protection.
