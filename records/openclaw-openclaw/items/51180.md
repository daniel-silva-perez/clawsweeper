## Summary

Keep this PR open. Current main still has the live `channels status --probe` truncation this PR targets, so the work is not obsolete; however, the patch should get maintainer follow-up because it overlaps with open PR #51179 and needs a clear allowlist display/privacy policy plus focused tests before landing.

## What This Changes

This PR changes the live `channels status --probe` formatter to show up to five `allowFrom` entries with a `,+N more` suffix instead of hard-truncating after two entries.

## Best Possible Solution

Use one canonical allowlist display helper for the live gateway status formatter and the config/status summary path, choose a clear show-all versus capped/redacted policy, render overflow with an unmistakable marker, and cover two-entry, three-entry, exact-limit, and overflow cases in focused tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to pass a live gateway status payload with three `allowFrom` entries into `formatGatewayChannelsStatusLines()` on current main; the formatter slices to two entries at `src/commands/channels/status.ts:96`. The original bug report also provides a WhatsApp `channels status --probe` reproduction path with three or more allowlisted numbers.

## Solution Assessment

No, not yet. The PR is a narrow and useful live-formatter fix, but the best maintainable solution should coordinate with #51179, share the formatting policy, avoid an ambiguous overflow token, and add regression tests before merge.
