## Summary

Keep open. Current main still has enough of the reported stale-provider context-window collision surface that this PR is not obsolete, but the branch is not merge-ready because it changes merge-mode provider retention semantics, carries unresolved provider-context accounting bugs, and mixes in unrelated embedded-runner/cron terminal-error behavior.

## What This Changes

This PR removes existing-provider seeding from `models.json` merge mode, switches several session/status/runtime context-window lookups toward provider-qualified resolution, and adds focused tests plus embedded-runner/cron terminal-error handling changes.

## Best Possible Solution

Revise and narrow the branch: explicitly decide and document merge-mode orphan-provider semantics, preserve or migrate legitimate `models.json`-only providers, use provider-aware context resolution in gateway defaults, CLI session display, and cron accounting, keep persisted session `contextTokens` as a fallback when provider/model are known, and split embedded-runner or cron terminal-error changes into separate reviewable work.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Preserve models.json-only providers in merge mode:** `src/agents/models-config.merge.ts:221`
  - body: Removing the existing-provider seed means every provider that exists only in the agent `models.json` is omitted from the next merge result. Current docs and tests treat `models.json` as a merge source, so this can delete manually maintained providers when the catalog regenerates; constrain the cleanup to known generated orphans or introduce an explicit ownership/migration rule.
  - confidence: 0.87
- **[P1] Keep persisted context as a fallback:** `src/auto-reply/reply/agent-runner.ts:525`
  - body: `contextTokensOverride` is authoritative in `resolveContextTokensForModel`, so passing `activeSessionEntry?.contextTokens` there bypasses provider/model lookup. In the stale-window scenario this PR is meant to fix, reply runs can keep writing the poisoned context window instead of correcting it from the known provider and model.
  - confidence: 0.9
- **[P1] Keep follow-up context as a fallback:** `src/auto-reply/reply/followup-runner.ts:282`
  - body: The follow-up runner has the same stale-value path: a persisted `sessionEntry.contextTokens` value passed as `contextTokensOverride` prevents the provider-qualified lookup from running. That leaves follow-up accounting and thresholds stuck on the old context window after the provider/model are known.
  - confidence: 0.89
- **[P2] Preserve mutating-tool retry warnings:** `src/agents/pi-embedded-runner/run.ts:1080`
  - body: The embedded-runner refactor can prefer a friendly transport error over the existing incomplete-turn message that warns users when mutating tools may already have executed. If a disconnect happens after a mutating tool call, users may retry without the side-effect caution they previously saw.
  - confidence: 0.76
