## Summary

Keep open. The stale context-window collision is a real area, but current main does not fully implement this PR's intended fix, and the branch still needs maintainer revision because it changes merge-mode provider retention semantics and carries unresolved review findings around stale persisted context values.

## What This Changes

The PR changes generated models.json provider merging, switches several session/status/runtime context-token paths toward provider-qualified resolution, adds focused regression tests, and also changes embedded-runner and cron empty-turn error handling.

## Best Possible Solution

Keep this PR open for maintainer-led revision: decide the merge-mode deletion contract, narrow the branch to provider-aware context resolution, keep persisted session contextTokens as fallback when provider/model metadata is known, and split terminal-error behavior into separate work.

## Reproduction Assessment

Yes for source-level reproduction. Current main still shows the merge retention path, gateway default bare lookup, CLI model-only lookup, and cron bare lookup; I did not run tests because this review is explicitly read-only.

## Solution Assessment

No, not as currently proposed. Provider-qualified context resolution is the right direction, but this branch should be rebased and narrowed, with explicit merge-mode semantics and the stale-session fallback review findings addressed before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Keep persisted context as fallback in reply accounting:** `src/auto-reply/reply/agent-runner.ts:522`
  - body: Passing activeSessionEntry?.contextTokens through contextTokensOverride makes a stored stale value authoritative. In the stale-provider collision this PR is meant to fix, that bypasses provider/model lookup and keeps persisting the wrong window instead of correcting it.
  - confidence: 0.92
- **[P1] Keep session context as fallback in follow-up accounting:** `src/auto-reply/reply/followup-runner.ts:282`
  - body: sessionEntry?.contextTokens is also passed as contextTokensOverride here, so follow-up turns with known provider/model metadata can keep reusing a poisoned context window. The stored session value should only be the fallback after provider-aware lookup fails.
  - confidence: 0.9
- **[P2] Preserve the mutating-tool retry warning:** `src/agents/pi-embedded-runner/run.ts:1082`
  - body: The new empty-turn error text prefers the provider-friendly error over the mutating-tool warning. If a turn already executed a likely mutating tool before a disconnect, users lose the existing caution that tool actions may already have happened before retrying.
  - confidence: 0.78
