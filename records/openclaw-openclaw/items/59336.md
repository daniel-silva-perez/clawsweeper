## Summary

Keep this PR open. The Raw mode regression is still present on current main and the PR targets the right config/redaction surface, but the proposed delete-based safe-bin normalization is not merge-ready because it can expose inherited exec safe-bin policy values on a security-sensitive path.

## What This Changes

The PR deletes empty normalized exec safe-bin config keys, adds undefined-key stripping plus an optional source comparison target to raw-redaction fallback checks, and passes `snapshot.parsed` as that comparison target.

## Best Possible Solution

Keep the source-shape round-trip comparison direction, but revise the implementation so normalization remains prototype-safe: preserve an own cleared value or clone onto a null/plain own-property object, and put undefined-key stripping in the comparison helper rather than relying on `delete` for exec policy fields.

## Reproduction Assessment

Yes. For the underlying bug, current main can be reproduced with a snapshot whose materialized `snapshot.config.tools.exec` has own `safeBinProfiles: undefined` while the raw-parsed object has no such key, causing `isDeepStrictEqual` to fail. For the PR finding, an exec config object with inherited `safeBinProfiles` or `safeBinTrustedDirs` becomes observable after the proposed `delete` lines remove the own cleared fields.

## Solution Assessment

No. Comparing restored raw text against the raw/source shape and ignoring undefined-valued keys is the right narrow direction, but deleting safe-bin fields in the materialized config is not the best fix because those fields feed exec approval policy. A safer solution should make the raw round-trip comparison tolerant without changing cleared exec-policy object shape in a way that exposes inherited properties.
