## Summary

Keep open. Current main still lacks a generic stable `openclaw/plugin-sdk/embeddings` SDK subpath, and the only documented/exported embedding helper facade remains the memory-core-named path; the related implementation PR was closed unmerged rather than superseding this issue.

## What This Changes

Expose a stable documented embedding-provider SDK subpath for third-party memory-slot plugins to reuse embedding factories, registry access, types, and batch helpers without importing the memory-core-named facade.

## Best Possible Solution

Add an owner-approved, additive SDK subpath such as `openclaw/plugin-sdk/embeddings`, keep the existing memory-core-named path backward compatible, and update package exports, SDK entrypoint metadata, docs, API/export checks, and focused contract coverage together.

## Reproduction Assessment

Yes. Static inspection on current main reproduces the gap: there is no `src/plugin-sdk/embeddings.ts`, export-map entry, SDK entrypoint registry entry, or docs row for `openclaw/plugin-sdk/embeddings`.

## Solution Assessment

Unclear pending SDK owner input. A narrow additive subpath appears consistent with the plugin boundary rules, but the stable name and exported contract should be approved before a repair PR is queued.
