## Summary

Keep open. Current main still lacks a generic stable `openclaw/plugin-sdk/embeddings` SDK subpath; the embedding helper surface exists only through the memory-core-named path, and the related implementation PR was closed unmerged rather than superseded.

## What This Changes

Request a stable, documented plugin SDK subpath for embedding provider APIs so third-party memory-slot plugins can use embedding helpers without depending on memory-core internals.

## Best Possible Solution

Add an owner-approved, backwards-compatible SDK subpath such as `openclaw/plugin-sdk/embeddings`, keep the existing memory-core-named path working, and update package exports, SDK entrypoint metadata, docs, API/export checks, and focused contract tests together.

## Reproduction Assessment

Yes. The missing public surface is reproducible by static inspection on current main: there is no `src/plugin-sdk/embeddings.ts`, package export, entrypoint registry item, or docs entry for `openclaw/plugin-sdk/embeddings`.

## Solution Assessment

Unclear pending SDK owner input. A narrow additive SDK subpath appears maintainable and consistent with the plugin boundary rules, but maintainers should decide the stable name and exported contract before implementation.
