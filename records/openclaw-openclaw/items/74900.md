## Summary

Keep open. Current main still exposes embedding helpers only through the memory-core-named SDK facade, while this request is specifically for a stable plugin-neutral public SDK subpath. The related implementation PR was closed unmerged, so this remains a real SDK contract decision rather than an implemented, duplicate, or ClawHub-only cleanup.

## What This Changes

Request a stable documented `openclaw/plugin-sdk/embeddings` subpath that exposes embedding provider helpers independently of the active `memory-core` plugin while preserving the existing memory-core-named path.

## Best Possible Solution

Add a narrow, backwards-compatible, owner-approved generic embeddings SDK subpath, keep the memory-core-named facade working, and update the export map, SDK entrypoint metadata, docs, API baseline/export checks, and focused contract coverage together.

## Reproduction Assessment

Yes. Static inspection against current main reproduces the missing public surface: there is no `src/plugin-sdk/embeddings.ts`, package export, entrypoint registry item, or docs entry for `openclaw/plugin-sdk/embeddings`.

## Solution Assessment

Unclear pending SDK-owner input. A narrow additive subpath appears consistent with the plugin boundary rules, but maintainers should decide the stable name, exported symbol set, and registry lifecycle contract before implementation.
