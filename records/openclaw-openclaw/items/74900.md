## Summary

Keep open. Current main still exposes the embedding helpers only through the memory-core-named SDK subpath, and there is no generic `openclaw/plugin-sdk/embeddings` export or docs entry. The request is a real public SDK/API-shape issue rather than stale cleanup; the prior related PR was closed unmerged and this issue is the narrower owner-input follow-up.

## What This Changes

Request a stable, documented plugin SDK subpath for memory embedding provider types, factories, registry helpers, and batch helpers that is not tied to the memory-core-named import path.

## Best Possible Solution

Add an owner-approved, backwards-compatible SDK subpath such as `openclaw/plugin-sdk/embeddings` that re-exports the existing memory embedding contracts/helpers from the current facade, keep the old memory-core-named path working, and update the SDK entrypoint registry, package exports, docs, and API/export checks together.

## Reproduction Assessment

Yes for the missing SDK surface: current main has no `src/plugin-sdk/embeddings.ts`, no package export, and no docs entry for `openclaw/plugin-sdk/embeddings`. A runtime plugin reproduction is not necessary to confirm the requested public subpath is absent.

## Solution Assessment

Unclear pending SDK owner input. A narrow generic subpath appears maintainable and consistent with the plugin API direction, but the issue should stay open until maintainers confirm the exact stable contract and whether bundled plugins should migrate immediately or only docs should prefer the new path.
