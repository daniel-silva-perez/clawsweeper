## Summary

Keep this open. Current main still records message-tool sends as pending at tool start, discards failed sends without adding suppression state, and suppresses both message-end block replies and final payloads only from committed successful sends; that leaves the failed-send plus in-flight retry duplicate-delivery window reported here. The existing ClawSweeper comment remains consistent with the current source, and adjacent open PR #61183 addresses adjusted tool arguments rather than this retry suppression bug.

## What This Changes

The issue reports duplicate visible delivery when a failed messaging tool send is retried in the same assistant turn and the automatic text reply is emitted before the retry result commits.

## Best Possible Solution

Fix the shared embedded messaging-tool lifecycle by tracking same-turn in-flight retry sends separately from committed successful deliveries, then make message_end block replies and final payload suppression consider same-target same-text retry state while preserving fallback delivery when no send succeeds.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is: failed message tool result, retry tool_execution_start with the same target/text, then message_end before retry tool_execution_end; current main still emits the block/final text because suppression sees only committed successful sends.

## Solution Assessment

Yes. A narrow shared lifecycle fix is the best direction; this should not be Telegram-specific, and it needs focused coverage for failed first send, in-flight retry, message_end suppression, retry success, and retry failure fallback.

## Repair Work Prompt

Fix the failed messaging-tool send retry duplicate-delivery bug described in https://github.com/openclaw/openclaw/issues/55640. Work in the shared embedded message-tool lifecycle, not Telegram-specific code. Current main records pending message sends in `src/agents/pi-embedded-subscribe.handlers.tools.ts`, deletes pending entries on failed tool results, and only exposes committed successful sends to message_end/final-payload suppression. Add a retry-aware in-flight suppression state that lets same-turn, same-target, same-text retries suppress automatic block replies and final payloads while preserving fallback delivery if no send ever succeeds. Keep provider/account/thread target semantics intact and do not suppress cross-target sends. Coordinate with adjacent PR #61183, which touches adjusted message-tool bookkeeping but does not solve this issue. Add focused regressions for: first `message` send fails, retry `tool_execution_start` records the same target/text, `message_end` arrives before retry result, block reply is suppressed, retry success commits normally; also cover retry failure/no-success fallback so the only visible reply is not lost. Changelog is expected because this is user-facing duplicate delivery behavior. Do not broaden text-leak policy from #25592 beyond this narrow retry lifecycle.
