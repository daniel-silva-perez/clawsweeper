## Summary

Keep this issue open. Current main still has the reported failed-send/retry window: messaging sends are tracked as pending at tool start, but pending entries are deleted on failed tool results and suppression only consults committed successful sends. If a retry starts and a `message_end` block reply is emitted before the retry result commits, the assistant text is still delivered even though the retry message tool call is in flight.

## Best Possible Solution

Keep this issue open and fix the embedded messaging-tool lifecycle. Track same-turn pending/retried message sends separately from committed deliveries, then let both block-reply and final-payload suppression consider a same-target, same-text in-flight retry while preserving fallback delivery if no send ever succeeds. Add a focused regression test for failed first send, retry start, `message_end` before retry result, and retry success.

## Evidence

## Risks / Open Questions

- A naive pending-send suppression set could hide the only visible reply if the retry also fails or never returns a tool result.
- The fix must preserve same-target routing semantics so a message sent to another channel, account, or Telegram topic does not suppress the current conversation reply.
- The broader open text-leak tracker #25592 overlaps the delivery boundary, but this issue has a narrower failed-send/retry repro that remains actionable independently.
