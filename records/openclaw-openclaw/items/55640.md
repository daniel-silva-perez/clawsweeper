## Summary

Keep this issue open. Current main still has the reported retry window: message tool sends are recorded as pending at tool start, but failed sends are discarded and in-flight retry state is not considered by either block-reply suppression or final-payload suppression. The broader #25592 text-leak tracker overlaps, but this issue has a focused failed-send/retry duplicate-delivery repro that remains actionable on its own.

## Best Possible Solution

Keep this issue open and fix the embedded messaging-tool lifecycle. Track same-turn pending/retried message sends separately from committed deliveries, then let both block-reply and final-payload suppression consider a pending same-target, same-text retry while still falling back to normal replies when no successful or in-flight retry exists. Add a focused regression test for failed first send, retry start, message_end before retry result, and retry success.

## Risks / Open Questions

- A naive pending-send suppression set could hide the only visible reply if the retry also fails or never returns a tool result.
- The fix must preserve same-target routing semantics so a message sent to another channel or account does not suppress the current conversation reply.
- The broader #25592 work may overlap the delivery boundary, but it does not replace this focused failed-send/retry regression.
