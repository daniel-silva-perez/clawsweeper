## Summary

Keep this issue open. The requested Android/iOS reverse-proxy transport auth is not implemented on current main, and the same author has an open implementation candidate, PR #73168, explicitly linked as closing this issue. Repository cleanup rules say not to close an issue while an open PR with closing syntax is still pending.

## Best Possible Solution

Keep the issue open and use PR #73168 as the current implementation candidate. The best path is to review or revise that PR so Android and, if still in scope, iOS can carry proxy-layer bearer/basic credentials from setup/manual connection into the initial HTTPS/WebSocket upgrade without weakening existing OpenClaw pairing, token/password auth, or TLS pinning semantics.
