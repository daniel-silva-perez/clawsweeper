## Summary

Keep this issue open. Current main still lacks proxy-auth fields in the setup payload and mobile clients still build the initial WebSocket connection without bearer/basic/custom proxy headers; the issue is also paired with open implementation candidate #73168, which should be reviewed or closed before this issue is resolved.

## What This Changes

The issue requests transport-level proxy authentication support for mobile clients so Android and iOS can pass reverse-proxy credentials during the initial HTTPS/WebSocket connection before OpenClaw pairing auth runs.

## Best Possible Solution

Keep this issue open and use #73168 as the current implementation candidate. The desired end state is an additive, documented mobile transport-auth design that preserves OpenClaw pairing auth while giving Android and iOS a scoped way to send proxy credentials on the intended HTTPS/WebSocket transport path.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: current setup-code generation has no proxy-auth fields, Android builds the WebSocket request without headers, and iOS creates WebSocket tasks from only a URL, so a reverse proxy requiring Authorization before upgrade cannot be satisfied by current main.

## Solution Assessment

Unclear for the final implementation. The issue should remain open because #73168 is a plausible candidate, but maintainers still need to decide whether the first fix is Android-only bearer/basic support or a broader cross-platform setup-code schema covering headers, cookies, and OIDC.
