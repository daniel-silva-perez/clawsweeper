## Summary

Keep this PR open. Current main still does not implement the Windows-drive `sessionFile` handling described by the PR, and the submitted patch is not safe to close as implemented because it collapses Windows paths to a basename without preserving agent/session-store context.

## Best Possible Solution

Keep this PR open but request a revision: handle Windows drive-letter `sessionFile` values before POSIX containment using `path.win32` or an existing Windows-path helper, preserve `agents/<agentId>/sessions/<file>` context instead of reducing to the basename, use the same realpath/containment rules as the current resolver, reject degenerate paths safely, and add focused tests for both slash styles plus cross-agent and fallback cases.

## Risks / Open Questions

- Closing this PR as implemented would lose a still-current Windows-on-POSIX session path bug; current main still treats drive-letter paths as relative on POSIX.
- Landing the submitted patch unchanged risks cross-agent transcript aliasing because it resolves `C:\...\agents\bot2\sessions\sess-1.jsonl` as `<current sessionsDir>/sess-1.jsonl`.
- The patch needs regression coverage for backslash drive paths, forward-slash drive paths, trailing separators or drive roots, symlink/realpath consistency, and cross-agent Windows-authored session paths.
