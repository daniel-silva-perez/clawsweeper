## Summary

Keep open. Current main still has source paths matching the reported Active Memory timeout, embedded abort propagation, and prompt-error carryover surfaces, and the related broader report is closed rather than a canonical open tracker or merged fix.

## What This Changes

The issue reports that Active Memory embedded recall aborts embedded `memory_search` tool calls, reports completed `NONE` responses as timeouts, and leaks prior prompt-error entries into later recall prompts.

## Best Possible Solution

Keep Active Memory bounded by a hard deadline, but separate deadline enforcement from ordinary memory-tool dispatch until the deadline actually fires; classify a completed `NONE` recall as `empty`; prevent stale `openclaw:prompt-error` custom entries from entering the next recall prompt; and cover those paths with focused Active Memory and embedded-runner regression tests.

## Reproduction Assessment

Yes. The issue provides a high-confidence reproduction path using Active Memory with persisted transcripts plus a direct `memory_search` control, and a second user independently confirmed the behavior on v2026.4.27.

## Solution Assessment

Unclear because there is no proposed patch. The narrow maintainable repair is in Active Memory's embedded-run timeout lifecycle plus the embedded-runner custom-entry filtering path, not model tuning or a generic timeout-doctor workaround.

## Repair Work Prompt

The observable bug is that Active Memory embedded runs can return `Aborted` for `memory_search` tool calls, then classify a completed `NONE` assistant response as `status=timeout`, and prior `openclaw:prompt-error` custom entries can leak into the next recall prompt. Keep the fix scoped to Active Memory and embedded-runner lifecycle/filtering.
