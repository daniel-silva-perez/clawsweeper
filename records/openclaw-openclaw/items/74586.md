## Summary

Keep open. The report is fresh, source-localized, and independently confirmed on v2026.4.27; current main has partial Active Memory timeout recovery but still leaves the embedded abort/tool path and empty-or-NONE timeout classification plausibly unresolved.

## What This Changes

Active Memory should stop aborting embedded memory_search calls prematurely, classify a completed NONE recall as an empty result instead of a timeout, and prevent prior prompt-error metadata from leaking into the next recall run.

## Best Possible Solution

Keep the Active Memory hard deadline, but separate deadline enforcement from ordinary memory tool execution until the deadline actually expires; treat a completed NONE recall as a clean empty result, and filter prior prompt-error custom entries out of fresh Active Memory recall prompts.

## Reproduction Assessment

Yes. The issue gives a high-confidence reproduction path with Active Memory enabled, persistTranscripts true, direct chat, and direct memory_search as a control; a second user confirmed the same behavior on v2026.4.27.

## Solution Assessment

Unclear because there is no proposed patch here, and current main's partial timeout recovery is not enough proof. The narrowest maintainable fix is a targeted Active Memory/embedded-runner regression and runtime change rather than a generic AI API timeout-doctor workaround.
