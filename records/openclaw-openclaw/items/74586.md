## Summary

Keep open. The fresh report has a precise transcript/control reproduction and an independent confirmation, while current main still contains the same Active Memory watchdog-to-embedded-run abort path, the same tool wrapper that can surface "Aborted", and the same timeout recovery path that maps empty or NONE output back to `status=timeout`.

## What This Changes

Active Memory should stop aborting embedded `memory_search` calls prematurely, classify completed NONE/empty recall as a clean empty result instead of timeout, and avoid carrying prior prompt-error events into the next recall run.

## Best Possible Solution

Keep Active Memory's hard deadline, but separate deadline enforcement from normal memory tool calls until the deadline actually expires; then treat a completed `NONE` recall as `empty`, preserve useful partial summaries only when the run truly times out, and filter prior `openclaw:prompt-error` custom entries out of fresh Active Memory recall prompts.

## Reproduction Assessment

Yes. The issue gives a high-confidence path using Active Memory with persisted transcripts and a direct `memory_search` control, and a second user confirmed the same behavior on v2026.4.27 with direct chat and a different configuration.

## Solution Assessment

Unclear because there is no proposed patch here. The narrow maintainable direction is a targeted Active Memory and embedded-runner regression fix, not a generic AI timeout-doctor workaround or a model tuning change.
