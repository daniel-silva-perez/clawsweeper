## Summary

Keep open. Current main still routes Active Memory's watchdog abort signal through the embedded runner and tool wrapper, so the reported `memory_search` `Aborted` symptom and timeout classification race remain plausible; the existing cleanup/partial-timeout paths do not prove that a completed `NONE` reply after embedded tool aborts is handled as a clean empty result.

## What This Changes

The issue asks OpenClaw to stop aborting Active Memory embedded `memory_search` calls, classify completed embedded `NONE` replies correctly, and prevent stale prompt-error entries from leaking into later recall runs.

## Best Possible Solution

Preserve Active Memory's hard deadline while decoupling it from ordinary memory tool execution until the budget actually expires, treat completed `NONE` responses as `empty` instead of timeout, and prevent prior prompt-error custom entries from entering the next recall prompt.

## Repair Work Prompt

Fix the Active Memory embedded recall abort/timeout bug reported in https://github.com/openclaw/openclaw/issues/74586. Observable behavior: inside the Active Memory embedded run, `memory_search` tool calls can return `AbortError: Aborted`, the assistant can still complete with `NONE`, and Active Memory records `status=timeout` with no summary; prior prompt-error custom entries may also appear in the next recall prompt. Keep the fix inside the Active Memory plugin and the generic embedded-runner/tool abort seams as needed. Preserve the hard Active Memory deadline, but do not let the deadline abort normal memory tool execution before the budget actually expires, and do not discard completed embedded payloads solely because cleanup or a late signal fired. Treat completed `NONE` as an `empty` result. Filter or retire stale `openclaw:prompt-error` context so a prior failed recall does not contaminate the next Active Memory prompt. Add focused regression tests in Active Memory first; touch embedded-runner tests only if the shared runner behavior changes. Do not change model fallback semantics, bundled provider defaults, unrelated memory-core behavior, or broad plugin SDK contracts. Add a single-line changelog fix entry with `Thanks @islandpreneur007` if runtime behavior changes.
