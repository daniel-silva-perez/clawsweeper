## Summary

Keep open. The issue is fresh, has two independent production reproductions with persisted transcript/control evidence, and current main still contains the same Active Memory watchdog signal, embedded-runner abort propagation, timeout classification, and prompt-error carryover surfaces described in the report.

## What This Changes

Fix Active Memory embedded recall so `memory_search` calls are not aborted prematurely, completed `NONE` recalls are not reported as timeouts, and stale prompt-error entries do not leak into later recall prompts.

## Best Possible Solution

Keep Active Memory bounded, but separate deadline enforcement from ordinary memory tool dispatch until the deadline actually fires; classify a completed `NONE` recall as `empty`; filter prior `openclaw:prompt-error` custom entries out of new recall prompts; and add focused regression coverage around those paths.

Yes. The issue provides a high-confidence reproduction path using Active Memory with persisted transcripts plus a direct `memory_search` control, and a second user independently confirmed the same behavior on v2026.4.27.

## Solution Assessment
