## Summary

Keep open. Current main still appends preserved existing service PATH entries during forced gateway installs, and that preservation filter does not exclude the same stale version-manager/package-manager segments that the service audit reports as non-minimal.

## What This Changes

The issue asks for `openclaw gateway install --force` to generate a gateway service PATH that `openclaw doctor` and `gateway status` accept as minimal instead of recreating non-minimal PATH warnings.

## Best Possible Solution

Make service reinstall planning converge on the same PATH contract as the audit: preserve safe existing PATH additions only when they will not be reported as non-minimal, and add a regression test where a forced reinstall starts from an old unit PATH containing stale version-manager/package-manager entries.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call `buildGatewayInstallPlan` with an `existingEnvironment.PATH` containing stale `.volta`, `.asdf`, `.nvm/current`, `.fnm/current`, or pnpm entries that are absent from the computed minimal set, then audit the resulting environment and observe `gateway-path-nonminimal`.

## Solution Assessment

Yes. The narrowest maintainable fix is to make preserved existing PATH entries follow the same non-minimal classification as the audit, while keeping explicit configured roots and safe custom directories intact.
