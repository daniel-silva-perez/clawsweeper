## Summary

Keep open. Current main still has the repair loop described here: fresh service PATH generation gates hard-coded version-manager fallbacks on directory existence, but forced reinstall and doctor repair still append preserved PATH entries from the existing service, and the audit then flags those same stale entries as non-minimal.

## What This Changes

The issue asks for Linux gateway service reinstall/doctor repair to generate a PATH that the service audit accepts as minimal instead of recreating version-manager/package-manager entries it warns about.

## Best Possible Solution

Make service reinstall planning converge on the same PATH contract as the audit: preserve existing PATH additions only when they are audit-clean or already part of the computed minimal service PATH, and add a regression test for a forced reinstall starting from an old unit PATH with stale version-manager/package-manager entries.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call `buildGatewayInstallPlan` with an `existingEnvironment.PATH` containing stale `.volta`, `.asdf`, `.nvm/current`, `.fnm/current`, or pnpm entries absent from the computed minimal set, then audit the planned environment and observe `gateway-path-nonminimal`.

## Solution Assessment

No. Current main is only a partial fix because it filters missing hard-coded fallback directories during fresh PATH generation, but it still preserves stale existing PATH segments that the audit rejects during forced reinstall and doctor repair.

## Repair Work Prompt

Fix the Linux gateway service PATH repair loop from https://github.com/openclaw/openclaw/issues/75220. Keep the change scoped to service reinstall/doctor planning for stale preserved PATH entries: `openclaw gateway install --force` and doctor repair should not preserve existing unit PATH segments that `auditGatewayServiceConfig` will immediately report as `gateway-path-nonminimal`, while still preserving safe custom PATH additions and explicit env-configured roots that belong in the computed minimal service PATH. Likely implementation area is `src/commands/daemon-install-helpers.ts`, possibly sharing/centralizing the audit-style minimal/non-minimal classification with `src/daemon/service-audit.ts` or `src/daemon/service-env.ts` if that avoids duplication. Add regression coverage where `buildGatewayInstallPlan` starts from an existing Linux service PATH containing stale missing `.volta`, `.asdf`, `.nvm/current`, `.fnm/current`, and/or pnpm entries and produces a PATH that the audit accepts. Do not reopen the broader Linux minimal-PATH policy debate, do not remove support for safe custom PATH entries, and do not change unrelated macOS #75233 behavior unless the same generic helper fix naturally covers it. Changelog entry is expected because this is user-facing daemon/doctor behavior; credit the reporter/related contributor handles according to repo policy.
