## Summary

Keep open. Current main still lets the Linux service reinstall and doctor repair paths append preserved `PATH` segments from the existing systemd unit, while the audit still reports those same version-manager/package-manager segments as non-minimal. The related open PR #71619 overlaps Linux minimal-PATH policy but only touches `service-env` code/tests, so it does not cover this issue's stale existing-unit preservation path.

## What This Changes

Fix `openclaw gateway install --force` and doctor repair so the generated Linux gateway service `PATH` is accepted by OpenClaw's own minimal-PATH audit.

## Best Possible Solution

Make reinstall planning converge on the same PATH contract as service audit: preserve safe existing PATH additions only when they will not be reported as non-minimal, and add a regression test where a forced reinstall starts from an old unit PATH containing stale version-manager/package-manager entries.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call `buildGatewayInstallPlan` with `existingEnvironment.PATH` containing stale `.volta`, `.asdf`, `.nvm/current`, `.fnm/current`, or pnpm entries that are absent from the computed minimal set, then audit the resulting environment and observe `gateway-path-nonminimal`.

## Solution Assessment

Yes. The narrow maintainable fix is to align the preserved-PATH filter with the audit contract, rather than relaxing doctor warnings or removing all custom PATH preservation.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/75220. On current main, `gateway install --force` and doctor repair pass the existing service environment into `buildGatewayInstallPlan`; `mergeServicePath` then appends preserved existing `PATH` entries after the new minimal service path. Add a narrow repair so preserved existing PATH segments that would be reported by the service audit as non-minimal version-manager/package-manager entries are not re-appended when absent from the newly computed minimal path. Preserve neutral custom absolute PATH additions, and keep the existing temp/proc/workspace hardening. Add a regression test in `src/commands/daemon-install-helpers.test.ts` where `serviceEnvironment.PATH` is minimal and `existingEnvironment.PATH` contains stale `.volta`, `.asdf`, `.nvm/current`, `.fnm/current`, and `.local/share/pnpm` entries plus one safe custom directory; expect the stale entries to be dropped and the safe directory preserved. Do not broaden into #71619's Linux minimal-PATH policy changes, do not change dependency pins or lockfiles, and add a `CHANGELOG.md` fix entry with appropriate reporter credit if this user-facing behavior changes.
