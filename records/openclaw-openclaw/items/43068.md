## Summary

Keep this PR open. Current main still lacks sandbox-only Python/Node research-script import templates, so the intended hardening is not implemented. The PR targets a real exec hardening gap, but the proposed scanner has unresolved security-relevant parser bypasses that should be fixed before merge.

## Best Possible Solution

Keep this PR open and request changes. The best path is to rebase onto current main, add sandbox-only import-template validation after the existing safe script preflight, harden parsing for multiline Node import/re-export syntax and whitespace-separated dynamic require/import calls, handle Python bare-relative denied-name imports, and add regression tests proving sandbox-vs-host behavior plus the identified bypasses.

## Risks / Open Questions

- Closing this PR would leave current main without sandbox-only import hardening for direct Python/Node research scripts.
- Merging the PR as-is could create false confidence in sandbox import enforcement because standard multiline Node syntax and whitespace-separated dynamic calls can bypass the proposed text scanner.
- The PR touches command-execution preflight behavior. I did not see workflow, lockfile, dependency-source, release-script, or package-publishing changes in the provided diff, so the main security concern is correctness of the new enforcement boundary rather than supply-chain expansion.
