## Summary

Keep open. Current main still does not implement sandbox-only Python/Node research import-template validation, so the PR is not obsolete; however, the proposed scanner has concrete command-execution hardening bypasses and the branch predates the current exec preflight refactor, so it should not merge as-is.

## What This Changes

The PR adds sandbox-only Python/Node research import-template validation to exec preflight, with focused tests and sandboxing docs.

## Best Possible Solution

Rebase onto current main, integrate sandbox-only import validation into the current exec preflight path without regressing the existing fail-closed parser or host-exec policy behavior, replace the line-by-line Node scanner with parser-backed or multiline-aware import extraction, handle Python bare-relative denied-name imports, and add focused regression tests for sandbox-vs-host behavior plus the reported bypasses.

## Reproduction Assessment

Yes. The PR body gives a concrete sandboxed `node bad.js` / `python bad.py` path for the intended hardening, and the parser bypasses can be reproduced from PR-head source by using multiline Node imports or whitespace before `require` / `import` parentheses.

## Solution Assessment

No. The requested direction is plausible, but this patch is not the best solution yet because a regex line scanner misses standard JavaScript syntax and the stale branch must be reconciled with current-main preflight behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Parse multiline Node import declarations:** `src/agents/sandbox-execution-templates.ts:436-471`
  - body: The validator scans one trimmed line at a time, so `import {\n  execSync\n} from "node:child_process"` never sends the denied specifier to `assertNodeImportAllowed`. Since multiline imports and re-exports are valid JavaScript, sandboxed research scripts can bypass the new Node template preflight.
  - confidence: 0.95
- **[P1] Handle whitespace before require/import parentheses:** `src/agents/sandbox-execution-templates.ts:473-484`
  - body: JavaScript permits `require ("node:child_process")` and `import ("node:child_process")`, but these regexes only recognize `require(` / `import(`. Those calls are neither validated as literal imports nor rejected as dynamic calls, so denied runtime modules can pass sandbox preflight.
  - confidence: 0.93
- **[P2] Check bare relative Python imported names:** `src/agents/sandbox-execution-templates.ts:287-289`
  - body: For `from . import subprocess`, the source specifier becomes empty after stripping dots and returns before checking the imported name. A sandbox script can therefore hide a denied-looking local wrapper behind a bare relative import, contrary to the template's stated denial policy.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff is command-execution sandbox hardening, but the proposed parser has concrete bypasses that would weaken the documented security guarantee if merged unchanged.
