## Summary

Keep this PR open. Current main implements the basic Feishu tool precedence change for matching Feishu account IDs, but it still does not implement the PR discussion’s final channel-origin guard: non-Feishu sessions can pass a colliding `agentAccountId` that wins over `channels.feishu.defaultAccount`. The earlier duplicate-close reports cover the basic precedence, not this stricter remaining behavior.

## Best Possible Solution

Keep this PR open for maintainer review, rebase, or a narrow replacement patch. The desired behavior is explicit tool `accountId` first; normalized inherited account only when `ctx.messageChannel` is Feishu and the account maps to an enabled Feishu account; then configured `channels.feishu.defaultAccount`; then normal Feishu default selection. Thread `messageChannel` through wiki, drive, perm, bitable, doc/media/scopes account resolution and add tests for Feishu-origin routing, non-Feishu collision fallback, unknown-account fallback, normalized account IDs, explicit override, and configured default fallback.

## Risks / Open Questions

- Closing this PR as implemented would leave the cross-channel collision case unresolved: a non-Feishu session whose `agentAccountId` matches a configured Feishu account can still override `channels.feishu.defaultAccount`.

- The branch is older than current main and should be rebased or replaced with a narrow patch that reconciles the channel-origin guard with current fallback and SecretRef behavior.
