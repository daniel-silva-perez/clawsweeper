## Summary

Keep open. The issue comments correctly identify a recurring live-docs-versus-release drift class: the original `isolatedSession` mismatch is fixed in the latest release, but current main now documents and accepts `heartbeat.skipWhenBusy` while `v2026.4.27` does not. The docs publish workflow mirrors main docs to the public docs repo, so released users can still follow live docs into a strict-schema unrecognized-key failure.

## What This Changes

Keep live docs in step with released OpenClaw versions, or clearly signal feature availability, so released users do not copy unsupported heartbeat config keys.

## Best Possible Solution

Define an explicit compatibility policy for `docs.openclaw.ai`, then implement the matching publish workflow, site metadata, generated config-doc compatibility checks, and/or version-aware validation errors so users can tell whether a documented config key is available in their installed OpenClaw version. Treat #71301 as complementary bundled-docs work rather than a complete replacement for the public live-docs policy gap.

## Reproduction Assessment

Yes. The original `isolatedSession` mismatch is fixed in `v2026.4.27`, but the same failure class reproduces by comparing current live-docs content for `heartbeat.skipWhenBusy` with the strict `v2026.4.27` heartbeat schema that does not accept that key.

## Solution Assessment

Unclear for implementation because there is no proposed patch and the right approach is a policy choice. A validation-only error would reduce confusion, but it would not fully solve live docs advertising unreleased config without a docs/versioning policy.
