## Summary

Keep open. Current main still does not carry a gateway runtime version through the `openclaw gateway status` service-status JSON/text path, and the issue is already linked to open PR #73225 with closing syntax, so the right cleanup is to review and land or close that PR rather than close this issue early.

## What This Changes

Expose the running Gateway version in `openclaw gateway status --json` and the related status surface so post-update checks can verify which build is serving traffic.

## Best Possible Solution

Land a reviewed fix that propagates the Gateway runtime version through the daemon probe, status gatherer, JSON/text printer, and docs, with tests for both handshake metadata and `status.runtimeVersion` fallback. After that PR merges, this issue can close as implemented.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: current main’s `gateway status` gather/print/probe path does not define or return a version field, and `--require-rpc` discards the `status` RPC payload that already contains `runtimeVersion`.

## Solution Assessment

Yes for the direction, but not yet merged. The narrow maintainable solution is to reuse existing handshake metadata and `status.runtimeVersion` fallback in the existing status pipeline, which is exactly the path open PR #73225 appears to target.
