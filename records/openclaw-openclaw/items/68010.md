## Summary

Keep this PR open. Current main still eagerly resolves the compatibility provider env-var export, so the lazy-proxy production change is not obsolete; however, the branch needs review fixes because its regression test does not actually prove laziness and its provider env-var hardening is security-sensitive and incomplete as written.

## What This Changes

The PR aliases `PROVIDER_ENV_API_KEY_CANDIDATES` to the lazy provider auth env-var proxy, adds a compat lazy-loading regression test, and adds provider env-var candidate hardening in `src/secrets/provider-env-vars.ts`.

## Best Possible Solution

Land a narrow compat-export fix that delegates to the existing lazy proxy with a post-import, pre-property-access regression test. Treat the provider env-var trust, denylist, defensive-copy, and cache-boundary changes as either a clearly reviewed security hardening patch with targeted tests or a separate follow-up.

## Reproduction Assessment

Yes. The current-main reproduction is static and high-confidence: importing `src/agents/model-auth-env-vars.ts` evaluates the top-level const at line 39, which calls the resolver path that loads plugin manifest metadata; the PR test defect is visible because the not-called assertion runs before the dynamic import is awaited.

## Solution Assessment

No, not as-is. Delegating the compatibility export to `PROVIDER_AUTH_ENV_VAR_CANDIDATES` is the narrow maintainable production fix, but the PR should not merge until the regression test is corrected and the security-sensitive provider env-var policy changes are split or explicitly approved with coverage.

Full review comments:

- **[P2] Assert laziness after importing the module:** `src/agents/model-auth-env-vars.test.ts:39-42`
  - body: The not-called assertion runs before `await modPromise`, so the module has not necessarily evaluated yet. With the old eager export, the assertion can pass, then the registry is called during `await modPromise`, and the final positive assertion still passes; move the not-called check after the import resolves and before accessing `PROVIDER_ENV_API_KEY_CANDIDATES`.

  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff touches environment-secret discovery and partially changes provider env-var hardening, but unresolved trust and cache-mutation concerns remain.

Concerns:

- **[high] Workspace plugin env vars remain trusted by default:** `src/secrets/provider-env-vars.ts:59`
  - body: The PR adds regex and denylist filtering, but the default `shouldUsePluginProviderEnvVars` behavior still includes workspace plugin `providerAuthEnvVars` unless callers pass `includeUntrustedWorkspacePlugins: false`; runtime `resolveEnvApiKey` builds lookup params without that flag, so untrusted workspace metadata can still influence which host env vars are read as provider credentials.
  - confidence: 0.8
- **[medium] Descriptor trap still exposes cached arrays:** `src/secrets/provider-env-vars.ts:203`
  - body: The PR copies arrays in the proxy `get` trap, but `getOwnPropertyDescriptor` still returns the original cached array as `value`, leaving a concrete mutation bypass for shared provider env-var candidates.
  - confidence: 0.86
- **[medium] Parameterless global lazy cache needs trust-boundary review:** `src/secrets/provider-env-vars.ts:224`
  - body: Aliasing the compat export to the parameterless global lazy proxy means first access can cache candidate maps derived from ambient workspace/config state; maintainer review should decide whether this cache is safe across workspaces or needs parameterized/reset behavior.
  - confidence: 0.68
