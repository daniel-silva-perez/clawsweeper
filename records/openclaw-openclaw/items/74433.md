## Summary

Keep open. Current main still lacks the requested durable env-token diagnostic, and this PR is not merge-ready because it detects broad process-env/config co-presence rather than the stale state-dir `.env` shadowing case, leaves the linked status surfaces without remediation, and does not cover the new audit path.

## What This Changes

The PR adds a doctor warning, a `gateway.env_token_overrides_config` security-audit finding, and doctor-security tests for cases where `OPENCLAW_GATEWAY_TOKEN` and `gateway.auth.token` are both configured.

## Best Possible Solution

Land a shared, redacted local-gateway auth-shadow diagnostic that detects a non-empty `OPENCLAW_GATEWAY_TOKEN` from the trusted state-dir `.env` shadowing a distinct configured `gateway.auth.token`, surfaces it consistently in doctor, status, gateway status, and deep security audit, and leaves credential precedence unchanged.

## Reproduction Assessment

Yes. A source-level reproduction is clear: configure a local gateway token plus a different state-dir `OPENCLAW_GATEWAY_TOKEN`, then compare local client credential selection with gateway server auth; a second reproduction for the PR bug is an env-template or env SecretRef config that points at `OPENCLAW_GATEWAY_TOKEN` and should not be reported as two competing sources.

## Solution Assessment

No. A warning is the right product direction, but this implementation is too broad and incomplete; the safer fix is a shared diagnostic scoped to durable state-dir env shadowing a distinct local config token, reused by the affected status and audit surfaces.
