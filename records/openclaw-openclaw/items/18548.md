## Summary

Keep open. Current main still drops the middle of oversized bootstrap files by deterministic prefix/suffix slicing, so the central relevance-aware bootstrap request remains unresolved. The compaction and observation-extraction parts have adjacent plugin surfaces, but the issue still bundles separate architecture decisions that need maintainer scoping rather than cleanup closure.

## What This Changes

The issue requests relevance-aware bootstrap file selection, incremental/non-bulk compaction behavior, and lower-overhead observation extraction from existing LLM responses.

## Best Possible Solution

Split the work into narrower trackers: implement relevance- or structure-aware bootstrap selection around `buildBootstrapContextFiles` and embedded bootstrap injection, keep incremental compaction aligned with compaction-provider/context-engine contracts, and leave observation extraction plugin-owned unless a specific missing SDK hook is identified.

## Reproduction Assessment

Yes for the central bootstrap behavior: current main shows oversized files flowing through `buildBootstrapContextFiles` into fixed prefix/suffix slices, and the unit test asserts that shape. The compaction and observation parts are architecture proposals rather than one reproducible bug.

## Solution Assessment

No, keeping this as one bundled implementation item is not the best way to solve it. The maintainable path is to split bootstrap relevance from compaction-provider/context-engine work and from observation-hook/plugin questions.
