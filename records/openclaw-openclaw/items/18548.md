## Summary

Keep open. Current main still trims oversized bootstrap files by deterministic prefix and suffix slices, so the central bootstrap relevance request remains unresolved. The compaction and observation-extraction portions now have adjacent plugin surfaces, but the issue still needs maintainer scoping because it bundles three separate architecture directions.

## What This Changes

The issue requests relevance-aware bootstrap file selection, less fragile incremental compaction, and observation extraction from existing LLM responses.

## Best Possible Solution

Split this into narrower tracks: implement relevance- or structure-aware bootstrap selection around `buildBootstrapContextFiles` and embedded bootstrap injection; keep incremental compaction aligned with the compaction-provider and context-engine contracts; and leave observation extraction plugin-owned unless maintainers identify a concrete missing SDK hook.

## Reproduction Assessment

Yes for the central bootstrap behavior: current main shows oversized files flowing through `buildBootstrapContextFiles` into fixed prefix/suffix slices, and the unit test asserts that shape. The compaction and observation parts are broader architecture proposals rather than a single reproducible bug in this issue.

## Solution Assessment

No, keeping this as one bundled implementation item is not the best way to solve it. The narrow maintainable path is to split bootstrap relevance from compaction-provider/context-engine work and from observation-hook/plugin questions.
