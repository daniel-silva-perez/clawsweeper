## Summary

Keep open. Current main still handles oversized bootstrap files with fixed prefix/suffix slicing, so the central relevance-aware bootstrap request remains unresolved; the compaction and observation-extraction portions have adjacent plugin/context-engine surfaces but still need maintainer scoping rather than closure.

## What This Changes

The issue requests three architecture improvements: content-aware bootstrap selection, incremental compaction instead of bulk summarization, and plugin-driven observation extraction from existing LLM responses.

## Best Possible Solution

Split this into narrower outcomes: add a relevance- or structure-aware bootstrap selection path around buildBootstrapContextFiles and embedded bootstrap injection, keep incremental compaction aligned with the context-engine and compaction-provider contracts, and leave observation extraction plugin-owned unless maintainers identify a specific missing SDK hook.

## Reproduction Assessment

Yes for the central bootstrap behavior: current main still routes oversized bootstrap files through deterministic prefix/suffix slicing, and tests assert that shape. The compaction and observation portions are architecture proposals rather than a single reproducible defect.

## Solution Assessment

No, not as one bundled implementation item. The maintainable path is to split bootstrap relevance from compaction-provider/context-engine work and from observation-hook/plugin questions.
