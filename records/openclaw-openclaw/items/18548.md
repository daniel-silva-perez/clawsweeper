## Summary

Keep open. Current main has improved the bootstrap ratio arithmetic, but oversized bootstrap files are still selected by deterministic prefix/suffix slicing with the middle dropped by position, so the central relevance-aware bootstrap request is unresolved. The compaction and observation-extraction parts have adjacent plugin/context-engine surfaces, but the issue bundles multiple architecture decisions and should be split or prioritized by maintainers.

## What This Changes

The issue requests three architecture improvements: relevance-aware bootstrap file injection, less fragile incremental compaction, and observation extraction from existing LLM responses.

## Best Possible Solution

Split this into narrower outcomes: add a relevance- or structure-aware bootstrap selection path around buildBootstrapContextFiles and embedded bootstrap injection, keep incremental compaction aligned with the context-engine and compaction-provider contracts, and leave observation extraction plugin-owned unless maintainers identify a specific missing SDK hook.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the central bootstrap behavior: current main still routes oversized bootstrap files through deterministic prefix/suffix slicing, and tests assert that shape. The compaction and observation portions are architecture proposals rather than a single reproducible defect.

## Solution Assessment

Is this the best way to solve the issue? No, not as one bundled implementation item. The maintainable path is to split bootstrap relevance from compaction-provider/context-engine work and from observation-hook/plugin questions.
