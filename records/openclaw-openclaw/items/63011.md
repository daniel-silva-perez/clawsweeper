## Summary

Keep open. Current main still leaves the requested outbound send contract unresolved: gateway `send` still turns hook-cancelled empty delivery into `No delivery result`, `message_sending` still receives only channel/account/conversation context, and hook cancellation still produces no structured blocked/auditable result. The issue is also paired with same-author draft PR #63007, so it should remain open for maintainer review rather than be closed or sent to a competing repair lane.

## What This Changes

Request one canonical outbound `message.send` path that runs the loaded runtime's `message_sending` hook, returns real delivery results, surfaces guard denials, and preserves agent/session/request context for audit correlation.

## Best Possible Solution

Ship a maintainer-approved canonical outbound-send contract that threads agent/session/request identity into `message_sending`, returns explicit guarded-denial outcomes on cancellation, preserves provider delivery results on success, and either aligns direct CLI, gateway `send`, and plugin-action paths or documents one supported canonical path.

## Reproduction Assessment

Yes. A source-level reproduction is clear: make a `message_sending` hook return `cancel: true`, then `deliverOutboundPayloads` skips delivery and returns no result, causing gateway `send` to throw `No delivery result`; the issue also provides a concrete Telegram/Tessera live sequence.

## Solution Assessment

Unclear. The open draft PR #63007 is a plausible narrow fix for making gateway `send` a canonical guarded path, but maintainers still need to decide whether the issue requires aligning direct CLI/plugin-action sends too.
