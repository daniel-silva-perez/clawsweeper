## Summary

Keep open. Current main still has split outbound semantics: gateway `send` treats hook-cancelled empty delivery as `No delivery result`, `message_sending` receives only channel/account/conversation context, cancellation does not produce a structured blocked/auditable result, and Telegram CLI sends can still route through gateway `message.action` plugin dispatch rather than the shared delivery contract. The same-author draft PR #63007 remains open and targets the central gateway identity/guard-denial gap, so this issue should not be closed before that PR is reviewed, landed, or replaced.

## What This Changes

The issue requests one canonical outbound `message.send` path that runs through `message_sending`, returns delivery or explicit guard-denial outcomes, and preserves agent/session/request identity for audit correlation.

## Best Possible Solution

Review and either land #63007 or replace it with a maintainer-approved canonical outbound-send contract that threads agent/session/request identity into `message_sending`, returns explicit guarded-denial outcomes on hook cancellation, preserves real provider delivery results, and clearly aligns or documents the direct CLI, gateway `send`, and plugin-action paths.
