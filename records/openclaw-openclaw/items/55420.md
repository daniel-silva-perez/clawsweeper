## Summary

Keep open. Current main still does not implement `tools.sessions.visibility="allowlist"` or a companion `tools.sessions.allowlist`; the runtime visibility type, shared guard, strict config schema, generated/typed config surface, docs, and tests all still model only `self`, `tree`, `agent`, and `all`. The related #48785 is adjacent but not a canonical duplicate because it asks for per-capability visibility knobs rather than session-key pattern allowlisting.

## Best Possible Solution

Keep this issue open and implement it in the shared session-tools visibility layer. Add `allowlist` to the runtime/type/schema/docs/generated config surfaces, add a strict `tools.sessions.allowlist: string[]`, match normalized canonical session keys in the shared guard used by list/history/send/status, and add focused tests for allowed and denied same-agent and cross-agent keys, sandbox clamping, sessionId resolution, overbroad patterns, and `tools.agentToAgent` interaction.

## Risks / Open Questions

- This is a core session-isolation change; an implementation should match against canonical resolved session keys before list/history/send/status access is granted.
- Cross-agent access should remain gated by `tools.agentToAgent` unless maintainers intentionally broaden that contract.
- Sandbox clamping needs explicit behavior for `allowlist`, especially whether sandboxed sessions are forced back to `tree` or may use a narrowed allowlist.
