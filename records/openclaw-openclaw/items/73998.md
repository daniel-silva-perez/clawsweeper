## Summary

Keep open. Current main still lacks the PR's intended Feishu WebSocket lifecycle handling, and the proposed patch is directionally right but has a blocking gap: its ClientConfig guard does not cover the missing/non-OK endpoint response path that underlies the PingInterval/system-busy report, so the SDK can still stay in its default infinite reconnect loop without surfacing monitor-level recovery.

## What This Changes

This PR updates Feishu WebSocket construction to use a guarded Lark SDK httpInstance and lifecycle hooks, rewrites monitor retry handling around SDK ready/error/reconnect callbacks, adds focused Feishu tests, and adds a changelog entry.

## Best Possible Solution

Land a narrow Feishu-owned repair that keeps the SDK lifecycle/httpInstance composition approach, avoids global prototype mutation, and explicitly handles missing or invalid endpoint ClientConfig/non-OK startup responses with a finite, test-backed recovery policy that reaches monitor retry without stacking a second backoff while the SDK reconnect loop is active.

## Reproduction Assessment

Yes. A focused code-level reproduction is to stub the WS endpoint response with missing ClientConfig or an invalid ReconnectCount and observe that the PR leaves the response/default reconnect count in a state where the SDK reads ClientConfig.PingInterval or reconnects indefinitely instead of surfacing monitor onError.

## Solution Assessment

No. The current PR is the right boundary and API direction, but it is not yet the best fix because the guard only repairs an existing ClientConfig object and the invalid reconnect-count fallback is infinite; the safer fix is to make the malformed endpoint path deterministically reach Feishu's monitor recovery and cover it in tests.
