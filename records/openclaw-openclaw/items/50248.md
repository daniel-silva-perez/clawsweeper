## Summary

Keep open. Current main and the latest release still use one resolved transcript path for `sessions cleanup --fix-missing`, so stale persisted `sessionFile` metadata can still cause valid entries to be pruned when the canonical `<sessionId>.jsonl` transcript exists. PR #69954 is open and explicitly tied to this issue, so the issue should remain open until that or an equivalent fix lands.

## What This Changes

Request that `sessions cleanup --fix-missing` avoid pruning fresh or valid cron/session entries when persisted `sessionFile` metadata is stale and a canonical `<sessionId>.jsonl` transcript exists.

## Best Possible Solution

Review and land PR #69954 if its behavior is acceptable, or replace it with a focused cleanup fix that checks the canonical `<sessionId>.jsonl` candidate before pruning, repairs stale `sessionFile` metadata when that canonical file exists, preserves true-missing cleanup, and leaves broader doctor/orphan archival behavior to separate tracking such as #73471.
