## Summary

Keep open. Current main still lets `sessions cleanup --fix-missing` prune from one resolved `sessionFile` path, so stale metadata can still remove a valid session whose canonical transcript exists; open PR #69954 directly targets this issue and should be reviewed or replaced before the issue is closed.

## What This Changes

The issue reports that `openclaw sessions cleanup --fix-missing` can prune fresh cron or heartbeat sessions when persisted `sessionFile` metadata points at a missing stale transcript even though `<sessionId>.jsonl` exists.

## Best Possible Solution

Land PR #69954 if its behavior is acceptable, or replace it with a focused cleanup fix that checks the canonical `<sessionId>.jsonl` candidate before pruning, repairs stale generated `sessionFile` metadata when the canonical transcript exists, preserves valid custom transcript paths and true-missing pruning, and adds regression coverage for stale-path, custom-path, and both-missing cases.
