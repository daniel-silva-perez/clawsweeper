## Summary

Keep open. Current main still treats a missing persisted `sessionFile` as authoritative in `sessions cleanup --fix-missing`, while the safer transcript candidate resolver can already find the canonical `<sessionId>.jsonl` path. PR #69954 is open and explicitly closes this issue, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue reports that `sessions cleanup --fix-missing` can prune valid fresh cron or heartbeat sessions when persisted `sessionFile` metadata is stale but the canonical `<sessionId>.jsonl` transcript exists.

## Best Possible Solution

Review and land PR #69954 if its fallback and repair behavior is acceptable; otherwise replace it with a focused cleanup fix that checks the canonical `<sessionId>.jsonl` transcript before pruning, repairs stale `sessionFile` metadata when that canonical file exists, preserves true-missing pruning, and leaves broader doctor/orphan archival behavior to separate follow-up work.
