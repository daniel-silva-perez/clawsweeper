## Summary

Keep open. Current main still has the cleanup safety bug: `--fix-missing` trusts one resolved `sessionFile` path and can prune the store entry when that stale path is missing, even though other transcript readers already know to try the canonical session-id transcript. The linked PR #69954 remains open and directly targets this issue, so the right next step is maintainer review of that implementation or an equivalent replacement.

## What This Changes

`sessions cleanup --fix-missing` should not prune a session entry solely because its persisted `sessionFile` is stale when a valid canonical `<sessionId>.jsonl` transcript exists.

## Best Possible Solution

Land PR #69954 if its behavior and accounting are acceptable, or replace it with a focused cleanup fix that checks the canonical `<sessionId>.jsonl` candidate before pruning, repairs stale generated `sessionFile` metadata when that canonical transcript exists, preserves custom transcript paths and true-missing pruning, and adds regression tests for stale stored path, valid custom path, and both-missing cases.
