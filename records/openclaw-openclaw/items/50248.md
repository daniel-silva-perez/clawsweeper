## Summary

Keep open. Current main still uses one resolved transcript path for `sessions cleanup --fix-missing`, so a stale persisted `sessionFile` can still make cleanup prune an entry even when the canonical `<sessionId>.jsonl` exists. The linked PR #69954 is open and directly targets this issue, so the issue should remain open until that or an equivalent fix lands.

## What This Changes

The issue asks for `sessions cleanup --fix-missing` to stop pruning fresh or otherwise valid cron/heartbeat sessions when stored `sessionFile` metadata is stale but a valid transcript exists under the current session id.

## Best Possible Solution

Land PR #69954 if its behavior is acceptable, or replace it with a focused cleanup fix that checks the canonical `<sessionId>.jsonl` candidate before pruning, repairs stale `sessionFile` metadata when that canonical file exists, preserves true-missing pruning, and leaves broader doctor/orphan archive semantics to their own tracked follow-up.
