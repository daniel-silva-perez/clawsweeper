## Summary

Keep open. Current main still lets `sessions cleanup --fix-missing` prune from one resolved `sessionFile` path, so stale metadata can still hide a valid canonical `<sessionId>.jsonl`; PR #69954 is open and explicitly targets this issue, so this issue should remain open until that or an equivalent fix lands.

## What This Changes

The issue asks OpenClaw to stop `sessions cleanup --fix-missing` from pruning fresh cron or heartbeat sessions when `sessions.json` has a stale `sessionFile` but the canonical transcript for the current `sessionId` still exists.

## Best Possible Solution

Review and land PR #69954 if its behavior is acceptable, or replace it with a focused cleanup fix that checks the canonical `<sessionId>.jsonl` candidate before pruning, repairs stale `sessionFile` metadata when that canonical file exists, preserves true-missing pruning, and leaves broader doctor/orphan archive semantics to separate follow-up work.
