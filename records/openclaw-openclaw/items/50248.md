## Summary

Keep open. Current main still checks a single resolved `sessionFile` path for `sessions cleanup --fix-missing`, so stale metadata can still cause valid session entries to be pruned when the canonical `<sessionId>.jsonl` transcript exists; the open linked PR #69954 directly targets that remaining bug.

## What This Changes

The issue requests a safer `sessions cleanup --fix-missing` path so fresh cron/run sessions with stale `sessionFile` metadata are not pruned when their canonical transcript file still exists.

## Best Possible Solution

Review and land PR #69954 if its behavior is acceptable, or replace it with an equivalent focused cleanup fix that checks the canonical `<sessionId>.jsonl` candidate before pruning, repairs stale `sessionFile` metadata when that canonical transcript exists, and leaves broader doctor/orphan archive semantics to separate follow-up work.
