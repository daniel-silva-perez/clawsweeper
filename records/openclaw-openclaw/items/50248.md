## Summary

Keep open. Current main and the latest release still let `sessions cleanup --fix-missing` trust one resolved `sessionFile` path, so stale persisted metadata can still prune entries whose canonical `<sessionId>.jsonl` transcript exists. PR #69954 is open with closing syntax for this issue, so the issue should remain open until that or an equivalent fix lands.

## What This Changes

The issue reports that `sessions cleanup --fix-missing` can prune fresh cron, heartbeat, or run sessions when persisted `sessionFile` metadata points at a stale missing file while the canonical `<sessionId>.jsonl` transcript exists.

## Best Possible Solution

Review and land PR #69954 if its behavior and accounting are acceptable, or replace it with a focused cleanup fix that checks the canonical `<sessionId>.jsonl` candidate before pruning, repairs stale generated `sessionFile` metadata when that canonical file exists, preserves true-missing cleanup, and leaves broader doctor/orphan archival behavior to separate tracking.

## Reproduction Assessment

Yes. A deterministic fixture can create a store entry with `sessionId=<id>`, a stale missing `sessionFile`, and an existing canonical `<id>.jsonl`; current main's source shows `--fix-missing` resolves only the stale path and would prune the entry.

## Solution Assessment

Yes. A narrow cleanup fallback/repair is the best path because producer-side rollover mitigations do not repair existing stale metadata, while broad doctor/orphan archival semantics are a separate issue.
