## Summary

Keep open. The unsafe cleanup behavior still exists on current main and in the latest release, while PR #69954 is open with closing syntax for this issue and should be reviewed or replaced before the issue is closed.

## What This Changes

The issue reports that `sessions cleanup --fix-missing` can prune valid fresh cron/run sessions when stored `sessionFile` metadata points to a stale missing transcript while the canonical `<sessionId>.jsonl` exists.

## Best Possible Solution

Review and land PR #69954 if its behavior is acceptable, or replace it with an equivalent focused cleanup fix that checks the canonical `<sessionId>.jsonl` candidate before pruning, repairs stale `sessionFile` metadata when that canonical file exists, and preserves true-missing cleanup behavior.

## Reproduction Assessment

Yes. A deterministic fixture can create a store entry with `sessionId=<id>`, a stale missing `sessionFile`, and an existing canonical `<id>.jsonl`; current main's cleanup source shows that `--fix-missing` would still prune from the stale path only.

## Solution Assessment

Yes. A narrow cleanup fallback/repair is the best fix for this issue because producer-side rollover changes do not repair existing stale metadata, while broader doctor/orphan archival semantics should remain separate.
