## Summary

Current main still does not implement this PR's intended Discord steer-mode bypass. The current Discord plugin always sends inbound jobs through the per-key worker queue before reply queue logic resolves `steer` or `steer-backlog`, and this PR targets the removed `src/discord` layout rather than the current `extensions/discord` implementation.

## Best Possible Solution

Keep this PR open for maintainer follow-up. The best path is to rebase it onto `extensions/discord`, carry enough session context into the inbound worker to resolve the effective queue mode, bypass the outer per-session `KeyedAsyncQueue` only for `steer` and `steer-backlog`, preserve serialization for non-steer modes, and add regression tests for steer, steer-backlog, session override precedence, replay dedupe, timeout fallback, deactivation, and run-state accounting.

## Risks / Open Questions

- The PR is stale against the current plugin layout and needs a rebase from `src/discord` to `extensions/discord`.
- A config-only bypass would still mishandle session-level `/queue` overrides, including cases where global Discord mode is steer but the active session asks for serialized behavior.
- Bypassing the outer worker can affect replay dedupe, timeout fallback delivery, deactivation, and busy/run-state accounting, so the rebased change needs focused regression coverage.
