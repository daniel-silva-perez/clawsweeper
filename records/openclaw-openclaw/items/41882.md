## Summary

Keep this PR open. The PR targets a real remaining Discord queue bug: current main still serializes every Discord inbound job through the outer per-session KeyedAsyncQueue before the reply queue layer can apply steer or steer-backlog behavior. The PR discussion is useful, including the follow-up that added steer-backlog, but the patch is stale against the current plugin path and still needs a rebased implementation under extensions/discord with session override handling considered.

## Best Possible Solution

Keep this PR open for maintainer follow-up or replace it with a rebased implementation under extensions/discord. The best fix is to resolve the effective Discord queue mode early enough to bypass the outer per-session KeyedAsyncQueue for steer and steer-backlog, while preserving serialization for collect, followup, interrupt, and other non-steer modes and honoring session override precedence where available. Add regression coverage for steer, steer-backlog, non-steer serialization, session override precedence, replay dedupe, timeout fallback, deactivation, and busy/run-state accounting.
