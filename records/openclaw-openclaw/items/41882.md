## Summary

Keep this PR open. Current main still serializes Discord inbound jobs through the bundled Discord plugin's outer per-session KeyedAsyncQueue before the core steer/steer-backlog logic can run. The PR remains a plausible fix direction, but it targets the removed src/discord layout and still needs a current extensions/discord implementation that resolves the effective queue mode safely, including session overrides.

## Best Possible Solution

Keep this PR open for maintainer follow-up. The best path is to rebase the fix onto extensions/discord, bypass the outer per-session KeyedAsyncQueue only when the effective queue mode is steer or steer-backlog, preserve serialization for collect/followup/interrupt, honor session-level queue overrides, and add focused regression coverage for steer, steer-backlog, non-steer serialization, deactivation, replay dedupe, timeout fallback, and run-state status accounting.

## Risks / Open Questions

- The PR cannot merge mechanically because it edits the removed src/discord layout instead of the current extensions/discord plugin files.
- A config-only bypass can mis-honor per-session /queue overrides, including cases where global Discord config is steer but the session was switched to collect, followup, or interrupt.
- Bypassing the outer worker changes same-session concurrency around replay dedupe, timeout fallback delivery, run-state status accounting, and deactivation.
- A rebased implementation must avoid importing core auto-reply internals from bundled plugin production code unless a plugin-SDK-safe queue settings seam is added.
