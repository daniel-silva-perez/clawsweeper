## Summary

Current main does not implement this PR's intended Discord steer-mode bypass. The Discord plugin still enqueues every inbound job through the per-key outer `KeyedAsyncQueue` before downstream reply queue logic can apply `steer` or `steer-backlog`, while the PR itself targets the old removed `src/discord` layout and remains non-mergeable without a rebase onto `extensions/discord`.

## Best Possible Solution

Keep this PR open for maintainer follow-up rather than closing it. The best path is to rebase the change onto `extensions/discord`, compute the effective queue mode with session overrides included, bypass the outer per-session `KeyedAsyncQueue` only for `steer` and `steer-backlog`, preserve serialization for non-steer modes, and add focused regression tests around steer, steer-backlog, non-steer serialization, deactivation, replay dedupe, timeout fallback, and run-state status accounting.

## Risks / Open Questions

- Current PR is stale against the codebase layout and needs rebasing from `src/discord` to `extensions/discord`.
- A config-only bypass would still mishandle session-level `/queue` overrides, including cases where global Discord mode is steer but a session override asks for serialized collect/followup/interrupt behavior.
- Bypassing the outer worker can affect replay dedupe, timeout fallback delivery, run-state accounting, and deactivation behavior; these need focused regression coverage in the current Discord plugin implementation.
