## Summary

Keep this PR open. Current main still serializes Discord inbound jobs by session/channel queue key before the downstream reply queue logic can apply `steer` or `steer-backlog`, so the intended behavior is not implemented on main. The PR is also stale against the current plugin layout because it targets removed `src/discord` paths instead of `extensions/discord`.

## Best Possible Solution

Keep this PR open for maintainer follow-up or replace it with a rebased change against `extensions/discord`. The best fix is to resolve the effective queue mode before the Discord inbound worker queue, bypass the outer per-session `KeyedAsyncQueue` only for `steer` and `steer-backlog`, preserve serialization for non-steer modes, and add regression tests for steer, steer-backlog, session override precedence, replay dedupe, timeout fallback, deactivation, and run-state accounting.
