## Summary

Keep open. Current main still matches the reported failure mode: the gateway pins the startup HTTP-route registry, later plugin registry updates do not update a pinned route surface, and an open unmerged PR already targets this exact bug with closing syntax.

## What This Changes

The issue asks for workspace plugin `api.registerHttpRoute(...)` routes to dispatch to their handlers instead of returning 404 after successful registration.

## Best Possible Solution

Keep this issue open while maintainers review and land PR #73877 or an equivalent narrow fix that makes dynamically loaded workspace plugin HTTP routes visible to the gateway dispatcher without regressing startup plugin routes or channel registry stability.

## Reproduction Assessment

Yes. The issue gives a concrete workspace-plugin Docker curl reproduction, and current-main source inspection shows the matching route-registry pin/update path that can leave later registered routes outside the dispatcher surface.

## Solution Assessment

Yes at the product boundary: the fix belongs in the gateway/plugin runtime registry path so dispatch resolves the registry that actually owns workspace plugin HTTP routes. The exact patch should still get maintainer review because existing tests intentionally cover pinned-route behavior.
