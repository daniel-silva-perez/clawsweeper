## Summary

Keep open. Current main still has the registry-pin path that can leave HTTP routes registered through `api.registerHttpRoute(...)` in a later plugin registry invisible to the gateway dispatcher, and open PR #73877 already targets this issue with closing syntax, so this issue should stay open until that PR or an equivalent fix lands.

## What This Changes

Workspace plugin HTTP routes registered with `api.registerHttpRoute(...)` should be reachable through the gateway dispatcher instead of returning 404 after successful plugin registration.

## Best Possible Solution

Keep this issue open until maintainers review and land #73877 or an equivalent narrow fix that makes workspace/dynamically loaded plugin HTTP routes visible to the gateway dispatcher while preserving the channel registry protection that prevents unrelated plugin-registry churn from breaking channel webhooks.

## Reproduction Assessment

Yes. The issue provides concrete Docker/workspace-plugin curl steps, and current-main source inspection shows a matching stale-registry path when a route is registered into a later plugin registry after the startup HTTP-route registry has been pinned.

## Solution Assessment

Yes at the product boundary. The narrow maintainable fix belongs in the gateway/plugin runtime registry path so dispatch resolves the registry containing workspace plugin routes; #73877 is the active maintainer-review object for that direction.
