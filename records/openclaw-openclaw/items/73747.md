## Summary

Keep open. This is a fresh, concrete Docker/workspace-plugin route-dispatch report against v2026.4.26. Current main still documents the route API and still has a plausible registry split where inspection can show a route while the HTTP dispatcher resolves a pinned or different route registry; I found no route-dispatch fix after the previous ClawSweeper pass.

## Best Possible Solution

Reproduce on current main in Docker with a minimal config-loaded workspace plugin under the default agent workspace that registers /plugins/webhooks/gmail-push, then verify both plugins inspect route count and a real POST handler invocation. If it reproduces, update Gateway startup/runtime wiring so the HTTP dispatcher resolves the same registry that loaded the workspace plugin, including deferred reload paths, and add an integration regression for the inspect-count plus POST path. If it does not reproduce, document the container restart, effective config, Gateway URL, auth mode, and route checks needed to diagnose stale deployment state.
