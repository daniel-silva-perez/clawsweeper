## Summary

Keep open. Current main still has the stale HTTP-route registry path: the gateway pins the startup route registry, later plugin registry activation cannot update that pinned surface, and the dispatcher can therefore miss routes registered into the later workspace/deferred plugin registry. Open PR #73877 already targets the same bug with closing syntax, so this should stay open until that PR or an equivalent fix lands.

## What This Changes

Workspace plugin HTTP routes registered with `api.registerHttpRoute(...)` should dispatch through the gateway instead of returning 404 after successful plugin load and route registration.

## Best Possible Solution

Make the gateway/plugin runtime route surface resolve the registry that owns workspace/deferred HTTP routes, with regression coverage for the reported dispatch path, while preserving the separate channel registry protection.

## Reproduction Assessment

Yes. The issue supplies concrete Docker/workspace-plugin curl steps, and current-main source inspection shows the matching stale-registry path when routes are registered into a later plugin registry after the startup HTTP-route registry is pinned.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the gateway/plugin runtime registry surface rather than in documentation or plugin workarounds; #73877 is the active implementation candidate for that direction.
