## What This Changes

The issue reports that workspace plugin HTTP routes register and appear in inspection output, but gateway HTTP dispatch returns 404 instead of invoking the registered handler.

## Best Possible Solution

Keep this issue open while maintainers review #73877 or an equivalent narrow fix that makes dynamically loaded workspace plugin HTTP routes visible to the gateway dispatcher without regressing route stability for startup/plugin-owned routes.

## Reproduction Assessment

Yes. The issue gives a concrete Docker/workspace-plugin curl reproduction, and current-main source inspection shows a matching code path where runtime state is created before deferred plugin reload while the HTTP-route surface remains pinned.

## Solution Assessment

Yes, at the product boundary: the fix should live in the gateway/plugin runtime registry path so dispatch resolves the registry that actually contains workspace plugin routes. The exact PR implementation still needs maintainer review because current tests intentionally cover pinned-route behavior.
