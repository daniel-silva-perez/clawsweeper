## Summary

Keep open. This is a fresh, concrete Docker/workspace-plugin report against v2026.4.26, and current main does not contain a clear fix proving that the reported inspect-time route registration and request-time 404 mismatch is resolved. The code still has a plausible startup/runtime registry mismatch path, and existing coverage is unit or mocked rather than the reporter's end-to-end Docker workspace-plugin route path.

## Best Possible Solution

Reproduce on current main in Docker with a minimal trusted config-loaded workspace plugin under the default agent workspace that registers `/plugins/webhooks/gmail-push`; verify both `plugins inspect` route count and an actual POST handler invocation. If it reproduces, fix the Gateway startup/runtime HTTP-route registry wiring so any registry loaded or reloaded for startup plugins is the one pinned or resolved by the dispatcher, then add an integration regression for the exact workspace-plugin POST path. If it does not reproduce, document the container restart, config load path, and route/auth checks needed to diagnose stale inspect-vs-runtime state.
