## Summary

Keep open. Current main still contains the startup HTTP-route registry pin and the pinned-surface sync guard that can leave routes registered by a later workspace plugin registry invisible to request dispatch; open PR #73877 already targets this issue with closing syntax, so the issue should remain open until that fix or an equivalent one lands.

## What This Changes

Workspace plugin HTTP routes registered with `api.registerHttpRoute(...)` should be reachable through the gateway dispatcher instead of returning 404 after successful plugin registration.

## Best Possible Solution

Keep the issue open and resolve it through a narrow gateway/plugin-runtime fix that makes workspace and dynamically loaded plugin HTTP routes visible to dispatch while preserving the channel registry protection that avoids unrelated plugin-registry churn breaking channel webhooks.

## Reproduction Assessment

Yes. The issue provides concrete Docker/workspace-plugin curl steps, and current-main source inspection shows the same stale-registry path when a route is registered into a later plugin registry after the startup HTTP-route registry is pinned.

## Solution Assessment

Yes at the product boundary. The maintainable fix belongs in the gateway/plugin runtime registry path; #73877 is the active review object for that direction, and the issue should close only after that or an equivalent fix lands.
