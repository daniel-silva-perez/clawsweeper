## Summary

Keep open. Current main still supports plugin HTTP routes and has registry-pinning/unit coverage, but I did not find a current-main change or integration proof that resolves this fresh Docker/workspace-plugin report where inspect sees `HTTP routes: 1` while live HTTP dispatch returns 404.

## Best Possible Solution

Reproduce against current main in Docker with a minimal trusted workspace/config-path plugin that registers `/plugins/webhooks/gmail-push`; verify both `plugins inspect` and a real HTTP POST handler invocation. If it reproduces, fix the Gateway startup/runtime route-registry or startup activation wiring so the live dispatcher uses the same registered route set, then add an integration regression for that path. If it does not reproduce, document the operational distinction between inspect-time plugin loading and the live Gateway process/restart state for container deployments.
