## Summary

Keep this PR open. Current main still does not implement the PR’s intended `/bash` exec policy inheritance: `handleBashChatCommand` forwards timeout and notification defaults into `createExecTool`, but not `security`, `ask`, `pathPrepend`, safe-bin policy, `strictInlineEval`, or `approvalRunningNoticeMs`. The lower-level exec tool already supports those defaults, so this remains a focused, plausible bug fix rather than obsolete work. The submitted PR also has review-thread gaps to resolve before merge, especially forwarding `approvalRunningNoticeMs` and likely `strictInlineEval`.

## Best Possible Solution

Keep this PR open and update it, or apply an equivalent current-main patch, so `/bash` forwards the relevant `tools.exec` policy fields from `handleBashChatCommand` into `createExecTool`: `security`, `ask`, `pathPrepend`, `safeBins`, `safeBinTrustedDirs`, merged/appropriate `safeBinProfiles`, `strictInlineEval`, and `approvalRunningNoticeMs`, while preserving `commands.bash` and elevated allowlist gates. Add a focused regression test that mocks `createExecTool` and verifies the inherited defaults, then run the targeted test and normal changed validation through the OpenClaw path.

## Risks / Open Questions

- Merging the submitted PR as-is would still leave partial policy drift because the review thread correctly notes `approvalRunningNoticeMs` is not forwarded; current main also exposes `strictInlineEval` as an exec default, so the final patch should address that policy field deliberately.
- Forwarding `security` and `ask` into `/bash` can intentionally reduce prompts for operators who configured permissive exec policy. That is consistent with existing `tools.exec` semantics, but regression tests should cover permissive and approval-backed behavior so elevated gating is not bypassed accidentally.
