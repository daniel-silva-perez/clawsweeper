## Summary

Keep this PR open. Current main still does not implement the intended `/bash` exec policy inheritance: the `/bash` chat command forwards timeout and notification defaults into `createExecTool`, but not the configured exec policy fields such as `security`, `ask`, `pathPrepend`, safe-bin policy, `strictInlineEval`, or `approvalRunningNoticeMs`. The PR is a focused fix candidate, but the review thread’s missing-field concerns remain valid before merge.

## Best Possible Solution

Keep this PR open and update it, or apply an equivalent current-main patch, so `/bash` builds its exec defaults from the same effective exec policy as the normal exec tool path. The patch should forward `security`, `ask`, `pathPrepend`, `safeBins`, `safeBinTrustedDirs`, merged `safeBinProfiles`, `strictInlineEval`, `approvalRunningNoticeMs`, timeout, and notification defaults while preserving `commands.bash` and elevated allowlist gates. Add a focused regression test that mocks `createExecTool` and verifies the inherited defaults, including the fields called out in review.

## Risks / Open Questions

- Merging the submitted PR as-is would still leave partial policy drift because the review thread correctly notes `approvalRunningNoticeMs` is not forwarded, and current main also has `strictInlineEval` as an exec default that the PR does not cover.
- If the final patch only forwards global `tools.exec`, per-agent exec policy may still diverge from the normal exec path, which already merges per-agent and global policy.
- Forwarding `security=full` and `ask=off` into `/bash` intentionally reduces approval prompts for operators who configured that policy, so regression coverage should also confirm that `commands.bash` and `tools.elevated` gating remain required.
