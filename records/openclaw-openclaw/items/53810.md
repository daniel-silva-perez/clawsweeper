## Summary

Keep open. Current origin/main still only handles Qwen/ModelStudio thinking as a request payload patch and only creates OpenAI-compatible tool calls from structured `choice.delta.tool_calls`; tagged marker sections embedded in reasoning/thinking output are not promoted. The affected Qwen catalog models remain bundled and documented, and a child session that reaches `done` still maps to an ok subagent completion, so the reported premature-success path remains plausible.

## What This Changes

The issue asks OpenClaw to normalize provider/model-specific tagged tool-call markup emitted inside thinking/reasoning output so subagents do not announce success before executing those tools.

## Best Possible Solution

Add provider-owned Qwen/ModelStudio response normalization, or a narrowly gated shared OpenAI-compatible helper reused by Qwen and Kimi, that extracts only validated standalone tagged tool-call sections from text/thinking/reasoning content, emits structured `toolCall` blocks, and sets `stopReason` to `toolUse` only after successful extraction.
