## Summary

Keep open. Current main has a partial mitigation for visible reply leaks, but it does not fully implement the requested per-agent auto-reply suppression or a shared final/public-vs-internal delivery boundary, and the broader related audience-boundary issue remains open.

## What This Changes

The issue requests preventing assistant text from being automatically sent to channels when tool calls follow, either through a per-agent `autoReply=false` directive or by sending only final assistant text after tool calls complete.

## Best Possible Solution

Define a shared visible-delivery contract that distinguishes public/final assistant text from internal/commentary text before channel dispatch, then wire it through provider stream parsing, transcript helpers, source-reply policy, Feishu/generic dispatch, config docs, and regression tests while preserving explicit `message` tool sends.

## Reproduction Assessment

Partly. Current source and tests give high-confidence proof of the residual policy gap: group/channel turns are suppressed by default, but direct chats remain automatic; the reporter's exact Feishu text-plus-tool sequence still needs a focused fixture or live transcript reproduction.

## Solution Assessment

No for the suggested per-agent flag alone. A narrow `autoReply=false` control would help, but the maintainable fix should cover final-only delivery and public/internal audience semantics at the shared extraction and channel-dispatch boundaries.
