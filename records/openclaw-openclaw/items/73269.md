## Summary

Keep open. Current main has a partial mitigation through source visible-reply modes, but direct chats still default to automatic delivery and there is no per-agent auto-reply disablement or explicit public/internal assistant audience boundary; the related broader audience-boundary report is still open.

## What This Changes

The issue requests suppressing automatic channel delivery of assistant text when the same response also performs tool calls, either via a per-agent `autoReply=false` directive or by delivering only final intended assistant text.

## Best Possible Solution

Define a shared visible-delivery or assistant-audience contract that covers final, block, preview, partial, and source-reply delivery across direct and group/channel turns, while preserving explicit `message(action=send)` as the visible-send path and documenting the config/schema behavior.

## Reproduction Assessment

Partly. Current source and unit tests give a high-confidence reproduction of the residual policy gap: direct chats remain automatic while message-tool-only turns suppress automatic delivery; the exact Feishu text-then-tool-call transcript still needs a focused synthetic or live fixture.

## Solution Assessment

No. The existing `messages.visibleReplies` mechanism is a useful partial mitigation, but the maintainable fix should be a reviewed delivery/audience boundary rather than only a per-agent `autoReply` flag.
