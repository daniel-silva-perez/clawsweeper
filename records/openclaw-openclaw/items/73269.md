## Summary

Keep open. Current main has a partial mitigation for this class of leaks: group/channel source turns default to message-tool-only visible delivery, operators can opt direct chats into the same behavior globally, and Feishu suppresses generic block chunks. It does not fully implement the requested per-agent auto-reply disablement or an explicit public/internal or final-only delivery boundary, and the related broader audience-boundary report remains open.

## What This Changes

The issue requests suppressing automatic channel delivery of intermediate assistant text when an assistant response also includes later tool calls, with Feishu given as an example surface.

## Best Possible Solution

Unify visible delivery around a shared source-reply or assistant-audience contract that can suppress automatic final, block, preview, and partial delivery per intended scope while still allowing explicit message-tool sends. That should cover direct and group/channel conversations, Feishu streaming cards, generic channel dispatch, transcript extraction, docs, config schema/help, and regression tests.

## Reproduction Assessment

Partly. The current unit tests and source give a high-confidence reproduction of the residual policy gap: direct chats still default to automatic delivery, while group/channel turns are suppressed; the exact reporter sequence of assistant text followed by tool calls would still need a focused synthetic or live Feishu transcript fixture.

## Solution Assessment

No. The current messages.visibleReplies mechanism is a useful workaround and partial fix, but it is not the best complete solution for this issue because it lacks the requested per-agent/directive-level control and does not create an explicit public/internal or final-only assistant text boundary.
