## Summary

Keep open. Current main has a useful mitigation through `messages.visibleReplies` and group/channel `message_tool` defaults, but it still does not provide the requested per-agent auto-reply disablement, a final-only delivery mode, or the broader public/internal visible-delivery boundary tracked by related #74928.

## What This Changes

The issue asks OpenClaw to stop automatically sending assistant text to channels when the same assistant turn continues into tool calls, either via an `autoReply=false` directive or by delaying visible delivery until final text is known.

## Best Possible Solution

Define one shared visible-delivery contract that can mark assistant text as public or internal and suppress automatic preview, block, partial, and final delivery unless policy allows it, while preserving explicit `message(action=send)` output. If maintainers prefer a narrower interim, add an explicit per-agent/source delivery override that maps into the same shared dispatch policy instead of relying on skill prompt rules.

## Reproduction Assessment

Partly. Current source and unit tests provide a high-confidence reproduction of the residual policy gap: direct/source chats default to automatic visible delivery while `message_tool_only` suppresses automatic delivery; the exact Feishu text-then-tool-call leak still needs a focused fixture or live transcript-free check.

## Solution Assessment

No. The current `messages.visibleReplies` mechanism is a useful mitigation, but it is not the full requested solution because it lacks per-agent control and does not establish a public/internal or final-only assistant text boundary across delivery surfaces.
