## Summary

Keep open. Current main has a useful partial mitigation for group/channel rooms, but the reported delivery boundary is not fully solved: direct/source chats still default to automatic final delivery, there is no per-agent auto-reply directive, and the related public/internal assistant-audience gap remains open.

## What This Changes

The issue requests suppressing automatic channel delivery of assistant text when tool calls follow, either with a per-agent auto-reply disablement or by delivering only final user-facing assistant text.

## Best Possible Solution

Define a shared visible-delivery contract that distinguishes internal, preview/block, and final public assistant text before channel dispatch. It should cover direct and group/channel conversations, Feishu streaming cards, generic dispatch, transcript visible-text helpers, config schema/docs, and explicit message-tool sends without relying on skill prompt workarounds.

## Reproduction Assessment

Unclear. Current source and unit tests give high-confidence proof of the residual policy gap, but the exact reporter sequence in Feishu would still need a focused synthetic or live fixture with assistant text followed by tool calls.

## Solution Assessment

No. The current messages.visibleReplies mechanism is a useful mitigation, but it is not the complete fix because it lacks the requested per-agent control and does not establish an explicit public/internal or final-only assistant text boundary.
