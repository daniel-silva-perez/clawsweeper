## Summary

Keep open. Current main has a shipped visible-reply mitigation, but it is only a partial answer: group/channel turns default to message-tool-only delivery and direct chats can opt in globally, while direct/source chats still default to automatic delivery and there is no per-agent `directives.autoReply` or shared public/internal audience contract. The broader related #74928 remains open and is likely the right place for the shared boundary decision.

## What This Changes

The issue requests suppressing automatic channel delivery for assistant text when tool calls follow, either through per-agent auto-reply control or final-only/public-delivery semantics.

## Best Possible Solution

Define one shared visible-delivery contract that can distinguish public assistant text from internal/runtime text and apply it consistently across preview, block, tool-progress, and final delivery. If maintainers choose #74928 as the canonical tracker, preserve this issue’s Feishu/direct-chat reproduction context there before closing this as superseded.

## Reproduction Assessment

Partly. Current source and tests give a high-confidence reproduction of the residual policy gap: direct chats default to automatic delivery while message-tool-only mode suppresses automatic callbacks and finals; the exact Feishu text-then-tool-call leak still needs a focused fixture or live check.

## Solution Assessment

No. The shipped `messages.visibleReplies` mitigation is useful, but it is not the complete requested solution because it is global/direct opt-in, not per-agent, and it does not add an explicit public/internal or final-only assistant text boundary.
