## Summary

Keep open. Current main has a useful partial mitigation for group/channel rooms and a global direct-chat workaround, but it still does not implement the requested per-agent auto-reply disablement or a shared final-only/public-vs-internal delivery boundary; the related broader audience-boundary report remains open.

## What This Changes

The issue requests suppressing automatic user-visible channel delivery of assistant text when later tool calls follow, either through `agents.<id>.directives.autoReply=false` or by sending only final assistant text after tool calls complete.

## Best Possible Solution

Define one shared visible-delivery contract that can distinguish public assistant text from internal/runtime narration and suppress automatic preview, block, partial, and final channel delivery unless policy allows it, while preserving explicit `message(action=send)` output. That should cover direct and group/channel conversations, Feishu streaming cards, generic dispatch, transcript extraction, config schema/help, docs, and regression tests.

## Reproduction Assessment

Partly. Current source and unit tests give a high-confidence reproduction of the residual policy gap: direct/source chats still default to automatic delivery while message-tool-only mode suppresses visible delivery; the exact Feishu text-before-tool-call transcript still needs a focused fixture or live check.

## Solution Assessment

No. The current `messages.visibleReplies` and `messages.groupChat.visibleReplies` behavior is a useful mitigation, but it is not the complete requested fix because it lacks per-agent/directive control and an explicit public/internal or final-only delivery boundary.
