## Summary

Keep open. Current main still lacks the intended explicit active-agent routing surface, but this PR should not merge as-is because the latest diff can still turn fallback-derived policy state into an authoritative outbound message route, and its changelog entry violates the repository attribution rule.

## What This Changes

The PR threads an `agentId` option through OpenClaw coding/tool construction into the message tool, changes tool-policy session fallback resolution, adds focused routing/default-agent tests, and adds a changelog entry.

## Best Possible Solution

Land a rebased routing fix that uses a separate trusted active/message-routing agent id only from runtime call sites that truly know it, keeps session/default fallback resolution config-aware at message execution time, adds coverage for config-less opaque-session default-agent behavior, and credits the changelog entry.

## Reproduction Assessment

Yes. A high-confidence static/unit reproduction is to construct coding tools without config for an opaque session key while the loaded runtime config has a non-main default agent, then execute a message send and verify the PR can make the fallback id authoritative before the message route is resolved from config.

## Solution Assessment

No. The direction is valid, but this implementation conflates trusted explicit routing with policy/session fallback inference; the narrower maintainable fix is a distinct active-agent route input plus fallback coverage at the message action boundary.
