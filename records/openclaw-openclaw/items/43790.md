## Summary

Keep open. Current main still lacks the intended explicit message-tool agent routing, but this PR is not ready to merge because it forwards a policy-derived fallback as an authoritative agent id when config is unavailable, which can bypass the message tool's later config-aware routing; the branch is also dirty and its changelog entry is missing required credit.

## What This Changes

The PR threads an `agentId` option from `createOpenClawCodingTools` through `createOpenClawTools` into `createMessageTool`, changes tool-policy session fallback resolution, adds focused routing tests, and adds a changelog bullet.

## Best Possible Solution

Land a rebased routing fix that keeps trusted runtime active-agent state separate from policy/session fallback, only passes an explicit message-tool agent id when the caller truly knows the active embedded agent, preserves config-aware fallback inside the message action path, adds regression coverage for the no-config opaque-session default-agent case, and credits the changelog entry correctly.

## Reproduction Assessment

Yes. A high-confidence static/unit reproduction is to build coding tools without config for an opaque session key while persisted config has a non-main default agent; the PR can compute `main` before config is loaded and then make that value authoritative for outbound message routing.

## Solution Assessment

No. The direction is right, but this diff conflates a trusted explicit agent id with an inferred policy fallback; the safer solution is to add a separate explicit active-agent input and leave inferred fallback resolution inside the config-aware message action path.
