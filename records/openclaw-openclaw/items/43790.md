## Summary

Keep open. Current main still does not expose an explicit `agentId` path into `createMessageTool`, so the intended bug fix is not already implemented; however, the PR branch still has two merge-blocking issues: it can turn an inferred fallback agent into an authoritative message route, and its changelog entry lacks the required attribution.

## What This Changes

The PR threads an explicit active agent id through OpenClaw tool construction into the message tool so message sends prefer that id over session-key-derived routing.

## Best Possible Solution

Land a guarded version of the tool-layer routing fix: pass a trusted explicit agent id only when the runtime caller genuinely supplied one, preserve lazy config-aware fallback for inferred or opaque session keys, cover both precedence and fallback behavior in focused tests, and add the required changelog credit.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve lazy config fallback for opaque session keys:** `src/agents/pi-tools.ts:491`
  - body: Passing `agentId` here makes the value from `resolveEffectiveToolPolicy` authoritative. When `options.config` is absent and the session key is legacy or opaque, that value can be the fallback `main`; `createMessageTool` then prefers it over the config-aware fallback it would compute after loading config, so sends can be attributed to the wrong default agent. Forward only trusted explicit agent ids, or preserve provenance so inferred fallbacks stay lazy-config aware.
  - confidence: 0.9
- **[P2] Credit the changelog entry:** `CHANGELOG.md:454`
  - body: The added changelog bullet is missing the required `Thanks @...` attribution. Repository policy requires every added changelog entry to credit at least one GitHub username, so this needs to be fixed before landing.
  - confidence: 0.95
