## Summary

Keep this PR open. Current main still does not implement the central behavior: the message tool cannot receive a trusted explicit active agent id, so tool-driven sends still route through the session-key-derived agent id. The lower-level gateway send path already supports explicit-agent precedence, which reinforces that the remaining gap is tool-layer propagation and focused regression coverage.

## Best Possible Solution

Keep this PR open and review or rebase a guarded patch: add a trusted agentId option to createOpenClawTools/createMessageTool, pass the active embedded agent from createOpenClawCodingTools, prefer that explicit value only when it is genuinely supplied by runtime code, preserve config-aware and lazy-config fallback for opaque/non-agent session keys, add focused tests for explicit override plus fallback behavior, and fix the changelog attribution before landing.

## Risks / Open Questions

- Wrong-agent attribution for message-tool sends in shared-session or delegated embedded flows remains possible on current main when the active embedded agent differs from the session-key-derived agent.
- Before landing, the PR should avoid treating a fallback main computed without runtime config as authoritative for opaque/non-agent session keys; that could bypass config-aware default-agent fallback after lazy config loading.
- Explicit routing affects per-agent account/session attribution, so the agentId override must stay sourced from trusted runtime context and not from user/model-provided message tool parameters.
