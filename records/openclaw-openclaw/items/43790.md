## Summary

Keep open. Current main still lacks the requested explicit message-tool routing, but the PR branch should not merge as-is because it can turn a policy/session fallback into an authoritative outbound route and its changelog entry misses the required attribution.

## What This Changes

The PR threads an agentId option through OpenClaw coding/tool construction into the message tool, changes effective tool-policy session fallback resolution, adds focused routing tests, and adds a changelog bullet.

## Best Possible Solution

Land a rebased routing fix that separates trusted active-agent state from inferred session/policy fallback: pass an explicit message-routing agent id only when the runtime truly knows it, leave config-aware fallback inside the message action path, cover the no-config opaque-session default-agent case, and credit the changelog entry correctly.

## Reproduction Assessment

Yes. A high-confidence static/unit reproduction is to construct coding/message tools without config for an opaque session key while runtime config has a non-main default agent; the PR can precompute main and make it authoritative before the message tool's lazy config-aware fallback runs.

## Solution Assessment

No. The direction is valid, but this implementation conflates trusted explicit routing with fallback inference; the safer alternative is a distinct active-agent input passed only from runtime call sites that know the active embedded agent.
