## Summary

Keep open. Current main still lacks the explicit active-agent path into the message tool, so the underlying wrong-agent routing problem remains real; however, the PR branch still needs changes before merge because it can make an inferred fallback agent authoritative in config-unavailable embedded runs and its changelog entry is missing required credit.

## What This Changes

The PR threads an explicit agentId through OpenClaw tool construction into createMessageTool, prefers it over session-derived routing, adjusts tool-policy session fallback behavior, adds focused routing tests, and adds a changelog entry.

## Best Possible Solution

Land a guarded version of the tool-layer routing fix: only pass a trusted explicit agent id when the runtime caller genuinely supplied one, preserve config-aware and lazy-config fallback for inferred/opaque session keys, cover both override and fallback cases in focused tests, and add the required changelog credit.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve lazy config fallback for opaque session keys:** `src/agents/pi-tools.ts:491`
  - body: Passing `agentId` here makes the value from `resolveEffectiveToolPolicy` authoritative. When `options.config` is absent and the session key is opaque, that value can be the fallback `main`, so `createMessageTool` will bypass its lazy config-aware default-agent fallback and route sends under the wrong agent. Only forward an agent id when it was explicitly supplied by trusted runtime context, or preserve provenance so inferred fallbacks remain lazy-config aware.
  - confidence: 0.9
- **[P2] Credit the changelog entry:** `CHANGELOG.md:454`
  - body: This added changelog bullet is missing the required `Thanks @...` attribution. The repository changelog policy requires every added entry to include contributor credit, so this needs to be fixed before landing.
  - confidence: 0.95

## Security Review

Status: needs_attention

Summary: Needs attention because the PR's explicit agent forwarding can make an inferred fallback agent authoritative in config-unavailable embedded runs; no supply-chain changes were found.

Concerns:

- **[medium] Fallback agent can become an authoritative route:** `src/agents/pi-tools.ts:491`
  - body: The added `agentId` forwarding can turn an inferred `main` fallback into the route used by message sends when runtime config was not available yet. For users with a non-main default agent, that can bind or attribute outbound messages to the wrong agent account/session, which changes per-agent routing scope even though the value is from runtime context rather than model input.
