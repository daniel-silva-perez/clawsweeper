## Summary

Keep open. Current main still lacks the explicit message-tool agent routing this PR is trying to add, so the contribution is relevant, but the current PR head is dirty and still has blocking review issues in a routing-sensitive outbound message path: a policy-derived fallback agent can become an authoritative message route, and the changelog entry is missing required attribution.

## What This Changes

The PR adds an explicit `agentId` through OpenClaw tool and message-tool construction, changes tool-policy session fallback resolution, adds focused routing tests, and adds a changelog bullet.

## Best Possible Solution

Land a rebased routing fix that passes only trusted runtime-supplied active agent ids to the message tool, preserves execution-time config-aware fallback for inferred or opaque sessions, adds focused tests for both explicit precedence and no-config opaque-session default-agent behavior, and credits the changelog entry correctly.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to build coding tools with no config and an opaque session key while persisted config has a non-main default agent; the PR computes a fallback `main` before config is loaded and then makes that value authoritative for the message tool.

## Solution Assessment

No. The direction is right, but this diff conflates an explicit active agent id with a policy-derived fallback; the safer fix is to keep those values separate and only override message-tool session resolution when the caller supplied trusted runtime agent state.
