## Summary

Keep open. Current main still lacks the public-facing agent mode this PR is trying to add, but the branch is not ready to merge because it still misses default-only public-mode enforcement and has a streamed OpenResponses trusted-proxy mismatch; it also needs maintainer reconciliation with the newer compat HTTP operator-scope contract.

## What This Changes

The branch adds `publicMode` agent/default config, prompt/runtime redaction, skills snapshot suppression, `/v1` owner-trust header handling, docs, and targeted tests for OpenAI-compatible chat completions and OpenResponses.

## Best Possible Solution

Rebase this PR on current main, integrate public-facing agent trust with the existing compat HTTP operator-scope model or explicitly justify a separate owner header, then finish the implementation so per-agent and default-only `publicMode` consistently redacts prompt/runtime details, suppresses skills snapshots, and applies identical sender-owner behavior across streaming and non-streaming `/v1/chat/completions` and `/v1/responses`.

## Reproduction Assessment

Yes. The default-only bug reproduces by setting `agents.defaults.publicMode: true` without an `agents.list` entry and reading the PR's `resolveAgentPublicMode()` path; the streamed OpenResponses mismatch reproduces by following the PR diff where the streamed `runResponsesAgentCommand()` call omits `trustedProxies`.

## Solution Assessment

No, not as currently proposed. The feature direction is useful, but the branch needs the two concrete fixes above and a rebase/design pass against the current `x-openclaw-scopes` owner semantics before it is the narrowest maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Honor defaults when no agent entry exists:** `src/agents/agent-scope.ts:153-154`
  - body: `resolveAgentPublicMode()` delegates to `resolveAgentConfig()`, which returns `undefined` before reading `agents.defaults` when the requested agent is not listed. In a default-only config, `agents.defaults.publicMode: true` therefore evaluates to false and the prompt/ingress protections are skipped.
  - confidence: 0.9
- **[P2] Pass trusted proxies to streamed responses:** `src/gateway/openresponses-http.ts:755`
  - body: The streamed `/v1/responses` path calls `runResponsesAgentCommand()` with `publicMode` but without `trustedProxies`. Since `x-openclaw-sender-is-owner: true` only works when the remote address matches `trustedProxies`, trusted upstreams are treated as non-owner only in streaming mode.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The PR touches owner/non-owner gateway semantics and still has concrete trust-boundary gaps that should be fixed before merge.

Concerns:

- **[medium] Default-only publicMode is not enforced:** `src/agents/agent-scope.ts:153`
  - body: Operators can reasonably set `agents.defaults.publicMode: true` without listing the default agent, but the PR resolves public mode through `resolveAgentConfig()` and returns false when no entry exists. That can leave runtime and workspace metadata exposed despite the public-mode default.
  - confidence: 0.9
- **[medium] Streamed responses ignore trusted-proxy owner assertions:** `src/gateway/openresponses-http.ts:755`
  - body: The streamed OpenResponses call omits `trustedProxies`, so the new owner header cannot be honored for trusted upstreams in that path. This creates a security-boundary inconsistency between stream and non-stream requests.
  - confidence: 0.86
- **[medium] Owner header needs reconciliation with current scope model:** `src/gateway/http-utils.ts:24`
  - body: Current main derives compat HTTP owner semantics from shared-secret auth and `x-openclaw-scopes` / `operator.admin`. Adding a second owner header without a clear relationship to that contract risks confusing deployments and docs for the gateway trust boundary.
  - confidence: 0.78
