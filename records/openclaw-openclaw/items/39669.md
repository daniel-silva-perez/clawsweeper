## Summary

Keep open. Current main still has the reported busy-followup shape: the active-run queue branch enqueues the message, cleans up typing, and returns without a generic queued-feedback callback or a Feishu/Lark receipt path; the related implementation work is either open and unmerged or closed unmerged.

## What This Changes

The issue asks for immediate visible acknowledgement, such as a reaction, typing signal, or short receipt, when a user message is queued behind an active agent run.

## Best Possible Solution

Add a generic, non-blocking queued-followup feedback seam at the active-run enqueue-followup boundary, then wire Feishu/Lark and any channel without existing initial status reactions to an immediate queued receipt or reaction. The hook should fire only after a followup is actually enqueued, swallow/log notification failures, avoid duplicate status signals on channels that already set initial queued reactions, and include focused queue/Feishu regression coverage.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is available: with an active run and a steer/followup queue mode, resolveActiveRunQueueAction selects enqueue-followup, and runReplyAgent enqueues then returns undefined without a queued-feedback callback; existing tests already assert the undefined return/enqueue behavior.

## Solution Assessment

Yes, the best direction is a narrow generic queued-receipt callback or typed hook rather than a Feishu-only special case. #39117 may be the active vehicle, but it should be reviewed for this exact behavior and kept non-blocking so queue reliability is preserved.
