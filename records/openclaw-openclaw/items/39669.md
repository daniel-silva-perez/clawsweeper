## Summary

Keep this issue open. Current main still has no generic queued-followup feedback seam for the busy `enqueue-followup` path: the run is enqueued, typing is cleaned up, and the call returns without a callback, receipt payload, queued reaction, or channel-facing hook. Feishu still wires feedback through reply-start typing callbacks only. Related implementation work exists, but it is not merged: #39117 is open and #39676 is closed unmerged.

## Best Possible Solution

Keep this issue open. The best fix is a generic, non-blocking queued-followup feedback seam in the active-run `enqueue-followup` path, such as an `onQueued` callback or typed receipt hook on `GetReplyOptions`/the shared reply dispatcher. It should fire only when a followup is actually enqueued, swallow/log callback failures so queue reliability is preserved, wire Feishu/Lark to an immediate reaction or receipt, avoid duplicate signals on channels that already set status reactions, and add focused tests for busy queued followups. #39117 and the closed #39676 are useful prior implementation context, but neither is shipped on current main.
