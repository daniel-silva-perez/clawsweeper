## Summary

Keep this issue open. Current main still matches the reported OpenShell name-collision bug: the bundled OpenShell plugin depends on unscoped `openshell@0.1.0`, the lockfile resolves that package to a Telegram bridge bot with `telegraf`, and the resolver still prefers the bundled package over the documented PATH CLI when `command` is the default `openshell`.

## Best Possible Solution

Remove the unscoped `openshell@0.1.0` runtime dependency and update the resolver so the default `command: "openshell"` uses the PATH CLI. Adjust the package-manifest contract and OpenShell CLI helper tests to cover the reporter's case; only support a packaged OpenShell CLI later if it is a verified/scoped dependency and documented explicitly.

## Risks / Open Questions

- Users who enable the OpenShell sandbox backend with the default `command: "openshell"` can still hit the unrelated Telegram package before the standalone OpenShell CLI on PATH, breaking sandboxed exec.
- A fix should avoid adding another unverified package-resolution assumption; any bundled CLI package should be scoped/verified, or the default path should use the documented standalone CLI.
