## Summary

Keep this issue open. Current main still carries the reported name-collision shape: the OpenShell bundled plugin declares the unscoped `openshell@0.1.0` runtime dependency, the lockfile resolves that package with Telegram-related dependencies, and the CLI resolver intentionally prefers a resolved bundled `openshell` binary over the PATH command when the configured command is the default `openshell`. The docs still describe the intended dependency as an installed OpenShell CLI on PATH, so the reporter’s root-cause comment remains relevant rather than obsolete.

## Best Possible Solution

Remove the unscoped `openshell@0.1.0` bundled runtime dependency unless a verified OpenShell CLI package exists, change the default resolver so `command: "openshell"` uses the PATH CLI, and update the OpenShell tests to cover the reporter’s scenario. If a packaged CLI is later supported, it should be scoped/verified and documented explicitly.

## Risks / Open Questions

- Leaving this unresolved can keep breaking sandboxed exec for users who enable the OpenShell backend with the default `command: "openshell"` and have the bundled runtime dependency installed.
- A fix should avoid replacing one unverified package-resolution assumption with another; the OpenShell backend should prefer the documented standalone CLI or validate any bundled package identity before using it.
