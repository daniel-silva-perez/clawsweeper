## Summary

Keep this issue open. Current main still contains the reported OpenShell npm name-collision path: the bundled OpenShell plugin depends on unscoped `openshell@0.1.0`, the resolver still prefers that bundled binary over the PATH CLI for the default `command: "openshell"`, and the upstream npm package is a Telegram bridge that throws when `TELEGRAM_BOT_TOKEN` is missing.

## Best Possible Solution

Remove the unscoped `openshell@0.1.0` runtime dependency from the bundled OpenShell plugin, update `resolveOpenShellCommand` so the default `command: "openshell"` uses the PATH CLI, and update the package-manifest contract plus OpenShell CLI helper tests to cover the reporter's fallback case. Only add packaged OpenShell CLI support later if there is a verified scoped dependency or explicit documented package source.

## Risks / Open Questions

- Users who enable the OpenShell sandbox backend with the documented default command can still execute the unrelated Telegram bridge package before the standalone OpenShell CLI on PATH, causing sandboxed exec to fail when `TELEGRAM_BOT_TOKEN` is not set.
- Leaving the unscoped dependency in the bundled plugin keeps packaging and contract tests aligned with the wrong package, so future releases may continue reinstalling the bad module unless the dependency and tests are changed together.
