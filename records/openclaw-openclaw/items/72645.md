## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Fix the skill attrs type before typechecking:** `extensions/diagnostics-otel/src/service.ts:2226-2228`
  - body: `skillExecutionBaseAttrs` is typed as `Record<string, string | number | boolean>`, but this `attrs` object is declared as `Record<string, string | number>` while spreading it. TypeScript can reject the possible boolean values here; the same pattern appears in `recordSkillExecutionError` below, so the branch needs the attr type narrowed or widened before merge.
  - confidence: 0.87
- **[P1] Keep session identifiers out of exported run attrs:** `extensions/diagnostics-otel/src/service.ts:1166-1170`
  - body: This adds raw `sessionKey` and `sessionId` to `addRunAttrs`, which is reused by exported spans sent to an external OTLP collector. Current OTel docs promise spans never include session keys, so these identifiers should be dropped, hashed, or explicitly handled through a reviewed privacy contract before being exported.
  - confidence: 0.9
- **[P1] Remove unbounded IDs from metric attribute sets:** `extensions/diagnostics-otel/src/service.ts:1407-1412`
  - body: `messageProcessingBaseAttrs` is reused for `messageProcessingCounter.add` and duration histogram recording, so every run/message/chat/session value becomes an OTel metric label. That creates unbounded time-series cardinality in Prometheus-style backends; keep these IDs on spans/log correlation only or replace them with bounded attributes for metrics.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff introduces concrete privacy and collector-cardinality concerns by exporting raw session identifiers and per-message IDs through OTLP attributes.

Concerns:

- **[medium] Raw session identifiers exported to OTLP:** `extensions/diagnostics-otel/src/service.ts:1166`
  - body: The PR adds `openclaw.session_key` and `openclaw.session_id` to shared span attribute helpers even though current public OTel docs say spans never include session keys. External collectors may retain those identifiers outside the OpenClaw trust boundary.
  - confidence: 0.9
- **[medium] Unbounded IDs added as metric labels:** `extensions/diagnostics-otel/src/service.ts:1407`
  - body: The new message-processing metric attrs include run, message, chat, and session identifiers. These can create one time series per message/session and can overload or greatly increase retention cost in Prometheus-compatible collectors.
  - confidence: 0.86
- **[low] No new third-party supply-chain source found:** `extensions/qa-lab/package.json:8`
  - body: The qa-lab dependency added by the PR is a workspace link to `@openclaw/qa-channel`, and the lockfile hunk shown in the current diff does not introduce a new external package source. The concern is scope, not a concrete supply-chain regression.
  - confidence: 0.78
