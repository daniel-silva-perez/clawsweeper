## Summary

Keep open. Current main does not implement the proposed message/skill lifecycle telemetry, and the open PR still needs maintainer review because the latest head has concrete type, privacy, and metric-cardinality blockers before it can merge safely.

## What This Changes

The PR adds `message.processing.*` and `skill.execution.*` diagnostics events, emits them from the embedded runner, exports corresponding diagnostics-otel spans/metrics, adds skill observability docs, and adjusts qa-lab/test dependency surfaces.

## Best Possible Solution

Keep this PR open for maintainer review, then revise it into a narrower telemetry contract that keeps raw session/message/chat identifiers out of exported metric labels and external span attrs unless a reviewed hashing/redaction policy is added, fixes the TypeScript attr typing, aligns docs with the existing OTel/Prometheus privacy model, and adds focused producer/exporter/stability tests for the final event contract.

Full review comments:

- **[P1] Fix the skill attrs type before typechecking:** `extensions/diagnostics-otel/src/service.ts:2226-2227`
  - body: `skillExecutionBaseAttrs` can return boolean-valued attrs, but this object is declared as `Record<string, string | number>` while spreading it. TypeScript can reject the same pattern here and in `recordSkillExecutionError`, so the branch needs the base type narrowed or the local attrs widened before merge.
  - confidence: 0.87
- **[P1] Keep session identifiers out of shared run attrs:** `extensions/diagnostics-otel/src/service.ts:1166-1170`
  - body: This adds raw `sessionKey` and `sessionId` to `addRunAttrs`, which is reused by exported OTLP spans. Current OTel docs promise spans never include session keys, so these identifiers should be removed, hashed, or handled through a reviewed privacy contract before export.
  - confidence: 0.91
- **[P1] Remove raw IDs from message metric attributes:** `extensions/diagnostics-otel/src/service.ts:1407-1412`
  - body: `messageProcessingBaseAttrs` includes run, message, chat, session key, and session id values, then the same attrs are passed to the counter and histogram. That creates unbounded metric labels in Prometheus-style backends; keep these values off metrics or replace them with bounded dimensions.
  - confidence: 0.89
- **[P3] Add docs metadata and navigation:** `docs/diagnostics/otel-skill-observability.md:1`
  - body: The new diagnostics doc starts directly with a heading and is not added to the docs navigation. `docs:list` relies on front matter metadata, and users will not discover this page from the existing OTel docs unless it is either folded into `gateway/opentelemetry` or given front matter plus a docs nav entry.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff introduces concrete privacy and collector-cardinality concerns by exporting raw session/message identifiers through OTLP attributes and metrics.

Concerns:

- **[medium] Raw session identifiers exported to OTLP:** `extensions/diagnostics-otel/src/service.ts:1166`
  - body: The PR adds `openclaw.session_key` and `openclaw.session_id` to shared span attrs even though current public OTel docs say spans never include session keys; external collectors may retain those identifiers outside the OpenClaw trust boundary.
  - confidence: 0.91
- **[medium] Unbounded identifiers used as metric labels:** `extensions/diagnostics-otel/src/service.ts:1407`
  - body: The new message-processing metric attrs include run, message, chat, session key, and session id values. These can create one time series per message/session and can overload or greatly increase retention cost in Prometheus-compatible collectors.
  - confidence: 0.89
