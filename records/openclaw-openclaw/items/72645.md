## Summary

Keep open. Current main does not implement the requested per-message and skill lifecycle telemetry, so this PR is not obsolete, but the branch still needs maintainer review before merge because it exports raw session identifiers and unbounded per-message identifiers through OTLP telemetry paths and adds an unwired docs page.

## What This Changes

The PR adds `message.processing.*` and `skill.execution.*` diagnostic events from the embedded runner, exports matching diagnostics-otel spans and metrics, adds a skill observability guide, and adjusts qa-lab dependency/test surfaces.

## Best Possible Solution

Land a narrowed telemetry contract that keeps high-cardinality identifiers on spans/log correlation only where privacy policy allows, keeps metrics bounded, documents the final attribute set in the existing OpenTelemetry docs surface, and preserves focused producer/exporter/stability tests for the new events.

## Reproduction Assessment

Yes for the review findings: static inspection of PR head `6582f290e888469701069c2203bb7b899c913423` shows the raw session identifiers and unbounded IDs at the changed lines. Current-main search confirms the requested lifecycle events are not already implemented.

## Solution Assessment

No. The direction may be useful, but this is not the best mergeable form until session identifiers are removed, hashed, or made explicitly opt-in under a reviewed policy, metric labels are bounded, and the docs page is wired into the public docs structure.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

  - body: This shared helper now copies raw `sessionKey` and `sessionId` into exported OTLP span attributes. The current public OTel contract says spans never include session keys, so these identifiers should be removed, hashed, or gated behind a reviewed opt-in privacy contract before merge.
  - confidence: 0.9

That creates unbounded time-series cardinality in Prometheus-style backends; keep these IDs on spans/log correlation only or replace them with bounded attributes.
  - confidence: 0.88
- **[P2] Wire the new OTel guide into docs metadata:** `docs/diagnostics/otel-skill-observability.md:1`
  - body: The new diagnostics page starts directly with an H1 and is not added to the Mintlify navigation, so it lacks the normal front matter/read_when metadata and is not discoverable from the docs sidebar. Add the metadata and sidebar entry, or fold the content into the existing OpenTelemetry page.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: Security review needs attention because the PR exports raw session identifiers and unbounded per-message identifiers through external OTLP telemetry paths.

Concerns:

- **[medium] Raw session identifiers exported to OTLP:** `extensions/diagnostics-otel/src/service.ts:1166`
  - body: The PR adds `openclaw.session_key` and `openclaw.session_id` to a shared span attribute helper. External collectors can retain those routing identifiers outside the OpenClaw trust boundary, and current docs explicitly say session keys are not exported in spans.
  - confidence: 0.9
- **[medium] Unbounded IDs added as metric labels:** `extensions/diagnostics-otel/src/service.ts:1407`
  - body: The new message-processing metric attributes include run, message, chat, and session identifiers. These can create one time series per message/session and can overload or greatly increase retention cost in Prometheus-compatible collectors.
  - confidence: 0.88
