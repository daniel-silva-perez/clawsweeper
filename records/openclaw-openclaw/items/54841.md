## Summary

Keep open. Current main still uses the configured Feishu webhookPath for logging/status/rate-limit identity but does not compare incoming request URLs to that path before body parsing, signature validation, challenge handling, or dispatcher invocation. PR #54904 is open, unmerged, and explicitly tracks this fix, so this issue should remain open until that PR or an equivalent Feishu plugin fix lands.

## What This Changes

The issue requests Feishu webhook mode to reject requests whose URL path does not match channels.feishu.webhookPath before payload parsing or dispatch.

## Best Possible Solution

Review and land PR #54904, or an equivalent Feishu plugin patch, that normalizes the configured webhookPath, parses the incoming request pathname, returns 404 for mismatches before body parsing/signature validation/dispatch, preserves query-string acceptance on the configured path, and adds focused regression coverage.
