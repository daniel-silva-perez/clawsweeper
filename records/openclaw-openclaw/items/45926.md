## Summary

Keep this issue open. Current main still returns immediately when `sessions_send` waits with `timeoutSeconds > 0` and `agent.wait` reports `timeout`, so the A2A announce flow is not scheduled on that path. Related prior cleanup covered other announce-routing cases, but not this timeout branch.

## Best Possible Solution

Patch `src/agents/tools/sessions-send-tool.ts` so the timeout branch calls `startA2AFlow(undefined, runId)` before returning the synchronous timeout result, with a focused regression test in the `sessions_send` tool coverage that mocks `agent.wait` as timeout and verifies the announce flow starts asynchronously. Keep the existing parent-owned ACP child skip behavior intact.

## Risks / Open Questions

- A fix should preserve the current `skipA2AFlow` guard for parent-owned background ACP child sessions, because that guard prevents duplicate completion delivery and parent/child echo loops.
- A regression test should assert that timeout returns synchronously as `status: "timeout"` while still scheduling `runSessionsSendA2AFlow` with the timed-out `runId`; otherwise the bug can recur silently.
