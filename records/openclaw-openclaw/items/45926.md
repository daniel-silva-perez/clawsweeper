## Summary

Keep open. Current main still returns a hard `sessions_send` timeout before launching the A2A announce flow, and the latest release has the same behavior; open PRs #58133 and #68196 target the same timeout/delivery contract but are still unmerged and need maintainer consolidation or revision.

## What This Changes

Make `sessions_send` keep agent-to-agent announce delivery alive asynchronously when the caller's initial wait times out after the target run has started.

## Best Possible Solution

Land one focused, backward-compatible fix that starts the existing A2A flow for true wait-window expiry after a target run has been accepted, carries baseline protection into any deferred reply read, preserves the owned-ACP-child skip guard, and adds regression coverage for delayed reply, no-new-reply, second-wait-timeout, and terminal-timeout cases.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to mock `waitForAgentRunAndReadUpdatedAssistantReply` returning `timeout` after `startAgentRun`; current main then returns at `src/agents/tools/sessions-send-tool.ts:379` before any A2A announce launch.

## Solution Assessment

Yes, with guardrails. Reusing the existing delayed A2A flow is the narrow maintainable direction, but the implementation should distinguish soft wait expiry from terminal timeout and preserve the pre-run reply baseline.
