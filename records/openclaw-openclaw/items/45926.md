## Summary

Keep open: current main still has the reported timeout path. The caller-wait timeout branch returns before starting the existing A2A announce helper, while adjacent fire-and-forget and successful-wait paths do start it.

## What This Changes

Make `sessions_send` start the A2A announce flow asynchronously when `timeoutSeconds > 0` and the caller wait times out, so the target can still deliver its result later.

## Best Possible Solution

Update the `sessions_send` timeout branch to invoke the existing A2A flow with the target `runId` before returning timeout to the caller, add focused regression coverage for the timeout branch, and leave broader announce-timeout policy to the adjacent A2A timeout work.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: make `agent.wait` return `timeout` from `waitForAgentRunAndReadUpdatedAssistantReply`, and the handler returns from `src/agents/tools/sessions-send-tool.ts:379` before any `startA2AFlow` call.

## Solution Assessment

Yes. The narrowest maintainable fix is to reuse the existing `startA2AFlow(undefined, runId)` / `waitRunId` helper path in the timeout branch while preserving the ACP owned-child skip guard; changing the global announce timeout cap is a separate decision.
