## Summary

Keep open: current main still returns from the `sessions_send` wait-timeout branch before starting the existing A2A announce flow. The remaining work is valid and narrow, and open PR #58133 already proposes the matching fix with regression coverage, so this issue should not be closed or duplicated.

## What This Changes

Make `sessions_send` keep the agent-to-agent announce delivery alive asynchronously when the caller's initial wait times out.

## Best Possible Solution

The desired end state is that a caller wait timeout no longer suppresses delayed target delivery: the timeout branch should hand the in-flight `runId` to the existing A2A flow, preserve the owned-ACP-child skip guard, and cover the delayed-timeout path in focused sessions tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to make `waitForAgentRunAndReadUpdatedAssistantReply` return `timeout` after `startAgentRun`; current main returns from `src/agents/tools/sessions-send-tool.ts:379` before any A2A announce launch.

## Solution Assessment

Yes, at the implementation boundary. Reusing `startA2AFlow(undefined, runId)` in the timeout branch is the narrow maintainable fix; maintainers should review whether the returned tool status should be `timeout` or `accepted` with pending delivery, as PR #58133 proposes.
