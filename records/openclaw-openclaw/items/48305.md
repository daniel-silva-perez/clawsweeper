## Summary

Keep open. Current main already has generic restart-reason plumbing, but the config-reload path still computes restart reasons and drops them at both restart call sites. This PR targets a real remaining gap, but the branch should be refreshed against current restart infrastructure and should not merge the added stack-trace diagnostic unchanged.

## What This Changes

The PR branch adds a restart reason parameter to gateway restart emission and deferral, passes config-reload reasons from reload handlers, and logs emitted or missing restart reasons.

## Best Possible Solution

Refresh the branch onto current main and keep the code change to the config-reload caller: pass `config-reload: ${reasons}` into both `deferGatewayRestartUntilIdle({ reason })` and `emitGatewayRestart(reason)`, then add a focused reload test that observes the propagated reason through the existing restart plumbing. Leave the generic restart infrastructure intact unless a narrowly justified diagnostic is added without full stack traces.

## Reproduction Assessment

Yes. A source-level reproduction is clear: invoke the config reload restart path with `restartReasons: ["gateway.port"]` on current main and the handler computes `gateway.port` but emits or defers the restart without passing a reason.

## Solution Assessment

No, not as currently proposed. The two config-reload call-site changes are the right minimal fix, but the stale `src/infra/restart.ts` edits duplicate older plumbing and add a broad stack-trace warning that should be removed or tightly scoped.
