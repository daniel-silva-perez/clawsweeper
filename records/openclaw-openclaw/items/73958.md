## Summary

Keep this PR open. Current main still forces Feishu/Lark thread delivery from inbound `thread_id` or `root_id`, while this PR is a narrow replacement for the closed #40147 work and preserves the separate #39507/#67783 semantics boundaries; the patch looks functionally correct for its stated scope, with one low-priority cleanup around now-inert dispatcher metadata.

## What This Changes

The PR changes Feishu group session routing so only `root_id` sets `threadReply`, stops the reply dispatcher from forcing `replyInThread` from that metadata flag, updates Feishu routing/dispatcher tests, and adds a changelog entry.

## Best Possible Solution

Use this PR, or an equivalent current-main patch, as the narrow fix for root-id anchored group routing and thread-id-only config honoring. Before landing, remove or clearly justify the now-inert `threadReply` dispatcher plumbing, keep #67783 separate, and leave the stricter disabled-as-hard-off decision to the #39507 review unless maintainers choose to fold that policy in here.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: a Feishu group event with `thread_id`, no `root_id`, and `replyInThread: "disabled"` flows through `resolveFeishuGroupSession` to `replyInThread: true`, and the current test at `extensions/feishu/src/bot.test.ts:2668` asserts that forced behavior.

## Solution Assessment

Yes for this PR's stated boundary. The patch stays inside the Feishu plugin, preserves existing send helpers and API contracts, and adds focused tests; the safer refinement is to remove or document the unused `threadReply` dispatcher parameter so future routing changes do not assume it still has an effect.
