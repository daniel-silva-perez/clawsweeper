## Summary

Keep this issue open. Current main has Telegram outbound chunking/retry and a write-ahead delivery recovery queue, but it still lacks configurable Telegram notification batching, digest aggregation, grouping by type/severity/source, max-count/interval flushing, and shutdown flush behavior. The related open #69165 is complementary queue/rate-limit work, not a replacement for this digest feature.

## Best Possible Solution

Keep this issue open and design an opt-in Telegram plugin batching/digest feature with typed config, strict schema, docs, and focused tests for interval flushing, max batch size, grouping policy, shutdown flush, error bypass behavior, media handling, and interaction with the existing delivery recovery queue. Keep #69165 as the separate transport queue/rate-limit/backoff track unless maintainers intentionally merge both designs under one shared outbound delivery abstraction.
