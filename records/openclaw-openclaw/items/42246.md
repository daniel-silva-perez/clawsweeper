## Summary

Keep this issue open. Current main has Telegram retry handling, text chunking, preview streaming controls, and a generic write-ahead delivery recovery queue, but it still does not expose or implement configurable outbound Telegram batching, digest formatting, grouping, max-batch flushing, or shutdown flushing. The related open #69165 is complementary transport queue/rate-limit work, not a replacement for digest aggregation.

## What This Changes

Request an opt-in Telegram outbound notification batching and digest layer that aggregates many short Telegram messages over a configurable window before sending.

## Best Possible Solution

Add an opt-in Telegram plugin batching feature with schema, config metadata, docs, and targeted tests. It should aggregate only eligible outbound notification-style text payloads, preserve immediate delivery for critical or unsupported payloads, and coordinate with #69165 so digesting and rate limiting do not become competing queues.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. We do have a high-confidence verification path by inspecting current main: Telegram config, docs, and outbound send code have retry/chunking/recovery behavior but no digest batching surface.

## Solution Assessment

Yes for the broad direction: a Telegram-owned, opt-in batching layer is the maintainable boundary because channel plugins own outbound sending. The exact policy is not fully settled, so the safest next step is maintainer design review before an automated implementation PR.
