## Summary

Keep open. Current main still leaves the reported gateway-to-DM recall path unresolved because `experimental.sessionMemory` enables the builtin sessions source, while QMD transcript export remains a separate opt-in and session hits are still filtered by the default `tree` visibility scope.

## What This Changes

The issue requests that `memory_search` with `experimental.sessionMemory` and `sources: ["memory", "sessions"]` return gateway-dispatched session transcript content across sessions, including QMD/Gemini-backed setups.

## Best Possible Solution

Define one explicit transcript recall contract: either make `experimental.sessionMemory` plus `sources: ["sessions"]` enable and diagnose both builtin and QMD session indexing, or keep QMD opt-in separate while surfacing clear docs/status guidance for `memory.qmd.sessions.enabled: true` and an intentional visibility setting such as `tools.sessions.visibility: "agent"` for same-agent gateway-to-DM recall. Add regression coverage for a gateway-style same-agent session queried from an unrelated DM session.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: with `memory.backend: "qmd"` and only `experimental.sessionMemory: true`, QMD resolves `sessions.enabled` to false, so no session collection/export is created; with default `tools.sessions.visibility: "tree"`, unrelated same-agent session hits are filtered even if indexed.

## Solution Assessment

Unclear. The requested behavior is valid, but the narrowest maintainable fix needs maintainer judgment on whether QMD should follow the experimental session-memory flag automatically or remain an explicit transcript-export opt-in with stronger diagnostics and documentation.
