## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks the requested `plugin_updated` lifecycle hook, so the contribution is useful, but the PR broad-loads all plugin runtimes before targeted dispatch and needs a current-main refresh onto the active hook contract surface.

## What This Changes

The PR adds a typed `plugin_updated` hook, emits lifecycle events from plugin updates, dispatches them from the plugin update CLI, adds docs/tests, and includes an unrelated formatter `maxBuffer` tweak.

## Best Possible Solution

Refresh the lifecycle feature onto current main as a narrow plugin contract change: define `plugin_updated` in the active hook types, emit events only for real successful updates, persist install records through the current ledger path, load only updated plugin IDs before targeted dispatch, and document the public hook behavior.

## Reproduction Assessment

Yes. Source inspection is enough to reproduce the review finding: the PR head broad-loads `scope: "all"` before targeted dispatch, while current main exposes `onlyPluginIds` for scoped runtime loading.

## Solution Assessment

No, not as written. The feature direction is sound, but the maintainable implementation should use the current `hook-types.ts` contract and explicit updated-plugin runtime scope instead of broad registry materialization.
