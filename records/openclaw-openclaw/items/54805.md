## Summary

Keep this PR open, but it needs changes before merge. Current main still lacks the requested `plugin_updated` lifecycle hook, and the PR covers the right product gap, but the branch is stale against the current hook contract/update flow and its dispatch path broadens plugin runtime loading to all plugins before a targeted hook call.

## What This Changes

The PR adds a typed `plugin_updated` plugin hook, updater lifecycle event emission, CLI dispatch after plugin updates, docs/tests, and an unrelated Windows formatter `maxBuffer` tweak.

## Best Possible Solution

Refresh the implementation onto current main as a narrow plugin lifecycle change: add the hook contract in `src/plugins/hook-types.ts`, emit events only for real successful version changes, persist install records through the current update path, load only the updated plugin IDs before targeted dispatch, and align docs/tests/changelog with that behavior.

## Reproduction Assessment

Yes. Source inspection on current main shows `plugin_updated` is absent from the active hook contract and update command, while the PR diff shows the dispatch regression via `ensurePluginRegistryLoaded({ scope: "all" })` before targeted hook dispatch.

## Solution Assessment

No, not as written. The direction is good, but the maintainable fix should use the current `hook-types.ts` surface and `onlyPluginIds`-scoped registry loading instead of broad runtime materialization.

## Review Findings

- **[P2] Load only updated plugins for lifecycle dispatch:** `src/cli/plugins-update-command.ts:198`
  - body: The new dispatch path calls `ensurePluginRegistryLoaded({ scope: "all" })` before invoking targeted `plugin_updated` handlers.
