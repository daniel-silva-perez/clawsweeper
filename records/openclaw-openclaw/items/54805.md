## Summary

Keep this PR open: current main still lacks the requested post-update plugin lifecycle hook, and the PR discussion shows the initial no-op update/type drift review comments were addressed. It should not be merged as-is because the branch is stale against the current hook-type and plugin-install-index refactors, and the dispatch path should avoid loading every plugin for a targeted update hook.

## What This Changes

The PR adds a typed `plugin_updated` hook and dispatch path for successful plugin updates, updates docs/tests, and adds a Windows formatter `maxBuffer` change in `scripts/lib/format-generated-module.mjs`.

## Best Possible Solution

Refresh the implementation onto current main by adding the hook contract in `src/plugins/hook-types.ts`, emitting lifecycle events only for real successful version changes, persisting the installed-plugin index through the current update path, refreshing/loading only the updated plugin IDs for targeted dispatch, and keeping the docs/tests aligned. The unrelated formatter `maxBuffer` fix should be split out or kept only if the refreshed PR still needs it for CI.

## Reproduction Assessment

Yes. Current main can be checked by source inspection: `plugin_updated` is absent from the typed hook registry, `updateNpmInstalledPlugins` has no lifecycle callback, and `runPluginUpdateCommand` has no dispatch step.

## Solution Assessment

No, not as written. The PR direction is the right product shape, but the current branch is stale and its `scope: "all"` registry load is broader than the targeted lifecycle hook requires; a scoped, rebased implementation would be the safer maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.78

Full review comments:

- **[P2] Load only updated plugins for the lifecycle dispatch:** `src/cli/plugins-update-command.ts:198`
  - body: The dispatch path materializes the plugin registry with `scope: "all"` before invoking targeted `plugin_updated` handlers. That can execute unrelated plugin runtime registration during a single-plugin update, which conflicts with the targeted lifecycle design and the current loader's laziness expectations. Use the updated plugin IDs with the scoped loader/current refresh path before calling `runPluginUpdatedForPlugin`.
  - confidence: 0.78

## Security Review

Status: needs_attention
