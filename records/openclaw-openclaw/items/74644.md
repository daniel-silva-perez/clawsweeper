## Summary

Keep open: current main still resolves media-provider execution auth and requires an API key before the audio provider implementation runs, and the linked implementation PR is open rather than merged. The report is actionable and already has a concrete candidate fix in #75005.

## What This Changes

Allow media-understanding audio providers to opt into a local/no-auth execution path instead of being rejected by generic provider API-key resolution before transcription runs.

## Best Possible Solution

Land a narrow additive media-understanding provider auth opt-in, with regression tests for plugin-only local audio/video providers and docs for plugin authors, while keeping normal credential resolution for existing providers and remote no-key cases.

## Reproduction Assessment

Yes. The static path is high-confidence: configure `tools.media.audio.models` to use a plugin-only provider with `transcribeAudio` and no credentials, and current main reaches `resolveApiKeyForProvider`/`requireApiKey` before the provider callback; I did not run tests because this review is read-only.

## Solution Assessment

Yes, the linked direction appears to be the right maintainable shape: an explicit opt-in synthetic/no-auth marker for media providers avoids treating missing credentials as consent while unblocking local STT providers. The final PR still needs normal review and validation before this issue should close.
