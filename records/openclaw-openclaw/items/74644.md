## Summary

Keep open: current main still resolves media audio/video provider execution through model auth and requires an API key before invoking the provider implementation, and an open unmerged PR, #75005, explicitly references this issue as its fix. The issue should stay open until that PR is reviewed and merged or replaced.

## What This Changes

The issue asks OpenClaw to let local or no-auth media-understanding audio/STT providers opt out of auth-store API key requirements before transcription runs.

## Best Possible Solution

Add or land a narrow explicit media-provider no-auth/synthetic-auth opt-in that is checked before requiring an API key, while preserving the current fail-closed behavior for remote or unmarked providers and documenting the plugin metadata contract.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure an audio media provider with no usable auth and a `transcribeAudio` implementation; current main resolves provider auth at `src/media-understanding/runner.entries.ts:620` before the call at `src/media-understanding/runner.entries.ts:644`, so the provider never runs.

## Solution Assessment

Unclear until #75005 is reviewed. Its stated direction is the right narrow shape, but current main is not sufficient because the media runner still treats missing provider auth as fatal before plugin execution.
