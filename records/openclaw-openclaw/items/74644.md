## Summary

Keep open. Current main still resolves media audio/video provider execution auth through `resolveApiKeyForProvider` and `requireApiKey` before invoking plugin media providers, while the provider request types require an `apiKey` and the manifest/docs do not expose a media-provider no-auth opt-in. The linked open PR #75005 uses closing syntax for this issue, so the issue should stay open until that PR is reviewed and either merged or rejected.

## What This Changes

The issue requests a plugin-facing way for media understanding audio/STT providers to run local or no-auth transcription without being blocked by API-key resolution.

## Best Possible Solution

Review #75005 as the canonical implementation path: add a narrow, explicit, fail-closed opt-in for local/no-auth media providers, with tests proving marked local audio providers reach `transcribeAudio` without credentials while unmarked providers still fail closed.

## Reproduction Assessment

Yes. The source path is enough to reproduce with high confidence: configure an audio media provider without a literal/configured/auth-store credential and the runner reaches `requireApiKey` before invoking the provider's `transcribeAudio`.

## Solution Assessment

Yes in direction. An explicit synthetic/no-auth opt-in for media providers is the narrow maintainable fix because it avoids weakening the default auth gate for ordinary remote providers; the exact implementation should be judged in PR #75005.
