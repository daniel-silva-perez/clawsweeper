## Summary

Keep open. Current main still resolves media provider auth and calls `requireApiKey` before invoking audio/video media provider implementations, while the media provider request type requires an `apiKey`; the manifest metadata/docs do not yet expose an explicit no-auth media execution opt-in. The issue is also paired with open closing PR #75005, so it should stay open until that implementation is reviewed and merged or rejected.

## What This Changes

The issue requests a media-understanding auth path that lets local/no-auth STT providers opt into non-secret or synthetic auth instead of failing before `transcribeAudio` runs.

## Best Possible Solution

Review and land a narrow, fail-closed media-provider opt-in for synthetic/no-auth execution, with tests proving local audio providers reach `transcribeAudio` without configured credentials while unmarked remote providers still fail closed.

## Reproduction Assessment

Yes. The source path is enough to reproduce with high confidence: configure an audio media provider without a literal/configured/auth-store API key and the runner reaches `requireApiKey` before invoking the provider's `transcribeAudio`.

## Solution Assessment

Yes in direction: an explicit opt-in synthetic/no-auth hook for media providers is the narrow maintainable fix because it preserves the existing auth gate for ordinary remote providers. The exact implementation should be evaluated in PR #75005 before closing this issue.
