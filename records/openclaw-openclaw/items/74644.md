## Summary

Keep open. Current main still routes media audio/video provider execution through model-auth API-key resolution before invoking the provider, and the only direct fix is the open, unmerged PR #75005 that references this issue with closing syntax.

## What This Changes

Allow media-understanding audio/STT providers that intentionally represent local or no-auth services to run without requiring auth-store API key credentials.

## Best Possible Solution

Review and land PR #75005, or an equivalent narrow fix, so media-understanding providers can explicitly opt into synthetic/no-auth execution while unmarked remote providers continue to fail closed and the behavior is covered by local audio/video regression tests.

## Reproduction Assessment

Yes. The source path is deterministic: configure a plugin-only `local-audio` media provider with an explicit audio model entry and no auth profile/API key, then `resolveProviderExecutionAuth` throws before `transcribeAudio` is called; PR #75005 includes the matching regression test shape.

## Solution Assessment

Yes, directionally. An explicit provider-owned media synthetic-auth opt-in is narrower and safer than bypassing `requireApiKey` globally because it preserves fail-closed behavior for unmarked providers while allowing local/no-auth plugins to declare intent.
