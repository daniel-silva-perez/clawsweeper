## Summary

Keep open. Current main has scoped exec/process primitives and active-run abort handling, but it still lacks the requested operator-facing Gateway/RPC/CLI surface to list and terminate all running exec child processes for an arbitrary stored session. The existing HTTP kill route is subagent-run control, not session-wide exec cleanup, and the prior issue discussion reached the same conclusion.

## What This Changes

The issue requests a session-level command or abort enhancement that lists and terminates all running exec child processes spawned by a session.

## Best Possible Solution

Add an explicit additive Gateway protocol and CLI surface that resolves a canonical session key, lists active supervised exec records for that session scope, terminates matching active runs through supervisor cancellation with process-tree fallback where appropriate, returns per-run status, and documents authorization/isolation rules without silently changing default sessions.abort behavior.

## Reproduction Assessment

Yes for the missing operator surface: current-main source and docs inspection shows the requested sessions kill/exec-list commands and abort flags are absent while lower-level scoped process primitives exist. A live zombie-process scenario was not rerun because this review was read-only and the source-level absence is sufficient to keep the feature open.

## Solution Assessment

Unclear as an implementation plan: session-wide list/kill is the right end state, but the exact RPC/CLI shape and authorization model need maintainer judgment. The safer direction is an additive, explicitly scoped operator API rather than overloading sessions.abort silently.
