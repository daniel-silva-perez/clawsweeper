## Summary

Keep open. Current main has scoped exec/process primitives and same-agent process-tool controls, but it still lacks the requested operator-facing Gateway/RPC/CLI surface to list and terminate all active exec child processes for an arbitrary stored session; the existing abort and HTTP kill paths cover active chat/subagent control rather than session-wide exec cleanup.

## What This Changes

The issue requests a sessions kill/list capability that terminates all still-running child exec processes associated with a session.

## Best Possible Solution

Add an explicit, additive Gateway protocol and CLI surface that resolves a canonical session key, lists active supervised exec/process records for that session scope, terminates matching active runs through supervisor cancellation with process-tree fallback where appropriate, returns per-run status, and documents the authorization and isolation rules without silently changing default sessions.abort behavior.

## Reproduction Assessment

Yes for the missing operator surface: current-main source and docs inspection shows no sessions kill/exec-list command and no abort flag for child exec termination, while lower-level scoped process primitives exist. A live zombie-process scenario was not rerun because this was a read-only review and the source-level absence is sufficient to keep the feature open.

## Solution Assessment

Unclear as an implementation plan: the requested session-wide list/kill capability is the right end state, but the exact Gateway/RPC/CLI contract and authorization model need maintainer design. The safer direction is an additive, explicitly scoped operator API rather than overloading sessions.abort silently.
