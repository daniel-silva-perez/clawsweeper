## Summary

Keep open. Current main has better process-supervisor primitives and per-process `process` controls, but it still does not provide a Gateway/RPC/CLI operator surface to list and terminate all exec runs for a canonical session; `sessions.abort` and the HTTP session-kill route do not enumerate or cancel the session's process-supervisor scope. Related #39960 covered active stop/foreground abort behavior, while #38721 is adjacent gateway-shutdown cleanup rather than the requested session operator feature.

## What This Changes

The issue requests a `sessions_kill` or equivalent session-level CLI/Gateway operator action that can list and terminate all child exec processes spawned by a session.

## Best Possible Solution

Keep this open for maintainer design of an additive Gateway/RPC/CLI operator surface that resolves a canonical session key, lists managed exec/process runs for that exact session, terminates matching runs through supervisor cancellation plus process-tree fallback, returns per-run status, and documents authorization/isolation semantics without changing default `sessions.abort` behavior.

## Risks / Open Questions

- A naive session-wide kill could terminate unrelated processes if canonical session keys, process scopes, node-host scopes, or sandbox scopes are not resolved strictly.
- Changing default `sessions.abort` behavior could break the current contract that background exec runs survive tool-run abort signals unless an explicit kill action is requested.
- The operator surface needs authorization and isolation design before implementation, including which process metadata can be exposed and which clients may kill runs.
- Related #38721 is a narrower shutdown cleanup candidate; broadening that work into session-level operator control without maintainer design would mix two different product contracts.
