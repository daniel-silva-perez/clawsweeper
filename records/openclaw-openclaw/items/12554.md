## Summary

Current main still lacks the requested session-wide operator surface to list and terminate all exec/process runs associated with an OpenClaw session. The repo has useful per-process controls and active-run abort wiring, but `sessions.abort` still only accepts `key`/`runId` and delegates to `chat.abort`; CLI/docs expose no `sessions exec-list`, `sessions kill`, or `abort --kill-exec` equivalent.

## Best Possible Solution

Keep this issue open for maintainer design of an additive Gateway/RPC/CLI operator surface: resolve the canonical session key, list managed exec/process runs for that session, terminate matching runs through supervisor cancellation plus process-tree fallback, return per-run status, document the contract, and cover authorization, session isolation, background exec, sandbox/node-host behavior, and subagent interactions.

## Risks / Open Questions

- A session-wide kill API must preserve authorization and session isolation; operators must not be able to inspect or terminate unrelated sessions' exec processes.
- Normal abort/stop should not unexpectedly kill intentionally backgrounded exec jobs unless maintainers explicitly choose that behavior; an explicit operator action or explicit abort flag is safer.
- Gateway, sandbox, and node-host exec paths may need separate process-record and cancellation handling so the feature works beyond in-process gateway exec.
- The existing `POST /sessions/:sessionKey/kill` route already means subagent kill, so a new API needs clear naming to avoid breaking or confusing existing clients.
