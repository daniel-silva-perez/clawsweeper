## Summary

Keep open. Current main has scoped exec/process primitives and per-session process-tool isolation, but it still lacks the requested operator-facing Gateway/RPC/CLI surface to list and terminate all active exec child processes for an arbitrary stored session. The prior issue comments reached the same conclusion, and the current source still matches that assessment.

## What This Changes

The issue requests a session-level command/API that can list running exec processes for a session and terminate all child exec processes spawned by that session.

## Best Possible Solution

Add an explicit additive Gateway protocol and CLI surface that resolves a canonical session key, lists active supervised exec/process records for that session scope, terminates matching active runs through supervisor cancellation with process-tree fallback where appropriate, returns per-run status, and documents authorization/isolation rules without silently changing default sessions.abort behavior.

## Reproduction Assessment

Yes for the missing operator surface: current-main source and docs inspection shows no sessions kill/exec-list commands and no abort flag for child exec termination, while lower-level scoped process primitives exist. A live zombie-process scenario was not rerun because this review was read-only and the source-level absence is enough to keep the feature open.

## Solution Assessment

Unclear as an implementation plan: the requested session-wide list/kill capability is the right end state, but the exact Gateway/RPC/CLI contract and authorization model need maintainer design. The safer direction is an additive, explicitly scoped operator API rather than overloading sessions.abort silently.
