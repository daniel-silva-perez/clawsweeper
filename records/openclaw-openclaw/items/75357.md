## Summary

Keep open. Current main has the usage-aware OpenClaw completions transport, but the standalone simple-completion routing still only switches to that transport when proxy/TLS overrides exist; for generic local OpenAI-compatible endpoints, OpenClaw can also force `compat.supportsUsageInStreaming` false, so the pinned dependency path may omit `stream_options.include_usage` and keep recording zero usage. The broader #73990 is related, but this report is a narrower, actionable route gap.

## What This Changes

`openai-completions` simple no-proxy/no-TLS sessions should request streaming usage from compatible local/OpenAI-compatible endpoints and persist non-zero assistant usage in session JSONL.

## Best Possible Solution

Add a narrow routing fix that makes no-proxy/no-TLS `openai-completions` simple completions use the usage-aware OpenClaw transport, or an equivalent generic seam, when the endpoint is local or otherwise known to support streaming usage. Preserve provider-owned streams and unsupported provider behavior, and add regression coverage for a final usage-only streaming chunk.

## Reproduction Assessment

Yes. The issue includes a concrete llama.cpp curl reproduction, and the source-level route check shows the simple no-proxy path can still avoid the OpenClaw transport that requests and parses streaming usage; a live endpoint was not available in this read-only pass.

## Solution Assessment

Yes. The narrowest maintainable fix is to reuse the existing usage-aware OpenAI completions transport/routing seam rather than adding provider-specific session-log repair or asking users to configure fake proxy/TLS overrides.
