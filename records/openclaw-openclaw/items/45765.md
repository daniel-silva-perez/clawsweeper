## Summary

Keep open: current `main` still has the reported nested `OPENCLAW_HOME` path behavior in both state-dir and config-dir resolution, and the issue is already paired with an open fix PR that should be reviewed or replaced rather than spawning another cleanup action.

## What This Changes

The issue reports that setting `OPENCLAW_HOME` to an existing `.openclaw` directory causes OpenClaw to write config and state into a nested `.openclaw/.openclaw` directory.

## Best Possible Solution

Implement a focused core path-resolution fix that prevents double nesting when the explicit home path is already a known state-directory basename, covers both `resolveStateDir()` and `resolveConfigDir()`, adds regression tests for tilde and legacy state names, and documents or repairs the migration edge case for users who already have nested state.

## Reproduction Assessment

Yes. The static call path on current `main` is deterministic: `OPENCLAW_HOME=~/.openclaw` is resolved as the effective home, then both state and config fallback logic append another `.openclaw`.

## Solution Assessment

No, current `main` is not the fix yet. A complete solution should touch both path helpers and tests; the open one-file PR may need expansion or replacement before merge.
