## Summary

Keep open: current main still deterministically produces the reported nested path for an explicit `OPENCLAW_HOME` ending in `.openclaw`, and the issue already has an open related fix PR that should be reviewed or replaced rather than closed early.

## What This Changes

Prevent `OPENCLAW_HOME=/home/user/.openclaw` from resolving config or state paths under `/home/user/.openclaw/.openclaw`.

## Best Possible Solution

Land a focused core path-resolution fix that only bypasses the extra `.openclaw` append when the explicit home path already names a known state directory, covers both state and config helpers, preserves `/srv/app -> /srv/app/.openclaw`, and adds regression tests plus any needed doctor or migration note for already-nested installs.

Yes. The static path through current main is enough: set `OPENCLAW_HOME` to a path whose basename is `.openclaw`, then both state and config fallback logic append another `.openclaw`.
