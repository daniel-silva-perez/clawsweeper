## Summary

Keep open: current main and the latest release still append an extra `.openclaw` segment after an explicit `OPENCLAW_HOME` that already points at `.openclaw`. The discussion and related PRs identify the right bug area, but no merged/shipped fix is present yet.

## What This Changes

The issue requests that `OPENCLAW_HOME=/path/.openclaw` stop producing nested state/config paths like `/path/.openclaw/.openclaw` during onboarding and path resolution.

## Best Possible Solution

Land a focused compatibility fix that bypasses the extra `.openclaw` append only when an explicit home path already names a known state directory, applies the rule consistently to state and config path helpers, and adds regression coverage for `.openclaw`, legacy dirnames, tilde expansion, and preserved `/srv/app` parent-home behavior.

## Reproduction Assessment

Yes. A static trace on current main reproduces the path construction: `OPENCLAW_HOME=/home/tpam/.openclaw` is preserved by the home resolver, then both state and config helpers append another `.openclaw`.

## Solution Assessment

No. Current main is not fixed; the narrow maintainable solution is a basename/state-dir guard applied consistently, rather than treating every explicit `OPENCLAW_HOME` as the final state/config directory.
