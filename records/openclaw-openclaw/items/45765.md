## Summary

Keep open: current main still routes an explicit `OPENCLAW_HOME=/home/user/.openclaw` through helpers that append another `.openclaw`, and the open related PR has not landed and is unsafe as-is.

## What This Changes

The issue requests that OpenClaw avoid creating `~/.openclaw/.openclaw` when `OPENCLAW_HOME` already points at the `.openclaw` state directory.

## Best Possible Solution

Land a focused compatibility fix that updates `resolveStateDir()` and `resolveConfigDir()` together, bypasses the extra `.openclaw` append only when an explicit home path already names a known state directory, preserves general service-home behavior, and adds regression tests for both the fixed and preserved cases.

## Reproduction Assessment

Yes. A static source trace on current main is enough: set `OPENCLAW_HOME` to a path whose basename is `.openclaw`, then both state and config helpers append another `.openclaw` segment.

## Solution Assessment

No. Current main is not fixed, and the open PR's direct-use approach is broader than the documented contract; the safer solution is a basename/state-dir guard applied consistently to both helpers.
