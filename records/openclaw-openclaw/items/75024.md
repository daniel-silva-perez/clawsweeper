## Summary

Keep this PR open. Current main still maps Feishu `ctx.rootId` only from `root_id` while topic-session routing depends on `rootId` or `threadId`, and the PR is a narrow Feishu parser/type/test change with no blocking review finding.

## What This Changes

The PR changes Feishu message parsing to fall back from `root_id` to `reply_target_message_id`, `parent_id`, and `upper_message_id`, adds `upper_message_id` to the event type, and adds parser tests for fallback order.

## Best Possible Solution

Land a narrow Feishu plugin fix that preserves existing `root_id` and `thread_id` precedence, normalizes reply/parent ancestors only when the root is missing, and keeps regression coverage around parser fallback plus topic-session routing.

## Reproduction Assessment

Yes. A focused reproduction is a Feishu group topic reply event with no `root_id` or `thread_id` but with `reply_target_message_id` or `parent_id`; on current main, `parseFeishuMessageEvent` leaves `ctx.rootId` unset, so topic routing cannot use that ancestor as the session root.

## Solution Assessment

Yes, with optional strengthening. Normalizing the fallback in `parseFeishuMessageEvent` is the narrowest maintainable boundary because downstream routing and reply dispatch already consume `ctx.rootId`; a dispatch-level regression test would make the proof stronger.
