## Summary

Keep this PR open for maintainer review. Current main still only maps Feishu `root_id` into `ctx.rootId`, while downstream group-topic routing depends on `ctx.rootId`/`ctx.threadId`; the PR is a narrow Feishu plugin change with focused parser coverage and no blocking review finding found in this read-only pass.

## What This Changes

The PR changes Feishu message parsing to derive `ctx.rootId` from `root_id` first, then `reply_target_message_id`, `parent_id`, and `upper_message_id`, with parser tests and a type update.

## Best Possible Solution

Land a narrow Feishu plugin fix that preserves existing `root_id` and `thread_id` precedence, normalizes reply/parent ancestors only when the root is missing, and keeps regression coverage around parser fallback plus, ideally, topic-session routing.

## Reproduction Assessment

Yes. A focused reproduction is a Feishu group/topic reply event with no `root_id` or `thread_id` but with `reply_target_message_id` or `parent_id`; current main leaves `ctx.rootId` unset in `parseFeishuMessageEvent`, so routing cannot use that ancestor as the topic root.

## Solution Assessment

Yes, with optional strengthening. Normalizing the fallback in `parseFeishuMessageEvent` is the narrowest maintainable boundary because downstream routing and reply dispatch already consume `ctx.rootId`; a dispatch-level regression test would make the proof stronger.
