## Summary

Keep this PR open for maintainer review. Current main still normalizes Feishu `ctx.rootId` only from `root_id`, while downstream topic-session routing depends on `ctx.rootId`; the PR is a narrow Feishu plugin change and I found no discrete blocking correctness or security finding in the proposed diff.

## What This Changes

The PR updates Feishu message parsing to fall back from `root_id` to `reply_target_message_id`, `parent_id`, and `upper_message_id`, adds `upper_message_id` to the event type, and adds parser coverage for the fallback order.

## Best Possible Solution

Land a narrow Feishu plugin fix that preserves existing `root_id` and `thread_id` precedence, normalizes reply/parent ancestors only when the root is missing, and keeps regression coverage around parser fallback plus topic-session routing.

## Reproduction Assessment

Yes. A focused reproduction is a Feishu group topic reply event with no `root_id` or `thread_id` but with `reply_target_message_id` or `parent_id`; on current main, `parseFeishuMessageEvent` leaves `ctx.rootId` unset, so topic routing cannot use that ancestor as the session root.

## Solution Assessment

Yes, with optional strengthening. Normalizing the fallback inside `parseFeishuMessageEvent` is the narrowest maintainable boundary because downstream routing and reply dispatch already consume `ctx.rootId`; a dispatch-level test would make the proof stronger.
