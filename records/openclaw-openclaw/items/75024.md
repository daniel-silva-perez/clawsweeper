## Summary

Keep open. The PR addresses a real Feishu threading gap still present on current main, and the parser-level fallback is a narrow functional fix, but the user-facing `fix(feishu)` change is missing the repo-required changelog entry before merge.

## What This Changes

The PR updates Feishu message parsing to fall back from `root_id` to reply/parent ancestor fields, adds `upper_message_id` to the event type, and adds parser tests for fallback order.

## Best Possible Solution

Merge a narrow Feishu plugin fix after adding the required Unreleased changelog entry and, if maintainers want stronger proof, a dispatch-level regression that shows the fallback root keeps topic-session routing stable.

## Reproduction Assessment

Yes. A focused reproduction is a Feishu group/topic reply event with no `root_id` or `thread_id` but with `reply_target_message_id` or `parent_id`; current main leaves `ctx.rootId` unset in `parseFeishuMessageEvent`, so downstream topic routing cannot use that ancestor as the session root.

## Solution Assessment

Yes, with the changelog gap fixed. Normalizing the fallback inside `parseFeishuMessageEvent` is the narrowest maintainable boundary because downstream session routing and reply dispatch already consume `ctx.rootId`.
