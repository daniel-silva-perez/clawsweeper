## Summary

Keep this PR open for maintainer review. Current main still leaves `ctx.rootId` unset when Feishu omits `root_id`, while downstream Feishu topic routing uses that normalized root to keep group-topic sessions together; the PR is a narrow parser/type/test change with no blocking review finding from this pass.

## What This Changes

The PR changes Feishu message parsing to fall back from `root_id` to `reply_target_message_id`, `parent_id`, and `upper_message_id`, extends the event type, and adds parser coverage for fallback order.

## Best Possible Solution

Land a narrow Feishu plugin fix that normalizes the best available ancestor into `ctx.rootId` before routing, preserves existing `root_id` and `thread_id` precedence, and adds enough regression coverage to prove session grouping stays stable when Feishu omits `root_id`.

## Reproduction Assessment

Yes. A focused parser-level reproduction is a Feishu group reply event with no `root_id` but with `reply_target_message_id`, `parent_id`, or `upper_message_id`; on current main, `parseFeishuMessageEvent` leaves `ctx.rootId` unset, which prevents topic routing from using that ancestor as the session root.

## Solution Assessment

Yes, with optional strengthening. Normalizing the fallback inside `parseFeishuMessageEvent` is the narrowest maintainable boundary because downstream routing already consumes `ctx.rootId`; a dispatch-level regression test would make the landing proof stronger.
