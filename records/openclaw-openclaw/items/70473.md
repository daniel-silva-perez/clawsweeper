## Summary

Keep open. Current main still has the narrower overflow token extractor and only forwards `currentTokenCount` when a provider message parses into the existing formats, while this PR adds still-missing MiniMax/Kimi/additive extraction plus a synthetic over-budget fallback for confirmed unparseable overflow text. Related #70479 is complementary session-preservation work, not a replacement for this PR.

## What This Changes

The PR broadens provider overflow token-count extraction, forwards a synthetic `currentTokenCount` for confirmed unparseable overflows, and adds focused agent regression tests plus compaction docs.

## Best Possible Solution

Land this narrow overflow-budget extraction and fallback path with the proposed tests and docs after normal maintainer review and CI, while keeping #70479 as the separate session-preservation follow-up.

## Reproduction Assessment

Yes. The current-main code path is clear: provider messages that are already classified as overflow can still return `undefined` from `extractObservedOverflowTokenCount`, causing the runner to omit `currentTokenCount`; the PR’s added tests exercise that gap.

## Solution Assessment

Yes. The PR uses the existing generic `currentTokenCount` context-engine contract and keeps the change scoped to provider-error parsing, overflow compaction routing, regression tests, and docs, which is the narrow maintainable fix for the reported behavior.
