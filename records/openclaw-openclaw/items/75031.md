## Summary

Keep open. Current main still has the backup staging bug, and this PR takes the right primary direction, but the patch is incomplete because the shared temp-dir helper can fall back to `os.tmpdir()`, which may still be the state-local `TMPDIR` involved in the linked bug.

## What This Changes

This PR switches backup archive staging from `os.tmpdir()` to `resolvePreferredOpenClawTmpDir()` and adds a regression test for `TMPDIR` pointing inside the OpenClaw state directory.

## Best Possible Solution

Land a focused backup fix that stages the manifest outside every included backup source, preserves the existing atomic archive publication flow, and adds regression coverage for both the macOS state-local `TMPDIR` case and the resolver fallback/containment guard.

## Reproduction Assessment

Yes. The source path is high-confidence: macOS services set `TMPDIR` under the state directory, current backup creation stages `manifest.json` under `os.tmpdir()`, and the tar input also walks the state source tree.

## Solution Assessment

No. Using `resolvePreferredOpenClawTmpDir()` is the right starting point, but the maintainable fix should also verify that the resolved staging root is outside all included backup assets because the helper fallback can still derive from `TMPDIR`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Guard the resolved temp root against backup sources:** `src/infra/backup-create.ts:353`
  - body: `resolvePreferredOpenClawTmpDir()` can fall back to `os.tmpdir()/openclaw-uid` when `/tmp/openclaw` is unavailable or unsafe. Under the macOS LaunchAgent case from #75007, `os.tmpdir()` may still be `~/.openclaw/tmp`, so this can stage `manifest.json` under the state asset and reproduce the duplicate manifest. Resolve the temp root and reject or choose another root when it is within any included backup asset.
  - confidence: 0.87
