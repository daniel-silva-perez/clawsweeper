## Summary

Keep open. The updated PR addresses the earlier review direction only partially: if the preferred OpenClaw temp dir is inside a backed-up source, the new fallback returns `os.tmpdir()`, which is also `~/.openclaw/tmp` in the macOS LaunchAgent environment that triggers the bug, so the archive staging directory can still land inside the state payload and duplicate the root manifest.

## What This Changes

This PR imports `resolvePreferredOpenClawTmpDir()`, adds a backup-local temp-root guard with an `os.tmpdir()` fallback, and adds a regression test for a preferred temp root under the OpenClaw state directory.

## Best Possible Solution

Stage the backup manifest in a temp root that has itself been checked against every included backup source. If both the preferred resolver result and `os.tmpdir()` are inside the payload, choose another verified external candidate or fail with an actionable error, and cover the LaunchAgent case by making both the preferred result and `os.tmpdir()` state-local in the regression test.

## Reproduction Assessment

Yes. A high-confidence reproduction is the macOS LaunchAgent setup from #75007: `TMPDIR` resolves to `<stateDir>/tmp`, the backup includes the state directory, and staging `manifest.json` under that tree lets tar pack it once explicitly and once through the state walk.

## Solution Assessment

No. The PR starts in the right area, but falling back from a state-local preferred root to `os.tmpdir()` recreates the same state-local staging path under the real service environment; the fix should only return a root after the candidate passes the same source-containment check.
