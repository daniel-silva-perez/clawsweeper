## Summary

Keep this PR open. Current main still does not implement persistent TTS voice switching via `persist=true`: directive results have no `persist` field, TTS prefs do not store provider-keyed voices, the public TTS runtime facade has no `getTtsVoice`/`setTtsVoice`, and successful TTS paths do not write a selected voice after synthesis. The PR is stale against the current speech-core/provider-plugin refactor, but the feature remains unimplemented and useful if ported carefully.

## Best Possible Solution

Keep this PR open as a stale-but-valid feature candidate. The best path is to port the design to the current speech-core/provider-plugin architecture, add a typed provider-keyed voice preference API if maintainers want this behavior, update docs and focused tests, and re-review the durable preference write policy before merge.

## Risks / Open Questions

- The PR targets older core paths and would need a real port to current `extensions/speech-core`, provider directive parsers, Discord voice code, and plugin SDK facades before it can merge.
- The feature lets model-emitted directives change durable local TTS preferences, so a current implementation should keep `persist` gated by `modelOverrides.allowVoice`, provider-specific validation, and best-effort prefs writes.
- Multiple TTS entry points now exist, including auto replies, Discord voice, telephony, CLI, and Gateway conversion; a port should explicitly define where persisted voices are read and where they may be written.
