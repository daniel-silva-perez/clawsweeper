## Summary

Keep open. Current main does not implement persistent TTS voice switching, and the PR is not landable as-is because it targets pre-refactor `src/tts` and `src/discord` paths instead of the active speech-core runtime, plugin SDK facade, and Discord extension voice path.

## What This Changes

The PR adds provider-keyed TTS voice preferences, parses `persist=true` directives, applies persisted OpenAI/ElevenLabs voices during synthesis, writes voice prefs after successful TTS, and adds TTS tests.

## Best Possible Solution

Rework the feature against the current provider-aware TTS architecture: add a typed persistence signal to `src/tts/directives.ts` and `TtsDirectiveParseResult`, add provider-keyed voice prefs plus facade/runtime APIs in speech-core, resolve persisted voices in normal and telephony synthesis, write prefs only after successful synthesis as best-effort, wire `extensions/discord/src/voice/tts.ts`, and update focused tests and TTS docs.

## Reproduction Assessment

Not applicable as a user bug report, but the PR defects are reproducible by source inspection: current main has no `persist` field, no voice-pref runtime API, and active TTS/Discord paths differ from the files the branch edits.

## Solution Assessment

No. The proposed approach is not the best solution now because it patches obsolete core and Discord files; the safer path is an additive speech-core/plugin-SDK implementation with explicit maintainer agreement on durable model-written preferences.
