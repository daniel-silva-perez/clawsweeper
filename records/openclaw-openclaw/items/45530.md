## Summary

Keep open. Current main does not implement persistent TTS voice switching, and this PR is not ready to merge because it targets pre-refactor TTS and Discord paths rather than the active speech-core runtime and plugin SDK facade.

## What This Changes

The PR branch adds provider-keyed TTS voice preferences, parses `persist=true`, applies persisted OpenAI/ElevenLabs voices during synthesis and telephony, writes voice prefs after successful TTS, and adds focused TTS tests.

## Best Possible Solution

If maintainers want this capability, rebuild it against the current speech-core/plugin architecture: add an explicit persistence signal to the provider-aware directive parser and result type, add provider-keyed voice prefs and facade/runtime APIs in speech-core, resolve persisted voices in normal and telephony synthesis, write prefs only after successful synthesis as best-effort, wire the current Discord plugin voice path, and update focused tests plus TTS docs.

## Reproduction Assessment

Not applicable as a user bug report. The PR defects are reproducible by source and PR inspection: current main has no `persist` signal, no voice-pref runtime API, and the active Discord voice path differs from the files changed by the branch.

## Solution Assessment

No. The proposed patch is not the best way to solve the feature now because it patches obsolete core and Discord files; the safer solution is an additive speech-core/plugin-SDK implementation after maintainer agreement on durable model-written preferences.
