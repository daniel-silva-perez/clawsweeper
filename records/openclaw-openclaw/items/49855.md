## Summary

Keep this issue open. Current main still downloads Feishu/Lark video and file message resources with `type=file`, has no `type=media` retry/fallback for file-type videos after Feishu 5xx responses, and open unmerged PRs #49967 and #50164 explicitly reference this issue as implementation candidates.

## Best Possible Solution

Keep the issue open and review the focused Feishu fix. The best product/code path is in `extensions/feishu`: add a conservative retry from `type=file` to Feishu's media resource type on 502/503/504 or other vetted server errors, add targeted tests for preserving non-video file behavior and retrying video-like file messages, and then close the issue only after a PR lands on main.
