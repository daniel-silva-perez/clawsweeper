## Summary

Keep this issue open. Current main has partial WebChat TTS delivery plumbing, but the browser Control UI path still does not clearly provide playable or downloadable audio: backend TTS audio is converted into base64 `type: "audio"` content blocks, then `chat.final`/`chat.history` projection removes the base64 payload, and the UI renders audio controls only for attachment items or transcript media paths, not those audio content blocks.

## Best Possible Solution

Keep this issue open and fix the WebChat TTS delivery path end to end. The conservative path is to turn trusted local TTS output into a renderable/downloadable WebChat attachment or authenticated media URL, or teach the UI to safely normalize/render gateway audio content blocks. Add regression coverage across `chat.send`, `chat.final`, `chat.history`, and Control UI rendering, while keeping the existing local-media trust and size-limit checks.
