## Summary

Keep open. Current main has partial WebChat TTS plumbing, but the observable WebChat playback/download problem is not clearly solved: TTS tool media is generated and promoted into the WebChat reply path, yet the gateway projects base64 audio blocks with the playable data removed and the Control UI renderer only plays audio when it is represented as an attachment URL.

## Best Possible Solution

Keep this issue open and fix the WebChat delivery path end to end: turn trusted local TTS output into a renderable/downloadable WebChat attachment or authenticated media URL, teach the UI to normalize/render that representation, and add regression coverage for tts tool results through chat.send, chat.final, and chat.history. Keep #57297 as the separate observability/status follow-up.
