## Summary

Keep this issue open. Current main still has the same end-to-end gap: TTS succeeds and produces trusted local media, WebChat backend can turn that media into raw base64 audio content blocks, but chat final/history projection strips the playable bytes and the Control UI only renders audio controls for normalized attachment items, not raw `type: "audio"` blocks. The linked same-author observability issue #57297 is also still open and remains a companion, not a replacement.

## Best Possible Solution

Keep this issue open and fix WebChat TTS delivery end to end. A conservative implementation would convert trusted local TTS output into a renderable/downloadable WebChat attachment or authenticated media URL, or safely normalize/render gateway audio content blocks in the Control UI. Add regression coverage across `chat.send`, `chat.final`, `chat.history`, and Control UI rendering, and coordinate logging/status work with #57297.
