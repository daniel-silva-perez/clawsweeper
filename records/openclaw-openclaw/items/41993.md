## Summary

Keep open. Current main still has the reported strict mixed-DNS behavior: `web_fetch` reaches the SSRF-pinned fetch path, DNS answers are checked with an any-blocked assertion, and IPv4-first pinning only happens after that assertion. The related implementation PRs in the provided context are closed and unmerged, and the shipped RFC2544 opt-in only covers the `198.18.0.0/15` fake-IP case, not mixed public IPv4 plus blocked special-use IPv6 answers like `::`, `::1`, or Teredo.

## Best Possible Solution

Keep this issue open. Implement a narrow change in `src/infra/net/ssrf.ts` that classifies resolved DNS answers, filters out blocked addresses, throws only when no allowed answer remains, and builds pinned lookup/dispatcher candidates only from the filtered allowed set. Add focused regression coverage in `src/infra/net/ssrf.pinning.test.ts` and, if useful, `src/agents/tools/web-fetch.ssrf.test.ts` for public IPv4 plus `::`, public IPv4 plus blocked Teredo/reserved IPv6, all-blocked DNS still failing, IPv4-first ordering after filtering, and pinned lookup results excluding blocked addresses.

## Risks / Open Questions

- This is SSRF-sensitive code: the fix should filter blocked DNS answers before constructing pinned lookup or dispatcher candidates, while still failing when every resolved answer is blocked.
- A broad private-network or RFC2544-only workaround would either weaken SSRF policy or leave the reported special-use IPv6 mixed-answer failure in place.
- Because related PRs are closed and unmerged, the best path is a fresh narrow patch or a maintainer-reopened implementation candidate with focused regression tests.
