## Summary

Keep open. Current main still has the reported strict mixed-DNS behavior: the SSRF resolver checks every DNS answer and throws when any resolved address is blocked, before IPv4-first pinning runs. `web_fetch` still reaches that guarded path and rethrows SSRF blocks, so a public IPv4 answer mixed with `::`, `::1`, or blocked Teredo-style IPv6 can still fail.

## Best Possible Solution

Keep this issue open. Implement a narrow change in `src/infra/net/ssrf.ts` that classifies DNS answers, filters out blocked addresses, throws only when no allowed answer remains, and builds pinned lookup/dispatcher candidates only from the filtered allowed set. Add regression coverage in `src/infra/net/ssrf.pinning.test.ts` and, if useful, `src/agents/tools/web-fetch.ssrf.test.ts` for public IPv4 plus `::`, public IPv4 plus blocked Teredo/reserved IPv6, all-blocked DNS still failing, IPv4-first ordering after filtering, and blocked addresses excluded from pinned lookup results.

## Risks / Open Questions

- This is SSRF-sensitive code: the fix should filter blocked DNS answers before constructing pinned lookup or dispatcher candidates, while still failing when every resolved answer is blocked.
- Broad private-network or RFC2544-only workarounds either weaken the SSRF boundary or leave the reported special-use IPv6 mixed-answer failure in place.
- The closed related PRs are useful implementation references, but none is merged, so closing now would lose the active bug tracker for the remaining work.
