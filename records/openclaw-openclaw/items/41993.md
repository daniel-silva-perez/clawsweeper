## Summary

Keep open. Current main still rejects mixed DNS answers when any resolved address is blocked, before IPv4-first pinning can use an allowed public IPv4 address. The related implementation PRs in the provided issue context are closed and unmerged, and the shipped RFC2544 opt-in only covers the 198.18.0.0/15 fake-IP/proxy case, not public IPv4 mixed with special-use IPv6 like `::`, `::1`, or Teredo.

## Best Possible Solution

Keep this issue open. Implement a narrow fix in `src/infra/net/ssrf.ts` that classifies resolved DNS answers, filters out blocked addresses, throws only when no allowed answer remains, and builds pinned lookup/dispatcher candidates from the filtered allowed set. Add focused regression coverage in `src/infra/net/ssrf.pinning.test.ts` and, if useful, `src/agents/tools/web-fetch.ssrf.test.ts` for public IPv4 plus `::`, public IPv4 plus blocked Teredo/reserved IPv6, all-blocked DNS still failing, IPv4-first ordering after filtering, and pinned lookup results excluding blocked addresses.

## Risks / Open Questions

- This is SSRF-sensitive code: the fix should filter blocked DNS answers before constructing pinned lookups or dispatcher override candidates, while still failing when every resolved answer is blocked.
- Treating the RFC2544 fake-IP opt-in as the complete fix would leave the reported special-use IPv6 mixed-answer failure in place.
- The checkout history is shallow/grafted, so provenance of the original v2026.3.8 regression could not be traced locally without fetching more history; current main behavior is still clear from source.
