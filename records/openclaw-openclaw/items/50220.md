## Summary

Keep open. Current main preserves explicit node workdirs at the gateway layer, but the node-host approval path still canonicalizes `cwd` inside `system.run.prepare` with `lstatSync`/`statSync`/`realpathSync` and returns the reported `INVALID_REQUEST` message when that local macOS launchd check fails. Hollychou924's firmlink/dev+ino comment is useful direction, but this is an exec-approval binding path and needs careful maintainer review rather than stale cleanup.

## What This Changes

The issue asks for macOS headless node-host `system.run` approval to accept valid `/Users/<user>` working directories instead of denying them as non-canonical cwd paths.

## Best Possible Solution

Keep the issue open and fix it in the node-host approval canonicalization path only after macOS launchd verification. The desired end state is a narrowly scoped macOS-safe cwd approval rule that accepts real `/Users/<user>` firmlink-backed directories while preserving symlink-component rejection, cwd identity binding, and pre-execution drift checks.

## Reproduction Assessment

Unclear. The issue has detailed steps and the current source still contains the exact denial path, but this review did not have a macOS launchd/TCC node-host environment to reproduce the `/Users/<user>` failure live.

## Solution Assessment

No. Closing this as implemented by the existing gateway workdir-forwarding fix would not solve the node-local `resolveCanonicalApprovalCwdSync` failure described here; the safer path is owner review plus targeted macOS coverage for the approval-plan canonical cwd behavior.
