## Summary

Keep this open. Current main has built-in validation, SecretRef preflight, base-hash guarded writes, and an agent-tool allowlist for config.patch/config.apply, but it still does not expose the requested synchronous plugin/API-level config pre-apply veto before config mutations are persisted. A recent maintainer comment also explicitly kept this issue open and tied it to the config validation/mutation RFC direction.

## Best Possible Solution

Keep this open and coordinate with the config validation/mutation RFC. The best implementation is a documented synchronous config-write interception point invoked by config.set, config.patch, config.apply, and shared config mutation helpers after pending config computation plus built-in validation/preflight, but before persistence. It should expose current config, pending config, action, raw patch/full payload, baseHash, actor/client metadata, changed paths, and a typed reject result, with regression tests proving plugin rejection leaves disk and active runtime config unchanged.

## Risks / Open Questions

- Closing now would lose track of a real plugin API gap: external validators can exist, but current main does not give them a mandatory pre-persistence interception point.
- The eventual implementation is security-sensitive and needs explicit decisions on hook ordering, SecretRef preflight ordering, fail-open/fail-closed behavior, recursion avoidance, actor metadata, and reload/restart semantics.
- A partial implementation that only wraps config.patch/config.apply would miss shared helper and plugin-runtime writes through replaceConfigFile/mutateConfigFile.
