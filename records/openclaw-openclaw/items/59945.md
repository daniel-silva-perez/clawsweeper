## Summary

Keep this issue open. The provided discussion confirmed a real runtime-snapshot path, and current main still publishes runtime config snapshots verbatim while Slack startup reads bot/app tokens from that active config. The config write RPC has redaction restoration, but that restoration is not applied at runtime snapshot publication; the claimed Slack sentinel regression test is also not present on main. The restart loop wording is partly less current because the channel manager now caps restarts, but the underlying Slack credential/sentinel failure is still not clearly fixed.

## Best Possible Solution

Fix this in the shared config/runtime layer, not Slack-specific code: before publishing a runtime snapshot with a paired source snapshot, restore or reject any redaction sentinels so live runtime config can never expose __OPENCLAW_REDACTED__ as credential data. Add the regression test described in the issue discussion using Slack botToken/appToken sentinel leakage, and keep the channel restart cap as a separate safety net.
