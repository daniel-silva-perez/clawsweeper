## Summary

Keep open. Current main still publishes runtime config snapshots without restoring or rejecting redaction sentinels, while Slack startup reads bot/app tokens from that active config path. The restart-loop wording is partly outdated because channel restarts are now capped, but the credential/sentinel root cause described in the issue discussion remains actionable.

## What This Changes

The issue asks OpenClaw to stop Slack Socket Mode startup from consuming `__OPENCLAW_REDACTED__` as live token config and repeatedly restarting despite valid Slack tokens.

## Best Possible Solution

Fix this in the shared config/runtime publication path: when both runtime and source snapshots are available, restore or reject any redaction sentinels before exposing live runtime config. Cover it with a regression test using Slack `botToken` and `appToken` sentinel leakage, and keep Slack-specific code limited to consuming already-resolved credentials.

## Reproduction Assessment

Yes, at source/unit level: publish a runtime snapshot containing Slack token sentinels with a paired source snapshot containing real values, then read the active runtime config through the Slack startup/account path. A live Slack gateway repro was not run in this read-only review.

## Solution Assessment

Yes, the shared runtime snapshot layer is the right fix boundary. A Slack-only workaround would leave the same sentinel publication bug available to other channels and provider/plugin credential paths.
