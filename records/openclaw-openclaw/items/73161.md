## Summary

Keep this PR open, but do not merge it as-is. The stale subagent cancellation bug still exists on current main, while this branch is stale/dirty against main, overlaps current runtime and Discord liveness work, and introduces blocking runtime, security-audit, config-doc, and message-tool schema regressions.

## What This Changes

The PR changes task runtime postbuild aliasing and subagent cancellation, Discord and Telegram liveness signals, outbound/message-tool sanitization, failover classification, security trust-model config, `.gitignore`, and coding-agent guidance.

## Best Possible Solution

Split and rebase the work: keep a narrow stale subagent cancellation fix against current `src/tasks/task-registry.ts`, drop the synthetic runtime wrapper in favor of the existing stable tsdown entries, only adjust Discord liveness if current transport-activity monitoring is insufficient, and handle `security.trustModel` as a separate secops-reviewed config/docs change with generated baselines updated.

## Reproduction Assessment

Yes. The stale task bug has a focused reproduction by mocking `killSubagentRunAdmin` to return `{ found: false, killed: false }` for a running subagent task; the PR regressions are reproducible by inspecting the generated wrapper code, security severity change, and config-doc baseline requirements.

## Solution Assessment

No. The PR addresses real symptoms, but the current branch is broad, stale, and introduces blocking regressions; the narrow maintainable solution is a rebased split that preserves current-main runtime and security contracts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Preserve the real subagent runtime initializer:** `scripts/runtime-postbuild.mjs:14-30`
  - body: The synthetic `subagent-registry.runtime.js` wrapper defines `ensureContextEnginesInitialized()` as a local boolean guard, but the real initializer registers the legacy context engine. After postbuild writes this wrapper, packaged subagent flows can lose the default context-engine registration.
  - confidence: 0.87
- **[P1] Keep trusted proxy misconfiguration visible:** `src/security/audit.ts:413-417`
  - body: Demoting `gateway.trusted_proxies_missing` to `info` under `security.trustModel="personal-assistant"` hides an actionable reverse-proxy spoofing warning. A single trusted operator still needs a warning when Control UI is exposed through a proxy without `gateway.trustedProxies`.
  - confidence: 0.8
- **[P2] Update generated config documentation artifacts:** `src/config/zod-schema.ts:458-465`
  - body: Adding top-level `security.trustModel` changes the public config schema surface, but the PR does not update the generated config baseline/hash artifacts required by `pnpm config:docs:check`. The generated baseline will drift until those artifacts are regenerated and committed.
  - confidence: 0.9
- **[P2] Preserve cross-channel action validation:** `src/agents/tools/message-tool.ts:453-460`
  - body: When a current channel is set, this returns only that channel's actions and drops actions from other configured channels. Current main intentionally keeps cross-channel-safe actions in the schema so isolated or cron agents can invoke configured cross-channel actions without validation errors.
  - confidence: 0.74
