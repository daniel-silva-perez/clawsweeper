## Summary

Keep this PR open. Current main still drops the reported Control UI live chat events before runId matching, and the PR is a focused fix with delta/final regression coverage, a clean current PR head, and no blocking review findings from this pass.

## What This Changes

The PR widens Control UI `handleChatEvent` to accept active-run chat events by matching `runId` when `sessionKey` differs, then adds delta/final regression tests and a changelog entry.

## Best Possible Solution

Land this focused PR after normal maintainer approval and branch-protection checks, so the linked bug closes through the merge while preserving strict session-key filtering for non-active-run events.

## Reproduction Assessment

Yes. A focused unit-level reproduction is `sessionKey: "main"`, `chatRunId: "run-1"`, and a `chat` payload with `sessionKey: "agent:main:main"` plus `runId: "run-1"`; current main returns null at the strict session-key guard, while the PR tests the expected delta and final delivery paths.

## Solution Assessment

Yes. Matching the gateway event to the active client-generated run id is narrower than changing gateway broadcast behavior or broadly normalizing every UI session key, and the PR keeps mismatched non-active events filtered out.
