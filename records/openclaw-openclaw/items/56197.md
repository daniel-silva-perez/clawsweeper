## Summary

Keep open. Current main still does not prove the reported Feishu plus Z.AI/AutoGLM auth failure fixed: runtime-auth retry is gated on prepared runtime-auth state, Z.AI remains registered as API-key auth without a runtime exchange hook, and the Feishu dispatch path can still deliver an embedded-run error payload whose formatter preserves a plain `HTTP 401: Invalid token` message.

## What This Changes

The issue requests Feishu embedded-agent auth failures from the Z.AI/AutoGLM route to refresh/retry or show safe re-authentication copy instead of returning raw HTTP 401 text.

## Best Possible Solution

Add a narrow auth recovery path that covers the reported Z.AI/AutoGLM credential shape, either through provider-owned `prepareRuntimeAuth` support for the short-lived token exchange or a generic one-shot credential re-resolution retry for auth-classified 401s. The fallback user copy should translate provider auth failures into clear re-authentication guidance rather than status-wrapper text.

## Reproduction Assessment

Unclear. The issue gives a concrete Feishu plus Z.AI/AutoGLM token-expiry path and current source shows the raw 401 reply remains plausible, but there is no live token-expiry run or focused current-main regression test for this exact route.

## Solution Assessment

Unclear. The existing runtime-auth retry hook is a good solution for providers that implement `prepareRuntimeAuth`, but it is not sufficient for the reported Z.AI API-key route; maintainers should choose the provider/core boundary before changing auth semantics.
