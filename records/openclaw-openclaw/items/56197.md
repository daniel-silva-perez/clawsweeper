## Summary

Keep open. Current main has a generic runtime-auth retry path, but it only runs when the provider has an active runtime auth state; the bundled Z.AI provider path in this report is API-key based and does not register a runtime auth exchange hook, and the generic HTTP error formatter can still produce `HTTP 401: Invalid token` for non-JSON status errors.

## What This Changes

The issue requests that Feishu embedded-agent replies recover from expired AutoGLM/ZAI JWT auth by refreshing and retrying, or at least avoid sending the raw HTTP 401 error to the Feishu user.

## Best Possible Solution

Add a narrow embedded-run auth recovery path that refreshes or re-resolves the relevant provider credential before surfacing a 401, retries once only when the request is safe to replay, and adds Feishu/ZAI-style regression coverage. At minimum, generic auth failures should be converted to user-friendly re-authentication copy instead of raw HTTP text.

## Reproduction Assessment

Unclear. The report has concrete steps and the current source shows a plausible path to the raw 401, but I did not have a live Feishu + AutoGLM/ZAI setup to reproduce the exact token-expiry timing on current main.

## Solution Assessment

Unclear. The current runtime-auth retry design is the right seam for providers with `prepareRuntimeAuth`, but it does not cover the reported ZAI/AutoGLM path; maintainers should choose between a provider-specific runtime auth hook, a generic config/auth re-resolution retry, and improved auth-error copy.
