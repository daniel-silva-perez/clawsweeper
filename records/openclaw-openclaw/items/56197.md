## Summary

Keep open. Current main still does not prove the reported Feishu plus Z.AI/AutoGLM expired-token path fixed: the runtime-auth retry only runs when prepared runtime-auth state exists, Z.AI is still registered as ordinary API-key auth without a runtime exchange hook, and the Feishu path still dispatches embedded-run final/error payloads through the generic reply pipeline. The newer main changes since `v2026.4.27` are adjacent lifecycle, empty-message, and formatter work, not the missing Z.AI/AutoGLM auth recovery.

## What This Changes

The issue asks the Feishu embedded-agent path using Z.AI/AutoGLM to refresh and retry expired-token HTTP 401 failures, or at least avoid returning raw `HTTP 401: Invalid token` text to the Feishu user.

## Best Possible Solution

Add a narrow auth recovery path for the reported Z.AI/AutoGLM credential shape, preferably provider-owned if the short-lived token exchange is Z.AI/AutoGLM-specific. The residual failure path should classify auth 401s into clear re-authentication guidance without exposing raw token material or status-wrapper text to Feishu users.

## Reproduction Assessment

Unclear. The issue gives concrete steps and logs, and current source shows the no-refresh path for plain Z.AI API-key auth, but there is no live Feishu plus expired AutoGLM token run or focused current-main regression test for this exact route.

## Solution Assessment

Unclear. The existing prepared-runtime-auth retry is a good solution for providers that implement `prepareRuntimeAuth`, but it is not sufficient for the reported Z.AI API-key route; maintainers should choose the provider/core boundary before changing auth semantics.
