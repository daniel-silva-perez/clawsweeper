## Summary

Keep open. Current main still only retries auth failures through the prepared runtime-auth state path, while the reported Z.AI/AutoGLM route is registered as API-key auth without a `prepareRuntimeAuth` hook. The Feishu dispatch path can still deliver embedded-run error payloads, and the shared formatter can still render a plain status-prefixed provider failure as `HTTP 401: Invalid token`, so the observable user-facing bug is not proven fixed.

## What This Changes

The issue requests that Feishu embedded-agent runs recover from expired Z.AI/AutoGLM credentials by refreshing and retrying on HTTP 401, or at least avoid replying to the Feishu user with the raw provider error.

## Best Possible Solution

Add a narrow auth recovery path that covers the reported Z.AI/AutoGLM credential shape: either provider-owned `prepareRuntimeAuth` support for the short-lived token exchange or a generic one-shot credential re-resolution retry for auth-classified 401s. The final user-facing fallback should translate provider auth failures into clear re-authentication copy, with regression coverage for Feishu delivery and a Z.AI-style auth failure that lacks prepared runtime-auth state.

## Reproduction Assessment

Unclear. The issue provides concrete steps and current source shows a plausible path to the raw 401 reply, but there is no live Feishu plus AutoGLM/Z.AI token-expiry reproduction or focused current-main regression test for this exact route.

## Solution Assessment

Unclear. The existing runtime-auth retry hook is a good solution for providers that implement `prepareRuntimeAuth`, but it does not cover the reported API-key Z.AI/AutoGLM path; maintainers should choose the owner boundary before a fix PR changes auth semantics.
