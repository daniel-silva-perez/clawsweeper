## Summary

Keep open. Current main still has only the runtime-auth retry path for providers that establish `runtimeAuthState`, while the reported Z.AI/AutoGLM route is API-key based and the Z.AI provider registration does not expose a `prepareRuntimeAuth` hook. The generic error formatter can still render plain status-prefixed failures as `HTTP 401: Invalid token`, so the observable Feishu user-facing problem is not proven fixed.

## What This Changes

The issue asks for Feishu embedded-agent runs through Z.AI/AutoGLM to refresh or re-resolve expired provider credentials on HTTP 401 and avoid replying to Feishu users with raw `HTTP 401: Invalid token` text.

## Best Possible Solution

Add a narrow embedded-run auth recovery path that either gives the Z.AI/AutoGLM route a refreshable runtime-auth hook or re-resolves the provider credential once on an auth-classified 401 before surfacing failure. The user-facing fallback should convert raw provider auth text into clear re-authentication copy, with regression coverage around Feishu delivery and Z.AI-style auth failures.

## Reproduction Assessment

Unclear. The report has concrete steps and current source shows a plausible path to the raw 401 reply, but there is no live Feishu plus AutoGLM/Z.AI token-expiry reproduction or focused regression test proving the exact timing on current main.

## Solution Assessment

Unclear. The existing runtime-auth retry seam is appropriate for providers with `prepareRuntimeAuth`, but it does not cover the reported Z.AI/AutoGLM API-key path; maintainers should choose between a provider-owned runtime auth hook, a generic credential re-resolution retry, and improved auth-error copy.
