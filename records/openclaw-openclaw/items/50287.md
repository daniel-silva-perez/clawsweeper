## Summary

Keep open. Current main still accepts the disputed `audio` and `video` values in configured provider model inputs through schema, public types, generated schema, and configured-catalog normalization, while docs and runtime checks remain centered on text/image semantics; resolving this requires a maintainer contract decision rather than cleanup closure.

## What This Changes

The issue asks to add CI-enforced validation that rejects unsupported `models.providers.*.models[*].input` modality configurations, with targeted tests and docs updates.

## Best Possible Solution

Define one explicit configured-model input contract and enforce it consistently across source schema, exported types, generated schema, catalog normalization, tests, and docs: either reject unsupported values such as `audio`/`video`, or implement and document their runtime meaning so CI can validate them intentionally.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists on current main: a configured provider model with `input: ["audio"]` or `input: ["video"]` is accepted by the schema/type/generated-schema path and preserved by configured-catalog normalization.

## Solution Assessment

Unclear. Schema narrowing would match the current docs but is not clearly the best fix until maintainers choose whether to break the public config contract or formalize audio/video as supported configured-model metadata.
