## Summary

Keep open. Current main still does not implement the requested guardrail: configured provider model inputs still accept `audio` and `video` through schema/type/generated-schema validation and an explicit configured-catalog preservation test, while docs and runtime capability checks still describe/use text and image semantics. Tightening this now would change a public config contract, so this needs maintainer API/product direction before a fix PR chooses the behavior.

## What This Changes

The issue requests schema validation, tests, CI coverage, and docs so unsupported `models.providers.*.models[*].input` modalities are rejected before deployment.

## Best Possible Solution

Define one explicit configured-model input contract and enforce it consistently across source schema, exported types, generated schema, catalog normalization, tests, and docs: either reject unsupported values such as `audio`/`video`, or implement and document their runtime meaning so they are no longer silently accepted metadata.

## Reproduction Assessment

Yes. A static reproduction path is clear on current main: a configured provider model with `input: ["audio"]` or `input: ["video"]` is accepted by the schema/type/generated-schema path, and no later validation check rejects those values.

## Solution Assessment

Unclear. Directly narrowing the schema to `text`/`image` would match the docs and runtime checks, but it would also change a public config/type contract and contradict an explicit preservation test; a maintainer should first choose whether to reject or formalize `audio`/`video`.
