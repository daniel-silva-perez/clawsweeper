## Summary

Keep open. Current main still does not implement the requested guardrail: configured provider model inputs accept `audio` and `video`, validation relies on that schema, and the configured catalog path has an explicit test preserving those values. Because rejecting them would narrow a public config/schema/type contract, this needs maintainer product/API direction before a fix PR should choose the behavior.

## What This Changes

The issue requests schema validation, tests, CI coverage, and docs so unsupported `models.providers.*.models[*].input` modalities such as `audio` are rejected before deployment.

## Best Possible Solution

Define one configured-model input contract and enforce it consistently across source schema, exported types, generated schema, catalog normalization, tests, and public docs: either reject unsupported values such as `audio`/`video`, or implement and document their runtime meaning.
