## Summary

Keep open. Current main and the latest release still accept and preserve `audio` and `video` in configured provider model inputs, while operator docs and normal model-turn capability checks only describe text/image behavior. The requested guardrail is real, but changing it now would reverse a shipped public config/type contract and an explicit preservation test, so this needs maintainer product/API direction rather than conservative cleanup closure.

## What This Changes

The issue requests validation guardrails for `models.providers.*.models[*].input` so unsupported modalities fail before deployment, with targeted tests, CI coverage, and docs updates.

## Best Possible Solution

Define one explicit configured-model input contract and enforce it consistently across source schema, exported types, generated schema, catalog normalization, runtime capability checks, tests, and docs: either reject unsupported values such as `audio`/`video`, or implement and document their runtime meaning so they are no longer silently accepted metadata.
