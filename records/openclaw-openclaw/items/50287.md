## Summary

Keep open. Current main still does not implement the requested guardrail: `models.providers.*.models[*].input` accepts `audio` and `video` in schema/types/generated schema, config validation relies on that schema, and a current provider-catalog test explicitly preserves configured audio/video modalities. Docs and runtime paths still primarily describe/use text/image model-input behavior, so this needs maintainer product-contract review before an automated tightening PR.

## Best Possible Solution

Keep this open for maintainer contract review. If `models.providers.*.models[*].input` is intended only for chat-turn native inputs, tighten the schema/type/generated schema/provider-catalog handling and add focused config-validation tests for a valid text/image case plus an invalid audio/video case. If audio/video are intended model metadata now, update docs and runtime capability handling so those modalities have a defined meaning instead of silently passing config validation.
