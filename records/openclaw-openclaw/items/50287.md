## Summary

Keep open. Current main does not satisfy the requested guardrail: the core config schema still accepts `audio` and `video` for `models.providers.*.models[*].input`, so an unsupported `audio` modality would not fail early during config validation. The docs and model-catalog normalization point toward the supported chat-model input set being text/image, so this remains actionable schema, test, and docs alignment work rather than a duplicate close.

## Best Possible Solution

Keep this issue open and implement the guardrail in core config validation: tighten `ModelDefinitionSchema.input` and `ModelDefinitionConfig.input` to the supported deployment modalities, or explicitly add/document real audio/video runtime support before allowing those values. Add targeted config-validation tests for invalid `audio` and valid text/image cases, and update the model/fallback docs so CI and contributor expectations match the schema.
