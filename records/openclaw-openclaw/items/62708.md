## Summary

Keep open. Current main has not implemented the requested cleanup: the legacy field is still declared, still written and cleared by CLI-session helpers, still preserved across session flows, and still read by Claude CLI history import compatibility code. The request remains concrete, but removal now requires a maintainer call because current tests explicitly protect legacy session resume behavior.

## What This Changes

The issue requests removing the legacy `claudeCliSessionId` session-store field and its compatibility reads, writes, passthroughs, and tests now that provider-scoped CLI session fields exist.

## Best Possible Solution

Decide whether OpenClaw is ready to retire this persisted-session compatibility field. If yes, remove `claudeCliSessionId` from the session type, CLI helper fallback and dual-write paths, session preservation/reset patches, Claude CLI history fallback, and tests while keeping `cliSessionBindings` and `cliSessionIds` behavior intact; if old-session resume compatibility still matters, keep the fallback and document why.

## Reproduction Assessment

Not applicable as a runtime bug, but the current state is reproducible with high confidence by statically inspecting current main. `rg claudeCliSessionId` and the focused file reads show active type, read, write, preservation, and test coverage for the legacy field.

## Solution Assessment

Unclear without maintainer compatibility judgment. The requested deletion is the narrow cleanup if the project accepts dropping old Claude-only session resume support, but the safer alternative is to keep the fallback until a deliberate deprecation or doctor migration path is chosen.
