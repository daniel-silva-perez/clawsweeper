## Summary

Keep this PR open. Current main still does not route Feishu lifecycle or transport activity into the gateway health snapshot, and the latest PR context still appears incomplete because it does not modify the Feishu channel entrypoint or account monitor where the status callback must be threaded.

## Best Possible Solution

Keep this PR open for maintainer review or contributor revision. The best path is a Feishu-owned implementation that passes ctx.setStatus from channel.ts through monitorFeishuProvider and monitorSingleAccount into both transport modes, publishes connected/mode/lastConnectedAt on start, connected:false plus disconnect/error details on abort/error/shutdown, updates lastTransportActivityAt from real transport activity, and adds focused Feishu tests proving the runtime snapshot drives the gateway health policy. Coordinate with #39487 and #39489 so only one canonical Feishu health-status fix lands.
