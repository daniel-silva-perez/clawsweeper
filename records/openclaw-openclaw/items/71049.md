## Summary

Current main still does not expose an MS Teams config option to disable the personal-chat streaminfo/informative status path. Personal chats still create a TeamsHttpStream, send an informative update at reply start, register partial reply streaming whenever that stream exists, and the strict config schema only accepts typingIndicator/blockStreaming as separate controls. The issue is recent, valid, and narrow enough for one focused MS Teams plugin PR.

## Best Possible Solution

Keep this issue open and implement a narrow MS Teams plugin config option, defaulting to current behavior, that lets personal chats disable streaminfo/informative status updates and fall back to normal final message delivery with native typing keepalive when typingIndicator is enabled. The change should stay in the MS Teams plugin/config surface and explicitly preserve the long-tool-chain keepalive path.

## Clownfish Work Prompt

Implement a narrow MS Teams plugin-owned config option for this issue: let `channels.msteams` disable the Copilot-style personal-chat streaminfo/informative status path while defaulting to current behavior. Scope the fix to the MS Teams plugin/config surface. Do not add owner-specific policy to core.

Expected behavior: with the new option unset/default, personal chats continue using `TeamsHttpStream` and informative status as today. With the option disabled, personal chats should not create/use the streaminfo controller, should not call `sendInformativeUpdate`, should not register `replyOptions.onPartialReply` for Teams streaminfo, and should deliver one normal final reply. If `typingIndicator` is enabled or unset, preserve the native typing keepalive for personal chats so long tool chains still keep the TurnContext alive; if `typingIndicator: false`, do not force native typing.

Tests should cover default behavior, opt-out behavior in personal chats, no stream partial callback when disabled, native typing keepalive behavior while disabled, and strict schema acceptance/rejection for the new key. Keep the changelog entry user-facing with proper attribution to the reporter, not Codex/OpenClaw maintainers.
