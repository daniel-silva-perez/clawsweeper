## Summary

Keep this PR open. Current main does not implement `channels.whatsapp.mediaLocalRoots`, does not validate that field in the WhatsApp schema, and does not merge WhatsApp channel/account configured roots into outbound WhatsApp media delivery. The existing generic media-root plumbing can carry roots once supplied, but the PR’s requested WhatsApp-scoped configuration path is still missing.

## Best Possible Solution

Keep the PR open as a plausible focused implementation candidate. The best path is to update the PR so WhatsApp channel-level and account-level `mediaLocalRoots` are typed, schema-validated, documented/generated, and merged into outbound context with correct `defaultAccount` resolution when `accountId` is omitted, plus regression tests for channel-level, account-level, default-account, and non-WhatsApp behavior.

## Risks / Open Questions

- Security review: the PR adds an operator-controlled local filesystem allowlist for outbound WhatsApp media reads. That is a legitimate feature, but it should require clear absolute-path validation or equivalent runtime guarantees, reject unsafe roots consistently, and include tests for relative/empty/root entries before merge.
- The current PR discussion notes that account-level roots may be skipped when WhatsApp resolves `defaultAccount` while `accountId` is omitted; merging without resolving that would leave a common multi-account path broken.
- The PR appears to need generated config/docs baseline updates, so it is not merge-ready as-is even though the core idea is focused and additive.
