## Summary

Keep open. Current main still lacks the requested WhatsApp-scoped outbound media roots, but this PR should not merge as-is because it hard-codes WhatsApp config policy into core outbound delivery, leaves generated bundled channel metadata stale, accepts `file://` root entries that downstream local-root enforcement does not normalize, and is currently dirty against main.

## What This Changes

The PR adds WhatsApp channel/account `mediaLocalRoots` config, schema validation, outbound media-root merging, docs baseline hash updates, and regression coverage for channel, explicit-account, and default-account roots.

## Best Possible Solution

Keep the feature direction, but resolve WhatsApp-configured roots in the WhatsApp-owned outbound adapter or a generic documented plugin seam, normalize or reject `file://` root entries consistently with the local media loader, regenerate bundled channel metadata plus docs baselines, and rebase the branch onto current main.

## Reproduction Assessment

Yes. A focused test can configure `channels.whatsapp.mediaLocalRoots: ["file:///tmp/wa-media"]`, send a local media path under that directory, and observe that the accepted root is not treated as an allowed local root because root entries are not converted from file URLs. Current-main source checks also reproduce that the requested config field is still absent.

## Solution Assessment

No. The requested feature is plausible, and the default-account coverage is a useful improvement, but the current implementation is not the narrowest maintainable fix because it puts WhatsApp-specific config resolution in core and leaves generated channel metadata plus file-url root handling incomplete.
