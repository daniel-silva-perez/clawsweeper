## Summary

Keep open. Current main does not implement configured WhatsApp media local roots, and the PR is useful but not ready as written because it misses implicit default-account account roots, generated config metadata, and stricter validation for a local-file allowlist setting.

## What This Changes

The PR adds WhatsApp channel/account `mediaLocalRoots` config, validates the new field in the WhatsApp schema, merges configured roots into outbound WhatsApp delivery context, and adds channel-level regression coverage.

## Best Possible Solution

Keep the PR open and revise it so WhatsApp root resolution follows the same default-account contract as the send path, generated config docs/metadata are updated, and `mediaLocalRoots` entries are validated or normalized consistently before they reach the media loader.

## Reproduction Assessment

Yes. A focused unit test can configure `channels.whatsapp.defaultAccount: "work"` with `accounts.work.mediaLocalRoots`, call WhatsApp `deliverOutboundPayloads` for media without `accountId`, and assert the outbound sender receives the account root; the PR's `params.accountId` guard would omit it.

## Solution Assessment

No. The feature direction is plausible, but the implementation should resolve the effective WhatsApp account before merging account roots and include generated config artifacts plus root-entry validation before merge.

## Review Findings

Full review comments:

- **[P2] Resolve implicit account roots before merging:** `src/infra/outbound/deliver.ts:271-274`
  - body: `sendMessageWhatsApp` resolves an omitted `accountId` through the configured WhatsApp default account, but this new lookup only reads `accounts[params.accountId]` when an explicit account was passed. Default-account setups that place `mediaLocalRoots` under `accounts.<id>` will still miss those roots and fail local media sends.
  - confidence: 0.91
- **[P2] Regenerate config metadata for the new field:** `src/config/zod-schema.providers-whatsapp.ts:58`
  - body: Adding `channels.whatsapp.mediaLocalRoots` changes the public config schema, but the PR does not update generated config metadata or `docs/.generated/config-baseline.sha256`. The config drift gate will fail and generated config surfaces will not expose the new setting.
  - confidence: 0.86
- **[P2] Validate configured media roots:** `src/config/zod-schema.providers-whatsapp.ts:58`
  - body: This setting expands the local media read allowlist, but `z.array(z.string())` accepts empty and relative entries that can resolve to unintended directories later. Require non-empty absolute paths or normalized `file://` paths before accepting the config.
  - confidence: 0.78
