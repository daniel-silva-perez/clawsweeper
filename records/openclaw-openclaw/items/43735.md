## Summary

Keep open. Current main supports workspace skill scanning, but the persisted skills snapshot refresh path still depends on process-local version state that can restart at 0, so stale version-0 snapshots can be reused after a process restart. The related open draft maintainer PR #71497 targets this root cause, so this should stay open for that PR or an equivalent fix rather than be closed or queued as a competing repair.

## What This Changes

The issue reports that valid workspace skills are only partially listed in `<available_skills>`, leaving agents unaware of skills that exist under the workspace skills directory.

## Best Possible Solution

Land #71497 or an equivalent restart-safe snapshot invalidation fix that rebuilds persisted skill snapshots after gateway and CLI restarts when the effective skill inventory changed, while preserving source precedence, eligibility gates, containment checks, and prompt-size limits.

## Reproduction Assessment

Yes. A high-confidence static reproduction path is a persisted session `skillsSnapshot` with version 0, workspace skills changed while OpenClaw is stopped, then a restart where `getSkillsSnapshotVersion()` is still 0 and both gateway and CLI refresh checks treat the old snapshot as current.

## Solution Assessment

No, current main is not the solved state because the version-only refresh predicate is not restart-safe for version-0 snapshots. The safer path is durable inventory-based invalidation or an equivalent startup version contract, not forcing every skill into the prompt unboundedly.
