## Summary

Keep open. Current main documents and scans workspace skills, but gateway and CLI session refresh still depend on process-local snapshot versions, so a persisted version-0 skills snapshot can survive a restart even when the effective skill inventory changed. An open draft maintainer PR already targets this restart-refresh path, so this should stay in maintainer follow-up rather than be closed or queued as a competing repair.

## What This Changes

The issue requests reliable discovery and surfacing of valid workspace skills in agent context instead of a partial or stale available-skills list.

## Best Possible Solution

Keep this open until #71497 or an equivalent fix lands and proves gateway, CLI, and cron skill snapshots rebuild after restart when the effective skill inventory changes. The durable end state should use restart-safe invalidation, such as an inventory fingerprint or equivalent version contract, while preserving source precedence, containment checks, eligibility gates, and prompt-size limits.

## Reproduction Assessment

Yes. A high-confidence static reproduction path is an existing persisted skillsSnapshot with version 0, workspace skills changed while OpenClaw is stopped, then a restart where getSkillsSnapshotVersion() is still 0 and the gateway or CLI refresh check treats the old snapshot as current.

## Solution Assessment

No, current main is not the best solved state because version-only refresh is not restart-safe for version-0 snapshots. The safer solution is durable inventory-based invalidation or an equivalent startup version contract, not unboundedly forcing every skill into the prompt.
