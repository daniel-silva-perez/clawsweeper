## Summary

Keep open. Current main still has the restart-safe invalidation gap for persisted skillsSnapshot data: workspace skills are loaded when a snapshot is rebuilt, but persisted session snapshots are considered fresh using only in-memory version counters and skill filters. The linked diagnosis in #44898 and the attempted fix in #48089 remain relevant because #48089 was closed unmerged, and current main does not persist or compare a deterministic skill inventory fingerprint.

## Best Possible Solution

Keep this issue open and fix the current skillsSnapshot contract. The likely maintainer-grade path is to persist and compare a deterministic inventory fingerprint for the visible skill roots and relevant inputs: SKILL.md metadata/content identity, root paths, effective skill filter, gating/config inputs, remote eligibility inputs where applicable, and prompt limit settings. Add regression coverage for an existing session whose version-0 snapshot survives a process restart after a workspace skill is added while no watcher is active. Also preserve the documented filtering and truncation behavior so users get clearer diagnostics without turning every skill directory into a guaranteed prompt entry.

## Risks / Open Questions

- A fix should not promise that every directory under skills/ always appears in <available_skills>; the concrete remaining issue is restart-safe invalidation or diagnostics for persisted session snapshots.
