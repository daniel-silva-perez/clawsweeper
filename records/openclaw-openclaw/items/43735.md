## Summary

Keep open. Current main still has the workspace skills loader and prompt-budget improvements, but the persisted session snapshot is invalidated by process-local version/filter state rather than a durable skills inventory fingerprint, and an open draft maintainer PR is already aimed at the same restart-stale snapshot path.

## What This Changes

The issue reports that valid workspace skills under `~/.openclaw/workspace/skills/` are only partially listed in the agent `<available_skills>` context.

## Best Possible Solution

Keep this open until #71497 or an equivalent fix lands and proves gateway, CLI, and cron skill snapshots rebuild after restart when the effective workspace skill inventory changes. A durable implementation should compare a deterministic inventory fingerprint while preserving source precedence, realpath containment, allowlists, eligibility gates, and prompt-size limits.

## Reproduction Assessment

Yes. The high-confidence static reproduction path is an existing session with a persisted version-0 `skillsSnapshot`, skills changed while the process is stopped, then a restart where `getSkillsSnapshotVersion()` returns 0 and the gateway/CLI refresh checks treat version 0 as current.

## Solution Assessment

No. The current version-only refresh is not the narrowest durable fix for restart-time inventory drift; forcing every skill unbounded into the prompt would also be wrong because prompt budgets are intentional, so the safer direction is inventory-based invalidation plus the existing compact/truncation behavior.
