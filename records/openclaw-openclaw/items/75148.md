## Summary

Keep this PR open. It addresses a real current-main fallback gap, but the current PR head still computes sandbox-based message-tool availability from the dispatch session key instead of the runtime policy session key used by the embedded runner, so one visible-reply fallback path can remain misclassified.

## What This Changes

The PR adds dispatch-side group, sandbox, and subagent tool-policy checks for `message` availability, plus regression tests for group and sandbox policies that remove the `message` tool.

## Best Possible Solution

Dispatch should determine `messageToolAvailable` from the same effective policy inputs as the embedded runner: group policy from the verified dispatch/group context, sandbox and subagent policy from the runtime policy session key, and focused tests for group, sandbox-all, and non-main direct-channel sandbox shapes.

## Reproduction Assessment

Yes. The bug is reproducible by tracing a direct channel turn that dispatches through `agent:main:main` while `resolveRuntimePolicySessionKey` derives a sandboxed peer key under `sandbox.mode: "non-main"`; the PR currently checks sandbox status against the dispatch key instead.

## Solution Assessment

No as written. Adding group, sandbox, and subagent policy checks is the right direction, but the sandbox/subagent portion should use the runner's runtime policy session key or a shared effective-availability helper.
