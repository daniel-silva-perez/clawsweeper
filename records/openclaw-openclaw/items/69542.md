## Summary

Keep open. Current main still lacks `acp.fallbacks` and ACP backend failover, so this PR is not obsolete or implemented. The PR is also not ready to merge as-is because its ACP turn-loop changes have unresolved correctness concerns and it mixes unrelated install-smoke/runtime-dependency staging changes into a security-sensitive execution/package-resolution surface.

## What This Changes

The PR adds an `acp.fallbacks` config field and wraps ACP `runTurn` in a backend failover loop, while also modifying install-smoke JSON parsing and bundled plugin runtime dependency staging.

## Best Possible Solution

Keep this PR open and request changes. The ACP failover work should be narrowed to the ACP control-plane/config/docs surface, derive candidates from resolved session metadata, close or retire old handles before switching, avoid failover after output or tool effects start, define init-stage `ACP_SESSION_INIT_FAILED` behavior, use precise retry classification, preserve failure bookkeeping/observability, add focused manager tests, split the unrelated install-smoke/runtime-deps work, and correct the wrong closing reference.

## Risks / Open Questions

- The PR branch can retry on another backend after visible output or tool-call events have already started, which can duplicate streamed text or side effects.
- Candidate backend selection is derived from global config before session metadata, which can regress binding/agent/runtime metadata and auto-selected backend behavior.
- Clearing cached runtime state before switching backends without closing or retiring the old handle can leak persistent backend sessions and concurrency accounting.
- Broad substring matching such as `rate` can misclassify unrelated `ACP_TURN_FAILED` messages as failover-worthy.
- Intermediate backend failures may be undercounted in observability when failover later succeeds.
- The unrelated workflow/runtime-deps edits touch code-execution and package-resolution paths and should not ride along with the ACP feature.
- The PR body references unrelated #4698, which could close the wrong historical issue if merged unchanged.
