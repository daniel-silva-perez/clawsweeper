## Summary

Keep this PR open. Current main still only emits `message:received` and `message:sent` internal hooks; it does not drain hook-produced `event.messages` or deliver them back to a user. The linked bug remains unresolved on main, and the maintainer successor #41375 is open but unmerged, so this PR is still an implementation candidate that needs consolidation rather than cleanup closure.

## Best Possible Solution

Keep this PR or an explicit maintainer successor open until #39688 is resolved. The best path is to consolidate with #41375, decide and document whether `message:sent` is replyable, implement a shared reply-drain helper only where a safe target exists, preserve session/media context, make hook-loop suppression durable for any replayable delivery path, and add regression tests plus hooks docs updates for the final contract.

## Risks / Open Questions

- The PR's `message:received` route-reply path omits `sessionKey`, so hook replies can diverge from normal agent-scoped response-prefix behavior.
- The PR's recursive `message:sent` delivery omits the originating `session`, weakening agent-scoped media-root preservation if the surface later handles more than text.
- The proposed in-memory `skipMessageHooks` flag is not represented in the current queue/recovery schema; queued or replayed hook-generated replies could re-emit message hooks unless the final design guarantees the path is unqueued or persists suppression.
- The desired `message:sent` reply contract is product-sensitive. Maintainer PR #41375 explicitly limits auto-delivery to replyable surfaces and does not make lifecycle-only hooks reply-capable.
- The PR was authored against an older base in the provided context and should be rebased or consolidated with the maintainer successor before merge evaluation.
