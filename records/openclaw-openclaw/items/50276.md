## Summary

Keep open. Current main still lacks the outbound message-action mapping from parsed `replyToCurrent` to a Slack `threadId`, so the PR is not obsolete, but the current PR head is not mergeable as-is because it redeclares an imported resolver, adds tests that call undefined helpers, and changes Slack `replyToMode="off"` behavior without resolving the documented plugin policy. Earlier review discussion found the scope useful, but the later blockers remain concrete.

## What This Changes

The PR adds Slack-specific `[[reply_to_current]]` handling in outbound message-action sends and adds same-channel and cross-channel Slack threading tests.

## Best Possible Solution

Rework the PR on current main through the existing outbound threading seam, remove the duplicate helper, add runnable tests for same-channel, cross-channel, explicit `threadId`/`replyTo` precedence, and first-mode accounting, and update Slack docs/plugin metadata if `off` mode is intentionally changed.

  - body: `message-action-runner.ts` already imports `resolveAndApplyOutboundThreadId` from `./message-action-threading.js`. Declaring another function with the same name in this module is invalid TypeScript/ESM and prevents the runner from compiling or loading.

  - body: The added tests call `mockHandledSendAction()` and `runThreadingAction(...)`, but this test file does not define or import those helpers. As written, the new coverage fails before it can validate the Slack behavior.
  - confidence: 0.94
- **[P2] Align the off-mode override with Slack policy:** `src/infra/outbound/message-action-runner.ts:117-120`
  - body: This helper forces `replyToMode: "all"`, so `[[reply_to_current]]` threads even when Slack `replyToMode` is `off`.   - confidence: 0.86
