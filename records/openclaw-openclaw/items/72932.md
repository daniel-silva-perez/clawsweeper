## Summary

Keep this PR open. The branch is not safe to merge or close as implemented: current main already has a safer version of the fresh `agentId` + `sessionId` routing pattern, but the PR branch still drops the requested agent in the gateway path, swallows background startup-maintenance failures, and carries stale Bonjour-core changes after that code moved out of current main.

## What This Changes

The PR changes CLI and gateway session-key routing for `agentId` plus `sessionId`, adds regression tests, defers selected gateway startup work behind `OPENCLAW_GATEWAY_FAST_CONFIG`, and changes Bonjour unhandled-rejection cleanup lifetime.

## Best Possible Solution

Update the branch against current main, keep or add only the missing regression coverage for fresh agent-scoped session IDs, implement any still-wanted fast-startup behavior against the current startup architecture with logged background failures, and handle Bonjour only in its current plugin-owned location if the cleanup still applies.

## Reproduction Assessment

Yes. The PR regression is reproducible by static call-path inspection: send gateway `agent` params with both `agentId` and `sessionId`, observe `requestedSessionKey` becomes undefined and no `agentId` is passed to ingress on the PR branch.

## Solution Assessment

No. The current PR is not the best fix because it solves the CLI half but regresses the gateway half; the safer solution is the current-main pattern of deriving an explicit session-id key or forwarding a validated matching `agentId`, plus explicit logging for deferred startup failures.

Full review comments:

- **[P1] Preserve agent routing for fresh session-id requests:** `src/gateway/server-methods/agent.ts:506`
  - body: When `agentId` and `sessionId` are both present, this branch clears `requestedSessionKey` and then dispatches `agentCommandFromIngress` without `agentId`. That lets the runner resolve the fresh session against the default agent/store, so a request for `mybot` can run as `main`; pass the validated agent id through or derive an explicit `agent:<id>:explicit:<sessionId>` key.
  - confidence: 0.91
- **[P2] Log background startup maintenance failures:** `src/gateway/server-startup-plugins.ts:72-73`
  - body: In fast gateway mode, `Promise.allSettled` consumes failures from channel maintenance and session migration, but the callback ignores the results. Operators get no warning if either background task fails, so inspect rejected results and log them before tracing completion.
  - confidence: 0.88
- **[P2] Clean up the Bonjour rejection handler on startup errors:** `src/infra/bonjour.ts:332-333`
  - body: The unhandled-rejection handler is registered before `createCycle()`, but it is only cleaned up from `stop()`. If `createCycle()` or listener setup throws before an advertiser is returned, the catch restores console logging while leaving the process-level handler installed.
  - confidence: 0.84
- **[P3] Restore fast-startup env in the test:** `src/gateway/server-startup-post-attach.test.ts:267`
  - body: This test sets `OPENCLAW_GATEWAY_FAST_CONFIG` and deletes it only on the happy path. If an import or assertion throws first, later tests inherit fast gateway mode; save the previous value and restore it in a `finally`.
  - confidence: 0.83
- **[P3] Measure plugin trace stages from their own start:** `src/gateway/server-startup-plugins.ts:82`
  - body: Several trace calls pass `pluginBootstrapStartedAt` into a helper whose `stage_ms` is computed from the supplied start time, so labels like `initSubagentRegistry` and config resolution report elapsed-since-bootstrap instead of stage duration. Capture a local timestamp for each stage or rename the metric.
  - confidence: 0.78
