## Summary

Keep this PR open for maintainer or author follow-up. The session-routing problem is partly covered on current main, but this PR is dirty against main, still contains blocking gateway and operational findings, and includes fast-startup behavior that is not implemented in current main.

## What This Changes

The PR changes CLI and gateway session-key routing for `agentId` plus `sessionId`, adds regression tests, defers selected gateway startup work behind `OPENCLAW_GATEWAY_FAST_CONFIG`, and changes Bonjour unhandled-rejection cleanup lifetime.

## Best Possible Solution

Rebase or split the work against current main: preserve the current-main session-id routing pattern, decide whether fast startup should be introduced in the current startup architecture with explicit failure logging, and drop or reapply any Bonjour cleanup only in the current `extensions/bonjour` implementation.

## Reproduction Assessment

Yes. The PR regression is reproducible by static call-path inspection: send gateway `agent` params with both `agentId` and a fresh `sessionId`, and the PR head clears `requestedSessionKey` while dispatching without `agentId`.

## Solution Assessment

No. The current PR is not the best complete solution because the gateway half regresses agent selection, the fast-startup branch hides background failures, and the Bonjour change targets a path that has moved on main.

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve agent routing for fresh session-id requests:** `src/gateway/server-methods/agent.ts:506-510`
  - body: When `sessionId` is present, this branch clears `requestedSessionKey`; the later gateway dispatch sends no `agentId`, so a fresh request for a named agent can run against the default agent/store. Forward the validated agent id or derive an explicit `agent:<id>:explicit:<sessionId>` key here.
  - confidence: 0.91
- **[P2] Log background startup maintenance failures:** `src/gateway/server-startup-plugins.ts:72-73`
  - body: `Promise.allSettled` consumes failures from channel maintenance and session migration, but the callback ignores rejected results. Operators get no warning when deferred startup work fails, so inspect the settled results and log rejected reasons.
  - confidence: 0.88
- **[P2] Clean up the Bonjour rejection handler on startup errors:** `src/infra/bonjour.ts:332-333`
  - body: The unhandled-rejection handler is registered before `createCycle()`, but the outer catch only restores console logging. If `createCycle()` or listener setup throws before an advertiser is returned, the process-level handler leaks for the rest of the process.
  - confidence: 0.84
- **[P3] Restore fast-startup env in the test:** `src/gateway/server-startup-post-attach.test.ts:307`
  - body: This test sets `OPENCLAW_GATEWAY_FAST_CONFIG` and deletes it only on the happy path. If an import or assertion throws first, later tests inherit fast gateway mode; save the previous value and restore it in a `finally`.
  - confidence: 0.83
- **[P3] Measure plugin trace stages from their own start:** `src/gateway/server-startup-plugins.ts:82`
  - body: Several trace calls pass `pluginBootstrapStartedAt` into a helper whose `stage_ms` is computed from the supplied start time, so those stage labels report elapsed-since-bootstrap rather than that stage's duration. Capture a local timestamp for each measured stage.
  - confidence: 0.78
