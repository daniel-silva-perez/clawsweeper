## Summary

Keep open. The core session-routing problem is largely addressed on current main, and the Bonjour cleanup surface has moved into the bundled Bonjour plugin, but this PR still carries an unresolved gateway routing regression and a separate fast-startup behavior change that is not implemented on current main.

## What This Changes

The PR changes CLI and gateway session-key routing for agent plus session-id requests, adds regression tests, defers selected gateway startup maintenance behind a fast-startup environment flag, and adjusts Bonjour unhandled-rejection cleanup lifetime.

## Best Possible Solution

Split or revise the branch so the session-routing fix matches the current-main pattern and only adds missing coverage, then handle fast gateway startup as a separate maintained behavior change with logged background failures and current startup architecture tests. Any Bonjour follow-up should target `extensions/bonjour`, not the removed core path.

## Reproduction Assessment

Yes. The PR regression is reproducible by static call-path inspection: send a gateway `agent` request with both `agentId` and `sessionId` and no `sessionKey`; the PR branch clears the key and does not pass the agent id into ingress.

## Solution Assessment

No. The current PR is not the best way to solve the issue because it mixes a correct session-routing direction with an incomplete gateway implementation and unrelated startup/Bonjour changes; the safer path is to reuse current main's explicit session-key/validated-agent pattern and review fast startup independently.

Full review comments:

- **[P1] Preserve agent routing for fresh session-id requests:** `src/gateway/server-methods/agent.ts:506-510`
  - body: When `agentId` and `sessionId` are both present, this branch clears `requestedSessionKey`, but the later gateway dispatch on the PR branch does not forward `agentId`. Fresh agent-scoped sessions can therefore run against the default agent/store; pass the validated agent id through or derive an explicit agent session key.
  - confidence: 0.91
- **[P2] Log background startup maintenance failures:** `src/gateway/server-startup-plugins.ts:72-73`
  - body: In fast gateway mode, `Promise.allSettled` consumes failures from channel maintenance and session migration, but the callback ignores rejected results. Operators get no warning if startup maintenance failed, so inspect the results and log rejected reasons before tracing completion.
  - confidence: 0.88
- **[P2] Clean up the Bonjour rejection handler on startup errors:** `src/infra/bonjour.ts:332-333`
  - body: The unhandled-rejection handler is registered before the advertiser cycle is created, but the PR only cleans it from `stop()`. If cycle creation or listener setup throws before an advertiser is returned, the process-level handler leaks; clean it in the startup error path too.
  - confidence: 0.84
- **[P3] Restore fast-startup env in the test:** `src/gateway/server-startup-post-attach.test.ts:267`
  - body: The test sets `OPENCLAW_GATEWAY_FAST_CONFIG` and deletes it only on the happy path. If an import or assertion throws, later tests inherit fast gateway mode; save the prior value and restore it in a `finally`.
  - confidence: 0.83
- **[P3] Measure plugin trace stages from their own start:** `src/gateway/server-startup-plugins.ts:82`
  - body: Several trace calls pass the overall bootstrap start time into a helper whose `stage_ms` is computed from the supplied timestamp, so the per-stage metric reports elapsed-since-bootstrap. Capture a local timestamp for each measured stage or rename the metric.
  - confidence: 0.78
