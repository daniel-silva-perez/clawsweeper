## Summary

Keep open. Current main still does not implement Firecrawl browser sign-in, and the discussion already establishes the feature as useful, but this branch is not merge-ready because it bypasses the provider-owned setup seam and has a concrete SecretRef/plaintext credential-storage regression.

## What This Changes

The PR adds a Firecrawl PKCE browser-auth helper with `source=openclaw`, wires it into web-search onboarding, syncs the returned API key into Firecrawl fetch config, and adds onboarding/auth tests.

## Best Possible Solution

Implement Firecrawl browser sign-in inside the Firecrawl plugin as a provider-owned `runSetup`, keep the PKCE helper under `extensions/firecrawl`, persist web-search and web-fetch credentials through plugin-scoped SecretInput/SecretRef-aware paths, and update Firecrawl docs after that lands.

## Reproduction Assessment

Yes for the review findings: the credential-storage bug is reproducible by following the PR success path with `secretInputMode: "ref"`, where the raw browser-auth key is still passed into the fetch sync helper. Current-main behavior is also clear by inspection: no Firecrawl browser-auth path exists yet.

## Solution Assessment

No. The feature direction is useful, but the current patch is not the best solution because provider-specific auth/setup now belongs behind the Firecrawl provider `runSetup` hook and must preserve SecretRef semantics across both search and fetch config.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Honor SecretRef mode before storing browser auth keys:** `src/commands/onboard-search.ts:423`
  - body: `resolveSearchSecretInput` can return an env SecretRef, but this success path still passes raw `auth.apiKey` into the Firecrawl fetch sync helper. A setup running with `secretInputMode: "ref"` would persist the plaintext Firecrawl key even though the operator requested reference-only secret storage.
  - confidence: 0.91
- **[P2] Keep Firecrawl setup behind the provider runSetup hook:** `src/commands/onboard-search.ts:406-414`
  - body: This adds a Firecrawl-specific branch to the shared onboarding command. Current main routes web-search setup through `src/flows/search-setup.ts` and calls each provider's `runSetup`, so this browser-auth flow should live in the Firecrawl provider instead of core setup.
  - confidence: 0.88
- **[P2] Persist Firecrawl fetch keys to plugin config:** `src/commands/firecrawl-browser-auth.ts:126-138`
  - body: The helper writes to legacy `tools.web.fetch.firecrawl.apiKey`, which current docs and doctor migration treat as compatibility input. New setup should update `plugins.entries.firecrawl.config.webFetch.apiKey` or a SecretRef through the Firecrawl provider-owned path.
  - confidence: 0.9
- **[P2] Restore fetch in test teardown:** `src/commands/firecrawl-browser-auth.test.ts:28-30`
  - body: After `vi.unstubAllGlobals()` this immediately re-stubs `fetch`, so the suite leaves a mocked global fetch installed for later files in non-isolated runs. Let `beforeEach` set the stub and keep teardown restorative.
  - confidence: 0.86
- **[P3] Stop the progress spinner on auth exceptions:** `src/commands/firecrawl-browser-auth.ts:104-115`
  - body: Once `prompter.progress` returns, an unexpected throw before either `spin.stop` call falls into the outer catch and leaves the terminal spinner active. Wrap the wait/return block in a local try/finally or catch that stops the spinner.
  - confidence: 0.78
