## Summary

Keep open. The feature is still valid and current main does not provide Firecrawl browser auth during web-search setup, but this branch is not merge-ready because it patches a stale core onboarding path and has credential-storage regressions around SecretRef mode and legacy Firecrawl fetch config.

## What This Changes

The PR adds a Firecrawl PKCE browser-auth helper, wires it into web-search onboarding, syncs the returned API key into Firecrawl fetch config, and adds tests.

## Best Possible Solution

Implement Firecrawl browser sign-in inside the Firecrawl plugin as a provider-owned `runSetup`, keep the PKCE helper under `extensions/firecrawl`, persist only plugin-scoped webSearch/webFetch credentials or SecretRefs as appropriate, and document the browser sign-in path on the Firecrawl docs page after the current-path implementation lands.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Honor SecretRef mode before storing browser auth keys:** `src/commands/onboard-search.ts:421`
  - body: `resolveSearchSecretInput` can return an env SecretRef, but this success path still passes the raw `auth.apiKey` into the Firecrawl fetch sync helper. A setup running with `secretInputMode: "ref"` would persist the plaintext Firecrawl key even though the user requested references only.
  - confidence: 0.91
- **[P2] Keep Firecrawl setup behind the provider runSetup hook:** `src/commands/onboard-search.ts:403-421`
  - body: This adds a Firecrawl-specific branch to the shared onboarding command. Current main has moved web-search setup to `src/flows/search-setup.ts` and calls each provider's `runSetup`, so this should live in the Firecrawl provider instead of hard-coding a bundled provider into core setup.
  - confidence: 0.88
- **[P2] Persist Firecrawl fetch keys to plugin config:** `src/commands/firecrawl-browser-auth.ts:122-138`
  - body: The helper writes to legacy `tools.web.fetch.firecrawl.apiKey`, which current docs and doctor migration treat as compatibility input. New setup should update `plugins.entries.firecrawl.config.webFetch.apiKey` or a SecretRef through the Firecrawl provider-owned path.
  - confidence: 0.9
- **[P2] Restore fetch in test teardown:** `src/commands/firecrawl-browser-auth.test.ts:28-29`
  - body: After `vi.unstubAllGlobals()` this immediately re-stubs `fetch`, so the suite leaves a mocked global fetch installed for later files in non-isolated runs. Let `beforeEach` set the stub and keep teardown restorative.
  - confidence: 0.86
- **[P3] Stop the progress spinner on auth exceptions:** `src/commands/firecrawl-browser-auth.ts:104-118`
  - body: Once `prompter.progress` returns, an unexpected throw before either `spin.stop` call falls into the outer catch and leaves the terminal spinner active. Wrap the wait/return block in a local try/finally or catch that stops the spinner.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff adds no new dependency or CI surface, but it has a concrete credential-handling regression around plaintext Firecrawl API key persistence.

Concerns:

- **[high] Plaintext key bypasses SecretRef mode:** `src/commands/onboard-search.ts:421`
  - body: Browser auth returns a raw Firecrawl API key; the PR stores that raw value through the fetch sync helper even after the search credential path may have been converted to a SecretRef. This can leave plaintext credentials in config when the operator asked for reference-only secret storage.
  - confidence: 0.91
