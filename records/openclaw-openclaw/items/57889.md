## Summary

Keep open. Current main does not implement this PR's early-allow behavior, and the PR is not merge-ready because it widens a gateway auth/device-identity boundary for Tailscale-authenticated device-less operator sessions that current code, tests, and docs still reject or scope-clear.

## What This Changes

This PR adds an early WebSocket connect allow path for device-less operator sessions authenticated by Tailscale or trusted-proxy when `sharedAuthOk` is false, plus a connect-policy test documenting the mismatch case.

## Best Possible Solution

Resolve the gateway trust policy first, then implement the narrow approved path in the shared WebSocket auth policy with docs and an integration test that covers connect plus an `operator.write` RPC. If device-less Tailscale/trusted-proxy operators should remain outside the exception list, close this PR with that policy decision and keep or redirect the linked token-auth reports separately.

## Reproduction Assessment

Yes. Current main has focused tests for the relevant boundary: Tailscale-only device-less operators are rejected, and Tailscale-routed shared-token operators connect with cleared scopes. The PR's early return would change that path before those policy checks run.

## Solution Assessment

No. The current PR solves the symptom by bypassing `evaluateMissingDeviceIdentity()` in `message-handler.ts`, and the condition is not limited to the token/password shared-auth scenario named by the title. A safer fix would live in the central policy helper and explicitly encode the approved credential, locality, client, and scope-preservation conditions.
