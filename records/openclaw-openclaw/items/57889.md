## Summary

Keep open. Current main does not implement this PR's final `operatorTransportMismatch` early-allow behavior for Tailscale-routed device-less operator WebSocket connects, and the current code/docs still treat device-less operator scope preservation as a narrow, security-sensitive exception. The PR remains a plausible implementation candidate for the linked gateway scope reports, but it needs maintainer product/security review rather than conservative cleanup closure.

## Best Possible Solution

Keep this PR open for maintainer review or replacement with a policy-aligned fix. If maintainers want device-less Tailscale/trusted-proxy operator sessions to retain scopes, the fix should be implemented in the gateway WebSocket auth path with an integration test covering connect plus an `operator.write` RPC. If maintainers want the current device-identity policy to stand, close this PR together with a clear resolution on the canonical linked reports such as #51396 and #48229.

## Risks / Open Questions

- The PR touches the gateway WebSocket auth/scope boundary and would allow a device-less operator path to proceed with self-declared scopes before the normal missing-device decision, so it needs explicit maintainer security review.
- The added PR test described in the provided diff documents helper behavior but does not exercise a full WebSocket handshake with Tailscale/trusted-proxy headers and a subsequent `operator.write` call.
- Current main has newer, narrower policy around direct-local backend shared-auth and Tailscale Control UI with browser device identity; merging this PR as-is may conflict with that documented trust model.
