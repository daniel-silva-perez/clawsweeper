## Summary

Keep open. Current main still performs npm global activation by renaming the live OpenClaw package tree from the active update process, while the Gateway `update.run` path still runs the package update before any restart. The service-stop mitigation reduces one lock source but does not prove the Windows self-update EBUSY bug is fixed, and #44614 remains the open detached-updater candidate.

## What This Changes

The issue reports that Windows npm-global `openclaw update` fails with EBUSY during rename because the running OpenClaw process locks `node_modules/openclaw` while updating itself.

## Best Possible Solution

Keep this as the canonical Windows npm-global self-update EBUSY report until #44614 or an equivalent design lands. The fix should hand package replacement to a detached Windows-safe helper or otherwise ensure the active CLI/Gateway process has released the live package tree before npm/global-root activation, with regression coverage for EBUSY/EPERM live-root activation and a real Windows update smoke path.
