## Summary

Keep open. Current main still performs the npm global package-root activation in the active updater process, and the Gateway `update.run` path still schedules restart only after the package update succeeds. The service-stop mitigation reduces one lock source, but it does not prove the Windows self-update EBUSY case is fixed; the open #44614 PR is still the focused detached-updater candidate.

## What This Changes

The issue reports that `openclaw update` fails on Windows npm-global installs because the active OpenClaw process locks the package directory while npm tries to rename it.

## Best Possible Solution

Keep this as the canonical Windows npm-global self-update EBUSY report until #44614 or an equivalent design lands. The desired end state is a Windows-safe update path that releases the active OpenClaw CLI/Gateway process from the live package tree before npm replaces it, with regression coverage for EBUSY/EPERM activation and a real Windows update smoke path.
