## Summary

Keep open. Current main still performs the npm-global update by swapping the live global package directory from the active update flow, so the Windows file-lock failure described by the report is still plausible; the related open PR #44614 is the active implementation candidate and should be reviewed rather than closing this issue early.

## What This Changes

The issue asks for `openclaw update` on Windows npm-global installs to avoid `EBUSY` rename failures caused by replacing the package tree while OpenClaw is still running from it.

## Best Possible Solution

Keep this as the canonical Windows npm-global self-update `EBUSY` report until a Windows-safe updater lands. The desired end state is a detached or exit-and-relaunch update flow that releases the active OpenClaw process from the live npm package tree before replacement, with regression coverage for `EBUSY`/`EPERM` behavior and a real Windows update smoke path.

## Reproduction Assessment

Yes. The issue provides concrete Windows Server/npm-global steps and an `EBUSY` rename log, and current main still contains the same live package-root rename path; native Windows smoke was not run in this read-only review.

## Solution Assessment

No. The current service-stop mitigation can help some managed CLI updates, but it is not the narrowest complete fix for self-replacement from a running Windows process; the detached helper design in #44614 or an equivalent handoff is the better product/code direction.
