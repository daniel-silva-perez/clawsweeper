## Summary

Keep this issue open. Current main still uses a staged npm install followed by an in-process rename swap of the live global package tree. The CLI now has partial mitigations, including blocking package updates from inside the gateway service and stopping a managed service before package replacement, but that does not prove the active CLI process has released the Windows-locked package tree. The Gateway `update.run` path also still calls the updater before scheduling a restart, so the reported Windows self-update EBUSY problem remains plausible and is better handled by the open detached-updater candidate #44614 or an equivalent design.

## Best Possible Solution

Keep this issue open as the canonical Windows npm global self-update EBUSY bug. Review and harden #44614, or land an equivalent Windows-safe updater that hands package replacement to a detached helper after the active CLI/Gateway process has released the live package tree. Add regression coverage for EBUSY/EPERM during live-root activation plus a real Windows manual or smoke update path.
