## Summary

Keep open. Current main still performs a live npm-global package-root swap from the active update flow, and the Gateway update RPC only schedules a restart after a successful update result, so the Windows self-lock EBUSY failure remains a real open bug. The related open PR #44614 is the active detached-updater candidate and should be reviewed, landed, or replaced rather than closing this issue early.

## What This Changes

Running `openclaw update` from a Windows npm-global install should update OpenClaw without EBUSY rename failures caused by active OpenClaw processes locking the installed package tree.

## Best Possible Solution

Keep this as the canonical Windows npm-global self-update EBUSY report until a Windows-safe update flow lands. The desired end state is a detached or exit-and-relaunch updater that releases the active OpenClaw process from the live npm package tree before replacement, while preserving npm fallback behavior, managed-service restart verification, result diagnostics, and regression coverage for Windows EBUSY/EPERM paths.

## Reproduction Assessment

Yes. The issue provides concrete Windows Server npm-global steps and an EBUSY rename log, and current main still contains the live package-root rename path that matches the reported failure. I did not run a live Windows smoke in this read-only pass.

## Solution Assessment

No. Current main's service-stop and inside-Gateway guard are useful partial mitigations, but they do not release the active CLI/Gateway process before replacing the live npm package tree; a reviewed detached handoff such as #44614, or an equivalent maintainer design, is the narrower complete fix.
