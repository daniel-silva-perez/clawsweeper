## Summary

Keep open. Current main still activates npm-global updates by renaming the live package root from the running update flow, and there is no detached Windows updater/result path on main; the related open PR #44614 remains an implementation candidate, not a reason to close this issue before merge.

## What This Changes

The report asks for `openclaw update` on Windows npm-global installs to avoid `EBUSY` rename failures when the running CLI or Gateway locks the installed package tree.

## Best Possible Solution

Keep this as the canonical Windows npm-global self-update `EBUSY` report until a Windows-safe updater lands. The desired end state is a flow that releases the active CLI/Gateway process from the live package tree before npm replaces it, with regression coverage for `EBUSY`/`EPERM` activation and a real Windows update smoke path.

## Reproduction Assessment

Yes. The issue includes a concrete Windows npm-global reproduction and `EBUSY` rename log, and current main still contains the same lock-sensitive live package-root rename path; native Windows smoke was not run during this read-only review.

## Solution Assessment

No. Current main's service-stop mitigation helps some shell updates but does not fully solve self-replacement from the active package tree; a detached helper or equivalent exit-and-relaunch update design is the narrower maintainable fix.
