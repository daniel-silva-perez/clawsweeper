## Summary

Keep open. Current main still performs npm global package activation from the active updater process, and the Gateway `update.run` path still restarts only after the update runner succeeds; the focused detached Windows updater in #44614 is open but has not landed.

## What This Changes

The issue requests that `openclaw update` on Windows npm-global installs avoid EBUSY rename failures caused by a running OpenClaw process locking its own package tree.

## Best Possible Solution

Keep this as the canonical Windows npm-global self-update EBUSY report until #44614 or an equivalent design lands. The desired end state is a Windows-safe update path that releases the active CLI/Gateway process from the live package tree before npm replaces it, with regression coverage for EBUSY/EPERM activation and a real Windows update smoke path.
