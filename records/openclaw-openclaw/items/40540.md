## Summary

Keep this issue open. Current main added an npm staging path, but it still activates the staged package by renaming the live global package directory from the active OpenClaw process, and gateway/CLI restart handling still happens only after a successful update. That is not enough evidence to close the reported Windows EBUSY self-update bug, especially with #44614 still presented as the detached-update implementation candidate.

## Best Possible Solution

Keep this issue open as the canonical Windows npm global self-update bug. Review #44614 or land an equivalent Windows-safe design that releases the active CLI/gateway package-tree locks before replacing the global package root, then add regression coverage for staged swap EBUSY behavior plus a real Windows smoke/manual update path.
