## Summary

Keep open. Current main still performs the npm-global package replacement from the running update process and only restarts the Gateway after the update runner reports success, so the reported Windows file-lock failure is not fixed. The related open PR #44614 remains the existing implementation candidate for a detached Windows updater design.

## What This Changes

The issue reports that `openclaw update` on Windows npm-global installs fails with `EBUSY` while npm renames the live `node_modules/openclaw` package root.

## Best Possible Solution

Keep this as the canonical Windows npm-global self-update EBUSY report until #44614 or an equivalent design lands. The desired end state is a Windows-safe update path that releases the active CLI/Gateway process from the live package tree before npm replaces it, with regression coverage for EBUSY/EPERM activation and a real Windows update smoke path.
