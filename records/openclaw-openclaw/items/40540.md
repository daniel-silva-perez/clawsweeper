## Summary

Keep open. Current main still performs the npm-global package swap from the active update flow, and the Gateway `update.run` path only schedules a restart after the update result succeeds, so the Windows self-lock `EBUSY` failure remains plausible. The related open PR #44614 is the active detached-updater candidate and should be reviewed or replaced rather than closing this issue early.

## What This Changes

The issue asks for `openclaw update` on Windows npm-global installs to avoid `EBUSY` rename failures caused by replacing the OpenClaw package tree while OpenClaw is still running from it.

## Best Possible Solution

Keep this as the canonical Windows npm-global self-update `EBUSY` report until a Windows-safe update flow lands. The desired end state is a detached or exit-and-relaunch updater that releases the active OpenClaw process from the live npm package tree before replacement, while preserving npm fallback behavior, service restart verification, and regression coverage for Windows `EBUSY` and `EPERM` paths.

## Reproduction Assessment

Yes. The issue provides concrete Windows Server npm-global steps and an `EBUSY` rename log, and current main still contains the same live package-root rename path that matches the reported failure. I did not run a live Windows smoke in this read-only pass.

## Solution Assessment

No. Current main's service-stop and post-update respawn hardening are useful partial mitigations, but they do not release the active Gateway or CLI process before the live package-root swap; a reviewed detached handoff such as #44614, or an equivalent maintainer design, is the narrower complete fix.
