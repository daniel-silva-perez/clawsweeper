## Summary

Keep open. Current main still lacks the requested Dashboard/WebChat user-selected reply workflow: there is no message context-menu handler, no reply callback or selected quote state in the chat props/composer, and the existing reply pill only renders inbound assistant directive metadata. The closed related #29732 points back here as the canonical tracker, while the open reply-threading PR in the provided context is a gateway/channel feature and does not implement this UI request.

## What This Changes

Add a Dashboard/WebChat message reply workflow where users can right-click a message, choose Reply, quote or reference the selected message, and continue typing in the composer.

## Best Possible Solution

Implement this in the Control UI chat owner area: add accessible message-level Reply controls plus a right-click menu, keep selected reply target state local to the composer, focus the textarea, show a dismissible quote preview, and prepend sanitized quoted Markdown on send. Defer protocol-level reply metadata, storage migrations, and channel threading unless maintainers promote a broader design.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path exists: inspect current main's Control UI chat renderer, composer, and send path and confirm there is no context-menu handler, reply action callback, selected quote preview state, or user reply field in chat.send.

## Solution Assessment

Yes. A UI-only quoted-Markdown reply workflow is the narrowest maintainable first fix for the reported Dashboard/WebChat problem; native threaded reply metadata would be a broader follow-up, not a prerequisite for this issue.
