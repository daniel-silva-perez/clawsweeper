## Summary

Current main still has the reported Telegram webhook loss window. The webhook handler returns 200 after validating the request body, then dispatches `bot.handleUpdate(...)` in a detached async task with no durable enqueue/replay path; existing tests and docs still describe/assert that asynchronous post-ACK behavior. The related contributor PRs remain closed unmerged, and the newer #55792 is broader inbound catch-up work rather than a direct replacement for durable replay of already-ACKed Telegram webhook payloads.

## Best Possible Solution

Keep this issue open. The best fix belongs in `extensions/telegram`: add a durable inbound webhook queue for validated raw Telegram updates before returning 200, replay pending entries on startup through the existing bot pipeline, and remove entries only after processing and deferred work settle. Use #40056, #42185, and #42523 as prior art, and coordinate with #55792 for broader cross-channel restart catch-up without treating it as a replacement for Telegram webhook payload persistence.

## Risks / Open Questions

- Users can still lose Telegram webhook messages if the gateway exits after returning 200 and before or during `bot.handleUpdate` and deferred Telegram work.
- A durable fix needs careful at-least-once semantics: persist only authenticated validated updates, replay pending entries on startup, delete only after processing and deferred work settle, and tolerate duplicate Telegram updates.
