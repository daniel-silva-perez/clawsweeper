## Summary

Current main still has the reported Telegram webhook loss window. Webhook mode returns 200 after body validation, then processes the update asynchronously with no durable enqueue/replay path; the related contributor PRs are closed unmerged, so they remain prior art rather than a shipped fix.

## Best Possible Solution

Keep this issue open. The best fix belongs in `extensions/telegram`: add a durable inbound webhook queue for validated raw Telegram updates before returning 200, replay pending entries on startup through the existing bot pipeline, remove entries only after processing and deferred work settle, and use #40056, #42185, and #42523 as prior art for maintainer-reviewed implementation and regression tests.

## Evidence

## Risks / Open Questions

- Users can still lose Telegram webhook messages if the gateway exits after returning 200 and before or during `bot.handleUpdate` and deferred Telegram work.
- A durable queue needs careful at-least-once semantics: persist only authenticated validated updates, replay pending entries on startup, delete only after processing and deferred work settle, and tolerate duplicate Telegram updates.
