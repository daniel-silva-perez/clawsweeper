## Summary

Keep open. Current main still acknowledges validated Telegram webhook payloads before processing them, with no durable queue or replay path for already-ACKed updates. The related contributor PRs were closed unmerged, and #55792 is related broader inbound catch-up work rather than a direct replacement for this Telegram webhook persistence gap.

## Best Possible Solution

Keep this issue open. The best fix belongs in extensions/telegram: add a durable inbound webhook queue for validated raw Telegram updates before returning 200, replay pending entries on startup through the existing bot pipeline, and remove entries only after processing and deferred work settle. Use #40056, #42185, and #42523 as prior art, and coordinate with #55792 without treating it as a replacement for Telegram webhook payload persistence.

## Risks / Open Questions

- Users can still lose Telegram webhook messages if the gateway exits after returning 200 and before or during bot.handleUpdate and deferred Telegram work.
- A durable fix needs careful at-least-once semantics: persist only authenticated validated updates, replay pending entries on startup, delete only after processing and deferred work settle, and tolerate duplicate Telegram updates.
