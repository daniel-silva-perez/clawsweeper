## Summary

Keep open. Current main still acknowledges validated Telegram webhook payloads before processing them, with no durable raw-update queue or replay path. The prior contributor PRs that attempted this fix are closed unmerged, and the broader open catch-up issue does not directly replace the Telegram webhook persistence gap.

## Best Possible Solution

Keep this issue open. The best fix belongs in extensions/telegram: durably enqueue the validated raw Telegram webhook update before returning 200, replay pending entries during webhook startup before accepting new deliveries, and remove entries only after bot.handleUpdate plus deferred Telegram work settle. Use #40056, #42185, and #42523 as prior art and coordinate with #55792 without folding this Telegram-specific persistence bug into the broader catch-up request.

## Risks / Open Questions

- Users can still lose Telegram webhook messages when the gateway exits after returning 200 and before or during bot.handleUpdate/deferred Telegram work.
- A durable fix needs careful at-least-once semantics: persist only authenticated validated payloads, replay on startup, delete only after processing settles, and tolerate duplicate Telegram updates.
- Treating #55792 as a replacement would leave the Telegram-specific ACK-before-processing loss mode unresolved.
