## Summary

Keep open. Current main and the latest release still acknowledge validated Telegram webhook payloads before durable handoff or processing, tests and docs describe that async post-ACK behavior, no webhook queue/replay implementation exists, and prior queue PRs remain closed unmerged; #55792 is broader and does not directly replace this Telegram-specific persistence gap.

## Best Possible Solution

Keep this issue open. The likely fix belongs in extensions/telegram: durably enqueue the validated raw Telegram webhook update before returning 200, replay pending entries during webhook startup before accepting new deliveries, and remove entries only after bot.handleUpdate plus deferred Telegram work settle. Coordinate with #55792 for broader channel catch-up, but keep this Telegram webhook persistence bug tracked directly.

## Risks / Open Questions

- Validated webhook updates remain process-memory-only after the 200 response, so restarts or aborts can still lose them.
- A durable fix needs careful at-least-once semantics: authenticate and validate before persistence, replay deterministically, delete only after processing settles, and tolerate duplicate Telegram updates.
- Treating #55792 as the only follow-up could leave this Telegram-specific ACK-before-processing gap unresolved.
