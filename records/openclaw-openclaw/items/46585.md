## Summary

Keep open. Current main and the latest release still have the reported selection shape: the CLI forwards `deviceId` and `facing` together, while the macOS picker returns an explicit device-id match before applying the requested front/back position. This is a valid maintainer follow-up rather than a closeable stale item.

## What This Changes

The issue requests correct front/back selection, or at least explicit unsupported handling, when `openclaw nodes camera snap` targets an iPhone Continuity Camera device with `--facing front` or `--facing back`.

## Best Possible Solution

The macOS camera path should resolve a Continuity device plus requested facing to the actual AVFoundation front/back capture device when the platform exposes one; if it cannot, OpenClaw should return a clear unsupported warning/error and document the limitation instead of silently producing a rear-camera capture labeled as front.

## Reproduction Assessment

Yes, for a maintainer with the required hardware: pair an iPhone as a Continuity Camera, run `openclaw nodes camera list --node <mac-node>`, then compare `snap --device-id <continuity-id> --facing front` with `--facing back`. I could not run that hardware path here, but current source inspection matches the reported behavior.

## Solution Assessment

Unclear until live AVFoundation verification. A source-only reorder may be wrong; the maintainable fix is API-backed Continuity device resolution when available, otherwise explicit unsupported behavior plus docs/CLI guidance.
