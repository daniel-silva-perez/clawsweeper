## Summary

Current main still does not resolve the reported Continuity Camera front/back behavior. The macOS camera path returns the requested deviceId match before applying the requested facing, the camera list exposes Continuity/external devices as raw devices with position labels, the CLI still forwards deviceId and a single facing together, and the docs still describe facing plus deviceId without a Continuity Camera caveat. The related closed allowlist reports address whether camera.snap is available, not whether an iPhone Continuity Camera front camera can be selected.

## Best Possible Solution

Keep this open for a focused macOS camera investigation. First verify on real macOS with iPhone Continuity Camera whether AVFoundation exposes a selectable front camera path for the reported device. If it does, implement the selection in CameraCaptureService and add narrow tests around deviceId/facing semantics. If it does not, make the CLI/tool/docs explicit that facing cannot be honored for single external/unspecified Continuity devices, preferably with an actionable warning or error instead of silently returning the rear camera.
