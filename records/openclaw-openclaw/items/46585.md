## Summary

Current main still does not resolve the reported iPhone Continuity Camera front/back selection bug. The macOS capture path still accepts the reported --device-id plus --facing front combination, but the selected device ID wins before facing is applied. Continuity/external macOS cameras are exposed as raw devices with position labels, and docs still do not explain any Continuity Camera limitation. The related closed reports are about camera.snap/camera.clip allowlisting and do not supersede this front/back selection issue.

## Best Possible Solution

Keep this open for focused macOS camera follow-up. First verify on real macOS with iPhone Continuity Camera whether AVFoundation exposes distinct selectable front/back capture devices or another API for the reported external/Continuity device. If it does, implement selection in apps/macos/Sources/OpenClaw/CameraCaptureService.swift and add narrow tests around deviceId/facing semantics. If it does not, make the CLI/tool/docs explicit that --facing cannot be honored for single external/unspecified Continuity Camera devices, preferably with an actionable warning or error instead of silently returning the rear camera.
