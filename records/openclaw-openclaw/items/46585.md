## Summary

Keep open. Current main and the latest release still have the reported macOS camera selection behavior: an explicit Continuity Camera device id is selected before the requested facing is considered, while the CLI still forwards `deviceId` and `facing` together and the docs do not warn that a single external/unspecified Continuity Camera may not honor front/back selection.

## What This Changes

The issue requests that iPhone Continuity Camera snapshots honor `--facing front` versus `--facing back` when capturing through `openclaw nodes camera snap` with an explicit device id.

## Best Possible Solution

The macOS camera path should either resolve a Continuity device id plus requested facing to the actual AVFoundation front/back capture device when the platform exposes one, or return an explicit unsupported error/warning and document the limitation instead of silently labeling a rear-camera capture as front.

## Reproduction Assessment

Yes, for a maintainer with the required hardware: pair an iPhone as a Continuity Camera, run `openclaw nodes camera list --node <mac-node>`, then compare `snap --device-id <continuity-id> --facing front` with `--facing back`. I could not live-run that hardware path here, but current source inspection matches the reported behavior because the explicit device id is selected before facing is applied.

## Solution Assessment

Unclear until live AVFoundation verification. If current macOS exposes separate continuity front/back devices, mapping that selection in `CameraCaptureService` is the narrow fix; otherwise the maintainable solution is an explicit unsupported response plus docs/CLI guidance.
