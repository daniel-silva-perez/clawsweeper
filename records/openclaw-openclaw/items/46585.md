## Summary

Keep this issue open. Current main still treats `--facing front` plus a specific `--device-id` as a valid request, passes both fields through to the macOS node, and the macOS camera picker returns the exact device-id match before applying the requested facing. The docs still advertise `front|back` facing and macOS `--device-id` usage without a Continuity Camera limitation or unsupported-facing error, and the latest shipped release has the same relevant camera code as current main.

## Best Possible Solution

Keep this open for a macOS camera backend fix. The maintainable path is to make `CameraCaptureService.pickCamera` honor facing when a selected device exposes enough position/device metadata, and otherwise fail loudly for unsupported Continuity/external device facing requests instead of silently returning the same rear-camera capture. Add focused macOS selection tests or an injectable picker seam, plus a docs note if Apple exposes only one Continuity Camera feed.
