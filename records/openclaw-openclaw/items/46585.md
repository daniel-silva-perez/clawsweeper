## Summary

Keep open. Current main still has the reported selection shape: the CLI sends `facing` and `deviceId` together, while the macOS picker returns the explicit device-id match before applying front/back selection. The latest release has the same relevant files, so this is still a valid macOS Continuity Camera follow-up rather than stale cleanup.

## What This Changes

The issue reports that iPhone Continuity Camera captures on macOS use the rear camera for both `--facing front` and `--facing back` when a Continuity `--device-id` is supplied.

## Best Possible Solution

The macOS camera path should resolve a Continuity device plus requested facing to the actual AVFoundation front/back capture device when the platform exposes one; if it cannot, OpenClaw should return a clear unsupported warning/error and document the limitation instead of silently producing a rear-camera capture for a front request.

## Reproduction Assessment

Yes, for a maintainer with the required hardware: pair an iPhone as a Continuity Camera, run `openclaw nodes camera list --node <mac-node>`, then compare `snap --device-id <continuity-id> --facing front` with `--facing back`. Source inspection of current main matches the reported behavior, but I could not run the hardware path here.

## Solution Assessment

Unclear until live AVFoundation verification. The safest solution is API-backed Continuity device resolution when available, otherwise explicit unsupported behavior plus docs/CLI guidance; a blind picker reorder is not enough evidence.
