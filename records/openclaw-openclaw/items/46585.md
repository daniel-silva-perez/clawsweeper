## Summary

Keep open. Current main still has the reported behavior shape: the CLI sends `facing` and `deviceId` together, and the macOS camera picker returns the explicit device-id match before applying front/back selection. The latest release has the same relevant camera files, and the existing docs/tests do not document or validate a Continuity Camera front/back limitation.

## What This Changes

The issue requests that macOS iPhone Continuity Camera captures respect `--facing front` when a Continuity Camera `--device-id` is supplied, instead of capturing rear-camera output for both front and back.

## Best Possible Solution

The macOS camera path should resolve a Continuity device plus requested facing to the actual AVFoundation front/back capture device when the platform exposes one; when it cannot, OpenClaw should return an explicit unsupported warning or error and document the limitation instead of silently capturing rear-camera output for a front request.

## Reproduction Assessment

Yes, for a maintainer with the required hardware: pair an iPhone as a Continuity Camera, run `openclaw nodes camera list --node <mac-node>`, then compare `snap --device-id <continuity-id> --facing front` with `--facing back`. Source inspection of current main matches the reported selection path, but I could not execute the hardware path here.

## Solution Assessment

Unclear until live AVFoundation verification. The current implementation is not the best end state because device-id selection can bypass facing; the maintainable fix is API-backed Continuity resolution when possible, otherwise explicit unsupported behavior plus docs.
