## Summary

Keep open. Current main still has the reported macOS Continuity Camera surface: the macOS capture service resolves an explicit deviceId before applying facing, the CLI still sends deviceId and facing together, and the docs do not state that a single external/unspecified Continuity Camera may not honor front/back selection. The related closed allowlist reports are adjacent availability bugs, not a fix for this behavior.

## What This Changes

The issue asks for `openclaw nodes camera snap --device-id <iPhone Continuity Camera> --facing front` to select the iPhone front camera, or at least stop silently capturing from the rear camera when that selection is unsupported.

## Best Possible Solution

The macOS camera path should either select a real front/back Continuity capture device when AVFoundation exposes one, or return an explicit unsupported warning/error and document that facing cannot be honored for a single external/unspecified Continuity device.

## Reproduction Assessment

Yes for a maintainer with the required hardware: pair an iPhone as a Continuity Camera, run `nodes camera list`, then compare `nodes camera snap --device-id <continuity-id> --facing front` with `--facing back`. Static inspection of current main matches that path because the same deviceId match wins before facing is applied.

## Solution Assessment

Unclear until live AVFoundation verification. If a separate Continuity front-camera device is available, implementing that mapping in `CameraCaptureService` is the best fix; otherwise the maintainable solution is an explicit unsupported response plus docs/CLI guidance rather than silently returning rear-camera captures. citeturn0search0turn0search3turn0search4
