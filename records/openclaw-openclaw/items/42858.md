## Summary

Keep open. Current main still does not support the reported `bailian` provider id or `BAILIAN_API_KEY` auth candidate, and the auth resolver returns null when a normalized provider has no candidate, evidence, or setup resolver. Closed target-repo PR attempts did not ship, and the open external fork PR is an implementation candidate rather than current OpenClaw behavior.

## What This Changes

The issue requests Bailian provider env-based auth so `BAILIAN_API_KEY` resolves for configs using provider id `bailian`.

## Best Possible Solution

If Bailian naming is accepted, add it through the Qwen/plugin-owned auth surface with `BAILIAN_API_KEY` coverage and regression tests for both provider normalization/auth lookup and the runtime model-auth path. If the alias is intentionally unsupported, document the supported provider ids and env names clearly in the Qwen provider docs and close with that product decision.

## Reproduction Assessment

Yes. The code path is enough to reproduce the resolver failure: `resolveEnvApiKey("bailian")` normalizes to `bailian`, finds no candidate/evidence entry, and falls through to null; the reporter also confirmed that adding the Bailian mapping made `bailian/qwen3.5-plus` work in their environment.

## Solution Assessment

Unclear. A one-line env-map addition addresses the immediate lookup, but the maintainable fix should align with the manifest-owned Qwen provider ids and decide whether `bailian` is an official compatibility alias; otherwise docs should explicitly steer users to the supported ids and env vars.
