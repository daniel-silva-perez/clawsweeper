## Summary

Keep this issue open. Current main at bdfb408ce6eeb724a638c106104dbbcae50b612c still lacks the reported `bailian` provider id plus `BAILIAN_API_KEY` auth path. The bundled Qwen path supports `qwen`, `qwencloud`, `modelstudio`, and `dashscope` with QWEN/MODELSTUDIO/DASHSCOPE env vars, but not Bailian naming. Related target-repo PR attempts are closed unmerged, and the open external fork PR is still an implementation candidate rather than shipped main behavior.

## Best Possible Solution

Keep this issue open for maintainer follow-up. If Bailian naming should be supported, prefer a focused manifest-owned Qwen compatibility change that adds the `bailian` provider/auth alias and `BAILIAN_API_KEY` env candidate, with regression coverage for `resolveEnvApiKey("bailian")` and the runtime model-auth path. If maintainers reject that alias, document the supported `qwen`/`modelstudio`/`dashscope` provider ids and current env SecretRef syntax, then close with that product decision.
