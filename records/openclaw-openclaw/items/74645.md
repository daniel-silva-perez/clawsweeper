## Summary

Keep this PR open for normal maintainer review. Current main still routes non-local macOS Talk providers such as OpenAI to local system speech, while the PR narrowly switches that path to Gateway `talk.speak`, preserves the explicit ElevenLabs/MLX/system paths, and has no blocking functional or security finding.

## What This Changes

The PR adds a macOS Talk playback plan that sends unsupported local Talk providers through Gateway `talk.speak`, carries a configured voice ID for remote providers, and updates the Talk runtime speech-plan test for OpenAI/custom providers.

## Best Possible Solution

Land this PR or an equivalent narrow macOS runtime change that makes remote/custom Talk providers use Gateway `talk.speak` as the primary playback path, keeps explicit `system` and `mlx` local behavior unchanged, and leaves provider-selection UI as a separate product decision if still desired.

Yes. A code-level reproduction is high confidence: on current main, `TalkModeRuntime.playbackPlan(provider: "openai", apiKey: nil, voiceId: "onyx")` reaches the default branch and returns `.systemVoiceOnly`, so no `talk.speak` call is attempted.

## Solution Assessment

Yes for the routing bug. The PR uses the existing macOS `playGatewayTalkSpeak` helper and gateway RPC contract instead of adding a new API, while preserving the direct ElevenLabs, MLX, and system-voice paths.
