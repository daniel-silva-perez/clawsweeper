## Summary

Keep this PR open. Current main still has the linked update-status misdetection, but the PR is not merge-ready because it adds a second, weaker global-install detector and misses supported custom Bun install roots that the existing update-global detector already handles.

## What This Changes

The PR adds a path-heuristic global install manager detector, wires package-install update status to it, and adds unit tests for pnpm, Bun, and npm global path examples.

## Best Possible Solution

Route package-root status detection through the existing global-install ownership logic, keep repo package-manager detection for git checkouts, and add focused regression tests for npm-global OpenClaw metadata declaring pnpm plus custom `BUN_INSTALL` Bun roots.

## Reproduction Assessment

Yes. A source-level reproduction is clear: use a non-git OpenClaw package root with `package.json.packageManager` set to pnpm, as the published package does, and current `checkUpdateStatus` returns that metadata-derived manager while `openclaw update status` renders it as `Install`.

## Solution Assessment

No. The PR fixes the right user-visible problem, but the implementation is not the best path because it adds a weaker duplicate detector instead of calling or adapting `detectGlobalInstallManagerForRoot`, which already handles real roots, symlinks, npm prefixes, and `BUN_INSTALL`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Handle custom Bun global roots:** `src/infra/detect-package-manager.ts:51`
  - body: This check only recognizes Bun when the package path contains `/.bun/`, but OpenClaw already supports custom Bun roots through `BUN_INSTALL`. After this PR, `checkUpdateStatus` relies on this function for all package installs, so a package under a custom Bun global root such as `/opt/bun/install/global/node_modules/openclaw` is reported as npm. Reuse the existing global-root detector or consult `BUN_INSTALL` here.
  - confidence: 0.9
