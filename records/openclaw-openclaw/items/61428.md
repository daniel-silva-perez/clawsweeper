## Summary

Keep open. Current main does not implement hook-mapping `sessionTarget`, and the PR still needs maintainer/security review because the proposed main-session rewrite happens after hook session-prefix validation and leaves config metadata/replay semantics out of sync.

## What This Changes

The PR adds a `sessionTarget?: "main" | "isolated"` hook mapping field and threads it through mapping normalization and dispatch so configured mapped webhooks can run an agent turn against the agent main session while direct `/hooks/agent` remains isolated.

## Best Possible Solution

Keep the PR open for maintainer review, then rework the feature on current main so effective session targets are resolved before prefix validation and idempotency hashing, cron job shape remains consistent, generated config metadata/docs/changelog are updated, and focused hook mapping tests cover main, isolated, agentId, prefix, and replay behavior.

## Reproduction Assessment

Yes for the review findings. The static path is a mapped hook with `sessionTarget: "main"`: current main validates one session key in `createHooksRequestHandler`, while the PR changes `dispatchAgentHook` to execute against a different main-session key; the replay and metadata gaps are directly visible in source.

## Solution Assessment

No. Mapping-level main targeting may be the right product direction, but this implementation should resolve the effective key earlier, include `sessionTarget` in replay scope, avoid or explicitly update the cron `main`/`agentTurn` invariant, and regenerate public config surfaces before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Revalidate prefixes against the effective session key:** `src/gateway/server/hooks.ts:49-55`
  - body: When a mapping uses `sessionTarget: "main"`, this branch replaces the already validated dispatch key with `resolveAgentMainSessionKey(...)`. A config can therefore validate one key but execute against a different main-session key, which can both bypass and incorrectly reject `hooks.allowedSessionKeyPrefixes`. Resolve the effective key before prefix validation, or re-check the rewritten key before dispatch.
  - confidence: 0.88
- **[P2] Keep sessionTarget in replay dedupe scope:** `src/gateway/server-http.ts:793`
  - body: Mapped hook idempotency still hashes a dispatch scope without `sessionTarget`, but this PR makes `isolated` and `main` dispatches behave differently. Two otherwise identical mapped requests with the same idempotency key can return the first run id even when the second should target a different session. Include the requested/effective session target in the replay scope.
  - confidence: 0.86
- **[P2] Regenerate config metadata for sessionTarget:** `src/config/zod-schema.hooks.ts:47`
  - body: Adding `sessionTarget` to the strict hook mapping schema extends the public config contract, but the generated schema/help/label metadata still lacks `hooks.mappings[].sessionTarget`. Schema-driven UI and docs tooling will not expose or preserve the new option until those generated surfaces are updated.
  - confidence: 0.9
- **[P2] Preserve the cron job sessionTarget invariant:** `src/gateway/server/hooks.ts:70`
  - body: This sets hook agent-turn jobs to `sessionTarget: "main"`, but cron validation currently defines `main` jobs as `systemEvent` jobs and `agentTurn` jobs as isolated/current/session-target jobs. The hook path bypasses `createJob` today, but downstream cron helpers and tests rely on that shape. Use a `session:<effective main key>` target for the agent turn or update the cron contract and tests intentionally.
  - confidence: 0.77

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete hook session-policy concern: the effective main-session key is chosen after prefix validation.

Concerns:

- **[medium] Effective session key bypasses prefix policy:** `src/gateway/server/hooks.ts:49`
  - body: For `sessionTarget: "main"`, the code validates the pre-dispatch hook/default session key, then replaces it with the agent main session key in `dispatchAgentHook`. Operators using `hooks.allowedSessionKeyPrefixes` can therefore get enforcement against a key that is not actually used for execution.
