## Summary

Current main still hardcodes webhook hook agent runs to isolated sessions, so the requested feature is not implemented. This PR should stay open rather than close, but it should not merge as-is because the diff rewrites the execution session after prefix-policy validation, creates a CronJob shape current main explicitly rejects, and leaves generated config metadata and replay scope out of sync. The author did address the first bot round around schema, agent scoping, and normalization; later review issues remain.

## What This Changes

The PR adds a `sessionTarget` option to hook mapping config, schema, mapping resolution, action dispatch, and gateway hook execution so mapped webhook agent runs can target the main session instead of the default isolated session.

## Best Possible Solution

Rebase the PR onto current main and implement the feature around the current split hook request handler. The effective target session should be resolved once, validated against `hooks.allowedSessionKeyPrefixes`, included in replay dedupe scope, and passed to dispatch without a later security-relevant rewrite; generated config metadata, docs/changelog, and focused hook tests should move with the public config key.

## Reproduction Assessment

Yes for the main policy regression: a mapping with `sessionTarget: "main"` can validate one hook key and then execute on a different agent main key after the PR's dispatch rewrite. The unsupported CronJob shape is also reproducible by comparing the PR's `sessionTarget: "main"` agent-turn job against current `assertSupportedJobSpec` and `assertMainSessionAgentId` checks.

## Solution Assessment

No. Mapping-level `sessionTarget` is a reasonable narrower direction than #58023, but this implementation needs to validate the effective session before dispatch, align with the cron session contract, update generated config metadata, and add regression tests before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Revalidate the effective main session key:** `src/gateway/server/hooks.ts:47-52`
  - body: When `sessionTarget` is `"main"`, this rewrites `sessionKey` after the hook request handler has already applied `hooks.allowedSessionKeyPrefixes` to the generated or mapped key. A mapping can therefore pass policy with a `hook:*` key and execute on an `agent:*` main key, or be rejected even when the final main key is allowed. Resolve the effective main key before prefix validation and replay hashing so policy covers the key that actually runs.
  - confidence: 0.9
- **[P2] Use a supported cron target for agent turns:** `src/gateway/server/hooks.ts:70`
  - body: This sets `job.sessionTarget` to `"main"` while the payload remains `kind: "agentTurn"`. Current cron validation explicitly reserves `sessionTarget: "main"` for `systemEvent` jobs and rejects non-default agents, so this direct construction bypasses the cron contract and can break downstream assumptions. Either model this as a supported session-bound agent turn or extend the cron contract with tests before dispatching this shape.
  - confidence: 0.86
- **[P2] Include sessionTarget in hook replay scope:** `src/gateway/server-http.ts:793`
  - body: Mapped hook dispatch now changes execution semantics based on `sessionTarget`, but the idempotency replay scope still keys only on agent, sessionKey, message, delivery, and model fields. Two otherwise identical mapped retries can return the stale run id even when one targets isolated execution and another targets the main session. Add the resolved session target or effective session key to the replay scope.
  - confidence: 0.78
- **[P2] Regenerate hook config metadata:** `src/config/zod-schema.hooks.ts:47`
  - body: Adding `sessionTarget` to the strict hook mapping schema creates a new public config key, but the generated schema/help/label metadata is not updated. Schema-driven UI or docs tooling will not expose the option and may hide or drop it during edits. Regenerate the config metadata and include the new path in the generated surfaces.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete session-policy concern by validating one hook session key and then executing a different main-session key.

Concerns:

- **[medium] Prefix policy applies to the wrong session key:** `src/gateway/server/hooks.ts:47`
  - body: For `sessionTarget: "main"`, dispatch replaces the prevalidated hook session key with the agent's main session key. That can bypass `hooks.allowedSessionKeyPrefixes` when the original key is allowed but the effective main key is not, or reject valid main-target mappings when only the final key is allowlisted.
