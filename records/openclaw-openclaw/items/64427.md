## Summary

Keep this issue open. Current main still accepts and documents Slack `dmHistoryLimit`, and generic session-history trimming can apply to existing DM transcripts, but the Slack plugin still does not plumb a DM-specific history limit or fetch/inject prior Slack 1:1 DM conversation history when a fresh DM session starts. Related #67020 is open and tracks the same gap, but this issue is older and has the stronger reproduction, so it should remain canonical unless maintainers choose otherwise.

## Best Possible Solution

Keep this issue open and fix it in the Slack plugin: derive an effective DM history limit using `channels.slack.dms.<user>.historyLimit` before `channels.slack.dmHistoryLimit`, carry it through `SlackMonitorContext`, fetch recent `conversations.history` for fresh top-level `channel_type: "im"` sessions, exclude the current inbound message, inject the context consistently with existing inbound history, and add focused Slack tests for DM backfill, override precedence, reset/fresh-session behavior, and no duplicate current-message injection.
