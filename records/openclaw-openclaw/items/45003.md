## Summary

Keep open. Current main still archives reset transcripts, but it does not provide the requested bulk `.jsonl.reset.*` restore/re-index utility, and the gateway/tool contracts in this checkout still do not expose archived reset history through `chat.history` or `sessions_history`. The same-author open PRs #60409 and #73883 are related implementation candidates, so this issue should stay open until maintainers review/land them or explicitly choose a different recovery design.

## Best Possible Solution

Keep this issue open while maintainers review the paired recovery PRs. The best path is either to land a conservative archive-reading surface for `chat.history` and `sessions_history` and then decide whether a restore script is still needed, or to add a first-party `openclaw sessions restore`/doctor repair workflow with dry-run, backup guidance, stable session identity preservation, original/restored path metadata, and tests around `.jsonl.reset.*` re-indexing.

## Risks / Open Questions

- A restore/re-index command would mutate local session state, so it should preserve stable session identity, original/restored paths, reset timestamps, and encourage backup/dry-run behavior before any rename.
- Open PRs #60409 and #73883 may solve much of the recovery gap without a bulk mutating restore script; maintainers should decide whether the original script remains necessary after those PRs are reviewed.
