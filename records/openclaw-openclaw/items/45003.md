## Summary

Keep open. Current main archives reset transcripts and has some adjacent archive-aware behavior, but it does not provide the requested bulk restore/re-index utility, `sessions_history` archived-session selector, or UI/logical-history recovery path. The related open PR and open issues are implementation candidates/follow-ups, not proof that this recovery request is obsolete.

## Best Possible Solution

Keep this open and decide the product path: either add a conservative first-party restore/re-index command or doctor repair for `.jsonl.reset.*` archives, or explicitly supersede it after archived reset sessions are first-class in `chat.history`, `sessions_history`, and the UI. The restore path should preserve stable session identity and original/restored transcript metadata for downstream history viewers.

## Risks / Open Questions

- A restore utility would mutate private transcript files and `sessions.json`; the safest implementation needs dry-run output, backup-first behavior, collision handling, idempotency, stable identity/provenance metadata, and retention warnings.
- Open #60409 may partially address `chat.history` fallback, but it is a narrower unmerged candidate and does not replace a first-party restore/re-index recovery command.
- Closing this now would lose the user-facing recovery path while current main still leaves archived reset sessions outside the normal `sessions_history`/UI restore workflow.
