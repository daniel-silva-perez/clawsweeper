## Summary

Keep this issue open. Current main still archives reset transcripts and has improved daily-reset freshness semantics, but it does not provide the requested bulk `.jsonl.reset.*` restore/re-index utility or first-class archived reset history access through `sessions_history`, `chat.history`, or the sessions UI/listing surface. The same-author open PR #60409 is related implementation work, so this issue should remain open until maintainers review that path or explicitly choose a broader recovery design.

## Best Possible Solution

Keep this issue open until maintainers choose the recovery product path: add a conservative first-party restore/re-index command or doctor repair for `.jsonl.reset.*` archives, or explicitly supersede it once archived reset sessions are first-class in `chat.history`, `sessions_history`, and session listing/UI. Preserve stable session identity plus original/restored path and reset timestamp metadata for downstream history viewers.

## Risks / Open Questions

- A restore/re-index command would mutate user transcript files, so it should be backup-first, dry-run capable, idempotent, and careful about current `.jsonl` collisions.
- Archive retention can prune `.reset.*` files before users discover them, so recovery UX should surface archives before or during cleanup.
- Open PR #60409 appears to cover one `chat.history` fallback path; landing it alone would not satisfy bulk restore/re-index, session list/UI discoverability, or stable identity metadata for downstream viewers.
