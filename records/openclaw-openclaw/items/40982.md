## Summary

Keep open. Current main still retains the reported effective 180-second resumed CLI no-output cap, the latest release tag still ships it, and open PR #41022 remains the active closing-syntax implementation path rather than a reason to close this issue before merge.

## What This Changes

Request raising or removing the CLI no-output watchdog defaults/config so long-running CLI agent requests are not killed after about 180 seconds of silence.

## Best Possible Solution

Ship a narrow CLI watchdog update that removes the effective 180-second cap for normal long-running CLI sessions, with defaults, resolver behavior, schema/types, tests, and public config behavior aligned; handle Monitor-specific expected-idle behavior as a scoped follow-up if the default raise does not cover it.

## Reproduction Assessment

Yes. A resumed CLI run with an overall timeout above three minutes and no stdout deterministically resolves to a 180000ms no-output watchdog on current main, and the Apr 29 production log in the discussion matches that exact timeout.

## Solution Assessment

Unclear. Raising the defaults to 30 minutes is a narrow maintainable first step, but the best final shape should still decide whether OpenClaw supports explicit no-cap semantics and whether Monitor-backed expected idle needs a separate exemption.
