## Summary

Keep this open. Current main and the latest shipped release still preserve a positive 180-second resumed CLI no-output cap with no 0/null disable semantics, and open PR #41022 is the active implementation candidate linked with closing syntax, so closing or spawning a duplicate repair lane would be premature.

## What This Changes

The issue asks to raise or remove the CLI no-output watchdog defaults so fresh and resumed CLI sessions are not killed after about 180 seconds, ideally with configurable disable semantics.

## Best Possible Solution

Land or revise #41022, or a successor, so shared CLI watchdog defaults no longer terminate legitimate long-running CLI turns at 180 seconds, the supported disable/configuration semantics are explicit, global timeout enforcement remains intact, and regression coverage covers fresh and resumed backend resolution plus any disable path that is accepted.
