## Summary

Keep open. Current main still resolves resumed CLI sessions to a 180-second no-output watchdog by default, the latest release tag still contains the same cap, and open PR #41022 is the active closing-syntax implementation path rather than a reason to close this issue before merge.

## What This Changes

The issue requests raising or removing the CLI no-output watchdog defaults so long-running CLI requests are not killed after about three minutes of silence.

## Best Possible Solution

Ship a narrow CLI watchdog fix that removes the practical 180-second cap for normal long-running CLI sessions, keeps plugin-exported defaults and bundled CLI backend behavior aligned, and treats explicit no-cap or Monitor expected-idle semantics as either part of that reviewed change or a clearly scoped follow-up.

## Reproduction Assessment

Yes. With a resumed CLI backend, no watchdog override, and an overall timeout above 600000ms, `resolveCliNoOutputTimeoutMs` computes 180000ms; the runner and Claude live-session paths then enforce that value as a no-output timeout, matching the Apr 29 production log.

## Solution Assessment

Yes for the triage decision: this should stay open while #41022 is reviewed, landed, revised, or replaced. The implementation direction is plausible, but the final patch should deliberately decide whether default-only changes are enough or whether disable semantics belong in the same fix.
