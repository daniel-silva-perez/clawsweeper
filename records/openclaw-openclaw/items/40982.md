## Summary

Keep open: current main still resolves resumed CLI no-output timeouts through the 180-second resume cap, the latest release tag preserves that behavior, and an open closing-linked PR remains the active implementation path rather than a reason to close this issue before merge.

## What This Changes

The issue requests raising or removing the 3-minute CLI no-output watchdog cap for fresh and resumed CLI requests, with optional per-backend tuning or disable semantics.

## Best Possible Solution

Land or revise #41022, or a narrow successor, so shared CLI watchdog defaults no longer terminate legitimate long-running CLI turns at 180 seconds; if disable/configuration semantics are accepted, make them explicit in the schema, resolver, and regression coverage while keeping global timeout enforcement intact.
