## Summary

Keep open. Current main does not contain the PR's cron-run session-key remapping helper, so the work is not obsolete, but the latest PR diff still leaves two changed enqueue paths out of alignment with their matching wake requests for custom `session.mainKey` cron-run sessions.

## What This Changes

The PR adds `resolveEventSessionKey` and extends `scopedHeartbeatWakeOptions` to remap cron-run session keys to an agent main key across bash exec, CLI watchdog, ACP stream, and gateway node exec paths, with session-key and exec-event classification tests.

## Best Possible Solution

Keep this PR as the narrow #52305 implementation candidate, but compute or load the relevant `mainKey` once per affected path and pass the same value to both `resolveEventSessionKey` and `scopedHeartbeatWakeOptions` before running targeted heartbeat/session-key tests and the changed gate.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Pass mainKey into the node-event wake remap:** `src/gateway/server-node-events.ts:686`
  - body: This enqueue now remaps cron-run events with `cfg.session?.mainKey`, but the matching wake below still calls `scopedHeartbeatWakeOptions(sessionKey, ...)` without that value. With `session.mainKey: "captain"`, the event is queued under `agent:<id>:captain` while the wake targets `agent:<id>:main`, so heartbeat can still drain the wrong queue and lose the completion.
  - confidence: 0.91
- **[P2] Keep ACP enqueue and wake keys aligned:** `src/agents/acp-spawn-parent-stream.ts:198`
  - body: The ACP stream enqueue path uses `loadConfig().session?.mainKey`, but `wake()` still remaps `parentSessionKey` without that same main key. Custom-main-key cron-run sessions will enqueue ACP progress under the configured main key and wake the default `:main` queue, leaving the queued event invisible to heartbeat.
  - confidence: 0.89
