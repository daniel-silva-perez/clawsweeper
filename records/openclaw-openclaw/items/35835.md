## Summary

Keep this issue open. Current main still has image-aware read behavior only: the read wrapper normalizes/sanitizes image blocks, the sandbox read hook only exposes image MIME detection, and the OpenAI/Anthropic/Google tool-result transports only forward text/images. Existing audio support covers MIME knowledge, transcription/media-understanding, TTS, realtime voice, and provider media-understanding paths, but not native audio blocks returned from read("*.audio") into the active model turn. The related #46607/#46601 work is a narrower Google/Gemini headless @file payload path and does not supersede this general read-tool request.

## Best Possible Solution

Keep this issue open and implement a focused cross-cutting feature: add a stable audio content block/session contract for read results, extend read MIME detection and guards for supported audio formats, forward audio blocks only through capable provider transports such as Google/Gemini and OpenAI, return a clear unsupported-model/tool-result message when the selected model cannot accept audio, and add read plus transport regression coverage. Treat #46607/#46601 as related but narrower Google headless @file work, not as the canonical fix for this issue.
