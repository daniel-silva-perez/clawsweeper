## Summary

Keep open. Current main still implements multimodal `read` only for images: the upstream read dependency exposes only an image MIME hook and text/image blocks, OpenClaw’s wrapper normalizes/sanitizes image blocks only, sandbox read MIME detection filters to `image/*`, and OpenAI/Google/Anthropic tool-result transports forward text/images but not audio. Existing audio support covers inbound media understanding, STT, TTS, and realtime voice, but not native audio attachments returned by `read("*.mp3")` into the active model turn. The related #46607/#46601 work is a narrower Google/Gemini headless `@file` path and does not supersede this general read-tool request.

## Best Possible Solution

Keep this issue open. The best path is a maintained design/implementation that adds a stable audio content block for read results, detects and guards supported audio MIME types in the read path, forwards audio only through capable provider transports such as Google/Gemini and OpenAI, returns a clear unsupported-model/tool-result message otherwise, and adds regression coverage across read plus provider conversion. Treat #46607/#46601 as related but narrower Google headless `@file` work, not the canonical fix for this issue.
