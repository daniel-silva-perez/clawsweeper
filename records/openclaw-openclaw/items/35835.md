## Summary

Current main still does not implement audio-file multimodal parity for the read tool. The read wrapper and upstream read dependency are still text/image-oriented, the session content contract has text/image blocks but no audio block, and OpenAI/Google/Anthropic tool-result transports only forward text plus images. Existing audio support covers STT, TTS, inbound media understanding, realtime voice, and narrower Google headless @file work, but not read("*.mp3") returning a native audio attachment to the active model turn.

## Best Possible Solution

Keep this issue open for maintainer design. The best path is to add a stable audio content-block contract for read/tool results, teach the read path to detect and guard supported audio MIME types, forward audio only through capable providers such as Google/Gemini and OpenAI, return a clear unsupported-model message otherwise, and add focused regression coverage across read plus provider conversion. Treat #46601/#46607 as related but narrower Google headless @file work, not the canonical fix for this issue.
