## Summary

Keep open. Current main still omits ordinary non-verbose tool failure reasons from chat-visible warnings and only clears internal retry failure state after a matching successful mutating action; it does not emit the requested retry-success confirmation. The remaining change needs maintainer policy on sanitized diagnostics across internal and external chat surfaces.

## What This Changes

The issue requests chat-visible tool failure warnings to include a useful failure reason and to confirm when a retry later succeeds.

## Best Possible Solution

Define sanitized, allowlisted failure summaries by tool and channel, then add a retry-success confirmation only when a previously surfaced failure is resolved by a matching successful mutating action. Coordinate the broader diagnostic direction with #60946, the external-channel privacy constraints from #54757, and the narrower edit-mismatch work in #67276.

## Reproduction Assessment

Yes. A high-confidence unit-level reproduction exists by building a verbose-off payload with a mutating lastToolError and observing that the warning omits the detail, then simulating a failed edit followed by a matching successful edit and observing that only lastToolError state is cleared.

## Solution Assessment

No current implementation fully solves the issue. The safest direction is not raw error surfacing; it is sanitized reason summaries plus explicit retry-success semantics with channel-aware privacy rules.
