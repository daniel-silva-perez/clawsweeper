## Summary

Current main still only partially addresses the report: visible non-verbose tool failure warnings generally omit the underlying `lastToolError.error`, and same-action retry success clears the stored failure state without emitting an explicit retry-success confirmation. The linked implementation PR #46592 is closed unmerged and explicitly did not cover retry confirmation, while related #60946 overlaps model-facing diagnostics but does not supersede this chat-surfacing policy issue.

## What This Changes

Request that user-facing chat messages for failed tool calls include a safe failure reason and that successful retries visibly confirm the prior failure was resolved.

## Best Possible Solution

Keep this issue open for maintainer product/security policy. The best path is to define a safe redacted or allowlisted failure-reason policy for chat-visible tool warnings, decide whether same-action retry success should emit a confirmation only after a failure warning was actually delivered, and coordinate that with #54757's external-channel privacy concern and #60946's model-facing rich diagnostics work.

## Risks / Open Questions

- Surfacing raw `lastToolError.error` in ordinary chat can expose local paths, command output, provider responses, channel targets, or secrets to Discord/Telegram/WhatsApp-style monitoring surfaces.
- Existing tests intentionally assert that default non-verbose warning summaries omit raw details, so a global suffix change would alter current privacy and noise expectations.
- Retry-success confirmation could add channel noise unless it is tied to a previously delivered failure warning and a true same-action success.
- Related #54757 asks for less channel delivery of tool errors, while #60946 asks for richer diagnostics; this needs a coherent policy rather than a one-line formatter change.
