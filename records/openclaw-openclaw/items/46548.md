## Summary

Keep open. Current main still only partially addresses the report: ordinary non-verbose chat-visible tool warnings continue to omit the underlying failure reason, and a matching successful retry clears internal failure state without emitting the requested retry-success confirmation. The remaining design is tied to channel/privacy policy, so it should not be closed or auto-fixed without maintainer direction.

## What This Changes

Request that chat-visible tool failure messages include a useful failure reason and that a later successful retry be confirmed to the operator.

## Best Possible Solution

Define a safe chat-visible diagnostics policy for tool failures, including redacted or allowlisted reason summaries by tool and channel, then add retry-success confirmation only when a previously surfaced failure is resolved by the same successful mutating action. Keep this coordinated with #60946 for richer diagnostics, #54757 for external-channel privacy, and #67276 for the narrower edit-mismatch case.

## Reproduction Assessment

Yes. A high-confidence unit-level reproduction exists: build a verbose-off payload with a mutating `lastToolError` and the warning omits the detail, then follow the tool handler retry path and observe that it only clears `lastToolError`. Current tests encode both behaviors.

## Solution Assessment

No current implementation is the full solution. The closed unmerged PR #46592 was not sufficient because it surfaced raw/truncated reasons without solving retry confirmation or channel privacy policy; the safer path is sanitized reason summaries plus explicit retry-success semantics gated by tool/channel policy.

## Risks / Open Questions

- Surfacing raw or loosely summarized tool errors to chat can expose local paths, command output, provider text, or other sensitive data without an allowlist/redaction policy.
- Retry-success confirmation needs precise same-action correlation and should only fire when the original failure was actually surfaced, or it can create noisy or misleading chat messages.
- Open related work is only partial: #60946 is broader model-facing diagnostics, #67276 is a narrow edit mismatch warning, and #54757 covers external-channel suppression rather than operator-facing failure wording.
