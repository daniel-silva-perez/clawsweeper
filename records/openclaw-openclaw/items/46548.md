## Summary

Keep open. Current main still only partially addresses the report: ordinary non-verbose chat-visible tool warnings show a generic failure summary without the underlying reason, and a matching successful retry clears internal state without emitting a user-visible retry-success confirmation. The remaining change is valid but needs maintainer policy because broadening chat-visible diagnostics can leak paths, command output, provider text, or channel-private data.

## What This Changes

The issue requests that chat-visible tool failure warnings include a failure reason and later confirm when a retried tool action succeeds.

## Best Possible Solution

Define sanitized, allowlisted failure summaries by tool and channel, then add retry-success confirmation only when a previously surfaced failure is resolved by a matching successful mutating action. Keep that policy aligned with #60946, the privacy constraints surfaced by #54757, and the narrower edit-mismatch work in #67276.

## Reproduction Assessment

Yes. A high-confidence unit-level reproduction exists by building a verbose-off payload with a mutating `lastToolError` and observing that the warning omits the detail, then simulating a failed edit followed by a matching successful edit and observing that only `lastToolError` state is cleared.

## Solution Assessment

No current implementation fully solves the issue. The safest direction is not raw error surfacing; it is sanitized reason summaries plus explicit retry-success semantics with channel-aware privacy rules.
