## Summary

Keep open. Current main still only partially addresses the request: normal chat-visible tool warnings omit the underlying failure reason unless verbose/detail policy allows it, and a matching successful retry clears internal error state without emitting a user-visible retry-success confirmation. The remaining change is security-sensitive because broader chat-visible tool details can leak local paths, provider output, or command data.

## What This Changes

Request that chat-visible tool failure messages include a useful failure reason and that a later successful retry be confirmed to the operator.

## Best Possible Solution

Define a safe chat-visible diagnostics policy for tool failures, including redacted or allowlisted reason summaries by tool and channel, then add retry-success confirmation only when a previously surfaced failure is resolved by the same successful mutating action. Coordinate the policy with #54757, the richer diagnostics direction in #60946, and the narrower edit-mismatch approach in #67276.

## Reproduction Assessment

Yes. The current unit-level path is high confidence: build a verbose-off payload with a mutating `lastToolError` and the warning omits the detail, then follow the handler retry path and it only clears `lastToolError`. The existing tests already encode both parts of that current behavior.

## Solution Assessment

No current implementation is the full solution, and the closed unmerged PR #46592 is not the best complete fix because it surfaced raw/truncated reasons without resolving retry confirmation or channel privacy policy. A safer solution is sanitized reason summaries plus explicit retry-success semantics gated by channel/tool policy.
