## Summary

Current main still only partially addresses the report. Non-verbose chat warnings for visible mutating tool failures still omit the underlying `lastToolError.error`, while retry handling clears the stored failure when the same mutating action later succeeds instead of sending an explicit retry-success confirmation. The linked implementation PR #46592 is closed unmerged and did not cover retry confirmation.

## Best Possible Solution

Keep this issue open. The best path is a maintainer-defined policy for safe, redacted or allowlisted failure reasons on user-facing chat warnings, plus a decision on whether same-action retry success should remain silent suppression or emit a confirmation only when a failure warning was already delivered.

## Risks / Open Questions

- Surfacing raw `lastToolError.error` in normal chat can expose local paths, command output, provider responses, channel targets, or secrets to Discord/Telegram/WhatsApp-style monitoring surfaces.
- Existing tests intentionally assert that default non-verbose mutating and non-mutating warning summaries omit raw details, so changing this globally may break current privacy and noise expectations.
- Explicit retry-success confirmation could add channel noise unless it is tied only to a previously delivered failure warning and only after a true same-action retry succeeds.
- Related open privacy report #54757 asks for less channel delivery of tool errors, while #60946 asks for richer tool errors; this needs a coherent policy rather than a one-line suffix change.
