## Summary

Keep open. Current main still only partially addresses the report: non-verbose chat-visible tool warnings can still omit ordinary failure reasons, and a matching successful retry clears `lastToolError` state without emitting a user-visible retry-success confirmation. The remaining work is real but security-sensitive because raw tool errors can expose paths, file contents, command output, or provider text to external chat channels.

## What This Changes

The issue requests chat-visible tool failure messages that include a safe failure reason and a confirmation when a retry later succeeds.

## Best Possible Solution

Define a safe chat-visible diagnostics policy for tool failures, using redacted or allowlisted reason summaries by tool and channel, then add retry-success confirmation only when a previously surfaced failure is resolved by a matching successful action. Keep the external-channel privacy work in #54757, richer model-facing diagnostics in #60946, and the narrow edit-mismatch direction in #67276 aligned rather than exposing raw tool errors globally.
