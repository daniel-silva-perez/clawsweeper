## Summary

Current main only partially addresses the report. The retry-noise path is handled by clearing `lastToolError` when the same mutating action later succeeds, but ordinary non-verbose chat warnings still omit the tool error reason. Existing tests explicitly assert that mutating write/edit failure warnings do not contain the failure text. The related PR #46592 tried to add the failure reason but was closed unmerged and explicitly left retry confirmation out of scope, so this issue still tracks real remaining work.

## Best Possible Solution

Keep this issue open. Implement a narrow change in `src/agents/pi-embedded-runner/run/payloads.ts` that includes a sanitized, bounded failure reason for user-visible tool warnings in normal mode, with focused tests updating the current `absentDetail` expectations. Preserve the existing retry-success clearing behavior so transient failed-then-successful mutating retries do not leave stale failure messages.
