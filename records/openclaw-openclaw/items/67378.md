## Summary

Keep this open. Current main still sends proactive WhatsApp phone-number targets through plain PN JID normalization, while LID handling is wired for inbound/reverse lookup and quoted-message preservation only. The issue discussion also says there was no PR, only the reporter's local compiled-bundle workaround.

## Best Possible Solution

Keep this issue open and fix the WhatsApp plugin outbound path. Add an account-aware recipient resolver with access to the active listener/auth directory and Baileys LID mapping, resolve plain phone-number targets to mapped `@lid` JIDs when available, preserve `@s.whatsapp.net` fallback for unmapped numbers, and use the resolved JID consistently for text/media, polls, reactions, composing presence, quoted-message lookup, logging, and returned `toJid`. Add focused regression tests for mapped and unmapped proactive sends.
