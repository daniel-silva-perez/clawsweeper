## Summary

Keep this issue open. Current main still normalizes proactive WhatsApp phone-number targets to PN JIDs in the final Web send API, while the existing LID support is wired for inbound/reverse lookup and preserving already-qualified LID participants. The issue discussion also confirms there was no PR, only the reporter's local compiled-bundle workaround.

## What This Changes

The issue requests outbound WhatsApp phone-number sends to resolve stored LID mappings before falling back to PN JIDs, preventing duplicate ghost chats.

## Best Possible Solution

Add a WhatsApp-plugin-owned, account-aware outbound recipient resolver that can access the active listener's auth directory and Baileys LID mapping store, resolves plain phone-number targets to mapped `@lid` JIDs when available, preserves already-qualified group/LID/PN JIDs, and falls back to PN JIDs for unmapped numbers. Apply it consistently to text/media, polls, reactions, composing presence, quoted-message lookup, logs, and returned `toJid`, with focused regression tests for mapped and unmapped sends.
