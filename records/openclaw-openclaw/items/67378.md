## Summary

Keep this issue open. Current main still routes proactive WhatsApp phone-number sends through plain PN JID normalization, while the existing LID support is inbound/reverse-only. The issue discussion also says there was no PR, only a local compiled-bundle workaround by the reporter.

## Best Possible Solution

Keep this issue open and fix the WhatsApp plugin outbound path. Add an account-aware recipient resolver with access to the active listener/auth directory and Baileys LID mapping, resolve plain phone-number targets to mapped @lid JIDs when available, keep unmapped numbers falling back to @s.whatsapp.net, and route text/media, polls, reactions, composing presence, quoted-message lookup, logs, and returned toJid through the resolved JID. Add focused regression tests for mapped and unmapped proactive sends.
