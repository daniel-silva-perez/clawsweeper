## Summary

Keep this PR open because current main still drops Slack forwarded/shared attachment content from thread-history seeding, and the PR addresses a real context-loss path. The branch needs attention before merge: current main moved the thread-history implementation into `thread.ts`, while the PR patch targets the older `media.ts` location, and the latest PR code still has a small retention/rendering mismatch noted in review comments.

## What This Changes

The PR adds forwarded Slack attachment handling to thread-history context and extends Slack media tests for shared-message text, forwarded-only messages, ignored non-share attachments, and image-only shares.

## Best Possible Solution

Move the PR's thread-history forwarded-attachment logic onto `extensions/slack/src/monitor/thread.ts`, preferably by sharing the existing forwarded attachment formatting/cap semantics rather than duplicating them, and cover the capped readable-share edge case in Slack thread-history tests.

## Reproduction Assessment

Yes. A focused unit reproduction is a mocked `conversations.replies` result with a thread message whose `text` is empty and whose Slack `attachments` contain `is_share: true` plus text/fallback; current main filters that message out in `resolveSlackThreadHistory`.

## Solution Assessment

No. The requested behavior is right, but the current PR is no longer the best merge-ready form because it edits the pre-refactor location and its retention predicate is not fully aligned with the capped rendering range.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Align retention with the rendered forwarded attachments:** `extensions/slack/src/monitor/media.ts:488-492`
  - body: The updated guard scans every shared attachment for readable text, but rendering first slices the shared attachments to `MAX_SLACK_FORWARDED_ATTACHMENTS` and only then extracts text/fallback. If the first eight shared attachments are image-only and a later one has text, this keeps a textless/fileless message but renders no forwarded text, so the blank entry still consumes a history slot and can evict useful messages. Use the same capped readable subset for retention and mapping, or filter readable shares before applying the cap.
  - confidence: 0.86
- **[P2] Rebase the fix onto the thread-history module:** `extensions/slack/src/monitor/media.ts:445`
  - body: Current main now re-exports `resolveSlackThreadHistory` from `extensions/slack/src/monitor/thread.ts`, while this PR changes the old implementation in `media.ts`. As written, the branch is stale against the current layout and will either conflict or miss the live implementation after rebase. Move the forwarded-attachment handling and tests to the current `thread.ts` surface before merge.
  - confidence: 0.82
