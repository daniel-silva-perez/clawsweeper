## Summary

Current main still contains the reported failure path. An explicit `gateway.bind="loopback"` still probes `127.0.0.1`, falls back to `0.0.0.0` when that preflight fails, and then `resolveGatewayRuntimeConfig` rejects that non-loopback result with the exact error reported here. The issue should stay open while the targeted open PR #65635 is reviewed, landed, or deliberately rejected.

## Best Possible Solution

Review and either land or deliberately reject #65635. The expected fix is to keep explicit `gateway.bind="loopback"` pinned to `127.0.0.1` after a failed loopback preflight, while preserving `auto` container behavior, tailnet/custom fallback semantics, and non-loopback auth/origin guards. If it lands, it should include focused regression coverage and a changelog entry with reporter/contributor credit.
