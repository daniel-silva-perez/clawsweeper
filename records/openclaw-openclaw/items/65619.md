## Summary

Current main still contains the reported failure path: explicit `gateway.bind="loopback"` can fall back to `0.0.0.0` when the loopback preflight fails, and runtime config then throws the exact non-loopback bind error. A targeted open PR, #65635, already implements the likely fix, so this issue should stay open until that PR is reviewed, rebased if needed, and either landed or rejected.

## Best Possible Solution

Review and land, or deliberately reject, #65635. The expected fix is to keep explicit `gateway.bind="loopback"` pinned to `127.0.0.1` after a failed loopback preflight while preserving `auto` container behavior, custom/tailnet fallback semantics, and non-loopback auth/origin guards. The fix should include focused regression coverage for the failed preflight path and a changelog entry with proper reporter/contributor credit if it ships.
