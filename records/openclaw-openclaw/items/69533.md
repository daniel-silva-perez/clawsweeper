## Summary

Keep open. Current main still hardcodes the Dream Diary narrative system prompt and user prompt, while the memory-core Dreaming schema, resolver, docs, and Dreams UI expose only enablement/cadence/model/status style controls. The related #65097 covers the language subset and #69354 was closed as a duplicate of that subset, but this issue is broader because it asks for user-customizable prompt/template output with placeholders.

## Best Possible Solution

Implement a memory-core-owned Dream Diary prompt/template contract under plugins.entries.memory-core.config.dreaming. Preserve the current prompt and prompt-builder output as the default, support a small documented placeholder set, thread the resolved config through light/REM/deep narrative generation, expose it through the existing config/UI surface, and update docs, changelog, and focused backend/UI tests.
