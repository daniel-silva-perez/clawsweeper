## Summary

Keep this issue open. Current main still hardcodes the Dream Diary system prompt and user prompt in memory-core, the resolved Dreaming config/schema/docs only cover enablement, cadence, timezone, model/execution, storage, and phases, and the Dreams UI currently exposes status/diary/actions plus the enable toggle rather than prompt editing. The language portion overlaps with #65097, but this issue is broader because it asks for a user-configurable prompt/template with placeholders.

## Best Possible Solution

Keep this open and implement a memory-core-owned Dreaming prompt/template contract. The likely path is to add a documented schema-bounded field under plugins.entries.memory-core.config.dreaming, preserve the current prompt as the default, support a small explicit placeholder set such as date/time/count/themes/snippets/promotions, thread the resolved template into Dream Diary narrative generation, expose it through the Settings/Dreaming config surface, update docs/changelog, and add focused resolver/schema/narrative/UI coverage.
