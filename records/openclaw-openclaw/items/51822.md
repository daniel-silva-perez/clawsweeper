## Summary

Keep this PR open. Current main still accepts credential-bearing cron webhook URLs at the shared normalization boundary, so the PR’s intended hardening is not implemented. The submitted diff is narrow and relevant, but it still needs maintainer follow-up because rejecting already-stored credential URLs would route them through a raw invalid-delivery log path, legacy notify fallback can become silent, doctor migration still copies cron.webhook verbatim, and regression coverage is missing.

## Best Possible Solution

Keep the PR open as a valid hardening candidate, but revise it to reject URL userinfo at normalizeHttpWebhookUrl, redact invalid stored webhook targets before logging, handle or warn on legacy cron.webhook and doctor migration paths, and add focused regression tests for primary delivery plus failure destinations. No dependency or CI supply-chain changes are needed for this fix.

## Risks / Open Questions

- As submitted, already-stored webhook URLs with embedded credentials can become invalid and then be logged through the raw deliveryTo warning path.
- Legacy notify:true jobs using cron.webhook with embedded credentials can silently stop delivering after the normalization change because the legacy fallback has no invalid-target warning path.
- openclaw doctor --fix can migrate a legacy credential-bearing cron.webhook verbatim into delivery.to, creating an invalid webhook job under the proposed validation.
- Without focused regression tests for primary webhook delivery and failureDestination userinfo cases, the security hardening can regress quietly.
