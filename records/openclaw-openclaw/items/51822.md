## Summary

Keep this PR open. Current main still accepts cron webhook URLs with embedded credentials, so the intended hardening is not implemented or superseded. The PR is a focused security-hardening candidate, but the one-file guard still needs redaction, legacy migration/warning behavior, and regression coverage before it is safe to land.

## Best Possible Solution

Keep the PR open as a valid cron webhook hardening candidate. The best path is to reject URL userinfo in the shared cron webhook URL normalizer, redact every invalid or failed webhook log path with the existing sensitive-URL helpers, add explicit warning or migration behavior for legacy `cron.webhook`, prevent doctor from migrating credential-bearing legacy webhook URLs without guidance, and add targeted regression tests. Keep #55365 separate because it covers the browser tool URL surface, not cron webhooks.

## Risks / Open Questions

- Merging the one-file guard as-is can make existing credential-bearing stored webhook URLs invalid and then log the raw URL in the skipped-delivery warning.
- Legacy `notify: true` jobs using `cron.webhook` with URL userinfo can stop delivering without a clear warning, while `openclaw doctor --fix` can still migrate that value into `delivery.to`.
- Without focused tests for userinfo rejection, redaction, runtime stored-job behavior, and legacy doctor migration, this security behavior can regress quietly.
