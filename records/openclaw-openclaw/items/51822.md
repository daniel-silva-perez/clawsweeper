## Summary

Keep this PR open. Current main still accepts cron webhook URLs with embedded credentials through the shared normalizer, and the PR's intended hardening is not implemented or superseded. The proposed diff is narrow and security-relevant, but it needs follow-up for stored/legacy URL redaction, doctor migration behavior, and regression tests before it is safe to land.

## Best Possible Solution

Keep the PR open as a valid hardening candidate. The best fix is to reject URL userinfo at normalizeHttpWebhookUrl, redact invalid stored webhook targets before logging, add a clear legacy cron.webhook warning or migration behavior, prevent doctor from copying now-invalid credential-bearing URLs into delivery.to, and add focused regression tests for primary delivery, failureDestination, and legacy migration paths. The related #55365 tracks a browser-tool surface and does not supersede this cron-specific PR.

## Risks / Open Questions

- As submitted, stored webhook URLs with embedded credentials can become invalid and then be logged through the raw deliveryTo warning path.
- Legacy notify:true jobs using cron.webhook with embedded credentials can silently stop webhook delivery because the legacy fallback has no invalid-target warning path.
- openclaw doctor --fix can migrate a legacy credential-bearing cron.webhook verbatim into delivery.to, creating an invalid webhook job after the proposed validation change.
- Without focused tests, userinfo rejection for primary webhook delivery and failureDestination can regress quietly.
