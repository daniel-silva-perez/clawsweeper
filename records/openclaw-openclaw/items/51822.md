## Summary

Keep this PR open. Current main still accepts cron webhook URLs with URL userinfo, so the hardening is not implemented or superseded, but the one-file guard is not merge-ready because it can log rejected stored credential URLs raw and leaves legacy cron.webhook migration behavior unsafe.

## What This Changes

This PR adds a guard in src/cron/webhook-url.ts so normalizeHttpWebhookUrl returns null when a parsed http(s) webhook URL contains a username or password.

## Best Possible Solution

Reject URL userinfo in the shared cron webhook URL normalizer, but land it together with redacted invalid-target logging, explicit legacy cron.webhook warning or migration guidance, doctor behavior that refuses or warns instead of copying invalid credential URLs, and targeted tests across primary delivery, failure destinations, runtime stored-job dispatch, agent/gateway validation, and doctor migration. Add a user-facing changelog or release-note entry because this intentionally breaks credential-bearing webhook URLs.

## Reproduction Assessment

Yes. Source inspection on current main shows the normalizer accepts credential-bearing http(s) URLs, and the PR's changed null path can be traced to the raw deliveryTo warning for stored webhook jobs without needing live webhook delivery.

## Solution Assessment

No. The shared normalizer is the right seam for the core rejection, but the PR is incomplete until every invalid-url log path is redacted and legacy migration/runtime behavior is handled with tests.

## Security Review

Concerns:
