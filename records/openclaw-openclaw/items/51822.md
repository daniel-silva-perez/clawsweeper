## Summary

Keep this PR open. Current main still accepts cron webhook URLs with embedded credentials, so the intended hardening is not implemented or superseded. The PR is narrowly scoped and security-relevant, but the review discussion correctly identifies follow-up needed around raw logging of newly rejected stored URLs, legacy cron.webhook migration behavior, and regression coverage before this is ready to land.

## Best Possible Solution

Keep the PR open as a valid hardening candidate. The best path is to reject URL userinfo in normalizeHttpWebhookUrl, redact any invalid stored webhook target before logging, add explicit legacy cron.webhook warning or migration behavior, prevent doctor from copying credential-bearing cron.webhook values into delivery.to without guidance, and add focused regression tests for primary delivery, failure destinations, runtime invalid-target warnings, and legacy migration. The related #55365 should remain separate because it covers the browser tool surface, not cron webhooks.

## Risks / Open Questions

- Merging the submitted one-file guard as-is can turn stored credential-bearing webhook URLs into invalid targets and then log the raw delivery.to value on every finished cron run.
- Legacy notify:true jobs that depend on cron.webhook can silently stop webhook delivery after this validation change because legacy fallback invalid-target handling is not surfaced like explicit delivery.mode='webhook'.
- openclaw doctor --fix can migrate a credential-bearing cron.webhook verbatim into delivery.to, creating a now-invalid webhook job through the documented repair path.
- Without focused tests for primary webhook delivery, failureDestination webhooks, runtime stored-job warnings, and doctor legacy migration, this security behavior can regress quietly.
