## Summary

Keep open. The PR is useful for #72074 and the Greptile duplicate-token comment was addressed with `replaceAll`, but the current patch still computes `{agentId}` from the raw tool argument before native and ACP spawn resolution, so default-target spawns produce labels with an empty agent id. It also adds a public config key without the generated config schema/docs/changelog follow-through required for this repository.

## What This Changes

The PR adds optional `labelTemplate` config under `agents.defaults.subagents` and `agents.list[].subagents`, then uses it to synthesize a `sessions_spawn` label when no explicit label is provided.

## Best Possible Solution

Land a narrowed version that applies `labelTemplate` after resolving the effective native or ACP target agent, preserves explicit-label precedence and existing spawn behavior, updates generated config schema/docs/changelog, and adds focused coverage for default, explicit-target, explicit-label, and duplicate-token cases.

## Reproduction Assessment

Yes. Configure `agents.defaults.subagents.labelTemplate` as `{agentId}-{taskPreview}` and call `sessions_spawn` from `agent:main` without `agentId`; the PR computes the label before `spawnSubagentDirect` resolves the omitted target to `main`, so `{agentId}` becomes empty.

## Solution Assessment

No. The maintainable fix is to resolve the effective child target first, or move label-template rendering into the native/ACP spawn paths after target resolution, then share a small helper/test matrix for token rendering and precedence.
