## Summary

Keep this PR open. The branch addresses a valid cron transcript-mirroring bug, but the proposed implementation still misses documented bare custom-session targets and can mirror text that was filtered out of the actual outbound delivery.

## What This Changes

The PR changes isolated cron direct delivery to resolve a delivery session key, pass that session into outbound delivery, add a transcript mirror object, and add a focused custom-session regression test.

## Best Possible Solution

Land a cron-owned fix that resolves destination session targets to the same agent-scoped store key used by the cron run, mirrors the normalized payload projection that was actually delivered, and covers bare `session:custom-id`, fully qualified session keys, and silent-token/media cases.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: documented `session:custom-id` jobs run under an agent-scoped store key, while the PR mirrors to the raw custom id that transcript append cannot find.

## Solution Assessment

No. Using the outbound mirror path is the right direction, but this branch should normalize the mirror key to the stored cron session and derive mirror text/media from the filtered delivery payloads instead of raw output text.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Use the stored cron session key for mirroring:** `src/cron/isolated-agent/delivery-dispatch.ts:585-586`
  - body: For a target like `session:project-alpha-monitor`, the cron run stores and uses `agent:<id>:project-alpha-monitor`, but this helper returns the raw `project-alpha-monitor`. The transcript append path looks up the session store by the key it receives, so documented bare custom-session jobs will still deliver to chat without appending to the destination transcript.
  - confidence: 0.88
- **[P2] Mirror the actual delivered payload:** `src/cron/isolated-agent/delivery-dispatch.ts:617`
  - body: `payloadsForDelivery` strips silent-token text before sending, but the mirror writes `outputText || synthesizedText` and does not pass media URLs. A media delivery after `NO_REPLY` filtering can record hidden text that was not sent and lose the media filename mirror semantics.
  - confidence: 0.82
