## Summary

This PR should stay open: current main does not implement namespaced `/models` grouping, and the PR is useful work, but the latest PR head still has a blocking channel-surface regression where Telegram can bypass the new grouped/scoped text and lose namespace scope during pagination.

## What This Changes

The PR adds parsing and text output for `/models <provider>/<group>`, groups mostly namespaced provider catalogs, preserves flat listing with `all`, and adds command tests for aggregator and local-model catalog behavior.

## Best Possible Solution

Keep the namespace grouping feature, but update the branch so grouped/scoped `/models` either bypasses interactive channel rendering or threads namespace scope through the channel adapter and Telegram callback data. Then add Telegram coverage for grouped catalogs and scoped pagination, and document the new drilldown syntax.

## Reproduction Assessment

Yes. A high-confidence static repro is Telegram `/models litellm` at PR head with multiple namespaced groups: Telegram can return the existing flat interactive keyboard before grouped text is built. A paginated scoped list such as `/models litellm/deepseek` also loses scope because callbacks carry only provider and page.

## Solution Assessment

No. The text-surface implementation is close, but the maintainable fix needs a scope-aware interactive path or an intentional text-only fallback for grouped/scoped views; landing the current ordering would leave Telegram behavior inconsistent with the feature.

## Review Findings
