## Summary

Keep this PR open: current main does not implement namespaced `/models` grouping, but the proposed patch still has a concrete Telegram/channel-regression risk because interactive model-list rendering can bypass the new grouped text and pagination callbacks do not preserve namespace scope.

## What This Changes

The PR adds `/models <provider>/<group>` parsing, grouped text output for namespaced aggregator catalogs, scoped model listings, and colocated auto-reply command tests.

## Best Possible Solution

Keep the namespace grouping feature, but either thread namespace scope through the channel adapter and Telegram callback data or bypass interactive rendering for grouped/scoped views. Add Telegram coverage for grouped catalogs and scoped pagination, and update the slash-command docs for `/models <provider>/<group>`.

## Reproduction Assessment

Yes. A high-confidence static reproduction is Telegram `/models litellm` on the PR branch with multiple namespaced groups: the command can return provider-only interactive channel data before grouped text is built. A scoped list with multiple pages also reproduces scope loss because Telegram pagination callbacks encode only provider and page.

## Solution Assessment

No. The text fallback implementation is close, but the maintainable fix needs namespace-aware channel rendering/callbacks, or an intentional text-only path for grouped and scoped listings.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Thread namespace scope through interactive model lists:** `src/auto-reply/reply/commands-models.ts:608-621`

Scoped pagination also drops the group because callbacks still encode only provider/page. Pass the prefix through the adapter/callback path, or skip interactive rendering for grouped/scoped views.
