## Summary

Keep open. Current main does not implement namespaced `/models` grouping, and the PR is a useful active implementation, but the branch still has a blocking interactive-channel bug: Telegram can return unscoped provider lists before the new grouped text path runs.

## What This Changes

The PR adds `/models <provider>/<group>` parsing, grouped text output for namespaced provider catalogs, a flat `all` escape hatch, and colocated command tests.

## Best Possible Solution

Keep the feature direction, but make namespace scope explicit in the channel list contract and Telegram callback parser/handler, or deliberately bypass interactive rendering for grouped and scoped lists. Add Telegram coverage for grouped catalogs and scoped pagination, then update the slash-command docs for the new syntax.

## Reproduction Assessment

Yes. A high-confidence static reproduction is Telegram `/models litellm` on the PR branch with multiple namespaced groups: `buildModelsListChannelData` can return before grouped text is built. Scoped pagination also reproduces scope loss because `mdl_list_<provider>_<page>` reloads all models for the provider.

## Solution Assessment

No. The text fallback implementation is close, but the maintainable fix needs to thread namespace scope through the interactive channel adapter and Telegram callback data, or avoid the interactive renderer for grouped/scoped views.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve namespace scope in interactive model lists:** `src/auto-reply/reply/commands-models.ts:552`
  - body: This scopes `models` for `/models <provider>/<group>`, but the interactive channel path still receives only `provider` and returns before grouped text runs. Please thread the namespace through the adapter/callback path or bypass interactive rendering for grouped/scoped views.
