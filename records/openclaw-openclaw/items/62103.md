## Summary

Keep open. Current main has independent Google Vertex support for ADC/env credentials, manifest ownership, endpoint classification, and a dedicated Vertex transport path, but it does not clearly implement this PR's OAuth onboarding/location-prompt workflow. The PR is also not merge-ready as submitted because its standalone provider is not declared in the PR-head manifest and its catalog path ignores OAuth-profile auth.

## What This Changes

The PR registers `google-vertex` as a standalone Google provider with OAuth onboarding, Vertex location/project URL helpers, env-driven catalog setup, config normalization changes, setup alias removal, and focused tests.

## Best Possible Solution

Rebase or replace the work against current main, preserve the shipped manifest/ADC/dedicated-transport path, and make an explicit maintainer decision on whether Vertex should also get OAuth onboarding with location collection. If yes, add it with manifest auth-choice metadata, auth-aware catalog discovery, endpoint-routing tests, and docs; if no, close this PR as superseded after documenting the ADC-only product direction.

## Reproduction Assessment

Yes for the review findings: the PR-head manifest and catalog code show the provider ownership and OAuth-catalog gaps without needing live Google credentials. Live endpoint behavior was not reproduced in this read-only sweep.

## Solution Assessment

No. The direction overlaps useful current-main Vertex work, but the PR as submitted is not the narrowest maintainable fix because it bypasses manifest ownership and uses API-key-only catalog gating for an OAuth-oriented provider.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Declare google-vertex in the plugin manifest:** `extensions/google/index.ts:181-182`
  - body: This registers a standalone `google-vertex` provider at runtime, but the PR-head manifest still declares only `google` and `google-gemini-cli`. After removing the setup/runtime hook alias, manifest-driven provider ownership and provider-scoped loads can miss this plugin for `google-vertex`, causing auth or model flows to treat it as unknown until the manifest is updated too.
  - confidence: 0.86
- **[P2] Use auth-aware lookup for the Vertex catalog:** `extensions/google/index.ts:293-295`
  - body: The catalog returns `null` unless `resolveProviderApiKey('google-vertex')` produces an API-key-style value, but OAuth profiles are exposed through `resolveProviderAuth`, not this API-key resolver. A user with a Vertex OAuth profile and project/location config can therefore fail to get the provider config generated; gate this path on auth-aware availability instead.
  - confidence: 0.8
