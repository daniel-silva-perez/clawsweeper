## Summary

Keep this issue open. Current main has compaction continuity mitigations, post-compaction startup refresh, and a CLI-specific next-user-message wrapper, but I did not find a generic embedded/default-session contract or regression that demotes an old compaction summary when a fresh unrelated user question arrives after the compaction boundary.

## Best Possible Solution

Keep this issue open. Add a focused regression for the embedded/default context assembly path: create a transcript with an old-topic compaction summary, append a fresh unrelated user message after the compaction boundary, and assert the assembled prompt/trajectory makes the latest user request the active task while treating the summary as historical background. If it fails, fix the generic prompt/context assembly path rather than adding extension-specific keyword heuristics.
