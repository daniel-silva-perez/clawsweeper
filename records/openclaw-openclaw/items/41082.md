## Summary

Keep this issue open. Current main has useful compaction safeguards and a CLI-only reseed wrapper, but the default embedded/legacy path still passes prior messages through and submits the fresh prompt without a generic guarantee that a new unrelated post-compaction ask is treated as the active task over stale summary context.

## Best Possible Solution

Add a focused embedded/default-session regression that builds a transcript with an old-topic compaction summary, appends a fresh unrelated user prompt after the compaction boundary, and asserts the assembled prompt or trajectory treats the latest user request as the active task while preserving the summary as historical background. If it fails, fix the generic context/prompt assembly path rather than adding channel-specific or keyword-only heuristics; then update the compaction/session docs to make the intended semantics explicit.
