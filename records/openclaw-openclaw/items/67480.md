## Summary

Keep this PR open. Current main still has the nested skills-root bug, and the PR targets the right area, but the latest diff still needs revision because its root-validity probe does not cover documented grouped skills and bypasses existing directory-scan caps.

## What This Changes

The PR adds a root-validity probe to `resolveNestedSkillsRoot()`, threads `maxSkillFileBytes` into that probe, and adds a regression test for a direct root skill alongside a nested `skills/` decoy.

## Best Possible Solution

Revise this PR so nested-root detection first proves the configured directory has at least one loadable direct or one-level grouped skill using the same containment, size, and scan-limit rules as normal discovery, then keep regression coverage for both direct and grouped conflict layouts.

## Reproduction Assessment

Yes. Current main can be reproduced with a configured skills root containing `my-skill/SKILL.md` plus `skills/decoy/SKILL.md`; the existing resolver redirects to `dir/skills` before root discovery, so `my-skill` is skipped. A grouped variant such as `group/my-skill/SKILL.md` plus `skills/decoy/SKILL.md` remains uncovered by the PR test.

## Solution Assessment

No. The proposed direction is right, but the current diff is not the best final fix because the probe does not mirror grouped-skill discovery and scans the root directory without the existing caps. A small shared root-probe helper aligned with actual loader semantics would be safer.
