## Summary

Keep open. Current main has partial mitigations, but the issue’s central recovery problem still remains for same-provider/shared-profile billing fallback chains, and configure/plugin re-auth does not clear stale provider usage state.

## What This Changes

The issue requests recovery from persisted provider auth cooldowns after billing or re-authentication has already been resolved.

## Best Possible Solution

Treat same-provider shared-profile billing fallback chains as having no effective alternate billing fallback for recovery probes, and clear provider/profile-scoped stale usage state after successful configure/onboarding auth refresh while preserving unrelated profile order and cooldown history.

## Reproduction Assessment

Yes. The issue provides step-by-step Anthropic and GitHub Copilot reproductions, and current source/tests corroborate the remaining paths: billing-disabled primaries with fallbacks are skipped far from expiry, and configure-based plugin auth lacks direct-login stale-lockout cleanup.

## Solution Assessment

Yes, with a narrower version of the proposed direction. Targeted billing probes plus provider-scoped cooldown clearing after successful auth refresh are more maintainable than removing all persisted cooldown state because they preserve backoff protection while fixing recovery.
