## Summary

Keep open. Current main has partial mitigation, but the reported recovery gaps remain: billing-disabled profiles in a fallback chain can still be skipped until near expiry, and the configure/plugin auth refresh path can write fresh credentials without clearing the matching stale auth-state usage entry.

## What This Changes

The issue asks OpenClaw to recover provider auth profiles promptly after billing top-up or re-auth instead of leaving persisted `auth-state.json` cooldown or disabled windows blocking runtime use for hours.

## Best Possible Solution

Keep persisted cooldown protection for unresolved provider-health failures, but add targeted recovery: allow throttled billing probes when configured fallbacks do not provide a real alternate billing/auth path, and clear provider/profile-scoped stale usage state after successful configure/onboarding auth refresh while preserving unrelated provider state.

## Reproduction Assessment

Yes. The issue and follow-up comment give concrete Anthropic billing and GitHub Copilot re-auth scenarios, and current source/tests still encode the same remaining behavior: far-from-expiry billing primaries with fallbacks are skipped, while configure/plugin auth refresh lacks the direct-login cleanup path.

## Solution Assessment

Yes. The narrow maintainable direction is targeted recovery rather than dropping persisted cooldowns wholesale: recover user-fixed states promptly while preserving cooldowns for unresolved failures and cross-provider fallback preference.
