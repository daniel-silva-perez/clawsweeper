## Summary

Keep open. Current main still has partial recovery only: billing disable windows are persisted for hours, same-provider/shared-profile billing fallback chains still skip far from expiry, and configure/plugin auth refresh does not clear stale usage state the way direct model auth login does.

## What This Changes

Request prompt recovery from persisted provider auth cooldowns after billing top-up or successful re-authentication, without requiring manual edits to auth-state.json.

## Best Possible Solution

Treat same-provider shared-profile billing fallback chains as having no effective alternate billing fallback for recovery probes, and clear provider/profile-scoped stale usage state after successful configure/onboarding auth refresh while preserving unrelated provider state and normal cooldown backoff for unresolved failures.

## Reproduction Assessment

Yes. The issue gives step-by-step Anthropic and GitHub Copilot reproductions, and current source/tests corroborate the remaining paths: billing-disabled primaries with fallbacks are skipped far from expiry, and configure-based plugin auth lacks direct-login stale-lockout cleanup.

## Solution Assessment

Yes, with a narrow version of the requested direction. Targeted billing probes plus provider-scoped cooldown clearing after successful auth refresh are safer than removing all persisted cooldowns because they preserve backoff protection while fixing recovery.

Fix openclaw/openclaw#70903.
