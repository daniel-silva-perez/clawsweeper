## Summary

Keep open. Current main has partial mitigation for this report, but the remaining lockout behavior is still present: billing-disabled primary profiles with configured fallbacks are skipped until near expiry, and the configure/plugin auth refresh path can upsert fresh credentials without clearing stale usage state.

## What This Changes

The issue requests provider cooldown recovery so persisted auth-state lockouts do not keep users blocked for hours after billing recovery or successful re-auth.

## Best Possible Solution

Keep persisted cooldowns for unresolved provider-health failures, but recover user-fixed states promptly: treat same-provider shared-profile billing chains as having no real alternate billing fallback, and clear provider/profile-scoped stale usage state after successful configure/onboarding auth refresh while preserving unrelated provider state.

## Reproduction Assessment

Yes. The issue and follow-up comment give concrete Anthropic and GitHub Copilot scenarios, and current source/tests still encode the remaining behavior: far-from-expiry billing primaries with fallbacks are skipped, while configure/plugin auth refresh lacks the direct-login cleanup path.

## Solution Assessment

Yes. The maintainable direction is targeted recovery rather than dropping persisted cooldowns wholesale: add same-provider billing recovery and auth-refresh cleanup while preserving cooldown protection for unresolved failures and unrelated fallback providers.
