## Summary

Keep open. Current main has partial recovery mitigations, but it still preserves long billing disable windows and skips far-from-expiry billing probes when configured fallbacks exist, which leaves same-provider/shared-profile fallback chains blocked after billing recovery. The re-auth path is also inconsistent: `models auth login` clears stale provider lockouts, while `openclaw configure` and plugin auth application do not clear matching persisted usage state.

## What This Changes

The issue asks OpenClaw to stop persistent provider cooldown state from blocking users for hours after billing or token recovery, especially for single-provider or shared-profile fallback setups.

## Best Possible Solution

Add a targeted recovery path that treats same-provider shared-profile billing fallback chains as having no effective alternate billing fallback, and clear provider/profile-scoped stale usage state after successful configure/plugin auth refresh while preserving persisted backoff for unrelated profiles and true ongoing failures.

## Reproduction Assessment

Yes. A high-confidence local reproduction can seed `auth-state.json` with a future billing `disabledUntil` for the only profile used by a same-provider fallback chain, then verify current fallback logic skips the primary far from expiry; the issue discussion also gives concrete Anthropic and GitHub Copilot real-world steps.

## Solution Assessment

Yes. A narrow same-provider billing probe plus provider-scoped stale-state cleanup after successful auth refresh is more maintainable than deleting all persisted cooldown state, because it preserves circuit-breaker protection while fixing user recovery.
