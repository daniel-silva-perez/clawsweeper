## Summary

Keep open. Current main partially improves the report by preventing mid-window retry extension, probing billing-disabled primaries when there are no fallbacks, and clearing stale lockouts for direct `models auth login`, but the reported shared-profile/same-provider fallback case still skips far from expiry and configure/plugin auth refresh still does not clear stale usage state.

## What This Changes

The issue requests a reliable recovery path when persisted provider auth cooldown or disabled state remains after billing recovery or successful provider re-authentication.

## Best Possible Solution

Keep persisted cooldowns for unresolved provider health failures, but treat same-provider shared-profile billing chains as having no real alternate billing fallback and clear provider/profile-scoped stale usage state after successful configure/onboarding auth refresh, while preserving unrelated provider state.

## Reproduction Assessment

Yes. The issue gives concrete Anthropic and GitHub Copilot steps, and current source/tests reproduce the remaining behavior: billing-disabled primaries with configured fallbacks are skipped far from expiry, and configure/plugin auth refresh lacks direct-login stale-lockout cleanup.

## Solution Assessment

Yes, the best direction is targeted recovery rather than deleting persisted cooldowns wholesale. Same-provider billing probing plus provider-scoped cleanup after successful auth refresh fixes the user lockout while preserving backoff protection and cross-provider fallback behavior.
