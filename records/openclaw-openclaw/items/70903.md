## Summary

Keep open. Current main has partial mitigations, but the reported recovery failure still exists for same-provider shared-profile billing fallback chains and for configure-based provider re-auth paths that do not clear stale persisted usage state.

## What This Changes

The issue asks OpenClaw to stop stale persisted provider cooldown or disabled state from blocking recovered auth profiles after billing recovery or provider re-auth.

## Best Possible Solution

Finish the recovery path by treating same-provider shared-profile fallback chains as having no effective billing fallback for throttled probes, and clear provider/profile-scoped stale usage state after successful configure/onboarding auth refresh while preserving unrelated profile order and cooldown history.

## Reproduction Assessment

Yes. The issue gives step-by-step Anthropic and GitHub Copilot reproductions, and current source/tests corroborate both shapes: billing-disabled primaries with fallbacks are skipped until near expiry, and configure-based plugin auth lacks the stale-lockout cleanup used by direct model auth login.

## Solution Assessment

Yes. Targeted billing probes plus provider-scoped cooldown clearing after successful auth refresh are the narrowest maintainable fix; removing all persisted cooldown state would overcorrect and weaken retry-storm protection.

Implement two focused behaviors:
1. Billing recovery for same-provider shared-profile fallback chains: when all fallback candidates for a provider effectively share the same disabled auth profile(s) and no usable alternate provider/profile exists, allow a throttled billing probe instead of skipping until near disabledUntil expiry.
