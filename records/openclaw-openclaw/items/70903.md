## Summary

Keep open. Current main has partial recovery work, including immutable active lockout windows and direct `models auth login` cleanup, but the reported user problem still exists for same-provider shared-profile fallback chains and for `openclaw configure --section model` re-auth flows that leave stale auth-state cooldowns behind.

## What This Changes

The issue requests that persisted provider cooldown or disabled state recover promptly after billing top-up or successful re-auth instead of blocking model use for hours.

## Best Possible Solution

Finish the shared auth recovery path: treat same-provider fallback chains that only share disabled profiles as no effective fallback for billing recovery probes, clear stale usage state for the affected provider/profile after successful configure-based re-auth, and keep unrelated auth profile ordering and cooldown history intact.

## Reproduction Assessment

Yes. The issue provides concrete steps and current source/tests corroborate the failure shape: billing-disabled primaries with fallback candidates are skipped until near expiry, and configure re-auth does not call the stale-lockout cleanup used by direct model auth login.

## Solution Assessment

Yes, the narrow maintainable direction is targeted recovery: throttled billing probes for shared-profile single-provider chains plus provider/profile-scoped cooldown clearing after successful re-auth. Removing all persisted cooldown state would overcorrect and could reintroduce provider retry storms.

## Repair Work Prompt

Fix openclaw/openclaw#70903 on current main. Current behavior still blocks users after billing recovery or configure re-auth because billing-disabled same-provider fallback chains sharing the same auth profile are counted as effective fallbacks, and `openclaw configure --section model` re-auth upserts profiles without clearing stale `usageStats` cooldown/disabled state. Implement a narrow fix: in model fallback billing cooldown decisions, treat same-provider fallback candidates whose resolved profile set is the same disabled provider/profile set as no effective fallback for the purpose of the throttled primary billing recovery probe, while keeping cross-provider fallbacks preferred and keeping probe throttling. Also clear cooldown/disabled usage state only for the affected provider/profile IDs after successful provider auth through the configure/plugin auth-choice path, preserving unrelated profiles, `lastGood`, auth order, and unrelated cooldown history. Do not remove persisted cooldowns wholesale. If #74909 has landed, adapt to it; if it is still open, do not duplicate its embedded-runner failure-policy changes except as needed for configure re-auth clearing. Add regression coverage for an Anthropic same-profile Opus fallback billing recovery probe and for GitHub Copilot configure re-auth clearing a stale `format` cooldown. Update docs/changelog if the user-visible recovery behavior changes; changelog credit should include Thanks @mattglover11 and @rocke2020.
