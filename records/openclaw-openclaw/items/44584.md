## Summary

Current main still does not fully resolve the reported Discord thread metadata reinjection or the CLI delivery-context session routing problem. The metadata half overlaps with open #41355, but the CLI routing half is not superseded by that narrower issue, and current source/docs still route `openclaw agent --channel discord --to ...` as delivery metadata rather than as a session lookup by Discord target/account.

## Best Possible Solution

Keep this open. Implement a focused source fix that adds delivery-context session reuse for CLI/gateway agent runs by normalized `{channel, to, accountId}` with Discord bare-ID and `channel:<id>` equivalence, while preserving explicit session selectors. In the same PR or a coordinated companion to #41355, stop Discord thread starter/history and volatile metadata from being re-injected on follow-up turns, with regression tests and a docs update if `--channel` gains session-reuse semantics.
