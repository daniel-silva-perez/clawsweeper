## Summary

Keep this issue open. Current main still drops Feishu/Lark custom sticker context on the inbound agent path: plain text parsing returns only `parsed.text`, standalone `sticker` messages become `<media:sticker>`, and the existing emotion renderer only covers rich-text `post` elements. The provided related Feishu reaction and Telegram custom-emoji items are adjacent but do not cover this inbound Feishu sticker-label gap.

## What This Changes

The issue requests preserving Feishu/Lark custom bracket-style sticker context by converting stickers to Unicode emoji equivalents or stable text labels such as `[OK]` and `[完成]`.

## Best Possible Solution

Fix this in the bundled Feishu plugin by preserving sticker labels or stable text fallbacks from inbound payloads, using Unicode mappings only where the mapping is verified. Add focused Feishu parser/inbound tests for plain text with sticker metadata, standalone `sticker` messages, rich-text `emotion` elements, Chinese bracket labels, and fallback behavior when no label is available.

## Reproduction Assessment

Unclear for the exact live Feishu mixed-payload shape because the issue lacks raw event JSON and no Feishu credentials were available for live replay. Source-level reproduction is clear: current `main` drops anything outside `parsed.text` for text messages and renders standalone stickers as `<media:sticker>`.

## Solution Assessment

Yes, a Feishu plugin parser fix is the right direction; this should not require a core channel API change. The safer implementation is label/text preservation first, with Unicode conversion only for verified mappings.
