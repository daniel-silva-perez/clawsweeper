## Summary

Keep this issue open. Current main still does not preserve Feishu/Lark custom sticker meaning on the inbound agent path: text messages return only the parsed text, standalone sticker messages are reduced to a generic media placeholder, and the rich-text `emotion` renderer only helps for `post` payloads that already carry usable emoji/text fields. The related reports in the provided context cover Feishu reaction API emoji normalization and Telegram custom emoji entities, not this inbound Feishu sticker-label preservation gap.

## Best Possible Solution

Keep this open and fix it in the bundled Feishu plugin. Capture actual Feishu/Lark mixed text plus custom sticker payloads, then extend the inbound parser to preserve stable sticker labels as text and only map to Unicode where the mapping is unambiguous. Add focused tests for plain text with sticker metadata, standalone sticker placeholders, rich-text `emotion` elements, and Chinese bracket labels.
