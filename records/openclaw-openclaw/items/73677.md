## Summary

Keep this open. Current main still does not preserve Feishu/Lark custom sticker labels in the agent-visible inbound message, and the remaining blocker is confirming the raw Feishu payload fields and the desired mapping policy before changing parser behavior.

## What This Changes

The issue requests Feishu/Lark inbound custom stickers such as [OK], [完成], and [调皮] be preserved for the agent as Unicode emoji or bracketed text.

## Best Possible Solution

Implement this in the Feishu plugin inbound parser by preserving verified sticker labels or stable text fallbacks from raw event payloads, then add focused parser and dispatch tests for mixed text plus sticker metadata, standalone sticker messages, rich-text emotion elements, Chinese bracket labels, and unknown-label fallback behavior.

## Reproduction Assessment

Unclear for the exact live Feishu event because the report does not include raw payload JSON. Source inspection gives a high-confidence static path: current main returns only parsed.text for text messages and <media:sticker> for standalone sticker messages.

## Solution Assessment

Yes for the direction: a Feishu plugin parser fix is the narrowest maintainable solution and should not require a core channel API change. Unicode mapping should wait for verified Feishu names or an explicit maintainer mapping policy.
