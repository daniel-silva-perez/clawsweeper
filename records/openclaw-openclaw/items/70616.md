## Summary

Keep open. Current main still only partially addresses the request: it supports `dreaming.model`, `dreaming.execution.defaults.model`, and per-phase `execution.model`, but the requested direct `dreaming.execution.model` shape and configured fallback model chains are absent from strict schema, resolver types, runtime retry order, docs, and tests.

## What This Changes

The issue asks to add an `execution` section to memory-core dreaming config so Dream Diary can run on cheaper background models and use configured fallback models.

## Best Possible Solution

Keep this open until maintainers choose the Dreaming model API. If accepted, implement a backward-compatible fallback surface across the manifest, resolver, Dream Diary attempt ordering, trust/allowlist behavior, docs, and focused tests; otherwise document `dreaming.model` plus session-default retry as the supported behavior.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: the strict memory-core manifest rejects the requested direct/global keys, and the resolver/runtime only support one model followed by a session-default retry.

## Solution Assessment

No. The current partial `dreaming.model` implementation is useful for cheaper primary Dream Diary runs, but it does not solve configured fallback chains or the direct shape requested here; a maintainable fix needs the API decision first.
