## Summary

Keep open. Current main still has a partial model override implementation: strict memory-core schema accepts `dreaming.model`, `dreaming.execution.defaults.model`, and per-phase `execution.model`, while the requested direct `dreaming.execution.model` shape and configured fallback chains are still absent from schema, resolver types, runtime attempts, docs, and tests.

## What This Changes

Request to let memory-core dreaming config choose cheaper background Dream Diary models and define fallback models for dreaming execution.

## Best Possible Solution

Keep this open for a maintainer API decision. If accepted, add a backward-compatible configured fallback surface, align the strict plugin manifest, resolver types, Dream Diary model attempt order, trust/allowlist handling, docs/config reference, and focused tests; otherwise document that `dreaming.model` plus session-default retry is the canonical supported behavior.

## Reproduction Assessment

Yes. The current manifest and resolver provide a high-confidence static reproduction: the requested `dreaming.execution.model` and fallback list are not schema properties under strict `additionalProperties: false`, and runtime/tests only retry with the session default.

## Solution Assessment

No. The current `dreaming.model` and `execution.defaults.model` behavior is a useful partial workaround, but it does not solve configured fallback chains or the direct shape requested in the issue; a maintainable fix needs the public API shape chosen first.
