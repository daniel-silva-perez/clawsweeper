## Summary

Keep this issue open. Current main partially addresses the cost-control part by adding `dreaming.model`, `dreaming.execution.defaults.model`, per-phase `execution.model`, and a retry to the session default when the configured Dream Diary model is unavailable. It still does not implement the requested direct `dreaming.execution.model` shape or explicit configured fallback model chains; docs now describe `dreaming.model` and say fallback is only a session-default retry, while the plugin subagent runtime still exposes `model` but no `fallbacks` parameter.

## Best Possible Solution

Keep this issue open for a maintainer/API decision on the remaining fallback-chain behavior. The likely product path is to either bless the current `dreaming.model` surface plus add a compatible `dreaming.fallbacks` or `dreaming.execution.defaults.fallbacks` shape, or explicitly decide that retry-to-session-default is sufficient and close with that scope. If explicit fallbacks are supported, update the memory-core manifest, resolver types, generated config/help/docs, Dream Diary narrative runtime wiring, and tests while preserving plugin subagent trust/allowlist checks.
