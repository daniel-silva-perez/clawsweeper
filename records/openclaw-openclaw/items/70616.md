## Summary

Current main only partially satisfies this request. It supports `dreaming.model`, `dreaming.execution.defaults.model`, per-phase `execution.model`, and a retry to the session default when the configured Dream Diary model is unavailable. It still does not support the requested direct `dreaming.execution.model` shape or user-configured fallback model chains, and the remaining work needs a maintainer API decision.

## Best Possible Solution

Keep this issue open for maintainer/API decision. A good resolution would either explicitly bless the current `dreaming.model` plus session-default retry as sufficient, or add a backward-compatible configured fallback surface such as `dreaming.execution.defaults.fallbacks` and possibly a direct `dreaming.execution.model` alias, then update the memory-core manifest, resolver types, Dream Diary runtime wiring, plugin subagent/gateway contract if needed, docs, and focused tests.
