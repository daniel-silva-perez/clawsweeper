## Summary

Keep open. Current main still has the divergent lifecycle paths described in the report: plugin SDK subagent runs dispatch the Gateway `agent` method and get task/abort/plugin-owner handling, while `subagent_ended` is emitted only from registry-backed `SubagentRunRecord` completion. I found no current-main bridge or regression test that makes `api.runtime.subagent.run()` participate in that registry lifecycle.

## What This Changes

The issue asks for `api.runtime.subagent.run()` plugin SDK runs to fire registered `subagent_ended` hooks when the child run completes.

## Best Possible Solution

Add a narrow Gateway/plugin-runtime lifecycle bridge so plugin SDK subagent runs create or attach to a registry-backed completion record before dispatch, then complete through the existing registry lifecycle exactly once. Preserve plugin owner scoping, idempotency, abort/task tracking, delete-session permissions, and existing `sessions_spawn` behavior; add focused regression coverage for a plugin `api.runtime.subagent.run()` plus `api.on("subagent_ended")` flow.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: call `api.runtime.subagent.run()`, follow it through `createGatewaySubagentRuntime().run()` into the Gateway `agent` handler, and observe that no `SubagentRunRecord` is created, while the only `subagent_ended` emission path requires such a record. I did not run a live/plugin test because this review was required to keep the checkout read-only.

## Solution Assessment

Yes, directionally. The suggested registry registration/bridge is the narrowest maintainable fix, but it should be implemented as a scoped plugin-runtime lifecycle path rather than a broad change to every Gateway `agent` RPC run.
