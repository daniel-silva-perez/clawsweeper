## Summary

Keep open. Current main still routes `api.runtime.subagent.run()` through the generic gateway `agent` RPC, while the natural `subagent_ended` lifecycle is driven by `SubagentRunRecord` entries that are only registered by the subagent tool/spawn paths.

## What This Changes

Ensure `subagent_ended` hooks fire for plugin SDK subagent runs launched through `api.runtime.subagent.run()`.

## Best Possible Solution

Add a narrow registry-backed lifecycle path for plugin SDK subagent launches so natural completion emits `subagent_ended` once, while preserving existing `sessions_spawn` behavior, plugin-owned session cleanup policy, and generic agent RPC semantics.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: register a plugin `subagent_ended` hook, call `api.runtime.subagent.run()`, and wait for `agent.wait`; current main completes through gateway dedupe/task state without ever registering a subagent run record.

## Solution Assessment

Yes, with a small refinement. The suggested fix direction is right, but the maintainable implementation should reuse or factor the existing subagent registry lifecycle rather than duplicating hook emission in the gateway agent handler.
