## Summary

Keep this PR open. Current main still forwards the SSRF guard dispatcher on the Slack inbound media path, and the current test suite asserts that dispatcher is preserved. The PR is not implemented on main and remains a plausible implementation candidate for the still-open Slack attachment failure, but it changes the SSRF/pinned-DNS transport boundary and needs maintainer review rather than cleanup closure.

## Best Possible Solution

Keep this PR open for maintainer review. Rebase it onto current main, update the raw-fetch allowlist line numbers, reproduce or disprove the linked Slack attachment failure against the current Undici runtime path, then either land a Slack-specific dispatcher-stripping fix with a regression test that locks the chosen SSRF contract or close it with an explicit maintainer decision to preserve pinned dispatcher enforcement.
