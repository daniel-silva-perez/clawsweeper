## Summary

Keep this PR open. Current main still does not implement the dispatcher-stripping change, the linked Slack attachment bug remains open, and the patch touches the SSRF/pinned-DNS transport boundary enough that it should receive maintainer review rather than conservative cleanup closure.

## What This Changes

The branch strips `dispatcher` from Slack media fetch init before routing to runtime/global fetch, updates the Slack media regression assertion, and adjusts raw-fetch allowlist line numbers.

## Best Possible Solution

Keep the PR open, rebase it onto current main, and make an explicit maintainer decision on Slack's SSRF contract: either land a Slack-specific dispatcher-stripping/preflight-only pattern with a regression test and updated raw-fetch allowlist, or fix the Undici incompatibility while preserving pinned dispatcher enforcement.

## Reproduction Assessment

Yes for the reported behavior path: #63905 gives concrete Slack container/sandbox steps and logs, and current main still routes Slack inbound media through `createSlackMediaFetch` with an SSRF dispatcher. I did not run a live Slack workspace/container repro in this read-only review.

## Solution Assessment

Unclear. The PR is narrow and has a BlueBubbles precedent, but it strips the guard-created dispatcher from the actual request; the safer maintainable path is an explicit, reviewed Slack transport contract rather than an implicit dispatcher drop.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Preserve the guarded dispatcher contract:** `extensions/slack/src/monitor/media.ts:88-90`
  - body: This removes the `dispatcher` that `fetchWithSsrFGuard` just attached before making the actual Slack media request. Please either preserve a dispatcher-compatible fetch path or encode an explicitly reviewed preflight-only/pinDns=false contract for Slack.
