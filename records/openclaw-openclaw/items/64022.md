## Summary

Keep this PR open. Current main still forwards the SSRF guard dispatcher on the Slack inbound media path, and the current test suite asserts that behavior. The PR remains a plausible implementation candidate for the still-open Slack attachment bug, but it changes a pinned-DNS/SSRF transport boundary and needs maintainer review rather than conservative cleanup closure.

## Best Possible Solution

Keep this PR open for maintainer review. Rebase it onto current main, update the raw-fetch allowlist line numbers, reproduce or disprove #63905 against the current runtime Undici path, then either land a Slack-specific fix with a regression test that locks the chosen dispatcher/SSRF contract or close it with an explicit maintainer decision that Slack media must preserve pinned dispatcher enforcement.
