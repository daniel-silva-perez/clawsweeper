## Summary

Keep this PR open. Current main still does not implement `gateway.auth.scopes`, still clears scopes for device-less shared-token/password WebSocket sessions, and still has tests/docs describing the current no-scopes behavior. The PR remains a plausible gateway auth feature, but it needs a rebase and security-focused tightening before merge.

## Best Possible Solution

Keep this PR open for contributor or maintainer follow-up. The best path is to rebase onto current main, add `gateway.auth.scopes` through the config type/schema/help/labels/docs, carry it through `auth-resolve.ts`, restore configured scopes only for device-less connections whose actual auth method is token or password, and add focused gateway tests for configured scopes, default cleared scopes, and trusted-proxy/Tailscale non-elevation.

## Risks / Open Questions

- The PR touches a security-sensitive gateway auth path by granting operator scopes to sessions without paired device identity.
- The live PR head uses `sharedAuthOk` as the regrant guard; on current main `sharedAuthOk` includes successful trusted-proxy auth, so a rebase must restrict scope restoration to actual token/password auth or explicitly exclude trusted-proxy/Tailscale paths.
- The submitted diff has no regression tests for configured scopes, absent scopes, or trusted-proxy/Tailscale non-elevation.
- The PR is stale against the current `auth-resolve.ts` resolver split and would need conflict-aware rework before it can be reviewed as mergeable.
