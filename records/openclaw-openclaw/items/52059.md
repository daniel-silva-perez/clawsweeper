## Summary

Keep this PR open. Current main still does not implement the requested configured scope regrant, and the submitted branch is stale against the current gateway auth resolver while also retaining a token/password guard bug that can elevate trusted-proxy sessions.

## What This Changes

The PR adds optional `gateway.auth.scopes` config metadata and resolver/handshake plumbing to regrant configured scopes to device-less shared-auth WebSocket connections.

## Best Possible Solution

Rebase the feature onto current main, add `gateway.auth.scopes` through the config type, strict schema, labels/help, generated docs surface, and `auth-resolve.ts`, then restore configured scopes only for device-less sessions whose actual auth method is token or password. The final patch should include focused tests for configured regrant, default cleared scopes, trusted-proxy non-elevation, and Tailscale non-elevation.

## Reproduction Assessment

Yes. Current-main code and tests provide a high-confidence reproduction path: connect as an operator with valid shared token/password and no device identity, request privileged scopes, and observe `hello-ok.auth.scopes` is empty and privileged RPCs fail with missing scope.

## Solution Assessment

No. The feature direction may be maintainable, but this submitted patch is not the best current solution because it targets the old resolver shape and uses `sharedAuthOk`, which also covers trusted-proxy auth; the safer fix is an `authMethod === "token" || authMethod === "password"` gate after rebase.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Restrict scope regranting to token/password auth:** `src/gateway/server/ws-connection/message-handler.ts:544`
  - body: `sharedAuthOk` is also true for successful trusted-proxy auth, so this block can grant `gateway.auth.scopes` to device-less trusted-proxy Control UI sessions. That contradicts the token/password-only contract and can unexpectedly elevate those sessions to scopes like `operator.admin`.
  - confidence: 0.95

## Security Review

Status: needs_attention

Summary: The PR has a concrete gateway-auth security concern because the configured scope regrant is gated by `sharedAuthOk`, which also includes trusted-proxy auth.

Concerns:

- **[medium] Trusted-proxy sessions can receive token scopes:** `src/gateway/server/ws-connection/message-handler.ts:544`

  - confidence: 0.95
