## Summary

Keep this PR open. Current main still lacks the proposed `gateway.auth.scopes` config, resolver plumbing, runtime regrant, docs, and regression tests. The PR remains a plausible gateway auth feature, but the live head is stale against the current `auth-resolve.ts` split and still needs a security-tight rebase before it can be considered mergeable.

## Best Possible Solution

Keep this PR open for contributor or maintainer follow-up. The best path is to rebase onto current main, add `gateway.auth.scopes` through the config type/schema/generated hints/docs, carry it through `auth-resolve.ts`, restore configured scopes only for device-less sessions whose actual auth method is token or password, and add focused gateway tests covering configured scopes, absent scopes, and trusted-proxy/Tailscale non-elevation.

## Risks / Open Questions

- The PR touches a security-sensitive gateway auth path by granting operator scopes to sessions without paired device identity.
- The live patch gates scope restoration on `sharedAuthOk`; on current main that includes trusted-proxy auth, so a rebase must restrict restoration to actual token/password auth or explicitly exclude trusted-proxy and Tailscale paths.
- The PR currently has no regression tests for configured scopes, default cleared scopes, or trusted-proxy/Tailscale non-elevation.
- The branch is stale against the current `auth-resolve.ts` resolver split and would need conflict-aware rework before merge review.
