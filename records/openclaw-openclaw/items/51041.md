## Summary

Keep this issue open. Current main still does not expose Discord slash interaction response controls to `api.registerCommand` handlers: the plugin command context is generic, Discord native commands defer before dispatch, handlers return a `ReplyPayload`, and Discord delivery applies the configured slash-command ephemeral mode. The existing Discord `respond` helper is only wired to component/modal interactive handlers, not native plugin commands.

## Best Possible Solution

Keep this issue open and design an additive, typed Discord responder seam for native `api.registerCommand` handlers, likely modeled on the existing Discord interactive `respond` helper but injected only after OpenClaw authorization and route resolution. Avoid exposing raw Discord objects, preserve existing command auth/policy behavior, support defer/reply/followUp/edit semantics with per-message ephemeral flags, and add focused Discord native command tests plus public plugin docs.

## Risks / Open Questions

- Exposing raw Discord/Carbon interaction objects would couple third-party plugins to channel internals and could bypass OpenClaw auth, routing, expiry, or delivery accounting unless wrapped in a narrow SDK responder abstraction.
- Discord interaction lifecycle rules make double-response and expired-interaction handling easy to regress; coverage should include guild channels, group DMs, deferred interactions, mixed ephemeral/public follow-ups, and edit behavior.
- This would expand the public plugin SDK, so the implementation should update types, API baseline/export checks, plugin docs, and Discord native command tests together.
