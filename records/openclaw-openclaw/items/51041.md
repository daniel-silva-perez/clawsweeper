## Summary

Keep this issue open. Current main still does not expose Discord slash interaction response controls to `api.registerCommand` handlers: the command context is generic, Discord native commands defer the interaction before dispatch, plugin handlers return a `ReplyPayload`, and the Discord channel delivers that payload using the configured slash-command ephemeral mode. Existing Discord component interactive handlers have a responder API, but that separate surface is not wired into native plugin command handlers.

## Best Possible Solution

Keep the issue open and design a small typed Discord responder seam for native `api.registerCommand` handlers, likely modeled after the existing Discord interactive `respond` helper but scoped to command interactions. Preserve current authorization and route resolution before handler execution, avoid exposing raw Carbon objects, support explicit defer/reply/followUp/edit semantics with per-message ephemeral flags, and add Discord native command tests for mixed private initial response plus public follow-up in guild channels and group DMs. Update plugin SDK types and public docs once the API shape is settled.

## Risks / Open Questions

- Exposing raw Discord/Carbon interaction objects would couple third-party plugins to channel internals and could bypass OpenClaw auth, routing, and delivery accounting if not wrapped in a narrow SDK responder abstraction.
- A Discord-specific command context field would expand the public plugin SDK, so it needs backwards-compatible typing, docs, and regression coverage for guild channels, group DMs, deferred interactions, and expired interaction handling.
