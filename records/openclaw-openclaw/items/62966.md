## Summary

Current main still does not expose `ReplyToId` or `ReplyToBody` to plugin `inbound_claim` handlers. The PR remains a narrow additive implementation candidate for a real plugin hook API gap, and the provided diff does not introduce an observed CI, dependency, package, or supply-chain security concern.

## Best Possible Solution

Keep this PR open and review it as a small plugin hook API fix. The best path is to add optional `replyToId` and `replyToBody` to the canonical inbound hook context, map them from `FinalizedMsgContext`, expose them in `inbound_claim` metadata, and add a focused regression test in `src/hooks/message-hook-mappers.test.ts`.
