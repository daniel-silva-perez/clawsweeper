## Summary

Keep this PR open. Current main still constructs the resolveExtraParams per-model lookup key with raw `${provider}/${modelId}`, so provider-prefixed IDs such as `openrouter/auto` still miss canonical `agents.defaults.models["openrouter/auto"].params`. The closed source PR #44319 and this replacement PR both target that same narrow bug with a canonical `modelKey` plus legacy fallback approach and regression coverage.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to repair mergeability if needed, land the narrow `resolveExtraParams` change using `modelKey` plus `legacyModelKey` fallback, keep the canonical `openrouter/auto` and legacy double-prefixed regression coverage, preserve @HenryXiaoYang's source PR credit, and avoid unrelated workflow, dependency, or release-script changes.
