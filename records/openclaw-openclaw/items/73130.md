## Summary

Keep open. Current main still lacks the proposed credential blast-radius guard, but the PR is not mergeable as-is because the new destructive-command matcher misses common split-flag root deletion forms and the approval path has a malformed-response diagnostic edge case.

## What This Changes

The PR adds a names-only credential blast-radius classifier, hook-level guard, focused tests, and an audit note that block selected destructive calls or route approval-required calls with credential-name metadata.

## Best Possible Solution

Land a maintainer-reviewed version that uses a robust rm argument parser or equivalent matcher, handles malformed approval decisions with accurate unavailable-route messaging, and updates the exec-approval policy/docs to describe the new destructive-action guard.

## Reproduction Assessment

Yes. The main correctness issue reproduces by evaluating the proposed `shell/rm-rf-root` regex against `rm -r -f /`, `rm -v -r /`, or `rm -r -v /*`; those recursive root deletion forms are not classified. Current main cannot reproduce the new guard behavior because the classifier is absent.

## Solution Assessment

No. The hardening direction is reasonable, but this revision is not the best mergeable solution until the destructive command matcher covers common catastrophic forms and the approval/policy surface is reviewed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Parse split rm flags before root targets:** `src/agents/credential-blast-radius-classifier.ts:330-331`
  - body: The new root/home deletion guard misses common catastrophic commands when non-recursive flags are separate tokens before the protected target. `rm -r -f /`, `rm -v -r /`, and `rm -r -v /*` do not match this pattern, so the guard can allow the root wipe it is meant to block. Parse the rm argv or allow arbitrary flag groups while independently requiring a recursive flag before the protected target.
  - confidence: 0.91
- **[P3] Treat undefined approval decisions as unavailable:** `src/agents/pi-tools.before-tool-call.ts:452-453`
  - body: If the approval request response has its own `decision` property with value `undefined`, this path skips the no-route branch and later reports a timeout. That is fail-closed, but it gives the wrong operator/debug signal for an unavailable approval route; handle `null` and `undefined` together here.
  - confidence: 0.82
