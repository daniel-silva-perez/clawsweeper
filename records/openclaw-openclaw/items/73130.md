## Summary

Keep open. Current main does not already contain this guard, and the PR is useful security hardening work, but it is not merge-ready because the new catastrophic `rm` matcher misses common split-flag forms and the core exec-policy impact needs maintainer review.

## What This Changes

The PR adds a names-only credential blast-radius classifier, tests, an audit note, and before-tool-call guardrails that block or route selected destructive tool calls through approval metadata.

## Best Possible Solution

Land a maintainer-reviewed version that closes the split-flag `rm` bypass, handles malformed approval decisions with accurate unavailable-route messaging, and updates the exec-approval documentation or policy model to describe the new destructive-action guard.

## Reproduction Assessment

Yes. The main correctness issue reproduces by checking the proposed regex against `rm -r -f /` or `rm --force -r /`; those commands are not classified even though they recursively target root. Current main cannot reproduce the new guard behavior because the classifier and guard are absent there.

## Solution Assessment

No. The hardening direction is reasonable, but the current patch is not the best mergeable solution until the command parser/matcher covers common catastrophic forms and the policy/docs interaction is reviewed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Catch split rm flags before root targets:** `src/agents/credential-blast-radius-classifier.ts:330`
  - body: The new root/home deletion guard misses common catastrophic commands when a non-recursive flag appears before the target. For example, `rm -r -f /` and `rm --force -r /` do not match this pattern, so the guard allows exactly the class of root wipes it is meant to block. Parse the rm arguments or allow arbitrary flag groups while separately requiring a recursive flag before the protected target.
  - confidence: 0.91
- **[P3] Treat undefined approval decisions as unavailable:** `src/agents/pi-tools.before-tool-call.ts:452-453`
  - body: If the approval request response includes a `decision` property with value `undefined`, this path skips the no-route branch and later reports a timeout. That is still fail-closed, but it gives the wrong operator/debug signal for an unavailable approval route; handle `null` and `undefined` together here.
  - confidence: 0.82
