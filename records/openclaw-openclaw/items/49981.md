## Summary

Keep this PR open. Current main still does not implement the PR’s intended HTTP-level timeout threading for remote embedding batch upload, create, status, error-read, and result-download calls. The PR is stale against the current memory-host SDK/plugin layout and has review feedback that should be addressed in a fresh port, but the underlying bug is still present.

## Best Possible Solution

Keep this PR open as useful but stale work. The best maintainer path is a fresh port that adds optional timeoutMs or shared deadline-budget support to packages/memory-host-sdk/src/host/{remote-http,post-json,batch-http,batch-upload}.ts, threads the remaining budget through extensions/{openai,voyage,google}/embedding-batch.ts upload/create/status/error-read/download calls, handles aborts without resubmitting already accepted batches, removes unrelated churn, and adds focused regression tests for SDK forwarding plus provider batch wait/download paths.

## Risks / Open Questions

- The PR branch appears stale against current main because the active implementation is now split across packages/memory-host-sdk and provider plugins, while the PR diff targets older src/memory paths plus unrelated formatting/test-support churn.
- A direct port that retries the whole batch operation on AbortError can duplicate provider jobs or leave orphaned remote files after upload/create has already succeeded.
- A correct fix needs one remaining deadline budget across upload, create, polling, terminal error reads, result downloads, and retry attempts; passing the original timeout to each phase can still exceed the documented full-operation timeout.
- The timeout port must preserve fetchWithSsrFGuard, SSRF policy, and proxy handling; bypassing the shared helper would be a security regression.
