## Summary

Keep this PR open. It targets a real regression linked to #74868, but the updated branch still computes `message` availability from only part of the runtime tool policy model, so some group/channel turns can remain suppressed while the agent still lacks the `message` tool.

## What This Changes

The PR adds `messageToolAvailable` plumbing to source reply delivery-mode resolution, computes message availability in dispatch, exposes the option through the plugin SDK helper, adds resolver unit tests, and includes an unrelated generated Persian locale correction.

## Best Possible Solution

Keep the fallback behavior, but base it on the same effective availability model as the runtime tool inventory, or move the decision to a boundary that already has provider/model, group, sandbox, subagent, and runtime-disable context. Add dispatch-level coverage for at least one provider or group/sandbox allowlist that excludes `message`.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a group/channel turn with default visible replies plus a provider-specific, group, sandbox, or subagent allowlist that excludes `message`; the PR guard can still return available while `createOpenClawCodingTools` filters the tool out.

## Solution Assessment

No. Falling back to automatic delivery is the right product behavior, but this implementation is not the best fix until it shares the full runtime tool availability path instead of approximating it in dispatch.
