## Summary

Keep this PR open, but do not merge it as-is. The linked bug in #74868 is real on current main, and the branch is useful, but the proposed guard misses channel-provided requested modes and uses only a subset of the runtime tool-policy stack.

## What This Changes

The PR adds message-tool availability plumbing to source reply delivery-mode resolution, auto-reply dispatch, the channel reply SDK helper, resolver tests, and one Persian locale string correction.

## Best Possible Solution

Keep the fallback behavior, but apply it after requested mode resolution and derive message-tool availability from the actual runtime inventory or an equivalent full-policy helper. The fix should preserve intentional sendPolicy denial and include dispatch-level coverage for Discord/Slack requested modes plus provider, group, sandbox, subagent, and disableMessageTool filtering.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a Discord or Slack group/channel turn with default visible replies and a policy path that removes message; current main resolves message_tool_only, suppresses automatic source delivery, and the agent lacks the tool needed to send visible output.

## Solution Assessment

No. Falling back to automatic delivery is the right direction, but this PR is not the best fix yet because it misses requested channel modes and does not use the same effective availability as the runtime tool inventory.
