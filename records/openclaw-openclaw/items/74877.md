## Summary

Keep this PR open. It addresses a real reply-delivery regression tracked by #74868, but the current branch is not merge-ready because the fallback can be bypassed by channel-provided delivery modes and its availability check does not match the runtime tool-policy stack.

## What This Changes

The PR adds a `messageToolAvailable` option to source reply delivery-mode resolution, computes message tool availability in auto-reply dispatch, forwards the option through the channel reply SDK helper, adds resolver tests, and changes one Persian locale string.

## Best Possible Solution

Keep the fallback behavior, but derive it from the same effective tool availability used to build the runtime tool inventory or move the decision to a boundary that already has provider/model, channel, group, sandbox, subagent, and runtime-disable context. The fix should cover channel-computed requested modes and preserve intentional `sendPolicy: deny` and suppression semantics.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a Discord or Slack group/channel turn with default visible replies and a policy path that removes `message`; current main computes `message_tool_only`, suppresses automatic delivery, and the agent lacks the tool needed to send visible output.

## Solution Assessment

No. Falling back to automatic delivery is the right product direction, but this PR is not the best fix yet because it does not handle requested channel modes and checks only part of the runtime tool-policy stack.

  - confidence: 0.9
