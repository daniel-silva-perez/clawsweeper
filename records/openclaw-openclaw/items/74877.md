## Summary

Keep this PR open. The fallback behavior is the right direction and the second commit addressed the earlier profile-only gap, but the branch still checks only part of the tool-availability contract, so current configurations can still suppress automatic delivery while the runtime tool inventory withholds `message`.

## What This Changes

The PR adds `messageToolAvailable` plumbing to source reply delivery-mode resolution, computes message-tool availability during dispatch, passes the option through the plugin SDK helper, and adds unit coverage for the fallback behavior.

## Best Possible Solution

Update the branch so message-tool fallback is based on the same effective availability model as the runtime tool inventory, or move the decision to a point that has the full provider/model, group, sandbox, subagent, and runtime-disable context. Add dispatch-level regression coverage for at least one provider or sandbox/group allowlist that excludes `message`.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a group/channel turn with default visible replies plus a provider-specific, group, or sandbox tool allowlist that excludes `message`; the PR guard can report `message` available while `createOpenClawCodingTools` filters it out later.

## Solution Assessment

No. Falling back to automatic delivery is the right product behavior, but this implementation should reuse or share the full effective-tool policy path instead of approximating availability at dispatch with a partial policy list.
