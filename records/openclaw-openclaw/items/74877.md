## Summary

Keep this PR open. The fallback behavior is needed, but the proposed patch still misses the main channel path because explicit `sourceReplyDeliveryMode` values bypass the new availability fallback, and the dispatch-side availability check does not match the full runtime tool filtering stack.

## What This Changes

The PR adds `messageToolAvailable` plumbing to source reply delivery-mode resolution, computes message-tool availability in auto-reply dispatch, exposes the option through the channel reply SDK helper, adds resolver tests, and includes a generated Persian locale string correction.

## Best Possible Solution

Keep the fallback behavior, but derive it from the same effective tool availability used by the runtime tool inventory, or move the decision to a boundary that already has channel/plugin, provider/model, group, sandbox, subagent, and runtime-disable context. The fix should cover config-derived channel modes as well as dispatch defaults, and should preserve `sendPolicy: deny` and intentional suppression semantics.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a Discord or Slack group/channel turn with default visible replies plus a tool policy stage that removes `message`; the channel computes `message_tool_only` before dispatch, and the PR's dispatch guard cannot override that requested mode.

## Solution Assessment

No. Falling back to automatic delivery is the right product behavior, but this implementation is not the best fix because it checks only a subset of policy inputs and does not handle channel-provided `sourceReplyDeliveryMode` values that are already `message_tool_only`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Let unavailable tools override explicit channel modes:** `src/auto-reply/reply/source-reply-delivery-mode.ts:16-17`
  - body: `resolveSourceReplyDeliveryMode` still returns `params.requested` before the new availability fallback. Discord and Slack compute `message_tool_only` first and pass it into dispatch as `replyOptions.sourceReplyDeliveryMode`, so even when the new dispatch check finds `message` unavailable, the resolver keeps suppressing automatic delivery. Move the availability fallback after config-derived requested modes, or stop passing those computed modes as authoritative requests.
  - confidence: 0.94
- **[P2] Check the full runtime tool policy stack:** `src/auto-reply/reply/dispatch-from-config.ts:596-607`
  - body: The new availability check only evaluates profile/provider-profile/global/agent policies and calls `resolveEffectiveToolPolicy` without provider/model context. Runtime tool creation also applies provider, agent-provider, group, sandbox, and subagent policies, so those configurations can still remove `message` while this guard reports it available.
  - confidence: 0.9
