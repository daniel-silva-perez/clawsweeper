## Summary

Keep this PR open. Current main still does not implement `skills.allowExternalSkillsIn`: escaped skill realpaths are still rejected, the public config type and strict schema omit the field, generated config schema/help omit the field, docs still describe strict root containment, and tests lock in default rejection of symlink escapes. The PR remains a plausible focused implementation candidate, but it changes the core skills/config trust boundary and needs maintainer product/security review rather than automated closure.

## Best Possible Solution

Keep this PR open for maintainer review. If OpenClaw wants trusted external symlinked skill roots, land a focused core skills/config change with safe realpath containment, generated config/docs artifacts, and regression tests for default rejection, allowed external containment, prefix-boundary bypasses, canonical prefix resolution, empty entries, and target-workspace sync behavior. If maintainers reject the trust expansion, close it only after explicit product/security judgment and point overlapping reports to the chosen symlink policy.

## Risks / Open Questions

- The PR expands a core skills loading trust boundary: configured external realpath prefixes can make files outside the workspace, managed, bundled, or extra skill root load as skills.
- Loaded skills are prompt/runtime-visible and can be copied into target workspaces by `syncSkillsToWorkspace`, so the feature affects data access and command/tool execution posture.
- Because this introduces a public config key, `types.skills.ts`, `zod-schema.ts`, generated schema/baseline artifacts, docs, and regression tests need to remain aligned before merge.
