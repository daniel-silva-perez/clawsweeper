## Summary

Keep this issue open. Current main has partial mitigations for large session stores, but the reported core behavior remains: gateway paths still build one combined store by reading every discovered agent `sessions.json`, then apply `agentId` and other filters after aggregation. The related stabilization PR was closed unmerged, and the structuredClone/cache memory concern remains visible in current source and related open context.

## Best Possible Solution

Keep the issue open and fix it in the session-store/gateway boundary: add a targeted per-agent or indexed lookup path for hook/run/session resolution, make `sessions.list` honor `agentId`/limit without loading unrelated stores, and revisit session-store cache cloning/serialized retention with regression coverage for large multi-agent stores. Existing cleanup and maintenance behavior should stay as defense in depth, not as the sole fix.

## Risks / Open Questions

- Closing now would leave current gateway paths that still load and merge all discovered agent session stores before filtering, which is the main OOM pattern reported here.
