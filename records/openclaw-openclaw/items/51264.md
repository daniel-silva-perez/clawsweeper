## Summary

Keep open. Current main still has the reported eager aggregation path: gateway callers build a combined session store by loading every discovered agent sessions.json, then apply agentId/sessionId/label/limit filters afterward. Recent maintenance work reduces some store growth and clone pressure, but it does not remove the all-store load behavior at the center of this OOM report.

## Best Possible Solution

Keep this issue open and fix the gateway/session-store boundary: add a targeted per-agent or indexed lookup path for hook/run/session resolution, make sessions.list honor agentId and limit before loading unrelated stores, use streaming or pagination for true all-agent listings, and separately land a safe cache-clone reduction for the structuredClone memory path. Add regression coverage with multiple large agent stores proving agent-scoped listing and resolution do not read unrelated stores.
