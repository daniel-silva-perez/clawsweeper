## Summary

Keep open. Current main has partial accounting improvements, but the reported failure modes are still source-backed: cache-read tokens can still feed prompt/context snapshots, the compaction safeguard can still write an empty structured fallback after strict real-message filtering, and preflight compaction still runs before the memory-flush gate.

## What This Changes

The issue requests compaction fixes so anomalous cached-token accounting cannot trigger destructive compaction, large non-empty sessions do not get empty fallback summaries, and memory flush runs before compaction commits.

## Best Possible Solution

Implement a narrow agents/compaction repair: use an actual prompt/context snapshot or provider-normalized prompt footprint for compaction pressure, reject or fall through from empty fallback summaries for large non-empty preparations, and require memory flush to complete or be recorded before destructive preflight/context compaction commits. Preserve legitimate prompt-cache context behavior covered by #66520 while guarding against anomalous aggregated cache telemetry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path exists: feed inflated cacheRead into the current prompt-token derivation, run a high-token compaction preparation where all entries fail isRealConversationMessage, and observe that runReplyAgent schedules preflight compaction before memory flush.

## Solution Assessment

No. Current main is not the best solution because it separates provider usage.total from diagnostics but still lets cacheRead affect context snapshots and still allows empty fallback compaction before memory flush; the safer fix is provider-aware context pressure plus non-destructive fallback and ordering guards.
