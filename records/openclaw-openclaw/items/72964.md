## Summary

Keep open. Current main still has source-backed compaction reliability gaps: prompt/context accounting can inherit provider cache-read spikes, the safeguard can write an empty structured fallback after strict real-message filtering, and the main reply path still runs preflight compaction before the memory-flush gate. The related open PR #66716 is adjacent cache-accounting work for #66520, not a merged fix for this report.

## What This Changes

Reports that compaction can overcount Anthropic prompt-cache reads, emit an empty fallback summary for large non-empty preparations, and run preflight compaction before the pre-compaction memory flush.

## Best Possible Solution

Implement a focused agents/compaction repair that uses a provider-normalized prompt footprint or explicit prompt snapshot for compaction pressure, falls through or refuses destructive empty fallback summaries for large non-empty preparations, and ensures the memory-flush cycle is recorded before destructive preflight/context compaction commits.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path exists: feed inflated cacheRead into prompt-token derivation, build a high-token compaction preparation where all entries fail isRealConversationMessage, and observe that runReplyAgent reaches preflight compaction before memory flush.

## Solution Assessment

No. Current main is not the best solution because it documents provider totals as non-authoritative for live context but still lets cacheRead influence fallback prompt snapshots and still allows an empty fallback compaction before the memory-flush gate.
