## Summary

Keep open. Current main has partial accounting improvements, but the reported failure modes are still present: cache-read tokens can still feed compaction pressure, the compaction safeguard can still commit an empty structured fallback for a high-token preparation with no entries passing the real-message filter, and preflight compaction still runs before the memory-flush gate.

## What This Changes

The issue requests safer compaction triggering and summarization so cached-token usage spikes cannot prematurely compact real conversations, empty fallback summaries are not committed for high-token sessions, and memory flush runs before destructive compaction.

## Best Possible Solution

Implement a narrow agents/compaction fix: derive compaction pressure from the actual prompt/context envelope or a sanitized provider prompt snapshot that cannot be inflated by billing-only cache spikes, refuse or fall through from the empty fallback summary path for large token preparations, and ensure memory flush completes or is recorded before any destructive preflight/context-engine compaction commits. Update the token-threshold documentation and focused tests for Anthropic cache spikes, fallback-summary gating, and flush-before-compact ordering.

## Reproduction Assessment

Yes. Without the private production trace, current-main source checks still give a high-confidence reproduction path: feed an Anthropic-style usage snapshot with inflated cacheRead into derivePromptTokens, run a compaction preparation whose messages fail isRealConversationMessage with high tokensBefore, and observe that runReplyAgent schedules preflight compaction before memory flush.

## Solution Assessment

No. The current implementation is not the best way to solve the issue because it still treats cacheRead as active compaction pressure and still allows an empty fallback summary before memory flush; the safer path is to separate billing/cache telemetry from active context pressure and make fallback compaction non-destructive under high-token or pending-flush conditions.
