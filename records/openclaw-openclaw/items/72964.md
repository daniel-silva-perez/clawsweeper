## Summary

Keep open. Current main has partial diagnostics/accounting improvements, but the reported destructive compaction path is still source-backed: cache-inclusive usage can still become prompt/context pressure, the empty-preparation fallback still commits a structured boundary after strict filtering, and the reply runner still runs preflight compaction before the memory-flush gate. The related #66716 PR remains open and addresses the opposite #66520 under-compaction case, so it is context to preserve rather than a superseding fix for this issue.

## What This Changes

The issue requests a reliability fix for cache-read-driven compaction pressure, empty fallback compaction summaries on large non-empty conversations, and pre-compaction memory-flush ordering.

## Best Possible Solution

Implement a focused agents/compaction repair that uses a provider-normalized prompt footprint or explicit prompt snapshot for compaction pressure, treats anomalous aggregated cache telemetry defensively, falls through or cancels instead of committing an empty fallback for large non-empty preparations, and ensures memory flush completes or is recorded before destructive compaction. Keep the genuinely empty heartbeat/cron boundary behavior and legitimate prompt-cache compaction behavior intact.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path exists: feed cacheRead-heavy usage into `derivePromptTokens` or the context-engine after-turn runtime, invoke the compaction safeguard with high `tokensBefore` and messages that fail `isRealConversationMessage`, and inspect `runReplyAgent` ordering where preflight compaction runs before memory flush. I did not run tests because this review is explicitly read-only and Vitest writes cache artifacts.

## Solution Assessment

No. Current main is not the best solution because it improves diagnostics and prompt snapshots but still allows cache-inclusive snapshots and unconditional empty fallback boundaries, and it still orders preflight compaction before memory flush. The safer direction is a narrow provider-aware context-pressure fix plus fallback and flush-order guards.
