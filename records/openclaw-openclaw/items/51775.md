## Summary

Keep this PR open. Current main cancels active tracked chat/agent runs through abort signals, but it still does not implement safe queue and subagent cleanup from `chat.abort`; the PRâ€™s direct cleanup calls are too broad for the existing run and owner boundaries.

## What This Changes

The PR adds embedded Pi abort, session queue clearing, and subagent stop calls to both `chat.abort` paths and adds tests for those cleanup calls.

## Best Possible Solution

Keep the current abort-signal path for active runs, then add queue and subagent cleanup only behind a contract that can prove the abort is authorized for all affected session work. For `runId`, either make cleanup genuinely run-scoped or limit broader cleanup to the last active run in that session, with tests for concurrent runs and mixed-owner devices.

## Reproduction Assessment

Yes. A unit-level reproduction can be built by creating two active runs or queued follow-ups under the same session and aborting only one run or one owner; the PRâ€™s added session-keyed cleanup would clear shared queues/subagents outside the authorized target.

## Solution Assessment

No. Mirroring `/stop` directly is not the best fix because `/stop` is session-targeted, while `chat.abort(runId)` and owner-filtered session aborts have narrower authorization and scoping semantics.
