## Summary

Keep this PR open. Current main partially addresses Stop-button cancellation through abort-signal plumbing for active chat and agent runs, but it still does not implement the PR’s stated queue and sub-agent cleanup behavior in `chat.abort`. The PR’s direct cleanup approach also still has unresolved run-scoping and owner-isolation risks that the author acknowledged in discussion.

## Best Possible Solution

Revise or replace this PR with a narrower design: keep the current abort-signal cancellation for active chat/agent runs, add queue and sub-agent cleanup only when the abort is truly session-wide and authorized for all affected work, and introduce run-scoped ownership for queue/sub-agent cleanup before doing it from `chat.abort(runId)`. Maintainers should answer the author’s scoping questions before this PR is merged or closed.

## Risks / Open Questions

- Merging the PR as written can turn `chat.abort(runId)` from targeted run cancellation into session-wide cleanup because the affected queues and embedded run state are keyed by session, not run.
- The sessionKey path can undermine existing owner isolation when multiple clients share a session key: one authorized run abort could still clear queues or stop sub-agents for unrelated active work.
- Current main still lacks regression coverage for safe queue and sub-agent cleanup that preserves both runId targeting and multi-owner authorization boundaries.
