## Summary

Keep this PR open. Current main has abort-signal cancellation for active chat/agent runs, but it still does not implement the PR's stated session queue and sub-agent cleanup behavior for `chat.abort`, and the PR's direct cleanup approach still carries unresolved run-scoping and owner-isolation risks acknowledged in review discussion.

## Best Possible Solution

Revise or replace this PR with a narrower implementation: keep current abort-signal cancellation for active chat/agent runs, add queue and sub-agent cleanup only when the abort is truly session-wide and authorized for all affected work, and introduce run-scoped queue/sub-agent ownership before doing cleanup from `chat.abort(runId)`. Maintainers should review the open scoping questions before merging or closing the PR.

## Risks / Open Questions

- The PR can turn `chat.abort(runId)` from targeted run cancellation into session-wide cleanup because queues and embedded session aborts are keyed by session, not run.
- The sessionKey path can undermine current owner isolation when multiple clients share a session key: one authorized run abort could still clear queues or stop sub-agents for unrelated active work.
- Current main still lacks regression coverage for safe queue/sub-agent cleanup that preserves both runId targeting and multi-owner authorization boundaries.
