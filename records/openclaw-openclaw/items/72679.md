## Summary

Keep open. Current main still has the Nextcloud Talk reaction dispatch gap, but this PR is not merge-ready because it mishandles shared reaction-removal semantics and bypasses the shared message-id resolver; there is also an older overlapping open PR for the same bug.

## What This Changes

The PR adds a Nextcloud Talk `react` channel message action adapter, registers it on the plugin, and adds unit tests for dispatching reactions through the existing sender.

## Best Possible Solution

Land one consolidated Nextcloud Talk actions adapter that registers `react`, resolves room/message context through the shared SDK helpers, implements or explicitly rejects reaction removal, includes request-shape coverage for the sender, and closes #70110 while retiring the duplicate PR path.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: on current main, `sendReactionNextcloudTalk` exists, but Nextcloud Talk has no `actions.handleAction`, so `dispatchChannelMessageAction` returns `null` before any reaction request is sent.

## Solution Assessment

No. Registering a channel message action adapter is the right seam, but this implementation should use `resolveReactionMessageId` and handle `remove: true` safely before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Honor reaction removal requests:** `extensions/nextcloud-talk/src/channel.adapters.ts:84`
  - body: The shared `message react` contract accepts `remove: true`, but this handler ignores it and always calls the POST/add sender. A removal request would add or refresh the emoji instead; implement the DELETE path or reject removal explicitly.
  - confidence: 0.88
- **[P2] Use the shared message-id resolver:** `extensions/nextcloud-talk/src/channel.adapters.ts:76`
  - body: This only accepts string `params.messageId` and string `toolContext.currentMessageId`, so the `message_id` alias and numeric current-message fallback are ignored. Use `resolveReactionMessageId` and stringify the resolved value before sending.
  - confidence: 0.84
