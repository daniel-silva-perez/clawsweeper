## Summary

Keep this PR open for maintainer review. Current main still has the reported Nextcloud Talk gap, because the reaction HTTP sender exists but the channel plugin does not register a message action adapter; however, this patch has blocking correctness issues around shared `react` semantics and overlaps with the older open implementation PR #72348, so the right path is to consolidate rather than close as implemented.

## What This Changes

The PR adds a Nextcloud Talk `react` message action adapter, registers it on the channel plugin, and adds unit tests for dispatching reactions through the existing sender.

## Best Possible Solution

Land one consolidated Nextcloud Talk actions adapter that registers `react`, resolves current room/message context with the shared SDK helpers, safely handles or rejects reaction removal, includes sender/request-shape coverage, and then closes #70110 while retiring the duplicate PR path.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: on current main, `sendReactionNextcloudTalk` exists, but `nextcloudTalkPlugin` has no `actions.handleAction`, so the shared dispatcher returns null for Nextcloud Talk `react` actions before any HTTP request can be made.

## Solution Assessment

No. Registering a channel message action adapter is the correct seam, but this implementation should not map `remove: true` to an add reaction and should use the shared message-id resolver before it is merge-ready.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Honor reaction removal requests:** `extensions/nextcloud-talk/src/channel.adapters.ts:84`
  - body: The shared `message` tool exposes `remove` for `react`, but this handler ignores it and always calls `sendReactionNextcloudTalk`, which sends the POST/add path. A request to remove an emoji would add or refresh the reaction instead; implement the DELETE sender or reject removal requests explicitly.
  - confidence: 0.88
- **[P2] Use the shared message-id resolver:** `extensions/nextcloud-talk/src/channel.adapters.ts:76`
  - body: This only accepts string `params.messageId` and string `toolContext.currentMessageId`, so the documented `message_id` alias and numeric current-message fallback are ignored. Use `resolveReactionMessageId` and stringify the result before sending.
  - confidence: 0.84
