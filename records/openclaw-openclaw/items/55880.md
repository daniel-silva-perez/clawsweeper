## Summary

Keep this issue open. Current main still has only partial coverage for the reported macOS auto-upgrade failure: installed-but-unloaded LaunchAgents can be re-bootstrapped, but the service-not-installed path is still not automatically repaired during update/restart, non-interactive update doctor mode deliberately skips runtime service install/restart repairs, and cron failure notifications still have fire-and-forget delivery paths that are not reflected back into the persisted run delivery status. The fresh 2026-04-28 report against v2026.4.26 remains valid because the only relevant post-review change on main is a daemon service environment hardening change, not a fix for update recovery or notification accounting.

## Best Possible Solution

Keep this issue open and fix it as a targeted macOS self-update recovery bug: make package-update recovery share a safe doctor/gateway repair path for installed-but-unloaded and missing LaunchAgent states, wait for gateway readiness long enough to avoid the startup race, and record failure-notification delivery attempts/results separately from the main cron run delivery state. Also reproduce whether missing optional provider or memory credentials can still block openclaw update --yes --json, and make that preflight warning-only unless the update path actually depends on those credentials.

## Risks / Open Questions

- Automatically reinstalling or bootstrapping a missing LaunchAgent during update must preserve external-supervisor and user-managed gateway setups; service repair policy needs to remain explicit and conservative.
- Making failure notifications authoritative may require bounded waits, retries, or separate accounting so cron finalization does not hang on Discord/channel delivery.
- The provider credential preflight question still needs a narrow reproduction before changing update semantics, because current doctor memory-search checks appear warning-oriented in source but the issue reports an update-blocking failure.
