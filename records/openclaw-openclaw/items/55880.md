## Summary

Keep open. Current main still only partially addresses the reported macOS update/recovery flow: installed-but-unloaded LaunchAgents can be re-bootstrapped, but missing service installs are not restored during package-update restart, non-interactive update doctor mode skips runtime service repairs, and cron failure notifications are dispatched fire-and-forget rather than reflected in the persisted delivery status. The fresh v2026.4.26 report keeps this current.

## Best Possible Solution

Keep this open and fix it as a targeted macOS self-update recovery issue: share a conservative doctor/gateway repair path for installed-but-unloaded and missing LaunchAgent states during package-update recovery, wait for gateway readiness robustly, and record failure-notification delivery attempts separately from the main cron run delivery result. Also reproduce whether missing optional provider or memory credentials can still block `openclaw update --yes --json` and make that warning-only unless required by the update path.
