## Summary

Keep this PR open. Current main still responds to device pair removal, token rotation, and token revocation before queueMicrotask-scheduled device disconnects, has no GatewayWsClient invalidation state, and only performs a per-RPC stale-session recheck for shared gateway auth, not device-token clients. The proposed PR remains a plausible targeted security hardening patch, but it should be rebased and reviewed as auth/token-sensitive gateway work.

## Best Possible Solution

Keep this PR open, rebase it onto current main, and review it as a targeted gateway security hardening change. The desired endpoint is a small patch that synchronously marks affected device-token clients invalid before the admin response is emitted, rejects later RPCs at the top of WS per-request dispatch, preserves the deferred hard close so the admin response can flush, and keeps focused tests for remove, rotate, revoke, request-context filtering, and dispatch rejection ordering.
