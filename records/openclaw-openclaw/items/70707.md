## Summary

Keep this PR open for maintainer review. Current main still has the described queued-RPC window for device-token clients, and the PR is a narrow gateway hardening patch that addresses it without introducing a definite code-review blocker; remaining work is maintainer judgment plus CI triage.

## What This Changes

The PR adds a Gateway WebSocket client invalidation flag, a request-context method to mark device clients invalid, pre-response invalidation for device removal/token rotation/token revocation, a pre-dispatch WebSocket guard, and focused handler/context tests.

## Best Possible Solution

Review and land this PR, or an equivalent narrow mainline patch, so affected device-token clients are marked invalid before successful invalidation responses are emitted and any later queued RPC on that socket is rejected before handler logic runs.

## Reproduction Assessment

Yes. Current main gives a high-confidence code reproduction: the affected handlers send success before queueMicrotask disconnect, and request dispatch only rechecks shared gateway auth, not device-token invalidation.

## Solution Assessment

Yes. The PR's in-memory invalidated flag plus pre-dispatch guard is the narrowest maintainable fix I found because it preserves admin response flushing while closing the queued-RPC window; a direct WS dispatch regression test would strengthen coverage but is not a required design change.
