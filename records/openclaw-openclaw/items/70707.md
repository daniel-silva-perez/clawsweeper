## Summary

Keep this PR open. Current main still has the response-before-deferred-disconnect window for device-token clients, while this PR is a narrow gateway auth hardening patch with no blocking code-review findings; it needs maintainer review and CI triage rather than cleanup closure.

## What This Changes

The PR adds synchronous device-client invalidation before device removal, token rotation, and token revocation success responses, adds a WebSocket invalidated-client dispatch guard, and extends gateway handler/request-context tests.

## Best Possible Solution

Review and land this PR, or an equivalent narrow mainline patch, so affected device-token clients are marked invalid before successful invalidation responses are emitted and any later queued RPC on that socket is rejected before handler logic runs.

## Reproduction Assessment

Yes. Current main source gives a high-confidence code reproduction: the three affected handlers respond before queueMicrotask disconnect, and the dispatcher only rechecks shared gateway auth before handler dispatch; no live pipelined-socket harness was run in this read-only review.

## Solution Assessment

Yes. The in-memory client flag plus pre-dispatch guard is a narrow maintainable fix that preserves admin response flushing while closing the queued-RPC window; a direct WS dispatch test would improve coverage but I did not find a smaller safer fix.
