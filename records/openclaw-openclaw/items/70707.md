## Summary

Keep this PR open. Current main still responds before deferred device-client disconnect and has no device-token per-RPC invalidation guard, while the PR is a narrow gateway auth hardening patch with no blocking code-review findings; it needs maintainer review, CI triage, and merge judgment rather than cleanup closure.

## What This Changes

The PR adds synchronous device-client invalidation before device removal, token rotation, and token revoke responses, adds an invalidated-client WebSocket dispatch gate, and extends gateway handler/request-context tests.

## Best Possible Solution

Review and land this PR, or an equivalent narrow mainline patch, so affected device-token clients are marked invalid before successful invalidation responses are emitted and any later queued RPC on that socket is rejected before business logic runs.

## Reproduction Assessment

Yes. The code-level reproduction path is high-confidence: current main responds before queueMicrotask disconnect in the three affected handlers, and the current WS dispatcher only rechecks shared gateway auth before handler dispatch. A live pipelined-socket harness was not run because this review was read-only.

## Solution Assessment

Yes, with a coverage caveat. The in-memory client flag plus pre-dispatch guard is the narrowest maintainable fix that preserves the successful admin response flush while closing the queued-RPC window; adding one direct WS dispatch test would strengthen the proof before merge.
