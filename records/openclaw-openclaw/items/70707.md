## Summary

Keep this PR open for maintainer review. Current main still has the described queued-RPC window for device-token clients, and the PR is a narrow gateway hardening patch with no definite code-review blocker found.

## What This Changes

The PR adds Gateway WebSocket client invalidation state, a request-context invalidation method, pre-response invalidation for device removal/token rotation/token revocation, a pre-dispatch invalidated-client guard, and focused gateway tests.

## Best Possible Solution

Land this PR, or an equivalent narrow mainline patch, so affected device-token clients are marked invalid before successful invalidation responses are emitted and any later queued RPC on that socket is rejected before handler logic runs.

## Reproduction Assessment

Yes. Current main gives a high-confidence code reproduction: the affected handlers send success before queueMicrotask disconnect, and request dispatch only rechecks shared gateway auth, not device-token invalidation.

## Solution Assessment

Yes. The in-memory invalidated flag plus pre-dispatch guard is a narrow maintainable fix because it preserves admin response flushing while closing the queued-RPC window; a direct WS dispatch regression test would strengthen coverage but is not a required design change.
