## Summary

Keep this PR open. Current main still scans fenced-code content in `extractWikiLinks`, so the reported false-positive lint path is real and the PR is still relevant. The latest PR head fixes the reported Python example and addresses earlier strip-order feedback, but the new fence regex can treat a shorter internal triple-backtick line as the end of a longer four-backtick fence, so code links can still leak into link extraction.

## What This Changes

The PR updates memory-wiki link extraction to strip fenced code blocks, HTML comments, and image links before scanning wiki links, and adds focused `extractWikiLinks` unit coverage.

## Best Possible Solution

Revise the extractor so fenced-code stripping respects the opening fence length, preferably with a small line scanner or a regex that cannot close a four-backtick block on a three-backtick line. Keep the focused extractor tests and add a regression for a longer opening fence containing a shorter internal fence before the real close.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Respect the opening fence length:** `extensions/memory-wiki/src/markdown.ts:74`
  - body: The new pattern treats any line with at least three backticks as a closing fence. In a valid four-backtick block, an internal triple-backtick line is code content, but this replacement stops there and lets later code lines get scanned as prose links. Track the opener length or use a small scanner, and add a regression for a shorter internal fence before the real close.
  - confidence: 0.91
