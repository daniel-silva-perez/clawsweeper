## Summary

Keep this PR open. Current `main` still has the reported memory-wiki false-positive path, and the PR remains a focused, relevant fix, but the latest patch is not merge-ready because its fenced-code regex can close a longer backtick fence on a shorter triple-backtick line and still extract fake links from valid fenced code content.

## What This Changes

This PR strips fenced code blocks, HTML comments, and image markdown before memory-wiki link extraction, and adds focused `extractWikiLinks` coverage for code, comment, image, local-link, external-link, and anchor cases.

## Best Possible Solution

Revise the PR so the prefilter tracks each backtick fence opener and only closes it with a same-type fence of at least the opener length, while preserving the existing outside-code, HTML-comment, image, external-link, and anchor behavior. Add a regression where a four-backtick fence contains a shorter triple-backtick marker before markdown-like code text, then run the targeted memory-wiki markdown test.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Honor opener length when closing fences:** `extensions/memory-wiki/src/markdown.ts:74`
  - body: The new regex treats any line with 3+ backticks as the closing fence, even when the opener used more backticks. In a valid four-backtick block that contains a literal ``` line before `[fake](bad)`, the replacement ends early and `extractWikiLinks` still extracts `bad`, so `wiki lint` can still report false broken-link targets from fenced code. Track the opener length or use a small fence scanner before extracting links.
  - confidence: 0.9
