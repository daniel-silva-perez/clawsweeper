## Summary

Keep open. Current main and the latest tagged release still render the shared status footer with only the session key plus last-updated/no-activity text, while the session start timestamp and lifecycle fallback already exist and are documented. This remains a narrow status-rendering feature, not an implemented or stale cleanup item.

## What This Changes

Add elapsed session duration or a started timestamp to the shared status footer's Session line.

## Best Possible Solution

Add the session-start visibility in the shared status renderer so `/status` and `session_status` get identical output, using `sessionStartedAt` first and the lifecycle fallback when needed, with focused tests for the footer behavior.

## Reproduction Assessment

Yes for the missing behavior. A source-level check against current main shows `buildStatusMessage()` ignores `sessionStartedAt` and renders only the session key plus last update/no-activity text even though the timestamp source is available.

## Solution Assessment

Yes, the narrowest maintainable fix is to update the shared status rendering path rather than duplicating formatting in `/status` or `session_status`. The only open choice is whether maintainers want `started <timestamp>` or `duration <elapsed>` as the displayed segment.
