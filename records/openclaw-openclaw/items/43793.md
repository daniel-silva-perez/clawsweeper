## Summary

Keep open. The requested TUI disconnect guidance is still a valid gap on current main, but this PR is not merge-ready: it calls a reconnect method outside the generic TUI backend contract and reintroduces `StrictHostKeyChecking=accept-new` in a macOS SSH probe that current main already hardened to strict host-key checking.

## What This Changes

The PR adds TUI gateway-disconnect guidance and Ctrl+R manual reconnect handling, exposes GatewayClient/GatewayChatClient reconnect controls, adds macOS remote-gateway probe/auth UI handling, and adjusts a diffs cleanup test and gateway callsite guard.

## Best Possible Solution

Rebase and narrow the PR so manual reconnect is represented as a typed optional `TuiBackend` capability or a gateway-only path, keep all macOS SSH probes on the shared strict host-key options, update TUI shortcut docs/tests, and avoid carrying unrelated cleanup unless it is still required after rebase.

## Reproduction Assessment

Yes. The backend bug is reproducible by static inspection: the PR diff calls `client.reconnect()` in `runTui`, while current main's `TuiBackend` and `EmbeddedTuiBackend` do not expose that method. The security regression is also directly visible in the PR diff through `StrictHostKeyChecking=accept-new`.

## Solution Assessment

No. The UX direction is reasonable, but this is not the best mergeable solution yet; reconnect should go through a typed backend capability or gateway-specific guard, and the macOS diagnostic path should reuse current main's strict SSH option fragments.
