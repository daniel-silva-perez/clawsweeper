## Summary

Keep open. Current main does not implement the requested TUI reconnect UX, but this PR still has blocking merge issues: it calls a gateway-only reconnect method through the generic TUI backend path and weakens macOS SSH host-key verification in the remote probe.

## What This Changes

The PR adds TUI gateway-disconnect hints and Ctrl+R reconnect handling, gateway reconnect timer cleanup, macOS remote-gateway probe/auth UI handling, and a diffs cleanup-test adjustment.

## Best Possible Solution

Land a rebased and narrowed version that adds a typed optional reconnect capability through the TUI backend, keeps local/embedded/Crestodian modes safe, shows reconnect guidance only for reconnectable gateway disconnects, and preserves strict macOS SSH host-key checks.

## Reproduction Assessment

Yes. The backend issue is reproducible by source inspection: current runTui stores the client as TuiBackend, and the PR calls client.reconnect() without adding that method to the interface or non-gateway implementations. The SSH issue is reproducible from the PR diff line that adds StrictHostKeyChecking=accept-new.

## Solution Assessment

No. The requested UX is reasonable, but the current patch is not the safest implementation; reconnect should be an optional backend capability or gateway-only path, and SSH diagnostics should reuse the shared strict host-key options.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Route reconnect through the TUI backend:** `src/tui/tui.ts:947`
  - body: `runTui` is now a generic `TuiBackend` consumer, and current main has embedded/local and Crestodian backends that do not expose `reconnect`. Calling `client.reconnect()` directly here makes the branch stale against the backend abstraction and can fail typechecking or break non-gateway TUI modes after rebase. Add an optional backend capability or a gateway-mode guard.
  - confidence: 0.9
- **[P1] Keep SSH probes on strict host-key checks:** `apps/macos/Sources/OpenClaw/RemoteGatewayProbe.swift:190`
  - body: The diagnostic SSH probe should not auto-trust first-seen host keys. Current main centralizes SSH calls on `CommandResolver.strictHostKeyCheckingSSHOptions` with `StrictHostKeyChecking=yes`; this added `accept-new` option weakens the remote-host verification model. Reuse the shared strict options here.
  - confidence: 0.88
