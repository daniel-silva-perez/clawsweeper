## Summary

Keep this PR open, but do not merge it as-is. The patch addresses the right destination-field problem for one direct gateway cron call, but normal main-session cron execution can still move through the queued heartbeat wake path, where `{ target: "last" }` is merged back over the full agent heartbeat config and can retain inherited `to`/`accountId`.

## What This Changes

The PR strips inherited heartbeat `to` and `accountId` fields from cron `runHeartbeatOnce` overrides when `target` is forced to `last`, and adds a gateway cron regression test plus a changelog entry.

## Best Possible Solution

Apply the destination-field omission at a boundary shared by direct and queued cron-forced `target: "last"` wakes, or teach the heartbeat wake runner's targeted merge to treat that cron override as non-inheriting for explicit destinations. Add regression coverage that runs the actual main-session cron job path rather than only invoking the adapter dependency directly, while preserving normal explicit heartbeat destination semantics outside cron-forced wakes.

## Reproduction Assessment

Yes. The high-confidence reproduction path is source-backed: a main-session cron job marks itself active, direct heartbeat execution can skip as `cron-in-progress`, the queued wake then merges `{ target: "last" }` with configured heartbeat defaults, and delivery resolution still honors inherited `to`/`accountId`.

## Solution Assessment

No. The helper is directionally correct for direct `runHeartbeatOnce`, but the narrow maintainable fix must cover queued `requestHeartbeatNow` cron wakes as well, otherwise the linked bug remains possible on the normal fallback path.

## Review Findings

- **[P1] Sanitize queued cron heartbeat wakes too:** `src/gateway/server-cron.ts:294-296`

The linked route leak can still happen for queued/fallback wakes, so apply the same omission at the shared wake boundary and cover the actual cron execution path.
