## Summary

Keep open. Current main appears to have changed the subagent model path so explicit subagent model selections are persisted without the old `sessions.patch` allowlist failure, but the `models.json` generation half still has a real remaining gap: explicit custom providers with non-empty `models` and no resolved `apiKey` can still flow into the generated registry payload.

## What This Changes

The issue asks OpenClaw to avoid writing unresolved custom providers into generated `models.json` and to align or clarify session/subagent admission for discovered Codex models.

## Best Possible Solution

Add a generator-level guard that omits or neutralizes only providers PI would reject because they have custom models but no resolved credential marker, while preserving valid keyless/auth-mode providers. Keep `models list --all` as catalog visibility, and only adjust session admission or error wording if a current `sessions_spawn` repro still shows a mismatch.

## Reproduction Assessment

Yes for the generator path: a provider with `models.providers.<id>.models` and no resolvable auth still flows through normalization unchanged and is serialized. Unclear for the session-admission half on current main because the subagent path now persists explicit model selections directly and has coverage for `openai-codex/gpt-5.4`.

## Solution Assessment

Unclear as a complete two-part solution. Dropping unresolved providers is the right direction, but a blanket strip of every missing-`apiKey` provider is not the safest fix; the implementation should encode PI's credential requirement while preserving documented keyless or marker-backed OpenClaw provider modes.
