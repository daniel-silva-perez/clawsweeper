## Summary

Keep open. Current main appears to have partially mitigated the Codex discovery failure through the newer PI registry behavior plus OpenClaw's provider catalog augmentation, and subagent model overrides are now persisted before the first child turn. The requested generator behavior is still not fully implemented, though: unresolved custom-provider model rows can still be written unchanged, and session admission can still reject a model that is visible in `models list --all` when it is outside the configured allowlist.

## What This Changes

The issue asks for generated `models.json` to exclude or neutralize unresolved custom-provider model rows, and for session/subagent model admission to align with or clearly explain divergence from live model discovery.

## Best Possible Solution

Add a narrow normalization or validation path for generated `models.json` that neutralizes only genuinely unresolved custom-provider entries without breaking built-in provider auth profiles, keyless `aws-sdk` providers, or provider-owned synthetic auth. Separately, make session rejection explain when `agents.defaults.models` or subagent allowlist policy is the reason a discoverable `--all` model is not admissible.

## Reproduction Assessment

Unclear. The source path still supports the unresolved-provider row described in the report, and the allowlist rejection path is present, but this review did not execute the original live macOS reproduction; current main also changes the Codex poisoning symptom through newer registry/catalog behavior.

## Solution Assessment

Unclear. Current main is a partial solution because it mitigates Codex discovery and persists subagent model overrides, but it does not fully solve the generator contract or provide a clearer allowlist-specific rejection for discoverable models.
