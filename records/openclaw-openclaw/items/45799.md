## Summary

Keep open. Current main still retries Telegram `getFile()` only; the subsequent Bot API file download still makes one `fetchRemoteMedia()` call and the existing unit test explicitly preserves that no-retry behavior. The related proxy/SSRF propagation work has shipped separately, but it does not cover transient proxy outages during the media download step.

## What This Changes

The issue requests bounded retry/backoff for Telegram inbound media file downloads when an operator-configured HTTP proxy is temporarily unavailable.

## Best Possible Solution

Add a narrow Telegram plugin retry around the remote media download call, preserving the existing transport, dispatcherAttempts, SSRF policy, and explicit-proxy DNS handling while replacing the no-retry regression test with retry and non-retry coverage.

## Reproduction Assessment

Yes. The current unit test `does not catch errors from fetchRemoteMedia (only getFile is retried)` is a high-confidence reproduction of the code path: mocked `getFile()` succeeds, the first download fails, and `resolveMedia()` rejects without retrying.

## Solution Assessment

Yes. Bounded `retryAsync()` around only the `fetchRemoteMedia()` download step is the narrowest maintainable fix; automatic direct fallback is not the best solution because explicit proxy configuration is an operator policy boundary.
