## Summary

Keep this issue open. Current main has shipped related Telegram media proxy/SSRF fixes, but the specific requested behavior, retry/backoff when an operator-configured Telegram HTTP proxy drops during the actual media file download, is still not implemented for explicit proxy transports. `getFile()` is retried; the subsequent Bot API file fetch is not wrapped in its own retry loop, and explicit proxy paths intentionally get no fallback attempts.

## Best Possible Solution

Keep the issue open and implement a narrow Telegram media download retry/backoff path around the actual `fetchRemoteMedia()` stage. Add focused tests for transient proxy/network failure succeeding after retry, exhausted transient retries, and no retry for permanent HTTP, size, or policy failures. Preserve the existing explicit-proxy SSRF behavior and avoid automatic direct fallback unless maintainers make that product/security decision.

## Risks / Open Questions

- Closing as implemented would conflate shipped proxy/SSRF propagation fixes with the still-missing retry/backoff behavior for transient explicit-proxy media download failures.
- The eventual fix should retry only transient download failures and should not retry permanent `http_error`, `max_bytes`, SSRF policy, or Telegram file-size failures.
- Automatic direct fallback would bypass an operator-configured proxy policy; the safer first implementation is to retry the proxied download unless maintainers explicitly approve direct fallback.
