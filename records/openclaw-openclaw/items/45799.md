## Summary

Keep open. Current main still retries only Telegram `getFile()`; the Bot API file download path calls `fetchRemoteMedia()` once, and the current regression test explicitly asserts that download failures are not caught or retried. The related proxy/DNS propagation work shipped separately, and prior PR #46013 targeted the right fix but was closed unmerged.

## Best Possible Solution

Implement a narrow Telegram plugin fix: wrap only the remote `fetchRemoteMedia()` call inside `downloadAndSaveTelegramFile()` with bounded `retryAsync()` backoff, retry transient `MediaFetchError` code `fetch_failed`, keep permanent `http_error` and `max_bytes` failures non-retryable unless maintainers explicitly choose a status-code policy, preserve all current transport/proxy/SSRF arguments, and replace the current no-retry test with focused retry and non-retry coverage. Do not add direct fallback.
