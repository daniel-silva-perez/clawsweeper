## Summary

Keep open. Current main still retries the Telegram Bot API `getFile()` step but leaves the later Bot API file download as a single `fetchRemoteMedia()` call; the colocated retry test still asserts that download failures are not retried. The related proxy/SSRF fixes have shipped, but they address proxy propagation and DNS pinning rather than bounded backoff for transient download-stage failures, and PR #46013 was closed unmerged.

## What This Changes

The issue requests bounded retry/backoff for Telegram inbound media file downloads when the configured proxy or network path drops after `getFile()` succeeds.

## Best Possible Solution

Add a bounded Telegram plugin retry around only the remote Bot API media download stage, retrying transient `MediaFetchError` download failures while preserving the configured transport, dispatcher attempts, explicit-proxy DNS behavior, SSRF policy, local Bot API file-path handling, and permanent-failure behavior.

## Reproduction Assessment

Yes. The current unit test mocks a successful `getFile()` followed by a failed `fetchRemoteMedia()` and asserts that `resolveMedia()` rejects without retrying, directly reproducing the missing download-stage retry path described here.

## Solution Assessment

Yes. A small `retryAsync()` wrapper around the remote `fetchRemoteMedia()` download step is the narrowest maintainable fix; direct fallback is not the best solution because it can bypass a configured proxy and SSRF policy boundary.
