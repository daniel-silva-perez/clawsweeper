## Summary

Keep open. Current main still retries only the Telegram Bot API `getFile()` step; the subsequent Bot API file download is still a single outer `fetchRemoteMedia()` call, and the colocated retry test explicitly asserts that download failures are not caught or retried. The related Telegram proxy/SSRF DNS fixes have shipped, but they solve proxy propagation and DNS pinning, not bounded backoff for transient download-stage failures; the implementation PR that directly targeted this issue is closed unmerged.

## What This Changes

The issue requests bounded retry/backoff for Telegram inbound media file downloads when a configured proxy or network path drops after `getFile()` succeeds.

## Best Possible Solution

Add a bounded Telegram plugin retry around only the remote media download stage, retrying transient `MediaFetchError` download failures while preserving the configured Telegram transport, dispatcher attempts, explicit-proxy DNS behavior, SSRF policy, local Bot API file-path handling, and permanent-failure behavior.

## Reproduction Assessment

Yes. The current unit test mocks a successful `getFile()` followed by a failed `fetchRemoteMedia()` and asserts that `resolveMedia()` rejects without retrying, directly reproducing the missing download-stage retry path described here.

## Solution Assessment

Yes. A small `retryAsync()` wrapper around the `fetchRemoteMedia()` download step is the narrowest maintainable fix; automatic direct fallback is not the best solution because it could bypass a configured proxy and the SSRF policy boundary.
