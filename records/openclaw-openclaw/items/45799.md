## Summary

Current main still leaves the specific retry gap reported here open: Telegram `getFile()` is retried, but the Bot API file download path still calls `fetchRemoteMedia()` once, and the current retry test explicitly preserves that no-retry behavior. Related proxy/SSRF propagation fixes appear to have shipped separately for the broader proxy family, and the linked implementation PR #46013 was closed unmerged, so this issue should remain open for a narrow Telegram media download retry fix.

## Best Possible Solution

Implement a narrow Telegram plugin fix: wrap only the remote `fetchRemoteMedia()` call inside `downloadAndSaveTelegramFile()` with bounded `retryAsync()` backoff, retry transient `MediaFetchError` code `fetch_failed`, keep permanent `http_error` and `max_bytes` failures non-retryable unless maintainers explicitly choose a status-code policy, preserve all current transport/proxy/SSRF arguments, and replace the current no-retry test with focused retry and non-retry coverage. Do not add direct fallback by default.
