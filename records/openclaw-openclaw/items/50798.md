## Summary

Keep open. Current main still lacks a supported way for a coordinator to post the initiating instruction visibly into an ACP-bound Discord thread while bypassing main-session outbound metadata creation. Recent A2A work adds visible announce/reply delivery after `sessions_send`, but that does not expose `visible`/`proxyOnly` on public send contracts and does not solve the reported proxy-only instruction path.

## What This Changes

Request a visible coordinator-to-ACP thread messaging path that delivers into the bound Discord thread without creating `agent:main:discord:channel:<thread_id>` session stubs.

## Best Possible Solution

Add a narrow, explicit visible proxy-delivery contract after maintainer API review. It should post the coordinator-authored instruction through the requested Discord account into the ACP-bound thread, bypass outbound main-session entry creation only for that authorized call, preserve normal metadata writes elsewhere, and document the supported ACP coordination workflow.

## Risks / Open Questions

- The public API shape is still undecided: `sessions_send visible`, `message proxyOnly`, and an ACP-bound-thread exclusion guard have different compatibility and UX tradeoffs.
- A broad skip of outbound session metadata for Discord thread sends could regress normal transcript mirroring, session continuity, and reply attribution for ordinary message sends.
- Dual-bot Discord relay behavior needs explicit security review around `allowBots`, mention gating, account identity, and auditability so proxy delivery does not become an implicit routing bypass.
