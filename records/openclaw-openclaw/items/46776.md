## Summary

Keep open. Current main still excludes broadcast-only Feishu `@all` from bot mention detection, configured Feishu broadcast dispatch is observer-style with only the active agent replying visibly, and direct/group conversations still route to distinct session keys. The related open Feishu broadcast item covers only part of the multi-bot delivery problem, while DM/group continuity needs a maintainer product decision.

## What This Changes

The issue reports that Feishu `@all` messages in multi-agent groups do not reach every agent and that the same agent does not share conversation context between direct and group sessions.

## Best Possible Solution

Split the work into two explicit maintainer decisions: an opt-in Feishu `@all` or multi-mention fan-out mode where each targeted agent can receive and visibly reply once, and a documented DM/group continuity model that either adds a safe opt-in session-linking mechanism or directs users to memory/session tools instead of shared transcripts. Coordinate the multi-bot delivery half with #43563.

## Reproduction Assessment

Yes. There is a high-confidence source-level reproduction path: broadcast-only `@all` parses as `mentionedBot=false`, mention-gated group handling returns before dispatch, and direct/group routing builds distinct session keys; no live Feishu credentials were visible in `~/.profile` for an end-to-end run.

## Solution Assessment

Unclear. The current behavior is deliberate and documented, but it does not satisfy the requested workflow; the safest path needs maintainer product judgment because visible multi-agent fan-out and cross-surface transcript sharing have privacy, duplication, and loop-control implications.
