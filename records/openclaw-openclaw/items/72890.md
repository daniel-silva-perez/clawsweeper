## Summary

Keep open. Current main still lacks a config-backed adaptive-thinking capability for opaque custom `anthropic-messages` model IDs, so the reported custom proxy case remains valid and narrow enough for a focused repair PR.

## What This Changes

Request support for configured custom Anthropic-compatible provider models to declare adaptive-thinking support so non-Claude-named proxy model IDs receive adaptive Anthropic payloads.

## Best Possible Solution

Add a typed config-backed adaptive-thinking capability for configured `anthropic-messages` models, feed it into thinking-profile validation and Anthropic transport option/payload resolution, keep the Claude model-id fallback, and document a generic custom proxy example without customer-specific prefixes.

## Reproduction Assessment

Yes. A focused test can configure a custom `models.providers.my-proxy` entry with `api: "anthropic-messages"`, opaque id `zyb-high`, `reasoning: true`, and an adaptive-capable metadata field; current main has no accepted field and, if forced into a non-off reasoning run, takes the budget-token branch instead of `thinking.type: "adaptive"`.

## Solution Assessment

Yes. A typed config/compat capability is the narrowest maintainable path because it supports opaque proxy IDs without hardcoding corporate prefixes or requiring a simple configured provider to become a provider plugin.
