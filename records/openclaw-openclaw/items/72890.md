## Summary

Keep open. Current main still gates Anthropic adaptive-thinking payloads on hardcoded Claude model-id substrings, while the configured custom-provider schema and docs do not provide a typed way for an opaque Anthropic-compatible model id to opt into that behavior. The request is valid and narrow enough for a focused fix.

## What This Changes

The issue requests that custom Anthropic-compatible provider models be able to declare adaptive-thinking support through model metadata instead of relying only on hardcoded model-id matching.

## Best Possible Solution

Add a typed config-backed adaptive-thinking capability for configured Anthropic Messages models, wire it into both thinking-profile validation and the Anthropic payload decision, and document the field for custom provider proxies without adding customer-specific model-id prefixes.

## Reproduction Assessment

Yes. A unit reproduction can configure a custom `anthropic-messages` model with an opaque id, `reasoning: true`, and a thinking level; current main will take the budget-token branch because `supportsAdaptiveThinking(model.id)` only matches Claude-name substrings.

## Solution Assessment

Yes, if implemented as a typed config/compat capability consumed by the existing thinking and Anthropic transport paths. Hardcoding proxy prefixes would not be the best solution because it would repeat the same upgrade-fragile pattern reported here.

## Repair Work Prompt

Fix openclaw/openclaw#72890. Custom `models.providers` entries using `api: "anthropic-messages"` and opaque model ids cannot opt into Claude adaptive thinking, because current runtime checks only model-id substrings. Add a typed config-backed capability, preferably in the existing model `compat` contract unless the local pattern strongly suggests another typed field, so configured custom Anthropic Messages models can declare adaptive-thinking support. Wire that signal into the Anthropic Messages transport so any reasoning/thinking level for a declared adaptive-capable model sends `thinking: { type: "adaptive" }` and `output_config.effort` instead of `thinking: { type: "enabled", budget_tokens: ... }`. Also make thinking-policy validation expose/accept `/think adaptive` for those configured models. Do not add customer-specific prefixes such as `zyb-`, do not broaden adaptive defaults for unrelated models, and keep direct Anthropic/Bedrock behavior unchanged. Update config schema/types/generated docs as required, update user-facing docs for custom providers, and add a changelog entry crediting `Thanks @fuyizheng3120`.
