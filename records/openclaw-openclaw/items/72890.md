## Summary

Keep open. Current main still routes Anthropic adaptive-thinking support through hardcoded Claude model-id substrings, and configured custom `anthropic-messages` model metadata has no typed adaptive-thinking override that reaches either thinking-profile validation or the Anthropic payload builder. The request is valid, narrow, and suitable for one focused fix PR.

## What This Changes

The issue requests a config-declared adaptive-thinking capability for custom Anthropic-compatible provider model IDs so opaque proxy IDs can opt into adaptive thinking instead of relying on hardcoded Claude model-name substrings.

## Best Possible Solution

Add a typed config-backed adaptive-thinking capability for configured `anthropic-messages` models, consume it in thinking-profile validation and the Anthropic payload option resolver, keep the hardcoded Claude model-id list as fallback, and document the custom-provider proxy example without adding customer-specific prefixes.

## Reproduction Assessment

Yes. A unit reproduction can configure `models.providers.my-proxy` with `api: "anthropic-messages"`, `id: "zyb-high"`, `reasoning: true`, and a thinking level; current main will miss `supportsAdaptiveThinking(model.id)` and build a budget-token thinking payload instead of adaptive thinking.

## Solution Assessment

Yes. A typed config/compat capability is the narrowest maintainable fix because it lets opaque proxy model ids declare the behavior explicitly; hardcoding proxy prefixes would preserve the upgrade-fragile pattern reported here.
