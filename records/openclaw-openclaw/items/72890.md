## Summary

Keep open. Current main still gates Anthropic adaptive-thinking transport behavior on Claude model-id substrings, so an opaque custom `anthropic-messages` proxy model cannot opt into adaptive payloads through config. The report is actionable and localized enough for a focused repair PR.

## What This Changes

Request a config-declared adaptive-thinking capability for custom Anthropic-compatible model providers so opaque proxy model IDs can receive adaptive thinking payloads.

## Best Possible Solution

Add a typed model compat/config opt-in for Anthropic adaptive thinking, consume it in thinking-profile validation and Anthropic transport option resolution, keep the Claude model-id allowlist as fallback, and document a generic custom proxy example without adding customer-specific prefixes.

## Reproduction Assessment

Yes. A focused source-level reproduction can configure a custom `anthropic-messages` model with an opaque id, `reasoning: true`, and non-off thinking; current main follows the non-adaptive branch because only `model.id` substring checks reach the adaptive payload path.

## Solution Assessment

Yes. A config-backed explicit opt-in is the narrowest maintainable fix because it handles corporate/API-aggregator proxy IDs without hardcoding private prefixes or requiring a full provider plugin for a simple custom provider entry.
