## Summary

Keep open. Current main still does not honor configured ACP bindings for Slack, while the PR is a focused Slack-owned implementation and I did not find a concrete blocking bug in the supplied diff. The earlier Greptile verbose-log finding was addressed by the second PR commit.

## What This Changes

The PR adds Slack configured ACP binding normalization/matching, routes inbound Slack messages through configured ACP bindings when no runtime binding is active, gates dispatch on configured target readiness, adds Slack session-key regression tests, and cleans up two unrelated extension test lint issues.

## Best Possible Solution

Land a Slack-owned fix that uses the existing configured-binding SDK path, preserves runtime binding precedence, drops unavailable configured ACP targets instead of falling back to `main`, and keeps regression coverage for stable configured Slack ACP session keys. Splitting the unrelated test lint cleanup is optional maintainer hygiene, not a functional blocker.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: current main cannot compile Slack `bindings[].type = "acp"` because Slack has no binding provider, and inbound Slack routing never calls the configured-binding resolver.

## Solution Assessment

Yes. The PR is the narrowest maintainable direction because it keeps Slack-specific normalization and routing in the Slack plugin while reusing the existing generic configured-binding registry and readiness helper already used by other channels.
