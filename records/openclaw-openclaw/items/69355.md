## Summary

Keep this PR open. Current main still lacks the proposed non-blocking `openclaw doctor` rebuild/pristine npm-release guard: there is no `doctor-rebuild-guard` implementation or test, the doctor flow only calls the existing source-install and startup notes before normal health contributions, and the doctor docs do not describe install freshness warnings. The PR is not obsolete, but it needs review updates for the current doctor contribution architecture plus the existing functional/security review comments before merge.

## Best Possible Solution

Keep this PR open and ask for an update rather than closing it. The best path is to adapt the guard into the current doctor health contribution flow, reuse existing bounded registry-fetch patterns where possible, compare package and build-info versions, distinguish missing vs malformed metadata, honor cache freshness in offline/fallback paths, document the final behavior, and fix the changelog attribution before maintainer review/merge.

## Risks / Open Questions

- The proposed PR adds a normal `openclaw doctor` npm registry network fetch; without a bounded timeout, doctor can stall on blackholed or slow outbound HTTPS.
- The proposed cache/offline behavior can revive stale registry data and produce false `unreleased` warnings when current registry data is unavailable.
- The proposed metadata reader can evaluate the wrong version when `package.json.version` and `dist/build-info.json.version` diverge, weakening the corruption/rebuild signal.
- Current main has refactored doctor checks into the health contribution registry and lazy imports, so the PR likely needs rebase/integration updates before it matches current architecture.
- The proposed changelog entry shown in the PR context lacks the required `Thanks @...` attribution and still has `#TBD`, so it should be fixed before landing.
