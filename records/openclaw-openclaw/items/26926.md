## Summary

Keep this issue open. Current main still supports Synology Chat outbound URL-based media delivery, but inbound webhooks remain text-only: the payload type has no attachment fields, parsing requires `text`, delivery passes only text/user metadata, inbound context does not populate `MediaPath`/`MediaUrl`/`MediaType` fields, and the public docs only describe outbound files.

## Best Possible Solution

Implement inbound attachment support inside `extensions/synology-chat` after confirming Synology Chat's attachment webhook/API contract: extend the typed payload/parser, accept attachment-bearing messages without requiring text, safely download or stage accepted images, CSVs, and PDFs, populate the existing inbound media context fields, update `docs/channels/synology-chat.md`, and add focused webhook/context tests. Keep Synology-specific behavior in the plugin unless a missing generic plugin seam is proven.
