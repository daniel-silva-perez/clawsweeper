## Summary

Keep this issue open. Current main at ab5c8025c9d0a31873d311ae3e73f3664b0a1185 still handles Synology Chat inbound webhooks as text-only; the plugin has outbound URL-based file delivery, but no inbound image, CSV, PDF, filename, MIME type, or staged media path support. The prior Codex review comment from steipete identified the same gap, and current source, docs, and tests still agree.

## Best Possible Solution

Keep this issue open. The best path is to implement inbound attachment handling inside the bundled Synology Chat plugin after confirming Synology Chat's attachment webhook/API shape: extend the typed payload/parser, safely download or stage accepted images, CSVs, and PDFs through existing media-safe helpers, expose MediaPath(s), MediaUrl(s), MediaType(s), and filename metadata in the inbound context, update https://docs.openclaw.ai/channels/synology-chat, and add focused webhook plus inbound-context regression tests. Avoid moving Synology-specific behavior into core unless a missing generic plugin seam is proven.
