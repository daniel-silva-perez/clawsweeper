## Summary

Keep open. Current main still treats `[[reply_to_current]]` as a live reply directive across parser, final payload, streaming, and outbound normalization paths, and the strict per-agent config schema/docs do not expose an agent-scoped disable policy. The related #72627 overlaps but is broader because it also asks for auto-send suppression, so this remains the focused runtime-guard request.

## What This Changes

The issue requests an opt-in per-agent directive setting that disables runtime processing of `[[reply_to_current]]` while preserving explicit message-tool replies.

## Best Possible Solution

Add a backward-compatible per-agent directive policy at the shared parsing/threading boundary, default it to enabled, and apply the disabled state consistently to final payloads, streaming chunks, and outbound normalization without changing Feishu-specific delivery or explicit message-tool replies.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass text containing `[[reply_to_current]]` and a `currentMessageId` through `parseInlineDirectives` or `applyReplyThreading` and observe `replyToCurrent` or `replyToId` being emitted with no active-agent disable policy consulted.

## Solution Assessment

Yes. The requested opt-in runtime guard is the narrow maintainable direction if scoped to `reply_to_current`; a Feishu-only workaround or a broader auto-send suppression change would be a less appropriate first fix for this issue.
