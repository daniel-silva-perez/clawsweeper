## Summary

Current main still treats `[[reply_to_current]]` as shared reply metadata and has no strict per-agent directive policy, so the requested runtime guard is not implemented. The broader same-author #72627 overlaps this request, but this issue has narrower Feishu-specific impact and remains a valid focused repair candidate unless maintainers choose to consolidate the trackers.

## What This Changes

Request a per-agent configuration directive that prevents the runtime from processing accidental `[[reply_to_current]]` tags for selected agents.

## Best Possible Solution

Add a documented, backward-compatible per-agent directive policy and enforce it at the shared reply directive parsing/threading boundary for both final payloads and streaming chunks, while leaving Feishu-specific delivery code and explicit message-tool thread replies unchanged.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to pass text containing `[[reply_to_current]]` plus a `currentMessageId` through the directive parsing or reply payload threading helpers and observe that current main still produces `replyToId` for any agent.

## Solution Assessment

Yes, the requested direction is a narrow maintainable fix if implemented as shared parser policy resolved from the active agent. The safer variant gates only `reply_to_current` for opted-in agents and preserves explicit reply ids and non-reply directives.
