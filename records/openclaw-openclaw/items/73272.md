## Summary

Current main still processes `[[reply_to_current]]` through the shared directive parser, final reply payload threading, streaming chunks, and outbound normalization without an agent-scoped disable policy, while the strict per-agent config schema does not expose the requested `directives.replyToCurrent` field. This should stay open as a narrow valid repair candidate; the broader same-author #72627 overlaps but also asks for auto-send suppression, so this issue remains the focused runtime-guard slice.

## What This Changes

The issue requests a per-agent runtime directive setting that disables automatic processing of `[[reply_to_current]]` for selected agents.

## Best Possible Solution

Add a backward-compatible per-agent directive policy for `reply_to_current` at the shared parsing/threading boundary, apply it consistently to final payloads, streaming chunks, and outbound normalization, and document the config without changing Feishu-specific delivery or explicit `message` tool replies.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass text containing `[[reply_to_current]]` plus a `currentMessageId` through `parseInlineDirectives` or `applyReplyThreading` and observe that current main emits `replyToCurrent` or `replyToId` without consulting an active agent policy.

## Solution Assessment

Yes. The requested direction is the narrow maintainable fix if scoped to an opt-in per-agent disable for `reply_to_current`; the broader auto-send suppression from #72627 should remain separate unless a maintainer explicitly expands the PR scope.
