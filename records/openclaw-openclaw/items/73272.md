## Summary

Keep open. Current main still treats `[[reply_to_current]]` as a core inline delivery directive, including explicit-tag handling that can bypass channel `replyToMode: "off"`, and there is no per-agent `directives.replyToCurrent` config surface to disable that parsing for one agent.

## What This Changes

The issue requests a per-agent directive such as `agents.<id>.directives.replyToCurrent=disabled` so selected agents cannot trigger runtime processing of `[[reply_to_current]]`.

## Best Possible Solution

Add a documented per-agent reply-directive policy, resolve it from the active session agent, and apply it at the shared directive parsing boundary before channel adapters see `replyToId`; keep channel-specific Feishu behavior unchanged except for receiving no reply id when that policy disables `reply_to_current`.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is `applyReplyThreading` with a payload containing `[[reply_to_current]]` and a current message id; current tests already assert that this resolves to `replyToId`, including explicit-tag paths that survive channel off mode.

## Solution Assessment

Yes, the requested direction is a reasonable narrow fix if implemented as a shared parser policy rather than a Feishu-only workaround. The safer shape is to gate only `reply_to_current` for the selected agent and leave other delivery directives intact.
