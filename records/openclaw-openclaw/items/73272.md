## Summary

Keep open. Current main still treats `[[reply_to_current]]` as a shared inline delivery directive, has no `agents.list[].directives.replyToCurrent` or equivalent per-agent config surface, and Feishu outbound delivery forwards resulting `replyToId` values into Feishu reply calls. The existing ClawSweeper comment on the issue is consistent with current source and tests.

## What This Changes

Add per-agent configuration to disable runtime processing of `[[reply_to_current]]` so selected agents cannot turn accidental tag text into reply metadata.

## Best Possible Solution

Introduce a generic per-agent reply-directive policy, resolve it for the active session agent before shared final and streaming directive parsing, and document the supported config. Feishu should remain a consumer of reply metadata rather than owning a Feishu-specific workaround.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is `applyReplyThreading` with a payload containing `[[reply_to_current]]`, a current message id, and implicit threading denied or channel off mode; current tests assert that explicit tags still produce `replyToId`.

## Solution Assessment

Yes. The requested direction is the narrowest maintainable solution if implemented as a shared parser policy rather than a Feishu-only adapter override, with the guard scoped only to `reply_to_current`.
