## Summary

Current main still processes `[[reply_to_current]]` as shared reply metadata and lacks a strict per-agent directive policy, so the requested runtime guard is not implemented. The broader same-author #72627 overlaps, but this issue remains a focused, actionable repair candidate for disabling only this directive for selected agents.

## What This Changes

Request a per-agent configuration directive that prevents the runtime from processing accidental `[[reply_to_current]]` tags for selected agents.

## Best Possible Solution

Add a documented, backward-compatible per-agent directive policy and enforce it at the shared reply directive parsing/threading boundary for final payloads, streaming chunks, and outbound normalization, while leaving Feishu-specific delivery and explicit message-tool replies unchanged.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to pass text containing `[[reply_to_current]]` plus a `currentMessageId` through `parseInlineDirectives` or `applyReplyThreading` and observe that current main produces `replyToId` without considering the active agent.

## Solution Assessment

Yes. The requested direction is a narrow maintainable fix if implemented as shared parser policy resolved from the active agent; the safer variant gates only `reply_to_current` for opted-in agents and preserves explicit reply ids plus non-reply directives.
