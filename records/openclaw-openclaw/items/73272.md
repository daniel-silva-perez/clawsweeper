## Summary

Keep open. Current main still treats `[[reply_to_current]]` as an unconditional assistant-output directive across parser, final reply threading, streaming, embedded Pi, and outbound normalization, while the strict per-agent config schema has no `directives.replyToCurrent` policy. The related #72627 is open and broader because it also asks for auto-send suppression, so this focused runtime-guard slice remains actionable.

## What This Changes

Add an opt-in per-agent directive setting to disable runtime processing of `[[reply_to_current]]` for selected agents.

## Best Possible Solution

Add a backward-compatible, opt-in per-agent directive policy for `reply_to_current`, apply it consistently at shared directive parsing and reply-normalization boundaries, document the config, and leave auto-send suppression to the broader related item unless maintainers intentionally expand scope.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass assistant text containing `[[reply_to_current]]` and a `currentMessageId` through `parseInlineDirectives` or `applyReplyThreading` and observe `replyToCurrent` / `replyToId` emitted without consulting agent config; strict agent config currently has no accepted directive policy field.

## Solution Assessment

Yes, if scoped narrowly. An opt-in runtime guard at shared directive boundaries is the maintainable fix for accidental tag processing; a channel-specific Feishu workaround or bundled prompt rule would not physically prevent the runtime behavior.
