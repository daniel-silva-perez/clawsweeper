## Summary

Keep open. Current `main` still processes `[[reply_to_current]]` unconditionally through parser, final reply, streaming, embedded Pi, and outbound paths, and the strict per-agent config schema still has no `directives.replyToCurrent` policy. The related #72627 remains open and broader because it also asks for auto-send suppression, so this issue is still useful as the focused runtime-guard slice.

## What This Changes

The issue requests an opt-in per-agent runtime directive that disables processing of accidental `[[reply_to_current]]` output while preserving explicit message-tool replies.

## Best Possible Solution

Add a backward-compatible, opt-in per-agent directive policy for `reply_to_current` at the shared directive parsing and reply-normalization boundaries, apply it consistently to final replies, streaming chunks, embedded Pi handling, and outbound payloads, and document the config without changing Feishu-specific delivery or explicit message-tool replies.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass assistant text containing `[[reply_to_current]]` plus a `currentMessageId` through `parseInlineDirectives` or `applyReplyThreading` and observe `replyToCurrent` / `replyToId` emitted without consulting any agent policy; config validation has no accepted `agents.list[].directives.replyToCurrent` field.

## Solution Assessment

Yes, if scoped narrowly. The requested opt-in runtime guard is the maintainable fix for accidental tag processing; the broader auto-send suppression request in #72627 should remain separate unless maintainers intentionally expand the work.
