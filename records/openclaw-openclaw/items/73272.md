## Summary

Keep open. Current main still parses `[[reply_to_current]]` as a shared inline delivery directive, preserves explicit reply tags even when channel reply mode is off, and has no per-agent config/schema/docs surface for disabling only this directive. The older #72627 overlaps with this request, but it is broader, so this narrower issue is still a useful implementation target that can be clustered with that report.

## What This Changes

The issue requests a per-agent runtime/config directive to disable processing of `[[reply_to_current]]` so selected agents cannot trigger current-message reply threading accidentally.

## Best Possible Solution

Add a documented per-agent reply-directive policy, resolve it from the active session agent, and enforce it at the shared directive parsing/threading boundary for final payloads, streaming chunks, and outbound payload normalization while leaving Feishu delivery unchanged except for receiving no reply id when the policy disables `reply_to_current`.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is `applyReplyThreading({ payloads: [{ text: "[[reply_to_current]] hi" }], replyToMode: "off", replyToChannel: "slack", currentMessageId: "42" })`, which current tests show resolves and preserves `replyToId` for explicit tags; the same parser path is shared before Feishu delivery.

## Solution Assessment

Yes, the requested direction is a narrow maintainable fix if implemented as a shared per-agent parser policy rather than a Feishu-only workaround. The safer variant is to gate only `reply_to_current` and keep explicit id replies plus other directives intact.
