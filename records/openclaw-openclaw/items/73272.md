## Summary

Current `main` still lacks the requested per-agent disable policy: shared parsing, final reply threading, streaming directives, and outbound normalization continue to treat `[[reply_to_current]]` as an active reply directive, while the strict agent config schema exposes no `directives.replyToCurrent` field. The open related #72627 is broader and not merged, so this should stay open as the focused runtime-guard repair slice.

## What This Changes

The issue requests an opt-in per-agent runtime directive that disables processing of accidental `[[reply_to_current]]` output while preserving explicit message-tool replies.

## Best Possible Solution

Add a backward-compatible, opt-in per-agent directive policy for `reply_to_current` at the shared directive parsing and reply-normalization boundaries, apply it consistently to final replies, streaming chunks, and outbound payloads, and document the config without changing Feishu-specific delivery or explicit message-tool replies.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass text containing `[[reply_to_current]]` plus a `currentMessageId` through `parseInlineDirectives` or `applyReplyThreading` and observe `replyToCurrent` / `replyToId` being emitted without consulting an agent policy; config validation has no accepted `agents.list[].directives.replyToCurrent` field.

## Solution Assessment

Yes, if scoped narrowly. The requested opt-in runtime guard is the maintainable fix for accidental tag processing; the broader auto-send suppression request in #72627 should remain separate unless maintainers intentionally expand the work.
