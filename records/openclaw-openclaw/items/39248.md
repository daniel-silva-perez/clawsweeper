## Summary

Keep this issue open. Current main still routes native `sessions_spawn` child sessions through the same non-main sandbox classification and non-blocking gateway dispatch path, and I found no current regression test or live proof that a Docker-backed child under `agents.defaults.sandbox.mode: "non-main"` initializes, records session/transcript progress, and surfaces sandbox-start failures. Related #57992 is open and overlaps on silent sandbox-container failure, but it is not a clear replacement for this deterministic `sessions_spawn` repro.

## Best Possible Solution

Keep this issue open and add a targeted regression or live Testbox proof for native `sessions_spawn` with `agents.defaults.sandbox.mode: "non-main"`. The fix should prove that the child Docker-backed run actually starts, writes or updates transcript/session state, records `systemSent` or token progress where applicable, and surfaces sandbox initialization failures in logs/session status instead of leaving an accepted child run stuck with zero messages.
