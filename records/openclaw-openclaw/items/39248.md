## Summary

Keep this issue open. Current main still classifies native subagent child sessions as non-main sandbox candidates and dispatches the child run through a non-blocking gateway call that returns the accepted ack before sandbox initialization is proven. I found no regression test or live proof on current main that `sessions_spawn` with `agents.defaults.sandbox.mode: "non-main"` initializes the Docker-backed child and writes transcript/token state. Related #57992 overlaps, but it is also open and describes a transient container-creation failure rather than clearly superseding this deterministic repro.

## Best Possible Solution

Keep the issue open and add a targeted regression test or live Testbox proof for native `sessions_spawn` with `agents.defaults.sandbox.mode: "non-main"`. The fix should prove the child run actually starts, creates or updates transcript/session state, records token or `systemSent` progress where applicable, and surfaces sandbox initialization failures in logs/session status instead of leaving an accepted child run stuck at zero messages.
