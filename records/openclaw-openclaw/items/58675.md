## Summary

Keep open. Current main still lacks GitHub Copilot chat model discovery, and the PR is an active, useful implementation, but the latest diff still has a blocking security issue: the new authenticated discovery fetch can use an operator-configured Copilot base URL while building an SSRF policy that skips private-network checks for that hostname.

## What This Changes

The PR adds GitHub Copilot chat model discovery via an authenticated `/models` request, merges discovered models and IDE headers into provider catalog resolution, deep-merges Copilot headers with explicit provider config, and adds related discovery tests.

## Best Possible Solution

Keep the provider-owned additive discovery design, but build the discovery SSRF policy from a hostname allowlist that does not skip private-network checks, merge in the configured provider request private-network opt-in only when enabled, and add a focused regression test for configured private/link-local discovery base URLs before rerunning the targeted Copilot discovery checks.

## Reproduction Assessment

Yes for the review finding. On the PR head, configure `models.providers.github-copilot.baseUrl` to a private, link-local, or private-DNS HTTP(S) endpoint without `models.providers.github-copilot.request.allowPrivateNetwork`; the catalog path passes that URL into `discoverCopilotModels`, whose `allowedHostnames` policy skips the private-network guard.

## Solution Assessment

No, not as currently proposed. The discovery feature is the right product shape, but the fetch policy should preserve the existing provider private-network contract before this is the narrowest maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Require private-network opt-in for discovery fetches:** `extensions/github-copilot/discovery.ts:89`
  - body: `discoverCopilotModels` builds its policy with `ssrfPolicyFromHttpBaseUrlAllowedHostname(baseUrl)`, and current SSRF handling skips private-network checks for `allowedHostnames`. Since `baseUrl` can come from `models.providers.github-copilot.baseUrl`, a private or link-local target can receive the Copilot bearer token without `request.allowPrivateNetwork`; use a hostname allowlist plus the provider request opt-in instead.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff still introduces a security-sensitive Copilot discovery request that can bypass the provider private-network opt-in.

Concerns:

- **[high] Discovery can bypass private-network opt-in:** `extensions/github-copilot/discovery.ts:89`
  - body: The discovery fetch sends `Authorization: Bearer <copilotToken>` to the configured base URL while using an `allowedHostnames` policy that skips private-network checks. A configured private, link-local, or private-DNS target should require the provider request opt-in before receiving that token.
