## Summary

Keep open. Current main still returns an empty GitHub Copilot provider catalog and has no chat-model discovery helper, so this PR is not obsolete. The provider-owned discovery design is useful, but the latest PR head still introduces a security-sensitive fetch policy problem for configured Copilot discovery base URLs.

## What This Changes

The PR adds a GitHub Copilot `/models` discovery helper, wires discovered models and Copilot IDE headers into provider catalog resolution, deep-merges Copilot headers with explicit provider config, and adds discovery/implicit-provider tests.

## Best Possible Solution

Keep the provider-owned additive discovery approach, but build discovery fetch policy from a hostname allowlist that does not skip private-network checks, merge in `models.providers.github-copilot.request.allowPrivateNetwork` only when explicitly enabled, and add a regression test for private/loopback configured base URLs before rerunning targeted Copilot discovery checks.

## Reproduction Assessment

Yes for the review finding: on the PR head, configure `models.providers.github-copilot.baseUrl` to a private or loopback HTTP endpoint without `request.allowPrivateNetwork`; the catalog path passes that base URL to `discoverCopilotModels`, which uses an allowed-hostname policy that current SSRF code treats as a private-network bypass. This is source-verifiable; I did not run tests because this was a read-only review.

## Solution Assessment

No, not as currently proposed. The discovery feature is the right shape, but the fetch policy should honor the existing model-provider private-network contract before this is the narrowest maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Require private-network opt-in for discovery fetches:** `extensions/github-copilot/discovery.ts:91`
  - body: `discoverCopilotModels` can be called with `models.providers.github-copilot.baseUrl`, but this policy is built with `allowedHostnames`. Current SSRF code skips private-network checks for allowed hostnames, so a configured `http://127.0.0.1` or metadata-style base URL can receive the Copilot bearer token without `request.allowPrivateNetwork`; use a hostname allowlist plus the provider request opt-in instead.
  - confidence: 0.89

## Security Review

Status: needs_attention

Summary: The diff still introduces a security-sensitive Copilot discovery fetch that can bypass the provider private-network opt-in.

Concerns:

- **[high] Discovery can bypass private-network opt-in:** `extensions/github-copilot/discovery.ts:91`
