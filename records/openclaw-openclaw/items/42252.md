## Summary

Keep open. Current main has useful adjacent improvements for gateway service drift checks, `gateway status` degraded/read-probe reporting, and Telegram duplicate-poller hints, but `openclaw doctor` still does not provide the requested consolidated mixed LaunchAgent/runtime/RPC diagnosis or final healthy/degraded/down verdict. The mixed output path remains: an RPC close can print `Gateway not running.` and a later daemon phase can still print a runtime summary.

## Best Possible Solution

Keep this issue open and implement a doctor-owned gateway diagnosis collector. It should gather service installed/loaded state, LaunchAgent supervision, runtime PID/status, RPC/read-probe result, port listener ownership, service config drift, extra gateway-like services, recent channel-specific signals such as Telegram `getUpdates` 409 conflicts, and last gateway errors. Then `openclaw doctor` should print one final healthy/degraded/down verdict with deterministic next steps and, if maintainers want it, a compact JSON diagnosis block.
