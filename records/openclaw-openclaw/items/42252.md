## Summary

Keep open. Current main has useful adjacent work for gateway status, LaunchAgent/service drift, port/runtime diagnostics, and Telegram 409 duplicate-poller hints, but `openclaw doctor` still does not produce the consolidated healthy/degraded/down gateway diagnosis this issue asks for. The doctor flow still emits gateway RPC health and daemon runtime/service repair as separate notes, so the mixed-message UX remains possible.

## Best Possible Solution

Keep this issue open and implement a doctor-owned gateway diagnosis collector. It should gather LaunchAgent/service installed and loaded state, runtime PID/status, RPC/read-probe result, port listener ownership, service config drift, extra gateway-like services, recent Telegram `getUpdates` 409 signals, and recent gateway errors, then print one final healthy/degraded/down verdict with probable causes, deterministic next steps, and an optional compact JSON diagnosis block.
