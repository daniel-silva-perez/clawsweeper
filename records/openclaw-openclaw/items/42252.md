## Summary

Keep open. Current main has adjacent gateway health, service drift, runtime, port, and Telegram 409 diagnostics, but doctor still emits them as separate checks rather than one consolidated healthy/degraded/down gateway diagnosis with probable causes, deterministic next steps, and an optional machine-readable diagnosis block.

## Best Possible Solution

Keep this issue open and implement a doctor-owned gateway diagnosis collector. It should gather service installed/loaded state, runtime PID/status, RPC probe result, port listener ownership, service config drift, extra gateway-like services, recent Telegram getUpdates 409 signals, and recent gateway errors, then print one final healthy/degraded/down verdict with probable causes, deterministic next steps, and an optional compact JSON diagnosis block.
