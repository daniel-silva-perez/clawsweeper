## Summary

Keep open. Current main has adjacent gateway status, service-drift, runtime, port, and Telegram 409 diagnostics, but `openclaw doctor` still does not produce the consolidated healthy/degraded/down gateway verdict, ordered recovery plan, or optional diagnosis JSON requested here. The mixed-message path remains possible because doctor still reports RPC health and daemon runtime/service facts as separate phases.

## Best Possible Solution

Keep this open and implement a doctor-owned gateway diagnosis collector. It should gather LaunchAgent/service installed and loaded state, runtime PID/status, RPC or read-probe result, port listener ownership, service config drift, extra gateway-like services, recent Telegram getUpdates 409 signals, and recent gateway errors, then print one final healthy/degraded/down verdict with probable causes, deterministic next steps, and an optional compact JSON diagnosis block.
