## Summary

Keep open. Current main has partial Docker friendliness for the raw image CMD and the scripted setup path, but the specific Dockge-style “paste Compose + .env and start” workflow is still not implemented. The bundled Compose service overrides the Dockerfile CMD, does not pass `--allow-unconfigured`, requires config/workspace path variables, and the docs still require setup/onboarding or a manual pre-start config write before `docker compose up`.

## Best Possible Solution

Keep this open and implement a first-class Dockge/Compose bootstrap path. The safest direction is a Compose-compatible entrypoint or documented template that can create a minimal local gateway config on first start, persist it under the mounted config directory, generate or require a gateway token, set safe bind/origin defaults, and preserve the current startup/auth guardrails for normal runtime.

## Risks / Open Questions

- A naive Compose-only change that starts unconfigured while binding to LAN could conflict with the existing auth and Control UI origin safety guardrails.
- The current `.env.example` does not provide the Compose-required `OPENCLAW_CONFIG_DIR` and `OPENCLAW_WORKSPACE_DIR`, so the Dockge paste workflow may need both config generation and clearer env defaults.
