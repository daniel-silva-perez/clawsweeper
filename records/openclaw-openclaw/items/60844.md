## Summary

Keep open. Current main still treats heartbeat session isolation as explicit opt-in, and the current tests/docs still describe unset `isolatedSession` as shared-session behavior, so this PR is not obsolete. The proposed direction is narrow, but the branch needs a current-main update before merge because default-on isolation must preserve newer base-session event consumption, delivery-routing, and docs surfaces.

## What This Changes

This PR makes heartbeat runs use an isolated `:heartbeat` session by default, keeps `isolatedSession: false` as the shared-session opt-out, updates heartbeat docs/type comments, and adds regression coverage for the default, opt-out, and main-session abort-state behavior.

## Best Possible Solution

Rebase the PR onto current main and apply the default flip only on top of the current heartbeat runner: keep `consumeInspectedSystemEvents`, stale isolated-key canonicalization, untrusted-event classification, and isolated delivery-route suppression intact; then update the current heartbeat/config-agent docs and add focused cron/wake event-consumption coverage for the new default.

## Reproduction Assessment

Yes for the PR finding: PR head 753dfe5 switches omitted `isolatedSession` to the `:heartbeat` session, while its successful-run path does not contain the current main base-event consumption call. Current main's isolated-key tests show the expected invariant that base-session cron events are consumed and not replayed.

## Solution Assessment

No as currently staged. The one-line opt-out condition is the narrow implementation shape, but the branch is stale enough that the safe solution is to rebase it onto current main and preserve the newer event-drain and delivery-routing behavior before merge.
