## Summary

Keep open: current main does not implement the requested pairing limit overrides, and the proposed PR is not ready to merge because it adds security-sensitive env knobs with a malformed-integer parser defect and no docs/threat-model alignment.

## What This Changes

This PR changes `src/pairing/pairing-store.ts` to resolve pending-pairing TTL/max from `OPENCLAW_PAIRING_PENDING_TTL_MINUTES` and `OPENCLAW_PAIRING_PENDING_MAX` at module load, and adds resolver unit tests in `src/pairing/pairing-store-env.test.ts`.

## Best Possible Solution

If maintainers want operator-tunable DM pairing limits, land a stricter version with full-string integer validation, explicit docs and threat-model updates for env names/defaults/bounds, and tests that cover both resolver behavior and store behavior under overridden TTL/max values.

## Reproduction Assessment

Yes. The parser defect is reproducible by inspection of the proposed `Number.parseInt` call: `3.7` and `3abc` become `3`, and the supplied PR test expects fractional acceptance even though the PR says malformed or fractional input should fall back.

## Solution Assessment

No. The proposed direction may be acceptable, but this patch is not the best merge shape until the integer parser is strict and the security/docs contract for configurable pairing limits is reviewed and documented.

## Review Findings

- **[P2] Reject malformed integer environment values:** `src/pairing/pairing-store.ts:71`
  - body: The new parser uses `Number.parseInt`, so values like `OPENCLAW_PAIRING_PENDING_TTL_MINUTES=3.7` or `3abc` are accepted as `3` instead of falling back. The PR describes these knobs as integer-only and says malformed/fractional input should degrade to defaults; silently applying a truncated TTL/max can change the pairing security window in a misconfigured deployment.
