## Summary

Keep open: current main still has hard-coded DM pairing pending TTL/max values, so this PR is not obsolete, but the proposed parser accepts malformed integer prefixes despite the stated integer-only/fallback contract and the change needs maintainer review because it alters security-sensitive pairing limits.

## What This Changes

This PR adds module-load environment overrides for DM pairing pending request TTL and max count in `src/pairing/pairing-store.ts`, plus new Vitest coverage for the resolver.

## Best Possible Solution

If maintainers want operator-tunable DM pairing limits, land a stricter version that rejects malformed integer input, documents the env names/defaults/bounds in pairing and security docs, and includes targeted tests for both resolver behavior and store behavior under overridden TTL/max values.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Reject malformed integer environment values:** `src/pairing/pairing-store.ts:71`
  - body: The new parser uses `Number.parseInt`, so values like `OPENCLAW_PAIRING_PENDING_TTL_MINUTES=3.7` or `3abc` are accepted as `3` instead of falling back. The PR describes these knobs as integer-only and says malformed/fractional input should degrade to defaults; silently applying a truncated TTL/max can change the pairing security window in a misconfigured deployment. Use strict integer validation and update the fractional test to expect the default.
  - confidence: 0.88
