## Summary

Keep open. Current main still keys normal Feishu inbound messages at the account/chat level, while the conversation router already supports group sender/topic scoped peer IDs. The PR is a narrow, coherent fix that reuses the existing group-session resolver and passes the resolved account config into the queue key path; I found no blocking review findings in the diff, but normal maintainer review and CI are still needed.

## What This Changes

The PR threads Feishu account config into message-handler registration, updates `getFeishuSequentialKey` to use Feishu group-session peer resolution for group chats, adds sequential-key tests for group/session scopes, and updates two Matrix approval test fixtures.

## Best Possible Solution

Land the targeted queue-key alignment after maintainer review and green CI, keeping `resolveFeishuGroupSession` as the single source of truth for Feishu group peer IDs so routing and serialization stay in sync.

## Reproduction Assessment

Yes. The reproduction path is code-level and testable: configure Feishu with `groupSessionScope` such as `group_topic`, send two normal group messages in different roots/topics, and current main will enqueue both under the same `feishu:<accountId>:<chatId>` key even though routing resolves different peer IDs.

## Solution Assessment

Yes. Reusing `resolveFeishuGroupSession` and the resolved per-account Feishu config is the narrowest maintainable fix because it avoids duplicating topic/sender fallback rules and preserves the existing per-chat `/stop` and `/btw` lanes.
