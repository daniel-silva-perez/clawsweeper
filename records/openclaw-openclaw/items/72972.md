## Summary

Keep open. Current main still serializes normal Feishu inbound work at the whole-chat key while routing can already split grouped sessions by topic or sender, and this PR is a narrow, plausible alignment of those two surfaces. I found no blocking functional or security findings; the remaining action is maintainer review, freshness/rebase handling, and CI validation.

## What This Changes

The PR threads Feishu account config into event-handler registration, updates Feishu sequential queue keys to reuse grouped-session peer resolution, adds Feishu sequential-key coverage, and refreshes Matrix approval test fixtures.

## Best Possible Solution

Land the targeted queue-key alignment after maintainer review and green CI, keeping `resolveFeishuGroupSession` as the single source of truth for Feishu group peer IDs so routing and serialization stay in sync.

## Reproduction Assessment

Yes. Configure Feishu with `groupSessionScope: "group_topic"` or `"group_topic_sender"`, send normal group messages in different topics or roots in the same chat, and current main will route them to different peer IDs while queuing them under the same `feishu:<accountId>:<chatId>` key.

## Solution Assessment

Yes. Reusing `resolveFeishuGroupSession` with the resolved Feishu account config is the narrow maintainable fix because it avoids duplicating topic/sender fallback rules and preserves the existing per-chat `/stop` and `/btw` lanes.
