## Summary

Keep open. Current main still serializes normal Feishu group messages at the chat-level queue key while routing can already split those same messages by group topic or sender, and the PR's code change aligns the queue key with the existing group-session resolver. I found no functional or security blocker in the patch, but the user-facing Feishu fix is missing the required changelog entry and needs normal maintainer review plus fresh CI before merge.

## What This Changes

The PR threads Feishu account config into receive-handler registration, makes `getFeishuSequentialKey` use Feishu group-session peer resolution for group chats, adds Feishu sequential-key scope coverage, and updates Matrix approval test fixtures.

## Best Possible Solution

Land the targeted queue-key alignment after adding the required changelog entry and getting green relevant CI, keeping `resolveFeishuGroupSession` as the single source of truth for Feishu group peer IDs so routing and serialization stay aligned.

## Reproduction Assessment

Yes. A code-level reproduction is clear: configure Feishu with `groupSessionScope: "group_topic"` or `"group_topic_sender"`, send two normal group messages in different roots/topics, and current main queues both under `feishu:<accountId>:<chatId>` even though routing resolves different peer IDs.

## Solution Assessment

Yes for the code path. Reusing the existing Feishu group-session resolver is the narrowest maintainable fix because it preserves `/stop` and `/btw` chat-level lanes while avoiding duplicate topic/sender fallback logic; the patch still needs the required changelog entry.
