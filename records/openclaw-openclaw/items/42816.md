## Summary

Keep open. Current main has several restart-adjacent mechanisms, but they are narrower than the requested generic checkpoint, resume, verify, and proactive report loop for work that reboots the host or kills the OpenClaw process.

## What This Changes

The issue requests durable task state persistence before service or host restarts and automatic verification plus result reporting to the original user after OpenClaw comes back online.

## Best Possible Solution

Build a generic durable continuation contract on top of the existing restart sentinel, task ledger, Task Flow, and recovery hooks: persist the session route, expected checks, notification policy, and authorized continuation before restart-causing work, then run bounded verification after startup and deliver the final result to the original channel.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is code and docs inspection: normal interactive chat is not generally task-backed, restart sentinels are explicit-path only, and recovery hooks do not provide generic post-reboot verification plus user delivery.

## Solution Assessment

No, current main is not the complete solution for the requested behavior. The maintainable path is to extend the existing restart/task/flow surfaces rather than add a separate persistence system.
