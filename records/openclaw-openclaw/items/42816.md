## Summary

Keep open. Current main has useful partial machinery: gateway agent RPC runs can be recorded in the task ledger, Task Flow can persist deliberate multi-step orchestration, explicit gateway/config/update restarts can write restart sentinels with delivery context, and package-manager updates verify the restarted version. That still does not satisfy this issue’s broader request for a generic checkpoint/resume/report loop for arbitrary work that reboots the host or kills the OpenClaw process. Main-session recovery resumes with `deliver: false`, task maintenance can mark disappeared backing state as `lost`, sub-agent docs still say pending announce-back work is lost across gateway restarts, and I did not find a general expected-outcome/pending-notification checkpoint API.

## Best Possible Solution

Keep this issue open as the broader product gap. The best path is a generic durable continuation contract layered on the existing restart sentinel, task ledger, and Task Flow primitives: before an owner-authorized restart/reboot/process-killing operation, persist the session key, requester delivery route, expected checks, intended outcome, and notification policy; on startup, run a bounded verification step or resume turn and deliver a final report to the original channel. Add coverage for explicit gateway restart/config/update paths and an arbitrary host reboot/process-death path, plus docs that define which operations are recoverable.

## Risks / Open Questions

- Existing restart sentinels, task records, Task Flow, and sub-agent recovery are real partial solutions, so a future implementation should extend those rather than adding a parallel persistence system.
- A full generic post-reboot continuation feature would need tight owner authorization, bounded verification semantics, and careful delivery routing to avoid surprising post-startup command execution.
