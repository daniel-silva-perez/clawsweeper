## Summary

Keep open. Current main has useful partial restart machinery, but it does not yet implement the requested general task checkpoint/resume/report loop for work that reboots the host or kills the OpenClaw process. Background tasks are durable for detached work, explicit gateway restart/update/config paths can write restart sentinels, and restart-aborted sessions have recovery hooks, but normal interactive chat turns are not task records, main-session recovery re-enters with `deliver: false`, and sub-agent docs still say pending announce-back work is lost across gateway restarts.

## Best Possible Solution

Keep this issue open as the broader product gap. The best path is to build a generic durable continuation/checkpoint contract on top of the existing restart sentinel and task/Task Flow registries: when an owner-authorized tool or workflow is about to restart the gateway or reboot the host, persist the session key, delivery route, intended verification checks, expected outcomes, and notification policy; on startup, run a bounded verification turn or deterministic check and deliver a final result to the original channel. It should include tests for explicit gateway restart, config/update restart, and an arbitrary host reboot/process-death path, plus docs explaining which operations are recoverable.

## Risks / Open Questions

- The existing restart sentinel, task ledger, Task Flow, and subagent recovery pieces are real partial solutions; any implementation should extend them rather than adding a separate persistence system.
- A full feature touches owner-only host control and post-boot automation, so it needs careful authorization and bounded verification semantics before running or reporting after a reboot.
