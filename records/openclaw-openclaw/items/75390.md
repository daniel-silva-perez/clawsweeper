## Summary

Keep this issue open. Current main and the latest tagged release still define both visibleReplies config paths as strict string enums, and the config reload path skips invalid snapshots; the paired open PR #75392 is the right active implementation candidate, so the issue should not be closed before that PR merges or is replaced.

## What This Changes

The issue requests accepting boolean values for `messages.visibleReplies` and `messages.groupChat.visibleReplies` instead of letting them invalidate config and stop channel reloads.

## Best Possible Solution

Keep the issue open and review/land #75392, or an equivalent narrow fix, that normalizes boolean values at the config schema boundary while preserving the downstream `"automatic" | "message_tool" | undefined` runtime type and adding targeted schema tests for both config paths.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: putting `true` or `false` at either reported config path reaches strict `z.enum` schemas through `OpenClawSchema.safeParse`, which makes the snapshot invalid and causes reload to skip the update.

## Solution Assessment

Yes. Schema-boundary coercion is the narrowest maintainable solution because startup, hot reload, doctor, and config command paths all flow through the same schemas, and downstream channel behavior can stay typed to the existing enum values.
