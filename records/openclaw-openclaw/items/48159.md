## Summary

Keep this issue open. Current main still lacks a hard per-session ephemeral/no-persistence contract: there is no `session.ephemeral` schema field, no `SessionEntry` privacy state, transcript/session-store writes still persist files and metadata, memory flush still runs through the normal persistence pipeline, and plugin/session hook context has no `isEphemeral()`-style signal. The related #62436 remains useful but narrower: it covers stateless HTTP passthrough, not secure chat across all session/channel/plugin layers.

## Best Possible Solution

Keep this issue open for a core design and implementation pass. The best path is an additive `ephemeral` session contract in config and runtime session state, hard no-write gates in transcript/session-store/compaction/memory-flush paths, a documented plugin SDK/session context persistence signal, clear command/UI/status verification surfaces, and focused tests proving that ephemeral sessions do not create durable transcripts, searchable session memory, memory files, or surviving session metadata. Treat #62436 as a narrower HTTP passthrough companion, not the canonical replacement for this secure chat request.

## Risks / Open Questions

- Closing this now would hide an unresolved privacy/security feature request: current main still persists transcripts, session metadata, compaction summaries, and memory-flush side effects without a hard per-session opt-out.
- The implementation surface is cross-cutting. A partial prompt-only or UI-only mode would not satisfy the issue because persistence must be gated in transcript writes, session store updates, memory flush, compaction/context-engine flows, and plugin SDK contracts.
- The HTTP docs use “stateless per request” to mean a new session key per call, while the code still uses the normal agent/session pipeline. That wording can continue to confuse users until #62436 or this broader issue is addressed.
