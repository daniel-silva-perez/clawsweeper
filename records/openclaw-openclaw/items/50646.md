## Summary

Keep open. Current main still has the reported observability gap: default session-tool visibility is restricted to `tree`, `sessions_list` silently filters rows that fail the visibility guard, and the returned payload only reports the visible count and rows. The related #50621 comment is useful context for session-state visibility pain, but it does not supersede this narrower UX issue.

## What This Changes

Request that session-listing tools disclose when returned sessions are scoped by `tools.sessions.visibility` so restricted results are not mistaken for the complete store.

## Best Possible Solution

Add an additive visibility disclosure to the session-listing surface that the model sees, including the effective visibility mode and a clear restricted-results note whenever the mode is not `all` or sandbox/spawned scoping applies. Keep the existing default access policy unchanged, and update docs/tests so the restricted count cannot be mistaken for the full store.

## Reproduction Assessment

Yes. A focused unit reproduction can mock `sessions.list` with multiple sessions, run `sessions_list` under default `tree` visibility from one requester, and observe that hidden rows are dropped while the result contains only `count` and `sessions` with no scope note.

## Solution Assessment

Yes, the requested direction is the right fix if implemented as additive disclosure. Changing the default visibility or broadening access would be riskier; the maintainable path is to expose effective scope/filtered metadata without changing authorization behavior.

## Repair Work Prompt

Fix the session-list visibility observability gap from https://github.com/openclaw/openclaw/issues/50646. Current `sessions_list` applies `tools.sessions.visibility` and silently drops rows, then returns only `count` and `sessions`; add an additive disclosure visible in the tool JSON/text whenever effective visibility is restricted (`self`, `tree`, `agent`, sandbox clamp, or spawned-only restriction). Include the effective visibility mode and a clear note such as `Session visibility is restricted (tools.sessions.visibility=tree). Showing only current session scope.` Include a filtered count only when it can be computed from already-fetched rows without broadening access or adding expensive extra scans. Do not change default visibility, do not loosen cross-agent or sandbox access, and keep any Gateway protocol additions backward-compatible. Update focused tests and docs for the new disclosure; no changelog entry is needed unless maintainers treat this as user-facing release-note material.
