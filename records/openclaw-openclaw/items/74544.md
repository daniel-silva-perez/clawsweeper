## Summary

Keep open. Current main still has the long `memory status --deep` embedding probe path with no non-TTY progress fallback, and this issue is paired with open PR #75106 using closing syntax, so it should remain open until that PR is reviewed and merged or replaced.

## What This Changes

The issue reports that `openclaw memory status --deep` can wait for the local embedding batch timeout with no progress output before returning an embeddings timeout while vector and FTS remain ready.

## Best Possible Solution

Review and land a maintainer-approved version of #75106, or replace it with an equivalent narrow fix, so the deep status probe has a bounded diagnostic budget and visible non-TTY progress while indexing keeps the longer batch timeout. Keep residual probe-readiness ambiguity tracked in #47235.

## Reproduction Assessment

Yes. The reporter supplied a corrected run showing completion only after the internal 600s embedding batch timeout, and current main shows `--deep` awaiting `probeEmbeddingAvailability()`, which calls `embedBatchWithRetry(["ping"])` under the local 600s batch timeout while non-TTY progress without fallback is a no-op.

## Solution Assessment

Yes for the CLI diagnostic bug. The linked PR direction is narrow because it bounds only the status probe, preserves the longer indexing timeout, adds a parseable timeout state, and documents the behavior; the provider timeout root cause can remain separate if needed.
