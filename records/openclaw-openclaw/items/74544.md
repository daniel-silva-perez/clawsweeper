## Summary

Keep open. Current main still routes `memory status --deep` through a live embedding readiness probe that uses the normal 600-second local/self-hosted batch timeout before the final status or JSON payload is rendered. The reporter’s correction narrows this from an infinite hang to a long, silent diagnostic wait plus a real local embedding timeout; adjacent open reports cover status/probe ambiguity but do not supersede this specific progress/timeout behavior.

## What This Changes

The issue asks for `openclaw memory status --deep` to avoid appearing hung during a long embedding readiness probe and to clarify or fix the underlying 600-second local embedding timeout on Windows.

## Best Possible Solution

Add a diagnostic-specific budget and durable progress/error reporting for the deep embedding readiness probe, preserving the longer inline batch defaults for real indexing. The status output should distinguish persisted vector/FTS/index health from the live provider probe and return parseable human/JSON results within a reasonable diagnostic window.

## Repair Work Prompt

In the `memory-core` plugin, make `openclaw memory status --deep` use a diagnostic-specific embedding readiness budget instead of inheriting the 600-second inline batch timeout used for indexing.
