## Summary

Keep this issue open. Current main still only partially addresses the report: the initial sessions_send agent call is internal/non-delivered, but the follow-up A2A announce flow can still post a non-ANNOUNCE_SKIP reply to the resolved target channel/thread, including Slack-like thread-bound sessions. The public sessions.send schema also still has no silent/internal-only option.

## Best Possible Solution

Keep this open and decide the product contract for inter-agent delivery: either suppress A2A announce delivery for sessions_send by default, add an explicit silent/internal-only option with safe defaults, or route the exchange through an internal bus while preserving the target session transcript. Add a regression covering Slack channel and thread-bound session keys so both the original thread-reply leak and the later top-level-channel leak cannot return.
