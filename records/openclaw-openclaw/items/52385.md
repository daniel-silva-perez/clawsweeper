## Summary

Keep this issue open. Current main has a partial fix: the initial `sessions_send` agent turn is routed as an internal, non-delivered agent call, which likely addresses the reported source-agent escalation leak. But the current agent-to-agent flow still has a separate announce step that can post any non-`ANNOUNCE_SKIP` reply to the target channel/thread, and the public `sessions.send` RPC schema still has no `silent`/internal-only option. That means the requested hard guarantee that inter-agent `sessions_send` traffic never renders in user-facing Slack surfaces is not fully implemented.

## Best Possible Solution

Keep the issue open and land a hard product/code decision for inter-agent delivery semantics: either suppress the A2A announce send for agent-to-agent `sessions_send` by default, add an explicit `silent`/`announce` option with safe defaults, or route these exchanges through a purely internal bus while preserving the target session transcript. Add a regression around Slack/thread-bound session keys so the original thread-reply and top-level-channel leak classes cannot return.

## Risks / Open Questions

- The current implementation depends on an LLM-controlled `ANNOUNCE_SKIP` response to avoid the A2A announce post for target channel sessions, so Slack surface suppression is not deterministic.
- There is no targeted regression test proving `sessions_send` into a Slack thread-bound target cannot produce a visible Slack reply through the announce path.
