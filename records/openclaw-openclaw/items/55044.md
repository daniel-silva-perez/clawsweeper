## Summary

Current main still has a plausible user-intent bug: explicit stop/no-action paths persist the same `abortedLastRun` marker that restart recovery uses to synthesize resume turns, without a separate user-abort/recovery-suppression state. The prior ClawSweeper comment keeping this open remains consistent with the current checkout at `68561a8c948b49e85ae012646de550a49d6c63a3`.

## Best Possible Solution

Keep this issue open and split explicit user-abort state from restart-interruption recovery state. `/stop`, `chat.abort`, and recognized no-action abort phrases should persist a terminal user-abort or recovery-suppression marker, while restart recovery should only resume sessions marked by stale transcript locks or legitimate restart-aborted subagent/main-session runs. Add regression coverage proving `/stop` followed by gateway startup does not call `callGateway`, while true restart-interrupted sessions still resume.
