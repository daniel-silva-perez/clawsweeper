## Summary

Keep this issue open. Current main still uses the same persisted `abortedLastRun` marker for explicit user aborts and restart-interrupted work, while main-session restart recovery still treats any running session with that marker and a resumable transcript tail as eligible for a synthetic resume turn.

## What This Changes

The issue requests that explicit user stop/no-action instructions stop active work and prevent later internal continuation or retry messages.

## Best Possible Solution

Persist explicit user abort intent separately from restart-interruption recovery, and make synthetic resume eligible only for restart-marked interrupted work. Explicit `/stop`, `chat.abort`, and recognized no-action abort phrases should leave the session terminal or recovery-suppressed even across Gateway restarts.
