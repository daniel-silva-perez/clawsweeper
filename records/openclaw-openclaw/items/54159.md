## Summary

Keep open. Current main and `v2026.4.27` still await `reader.cancel()` in the shared overflow path, so the core bug is not implemented away; the PR is also stale against current main because its Matrix edit targets a copied reader loop that now delegates through the SDK wrapper, and its agent import cleanup no longer matches the refactored files.

## What This Changes

The PR branch changes the shared bounded-response reader to fire-and-forget overflow cancellation, adds a unit test for a never-resolving cancel, applies the same pattern to an old Matrix reader copy, and removes two agent imports.

## Best Possible Solution

Land a narrow current-main change in `src/media/read-response-with-limit.ts` that does not await overflow cancellation, add the focused never-resolving-cancel unit regression in `src/media/read-response-with-limit.test.ts`, let Matrix inherit the behavior through its SDK wrapper, and keep unrelated agent cleanup out of the patch.

## Reproduction Assessment

Yes. A mocked streamed response whose first chunk exceeds `maxBytes` and whose `reader.cancel()` never resolves reproduces the shared-helper hang on current main; live Discord impact is now bounded by its 120s total attachment timeout rather than fully unbounded.

## Solution Assessment

No, not as currently shaped. The shared reader change and focused unit test are the right maintainable fix, but the stale Matrix hunk and unrelated agent import cleanup should be dropped or rebased away.
