## Summary

Keep open. Current main still lacks the under-cap optimizer-failure fallback, so this PR is not obsolete or implemented-on-main; however the branch is dirty against current main and is missing the required changelog entry for a user-facing media fix.

## What This Changes

The PR changes shared media loading to return original under-cap non-HEIC image bytes when optimization fails, preserve the original content type, rethrow image guard errors, keep optimizer causes, and add regression coverage.

## Best Possible Solution

Rebase the PR onto current main, keep the fallback at the `loadWebMedia` boundary with pixel-limit, unknown-dimension, HEIC, and over-cap paths fail-closed, add the required Unreleased changelog entry, then run focused media tests plus the changed gate before landing and closing #73424 and #73148 if maintainers accept the policy.

## Reproduction Assessment

Yes. On current main, an under-cap image with an unavailable optimizer backend still propagates the optimizer failure because `optimizeAndClampImage` has no fallback catch; the PR adds a mock-backed regression test for that path.

## Solution Assessment

Mostly yes, but the branch is not merge-ready. Solving this at `loadWebMedia`/`optimizeAndClampImage` is narrower and preserves MIME better than returning original bytes from `optimizeImageToJpeg`, but the PR must be rebased over current diagnostics and documented in the changelog.
