## Summary

Keep this PR open: current main still lacks the under-cap original-image fallback, so the PR is not obsolete, but the patch needs revision before merge because the fallback accepts original bytes after arbitrary optimizer failures instead of proving the original is a valid sendable image. Current main has already picked up the diagnostic-cause behavior, so the PR also needs a rebase that preserves that shipped-on-main change.

## What This Changes

The PR adds a shared media fallback that returns original under-cap non-HEIC image bytes when optimization fails, preserves original MIME metadata on fallback, rethrows guard and HEIC failures, surfaces optimizer causes, and adds regression tests.

## Best Possible Solution

Land a rebased consolidation only after the fallback is gated on a buffer-verified image or a narrowly recognized optimizer-backend-unavailable failure, while keeping #58226's resource guard fail-closed and current main's diagnostic-cause behavior. If accepted, it can close #73424 and #73148 and supersede the narrower fallback/diagnostic PRs.

## Reproduction Assessment

Yes. The related reports and current-main code provide a high-confidence path for the original bug: make image optimization throw, then `loadWebMedia` currently propagates the failure with no original-byte fallback. The introduced finding is also reproducible by using under-cap bytes classified as image only by extension or header and forcing the optimizer to throw.

## Solution Assessment

No. The consolidation direction is maintainable, but this implementation is too broad because it treats every non-guard optimizer error as permission to send original bytes; the safer fix is to validate the original image or narrow fallback to backend-unavailable failures before returning it.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Validate image bytes before fallback:** `src/media/web-media.ts:406-412`
  - body: The new catch path returns the original buffer whenever it is under the cap and not HEIC. Local MIME detection can classify a file as an image from its extension after sniffing fails, so non-image bytes in `bad.png` or an unsupported image with unknown dimensions can now be forwarded when `resizeToJpeg` throws. Please gate this fallback on a buffer-verified image or a known optimizer-backend-unavailable error instead of accepting every optimizer failure.
  - confidence: 0.84
