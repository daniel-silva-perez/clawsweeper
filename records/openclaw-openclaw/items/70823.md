## Summary

Keep this open. Current main still treats plugin slots as bare string plugin IDs across config types, validation, normalization, activation, write helpers, tests, and docs, so the requested owner/provenance object and overwrite invariant remain a real core plugin-contract RFC rather than a stale or ClawHub-only item.

## What This Changes

The issue requests a backwards-compatible `plugins.slots.<slot>` ownership object with `owner` and provenance fields while continuing to accept legacy bare-string slot configs.

## Best Possible Solution

Keep the RFC open for maintainer design review, then implement a versioned backwards-compatible slot ownership contract: read-side string-or-object parsing, canonical owner-object normalization, write-side provenance, explicit override/takeover policy, generated schema/help/docs updates, and focused tests across activation, install, update, uninstall, validation, defaults, and `"none"` behavior.

## Reproduction Assessment

Not applicable as a runtime bug reproduction; this is a design RFC. The current-main source checks give high confidence that object slot values are not accepted or normalized today.

## Solution Assessment

Unclear until maintainers settle the contract. The requested direction is plausible, but the safest path is to approve the schema and takeover semantics first, then land read-side compatibility before canonical writes and enforcement.
