## Summary

Keep open. Current main still models plugin slots as string selectors, not machine-readable ownership records, and this RFC affects the public config/plugin-slot contract rather than an optional plugin that can be moved to ClawHub. The proposal is credible but needs maintainer API/product judgment before implementation.

## What This Changes

Request a backwards-compatible schema extension so `plugins.slots.<slot>` can be read as an object with `owner`, `claimed_at`, and `claimed_by_version`, followed by canonical writes and overwrite protections.

## Best Possible Solution

Keep this RFC open for maintainer design review. If accepted, land a narrow read-side normalization helper first, then separately decide canonical write metadata and overwrite enforcement so existing bare-string configs and third-party plugins keep working.

## Reproduction Assessment

Not applicable as a feature/RFC rather than a runtime bug. The current-main source check is high confidence: types, schema, normalization, selection logic, tests, and docs all still use bare string slot ids.

## Solution Assessment

Unclear. The proposed direction is plausible, but the best maintainable path should be a staged, versioned config/SDK contract with read-side compatibility first and separate maintainer decisions for provenance fields and force/override policy.
