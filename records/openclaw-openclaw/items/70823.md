## Summary

Keep this RFC open. Current main still models plugin slots as bare string plugin IDs in the public config type, runtime validation, normalization, activation, install/write selection, uninstall/update paths, tests, generated schema, and docs. The proposed owner/provenance object and overwrite invariant remain unresolved core plugin-contract work, not something already implemented or safely movable to ClawHub alone.

## What This Changes

The issue requests a backwards-compatible `plugins.slots.<slot>` schema that can expose a machine-readable owner and provenance fields instead of only a bare plugin ID string.

## Best Possible Solution

Keep the RFC open for maintainer design review, then implement a versioned backwards-compatible slot ownership contract: read-side string-or-object parsing, canonical owner-object normalization, write-side provenance, explicit override/takeover policy, generated schema/help/docs updates, and focused tests across activation, install/update, uninstall, validation, defaults, and `"none"` behavior.

## Reproduction Assessment

Not applicable as a runtime bug reproduction; this is a design RFC. Current-main source inspection gives high confidence that owner-object slot values are not accepted or normalized today because the types, schema, normalization, activation checks, mutation paths, and docs all still use string slot IDs.

## Solution Assessment

Unclear until maintainers settle the contract. The requested direction is plausible, but the safest path is to approve the schema and takeover semantics first, then land read-side compatibility before canonical writes and enforcement.
