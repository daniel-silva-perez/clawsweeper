## Summary

Keep open. Current `main` still treats plugin slots as string selectors in the public config type, runtime schema, normalization, install/selection writes, validation, docs, and activation checks. The request is a core plugin/config-contract RFC with a plausible external motivator, but the owner object shape and overwrite policy need maintainer API judgment before implementation.

## What This Changes

The issue proposes evolving `plugins.slots.<slot>` from a bare plugin id string into a backward-compatible ownership record with fields such as `owner`, `claimed_at`, and `claimed_by_version`.

## Best Possible Solution

Have maintainers decide the minimal slot ownership contract first. If accepted, stage the work as a backward-compatible read model and schema/docs/tests update, then canonical string-to-object writes, and only then an explicit overwrite/force invariant once compatibility behavior is settled.

## Reproduction Assessment

Not applicable as a bug reproduction; this is an RFC. The source-level verification is high-confidence: current `main` only accepts, normalizes, writes, validates, documents, and compares string slot selectors.

## Solution Assessment

Unclear until maintainer design review. The staged rollout proposed in the issue is plausible, but the best field set, provenance semantics, and overwrite policy are public API decisions.
