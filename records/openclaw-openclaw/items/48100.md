## Summary

Keep this PR open. Current main still does not implement `after_agent_complete` or an equivalent post-agent/pre-payload hook with suppress and reinject semantics. The speedtrap use case is concrete, but the PR remains core plugin API work that needs maintainer design and security review before merge.

## Best Possible Solution

Keep this PR open for maintainer design and security review. If pursued, rebase it onto the current `hook-types.ts` layout, decide whether the right core seam is `after_agent_complete` or a streaming-aware pre-delivery/finalization hook, add explicit prompt-mutation and conversation-access gates, and cover no-hook fast path, suppress, reinject cap, streaming behavior, and transcript state with focused tests.

## Risks / Open Questions

- The proposed hook can conflict with streaming because block replies can be queued or sent during `runAgentTurnWithFallback` before a post-completion suppress/reinject decision.
- Suppress/reinject semantics may leave transcript, context-engine, or tool-result state containing a draft assistant turn that was not delivered to the user.
- The hook needs explicit `allowPromptInjection` and raw conversation access policy coverage before being exposed to non-bundled plugins.
