## Summary

Keep this PR open. Current main still does not implement `after_agent_complete` or an equivalent post-agent, pre-payload hook with suppress and reinject semantics, and the existing hooks only cover pre-agent claiming, final-answer revision, delivery dispatch, or after-the-fact observation. This is core plugin API work rather than ClawHub cleanup, and it still needs maintainer design/security review before merge.

## Best Possible Solution

Keep this PR open for maintainer design and security review. If pursued, rebase it onto the current `hook-types.ts` hook-contract layout, decide whether the right core seam is `after_agent_complete` or a streaming-aware pre-delivery/finalization hook, add explicit prompt-mutation and conversation-access gates, and cover the fast path, suppress, reinject cap, streaming, and transcript-state behavior with tests.

## Risks / Open Questions

- Partial streaming can deliver stale content before a post-completion suppress/reinject decision unless the design explicitly defers delivery.
- Suppress or reinject after the embedded agent run may leave undelivered draft assistant turns in transcript/context state; that may be acceptable for speedtrap, but it needs a maintainer-owned contract.
- The hook observes raw assistant output and can mutate prompt flow, so non-bundled plugin exposure needs explicit prompt-injection and conversation-access policy coverage.
- The PR touches core plugin API and reply-runner execution, so it needs focused tests for no-hook fast path, suppress, reinject cap, streaming behavior, and transcript semantics.
