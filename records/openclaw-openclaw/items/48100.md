## Summary

Keep this PR open. Current main still does not implement `after_agent_complete` or an equivalent post-agent/pre-delivery hook with suppress/reinject semantics. The existing `before_agent_finalize` and `before_agent_reply` hooks overlap only partially and do not cover the PR's central speedtrap-style stale-response use case.

## Best Possible Solution

Keep this PR open for maintainer design review. The best path is to rebase it against current hook contracts, decide whether the speedtrap use case belongs in an extension of `before_agent_finalize`/delivery hooks or a new post-run hook, and add focused tests for suppression, reinjection, streaming behavior, transcript state, and plugin policy gating.

## Risks / Open Questions

- The PR diff touches a plugin lifecycle API that can mutate prompt flow and observe raw response content; it needs explicit policy review for both `allowPromptInjection` and raw conversation access before merge.
- Provided review comments and author replies leave unresolved runtime design concerns around partial/block streaming, persisted assistant turns after suppression, and reinjection from a session state that already contains the first draft.
- The proposed API now overlaps with shipped `before_agent_finalize`, `before_agent_reply`, and delivery hooks, so maintainers should decide whether to extend an existing hook or accept a separate post-run hook before merging.
