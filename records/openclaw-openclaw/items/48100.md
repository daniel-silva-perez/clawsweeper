## Summary

Keep this PR open. Current main still does not implement `after_agent_complete` or an equivalent post-agent, pre-payload hook with suppress and reinject semantics. The proposal is focused core plugin API work for a concrete speedtrap/stale-response use case, and it still needs maintainer design, test, and security review rather than cleanup closure.

## Best Possible Solution

Keep this PR open for maintainer design and security review. If pursued, rebase it onto the current `src/plugins/hook-types.ts` hook-contract layout, decide whether the right seam is `after_agent_complete` or a streaming-aware pre-delivery/finalization hook, enforce prompt-mutation and conversation-access gates, and add tests for the fast path, suppress, reinject cap, streaming behavior, and transcript-state semantics.

## Risks / Open Questions

- Partial streaming may deliver user-visible content before a post-completion suppress/reinject decision, unless the design explicitly disables or defers streaming for this hook.
- Suppress/reinject after the embedded run may leave draft assistant turns in transcript or context state; the PR discussion treats this as an intentional v1 tradeoff, but it needs a maintainer-owned contract.
- The hook observes raw assistant output and can mutate prompt flow, so non-bundled plugin exposure needs explicit prompt-injection and conversation-access policy tests.
- The PR touches the core plugin API and reply-runner execution path but still needs focused tests for no-hook fast path, suppress, reinject cap, streaming behavior, and transcript semantics.
