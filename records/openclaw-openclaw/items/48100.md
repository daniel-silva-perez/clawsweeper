## Summary

Keep this PR open. Current main still does not implement `after_agent_complete` or an equivalent post-agent/pre-delivery hook with suppress and reinject semantics. The PR is a focused core plugin API proposal, but it needs maintainer design and security review before it is mergeable.

## Best Possible Solution

Keep this PR open for maintainer design and security review. The best path is to rebase it onto the current `hook-types.ts` contract layout, decide whether this should be a new post-run hook or an extension of existing finalize/delivery seams, add both prompt-mutation and raw-conversation policy gates, and cover suppress, reinject, streaming, transcript state, and no-hook fast-path behavior with focused tests.

## Risks / Open Questions

- The proposed hook can observe raw assistant response text and mutate prompt flow through reinjection, so it needs explicit `allowConversationAccess` and `allowPromptInjection` policy coverage before merge.
- Streaming remains a functional risk: current block/partial delivery paths can enqueue or send chunks during `runAgentTurnWithFallback`, before any post-completion hook decision.
- Suppress/reinject semantics may leave transcript and context-engine state containing a draft the user never saw; the PR discussion treats this as a v1 tradeoff, but maintainers should decide whether that is acceptable for a core hook contract.
