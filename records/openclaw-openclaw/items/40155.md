## Summary

Keep open. Current main still has credible Telegram/WebChat divergence paths for this report: Telegram can pass a route-level skill filter into the shared reply pipeline while WebChat `chat.send` has no matching request field, and the vLLM path still has documented/tested behavior where local OpenAI-compatible servers can return empty or text-only tool calls.

## What This Changes

The issue reports that Telegram should expose the same himalaya skill/tool behavior as WebUI chat when using a vLLM OpenAI-compatible gpt-oss-20b model.

## Best Possible Solution

Keep this open until a maintainer or repair pass captures the effective skill snapshot and provider payload for the same agent/session through both WebChat and Telegram. If Telegram filters out himalaya unexpectedly, fix Telegram filter inheritance or diagnostics with a regression test; if the tool list matches but vLLM emits empty/text-only tool calls, handle or document that provider/model path directly.

## Reproduction Assessment

No. The report gives credible steps, but this read-only review did not run Telegram plus vLLM gpt-oss-20b plus himalaya, and the missing Telegram route config/logs are necessary for a high-confidence reproduction.

## Solution Assessment

Unclear. A Telegram skill-filter fix is the best path only if the effective Telegram skill snapshot differs from WebChat; otherwise the narrower solution is in vLLM/OpenAI-compatible tool-call handling or setup guidance.
