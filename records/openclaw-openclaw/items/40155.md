## Summary

Keep this issue open. Current main still has a credible Telegram/WebChat divergence path for skill availability: Telegram can resolve and forward per-DM/group/topic skill filters, including an intentional empty list, into the shared reply pipeline, while WebChat `chat.send` has no matching request field. The vLLM docs also still document empty/text-only tool-call behavior for some local OpenAI-compatible model stacks, so the reported vLLM plus himalaya mismatch is not proven fixed.

## Best Possible Solution

Keep the issue open. Reproduce on current main with the same agent/session, vLLM gpt-oss-20b, himalaya, and Telegram route; capture Telegram DM/group/topic config, effective `replyOptions.skillFilter`, built `skillsSnapshot`, runtime tool list, provider payload/tool_choice, and structured-versus-text tool-call events for both WebChat `chat.send` and Telegram. If Telegram unintentionally filters out himalaya, fix the Telegram filter inheritance or add diagnostics and a Telegram dispatch regression test. If the effective tool set matches but vLLM emits empty or text-only tool calls, fix or document that provider/model path directly.
