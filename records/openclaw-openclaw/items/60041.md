## Summary

Keep open. Current main still matches the reported Twitch failure modes: the generated channel config schema still contains the `allOf` plus `additionalProperties: false` shape that can reject valid Twitch configs during validation, and the Twitch gateway start path still returns after monitor setup so the shared channel manager can treat a successful connect as a clean exit and schedule auto-restarts. An open unmerged related PR, #74773, explicitly targets this issue, so this issue should remain open until that PR is reviewed and the remaining lifecycle gap is handled.

## What This Changes

The issue requests fixing the Twitch post-update regression where an enabled Twitch channel never stabilizes, plus clearer current config schema examples, validation, onboarding parity, and diagnostics.

## Best Possible Solution

Land a complete Twitch-owned fix that flattens the channel config schema used by validation/generated metadata and keeps the passive Twitch monitor alive until abort, with regression tests for both config validation and the non-returning gateway lifecycle. If #74773 only solves the schema side, follow it with a narrow lifecycle patch before closing this issue.

## Reproduction Assessment

Yes. There is a high-confidence source-level reproduction: a successful Twitch monitor setup can resolve `startAccount`, and current gateway lifecycle code records that as a clean channel exit and starts auto-retry; the generated schema also visibly reproduces the validation mismatch shape.

## Solution Assessment

No, not fully yet. The open PR is a plausible narrow fix for the schema validation half, but the best maintainable solution also needs the Twitch gateway lifecycle to follow the existing passive-monitor pattern and prove it with tests.
