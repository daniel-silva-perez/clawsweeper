## Summary

Current main still does not implement Feishu send-time cross-bot broadcast. The Feishu plugin has inbound broadcast fan-out for eligible received group events, but outbound replies, message-tool sends, and direct sends still go straight through Feishu send APIs without synthesizing peer-bot inbound events; the related mention-identity reports are adjacent, not replacements for this request.

## What This Changes

The issue requests opt-in Feishu cross-bot broadcast so messages sent by one Feishu bot in a group are delivered as synthetic inbound events to other registered OpenClaw Feishu bots in that group.

## Best Possible Solution

Implement a Feishu-owned, opt-in send-time synthetic inbound fan-out after successful outbound group sends, with origin metadata, peer account/group scoping, dedupe, loop limits, schema/docs/tests, and only a generic channel/plugin seam if another channel proves the same API is needed.

## Risks / Open Questions

- A send-time synthetic bot-to-bot path can create reply loops, token amplification, and confusing authorship unless it is opt-in, scoped by account/group, deduped, and guarded by a consecutive-bot-turn limit that resets on human messages.
- Existing top-level broadcast behavior and Feishu inbound broadcast partially overlap the multi-agent use case, so a fix needs explicit semantics for observer agents and should avoid making existing broadcast sessions unexpectedly emit Feishu replies.
- Related #40768 and closed #42410 cover Feishu mention identity, while open #46776 overlaps Feishu multi-agent reliability; none is a clean replacement for this bot-sent-message visibility request.
