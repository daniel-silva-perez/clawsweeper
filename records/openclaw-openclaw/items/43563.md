## Summary

Keep open. Current main still implements Feishu broadcast only for platform-delivered inbound group events; outbound Feishu sends and final replies return after the Feishu API call without any post-send synthetic event fan-out to peer bot accounts. The request also overlaps an open Feishu lost-message report, so it needs maintainer product and channel-architecture judgment rather than cleanup closure.

## What This Changes

Request opt-in Feishu cross-bot broadcast so bot-authored group messages are synthetically delivered to other registered OpenClaw Feishu bots in the same group, with loop protection and configuration.

## Best Possible Solution

Keep this as the canonical maintainer design item for Feishu send-time cross-bot delivery. If approved, implement an opt-in Feishu-owned post-send fan-out after successful group sends, mark injected turns as synthetic bot-originated, enforce dedupe and consecutive-bot limits, update config schema/help/docs, and cover final replies plus message-tool sends in focused Feishu tests.

## Reproduction Assessment

Yes. The reporter gives a concrete multi-bot Feishu group setup, and current source provides a high-confidence code-level reproduction: outbound sends go through `sendMessageFeishu`/`sendCardFeishu` and no current path injects a peer-bot inbound event after the API call.

## Solution Assessment

Unclear. Send-time fan-out is a plausible narrow product fix for Feishu’s platform limitation, but the best maintainable solution depends on maintainer decisions about shared channel contracts, synthetic-event metadata, loop limits, and cross-channel applicability.
