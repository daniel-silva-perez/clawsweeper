## Summary

Keep open. Current main still lacks the requested Feishu send-time cross-bot broadcast: the existing Feishu broadcast path fans out an already received inbound event to multiple agents, while outbound replies, message-tool sends, cards, and media still call Feishu send APIs without synthesizing peer-bot inbound events. The related mention-identity issues are adjacent but do not replace this bot-sent-message visibility request.

## What This Changes

Request opt-in Feishu send-time synthetic cross-bot broadcast so bot messages sent to a group are delivered to other registered bots in the same group, with loop guards and Discord-style multi-bot parity.

## Best Possible Solution

Implement a Feishu-owned, opt-in send-time synthetic inbound fan-out after successful outbound group sends, with origin metadata, peer account/group scoping, dedupe, loop limits, schema/docs/tests, and only a generic channel/plugin seam if another channel proves the same API is needed.

## Risks / Open Questions

- A synthetic bot-to-bot path can create reply loops, token amplification, and confusing authorship unless it is opt-in, scoped by account/group, deduped, and guarded by a consecutive-bot-turn limit that resets on human messages.
- Existing top-level broadcast and Feishu inbound broadcast partially overlap the multi-agent use case, so a fix needs explicit semantics for observer agents and should avoid making current broadcast sessions unexpectedly emit Feishu replies.
- Related #46776 overlaps Feishu multi-agent reliability but is broader; this issue remains the clearer tracker for bot-sent-message visibility.
