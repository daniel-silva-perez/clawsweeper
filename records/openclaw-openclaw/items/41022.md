## Summary

Keep this PR open. Current main still carries the old 180-second resumed CLI watchdog cap, while the PR is a narrow constants-only implementation path for the linked watchdog issue; I did not find a concrete diff-introduced correctness or security bug that would justify closing it as obsolete or blocking it in this sweep.

## What This Changes

The PR raises the fresh CLI watchdog minimum/default cap and the resumed CLI watchdog cap in `src/agents/cli-watchdog-defaults.ts`.

## Best Possible Solution

Review and land this PR or a successor if maintainers accept the 30-minute default cap policy, ideally with focused resolver/default coverage; keep Monitor-specific idle handling and any no-cap configuration semantics as separate scoped follow-up work.

## Reproduction Assessment

Yes. A resumed CLI run with a long outer timeout currently resolves through the resume watchdog defaults with `maxMs: 180_000`, and the runner passes that value to process supervision as `noOutputTimeoutMs`; linked issue comments also report live 180-second terminations.

## Solution Assessment

Yes for the narrow cap-raise request. The constants-only change is the smallest maintainable mitigation for the hard 180-second cap, though it is not the full solution for Monitor-aware idle sessions or a configurable no-cap mode.
