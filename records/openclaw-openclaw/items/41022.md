## Summary

Keep open: current main and the latest release still have the 180-second resumed CLI watchdog cap, and the branch is a narrow constants-only mitigation with no clear correctness or security blocker. Because this is an active contributor PR tied to #40982 and related #71803, the remaining step is maintainer policy review and validation rather than closure or an automated replacement.

## What This Changes

The branch changes `src/agents/cli-watchdog-defaults.ts` to raise the fresh watchdog minimum and maximum and the resumed watchdog maximum defaults.

## Best Possible Solution

Land this PR or an equivalent successor if maintainers accept the new default cap policy, ideally with focused assertions for fresh/resume default resolution. Keep configurable no-cap behavior and Monitor expected-idle watchdog suspension as separate follow-up work under the related open reports.

## Reproduction Assessment

Yes. A code-level reproduction is high-confidence: on current main, a resumed CLI run with a long outer timeout resolves through `CLI_RESUME_WATCHDOG_DEFAULTS.maxMs: 180_000`, and the runner passes that value to the live-session or supervisor no-output watchdog.

## Solution Assessment

Yes for the narrow cap-raise mitigation. The constants-only change is the smallest maintainable way to remove the hard 180-second resumed ceiling, but it is not the full solution for disabling the watchdog or exempting expected idle Monitor waits.
