## Summary

Keep this PR open. Current main still has the old CLI no-output watchdog defaults, bundled CLI backends still consume those defaults, and the resolver/enforcement path still turns the 180000 ms resume max into a real subprocess cancellation. The PR is a narrow implementation candidate for the linked open issue, not obsolete cleanup.

## Best Possible Solution

Keep the PR open as a plausible implementation candidate. Rebase or reapply the constants change on current main, add focused tests for default fresh/resume watchdog resolution and the global-timeout clamp, then review and land if maintainers accept the longer default wait. Treat null/0 disable-cap behavior as a separate product/API decision tied to the linked open issue #40982.

## Risks / Open Questions

- The PR currently has no focused regression test proving the new fresh/resume default resolution or the existing global-timeout clamp.
- Raising the no-output cap can leave genuinely stalled silent CLI subprocesses alive longer; the existing resolver still keeps the watchdog below the global session timeout.
- The linked issue also proposes null/0 disable-cap behavior, which this PR intentionally leaves as separate maintainer scope.
