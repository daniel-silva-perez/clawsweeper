## Summary

Keep open. Current main and the latest tagged release still carry the 180s resumed-session watchdog cap, while this PR is a narrow constants-only mitigation for a still-open watchdog issue; the remaining action is maintainer review of the chosen defaults and validation scope, not cleanup closure.

## What This Changes

The PR changes `src/agents/cli-watchdog-defaults.ts` to raise the fresh watchdog minimum from 180000 to 300000 ms, the fresh maximum from 600000 to 1800000 ms, and the resumed watchdog maximum from 180000 to 1800000 ms.

## Best Possible Solution

Review and land this PR or a successor with maintainer-approved CLI watchdog defaults plus focused resolver/backend coverage, while leaving the broader configurable-disable and Monitor expected-idle semantics tracked by the related issues if they are outside this narrow cap lift.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: with current main defaults, a resumed CLI run with a long outer timeout is still bounded by `CLI_RESUME_WATCHDOG_DEFAULTS.maxMs = 180000`, and the runner passes that value to the process supervisor as `noOutputTimeoutMs`.

## Solution Assessment

Yes for the narrow mitigation. Raising these constants is the smallest code change that removes the hard 3-minute resumed-session cap, and the resolver already prevents the watchdog from exceeding the global session timeout; it is not the complete solution for configurable disable or Monitor-aware expected idle.
