## Summary

Keep this PR open for maintainer review. Current main and the latest release still carry the 180-second resumed CLI watchdog cap, while the PR's one-file constants change appears to address the linked timeout symptom without introducing an obvious correctness or security regression.

## What This Changes

The PR raises the fresh CLI watchdog minimum and maximum and the resumed CLI watchdog maximum in `src/agents/cli-watchdog-defaults.ts`.

## Best Possible Solution

Land this PR or an equivalent successor if maintainers accept the 30-minute default cap policy, with focused coverage for default fresh/resume timeout resolution. Keep Monitor-specific expected-idle handling and configurable no-cap semantics as separate follow-up work under the related open reports.

## Reproduction Assessment

Yes. On current main, a resumed CLI run with a long outer timeout resolves through `CLI_RESUME_WATCHDOG_DEFAULTS.maxMs: 180_000`, and the runner passes that value to live-session or supervisor process termination as `noOutputTimeoutMs`.

## Solution Assessment

Yes for the narrow cap-raise mitigation. The constants-only change is the smallest maintainable way to remove the hard 180-second resumed cap, but it should not be treated as the full solution for Monitor-aware idle sessions or disabling the watchdog entirely.
