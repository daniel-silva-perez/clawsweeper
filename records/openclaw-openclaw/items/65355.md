## Summary

Keep open. Current main still routes active remote configured gateways and explicit non-loopback probe targets through a 1500ms effective timeout, and the latest tagged release contains the same cap, so the reported false-negative path is still source-backed.

## What This Changes

Adjust `openclaw gateway probe --timeout` so healthy non-loopback remote `configRemote` or explicit gateway targets are not falsely failed by an internal 1500ms effective probe budget.

## Best Possible Solution

Update the gateway probe budget policy so active configured remote targets and explicit non-loopback remote URLs either honor the caller's timeout or expose a clearly documented effective per-target budget. Keep the intentionally short inactive local-loopback and SSH-tunnel behavior unless maintainers deliberately change those policies, and add regression coverage for `gateway.mode="remote"`.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: `resolveTargets` creates an active `configRemote`, `resolveProbeBudgetMs(15000, active non-loopback configRemote)` returns 1500, and `runGatewayStatusProbePass` passes that capped value into `probeGateway`.

## Solution Assessment

Yes. The narrowest maintainable fix is in `src/commands/gateway-status/helpers.ts` with call-site regression coverage, preserving unrelated probe target behavior while making the user-facing timeout truthful for active remote probes.
