## Summary

Keep open. Current main still routes bare `openclaw` to Crestodian with no config-state check, while this PR is a focused implementation candidate for the fresh-install onboarding behavior and I did not find a definite blocking correctness bug in the diff.

## What This Changes

The PR adds a fresh-install config snapshot gate to bare CLI startup, routes missing or valid-empty configs to classic onboarding, preserves Crestodian for configured or invalid configs, and updates related CLI docs plus unit and Docker-smoke coverage.

## Best Possible Solution

If maintainers accept the UX change, land this PR or an equivalent scoped route: check the config snapshot immediately before the bare-root Crestodian branch, send missing or valid-empty configs to classic onboarding, keep explicit Crestodian and invalid-config repair paths intact, and include the matching docs, smoke coverage, and changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: bare `openclaw` satisfies `shouldStartCrestodianForBareRoot`, reaches the `runCrestodian` branch, and current tests/docs/smoke coverage assert that behavior rather than fresh-install onboarding.

## Solution Assessment

Yes, conditionally on the product decision. The PR's config-snapshot gate in `runCli` is the narrowest maintainable place to distinguish fresh installs from configured or invalid-config repair paths; the safer merge path is to add the changelog entry and run the Docker/changed gates before landing.
