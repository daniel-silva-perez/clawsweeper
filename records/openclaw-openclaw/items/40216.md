## Summary

Keep open. Current main has a partial CJK-aware estimator in the active pruning path, but the PR branch targets the removed `pi-extensions` path and still has a soft-trim unit mismatch, so it should not be merged as-is and the intended behavior is not fully implemented on main.

## What This Changes

The PR branch adds CJK/non-Latin weighted string-length estimation to context-pruner sizing and adds unit tests for ASCII, CJK, mixed text, empty strings, and supplementary-plane CJK.

## Best Possible Solution

Port the useful behavior into `src/agents/pi-hooks/context-pruning`, reuse `src/utils/cjk-chars.ts` for every text-bearing estimate including serialized tool-call arguments, make soft-trim eligibility and head/tail extraction use coherent units, and add focused regressions for CJK soft trimming and tool-call arguments while preserving ASCII behavior.

## Reproduction Assessment

Yes. A focused source-level reproduction is a CJK-heavy prunable `toolResult` or assistant `toolCall` argument in the active pruner: soft-trim eligibility still uses raw `p.length`, and tool-call arguments still use raw serialized length, so CJK content remains undercounted in those paths.

## Solution Assessment

No. The PR direction is useful, but the current patch is not the best fix because it edits the obsolete `pi-extensions` path and makes soft-trim eligibility weighted while extraction still slices raw UTF-16 units.
