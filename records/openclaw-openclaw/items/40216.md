## Summary

Keep open. The contribution targets a real CJK token-estimation gap, but current main moved the shipped pruner from `pi-extensions` to `pi-hooks`, and the active path still has CJK gaps for soft-trim eligibility and serialized tool-call arguments; merging this branch as-is would not fix shipped behavior.

## What This Changes

The PR adds a local CJK/non-Latin weighted string estimator under `src/agents/pi-extensions/context-pruning`, uses it for pruner message and tool-result estimates including serialized tool-call arguments, and adds estimator tests.

## Best Possible Solution

Port the useful behavior to `src/agents/pi-hooks/context-pruning`, reuse `src/utils/cjk-chars.ts`, apply CJK-aware estimation to serialized tool-call arguments and soft-trim eligibility/extraction in one consistent unit, and keep the removed `pi-extensions` path out of the fix.

## Reproduction Assessment

Yes. A source-level reproduction exists on current main: use a CJK-heavy prunable `toolResult` whose raw length is under `softTrim.maxChars` but whose weighted estimate exceeds it, or an assistant `toolCall` with CJK arguments; the active code still uses raw length in both paths.

## Solution Assessment

No. The direction is useful, but this patch is not the best way to solve the issue because it edits obsolete files and its weighted soft-trim eligibility change is not paired with weighted head/tail extraction semantics.
