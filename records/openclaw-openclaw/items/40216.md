## Summary

Keep open. The PR captures a real CJK token-estimation problem, but the branch targets the old `pi-extensions` pruner while current `main` runs `pi-hooks`, and the active pruner still undercounts CJK in soft-trim eligibility and serialized tool-call arguments.

## What This Changes

The PR branch adds a CJK/non-Latin weighted string-length estimator to the old `pi-extensions` context pruner, wires it into message and soft-trim size estimates, and adds estimator unit tests.

## Best Possible Solution

Port the useful CJK-aware estimation behavior into `src/agents/pi-hooks/context-pruning`, use the shared `src/utils/cjk-chars.ts` helper for every text-bearing estimate including serialized tool-call arguments, keep soft-trim eligibility and extraction in coherent units, and add focused regressions for CJK tool results and tool-call arguments while preserving ASCII behavior.

## Reproduction Assessment

Yes. A source-level reproduction is a CJK-heavy prunable `toolResult` or assistant `toolCall` argument in the active pruner: current main still uses raw `p.length` for soft-trim eligibility and raw serialized length for tool-call arguments.

## Solution Assessment

No. The PR direction is useful, but the current patch is not the best solution because it edits the obsolete `pi-extensions` path and introduces a weighted soft-trim eligibility path without matching weighted extraction semantics.
