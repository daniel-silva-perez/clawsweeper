## Summary

Keep open. Current main has useful partial hardening, but it still does not implement the requested compaction circuit breaker: fallback compaction model is only a static override, repeated failures do not force-prune conversation history, channel ingestion is still tied to the active session lane/compaction wait path, and status exposes checkpoints rather than live/failing compaction health. The open PR #74470 is relevant for the fallback-model slice, but it is unmerged and does not cover the broader force-truncate, status, or transport-decoupling requests.

## What This Changes

Request a compaction circuit breaker that retries with a fallback model, force-truncates after repeated failures, avoids taking down Telegram polling, and exposes compaction health in gateway status.

## Best Possible Solution

Define and implement a core compaction failure policy that tracks per-session compaction failures, uses an approved fallback route/model for implicit compaction, applies a documented lossy recovery after the threshold, exposes live/failing compaction state through session/gateway status, and keeps Telegram/channel transport resilient without adding Telegram-specific policy to core.
