## Summary

Keep open. Current main still only has partial compaction hardening: static compaction model override, bounded retry/give-up paths, and cleanup timeouts, but not the requested fallback circuit breaker, forced lossy transcript pruning, live failing-compaction status, or transport decoupling. The open PR #74470 is relevant to the fallback slice but remains unmerged and does not cover the broader issue.

## What This Changes

Request adding a compaction circuit breaker with fallback compaction model selection, forced lossy truncation after repeated failures, non-blocking Telegram/channel ingestion during compaction, and gateway status health visibility.

## Best Possible Solution

Define a core compaction failure policy that tracks per-session compaction failures, uses an approved fallback route/model for implicit compaction, applies documented lossy recovery after a threshold, exposes live/failing compaction state through session/gateway status, and keeps Telegram/channel transport resilient without putting Telegram-specific policy in core.
