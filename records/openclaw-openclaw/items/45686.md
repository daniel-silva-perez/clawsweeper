## Summary

Current main has partial compaction hardening, but it does not implement the requested circuit breaker. Compaction can use a static configured override, timeout/overflow paths retry compaction a bounded number of times, Telegram polling error handling is narrower than in the report, and status shows checkpoint counts, but there is no automatic fallback model after repeated compaction failures, no force-truncate of oldest conversation history after N failed compactions, and no gateway status health state for in-flight or failing compaction.

## What This Changes

The issue requests a compaction circuit breaker with fallback compaction model, force-truncation after repeated failures, non-blocking Telegram transport behavior during compaction, and gateway status visibility.

## Best Possible Solution

Add a maintainer-scoped core compaction circuit breaker around embedded/context-engine compaction: track per-session failure state, retry compaction through a configured fallback model or fallback route, apply a documented lossy prune/reset policy after the failure threshold, expose compaction health through gateway/session status, and make the Telegram/channel path resilient to compaction stalls without inventing Telegram-specific core behavior.
