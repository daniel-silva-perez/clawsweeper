## Summary

Current main still only partially hardens compaction failure handling, so this issue should stay open. The code now has static compaction model selection, bounded retry/give-up behavior, timeout cleanup, and tool-result truncation, but it does not implement the requested repeated-failure fallback circuit breaker, forced lossy transcript pruning after failed compactions, live failing-compaction status, or full channel transport decoupling; the related fallback PR #74470 remains open and covers only one slice.

## What This Changes

The issue requests a compaction failure circuit breaker with fallback-model retries, forced truncation after repeated compaction failures, Telegram transport isolation, and gateway status visibility.

## Best Possible Solution

Define a core compaction failure policy that tracks per-session compaction failures, uses an approved fallback route/model for implicit compaction, applies documented lossy recovery after a threshold, exposes live/failing compaction state through session or gateway status, and keeps channel transports resilient without putting Telegram-specific policy into core.
