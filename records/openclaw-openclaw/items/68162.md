## Summary

Keep this open. The report is recent, has concrete corroborating evidence from two released versions, and current main still lacks a focused Control UI reconnect regression proving that the same backing gateway session survives reconnect/page refresh without a reset/new-session rollover.

## What This Changes

The issue requests that Control UI/webchat resume the existing gateway session after WebSocket reconnects instead of creating a fresh session and losing visible context.

## Best Possible Solution

Add a targeted reconnect/page-refresh regression and make Control UI session continuity explicit: reconnecting or remounting with the same selected session should reload history and send follow-ups against the same backing session unless the user intentionally starts a new session or the configured reset policy legitimately expires it.
