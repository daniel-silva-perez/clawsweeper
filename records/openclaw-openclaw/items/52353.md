## Summary

Keep this issue open. Current main still surfaces embedded-run timeout failures to users, but it does not emit the requested plugin-observable lifecycle timeout event: the embedded run timeout handler logs and aborts, lifecycle error events lack `errorType`, and plugin runtime event listeners only see what is emitted onto the global agent-event bus.

## Best Possible Solution

Keep this open and implement an additive embedded-run lifecycle event contract: emit one plugin-observable `lifecycle:error` event for terminal embedded run/model timeout, with optional structured fields such as `errorType: "timeout"`, timeout kind, run/session ids, provider/model/profile context, duration, and recovery/snapshot metadata when available. Add focused tests for run-deadline timeout, idle-timeout retry/final surfacing, compaction timeout/grace, and plugin `api.runtime.events.onAgentEvent()` delivery, then document the optional lifecycle error fields in the plugin runtime docs.

## Risks / Open Questions

- A fix needs to avoid duplicate terminal lifecycle events because embedded runs already emit normal start/end/error events through the subscription path.
- The timeout signal should distinguish retried idle timeouts, compaction grace, and terminal run timeouts so plugins do not retry too early or too often.
- Provider/model/profile context should stay sanitized and avoid leaking credentials or raw provider error bodies into plugin-visible payloads.
