## Summary

Keep this issue open. Current main has improved embedded-run timeout handling and user-facing timeout payloads, but the requested plugin-observable timeout agent event is still missing: the embedded run deadline handler logs and aborts without emitting `emitAgentEvent`, lifecycle error events are only derived from assistant error messages and do not carry `errorType`, and the plugin runtime’s `api.runtime.events.onAgentEvent()` subscribes directly to the global agent-event bus that would need this signal. The related timeout-configurability work does not supersede this event/API request, and Ryce’s comment reinforces the recovery/snapshot use case rather than resolving it.

## Best Possible Solution

Keep the issue open and implement an additive embedded-run lifecycle event contract: emit a single `lifecycle:error` event on terminal run/model timeout with optional structured fields such as `errorType: "timeout"`, timeout kind, run/session ids, provider/model/profile context, duration, and recovery/snapshot metadata if available. Add focused tests around run deadline timeout, LLM idle timeout retry/final surfacing, compaction timeout, and plugin `api.runtime.events.onAgentEvent()` delivery, then document the new optional lifecycle error fields in the plugin runtime docs.
