## Summary

Keep open. Current main still lacks the requested TUI preservation flag, and this PR remains the canonical implementation path for #11669, but the proposed preserve mode leaves earlier partial assistant chunks behind when final output is suppressed and can append a duplicate final snapshot.

## What This Changes

The PR adds an opt-in `--preserve-intermediate` TUI flag, threads it through CLI options into `ChatLog`, appends streamed assistant updates as separate entries, and adds focused docs and tests.

## Best Possible Solution

Land the feature after preserve mode tracks all assistant components created for a run, removes every preserved chunk when that run is explicitly suppressed, skips or reuses an unchanged final snapshot, and documents the log-budget tradeoff while keeping default TUI behavior unchanged.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Drop preserved chunks when final output is suppressed:** `src/tui/components/chat-log.ts:100-102`
  - body: With preserve mode enabled, each displayed delta appends a new assistant component while `streamingRuns` points only at the newest one. Existing final-event suppression paths call `dropAssistant` for no-message and external `(no output)` finals, so this removes only the last chunk and leaves earlier partial text visible as a phantom reply.
  - confidence: 0.9
- **[P3] Skip unchanged final snapshots:** `src/tui/components/chat-log.ts:115-119`
  - body: `finalizeAssistant` always appends another component in preserve mode, even when the final text is identical to the latest displayed delta. Normal final payloads can repeat the streamed text, which leaves duplicate trailing assistant entries and consumes the log budget sooner.
  - confidence: 0.78
