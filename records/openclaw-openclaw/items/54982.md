## Summary

Keep open. Current main still lacks the requested TUI option, so this PR is not obsolete, but the proposed `preserveIntermediate` implementation has a blocking edge-case regression: existing no-output final paths call `dropAssistant`, and the patch would only drop the latest preserved component while leaving earlier streamed chunks visible as phantom assistant output.

## What This Changes

The PR adds an opt-in `--preserve-intermediate` TUI flag that preserves assistant streaming updates as separate chat-log entries, with option wiring, docs, and focused tests.

## Best Possible Solution

Keep the feature path open, but require the preserve mode to track all preserved components for a run, drop them all when the run is explicitly suppressed, and avoid duplicating an unchanged final snapshot before this lands.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Drop all preserved chunks on suppressed output:** `src/tui/components/chat-log.ts:99-103`
  - body: With `preserveIntermediate` enabled, `updateAssistant` appends a fresh component for each displayed delta while `streamingRuns` still points only at the newest one. Existing final-event paths call `dropAssistant` for no-message or suppressed `(no output)` runs, so this would remove only the last chunk and leave earlier partial assistant text visible as a phantom reply.
  - confidence: 0.9
- **[P3] Avoid duplicating the final snapshot:** `src/tui/components/chat-log.ts:112-116`
  - body: `finalizeAssistant` appends another component in preserve mode even when the final text is identical to the latest displayed delta. `TuiStreamAssembler` suppresses unchanged deltas but still returns final text at completion, so repeated final payloads can leave duplicate trailing assistant entries and consume the log budget sooner.
  - confidence: 0.78
