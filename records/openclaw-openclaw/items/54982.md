## Summary

Keep open. Current main still lacks the requested TUI preservation option, and this PR remains the canonical implementation path for the closed feature request, but the patch currently regresses no-output suppression in preserve mode and can append duplicate final snapshots.

## What This Changes

The branch adds an opt-in `--preserve-intermediate` TUI flag, threads it into `ChatLog`, appends streamed assistant snapshots as separate log entries, and adds focused docs and tests.

## Best Possible Solution

Land the feature after preserve mode tracks all assistant components created for a run, removes every preserved fragment when that run is explicitly suppressed, avoids adding an unchanged final snapshot, and documents the log-budget tradeoff while keeping default TUI behavior unchanged.

## Reproduction Assessment

Yes. A high-confidence reproduction follows directly from the PR code path: in preserve mode, call `updateAssistant` multiple times for one run, then route a no-message or external `(no output)` final through `dropAssistant`; only the latest component is removed and earlier preserved chunks remain visible.

## Solution Assessment

No. The requested flag is a reasonable narrow solution, but the current implementation is not the best mergeable form because it preserves stale chunks after suppressed finals and duplicates unchanged final text; the safer fix is to keep per-run preserved component state and finalize/drop that state deliberately.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Existing final-event suppression paths call `dropAssistant` for no-message and external `(no output)` finals, so this removes only the last chunk and leaves earlier partial text visible as a phantom reply.
  - confidence: 0.9

  - confidence: 0.78
