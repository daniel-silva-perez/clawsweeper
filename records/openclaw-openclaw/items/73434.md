## Summary

Keep open. Current main already implements and ships the pending-tool-task drain portion, but this PR still contains meaningful unmerged work for synthetic tool-result persistence and per-tool-call timeouts. The PR should not merge as-is because its default 60-second wrapper caps every tool call, including tools like exec that already have longer documented per-tool/per-call timeout semantics.

## What This Changes

The PR enables synthetic missing tool results by default, adds an `agents.defaults.toolCallTimeoutSeconds` config knob, wraps SDK tool execution with an aborting timeout, threads that setting into embedded runs and compaction, and adds agent timeout/transcript-policy tests.

## Best Possible Solution

Keep the shipped pending-drain implementation on main, then revise this PR into a narrower runtime change that preserves existing tool-owned timeout behavior by default, makes synthetic result persistence provider-policy aware, documents the new config surface, and validates the changed agent/config lanes.

## Reproduction Assessment

Yes. Source-level reproduction is high-confidence: at PR head, unset `toolCallTimeoutSeconds` becomes a 60s wrapper around every tool, while current exec docs and implementation allow 1800s by default and longer per-call timeouts; a long `exec timeout=300` call would now fail early. I did not execute a live repro because this review is read-only.

## Solution Assessment

No. The current-main drain fix is the right narrow solution for the already-shipped Layer 3 behavior, but the PR's timeout layer should respect existing per-tool timeout contracts or be opt-in/disabled by default rather than imposing a new global 60s cap.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve tool-owned timeout semantics:** `src/agents/pi-tool-definition-adapter.ts:293-296`
  - body: With `toolCallTimeoutSeconds` unset, this defaults `timeoutMs` to 60s and then wraps every registered tool. That preempts existing tool-specific contracts: `exec` documents `tools.exec.timeoutSec` as 1800s by default and lets a call set `timeout` or `timeout: 0`, but those calls would now be aborted by the outer wrapper after 60s. Please make the wrapper preserve tool-owned/per-call timeouts by default, or scope the new timeout so long-running tools are not capped unexpectedly.
  - confidence: 0.91
