## Summary

Keep open. Current main already has the bounded pending-tool-task drain portion, but it does not implement the PR's global synthetic-tool-result default or per-tool-call timeout surface. The PR still contains meaningful runtime work, but the current patch needs maintainer follow-up because its default 60s wrapper can override existing long-running tool semantics.

## What This Changes

This PR enables synthetic tool-result repair by default for all transcript policies, adds a configurable per-tool-call timeout through the Pi tool adapter, threads that config from embedded runners, and adds config/schema/test coverage.

## Best Possible Solution

Keep this PR open as an implementation candidate, but narrow it against current main: retain the already-shipped pending-drain layer, make per-tool timeout behavior compatible with existing tool-specific timeouts, and handle global synthetic-result defaults through explicit provider/runtime compatibility review plus aligned docs and config metadata.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partly: the linked lost-tool-result reports include concrete stuck-session traces, and current main has a bounded-drain implementation for the pending-task path. For the PR-introduced timeout regression, a foreground or non-backgrounded tool expected to run longer than 60 seconds with no agents.defaults.toolCallTimeoutSeconds would exercise the conflict.

## Solution Assessment

Is this the best way to solve the issue? No as currently written: the synthetic repair and timeout layers are plausible, but applying a global 60s default at the adapter layer is broader than the existing tool contracts. A safer fix would preserve existing tool-specific timeout defaults, make the new timeout opt-in or tool-aware, and update provider/config docs before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Preserve existing long-running tool timeouts:** `src/agents/pi-tool-definition-adapter.ts:293-296`
  - body: This default applies a 60s timeout to every adapted tool when users have not configured `agents.defaults.toolCallTimeoutSeconds`. Existing tools such as `exec` already expose their own timeout contract (`tools.exec.timeoutSec` defaults to 1800s and per-call `timeout: 0` disables it), so foreground/non-backgrounded commands or other long-running tools can now be aborted by the wrapper long before their documented timeout. Preserve current behavior by disabling the adapter timeout by default or making it tool-aware.
  - confidence: 0.86
