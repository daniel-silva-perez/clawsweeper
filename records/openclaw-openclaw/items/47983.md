## Summary

Keep open. Current main still has selected-file handling and downstream non-video attachment support, but the visible Control UI/WebChat attach button is still routed through an unscoped document-wide hidden-input lookup, and the existing tests do not cover the reported visible-button-to-file-picker activation path. Related #72299 shipped the post-selection attachment pipeline; open #72612 and #74082 cover adjacent attachment safety/video work rather than explicitly fixing this picker trigger.

## What This Changes

The issue reports that the WebChat/Control UI upload button does nothing when clicked in OpenClaw 2026.3.13 on Chrome for Windows.

## Best Possible Solution

Keep the existing attachment serialization and Gateway media path intact, but make the chat composer Attach file control target its own file input through a component-scoped ref/createRef or label-backed input, add an explicit button type, and add a regression test that clicks the visible control and verifies the hidden input activation path.

## Reproduction Assessment

Unclear. The issue gives a concrete Chrome/Windows reproduction and current main still exposes the same visible-button path, but this read-only review did not run a live browser file-picker smoke.

## Solution Assessment

Yes. A narrow Control UI fix to the button-to-input activation path plus a view-level regression test is the most maintainable solution; the downstream attachment handling should not be changed for this bug.
