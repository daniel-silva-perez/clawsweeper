## Summary

Keep open. Current main still lacks a terminal chat runId dedupe mechanism for retransmitted Control UI `final` events, so this PR is not obsolete or implemented on main. However, the PR branch should not merge as-is: it bundles unrelated core/runtime/provider/Discord/ops changes with the UI fix, and review comments plus the diff show correctness and security-sensitive risks in the proposed patch.

## Best Possible Solution

Keep this PR open for maintainer follow-up, but do not merge the current branch as-is. The best path is a minimal UI-only replacement or revision that deduplicates terminal chat events by runId across own-run `final`/`aborted` and other-run/sub-agent final paths, preserves the dedupe record through same-session history reloads, clears it only on real session/agent switches, and adds focused tests in `ui/src/ui/controllers/chat.test.ts` and/or `ui/src/ui/app-gateway.node.test.ts`. Drop the unrelated Google Vertex, Discord, sandbox, AGENTS, and gaxios changes from this PR.

## Risks / Open Questions

- Current main still appears vulnerable to duplicate assistant messages if the same Control UI final event is retransmitted after `chatRunId` has been cleared.
- Merging the PR as-is would mix a narrow UI bug fix with unrelated AGENTS, Google Vertex auth, sandbox policy, Discord monitor/config, and gaxios compatibility changes.
- The PR’s `globalThis.window` shim is a process-wide Node runtime behavior change and is security/supply-chain sensitive because it can route dependencies into browser-specific code paths.
- The PR’s own dedupe implementation appears incomplete for history reloads and other-run/sub-agent final events, so it needs revision and focused regression tests before merge.
