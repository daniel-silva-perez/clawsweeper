## Summary

Keep open. Current main still has the cron/main-session prompt-contamination behavior this PR targets, so the work is not obsolete; however, the PR should not land as-is because it drops the no-relay cron prompt guard and blocks operational edits to legacy rich main-session jobs.

## What This Changes

The PR adds a main-session cron systemEvent safety layer, changes cron heartbeat prompts to a minimal `SYSTEM_WAKE` token shape, and updates cron tool/system prompt guidance plus focused cron/heartbeat tests to prefer isolated `agentTurn` reminders.

## Best Possible Solution

Keep this PR open as an implementation candidate, but rebase or replace it with a narrow patch that tokenizes main-session cron wakes, preserves strict internal-only behavior when user delivery is disabled, allows non-payload edits to legacy unsafe main-session jobs, updates docs/tool guidance, and preserves isolated `agentTurn` reminder flows.

## Reproduction Assessment

Yes. Static source and tests give a high-confidence path: current main’s `buildCronEventPrompt` and heartbeat runner still produce relay-style cron prompts, and the PR head can be checked by calling `buildCronEventPrompt(["Cron: rotate logs"], { deliverToUser: false })`, which no longer returns strict internal-only wording.

## Solution Assessment

No. The direction is right, but the proposed patch is not the best landing shape until it restores no-relay semantics and scopes validation so legacy rich main-session jobs can still be disabled or repaired while execution degrades safely.

## Review Findings

Full review comments:

- **[P1] Restore no-relay cron wake behavior:** `src/infra/heartbeat-events-filter.ts:23`
  - body: `buildCronEventPrompt` now ignores the `deliverToUser` option and always tells the heartbeat to follow up if context requires it. For no-target or alerts-disabled cron wakes, that leaves a main-session turn that can still produce transcript residue instead of the prior strict internal/`HEARTBEAT_OK` path.
  - confidence: 0.88

  - body: `applyJobPatch` calls `assertSupportedJobSpec` after every patch, so existing main-session jobs with old reminder prose now throw even for unrelated edits such as `enabled:false`. Users need to be able to disable or repair those legacy jobs while execution falls back to a safe token.
  - confidence: 0.84
