## Summary

Keep this open. Current main still has a stop-reason-only abort gap: the runner can preserve `stopReason: "aborted"`, and persisted session state maps that to killed, but lifecycle emission, `agent.wait`, and the subagent registry still treat a lifecycle `end` without `aborted: true` as successful. The issue is also paired with an open linked fix PR using closing syntax, so the maintainer path is to review, land, or replace that PR rather than close early.

## What This Changes

The report asks OpenClaw to classify empty-output aborted subagent terminal turns as parent-visible failed or killed outcomes so parent wait/announce flows do not silently black-hole.

## Best Possible Solution

Land or replace a narrow core fix that normalizes a final assistant `stopReason: "aborted"` into a failed/killed parent-visible outcome across lifecycle emission, `agent.wait`, subagent registry completion, announce formatting, and focused regression tests, without adding a new public status union or provider retry policy unless maintainers explicitly choose that scope.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to produce a terminal lifecycle `end` carrying `stopReason: "aborted"` but no `aborted: true`; current main persists that shape as killed in one helper, while `agent.wait` and the subagent registry still classify it as successful.

## Solution Assessment

Yes, the narrow lifecycle/classification fix described by the linked PR is the maintainable direction. The issue should stay open until that PR or an equivalent reviewed target-repo patch lands and covers both producer and consumer paths.
