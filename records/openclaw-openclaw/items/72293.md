## Summary

Keep this issue open. Current main still only partially handles terminal `stopReason: "aborted"`: persisted session state maps it to killed, but `agent.wait` and the subagent registry still classify a lifecycle `end` carrying only that stop reason as successful, so the reported parent-visible black-hole remains plausible. The issue is also paired with an open, unmerged fork PR that uses GitHub closing syntax for this issue, so the maintainer path is to review, land, or replace that PR rather than close early.

## What This Changes

The issue requests that empty-content subagent runs ending with `stopReason: "aborted"` become parent-visible failures instead of successful silent completions.

## Best Possible Solution

Land or replace a narrow core fix that consistently treats a final assistant `stopReason: "aborted"` as a terminal failure/killed outcome across lifecycle emission, `agent.wait`, direct subagent registry completion, persisted session state, announce formatting, and regression tests for empty-content aborted assistant turns.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to emit a lifecycle `end` event with `stopReason: "aborted"` and no `aborted: true`; current main persists that as killed in the session helper but `agent.wait` and the subagent registry still classify it as `ok`.

## Solution Assessment

No, not yet. The current main behavior is only a partial solution; the narrow maintainable fix should reuse existing public status unions where possible and classify stopReason-only aborts as a parent-visible failure/killed outcome with focused regression coverage.
