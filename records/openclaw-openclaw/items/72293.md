## Summary

Keep this open. Current main still preserves a final assistant `stopReason: "aborted"`, but the parent-visible lifecycle consumers only treat lifecycle `error` or `aborted: true` as unsuccessful, so a stop-reason-only abort can still be completed as `ok`; the issue is also paired with an open linked fix PR using closing syntax, so it should not be closed before that PR or an equivalent fix lands.

## What This Changes

The issue requests that aborted, empty-output subagent terminal turns become parent-visible aborted or failed outcomes instead of being treated as successful `done` runs with no useful announcement.

## Best Possible Solution

Land or replace a narrow core fix that normalizes final assistant `stopReason: "aborted"` into a failed or timed-out parent-visible outcome across lifecycle emission, `agent.wait`, subagent registry completion, announce status, and focused regression tests, without adding provider retry policy or a new public status union unless maintainers explicitly choose that scope.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to emit lifecycle `phase: "end"` with `stopReason: "aborted"` and no `aborted: true`; current main persists that shape as killed in session state but still lets `agent.wait` and the subagent registry classify it as successful.

## Solution Assessment

Yes, the narrow classification fix is the best direction. The open linked PR’s scope matches the maintainable boundary, but this issue should stay open until that PR or an equivalent reviewed target-repo patch lands.
