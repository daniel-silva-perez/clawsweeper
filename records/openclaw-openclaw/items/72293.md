## Summary

Keep this issue open. Current main still has the same stop-reason-only abort gap: persisted session state understands `stopReason: "aborted"`, but lifecycle emission, `agent.wait`, and the subagent registry can still treat a terminal lifecycle `end` with only that stop reason as successful, so the reported parent-visible black-hole remains plausible. The issue is also paired with an open unmerged fork PR that uses closing syntax, so the maintainer path is to review, land, or replace that PR rather than close early.

## What This Changes

The issue reports that aborted subagent runs with empty assistant content are marked as successful/done and fail to deliver a parent-visible completion announcement.

## Best Possible Solution

Land or replace a narrow core fix that consistently classifies a final assistant `stopReason: "aborted"` as a parent-visible failed/killed terminal outcome across lifecycle emission, `agent.wait`, subagent registry completion, announce formatting, and focused regression tests, without adding a new public status union or provider retry policy unless maintainers explicitly choose that scope.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to emit or produce a lifecycle `end` carrying `stopReason: "aborted"` but no `aborted: true`; current main persists that shape as killed in one helper but `agent.wait` and the subagent registry still classify it as successful.

## Solution Assessment

No. Current main is only a partial solution; the linked PR direction appears aligned, but the maintainable fix still needs to be reviewed or replaced and merged in the target repository.
