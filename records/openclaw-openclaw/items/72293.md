## Summary

Keep this issue open. Current main has partial progress: lifecycle events can now carry `stopReason: "aborted"`, and session lifecycle state can map that to `killed` with `abortedLastRun: true`; however `agent.wait` and the subagent registry still classify a plain lifecycle `end` with only `stopReason: "aborted"` as successful completion, so the parent-visible failure path remains incomplete. The linked fork PR is still open and unmerged, so the issue should stay open while that fix is reviewed or replaced.

## What This Changes

The issue reports aborted subagent assistant turns with empty output being classified as successful completion instead of a parent-visible failure.

## Best Possible Solution

Land a narrow core fix that treats final assistant `stopReason: "aborted"` as a terminal failure/killed outcome across lifecycle production, `agent.wait`, subagent registry completion, persisted session state, and announce formatting, with regression tests for an empty-content aborted assistant turn producing parent-visible failure/no-output completion.

## Reproduction Assessment

Yes. A source-level reproduction is clear: emit a lifecycle `end` event with `stopReason: "aborted"` and no `aborted: true`; current `agent.wait` returns `ok` and the direct subagent registry listener completes with `outcome.status: "ok"`, even though session lifecycle state would persist it as killed.

## Solution Assessment

No. Current main is only a partial solution because it can propagate and persist aborted stop reasons, but the parent-facing wait and subagent completion paths still need to classify that same terminal state as failed/killed instead of successful.
