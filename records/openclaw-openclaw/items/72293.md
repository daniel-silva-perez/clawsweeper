## Summary

Keep this open. Current main still has the stop-reason-only abort gap: persisted session state maps `stopReason: "aborted"` to killed, but `agent.wait` and direct subagent lifecycle completion still treat a lifecycle `end` with only that stop reason as successful. The item is also paired with an open, unmerged fork PR that uses closing syntax, so the issue should remain open until that PR is reviewed, landed, or replaced.

## What This Changes

The issue reports that aborted subagent assistant turns with empty content are classified as successful completion, so parent-visible announce/wait behavior can hang instead of surfacing a failure.

## Best Possible Solution

Land or replace a narrow core fix that treats final assistant `stopReason: "aborted"` as a parent-visible terminal failure across lifecycle emission, `agent.wait`, subagent registry completion, announce formatting, and regression tests, while preserving existing retry grace for transient `aborted: true` lifecycle events.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to emit a lifecycle `end` event with `stopReason: "aborted"` and no `aborted: true`; current main persists that status as killed in the session helper but still lets `agent.wait` and the subagent registry classify it as successful.

## Solution Assessment

No. Current main is only a partial solution; the maintainable fix should centralize or consistently reuse terminal abort classification and add focused tests for empty-content aborted assistant turns through lifecycle, wait, and subagent announce completion.
