## Summary

Keep open. Current main still unconditionally resolves the runtime plugin registry from the inbound dispatch path, and the paired #74118 implementation PR remains open and unmerged, so the issue should stay open until that fix or an equivalent one lands.

## What This Changes

The issue asks OpenClaw to avoid per-message plugin registry rebuilds by making `ensureRuntimePluginsLoaded` a no-op when a valid active runtime plugin registry already exists.

## Best Possible Solution

Keep this issue open until #74118 or an equivalent regression-covered fix lands. The desired end state is that inbound dispatch reuses an already-active runtime plugin registry without re-running plugin registration, while preserving explicit registry invalidation on config or gateway reload changes.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: inbound dispatch calls `ensureRuntimePluginsLoaded`, the helper always calls `resolveRuntimePluginRegistry`, and the resolver can fall through to `loadOpenClawPlugins` when the active cache key is incompatible. I did not run live timing because this review is read-only.

## Solution Assessment

Unclear. The #74118 fast path is a plausible narrow fix for this helper, but it should be reviewed against registry invalidation behavior; if maintainers want one answer for multiple strict-cache-key hot paths, a loader-level compatibility change may be the better long-term fix.
