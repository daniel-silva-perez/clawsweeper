## Summary

Keep open. Current main still has the reported hot-path shape, and #74118 is an open, unmerged fix PR paired with this issue, so closing now would skip the normal review/merge path.

## What This Changes

The issue reports that `ensureRuntimePluginsLoaded` rebuilds the runtime plugin registry on every inbound dispatch because dispatch-time load options miss the gateway startup registry cache key.

## Best Possible Solution

Keep this issue open until #74118 or an equivalent fix lands, with regression coverage proving that an already-active runtime plugin registry makes inbound dispatch a no-op for plugin loading.

## Reproduction Assessment

Yes. The static reproduction path is visible on current main: inbound dispatch calls `ensureRuntimePluginsLoaded`, the helper always calls the runtime resolver with explicit options, and the resolver reloads when the active cache key is not compatible.

## Solution Assessment

Unclear until PR review. The proposed active-registry fast path is a narrow, plausible fix for this call site, but maintainers should decide whether they prefer that helper-level change or a broader loader cache-compatibility fix.
