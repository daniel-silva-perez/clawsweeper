## Summary

Keep open. Current main and the latest release tag still have the reported hot-path shape, and the paired implementation PR #74118 is open and unmerged, so the correct cleanup path is to review or land that PR rather than close this issue early.

## What This Changes

Request to avoid per-inbound-dispatch plugin-registry rebuilds by making ensureRuntimePluginsLoaded a no-op when the active runtime plugin registry is already loaded.

## Best Possible Solution

Land a reviewed fix that makes inbound dispatch reuse the already-active runtime plugin registry, with focused regression coverage proving no rebuild when a valid active registry exists; if maintainers prefer a broader design, solve the cache-key compatibility gap in the loader and keep this issue as the user-facing tracker until that lands.

## Reproduction Assessment

Yes. The high-confidence reproduction is the current static path: inbound dispatch calls ensureRuntimePluginsLoaded, that helper always resolves with reduced options, and the resolver falls through to loadOpenClawPlugins when the active cache key differs from the gateway boot key. I did not run a live hosted gateway because this review is read-only.

## Solution Assessment

Unclear until #74118 is reviewed. The proposed active-registry fast path is the narrowest call-site fix for this issue, while a loader-level subset-compatibility change may be preferable if maintainers want to solve the same cache-key mismatch for multiple hot paths.
