## Summary

Keep this open. Current main improves the default case by steering active runs, and Telegram has a narrow abort supersession fence, but the requested general same-session pre-send queue policy is still missing: collect/followup messages are queued for a later turn and completed final payloads are still preserved for delivery.

## What This Changes

The issue requests an opt-in pre-send same-session queue check that suppresses or defers a completed reply when newer inbound messages are already pending.

## Best Possible Solution

Add a tested, opt-in core queue/delivery policy that checks same-session pending followups immediately before eligible visible final delivery, suppresses or defers only stale final payloads, and carries enough suppressed reply/tool context into the next collected turn. The design should explicitly define streaming preview cleanup, side-effect disclosure, routed delivery, plugin hook order, and whether content-aware classification is a later layer.
