## Summary

Keep open. Current main still handles newer same-session messages by enqueueing them for a later collect/followup turn while returning the active run's completed final payload unchanged, so the requested pre-send stale-final suppression is not implemented. The only concrete suppression I found is Telegram's narrow authorized-abort fence, and the discussion raises policy questions around side effects, streaming previews, routed replies, and async branch supersession that need maintainer design rather than stale cleanup.

## What This Changes

Request to suppress or defer an active run's completed final reply when newer same-session inbound messages are already queued, so followup/collect handling answers the updated request without first sending stale text.

## Best Possible Solution

Add a tested, opt-in core queue/delivery policy that checks same-session pending followups immediately before eligible visible final delivery, suppresses or defers only stale final payloads, and carries enough suppressed reply/tool context into the next collected turn. The design should explicitly define streaming preview cleanup, side-effect disclosure, routed delivery, plugin hook order, and whether content-aware classification is a later layer.
