## Summary

Keep this open. Current main still enqueues newer same-session messages for a later followup/collect turn while preserving the active run's completed final payload for delivery, so the requested delivery-time stale-reply suppression is not implemented. The discussion also raises policy questions around side effects, streaming previews, routed delivery, plugin hooks, and async branch supersession that need maintainer design rather than stale cleanup.

## What This Changes

The issue requests a pre-send queue check that suppresses or defers a completed reply when newer same-session inbound messages are already pending.

## Best Possible Solution

Add a tested, opt-in core queue/delivery policy that checks same-session pending followups immediately before eligible visible final delivery, suppresses or defers only stale final payloads, and carries enough suppressed reply/tool context into the next collected turn. The design should explicitly define streaming preview cleanup, side-effect disclosure, routed delivery, plugin hook order, and whether content-aware classification is a later layer.
