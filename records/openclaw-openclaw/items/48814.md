## Summary

Keep this issue open. Current main still implements queued follow-up/collect behavior after the active run rather than a general delivery-time same-session pending-queue suppression policy; the only stale-final suppression I found is a narrower Telegram authorized-abort fence, and the discussion raises product/API questions that need maintainer design.

## What This Changes

Request an opt-in pre-send same-session queue check that suppresses or defers stale final replies when newer inbound messages are pending.

## Best Possible Solution

Add a tested, opt-in core queue/delivery policy that checks same-session pending follow-ups immediately before eligible visible final delivery, suppresses or defers only stale final payloads, and carries suppressed reply plus tool-result context into the next collected turn. The design should explicitly cover streaming preview cleanup, side-effect disclosure, routed delivery, plugin hook order, and whether content-aware classification is a later layer.

## Reproduction Assessment

Yes. A code-level reproduction is clear: with `collect` or `followup`, a newer same-session inbound message during an active run takes the `enqueue-followup` path, while `finalizeWithFollowup` preserves the active run's completed final payload for delivery.

## Solution Assessment

No, not as a one-line blanket queue-size check. The requested direction is valid, but the maintainable fix should be an opt-in queue/delivery policy with explicit semantics for side effects, streaming previews, routing, and hook ordering.
