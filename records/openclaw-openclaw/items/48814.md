## Summary

Keep this open. Current main still queues active same-session messages for a later follow-up turn and returns the completed final payload for normal delivery; I found Telegram-specific abort-fence suppression for explicit abort commands, but not the requested general pre-send pending-followup check for collect/followup mode.

## What This Changes

The issue requests an opt-in pre-send queue check that suppresses or defers a completed reply when newer same-session inbound messages are already pending.

## Best Possible Solution

Add an opt-in core queue/delivery policy that checks same-session pending followups immediately before visible final delivery, suppresses or defers only eligible final payloads, and injects enough suppressed reply/tool context into the next collected turn. The design should explicitly cover preview/block streaming, tool side effects, routed follow-up delivery, and whether content-aware classification is a later feature.

## Risks / Open Questions

- A final-reply drop could hide tool calls or external side effects that already occurred during the superseded turn unless suppressed reply and tool context are preserved for the next turn.
- Block streaming, preview edits, tool-progress updates, or message_sending hooks may already have emitted visible content before a final-only pre-send check runs.
- Changing default delivery behavior across all channels could surprise existing users, so the policy likely needs an explicit opt-in and focused channel/streaming tests.
