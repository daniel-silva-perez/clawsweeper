## Summary

Keep this issue open. Current main still has collect/followup behavior that queues newer inbound messages for a follow-up turn, while final reply finalization returns the completed payload and only schedules the queue drain; I did not find a same-session pre-send queue-depth check that suppresses a stale final reply before channel delivery. The issue discussion is concrete and includes useful design constraints around tool side effects, smarter routing, and late async/subagent results.

## Best Possible Solution

Keep this open for maintainer design. The likely path is an opt-in core queue/delivery policy that checks same-session pending followups before visible final delivery, suppresses or defers only the appropriate final payloads, and preserves enough suppressed reply/tool context for the next collected turn. The design should explicitly cover block streaming, tool side effects, routed follow-up delivery, and whether smarter content-aware classification belongs in a later follow-up.
