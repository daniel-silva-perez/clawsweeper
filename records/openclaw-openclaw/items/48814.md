## Summary

Keep this issue open. Current main still treats `collect`/`followup` as queued work that drains after the active run, and the final delivery paths do not check for pending same-session inbound messages before sending the completed reply. There is a narrower Telegram abort-race suppression, but it is tied to explicit abort/pre-dispatch behavior, not the general pre-send queue check requested here.

## Best Possible Solution

Keep this issue open and design a core, channel-agnostic supersession policy at the reply-delivery boundary. The likely implementation should check same-session pending followup queue state before final outbound delivery, decide whether to suppress/defer the current payload, preserve useful suppressed response and tool-result context for the next turn, and add regression tests for collect mode, routed channel delivery, block streaming, and explicit abort/correction cases.

## Risks / Open Questions

- A naive queue-depth suppression could hide completed tool side effects; the next turn should receive suppressed reply/tool-result context or explicit branch-state metadata.
- Block streaming and direct `routeReply` delivery can send outside the final dispatcher path, so a complete fix needs coverage across final replies, block replies, routed channel delivery, and followup-queue delivery.
