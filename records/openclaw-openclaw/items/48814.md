## Summary

Keep this open. Current main still preserves the active run's completed final payload while newer same-session messages are queued for a later followup/collect turn; the newer Telegram stale-final guard is limited to authorized abort supersession and does not implement the requested general pre-send pending-queue policy.

## What This Changes

Request to add a pre-send same-session queue check that suppresses or defers stale final replies when newer inbound messages are already pending.

## Best Possible Solution

Add a tested, opt-in core queue/delivery policy that checks same-session pending followups immediately before eligible visible final delivery, suppresses or defers only stale final payloads, and carries enough suppressed reply/tool context into the next collected turn. The design should explicitly define streaming preview cleanup, side-effect disclosure, routed delivery, plugin hook order, and whether content-aware classification is a later layer.

## Risks / Open Questions

- A blanket suppression rule could incorrectly drop replies when pending messages are unrelated, cross-routed, or belong to a different channel/thread batch.
- Suppressing only final text without carrying tool calls, tool results, and side-effect context could hide actions that already happened during the superseded turn.
- The policy must define interaction with streaming previews, block replies, message_sending and reply_dispatch hooks, routed delivery, and Telegram abort-supersession cleanup before implementation.
