## Summary

Keep this open. Current main still implements collect/followup as later-turn queueing while preserving the active run's completed final payload for delivery; the only stale-final suppression I found is Telegram's authorized abort fence, not a generic pending-message pre-send policy. The discussion also raises side-effect, content-routing, and async-branch concerns that need maintainer policy design rather than stale cleanup.

## What This Changes

The issue requests a pre-send same-session queue check that suppresses or defers a completed reply when newer inbound messages are already pending.

## Best Possible Solution

Add a tested, opt-in core queue/delivery policy that checks same-session pending followups immediately before eligible visible final delivery, suppresses or defers only stale final payloads, and carries enough suppressed reply/tool context into the next collected turn. The design should explicitly define streaming preview cleanup, side-effect disclosure, routed delivery, plugin hook order, and whether content-aware classification is a later layer.
