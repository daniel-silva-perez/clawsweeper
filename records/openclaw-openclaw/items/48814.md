## Summary

Keep this issue open. Current main still has the reported timing gap: active same-session messages in collect/followup mode are queued for a later follow-up turn, while finalization schedules the queue drain and returns the completed final payload for normal delivery. I did not find a built-in pre-send same-session queue-depth check that suppresses or defers stale final replies before channel delivery.

## What This Changes

Add a pre-send same-session queue check that suppresses or defers a completed reply when newer inbound messages are pending.

## Best Possible Solution

Keep this open for maintainer design. The best path is an opt-in core queue/delivery policy that checks same-session pending followups immediately before visible final delivery, suppresses or defers only appropriate final payloads, and preserves enough suppressed reply/tool context for the next collected turn. The design should explicitly cover block streaming, preview edits, tool side effects, routed follow-up delivery, and whether content-aware classification belongs in a later follow-up.
