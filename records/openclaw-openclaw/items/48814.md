## Summary

Keep this issue open. Current main still queues newer same-session messages for a later collect/followup turn while preserving the active run's completed final payload for delivery, and the narrower Telegram abort fence only covers authorized abort supersession. The issue discussion raises valid product/API questions around side effects, streaming previews, routed delivery, and async branch supersession that need maintainer design before an automated repair PR.

## What This Changes

Request an opt-in pre-send same-session queue check that suppresses or defers stale final replies when newer inbound messages are pending.

## Best Possible Solution

Add a tested, opt-in core queue/delivery policy that checks same-session pending followups immediately before eligible visible final delivery, suppresses or defers only stale final payloads, and carries enough suppressed reply/tool context into the next collected turn. The design should explicitly define streaming preview cleanup, side-effect disclosure, routed delivery behavior, plugin hook order, and whether content-aware classification is a later layer.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: with `collect` or `followup`, a newer same-session inbound message during an active run takes the `enqueue-followup` path, while the active run still returns `finalizeWithFollowup(finalPayloads)`, preserving the completed final payload for delivery.

## Solution Assessment

No, not as a one-line blanket queue-size check. The requested direction is valid, but the maintainable solution should be an opt-in queue/delivery policy with explicit semantics for side effects, streaming previews, routed delivery, and hook ordering.
