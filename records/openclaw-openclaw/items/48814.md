## Summary

Keep this open. Current main still queues newer same-session messages for a later collect/followup turn while preserving the active run's completed final payload for delivery, and the Telegram abort fence only covers authorized abort supersession. The discussion adds concrete queue, side-effect, and async-branch concerns, so this is still a valid product/runtime design gap rather than stale cleanup.

## What This Changes

The issue requests an opt-in pre-send queue check that suppresses or defers a completed reply when newer same-session inbound messages are pending.

## Best Possible Solution

Add a tested, opt-in core queue/delivery policy that checks same-session pending follow-ups immediately before eligible visible final delivery, suppresses or defers only the stale final payload, and carries enough suppressed reply/tool context into the next collected turn. The design should explicitly define behavior for streaming previews, side-effectful tools, routed follow-up delivery, and whether content-aware classification is a later layer.

## Risks / Open Questions

- A final-only pre-send check may still miss visible preview edits, block streaming chunks, progress updates, or plugin hooks that already emitted content before final delivery.
- Suppressing a reply after tool calls have executed could hide real side effects unless the next turn receives preserved reply, tool-call, and tool-result context.
- Changing collect/followup delivery defaults across channels could surprise existing users, so the policy likely needs explicit opt-in semantics and channel/streaming coverage.
