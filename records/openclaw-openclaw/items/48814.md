## Summary

Keep this open. Current main still implements collect/followup as a later queued turn and still returns the completed active run's final payload for normal delivery. The issue discussion correctly calls out a delivery-time policy gap, and the newer Telegram abort-fence work only covers explicit authorized abort supersession, not arbitrary newer pending same-session messages.

## What This Changes

The issue requests an opt-in pre-send queue check that suppresses or defers a completed reply when newer same-session inbound messages are pending, so the next turn can respond to the superseding input instead of emitting a stale answer first.

## Best Possible Solution

Add an opt-in core queue/delivery policy that checks same-session pending follow-ups immediately before visible final delivery, suppresses only eligible final payloads, and injects enough suppressed reply/tool context into the next collected turn. The design should explicitly cover preview/block streaming, side-effectful tools, routed follow-up delivery, and whether content-aware classification is a later feature.

## Risks / Open Questions

- Suppressing a final reply after tool calls have already executed could hide real side effects unless the next turn receives preserved reply, tool-call, and tool-result context.
- Preview edits, block streaming, progress updates, or message_sending plugin hooks may already have emitted visible content before a final-only pre-send check runs.
- Changing default collect/follow-up delivery across all channels could surprise existing users, so the policy likely needs an explicit opt-in and channel/streaming coverage.
