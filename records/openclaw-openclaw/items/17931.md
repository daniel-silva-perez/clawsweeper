## Summary

Keep open. Current main still does not generate read-only skill directory mounts for sandbox containers: non-rw sandboxes still copy eligible skills into the sandbox workspace, rw sandboxes still expose workspace skills through the writable workspace mount, and the shipped #22669 work only covers manually configured custom bind precedence rather than automatic per-skill read-only overlays.

## Best Possible Solution

Keep this issue open for maintainer/secops design. The likely fix should derive eligible skill source directories from the skills loader, append Docker read-only skill overlays after broader workspace mounts, avoid writable duplicate skill copies for ro/none when source mounts are available, model the mounts in config hashing and fs bridges, define SSH/OpenShell behavior, and update sandbox/skills docs plus tests across all workspaceAccess modes.
