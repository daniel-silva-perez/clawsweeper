## Summary

Current main still has the reported gap. Non-rw sandbox sessions copy eligible skills into the writable sandbox workspace, rw sessions expose workspace skills through the writable workspace mount, and Docker container creation has no automatic per-skill read-only overlay mounts. The related #22669 fix only makes explicitly configured overlapping binds override workspace mounts, so this issue still needs maintainer/product direction.

## Best Possible Solution

Keep this issue open until OpenClaw either implements automatic read-only handling for eligible skill directories in sandboxed sessions or explicitly rejects that security/product direction. A strong fix should derive eligible skill source directories from the existing skill loader, overlay read-only skill binds after broader workspace mounts for Docker `rw`, avoid writable copied skill state for `ro`/`none` when source mounts are available, define non-Docker backend behavior, and update fs-bridge/config-hash/docs/tests across all `workspaceAccess` modes.

## Risks / Open Questions

- A correct fix needs deterministic skill source derivation across workspace, project `.agents`, personal, managed, bundled, extra, and plugin skill sources without weakening path confinement.
- Docker overlapping mounts are only one backend path; maintainers need to define fallback or equivalent behavior for non-Docker sandbox backends.
- Container config hashing/recreate behavior must include any generated mount set so stale containers do not keep the old writable layout.
