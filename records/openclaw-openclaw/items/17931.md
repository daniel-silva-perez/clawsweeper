## Summary

Current main still does not implement automatic read-only skill directory mounts for sandbox containers. Non-rw sandbox sessions still copy eligible skills into the writable sandbox workspace, rw sessions still expose workspace skills through the writable workspace mount, and the #22669 bind-order fix only enables a manual overlapping-bind workaround.

## Best Possible Solution

Keep this issue open until OpenClaw either implements automatic read-only handling for eligible skill directories in sandboxed sessions or explicitly rejects that security/product direction. A strong implementation would derive deterministic eligible skill source directories from the existing skill loader, add read-only overlays after broader workspace mounts for Docker rw mode, avoid writable copied skill state for ro/none when source mounts are available, define fallback behavior for non-Docker backends, and update fs-bridge/config-hash/docs/tests across all workspaceAccess modes.
