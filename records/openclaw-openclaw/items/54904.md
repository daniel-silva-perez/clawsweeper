## Summary

Current main still lacks the Feishu webhook path check, and the proposed branch appears focused and correct after the earlier parser and normalization review comments were addressed. This should stay open for normal maintainer validation and landing rather than be closed or replaced.

## What This Changes

The PR normalizes and enforces Feishu webhook request paths before body parsing, signature verification, and dispatch, with transport tests for wrong-path rejection, query-string acceptance, normalized configured paths, malformed request targets, and network-path-style bypass attempts.

## Best Possible Solution

Land this PR or an equivalent Feishu-owned change that enforces the normalized configured webhook path before body parsing, signature verification, and event dispatch, while preserving the bypass and malformed-target regression coverage.

## Reproduction Assessment

Yes. On current main, source inspection gives a high-confidence reproduction path: configure Feishu webhook mode with a custom webhookPath, send a correctly signed POST to a different path on the same listener, and the handler can still reach eventDispatcher.invoke because no req.url/path comparison exists.

## Solution Assessment

Yes. A Feishu-local guard using the existing normalizeWebhookPath helper is the narrowest maintainable fix; it avoids adding core-specific channel policy and keeps the check before expensive or security-sensitive request processing.
