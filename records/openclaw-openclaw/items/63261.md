## Summary

Keep open. Current main f2405c830bd925bd24f91b6bac5d417c6d36e5ba still has the reported Discord runtime owner-auth gap: the production Discord config adapter reads only legacy dm.allowFrom, while docs, config normalization, account merging, and the Discord allowlist path treat top-level channels.discord.allowFrom as canonical. That means resolveCommandAuthorization can still build ownerList=[] and senderIsOwner=false for the reported config, hiding owner-only tools such as cron and gateway.

## Best Possible Solution

Land a focused Discord plugin fix that resolves owner allowFrom from account.config.allowFrom ?? account.config.dm?.allowFrom in extensions/discord/src/shared.ts, with regression coverage through the production Discord plugin adapter and resolveCommandAuthorization. Consider aligning extensions/discord/src/security.ts to the same top-level-or-legacy contract, but keep the separate #39341/#72467 security-audit scope distinct. Add a user-facing changelog entry because this restores owner-only tools for Discord operators.
