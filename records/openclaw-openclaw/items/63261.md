## Summary

Keep this issue open. Current main at 39f810911c382f92ea80b2fb406fdf7dbd405721 still has the reported Discord owner-auth gap: the production Discord config adapter returns only legacy dm.allowFrom, while the canonical config, docs, account merge behavior, and adjacent Discord allowlist path use top-level allowFrom first. That can still produce ownerList=[] and senderIsOwner=false for a Discord DM sender allowlisted only via channels.discord.allowFrom, which then hides owner-only tools such as cron and gateway.

## What This Changes

Make Discord owner authorization use canonical top-level channels.discord.allowFrom, falling back to legacy channels.discord.dm.allowFrom, so allowlisted DM senders retain owner-only tools.

## Best Possible Solution

Land a focused Discord plugin fix in extensions/discord/src/shared.ts so discordConfigAdapter.resolveAllowFrom returns account.config.allowFrom ?? account.config.dm?.allowFrom, add regression coverage through the production Discord adapter plus resolveCommandAuthorization, and add a user-facing changelog entry because this restores Discord owner-only tools. Keep the broader security-audit follow-up in #39341 separate.
