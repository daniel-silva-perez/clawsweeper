## Summary

Keep open. Current main still has the reported Discord runtime bug: the Discord plugin config adapter resolves owner candidates from legacy `dm.allowFrom` only, while docs/schema/config normalization treat top-level `channels.discord.allowFrom` as canonical. The prior fix PRs #63453 and #63515 were closed unmerged, and the currently open #72467 is a separate security-audit fix, not the runtime owner-auth path.

## Best Possible Solution

Land a focused Discord plugin fix that resolves allowFrom from `account.config.allowFrom ?? account.config.dm?.allowFrom` in the Discord runtime config adapter, with regression tests through the production adapter/command-auth path. Also align the Discord DM security resolver if maintainers agree it is the same contract. Preserve empty top-level list semantics, avoid core fallback changes, and treat #63453/#63515 as useful prior unmerged context rather than shipped fixes.

## Clownfish Work Prompt

Fix the Discord runtime owner-auth bug where a DM sender allowlisted via top-level `channels.discord.allowFrom` is not treated as owner. On current main, `extensions/discord/src/shared.ts` has `discordConfigAdapter.resolveAllowFrom` reading only `account.config.dm?.allowFrom`, while docs/schema/config normalization make top-level `allowFrom` canonical. Update the Discord plugin adapter to use `account.config.allowFrom ?? account.config.dm?.allowFrom`, preserving `allowFrom: []` as an intentional empty top-level list. Add regression coverage that exercises the production Discord adapter and `resolveCommandAuthorization` path so a config with `channels.discord.allowFrom: ["123"]`, no `commands.ownerAllowFrom`, and a Discord DM sender `123` produces `ownerList: ["123"]` and `senderIsOwner: true`. Also inspect/fix `extensions/discord/src/security.ts` DM policy resolver if it should follow the same top-or-nested contract, with focused coverage. Do not change core command-auth fallback semantics or owner-only tool policy. Include a user-facing changelog fix entry thanking the reporter/related contributors per repo rules. Prior context: #63453 and #63515 were closed unmerged; #72467 is adjacent security-audit work for #39341 and is not a runtime fix for this issue.
