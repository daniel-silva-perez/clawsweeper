## Summary

Keep open. Current main still contains several concrete, heterogeneous audit leads: the tracked root repair artifacts remain, the Pi MCP harness still dynamically imports filesystem paths from `require.resolve()`, and the browser-evaluate and slash-command areas have partial docs/tests but still need owner-level policy calibration rather than cleanup closure.

## What This Changes

The issue requests maintainer review of a community `agchk` architecture scan with concrete leads around root repair artifacts, browser evaluation trust boundaries, slash-command policy, secret-fixture calibration, Pi MCP dynamic imports, and timer delay clamping.

## Best Possible Solution

Keep this open long enough for maintainers to split it into focused follow-ups: land or replace the root-artifact cleanup, decide and test Pi MCP file-URL normalization, separately document or tighten browser-evaluate trust-boundary policy, review slash-command catalog/queueing against gateway method scopes, and re-scope timer/scanner-calibration leads against current main.

## Reproduction Assessment

Yes for the source-level leads: current-main inspection confirms the tracked root artifacts, direct Pi MCP dynamic imports, browser dynamic evaluation path with config gating, and local slash-command dispatch/gateway execution split. The external `agchk` severity totals were not rerun, but they are not needed to reproduce the remaining concrete source leads.

## Solution Assessment

No single change is the best way to solve this issue. The maintainable path is to split narrow cleanup and portability fixes from browser/slash-command policy decisions, then validate each with targeted tests and docs checks.
