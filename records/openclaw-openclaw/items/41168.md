## Summary

Keep this issue open. Current main still defaults a TUI launch with no explicit session to `global` or the agent main session, then loads that history directly. It does not select the latest user-active non-heartbeat session. The storage concern has newer maintenance/snapshot mitigations, but the central TUI resume behavior remains actionable.

## Best Possible Solution

Keep this issue open. Implement a TUI startup resume policy for the absent-`--session` case: after agent metadata is known, list sessions for the current agent, expose or derive `lastInteractionAt`, exclude heartbeat/cron/system-only sessions, select the latest real user-active session, and then load that history. Preserve explicit `--session` behavior, add TUI/gateway regression coverage, update TUI docs, and split any remaining storage growth into a focused session-retention or skill-snapshot issue if it still reproduces.

## Risks / Open Questions

- Changing the absent-`--session` startup target is user-visible and needs a precise fallback policy for explicit `--session`, global scope, workspace-derived agent selection, missing stores, heartbeat-isolated sessions, cron/system rows, and main-session fallback.
- The current Gateway session list uses `updatedAt`, which can be touched by heartbeat or system bookkeeping; implementing this with `updatedAt` alone could recreate the reported wrong-session behavior.
- The storage-growth portion may need a narrower follow-up if it still reproduces after current session maintenance and skill snapshot behavior, with concrete store sizes and reproduction steps.
