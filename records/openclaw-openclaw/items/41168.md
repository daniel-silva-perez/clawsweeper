## Summary

Keep this issue open. Current main still defaults a no-`--session` TUI launch to the current agent's main session and loads that history directly; it does not choose the latest user-active non-heartbeat session. The storage side has caps and maintenance, but per-session skill snapshots are still persisted, so that part also needs focused verification rather than closure.

## Best Possible Solution

Keep this issue open. Implement a TUI startup resume policy for the no-`--session` case after agent metadata is known: list sessions for the current agent, expose or derive `lastInteractionAt`, filter heartbeat/cron/system-only rows, select the latest real user-active session, then load that history. Preserve explicit `--session` and global behavior, add TUI/gateway regression coverage, update the TUI docs, and separately verify whether per-session `skillsSnapshot` storage needs dedupe or a narrower storage-retention follow-up.

## Risks / Open Questions

- Changing the absent-`--session` startup target is user-visible; explicit `--session`, global scope, workspace-derived agent selection, missing stores, and main-session fallback need a compatibility policy.
- A resume policy based only on `updatedAt` could recreate the reported heartbeat-selection bug because heartbeat/system bookkeeping may mutate rows without a real user interaction.
- The storage-growth report combines session selection and skill snapshots; after the resume fix, maintainers should verify whether snapshot dedupe, snapshot-size reduction, or a separate retention issue is still needed.
