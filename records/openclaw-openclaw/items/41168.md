## Summary

Current main still does not implement the reported TUI startup behavior. A TUI launch without an explicit session still resolves to the current agent's main session, refreshes agent metadata, and loads that resolved history directly; it does not select the latest user-active non-heartbeat session. Session maintenance and skill snapshot behavior have improved the storage side, but they do not close the central TUI resume bug.

## Best Possible Solution

Keep this issue open. Implement a TUI startup resume policy for the absent-`--session` case: after agent metadata is known, list sessions for the current agent, expose or derive `lastInteractionAt`, exclude heartbeat/cron/system-only rows, select the latest real user-active session, and then load that history. Preserve explicit `--session` and global-scope behavior, add TUI/gateway regression coverage, update TUI docs, and split any remaining storage growth into a focused session-retention or skill-snapshot issue if it still reproduces.

## Risks / Open Questions

- Changing the absent-`--session` startup target is user-visible; explicit `--session`, global scope, workspace-derived agent selection, missing stores, and main-session fallback all need a precise compatibility policy.
- Using `updatedAt` alone for latest-session selection could recreate the reported bug because heartbeat or system bookkeeping can update a row without a real user/channel interaction.
- The storage-growth portion should be verified separately after current session maintenance and skill-snapshot behavior; if it still reproduces, it likely deserves a focused retention/snapshot-size issue with concrete store sizes and reproduction steps.
