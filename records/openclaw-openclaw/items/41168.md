## Summary

Current main still does not implement the requested no-`--session` TUI resume policy, and the storage-bloat half is only partly mitigated. The TUI still resolves an empty session argument to the agent main session, loads that history on connect, and later refreshes metadata only for the already-selected key. `skillsSnapshot` is still part of `SessionEntry` and the session store is serialized with plain `JSON.stringify`, while open PR #48425 targets that narrower storage problem but is not merged.

## Best Possible Solution

Keep this issue open. Implement a TUI startup path for no explicit `--session`: after agent metadata is available, list sessions for the selected agent, expose or derive `lastInteractionAt`, filter heartbeat/cron/system-only sessions, choose the latest real user-active session, and fall back to the configured main session when none exists. Preserve explicit `--session` and global-scope behavior, add TUI/gateway regression coverage, update https://docs.openclaw.ai/web/tui, and handle the `skillsSnapshot` bloat through PR #48425 or a canonical storage issue.
