## Summary

Current main still does not implement the reported TUI startup behavior. With no explicit `--session`, the TUI still resolves to `global` or the agent main session and loads that history; it does not choose the latest user-active non-heartbeat session. Storage has current mitigations through compact skill snapshots, but the central TUI resume bug remains actionable.

## Best Possible Solution

Keep this issue open. Implement a TUI startup resume policy for the absent-`--session` case: after agent metadata is known, list sessions for the current agent, expose or derive a real user-activity timestamp such as `lastInteractionAt`, exclude heartbeat/cron/system-only sessions, select the latest user-active session, and load that history. Preserve explicit `--session` behavior, add TUI/gateway regression coverage, and update the TUI docs. Split any remaining storage growth into a narrower session-retention or skill-snapshot issue if it still reproduces.
