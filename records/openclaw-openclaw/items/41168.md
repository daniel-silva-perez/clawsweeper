## Summary

Current main still does not implement the requested no-`--session` TUI resume policy. The TUI resolves an empty session argument to the agent main session, loads history for that key on connect, and only refreshes metadata for the already-selected key. The storage half is partly mitigated by session maintenance, but per-session `skillsSnapshot` persistence still exists and has related open follow-up work, so this issue should remain open or be split into narrower tracked items.

## Best Possible Solution

Keep this issue open. Implement a TUI no-`--session` resume policy after agent metadata is available: list sessions for the selected agent, expose or derive `lastInteractionAt`, filter heartbeat/cron/system-only sessions, select the latest real user-active session, and fall back to the configured main session when none exists. Preserve explicit `--session` and global-scope behavior, add TUI/gateway regression coverage, update the TUI docs, and track the `skillsSnapshot` storage work through the existing narrower storage-bloat PR/issues or split it from this combined report.
