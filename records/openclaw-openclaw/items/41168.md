## Summary

Keep this issue open. Current main still defaults a no-`--session` TUI launch to the agent main session and loads that selected key; it does not choose the latest user-active non-heartbeat session. The storage side is also not fully resolved: session-store maintenance now bounds old rows, but new/first-turn sessions still persist a full per-session `skillsSnapshot`.

## Best Possible Solution

Keep this issue open. Implement a no-`--session` TUI resume policy after agent metadata is available: list sessions for the selected agent, expose or derive `lastInteractionAt`, filter heartbeat/cron/system-only rows, choose the latest real user-active session, and fall back to the main session when no user session exists. Preserve explicit `--session` and global-scope behavior, add TUI/gateway regression coverage, update the TUI docs, and separately settle whether per-session `skillsSnapshot` storage needs dedupe or a narrower storage follow-up.

## Risks / Open Questions

- A fix based only on `updatedAt` could still pick heartbeat, cron, exec-notification, or bookkeeping-updated sessions over the last real user session.
- The TUI likely needs either `lastInteractionAt` exposed through the sessions API or a carefully derived transcript/user-message signal before it can resume the right session reliably.
- The storage complaint combines row-retention and per-session skill snapshot duplication; maintainers should decide whether dedupe, snapshot-size reduction, retention tuning, or a narrower follow-up issue is the right product fix.
