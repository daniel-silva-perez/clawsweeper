## Summary

Keep this PR open. Current main has not implemented the proposed short aliases, and the PR still needs maintainer review or a revision because `/m` is not covered by the model directive parser path that actually handles `/model` switching.

## Best Possible Solution

Keep the PR open and ask for a focused revision: either drop `/m` from the alias set, or wire `/m` through the same model directive path as `/model` with regression tests for `/m <provider/model>`, `/m` status behavior, and configured model-alias collision handling. The other proposed aliases look mechanically consistent with the registry/handler pattern, and the separated `/model` versus `/models` docs from the PR should be retained if the feature proceeds.

## Risks / Open Questions

- Merging the PR as-is could advertise `/m` as a model-switching shortcut while raw `/m ...` is not recognized by the model directive parser.
- Registering `/m` as a built-in alias would reserve that token and can shadow user-configured model aliases named `m`, so this needs explicit product/compatibility judgment or a tested migration path.
