## Summary

Keep open. Current main still has the automatic daily/idle rollover gap, and this PR remains a plausible implementation candidate, but the proposed patch still has a behavior-sensitive latency issue and newer open alternatives exist, so conservative cleanup should leave it for maintainer review.

## What This Changes

The PR emits automatic daily/idle session reset hook events, expands the bundled session-memory hook to handle them, and updates related docs and tests.

## Best Possible Solution

Land one focused canonical fix that gives automatic rollover the same memory-preservation guarantee as manual resets, passes a stable pre-reset or archived transcript reference, keeps expensive hook work off the reply-critical path, and updates the bundled hook metadata, handler, docs, and regression tests.

## Reproduction Assessment

Yes. Configure an enabled session-memory hook, let a session go stale by daily or idle policy, then send the next message: current main rotates and archives the session without any internal session reset hook, while this PR's patch can be inspected to show the awaited hook path.

## Solution Assessment

No, not as-is. The direction is right, but the hook dispatch should be non-blocking or otherwise bounded with stable transcript access rather than awaited inline during inbound session initialization.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Run automatic reset hooks off the reply path:** `src/auto-reply/reply/session.ts:685`
  - body: This new stale-rollover branch awaits `triggerInternalHook(...)` before archiving. With `session-memory` subscribed, `triggerInternalHook` awaits each handler and the handler can call `generateSlugViaLLM` with a 15s timeout, so the first user message after idle/daily expiry can block on memory housekeeping and hit channel timeouts. Dispatch this work asynchronously or pass a stable transcript path so memory capture does not delay replies.
  - confidence: 0.86
