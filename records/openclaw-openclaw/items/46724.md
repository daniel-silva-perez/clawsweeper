## Summary

Keep open. This is a real core gateway/protocol feature request, not an already-implemented behavior or a ClawHub-only plugin idea. Current main models the Gateway as a single control plane with operator/node clients; there is discovery for finding gateways, but no gateway-to-gateway authenticated role, routing method, or federation/HA semantics.

## Best Possible Solution

Keep this issue open as a core design request. The best path is maintainer product/design review for additive Gateway protocol work: define gateway peer identity/auth, role or capability semantics, cross-gateway routing and ACLs, session/presence ownership, HA/failover behavior, schema/docs/client updates, and contract tests. If maintainers want external implementations, first add a narrow documented plugin/API seam that can safely express gateway federation.

## Risks / Open Questions

- A future implementation would expand the Gateway trust boundary and needs explicit security review for gateway identity, pairing, scopes, replay/idempotency, routing authorization, and downgrade/version handling.
- The existing Bonjour `role=gateway` string is only unauthenticated discovery metadata; treating it as implemented federation would close a still-missing core capability prematurely.
