## Summary

Keep open. Current `main` still accepts Anthropic `type: "token"` profiles and even prefers token profiles ahead of API-key profiles in automatic auth order, while the Anthropic docs still describe setup-token/paste-token as supported and do not clearly warn Enterprise/API-key users to use `type: "api_key"`. The boundary-aware Anthropic transport now appears to choose `x-api-key` vs `Authorization: Bearer` from the credential prefix rather than the auth-profile mode, so the exact header-routing failure may already be avoided in that runtime path, but the requested validation/diagnostic/docs cleanup is not implemented and removing token support would conflict with the supported Anthropic setup-token path.

## What This Changes

The issue requests validation, diagnostics, and documentation for Anthropic auth profiles so Enterprise API keys are configured as `api_key` instead of the bearer-token profile path.

## Best Possible Solution

Keep the issue open and decide the product boundary explicitly: preserve supported Anthropic setup-token/Claude CLI token auth, but add a targeted diagnostic/docs path for Anthropic API-key-shaped credentials stored as `type: "token"` and document that Enterprise/API-only Anthropic accounts should use `type: "api_key"` with `key`. If behavior changes, add focused auth-profile and Anthropic transport/status tests rather than removing token support globally.
