## Summary

Keep open. Current main still accepts Anthropic `type: "token"` auth profiles and intentionally exposes Anthropic setup-token/paste-token as supported token auth, so removing token support would be too broad. The Anthropic transport now appears to route `sk-ant-api...` values through the API-key path by credential prefix, but the requested validation, diagnostic warning, and clearer Enterprise/API-key documentation are still missing.

## What This Changes

The issue requests Anthropic auth-profile validation, diagnostics, and docs so Enterprise/API-only credentials are stored as `api_key` rather than misleading `token` profiles that can cause repeated Anthropic 529 failures.

## Best Possible Solution

Preserve supported Anthropic token auth, then add a targeted diagnostic/docs path for Anthropic API-key-shaped credentials stored as `type: "token"`, steering Enterprise/API-only users to `type: "api_key"` with `key` while keeping `sk-ant-oat...` and setup-token flows on the bearer-token path.
