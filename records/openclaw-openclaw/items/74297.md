## Summary

Keep open. Current main appears to avoid the original wrong-header failure for API-key-shaped Anthropic credentials by sending non-`sk-ant-oat` credentials as `x-api-key`, but it still accepts Anthropic `type: "token"` profiles, automatic profile ordering can prefer token profiles over API-key profiles, and the Anthropic docs/status paths do not clearly warn Enterprise/API-only users to store `sk-ant-api` keys as `type: "api_key"` with `key`.

## What This Changes

The issue requests validation, diagnostics, and documentation so Anthropic Enterprise/API-only credentials are stored and explained as `api_key` profiles rather than `token` profiles.

## Best Possible Solution

Preserve Anthropic setup-token and Claude CLI token support, add a targeted no-secret diagnostic/status warning for Anthropic `type: "token"` profiles that contain API-key-shaped `sk-ant-api` credentials, and update the public Anthropic provider docs to show Enterprise/API-only accounts should use `type: "api_key"` with `key`.

## Reproduction Assessment

No, not for the exact 529 loop against current main without a live Anthropic Enterprise credential. There is a source-level reproduction for the remaining gap: current main accepts/selects Anthropic token profiles and lacks a warning, while transport tests show API-key-shaped values now route as `x-api-key`.

## Solution Assessment

Yes, the best fix is a narrow diagnostic and docs change, not global removal of token support. Removing `token` would conflict with the supported Anthropic setup-token path and other bearer-token providers.
