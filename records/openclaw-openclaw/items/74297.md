## Summary

Keep open. Current main avoids the exact wrong-header transport path for `sk-ant-api`-shaped values, but it still accepts Anthropic `token` profiles and treats API-key-shaped token entries as static/healthy, so the requested diagnostics and clearer docs remain actionable.

## What This Changes

Add validation, diagnostics, and documentation so Anthropic Enterprise/API-only credentials use `type: "api_key"` rather than `type: "token"` in `auth-profiles.json`.

## Best Possible Solution

Preserve Anthropic setup-token and generic token-profile support, but add an Anthropic-owned warning in doctor/status or the auth-profile health path when a token profile value looks like an Anthropic API key, and document the exact `api_key`/`key` shape for API-only and Enterprise accounts.

## Reproduction Assessment

Partially yes. Current-main source gives a high-confidence static reproduction by storing an Anthropic `type: "token"` profile with an API-key-shaped value: it is accepted, can be ordered before API-key profiles, and is not reported by doctor/auth health; the live provider-side 529 loop still needs real Enterprise credentials.

## Solution Assessment

Not applicable as a patch review. The safest direction is a narrow provider-owned diagnostic plus docs update, not removing the supported `token` mode or Anthropic setup-token path.
