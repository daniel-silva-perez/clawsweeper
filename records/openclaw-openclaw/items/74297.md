## Summary

Keep open. Current main appears to avoid the exact wrong-header path for API-key-shaped Anthropic credentials, but Anthropic `type: "token"` profiles are still accepted and can be auto-selected without a targeted doctor/status warning when the stored value looks like an API key. Token support is intentional, so the right cleanup is provider-owned diagnostics and docs rather than removing the schema mode.

## What This Changes

The issue asks OpenClaw to validate or warn on Anthropic `type: "token"` auth profiles that contain Enterprise/API-key credentials and to document the token vs `api_key` auth-profile distinction.

## Best Possible Solution

Preserve Anthropic setup-token and Claude CLI token support, but add a provider-owned doctor/status warning for Anthropic token profiles whose stored value looks like an API key, and document that API-only/Enterprise Anthropic accounts should use `type: "api_key"` with `key`.

## Reproduction Assessment

Partially yes. Current-main source gives a high-confidence static reproduction for the missing validation/docs path by storing an Anthropic token profile with an API-key-shaped value, but the live 529 loop needs an Enterprise credential and was not reproduced here.

## Solution Assessment

Not applicable as a patch review. Removing `token` from the global schema is not the best fix because Anthropic setup-token auth is intentionally supported; a narrow Anthropic diagnostic plus docs update is safer.
