## Summary

Keep open. Current main contains the related text-only Discord intent fix, but the default voice-enabled Discord runtime path remains active and I found no source, test, docs, changelog, or history evidence that the reported idle CPU spin has been fixed.

## What This Changes

The issue requests investigation and remediation for sustained gateway CPU usage when the embedded Discord plugin runs with voice enabled after bot accounts connect.

## Best Possible Solution

Keep the issue open and investigate in the Discord plugin. Reproduce/profile gateway idle CPU with `channels.discord.voice.enabled=true`, isolate whether the loop is in gateway voice-state dispatch, the voice plugin adapter, `DiscordVoiceManager` lifecycle, auto-join/session recovery, or `@discordjs/voice`, then make enabled voice idle safely or deliberately change the runtime/default with docs, changelog, and focused regression coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Unclear for current main: the report gives concrete v2026.4.26 observations and an A/B workaround, but this review did not live-profile current main with real Discord bot accounts; source inspection only confirms the implicated path remains active.

## Solution Assessment

Is this the best way to solve the issue? Not applicable for a proposed patch, and closing via the #73709 text-only fix would not solve this report. The narrowest maintainable solution is to profile the enabled voice path first, then patch the actual loop or explicitly change the voice default with maintainer approval.
