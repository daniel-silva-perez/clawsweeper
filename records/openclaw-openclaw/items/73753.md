## Summary

Keep open. Current main has the related text-only Discord intent fix from #73709, but the default voice-enabled path still requests `GuildVoiceStates`, installs the Discord voice plugin, creates `DiscordVoiceManager`, and registers the ready listener; I found no source, docs, changelog, or history evidence that the reported enabled-voice idle CPU spin is fixed.

## What This Changes

The issue reports sustained `openclaw-gateway` CPU usage after embedded Discord bot accounts connect when `channels.discord.voice.enabled` is true or omitted.

## Best Possible Solution

Keep the issue open and investigate in the Discord plugin. Profile `openclaw-gateway` on Linux with embedded Discord and `channels.discord.voice.enabled=true` after bot accounts connect, isolate the hot loop in the enabled voice path, then either make default voice idle safely or deliberately change the voice default/runtime behavior with docs, changelog, and focused regression coverage.

## Reproduction Assessment

Unclear. The report gives concrete v2026.4.26 observations and an A/B workaround, but this review did not live-profile current main with real Discord bot accounts; source inspection confirms the implicated enabled-voice path remains active.

## Solution Assessment

Not applicable for a proposed patch. Closing via the #73709 text-only intent fix would not solve this report; the narrow maintainable path is to profile enabled voice first, then patch the actual loop or make a maintainer-approved default/runtime change.
