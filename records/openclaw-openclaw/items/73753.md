## Summary

Keep open. Current main still leaves the default Discord voice-enabled path active, and the shipped related fix only covers the text-only `voice.enabled=false` intent-selection bug, not the enabled-voice idle CPU spin reported here.

## What This Changes

The issue reports sustained `openclaw-gateway` CPU spin after embedded Discord bot accounts connect when Discord voice remains enabled by default.

## Best Possible Solution

Investigate in the bundled Discord plugin on current main with `channels.discord.voice.enabled=true`, profile the idle gateway after real bot connections, then land a focused runtime fix or an explicit maintainer-approved voice default change with Discord regression coverage, docs, and a changelog entry.

Do we have a high-confidence way to reproduce the issue? Unclear: the report gives concrete v2026.4.26 A/B observations and current source still follows the implicated enabled-voice path, but current main has not been live-profiled here with real Discord bot accounts.

## Solution Assessment

Is this the best way to solve the issue? Not applicable because there is no patch. Closing via the #73709 text-only intent fix would not solve this enabled-voice report; the maintainable path is profiling first, then patching the actual loop or making a deliberate default change.
