## Summary

Keep open. Current main contains the related text-only intent-selection fix for #73709, but it still leaves the reported default voice-enabled path active: Discord voice is enabled by default, `GuildVoiceStates` is requested when voice is enabled, and startup still creates the Carbon `VoicePlugin` plus `DiscordVoiceManager` for that path. The issue is recent and reports a remaining CPU spin specifically when voice stays enabled, so there is not enough evidence to close it as implemented or duplicate.

## Best Possible Solution

Keep the issue open and investigate in the Discord plugin. A good fix path is to reproduce/profile the gateway on Linux with `channels.discord.voice.enabled=true`, isolate whether the spin is in the Carbon `VoicePlugin`, `@discordjs/voice`, voice-state handling, auto-join/session lifecycle, or OpenClaw listener wiring, then make the enabled voice runtime idle safely or more explicitly opt-in. Add a focused regression/perf test or lifecycle test around the identified loop, plus docs/changelog if behavior or defaults change.
