## Summary

Keep open. Current main still keeps the default Discord voice-enabled path active, and the only shipped/current evidence I found addresses the separate text-only `voice.enabled=false` intent bug from #73709 rather than the enabled-voice idle CPU spin reported here.

## What This Changes

The issue reports sustained gateway CPU usage after embedded Discord connects when `channels.discord.voice.enabled=true`, the default Discord voice runtime path.

## Best Possible Solution

Keep this open and investigate in the bundled Discord plugin with a live/profiled enabled-voice setup. The desired end state is a focused runtime or dependency workaround that makes default Discord voice idle safely after connect, or an explicit maintainer-approved default/config change with Discord regression coverage, docs, and changelog updates.

## Reproduction Assessment

Unclear. The report gives concrete v2026.4.26 A/B observations on Linux and macOS, and current source inspection confirms the implicated enabled-voice path remains active, but this read-only review did not live-profile current main with real Discord bot accounts.

Not applicable for a proposed patch because there is none. Closing via the #73709 text-only intent fix would not solve this enabled-voice report; the maintainable path is to profile the enabled runtime first, then patch the actual loop or intentionally change the default with maintainer approval.
