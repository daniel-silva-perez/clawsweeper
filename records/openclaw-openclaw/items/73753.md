## Summary

Keep open. Current main contains the related text-only intent fix for #73709, but the reported default enabled-voice path is still active: Discord voice defaults on, `GuildVoiceStates` is included for that default, Carbon `VoicePlugin` is registered, and `DiscordVoiceManager` plus its ready listener are loaded when voice is enabled. I found no current source, test, docs, or changelog evidence that the enabled-voice idle CPU spin has been fixed or superseded.

## Best Possible Solution

Keep this issue open and investigate in `extensions/discord`: reproduce/profile gateway idle CPU with `channels.discord.voice.enabled=true`, isolate whether the loop is in Carbon `VoicePlugin`, `@discordjs/voice`, voice-state event handling, auto-join/session lifecycle, or OpenClaw listener wiring, then make enabled voice idle safely or make the runtime more explicitly opt-in. Add focused regression or lifecycle/perf coverage around the identified loop and update docs/changelog if defaults or behavior change.
