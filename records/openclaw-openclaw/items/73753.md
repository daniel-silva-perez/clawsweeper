## Summary

Keep open. Current main still keeps the default Discord voice-enabled path active: it requests `GuildVoiceStates`, installs the voice plugin, loads the voice manager, dispatches voice gateway updates, and registers the ready listener. The shipped #73709 fix covers text-only configs with `voice.enabled=false`, but I found no source, test, docs, changelog, or release evidence that the reported default enabled-voice idle CPU spin has been fixed.

## What This Changes

Fix the Discord voice-enabled gateway path so embedded Discord can connect with `channels.discord.voice.enabled=true` without sustained idle CPU spin after bot accounts connect.

## Best Possible Solution

Keep this open and investigate in the bundled Discord plugin. Reproduce and profile `openclaw-gateway` on Linux with embedded Discord and `channels.discord.voice.enabled=true`, isolate the specific voice-state, adapter, auto-join, or dependency loop, then land a focused runtime fix or an explicit maintainer-approved default change with Discord regression coverage and user-facing docs/changelog updates.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Unclear for current main: the report gives concrete v2026.4.26 A/B observations and source inspection confirms the implicated enabled-voice path remains active, but I did not live-profile current main with real Discord bot accounts.

## Solution Assessment

Is this the best way to solve the issue? Not applicable for a proposed patch because there is none; closing via the #73709 text-only intent fix would not solve this enabled-voice report. The narrowest maintainable path is to profile the enabled voice runtime first, then patch the actual loop or deliberately change the voice default with maintainer approval.
