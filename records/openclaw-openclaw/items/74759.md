## Summary

Keep this open. Current main has a configured SecretRef foundation, including env/file/exec providers and snapshot-based command resolution, but it still does not implement the requested lazy agent-callable `secret.get`/`secret.has`/`secret.list` broker. The issue discussion already identifies the key remaining product/security choice: either ship a trusted bundled implementation or first extend the plugin API so external plugins can return redacted, session-scoped secret results safely.

## What This Changes

Add a first-class secret broker plugin/tool surface exposing `secret.get`, `secret.has`, and `secret.list` over backends such as Bitwarden, Keychain, 1Password, and Vault with redaction and session-scope safeguards.

## Best Possible Solution

Define a trusted secret capability contract on top of the existing SecretRef provider foundation, including `has`/`list` metadata semantics, redacted or non-persistent `get` results, and owner/direct-session gating before the tool is offered or executed. Then implement the first backend either as a maintainer-approved bundled plugin or after a reviewed plugin API extension makes the same safety guarantees available to external plugins.

## Reproduction Assessment

Not applicable as a feature and security-design request. The current-main gap is high-confidence from exact source search plus docs/code inspection: configured SecretRefs exist, but lazy agent-callable `secret.*` tools and their safe redaction/session-scope contract do not.

## Solution Assessment

Unclear. The requested generic plugin direction is not sufficient by itself because external plugins cannot currently register the trusted tool-result middleware needed for pre-model redaction; the safer path is a maintainer choice between a bundled trusted implementation and a narrow plugin API trust extension first.

## Security Review

Status: needs_attention

Summary: This issue proposes returning live credential material through agent tools, so it needs a trusted redaction/non-persistence and session-scope contract before implementation.

Concerns:

- **[high] Define redacted secret result semantics before get:** `src/plugins/registry.ts:369`
  - body: A `secret.get` tool would intentionally return credential material.   - confidence: 0.93
- **[high] Gate secret tools by trusted session scope:** `docs/channels/groups.md:81`
  - body: OpenClaw supports group/channel sessions where trigger authorization is separate from context visibility. Secret tools should be hidden or blocked before execution for group/channel and otherwise untrusted sessions, not only redacted after a result exists.
  - confidence: 0.88
