## Summary

Keep this issue open. Current main has a substantial SecretRef system for configured credentials, including exec-backed 1Password/Vault-style providers, but it does not expose lazy agent-callable `secret.get`/`secret.has`/`secret.list` tools. The follow-up discussion correctly narrows the remaining work to a maintainer product/security choice because the pre-model tool-result rewrite seam is bundled-only today.

## What This Changes

Add a generic secret broker plugin or plugin API surface exposing `secret.get`, `secret.has`, and `secret.list` over Bitwarden, Keychain, 1Password, and Vault with redaction and session-scope guards.

## Best Possible Solution

Define a trusted secret capability contract that reuses the existing SecretRef provider foundation, adds explicit redacted or non-persistent secret result semantics, and enforces owner/session trust policy before exposing any Bitwarden, Keychain, 1Password, or Vault-backed agent tools.
