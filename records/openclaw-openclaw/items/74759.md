## Summary

Keep this open. Current main has a strong configured SecretRef system with env/file/exec providers, but it still lacks lazy agent-callable `secret.get`/`secret.has`/`secret.list` tools and the trusted redaction plus session-scope contract needed before returning live credential material. The follow-up discussion also converges on this being a maintainer architecture/product choice rather than a straightforward plugin patch.

## What This Changes

Add a generic secret broker plugin with agent-callable `secret.get`/`secret.has`/`secret.list` over Bitwarden, Keychain, 1Password, and Vault, with transcript redaction and trusted session guards.

## Best Possible Solution

Define a trusted secret capability contract on top of the existing SecretRef provider foundation: `has`/`list` should expose names only, `get` should return a redacted or non-persistent result shape, and the harness should block or hide the tools before execution unless the session satisfies an explicit trusted owner-only scope. After that contract is approved, implement the first backend either as a bundled trusted plugin or as an external plugin using the new API, with docs and focused tests for redaction, persistence, and session gating.

## Reproduction Assessment

Not applicable as a feature and security-design request. The current-main gap is high-confidence from source and docs inspection: configured SecretRefs exist, while lazy `secret.*` agent tools and a safe trusted result/session-scope API do not.

## Solution Assessment

Unclear. The requested plugin direction is plausible, but not sufficient by itself because current external plugins cannot register the trusted result middleware or trusted pre-tool policy needed for this surface; the safer path is a maintainer-approved bundled implementation or a small plugin API extension first.
