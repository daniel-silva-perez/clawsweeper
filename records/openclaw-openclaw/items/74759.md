## Summary

Keep this open. Current main has substantial SecretRef support for configured credentials, including exec/file/env providers, but it does not implement lazy agent-callable `secret.get`/`secret.has`/`secret.list` tools, and the issue discussion correctly identifies the remaining work as a maintainer security/product decision rather than a cleanup close.

## What This Changes

Add a generic secret broker plugin/API surface so agents can resolve named secrets at call time through backends such as Bitwarden, macOS Keychain, 1Password, or Vault.

## Best Possible Solution

Define a trusted secret capability contract that reuses the existing SecretRef provider foundation, adds explicit redacted or non-persistent result semantics, and enforces owner/session trust policy before exposing any Bitwarden, Keychain, 1Password, or Vault-backed agent tools.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: the requested tool names are absent, and the existing `secrets.resolve` path only handles known configured SecretRef target IDs from the active runtime snapshot.

## Solution Assessment

No current implementation is available to accept, and a plain plugin tool is not the best path because normal tool results are model-visible while the pre-model rewrite seam is bundled-only. The safer path is maintainer/security review of either a trusted bundled plugin or an additive plugin API extension for redacted secret results and session-scope gating.
