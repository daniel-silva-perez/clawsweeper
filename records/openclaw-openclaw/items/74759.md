## Summary

Keep this open. Current main has a mature SecretRef system for configured credentials and exec-backed providers, but it does not provide lazy agent-callable `secret.get`/`secret.has`/`secret.list`; the remaining work is a security-sensitive product/API decision around redacted result semantics and trusted session scope.

## What This Changes

Request to add a secret broker plugin exposing first-class agent tools for `secret.get`, `secret.has`, and `secret.list` over Bitwarden, macOS Keychain, 1Password, or Vault with transcript redaction and session trust guards.

## Best Possible Solution

Define a trusted secret capability contract that reuses the existing SecretRef provider foundation, adds explicit redacted or non-persistent secret result semantics, and enforces owner/session scope before any Bitwarden, Keychain, 1Password, or Vault-backed tool can return credential values.
