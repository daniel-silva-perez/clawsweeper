## Summary

Keep open. Current main has friendlier restart wording and preserves work that was already in the command queue, but it still rejects new inbound agent work during the gateway drain window before it can be appended and replayed. The related Kafka PR is closed/unmerged and would be observability, not native retry of the original user message; the closed restart-resume issue covers adjacent interrupted-session recovery, not drain-window inbound queuing.

## Best Possible Solution

Keep this issue open. The best core fix is to change command queue and restart handling so drain-window inbound agent tasks are accepted into preserved FIFO lanes without starting new work until `resetAllLanes()` after restart, then update the current rejection tests and add regression coverage for multiple messages, per-session/channel ordering, and SIGUSR1 replay. Treat durable full stop/start replay and provider-attribution copy as separately scoped follow-ups.
