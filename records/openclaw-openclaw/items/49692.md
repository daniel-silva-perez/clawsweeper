## Summary

Keep open. Current main has friendlier restart wording, but drain-window inbound work is still rejected before it enters the preserved command queue, so the original user message can still be lost. The related Kafka plugin PR was closed unmerged and would not replace native in-process replay for this restart UX path.

## What This Changes

The issue requests softer gateway restart/drain UX and automatic FIFO replay of inbound messages that arrive while the gateway is draining for restart.

## Best Possible Solution

Native in-process restart handling should accept inbound command entries during the SIGUSR1 drain window, preserve per-lane FIFO ordering, avoid starting new work while draining, and let resetAllLanes replay the preserved queue after the gateway starts again. The friendly fallback message can remain for truly non-replayable failures, while durable file-backed recovery and provider attribution should stay separate follow-ups.

## Repair Work Prompt

Fix drain-window inbound message loss for SIGUSR1 in-process gateway restarts. Modify the command queue so markGatewayDraining prevents starting new work during the drain window but enqueueCommandInLane still accepts tasks into the correct lane queue and returns a promise. Newly accepted drain-window entries must not run until resetAllLanes clears the draining flag and drains preserved queues, preserving per-lane FIFO and existing lane concurrency/task-timeout/clearCommandLane/waitForActiveTasks semantics. Update command-queue tests to assert drain-window entries are queued but not executed until resetAllLanes. Update restart/fallback tests as needed so the friendly retry message remains only for real non-replayable drain failures. Keep full stop/start durable file-backed queues, provider attribution, Kafka/event-log work, and broad channel rewrites out of this PR. Add a user-facing changelog fix entry crediting @skillz-xx if behavior changes.
