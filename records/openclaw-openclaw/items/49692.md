## Summary

Keep open. Current main has friendlier restart wording, but drain-window inbound work is still rejected before it enters the preserved command queue, so the triggering message can still be lost and require a manual resend. The closed Kafka plugin PR discussed in the thread was not merged and would not replace native in-process replay for this UX path.

## What This Changes

The issue requests friendlier user-facing gateway restart drain messaging and automatic retry/replay of inbound messages rejected during restart drain.

## Best Possible Solution

Native in-process restart handling should accept inbound command entries during the drain window, keep them queued without starting new work while the gateway is draining, and let the existing restart reset pump preserved lane queues in FIFO order after startup resumes, while tracking durable full-stop recovery separately.

## Repair Work Prompt

Fix the in-process gateway restart drain-window message loss reported here. Current main calls `markGatewayDraining()` during SIGUSR1 restart and `enqueueCommandInLane()` rejects before pushing a `QueueEntry`; `resetAllLanes()` can only replay entries that were already accepted. Change the command queue so new inbound command entries accepted during `gatewayDraining` are queued but not pumped while the gateway is draining, then are drained in FIFO order after `resetAllLanes()` clears the flag. Preserve active-work draining semantics and do not start new user work during shutdown. Update tests that currently expect `GatewayDrainingError` rejection, add coverage for multiple drain-window enqueues replaying after reset, and keep the friendlier fallback restart copy for any remaining abort/cleared-lane paths. Leave durable file-backed full stop/start persistence, provider-overload attribution, and broad session recovery manifests out of scope. Add a user-facing changelog entry if behavior changes, crediting @skillz-xx and related reporters as appropriate. Do not change plugin/provider APIs unless a small generic seam is strictly required.
