## Summary

Keep open. Current main has friendlier restart copy, but it still rejects inbound command work during the drain window before the work can enter the preserved queue, so the reported lost-message/manual-resend behavior remains. The related Kafka plugin PR was closed unmerged, #57425 is a broader open umbrella, and several overlapping open restart-recovery PRs need maintainer selection rather than another automatic close or duplicate repair job.

## What This Changes

The issue requests friendlier gateway restart/drain messaging and automatic replay of inbound messages rejected while the gateway is draining.

## Best Possible Solution

Native in-process SIGUSR1 restart handling should accept and preserve inbound command entries during the drain window, keep per-lane FIFO ordering, avoid starting new work while draining, and let `resetAllLanes()` replay preserved entries after restart. Keep the friendly fallback message for truly non-replayable drain failures, and handle durable crash/full-stop recovery, provider attribution, and broader session manifests through #57425 or a reviewed follow-up PR.

## Reproduction Assessment

Yes. The high-confidence reproduction path is unit-level: call `markGatewayDraining()`, then `enqueueCommand()`, and current main rejects with `GatewayDrainingError` before any `QueueEntry` is preserved.

## Solution Assessment

Yes for the narrow remaining bug: accepting but not draining new command entries during SIGUSR1 drain is the smallest maintainable in-process fix. Broader pending-inbound stores or drain manifests may be useful, but they need maintainer/security review because they expand persistence and replay semantics.
