## Summary

Keep open. Current main still has the friendlier restart message, but inbound work during the drain window is rejected before it can enter the command queue, so the original message can still be lost and require a manual resend. The closed Kafka producer PR and the broader restart-recovery tracker do not replace the narrow native in-process replay fix requested here.

## What This Changes

The issue requests softer gateway restart/drain UX and automatic FIFO retry of inbound messages that arrive while the gateway is draining for restart.

## Best Possible Solution

Native in-process restart handling should accept inbound command entries during the SIGUSR1 drain window, preserve per-lane FIFO ordering, avoid starting new work while the gateway is draining, and let resetAllLanes replay the preserved queue after the gateway starts again. The friendly fallback message should remain for truly non-replayable drain failures, while file-backed durable recovery and provider attribution should stay separate follow-ups.

## Reproduction Assessment

Yes. The high-confidence reproduction path is the current command-queue unit path: call markGatewayDraining, then enqueueCommand, and main still rejects with GatewayDrainingError before any QueueEntry is preserved.

## Solution Assessment

Yes for the narrow remaining bug: changing the command queue to accept but not drain new entries during SIGUSR1 draining is the smallest maintainable fix. A durable on-disk queue for full stop/start restarts is a separate product decision.
