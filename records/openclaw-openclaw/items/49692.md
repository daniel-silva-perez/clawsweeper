Keep open because current main only improves the user-facing restart wording; it still rejects drain-window inbound work before it can enter the preserved command queue and be replayed after restart.

## What This Changes

The issue requests friendlier gateway restart/drain copy and automatic FIFO retry of inbound messages that arrive while the gateway is draining for restart.

## Best Possible Solution

Implement native drain-window queuing for in-process restarts: accept inbound command queue entries while draining, hold them without starting new work until the restart reset, replay preserved lanes in FIFO order, and update rejection tests to cover multi-message ordering. Keep durable full-process restart persistence and provider-specific overload copy as separately scoped follow-ups.

## Repair Work Prompt

Fix the remaining drain-window retry gap for #49692. Current main has friendly restart copy, but `enqueueCommandInLane()` rejects when `queueState.gatewayDraining` is true, so inbound agent work during an in-process SIGUSR1 restart is lost before `resetAllLanes()` can replay preserved queues. Implement a narrow core fix in the command queue/restart path so new command entries submitted while the gateway is draining are appended to their normal FIFO lane but are not drained until the in-process restart calls `resetAllLanes()`. Preserve existing lane ordering and concurrency behavior, and avoid starting new agent work during the draining lifecycle. Update tests that currently expect `GatewayDrainingError` rejection, add regression coverage for multiple messages queued during drain and replayed after reset in order, and keep the user-facing friendly restart copy intact for fallback restart aborts/cleared lanes. Do not implement durable full stop/start persistence, Kafka/event-log work, or provider-overload attribution in this PR. Because this is user-facing behavior, add a single-line changelog entry under the active Fixes section with `Thanks @skillz-xx`.
