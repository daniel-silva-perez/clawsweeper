## Summary

Keep open. Current main still creates Feishu synthetic reaction message IDs with the reported `:reaction:<emoji>:<uuid>` suffix and can pass that suffixed ID into Feishu API `message_id` path parameters. The linked #39558 PR is still open and explicitly targets this bug, so it is an implementation candidate rather than a reason to close before merge.

## Best Possible Solution

Keep this issue open and review, land, or replace #39558 with a narrow Feishu plugin fix that adds a well-tested reaction message ID normalizer for API-boundary `message_id` path parameters, while preserving internal synthetic IDs for dedupe/session behavior. Regression coverage should include reply dispatch plus typing, reaction, media/card, and message get/edit paths as applicable.
