## Summary

Current main still preserves Feishu synthetic reaction message IDs with the reported `:reaction:<emoji>:<uuid>` suffix and can pass those IDs into Feishu `message_id` API path parameters. The provided GitHub context also shows open PR #39558 still has closing syntax for this issue, so the issue should remain open until that PR is reviewed, landed, or replaced.

## Best Possible Solution

Keep this issue open and review, land, or replace PR #39558 with a narrow Feishu plugin fix that adds a tested reaction message ID normalizer for Feishu API-boundary `message_id` path parameters while preserving internal synthetic IDs for dedupe and session routing. Regression coverage should include reply dispatch plus typing, reactions, message get/edit, media replies, and card/streaming update paths where those IDs cross the Feishu API boundary.
