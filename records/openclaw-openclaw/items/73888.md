## Summary

Keep this PR open because current main still does not implement the user-bus env recovery, but the proposed patch needs one narrow repair before merge: it can forward parsed service-unit environment values into `systemctl` children when synthesizing DBus env.

## What This Changes

The PR updates daemon systemd helpers to synthesize missing user DBus environment for direct `systemctl --user` calls, prefer machine scope for root-effective non-root targets, narrow systemd-unavailable classification, and add tests plus a changelog entry.

## Best Possible Solution

Repair the PR by keeping the direct user-bus recovery and classifier fixes, but narrow the synthesized child env to `process.env` plus only the DBus/XDG values needed by `systemctl`, with a regression that service-unit secrets are not forwarded.

## Reproduction Assessment

Yes. #63561 provides concrete Ubuntu status/install steps, and current-main source inspection reproduces the missing recovery path: direct `systemctl --user` calls have no synthesized bus env and the classifier still mishandles a related bus error string.

## Solution Assessment

No, not as-is. The direction is the right narrow fix for #63561, but the implementation should avoid spreading parsed service environment values into `systemctl` child processes before it lands.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Avoid forwarding service secrets to systemctl:** `src/daemon/systemd.ts:394-402`
  - body: `resolveSystemctlUserBusEnv` builds the child env from `{ ...process.env, ...env }`. Status paths pass an env merged with parsed service `Environment`/`EnvironmentFile` values, including gateway tokens and other credentials, so when bus vars are missing this forwards service-only secrets to every `systemctl` child. Build the child env from the inherited process env plus only the DBus/XDG overrides needed for the probe.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The patch introduces a concrete local secret-exposure risk by passing merged service-unit environment values to `systemctl` when synthesizing user-bus env.

Concerns:

- **[medium] Service secrets can be forwarded to systemctl:** `src/daemon/systemd.ts:394`
  - body: `readGatewayServiceState` merges parsed service environment into the env passed to systemd probes, and the PR then spreads that env into the child process environment. Service unit EnvironmentFile values can include gateway tokens or provider credentials, so the fix should only add the DBus/XDG variables instead of forwarding all service env.
  - confidence: 0.86
