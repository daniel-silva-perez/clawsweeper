## Summary

Keep open. Current main still ties OpenAI prompt-cache affinity to the per-run session identity, while isolated cron intentionally creates a fresh session id and run-scoped key for each run, so the reported cross-run first-turn cache instability remains a real runtime gap.

## What This Changes

Requests stable cross-run OpenAI prompt-cache affinity for recurring isolated cron jobs, especially by deriving `prompt_cache_key` from cron job identity instead of per-run session identity.

## Best Possible Solution

Add a cron-owned prompt-cache affinity identity that is separate from runtime session identity: isolated cron should keep fresh transcripts, run-scoped session keys, active-run ownership, and cleanup behavior, while OpenAI-family payloads receive a stable job-scoped cache key when caching is enabled.

## Reproduction Assessment

Yes for the architectural reproduction path. Static inspection shows isolated cron creates a fresh random session id per run and OpenAI request builders use that session id as `prompt_cache_key`; a focused unit test can reproduce the changing key without live provider calls.

## Solution Assessment

Yes, the requested stable `prompt_cache_key` direction is the narrow maintainable fix. Reusing the cron session id or run session key would be the wrong fix because those identities also protect transcript isolation and runtime ownership.

## Repair Work Prompt

Implement stable OpenAI prompt-cache affinity for recurring isolated cron jobs without weakening isolated cron session/runtime isolation. Add a separate prompt-cache key/affinity field, not a replacement for `sessionId` or `sessionKey`, and have OpenAI-family request builders use it for `prompt_cache_key` when cache retention is enabled and the provider/API supports that field. For default isolated `cron:<jobId>` jobs, derive a stable bounded key from the cron job identity, agent identity, and preferably the resolved model/provider or a short hash so repeated runs share routing while distinct jobs do not collide. Preserve fresh transcript ids, run-scoped session keys, active-run ownership, MCP/browser cleanup, and provider compatibility stripping for non-OpenAI/proxy endpoints. Do not add broad cron cache-affinity config unless needed for the narrow fix. Add focused tests proving two isolated runs keep different runtime session ids/keys but produce the same OpenAI `prompt_cache_key`, plus transport tests for Responses and opted-in Completions behavior. Add a user-facing changelog fix entry if the repo’s active changelog requires it, crediting @lubao515, and avoid touching unrelated prompt-cache drift issues.
