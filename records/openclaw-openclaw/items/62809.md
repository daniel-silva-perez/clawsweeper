## Summary

Keep this issue open. Current main still gives isolated cron runs fresh run-scoped session identity, while OpenAI prompt-cache routing uses the model call session id/key path rather than a stable cron-job affinity key. The reported requested improvement is not implemented, and the existing run-scoped isolation is intentional for context inheritance, so the fix needs a separate cache-affinity identity rather than reusing transcript/session state.

## Best Possible Solution

Add a narrow cron cache-affinity seam that derives a stable OpenAI prompt-cache identity from the cron job and selected model, then passes that identity to OpenAI-family transports separately from `sessionId`/`sessionKey`. Cover it with a unit test proving isolated cron still uses fresh transcript/session ids while reusing the stable cache key across runs, plus a targeted live cache probe if maintainers want provider-side proof.
