## Summary

Keep this PR open. Current main still lacks the requested post-compaction checkpoint fallback, and the PR is a narrow implementation candidate for the active status bug, but the branch can report an older checkpoint's `tokensAfter` when the actual latest checkpoint has no token count, so it needs author or maintainer follow-up before merge.

## What This Changes

The PR updates shared status rendering to fall back from stale session totals to compaction checkpoint `tokensAfter` and adds a regression test in `src/auto-reply/status.test.ts`.

## Best Possible Solution

Land a focused status formatter change that selects the actual latest compaction checkpoint first, uses its `tokensAfter` only when it is a valid non-negative number, leaves stale totals unknown when the latest checkpoint has no post-compaction count, and keeps regression coverage for both the normal fallback and the newer-checkpoint-without-token case.

## Reproduction Assessment

Yes. The supplied test shape is enough for the intended bug: call `buildStatusMessage()` with `totalTokensFresh: false`, a stale `totalTokens`, and a checkpoint with `tokensAfter`; the PR should show the checkpoint value instead of the stale total or unknown. A second high-confidence counterexample for the PR is two checkpoints where the newer checkpoint lacks `tokensAfter`.

## Solution Assessment

No. The requested direction is right, but the current PR implementation is not the best fix because it finds the newest checkpoint only after filtering out checkpoints without `tokensAfter`; the safer alternative is to choose the newest checkpoint first and then validate that checkpoint's `tokensAfter`.
