## Summary

Keep this PR open. It is the active, focused implementation candidate for the related stale post-compaction status bug, and current main still does not read compaction checkpoint totals in the shared status formatter. The branch is not ready to merge as-is because it can report an older checkpoint when the actual latest checkpoint lacks `tokensAfter`, plus it carries a small unreachable branch in the new fallback logic.

## What This Changes

The PR updates the shared status message builder to fall back to compaction checkpoint `tokensAfter` when session totals are stale and adds a regression test for the post-compaction Context line.

## Best Possible Solution

Update the formatter so stale totals use `tokensAfter` only from the actual latest compaction checkpoint when that checkpoint has a valid nonnegative value, leave context unknown otherwise, remove the unreachable branch, and keep regression coverage tied to #72140.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Use the actual latest checkpoint before reading tokensAfter:** `src/status/status-message.ts:264-274`
  - body: `tokensAfter` is optional, but this helper filters first and then picks the newest checkpoint among the remaining entries. If a later compaction checkpoint lacks `tokensAfter`, status would reuse an older checkpoint's post-compaction count as if it were current. Pick the latest checkpoint first, then use its `tokensAfter` only when it is valid.
  - confidence: 0.84
- **[P3] Remove the unreachable checkpoint fallback branch:** `src/status/status-message.ts:656-657`
  - body: After the new fallback expression, `totalTokens` is only `undefined` when `totalTokensFresh === false`, which the preceding `if` already handles; otherwise `usageFallbackTotal` is numeric. Dropping this branch keeps the stale-total path easier to audit without changing behavior.
  - confidence: 0.91
