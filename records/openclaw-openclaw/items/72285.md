## Summary

Keep this PR open, but do not merge the current branch as-is. Current main still has the stale-status gap this PR is trying to fix, yet the PR fallback can report an older checkpoint when the actual newest checkpoint lacks `tokensAfter`, and the existing regression test does not cover that case.

## What This Changes

The PR changes shared status rendering to use compaction checkpoint `tokensAfter` when session totals are stale, and adds a regression test for that status output.

## Best Possible Solution

Land a focused status formatter change that selects the actual newest compaction checkpoint first, uses that checkpoint's `tokensAfter` only when it is a valid non-negative number, leaves stale totals unknown when the newest checkpoint has no post-compaction count, removes the dead fallback branch, and adds regression coverage for both the normal fallback and the newer-checkpoint-without-token case.

## Reproduction Assessment

Yes. The reported bug can be reproduced with `buildStatusMessage()` using `totalTokensFresh: false`, a stale `totalTokens`, and a checkpoint with `tokensAfter`; the PR-specific counterexample is two checkpoints where the newer one lacks `tokensAfter` and the older one has it.

## Solution Assessment

No. The requested data source is appropriate, but this PR is not the best implementation because it filters before choosing the latest checkpoint; the safer implementation chooses the latest checkpoint first and then validates its `tokensAfter`.
