## Summary

Keep open for maintainer/security review because the patch is not safe as written: current main intentionally keeps explicit device scope-upgrade requests on the manual approval path, while this PR would let local/shared-secret scope-upgrades become silent and auto-approved.

## What This Changes

The PR removes the scope-upgrade-specific `silent: false` guard in `requestDevicePairing`, allowing `allowSilentLocalPairing` to decide whether scope-upgrade pairing requests are silent.

## Best Possible Solution

Preserve manual approval for explicit device scope-upgrades, then solve any remaining VPS/local backend case through the existing narrow device-less `gateway-client` backend shared-auth path or another scoped locality/backend seam with positive coverage for that scenario and negative coverage proving paired device scopes cannot widen silently.

## Reproduction Assessment

Yes. Current main has focused tests that reproduce the sensitive path: a paired read-scoped operator reconnects locally requesting `operator.admin` and must receive `PAIRING_REQUIRED` while `approvedScopes` stays at `operator.read`.

## Solution Assessment

No. Removing the blanket scope-upgrade guard is too broad; the maintainable fix should distinguish trusted device-less backend loopback RPCs from explicit device identities, device tokens, browser, node, and other scope-upgrade paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve explicit approval for scope upgrades:** `src/gateway/server/ws-connection/message-handler.ts:1003-1005`
  - body: With this deletion, local/shared-secret `scope-upgrade` requests become silent because `shouldAllowSilentLocalPairing` returns true for that reason.
