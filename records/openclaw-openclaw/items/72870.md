## Summary

Keep this PR open for maintainer handling, but do not merge it as-is: the one-line change is a security regression because local/shared-secret scope-upgrade pairing would become silent and persist the requested scopes. The linked user report was closed after manual approval guidance, while current main preserves explicit scope-upgrade approval with regression tests and docs.

## What This Changes

The PR removes the `scope-upgrade` special case that forces device pairing requests to be non-silent, letting the existing local/bootstrap/trusted-CIDR silent-pairing flags decide silent approval.

## Best Possible Solution

Keep explicit device scope-upgrades on the normal approval path. If a real VPS/backend case remains, fix it only through the trusted device-less backend loopback/locality path and add positive coverage for that case plus negative coverage proving explicit device identities, device tokens, browsers, nodes, and remote clients cannot widen scopes silently.

## Reproduction Assessment

Yes. Current main has focused regression tests that exercise the sensitive path: a read-scoped paired operator or backend client reconnects locally requesting `operator.admin`, receives `PAIRING_REQUIRED`, and keeps `approvedScopes` unchanged.

## Solution Assessment

No. Removing the blanket `scope-upgrade` guard is broader than the reported backend/local problem and bypasses the intended manual approval boundary; the safer solution is a narrow backend/locality fix if one is still needed.
