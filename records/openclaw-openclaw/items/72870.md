## Summary

Keep open for maintainer review because the patch is not safe as written: current main intentionally keeps explicit scope-upgrade requests on the manual approval path, while this PR would allow local/shared-secret scope-upgrades to become silent and auto-approved.

## What This Changes

The PR removes the `scope-upgrade` override that forces device pairing requests to be non-silent in `src/gateway/server/ws-connection/message-handler.ts`.

## Best Possible Solution

Preserve manual approval for explicit device scope-upgrades, then fix the reported VPS/local backend case through a narrow backend/locality path with positive coverage for that scenario and negative coverage proving existing paired-device scopes cannot widen silently.

## Reproduction Assessment

Yes. Current main has focused tests that reproduce the sensitive path: a paired read-scoped operator reconnects locally requesting `operator.admin` and must receive `PAIRING_REQUIRED` while keeping `approvedScopes` unchanged.

## Solution Assessment

No. Removing the blanket scope-upgrade guard is too broad; the maintainable fix should distinguish trusted device-less backend loopback RPCs from explicit device identities, device tokens, browser, node, and other scope-upgrade paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve explicit approval for scope upgrades:** `src/gateway/server/ws-connection/message-handler.ts:1003-1005`
  - body: With this change, local/shared-secret `scope-upgrade` requests become silent because `shouldAllowSilentLocalPairing` returns true for that reason. The silent branch then approves the pending request with the requested scopes, so a paired `operator.read` device can reconnect asking for `operator.admin` and have its approved baseline widened, contrary to the current regression tests and docs.
  - confidence: 0.91
