## Summary

Keep open. Current main still accepts direct and mapped hook agent deliveries with 200 after optional idempotency-key replay dedupe, then starts an asynchronous isolated cron-style agent turn. The downstream embedded runner serializes work by session lane, but that queues duplicate deliveries instead of returning 409 Conflict when the same normalized hook sessionKey is already active.

## Best Possible Solution

Keep this issue open and implement an explicit hook ingress concurrency policy for direct /hooks/agent and mapped agent hooks. The likely fix should check the normalized dispatch sessionKey before creating the isolated run, return 409 Conflict when that sessionKey already has active or queued hook work according to the chosen contract, and add focused tests for direct hooks, mapped hooks, idempotency-key replay, target-agent rebinding, and the existing downstream session-lane serialization behavior.
