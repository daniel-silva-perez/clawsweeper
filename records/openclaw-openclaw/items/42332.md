## Summary

Keep open. Current main still accepts direct and mapped hook agent deliveries with 200 after optional idempotency-key replay, then dispatches an asynchronous isolated cron-style run. Work is serialized later by the embedded runner's per-session lane, but webhook ingress does not detect an already active normalized hook sessionKey and return 409 Conflict.

## What This Changes

The issue requests a webhook ingress guard that returns 409 Conflict when a hook delivery targets a sessionKey that already has active work.

## Best Possible Solution

Add a backward-compatible hook ingress concurrency policy for direct and mapped agent hooks. The guard should evaluate the normalized dispatch sessionKey after agent rebinding and session-prefix checks, preserve idempotency replay semantics, return 409 only under the chosen policy, and document the config/API contract.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. Send two non-idempotent POST /hooks/agent or mapped hook requests with the same allowed sessionKey while the first run is still active; current code accepts and queues both instead of returning 409.

## Solution Assessment

Is this the best way to solve the issue? Unclear as a default global behavior, because blanket 409 responses could be breaking. The best maintainable direction is an explicit, documented conflict-on-active hook policy that defaults to current accept-and-queue behavior.

## Repair Work Prompt

Implement a sessionKey-based concurrency guard for webhook agent deliveries. Add a backward-compatible hook ingress policy for direct /hooks/agent and hooks.mappings[] agent actions, defaulting to the current accept-and-queue behavior. Prefer a config/mapping-owned option such as onActiveSession: "accept" | "conflict" or equivalent; do not let untrusted request bodies silently opt themselves into privileged routing behavior unless the existing hook policy explicitly allows it. Resolve the normalized dispatch sessionKey after allowRequestSessionKey, allowedSessionKeyPrefixes, and target-agent rebinding. Apply idempotency replay before active-session conflict checks so exact retries still return the cached 200 run id. Track hook runs from before async dispatch until completion/error, and preserve wake hooks, cron scheduler overlap behavior, downstream session-lane serialization, existing hook security checks, and default 200 behavior. Add tests for direct hooks, mapped hooks, idempotency interaction, target-agent rebinding, completion cleanup, and queued/active conflict semantics.
