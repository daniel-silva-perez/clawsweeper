## Summary

Keep open. Current main accepts direct and mapped hook agent deliveries with `200` and dispatches them asynchronously into isolated cron-style runs. The same normalized hook `sessionKey` is serialized later by the embedded runner's per-session lane, and exact retries can be deduped with an idempotency key, but there is no hook ingress guard that detects an already active `sessionKey` and returns `409 Conflict`. Related #27193 was closed unmerged and addressed persistent hook session reuse, while #54325 was a cron-overlap issue, so neither supersedes the webhook delivery guard request.

## Best Possible Solution

Keep the issue open and implement an explicit, probably opt-in hook ingress policy for direct and mapped agent hooks. The fix should check the normalized dispatch `sessionKey` before creating the isolated run, return `409 Conflict` when that key already has active or queued hook work as defined by the product contract, and add tests for `/hooks/agent`, mapped hooks, idempotency-key interaction, target-agent session rebinding, and the existing session-lane serialization behavior.
