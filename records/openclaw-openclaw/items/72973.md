## Summary

Keep open. Current main still passes raw Content-Disposition and URL-derived basenames into outbound media labels, and this PR is a focused implementation with tests rather than something already shipped or obsolete. The remaining decision is maintainer review of the Unicode strip policy and coordination with the overlapping filename-decoder work in #71517.

## What This Changes

The PR adds a shared outbound filename control-character stripping helper, wires it into remote fetch and web media filename extraction, and adds unit plus fetch regression coverage.

## Best Possible Solution

Land a shared media-boundary filename display hardening fix after maintainer approval of the Unicode strip set, keeping the logic centralized in media extraction rather than duplicating it in channel adapters.

## Reproduction Assessment

Yes. Current main can be reproduced by feeding `fetchRemoteMedia` a mocked `Content-Disposition: filename*=UTF-8''report%E2%80%AEgpj%E2%80%8B.exe`; source inspection shows that value currently flows through `path.basename(decodeURIComponent(...))` without stripping.

## Solution Assessment

Mostly yes. Centralizing the strip at `fetch.ts` and `web-media.ts` is the narrowest maintainable boundary for Telegram, Discord, Matrix, and WhatsApp consumers, but the exact bidi-mark policy should be approved by a maintainer before merge.
