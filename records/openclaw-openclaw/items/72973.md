## Summary

Keep this PR open. Current main still has the raw outbound filename boundary the PR targets, and the branch is a plausible centralized hardening change; it needs maintainer review of the Unicode strip-set policy and coordination with the overlapping decoder work rather than conservative cleanup closure.

## What This Changes

The branch adds a shared outbound filename control-character stripper, applies it to Content-Disposition parsing and local/web media basename extraction, and adds helper plus fetch-path regression tests.

## Best Possible Solution

Adopt one shared outbound filename normalization/decoding path at the media extraction boundary, with an explicit tested Unicode policy, and fold this hardening into the shared decoder if #71517 becomes the canonical implementation.

## Reproduction Assessment

Yes. A focused `fetchRemoteMedia` mock with RFC 5987 `filename*=` containing RLO and ZWSP reproduces the current-main leak through `decodeURIComponent` plus `path.basename`; the read-only Node check confirmed the exact characters remain.

## Solution Assessment

Mostly yes. Centralizing the strip at shared media extraction sites is narrower and more maintainable than duplicating logic in Telegram, Discord, Matrix, and WhatsApp adapters, but the exact bidi-mark policy should be decided before merge.
