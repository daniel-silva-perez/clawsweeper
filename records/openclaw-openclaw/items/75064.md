## Summary

Keep this PR open. Current main still has module-local memory plugin state while Memory Wiki bridge/status read public artifacts through the SDK memory-host-core path, so the linked split-singleton bug remains plausible and source-backed. The patch is a narrow draft fix with relevant mirrored-runtime tests and no blocking review finding, but it still needs normal maintainer review and CI/Testbox validation before merge.

## What This Changes

The branch stores memory plugin state in `createPluginRuntimeStore` and adds mirrored-runtime regression coverage for memory-state plus Memory Wiki bridge/status artifact counting.

## Best Possible Solution

Merge a narrow, backward-compatible fix that makes the active memory capability registry visible across canonical and mirrored runtime module instances, with regression coverage for Memory Wiki bridge/status and without changing the public memory plugin API. A later host-owned capability service can be tracked separately if maintainers want to remove the remaining global-state compatibility layer.

## Reproduction Assessment

Yes. The linked issue gives a concrete 2026.4.27 path using `memory-core` plus `memory-wiki` bridge where `wiki.bridge.import` and `wiki status` report zero artifacts, and current main source shows a module-local registry that can split across runtime mirrors.

## Solution Assessment

Yes for the immediate regression. Reusing `createPluginRuntimeStore` is the narrowest existing mechanism for sharing in-process state across duplicate module instances while preserving the existing memory plugin API; a host-owned capability service is a broader follow-up, not required for this small fix.
