## Summary

Keep this PR open. Current main still has the module-local memory capability registry that can split across runtime mirrors, and this draft PR is a narrow, plausible fix with no concrete blocking defect found in review; it still needs maintainer review plus CI/Testbox validation before merge.

## What This Changes

The PR stores memory plugin registry state through `createPluginRuntimeStore` and adds mirrored-runtime regression coverage for memory-state plus Memory Wiki bridge/status artifact counting.

## Best Possible Solution

Merge a narrow, backward-compatible fix that makes the active memory capability registry visible across canonical and mirrored runtime module instances, with regression coverage for Memory Wiki bridge import/status and without changing the public memory plugin API. Track a broader host-owned capability service separately if maintainers want to remove this global-state compatibility layer later.

## Reproduction Assessment

Yes. The linked issue gives a concrete 2026.4.27 Gateway path using `memory-core` plus `memory-wiki` bridge where `wiki.bridge.import` and `wiki status` report zero artifacts, and current main source shows the module-local registry that can split across runtime mirrors.

## Solution Assessment

Yes for the immediate regression. Reusing `createPluginRuntimeStore` is the narrowest existing mechanism for in-process duplicate-module state sharing while preserving the memory plugin SDK API; a host-owned capability service is a broader follow-up, not required for this small fix.
