## What This Changes

The PR moves memory plugin registry state from a module-local object into `createPluginRuntimeStore` and adds mirrored-runtime regression tests for memory-state, Memory Wiki bridge import, and Memory Wiki status artifact counts.

## Best Possible Solution

Land a narrow, backward-compatible fix that makes the active memory capability registry visible across canonical and mirrored runtime module instances, with regression coverage for Memory Wiki bridge/status and without changing the public memory plugin API. Track a fuller host-owned capability service separately if maintainers want to replace the compatibility-layer global store later.

## Reproduction Assessment

Yes. The linked issue gives a concrete 2026.4.27 Gateway path using `memory-core` plus `memory-wiki` bridge where import/status report zero artifacts, and current main source still shows a module-local memory registry that can split across runtime mirrors.

## Solution Assessment

Yes for the immediate regression. Reusing `createPluginRuntimeStore` is the narrowest existing in-process duplicate-module sharing mechanism and preserves the current memory plugin SDK API; a canonical host-owned capability service is a broader follow-up, not required for this small fix.
