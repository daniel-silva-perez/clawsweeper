## Summary

Keep open. Current main still lacks a configurable model API request delay, and #49221 remains the matching open implementation candidate. The inspected config types, strict schema, docs, guarded fetch path, embedded run setup, and compaction setup cover timeouts/transport/retry handling but do not expose or enforce request-start pacing.

## What This Changes

Add a configurable delay between consecutive model API request starts for users hitting strict provider rate limits, especially GLM-style providers.

## Best Possible Solution

Keep this issue open and review #49221, or replace it with an equivalent core implementation that adds validated global and per-provider model request-delay config, applies it to normal embedded runs and compaction, documents the public config surface, and adds focused tests for provider-scoped spacing, zero disablement, and override precedence.

## Reproduction Assessment

Not applicable as a runtime bug reproduction, but yes for source-level verification: current main can be checked by inspecting the config schema/types, docs, provider fetch path, and embedded/compaction stream setup, all of which lack request pacing.

## Solution Assessment

Unclear until #49221 is reviewed. The proposed global plus per-provider requestDelayMs direction matches the maintainer boundary, but the final fix should prove provider-id scoping, zero overrides, shared-session behavior, and abortable waits.
