## Summary

Keep this issue open. Current main still has the core token-burn shape: built-in compaction retries each `generateSummary` call up to three times, `summarizeWithFallback()` converts a failed chunk summary into a fallback string, and `summarizeInStages()` then continues through later split chunks and the final merge. The proposed circuit-breaker PRs referenced from the issue were closed unmerged, and I did not find an equivalent guard on main.

## Best Possible Solution

Implement a bounded failure guard in the built-in compaction summarization path, with focused tests for multi-split `summarizeInStages()` behavior under persistent non-timeout model failures. The guard should stop later split/merge LLM attempts after a small consecutive-failure budget and return the previous summary or existing structured fallback, while keeping plugin provider behavior, abort handling, timeout handling, and quality-guard retries explicit.

## Risks / Open Questions

- A fix should scope any breaker carefully, likely by session and summarizer model/provider, so one bad compaction target does not suppress unrelated healthy compactions.
- The fix needs to preserve existing abort/timeout handling and the compaction-provider fallback contract; aborts and timeouts currently propagate differently from ordinary model errors.
