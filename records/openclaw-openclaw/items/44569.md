## Summary

Keep this PR open, but do not merge it as-is. Current main still drops config env.vars from sandbox exec, so the contribution is not obsolete; the proposed patch bypasses runtime env guards used by applyConfigEnvVars and changes the sandbox secret/env exposure contract without a clear docs/security decision.

## What This Changes

The PR threads collected config env.vars through exec defaults into buildSandboxEnv so sandbox exec commands receive config-level environment variables beneath sandbox env and per-call env overrides.

## Best Possible Solution

Revise this into a guarded sandbox env propagation change: build an effective config env map that preserves applyConfigEnvVars semantics, layer it below sandbox/backend env and per-call params, add regression coverage for forwarding, precedence, blocked keys, existing env precedence, and unresolved placeholders, then update public sandbox/env docs if maintainers decide env.vars should be part of the sandbox exec contract.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure env.vars and use sandbox exec; current main’s sandbox path constructs its env without config env vars, while gateway exec inherits process.env after config loading.

## Solution Assessment

No. The proposed direction addresses the right gap, but directly forwarding collectConfigRuntimeEnvVars is not the safest implementation because it skips runtime guards and changes sandbox exposure semantics; a guarded effective-env helper or equivalent applyConfigEnvVars-based path is the narrower maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve runtime guards for forwarded env vars:** `src/agents/pi-tools.ts:408`
  - body: This forwards collectConfigRuntimeEnvVars(options?.config) directly into sandbox exec defaults. That helper only normalizes and filters names; it does not skip keys that already exist in the runtime env or values that still contain unresolved ${VAR}, both of which applyConfigEnvVars intentionally skips before gateway/host exec inherits env. Sandbox exec can therefore receive stale config values or literal placeholders that the non-sandbox path would avoid.
  - confidence: 0.93
