## Summary

Keep this PR open. Current main at 9bb1e59447a5fdf2c0164672df6c748188b26442 still does not thread config env.vars into sandbox exec environments: sandbox exec builds a clean env from PATH/HOME, sandbox env, and per-call params only. The PR targets a real unresolved bug, but the submitted direct forwarding still needs maintainer/security revision because it changes sandbox secret exposure and does not mirror applyConfigEnvVars guards for existing env precedence or unresolved placeholders.

## Best Possible Solution

Keep this PR open and revise it into a guarded sandbox env propagation patch. The implementation should derive an effective config env map that mirrors applyConfigEnvVars semantics, layer it below sandbox/backend env and per-call params, add regression tests for forwarding, precedence, blocked keys, existing env precedence, and unresolved placeholders, and update https://docs.openclaw.ai/gateway/sandboxing if maintainers decide config env.vars is part of the sandbox exec contract.

## Risks / Open Questions

- Merging the PR as written could make sandbox exec receive config values that gateway/host exec intentionally skips when an existing runtime env value already exists.
- Merging the PR as written could forward literal unresolved ${VAR} placeholders into sandbox commands, diverging from applyConfigEnvVars behavior.
- Forwarding config env.vars into sandbox exec expands configured secret exposure to sandboxed commands; that may be desired, but current docs describe sandbox env isolation.
- The PR body's sanitizer rationale overstates exec-time coverage: Docker create sanitizes configured container env, but docker exec argument construction forwards non-PATH env values it is handed.
- Adjacent open work in #40290 and #57427 should be coordinated so env.vars, sandbox.docker.env, sanitizer bypasses, and OpenShell placeholder semantics do not drift.
