## Summary

Keep open. The checked-out main still starts model-pricing refresh during initial Gateway runtime setup, while this PR remains an open, focused implementation to defer that optional work until post-attach and sidecar readiness. I did not find a blocking correctness or security issue in the proposed diff; the remaining action is maintainer review, CI/rebase hygiene, and landing or adapting the PR.

## What This Changes

This PR moves model-pricing refresh startup from `startGatewayRuntimeServices()` to `activateGatewayScheduledServices()`, gates scheduled activation on post-attach plus sidecar readiness, stores the pricing stop handle on runtime state, preserves plugin lookup metadata, and updates gateway runtime-service tests.

## Best Possible Solution

Review and land or adapt this focused lifecycle change so pricing stays enabled by default but cannot start until post-attach work and sidecars/channels are ready. Keep the broader Windows/macOS runtime degradation issues open for deeper network, timer, and fetch-timeout investigation.

## Reproduction Assessment

Yes for the startup-order behavior: current main statically starts pricing from `startGatewayRuntimeServices()` before `startListening()` and post-attach sidecar readiness. The broader Windows/macOS network degradation was not live-reproduced in this read-only review.

## Solution Assessment

Yes, this appears to be the narrowest maintainable fix for the PR’s stated goal: move optional pricing refresh into the existing scheduled-services activation path and gate it on readiness, without adding another disable switch or changing fetch timeouts. Broader runtime degradation should remain tracked separately.
