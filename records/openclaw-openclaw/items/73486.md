## Summary

Keep this PR open. Current main still starts optional model-pricing refresh during initial Gateway runtime setup, while this PR is a focused lifecycle fix that moves that refresh behind post-attach and sidecar readiness; I found no blocking functional or security issue in the current PR diff.

## What This Changes

The PR moves Gateway model-pricing refresh out of initial runtime service startup and into scheduled-service activation gated on post-attach and sidecar readiness, with updated lifecycle tests.

## Best Possible Solution

Land or adapt this PR after maintainer review and required checks, keeping pricing enabled by default but starting its optional remote catalog refresh only after Gateway post-attach work and sidecars/channels reach the ready path. Leave the broader pricing timeout/cache/circuit-breaker work to the related follow-up PRs and #73323.

## Reproduction Assessment

Yes. The narrow lifecycle issue is reproducible from current-main source and tests: initial runtime setup still starts model pricing before the post-attach scheduled-services phase, and the test suite asserts that behavior. The broader user-facing timeout degradation has live reports in #73323 and related issues, but this PR only needs the startup-order proof.

## Solution Assessment

Yes. This is a narrow, maintainable fix for the startup-order problem: it keeps pricing behavior and plugin lookup metadata intact while moving the optional refresh behind sidecar readiness. Timeout reduction, disk cache, and circuit breaker behavior are safer as separate follow-up work such as #75067.
