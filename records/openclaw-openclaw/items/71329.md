## Summary

Keep this PR open. Current main still gates Slack lifecycle status reactions on the ack-reaction path before dispatch can enable the controller, while the documented `messages.statusReactions.enabled` flag says Slack should support lifecycle reactions; the PR is a focused implementation candidate with one small test assertion gap and a rebase concern against newer main.

## What This Changes

The PR changes Slack message preparation so `messages.statusReactions.enabled === true` can create the lifecycle status-reaction handle without requiring the ack-reaction gate, and adds prepare tests for explicit enable and explicit disable behavior.

## Best Possible Solution

Adapt and review this narrow Slack change on current main: let an explicit `messages.statusReactions.enabled === true` activate lifecycle reactions independently of the ack-reaction scope gate, keep `enabled: false` as an explicit disable, preserve message-tool-only suppression, and tighten the regression tests around both explicit enable and explicit disable behavior.

## Reproduction Assessment

Yes. A focused current-main check is a timestamped Slack DM with `messages.statusReactions.enabled: true` and the default `group-mentions` ack scope; `prepareSlackMessage` still produces no `ackReactionPromise`, and dispatch requires that handle before lifecycle reactions run.

## Solution Assessment

Yes, with rebase adjustments. The PR's activation approach is the narrowest maintainable fix for the documented flag, but it should be applied on top of current main's `shouldSendAckReaction` logic and its explicit-disable test should assert the fallback ack promise.
