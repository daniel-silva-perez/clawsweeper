## Summary

Keep open. Current main still has the Gmail watcher bootstrap on the awaited sidecar startup path, and the issue discussion now has a reporter-confirmed bisection showing that Gmail watcher startup, not model-pricing, caused the original multi-minute delay.

## What This Changes

Request that Gateway startup stop blocking on hook/Gmail watcher initialization, especially network-dependent watcher setup, while keeping pricing/bootstrap work lazy or non-blocking.

## Best Possible Solution

Keep this issue open and fix the Gmail-owned startup path: start the Gmail watcher bootstrap in the background from sidecar startup, make that background state visible in logs/startup trace, preserve the existing opt-out and error handling, and leave broader channel/Bonjour startup stalls to their narrower follow-up.

## Reproduction Assessment

Yes. Configure `hooks.enabled=true` with `hooks.gmail.account`, make `gog gmail watch start` slow or timing out, and start the Gateway through the normal CLI path; current main awaits that watcher before channel startup and `gateway ready`.

## Solution Assessment

Yes for the localized Gmail bottleneck: backgrounding `startGmailWatcherWithLogs` from `sidecars.gmail-watch` is the narrowest maintainable fix. A broad hook timeout/skip flag should not be the first change unless multiple hook owners need a shared startup contract.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/61278. Current main still awaits Gmail watcher startup in `src/gateway/server-startup-post-attach.ts`, so a slow `gog gmail watch start` or Tailscale setup can delay channel startup and `gateway ready` by up to about 120 seconds. Make the Gmail watcher bootstrap non-blocking from the sidecar startup path: preserve `OPENCLAW_SKIP_GMAIL_WATCHER`, preserve existing `startGmailWatcherWithLogs` success/warn/error logging, add an explicit log or trace signal that Gmail watcher bootstrap is continuing in the background, and do not change model-pricing, Bonjour, or general channel startup behavior in this PR. Add a regression test in `src/gateway/server-startup-post-attach.test.ts` with `hooks.enabled=true` and `hooks.gmail.account` where mocked `startGmailWatcherWithLogs` stays pending, then assert channel startup/sidecar completion is not delayed. Update `CHANGELOG.md` with a user-facing fix entry crediting the reporter(s), and update docs only if the visible Gmail watcher readiness semantics/log wording need explanation.
