## Summary

Keep open. The issue discussion corrected the initial model-pricing theory and localized the delay to Gmail watcher startup. Current main still awaits Gmail watcher bootstrap before channel startup/sidecar readiness, so the reported startup-blocking path remains actionable.

## Best Possible Solution

Keep this open and fix the Gmail watcher owner path: start Gmail watcher bootstrap in the background from gateway sidecar startup, log that it is continuing asynchronously, preserve existing skip/error handling, and add a regression test that injects a slow watcher and proves channel startup/readiness is not delayed.
