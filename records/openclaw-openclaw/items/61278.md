## Summary

Keep this issue open. The provided discussion already corrected the initial model-pricing theory and localized the real bottleneck to Gmail watcher startup. Current main still awaits Gmail watcher bootstrap before channel startup, so the central user problem is still actionable rather than implemented or obsolete.

## Best Possible Solution

Keep this open and fix the Gmail watcher owner path: move Gmail watcher bootstrap out of the awaited sidecar/channel critical path, log that it is continuing in the background, preserve the existing skip/error handling, and add a regression test that injects a slow `startGmailWatcherWithLogs` and proves channel startup/readiness is not delayed by it.

## Evidence
