## Summary

Keep this PR open. Current main already covers the config method scope-enforcement part through reserved `config.*` admin handling, but it does not implement the PR's central HMAC sidecar signing/verification, integrity-warning snapshot field, reload rejection path, or HMAC security-audit finding. The PR is also security-sensitive and not merge-ready based on the provided review comments plus the diff security pass.

## What This Changes

This PR adds config HMAC sidecar signing/verification and integrity warnings, marks config-mutating gateway methods as admin scoped with inline handler checks, adds config audit/security-audit docs and findings, and adds a CodexBar onboarding option.

## Best Possible Solution

A maintainer/security-owned version should first settle the config-integrity threat model, then land a narrow core implementation with an explicit migration/bootstrap story, injected dependencies, consistent write-failure semantics, focused tamper/reload/upgrade tests, and docs. The existing reserved `config.*` admin-scope enforcement should be preserved, and unrelated onboarding changes should be split out.

## Risks / Open Questions

- The PR states that details were reported to the security contact, so maintainers should treat the design as security-sensitive rather than routine cleanup.
- The HMAC bootstrap behavior in the PR can reject existing valid configs that have a gateway token but no `.sig` file yet.
- The PR signs sidecars best-effort but enforces missing signatures on read, so a signature write failure can become a later startup/reload failure.
- The token lookup bypasses injected filesystem/home/env dependencies, making tests likely to miss the integrity path and making the implementation inconsistent with config IO dependency contracts.
- The PR mixes unrelated CodexBar onboarding changes into a config hardening change, increasing review scope and product risk.
