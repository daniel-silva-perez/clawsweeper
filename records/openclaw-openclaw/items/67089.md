## Summary

Keep open. Current main already covers the admin-scope portion, but the central HMAC integrity/reload/audit behavior is still absent, and the PR as submitted has compilation plus security-design blockers that require maintainer/security review.

## What This Changes

The PR adds config HMAC sidecar signing and verification, config reload and security-audit integration, explicit config method admin-scope checks, config-protection docs, and a CodexBar onboarding option.

## Best Possible Solution

A mergeable path should be a maintainer/security-owned config-integrity design that keeps the existing reserved `config.*` admin-scope behavior, adds a backwards-compatible signature bootstrap or migration plan, makes write and read failure semantics consistent, uses injected dependencies, splits unrelated onboarding work, and includes focused tamper, reload, upgrade, and write-failure tests.

## Reproduction Assessment

Yes for the current PR blockers: the diff itself shows the parse-breaking `src/config/io.ts` change, and the upgrade lockout follows directly from the added `no-sig && suspicious` rejection path for established configs with a gateway token. I did not run tests because this review was explicitly read-only.

## Solution Assessment

No. The proposed direction is plausible security work, but this implementation is not the narrowest maintainable fix until the bootstrap/migration story, failure semantics, dependency injection, docs, and unrelated onboarding change are resolved.

  - body: The options object passed into `io.writeConfigFile` is opened in this call but the changed line closes only the function call. TypeScript will fail to parse this file, so the branch cannot build.

  - body: This path rejects established configs when the `.sig` file is missing. Existing installations with a normal config and a gateway token have no sidecar yet, so first boot after upgrade can drop a valid config by returning `{}`.

  - body: The sidecar writer swallows every write error, but later reads can reject missing or empty signatures. A successful config write followed by a failed `.sig` write can therefore become a self-inflicted startup failure with no immediate signal.
  - confidence: 0.91

  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The patch is security-sensitive and introduces concrete availability and assurance risks in the proposed config integrity enforcement.

Concerns:

- **[high] Missing signatures can lock out existing configs:** `src/config/io.ts:1109`
  - body: The PR treats missing signatures on established configs as suspicious and rejects them, but existing installations will not have generated `.sig` sidecars before this feature lands.
  - confidence: 0.94
- **[medium] Sidecar write failures fail open before reads fail closed:** `src/config/io.hmac-integrity.ts:82`
  - body: Signature write errors are silently ignored while read paths can reject missing signatures later, so an I/O error during persistence can turn into a later startup failure.
  - confidence: 0.91
- **[low] HMAC key resolution bypasses injected dependencies:** `src/config/io.hmac-integrity.ts:23`
  - body: Direct `os.homedir()` and `process.env` reads make tests and alternate homes use a different token source from the config IO dependency graph, weakening test coverage for the new security path.
  - confidence: 0.88
