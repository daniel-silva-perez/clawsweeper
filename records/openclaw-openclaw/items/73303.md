## Summary

Keep open. Current main still allows managed Gateway restarts to wait up to the default 300000ms drain budget while active tasks, embedded runs, or bundled runtime-deps installs remain active, which matches the reported multi-minute LaunchAgent pre-start gap. The latest runtime-deps performance work reduces one possible source of startup/restart pressure, but it does not change the restart lifecycle policy behind this issue.

## What This Changes

The issue reports macOS LaunchAgent Gateway restarts on v2026.4.26 taking several minutes before the new process starts and asks whether shutdown, plugin/channel teardown, or a known regression is responsible.

## Best Possible Solution

Define the operator restart lifecycle policy explicitly, then implement a clear safe-drain versus force-restart path with progress logging and documented timeout semantics so managed restarts are prompt without silently discarding active reply delivery guarantees.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the likely mechanism: current code and fake-timer tests show restart requests can wait until the 300-second default expires while active work remains, and SIGTERM with a service-manager restart intent enters that draining restart path. The exact reporter host and plugin mix were not live-reproduced.

## Solution Assessment

Is this the best way to solve the issue? No. The current bounded drain prevents an infinite hang, but it still permits the multi-minute delay reported here and needs a maintainer decision on the default restart policy or an operator-controlled force path.
