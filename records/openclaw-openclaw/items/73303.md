## Summary

Keep open. Current main still allows Gateway restarts to wait on active tasks, embedded runs, and bundled runtime dependency installs before `server.close`, with a default cap of 300 seconds; that matches the reported multi-minute pre-start gap and is only partially mitigated on main.

## What This Changes

The issue requests investigation and mitigation for macOS LaunchAgent Gateway restarts that can appear stuck for several minutes before the replacement process starts.

## Best Possible Solution

Define the Gateway lifecycle policy for operator restarts: either make manual/service-manager restarts fast by default with explicit cancellation/progress semantics, or expose a clear safe-drain versus force-restart path while preserving active reply delivery guarantees.

## Reproduction Assessment

Yes for the likely mechanism: current code and fake-timer tests show restart requests can wait until the 300-second default deferral expires when active work remains. The exact launchd host and plugin mix from the report were not live-reproduced.

## Solution Assessment

No. The current main behavior is a partial mitigation and a documented knob, but it still permits the multi-minute delay the issue reports and needs a maintainer decision on restart semantics.
