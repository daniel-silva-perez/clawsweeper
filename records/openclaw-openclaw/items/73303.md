## Summary

Keep open. Current main still allows managed Gateway restarts to wait on active tasks, embedded runs, and bundled runtime-dependency installs with a 300000ms default cap before the old process exits, which matches the reported multi-minute macOS LaunchAgent pre-start delay. The v2026.4.29 change bounds previously unbounded waits and a newer runtime-deps fix reduces one possible stall source, but neither changes the default restart-drain policy enough to close this operational issue.

## What This Changes

Investigate and fix macOS LaunchAgent Gateway restarts that can wait several minutes between SIGTERM and the replacement Gateway process starting, including shutdown drain, plugin/channel teardown, and workaround status.

## Best Possible Solution

Define the Gateway restart lifecycle contract for operator/service-manager restarts, then implement a clear fast-restart or force-restart path with progress/cancellation semantics while preserving deliberate safe-drain behavior where active reply delivery must win.

## Reproduction Assessment

Yes for the likely mechanism: current source and fake-timer tests show restart requests can wait until the 300-second default expires while active work remains. The exact macOS LaunchAgent environment and external plugin mix from the report were not live-reproduced.

## Solution Assessment

No. The current v2026.4.29 behavior is a bounded mitigation and configurable drain knob, but it still permits the multi-minute delay reported here; the narrower maintainable fix is an explicit restart policy rather than treating this as already implemented.
