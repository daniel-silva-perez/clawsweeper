## Summary

Keep open. Current main still has an intentional restart drain path that can wait up to the default 300 seconds before closing the old Gateway process and letting a LaunchAgent restart create the replacement process, which matches the reported multi-minute pre-start gap. The discussion has multiple independent “same issue” reports, and the newer startup/reload fixes and configuration knob are related mitigations rather than proof that this user-facing restart delay is resolved.

## What This Changes

The issue asks OpenClaw to investigate and reduce macOS LaunchAgent Gateway restarts that can appear hung for roughly 3-4 minutes on v2026.4.26, including shutdown drain, plugin/channel teardown, and possible workarounds.

## Best Possible Solution

Keep this open for a Gateway lifecycle policy decision: manual restarts should either expose clear drain progress and a safe fast-restart option, or use a shorter/default drain policy for operator-requested restarts while preserving protection for active replies, embedded runs, and runtime dependency installs.

## Reproduction Assessment

Yes for the likely mechanism: current source and tests show a restart with active work can wait up to the configured default 300-second drain before the old Gateway closes. The exact reporter machine was not live-tested.

## Solution Assessment

No. The current five-minute bound and deferralTimeoutMs knob prevent unbounded waits, but they are not the narrowest complete solution for users seeing a 3-4 minute LaunchAgent restart delay by default.
