## Summary

Keep open. Current main still allows a managed Gateway restart to wait for the default 300000ms drain budget before the old process closes, which can gate launchd's replacement process and matches the reported multi-minute pre-start gap.

## What This Changes

The issue reports macOS LaunchAgent Gateway restarts on v2026.4.26 taking several minutes before the new process starts and asks whether shutdown, plugin/channel teardown, or a known regression is responsible.

## Best Possible Solution

Define an explicit operator restart policy with a clear safe-drain path and a fast force-restart path, then document timeout/progress semantics so managed restarts are prompt without silently discarding active work guarantees.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the likely mechanism: current code and fake-timer tests show a managed restart can wait until the 300-second default drain budget expires while active work remains. The exact reporter machine and plugin mix were not live-reproduced.

## Solution Assessment

Is this the best way to solve the issue? No. The current bounded drain avoids an infinite hang, but it still permits the multi-minute delay reported here and needs a maintainer decision on default restart behavior or an operator-controlled force path.
