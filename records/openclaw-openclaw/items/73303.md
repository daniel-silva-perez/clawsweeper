## Summary

Keep open. Current main still allows a managed Gateway restart to wait up to the default 300000ms drain budget before the old process exits, which matches the reported multi-minute LaunchAgent pre-start delay. Recent runtime-deps fixes reduce one possible source of waits, but they do not resolve the restart lifecycle policy behind this issue.

## What This Changes

Investigate macOS LaunchAgent Gateway restarts on v2026.4.24-.26 that can spend several minutes between SIGTERM and the next Gateway startup, including shutdown drains and plugin/channel teardown latency.

## Best Possible Solution

Define the Gateway lifecycle policy for operator/service-manager restarts, then implement an explicit safe-drain versus force-restart path with progress logging and documented timeout semantics so operators can restart promptly without silently losing active reply delivery guarantees.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the likely mechanism: current code and fake-timer tests show restart requests can wait until the 300-second default expires while active work remains. The exact LaunchAgent host and external plugin mix from the report were not live-reproduced.

## Solution Assessment

Is this the best way to solve the issue? No. The current bounded drain prevents an infinite hang, but it still permits the multi-minute delay reported here and needs a clearer restart policy or operator-controlled force path.
