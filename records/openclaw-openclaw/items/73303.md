## Summary

Keep open. Current main still intentionally permits a Gateway restart to wait on active tasks, embedded runs, pending replies, and bundled runtime dependency installs with a 300000ms default cap before the old process exits, so the reported multi-minute LaunchAgent pre-start delay remains a valid lifecycle-policy issue. A newer runtime-deps fix reduces one possible source of repeated install waits, but it does not change the default restart-drain behavior that matches the main symptom.

## What This Changes

The issue requests investigation and mitigation for v2026.4.26 macOS LaunchAgent Gateway restarts that can wait several minutes before a new process starts.

## Best Possible Solution

Define the Gateway lifecycle policy for operator/service-manager restarts, then implement either fast restarts with explicit cancellation/progress semantics or a clear safe-drain versus force-restart path that preserves active reply delivery guarantees where intended.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the likely restart-drain mechanism: current code and fake-timer tests show restart requests can wait until the 300-second default expires while active work remains. The exact LaunchAgent machine and external plugin mix from the report were not live-reproduced.

## Solution Assessment

Is this the best way to solve the issue? No. The current behavior is a bounded mitigation and documented knob, but it still permits the multi-minute delay reported here and needs an explicit maintainer decision on restart semantics.
