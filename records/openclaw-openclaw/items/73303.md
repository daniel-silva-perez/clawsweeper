## Summary

Keep open. Current main still has an intentional restart drain path that can wait up to 300 seconds before the Gateway closes and lets launchd start a fresh process, which matches the reported 3-4 minute pre-start gap. The code also has bounded channel teardown and documented configuration, but the central user pain is not resolved by current main.

## What This Changes

The issue asks OpenClaw to investigate and mitigate macOS LaunchAgent Gateway restarts in v2026.4.26 that can delay fresh startup by about 3-4 minutes, with possible channel/plugin teardown or startup-latency involvement.

## Best Possible Solution

Keep this open for a Gateway lifecycle decision: manual restarts should either surface clear drain progress and a safe fast-restart path, or use a different default drain policy than the current 300-second wait while preserving protection for active replies, embedded runs, and runtime dependency installs. The v2026.4.27 startup improvements should be treated as related evidence, not as proof that this specific restart-delay issue is fixed.

## Reproduction Assessment

Yes for the likely mechanism: current tests and code show that a restart with active tasks, embedded runs, or runtime dependency installs can wait up to the configured deferral timeout before closing the old Gateway process. The exact macOS environment from the report was not live-reproduced.

## Solution Assessment

No. The current configuration knob and startup fixes help diagnosis or related latency, but they are not the narrowest complete solution for users seeing a default 3-4 minute LaunchAgent restart delay.
