## Summary

Keep open. Current `main` still allows a restart to wait up to five minutes for active Gateway work before `server.close()` and process replacement, which matches the reported 3-4 minute LaunchAgent pre-start gap. Recent changes bound indefinite waits and improve adjacent startup/runtime-deps paths, but they do not resolve the central restart-latency policy issue.

## What This Changes

The issue asks to investigate and reduce macOS LaunchAgent Gateway restarts that can spend several minutes between `SIGTERM` and fresh startup, including shutdown, plugin teardown, and regression/workaround causes.

## Best Possible Solution

Keep this open for a Gateway lifecycle policy decision: manual or operator-requested restarts should either have a safe fast-restart mode or clearer drain progress with a shorter default, while still protecting active replies, embedded runs, and runtime dependency installs from avoidable loss.

## Reproduction Assessment

Yes for the likely mechanism: current code and tests show that restarts with active tasks, embedded runs, or runtime dependency installs can wait up to the configured deferral timeout before the old Gateway closes. The exact reported macOS LaunchAgent setup was not live-reproduced.

## Solution Assessment

No. The current five-minute bound is safer than an indefinite stall, but it is not a complete solution for users seeing a default 3-4 minute restart gap; the narrower maintainable fix needs an explicit restart policy or UX decision.
