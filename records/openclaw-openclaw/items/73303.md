## Summary

Keep open. Current main still permits manual or service-manager Gateway restarts to wait up to the default 300-second restart deferral before `server.close` runs, which matches the reported multi-minute pre-start gap; the latest shipped behavior bounds the wait but does not make operator restarts fast or clearly distinguish safe-drain from force-restart semantics.

## What This Changes

The issue asks OpenClaw to investigate and fix macOS LaunchAgent Gateway restarts that can linger for several minutes before the replacement process starts.

## Best Possible Solution

Keep this open for a Gateway lifecycle policy decision: operator/service-manager restarts should either become fast by default with explicit cancellation and progress reporting, or expose a documented safe-drain versus force-restart path that preserves reply delivery and active-run protection where intended.

## Reproduction Assessment

Yes for the likely mechanism: current code and tests show SIGTERM-with-restart-intent and SIGUSR1/config restarts can wait on active work until the default 300-second deferral. The exact macOS LaunchAgent environment from the report was not live-reproduced.

## Solution Assessment

No. The current 300-second bounded default and `deferralTimeoutMs` knob are a mitigation and diagnostic surface, but they are not the narrowest complete solution for users who experience manual restarts as stuck for several minutes.
