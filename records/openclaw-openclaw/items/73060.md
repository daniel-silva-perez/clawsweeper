## Summary

Keep open. Current main still drops explicit Mattermost `threadId` on direct message-tool `send` handling, and the PR is a narrow, valid fix for that path; the only concrete issue found is the stale nearby fallback comment.

## What This Changes

The PR updates Mattermost message-tool send handling to use `threadId` as a fallback between explicit `replyToId` and legacy `replyTo`, with regression tests for trimming and precedence.

## Best Possible Solution

Land this focused Mattermost plugin fix after updating the stale fallback comment, while keeping #67784 open for the remaining media-path and local-root message-tool work.

## Reproduction Assessment

Yes. Call `mattermostPlugin.actions.handleAction` with `action: "send"`, a Mattermost channel target, and `threadId` but no `replyToId`; current main forwards no `replyToId` to `sendMessageMattermost`, so no Mattermost `rootId` is set.

## Solution Assessment

Yes, with a small comment cleanup. The `replyToId ?? threadId ?? replyTo` order is the narrow maintainable fix because it matches existing Mattermost outbound helpers while preserving explicit `replyToId` precedence.
