## Summary

Keep open. Current main still drops `threadId` on the Mattermost direct message-tool `send` path, while this PR makes the narrow fallback change and adds focused regression coverage; the only concrete follow-up I found is the stale nearby comment.

## What This Changes

The PR updates Mattermost message-tool send handling to use `threadId` as a fallback between explicit `replyToId` and legacy `replyTo`, with tests for trimmed `threadId` and `replyToId` precedence.

## Best Possible Solution

Land this focused Mattermost plugin fix after updating the stale fallback comment and refreshing validation, while keeping #67784 open for the remaining media-path and local-root message-tool work.

## Reproduction Assessment

Yes. Call `mattermostPlugin.actions.handleAction` with `action: "send"`, a Mattermost channel target, and `threadId` but no `replyToId`; current main forwards no `replyToId` to `sendMessageMattermost`, so no Mattermost `rootId` is set.

## Solution Assessment

Yes, with a small comment cleanup. The `replyToId ?? threadId ?? replyTo` order is the narrow maintainable fix because it matches the existing Mattermost outbound helpers and preserves explicit `replyToId` precedence.
