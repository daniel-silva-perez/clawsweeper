## Summary

Keep open. Current main still drops `threadId` on the Mattermost shared message-tool `send` action path, while the PR narrowly adds the missing fallback and tests the intended precedence. The only concrete review issue I found is a stale adjacent comment, which is non-blocking but should be updated before landing.

## What This Changes

The PR adds a `threadId` fallback to Mattermost message-tool send handling and adds regression coverage for trimmed `threadId` plus `replyToId` precedence.

## Best Possible Solution

Land this narrow Mattermost plugin fix after updating the stale comment and refreshing the targeted/changed checks, while leaving #67784 open for the remaining media-path and local-root message-tool work.

## Reproduction Assessment

Yes. A focused reproduction is to call `mattermostPlugin.actions.handleAction` with `action: "send"`, a Mattermost channel target, and `threadId` but no `replyToId`; current main forwards no `replyToId` to `sendMessageMattermost`, so the Mattermost post has no root id.

## Solution Assessment

Yes, with a small comment cleanup. The `replyToId ?? threadId ?? replyTo` order is the narrow maintainable fix because it matches the existing plugin outbound helpers and the shared thread-aware route precedence while preserving explicit `replyToId` priority.
