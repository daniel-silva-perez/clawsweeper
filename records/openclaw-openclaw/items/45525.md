## Summary

Keep this PR open. Current main still does not detect dynamic `require()` or `import()` module loading, so the hardening goal remains relevant, but the proposed first-token regex classifier leaves security bypasses and install-blocking false positives in a critical scan path.

## What This Changes

The PR adds critical `dynamic-require` and `dynamic-import` line rules to `src/security/skill-scanner.ts` for non-literal `require()` and `import()` arguments.

## Best Possible Solution

Replace the first-character regex approach with an argument-aware module specifier classifier in the scanner, then add focused tests for variables, concatenation, comments between tokens, commented literal imports, static and interpolated template literals, multiline forms, and normal static literal imports.

## Reproduction Assessment

Yes. The PR behavior is reproducible with a deterministic probe against the exact proposed regexes, and a parser probe confirms the bypass and false-positive syntax forms are valid JavaScript.

## Solution Assessment

No. A line regex that only inspects the first non-whitespace character after `(` is not the best way to solve a security scanner gap; a small tokenizer or parser-backed helper is the narrower maintainable fix for classifying module specifier arguments.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Classify the full module specifier:** `src/security/skill-scanner.ts:178-185`
  - body: The regex skips any call whose first argument token is a quote, so `require("./" + userInput)` and `import("./" + name)` avoid the new critical finding even though the module target is still resolved dynamically.
  - confidence: 0.93
- **[P1] Detect calls split by comments:** `src/security/skill-scanner.ts:178-185`
  - body: JavaScript permits comments between the callee and `(`, but these patterns only allow whitespace, so `require/*x*/(userInput)` and the matching `import` form bypass the intended critical rule.
  - confidence: 0.9
- **[P2] Skip commented literal import specifiers:** `src/security/skill-scanner.ts:185`
  - body: A literal import with a leading comment, such as `import(/* webpackChunkName: "x" */ "./mod")`, starts with `/` after the paren and is reported as critical even though the specifier is static.
  - confidence: 0.87
- **[P2] Treat static template literals as constants:** `src/security/skill-scanner.ts:178-185`
  - body: The pattern excludes only single and double quotes, so `import(`./fixed.js`)` and the matching `require` form are classified as critical even when the template contains no interpolation.
  - confidence: 0.88
