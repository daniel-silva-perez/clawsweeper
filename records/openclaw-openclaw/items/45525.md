## Summary

Keep this PR open, but do not merge it as-is. The dynamic module-loading blind spot is real, while the proposed regex-only line rules still miss runtime-resolved specifiers and introduce critical false positives in an install-blocking scanner.

## What This Changes

The PR adds two critical regex-based line rules to `src/security/skill-scanner.ts` to flag `require()` and dynamic `import()` calls whose argument does not start with a single or double quote.

## Best Possible Solution

Implement argument-aware module specifier classification for `require()` and dynamic `import()`, using a small tokenizer or parser-backed helper consistent with repo tooling, then add focused scanner tests for variables, concatenation, comments, template literals with and without interpolation, multiline literals, and normal static import forms.

## Reproduction Assessment

Yes. The PR behavior is reproducible with deterministic regex probes against the proposed patterns, and a parser probe confirms the bypass and false-positive syntax forms are valid JavaScript.

## Solution Assessment

No. A first-character line regex is not the best way to classify JavaScript module specifiers in an install-blocking security path; an argument-aware scanner is the safer maintainable direction.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Classify the full module specifier:** `src/security/skill-scanner.ts:178-185`
  - body: The regex only checks the first non-whitespace character after `(`, so `require("./" + userInput)` and `import("./" + name)` skip the critical rule even though the resolved module target is still dynamic.
  - confidence: 0.93
- **[P1] Detect comment-separated dynamic load calls:** `src/security/skill-scanner.ts:178-185`
  - body: JavaScript allows comments between the callee and `(`, but these patterns only allow whitespace. `require/*x*/(userInput)` and the matching `import` form bypass the new critical rule.
  - confidence: 0.9
- **[P2] Skip commented literal import specifiers:** `src/security/skill-scanner.ts:185`
  - body: A literal import with a leading comment, such as `import(/* webpackChunkName: "x" */ "./mod")`, starts with `/` after the paren, so the new rule reports a critical dynamic import even though the specifier is static.
  - confidence: 0.87
- **[P2] Treat static template literals as constants:** `src/security/skill-scanner.ts:178-185`
  - body: The lookahead excludes only single and double quotes, so `import(`./fixed.js`)` and the matching `require` form are reported as critical even when the template has no interpolation.
  - confidence: 0.88
