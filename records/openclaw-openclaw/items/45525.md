## Summary

Keep this PR open. Current main still lacks dynamic require/import detection, but the proposed scanner rules are not safe to merge because they miss dynamic module paths that begin with literals or comments and can create critical false positives on static imports.

## What This Changes

The PR adds two critical line-based skill scanner rules in `src/security/skill-scanner.ts` for `require()` and dynamic `import()` calls whose first argument does not start with a single or double quote.

## Best Possible Solution

Implement argument-aware module-load scanning that classifies the complete `require()` or dynamic `import()` specifier, with focused coverage for variables, concatenation, template literals with and without interpolation, comments around the callee and argument, multiline literals, and normal static imports.

## Reproduction Assessment

Yes. The patch behavior follows directly from the regexes and `scanSource` line-rule loop: examples such as `require("./" + name)`, `require/*x*/(name)`, `import(/* webpackChunkName: "x" */ "./mod")`, and `import(`./fixed.js`)` exercise the bypass and false-positive cases deterministically.

## Solution Assessment

No. First-character regex checks are not the best fix for JavaScript module specifier classification; a small argument-aware helper or parser-backed scanner would be more maintainable and safer for an install-blocking security path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Classify the complete module specifier:** `src/security/skill-scanner.ts:181-188`
  - body: The regexes only inspect the first non-whitespace character after `(`. Calls such as `require("./" + name)` and `import("./" + name)` begin with a quote, so they skip the critical rule even though the resolved module target is runtime-controlled.
  - confidence: 0.93
- **[P1] Handle comments before dynamic-load calls:** `src/security/skill-scanner.ts:181-188`
  - body: JavaScript permits comments between the callee and `(`, but these patterns only allow whitespace there. `require/*x*/(userInput)` and `import/*x*/(userInput)` bypass the new critical rule while still dynamically loading modules.
  - confidence: 0.9
- **[P2] Skip comments before literal import specifiers:** `src/security/skill-scanner.ts:188`
  - body: A static import such as `import(/* webpackChunkName: "x" */ "./mod")` starts with `/` after the paren, so the new `dynamic-import` rule reports a critical dynamic load even though the module specifier is a literal after the comment.
  - confidence: 0.87
- **[P2] Treat constant template specifiers as static:** `src/security/skill-scanner.ts:181-188`
  - body: The lookahead excludes only single and double quotes, so constant backtick specifiers like `import(`./fixed.js`)` or `require(`./fixed.js`)` are reported as critical even when they contain no interpolation, creating install-blocking false positives.
  - confidence: 0.88
