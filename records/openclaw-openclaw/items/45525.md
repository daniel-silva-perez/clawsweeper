## Summary

Keep this PR open, but it is not ready to merge. Current main still lacks the requested dynamic module-load scanner rules, while this PR's regex-only approach leaves bypasses and adds critical false positives in an install-blocking security path.

## What This Changes

The PR adds two critical line-based rules to `src/security/skill-scanner.ts` for `require()` and dynamic `import()` calls whose first argument does not start with a single or double quote.

## Best Possible Solution

Implement argument-aware module specifier classification for `require()` and dynamic `import()`, preferably through a small parser-backed helper or TypeScript AST path already consistent with repo tooling, with focused tests for variables, concatenation, comments, template literals with and without interpolation, multiline literals, and ordinary static imports.

## Reproduction Assessment

Yes. The proposed behavior is reproducible with deterministic regex probes against the PR patterns: concatenated and comment-split dynamic loads miss, while commented literal imports and constant template literals are flagged as critical.

## Solution Assessment

No. First-character regex checks are not the best way to classify JavaScript module specifiers in an install-blocking security path; an argument-aware scanner is safer and more maintainable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Classify the full module specifier:** `src/security/skill-scanner.ts:181-188`
  - body: The regex only checks the first non-whitespace character after `(`, so calls like `require("./" + name)` or `import("./" + name)` skip the critical rule even though the resolved module path is runtime-controlled.
  - confidence: 0.93
- **[P1] Handle comments around dynamic-load calls:** `src/security/skill-scanner.ts:181-188`
  - body: JavaScript permits comments between the callee and `(`, but these patterns only allow whitespace there. `require/*x*/(userInput)` and the equivalent `import` form bypass the new critical rule.
  - confidence: 0.9
- **[P2] Skip commented literal import specifiers:** `src/security/skill-scanner.ts:188`
  - body: A static import like `import(/* webpackChunkName: "x" */ "./mod")` starts with `/` after the paren, so the new `dynamic-import` rule reports a critical dynamic load even though the actual specifier is a literal.
  - confidence: 0.87
- **[P2] Classify constant template specifiers as static:** `src/security/skill-scanner.ts:181-188`
  - body: The lookahead excludes only single and double quotes, so `import(`./fixed.js`)` or `require(`./fixed.js`)` is reported as critical despite being a constant module specifier with no interpolation.
  - confidence: 0.88
