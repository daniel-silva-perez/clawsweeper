## Summary

Keep this PR open. Current main still lacks dynamic require()/import() detection, so the hardening goal remains relevant, but the proposed regex-only patch has concrete scanner bypasses and critical false positives that should block merge.

## What This Changes

The PR adds two critical LINE_RULES in src/security/skill-scanner.ts for dynamic require() and import() calls.

## Best Possible Solution

Replace the first-character regex approach with scanner logic that classifies the full require()/import() argument well enough for this security surface, then add focused tests in src/security/skill-scanner.test.ts for variable specifiers, concatenated specifiers, template literals with and without interpolation, inline comments, multiline literal imports, and ordinary static literals.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Detect concatenated module specifiers:** `src/security/skill-scanner.ts:178-185`
  - body: The new patterns only inspect the first non-whitespace character after `(`, so `require("./" + name)` and `import("./" + name)` look like static literals and avoid a critical finding even though the resolved module target is runtime-controlled.
  - confidence: 0.92
- **[P1] Allow comments before the call parenthesis:** `src/security/skill-scanner.ts:178-185`
  - body: JavaScript permits comments between the callee and `(`, but the regex only allows whitespace there. A call such as `require/*x*/(userInput)` bypasses the new critical rule while still loading a runtime-selected module.
  - confidence: 0.9
- **[P2] Skip leading comments before literal import specifiers:** `src/security/skill-scanner.ts:185`
  - body: For `import(/* webpackChunkName: "x" */ "./mod")`, the first character after `(` is `/`, so the new dynamic-import rule reports a critical finding even though the specifier is a static literal after the comment.
  - confidence: 0.86
- **[P2] Treat static template literals as literals:** `src/security/skill-scanner.ts:178-185`
  - body: Because the lookahead excludes only single and double quotes, constant backtick specifiers like `import(`./fixed.js`)` are reported as critical even when they contain no interpolation. That creates user-visible false-positive install blocks.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff is security-sensitive and needs attention because the proposed scanner rule can miss dynamic module loads while adding critical false positives.

Concerns:

- **[high] Dynamic module-load bypasses remain:** `src/security/skill-scanner.ts:178`
  - body: The added regexes miss concatenated specifiers and comment-split calls, so a malicious skill/plugin can still runtime-resolve modules without the intended critical scanner finding.
  - confidence: 0.9
- **[medium] Critical false positives can block safe installs:** `src/security/skill-scanner.ts:185`
  - body: Static template literals and literal imports with leading comments can be classified as critical dynamic loads, which can block gateway-backed skill/plugin install paths by default.
  - confidence: 0.86
