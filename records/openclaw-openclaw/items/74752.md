## Summary

Keep open. Current main still has a plausible Telegram delivery gap: the TTS tool and shared delivery preserve `audioAsVoice`, and the low-level Telegram sender can call `sendVoice`, but the registered Telegram plugin payload path still does not map that flag to `asVoice`. The existing ClawSweeper comment correctly kept this open, and overlapping PR #74663 is not a complete main-branch fix.

## What This Changes

The issue requests that generated Microsoft TTS audio from the `tts` tool be delivered to Telegram as a voice message instead of resulting in a regular text-only reply.

## Best Possible Solution

Carry `audioAsVoice` from the TTS-generated reply payload through the registered Telegram plugin payload/send-media path into `sendMessageTelegram` as `asVoice`, while keeping the existing direct bot delivery behavior intact. Add a regression that starts from a Telegram-origin TTS tool result with mocked `.ogg` or `.mp3` output and proves the final Telegram send calls `sendVoice`.
