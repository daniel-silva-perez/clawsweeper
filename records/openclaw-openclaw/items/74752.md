## Summary

Keep open. Current main still shows the reported TTS-to-Telegram voice delivery gap, and an open closing PR plus an overlapping maintainer PR are already tracking adjacent parts of the fix, so this issue should remain open until maintainers land or replace a coordinated patch.

## What This Changes

The issue requests that TTS-generated audio be delivered automatically as a Telegram voice message instead of returning `(spoken)` while sending a regular text fallback.

## Best Possible Solution

Land one coordinated narrow fix that lets trusted TTS local media survive tool-result filtering and carries the voice flag through Telegram outbound media/payload delivery into `sendMessageTelegram` as `asVoice`, with regression coverage for a TTS-originated Telegram voice payload.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass a TTS result with local `/tmp/...ogg` media, `trustedLocalMedia: true`, and `audioAsVoice: true` through the tool-result handler when `builtinToolNames` is present but lacks raw `tts`; current main can filter the local media away before queueing, and the Telegram outbound adapter also drops the voice flag.

## Solution Assessment

Unclear whether PR #74901 alone is the best complete fix because it targets the trusted-local-media filter while current main also shows a Telegram voice-flag forwarding gap. The safer maintainable path is one consolidated patch covering both boundaries and tests.
