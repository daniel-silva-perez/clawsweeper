## Summary

Keep open. Current main still shows a credible TTS-to-Telegram voice delivery gap, and open PR #74901 explicitly closes this issue while open PR #74920 overlaps the adjacent media/voice flag delivery path, so this issue should remain open until maintainers land or replace the coordinated fix.

## What This Changes

Make the TTS tool deliver generated Telegram audio as a voice message instead of returning `(spoken)` while the user receives text.

## Best Possible Solution

Land a coordinated narrow fix that lets trusted TTS local media survive tool-result filtering and carries `audioAsVoice` through Telegram outbound media/payload delivery into `sendMessageTelegram` as `asVoice`, with regression coverage for a TTS-originated Telegram voice payload.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass a TTS result with local `/tmp/...ogg` media, `trustedLocalMedia: true`, and `audioAsVoice: true` through the tool-result handler when `builtinToolNames` is present but lacks raw `tts`; current main can filter the media away before queueing, and the Telegram adapter also does not forward the voice flag.

## Solution Assessment

Unclear whether PR #74901 alone is the best complete fix because it addresses the trusted-local-media gate while current main also shows a Telegram `audioAsVoice` forwarding gap. The safest maintainable path is one coordinated patch covering both boundaries and their tests.
