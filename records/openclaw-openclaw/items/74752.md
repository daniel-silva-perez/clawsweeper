## Summary

Keep open. Current main still shows a credible TTS-to-Telegram voice delivery gap, and open PR #74901 uses closing syntax for this issue while PR #74920 overlaps the same media-delivery path, so the issue should remain open until maintainers land or replace the fix.

## What This Changes

The issue reports that the TTS tool returns `(spoken)` but Telegram receives a text reply instead of an automatic voice message.

## Best Possible Solution

Land or replace the linked work with one coordinated patch that lets trusted TTS local media survive tool-result filtering without weakening alias/MCP safeguards, carries `audioAsVoice` through Telegram `sendMedia` and `sendPayload` as `asVoice`, and adds regression coverage for a TTS-originated Telegram voice payload.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass a TTS result with local `/tmp/...ogg` media, `trustedLocalMedia: true`, and `audioAsVoice: true` through the tool-result handler when `builtinToolNames` is present but lacks raw `tts`; current main can filter the media away before queueing, and the Telegram outbound adapter also drops the voice flag before low-level send.

## Solution Assessment

Unclear whether PR #74901 alone is the best complete fix because it targets the trusted-local-media gate while current main also shows a Telegram voice-flag forwarding gap. The safer solution is a coordinated patch covering both boundaries and their tests.
