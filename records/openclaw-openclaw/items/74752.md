## Summary

Keep open. Current `main` still has a concrete Telegram voice-delivery gap: the TTS tool and shared outbound layer preserve `audioAsVoice`, and the low-level Telegram sender can call `sendVoice`, but the registered Telegram plugin path builds send options without forwarding that flag as `asVoice`. The earlier ClawSweeper comment and overlapping #74663 context remain accurate: #74663 is useful but does not cover the registered plugin path.

## What This Changes

The issue requests that audio generated by the TTS tool be automatically delivered to Telegram as a voice message instead of surfacing `(spoken)` text or a non-voice delivery.

## Best Possible Solution

Wire the existing voice intent through the registered Telegram plugin: carry `ctx.audioAsVoice` or `payload.audioAsVoice` into Telegram send options as `asVoice`, keep the legacy adapter aligned, and add a regression that starts from Telegram-origin TTS media and proves the final Telegram send calls `sendVoice`.

## Reproduction Assessment

Yes for a high-confidence code-level regression: synthesize a Telegram outbound/TTS media payload with `mediaUrl` and `audioAsVoice: true` through the registered plugin path and assert `sendMessageTelegram` receives `asVoice: true` or the mocked API reaches `sendVoice`. A live Telegram/Microsoft reproduction was not performed here.

## Solution Assessment

Yes, the narrow maintainable fix is to propagate the already-existing `audioAsVoice` flag through Telegram’s plugin-owned send option builder. #74663 is not sufficient as-is because it only covers the legacy outbound adapter, not the registered plugin path used by normal delivery.
