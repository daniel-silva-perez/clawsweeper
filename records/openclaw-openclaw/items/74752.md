## Summary

Keep open. Current main still has a concrete TTS-to-Telegram delivery gap: the TTS tool emits trusted, voice-capable local media, but the tool-result filter can still drop that local media before queueing, and the Telegram outbound adapter still does not forward the voice flag to the low-level `asVoice` option. Open PR #74901 explicitly closes this issue, and open PR #74920 overlaps the adjacent media delivery path, so the issue should stay open until maintainers consolidate and land the fix.

## What This Changes

The issue requests that TTS tool output marked as spoken audio be delivered to Telegram as a voice message instead of falling back to text.

## Best Possible Solution

Land a coordinated narrow fix that honors TTS-originated `trustedLocalMedia` during tool-result filtering and carries `audioAsVoice` through Telegram outbound media/payload delivery as `asVoice`, with focused regression coverage for a local TTS OGG/Opus result sent to Telegram as a voice message.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass a TTS tool result with local `details.media.mediaUrl`, `trustedLocalMedia: true`, and `audioAsVoice: true` through the tool-result handler while `builtinToolNames` is defined without `tts`; current main filters the local media out before queueing, and the Telegram adapter also lacks voice-flag forwarding.

## Solution Assessment

Unclear whether PR #74901 alone is the best complete fix because it targets the trusted-local-media filter, while current main still shows a Telegram `audioAsVoice` forwarding gap. The safest maintainable path is a consolidated or ordered fix covering both boundaries without broadening local file trust for untrusted tools.
