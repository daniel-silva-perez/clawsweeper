## Summary

Keep open. Current main still logs the manifest/package mismatch for the reported OpenClawBrain package/id pair, and the PR is not already implemented on main; the branch is useful but needs maintainer review because it adds a plugin-specific core alias and one discovery test does not actually exercise the new alias path.

## What This Changes

The PR adds an `@openclawbrain/openclaw` to `openclawbrain` alias in plugin discovery, suppresses the install-time manifest/package mismatch info log for that known pair, and adds targeted discovery/install regression tests.

## Best Possible Solution

Keep the manifest id as the canonical install/config key and, if this compatibility exception is accepted, implement the OpenClawBrain equivalence through one shared helper or metadata-backed table used by both discovery and install, with tests that cover the actual package-name alias and the install log suppression paths.

## Reproduction Assessment

Yes. A native plugin fixture with package name `@openclawbrain/openclaw` and manifest id `openclawbrain` follows current main into `packageNameMatchesId(...) === false` and the install info log; the discovery alias path is only reproducible when no manifest id wins first.

## Solution Assessment

Unclear. The install warning suppression is narrow, but the current PR shape duplicates alias state and hardcodes a plugin-specific exception in core; a shared equivalence helper or manifest/registry-backed compatibility contract would be safer if maintainers want this behavior.
