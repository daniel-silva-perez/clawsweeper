## Summary

Keep open. Current main still emits the OpenClawBrain manifest/package mismatch info message during install, so the PR is not obsolete; however the branch is dirty against current main and the discovery side has since moved to manifest-first identity, so this needs maintainer review and a current-main update before merge.

## What This Changes

The PR adds a hard-coded `@openclawbrain/openclaw` to `openclawbrain` alias in plugin discovery, suppresses the install mismatch info log for that alias, and adds discovery/install regression tests.

## Best Possible Solution

Keep the PR open, but update it against current main: preserve current manifest-first discovery behavior, implement any accepted OpenClawBrain install-name equivalence through one shared helper or metadata-backed contract, and keep targeted install/discovery regression coverage aligned with the current plugin identity flow.

## Reproduction Assessment

Yes. A deterministic source-level reproduction exists for the install chatter: install a native plugin whose `package.json` name is `@openclawbrain/openclaw` and whose manifest id is `openclawbrain`; current main's equivalence helper evaluates false and the install path logs the mismatch message.

## Solution Assessment

Unclear as-is. The install suppression is narrow and likely useful, but the PR should be rebased onto current main and avoid two private alias maps; a centralized equivalence helper or metadata-based contract would be the safer maintainable direction.
