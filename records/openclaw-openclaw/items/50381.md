## Summary

Keep this PR open. Current main still exposes `openclaw logs` as gateway-file-only tailing over `logs.tail`; it does not have `--source gateway|llm|all`, cache-trace-backed LLM logs, source metadata, hints, or pretty LLM trace rendering. The PR remains a plausible focused feature, but the existing review comments and a security/privacy pass need maintainer follow-up before merge.

## Best Possible Solution

Keep this PR open for maintainer/contributor follow-up. The best path is to rebase it onto current main, fix the review comments, decide the privacy/scope contract for cache-trace-backed LLM logs, add/update docs for `openclaw logs --source llm`, and rerun targeted CLI/gateway tests plus the appropriate changed gate before merge.

## Risks / Open Questions

- Cache-trace output can include prompt, system, and message content by default; exposing it through `logs.tail` under the existing read scope may broaden access to sensitive diagnostics.
- The PR's reported `loadConfig()` use in the LLM log path could break `openclaw logs --source llm` and `--source all` exactly when config is invalid and logs are needed for troubleshooting.
- The reported `--plain` ANSI leak and `toPublicSource` fallback bug should be fixed before merge to preserve CLI output contracts and older-gateway compatibility.
