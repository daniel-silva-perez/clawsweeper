## Summary

Keep this PR open. Current main still does not implement the PR's two intended Android fixes: the Screen tab active branch still renders no first-visit content, and initial chat load still uses the hard-coded 50-session bootstrap request instead of the proposed 200-session initial list. The PR is focused and not security-sensitive, but it still needs Android review because the Screen-tab approach has an unresolved WebView teardown concern and the PR body lacks verification detail.

## Best Possible Solution

Keep this PR open for normal Android maintainer review. The best path is to keep the chat fix on the single loadChat -> ChatController.bootstrap path with an explicit higher initial sessions limit, revise the Screen tab fix so ScreenTabScreen is composed in one stable keep-alive slot without WebView disposal/flicker, and add focused regression coverage or explicit real-device manual verification for both behaviors before merge.
