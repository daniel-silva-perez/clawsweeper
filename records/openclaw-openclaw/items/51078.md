## Summary

Keep this PR open. Current main still lacks the requested Android Screen-tab first-open behavior and the higher initial chat session-list limit, but the branch has an actionable Screen tab lifecycle regression to fix before it is merge-ready.

## What This Changes

The PR threads an optional chat session-list limit through the Android chat load path and changes the Screen tab branch to render `ScreenTabScreen` on first visit.

## Best Possible Solution

Land a rebased Android fix that composes the Screen tab from one stable keep-alive slot, threads the higher initial chat session-list limit through the existing single bootstrap `sessions.list` request, preserves current main's device-specific session handling, and includes focused regression coverage or explicit Android manual verification.

## Reproduction Assessment

Yes for code-level reproduction. Selecting the Screen tab first sets `screenTabStarted` from `LaunchedEffect(activeTab)` while the current active branch renders `Unit` on main; the PR's fallback then moves into the keep-alive branch after recomposition, and initial chat load still traces to `fetchSessions(limit = 50)` on main.

## Solution Assessment

No, the PR is not the best fix as-is. The chat-limit direction is the right seam after the earlier race review, but the Screen change should be reworked so `ScreenTabScreen` is only ever mounted in one stable composition slot.
