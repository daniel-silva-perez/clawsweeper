## Summary

Keep open. The PR targets a real SQLite binding failure, but the proposed fix is incomplete for the claimed agent/runtime path because it serializes only at the SQLite bind boundary while active `TaskRecord` objects can still carry non-string `task` values into in-memory consumers.

## What This Changes

The PR adds a helper to stringify non-string task payloads before SQLite binding and adds a regression test for storing a structured task payload as JSON.

## Best Possible Solution

Normalize task values before they become `TaskRecord` instances, or deliberately widen `TaskRecord.task` and all display/delivery consumers to handle structured values. That keeps the in-memory registry, observer events, fallback stores, and SQLite persistence on the same contract.

## Reproduction Assessment

Yes for the reported SQLite failure: a focused `node:sqlite` in-memory bind check reproduces object and array values failing to bind. For the PR’s remaining bug, code inspection shows the structured value would still be kept in memory and later hit string-only consumers.

## Solution Assessment

No. Binding-time serialization is too late for the claimed runtime path; the safer narrow fix is to coerce at `createTaskRecord`/registry ingress so the persisted value and active record agree.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.78

Full review comments:

- **[P2] Normalize task values before storing records:** `src/tasks/task-registry.store.sqlite.ts:156`
  - body: The patch only stringifies `record.task` in the SQLite bind map. If a structured payload reaches `createTaskRecord`, SQLite no longer throws, but the active registry record and observer event still carry an object even though consumers such as task status rendering call `task.task.trim()`. Please normalize the value before constructing/storing the `TaskRecord`, or widen the type and update all consumers.
  - confidence: 0.78
