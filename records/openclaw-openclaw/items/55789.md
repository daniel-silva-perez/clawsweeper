## Summary

Keep open. Current main still accepts reserved internal session keys through the OpenAI-compatible and OpenResponses request-context path, while the PR head appears to address that gap narrowly; the remaining action is maintainer gateway/security review plus landing validation, not closure or an automated replacement.

## What This Changes

The PR adds gateway request-context validation that rejects reserved `x-openclaw-session-key` prefixes, maps the dedicated validation error to 400 responses in the OpenAI-compatible and OpenResponses handlers, and adds regression coverage.

## Best Possible Solution

Land a narrow guard for OpenAI-compatible and OpenResponses header overrides, align the public endpoint docs with the reserved-namespace rejection, and coordinate any shared internal-session helper with the separate session-history work in #55928.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: send an OpenAI-compatible or OpenResponses request with `x-openclaw-session-key: agent:main:subagent:worker`; `resolveSessionKey` returns it unchanged and the handlers use that context.

## Solution Assessment

Yes. The PR’s central request-context check plus dedicated error type is the narrowest maintainable fix for these endpoints; a small docs update is the main cleanup before landing.
