## Summary

Keep open. Current main still accepts explicit internal-looking session keys on the OpenAI-compatible and OpenResponses HTTP endpoints, and this PR is a narrow relevant fix that should go through maintainer gateway/security review, docs alignment, and CI/Testbox validation rather than being closed or replaced.

## What This Changes

The PR adds reserved-prefix validation for explicit `x-openclaw-session-key` headers in gateway request-context resolution, maps that dedicated error to 400 responses in `/v1/chat/completions` and `/v1/responses`, and adds request-context plus endpoint regression tests.

## Best Possible Solution

Land the narrow header-override guard with a short public docs note for both HTTP compatibility pages, keep #55928 as the separate session-history follow-up, and only extract a shared internal-session helper if maintainers want the same policy across additional HTTP surfaces.

## Reproduction Assessment

Yes. Against current main, a request with `x-openclaw-session-key: agent:main:subagent:worker` is accepted by `resolveSessionKey` and reaches both endpoint handlers; source inspection gives a high-confidence reproduction path without running a live server in this read-only review.

## Solution Assessment

Yes, with docs follow-through. Blocking in shared request-context resolution and translating only the dedicated `GatewaySessionKeyOverrideError` to 400 is the narrowest maintainable fix for these two endpoints while avoiding the earlier internal-error exposure problem.
