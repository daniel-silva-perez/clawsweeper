## Summary

Keep open for maintainer review and merge consideration. Current main still accepts the reserved session-key override path, while the refreshed PR head appears to address the stated OpenAI-compatible and OpenResponses HTTP surfaces without a blocking correctness issue.

## What This Changes

The PR adds reserved `x-openclaw-session-key` prefix validation in gateway request-context resolution, maps that validation error to 400 for the OpenAI-compatible and OpenResponses HTTP handlers, and adds request-context plus endpoint regression tests.

## Best Possible Solution

Review and land this narrow guard for the OpenAI-compatible and OpenResponses HTTP surfaces, then track any broader internal-session HTTP policy in separate focused work such as the session-history follow-up.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence path: send an OpenAI-compatible or OpenResponses request with `x-openclaw-session-key: agent:main:subagent:worker`; the explicit key is returned unchanged by `resolveSessionKey` and used by the endpoint context.

## Solution Assessment

Yes. The PR uses the narrowest maintainable shape for the stated endpoints by centralizing reserved-prefix detection in request-context resolution and translating only the dedicated validation error to a 400 client response.
