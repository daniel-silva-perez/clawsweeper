## Summary

Keep open: current main still forwards explicit reserved session-key overrides, while the PR head adds a focused guard and narrowed 400 mapping. I found no blocking correctness or supply-chain finding in the proposed diff, so this remains a valid implementation candidate for maintainer review and landing.

## What This Changes

This PR adds reserved session-key prefix validation to gateway HTTP request-context resolution, maps that validation failure to 400 `invalid_request_error` in the OpenAI-compatible and OpenResponses handlers, and adds request-context plus endpoint regression tests.

## Best Possible Solution

Review and land the narrow guarded validation for the OpenAI-compatible and OpenResponses HTTP surfaces, preserving ordinary custom session keys and the dedicated error type so unrelated context/config failures are not exposed as client input errors. If maintainers want broader HTTP hardening, track the tools-invoke/session-history surfaces separately rather than expanding this PR ad hoc.

## Reproduction Assessment

Yes. On current main, source inspection gives a high-confidence reproduction path: send an OpenAI-compatible or OpenResponses HTTP request with `x-openclaw-session-key: agent:main:subagent:worker`; `resolveSessionKey` returns it unchanged and the endpoint passes it into the agent command path.

## Solution Assessment

Yes. The PR is the narrowest maintainable fix for the stated endpoints because it centralizes reserved-prefix detection in `resolveGatewayRequestContext` and only maps the dedicated override error to a 400; the safer alternative for broader ingress would be a separate follow-up rather than a mixed-scope patch.
