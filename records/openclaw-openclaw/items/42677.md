## Summary

Keep this PR open. Current main still does not implement the PR's intended bundled `session-memory` behavior for automatic stale-session rollover: the bundled hook is registered for explicit `/new` and `/reset` command events only, and its handler exits for non-command events. Current main does have plugin `session_end`/`session_start` lifecycle hooks for stale daily/idle rollovers, but bundled `session-memory` is not wired to that path. The PR remains relevant and should be refreshed rather than closed.

## Best Possible Solution

Keep this PR open for maintainer or contributor follow-up. Rebase it onto current main, choose the canonical integration path for automatic session boundaries, and update bundled `session-memory` handler metadata, docs, and tests so daily and idle rollover persist the prior session exactly once while explicit `/new` and `/reset`, including ACP-bound stale-session flows, do not double-write.

## Risks / Open Questions

- A naive `isNewSession && !resetTriggered` automatic rollover hook can duplicate `session-memory` writes for stale ACP-bound explicit `/new` or `/reset`, because ACP reset handling sets `resetHookTriggered` later in command processing.
- Current main now has plugin `session_end`/`session_start` lifecycle hooks for stale rollovers, so maintainers should choose whether bundled `session-memory` should consume that lifecycle path or add the PR's proposed internal `session:rollover` event.
