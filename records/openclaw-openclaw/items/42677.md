## Summary

Keep open. Current main still does not implement this PR's intended behavior: bundled session-memory only handles explicit /new and /reset command events, while automatic daily/idle rollover archives old transcripts and emits plugin lifecycle hooks rather than an internal session:rollover event that the bundled hook can persist. The PR remains a plausible focused fix, but it needs maintainer follow-up around the current lifecycle direction and the ACP duplicate-write review concern.

## Best Possible Solution

Keep this PR open for maintainer or contributor follow-up. Rebase it onto current main, choose the canonical integration path for automatic session boundaries, and update bundled session-memory handler metadata/docs/tests so daily and idle rollover persist the prior session exactly once while explicit /new and /reset, including ACP-bound resets, do not double-write.

## Risks / Open Questions

- A naive rollover predicate based only on !resetTriggered can still double-run session-memory for stale ACP-bound explicit reset/new flows, because ACP reset handling marks resetHookTriggered during command handling after session initialization.
- Current main now exposes automatic session boundaries through plugin session_end/session_start hooks. Maintainers should choose whether bundled session-memory should integrate with that lifecycle path or add the PR's new internal session:rollover event, then cover both daily and idle rollover.
