## Summary

Keep open. Current main still does not implement this PR's intended behavior: bundled session-memory is registered and executed only for explicit `command:new` and `command:reset`, while automatic daily/idle rollover archives the old transcript and emits plugin lifecycle hooks rather than an internal rollover event that the bundled hook can persist. The PR remains relevant, but it needs a rebase and maintainer follow-up on the lifecycle seam plus the ACP duplicate-write review concern.

## Best Possible Solution

Keep this PR open for maintainer or contributor follow-up. Rebase it onto current main, choose the canonical integration path for automatic session boundaries, and update bundled session-memory handler metadata, docs, and tests so daily and idle rollover persist the prior session exactly once while explicit `/new` and `/reset`, including ACP-bound stale-session flows, do not double-write.

## Risks / Open Questions

- A naive `isNewSession && !resetTriggered` rollover hook can double-persist for stale ACP-bound explicit `/new` or `/reset`, because ACP reset handling marks `resetHookTriggered` later in command handling.
- Current main now has plugin `session_end`/`session_start` lifecycle hooks for automatic rollover. Maintainers should choose whether bundled session-memory should consume that lifecycle path or add the PR's proposed internal `session:rollover` event.
