## Summary

Keep this PR open. Current main still leaves bundled `session-memory` wired only to explicit `/new` and `/reset` internal command events; automatic daily/idle rollover now has plugin `session_end`/`session_start` lifecycle coverage, but that path is not connected to the bundled internal hook metadata or handler. The PR remains relevant, though it should be rebased and reconciled with the newer plugin lifecycle path before merge.

## Best Possible Solution

Keep this PR open for contributor or maintainer follow-up. Rebase it onto current main, decide the canonical integration path with the newer plugin session lifecycle, and update handler metadata, docs, and tests so daily/idle rollover persists the prior bundled session-memory snapshot exactly once while explicit `/new`, `/reset`, and ACP-bound reset flows do not double-write.

- Current main now has plugin `session_end`/`session_start` lifecycle hooks, so the refreshed PR should choose whether bundled `session-memory` should use that lifecycle path or a new typed internal `session:rollover` event.

- Hook handlers run in the Gateway process and this PR touches the memory-write path, so the final implementation should preserve the existing trusted-hook boundary and avoid duplicate LLM slug generation or duplicate memory files.
