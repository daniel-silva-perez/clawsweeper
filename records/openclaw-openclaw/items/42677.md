## Summary

Keep this PR open. Current main still leaves bundled `session-memory` wired only to explicit `/new` and `/reset` command events; automatic daily/idle rollover now has plugin `session_end`/`session_start` lifecycle coverage, but that path is not connected to the bundled internal hook.

## Best Possible Solution

Keep this PR open for maintainer or contributor follow-up. Rebase it onto current main, decide whether bundled `session-memory` should consume the current plugin session lifecycle path or a new typed internal `session:rollover` event, and update handler metadata, docs, and tests so daily/idle rollover persists the prior session exactly once while explicit `/new`, `/reset`, and ACP-bound reset flows do not double-write.

## Risks / Open Questions

- Current main now has plugin `session_end`/`session_start` lifecycle hooks, so the refreshed PR should choose a canonical integration path instead of adding a parallel rollover contract by default.
- A naive automatic-rollover predicate can double-write memory snapshots around explicit stale `/new` or `/reset` flows, especially ACP-bound reset handling; this needs a targeted regression test before merge.
- The PR is not a supply-chain concern from the provided diff, but hook handlers execute inside the Gateway process, so any refreshed implementation should preserve the existing hook boundary and eligibility checks.
