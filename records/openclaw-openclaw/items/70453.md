## Summary

Keep open. Current main still sends Google Gemini LLM traffic through the shared guarded model fetch path, and that path does not opt ambient HTTP_PROXY/HTTPS_PROXY into trusted env-proxy mode. The related #72057 report is already closed as a duplicate, while #72480 is still an open/unmerged implementation candidate for the same gap.

## What This Changes

The issue reports that CLI and WebUI Gemini LLM calls hang or fail silently when HTTP_PROXY/HTTPS_PROXY are set to route requests through a local HTTP proxy.

## Best Possible Solution

Land #72480 or an equivalent narrow provider guarded-fetch change that selects trusted env-proxy mode only when no explicit provider dispatcher policy exists and shouldUseEnvHttpProxyForUrl(url) is true, with regression coverage for ambient proxy use and explicit provider override preservation.

## Reproduction Assessment

Yes for the core proxy-routing gap: an HTTPS Google LLM request with ambient HTTPS_PROXY and no models.providers.google.request.proxy override still reaches fetchWithSsrFGuard without trusted env-proxy mode on current main. The exact CLI/WebUI silent-hang presentation needs live proxy/Gemini validation during the fix.

## Solution Assessment

Yes: the narrow maintainable fix belongs at the shared provider guarded-fetch seam, not in Google-specific transport code. The open #72480 direction matches that boundary but still needs maintainer/security review, rebase, and CI before this issue can close.
