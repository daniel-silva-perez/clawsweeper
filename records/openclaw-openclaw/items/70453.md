## Summary

Keep open. Current main still sends Google Generative AI LLM requests through the shared guarded model fetch path, and that path does not opt bare ambient HTTP_PROXY/HTTPS_PROXY into trusted env-proxy mode. The related #72057 report is already closed as a duplicate of this issue, while #72480 is still an open implementation candidate rather than a merged fix.

## What This Changes

The issue requests that Google/Gemini CLI and WebUI model calls respect HTTP(S) proxy environment variables and surface explicit failures instead of hanging silently.

## Best Possible Solution

Land #72480 or an equivalent narrow change in buildGuardedModelFetch that selects trusted env-proxy mode only when no explicit provider dispatcher policy exists and shouldUseEnvHttpProxyForUrl(url) is true, with regression coverage for ambient proxy use and explicit provider override preservation. If maintainers choose not to support bare ambient proxy env for this path, document the supported proxy configuration and make Google LLM proxy failures surface clearly in CLI/WebUI.

## Reproduction Assessment

Yes for the proxy-routing gap: an HTTPS Google LLM request with ambient https_proxy and no managed/provider proxy override still reaches fetchWithSsrFGuard without trusted env-proxy mode on current main. The full silent-hang UI symptom needs a live proxy/Gemini run or targeted integration proof during the fix.

## Solution Assessment

Yes: the narrow maintainable solution is a guarded-fetch seam change, not Google-specific proxy handling. The open #72480 direction matches that boundary but still needs maintainer and security review before this issue can close.
