## Summary

Keep open. Current main already ships the Gemini Embedding 2 provider path, configurable output dimensions, and opt-in image/audio multimodal indexing for files under memorySearch.extraPaths, but it still does not implement the requested conversation-history attachment indexing workflow for chat or Telegram media.

## Best Possible Solution

Keep this open and narrow the remaining work to session-memory attachment indexing. Extend the session-memory indexing path to discover eligible MediaPath/MediaPaths plus MIME metadata from chat transcripts, resolve them through existing managed-media and sandbox boundaries, create structured EmbeddingInput chunks for supported attachments, and add regression coverage for chat/Telegram attachment recall. Preserve the existing Gemini Embedding 2, outputDimensionality, and extraPaths image/audio behavior, and document the distinction between file-backed multimodal memory and conversation-history attachment indexing.

- A correct implementation needs careful media path resolution, managed-media and sandbox boundary checks, MIME validation, provider capability checks, file-size limits, and fallback behavior.
- Current documented Gemini multimodal memory covers image/audio extraPath files only; video and PDF memory embedding remain outside this shipped scope.
