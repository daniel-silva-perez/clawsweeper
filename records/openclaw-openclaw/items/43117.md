## Summary

Keep open. Current main implements the Gemini Embedding 2 subset, configurable output dimensions, and opt-in image/audio indexing for files under memorySearch.extraPaths, but it still does not implement the central requested workflow: embedding conversation-history attachments such as Telegram screenshots/photos directly from session memory. The provided discussion is useful and accurate: one commenter documented the shipped extraPaths workaround, while the original report and +1 comment ask for searchable chat attachments.

## Best Possible Solution

Keep this issue open and narrow the next implementation to session-memory attachment indexing. Extend the session transcript memory path to discover eligible MediaPath/MediaPaths/MediaType/MediaTypes metadata, resolve files through existing managed-media and sandbox safety boundaries, and emit structured EmbeddingInput parts for supported attachments while preserving the shipped Gemini Embedding 2, output-dimensionality, and extraPaths image/audio behavior. Add regression coverage for chat or Telegram attachment recall and update docs to distinguish file-path media indexing from conversation-history attachment indexing; treat video/PDF as a deliberate follow-up unless implemented in the same scoped change.
