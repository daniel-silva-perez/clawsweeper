## Summary

Keep open. Current main implements the useful subset called out in the discussion: Gemini Embedding 2 preview, configurable output dimensions, and opt-in image/audio indexing for files under memorySearch.extraPaths. It still does not implement the central conversation-history attachment workflow: chat transcripts can persist MediaPath/MediaPaths/MediaType/MediaTypes, but session memory indexing flattens only string/text content into text chunks and never turns those media fields into structured embedding inputs. Video and PDF are also still outside the current memory multimodal modality set.

## Best Possible Solution

Keep this issue open and narrow it to session-memory attachment indexing. Extend the session transcript indexing path to collect eligible persisted media path/type fields, resolve them through the existing media and sandbox safety boundaries, and pass structured EmbeddingInput parts for supported attachments while preserving the shipped Gemini Embedding 2 preview, output-dimensionality, and extra-path image/audio behavior. Add regression coverage for chat or Telegram attachment recall and update docs to distinguish extra-path media indexing from conversation-history attachment indexing, plus any future video/PDF support.
