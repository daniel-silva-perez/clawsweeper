## Summary

Keep open. Current main implements the useful subset called out in the discussion: Gemini Embedding 2 preview, configurable output dimensions, and image/audio multimodal file indexing from memorySearch.extraPaths. The central requested workflow still remains: conversation-history attachments such as Telegram screenshots/photos are persisted as MediaPath/MediaPaths/MediaType/MediaTypes outside message.content, while the session-memory exporter only extracts text from message.content and the session sync path indexes that rendered text. Video and PDF are also still outside the memory multimodal modality set.

## Best Possible Solution

Keep this issue open and narrow the next implementation to session-memory attachment indexing. Extend the session transcript indexing path to collect eligible persisted MediaPath/MediaPaths/MediaType/MediaTypes fields, resolve them through existing managed-media and sandbox safety boundaries, and emit structured EmbeddingInput parts for supported attachments while preserving the shipped Gemini Embedding 2, output-dimensionality, and image/audio extraPaths behavior. Add regression coverage for chat or Telegram attachment recall and update docs to distinguish file-path media indexing from conversation-history attachment indexing, with video/PDF support either implemented deliberately or split into a follow-up.

## Risks / Open Questions

- Closing now would lose the main user-visible request: searching prior chat or Telegram attachments without manually placing those media files under memorySearch.extraPaths.
- A real implementation needs careful media path resolution through managed-media and sandbox boundaries, MIME validation, size limits, provider capability checks, and explicit fallback behavior.
- Video and PDF embeddings remain separate gaps from the currently shipped image/audio file-indexing path.
