## Summary

Keep open. Current main has the shipped Gemini Embedding 2 subset: configurable output dimensions and structured image/audio embeddings for file-based memory indexing. The central request still remains: conversation-history attachments such as chat or Telegram screenshots/photos are stored as media path/type metadata, while session memory indexing extracts only text from message.content and indexes that rendered text.

## Best Possible Solution

Keep this issue open and narrow the next implementation to session-memory attachment indexing. Extend the session transcript memory path to collect eligible MediaPath/MediaPaths/MediaType/MediaTypes fields, resolve them through existing managed-media and sandbox safety boundaries, and emit structured EmbeddingInput parts for supported attachments while preserving the shipped Gemini Embedding 2, output-dimensionality, and image/audio file-indexing behavior. Add regression coverage for chat or Telegram attachment recall and update docs to distinguish file-path media indexing from conversation-history attachment indexing; split video/PDF into a deliberate follow-up unless implemented in the same scoped change.

## Risks / Open Questions

- Closing now would drop the main user-visible workflow: searching prior chat or Telegram attachments without manually placing media files in a memory-indexed filesystem location.
- A correct implementation needs careful media path resolution through managed-media and sandbox boundaries, MIME validation, size limits, provider capability checks, and explicit fallback behavior.
- Video and PDF support remain separate gaps from the currently shipped image/audio multimodal memory path.
