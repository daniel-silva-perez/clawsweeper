## Summary

Keep open. Current main implements the Gemini Embedding 2 subset, configurable output dimensions, and image/audio multimodal indexing for file-backed `memorySearch.extraPaths`, but it still does not implement the requested conversation-history attachment workflow. Chat and Telegram transcript messages can carry `MediaPath`/`MediaPaths` metadata, yet memory session indexing still flattens only text and indexes `entry.content`.

## Best Possible Solution

Keep this issue open and narrow the remaining work to session-memory attachment indexing. Extend the memory session indexing path to discover eligible `MediaPath`/`MediaPaths` plus MIME metadata, resolve files through existing managed-media and sandbox boundaries, emit structured `EmbeddingInput` chunks for supported attachments, and add regression coverage for chat/Telegram attachment recall. Preserve the existing Gemini Embedding 2, `outputDimensionality`, and extraPaths image/audio behavior, and document the distinction between file-backed multimodal memory and conversation-history attachment indexing.

- Video and PDF embedding remain outside the currently documented image/audio extraPaths implementation.
