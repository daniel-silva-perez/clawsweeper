## Summary

Keep this PR open. Current main still does not implement the PR's intended behavior: local Gateway calls are still constructed as loopback, tests still assert loopback for tailnet local calls, tailnet/custom server binds can still be non-loopback-only, and the CLI close formatter still has no mode-aware auth hint. The PR remains a plausible implementation candidate, but it is not merge-ready as written because the client-side non-loopback URL approach conflicts with the existing plaintext WebSocket security policy and the proposed auth hint needs mode-specific handling.

## Best Possible Solution

Keep this PR open for maintainer follow-up. Maintainers should decide whether #40247's server-side loopback-alias approach is the canonical fix for tailnet/custom local calls; if so, this PR should narrow to a tested, mode-aware auth diagnostic. If client-side host resolution remains desired, it should be implemented in `src/gateway/connection-details.ts`, preserve the existing plaintext WebSocket security policy, update diagnostics such as `urlSource`, and add focused tests for tailnet/custom bind plus token/password/trusted-proxy auth behavior.

## Risks / Open Questions

- The PR is not merge-ready without resolving the plaintext non-loopback WebSocket policy interaction for default `gateway.tls.enabled=false` tailnet/custom setups.
