## Summary

Keep this PR open because current main still lacks the Copilot `/models` catalog fetch and GPT-5.5 capability mapping, but the proposed patch needs one correctness fix before merge: it maps every Copilot `/models` entry into chat model metadata even though the same endpoint can include embedding-only models.

## What This Changes

The branch updates the GitHub Copilot plugin and tests to fetch and map `/models` metadata for GPT-5.5 while also carrying several formatting-only changes in unrelated files.

## Best Possible Solution

Land a provider-owned Copilot catalog patch after it filters `/models` to chat/inference models only, keeps embedding discovery separate, preserves GPT-5.5 capability and xhigh support, and leaves unrelated formatting churn out of the branch.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Filter embedding-only Copilot catalog entries:** `extensions/github-copilot/index.ts:145-147`
  - body: The Copilot `/models` endpoint is already used by the embedding provider and can return entries such as `text-embedding-3-small` with `/v1/embeddings` or empty endpoints. This loop maps every entry with an id into `provider.models`, so embedding-only IDs would be advertised as chat models using the Responses transport and fail when selected. Filter to chat/Responses/Messages entries, or explicitly skip embedding-only models before mapping.
  - confidence: 0.86
