## Summary

Keep this PR open. Current main still handles direct leading `403 <html>...` auth pages, but it does not implement the PR's central embedded-wrapper case such as `Error: fetch failed: 403 <html>...`; that path can still miss `auth_html_403` classification and fall through to transport-style copy. The provided PR context also shows the earlier mojibake ellipsis review was followed by a second fix commit, so the best next step is maintainer review of the updated PR rather than cleanup closure.

## Best Possible Solution

Keep the PR open as a focused implementation candidate. The best path is to review the updated head, confirm the ellipsis fix is present, consider sharing the HTML auth-copy predicate, rerun the targeted formatter, sanitizer, and runtime-classification tests, then merge or request changes based on that review.
