## Summary

Keep this issue open. Current main supports named profile homes and has useful gateway status/config-mismatch diagnostics, but it does not fully implement the requested authoritative multi-profile truth surface: custom profile casing is still preserved, service labels are still derived from the raw custom profile, and status/restart/agent output does not consistently report the resolved profile, service boundary, config/state paths, and session store as a single authoritative target report.

## Best Possible Solution

Keep this open and narrow it to core CLI/gateway behavior: define canonical custom-profile casing or fail loudly on ambiguous casing, make status/restart/agent surfaces print the resolved profile, config path, state dir, service label/process boundary, port/probe target, and session store path, and add macOS LaunchAgent tests for multiple `~/.openclaw-*` homes and case-mismatched profile names. Treat the current multiple-gateway docs and config-mismatch diagnostics as the baseline, not a complete fix.
