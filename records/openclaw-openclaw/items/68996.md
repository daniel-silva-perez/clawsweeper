## Summary

Keep this PR open. Current main still does not implement the central Gemma transport fix: dynamic Gemma resolution only patches reasoning, explicit provider-model normalization does not force Gemma model APIs, and runtime resolution can still prefer configured OpenAI-compatible transport over discovered native metadata. The PR addresses a real Google provider bug and the author supplied useful endpoint/config verification, but review comments identify unresolved google-gemini-cli/baseUrl and provider-normalization edge cases that need revision before merge.

## Best Possible Solution

Keep the PR open and revise it before merge: implement Gemma native routing in the Google plugin owner code for both dynamic and explicit model configs, preserve compatible google-gemini-cli/baseUrl behavior, avoid provider-level baseUrl rewrites unless the affected model API is actually switched to google-generative-ai, and add regression assertions for api and baseUrl on google, google-gemini-cli, and explicit config paths. After the revised fix lands, use the landed commit to close remaining Gemma transport reports such as #63685 if covered.
