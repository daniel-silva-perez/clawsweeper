## Summary

Keep this issue open. Current `origin/main` still has no `brave.http` diagnostics flag, and the Brave plugin makes outbound Brave API calls plus cache reads/writes without request/status/timing/cache diagnostic logging. The feature is narrow and actionable in the bundled Brave plugin using existing diagnostics and logging SDK surfaces.

## What This Changes

The issue requests a Brave-specific `brave.http` diagnostics flag for targeted web search request, response, timing, and cache hit/miss logging.

## Best Possible Solution

Add an opt-in Brave diagnostics path inside the Brave plugin: pass the full config or a precomputed `brave.http` flag into the runtime, emit sanitized request/status/duration/cache hit/miss logs for both `web` and `llm-context` modes, and document the flag in the diagnostics and Brave Search docs.

## Repair Work Prompt

Implement `brave.http` diagnostics for the bundled Brave Search plugin. In `extensions/brave`, thread the full OpenClaw config or a precomputed `isDiagnosticFlagEnabled("brave.http", config)` boolean from `createBraveWebSearchProvider().createTool(ctx)` into the runtime. Add sanitized diagnostic logging around both Brave Search API modes: outbound endpoint/mode/query/params, response status and elapsed duration, and cache hit/miss/write behavior. Do not log `X-Subscription-Token`, SecretRef-resolved credential material, or raw headers. Keep behavior unchanged when the flag is disabled. Add focused tests in the Brave provider test file for disabled logging, enabled request/status timing logging, and cache hit/miss logging. Update `docs/diagnostics/flags.md`, `docs/tools/brave-search.md`, and the duplicate Brave Search doc if it remains content-bearing.
