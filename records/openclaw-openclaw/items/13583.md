## Summary

Keep this open. Current main has adjacent pre-tool blocking, final-answer revision, reply-dispatch, and outbound message-cancel hooks, but it still lacks the requested fail-closed response gate with mandatory tool-call accounting, typed validators, two-phase planning, bounded retry/fallback behavior, and audit records. A maintainer-facing review in the issue already classified this as a core/plugin API gap requiring follow-up.

## What This Changes

The issue requests a pre-response enforcement layer that prevents final output delivery until configured mandatory tools and validators have run and passed.

## Best Possible Solution

Keep this as the canonical response-gate policy-as-code design item. The desired end state is a versioned core/plugin API for mandatory tool-call accounting, typed response validators, fail-closed final-response gating, bounded retry/fallback behavior, and audit records, with domain-specific policy packs implemented as plugins or ClawHub packages.

## Reproduction Assessment

No high-confidence repo-local reproduction exists yet. The discussion gives credible production and canary evidence, and current-main code inspection confirms the missing response-gate surface, but there is no deterministic failing test that demonstrates prompt-only mandatory tool rules being bypassed and then prevented by a gate.

## Solution Assessment

Yes, keeping this open for explicit design is the best current path. Existing hooks are useful adjacent mechanisms, but the requested behavior needs a versioned core/plugin API rather than treating message_sending or before_agent_finalize as a complete hard gate.
