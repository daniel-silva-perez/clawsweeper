## Summary

Keep this open. Current main still has adjacent pre-tool blocking, final-answer revision, reply-dispatch, and outbound message-cancel hooks, but it does not implement the requested fail-closed response gate with mandatory tool-call accounting, typed validators, two-phase planning, bounded retry/fallback behavior, or audit records. The issue discussion usefully links complementary external/plugin work, while preserving the core gap that plugins cannot yet mechanically prevent final response delivery until required tools and validators pass.

## What This Changes

The issue requests a fail-closed pre-response enforcement layer, using a two-phase protocol and validators to require configured tool calls or policy checks before final output delivery.

## Best Possible Solution

Keep this as the canonical response-gate policy-as-code design item. The desired end state is a versioned core/plugin API for mandatory tool-call accounting, typed response validators, fail-closed final-response gating, bounded retry/fallback behavior, and audit records, with domain-specific policy packs implemented as plugins or ClawHub packages.

## Reproduction Assessment

No high-confidence repo-local reproduction exists yet. The discussion gives credible production and canary evidence, and current-main code inspection confirms the missing response-gate surface, but there is no deterministic failing test for mandatory prompt-rule bypass and enforcement.

## Solution Assessment

Yes, keeping this open for explicit maintainer design is the best current path. Existing before_tool_call, before_agent_finalize, reply_dispatch, and message_sending hooks are useful adjacent mechanisms, but treating them as the requested hard response gate would leave the mandatory-tool ledger, validator contract, fail-closed policy, and audit model unspecified.
