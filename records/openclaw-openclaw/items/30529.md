## Summary

Keep this issue open. Current main has automatic Discord typing support for inbound reply and heartbeat/component paths, but it still does not expose a shared agent-callable `typing` or `typing-indicator` action through the `message` tool. The prior keep-open review remains accurate against current main SHA 14f140d6f0d31fed802869622c38c7e8f8e19587.

## Best Possible Solution

Keep the issue open until maintainers either add a shared `typing` or `typing-indicator` message-tool action, or explicitly decide that automatic inbound reply and heartbeat typing are the intended OpenClaw boundary. A complete implementation would update the shared action registry, message-tool schema, CLI normalization/target mode, Discord discovery, Discord action dispatch, lifecycle semantics, regression tests, and public docs.

## Risks / Open Questions

- The existing automatic Discord typing path partially satisfies the title, so maintainers should explicitly decide whether automatic inbound/heartbeat typing is the intended product boundary or whether manual message-tool control should be added.
- Discord typing is a temporary TTL cue, not a durable start/stop state; any manual action needs clear one-shot, refresh, and cleanup semantics to avoid repeating prior stuck-indicator lifecycle bugs cited in the issue discussion.
