## Summary

Keep this issue open. Current main has Discord typing infrastructure for automatic inbound replies and heartbeat delivery, but it still does not expose the requested agent-callable `typing` or `typing-indicator` action through the shared `message` tool. The related comments point to lifecycle bugs and cross-provider design concerns, not to a merged implementation or canonical successor.

## Best Possible Solution

Keep the issue open until maintainers either add a shared `typing` or `typing-indicator` message-tool action, or explicitly decide that automatic typing is sufficient. A concrete implementation should add the shared action name, target-mode/schema support, Discord `describeMessageTool` advertising, a `handleDiscordMessageAction` route to `sendTypingDiscord`, lifecycle semantics, Discord regression coverage, and public docs. If cross-provider parity is desired, design the action so Telegram, Slack, Discord, and similar plugins can opt in through the channel plugin contract.

## Risks / Open Questions

- Manual typing needs explicit one-shot, refresh, and stop semantics so it does not recreate the stuck-indicator lifecycle bugs cited in #26437, #27174, #27419, and #26416.
- Maintainers may decide automatic inbound-reply and heartbeat typing is the intended product boundary, but that would be a product decision rather than evidence that the requested message-tool action already exists.
