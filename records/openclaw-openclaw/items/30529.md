## Summary

Keep this issue open. Current main has automatic Discord typing for inbound reply pipelines and heartbeat targets, but it still does not expose an agent-callable `typing` or `typing-indicator` action through the shared `message` tool. The action registry, agent schema, CLI normalization, Discord discovery, and Discord dispatcher all still omit that surface, so the requested manual message-tool behavior remains unimplemented.

## Best Possible Solution

Keep the issue open until maintainers either add a shared `typing`/`typing-indicator` message-tool action or explicitly decide that automatic inbound and heartbeat typing are the intended product boundary. A concrete implementation should update the shared action registry and target mode, expose the action through Discord discovery, route it to `sendTypingDiscord`, define bounded lifecycle semantics, add Discord regression tests, and document the behavior in the public message-tool and typing-indicator docs.

## Risks / Open Questions

- Manual typing control needs explicit one-shot, refresh, timeout, and cleanup semantics so it does not recreate the stuck-indicator lifecycle bugs cited in the discussion.
- A generic cross-channel typing action would touch shared SDK/action enums, target-mode mapping, CLI routing, plugin discovery, Discord dispatch, docs, and regression coverage.
- Discord typing is a native one-shot/TTL cue rather than a durable start/stop state, so the public contract should avoid promising unsupported stop semantics.
