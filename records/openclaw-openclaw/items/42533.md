## Summary

Keep open. Current main still matches the reported macOS speech-input failure: Push-to-Talk, Talk Mode, Voice Wake, and the Voice Wake tester all append hardware-format audio tap buffers directly to `SFSpeechAudioBufferRecognitionRequest`, with no macOS app downmix/conversion layer or multi-channel regression coverage found.

## Best Possible Solution

Keep this issue open and fix the shared macOS Speech capture path. Add a reusable helper that converts or downmixes `AVAudioPCMBuffer` from the hardware tap into a Speech-compatible mono/native format before appending it, wire that through `VoicePushToTalk`, `TalkModeRuntime`, `VoiceWakeRuntime`, and `VoiceWakeTester`, then add focused conversion coverage plus manual validation on a greater-than-2-channel macOS input device. If reusing the existing Swabble converter is the right direction, expose it through the proper product/API instead of leaving the macOS app on `SwabbleKit` only.
