## Summary

Keep open. Current main at 189c91eae6e1f4f280179961bca2c4153cf42e86 still matches the reported macOS speech-input failure mode: PTT, Talk Mode, Voice Wake runtime, and the Voice Wake tester all capture the hardware input format and append raw buffers directly to SFSpeechAudioBufferRecognitionRequest. No macOS app downmix/conversion layer or multi-channel regression coverage was found. The previous ClawSweeper comment remains accurate against current main.

## Best Possible Solution

Keep this issue open and fix the macOS speech capture pipeline. Add a shared helper that converts or downmixes AVAudioPCMBuffer input from the hardware tap into a Speech-compatible mono/native format before appending it, wire it through VoicePushToTalk, TalkModeRuntime, VoiceWakeRuntime, and VoiceWakeTester, then add focused conversion coverage plus manual validation on a >2-channel macOS input device.
