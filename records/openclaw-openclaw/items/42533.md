## Summary

Keep open. Current main still matches the reported macOS speech-input failure: Push-to-Talk, Talk Mode, Voice Wake runtime, and the Voice Wake tester all tap the hardware input format and append raw buffers directly to `SFSpeechAudioBufferRecognitionRequest`, with no macOS app downmix/conversion helper or multi-channel regression coverage found.

## Best Possible Solution

Keep this issue open and fix the macOS Speech capture pipeline. Add a shared helper that converts or downmixes `AVAudioPCMBuffer` input from the hardware tap into a Speech-compatible mono/native format before appending it, wire it through `VoicePushToTalk`, `TalkModeRuntime`, `VoiceWakeRuntime`, and `VoiceWakeTester`, then add focused conversion coverage plus manual validation on a greater-than-2-channel macOS input device.
