## Summary

Keep open. Current main still matches the reported macOS Speech input bug: Push-to-Talk, Talk Mode, Voice Wake, and the Voice Wake tester append hardware tap buffers directly to `SFSpeechAudioBufferRecognitionRequest` with no downmix or conversion layer for multi-channel input devices.

## Best Possible Solution

Keep this issue open and fix the shared macOS Speech capture path. Add a reusable helper that converts or downmixes `AVAudioPCMBuffer` from the hardware tap into a Speech-compatible mono/native format before appending it, wire it through `VoicePushToTalk`, `TalkModeRuntime`, `VoiceWakeRuntime`, and `VoiceWakeTester`, then add focused conversion coverage plus manual validation on a greater-than-2-channel macOS input device.
