## Summary

Current `main` still appears to expose the reported macOS Speech capture bug: the PTT, Talk Mode, Voice Wake, and Voice Wake tester paths install taps using the hardware input format and append captured `AVAudioPCMBuffer`s directly into `SFSpeechAudioBufferRecognitionRequest` with no shared mono downmix or format conversion layer for multi-channel input devices. The prior ClawSweeper comment is consistent with the current checkout, and no related item in the provided context supersedes this concrete bug.

## Best Possible Solution

Keep this issue open and fix the shared macOS Speech capture path. The likely implementation should add a reusable conversion/downmix helper that normalizes hardware tap buffers into a Speech-compatible mono/native format before appending, then wire it through `VoicePushToTalk`, `TalkModeRuntime`, `VoiceWakeRuntime`, and `VoiceWakeTester`. Add focused conversion coverage where possible plus manual validation on a multi-channel macOS input interface.
