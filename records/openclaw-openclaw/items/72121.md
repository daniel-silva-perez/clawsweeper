## Summary

Keep open. Current main and the latest shipped tag still reject the reported `sk-ant-api03-...` value on the `models auth paste-token --provider anthropic` path, and the linked implementation PR is closed unmerged rather than a landed fix.

## What This Changes

The issue asks OpenClaw to accept standard Anthropic `sk-ant-api03-...` API keys in `openclaw models auth paste-token --provider anthropic`, or to make the separate API-key auth path explicit.

## Best Possible Solution

Widen the shared Anthropic paste-token validator to accept both `sk-ant-oat01-` setup tokens and `sk-ant-api03-` API keys, add focused validator and command-path regression coverage, and lightly adjust Anthropic token-path messaging if needed so API-key users are not told they pasted only a setup token.

## Reproduction Assessment

Yes. The current source gives a high-confidence reproduction without a real secret: an Anthropic `paste-token` value starting with `sk-ant-api03-` reaches a validator that only accepts `sk-ant-oat01-` and returns the reported error.

## Solution Assessment

Yes. A narrow validator/test update is the most maintainable fix because it keeps the existing command path and token storage behavior while making the accepted prefix set match the runtime’s existing treatment of `sk-ant-api03-` as a regular Anthropic API key; the safer alternative is explicit CLI redirection to the existing `api-key` method if maintainers reject API keys on `paste-token`.
