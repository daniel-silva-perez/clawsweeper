## Summary

Keep open. Current main still rejects the reported Anthropic `sk-ant-api03-` input in the `paste-token` path before any profile is written, while Anthropic API-key auth exists as a separate `api_key` contract and the focused fix PR was closed unmerged.

## What This Changes

The issue asks OpenClaw to accept standard Anthropic `sk-ant-api03-` API keys through `openclaw models auth paste-token --provider anthropic`, or clearly direct users to the API-key auth path instead of rejecting them as invalid setup tokens.

## Best Possible Solution

Anthropic `paste-token` should distinguish credential families: `sk-ant-oat01-` remains setup-token auth, while `sk-ant-api03-` is either routed into the existing API-key profile/mode path or rejected with an actionable API-key command that does not imply the key is invalid.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: with provider `anthropic`, any pasted value beginning `sk-ant-api03-` reaches `validateAnthropicSetupToken`, fails the only-prefix check, and returns the reported error before profile storage.

## Solution Assessment

No. The current behavior is not the best way to solve the issue because OpenClaw already has a separate Anthropic API-key contract; the fix should bridge or explain that split rather than treating standard API keys as invalid setup tokens.
