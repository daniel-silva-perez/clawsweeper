## Summary

Keep open. Current main and the latest provided release still route Anthropic `paste-token` input through an `sk-ant-oat01-`-only setup-token validator, while the related focused PR was closed unmerged and the broader open relaxation PR is not a safe substitute for the API-key/token distinction.

## What This Changes

The issue asks OpenClaw to handle standard Anthropic `sk-ant-api03-` API keys in the `openclaw models auth paste-token --provider anthropic` flow, or to give a clear API-key auth path instead of a setup-token prefix error.

## Best Possible Solution

Anthropic auth should distinguish setup tokens from standard API keys in the CLI path: `sk-ant-oat01-` should remain setup-token auth, while `sk-ant-api03-` should either be persisted as an API-key profile through the existing API-key contract or rejected with a direct API-key command that does not imply the key is invalid.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: any non-empty Anthropic pasted value beginning with `sk-ant-api03-` reaches `validateAnthropicSetupToken`, fails the `sk-ant-oat01-` prefix check, and returns the reported error before profile storage.

## Solution Assessment

No. The current behavior is not the best way to solve the issue because OpenClaw already has a distinct Anthropic API-key path, and a broad prefixless token relaxation would blur API-key and OAuth/setup-token semantics.
