## Summary

Keep open. Current main still rejects `sk-ant-api03-...` on the reported `models auth paste-token --provider anthropic` path because that path calls a validator that only accepts `sk-ant-oat01-`; the latest release tag shows the same behavior. The separate Anthropic API-key auth method exists, but it does not fix the generic paste-token path described in this issue, and the targeted implementation PR was closed unmerged while #50975 remains broader than this bug.

## What This Changes

Allow `openclaw models auth paste-token --provider anthropic` to accept standard `sk-ant-api03-...` Anthropic API keys, or clearly route those users to the Anthropic API-key auth path instead of rejecting them as non-setup tokens.

## Best Possible Solution

Keep this issue open until the Anthropic paste-token path either accepts the explicit `sk-ant-api03-` API-key family with focused regression coverage, or rejects it with a clear pointer to the existing Anthropic API-key auth method. Avoid broad prefixless acceptance unless maintainers intentionally approve that validation policy.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence without a real secret: any pasted Anthropic value starting with `sk-ant-api03-` reaches `validateAnthropicSetupToken`, fails the `sk-ant-oat01-` prefix check, and returns the reported error.

## Solution Assessment

Yes, a narrow explicit-prefix validator update plus tests is the smallest maintainable fix if `paste-token` is intended to be a generic manual credential paste path. If maintainers want semantic separation, the safer alternative is a targeted `sk-ant-api03-...` rejection that directs users to `models auth login --provider anthropic --method api-key` or the equivalent API-key route.
