## Summary

Keep open. Current `main` and `v2026.4.29` still reject the reported Anthropic `paste-token` input through the `sk-ant-oat01-`-only validator, and the exact targeted PR was closed unmerged. The API-key auth route exists, so the remaining fix should be narrow rather than treating this as resolved or relying on the broader open relaxation PR.

## What This Changes

The issue asks for `openclaw models auth paste-token --provider anthropic` to accept `sk-ant-api03-` API keys or clearly direct users to the separate Anthropic API-key auth path.

## Best Possible Solution

Keep this open until Anthropic `paste-token` handles `sk-ant-api03-` intentionally: either route it through API-key storage or, more conservatively, reject that prefix with a direct pointer to `openclaw models auth login --provider anthropic --method api-key --set-default`. Avoid broad prefixless token acceptance.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: any non-empty Anthropic pasted value starting with `sk-ant-api03-` reaches `validateAnthropicSetupToken`, fails the `sk-ant-oat01-` prefix check, and returns the reported error before profile storage.

## Solution Assessment

No. The current behavior is not the best solution because the rejection does not guide users to the existing API-key path, and #50975's broad length-based relaxation is less precise than handling the explicit `sk-ant-api03-` family or redirecting it to API-key auth.
