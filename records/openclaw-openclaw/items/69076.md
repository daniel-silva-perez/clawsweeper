## Summary

Current main still does not implement this PR’s intended behavior. Manual-cancel exec exits are still classified through the failed path, notify-on-exit still wakes for manual-cancelled background execs with output, process kill/remove still return failed tool details after successful termination requests, and the Telegram sent-message cache still stores timestamp-only records. The PR remains a focused implementation candidate, with the main review question being the command/process cancellation semantics boundary.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to validate and land a version that classifies explicit manual cancellations as `killed`, suppresses notify-on-exit for killed sessions, maps successful kill/remove control actions to completed tool details, preserves failed/timeout/spawn-error behavior, and keeps the Telegram metadata compatibility coverage if that diagnostic change remains in scope.
