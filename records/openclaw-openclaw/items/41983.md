## Summary

Keep this PR open. Current main still aggregates known control-plane write clients by `deviceId+clientIp` and only uses `connId` as a fallback when both identity and IP are unknown, so the PR's intended per-connection isolation is not implemented or superseded. The PR is a focused gateway policy/security change, not a cleanup-only change, and needs maintainer review before merge or manual close.

## Best Possible Solution

Keep the PR open for maintainer policy review or an author update. If maintainers want per-connection write budgets, rebase onto current main, preserve bucket pruning and the 10,000-entry cap, rename/update the stale keying test, add a same-device/IP multi-connection regression test, and update gateway docs/changelog for the new rate-limit scope. If maintainers prefer the current aggregate `deviceId+clientIp` policy, close the PR manually with that product/security rationale.
