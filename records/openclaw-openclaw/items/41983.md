## Summary

Keep this PR open. Current main still aggregates known control-plane write clients by `deviceId+clientIp` and only falls back to `connId` when both identity and IP are unknown, so the PR's intended per-connection isolation is not implemented or superseded. The earlier review discussion remains relevant: this is a gateway control-plane policy/security change with a real memory-management concern to preserve on rebase, not a cleanup-only PR to auto-close.

## Best Possible Solution

Keep this PR open for maintainer policy review or an author update. If per-connection write budgets are desired, rebase onto current main, preserve bucket pruning and the 10,000-entry cap, rename/update the stale keying test, add a same-device/IP multi-connection regression test, and update gateway docs/changelog for the new rate-limit scope. If maintainers prefer the current aggregate `deviceId+clientIp` policy, close the PR manually with that product/security rationale.
