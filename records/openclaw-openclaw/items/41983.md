## Summary

Keep this PR open. Current main still aggregates control-plane write budgets by `deviceId|clientIp` whenever either identity signal is present, while this PR proposes adding `connId` to isolate concurrent gateway sessions. That changes throttling for `config.apply`, `config.patch`, and `update.run`, so it remains a real gateway policy/security decision rather than conservative cleanup-close material.

## Best Possible Solution

Keep this PR open for maintainer policy review or an author update. If per-connection control-plane write budgets are desired, rebase onto current main, preserve the bucket pruning and 10,000-entry cap, add a same-device/IP multi-connection regression test plus memory-bound coverage for the higher-cardinality key path, update the gateway docs/changelog, and fill in the security impact. If maintainers prefer the current aggregate `deviceId+clientIp` policy, close this PR manually with that product/security rationale.
