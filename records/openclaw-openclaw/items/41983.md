## Summary

Keep this PR open. Current main still aggregates control-plane write rate limits by `deviceId+clientIp` for identified or known-IP clients, and only uses `connId` as a fallback when both identity and IP are missing. The PR's intended per-connection isolation is therefore not implemented or superseded, but it needs maintainer policy review and an author update before merge because it changes gateway control-plane write-budget semantics.

## Best Possible Solution

Keep this PR open for maintainer policy review or an author update. If per-connection control-plane write budgets are desired, rebase onto current main, preserve the pruning and 10,000-entry cap, update the stale keying test, add a same-device/IP multi-connection regression test, and update the gateway docs/changelog. If maintainers prefer the current aggregate `deviceId+clientIp` policy, close the PR manually with that product/security rationale.
