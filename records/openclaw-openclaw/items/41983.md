## Summary

Keep this PR open. Current main still keys known control-plane write clients by deviceId+clientIp and only falls back to connId when both identity and IP are unknown, so the PR's intended per-connection isolation is not implemented or superseded. The PR is a real gateway policy/security change, not cleanup-only churn.

## Best Possible Solution

Keep this PR open for maintainer policy review or an author update. If per-connection write budgets are desired, rebase onto current main, preserve the bucket pruning and 10,000-entry cap, update the stale keying test, add a same-device/IP multi-connection regression test, and update the gateway docs/changelog for the new rate-limit scope. If maintainers prefer the current aggregate `deviceId+clientIp` policy, close the PR manually with that product/security rationale.
