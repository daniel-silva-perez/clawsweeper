## Summary

Keep open. Current main still lacks the Feishu user-facing fallback/hint for blocked local `MEDIA:` paths, so this PR is not obsolete; however the patch needs maintainer review because it has a one-time fallback guard race and should be rebased against the current dispatcher/docs plus the complementary media-root work in #51375.

## What This Changes

The PR adds Feishu blocked local `MEDIA:` path detection with a runtime log hint and one-time final-reply plaintext fallback, plus dispatcher tests and setup/FAQ documentation notes.

## Best Possible Solution

Rebase the runtime fallback onto current main, set the fallback guard before any awaited cleanup, keep tests around both serial and overlapping final failures, and align the docs with the current `tools.fs.workspaceOnly` trust model while coordinating with #51375 for allowed-root plumbing.

## Reproduction Assessment

Yes. A focused reproduction is a Feishu final reply containing a local `MEDIA:` path outside allowed roots; `loadWebMedia`/`assertLocalMediaAllowed` throws `LocalMediaAccessError` with `path-not-allowed`, and current main's Feishu `onError` only logs and closes out.

## Solution Assessment

No, not as written. The direction is narrow and maintainable, but the one-time fallback guard is set after an awaited cleanup even though the shared dispatcher does not await `onError`, so the patch should be adjusted before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Set the fallback guard before awaiting cleanup:** `extensions/feishu/src/reply-dispatcher.ts:402-403`
  - body: `createReplyDispatcher` invokes `onError` without awaiting it, but this handler waits for `closeStreaming()` before setting `localMediaFallbackSent`. If two final media sends fail in the same reply cycle, both handlers can observe the guard as false and send duplicate fallback messages. Set the guard before the first await and cover overlapping final failures.
