## Summary

Keep open. Current main still lacks the Feishu-specific user-facing fallback when a blocked local `MEDIA:` path fails during final reply delivery, and the PR’s runtime direction remains useful. It should get maintainer review and a current-main rebase because the docs text has since moved to a more nuanced `tools.fs.workspaceOnly` model and #51375 covers the complementary root-plumbing fix.

## What This Changes

This PR adds Feishu blocked local `MEDIA:` path detection with a runtime hint and one-time final-reply plain-text fallback, plus dispatcher tests and setup/FAQ documentation notes.

## Best Possible Solution

Keep the narrow Feishu recovery behavior, adapt it onto current main’s dispatcher state, retain the current `tools.fs.workspaceOnly` docs model instead of duplicating older wording, and coordinate with #51375 so allowed local media sends work while genuinely disallowed paths produce a clear fallback.

## Reproduction Assessment

Yes. A focused reproduction is a Feishu final reply containing a local `MEDIA:` path outside allowed roots; current media loading throws `LocalMediaAccessError` with `path-not-allowed`, and current Feishu `onError` only logs and closes out.

## Solution Assessment

Partly yes. The runtime fallback is a narrow maintainable Feishu-channel recovery path, but the docs portion should be reconciled with current main and the final implementation should account for the related media-root plumbing in #51375.
