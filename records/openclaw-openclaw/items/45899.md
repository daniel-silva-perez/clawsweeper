## Summary

Keep open. Current main still lacks the Feishu local-media blocked-path user fallback, and the PR has plausible narrow value, but the branch should not merge as written because its fallback guard is set only after awaited streaming cleanup. It also needs a rebase/docs alignment with the current media trust-model docs and coordination with the complementary open Feishu media-roots PR.

## What This Changes

The PR adds Feishu detection for disallowed local `MEDIA:` path errors, logs a hint, sends a one-time final-reply plaintext fallback, covers that path with dispatcher tests, and adds setup/FAQ documentation notes.

## Best Possible Solution

Rebase the runtime fallback onto current main, set the fallback guard before any awaited cleanup, add coverage for overlapping final failures as well as serial failures, and keep the docs consistent with the current `tools.fs.workspaceOnly` media trust model while coordinating with #51375 for allowed-root plumbing.

## Reproduction Assessment

Yes. A focused reproduction is a Feishu final reply containing a local `MEDIA:` path outside allowed roots; `loadWebMedia`/`assertLocalMediaAllowed` throws `LocalMediaAccessError` with `path-not-allowed`, and current main's Feishu `onError` only logs and closes out.

## Solution Assessment

No, not as written. The direction is narrow and maintainable, but the one-time fallback guard must be set before the first await because the shared dispatcher does not await `onError`.
