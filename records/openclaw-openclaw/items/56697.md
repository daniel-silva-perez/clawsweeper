## Summary

Keep this PR open: current main still does not add tool-specific examples to empty/missing parameter validation errors, so the requested UX improvement remains useful. The proposed patch should not merge as-is because its `edit` example no longer matches the current `edits[]` contract, and the prior review comments correctly flagged additional cleanup in the same diff.

## What This Changes

The PR changes `src/agents/pi-tools.params.ts` to append hard-coded read/write/edit examples to parameter validation errors, pass the tool name into the error formatter, and collect accepted key labels during missing-parameter checks.

## Best Possible Solution

Revise this PR or replace it with a focused patch that matches current `REQUIRED_PARAM_GROUPS`, uses a valid `edit({"path":..., "edits":[...]})` example, removes dead code, and adds regression tests for empty-object and missing-field validation messages. Keep broader retry-loop prevention separate, since #68070 is tracking that surface.

## Reproduction Assessment

Yes for the validation-message surface: current main can be checked through `assertRequiredParams({}, REQUIRED_PARAM_GROUPS.read, "read")` or the existing empty-record test, and it emits the generic missing-parameter message without an example. Live Kimi self-correction speed was not reproduced here.

## Solution Assessment

No, not as currently written. The direction is narrow and plausible, but the patch should be rebased to the current canonical tool schema, use a valid edit payload example, and add tests before merge.
