## Summary

Current main still lacks the requested native parent-scoped send path for sub-agents. Broad opt-in `sessions_send`, announce-chain delivery, and ACP `streamTo: "parent"` exist, but there is no `sessions_send_parent` tool, no `sessionKey: "parent"` resolver, and no direct-parent-only visibility rule enforcing the no-cross-agent/no-chaining constraints.

## Best Possible Solution

Keep this open for maintainer design. The best direction is either a dedicated `sessions_send_parent` tool or a `sessionKey: "parent"` resolver that derives the direct parent from trusted spawn metadata, enforces same-tree/direct-parent-only access, bypasses broad arbitrary-session visibility, avoids ancestor auto-forwarding, and adds policy, resolver, docs, and regression tests for allowed parent sends plus denied sibling, ancestor, cross-agent, and missing-parent cases.

## Risks / Open Questions

- A native parent-send path changes session messaging and visibility boundaries; an overly broad implementation could allow cross-agent, sibling, or ancestor messaging that the issue explicitly rejects.
- Existing opt-ins are useful but high-trust: `tools.subagents.tools.allow/alsoAllow` plus `tools.sessions.visibility=agent/all` can expose more than the direct parent, so they do not satisfy the requested security shape.
- ACP `streamTo: "parent"`, sub-agent announce delivery, and related duplicate-message issues are adjacent but not substitutes for a child-initiated native parent-question/progress tool for normal sub-agents.
