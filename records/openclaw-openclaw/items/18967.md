## Summary

Current main still does not provide a native parent-scoped session-send path for native sub-agents. The existing behavior supports announce-chain delivery, ACP-only streamTo:"parent", and a broad config opt-in that can re-enable sessions_send, but none of those implement a trusted direct-parent-only send API with same-tree and no-chaining guarantees.

## Best Possible Solution

Keep this issue open for maintainer design. The best implementation is a dedicated sessions_send_parent tool or a sessionKey:"parent" resolver that derives the direct parent from trusted spawn/session metadata, enforces same-tree/direct-parent-only access, rejects sibling/ancestor/cross-agent/missing-parent targets, avoids automatic forwarding beyond that parent, and updates policy, resolver, docs, and regression tests.

## Risks / Open Questions

- A broad sessions_send opt-in is not equivalent to a direct-parent-only API and can widen session access when paired with broader tools.sessions.visibility settings.
- A native parent send implementation would touch cross-session messaging policy, so it needs explicit maintainer security/design review before an automated fix PR.
- ACP streamTo:"parent" is a separate ACP-only streaming path and should not be treated as solving native sub-agent parent sends.
