## Summary

Current main still does not implement a native parent-scoped send path for sub-agents. It has broad opt-in paths that can re-enable `sessions_send`, plus announce/ACP parent-stream behavior, but no `sessions_send_parent` tool, no `sessionKey: "parent"` resolver, and no direct-parent-only visibility rule that enforces the requested no-cross-agent/no-chaining constraints.

## Best Possible Solution

Keep this open for a focused design/implementation of a native parent-scoped send path. The best code direction is either a dedicated `sessions_send_parent` tool or an explicit `sessionKey: "parent"` resolver that derives the direct parent from trusted spawn metadata, enforces same agent tree and direct-parent-only access, bypasses broad arbitrary-session send visibility, and does not auto-forward to ancestors. Add policy, resolver, visibility, docs, and regression tests covering allowed direct parent send plus denied sibling, ancestor, cross-agent, and missing-parent cases.

## Risks / Open Questions

- A native parent-send path touches session visibility and tool policy boundaries; a loose implementation could accidentally allow cross-agent or cross-tree messaging.
- Existing broad opt-ins (`tools.subagents.tools.allow/alsoAllow` plus `tools.sessions.visibility=agent/all`) are useful but do not satisfy the issue's parent-only/no-chaining safety constraints.
- ACP `streamTo: "parent"` and sub-agent announce delivery are easy to confuse with this request, but they do not provide a native child-initiated question/progress tool for normal sub-agents.
