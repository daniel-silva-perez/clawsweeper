## Summary

Keep this PR open, but it should not merge as-is. The reported Grok/xAI web_search defects are still visible on current main, while the PR applies the fix to a stale provider file that current main no longer contains or loads and also carries unrelated status-command changes.

## What This Changes

The PR changes an older Grok web_search provider to guard malformed xAI response entries, convert internal timeout AbortErrors into structured tool errors, and use a 60s Grok timeout default, while also changing status-command loading and status model alias handling.

## Best Possible Solution

Port only the Grok/xAI web_search fixes into the current xAI plugin runtime: harden the shared Responses parser, convert provider-internal timeout aborts into structured tool errors while preserving the original cause, keep explicit timeout overrides, use a Grok-specific 60s default when unset, and add focused regression coverage. Leave the status-command changes out of the repair.

## Reproduction Assessment

Yes. The parser crash is reproducible by calling `extractXaiWebSearchContent` on current main with null entries in `output` or `content`, and #58733 gives a current-main timeout-boundary reproduction where a 30s request fails while a 60s request succeeds.

## Solution Assessment

No. The intended fix direction is sound, but this PR is not the best merge shape because it targets a removed/unloaded provider file and mixes unrelated status-command edits; a narrow replacement against the live xAI runtime is safer.
