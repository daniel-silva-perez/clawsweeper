## Summary

Keep open. Current main still resolves the status auth label from the session authProfileOverride while provider usage auth resolves OAuth credentials from provider order only, so the linked bug is not implemented on main. The PR is useful source work, including the later plugin-provider override fix, but it needs a current-main port because the status text builder has moved and the linked bug remains open.

## What This Changes

The PR threads a session-selected OAuth profile into provider usage summary resolution, validates preferred profile/provider matching, preserves it through plugin provider overrides, adds regression coverage, and includes Swift ModelChoice alias fields.

## Best Possible Solution

Land a narrow current-main implementation that passes the active session OAuth profile into usage summary resolution, validates that the preferred profile belongs to the resolved usage credential provider, preserves it through plugin provider override lookups, and covers both the direct openai-codex case and plugin override case with focused tests.

## Reproduction Assessment

Yes. With multiple openai-codex OAuth profiles and sessionEntry.authProfileOverride selecting the active profile, current main has a source-level reproduction path: the displayed label tries the override first, while usage auth still walks provider order only.

## Solution Assessment

Mostly yes. The PR's preferredProfileIds seam with a provider-match guard is the narrow maintainable direction, but it should be ported to the current shared status builder and should drop unrelated Swift alias churn that is already present on main.
