## Summary

Keep open. Current main already has the normal top-level nullable `finish_reason` and dedicated empty-delta stop chunk path, but the streaming error path still combines the error text with `finish_reason: "stop"`; this PR addresses useful work, but it should not merge as-is because it bypasses the current usage-aware finalizer and adds a test assertion that conflicts with the helper's nullable `finish_reason` output.

## What This Changes

This PR changes the Gateway `/v1/chat/completions` streaming helper and tests so streamed chat chunks use top-level `finish_reason`, terminal stop chunks use empty deltas, lifecycle end emits a stop chunk, and streaming errors emit error text before a separate stop signal.

## Best Possible Solution

Carry forward the error-path split as a small current-main patch: emit the error text as a nullable non-final content chunk, let the existing finalization path emit the empty-delta stop chunk and optional usage chunk, preserve disconnect cleanup, and update the focused Gateway streaming tests plus a user-facing changelog fix entry.

## Reproduction Assessment

Yes. The focused `stream=true` test path with a mocked rejected `agentCommand` exercises the bug, and current main's test already asserts the old combined error text plus stop choice behavior.

## Solution Assessment

No. The requested direction is right, but this branch is not the best implementation because it closes streams outside `maybeFinalize()` and has a mismatched test; the safer fix is to split only the error content and stop signal while preserving the existing finalizer contract.
