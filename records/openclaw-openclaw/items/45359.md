## Summary

Keep open. Current main has related OpenAI Codex OAuth refresh hardening and Codex `server_error` failover classification, but I did not find evidence that it fixes the narrower reported workflow where `openai-codex/gpt-5.4` fails immediately after OAuth refresh and recovers only after manually switching models.

## What This Changes

The issue requests that `openai-codex/gpt-5.4` continue working after OAuth refresh without requiring a manual switch to `openai-codex/gpt-5.3-codex` and back.

## Best Possible Solution

Keep this open and investigate the Codex-specific post-refresh path: collect current request IDs/timestamps on v2026.4.27 or newer, then add a narrowly gated refresh/rebind and one-time same selected model retry only when the failed turn is replay-safe.

## Reproduction Assessment

Unclear. The issue gives plausible manual steps, but no request IDs/logs are included and I did not live-exhaust an OAuth account; code inspection only confirms that the exact regression path is not covered.

## Solution Assessment

No, the current generic OAuth refresh and configured fallback handling are not sufficient for the reported manual-switch recovery pattern. The maintainable fix should be a Codex-specific auth/transport refresh or session rebind plus a replay-safe same-model retry, not broad silent fallback for user-selected models.
