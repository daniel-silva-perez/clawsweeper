Keep open. Current main has OpenAI Codex OAuth refresh hardening, Codex server_error fallback classification, and a newer boundary-aware OAuth bearer injection fix, but I did not find evidence that the exact reported workflow is fixed: openai-codex/gpt-5.4 returns server_error immediately after OAuth refresh and recovers only after switching to another model and back.

## Best Possible Solution

Keep this issue open for focused maintainer investigation. The best path is to collect current request IDs/timestamps on current main or the next release, then add a replay-safe OpenAI Codex post-refresh recovery path that refreshes or rebuilds the Codex auth/transport context and retries the same selected model once only before any side effects have committed.
