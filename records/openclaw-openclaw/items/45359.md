## Summary

Keep open. Current main has substantial related hardening for OpenAI Codex OAuth refresh and Codex `server_error` fallback, but I did not find a source/test path that proves the exact reported workflow is fixed: GPT-5.4 returning `server_error` immediately after OAuth refresh and recovering without the user manually switching models. The existing coverage proves token refresh persistence and configured fallback handoff, not refresh-triggered same-model/session rebind recovery.

## Best Possible Solution

Keep this issue open and add a focused OpenAI Codex regression path: when `openai-codex/gpt-5.4` returns a replay-safe `server_error` immediately after OAuth refresh, invalidate/rebuild the PI/OpenAI Codex auth or transport session context and retry once on the same selected model before surfacing the error or using configured fallbacks. If maintainers believe the current OAuth hardening already fixed it, ask the reporter for request IDs/timestamps on current `v2026.4.26+` to confirm before closing.

## Evidence

- **Issue context:** The reporter describes `openai-codex/gpt-5.4` intermittently returning upstream `server_error` after limits/OAuth refresh, with recovery only after manually switching to `openai-codex/gpt-5.3-codex` and back. The follow-up comment is still a hypothesis and offers request IDs/timestamps, but the supplied context does not include those logs.
- **OpenAI Codex provider refresh hook exists:** The OpenAI Codex plugin refreshes OAuth credentials via `refreshOpenAICodexToken`, merges refreshed fields back into the existing credential, and registers that function as the provider `refreshOAuth` hook. This supports normal refresh, but it is not a retry/rebind path for a post-refresh `server_error` on GPT-5.4.
