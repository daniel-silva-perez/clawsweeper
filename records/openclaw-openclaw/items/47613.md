## Summary

Keep open. The underlying Gateway/WebSocket reasoning-stream gap is still visible on current main, but this branch is not a safe merge as-is because it would not preserve today’s reasoning visibility gate and it does not cover the current Gateway path that must actually pass the session reasoning level into the embedded runner.

## What This Changes

The PR changes `src/agents/pi-embedded-subscribe.ts` so `reasoningMode: "stream"` can broadcast `stream: "thinking"` agent events without requiring `onReasoningStream`, while still calling that callback when present.

## Best Possible Solution

Land one canonical Gateway reasoning-stream implementation that removes only the channel-callback dependency, preserves stream-mode, thinking-enabled, and silent-turn gates, forwards session reasoning into the embedded runner, adds no-callback WebSocket regression coverage, and updates `docs/tools/thinking.md` if Gateway streaming is now supported.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a Gateway `chat.send` run for a session configured with `reasoningLevel: "stream"`: the Gateway path does not pass `onReasoningStream`, and current main returns before emitting `stream: "thinking"`. Full end-to-end proof should also verify the session reasoning level reaches `runEmbeddedPiAgent`.

## Solution Assessment

No. The direction is useful, but this exact patch is not the best fix against current main because it makes `streamReasoning` depend only on stream mode, drops the current `canShowReasoning` protection, and does not address the current Gateway reasoning-level forwarding and docs/test gaps.
