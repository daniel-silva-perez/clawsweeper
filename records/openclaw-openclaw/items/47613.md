## Summary

Keep open. The PR targets a real current-main Gateway reasoning-stream gap, but the submitted patch is not safe to merge as-is against current main because it drops the current reasoning visibility gate and does not address the Gateway command path that still fails to forward session reasoning into the embedded runner.

## What This Changes

The PR changes `src/agents/pi-embedded-subscribe.ts` so `streamReasoning` no longer depends on `onReasoningStream` and `emitReasoningStream()` broadcasts `stream: "thinking"` events while calling the channel callback only when present.

## Best Possible Solution

Land one canonical Gateway reasoning-stream implementation that removes only the channel-callback dependency, preserves stream-mode, thinking-enabled, and silent-turn gates, forwards session `reasoningLevel` into the embedded runner, adds no-callback WebSocket regression coverage, and updates `docs/tools/thinking.md` if Gateway streaming becomes supported.

## Reproduction Assessment

Yes. A high-confidence source reproduction is a Gateway `chat.send` run for a session patched with `reasoningLevel: "stream"`: current main stores that value, but the Gateway path does not pass `onReasoningStream` and `runAgentAttempt()` still omits `reasoningLevel` when calling `runEmbeddedPiAgent()`.

## Solution Assessment

No. The direction is useful, but this exact patch is incomplete and unsafe against current main; the narrower maintainable fix is to decouple only the callback gate while preserving visibility gates and plumbing session reasoning through the Gateway command path.
