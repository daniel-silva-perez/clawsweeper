## Summary

Keep open: the PR targets a real current Gateway/WebSocket reasoning-stream gap, but it is not ready to merge against current main because the proposed `streamReasoning` assignment would drop the current reasoning-visibility guard during rebase. The overlapping open PRs should be reconciled by a maintainer before choosing the canonical fix.

## What This Changes

The PR modifies `src/agents/pi-embedded-subscribe.ts` so `reasoningMode: "stream"` can emit `stream: "thinking"` agent events without an `onReasoningStream` callback, while still invoking the callback when present.

## Best Possible Solution

Land one canonical Gateway reasoning-stream implementation that decouples the WebSocket/node broadcast from channel-specific callbacks, preserves explicit stream mode, thinking-enabled, and silent-turn gates, adds no-callback Gateway/WebSocket regression coverage, and updates `docs/tools/thinking.md` if Gateway streaming becomes supported.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to set a Gateway session to `reasoningLevel: "stream"` with thinking enabled and send through `chat.send`: the Gateway path does not pass `onReasoningStream`, and current main keeps `streamReasoning` false or returns before emitting `stream: "thinking"`.

## Solution Assessment

No. The direction is useful, but this exact patch is not the best solution against current main because it removes the callback requirement by replacing the whole stream gate; the safer fix removes only the callback dependency while preserving `canShowReasoning` and `silentExpected` behavior.
