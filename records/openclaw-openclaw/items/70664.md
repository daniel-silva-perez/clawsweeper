## Summary

Keep open. Current main still has the reported Unix/macOS behavior: `lsof -Fpc` listener PIDs are filtered by the `c` command token before full argv verification, so a Node-backed gateway reported as `cnode` can be dropped. The issue is also paired with open, unmerged PR #70681, so it should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue reports that macOS/Unix stale gateway PID cleanup filters `lsof -Fpc` results by command name, dropping Node-backed `openclaw-gateway` listeners before full argv verification can occur.

## Best Possible Solution

Land #70681 or an equivalent narrow fix that treats `lsof` as a raw listener-PID source, verifies Unix/macOS candidates with full argv inspection before cleanup and port-free polling, preserves self and ancestor exclusions, and updates regression coverage for `cnode` gateway listeners and unrelated non-gateway listeners.

## Reproduction Assessment

Yes. The issue gives a concrete macOS reproduction, and current-main inspection gives a focused deterministic path: feed `lsof -Fpc` output containing `p<PID>\ncnode` for a real gateway and `findGatewayPidsOnPortSync` drops it before argv verification can inspect the process.

## Solution Assessment

Yes. The proposed direction of verifying Unix listener PIDs by full argv is the narrowest maintainable fix because it mirrors the Windows path and avoids trusting `lsof`'s kernel command field for a Node process whose visible title is rewritten after exec.
