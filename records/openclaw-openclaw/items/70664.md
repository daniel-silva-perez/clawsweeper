## Summary

Keep open. Current main still has the reported Unix/macOS behavior: listener PIDs from `lsof -Fpc` are filtered by the `c` command token before full argv verification can run, so a real Node-backed gateway reported as `cnode` is dropped. The issue is also paired with open, unmerged PR #70681, so it should remain open until that PR or an equivalent fix lands.

## What This Changes

Requests fixing macOS stale gateway cleanup so Unix listener PIDs are verified through full process argv instead of being dropped by the `lsof` `p_comm` command name.

## Best Possible Solution

Land #70681 or an equivalent narrow fix that treats `lsof` as a listener-PID source only, verifies Unix/macOS candidates with full argv inspection before cleanup or port-free polling, preserves self and ancestor exclusions, and updates regression coverage for `cnode` gateway listeners and unrelated non-gateway listeners.

## Reproduction Assessment

Yes. The report gives a concrete macOS reproduction, and current-main inspection shows a deterministic focused reproduction: feed `lsof -Fpc` output containing `p<PID>\ncnode` for a real gateway and `findGatewayPidsOnPortSync` drops it before the existing `ps`-based verifier can inspect argv.

## Solution Assessment

Yes. Verifying Unix listener PIDs by full argv is the narrowest maintainable fix because it mirrors the Windows path and avoids trusting `lsof`'s kernel command field for a Node process whose visible title is rewritten after exec.
