## Summary

Keep open. Current main has a generic `registerControlUiDescriptor` and `plugins.uiDescriptors` projection, but it does not implement the requested slot vocabulary or Control UI behavior for chat modes, input guards, event classifiers, status surfaces, panel/native-page rendering, or plugin-owned UI actions. The active related PRs are explicitly partial follow-ups, not a full closeout.

## What This Changes

Request to add a declarative, data-only Control UI plugin contribution surface for chat modes, approval cards, event classifiers, input guards, status surfaces, and safe host-rendered plugin descriptors.

## Best Possible Solution

Keep this as the broad Control UI contribution-slot design thread until maintainers choose an additive, versioned, host-owned descriptor model. A complete solution should include typed schemas, gateway projection, Control UI consumers, plugin disablement semantics, sanitized browser rendering, and fixture coverage for enabled/disabled plugins and plugin-originated actions.

## Reproduction Assessment

Not applicable; this is an RFC/feature request rather than a bug. The gap is verifiable by source inspection on current main: only generic descriptor projection exists, with no slot-specific API or browser-side consumer for the requested contribution classes.

## Solution Assessment

Unclear. The requested direction is plausible, but the safest first implementation likely needs maintainer approval around a manifest-first, host-owned component catalog and versioned data-only schemas rather than a broad mutable runtime registry.
