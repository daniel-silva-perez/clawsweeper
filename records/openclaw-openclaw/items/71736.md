## Summary

Keep open. Current main only has a narrow generic Control UI descriptor registry and `plugins.uiDescriptors` projection; it does not provide the requested contribution-slot vocabulary or browser-side rendering/guard behavior, and the active related PRs are explicitly described as partial foundation work rather than a full closeout.

## What This Changes

Request data-driven Control UI plugin contribution slots for chat modes, approval/card rendering, event classifiers, input guards, status surfaces, safe host-owned descriptors, and enable/disable projection semantics.

## Best Possible Solution

Keep this as the broad Control UI contribution-slot design thread until maintainers choose an additive, versioned, host-owned descriptor model. The eventual implementation should include typed schemas, gateway projection, Control UI consumers, plugin disablement semantics, sanitization against arbitrary browser code, and fixture coverage for enabled/disabled plugins and plugin-originated actions.

## Reproduction Assessment

Not applicable; this is an RFC/feature request rather than a bug report. The verification path is source, docs, tests, and linked discussion inspection against current main, which shows a partial generic descriptor foundation but no full contribution-slot implementation.

## Solution Assessment

Unclear. The requested direction is plausible, but the safest maintainable first slice likely needs maintainer product/API judgment, with a manifest-first host-owned component catalog for panels/status surfaces safer than a broad mutable runtime registry.
