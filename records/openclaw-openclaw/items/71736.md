## Summary

Keep open. Current main only has a generic Control UI descriptor registration/projection foundation, while the requested broader slot vocabulary and browser rendering behavior remain missing. The issue is also paired with open same-author follow-up PRs and adjacent prototype discussion, so it should stay as a maintainer design thread rather than be swept closed.

## What This Changes

Request a declarative plugin SDK and Control UI contribution-slot model for plugin chat modes, approval cards, event classifiers, input guards, status surfaces, and safe host-owned rendering.

## Best Possible Solution

Keep this issue as the broad Control UI contribution-slot design thread. The eventual implementation should be additive, typed, and host-owned: descriptor schemas for the chosen slot vocabulary, gateway projection, Control UI consumers, plugin disablement semantics, sanitization against arbitrary browser code, and fixture coverage for enabled/disabled plugins and plugin-originated actions.

## Reproduction Assessment

Not applicable; this is an RFC/feature request rather than a bug report. The verification path is source/docs/test inspection against current main plus the active related discussion, which shows a generic descriptor foundation but no full contribution-slot implementation.

## Solution Assessment

Unclear. The requested direction is plausible, but the best maintainable solution needs maintainer API judgment; a manifest-first, host-owned component catalog for an initial panels/status-surface slice may be safer than starting with one broad mutable runtime registry.
