## Summary

Keep this open. Current main has the generic Control UI descriptor registration and Gateway projection foundation, but it does not implement the broader contribution-slot vocabulary or Control UI browser consumption requested here; the discussion is active and tied to open follow-up PRs plus an adjacent manifest-first panel prototype.

## What This Changes

The issue requests a declarative plugin SDK, Gateway, and Control UI surface for plugin chat modes, approval-card families, event classifiers, input guards, status surfaces, and related safe host-owned UI contribution descriptors.

## Best Possible Solution

Keep this as the design thread for the broader Control UI contribution-slot system. Maintainers should decide the manifest/runtime shape and initial host-owned component catalog, then land narrower additive slices that extend the current generic descriptor foundation into client-consumed slots with sanitization, disablement semantics, fixture coverage, and public SDK/docs alignment.

## Reproduction Assessment

Not applicable; this is an RFC/feature request rather than a bug. The high-confidence verification path is source and docs inspection against current main, which shows generic descriptor registration/projection but no broader slot vocabulary or Control UI client rendering.

## Solution Assessment

Unclear; a safe descriptor-based, host-rendered contribution model matches the repository direction, but the broad registerControlUiContribution shape needs maintainer API judgment. A manifest-first panel/status slice like the prototype discussed in comments may be a safer first increment than a broad mutable registry, while preserving room for chat modes, guards, classifiers, and status surfaces.
