## Summary

Keep open. Current main has a generic Control UI descriptor registry and a `plugins.uiDescriptors` gateway projection, but it does not implement the broader contribution-slot model requested here or any browser-side consumer for those descriptors; the linked discussion and open follow-up PRs also treat the current surface as partial.

## What This Changes

Add data-driven Control UI plugin contribution slots for plugin chat modes, approval cards, event classifiers, input guards, and status surfaces using safe host-owned descriptor rendering.

## Best Possible Solution

Keep this issue as the broad Control UI contribution-slot design thread until maintainers choose an additive, versioned, host-owned descriptor model. The eventual implementation should include typed schemas, gateway projection, Control UI consumers, plugin disablement semantics, sanitization against arbitrary browser code, and fixture coverage for enabled/disabled plugins and plugin-originated actions.

## Reproduction Assessment

Not applicable; this is an RFC/feature request, not a bug report. The verification path is source/docs/test inspection against current main plus the linked discussion, which shows a narrower generic descriptor foundation but no full contribution-slot implementation.

## Solution Assessment

Unclear. The requested direction is plausible, but the best maintainable first slice likely needs maintainer product/API judgment, with a manifest-first host-owned component catalog for panels/status surfaces safer than a broad mutable runtime registry.
