## Summary

Keep this open. Current main has a generic plugin Control UI descriptor registration and Gateway projection seam, but it does not implement the requested slot vocabulary or browser-side consumption for chat modes, input guards, event classifiers, approval-card families, status surfaces, panels, or native pages; the discussion is active and tied to open follow-up PRs, so this needs maintainer API/product review rather than cleanup closure.

## What This Changes

The issue requests a first-class plugin SDK and Control UI contribution-slot model for plugin chat modes, approval cards, event classifiers, input guards, and status surfaces.

## Best Possible Solution

Keep this as the Control UI contribution-slot design thread until maintainers choose the descriptor model and slot vocabulary, then land additive SDK/Gateway schemas, host-owned browser renderers, disablement semantics, sanitization, fixture coverage, and docs without allowing arbitrary plugin JavaScript in the Control UI.

## Reproduction Assessment

Not applicable; this is an RFC/feature request, not a bug report. The high-confidence verification path is current-main source and docs inspection, which shows only the generic descriptor foundation and no Control UI consumer for the requested slots.

## Solution Assessment

Unclear; the requested direction is plausible, but a broad mutable `registerControlUiContribution(...)` registry may not be the narrowest maintainable first step. A manifest-first, slot-specific descriptor slice for panels/status plus follow-up chat-mode/input-guard/event-classifier contracts looks safer to review.
