## Summary

Keep open. Current main has the generic host-hook descriptor foundation, but it does not implement the broader Control UI contribution-slot model requested here, and the active discussion/open PR stack explicitly treats this issue as still unresolved.

## What This Changes

Add data-driven plugin contribution slots for Control UI chat modes, approval cards, event classifiers, input guards, status surfaces, and safe host-owned rendering.

## Best Possible Solution

Keep this as the Control UI contribution-slot design thread. Maintainers should choose a manifest-first or runtime projection model, define a host-owned component catalog, then land additive slices with Gateway projection, Control UI rendering, disablement semantics, sanitization, and fixture coverage.

## Reproduction Assessment

Not applicable; this is an RFC/feature request, not a bug report. The verification path is source/docs inspection against current main, which shows generic descriptor plumbing but no broader slot vocabulary or Control UI consumer.

## Solution Assessment

Unclear; the requested direction is plausible, but the narrowest maintainable shape needs maintainer API judgment. A manifest-first, host-owned component slice for panels/status may be safer as a first increment than a broad mutable runtime registry.
