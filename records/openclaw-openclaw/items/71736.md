## Summary

Keep this open. Current main has a useful generic foundation for plugin Control UI descriptors, but it does not implement the specific contribution-slot vocabulary requested here, and the Control UI client does not yet consume those descriptors. The discussion is also active, with same-author follow-up PRs and a concrete external panel/status prototype from 1fanwang, so this should stay as a maintainer/API design thread rather than be swept closed.

## What This Changes

Request a first-class, data-driven Control UI plugin contribution surface for chat modes, approval cards, event classifiers, input guards, and status surfaces using safe host-owned descriptors.

## Best Possible Solution

Keep this issue as the Control UI contribution-slot design thread. Maintainers should decide whether the shipped generic descriptor model is the intended base, then accept or request a narrower follow-up that adds the missing host-owned slot descriptors, Gateway projection, Control UI rendering behavior, disablement semantics, sanitization, and fixture coverage without arbitrary plugin browser JavaScript.

## Reproduction Assessment

Not applicable; this is an RFC/feature request rather than a bug report. The high-confidence verification path is source and docs inspection against current main, which shows only the generic descriptor foundation and no client-side slot rendering.

## Solution Assessment

Unclear; the requested direction is plausible, but the best maintainable shape still needs maintainer API judgment. A manifest-first panel/status slice may be a safer first increment than a broad mutable runtime registry, while preserving a path for chat modes, guards, classifiers, and status surfaces.
