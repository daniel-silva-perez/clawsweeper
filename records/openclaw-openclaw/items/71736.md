## Summary

Keep open. Current main contains only a narrow generic Control UI descriptor registry and `plugins.uiDescriptors` projection; it does not implement the requested slot vocabulary, manifest/runtime design, or browser-side consumers for chat modes, input guards, event classifiers, status surfaces, panels/native pages, or richer approval-card families. The issue discussion also identifies active follow-up PRs and a forked `controlUiPanels` prototype as partial/adjacent work, not a closeout.

## What This Changes

Request an SDK, Gateway, and Control UI contribution-slot model for plugin-provided chat modes, approval cards, event classifiers, input guards, and status/status-panel surfaces using safe host-owned rendering.

## Best Possible Solution

Keep this as the broad Control UI contribution-slot design thread until maintainers choose an additive, versioned, host-owned descriptor model. The eventual implementation should define typed schemas, gateway projection, Control UI consumers, plugin disablement semantics, sanitization against arbitrary browser code, and fixture coverage for enabled/disabled plugins plus plugin-originated actions.

## Reproduction Assessment

Not applicable; this is an RFC/feature request rather than a bug report. The verification path is source, docs, tests, and discussion inspection against current main, which shows partial generic descriptors but no full contribution-slot implementation.

## Solution Assessment

Unclear. The requested direction is plausible, but the safest maintainable first slice likely needs maintainer API/product judgment, with manifest-first data descriptors and a host-owned component catalog preferred over arbitrary plugin JavaScript or a broad mutable browser registry.
