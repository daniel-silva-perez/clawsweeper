## Summary

Keep open. Current main still has no state-only hook permission, and `agent_end` remains gated as a raw conversation hook. There is an adjacent agent-event subscription API that may be a better state-only path, but choosing/documenting that versus adding a new minimal lifecycle hook is a product and security design decision.

## What This Changes

Request a fine-grained, state-only plugin hook permission or minimal lifecycle event so display/status plugins can observe agent activity without granting raw conversation access.

## Best Possible Solution

Keep raw-content hooks behind `allowConversationAccess` and provide a documented minimal lifecycle/state path for non-bundled plugins, either by formalizing filtered agent-event subscriptions for state monitors or by adding a new sanitized hook/event with session/run state only. Update the config schema/help, plugin hook docs, SDK docs, loader tests, and harness emission tests around that chosen boundary.

## Security Review

Status: needs_attention

Summary: This issue is a security-relevant permission design request; any implementation must preserve the raw conversation gate while exposing only minimal state.

Concerns:

- **[medium] Keep state access away from raw agent_end payloads:** `src/plugins/hook-types.ts:272`
  - body: `agent_end` currently carries `messages` and the embedded runner passes the conversation snapshot into it. A new state permission must not allow the existing payload to run for non-bundled plugins without `allowConversationAccess`.
  - confidence: 0.94
- **[medium] Do not treat content-bearing hooks as state-only:** `src/plugins/hook-types.ts:198`
  - body: `before_agent_reply` carries `cleanedBody`, and `before_tool_call` carries tool params. A state-only solution should use a minimal event shape or carefully filtered subscription instead of broadening these hooks.
  - confidence: 0.87
- **[low] Document safe agent-event stream use if chosen:** `src/plugins/host-hooks.ts:106`
  - body: `registerAgentEventSubscription` can filter streams, but some streams include assistant text or tool/result details. If this is the intended state API, docs and tests should steer state monitors to safe lifecycle/tool-phase data only.
  - confidence: 0.78
