## Summary

Keep open. Current main still has only the raw-conversation `allowConversationAccess` gate for `agent_end`, with no `allowStateAccess` or equivalent state-only permission. The linked OpenClaw Interaction Bridge is a concrete plugin ecosystem example: its README documents that status updates need `allowConversationAccess` on OpenClaw 2026.4.26+ even though it says no conversation data is read. ([github.com](https://github.com/snarflakes/openclaw-interaction-bridge/blob/development/README.md))

## What This Changes

Add a state-only plugin permission or minimal lifecycle event path so non-bundled plugins can track agent run state without granting raw conversation access.

## Best Possible Solution

Keep raw-content hooks behind `allowConversationAccess` and expose a documented minimal lifecycle/state contract for non-bundled plugins, either by formalizing filtered lifecycle agent-event subscriptions or adding a sanitized hook/event shape with only session/run state. Update config schema/help, plugin hook docs, SDK docs, and loader/host-hook tests around the chosen boundary.

  - confidence: 0.94

  - body: `before_agent_reply` carries user-facing text and `before_tool_call` carries tool params. A state API should use a sanitized event shape or filtered lifecycle stream instead of broadening those hooks.

  - body: `registerAgentEventSubscription` can support workflow monitors, but agent event streams can include assistant or tool details. Docs and tests should steer state monitors to lifecycle-only data.
