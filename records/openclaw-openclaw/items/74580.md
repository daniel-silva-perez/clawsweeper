## Summary

Keep open. Current main still treats `agent_end` as a raw conversation hook, still blocks non-bundled `agent_end` registration without `allowConversationAccess`, and has no state-only permission or turn-ended plugin contract. The related discussion usefully narrows the request to state-only `agent_end`/turn-ended access for plugins like OpenClaw Interaction Bridge, which is a real plugin API and security-boundary question rather than cleanup.

## What This Changes

Add a state-only permission or lightweight lifecycle contract so non-bundled plugins can detect agent turn completion without receiving raw conversation content.

## Best Possible Solution

Keep raw-content hooks behind `allowConversationAccess` and define a minimal lifecycle or turn-ended contract that exposes only run/session/state/timing fields. If maintainers choose the existing event-subscription seam instead, document it explicitly for state-display plugins and add contract tests proving no conversation text reaches that path.

## Reproduction Assessment

Yes. Load a non-bundled plugin that calls `api.on("agent_end", ...)` without `plugins.entries.<id>.hooks.allowConversationAccess=true`; the current loader test already codifies that registration is blocked.

## Solution Assessment

Unclear. The proposed `allowStateAccess` flag is plausible, but the existing `agent_end` payload contains raw `messages`, so the safer fix is a sanitized lifecycle/turn-ended contract or a documented event-subscription path rather than simply relaxing the existing hook gate.
