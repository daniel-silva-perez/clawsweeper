## Summary

Keep open. Current main still gates non-bundled `agent_end` behind full conversation access, the payload still includes raw messages, and there is no state-only hook permission or documented turn-ended plugin contract. The existing agent event subscription seam is adjacent, but the follow-up discussion correctly narrows this to a plugin API and security-boundary decision rather than cleanup.

## What This Changes

Add a state-only permission or lightweight lifecycle/turn-ended plugin contract so non-bundled plugins can detect agent turn completion without receiving raw conversation content.

## Best Possible Solution

Keep raw-content hooks behind `allowConversationAccess` and add or document a minimal lifecycle/turn-ended contract that exposes only run/session/state/timing fields. If maintainers choose the existing event-subscription seam, document it explicitly for state-display plugins and add contract tests proving no conversation text reaches that path.

## Reproduction Assessment

Yes. Load a non-bundled plugin that calls `api.on("agent_end", ...)` without `plugins.entries.<id>.hooks.allowConversationAccess=true`; current loader tests already codify that registration is blocked.

## Solution Assessment

Unclear. The proposed `allowStateAccess` flag is plausible, but the existing `agent_end` payload contains raw `messages`, so a sanitized lifecycle/turn-ended contract or explicitly documented event-subscription path is safer than relaxing the existing gate.
