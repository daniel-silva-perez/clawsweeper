## Summary

Keep open. Current main still treats `agent_end` as a conversation-content hook, still blocks non-bundled `agent_end` registrations without `allowConversationAccess`, and has no `allowStateAccess` or equivalent lightweight turn-ended plugin contract. The issue discussion also narrowed the request to state-only `agent_end`/turn-ended access, which is a valid plugin API/security boundary question rather than stale cleanup.

## What This Changes

Request a plugin API permission or event that lets external plugins observe sanitized agent turn/lifecycle end state without granting raw conversation access to `agent_end`.

## Best Possible Solution

Keep raw-content hooks behind `allowConversationAccess` and define a minimal lifecycle/turn-ended contract that exposes only run/session/state fields. If maintainers choose the existing event subscription seam instead, document it explicitly for status-display plugins, clarify per-run versus per-turn semantics, and add contract tests that prove no conversation text is delivered.

## Reproduction Assessment

Yes: the current limitation is reproducible by loading a non-bundled plugin that registers `agent_end` without `allowConversationAccess`; `src/plugins/loader.test.ts` already codifies that it is blocked. This is a feature/API gap rather than a runtime crash.

## Solution Assessment

Unclear: the proposed `allowStateAccess` flag is reasonable, but not proven to be the safest long-term shape because the current `agent_end` payload contains raw messages. A sanitized lifecycle-only event or documented lightweight turn-ended subscription would be safer than reclassifying the existing hook payload.
