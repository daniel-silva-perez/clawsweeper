## Summary

Keep open. The report is fresh, concrete, and not safely closeable from current `main`: `openclaw status` and `openclaw cron list` still traverse gateway RPC/read paths, and I found no current-main commit or test proving the WSL2 hang symptom is fixed. The cross-referenced #73381 is useful context but tracks an agent/sendPolicy failure path, while the reported `status` and `cron.list` paths do not directly invoke the gateway `agent` method.

## Best Possible Solution

Keep this issue open and ask for targeted diagnostics: `openclaw status --debug --timeout 5000`, `openclaw cron list --timeout 5000 --json`, gateway logs around the hang, and whether the CLI eventually exits with `gateway timeout`. In parallel, review #73381 for the narrower sendPolicy/agent bug, but do not use it as the sole canonical tracker unless maintainers confirm the same root cause for `status` and `cron.list`.
