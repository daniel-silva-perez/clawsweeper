## Summary

Keep this open. Current main still has the Control UI `AUTH_TOKEN_MISMATCH` path that can continue reconnecting when no cached device-token retry is available, and `chat.history` still reads/parses the full transcript synchronously before applying message and byte limits. The Slack Socket Mode timeout tuning is present, but the follow-up report says it only lengthened disconnect intervals and did not resolve inbound event loss.

## What This Changes

The issue requests stopping stale Control UI auth retry loops and making `chat.history` non-starving so a UI auth problem cannot degrade Slack Socket Mode connectivity.

## Best Possible Solution

Fix both root paths in core: make browser `AUTH_TOKEN_MISMATCH` terminal when no trusted cached-device-token retry can be attempted or after the single retry is exhausted, and refactor `chat.history` to avoid synchronous full-transcript reads/parses before recent-message and byte limits while preserving transcript semantics.

## Reproduction Assessment

Yes for the core code paths: current tests and source reproduce the Control UI no-retry `AUTH_TOKEN_MISMATCH` reconnect, and static inspection shows `chat.history` still performs synchronous full-file transcript work before limits. The live Slack disconnect timing remains to be validated with Socket Mode after a patch.

## Solution Assessment

Yes, the issue’s proposed direction is the right fix boundary. Slack Socket Mode ping/pong timeout tuning is a useful mitigation, but the maintainable solution is to stop the UI auth loop and make Gateway history reads non-starving.
