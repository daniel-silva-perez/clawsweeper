## Summary

Keep this issue open. Current main still has the two root paths described in the report: the browser Control UI can continue reconnecting after `AUTH_TOKEN_MISMATCH` when no cached device-token retry was possible, and `chat.history` still synchronously reads/parses the transcript before applying message and byte limits. The Slack Socket Mode timeout tuning and open PR #44897 are mitigations for symptoms, not a fix for the auth retry loop or event-loop starvation risk.

## What This Changes

The issue requests stopping stale Control UI auth retry loops and making `chat.history` bounded/non-blocking enough that UI auth failures cannot degrade Slack Socket Mode connectivity.

## Best Possible Solution

Fix both root paths in core: make browser and shared Gateway clients treat `AUTH_TOKEN_MISMATCH` as terminal when no trusted cached-device-token retry can be attempted or after that retry is exhausted, and refactor history loading so `chat.history` can return a recent bounded window without synchronous full-transcript reads/parses before limits while preserving transcript branch semantics.

## Reproduction Assessment

Yes for the core code paths: the current Control UI test reproduces reconnect after `AUTH_TOKEN_MISMATCH` with no cached retry, and source inspection shows `chat.history` still performs synchronous full-file transcript work before limits. The live Slack timing should be rechecked after a patch.

## Solution Assessment

Yes, the requested root-fix direction is the narrow maintainable path. PR #44897's frontend timeout and Slack pong-timeout tuning help symptoms, but they do not remove the auth retry loop or the event-loop-starving history path.
