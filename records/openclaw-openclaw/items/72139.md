## Summary

Keep this issue open. Current main still leaves both reported root paths in place: the Control UI can reconnect after `AUTH_TOKEN_MISMATCH` when no cached device-token retry was possible, and `chat.history` still synchronously reads/parses transcript data before message and byte limits are applied. The open WebUI timeout PR and Slack Socket Mode ping/pong tuning are useful mitigations, but they do not resolve the auth retry loop or the event-loop starvation risk described here.

## What This Changes

The issue requests stopping stale Control UI auth retry loops and making `chat.history` bounded/non-blocking enough that UI auth failures cannot degrade Slack Socket Mode connectivity.

## Best Possible Solution

Fix both root paths in core: treat `AUTH_TOKEN_MISMATCH` as terminal when no trusted cached-device-token retry can be attempted or after that retry is exhausted, and add a recent-window transcript loading path for `chat.history` that avoids synchronous full-transcript reads before limits while preserving existing display semantics.

## Reproduction Assessment

Yes. Current tests/source reproduce the Control UI reconnect-after-mismatch path with no cached token, and source inspection shows `chat.history` still performs synchronous full-file transcript work before applying limits; the exact Slack timing should be rechecked after a patch.

## Solution Assessment

Yes, the issue's requested direction is the narrow maintainable fix. PR #44897's frontend timeout and Slack pong-timeout tuning are symptom mitigations and should not replace fixing the auth reconnect gate and history loading path.
