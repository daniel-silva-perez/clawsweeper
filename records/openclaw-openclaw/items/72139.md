## Summary

Keep this issue open. Current main still leaves the Control UI able to reconnect after an `AUTH_TOKEN_MISMATCH` when no cached device-token retry was possible, and `chat.history` still synchronously reads/parses the full transcript before applying request, message, or byte limits; the Slack Socket Mode timeout tuning is only a mitigation, not the root fix.

## What This Changes

The issue asks OpenClaw to stop stale Control UI auth retry loops and make `chat.history` non-starving so browser auth failures cannot degrade Slack Socket Mode connectivity.

## Best Possible Solution

Fix both root paths in core: make browser and shared Gateway clients treat `AUTH_TOKEN_MISMATCH` as terminal when no trusted cached-device-token retry can be attempted or after that retry is exhausted, and refactor history loading so `chat.history` can return a recent bounded window without synchronous full-transcript reads/parses before limits while preserving transcript branch semantics.

## Reproduction Assessment

Yes for the core code paths: the current Control UI test reproduces reconnect after `AUTH_TOKEN_MISMATCH` with no cached retry, and source inspection shows `chat.history` still performs synchronous full-file transcript work before limits. The exact live Slack disconnect timing remains to be rechecked with Socket Mode after a patch.

## Solution Assessment

Yes, the issue's requested direction is the right fix boundary. PR #44897's frontend timeout and the Slack pong-timeout tuning are partial mitigations, but they do not remove the auth retry loop or the event-loop-starving history path.
