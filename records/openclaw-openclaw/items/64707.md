## Summary

Keep this issue open. Current main still has the shared fallback/default-account status path on the Feishu surface, latest released code does not differ in the implicated files, and same-author PR #64712 remains the active implementation candidate rather than a merged fix.

## What This Changes

The issue asks OpenClaw to stop showing or creating an unconfigured Feishu `default` account after gateway restart when the user only intended explicitly configured Feishu accounts.

## Best Possible Solution

Keep the issue open while #64712, or a replacement, is reviewed. The end state should distinguish explicitly configured Feishu accounts from synthetic fallback ids across channel summary and gateway status, avoid surfacing an unauthored `default` account, and preserve useful diagnostics for real but misconfigured default accounts.

## Reproduction Assessment

Unclear for the exact multi-account gateway-restart persistence claim; this read-only review did not run a live Feishu gateway. The implicated synthetic status path is high-confidence from current source and tests: empty or fallback account ids can still produce a default-account status line.

## Solution Assessment

Unclear whether the exact PR implementation is the final best fix until review, but the narrow maintainable direction is to fix shared status/account enumeration rather than add Feishu-only credential special cases.
