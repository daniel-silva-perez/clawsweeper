## Summary

Keep open. Current main still has the reported Gateway behavior: `handleAgent` rejects on `sendPolicy=deny` before computing whether the request actually asks for delivery, while the embedded agent path only checks send policy when `deliver` is true. The paired closing PR #73500 is open and unmerged, so this issue should wait for that PR or an equivalent focused fix to land.

## What This Changes

The issue reports that Gateway `agent` requests without external delivery are rejected when the session send policy is `deny`.

## Best Possible Solution

Land #73500 or an equivalent focused change in `src/gateway/server-methods/agent.ts` that enforces `sendPolicy=deny` only for `request.deliver === true`, plus regression tests proving non-delivery deny-policy Gateway calls run and explicit delivery calls remain blocked.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: a Gateway `agent` call with a session resolving to `sendPolicy=deny` and `deliver` omitted or false reaches the rejection before `wantsDelivery` is computed.

## Solution Assessment

Yes. Gating the Gateway policy check on `request.deliver === true` mirrors the embedded `agent-command.ts` behavior and preserves the outbound-delivery safety check.
