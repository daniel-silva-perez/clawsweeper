## Summary

Keep open. Current main still matches the reported bug: the Gateway `agent` RPC resolves and enforces `sendPolicy=deny` before it computes whether the request wants external delivery, while the embedded agent path only applies that policy when `deliver` is true. The linked #73500 PR remains open and directly targets this issue, so this should wait for that PR or an equivalent focused fix to merge.

## What This Changes

Allow Gateway `agent` invocations that do not request external delivery to run when the session resolves `sendPolicy=deny`, while still blocking explicit delivery sends.

## Best Possible Solution

Land #73500 or an equivalent focused change in `src/gateway/server-methods/agent.ts` that enforces `sendPolicy=deny` only when `request.deliver === true`, with regression coverage for non-delivery deny-policy calls being accepted and explicit delivery deny-policy calls being rejected.

## Reproduction Assessment

Yes. A source-level reproduction is clear: with a session entry or config resolving to `sendPolicy=deny`, a Gateway `agent` request with `deliver` omitted or false reaches the rejection at `src/gateway/server-methods/agent.ts:1049` before the handler derives delivery intent.

## Solution Assessment

Yes. The paired PR direction is the narrowest maintainable fix because it mirrors the embedded `agent-command.ts` policy gate and preserves the denial path for actual external delivery.
