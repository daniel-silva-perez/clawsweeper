## Summary

Keep this PR open. Current main still only partially overlaps the branch: `sessions_send` announce routing now has plugin-owned session-key parsing and thread propagation, but native ACP `sessions_spawn(runtime:"acp", thread:true, mode:"session")` still does not share the `/acp spawn` current-context placement and ownership path, and the proposed branch still has blocking Feishu and current-binding routing/security regressions.

## What This Changes

The PR adds shared ACP origin/placement/canonicalization and bound-delivery routing across ACP spawn, session send, subagent announce, and Discord/Telegram/Feishu channel adapters, with focused tests and generated API/config baselines.

## Best Possible Solution

Keep the PR open for maintainer or contributor follow-up. The best end state is a fresh rebase or narrower replacement that shares one origin-aware placement resolver between `/acp spawn` and native `sessions_spawn`, preserves Feishu action handlers, enforces the `boundBy` ownership gate for current bindings, routes bootstrap/completion through a binding-aware channel seam, and keeps focused regression coverage for Discord DM/thread/parent-channel, Telegram DM/topic, Feishu topic routing, bound completion delivery, and threaded `sessions_send` announce delivery.

## Reproduction Assessment

Yes. The PR body provides concrete live scenarios, and current main has static/test evidence: `src/agents/acp-spawn.test.ts` still expects a child binding even when `agentThreadId` is present, while the slash ACP path has the current-context resolver and ownership gate missing from native ACP binding.

## Solution Assessment

No, this PR is not currently the best mergeable solution as submitted. The direction is useful, but the patch needs to preserve existing Feishu handlers, add the current-binding ownership check, and make bootstrap/completion delivery use the same binding-aware reverse-target seam before it is safe to land.

  - body: The diff removes the Feishu handlers for pins, channel/member lookup, channel-list, and reactions after the `edit` branch, but the plugin still advertises those actions. Users can discover and invoke them, then every call falls through to unsupported-action behavior.

For current Discord/Telegram surfaces that can prefer malformed targets like `channel:user:<id>` or `channel:<chatId>` over the requester origin, so the initial ACP task can be misrouted even after binding succeeds.
  - confidence: 0.86
- **[P1] Enforce ownership before current ACP rebinds:** `src/agents/acp-spawn.ts:589`
  - body: When placement resolves to `current`, this path binds immediately and records `boundBy: "system"`; unlike `/acp spawn`, it does not check an existing conversation binding's `boundBy` sender first. A different participant can therefore rebind an already-bound ACP DM/thread and clear the previous ownership metadata.
  - confidence: 0.88
