## Summary

Keep open. Current main still only partially overlaps this PR: `sessions_send` announce routing has improved, but native ACP `sessions_spawn(runtime:"acp", thread:true, mode:"session")` still does not share the `/acp spawn` origin-aware placement and ownership path, and this PR still has unresolved functional/security-sensitive review concerns.

## Best Possible Solution

Keep this PR open for maintainer or contributor follow-up. Best path is a fresh rebase or narrower replacement that shares one origin-aware placement resolver between `/acp spawn` and native `sessions_spawn`, preserves Feishu action handlers, enforces the current-binding `boundBy` ownership gate, adds binding-aware reverse delivery for bootstrap/completion routing, fixes Telegram direct-vs-topic parsing at the parser or caller boundary, and keeps focused regression tests for Discord DM/thread/parent-channel, Telegram DM/topic, Feishu topic routing, bound completion delivery, and threaded `sessions_send` announce routing. Open PR #64322 is complementary for Discord per-thread session keys, not a superseding fix for this broader placement and delivery-routing scope.

## Risks / Open Questions

- Landing this PR as-is can regress Feishu message actions because the provided diff removes handlers that current main still advertises and tests.
- The native current-binding direction needs the same `boundBy` ownership gate as `/acp spawn`; otherwise a second participant could rebind an existing conversation surface.
- The PR's change from depth-based internal announce detection to key-shape-only detection needs maintainer review because legacy flat sessions with stored subagent ancestry could start delivering internal completion messages visibly.
- Security/supply-chain pass found no new dependency source, workflow/action ref, lockfile, release script, or secret-scope change in the provided file list; the main review blockers are routing correctness and the Feishu/action-handler regression.
