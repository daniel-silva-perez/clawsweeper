## Summary

Keep open. Current main still leaves the reported recovery deadlock in place: the gateway rejects `device.pair.reject` before dispatch for callers without `operator.pairing`, while the CLI only has local fallback paths for list and approve. The reporter's follow-up confirms this is blocking real gateway control-plane use, and related older fixes cover approve/list recovery rather than this reject or downgrade path.

## What This Changes

The issue requests a supported recovery path when a CLI paired with only `operator.read` receives repeated broader repair pairing requests that it cannot approve, reject, or narrow.

## Best Possible Solution

Add or document a bounded same-device recovery path for local repair deadlocks without silently granting broader scopes. The end state should let a limited CLI clear or replace only its own pending repair request, preserve explicit approval for scope upgrades, and include CLI, gateway pairing, docs, and regression coverage for the chosen semantics.

## Reproduction Assessment

Yes. A high-confidence source-backed reproduction is a paired device with only `operator.read`, then a CLI reconnect or reject attempt that requests `operator.pairing` or broader scopes; current main gates reject before dispatch and has no reject fallback. No live state mutation or test run was performed because this review was read-only.

## Solution Assessment

Unclear. The issue identifies a real bug, but the best fix depends on maintainer policy: self-reject fallback, narrower repair-scope requests, or a different bootstrap path each changes pairing recovery authority differently.
