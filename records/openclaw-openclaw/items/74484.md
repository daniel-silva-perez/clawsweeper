## Summary

Keep open. Current main still has only list/approve local fallback for scope-limited local pairing recovery, while `devices reject` and the relevant control-plane methods still require higher operator scope. The reporter's follow-up confirms this is an active recovery deadlock, and the remediation changes pairing authorization semantics rather than a safe cleanup close.

## What This Changes

The issue requests a supported recovery path for a CLI paired with only `operator.read` when broader repair or scope-upgrade requests keep being reissued and cannot be approved, rejected, downgraded, or narrowed from that CLI.

## Best Possible Solution

Define a reviewed recovery contract for scope-limited paired CLIs, likely combining a same-device local-loopback rejection path for pending repair requests with an explicit decision on narrow repair-scope requests or a documented bootstrap flow.
