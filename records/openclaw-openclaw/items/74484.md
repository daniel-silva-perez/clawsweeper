## Summary

Keep open. Current main has partially mitigated the original broad-bundle symptom by using least-privilege scopes for classified CLI gateway calls, but the recovery deadlock is still not fully resolved: `device.pair.reject` remains pairing-scope gated before the same-device handler can run, the CLI reject command has no local fallback, and same-device pending request replacement can still complicate approving or clearing a scope-upgrade repair request from a read-only CLI.

## What This Changes

The issue requests a supported recovery path for a CLI paired with only `operator.read` that cannot approve, reject, or downgrade recurring over-scoped gateway repair requests.

## Best Possible Solution

Define a bounded same-device recovery model for local loopback read-only CLI repair deadlocks. The end state should let the operator clear or replace only their own pending repair request, avoid silent scope widening, preserve explicit approval for scope upgrades, and update CLI behavior, gateway pairing tests, and the public devices docs together.

## Reproduction Assessment

Yes. A high-confidence source-backed reproduction is a paired operator device with only `operator.read`, then a pending scope-upgrade repair followed by `openclaw devices reject <requestId>` or an approve/list recovery path; current main still gates reject before dispatch and has no reject fallback. The old full default-bundle prompt is partly mitigated for classified methods by the shipped least-privilege CLI change.

## Solution Assessment

Unclear. The issue identifies a real remaining recovery gap, but the safest fix needs maintainer/security policy: a local same-device reject fallback, a no-reconnect local approval path, narrower repair request pinning, or another bootstrap flow all change pairing recovery authority differently.
