## Summary

Keep open. Current main still leaves a read-only paired CLI unable to reject its own pending scope-upgrade repair request: `device.pair.reject` is pairing-scope gated and the CLI reject command only calls the gateway RPC, while local fallback exists for list/approve only.

## What This Changes

The issue requests a supported recovery path for a CLI paired with only `operator.read` when an auto-reissued repair request asks for broader operator scopes that the CLI cannot approve or reject.

## Best Possible Solution

Add or document a bounded recovery path for same-device local-loopback repair deadlocks without silently granting broader operator scopes. The end state should let a limited CLI clear or replace only its own pending repair request, preserve explicit approval for scope upgrades, and cover the chosen semantics in CLI, gateway pairing, docs, and regression tests.

## Reproduction Assessment

Yes. A high-confidence source-backed reproduction is a paired CLI/device record with only `operator.read`, followed by a CLI reconnect or reject attempt that requests `operator.pairing` or broader scopes; current main keeps reject behind `operator.pairing` and provides no reject fallback.

## Solution Assessment

Unclear. The issue correctly identifies a real deadlock, but the best fix depends on maintainer policy around pairing recovery authority rather than a purely mechanical code change.
