## Summary

Keep this PR open. Current main still does not implement the PR's central health-snapshot behavior: channel account probes are still awaited sequentially with only a per-probe timeout, and the gateway refresh path does not pass any snapshot-wide budget. The legacy cron-ID part is partially covered by current doctor repair, but runtime cron load remains intentionally narrower and docs point users to `openclaw doctor --fix` for cron store migrations.

## Best Possible Solution

Keep this PR open or replace it with a rebased focused patch. The best path is to implement bounded, parallel health snapshots in current `src/commands/health.ts` and the gateway refresh path, add regression tests for deadline exhaustion and concurrent account probes, and make an explicit maintainer decision before changing cron ID repair from doctor-only persistence to runtime load-time persistence.

- The PR's runtime cron ID rewrite conflicts with the current doctor-first legacy cron store migration policy unless maintainers explicitly want load-time persistence.
