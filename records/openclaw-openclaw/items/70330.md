## Summary

Keep open. Current main still leaves the reported WebChat/session-continuity gap: admin-scoped WebChat can still reach session reset behavior through the generic `chat.send` slash-command path, the Control UI New Session flow still sends `/new`, and the gateway has no first-class reset provenance/audit payload that would explain an old/new session-id rotation after reconnect. The reporter’s follow-up narrows the timing to a post-reconnect `chat.send`, and related open reports #59839 and #70472 show this remains part of a broader session-history visibility/rotation problem rather than a closed duplicate.

## What This Changes

The issue requests that WebChat preserve or explicitly audit `agent:main:main` session continuity after gateway restart/reconnect so a normal visible send cannot silently rotate the active session and hide prior transcripts/checkpoints.

## Best Possible Solution

Keep the issue open and land a core fix that separates intentional WebChat session control from ordinary `chat.send`: use an explicit reset/create RPC or explicit session-control flag with trigger metadata, persist old/new session ids and archive paths as an auditable reset event, prevent reconnect/plain-send paths from rotating the active `agent:main:main` mapping, and expose reset archives/checkpoints through the session UI or a typed RPC.

## Reproduction Assessment

Unclear. The report has detailed incident evidence and current main still contains the reset path that can rotate an admin-scoped WebChat session, but I did not find a high-confidence automated restart/reconnect reproduction that proves a plain `chat.send("continue")` currently creates `*.jsonl.reset.*` without an explicit reset trigger.

## Solution Assessment

Yes for keeping this open; closing would hide a real remaining observability and continuity gap. For implementation, the narrowest maintainable path is an additive reset provenance/session-control change plus regression coverage before attempting broader reset-archive browsing UX.
