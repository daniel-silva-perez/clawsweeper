## Summary

Keep this issue open. Current main has partial safety work around reset confirmation and lifecycle hooks, but the reported WebChat/session-continuity gap is still present: Control UI/WebChat is admin-scoped, UI and local `/new`/`/reset` still route through generic `chat.send`, the gateway still parses and strips reset slash text on that path, and there is no persisted or protocol-visible reset audit marker tying old/new session ids and archive paths to a trigger.

## Best Possible Solution

Keep this open in OpenClaw core. Implement an explicit WebChat session-control path for New Session and typed `/new`/`/reset`, persist or expose reset provenance with trigger, session key, old/new session ids, post-reset-message presence, and archive path, and add a gateway restart/reconnect regression proving that a plain post-reconnect send keeps the existing `agent:main:main` session id and does not create a reset archive.

## Clownfish Work Prompt

Fix the WebChat/session reset ambiguity described by this issue. Scope the PR to OpenClaw core/UI only: add an explicit WebChat reset/new control path and durable reset provenance so a WebChat reset cannot look like a normal first message after reconnect.

Expected implementation boundary:
- Do not remove documented typed `/new` and `/reset` support. For WebChat, intercept confirmed typed reset commands and the New Session button in the client and call an explicit reset path, or add an additive `chat.send` session-control flag with trigger metadata. Avoid silently relying on bare control text from the UI.
- Extend the reset path around `performGatewaySessionReset` so reset provenance is returned and/or persisted in a user-visible durable place: `sessionKey`, `oldSessionId`, `newSessionId`, trigger/source (`ui-button`, `typed-slash-command`, `gateway:agent`, `sessions.reset`, etc.), archive path(s), timestamp, and whether a post-reset message was present.
- Preserve compatibility for non-WebChat/admin callers unless the protocol change is explicitly additive and tests cover the old behavior.
- Add regression coverage for an existing `agent:main:main` session id A, WebChat reconnect/history/list, then plain `chat.send("continue")`: session id A must remain active and no `A.jsonl.reset.*` archive should be created.
- Add/update Control UI tests proving New Session/typed reset confirmation no longer sends unaudited bare `/new` through generic chat send, or sends it only with explicit session-control metadata.
- Add docs/changelog if user-facing behavior or protocol output changes. Changelog credit should include `Thanks @Squirbie` and any related reporter whose scenario is directly fixed; do not add internal-only entries.
