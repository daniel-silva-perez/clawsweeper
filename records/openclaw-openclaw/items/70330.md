## Summary

Keep open. Current main still allows WebChat session rotation through generic chat-send and implicit rollover paths without a durable reset/rotation audit record, and the issue’s post-reconnect timeline remains actionable rather than obsolete.

## What This Changes

The issue asks for WebChat session continuity and reset provenance so a normal post-reconnect message cannot silently rotate the active `agent:main:main` session or hide prior transcripts/checkpoints without an explicit, auditable reset path.

## Best Possible Solution

Land an additive core/UI fix that separates intentional WebChat session control from ordinary `chat.send`, records reset/rollover provenance with old/new session ids and archive paths, returns enough typed metadata for WebChat to show what happened, and makes reset archives/checkpoints discoverable without requiring manual `sessions.json` repair.

## Reproduction Assessment

Unclear. The report has detailed disk/log evidence and current main contains the rotation and archive paths, but I did not find a high-confidence automated restart/reconnect test proving that a plain post-reconnect `chat.send("continue")` rotates the session without an explicit or scheduled reset trigger.

## Solution Assessment

Yes for keeping this open and queueing a narrow first PR. The safest implementation path is additive provenance plus a typed WebChat reset flow and targeted regression tests before broader history-browser UX changes.

Implement an additive first PR that: (1) stops the Control UI New Session button from using generic `chat.send("/new")` and routes it through a typed session reset/create RPC or an explicit session-control flag; (2) adds reset/rollover provenance for every `performGatewaySessionReset`/implicit session rollover path, including sessionKey, oldSessionId, newSessionId, trigger/source, timestamp, archive transcript path(s), and whether a post-reset message was present; (3) returns or broadcasts enough typed metadata for WebChat/Control UI to show that a reset occurred instead of making the first new message look like an ordinary continuation; (4) prevents a plain post-reconnect `chat.send("continue")` from consuming stale reset UI/draft/idempotency state; and (5) adds targeted regression coverage.
