## Summary

Keep open. Current main still has the reported PI auth bridge behavior: keyRef/tokenRef-backed auth profiles are valid in the canonical auth resolver, but the PI discovery credential map only converts inline key/token material, so the OpenRouter symptom remains a real narrow bug rather than stale cleanup.

## What This Changes

Make PI model discovery resolve keyRef/tokenRef-backed auth profiles so OpenRouter and other provider models are treated as authenticated in model discovery and pickers.

## Best Possible Solution

Update the PI auth discovery bridge to use the canonical auth-profile credential resolver for api_key and token profiles under no-prompt/read-only semantics, keep legacy auth.json scrubbing separate, and add regression coverage for keyRef-only and tokenRef-only PI discovery plus the model-list/picker surface if needed.

## Reproduction Assessment

Yes. A store containing `openrouter:default` with `type: "api_key"` and only `keyRef` follows the current `resolvePiCredentialsForDiscovery` path into `resolvePiCredentialMapFromStore`, where the converter checks only `cred.key` and drops the provider.

## Solution Assessment

Yes, the issue’s proposed direction is the best maintainable fix: reuse `resolveApiKeyForProfile` or a shared no-prompt extraction of that resolver instead of adding a second PI-specific SecretRef resolver. The main design choice is how to handle the async boundary safely.
