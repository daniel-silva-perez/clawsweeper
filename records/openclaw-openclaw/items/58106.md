## Summary

Keep open. Current main still has the reported generic PI auth-bridge gap: keyRef/tokenRef-backed auth profiles are supported by the canonical auth resolver and persisted without inline plaintext, but PI credential discovery still converts only inline fields before constructing PI auth storage.

## What This Changes

The issue requests that PI model/auth discovery resolve keyRef/tokenRef-backed auth profiles so OpenRouter and other provider models are treated as authenticated in model picker and model-listing flows.

## Best Possible Solution

Fix the generic auth-profile-to-PI bridge so PI credentials are built from the same credential-resolution semantics used at runtime, then add a keyRef-only regression test that proves OpenRouter/provider auth is visible to PI availability without any OpenRouter-specific branch.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an auth store with an api_key OpenRouter profile containing keyRef and no inline key: current PI conversion returns no credential, then the pinned PI registry treats the provider as unauthenticated for getAvailable().

## Solution Assessment

Yes. Reusing resolveApiKeyForProfile() or a shared resolver in the PI credential builder is the narrow maintainable fix because it keeps secret-ref semantics centralized and avoids duplicating provider-specific or OpenRouter-specific auth logic.
