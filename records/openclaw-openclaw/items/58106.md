## Summary

Keep open. Current main still has the reported PI auth discovery gap: model discovery builds PI credentials from raw auth-profile objects through a synchronous converter that only accepts inline plaintext `key`/`token` fields, while the canonical auth-profile resolver already handles `keyRef`/`tokenRef`. That means a properly stored OpenRouter profile with only `keyRef` can still be absent from PI discovery credentials and therefore from picker/listing surfaces backed by that registry.

## Best Possible Solution

Keep this issue open. Implement a generic PI discovery credential bridge that resolves `api_key` and `token` profiles through `resolveApiKeyForProfile()` or an equivalent shared auth-profile resolver, without persisting resolved plaintext secrets, then add regression coverage for keyRef-only OpenRouter/provider profiles in `src/agents/pi-model-discovery.auth.test.ts` and verify the `/models`/Discord picker path uses the resolved credential state.
