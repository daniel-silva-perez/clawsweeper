## Summary

Keep open. Current main still converts auth profiles to PI credentials through a synchronous helper that only reads inline `key`/`token` fields, while the documented and persisted auth-profile path supports `keyRef`/`tokenRef`; pi-coding-agent then filters unauthenticated providers out of available model results, matching the reported OpenRouter picker failure.

## What This Changes

Resolve secret-reference-backed auth profiles during PI model discovery so OpenRouter and other providers with `keyRef` or `tokenRef` credentials are treated as authenticated in model picker/listing surfaces.

## Best Possible Solution

Keep the issue open and fix the generic auth-profile-to-PI bridge: reuse `resolveApiKeyForProfile()` or an equivalent shared resolver inside the PI credential builder for `api_key` and `token` profiles, keep OpenRouter out of the core logic, preserve secret-ref safety, and add regression coverage for keyRef-only discovery.

## Reproduction Assessment

Yes. A high-confidence reproduction is an auth store containing an `api_key` OpenRouter profile with `keyRef` and no inline `key`: current source returns no OpenRouter PI credential, and the pinned registry filters providers without `hasAuth()` from available models.

## Solution Assessment

Yes, the issue’s proposed direction is the narrowest maintainable fix: reuse the existing auth-profile secret resolver instead of adding a bespoke PI or OpenRouter-specific secret-ref path.

## Repair Work Prompt

Fix OpenRouter/PI model discovery auth for auth profiles whose credential material is stored only as `keyRef` or `tokenRef`. Current `src/agents/pi-auth-credentials.ts` only converts inline `key`/`token`, so `src/agents/pi-auth-discovery.ts` can omit providers from PI auth storage and pi-coding-agent hides their models from available picker/list results. Implement a generic fix in the auth-profile-to-PI bridge, preferably by reusing `resolveApiKeyForProfile()` or a shared resolver for `api_key` and `token` profiles. Do not special-case OpenRouter, do not add a second bespoke secret-ref resolver, do not persist resolved plaintext secrets into `auth-profiles.json` or `auth.json`, and preserve discovery/status no-keychain-prompt behavior. If async resolution is required, update the smallest safe set of PI discovery call sites or add a clearly named async path while preserving sync callers where necessary. Add regression coverage for an OpenRouter `api_key` profile with `keyRef` and no inline `key`, and preferably a tokenRef case if the same path changes. Changelog expected: add a single-line user-facing fix under the active version with `Thanks @ThalynLabs and @macclaw2006-eng`.
