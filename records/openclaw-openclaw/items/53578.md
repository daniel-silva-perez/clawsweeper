## Summary

Keep open. Current main still routes Gemini CLI OAuth turns through the external `gemini` CLI backend, and I found no current-main or shipped-release evidence that the repeated per-turn startup/auth latency reported here has been fixed or moved out of scope.

## Best Possible Solution

Keep this issue open for maintainer triage. The best path is to measure standalone `gemini` startup/auth latency versus OpenClaw wrapper overhead on an affected macOS OAuth setup, then either document/doctor a Gemini CLI Keychain remediation, add diagnostics that clearly flag slow external CLI startup/auth, or implement a supported mitigation that avoids repeated per-turn Gemini CLI credential initialization where the CLI/API contract allows it.
