## Summary

Keep open. Current main still lacks a validated, granular outbound-media root allowlist for Telegram or shared filesystem policy; the transport can consume forwarded roots, but config validation only exposes broad `tools.fs.workspaceOnly` behavior and Telegram's separate `trustedLocalFileRoots` for self-hosted Bot API file paths. Open PR #52951 may become the right shared solution, but it is still open and security-sensitive.

## What This Changes

The issue asks for a validated configuration surface that lets Telegram outbound local media sends use operator-approved host directories outside OpenClaw-managed roots.

## Best Possible Solution

The end state should be a shared, documented, schema-backed operator allowlist, likely `tools.fs.roots` or an equivalent filesystem policy surface, that feeds outbound media access across channels with config types, generated schema/help, docs, and regression tests aligned. A Telegram-only hidden knob is not the best long-term shape unless maintainers explicitly choose channel-scoped roots.

## Reproduction Assessment

Yes. The config-side failure is reproducible from current source because both relevant Zod objects are strict and omit the requested keys, while tests and runtime inspection show Telegram only consumes roots supplied by core.

## Solution Assessment

Yes, keeping this open for shared FS/media policy review is the best current action. The likely fix should land through #52951 or an equivalent shared allowlist rather than a separate Telegram-only validator exception.
