## Summary

Keep open. Current main still does not provide a validated granular outbound-media root allowlist for Telegram or shared filesystem policy: Telegram can consume forwarded `mediaLocalRoots`, but config validation exposes only broad `tools.fs.workspaceOnly` behavior plus Telegram's separate `trustedLocalFileRoots` for self-hosted Bot API file paths. Related PR #52951 appears to target the right shared direction, but it is still open and security-sensitive.

## What This Changes

The issue asks for a validated configuration surface that lets Telegram outbound local media sends use operator-approved host directories outside OpenClaw-managed roots.

## Best Possible Solution

Land a shared, documented, schema-backed operator allowlist, likely `tools.fs.roots` or an equivalent filesystem policy surface, that feeds local media access across channels with config types, generated schema/help, docs, and regression tests aligned.

## Reproduction Assessment

Yes. The config-side failure is reproducible from current source because the relevant strict Zod objects omit `mediaLocalRoots`, while Telegram outbound code only consumes roots supplied by core.

## Solution Assessment

No for a Telegram-only validator exception as the final design. The safer maintainable solution is a shared filesystem/media root policy, with #52951 or an equivalent PR reviewed for security and product fit.
