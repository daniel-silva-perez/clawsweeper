## Summary

Keep open. Current main still has the memory-core promotion writer appending new deep-dreaming sections to `MEMORY.md` without a durable disk-side budget, compaction, or archive handoff, and the paired fix PR is open and unmerged. The bootstrap layer truncates injected context, so the exact Gateway-freeze mechanism still needs logs, but the unbounded file-growth bug remains valid.

## What This Changes

The issue requests bounding `MEMORY.md` growth from memory-core dreaming promotions so oversized durable memory cannot overflow bootstrap/session startup behavior or wedge the Gateway.

## Best Possible Solution

Review and land PR #74088, or replace it with an equivalent memory-core fix that enforces a durable budget before promotion writes, preserves user-authored `MEMORY.md` content, and adds regression coverage for near-limit and over-limit promotion runs.

## Reproduction Assessment

Yes for the disk-growth path: the current `applyShortTermPromotions` implementation appends existing `MEMORY.md` plus each new section without a size guard. Unclear for the full Gateway freeze because the issue provides no logs and current bootstrap injection truncates oversized files rather than injecting them whole.

## Solution Assessment

Yes, the right boundary is the bundled memory-core promotion writer, not bootstrap alone. The exact PR implementation still needs review, but a narrow compaction/archive strategy for dreaming-owned sections is the maintainable direction.
