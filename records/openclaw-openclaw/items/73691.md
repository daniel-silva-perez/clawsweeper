## Summary

Keep open. Current main still has the memory-core deep promotion path appending to `MEMORY.md` without a durable disk-side budget, compaction, or archive layer, and the open paired PR #74088 has closing syntax but is not merged into `main`.

## What This Changes

The issue requests bounding or compacting memory-core dreaming promotions so `MEMORY.md` cannot grow past bootstrap limits and contribute to Gateway lockups.

## Best Possible Solution

Land PR #74088 after review, or replace it with an equivalent memory-core fix that applies a durable budget before promotion writes, preserves user-authored `MEMORY.md` content, and adds regression coverage for near-limit and over-limit promotion runs.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Partially: the unbounded disk growth is clear from the current append-only `applyShortTermPromotions` path and can be reproduced with repeated new promotion candidates, but the full Gateway freeze/session-write-lock timeout still lacks logs or a focused reproduction.

## Solution Assessment

Is this the best way to solve the issue? Yes on boundary: the fix belongs in bundled `memory-core` promotion writing. The exact compaction/archive policy should be settled in review of #74088 or an equivalent replacement before closing this issue.
