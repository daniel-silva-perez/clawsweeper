## Summary

Keep open. Current main has bootstrap truncation and visibility defenses, so the exact bootstrap-overflow/freeze mechanism is not proven from the issue alone, but the core reported product gap is still present: memory-core dreaming/deep promotion appends to root MEMORY.md without a durable file-size budget, compaction, or archive layer.

## Best Possible Solution

Keep this issue open and fix it in the bundled memory-core owner surface: add a durable MEMORY.md budget check before promotion writes, choose compaction or an archive layer for older promoted entries, expose status/doctor visibility for oversized root memory, and add regression tests for near-limit and over-limit promotion runs. Treat the session-write-lock/Gateway freeze as a related verification point that still needs logs or a focused reproduction.
