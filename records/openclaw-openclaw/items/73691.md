## Summary

Keep open. Current main has bootstrap truncation and warning/reporting defenses, and bootstrap context is resolved before the session write lock, so the exact overflow-to-lock-timeout mechanism is not proven. The central product gap remains: memory-core dreaming/deep promotion still appends to root MEMORY.md without a durable size budget, compaction, or archive layer.

## Best Possible Solution

Keep this issue open and fix it in the bundled memory-core owner surface: add a durable MEMORY.md budget check before promotion writes, choose compaction or an archive layer for older promoted entries, surface oversized root memory through status/doctor visibility, and add regression tests for near-limit and over-limit promotion runs. Treat the Gateway freeze/session-write-lock timeout as a related verification point that still needs logs or a focused reproduction.
