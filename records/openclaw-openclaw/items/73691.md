## Summary

Keep open: current main still has the memory-core promotion path appending to `MEMORY.md` without a durable file-size budget, compaction, or archive layer. The paired fix PR #74088 uses closing syntax for this issue, but it remains open and unmerged, so this issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue requests bounding memory-core dreaming promotions so `MEMORY.md` cannot grow past bootstrap/session limits and wedge the Gateway after extended use.

## Best Possible Solution

Review and land PR #74088, or replace it with an equivalent memory-core fix that applies a durable size budget before promotion writes, preserves user-authored `MEMORY.md` content, and adds regression coverage for repeated near-limit and over-limit promotion runs.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the unbounded `MEMORY.md` growth path: repeated `applyShortTermPromotions` calls with new candidates append existing content plus a new section. Unclear for the full Gateway freeze because the report has no logs or focused lock-timeout reproduction.

## Solution Assessment

Is this the best way to solve the issue? Yes on boundary: the fix belongs in the bundled `memory-core` promotion writer rather than bootstrap truncation alone. The exact compaction/archive policy should be settled in review of PR #74088 or an equivalent replacement.
