## Summary

Keep open. Current main still has the reported Bedrock credential-detection path: the memory adapter calls `hasAwsCredentials()`, and that helper still delegates to the AWS SDK `defaultProvider()` without an `AWS_EC2_METADATA_DISABLED` guard, an IMDS-safe passive-detection path, or a retry/backoff cache. The related fix PR #64944 was closed unmerged, and the merged #71245 only added the Bedrock auto-selection gate that now invokes this same probe.

## What This Changes

Stop OpenClaw startup and memory embedding auto-selection from repeatedly probing AWS IMDS at `169.254.169.254` in sandboxed non-AWS environments after the Bedrock credential-provider dependency was added.

## Best Possible Solution

Add a focused Amazon Bedrock plugin fix that separates passive memory-embedding credential eligibility from actual Bedrock runtime authentication, so startup auto-selection does not repeatedly contact IMDS in non-AWS sandboxes while real Bedrock calls still use the AWS SDK default credential chain. The fix should lock in behavior with unit tests for `AWS_EC2_METADATA_DISABLED`, IMDS suppression or bounded backoff/coalescing, env restoration, and normal static/profile/SSO/role-compatible paths.
