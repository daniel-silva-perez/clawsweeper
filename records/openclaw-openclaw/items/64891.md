## Summary

Keep open. Current main still has the Bedrock memory credential-detection path that can probe AWS IMDS during startup: the Bedrock memory adapter calls `hasAwsCredentials()`, and that helper still invokes the AWS SDK `defaultProvider()` without an IMDS-disabled passive-detection path, memoization, or backoff. The closed #64944 attempted to address this directly but was not merged; the merged #71245 fixed a related auto-selection failure while preserving the same credential-chain probe, so the reported IMDS request flood remains plausible on current main.

## What This Changes

The issue requests that OpenClaw stop making repeated AWS IMDS requests to `169.254.169.254:80` during startup in sandboxed/non-AWS environments when Bedrock credential auto-detection runs.

## Best Possible Solution

Add a focused Amazon Bedrock plugin fix that separates passive memory-embedding credential eligibility from actual Bedrock runtime authentication, so startup auto-selection does not repeatedly contact IMDS in non-AWS sandboxes while explicit Bedrock usage and real EC2/ECS/EKS role flows still use the AWS SDK default credential chain. Lock it with tests for `AWS_EC2_METADATA_DISABLED`, IMDS suppression or bounded coalescing/backoff, env restoration, and normal static/profile/SSO/role-compatible paths.

## Repair Work Prompt

Fix the Bedrock memory auto-selection IMDS request regression reported in openclaw/openclaw#64891. Current `extensions/amazon-bedrock/memory-embedding-adapter.ts` calls `hasAwsCredentials()` during provider auto-selection; `extensions/amazon-bedrock/embedding-provider.ts` then invokes `@aws-sdk/credential-provider-node` defaultProvider, which can probe AWS IMDS at `169.254.169.254` in non-AWS sandboxes. Implement a focused Amazon Bedrock plugin fix that prevents repeated IMDS probes during passive memory embedding eligibility checks without changing actual Bedrock runtime authentication for explicit Bedrock usage or legitimate AWS SDK default-chain users. Use #64944 as prior context but do not blindly reapply the global IMDS-disable approach because review feedback noted EC2 instance-role risk. Do not move Bedrock-specific policy into core unless a tiny generic seam is truly needed. Add a user-facing changelog fix entry with `Thanks @vnicolici` and credit prior diagnostic context if used.
