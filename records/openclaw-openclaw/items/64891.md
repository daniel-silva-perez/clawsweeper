## Summary

Keep open. Current main and the latest release still route Bedrock memory auto-selection through `hasAwsCredentials()`, and that helper still calls the AWS SDK `defaultProvider()` without an IMDS-safe passive detection path, memoization, or backoff. The prior direct fix PR #64944 is closed unmerged, while merged #71245 added the adapter credential gate that now invokes the same probe rather than removing the reported IMDS traffic source.

## What This Changes

The issue requests that OpenClaw stop making repeated AWS IMDS requests to `169.254.169.254:80` during startup in sandboxed or non-AWS environments when Bedrock credential auto-detection runs.

## Best Possible Solution

Add a focused Amazon Bedrock plugin fix that separates passive memory-embedding credential eligibility from actual Bedrock runtime authentication, so startup auto-selection does not repeatedly contact IMDS in non-AWS sandboxes while explicit Bedrock usage and real AWS SDK default-chain flows still work. Lock the behavior with unit tests for `AWS_EC2_METADATA_DISABLED`, no repeated IMDS/defaultProvider probes during auto-selection, env restoration if env is temporarily changed, and static/profile/SSO/role-compatible paths.

## Reproduction Assessment

Yes. The issue’s live reproduction is detailed, and current source gives a high-confidence static reproduction path: memory auto-selection calls the Bedrock adapter, the adapter calls `hasAwsCredentials()`, and that helper invokes the AWS SDK default credential chain that can fall through to IMDS.

## Solution Assessment

No current solution is in place. The prior #64944 direction addressed the unwanted network calls but had EC2 instance-role regression risk; the safer fix should be scoped to passive eligibility detection or bounded/coalesced probing while preserving runtime Bedrock auth semantics.
