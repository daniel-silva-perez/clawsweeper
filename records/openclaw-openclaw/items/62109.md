## Summary

Keep open. The issue is recent, concrete, and reports a core custom OpenAI-compatible provider path rather than optional plugin work. Current main has generic embedded transport and proxy-style custom-provider handling, but I did not find an IDN/punycode-specific fix, regression test, changelog entry, or canonical related issue that supersedes this report.

## Best Possible Solution

Keep the issue open and add a focused reproduction around a custom `api: "openai-completions"` provider using Unicode and punycode `baseUrl` values in the embedded/Web UI path. Compare the outgoing URL, Host/SNI-sensitive behavior, headers, and request body with an ASCII control; then either normalize or preserve host handling consistently in `src/agents/openai-transport-stream.ts` and the guarded fetch path, or improve diagnostics if the 403 is truly returned by the upstream edge rather than OpenClaw.
