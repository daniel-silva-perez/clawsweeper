## Summary

Keep open. Current main still routes custom OpenAI-compatible interactive traffic through the OpenAI SDK plus OpenClaw's guarded fetch/SSRF dispatcher, but I found no IDN/punycode-specific normalization, regression coverage, duplicate tracker, or release-proven fix for this reported failure mode.

## What This Changes

The issue requests that custom OpenAI-compatible provider `baseUrl` values work consistently in interactive Web UI/local TUI runs when the hostname is Unicode/IDN, punycode, plain ASCII, or an IP address.

## Best Possible Solution

Add a focused transport-level reproduction using a local OpenAI-compatible test server with deterministic IDN/punycode hostname handling, then fix the narrow layer that diverges between configured `baseUrl`, SDK request URL, guarded-fetch DNS pinning, Host/SNI, and headers while preserving existing SSRF protections.
