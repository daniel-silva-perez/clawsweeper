## Summary

Keep open. Current main has several partial hardening changes for this failure chain, but it still does not implement the requested end-to-end behavior: gateway drain still rejects new work and surfaces restart text, subagent completion delivery remains best-effort across restarts, compaction timeout recovery is bounded rather than a configurable circuit breaker/reset path, and fallback/session override behavior during live config changes remains a product/code decision rather than a completed fix.

## What This Changes

Investigate and harden a 2026.4.8 large-session failure chain involving context overflow, compaction timeout, gateway drain rejection, subagent announce loss, and stale fallback-chain behavior after config changes.

## Best Possible Solution

Keep this issue open as the umbrella until maintainers split or land the remaining work. The likely path is to keep #55412 or a successor as the GatewayDrainingError retry canonical item, decide whether compaction needs a timeoutAction/circuit-breaker/reset policy, make subagent completion announces durable or explicitly bounded across restart windows, and clarify how hot-applied fallback config interacts with active auto-fallback session overrides during drain.
