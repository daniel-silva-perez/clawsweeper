## Summary

Keep open. Current main has meaningful hardening around config hot reload, bounded timeout compaction, gateway drain visibility, and subagent announce fallback, but it still preserves several observable parts of the reported failure chain: drain windows reject new work, GatewayDrainingError is surfaced as a restart response instead of retrying the original turn, announce cleanup can still give up, and timeout compaction remains bounded rather than a full recovery policy.

## What This Changes

The issue asks OpenClaw to investigate and fix a large Telegram-session failure chain where overflow/compaction timeouts cascade into gateway drain rejections, subagent completion announce loss, and stale runtime fallback choices until restart.

## Best Possible Solution

Keep this as the umbrella until maintainers split it into narrower canonical follow-ups or land one coordinated reliability fix covering restart/drain retry semantics, durable or explicitly bounded subagent completion delivery, compaction timeout circuit-breaker/reset behavior, and documented live fallback/session override semantics after config changes.
