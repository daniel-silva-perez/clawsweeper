## Summary

Keep open. Current main only partially addresses the reported v2026.4.29 Discord regression: the post-release change quiets repeated native slash-command deploy 429 logs, but source and tests still preserve three deploy attempts on a 429, and `users/@me` bot identity fetch remains a hard startup dependency. The report has concrete runtime evidence and is distinct from the earlier SecretRef mismatch in #74765.

## What This Changes

The issue asks for Discord startup to avoid repeated OAuth identity fetch timeouts and native slash-command deploy 429 bursts after upgrading to v2026.4.29.

## Best Possible Solution

Discord startup should avoid amplifying REST trouble: use the existing application ID bypass where possible, avoid native command deployment while identity preflight is failing or rate-limited, and treat deploy 429s as a single non-fatal skip/cooldown that leaves existing commands active. Add targeted Discord monitor/deploy tests for the 429 and `fetchUser("@me")` timeout paths.

## Reproduction Assessment

Partially yes. The native slash-command deploy burst is reproducible from current tests and source because a 429 leads to three deploy attempts; the OAuth/user `@me` timeout needs the reporter's network/Discord conditions, but the hard startup fetch path is visible in current code.

## Solution Assessment

No. Current main's post-release change is useful but only quiets repeated deploy 429 logs; it does not fully prevent the retry burst or make bot identity timeout behavior non-amplifying.
