## Summary

Keep open. Current main still only partially addresses the reported Discord v2026.4.29 startup regression: repeated native deploy 429 logs were collapsed, but source and tests still preserve three deploy attempts on a route 429, and bot identity lookup through `fetchUser("@me")` remains a hard startup failure path. The report is concrete, current-release-specific, and distinct from the SecretRef mismatch tracked in #74765.

## What This Changes

The issue reports a v2026.4.29 Discord startup regression where OAuth identity preflight requests time out and native slash-command deploy PATCH requests hit Discord 429 rate limits during startup.

## Best Possible Solution

Discord startup should avoid amplifying REST trouble: use the existing application ID/token-derived identity path where it is valid, skip or cool down native command deployment when identity preflight is failing or Discord returns startup 429s, and treat deploy 429s as one non-fatal warning that leaves existing commands active. Add focused Discord monitor/deploy tests for route 429 and `fetchUser("@me")` timeout/rate-limit paths.

## Reproduction Assessment

Partially yes. The native slash-command deploy burst is reproducible from current source and tests because a 429 leads to three deploy attempts; the `/users/@me` timeout itself depends on the reporter's Discord/network conditions, but the hard-fail startup path is directly visible in current code.

## Solution Assessment

No. Current main's log-collapse change is useful but not the narrow complete fix because it still performs the retry burst and still treats bot identity fetch timeout as a startup-blocking failure.
