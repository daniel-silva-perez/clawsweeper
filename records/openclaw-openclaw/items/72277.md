## Summary

Keep open. Current main still matches the narrowed failure mode: exec completions preserve a deliveryContext, and heartbeat preflight reads it, but explicit heartbeat targets do not promote that queued deliveryContext into the full turn source needed for Slack thread routing. Open draft PR #71582 remains the active implementation candidate rather than a reason to close this issue.

## What This Changes

The issue requests that Slack async exec completion replies stay in the originating Slack thread when heartbeat delivery is configured with an explicit Slack target.

## Best Possible Solution

Preserve or prefer queued exec-event deliveryContext as the source route during heartbeat delivery resolution, including Slack threadId, even when heartbeat.target and heartbeat.to are explicit. Add a focused Slack or generic target-resolution regression for explicit heartbeat targets while preserving scheduled-heartbeat behavior that avoids stale inherited threads.

## Reproduction Assessment

Yes. The issue gives a high-confidence live Slack reproduction, and current main has a deterministic code-level reproduction path where the queued event context exists but explicit heartbeat target resolution does not pass channel/to/accountId as turn-source metadata.

## Solution Assessment

Yes for the requested direction, but it is not implemented on main. The narrow maintainable fix belongs in generic heartbeat/outbound routing, with Slack proving that a supplied threadId becomes thread_ts rather than changing Slack send behavior itself.
