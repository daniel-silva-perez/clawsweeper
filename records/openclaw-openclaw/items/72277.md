## Summary

Keep open. Current main still has the reported resolver-level gap: async exec completion events can carry a Slack thread deliveryContext, but explicit heartbeat targets do not preserve the queued turn-source route into session target resolution, so the outbound Slack send can receive no threadId and post at channel root. Draft PR #71582 remains the active implementation candidate rather than a reason to close this issue.

## What This Changes

Ensure Slack async exec completion relays stay in the originating Slack thread even when heartbeat delivery is configured with an explicit Slack target and destination.

## Best Possible Solution

Preserve or prefer queued exec-event deliveryContext, including Slack threadId, in the generic heartbeat/outbound resolver even when heartbeat.target and heartbeat.to are explicit. Add focused regression coverage for explicit Slack heartbeat targets while keeping existing stale-thread suppression for normal scheduled heartbeat delivery.

## Reproduction Assessment

Yes. The issue provides a concrete Slack thread async exec reproduction, and current main has a deterministic code-level reproduction path where the event thread exists but explicit heartbeat target resolution does not pass a parsed turn-source channel/to alongside it.

## Solution Assessment

Yes for the direction, but it is not implemented on main. The narrow maintainable fix belongs in generic heartbeat delivery resolution rather than Slack send, because Slack already uses thread_ts when core passes a valid threadId.
