## Summary

Keep open. Current main still matches the reported failure mode: async exec completions capture and enqueue a deliveryContext, but explicit heartbeat targets only pass the raw threadId without treating the queued deliveryContext as the full turn source, so Slack can receive no threadId and post the relay at channel root. The open draft PR #71582 remains the active implementation candidate rather than a reason to close this issue.

## What This Changes

The issue reports that Slack async exec completion replies can escape the originating Slack thread when the exec-event wake is delivered through an explicit Slack heartbeat target.

## Best Possible Solution

Preserve or prefer queued exec-event deliveryContext, including Slack threadId, in the generic heartbeat/outbound resolver even when heartbeat.target and heartbeat.to are explicit. Add a focused Slack regression for explicit heartbeat targets while preserving the existing scheduled-heartbeat behavior that avoids stale inherited threads.

## Reproduction Assessment

Yes. The issue provides a concrete Slack thread async exec reproduction, and current main has a deterministic code-level reproduction path where the event thread exists but explicit heartbeat target resolution does not pass a parsed turn-source channel/to alongside it.

## Solution Assessment

Yes for the proposed direction, but it is not implemented on main. The narrow maintainable fix belongs in generic heartbeat delivery resolution, because Slack already sends thread_ts when core passes a valid threadId.
