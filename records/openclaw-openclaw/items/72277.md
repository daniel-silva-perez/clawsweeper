## Summary

Keep open. Current main still only treats queued system-event delivery context as turn-source routing when the heartbeat target is `last`; an explicit Slack heartbeat target/to can still drop the originating thread before assistant context and Slack outbound delivery receive it. The related open draft PR #71582 remains the active implementation candidate for this routing gap.

## What This Changes

Slack async exec completion relays should stay in the originating Slack thread when the completion wake is routed through an explicit Slack heartbeat target/to.

## Best Possible Solution

Preserve the queued exec-event deliveryContext, including Slack threadId, through heartbeat delivery resolution even when the configured heartbeat target is an explicit Slack target/to. The fix should live in the generic heartbeat/outbound routing path, keep stale-thread suppression for ordinary scheduled heartbeats, and add a focused regression for explicit Slack heartbeat target routing.

## Reproduction Assessment

Yes. The reporter provided a concrete Slack async exec reproduction, and current main’s source path reproduces the route loss at the resolver level for explicit heartbeat targets: the event thread exists, but it is not treated as an active turn-source target before heartbeat mode clears thread routing.

## Solution Assessment

Yes for the direction, not yet implemented. The narrow maintainable solution is to carry or prefer exec-event deliveryContext in generic heartbeat target resolution rather than patching Slack send, because Slack only receives whatever threadId the core resolver preserves.
