## Summary

Current main still appears vulnerable to the reported edge case: exec completions capture a deliveryContext with threadId, but heartbeat delivery target resolution only applies that turn-source route when heartbeat target is "last". With an explicit Slack heartbeat target/to, the event threadId is dropped before the assistant context and outbound Slack send path receive it. An open PR, #71582, appears to be the active implementation candidate for this exec-event route-clearing/routing problem, so this issue should stay open pending review/landing or explicit maintainer closure of that PR.

## Best Possible Solution

Keep this issue open and route it to the active exec-event heartbeat routing work in #71582. The best fix should make exec-event wakes prefer or carry the event deliveryContext, including Slack threadId, even when the configured heartbeat target is an explicit Slack target/to, while preserving stale-thread suppression for ordinary scheduled heartbeats and adding a focused Slack explicit-target regression test.
