## Summary

Keep open. Current main still matches the narrowed failure mode: async exec completion events carry a deliveryContext, but explicit heartbeat targets do not turn that queued context into a full turn-source route, so Slack thread routing can lose the originating thread. Open draft PR #71582 remains the active implementation candidate, not a reason to close this issue before it lands or is replaced.

## What This Changes

The issue requests that Slack async exec completion relays stay in the originating channel thread when heartbeat delivery is configured with an explicit Slack target.

## Best Possible Solution

Preserve or prefer queued exec-event deliveryContext as the source route during heartbeat delivery resolution, including Slack threadId, even when heartbeat.target and heartbeat.to are explicit. Add a focused generic outbound or Slack regression for explicit heartbeat targets while preserving scheduled-heartbeat behavior that avoids stale inherited threads.

## Reproduction Assessment

Yes. The issue provides a high-confidence live Slack reproduction, and current main has a deterministic code-level path where an explicit heartbeat target receives only a bare turnSourceThreadId that the heartbeat resolver will not honor without turnSourceChannel.

## Solution Assessment

Yes for the requested direction, but it is not implemented on main. The narrow maintainable fix belongs in generic heartbeat/outbound target resolution, with Slack coverage proving the preserved threadId becomes thread_ts rather than changing Slack send behavior itself.
