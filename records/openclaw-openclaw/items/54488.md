## Summary

Keep open. Current main still has the central scheduler fairness problem: active inbound work is appended to the same followup FIFO queue, the drain continues until that queue is empty, and each followup turn runs through the same per-session lane. Recent context-engine maintenance changes reduce one contributor, but they do not bound or prioritize the followup drain itself.

## What This Changes

The issue requests fair session-lane scheduling so followup drains cannot monopolize a session and delay later inbound Discord, WhatsApp, or other channel messages for many minutes.

## Best Possible Solution

Add scheduler-level fairness for the auto-reply followup path: bound consecutive followup turns or elapsed drain time, give fresh user/control inbound work a way to run ahead of accumulated system/followup backlog on the same session key, and keep ordering guarantees within each priority class with focused regression coverage.

## Repair Work Prompt

Fix the session-lane starvation reported in this issue on current main. Implement bounded/fair followup draining in the core auto-reply/session scheduler so accumulated followup/system-event work cannot monopolize `session:<key>` and later fresh user/control inbound work can run before the rest of the backlog. Preserve ordering within the same priority class and keep collect-mode authorization grouping, cross-channel routing, debounce, cap, and drop-policy behavior intact. Do not hardcode Discord, WhatsApp, Telegram, provider ids, or model behavior. Do not try to solve the broader timeout-attribution umbrella in #65504 or the durable subagent lifecycle notification architecture in #66638 in the same PR. Add focused regression coverage that enqueues multiple followups, then a later fresh inbound/control message on the same session key during an active followup drain, and proves the fresh work is not starved behind the whole backlog. If the behavior change gets a changelog entry, credit `Thanks @Flakedict`.
