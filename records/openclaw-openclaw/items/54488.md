## Summary

Keep this issue open. Current main still has the central scheduler fairness problem: busy inbound messages are appended to the same followup queue, the drain runs that queue to empty without a max-turn/yield policy, and each followup can run a full embedded agent turn through the same FIFO per-session lane. Related #65504 and #66638 remain adjacent but do not supersede this specific followup-drain starvation bug.

## Best Possible Solution

Keep this issue open and fix it in the core auto-reply/session scheduler. The likely path is a bounded or fair followup drain, explicit priority/yielding for fresh user messages over accumulated followup/system-event work on the same queue key, careful handling of heartbeat/system events, and a focused regression test that enqueues multiple slow followups plus a later fresh inbound message on the same session key and proves the fresh turn is not starved behind the whole backlog.
