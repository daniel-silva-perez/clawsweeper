## Summary

Keep open. Current main still has the core fairness problem: busy inbound work is appended to the same followup FIFO, the followup drain runs until the queue is empty, and each followup turn still runs through the same per-session FIFO lane. Recent context-engine deferred maintenance and heartbeat busy-lane skips reduce adjacent latency contributors, but they do not bound or prioritize the followup drain itself.

## What This Changes

The issue requests scheduler fairness for followup drains so fresh inbound user/control messages are not starved behind accumulated followup/system work on the same session lane.

## Best Possible Solution

Add scheduler-level fairness to the auto-reply followup path: bound consecutive followup turns or drain elapsed time, give fresh user/control inbound work a way to run ahead of accumulated system/followup backlog on the same session key, and preserve FIFO ordering within each class with focused regression coverage.
