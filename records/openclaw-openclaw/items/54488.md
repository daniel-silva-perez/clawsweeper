## Summary

Keep open. Current main still routes active inbound work into the same followup FIFO, drains that queue until empty, and runs every followup through the same per-session FIFO lane. Recent deferred context-engine maintenance reduces one latency contributor, but it does not bound or prioritize the followup drain itself, and the adjacent open fast-lane PR does not close this issue.

## What This Changes

The issue reports that followup queue draining can monopolize a session lane for 20-30 minutes, delaying new Discord and WhatsApp inbound messages behind accumulated followup and system work.

## Best Possible Solution

Add scheduler-level fairness to the auto-reply followup path: bound consecutive followup turns or drain elapsed time, give fresh user/control inbound work a way to run before accumulated system/followup backlog on the same session key, and preserve FIFO ordering within each class with focused regression coverage.
