## Summary

Keep this PR open. Current main still does not implement immediate live rendering of inbound user turns in TUI `--session` mode, and the PR is still a plausible implementation candidate for the linked #45388 workflow. It should not be closed as implemented because main lacks the proposed `user-message` chat event and the TUI still does not subscribe to the existing `session.message` transcript stream.

## Best Possible Solution

Keep this PR open for maintainer follow-up. The strongest path is either to have TUI `--session` subscribe to `sessions.messages.subscribe` for the active session and render `session.message` user turns with local echo/dedupe protection, or to emit a versioned user-turn event from the shared external-channel ingress/transcript pipeline used by Telegram and other channels. Update protocol schema/docs and focused TUI/gateway regression coverage, and split the unrelated model-selection test change.

## Risks / Open Questions

- Merging the PR as-is could leave the linked Telegram/external-channel workflow unfixed because the current hook is on `chat.send`, not the shared external ingress/transcript path.
- Adding raw inbound user text to the `chat` event family is a protocol and privacy surface; it needs explicit schema/docs/client compatibility review and strong session filtering.
- The PR bundles unrelated model-selection TDZ regression coverage from #45368, which should be split, dropped, or reviewed separately before landing the TUI fix.
