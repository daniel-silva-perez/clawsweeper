## Summary

Keep this PR open. Current main at e476523082a6fff3a0c3048077d2f0e52413fbf2 still does not implement immediate live rendering of inbound user turns in TUI --session mode: the chat protocol lacks the proposed user-message event, the TUI handler has no addUser event path, and the TUI backend does not subscribe to the existing session.message transcript stream. The PR remains a plausible implementation candidate for #45388, but the discussion also identifies a real unresolved design gap: the proposed broadcast is wired to chat.send, while external channel ingress paths call dispatchInboundMessage directly.

## Best Possible Solution

Keep this PR open for maintainer follow-up. The strongest path is either to have TUI --session subscribe to sessions.messages.subscribe for the active session and render session.message user turns with local echo/dedupe protection, or to emit a versioned user-turn event from the shared external-channel ingress/transcript pipeline used by Telegram and other channels. Update protocol docs/schema and focused TUI/gateway regression coverage, and split the unrelated #45368 model-selection test change.
