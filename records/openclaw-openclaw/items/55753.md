## Summary

Keep this PR open. Current main still unconditionally copies the Kubernetes ConfigMap files into the PVC-backed OpenClaw state on every pod start, so the data-loss problem described by the PR is still present. The PR is not obsolete, but it needs maintainer judgment because its one-time copy behavior conflicts with the documented Kubernetes redeploy flow that says ConfigMap changes are picked up by redeploying.

## Best Possible Solution

Keep the PR open and ask maintainers to decide the Kubernetes config ownership contract. If PVC files are user-owned after first boot, land a seed-only implementation with docs updated to say ConfigMap changes no longer reconcile existing PVC files. If ConfigMap remains the declarative source, replace the unconditional copy with a safer mechanism such as backup-before-overwrite, checksum/version-aware reconciliation, or an explicit reset/sync operation.
