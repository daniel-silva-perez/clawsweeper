## Summary

Keep open. Current main has related configurable-limit work, but it does not implement the requested default increase or byte-aware visible truncation, and this PR still has a correctness bug because its fallback budgets raw UTF-8 text while the hard cap budgets serialized JSON bytes.

## What This Changes

This PR raises the old gateway `chat.history` text cap to 64,000, adds raw UTF-8 byte-aware truncation before oversized placeholder replacement, and adds gateway regression tests for long ASCII and emoji-heavy assistant replies.

## Best Possible Solution

Implement this behavior in the shared chat display projection path, preserve long assistant text when the fully serialized history entry fits the hard cap, truncate UTF-8-heavy and JSON-escape-heavy text against the same serialized byte budget used by the oversized guard, and cover the gateway plus Control UI default/config path with focused tests and docs/config-help updates if the default changes.

## Reproduction Assessment

Yes. A current-main reproduction is clear from source: a default Control UI `chat.history` read of a 50,000-character ASCII assistant message is projected through the 8,000-character default cap, while the PR defect can be reproduced with JSON-escape-heavy text that passes raw byte budgeting but exceeds the later `JSON.stringify(message)` hard cap.

## Solution Assessment

No. The PR addresses the right user problem, but it is not the best current implementation because main moved the behavior into `src/gateway/chat-display-projection.ts` and the proposed byte fallback does not budget against serialized history bytes.
