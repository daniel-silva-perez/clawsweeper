## Summary

Keep this PR open. Current main still truncates chat.history display text by character count with an 8,000-character default before the 128 KiB JSON byte placeholder pass, so the intended behavior is not implemented on main. The PR is still relevant, but it is stale against the current chat-display-projection split and has an unresolved review concern around raw UTF-8 budgeting versus JSON-encoded bytes.

## What This Changes

This PR raises the chat.history text cap to 64,000 characters, adds byte-aware fallback truncation for UTF-8-heavy text, and adds gateway regression tests for long ASCII and emoji-heavy assistant history.

## Best Possible Solution

Current main should preserve long assistant history text that fits the serialized chat.history budget, degrade byte-heavy or JSON-escape-heavy content to a readable truncated prefix before placeholder replacement, and cover the default Control UI/WebChat path plus related history endpoints and docs.
