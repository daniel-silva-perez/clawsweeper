## Summary

Keep this PR open for contributor rework or maintainer review. Current main has related configurable-limit work, but it still defaults to an 8,000-character projection cap and has no serialized-byte-aware visible-prefix fallback, while this PR still has a blocking truncation-budget defect and is paired with open #53242 via closing syntax.

## What This Changes

The PR raises the `chat.history` text cap from 12,000 to 64,000, adds raw UTF-8 byte-aware truncation before the oversized-placeholder guard, and adds gateway regression tests for long ASCII and emoji-heavy assistant history.

## Best Possible Solution

Implement visible-prefix truncation in the shared chat display projection path, preserve long assistant text that fits the serialized history budget, truncate UTF-8-heavy or JSON-escape-heavy text against serialized bytes before placeholder replacement, and align the chosen default/config semantics with focused gateway and UI coverage.

## Review Findings

Full review comments:

- **[P2] Measure truncation against serialized history bytes:** `src/gateway/server-methods/chat.ts:425-427`
  - body: The fallback trims candidates with `Buffer.byteLength(candidate, "utf8")`, but the later oversized gate measures `JSON.stringify(message)`. JSON-escape-heavy text, such as control characters, can fit this raw text budget, expand past the 128 KiB single-message cap when serialized, and still be replaced entirely by the placeholder. Budget the prefix against the serialized history message bytes so these messages degrade to visible truncation instead of omission.
