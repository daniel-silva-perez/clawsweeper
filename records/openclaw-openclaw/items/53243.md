## Summary

Keep open: current main still truncates chat-history display text by character count in the shared projection path, and this PR's byte-aware fallback still budgets raw text bytes rather than the JSON-serialized message bytes used by the later oversized-message guard. Merged #58900 is related because it made the limit configurable, but it did not implement this PR's default increase or serialized-byte-aware visible-prefix fallback.

## What This Changes

The PR raises the old `chat.history` text cap to 64,000 characters, adds a UTF-8 byte-aware truncation fallback in gateway chat-history code, and adds gateway regression tests for long ASCII and emoji-heavy assistant messages.

## Best Possible Solution

Implement visible-prefix truncation in `src/gateway/chat-display-projection.ts`, preserve long assistant text when the fully serialized history message fits the hard cap, truncate UTF-8-heavy and JSON-escape-heavy text against serialized bytes before placeholder replacement, and align focused gateway/UI tests plus docs or config comments with the chosen default semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Measure truncation against serialized history bytes:** `src/gateway/server-methods/chat.ts:425-427`
  - body: The fallback trims candidates with `Buffer.byteLength(candidate, "utf8")`, but the later oversized gate measures `JSON.stringify(message)`. Escape-heavy text such as control characters can stay under this raw text budget, expand past the 128 KiB single-message cap when serialized, and still be replaced entirely by the placeholder. Budget the prefix against the JSON-encoded history message bytes so these messages degrade to visible truncation instead of omission.
