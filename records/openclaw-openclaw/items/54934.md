## Summary

Keep open. Current main already fixes the linked Feishu send/thread-reply failure from #54430 by ignoring native `params.card` on the message send path and only creating Feishu cards from normalized `presentation`; empty `card: {}` with text now falls through to text send. However, this PR also adds runtime-layer guards for direct `sendCardFeishu` and `editMessageFeishu`, and current main still accepts `{}` on those helper paths. That remaining part of the PR is small, relevant, and not clearly superseded.

## Best Possible Solution

Keep the current presentation-based Feishu send path, but apply or rework the small runtime-layer validation from this PR in `extensions/feishu/src/send.ts` so direct `sendCardFeishu` and `editMessageFeishu` reject empty card objects before calling Feishu. Add the focused unit tests from the PR or equivalent current-main tests, then close the PR once that guard is landed or explicitly deemed out of scope by a maintainer.
