## Summary

Keep open. Current main already avoids the reported Feishu message-tool failure by using the shared `presentation` contract for send/thread-reply, but this PR still contains useful direct helper hardening and is stale because its channel hunk targets the old raw `ctx.params.card` send path.

## What This Changes

The PR adds Feishu empty-card validation to the message action dispatcher and send/edit helpers, with tests for card-only rejection, text fallback, and direct helper rejection.

## Best Possible Solution

Keep the shipped presentation-based message send behavior. If maintainers want native card hardening, land a narrow follow-up around `sendCardFeishu`, `editMessageFeishu`, and any native outbound entry points, with focused tests and no shared message-tool raw-card support.

## Reproduction Assessment

Yes. The stale PR problem is reproducible by comparing the PR hunk against current main: the changed channel lines target raw `ctx.params.card`, while current main sends cards only from `ctx.params.presentation`; the remaining helper gap is visible in `sendCardFeishu({ card: {} })` and `editMessageFeishu({ card: {} })` source flow.

## Solution Assessment

No. The helper guard idea is reasonable, but the PR as written is not the best fix because it mixes that hardening with an obsolete channel dispatcher change; a smaller helper/native-outbound patch would be safer.
