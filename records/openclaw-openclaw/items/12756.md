## Summary

Keep this issue open. Current main still does not implement Telegram `message read`: the shared action name exists, but the Telegram plugin does not advertise, map, gate, dispatch, test, or document a Telegram read action. Closed PR #14367 is useful prior art around an inbound-store/no-`getUpdates` approach, but it was not merged and does not make current main functional.

## Best Possible Solution

Keep this issue open and implement the feature in the Telegram plugin, not core. The best path is a maintainer-approved no-`getUpdates` design, likely a bounded inbound message store or equivalent runtime-owned cache, then wire `read` through Telegram action discovery, action mapping, runtime aliases/dispatch, config gates, CLI `--target`/`to` plus topic/thread semantics, docs, and focused adapter/runtime/contract tests.
