## Summary

Keep open. Current main still has the shared and CLI `read` action, but Telegram does not expose, map, gate, dispatch, test, or document a Telegram message-history read path; the related inbound-store PR was closed unmerged, so this remains the live tracker.

## What This Changes

The issue requests Telegram support for the shared `message read` action so users can fetch recent Telegram group messages with limits and pagination-like filters.

## Best Possible Solution

Implement Telegram `message read` inside the Telegram plugin with explicit observed-only semantics, a bounded privacy-reviewed runtime-owned inbound cache or equivalent store, no secondary `getUpdates` polling, and full wiring through discovery, adapter mapping, runtime dispatch, config/schema help, CLI/channel docs, and focused tests.

## Reproduction Assessment

Yes. The current-main static path is high-confidence: `openclaw message read --channel telegram --target ...` reaches the shared `read` action, but the Telegram adapter has no `read` mapping and rejects unmapped actions as unsupported.

## Solution Assessment

Unclear until maintainers approve the Telegram read semantics. The user-facing `message read` surface is the right target, but the safer implementation is a Telegram-owned observed-message store with explicit retention and privacy limits rather than a separate history fetch.
