## Summary

Current main still does not implement Telegram `message read`. The shared action name exists, but the Telegram plugin does not advertise `read`, does not map it to a Telegram runtime action, has no read runtime handler or inbound message-store path, and the CLI docs still list `read` only for Discord/Slack/Matrix. The prior implementation PR #14367 is useful design context, but it is closed unmerged and absent from current main.

## Best Possible Solution

Keep this issue open. Implement Telegram `message read` end-to-end in the bundled Telegram plugin only after maintainers settle the inbound-store semantics: advertise `read` in Telegram discovery, map the shared action to a Telegram runtime handler, record bounded Telegram inbound messages safely, align `--target`/`to`, `limit`, `before`, `after`, and topic/thread handling, update docs, and add focused adapter/runtime/contract tests. Treat #14367 as prior art, not shipped proof.

## Risks / Open Questions

- A correct implementation cannot fetch arbitrary Telegram chat history through the Bot API; it likely needs a bounded store of messages the bot has already received, with documented restart, retention, pagination, and privacy semantics.
- Adding a Telegram inbound message store would handle user/group message content, so maintainers should explicitly choose in-memory vs persisted storage, retention defaults, and access-gating behavior before an automated fix PR attempts it.
- The closed prior PR #14367 is useful prior art but was not merged and its Greptile review noted end-to-end wiring and parameter alignment gaps.
