## Summary

Keep open. Current main still has a shared `read` message action, but the bundled Telegram plugin does not expose, map, dispatch, gate, test, or document Telegram message reads. The unmerged #14367 inbound-store PR and closed #24602 duplicate are useful context, but neither resolves the current main behavior.

## What This Changes

Request support for Telegram `openclaw message read` so Telegram chats and groups can be monitored with target, limit, before, and after-style parameters.

## Best Possible Solution

Implement Telegram `message read` in the Telegram plugin with a bounded runtime-owned inbound message store or equivalent cache, avoid secondary `getUpdates` polling, wire the action through discovery, mapping, runtime dispatch, config gating, CLI target/thread semantics, docs, and focused tests, and document the platform limitation that only observed recent messages can be read.

## Risks / Open Questions

- Telegram bots do not provide a full per-chat history endpoint, so the expected feature likely must be limited to messages observed while the OpenClaw Telegram runtime is running.
- A read implementation would retain inbound Telegram message content, so retention limits, opt-in behavior, privacy review, and documentation need maintainer judgment.
- The prior #14367 approach was closed unmerged and had reported end-to-end wiring issues, so it should be treated as prior art rather than a ready fix.
