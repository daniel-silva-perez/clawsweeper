## Summary

Keep open. Current main still lacks Telegram support for the shared `message read` action across discovery, adapter mapping, runtime dispatch, config gating, docs, and tests. The prior inbound-store PR #14367 is useful implementation context but closed unmerged, and the related duplicate #24602 is closed, so this issue remains the live tracker.

## What This Changes

The issue requests Telegram support for the shared `message read` action so agents can read recent Telegram group messages through `openclaw message read --channel telegram`.

## Best Possible Solution

Implement Telegram `message read` inside the Telegram plugin with a bounded runtime-owned inbound message store or equivalent observed-message cache, avoid secondary `getUpdates` polling, wire the action through discovery, adapter mapping, runtime dispatch, config gating, CLI target/topic semantics, docs, and focused tests, and document that Telegram can only return observed recent messages.
