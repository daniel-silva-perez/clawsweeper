## Summary

Keep open. Current main still has the shared `read` message action, but the bundled Telegram plugin does not advertise, map, gate, dispatch, test, or document Telegram message-history reads. The related implementation PR is closed unmerged and the duplicate issue is closed, so this remains the live tracker for the Telegram-specific gap.

## What This Changes

Add Telegram support for the shared `message read` action so `openclaw message read --channel telegram ...` can return recent Telegram chat messages for monitoring workflows.

## Best Possible Solution

Implement Telegram `message read` inside the Telegram plugin with explicit observed-message semantics, a bounded privacy-reviewed inbound cache or equivalent runtime-owned store, no secondary `getUpdates` polling, and complete wiring through discovery, action mapping, runtime dispatch, config schema/help, CLI/channel docs, and focused tests.

## Reproduction Assessment

Yes. The current-main code path is high-confidence: `openclaw message read --channel telegram --target ...` reaches the shared `read` action, but the Telegram adapter has no `read` mapping and rejects unmapped actions as unsupported.

## Solution Assessment

No current solution exists on main. The requested surface is the right user-facing action, but the maintainable implementation should be a Telegram-owned observed-message store with clear retention/privacy limits rather than trying to fetch historical chat messages through `getUpdates`.
