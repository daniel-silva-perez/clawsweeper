## Summary

Keep open. Current main still has the shared and CLI `read` action, but the Telegram plugin does not expose, map, gate, dispatch, test, or document Telegram message-history reads; the related inbound-store PR is closed unmerged, so this remains the live tracker.

## What This Changes

Add Telegram support for `openclaw message read` so agents can retrieve recent messages from Telegram chats or groups.

## Best Possible Solution

Implement Telegram `message read` inside the Telegram plugin with explicit observed-only semantics, a bounded privacy-reviewed runtime-owned inbound cache or equivalent store, no secondary `getUpdates` polling, and complete wiring through discovery, adapter mapping, runtime dispatch, config/schema help, CLI/channel docs, and focused tests.

## Reproduction Assessment

Yes. The current-main static path is high-confidence: `openclaw message read --channel telegram --target ...` reaches the shared `read` action, but the Telegram adapter has no `read` mapping and rejects unmapped actions as unsupported.

## Solution Assessment

Unclear until maintainers approve the Telegram read semantics. The user-facing `message read` action is the right surface, but the safer implementation is a Telegram-owned observed-message store with retention/privacy limits, not a separate `getUpdates` history fetch.
