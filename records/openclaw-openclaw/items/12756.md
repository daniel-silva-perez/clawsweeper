## Summary

Keep open. Current main still has the shared and CLI `read` action, but the Telegram plugin does not expose, map, gate, dispatch, test, or document Telegram message reads; the related inbound-store PR was closed unmerged and had end-to-end wiring concerns, so this remains a valid tracker.

## What This Changes

Request Telegram support for `openclaw message read` so agents can read recent observed messages from Telegram chats or groups.

## Best Possible Solution

Implement Telegram `message read` inside the Telegram plugin as an explicit observed-only read feature backed by a bounded, privacy-reviewed runtime-owned inbound cache or equivalent store, wired through discovery, adapter mapping, runtime dispatch, config/schema help, CLI/channel docs, and focused tests without secondary `getUpdates` polling.

## Reproduction Assessment

Yes. The current-main static path is high-confidence: the CLI accepts `message read` and maps the target, but Telegram has no `read` mapping or runtime alias, so the Telegram adapter rejects the action as unsupported.

## Solution Assessment

Unclear until maintainers approve the Telegram read semantics. The requested `message read` surface is the right user-facing target, but the maintainable implementation should be a Telegram-owned observed-message store with explicit retention and privacy limits rather than a direct Bot API history fetch.
