## Summary

Keep this issue open. Current main still does not implement Telegram `message read`: the shared action name exists, but Telegram does not advertise `read`, map it to a runtime action, dispatch it, gate it in config, or document it as supported. Closed PR #14367 is useful prior art for an inbound-store/no-`getUpdates` approach, and #24602 was closed as a duplicate of this tracking issue, but neither changes current main behavior.

## Best Possible Solution

Keep this issue open and implement the feature in the Telegram plugin, not core. The best path is a maintainer-approved no-`getUpdates` design, likely a bounded inbound message store or equivalent runtime-owned cache, then wire `read` through Telegram action discovery, action mapping, runtime aliases/dispatch, config gates, CLI `--target`/`to` plus topic/thread semantics, docs, and focused adapter/runtime/contract tests. PR #14367 is useful prior art but needs a current-main design/port rather than treating this as already fixed.
