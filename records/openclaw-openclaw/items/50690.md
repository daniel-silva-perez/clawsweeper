## Summary

Keep open. Current main has a generic outbound `message_sending` rewrite/cancel hook plus narrower commentary and isolated-cron stale-ack mitigations, but it still lacks the requested execution-state-backed fake-progress policy across user-facing delivery paths. The related outbound-boundary and pre-send-hook reports overlap without fully superseding this issue’s ack/progress classification, repeat suppression, progress-debt, and durable-state requirements.

## What This Changes

Add a runtime pre-send policy layer that classifies outbound assistant messages and blocks fake-progress acknowledgements, commentary leakage, repeated ack spam, silent post-start drift, and bursty low-value progress updates unless backed by real execution state.

## Best Possible Solution

Define one first-class outbound policy boundary, then implement fake-progress enforcement on top with durable execution state, typed ack/progress/blocker/completion classification, per-session repeat and progress-debt tracking, structured suppression metrics, and integration coverage through shared delivery plus channel/tool-specific sends.
