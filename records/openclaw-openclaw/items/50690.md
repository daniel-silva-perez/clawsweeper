## Summary

Keep open. Current main has useful partial mitigation, but it still does not implement the requested fake-progress pre-send policy: outbound hooks can rewrite or cancel content, commentary-phase text is suppressed in some assistant stream paths, and isolated cron retries stale acknowledgements, yet there is no shared runtime classifier that gates ack/progress/completion delivery on durable execution and progress state across user-facing channels. Ryce's keepmyclaw snapshot comment is relevant because the proposed gate depends on durable execution state, not only prompt or text filtering.

## What This Changes

The issue requests a runtime-enforced outbound pre-send policy layer that classifies assistant messages and blocks fake progress or repeated acknowledgements unless backed by execution/progress state.

## Best Possible Solution

Define one first-class outbound policy boundary, then implement this fake-progress policy on top with durable execution state, typed ack/progress/blocker/completion classification, per-session repeat and progress-debt tracking, structured suppression metrics, and integration coverage through shared outbound delivery plus channel/tool-specific send paths.
