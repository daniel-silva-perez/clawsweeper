## Summary

Keep open. Current main has useful Active Memory latency mitigations, including result caching, setup-grace timeout handling, partial timeout recovery, and circuit-breaker behavior, but it still does not implement warm standby or persistent embedded-agent session reuse; uncached recalls still allocate a fresh Active Memory subagent identity/session file and enter the embedded runner through the normal per-call setup path.

## What This Changes

The issue requests warm standby or persistent session reuse for Active Memory embedded-agent runs to reduce repeated cold-start latency.

## Best Possible Solution

Keep this as Active Memory embedded-agent warm reuse work. A maintainer-quality design should choose whether reuse is plugin-owned per parent session or exposed as a generic embedded-runner/session API, then implement bounded TTL/invalidation, isolation guarantees, and tests proving repeated Active Memory turns skip the expensive setup path without cross-session state leakage.

## Reproduction Assessment

Yes for the missing reuse behavior: two uncached Active Memory prompt-build paths can be inspected or mocked to show a new Active Memory session id/session file and a new runEmbeddedPiAgent call each time. The exact 10-30s latency magnitude still needs a live configured provider/gateway timing run.

## Solution Assessment

Unclear. Warm standby/session reuse is a plausible direction, but the best implementation boundary is not settled because it may belong in the Active Memory plugin or a generic embedded-runner API with stricter invalidation and isolation semantics.
