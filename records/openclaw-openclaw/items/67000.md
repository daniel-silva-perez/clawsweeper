## Summary

Keep open. Current main has active-memory latency mitigations, result caching, timeout partial recovery, and circuit-breaker behavior, but it still does not implement warm standby or persistent embedded-agent session reuse; uncached active-memory recalls still create a fresh sub-agent identity/session file and call the embedded runner through the normal per-call setup path.

## What This Changes

The issue requests warm standby or persistent session reuse for active-memory embedded-agent runs to reduce repeated cold-start latency.

## Best Possible Solution

Keep this as active-memory embedded-agent warm reuse work. A maintainer-quality design should choose whether the pool is plugin-owned per parent session or exposed as a generic embedded-runner/session API, then implement bounded TTL/invalidation, isolation guarantees, and tests proving repeated active-memory turns skip the expensive setup path without cross-session state leakage.

## Reproduction Assessment

Yes for the missing reuse behavior: two uncached active-memory prompt-build paths can be inspected or mocked to show a new active-memory session id/session file and a new runEmbeddedPiAgent call each time. The exact 10-30s latency magnitude still needs a live configured provider/gateway timing run.

## Solution Assessment

Unclear. Warm standby/session reuse is a plausible direction, but the best implementation boundary is not settled because it may belong in the active-memory plugin or a generic embedded-runner API with stricter invalidation and isolation semantics.
