## Summary

Keep open. Current main still matches the reported Copilot auth-order bug: the shared GitHub Copilot token helper builds a raw provider profile list and selects the first id, while the implicit catalog and embedding paths both call that helper. Open PR #48682 is already the implementation candidate, so closing this issue or queueing duplicate automated repair work would be premature.

## What This Changes

The issue reports that GitHub Copilot implicit auth profile selection ignores auth.order and picks the first stored auth profile by object insertion order.

## Best Possible Solution

Land or rework the plugin-owned fix so resolveFirstGithubToken iterates order-aware eligible Copilot profile ids while preserving environment-token precedence, then cover both implicit catalog discovery and Copilot memory embeddings with targeted regression tests.

## Reproduction Assessment

Yes. Configure two stored github-copilot token profiles, set auth.order.github-copilot to prefer the non-first profile, and run the implicit Copilot helper path with no env token; current main still reaches profileIds[0].

## Solution Assessment

Yes. The narrow maintainable fix belongs in the GitHub Copilot plugin helper using the existing order-aware auth seam, with #48682 as the current review candidate rather than a duplicate repair branch.
