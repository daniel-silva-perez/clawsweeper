## Summary

Keep open. Current main still matches the reported GitHub Copilot auth-order bug: the shared Copilot token helper builds a raw provider profile list and selects the first entry, while both implicit catalog discovery and the memory embedding path call that helper. Open PR #48682 already references this issue as the fix candidate, so the right cleanup action is maintainer review of that PR rather than closing this issue or queueing a duplicate repair branch.

## What This Changes

The issue requests that implicit GitHub Copilot auth profile selection honor `auth.order` instead of always using the first stored `auth.profiles` entry.

## Best Possible Solution

Land or rework a plugin-owned fix so `resolveFirstGithubToken` preserves environment-token precedence, then iterates order-aware eligible Copilot profile ids from the existing auth-order seam. Cover both implicit catalog discovery and Copilot memory embeddings with targeted regression tests, and avoid moving GitHub Copilot-specific policy into core.

## Reproduction Assessment

Yes. Configure two stored `github-copilot` token profiles, set `auth.order.github-copilot` to prefer the non-first profile, and run implicit Copilot discovery or embedding setup with no env token; current main still reaches `profileIds[0]`.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the GitHub Copilot plugin helper using the existing order-aware auth seam; #48682 is the current review candidate, so a duplicate repair branch is not the best path.
