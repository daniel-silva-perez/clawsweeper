## Summary

Keep open. Current main still matches the report: the shared GitHub Copilot token helper uses the raw provider profile list and selects the first stored profile, so configured `auth.order` is bypassed for implicit Copilot discovery. The issue is also paired with open PR #48682 using closing syntax, so it should stay open until that implementation is reviewed, reworked, merged, or closed.

## What This Changes

The issue requests that GitHub Copilot implicit auth profile selection honor `auth.order` instead of always using the first stored `auth.profiles` key.

## Best Possible Solution

Update the plugin-owned Copilot token selection path so `resolveFirstGithubToken` iterates order-aware eligible profiles, keeps env tokens first, and covers both catalog discovery and embeddings with focused regression tests.

## Reproduction Assessment

Yes. A static reproduction is clear: configure two stored `github-copilot` token profiles, set `auth.order.github-copilot` to the non-first profile, and run the implicit Copilot helper path with no env token; current code still chooses `profileIds[0]`.

## Solution Assessment

Yes, the narrow maintainable fix is to use the existing order-aware auth-profile helper inside the GitHub Copilot plugin helper rather than adding core provider-specific behavior.
