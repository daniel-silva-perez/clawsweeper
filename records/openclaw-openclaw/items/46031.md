## Summary

Keep open. Current main still matches the report: the GitHub Copilot token helper builds a raw provider profile list and selects `profileIds[0]`, so configured `auth.order` is bypassed for implicit Copilot discovery. The issue is also paired with open PR #48682 using closing syntax, so it should remain open until that PR is reviewed, reworked, merged, or closed.

## What This Changes

The issue requests that implicit GitHub Copilot profile selection respect `auth.order` instead of selecting the first stored `auth.profiles` key.

## Best Possible Solution

Update the plugin-owned Copilot token selection helper to iterate order-aware eligible profile IDs while preserving environment-token precedence, then cover implicit catalog discovery and embeddings with focused regression tests. The existing implementation candidate is PR #48682, so maintainers should review, land, rework, or close that PR rather than start a duplicate lane.

## Reproduction Assessment

Yes. Configure two stored `github-copilot` token profiles, set `auth.order.github-copilot` to prefer the non-first profile, and run the implicit Copilot helper path with no env token; current code still selects `profileIds[0]`.

## Solution Assessment

Yes. The narrow maintainable fix is to use the existing order-aware auth-profile helper inside the GitHub Copilot plugin helper, not to add provider-specific behavior in core.
