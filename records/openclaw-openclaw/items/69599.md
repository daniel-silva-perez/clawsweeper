## Summary

Keep this issue open. Current main still routes the Control UI New Session button and typed `/new`/`/reset` through `chat.send` on the active session key, while the independent fresh dashboard session path remains `sessions.create`. The bug is narrow and still actionable: Control UI should use `sessions.create` and switch to the returned session key without changing generic gateway/channel reset semantics.

## Best Possible Solution

Implement a Control UI-owned fresh-session path: the New Session button and Control UI typed `/new` or `/reset` should call `sessions.create` for the active agent, optionally set the current session as parent when available, switch the UI to the returned key using existing `switchChatSession` machinery, refresh sessions/history/avatar/commands, and preserve non-UI gateway `chat.send` reset semantics unless maintainers intentionally change that contract.
