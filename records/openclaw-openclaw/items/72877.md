## Summary

Keep open. Current main still uses the old native and isolated-cron precedence where the target agent primary model beats `agents.defaults.subagents.model`, so the PR is not obsolete. The branch still needs revision because its child-session persistence path marks every spawned model override as `auto`, which loses exact user-selection semantics for explicit `sessions_spawn.model`; current main already has a safer source-preserving version of that part.

## What This Changes

The PR reorders native and isolated-cron subagent model precedence, persists selected child-session model override fields, updates related tests, and pins Yuanbao external channel catalog metadata.

## Best Possible Solution

Revise this into one focused agent-model patch: keep the intended precedence as per-agent `subagents.model`, then global `agents.defaults.subagents.model`, then target agent primary model; preserve explicit spawned models as `modelOverrideSource: "user"`; and drop already-on-main catalog/runtime persistence noise after rebasing.

## Reproduction Assessment

Yes. The code-level reproduction is to call `sessions_spawn` with an explicit `model` while configured fallbacks exist; the PR head stores that child-session override as `auto`, and `resolveEffectiveModelFallbacks` treats auto overrides as fallback-eligible rather than exact user selections.

## Solution Assessment

No. The precedence direction and child-session override surface are plausible, but the PR should preserve explicit model source semantics and rebase onto current main so it does not reintroduce or duplicate already-shipped runtime persistence behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve explicit spawn model source:** `src/agents/subagent-spawn.ts:243`
  - body: `buildDirectChildSessionPatch` records every model patch as `modelOverrideSource: "auto"`, but explicit `sessions_spawn.model` values also flow through this helper. Auto-sourced overrides keep using configured fallbacks, while user-selected models are exact, so an unavailable explicit child model can silently fall through to another configured model instead of failing on the requested one. Carry the source from the spawn plan and mark explicit spawn models as `user`.
