## Summary

Keep open for maintainer review. The remaining precedence fix is not implemented on current main, but this PR has a blocking correctness issue: it records every spawned child model override as `auto`, so an explicit `sessions_spawn.model` can incorrectly use configured fallbacks instead of staying an exact user-selected model.

## What This Changes

The PR reorders native and isolated-cron subagent model precedence, persists selected model fields on child sessions, updates related tests, and pins the Yuanbao official external channel catalog npm source.

## Best Possible Solution

Revise toward one focused implementation: make the intended precedence explicit as per-agent `subagents.model`, then global `agents.defaults.subagents.model`, then agent primary model; persist child-session `providerOverride`/`modelOverride` while preserving `user` versus `auto` source; and keep unrelated catalog metadata out of the agent PR unless maintainers intentionally bundle it.

## Reproduction Assessment

Yes. The code-level reproduction is high confidence: call `sessions_spawn` with an explicit `model` while defaults include fallbacks, and the PR head stores that explicit model with `modelOverrideSource: "auto"`, which routes it through fallback behavior instead of exact user-selection behavior.

## Solution Assessment

No. The precedence direction is plausible and the child-session override seam is the right place to apply the selected model, but the patch should preserve explicit spawn models as `user` source and avoid mixing already-on-main catalog metadata into this agent fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve explicit spawn model source:** `src/agents/subagent-spawn.ts:243`
  - body: `buildDirectChildSessionPatch` now records every model patch as `modelOverrideSource: "auto"`, but explicit `sessions_spawn.model` values also flow through this helper. Auto-sourced overrides keep using configured fallbacks, while user-selected models are supposed to be exact, so an explicit child model can silently fall through to another configured model instead of failing on the requested one. Preserve the source from the spawn plan and mark explicit spawn models as `user`.
  - confidence: 0.91
