## Summary

Current main still does not implement the requested fast-degrade behavior for MCP/plugin failures. Bundled MCP materialization remains on the embedded reply path, only connection setup is time-bounded, catalog listing and tool execution still await SDK calls directly, catalog failures still reject rather than returning degraded empty capability, and key modifying plugin hooks on the reply path still await handlers without per-hook timeout/circuit behavior. The prior Codex review and the narrower Chrome MCP diagnostic item #65560 do not supersede this broader resilience request.

## Best Possible Solution

Keep this issue open and implement a core resilience layer around external MCP/plugin work: configurable short budgets for MCP catalog listing and tool calls, repeated-failure cooldown/circuit state, degraded empty-tool materialization for optional unhealthy MCP servers, status/log diagnostics that distinguish channel health from external capability health, and regression tests for slow listTools, slow callTool, repeated failure cooldown, and blocking plugin hook phases.
