## Summary

Keep this issue open. The report is concrete and recent, and current main has promising mitigations for WebChat history display and a pre-created session-file persistence quirk, but I did not find enough issue-linked proof or an exact regression test showing the reported persisted JSONL message-loss window is fully fixed. The related #58088 thread is still open and covers a different session-pointer eviction path, so it is not a clean duplicate/supersession.

## Best Possible Solution

Keep this issue open and have maintainers either add a focused regression test for WebChat sessions with pre-created header-only transcript files and reconnect/error paths, or explicitly link it to the fix/release that covers the reported JSONL data loss. If #58088 becomes the canonical remaining session-loss tracker, cross-link that and close this only after the WebChat-specific persistence angle is accounted for.
