## Summary

Keep open. Current main still does not expose the requested `openclaw/plugin-sdk/backup` subpath, and the PR is a focused additive SDK proposal, but it is not merge-ready because it changes a public SDK surface without the required API-baseline/docs follow-through and leaves a real plugin data-boundary question unanswered.

## What This Changes

The PR adds a new `openclaw/plugin-sdk/backup` subpath that re-exports backup plan helpers and types, then wires that subpath into `package.json` exports and the plugin SDK entrypoint list.

## Best Possible Solution

If maintainers want backup planning to become a supported plugin API, land a rebased narrow SDK subpath with package exports, entrypoint metadata, SDK subpath docs, API-baseline hash, export checks, and a documented data-boundary decision; otherwise keep backup archive creation as the only public helper and leave raw planning internals private.
