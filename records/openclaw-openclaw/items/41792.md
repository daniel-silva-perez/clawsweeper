## Summary

Keep open. Current main still lacks the requested `openclaw/plugin-sdk/backup` subpath, so the PR is not obsolete, but the proposed public SDK surface should not merge as-is because it exposes backup plan path inventory without documenting the supported contract or data boundary. The older build-output concern appears superseded by current main's generated entrypoint wiring from `scripts/lib/plugin-sdk-entrypoints.json`.

## What This Changes

The PR adds a new `openclaw/plugin-sdk/backup` subpath by creating `src/plugin-sdk/backup.ts`, adding a package export, and registering `backup` in SDK entrypoint metadata.

## Best Possible Solution

If maintainers want backup planning to become a supported plugin API, land a rebased narrow SDK subpath with docs that define the contract and local-data boundary, aligned package/entrypoint/API metadata, and SDK export validation; otherwise keep `createBackupArchive` as the public helper and leave raw planning internals private.
