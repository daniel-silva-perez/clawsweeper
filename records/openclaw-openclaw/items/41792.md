## Summary

Keep open. Current main still lacks the requested backup SDK subpath, and the PR remains a focused implementation candidate, but it should not merge as-is because it promotes backup path-planning data into a public SDK contract without the matching documentation, API-baseline alignment, and explicit trust-boundary review.

## What This Changes

The PR adds an `openclaw/plugin-sdk/backup` subpath that re-exports backup archive helpers, backup plan resolution/types, and backup create option/result types, then registers the subpath in package exports and SDK entrypoint metadata.

## Best Possible Solution

If maintainers want backup planning to become supported plugin API, land a rebased narrow SDK subpath with docs that define intended consumers, absolute-path exposure, trust assumptions, aligned SDK metadata/API-baseline updates, and validation. Otherwise keep lower-level backup planning private and continue exposing only the existing archive creation helper.
