## Summary

Keep open. Current main still lacks the requested backup SDK subpath, and the PR is directionally useful, but it promotes local backup path-planning internals into a public plugin contract without the docs, API metadata/baseline, and explicit local-data boundary expected for SDK expansion.

## What This Changes

The PR adds an `openclaw/plugin-sdk/backup` subpath that re-exports backup archive helpers, backup plan resolution/types, and backup create option/result types, then registers the subpath in package exports and SDK entrypoint metadata.

## Best Possible Solution

Land a rebased version only if maintainers want backup planning as a supported plugin API: keep the subpath narrow, document the local-data/trusted-plugin boundary, add SDK docs metadata and subpath docs, regenerate the API baseline, and verify package export/build alignment.

## Reproduction Assessment

Not applicable as a runtime reproduction case; this is an additive SDK PR. The static review path is high-confidence: current main lacks the subpath, the PR adds it, and the missing docs/metadata boundary is visible from the current SDK rules and metadata files.

## Solution Assessment

No. The proposed subpath is a plausible narrow shape, but the best maintainable solution needs explicit API/security-boundary documentation and generated SDK metadata/baseline updates before third-party plugins rely on it.
