## Summary

Keep open. Current main still has no supported way for `openclaw mcp serve` to request a narrower or explicit operator scope set: the CLI has no scope option, the MCP serve options do not carry scopes, and the bridge still constructs the Gateway client with the fixed read/write/approvals scope bundle. The linked implementation PR is closed unmerged, so no merged fix currently resolves the issue.

## What This Changes

Add a supported way for `openclaw mcp serve` to request narrower or explicit operator scopes, such as repeatable `--scope`, modes, config, or environment support.

## Best Possible Solution

Implement validated repeatable `openclaw mcp serve --scope <operator.scope>` plumbing through the CLI, `OpenClawMcpServeOptions`, `OpenClawChannelBridge`, and `GatewayClient`, preserving the current default scope bundle when omitted. Update MCP docs and focused CLI/server tests; keep per-session tool policy work and scope-filtered tool listing as separate follow-ups unless maintainers explicitly expand the scope.

## Reproduction Assessment

Yes. The source-only reproduction is high confidence: current main exposes no `--scope` option, the serve options cannot carry scopes, and the bridge hardcodes the fixed scope array before constructing `GatewayClient`.

## Solution Assessment

Yes. A validated repeatable `--scope` flag is the narrowest maintainable first fix because the Gateway client already accepts caller-provided scopes and the repo already has an operator-scope validation seam; config/env modes can be layered later if maintainers want them.
