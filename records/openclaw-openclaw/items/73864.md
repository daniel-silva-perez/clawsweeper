## Summary

Keep open. Current main still has the exact feature gap described: `openclaw mcp serve` has no scope or mode selector, its server options do not carry requested scopes, and `OpenClawChannelBridge` still connects to the Gateway with a fixed `[operator.read, operator.write, operator.approvals]` scope set. The broader related #49183 is about per-session tool policy and does not supersede this narrower MCP bridge pairing/request-scope feature.

## Best Possible Solution

Keep the issue open and implement a small, explicit MCP bridge scope-selection surface in core: add repeatable `openclaw mcp serve --scope <scope>` and/or named modes such as `read`, `write`, and `approvals`; thread that through `OpenClawMcpServeOptions` into `OpenClawChannelBridge` and `GatewayClient`; validate against `operator.read`, `operator.write`, and `operator.approvals`; add CLI/unit coverage for requested connect scopes plus docs at `docs/cli/mcp.md`. Treat optional `tools/list` filtering as a follow-up only if the chosen MCP UX needs it.

## Risks / Open Questions

- This is a security-sensitive least-privilege feature. An implementation should validate requested scopes against known operator scope constants and fail closed for unknown or empty effective modes.
- Adding a `--scope` flag without updating pairing/upgrade UX, docs, and tests could make requested vs approved device scopes confusing for users running profile-isolated bridge identities.
- If `tools/list` remains static, docs or tool-call error behavior should make invocation-time authorization clear so clients do not assume listed tools are callable under narrower scopes.
