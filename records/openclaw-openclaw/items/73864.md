## Summary

Keep open. Current main still has the reported MCP bridge least-privilege gap: `openclaw mcp serve` exposes no scope or mode selector, its serve options do not carry requested scopes, and `OpenClawChannelBridge` still constructs the Gateway client with the fixed `[operator.read, operator.write, operator.approvals]` scope set. The related #49183 is broader agent tool-policy work and does not supersede this narrower MCP connect-scope request.

## Best Possible Solution

Keep this issue open and implement a focused MCP bridge scope-selection surface in core after maintainer UX/security review. The likely path is to add repeatable `openclaw mcp serve --scope <scope>` and/or named modes such as read, write, and approvals; thread the validated scope list through `OpenClawMcpServeOptions` into `OpenClawChannelBridge` and `GatewayClient`; document the behavior in `docs/cli/mcp.md`; and add CLI/unit coverage proving the requested connect scopes. Treat `tools/list` filtering as a follow-up unless maintainers decide the initial UX requires it.
