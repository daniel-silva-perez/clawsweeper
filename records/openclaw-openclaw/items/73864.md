## Summary

Keep open. Current main still lacks a configurable operator-scope request path for `openclaw mcp serve`: the CLI has no `--scope` or mode option, the serve options cannot carry scopes, and the MCP bridge still constructs its `GatewayClient` with the fixed read/write/approvals scope set. The related implementation PR #74030 was closed unmerged, so it does not supersede this issue.

## What This Changes

Add a supported way for `openclaw mcp serve` to request narrower operator scopes, such as repeatable `--scope` flags, config/env support, or documented read/write/approvals modes.

## Best Possible Solution

Add validated repeatable `openclaw mcp serve --scope <operator.scope>` support, thread the resulting `OperatorScope[]` through `OpenClawMcpServeOptions` and `OpenClawChannelBridge` into `GatewayClient`, preserve the current read/write/approvals default when omitted, and document the static tool-list versus invocation-time authorization behavior.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: current main has no CLI or serve-option scope input, and the bridge hardcodes `[operator.read, operator.write, operator.approvals]` when constructing the gateway client.

## Solution Assessment

Yes. The narrowest maintainable first fix is validated repeatable `--scope` plumbing using the existing `OperatorScope`/`isOperatorScope` seam; tool-list filtering can stay out of scope unless maintainers want that UX in the same change.
