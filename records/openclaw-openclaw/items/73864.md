## Summary

Keep open. Current main still has the reported MCP bridge least-privilege gap: `openclaw mcp serve` has no scope or privilege mode option, its serve options cannot carry requested scopes, and `OpenClawChannelBridge` still constructs `GatewayClient` with the fixed `[operator.read, operator.write, operator.approvals]` request. The related #49183 is broader agent tool-policy work and does not supersede this narrower MCP connect-scope request.

## Best Possible Solution

Keep this issue open and implement a focused core MCP bridge scope-selection surface after maintainer UX/security review. The likely path is a validated repeatable `openclaw mcp serve --scope <scope>` and/or named modes such as `read`, `write`, and `approvals`; thread the selected scopes through `OpenClawMcpServeOptions` into `OpenClawChannelBridge` and `GatewayClient`; document the behavior in `docs/cli/mcp.md`; and add MCP/CLI coverage proving the requested connect scopes. Treat `tools/list` filtering as a follow-up unless maintainers decide initial UX requires it.
