## Summary

Keep open. Current main still does not include a configurable channel silence watchdog keyed to operator-visible output, and the existing typing, preview streaming, tool-progress, run-state, and outbound hook mechanisms do not provide the requested threshold/reset/repeat/max-beacon contract. The broader same-author #47591 remains open and overlaps this request, so this should be coordinated rather than closed in isolation.

## What This Changes

Requested a configurable gateway-level silence watchdog that sends operator-visible status beacons during active work after a visible-output silence threshold.

## Best Possible Solution

Define an opt-in, default-off generic runtime watchdog with documented operator-visible delivery timestamps, reset/cancel/repeat semantics, per-channel config/schema/docs, and focused coverage for outbound delivery, preview/block streaming, active run lifecycle, max-beacon behavior, and interaction with #47591’s delivery-confirmation design.

## Reproduction Assessment

Not applicable as a feature request rather than a bug report. The high-confidence check is source/docs inspection on current main, which shows the requested watchdog contract is absent while adjacent liveness mechanisms exist.

## Solution Assessment

Unclear until maintainer approval. The maintainable direction is a shared channel/runtime seam tied to real visible delivery, not per-channel ad hoc messages or model prompt rules.
