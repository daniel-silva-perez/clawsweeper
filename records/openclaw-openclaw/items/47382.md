## Summary

Keep open. Current main still does not implement a configurable gateway/channel silence watchdog keyed to operator-visible output; the adjacent streaming, tool-progress, outbound hook, and run-state mechanisms provide useful signals but not the requested threshold/reset/repeat/max-beacon contract. The broader same-author #47591 remains open and overlaps this feature, so this should be coordinated rather than closed in isolation.

## What This Changes

Add a configurable gateway-level channel silence watchdog that sends repeatable generic status beacons during active work when no operator-visible output has been delivered for a configured interval.

## Best Possible Solution

Define a generic, documented runtime/channel watchdog seam tied to real operator-visible delivery timestamps, with per-channel config/schema/docs, reset/cancel/repeat/max-beacon semantics, focused coverage for streaming and outbound delivery paths, and coordination with #47591’s response-delivery confirmation design.

## Reproduction Assessment

Not applicable as a feature request rather than a bug report. The high-confidence check is source and docs inspection on current main, which shows adjacent liveness mechanisms exist but the requested watchdog contract is absent.

## Solution Assessment

Unclear until maintainer approval. The maintainable direction is a shared channel/runtime seam keyed to actual visible delivery, not per-channel ad hoc messages or model prompt rules.
