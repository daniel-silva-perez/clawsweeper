## Summary

Keep this PR open. Current main still has the Telegram replyToMode/tool-progress preview mismatch and no doctor/docs warning, and this PR is the active implementation candidate for the linked bug. The approach is narrow and security-sensitive surfaces look clean, but the proposed scanner has one concrete false-negative edge case before merge.

## What This Changes

This PR adds a Telegram `openclaw doctor` warning plus Telegram/streaming docs and changelog notes for the `replyToMode` versus `streaming.preview.toolProgress` conflict, with regression tests and a small active-memory test cleanup.

## Best Possible Solution

Keep the PR active, patch the scanner to evaluate the implicit default/top-level Telegram config when no configured account records are present, then land the doctor warning and docs note so the linked bug closes through the implementation candidate.

## Reproduction Assessment

Yes. The current-main code path is reproducible by configuring Telegram with `replyToMode: "first"`, active preview streaming, and default tool progress, then sending an inbound text message that triggers a tool; code inspection shows quote-reply prevents `answerLane.stream`, and the reporter provided matching live traces.

## Solution Assessment

Mostly yes, but not fully yet. A doctor warning plus docs note is the narrowest maintainable fix for the intentional runtime mutual exclusion; the scanner should first cover the implicit-default `accounts: {}` path or derive effective Telegram account ids from the same runtime selection logic.

- **[P2] Scan the default account when accounts is empty:** `src/commands/doctor/shared/telegram-replyto-streaming.ts:74-87`
  - body: `hasRecord(accounts)` is true for `accounts: {}`, so the helper enters this branch and returns without evaluating the top-level Telegram config. Runtime account selection still creates the implicit `default` account when no account ids are configured, and `mergeTelegramAccountConfig` uses the top-level settings for that account, so a config like `replyToMode: "first", accounts: {}` still has the silent tool-progress conflict but doctor emits no warning. Fall back to scanning the top-level/default config when there are no record account entries.
