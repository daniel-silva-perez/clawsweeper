## Summary

Keep open. Current main still reaches the generated `models.json` preparation path during gateway startup, and merge mode still refreshes matching provider model arrays from config/catalog data while preserving only selected credential and endpoint fields. The remaining question is the intended product contract for manual per-agent model-list pruning, not whether the reported source path exists.

## What This Changes

The issue requests that manual per-agent `models.json` edits, especially pruning a provider's model list, survive gateway restarts instead of regenerated defaults or catalog entries restoring removed models.

## Best Possible Solution

Maintainers should make the `models.json` ownership contract explicit. If manual pruning is supported, add a durable deletion/tombstone or explicit replace mechanism and a gateway-restart regression. If the file remains generated catalog state, route persistent pruning through `models.providers`, `agents.defaults.models`, or a documented CLI/config flow and keep docs, setup, and doctor guidance aligned.

## Reproduction Assessment

Yes. The issue's restart steps map to a deterministic source path: gateway startup calls `ensureOpenClawModelsJson`, the planner compares generated contents to the existing file, and merge mode can append implicit catalog models that were manually removed.

## Solution Assessment

Unclear. Preserving direct edits may be the right user-facing fix, but the current docs and code also support treating `models.json` as regenerated state; maintainer product/API judgment is needed before choosing tombstones, replace mode, or documentation-only behavior.
