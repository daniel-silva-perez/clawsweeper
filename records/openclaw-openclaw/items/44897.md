## Summary

Keep this PR open. The reported Control UI hang is still a real gap on current main because `loadChatHistory` still awaits `chat.history` without a bounded timeout, but this branch should not merge as-is: its timeout path abandons the underlying browser RPC and its late-success path bypasses current main's history filtering and optimistic-tail reconciliation.

## What This Changes

This PR changes Control UI chat history loading to race the initial `chat.history` request against a 10-second timeout, clear the timeout timer, late-apply guarded successful responses, and bump a freshness token when sending chat messages.

## Best Possible Solution

Land a current-main-native Control UI history timeout that exits the empty loading state with a targeted error, explicitly cancels or removes the timed-out browser RPC, safely ignores or reconciles late responses, and preserves startup retry, per-state request versions, session guards, optimistic-tail preservation, active stream behavior, and hidden-history-message filtering.

## Reproduction Assessment

Yes. A high-confidence focused reproduction is to mock `state.client.request("chat.history")` with a never-settling promise; current main leaves `chatLoading` true, and the empty chat renderer shows the loading skeleton for that state.

## Solution Assessment

No. The `Promise.race` direction is useful, but the current PR is not the best fix because it does not cancel the underlying pending RPC and does not reuse the current history reconciliation path for late successes.
