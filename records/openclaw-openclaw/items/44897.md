## Summary

Keep open. The underlying Control UI hang is still real on current main, but this PR is stale against current chat-history contracts and still has blocking correctness issues around pending RPC cancellation and late-response reconciliation.

## What This Changes

This PR changes Control UI chat history loading to race the initial `chat.history` request against a 10-second timeout, clear the timeout timer, late-apply resolved history, and bump a freshness token when sending messages.

## Best Possible Solution

Land a current-main-native timeout path that exits the empty chat loading state with a targeted error, removes or cancels the timed-out browser RPC, and preserves the existing startup retry, per-state request version/session guards, hidden-history filtering, optimistic-tail preservation, active stream handling, and missing-scope behavior.

## Reproduction Assessment

Yes. A high-confidence focused reproduction is to mock `state.client.request("chat.history")` with a never-settling promise; current main keeps `loadChatHistory` pending with `chatLoading` true, and the empty chat renderer shows the loading skeleton for that state.

## Solution Assessment

No. A bounded timeout is the right direction, but this PR is not the best fix because it leaves the underlying browser RPC pending and applies late successful responses through an obsolete, narrower reconciliation path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

- **[P2] Cancel timed-out history RPCs:** `ui/src/ui/controllers/chat.ts:121-123`
  - body: When the timeout branch wins, the raced `requestPromise` still remains registered in `GatewayBrowserClient.pending`; that client removes entries only on a response or socket shutdown, so retries after a stalled `chat.history` can accumulate unresolved pending RPCs. The timeout path needs cancellation or explicit pending removal.
  - confidence: 0.86

  - confidence: 0.82
