## Summary

Keep this PR open, but it needs changes before merge. The reported hang is still credible on current main, while the PR head leaves timed-out browser RPCs registered and is stale against newer Control UI history-load contracts.

## What This Changes

This PR changes Control UI chat history loading to race the initial `chat.history` request against a 10-second timeout, clear the timeout timer, late-apply guarded successful responses, and bump a module-level freshness token when sending chat messages.

## Best Possible Solution

Land a current-main-native Control UI history timeout that exits the empty loading state with a targeted error, explicitly cancels or removes the timed-out browser RPC, safely ignores or handles late responses, and preserves current startup retry, per-state request-version, session, optimistic-tail, stream, and hidden-history-message behavior.

## Reproduction Assessment

Yes. A high-confidence focused reproduction is to mock `state.client.request("chat.history")` with a never-settling promise: current main leaves `chatLoading` true, and the empty chat renderer shows the loading skeleton for that state.

## Solution Assessment

No. The PR’s fixed `Promise.race` timeout is directionally useful, but it is not the best solution because it does not cancel the underlying `GatewayBrowserClient.pending` entry and does not adapt cleanly to current main’s history reconciliation contracts.
