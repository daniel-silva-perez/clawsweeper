## Summary

Keep open. Current main still has the reported user-visible hang path because `loadChatHistory` awaits `chat.history` without a timeout, but this PR is not safe to merge as-is because it does not cancel or remove the underlying pending browser RPC and its late-success path bypasses current main's history filtering and optimistic-tail reconciliation.

## What This Changes

This PR changes `ui/src/ui/controllers/chat.ts` to race the initial `chat.history` request against a 10-second timeout, clear the timeout timer, late-apply a delayed success, and bump a freshness token when sending chat messages.

## Best Possible Solution

Land a current-main-native timeout path that exits the empty chat loading state with a targeted error, cancels or explicitly removes the timed-out browser RPC, and uses the existing history reconciliation for both normal and safe late successes while preserving startup retry, request-version/session guards, hidden-history filtering, optimistic-tail preservation, active-stream handling, and missing-scope behavior.

## Reproduction Assessment

Yes. A high-confidence focused reproduction is to mock `state.client.request("chat.history")` with a never-settling promise; current main keeps `loadChatHistory` pending with `chatLoading` true, and the empty chat renderer shows the loading skeleton for that state.

## Solution Assessment

No. A bounded timeout is the right direction, but this PR is not the best fix because it leaves the underlying browser RPC pending and applies late successful responses through an obsolete, narrower reconciliation path.
