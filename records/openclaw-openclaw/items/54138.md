## Summary

Keep open. Current main still passes heartbeat final reply payload text through unchanged in `buildReplyPayloads`, and the final delivery path still passes `isHeartbeat` into that builder. The shared bracket-block stripper exists in assistant-visible text sanitization, but it is not applied on this heartbeat delivery path and its parser does not cover the reported same-line or non-JSON `[TOOL_CALL]` / `[TOOL_RESULT]` examples. The issue is also paired with open closing-syntax PR #54165, so it should stay open until that PR or an equivalent fix lands.

## What This Changes

Heartbeat replies should strip raw uppercase `[TOOL_CALL]...[/TOOL_CALL]` and `[TOOL_RESULT]...[/TOOL_RESULT]` bracket-format blocks before user-facing channel delivery.

## Best Possible Solution

Keep this issue open and review/land PR #54165, or land an equivalent focused fix in `src/auto-reply/reply/agent-runner-payloads.ts` that strips uppercase bracket tool-call/result blocks from heartbeat reply text before final channel delivery, with regression coverage in `src/auto-reply/reply/agent-runner-payloads.test.ts`. Track the broader interactive/XML leak separately if it still reproduces after the heartbeat fix.
