## Summary

Keep open. Current main still routes final heartbeat replies through `buildReplyPayloads` with `isHeartbeat` set, and that function still uses heartbeat payloads unchanged before channel delivery. The newer shared bracket sanitizer only covers standalone newline-delimited JSON-shaped bracket tool blocks, not the reported same-line or non-JSON `[TOOL_CALL]...[/TOOL_CALL]` / `[TOOL_RESULT]...[/TOOL_RESULT]` variants, and the focused payload-builder tests still have no heartbeat bracket regression case. The issue is also paired with open closing-syntax PR #54165, so it should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue requests that heartbeat-mode reply payloads strip uppercase `[TOOL_CALL]...[/TOOL_CALL]` and `[TOOL_RESULT]...[/TOOL_RESULT]` bracket-format blocks before Telegram/Discord or other user-facing delivery.

## Best Possible Solution

Keep this issue open and review/land PR #54165, or land an equivalent focused fix in `src/auto-reply/reply/agent-runner-payloads.ts` that strips uppercase `[TOOL_CALL]...[/TOOL_CALL]` and `[TOOL_RESULT]...[/TOOL_RESULT]` blocks from heartbeat reply text before final channel delivery, with regression coverage in `src/auto-reply/reply/agent-runner-payloads.test.ts`. The broader interactive/XML leak should be tracked separately if still reproducible.
