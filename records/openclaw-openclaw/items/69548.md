## Summary

Keep open. Current main does not implement the requested `dreaming.staggerMs` memory-core setting, so this PR is still the active implementation candidate. The latest branch still has a concrete P2 behavior defect: removing `dreaming.staggerMs` will not clear a previously persisted explicit stagger because the branch only compares stagger drift when the desired value is defined, while cron patching preserves omitted `staggerMs` values.

## What This Changes

The PR adds `dreaming.staggerMs` to memory-core manifest/docs/types, parses it in the memory host SDK, and passes/logs it on the managed dreaming cron schedule.

## Best Possible Solution

Expose `dreaming.staggerMs` as a documented memory-core setting whose create, update, explicit `0`, unset, and removal semantics are predictable, with focused tests covering config parsing, schedule creation, schedule updates, and clearing or documenting a previously configured stagger.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Clear stale stagger when config is removed:** `extensions/memory-core/src/dreaming.ts:294`
  - body: When an operator removes `dreaming.staggerMs`, `desiredStaggerMs` becomes `undefined`, so this condition skips the schedule update even if the stored managed cron job still has an old explicit stagger. `applyJobPatch` preserves omitted cron `staggerMs` values, so the job keeps using stale jitter instead of returning to the unset/default behavior described by the new docs.
  - confidence: 0.89
