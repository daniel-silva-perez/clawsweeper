## Summary

Keep this PR open. Current main still does not expose a memory-core `dreaming.staggerMs` setting, so the feature work is not obsolete, but the latest patch still has a concrete removal-semantics bug: removing the config field can leave an old explicit stagger persisted while the docs say unset returns to the cron default.

## What This Changes

The PR adds optional `dreaming.staggerMs` config/schema/docs plumbing, parses it in the memory host SDK, and passes/logs it on the managed memory-core dreaming cron schedule.

## Best Possible Solution

Ship the feature with a complete lifecycle contract for create, update, explicit `0`, unset, and removal. The implementation should either reset a previously explicit managed cron stagger when the config field is removed, or intentionally document and test preserve-on-removal semantics with `0` as the supported clear path.

## Reproduction Assessment

Yes. A focused unit reproduction is an existing managed dreaming cron job with `schedule.staggerMs` set, followed by desired config without `dreaming.staggerMs`; the PR's patch builder emits no schedule patch, and current cron patching preserves omitted stagger fields.

## Solution Assessment

No. Passing `staggerMs` through the memory config is the right direction, but the current patch is not the best complete solution until removal behavior matches the docs or is explicitly documented and tested as a deliberate limitation.
