## Summary

Keep this PR open. Current main still does not expose `dreaming.staggerMs`, and this PR is the active implementation for the related feature request, but the latest patch still has a concrete removal-semantics defect: removing the config field leaves a previously persisted explicit stagger in place while the new docs say unset should return to the cron default.

## What This Changes

The PR adds optional `dreaming.staggerMs` config, schema, docs, and host SDK plumbing so memory-core’s managed dreaming cron job can receive a cron stagger window.

## Best Possible Solution

Ship the feature with a complete lifecycle contract for create, update, explicit `0`, unset, and removal. The implementation should either clear/reset the persisted managed cron schedule when the config field is removed, or intentionally document and test preserve-on-removal semantics with `0` as the supported clear path.

## Reproduction Assessment

Yes. A focused unit reproduction is an existing managed dreaming cron job with `schedule.staggerMs` set, followed by desired config without `dreaming.staggerMs`; the PR's patch builder emits no schedule patch, and current cron patching preserves omitted stagger fields.

## Solution Assessment

No. Passing `staggerMs` through the memory config is the right direction, but the current patch is not the best complete solution until removal behavior matches the docs or is explicitly documented and tested as a deliberate limitation.

## Review Findings

Overall correctness: patch is incorrect
