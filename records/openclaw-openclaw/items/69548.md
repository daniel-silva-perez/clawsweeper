## Summary

Keep this PR open. Current main still does not expose `dreaming.staggerMs` for memory-core dreaming schedules, and this PR is the active implementation for the same request tracked in #69531. The latest patch addresses the earlier `0` and blank-string parsing concerns, but it still leaves a predictable removal-semantics defect: removing `dreaming.staggerMs` from config does not clear a previously stored explicit stagger because the PR skips the schedule patch when the desired value is unset while the cron service preserves omitted cron stagger fields.

## What This Changes

The PR adds optional `dreaming.staggerMs` config/schema/docs plumbing and passes the value through memory host config into the managed memory-core dreaming cron schedule and verbose logging.

## Best Possible Solution

Ship the feature with a complete lifecycle contract for create, update, explicit `0`, unset, and removal. The implementation should either clear/reset the persisted managed cron schedule when the config field is removed, or intentionally document and test preserve-on-removal semantics with `0` as the supported clear path.
