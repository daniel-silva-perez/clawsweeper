## Summary

Keep this PR open. Current main still does not expose `dreaming.staggerMs`, and the PR is a plausible implementation of the related feature request, but the latest diff still has a P2 behavior defect: removing the setting will not reset an existing managed dreaming cron job back to unset/default stagger semantics because the PR skips the comparison when the desired value is undefined while the cron patch path preserves omitted stagger values.

## What This Changes

The PR adds `dreaming.staggerMs` to the memory-core manifest/docs/types, parses it in the memory host SDK, and passes/logs it on the managed dreaming cron schedule.

## Best Possible Solution

Expose `dreaming.staggerMs` with predictable create, update, explicit `0`, unset, and removal semantics. The patch should either reset a previously explicit stagger to the appropriate cron default when the config is removed or intentionally document and test that removal preserves the prior value and explicit `0` is the supported clear path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Clear stale stagger when config is removed:** `extensions/memory-core/src/dreaming.ts:294`
  - body: When an operator removes `dreaming.staggerMs`, `desiredStaggerMs` is `undefined`, so this condition suppresses a schedule patch even if the existing managed job still stores a prior explicit stagger. Current cron patching preserves omitted cron `staggerMs`, so the old jitter remains instead of returning to the unset/default behavior documented by this PR.
