## Summary

Keep this PR open. Current main still does not expose or pass `dreaming.staggerMs` through memory-core, while the generic cron layer already supports `CronSchedule.staggerMs`; this PR remains the focused implementation path for closed feature request #69531. The branch has addressed the earlier `staggerMs: 0` and blank-string review findings, but it still needs maintainer review for clear/unset semantics, focused tests, public docs, and a rebase because GitHub reports the branch as dirty.

## What This Changes

The PR adds an optional `dreaming.staggerMs` memory-core config field and threads it through host SDK dreaming config, managed dreaming cron schedule creation/reconciliation, and verbose logging.

## Best Possible Solution

Keep this PR open as the canonical implementation path for #69531. Before merge, rebase it, add focused host-SDK and memory-core cron reconciliation tests, update public dreaming or memory config docs, and have a maintainer settle whether removing `dreaming.staggerMs` should clear a previous explicit value or whether `staggerMs: 0` is the documented clear/exact-timing behavior.
