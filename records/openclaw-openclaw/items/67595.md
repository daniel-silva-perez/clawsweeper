## Summary

Keep open. Current main has evidence that the LaunchAgent/service-token embedding half was fixed, and a maintainer explicitly called that out as a partial fix, but the separate `openclaw channels list` provider-usage failure still exists in the current CLI path.

## What This Changes

The issue reports two bugs: forced gateway install/repair keeps embedding `OPENCLAW_GATEWAY_TOKEN` in managed service metadata, and `openclaw channels list` emits or fails with `TypeError: fetch failed` when auth providers exist but no chat channels are configured.

## Best Possible Solution

Keep the service-install half as fixed and land a focused CLI/provider-usage change so `openclaw channels list` always returns the configured chat/auth listing in both human and JSON modes, treating usage enrichment failures as optional with regression coverage for empty chat channels plus auth profiles.
