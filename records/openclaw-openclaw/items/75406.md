## Summary

Keep open. Current main still matches the reported hot surfaces: the first embedded run still performs a broad runtime plugin load before model submission, bundled runtime dependency staging still has synchronous filesystem and install paths, and the pre-prompt token estimate is synchronous. Related main-only latency fixes exist, but there is not enough evidence that they fully solve this Docker first-reply event-loop starvation report.

## What This Changes

The issue reports first post-boot LLM calls in v2026.4.29 blocking the Node event loop for 20-79 seconds around runtime plugin loading or system-prompt assembly, causing Telegram and WhatsApp channel drops.

## Best Possible Solution

Keep this open until current main has a focused first post-boot Docker/Linux profile that separates runtime plugin resolution, bundled dependency staging, provider/catalog hooks, channel capability lookup, prompt construction, and token estimation. The durable end state should remove broad synchronous plugin/dependency work from the reply critical path or prewarm it outside channel timeout windows, with regression coverage for the selected boundary.

## Reproduction Assessment

Unclear. The reporter provides concrete v2026.4.29 Docker timings and the current code still has matching synchronous paths, but this review did not find a current-main profile, fixture, or test that reproduces the 20-79 second event-loop stall with high confidence.

## Solution Assessment

Unclear. Eager or worker-thread tokenizer initialization is not proven to be the right fix; the safer path is to profile the current runtime-plugins and system-prompt substeps, then scope or defer the confirmed blocking owner path.
