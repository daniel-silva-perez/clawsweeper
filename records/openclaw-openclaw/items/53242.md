## Summary

Keep open. Current main still truncates default chat.history text fields before the larger byte-budget checks, and the Control UI/WebChat path does not request a larger maxChars override. The linked same-author PR #53243 remains open and unmerged, so this issue should stay open until that PR or an equivalent gateway fix lands.

## What This Changes

The issue asks for chat.history to preserve long assistant text that fits the configured history byte budget instead of applying a small default character cap first.

## Best Possible Solution

Update the shared chat display projection default to raise or derive the text cap from the byte budget, keep the config/RPC override semantics, add a default-path regression test for long assistant text plus UTF-8-heavy truncation behavior, and update config help/docs if the shipped default changes.
