## Summary

Keep open. Current main still applies a small default character cap before the larger byte-budget checks, so the reported WebChat/Control UI truncation path remains valid. The issue is also paired with open, unmerged PR #53243 using closing syntax, so it should stay open until that PR or an equivalent gateway fix lands.

## What This Changes

The issue asks for `chat.history` to stop truncating long assistant text fields at a small default character cap when those messages still fit the per-message and overall history byte budgets.

## Best Possible Solution

Update the shared chat display projection default to raise or derive the text cap from the byte budget, preserve the existing config/RPC override semantics, add default-path regression coverage for long assistant text plus UTF-8-heavy truncation, and adjust config help/docs if the shipped default changes.
