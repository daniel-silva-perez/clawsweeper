## Summary

Keep open. Current main still inserts literal spaces into long non-copy-sensitive tokens, the copy-sensitive guard does not protect hyphen-only package names, and assistant-visible TUI content still flows through that sanitizer. The issue also has an open unmerged closing PR (#69340), so cleanup should wait for that PR to be reviewed, merged, replaced, or closed.

## What This Changes

The issue asks for TUI code-block and table rendering to stop injecting spurious spaces into long wrapped content so copied code remains valid.

## Best Possible Solution

Preserve copy-sensitive code and table content in the TUI without injecting literal display spaces, then add focused regression coverage for the reported hyphenated package-name case plus fenced code/table rendering before closing this issue.

## Reproduction Assessment

Yes. A code-level reproduction against current main shows the current sanitizer turns a 33-character hyphenated package token into a token with an inserted literal space, and the TUI assistant rendering path still uses that sanitizer.

## Solution Assessment

Unclear. PR #69340 is a plausible narrow fix for the package-name example, but the issue discussion also covers fenced code blocks and box-drawn tables, so the safest fix may need broader copy-sensitive/code-block preservation with tests.
