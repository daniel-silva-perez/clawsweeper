## Summary

Keep this issue open. Current main has partial local transcript-token estimation for preflight compaction and can backfill status/session rows from transcript usage metadata when a provider logged meaningful usage, but it still does not implement the requested shared local estimate for MiniMax-style missing or all-zero usage. `/status` can still collapse to stored/input-output totals, `/usage` and context notices intentionally hide stale context snapshots, and transcript fallback does not estimate from arbitrary stored messages when no usage snapshot exists.

## Best Possible Solution

Implement a shared estimated context snapshot path for sessions whose provider usage is missing or all-zero. Reuse `readSessionMessages` plus `estimateMessagesTokens` or the existing prompt estimator, carry a source/estimated flag through session store or Gateway session rows/events, render it clearly in `/status`, `/usage`, and context notices, and let compaction thresholds consume it without overwriting fresh provider snapshots. Add regression coverage for MiniMax-style all-zero usage and transcripts with messages but no assistant usage metadata.

## Risks / Open Questions

- A future fix must preserve provider-reported fresh usage as authoritative and keep heuristic estimates clearly marked with a distinct source/estimated flag.
- Raw transcript estimates can undercount provider tokenization, hidden prompts, tool schema overhead, cached prompt behavior, and provider wrappers, so compaction thresholds should retain conservative headroom.
- Large transcript parsing for display surfaces should be bounded or cached so `/status` and session lists remain fast.
