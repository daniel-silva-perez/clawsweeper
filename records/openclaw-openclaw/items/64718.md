## Summary

Keep this PR open. The branch addresses a real command-execution hardening gap that is not implemented on current main, but the proposed non-owner path still has a blocking fail-open case for partial operator exec defaults and overlaps with the broader dangerous-tool hardening in #64719.

## What This Changes

The branch passes `command.senderIsOwner` into auto-reply exec override resolution, ignores inline/session exec settings for non-owner senders, and adds unit tests for owner, non-owner, legacy, and explicit-default cases.

## Best Possible Solution

Make the non-owner auto-reply exec boundary fail closed by ignoring user-controlled inline/session exec settings and filling or requiring safe `security` and `ask` policy fields for any operator defaults, then coordinate that policy with #64719 before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Require non-owner exec defaults to fail closed:** `src/auto-reply/reply/get-reply-exec-overrides.ts:20`
  - body: The non-owner branch returns any `agentExecDefaults` object as-is. Because `security` and `ask` are optional, an agent config that only sets `host`, `node`, or another exec field still falls through to downstream host defaults of `security=full` and `ask=off`, preserving command execution for unauthenticated auto-reply senders. Merge in `deny`/`always` for missing fields or require an explicit safe policy before bypassing the deny default.
  - confidence: 0.9
