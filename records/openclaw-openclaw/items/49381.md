## Summary

Keep open. Current main has Feishu duplicate-final mitigations, including coalescing distinct streamed finals into one card until idle and exact-text final dedupe, but it still does not fully cover the reported failover coordination bug. The Feishu dispatcher still closes and resets the active streaming card on reply delivery error, generic final delivery still routes every non-reasoning final payload, non-streaming tests still preserve distinct finals, and the linked candidate fix #59771 remains open and unmerged.

## Best Possible Solution

Keep this issue open until #59771 or an equivalent fix lands. The fix should prevent a failed primary attempt from surfacing a user-visible final when a fallback attempt later succeeds, keep recovered Feishu streaming replies on one visible card where appropriate, and add regression coverage for the reported non-streaming failover path or explicitly split that remaining path into a narrower follow-up.
