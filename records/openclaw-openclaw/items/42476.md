## Summary

Keep this PR open. Current main still does not implement the intended media download anti-cache hardening: the active `/media/:id` handler only sets `X-Content-Type-Options: nosniff`, and the media tests do not assert `Cache-Control`, `Pragma`, or `Expires`. The PR remains a narrow, plausible security hardening change, but it needs to be ported from the older `attachMediaRoutes` shape to the current `createMediaRequestHandler` implementation.

## Best Possible Solution

Keep this PR open. Port the header setting into `createMediaRequestHandler` immediately after a matched `/media/:id` route and before validation/read/error branches, then expand media server tests to cover success, HEAD, expired, invalid, oversized, missing, and outside-workspace responses while preserving TTL expiry, cleanup timing, path validation, active-content download behavior, and file-serving semantics.
