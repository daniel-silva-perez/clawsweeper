## Summary

Keep this PR open. Current main still does not send explicit anti-cache headers for temporary `/media/:id` downloads; the active handler only sets `X-Content-Type-Options: nosniff`. The PR is a plausible security-hardening change, but it needs to be ported from the older route shape in the PR diff to the current `createMediaRequestHandler` implementation and should expand tests across the media response branches.

## Best Possible Solution

Keep this PR open. Port the anti-cache headers into `createMediaRequestHandler` after a matched `/media/:id` route and before all validation/read/error branches, then expand `src/media/server.test.ts` and `src/media/server.outside-workspace.test.ts` to assert `Cache-Control`, `Pragma`, and `Expires` across success, HEAD, expired, invalid, oversized, missing, and outside-workspace responses while preserving TTL expiry, cleanup timing, path validation, active-content download behavior, and HEAD semantics.
