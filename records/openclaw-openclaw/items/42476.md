## Summary

Keep this PR open. Current main still serves temporary `/media/:id` downloads through `createMediaRequestHandler` without the requested anti-cache headers; the handler only sets `X-Content-Type-Options: nosniff`. The PR is still a plausible security-hardening change, but its patch targets an older `attachMediaRoutes` shape, so it needs to be ported to the current handler and its tests expanded across the current media response branches.

## Best Possible Solution

Keep this PR open. Port the anti-cache headers into `createMediaRequestHandler` immediately after a matched `/media/:id` route and before validation/read/error branches, then expand `src/media/server.test.ts` and `src/media/server.outside-workspace.test.ts` to assert `Cache-Control: no-store, private, max-age=0, must-revalidate`, `Pragma: no-cache`, and `Expires: 0` across success, HEAD, expired, invalid, oversized, missing, and outside-workspace responses while preserving cleanup timing, TTL expiry, active-content download behavior, and HEAD semantics.
