## Summary

Keep open. Current main has partial direct-fallback hardening and bounded restart orphan recovery, but it still does not implement the requested durable completion spool/inbox: completion-mode delivery remains direct-first with conditional queue fallback, cleanup can still mark delivery failed after retry/expiry, restore/resume orphan handling can still prune missing-session runs, and the public docs still describe restart-time announce-back loss.

## What This Changes

Request durable subagent completion delivery so successful child results survive direct-announce timeouts, drain/restart gaps, and restore/orphan pruning.

## Best Possible Solution

Keep this issue as the planning thread and implement a core subagent lifecycle fix: persist a self-contained pending completion spool before cleanup can give up, make direct and queued delivery consume/update that spool, keep task delivery pending while the spool is pending, and retry or explicitly expire pending spooled completions during resume and orphan reconciliation.

## Reproduction Assessment

Yes. Current source and tests provide a high-confidence reproduction path: force completion-mode direct delivery to fail while the fallback queue returns none, or restore/resume an ended run with a missing child session, and current main records failure/prunes instead of preserving a pending durable completion spool.

## Solution Assessment

Not applicable as a patch review because this issue has no proposed diff. The durable spool direction is the right target, but the exact persisted schema, expiry, and idempotency behavior should be confirmed by maintainers before a repair PR changes core lifecycle state.
