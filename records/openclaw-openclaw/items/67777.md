## Summary

Keep open. Current main has partial subagent announce hardening and newer bounded orphan-recovery work, but it still lacks the requested durable completion spool/inbox: completion delivery remains direct-first with conditional queue fallback, cleanup can still mark delivery failed after retry/expiry, orphan reconciliation still prunes missing-session runs, and the public sub-agent docs still say pending announce-back work is lost on gateway restart.

## What This Changes

Request durable subagent completion delivery so successful child results survive direct-announce timeout, drain/restart, and restore/orphan-prune paths.

## Best Possible Solution

Keep this issue as the planning thread and implement a core subagent lifecycle fix: persist a self-contained pending completion spool before cleanup can give up, make direct and queued delivery consume/update that spool, keep task delivery pending while the spool is pending, and retry or explicitly expire pending spooled completions during resume and orphan reconciliation. Review #65037, #65000, and the newer bounded orphan-recovery behavior before choosing the final patch shape.

## Reproduction Assessment

Yes. Current tests and source provide a high-confidence reproduction path: force completion-mode direct delivery to fail while the fallback queue returns none, or remove the child session before steer-restart cleanup, and current main records failure/prunes before durable delivery instead of preserving a pending completion spool.

## Solution Assessment

Unclear as an implementation because this issue has no patch, but the proposed durable spool direction is the right target. The safest solution is a narrow persisted completion state integrated with existing registry/task delivery semantics, not another transient direct/queue retry layer.
