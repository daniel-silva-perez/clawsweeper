## Summary

Keep open. Current main still builds light dreaming from the configured lookback window and appends each successful Dream Diary narrative as a new timestamped entry; the existing dedupe fingerprint includes the timestamp, and the regression test preserves identical bodies at different times. The visible duplicate-output problem remains valid, but open PR #70332 already targets body-level diary dedupe and duplicate append skipping, so this should not be closed or queued as a duplicate repair job yet.

## What This Changes

The issue requests incremental or body-level deduplication for light-phase Dream Diary work summaries so unchanged work-summary blocks are not appended repeatedly across overnight cycles.

## Best Possible Solution

Land or rework a focused memory-core change that suppresses unchanged automatic Dream Diary output, with regression coverage for repeated light sweeps using unchanged input and for genuinely new snippets that must still append a new entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to seed daily or short-term recall material, run two light dreaming sweeps inside the same lookback window with deterministic narrative text, and observe that current `appendNarrativeEntry` can append the same body under different timestamps; the existing dedupe test also proves those different timestamps are preserved.

## Solution Assessment

Unclear. The issue's `lastEmittedAt` cutoff is a plausible narrow fix for light input, while PR #70332's body-level duplicate skip may be a broader and simpler diary-level guard; maintainers should choose the narrowest policy that removes unchanged automatic output without hiding legitimate new entries.
