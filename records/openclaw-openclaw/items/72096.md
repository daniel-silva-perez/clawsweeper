## Summary

Keep open. Current main still builds the light Dream Diary narrative from the configured lookback window and appends each successful narrative as a timestamped diary entry; the existing dedupe path includes the timestamp, so identical bodies at different cycle times are intentionally retained. Open PR #70332 directly targets body-level diary dedupe and duplicate append skipping, so this should stay open for that PR’s review rather than spawning a duplicate repair lane.

## What This Changes

The issue requests preventing light-phase Dream Diary work-summary content from repeating byte-for-byte across recurring overnight dreaming cycles, either through an incremental emitted cutoff or duplicate suppression.

## Best Possible Solution

Land or rework a targeted `memory-core` change that prevents unchanged automatic light Dream Diary material from being appended repeatedly, with tests covering repeated light sweeps inside one lookback window and genuinely new snippets that should still produce a new diary entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to run two light dreaming sweeps within the same lookback window with unchanged snippets and deterministic narrative output, then observe that current main can append the same body at different timestamps; I did not run it because this review is read-only.

## Solution Assessment

Unclear. The issue’s `lastEmittedAt` cutoff is a plausible input-level fix, while PR #70332’s body-level duplicate suppression may cover more diary sources; maintainers should choose the narrowest policy that suppresses unchanged automatic output without hiding legitimate new entries.
