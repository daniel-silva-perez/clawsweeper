## Summary

Keep open. Current main still builds light Dream Diary input from the configured lookback window and appends each successful narrative as a new timestamped diary entry; existing dedupe keeps identical bodies when timestamps differ. The bug remains valid, but open PR #70332 already targets body-level diary dedupe and duplicate append skipping, so this should not be closed or sent to a second repair lane yet.

## What This Changes

The issue requests incremental or body-level deduplication for light-phase Dream Diary work summaries so unchanged work-summary narrative blocks are not repeatedly appended across overnight cycles.

## Best Possible Solution

Land or rework a focused memory-core change that suppresses unchanged automatic Dream Diary output, with regression coverage for repeated light sweeps over unchanged input and for genuinely new snippets that must still append a new entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to seed stable short-term recall or daily memory input, run two light dreaming sweeps inside the same lookback window with deterministic narrative text, and observe that current append logic can write the same body under different timestamps.

## Solution Assessment

Unclear. The issue's lastEmittedAt cutoff is plausible for light summaries, but PR #70332's body-level duplicate skip may be a better central guard if it preserves legitimate distinct diary prose.
