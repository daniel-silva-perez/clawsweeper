## Summary

Keep open. Current main still builds light dreaming from a fixed recent-entry lookback and appends timestamped Dream Diary entries without cross-timestamp body or input duplicate suppression; the existing dedupe path intentionally preserves the same body when timestamps differ. Open PR #70332 appears to target this gap, so this issue should remain open for review rather than be closed.

## What This Changes

The issue asks memory-core dreaming to stop repeating identical light-phase work-summary material across cycles in `DREAMS.md`, either by tracking an emitted cutoff or deduping repeated output.

## Best Possible Solution

Land or rework a targeted memory-core change that suppresses repeated automatic Dream Diary material across repeated light sweeps, preferably with a phase input/body fingerprint stored under the dreaming state or checked during append, plus tests that repeated sweeps skip identical light narratives while genuinely new snippets still append.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to seed a daily note or short-term recall entry, trigger light dreaming multiple times with the same lookback window and a deterministic/fallback narrative, and inspect that the same candidate input can be appended as separate timestamped diary entries; current tests already assert the repeated light candidate input path.

## Solution Assessment

Unclear. Current main is not the best solution because it preserves same-body entries at different timestamps; PR #70332 is a plausible narrow fix, but maintainers should decide whether body-only dedupe or per-phase input fingerprints are safer before landing it.
