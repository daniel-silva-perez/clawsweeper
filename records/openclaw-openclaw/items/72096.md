## Summary

Keep open. Current main still builds light Dream Diary input from the configured lookback window and appends successful narrative text as a new timestamped diary entry; the existing dedupe path fingerprints timestamp plus body, so identical body text at different timestamps is intentionally retained. Open PR #70332 directly targets body-level diary dedupe and duplicate append skipping, so the next step is to review that fix shape rather than closing this report or queueing a duplicate repair.

## What This Changes

The issue reports repeated light-phase Dream Diary work summaries across scheduled dreaming cycles and asks for incremental or duplicate-suppressed light-phase output.

## Best Possible Solution

Land or rework a targeted `memory-core` change that prevents repeated automatic light Dream Diary material from being appended when the light input has not changed, with regression coverage for repeated light sweeps and for genuinely new snippets that should still produce a new diary entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to seed daily or short-term recall material, run two light dreaming sweeps inside the same lookback window with a deterministic narrative or fallback, and observe that current main can append the same body at different timestamps; I did not run this as a test because this review is read-only.

## Solution Assessment

Unclear. The issue's proposed `lastEmittedAt` cutoff is plausible for the light input path, while PR #70332's body-level dedupe may cover more repeated diary sources; maintainers should pick the narrowest policy that suppresses unchanged automatic output without hiding legitimate new diary entries.
