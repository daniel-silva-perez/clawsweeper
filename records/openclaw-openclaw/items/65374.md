## Summary

Keep this issue open. Current main still has a workspace-scoped Dreaming path: agents that resolve to the same workspace are grouped together, all of their session transcripts are read into one workspace corpus, and recall records use a workspace/day key without an agent filter. The discussion already narrowed this away from separate-gateway deployments and from the adjacent self-ingestion fixes.

## What This Changes

The issue requests agent-level isolation for memory-core Dreaming so shared-gateway multi-agent setups cannot pool session snippets, recall, or Dream Diary narrative across agent identities.

## Best Possible Solution

Define Dreaming's isolation contract explicitly, then make the default match the multi-agent docs: thread an agentId or isolation scope through workspace resolution, sweep execution, session-corpus storage, __dreaming_sessions__ recall keys, ranking/promotion, and Dream Diary generation. Add a regression test with two active agents sharing one workspace proving one agent cannot ingest, rank, promote, or narrate the other agent's transcript unless an explicit pooling mode is enabled.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists in current main: configure two agents with the same workspace, observe resolveMemoryDreamingWorkspaces returning one entry with both agentIds, then follow collectSessionIngestionBatches as it reads both agents' session files into the same workspace corpus.

## Solution Assessment

No. The current source-tagged, isolated-cron implementation is helpful but not the best solution for the isolated-agent contract because it still pools same-workspace agents before recall and narrative generation; the safer direction is agent-scoped Dreaming or an explicit documented pooling mode.
