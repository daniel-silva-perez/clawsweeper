## Summary

Keep this issue open. Current main has adjacent dreaming fixes, including isolated cron execution and source-tagged corpus lines, but the shared-workspace multi-agent isolation bug remains: dreaming still resolves work by workspace, gathers every agent mapped to that workspace, writes one corpus, and records recall under a workspace/day query without an agent filter.

## Best Possible Solution

Keep this open as a narrowed memory-core isolation bug. The best fix is to decide the intended scope, then either thread an active agent/scope through dreaming sweeps, session corpus storage, __dreaming_sessions__ recall keys, ranking, promotion, and Dream Diary generation, or document workspace-level pooling with a clear opt-out. Add a regression test with two active agents sharing one workspace proving Agent B cannot ingest or promote Agent A's transcript when agent isolation is enabled.
