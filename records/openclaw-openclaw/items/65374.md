## Summary

Keep this issue open. Current main has adjacent dreaming fixes, including source-tagged corpus lines and isolated cron targeting, but dreaming still has a workspace-scoped aggregation path: multiple agent IDs can be grouped under one normalized workspace, the managed sweep runs by deduped workspace path, session ingestion reads every agent attached to that workspace, and recall records use a workspace/day query without an active-agent filter. The issue is recent, has concrete follow-up reports, and prior discussion already narrowed it to multi-agent isolation rather than the older self-ingestion bug fixed by #65138.

## Best Possible Solution

Keep this open as a narrowed memory-core isolation bug. The best fix is to decide the intended scope, then thread an active agent or isolation scope through dreaming workspace resolution, session corpus storage, __dreaming_sessions__ recall, ranking, promotion, and Dream Diary generation. Add a regression test with two active agents sharing one workspace proving one agent cannot ingest, rank, promote, or narrate the other agent's transcript when agent isolation is enabled; if workspace-level pooling is intentional, document it clearly and provide an opt-out.
