## Summary

Current main still matches the reported Slack bug: Slack accepts and documents `dmHistoryLimit`, but the Slack monitor only derives and carries `historyLimit`, then gates pending-history recording, prompt injection, structured `InboundHistory`, and clearing on `isRoomish`, which excludes 1:1 DMs. The closed unmerged PR #67054 is useful prior work, and open #64427 overlaps on Slack DM history but is broader cross-session hydration rather than a clean supersession of this narrower monitor-wiring bug.

## What This Changes

Wire Slack `dmHistoryLimit` into 1:1 DM prompt/history assembly so non-threaded DM replies can include prior DM context.

## Best Possible Solution

Land a focused Slack monitor fix that derives `account.config.dmHistoryLimit`, carries it through `SlackMonitorContext`, computes an effective room-vs-DM history limit, records pending 1:1 DM history, uses that effective limit for prompt `combinedBody`, structured `InboundHistory`, and history clearing, and adds a regression test for a non-threaded 1:1 Slack DM. Keep cross-session history hydration and cross-channel parity as separate follow-up scope.

## Repair Work Prompt

Fix the Slack channel-monitor `dmHistoryLimit` bug reported here. Scope the PR to Slack monitor prompt/history assembly, not the broader #64427 cross-session `conversations.history` hydration work.

Expected implementation boundary:
- Derive an account-level `dmHistoryLimit` near the existing `historyLimit` derivation in `extensions/slack/src/monitor/provider.ts`, defaulting to 0 when unset unless existing config semantics require a documented fallback.
- Add `dmHistoryLimit` to `SlackMonitorContext` / `createSlackMonitorContext` and update Slack monitor test fixtures/helpers.
- In `extensions/slack/src/monitor/message-handler/prepare.ts`, compute an effective history limit for the current message: room/group-DM uses `ctx.historyLimit`, true 1:1 DM uses `ctx.dmHistoryLimit`.
- Record pending history for 1:1 DM messages when the effective DM limit is > 0, and use the effective limit for `combinedBody` history injection and structured `InboundHistory`.
- In dispatch/types, carry or recompute the effective history limit so history clearing is no longer gated solely by `prepared.isRoomish`.
- Add a regression test for a non-threaded Slack 1:1 DM with `dmHistoryLimit > 0` proving prior pending DM context appears in `Body`/`InboundHistory`, while default unset/0 behavior remains unchanged.
- Do not implement Slack API `conversations.history` session hydration in this PR; leave #64427 as separate scope unless maintainers explicitly ask to combine them.
- Add a single-line changelog fix entry with `Thanks @fmanjoo`; also credit useful confirmation/prior work if appropriate under repo changelog rules.
