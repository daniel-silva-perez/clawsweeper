For generic non-update gateway restarts, unmanaged Linux/custom-supervisor environments must not detach-spawn an orphaned gateway and then exit the original tracked PID. Prefer making restartGatewayProcessWithFreshPid() return disabled/in-process when no trusted supervisor is detected for non-update restarts, or an equivalent run-loop change that preserves the original PID. Do not change respawnGatewayProcessForUpdate() semantics: update.run must still be able to replace the OS process after package updates, including the #66865 hard-respawn/version-check behavior. Keep launchd/systemd/schtasks trusted-supervisor behavior unless the tests show a directly necessary adjustment. Update process-respawn and run-loop tests so unmanaged Linux no-supervisor generic restarts use in-process restart, and add/adjust coverage that config.restart-required changes still schedule SIGUSR1 but do not orphan under the generic restart path. Update doctor/platform docs so OPENCLAW_NO_RESPAWN=1 is described as custom-supervisor PID/orphan safety, not just startup overhead. Add a changelog fix entry with Thanks @mjamiv; include @iannwu only if the patch also materially addresses the launchd variant from the discussion. ## Risks / Open Questions

- Changing all restart modes broadly could regress update.run, which must replace the OS process after package files change so the new module graph loads.
- Changing supervisor detection by parent process name would be brittle; a narrower safer default for unmanaged non-update restarts is lower risk than guessing whether a custom parent will restart.
- Operators who intentionally rely on detached PID rotation for non-update restarts may need an explicit opt-in or clear migration note.
