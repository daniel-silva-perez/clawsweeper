## Summary

Current main only partially mitigates the reported restart/orphan class. Container config restarts now fall back to in-process restart, but unmanaged non-container Linux under a custom supervisor still falls through to detached spawn and original-PID exit; supervisor detection remains limited to launchd/systemd/schtasks, and doctor/docs still describe OPENCLAW_NO_RESPAWN as startup overhead rather than supervisor safety.

## Best Possible Solution

Keep this open and fix the non-update gateway restart contract in core: for SIGUSR1/config restarts without a trusted supervisor, use in-process restart by default instead of detached-spawning and exiting the tracked PID. Preserve the separate hard-respawn path for successful update.run, update doctor/docs to describe OPENCLAW_NO_RESPAWN as a supervisor-safety setting, and add regression coverage for unmanaged non-container Linux plus the existing container and update cases.

## Clownfish Work Prompt

Fix the remaining custom-supervisor SIGUSR1/config restart bug tracked by this issue. Current main still makes `restartGatewayProcessWithFreshPid()` detached-spawn and exit on unmanaged non-container Linux, which orphans the gateway under bash/Python/tini-style supervisors that track the original PID. Scope the fix to non-update restarts: preserve `respawnGatewayProcessForUpdate()` and the `update.run` hard process replacement behavior, including the existing `restart-unavailable` guard for unmanaged global installs. A conservative direction is to make non-update unmanaged restarts return `mode: "disabled"` with an explanatory detail so the run loop uses in-process restart, unless maintainers already have an explicit detached-respawn opt-in pattern to use. Keep launchd/systemd/schtasks behavior intact, keep Windows unmanaged disabled, and keep container in-process behavior intact. Update `openclaw doctor` and Linux/VPS/Raspberry Pi docs so `OPENCLAW_NO_RESPAWN=1` is described as supervisor-safety/restart-contract protection rather than only startup overhead. Add/update focused tests in process-respawn, run-loop if needed, doctor note tests, and update tests to prove update-run semantics were not changed. Add a user-facing changelog fix entry crediting @mjamiv and the launchd corroboration from @iannwu if included. Do not broaden this into update pipeline rewrites, service installer rewrites, or plugin API changes.

## Risks / Open Questions

- A fix that changes all restart modes could regress update.run, where the process must be replaced to load updated package files. Keep the update-specific `respawnGatewayProcessForUpdate()` path separate.
- Generic parent-process-name supervisor detection can misclassify foreground shells. A safer fix is to make non-update unmanaged SIGUSR1/config restarts in-process by default, with detached respawn as an explicit opt-in if maintainers still want that mode.
