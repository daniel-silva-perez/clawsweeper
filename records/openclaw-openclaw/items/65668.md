## Summary

Keep this issue open. Current main partially narrows the failure by disabling generic detached respawn on Windows without task markers and inside detected containers, but non-container Linux processes without systemd markers still use detached self-respawn, and the doctor/docs wording still presents `OPENCLAW_NO_RESPAWN=1` mainly as startup tuning rather than supervisor safety.

## What This Changes

The issue requests safer SIGUSR1/config.patch gateway restart behavior under custom supervisors, where detached self-respawn can orphan the gateway and cause EADDRINUSE restart loops.

## Best Possible Solution

Define a clear gateway restart contract for non-update config/SIGUSR1 restarts: avoid orphaning when an external supervisor owns lifecycle, keep update restarts on the fresh-process path when needed for new code, and update doctor plus public docs so `OPENCLAW_NO_RESPAWN=1` is documented as a supervisor-safety setting where applicable.

## Reproduction Assessment

Yes. Source and tests on current main show a Linux process without recognized supervisor hints and without `OPENCLAW_NO_RESPAWN=1` still takes the detached `spawned` path, and config writes still emit SIGUSR1 for restart-required changes.

## Solution Assessment

Unclear. The current container/Windows guard is a useful partial fix, but the report’s broader custom-supervisor case remains, and the safest default or detection strategy needs maintainer judgment before an automated behavior change.
