## Summary

Keep open. This draft PR is still an unmerged implementation candidate, and current main does not implement the requested opt-in `diagnostics.otel.mcp` canary for MCP JSON-RPC trace-context propagation, MCP diagnostic events, or MCP OpenTelemetry spans. The prior automated comment on this PR reached the same conclusion, and the related MCP reports in the provided context are broader MCP/plugin items rather than a canonical shipped implementation of this canary.

## Best Possible Solution

Review this draft as the implementation candidate for MCP trace-context propagation. The likely path is to keep it opt-in under `diagnostics.otel.mcp`, resolve the `traceparent` versus `tracestate` scope, verify the trust/privacy behavior with targeted tests plus an OTLP collector smoke when feasible, document the new config and semantics, and then merge or request changes. Do not close this PR unless current main later lands equivalent MCP canary behavior or maintainers reject the product direction.

## Risks / Open Questions

- Security/privacy review is still needed before merge because the PR accepts caller-provided MCP `params._meta.traceparent` as remote parent context and emits MCP request/session/tool metadata into diagnostics, even though the feature is opt-in/default-off in the patch.
- The PR body and config help mention `tracestate`, but the patch and current trace helper evidence show implementation around `traceparent` only; maintainers should either add `tracestate` handling or narrow the wording.
- The author explicitly noted no local Jaeger/OTLP trace-chain E2E was run, so collector-chain proof or an explicit maintainer decision on unit-only coverage remains useful.
- If accepted, user-facing OpenTelemetry docs need an update for `diagnostics.otel.mcp` and the MCP trace/span semantics.
