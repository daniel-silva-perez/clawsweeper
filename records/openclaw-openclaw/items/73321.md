## Summary

Keep open. Current main still lacks the proposed default-off MCP trace-context canary, so this PR is not obsolete; however, the branch should not merge as-is because the diff misses promised tracestate handling, misclassifies JSON-RPC error responses, and needs telemetry trust/privacy review.

## What This Changes

This PR adds an opt-in `diagnostics.otel.mcp` config surface, MCP JSON-RPC diagnostic events, OTEL span export, outbound bundled MCP traceparent injection, generated schema/help updates, and targeted tests.

## Best Possible Solution

Keep this PR as the implementation candidate, but require revisions before merge: either implement the full W3C traceparent/tracestate scope promised by the PR and config text or explicitly narrow the feature to traceparent-only, classify JSON-RPC protocol errors correctly, make trusted-event handling symmetric, document the canary in the OpenTelemetry docs, and validate the revised path with targeted tests plus the changed gate.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Propagate tracestate with the canary:** `src/gateway/mcp-http.trace.ts:64-65`
  - body: The PR body and config/help text promise `traceparent`/`tracestate` propagation via `params._meta`, but inbound extraction reads only `_meta.traceparent` and the outbound bundled MCP request only writes `traceparent`. With the canary enabled, vendor state and sampling hints carried in `tracestate` are silently dropped, so either implement `tracestate` handling or narrow the public surface to traceparent-only.
  - confidence: 0.88
- **[P2] Classify JSON-RPC error responses as MCP errors:** `src/gateway/mcp-http.trace.ts:141-142`
  - body: `completeMcpTraceScope` only treats `response.result.isError === true` as an error. JSON-RPC error objects returned by `jsonRpcError` for unsupported methods go through the completed path, so failed MCP requests export `mcp.request.completed` instead of `mcp.request.error`.
  - confidence: 0.9
- **[P2] Require trusted metadata for completed and error spans:** `extensions/diagnostics-otel/src/service.ts:2100-2118`
  - body: `recordMcpRequestStarted` rejects untrusted diagnostic events, but the completed/error handlers only check `tracesEnabled`. If an untrusted MCP diagnostic event is emitted, the OTEL plugin can still create a span even though no trusted start span was accepted; mirror the trusted guard across these handlers.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The patch adds no new dependencies, CI execution paths, or secret handling, but the telemetry trust boundary needs review because the canary accepts caller-provided MCP trace context and the OTEL handlers do not consistently require trusted metadata.

Concerns:

- **[medium] Remote MCP trace context becomes exported parent context:** `src/gateway/mcp-http.trace.ts:64`
  - body: When enabled, the canary parses caller-supplied `params._meta.traceparent` and uses it as parent context for exported spans. That is the feature goal, but it allows trace-correlation spoofing into an operator's collector unless the trust model and docs make the opt-in risk explicit.
  - confidence: 0.78
- **[low] Untrusted MCP events can create OTEL spans:** `extensions/diagnostics-otel/src/service.ts:2100`
  - body: The completed/error span handlers lack the `metadata.trusted` guard used by the started handler, so untrusted diagnostic events can produce exported MCP spans if such an event path is introduced.
  - confidence: 0.82
