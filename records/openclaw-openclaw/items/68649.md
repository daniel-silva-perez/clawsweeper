## Summary

Keep open. Current main still has the concrete PDF remote-fetch timeout gap: the PDF tool calls `loadWebMediaRaw` without an idle-read timeout, and `loadWebMediaRaw` still cannot forward one to `fetchRemoteMedia`. The lower fetch layer already supports and tests stalled body-read aborts, but related PR #68747 is closed unmerged, and broader lane recovery in #73581 does not supersede this PDF-specific bug.

## What This Changes

The issue requests timeout and recovery behavior for stalled remote PDF tool calls, plus separate recovery paths so slash commands and sessions do not remain queued behind a hung tool call.

## Best Possible Solution

Land a narrow PDF/media fix: add an optional idle-read timeout to `WebMediaOptions`, forward it through the HTTP branch into `fetchRemoteMedia`, set a conservative default for remote PDF loads, and add a PDF-tool regression test. Keep `/new`, `/restart`, and session watchdog recovery scoped to the broader lane-stall work.

## Reproduction Assessment

Yes for the narrow PDF-fetch hang: a remote PDF enters `pdf-tool.ts` and then `loadWebMediaRaw` without any idle timeout, while `fetchRemoteMedia` only aborts stalled body reads when `readIdleTimeoutMs` is supplied. The reporter also provided a transcript ending at a `pdf` tool call with no tool result and a lane wait warning.

## Solution Assessment

Yes, the narrow timeout plumbing proposed by #68747 is the best maintainable fix for the PDF-specific failure because the lower fetch layer already has the right contract. The slash-command bypass/watchdog portion should not be mixed into that patch; it belongs with the broader lane-stall issue.
