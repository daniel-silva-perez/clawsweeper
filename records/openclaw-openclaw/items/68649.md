## Summary

Keep open. Current main still has the concrete PDF remote-fetch gap: the `pdf` tool loads remote PDFs through `loadWebMediaRaw`, but neither the tool nor `WebMediaOptions` forwards a `readIdleTimeoutMs` into `fetchRemoteMedia`. The lower-level stalled-read timeout exists and is tested, and related PR #68747 proposed the right narrow fix, but that PR is closed unmerged and its head commit is not present in this checkout. The broader open lane-stall issue #73581 can track `/new`/`/restart` recovery, but it does not replace this source-backed PDF timeout bug.

## Best Possible Solution

Keep this issue open. Land a narrow fix like PR #68747: add optional `readIdleTimeoutMs` to `WebMediaOptions`, forward it through `loadWebMediaRaw` into `fetchRemoteMedia`, set a conservative default for remote PDF loads, and add a PDF-tool regression test that proves the timeout is passed. Track `/new`/`/restart` bypass and session-level watchdog recovery in the broader lane-stall thread #73581.
