## Summary

Keep open. Current main still leaves the PDF remote-download path without an idle body-read timeout, even though the lower fetch helper can abort stalled reads when that option is supplied; the direct fix PR is closed unmerged and the broader lane-stall issue does not supersede the PDF-specific gap.

## What This Changes

The issue reports that the PDF tool can hang indefinitely on a stalled large remote PDF and asks for a PDF fetch timeout plus stronger session recovery for queued /new and /restart commands.

## Best Possible Solution

Land a narrow PDF/media fix: add an optional readIdleTimeoutMs to WebMediaOptions, forward it through the HTTP branch into fetchRemoteMedia, set a conservative default for remote PDF loads, and add a PDF-tool regression test. Keep slash-command bypass and general session watchdog recovery in the broader lane-stall track.

## Reproduction Assessment

Yes for the PDF-specific failure path: a remote PDF enters pdf-tool.ts and then loadWebMediaRaw without any idle timeout, while fetchRemoteMedia only aborts stalled body reads when readIdleTimeoutMs is supplied. The reporter also provided a transcript ending at a pdf tool call with no tool result and a lane wait warning.

## Solution Assessment

Yes for the narrow repair: plumbing the existing idle-read timeout contract through WebMediaOptions and setting it from the PDF tool is the smallest maintainable fix. The session queue bypass/watchdog portion should be handled separately under the broader lane-stall issue rather than mixed into this patch.
