## Summary

Keep this issue open. Current main still lacks the reported fallback from trusted `read` image tool output to the original tool-call args when `details.path` is absent, and open PR #46268 still references this issue with closing syntax. The downstream pending-media merge and Feishu media send path exist, so the remaining gap is populating pending media for the `read` image result shape that only preserves the file path in the start args.

## Best Possible Solution

Keep this issue open and review or consolidate PR #46268 and PR #50359. The fix should thread stored `tool_execution_start` args into embedded media extraction, fall back to `path` / `file_path` / `filePath` only when trusted image content exists and no explicit media path is present, then preserve that media through pending-tool-media merge and the Feishu `sendMediaFeishu` path with focused regression tests.
