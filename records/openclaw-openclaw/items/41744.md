## Summary

Keep this issue open. Current main still does not promote trusted `read` image results whose media path exists only in the original tool-call args, and PR #46268 is still open with closing syntax for this issue. The downstream pending-media merge and Feishu media send path exist, so the remaining work is the focused `startArgs.path` / `file_path` / `filePath` fallback with trust-scoped tests.

## Best Possible Solution

Keep this issue open and review or consolidate PR #46268 and PR #50359. The implementation should thread `tool_execution_start` args into embedded tool-result media extraction, fall back to `path` / `file_path` / `filePath` only when trusted image content exists and no explicit media path is present, then verify that pending media is merged into the final reply and delivered through Feishu `sendMediaFeishu` with focused regression tests.
