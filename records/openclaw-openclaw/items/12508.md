## Summary

Current main still has the mutable, sequential plugin hook pipeline described in the report, without hook-chain integrity state, per-handler mutation audit records, or a per-hook registration allowlist. The prior issue discussion also kept this open for core/plugin-runtime security design, and related #18571 is narrower outbound verification work rather than a replacement.

## What This Changes

Request hook-chain integrity protection for mutable plugin hook execution, including audit logging, integrity checks, and per-hook registration policy.

## Best Possible Solution

Define host-owned hook provenance and per-handler mutation audit semantics in the plugin runtime, explicitly distinguish in-process native plugins from any isolated-plugin IPC path, then add a backwards-compatible per-hook permission policy with docs and focused runtime/config tests.
