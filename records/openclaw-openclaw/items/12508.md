## Summary

Keep open. Current main still documents and implements mutable, sequential plugin hooks without hook-chain integrity checks, per-handler modification audit records, or a per-hook manifest permission allowlist. The prior maintainer review in the thread also explicitly kept this open for core/plugin-runtime security design.

## Best Possible Solution

Keep this open for maintainer-owned core/plugin-runtime security design. The likely path is host-owned hook provenance and per-handler modification audit semantics in src/plugins/hooks.ts and call sites, plus a backward-compatible per-hook permission/config or manifest policy. Related outbound verification work such as #18571 can inform message_sending, but it should not replace the broader hook-chain integrity, audit, and permission design requested here.
