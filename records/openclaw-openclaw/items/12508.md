## Summary

Keep this issue open. Current main still has trusted in-process plugin hooks that can intentionally rewrite tool parameters, transcript-adjacent messages, and outbound content through sequential hook chains. Existing docs and tests cover those mutation paths as supported behavior, while current policy only has coarse prompt/conversation hook gates, not host-owned integrity verification, per-handler modification audit records, or per-hook manifest permissions.

## Best Possible Solution

Keep this issue open for core plugin-runtime security design. The best path is to design host-owned hook provenance and audit semantics across src/plugins/hooks.ts and the specific tool/message/outbound call sites, then add manifest/config policy for per-hook permissions without breaking legitimate documented plugin rewrites. Related narrower work such as #18571 can inform the outbound verification portion, but it does not replace this broader integrity/audit/permission design.
