## Summary

Keep open. Current main still has mutable, sequential plugin hooks without hook-chain integrity state, per-handler modification audit records, or a per-hook permission allowlist, and the issue thread already includes maintainer-facing reviews keeping this for security design rather than cleanup closure.

## What This Changes

The issue requests hook-chain integrity protection, modification auditing, and hook permissioning to detect or constrain plugin message tampering.

## Best Possible Solution

Define host-owned hook provenance and per-handler modification audit semantics in the plugin runtime, explicitly distinguish in-process native plugins from any future isolated-plugin IPC path, then add a backwards-compatible per-hook permission policy with docs and focused runtime/config tests.
