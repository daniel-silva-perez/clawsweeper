## Summary

Keep open. Current main still exposes mutable sequential plugin hooks without hook-chain integrity state, per-handler modification audit records, or a per-hook registration allowlist; the prior issue discussion also kept this for maintainer security-design work, and #18571 is narrower outbound verification work rather than a replacement.

## What This Changes

Request to add hook-chain integrity checks, modification auditing, and per-hook permissions for plugin hook message mutation.

## Best Possible Solution

Define host-owned hook provenance and per-handler modification audit semantics in the plugin runtime, distinguish in-process native plugins from any future isolated-plugin IPC path, then add a backwards-compatible per-hook permission policy with docs and focused runtime/config tests.
