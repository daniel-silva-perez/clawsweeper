## Summary

Keep open. Current main still matches the reported security-sensitive gap: plugin hooks can run in priority order and mutate messages/results without hook-chain integrity state, per-handler mutation provenance, or a per-hook registration allowlist. The existing maintainer-thread reviews also kept this open, and related #18571 is narrower outbound-verification work rather than a replacement for this broader plugin-runtime design.

## What This Changes

Request to add tamper-evident hook-chain integrity, mutation audit logging, and per-hook permission controls for plugin hook execution.

## Best Possible Solution

Define host-owned hook provenance and per-handler mutation audit semantics in the plugin runtime, explicitly separate the in-process native-plugin trust model from any isolated IPC path, then add a backwards-compatible per-hook permission policy with docs and focused runtime/config tests.
