## Summary

Keep open. Current main still exposes mutable, sequential plugin hooks without hook-chain integrity state, per-handler modification audit records, or a per-hook permission allowlist; the prior thread review also kept this for maintainer security-design work, and #18571 is a narrower outbound-verification request rather than a replacement.

## What This Changes

The issue requests hook-chain integrity protection, audit logging, and per-hook permission controls for mutable plugin hook execution.

## Best Possible Solution

Define host-owned hook provenance and per-handler modification audit semantics in the plugin runtime, explicitly distinguish in-process native plugins from any future isolated-plugin IPC path, then add a backwards-compatible per-hook permission policy with docs and focused tests around the runner, registry, loader, and transcript/message call sites.
