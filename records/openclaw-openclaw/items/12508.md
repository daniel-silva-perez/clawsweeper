## Summary

Keep open. Current main still has mutable, sequential plugin hooks without hook-chain integrity state, per-handler mutation audit records, or a per-hook registration allowlist; prior thread reviews also kept this open as security-sensitive plugin-runtime design work, and related #18571 is narrower outbound verification work rather than a replacement.

## What This Changes

Request adds hook-chain integrity protection for plugin hooks, including tamper detection, audit logging around sequential message modifications, and a per-hook permission model.

## Best Possible Solution

Define host-owned hook provenance and redacted per-handler mutation audit semantics in the plugin runtime, explicitly separate in-process native-plugin trust from any isolated IPC path, then add a backwards-compatible per-hook registration policy with schema, docs, and focused runtime/config tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: register ordered modifying hooks such as before_message_write, tool_result_persist, or message_sending and current main passes rewritten results through sequential handlers without checksum, provenance, or audit verification; existing tests already prove before_message_write mutations persist.

## Solution Assessment

Unclear whether the issue's checksum proposal is the best standalone fix, because native plugins currently share the Gateway process trust boundary. The maintainable path is a maintainer-defined security model first, then provenance/audit and per-hook policy that match that model.
