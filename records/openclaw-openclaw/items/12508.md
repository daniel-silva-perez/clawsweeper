## Summary

Keep open. Current main still has mutable, sequential plugin hooks without hook-chain integrity state, per-handler mutation audit records, or a per-hook registration allowlist; prior thread reviews also kept this open as security-sensitive plugin-runtime design work, and related #18571 is narrower outbound verification work rather than a replacement.

## What This Changes

The issue requests hook-chain integrity checks, audit logging, and per-hook permission controls to detect or prevent silent plugin hook message tampering.

## Best Possible Solution

Define host-owned hook provenance and per-handler mutation audit semantics in the plugin runtime, explicitly separate the in-process native-plugin trust model from any isolated IPC path, then add a backwards-compatible per-hook permission policy with docs and focused runtime/config tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: register ordered modifying hooks such as before_message_write or message_sending and the current runner will pass mutable results through sequential handlers without any checksum/provenance/audit guard; existing tests already prove before_message_write mutations persist.

## Solution Assessment

Unclear whether the issue's checksum proposal is the best standalone fix, because native plugins currently share the Gateway process trust boundary. The maintainable path is a maintainer-defined security model first, then provenance/audit and per-hook policy that match that model.
