## Summary

Keep open. Current main still has mutable sequential plugin hooks without hook-chain integrity state, per-handler modification audit records, or a per-hook registration allowlist; prior issue discussion also kept this for maintainer security-design work, and related #18571 is narrower outbound verification work rather than a replacement.

## What This Changes

The issue requests hook-chain integrity protection for plugin hook execution, including tamper detection, audit logging of hook modifications, and per-hook registration permissions.

## Best Possible Solution

Define host-owned hook provenance and per-handler modification audit semantics in the plugin runtime, explicitly distinguish in-process native plugins from any future isolated-plugin IPC path, then add a backwards-compatible per-hook permission policy with docs and focused runtime/config tests.
