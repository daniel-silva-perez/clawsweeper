## Summary

Keep open. Current main still exposes mutable plugin hook surfaces without hook-chain integrity checks, per-handler modification audit records, or per-hook manifest permission declarations. The April 26 maintainer review comment already kept this open for core/plugin-runtime security design, and the current source at 1b25dcf57a9f still matches that assessment.

## Best Possible Solution

Keep this open for maintainer-owned core/plugin-runtime security design. The likely direction is host-owned hook provenance and per-handler modification audit semantics in src/plugins/hooks.ts and call sites, plus a backward-compatible per-hook permission/config or manifest policy. Related outbound verification work such as #18571 can inform message_sending, but should not replace the broader hook-chain integrity/audit/permission design requested here.
