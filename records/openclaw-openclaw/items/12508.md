## Summary

Keep open. Current main still exposes mutable sequential plugin hook paths without hook-chain integrity state, per-handler mutation audit records, or a per-hook registration allowlist, and the existing discussion already kept this for security-sensitive plugin-runtime design rather than a narrow cleanup close.

## What This Changes

Request hook-chain integrity protection, audit logging, and per-hook permission controls for plugin hooks that can modify messages, tool calls, and agent context.

## Best Possible Solution

Define host-owned hook provenance and redacted per-handler mutation audit semantics in the plugin runtime, explicitly separate native in-process plugin trust from any future isolated IPC path, then add a backwards-compatible per-hook registration policy with schema, docs, and focused runtime/config tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: register ordered modifying hooks such as message_sending, tool_result_persist, or before_message_write, and current main passes rewritten results through sequential handlers without checksum, provenance, or audit verification; existing tests already prove before_message_write mutations persist.

## Solution Assessment

Unclear whether the issue's checksum proposal is the best standalone fix, because native plugins currently share the Gateway process trust boundary. The maintainable path is a maintainer-defined security model first, then provenance/audit and per-hook policy that match that model.
