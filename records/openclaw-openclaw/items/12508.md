## Summary

Keep open. Current main still exposes trusted in-process plugin hooks that can sequentially rewrite tool parameters, transcript-adjacent messages, and outbound content. Docs and tests treat those mutations as supported behavior, while current policy only has coarse prompt/conversation hook gates, not hook-chain integrity verification, per-handler modification audit records, or per-hook manifest permissions.

## Best Possible Solution

Keep this open for maintainer-owned core plugin-runtime security design. The likely path is host-owned hook provenance and modification audit semantics in src/plugins/hooks.ts and the concrete tool/message/outbound call sites, plus a backwards-compatible per-hook permission/config or manifest policy in the plugin registry/config surface. Related outbound verification work such as #18571 can inform the message_sending portion, but it does not replace the broader integrity/audit/permission design requested here.
