## Summary

Keep open. Current main still exposes mutable, sequential plugin hooks without hook-chain integrity state, per-handler modification audit records, or a per-hook permission allowlist; the prior maintainer-thread review also kept this for plugin-runtime security design, and related #18571 is narrower outbound verification work rather than a replacement.

## What This Changes

Request host-owned hook-chain integrity protection with tamper detection/checksums, audit logging for hook modifications, and manifest/profile-based per-hook permission controls for plugin hooks.

## Best Possible Solution

Define host-owned hook provenance and per-handler modification audit semantics in the plugin runtime, explicitly cover in-process versus future isolated plugin behavior, then add a backwards-compatible per-hook permission policy with docs and focused tests around the runner, registry, loader, and transcript/message call sites.
