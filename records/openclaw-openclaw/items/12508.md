## Summary

Keep open. Current main still has mutable sequential plugin hooks without hook-chain integrity state, per-handler modification audit records, or a per-hook registration allowlist. The issue discussion already kept this for maintainer security-design work, and related #18571 is narrower outbound-verification work rather than a replacement.

## What This Changes

The issue requests hook-chain integrity protection, audit logging, and per-hook permissions so plugins cannot silently tamper with messages or hook payloads as they pass through sequential plugin hooks.

## Best Possible Solution

Define host-owned hook provenance and per-handler modification audit semantics in the plugin runtime, explicitly distinguish in-process native plugins from any future isolated-plugin IPC path, then add a backwards-compatible per-hook permission policy with docs and focused runtime/config tests.
