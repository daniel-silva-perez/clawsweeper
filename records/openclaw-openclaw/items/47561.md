## Summary

Keep open. Current main still derives Telegram direct-message session keys from the active `session.dmScope`; switching from `per-channel-peer` back to `main` still routes the same DM to `agent:<agentId>:main`. I found only the inverse cleanup path for main-to-isolated DM routing, not a migration, rebind, warning, or preference for the fresher `/new` session when the routing mode returns to unified.

## Best Possible Solution

Keep this issue open and fix the generic session/routing layer with channel coverage. The durable path is to persist reset intent or conversation-to-session binding metadata when `/new` rotates a session, then on later dmScope/routing changes either rebind the fresh session to the new canonical key, warn that the fresh session was superseded, or expose a doctor diagnostic. Add regression coverage for the `per-channel-peer` to `main` transition and keep the existing inverse cleanup tests.
