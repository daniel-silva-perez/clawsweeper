## Summary

Keep open. Current main still recomputes Telegram/direct session keys from the active `session.dmScope`, so reverting from `per-channel-peer` back to `main` routes the same DM back to `agent:<agentId>:main`. I found only the inverse cleanup path, which clears stale main-session delivery metadata when moving from unified main to per-channel DM keys; I did not find a migration, rebind, warning, or preference for the fresher per-channel `/new` session when routing later returns to unified mode. Related #69599 and closed #47504 cover adjacent fresh-session and channel-freeze symptoms, but neither supersedes this routing-mode transition case.

## Best Possible Solution

Keep this issue open and fix it in the generic session/routing layer with channel coverage. A durable fix should persist reset intent or conversation-to-session binding metadata when `/new` rotates a session, then on later dmScope/routing changes either rebind the fresh session to the new canonical key, explicitly warn that the fresh session was superseded, or provide a doctor diagnostic. Add regression tests for the `per-channel-peer` to `main` transition and keep the existing inverse cleanup tests.
