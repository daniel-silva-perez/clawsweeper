## Summary

Current main does not yet satisfy the reported routing-mode transition bug. Telegram/direct routing still derives the active session key from the current `session.dmScope`; when that setting is `main`, a DM resolves to `agent:<agentId>:main`, while `per-channel-peer` resolves to `agent:<agentId>:telegram:direct:<peer>`. Existing cleanup only handles the opposite direction by retiring stale main-session delivery metadata when moving to per-channel DM keys. I did not find a core path that migrates, rebinds, warns about, or prefers a newer per-channel `/new` session when the same conversation later resolves back to the main session.

## Best Possible Solution

Keep this issue open and fix it in the generic session/routing layer, with channel coverage. The likely path is to persist conversation-to-session reset intent or migration metadata when `/new` rotates a session, then on later dmScope/routing changes either rebind the fresh session to the new canonical key, explicitly warn that the fresh session was superseded, or provide a doctor diagnostic. Add regression tests for the per-channel-peer to main transition, not only the existing main to per-channel-peer cleanup.
