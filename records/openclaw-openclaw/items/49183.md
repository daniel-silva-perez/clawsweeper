## Summary

Keep open. Current main has adjacent tool-policy pieces, including global/agent/provider policies and group `toolsBySender`, but it does not implement the RFC's operator-scope-to-session-tool-policy bridge or a `toolsByScope` config surface. The related discussion links useful narrower slices (#53530, #53496, #58434), but those do not fully supersede the scope-based RBAC request.

## Best Possible Solution

Keep this RFC open unless a maintainer explicitly consolidates it into #43673. The best implementation path is to add a typed, documented session/scope-aware policy input to the shared tool-policy resolver, thread authenticated gateway scopes and relevant sender identity into agent/tool resolution, preserve owner-only gates, and cover the behavior with schema docs plus tests proving narrower operator scopes cannot access denied tools while admin/full-operator sessions retain intended access.

## Risks / Open Questions

- Closing as implemented would overstate current behavior: the existing sender/group policies and per-agent policies do not let an `operator.read` session automatically receive a different agent tool set than an `operator.admin` session.
- Closing as a duplicate of #43673 would be plausible only after maintainer consolidation; this issue contains a narrower concrete design and active related implementation discussion that is not fully captured by the broader org/team umbrella.
