## Summary

Keep this issue open. Current main has useful diagnostics, replay hygiene, and context-pressure mitigations, but I did not find a source-backed fix for the reported long-session GateAI/Claude-on-Telegram path where live tool calls arrive at OpenClaw with `{}` or partial required params. The related merged PR only made missing-param failures more visible; it explicitly did not change parameter transport or validation behavior.

## Best Possible Solution

Keep the issue open and treat it as a live long-session tool-argument reliability bug. The best next step is to capture a current-main trajectory or provider payload for Telegram + GateAI/Claude after many turns, then add a targeted regression around the actual failure point: stream argument assembly if args are emitted but lost, provider/tool schema injection if the model stops seeing required schemas, or a broader provider-scoped repair if GateAI’s Anthropic-compatible stream needs the same treatment as the existing Kimi/openai-completions repair. The existing received-param logging from PR #55317 should be kept as diagnostic support, but it is not the fix.
