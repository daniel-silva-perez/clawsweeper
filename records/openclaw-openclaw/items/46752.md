## Summary

Keep this PR open. Current main still has unbounded tool-schema parameters that match the hardening goal, but this patch is not merge-ready because several proposed caps conflict with current runtime/provider behavior and the browser schema has moved to the bundled browser plugin path.

## What This Changes

The PR adds schema-only maxItems, maximum, minimum, and description metadata across agent tool schemas for browser, canvas, gateway, memory, message, node, session, subagent, and web-fetch tools.

## Best Possible Solution

Rebase and narrow the hardening so each maxItems, maximum, and maxLength is backed by the current runtime, provider API, protocol schema, or user-facing docs; update the browser plugin-owned schema path; and add focused contract tests that fail when schema bounds fall out of sync.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Allow node media calls to wait for max recordings:** `src/agents/tools/nodes-tool.ts:113`
  - body: The new 120s timeoutMs cap is lower than valid media durations: current nodes allow 300000 ms recordings, and screen_record can run for that full duration. A caller recording near the documented max can no longer set a gateway timeout long enough for the operation to complete.
  - confidence: 0.93
- **[P2] Keep sessions_list compatible with 200-row lookups:** `src/agents/tools/sessions-list-tool.ts:36`
  - body: Capping sessions_list.limit at 50 rejects a pattern OpenClaw still uses internally: announce-target resolution asks sessions.list for 200 rows, and the gateway protocol has no maximum. Larger installs can lose the ability to find older or less-active sessions.
  - confidence: 0.9
- **[P2] Preserve long-running subagent timeouts:** `src/agents/tools/sessions-spawn-tool.ts:44`
  - body: The runtime and docs still allow any non-negative runTimeoutSeconds value, with 0 meaning no timeout. Adding maximum: 3600 makes legitimate long-running or no-timeout subagent jobs fail schema validation before reaching the existing runtime behavior.
  - confidence: 0.91
- **[P2] Use the browser backend snapshot limit:** `src/agents/tools/browser-tool.schema.ts:98`
  - body: The PR lowers browser snapshot limit to 1000, but current browser snapshot backends clamp at 2000. This rejects large-page snapshot requests that the implementation can still serve, reducing useful browser context on complex pages.
  - confidence: 0.84
- **[P2] Permit Discord media channel type:** `src/agents/tools/message-tool.ts:329`
  - body: Discord's GuildMedia channel type is 16, and the current Discord runtime already handles it as forum-like. Capping channel management type at 15 blocks valid channel-create calls for that supported channel type.
  - confidence: 0.88
