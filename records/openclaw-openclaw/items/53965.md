## Summary

Keep this PR open, but do not merge it as-is. Current main still routes the affected secrets apply writes through the unguarded sync writer, while the PR only adds an unused helper, so it would not fix the intended Windows-mounted Docker volume crash.

## What This Changes

The PR adds a new `src/shared/fs-atomic-fix.ts` helper that writes text through a temp file, ignores `chmodSync` failures, and renames the temp file into place.

## Best Possible Solution

Replace or revise this PR with a focused change on the active secrets write path. The desired end state is a tested Windows/Docker-safe persistence path for secrets apply that preserves directory creation, uses the existing hardened atomic-write conventions where practical, and keeps unexpected credential-file permission failures visible.

## Risks / Open Questions

- Merging this PR as-is would not change runtime behavior because the new helper has no callers.
- If the helper were wired in unchanged, it would silently swallow all `chmodSync` failures on credential-bearing writes instead of narrowly handling the expected Windows/Docker filesystem limitation.
- The helper omits existing parent-directory creation behavior, so it would regress writes when the destination directory does not already exist.
- Related Windows auth-profile persistence work is also tracked around #62099 and #67077, so maintainers should avoid splitting overlapping fixes without deciding the boundary.
