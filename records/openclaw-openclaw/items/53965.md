## Summary

Keep this PR open for maintainer handling, but it should not be treated as implemented on current main. Main has some later Windows atomic-write hardening in generic config/JSON helpers, but the active secrets apply sync writer still has the unguarded `chmodSync` path this PR and its review comments discuss, and the PR’s added helper is not wired into any caller.

## Best Possible Solution

Request changes or supersede this PR with a focused patch on the active write paths: update `src/secrets/shared.ts` or route secrets apply through the hardened JSON/file helpers, preserve `ensureDirForFile`, handle expected Windows/NTFS/CIFS chmod and rename failures deliberately, and add a regression test that exercises `runSecretsApply` with mocked `EPERM`. Also consider whether the include writer in `src/config/mutate.ts` needs the same Windows rename fallback.

## Risks / Open Questions

- Merging the PR as-is would falsely suggest the Windows-mounted Docker crash is fixed because the new helper has no callers.
- The active `src/secrets/shared.ts` write path can still throw on `chmodSync` during `openclaw secrets apply` auth/env writes or rollback restores.
- Reusing the PR helper directly would regress parent-directory creation and silently swallow all chmod errors, which is sensitive for config/auth material.
