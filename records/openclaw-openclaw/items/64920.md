## Summary

Keep open. The PR is aimed at a real Feishu queueing bug: current main still routes Feishu message processing through chat-wide sequential keys, while current session routing and docs already distinguish native `topic_group` semantics. The proposed branch is not merge-ready because it is stale/dirty and still misses current-main `topic_group` behavior in the proposed queue-lane path; the Lark plugin context in the PR body is useful supporting evidence for the direction, but this should land as a narrower Feishu-only update.

## What This Changes

The PR adds Feishu topic-aware sequential queue keys and tests, extracts group-session scope resolution, wires Feishu dispatch to pass that scope, updates the changelog, and carries unrelated cleanup across other channel, agent, gateway, and plugin SDK files.

## Best Possible Solution

Land a narrow Feishu-only implementation on current main that makes the sequential queue key use the same group/session scope and topic-id resolver as `resolveFeishuGroupSession`, including native `topic_group` thread-first semantics, while preserving chat-wide `/stop` and `/btw` lanes and normal-group/DM FIFO behavior.
