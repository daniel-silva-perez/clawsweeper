## Summary

Keep open. The user problem is real on current main because Feishu sequential keys still collapse topic metadata to one chat-wide queue, but this PR is not ready to merge against current main: its new routing gates exclude native `topic_group` events and its lane-id precedence no longer matches the current Feishu session resolver.

## What This Changes

The PR adds Feishu topic-aware sequential queue keys and tests, extracts group-session scope resolution, wires the Feishu monitor to pass that scope, updates the changelog, and carries several unrelated CI/type/lint cleanup changes outside Feishu.

## Best Possible Solution

Land a narrow Feishu-only implementation that makes the sequential queue lane use the same group/session scope and topic-id resolver as `resolveFeishuGroupSession` for both `group` and `topic_group`, with focused tests for topic starters, topic replies, normal groups, DMs, `/stop`, and `/btw`; split unrelated core/channel cleanup into separate work.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Route topic_group through the topic-lane path:** `extensions/feishu/src/monitor.account.ts:421`
  - body: `topic_group` is a Feishu group chat type on current main, but this gate only resolves `groupSessionScope` for `chat_type === "group"`. Native topic-group messages therefore enter `getFeishuSequentialKey` with no topic scope and keep the old chat-wide lane, so the PR's central case still serializes behind other topics. Include `topic_group` here and in the key function, preferably via the existing group-chat helper.
  - confidence: 0.93
- **[P2] Mirror topic_group thread-id precedence:** `extensions/feishu/src/sequential-key.ts:74`
  - body: Current session routing uses `thread_id ?? root_id` for `topic_group` events so a topic starter and its replies share one topic peer. This helper uses `root_id ?? thread_id`; after `topic_group` is admitted to the lane path, replies with both fields would queue on a different lane from the session key and could interleave shared-session history.
  - confidence: 0.88
