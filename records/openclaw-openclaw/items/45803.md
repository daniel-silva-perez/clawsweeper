## Summary

Keep this issue open. Current main still does not add `allowed_mentions` to the Discord final outbound text/media/forum/webhook send payloads, and the related implementation PR was closed unmerged. The root-cause claim needs maintainer care because current Discord docs say regular bot messages default to parsing all mention types when `allowed_mentions` is omitted, while webhooks/interactions default to user mentions only.

## Best Possible Solution

Keep this open for a focused Discord plugin fix or maintainer triage. The next best step is to live-verify the failing notification path, then either add a narrow `allowed_mentions` policy with regression tests across final text/media/forum/webhook sends, or close manually with evidence that current Discord API defaults already notify correctly for the affected send type.

## Risks / Open Questions

- Blindly adding `parse: ["users", "roles"]` could change mention-safety behavior, especially for role pings and webhook sends. The implementation should explicitly decide whether final bot sends, webhooks, forum starter messages, media captions, polls, and stickers share the same policy.
- The issue may be describing a real notification failure, but official Discord docs suggest omitted `allowed_mentions` is not enough to explain regular bot-message user/role pings by itself; a live Discord repro or logs would make the fix direction firmer.
