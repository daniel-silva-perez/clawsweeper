## Summary

Keep this PR open. Current main still rejects symlinked personal/project `.agents/skills` directories, so the branch is not obsolete, but the proposed containment exception changes a skill-loading trust boundary and still needs maintainer review plus two concrete fixes before merge.

## What This Changes

The PR adds a logical-path containment fallback for `agents-skills-personal` and `agents-skills-project`, with tests for symlinked personal/project skills and managed-source symlink rejection.

## Best Possible Solution

Land this PR or an equivalent narrow patch only if maintainers want symlinked personal and project `.agents/skills` directories to be supported. Keep strict physical containment for managed, bundled, workspace, extra, and plugin-owned skill roots; gate or adapt the symlink tests for Windows; and update the public Skills security docs at `https://docs.openclaw.ai/tools/skills` to describe the accepted exception.

- **[P2] Gate symlink tests on Windows:** `src/agents/skills.agents-skills-directory.test.ts:145-148`
  - body: The new tests call `fs.symlink()` unconditionally, but nearby skill symlink coverage is skipped on Windows because symlink creation and link type behavior are platform-sensitive there.   - confidence: 0.79

  - body: This fallback permits personal/project `.agents/skills` entries whose real target is outside the configured root, but the public Skills security docs do not describe that exception or which roots remain physically contained. Since this changes a skill-loading trust boundary, document the accepted behavior before merging.
