## Summary

Keep this PR open. Current main still rejects symlinked personal/project `.agents/skills` entries through strict realpath containment, and the proposed branch is a narrow implementation candidate rather than obsolete work. It needs maintainer review because the change intentionally broadens the skill-loading trust boundary for user-managed sources and should be reconciled with docs/security wording and #46690.

## What This Changes

The PR adds a logical-path containment fallback for personal and project `.agents/skills` sources and tests symlinked personal/project skill loading plus managed-source symlink rejection.

## Best Possible Solution

Land this PR or an equivalent narrow patch only if maintainers want user-managed `.agents/skills` symlinks as a supported workflow. Preserve strict physical containment for managed, bundled, workspace, and extra skill roots; add or keep focused tests for personal/project acceptance and managed rejection; clarify whether symlinked `SKILL.md` files are allowed; and update the public Skills security docs.

## Risks / Open Questions

- The proposed fallback is source-wide for `agents-skills-personal` and `agents-skills-project`, so it would also accept an in-root skill directory whose `SKILL.md` is a symlink to an external file unless maintainers narrow or explicitly bless that behavior.
- Project `.agents/skills` can be repository-controlled in some workflows; allowing external symlink targets there is a trust-boundary decision that should be documented if accepted.
- The PR overlaps conceptually with #46690, which asks for better handling of skill directories outside canonical paths; maintainers should decide whether symlinked external skills are an accepted escape hatch or should be surfaced diagnostically.
