## Summary

Current main still matches the reported Telegram multi-account inconsistency. Account resolution intentionally drops channel-level `groups` when two or more Telegram accounts are configured, while the mention policy still falls back to channel-level groups, so `requireMention` can work from the same block that prompt/skill/topic resolution cannot see. The linked implementation candidate #64650 is closed and unmerged, so there is no mainline fix to close against.

## Best Possible Solution

Keep this issue open. The best fix is a Telegram-owned resolver change, likely in `extensions/telegram/src/bot-core.ts`, that resolves prompt/skill/topic context from `channels.telegram.accounts.<accountId>.groups` first and falls back to `channels.telegram.groups` only for message context construction, not account routing. Add a regression test in `extensions/telegram/src/bot.create-telegram-bot.test.ts` for two configured Telegram accounts with channel-level group/topic `systemPrompt` and `skills`. If maintainers decide channel-level group prompts should remain unsupported in multi-account mode, add a Telegram doctor warning and public docs clarification instead of leaving the partial behavior silent.

## Risks / Open Questions

- A naive fix in `mergeTelegramAccountConfig` could reintroduce #30673 by making routing/delivery inherit channel-level groups wholesale in multi-account setups.
- A prompt-only fallback should keep account-scoped `groups` authoritative, so channel-level prompt or skill config does not override an account that intentionally defines its own group map.
