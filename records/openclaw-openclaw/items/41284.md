## Summary

Keep this PR open. It targets a real iOS signing gap that still exists on current main, but the proposed regression test still uses a fake Python executable that bypasses the new inline plist parser, so the patch is not merge-ready without maintainer acceptance of the live macOS proof or a test update.

## What This Changes

The PR updates `scripts/ios-team-id.sh` to parse `defaults export` Xcode preferences for `IDEProvisioningTeamByIdentifier` and `IDEProvisioningTeams`, preserves the `plutil` fallback, and adjusts the iOS team-id test fixture and expected selection.

## Best Possible Solution

Land the focused signing-helper support after the identifier-key regression case runs the actual inline Python parser against mocked `defaults` and `plutil` tools, while preserving explicit env overrides, existing `IDEProvisioningTeams`, managed-profile fallback, legacy defaults, and optional keychain fallback behavior.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists with Xcode preferences that only contain `IDEProvisioningTeamByIdentifier`: current main only extracts `IDEProvisioningTeams`, and the consumer warning path matches the PR body.

## Solution Assessment

No, not as proposed yet. Parsing exported Xcode preferences while retaining the `plutil` fallback is the right narrow direction, but the changed smoke test should execute the real parser before the patch is merge-ready.
