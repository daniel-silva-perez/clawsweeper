## Summary

Keep open. Current main has partial safety nets for repetitive tool use, unknown tool loops, run retry limits, transcript repair, and parameter-validation guidance, but it does not implement the broader self-healing execution loop requested here: default deterministic validation-error circuit breaking, repair planning, verifier-gated retries, duplicate outbound-message suppression, or persisted per-turn repair state outside compaction. The related open bug is narrower symptom evidence rather than a canonical replacement for this architecture item.

## What This Changes

The issue proposes a finite self-healing execution mechanism for tool failures, including error classification, repair planning, per-turn state guards, verifier-gated retries, duplicate suppression, and compaction-safe runtime memory.

## Best Possible Solution

Keep this open as a core runtime design item and split it into scoped follow-ups: a default-safe deterministic validation/schema failure circuit breaker, per-run state stored outside transcript compaction, verifier-gated repair/retry hooks, provider/gateway/session repair actions that reuse existing failover and doctor infrastructure, and duplicate outbound-message suppression aligned with existing idempotency paths.

## Reproduction Assessment

Partially. The broader proposal is architectural, but the related open Feishu/Kimi report provides a concrete symptom path involving repeated `exec` validation failures and repeated outbound messages; current-main inspection shows the requested default deterministic breaker and repair state are still missing.

## Solution Assessment

Unclear. The requested direction is reasonable, but the best implementation should be staged through existing agent-runtime seams rather than a single broad self-healing subsystem that bypasses current loop detection, failover, transcript repair, and doctor logic.
