## Summary

Keep this issue open. Current main has partial protections for tool loops, unknown tools, bounded retries, model fallback, and compaction recovery, but it does not implement the proposal's broader self-healing execution loop with error classification, repair planning, verifier-gated retry, deterministic validation/schema circuit breaking, duplicate message suppression, and persisted per-turn repair state outside transcript compaction. The related closed bug #71369 is useful symptom context, not a canonical tracker for the remaining architecture work.

## Best Possible Solution

Keep this issue open as a core runtime/product design item. The best path is to split it into scoped follow-ups: a default-safe deterministic validation/schema failure circuit breaker, per-run state guard outside transcript compaction, verifier-gated repair/retry hooks in the embedded runner and gateway surfaces, duplicate message/call suppression where idempotency is missing, and provider/gateway/session repair actions that reuse existing loop detection, model fallback, compaction, and doctor/fix infrastructure.

## Risks / Open Questions

- The proposal is broad and should be split before implementation so it reuses existing loopDetection, model fallback, compaction, and doctor/fix paths instead of creating parallel retry machinery.
- Default-enabling broader loop blocking could create false positives for legitimate polling or long-running tool workflows; current docs intentionally keep generic detection opt-in.
- Repair actions around providers, gateway restart, and session recovery need explicit verifier and side-effect boundaries so self-healing does not mask unsafe or non-retryable failures.
