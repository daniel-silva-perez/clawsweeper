## Summary

Keep this issue open. Slack multi-account Socket Mode is a supported in-repo path, and current main still contains the account-scoped startup, event filtering, and DM routing surfaces implicated by the report, but I found no source, test, changelog, or history evidence that the specific secondary-workspace inbound DM failure has been fixed or superseded. The follow-up comment on 2026-04-21 is especially useful because it shows the same secondary workspace works when isolated in a single-workspace OpenClaw instance, pointing back to multi-account routing/account resolution rather than Slack app setup alone.

## Best Possible Solution

Keep the issue open. The best next step is a focused Slack plugin investigation: add a regression harness or live diagnostic path for two configured Slack accounts in Socket Mode, assert that a secondary-account `message.im` event reaches `monitorSlackProvider`, survives `shouldDropMismatchedSlackEvent`, and routes with the secondary `accountId` plus workspace-specific Slack user id. Then patch the specific drop/startup/routing branch that fails.
