## Summary

Keep this PR open. Current main still lacks a heartbeat-specific skill allowlist: the strict heartbeat schema, exported heartbeat config type, runner reply options, config inventory, and canonical docs do not expose `allowSkills`. The PR is a focused implementation candidate, but it needs a rebase/docs alignment pass because current main moved the agent heartbeat reference to `docs/gateway/config-agents.md`.

## What This Changes

The PR adds `agents.defaults.heartbeat.allowSkills` and `agents.list[].heartbeat.allowSkills` to config schema/types/help/generated metadata/docs/tests and threads the merged heartbeat value into heartbeat reply runs as `skillFilter`.

## Best Possible Solution

Land this PR or an equivalent small additive change after rebasing to current main, documenting `allowSkills` on the current canonical agent heartbeat config page, regenerating/checking schema metadata, and preserving unset versus empty-array behavior through the existing `skillFilter` merge seam.

## Reproduction Assessment

Yes. On current main, a config containing `agents.defaults.heartbeat.allowSkills` is outside strict `HeartbeatSchema`, and heartbeat reply options never include `skillFilter` even though `getReplyFromConfig` already supports that seam.

## Solution Assessment

The code direction is the narrow maintainable fix: reuse the existing `skillFilter` intersection behavior instead of adding a new skills loader path. As submitted against current main, it is incomplete until the docs/rebase alignment issue is fixed.
