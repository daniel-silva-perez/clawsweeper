## Summary

Keep open. Current main still formats the per-response `/usage` footer from turn token/cache/cost/session data only; the existing `/status` path already loads provider quota windows, and the GitHub Copilot plugin already exposes the requested Premium/Chat windows. The direct implementation PR is closed unmerged, while the broader open footer PR overlaps related plumbing but does not currently claim this specific provider-quota footer request.

## What This Changes

The issue requests adding GitHub Copilot Premium/Chat quota-window percentages to the per-response `/usage` footer when `/usage tokens` or `/usage full` is enabled.

## Best Possible Solution

Implement a generic reply-footer enhancement that appends a compact formatUsageWindowSummary result for the active provider's usage windows when `/usage tokens` or `/usage full` is enabled, while keeping provider-specific auth and fetching inside plugins. Add reply-runner regression coverage, update token-use docs, and include a user-facing changelog entry credited to the reporter.

Yes. A high-confidence code-level reproduction is to enable responseUsage, run the reply path with mocked active provider usage windows, and observe that current main only appends token/cache/cost/session text while the status path can already format provider windows.

## Solution Assessment

Yes. The narrow maintainable solution is a generic provider-window append in the reply footer path, not GitHub Copilot-specific API calls in core; it should reuse the existing provider usage loader/formatter and plugin-owned Copilot usage fetcher.
