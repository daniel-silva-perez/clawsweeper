## Summary

Keep this issue open. Current main still has global Voice Call settings for inbound greeting, response agent/model/system prompt, realtime consult behavior, and TTS voice selection, while provider webhook events only capture the dialed `to` number without resolving a per-number route. The provided discussion also includes a maintainer comment from 2026-04-25 explicitly keeping this open, and #59245 is broader outbound task-call work rather than a duplicate for inbound per-number persona routing.

## Best Possible Solution

Keep this issue open. The likely implementation belongs in `extensions/voice-call`: add a typed and documented per-number route table to config, manifest metadata, UI hints, and docs; normalize provider `to` values at call creation; store a stable route id or effective config on the call; and make inbound greeting, response agent/model/system prompt, realtime consult settings, and TTS voice selection use that effective config with global fallback. Add focused tests for route matching, fallback behavior, provider number normalization, and stable per-call routing. Keep #59245 separate because it tracks broader outbound task-call behavior.
