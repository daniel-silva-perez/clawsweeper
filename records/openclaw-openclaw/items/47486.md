## Summary

Keep open. Current main still has no `sessions_broadcast` implementation, and the PR remains a useful but non-merge-ready implementation: the author addressed the initial fire-and-forget/messaging-profile comments in the follow-up commit, but current-main review still shows invalid gateway provenance, an undeclared TypeBox import, and missing policy coverage for a new cross-session injection primitive.

## What This Changes

The PR adds a core `sessions_broadcast` agent tool, registers it in the OpenClaw tool catalog and tool factory, and adds unit tests for broadcasting system-event messages to active sessions with optional channel filtering.

## Best Possible Solution

If maintainers want this capability in core, land it as a policy-aware sibling of `sessions_send`: use the declared `typebox` dependency, reuse or formally extend gateway input provenance, annotate received content as non-user inter-session/system data, add matching deny/mutation coverage across subagents, Gateway HTTP, and realtime direct-tool surfaces, and document the broadcast boundary versus narrower admin notification or peer-delivery follow-ups such as #53228.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: the PR sends `inputProvenance.kind: "broadcast"`, while current main's gateway schema only accepts `external_user`, `inter_session`, and `internal_system`; the policy gap is visible by comparing the new default registration with existing `sessions_send` deny surfaces.

## Solution Assessment

No. The PR is a useful implementation sketch, but it is not the best mergeable solution because it bypasses current dependency, gateway provenance, and restricted-tool contracts for cross-session injection.
