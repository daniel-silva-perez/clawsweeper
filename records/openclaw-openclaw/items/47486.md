## Summary

Keep open. Current main still does not implement `sessions_broadcast`, so the contribution is not obsolete, and the follow-up commit addressed the first automated review's delivery-tracking and profile comments. The branch is not merge-ready because it adds a new core cross-session injection primitive with an undeclared TypeBox import, an invalid gateway provenance kind, and missing deny/mutation coverage for the same restricted surfaces that already gate `sessions_send`.

## What This Changes

The PR adds a core `sessions_broadcast` agent tool, registers it in the tool catalog and default OpenClaw tool factory, and adds unit coverage for broadcasting messages to active sessions with optional channel filtering.

## Best Possible Solution

If maintainers want this capability in core, land it as a policy-aware sibling of `sessions_send`: import from `typebox`, use or extend the gateway provenance contract deliberately, annotate received content as non-user inter-session or internal-system data, add matching restricted-surface and mutation-policy coverage, and document how this differs from narrower admin notification work such as #53228.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: the PR sends `inputProvenance.kind: "broadcast"` while current main's gateway schema only accepts `external_user`, `inter_session`, and `internal_system`; the policy gap is visible by comparing the new default registration with existing `sessions_send` deny surfaces.

## Solution Assessment

No. The proposed branch is a useful implementation sketch, but it is not the best mergeable solution until it follows the repository's declared dependency, gateway provenance, cross-session trust, and restricted-tool policy contracts.
