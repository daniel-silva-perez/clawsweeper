## Summary

Keep open. Current main does not implement `sessions_broadcast`, so the PR is not obsolete, but the proposed core cross-session injection tool is not merge-ready because it violates current dependency/provenance contracts and bypasses comparable restricted-tool policy surfaces.

## What This Changes

The PR adds a core `sessions_broadcast` agent tool, registers it in the catalog and default OpenClaw tool factory, and adds unit tests for broadcasting messages to active sessions with optional channel filtering.

## Best Possible Solution

If maintainers want this capability in core, land it as a policy-aware sibling of `sessions_send`: use the declared `typebox` package, use or deliberately extend the Gateway provenance contract, mark received content as non-user inter-session or internal-system data, add matching subagent/Gateway HTTP/realtime restrictions and regression tests, document the trust boundary, and rebase onto current main. Otherwise, keep the broader broadcast work out of core and pursue the narrower admin-scoped notification design tracked separately.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: the PR sends `inputProvenance.kind: "broadcast"` through the validated `agent` Gateway method while current main only accepts three provenance kinds, and the policy gap is visible by comparing the new default registration with the existing `sessions_send` deny surfaces.

## Solution Assessment

No. The branch is a useful implementation sketch, but it is not the best mergeable solution until it follows the repository's declared dependency, Gateway provenance, cross-session trust, and restricted-tool policy contracts.
