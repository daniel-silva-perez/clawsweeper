## Summary

Keep this PR open. Current main still does not implement Telegram forum topic deletion through the message action surface, and the PR is a plausible implementation candidate for a bundled Telegram plugin capability. It should not be closed as implemented or duplicate cleanup: the related umbrella #11414 is still open, and this PR targets one concrete missing part of that work. It is also not merge-ready as-is because the review discussion still has a security-relevant parsing concern for destructive IDs and the patch touches shared parser/tool paths beyond Telegram.

## Best Possible Solution

Keep the PR open as useful implementation work. The best path is to rebase it onto current main, keep the behavior owned by the Telegram plugin, add `topic-delete` to the shared action contract and outbound target spec if maintainers want the alias, implement `deleteForumTopicTelegram` plus a runtime `deleteForumTopic` branch, document the user-facing action, and add focused tests for message delete vs topic delete routing, gating, cfg pass-through, and decimal-only destructive ID validation. Keep #11414 as the broader umbrella for remaining Telegram forum topic lifecycle work.

## Risks / Open Questions

- Security review: forum topic deletion is destructive. A rebased patch should enforce explicit `topicId`/`threadId` and decimal-only integer parsing before calling Telegram, and should not silently fall back from invalid message-delete input to topic deletion.
- The PR changes shared parser/tool surfaces in addition to Telegram plugin files, so maintainers should verify no non-Telegram tool callers get unintended stricter or looser number parsing.
- The PR is stale against the current plugin-owned layout and still references older shared tool/action paths in the provided diff, so it needs a clean rebase/port before normal merge evaluation.
