## Summary

Current main still has the reported exec-approval identity mismatch: executable resolution computes both `resolvedPath` and `resolvedRealPath`, allowlist path matching still checks path-shaped patterns against `resolvedPath` only, and approval-backed execution pins to `resolvedRealPath` when available. The issue also has open PR #73543 with closing syntax, so it should remain open until that PR or an equivalent fix lands.

## Best Possible Solution

Keep this issue open while maintainers review #73543. The best fix is either to land that PR with a clearly scoped matcher fix and follow-up tracking for remaining realpath-trust surfaces, or expand it into a coherent executable-identity model across allowlist matching, safeBins, skill auto-allow, Allow Always persistence, audit metadata, and diagnostics, with regression tests for realpath allowlisting through symlinks and symlink target drift.
