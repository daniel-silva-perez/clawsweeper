## Summary

Current main still appears to have the reported exec approval asymmetry: executable resolution records both `resolvedPath` and `resolvedRealPath`, allowlist path matching still compares path-shaped patterns only against `resolvedPath`, and approval-backed execution pins to `resolvedRealPath` when present. The provided GitHub context also shows an open, unmerged PR (#73543) with closing syntax for this issue, so the issue should remain open for that implementation candidate rather than be closed now.

## Best Possible Solution

Review and either land #73543 or replace it with an equivalent fix that makes approval matching, durable allow-always persistence/audit metadata, safe-bin or skill trust where applicable, and execution use a coherent executable identity. Add focused regression tests for real-path allowlist matching through a symlink and for symlink-target drift.
