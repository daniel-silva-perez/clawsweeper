## Summary

Keep open. Current main still has the reported exec-approval identity mismatch, and open PR #73543 already references this issue with closing syntax, so the issue should stay open until that PR or an equivalent fix lands.

## What This Changes

Exec approval matching, persistence, and related trust checks should use a consistent executable identity when a command resolves through symlinks.

## Best Possible Solution

Land #73543 or an equivalent focused fix after security-aware review, with regression coverage for realpath allowlist matching through symlinks and symlink target drift. If the merged change intentionally handles only the matcher, split explicit follow-up for persistence, safe-bin trust, skill auto-allow, and audit metadata so the executable identity model is consistent.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a resolution with `resolvedPath` set to `/opt/homebrew/bin/rg` and `resolvedRealPath` set to `/opt/homebrew/Cellar/ripgrep/14.1.1/bin/rg`: current `matchAllowlist()` checks only the symlink path while approved execution later pins the realpath.

## Solution Assessment

Unclear. Consulting the realpath in the matcher is the narrow primary fix, but maintainers should explicitly choose whether symlink-path allowlist entries remain valid after target changes or whether durable approvals should canonicalize to the executed realpath.
