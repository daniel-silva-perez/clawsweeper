## Summary

Keep open. Current main still has the reported exec-approval identity mismatch: allowlist matching uses the PATH-resolved symlink path while approved execution pins to the realpath. The linked open PR #73543 uses closing syntax for this issue, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue asks to canonicalize exec approval identity so allowlist/trust checks, durable approval persistence, metadata, and execution use the same executable identity instead of mixing symlink paths with realpaths.

## Best Possible Solution

Review and land #73543 or an equivalent focused fix with regression coverage for real-path allowlist matching through a symlink and symlink target drift. Before closing, either extend the fix to the adjacent identity surfaces named in the issue or leave a narrower follow-up that clearly tracks any remaining canonical-identity gaps.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a resolution with `resolvedPath` set to `/opt/homebrew/bin/rg` and `resolvedRealPath` set to `/opt/homebrew/Cellar/ripgrep/14.1.1/bin/rg`: current `matchAllowlist()` misses a real-path pattern while approved execution later pins to the realpath.

## Solution Assessment

Unclear until the open PR is reviewed. Matching the realpath in `matchAllowlist()` is the narrow fix for the primary allowlist miss, but the broader persistence and trust surfaces need an explicit maintainer scope decision before this issue is closed.

## Security Review

Status: needs_attention

Summary: This issue is security-sensitive because current main can approve one executable path identity while executing another.
