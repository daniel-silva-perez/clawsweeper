## Summary

Keep open. Current main still has the reported exec-approval identity mismatch, and open PR #73543 already references this issue with closing syntax, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue requests a canonical executable identity for exec approval checks so allowlist, trust, persistence, and approval-backed execution do not diverge when binaries are reached through symlinks.

## Best Possible Solution

Land #73543 or an equivalent focused fix after security-aware review, with regression coverage for realpath allowlist matching through symlinks and symlink target drift. If the merged fix is matcher-only, keep or split follow-up tracking for persistence, safe-bin trust, skill auto-allow, and audit metadata so the approval identity model is explicit.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a resolution with `resolvedPath` set to `/opt/homebrew/bin/rg` and `resolvedRealPath` set to `/opt/homebrew/Cellar/ripgrep/14.1.1/bin/rg`: current `matchAllowlist()` checks only the symlink path while approval-backed execution later pins the realpath.

## Solution Assessment

Unclear. Matching `resolvedRealPath` in the allowlist matcher is the narrow primary fix, but the issue is broader than one matcher and needs maintainer scoping around persistence, safe-bin trust, skill auto-allow, and audit fields.

## Security Review

Status: needs_attention

Summary: The issue is security-sensitive because current main can approve one executable path identity while executing another.

Concerns:

- **[medium] Approval identity can drift across symlink target changes:** `src/infra/exec-command-resolution.ts:376`
  - body: Path-shaped allowlist matching checks `resolvedPath`, but approval-backed execution uses `resolvedRealPath` when available. An approval tied to a symlink path can therefore continue matching after the symlink target changes while a different real binary is executed.
