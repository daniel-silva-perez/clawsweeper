## Summary

Keep open. Current main still parses provider-level `token`/`oauth` auth overrides but does not apply them when walking auth-profile candidates, so the PR targets a real remaining behavior gap. The submitted patch should not merge as-is because its no-match fallback can still select mismatched profiles and the availability check remains out of sync with the new resolver behavior.

## What This Changes

The PR changes `src/agents/model-auth.ts` so `resolveApiKeyForProvider` filters auth-profile candidates by provider-level `auth: "token"` or `auth: "oauth"` before resolving credentials.

## Best Possible Solution

Centralize provider-level token/oauth profile eligibility, use it from both `resolveApiKeyForProvider` and `hasAvailableAuthForProvider`, avoid silent fallback to mismatched profile types, and add focused regressions for mixed OAuth/token stores, no matching profile type, and unresolved matching profiles.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Do not fall back to mismatched auth profiles:** `src/agents/model-auth.ts:357`
  - body: With `models.providers.<id>.auth: "token"` and only OAuth profiles available, `filteredOrder` is empty so `finalOrder` reuses the original `order` and can select OAuth anyway. That preserves the auth-routing bug this PR is trying to fix; the override should fail or report no available matching profile instead of silently trying the mismatched type.
  - confidence: 0.91
- **[P2] Keep availability checks aligned with filtered resolution:** `src/agents/model-auth.ts:350-357`
  - body: The new filtering only applies in `resolveApiKeyForProvider`. If a matching token profile is present but cannot resolve, `hasAvailableAuthForProvider` can still return true by seeing an OAuth profile that runtime resolution will skip, producing false provider readiness. Share the same auth-mode eligibility helper with the availability path.
  - confidence: 0.82
