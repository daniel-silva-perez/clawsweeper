## Summary

Keep open. Current `main` still routes bare Feishu `oc_...` outbound targets as `direct`, while the related OpenChat inbound correction in #52342 is still open and does not make this report obsolete. This is a valid coordination/follow-up item rather than a stale cleanup close.

## What This Changes

The issue asks OpenClaw to use one Feishu OpenChat session for `oc_...` chats across outbound sends, cron delivery, and inbound webhook messages instead of splitting direct and group session keys.

## Best Possible Solution

Centralize Feishu chat-id classification in the Feishu plugin so `oc_...` is treated as a group/OpenChat everywhere it matters: inbound bot routing, mention-forward logic, outbound `message send`/cron session route construction, and session-key regression tests. Existing duplicate session files, if repaired automatically, should go through an explicit doctor/fix-style path or a narrowly reviewed migration rather than an implicit core startup merge.

## Reproduction Assessment

Unclear for the full duplicate-session scenario on current `main`, because #52342's inbound `oc_` p2p correction is not merged yet. The outbound mismatch is high-confidence by source inspection: bare `oc_...` Feishu targets still resolve to `agent:<agent>:feishu:direct:<oc_id>` unless the caller explicitly prefixes `chat:` or `group:`.

## Solution Assessment

Partly yes. Sharing one Feishu-owned `oc_` classifier across inbound and outbound is the narrow maintainable fix, but the broader proposed startup reconciliation is not clearly the best first step without maintainer review of state-repair policy.
