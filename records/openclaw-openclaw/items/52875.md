## Summary

Keep open. Current main still rejects missing target sessions instead of bootstrapping them, lacks a registered `sessions.start`/`sessions_start` surface, and does not return the discussed delivery metadata on timeout. Open PR #52921 is still the linked implementation candidate, so closing before maintainer review or merge would strand the reported regression.

## What This Changes

The issue requests a recovery path for agent-to-agent `sessions_send` when a configured target agent has no active session, including start/prewarm behavior and clearer timeout or delivery acknowledgement state.

## Best Possible Solution

Review PR #52921 as the current implementation candidate. If it is correct and safe, land it and let the linked report close through GitHub; otherwise keep this as the canonical report and implement a focused replacement covering idempotent session start, `sessions_send` auto-bootstrap for known agent configs, structured not-found errors, delivery metadata on accepted/timeout paths, regression tests, and docs updates.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is calling `sessions.send` with a session key that has no stored `sessionId`: current `handleSessionSend` returns `INVALID_REQUEST` immediately, and the agent label path fails at `sessions.resolve` before any start flow can run.

## Solution Assessment

Unclear until PR #52921 is reviewed. The intended direction is maintainable if it is implemented as an idempotent start/auto-bootstrap path that preserves existing session visibility, permissions, and protocol compatibility; active/waiting/read receipts should be scoped separately if they complicate the fix.
