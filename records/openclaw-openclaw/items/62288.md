## Summary

Keep open. Current main has useful partial fixes for the browser default and Chrome MCP error wording, but it still does not meet the maintainer-stated canonical acceptance bar for existing-session attach diagnostics and target selection.

## What This Changes

Request to make the browser `user` existing-session attach path less brittle, prefer the managed `openclaw` lane by default, and improve fallback/status/doctor diagnostics for Chrome MCP attach failures.

## Best Possible Solution

Keep this as the canonical browser existing-session reliability issue until the browser plugin reports structured per-profile health and attach reason codes, avoids browser-internal targets for implicit selection, preserves explicit `profile="user"` behavior without silent managed fallback, and covers the behavior with targeted tests and user-facing docs where behavior changes.

## Reproduction Assessment

Yes for the remaining repo-side gaps: a unit test can stub an existing-session status failure and observe `/doctor` returning an error instead of a multi-profile health report, and can list `chrome://inspect` before a real tab to show implicit selection still chooses the first page. The exact macOS/Windows DevToolsActivePort failure still needs live Chrome MCP verification.

## Solution Assessment

No. Current main is a partial improvement, but the narrow maintainable solution is still a browser-plugin diagnostics and target-selection change, not closing this as fixed.

## Repair Work Prompt

Fix the canonical browser existing-session attach/fallback diagnostics issue. On current main, keep the managed `openclaw` profile as the default and keep explicit `profile="user"` from falling back to sandbox or managed browser. Add structured existing-session health/failure information to browser `/`, `/doctor`, and `/profiles` so callers can see managed browser healthy while `user`/existing-session is unhealthy, including actionable reason codes where possible for DevToolsActivePort/Could-not-connect, `npx`/spawn failures, no tabs, and approval/attach readiness failures. Update implicit target selection so existing-session/default tab selection skips browser-internal/devtools pages such as `chrome://inspect`, `devtools://`, `chrome-extension://`, and similar internal URLs while still listing them and allowing explicit targetId/tabId selection. Add focused regression tests for the status/doctor/profile diagnostics and internal-tab selection behavior. Do not change core plugin APIs, do not add dependencies, do not solve managed-profile persistence, and do not make `profile="user"` silently use `openclaw`. Update docs only for user-visible diagnostic/selection behavior.
