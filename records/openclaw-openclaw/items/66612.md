## Summary

Keep open. Current main and the latest release still expose only one memory-flush write target and reject `SESSION_HANDOFF.md` through the exact-path write wrapper. The default bundled prompt now targets only the daily memory file, so this is no longer a default-prompt mismatch, but the configurable-prompt/additional-write-target bug remains valid; PR #66765 is closed unmerged, #74198 was closed as a duplicate of this issue, and #63664/#68995 cover adjacent but broader write-policy variants.

## What This Changes

Allow pre-compaction memory flush writes to `SESSION_HANDOFF.md` or another explicitly configured safe path instead of only `memory/YYYY-MM-DD.md`.

## Best Possible Solution

Keep this issue open as the canonical `SESSION_HANDOFF.md`/additional-target tracker. The maintainable end state is a typed memory-flush write-target contract in config or `MemoryFlushPlan`, with explicit validated overwrite targets, the daily memory file kept append-only, traversal and sibling-prefix rejection, docs/schema updates, and regression tests across the plan, runner, tool wrapper, and config surfaces.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: configure the memory flush prompt to request `SESSION_HANDOFF.md`, trigger a memory run, and the current write wrapper will reject that path because the only forwarded allowed target is `memory/YYYY-MM-DD.md`.

## Solution Assessment

No for current main. The best fix is not to infer paths from prompt text alone, but to add an explicit allowlist/target contract that the memory plugin, config schema, embedded runner, and write wrapper can validate and enforce consistently.
