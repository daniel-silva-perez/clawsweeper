## Summary

Current main still keeps memory-flush writes scoped to one append-only daily memory file. A configured flush prompt that asks for SESSION_HANDOFF.md would still be rejected by the write wrapper. The bundled default prompt/docs now describe only memory/YYYY-MM-DD.md, but that does not resolve the configurable-prompt case reported here. The prior implementation PR #66765 is closed and unmerged, and related open issues #63664 and #68995 overlap without superseding this specific handoff-file request.

## Best Possible Solution

Keep this open and fix the memory-flush write contract deliberately: add an explicit validated additional-write-target mechanism to the memory flush plan/config or plugin contract; thread it through the embedded runner and tool creation; allow full overwrite for SESSION_HANDOFF.md only when explicitly configured while preserving append-only behavior for the daily memory file; reject unrelated paths and traversal attempts; and add regression tests for allowed handoff writes, daily-memory append behavior, unrelated-path rejection, and path escapes.

## Risks / Open Questions

- Allowing arbitrary paths inferred from free-form prompt text would widen flush-time write authority; the fix should use explicit, validated targets with workspace-root, traversal, and sandbox checks.
- The bundled default prompt no longer mentions SESSION_HANDOFF.md, so the remaining bug should be framed as a configurable additional-write-target contract gap rather than a current default-prompt mismatch.
- Related open issues #63664 and #68995 touch the same write-policy area, so the final design should coordinate the memory-flush write contract instead of only special-casing one filename.
