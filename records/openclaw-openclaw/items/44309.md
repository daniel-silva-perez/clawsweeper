## Summary

Keep open. Current main still lacks an explicit one-way dispatch/handoff mode for `sessions_send`: the tool schema exposes only target/message/timeout fields, fire-and-forget still starts the A2A follow-up path, and the A2A helper still performs requester/target reply-back turns plus announce delivery. Related open items are adjacent lifecycle work, not a canonical replacement for this per-call dispatch semantic.

## Best Possible Solution

Add a backward-compatible explicit delivery intent for `sessions_send`, preferably `mode: "conversation" | "dispatch"` with current behavior as the default. In dispatch mode, enqueue the message into the target session through the existing internal-channel/inter-session provenance path, return a minimal delivery acknowledgement, and skip `runSessionsSendA2AFlow` so there is no requester-side reply reinjection and no announce unless a separate explicit option is later designed. Update focused tests and the public session tool docs.

## Repair Work Prompt

Implement a backward-compatible one-way dispatch mode for `sessions_send` to address this issue. Use a minimal explicit delivery intent such as `mode: "conversation" | "dispatch"`, with existing conversation/announce behavior as the default. In `mode: "dispatch"`, enqueue/persist the message into the target session using the existing internal channel, nested lane, and inter-session provenance path, then return a minimal accepted/delivery acknowledgement. Do not call `runSessionsSendA2AFlow` for dispatch mode, do not inject the target reply back into the requester session, and do not run the announce step. Define and document `timeoutSeconds` behavior conservatively for dispatch; prefer delivery acknowledgement over waiting for a target model reply. Add focused tests proving default behavior is unchanged and dispatch skips reply-back/announce for fire-and-forget and any supported acknowledgement path. Update public docs in `docs/concepts/session-tool.md`; add a user-facing changelog entry with reporter credit if the active changelog section expects one. Do not solve #39476, #33478, #8299, #68622, or #46660 in this PR except to avoid conflicts with nearby touched areas.
