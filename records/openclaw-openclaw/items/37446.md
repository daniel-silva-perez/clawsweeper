## Summary

Keep this issue open. Current main still lacks a stable caller-supplied or deterministic recovery identity for `sessions_spawn`; native, ACP, and restart-orphan recovery paths still mint fresh child run/session identities, so a lost acknowledgement can still create an independent retry instead of converging on the original run.

## Best Possible Solution

Add an additive core recovery contract for `sessions_spawn`: accept a caller-supplied or deterministic requester-scoped recovery key, persist a mapping from that key to the child session/run, and make native, ACP, and restart-recovery paths return or reuse the existing child run after timeout or lost acknowledgement. Add regression tests for native and ACP retry convergence, document the contract in the sub-agent docs, and keep the linked ClawHub skill as operational guidance for service-specific idempotency rather than treating it as the core fix.

## Risks / Open Questions

- A parent retry after a lost `sessions_spawn` acknowledgement can still create a second child run with a fresh child session key and fresh gateway idempotency key.
- External side effects performed by the first child run, such as comments, PRs, charges, or database writes, remain vulnerable unless every downstream tool/service implements its own idempotency check.
- A core fix needs to be additive and backwards-compatible so existing third-party plugin and ACP spawn behavior keeps working.
