## Summary

Keep this issue open. Current main still lacks a stable caller-supplied or deterministic recovery identity for `sessions_spawn`; both native and ACP spawn paths create fresh child session keys and fresh gateway `agent` idempotency keys for each spawn attempt. The linked ClawHub multi-agent deployment skill is useful operational guidance, but it cannot make core `sessions_spawn` retries converge on an existing child run after a lost acknowledgement.

## Best Possible Solution

Add an additive `sessions_spawn` recovery contract in OpenClaw core: accept a caller-supplied or deterministic recovery/idempotency key, persist a requester-session scoped mapping from that key to the child session/run, and make both native and ACP spawn paths return or reuse the existing child run after timeout or lost acknowledgement. Add regression coverage for native and ACP retries, update the sub-agent docs, and keep the linked ClawHub skill as operational guidance rather than treating it as the fix.

## Risks / Open Questions

- A retry after a lost `sessions_spawn` acknowledgement can still create an independent child run because the retry path has no stable key to map back to the first child session/run.
- The ClawHub skill linked in the discussion can reduce operational risk with external checks and service-specific idempotency, but it cannot enforce core `sessions_spawn` exactly-once behavior without an OpenClaw recovery contract.
