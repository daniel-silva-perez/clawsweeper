## Summary

Keep this issue open. Current main still has gateway-level idempotency when the same `agent` idempotency key is reused, but `sessions_spawn` itself does not accept or derive a stable recovery identity, and both native and ACP spawn paths still mint fresh child session keys plus fresh gateway idempotency keys on each invocation.

## Best Possible Solution

Add an additive `sessions_spawn` recovery contract: accept a caller-supplied or deterministic requester-scoped recovery key, persist a mapping from that key to the child session/run, and make native, ACP, and restart-recovery paths return or reuse the existing child run after timeout or lost acknowledgement. Add regression tests for native and ACP retry convergence, document the contract in the sub-agent docs, and keep the linked ClawHub skill as operational guidance for service-specific idempotency.

## Risks / Open Questions

- A parent retry after a lost `sessions_spawn` acknowledgement can still create a second child session/run with a fresh gateway idempotency key.
- Downstream external effects from the first child run, such as comments, PRs, charges, or database writes, still depend on service-specific idempotency rather than a core sub-agent recovery contract.
- A fix needs to be additive and backwards-compatible for native sub-agents, ACP spawns, and third-party plugin callers.
