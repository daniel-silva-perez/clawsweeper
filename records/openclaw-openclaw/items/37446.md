## Summary

Keep open. Current main still lacks a stable `sessions_spawn` recovery identity: gateway `agent` calls dedupe only when the same idempotency key is reused, while native and ACP sub-agent spawn paths still create fresh child session keys plus fresh gateway idempotency keys for each invocation. The linked ClawHub skill is useful operational guidance, but it does not remove the core retry gap.

## What This Changes

The issue requests an idempotent recovery identity for sub-agent POST/retry flows so lost acknowledgements do not create duplicate external side effects.

## Best Possible Solution

Add an additive `sessions_spawn` recovery contract that accepts or derives a requester-scoped recovery key, persists the key-to-child-run mapping, and makes native and ACP retry paths return the original accepted child run after timeout or lost acknowledgement. Keep service-specific idempotency guidance in ClawHub/docs, but make OpenClaw core responsible for not starting duplicate child sessions for the same recovered spawn request.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: invoke `sessions_spawn`, lose the parent acknowledgement, then invoke the same tool call again; current main has no stable recovery parameter and both spawn implementations generate new child session and idempotency IDs.

## Solution Assessment

Yes, an additive `sessions_spawn` recovery identity is the narrowest maintainable fix. Gateway-level idempotency and the ClawHub deployment skill are useful but insufficient because they do not make repeated parent spawn calls converge before a second child run is created.
