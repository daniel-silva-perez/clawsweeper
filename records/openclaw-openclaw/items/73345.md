## Summary

Keep open. The PR addresses a real gap because current main still does not accept or forward a `/hooks/agent` thread value, but the proposed patch is not merge-ready: once `thread` becomes part of the delivery destination, the replay/idempotency scope also needs to include it so a same-key request to a different thread is not incorrectly deduped.

## What This Changes

The PR adds an optional `/hooks/agent` `thread` payload field, maps it to `CronJob.delivery.threadId`, and adds gateway hook coverage for threaded and non-threaded deliveries.

## Best Possible Solution

Revise the PR so `thread` is treated as part of the delivery destination everywhere the hook request is scoped: normalize it, forward it to cron delivery, include it in replay dedupe scope, add a same-idempotency-key/different-thread regression, and update the public webhook docs before merge.
