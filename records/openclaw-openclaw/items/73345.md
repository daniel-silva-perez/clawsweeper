## Summary

Keep open. The PR is useful and narrow, but the current diff only partially treats `thread` as a delivery destination: replay dedupe would still ignore it, numeric thread IDs would be dropped, and the public docs/changelog are not updated.

## What This Changes

The PR adds optional `thread` parsing to `/hooks/agent`, forwards it to `CronJob.delivery.threadId`, and adds gateway hook coverage for payloads with and without `thread`.

## Best Possible Solution

Make `thread` a first-class optional `/hooks/agent` delivery destination across normalization, cron delivery handoff, replay scope, regression tests, public docs, and changelog before merge.

## Reproduction Assessment

Yes. From the PR diff and current replay code, two `/hooks/agent` requests with the same idempotency key and different `thread` values would hash the same replay scope and can return the first runId instead of dispatching to the new thread.

## Solution Assessment

No. Parsing and forwarding `thread` is the right direction, but the maintainable fix must also scope replay by normalized thread, preserve numeric thread IDs, and update the user-facing docs/changelog.

  - body: After this PR makes `thread` part of the delivery target, `/hooks/agent` replay dedupe still hashes a `dispatchScope` that omits the thread value.   - confidence: 0.84

  - confidence: 0.88
