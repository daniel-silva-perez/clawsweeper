## Summary

Keep open. Current main still lacks `/hooks/agent` thread propagation, and the PR is a useful narrow adapter change, but it is not merge-ready because the new delivery target dimension is not included in the idempotency replay scope.

## What This Changes

The PR adds an optional `thread` field to `/hooks/agent` payload normalization, forwards it to `CronJob.delivery.threadId`, and adds gateway hook regression coverage.

## Best Possible Solution

Land the gateway hook adapter with `thread` treated as part of the delivery destination, included in replay scoping and public docs, while leaving the existing cron delivery contract intact.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Scope idempotency replay by thread:** `src/gateway/server/hooks.ts:69`
  - body: After this line makes `thread` part of the delivery target, two `/hooks/agent` requests that reuse an idempotency key but change only `thread` will still share the same replay cache key because `dispatchScope` does not include it. The second request can return the first runId and skip delivery to the new topic/thread; add the normalized thread value to the replay scope and cover that case.
