## Summary

Keep open. The requested behavior is valid and current main does not yet accept a `/hooks/agent` `thread`, but the PR is incomplete because it makes `thread` part of the delivery target without adding it to the `/hooks/agent` replay dedupe scope, so a reused idempotency key can suppress delivery to a different thread.

## What This Changes

The PR adds optional `thread` parsing to `/hooks/agent`, forwards it as `CronJob.delivery.threadId`, and adds gateway hook coverage for payloads with and without `thread`.

## Best Possible Solution

Revise this PR so `thread` is treated as part of the hook delivery destination everywhere: normalize it, forward it to cron delivery, include it in `/hooks/agent` replay scope, cover same-idempotency-key/different-thread behavior, and document the public webhook field before merge.

## Reproduction Assessment

Yes. On the PR branch, send two `/hooks/agent` requests with the same idempotency key and identical payloads except `thread`; because replay scope omits `thread`, the second request can return the first runId instead of dispatching to the new thread.

## Solution Assessment

No. Parsing and forwarding `thread` is the right direction, but the maintainable fix must also scope replay dedupe by the normalized thread and update the public webhook documentation.
