## Summary

Keep open. Current main still lacks pending-history media inclusion for Feishu group chats, and the PR is pointed at the right owner path, but the branch should not merge as-is because its history-candidate filter misses media on its own base branch and its advertised media cap is enforced after potentially expensive downloads have already happened.

## What This Changes

The PR adds raw Feishu message content to fetched message metadata and prepends best-effort resolved media from recent pending group-history entries to the inbound Feishu media payload.

## Best Possible Solution

Rebase the change onto current main, keep the rawContent plumbing, resolve pending-history media through a helper that can identify Feishu media entries robustly on the rebased code, enforce the item/download cap before downloading rich post assets, and add targeted Feishu tests for a pre-mention image plus a multi-image post cap case.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Recognize raw pending media bodies:** `extensions/feishu/src/bot.ts:768-770`
  - body: On this branch, pending group history is recorded from ctx.content, and Feishu image/file/audio events still leave ctx.content as raw JSON such as {"image_key":"..."}. The new filter only matches rendered placeholders, so it skips the screenshots this PR is meant to attach.
  - confidence: 0.88
- **[P2] Cap post media before downloading:** `extensions/feishu/src/bot.ts:803-805`
  - body: The 10-item guard runs after resolveFeishuMediaList returns, but post messages download every embedded image/media before returning. A single rich pending post can still download far more than 10 files, defeating the stated latency and cost bound.
  - confidence: 0.9
