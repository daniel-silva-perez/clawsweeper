## Summary

Keep open. Current main still leaves the Feishu pending-history media gap, and the PR is useful source work, but the proposed patch is not merge-ready because the promised 10-item cap is applied after bulk post downloads and the history-media prepend would shift current-main audio transcription indexes if rebased in the same position.

## What This Changes

The PR preserves raw Feishu message body content and prepends best-effort media resolved from recent pending Feishu group-history entries to inbound agent media context.

## Best Possible Solution

Rebase the PR onto current main, keep the rawContent plumbing, merge pending-history media before audio preflight or recompute indexes afterward, enforce the media-item limit before downloading rich post assets, and add focused Feishu regressions for pre-mention images, multi-image post bounds, and audio plus history media ordering.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Cap rich history media before downloading:** `extensions/feishu/src/bot.ts:802-805`
  - body: The 10-item guard runs after resolveFeishuMediaList returns, but that helper downloads every embedded image/media in a rich post before returning. One pending post can still fetch far more than 10 assets, defeating the latency and cost bound this change promises.
  - confidence: 0.91
- **[P2] Keep audio indexes aligned after prepending history media:** `extensions/feishu/src/bot.ts:815-827`
  - body: On current main, audio preflight computes MediaTranscribedIndexes from the triggering-message media list before this history block would prepend older media. If a triggering audio message also has pending history media, the final MediaPaths order shifts and the transcribed index can point at the wrong item unless the merge happens before preflight or the index is recomputed.
  - confidence: 0.78
