## Summary

Keep open. Current main still lacks pending-history media attachment, and the PR is a useful Feishu plugin fix, but the proposed patch is not safe to merge because it enforces the 10-item cap only after resolving/downloading rich post assets and can leave MediaTranscribedIndexes pointing at the wrong media after prepending history media.

## What This Changes

The PR preserves raw Feishu message body content and prepends best-effort media resolved from recent pending Feishu group-history entries to the inbound agent media payload.

## Best Possible Solution

Land a rebased Feishu plugin fix that keeps fetched rawContent available, resolves only a bounded amount of pending-history media before building the final inbound context, and keeps audio transcript metadata aligned with the final MediaPaths order.

## Reproduction Assessment

Yes. A source-level reproduction is clear: use a mention-required Feishu group with a pending rich post containing more than 10 embedded media items before a bot mention, or a pending image plus a triggering audio message; the first path over-downloads before the cap and the second shifts MediaPaths after the audio index is computed.

## Solution Assessment

No, not yet. The rawContent plumbing and pending-history lookup are the right direction, but the fix should enforce the media cap before download and either merge history media before audio preflight or recompute MediaTranscribedIndexes after merging.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Cap rich history media before downloading:** `extensions/feishu/src/bot.ts:802-805`
  - body: The 10-item guard runs after resolveFeishuMediaList returns, but that helper downloads every embedded image/media in a rich post before returning. A single pending post can still fetch far more than 10 assets, defeating the latency and cost bound this change promises.
  - confidence: 0.91
- **[P2] Keep audio indexes aligned after prepending history media:** `extensions/feishu/src/bot.ts:815-827`
  - body: Audio preflight computes MediaTranscribedIndexes from the triggering-message mediaList before this block prepends history media and rebuilds mediaPayload. If the triggering message is audio and history media is prepended, the final MediaPaths order shifts and the transcribed index can point at the wrong item.
  - confidence: 0.78
