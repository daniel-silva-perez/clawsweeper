## Summary

Keep open. Current main still uses a 30-second TUI streaming watchdog fallback with only an internal `streamingWatchdogMs` handler option, while normal TUI startup, CLI options, config schema, slash commands, and docs expose no user-facing threshold or disable control. Related #69026 and #69978 remain open rather than merged replacements.

## What This Changes

Request to add a user-configurable TUI streaming watchdog timeout or disable control so long-reasoning models do not hit the hardcoded 30-second silence warning.

## Best Possible Solution

Add a validated user-facing TUI streaming watchdog control, plumb it through `TuiOptions` into `createEventHandlers`, document it on the public TUI/config pages, and cover default, configured threshold, disabled watchdog, and any chosen runtime command behavior with focused tests.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. The current handler arms a 30-second default watchdog when no override is passed, the normal TUI path passes no override, and existing fake-timer tests show the watchdog warning/reset behavior through the same handler path.

## Solution Assessment

Is this the best way to solve the issue? Unclear on the exact UX, but the best direction is a TUI-owned user-facing setting or command; changing unrelated provider/model timeouts would not address the client-side watchdog shown in current main.
