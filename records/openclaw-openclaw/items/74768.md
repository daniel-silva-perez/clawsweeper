## Summary

Keep open. Current main still ignores the deprecated `embeddedHarness` runtime policy in the shared resolver even though current types, schema, and doctor migration still accept that config shape, while this PR provides a focused compatibility change with targeted tests and no definite review finding.

## What This Changes

The PR updates agent runtime policy resolution to merge legacy `embeddedHarness` runtime/fallback values into missing `agentRuntime` fields and adds harness selection, `sessions_spawn`, and gateway startup plugin activation tests.

## Best Possible Solution

Land this PR or an equivalent central resolver compatibility fix after maintainer review, keeping explicit `agentRuntime` precedence, fill-missing legacy migration semantics, plugin disablement behavior, and the documented native Codex versus explicit ACP route split.

## Reproduction Assessment

Yes. A config using only `agents.defaults.embeddedHarness` or `agents.list[].embeddedHarness` is accepted by current types/schema and migrated by doctor, but current main's shared resolver ignores that field before harness selection and startup activation consume policy.

## Solution Assessment

Yes. Centralizing the compatibility behavior in `resolveAgentRuntimePolicy` is the narrowest maintainable fix because the relevant downstream consumers already route through that resolver; the safer alternative would be duplicating legacy handling in each consumer, which would be easier to drift.
