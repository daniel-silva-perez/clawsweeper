## Summary

Keep open for maintainer review. Current main still ignores legacy `embeddedHarness` in shared runtime-policy resolution, while this PR targets that live compatibility gap with a narrow resolver change and focused regression tests; I did not find a definite blocking defect in the proposed patch.

## What This Changes

The PR maps legacy `embeddedHarness.runtime` and `embeddedHarness.fallback` into shared agent runtime policy resolution, adds harness-selection and startup-plugin regression tests, and verifies `sessions_spawn` keeps configured Codex agents on the subagent path unless ACP is explicit.

## Best Possible Solution

Land this PR or an equivalent shared-resolver compatibility change after maintainer review, preserving explicit `agentRuntime` precedence, plugin enable/disable semantics, and the native Codex versus ACP routing boundary documented for agents.

## Reproduction Assessment

Yes. A config that only sets legacy `embeddedHarness: { runtime: "codex", fallback: "none" }` reproduces the current-main gap by inspection because shared policy resolution ignores `embeddedHarness`, and harness selection plus startup activation both depend on that resolver.

## Solution Assessment

Yes. Updating the shared resolver is the narrowest maintainable fix because the affected harness selection, runtime metadata/alias consumers, and startup activation already share that policy function; I did not identify a smaller safer patch.
