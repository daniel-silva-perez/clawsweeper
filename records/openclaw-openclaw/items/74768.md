## Summary

Keep open. Current main still ignores legacy `embeddedHarness` in shared runtime-policy resolution, while this PR targets that gap with a focused resolver change and regression tests; I found no definite blocking review finding, so the remaining action is normal maintainer review.

## What This Changes

The PR updates shared agent-runtime policy resolution to merge deprecated `embeddedHarness` fields into `agentRuntime`, with tests for harness selection, startup plugin activation, and `sessions_spawn` routing.

## Best Possible Solution

Land this PR or an equivalent shared-resolver compatibility change after maintainer review, preserving explicit `agentRuntime` field precedence, plugin enable/disable semantics, and the native Codex versus explicit ACP routing boundary.

## Reproduction Assessment

Yes. A current-main config that only sets legacy `embeddedHarness: { runtime: "codex", fallback: "none" }` reproduces by inspection because the shared resolver ignores `embeddedHarness`, while harness selection and startup activation both consume that resolver.

## Solution Assessment

Yes. Updating the shared resolver is the narrowest maintainable fix because the affected consumers already centralize on it, and the merge behavior matches the existing doctor migration semantics of filling missing `agentRuntime` fields from `embeddedHarness`.
