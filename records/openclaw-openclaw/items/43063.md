## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks the requested metered/provider billing JSON surface, and the PR remains a plausible direction, but the patch has blocking correctness issues and needs maintainer design review around provider-owned usage hooks and billing-data exposure.

## What This Changes

The PR adds an `auth.providerUsage` JSON object to `openclaw models status --json`, introduces a core OpenRouter metered billing fetcher with tests, and hoists provider usage fetching ahead of output rendering.

## Best Possible Solution

Define a small additive provider-usage JSON contract in core, extend the usage snapshot types only as needed, and keep OpenRouter-specific credential resolution, billing requests, and response parsing in `extensions/openrouter` via `resolveUsageAuth` and `fetchUsageSnapshot` or a backwards-compatible SDK extension if multi-key reporting requires it. Fetch metered billing only for output modes that render it, report partial/non-2xx failures accurately, add command and provider tests, and document the new JSON fields at `https://docs.openclaw.ai/cli/models`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Use the awaited OpenRouter usage type:** `src/commands/models/list.status-command.ts:346`
  - body: `ReturnType<typeof loadOpenRouterMeteredUsage>` is the Promise type. This variable stores the awaited snapshot and later reads `.kind`, `.account`, and `.keys`, so typechecking will fail; use `Awaited<ReturnType<...>>` or an explicit snapshot type.
  - confidence: 0.95
- **[P2] Gate OpenRouter billing fetches to JSON output:** `src/commands/models/list.status-command.ts:377-380`
  - body: This call runs for every non-plain `models status` invocation, but `openrouterUsage` is only rendered in the JSON payload. With an OpenRouter key configured, the default rich status path gains an unnecessary external billing request and timeout risk.
  - confidence: 0.91
- **[P2] Mark partial key fetch failures as errors:** `src/infra/provider-usage.openrouter.ts:272-281`
  - body: Key request failures are not reflected in provider status when any key succeeds, and non-2xx key responses are converted to `null` before this status logic. A revoked or timed-out key can be omitted while OpenRouter still reports `status: "ok"`.
  - confidence: 0.88
