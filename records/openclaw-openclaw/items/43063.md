## Summary

Keep this PR open. Current main still lacks the requested metered provider billing JSON surface, but the PR branch is not merge-ready: it introduces a type error in the status command, calls the wrong OpenRouter key endpoint, can report incomplete billing data as healthy, and needs maintainer approval for the new billing-data JSON contract.

## What This Changes

The PR adds `auth.providerUsage` JSON output for windowed provider usage and OpenRouter metered billing, introduces a core OpenRouter billing fetcher, and updates focused models status tests.

## Best Possible Solution

Extend the generic provider-usage JSON contract deliberately, keep OpenRouter-specific credential resolution and billing HTTP parsing inside the OpenRouter plugin via the existing usage hooks, fetch metered billing only for the JSON surface that renders it, report non-2xx and partial failures accurately, and document the new fields before merging.

## Reproduction Assessment

Yes. The blocking issues are reproducible by static inspection of the PR head: `openrouterUsage` is typed as the promise `ReturnType` while storing an awaited value, and the OpenRouter key usage request targets an endpoint that does not match upstream docs.

## Solution Assessment

No. The requested direction is useful, but this PR is not the best implementation until it follows the provider-owned usage boundary, fixes type safety, uses the documented OpenRouter endpoints, gates billing calls to `--json`, and treats partial billing failures as degraded status.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Use the awaited OpenRouter usage type:** `src/commands/models/list.status-command.ts:346`
  - body: `ReturnType<typeof loadOpenRouterMeteredUsage>` is the promise type, but this variable stores the awaited snapshot and is later read as an object. This should fail typechecking; use `Awaited<ReturnType<...>>` or an explicit snapshot type.
  - confidence: 0.96
- **[P2] Gate OpenRouter billing fetches to JSON output:** `src/commands/models/list.status-command.ts:377-380`
  - body: `openrouterUsage` is only emitted in the JSON payload, but this call runs for every non-plain status invocation. Default rich status can now make authenticated OpenRouter billing requests and wait on timeouts even though it never renders the data.
  - confidence: 0.93
- **[P2] Call OpenRouter's current-key endpoint:** `src/infra/provider-usage.openrouter.ts:181`
  - body: The key-usage fetch requests `/api/v1/auth/key`, while OpenRouter documents the current-key endpoint as `/api/v1/key`. Valid keys can therefore produce failed or empty key usage in the new JSON payload.
  - confidence: 0.91
- **[P2] Treat non-2xx key responses as provider errors:** `src/infra/provider-usage.openrouter.ts:193-194`
  - body: A 401/403/500 key response becomes `null`, and the caller only records thrown errors. Invalid or revoked keys can be silently omitted while the provider still reports an apparently healthy status with incomplete data.
  - confidence: 0.9
- **[P2] Mark partial key fetch failures as degraded:** `src/infra/provider-usage.openrouter.ts:272-280`
  - body: When one key throws but another succeeds, `keyErrors` is ignored unless there are zero successful keys. Multi-key setups can therefore return `status: "ok"` while omitting failed profiles from the JSON payload.
  - confidence: 0.88
