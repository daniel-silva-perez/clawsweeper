## Summary

Keep this PR open. Current main still does not implement the intended `openclaw models status --json` billing/usage schema: the JSON auth payload has no `providerUsage`, the shared provider usage type remains window-only, OpenRouter is not a usage provider, and the OpenRouter plugin does not implement usage hooks. The PR remains a plausible implementation candidate, but it needs maintainer design/security review before merge because it introduces provider billing calls and account/per-key spend-like JSON output.

## Best Possible Solution

Keep this PR open as an implementation candidate. Rebase it onto current main, decide the public `models status --json` windowed/metered schema, and put OpenRouter-specific billing auth/fetch/parsing in `extensions/openrouter` through `resolveUsageAuth` and `fetchUsageSnapshot` where possible. If the existing single-token snapshot contract cannot model multi-key/account data, add a narrow backwards-compatible provider usage SDK extension, then cover JSON output, fetch gating, partial/non-2xx failures, and docs at `https://docs.openclaw.ai/cli/models`.

## Risks / Open Questions

- The PR’s proposed OpenRouter-specific billing auth, HTTP requests, and parsing live in core `src/infra` in the supplied diff, while current main has provider-owned usage hooks intended for this provider-specific logic.
- The proposed JSON exposes account credits, remaining balance, per-key usage, and limits. That is useful, but it is more sensitive than current usage-window summaries and needs explicit maintainer product/security review.
- The current usage contract is single-snapshot/window-based, and the auth resolver returns one token per usage provider. Multi-key/account metered reporting may require a small backwards-compatible contract extension or a provider-owned aggregation design.
- Open review context still raises merge-readiness concerns around output gating, non-2xx responses, and partial key/credits failures accurately affecting status instead of silently producing healthy-looking JSON.
