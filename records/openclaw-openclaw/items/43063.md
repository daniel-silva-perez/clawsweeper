## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks the requested JSON provider-usage surface, while the latest PR head still has blocking correctness issues plus billing-data privacy/API-boundary decisions that need maintainer-guided rework.

## What This Changes

The PR adds an OpenRouter metered billing fetcher and wires windowed/metered provider usage into `openclaw models status --json`, with focused status-command and fetcher tests.

## Best Possible Solution

Extend the generic provider-usage JSON contract deliberately, keep OpenRouter credential resolution and billing endpoint parsing inside `extensions/openrouter` through the existing usage hooks, fetch metered billing only for the JSON surface that renders it, report non-2xx and partial failures accurately, and document the new `models status --json` fields.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Use the awaited OpenRouter usage type:** `src/commands/models/list.status-command.ts:346`
  - body: `ReturnType<typeof loadOpenRouterMeteredUsage>` is the promise type, but this variable stores the awaited snapshot and is later read as an object. This should fail typechecking; use `Awaited<ReturnType<...>>` or the explicit snapshot type.
  - confidence: 0.95
- **[P2] Call OpenRouter's current-key endpoint:** `src/infra/provider-usage.openrouter.ts:181`
  - body: The key-usage fetch requests `/api/v1/auth/key`, while OpenRouter documents the current-key endpoint as `/api/v1/key`. Valid keys can therefore produce failed or empty key usage in the new JSON payload.
  - confidence: 0.9
- **[P2] Mark non-2xx OpenRouter responses as errors:** `src/infra/provider-usage.openrouter.ts:193-194`
  - body: Non-2xx key responses are converted to `null`, and the caller only records thrown errors. Invalid or revoked keys can be silently omitted while the provider still reports `status: "ok"` with incomplete data.
  - confidence: 0.9
- **[P2] Gate OpenRouter billing calls to JSON output:** `src/commands/models/list.status-command.ts:377-380`
  - body: `openrouterUsage` is only rendered in the JSON payload, but the final patch calls the billing fetcher for every non-plain status invocation. Default rich status and `--check` can make unnecessary authenticated billing requests and wait on timeouts.
  - confidence: 0.92

## Security Review

Status: needs_attention

Summary: The diff introduces authenticated OpenRouter billing requests and exposes account/key spend metadata in JSON, so the privacy and public-contract surface needs maintainer approval before merge.
