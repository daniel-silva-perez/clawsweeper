## Summary

Current main still does not implement this PR's intended behavior. `openclaw models status --json` has no unified `auth.providerUsage` schema, the provider usage contract is still window-based, and OpenRouter is not registered as a usage provider. The PR remains a plausible implementation candidate, but it needs maintainer/design work around the provider-owned usage hook boundary, billing-data sensitivity, output gating, and error handling before it should merge or close.

## Best Possible Solution

Keep this PR open as an implementation candidate. Rebase it onto current main, decide the generic windowed/metered JSON contract in the core models status surface, and implement OpenRouter-specific billing auth/fetch/parsing in `extensions/openrouter` through `resolveUsageAuth` and `fetchUsageSnapshot` or add a small backwards-compatible SDK extension if the existing single-token snapshot contract is insufficient for multi-key/account reporting. Gate billing fetches to output modes that render them, mark non-2xx and partial failures accurately, add command/fetcher tests, and document the new public JSON fields at `https://docs.openclaw.ai/cli/models`.

## Risks / Open Questions

- The PR adds secret-bearing OpenRouter billing calls and exposes account credits/per-key spend-like usage data; that needs explicit maintainer review for privacy and user expectations.
- The proposed implementation places OpenRouter-specific auth, HTTP requests, and response parsing in core `src/infra` even though current main has provider-owned usage hooks for this purpose.
- The PR still needs careful handling for output gating and partial/non-2xx OpenRouter failures so default status paths do not gain unnecessary latency and JSON health does not look healthy when billing/auth calls fail.
