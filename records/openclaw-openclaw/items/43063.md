## Summary

Keep this PR open. Current main still lacks the requested provider billing data in `openclaw models status --json`, but the PR is not merge-ready because the latest patch has blocking correctness issues around typing, provider ownership, OpenRouter endpoint selection, error reporting, and avoidable billing calls outside the JSON-only surface.

## What This Changes

The PR adds an `auth.providerUsage` JSON schema to `openclaw models status --json`, maps existing windowed provider usage into it, and adds a new OpenRouter metered billing fetcher with tests.

## Best Possible Solution

Rework this as a deliberate provider-usage contract change: put OpenRouter credential resolution and billing HTTP parsing in the OpenRouter provider plugin via the existing usage hooks or a generic metered extension of that seam, expose a documented versioned JSON shape, fetch metered billing only for `--json`, and preserve degraded/error status for partial provider failures.

## Reproduction Assessment

Yes. Static inspection reproduces the blocking issues: current main lacks `auth.providerUsage`, while the PR diff has the wrong awaited type, an OpenRouter core fetcher, a documented-endpoint mismatch, non-JSON billing calls, and silent non-2xx/partial-failure paths.

## Solution Assessment

No. The feature direction is useful, but this patch is not the best implementation until it follows the provider-owned usage boundary, fixes the type error, uses OpenRouter's documented endpoints, gates billing calls to the JSON surface, and documents the billing-data contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Use the awaited OpenRouter usage type:** `src/commands/models/list.status-command.ts:346`
  - body: `ReturnType<typeof loadOpenRouterMeteredUsage>` is the Promise type, but this variable stores the awaited value and is later read as an object. This should fail typechecking; use `Awaited<ReturnType<...>>` or an explicit snapshot type.
  - confidence: 0.96
- **[P2] Gate OpenRouter billing fetches to JSON output:** `src/commands/models/list.status-command.ts:377-380`
  - body: `openrouterUsage` is only emitted in the JSON payload, but the fetch runs for every non-plain status invocation. Default rich status can now make authenticated OpenRouter billing requests and wait on timeouts even though it never renders the data.
  - confidence: 0.94
- **[P2] Keep OpenRouter billing inside the provider plugin:** `src/infra/provider-usage.openrouter.ts:1-10`
  - body: This adds an OpenRouter-specific billing fetcher under core infra even though the provider-usage hook already lets plugins own provider-specific usage HTTP parsing. Move this behavior into the OpenRouter provider plugin or extend the generic usage seam.
  - confidence: 0.9
- **[P2] Call OpenRouter's documented key endpoint:** `src/infra/provider-usage.openrouter.ts:181`
  - body: The key-usage fetch requests `/api/v1/auth/key`, while OpenRouter's current OpenAPI documents the current-key endpoint as `/key`. Valid keys can therefore produce failed or empty key usage in the new JSON payload.
  - confidence: 0.95
- **[P2] Treat non-2xx key responses as provider errors:** `src/infra/provider-usage.openrouter.ts:193-195`
  - body: A 401, 403, or 500 key response becomes `null`, and the caller only records thrown errors. Invalid or revoked keys can be silently omitted while the provider still reports an apparently healthy status with incomplete data.
  - confidence: 0.91
- **[P2] Mark partial key fetch failures as degraded:** `src/infra/provider-usage.openrouter.ts:272-280`
  - body: When one key throws but another succeeds, `keyErrors` is ignored unless there are zero successful keys. Multi-key setups can therefore return `status: "ok"` while omitting failed profiles from the JSON payload.
  - confidence: 0.89
- **[P2] Surface credits endpoint failures as provider errors:** `src/infra/provider-usage.openrouter.ts:153-155`
  - body: OpenRouter documents `/credits` as management-key-only, but a non-2xx response is converted to `null` and can still return an `ok` provider when key usage succeeds. That hides real auth or permission failures for account totals.
  - confidence: 0.87
