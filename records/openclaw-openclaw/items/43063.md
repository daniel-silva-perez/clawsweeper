## Summary

Keep this PR open. Current main still does not expose a provider billing payload in `models status --json`, but the PR head is not merge-ready: it has a TypeScript type error, still performs OpenRouter billing requests on non-JSON status output, uses the wrong OpenRouter key endpoint, masks several billing/auth failures as healthy output, and needs maintainer approval for the new billing-data JSON contract.

## What This Changes

The PR adds an OpenRouter metered billing fetcher, maps existing windowed usage snapshots into a new `auth.providerUsage` JSON shape, and updates `models status` tests.

## Best Possible Solution

Extend the generic provider-usage contract deliberately, keep OpenRouter-specific credential resolution and HTTP parsing inside the OpenRouter plugin, expose metered snapshots through a versioned provider-usage shape, fetch billing data only for the JSON surface that emits it, report non-2xx and partial failures accurately, and document the new JSON fields before merge.

## Reproduction Assessment

Yes. Static inspection of the PR head reproduces the blocking issues: the awaited value is typed as a Promise, OpenRouter billing is called in non-JSON status paths, the key endpoint differs from OpenRouter's current OpenAPI spec, and non-2xx responses are collapsed to missing data without a degraded status.

## Solution Assessment

No. The requested feature is useful, but this patch is not the best implementation until it follows the provider-owned usage boundary, fixes type safety, uses documented OpenRouter endpoints, gates metered billing to `--json`, and treats partial billing failures as degraded/error states.

Overall confidence: 0.94

Full review comments:

- **[P1] Use the awaited OpenRouter usage type:** `src/commands/models/list.status-command.ts:346`
  - body: `ReturnType<typeof loadOpenRouterMeteredUsage>` is the Promise type, but this variable stores the awaited snapshot and is later read as an object. This should fail typechecking; use `Awaited<ReturnType<...>>` or an explicit snapshot type.
  - confidence: 0.96
- **[P2] Gate OpenRouter billing fetches to JSON output:** `src/commands/models/list.status-command.ts:377-380`
  - body: `openrouterUsage` is only emitted in the JSON payload, but this call runs for every non-plain status invocation. Default rich status can now make authenticated OpenRouter billing requests and wait on timeouts even though it never renders the data.
  - confidence: 0.94
- **[P2] Keep OpenRouter billing inside the provider plugin:** `src/infra/provider-usage.openrouter.ts:1`
  - body: This adds an OpenRouter-specific billing fetcher under core infra even though the provider-usage hook already makes plugins own provider-specific HTTP parsing. Move this behavior into the OpenRouter provider plugin or a generic metered usage seam.
  - confidence: 0.88
- **[P2] Call OpenRouter's documented key endpoint:** `src/infra/provider-usage.openrouter.ts:181`
  - body: The key-usage fetch requests `/api/v1/auth/key`, while OpenRouter's current OpenAPI spec documents the current-key endpoint as `/key`. Valid keys can therefore produce failed or empty key usage in the new JSON payload.
  - confidence: 0.95
- **[P2] Treat non-2xx key responses as provider errors:** `src/infra/provider-usage.openrouter.ts:193-195`
  - body: A 401, 403, or 500 key response becomes `null`, and the caller only records thrown errors. Invalid or revoked keys can be silently omitted while the provider still reports an apparently healthy status with incomplete data.
  - confidence: 0.91
- **[P2] Mark partial key fetch failures as degraded:** `src/infra/provider-usage.openrouter.ts:272-280`
  - body: When one key throws but another succeeds, `keyErrors` is ignored unless there are zero successful keys. Multi-key setups can therefore return `status: "ok"` while omitting failed profiles from the JSON payload.
  - confidence: 0.89
- **[P2] Surface credits endpoint failures as provider errors:** `src/infra/provider-usage.openrouter.ts:153-155`
  - body: OpenRouter documents `/credits` as requiring a management key, but a non-2xx response is converted to `null` and can still return an `ok` provider when key usage succeeds. That hides real auth or permission failures for the account totals.
  - confidence: 0.87
