## Summary

Keep open. Current main still lacks the requested `auth.providerUsage` JSON surface, and the PR head is not merge-ready because static inspection shows a typecheck blocker plus OpenRouter billing fetch, endpoint, error-reporting, and provider-boundary issues.

## What This Changes

The PR adds a new OpenRouter metered billing fetcher, maps existing windowed provider usage into a new `auth.providerUsage` JSON schema for `openclaw models status --json`, and updates model status tests.

## Best Possible Solution

Rework this as a deliberate provider-usage contract change: keep OpenRouter-specific credential resolution and HTTP parsing in the OpenRouter provider plugin or a generic metered-usage seam, expose a documented versioned JSON shape, fetch billing only for the JSON surface, and preserve accurate degraded/error status for partial failures.

## Reproduction Assessment

Yes. Static inspection of the PR head reproduces the main failures: the awaited value is typed as a Promise, OpenRouter billing is fetched in non-JSON status, OpenRouter's OpenAPI lists `/key` rather than `/auth/key`, and non-2xx responses are converted to missing data.

## Solution Assessment

No. The requested feature is useful, but this patch is not the best implementation until it follows the provider-owned usage boundary, fixes type safety, gates metered billing to `--json`, uses documented OpenRouter endpoints, and documents the new billing-data contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Use the awaited OpenRouter usage type:** `src/commands/models/list.status-command.ts:346`
  - body: `ReturnType<typeof loadOpenRouterMeteredUsage>` is the Promise type, but this variable stores the awaited snapshot and is later read as an object. This should fail typechecking; use `Awaited<ReturnType<...>>` or an explicit snapshot type.
  - confidence: 0.96
- **[P2] Gate OpenRouter billing fetches to JSON output:** `src/commands/models/list.status-command.ts:377-380`
  - body: `openrouterUsage` is only emitted in the JSON payload, but this call runs for every non-plain status invocation. Default rich status can now make authenticated OpenRouter billing requests and wait on timeouts even though it never renders the data.
  - confidence: 0.94
- **[P2] Keep OpenRouter billing inside the provider plugin:** `src/infra/provider-usage.openrouter.ts:1-10`
  - body: This adds an OpenRouter-specific billing fetcher under core infra even though the provider-usage hook already makes plugins own provider-specific HTTP parsing. Move this behavior into the OpenRouter provider plugin or add a generic metered usage seam.
  - confidence: 0.88
- **[P2] Call OpenRouter's documented key endpoint:** `src/infra/provider-usage.openrouter.ts:181`
  - body: The key-usage fetch requests `/api/v1/auth/key`, while OpenRouter's current OpenAPI spec documents the current-key endpoint as `/key`. Valid keys can therefore produce failed or empty key usage in the new JSON payload.
  - confidence: 0.95
- **[P2] Treat non-2xx key responses as provider errors:** `src/infra/provider-usage.openrouter.ts:193-195`
  - body: A 401, 403, or 500 key response becomes `null`, and the caller only records thrown errors. Invalid or revoked keys can be silently omitted while the provider still reports an apparently healthy status with incomplete data.
  - confidence: 0.91
- **[P2] Mark partial key fetch failures as degraded:** `src/infra/provider-usage.openrouter.ts:272-280`
  - body: When one key throws but another succeeds, `keyErrors` is ignored unless there are zero successful keys. Multi-key setups can therefore return `status: "ok"` while omitting failed profiles from the JSON payload.
  - confidence: 0.89
- **[P2] Surface credits endpoint failures as provider errors:** `src/infra/provider-usage.openrouter.ts:153-155`
  - body: OpenRouter documents `/credits` as requiring a management key, but a non-2xx response is converted to `null` and can still return an `ok` provider when key usage succeeds. That hides real auth or permission failures for account totals.
  - confidence: 0.87
