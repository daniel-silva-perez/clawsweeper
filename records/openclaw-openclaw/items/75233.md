## Summary

Keep open. Current main still generates the macOS gateway service PATH from user/version-manager directories and merges preserved plist PATH entries into the install environment, while docs and tests describe that as the expected policy. The open closing PR #75246 targets this exact behavior, so the issue should stay open until that PR lands or is replaced.

## What This Changes

Request that macOS gateway LaunchAgent installs write a deterministic minimal PATH by default instead of persisting interactive-shell or Node version-manager PATH entries.

## Best Possible Solution

Land #75246 or an equivalent maintainer-approved macOS-only fix that separates the gateway executable path from child subprocess PATH, stops preserving old macOS plist PATH entries, and updates audit logic, docs, changelog, and focused service-env/install tests together.

## Reproduction Assessment

Yes for the source-level behavior: build a macOS service environment with HOME plus version-manager roots or existing plist PATH entries and inspect getMinimalServicePathParts/buildGatewayInstallPlan output. The broader launchd timeout symptoms were not live-reproduced here.

## Solution Assessment

Unclear until maintainers approve the macOS LaunchAgent PATH policy. The linked PR appears narrowly aligned with the report, but it intentionally changes behavior that current docs and tests describe as supported.
