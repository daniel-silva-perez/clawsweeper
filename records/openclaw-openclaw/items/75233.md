## Summary

Keep open. Current main still generates the macOS gateway service PATH from helpers that include user-bin and Node version-manager directories, and the install/repair path persists that computed PATH, so the requested deterministic canonical LaunchAgent PATH is not implemented.

## What This Changes

The issue requests that macOS gateway LaunchAgent installation write a deterministic minimal PATH instead of capturing or generating Volta/asdf/fnm/pnpm-style version-manager directories.

## Best Possible Solution

Define a maintainer-approved macOS gateway service PATH policy, then update the installer, doctor audit/repair logic, docs, and focused tests together. The maintainable end state should separate the absolute Node executable used to launch the gateway from the subprocess search PATH inherited by gateway children.

## Reproduction Assessment

Yes for the core source-level behavior: with a macOS HOME and existing version-manager directories, current helpers include those directories before system dirs and the install plan persists that PATH. The broader timeout/version-mismatch symptoms need a live macOS LaunchAgent run to reproduce independently.

## Solution Assessment

Unclear. A canonical system PATH is plausible for deterministic update behavior, but current docs/tests intentionally preserve version-manager roots and related reports ask for version-manager compatibility, so maintainers need to choose the policy before implementation.
