## Summary

Keep open. Current main still ignores `cwd` for native `sessions_spawn`, so the contribution targets a real gap, but the current PR is not merge-ready because its attachment override can write cross-agent attachments into the requester or `cwd` workspace while the child still runs from the target agent workspace.

## What This Changes

The PR forwards the `sessions_spawn` `cwd` parameter into native subagent workspace inheritance and adjusts subagent attachment materialization to use that overridden workspace.

## Best Possible Solution

Keep the native `cwd` support, but compute one effective child workspace after applying the cross-agent target-agent rule and use that same workspace for session metadata, runtime startup, registry state, and attachment materialization.

## Reproduction Assessment

Yes. Current-main inspection shows `cwd` is read but not passed to native `spawnSubagentDirect`; the PR regression is reproducible by tracing a cross-agent native `sessions_spawn` call with attachments and an inherited requester workspace.

## Solution Assessment

No. The initial `cwd || opts?.workspaceDir` direction is plausible for same-agent native spawns, but the attachment follow-up should not use requester `ctx.workspaceDir` unconditionally; it should align with the final resolved child workspace.
