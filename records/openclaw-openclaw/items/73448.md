## Summary

Keep open. Current main still accepts `asVoice` in the shared message tool schema but does not preserve that flag through the shared runner, core send service, direct send payload plan, or gateway send contract; open PR #73483 uses closing syntax for this issue and PR #74920 overlaps the same send path, so the issue should stay open until one implementation lands or is superseded.

## What This Changes

The issue requests preserving the message tool `asVoice` flag through shared send paths so audio media can be delivered as native voice messages where supported.

## Best Possible Solution

Land one narrow shared pass-through that maps message-tool and CLI `asVoice` to downstream `audioAsVoice` through the send runner, core send service, direct/gateway send contracts, generated protocol clients if needed, docs, and focused regression tests, while leaving plugin-owned transcoding behavior in the channel plugins.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: `asVoice: true` is accepted by the shared message tool schema, but the inspected shared runner, core send service, and gateway send path have no field that can carry it to delivery.

## Solution Assessment

Yes. The narrow maintainable fix is a shared pass-through for the existing voice intent, because lower-level payload delivery and Feishu media handling already honor `audioAsVoice` once it reaches them.
