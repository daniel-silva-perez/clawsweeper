## Summary

Keep open. Current main still accepts `asVoice` in the shared message tool schema but does not carry that intent through the shared send runner, outbound send service, `sendMessage`, or gateway `send` contract, and PR #73483 remains open with explicit closing linkage to this issue.

## What This Changes

The issue requests that the shared message send path read the `asVoice` parameter and propagate it as downstream `audioAsVoice` so supported channels can send native voice messages.

## Best Possible Solution

Land one narrow, additive implementation that maps shared message-tool and CLI `asVoice` to downstream `audioAsVoice` through `handleSendAction`, `executeSendAction`, `sendMessage`, gateway `send`, generated client models where applicable, docs/help, and focused regression tests for both direct and gateway delivery.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: call the shared message tool with audio media and `asVoice: true`; current main accepts the parameter in the schema but has no field to carry it to shared outbound or gateway delivery.

## Solution Assessment

Yes. The best fix direction is a narrow pass-through in the shared send contract, not a Feishu-only workaround, because generic delivery and Feishu already support `audioAsVoice` once it reaches the lower media seam.
