## Summary

Keep open. Current main still exposes `asVoice` on the shared `message` send schema, but the shared `handleSendAction` path only reads `gifPlayback`, `forceDocument`/`asDocument`, `bestEffort`, and `silent`, then calls `executeSendAction` without any voice flag. `executeSendAction` and `sendMessage` also have no `audioAsVoice`/`asVoice` parameter, so the generic outbound payload plan is built without preserving the tool parameter. Feishu’s plugin action path does separately read `asVoice` and has a regression test for that direct plugin path, so the Feishu-specific claim is partially mitigated, but the schema-to-generic-send propagation gap reported here is still actionable on main.

## Best Possible Solution

Keep this issue open and fix the shared send pipeline: read `asVoice` in `handleSendAction`, carry it as `audioAsVoice` through `executeSendAction` and `sendMessage`, include it in the outbound payload plan, and add a focused regression test in the shared message-action/outbound path. Preserve the existing Feishu plugin action behavior and its test.
