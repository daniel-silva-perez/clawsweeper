## Summary

Keep open. Current main still accepts `asVoice` at the shared message tool schema but does not carry that intent through the shared core or gateway send paths, and an open linked PR already claims the issue for closure.

## What This Changes

The issue requests propagating the message tool `asVoice` parameter as downstream `audioAsVoice` so audio media can be delivered as native voice messages where supported.

## Best Possible Solution

Land one reconciled, narrow pass-through implementation that maps `asVoice` to `audioAsVoice` across the shared action runner, outbound send service, `sendMessage`, gateway schema/handler, generated clients, CLI/docs, and focused regression tests.

## Reproduction Assessment

Yes. A static reproduction is high-confidence: call the shared message send path with audio media and `asVoice: true`; current main accepts the schema field but has no shared core or gateway field that can deliver it as `audioAsVoice`.

## Solution Assessment

Yes. The maintainable solution is an additive pass-through of the existing semantic rather than a channel-specific workaround, because lower delivery and channel adapters already know how to honor `audioAsVoice` once it reaches them.
