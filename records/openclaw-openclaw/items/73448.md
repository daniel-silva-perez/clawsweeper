## Summary

Keep open. Current main still accepts `asVoice` in the shared message tool schema but does not propagate that intent through the shared send runner, outbound send service, `sendMessage`, or gateway `send` contract, and the explicitly linked fix PR remains open and unmerged.

## What This Changes

The issue requests that shared message sends read `asVoice` and pass it downstream as `audioAsVoice` so supported channels can deliver native voice messages.

## Best Possible Solution

Land one narrow, additive implementation that maps shared message-tool and CLI `asVoice` to downstream `audioAsVoice` through the shared action runner, outbound send service, direct and gateway delivery contracts, generated clients where applicable, docs/help, and focused direct/gateway regression tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: pass audio media with `asVoice: true` to the shared message tool; current main accepts the schema field but has no shared outbound or gateway field that can carry it to delivery.

## Solution Assessment

Yes. The narrow maintainable direction is a pass-through fix in the shared send contract rather than a Feishu-only workaround, because generic delivery and Feishu already support `audioAsVoice` once it reaches the lower media seam.
