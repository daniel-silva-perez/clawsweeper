## Summary

Keep open. Current main still exposes `asVoice` in the shared message tool schema but the shared send runner, core send service, direct send path, and gateway send contract have no `asVoice`/`audioAsVoice` pass-through; open PR #73483 claims the fix with closing syntax and PR #74920 overlaps, so the issue should remain open until one implementation lands or is replaced.

## What This Changes

Propagate the message tool `asVoice` parameter through shared sends so audio media reaches delivery as `audioAsVoice` for native voice-message capable channels.

## Best Possible Solution

Land one narrow, tested pass-through that maps message-tool and CLI `asVoice` to downstream `audioAsVoice` through `handleSendAction`, `executeSendAction`, `sendMessage`, gateway `send`, generated protocol clients where needed, docs, and focused regression tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: `asVoice: true` is accepted by the shared message tool schema, but the inspected shared runner and gateway/core send paths have no field that can carry it to delivery.

## Solution Assessment

Yes. The best fix is a shared pass-through for the existing voice intent rather than a Feishu-only workaround, because lower-level delivery and Feishu media handling already understand `audioAsVoice` once it reaches them.
