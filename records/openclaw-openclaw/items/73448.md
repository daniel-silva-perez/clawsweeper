## Summary

Keep open. Current main still exposes `asVoice` in the shared message tool schema but does not propagate a voice intent through the shared send runner, `executeSendAction`, `sendMessage`, or gateway send contract; linked PR #73483 is open and unmerged, so this issue should remain open until that fix or an equivalent one lands.

## What This Changes

The issue reports that message tool `send` accepts `asVoice` but drops it before shared outbound delivery, causing supported channels to send audio as a regular attachment instead of a native voice message.

## Best Possible Solution

Land one narrow implementation that maps `asVoice` to downstream `audioAsVoice` through the shared action runner, outbound send service, `sendMessage`, gateway send schema/handler, generated clients where applicable, CLI/docs, and focused regression tests.

## Reproduction Assessment

Yes. A static reproduction is high-confidence: pass `asVoice: true` to shared message-tool `send` with audio media; current main accepts the schema field but the inspected shared send and gateway paths have no field that can carry it to delivery.

## Solution Assessment

Yes, the best direction is a narrow pass-through fix rather than a channel-specific workaround. The open implementation candidate should be reviewed or replaced with an equivalent scoped patch before this issue closes.
