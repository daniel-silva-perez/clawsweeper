## Summary

Keep this PR open. Current main has only partial Telegram Business plumbing: business update names appear in fallback allowed-update constants, but there is no `business_message` or `business_connection` handler, no outbound `business_connection_id` routing, and no business-specific tests/docs. The PR is a plausible implementation candidate for open #20786, but it needs revision for the existing P1 review feedback and the outbound API mutation risk before it should merge.

## Best Possible Solution

Keep this PR open as the implementation candidate for Telegram Business support, but require a revised patch: explicitly subscribe to all needed business updates, route `business_message` through the inbound pipeline, scope business connection state per Telegram account, clear or update state on disconnect/edit events, handle outbound `business_connection_id` through a narrow safe seam, and add focused tests for multi-account and disconnect cases.

## Risks / Open Questions

- Current main does not implement the PR's central behavior, so closing as implemented would leave Telegram Business messages unrouted.
- The PR's proposed module-scope map can mix mappings across multiple Telegram accounts and inject the wrong `business_connection_id`.
- The PR does not remove mappings on Telegram Business disconnect/edit events, leaving stale outbound state.
- The outbound API transformer mutates payloads broadly; it should be narrowed, account-aware, and covered by regression tests before merge.
