## Summary

Keep this PR open. Current main still lacks Telegram Business message/connection handlers and outbound Business reply wiring, but the proposed patch has blocking correctness and account-isolation defects that need maintainer handling before merge.

## What This Changes

The PR adds Telegram Business allowed update names, Business update context fields, business_connection and business_message handlers with an in-memory connection map, an outbound API transformer, and focused handler tests.

## Best Possible Solution

Ship a refreshed Telegram plugin implementation that handles Business connect and disconnect state per account, routes business_message through the existing DM authorization and inbound pipeline, updates dedupe and sequential keys for Business payloads, injects business_connection_id only for supported outbound methods using user_chat_id-based routing, and documents Telegram Business setup plus access-control behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Avoid pre-consuming the Business message dedupe key:** `extensions/telegram/src/bot-handlers.runtime.ts:1765-1767`
  - body: `handleInboundMessageLike` already calls `shouldSkipUpdate` for the same context. With the update tracker middleware active, this added call marks the Business message key handled, so the second call inside the shared inbound path skips the message before processing.
  - confidence: 0.9
- **[P1] Scope Business connection mappings per account:** `extensions/telegram/src/bot-handlers.runtime.ts:97`
  - body: `businessConnectionMap` is module-scoped, but Telegram supports multiple configured accounts. A Business update on one account can overwrite the same numeric chat ID for another account, causing outbound API calls to use the wrong `business_connection_id`.
  - confidence: 0.88
- **[P1] Remove Business mappings on disconnect:** `extensions/telegram/src/bot-handlers.runtime.ts:1744`
  - body: Business connection updates include `is_enabled: false` for disabled connections, but the handler always stores the mapping. After disconnect, outbound calls can keep injecting a stale Business connection ID.
  - confidence: 0.87
- **[P2] Use user_chat_id for connection routing:** `extensions/telegram/src/bot-handlers.runtime.ts:1740`
  - body: The Telegram Business connection contract exposes `user_chat_id` as the private chat identifier. Mapping `user.id` instead can route replies to the wrong target and does not match the Bot API type contract.
  - confidence: 0.78
- **[P2] Wire Business payloads into update keys:** `extensions/telegram/src/bot-updates.ts:24-34`
  - body: The patch adds Business fields to the context type, but `buildTelegramUpdateKey` and `getTelegramSequentialKey` still never read Business payloads, so semantic dedupe and per-chat serialization remain incomplete for Business updates.
  - confidence: 0.82
- **[P3] Do not exclude Business chat actions:** `extensions/telegram/src/bot-handlers.runtime.ts:141`
  - body: The locked Telegram type definitions include `business_connection_id` on `sendChatAction`, and OpenClaw uses chat actions for typing state before replies. Business chats should get the same account-scoped parameter for supported typing actions.
  - confidence: 0.73

## Security Review

Status: needs_attention

Summary: No dependency, workflow, or package metadata changes were found, but the patch introduces account-isolation risks in Telegram Business outbound API mutation.

Concerns:

- **[medium] Shared Business map crosses Telegram account boundaries:** `extensions/telegram/src/bot-handlers.runtime.ts:97`
  - body: The module-scoped map is shared by every Telegram account instance while the API transformer mutates outbound payloads from that map, so a Business connection observed on one account can affect outbound calls for another configured account using the same numeric chat ID.
  - confidence: 0.86
- **[low] Disconnected Business chats retain outbound authority:** `extensions/telegram/src/bot-handlers.runtime.ts:1744`
  - body: The handler ignores disabled Business connection updates, leaving stale Business connection IDs available for later outbound API calls after a user disconnects the bot.
  - confidence: 0.82
