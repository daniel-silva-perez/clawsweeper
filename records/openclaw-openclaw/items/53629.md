## Risks / Open Questions

- The PR's module-scope businessConnectionMap can share chat-to-business_connection_id mappings across multiple Telegram accounts, which can inject the wrong connection id into outbound calls.
- The PR stores business_connection updates but does not remove mappings on disconnect, leaving stale business_connection_id values for future outbound requests.
- The PR's business_message handler appears to call shouldSkipUpdate before handleInboundMessageLike calls the same dedupe check, which can skip the business message pipeline under current tracker semantics.
- The outbound API transformer adds business_connection_id to every numeric chat_id payload except sendChatAction; supported method coverage should be verified against Telegram Bot API behavior before merge.
- The added tests are useful scaffolding, but several assertions only prove registration/no-throw behavior and do not cover account scoping, disconnect cleanup, dedupe interaction, or method-specific outbound injection.
