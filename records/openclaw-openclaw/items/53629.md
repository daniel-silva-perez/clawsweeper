## Summary

Keep this PR open. Current main only has partial Telegram Business delivery support through allowed-update constants; it still lacks runtime handlers, Business-aware update keys/sequential routing, and outbound `business_connection_id` handling. The PR is useful source work, but the branch needs maintainer review or a refreshed replacement because its proposed outbound transformer and connection map have account-isolation and Telegram API contract problems.

## What This Changes

The PR adds Telegram Business update names, extends Telegram update-key context, registers `business_connection` and `business_message` handlers with an in-memory connection map plus outbound API transformer, and adds focused Telegram Business handler tests.

## Best Possible Solution

Refresh or replace the branch with an account-scoped Telegram Business implementation that handles connect and disconnect state, routes `business_message` through the existing DM authorization/inbound pipeline, updates dedupe and sequential routing for Business updates, injects `business_connection_id` only for supported outbound methods using the correct `user_chat_id` mapping, and adds Telegram docs/changelog plus focused tests.

## Risks / Open Questions

- The PR's module-scoped Business connection map can mix mappings across multiple Telegram accounts and inject the wrong `business_connection_id` into outbound calls.
- The proposed handler stores mappings but does not remove them when Telegram sends a disabled Business connection update, so stale IDs could keep breaking later sends.
- The visible patch maps a Business connection through `user.id` even though Telegram exposes `user_chat_id` for the private chat, which may route replies to the wrong identifier.
- The outbound API transformer mutates every numeric-`chat_id` payload except one method; this is broader than the PR title and needs a method-aware allowlist that matches current Telegram Bot API support.
