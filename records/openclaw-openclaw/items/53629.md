## Summary

Keep this PR open. Current main still lacks the Telegram Business runtime behavior this PR is trying to add; the PR is a useful implementation candidate, but it needs maintainer review because the proposed outbound API mutation and connection mapping have account-isolation and disconnect-state defects.

## What This Changes

The PR adds Telegram Business update names, extends Telegram update-key context, registers `business_connection` and `business_message` handlers with an in-memory connection map and outbound API transformer, and adds focused Telegram Business handler tests.

## Best Possible Solution

Ship a refreshed Telegram plugin implementation that handles Business connect and disconnect state per account, routes `business_message` through the existing DM authorization and inbound pipeline, updates dedupe/sequential keys for Business updates, injects `business_connection_id` only for supported outbound methods using the correct `user_chat_id` mapping, and documents the Telegram Business setup and access-control behavior.

## Risks / Open Questions

- The proposed module-scoped Business connection map can mix connection IDs across multiple Telegram accounts and mutate outbound API calls for the wrong account.
- Disabled Business connections are not removed from the proposed map, so stale `business_connection_id` values may continue to be injected after disconnect.
- Business chat support routes personal DM conversations through OpenClaw, so access-control behavior and docs need explicit maintainer review before shipping.
- The new tests mostly assert registration/no-throw behavior and do not deeply prove account isolation, disconnect cleanup, persistence/restart behavior, or outbound transformer payload correctness.
