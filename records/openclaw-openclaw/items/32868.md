## Summary

Keep open. Current main still has the same-channel block-streaming await gap: block replies are queued on the dispatcher, but `sendBlockReply` is still boolean-only and `dispatch-from-config` does not wait for same-channel dispatcher delivery before the tool-start flush can complete. The issue remains distinct from open PR #65541, which covers disabled block streaming rather than enabled `text_end` same-channel delivery ordering.

## Best Possible Solution

Keep this issue open as the canonical core bug. The best fix is a focused core change that makes accepted same-channel block delivery awaitable, waits for that delivery before tool execution proceeds under `blockStreamingBreak: "text_end"`, preserves the already-awaited cross-channel route, updates ACP/dispatcher contracts and mocks consistently, and adds a regression test proving the actual channel `deliver` promise completes before the next tool starts.

## Repair Work Prompt

Fix this issue's same-channel block streaming delivery ordering bug. Current behavior: with `blockStreamingDefault: "on"` and `blockStreamingBreak: "text_end"`, `dispatch-from-config` awaits cross-channel `sendPayloadAsync(...)` but same-channel delivery only calls `dispatcher.sendBlockReply(...)`; `sendBlockReply` returns `boolean` while the real `options.deliver(...)` runs later on the dispatcher `sendChain`. Expected behavior: when a block reply is accepted for same-channel delivery, the block reply flush awaited before tool execution must not resolve until the actual dispatcher delivery promise settles. Keep the existing cross-channel path intact. Update `ReplyDispatcher` return semantics or add an equally narrow awaitable seam, update ACP delivery handling and all dispatcher mocks/types consistently, and add regression coverage that fails unless the real dispatcher `deliver` promise completes before the next tool starts. Do not fold in disabled-block-streaming behavior from #65541 except to preserve compatibility. Add a user-facing changelog fix entry if the repo's active changelog requires it, crediting the reporter/commenters with real GitHub usernames and never `@codex`. Before handoff, run targeted tests locally and `pnpm check:changed` in Testbox because this touches shared core reply delivery contracts.
