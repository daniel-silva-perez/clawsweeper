## Summary

Keep this issue open. Current main still has the same same-channel block-streaming delivery gap: the block pipeline can wait for the `onBlockReply` callback, but the same-channel callback only enqueues dispatcher delivery and returns before the channel `deliver` promise settles. The WeCom workaround and duplicate WhatsApp context are useful impact evidence, but they do not replace a core awaitable delivery contract.

## Best Possible Solution

Keep this issue open as the canonical core bug. The best fix is a focused core change that makes same-channel block delivery awaitable, awaits it before tool execution proceeds for `blockStreamingBreak: "text_end"`, updates ACP/plugin SDK dispatcher contracts and mocks consistently, and adds a regression test proving actual channel delivery completes before the next tool starts.

## Clownfish Work Prompt

Fix the same-channel block-streaming delivery ordering bug tracked by this issue. Current main still queues same-channel `dispatcher.sendBlockReply(...)` and returns before actual channel delivery, so `blockStreamingBreak: "text_end"` block replies can arrive after tool execution starts.

Expected boundary: make same-channel block delivery awaitable and await it before tool execution proceeds. Update `ReplyDispatcher.sendBlockReply` and related dispatcher return types in a backward-compatible way, update `dispatch-from-config.ts` to await same-channel block delivery, update ACP/direct delivery handling and plugin SDK exports/mocks consistently, and add a regression test that proves the channel delivery promise settles before tool execution continues. Preserve cross-channel behavior and existing boolean/drop semantics for skipped payloads. Do not add channel-specific IDs or provider policy to core.
