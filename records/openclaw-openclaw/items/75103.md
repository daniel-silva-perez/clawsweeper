## Summary

Keep open for normal maintainer review. Current main still lacks start-time disambiguation in the Plugin SDK file lock, while this PR applies the same raw process-start-time pattern already used by adjacent lock helpers; I found no blocking correctness issue in the diff.

## What This Changes

The PR adds an optional lock-owner `starttime` to `src/plugin-sdk/file-lock.ts`, compares it with `getProcessStartTime` during staleness checks, and adds unit coverage for Linux mismatch recovery plus legacy fallback behavior.

## Best Possible Solution

Land the narrow Plugin SDK lock hardening after focused tests and the changed-file gate pass, preserving the existing public API and legacy/non-Linux fallback behavior without broadening this PR into a shared lock refactor.

## Reproduction Assessment

Yes. A high-confidence unit reproduction is to precreate a `.lock` file with the current PID, a fresh `createdAt`, and a mismatched Linux `starttime`; current main keeps retrying because the PID is alive, while the PR reclaims and rewrites the lock.

## Solution Assessment

Yes. The proposed fix is the narrowest maintainable path because it reuses the existing raw `/proc` start-time helper, keeps the lock payload addition internal and optional, and leaves legacy/non-Linux behavior unchanged.
