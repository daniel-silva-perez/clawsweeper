## Summary

Keep open. Current main still has the Plugin SDK file-lock gap: it records only `pid` and `createdAt`, so a live recycled PID can keep a fresh leftover lock until timeout. The PR is a narrow, reviewable implementation that mirrors the already-shipped raw `/proc` start-time pattern used by adjacent lock helpers, and I found no PR-introduced correctness or security finding; the remaining action is maintainer review plus resolving/rerunning the current unrelated CI lint failure.

## What This Changes

The PR adds an optional internal `starttime` field to Plugin SDK file-lock owner payloads, compares it during stale-lock checks, and adds focused unit coverage for Linux mismatch recovery and legacy fallback behavior.

## Best Possible Solution

Land the narrow Plugin SDK lock hardening after maintainer review and clean validation, preserving the existing public file-lock API, internal optional payload compatibility, and legacy/non-Linux fallback behavior without expanding this into a broader lock refactor.

## Reproduction Assessment

Yes. A high-confidence unit reproduction is to precreate a `.lock` file with the current live PID, fresh `createdAt`, and a mismatched Linux `starttime`; current main treats the PID as alive and retries, while the PR reclaims and rewrites the lock.

## Solution Assessment

Yes. The proposed approach is the narrowest maintainable fix because it reuses the existing raw `/proc` start-time helper, keeps the payload addition internal and optional, and matches adjacent lock-helper behavior without changing SDK signatures.
