## Summary

Keep open. The report is consistent with current main: runtime snapshots can resolve Gemini web_search SecretRefs, but web_search execution still accepts a caller-supplied config and keeps it when it does not match the active source snapshot, allowing a stale raw config to reach Gemini provider execution.

## What This Changes

The issue requests that Gemini-backed managed web_search executions use the resolved active runtime secret snapshot instead of raw plugin-scoped SecretRefs, including existing sessions after secrets reload.

## Best Possible Solution

Managed web_search execution should late-bind the active resolved runtime config for gateway/session-owned tool calls, while preserving explicit custom config behavior for SDK/runtime callers. Add a regression test for an existing wrapper holding a stale raw Gemini plugin SecretRef after an active resolved snapshot is installed.

## Reproduction Assessment

Yes. The issue includes concrete config and error output, and current source gives a high-confidence static reproduction path: pass a stale raw config that does not match runtimeSourceConfig into runWebSearch after a resolved runtime snapshot is active, then Gemini reads the raw plugin-scoped apiKey.

## Solution Assessment

Unclear. The requested direction is right, but the maintainable fix should be chosen carefully at the managed tool/runtime boundary so resolved snapshots win for session tools without silently ignoring intentionally supplied custom configs.
