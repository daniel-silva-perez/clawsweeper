## Summary

Keep this PR open. Current main still does not bind the cached GitHub Copilot API token to the incoming GitHub credential, so auth-profile rotation can still reuse another profile's valid cached Copilot token. The PR is also paired with the same-author open bug #43658 via closing syntax, so it remains an implementation candidate rather than something to close before merge.

## Best Possible Solution

Keep this PR open for maintainer review. Land it, or an equivalent fix, if the credential-fingerprint cache ownership design is acceptable; otherwise request a revised per-profile cache-key design. Keep #43658 open until the active GitHub Copilot plugin path and the legacy agent helper both stop reusing Copilot API tokens across different GitHub auth profiles.

## Risks / Open Questions

- The PR persists a stable 16-hex-character SHA-256-derived fingerprint of a GitHub PAT; maintainers should explicitly accept that credential-derived cache marker or ask for a per-profile cache-key design.
- The PR still uses one shared cache file, so alternating healthy GitHub Copilot profiles can evict each other and force token refetches. That is an efficiency follow-up, not a reason to close the correctness fix.
- The PR adds regression coverage for the plugin token helper, while the legacy src/agents helper is patched by parity but has no direct new regression test in the shown diff.
