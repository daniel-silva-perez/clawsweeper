## Summary

Keep this PR open. Current main still reuses any valid GitHub Copilot API token from the shared cache before checking which GitHub credential/profile is active, and the same-author bug #43658 remains open with this PR as the implementation candidate. The PR diff is focused and plausibly addresses the bug, but it has not landed on main.

## Best Possible Solution

Keep this PR open for maintainer review. Land it or an equivalent fix if the fingerprint-based cache ownership design is acceptable; otherwise ask for a revised per-profile cache-slot design. Keep #43658 open until both the active GitHub Copilot plugin path and the legacy agent helper stop reusing Copilot API tokens across different GitHub auth profiles.

## Risks / Open Questions

- The proposed fix stores a persistent credential-derived 64-bit SHA-256 prefix in the Copilot token cache. It is not the PAT, but maintainers should explicitly accept that marker in the credentials directory.
- The PR still uses one shared cache file, so alternating healthy profiles can evict each other's Copilot API tokens and force extra token exchanges. That is an efficiency follow-up, not a reason to close the correctness fix.
- The PR patches the legacy src/agents helper by parity, but the shown regression coverage is in the plugin token test surface only.
