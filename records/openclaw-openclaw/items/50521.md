## Summary

Keep this PR open. Checked-out main and the latest tagged release still lack explicit NUL rejection in the shared archive entry validator, while ZIP and TAR extraction both route entry names through that helper; the PR is narrow and I found no blocking review finding in the proposed patch.

## What This Changes

This PR adds a null-byte rejection to `validateArchiveEntryPath()` and two archive-path regression tests for NUL bytes in file and directory components.

## Best Possible Solution

Land this PR, or an equivalent minimal patch, by keeping the NUL-byte rejection in the shared archive path validator before normalization and retaining focused regression coverage for both filename and directory-component cases.

## Reproduction Assessment

Yes. Current main can be checked by reading `validateArchiveEntryPath()` and with a read-only Node probe mirroring its predicates; both show NUL-containing archive entry strings are not rejected before path normalization.

## Solution Assessment

Yes. The proposed guard is the narrowest maintainable fix because it lives in the shared validator used by both ZIP and TAR extraction and runs before string normalization can obscure the original entry path.
