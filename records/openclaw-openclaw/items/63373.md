## Summary

Keep open. Current main has broad assistant-visible sanitization for XML-style `<tool_call>/<tool_result>` leaks and local-model `[tool] ... [END_TOOL_REQUEST]` blocks, but I did not find coverage for the exact reported uppercase bracketed transcript form `[TOOL_CALL] ... [/TOOL_CALL]` with non-JSON `tool => ...` payload syntax. The related closed items address adjacent XML/WebChat/parallel-tool repair cases, not this Telegram outbound marker shape with enough source/test proof to close it.

## Best Possible Solution

Keep this issue open and add a narrowly tested shared user-visible sanitization guard for internal bracketed transcript markers such as `[TOOL_CALL]...[/TOOL_CALL]` and `[TOOL_RESULT]...[/TOOL_RESULT]`, ideally in the shared assistant-visible sanitizer with tests for the exact reported Telegram payload shape. Because the report is a leak class, also consider a final outbound-delivery guard so channel adapters cannot send malformed transcript internals even if a repair path races ahead of normal extraction.
