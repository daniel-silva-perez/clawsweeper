## Summary

Keep open. The PR targets a real current-main delay: cleanup still awaits idle before clearing pending tool results and releasing the session lock. The proposed fast path is narrow and I found no blocking correctness or security issue, but the user-facing perf change still needs the required changelog entry and maintainer/CI validation before merge.

## What This Changes

The PR makes `flushPendingToolResultsAfterIdle` treat `timeoutMs <= 0` as an immediate timeout, passes that fast path during aborted embedded-run cleanup, and adds unit coverage for bypassing `waitForIdle`.

## Best Possible Solution

Land the narrow abort/timeout cleanup fast path after adding the changelog entry and verifying the focused agent guard test plus the changed gate, while preserving the existing idle wait for non-aborted cleanup.

## Reproduction Assessment

Yes. Current main statically reproduces the delay path: a never-idle agent makes cleanup await `waitForAgentIdleBestEffort` for the configured/default timeout before clearing pending tool results and releasing the session lock.

## Solution Assessment

Yes, with the changelog addition. The proposed helper-level `timeoutMs <= 0` convention is a narrow maintainable fix because callers opt into it only for abort/timeout cleanup, leaving the normal idle-wait race protection intact.
