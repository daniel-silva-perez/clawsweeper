Current main still has the lock-holding idle wait, and the latest PR diff appears to address the earlier ClawSweeper finding by passing the runner's internal abort and timeout state into final cleanup; I found no blocking correctness issue in the current patch.

## What This Changes

The PR adds an immediate timeout path for pending-tool-result cleanup, threads abort/timeout state into embedded attempt cleanup, and adds focused coverage for `timeoutMs <= 0`.

## Best Possible Solution

Land a narrow version of this change after maintainer review and CI: normal successful cleanup should still wait for idle to preserve tool results, while external aborts and internal timeout cleanup should clear pending tool results immediately so the session lock is released promptly.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: on current main, a never-idle agent keeps `flushPendingToolResultsAfterIdle` pending until the 30s default, and cleanup does not release `sessionLock` until that await completes.

## Solution Assessment

Yes. The latest patch uses the narrow helper-level immediate-timeout path and now derives final cleanup from the runner's internal abort/timeout flags; the test could assert `waitForIdle` is not called, but that is a coverage strengthening rather than a blocking defect.
