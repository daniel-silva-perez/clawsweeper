## Summary

Keep open. Current main still takes the idle-wait path during abort cleanup, while this PR is a narrow performance fix for a real lock-hold path and has no blocking correctness finding from the provided diff; it still needs normal maintainer review and CI against current main before merge.

## What This Changes

The branch adds an abort fast path that treats nonpositive tool-result flush timeouts as immediate, passes `timeoutMs: 0` from embedded-runner abort cleanup paths, and extends the wait-before-flush unit coverage.

## Best Possible Solution

Merge this abort-only fast path, or an equivalent maintainer-authored patch, after current-main validation confirms interrupted runs skip the idle wait while successful/non-aborted teardown still waits for real tool results to land.

## Reproduction Assessment

Yes. Current main has a high-confidence source-level reproduction: use a guarded session manager with an agent whose `waitForIdle` never resolves, call `flushPendingToolResultsAfterIdle` with `clearPendingOnTimeout: true`, and cleanup waits until the timeout before clearing and releasing the lock.

## Solution Assessment

Yes. The proposed direction is the narrowest maintainable fix because it reuses the existing clear-on-timeout behavior and only makes abort/interruption cleanup pass an immediate timeout, preserving the prior safety wait for ordinary teardown.
