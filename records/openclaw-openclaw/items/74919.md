## Summary

Keep open for normal maintainer review. Current main still waits on the idle path before releasing the session lock, and the PR appears to be a narrow, relevant fix with no definite blocking review finding from read-only inspection.

## What This Changes

The PR adds an immediate-timeout path for pending tool-result cleanup, threads abort state into embedded attempt cleanup, and adds focused helper coverage for zero or negative timeouts.

## Best Possible Solution

Land a narrow version after maintainer review and CI: successful cleanup should still wait for idle to preserve real tool results, while aborted or timed-out cleanup should clear pending tool results immediately so the session lock can be released promptly.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: use an agent whose `waitForIdle` never resolves, then current main keeps `flushPendingToolResultsAfterIdle` pending until the 30s default and cleanup does not release `sessionLock` until that await completes.

## Solution Assessment

Yes. The proposed helper-level `timeoutMs <= 0` short-circuit plus abort-state threading is the narrowest maintainable shape because it preserves the normal successful idle wait while removing the abort cleanup delay.
