## Summary

Keep this PR open. Current main still lacks the requested zh-CN setup localization, CLI locale resolution, and persisted `cli.locale` contract, so the PR is not obsolete; however, the latest head still has merge-blocking issues around locale preservation on reset, missing public config/env documentation, and security-sensitive token-storage copy.

## What This Changes

This PR adds an English/zh-CN CLI setup message catalog, routes setup and finalization copy through it, adds `cli.locale` to config types/schemas, and resolves locale from `OPENCLAW_LOCALE`, config, `LANG`, or an interactive prompt.

## Best Possible Solution

Keep this as the consolidated zh-CN setup localization candidate, but make locale resolution return the effective locale for env, config, LANG, and prompt paths; preserve it through reset and all config writes; document `cli.locale` and `OPENCLAW_LOCALE`; align token-storage copy with the current sessionStorage behavior; and add targeted setup tests before changed-gate validation.

## Reproduction Assessment

Yes. The reset bug is reproducible by inspection: start with `OPENCLAW_LOCALE=zh-CN` or `baseConfig.cli.locale: "zh-CN"`, choose existing config handling `reset`, and the PR head rebuilds `{}` through a no-op `applySelectedLocale` because `selectedLocale` is `undefined`.

## Solution Assessment

No. The overall direction is plausible for the requested feature, but the current PR head is not the best mergeable solution until the effective locale is preserved, the new public config/env surface is documented, and the token-storage note matches current Control UI behavior.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Preserve the effective locale through reset:** `src/wizard/setup.ts:81-90`
  - body: When the locale comes from `OPENCLAW_LOCALE` or existing `baseConfig.cli.locale`, this helper sets the environment but returns `undefined`. The reset path later calls `applySelectedLocale({})`, which no-ops for `undefined`, so a user who resets setup loses their existing locale preference instead of preserving it.
  - confidence: 0.94
- **[P2] Document the new CLI locale contract:** `src/config/types.cli.ts:4`
  - body: This adds `cli.locale` as a persisted config key and `OPENCLAW_LOCALE` as behavior, but the PR does not update the public configuration/onboarding docs or schema help text. That leaves a new user-facing config surface undiscoverable and out of sync with the repo's config contract expectations.
  - confidence: 0.82
- **[P2] Keep token storage copy aligned with sessionStorage:** `src/i18n/cli.ts:220`
  - body: The new token note says the Web UI stores a token copy in `localStorage`, but current Control UI code persists tokens in `sessionStorage` and current docs call out that behavior. This setup copy is security-sensitive and should not tell users the token is stored in a different browser storage bucket.
  - confidence: 0.9
