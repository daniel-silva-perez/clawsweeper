## Summary

Keep this PR open. Current main still lacks the requested CLI zh-CN setup localization, but the latest PR head is not mergeable because it can drop the effective locale during reset, adds an undocumented config/env surface, and introduces security-sensitive token-storage copy that contradicts current Control UI behavior.

## What This Changes

This PR adds an English/zh-CN CLI setup message catalog, locale resolution and persistence through `cli.locale`/`OPENCLAW_LOCALE`/`LANG`, and localized setup/finalization prompts.

## Best Possible Solution

Keep this as the consolidated zh-CN setup localization candidate, but make locale resolution return and persist the effective locale for env, config, LANG, and prompt paths; document `cli.locale` and `OPENCLAW_LOCALE` in the config/onboarding docs and schema help; align token-storage copy with sessionStorage; and add focused reset/persistence tests before changed-gate validation.

## Reproduction Assessment

Yes. By inspection, running setup with an existing `cli.locale: "zh-CN"` or `OPENCLAW_LOCALE=zh-CN`, choosing existing config handling `Reset`, and saving the rebuilt config exercises the no-op `applySelectedLocale({})` path that drops the locale.

## Solution Assessment

No. The overall feature direction is maintainable, but this PR is not the best mergeable solution until the effective locale is preserved through resets and all config writes, the new public contract is documented, and the security-sensitive token copy matches current Control UI behavior.
