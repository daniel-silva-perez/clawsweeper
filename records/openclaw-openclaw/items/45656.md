## Summary

Keep open. Current main still matches the concrete v2 Control UI composer regression: the visible bordered composer contains non-textarea chrome that does not delegate clicks to the textarea, and the composer controls remain small icon-only buttons. Recent slash-command accessibility work improved ARIA wiring but did not address the reported hit target or discoverability problem.

## What This Changes

The issue requests restoring v2 chat composer usability by making the visible chat line reliably clickable, reducing wasted layout space, and improving small icon-only control discoverability compared with v1.

## Best Possible Solution

Fix the Control UI composer so all non-control visible entry chrome focuses the textarea, the apparent input area matches the real text-entry target, and composer controls have practical hit targets and discoverability while preserving Send, Queue, Stop, attachments, Talk, paste, auto-height, slash-command, and Shift+Enter behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path is to open the Control UI composer and click inside the bordered lower composer area outside the textarea and buttons; current main has no wrapper click handler to focus the textarea from that chrome.

## Solution Assessment

Yes for a narrow repair. Focus delegation plus targeted hit-target/discoverability improvements are the maintainable fix; a full rollback to the v1 composer layout is a separate product decision.
