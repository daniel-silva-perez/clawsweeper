## Summary

Keep open. Current main still matches the concrete Control UI composer regression: the visible bordered composer includes non-textarea chrome that does not focus the textarea when clicked, and the controls remain small icon-only buttons with tooltip-dependent labels.

## What This Changes

The issue asks for v2 Control UI chat composer usability to be restored or improved, specifically the click target, wasted space, and tiny icon-only controls.

## Best Possible Solution

Fix the Control UI composer so all non-control visible text-entry chrome focuses the textarea, align the apparent entry area with the actual input target, and improve practical control hit targets/discoverability while preserving Send, Queue, Stop, attachment, Talk, paste, auto-height, and Shift+Enter behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path is to open the Control UI composer and click inside the bordered lower composer area outside the textarea and buttons; current main has no wrapper click handler to focus the textarea from that chrome.

## Solution Assessment

Yes for a narrow repair. Composer focus delegation plus focused hit-target/discoverability changes are the maintainable fix, while a full rollback to the v1 layout is a separate product decision.
