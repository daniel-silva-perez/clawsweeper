## Summary

Keep open: current main still lacks built-in tool-call output for `/v1/responses`, and this active PR is the canonical implementation paired with #75074 after #75107 was closed as the duplicate attempt. I found no blocking correctness issue in the diff, but it still needs maintainer API/product review and a rebase because the live PR is currently dirty against latest main.

## What This Changes

The PR adds a default-off Gateway Responses config flag that emits built-in agent tool-start events as JSON/SSE `function_call` output items, with config schema, docs, changelog, and e2e coverage.

## Best Possible Solution

Land a maintainer-approved version that keeps the behavior opt-in, consumes the existing sanitized run-scoped tool-start events, preserves client-tool delegation semantics, and ships with schema, docs, changelog, and JSON/SSE ordering coverage.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence reproduction path: emit a run-scoped `stream: "tool"`, `phase: "start"` event during a `/v1/responses` turn and current main has no code path that forwards it into `output`. The PR tests model that path with mocked agent events.

## Solution Assessment

Mostly yes. The proposed implementation is narrow, additive, and uses the existing sanitized event stream rather than adding a new runtime channel; the remaining decision is maintainer approval of the public flag/output semantics and a rebase onto current main.
