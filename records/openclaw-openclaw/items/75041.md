## Summary

Keep open. Current main still builds the gateway runtime from `node:24-bookworm-slim` and the default runtime package list still omits `python3`; the documented apt hook only helps local rebuilds, not users pulling the published image. The related open PRs/issues are adjacent fs-safe and `python` symlink work, not a shipped fix for user workspace `.py` scripts in the gateway image.

## What This Changes

The issue requests restoring or documenting `python3` availability for user-authored workspace Python scripts in the published gateway Docker image after the slim Node runtime dropped the interpreter.

## Best Possible Solution

Restore baseline `python3` in the published gateway runtime image, add a Dockerfile regression assertion and image smoke proof, and update Docker docs plus the changelog. If maintainers intentionally keep the image interpreter-free, the docs and `.py` execution failure should make that breaking change explicit and actionable.

## Reproduction Assessment

Yes. The issue provides direct `docker run` commands comparing affected and last-known-good published tags, and current main source still predicts the missing-`python3` result for the gateway runtime image.

## Solution Assessment

Yes. Re-adding baseline `python3` is the narrowest compatibility restoration for workspace `.py` scripts in official images; the current opt-in apt hook is useful for local rebuilds but does not fix pulled release tags.
