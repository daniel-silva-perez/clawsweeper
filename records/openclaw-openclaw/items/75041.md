## Summary

Keep open. Current main still builds the gateway runtime from `node:24-bookworm-slim` and the default runtime apt package list does not include `python3`; the existing opt-in package hook only helps users who rebuild locally. The related open PRs address adjacent internal helper or `python` symlink behavior, not the reported published-image contract for user workspace Python scripts.

## What This Changes

Restore or clearly document the gateway Docker runtime contract so user-authored workspace `.py` scripts keep working in the published slim-based image.

## Best Possible Solution

Restore baseline `python3` in the published gateway runtime image, add a Dockerfile regression test for the runtime package list, and update Docker docs plus the changelog. If maintainers intentionally keep the image interpreter-free, the docs and `.py` execution diagnostic should make that breaking change explicit and actionable.
