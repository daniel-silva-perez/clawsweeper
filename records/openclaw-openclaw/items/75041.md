## Summary

Keep open. Current main still builds the gateway runtime from `node:24-bookworm-slim`, still omits `python3` from the default runtime package list, and only documents `OPENCLAW_DOCKER_APT_PACKAGES` as a local-build escape hatch. The related fs-safe fallback and `python` symlink work is adjacent, but it does not restore user-authored workspace `.py` scripts in pulled gateway images.

## What This Changes

The issue requests restoring or clearly documenting `python3` availability for user-authored workspace Python scripts in the published gateway Docker image after the 2026.4.x slim runtime stopped providing the interpreter.

## Best Possible Solution

Restore baseline `python3` in the published gateway runtime image, add a Dockerfile regression assertion and image smoke proof, and update Docker docs plus the changelog. If maintainers intentionally keep the image interpreter-free, the docs and `.py` execution failure should make that breaking change explicit and actionable.

## Reproduction Assessment

Yes. The issue provides a concrete `docker run` comparison across published tags, and current main source still predicts the missing-`python3` result for the gateway runtime image.

## Solution Assessment

Yes. Re-adding baseline `python3` is the narrowest compatibility restoration for existing workspace `.py` scripts; the current optional apt hook is useful for local rebuilds but does not fix pulled release images.
