## Summary

Keep open. Current main and the latest v2026.4.27 release still build the gateway runtime from `node:24-bookworm-slim` and only install `ca-certificates procps hostname curl git lsof openssl`, so `python3` is still absent from the published default image. The related open work is adjacent but not sufficient: #73791 targets internal `fs-safe` fallback behavior, and #71235 only creates a `python` symlink when `python3` already exists.

## What This Changes

Request to restore or clearly document Python availability in the Docker gateway runtime because v2026.4.x slim images no longer include `python3` and workspace `.py` scripts fail.

## Best Possible Solution

Reestablish a clear Docker runtime contract for workspace Python: preferably install baseline `python3` in the published gateway runtime image and cover it with Dockerfile tests, docs, and a changelog entry; if maintainers intentionally keep the image interpreter-free, document the breaking change and make `.py` execution failures point to the supported rebuild path.

## Reproduction Assessment

Yes. The issue provides direct `docker run` commands comparing `ghcr.io/openclaw/openclaw:2026.4.27` against v2026.4.23, and current main still uses `node:24-bookworm-slim` with no `python3` in the default apt package list.

## Solution Assessment

Yes, restoring baseline `python3` is the narrowest compatibility fix if the goal is to preserve the pre-slim workspace script behavior. A docs-only escape hatch is safer for image size but would leave the reported upgrade regression in the published image.
