## Summary

Keep open. Current main still builds the gateway runtime from `node:24-bookworm-slim` and the default runtime package list still excludes `python3`; the opt-in apt package hook only helps rebuilt/local images. The related open PRs cover adjacent `fs-safe` fallback and `python` symlink behavior, not the published gateway image contract for workspace Python scripts.

## What This Changes

The issue asks OpenClaw to restore or clearly document `python3` support for user-authored workspace `.py` scripts in the published gateway Docker image.

## Best Possible Solution

Restore baseline `python3` in the published gateway runtime image, add a Dockerfile regression assertion and image smoke proof, and update Docker docs plus the changelog. If maintainers intentionally keep the image interpreter-free, document that as a breaking runtime contract and make `.py` execution failures point to the documented rebuild path.

## Reproduction Assessment

Yes. The issue provides direct `docker run` comparisons for v2026.4.27 versus v2026.4.23, and current main source still shows the slim runtime package list without `python3`.

## Solution Assessment

Yes. Re-adding baseline `python3` is the narrowest compatibility restoration for workspace `.py` scripts in the published image; the existing opt-in apt hook is useful for local rebuilds but does not fix users pulling `ghcr.io/openclaw/openclaw:<version>`.
