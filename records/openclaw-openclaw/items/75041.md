## Summary

Keep open. Current main and the latest released Dockerfile still build the gateway runtime from `node:24-bookworm-slim` and install a default apt package list that omits `python3`; the documented apt hook only helps local rebuilds, not users pulling the published image. The related open PRs cover the `python` symlink or internal fs-safe fallback paths, not this published-image contract for workspace Python scripts.

## What This Changes

The issue asks OpenClaw to restore or clearly document `python3` availability in the published gateway Docker image because v2026.4.x slim images break user-authored workspace `.py` scripts.

## Best Possible Solution

Restore baseline `python3` in the published gateway runtime image, add Dockerfile/test coverage that asserts the intended interpreter contract, and update Docker docs plus the changelog. If maintainers intentionally want the gateway image to stay interpreter-free, the product should document that breaking change and surface a clear missing-`python3` error when workspace Python scripts are invoked.

## Reproduction Assessment

Yes. The issue gives direct `docker run` commands comparing v2026.4.27 and v2026.4.23, and source inspection of current main plus the v2026.4.27 Dockerfile confirms the published runtime path still omits `python3`.

## Solution Assessment

Yes, with maintainer review for the image-size tradeoff. Re-adding baseline `python3` is the narrowest compatibility restoration for users pulling the official gateway image; the existing build arg is useful for custom images but does not fix the published-image regression.
