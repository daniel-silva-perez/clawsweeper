## Summary

Keep open. Current main still builds the published gateway runtime from the slim Node base and does not install `python3` by default, so user-authored or shipped `.py` scripts can still fail in pulled v2026.4.x gateway images. The related fs-safe fallback and unversioned `python` symlink work are adjacent but do not restore `python3` availability for workspace scripts.

## What This Changes

Restore or explicitly support `python3` availability for user-authored workspace `.py` scripts in the published gateway Docker image after the v2026.4.x slim-runtime switch removed it.

## Best Possible Solution

Restore baseline `python3` in the published gateway runtime image, add a Dockerfile regression assertion and image smoke proof, and update Docker docs plus the changelog. If maintainers intentionally keep the gateway image interpreter-free, the docs and `.py` execution failure should make that breaking change explicit and actionable.

## Reproduction Assessment

Yes. The issue gives direct `docker run` comparisons between v2026.4.27 and v2026.4.23, and current main source confirms the slim runtime still omits `python3` from the default package list.

## Solution Assessment

Yes. Re-adding baseline `python3` is the narrowest compatibility restoration for workspace `.py` scripts in pulled gateway images; the current apt hook only helps local rebuilds, and the related symlink/fs-safe work does not make `python3` exist.
