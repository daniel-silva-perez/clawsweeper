## Summary

Keep open. Current main still has the Zalo proactive-send mismatch: the Zalo plugin advertises a numeric-only target id predicate, while the Zalo send layer itself accepts arbitrary non-empty string chat_id values. The CLI/action path resolves --target before dispatching the plugin send action, so a hex-like chat_id can still fall through to directory lookup and produce the reported Unknown target error unless it happens to be present in a directory/allowFrom entry.

## Best Possible Solution

Update the Zalo plugin's target resolver to accept real Zalo Bot Platform chat_id strings, including hex-like non-numeric IDs after optional zalo:/zl: prefix normalization. Add a regression test through the message action or target resolver path for a value like 3becaa50ae12474c1e03, and update the Zalo channel docs/allowFrom wording so they no longer imply chat IDs must be numeric.

## Risks / Open Questions

- A fix should stay Zalo-plugin-owned. Relaxing the shared isNumericTargetId helper globally could change target resolution behavior for other channels that intentionally distinguish numeric IDs from names or handles.
- No live Zalo send was attempted during this read-only review because it would send a real outbound message; the code path is enough to keep the issue open.
