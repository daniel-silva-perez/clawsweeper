## Summary

Current main still has the reported Zalo proactive-send mismatch. The Zalo plugin normalizes `zalo:`/`zl:` prefixes, but its target resolver still delegates id detection to the shared numeric-only `isNumericTargetId` predicate; the message action runner resolves `--target` before dispatching Zalo's send action, so a hex-like Zalo `chat_id` can still be rejected as an unknown target unless it is present in directory/allowFrom data.

## Best Possible Solution

Keep this issue open. Update the Zalo plugin's `targetResolver.looksLikeId` to accept real Zalo Bot Platform `chat_id` strings, including hex-like non-numeric IDs after optional `zalo:`/`zl:` prefix normalization. Add a regression test through `resolveChannelTarget` or the message action runner for a value like `3becaa50ae12474c1e03`, and update the Zalo CLI/docs wording so delivery targets are documented as Zalo chat IDs, not numeric-only examples.
