## Summary

Keep open. Current main still has the loose `channels status --timeout` path: `channelsStatusCommand` uses raw `Number(...)`, forwards the value to both gateway request params and transport timeout, and has no focused invalid-timeout regression test. The PR remains useful, but it needs a small rebase/update because current main now has a probe-aware 30,000 ms fallback that the original patch should preserve.

## Best Possible Solution

Keep the PR open and ask for a current-main update: import `parseTimeoutMsWithFallback` in `src/commands/channels/status.ts`, compute the fallback as `opts.probe ? 30_000 : 10_000`, pass `{ invalidType: "error" }`, and add a dedicated option-validation regression test outside the SecretRef fallback describe block. Coordinate with #40953 if maintainers want one broader timeout-validation PR instead of focused per-command fixes.
