## Summary

Keep this PR open. Current main still implements `channels status --timeout` with raw `Number(...)` parsing and forwards that value into the gateway request, so the PR’s intended validation behavior is not implemented on main. The PR remains a narrow implementation candidate; the provided diff is limited to the command and focused tests, and the latest branch repair appears to address the earlier test-placement review note.

## Best Possible Solution

Keep this PR open for maintainer review and landing, or apply the equivalent focused change on main: use `parseTimeoutMsWithFallback(opts.timeout, opts.probe ? 30_000 : 10_000, { invalidType: "error" })` in `src/commands/channels/status.ts`, keep command-level invalid-timeout coverage, confirm the intended `--probe` default at the CLI boundary, then run the normal changed gate before merge.
