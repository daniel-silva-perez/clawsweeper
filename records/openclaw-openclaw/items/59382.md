## Summary

Keep this PR open. Current main still parses `channels status --timeout` with raw `Number(...)` and forwards the result into the gateway status call instead of validating through the shared CLI timeout parser. The PR remains a narrow implementation candidate, and the provided diff does not show security-sensitive workflow, dependency, lockfile, install, release, package, or secrets changes.

## Best Possible Solution

Review and land this PR, or apply the equivalent focused change on main: use `parseTimeoutMsWithFallback(opts.timeout, opts.probe ? 30_000 : 10_000, { invalidType: "error" })` in `src/commands/channels/status.ts`, keep command-level coverage for invalid timeout input, and confirm the desired `--probe` default at the CLI boundary.
