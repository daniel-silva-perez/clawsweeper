## Summary

Keep this PR open. Current main still parses `channels status --timeout` with raw `Number(...)`, so invalid values are not validated through the shared CLI timeout helper before the gateway call. The PR remains a narrow implementation candidate; the current PR context also shows only command code and focused tests, with no apparent security-sensitive workflow, dependency, lockfile, install, release, package, or secrets changes.

## Best Possible Solution

Review and land this PR, or apply the equivalent focused change on main: import and use `parseTimeoutMsWithFallback(opts.timeout, opts.probe ? 30_000 : 10_000, { invalidType: "error" })` in `src/commands/channels/status.ts`, keep command-level coverage for invalid timeout input, and explicitly settle the probe default behavior at the CLI boundary.
