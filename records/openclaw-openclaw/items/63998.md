## Summary

Keep this issue open. Current main has useful partial mitigation through opt-in byte-triggered preflight compaction and successor transcript rotation, but it still does not provide a startup-safe recovery guard for the reported crash-restart/OOM loop. Default configuration leaves the byte guard disabled, and restart/run recovery paths can still materialize a pathological JSONL transcript before quarantine, bounded rotation, or degraded-session handling. I also did not find a count-based circuit breaker for accumulated `openclaw:bootstrap-context:full` markers.

## Best Possible Solution

Keep this issue open and implement a core startup-safe transcript recovery guard. The guard should stat/classify active JSONL files before repair, restart recovery, `readSessionMessages`, and `SessionManager.open`; rotate, quarantine, or compact pathological transcripts into a bounded successor without first materializing them; mark offending sessions degraded so startup does not retry indefinitely; add a count-based circuit breaker for repeated `openclaw:bootstrap-context:full` markers; reuse existing media offload/dedup behavior where applicable; and add focused regression tests for oversized restart loops and marker accumulation.

## Risks / Open Questions

- Closing as implemented would overstate the opt-in byte-trigger compaction as a full crash-loop recovery fix for default users.
- Already pathological active transcripts can still create memory pressure before repair, restart recovery, or `SessionManager.open` reaches a bounded recovery decision.
- Repeated full-bootstrap marker accumulation remains only indirectly mitigated by optional continuation-skip behavior; there is no count-based circuit breaker in the inspected path.
