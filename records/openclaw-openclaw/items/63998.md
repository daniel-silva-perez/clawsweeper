## Summary

Keep this open. Current main still has partial mitigations rather than a default startup-safe recovery guard: the active transcript byte guard is opt-in, and the repair, restart-recovery, and gateway history paths can still read or open a pathological transcript before making a bounded quarantine decision.

## What This Changes

The issue asks OpenClaw to prevent a crash-restart loop where oversized session JSONL transcripts and repeated bootstrap entries are reloaded and expanded until the Gateway OOMs.

## Best Possible Solution

Implement a core startup-safe transcript recovery guard that stats and classifies active JSONL files before repair, restart recovery, readSessionMessages, or SessionManager.open; rotate, quarantine, or compact pathological transcripts into a bounded successor without first loading the whole file; add a total-count circuit breaker for repeated full-bootstrap markers; keep media deduplication as follow-up unless required for the bounded recovery path; and cover this with focused recovery tests and docs.

## Reproduction Assessment

Yes. The issue and 2026-04-30 confirmation give concrete sizes, counts, logs, and restart cadence, and source inspection identifies the current whole-file read/open paths that would exercise the failure mode.

## Solution Assessment

Yes for the recommended repair direction, but no for closing based on current main. The narrow maintainable path is startup-safe transcript classification/quarantine plus a bootstrap marker cap; the existing opt-in compaction guard is useful but does not protect default restart recovery.
