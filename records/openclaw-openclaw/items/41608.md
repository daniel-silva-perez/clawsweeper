## Summary

Current main still does not normalize Slack target-style route binding peer IDs before generic matching. Slack inbound routing passes the raw Slack channel/user ID into `resolveAgentRoute`, while route bindings store only trim-normalized peer IDs and exact-match them. A binding configured as `channel:C...` can still miss an inbound runtime peer of `C...` and fall back to the default `main` agent. The related implementation candidate, PR #42162, is closed and unmerged in the provided GitHub context, so this issue should stay open.

## Best Possible Solution

Keep this issue open. The best fix is to normalize Slack route-binding peer IDs before generic matching so target-style forms such as `channel:C123`, `user:U123`, and `<@U123>` match the raw Slack IDs received at runtime, while preserving existing raw Slack ID configs and canonical raw IDs in session keys. Add focused regression tests for channel and direct bindings plus config help or warnings if raw provider IDs remain the preferred documented form.
