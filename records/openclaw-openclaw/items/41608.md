## Summary

Keep this issue open. Current main still passes raw Slack event peer IDs into generic route matching, while configured binding peer IDs are trim-normalized and matched literally. A Slack binding written as `channel:C...` can still miss the runtime peer ID `C...`, fall through to the default agent, and produce the reported `agent:main:slack:channel:...` session key. The related PR that targeted this issue is closed and unmerged in the supplied GitHub context, so there is no shipped fix to close against.

## Best Possible Solution

Keep this issue open. A conservative fix should normalize Slack route-binding peer IDs before generic route matching while preserving canonical raw Slack IDs in session keys. It should accept raw IDs plus target-style forms such as `channel:C123`, `user:U123`, and `<@U123>`, add focused Slack channel/group and DM binding regression tests, and update config help/docs if raw IDs remain the preferred documented binding form.
