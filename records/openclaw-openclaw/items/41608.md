## Summary

Keep this issue open. Current main still appears to miss the reported Slack route-binding shape: Slack inbound routing passes raw Slack IDs into the generic route resolver, while route binding peer IDs are only trim-normalized and then compared literally. A binding configured as `channel:C...` can still fall through to the default agent and create `agent:main:slack:...` session keys. The related PR #42162 was closed unmerged, so it is not shipped evidence of a fix.

## Best Possible Solution

Normalize Slack route-binding peer IDs before generic route matching while preserving the canonical raw Slack ID session-key shape. A conservative fix should accept raw IDs and target-style IDs such as `channel:C123`, `user:U123`, and `<@U123>`, add focused regression tests for Slack channel/group and DM bindings, and update config help or doctor guidance if raw IDs remain the preferred documented form.
