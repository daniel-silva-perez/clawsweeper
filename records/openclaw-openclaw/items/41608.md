## Summary

Current main still appears to have the reported Slack route-binding gap. Slack inbound routing passes raw Slack IDs into the generic route resolver, while configured route-binding peer IDs are only trim-normalized and then compared literally. A binding using `channel:C...` or `user:U...` can still miss and fall back to the default agent, producing the observed `agent:main:slack:channel:...` session key. The related PR #42162 was closed unmerged, so it is not shipped evidence of a fix.

## Best Possible Solution

Keep this issue open. A conservative fix should normalize Slack route-binding peer IDs before generic route matching while preserving the canonical raw Slack ID session-key shape. It should accept raw IDs plus target-style forms such as `channel:C123`, `user:U123`, and `<@U123>`, add focused Slack channel/group and DM binding regression tests, and update config help or doctor guidance if raw IDs remain the preferred documented form.
