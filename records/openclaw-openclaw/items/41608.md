## Summary

Keep open. Current main still routes Slack inbound messages with raw Slack peer IDs, stores configured binding peer IDs with only generic trim normalization, and compares peer IDs exactly. A binding configured as `channel:C...` can still miss a runtime peer of `C...` and fall back to the default `main` agent. The linked implementation candidate, #42162, directly addressed this but is closed and unmerged; the later referenced fork merge is not in current `openclaw/openclaw` main.

## Best Possible Solution

Keep this issue open. The best fix is to normalize Slack route-binding peer IDs before generic matching, accepting target-style forms such as `channel:C123`, `user:U123`, and `<@U123>` while preserving existing raw Slack ID configs and raw canonical IDs in session keys. Add focused regression coverage for channel and direct bindings plus config help/warnings if raw provider IDs remain the preferred documented form.
