## Summary

Current main still does not normalize Slack route-binding peer IDs such as `channel:C...` before generic route matching. Slack inbound routing passes raw Slack IDs, route bindings keep peer IDs literal after trim-normalization, and peer matching is exact, so the reported binding can still fall through to the default `main` agent.

## Best Possible Solution

Keep this issue open. The conservative fix belongs in the Slack routing/binding normalization path: normalize Slack route-binding peer IDs before generic matching, preserve canonical raw Slack IDs in session keys, add focused regression coverage for prefixed Slack channel/user binding IDs, and update config help/docs if raw provider IDs remain the preferred form.

## Evidence
