## Summary

Keep this issue open. Current main still routes Slack inbound events into the generic resolver with raw Slack IDs, while route-binding peer IDs are trim-normalized and matched literally. A binding configured as `channel:C...` can still miss a runtime peer id of `C...` and fall back to the default agent, which matches the reported `agent:main:slack:channel:...` behavior. The related PR #42162 is closed and unmerged in the provided GitHub context, so it is not shipped evidence of a fix.

## Best Possible Solution

Keep this issue open. A conservative fix should normalize Slack route-binding peer IDs before generic route matching while preserving canonical raw Slack ID session keys. It should accept raw IDs plus target-style forms such as `channel:C123`, `user:U123`, and `<@U123>`, add focused Slack channel/group and DM binding regression tests, and update config help or docs if raw IDs remain the preferred documented binding form.
