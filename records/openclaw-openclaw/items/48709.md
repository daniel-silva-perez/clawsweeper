## Summary

Keep open. Current main still stores Gemini visible text signatures on assistant text blocks and replays them as Gemini thoughtSignature on same-provider history, while the generic replay sanitizer only targets thought_signature/thoughtSignature fields. The mixed text-before-tool path now has partial flush protection, but the signature bloat and Gemini tagged-reasoning/final-tag suppression portions remain unresolved.

## What This Changes

Request that Gemini 2.5 Pro sessions stop accumulating and replaying large textSignature blobs, avoid suppressing embedded think/final-tag text, and deliver text emitted before tool calls.

## Best Possible Solution

Handle this at the Google provider and replay boundary: keep only Gemini signatures that are required for valid continuation, bound or avoid replaying stale visible-text signatures, and move Gemini reasoning delivery toward native thinking or a provider-specific fallback that cannot silently suppress final-less visible replies.

## Reproduction Assessment

Yes. The issue has a high-confidence source-level reproduction path: a same-provider Google assistant text block with textSignature is serialized back to Gemini as thoughtSignature, and enforced final-tag delivery suppresses text without a <final> start tag; the live Gemini/Telegram loop was not run in this read-only pass.

## Solution Assessment

No current-main solution is complete. The text-before-tool flush path is partially addressed, but the remaining signature persistence/replay and Gemini tagged-versus-native reasoning choices need a provider-owner decision before a narrow implementation should proceed.
