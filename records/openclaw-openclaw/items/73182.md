## Summary

Keep open. Current main has partial mitigation, including a global reasoning visibility default and an Opus 4.7 thinking default of off, but source, docs, and tests still encode non-off API-thinking defaults for Claude 4.6 and generic reasoning-capable catalog models, unset cron payloads still fall through to the default resolver, and the Microsoft Teams renderer still lacks a reasoning-payload suppression guard.

## What This Changes

The issue requests cost-safe opt-in API thinking defaults for Claude/reasoning-capable models, global and cron-safe default handling, startup/doctor disclosure for billing-impacting reasoning, and separation of API thinking from visible reasoning output in messaging channels.

## Best Possible Solution

Define one explicit product policy: API thinking is off unless explicitly configured or required by the provider/model, cron jobs inherit the same resolved agent/global policy without silent medium defaults, reasoning payloads are suppressed in messaging channels unless the surface explicitly requests them, and docs/release notes call out billing-affecting default changes before upgrade.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is available by resolving default thinking for a reasoning-capable catalog model or Claude 4.6 with no explicit thinking setting, and by passing a reasoning-marked payload through the Teams `renderReplyPayloadsToMessages` path.

## Solution Assessment

No. Current main is only a partial mitigation: global reasoning visibility config and Opus 4.7 off are present, but the broader cost-safe API-thinking default, cron audit/inheritance behavior, release-note disclosure, and Teams suppression gap remain unresolved.
