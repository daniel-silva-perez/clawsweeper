## Summary

Keep open. Current main has partial mitigation, including global reasoning visibility config and an Opus 4.7 API-thinking default of off, but the central cost-safety and disclosure request is not fully implemented: Claude 4.6 and generic reasoning-capable catalog models still resolve to non-off API thinking, unset cron agent-turn payloads still inherit the resolver, model reasoning visibility can still flip on when thinking is off, and Microsoft Teams still lacks the reasoning-payload suppression guard present in Mattermost.

## What This Changes

The issue requests opt-in API thinking for Claude/reasoning-capable models, a global default, startup or doctor warnings for enabled thinking, cron inheritance/auditing, and separation of API-level thinking from chat-visible reasoning output.

## Best Possible Solution

Define one explicit product policy: API thinking is off unless configured or required by the provider/model, cron jobs inherit the same resolved agent/global policy without silent medium defaults, reasoning payloads are suppressed in messaging channels unless the surface explicitly supports and requests them, and docs/release notes call out billing-affecting default changes before upgrade.

## Reproduction Assessment

Yes. The current source and tests provide a high-confidence reproduction path: leave thinking unset on Claude 4.6 or a reasoning-capable catalog model, or on an isolated cron agentTurn payload, and the resolver reaches a non-off thinking default; Teams rendering can be checked by passing a reasoning-marked payload through renderReplyPayloadsToMessages.

## Solution Assessment

No. Current main is only a partial mitigation because it special-cases Opus 4.7 and exposes reasoning visibility defaults, but it does not fully make API thinking opt-in or suppress Teams reasoning payloads.
