## Summary

Keep open: current main still treats a WhatsApp QR-login 408 as a terminal login failure, so the PR's intended behavior is not implemented. The source patch is useful but should not merge as-is because its retry path bypasses the shared socket-replacement contract and can churn sockets under persistent timeouts.

## What This Changes

The PR adds a 408 timeout branch to WhatsApp QR-login waiting that closes and recreates the socket, plus a timed-out log branch in the WhatsApp session socket handler.

## Best Possible Solution

Move the useful 408 handling into the shared WhatsApp login waiter as a bounded `DisconnectReason.timedOut` restart path that preserves `authDir`, `isLegacyAuthDir`, `socketTiming`, `onQr`, and `onSocketReplaced`, with regression coverage for retry success, retry exhaustion, QR refresh preservation, and unchanged 401/515 behavior.

## Reproduction Assessment

Yes. A focused unit reproduction can mock `waitForWaConnection` to reject with status 408: current main returns a terminal QR-login failure, while the PR can be inspected to show callback loss and unbounded retry without needing a live WhatsApp account.

## Solution Assessment

No. The PR targets the right symptom, but the outer `waitForWebLogin` retry is not the best implementation because the existing shared waiter already owns socket replacement and callback preservation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve QR callbacks when retrying 408 sockets:** `extensions/whatsapp/src/login-qr.ts:287-289`
  - body: The 408 retry creates the replacement socket with only `authDir`. `createWaSocket` captures `onQr` when the socket is constructed, so fresh QR rotations after the timeout will not update active login state or `qrDataUrl`, leaving users with a stale QR.
  - confidence: 0.91
- **[P2] Bound repeated 408 socket recreation:** `extensions/whatsapp/src/login-qr.ts:283-298`
  - body: A persistent 408 re-enters this branch each time the replacement waiter fails until the outer wait deadline, repeatedly closing and opening sockets. Add an explicit retry cap or handle 408 in the shared waiter with a bounded restart contract.
  - confidence: 0.86
- **[P3] Fix the timeout log comment:** `extensions/whatsapp/src/session.ts:150`
  - body: The new session branch only logs the timeout; it does not trigger reconnect from this handler. The comment should not imply reconnect happens here because recovery belongs in the login waiter or connection controller.
  - confidence: 0.78
