## Summary

Keep open. Current main still does not implement a 408 QR-login socket retry, and this PR is useful but not merge-ready: the proposed retry is unbounded, does not preserve the QR callback or socket timing/options, and lacks focused 408 regression coverage. The security pass did not find CI, dependency, lockfile, package, publishing, or secrets changes; the main risk is operational resource churn from repeated socket recreation under persistent 408s.

## What This Changes

This PR modifies the WhatsApp Web QR-login flow to recreate the socket after a 408 timeout in `waitForWebLogin` and adds a 408 timeout log branch in the WhatsApp session socket handler.

## Best Possible Solution

Land a narrow WhatsApp plugin fix that treats Baileys 408/time-out as a bounded retryable QR-login condition, preferably in the shared login-waiter path, while preserving `authDir`, legacy auth handling, socket timing, QR callbacks, socket replacement state, 515 behavior, and 401 logout handling. Keep the sendUnifiedSession handshake work and post-408 listener recovery work separate unless tests or live proof show overlap.
