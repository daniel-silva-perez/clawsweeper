## Summary

Keep open. Current main still does not implement bounded 408/timedOut QR-login socket retry, and the PR's proposed implementation is not merge-ready because it recreates the socket outside the existing state-preserving waiter contract and can churn sockets repeatedly until the wait deadline.

## What This Changes

The PR adds a WhatsApp Web QR-login branch that closes and recreates the socket after 408 timeout errors, plus a session log branch for `DisconnectReason.timedOut`.

## Best Possible Solution

Port the useful idea into the shared WhatsApp login waiter as a bounded `DisconnectReason.timedOut` retry path that preserves `authDir`, `isLegacyAuthDir`, socket timing, QR callbacks, and socket replacement state, with regression tests for retry success, retry exhaustion, QR refresh preservation, and unchanged 401/515 behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve QR callbacks on 408 socket retries:** `extensions/whatsapp/src/login-qr.ts:287-289`
  - body: The retry creates the replacement socket with only `authDir`. `createWaSocket` captures `onQr` when the socket is constructed, so a fresh QR emitted after the timeout will not update the active login state and users can keep seeing the expired QR. Reuse the shared restart path or pass the same QR/timing callbacks when recreating the socket.
  - confidence: 0.91
- **[P2] Bound repeated 408 socket recreation:** `extensions/whatsapp/src/login-qr.ts:283-298`
  - body: A persistent 408 re-enters this branch each time the new waiter fails until the outer wait deadline, repeatedly closing and opening sockets without a per-login cap. Track a bounded retry count, or handle 408 in the shared waiter with an explicit maximum, so network faults do not churn sockets and logs for the full wait window.
  - confidence: 0.86
