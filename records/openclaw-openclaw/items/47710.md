## Summary

Keep open. Current main still has only the existing one-time 515 QR-login socket restart path; a 408/timedOut login error still falls through to terminal failure. The PR branch is useful source work but is not mergeable as-is because its 408 retry is unbounded and recreates the socket without preserving the QR callback or socket timing options already used by the shared login waiter.

## What This Changes

The PR adds WhatsApp Web QR-login handling that closes and recreates the socket after a 408 timeout in `waitForWebLogin`, plus a 408 timeout log branch in the WhatsApp session socket handler.

## Best Possible Solution

Land a narrow WhatsApp plugin implementation that treats Baileys 408/timedOut during QR login as a bounded retryable socket-restart condition in the shared login waiter, with tests proving retry success, retry exhaustion, callback/timing preservation, and unchanged 515/401 behavior. Keep sendUnifiedSession handshake work and post-408 listener recovery as separate tracks unless tests or live evidence show a shared root cause.
