## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks bounded QR-login recovery for a 408 timeout, while the proposed patch recreates sockets outside the existing restart contract and can retry repeatedly under a persistent timeout.

## What This Changes

The PR adds WhatsApp Web QR-login socket recreation on 408 timeout in `waitForWebLogin` and logs `DisconnectReason.timedOut` in the WhatsApp session `connection.update` handler.

## Best Possible Solution

Port the idea into the shared WhatsApp login waiter as a bounded `DisconnectReason.timedOut`/408 restart path that preserves `authDir`, `isLegacyAuthDir`, socket timing, QR callbacks, and active-login socket replacement state, with regression tests for retry success, retry exhaustion, refreshed QR propagation, and unchanged 401/515 behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve QR callbacks on 408 retries:** `extensions/whatsapp/src/login-qr.ts:287-289`
  - body: The 408 retry creates the replacement socket with only `authDir`. `createWaSocket` captures `onQr` at construction, so a fresh QR emitted after the timeout is not written back into active login state; users can keep seeing the expired QR. Reuse the shared restart path or pass the same QR/timing callbacks when recreating the socket.
  - confidence: 0.91
- **[P2] Bound repeated 408 socket retries:** `extensions/whatsapp/src/login-qr.ts:283-298`
  - body: A persistent 408 re-enters this branch each time the replacement waiter fails until the outer wait deadline, repeatedly closing and opening sockets without a per-login cap. Track a bounded retry count, or handle 408 in the shared waiter with an explicit maximum, so network faults do not churn sockets and logs for the full wait window.
  - confidence: 0.86
