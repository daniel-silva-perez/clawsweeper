## Summary

Keep open. Current main still treats WhatsApp QR-login 408 failures as terminal, and this PR is useful but not merge-ready because its retry path recreates the socket outside the existing state-preserving restart contract and has no retry bound. Related merged WhatsApp work covers QR rotation, 515 restart, proxy/socket timing, and listener recovery, but not this QR-timeout retry.

## What This Changes

The PR adds WhatsApp Web QR-login socket recreation on 408 timeout in `waitForWebLogin` and logs `DisconnectReason.timedOut` in the session `connection.update` handler.

## Best Possible Solution

Implement the retry in the shared WhatsApp login waiter as a bounded `DisconnectReason.timedOut`/408 restart path that reuses the existing replacement machinery, preserves QR callbacks, socket timing, auth-dir context, and active-login socket state, and documents only the QR-login timeout case it actually fixes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve QR callbacks on 408 retries:** `extensions/whatsapp/src/login-qr.ts:287-289`
  - body: The retry creates the replacement socket with only `authDir`. `createWaSocket` captures `onQr` at construction, so a fresh QR emitted after the timeout is not written back to active login state and users can keep seeing the expired QR. Reuse the shared restart path or pass the same QR/timing callbacks when recreating the socket.
  - confidence: 0.91
- **[P2] Bound repeated 408 socket retries:** `extensions/whatsapp/src/login-qr.ts:283-298`
  - body: A persistent 408 re-enters this branch each time the new waiter fails until the outer wait deadline, repeatedly closing and opening sockets without a per-login cap. Track a bounded retry count, or handle 408 in the shared waiter with an explicit maximum, so network faults do not churn sockets and logs for the full wait window.
  - confidence: 0.86
