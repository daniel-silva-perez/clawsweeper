## Summary

Keep open. Current main still treats WhatsApp QR-login 408 timeouts as terminal failures, and this PR targets a real gap, but the patch is not merge-ready because its replacement socket drops the active QR/timing callback contract and can recreate sockets repeatedly until the wait deadline.

## What This Changes

The PR adds a 408 timeout branch to WhatsApp QR-login waiting that closes and recreates the socket, plus a timed-out log branch in the WhatsApp session socket handler.

## Best Possible Solution

Move the useful 408 handling into `waitForWhatsAppLoginResult` as a bounded `DisconnectReason.timedOut` restart path that preserves `authDir`, `isLegacyAuthDir`, `socketTiming`, `onQr`, and `onSocketReplaced`, with tests for retry success, retry exhaustion, QR refresh preservation, and unchanged 401/515 behavior.

## Reproduction Assessment

Yes. A focused unit reproduction can mock `waitForWaConnection` to reject with status 408: current main returns a terminal QR-login failure, while this PR can be inspected to show callback loss and unbounded retry behavior without a live WhatsApp account.

## Solution Assessment

No. The PR targets the right symptom, but the best maintainable fix belongs in the shared WhatsApp login waiter, where socket replacement already has the callback-preserving restart contract and can enforce a bounded retry policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve QR callbacks and socket timing:** `extensions/whatsapp/src/login-qr.ts:287-289`
  - body: The 408 retry creates the replacement socket with only `authDir`. `createWaSocket` captures `onQr` and timing options when the socket is constructed, so fresh QR rotations after the timeout will not update active login state and the retry can also lose configured socket timings.
  - confidence: 0.91
- **[P2] Bound repeated 408 socket recreation:** `extensions/whatsapp/src/login-qr.ts:283-298`
  - body: A persistent 408 re-enters this branch each time the replacement waiter fails until the outer wait deadline, repeatedly closing and opening sockets. Add an explicit retry cap or move 408 handling into the shared waiter with a bounded restart contract.
  - confidence: 0.87
- **[P3] Match timedOut exactly:** `extensions/whatsapp/src/login-qr.ts:283`
  - body: `String(login.errorStatus).includes('408')` is an imprecise match for a numeric status and can retry on unrelated values such as 1408 or 4080. Use `DisconnectReason.timedOut` or an exact 408 comparison.
  - confidence: 0.8
- **[P3] Fix the timeout reconnect comment:** `extensions/whatsapp/src/session.ts:150`
  - body: The session handler branch only logs the 408 timeout; it does not trigger a reconnect from that handler. The comment should not imply recovery happens there.
  - confidence: 0.78
