## Summary

Keep open: current main still treats WhatsApp QR-login 408 errors as terminal, so the PR’s underlying bug is not implemented. The proposed patch is not mergeable as-is because it recreates sockets outside the shared login-waiter contract, drops QR/timing/state callbacks, and has no bounded retry policy.

## What This Changes

The PR adds WhatsApp Web QR-login socket recreation after a 408 timeout in `waitForWebLogin` and adds a timed-out connection log branch in `createWaSocket`.

## Best Possible Solution

Implement the 408/timedOut retry inside the shared WhatsApp login waiter as a bounded socket-restart path that preserves auth directory, legacy-auth flag, socket timing, QR refresh callbacks, and active-login socket replacement state, with regression tests for retry success, retry exhaustion, refreshed QR propagation, and unchanged 401/515 behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve QR callbacks on 408 retries:** `extensions/whatsapp/src/login-qr.ts:287-289`
  - body: The retry creates the replacement socket with only `authDir`. `createWaSocket` captures `onQr` when the socket is constructed, so a fresh QR emitted after the timeout is not written back into active login state and users can keep seeing the expired QR. Reuse the shared restart path or pass the same QR/timing/replacement callbacks.
  - confidence: 0.91
- **[P2] Bound repeated 408 socket retries:** `extensions/whatsapp/src/login-qr.ts:283-298`
  - body: A persistent 408 re-enters this branch each time the replacement waiter fails until the outer wait deadline, repeatedly closing and opening sockets without a per-login cap. Track a bounded retry count, or handle 408 in the shared waiter with an explicit maximum, so network faults do not churn sockets and logs for the full wait window.
  - confidence: 0.86
