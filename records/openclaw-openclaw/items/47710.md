## Summary

Keep this PR open: the 408 WhatsApp QR-login problem is not implemented on current main, but the proposed retry path is not merge-ready because it bypasses the existing socket-replacement callback contract and has no retry bound.

## What This Changes

The PR adds WhatsApp QR-login handling that closes and recreates the socket after a 408 timeout in `waitForWebLogin`, plus a timed-out log branch in the WhatsApp session socket handler.

## Best Possible Solution

Port the useful idea into the shared WhatsApp login waiter as a bounded 408/timedOut socket-restart path that preserves `authDir`, `isLegacyAuthDir`, `socketTiming`, `onQr`, and `onSocketReplaced`, with regression coverage for retry success, retry exhaustion, refreshed QR propagation, and unchanged 401/515 behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve QR callbacks on 408 retries:** `extensions/whatsapp/src/login-qr.ts:287-289`
  - body: The 408 retry creates the replacement socket with only `authDir`. `createWaSocket` captures `onQr` when the socket is constructed, so any fresh QR emitted after the timeout is not written back to the active login state and users can keep seeing an expired QR.
  - confidence: 0.91
- **[P2] Bound repeated 408 socket retries:** `extensions/whatsapp/src/login-qr.ts:283-298`
  - body: A persistent 408 re-enters this branch each time the replacement waiter fails until the outer wait deadline, repeatedly closing and opening sockets. Add an explicit retry cap or handle 408 in the shared waiter with the same kind of bounded restart contract as the 515 path.
  - confidence: 0.86
