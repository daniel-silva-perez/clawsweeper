## Summary

Keep open. Current main still has no `===DONE_OK===` cron classifier contract, so the PR is addressing a real cron false-positive surface, but the proposed implementation is not merge-ready because it trusts the sentinel anywhere in payload history and can hide a later real failure.

## What This Changes

The PR changes the isolated cron payload resolver and tests so `===DONE_OK===` in final assistant text or payload text makes trailing error payloads non-fatal unless a run-level error is present.

## Best Possible Solution

Keep the draft open for maintainer review. If the direction is accepted, rebase onto current main and implement a narrow cron success contract that trusts only an intentional final success signal or equivalent structured metadata, without weakening structured failure signals, denial markers, run-level failures, or real trailing-error handling.

## Reproduction Assessment

Yes. A focused call to `resolveCronPayloadOutcome` with successful assistant output followed by a final `isError: true` payload reproduces current-main fatal classification, and `run.ts` persists that flag as `status: "error"`.

## Solution Assessment

No, not as written. The sentinel direction may be acceptable, but matching against every payload text is too broad because earlier or quoted sentinel text can suppress a later real failure; the safer fix is to constrain the trusted success signal and rebase it into the current classifier.
