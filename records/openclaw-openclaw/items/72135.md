## Summary

Keep open. Current main still has no native `DONE_OK`/`CRON_OK` success sentinel, so the draft is not obsolete, but the PR should not merge as written because it trusts the sentinel from any payload text and needs reconciliation with the newer current-main cron failure, denial, and run-level classifier paths.

## What This Changes

The draft PR adds `===DONE_OK===` sentinel detection to isolated cron payload resolution and tests so trailing error payloads can be classified as non-fatal unless a run-level error is present.

## Best Possible Solution

If maintainers accept the concept, implement the sentinel on current main as a narrow cron classifier contract: trust only a canonical final success signal or tightly scoped last successful output, clear only the structured trailing-error false positive, keep typed failure signals/denials/run-level errors fatal, and add focused tests in the current resolver test layout plus docs if the token is public.

## Reproduction Assessment

Yes. A direct unit call to `resolveCronPayloadOutcome` with a successful text payload or final assistant text followed by a final `isError: true` payload reproduces current-main fatal classification, and `run.ts` maps that flag to `status: "error"`.

## Solution Assessment

No, not as written. A success sentinel may be a reasonable fix direction, but scanning every payload text is too broad and the patch must be rebased into the current classifier so newer fatal signal paths still win.
