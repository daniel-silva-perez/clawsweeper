## Summary

Keep open. Current main still does not implement a `===DONE_OK===` cron success sentinel, so this draft is not obsolete, but the proposed broad payload-text scan is not the best mergeable shape because it can clear a later real tool error and the branch predates the current classifier’s typed failure and denial paths.

## What This Changes

The PR changes the isolated cron payload resolver and tests so `===DONE_OK===` in final assistant text or any payload text makes trailing error payloads non-fatal unless a run-level error is present.

## Best Possible Solution

Keep the draft open for maintainer product review. If accepted, rebase it onto current main and implement the sentinel as a narrow cron classifier contract with focused resolver tests in the current test layout, explicit token/scope semantics, and no weakening of typed failure signals, denial markers, run-level errors, or real trailing-error handling.

## Reproduction Assessment

Yes. A focused unit call to `resolveCronPayloadOutcome` with a successful payload or final assistant text followed by a final `isError: true` payload reproduces the current fatal classification, and `src/cron/isolated-agent/run.ts` shows that flag becomes `status: "error"`.

## Solution Assessment

No, not as written. The direction may be acceptable, but the implementation should restrict where the sentinel is trusted and be rebased into the current classifier so it cannot override newer fatal-signal and denial paths.
