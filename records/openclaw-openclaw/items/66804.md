## Summary

Keep open. Current main has meaningful Active Memory timeout mitigations, but it still does not implement the reported fallback-after-timeout behavior and still returns no injected memory when the selected recall model times out without partial text. The recent open related reports also show the Active Memory timeout cluster is not fully resolved.

## What This Changes

Active Memory should complete or fall back when MiniMax-M2.7 times out instead of returning `summaryChars=0` after the configured timeout.

## Best Possible Solution

Keep this open until maintainers either choose a canonical Active Memory timeout issue and link these MiniMax-M2.7 logs there, or implement a deliberate runtime behavior for slow recall models: bounded fail-open, local memory-search summary recovery, or a new explicit timeout-fallback model setting that does not overload the current selection-only `modelFallback`.

## Reproduction Assessment

Yes, externally: configure Active Memory to use `minimax/MiniMax-M2.7` or `minimax-portal/MiniMax-M2.7`, enable logging, and watch for `status=timeout` with `summaryChars=0`; multiple users provided matching logs. I did not perform a live MiniMax run in this read-only review.

## Solution Assessment

No, current main is not the complete solution for this report. It improves hard deadlines, partial transcript recovery, thinking defaults, and repeated-timeout suppression, but it does not complete recall or switch to another/local fallback when MiniMax times out.
