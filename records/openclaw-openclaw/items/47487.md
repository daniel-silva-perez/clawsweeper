## Summary

Keep open. Current main still widens restricted tool profiles when a global or agent `tools.exec` section exists, and that implicit grant is unioned with explicit `alsoAllow`; the focused fix PR was closed unmerged. Because this touches an exec authorization boundary, it needs maintainer policy review before changing behavior.

## What This Changes

The issue requests that an agent using `tools.profile: "messaging"` with an explicit `alsoAllow` list excluding `exec` not receive `exec` merely because `tools.exec` is configured.

## Best Possible Solution

Revise the tool-policy contract so global `tools.exec` configuration cannot widen restricted agent profiles, preserve only intentionally explicit per-agent opt-in semantics, add regression coverage for `messaging` plus explicit `alsoAllow`, document the global-versus-agent behavior, and review the analogous `tools.fs` path in the same policy pass.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: configure global `tools.exec`, set an agent to `tools.profile: "messaging"` with `alsoAllow` excluding `exec`, and `resolveEffectiveToolPolicy()` still produces a `profileAlsoAllow` union containing `exec` and `process`.

## Solution Assessment

No. Current main's implicit grant mixes exec safety configuration with exec access; the safer maintainable fix is to require explicit profile/allow opt-in for runtime access while keeping deny precedence intact.
