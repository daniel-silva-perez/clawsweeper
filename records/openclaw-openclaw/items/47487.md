## Summary

Keep open. Current main still matches the reported policy bypass: a configured `tools.exec` section is converted into implicit `exec`/`process` profile allowances and then unioned with explicit `alsoAllow`, so omitting `exec` from a restricted profile's `alsoAllow` does not block it. The focused fix PR #47991 is closed unmerged, while #42165 is an adjacent filesystem analogue and #69190 is broader rather than a canonical replacement.

## What This Changes

Ensure restricted agent tool profiles are not widened to `exec` merely because `tools.exec` is configured outside the explicit allowed tool list.

## Best Possible Solution

Revise the tool-policy contract so global `tools.exec` configuration does not widen restricted agent profiles, preserve deny precedence, add focused regression coverage for `messaging` plus explicit `alsoAllow`, document global-versus-agent semantics, and handle the `tools.fs` analogue consistently.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure global `tools.exec`, set an agent to `tools.profile: "messaging"` with `alsoAllow` excluding `exec`, and `resolveEffectiveToolPolicy()` still produces `profileAlsoAllow` containing `exec` and `process`.

## Solution Assessment

No. Current main conflates exec safety configuration with exec access; the safer maintainable fix is to require explicit profile/allow opt-in for runtime access while preserving deny precedence and documenting any intentional per-agent exception.
