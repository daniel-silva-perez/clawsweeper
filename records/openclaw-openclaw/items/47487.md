## Summary

Keep open. Current main still widens restricted tool profiles when a global or agent `tools.exec` block exists, and the focused fix PR linked from the issue was closed unmerged. The report remains valid but touches an exec authorization boundary that needs maintainer policy judgment.

## What This Changes

The issue asks for an agent with `tools.profile: "messaging"` and an explicit `alsoAllow` list excluding `exec` to block `exec`, even when global `tools.exec` safety settings are configured.

## Best Possible Solution

Revise the tool-policy contract so global `tools.exec` configuration cannot widen restricted agent profiles, preserve any intended per-agent explicit opt-in semantics, add regression coverage for `messaging` plus explicit `alsoAllow`, document the global-versus-agent behavior, and review the analogous `tools.fs` path in the same policy pass.

## Reproduction Assessment

Yes. The source path is enough to reproduce at policy-resolution level: configure global `tools.exec`, set an agent to `tools.profile: "messaging"` with explicit `alsoAllow` excluding `exec`, and `resolveEffectiveToolPolicy()` still unions in `exec` and `process`.

## Solution Assessment

No. Current main's implicit global tool-section grant is not the best way to solve the issue because it mixes exec safety configuration with exec access; the safer direction is to require explicit profile/allow opt-in for access and keep deny precedence intact.
