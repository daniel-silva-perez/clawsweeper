## Summary

Keep open. Current main still applies cron update patches to the stored job before cron expression validation can fail, so disabled schedule edits can persist invalid cron expressions and a failed enable can leave the live in-memory job enabled. The merged #74193 gateway fix changes the error classification but does not make service updates atomic, and the open #74068 follow-up is scoped to narrowing that gateway catch.

## What This Changes

Reject invalid cron expressions atomically during `cron edit` and `cron enable` so disabled jobs keep their prior valid schedule and enabled state after validation failures.

## Best Possible Solution

Make cron updates build and validate a staged post-patch job first, including cron parsing when the schedule changes or a job is being enabled, then commit the staged job only after next-run computation succeeds. Add focused service regression coverage for disabled invalid edits and failed enable rollback, plus gateway coverage for the visible `INVALID_REQUEST` behavior.

## Repair Work Prompt

Fix this issue: `openclaw cron edit <id> --cron <invalid>` currently persists invalid cron expressions on disabled jobs, and `openclaw cron enable <id>` can return `INVALID_REQUEST` while leaving the live job enabled. Keep the fix narrow. In `src/cron/service/ops.ts` / `src/cron/service/jobs.ts`, stage the full post-patch job on a clone, validate/apply schedule computation before mutating the stored job, and commit only after validation succeeds. Ensure disabled schedule edits still validate cron syntax, and failed enable attempts leave the previous `enabled` and `schedule` values intact. Preserve existing legacy persisted-job schedule-error isolation behavior. Add focused regression tests in `src/cron/service.issue-regressions.test.ts` and, if needed, gateway coverage in `src/gateway/server-methods/cron.validation.test.ts`. Do not broaden Croner error classification beyond the existing gateway behavior or take over the separate #74068 scope. Add a single-line user-facing changelog fix entry with `Thanks @bittoby`.
