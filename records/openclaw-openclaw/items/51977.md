## Summary

Keep this issue open. Current main has important mitigations for the reported cron amplification path, including stable-key cron persistence, cron run-session retention, session-store entry caps, and disk-budget cleanup. However, the central rendered-prompt retention problem is still present: `skillsSnapshot.prompt` remains part of the persisted `SessionEntry` shape, normal reply and CLI paths still write full snapshots into `sessions.json`, reset still copies the snapshot forward, follow-up run payloads still carry it, and the session-store cache still retains both structured-cloned store objects and serialized JSON strings. The discussion and related ClawSweeper context are directionally supported by current source: this is improved, but not fully fixed.

## Best Possible Solution

Keep the issue open and fix the remaining root cause by making persisted session state metadata-only: store skill list/filter/version/hash, rebuild or share the rendered prompt lazily, strip `skillsSnapshot.prompt` from `sessions.json` and queued/reset/follow-up payloads, and add regression coverage that large rendered skills prompts are not serialized into retained session entries. Keep the current cron stable-key and session-maintenance mitigations as guardrails.
