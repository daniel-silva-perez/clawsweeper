## Summary

Keep this issue open. Current main has real mitigations for the original cron amplification path, but it still persists the full rendered `skillsSnapshot.prompt` in `SessionEntry`, serializes it into `sessions.json`, keeps parsed and serialized cache forms, copies it through reset, and carries it in queued follow-up run payloads. The central retained rendered-prompt problem is not fixed yet.

## Best Possible Solution

Keep the issue open and fix the root persistence model: make session skill snapshots metadata-only in durable state, store skill list/filter/version/hash rather than rendered prompt text, rebuild or share the rendered prompt lazily by hash/version/filter, strip full snapshots from reset and queued follow-up payloads, and add regression coverage proving `sessions.json`, reset, follow-up, and cron paths do not retain large rendered skill prompts. Keep the current cron stable-key and session-maintenance mitigations as guardrails.
