## Summary

Keep open. Current main still has an iMessage DM-history contract mismatch: the schema/types accept `dmHistoryLimit` and the docs still list iMessage as supported, but the runtime path only builds supplemental history for group messages. The reporter's `attributedBody` detail remains useful for an eventual implementation, but the current-main root cause is broader than a bad `message.text` query: direct-message history is not implemented for the iMessage plugin.

## What This Changes

The issue requests fixing iMessage direct-message history injection so `dmHistoryLimit` preserves decoded outbound message text instead of injecting blank history entries.

## Best Possible Solution

Maintainers should choose the iMessage contract explicitly: either add a decoded iMessage DM-history adapter in the iMessage plugin, using `imsg` output that preserves outbound message text, or stop advertising iMessage as supporting `dmHistoryLimit` while keeping backward-compatible config acceptance if needed.

## Reproduction Assessment

Yes for the current-main contract gap: direct iMessage dispatches have no `historyKey`, and `buildIMessageInboundContext` only sets history for groups. The exact blank-outbound macOS database behavior is not live-reproduced here, but the source path confirms direct-message history is not injected at all.

## Solution Assessment

Unclear. Using decoded `imsg` history is the right runtime direction if iMessage DM history remains supported, but a docs-honest de-advertising path was also proposed and closed unmerged; maintainer product judgment should decide which contract OpenClaw wants for the legacy iMessage plugin.
