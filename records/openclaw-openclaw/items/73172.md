## Summary

Keep open. Current main still has the iMessage DM-history contract mismatch: config/schema/docs expose `dmHistoryLimit` for iMessage, but the iMessage monitor only resolves group `historyLimit` and only attaches `InboundHistory` for group decisions. The reporter's `attributedBody` evidence remains useful for any runtime implementation, but the source-backed current-main issue is broader than blank outbound rows: direct-message history loading is not implemented for the legacy iMessage plugin, and the docs-only fix PR #73775 was closed unmerged.

## What This Changes

The issue asks for iMessage direct-message history injected by `dmHistoryLimit` to use decoded `imsg` history/RPC output so outbound messages are not blank when macOS stores text in `attributedBody` instead of `message.text`.

## Best Possible Solution

Maintainers should make the iMessage contract explicit: either add a decoded iMessage direct-message history adapter in the iMessage plugin with regression coverage for outbound message text, or remove iMessage from the documented `dmHistoryLimit` supported list while keeping backward-compatible config parsing if needed.

## Reproduction Assessment

Yes for the current-main contract gap: direct iMessage decisions have no `historyKey`, and `InboundHistory` is only populated for group decisions. The reporter's specific macOS database `attributedBody` evidence was not live-verified here, but it is relevant implementation guidance if runtime support is kept.

## Solution Assessment

Unclear. Using decoded `imsg` history/RPC output is the right implementation direction if OpenClaw intends to support iMessage DM history, but the docs-honest de-advertising path in #73775 was also plausible and closed unmerged, so an automated repair should not choose the contract on its own.
