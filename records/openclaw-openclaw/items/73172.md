## Summary

Keep open. Current main still has the iMessage direct-message history contract mismatch: config and docs accept/advertise `dmHistoryLimit` for iMessage, while the runtime only resolves and injects pending group history. The reporter's `attributedBody` evidence remains useful for a runtime implementation, but current source shows the broader problem is that iMessage DM history is not implemented at all; the related docs-only PR #73775 is closed unmerged and does not supersede this issue.

## What This Changes

The issue requests that iMessage `dmHistoryLimit` history injection use decoded `imsg` history/RPC output so outbound Messages content is not injected as blank.

## Best Possible Solution

Choose and make the iMessage contract explicit: either implement a decoded iMessage DM-history adapter in the iMessage plugin, using `imsg` output that preserves outbound text, or stop advertising iMessage as supporting `dmHistoryLimit` while preserving backward-compatible config acceptance if needed.

## Reproduction Assessment

Yes for the current-main contract gap: direct iMessage dispatches do not get a `historyKey`, and `buildIMessageInboundContext` only injects history for groups. The reporter's narrower blank outbound `attributedBody` behavior still needs a macOS/imsg live check if runtime support is implemented.

## Solution Assessment

Unclear. The requested decoded `imsg` history/RPC direction is the right runtime shape if iMessage DM history remains supported, but a docs-honest de-advertising path was also proposed and not merged, so maintainers should choose the intended contract first.
