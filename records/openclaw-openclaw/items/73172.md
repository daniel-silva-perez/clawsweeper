## Summary

Keep open. Current main still exposes and documents iMessage `dmHistoryLimit`, while the iMessage runtime only builds pending history for group messages; the report remains an unresolved contract gap, even though the exact `chat.db` `attributedBody` root cause was not live-verified here.

## What This Changes

The issue asks OpenClaw to make iMessage direct-message history configured by `dmHistoryLimit` include decoded outbound message text instead of blank history entries.

## Best Possible Solution

Make the iMessage DM-history contract explicit: either implement a decoded direct-message history adapter in `extensions/imessage` using `imsg` history/RPC output with regression coverage for outbound text, or remove iMessage from documented DM-history support while preserving the accepted config field for compatibility.

## Reproduction Assessment

Yes for the current-main contract gap: configure `channels.imessage.dmHistoryLimit`, and source/docs show it is accepted and advertised, but iMessage direct-message dispatch has no `historyKey` and `InboundHistory` is only populated for groups. The reporter's exact `attributedBody` database evidence is plausible implementation guidance but was not live-reproduced here.

## Solution Assessment

Unclear. A decoded `imsg` history/RPC adapter is the right implementation if OpenClaw intends to support legacy iMessage DM history, but the docs-honest alternative in #73775 was also plausible and closed unmerged, so this needs maintainer product direction first.
