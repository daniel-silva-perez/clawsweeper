## Summary

Keep open. Current main still has an iMessage DM-history contract mismatch: config and docs expose `dmHistoryLimit` for iMessage, but the iMessage runtime only resolves and injects pending group history. The reporter's `attributedBody` detail remains useful for a runtime implementation, but the source-backed current-main bug is broader: direct-message history is not implemented for this plugin, and the related docs-only PR #73775 is closed unmerged.

## What This Changes

The issue requests working iMessage direct-message history for `dmHistoryLimit`, with outbound text retrieved through a decoded `imsg` history/RPC path instead of blank database text fields.

## Best Possible Solution

Maintainers should make the iMessage contract explicit: either add a decoded iMessage direct-message history adapter in the iMessage plugin, with regression coverage for outbound message text, or remove iMessage from the documented `dmHistoryLimit` support list while preserving backward-compatible config parsing if needed.

## Reproduction Assessment

Yes for the current-main contract gap: a direct iMessage dispatch has no `historyKey`, and `InboundHistory` is only populated for group decisions. The reporter's exact macOS database `attributedBody` evidence is not live-verified here, but it is relevant implementation guidance if runtime support is kept.

## Solution Assessment

Unclear. Using decoded `imsg` history/RPC output is the right implementation direction if OpenClaw intends to support iMessage DM history, but a docs-honest de-advertising path was also proposed in #73775 and closed unmerged, so this needs maintainer product judgment before an automated repair chooses a contract.
