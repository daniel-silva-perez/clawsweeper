## Summary

Keep open. Current main still has an iMessage history contract gap: config/docs and generic session-history limiting expose `dmHistoryLimit` for iMessage, but the iMessage runtime has no decoded direct-message history/backfill adapter and its `InboundHistory` path remains group-only. The related docs-only PR #73775 was closed unmerged, so it does not supersede this issue.

## What This Changes

The issue requests that iMessage `dmHistoryLimit` history injection preserve outbound text by using decoded `imsg` history/RPC output instead of blank `message.text` database values.

## Best Possible Solution

Maintainers should make the iMessage contract explicit: either add a decoded direct-message history adapter in the iMessage plugin using `imsg` history/RPC output with regression coverage for outbound text, or remove iMessage from documented DM-history support while keeping existing config parsing compatible.

## Reproduction Assessment

Yes for the current-main contract gap: a direct iMessage dispatch has no `historyKey`, and `InboundHistory` is only populated for group decisions. The reporter's exact macOS `attributedBody` evidence remains useful implementation guidance, but it was not live-verified in this review.

## Solution Assessment

Unclear. Using decoded `imsg` history/RPC output is the right implementation direction if OpenClaw intends to support legacy iMessage DM history, but a docs-honest removal path was also proposed in #73775 and closed unmerged, so maintainers should choose the contract before an automated repair writes code.
