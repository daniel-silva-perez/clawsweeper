## Summary

Keep open. Current main still has an iMessage DM-history contract gap: config/schema/docs expose `dmHistoryLimit` for iMessage, but the iMessage monitor only builds pending history for group messages and has no decoded direct-message history/backfill adapter. The reporter's `attributedBody` evidence remains useful implementation guidance, while the prior docs-only PR #73775 was closed unmerged and does not supersede the issue.

## What This Changes

The issue asks OpenClaw to make iMessage `dmHistoryLimit` provide non-blank direct-message history by using decoded `imsg` history/RPC output instead of relying on raw Messages `text` values.

## Best Possible Solution

Maintainers should make the iMessage contract explicit: either add a decoded direct-message history adapter in the iMessage plugin using `imsg` history/RPC output with regression coverage for outbound text, or remove iMessage from documented DM-history support while keeping existing config parsing compatible.

## Reproduction Assessment

Yes for the current-main contract gap: a direct iMessage dispatch has no `historyKey`, and `InboundHistory` is only populated for group decisions. The reporter's exact `attributedBody` SQL evidence is credible implementation guidance but was not live-verified in this read-only review.

## Solution Assessment

Unclear. Using decoded `imsg` history/RPC output is the right implementation direction if OpenClaw intends to support legacy iMessage DM history, but the closed docs-only PR shows maintainers still need to choose between implementation and de-advertising the unsupported contract.
