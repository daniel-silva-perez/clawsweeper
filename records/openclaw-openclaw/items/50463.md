## Summary

Keep this PR open. Current main still does not implement the requested model-visible hint when a registered skill name is called as an unknown tool. The PR is a focused contribution rather than obsolete, but it still needs rebase and functional rework against the currently pinned pi-agent contract before it can land.

## Best Possible Solution

Keep this PR open as a focused implementation candidate. The best path is to rebase onto current main, implement the hint at a supported missing-tool-result path for the pinned pi-agent package, avoid private or nonexistent hook APIs, define visible-skill matching semantics deliberately, preserve real tool-collision behavior, and add targeted regression tests for both rewrite and non-rewrite cases.

## Risks / Open Questions

- The PR's current hook wiring appears incompatible with the pinned `@mariozechner/pi-agent-core@0.70.5` public surface because it uses `setAfterToolCall`, which is not present in the inspected published types.
- The common missing-tool result path in the pinned agent loop appears to bypass `afterToolCall`, so the proposed hook may not affect the exact behavior it targets without a different integration point or dependency change.
- The review discussion still leaves matching semantics to settle: optional `resolvedSkills`, `$`-prefixed skill spellings, skill names with spaces or punctuation, and real tool/skill collisions.
- No regression tests are present on current main for skill-match rewrite or non-skill unknown-tool behavior.
