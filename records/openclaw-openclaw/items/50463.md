## Summary

Keep this PR open. Current main still does not implement the requested model-visible hint when a skill name is called as an unknown tool, and this PR is a focused contribution rather than obsolete. It needs rebase and implementation rework against the currently pinned pi-agent contract before it can land.

## Best Possible Solution

Keep this PR open as a useful focused candidate. The best path is to rebase onto current main, implement the hint at a supported missing-tool-result path for the pinned pi-agent packages, avoid private or nonexistent hook APIs, deliberately define visible-skill matching semantics, preserve real tool collision behavior, and add targeted regression tests for both rewrite and non-rewrite cases.

- The PR appears incompatible with current `@mariozechner/pi-agent-core@0.70.5` because it uses `setAfterToolCall`, which is not present in the inspected public types.
- The common missing-tool result path in the pinned agent loop appears to bypass `afterToolCall`, so the proposed hook may not affect the exact behavior it targets without dependency or integration changes.
- No regression tests are added for skill-match rewrite, non-skill unknown tools, tool/skill collisions, `$`-prefixed skill spellings, punctuation/space skill names, or snapshots without `resolvedSkills`.
- The unknown-tool parser remains narrower than documented/raw skill names because it only captures `[a-z0-9_-]+`.
- No security or supply-chain regression was found in the PR diff; the remaining concerns are functional correctness and dependency-contract fit.
