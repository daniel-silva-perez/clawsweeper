## Summary

Keep this PR open. Current main still does not implement a model-visible hint when a skill name is called as an unknown tool, and this PR is a plausible focused candidate, but it is not obsolete or safely closeable as implemented-on-main. The PR also still needs maintainer review around hook contract/API compatibility, parser coverage, and regression tests before it should be merged.

## Best Possible Solution

Keep this PR open as a useful focused candidate. The best path is to rebase it onto current main, verify the supported pi-agent-core result-rewrite path for unknown-tool results, avoid private hook fields if possible, share or harden the unknown-tool parser, handle visible-skill matching intentionally, skip real tool collisions, and add targeted tests for skill match, non-skill unknown tool, collisions, `$`/punctuation names, and the pinned missing-tool path.

## Risks / Open Questions

- The PR relies on `setAfterToolCall` and a private `_afterToolCall` field; that hook composition needs explicit validation against the pinned `@mariozechner/pi-agent-core` 0.70.5 contract before merge.
- The PR diff contains no regression test files, so the intended behavior is not protected for skill match, non-skill unknown tool, tool/skill collision, and current missing-tool paths.
- The proposed `resolveUnknownToolName` regex only captures `[a-z0-9_-]+`, while current skill prompts render raw skill names and command sanitization exists separately, so `$` prefixes, spaces, punctuation, and mixed skill names need either support or a documented scope decision.
- The PR discussion contains unresolved disagreement about whether prepare-time/immediate unknown-tool results reach the chosen hook path; that should be settled with source-backed proof or a targeted test.
