## Summary

Keep open. Current main still routes the slug-generator embedded helper through shared auth-profile failure handling, so a billing-classified helper error can still mark the shared profile disabled; the repository already has the narrower local failure-policy seam needed for a focused fix.

## What This Changes

The issue asks for slug-generator helper failures, especially billing-classified Anthropic extra-usage HTTP 400s, to stay local instead of disabling the shared auth profile for five hours.

## Best Possible Solution

Add `authProfileFailurePolicy: "local"` to the slug-generator embedded run and lock it with regression coverage; keep broader slug-model decoupling under #33962 and only revisit global billing classification if maintainers decide extra-usage text should not be treated as a shared auth-health signal.

## Reproduction Assessment

Yes. A high-confidence source reproduction path exists: let the session-memory slug helper receive a billing-classified prompt error, and because the helper call omits the local policy, `run.ts` can write shared billing failure state for the active auth profile.

## Solution Assessment

Yes for a narrow fix. Using the existing local auth-profile failure policy on the slug-generator helper is the smallest maintainable repair and avoids a broad classifier change that could affect real billing/quota fallback behavior elsewhere.
