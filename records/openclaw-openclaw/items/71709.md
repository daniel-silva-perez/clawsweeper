## Summary

Keep open. Current main and the latest release still let the slug-generator embedded helper use shared auth-profile failure handling, so a billing-classified helper error can still write profile-wide disabled state; related PR #65134 overlaps but is open, broader than this bug, and not yet an implemented fix.

## What This Changes

The issue requests that slug-generator helper failures stay lane-local and not mark a shared auth profile as `disabled:billing` after a transient HTTP 400 or extra-usage rejection.

## Best Possible Solution

Make slug generation helper-local for auth health: either pass `authProfileFailurePolicy: "local"` on the current embedded slug run with focused regression tests, or land a narrowed simple-completion replacement that removes the embedded-run auth-state write path while keeping broader lightweight-model work tracked separately by #33962.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: make the slug-generator embedded run encounter a billing-classified prompt error such as the Anthropic extra-usage message, and because the call omits `authProfileFailurePolicy: "local"`, the runner can mark the shared profile as billing-disabled.

## Solution Assessment

No, current main is not the best end state because the helper still shares profile-wide auth failure handling. The smallest maintainable repair is to use the existing local helper policy for `temp:slug-generator`, while #65134 needs maintainer review before it can serve as the broader replacement.
