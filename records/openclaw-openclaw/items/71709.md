## Summary

Keep open. Current main still matches the reported slug-generator auth-health leak: the helper runs on `temp:slug-generator` without local auth-failure scoping, while the embedded runner already has a local policy that would prevent helper failures from poisoning shared auth-profile state. The prior ClawSweeper comment identified the same narrow fix, and the current checkout still has that gap.

## What This Changes

The issue asks OpenClaw to prevent low-priority LLM slug-generation failures, especially Anthropic CLI HTTP 400 extra-usage responses, from disabling a shared auth profile with a five-hour billing cooldown.

## Best Possible Solution

Add `authProfileFailurePolicy: "local"` to the slug-generator embedded run and cover it in `src/hooks/llm-slug-generator.test.ts`, preserving the existing provider/model and timeout behavior. Leave broader HTTP 400 billing-classification policy changes to separate maintainer review.

## Reproduction Assessment

Yes. The high-confidence reproduction path is static on current main: slug generation runs through `runEmbeddedPiAgent` without local policy, the runner persists classified prompt failures, the reported extra-usage text is billing-classified, and billing disables default to five hours. I did not live-trigger Anthropic, but the code path matches the incident.

## Solution Assessment

Yes for the narrow repair: scoping the slug helper with `authProfileFailurePolicy: "local"` uses an existing runner contract and limits the change to the low-priority helper lane. A classifier-only fix is less safe as the first step because the same billing patterns are intentionally used for real provider failover and cooldown behavior.
