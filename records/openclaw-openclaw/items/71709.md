## Summary

Keep open. Current main still routes session-memory slug generation through `runEmbeddedPiAgent` without the existing local auth-profile failure policy, while the embedded runner persists billing-classified prompt failures to shared auth state; the reported Anthropic extra-usage text is intentionally classified as billing for normal fallback paths, so the helper-scoping bug remains valid.

## What This Changes

The issue requests lane-local handling for session-memory slug-generator LLM failures so an internal helper error cannot mark a shared auth profile as billing-disabled for five hours.

## Best Possible Solution

Add `authProfileFailurePolicy: "local"` to the slug-generator embedded run, cover it in `src/hooks/llm-slug-generator.test.ts`, and leave broader dedicated slug-model or congestion behavior under #33962. Do not loosen the global Anthropic extra-usage billing classifier, because that is intentionally used for normal same-turn fallback.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists by sending an Anthropic extra-usage or billing-classified prompt failure through `generateSlugViaLLM`; current main passes it into the shared embedded runner without the local policy, allowing shared auth-state writes.

## Solution Assessment

Yes for a narrow repair. The existing local policy seam is the smallest maintainable fix because it scopes internal helper failures without changing normal provider billing/fallback behavior.
