## Summary

Keep open. Current main and the latest release still allow the slug-generator embedded helper to use shared auth-profile failure handling, while Anthropic extra-usage text remains billing-classified and billing disables still use a five-hour default window.

## What This Changes

The issue asks that slug-generator helper failures, especially HTTP 400 extra-usage errors, not mark the shared auth profile as billing-disabled for five hours.

## Best Possible Solution

Use the existing embedded-runner local auth-profile failure policy for the slug-generator helper and add regression coverage that locks the helper out of shared auth-health writes; keep the broader lightweight-slug-model work tracked separately by #33962.

## Reproduction Assessment

Yes. A high-confidence source reproduction path exists: make the slug-generator embedded run receive a billing-classified prompt error and, because the call omits `authProfileFailurePolicy: "local"`, the runner can mark the shared profile as billing-disabled.

## Solution Assessment

Yes for the narrow repair candidate. Adding `authProfileFailurePolicy: "local"` to the slug-generator embedded run is the smallest maintainable fix because it uses an existing runner contract and avoids broad changes to provider billing classification.
