## Summary

Keep this open. Current main still has the reported reliability gap for normal Gateway/agent-command delivery: successful recovered or fallback runs can return clean payloads while the delivery helper discards the lower-level delivery result, and the Gateway marks the run succeeded from agent completion rather than channel delivery completion. The issue discussion also links the active draft #53961, which is relevant but not merged into current main.

## What This Changes

Reported that recovered or fallback agent runs can record a clean assistant response in the transcript while Discord or Telegram delivery silently drops the outbound message.

## Best Possible Solution

Keep this as the canonical bug for recovered/fallback agent-run delivery drops across channel deliveries. Land or revise #53961, or an equivalent structured delivery-status contract, then add a narrow post-condition at the agentCommandFromIngress/deliverAgentCommandResult boundary that re-resolves fresh session delivery context and retries only when delivery was requested but not actually attempted or definitively failed.

## Reproduction Assessment

Yes. The issue has high-confidence field reproductions across Discord and Telegram, and current main exposes a focused unit-test seam by mocking deliverOutboundPayloads to return [] after a successful agent result and verifying the Gateway still reports success.

## Solution Assessment

Yes, the best direction is a structured delivery status plus a carefully scoped fresh-context retry. A provider-specific overloaded_error fix would be too narrow, and a blanket retry after any empty result risks duplicate sends.
