## Summary

Keep open. Current main still follows the reported failing path: the OpenAI plugin treats the ChatGPT Codex backend as a native route, emits turn-state metadata for it, the Responses builder includes that metadata in the request body, and the final SDK send path merges the same metadata again immediately before client.responses.create. The closed PR #74029 did not merge, and the follow-up comment correctly identifies why a builder-only strip would not cover the live send path.

## What This Changes

The issue requests that OpenClaw stop sending unsupported top-level metadata on openai-codex Responses requests to the ChatGPT Codex backend and preferably improve the misleading 400 error surface.

## Best Possible Solution

Fix this in the OpenAI provider-owned transport policy by preserving the x-openclaw session and turn headers for openai-codex while omitting request-body metadata for that provider, then add a regression test that covers the final post-merge params handed to the SDK send path.

## Reproduction Assessment

Yes. The issue provides concrete live reproduction and current main has a high-confidence static reproduction path: openai-codex resolves metadata, buildOpenAIResponsesParams accepts it, and mergeTransportMetadata re-adds it before client.responses.create.

## Solution Assessment

Yes. Gating metadata at resolveOpenAITransportTurnState for the normalized openai-codex provider is the narrowest durable fix because it covers both the builder spread and the later post-onPayload merge; a builder-only strip is not sufficient on current main.
