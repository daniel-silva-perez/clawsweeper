## Summary

Close as implemented on current main. The send path now strips `metadata` and other unsupported native ChatGPT/Codex Responses payload fields after payload hooks and before `client.responses.create`, which covers the reporter's follow-up concern that a builder-only change would be re-contaminated later.

## What This Changes

The issue asks OpenClaw to stop sending top-level `metadata` in native `openai-codex` Responses requests to the ChatGPT backend because that field caused HTTP 400 rejections.

## Best Possible Solution

Keep the current main implementation: sanitize unsupported native ChatGPT/Codex Responses fields at the final transport boundary while preserving custom Codex-compatible endpoints, then let the next release carry the fix to npm users.

## Reproduction Assessment

Yes. The issue includes a concrete live OAuth/agent reproduction and raw backend response, and current main has source and regression-test proof that the offending `metadata` field is removed before the native Codex request is sent.

## Solution Assessment

Yes. The main fix is the narrow maintainable solution because it handles both builder output and post-hook send-path mutation, while avoiding a blanket strip that would change custom compatible endpoints.
