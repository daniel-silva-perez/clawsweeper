## Summary

Keep open: current main still routes exec approval follow-up agent turns by the stable session key, while /new and /reset replace the session id stored under that same key. I did not find a source or test guard that captures the original session instance and drops stale approval follow-ups after the key is rebound.

## What This Changes

The issue requests that async exec approval follow-ups remain bound to the original session instance and be dropped after /new or /reset rebinds the same session key to a new session id.

## Best Possible Solution

Add an explicit session-instance guard for exec approval follow-ups: capture the current session id when the approval follow-up target is created, carry it through the async completion path, and before launching the follow-up agent turn verify that the session key still maps to that same id. If it does not, log/drop the stale follow-up and do not fall back to direct channel delivery.

## Reproduction Assessment

Yes. The issue provides concrete local reproduction data, and current main's source path independently shows the same route: pending approval follow-ups retain only sessionKey, while /new resets that key to a new sessionId before the later agent follow-up runs.

## Solution Assessment

Yes. The proposed expectedSessionId-style guard is the narrowest maintainable fix; adding a separate guard is safer than passing the old id as agent.sessionId because the current agent handler treats sessionId as a requested writable session id rather than a stale-drop precondition.

Current main builds ExecApprovalFollowupTarget with only sessionKey and later calls the Gateway agent method by that stable key; sessions.reset remints sessionId under the same key, so an old approval follow-up can resume the new session.
