## Summary

Keep this issue open. Current main still routes async exec approval follow-ups by logical sessionKey without carrying the original sessionId, so a follow-up can target whatever sessionId is currently bound to that key after /new or /reset. Related prior cleanup fixed adjacent approval delivery and webchat/channel routing problems, but not this session-instance guard.

## Best Possible Solution

Plumb the active sessionId from the exec tool context into the approval follow-up target as an expected/original session id. Before sending the follow-up agent turn, reload the current sessionKey mapping and drop/log the follow-up if the current sessionId differs. Add a focused regression test that creates an approval, resets the same sessionKey to a new sessionId, resolves the old approval, and asserts no follow-up is delivered into the new transcript.
