## Summary

Keep open. Current main still routes async exec approval follow-ups by `sessionKey` without carrying or checking the original `sessionId`, so a reset or `/new` can still rebind the logical key before the pending approval resolves.

## What This Changes

The issue requests binding exec approval follow-ups to the original session instance and dropping stale follow-ups after `/new` or `/reset` rebinds the same `sessionKey` to a different `sessionId`.

## Best Possible Solution

Add an original-session guard to async exec approval follow-up delivery: capture the active `sessionId` when the approval request creates its follow-up target, compare it with the current mapping for that `sessionKey` immediately before dispatch, and drop/log the follow-up when the mapping has changed. Cover approved, denied, and reset-after-request cases with focused regression tests.

## Reproduction Assessment

Yes. The issue provides concrete local reproduction data, and static inspection of current main confirms the pending follow-up path still stores only `sessionKey` while the gateway `agent` method resolves whatever session id is current for that key.

## Solution Assessment

Yes, the proposed `expectedSessionId` guard is the narrowest maintainable fix for the reported leak. The safer implementation detail is to treat a mismatch as a stale follow-up to drop, not as a reason to fall back to direct channel delivery.
