## Summary

Keep this PR open. Current main at 891c7d9f1c1f5e0f52935fadfeacf695b0ef1eb8 still does not implement native audio/video forwarding: there is no `nativeForwarding` config surface or native media store, media understanding still emits text transcript/description outputs for audio/video, and the embedded runner still passes only `ImageContent[]` to `activeSession.prompt()`. The PR remains plausible feature work, but it is not obsolete or already shipped, and the existing review discussion identifies real cleanup, dependency-contract, and raw-media egress risks that need maintainer review before merge.

## Best Possible Solution

Keep this PR open as useful but unfinished feature work. The best path is to revise it against current main with a typed native audio/video content contract in the pi-ai/pi-coding-agent layer, add `tools.media.audio/video.nativeForwarding` to the public config schema/help/generated docs, thread one resolved run/session key through capture and consumption, add cleanup and native-command redirect regression tests, and document provider support, payload limits, and fallback behavior for models that reject native audio/video blocks.

## Risks / Open Questions

- Merging the PR as-is would broaden raw audio/video egress to model providers; this needs an explicit opt-in contract, provider capability behavior, payload limits, and docs/schema alignment.
- The review-discussed module-level base64 store can leak up-to-25 MiB media blocks when the capture key and embedded-runner session key diverge in native-command redirect flows.
- The PR force-casts native audio/video blocks through an image-only pi-ai/pi-coding-agent contract, creating deployment-ordering and prompt-shape risk until the upstream types and adapters support audio/video content.
- The PR does not include tests or generated config/docs updates for the new public config keys, native-command redirect cleanup, provider rejection behavior, or raw-media forwarding limits.
