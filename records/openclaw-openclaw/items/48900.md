## Summary

Keep open. Current main does not implement this PR's intended native audio/video forwarding: there is no `nativeForwarding` config surface or native-media store, media understanding still emits text transcript/description outputs, and the embedded runner still submits only `ImageContent[]` through the `images` prompt option. The PR remains plausible feature work, but it is not obsolete or already shipped, and the provided review discussion identifies unresolved correctness and dependency-contract risks that should be fixed before landing.

## Best Possible Solution

Keep this PR open as useful but unfinished feature work. The best path is to revise it against current main with a typed native media content contract from `@mariozechner/pi-ai`, add `tools.media.audio/video.nativeForwarding` to the public config schema/help/generated docs, thread one resolved session/run key through capture and consumption, add cleanup and native-command redirect regression tests, and document provider capability/fallback behavior for models that reject native audio/video blocks.

## Risks / Open Questions

- Merging the PR as provided would broaden raw audio/video egress to model providers, so it needs an explicit opt-in contract, config schema/help/docs, provider capability checks, payload limits, and clear fallback behavior.
- The PR's module-level base64 store has an identified cleanup/keying bug in native-command redirect flows; that should be fixed with regression coverage before maintainers consider landing it.
- The proposed cast around `@mariozechner/pi-ai` image content creates a deployment-ordering/type-contract risk unless the dependency gains typed audio/video content support or OpenClaw guards the path at runtime.
