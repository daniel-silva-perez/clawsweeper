## Summary

Keep this PR open. Current main still does not implement native audio/video forwarding, and the PR remains useful feature work, but the proposed patch is not merge-ready because it misses the strict public config schema, stores native media under a key that can diverge from the prompt-consumption key, and force-casts audio/video blocks through the current image-only pi-ai prompt contract.

## What This Changes

This PR adds an opt-in `tools.media.audio/video.nativeForwarding` config type, a session-keyed native media store, raw audio/video attachment capture during media understanding, and prompt-time merging of those blocks into embedded-runner media submission.

## Best Possible Solution

Revise the feature behind a typed native media prompt contract from the pi-ai layer, add the config schema/help/generated docs for `tools.media.audio/video.nativeForwarding`, key capture and consumption with the same resolved session/run identifier, and cover native-command redirect cleanup plus provider-rejection fallback behavior with focused tests.

## Reproduction Assessment

Yes. Current main can be checked with repository search and the media/config/prompt code paths: there is no native-forwarding symbol, the strict schema lacks the option, audio/video understanding returns text outputs, and the embedded runner only submits image blocks.

## Solution Assessment

No, not as currently proposed. The direction is plausible, but the mergeable solution needs schema/docs alignment, a dependency-backed audio/video prompt type, deterministic cleanup keying, and regression tests before raw media forwarding is safe to ship.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Add the public schema for nativeForwarding:** `src/config/types.tools.ts:96-103`
  - body: The PR declares `nativeForwarding` only in the TypeScript config type, but `ToolsMediaUnderstandingSchema` remains strict and does not accept that key. Users configuring `tools.media.audio.nativeForwarding` or `tools.media.video.nativeForwarding` through validated config paths will have an invalid public option, so the schema/help/generated docs and config tests need to be updated with the new field.
  - confidence: 0.91
- **[P1] Use the resolved session key for native media blocks:** `src/media-understanding/apply.ts:599-601`
  - body: `applyMediaUnderstanding` stores native media under `ctx.SessionKey`, while the embedded runner consumes and discards under `params.sessionKey ?? params.sessionId`. Native command redirects can set `params.sessionKey` to `CommandTargetSessionKey`, leaving the blocks stored under the source key unconsumed and retained for the process lifetime.
  - confidence: 0.9
- **[P2] Gate native blocks on a typed prompt contract:** `src/agents/pi-embedded-runner/run/attempt.ts:2572-2574`
  - body: The current pinned `@mariozechner/pi-coding-agent` prompt options accept `images?: ImageContent[]`, and `@mariozechner/pi-ai` user content is text/image only. Casting `NativeMediaBlock[]` into that image array can send unsupported `{type: "audio"|"video"}` blocks unless the companion dependency has already shipped and is pinned here, so the PR should either update the dependency contract in the same change or fail clearly before injection.
  - confidence: 0.87

## Security Review

Status: needs_attention

Summary: The diff does not alter workflows, dependencies, lockfiles, or install scripts, but it introduces raw audio/video forwarding and a process-memory media store that need explicit privacy and cleanup guarantees before merge.

Concerns:

- **[medium] Raw media can remain in process memory after redirects:** `src/media-understanding/apply.ts:599`
  - body: The PR stores base64 audio/video blocks under the source `SessionKey`, but redirected native-command runs consume and discard under the resolved target session key. That mismatch can retain private raw media buffers until process exit.
  - confidence: 0.88
- **[low] Raw media egress needs a documented opt-in contract:** `src/config/types.tools.ts:96`
  - body: The feature forwards original audio/video bytes to the model provider in addition to transcription/description. That is intended, but the PR does not add the strict schema/help/docs surface or provider capability/fallback contract that would make the privacy boundary clear to operators.
  - confidence: 0.78
