## Summary

Keep open. Current main still has separate controls for builtin SQLite recall, QMD recall, compaction-time Markdown writes, and fallback behavior, but it does not provide the requested strict memory-mode contract that makes those behaviors consistent across users. The related draft PR #70540 is relevant memory work, but it is only a first `memory_add` slice and does not settle strict modes, disabled-memory semantics, QMD fallback policy, or workspace-scoped SQLite storage.

## What This Changes

The issue asks for a strict, consistent memory management contract across builtin SQLite memory, QMD Markdown memory, disabled memory, and workspace-scoped storage.

## Best Possible Solution

Treat this as a focused maintainer design task for memory-core, config/schema, memory-host storage, and docs. The end state should define one coherent memory-mode contract or documented successor covering durable writes, recall backend, QMD fallback, disabled memory, workspace and agent storage scope, migration for existing Markdown/SQLite artifacts, and operator guidance.
