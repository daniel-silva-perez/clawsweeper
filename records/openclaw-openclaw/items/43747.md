## Summary

Keep open. Current main still has separate memory backend, semantic index, QMD fallback, and compaction-time Markdown flush controls, so it does not implement the strict unified memory-mode contract requested in the discussion. Related memory CRUD work in #70049 and draft PR #70540 is useful context, but it only covers a `memory_add`/daily-block slice and does not settle `memory.type`, qmd-only behavior, disabled memory, or workspace-scoped storage semantics.

## What This Changes

The issue asks OpenClaw to replace scattered memory behavior with a strict, unified memory-management contract so users get consistent SQLite, QMD, mixed, or disabled memory behavior across machines.

## Best Possible Solution

Define one coherent memory-mode contract that covers durable writes, recall backend, QMD fallback policy, disabled memory, workspace and agent storage scope, migration for existing Markdown and SQLite artifacts, and user-facing config/docs guidance.

## Reproduction Assessment

Not as a high-confidence end-user runtime reproduction: the report lacks the three users' configs and logs. The requested missing contract is verifiable by static inspection of current main: `memory.type` is absent and backend, search, QMD fallback, and memory flush remain separate controls.

## Solution Assessment

No. The current scattered configuration is not the requested solution, and a small automated `memory.type` alias would not be the best fix because it would have to choose public semantics, migrations, fallback behavior, and storage scope without maintainer product design.
