## Summary

Keep open. Current main documents and implements several memory controls, but it still does not provide the requested strict memory-mode contract; builtin SQLite search, QMD Markdown indexing, daily Markdown flushes, fallback behavior, active memory, and storage scoping remain separate surfaces. The open draft PR #70540 is a useful first slice for `memory_add`, but it does not settle the requested builtin-only/qmd-only/mixed contract or workspace-scoped SQLite behavior.

## What This Changes

The issue requests a strict, consistent memory management contract so builtin SQLite, QMD Markdown memory, mixed mode, and workspace or agent scoping behave predictably across installations.

## Best Possible Solution

Treat this as a focused maintainer design task for `memory-core`, config/schema, memory-host storage, and docs. The end state should define one coherent memory-mode contract or documented successor covering durable writes, recall backend, QMD fallback, workspace and agent storage scope, migration for existing Markdown/SQLite artifacts, and operator guidance.
