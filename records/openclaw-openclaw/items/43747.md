## Summary

Keep open. Current main still exposes memory through separate backend, memory-search, QMD, plugin-slot, and compaction-flush controls; it does not implement the strict unified memory-mode contract requested in the discussion.

## What This Changes

The issue requests a strict, unified memory-management definition so users do not diverge between builtin SQLite memory, daily Markdown/QMD memory, or no memory.

## Best Possible Solution

Define one coherent public memory-mode contract that aligns config schema, runtime backend selection, plugin-slot disablement, QMD fallback policy, daily Markdown flush behavior, workspace and agent storage scope, docs, generated config metadata, and compatibility for existing memory.backend users.

## Reproduction Assessment

No for the original three-user divergence: the report lacks each user's configs, logs, plugin-slot state, QMD setup, and embedding provider state. Yes for the contract gap itself: static inspection shows current main still has no memory.type-style enum and keeps the relevant controls split.

## Solution Assessment

Unclear. A unified public mode may be the right direction, but the proposed built-in-only/qmd-only/mixed shape needs maintainer design because a thin alias would not settle migration, fallback, disabled-memory, plugin-slot, or workspace storage semantics.
