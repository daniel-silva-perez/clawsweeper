## Summary

Keep open. Current main still exposes memory as separate backend, QMD, memory-search, memory-plugin-slot, and compaction-flush controls; it does not implement the strict unified memory-mode contract requested in the discussion.

## What This Changes

The issue asks to replace scattered memory behavior and configuration with a strict unified `memory.type` contract covering built-in, QMD, mixed, disabled, and workspace-scoped storage behavior.

## Best Possible Solution

Define one coherent public memory-mode contract that aligns config schema, runtime backend selection, plugin-slot disablement, QMD fallback policy, daily Markdown flush, workspace and agent storage scope, migration for existing artifacts, generated docs, and compatibility for current `memory.backend` users.

## Reproduction Assessment

Unclear for the original three-user runtime divergence because the report lacks each user's configs, logs, plugin-slot state, and QMD/embedding setup. The missing contract itself is reproducible by static inspection: current main has no `memory.type` and still splits backend, QMD, memory-search, plugin-slot, and compaction-flush controls.

## Solution Assessment

Unclear. A unified public mode may be the right direction, but the exact `built-in-only`/`qmd-only`/`mixed` proposal needs maintainer design before implementation because a thin alias would not settle compatibility, migration, fallback, disabled-memory, or workspace storage semantics.
