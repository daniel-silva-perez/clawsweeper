## Summary

Keep open. Current main documents and implements the memory pieces more clearly, but it still does not provide the strict single memory-mode contract requested in the discussion. Builtin search, QMD selection, daily Markdown flushes, QMD fallback, active-memory recall, and SQLite store scoping remain separate controls, so team members can still see different behavior from config, credentials, or local dependency differences.

## Best Possible Solution

Keep this as a focused maintainer design task. Define either memory.type or a documented successor that covers durable write target, search/index backend, QMD fallback strictness, active-memory interaction, per-agent/per-workspace SQLite scoping, migration behavior for users with both Markdown and SQLite artifacts, config schema/help/docs, and doctor guidance.
