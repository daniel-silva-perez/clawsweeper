## Summary

Keep this PR open. Current main still accepts Telegram `agentId` only for topic config and binding-based routing; it does not accept or consume `channels.telegram.accounts.<id>.agentId`. The PR remains a focused implementation candidate for the linked Telegram multi-account regression in #62985, with review follow-ups already addressed in the PR discussion.

## Best Possible Solution

Review, rebase, and land this PR or an equivalent focused patch for #62985: accept `channels.telegram.accounts.<id>.agentId` in the account schema/type, keep top-level `channels.telegram.agentId` rejected, consume the account default only when no explicit binding matches, keep explicit bindings and topic-specific `agentId` ahead of the account default, and add schema plus Telegram route/message-context regression coverage. Update Telegram docs and changelog if maintainers treat the account-level field as a user-facing config addition.
