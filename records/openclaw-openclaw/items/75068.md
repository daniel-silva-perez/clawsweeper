## Summary

Keep open. Current `main` and the latest release still preflight-reject the reported Brave `llm-context` freshness/date filters, while the upstream Brave LLM Context docs list `freshness` and date-range freshness support; an open related PR already targets the narrow fix.

## What This Changes

The issue requests that Brave `web_search` in `llm-context` mode stop falsely rejecting freshness and date filters, and asks maintainers to clarify or relax `ui_lang` handling and add `count` support.

## Best Possible Solution

Keep this issue open until a fix lands that unblocks Brave LLM Context freshness/date filters, adds focused provider tests for the URL parameters and cache key, and updates the Brave Search plus web-search docs; handle `ui_lang` and `count` as explicit API-contract decisions rather than folding them silently into the minimal bug fix.

## Reproduction Assessment

Yes. With Brave configured for `llm-context` and any API key present, `web_search` calls containing `freshness`, `date_after`, or `date_before` deterministically return the local `unsupported_*` payloads before any Brave request is sent.

## Solution Assessment

Unclear. Removing the freshness/date preflight guards and wiring date ranges into `runBraveLlmContextSearch` is the narrow maintainable fix, but `ui_lang` should likely remain rejected or be documented as unsupported unless Brave documents it, and `count` needs a separate settled contract.
