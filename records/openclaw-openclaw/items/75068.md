## Summary

Keep open. Current main and latest release still preflight-reject Brave `llm-context` freshness and date filters before contacting Brave, while Brave's LLM Context docs list `freshness` support; the related open PR #51005 targets the same bug and should be reviewed or replaced rather than starting a competing repair lane.

## What This Changes

The issue reports that Brave `web_search` in `llm-context` mode rejects `freshness`, `date_after`/`date_before`, and `ui_lang` before reaching the Brave API.

## Best Possible Solution

Keep this as the canonical bug report, then review and port PR #51005 or replace it with a narrow Brave plugin fix that unblocks validated freshness/date filters, adds focused runtime tests, and updates Brave Search plus web-search docs while handling `ui_lang` as a separate explicit contract choice.

## Reproduction Assessment

Yes. With Brave configured in `llm-context` mode, a `web_search` call containing `freshness`, `date_after`, or `date_before` deterministically returns a local `unsupported_*` payload in `executeBraveSearch` before any Brave request is sent.

## Solution Assessment

Unclear. Removing the freshness/date preflight guards and mapping closed date ranges in the Brave runtime is the narrow maintainable direction, but `ui_lang`, `count`, and one-sided date bounds should be decided separately against the Brave contract.
