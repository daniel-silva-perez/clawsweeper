## Summary

Keep open. Current main and the latest release still preflight-reject Brave `llm-context` freshness and date filters before dispatch, while Brave documents `freshness` and custom date-range support for that endpoint. The `ui_lang` part is less settled because Brave does not document it for LLM Context, and open PR #51005 is already the active implementation path for the freshness/date portion.

## What This Changes

The issue requests that Brave `web_search` in `llm-context` mode stop rejecting `freshness`, `date_after`/`date_before`, and possibly `ui_lang`.

## Best Possible Solution

Keep this issue open as the canonical bug report, then review or replace #51005 so Brave `llm-context` accepts validated freshness/date filters, maps date ranges to Brave `freshness`, keeps cache keys distinct, and updates the Brave/web-search docs; handle `ui_lang` explicitly as unsupported or product-approved passthrough.

## Reproduction Assessment

Yes. With any configured Brave API key value, a `web_search` call using Brave `llm-context` plus `freshness`, `date_after`, or `date_before` deterministically returns the local `unsupported_*` response before any network request is made.

## Solution Assessment

Unclear for the whole report. The narrow maintainable fix is to remove the freshness/date preflight guards and wire date ranges into the LLM Context `freshness` parameter; `ui_lang` should likely stay rejected or be documented separately unless Brave adds it to the endpoint contract.
