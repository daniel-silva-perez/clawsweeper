## Summary

Keep open. Current `main` still rejects Brave `llm-context` `freshness`, `date_after`/`date_before`, and `ui_lang` before calling Brave; the upstream Brave LLM Context contract supports `freshness` and custom date ranges, while `ui_lang` remains undocumented. The already-open related PR #51005 targets the valid freshness/date portion, so this issue should stay open for maintainer review rather than be closed or queued as a competing repair.

## What This Changes

The issue asks OpenClaw's Brave `web_search` provider to stop preflight-rejecting `freshness`, `date_after`/`date_before`, and possibly `ui_lang` in `llm-context` mode, with an optional follow-up to pass `count`.

## Best Possible Solution

Keep this as the canonical bug report, then review or replace #51005 so the Brave plugin runtime unblocks documented freshness/date filtering, adds focused provider tests for the actual LLM Context URL parameters, and updates `docs.openclaw.ai/tools/brave-search` plus the web-search overview. Keep `ui_lang` rejected or clarify the docs unless Brave documents support, and decide `count` as a separate scoped enhancement if maintainers want it included.

## Reproduction Assessment

Yes. With Brave configured in `llm-context` mode, any `web_search` call carrying `freshness`, `date_after`/`date_before`, or `ui_lang` deterministically hits local `unsupported_*` returns in `executeBraveSearch` before a network request is made.

## Solution Assessment

Unclear. Unblocking documented `freshness` and date range filters in the Brave plugin runtime is the narrow maintainable fix, but allowing `ui_lang` is not clearly supported by Brave's contract and `count` is adjacent scope rather than the core regression.
