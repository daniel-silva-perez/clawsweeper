## Summary

Keep open. Current main and the latest release still preflight-reject Brave `llm-context` `freshness` and `date_after`/`date_before`, while Brave's current LLM Context contract documents `freshness`; `ui_lang` is less clear because Brave does not document it. The issue is also already paired with open PR #51005, so this should stay open for maintainer review of that fix or a deliberate replacement.

## What This Changes

The issue asks OpenClaw to stop falsely rejecting Brave `llm-context` `freshness`, `date_after`/`date_before`, and possibly `ui_lang`, and also notes missing `count` forwarding.

## Best Possible Solution

Keep this issue open as the bug report, then review, land, or replace PR #51005 so the Brave plugin runtime unblocks documented freshness/date filtering for `llm-context`, adds focused tests for request parameters and cache keys, and updates the Brave Search and web-search docs to match the settled `ui_lang` and `count` policy.

## Reproduction Assessment

Yes. With a configured Brave key, a Brave `web_search` call in `llm-context` mode with `freshness`, `date_after`, or `date_before` deterministically returns the local `unsupported_*` payload before any Brave request is sent.

## Solution Assessment

Unclear. Removing the freshness/date preflight guards and mapping date ranges to Brave `freshness` is the narrow maintainable fix, but `ui_lang` and `count` should be handled as separate contract decisions because Brave documents `count` but not `ui_lang`.
