## Summary

Keep open. Current main still has the Brave `llm-context` preflight guards that reject `freshness`, `date_after`/`date_before`, and `ui_lang`, while the dependency docs support at least `freshness`/range filtering and an open related PR already targets this bug.

## What This Changes

The issue requests that Brave `web_search` in `llm-context` mode stop rejecting supported freshness and date filtering parameters, clarify `ui_lang`, and optionally forward `count` to Brave's LLM Context endpoint.

## Best Possible Solution

Keep this issue open as the canonical bug report, review or replace the linked fix so the active Brave plugin runtime unblocks validated freshness/date filters, and update focused Brave tests plus the public Brave Search/web-search docs to match the settled parameter contract.

## Reproduction Assessment

Yes. A Brave `web_search` call in `llm-context` mode with `freshness`, `date_after`, or `date_before` deterministically hits local `unsupported_*` returns in `executeBraveSearch` before any Brave request is sent.

## Solution Assessment

Unclear. Removing the freshness/date preflight guards in the active runtime is the narrow maintainable direction, but `ui_lang`, `count`, and open-ended date-range mapping should be decided separately.
