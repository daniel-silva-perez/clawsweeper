## Summary

Keep open. Current main and the latest release still preflight-reject Brave `llm-context` `freshness`, `date_after`/`date_before`, and `ui_lang`; the dependency contract supports at least freshness/range filtering, and an open related PR already targets the bug.

## What This Changes

The issue asks OpenClaw to stop falsely rejecting supported Brave LLM Context freshness/date filters, clarify `ui_lang`, and optionally forward `count` for Brave `web_search` in `llm-context` mode.

## Best Possible Solution

Keep this issue open as the canonical bug report, then review or replace the linked fix so the active Brave plugin runtime unblocks validated freshness/date filters, adds focused Brave tests, and updates both Brave Search and web-search docs to match the settled parameter contract.

## Reproduction Assessment

Yes. A Brave `web_search` call in `llm-context` mode with `freshness`, `date_after`, or `date_before` deterministically hits local `unsupported_*` returns in `executeBraveSearch` before any Brave request is sent.

## Solution Assessment

Unclear. Removing the freshness/date preflight guards in `extensions/brave/src/brave-web-search-provider.runtime.ts` is the narrow maintainable direction, but `ui_lang`, `count`, and open-ended date-range mapping should be decided separately.
