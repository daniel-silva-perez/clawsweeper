## Summary

Current `main` still contains the Brave `llm-context` preflight rejection for `freshness`, `date_after`/`date_before`, and `ui_lang`. The report is fresh, source-backed, and already has a related open implementation PR, so this should stay open for maintainer review rather than be swept closed.

## What This Changes

The issue requests that Brave `web_search` in `llm-context` mode stop rejecting supported freshness/date filtering parameters, clarify or relax `ui_lang`, and optionally forward `count` to Brave's LLM Context endpoint.

## Best Possible Solution

Keep this issue open as the canonical bug report and review the linked fix work so the Brave plugin removes the incorrect LLM Context freshness/date rejection in the active runtime, adds focused tests for native and range freshness, and updates the Brave Search docs to match the settled parameter contract.

## Reproduction Assessment

Yes. The current-main reproduction path is a Brave `web_search` call in `llm-context` mode with `freshness` or `date_after`/`date_before`; source inspection shows `executeBraveSearch` returns an `unsupported_*` error before reaching the Brave API.

## Solution Assessment

Partly. Removing the `freshness` and date-filter preflight block is the narrow maintainable direction, but `ui_lang`, `count`, and one-sided date ranges should be settled explicitly before broadening the patch.
