## Summary

Keep open. Current main still preflight-rejects Brave `llm-context` `freshness` and date filters before dispatch even though Brave documents those LLM Context parameters; `ui_lang` is less certain because it is not in Brave's documented LLM Context contract. PR #51005 remains the active open implementation path for the freshness/date portion but has not merged.

## What This Changes

The issue asks to stop Brave `web_search` in `llm-context` mode from rejecting `freshness` and `date_after`/`date_before`, reconsider `ui_lang`, and optionally pass `count` through.

## Best Possible Solution

Land a narrow Brave plugin fix that removes the llm-context freshness/date guards, converts validated date filters into Brave `freshness` ranges, keeps cache keys/tests/docs aligned, and makes an explicit scoped decision on `ui_lang` and `count`.

## Reproduction Assessment

Yes. With a Brave API key configured and Brave set to `mode: "llm-context"`, a `web_search` call containing `freshness`, `date_after`, or `date_before` deterministically returns the local `unsupported_*` payload before any network request.

## Solution Assessment

Unclear for the whole report. The best narrow path is to wire documented freshness/date filters into the existing Brave llm-context request builder; `ui_lang` should likely remain rejected or be separately documented unless Brave adds it to the endpoint contract, and `count` can be a small follow-up if maintainers accept that scope.
