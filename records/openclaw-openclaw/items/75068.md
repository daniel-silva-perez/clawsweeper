## Summary

Keep open. Current main still preflight-rejects Brave `llm-context` `freshness` and date filters before dispatch, while Brave's LLM Context endpoint documents `freshness`, `count`, and custom date ranges; the `ui_lang` part remains less certain because it is not in the documented endpoint contract. Open PR #51005 is the active implementation path for the freshness/date portion, but it has not merged.

## What This Changes

The issue requests that Brave `web_search` in `llm-context` mode stop locally rejecting `freshness` and `date_after`/`date_before`, reconcile `ui_lang`, and optionally pass `count` through.

## Best Possible Solution

Keep this issue open until PR #51005 or a replacement lands the provider-owned fix: remove the freshness/date preflight guards, translate `date_after`/`date_before` into Brave `freshness` ranges for llm-context, keep cache keys and tests aligned, update Brave/web-search docs, and make an explicit maintainer decision on `ui_lang` and `count` scope.

## Reproduction Assessment

Yes. With a configured Brave API key and `plugins.entries.brave.config.webSearch.mode` set to `llm-context`, a `web_search` call containing `freshness`, `date_after`, or `date_before` deterministically returns the local `unsupported_*` payload before any network request.

## Solution Assessment

Unclear for the whole report. The narrow maintainable fix is to wire validated freshness/date filters into the existing Brave llm-context request builder; `ui_lang` should likely remain rejected or be separately documented unless Brave adds it to the endpoint contract, and `count` can be a small follow-up if maintainers accept that part of the request.
