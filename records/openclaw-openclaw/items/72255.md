## Summary

Keep open. Current main still allows a stored provider credential to be selected when no matching `auth.profiles` declaration exists, and the Pi embedded run path can inject that resolved credential into runtime auth without an undeclared-profile guard. The related stale-key and silent-fallback issues are adjacent context, but they do not fully cover this removed-declaration billing route.

## What This Changes

The issue asks OpenClaw to stop silently using per-agent stored auth-profile credentials after their `openclaw.json` declarations are removed, or at least warn and report that drift before runtime use.

## Best Possible Solution

Adopt a declaration-aware credential authority model: runtime selection should ignore or loudly flag stored provider profiles whose ids are no longer declared for that provider, while doctor or models status reports stored-but-undeclared profiles with non-destructive remediation.

## Reproduction Assessment

Yes. A source-level reproduction is to provide config with no `auth.profiles` entry for `anthropic`, keep a valid `anthropic:*` credential in the auth store, and observe `resolveAuthProfileOrder` returning that stored profile, which the Pi auth controller then resolves and injects into `authStorage`.

## Solution Assessment

No. The current silent reuse behavior is not the best way to solve the issue; the narrow maintainable direction is a resolver allowlist or selection-time warning plus diagnostic drift reporting, not hidden billing through removed declarations or automatic secret deletion.
