## Summary

Keep open. Current main still lets stored provider credentials participate in runtime auth selection when the matching provider has no declared auth profile, and the Pi embedded path can inject the selected credential into runtime auth without an undeclared-profile guard. The report is recent, security/billing-sensitive, and not fully covered by the adjacent stale-key or silent-fallback reports.

## What This Changes

The issue requests that Pi/runtime auth stop silently selecting credentials left in per-agent auth-profiles.json after their openclaw.json auth profile declarations are removed.

## Best Possible Solution

Adopt a declaration-aware credential authority model for runtime selection: ignore or loudly warn on stored provider profiles whose ids are no longer declared for that provider, and add non-destructive doctor/models-status diagnostics for stored-but-undeclared profiles.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to use config with no declared auth.profiles entry for provider anthropic, keep an anthropic:* credential in auth-profiles.json, and observe resolveAuthProfileOrder returning the stored profile before the Pi auth controller resolves and injects it.

## Solution Assessment

No. The current silent stored-profile fallback is not the best solution for a removed declaration; the safer maintainable path is a resolver allowlist or explicit warning plus diagnostic drift reporting, while preserving legacy migration recovery deliberately.
