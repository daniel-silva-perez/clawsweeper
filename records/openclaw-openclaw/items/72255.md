## Summary

Keep open. Current main still has a plausible path for stored auth-profile credentials to be selected by the embedded Pi runtime without requiring a matching `auth.profiles` declaration, and doctor/status health does not appear to warn about declared-config/store drift. The related #40363 and #43945 reports cover adjacent stale-key and silent-fallback behavior, but they do not supersede this removed-declaration/Pi credential-selection case.

## Best Possible Solution

Keep this open and make `auth.profiles` declaration authority explicit at selection time: Pi/model-auth should ignore or loudly warn on stored credentials whose profile ID is not declared when config declarations exist for that auth provider, while preserving intentional `auth.order` behavior only for declared IDs. Add `openclaw doctor`/models-status detection for stored-but-undeclared profiles with a safe repair/quarantine path, plus focused tests for removed Anthropic setup-token credentials, stored `auth.order`, and `claude-cli` provider alias handling.
