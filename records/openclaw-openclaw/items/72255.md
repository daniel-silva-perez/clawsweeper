## Summary

Keep open. Current main still lets runtime auth profile selection use stored provider credentials when no matching `auth.profiles` declaration exists, and the Pi path consumes that shared order without a visible undeclared-profile guard. The related reports #40363 and #43945 are adjacent stale-auth/fallback cases, but they do not fully cover this removed-declaration billing route.

## What This Changes

The issue requests strict or visible handling for per-agent auth-profile credentials that remain in `auth-profiles.json` after their declarations are removed from `openclaw.json`.

## Best Possible Solution

Make declaration authority explicit in runtime auth selection: stored credentials whose profile ids are no longer declared should either be ignored for normal runtime use or selected only with a high-visibility warning, while `openclaw doctor` or `models status` reports stored-but-undeclared profiles and offers a non-destructive cleanup path.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to leave a valid provider credential in the auth store, remove the matching `auth.profiles` declaration, and observe `resolveAuthProfileOrder` returning stored provider profiles that the Pi auth controller can resolve and inject.

## Solution Assessment

No. The current silent reuse behavior is not the best way to solve the issue; a narrow resolver allowlist or selection-time warning plus doctor/status drift detection is safer than hidden billing through removed declarations or destructive startup pruning.
