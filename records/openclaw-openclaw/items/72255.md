## Summary

Keep open. Current main still allows stored provider credentials from `auth-profiles.json` to become runtime auth candidates when there is no matching `auth.profiles` declaration, and the embedded Pi auth controller consumes those candidates without an undeclared-profile warning. The related stale-auth reports are adjacent but do not fully cover this removed-declaration billing path.

## What This Changes

The issue requests that runtime auth ignore or visibly warn on per-agent auth-profile credentials after their `openclaw.json` declarations are removed.

## Best Possible Solution

Make declaration authority explicit in runtime selection: either ignore stored profiles that are no longer declared for normal runtime use, or select them only with a high-visibility warning and a clear `openclaw doctor` or `models status` drift report plus non-destructive cleanup guidance.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to leave a valid credential in `auth-profiles.json`, remove the matching `auth.profiles` declaration, and observe `resolveAuthProfileOrder` returning the stored provider profile that Pi can inject for runtime auth.

## Solution Assessment

No. The current silent stored-profile fallback is not the best way to handle removed declarations; a declaration allowlist or explicit warning plus drift detection is safer than hidden billing through retired credentials or destructive startup pruning.
