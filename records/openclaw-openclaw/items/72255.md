## Summary

Keep open. Current main still allows the embedded Pi runtime to select stored auth-profile credentials without requiring a matching `auth.profiles` declaration, and the existing health/doctor path does not flag stored-but-undeclared profile drift.

## What This Changes

The issue requests that removed `auth.profiles` declarations stop their leftover per-agent `auth-profiles.json` credentials from being silently used by the Pi runtime, or at least produce a loud warning/doctor signal.

## Best Possible Solution

Make declaration authority explicit in the auth resolver and Pi candidate path: stored credentials whose profile IDs are no longer declared should be ignored or loudly warned about when declarations exist for that auth provider, and `openclaw doctor`/models status should report stored-but-undeclared profiles with a safe quarantine or repair flow.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to construct a config with no declared `anthropic:claude-set-up-token`, keep that provider credential in the auth store, and observe `resolveAuthProfileOrder` feeding it to the Pi auth controller candidate path.

## Solution Assessment

No, the current behavior is not the best way to solve the issue because it treats the per-agent store as selectable runtime state even after the user removes the declaration they reason about. A narrow resolver-level declaration allowlist plus doctor/status detection is safer than startup-time destructive pruning.
