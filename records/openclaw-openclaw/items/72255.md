## Summary

Keep open. Current main still permits stored provider credentials to be selected when no matching `auth.profiles` declaration exists, and the Pi runner can inject that selected credential into runtime auth without an undeclared-profile guard. The adjacent stale-auth/fallback reports are related, but they do not fully supersede this removed-declaration billing route.

## What This Changes

The issue asks OpenClaw to stop silently using per-agent stored auth-profile credentials after their `openclaw.json` declarations are removed, or at least warn/report that drift before runtime use.

## Best Possible Solution

Make runtime auth selection declaration-aware without destructive startup cleanup: normal Pi/runtime selection should ignore or loudly warn on stored profiles whose ids are no longer declared for that provider, and doctor or models status should surface stored-but-undeclared profiles with a non-destructive remediation path.

## Reproduction Assessment

Yes. A source-level reproduction is to provide config with no `auth.profiles` entry for `anthropic`, keep a valid `anthropic:*` credential in the auth store, and observe `resolveAuthProfileOrder` returning that stored profile, which the Pi auth controller then resolves and injects into `authStorage`.

## Solution Assessment

No. The current silent reuse behavior is not the best way to solve the issue; the narrowest maintainable path is a resolver allowlist or selection-time warning plus diagnostic drift reporting, not hidden billing through removed declarations or automatic secret deletion.
