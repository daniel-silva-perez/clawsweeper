## Summary

Keep this PR open. Current main covers the MiniMax-style JSON `api_error` body path, but it still does not implement the PR's explicit direct HTTP 520 handling in the transient HTTP-prefix set or structured status classifier. The PR also needs a narrow rebase because current main has a safer `api_error` transient-signal guard than the branch described in the PR discussion.

## Best Possible Solution

Keep this PR open for a narrow rebase or replacement patch: add 520 to `TRANSIENT_HTTP_ERROR_CODES`, classify direct HTTP 520 as `timeout`, and add focused tests for `isTransientHttpError`, `classifyFailoverReasonFromHttpStatus(520)`, and `resolveFailoverReasonFromError({ status: 520 })`, while preserving current main's safer JSON `api_error` transient-signal, billing/auth, and format/context guards.
