## Summary

Keep this PR open. Current main implements the safer MiniMax-style JSON `api_error` fallback from merged #49611, but it still has not implemented this PR's explicit HTTP 520 status handling: 520 is absent from the transient status set and the direct HTTP status classifier has no 520 branch. The PR should be rebased or replaced with a narrow 520 patch that preserves main's stricter transient-signal guard for `api_error` payloads.

## Best Possible Solution

Rebase or replace this PR with a narrow patch that adds 520 to `TRANSIENT_HTTP_ERROR_CODES`, adds an explicit 520 branch in `classifyFailoverClassificationFromHttpStatus`, and adds focused tests for `isTransientHttpError`, `classifyFailoverReasonFromHttpStatus`, and any signal path that should treat HTTP 520 as timeout, while preserving current main's transient-signal `api_error` guard from #49611.
