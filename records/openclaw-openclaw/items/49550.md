## Summary

Keep this PR open. Current main already implements the safer raw JSON `api_error` half, including MiniMax-style `unknown error, 520` payloads and guards for billing/auth/format/context cases, but it still does not implement the PR’s explicit HTTP 520 status handling. The PR is also stale against the stricter current classifier and should not be merged as-is.

## Best Possible Solution

Rework this PR or replace it with a smaller one that preserves current main’s safer `api_error` transient-signal classifier, then adds only the missing HTTP 520 status support in `TRANSIENT_HTTP_ERROR_CODES` and `classifyFailoverClassificationFromHttpStatus`, with focused tests for `isTransientHttpError("520 ...")`, `classifyFailoverReasonFromHttpStatus(520)`, and the existing non-transient api_error guard cases.
