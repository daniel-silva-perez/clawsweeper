## Summary

Keep open. Current main still mutates plugin config before proving the requested id exists, so the PR's intended fix is not implemented on main. The PR is a focused candidate for linked #73551, but the posted diff still needs fixup because it accepts display-name matches while forwarding the raw input string into config mutation helpers, which can persist the display name as a plugin config key instead of the canonical plugin id.

## Best Possible Solution

Keep this PR open for review and fixup. The best path is to land a corrected version that resolves the requested plugin through `buildPluginSnapshotReport` before any config write, forwards the canonical `targetPlugin.id` to enable/disable and slot selection, exits 1 without `replaceConfigFile` or registry refresh on a miss, and adds regression coverage for missing ids plus display-name-to-canonical-id behavior. Keep linked #73551 open until this PR or an equivalent fix merges.
