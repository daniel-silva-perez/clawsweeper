## Summary

Keep this PR open. Current main still lacks pre-write registry validation for `plugins enable` and `plugins disable`, and the PR remains a focused implementation candidate for the linked bug. The posted patch should not be closed as implemented because it still forwards raw user input after a display-name match, which can write the wrong config key.

## Best Possible Solution

Keep this PR open for maintainer review and fixup. The best path is to land a corrected implementation that resolves the requested plugin from the registry before any config write, exits non-zero without `replaceConfigFile` or registry refresh on a miss, forwards the canonical `targetPlugin.id` to enable/disable and slot selection, and adds regression coverage for missing ids plus display-name-to-canonical-id behavior. Keep linked #73551 open until this PR or an equivalent fix merges.

## Risks / Open Questions

- Current main can still persist stale `plugins.entries.<id>` records for typoed or nonexistent plugin ids and then report a successful enable/disable.

- A corrected patch needs to preserve built-in channel alias behavior, slot selection, include-backed config writes, and registry refresh semantics for valid enable/disable operations.
