## Summary

Keep open for normal PR review. Current main does not implement the proposed pricing-cache resilience work, but the PR branch has concrete correctness findings around config-pricing precedence and cooldown scheduling, plus a sequencing dependency on the still-open #73486.

## What This Changes

The PR lowers the Gateway pricing fetch timeout, adds disk-backed pricing cache warming and an in-memory circuit breaker, adds `startupBlockedReason` to startup-pending readiness responses, and updates Gateway tests.

## Best Possible Solution

Keep the resilience direction, but make disk cache warming overlay or follow current config and manifest seeding, schedule retries using remaining cache TTL and circuit cooldown, and coordinate with #73486 so optional pricing refresh cannot remain on the startup-critical path.

## Reproduction Assessment

Yes for the review findings: the PR head shows the fresh-cache return before `collectSeededPricing`, and the circuit-open branch calls the 24h `scheduleRefresh` helper. Focused unit tests with a fresh cache plus fake timers should reproduce both issues; runtime tests were not executed because this sweep was read-only.

## Solution Assessment

No. The proposed approach is maintainable in shape, but this implementation is not ready until it preserves current config-pricing precedence and uses cooldown/TTL-aware scheduling instead of the generic 24h refresh helper.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Apply current config before trusting disk cache:** `src/gateway/model-pricing-cache.ts:1156-1159`
  - body: The fresh-cache branch returns before `collectSeededPricing` runs. If a user edits `models.providers.*.models[].cost` or changes configured refs while a pricing file is still fresh, startup keeps the old disk map and local pricing no longer wins until the cache expires. Load or merge disk data after seeding current config, or overlay configured prices before returning.
  - confidence: 0.91
- **[P2] Retry after the circuit breaker cooldown:** `src/gateway/model-pricing-cache.ts:1199`
  - body: When the breaker is open, this branch schedules via `scheduleRefresh`, which always waits the 24h cache TTL even though the log reports the remaining 10-minute pause. After the breaker opens, a recovered pricing service can be ignored until the daily timer rather than the cooldown.
  - confidence: 0.9
- **[P3] Schedule disk refresh for the remaining TTL:** `src/gateway/model-pricing-cache.ts:1158`
  - body: After loading a fresh disk cache, this schedules the next refresh for 24h from startup instead of the remaining time until the loaded cache expires. A 23-hour-old cache can therefore remain in use for almost another day, and repeated restarts before the timer fires can keep postponing live refreshes.
  - confidence: 0.84
