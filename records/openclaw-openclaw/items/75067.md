## Summary

Keep this PR open. The resilience direction is relevant and current main still lacks the requested timeout, disk-cache, circuit-breaker, and readiness-field changes, but the PR head still has blocking correctness issues around pricing precedence and retry scheduling, and it needs maintainer/author revision before merge.

## What This Changes

The PR reduces Gateway model-pricing fetch timeout to 10s, adds disk-backed pricing cache warming/persistence and an in-memory circuit breaker, adds `startupBlockedReason` to startup-pending readiness responses, and updates Gateway tests.

## Best Possible Solution

Revise the branch so disk warming overlays current configured and manifest pricing, retry timers use the actual cache TTL or circuit-breaker cooldown, and the sequencing with #73486 ensures pricing refresh cannot block Gateway readiness.

## Reproduction Assessment

Yes for the review findings. The PR head shows the fresh disk-cache return before config seeding and both problematic paths using the fixed 24h scheduler; focused fake-timer and disk-warm coverage should reproduce the stale-config and delayed-retry cases.

## Solution Assessment

No. The architecture is reasonable, but this implementation is not the best fix until it preserves current pricing precedence and schedules retries from the real remaining TTL or breaker cooldown instead of the generic daily refresh helper.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Apply configured pricing before disk cache return:** `src/gateway/model-pricing-cache.ts:1182-1185`
  - body: A fresh disk cache returns before manifest/config seeding runs. If an operator changes `models.providers.*.models[].cost`, stale disk data can override local config for the cache window; overlay configured and manifest pricing before returning.
  - confidence: 0.91
- **[P2] Retry when circuit cooldown expires:** `src/gateway/model-pricing-cache.ts:1232`
  - body: The circuit-open branch logs the remaining cooldown but calls `scheduleRefresh`, which waits `CACHE_TTL_MS` (24h). After three failures, recovery can be ignored until the daily refresh instead of the 10-minute breaker cooldown.
  - confidence: 0.9
- **[P3] Schedule disk refresh from remaining TTL:** `src/gateway/model-pricing-cache.ts:1184`
  - body: A fresh disk cache always schedules `CACHE_TTL_MS` from process start. A 23-hour-old cache can therefore stay in use for almost another day, and restarts keep pushing the live refresh out; schedule from the loaded cache's remaining TTL.
  - confidence: 0.84
