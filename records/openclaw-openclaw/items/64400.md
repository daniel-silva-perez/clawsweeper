## Summary

Keep this issue open. Current main still sends doctor Phase A through the heavyweight `health` RPC under the 3000ms non-interactive budget, and a failed health gate can still reach the running-service restart prompt. The related PR #64511 was closed unmerged, while a later commenter also reports slow Windows `status`/`gateway status` behavior, so this is not obsolete.

## Best Possible Solution

Keep this issue open and implement the narrow core liveness fix: make doctor Phase A use a lightweight gateway liveness signal such as `status` before any service restart repair, classify accepted local probes as reachable/degraded when only follow-up detail RPCs time out, and add regression tests proving slow `health` does not trigger a false running-service restart. Treat any unauthenticated loopback auth relaxation as a separate maintainer security decision.
