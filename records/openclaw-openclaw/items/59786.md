## Summary

Keep open. Current main still treats exec approval allowlists as executable/path-name allowlists for the portable path. There is a Windows-only `argPattern` hook, plus exact-command durable `allow-always` reuse, but neither provides the requested documented, cross-platform subcommand-level allowlisting for cases like allowing `openclaw cron list` while still prompting for `openclaw gateway restart`.

## Best Possible Solution

Implement this in the core exec approval policy surface: add a documented, portable argv/subcommand condition model for allowlist entries, expose it through `openclaw approvals allowlist` and the Control UI, and add tests showing safe `openclaw cron ...` subcommands can be pre-approved while gateway restarts, approval mutations, and config writes still require approval.

## Risks / Open Questions

- Closing this as implemented would overstate the current protection: on POSIX hosts, allowlisting the `openclaw` executable still permits every subcommand through the allowlist match.
- This is a core exec-approval policy gap, not a ClawHub/plugin cleanup item, because enforcement happens in the gateway/node-host exec approval path.
