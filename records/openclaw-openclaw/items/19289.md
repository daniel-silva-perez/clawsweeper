## Summary

Keep this issue open. Current main still does not provide a browser backend/plugin registration interface for profile-selectable third-party MCP/custom drivers. It supports the bundled browser plugin, whole-tool replacement, managed/CDP profiles, and Chrome DevTools MCP existing-session profiles, but the config/schema/runtime/API surfaces are still closed to built-in driver modes.

## Best Possible Solution

Keep this issue open for maintainer design. The best path is an additive browser backend/adapter extension point in core and the Plugin SDK: define a typed browser backend contract, let plugins register driver ids, broaden browser profile config/schema/help/client/CLI/control routes to accept registered ids, route runtime dispatch through that registry, document the contract in browser and plugin SDK docs, and add contract tests proving MCP/custom adapters can bind without replacing the bundled browser plugin wholesale. Treat #52190 as backend-specific prior art and #55840 as related extension-relay demand, not as canonical replacements for this API gap.
