## Summary

Keep this issue open. Current main still has no browser backend/driver registration interface for third-party MCP or custom adapters: profile config and validation are closed to built-in driver values, runtime resolution collapses profiles to `openclaw` or `existing-session`, and the Plugin API documents/registers many capability types but not browser backends. The prior maintainer-side Codex review in the provided discussion reached the same conclusion, and current-main inspection at 1b25dcf57a9fca5d00cf275046e20a2e3ea3730c did not find a new seam that would make this a ClawHub-only close.

## Best Possible Solution

Keep this open for maintainer design. The best path is an additive browser backend/adapter extension point in core and the Plugin SDK: define a typed backend contract, allow plugins to register driver ids, broaden browser profile config/schema/help/client/CLI/control routes to accept registered ids, route runtime dispatch through that registry, document the contract in browser and plugin SDK docs, and add contract tests proving MCP/custom adapters can bind without replacing the bundled browser plugin wholesale. Treat #52190 and #55840 as related backend/relay demand, not replacements for this API gap.
