## Summary

Keep this issue open. Current main still has no browser backend or driver registration interface for third-party MCP/custom adapters: browser profile driver config is limited to built-in values, schema and route validation reject arbitrary driver ids, runtime resolution collapses all non-existing-session profiles to the managed OpenClaw driver, and the Plugin SDK registration surface does not expose a browser-backend capability. The existing maintainer-side Codex review in the issue reached the same conclusion, and #52190/#55840 are related demand signals rather than replacements for this core/plugin API gap.

## Best Possible Solution

Keep this open for maintainer design. The best path is an additive browser backend/adapter extension point in core and the Plugin SDK: define a typed backend contract, allow plugins to register driver ids, broaden browser profile config/schema/help/client/CLI/control routes to accept registered ids, route runtime dispatch through that registry, document the contract in browser and plugin SDK docs, and add contract tests proving MCP/custom adapters can bind without replacing the bundled browser plugin wholesale. Treat #52190 and #55840 as related demand signals, not replacements.

## Risks / Open Questions

- A browser backend adapter seam is a core/plugin contract and security boundary: third-party drivers could control authenticated browser sessions, route network requests, or execute page actions, so maintainer design should define trust, permissions, SSRF behavior, and lifecycle cleanup before implementation.
- The existing built-in Chrome MCP existing-session support is not enough to satisfy this issue because it is a fixed driver mode, not a general registration surface for MCP/custom browser engines.
- Related #52190 and #55840 show demand for alternate browser backends and extension relay behavior, but neither supersedes the generic API gap tracked here.
