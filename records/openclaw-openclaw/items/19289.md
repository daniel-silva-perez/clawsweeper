## Summary

Keep this issue open. Current main still does not expose a browser backend/driver registration interface for third-party MCP/custom adapters. The current surfaces allow replacing the bundled browser plugin wholesale, managed/CDP profiles, and built-in Chrome DevTools MCP existing-session profiles, but profile driver config, schema validation, runtime resolution, profile creation, and SDK docs remain closed to built-in driver modes.

## Best Possible Solution

Keep this open for maintainer design. The best path is an additive browser backend/adapter extension point in core and the Plugin SDK: define a typed browser backend contract, allow plugins to register driver ids, broaden browser profile config/schema/help/client/CLI/control routes to accept registered ids, route runtime dispatch through that registry, document the contract in browser and plugin SDK docs, and add contract tests proving MCP/custom adapters can bind without replacing the bundled browser plugin wholesale. Treat #52190 as backend-specific prior art and #55840 as related extension-relay demand, not as canonical replacements for this API gap.

## Risks / Open Questions

- Closing this now would lose tracking for a real core/plugin API gap: third-party code can replace the entire browser tool, but cannot register a profile-selectable backend id for the bundled browser runtime.
- A browser backend adapter seam is security-sensitive because native plugins run in-process and browser control crosses CDP, existing sessions, gateway auth, and SSRF/private-network policy boundaries.
- The likely fix is broad public API work across config, schema/help generation, browser runtime dispatch, Plugin SDK exports, CLI/control routes, docs, and contract tests, so it is not a safe one-off cleanup PR.
