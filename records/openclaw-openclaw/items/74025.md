## Summary

Keep open. Current main still has Lobster run/resume support only through the optional agent tool path, and Task Flow has status/inspection/cancel surfaces but no explicit channel slash command to manually run or resume a workflow. The feature is plausible, but the command syntax, authorization, owner/session binding, JSON argument handling, and approval/resume semantics need maintainer design first.

## What This Changes

Request to add a slash command that manually executes or resumes Task Flow/Lobster workflows with optional JSON arguments.

## Best Possible Solution

Keep the issue open for maintainer design, then implement a Lobster-owned plugin command if approved. The command should reuse the existing Lobster runner and managed TaskFlow runtime, bind to the active authorized session, parse JSON arguments deliberately, preserve approval/resume semantics, and document channel usage and failure cases.

## Reproduction Assessment

Not applicable as a runtime bug, but there is a high-confidence verification path. Source and docs inspection on current main show no /lobster, /taskflow, or flow-run command, while /tasks rejects arguments and only reports status.

## Solution Assessment

Unclear until maintainers settle the command contract. A plugin-owned command wrapping existing Lobster/TaskFlow execution appears narrower than adding a core workflow command, but the security and UX details are not settled enough for an automated fix PR.

## Risks / Open Questions

- A channel command for Lobster/Task Flow can trigger side-effectful local workflows, so owner-only behavior, authorization level, tool policy, approval/resume handling, JSON parsing, and safe error text need explicit design.
- Task Flow has managed and mirrored modes; a generic command could create ambiguous flow ownership or bypass intended session/requester binding if it is added without a narrow contract.
