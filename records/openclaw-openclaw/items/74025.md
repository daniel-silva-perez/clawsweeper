## Summary

Keep open. Current main still lacks the requested channel slash command for manually running Lobster or Task Flow workflows; the existing surfaces expose Lobster as an optional agent tool and Task Flow as inspect/cancel state management, not user-invoked workflow execution. Because this would expose local workflow execution from chat, the next step should be maintainer design rather than automatic cleanup or closure.

## What This Changes

Requests a chat slash command to manually execute Task Flow or Lobster workflows with optional JSON arguments.

## Best Possible Solution

Add an owner-reviewed Lobster plugin slash command that reuses the existing embedded Lobster runner and managed Task Flow runtime, with explicit authorization, structured JSON argument parsing, safe cwd/path handling, approval/resume support, user-safe errors, docs, and focused command/tool tests.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. We do have a high-confidence verification path for the missing capability by inspecting the current slash-command registry, /tasks handler, Task Flow CLI/docs, and Lobster plugin entrypoint on current main.

## Solution Assessment

Unclear until maintainer design is settled. The narrowest maintainable direction appears to be a plugin-owned api.registerCommand implementation in the Lobster plugin, not a new core built-in command, but it needs explicit security and UX decisions first.
