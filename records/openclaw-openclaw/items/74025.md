## Summary

Current main still lacks an explicit chat slash command for manually running Lobster or Task Flow workflows; existing surfaces are status/inspection/cancel commands plus the optional Lobster agent tool.

## Best Possible Solution

Keep this open for maintainer design. The likely product path is a Lobster-owned plugin command, probably registered from extensions/lobster via api.registerCommand(), that supports a small command grammar for run/resume plus optional JSON args, reuses the existing embedded runner and managed TaskFlow runtime, preserves sandbox/cwd/approval constraints, and updates docs/tests. If maintainers want this outside the bundled plugin, first define or document the safe public API an external ClawHub plugin would use.

## Risks / Open Questions

- A channel-triggered workflow command can execute local commands and side effects, so the design needs explicit owner/authorization gating, cwd/path limits, sandbox behavior, approval/resume semantics, and safe error output.
- Plugin command context is not identical to agent tool context; an implementation should preserve the existing Lobster tool safety checks and session-bound managed TaskFlow behavior instead of bypassing them.
- Closing this as ClawHub-only would be premature because the bundled Lobster runner is private to the Lobster plugin and there is not yet a clearly documented external plugin API for invoking that exact runtime safely from a command.
