## Summary

Current main still does not provide the requested explicit chat slash command for manually running Task Flow or Lobster workflows. The existing surfaces are `/tasks` status, `openclaw tasks flow list|show|cancel`, and the optional `lobster` agent tool. The plugin command API looks capable of hosting this, but channel-triggered workflow execution is security-sensitive and needs maintainer design before a fix PR.

## Best Possible Solution

Keep this issue open for maintainer design. The likely best implementation is a Lobster-owned plugin command using `api.registerCommand()` and the existing Task Flow/Lobster runtime, gated to authorized or owner contexts with structured JSON parsing, safe error reporting, approval/resume semantics, docs, and focused tests. If maintainers do not want this bundled, the same capability can be moved to a ClawHub plugin unless a missing generic plugin API is identified.

## Risks / Open Questions

- A channel slash command that runs Lobster workflows can trigger local in-process workflow execution from chat, so authorization, owner routing, approval/resume handling, and audit output need explicit design.
- Passing JSON arguments, workflow paths, pipeline strings, or cwd values through a command parser could reintroduce command-injection or path-boundary bugs if implemented casually.
- Native command providers have different argument limits and menu semantics, so command syntax and JSON handling need provider-aware design before implementation.
