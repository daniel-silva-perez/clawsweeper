## Summary

Keep open. Current main still exposes Lobster execution through the optional agent tool path and Task Flow inspection/cancel surfaces, but there is no channel slash command that manually runs or resumes a Lobster/Task Flow workflow. The feature is plausible, but command syntax, authorization, session/owner binding, JSON argument parsing, and approval/resume behavior need maintainer design first.

## What This Changes

Request a slash command to manually run or resume Task Flow/Lobster workflows from a channel, with optional JSON arguments.

## Best Possible Solution

Define a Lobster-owned plugin command contract, then implement it against the existing Lobster runner and managed TaskFlow runtime. The command should be explicitly authorized, bound to the active conversation/session, parse JSON arguments deliberately, preserve approval/resume semantics, and update slash-command plus Lobster/Task Flow docs.

## Reproduction Assessment

Not applicable as a runtime bug, but there is a high-confidence verification path. Source and docs inspection on current main show no /lobster, /taskflow, or flow-run command, while /tasks rejects arguments.

## Solution Assessment

Unclear until maintainers settle the command contract. A plugin-owned command wrapping the existing Lobster/TaskFlow execution path appears narrower than adding a core workflow command, but the security and UX choices are not specified enough for an automated fix PR.
