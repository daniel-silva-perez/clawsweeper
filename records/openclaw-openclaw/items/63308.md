## Summary

Keep open. Current main still exposes `sessionKey`, `label`, and `agentId` as independent optional `sessions_send` fields and still returns the reported mixed-selector error before resolving the target; the latest release tag has the same guard. Existing open PRs already cover the runtime and schema/docs paths, so this should stay open for maintainer consolidation rather than being closed or queued again.

## What This Changes

The issue asks for `sessions_send` to stop encouraging or failing mixed `sessionKey` plus `label` calls by adding clearer selector guidance and/or treating `sessionKey` as authoritative when both selectors are present.

## Best Possible Solution

Keep this issue open until a canonical fix lands that makes `sessionKey` authoritative when present, ignores redundant `label` and `agentId` on that path, documents the two valid addressing modes in schema/tool/docs text, and adds focused regression coverage for `sessionKey`-only, `label`-only, and mixed-selector calls.

## Reproduction Assessment

Yes. A focused check can call `createSessionsSendTool().execute` with both `sessionKey` and `label`; current source returns the exact reported error before any gateway send resolution.

## Solution Assessment

Yes, the narrow maintainable direction is `sessionKey` precedence plus schema/docs guidance. The open schema-only path is helpful but incomplete unless the runtime also tolerates redundant selectors from real caller paths.
