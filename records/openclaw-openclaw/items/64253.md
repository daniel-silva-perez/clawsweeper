## Summary

Keep this issue open. Current main has useful mitigations for active requesters and default queue steering, but the reported failure shape still exists for completion delivery: completion announces are direct-first, dormant or non-wakeable requesters still use a Gateway `agent` RPC with `expectFinal: true`, the default announce timeout remains 120000 ms, and `gateway timeout` is still classified as retryable.

## What This Changes

The issue requests a fix for Windows Gateway unresponsiveness under sustained subagent completion-announcement load, where 120s announce timeouts cascade into provider failover, lane backlog, and silent channels until restart.

## Best Possible Solution

Keep this as the focused Windows Gateway/subagent completion-timeout report. Maintainers should reproduce or add a stress regression for concurrent completion announcements against saturated Gateway `agent` calls, then choose a deliberate delivery change such as shorter direct completion timeout, earlier fallback, bypassing requester `agent` lanes for final delivery, isolating announce delivery from busy lanes, admission control for subagent fanout, or Windows unresponsive-Gateway recovery.
