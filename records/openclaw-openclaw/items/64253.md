## Summary

Keep this issue open. Current main still preserves the reported failure shape for completion announcements: manual subagent completion delivery is direct-first, dormant requesters still go through a Gateway `agent` call with `expectFinal: true`, the default announce timeout is still 120000 ms, and `gateway timeout` is still treated as transient/retryable. Recent yielded-subagent and fallback work helps adjacent paths, but it does not remove the long direct Gateway RPC path implicated by this Windows unresponsive-Gateway report.

## Best Possible Solution

Keep this open as the focused Windows Gateway/subagent completion-timeout report. The maintainer path is to reproduce or add a stress regression for concurrent subagent completion announcements against saturated Gateway `agent` calls, then decide whether completion delivery should use a shorter direct timeout, fall back before exhausting direct RPC retries, bypass requester `agent` lanes for final delivery, isolate announce delivery from busy lanes, add admission control for subagent fanout, or add an unresponsive-Gateway recovery path for Windows service installs. Cross-link #47975 and #71699 as related but not canonical duplicates.
