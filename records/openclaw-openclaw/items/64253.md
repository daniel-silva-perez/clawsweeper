## Summary

Keep this issue open. Current main still preserves the reported failure shape for dormant or non-wakeable completion requesters: subagent completion delivery is direct-first, the direct path calls Gateway `agent` with `expectFinal: true`, the default timeout is still 120000 ms, and `gateway timeout` remains transient/retryable. Recent active-requester and direct-send fallback paths may reduce some losses, but they do not remove the long direct `agent` RPC retry path implicated by the Windows unresponsive-Gateway report.

## Best Possible Solution

Keep this open as the focused Windows Gateway/subagent completion-timeout report. The maintainer path is to reproduce or add a stress regression for concurrent subagent completion announcements against saturated Gateway `agent` calls, then decide whether completion delivery should use a shorter direct timeout, fall back before exhausting the direct RPC, bypass the requester `agent` lane for final delivery, isolate announce delivery from busy lanes, add admission control for subagent fanout, or add an unresponsive-Gateway recovery path for Windows service installs. Cross-link #47975 and #71699 as related but not canonical duplicates.
