## Summary

Keep open. Current main still does not auto-select the only enabled account for omitted-account message sends: the shared runner preserves only explicit/default/bound account IDs, channel selection is channel-only, and WhatsApp still falls back to its default-account resolver. The related fix PR #20801 is closed unmerged, so this remains a valid narrow fix candidate.

## Best Possible Solution

Implement a shared outbound message-account resolver after channel selection. Preserve explicit accountId, input.defaultAccountId, and agent binding first; then use the selected channel plugin config adapter to prefer an enabled configured default account, auto-select exactly one enabled/configured account, and throw clear zero/multiple-enabled-account errors. Add focused shared-runner/channel-selection tests, WhatsApp regression coverage for omitted account sends, docs updates for routing/message behavior, and a user-facing changelog fix entry.

## Clownfish Work Prompt

Implement the fix for openclaw/openclaw#20756: when the `message` tool or `openclaw message` sends without `accountId`, auto-select the only usable account for the selected channel instead of falling through to a disabled/default account. Use #20801 only as prior context; it is closed unmerged.

Expected boundary:
- Keep core extension-agnostic. Do not hard-code WhatsApp or bundled channel IDs in shared core.
- Add a shared outbound message account resolver after channel selection and before plugin dispatch/hydration needs account-specific behavior.
- Preserve existing precedence first: explicit `params.accountId`, `input.defaultAccountId`, and current agent binding account.
- If no account is selected, use the selected channel plugin config adapter (`listAccountIds`, `resolveAccount`, `defaultAccountId`, `isEnabled`, `isConfigured` where present) to select an enabled configured default account; otherwise auto-select exactly one enabled/configured account.
- If no enabled/configured accounts exist, return a clear error. If multiple enabled/configured accounts exist and no default is usable, require explicit `accountId` with a clear error.
- Do not change global `createAccountListHelpers` / `resolveListedDefaultAccountId` behavior for setup, login, status, heartbeat, or account management.
- Add focused tests for the shared resolver/channel-selection behavior and `runMessageAction` plugin dispatch context. Include a WhatsApp omitted-account regression where `default` is disabled and a single named account is enabled.
