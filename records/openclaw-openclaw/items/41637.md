## Summary

Keep this PR open because the underlying Feishu multi-account webhook bug is still present on current main, but the proposed patch is not merge-ready. It introduces a useful shared-listener direction while regressing current webhook signature hardening, pre-auth body limits, and shared-server error handling.

## What This Changes

The PR adds a Feishu webhook server pool keyed by host:port, per-account route registration, pooled dispatch by token/app_id, request body replay, and pool-aware cleanup in Feishu monitor state and transport code.

## Best Possible Solution

Land a refreshed Feishu plugin fix that shares one webhook listener per host:port while preserving current raw-body signature validation, timing-safe comparison, 64 KiB / 5 second pre-auth limits, per-account abuse controls or an equivalent pooled pre-route guard, explicit joined-account error propagation, and focused tests for startup coalescing, dispatch, cleanup, malformed bodies, and server-error paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve raw-body signature verification:** `extensions/feishu/src/monitor.transport.ts:367-369`
  - body: The PR head validates signatures over JSON.stringify of the parsed payload and compares with ===. Current main signs the exact raw request body with a timing-safe comparison before JSON parsing, so valid signed requests whose JSON formatting differs can be rejected and the current hardening is lost.
  - confidence: 0.9
- **[P1] Keep the tightened pre-auth body limits:** `extensions/feishu/src/monitor.state.ts:31-32`
  - body: The PR head changes Feishu webhook limits to 1 MiB and 30 seconds, while current main uses 64 KiB and 5 seconds before authentication. The pooled path buffers unauthenticated bodies before routing, so relaxing these limits increases exposure and regresses existing coverage.
  - confidence: 0.88
- **[P1] Propagate shared-server errors to joined accounts:** `extensions/feishu/src/monitor.transport.ts:282-295`
  - body: When the pooled server emits error, this handler clears state for every route but only rejects the creator account's monitor promise. Joined accounts only resolve on abort, so they can remain marked as running after listener state has been torn down.
  - confidence: 0.87
- **[P2] Return malformed pooled JSON as 400:** `extensions/feishu/src/monitor.transport.ts:473-477`
  - body: In multi-account mode, malformed JSON makes route resolution return no route and the dispatcher maps that to 404. Current account handling returns 400 Invalid JSON, so pooled mode changes webhook error semantics and hides malformed payloads as routing misses.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff has no dependency, CI, lockfile, or publishing changes, but it modifies Feishu webhook authentication and unauthenticated buffering in ways that are not safe as-is.

Concerns:

- **[high] Stale signature hardening:** `extensions/feishu/src/monitor.transport.ts:65`
  - body: The PR head validates signatures over parsed JSON reserialization with a direct string comparison instead of current main's raw-body timing-safe check, which can reject valid signed requests and weakens the authentication boundary.
  - confidence: 0.9
- **[medium] Relaxed unauthenticated body limits:** `extensions/feishu/src/monitor.state.ts:31`
  - body: The PR head raises Feishu webhook pre-auth limits to 1 MiB and 30 seconds while current main tightened them to 64 KiB and 5 seconds with regression tests.
  - confidence: 0.88
- **[medium] Unroutable requests bypass account rate limits:** `extensions/feishu/src/monitor.transport.ts:428`
  - body: The pooled dispatcher omits rate limiting before buffering and only reaches per-account rate limits after successful routing, so invalid-token floods rely only on size and timeout caps.
  - confidence: 0.76
