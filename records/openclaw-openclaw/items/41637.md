## Summary

Keep this PR open. Current main still creates one Feishu webhook HTTP server per account and listens on that account's configured host/port, while gateway account startup is concurrent and Feishu docs still show a shared default webhook port. The PR remains a focused candidate for a real multi-account webhook bug, but it is not merge-ready because the provided review discussion identifies unresolved lifecycle, routing, and request-hardening regressions in the proposed pooled-server implementation.

## Best Possible Solution

Keep the PR open as a real Feishu plugin bug-fix candidate. The best implementation is a rebased pooled webhook listener keyed by host:port that registers per-account routes, routes by unambiguous token/app/path data without cross-account delivery, preserves the current 64 KiB body cap, 5 second body timeout, signature-before-JSON behavior, and per-account/path rate limiting, and correctly propagates bind/server errors and abort cleanup for creator and joiner accounts. Add regression tests for two webhook accounts sharing the same host:port, route disambiguation, bind failure, account stop cleanup, and the existing request-hardening guarantees.
