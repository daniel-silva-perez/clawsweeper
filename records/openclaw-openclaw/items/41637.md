## Summary

Keep this PR open. The reported Feishu multi-account webhook failure is still real on current main, but the PR is not merge-ready because the branch is stale against later Feishu webhook security hardening and still has concrete pooled-routing and lifecycle issues.

## What This Changes

The PR adds a Feishu webhook server pool keyed by host:port, with per-account route registration, pooled cleanup, and buffered body replay for multi-account routing.

## Best Possible Solution

Implement the shared host:port webhook server inside the Feishu plugin, but rebase it onto current main, preserve the existing webhook security guard behavior, settle all joined account monitor promises on pooled server failure, route only on unambiguous account identity, and add targeted multi-account webhook tests for startup, abort, malformed bodies, rate limiting, and shared-token cases.

## Reproduction Assessment

Yes. Configure two enabled Feishu accounts in webhook mode with the same `webhookHost:webhookPort`; current main starts per-account webhook monitors and each monitor calls `server.listen`, so the second account has a clear `EADDRINUSE` path.

## Solution Assessment

No. The Feishu-plugin-local pool is the right boundary, but this PR branch is not the best mergeable form until it is rebased over current webhook hardening and fixes the pooled error/routing edge cases.

  - body: The shared server error handler clears state for every registered route but only rejects the creator account's monitor promise. Accounts that joined through `joinExistingServer` keep waiting for abort, so the gateway can leave them marked running with no webhook listener after a server error.
  - confidence: 0.84
- **[P2] Rate-limit requests before pooled buffering:** `extensions/feishu/src/monitor.transport.ts:429-433`
  - body: The multi-account dispatcher only checks method/content-type before buffering, and unroutable requests return 404 before any per-account handler runs. That regresses current pre-auth throttling for invalid webhook floods and lets attackers consume the body parser up to the full size/timeout budget without being counted.
  - confidence: 0.8
- **[P2] Disambiguate token-only route matches:** `extensions/feishu/src/monitor.transport.ts:519-531`
  - body: After the token+appId pass, the token-only fallback returns the first matching route. Multiple Feishu accounts can share an inherited verification token, so payloads without app_id, including url_verification, can be delivered to the wrong account instead of matching path or rejecting ambiguity.
  - confidence: 0.81

  - confidence: 0.76
