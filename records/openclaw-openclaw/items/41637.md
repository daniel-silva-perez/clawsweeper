## Summary

Keep this PR open. Current main still creates one Feishu webhook HTTP server per account, so the reported shared host:port collision remains real, but this PR branch is not merge-ready because it is stale against current webhook hardening and still has pooled-listener lifecycle and request-validation regressions.

## What This Changes

This PR adds a Feishu webhook server pool keyed by host:port, registers per-account routes, dispatches pooled requests by token or app_id, and updates Feishu monitor cleanup for shared servers.

## Best Possible Solution

Rebase or replace this PR with a focused Feishu plugin fix that keeps one listener per webhook host:port while preserving current raw-body signature validation, 64 KiB / 5 s pre-auth body limits, per-account rate limiting, unambiguous route selection, lifecycle/error propagation for all joined accounts, and targeted regression tests for shared startup, dispatch, cleanup, and server-error cases.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Preserve raw-body signature verification:** `extensions/feishu/src/monitor.transport.ts:65-69`
  - body: The PR head still computes signatures from `JSON.stringify(parsedPayload)` and compares with `===`. Current main validates the exact raw request body with a timing-safe comparison; rebasing this as-is would break valid signed Feishu webhooks whose body formatting differs and would drop the current timing-safe check.
  - confidence: 0.9
- **[P1] Keep tightened pre-auth body limits:** `extensions/feishu/src/monitor.state.ts:31-32`
  - body: The PR head still uses 1 MiB / 30 s webhook body limits, while current main and its security tests enforce 64 KiB / 5 s. The pooled path buffers unauthenticated bodies before routing, so relaxing these limits increases resource exposure and regresses the existing hardening.
  - confidence: 0.88
- **[P1] Propagate shared server errors to joiners:** `extensions/feishu/src/monitor.transport.ts:282-295`
  - body: On `server.error`, this clears state for every registered route but only rejects the creator account's monitor promise. Joined accounts can stay pending and appear running with no listener, so gateway restart/backoff will not repair those accounts.
  - confidence: 0.87
- **[P2] Return malformed pooled JSON as 400:** `extensions/feishu/src/monitor.transport.ts:475-477`
  - body: In multi-account mode, malformed JSON makes `resolveRoute` return `undefined`, and the dispatcher maps that to `404 Not Found`. The existing account handler returns `400 Invalid JSON`, so pooled mode changes webhook error semantics and anomaly accounting for malformed requests.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff has no CI, dependency, lockfile, or package-publishing changes, but it modifies Feishu webhook authentication and pooled request buffering in ways that need security review before merge.

Concerns:

- **[high] Stale signature hardening:** `extensions/feishu/src/monitor.transport.ts:65`
  - body: The PR head validates signatures over a parsed JSON reserialization and direct string comparison instead of current main's raw-body, timing-safe check, which can reject valid signed payloads and lose current security hardening.
  - confidence: 0.9
- **[medium] Relaxed unauthenticated body limits:** `extensions/feishu/src/monitor.state.ts:31`
  - body: The PR head uses 1 MiB and 30 s body limits while current main tightened Feishu webhook pre-auth limits to 64 KiB and 5 s with regression coverage.
  - confidence: 0.88
- **[medium] Unroutable pooled requests bypass per-account rate limits:** `extensions/feishu/src/monitor.transport.ts:428`
  - body: The pooled dispatcher deliberately runs only method/content-type guards before buffering; requests that cannot be routed never reach the per-account rate limiter, so invalid-token floods rely only on body size/time caps.
  - confidence: 0.76
