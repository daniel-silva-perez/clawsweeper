## Summary

Keep this PR open. The underlying Feishu webhook bug is still real on current main, but the proposed patch is not merge-ready because the pooled handler regresses the current raw-body webhook security flow and leaves lifecycle/routing edge cases unresolved.

## What This Changes

The PR adds a Feishu webhook server pool keyed by host:port, dispatches pooled requests by verification token/app_id, replays buffered bodies to per-account handlers, and makes stop cleanup aware of shared servers.

## Best Possible Solution

Implement shared Feishu webhook binding in the Feishu plugin, but rebase it onto the current raw-body guard/signature flow, keep pooled lifecycle state consistent for every account task, reject ambiguous route matches instead of choosing the first account, and add focused pooled multi-account tests before merge.

## Reproduction Assessment

Yes. Current main starts Feishu accounts concurrently and each webhook account binds its own http.Server, so two webhook accounts sharing host:port reproduce the EADDRINUSE failure path. The PR-specific regressions are reproducible with existing webhook security tests plus a new two-account pooled lifecycle/routing test.

## Solution Assessment

No. Sharing a host:port pool is the right direction, but this branch is not the best mergeable form because it is built around a stale parsed-body webhook handler and has unresolved pooled error/routing semantics; a narrow rebase plus tests is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve raw-body signature verification in pooled handlers:** `extensions/feishu/src/monitor.transport.ts:345-369`
  - body: The new guarded handler reads and parses JSON before signature verification, then validates the parsed payload. Current main verifies the raw request body before JSON parsing, so this regresses the auth boundary and can reject valid signed payloads whose raw JSON serialization differs. Rebase this helper onto the current raw-body `readWebhookBodyOrReject` flow.
  - confidence: 0.9
- **[P2] Propagate shared server errors to joined accounts:** `extensions/feishu/src/monitor.transport.ts:282-295`
  - body: When the pooled server emits `error`, this handler deletes state for every registered route but only rejects the creator account's monitor promise. Accounts that joined through `joinExistingServer()` only resolve on abort, so they can remain marked running with no webhook socket or restart trigger.
  - confidence: 0.86
- **[P2] Reject ambiguous token-only route matches:** `extensions/feishu/src/monitor.transport.ts:519-522`
  - body: If multiple accounts inherit the same top-level verification token, the token-only fallback returns the first matching route when `app_id` is absent. That can send a valid event or url_verification challenge to the wrong account; include app_id/path disambiguation or reject ambiguous matches.
  - confidence: 0.82
- **[P3] Return malformed pooled webhooks as bad requests:** `extensions/feishu/src/monitor.transport.ts:474-478`
  - body: Malformed JSON makes `resolveRoute()` return `undefined`, which the dispatcher maps to 404. Current webhook behavior distinguishes invalid bodies from unroutable requests with a 400 response, so pooled mode should preserve that error semantics.
  - confidence: 0.79

## Security Review

Status: needs_attention

Summary: The patch needs security attention because the pooled handler is based on an older parsed-body signature flow and can misroute shared-token accounts.

Concerns:

- **[high] Parsed-body signature validation regression:** `extensions/feishu/src/monitor.transport.ts:345`
  - body: The PR's new handler parses JSON before authentication and validates the parsed payload instead of the raw body. Current main intentionally authenticates the raw body first, so this weakens the webhook auth boundary and risks rejecting or accepting behavior that differs from Feishu's signed bytes.
  - confidence: 0.9
- **[medium] Ambiguous shared-token routing:** `extensions/feishu/src/monitor.transport.ts:519`
  - body: The route resolver accepts the first token-only match. Because the config allows accounts to inherit a top-level verification token, pooled mode can route one account's webhook to another account when app_id is missing, crossing account boundaries.
  - confidence: 0.82
