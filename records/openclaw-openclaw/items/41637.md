## Summary

Keep open. The underlying Feishu multi-account webhook bug still exists on current main, but this PR head is stale and not merge-ready because it would regress current webhook signature/body-limit hardening and still has shared-listener lifecycle gaps.

## What This Changes

This PR adds a Feishu webhook server pool keyed by host:port, registers per-account routes, dispatches pooled requests by token/app_id, replays buffered requests to account handlers, and updates Feishu monitor cleanup.

## Best Possible Solution

Land a refreshed Feishu plugin fix that keeps one listener per webhook host:port while preserving current raw-body signature validation, timing-safe comparison, 64 KiB / 5 s pre-auth limits, sensible abuse controls for unroutable pooled requests, per-account lifecycle semantics, and focused regression tests for startup, dispatch, cleanup, and server-error paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Preserve raw-body signature verification:** `extensions/feishu/src/monitor.transport.ts:365-369`
  - body: The shared handler still validates signatures over a parsed JSON object that is reserialized with `JSON.stringify` and compared with `===`. Current main signs the exact raw request body and uses timing-safe comparison; this PR would reject valid signed payloads whose formatting differs and drop the current hardening.
  - confidence: 0.9
- **[P1] Keep the tightened pre-auth body limits:** `extensions/feishu/src/monitor.state.ts:31-32`
  - body: The PR head still uses 1 MiB and 30 s webhook body limits, while current main enforces 64 KiB and 5 s before authentication. The pooled path buffers unauthenticated bodies before routing, so relaxing these limits increases exposure and regresses existing security tests.
  - confidence: 0.88
- **[P1] Propagate shared server errors to joiners:** `extensions/feishu/src/monitor.transport.ts:282-295`
  - body: When the pooled server emits `error`, this handler removes state for every registered route but only rejects the creator account's monitor promise. Joined accounts can remain pending and appear running even though their listener state has been torn down, so restart/backoff will not repair them.
  - confidence: 0.87
- **[P2] Return malformed pooled JSON as 400:** `extensions/feishu/src/monitor.transport.ts:473-477`
  - body: In multi-account mode, malformed JSON makes route resolution return undefined and the dispatcher maps that to `404 Not Found`. The existing account handler returns `400 Invalid JSON`, so pooled mode changes webhook error semantics for malformed requests.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff has no dependency, CI, lockfile, or publishing changes, but it modifies Feishu webhook authentication and unauthenticated request buffering in security-sensitive ways that are not safe as-is.

Concerns:

- **[high] Stale signature hardening:** `extensions/feishu/src/monitor.transport.ts:65`
  - body: The PR head validates Feishu signatures over parsed JSON reserialization with a direct string comparison instead of current main's raw-body timing-safe check, which can reject valid signed requests and loses current hardening.
  - confidence: 0.9
- **[medium] Relaxed unauthenticated body limits:** `extensions/feishu/src/monitor.state.ts:31`
  - body: The PR head uses 1 MiB and 30 s pre-auth body limits while current main tightened Feishu webhook limits to 64 KiB and 5 s with regression coverage.
  - confidence: 0.88
- **[medium] Unroutable pooled requests bypass per-account rate limits:** `extensions/feishu/src/monitor.transport.ts:428`
  - body: The pooled dispatcher runs only method/content-type checks before buffering; requests that cannot be routed never reach the per-account rate limiter, so invalid-token floods rely only on body size/time caps.
  - confidence: 0.76
