## Summary

Keep open. Current main still has the per-account Feishu webhook listener behavior that this PR targets, but the PR head is dirty against main and is not merge-ready because it would regress current webhook signature, throttling, routing, and shared-listener lifecycle behavior.

## What This Changes

The PR adds a Feishu webhook server pool keyed by host:port, registers per-account routes, dispatches pooled requests by verification token or app_id, and changes Feishu monitor cleanup to remove pooled routes before closing a shared server.

## Best Possible Solution

Keep the shared-server fix inside the Feishu plugin, but refresh it over current main with a pool design that preserves raw-body signature validation, the 64 KiB / 5 s pre-auth body budget, per-account throttling semantics, joined-account lifecycle/error propagation, unambiguous route matching, and focused multi-account webhook tests.

## Reproduction Assessment

Yes. Configure two enabled Feishu accounts in webhook mode with the same webhookHost:webhookPort; current main starts each account through monitorWebhook, and each invocation creates and listens its own http.Server.

## Solution Assessment

No. A Feishu-plugin-local pooled server is the right boundary, but this branch is dirty against main and its current head is not the narrowest maintainable fix because it regresses newer webhook security and lifecycle behavior.

  - confidence: 0.84

  - body: In multi-account mode the dispatcher buffers and parses requests before routing, but the pre-buffer guard omits the existing rate limiter. Invalid-token floods can consume body timeout and memory budget without being counted by the current per-account limiter.
  - confidence: 0.82

  - confidence: 0.81

  - body: Malformed JSON makes route resolution fail and the dispatcher returns 404. Current webhook handling returns an invalid-body response for signed malformed JSON, so pooled mode changes error semantics and hides bad webhook bodies as unknown routes.
  - confidence: 0.76

## Security Review

Status: needs_attention

Summary: The PR changes unauthenticated Feishu webhook ingress and has concrete regressions around signature validation, request throttling, route isolation, and body-budget hardening.

Concerns:

- **[medium] Raw-body signature hardening can be lost:** `extensions/feishu/src/monitor.transport.ts:365`
  - body: The branch validates signatures using parsed JSON serialization and direct string equality, while current main validates the exact raw body with safeEqualSecret. The shared-server implementation must preserve the current raw-body contract.
  - confidence: 0.87
- **[medium] Unroutable pooled requests bypass throttling:** `extensions/feishu/src/monitor.transport.ts:428`
  - body: The pooled dispatcher buffers and parses before route resolution without using the existing rate limiter, and route misses return before per-account guards run. Invalid-token traffic can consume request resources without being counted.
  - confidence: 0.82
- **[medium] Ambiguous token-only routing can cross accounts:** `extensions/feishu/src/monitor.transport.ts:519`
  - body: The token-only fallback selects the first matching route when multiple accounts share a verification token, which can misdeliver webhook payloads across accounts when app_id is absent or ambiguous.
  - confidence: 0.8
- **[low] Stale body budget is broader than current main:** `extensions/feishu/src/monitor.state.ts:31`
  - body: The branch carries 1 MiB and 30 s webhook body limits, while current main uses 64 KiB and 5 s. A rebase should keep the tighter unauthenticated request budget unless maintainers intentionally change it.
  - confidence: 0.78
