## Summary

Keep open. Current main still has the reported Feishu multi-account webhook port-conflict path, but this branch is not merge-ready because it is stale against current webhook hardening and introduces concrete lifecycle, routing, and unauthenticated ingress regressions.

## What This Changes

The branch adds a Feishu webhook server pool in `monitor.state.ts` and `monitor.transport.ts` so accounts sharing a host:port register per-account routes on one HTTP server with buffered multi-account routing and pooled cleanup.

## Best Possible Solution

Implement the shared host:port server inside the Feishu plugin, but rebase it onto current main, preserve raw-body `safeEqualSecret` signature checks and the current 64 KiB / 5 s pre-auth body budget, keep pre-body throttling effective for unroutable requests, propagate pooled server failures to every joined monitor task, reject ambiguous route matches, and add focused multi-account webhook tests for startup, abort, malformed bodies, rate limiting, signed payloads, and shared-token cases.

## Reproduction Assessment

Yes. Configure two enabled Feishu accounts in webhook mode with the same `webhookHost:webhookPort`; current main creates and listens a separate HTTP server per account, so the second listener has a clear `EADDRINUSE` path.

## Solution Assessment

No. A Feishu-plugin-local server pool is the right boundary, but this branch is not the best mergeable fix until it is rebased over current webhook hardening and the pooled lifecycle, throttling, and ambiguous-routing edge cases are fixed.

  - body: This branch validates signatures from `JSON.stringify(bodyResult.value)` and direct equality.   - body: The pooled `server.on("error")` path clears state for every registered route but only rejects the creator account promise. Accounts that joined the pool keep waiting for abort, leaving them marked running after their webhook listener and route were torn down.
  - confidence: 0.84
- **[P2] Rate-limit requests before pooled buffering:** `extensions/feishu/src/monitor.transport.ts:428-433`
  - body: In multi-account mode, only method/content-type checks run before buffering, and route misses return before any per-account handler sees the request. Invalid-token floods can therefore consume the body-buffer timeout/size budget without being counted by the existing webhook rate limiter.
  - confidence: 0.82
- **[P2] Reject ambiguous token-only route matches:** `extensions/feishu/src/monitor.transport.ts:519-531`
  - body: After the token+appId pass, the token-only fallback returns the first matching route. If accounts share an inherited verification token and a payload lacks `app_id`, events can be delivered to the wrong account instead of being matched by a stronger key or rejected as ambiguous.
  - confidence: 0.81
- **[P3] Return malformed JSON as an invalid body:** `extensions/feishu/src/monitor.transport.ts:473-478`
  - body: Malformed JSON makes `resolveRoute` return no route, which the pooled dispatcher maps to 404. Current handling returns 400 for invalid JSON, so multi-account mode would hide bad webhook bodies as unknown routes.
  - confidence: 0.76
