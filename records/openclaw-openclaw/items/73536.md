## Summary

Keep this PR open. Current main still does not pass an MCP RequestOptions timeout to bundled MCP client.callTool calls, so the intended #60967 fix is not implemented. The PR is a plausible implementation candidate, but it still needs maintainer review because the patch reuses the 30-second connection timeout as the per-call timeout, which can regress the SDK's current 60-second unconfigured request default, and it lacks a focused timeout propagation test.

## Best Possible Solution

Keep this PR open as the implementation candidate for #60967, but require a small update before merge: pass a per-tool request timeout through Client.callTool without regressing the SDK's 60-second default for unconfigured servers, fix the config-key wording, and add a focused test that captures the third callTool argument. If maintainers want separate semantics, add and document a dedicated request/tool timeout field instead of overloading connectionTimeoutMs.
