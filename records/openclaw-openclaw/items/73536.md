## Summary

Keep this PR open. Current main still does not implement the intended bundled MCP per-request timeout, and the linked bug remains actionable. The PR is a plausible implementation candidate, but the review concern about changing the unconfigured default from the MCP SDK's 60 seconds to OpenClaw's 30-second connection default still needs maintainer resolution and a focused regression test.

## Best Possible Solution

Keep this PR open and review it as the implementation candidate for #60967. The best path is a narrow fix in `src/agents/pi-bundle-mcp-runtime.ts` that passes a request timeout to `Client.callTool(..., undefined, { timeout })` while preserving the MCP SDK's 60-second default for unconfigured servers, or introduces a separate documented request/tool timeout field. Add a focused test that captures the third SDK argument, then land this PR or a maintainer-authored equivalent so the linked bug can close after merge.

## Risks / Open Questions

- Merging the PR as written could silently lower unconfigured bundled MCP tool-call timeout behavior from the SDK default of 60 seconds to OpenClaw's 30-second connection-timeout default.
- There is no focused regression test on current main or in the provided PR patch that asserts `Client.callTool(..., undefined, { timeout })` receives the intended third argument.
- If `connectionTimeoutMs` is reused for tool calls, docs and changelog wording should make that expanded meaning explicit; otherwise a separate request/tool timeout field would avoid overloading a connection setting.
