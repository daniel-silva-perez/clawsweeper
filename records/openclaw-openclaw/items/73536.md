## Summary

Keep open. Current main still lacks a per-request timeout for bundled MCP tool calls, and this PR is an active implementation candidate for the linked long-running MCP timeout bug. The code direction is narrow and matches the MCP SDK contract, but the branch should not merge as-is because it changes public `connectionTimeoutMs` semantics without updating docs or `CHANGELOG.md`, and the exact head currently has failing checks.

## What This Changes

The PR adds a resolved bundled-MCP request timeout derived from explicit `connectionTimeoutMs` or the MCP SDK 60s default, threads it through MCP transport/session runtime, and passes it as `RequestOptions.timeout` to `Client.callTool()` with config regression tests.

## Best Possible Solution

Land this PR or an equivalent current-main patch after adding public docs and a changelog entry, preserving the 60s request-timeout fallback for unconfigured servers while applying an explicit operator timeout to long-running tool calls.

## Reproduction Assessment

Yes. A static reproduction is high-confidence: configure a bundled MCP server with `connectionTimeoutMs` above 60,000 ms and call a tool that runs longer than 60s; current main never passes that value to `Client.callTool()`, so the SDK default still applies. I did not run a live slow MCP server because this review was read-only.

## Solution Assessment

No, not as-is. Passing `RequestOptions.timeout` with a 60s fallback is the right narrow code direction, but the mergeable solution should update docs/changelog and clear CI; a separate public `requestTimeoutMs` key is a broader follow-up decision, not required for this bug.
