## Summary

Keep this PR open. Current main still does not pass a per-request timeout to bundled MCP `callTool` calls, and the linked bug report remains open. The PR is a plausible implementation candidate, but it still needs maintainer review for the default-timeout regression noted in review and ideally a focused regression test.

## Best Possible Solution

Keep this PR open and review it as the implementation candidate for #60967. The best path is a narrow fix in `src/agents/pi-bundle-mcp-runtime.ts` that passes a request timeout to `Client.callTool(..., undefined, { timeout })` while preserving the SDK's 60-second default for unconfigured servers, or introduces a separate documented request/tool timeout field. Add a focused test that captures the third SDK argument, then land the PR or a maintainer-authored equivalent and let the linked bug close after merge.

## Risks / Open Questions

- Merging the PR as written could silently lower unconfigured bundled MCP tool-call timeout behavior from the SDK default 60 seconds to OpenClaw's default 30-second connection timeout.
- There is no focused regression test on current main or in the PR patch that asserts the third positional `Client.callTool(..., undefined, { timeout })` argument is passed.
- If `connectionTimeoutMs` is intentionally reused for tool calls, docs and changelog wording should make that expanded meaning explicit; otherwise a separate request/tool timeout config would avoid overloading the connection setting.
