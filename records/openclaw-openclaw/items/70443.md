## Summary

Keep open. Current main still lacks any `--offline` handling for Docker setup, and the script continues to build or pull images before the later permission repair, onboarding, gateway config sync, and compose startup steps. The reporter's linked workaround is useful evidence of the desired flow, but it needs an in-repo implementation that preserves default online behavior.

## What This Changes

Add an explicit `--offline` flag to `scripts/docker/setup.sh` so airgapped reruns can use preloaded Docker images while still running the rest of setup.

## Best Possible Solution

Add a narrow `--offline` mode in the Docker setup script, keep the no-flag build/pull behavior unchanged, verify the required main image with `docker image inspect`, warn rather than fail for a missing optional sandbox image, and document/test the airgapped rerun workflow.

## Reproduction Assessment

Yes. Passing `--offline` to current main is a high-confidence static reproduction because the script does not parse arguments and still reaches the normal build/pull branch before permission repair and onboarding.

## Solution Assessment

Yes. The explicit opt-in flag is the narrowest maintainable direction; the linked external patch is useful prior art but should be adapted rather than copied because it changes default local-build behavior.
