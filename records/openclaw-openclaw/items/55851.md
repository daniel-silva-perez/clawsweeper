## Summary

Keep this PR open. Current main still uses generic rate-limit, overloaded, and transport error copy, and the formatter/payload APIs still do not carry profileId or trigger into user-facing assistant error text. The PR remains a plausible implementation candidate, but it needs rebase/review work around transient api_error failover behavior, focused context tests, and a privacy decision on surfacing raw provider request IDs.

## Best Possible Solution

Keep this PR open as a focused implementation candidate. The best path is to rebase onto current main, preserve the existing broad transient api_error failover behavior, add direct tests for the context builder and formatter/payload plumbing, keep console diagnostics assertions strong, and decide whether provider request IDs should be omitted, hashed, or explicitly allowed in user-facing channel messages. If merged, it can resolve the still-open provider-name/error-context request such as #51336 and related closed context reports.
