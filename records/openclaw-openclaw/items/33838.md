## Summary

Keep this issue open. Current main still models node-declared commands as `string[]` in connect, pairing, persistence, runtime registry, discovery, and generated client surfaces; invoke params/results remain unknown payloads with only envelope-level validation. The related #8287 is adjacent because it asks for node-registered agent tools with parameter schemas, but it is broader and does not supersede this command-schema protocol request. No open PR found with closing syntax for this issue.

## Best Possible Solution

Keep open for a maintainer-designed additive gateway protocol change. Preserve `commands: string[]` compatibility, add optional per-command metadata/schema declarations, carry them through connect, node pairing request/persistence, registry/session state, `node.list`/`node.describe`, generated Swift/Kotlin clients, node-host/plugin runtime APIs, docs, and focused tests. Validation semantics for `node.invoke` params/results should be explicit.
