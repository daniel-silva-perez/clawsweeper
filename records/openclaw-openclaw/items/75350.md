## Summary

Keep open, but the revised PR is still not merge-ready. It preserves DeepSeek V4 tool-call replay now, but it removes the current main behavior that backfills `reasoning_content` on plain assistant replay messages; that behavior is covered by current SDK/plugin/live tests, provider docs, and the changelog entry for #73417.

## What This Changes

The PR changes the DeepSeek V4 OpenAI-compatible thinking wrapper so enabled-thinking requests strip `reasoning_content` from assistant messages without `tool_calls` while preserving or backfilling it for assistant messages with tool calls.

## Best Possible Solution

Keep the enabled-thinking DeepSeek V4 replay path preserving or backfilling `reasoning_content` for plain and tool-call assistant replay unless maintainers obtain live evidence for a narrower endpoint-specific failure. If that split is real, implement it with provider/turn-aware tests, live proof, docs, and a changelog entry while preserving the #71372 and #73417 replay fixes.

## Reproduction Assessment

Yes. Current SDK, DeepSeek plugin, and live-test fixtures construct enabled DeepSeek V4 replay payloads with prior plain assistant messages and assert `reasoning_content` is present; this PR would remove it from those payloads.

## Solution Assessment

No. The revised helper is better than the initial blanket strip because it preserves tool-call turns, but deleting plain assistant `reasoning_content` conflicts with current shipped behavior and tests; a safer fix needs live evidence and a narrower rule before changing the replay contract.
