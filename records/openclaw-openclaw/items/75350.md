## Summary

Keep open, but this patch should not merge as written. The current enabled-thinking path is intentionally preserving/backfilling DeepSeek V4 `reasoning_content`, and DeepSeek's current thinking-mode docs require that field to be passed back for thinking-mode tool-call turns; the PR's blanket strip would reintroduce the same 400 replay failure that existing tests and changelog entries cover. ([api-docs.deepseek.com](https://api-docs.deepseek.com/guides/thinking_mode))

## What This Changes

The PR changes the DeepSeek V4 OpenAI-compatible thinking wrapper to strip `reasoning_content` from outgoing messages in the thinking-enabled branch instead of ensuring assistant replay messages carry it.

## Best Possible Solution

Keep the enabled-thinking replay path preserving or backfilling `reasoning_content` for DeepSeek V4 tool-call continuations. If maintainers can live-reproduce a rejection on plain no-tool follow-up turns, the maintainable fix should be turn-boundary-aware and covered by DeepSeek unit/live tests, docs, and changelog, rather than stripping the field for every enabled-thinking request.

## Reproduction Assessment

Yes. Existing unit/live test fixtures build the enabled DeepSeek V4 replay payload and assert `reasoning_content` is present; replacing the enabled-path helper with `stripDeepSeekV4ReasoningContent` would make those assertions fail and conflicts with DeepSeek's documented 400 path for tool-call thinking replay.

## Solution Assessment

No. A blanket strip in the enabled path is not the best way to solve the reported 400 because it removes the field DeepSeek requires for thinking-mode tool-call replay; a safer alternative would distinguish no-tool follow-up cleanup from tool-call continuation replay if live evidence proves that split is needed.
