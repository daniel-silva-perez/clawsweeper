## Summary

Keep open. Current main still lacks Volcengine Ark Responses session caching, so the PR is not obsolete, but the proposed patch is not ready to merge because the cache can reuse stale system/developer prompts and the retry path can mix failed-attempt events with retry output.

## What This Changes

The PR adds a Volcengine provider stream wrapper, unit tests, and docs for opt-in Ark Responses session caching using `previous_response_id`, `caching`, `store`, `expire_at`, and optional thinking settings.

## Best Possible Solution

Keep the work provider-local, but require a rebase onto current main, add regression tests for prompt/system-message invalidation and stale-response retry behavior, preserve the current Volcengine TTS docs, and then validate the Volcengine plugin tests plus the changed gate before merge.

## Reproduction Assessment

Yes for the blocking review findings. A focused fake stream can emit a partial event before a stale `previous_response_id` error to show retry output mixing, and a two-turn same-session payload with changed `context.systemPrompt` follows directly from the PR code plus the pi-ai v0.70.6 input serialization contract.

## Solution Assessment

No, not as currently written. The provider-local wrapper is the right direction, but it needs to hash or otherwise invalidate system/developer input changes and retry without reusing an already-emitted output stream.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Invalidate cache when the system prompt changes:** `extensions/volcengine/cache.ts:277-278`
  - body: With a stable `sessionId`, the cache key stays fixed and this hash only looks at `tools` and `instructions`. pi-ai serializes `context.systemPrompt` as leading `system`/`developer` items in `input`, so a prompt/profile change can reuse `state.responseId`; the continuation path then trims the request and drops the updated prompt. Include those input prompt items in the invalidation hash or clear the response id before trimming.
  - confidence: 0.88
- **[P2] Retry without reusing emitted stream output:** `extensions/volcengine/cache.ts:354`
  - body: If the stale `previous_response_id` error arrives after any start or partial events, those events have already been pushed to `output`. Recursing into `pipeStream(output, false)` appends the retry attempt to the same stream, so consumers can see mixed failed-attempt and retry output. Buffer the first attempt until the retry decision is known or create a fresh stream for the retry.
  - confidence: 0.8
