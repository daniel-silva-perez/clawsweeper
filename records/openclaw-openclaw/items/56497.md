## Summary

Keep open. Current main still lacks Volcengine Ark Responses session caching, and the PR is provider-local in the right area, but the proposed patch still has blocking correctness issues before merge.

## What This Changes

The PR adds a Volcengine stream wrapper, unit tests, and docs for opt-in Ark Responses session caching with `previous_response_id`, `caching`, `store`, `expire_at`, and optional thinking settings.

## Best Possible Solution

Keep the implementation provider-local, cover both `volcengine` and `volcengine-plan`, invalidate cache state when system/developer input changes, retry stale response IDs without mixing output streams, preserve current Volcengine TTS docs during rebase, and validate the focused plugin tests plus the changed gate.

## Reproduction Assessment

Yes. A fake `openai-responses` stream can emit a partial event before a stale-response error to show mixed retry output, and a same-session payload with a changed `context.systemPrompt` follows from pi-ai's input serialization contract to show stale prompt reuse.

## Solution Assessment

No, not as currently written. The provider-local wrapper is the right boundary, but the current implementation misses the coding provider hook path and needs stricter cache invalidation and retry isolation before it is the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Register the wrapper for volcengine-plan:** `extensions/volcengine/index.ts:81`
  - body: `wrapStreamFn` is attached only to the `volcengine` provider entry, while hook lookup matches `provider.id`, `aliases`, or `hookAliases`. Runs using the default `volcengine-plan/*` provider therefore never reach this cache wrapper even with `params.cache.enable`, so the feature silently does not work on the coding surface. Add a hook alias or equivalent registration and cover both provider ids in tests.
  - confidence: 0.86
- **[P1] Invalidate cache when system input changes:** `extensions/volcengine/cache.ts:277-278`
  - body: The cache hash only includes `target.tools` and `target.instructions`, but pi-ai places `context.systemPrompt` into the Responses `input` as a leading system/developer item. With the same session id, a prompt/profile change can reuse `state.responseId`; continuation mode then trims `input` and drops the updated prompt, so the next request runs under stale instructions.
  - confidence: 0.88
- **[P2] Retry without reusing emitted stream output:** `extensions/volcengine/cache.ts:354`
  - body: When a stale `previous_response_id` error triggers the retry, `pipeStream(output, false)` appends the retry attempt to the same stream object. If the failed attempt emitted any start or partial events before the error, consumers see failed-attempt output followed by retry output. Buffer until the retry decision is known or retry through a fresh output path.
  - confidence: 0.82
