## Summary

Keep open. Current main does not contain Volcengine session caching, and the provider-local boundary is reasonable, but the proposed branch still has blocking correctness gaps that would make the feature unreliable on the default coding provider and after prompt changes.

## What This Changes

The PR adds an opt-in Volcengine Ark Responses session-cache wrapper, tests, and docs for `params.cache` with `previous_response_id`, `caching`, `store`, `expire_at`, and optional thinking settings.

## Best Possible Solution

Keep the feature provider-local, but update the branch so session caching covers both Volcengine provider IDs, invalidates on all prompt/tool state that can affect Responses continuation, retries stale response IDs without exposing failed-attempt output, and documents defaults that match the implementation.

## Reproduction Assessment

Yes. A unit-style fake `openai-responses` stream can emit a partial event before a stale-response error to demonstrate mixed retry output, and a same-session run with changed `context.systemPrompt` follows pi-ai's input serialization path to show stale prompt reuse.

## Solution Assessment

No. The provider-local wrapper is the right direction, but the current patch is not the best mergeable solution until it covers `volcengine-plan`, includes system/developer input in invalidation, isolates retry output, and aligns docs with code defaults.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Register the cache hook for the coding provider:** `extensions/volcengine/index.ts:81`
  - body: `wrapStreamFn` is attached only to the `volcengine` plugin entry, while hook lookup matches the runtime provider id against `id`, `aliases`, or `hookAliases`. The default Volcengine model path is `volcengine-plan/*`, so cache-enabled coding requests never reach this wrapper. Add a hook alias or equivalent registration and cover both provider ids in tests.
  - confidence: 0.87
- **[P1] Include system input in cache invalidation:** `extensions/volcengine/cache.ts:277-278`
  - body: The cache hash only includes `target.tools` and `target.instructions`, but pi-ai serializes `context.systemPrompt` into Responses `input` as a leading system/developer item. With the same session id, a prompt change can reuse `state.responseId`; continuation mode then trims `input` and drops the updated prompt, so the next request runs under stale instructions.
  - confidence: 0.88
- **[P2] Retry stale response IDs through fresh output:** `extensions/volcengine/cache.ts:354`
  - body: When a stale `previous_response_id` error triggers retry, `pipeStream(output, false)` appends the retry attempt to the same event stream. If the failed attempt emitted start or partial events before the error, consumers see failed-attempt output followed by retry output. Buffer until the retry decision is known or retry through a fresh output path.
  - confidence: 0.82
- **[P3] Align thinking defaults between docs and code:** `docs/providers/volcengine.md:72-73`
  - body: The docs say omitted `cache.thinking` falls back to `true`, but `resolveCacheConfig` leaves it `undefined` and the wrapper only sends `target.thinking` when the setting is explicitly boolean. Users following the docs will not get the documented default; either change the code/tests or change the docs to match the actual contract.
  - confidence: 0.79
