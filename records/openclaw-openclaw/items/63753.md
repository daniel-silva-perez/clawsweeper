## Summary

Keep open. Current main still has the macOS Talk routing gap: it fetches `talk.config`, but unsupported/non-local providers fall through to local system speech instead of using `talk.speak`, and there is already an open unmerged closing PR for this exact issue.

## What This Changes

The issue requests that macOS Talk mode use Gateway `talk.speak` for configured non-local TTS providers, respect `talk.providers`, expose provider settings, and fall back to local TTS only when gateway speech is unavailable.

## Best Possible Solution

Review and land #74645 or an equivalent macOS runtime fix that makes every non-local/non-MLX/non-system Talk provider use Gateway `talk.speak` as the primary playback path, keeps explicit local paths local, and adds regression coverage for an OpenAI/custom provider config. Treat the provider-selection settings UI as a follow-up only if maintainers still want that surface after the routing bug is fixed.

## Reproduction Assessment

Yes. With a Gateway config such as `talk.provider: "openai"`, current macOS code parses the active provider, logs it as unsupported, and `playbackPlan` sends it to `.systemVoiceOnly`, so no `talk.speak` call is expected.

## Solution Assessment

Yes for the runtime bug: the narrow maintainable fix is in the macOS Talk runtime, routing provider-agnostic TTS through `talk.speak` while preserving local `mlx` and `system` behavior. The broader settings UI request is not required to prove or fix the core playback regression.
