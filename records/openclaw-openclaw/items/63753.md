## Summary

Keep this issue open. Current main still has the macOS Talk routing gap: the macOS app fetches Gateway Talk config, but playback planning only treats `elevenlabs`, `mlx`, and `system` as executable paths, with unknown providers such as `openai` falling through to local system voice. Gateway `talk.speak` and OpenAI speech support exist, and Android already uses `talk.speak` as the primary Talk playback path, so this remains a real macOS parity/product bug rather than cleanup noise.

## Best Possible Solution

Implement macOS Talk playback so Gateway `talk.speak` is the primary path for every configured non-local provider such as OpenAI, Microsoft, Google, or extension speech providers. Keep explicit local paths for `mlx` and `system`; if direct ElevenLabs streaming remains for latency, make it an optimized ElevenLabs path with Gateway `talk.speak` fallback before AVSpeechSynthesizer. Add macOS regression tests for an `openai` or custom provider routing to `talk.speak`, and add/clarify Talk mode settings or docs for provider selection.
