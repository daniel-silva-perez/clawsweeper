## Summary

Keep open. The report is concrete and current source still has an embedded prompt-discovery path that can synchronously read channel credentials from the config it is handed; existing caller-side SecretRef resolution is helpful, but it is not enough evidence to close this crash report without an end-to-end Telegram/Discord regression proving raw SecretRefs cannot reach those helpers.

## What This Changes

Embedded Telegram and Discord reply runs should not crash during prompt/action discovery when channel credentials are configured with non-env SecretRefs.

## Best Possible Solution

Harden the embedded reply prompt-discovery boundary so Telegram/Discord prompt capabilities, reactions, and action discovery either receive the resolved active runtime snapshot or treat unresolved discovery credentials as unavailable without aborting the agent run, then cover that behavior with a focused SecretRef regression.

## Reproduction Assessment

Yes. The issue gives a concrete operator path and sanitized logs; source inspection confirms that if raw non-env SecretRefs reach Telegram/Discord discovery helpers, those helpers can throw before model output.

## Solution Assessment

No. The current caller-side resolver is a useful part of the solution, but the safer maintainable fix is to make the embedded discovery boundary itself runtime-snapshot-aware or failure-tolerant for credential-only discovery.
