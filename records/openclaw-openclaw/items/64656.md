## Summary

Keep this PR open for maintainer review rather than cleanup-close: current main does not implement the proposed `failoverState` or `memoryConversationKey` surfaces, and the branch still contains correctness issues in the context-engine identity split and legacy reset behavior. The failover reset portion overlaps shipped main behavior, but the PR also changes broader session, context-engine, and overflow semantics that are not safely superseded by the related merged PRs.

## What This Changes

This PR changes session fallback persistence to a structured `SessionEntry.failoverState`, adds a `memoryConversationKey` for LCM/context-engine identity across `/new` and `/reset`, and adds structured overflow diagnostics plus a compaction/truncation circuit breaker.

## Best Possible Solution

Rebase and narrow this branch under maintainer review: decide whether OpenClaw should keep the current source-field failover contract or introduce a versioned `failoverState`, implement any LCM conversation-key split consistently across every context-engine lifecycle call while preserving legacy `/reset` continuity, and keep only overflow diagnostics or breaker behavior that improves the existing bounded recovery path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Use one context-engine key for every lifecycle call:** `src/agents/pi-embedded-runner/run.ts:876`
  - body: The PR switches timeout/overflow compaction to `contextEngineSessionKey`, but the attempt lifecycle still calls bootstrap, assemble, afterTurn, ingest, and maintenance without passing `memoryConversationKey`. Once a memory key exists, compaction can operate on a different conversation than the one being assembled and finalized, so recovery may compact the wrong thread and leave the active overflow unchanged.
  - confidence: 0.9
- **[P2] Preserve legacy identity on first reset:** `src/gateway/session-reset-service.ts:604-606`
  - body: For existing sessions that do not yet have `memoryConversationKey`, `/reset` falls back to `randomUUID()`. That rotates legacy sessions away from their existing LCM `sessionKey` identity even though reset is supposed to preserve continuity; seed the key from the existing session identity instead of generating a new one for this path.
  - confidence: 0.88
- **[P2] Clear stale failover state after primary recovery:** `src/auto-reply/reply/agent-runner-execution.ts:126-138`
  - body: The new `failoverState` is displayed as an active temporary failover, but model selection no longer reads it on the next turn and the PR only clears it on reset or explicit user model changes. After a later turn succeeds on the primary model, `/status` can still report the old fallback as active; clear or recompute this state when a normal primary run succeeds.
  - confidence: 0.78
