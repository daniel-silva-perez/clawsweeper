## Summary

Keep open: current main does not implement the branch's failoverState, memoryConversationKey, or overflow diagnostic design, and the PR still has correctness issues around context-engine key routing, legacy reset continuity, and stale failover status. Because it touches core session semantics and overflow recovery, it needs maintainer design review rather than an autonomous replacement fix.

## What This Changes

This PR changes runtime fallback persistence to SessionEntry.failoverState, adds memoryConversationKey for LCM/context-engine identity, and adds overflow diagnostics plus a compaction/truncation circuit breaker.

## Best Possible Solution

Rebase and narrow this under maintainer review: choose whether OpenClaw should keep the current auto-sourced override contract or introduce a versioned failoverState, thread any LCM identity split through every context-engine lifecycle call while preserving legacy /reset continuity, and keep only overflow diagnostics or breaker behavior that improves the existing bounded recovery path with targeted regression coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Use one context-engine key for every lifecycle call:** `src/agents/pi-embedded-runner/run.ts:876`
  - body: The PR switches overflow compaction to contextEngineSessionKey, but it does not update the attempt lifecycle call sites in attempt.ts to pass memoryConversationKey into bootstrap, assemble, or finalize. After /new or /reset, compaction can run against the memory conversation while assembly/finalization still use the runtime session key, so recovery may compact the wrong thread.
  - confidence: 0.9
- **[P2] Preserve legacy identity on first reset:** `src/gateway/session-reset-service.ts:604-606`
  - body: For existing sessions without memoryConversationKey, /reset falls back to randomUUID(). That rotates legacy sessions away from their existing LCM sessionKey identity even though reset is supposed to preserve continuity; seed the key from the current session identity instead.
  - confidence: 0.88
- **[P2] Clear stale failover state after primary recovery:** `src/auto-reply/status.ts:810-812`
  - body: failoverState is displayed as an active temporary failover, but the runner only clears it on reset, explicit user model changes, or rollback. A later turn that succeeds on the primary model can leave stale active state, so /status may keep reporting the old fallback as active.
  - confidence: 0.78
