## Summary

Keep this PR open. Current main still lacks the Feishu outbound success logging the PR targets: reply-dispatcher send paths still await text/card/media sends without logging the returned Feishu message IDs. The PR is not obsolete, but it needs revision before merge because the proposed log mode is derived from the requested reply target, while current send helpers can fall back from reply to top-level send for withdrawn/not-found non-thread replies.

## Best Possible Solution

Keep the PR open and ask for a small rebase/revision: add success logging on the current reply-dispatcher text/card/media paths, preserve direct-send and reply/thread tests, and make the logged mode reflect the actual send path. The cleanest path is likely to have Feishu send helpers return or otherwise expose whether delivery used reply or fallback direct send, then have the dispatcher log that result instead of deriving mode only from replyToMessageId.

## Risks / Open Questions

- Merging the PR as-is could produce misleading `outbound ... reply ok` logs for text/card sends that actually fell back to top-level direct delivery after a withdrawn or missing reply target.
- The proposed success logs include Feishu chat IDs, reply target IDs, and message IDs. That is useful operational telemetry, but reviewers should confirm it is acceptable for normal runtime logs.
