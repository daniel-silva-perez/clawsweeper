## Summary

Keep open. Current main partially addresses custom-provider modality metadata by inferring or asking about image input, and the related runtime catalog issue was handled separately, but the onboarding flow still does not prompt for context window or max output tokens and only exposes text/image input controls.

## What This Changes

Add optional onboarding prompts so OpenAI-compatible custom provider models can set context window, maximum output tokens, and supported modalities without manual config editing.

## Best Possible Solution

Implement this in the custom provider onboarding/config layer: collect optional model-level contextWindow and maxTokens values, seed them from existing config or defaults, and expose a clear modality picker that writes supported `input` metadata without changing unrelated runtime routing. Keep docs and targeted tests aligned, and preserve existing custom model values on re-onboard unless the user explicitly changes them.

## Reproduction Assessment

Yes. A code-path inspection against current main is enough to reproduce the gap: the custom provider flow reaches only the image-input prompt and then writes hard-coded/default contextWindow and maxTokens values.

## Solution Assessment

Yes, with scope control. The maintainable path is to extend `promptCustomApiConfig` and `applyCustomApiConfig` to write the already-supported model metadata fields, while treating audio/video as a verified/documented metadata choice rather than a hidden runtime behavior change.
