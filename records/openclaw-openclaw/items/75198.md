## Summary

Keep open. The linked bug is valid and the shared model-selection/session-patch boundary is the right place to fix it, but the PR has a correctness gap: the new provider-qualified alias branch normalizes the provider prefix before comparing it to alias targets, which conflicts with existing exact configured-provider-id behavior.

## What This Changes

The PR canonicalizes one-slash provider-qualified model aliases in shared model selection, adds model-selection and sessions-patch regression tests, and adds an Unreleased changelog fix entry for the linked session-switch alias bug.

## Best Possible Solution

Keep the generic shared resolver approach, but match provider-qualified aliases using the same provider identity semantics used to build the alias target, or index aliases by provider plus alias. Add a regression for an exact configured provider id such as modelstudio/Alias and then land the focused fix with the current sessions.patch coverage.

## Reproduction Assessment

Yes. The linked issue provides concrete config/log symptoms for lmstudio-moe/MoE, and source inspection shows the current-main path where that value is parsed as provider lmstudio-moe with model MoE instead of resolving the MoE alias.

## Solution Assessment

No, not as written. The shared resolver boundary is the right solution, but the comparison should preserve exact configured provider ids before legacy normalization so provider-qualified aliases work for all supported provider-id shapes.
