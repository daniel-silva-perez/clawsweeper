## Summary

Keep this PR open. Current main still does not implement the proposed Control UI API-key/provider panel, and the proposed branch is not merge-ready because it mixes the dashboard feature with FireClaw/WhatsApp/Docker publishing changes and introduces concrete correctness plus supply-chain issues.

## What This Changes

The PR adds a Control UI multi-provider API-key panel with curated model catalogs and also changes WhatsApp topbar setup, FireClaw-specific UI/gateway files, Docker publishing workflow, docs, scripts, and assorted runtime/tests.

## Best Possible Solution

Keep the PR open for maintainer review, then split the provider/API-key Control UI work from fork-specific FireClaw/WhatsApp/Docker publishing changes. The provider panel should build on the existing configured models.list/auth visibility contract, refresh config hashes before writes, surface session/model refresh failures, and add focused UI/gateway tests before merge consideration.

## Reproduction Assessment

Yes for the patch regressions: the model-filter bug is reproducible by having one provider available through env/auth-profile credentials and then configuring another provider through the new panel, after which the PR's filter drops the auth-backed provider from the picker. The session-switch and Docker-tag issues are also directly reproducible from the PR head code paths without needing a live provider key.

## Solution Assessment

No. The PR's current approach is not the best way to solve the requested API-key setup because it bypasses the existing Gateway model visibility semantics, suppresses important write failures, and bundles unrelated fork/publishing changes; a narrower Control UI feature on top of current Gateway contracts is safer.

Full review comments:

- **[P1] Guard fireclaw-latest publishing to trusted refs:** `.github/workflows/docker-fireclaw.yml:47`
  - body: This workflow runs on both main and the feature branch, but `fireclaw-latest` is emitted unconditionally while the job has `packages: write`. Any push to the feature branch can overwrite the shared GHCR tag with an unmerged image, so restrict that tag to the default branch or remove this workflow from the feature PR.
  - confidence: 0.94
- **[P1] Preserve auth-backed models in the picker:** `ui/src/ui/fireclaw/api-key-ui.ts:21`
  - body: Filtering `chatModelCatalog` to only providers with `apiKey`/`auth` in config drops models that current `models.list` exposes through env vars, auth profiles, or custom local auth. Once a user saves any provider key through this panel, other valid providers can disappear from the chat model picker even though inference still works.
  - confidence: 0.9
- **[P2] Surface failed session model switches after saving:** `ui/src/ui/controllers/api-key.ts:134-137`
  - body: After writing config, this immediately calls `sessions.patch` and ignores any error. Because Gateway config reload is debounced and `sessions.patch` validates against the runtime model catalog, the switch can fail while the UI still reports success and leaves the active session on the old model.
  - confidence: 0.88
- **[P2] Refresh stale config hashes before retrying saves:** `ui/src/ui/controllers/api-key.ts:25-27`
  - body: `ensureConfigLoaded` returns as soon as any snapshot exists, so after another client or CLI changes config, this panel keeps retrying writes with the same stale `baseHash`. A hash-mismatch failure should force a fresh `config.get` before retrying instead of trapping the user in repeated failures.
  - confidence: 0.84
- **[P3] Fix mojibake in the search shortcut label:** `ui/src/ui/app-render.ts:1259-1263`
  - body: The branch rewrites the command-palette title and keyboard hint with mojibake (`â€¦`, `âŒ˜`) instead of the existing ellipsis and command symbol. This is a visible UI regression in the top bar and should be restored before merge.
