## Summary

Keep this PR open for maintainer review. Current main does not implement the PR's stated dashboard API-key/provider-catalog panel, and the PR still carries substantial unmerged product work plus unrelated FireClaw/WhatsApp and CI publishing changes that should not be swept closed as implemented.

## Best Possible Solution

Keep this PR open for explicit maintainer review. The best path is to split the dashboard API-key/provider-catalog work from the FireClaw/WhatsApp/CI publishing changes, preserve existing provider and agent model config during patches, use fresh config hashes, handle SecretRef/OAuth/env-backed providers, and add focused UI/gateway tests. The Docker publishing workflow and fork-branded code should be removed or kept in the fork unless a maintainer explicitly wants that surface in OpenClaw.

## Risks / Open Questions

- The PR mixes the stated dashboard API-key feature with FireClaw fork code, WhatsApp UI/settings behavior, gateway web-login fallback logic, generated/config/docs churn, release-skill edits, and Docker publishing workflow changes.
- The added Docker workflow would grant `packages: write` and publish `fireclaw-latest` from the feature branch, which is a concrete supply-chain/release risk if merged as-is.
- Review comments identify unresolved functional risks in the proposed API-key flow: SecretRef/OAuth/env-backed providers may be hidden, config patches may overwrite provider/model fallback settings, stale config hashes can cause repeated failures, and session/model refresh errors are suppressed.
