## Summary

Keep this issue open. Current main still implements foreground exec approval as an immediate `approval-pending` tool result with detached gateway/node follow-up execution, and the docs/tests still encode continuation rather than a hard turn barrier.

## What This Changes

The issue requests that foreground exec approval requests pause the agent turn until the approval is approved, denied, or times out.

## Best Possible Solution

Make foreground approval-required exec calls a core turn barrier until allow, deny, timeout, or unavailable-route resolution is known; preserve explicit background behavior and native approval delivery, handle heartbeat/automation deterministically, and update the docs plus approval regression tests that currently encode immediate continuation.

## Reproduction Assessment

Yes. Source and test inspection on current main provides a high-confidence reproduction path: a foreground approval-required gateway or node exec registers approval, starts a detached wait, returns `approval-pending`, and current tests allow multiple pending requests from sequential calls.

## Solution Assessment

Yes, the right solution belongs in the core agent/exec approval flow rather than a plugin or docs-only change. The safest path is to reuse the existing approval registration and `exec.approval.waitDecision` primitives while changing foreground semantics behind maintainer-reviewed timeout/background rules.
