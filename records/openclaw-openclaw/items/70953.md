## Summary

Keep this issue open. Current main still documents and implements foreground exec approval as an immediate `approval-pending` tool result, with gateway/node approval waits running detached and follow-up results emitted later. The heartbeat context remains relevant because heartbeat is a recognized run kind, while the current headless no-route fallback only treats `cron` as headless for inline deterministic denial.

## Best Possible Solution

Keep this issue open for maintainer-scoped design and implementation. The best fix is a core agent/exec change that makes foreground approval-required exec calls a hard turn barrier until allow, deny, timeout, or unavailable-route resolution is known, while preserving explicit background behavior, native approval cards, and deterministic heartbeat/automation failure handling. Update the current docs and regression tests that encode immediate `approval-pending` continuation.
