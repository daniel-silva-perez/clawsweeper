## Summary

Keep this issue open. Current main still treats foreground exec approval as an immediate model-visible `approval-pending` result with detached gateway/node follow-up execution, and the docs plus regression tests still encode continuation rather than a hard turn barrier.

## What This Changes

The issue requests that foreground exec approval requests pause the agent turn until approval, denial, timeout, or deterministic failure is known.

## Best Possible Solution

Make foreground approval-required exec calls a core turn barrier until allow, deny, timeout, unavailable-route, or registration failure is known; preserve explicit background behavior and native approval delivery, handle heartbeat/automation deterministically, and update the approval docs plus regression tests that currently encode immediate continuation.

## Reproduction Assessment

Yes. Source and test inspection on current main gives a high-confidence reproduction path: a foreground approval-required gateway or node exec registers approval, starts a detached wait, returns `approval-pending`, and current tests allow multiple pending requests from sequential calls.

## Solution Assessment

Yes at the product-direction level: this belongs in the core agent/exec approval flow, not a plugin or docs-only change. The exact patch should be maintainer-reviewed because the narrowest maintainable fix must reuse existing registration and `exec.approval.waitDecision` primitives without weakening timeout, background, or native approval semantics.
