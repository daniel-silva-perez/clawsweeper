## Summary

Keep this open. Current main still surfaces foreground exec approvals as an `approval-pending` tool result while a detached waiter handles the eventual approve/deny/timeout outcome, so the reported turn-control problem remains present.

## What This Changes

The issue asks for foreground exec approval requests to pause or otherwise hard-stop the agent turn until the approval is approved, denied, or times out, while keeping background/headless behavior separate.

## Best Possible Solution

Define one core exec-approval turn-control contract: foreground interactive exec approvals should either await the approve/deny/timeout result before the model can issue more tool calls, or terminate the current turn in a deterministic pending state that cannot trigger more commands until resolution. Keep explicit background/headless automation behavior separate, preserve approval audit binding, and update exec docs plus regression tests for gateway, node, heartbeat, timeout, and denial paths.

## Reproduction Assessment

Yes. A source-level reproduction is clear on current main: run a foreground gateway or node exec that requires approval, and the tool returns `details.status = "approval-pending"` while a detached waiter handles the later decision.

## Solution Assessment

Unclear. Awaiting the final decision matches the requested behavior, but current docs and code intentionally use an async pending/follow-up model, so maintainers should first choose whether to block the tool call or hard-stop the turn after returning pending.
