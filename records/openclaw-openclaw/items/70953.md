## Summary

Keep open. Current main still treats approval-required foreground exec as a soft pending result: it registers the approval, waits for the decision in detached follow-up work, and returns `approval-pending` to the active agent turn. The related #69386 remains open for broader denial-loop and queue-flood symptoms, but this issue has a distinct, focused control-flow request that is not implemented yet.

## What This Changes

This issue asks OpenClaw to pause foreground exec tool calls until the user resolves an approval request instead of returning `approval-pending` and letting the agent continue the same turn.

## Best Possible Solution

Implement a foreground approval gate in the gateway and node host exec paths: after registration, foreground calls should await approve, deny, or timeout and then return the final command or denial result in the same tool call. Keep explicit background execution and headless automation deterministic, and update docs/tests to describe the new boundary.

## Reproduction Assessment

Yes. A source-level reproduction is clear on current main: run a foreground gateway or node exec that requires approval, and the implementation returns `details.status: "approval-pending"` after registration while decision handling continues asynchronously.

## Solution Assessment

Yes directionally, with a narrower implementation boundary. The maintainable fix is to await approval only for foreground approval-gated exec calls, with explicit timeout/background/headless exceptions, rather than changing approval policy or adding broad loop-detection behavior.
