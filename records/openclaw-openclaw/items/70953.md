## Summary

Keep this open. Current main still returns foreground gateway/node exec approvals as model-visible `approval-pending` tool results while a detached waiter handles the approval decision and follow-up later, so the requested turn-pause behavior is not implemented.

## What This Changes

The issue requests that foreground exec approval requests pause the active agent turn until the user approves, denies, or the request times out.

## Best Possible Solution

Define one core exec-approval turn-control contract: foreground interactive exec approvals should either await the final approve/deny/timeout result before more model tool calls can run, or hard-stop the turn in a pending state that resumes only after resolution. Keep cron/headless/background behavior separate and add regression coverage for gateway, node, heartbeat, denial, timeout, and follow-up delivery paths.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: run a foreground gateway or node exec that requires approval, and current main returns `details.status = "approval-pending"` while a detached continuation waits for the final decision and sends follow-up output later.

## Solution Assessment

Unclear. Awaiting the approval decision matches the issue request, but current docs and tests intentionally encode an async pending/follow-up model, so maintainers should first choose whether the safest fix is blocking the tool call or hard-stopping the turn after returning pending.
