## Summary

Keep this issue open. Current main still treats foreground exec approval as an immediate `approval-pending` tool result with detached follow-up execution after approval resolution, not as a hard pause of the agent turn. The heartbeat comment remains relevant: heartbeat runs are not classified as the existing headless/cron fallback path, and per-call stricter `security`/`ask` arguments can still tighten a permissive local policy.

## Best Possible Solution

Keep this issue open for maintainer-scoped design and implementation. The best fix is a core agent/exec change that makes foreground exec approval a hard turn barrier until approval is allowed, denied, or timed out, while preserving explicit background behavior and making heartbeat/automation deterministic. The implementation should reuse the existing approval registration and `exec.approval.waitDecision` primitives where possible, update docs, and add regression coverage that prevents multiple unresolved foreground approval requests from the same turn.

## Risks / Open Questions

- Changing this behavior affects a security-sensitive approval boundary for host command execution, so it needs maintainer design before an autonomous fix PR.
- The right implementation boundary is not obvious: blocking inside the exec tool, suspending the agent loop, or adding an explicit resumable approval state each has different transcript, timeout, and delivery semantics.
- Heartbeat, cron, interactive chat, and explicitly backgrounded exec runs need separate rules so automation does not hang indefinitely or create approval noise.
- Existing tests currently preserve registration-only waiting and multiple pending approvals, so a fix will intentionally change current behavior and require focused regression updates.
