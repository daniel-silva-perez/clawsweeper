## Summary

Keep this open. Current main still treats foreground gateway/node exec approvals as model-visible `approval-pending` tool results while detached continuations wait for approve/deny/timeout and emit follow-up turns later, so the requested pause or hard-stop turn behavior is not implemented.

## What This Changes

The issue requests foreground exec approvals to pause or hard-stop the agent turn until the user approves, denies, or times out the request.

## Best Possible Solution

Define one core exec-approval turn-control contract: foreground interactive approvals should either await the final approve/deny/timeout result before more model tool calls can run, or hard-stop the turn in a pending state that resumes only after resolution. Keep cron/headless/background behavior separate and add regression coverage across gateway, node, heartbeat, denial, timeout, and follow-up delivery paths.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: run a foreground gateway or node exec that requires approval, and current main returns `details.status = "approval-pending"` while detached continuations wait for the final decision and send follow-up output later.

## Solution Assessment

Unclear. Awaiting the approval decision matches the issue request, but current docs and tests intentionally encode an async pending/follow-up model, so maintainers should choose whether the safest fix is blocking the tool call or hard-stopping/resuming the turn after approval resolution.
