## Summary

Keep open. Current main still preserves the reported Moonshot/Kimi payload inconsistency: enabled Moonshot thinking is converted to `thinking.type: "disabled"` when `tool_choice` is pinned, while the OpenAI-compatible transport can still emit `reasoning_effort`; current tests also lock in that behavior despite docs saying incompatible Moonshot tool choices are normalized to `auto`.

## What This Changes

The issue requests fixing Moonshot-compatible reasoning requests that can fail with HTTP 400 when `reasoning_effort` remains enabled but the Moonshot thinking wrapper disables `thinking` for pinned tool choices.

## Best Possible Solution

Update the Moonshot thinking wrapper so enabled thinking remains coherent by normalizing pinned or otherwise incompatible `tool_choice` values to `auto`, while preserving explicit `/think off` and explicitly configured disabled thinking. Add regression coverage for the direct wrapper path and provider stream-family hook so `reasoning_effort` cannot coexist with `thinking.type: "disabled"` on this path.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a Moonshot-compatible payload with `thinkingLevel: "low"`, pinned `tool_choice`, and independently emitted `reasoning_effort`; current wrapper code and tests show the payload becomes `thinking.type: "disabled"` while the reasoning effort path remains possible.

## Solution Assessment

No for the literal suggested replacement of `"disabled"` with `"enabled"`, because that could still leave a Moonshot-incompatible pinned tool choice. The narrower maintainable fix is the docs-aligned `tool_choice: "auto"` fallback when thinking is enabled.
