## Summary

Keep open. Current main still contains the Moonshot/Kimi payload inconsistency: enabled thinking with a pinned `tool_choice` is downgraded to `thinking.type: "disabled"`, while an already-built request can still carry `reasoning_effort`; the docs say incompatible Moonshot tool choices should be normalized to `auto`, and the related fix PRs were closed unmerged.

## What This Changes

The issue requests a fix for Moonshot-compatible reasoning requests so pinned tool choices do not produce an invalid `reasoning_effort` plus disabled-thinking payload.

## Best Possible Solution

Update the Moonshot thinking wrapper so enabled thinking stays coherent by normalizing pinned or otherwise incompatible `tool_choice` values to `auto`, while preserving explicit `/think off` and explicitly disabled thinking. Add regression coverage for the direct wrapper path and the provider stream-family hook so `reasoning_effort` cannot coexist with `thinking.type: "disabled"` on this path.

## Reproduction Assessment

Yes. A focused reproduction is a Moonshot-compatible payload with `thinkingLevel: "low"`, pinned `tool_choice`, and an existing or independently emitted `reasoning_effort`; current source and tests show the wrapper changes thinking to disabled while leaving the pinned choice intact.

## Solution Assessment

No for the literal issue suggestion to replace `"disabled"` with `"enabled"`, because that still leaves a Moonshot-incompatible pinned tool choice. The docs-aligned fix is to fall back to `tool_choice: "auto"` when thinking is enabled.
