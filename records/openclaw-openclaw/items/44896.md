## Summary

Keep open. Current main still has the reported Moonshot/Kimi payload conflict: the Moonshot thinking wrapper disables `thinking.type` for pinned tool choices without clearing independently emitted reasoning fields, while the OpenAI-compatible transport can still add `reasoning_effort` for reasoning-capable models. Existing tests preserve the conflicting pinned-tool policy, the docs describe a different `tool_choice: auto` contract, and the related fix PRs are closed unmerged.

## What This Changes

The issue requests fixing Moonshot/Kimi payload normalization so reasoning-enabled requests with pinned `tool_choice` do not send `reasoning_effort` together with `thinking.type: disabled`.

## Best Possible Solution

Make the Moonshot/Kimi wrapper produce a single coherent contract for enabled thinking plus pinned tool choices. The best aligned path is to normalize pinned `tool_choice` values to `auto` when thinking is enabled, matching the docs and preserving user-requested reasoning; if maintainers choose to preserve pinned tool choice instead, the wrapper must delete all conflicting reasoning fields and update docs/tests to match.
