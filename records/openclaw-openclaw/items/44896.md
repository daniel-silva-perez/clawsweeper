## Summary

Keep open. Current main still has the Moonshot/Kimi payload path that can produce an incoherent request: pinned tool choices force `thinking.type` to `disabled`, while the OpenAI-compatible reasoning path can still emit `reasoning_effort` for reasoning-enabled models. The existing docs and closed unmerged fix PRs also show this is unresolved rather than obsolete.

## What This Changes

The issue requests fixing Moonshot-compatible requests where reasoning is enabled and a pinned tool choice causes `thinking.type=disabled`, producing an API-rejected `reasoning_effort` plus disabled-thinking combination.

## Best Possible Solution

Make the Moonshot/Kimi wrapper produce one coherent payload contract for pinned tool choices, then align tests, docs, and changelog with that contract. The conservative current-main-compatible fix is to clear `reasoning_effort`, `reasoning`, and related reasoning payload fields whenever the wrapper intentionally forces `thinking.type` to `disabled`; if maintainers decide reasoning should win, change pinned tool choices to `auto` and update the existing pinned-choice tests accordingly.
