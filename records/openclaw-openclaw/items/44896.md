## Summary

Current main still appears vulnerable to the reported Moonshot/Kimi payload conflict: pinned tool choices force `thinking.type` to `disabled`, while the OpenAI-compatible transport can still emit `reasoning_effort`. The issue discussion, current tests, docs drift, latest release tag, and closed unmerged fix PRs all point to unresolved work rather than an obsolete report.

## What This Changes

The issue requests fixing Moonshot-compatible requests where reasoning is enabled and a pinned tool choice produces an API-rejected `reasoning_effort` plus disabled-thinking payload.

## Best Possible Solution

Make the Moonshot/Kimi wrapper produce one coherent payload contract for pinned tool choices, then align tests, docs, and changelog with that contract. The conservative current-main-compatible path is to clear `reasoning_effort`, `reasoning`, and related reasoning payload fields whenever the wrapper intentionally forces `thinking.type` to `disabled`; if maintainers decide reasoning should win, switch pinned choices to `auto` and update the existing pinned-choice tests accordingly.

Current main sets `thinking.type` to `disabled` for pinned Moonshot `tool_choice` objects but can leave `reasoning_effort`/`reasoning` payload fields in place, causing providers to reject combinations like `reasoning_effort=low` plus disabled thinking. Implement the conservative contract that matches current merged tests: when `createMoonshotThinkingWrapper` intentionally forces `thinking.type = "disabled"` for a pinned tool choice, also remove conflicting reasoning fields such as `reasoning_effort`, `reasoning`, and any camelCase payload variants if present. Related context: #47261, #47780, and #47802 tried the alternate `tool_choice: auto` approach but were closed unmerged; #68816 recently codified the pinned-tool disables-thinking behavior, so preserve that policy unless a maintainer explicitly chooses the alternate contract.
