## Summary

Keep open. Current main still has the contradictory Moonshot-compatible payload path: pinned tool choices force `thinking.type` to `disabled`, while the upstream OpenAI-compatible transport can still add `reasoning_effort` for reasoning-capable models. The prior fix PRs were closed unmerged, and the latest release still contains the same behavior.

## What This Changes

The issue requests that Moonshot-compatible requests avoid sending `reasoning_effort=low` together with `thinking.type=disabled` when a pinned `tool_choice` is present.

## Best Possible Solution

Make the Moonshot/Kimi wrapper produce one coherent request contract for pinned tool choices and align source, regression tests, and docs with that contract. Given current main’s explicit pinned-tool preservation tests, the smallest compatible fix is likely to clear `reasoning_effort` and any nested reasoning payload whenever the wrapper intentionally disables thinking; if maintainers want reasoning to win instead, switch pinned choices to `auto` and update the existing tests accordingly.
