## Summary

Keep open. Current main still has the Moonshot/Kimi payload inconsistency: enabled thinking with a pinned `tool_choice` is downgraded to `thinking.type: "disabled"`, while the OpenAI-compatible request path can still carry `reasoning_effort`; current tests also lock in that behavior despite docs saying incompatible Moonshot tool choices should become `auto`.

## What This Changes

The issue reports that Moonshot-compatible reasoning requests can fail with HTTP 400 when `reasoning_effort` remains enabled but the wrapper changes `thinking.type` to `disabled` for a pinned tool choice.

## Best Possible Solution

Update the Moonshot thinking wrapper so enabled thinking remains coherent by normalizing pinned or otherwise incompatible `tool_choice` values to `auto`, while preserving explicit `/think off` and explicitly configured disabled thinking. Add regression coverage for both the direct wrapper path and the provider stream-family hook so `reasoning_effort` cannot coexist with `thinking.type: "disabled"` on this path.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a Moonshot-compatible payload with `thinkingLevel: "low"`, pinned `tool_choice`, and an independently emitted `reasoning_effort`; current wrapper code and tests show the payload becomes `thinking.type: "disabled"` while the reasoning effort path remains possible.

## Solution Assessment

No for the literal suggested replacement of `"disabled"` with `"enabled"`, because that could still leave a Moonshot-incompatible pinned tool choice. The narrower maintainable fix is the docs-aligned `tool_choice: "auto"` fallback when thinking is enabled.
