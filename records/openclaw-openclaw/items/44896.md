## Summary

Keep open. Current main still contains the Moonshot/Kimi payload inconsistency: enabled thinking plus an incompatible pinned `tool_choice` is converted to `thinking.type: "disabled"`, while the OpenAI-compatible transport can independently emit `reasoning_effort` for reasoning-capable models. The public docs say incompatible Moonshot tool choices should be normalized to `auto`, and the related fix PRs were closed unmerged rather than shipped.

## What This Changes

The issue requests a fix for Moonshot/Kimi requests that can combine non-off reasoning effort with disabled native thinking after pinned tool-choice handling.

## Best Possible Solution

Update the Moonshot thinking wrapper so enabled thinking stays coherent with reasoning fields by normalizing pinned or otherwise incompatible `tool_choice` values to `auto`, while preserving explicit `/think off` and explicit disabled-thinking behavior. Add regression tests for both direct wrapper behavior and the provider stream-family hook so future changes cannot reintroduce `reasoning_effort` plus `thinking.type: disabled` on this path.

## Reproduction Assessment

Yes. A focused unit-level reproduction is a Moonshot/OpenAI-completions payload with `thinkingLevel: "low"`, a pinned `tool_choice` object, and an existing or independently emitted `reasoning_effort`; current source and tests show the wrapper changes thinking to disabled while leaving the pinned tool choice intact.

## Solution Assessment

No for the literal `"disabled"` to `"enabled"` replacement suggested in the issue, because that would still leave an incompatible pinned Moonshot `tool_choice`. The docs-aligned fix is to normalize the incompatible tool choice to `auto` when thinking is enabled, with regression coverage for explicit thinking-off behavior.
