## Summary

Current main still does not implement syntax highlighting for WebChat code blocks. The WebChat markdown renderer preserves code-block chrome, language labels, copy buttons, JSON collapse, sanitization, caching, and large-message fallbacks, but fenced and indented code content is still escaped directly into `<pre><code>` without token markup or highlighter classes.

## Best Possible Solution

Keep this issue open and implement a narrow Control UI/WebChat syntax-highlighting pass in the existing markdown renderer or a helper used by it. The implementation should preserve copy buttons, language labels, JSON collapse, cache behavior, large-message fallback, raw HTML escaping, and sanitized output, with focused tests for highlighted fenced code and regressions around existing code-block behavior.

## Risks / Open Questions

- A highlighting implementation must keep DOMPurify-compatible output and preserve raw HTML escaping in markdown content.
- Language auto-detection can add client-side cost and unstable output for large or language-less messages; explicit-language-first or bounded detection should be considered.
- Adding or promoting a syntax-highlighting dependency affects bundle size and supply-chain surface, so the dependency choice should be narrow and declared directly if used.
