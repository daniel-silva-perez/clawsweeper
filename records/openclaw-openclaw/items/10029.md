## Summary

Current main still does not implement syntax highlighting for WebChat code blocks. The Control UI markdown renderer preserves code-block chrome, language labels, copy buttons, JSON collapse, sanitization, caching, and large-message fallback, but fenced and indented code content is still escaped directly into `<pre><code>` without highlighted token markup or highlighter classes.

## Best Possible Solution

Keep this issue open and implement a narrow Control UI/WebChat syntax-highlighting pass in the existing markdown renderer or a helper used by it. The fix should preserve copy buttons, language labels, JSON collapse, cache behavior, raw HTML escaping, sanitized output, and oversized/plain-text fallback, with focused regression tests.

## Risks / Open Questions

- A highlighting implementation must keep DOMPurify-compatible output and preserve raw HTML escaping in markdown content.
- Language auto-detection can add client-side cost and unstable output for large or language-less messages; explicit-language-first or bounded detection should be considered.
- Adding or promoting a syntax-highlighting dependency affects bundle size and supply-chain surface, so the dependency choice should be narrow and declared directly if used.
