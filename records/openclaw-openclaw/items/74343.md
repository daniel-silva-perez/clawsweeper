## Summary

Keep open. The report describes a current Slack native slash-command multi-app routing problem, and current main still has no per-account native command name/prefix mapping or documented duplicate-command support boundary. The Slack handler resolves from the receiving Slack account/team/channel and drops mismatched app/team payloads, but it does not recover from Slack assigning an identical slash command to a different installed app.

## What This Changes

The issue requests fixing Slack native slash-command cross-routing when multiple Slack Apps in the same workspace register identical built-in command names.

## Best Possible Solution

Make the Slack native-command contract explicit: either document duplicate slash command names across multiple Slack Apps as unsupported and recommend unique per-app names, or add a per-account native command name/prefix mapping with regression coverage for account/channel-bound routing.

## Reproduction Assessment

Yes. The issue gives a high-confidence reproduction path using two Slack Apps in one workspace, duplicate native command names, and `commands.native: true`; current source inspection shows the exact Slack account/team/channel routing path involved, though this review did not run a live Slack workspace.

## Solution Assessment

Unclear. There is no proposed patch here, and the best fix needs maintainer product judgment between a documented unsupported boundary and a new per-account native command name/prefix mapping.
