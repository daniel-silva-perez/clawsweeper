## Summary

Keep open. Current main still lacks a Slack per-account native slash-command name/prefix mapping or duplicate-command ownership remap, and the Slack handler continues to resolve routing from the receiving Slack account plus team/channel rather than from an intended app ownership model. The related per-account prefix request remains open, so there is no merged canonical fix to close against.

## What This Changes

The issue requests correct Slack native slash-command routing when multiple Slack Apps in one workspace expose identical command names for different agents or channels.

## Best Possible Solution

Settle the Slack native-command contract, then either document duplicate command names across Slack Apps as unsupported with clear unique-name guidance, or add a typed per-account native command name/prefix mapping that is reflected in registration, dispatch, config schema/help, docs, and regression tests.

## Reproduction Assessment

Yes. The issue gives a concrete two Slack App setup with duplicate native command names, and current source inspection confirms the relevant Slack native command path has no duplicate-app ownership remap; live Slack verification was not run in this read-only sweep.

## Solution Assessment

Unclear. The requested behavior may not be maintainable with identical Slack command names if Slack owns command selection, so the safer path is a maintainer decision between documenting that limitation and implementing unique per-account native names/prefixes.
