## Summary

Keep open. Current main still does not provide a supported boundary for identical Slack native slash-command names across multiple Slack Apps: Slack command names are not namespaced, and OpenClaw still registers/dispatches native command names at the provider level while routing through the Slack account that received the payload. 

## What This Changes

The issue requests correct Slack native slash-command routing when multiple Slack Apps in one workspace register the same names such as `/model`, `/help`, or `/config`.

## Best Possible Solution

Maintainers should define the supported Slack multi-app native-command boundary. If identical names are unsupported, document the limitation and recommend unique per-app names; if multi-app native support is intended, implement account-level native command naming/prefix mapping across config validation, native spec generation, Slack registration guidance, dispatch lookup, and argument-menu callbacks, likely through the #38302 lane.

## Reproduction Assessment

Yes. The issue gives concrete Slack multi-app steps, and current main has a source-level reproduction path: native command names are provider-scoped while slash dispatch routes from the Slack account/app that receives Slack's payload.

## Solution Assessment

Unclear. There is no patch here, and directly supporting identical names conflicts with Slack's non-namespaced command behavior; the maintainable options are either documenting unique names as required or adding deliberate per-account native name mapping.
