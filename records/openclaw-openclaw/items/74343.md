## Summary

Keep open. Current main still has no implemented or documented support boundary for duplicate Slack native slash command names across multiple Slack Apps, and the Slack slash handler still routes from the app/account that receives Slack's payload. Slack's own docs say slash commands are not namespaced and duplicate names invoke the most recently installed command, so the remaining fix needs an explicit OpenClaw product choice rather than cleanup closure. ([docs.slack.dev](https://docs.slack.dev/interactivity/implementing-slash-commands))

## What This Changes

The issue reports Slack native slash commands cross-routing between agents when multiple Slack Apps in one workspace register the same command names.

## Best Possible Solution

Maintainers should decide the supported boundary for duplicate Slack native commands. If multi-app native support is intended, implement account-level native command naming/prefix mapping through config validation, native spec generation, Slack registration, dispatch lookup, and argument-menu callbacks; if identical names cannot be supported reliably, document that and recommend unique per-app command names.

## Reproduction Assessment

Yes. The issue provides concrete multi-app Slack steps, and current main has a source-level reproduction path: native command names are provider-scoped while dispatch routes from the Slack account/channel that receives Slack's payload.

## Solution Assessment

Unclear. There is no patch here, and the best path depends on maintainer product judgment between documenting duplicate identical Slack command names as unsupported or adding account-specific native command naming.
