## Summary

Keep open. Current main partially addresses the report with packaged Node compile-cache enablement, documented operator-managed `NODE_COMPILE_CACHE`, and `OPENCLAW_NO_RESPAWN`, but it still does not provide the requested first-class cache prewarming command, shipped prewarmed cache artifact, or postinstall cache priming; those remaining choices need maintainer product and security judgment.

## What This Changes

Request to add cold-start optimizations for container/server deployments by supporting prewarmed Node compile caches and a public way to skip the CLI self-respawn when operators manage the required environment.

## Best Possible Solution

Keep the issue open for a maintainer decision on the accepted product shape. The narrowest maintainable path appears to be an explicit, documented `openclaw warm-cache --output <dir>` image-build helper, while treating shipped cache artifacts and automatic postinstall prewarming as separate product/security decisions.

## Reproduction Assessment

Not applicable as a normal bug reproduction; this is a performance/product enhancement request. Source inspection on current main gives high-confidence verification that the respawn opt-out and runtime compile-cache support exist, while a first-class prewarm command or artifact does not.

## Solution Assessment

Unclear until maintainers choose the product boundary. A documented explicit prewarm command is likely safer than automatic postinstall priming or shipped cache artifacts, because it keeps install-time side effects and cache invalidation under operator control.
