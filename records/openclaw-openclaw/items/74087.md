## Summary

Keep open. Current main implements the public respawn escape hatch and operator-managed compile-cache tuning, but it still does not provide the requested first-class cache prewarming command, shipped prewarmed cache artifact, or postinstall cache priming path; those remaining options need maintainer product and security judgment.

## What This Changes

The issue requests a first-class way to prewarm and reuse Node compile caches for cold starts, plus a public opt-in to bypass CLI self-respawn in controlled container environments.

## Best Possible Solution

Keep the implemented OPENCLAW_NO_RESPAWN and packaged compile-cache scoping, then decide whether to add a narrow, explicit openclaw warm-cache --output helper with focused tests and VPS/container docs. Treat shipped cache artifacts and automatic postinstall prewarming as separate product/security decisions rather than bundling them into an automated cleanup fix.

## Reproduction Assessment

Not applicable as a conventional bug reproduction. The remaining issue is verifiable by source inspection: current main has compile-cache and no-respawn support, but no warm-cache command, shipped artifact, or postinstall cache priming implementation.

## Solution Assessment

Unclear until a maintainer chooses the accepted product shape. The narrowest maintainable path appears to be an explicit opt-in warm-cache command; automatic postinstall priming or shipped cache artifacts would carry broader security and portability tradeoffs.
