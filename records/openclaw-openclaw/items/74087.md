## Summary

Keep this issue open. Current main already has operator-managed compile-cache enablement and a public respawn opt-out, but it still does not provide the requested first-class cache prewarming command, shipped prewarmed cache artifact, or postinstall cache priming; those remaining choices need maintainer product and security judgment.

## What This Changes

Request to reduce container cold-start time by adding prewarmed Node compile-cache support and a supported way to bypass CLI self-respawn when operators provide the startup environment themselves.

## Best Possible Solution

Choose the accepted upstream shape first. A conservative endpoint would be a documented, explicit image-build helper such as openclaw warm-cache --output <dir> with focused tests and VPS/container docs, while treating shipped cache artifacts and automatic postinstall prewarming as separate decisions.

## Risks / Open Questions

- Automatic postinstall cache prewarming would add install-time work and code execution, so it should get explicit maintainer/security review before implementation.
- Prewarmed Node compile-cache artifacts are sensitive to Node version, architecture, package path, and update invalidation; stale cache behavior is already adjacent to #71348.
- A warm-cache helper that imports plugin entry points could run plugin module side effects unless the implementation constrains that path carefully.
- Respawn bypass docs must make operator responsibility clear because misconfiguration can skip OpenClaw's warning-suppression and CA-certificate bootstrap setup.
