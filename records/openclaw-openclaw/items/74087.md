## Summary

Keep open. Current main partially covers the request: packaged launchers enable a scoped Node compile cache, foreground gateway runs and `OPENCLAW_NO_RESPAWN=1` skip the CLI self-respawn path, and docs/doctor already recommend those knobs for small hosts. The remaining requested upstream surface, a first-class pre-warm command/artifact or postinstall cache priming flow, is not present and needs maintainer product/security judgment rather than cleanup closure.

## What This Changes

The issue requests a first-class pre-warmed Node compile-cache workflow and a documented opt-in path to bypass CLI self-respawn in controlled container environments.

## Best Possible Solution

Keep the existing env-based compile-cache and no-respawn behavior, then decide whether OpenClaw should add an explicit operator-run `openclaw warm-cache --output ...` command with docs and tests. Avoid automatic postinstall prewarming unless maintainers explicitly accept the install-time code-execution and cache-invalidation tradeoffs.

## Reproduction Assessment

Yes for the remaining repo gap: current main's command descriptors and repository search show no `warm-cache` command while source/docs show the existing env-based cache and no-respawn mechanisms. The exact wall-clock speedup remains externally reported and was not rebenchmarked in this read-only review.

## Solution Assessment

Unclear. The current env-based no-respawn and scoped compile-cache pieces are maintainable, but the remaining proposal has product and supply-chain choices; an explicit warm-cache command looks safer than automatic postinstall prewarming if maintainers accept the feature.
