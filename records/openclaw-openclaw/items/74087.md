## Summary

Keep this issue open. Current main covers the respawn escape hatch and operator-managed compile-cache tuning, but it still does not provide the requested first-class cache prewarming command, shipped prewarmed cache artifact, or postinstall cache priming; the remaining choices need maintainer product/security judgment.

## What This Changes

Request to reduce cold starts by adding first-class Node compile-cache prewarming or shipped cache support plus a public opt-in respawn skip for controlled container deployments.

## Best Possible Solution

Keep this issue open for a maintainer decision on the accepted product shape. The safest narrow path appears to be an explicit documented openclaw warm-cache --output <dir> image-build helper with focused unit tests and VPS/container docs, while treating shipped cache artifacts and automatic postinstall prewarming as separate security/product decisions. The existing OPENCLAW_NO_RESPAWN=1 env can remain the public respawn escape hatch unless maintainers want a CLI alias.

## Risks / Open Questions

- Automatic postinstall cache prewarming would expand install-time work/code execution and should get explicit maintainer/security review before implementation.
- Prewarmed Node compile-cache artifacts are sensitive to Node version, architecture, package path, and update invalidation; stale-cache behavior is already an adjacent concern in #71348.
- A respawn bypass must keep operator responsibility clear because it can skip OpenClaw bootstrap setup for warning suppression and CA certificate environment when misconfigured.
