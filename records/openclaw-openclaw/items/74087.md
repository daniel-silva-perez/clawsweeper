Keep open. Current main already covers the runtime compile-cache and no-respawn parts of the request, but it still lacks the requested first-class cache prewarm command/artifact or postinstall priming flow, and that remaining surface needs maintainer product and supply-chain judgment.

## What This Changes

Request to optimize cold starts by adding a pre-warmed Node compile-cache surface and an operator opt-in to bypass CLI self-respawn.

## Best Possible Solution

Keep the existing env-based compile-cache and no-respawn behavior, then have maintainers decide whether to add an explicit operator-run warm-cache command with tests and docs. Prefer an explicit command over automatic postinstall prewarming unless maintainers accept the install-time execution and cache-invalidation tradeoffs.
