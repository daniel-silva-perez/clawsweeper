## Summary

Keep open. Current main already covers the public no-respawn escape hatch and packaged Node compile-cache enablement, but it does not provide the requested first-class pre-warmed cache workflow or `openclaw warm-cache` command, so the central upstream ask remains unresolved and needs maintainer product/performance review.

## What This Changes

The issue requests a first-class way to pre-warm OpenClaw's Node compile cache for container cold starts and a documented opt-in to skip the CLI self-respawn when operators have already prepared the Node environment.

## Best Possible Solution

Keep the issue open and decide the remaining product shape: likely a dedicated, opt-in `openclaw warm-cache --output ...` command with tests and docs, while avoiding automatic postinstall priming or extension-entry imports until maintainers are comfortable with the lifecycle and side-effect model.

## Reproduction Assessment

Yes for triage: source inspection reproduces the remaining gap because current main has no `warm-cache` CLI/doc surface, while tests prove the existing compile-cache and no-respawn pieces. I did not run the reporter's timing benchmark because this review was read-only.

## Solution Assessment

Unclear. The no-respawn portion is already solved by `OPENCLAW_NO_RESPAWN` and foreground gateway skip behavior, but the proposed prewarm/postinstall approaches need maintainer design review before treating one as the best upstream fix.
