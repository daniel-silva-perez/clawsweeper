## Summary

Keep open. Current main has a generic `openclaw pairing approve --notify` option, but WhatsApp does not provide a `notifyApproval` pairing adapter, so approving a WhatsApp sender still cannot send the documented confirmation back to that sender.

## What This Changes

The issue requests that WhatsApp DM pairing approvals notify the newly approved sender, either via an approval flag or a configurable/default confirmation.

## Best Possible Solution

Wire WhatsApp into the existing pairing notification contract by adding a WhatsApp text approval adapter that sends `PAIRING_APPROVED_MESSAGE` through the active WhatsApp listener, and make the generic notify path preserve the approved request's account scope for multi-account senders.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: `openclaw pairing approve whatsapp <code> --notify` reaches `notifyPairingApproved`, but the WhatsApp pairing adapter lacks `notifyApproval`, so the function returns without sending anything.

## Solution Assessment

Yes for a narrow fix. Reusing the existing `--notify` pairing adapter path is the smallest maintainable solution; changing approvals to notify by default or replay blocked messages should be treated as a separate product decision.
