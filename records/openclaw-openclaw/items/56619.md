## Summary

Keep open. Current main has a generic opt-in `openclaw pairing approve --notify` path, but WhatsApp still declares only a pairing ID label and no approval notifier, so approving a WhatsApp pairing request cannot notify the newly approved sender.

## What This Changes

The issue requests that an approved WhatsApp sender receive an automatic confirmation after `openclaw pairing approve whatsapp <code>`, with optional configurability or replay of the blocked message.

## Best Possible Solution

Add a WhatsApp approval notification adapter that reuses the existing pairing notification contract and sends `PAIRING_APPROVED_MESSAGE` through the active WhatsApp listener for the approved request/account, while keeping notification opt-in and leaving replay/default-policy changes for separate product work.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: `openclaw pairing approve whatsapp <code> --notify` reaches `notifyPairingApproved`, but the WhatsApp pairing adapter lacks `notifyApproval`, so the helper returns without sending anything.

## Solution Assessment

Yes for the narrow repair. Reusing the existing `--notify` adapter path is the smallest maintainable fix; making notification default or replaying blocked messages would be a separate behavior change.
