## Summary

Keep open. Current main still runs Active Memory as a foreground before_prompt_build hook and awaits recall before the main prompt is built. Recent code bounds the wait and catches failures, but the reported product problem remains: a configured 30s Active Memory timeout can still sit on the user-visible direct-chat reply path, and fresh related issue context reports the same timeout class on v2026.4.26.

## Best Possible Solution

Keep this issue open as the broader Active Memory direct-chat latency and reliability tracker unless maintainers intentionally split it. A complete fix should move recall off the reply-critical path, add a very small fail-open foreground budget with optional late memory update, or add an explicit prewarm/readiness/indexing gate so cold memory work is not paid inside the first user turn. Preserve the existing hard-deadline and toolsAllow fixes, and document or repair the reindexing requirement if it remains part of the operational answer.
