## Summary

Keep open. Current main still runs Active Memory in the foreground `before_prompt_build` path and awaits the recall result before the main reply can be built. Recent mitigations bound the wait and make some failures fail open, but they do not implement the requested non-blocking/background behavior or a prewarm/readiness gate, and fresh issue context still reports slow or timeout behavior after reindexing.

## Best Possible Solution

Keep this issue open as the canonical Active Memory direct-chat latency/reliability tracker unless a maintainer intentionally splits it. A complete fix should make recall safe for primary chat through a non-blocking/background recall mode, a much smaller foreground fail-open budget with optional late memory update, or an explicit prewarm/readiness/indexing gate so cold memory work is not paid inside the first user turn. Preserve the existing hard-deadline and toolsAllow mitigations, and document or fix the reindexing requirement if `openclaw memory index --force` is part of the operational answer.
