## Summary

Keep open. Current main has added useful mitigations for Active Memory timeouts, including a hard prompt-hook deadline, bounded partial summaries, caching, and a timeout circuit breaker, but the core reported behavior is not fully solved: eligible direct-chat turns still await Active Memory in the foreground prompt-build path and can spend the configured timeout budget before the main reply proceeds. The open narrower single-shot request #72359 is related, but it does not fully replace this broader latency/reliability tracker.

## What This Changes

The issue asks for Active Memory in direct-chat sessions to complete quickly or fail open without blocking visible replies for roughly the configured timeout across model configurations.

## Best Possible Solution

Keep this as the broad Active Memory direct-chat latency and reliability tracker unless maintainers explicitly consolidate it. The end state should make primary chat safe by choosing one or more deliberate designs: an opt-in single-shot low-latency path such as #72359, a non-blocking/background recall path with optional late memory updates, or an explicit prewarm/readiness/indexing gate so cold memory work is not paid inside the first user turn. Existing hard-deadline, cache, partial-summary, toolsAllow, and circuit-breaker behavior should remain in place.
