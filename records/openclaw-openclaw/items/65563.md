## Summary

Current main still appears affected. The TaskFlow SQLite migrator copies legacy `owner_session_key` into `owner_key` only in one schema state and does not rebuild/drop the legacy NOT NULL column, while new `flow_runs` inserts only write `owner_key`. The webhooks `create_flow` path still relies on the route-bound session, and open PR #65767 is a same-author implementation candidate for this exact migration gap.

## Best Possible Solution

Keep this issue open and review PR #65767, or land an equivalent maintainer patch, that rebuilds or otherwise normalizes legacy `flow_runs` tables so `owner_session_key` no longer blocks new inserts, preserves existing `owner_key` data, and adds regression coverage for both full-legacy and partially migrated SQLite schemas plus managed flow creation.
