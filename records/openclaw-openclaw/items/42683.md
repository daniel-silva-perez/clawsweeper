## Summary

Keep open. Current main still lacks iMessage inbound reply-context enrichment when imsg omits resolved `reply_to_*` fields, and this PR is a useful implementation candidate, but it is not correct as-is because it targets the removed `src/imessage` path and depends on `reply_to_guid` despite the related iMessage analysis identifying `thread_originator_guid` as the reliable reply target.

## What This Changes

The PR adds a read-only chat.db lookup that fills iMessage `reply_to_id`, `reply_to_text`, and `reply_to_sender` before inbound dispatch, with unit tests for empty/missing databases, old schemas, missing quoted rows, and valid replies.

## Best Possible Solution

Port or supersede the idea in the active `extensions/imessage/src/monitor` implementation. The final fix should enrich missing inbound reply context before enqueue/dispatch, prefer upstream resolved fields or `thread_originator_guid`, skip local database reads for remote SSH-wrapper sessions, keep chat.db access read-only and failure-tolerant, and add regression coverage for GUID-only replies plus missing/old/locked database cases.

## Reproduction Assessment

Yes. The reported live path is clear: configure iMessage with `dbPath`, send an inline reply, and observe that imsg notifications without resolved `reply_to_*` fields reach current main without enrichment; the same gap is visible statically because `handleMessage` enqueues the parsed payload directly and `describeReplyContext` requires `reply_to_text`.

## Solution Assessment

No, not as-is. The approach is directionally useful, but the maintainable fix needs to land in the active extension path and use the reliable reply-origin source rather than `reply_to_guid` alone.
