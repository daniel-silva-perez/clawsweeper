## Summary

Keep this PR open. Current main still does not implement the intended iMessage chat.db/GUID fallback for inbound reply context, and the remaining work is real. The active code now lives under `extensions/imessage`, while this PR targets the removed `src/imessage` path, so it is useful as an implementation candidate but not merge-ready as-is.

## Best Possible Solution

Keep this PR open as a useful implementation candidate, or supersede it with a maintainer PR that ports the idea into `extensions/imessage/src/monitor`. The final fix should resolve iMessage GUID/thread-origin metadata into the existing `ReplyToId`/`ReplyToBody`/`ReplyToSender` flow, align with #10872 on the reliable origin field, skip local chat.db reads for remote sessions, and add regression coverage for GUID-only replies plus permission/schema/locking failures.
