## Summary

Keep this PR open. Current main still does not implement the intended iMessage chat.db/GUID fallback for missing inbound reply context: the active iMessage plugin only consumes already-populated `reply_to_*` fields, and this PR targets the removed `src/imessage/monitor/*` path instead of the current `extensions/imessage/src/monitor/*` implementation. The PR is useful as an implementation candidate, but it is not merge-ready against current main.

## Best Possible Solution

Keep this PR open as a useful implementation candidate, or supersede it with a maintainer PR that ports the idea into `extensions/imessage/src/monitor`. The final fix should enrich missing inbound reply context before `inboundDebouncer.enqueue`, use the existing `ReplyToId`/`ReplyToBody`/`ReplyToSender` flow, align with #10872 on the reliable origin GUID field, skip local database reads for remote sessions, and add regression coverage for GUID-only replies plus permission/schema/locking failures.

## Risks / Open Questions

- Closing this as implemented would leave GUID-only or chat.db-backed iMessage inline replies unresolved on current main.
- Merging this PR as-is would not update the active bundled plugin path on current main; the implementation needs to be ported from `src/imessage/monitor/*` to `extensions/imessage/src/monitor/*`.
- A port should resolve the #10872 concern around `thread_originator_guid` versus `reply_to_guid` before relying on one database field, and should preserve local-only gating so remote SSH-wrapper setups do not read the wrong local chat.db.
- The final implementation must keep chat.db access read-only and fail closed on permission, schema, missing-file, and locking errors.
