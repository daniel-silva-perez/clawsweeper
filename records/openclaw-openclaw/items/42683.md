## Summary

Keep this PR open. Current main still does not implement the intended iMessage fallback for inbound reply/quote context when imsg omits `reply_to_*`, and the PR remains a useful implementation candidate. It is not merge-ready as-is because the submitted patch targets the old `src/imessage/monitor/*` path, while current main runs the bundled plugin under `extensions/imessage/src/monitor/*`.

## Best Possible Solution

Keep this PR open as a useful implementation candidate, or supersede it with a maintainer PR that ports the idea into `extensions/imessage/src/monitor`. The final fix should enrich missing inbound reply context before `inboundDebouncer.enqueue`, use the existing `ReplyToId`/`ReplyToBody`/`ReplyToSender` flow, align with #10872 on the reliable origin GUID field, preserve local-only gating for remote iMessage sessions, and add regression coverage for GUID-only replies plus permission/schema/locking failures.

## Risks / Open Questions

- Closing this as implemented would leave GUID-only or chat.db-backed iMessage inline replies unresolved on current main.
- Merging this PR as-is would not update the active bundled plugin path on current main; the implementation needs to be ported from `src/imessage/monitor/*` to `extensions/imessage/src/monitor/*`.

- The chat.db fallback must stay read-only and fail closed on permission, missing-file, schema, and locking errors.
