## Summary

Keep open. Current main and v2026.4.29 contain partial Google Vertex support, but the reported default `google-vertex/*` path is still not proven fixed: implicit Google Vertex model resolution can still produce a non-Vertex API, native Vertex streaming is gated behind `model.api === "google-vertex"`, and ADC handling is authorized-user only while the report calls out service-account ADC. Open PR #60860 still uses closing syntax for this issue, so maintainer review should continue there or in an equivalent replacement.

## What This Changes

The issue asks for `google-vertex` to retain Vertex AI transport and ADC auth while sharing only Gemini model-id normalization with the Google provider.

## Best Possible Solution

Finish native Google Vertex support in the Google plugin so ordinary `google-vertex/*` model refs select a Vertex runtime model, construct Vertex project/location endpoints, use ADC/Bearer auth separately from AI Studio API-key auth, and either support service-account ADC or document and test the accepted ADC credential types.

## Reproduction Assessment

Yes. A source-level reproduction path exists: resolve `google-vertex/gemini-3.1-pro-preview` through the current forward-compat model path and it is still asserted to carry `api: "google-gemini-cli"`, while the native Vertex stream only activates for `api: "google-vertex"`.

## Solution Assessment

No. The current shipped implementation is useful partial progress, but the narrow maintainable fix still needs to connect implicit/configured `google-vertex` models to the native Vertex transport and settle ADC credential-type behavior; PR #60860 or a maintainer replacement is the right vehicle to review that.
