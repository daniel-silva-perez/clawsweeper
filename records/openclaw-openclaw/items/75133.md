## Summary

Keep open. Current main still drops compatible-bundle `activation` metadata before bundle registry records are built, and the PR’s approach is a narrow thread-through of the existing activation normalization path. I found no blocking functional or security issue by static review; validation still needs the PR branch’s normal CI because this review was read-only.

## What This Changes

The PR exports manifest activation normalization, stores normalized `activation` on Codex, Claude, and Cursor bundle manifests, copies it into bundle registry records, and adds registry coverage for a Claude bundle with `activation.onStartup: false`.

## Best Possible Solution

Land the narrow metadata-preservation fix after normal maintainer review and CI, keeping activation as control-plane metadata shared with native manifest parsing and avoiding any runtime-load behavior change beyond honoring the existing `onStartup` semantics for bundle records.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: on current main, a Claude/Codex/Cursor bundle manifest containing `activation` is normalized into a `BundlePluginManifest` without that field, then `buildBundleRecord` produces a registry record with `activation === undefined`.

## Solution Assessment

Yes. Reusing the existing manifest activation normalizer and threading the result through the bundle manifest and registry record is the narrowest maintainable fix; converting bundles to native plugins or special-casing the compatibility warning would solve the wrong layer.
