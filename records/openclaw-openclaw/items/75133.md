## Summary

Keep open. Current main still drops bundle activation metadata before bundle registry records are built, and this PR is a narrow implementation that threads the existing activation normalizer through that missing path. I did not find a concrete blocking review finding in the provided latest diff, but it still needs maintainer review and targeted CI before merge.

## What This Changes

The branch exports manifest activation normalization, stores normalized activation on Codex, Claude, and Cursor bundle manifests, copies it into bundle registry records, adds Claude bundle registry coverage for `activation.onStartup: false`, and clears service-mode environment state in a gateway CLI test.

## Best Possible Solution

Land the narrow metadata-preservation fix after maintainer review and targeted CI, keeping activation as manifest/control-plane metadata for bundle records rather than adding compatibility-warning special cases or forcing bundle authors to convert inert content bundles into native plugins.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: on current main, a bundle manifest with `activation` is parsed into a bundle manifest object with no activation field, then `buildBundleRecord` creates a registry record with activation undefined, which can hit the legacy implicit startup sidecar compatibility path.

## Solution Assessment

Yes. Reusing the existing manifest activation normalizer and threading the normalized metadata through bundle parsing and registry assembly is the narrowest maintainable fix for the reported behavior.
