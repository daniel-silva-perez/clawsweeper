## Summary

Keep this issue open. Current main still supports automatic model fallback chains, strict user-selected models, explicit empty fallback lists, fallback status events, and persisted auto fallback state, but it does not implement the requested interactive confirmation mode before moving from the primary model to a fallback. The provided related title matches are Feishu interactive-card/text-fallback items, not the same model failover policy request.

## Best Possible Solution

Keep this open and implement the feature in the core model fallback policy. A good path is to add a backwards-compatible `fallbackMode: "auto" | "confirm"` field to the agent model config/schema/docs, gate only interactive chat sessions before trying a non-primary fallback candidate, define deterministic defaults for cron and other non-interactive runs, reuse existing channel prompt or command-response infrastructure, and add focused tests for accepted, declined, picked, automatic, strict, and non-interactive fallback behavior.
