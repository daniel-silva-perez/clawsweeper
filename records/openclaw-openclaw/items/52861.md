## Summary

Keep open. Current main has partial prompt-cost mitigations for skills, bootstrap files, and compaction, but the embedded runner still materializes the full effective tool set into model-facing custom tool schemas for the run, so the central automatic per-turn tool relevance routing remains unimplemented.

## What This Changes

The issue asks OpenClaw to reduce repeated prompt/context tokens by loading tool schemas, skill metadata, workspace files, and history conditionally instead of injecting broad static context every turn.

## Best Possible Solution

Keep this as the umbrella prompt-overhead tracker and split implementation into policy-preserving pieces: add a per-turn relevance layer after effective tool policy but before customTools registration, keep skills on the existing compact catalog path, and treat bootstrap indexing/streaming as a separate context-engine or bootstrap-injection design.
