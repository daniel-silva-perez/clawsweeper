## Summary

Keep this issue open. Current main still lacks first-class outgoing Telegram custom emoji support for messages, captions, and replies: shared Markdown cannot represent a custom emoji id, Telegram rendering only emits the existing safe HTML subset, and send/caption delivery still routes through HTML parse mode without an `entities` or `caption_entities` send surface. The related closed PR #13905 is useful prior art but was not merged, and the related open PR #47562 is scoped to reactions rather than message/caption rendering.

## What This Changes

Request outgoing Telegram custom emoji support in messages, captions, and replies, including custom emoji identifiers and rendering/sending behavior wherever standard emoji formatting is supported.

## Best Possible Solution

Define the API representation first, then implement validated Telegram custom emoji entities through the Telegram plugin send, caption, reply, edit, delivery, and streaming paths plus the shared message tool, with focused docs and regression tests that preserve the current HTML fallback behavior.
