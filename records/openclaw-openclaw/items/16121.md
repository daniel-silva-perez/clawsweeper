## Summary

Keep this issue open. Current main still lacks first-class outgoing Telegram custom emoji support: the shared Markdown IR has no custom emoji span/entity, the Telegram renderer emits HTML parse-mode markup only, the send/message-tool paths do not accept or forward `entities` or `caption_entities`, and streaming previews also use rendered text plus `parse_mode`. Related #40848 remains open for a narrower message-tool entities surface, while PR #13905 was closed unmerged.

## Best Possible Solution

Keep this issue open and have maintainers decide the Telegram API shape before implementation. The likely path is a Telegram-owned validated entities/custom-emoji surface, or an approved shared Markdown/IR custom emoji span, then thread it through message-tool sends, captions, reply delivery, edits/streaming previews, docs, and focused Telegram tests while preserving existing HTML/plain-text fallback behavior and coordinating with #40848, #10944, and lessons from closed PR #13905.
