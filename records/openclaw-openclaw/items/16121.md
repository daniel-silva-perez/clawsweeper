## Summary

Keep this issue open. Current main still lacks first-class inline Telegram custom emoji support for Markdown-rendered messages, captions, action sends/edits, replies, and streaming previews. There is a trusted HTML path in some delivery flows, but the normal Telegram action/runtime surfaces still cannot carry custom emoji IDs or safe fallback emoji spans. Related #40848 is a narrower open entities request, and closed PR #13905 was an unmerged attempt with delivery reliability concerns, not shipped behavior.

## Best Possible Solution

Keep this issue open and implement the capability in the bundled Telegram plugin. The best path is a narrow Telegram-owned design that carries custom emoji id plus fallback emoji through either a validated entity surface or an explicit safe Markdown/IR span, then threads it through messages, captions, action sends/edits, replies/quotes, preview streaming, docs, and tests while preserving HTML/plain-text fallback behavior. Coordinate with #40848 and learn from the unmerged #13905 attempt rather than accepting raw HTML broadly.

## Risks / Open Questions

- A raw `<tg-emoji>` allowlist could weaken the existing safe escaping and parse-error fallback behavior if implemented without a narrow trusted path.
- Entity-based custom emoji support needs UTF-16 offset handling, chunking/splitting behavior, and graceful fallback when Telegram rejects malformed entities or account eligibility is missing.
- Related #40848 and #10944 overlap with this request, so maintainers should choose whether this is solved via validated entities, a safe Markdown syntax, parse-mode config, or a combination.
