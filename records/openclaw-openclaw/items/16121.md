## Summary

Keep this issue open. Current main still does not provide first-class Telegram inline custom emoji support for Markdown-rendered text, captions, message-tool actions, edits, replies, or preview streaming; it mostly renders Telegram HTML with plain-text fallback. Related #40848 is a narrower open entities request, and PR #13905 was closed unmerged, so there is no shipped implementation to close against.

## Best Possible Solution

Keep this open and implement the capability in the bundled Telegram plugin. The best path is a Telegram-owned design that carries custom emoji id plus fallback emoji through a validated entity surface or explicit safe Markdown/IR span, then threads it through text, captions, action sends/edits, replies/quotes, preview streaming, docs, and tests while preserving HTML/plain-text fallback behavior and coordinating with #40848.

## Risks / Open Questions

- A raw <tg-emoji> allowlist could weaken the current safe escaping and HTML parse-error fallback behavior if implemented without a narrow trusted path.
- Entity-based custom emoji support needs UTF-16 offset handling, chunking/splitting behavior, and graceful fallback when Telegram rejects malformed entities or the bot/account is not eligible.
- Related #40848 and #10944 overlap with this request, so maintainers should choose the product/API shape before implementation: validated entities, a safe Markdown/IR custom emoji span, parse-mode configuration, or a combination.
