## Summary

Keep this issue open. Current main still lacks first-class outgoing Telegram custom emoji support for messages and captions: the shared Markdown IR cannot represent a custom emoji id, Telegram rendering only emits the existing HTML subset, the send/caption paths still use `parse_mode: "HTML"`, and the shared message tool has no custom emoji/entity input surface. The closed PR #13905 is relevant prior art but was not merged, and open PR #47562 is scoped to reactions rather than message/caption rendering.

## What This Changes

Request Telegram custom emoji support for bot messages, captions, and replies, including custom emoji identifiers and rendering according to Telegram Bot API formatting rules.

## Best Possible Solution

Define the public representation first, then implement validated Telegram custom emoji entities through the Telegram plugin send, caption, reply, edit, delivery, and streaming paths plus the shared message tool, with docs and tests that preserve the current HTML fallback behavior.
