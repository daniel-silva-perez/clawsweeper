## Summary

Keep open. Current main still lacks first-class Telegram custom emoji support for outgoing messages, captions, edits, draft previews, and the shared message tool; the related open PR only targets reactions, and prior maintainer review already kept the broader tracker open for API-shape follow-up.

## What This Changes

The issue requests Telegram custom emoji support for messages, captions, and replies, including recognizing custom emoji identifiers and sending/rendering them wherever normal emoji formatting is supported.

## Best Possible Solution

Define a small public custom-emoji representation, then implement validated Telegram entity handling through send, caption, reply, edit, delivery, and streaming paths plus the shared message tool, with docs and focused regression coverage that preserves existing HTML fallback behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: current main has no custom-emoji IR span or outgoing entities/caption_entities send surface, and Telegram send/edit/delivery paths still rely on HTML parse_mode.

## Solution Assessment

Unclear. The feature direction is valid, but a narrow raw <tg-emoji> passthrough would not cover message-tool entities, captions, edits, draft streaming, chunking, or UTF-16 offset validation.
