## Summary

Keep this issue open. Current `main` still lacks first-class Telegram custom emoji support for messages, captions, and replies: the shared Markdown IR cannot carry a custom emoji id, the Telegram renderer only emits the existing safe HTML tags, and the send/caption paths still use HTML parse mode without an entity or caption-entity surface. The related open PR #47562 is about custom emoji reactions, which is adjacent but does not solve message/caption rendering; closed PR #13905 remains useful prior art but did not merge.

## What This Changes

Request Telegram custom emoji support in bot messages, captions, and replies, including recognizing custom emoji identifiers and rendering/sending them where regular emoji formatting is supported.

## Best Possible Solution

Define the representation first, then implement validated Telegram custom emoji entities through the Telegram plugin's send, caption, reply, edit, delivery, and streaming paths plus the shared message tool and focused docs/tests, while preserving current HTML fallback behavior for existing formatting.
