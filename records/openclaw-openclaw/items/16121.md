## Summary

Keep this issue open. Current main still does not provide first-class outgoing Telegram custom emoji support: the Markdown IR has no custom emoji span/entity, the Telegram send path still sends rendered HTML with parse_mode rather than message/caption entities, and the message tool/action path does not accept or forward Telegram entities. A maintainer already kept this open and related #40848 remains open for the narrower message-tool entity surface; closed PR #13905 is useful prior art but was not merged.

## Best Possible Solution

Keep this issue open for maintainer API-shape review. The best path is to decide whether Telegram custom emoji belongs in a Telegram-owned validated `entities` / `caption_entities` surface, a shared Markdown/IR custom emoji span, or both; then thread that through message-tool sends, direct send helpers, captions, reply delivery, edits/streaming previews, docs, and focused Telegram tests while coordinating with #40848, #10944, and the lessons from closed PR #13905.

## Risks / Open Questions

- A narrow allowlist for raw `<tg-emoji>` HTML would not solve the broader request because message-tool sends, captions, replies, and streaming previews still lack a validated entity surface.
- Telegram entity offsets use UTF-16 code units; an implementation that calculates offsets on JavaScript string indices or after HTML escaping can create parse errors or failed delivery.
- Entity sends must preserve existing Telegram chunking, parse-error fallback, stale preview handling, native quote replies, media captions, and plain-text degradation paths.
