## Summary

Keep open. Current main satisfies the Gemini half through the bundled Google/Gemini web_search provider, but the ZAI/GLM half is still absent from the ZAI plugin manifest, registration path, docs, and shared registration contract expectations. The prior Gemini PR is merged; the prior ZAI attempts are closed unmerged, so the remaining user-visible request is valid and still actionable.

## What This Changes

The issue requests native or bundled web_search support for ZAI/GLM and Google/Gemini so users can search without separate third-party search API credentials.

## Best Possible Solution

Add a bundled ZAI-owned web_search provider through the existing webSearchProviders plugin contract, with manifest metadata, lazy runtime execution, ZAI credential/config reuse, docs, changelog, and registration/runtime tests. Keep the existing Gemini provider intact and leave Gemini freshness behavior to #66498.

## Reproduction Assessment

Yes. The structural reproduction path is high-confidence: current main has no ZAI webSearchProviders manifest entry, no registerWebSearchProvider call, and no ZAI web_search runtime, while Gemini has all three.

## Solution Assessment

Yes. A provider-plugin implementation is the narrowest maintainable solution because search provider behavior is already plugin-owned for Gemini/Kimi/Grok and does not require a new core bypass or bundled-id special case.
