## Summary

Keep open. Current main implements the Gemini side through the bundled Google web_search provider, but the ZAI/GLM side is still missing from the bundled ZAI plugin manifest, runtime registration path, shared registration contract expectations, tests, and docs. The prior ZAI search PRs in the provided context are closed unmerged, while maintainer comments already confirmed this should remain open for ZAI follow-up.

## What This Changes

Add native provider-backed web_search support for ZAI/GLM and Google/Gemini so users can use provider-supplied search without configuring a separate third-party search API key.

## Best Possible Solution

Add a bundled ZAI-owned web_search provider through the existing webSearchProviders plugin contract, with manifest metadata, lazy runtime execution, ZAI credential/config reuse, docs, changelog, and registration/runtime tests. Keep the shipped Gemini provider intact and leave Gemini freshness behavior to #66498.

## Reproduction Assessment

Yes. The high-confidence structural reproduction is to inspect current main: ZAI has no webSearchProviders manifest entry, no registerWebSearchProvider call, no ZAI web_search runtime, and its registration contract expects media only, while Gemini has the full provider path.

## Solution Assessment

Yes. A provider-plugin implementation is the narrowest maintainable solution because OpenClaw already models Gemini, Kimi, Grok, and other search integrations as plugin-owned web_search providers; adding a ZAI special case directly into core model traffic would bypass that contract.
