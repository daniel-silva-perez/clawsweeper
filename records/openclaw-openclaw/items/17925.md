## Summary

Keep this issue open. Current main implements the Gemini side through a bundled `gemini` web_search provider using Google Search grounding, but the requested ZAI/GLM web_search support is still absent from the ZAI plugin, manifest contracts, docs, and registration tests. The related Zhipu/ZAI PRs in the provided issue context were closed unmerged, and the narrower Gemini freshness follow-up remains separate in #66498.

## Best Possible Solution

Keep this issue open until the bundled ZAI plugin implements GLM/ZAI native search through the existing provider-owned web_search contract, or maintainers explicitly choose a direct model-call passthrough design. The Gemini portion is already implemented; Gemini freshness semantics should remain tracked by #66498.

## Risks / Open Questions

- The remaining implementation shape still needs maintainer choice: a registered bundled `zai` web_search provider fits current architecture, while direct model-call passthrough would alter provider runtime behavior differently.
- Adding ZAI search affects auto-detection and default provider selection if it reuses `ZAI_API_KEY`; that priority and fallback behavior should be decided deliberately.
- ZAI search returns external web content, so any implementation needs the same guarded fetch, credential redaction, citation handling, caching, and prompt-injection wrapping standards used by existing web_search providers.
