## Summary

Keep open. Current main still resolves and writes doctor config through the invoking process environment without a doctor preflight comparison against the running service or gateway config path, and #63268 remains an open paired implementation PR for this issue. The prior discussion also supports a defensive warning/guard rather than closing this as already handled.

## What This Changes

Request a doctor preflight warning or guard when `openclaw doctor --fix` is about to write a config path that diverges from the running gateway or service config path, especially for root shells versus systemd service homes.

## Best Possible Solution

Review and land #63268 or a replacement that compares doctor’s planned write path with a generic running gateway or service config-path signal, emits a clear `OPENCLAW_CONFIG_PATH` remediation, and preserves existing config resolution semantics.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. The reporter provided a concrete root-shell/systemd-HOME mismatch path, and current source shows doctor resolves and writes config from the invoking environment without comparing the service config path; I did not run destructive `doctor --fix` because this review is read-only.

## Solution Assessment

Is this the best way to solve the issue? Unclear. A warning/guard in doctor preflight is the right narrow direction, but the safest implementation should use a generic service or running-gateway config signal rather than only probing hard-coded alternate home directories.
