## Summary

Keep this PR open. Current main has already moved Discord and Slack onto richer sendPayload implementations, but Google Chat, Mattermost, Microsoft Teams, and Synology Chat still do not expose sendPayload, so the central adapter-coverage work is not obsolete; the branch needs a rebase and rework around the current shared payload helpers before it is mergeable.

## What This Changes

The PR branch adds per-adapter sendPayload methods plus colocated tests for Discord, Google Chat, Mattermost, Microsoft Teams, Slack, and Synology Chat.

## Best Possible Solution

Keep the current Discord and Slack implementations, then add Google Chat, Mattermost, Microsoft Teams, and Synology Chat sendPayload support using the current shared sendTextMediaPayload-style helper or equivalent contract-preserving logic, with focused adapter and shared payload contract tests.

## Reproduction Assessment

Yes for the review findings: a focused unit test can invoke the proposed sendPayload with an implicit replyToId and multiple media URLs or chunks and observe that every delegated send reuses the same context. The original feature gap is source-verifiable on current main because rg finds no sendPayload in Google Chat, Mattermost, Microsoft Teams, or Synology Chat.

## Solution Assessment

No, the PR’s current copied per-adapter loops are not the best way to solve this on current main. The safer path is to rebase and use the existing shared payload helper semantics so reply-to fanout, formatting-aware chunking, and current Discord/Slack behavior stay aligned.

Full review comments:

- **[P2] Preserve reply fanout in sendPayload loops:** `extensions/msteams/src/outbound.ts:30-34`
  - body: The new loop forwards `{ ...ctx }` to every media send, so an implicit `replyToId` is reused for each attachment instead of being consumed after the first send. Current payload delivery uses reply-to fanout for multi-part sends; use the shared helper or createReplyToFanout so later media/chunks do not all thread to the original message.
  - confidence: 0.86
- **[P2] Pass formatting context to adapter chunkers:** `extensions/msteams/src/outbound.ts:39`
  - body: This chunks text with `msteamsOutbound.chunker(text, limit)` but drops the `ctx.formatting` options that current delivery/helpers pass through. Rich sendPayload deliveries can ignore configured formatting constraints such as line limits, so thread the formatting context or reuse the shared sendTextMediaPayload helper.
  - confidence: 0.82
- **[P2] Break the Synology self-reference cycle:** `extensions/synology-chat/src/channel.ts:41`
  - body: Changing the factory to infer `const plugin = { ... }` while `sendPayload` references `plugin.outbound` creates a self-referential initializer under the repo's strict TypeScript settings. Add an explicit ChannelPlugin type/satisfies annotation or route through local send functions to avoid TS7022-style inference failures.
  - confidence: 0.74
