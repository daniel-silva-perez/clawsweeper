## Summary

Keep this PR open. Current main only covers the Discord and Slack parts of the requested batch, while Google Chat, Mattermost, Microsoft Teams, and Synology Chat still lack sendPayload; however, the PR branch is stale against current main and its copied loops miss current shared helper semantics, so it should not merge as-is.

## What This Changes

The PR branch adds per-adapter sendPayload methods plus colocated tests for Discord, Google Chat, Mattermost, Microsoft Teams, Slack, and Synology Chat.

## Best Possible Solution

Keep the current Discord and Slack sendPayload implementations, then add Google Chat, Mattermost, Microsoft Teams, and Synology Chat support on top of current main using sendTextMediaPayload or equivalent contract-preserving logic, with focused adapter and shared payload contract tests.

## Reproduction Assessment

Yes for the review findings: a focused unit test can invoke the proposed sendPayload with an implicit single-use reply and multiple media URLs or text chunks, then assert later sends do not reuse the consumed replyToId and chunkers receive formatting context. The original feature gap is source-verifiable because current main has no sendPayload definitions in the four remaining adapter trees.

## Solution Assessment

No. The PR's copied per-adapter loops are not the best current solution; the safer path is to rebase and reuse the shared helper semantics so reply fanout, formatting-aware chunking, and existing Discord/Slack behavior stay aligned.

## Review Findings

Full review comments:

- **[P2] Preserve reply fanout in sendPayload loops:** `extensions/msteams/src/outbound.ts:30-34`
  - body: The new loop forwards `{ ...ctx }` to every media send, so an implicit single-use `replyToId` is reused for every attachment instead of being consumed after the first send. Use the shared helper or createReplyToFanout so later media and chunks do not all thread to the original message.
  - confidence: 0.86
- **[P2] Pass formatting context to adapter chunkers:** `extensions/msteams/src/outbound.ts:39`
  - body: This chunks text with `msteamsOutbound.chunker(text, limit)` but drops `ctx.formatting`, while the current helper passes formatting options through. Rich sendPayload deliveries can ignore configured formatting constraints such as line limits, so thread the formatting context or reuse sendTextMediaPayload.
  - confidence: 0.82
- **[P2] Break the Synology self-reference cycle:** `extensions/synology-chat/src/channel.ts:40`
  - body: Changing the factory to infer `const plugin = { ... }` while nested sendPayload closes over `plugin.outbound` creates a self-referential initializer under strict TypeScript inference. Add an explicit ChannelPlugin annotation or route through local send functions to avoid TS7022-style failures.
  - confidence: 0.74
