## Summary

Keep this PR open for maintainer review: current main does not implement session-scoped extra bootstrap files, but the proposed patch has a blocking backwards-compatibility and data-exposure concern because it removes the recognized-basename gate for every glob match.

## What This Changes

The PR adds a `sessions` map to the bundled `bootstrap-extra-files` hook, removes the extra-file basename allowlist, stabilizes extra-file ordering, and updates hook docs and tests.

## Best Possible Solution

Land one coordinated bootstrap-extra-files contract that supports exact session-scoped paths and custom context files while preserving a safe migration path for legacy glob configs, with docs, tests, and changelog credit aligned with #39120 and #73917.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Keep legacy globs from injecting every matched file:** `src/agents/workspace.ts:738-740`
  - body: This removes the recognized-basename gate for all resolved patterns. An existing config such as `paths: ["packages/*/*"]` used to load only AGENTS/TOOLS-style files, and current main even tests that `README.md` is skipped; after this patch, any matched workspace file is read into bootstrap context and sent to the model. Please require an explicit opt-in for arbitrary basenames or keep the bypass limited to exact custom paths/session entries so broad legacy globs do not become a data-exposure regression.
  - confidence: 0.87

## Security Review

Status: needs_attention

Summary: The diff broadens which workspace files can be injected into model context for existing glob configs, so the data-access impact needs maintainer review before merge.

Concerns:

- **[medium] Broad globs can expose arbitrary workspace files:** `src/agents/workspace.ts:738`
  - body: Removing the basename allowlist means every file matched by an existing configured glob can be read and included in bootstrap context, not only recognized bootstrap basenames. Workspace containment and size guards still apply, but they do not prevent accidental disclosure of in-workspace files to the model.
