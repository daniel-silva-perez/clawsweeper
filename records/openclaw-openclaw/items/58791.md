## Summary

Keep this PR open. Current main does not implement the proposed non-provider plugin API: `OpenClawPluginApi` has provider `createStreamFn`/`wrapStreamFn` hooks, but no `registerStreamFnWrapper`, no registry field for plugin-wide stream wrappers, and no runner application point for non-provider plugins. The idea is a plausible core plugin-API extension rather than a ClawHub-only plugin, but the current PR still needs maintainer design/security review and likely revisions before merge.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is to decide the SDK contract first: either add a constrained, tested non-provider stream interception seam with safe composition/return validation and documented ordering, or narrow the supported use case to provider-owned hooks/runtime hints. The implementation should update all plugin API builders/test helpers/docs and add focused tests for registration mode, wrapper composition, per-call execution, invalid wrapper returns, and interaction with provider/core wrappers.

## Risks / Open Questions

- The proposed API gives arbitrary loaded plugins the ability to replace the model stream function; the PR needs an explicit trust/security model, ordering rules, and tests proving provider/core enforcement wrappers cannot be bypassed unintentionally.
- The advertised per-call model routing use case is not fully proven by the patch as described: applying a wrapper after provider stream selection may not safely support cross-provider or cross-API model swaps.
- The PR patch shown in the provided context updates several test helpers, but current main has other OpenClawPluginApi helper surfaces such as src/plugin-sdk/plugin-test-api.ts that would also need to stay type-complete if the API is added.
