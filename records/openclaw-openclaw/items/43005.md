## Summary

Current main still does not implement the requested opt-in custom safe-header mechanism for fetch-guard cross-origin redirects. The redirect allowlist is fixed, `GuardedFetchOptions` has no additional-safe-headers field, provider/media paths still use the fixed helper, and tests still assert that `x-api-key` is stripped. The related #57588 tracks adjacent bearer-auth fallback behavior, not this redirect-header extension.

## Best Possible Solution

Keep this issue open. If maintainers accept the feature, add a narrow fail-closed fetch-guard extension point that validates header names, preserves the strict default, denies credential-like names unless there is an explicit trusted-host design, exposes the option only through the intended per-call/config/SDK surface, and adds tests for allowed custom headers, denied credential headers, same-origin redirects, cross-origin redirects, media downloads, and provider request config/schema behavior.

## Risks / Open Questions

- Allowing arbitrary custom headers across cross-origin redirects is security-sensitive; names such as `x-api-key`, `token`, `secret`, `cookie`, and `authorization` can leak credentials to attacker-controlled redirect targets if the option is too broad.
- `fetchWithSsrFGuard` is exported through plugin/runtime surfaces and used by provider, media, marketplace, memory-host, and plugin code paths, so any solution needs narrow defaults, validation, and regression coverage across the shared redirect behavior.
