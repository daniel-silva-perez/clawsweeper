## Summary

Current main still has only a fixed cross-origin redirect safe-header list, no `additionalSafeHeaders` or equivalent per-call/config option, and regression tests continue to assert that `x-api-key` is stripped on cross-origin redirects. This should remain open for maintainer security/product judgment.

## Best Possible Solution

Keep this issue open. If maintainers accept the enhancement, implement a narrow opt-in extension point in the fetch-guard/redirect-header owner path, preferably scoped by trusted destination or origin policy rather than a global header-name list. Preserve strict defaults, validate header names, handle credential-like names explicitly, expose the setting through the intended per-call/config/SDK surface, and add regression tests for allowed custom headers, denied credential headers, same-origin redirects, cross-origin redirects, media downloads, and provider request config behavior.

## Risks / Open Questions

- A broad header-name-only allowlist for cross-origin redirects could leak credential-like headers to attacker-controlled redirect targets.
- The shared fetch guard and media redirect helper affect provider, plugin, and media paths, so any accepted fix should remain fail-closed and have cross-surface regression coverage.
