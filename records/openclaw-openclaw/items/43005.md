## Summary

Current main still does not implement the requested opt-in custom safe-header mechanism for fetch-guard cross-origin redirects. The earlier ClawSweeper comment remains accurate against `891c7d9`: the redirect allowlist is fixed, `GuardedFetchOptions` has no additional-safe-headers field, provider/media paths still flow through the fixed helper, and regression tests still assert that `x-api-key` is stripped. The related #57588 is adjacent bearer-auth fallback hardening, not a duplicate or implementation of this request.

## Best Possible Solution

Keep this issue open. If maintainers accept the feature, implement a narrow fail-closed redirect-header extension point in the fetch-guard/redirect-header owner path, preferably scoped by trusted destination/origin policy rather than a global header-name-only list. Validate header names, preserve the strict default, handle credential-like names explicitly, expose it only through the intended per-call/config/SDK surface, and add regression tests for allowed custom headers, denied credential headers, same-origin redirects, cross-origin redirects, media downloads, and provider request config/schema behavior.

## Risks / Open Questions

- Allowing arbitrary custom headers across cross-origin redirects is security-sensitive; credential-like names such as `x-api-key`, `token`, `secret`, `cookie`, and `authorization` can leak to attacker-controlled redirect targets if the option is broad or header-name-only.
- `fetchWithSsrFGuard` is exported through plugin/runtime surfaces and used by provider, media, marketplace, memory-host, and plugin code paths, so any solution needs narrow defaults, validation, and regression coverage across the shared redirect behavior.
