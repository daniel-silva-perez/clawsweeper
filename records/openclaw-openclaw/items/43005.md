## Summary

Current main still does not implement an opt-in mechanism for custom safe headers on fetch-guard cross-origin redirects. The prior ClawSweeper review remains directionally accurate against bdfb408c: the allowlist is fixed, the guarded-fetch API has no additional-safe-headers option, provider/media paths still use the fixed helper, and tests explicitly assert that x-api-key is stripped on cross-origin redirects.

## Best Possible Solution

Keep this issue open for maintainer product/security judgment. If accepted, implement a narrow redirect-header extension point in the fetch-guard/redirect-header owner path, preferably scoped by trusted destination or origin policy rather than a global header-name list. Preserve strict defaults, validate header names, explicitly handle credential-like names, expose the setting only through the intended per-call/config/SDK surface, and add regression tests for allowed custom headers, denied credential headers, same-origin redirects, cross-origin redirects, media downloads, and provider request config/schema behavior.

## Risks / Open Questions

- A broad header-name-only allowlist for cross-origin redirects would be security-sensitive because credential-like headers such as x-api-key, token, secret, cookie, or authorization can leak to attacker-controlled redirect targets.
- fetchWithSsrFGuard is exported through plugin/runtime surfaces and used by provider, media, marketplace, memory-host, and plugin paths, so any fix should be narrow, fail-closed, and covered across shared redirect behavior.
