## Summary

Current main still has only a fixed cross-origin redirect safe-header allowlist. There is no per-call, config, or SDK surface for opting specific custom headers such as `x-api-key` back in, and current regression tests still require `x-api-key` to be stripped on cross-origin redirects.

## Best Possible Solution

Keep this issue open. If accepted, implement a narrow fetch-guard extension that preserves the current strict default, validates caller-provided header names, denies credential headers such as `authorization`, `cookie`, and `proxy-authorization` plus credential-like custom names unless deliberately designed otherwise, and plumbs the option only through the intended per-call/config or SDK surface. Add regression tests for allowed custom headers, denied credential headers, same-origin redirects, cross-origin redirects, media downloads if affected, and docs/schema updates if a user config surface is added. Treat #57588 as related security context, not the canonical tracker.

## Risks / Open Questions

- Allowing custom redirect-safe headers can leak API keys or other credentials across attacker-controlled cross-origin redirects unless the design validates header names, stays fail-closed by default, and blocks credential-like headers or scopes them tightly.
- `fetchWithSsrFGuard` is a shared core/SDK surface used by providers, plugins, media, marketplace, cron, tools, and memory-host code, so this needs maintainer security/product judgment rather than a broad config-only knob.
