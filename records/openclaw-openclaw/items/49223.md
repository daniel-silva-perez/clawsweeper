## Summary

Keep open. Current main still routes `sessions_send` target turns as generic `inter_session` input, still tells agents they may use `REPLY_SKIP`/`ANNOUNCE_SKIP`, and still suppresses visible delivery when the announce step returns `ANNOUNCE_SKIP`. Related #43848 overlaps the A2A skip behavior, but it tracks Telegram topic/transcript problems and is not a canonical replacement for this WhatsApp explicit-delivery request.

## Best Possible Solution

Keep the issue open and fix the core `sessions_send`/A2A path. The best direction is a backward-compatible delivery-intent signal or explicit prompt/runtime guard so clear requests to emit a normal channel message outrank generic `REPLY_SKIP`, `ANNOUNCE_SKIP`, and `NO_REPLY` guidance, while preserving those tokens for real coordination chatter. Add a focused WhatsApp group regression test covering authorized/always activation, explicit delivery wording, and forbidden skip-token outcomes.
