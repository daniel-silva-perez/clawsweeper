## Summary

Keep open. Current main still leaves the reported macOS supervisor stderr surface unbounded: launchd writes directly to `gateway.err.log`, while the existing `logging.maxFileBytes` rotation applies only to the structured Gateway JSONL logger, not launchd `StandardErrorPath`.

## What This Changes

The issue requests rate limiting or deduplication for repeated WebSocket heartbeat timeout lines and bounded rotation/max-size control for macOS `gateway.err.log`.

## Best Possible Solution

Add bounded retention for macOS launchd supervisor logs in the daemon layer, keeping it distinct from the structured JSONL logger. The robust end state is that `gateway.log` and `gateway.err.log` cannot grow past the chosen retention policy during long-running service operation, with install/restart tests and docs explaining that supervisor-log retention is separate from `logging.maxFileBytes`.

## Reproduction Assessment

Yes for the unbounded supervisor-log class: current source shows launchd stderr goes straight to `gateway.err.log` with no OpenClaw rotation hook. The exact heartbeat flood was not live-reproduced, and the exact message is not present in first-party source.

## Solution Assessment

Yes, bounded supervisor-log retention is the right first fix because it protects users from any noisy stderr producer, including the reported heartbeat line. Producer deduplication can be a follow-up once the upstream source of that exact message is identified.
