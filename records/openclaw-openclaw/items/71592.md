## Summary

Keep this issue open. Current main still advertises `/status` and `/compact` in TUI slash autocomplete, but the TUI command switch only handles `/gateway-status` locally and still sends unhandled slash text through `sendMessage(raw)`. In local embedded mode that path enters `EmbeddedTuiBackend` and starts an agent turn, so the reported local-mode fallthrough remains valid; the related open #51270 is adjacent `/model` work, not a fix for these commands.

## What This Changes

TUI local embedded mode should handle or explicitly reject advertised `/status` and `/compact` slash commands instead of sending them to the embedded model as ordinary user prompts.

## Best Possible Solution

Add a local embedded TUI command boundary for advertised shared slash commands. `/status` and `/compact` should either execute through reusable shared command logic or render an explicit local unsupported/status response, with focused regression tests proving they do not call the embedded agent path while gateway forwarding remains intact.

## Reproduction Assessment

Yes. The current-main static path is high-confidence: `/status` and `/compact` parse as commands, have no local `handleCommand()` cases, fall into `sendMessage(raw)`, and local mode routes `sendChat()` into `agentCommandFromIngress()`. Existing tests already prove the gateway-forwarding half for `/status`, but they do not cover the local-mode exception.

## Solution Assessment

No proposed patch is attached to this issue. The best fix is a narrow local-mode command handling or shared-command adapter change, not removing shared autocomplete globally or blocking all slash text.
