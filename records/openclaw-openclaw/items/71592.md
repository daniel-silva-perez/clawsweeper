## Summary

Keep open. Current main still matches the reported local TUI fallthrough: advertised shared commands include `/status` and `/compact`, but the TUI local command switch does not handle either command before its default branch sends the raw slash text into the embedded backend.

## What This Changes

The issue requests that TUI local embedded mode stop treating advertised `/status` and `/compact` slash commands as ordinary model prompts.

## Best Possible Solution

Add a narrow local embedded TUI command boundary for shared advertised commands. `/status` and `/compact` should execute reusable shared logic where feasible or render explicit local responses, with regression coverage proving they do not enter the embedded agent run path.

## Reproduction Assessment

Yes. The static current-main path is high confidence: slash input enters `handleCommand()`, the advertised shared commands lack local cases, the default branch calls `sendMessage(raw)`, and local mode forwards that to `agentCommandFromIngress()`.

## Solution Assessment

No patch is attached. The best fix is a narrow local-mode command handler or shared command adapter, not removing `/status` or `/compact` from the shared registry globally.

Fix the local embedded TUI slash-command fallthrough reported in https://github.com/openclaw/openclaw/issues/71592. In `openclaw tui --local`, advertised shared commands `/status` and `/compact` must not be recorded as user messages or sent into `EmbeddedTuiBackend.sendChat()` / `agentCommandFromIngress()`. Prefer a narrow local command boundary or shared command execution adapter that preserves gateway-mode behavior.
