## Summary

Keep this issue open. Current main has partial fixes for the hook and webhook-registry sides, but the BlueBubbles phone-format routing mismatch still appears current: inbound DM routing uses the sender handle verbatim, and generic binding matching is exact, so `+1...` and `1...` remain distinct and can still fall through to a catch-all binding. The related #54686 was closed unmerged and its own scope said it did not change hooks behavior or route matching.

## Best Possible Solution

Keep this issue open and implement a BlueBubbles-specific canonical handle/phone matching path for route bindings and conversation IDs, with regression coverage showing an inbound `1XXXXXXXXXX` handle matches a configured `+1XXXXXXXXXX` peer binding before the catch-all. Preserve the current hook session rebinding and pinned plugin HTTP-route registry behavior, and consider splitting webhook 404 and hook-context follow-ups only if maintainers want separate tracking.
