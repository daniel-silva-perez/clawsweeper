## Summary

Keep this issue open. Current main still routes bundled and forward-compatible GitHub Copilot Gemini model IDs to `openai-responses`; the shared OpenAI transport now appears to add Copilot IDE headers for `github-copilot`, but the default Gemini route still matches the reported unsupported `/responses` failure and docs/tests still preserve that behavior.

## What This Changes

The issue reports that GitHub Copilot Gemini/chat-completions-only models are routed to the unsupported Responses endpoint, while the completions override must preserve Copilot IDE auth headers.

## Best Possible Solution

Land a focused GitHub Copilot provider fix that routes known and forward-compatible Gemini Copilot model IDs to `openai-completions`, keeps Copilot IDE headers on that transport with regression coverage, updates the provider docs and changelog, and leaves the repeated fallback-cost warning as a separate follow-up.

Yes. A source-level reproduction is high confidence: resolving bundled or dynamic Copilot Gemini IDs on current main still selects `openai-responses`, which is the endpoint the reporter verified Copilot rejects for Gemini.

## Solution Assessment
