## Summary

Keep this issue open. Current main still auto-exposes update_plan for strict-agentic OpenAI/Codex GPT-5 runs and documents the mitigation, but it does not add a default cap or strict-agentic block/retry path for long consecutive update_plan-only tool bursts.

## Best Possible Solution

Keep this issue as the focused tracker for excessive consecutive update_plan-only bursts. The likely fix belongs in the embedded Pi/tool-loop layer: treat update_plan-only activity as non-progress even without assistant prose, add a small default cap or strict-agentic retry/block path for consecutive plan-tool-only calls, and add regression coverage for many update_plan calls before the next non-plan tool. Continue documenting tools.experimental.planTool=false as a mitigation, but do not treat it as the runtime fix.
