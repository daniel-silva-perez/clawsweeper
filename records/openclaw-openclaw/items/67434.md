## Summary

Current main still documents the mitigation and treats `update_plan` as non-progress only in the terminal planning-only retry path with visible assistant text; it does not prevent a long in-attempt sequence of consecutive `update_plan` tool calls before the next real tool, so the reported burst remains actionable.

## What This Changes

The issue asks OpenClaw to prevent excessive consecutive `update_plan`-only bursts in strict-agentic GPT-5/Codex runs and to make the plan-tool auto-exposure clearer to operators.

## Best Possible Solution

Add a bounded embedded-runner or tool-loop guard that treats consecutive `update_plan`-only activity as non-progress before it can consume hundreds of tool round trips, resets on the first non-plan tool, keeps the explicit `tools.experimental.planTool=false` mitigation, and documents any new user-facing cap or failure mode.
