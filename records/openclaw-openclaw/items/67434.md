## Summary

Keep this issue open. Current main documents and mitigates update_plan exposure, and the tool now returns compact results, but it still does not provide a default cap or strict-agentic block/retry path for long consecutive update_plan-only tool bursts before the next real tool call.

## Best Possible Solution

Keep this issue as the focused runtime tracker. The likely fix belongs in the embedded Pi/tool-loop layer: treat consecutive update_plan-only tool cycles as non-progress for strict-agentic GPT-5/Codex runs, add a small default cap or earlier steer/block path, reset the counter on meaningful non-plan progress, and add regression coverage for a long update_plan-only burst before the next real tool. Continue documenting tools.experimental.planTool=false as an operator mitigation, but do not treat it as the runtime fix.
