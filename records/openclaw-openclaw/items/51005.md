## Summary

Keep open. Current main still rejects the Brave `llm-context` `freshness`, `date_after`, and `date_before` paths, while Brave documents `freshness` and closed custom date ranges for the LLM Context endpoint; this PR is useful but needs maintainer follow-up because the branch targets an older provider file and its one-sided date range behavior still needs review.

## What This Changes

The PR removes Brave `llm-context` rejection guards for freshness/date filters, converts existing date filter parameters into Brave `freshness` ranges, adjusts cache keys, and updates docs plus web-search tests.

## Best Possible Solution

Port the fix into `extensions/brave/src/brave-web-search-provider.runtime.ts`, keep `ui_lang` rejected for `llm-context`, allow supported `freshness` and explicit date ranges to reach Brave, settle one-sided date range behavior deliberately, then update the Brave/web docs and focused regression tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Validate future date_after ranges:** `extensions/brave/src/brave-web-search-provider.ts:235-239`
  - body: The `date_after`-only branch builds `freshness=<date_after>to<today>`. If `date_after` is after today's UTC date, the generated range is reversed, so the request can fail at Brave instead of producing deterministic local behavior. Validate against the effective end date before calling the API.
  - confidence: 0.86
- **[P2] Do not cap date_before-only searches at 1970:** `extensions/brave/src/brave-web-search-provider.ts:240-241`
  - body: The `date_before`-only branch rewrites an open-ended before filter to `1970-01-01to<date_before>`, which silently drops older sources even though the tool parameter means results before that date. Use a documented open-ended form if Brave supports one, or make the boundary explicit instead of narrowing results.
  - confidence: 0.78
