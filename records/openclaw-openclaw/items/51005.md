## Summary

Keep open. The underlying bug still exists on current main and Brave's upstream docs support the requested `freshness` parameter for LLM Context, but this PR is stale against the active Brave plugin runtime and still has one-sided date-range semantics that need maintainer review before merge.

## What This Changes

The PR unblocks existing Brave `llm-context` `freshness`, `date_after`, and `date_before` web_search parameters, converts date filters into Brave `freshness` ranges, updates docs, and changes regression tests to assert query parameters on the Brave endpoint.

## Best Possible Solution

Port the fix into the active Brave plugin runtime, keep `ui_lang` rejected for `llm-context`, pass Brave-supported `freshness` and deliberate date-range values through the existing trusted endpoint path, and align Brave/web docs plus focused extension regression tests before merge.

- **[P2] Reject reversed date_after-only ranges:** `extensions/brave/src/brave-web-search-provider.ts:233-236`
  - body: When only `date_after` is set, the patch builds `freshness=<date_after>to<today>`. A future `date_after` produces a reversed range, so Brave receives an invalid request instead of OpenClaw returning a deterministic local validation error for the newly enabled path.
  - confidence: 0.83
- **[P2] Preserve open-ended date_before semantics:** `extensions/brave/src/brave-web-search-provider.ts:237-238`
  - body: `date_before` is documented as results before the given date, but this branch rewrites it to `1970-01-01to<date_before>`, silently excluding older sources. Historical searches in `llm-context` mode would get incorrectly scoped results.
  - confidence: 0.78
