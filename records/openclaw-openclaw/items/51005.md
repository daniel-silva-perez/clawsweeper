## Summary

Keep open. The underlying bug still exists on current main and the PR is aimed at the right Brave LLM Context capability, but the branch is stale against the active Brave plugin runtime and still needs maintainer/author follow-up before merge. The author’s note that the earlier Tavily PR was unrelated is supported by the related-item context; this remains Brave-specific work, not work superseded by Tavily.

## What This Changes

The PR unblocks Brave `llm-context` `freshness`, `date_after`, and `date_before` handling, converts date filters into Brave `freshness` ranges, updates web-search docs, and changes tests to assert the Brave LLM Context endpoint receives the expected query parameters.

## Best Possible Solution

Keep the useful Brave-specific change open, rebase it onto current main, implement the unblock in the active Brave runtime, keep `ui_lang` rejected for `llm-context`, settle one-sided range behavior deliberately, and update both the general web-search docs and the dedicated Brave docs with focused extension tests.

## Reproduction Assessment

Yes. On current main, configuring Brave with `mode: "llm-context"` and calling `web_search` with `freshness`, `date_after`, or `date_before` deterministically returns the local unsupported errors before any Brave network call.

## Solution Assessment

No. The intended direction is correct, but the current patch is not the best mergeable solution because it edits stale integration points and carries unresolved one-sided date-range behavior; the fix should land in the active Brave runtime with aligned docs and tests.
