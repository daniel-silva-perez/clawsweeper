## Summary

Current main still lacks a generic plugin-config env override or writable overlay before plugin config validation and activation. The issue should stay open because the discussion shows a policy-locked deployment problem that needs a core plugin configuration contract decision rather than ClawHub-only plugin work or stale cleanup.

## What This Changes

The issue requests an additive way for plugin configuration under `plugins.entries.<id>.config` to be overridden from declared environment variables and/or a writable overlay path so policy-locked sandboxes can reconfigure plugins without rebuilding images.

## Best Possible Solution

Add an additive host-side plugin-config resolution layer that can merge baked `plugins.entries.<id>.config` with an explicitly approved writable overlay and/or declared env mappings, validate the final merged object against the plugin manifest schema, expose only that final value through `api.pluginConfig`, and document precedence plus policy-locked sandbox behavior.

## Reproduction Assessment

Yes for the OpenClaw side: current-main source inspection shows `entry.config` is the only value resolved, validated, normalized, and passed into plugin APIs. The NemoClaw EACCES symptom would require that sandbox to live-reproduce, but it is not needed to confirm the missing OpenClaw seam.

## Solution Assessment

No current solution is in place. The best direction is an opt-in core resolution contract before manifest validation; asking every plugin to independently read arbitrary `process.env` would bypass the unified config and validation contract that this issue is trying to preserve.
