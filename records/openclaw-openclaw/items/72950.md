## Summary

Keep open. Current main still resolves plugin config from `plugins.entries.<id>.config`, validates that value directly, and passes it to plugin APIs without a generic env-var override or writable plugin-config overlay. The discussion also shows broader plugin impact, including the external openclaw-weixin comment, and #72944 was already closed in favor of this issue.

## What This Changes

Request an env-var override or writable overlay path for `plugins.entries.<id>.config` so policy-locked sandbox deployments can change plugin settings without rebuilding images.

## Best Possible Solution

Add an additive plugin-config resolution seam that merges baked `plugins.entries.<id>.config` with an explicitly supported writable overlay and/or plugin-declared env mappings, validates the merged result against the manifest schema, exposes only the final value through `api.pluginConfig`, and documents precedence plus sandbox operational behavior.

## Reproduction Assessment

Yes for the OpenClaw side. A focused current-main check is to set a literal `plugins.entries.demo.config` value and a matching env var: the schema, resolver, and loader still validate and pass `entry.config` only, while the issue body supplies the external NemoClaw read-only file symptom.

## Solution Assessment

No current solution is in place. The narrow maintainable direction is an additive, opt-in config resolution contract; making plugins read arbitrary env vars independently would bypass the unified config contract the issue is trying to preserve.
