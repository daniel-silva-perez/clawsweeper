## Summary

Keep open. Current main still has no generic plugin-config env override or writable-overlay merge before plugin config validation/activation; existing `${VAR}` substitution only works for placeholders already authored into the read-only config, not for overriding a baked literal or reading a writable overlay. This is a core plugin config contract question rather than ClawHub-only work because VISION.md explicitly points missing plugin API seams back to core design discussion.

## What This Changes

The issue requests an additive way to override or overlay `plugins.entries.<id>.config` so policy-locked sandbox deployments can change plugin settings without rebuilding images.

## Best Possible Solution

Add an additive plugin-config resolution seam that merges baked `plugins.entries.<id>.config` with a maintainer-approved writable overlay and/or explicitly declared env mappings, validates the final merged value against the plugin manifest schema, exposes only that final value through `api.pluginConfig`, and documents precedence plus NemoClaw/OpenShell operational behavior.

## Reproduction Assessment

Yes for the OpenClaw side: source inspection on current main shows `entry.config` is the only value resolved, validated, and passed into plugin APIs. The external NemoClaw EACCES symptom is supplied by the issue body/comments and would need that sandbox to live-reproduce, but it is not needed to confirm the missing OpenClaw config seam.

## Solution Assessment

No current solution is in place. The narrow maintainable direction is an additive, opt-in config resolution contract; telling plugins to read arbitrary `process.env` independently would bypass the unified config and manifest validation contract the issue is trying to preserve.
