## Summary

Keep open: current main still resolves, validates, and exposes only the baked `plugins.entries.<id>.config` object. The existing global `${VAR}` substitution helps when `openclaw.json` already contains placeholders, but it does not provide the requested plugin-declared env override or writable overlay path for policy-locked sandbox deployments, and follow-up comments show this affects more than one plugin environment.

## What This Changes

Request an env-var and/or writable-overlay path for `plugins.entries.<id>.config` so plugin configuration can be changed in policy-locked sandboxes without rebuilding the image.

## Best Possible Solution

Add an additive plugin-config resolution layer in core: start from baked `plugins.entries.<id>.config`, merge an explicitly approved writable overlay and/or plugin-declared env mappings, validate the final object against the plugin manifest schema, expose only the validated result through `api.pluginConfig`, and document precedence plus policy-locked sandbox behavior.

## Reproduction Assessment

Yes for the OpenClaw side: current-main source inspection shows `entry.config` is the only value resolved, validated, and passed into plugin APIs. The NemoClaw EACCES reproduction is supplied by the reporter and comments, but it is not required to confirm the missing OpenClaw seam.

## Solution Assessment

Partly: the host-side overlay/env-resolution direction is the maintainable shape if it stays explicit and schema-validated before plugin activation. The exact writable path, merge semantics, and precedence are still product/security decisions rather than a mechanical repair.
