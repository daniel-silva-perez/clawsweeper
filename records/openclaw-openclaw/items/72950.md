## Summary

Keep open. Current main still resolves plugin config from `plugins.entries.<id>.config` only, validates and passes that value directly to plugin APIs, and has no generic env-to-config mapping or writable plugin-config overlay. Existing `${VAR}` substitution and manifest env metadata are useful adjacent surfaces, but they do not override an already-authored baked plugin config value in a policy-locked sandbox. The related thread #72944 was closed in favor of this issue, and the openclaw-weixin comment supports this as a broader plugin-contract problem rather than cleanup-close material.

## Best Possible Solution

Keep this issue open as a plugin config contract/design task. The strongest path is an additive, opt-in mechanism such as manifest- or config-declared env mappings for selected plugin config keys, or a documented writable plugin-config overlay with clear precedence, schema validation, SecretRef handling, restart/reload semantics, and focused tests around `api.pluginConfig` resolution. If maintainers reject a core surface, document the policy-locked sandbox constraint and the supported plugin-author pattern explicitly.

## Risks / Open Questions

- Current docs/config help mention `plugins.entries.<id>.env` and `plugins.entries.<id>.apiKey`, but the inspected source schema/type do not accept those top-level plugin entry fields. Maintainers should resolve that drift while designing this surface.
- `OPENCLAW_CONFIG_PATH` can point the whole config at another file, but that is not a per-plugin overlay or env override and would require replacing the active config source rather than merging runtime-safe plugin config over baked defaults.
