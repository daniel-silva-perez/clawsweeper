## Summary

Keep open. Current main still has no generic plugin-config env override or writable-overlay merge before plugin config validation and activation, and the discussion shows this affects policy-locked plugin deployments beyond one observability setup. This is a core plugin configuration contract question, not a ClawHub-only plugin request, because the missing capability is the host-side seam that would let plugins receive a merged, validated config.

## What This Changes

Requested support for environment-variable and/or writable-overlay sources for `plugins.entries.<id>.config` so policy-locked sandbox deployments can change plugin config without rebuilding images.

## Best Possible Solution

Add an additive plugin-config resolution layer in core: merge baked `plugins.entries.<id>.config` with a maintainer-approved writable overlay and/or explicit env mappings, validate the final object against the plugin manifest schema, expose only that final value through `api.pluginConfig`, and document precedence plus policy-locked sandbox behavior.

## Reproduction Assessment

Yes for the OpenClaw side: source inspection on current main shows `entry.config` is the only value resolved, validated, normalized, and passed into plugin APIs. The external NemoClaw EACCES symptom is supplied by the issue body/comments and would need that sandbox to live-reproduce, but it is not needed to confirm the missing OpenClaw seam.

## Solution Assessment

No current solution is in place. The best direction is an opt-in host-side config resolution contract before manifest validation; telling plugins to independently read arbitrary `process.env` would bypass the unified config and validation contract this issue is trying to preserve.
