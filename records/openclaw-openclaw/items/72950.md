## Summary

Keep open. Current main still passes arbitrary plugin config to plugins from `plugins.entries.<id>.config` only, with no generic env-to-config mapping and no writable plugin-config overlay. General `${VAR}` substitution exists for authored config string values, and the bundled diagnostics OpenTelemetry path has its own env fallbacks, but neither solves the reported case where a baked plugin config value needs an opt-in runtime env override or writable overlay in a policy-locked sandbox. The related context also points to another affected external plugin, so this remains a valid plugin contract/design issue rather than cleanup-close material.

## Best Possible Solution

Keep this issue open as a plugin config contract/design task. The best path is an additive, opt-in mechanism such as manifest- or config-declared env mappings for selected plugin config keys, or a documented writable overlay with clear precedence, schema validation, SecretRef handling, restart/reload semantics, and tests around `api.pluginConfig` resolution. If maintainers reject a core surface, document the policy-locked constraint and the supported plugin-author pattern explicitly.

## Risks / Open Questions

- Generated help/docs mention `plugins.entries.<id>.env` and `apiKey`, but the inspected zod source schema and TypeScript type do not accept those fields. Maintainers should decide whether that documentation is stale or an incomplete feature.
- Generic `${VAR}` substitution can help when the baked config is authored as a placeholder, but it does not provide fallback/default semantics or override an already-authored literal plugin config value, which is the core policy-locked sandbox complaint.
