## Summary

Keep open. Current main still has the reported Slack tool exposure mismatch and the requested high-privilege Slack admin/app-manifest actions are not implemented, so this is not a conservative cleanup close. The issue is valid but needs maintainer/security direction before implementation.

## What This Changes

Expose Slack actions to Claude Code/ToolSearch and add opt-in Slack admin operations with user/app-configuration token plumbing.

## Best Possible Solution

Define the Claude/ACPX tool-exposure contract first, then split the work into a Slack-owned bridge or skill-loading fix plus a separate disabled-by-default admin action group with scoped token resolution, SecretRef-compatible config, scope documentation, and focused Slack/MCP tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: current main advertises a `slack` skill, registers Slack as a channel plugin, and exposes no Slack callable tool through the ACPX/OpenClaw-tools MCP bridge used by Claude Code.

## Solution Assessment

Unclear. The proposed direction is plausible, but the narrowest maintainable fix depends on maintainer choice between bridging Slack actions, suppressing unreachable skill guidance, or both; the admin operations should be reviewed as a separate opt-in feature.
