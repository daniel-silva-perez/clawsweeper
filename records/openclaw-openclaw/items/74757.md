## Summary

Keep open. Current main still has the reported mismatch: Slack actions are implemented as a Slack channel message-action surface, while the ACPX/OpenClaw-tools MCP bridge does not expose a Slack callable tool to Claude Code; the requested Slack admin and app-manifest actions are also absent. The discussion correctly narrows this to maintainer architecture and security direction rather than a conservative cleanup close.

## What This Changes

The issue requests exposing Slack message actions to Claude Code/ToolSearch and adding opt-in Slack admin/app-management actions such as channel creation, invites, member lookup, member listing, app manifest create/update, and channel archive.

## Best Possible Solution

Define the Claude/ACPX tool-exposure contract first, then split implementation into a Slack-owned bridge or skill-loading fix plus a separate opt-in admin action group. The admin path should use least-privilege token resolution, SecretRef-compatible config, clear scope documentation, tests across Slack action runtime and MCP/tool discovery, and stay disabled unless explicitly configured.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: current main advertises `slack` in the bundled skill, registers Slack as a channel plugin, and exposes no Slack tool through the OpenClaw-tools MCP bridge used by ACPX/Claude Code.

## Solution Assessment

Unclear. The proposed direction is plausible, but the narrowest maintainable fix depends on maintainer choice between bridging Slack actions, hiding unreachable skill guidance, or doing both; the high-privilege admin operations should be reviewed as a separate opt-in feature.
