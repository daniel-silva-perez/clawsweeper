## Summary

Keep open. Current main still has the reported mismatch: Slack is a bundled channel plugin with channel message actions and prompt skill text, but the Claude/ACPX OpenClaw MCP bridge does not expose a Slack callable tool, and the requested Slack admin/app-manifest action group is still absent. The reporter's follow-up correctly narrows this to maintainer architecture and security decisions rather than a cleanup-close candidate.

## What This Changes

The issue requests exposing Slack actions to Claude Code through the OpenClaw tool/MCP surface and adding opt-in Slack admin actions for channel, user, and app-manifest management.

## Best Possible Solution

Define the Claude/ACPX Slack tool-exposure contract first, then split implementation into a Slack-owned bridge or skill-loading fix and a separate disabled-by-default Slack admin action group. The admin path should be SecretRef-compatible, least-privilege, clearly documented with Slack scopes, and covered by Slack action-runtime/config tests plus ACP/MCP discovery tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: Slack prompt guidance is present, Slack actions are channel-message adapters, the OpenClaw tools MCP bridge returns only cron, and Claude CLI only allows `mcp__openclaw__*` tools.

## Solution Assessment

Unclear. The proposed direction is plausible, but the best maintainable fix needs maintainer choice between MCP bridging, skill visibility gating, or both; the admin operations should be reviewed and implemented as a separate opt-in security-sensitive feature.
