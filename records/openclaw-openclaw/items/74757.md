## Summary

Keep open. Current main partially overlaps the token-plumbing request by supporting Slack `userToken`, but the two central gaps remain: Slack channel actions are still not exposed as a Claude/ACPX-callable OpenClaw MCP tool, and the requested Slack admin/app-manifest actions plus `appConfigToken` surface are still absent. The reporter's follow-up and the prior ClawSweeper comment correctly frame this as maintainer architecture and security work, not conservative cleanup-close material.

## What This Changes

The issue requests making Slack channel actions callable from Claude Code/ToolSearch and adding opt-in Slack admin/app-manifest actions backed by additional token configuration.

## Best Possible Solution

Define the Claude/ACPX Slack tool-exposure contract first, then split implementation into a Slack-owned bridge or skill-loading fix and a separate disabled-by-default Slack admin action group with SecretRef-compatible tokens, least-privilege Slack scopes, owner/session gating, docs, and focused Slack plus MCP discovery tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: Slack prompt guidance is present, Slack actions are channel-message adapters, the OpenClaw tools MCP bridge has no Slack tool, and Claude Code is limited to OpenClaw MCP-prefixed tools.

## Solution Assessment

Unclear. The proposed direction is plausible, but the best maintainable fix depends on maintainer choice between MCP bridging, skill visibility gating, or both; the admin operations should be reviewed as a separate opt-in security-sensitive feature.
