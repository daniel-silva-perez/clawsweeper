## Summary

Keep open. Current main still matches the reported gaps: Slack remains a channel plugin with skill text that names a `slack` tool, while the Claude Code MCP path only exposes selected `mcp__openclaw__*` tools and the current OpenClaw tools bridge does not include Slack. The requested Slack admin and app-manifest actions are also absent, so this is valid but needs maintainer and security direction before implementation.

## What This Changes

Request to expose Slack actions to Claude Code/ToolSearch and add opt-in Slack admin/app-manifest actions with admin-token configuration.

## Best Possible Solution

Settle the Claude Code exposure contract first, then implement Slack admin as a separate, explicitly opt-in action group with documented token resolution, SecretRef-compatible `appConfigToken` or equivalent app-configuration credentials, least-privilege scope guidance, owner-aware gating, and focused tests for the MCP/tool bridge plus Slack action runtime.

## Reproduction Assessment

Yes. The static reproduction path is high confidence: inspect the Claude backend allowlist, the `openclaw-tools` MCP resolver, the Slack channel entry, and the Slack action dispatch tables on current main; together they show the tool is not Claude Code-discoverable and the requested admin actions are not implemented.

## Solution Assessment

Unclear. The issue's direction is plausible, but the best fix needs maintainer choice between an MCP bridge, harness-specific skill filtering, or both; the admin action work should be split behind a least-privilege, off-by-default design.
