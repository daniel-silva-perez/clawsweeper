## Summary

Keep open. Current main still leaves the core mismatch unresolved: Slack is implemented as a bundled channel plugin with message actions, while the ACPX/OpenClaw MCP bridge does not expose a Slack callable tool to Claude Code, and the requested Slack admin/app-manifest operations are still absent. The reporter's follow-up correctly narrows this to an architecture and security decision rather than a conservative cleanup close.

## What This Changes

Expose Slack channel actions to Claude/ACPX tool discovery and add opt-in Slack admin/app-management actions with token plumbing.

## Best Possible Solution

Define the Claude/ACPX tool-exposure contract first, then split the work into a Slack-owned bridge or skill-loading fix plus a separate opt-in admin action group. The admin path should use least-privilege token resolution, SecretRef-compatible config, clear scope documentation, tests across Slack action runtime and MCP/tool discovery, and should remain disabled unless explicitly configured.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: current main loads Slack as a channel message-action provider and prompt skill, while the OpenClaw MCP bridge exposes only cron and the requested Slack admin endpoints/config are absent.

## Solution Assessment

Unclear. The proposed direction is plausible, but the narrowest maintainable solution depends on maintainer choice between MCP bridging, skill visibility changes, or both; the admin operations should be reviewed as a separate opt-in security-sensitive feature.
