## Summary

Keep open. Current main still supports Slack as a bundled channel plugin with channel message actions, but it does not expose a callable Slack tool through the Claude/ACPX MCP surfaces, and the requested Slack admin/app-manifest actions are not implemented. The reporter's follow-up correctly identifies this as an architecture/security decision rather than a conservative cleanup close.

## What This Changes

Request to expose Slack channel actions to Claude Code/ToolSearch and add opt-in Slack admin actions/token plumbing for channel, app, invite, and user administration.

## Best Possible Solution

Define the Claude/ACPX exposure contract first, then split implementation into a Slack-owned callable bridge or skill-loading fix plus a separate opt-in Slack admin action group. The admin path should be SecretRef-compatible, documented with least-privilege scopes, disabled by default, and covered by Slack runtime plus MCP/tool bridge tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: current main loads Slack as a channel plugin/action adapter while the OpenClaw MCP bridge and Claude allowed-tools configuration do not expose a Slack callable tool, and exact searches show the requested admin/app-manifest endpoints are absent.

## Solution Assessment

Unclear. The requested direction is plausible, but the best fix depends on maintainer choice between MCP bridging, conditional skill loading, or both; the admin actions should be a separate opt-in security-reviewed change rather than bundled into an automatic cleanup PR.
