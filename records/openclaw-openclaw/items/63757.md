## Summary

Keep this issue open. Current main still handles Telegram reaction updates, applies authorization and notification filtering, and converts added emoji reactions into text-only system events. I did not find a stable structured plugin or internal hook that downstream integrations can consume for message id, emoji, add/remove operation, sender, account, session, or authorization context.

## What This Changes

Expose Telegram message reactions as a stable structured downstream hook/event for managed hooks or plugins, instead of requiring local core patches.

## Best Possible Solution

Add an additive typed reaction event seam after existing channel authorization/filtering, preferably a generic `message_reaction` plugin hook and matching `message:reaction` internal hook. The payload should include channel/account/session context, target message id, emoji, add/remove operation when known, sender metadata, authorization outcome, and bounded raw metadata; feedback storage should remain plugin or hook-owned.
