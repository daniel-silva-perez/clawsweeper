## Summary

Keep this issue open. Current main still receives Telegram `message_reaction` updates, authorizes and filters them, then exposes them only as human-readable system-event text. I did not find a stable structured internal hook or typed plugin hook carrying target message id, emoji, add/remove operation, sender metadata, account/session context, or authorization outcome. The prior ClawSweeper comment matches current main.

## Best Possible Solution

Keep this issue open. Add an additive typed reaction event seam after Telegram's existing authorization and notification filtering path, ideally as a cross-channel `message_reaction` plugin hook and/or `message:reaction` internal hook. The payload should include channel/account context, session or conversation context when available, Telegram chat id, target message id, emoji, add/remove operation, sender metadata, authorization result, and bounded raw metadata. Add Telegram handler tests, hook contract tests, docs for public plugin/internal hook behavior, and keep feedback recording as a hook/plugin built on the seam rather than hardcoding feedback storage into the Telegram channel.
