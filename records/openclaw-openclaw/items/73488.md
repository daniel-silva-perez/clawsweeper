## Summary

Current main still omits per-conversation identity from the provider plugin `createStreamFn` context, and the issue is paired with open PR #73492 using closing syntax. It should stay open until that implementation, or an equivalent narrow fix, is reviewed and merged or rejected.

## What This Changes

The issue requests optional `sessionId` and `sessionKey` fields in `ProviderCreateStreamFnContext`, propagated from conversation-aware runner paths so provider plugins can scope state by conversation.

## Best Possible Solution

Land a reviewed version of #73492 or an equivalent narrow API change that adds optional `sessionId` and `sessionKey` to provider stream context, forwards them from conversation-bearing paths, leaves one-off completions unset, and adds focused propagation coverage.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main exposes no `sessionId` or `sessionKey` to provider `createStreamFn`, while the runner path already has those values in scope before registering the provider stream.

## Solution Assessment

Yes. Optional context fields passed through `registerProviderStreamForModel` are the narrowest backward-compatible fix and avoid forcing provider plugins to infer conversation identity from prompts, paths, or first messages.
