## Summary

Keep open. Current main still does not expose per-conversation identity in the provider plugin createStreamFn context, and open PR #73492 already references this issue with closing syntax as the implementation candidate. The Glueclaw transcript-bleed report is relevant evidence of a real plugin API gap rather than an external-only support request.

## What This Changes

Expose per-conversation identity fields in provider plugin createStreamFn context so provider plugins can scope runtime state by OpenClaw conversation.

## Best Possible Solution

Land a reviewed version of #73492, or an equivalent narrow core change, that adds optional sessionId and sessionKey to ProviderCreateStreamFnContext, forwards them from conversation-bearing runner paths, leaves one-off completion paths unset, and adds focused propagation coverage for the plugin API contract.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main has sessionId/sessionKey available in conversation-bearing paths, but the createStreamFn context type and registerProviderStreamForModel call chain cannot deliver those values to provider plugins.

## Solution Assessment

Yes. Optional context fields plumbed through registerProviderStreamForModel are the narrowest backward-compatible fix; provider plugins can consume sessionKey when present without changing existing plugins that ignore the new fields.
