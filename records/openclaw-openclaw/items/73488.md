## Summary

Keep open. Current main still omits per-conversation identity from `ProviderCreateStreamFnContext` and from the `registerProviderStreamForModel` context object, and the issue is already paired with open PR #73492 using closing syntax, so it should remain open until that implementation is reviewed and merged or rejected.

## What This Changes

The issue requests adding optional `sessionId` and `sessionKey` fields to provider plugin `createStreamFn` context so provider plugins can scope per-conversation state.

## Best Possible Solution

Land a reviewed version of #73492 or an equivalent narrow change that adds optional `sessionId` and `sessionKey` to the provider stream context, forwards them from conversation-bearing paths, leaves one-off completions unset, and adds focused propagation coverage.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main exposes no `sessionId` or `sessionKey` to `createStreamFn`, while the main runner paths already have those values in scope before registering the provider stream.

## Solution Assessment

Yes. Optional context fields passed through `registerProviderStreamForModel` are the narrowest backward-compatible fix and avoid requiring provider plugins to infer conversation identity from brittle side channels.
