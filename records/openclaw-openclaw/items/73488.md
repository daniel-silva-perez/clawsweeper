## Summary

Keep open. Current main still does not expose per-conversation identity on the provider plugin `createStreamFn` context, and this issue is paired with open PR #73492 using closing syntax, so the right maintainer action is to review that PR rather than close the issue early or queue duplicate repair work.

## What This Changes

Provider plugin authors request optional `sessionId` and `sessionKey` fields on `ProviderCreateStreamFnContext`, plus propagation through conversation-bearing provider stream registration paths.

## Best Possible Solution

Land a reviewed version of #73492 or an equivalent narrow patch that adds optional `sessionId` and `sessionKey` to the provider stream context, forwards them from main turn, compaction, and BTW conversation paths, leaves one-off completions unset, and adds focused propagation coverage.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence on current main: register a provider `createStreamFn` and inspect its context during a conversation-bearing path, and it cannot receive `sessionId` or `sessionKey` even though those values are already in scope upstream.

## Solution Assessment

Yes. Optional context fields passed through `registerProviderStreamForModel` are the narrowest backward-compatible solution; it avoids brittle provider-side inference while preserving existing plugins that ignore the new fields.
