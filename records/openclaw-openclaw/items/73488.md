## Summary

Keep open. Current main still does not expose per-conversation identity on the provider plugin createStreamFn context, and open PR #73492 already references this issue as the implementation candidate. The downstream Glueclaw transcript-bleed report is relevant evidence of a real plugin API gap rather than an external-only workaround request.

## What This Changes

Add optional sessionId and sessionKey fields to ProviderCreateStreamFnContext and plumb them through provider stream registration so provider plugins can scope state per conversation.

## Best Possible Solution

Review and land #73492, or an equivalent narrow core change, that adds optional sessionId/sessionKey to the provider createStreamFn context, forwards them from conversation-bearing runner paths, leaves one-off completion paths unset, and adds focused propagation coverage for the plugin API contract.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main has sessionId and sessionKey available in the runner and BTW paths, but the createStreamFn context type and registerProviderStreamForModel call chain cannot deliver them to provider plugins.

## Solution Assessment

Yes. Optional context fields plumbed through registerProviderStreamForModel are the narrowest backward-compatible fix; provider plugins can prefer sessionKey when present without breaking existing plugins that ignore the new fields.
