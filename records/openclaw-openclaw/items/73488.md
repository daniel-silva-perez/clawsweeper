## Summary

Keep open. Current main still does not expose per-conversation identity to provider plugin `createStreamFn`, and an open linked implementation PR already uses closing syntax, so this issue should remain open until that PR is reviewed and merged or rejected.

## What This Changes

The issue requests optional `sessionId` and `sessionKey` fields in provider plugin `createStreamFn` context and propagation from conversation-bearing agent paths.

## Best Possible Solution

Land a reviewed version of PR #73492 or an equivalent narrow change that adds optional `sessionId` and `sessionKey` to the provider stream context, forwards them from conversation-bearing paths, leaves one-off completions unset, and adds focused propagation coverage.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main exposes no `sessionId` or `sessionKey` to `createStreamFn`, while the agent runner and compaction params already carry those values before provider stream registration.

## Solution Assessment

Yes. Optional fields on the existing provider stream context are the narrowest backward-compatible fix; they avoid asking provider plugins to infer conversation identity from prompts, filesystem locks, or first-message hashes.
