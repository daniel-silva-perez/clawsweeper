## Summary

Keep this PR open. Current main still has the close/response race the PR targets: `onClose` rejects immediately while `client.request(...)` is still awaiting, and there is no `requestInFlight` flag or normal-close deferral equivalent on main. Existing main tests cover formatted close errors, wrapper timeout handling, and teardown ordering, but not the same-turn `cron.list` response followed by normal close that this PR adds coverage for. The PR diff described in the provided context is narrow and does not introduce an obvious security or supply-chain concern.

## Best Possible Solution

Keep this PR open as a focused gateway bug-fix candidate. The best path is to rebase it onto current main, adapt the `requestInFlight`/normal-close deferral to the current `stopGatewayClient()` flow, keep the added close-race regression coverage, and verify abnormal disconnects still surface `formatGatewayCloseError(...)` details.
