## Summary

Close as not reproducible and not a merge candidate. Current main uses UTC ISO timestamps as the on-disk session contract and localizes/parses them at read/display time; the linked bug was already closed not planned after maintainer review of that behavior. The PR changes storage semantics to local-offset strings and still has correctness/security concerns in the wrapper, so merging it is not the right cleanup path.

## What This Changes

This PR adds a local-offset timestamp formatter and wraps or updates session transcript write paths so new JSONL headers and entries use ISO strings with local timezone offsets instead of `...Z` UTC strings.

## Best Possible Solution

Close this PR and keep UTC ISO timestamps as the transcript storage contract. If raw transcript inspection remains confusing, improve docs or viewer/CLI display wording around local-time rendering rather than changing JSONL persistence semantics.

## Reproduction Assessment

Yes. A focused source check plus a `TZ=Asia/Shanghai` Node check shows current main stores a correct UTC instant and displays/parses it back into local time; the reported 8-hour shift is a raw-string expectation mismatch, not a current-main persistence bug.

## Solution Assessment

No. The PR changes the on-disk contract instead of keeping UTC-at-rest with localized display, and its wrapper still misses upstream recovery rewrites and malformed timestamp handling.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Handle recovered headers after open-time rewrites:** `src/sessions/local-session-timestamps.ts:65-66`
  - body: When `SessionManager.open()` repairs an empty or corrupt transcript, upstream writes a fresh header with `new Date().toISOString()`, calls `_rewriteFile()`, and sets `flushed = true` before returning. This wrapper then calls `normalizePendingHeader()`, but the `flushed` guard returns immediately, so the recovered transcript still keeps a `...Z` header despite the PR's local-offset guarantee.
  - confidence: 0.91
- **[P3] Keep timestamp normalization best-effort:** `src/sessions/local-session-timestamps.ts:61`
  - body: `normalizeEntryTimestamp()` calls `formatLocalSessionTimestamp()` on persisted values, and `toDate()` throws for malformed timestamp strings. Existing transcript loading skips malformed JSON and tolerates imperfect data, so this wrapper can turn a bad historical timestamp into a branch/rewrite failure until the JSONL is manually repaired.
  - confidence: 0.79

## Security Review

Status: needs_attention

Summary: The diff adds no new dependency or privilege surface, but it introduces a low availability concern when malformed persisted timestamps are normalized.

Concerns:

- **[low] Malformed timestamp can abort session normalization:** `src/sessions/local-session-timestamps.ts:61`
  - body: The new normalizer throws `RangeError` for unparsable persisted timestamp values; paths that normalize existing `fileEntries` can fail repeatedly for a corrupted or user-edited transcript instead of leaving the bad value untouched.
  - confidence: 0.82
