## Summary

Keep this PR open. Current main still lacks native outbound WhatsApp mention metadata, and this PR is a relevant implementation candidate for the open WhatsApp mention feature, but it should not merge as-is because the branch is stale against current send-result behavior and the new resolver still has concrete LID edge-case bugs.

## What This Changes

This PR adds WhatsApp outbound @mention resolution for group text and media captions, wiring participant metadata from the WhatsApp monitor into send paths, with docs, changelog, and resolver/send API tests.

## Best Possible Solution

Port the mention resolver onto current main's WhatsApp send-result implementation, keep the scope limited to outbound group mention metadata for #39879, fix the LID priority/collision cases with tests, preserve the docs/changelog credit, and leave reply-target media injection to a separate follow-up.

## Reproduction Assessment

Yes. On current main, a focused unit call through `createWebSendApi().sendMessage("120363...@g.us", "hi @+15551234567")` would send a plain `{ text }` payload without `mentions`; the PR findings are reproducible with pure `resolveWhatsAppOutboundMentions` inputs that include both phone and LID participant fields or colliding digit keys.

## Solution Assessment

No, not as currently proposed. The feature boundary is appropriate for the WhatsApp plugin, but the branch needs to be rebased onto current send-result behavior and the resolver should prefer explicit LID data and handle ambiguous digit collisions before merge.

Full review comments:

- **[P2] Preserve the current send-result contract:** `extensions/whatsapp/src/inbound/send-api.ts:59`
  - body: This branch is based before the current `WhatsAppSendResult` flow: `sendMessage` returns only `{ messageId }`, while current main normalizes `providerAccepted`/`messageIds` and combines the separate audio text send. A rebase that keeps this shape would break auto-reply delivery tracking, so apply the mention wiring to the current `normalizeWhatsAppSendResult`/`combineWhatsAppSendResults` implementation.
  - confidence: 0.92
- **[P2] Prefer explicit LID participants when selecting mention JIDs:** `extensions/whatsapp/src/inbound/outbound-mentions.ts:129-136`
  - body: `chooseMentionJid` normalizes `id` before `lid`, so a participant shaped as `{ id: "15551234567@s.whatsapp.net", lid: "277038292303944@lid" }` records the LID digits with a phone JID target. That makes `@277038292303944` resolve to the phone JID and fail in LID-addressed groups; prefer a valid `lid` JID over a phone `id` when both are present.
  - confidence: 0.86
- **[P3] Disambiguate phone and LID digit collisions:** `extensions/whatsapp/src/inbound/outbound-mentions.ts:213`
  - body: The resolver always does `byPhone.get(digits) ?? byLid.get(digits)`, so if one participant's phone number equals another participant's LID digits, a visible LID token resolves to the phone user. Since this PR supports LID group tokens, preserve sender intent, for example by using the `+` prefix for phone lookup and bare LID tokens for `byLid` when both maps contain the key.
  - confidence: 0.74
