## Summary

Keep open. Current main has adjacent lifecycle and cleanup improvements, but it still does not fully implement the reported user-visible timeout state: normal Discord direct raw-runner failures still use generic copy, committed side effects before provider failure do not get an explicit partial-success warning, and diagnostic stuck-session recovery remains observe-only for active work by default.

## What This Changes

The issue requests clearer Discord agent-session health and user-facing timeout behavior after an embedded agent turn times out after possible side effects.

## Best Possible Solution

Add a shared terminal timeout/session-health transition that either marks or rotates the affected session before later routing, and surface channel-appropriate copy that says the run timed out and previous side effects may already have completed. Keep the existing conservative active-run recovery unless maintainers explicitly choose an abort/rotate policy for active work.

## Reproduction Assessment

Yes for source-level reproduction: current tests and code still show generic direct Discord raw-runner failure copy and observe-only active stuck-session recovery. A live Discord/Claude CLI timeout run would help tune the final policy, but it is not required to see the remaining source-level gap.

## Solution Assessment

Unclear because no patch is proposed. Current main's lifecycle backstop and cleanup are useful partial mitigations, but they are not the full fix for partial-success timeout copy or active-session routing policy.
