## Summary

Keep open. Current main has adjacent cleanup that reduces stuck processing after runtime failures, but it still does not fully satisfy this report: active stuck-session recovery remains observe-only by default, normal Discord direct pre-reply failures still use generic copy, and docs describe active embedded/reply/lane work as warning-only unless a maintainer chooses a stronger recovery policy.

## What This Changes

The issue asks OpenClaw to make Discord-routed agent timeout state unambiguous after side effects may have completed, instead of leaving a wedged session routable and surfacing only generic failure text.

## Best Possible Solution

Add a shared terminal timeout/session-health transition for embedded agent failures that marks or rotates the affected session before later routing, and surface channel-appropriate timeout copy that warns previous side effects may have completed. Keep Discord regression coverage, but avoid a Discord-only workaround or unconditional active-run abort without maintainer policy approval.

## Reproduction Assessment

Yes for source-level reproduction: current tests still assert generic Discord direct failure copy, and diagnostic recovery still skips active runs by default. A live Discord plus Claude CLI timeout run would strengthen policy selection, but the current-main gap is visible in source and tests.

## Solution Assessment

Unclear because no patch is proposed and the remaining fix depends on maintainer policy. The narrow maintainable direction is a shared session-health transition plus timeout/partial-success copy after deciding whether active side-effectful runs should be aborted, rotated, or marked dead.
