## Summary

Keep open. Current main has diagnostic stuck-session detection and a conservative recovery helper, but active embedded runs, active reply work, and active session-lane tasks still remain warning-only by default, while Discord direct-chat pre-reply runner failures still surface the same generic failure copy. The related open #73581 overlaps on general stuck-lane recovery, but this issue still has a distinct Discord/user-facing timeout and partial-success ambiguity that is not fully covered or implemented.

## What This Changes

The issue asks OpenClaw to make Discord-routed agent timeouts unambiguous by marking or resetting wedged sessions and by distinguishing partial-success timeout failures from total failures.

## Best Possible Solution

Add a generic agent/session health outcome for terminal embedded-agent timeouts that can mark the affected session failed or rotate it before later routing, and surface timeout-specific copy that warns previous side effects may have completed. Keep the fix in the shared agent/session runner path with Discord direct/group coverage rather than adding a Discord-only workaround.

## Reproduction Assessment

Yes for a source-level reproduction: the existing tests already show Discord direct runner failures receive the generic copy, and the recovery tests show active embedded runs are observe-only by default. A full live Discord/Claude CLI 900s reproduction would still be useful before landing behavior that resets or aborts live sessions.

## Solution Assessment

Unclear until maintainers choose the product policy. The narrowest maintainable fix is likely a shared timeout/session-health transition plus channel-appropriate user copy, not a Discord-only routing patch or unconditional active-run abort.
