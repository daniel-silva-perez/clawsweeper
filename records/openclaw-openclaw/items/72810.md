## Summary

Keep open. Current main has added lifecycle backstops and conservative stuck-session recovery, but it still does not fully provide the requested unambiguous Discord timeout outcome or partial-success user copy: active work remains observe-only by default, Discord direct pre-reply failures still use the generic failure text, and the broader recovery policy is still overlapped by open #73581.

## What This Changes

The issue requests that Discord-routed agent sessions become clearly failed, reset, or freshly routed after fatal embedded-agent timeouts, with user-facing copy that distinguishes late failure after side effects from total failure.

## Best Possible Solution

Add a shared terminal timeout/session-health outcome that can mark or rotate the affected session before later routing, and surface channel-appropriate timeout copy warning that previous side effects may already have completed. Keep the recovery policy generic across agent/session runners, with Discord coverage for direct and group paths.

## Reproduction Assessment

Yes for a source-level reproduction: current tests and code show Discord direct runner failures still receive generic copy, and recovery tests show active work remains observe-only by default. A live Discord/Claude CLI 900s reproduction would still be useful before changing active-run abort or rotation policy.

## Solution Assessment

Unclear until maintainers choose the recovery policy. The narrowest maintainable direction is a shared session-health transition plus timeout-specific channel copy, not a Discord-only workaround or unconditional active-run abort.
