## Summary

Keep open. Current main has adjacent reliability cleanup that reduces stuck processing after runtime failures, but it still does not fully satisfy this Discord timeout report: active stuck-session recovery remains observe-only by default, normal Discord direct pre-reply runner failures still use generic copy, and the docs still describe active embedded/reply/lane work as warning-only unless a maintainer chooses a stronger recovery policy.

## What This Changes

The issue requests unambiguous Discord session-health handling and partial-success-aware user feedback after embedded agent timeouts or late pre-reply failures.

## Best Possible Solution

Add a shared terminal timeout/session-health transition for embedded agent failures that marks or rotates the affected session before later routing, and surface channel-appropriate timeout copy that warns previous side effects may have completed. Keep the fix generic across chat channels, with Discord regression coverage, rather than adding a Discord-only workaround.

## Reproduction Assessment

Yes for source-level reproduction: current tests still assert generic Discord direct failure copy, and diagnostic recovery still skips active work by default. A live Discord plus Claude CLI timeout run would strengthen policy selection, but the current-main gap is visible in source, tests, and docs.

## Solution Assessment

Unclear because no patch is proposed and the remaining behavior depends on maintainer runtime policy. The narrow maintainable direction is a shared session-health transition plus timeout/partial-success copy after deciding whether active side-effectful runs should be aborted, rotated, or marked dead.
