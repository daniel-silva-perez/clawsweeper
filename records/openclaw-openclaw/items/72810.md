## Summary

Keep open. Current main still lacks the requested unambiguous Discord timeout and partial-success state: Discord direct pre-reply failures still use generic copy, diagnostics still avoid aborting active work by default, and the related stuck-session recovery work remains under maintainer review rather than merged proof.

## What This Changes

The issue requests that Discord-routed agent sessions become clearly failed, reset, or freshly routed after fatal embedded-agent timeouts, with user-facing copy that accounts for possible completed side effects.

## Best Possible Solution

Add a shared terminal timeout/session-health outcome that can mark or rotate the affected session before later routing, and surface channel-appropriate timeout copy warning that previous side effects may already have completed. Keep Discord coverage in tests, but implement the health transition in the shared agent/session runner path.

## Reproduction Assessment

Yes for source-level reproduction: current tests and code show Discord direct runner failures still receive generic copy, and recovery tests show active work remains observe-only by default. A live Discord/Claude CLI timeout run would still be useful before changing active-run abort or rotation policy.

## Solution Assessment

Unclear until maintainers choose the recovery policy. The narrowest maintainable direction is a shared session-health transition plus timeout-specific channel copy, not a Discord-only workaround or unconditional active-run abort.
