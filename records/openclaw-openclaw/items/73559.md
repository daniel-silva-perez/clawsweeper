## Summary

Keep open. Current main still matches the reported OpenAI Codex OAuth bearer propagation gap: the run resolves and forwards an API key, but the boundary-aware `openai-codex-responses` fallback returns the transport directly without injecting it. The linked PR #73588 is open and unmerged with an explicit closing reference, so the best maintainer path is to review/land or reject that PR rather than closing this issue first.

## Best Possible Solution

Keep this issue open and review #73588 or an equivalent fix. The likely code path is `src/agents/pi-embedded-runner/stream-resolution.ts`: wrap the boundary-aware fallback with the same resolved API key and run-signal handling used by provider-owned streams, then add focused tests proving `openai-codex/gpt-5.5` / `openai-codex-responses` receives `options.apiKey`. After the fix lands on main, close this as implemented.
