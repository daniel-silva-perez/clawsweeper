## Summary

Keep open. Current main intends `openai-codex/gpt-5.5` to use the Codex Responses transport, but the embedded boundary-aware stream path still has a credible OpenClaw-owned auth gap: it can select the OpenClaw Responses transport without wrapping it with the resolved OAuth bearer token. Related OAuth-scope reports are relevant context, but they do not clearly supersede this specific GPT-5.5 Control UI missing-bearer regression.

## Best Possible Solution

Add a regression test for an embedded Control UI run with `openai-codex/gpt-5.5` proving the resolved OAuth profile token reaches the `openai-codex-responses` request, then fix `resolveEmbeddedAgentStreamFn` or the transport selection so boundary-aware Codex Responses streams receive the bearer token and required Codex headers. Keep the adjacent OAuth-scope issues in view, but treat this report as a current runtime auth propagation bug until proven otherwise.
