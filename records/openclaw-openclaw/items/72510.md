## Summary

Keep this PR open. The direction is useful and the latest patch addresses the two Greptile null-vs-undefined notes, but it still does not cover the linked Telegram multi-bot repro and it adds a stale exact-string test that will fail against current main.

## What This Changes

The PR adds optional bot-targeting metadata to auto-reply history/template context, renders that metadata in inbound prompt context, and updates Telegram mention/reply parsing plus tests to populate the new fields.

## Best Possible Solution

Keep the additive metadata contract, but require the Telegram path to either receive a trustworthy sibling-bot list or preserve a separate explicit signal for skipped messages addressed to another visible mention. The final patch should rebase onto current main, update the byte-stability assertion, and prove current-bot, sibling-bot, reply-to-bot, and ambient history behavior with focused tests.

## Reproduction Assessment

Yes. The source-level path is high-confidence: current main records skipped Telegram group messages without target metadata, and the PR still calls the new detector with only the current bot username on that skipped-history branch.

## Solution Assessment

No. The core fields and prompt rendering are a reasonable shape, but the proposed end-to-end Telegram fix is incomplete until the skipped-history branch can identify or safely represent sibling-bot mentions, and the stale exact-string test is updated for current main.
