## Summary

Keep open. Current main still has session maintenance paths that can evict idle topic/thread conversation pointers: stale pruning and entry-count capping only preserve explicitly supplied keys, and disk-budget enforcement only skips the active key before deleting oldest store entries. The related items are adjacent, but none is a merged or canonical replacement for this focused maintenance-eviction bug.

## What This Changes

Request to prevent automatic session maintenance from deleting long-lived topic/thread session entries from sessions.json.

## Best Possible Solution

Add a shared session-maintenance protection predicate for long-lived external conversation sessions and use it consistently in save-time prune/cap, load-time prune/cap, disk-budget entry deletion, and cleanup previews. The predicate should protect topic/thread conversation keys and real external group/channel session pointers while still allowing ephemeral subagent/cron/system sessions to age out.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to create a store with an older topic/thread session key and enough newer ephemeral entries or disk pressure, then run the current maintenance helpers; the code paths shown above delete non-active old keys. I did not execute a mutation test because this review was required to remain read-only.

## Solution Assessment

The requested direction is valid, but the safest implementation is a shared typed helper rather than ad hoc substring checks in each loop. Protecting every entry with group-like metadata would be too broad because subagents can inherit group delivery metadata and still be intended maintenance candidates.
