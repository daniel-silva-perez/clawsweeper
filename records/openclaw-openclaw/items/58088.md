## Summary

Keep open. Current main still allows session maintenance to delete idle topic/thread session pointers: prune/cap paths only honor explicit preserve keys, disk-budget enforcement only skips the active key, and cleanup preview/reporting follows the same generic maintenance helpers. The related reports are adjacent, but none is a merged fix or exact replacement for this focused retention bug.

## What This Changes

The issue asks OpenClaw session maintenance to preserve long-lived topic/thread conversation session pointers instead of pruning them through generic age, count, or disk-budget cleanup.

## Best Possible Solution

Add a shared session-maintenance protection predicate for durable external conversation sessions and apply it consistently in save-time prune/cap, load-time prune/cap, disk-budget entry deletion, and cleanup preview/reporting. Keep configurable per-pattern retention rules as separate #50900 product work.

## Reproduction Assessment

Yes. A high-confidence static reproduction is a store with an older Telegram topic or generic thread key plus enough newer ephemeral entries or disk pressure; current helpers will delete the old non-active key. I did not run mutating tests because this review is read-only.

## Solution Assessment

No, not exactly as suggested. The requested protection is right, but ad hoc :topic:/:thread: substring checks are not the best implementation; a shared typed helper using existing thread/session parsing and entry metadata is safer and easier to apply across all maintenance paths.
