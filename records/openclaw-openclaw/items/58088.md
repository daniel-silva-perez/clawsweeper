## Summary

Keep open. Current main still lets session maintenance delete idle topic/thread pointers: stale pruning and entry capping only preserve caller-provided keys, and disk-budget enforcement only skips the active key before deleting oldest store entries. The related reports are adjacent, but none is a merged fix or exact replacement for this focused data-retention bug.

## What This Changes

The issue requests that automatic session maintenance preserve long-lived topic/thread group conversation session pointers instead of pruning them with generic disk, age, or count maintenance.

## Best Possible Solution

Add a shared session-maintenance protection predicate for durable external conversation sessions and apply it consistently in save-time prune/cap, load-time prune/cap, disk-budget entry deletion, and cleanup preview/reporting. Keep configurable retention rules as separate #50900 design work unless maintainers explicitly combine them.

## Reproduction Assessment

Yes. A high-confidence static reproduction is a store with an older Telegram topic or generic thread key plus enough newer ephemeral entries or disk pressure; the current helpers will delete the old non-active key. I did not run a mutating test because this review was required to remain read-only.

## Solution Assessment

Yes. The best narrow fix is a shared typed helper used by all maintenance paths, not ad hoc substring checks in each loop; it should protect real topic/thread conversation pointers while allowing ephemeral maintenance candidates to age out.
