## Summary

Current main still has the reported core behavior: session maintenance can evict non-active long-lived topic/thread/group session entries. The issue is concrete, recent, and not superseded by the related reports in the provided context.

## Best Possible Solution

Implement a core session-store protection predicate for long-lived conversation rows, likely covering generic `:thread:` keys, Telegram `:topic:` keys, entries with thread metadata, and group/channel chat types where appropriate. Apply it consistently to save-time pruning, save-time capping, disk-budget entry eviction, and load-time oversized-store cleanup, then add regression tests for Telegram topic keys and generic thread keys plus docs/config notes if retention semantics change.

## Evidence

There is no protection for `:topic:`, `:thread:`, `chatType: "group"`, or thread metadata.

## Risks / Open Questions

- Leaving this open is appropriate because the current code can still remove long-lived topic/thread/group session pointers during enforce-mode pruning, max-entry capping, disk-budget cleanup, and load-time oversized-store cleanup.
- A fix should avoid preserving all channel/group rows blindly if operators intentionally configure strict retention; the product decision likely needs a clear protected-session predicate and tests for topic/thread/group rows across all maintenance paths.
