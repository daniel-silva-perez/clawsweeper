## Summary

Keep open. The bootstrap EIO race remains plausible because current main still has raw `launchctl bootstrap` paths, but this PR is not mergeable as-is: it adds numeric-PID SIGKILL escalation and is stale against the current kickstart-first launchd restart lifecycle.

## What This Changes

The PR adds EIO-only `launchctl bootstrap` retries, a PID wait helper with SIGKILL escalation, applies those paths to launchd repair/install/restart, and expands launchd unit coverage.

## Best Possible Solution

Implement a shared bounded EIO-only `launchctl bootstrap` retry helper on current main and use it for `bootstrapLaunchAgentOrThrow`, `repairLaunchAgentBootstrap`, and launchd handoff bootstrap fallbacks while preserving the kickstart-first restart model. Add force-kill escalation only with explicit maintainer approval and launchd job identity verification before signaling.

## Reproduction Assessment

Yes for the blocking PR finding: the PR branch shows `process.kill(pid, "SIGKILL")` after only a numeric-PID liveness probe. The original EIO race has a concrete manual reproduction in the PR body, but this review did not rerun it live on macOS.

## Solution Assessment

No. EIO-only retry is the narrower maintainable direction, but this PR mixes it with unsafe PID-based force killing and an older restart lifecycle; a current-main retry-focused patch is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Verify the saved PID before SIGKILL:** `src/daemon/launchd.ts:440`
  - body: The branch waits, checks `kill(pid, 0)`, then sends SIGKILL to the saved PID. That only proves some process currently has the PID; if the LaunchAgent exited and the PID was reused, install or restart can terminate an unrelated process. Avoid SIGKILL here or verify the PID still belongs to the expected launchd job before signaling.
  - confidence: 0.86
