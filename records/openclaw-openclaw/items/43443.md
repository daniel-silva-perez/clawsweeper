## Summary

Keep open. The PR targets a plausible launchd EIO restart race that current main still does not fully address, but the branch is not merge-ready because it adds a PID-only SIGKILL escalation path and is based on an older launchd restart lifecycle.

## What This Changes

The PR adds EIO-only launchctl bootstrap retries, a PID wait helper with SIGKILL escalation, applies those paths to launchd repair/install/restart, and expands launchd unit coverage.

## Best Possible Solution

Implement a shared bounded EIO-only launchctl bootstrap retry helper on current main and use it for bootstrapLaunchAgentOrThrow, repairLaunchAgentBootstrap, and launchd handoff bootstrap attempts while preserving the current kickstart-first restart model. Add force-kill escalation only if maintainers explicitly want it and the code verifies launchd job identity before signaling.

## Reproduction Assessment

Yes for the blocking PR finding: branch inspection shows waitForPidExitWithEscalation sends SIGKILL after only a numeric-PID liveness probe. The original EIO failure also has a concrete manual reproduction in the PR body, but this review did not rerun it live.

## Solution Assessment

No. EIO-only retry is likely the narrow maintainable direction, but this PR combines it with unsafe PID-based force killing and an older restart lifecycle; a current-main retry-focused patch is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Verify the saved PID before SIGKILL:** `src/daemon/launchd.ts:440`
  - body: The branch waits, checks kill(pid, 0), then sends SIGKILL to the saved PID. That only proves some process currently has the PID; if the LaunchAgent exited and the PID was reused, install or restart can terminate an unrelated process. Avoid SIGKILL here or verify the PID still belongs to the expected launchd job before signaling.
  - confidence: 0.86
