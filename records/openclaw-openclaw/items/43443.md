## Summary

Keep this PR open. Current main has newer macOS LaunchAgent restart hardening, but it still lacks the PR's central bounded EIO-only `launchctl bootstrap` retry, and several bootstrap paths remain single-attempt. The PR is useful but stale against the current kickstart-first launchd flow and should not merge as-is without maintainer review of the proposed SIGKILL escalation.

## What This Changes

The PR adds bounded EIO retries around `launchctl bootstrap`, waits for prior LaunchAgent PIDs after bootout with SIGKILL escalation, updates install/restart/doctor repair paths, and expands launchd unit tests.

## Best Possible Solution

Add a shared bounded EIO-only bootstrap retry on current main and route all LaunchAgent bootstrap surfaces through it, including doctor repair, install activation, restart fallback recovery, unloaded-service restart recovery, and detached handoff scripts. Treat SIGKILL escalation as a separate maintainer decision; if retained, require process identity protection and tests for retry success/exhaustion, errno 125, non-EIO failures, doctor repair, install activation, restart fallback, and handoff behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Verify the saved PID before sending SIGKILL:** `src/daemon/launchd.ts:440`
  - body: The branch stores a previous PID once, waits up to 15 seconds, then calls `process.kill(pid, "SIGKILL")` if `kill(pid, 0)` still succeeds. A reused PID will also pass that probe, so reinstall/restart can terminate an unrelated user process. The final liveness check does not prove process identity; either avoid SIGKILL or verify the PID still belongs to the expected launchd job before killing it.
  - confidence: 0.86

## Security Review

Status: needs_attention

  - body: The proposed escalation sends SIGKILL to a saved PID after a delay based only on `kill(pid, 0)` liveness. PID reuse can make that probe succeed for a different process, creating a concrete availability and local safety risk in a service-management command.
