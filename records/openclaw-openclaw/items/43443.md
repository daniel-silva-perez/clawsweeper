## Summary

Keep open. Current main still has single-attempt LaunchAgent bootstrap paths, so the useful EIO retry part is not implemented, but this PR is dirty against the current launchd architecture and its SIGKILL escalation introduces a concrete PID-reuse safety issue before merge.

## What This Changes

The PR adds bounded EIO retries around launchctl bootstrap, waits for prior LaunchAgent PIDs with SIGKILL escalation, updates install/restart/doctor repair paths, and expands launchd unit tests.

## Best Possible Solution

Add a shared bounded EIO-only bootstrap retry on current main and route all LaunchAgent bootstrap surfaces through it, including doctor repair, install activation, restart fallback recovery, unloaded-service recovery, and detached handoff scripts. Treat SIGKILL escalation as a separate maintainer decision; if retained, verify process identity before killing and cover retry success/exhaustion, errno 125, non-EIO failures, doctor repair, install activation, restart fallback, and handoff behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Verify the saved PID before sending SIGKILL:** `src/daemon/launchd.ts:440`
  - body: The branch stores a previous PID, waits up to 15 seconds, then sends SIGKILL if kill(pid, 0) still succeeds. A reused PID will also satisfy that probe, so reinstall/restart can terminate an unrelated user process. Avoid SIGKILL here or verify the PID still belongs to the expected launchd job before killing it.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: No dependency or supply-chain changes were found, but the diff adds a process-control path that can SIGKILL an unrelated reused PID without identity verification.

Concerns:

- **[medium] SIGKILL can target a reused PID:** `src/daemon/launchd.ts:440`
  - body: The proposed escalation sends SIGKILL to a saved PID after a delay based only on kill(pid, 0) liveness. PID reuse can make that probe succeed for a different process, creating a local availability and safety risk in a service-management command.
  - confidence: 0.82
