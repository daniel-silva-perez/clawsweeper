## Summary

Keep open. The updated PR avoids the earlier default security broadening by making arbitrary system-trusted TLS pin repair opt-in, but the opt-in is not registered in OpenClaw's strict canonical config contract, so an operator who sets it would create a config the gateway rejects.

## What This Changes

The PR adds a macOS opt-in `gateway.remote.allowSystemTrustedPinRepair` path for stale TLS pin auto-repair on system-trusted remote `wss://` gateways, updates Swift helper tests, and adds a changelog entry for #74848.

## Best Possible Solution

Keep the default loopback/`.ts.net` repair boundary, and if the opt-in policy is accepted, make `gateway.remote.allowSystemTrustedPinRepair` a first-class config key across the shared type, zod schema, labels/help metadata, generated config docs, public docs, and validation tests.

## Reproduction Assessment

Yes. Current main can be checked at helper level with a non-loopback, non-`.ts.net` `wss://` URL and a `pinMismatch` failure where `systemTrustOk` is true; the PR's remaining defect is reproducible by validating config that contains `gateway.remote.allowSystemTrustedPinRepair` against the strict shared schema.

## Solution Assessment

No, not yet. The opt-in approach is safer than broadening default pin repair, but it is incomplete until the opt-in is accepted by the canonical config/schema/docs surfaces that the Gateway and Control UI use.
