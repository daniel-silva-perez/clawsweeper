## Summary

Keep this PR open, but do not merge it yet. The opt-in direction addresses the earlier security concern better than broadening the default TLS pin repair policy, but the advertised `gateway.remote.allowSystemTrustedPinRepair` key is not registered in the canonical strict config contract, so operators enabling it would create an invalid schema-backed config.

## What This Changes

The PR adds a macOS `gateway.remote.allowSystemTrustedPinRepair` opt-in for stale TLS pin auto-repair on arbitrary system-trusted `wss://` gateway hosts, adds Swift coverage for default/opt-in paths, and adds a changelog entry for #74848.

## Best Possible Solution

Keep the default loopback/`.ts.net` repair boundary, and if the opt-in policy is accepted, make `gateway.remote.allowSystemTrustedPinRepair` a first-class config key across the shared type, zod schema, labels/help metadata, generated config docs baseline, public docs, and validation tests before merging.

## Reproduction Assessment

Yes. At code level, current main returns false for a non-loopback, non-`.ts.net` `wss://gateway.example.com` pin mismatch even when `systemTrustOk` is true; the PR’s remaining defect is reproducible by validating config that contains the new `gateway.remote.allowSystemTrustedPinRepair` key against the strict schema.

## Solution Assessment

Not yet. The explicit opt-in is a safer direction than silently widening pin repair for every system-trusted host, but this implementation is incomplete until the new key is part of the canonical config contract and docs.
