## Summary

Keep open. The PR targets a real macOS stale TLS-pin reconnect problem from #74848, but the proposed policy silently replaces stored gateway pins for any certificate chain accepted by macOS system trust, which weakens the pin boundary and needs maintainer/security judgment before merge.

## What This Changes

The PR changes macOS stale gateway TLS pin repair so any `wss://` pin mismatch with a matching store key and `systemTrustOk == true` replaces the stored fingerprint, adds Swift coordinator tests for trusted and untrusted public hosts, and adds a changelog entry.

## Best Possible Solution

Keep stale-pin repair explicitly scoped and documented, or add an operator-confirmed/configured-fingerprint path before replacing pins for arbitrary remote hosts. Coverage should distinguish public WebPKI rotation from OS-trusted private-root or enterprise-proxy mismatches.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes at code level: construct a `GatewayTLSValidationFailure(kind: .pinMismatch, storeKey: host:443, systemTrustOk: true)` for a non-loopback, non-`.ts.net` `wss://` host and current main returns `false`, while the PR changes that path to `true`. The exact reporter environment from #74848 was not live-reproduced in this read-only pass.

## Solution Assessment

Is this the best way to solve the issue? No. The proposed implementation is narrow, but using `systemTrustOk` as the only remote-host policy check is not the safest maintainable solution for stored TLS pins; scoped allowlists, explicit user confirmation, public-root policy, or configured fingerprint flow would preserve the boundary better.

## Security Review

Status: needs_attention

Summary: The diff broadens TLS pin self-repair from known host classes to every system-trusted chain, which is a concrete security-sensitive change.

Concerns:

- **[high] Silent pin replacement for any system-trusted cert:** `apps/macos/Sources/OpenClaw/NodeMode/MacNodeModeCoordinator.swift:217`
  - body: The new policy would replace a stored gateway TLS pin whenever `SecTrustEvaluateWithError` succeeds, including OS-trusted private roots or local interception roots. That turns a pin mismatch into silent trust-on-new-cert for arbitrary remote hosts and should not merge without maintainer/security approval.
