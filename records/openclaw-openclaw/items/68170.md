## Summary

Keep open. Current `main` still has the reported broad sibling-home scan and still prints discovered sibling `.openclaw` paths in Doctor output; the latest shipped tag also contains that behavior. The related #68752 corroborates the same noisy multi-user-host warning, but it is closed and does not supersede this still-reproducible privacy/noise issue.

## What This Changes

The issue asks Doctor to stop broadly scanning sibling home directories for `.openclaw` state directories and to avoid cross-user path disclosure or false positives from backup-style sibling homes.

## Best Possible Solution

Keep this issue open and patch `src/commands/doctor-state-integrity.ts` so Doctor no longer enumerates sibling home directories by default. The conservative end state is a state-dir diagnostic scoped to the current effective home and configured/default state paths, with no verbatim non-current-user paths in output; add focused tests for `/home/alice` versus `/home/bob` and backup-sibling names, then update the public Doctor docs if the scan semantics change.

## Reproduction Assessment

Yes. On Linux or macOS, any readable sibling directory under `/home` or `/Users` with a `.openclaw` subdirectory is deterministically collected by `findOtherStateDirs` and printed by `noteStateIntegrity`; the current source is enough to reproduce without mutating this checkout.

## Solution Assessment

No current solution is in place. Among the issue's proposed options, scoping or removing broad sibling-home enumeration by default is safer than only adding an opt-out or backup-name filter, because it addresses both path disclosure and false positives.
