## Summary

Keep open. Current main has improved the Feishu proactive-send path since the original bypass theory: `message(action=send)` now reaches plugin action handling. The specific compatibility regression remains, though: raw Feishu interactive-card JSON supplied as the plain `message`/`text` value is still handled as text/post content. Feishu cards are produced from semantic `presentation` payloads or explicit `channelData.feishu.card` metadata, not by detecting native card JSON embedded in `message`. Related #53295 and #53310 covered other Feishu proactive/card failures and do not supersede this raw-card-JSON report.

## Best Possible Solution

Keep this issue open for Feishu-owned compatibility work. The likely fix belongs in `extensions/feishu/src/channel.ts` or a nearby Feishu helper: when `send` or `thread-reply` receives a plain `message`/`text` string, conservatively parse only valid Feishu v1/v2 or wrapped interactive card JSON and route it to `sendCardFeishu`; add regression tests proving native card JSON uses `msgType: "interactive"` while ordinary JSON remains text. If maintainers intentionally want to drop this compatibility path, update the Feishu/message docs to require semantic `presentation` or explicit provider metadata instead.

## Risks / Open Questions

- Maintainers may decide provider-native Feishu card JSON in `message` is no longer supported, but that should be an explicit product/docs resolution because the report describes a prior workflow as a regression.
