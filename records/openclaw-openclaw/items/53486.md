## Summary

Keep open. Current main still treats raw Feishu interactive-card JSON passed through the plain `message`/`text` value as text/post content; Feishu cards are sent only from semantic `presentation`/legacy `interactive` payloads or explicit `channelData.feishu.card` metadata. The older complete-bypass theory is partly obsolete because proactive sends now reach Feishu plugin action handling, but the reported compatibility regression remains.

## What This Changes

Restore Feishu proactive `message(action=send)` compatibility so native interactive-card JSON supplied as the plain `message` value is sent as an interactive Feishu card instead of raw text.

## Best Possible Solution

Keep this open and fix it in the Feishu plugin boundary. The narrow product/code path is to detect only valid Feishu native card JSON strings received as `message`/`text` for `send` and `thread-reply`, sanitize them through the existing native-card sanitizer, and route them to `sendCardFeishu`; ordinary JSON and malformed cards should remain text. Do not add provider-native card fields to the shared core message schema.
