## Summary

Keep this issue open. Current main still routes a Feishu `message(action=send)` plain `message` string through the text path unless the caller supplies structured `presentation`/`interactive` data or explicit `channelData.feishu.card`; the merged native-card payload work is adjacent but does not cover serialized card JSON in `message`. A fresh reporter confirmation keeps the regression current for 2026.4.25.

## What This Changes

Restore Feishu proactive sends so valid interactive-card JSON supplied in the `message` parameter renders as a Feishu interactive card instead of raw plain text.

## Best Possible Solution

Add a Feishu-owned compatibility parser for valid native interactive-card JSON strings on `send` and `thread-reply`, including schema-2, legacy top-level `elements`, and wrapped interactive-card shapes when they pass the local sanitizer. Route only sanitized cards to `sendCardFeishu`; leave malformed card JSON and unrelated JSON as normal text, and keep Feishu-specific parsing out of the shared message runner unless a generic plugin seam becomes necessary.
