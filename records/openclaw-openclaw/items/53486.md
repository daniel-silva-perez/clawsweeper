## Summary

Keep this open. Current main still does not detect valid Feishu native card JSON when it is supplied as the plain `message` string: the shared runner parses only structured presentation/delivery/interactive fields, and the Feishu `send`/`thread-reply` action sends a card only from normalized `presentation`; otherwise the string falls through to the post/text sender. The related native-card work in #72667 fixed explicit `channelData.feishu.card` payloads but does not cover this compatibility path.

## What This Changes

Restore Feishu `message(action=send)` compatibility so valid native interactive-card JSON supplied in the `message` parameter renders as a Feishu interactive card instead of plain text.

## Best Possible Solution

Add Feishu-local compatibility detection for valid native interactive-card JSON strings in `send` and `thread-reply`, sanitize through the existing native-card sanitizer, route valid cards to `sendCardFeishu`, and leave malformed card JSON or ordinary JSON as text. The shared message schema should remain provider-agnostic.

## Repair Work Prompt

Fix the Feishu regression tracked by #53486: valid Feishu native interactive-card JSON supplied as the plain `message` string for `message(action=send)` or `thread-reply` is currently sent as text/post instead of `msg_type: "interactive"`.

Implement this as a Feishu-owned compatibility fix, not a shared core schema change. Add a small detector/parser in the Feishu plugin path that recognizes only valid native Feishu card shapes from `text`/`message` strings, including the legacy v1 `header`/`elements` shape, schema 2.0 card shape, and wrapped `{ "type": "interactive", "card": ... }` when appropriate. Reuse or expose the existing native-card sanitizer in `extensions/feishu/src/outbound.ts` so valid cards are sanitized before `sendCardFeishu`. Ordinary JSON, malformed JSON, unsupported card shapes, and unsafe card content must continue as text or be sanitized away according to existing behavior.

Likely implementation boundary: `extensions/feishu/src/channel.ts` for `send`/`thread-reply`, possibly a local helper in `extensions/feishu/src/outbound.ts` or a new Feishu-local helper if sharing with outbound payload tests is cleaner. Do not add provider-native Feishu `card` fields to the shared message action schema and do not move Feishu-specific detection into core.

Add regression tests for `send` and `thread-reply` with raw card JSON in `message`, explicit non-card JSON remaining text, malformed JSON remaining text, and sanitizer behavior for unsafe native card content. Update changelog if repo policy treats this user-facing regression fix as requiring an entry; credit the reporters with `Thanks @ZenoRewn` and `Thanks @smallmj` if adding one.
