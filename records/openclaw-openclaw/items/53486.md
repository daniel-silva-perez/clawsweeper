## Summary

Keep this issue open. Current main still treats a valid Feishu card JSON string in the plain `message` parameter as text: the shared runner only parses structured `presentation`, `delivery`, and `interactive` fields, while the Feishu send handler only builds a card from normalized `presentation`. The merged native-card payload work covers explicit `channelData.feishu.card` payloads but not this compatibility path, and a fresh issue comment confirms the regression is still affecting 2026.4.25.

## What This Changes

The issue asks for Feishu `message(action=send)` to detect valid interactive-card JSON supplied in the `message` string and send it as a native interactive card instead of a post/text message.

## Best Possible Solution

Add Feishu-local compatibility detection for valid native interactive-card JSON strings in `send` and `thread-reply`, sanitize through the existing native-card sanitizer, route valid cards to `sendCardFeishu`, and leave malformed card JSON or ordinary JSON as text. Keep the shared message schema provider-agnostic and avoid moving Feishu-specific parsing into core unless a generic plugin seam is needed.
