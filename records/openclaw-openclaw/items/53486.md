## Summary

Keep this issue open. Current main still sends Feishu card JSON supplied through the plain `message` parameter down the text/post path; the implemented native-card support only covers structured `presentation`, `interactive`, or `channelData.feishu.card` payloads, and the related merged PR does not cover this compatibility regression.

## What This Changes

The issue requests that Feishu `message(action=send)` detect valid interactive-card JSON strings in the plain `message` field and send them as native Feishu interactive cards instead of plain text.

## Best Possible Solution

Add Feishu-local compatibility detection for valid native interactive-card JSON strings on `send` and `thread-reply`, support the known v1/v2/wrapped card shapes, sanitize through the existing native-card safety path, route valid cards to `sendCardFeishu`, and keep malformed card JSON or ordinary JSON on the text path. Keep the shared message tool provider-agnostic unless a generic plugin seam becomes necessary.

## Reproduction Assessment

Yes. The issue provides concrete steps, a sample card JSON payload, and a 2026-04-29 confirmation on OpenClaw 2026.4.25; current main’s source path also shows that a plain `message` string cannot become a Feishu native card today.

## Solution Assessment

Yes, the best fix is a narrow Feishu-owned compatibility path using the existing sanitizer and `sendCardFeishu` route. Moving Feishu-specific JSON parsing into shared core would be broader and less aligned with the plugin ownership boundary unless more channels need the same seam.
