## Summary

Keep this issue open. Current main still sends a valid Feishu native card JSON string from the plain `message` parameter as text: only structured `presentation`, `delivery`, and `interactive` params are parsed before dispatch, and the Feishu `send`/`thread-reply` handler only builds cards from normalized `presentation`. The merged native-card payload work covers explicit `channelData.feishu.card` payloads, but not this compatibility path, and the discussion includes a fresh confirmation on OpenClaw 2026.4.25.

## What This Changes

The issue asks for Feishu `message(action=send)` to detect valid interactive-card JSON supplied as the plain `message` string and send it as a native interactive card instead of post/text content.

## Best Possible Solution

Add Feishu-local compatibility detection for valid native interactive-card JSON strings in `send` and `thread-reply`, sanitize through the existing native-card sanitizer, route valid cards to `sendCardFeishu`, and leave malformed card JSON or ordinary JSON on the text path. Keep the shared message schema provider-agnostic unless a small generic plugin seam is clearly required.
