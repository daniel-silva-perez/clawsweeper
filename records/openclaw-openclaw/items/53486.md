## Summary

Keep open. Current main no longer matches the oldest bypass theory because Feishu `message(action=send)` reaches plugin action handling, but the reported compatibility regression is still present: a native Feishu card JSON string passed as `message` is treated as plain text/post content unless the caller uses semantic `presentation` or explicit `channelData.feishu.card` on a different outbound path. Related #53295 and #53310 do not supersede this raw-card-JSON case.

## Best Possible Solution

Keep this issue open for Feishu-owned compatibility work. The likely fix belongs in `extensions/feishu/src/channel.ts` or a small Feishu helper: when `send` or `thread-reply` receives a plain `message`/`text` string, conservatively parse only valid Feishu v1/v2 or wrapped interactive card JSON, sanitize it through the existing native-card path, and route it to `sendCardFeishu`. Add regression tests proving native card JSON sends `msgType: "interactive"` while ordinary JSON remains text. If maintainers choose to drop this compatibility path, update the Feishu/message docs to say callers must use `presentation` or explicit provider metadata.

## Risks / Open Questions

- A compatibility detector for raw card JSON must be conservative so ordinary JSON text is not silently converted into Feishu cards.
- Native Feishu cards can contain provider-native action/link fields, so any raw-card bridge should reuse existing sanitization before calling `sendCardFeishu`.
- Maintainers may intentionally prefer the newer semantic `presentation` contract, but that would need an explicit product/docs resolution because the report describes a prior workflow as a regression.
