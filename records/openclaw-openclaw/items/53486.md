## Summary

Keep this open. Current main still routes Feishu `message(action=send)` and `thread-reply` text through the text branch unless the payload uses semantic `presentation` or explicit `channelData.feishu.card`/`interactiveCard`; a raw Feishu card JSON string in `message` is not detected and would still be sent as a post/plain-text message.

## What This Changes

The issue requests restoring Feishu proactive sends so valid interactive-card JSON passed in the `message` parameter renders as a Feishu interactive card instead of plain text.

## Best Possible Solution

Add owner-local Feishu compatibility detection for valid native interactive-card JSON strings received as `message`/`text` on `send` and `thread-reply`, sanitize through the existing native-card sanitizer, and route to `sendCardFeishu`; malformed cards and ordinary JSON should continue as text, and the shared core message schema should stay provider-agnostic.
