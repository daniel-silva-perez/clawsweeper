## Summary

Keep this PR open. Current main does not implement the proposed per-file-signature cache for `loadPluginManifest`; it only has broader manifest-registry/installed-index caching and metadata snapshot reuse. The PR is recent, focused, draft, tied to the still-open CPU regression report, and its diff is limited to manifest loading plus a regression test, so it needs maintainer review/validation rather than cleanup closure.

## Best Possible Solution

Keep the draft PR open and review it as the focused implementation candidate for the manifest-load portion of #73647. The best path is to verify the CPU root cause against current main, confirm the cache invalidation and boundary-check behavior, add or adjust regression coverage for unchanged and changed manifest files, then run the targeted manifest loader test plus the appropriate changed gate/Testbox validation before merge.

## Risks / Open Questions

- The PR's cache invalidation policy should be reviewed carefully before merge: a signature of dev, inode, size, and mtimeMs is usually reasonable, but maintainers should confirm it is sufficient on the reported Docker/Synology filesystem and cannot mask manifest edits with unchanged metadata.
- The new cache should preserve the existing boundary-file safety checks on every call. The diff appears to keep `openBoundaryFileSync` before cache lookup, but this should be part of normal review.
