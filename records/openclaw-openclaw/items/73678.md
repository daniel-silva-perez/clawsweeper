## Summary

Keep this PR open. Current main still does not implement the proposed per-file-signature cache for direct `loadPluginManifest` calls; existing reuse is snapshot, manifest-registry, or installed-index registry scoped. The linked CPU regression #73647 is still open, and this draft PR is a focused implementation candidate rather than cleanup closure material.

## Best Possible Solution

Keep the draft PR open and review it as the focused implementation candidate for the direct manifest-load portion of #73647. Verify the current CPU root cause, confirm cache invalidation and boundary-check behavior, add or keep coverage for unchanged and changed manifest files, then run the targeted manifest loader test plus the appropriate changed gate/Testbox validation before merge.

## Risks / Open Questions

- The proposed cache invalidation needs maintainer review against Docker/Synology filesystem behavior; a dev/inode/size/mtime-style signature is plausible but should not mask manifest edits with unchanged metadata.
- The cache must preserve the current `openBoundaryFileSync` safety checks on every call before returning a cached result.
- The linked CPU regression may have more than one root cause, so this PR should be validated as the manifest-load slice rather than assumed to fully fix #73647.
