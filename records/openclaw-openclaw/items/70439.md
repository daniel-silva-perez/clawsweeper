## Summary

Keep open. Current main still has the Signal voice-note preflight gap described in the issue and in the maintainer follow-up: Signal preserves audio media metadata, but it does not call the shared audio preflight helper before mention/command handling and it does not mark a preflight transcript on the inbound context.

## What This Changes

Request to add Signal audio preflight so inbound Signal voice notes can be transcribed before agent, mention, and command handling, preferably through the shared/provider-backed path rather than a shell command hook.

## Best Possible Solution

Implement a Signal-owned provider-backed voice-note preflight after Signal access checks and attachment limits but before requireMention/command gating, then pass the transcript into body/command/mention handling while preserving media metadata and marking transcribed media so the shared media-understanding pass does not double-transcribe.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: with a Signal group that requires mentions and an audio-only attachment, current main can return at the no-mention branch before attachments are fetched; for messages that do pass, the inbound context carries media metadata but no Transcript or MediaTranscribedIndexes.

## Solution Assessment

Yes. The best current fix is the narrow provider-backed Signal hook using the existing transcribeFirstAudio helper, plus targeted Signal tests and docs; adding a generic shell preflight command would be a broader and more security-sensitive design.
