## Summary

Keep open. Current main still has the Signal voice-note preflight gap described by the reporter and maintainer follow-up: Signal now preserves audio metadata, but it does not run the shared audio preflight helper or attach transcript fields before mention/command handling.

## What This Changes

Expose a Signal audio-preflight path or hook so inbound Signal voice notes can be transcribed before the agent sees only a media placeholder.

## Best Possible Solution

Implement a Signal-owned provider-backed voice-note preflight using the existing transcribeFirstAudio helper after Signal access checks and attachment limits, then feed the transcript into body, command, and mention handling while preserving media metadata and marking the transcribed attachment to avoid duplicate media-understanding work.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: an audio-only Signal group message with requireMention can be dropped before attachments are fetched, and accepted Signal audio messages currently carry media metadata without Transcript or MediaTranscribedIndexes.

## Solution Assessment

Yes. The narrowest maintainable fix is a Signal plugin hook that reuses the existing provider-backed audio preflight helper; adding a generic shell preflight command would be broader and more security-sensitive than needed for parity.
