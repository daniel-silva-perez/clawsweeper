## Summary

Keep open. Current main partially fixed the older Signal audio metadata problem, but it still does not implement the requested Signal audio preflight or configurable preflight hook. Signal forwards audio attachments into the shared media-understanding path after routing, while Telegram/Discord/WhatsApp have explicit voice-note preflight paths that can produce a transcript before mention gating and mark the attachment as already transcribed.

## Best Possible Solution

Keep this open and implement a provider-backed Signal voice-note preflight using the existing shared `transcribeFirstAudio` path. It should run after DM/group access checks but before requireMention skips for eligible audio-only Signal messages, feed the transcript into mention/body handling, set `Transcript` and `MediaTranscribedIndexes`, and document Signal behavior. Do not add an arbitrary shell `preflight_cmd` without a separate security-reviewed design.

## Risks / Open Questions

- An arbitrary `preflight_cmd` would add a new command-execution surface around user-supplied media paths; the safer first fix is provider-backed preflight through the existing media pipeline.
- Signal currently avoids downloading attachments for mention-skipped group messages. A fix should preserve access checks, `ignoreAttachments`, `mediaMaxMb`, and blocked-sender behavior before any new preflight download/transcription.
- The preflight transcript must be treated as untrusted user content and must set `MediaTranscribedIndexes` so the shared media-understanding pass does not duplicate STT calls.
