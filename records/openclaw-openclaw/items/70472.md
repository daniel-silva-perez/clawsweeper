## Summary

Keep open. Current main still routes overflow and compaction-failure recovery through a session reset path that writes a new session id into the active session entry, and #70479 is an open, unmerged closing PR for this issue rather than a reason to close it before merge.

## What This Changes

The issue asks OpenClaw to preserve the live `agent:main:main` WebChat session mapping across context-overflow and compaction-checkpoint failures instead of silently remapping the session key to a fresh transcript.

## Best Possible Solution

Keep this issue open and review or land #70479, or an equivalent narrow fix, so compaction failure preserves the active session mapping unless the user explicitly requests a reset. The fix should also make any intentional recovery rotation auditable and cover the newer missing-transcript variant reported in #75151 if it shares the same reset-store ordering path.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to make the embedded runner return a context-overflow error payload or throw a compaction-failure error while `resetSessionAfterCompactionFailure` succeeds; current main then calls `resetReplyRunSession`, and the existing reset unit test asserts the active store row rotates to a generated session id.

## Solution Assessment

Yes for keeping this issue open. The current main behavior is not the best fix because replaying recent messages into a new transcript still changes the active session id; the maintainable direction is to preserve the existing logical session or make an intentional reset explicit and durable.
