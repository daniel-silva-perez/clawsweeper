## Summary

Keep open. Current main still creates a thread-scoped session for top-level Slack DMs when `replyToMode="all"`: routing derives `autoThreadId` from the message timestamp for non-thread replies, applies it to non-roomish DMs, and the shared session-key helper appends `:thread:<ts>`. The channel fix from the related reports is present for room/channel messages, but DM auto-threading is still explicitly preserved and covered by tests, so the reported per-channel-peer DM context loss is not fixed or superseded.

## Best Possible Solution

Keep this issue open and fix it in the Slack plugin owner surface if maintainers agree with the reported expectation. The likely patch belongs in `extensions/slack/src/monitor/message-handler/prepare-routing.ts`: avoid using `autoThreadId` for top-level direct messages when the desired DM session is the base per-peer/per-channel-peer session, while preserving real thread replies (`thread_ts` different from `ts` or `parent_user_id`) and documented direct-chat overrides. Add a focused regression test for `session.dmScope="per-channel-peer"` plus `replyToMode="all"` that expects `agent:<agentId>:slack:direct:<user>` without `:thread:`.
