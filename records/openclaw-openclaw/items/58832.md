## Summary

Keep open. Current main still matches the reported Slack DM behavior: top-level direct messages with `replyToMode="all"` derive the message's own timestamp as a routed thread id, and an existing regression test currently asserts that the DM session key contains a `:thread:` suffix. The earlier channel-session fix is present for room messages, but the direct-message path remains intentionally different.

## What This Changes

The issue requests that top-level Slack DMs with `dmScope: "per-channel-peer"` keep the persistent direct-message session key when `replyToMode` is not `off`, instead of appending `:thread:<message ts>`.

## Best Possible Solution

Fix the Slack plugin routing so only actual Slack thread replies add a `:thread:` suffix to direct-message inbound session keys; top-level DMs should keep the base `dmScope` session while still preserving `MessageThreadId` or reply-target metadata needed for Slack UI threading.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists in the current test suite: the top-level DM test with `replyToMode="all"` expects `SessionKey` to contain `:thread:500.000`, and the routing code deterministically derives that suffix from the message timestamp.

## Solution Assessment

Yes, the narrow maintainable fix belongs in Slack routing/tests rather than core session-key code. The safer implementation is to keep outbound reply threading metadata intact while preventing top-level direct messages from using the message timestamp as the inbound session-key thread id.
