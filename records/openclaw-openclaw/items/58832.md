## Summary

Keep open. Current main still routes ordinary top-level Slack DMs with `replyToMode="all"` through a per-message `:thread:<ts>` session key, and the existing tests still assert that behavior. The report is concrete and recent, but this is not a cleanup close because the behavior also appears to have been intentionally shipped earlier for Slack DM auto-threading.

## What This Changes

Stop top-level Slack direct messages with per-channel-peer DM scoping and non-off reply threading from being routed to per-message `:thread:<ts>` session keys.

## Best Possible Solution

Decide and document the intended Slack top-level DM policy. If base-session continuity is preferred for ordinary per-channel-peer DMs, change the Slack routing helper to suppress `autoThreadId` for non-thread direct messages while preserving real thread replies and any explicitly chosen assistant/native-thread roots, then update the focused Slack tests and user-facing docs or changelog.

## Reproduction Assessment

Yes. The current source-level path is enough to reproduce with high confidence: a direct Slack message, `replyToMode="all"`, and a top-level `ts` produce `autoThreadId`, and the existing Slack test asserts a `:thread:<ts>` session key for that case.

## Solution Assessment

Unclear. The reporter's requested direction is narrow and technically feasible, but current tests and changelog show the opposite DM auto-thread behavior was intentional, so maintainer product judgment should settle the Slack DM semantics before a fix branch is queued.
