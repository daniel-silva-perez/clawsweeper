## Summary

Keep this PR open. Current main still returns non-agent-scoped colon session keys verbatim in `sandbox explain`, so `telegram:slash:<id>` cannot be parsed for channel inference and still reports the elevated allowlist path incorrectly. The PR remains a focused implementation candidate, but it likely needs rebasing and coordination with #51584 because current main also still depends on the active plugin channel registry for `telegram` normalization.

## Best Possible Solution

Rebase this PR onto current main, keep the non-agent-scoped colon-key normalization, and include the downstream `elevated.channel` plus `allowedByConfig` assertions. Coordinate with #51584 or fold in a static channel-registry fallback so `telegram` resolves reliably in CLI diagnostics. If maintainers decide the supported contract is agent-scoped or target-session keys only, update the slash-command docs and resolve #51245 that way instead.
