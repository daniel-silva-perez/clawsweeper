## Summary

Keep this PR open. Current main still has the reported sandbox explain bug, and this PR is a useful but incomplete implementation candidate: it adds the needed session-key wrapping, but the channel resolver still depends on the active plugin registry, so CLI-only Telegram resolution likely still needs the static built-in fallback tracked by #51584.

## What This Changes

The PR normalizes non-agent-scoped colon session keys passed to `openclaw sandbox explain` into agent-scoped session keys and adds regression tests for Telegram slash session channel and elevated allowlist output.

## Best Possible Solution

Land one coherent sandbox explain fix that wraps non-agent-scoped colon keys, resolves built-in channel IDs without requiring an active plugin registry, preserves already agent-scoped keys and `global`, and keeps the end-to-end assertions for `sessionKey`, `elevated.channel`, and `allowedByConfig`.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `telegram:slash:<id>` remains unscoped on current main, `parseAgentSessionKey` then fails, and elevated diagnostics never get a channel to look up.

## Solution Assessment

No, not by itself. The session-key normalization is the right first half, but the branch should also include the static built-in channel fallback from #51584 or an equivalent resolver change before it can reliably fix CLI diagnostics.
