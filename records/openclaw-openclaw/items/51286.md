## Summary

Keep this PR open. Current main still returns non-agent-scoped colon session keys verbatim in `sandbox explain`, so the documented `telegram:slash:<userId>` diagnostic path still cannot infer `telegram` or report `allowedByConfig: true`. Recent Telegram metadata fixes help runtime target-session metadata, but they do not implement this PR’s CLI normalization.

## Best Possible Solution

Keep this PR as a focused implementation candidate. If maintainers want `openclaw sandbox explain --session telegram:slash:<id>` to remain supported as documented, rebase and land the normalization with the downstream `channel` and `allowedByConfig` assertions. If the intended contract is now agent-scoped or target-session keys only, update `docs/tools/slash-commands.md` and resolve the linked bug that way instead of closing this as already implemented.
