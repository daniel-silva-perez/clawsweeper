## Summary

Keep this PR open. Current main still returns non-agent-scoped colon session keys unchanged in `openclaw sandbox explain`, so `telegram:slash:<id>` cannot be inferred as Telegram and cannot satisfy `tools.elevated.allowFrom.telegram` through the CLI diagnostic path. Main has a separate Telegram native-command runtime fix that records agent-scoped slash session metadata, but that does not implement this PR’s specific backwards-compatible normalization for users running `sandbox explain --session telegram:slash:<id>`.

## Best Possible Solution

Keep this PR as a focused implementation candidate. If maintainers want `openclaw sandbox explain --session telegram:slash:<id>` to remain a supported diagnostic path, rebase the PR, keep the downstream `channel` and `allowedByConfig` assertions, and run the targeted sandbox-explain test before landing. If the intended contract is agent-scoped keys only, close the PR only after updating the slash-command/sandbox docs and resolving the linked open bug accordingly.
