## Summary

Keep open. Current main still treats Telegram `message_reaction` updates as passive session context: authorized added emoji reactions are enqueued as system events, but the handler does not dispatch a message or request a session heartbeat wake. The related semantics PR is still open and dirty, so it is not an implemented fix on main.

## What This Changes

Honor Telegram inbound reaction wake behavior so added emoji reactions can trigger an agent turn instead of only appending context.

## Best Possible Solution

Keep this issue open and fix it in the Telegram plugin boundary: after an authorized, configured inbound reaction is enqueued, request a session-scoped agent/heartbeat wake through a plugin-safe seam. Preserve passive notifications by default, document the supported opt-in config, and keep broader reaction semantics/custom emoji policy coordinated with #47677/#47687.

## Reproduction Assessment

Yes. The issue has multiple live field reproductions, and the source-level reproduction is high-confidence: invoke the Telegram `message_reaction` handler with an added emoji and it calls `enqueueSystemEvent` without any wake or dispatch path.

## Solution Assessment

No, current main is not the best solution because it records the reaction but leaves the agent asleep. The maintainable fix is a narrow Telegram-owned opt-in wake after authorization/routing, or an updated #47687 if maintainers want the broader semantics design to carry this bugfix.
