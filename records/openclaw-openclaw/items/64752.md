## Summary

Keep this issue open. Current main still treats Telegram `message_reaction` updates as authorized system-event notifications only; it does not schedule an agent/heartbeat turn, and recent issue discussion reports the same behavior on the latest v2026.4.26 release.

## Best Possible Solution

Keep this issue open. Implement a narrow Telegram-owned opt-in reaction wake path: preserve passive reaction notifications by default, define or honor a documented wake setting, enqueue the reaction context, then request a session-scoped agent/heartbeat turn for the resolved route. Add regression tests proving a reaction-only update wakes the agent when configured and remains context-only when disabled; leave richer reaction semantics and custom emoji policy to #47677 unless maintainers intentionally merge the designs.
