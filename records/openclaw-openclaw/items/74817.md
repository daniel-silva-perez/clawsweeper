## Summary

Keep open. Current main still lacks a Feishu bot-authored-message policy or relay implementation, and the broadcast `@all` behavior is real current behavior that is already being debated in open #75001 rather than resolved here.

## What This Changes

The issue requests Feishu handling for `@所有人`/`@all` mentions plus a built-in way to avoid or intentionally manage multi-bot group behavior when `requireMention` is disabled.

## Best Possible Solution

Split the work by Feishu semantics: keep #75001 as the canonical `@all` discussion, and have maintainers decide whether standard OpenClaw should add conservative app-sender filtering, document the Feishu platform limitation, or support a relay-style Feishu feature. If filtering is chosen, carry `sender_type` into `FeishuMessageContext`, add an explicit account/group policy with compatibility-preserving defaults, and update schema, dispatch tests, and docs together.

## Reproduction Assessment

Unclear. Static inspection gives a high-confidence reproduction for broadcast-only `@all` not satisfying mention gating, but the multi-bot echo-loop half needs live Feishu payloads or logs because the reporter's follow-up says Feishu does not deliver bot-to-bot events to standard bots.

## Solution Assessment

Unclear. A narrow `sender_type: "app"` ignore policy could be maintainable if those events are delivered, but the relay research and open #75001 mean maintainer product/API direction should come before an automated fix PR.
