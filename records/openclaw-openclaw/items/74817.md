## Summary

Keep open. Current main still does not provide a general Feishu `ignoreOtherBots`/`allowBots` policy, and broadcast-only `@all` is intentionally excluded from mention detection and separately tracked by #75001. The reporter's follow-up usefully points to a fork with a relay implementation, but that makes the remaining direction a Feishu product/API decision rather than something already fixed in core.

## What This Changes

Request that Feishu `@所有人` group mentions reach the bot and that multi-bot Feishu groups with `requireMention:false` avoid bot-to-bot echo loops, potentially through an ignore-other-bots policy or relay/broadcast behavior.

## Best Possible Solution

Split the remaining work into clear Feishu semantics: keep #75001 as the canonical `@all` discussion, and decide whether standard OpenClaw should add an explicit bot-authored-message policy or a relay/broadcast feature. If app-sender filtering is desired, carry `sender_type` into `FeishuMessageContext`, add account/group config with a compatibility-preserving default, and cover group dispatch, schema, and docs.

## Reproduction Assessment

Unclear. Source inspection gives a high-confidence static path for the current behavior, but there is no live Feishu event payload or log here proving whether bot-authored group messages are delivered to standard OpenClaw in the reporter's environment.

## Solution Assessment

Unclear. A narrow `sender_type: "app"` filter would be maintainable if Feishu sends those events, but the reporter's relay conclusion and the separate `@all` semantics issue mean maintainer product direction should come before an automated fix PR.
