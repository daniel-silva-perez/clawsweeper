## Summary

Keep open. Current main still lacks a Feishu policy for suppressing bot-authored inbound messages, and the `@all` half is already being tracked separately in #75001, so this needs maintainer product direction rather than cleanup closure.

## What This Changes

The issue requests Feishu support for `@所有人`/`@all` delivery semantics and a configurable way to avoid multi-bot echo loops when `requireMention` is disabled.

## Best Possible Solution

Keep #75001 as the canonical `@all` semantics discussion, then make a Feishu maintainer decision on the bot-to-bot side: either add a compatibility-preserving app-sender suppression policy by carrying `sender_type` through `FeishuMessageContext`, or explicitly scope relay/shared-history behavior as a separate Feishu plugin feature informed by the linked fork.

## Reproduction Assessment

Unclear. Source and tests give a high-confidence static path for the current `@all` suppression, but the multi-bot echo-loop side needs a live Feishu `im.message.receive_v1` payload or log because the reporter’s follow-up says Feishu suppresses bot-to-bot delivery at the platform layer.

## Solution Assessment

Unclear. A narrow `sender_type: "app"` filter would be maintainable if Feishu delivers those events, but the linked relay research and open `@all` issue mean maintainer product direction should come before an automated fix.
