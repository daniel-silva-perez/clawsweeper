## Summary

Keep open. Current main still has a real Feishu gap for the multi-bot echo-loop half: Feishu events can identify app-authored senders, but the inbound group path does not filter bot/app messages before dispatch when mention gating is disabled. The broadcast `@all` half is already documented and tested as not being a bot mention, so that part needs clearer expectation-setting unless Feishu sends payloads that identify the bot directly.

## What This Changes

Request Feishu support for treating group-wide `@所有人` as a bot trigger and for preventing multi-bot group echo loops when `requireMention:false` is configured.

## Best Possible Solution

Add Feishu-local bot-authored message suppression, aligned with the existing Slack/Discord `allowBots` default-false model, and document that broadcast-only `@all` is not a bot mention unless Feishu sends a direct bot mention or reliable broadcast metadata that OpenClaw can safely interpret.

## Reproduction Assessment

Yes for the echo-loop half: a Feishu group event with `sender.sender_type: "app"`, an admitted group, and `requireMention:false` has no current bot-authored drop before dispatch. The `@all` half is only partially reproducible from source/tests because actual delivery depends on Feishu’s event payload.

## Solution Assessment

Yes for a narrow repair: the maintainable fix belongs in the Feishu plugin’s inbound filtering/config/docs/tests, not in core. The safer direction is an explicit `allowBots` opt-in rather than making `@all` or bot-authored messages trigger implicitly.
