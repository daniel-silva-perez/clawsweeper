## Summary

Keep open. Current main still lacks user-role stripping for the memory plugin’s injected `<relevant-memories>` prefix in the Gateway history and Control UI paths, so the PR is still useful, but the current PR helper appears to remove all whitespace after the injected block and can still drop intentional indentation from the real user prompt.

## What This Changes

The PR adds a shared injected-memory-prefix sanitizer, wires it into Gateway and Control UI user-message display paths, adds regression tests, and updates the changelog.

## Best Possible Solution

Land a narrow shared sanitizer that recognizes only the memory plugin’s injected leading prefix/preamble, removes that prefix plus the generated separator, preserves ordinary user-authored tags and indentation, and covers Gateway history, Control UI extraction, and normalized grouped rendering with regression tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: `memory-lancedb` returns the leading `<relevant-memories>` block as `prependContext`, while the Gateway and UI user-message display paths do not strip that prefix.

## Solution Assessment

No, not quite. The PR has the right narrow shape and avoids arbitrary tag stripping, but the helper should preserve user-authored leading whitespace after an injected prefix before this is the best mergeable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve prompt indentation after removing memory prefix:** `src/shared/text/assistant-visible-text.ts:591-593`
  - body: `stripInjectedRelevantMemoriesPrefix` skips every whitespace character after the closing memory tag.
