## Summary

Keep this issue open. Current main still updates the configured default model separately from persisted session model state: `openclaw models set` writes config only, while session resolution and list/status display can continue to prefer stored runtime or override fields until an explicit patch/reset path clears them. The related status-display items do not provide a merged canonical fix for global default propagation into active sessions.

## What This Changes

The issue requests a refresh or migration path so existing active or persistent sessions can adopt a new global model default after `openclaw models set <model>` changes it.

## Best Possible Solution

Add a deliberate refresh path for sessions after a global default change: sessions carrying only stale runtime/default-derived identity should be able to move to the new default, while intentional per-session user overrides remain unchanged unless the user explicitly resets them.

## Reproduction Assessment

Yes. The high-confidence reproduction path is static from current source: create a session row with stored model metadata, change the default via `models set`, and the command has no session refresh path while session resolution/display can still prefer the stored metadata.

## Solution Assessment

Unclear. A session refresh or migration mechanism is the right direction, but the narrowest maintainable fix needs a product decision about when `models set` may mutate existing sessions versus requiring an explicit reset/migrate action.
