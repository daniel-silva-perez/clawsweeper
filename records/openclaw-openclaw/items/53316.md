## Summary

Keep open. The reported routing bug is still present on current main, but this PR patches the old iMessage helper body instead of the shared parser that current iMessage and BlueBubbles runtime paths use.

## What This Changes

The PR adds decimal-only guards before parsing iMessage `chat_id` values and adds iMessage regression tests for hex `chat_id` rejection plus allowlist false-match prevention.

## Best Possible Solution

Land a narrow shared-parser fix in `src/channels/plugins/chat-target-prefixes.ts` that requires decimal-only `chat_id` values before parsing, throws a neutral invalid-`chat_id` error for send targets, silently rejects invalid allowlist `chat_id` entries, covers shared/iMessage/BlueBubbles regressions, and adds an appropriate changelog fix credit.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Port validation to the shared chat target parser:** `extensions/imessage/src/target-parsing-helpers.ts:91-93`
  - body: On current main, `extensions/imessage/src/target-parsing-helpers.ts` is only a re-export and the live iMessage plus BlueBubbles paths call `src/channels/plugins/chat-target-prefixes.ts`. Adding the guard only to the old helper body leaves current runtime parsing able to truncate values like `chat_id:2abc` to `2`, so the validation needs to move to the shared helper.
  - confidence: 0.94
- **[P3] Broaden the invalid chat_id error hint:** `extensions/imessage/src/target-parsing-helpers.ts:91-93`
  - body: The new guard rejects every non-decimal `chat_id` value, including negative numbers, decimals, empty values, and arbitrary handles, but the error always says to use `chat_identifier` for hex identifiers. Use a neutral decimal-only error and optionally add the hex hint only for hex-looking input.
  - confidence: 0.83
