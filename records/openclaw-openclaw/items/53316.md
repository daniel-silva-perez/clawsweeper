## Summary

Keep open. The routing/privacy bug is still present on current main, but this PR is stale against the current parser layout and would not fix the live shared parser used by iMessage and BlueBubbles.

## What This Changes

The PR adds decimal-only guards to iMessage `chat_id` parsing plus iMessage regression tests for rejecting hex `chat_id` values and preventing allowlist false matches.

## Best Possible Solution

Land a narrow shared-parser fix in `src/channels/plugins/chat-target-prefixes.ts` that requires decimal-only `chat_id` values before parsing, throws a neutral invalid-`chat_id` error for send targets, silently rejects invalid allowlist `chat_id` entries, preserves `chat_guid` and `chat_identifier` behavior, and adds focused shared/iMessage/BlueBubbles regression coverage plus a user-facing changelog fix with reporter and contributor credit.

## Reproduction Assessment

Yes. Current main has a high-confidence static reproduction: JavaScript truncates the sample hex identifier to `2`, and the live shared send and allowlist parsers still pass that value through `Number.parseInt` without validating the full string.

## Solution Assessment

No. The PR's direction is correct, but the implementation is no longer the best fix because current main routes both iMessage and BlueBubbles through the shared SDK parser rather than the old iMessage-local helper body.
