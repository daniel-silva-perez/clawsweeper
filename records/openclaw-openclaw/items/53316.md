## Summary

Keep this PR open. Current main still parses leading-digit non-decimal `chat_id:` values with `Number.parseInt(value, 10)` in the shared chat-target parser, so the misrouting and allowlist false-match described by #17362 remain possible. The PR is useful but now stale because its original implementation target is only an SDK re-export on current main; the fix needs to move to the shared parser and cover the runtime allowlist path.

## What This Changes

The PR adds decimal-only validation before parsing iMessage `chat_id:` values and adds regression tests for hex target rejection and allowlist false matches.

## Best Possible Solution

Keep this PR open and update it, or replace it with a narrow maintainer branch, by adding decimal-only `chat_id:` validation in `src/channels/plugins/chat-target-prefixes.ts` for both target parsing and allowlist parsing, using a neutral invalid-`chat_id` error, adding iMessage and shared/BlueBubbles regression coverage, and adding a user-facing changelog fix entry because this affects message routing and allowlist behavior.

## Risks / Open Questions

- The shared parser is also used by BlueBubbles and plugin SDK consumers, so the final fix should be generic and covered outside the old iMessage-only helper body.
