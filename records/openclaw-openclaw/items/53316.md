## Summary

Keep this PR open. Current main still parses leading-digit non-decimal `chat_id:` values with `Number.parseInt(value, 10)` in the shared chat target helper, so the misrouting and allowlist false-match reported in #17362 remain possible. The PR's fix is still useful, but it now needs to be ported from the old iMessage helper body to the shared SDK-backed parser.

## Best Possible Solution

Keep this PR open and update it for current main: add a decimal-only guard in `src/channels/plugins/chat-target-prefixes.ts` for both `parseChatTargetPrefixesOrThrow` and `parseChatAllowTargetPrefixes`, use a neutral invalid-`chat_id` error message with optional `chat_identifier:` guidance, add regression coverage for the thrown target path and allowlist false-match path, and add a user-facing changelog fix entry because this affects message routing and access control.
