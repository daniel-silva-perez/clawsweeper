## Summary

Keep open. Current main still has the non-aborted fallback signal in the gateway stop path, and the PR's one-line change addresses a real lifecycle consistency issue without introducing an evident functional or security regression.

## What This Changes

The PR replaces the `stopChannel` fallback `AbortController` signal with an already-aborted `AbortSignal` when invoking `stopAccount`.

## Best Possible Solution

Land the narrow fallback change, ideally with a focused `server-channels.test.ts` regression that asserts `stopAccount` receives `abortSignal.aborted === true` when no active controller exists.

## Reproduction Assessment

Yes. A high-confidence reproduction is to stop a configured plugin account that has `gateway.stopAccount` but no active task/controller; current main passes `new AbortController().signal`, whose `aborted` flag is false.

## Solution Assessment

Yes. Using `AbortSignal.abort()` is the narrowest maintainable fix because it matches the primary path after `abort?.abort()`; a regression test would strengthen the PR without changing the solution.
