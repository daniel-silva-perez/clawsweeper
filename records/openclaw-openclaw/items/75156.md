## Summary

Keep this open. Current main still has the reported failure shape: mixed text+media block payloads are recorded as sent media, and the final payload dedupe can then strip the attachment while leaving the text; Telegram’s delivery layer itself supports captioned media, so the narrow fix belongs in auto-reply block/finalization rather than Telegram transport.

## What This Changes

The issue reports that Telegram direct-chat replies containing both visible text and a `MEDIA:` attachment can deliver the text while dropping the attachment on the first send.

## Best Possible Solution

Add a focused auto-reply regression fix that only treats block-streamed media as dedupe-safe when that block payload was media-only or otherwise definitely delivered as media, while preserving media-only duplicate suppression and leaving Telegram captioned-media delivery unchanged.

## Reproduction Assessment

Yes. The issue includes live Telegram DM steps, and current source gives a high-confidence unit reproduction: enqueue or model a mixed text+media block payload, have `getSentMediaUrls()` return that media, then `buildReplyPayloads` can return the final text with the media stripped.

## Solution Assessment

Yes. The reporter’s proposed direction is the narrowest maintainable fix: adjust block reply sent-media accounting and add regression coverage, rather than changing Telegram delivery, which already supports text captions on media sends.
