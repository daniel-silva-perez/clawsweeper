## Summary

Keep this open. Current main still has the reported failure shape: mixed text-plus-media block payloads are recorded as sent media, and final payload dedupe can strip that media while preserving the text. There is also an open PR that uses closing syntax for this issue, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Telegram direct-chat replies containing visible text plus a MEDIA attachment can deliver only the text on the first send because mixed block-streamed media may be treated as already delivered before final payload delivery.

## Best Possible Solution

Land a narrow auto-reply regression fix that records block-streamed media as dedupe-safe only when the block payload was media-only or otherwise definitely delivered as media, while preserving media-only duplicate suppression and Telegram captioned-media delivery.

## Reproduction Assessment

Yes. The issue gives live Telegram DM steps, and current main gives a high-confidence source reproduction: a mixed text+media block payload can add its media to sentMediaUrls, after which final payload dedupe can strip that media and leave only text.

## Solution Assessment

Yes. The proposed direction is the narrowest maintainable fix: adjust block reply sent-media accounting and add regression coverage, rather than changing Telegram transport code that already supports captioned media sends.
