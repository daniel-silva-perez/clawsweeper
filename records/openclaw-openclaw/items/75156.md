## Summary

Keep this open. Current main still has the reported failure shape in the block reply/final payload path, and the linked closing PR #75180 is open and unmerged, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Telegram direct-chat replies that combine visible text with a `MEDIA:` attachment should deliver both parts on the first assistant reply instead of dropping the attachment after block-stream media dedupe.

## Best Possible Solution

Review and land PR #75180, or an equivalent narrow patch, that changes block reply sent-media accounting so mixed text plus `MEDIA:` blocks do not prematurely mark their media as delivered while media-only duplicate suppression remains intact.

## Reproduction Assessment

Yes. The issue includes live Telegram DM steps, and current source gives a high-confidence unit reproduction: a mixed text+media block payload is recorded in `getSentMediaUrls()`, then final payload dedupe can strip that same media while leaving text.

## Solution Assessment

Yes. The proposed direction is the narrowest maintainable fix because Telegram already supports captioned media; the repair belongs in block reply media accounting plus regression coverage, not in the Telegram transport.
