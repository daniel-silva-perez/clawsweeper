## Summary

Keep this open. Current main still has the reported mixed text-plus-media dedupe path, and an open unmerged PR already references this issue with closing syntax, so the issue should remain open until that PR is reviewed and merged or rejected.

## What This Changes

The issue requests that Telegram assistant replies containing visible text plus a `MEDIA:` attachment deliver both parts on the first send instead of stripping the attachment during final reply dedupe.

## Best Possible Solution

Review PR #75180 as the active implementation candidate and either land its narrow media-only sent-media accounting fix with regression coverage or close it with a replacement that makes the same block/final reply boundary explicit.

## Reproduction Assessment

Yes. The issue includes live Telegram DM steps, and current source gives a high-confidence unit reproduction path: a mixed text+media block payload can populate `getSentMediaUrls()`, after which final payload construction can strip the same media while preserving text.

## Solution Assessment

Yes. The best fix is in auto-reply block sent-media accounting and regression tests, not in Telegram delivery, because Telegram already has captioned media delivery paths and tests.
