## Summary

Keep this issue open. Current main still has the reported mixed text-plus-media block-stream dedupe path, and an open unmerged PR already references this issue with closing syntax, so cleanup should wait for that PR to land or be rejected.

## What This Changes

Ensure Telegram direct replies that contain visible text plus a `MEDIA:` attachment deliver both parts on the first send.

## Best Possible Solution

Review and land PR #75180, or an equivalent narrow fix, so block streaming only marks media as dedupe-safe when delivery actually sent that media and regression coverage locks down mixed text-plus-media final delivery.

## Reproduction Assessment

Yes. The reporter supplied live Telegram DM steps and local validation, and current source gives a high-confidence unit reproduction path: stream a mixed text-plus-media block payload, observe that media enter `getSentMediaUrls()`, then see final payload media stripped while text remains.

## Solution Assessment

Yes. The narrow maintainable fix belongs in block-reply sent-media accounting with regression tests, because Telegram delivery already supports captioned media and the stripping occurs in generic final-payload dedupe.
