## Summary

Keep this open. Current main still records media from mixed text-plus-media block replies as already sent, then later dedupes final reply media against that set; an open unmerged PR already references this issue with closing syntax, so the issue should stay open until that PR is reviewed and merged or rejected.

## What This Changes

Fix Telegram direct-chat replies so a first assistant response containing visible text plus a `MEDIA:` attachment delivers both the text and the attachment instead of dropping the attachment during reply dedupe.

## Best Possible Solution

Review PR #75180 as the active implementation candidate and land its narrow media-only sent-media accounting fix with regression coverage, or close it with an equivalent replacement that makes the block-stream versus final-reply dedupe boundary explicit.

## Reproduction Assessment

Yes. The reporter supplied live Telegram DM steps and local validation, and current source gives a high-confidence unit reproduction path: send a mixed text-plus-media block payload, observe `getSentMediaUrls()` include that media, then build final payloads with block streaming enabled and see the same media stripped while text remains.

## Solution Assessment

Yes. The narrow maintainable fix belongs in block reply sent-media accounting and regression tests, not in Telegram delivery, because current Telegram delivery already has captioned-media support and the observed stripping occurs in generic final payload dedupe.
