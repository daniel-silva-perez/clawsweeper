## Summary

Keep this open. Current main still has the reported failure shape: mixed text+media block payloads are recorded as sent media, and final payload dedupe can then strip the attachment while preserving the text. The linked open PR #75180 uses closing syntax for this issue, so the issue should remain open until that PR is reviewed and either merged or closed.

## What This Changes

Ensure Telegram direct-chat assistant replies containing visible text plus a `MEDIA:` attachment deliver both the text/caption and the attachment on the first send.

## Best Possible Solution

Keep the issue open until a narrow block-reply finalization fix is merged: only treat block-streamed media as dedupe-safe when that block payload was media-only or otherwise definitely delivered as media, and add focused tests around the block pipeline plus final payload builder.

## Reproduction Assessment

Yes. The issue includes live Telegram DM steps, and current main gives a high-confidence source-level reproduction: enqueue a mixed text+`MEDIA:` block payload, let the block pipeline record its media URL, then final payload dedupe can strip that URL while leaving the text.

## Solution Assessment

Yes. The best fix direction is to adjust block reply sent-media accounting and regression coverage; changing Telegram delivery would be broader and unnecessary because Telegram already supports captioned media sends.
