## Summary

Keep this open. Current main still has the reported failure shape: mixed text+media block replies are recorded as sent media, and the later final-payload dedupe can strip that same media while leaving the text; Telegram’s delivery layer already supports captioned media, so the fix belongs in auto-reply block/finalization accounting.

## What This Changes

Telegram direct-chat replies containing visible text plus a `MEDIA:` attachment can lose the attachment on the first delivery when block-stream media dedupe treats the media as already sent.

## Best Possible Solution

Add a focused auto-reply regression fix that only treats block-streamed media as dedupe-safe when that media was definitely delivered as media, while preserving media-only duplicate suppression and leaving Telegram captioned-media delivery unchanged.

## Reproduction Assessment

Yes. The issue includes live Telegram DM steps, and the current source gives a high-confidence unit path: enqueue a mixed text+media block payload, let `getSentMediaUrls()` include that media, then `buildReplyPayloads` can strip the media from the final payload while preserving the text.

## Solution Assessment

Yes. The narrow maintainable solution is to adjust block reply sent-media accounting and add regression coverage; changing Telegram transport would be the wrong layer because it already sends media with captions.
