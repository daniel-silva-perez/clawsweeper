## Summary

Keep open. Current main has overlapping pre-delivery and finalization hooks plus a built-in empty-response continuation retry, but it still does not expose a `message:before_send`-style content-quality gate that can reject a completed response and route that rejection into the configured model fallback chain.

## What This Changes

Request a pre-delivery/finalization hook that can inspect generated content and reject empty or low-quality responses so OpenClaw retries through model fallbacks instead of delivering them.

## Best Possible Solution

Define a documented plugin/runner decision point before terminal payload acceptance that exposes content, provider/model, finish/stop reason, usage, run/session context, and a bounded decision result, then map replay-safe rejection into the existing fallback chain while leaving `message_sending` as delivery policy.

## Reproduction Assessment

Unclear. The issue gives a concrete provider/model/channel, and current tests simulate empty final answers, but there is no deterministic Gemini prompt/log here; code inspection is enough to verify the requested hook/fallback contract is absent.

## Solution Assessment

Unclear. A delivery-stage `message:before_send` is not clearly the safest implementation point because current delivery hooks only rewrite/drop payloads; a runner/finalization failover seam before terminal acceptance is likely safer.
