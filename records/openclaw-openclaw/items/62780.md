## Summary

Keep this issue open. Current main has adjacent pieces: `message_sending` can rewrite or cancel outbound delivery, and the model fallback runner can advance on built-in empty/incomplete result classifications. The requested surface is still missing: a pre-send/content-quality hook that can reject a completed model response with model/finish metadata and cause the configured fallback chain to run.

## Best Possible Solution

Keep the issue open and narrow it to an additive model-outcome/content-quality gate in the fallback decision path. A good implementation would expose bounded provider/model/finish metadata and visible response content, allow a typed rejection such as `format` or `empty_content`, feed that classification into `runWithModelFallback`, and document how it differs from delivery-only `message_sending` cancellation.
