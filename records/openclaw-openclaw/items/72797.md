## Summary

Keep open. Current main still does not implement a `resolve_exec_env` hook, and the PR is useful source work, but the patch needs maintainer review because it introduces a public plugin SDK seam that can bypass the existing host exec environment security policy.

## What This Changes

The PR adds a typed `resolve_exec_env` plugin hook, exposes a hook-runner method, calls it from exec environment construction, and adds focused hook tests.

## Best Possible Solution

Keep the PR open for maintainer review and revise it into an additive, documented exec-environment seam that preserves host env security, has explicit trusted-plugin or reserved-key semantics, passes the needed channel/account/thread/sender context, and is tested across gateway, node, and sandbox execution. If maintainers decide env injection is too broad, redirect the design to a narrower tool-execution metadata hook that also covers #73343.

## Reproduction Assessment

Not applicable as a feature PR, but there is a high-confidence inspection path: current main lacks the hook, and the PR patch visibly merges hook output after env sanitization. No runtime reproduction is required to see the policy bypass in the proposed diff.

## Solution Assessment

No, not as-is. The hook direction is plausible, but the safest maintainable solution should filter or otherwise govern reserved env keys and align the exec-env API with the related sender/account/thread/MCP context work.
