## Summary

This PR should stay open. Current main does not expose `resolve_exec_env`, does not return `runResolveExecEnv` from the hook runner, and constructs the exec environment without a plugin env injection point. The existing `before_tool_call` hook can rewrite generic tool params, but its context does not include channel/provider fields, so it does not obviously replace the PR's channel-specific exec-env seam. The PR is recent and active, and the discussion already narrowed the remaining review point to the trusted-plugin/security tradeoff around env clobbering.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to decide whether channel-specific exec environment injection belongs as a dedicated typed hook or as an extension of existing tool hooks; if the dedicated hook is accepted, land it with explicit docs/tests for host/sandbox/node behavior, reserved-key policy, merge ordering, error isolation, and the trusted-plugin security boundary.

## Risks / Open Questions

- Security review should focus on whether `resolve_exec_env` may set reserved host-exec keys such as `PATH`, `LD_*`, `DYLD_*`, `HOME`, proxy, TLS, or credential-routing variables after normal exec env sanitization, and whether that behavior must differ for gateway, node, and sandbox hosts.
- The PR adds a new public plugin hook name/type. If accepted, docs and SDK export/API generation expectations should be checked so plugin authors can discover the hook and its trust boundary.
- The proposed merge order gives later plugins override priority for the same env key. That behavior is plausible, but it should be documented and tested because it creates cross-plugin precedence semantics on a code-execution path.
