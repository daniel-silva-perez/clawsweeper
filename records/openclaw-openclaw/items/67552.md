## Summary

Current main still does not implement the requested shared tool-response freshness contract. The existing surfaces use ordinary `content` plus runtime-only `details`, and the named examples either return tool-specific payloads or ad hoc cache fields rather than a standardized `cached`/`age_ms`/`ttl_ms`/source timestamp envelope. This should stay open for maintainer design because making freshness agent-visible crosses the core tool-result, SDK, transcript, compaction, and provider replay boundaries.

## What This Changes

Request a standardized, backwards-compatible tool-response freshness metadata contract exposing whether data was cached, how old it is, its TTL, and its original fetch timestamp.

## Best Possible Solution

Define an optional, backwards-compatible freshness contract in the core/SDK tool-result surface, explicitly deciding what is model-visible versus runtime-only, then document it for plugin authors and adopt it in high-value cached tools such as `session_status`, `memory_search`, cron status/runs, and web/provider fetch/search tools with focused regression coverage.
