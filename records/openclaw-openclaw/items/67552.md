## Summary

Current main still does not implement the requested standardized tool-response freshness metadata contract. There are narrow freshness-related fields such as session `totalTokensFresh` and ad hoc cached flags in web tools, but no shared `ToolFreshnessMetadata` type, no `data`/`freshness` envelope, and no adoption in the named `session_status`, cached web/search/fetch, or cron tool-result paths.

## Best Possible Solution

Keep this issue open for maintainer design. The best path is a backwards-compatible optional freshness metadata contract in the core/SDK tool-result surface, with clear rules for what is model-visible versus runtime-only, public docs for plugin authors, regression tests for replay/compaction/transcript behavior, and targeted adoption in `session_status`, `memory_search`, cron status/runs, and cached web/provider tools.

## Risks / Open Questions

- The right design crosses the model-visible `content` versus runtime-only `details` boundary, provider replay, compaction, transcript persistence, middleware validation, and plugin SDK compatibility.
- A blanket `data`/`freshness` envelope could change model-visible tool-result bytes for many tools, so it needs explicit backwards-compatibility rules and targeted adoption tests.
