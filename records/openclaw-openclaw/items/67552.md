## Summary

Current main does not implement the requested shared tool-response freshness contract. There is no `ToolFreshnessMetadata` type or standardized `data`/`freshness` envelope, and the named tool surfaces still return ordinary `content` plus runtime `details` without `cached`, `age_ms`, `ttl_ms`, or source timestamp metadata.

## Best Possible Solution

Keep this issue open for maintainer design. The best path is a backwards-compatible optional freshness metadata contract in the core/SDK tool-result surface, with explicit rules for what is model-visible versus runtime-only, docs for plugin authors, regression tests, and targeted adoption in `session_status`, `memory_search`, cron status/runs, and cached web/provider tools.
