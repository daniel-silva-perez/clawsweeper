## Summary

Keep open. Current main still lacks `${VAR:-default}` fallback support, so the PR is not obsolete, but the latest diff still treats defaulted refs as unresolved env references in the helper that gates `env.vars` injection. That leaves the config chaining regression from the prior review unresolved and should block merge until the PR distinguishes required refs from fallback-resolvable refs.

## What This Changes

The branch adds a shared brace scanner plus updates config env substitution, env-ref write-back preservation, and unit tests to support `${VAR:-default}` fallbacks.

## Best Possible Solution

Land fallback support only after the parser or helper API can separate required unresolved refs from default-resolvable refs, and add a regression test where `env.vars.API_KEY` uses `${UPSTREAM:-demo}` and another config field consumes `${API_KEY}`.

## Reproduction Assessment

Yes. A high-confidence reproduction path is `env.vars.API_KEY = "${UPSTREAM:-demo}"` with another string field set to `${API_KEY}`: the PR still reports the env.vars value as an env reference, so the pre-substitution injection path skips it and the consumer remains unresolved.

## Solution Assessment

No. The feature direction is reasonable, but this is not the best merge shape until defaulted refs bypass the unresolved-ref skip or are resolved before `applyConfigEnvVars`; a small distinct token/helper such as a required-ref check would be safer.
