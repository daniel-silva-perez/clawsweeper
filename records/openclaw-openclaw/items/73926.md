## Summary

Keep open. Current main has a partial bare `/new`/`/reset` improvement, but the embedded runner still has the exact remaining empty-dispatch shape from the report: blank replay messages count as content before replay sanitization drops them, and runtime-only submissions can still call the provider with an empty visible prompt. Two open unmerged PRs already reference this issue with closing syntax, so the issue should stay open until one canonical fix lands.

## What This Changes

The issue requests that `/new` and `/reset` never dispatch an empty prompt plus empty provider-replayable message list, especially during active-memory/runtime-context races.

## Best Possible Solution

Add a single embedded-runner dispatch-boundary guard that evaluates the sanitized provider-replayable prompt state: non-blank visible prompt, images, or non-empty replayable messages. For bare `/new`/`/reset`, use the existing startup/reset prompt or marker path; for other empty submissions, skip/fail terminally with a diagnostic instead of forwarding an empty provider request.

## Reproduction Assessment

Yes. A source-level reproduction is clear: use an embedded attempt with an empty prompt and a stale blank user replay message; the current guard treats `messages.length > 0` as content, replay sanitization then drops the blank message, and Anthropic-style conversion can send an empty `messages` payload.

## Solution Assessment

Yes for the general direction, but the exact open PR choice still needs maintainer review. A narrow guard based on sanitized replayable content is safer than injecting a generic greeting into every empty prompt path because it preserves transcript hygiene and legitimate history/media submissions.
