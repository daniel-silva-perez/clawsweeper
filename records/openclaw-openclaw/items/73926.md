## Summary

Keep open. Current main fixes the bare-reset marker and blank visible prompt/replay-history subcases, and the formerly paired PR #73929 is closed as superseded by #74168, but the active-memory/runtime-only path still intentionally allows an empty visible prompt to reach the embedded runner. That leaves the issue’s provider-visible empty-input boundary only partially addressed.

## What This Changes

The issue asks OpenClaw to prevent `/new` or `/reset` active-memory races from dispatching an empty prompt and empty message list to Anthropic-compatible providers.

## Best Possible Solution

Define a provider-safe runtime-only contract at the embedded-runner or provider-adapter boundary so hidden runtime context can remain outside the visible transcript while provider calls never contain no model-visible input. The end state should preserve active-memory and heartbeat semantics while adding one explicit marker, skip path, or lower-level hidden-input mechanism that providers accept.

## Reproduction Assessment

Yes at source level. A run with non-empty runtime context and blank `transcriptPrompt` resolves to `prompt: ""`, bypasses the blank visible-prompt skip because `runtimeOnly` is true, and current tests preserve the active-memory blank transcript shape.

## Solution Assessment

No for the full issue. Current main is the right narrow fix for blank visible user prompts and bare chat resets, but it does not settle whether runtime-only active-memory runs may still submit an empty visible prompt to providers.
