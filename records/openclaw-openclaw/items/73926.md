## Summary

Keep open. The issue is paired with open PR #73929, which explicitly tracks this bug, and current main only partially covers the failure class: blank visible prompt submissions are now skipped, but runtime-only empty prompt submission remains an intentional, tested path.

## What This Changes

Prevent `/new` or `/reset` plus active-memory races from dispatching an empty prompt and empty replay-message payload to Anthropic/OpenAI-style providers.

## Best Possible Solution

Keep this issue open while maintainers review #73929 against current main. The durable endpoint should be a single embedded-runner boundary that never sends provider-visible empty input, while preserving runtime context outside user-visible transcript history.

## Reproduction Assessment

Yes. The empty-payload condition is reproducible at the source level with a blank transcript prompt through runEmbeddedAttempt; current tests prove the blank visible-prompt skip and also prove the runtime-only empty prompt path still exists.

## Solution Assessment

Unclear. PR #73929 targets the reported stale blank replay-history class, but current main already has a broader blank visible-prompt guard and still leaves runtime-only empty prompts untouched, so the best next step is maintainer review of that PR rather than closing this issue.
