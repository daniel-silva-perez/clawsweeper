## Summary

Keep this PR open. Current main still does not implement create-with-content for `feishu_doc`: the create schema has no `content` field, the runtime create branch only calls `createDoc`, and current create tests do not cover inserting markdown after document creation. The PR remains a focused implementation candidate for #48373, but it needs maintainer review around tests, empty-content semantics, response shape, docs, and whether `writeDoc` is the right helper for a newly-created document.

## Best Possible Solution

Keep this PR open and review it as the implementation candidate for #48373. The maintainer-quality path is to add `content` to the create schema, populate the new document through a tested helper, define empty-content behavior, return a clean create response, avoid unnecessary Feishu API calls if practical, and update the Feishu doc skill documentation to match the final schema and permission behavior. If maintainers decide create should stay title-only, close this PR only after the docs and linked bug are resolved around the explicit two-step create-then-write workflow.

## Risks / Open Questions

- The PR needs a focused regression test proving `feishu_doc` create with `content` creates the document and inserts converted markdown in one tool call.
- The proposed truthiness check for `p.content` would skip an explicitly provided empty string; maintainers should define whether empty or whitespace-only content is valid, rejected, or equivalent to omitted content.
- Reusing `writeDoc` is plausible but it calls `clearDocumentContent` first, adding unnecessary Feishu API calls for a fresh document unless a create-specific append/write helper is used.
- The Feishu doc skill docs are stale around create parameters and requester permissions, and open PR #50357 proposes documenting the opposite product behavior, so maintainers should settle behavior and docs together.
