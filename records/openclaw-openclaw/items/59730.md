## Summary

Keep open. Current main still uses an explicit multi-config Vitest project graph, package scripts still call specific config files with `--config`, docs still describe split lanes/configs, and there is no `VITEST_MODE`-based unified configuration.

## Best Possible Solution

Keep this issue open unless maintainers explicitly reject the refactor. A good implementation would first design a typed config registry or factory that preserves the current shard/lane behavior, then migrate `scripts/test-projects*`, package scripts, CI workflow calls, docs, and config tests together so contributors get one discoverable entry point without losing the current targeted-test scheduling.
