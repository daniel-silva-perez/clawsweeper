## Summary

Keep open. Current main has not consolidated Vitest configuration: the root config still enumerates many project configs, package scripts still call explicit `--config` lanes, and the testing/CI docs describe the split shard model as current behavior. This is a broad test-infrastructure architecture decision rather than a narrow repair candidate.

## What This Changes

Request to consolidate the multiple Vitest configuration files into one unified mode-based or factory-backed configuration and update scripts, CI, and docs accordingly.

## Best Possible Solution

Decide the desired test-harness direction first. If maintainers want the DX improvement, consolidate declarations into a typed registry or shared factory that preserves current lane names, sharding, worker/cache/isolation behavior, package scripts, CI routing, and docs; otherwise document that the split config model is intentional and close by maintainer decision.

## Reproduction Assessment

Not applicable as a refactor request, not a runtime bug. The current state is directly verifiable on main: 95 Vitest config files remain, `rootVitestProjects` enumerates split configs, package scripts still pass explicit `--config` paths, and no `VITEST_MODE` implementation exists.

## Solution Assessment

Unclear. The proposed single `VITEST_MODE` config would reduce file discovery overhead, but it is not clearly the best solution unless it can preserve the existing sharded CI and local scheduling contracts; a registry/factory consolidation may be safer than collapsing every lane into one config file.
