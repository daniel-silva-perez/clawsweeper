## Summary

Keep open. The PR addresses behavior that still appears missing on current main, but the proposed suppressed-source fallback is not merge-ready because it delivers raw suppressed payloads outside the normal reply dispatcher normalization path, which can leak exact silent tokens such as `NO_REPLY` and skip response-prefix/channel transforms.

## What This Changes

The branch adds Telegram direct `/status` delivery, Telegram group/topic suppressed-source reply fallback wiring, a core suppressed-source callback, follow-up failure replies, topic-aware sequential keys, and focused regression tests.

## Best Possible Solution

Keep the PR active, but require the fallback path to preserve normal reply normalization semantics before delivery and add a regression for suppressed fallback payloads containing exact silent tokens. Maintainers should also explicitly approve the Telegram privacy behavior change or narrow the fallback to a safer subset.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Normalize suppressed replies before fallback delivery:** `src/auto-reply/reply/dispatch-from-config.ts:1050`
  - body: The new suppressed-source fallback calls `onSuppressedSourceReply` with the raw payload instead of routing it through the normal dispatcher normalization. That means an exact `NO_REPLY` final has sendable text and can be posted visibly by Telegram, and response prefixes/channel transforms are skipped. Please normalize or otherwise filter fallback payloads with the same rules used by `createReplyDispatcher` before invoking the channel callback.
  - confidence: 0.88
