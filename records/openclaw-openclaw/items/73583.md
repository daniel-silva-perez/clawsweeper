## Summary

Keep this draft PR open. Current main has a Telegram control-lane bypass for read-only status commands, but native Telegram `/status` still builds a normal native command context and sends it through `dispatchReplyWithBufferedBlockDispatcher`; the PR's intended direct status reply fast path is not implemented on main. The supplied PR context is focused to Telegram source/tests plus changelog and does not show unrelated security-sensitive changes.

## Best Possible Solution

Review and land this PR or an equivalent Telegram-plugin fix: add a status-specific branch in `extensions/telegram/src/bot-native-commands.ts` that resolves `/status` with `resolveDirectStatusReplyForSession` and sends it via `deliverReplies` directly, while preserving authorization, target session selection, topic threading/reply-to behavior, and keeping `/fast status` on the generic native-command route. Include the regression tests proposed in the PR before merge.
