## Summary

Keep open. Current main still lacks the Telegram direct `/status` path, the suppressed-source fallback hook, Telegram fallback delivery for suppressed group/topic output, and follow-up failure replies; the latest PR branch also addresses the prior normalization finding. The added independent 2026-04-30 reproduction against v2026.4.27 supports this as an active user-facing Telegram group/topic regression that needs maintainer review and CI/Testbox validation rather than cleanup closure.

## What This Changes

The PR routes Telegram native `/status` through direct status delivery, adds a core suppressed-source fallback used by Telegram group/topic replies, routes queued follow-up failures back to the originating topic, adjusts Telegram topic sequential keys, and adds focused regression tests.

## Best Possible Solution

Keep this PR active and have a maintainer review the latest branch, resolve mergeability, run the changed Telegram/core reply tests plus CI/Testbox, and explicitly decide whether Telegram should have this compatibility fallback or whether the docs/config semantics need to be updated with the shipped behavior.

## Reproduction Assessment

Yes. Current main source shows the suppressed group/topic paths have no fallback, and the 2026-04-30 reproduction reports inbound processing and model completion with no Telegram send event on v2026.4.27, then recovery after downgrading to v2026.4.26.

## Solution Assessment

Unclear but plausible. The PR is a narrow, tested implementation for the observed Telegram gaps and the earlier raw-payload normalization bug is fixed, but the default group visibility semantics are product/privacy-sensitive and need maintainer approval before merge.
