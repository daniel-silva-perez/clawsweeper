## Summary

Keep this issue open. Current main still appears to preserve the reported forced-onboarding path for an existing packaged/Umbrel-style config: the installer detects an existing config and runs repair/doctor work, but then falls through to the interactive setup branch. The large model-list and OpenRouter onboarding parts are mitigated on main, but the existing-config wizard regression remains unresolved.

## Best Possible Solution

Fix the installer or Umbrel setup entrypoint so an existing `OPENCLAW_CONFIG_PATH` or legacy config exits onboarding after repair/doctor handling instead of falling through to `openclaw onboard`. Add a focused installer regression test for the existing-config skip path, then verify the packaged Umbrel setup can filter large model catalogs and select OpenRouter.
