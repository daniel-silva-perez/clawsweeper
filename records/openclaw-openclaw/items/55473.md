## Summary

Keep open. Current main still treats startup secret activation failures as fatal, so the underlying channel-local degradation problem remains real, and the maintainer discussion already called that out. This PR is not ready to merge as-is: the branch is dirty against current main, and the proposed recovery path has correctness gaps around restarting recovered channels and preserving explicit disable writes during degraded startup.

## What This Changes

The PR adds a gateway startup degradation path for missing env-backed channel secrets, carries a recoveryConfig snapshot for later secrets.reload recovery, adds a merge-patch helper, and adds gateway reload regression coverage.

## Best Possible Solution

Keep the PR open, but rework it on top of current main by moving the channel-degradation logic into the current startup secret activator structure, preserving current secrets.reload restart and rollback behavior, and adding focused tests for startup degradation, allowlist/traversal fatal failures, explicit channel-disable persistence, and immediate channel recovery after secrets.reload.

## Reproduction Assessment

Yes. Configure an active channel such as Discord with `token: { source: "env", provider: "default", id: "DISCORD_BOT_TOKEN" }`, leave that env var unset, and start the gateway; current main's startup secret activator still follows the fatal startup path.

## Solution Assessment

No. The direction is reasonable, but this branch is not the best current implementation because it predates current main's aux-handler reload restart path and introduces recoveryConfig edge cases that can leave channels offline or undo operator disable writes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Restart recovered channels during secrets.reload:** `src/gateway/server.impl.ts:1350-1355`
  - body: In the degraded-startup path the channel was never started because the active snapshot has `enabled: false`. This reload handler swaps in the recovery config and returns success without diffing/restarting channel runtimes, so after the env secret appears the channel can remain offline even though `secrets.reload` reports ok. Route recovery through the current reload-plan stop/start path.
  - confidence: 0.88
- **[P2] Preserve explicit channel disables in recoveryConfig:** `src/secrets/runtime.ts:310-311`
  - body: When an operator explicitly sets `channels.<id>.enabled=false` during a degraded session, both the old degraded sourceConfig and the new sourceConfig already contain `enabled: false`, so this delta is empty and recoveryConfig keeps the original enabled state. A later `secrets.reload` can re-enable a channel the operator disabled; track that explicit disable or merge against the pre-degrade baseline.
  - confidence: 0.78
