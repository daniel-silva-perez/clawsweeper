## Summary

Keep open. Current main still only supports the native Signal JSON-RPC/SSE transport, while this PR is the active implementation candidate for Signal container REST/WebSocket support; however the PR head still has blocking raw payload logging and config-schema validation issues, so it needs maintainer review and branch updates rather than cleanup closure.

## What This Changes

The PR adds a Signal native/container adapter, a bbernhard REST/WebSocket client, `channels.signal.apiMode` docs/types/tests, caller rewiring, a changelog fragment, and a direct `ws` runtime dependency for the Signal plugin.

## Best Possible Solution

Ship a Signal plugin-local transport adapter that preserves native JSON-RPC/SSE behavior while adding documented bbernhard REST/WebSocket mode, with config types/schema/help/docs kept in sync, raw payload logging removed or redacted, focused tests for both modes, and live container smoke proof or an explicit maintainer waiver.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Remove raw Signal payload logging:** `extensions/signal/src/client-container.ts:202`
  - body: The bbernhard receive stream carries full Signal envelopes, including message bodies and sender metadata. Logging the first 200 characters of every inbound frame sends private chat content to gateway logs during normal operation; log only redacted metadata or gate this behind an explicit debug path.
  - confidence: 0.93
- **[P2] Validate the apiMode setting:** `src/config/types.signal.ts:68`
  - body: This exposes `channels.signal.apiMode` in the public type/docs, but the PR head does not add `apiMode` to `SignalConfigSchema`, which is strict. Users following the new docs with `apiMode: "container"` can fail config validation before adapter routing can use the setting.
  - confidence: 0.91
