## Summary

Keep open. Current main still only supports the native Signal JSON-RPC/SSE transport, so this PR is not obsolete; however PR head has blocking review issues around raw Signal payload logging and a public `apiMode` setting that is not accepted by the strict config schema.

## What This Changes

The PR adds a Signal native/container adapter, a bbernhard REST/WebSocket client, `channels.signal.apiMode` docs/types/tests, Signal caller rewiring, a changelog fragment, and a direct `ws` runtime dependency for the Signal plugin.

## Best Possible Solution

Ship a Signal plugin-local transport adapter that preserves native JSON-RPC/SSE behavior while adding documented bbernhard REST/WebSocket mode, with config schema/help/docs generated in sync, raw payload logging removed or redacted, focused tests for both modes, and live container smoke proof or an explicit maintainer waiver.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Remove raw Signal payload logging:** `extensions/signal/src/client-container.ts:202`
  - body: The receive stream logs the first 200 characters of every WebSocket frame before parsing. Those frames can include private message text and sender/group/attachment metadata, so normal container receive would write Signal content to gateway logs; log redacted metadata only or guard this behind explicit debug output.
  - confidence: 0.93
- **[P2] Validate the apiMode setting:** `src/config/types.signal.ts:68`
  - body: `apiMode` is added to public `SignalConfig` and docs, but the strict `SignalConfigSchema` at PR head still extends `SignalAccountSchemaBase` without adding it. A user setting `channels.signal.apiMode: "container"` can fail config validation before adapter routing sees the value.
  - confidence: 0.91

## Security Review

Status: needs_attention

Summary: The new WebSocket dependency and transport are purpose-aligned, but the receive path currently logs raw Signal payload snippets.

Concerns:

- **[medium] Raw Signal payloads are logged:** `extensions/signal/src/client-container.ts:202`
  - body: `streamContainerEvents` logs the first 200 characters of each inbound WebSocket frame before parsing; those frames can contain private Signal message text, sender numbers/UUIDs, group metadata, and attachment metadata, so this should be removed or redacted before merge.
