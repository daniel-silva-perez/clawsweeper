## Summary

Keep open. Current main still only supports native Signal JSON-RPC/SSE, so the container REST/WebSocket goal remains relevant, but the PR head is not merge-ready because it logs raw inbound Signal WebSocket payloads and exposes an apiMode setting that strict config validation does not accept.

## What This Changes

The PR adds a Signal native/container adapter, a bbernhard REST/WebSocket client, Signal caller rewiring, channels.signal.apiMode docs/types/tests, a changelog fragment, and a direct ws runtime dependency for the Signal plugin.

## Best Possible Solution

Ship a Signal plugin-local transport adapter that preserves native JSON-RPC/SSE behavior while adding documented bbernhard REST/WebSocket mode, with config schema/help/docs generated in sync, raw payload logging removed or redacted, focused tests for both modes, and live container smoke proof or an explicit maintainer waiver.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Remove raw Signal payload logging:** `extensions/signal/src/client-container.ts:202`
  - body: The receive stream logs the first 200 characters of every WebSocket frame before parsing. Those frames can include private message text and sender/group/attachment metadata, so normal container receive would write Signal content to gateway logs; log redacted metadata only or guard this behind explicit debug output.
  - confidence: 0.93
- **[P2] Validate the apiMode setting:** `src/config/types.signal.ts:68`
  - body: apiMode is added to public SignalConfig and docs, but the strict SignalConfigSchema at PR head still extends SignalAccountSchemaBase without adding it. A user setting channels.signal.apiMode: "container" can fail config validation before adapter routing sees the value.
  - confidence: 0.91
