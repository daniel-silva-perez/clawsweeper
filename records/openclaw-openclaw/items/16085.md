## Summary

Keep open. Current main still only supports the native Signal JSON-RPC/SSE transport, while this PR is the active implementation candidate for bbernhard REST/WebSocket container support. The branch is not ready to merge because the latest head still has blocking native-mode, startup, config/schema, SDK-boundary, security logging, and CI failures.

## What This Changes

The PR adds a Signal native/container adapter, a bbernhard REST/WebSocket client, Signal caller rewiring, `channels.signal.apiMode` docs/types/tests, a changelog fragment, and a direct `ws` runtime dependency for the Signal plugin.

## Best Possible Solution

Ship a plugin-local Signal transport adapter that preserves current native JSON-RPC/SSE behavior, adds documented bbernhard REST/WebSocket mode through accepted SDK runtime/config seams, validates `apiMode` in the strict schema and generated docs, removes sensitive logging, and lands with targeted tests plus live container proof or an explicit maintainer waiver.

## Reproduction Assessment

Yes. The user problem is source-verifiable: current main uses `/api/v1/rpc`, `/api/v1/check`, and `/api/v1/events`, while bbernhard documents `/v2/send` and WebSocket `/v1/receive/<number>`; the PR defects are also reproducible by static source and live check-run inspection. This read-only review did not run a live container smoke.

## Solution Assessment

No. The adapter direction is plausible, but the current branch is not the best mergeable form until it preserves native SSE timeout behavior, keeps startup checks retryable, uses accepted SDK/config seams, validates the new config key, removes raw payload logging, and has green checks plus live container proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Forward the SSE timeout through the adapter:** `extensions/signal/src/client-adapter.ts:118-143`
  - body: `runSignalSseLoop` still passes `timeoutMs: 0` so native Signal monitors can stay idle without reconnect churn, but the adapter does not accept or forward `timeoutMs` to `nativeStreamEvents`. That makes the current call from `sse-reconnect.ts` a type error and, if bypassed, native mode falls back to the 10s default.
  - confidence: 0.92
- **[P1] Keep Signal readiness probes retryable:** `extensions/signal/src/client-adapter.ts:107-111`
  - body: `signalCheck()` now resolves auto mode before returning. While an auto-started native daemon is still booting, both probes can fail and `detectSignalApiMode()` throws; `waitForTransportReady()` does not catch check exceptions, so startup can fail immediately instead of polling until the daemon is ready.
  - confidence: 0.9
- **[P1] Remove raw Signal payload logging:** `extensions/signal/src/client-container.ts:202`
  - body: `streamContainerEvents` logs the first 200 characters of every inbound WebSocket frame before parsing it. Those frames can include private Signal message text plus sender, group, and attachment metadata, so normal container receive would write sensitive content to gateway logs.
  - confidence: 0.94
- **[P2] Use the narrow config runtime seam:** `extensions/signal/src/client-adapter.ts:9`
  - body: This bundled plugin production file imports the broad `openclaw/plugin-sdk/config-runtime` barrel and calls ambient `loadConfig()`. Repo boundary guards require narrow config subpaths and passing runtime config through the call path for plugin runtime code, so this will keep boundary checks red.
  - confidence: 0.91
- **[P2] Use the narrow fetch runtime seam:** `extensions/signal/src/client-container.ts:11`
  - body: The new container client imports `resolveFetch` from the broad `openclaw/plugin-sdk/infra-runtime` barrel. Bundled plugin source is guarded against that broad subpath, and existing plugin code uses `openclaw/plugin-sdk/fetch-runtime` for this helper.
  - confidence: 0.88
- **[P2] Add apiMode to the strict config schema:** `src/config/types.signal.ts:68`
  - body: `apiMode` is added to the public Signal config type and docs, but the strict `SignalConfigSchema` still does not accept this key. A user setting `channels.signal.apiMode: "container"` can fail config validation before the adapter sees it.
  - confidence: 0.92
