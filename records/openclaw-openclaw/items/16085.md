## Summary

Keep open. Current main remains native JSON-RPC/SSE only, and this PR is still the active implementation candidate for the real container REST/WebSocket request, but the branch has blocking review findings around sensitive payload logging, config-boundary rules, and config validation.

## What This Changes

The PR adds a Signal native/container adapter, bbernhard REST/WebSocket client support, Signal caller rewiring, `channels.signal.apiMode` docs/types/tests, a changelog fragment, and a direct `ws` runtime dependency for the Signal plugin.

## Best Possible Solution

Ship the Signal plugin-local transport adapter after removing or redacting raw payload logs, passing mode selection through an accepted runtime config seam, adding `apiMode` to strict schema plus generated config/help metadata, preserving native JSON-RPC/SSE behavior, and providing targeted native/container tests plus live container proof or an explicit maintainer waiver.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Remove raw Signal payload logging:** `extensions/signal/src/client-container.ts:202`
  - body: `streamContainerEvents` logs the first 200 characters of every inbound WebSocket frame before parsing it. These frames can contain private Signal message text plus sender, group, and attachment metadata, so normal container receive would write sensitive content to gateway logs; remove the log or replace it with redacted metadata behind explicit debug output.
  - confidence: 0.93
- **[P2] Use the accepted runtime config seam:** `extensions/signal/src/client-adapter.ts:9`
  - body: This production extension file imports `loadConfig` from the broad `openclaw/plugin-sdk/config-runtime` barrel and calls it during mode resolution. The current config-boundary guard rejects both broad config-runtime imports and ambient `loadConfig()` in production extension code, so the changed gate will fail; pass the already loaded config/mode through the call path or use an allowed narrow runtime config seam.
  - confidence: 0.9
- **[P2] Validate the apiMode setting:** `src/config/types.signal.ts:68`
  - body: `apiMode` is added to the public Signal config type and docs, but the strict `SignalConfigSchema` still extends `SignalAccountSchemaBase` without accepting this key. A user setting `channels.signal.apiMode: "container"` can fail config validation before the adapter sees it, so the Zod schema and generated config/help surfaces need the new enum.
  - confidence: 0.91
