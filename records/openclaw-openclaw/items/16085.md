## Summary

Keep open. Current main still only supports the native Signal JSON-RPC/SSE transport, while this PR targets a real bbernhard REST/WebSocket compatibility gap; however, the PR branch is not merge-ready because it introduces blocking startup, config-boundary, schema-validation, and security logging issues.

## What This Changes

The PR adds a Signal native/container adapter, a bbernhard REST/WebSocket client, Signal caller rewiring, `channels.signal.apiMode` docs/types/tests, a changelog fragment, and a direct `ws` runtime dependency.

## Best Possible Solution

Ship a Signal plugin-local transport adapter after removing or redacting raw payload logs, keeping startup readiness retryable, passing mode/config through accepted runtime seams, adding strict schema and generated config/help metadata, preserving native JSON-RPC/SSE behavior, and providing targeted tests plus live container proof or an explicit maintainer waiver.

## Reproduction Assessment

Yes for the compatibility gap: current main is wired to `/api/v1/rpc`, `/api/v1/check`, and `/api/v1/events`, while upstream bbernhard docs use `/v2/send` and `/v1/receive/<number>`. A live container smoke was not run in this read-only review.

## Solution Assessment

No, not in the current branch form. The adapter approach is plausible, but the implementation needs security logging cleanup, retry-safe mode detection, accepted config plumbing, strict schema/generated metadata updates, and live proof before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Keep startup Signal checks retryable:** `extensions/signal/src/client-adapter.ts:107`
  - body: `signalCheck()` now resolves auto mode before returning. When a spawned native daemon is still starting, both probes can fail and `detectSignalApiMode()` throws; `waitForTransportReady()` does not catch exceptions from its check, so default native autostart can fail immediately instead of polling until ready. Return `{ ok: false }` on probe failure or pass the intended mode into the readiness path.
  - confidence: 0.9
- **[P1] Remove raw Signal payload logging:** `extensions/signal/src/client-container.ts:202`
  - body: `streamContainerEvents` logs the first 200 characters of every inbound WebSocket frame before parsing it. These frames can contain private Signal message text plus sender, group, and attachment metadata, so normal container receive would write sensitive content to gateway logs; remove the log or replace it with redacted metadata behind explicit debug output.
  - confidence: 0.94
- **[P2] Use the accepted config seam:** `extensions/signal/src/client-adapter.ts:9`
  - body: This production extension file imports the broad `openclaw/plugin-sdk/config-runtime` barrel and calls ambient `loadConfig()` during mode resolution. Current config-boundary guards reject both patterns in bundled plugin production code, so the changed gate should fail; pass the loaded config or mode through the call path, or use an accepted runtime snapshot seam at an explicit boundary.
  - confidence: 0.91
- **[P2] Import fetch helpers from the narrow SDK subpath:** `extensions/signal/src/client-container.ts:11`
  - body: The new container client imports `resolveFetch` from the broad `openclaw/plugin-sdk/infra-runtime` barrel. The bundled-plugin import guard rejects that broad subpath, and existing channel plugins use `openclaw/plugin-sdk/fetch-runtime` for this helper, so this branch should switch to the narrow fetch runtime entrypoint.
  - confidence: 0.88
- **[P2] Validate the apiMode setting:** `src/config/types.signal.ts:68`
  - body: `apiMode` is added to the public Signal config type and docs, but the strict `SignalConfigSchema` still does not accept this key. A user setting `channels.signal.apiMode: "container"` can fail config validation before the adapter sees it, so the Zod schema and generated config/help surfaces need the new enum.
  - confidence: 0.92
