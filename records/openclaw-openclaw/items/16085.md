## Summary

Keep open. Current main still has only the native Signal JSON-RPC/SSE transport, so the container REST/WebSocket compatibility gap is real; however, the PR branch still has concrete merge blockers around raw inbound payload logging, retry behavior, plugin SDK boundary imports, and config schema coverage.

## What This Changes

The PR adds a Signal native/container adapter, a bbernhard REST/WebSocket client, caller rewiring, `channels.signal.apiMode` docs/types/tests, a changelog fragment, and a direct `ws` runtime dependency.

## Best Possible Solution

Ship a Signal plugin-local transport adapter after removing or redacting raw payload logs, keeping startup readiness retryable, passing mode/config through accepted runtime seams, adding strict schema and generated config/help metadata, preserving native JSON-RPC/SSE behavior, and providing targeted tests plus live container proof or an explicit maintainer waiver.

## Reproduction Assessment

Yes. Current main is wired to `/api/v1/rpc`, `/api/v1/check`, and `/api/v1/events`, while the upstream bbernhard container examples use `/v2/send` and WebSocket `/v1/receive/<number>`; a live container smoke was not run in this read-only review.

## Solution Assessment

No. The adapter approach is plausible, but the current branch is not the best mergeable form until the logging, retry, SDK boundary, schema/generated metadata, and live-proof gaps are addressed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Keep startup Signal checks retryable:** `extensions/signal/src/client-adapter.ts:107`
  - body: `signalCheck()` now resolves auto mode before returning. When a spawned native daemon is still starting, both probes can fail and `detectSignalApiMode()` throws; `waitForTransportReady()` does not catch exceptions from its check, so default native autostart can fail immediately instead of polling until ready.
  - confidence: 0.9
- **[P1] Remove raw Signal payload logging:** `extensions/signal/src/client-container.ts:202`
  - body: `streamContainerEvents` logs the first 200 characters of every inbound WebSocket frame before parsing it. Those frames can contain private Signal message text plus sender, group, and attachment metadata, so normal container receive would write sensitive content to gateway logs.
  - confidence: 0.94
- **[P2] Use the accepted config seam:** `extensions/signal/src/client-adapter.ts:9`
  - body: This production extension file imports the broad `openclaw/plugin-sdk/config-runtime` barrel and calls ambient `loadConfig()` during mode resolution. Current config-boundary guards reject that broad subpath in bundled plugin production code; pass the loaded config or mode through the call path instead.
  - confidence: 0.91
- **[P2] Import fetch helpers from the narrow SDK subpath:** `extensions/signal/src/client-container.ts:11`
  - body: The new container client imports `resolveFetch` from the broad `openclaw/plugin-sdk/infra-runtime` barrel. The bundled-plugin import guard rejects that broad subpath, and existing channel plugins use `openclaw/plugin-sdk/fetch-runtime` for this helper.
  - confidence: 0.88
- **[P2] Validate the apiMode setting:** `src/config/types.signal.ts:68`
  - body: `apiMode` is added to the public Signal config type and docs, but the strict `SignalConfigSchema` still does not accept this key. A user setting `channels.signal.apiMode: "container"` can fail config validation before the adapter sees it.
  - confidence: 0.92
