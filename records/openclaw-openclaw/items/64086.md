## Summary

Keep this issue open. Current `main` still lacks the requested plugin footer status contribution API and footer consumption path, and open PR #72557 explicitly resolves this issue, so the maintainer path is to review, land, or close that PR rather than close the issue early.

## What This Changes

Request a plugin API such as `registerStatusProvider` so context engine and other plugins can contribute small status indicators to the TUI footer without monkeypatching bundled TUI code.

## Best Possible Solution

Keep this issue open until a reviewed implementation lands that exposes a stable plugin status-provider seam and either wires it into the TUI footer or clearly tracks the render-path follow-up with the broader footer customization work.

## Reproduction Assessment

Yes. A read-only current-main check reproduces the gap: there is no `registerStatusProvider` or `PluginStatusProvider` in the plugin API/registry/SDK docs, and the TUI footer render path only uses core session/model/mode/token data.

## Solution Assessment

Unclear. A narrow plugin API is a plausible maintainable seam, but the best final shape depends on maintainer decisions about return structure, footer UX, and whether API storage and TUI consumption should ship together.
