## Summary

Keep open. Current main still lets the reported inline Control UI/WebChat HEIC/HEIF attachment path preserve the original image MIME into the current-run prompt image handoff, and #50320 is an open, unmerged implementation candidate that uses closing syntax for this issue.

## What This Changes

The issue requests converting HEIC/HEIF images uploaded through Control UI webchat to JPEG before the current-run provider prompt image handoff.

## Best Possible Solution

Land a narrow fix, from #50320 or a focused replacement, that converts HEIC/HEIF inline chat attachments to JPEG before `parsed.images` reaches `activeSession.prompt`, rechecks the converted output against image limits, preserves non-HEIC and offloaded-media behavior, and adds focused gateway attachment and prompt-dispatch regression coverage.

## Reproduction Assessment

Yes. The reporter provided a concrete Control UI HEIC upload path and hot-patch confirmation, and current main statically preserves `image/heic` from chat attachment parsing through `replyOptions.images` into `activeSession.prompt` when the prompt has no file refs.

## Solution Assessment

Yes, normalizing before the current-run provider call is the right fix boundary because persisted/history normalization happens too late for the in-flight model request. The only uncertainty is vehicle selection: #50320 may need maintainer review or splitting because it mixes unrelated fixes.
