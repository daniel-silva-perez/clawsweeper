## Summary

Keep this PR open. The incremental session-sync direction addresses the real performance problem tracked by #40919 and supersedes the stale-path approach in #59577, but the proposed patch is not merge-ready because hash-reused chunks can stay missing from vector recall and the new persisted offsets are inaccurate for chunk boundaries.

## What This Changes

The PR adds hash-based incremental indexing for session memory files, start/end offset schema and SDK exports, and memory-core tests/benchmarks for preserving unchanged session chunks.

## Best Possible Solution

Land a corrected incremental session-sync implementation that keeps the append/growth savings, rehydrates or repairs vector rows for reused chunks when vector state requires it, fixes offset accounting, and adds focused regression coverage before exposing the new metadata.

## Reproduction Assessment

Yes. A high-confidence reproduction for the main blocker is a session index whose existing chunks have embeddings in chunks but missing chunks_vec rows, followed by a targeted session growth sync; the PR reuses hash-matched chunks without calling replaceMemoryVectorRow for them. The offset issue is reproducible by chunking multi-line content at a small chunk size and checking that the next chunk startOffset points at the newline.

## Solution Assessment

No. Incremental sync is the right direction for ordinary session growth, but this implementation needs to preserve current selected-file repair behavior for vector rows and correct the offset helper before it is the best maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Rebuild vector rows for reused chunks:** `extensions/memory-core/src/memory/manager-embedding-ops.ts:864`
  - body: Hash-matched chunks only go through the metadata refresh path, so their stored embeddings are never written back to chunks_vec. If an earlier sync ran while sqlite-vec was unavailable, or the vector row is otherwise missing, a later targeted session sync leaves those reused chunks absent from vector recall because vector search joins through chunks_vec. Reinsert the existing embedding when vectorReady is true or fall back to a full file rewrite when vector rows are missing.
  - confidence: 0.86
- **[P3] Advance offsets past original separators:** `packages/memory-host-sdk/src/host/internal.ts:515`
  - body: After flushing a chunk, charOffset advances by currentRawChars - 1. For content like `abc\ndef`, the first chunk ends at offset 2 but the next chunk starts at offset 3, the newline, instead of offset 4. Since these offsets are newly persisted and exported, multi-line and split-line chunks will record incorrect ranges.
  - confidence: 0.84
