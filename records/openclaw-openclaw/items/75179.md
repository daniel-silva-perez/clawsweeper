## Summary

Keep this PR open. The underlying session-memory performance problem remains valid and the branch now targets the active memory-core paths, but the proposed patch is not ready because it breaks the documented force-reindex repair path for session files.

## What This Changes

The PR adds hash-based incremental indexing for session memory files, exports offset-aware chunking helpers through the memory host SDK, adds start/end offset columns to the memory index schema, and adds incremental/benchmark test coverage.

## Best Possible Solution

Keep the incremental session-sync direction, but preserve full-reindex semantics when `needsFullReindex` is true and lock both incremental growth and forced-repair behavior with focused memory-core tests.

## Reproduction Assessment

Yes. A high-confidence reproduction path is `manager.sync({ force: true })` or `openclaw memory index --force` on an unchanged session after deleting/corrupting vector rows or expecting same-model re-embedding; the proposed call site sends that path through hash-preserving incremental sync.

## Solution Assessment

No. Incremental sync is a reasonable non-force optimization, but the safest maintainable fix is to use it only for non-force session deltas or pass an explicit force mode that clears and re-embeds all session chunks.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve force reindex semantics for sessions:** `extensions/memory-core/src/memory/manager-sync-ops.ts:886`
  - body: This replacement also runs when `params.needsFullReindex` is true. The new incremental path preserves hash-matched chunks and embeddings, so `openclaw memory index --force` no longer performs the documented full session rebuild needed to repair stale or corrupted rows. Keep the existing `indexFile` path for force reindexes or make the incremental helper clear/re-embed when forced.
  - confidence: 0.93
- **[P3] Fix stored offsets for multi-line chunks:** `packages/memory-host-sdk/src/host/internal.ts:511`
  - body: `currentRawChars` already includes one synthetic newline per segment, but advancing by `currentRawChars - 1` after a multi-line flush points the next chunk at the separator newline instead of the next content character. Since these offsets are exported and written to the new schema columns, multi-line session chunks can get inaccurate `start_offset` values.
  - confidence: 0.82
