## Summary

Keep open. Current main still uses the full clear-and-reinsert session indexing path, so the incremental-session-sync work is not implemented yet; this PR is useful source work but not merge-ready because it regresses documented force-reindex semantics and stores incorrect offsets for some multi-line chunks.

## What This Changes

The PR adds hash-based incremental indexing for session memory files, start/end offset storage and SDK exports, schema updates, and memory-core tests/benchmarks for preserving unchanged session chunks.

## Best Possible Solution

Keep the incremental session-sync direction, but use it only for non-force session deltas or add an explicit forced mode that clears and rebuilds session chunks. Lock both append-growth efficiency and forced-repair behavior with focused memory-core tests, and correct the offset helper before exposing the new stored metadata.

## Reproduction Assessment

Yes. A high-confidence reproduction path is `manager.sync({ force: true })` or `openclaw memory index --force` on an unchanged session after deleting or corrupting indexed rows; the PR sends that path through hash-preserving incremental sync instead of the current clear-and-rebuild path.

## Solution Assessment

No. Incremental sync is a good fit for ordinary session growth, but this implementation should preserve the documented force-reindex repair behavior and fix offset accounting before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve force reindex semantics for sessions:** `extensions/memory-core/src/memory/manager-sync-ops.ts:886`
  - body: This replacement also runs when `params.needsFullReindex` is true. The new incremental path preserves hash-matched chunks and embeddings, so `openclaw memory index --force` no longer performs the documented full session rebuild needed to repair stale or corrupted rows. Keep the existing `indexFile` path for force reindexes or make the incremental helper clear/re-embed when forced.
  - confidence: 0.93
- **[P3] Advance offsets past separators after multi-line chunks:** `packages/memory-host-sdk/src/host/internal.ts:511`
  - body: After flushing a chunk with multiple lines, `currentRawChars - 1` advances only to the separator newline after the last emitted line. The next chunk then records a `startOffset` that points at that newline instead of the next content character, so the new persisted offsets are inaccurate for multi-line chunks.
  - confidence: 0.82
