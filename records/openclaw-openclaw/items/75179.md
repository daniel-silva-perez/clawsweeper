## Summary

Keep open. The PR targets a valid open session-memory performance problem, but current main does not already implement the change and the proposed diff still has concrete correctness issues around preserved chunks and stored offsets, so this needs normal maintainer review/requested changes rather than cleanup closure.

## What This Changes

The PR adds hash-based incremental indexing for session memory files, offset-aware chunking/schema fields, SDK exports, and incremental plus benchmark test coverage.

## Best Possible Solution

Merge an incremental session-sync implementation that preserves current repair semantics: reuse embeddings for unchanged chunks only when required vector/FTS rows are present or backfilled, store correct source offsets, and cover session growth plus degraded-vector recovery with focused memory-core tests.

## Reproduction Assessment

Yes. For the blocking finding, create or simulate session chunks with embeddings but missing vector rows, then run targeted `sync({ sessionFiles: [...] })` after the session grows; the PR reuses old hash-matched chunks without calling the vector write path. The offset issue can be checked by chunking multi-line content across a flush and comparing `startOffset` to the original content index.

## Solution Assessment

No. Incremental session sync is a reasonable direction for #40919, but this implementation should first verify/backfill preserved vector rows and fix offset tracking so the optimization does not regress recall repair or citation metadata.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Backfill vectors for reused chunks:** `extensions/memory-core/src/memory/manager-embedding-ops.ts:820-823`
  - body: Hash-matched chunks are moved into the metadata-refresh path and never call `replaceMemoryVectorRow`. If an earlier sync stored chunk embeddings while vector writes were degraded, or if vector rows were pruned, a later targeted session sync will keep those unchanged chunks out of semantic search even after vectors are available; check/backfill the vector rows for reused chunks or fall back to a full rewrite when they are missing.
  - confidence: 0.87
- **[P3] Advance offsets past line separators:** `packages/memory-host-sdk/src/host/internal.ts:515`
  - body: `currentRawChars` includes separator characters that are not all part of the emitted chunk text, but the flush path advances `charOffset` by `currentRawChars - 1`. After a line-boundary flush, the next chunk can start on the separator newline instead of the next content character, so stored `start_offset` values drift.
  - confidence: 0.82
