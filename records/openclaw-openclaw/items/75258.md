## Summary

Keep this PR open for normal maintainer review. Current main still lacks the early final-assembly and streaming-normalizer reasoning filters this PR adds, and the updated head addresses the earlier review-cycle cleanup suggestions without introducing a concrete blocking regression.

## What This Changes

The PR filters `ReplyPayload.isReasoning` out of final reply payload assembly and shared streaming normalization, then adds regression coverage for mixed reasoning plus visible replies.

## Best Possible Solution

Land the narrow delivery-boundary hardening after maintainer review and CI, while keeping the broader WhatsApp session instability tracked separately in #74886.

## Reproduction Assessment

Yes for the narrow boundary this PR targets: a mixed `ReplyPayload[]` containing `{ isReasoning: true }` plus a visible answer reproduces the final assembly gap on current main, and reasoning-marked partial/tool/block payloads exercise the streaming normalizer gap. The full WhatsApp bad-session state from #74886 is not reproduced by this PR alone.

## Solution Assessment

Yes for this PR’s scope: filtering marked internal reasoning payloads before final assembly and streaming callbacks is a narrow, maintainable defense-in-depth layer that matches the existing downstream dispatch policy. It is not the complete solution for the unrelated slowdown/fallback symptoms in the linked regression report.
