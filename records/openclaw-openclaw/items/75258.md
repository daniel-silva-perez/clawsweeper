## Summary

Keep this PR open for revision. The privacy hardening direction is valid for generic channel delivery, but the current diff suppresses all marked reasoning too early and would break opted-in reasoning lanes such as Telegram `/reasoning on`.

## What This Changes

The PR filters `isReasoning` payloads from final reply assembly and streaming normalization, then adds regression tests for mixed reasoning and visible reply payloads.

## Best Possible Solution

Revise the hardening so marked reasoning is suppressed at generic or capability-aware delivery boundaries while preserving explicit reasoning lanes such as Telegram `/reasoning on` and `/reasoning stream`; then add regression coverage for both generic WhatsApp suppression and Telegram opted-in delivery.

## Reproduction Assessment

Yes. The narrow privacy boundary can be reproduced with mixed `ReplyPayload[]` and reasoning-marked streamed callbacks, and the PR regression can be reproduced by a Telegram `reasoningLevel: "on"` path that emits `Reasoning:` as a separate block/final lane.

## Solution Assessment

No, not as-is. Filtering marked reasoning globally inside final assembly and the shared streaming normalizer is earlier than the channel capability boundary, so the safer fix is generic-dispatch suppression or a capability-aware seam that preserves dedicated reasoning lanes.

The Telegram path has explicit `Reasoning:` lane handling and tests for `reasoningLevel: "on"`, so mixed reasoning plus answer payloads would lose the separate reasoning message even when the user enabled it.
  - confidence: 0.88
