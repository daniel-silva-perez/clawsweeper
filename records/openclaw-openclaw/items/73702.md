## Summary

Keep this issue open. Current main has an older inter-session provenance mitigation, but the A2A reply path still forwards the other session's reply as the active prompt text for the caller/peer run, with provenance carried as metadata and system context rather than as a model-visible tool result or per-message untrusted envelope. That is close to the reported security boundary, and the existing tests prove `inputProvenance` is passed, not that the same-turn model input cannot be treated as a bare user instruction.

## Best Possible Solution

Keep this issue open and harden the A2A reply path so foreign-session replies are model-visible as untrusted data before the receiving model call. The most direct fix is in `runAgentStep`/`sessions_send`: deliver replies as a tool-result-like payload or wrap the active prompt with a synthetic inter-session envelope that includes source session/channel/tool and an explicit non-user instruction flag. Add a regression test that captures the model input for the requester-side reply step and proves the target reply cannot appear as an unmarked bare user instruction.

## Risks / Open Questions

- Closing now could treat persisted provenance as equivalent to same-turn prompt isolation, but the reported exploit is about how the forwarded reply is interpreted during the receiving run.
- The affected surface is security-sensitive cross-session agent integrity; a conservative maintainer cleanup should require a regression test that captures the actual model-bound input for the A2A reply step.
