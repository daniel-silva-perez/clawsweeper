## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks the PR's generic runtime-level 180s fallback, and the Greptile comment correctly identified the intended small surface, but the patch has a concrete regression: converting an omitted timeout into `180_000` before provider dispatch bypasses OpenAI's provider-owned Azure default of `600_000`.

## What This Changes

The PR adds a `DEFAULT_IMAGE_GENERATION_TIMEOUT_MS` runtime fallback of 180 seconds and tests default/config/per-request timeout precedence in image generation.

## Best Possible Solution

Preserve provider-owned timeout defaults while adding a generic fallback only where it cannot override a more specific provider policy. A safe revision would keep OpenAI/Azure defaults intact, add coverage for Azure plus the intended generic fallback, and avoid treating a built-in runtime default as a user/config override.

## Reproduction Assessment

Yes. The regression can be reproduced by comparing the PR diff with the existing Azure OpenAI provider test path: current main sends `600_000` when no timeout is provided, while the PR would pass `180_000` into the provider first.

## Solution Assessment

No. A runtime-wide unconditional default is not the best fix because providers already have capability- or endpoint-specific defaults; the safer approach is to apply the 180s fallback at provider/helper boundaries that lack their own default, or otherwise preserve an unset-caller signal.
