## Summary

Keep open. The PR addresses a plausible timeout gap, but the current diff is not landable because it converts an omitted timeout into an explicit 180s timeout before provider-owned defaults can run, which regresses Azure OpenAI image generation from its existing 600s default. It also needs the required changelog entry for a user-facing fix.

## What This Changes

The PR adds a 180000 ms image-generation runtime fallback and tests default/config/per-request timeout precedence in `src/image-generation/runtime.ts` and `src/image-generation/runtime.test.ts`.

## Best Possible Solution

Revise the timeout change so an unset caller/config timeout remains distinguishable from an explicit override, preserve OpenAI and Azure provider defaults, add coverage for both the intended generic fallback and Azure no-timeout behavior, and include the required Unreleased changelog entry.

## Reproduction Assessment

Yes. The regression is reproducible by following the existing Azure OpenAI provider test path: current main sends `600_000` when no timeout is supplied, while the PR diff would pass `180_000` into the provider before that default can be selected.

## Solution Assessment

No. A runtime-wide unconditional fallback is not the best fix because image providers already own endpoint-specific timeout policy; the safer direction is to apply a generic 180s fallback only at provider/helper boundaries that lack their own default.
