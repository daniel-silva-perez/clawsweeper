## Summary

Keep open. The PR addresses a real timeout pain point, but the patch is not merge-ready because its runtime-wide fallback turns an omitted timeout into an explicit 180s timeout before provider-owned defaults can apply, shortening Azure OpenAI image generation from the current 600s default.

## What This Changes

The PR adds a 180-second image-generation runtime timeout fallback and tests default, config, and per-request timeout precedence.

## Best Possible Solution

Preserve the unset-timeout signal for providers that already own longer defaults, and add a 180s fallback only at provider/helper boundaries that otherwise fall through to the generic 60s guarded HTTP default, with Azure regression coverage.

## Reproduction Assessment

Yes. The regression can be reproduced by comparing the PR diff to the current OpenAI provider path: main omits timeoutMs so Azure resolves to 600_000, while the PR would pass 180_000 before the provider can apply its Azure default.

## Solution Assessment

No. A runtime-wide unconditional default is too broad because provider implementations already encode endpoint-specific timeout policy; the narrower maintainable fix is provider/helper-level fallback behavior that does not override a more specific provider default.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve Azure OpenAI's longer default timeout:** `src/image-generation/runtime.ts:105`
  - body: Passing `timeoutMs: effectiveTimeoutMs` unconditionally converts an omitted timeout into `180_000` before the OpenAI provider sees the request. The provider currently uses `undefined` to select its own defaults, including `600_000` for Azure deployments, and existing tests assert that behavior. With this patch, Azure image generation without a user/config timeout would be shortened from 10 minutes to 3 minutes.
  - confidence: 0.9
