## Summary

Keep open. Current main still does not implement the requested exec-side repeated file-reread detector: loop detection remains based on generic exact tool+params hashing, known polling/no-progress detection, ping-pong detection, unknown-tool repeats, and the global circuit breaker. The related #64426 is a companion first-class read-tool proposal, not a replacement for this exec-runtime request.

## Best Possible Solution

Keep this issue open. The best path is a narrow core exec/tool-loop enhancement that classifies obvious read-only shell readbacks, normalizes cwd plus target file paths while collapsing line-window variants, tracks same-file and same-file-pair counts per run/session, emits a warning-only synthetic result before executing the next redundant read, logs structured trigger telemetry, and adds focused tests for sed -n, nl -ba | sed -n, cat, bounded rg, alternating file pairs, thresholds, progress resets, and false-positive cases. Coordinate with #64426 without treating it as a replacement for this exec-runtime work.

## Risks / Open Questions

- A low-threshold exec reread detector could false-positive in legitimate debugging unless it starts narrow, warning-only, and focused on obvious read-only command families.
- The implementation should preserve exec approvals, sandbox/gateway/node routing, script preflight behavior, and structured diagnostic telemetry while avoiding execution of the next redundant readback command.
- The exec-side semantics should coordinate with #64426 so first-class read-tool and shell readback guards share path normalization and progress-reset behavior where practical.
