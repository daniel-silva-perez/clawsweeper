## Summary

Keep this draft PR open for maintainer follow-up. The requested Google Meet human barge-in behavior is still missing from current main, but the proposed patch does not reliably stop OpenAI provider output because the new side-channel calls the barge-in hook after Google Meet has already acknowledged playback marks immediately.

## What This Changes

The PR adds an optional realtime voice `handleBargeIn()` bridge hook, Google Meet Chrome `bargeInInputCommand` config/metadata/docs, loopback input suppression during playback, local playback clearing on detected speech, and focused tests/generated config baseline updates.

## Best Possible Solution

Keep the draft available, but require a patch update that makes barge-in stop provider output reliably for the Google Meet OpenAI path, preserves the optional Google Meet-owned side-channel command model, keeps external bridge behavior unchanged, and lands with focused Google Meet/OpenAI realtime tests plus plugin SDK/config/docs checks.

## Reproduction Assessment

Yes for the source-level gap and PR blocker. Current main lacks the side-channel and hook, and the PR branch has a deterministic path where Google Meet uses immediately acknowledged marks while OpenAI only truncates when its mark queue is still populated.

## Solution Assessment

No, not as written. The direction is plausible, but provider cancellation/truncation should be independent of transport mark acknowledgements or should preserve the OpenAI response state until external barge-in is handled.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Keep provider truncation effective for barge-in:** `extensions/google-meet/src/realtime.ts:274`
  - body: The Google Meet command-pair bridge uses `markStrategy: "ack-immediately"`, so `session-runtime` immediately calls `acknowledgeMark()` and OpenAI clears `markQueue`, `responseStartTimestamp`, and `lastAssistantItemId`. When this new side-channel later calls `bridge?.handleBargeIn()`, OpenAI usually falls through to `onClearAudio()` without sending `conversation.item.truncate`, so the active response can keep streaming new audio after the local buffer clear.
