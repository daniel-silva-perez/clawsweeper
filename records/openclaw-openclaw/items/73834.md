## Summary

Keep this PR open for maintainer follow-up. Current main still lacks the requested Google Meet human barge-in behavior and the linked #73850 issue remains open, but the draft patch has a correctness issue: the new Google Meet side-channel calls the OpenAI barge-in hook in a path where marks are acknowledged immediately, so provider-side truncation will usually not happen.

## What This Changes

The PR adds an optional realtime `handleBargeIn()` bridge hook, Google Meet Chrome `bargeInInputCommand` config/metadata/docs, loopback input suppression during playback, local playback clearing on detected speech, and focused tests/generated config baseline updates.

## Best Possible Solution

Keep the draft available, but require a patch update that makes barge-in stop provider output reliably for the Google Meet OpenAI path, preserves the optional Google Meet-owned side-channel command model, keeps external bridge behavior unchanged, and lands with focused Google Meet/OpenAI realtime tests plus plugin SDK/config/docs checks.

## Reproduction Assessment

Yes for the source-level gap and the PR blocker. Current main lacks the side-channel and hook entirely, and the PR branch shows a deterministic path where Google Meet uses `ack-immediately` marks while OpenAI only truncates when its mark queue is still populated; no live Meet run was performed.

## Solution Assessment

No, not as written. The overall direction is plausible, but the implementation should make provider cancellation/truncation independent of transport mark acknowledgements or otherwise avoid clearing the OpenAI response state before external barge-in can truncate it.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Keep provider truncation effective for barge-in:** `extensions/google-meet/src/realtime.ts:274`
  - body: The Google Meet command-pair bridge uses `markStrategy: "ack-immediately"`, so `session-runtime` immediately calls `acknowledgeMark()` and the OpenAI bridge clears `markQueue`, `responseStartTimestamp`, and `lastAssistantItemId`. When this new side-channel later calls `bridge?.handleBargeIn()`, OpenAI usually falls through to `onClearAudio()` without sending `conversation.item.truncate`, so the active response can keep streaming new audio after the local buffer clear. Make the provider barge-in hook cancel/truncate independently of transport marks, or keep the needed response state until external barge-in is handled.
  - confidence: 0.87
