## Summary

Keep open. Current main still treats Twitter/X URLs as ordinary `web_fetch` inputs, has no `domainBlocklist`, `domainRoutes`, or tool-description override surface, and the existing `x_search` alternative is a separate tool that is not consulted by `web_fetch`; the related FxTwitter fallback PR was closed unmerged.

## What This Changes

The issue requests tool-level routing, blocking, or description overrides for Twitter/X URLs so agents stop choosing generic `web_fetch` when X-specific alternatives are available.

## Best Possible Solution

Add a maintainer-approved, generic `web_fetch` routing or blocking seam that can steer known bad domains before the normal fetch path, with config/schema/docs/tests, while avoiding hard-coded service-specific behavior in core unless explicitly accepted.

## Reproduction Assessment

Yes. A static reproduction path is clear: call `web_fetch` with an `x.com` or `twitter.com` status URL and current main will send that URL through the ordinary fetch path with no route rewrite or blocklist check.

## Solution Assessment

Unclear. The closed FxTwitter PR would address the visible Twitter/X symptom, but the safer maintainable direction is a generic tool-level routing or blocklist API plus tests, with any default X/Twitter behavior approved as a product/privacy decision.
