## Summary

Keep open. Current main still exposes `tools.sessions.visibility` and `tools.agentToAgent` as global runtime controls, while `agents.list[].tools` does not accept those per-agent keys; an open implementation PR also declares it resolves this issue, so the issue should remain open until that PR or a replacement lands.

## What This Changes

Add per-agent session visibility and agent-to-agent invocation scoping so administrators can enforce least-privilege access between agents.

## Best Possible Solution

Keep the issue open while maintainers review PR #59170 or a replacement. The end state should be one coherent per-agent access-control contract covering schema/types, docs, `sessions_list`, `sessions_history`, `sessions_send`, `session_status`, outbound A2A allowlists, and any chosen inbound invocation policy, with focused tests for requester identity and fail-closed behavior.

## Reproduction Assessment

Yes. The limitation is reproducible by inspecting current main: `agents.list[].tools` is strict and lacks `sessions`/`agentToAgent`, while the runtime session/A2A guards read only `cfg.tools` global settings.

Unclear. The requested direction is valid, but the open implementation PR needs maintainer review for access-control semantics, overlap with per-agent sandbox visibility work, and whether inbound `receivesFrom` belongs in the initial scope.
