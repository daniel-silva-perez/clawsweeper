## Summary

Keep open. Current main still resolves session visibility and agent-to-agent policy from global config only, while the linked implementation PR remains open and explicitly claims to resolve this issue. Because this is paired with an unmerged PR and the requested behavior is access-control-sensitive, it should stay open for maintainer review rather than be swept closed.

## What This Changes

Request per-agent scoping for session visibility and agent-to-agent invocation so orchestrator agents can access selected agents while specialists remain restricted.

## Best Possible Solution

Review or replace the linked implementation with a rebased, narrow core change that adds documented per-agent config, resolves policy from trusted requester agent context, applies it consistently to `sessions_list`, `sessions_history`, `sessions_send`, and `session_status`, and preserves backwards-compatible global defaults.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. Source inspection on current main shows the schema lacks `agents.list[].tools.sessions` and `agents.list[].tools.agentToAgent`, and the runtime session access helpers/tool callers read only global `tools.sessions` and `tools.agentToAgent`.

## Solution Assessment

Is this the best way to solve the issue? Unclear for the exact proposed shape. Per-agent policy is the right direction, but the safest solution needs maintainer review of inheritance, wildcard, and target-side receiving semantics before implementation is treated as the final access-control contract.
