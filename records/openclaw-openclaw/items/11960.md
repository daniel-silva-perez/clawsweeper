## Summary

Keep open. Current main still does not route automatic daily or idle stale rollovers through the reset persistence hook surface used by manual resets, so the primary memory-persistence request remains unresolved. The secondary hook-specific config schema mismatch is fixed, and open PR #71792 explicitly closes this issue while #61675 overlaps, so maintainers should review the existing implementation candidates rather than close this issue or queue a duplicate repair.

## What This Changes

Request a `session:reset` or `session:before-reset` hook, or equivalent reset persistence event, so automatic daily and idle session resets can save previous session memory before old context is cleared, plus allow hook-specific config such as `messages`.

## Best Possible Solution

Land #71792, #61675, or a maintainer-approved replacement that wires automatic daily and idle rollovers into the same effective pre-reset persistence surface as manual resets, supplies previous transcript data to session-memory and custom hooks, preserves typed lifecycle hooks, and adds daily plus idle regression coverage.

## Reproduction Assessment

Yes. Configure a daily or idle stale session and send the next user message: current main captures and archives the previous session, then emits only `session_end`/`session_start`, while session-memory still filters to `command:new` and `command:reset`.

## Solution Assessment

Unclear. The requested direction is valid, but maintainers still need to choose the maintainable event semantics from the open implementation candidates rather than letting another autonomous fix define a competing reset surface.
