## Summary

Keep open. Current main still does not route automatic daily or idle stale rollovers through the reset persistence surface used by session-memory; the secondary per-hook config schema mismatch is already fixed, but the primary auto-reset persistence request remains unresolved and is paired with open implementation PRs.

## What This Changes

The issue requests a hook/event surface that lets session-memory and custom hooks persist context when sessions reset automatically via daily or idle rollover.

## Best Possible Solution

Land #71792, #61675, or a maintainer-approved replacement that wires automatic daily and idle rollovers into the same effective pre-reset persistence surface as manual resets, supplies previous transcript data to session-memory and custom hooks, preserves typed lifecycle hooks, and adds daily plus idle regression coverage.

## Reproduction Assessment

Yes. Configure a daily or idle stale session and send the next user message: current main captures and archives the previous session, then emits only lifecycle hooks while session-memory remains gated to `command:new` and `command:reset`.

## Solution Assessment

Unclear. The requested direction is valid, but maintainers should select the durable event semantics from the existing implementation candidates rather than letting another autonomous repair introduce a competing reset surface.
