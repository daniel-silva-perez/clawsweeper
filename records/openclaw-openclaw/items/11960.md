## Summary

Keep open. Current main still does not route automatic daily or idle stale rollovers through the reset persistence surface that session-memory uses, so the primary memory-persistence request remains unresolved; only the secondary hook-config schema mismatch is fixed.

## What This Changes

The issue requests a `session:reset`/`session:before-reset` hook or equivalent so automatic daily and idle session resets can persist prior context, plus hook-specific config keys such as `messages`.

## Best Possible Solution

Land #71792, #61675, or a maintainer-approved replacement that wires automatic daily and idle rollovers into the same effective pre-reset persistence surface as manual resets, supplies previous transcript data to session-memory and custom hooks, preserves typed lifecycle hooks, and adds daily plus idle regression coverage.

## Reproduction Assessment

Yes. Configure a daily or idle stale session and send the next user message: current main captures and archives the previous session, then emits only `session_end`/`session_start`, while session-memory still filters to `command:new` and `command:reset`.

## Solution Assessment

Unclear. The requested direction is valid, but maintainers should select the durable event semantics from the open implementation candidates rather than letting another autonomous repair define a competing reset surface.
