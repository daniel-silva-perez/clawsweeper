## Summary

Keep open. Current main still supports loading shared skills from `~/.openclaw/skills` and `~/.agents/skills`, but native `openclaw skills install` remains workspace-targeted only. The earlier ClawSweeper comment is still accurate on current main: this is distinct from #56161 and #43735 because it asks for an intentional global/shared install target.

## Best Possible Solution

Implement an explicit opt-in global install target for native ClawHub skill installs, most likely `openclaw skills install <slug> --global`, targeting the managed shared root `~/.openclaw/skills`. Preserve existing workspace behavior by default, reject ambiguous `--global` plus `--agent`, print the resolved target path, keep existing safe slug/path validation and ClawHub origin metadata, and update focused CLI/installer tests plus skills/ClawHub docs. Treat a config default such as `skills.install.defaultTarget` as a follow-up unless maintainers explicitly want that policy now.

## Risks / Open Questions

- A global target broadens visibility of third-party skill content across agents, so the fix should be explicit opt-in and preserve the current workspace default.
- Lock/origin tracking should stay coherent per target; global installs should not accidentally write workspace lock state or make workspace updates manage global skills unexpectedly.
- `--global` combined with `--agent` would be ambiguous and should be rejected or clearly defined before implementation.
