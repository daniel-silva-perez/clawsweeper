## Summary

Keep open. Current main still has the reported lifecycle ambiguity: embedded runs emit only `start`, `end`, and `error`; sub-agent tracking ignores other lifecycle phases; and `end` events with `aborted:true` still collapse to timeout after a retry grace window. The requested `first-progress` and typed `startup-failed` distinction is not implemented and remains a concrete, narrow runtime fix.

## What This Changes

The issue requests additive sub-agent lifecycle phases for first progress and startup failures so parent sessions can distinguish silent startup aborts from slow or hung child runs.

## Best Possible Solution

Add the lifecycle signal as a backwards-compatible core runtime seam: emit first progress once from the first qualifying assistant text or tool call, emit a typed startup failure before silent aborts can look like normal timeouts, and keep existing `start`/`end`/`error` behavior intact for current consumers.

## Reproduction Assessment

Yes at the code-path level: emitting lifecycle `end` with `aborted:true` and no stop reason reproduces the timeout mapping in the current gateway wait/session/sub-agent paths, and existing tests encode that behavior. The full live startup-abort transcript fixture is not available in the checkout.

## Solution Assessment

Yes, the requested additive lifecycle events are the right shape if implemented with compatibility tests. A safer implementation should handle new phases explicitly in sub-agent registry/gateway wait consumers while leaving unknown-phase tolerance and existing terminal events unchanged.
