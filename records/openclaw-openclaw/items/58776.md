## Summary

Keep open. Current main has useful restart-aborted subagent recovery and timeout-grace handling, but it still lacks the requested typed startup/progress lifecycle surface: lifecycle emitters and consumers remain centered on `start`, `end`, and `error`, and plain aborted terminal events still resolve as timeout in the public event path.

## What This Changes

Add additive sub-agent lifecycle signals for first progress and typed startup failures so parent runs can distinguish silent startup aborts from slow starts or real timeouts.

## Best Possible Solution

Add a generic lifecycle-progress seam for embedded/native subagent runs: emit one `first-progress` event after the first qualifying visible assistant text or tool start, emit a typed `startup-failed` event when a run terminates before that point, preserve existing `start`/`end`/`error` events, and update SDK/docs/tests so existing consumers remain compatible.

## Reproduction Assessment

Yes. A focused reproduction can register a subagent run, emit lifecycle `{ phase: "end", aborted: true }` before any assistant/tool progress, and observe the registry/SDK timeout path; the live transcript corpus itself is not available in this checkout.

## Solution Assessment

Yes, the proposed additive lifecycle events are a maintainable direction if implemented generically from the existing assistant/tool/lifecycle streams. The safer implementation boundary is the embedded runner, subagent registry, Gateway wait/event normalization, SDK types, and docs, without plugin-specific IDs or startup policy in core.
