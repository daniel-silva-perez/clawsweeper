## Summary

Keep this PR open. Current main still resolves channel config forms from the channel root only, so the intended account-scoped fix is not implemented. The PR is still relevant for #51335, but the proposed diff has functional blockers around schema fallback and patch-path alignment before it is safe to merge.

## Best Possible Solution

Keep this PR open as a focused fix candidate, but require a revised implementation: pass the preferred/default account id into channel config rendering, resolve account-scoped values when present, fall back both schema and value handling to the root channel config for root-backed channels, align the patch path with the object actually displayed, trim account ids consistently, update extras/status consumers if they should reflect account-scoped fields, and add regression tests for account-scoped plus root-fallback cases.

## Risks / Open Questions

- Merging the current PR as written can make root-schema single-account channels, such as Microsoft Teams, render "Channel config schema unavailable" when a default account id is present.
- The proposed fallback can display root channel config while edits patch channels.<id>.accounts.<accountId>, so users may write to a different object than the one shown.
- resolveChannelExtras remains root-only on current main and in the PR, so account-scoped fields may still be omitted from status metadata.
- The PR body reports the targeted Vitest run was still hanging in the contributor checkout, so regression-test proof is incomplete.
