## Summary

Keep this PR open. Current main still resolves the Control UI channel config form only at the channel root, while account-scoped channel config is a current product surface. The PR is a relevant fix candidate for the linked account-path bug, but the provided review discussion identifies correctness blockers that need revision before merge.

## Best Possible Solution

Keep this PR open as a focused fix candidate, but require a corrected implementation before merge: pass the preferred/default account id into channel config rendering, resolve account-scoped values when present, fall back both schema and value handling to the root channel config for root-backed channels, choose a patch path that matches the object being displayed, update extras/status consumers if they should reflect account-scoped fields, and add regression tests for account-scoped and root-fallback cases.

## Risks / Open Questions

- The PR as written can regress root-schema single-account channels if schema lookup moves directly to `channels.<id>.accounts.<accountId>` without falling back to `channels.<id>`.

- The PR body says the targeted Vitest run was still hanging in the contributor checkout, and current main does not have the proposed focused regression test.
