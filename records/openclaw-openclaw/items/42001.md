## Summary

Keep this PR open. Current main still clears the in-memory Gateway Token when the Overview WebSocket URL changes, and current browser coverage still asserts that behavior. This PR is a plausible fix for #41545 with useful regression coverage, but it overlaps with #41546 and #41718 and changes credential-boundary semantics that maintainers should settle before choosing one implementation.

## Best Possible Solution

Keep this PR open for maintainer review or rework. The best path is to resolve #41545 with one canonical implementation that stops surprising token loss during ordinary Overview URL edits while preserving the selected-gateway credential boundary, then reuse the strongest regression coverage from this PR and the other open candidate PRs before closing duplicates.

## Risks / Open Questions

- Preserving a bearer token across arbitrary WebSocket URL edits can send a token selected for one gateway to a different endpoint when the user clicks Connect.
- Because `applySettings` saves on every input event, preserving the token during URL typing can write the existing token under new or partial gateway URL sessionStorage keys unless the chosen implementation accounts for that.
- Multiple open candidate PRs target #41545, so maintainers should choose one canonical fix and decide whether token preservation should be pure in-memory edit state, storage-aware behavior, or a broader credential-boundary adjustment.
