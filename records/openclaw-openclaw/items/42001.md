## Summary

Keep this PR open. Current main still has the targeted Control UI behavior: editing the Overview WebSocket URL clears the in-memory Gateway Token, and current browser coverage still asserts that clearing behavior. The PR is a plausible implementation candidate for #41545, but it overlaps with other open candidates and needs maintainer judgment on the selected-gateway credential boundary before merge or supersession.

## Best Possible Solution

Keep this PR open for maintainer review or rework. The best path is to resolve #41545 with one canonical Control UI change that avoids surprising token loss during normal Overview URL edits while preserving the documented selected-gateway credential boundary, then land regression coverage and close the overlapping PRs as superseded.

## Risks / Open Questions

- Preserving the current bearer token while the user edits the WebSocket URL can send a credential selected for one gateway to a different endpoint if the user then clicks Connect.
- Because applySettings saves on every input, a direct preserve-token change can write the old token under new or partial edited gateway URL sessionStorage keys.
- There are multiple open implementation candidates for #41545, so this should not be closed or merged as the canonical fix without maintainer selection.
