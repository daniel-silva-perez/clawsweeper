## Summary

Keep this PR open. Current main still clears the Gateway Token when the Overview WebSocket URL changes, and current browser coverage still asserts that behavior. This PR remains a plausible implementation candidate for #41545, but it overlaps with open PR #41546 and changes credential-boundary semantics, so maintainers should pick one canonical fix rather than closing this as obsolete.

## Best Possible Solution

Keep this PR open for maintainer review or rework. The best path is to resolve #41545 with one canonical implementation that stops surprising token loss during ordinary URL edits while preserving the selected-gateway credential boundary, then reuse the strongest regression coverage from this PR and #41546.

## Risks / Open Questions

- Preserving a bearer token across arbitrary WebSocket URL edits can carry a token selected for one gateway to a different endpoint when the user clicks Connect.
- Because `applySettings` saves on input and token storage is URL-scoped, preserving the token while typing can write the existing token under new or partial gateway URL sessionStorage keys.
- There are multiple open implementation candidates for #41545, including #41546, so maintainers should choose one credential-boundary design before landing or closing duplicates.
