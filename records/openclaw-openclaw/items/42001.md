## Summary

Keep this PR open. Current main still has the WebSocket URL input behavior this PR targets: editing the Overview Gateway Access URL clears the in-memory Gateway Token, and current browser coverage still asserts that clearing behavior. The PR is a plausible implementation candidate for the open #41545 bug, but it overlaps with other open candidate PRs and needs maintainer judgment on the selected-gateway credential boundary before it is merged, closed, or superseded.

## Best Possible Solution

Keep this PR open for maintainer review or rework. The best path is to resolve #41545 with one canonical Control UI change that avoids surprising token loss during ordinary Overview URL edits while preserving the documented selected-gateway credential boundary, then land regression coverage and close the overlapping PRs as superseded.

## Risks / Open Questions

- Preserving the current bearer token while the user edits the WebSocket URL can send a token selected for one gateway to a different endpoint if the user then clicks Connect.
- Because settings are saved on input, a direct preserve-token implementation can persist the existing token under new or partial gateway URL sessionStorage keys unless the implementation accounts for that.
- Multiple open candidate PRs target #41545, so maintainers should choose one canonical patch and close the remaining PRs only after that path lands or is explicitly superseded.
- The live PR API reports the branch is currently dirty against main, so this candidate may need a current-main rebase or rework before final review.
