## Summary

Keep this PR open. Current main still has the exact Control UI behavior this PR tries to change: editing the Overview WebSocket URL clears the in-memory Gateway Token, and current browser coverage still asserts that clearing behavior. The PR is a relevant implementation candidate for #41545, but because it changes credential handling and overlaps with open PR #41546, maintainers should choose a canonical security-aware fix rather than closing this as obsolete.

## Best Possible Solution

Keep this PR open for maintainer review or rework. The best fix should stop ordinary WebSocket URL edits from unexpectedly erasing a manually entered Gateway Token, while preserving the selected-gateway credential boundary with explicit behavior for when a URL change should require confirmation, token re-entry, or scoped token transfer. Review it alongside #41545 and #41546, then land one canonical implementation with regression coverage for Overview/navigation behavior.

## Risks / Open Questions

- Preserving the token across arbitrary WebSocket URL edits can carry a selected-gateway bearer token to a different endpoint when the user clicks Connect.
- Because applySettings saves on input and session token keys are URL-scoped, preserving the token while typing can write the existing token under new or partial gateway URL sessionStorage keys.
- There are multiple open implementation candidates for #41545, including #41546, so maintainers should avoid merging or closing one path without choosing the intended credential-boundary semantics.
