## Summary

Keep this PR open. Current main still does not implement the intended Telegram streaming fix: draft previews build one `replyParams` object up front, reuse it for each new `sendMessage` after `forceNewMessage()`, and the dispatch path does not pass `replyToMode` into `createTelegramDraftStream`. The provided PR context also shows a maintainer pushed a narrow repair to the branch, while an Aisle security finding on the touched thread fallback still needs maintainer resolution or an explicit split before landing.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to land or port the repaired Telegram draft-stream change: pass and honor `replyToMode`, apply `reply_to_message_id` only to the first successful single-use preview send, preserve `message_thread_id` on later rotated previews, keep `replyToMode: "all"` semantics, and retain regression tests for forceNewMessage plus thread context. Before landing, resolve or explicitly split the thread-not-found fallback security concern.
