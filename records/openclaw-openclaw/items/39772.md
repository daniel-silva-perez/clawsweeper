## Summary

Keep this PR open. Current main still does not implement the intended Telegram streaming fix: draft-stream builds one reply parameter object and reuses it for each rotated preview send, dispatch does not pass `replyToMode` into the draft-stream path, and current tests still do not cover dropping `reply_to_message_id` after `forceNewMessage()` while preserving thread parameters. The PR remains a focused implementation candidate for the Telegram Deleted-message streaming artifact, though maintainers should address or explicitly split the thread-fallback security finding before landing.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to land or port the repaired Telegram draft-stream change: gate single-use preview `reply_to_message_id` after the first successful preview send, preserve `message_thread_id`, pass and honor `replyToMode` so `all` keeps its existing semantics, and keep the proposed regression tests. Before landing, resolve or explicitly split the thread-not-found fallback security concern.
