## Summary

Keep this PR open. Current main still does not implement a NetBird bind mode, so this is not cleanup-closeable as implemented; the PR is the active implementation candidate for the closed NetBird request and has useful real-deployment context plus positive user interest. It still needs maintainer review because the patch affects Gateway network exposure, has blocking correctness/security-adjacent gaps, and has drifted behind current main’s setup/link-helper refactors.

## What This Changes

The PR adds a `gateway.bind: "netbird"` mode across config/schema, Gateway bind resolution, auto-TLS, CLI/status, onboarding, tests, and changelog using `wt*` NetBird interface detection.

## Best Possible Solution

Refresh this PR against current main, update the current shared Gateway link/status/setup/docs surfaces, seed the actual NetBird HTTPS origin safely, load the generated TLS fingerprint anywhere auto-TLS is used, and land only after maintainers accept the NetBird trust boundary. If service-specific bind modes are no longer desired in core, keep the user problem tracked while designing a generic extension/plugin API for VPN bind profiles.

  - body: For `gateway.bind="netbird"`, the patch seeds HTTPS localhost origins but the UI is meant to open on the detected NetBird IP. `checkBrowserOrigin()` matches exact origins, so a browser at `https://<netbird-ip>:<port>` can still fail with `origin not allowed` unless users manually add the VPN IP origin.
