## Summary

Keep open. Current main still has no NetBird bind mode, so the user problem is not implemented, but this PR is currently dirty/non-rebaseable and needs a maintainer/security refresh before merge because it touches Gateway network exposure and misses several current shared helper paths.

## What This Changes

The PR adds `gateway.bind: "netbird"` across Gateway config/schema, CLI/status/onboarding surfaces, NetBird `wt*` interface detection, bind resolution, auto-generated self-signed TLS, tests, and changelog.

## Best Possible Solution

Refresh the PR against current main, implement NetBird support in the current central bind, connection-details, control-ui-links, status, origin-seeding, setup, and docs paths, seed or require the actual NetBird HTTPS origin, load the generated TLS fingerprint for auto-TLS probes, add focused tests, and get maintainer sign-off on whether this should remain service-specific or become a generic VPN bind profile.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Seed the detected NetBird origin:** `src/config/gateway-control-ui-origins.ts:35-38`
  - body: For `gateway.bind="netbird"`, the default origins become HTTPS localhost entries, but the Control UI is meant to open from the detected NetBird IP. Since browser origins are exact-matched, `https://<netbird-ip>:<port>` can still be rejected unless that origin is seeded or explicitly required.
  - confidence: 0.88
- **[P2] Move NetBird link handling to the current helper:** `src/commands/onboard-helpers.ts:459-480`
  - body: This branch updates the older onboarding link helper, but current main exports `resolveControlUiLinks` from `src/gateway/control-ui-links.ts`. After rebasing, dashboard, status, setup, and onboarding output need NetBird host and TLS handling in that shared helper or they will keep rendering the wrong scheme/host.
  - confidence: 0.82
- **[P2] Load the auto-TLS fingerprint for status probes:** `src/cli/daemon-cli/status.gather.ts:198-200`
  - body: The status summary switches NetBird probes to `wss://`, but the later probe TLS-runtime path still follows only explicit `gateway.tls.enabled`. Default NetBird setups auto-generate a self-signed cert, so status probing can fail certificate verification without loading the generated fingerprint.
  - confidence: 0.84
- **[P2] Refresh local connection details for current main:** `src/gateway/call.ts:150-153`
  - body: The PR updates URL/TLS selection in `src/gateway/call.ts`, but current main delegates local target construction to `src/gateway/connection-details.ts`. Without moving the NetBird auto-TLS logic there during rebase, CLI calls can continue using the stale local URL behavior.
  - confidence: 0.78
- **[P3] Add the required changelog attribution:** `CHANGELOG.md:9`
  - body: The added user-facing changelog entry is missing the repository-required `Thanks @...` attribution on the same single-line bullet.
  - confidence: 0.93

## Security Review

Status: needs_attention

Summary: No dependency, workflow, lockfile, or package-publishing change was found, but the patch changes Gateway network exposure and relies on weak NetBird interface attribution.

Concerns:

- **[medium] Strengthen NetBird interface attribution:** `src/infra/netbird.ts:14`
  - body: The new detector treats any non-internal interface whose name starts with `wt` as NetBird before binding the Gateway and enabling TLS. On hosts with another `wt*` interface or non-default NetBird naming, this can expose the Gateway on an unintended private network, so the implementation should use stronger attribution or make the trust boundary explicit in config/docs.
  - confidence: 0.67
