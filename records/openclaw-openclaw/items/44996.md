## Summary

Keep this PR open. Current main still lacks NetBird bind-mode and auto-TLS support, and the related NetBird feature request discussion plus PR comments show real user demand. The branch is useful source work, but it is not merge-ready as written because current main has moved several gateway URL/link paths, the NetBird Control UI origin and status-probe TLS paths remain incomplete, and the new VPN bind profile needs maintainer security review.

## What This Changes

The PR adds `gateway.bind: "netbird"` across Gateway config/schema, CLI and onboarding surfaces, NetBird `wt*` interface detection, bind resolution, auto-generated self-signed TLS, connection URL handling, tests, and changelog.

## Best Possible Solution

Refresh the PR against current main and implement NetBird support through the current central bind resolution, connection-details, Control UI link, status, origin-seeding, setup/onboarding, docs, and schema paths. The end state should either be an explicitly reviewed NetBird bind profile with secure defaults or a maintainer-approved generic VPN bind profile that covers NetBird without hard-coding a fragile trust boundary.

## Reproduction Assessment

Yes. Current main can be checked by source: `gateway.bind: "netbird"` is absent from the type/schema/resolver, and the PR body provides a concrete NetBird deployment path that explains the HTTPS/WebCrypto failure the feature is meant to solve.

## Solution Assessment

No as written. The direction is plausible, but the patch must be moved onto current shared Gateway helpers, seed or require the actual NetBird HTTPS origin, load the generated self-signed TLS fingerprint for probes, and tighten the NetBird interface trust model before it is the narrow maintainable fix.

## Security Review

Status: needs_attention

Summary: Security review needs attention because the patch adds a new Gateway network exposure mode and auto-TLS path with interface attribution that is plausible but not yet strong enough for a security-sensitive bind profile.

Concerns:

- **[medium] Strengthen NetBird interface attribution:** `src/infra/netbird.ts:14`
  - body: The new detector treats any non-internal interface whose name starts with `wt` as NetBird before binding the Gateway and enabling TLS. NetBird docs support `wt0` as the normal interface, but the implementation should avoid silently trusting unrelated `wt*` interfaces or make the trust boundary explicit in config/docs.
