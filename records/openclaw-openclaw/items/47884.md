## Summary

Keep open. Current main has improved remote-memory HTTP handling, including env-proxy-aware guarded fetch, but I cannot close this Bailian/DashScope memory embedding report as implemented: Qwen/DashScope memory embeddings are still documented as planned, and the current OpenAI embedding adapter still resolves endpoint/auth through the hardcoded `openai` provider path rather than proving that a custom `bailian-embedding` provider with `api: "openai-completions"` is honored end to end.

## Best Possible Solution

Keep this issue open and resolve it with a targeted memory-embedding fix or documentation correction: add a regression test for a `models.providers.bailian-embedding`/DashScope-style OpenAI-compatible embedding provider, then either make `api: "openai-completions"` route to a memory embedding adapter while preserving the custom provider's `baseUrl`/`apiKey`, or document the supported `memorySearch.provider: "openai"` plus `memorySearch.remote.baseUrl/apiKey` configuration and surface a clearer diagnostic when the unsupported custom-provider shape is used.
