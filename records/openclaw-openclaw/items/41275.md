## Summary

Keep open. Current main still rejects `timeoutSeconds: 0` in the Control UI and drops an explicit zero from the payload, while the cron protocol and runtime already treat zero as valid no-timeout mode. The PR is directionally correct, but it is not merge-ready because it leaves the existing controller test expectation and user-facing timeout copy behind.

## What This Changes

The PR changes Control UI cron timeout validation and payload construction so explicit `timeoutSeconds: 0` is accepted and sent in agent-turn cron payloads.

## Best Possible Solution

Amend this branch so the Control UI accepts and preserves explicit `timeoutSeconds: 0`, still rejects negative and malformed timeout input, updates the controller tests for validation and payload construction, and refreshes the English timeout copy plus generated locale output if the i18n workflow requires it.

## Reproduction Assessment

Yes. Current main gives a high-confidence reproduction path: `validateCronForm` rejects `timeoutSeconds: "0"`, `buildCronPayload` omits explicit zero, and the controller test currently asserts the old invalid-zero behavior.

## Solution Assessment

No, not as currently proposed. The controller parsing and payload direction is narrow and maintainable, but the mergeable fix should update the stale test expectation and timeout copy alongside the code change.
