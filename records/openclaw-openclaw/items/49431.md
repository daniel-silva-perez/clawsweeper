## Summary

Keep open. Current main still lacks the pending-inbound and active-turn persistence proposed here, so the PR is not obsolete; however, the latest branch remains an XL gateway/agent recovery change with unresolved capture-path, prompt-boundary, and local-state hardening issues. The author addressed some earlier review items, but the remaining concerns need maintainer/security review before merge.

## What This Changes

The PR adds a file-backed pending-inbound and active-turn store, startup replay/recovery hooks, embedded-run lifecycle persistence, JSON-file helper changes, an exported system-event cap, and related tests.

## Best Possible Solution

Converge on one restart-resilience design: persist inbound work at the actual channel or auto-reply boundary before drain rejection, replay external content with trusted:false, harden any recovery store with bounded reads and safe backup behavior, and explicitly align with or supersede the durable message-store work tracked in #65650.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Persist drain-time inbound messages before replaying them:** `src/process/command-queue.ts:230-231`
  - body: The PR adds startup replay and an isGatewayDraining helper, but the diff does not wire writePendingInbound into any production inbound handler before enqueue rejection. When the gateway is draining, the inbound message can still be rejected without ever being serialized, so replay only works for entries created by tests or manual store edits.
  - confidence: 0.91
- **[P2] Mark replayed inbound previews untrusted:** `src/gateway/server-startup-post-attach.ts:428-431`
  - body: The replay path builds system-event text from stored sender and message-preview fields, but omits trusted:false. System events default to trusted, so user-controlled replay content is formatted as trusted System context instead of receiving the external-content downgrade.
  - confidence: 0.88
- **[P2] Bound and harden corrupt pending-store reads:** `src/infra/pending-inbound-store.ts:333-352`
  - body: readPendingInboundFile reads the whole store before validation and writes invalid raw content to a predictable .bak path. A large or symlinked state file can cause memory/disk amplification or redirect the backup write, so the store needs a hard byte cap and safe exclusive/no-follow backup behavior or no raw backup.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The branch introduces security-sensitive local state-file and prompt-boundary behavior that still needs hardening before merge; no workflow, dependency, lockfile, or publishing changes were identified.

Concerns:

- **[medium] Trusted replay of user-controlled inbound preview:** `src/gateway/server-startup-post-attach.ts:428`
  - body: Pending replay formats sender/text fields from external inbound data and enqueues the result without trusted:false, so prompt formatting treats replayed content as trusted System context.
  - confidence: 0.88
- **[high] Predictable corrupt-store backup can follow symlinks:** `src/infra/pending-inbound-store.ts:352`
  - body: The parse-failure path writes raw pending-store content to a fixed .bak path with fs.writeFile; if the state directory is writable or misconfigured, a preexisting symlink can redirect the overwrite.
  - confidence: 0.79
- **[medium] Oversized pending store can exhaust memory or disk:** `src/infra/pending-inbound-store.ts:333`
  - body: The pending store is read entirely before parsing, and corrupt content is backed up in full, so a large local state file can force high memory use and disk amplification during startup or recovery.
  - confidence: 0.83
