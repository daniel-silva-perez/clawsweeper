## Summary

Keep this PR open. Current main still lacks durable pending-inbound replay and disk-backed active-turn recovery, but the latest PR head is not merge-ready because it does not appear to persist inbound messages from any production drain path and still has security-sensitive state-file and prompt trust-boundary gaps.

## What This Changes

The PR adds a file-backed pending-inbound/active-turn store, replays stored inbound events during gateway startup, records active embedded turns on disk, exports gateway drain state, and adds store/system-event tests.

## Best Possible Solution

Keep the PR open only as an implementation candidate: either finish a narrow production capture path plus state-file and prompt-boundary hardening here, or have maintainers explicitly supersede it with a merged restart-resilience design such as the SQLite direction in #65650.

Full review comments:

- **[P2] Persist drain-time inbound messages before replaying them:** `src/process/command-queue.ts:230-231`
  - body: This PR adds `isGatewayDraining()` but the diff never uses it, or `writePendingInbound`, from any production inbound handler. Startup replay will only process manually/test-created store entries, so messages rejected by `GatewayDrainingError` during restart are still lost.
  - confidence: 0.9
- **[P2] Mark replayed inbound previews untrusted:** `src/gateway/server-startup-post-attach.ts:428-431`
  - body: The replay event text includes stored sender/text fields from an external inbound message, but `enqueueSystemEvent` is called without `trusted: false`. That means the existing formatter emits it as `System:` instead of `System (untrusted):`, weakening the prompt trust boundary even with tag sanitization.
  - confidence: 0.86
- **[P2] Bound and harden corrupt-store reads:** `src/infra/pending-inbound-store.ts:333-352`
  - body: `readPendingInboundFile` reads the entire store into memory and, on parse failure, writes the full raw content to a predictable `.bak` path. A corrupted or attacker-controlled state file can still cause memory/disk amplification, and the backup path needs exclusive/no-follow creation or a bounded alternative.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The latest PR head introduces security-sensitive local state-file and prompt-boundary behavior that needs hardening before merge.

Concerns:

- **[medium] Trusted replay of user-controlled inbound preview:** `src/gateway/server-startup-post-attach.ts:428`
  - body: Pending replay formats sender/text preview from an external inbound message and enqueues it without `trusted: false`, so downstream prompt formatting treats it as a trusted `System:` event instead of untrusted data.
  - confidence: 0.86
- **[high] Predictable corrupt-store backup can follow symlinks:** `src/infra/pending-inbound-store.ts:352`
  - body: The parse-failure path writes raw content to `${filePath}.bak` with `fs.writeFile`; if the state directory is writable or misconfigured, a preexisting symlink can redirect that overwrite.
  - confidence: 0.78
- **[medium] Oversized pending store can exhaust memory or disk:** `src/infra/pending-inbound-store.ts:333`
  - body: The pending store is read entirely before parsing, and corrupt content is backed up in full, so a large local state file can force high memory use and disk amplification during startup/recovery.
  - confidence: 0.82
