## Summary

Keep open. Current main still lacks the pending-inbound store, startup replay, and disk-backed active-turn recovery this PR proposes, and the latest PR diff still has blocking correctness and security issues around production capture, prompt trust, and state-file hardening.

## What This Changes

The PR adds a file-backed pending-inbound and active-turn JSON store, startup replay/recovery hooks, active embedded-run write/clear calls, JSON-file helper changes, and related tests.

## Best Possible Solution

Keep this PR as an implementation candidate only if it is finished end to end: persist real drain-time inbound messages at the channel/inbound boundary, replay external content as untrusted system events, harden bounded state-file reads and backups, and align the design with or explicitly supersede the SQLite restart-resilience plan in #65650.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Persist drain-time inbound messages before replaying them:** `src/process/command-queue.ts:230-232`
  - body: This PR adds `isGatewayDraining()` but never uses it, or `writePendingInbound`, from any production inbound handler. When the gateway starts rejecting new lane enqueues during drain, those inbound messages still are not serialized, so startup replay only handles manually or test-created store entries and the advertised lossless restart path remains ineffective.
  - confidence: 0.91
- **[P2] Mark replayed inbound previews untrusted:** `src/gateway/server-startup-post-attach.ts:428-431`
  - body: `eventText` includes stored sender and text fields from an external inbound message, but the call to `enqueueSystemEvent` omits `trusted: false`. The existing formatter therefore emits the replay as a trusted `System:` line instead of `System (untrusted):`, weakening the prompt trust boundary even with tag sanitization.
  - confidence: 0.86
- **[P2] Bound and harden corrupt-store reads:** `src/infra/pending-inbound-store.ts:333-352`
  - body: `readPendingInboundFile` reads the entire store before validating size and writes corrupt raw contents to a predictable `.bak` path with `fs.writeFile`. A corrupted or attacker-controlled state file can force memory/disk amplification, and a preexisting symlink can redirect the backup write; add a hard read cap and use bounded exclusive/no-follow backup creation or avoid raw backups.
  - confidence: 0.83

## Security Review

Status: needs_attention

Summary: The latest PR head introduces security-sensitive local state-file and prompt-boundary behavior that needs hardening before merge.

Concerns:

- **[medium] Trusted replay of user-controlled inbound preview:** `src/gateway/server-startup-post-attach.ts:428`
  - body: Pending replay formats sender/text preview from an external inbound message and enqueues it without `trusted: false`, so downstream prompt formatting treats it as a trusted `System:` event instead of untrusted data.
  - confidence: 0.86
- **[high] Predictable corrupt-store backup can follow symlinks:** `src/infra/pending-inbound-store.ts:352`
  - body: The parse-failure path writes raw content to `${filePath}.bak` with `fs.writeFile`; if the state directory is writable or misconfigured, a preexisting symlink can redirect that overwrite.
  - confidence: 0.78
- **[medium] Oversized pending store can exhaust memory or disk:** `src/infra/pending-inbound-store.ts:333`
  - body: The pending store is read entirely before parsing, and corrupt content is backed up in full, so a large local state file can force high memory use and disk amplification during startup or recovery.
  - confidence: 0.82
