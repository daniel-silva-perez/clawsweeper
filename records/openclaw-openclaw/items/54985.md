## Summary

Keep open. Current main does not implement this PR’s fixed `X-OpenClaw-*` request headers or `llm_output.isFallback`. Main has adjacent static provider headers and provider-owned transport turn-state hooks, but those do not cover the PR’s run id, fallback status, or turn-index contract for generic AI gateway tracing. The PR also still needs design/security review because it would add new outbound metadata on model-provider requests.

## Best Possible Solution

Do not close this PR as implemented. The best path is maintainer review of whether OpenClaw should expose a generic AI-gateway tracing header contract at all; if yes, implement it with an explicit privacy/security posture, propagate true fallback state from the fallback runner, make turn IDs unique across retries, document the header contract, and add focused tests for HTTP headers, fallback status, and retry behavior. If maintainers prefer plugin ownership, extend the provider transport hook with the missing run/fallback context instead of adding unconditional core headers.

## Risks / Open Questions

- Security review needed: the PR would send stable session/run/fallback/turn identifiers as outbound HTTP headers to model providers or gateways, which broadens metadata exposure and should be opt-in or explicitly documented if accepted.
- Correctness risk: the PR discussion reports duplicate `X-OpenClaw-Turn-Id` values across retry paths before assistant messages are persisted.
- Correctness risk: the PR discussion reports `X-OpenClaw-Fallback` can still be mislabeled in paths that derive fallback status from executed-attempt count rather than the selected fallback candidate.
- No source evidence or provided PR context shows regression tests locking the new header contract, fallback semantics, or retry uniqueness.
