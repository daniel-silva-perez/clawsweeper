## Summary

Keep open. Current main still lacks the requested before_prompt_build tool/skill narrowing contract, and the PR addresses a real plugin-owned prompt-surface problem with supporting field data in the discussion. The proposed patch is not ready to merge as-is because the skills filtering path can drop prompt/provider mutations, the tool catalog exposed to hooks omits client-hosted tools, and the snapshot skill path reloads live workspace skills instead of filtering the session snapshot.

## What This Changes

The PR extends before_prompt_build hooks with availableTools/availableSkills inputs and toolsAllow/skillsAllow outputs, then applies those outputs in the embedded runner to narrow per-turn tools and the injected skills catalog.

## Best Possible Solution

Keep the PR open, rebase it onto current main, preserve the current host-hook and prompt-helper behavior, and land the dynamic narrowing seam only after the tool catalog, skill snapshot, and system-prompt rebuild paths are covered by focused tests and matching docs/changelog updates.

## Reproduction Assessment

Yes. Current-main source inspection reproduces the missing capability: before_prompt_build has no tool/skill catalog inputs or allow-list outputs, and the PR-head bugs can be reproduced with a hook that returns skillsAllow plus system prompt mutations, or with a run that includes clientTools.

## Solution Assessment

No as written. The direction is maintainable and fits the plugin boundary better than MCP-server-side workarounds, but the patch should reuse the existing prompt transform/composition path, include every model-visible tool in the hook catalog, and filter snapshot-backed skills without reloading the workspace.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve hook system prompt edits when filtering skills:** `src/agents/pi-embedded-runner/run/attempt.ts:1946-1952`
  - body: When skillsAllow is set, this block rebuilds systemPromptText from appendPromptParams and applies it directly. That overwrites any hookResult.systemPrompt, prependSystemContext, or appendSystemContext that was applied just above, and it also skips the provider transform path used for the original system prompt. A plugin returning skillsAllow with other prompt mutations will silently lose those mutations for the turn.
  - confidence: 0.9
- **[P2] Include client tools in the hook-visible catalog:** `src/agents/pi-embedded-runner/run/attempt.ts:1839`
  - body: availableTools is built from effectiveTools only, but client-hosted tools are converted later into clientToolDefs and registered in allCustomTools. A plugin that derives toolsAllow from availableTools has no way to include those client tools, and the later setActiveToolsByName(hookResult.toolsAllow) can deactivate them. The hook input and filtering should account for all model-visible tool names or explicitly preserve client tools.
  - confidence: 0.86
- **[P2] Filter snapshot skills without reloading the workspace:** `src/agents/pi-embedded-runner/run/attempt.ts:1927-1938`
  - body: For snapshot-backed runs, this branch ignores params.skillsSnapshot and reloads current workspace skill entries. That can inject newer on-disk skill content or drop skills that were present in the session snapshot, breaking the snapshot consistency that the existing skills runtime preserves. Filter the snapshot-resolved skill data instead of loading fresh entries during the turn.
  - confidence: 0.78
