## Summary

Keep open. Current main still lacks the requested `thinking-events` Gateway capability and cap-gated thinking routing, so this PR is not obsolete; however, the proposed patch needs changes before merge because its reasoning replacement heuristic can corrupt delta-only thinking streams and its pre-registered recipient cleanup can leave stale reasoning subscribers alive.

## What This Changes

The PR adds a `thinking-events` Gateway client capability, raw reasoning delta tracking, run-scoped recipient routing for thinking events, related Gateway wiring, tests, and generated SDK baseline updates.

## Best Possible Solution

Land a corrected version of the cap-gated thinking-event design: keep the new opt-in Gateway capability, route thinking only to authorized run recipients, preserve raw accumulated/delta semantics for both full snapshots and pure delta events, finalize recipient registries on every early failure path, and update the Gateway protocol/docs/tests after rebasing onto current main.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve incremental reasoning deltas:** `src/agents/pi-embedded-subscribe.ts:653-654`
  - body: This replacement branch also handles pure `thinking_delta` fallback chunks, where the handler receives only the new token rather than a full snapshot. After an initial `"Checking"` chunk, a later `" files"` chunk has no prefix/suffix overlap with the prior text, so this code treats it as a replacement and emits truncated or empty `delta`/`text` instead of `"Checking files"`. The fix needs an explicit full-snapshot vs delta signal, or a fallback that appends known delta-only events.
  - confidence: 0.87
- **[P2] Finalize pre-registered recipients on dispatch failure:** `src/gateway/server-methods/chat.ts:1554-1559`
  - body: Recipients are registered before `dispatchInboundMessage`, but if dispatch rejects before any lifecycle event is emitted, nothing calls `markFinal` for the tool/thinking recipient registries. Those run-to-connection entries then remain for the registry TTL and can receive events for a later reused client run id after the shorter dedupe cache expires. Mark the pre-registered run final or remove it in the `.catch/.finally` failure path.
  - confidence: 0.83

## Security Review

Status: needs_attention

Summary: The diff improves the intended privacy model, but the early-failure cleanup gap can leave stale thinking-event subscribers able to receive reasoning for a reused run id.

Concerns:

- **[medium] Stale thinking subscribers after early dispatch failure:** `src/gateway/server-methods/chat.ts:1558`
  - body: `chat.send` pre-registers thinking recipients before dispatch, but the failure path can finish without a lifecycle event, so `thinkingEventRecipients.markFinal` is never called. Because these entries outlive the dedupe cache, a later reused client run id could send reasoning tokens to a stale connection.
  - confidence: 0.82
