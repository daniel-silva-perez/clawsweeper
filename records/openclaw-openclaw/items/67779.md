Keep open. Current main still matches the reported WebChat live-update failure mode: mixed-case non-default agent deep links can remain mixed-case in Control UI state, while gateway chat/session events use canonical lowercase session keys, and the UI handlers still require exact string equality before applying chat or transcript updates.

Make Control UI WebChat render streaming or final assistant replies for mixed-case non-default agent sessions without requiring a manual browser refresh.

## Repair Work Prompt

Fix the Control UI WebChat live-update bug for mixed-case non-default agent session deep links. Current source evidence: `applySettingsFromUrl` can store `agent:DB_Dude:main` as the active UI session, while the gateway canonicalizes agent session keys to lowercase and broadcasts `agent:db_dude:main`; `handleChatEvent` and `handleSessionMessageGatewayEvent` then drop those events via exact string comparisons. Implement a generic session-key canonicalization/equivalence helper in the UI layer, preferably near `ui/src/ui/session-key.ts`, and use it so active mixed-case agent sessions apply canonical lowercase `chat` and `session.message` events. Do not special-case `DB_Dude`, `ViewDude`, LM Studio, or the reporter's local paths. Preserve filtering for truly different sessions. Add focused regression tests for active `agent:DB_Dude:main` receiving `agent:db_dude:main` final/delta chat events and deferred or direct `session.message` reloads. If URL/session storage is canonicalized instead of comparison-time equivalence, update `app-settings` tests to prove mixed-case URL sessions are normalized safely. This is user-facing, so add a single-line changelog fix entry with `Thanks @rfwilliams` unless maintainers direct otherwise.
