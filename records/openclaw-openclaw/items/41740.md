## Summary

Keep open. Current main still has the reported TLS fingerprint gap in the shared operator approvals GatewayClient path used by Discord approval buttons and native approval runtimes. The prior related fixes were closed unmerged, so they remain useful prior art rather than implementation evidence.

## Best Possible Solution

Keep this issue open. The best fix belongs in the shared operator approvals Gateway bootstrap/client path: resolve the same local TLS runtime fingerprint used by callGateway when gateway.tls.enabled is true, the target is local wss://, and no explicit CLI/env URL override or remote URL is in effect; pass that fingerprint into GatewayClient; add focused regression coverage for resolveGatewayClientBootstrap or createOperatorApprovalsGatewayClient plus the Discord/native approval paths. The closed #41741 and #46232 are useful prior art but should not close this issue unless an equivalent tested fix lands on main.
