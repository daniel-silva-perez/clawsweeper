## Summary

Current main still has the reported gap. Discord approval buttons resolve through the shared operator-approvals Gateway client, and that client still constructs GatewayClient without a tlsFingerprint. The normal callGateway path still contains the local self-signed TLS fingerprint logic, so this approval path remains inconsistent.

## Best Possible Solution

Keep this issue open. Fix the shared operator-approval Gateway bootstrap/client path so it resolves the local gateway TLS runtime under the same local wss:// conditions as callGateway, passes tlsFingerprint to GatewayClient, and adds focused regression tests for local self-signed TLS plus override/remote cases. The closed #41741 and #46232 remain useful prior art but should not close the issue unless an equivalent tested fix lands on main.
