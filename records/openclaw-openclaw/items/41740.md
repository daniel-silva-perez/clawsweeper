## Summary

Keep open. Current main still leaves the reported local self-signed TLS gap in the shared operator-approval Gateway client path: Discord approval button resolution reaches the shared client, bootstrap still carries only URL/auth data, and GatewayClient only enables pinned self-signed TLS handling when tlsFingerprint is supplied. The prior closed PRs #41741 and #46232 are useful implementation context, but neither is merged into the inspected main behavior.

## Best Possible Solution

Keep this issue open. Fix the shared operator-approval Gateway bootstrap/client path so it resolves the local Gateway TLS runtime under the same conditions as callGateway, passes tlsFingerprint into GatewayClient, and adds focused regression tests for local self-signed wss plus CLI/env override and configured remote cases. The closed #41741 and #46232 are useful prior art but should not close this until an equivalent tested fix lands on main.
