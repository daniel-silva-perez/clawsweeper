## Summary

Keep open. Current main still has the reported source-level gap: Discord approval buttons and the native approval runtime both use the shared operator-approvals GatewayClient path, and that path still constructs GatewayClient without a tlsFingerprint. The local self-signed TLS fingerprint logic exists in callGateway, but it has not been carried into the operator-approvals bootstrap/client path. Related fix PRs #41741 and #46232 were closed unmerged, so they are useful prior art but not a reason to close this issue.

## Best Possible Solution

Keep this issue open. The best fix belongs in the shared operator approvals gateway bootstrap/client path: resolve the same local gateway TLS runtime fingerprint used by callGateway when gateway.tls.enabled is true, the target is local wss://, and no explicit CLI/env URL override or remote URL is in effect; pass that fingerprint into GatewayClient; add focused regression coverage for resolveGatewayClientBootstrap or createOperatorApprovalsGatewayClient plus the Discord/native approval paths. The closed PRs #41741 and #46232 are useful prior art but should not close this issue unless an equivalent tested fix lands on main.
