## Summary

Keep this PR open. Current main still does not implement BlueBubbles tapback/reaction exec approval resolution, and the PR is the active implementation candidate, but it needs security rework before merge. The latest maintainer review asks for GUID-bound approval resolution instead of parsing replied-to chat text, and current main still has only generic BlueBubbles reaction forwarding.

## Best Possible Solution

Keep this PR open for rework. The best path is a BlueBubbles approval resolver that records the delivered approval prompt message GUID, resolves only reactions against tracked pending approvals, verifies the reactor through the same approver model as native approval clients or `/approve`, uses the narrow approval gateway/runtime SDK surface such as `resolveApprovalOverGateway`, handles both real `processReaction` webhooks and any legacy tapback-message fallback, and adds focused tests for success, unauthorized senders, stale or untracked GUIDs, spoofed reply text, and gateway failure behavior.

## Risks / Open Questions

- Merging the current PR as-is could resolve privileged exec approvals from spoofable or forwarded replied-to text instead of a tracked bot-sent approval prompt GUID.
- The PR broadens the plugin SDK with raw `callGateway`, while current main already has narrower approval gateway helpers better suited to this flow.
- The PR handles tapback-shaped normal messages but does not cover the real BlueBubbles `processReaction` webhook path used for normalized reaction events.
- Using BlueBubbles command authorization alone may diverge from the approver model used by native approval clients and `/approve` handling.
