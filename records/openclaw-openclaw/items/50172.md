## Summary

Keep this PR open. Current main still does not implement BlueBubbles tapback/reaction exec approval resolution, and the PR remains a plausible implementation candidate, but the current diff is security-sensitive and needs rework before merge. The latest maintainer discussion specifically asks for GUID-bound approval resolution instead of parsing replied-to chat text.

## Best Possible Solution

Keep this PR open for rework. The best path is a BlueBubbles-native approval resolver that records the delivered approval prompt message GUID, resolves only reactions against tracked pending approvals, verifies the reactor through the same approver model as `/approve` or native approval clients, uses the narrow approval gateway/runtime SDK surface such as `resolveApprovalOverGateway`, handles both real `processReaction` webhooks and any legacy tapback-message fallback, and adds focused tests for success, unauthorized senders, stale or untracked GUIDs, spoofed reply text, and gateway failure behavior.

## Risks / Open Questions

- Merging the PR as-is could let approval resolution depend on spoofable/free-form replied-to text instead of a tracked bot-sent approval prompt GUID.
- The PR adds raw `callGateway` to `src/plugin-sdk/gateway-runtime.ts`, broadening the public plugin SDK surface beyond the narrow approval gateway helpers already present on main.
- The PR handles tapback-shaped normal messages but not the existing BlueBubbles `processReaction` webhook path used for real reaction events.
- Using BlueBubbles command authorization alone can diverge from the normal approval approver model used by native approval clients.
