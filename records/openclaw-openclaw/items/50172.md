## Summary

Keep open. Current main still does not implement BlueBubbles tapback approval resolution, but this PR is not ready to merge because the proposed resolver trusts reply text instead of binding reactions to a bot-sent pending approval message, misses the real BlueBubbles reaction webhook path, and widens the public gateway SDK surface for a feature that already has a scoped approval helper.

## What This Changes

This PR adds BlueBubbles thumbs-up/thumbs-down tapback handling for exec approval prompts, tests that shortcut, and a `callGateway` export from `plugin-sdk/gateway-runtime`.

## Best Possible Solution

Implement BlueBubbles approval reactions through a pending-approval binding keyed by the delivered bot message GUID, verify the reactor with the same approval-specific authorization model used by `/approve` or native approval capabilities, handle both text-shaped tapbacks and `message-reaction` webhooks through one helper, and resolve through the existing scoped approval gateway helper without expanding the generic SDK surface.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: on current main, a BlueBubbles `message-reaction` payload is routed to `processReaction` and only enqueued as a system event; in the PR diff, a `new-message` tapback with forged `replyTo.text` containing `Full id:` reaches `exec.approval.resolve`.

## Solution Assessment

No. The proposed shortcut is not the best way to solve the issue because it treats user-controlled reply text as authority and bypasses the existing approval-specific authorization/scoped gateway seams; a GUID-bound pending approval lookup is the safer narrow design.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Bind tapbacks to sent approval prompts:** `extensions/bluebubbles/src/monitor-processing.ts:887-899`
  - body: The resolver extracts `approvalId` from `replyToBody`, which can be forged, quoted, or cached from arbitrary inbound text. An allowlisted sender can therefore react to text shaped like an approval prompt and submit `exec.approval.resolve` without proving the reacted-to GUID is a bot-sent pending approval or that they are the intended approver. Track the outbound approval prompt GUID to approval id and authorize the reactor before resolving.
  - confidence: 0.94
- **[P2] Handle BlueBubbles reaction webhooks:** `extensions/bluebubbles/src/monitor-processing.ts:885`
  - body: This approval path only runs in `processMessage`, but real `message-reaction`/`reaction` webhook payloads are normalized as reactions and routed to `processReaction`. Those events will still be forwarded as agent-visible reaction events and never resolve approvals, so the feature will fail on installations that emit structured reaction webhooks.
  - confidence: 0.9
- **[P2] Use the scoped approval gateway helper:** `src/plugin-sdk/gateway-runtime.ts:4`
  - body: The feature only needs approval resolution, but this adds generic `callGateway` to the public plugin SDK. Use the existing `approval-gateway-runtime`/operator-approvals client path so the public surface stays limited to approval RPCs instead of promoting a broad gateway caller for this one channel shortcut.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The PR introduces an approval-resolution path with concrete spoofing risk and broadens a public gateway RPC SDK surface.

Concerns:

- **[high] Approval tapback spoofing:** `extensions/bluebubbles/src/monitor-processing.ts:887`
  - body: The new resolver trusts regex matches in `replyToBody` and only checks BlueBubbles command authorization before resolving exec approvals. Because the source text is not bound to a bot-sent approval prompt GUID or pending approval record, an authorized chat participant can craft or quote approval-shaped text and react to it to submit a privileged decision.
  - confidence: 0.94
- **[medium] Generic gateway RPC exposed to plugins:** `src/plugin-sdk/gateway-runtime.ts:4`
  - body: The PR exports `callGateway` from `plugin-sdk/gateway-runtime` even though approval resolution already has a scoped helper. That expands the public plugin contract around host gateway calls and should not be mixed into a channel UX change without explicit SDK/security review.
  - confidence: 0.78
