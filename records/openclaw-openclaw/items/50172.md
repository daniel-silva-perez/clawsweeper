## Summary

Keep this PR open. Current main still does not implement BlueBubbles tapback/reaction exec approval resolution: real reaction webhooks are routed to `processReaction` and enqueued as agent-visible system events, tapback-shaped normal messages are only rewritten into reply-tagged text, and the BlueBubbles plugin has no native approval capability. The PR remains a useful implementation candidate, but maintainer review and the PR diff still point to a security-sensitive design gap around parsing replied-to text and broad raw gateway access instead of binding reactions to tracked delivered approval message GUIDs.

## Best Possible Solution

Keep this PR open for rework. The best path is a BlueBubbles approval resolver that records the delivered approval prompt message GUID, resolves only reactions against tracked pending approvals, verifies the reactor with the same approval authorization model used by `/approve` or native approval clients, uses the narrow `resolveApprovalOverGateway` SDK surface, handles both real BlueBubbles reaction webhooks and any legacy tapback-message fallback, and adds focused tests for success, deny, unauthorized senders, stale or untracked GUIDs, spoofed reply text, and gateway failure behavior.

## Risks / Open Questions

- Closing this PR as implemented would be inaccurate because current main still only forwards BlueBubbles reactions to the agent instead of resolving exec approvals.
- Merging the PR as currently described could resolve privileged exec approvals from spoofed, quoted, or forwarded reply text rather than a tracked OpenClaw-sent approval prompt GUID.
- The PR's raw `callGateway` SDK export broadens plugin access to gateway RPC beyond the existing approval-specific `resolveApprovalOverGateway` helper, creating unnecessary public SDK privilege surface.
