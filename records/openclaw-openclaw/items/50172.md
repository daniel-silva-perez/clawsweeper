## Summary

Keep this PR open. Current main still does not implement BlueBubbles tapback/reaction exec approval resolution, so the contribution is not obsolete, but the PR’s current design remains security-sensitive and needs rework before it should land.

## Best Possible Solution

Keep this PR open for rework. The best path is a BlueBubbles-native implementation that records the delivered approval prompt message GUID, resolves only reactions against tracked pending approvals, verifies the reactor is authorized for that approval, uses the narrow approval gateway/runtime SDK surface, handles both `processReaction` webhooks and legacy tapback-message fallback where needed, and adds focused tests for success, unauthorized senders, stale/untracked GUIDs, and gateway failure behavior.

## Risks / Open Questions

- The PR diff parses approval IDs from `replyToBody`, which is free-form chat text unless bound to a bot-sent approval message GUID and pending approval state.
- The PR adds a raw `callGateway` export to a public plugin SDK subpath; current main has narrower approval-specific gateway helpers that better match this need.
- The PR handles tapback-like normal messages, but current BlueBubbles real reaction webhooks go through `processReaction`, so real installations emitting reaction events would still not resolve approvals.
- Using BlueBubbles `commandAuthorized` alone can diverge from the normal `/approve` approver authorization model.
