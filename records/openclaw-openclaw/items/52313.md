## Summary

Current main still has the reported respawn argument behavior: detached gateway respawn reuses `process.execArgv` plus `process.argv.slice(1)` for both normal SIGUSR1 restart and `update.run` respawn. The update path has a health fallback, but it still does not rewrite pnpm versioned package realpaths to a stable `openclaw.mjs` or package-root entrypoint, and the related implementation PR was closed unmerged.

## Best Possible Solution

Keep this issue open and implement a narrow respawn planner in `src/infra/process-respawn.ts`: detect pnpm versioned OpenClaw realpaths, resolve the stable package root or symlinked `openclaw.mjs` entrypoint, preserve source/dev entrypoints, and cover both `restartGatewayProcessWithFreshPid()` and `respawnGatewayProcessForUpdate()` with targeted regression tests. Reuse or adapt the stable symlink logic already present in `src/daemon/program-args.ts` where it fits, and treat closed PR #52319 as prior art rather than a landed fix.

## Risks / Open Questions

- Closing now would leave the package-manager versioned-realpath respawn failure untracked for normal SIGUSR1 restart paths, and current tests still assert argv reuse.
- The update health fallback reduces dead-gateway risk, but a stale package path can still fail to launch the updated package and fall back to the old in-process module graph instead of proving the stable wrapper behavior requested here.
- A future fix touches process spawning across Node, Bun, source checkouts, package installs, execArgv, and NODE_OPTIONS; it should stay focused and avoid broad unrelated script or lifecycle changes from the closed XL PR.
