## Summary

Keep open. Current main still has the reported broad private-network web-tools SSRF policy, so this PR is not obsolete; the latest branch is directionally useful but still has a blocking Plugin SDK API baseline gap and a smaller proxy-env-dependent test issue. The Dockerfile concern raised by Greptile appears to be cross-PR noise because this PR does not touch Dockerfile and #74359 tracks that separate Bun image work.

## What This Changes

The branch splits web-tools SSRF handling into strict trusted and explicit self-hosted helpers, switches SearXNG to the self-hosted helper, exports the helpers through Plugin SDK provider entrypoints, and adds guarded-fetch SSRF tests.

## Best Possible Solution

Land a maintainer-approved SSRF fix that keeps fixed public web-search providers strict by default, preserves a clearly named and validated self-hosted private-network path for SearXNG-style endpoints, updates generated SDK baseline/docs for any public helper exports, and lets #74357 close only after the reviewed fix merges.
