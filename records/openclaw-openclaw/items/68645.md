## Summary

Keep open. Current main does not implement the new `runTimeoutSeconds` wall-clock kill, and the PR is supported by concrete production symptoms plus related contributor comments, but the draft branch is currently dirty against main and the patch is incomplete for OpenClaw's public cron contracts and update paths.

## What This Changes

The PR adds a cron `agentTurn` `runTimeoutSeconds` wall-clock timeout, proactively finalizes timed-out cron task-ledger rows, and adds regression tests for timeout ledger closure and unresponsive isolated runners.

## Best Possible Solution

Rework the PR onto current main so cron keeps the existing timeout deferral and timeout cleanup semantics, adds a separate hard wall-clock ceiling, finalizes the task ledger idempotently on timeout, and wires the new field through protocol schemas, update merging, normalization, docs, and focused cron/task tests.

## Reproduction Assessment

Yes. The PR provides a production reproduction with stale task counts and proposes focused fake-timer tests; current main inspection confirms there is no separate `runTimeoutSeconds` path and that ledger finalization is still primarily in the apply-outcome path.

## Solution Assessment

No, not as currently proposed. The direction is sound, and the comments support the shape, but the patch is incomplete against the public cron API and current main's timeout implementation; a rebased, fully wired version is the safer fix.

  - body: Adding `runTimeoutSeconds` to the TypeScript payload type is not enough: cron RPC payload schemas still have `additionalProperties: false` and do not list this field, so Gateway clients that send the new option will be rejected before timer code sees it. Please add the field to the cron payload/create/update schemas and cover that path in tests.
  - confidence: 0.9

Legacy-style create/update inputs would persist an out-of-schema top-level key or fail strict validation, unlike the existing `timeoutSeconds` path this mirrors.
  - confidence: 0.86
