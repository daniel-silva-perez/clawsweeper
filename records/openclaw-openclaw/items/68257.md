## Summary

Keep open. Current main still lacks a top-level auth reload rule, so auth profile refresh paths still fall through to the conservative restart branch; this PR is a focused implementation for the paired open bug report and I did not find a blocking functional or security issue in the diff.

## What This Changes

This PR adds a top-level `auth` no-op reload rule in the gateway config reload planner and regression tests for OAuth auth-profile refreshes plus mixed `gateway.auth.*` restart batches.

## Best Possible Solution

Land this PR or an equivalent narrow patch on main so OAuth auth-profile refreshes do not restart the gateway, while preserving the existing `gateway.auth.*` restart behavior and keeping the regression tests with the reload planner.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence reproduction path: `auth.profiles.<provider>:<email>` matches no reload rule and therefore takes the unmatched-path restart branch in `buildGatewayReloadPlan()`.

## Solution Assessment

Yes, with one scope caveat. Adding an `auth` no-op rule is the smallest change that fixes current auth-profile refresh restarts while the existing `gateway` rule still covers `gateway.auth.*`; a stricter `auth.profiles`-only rule would be the safer alternative only if maintainers want future auth subkeys to keep default restart behavior.
