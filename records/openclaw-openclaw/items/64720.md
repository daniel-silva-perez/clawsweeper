## Summary

Keep this PR open, but do not merge the current branch as-is. Current main still leaves channel message bodies and thread context without external-content boundaries, so the security goal is valid; the branch fixes the earlier queue-summary regression but still classifies trust from reply-routing metadata and an incomplete internal-provider allowlist, which can wrap trusted heartbeat and cron prompts as attacker content.

## What This Changes

This PR adds `wrapExternalContent` wrapping for channel message bodies and thread-context bodies, extends the external-content source labels with `channel`, and adds inbound-text unit coverage for normalization, sanitization, provider classification, and channel-body wrapping.

## Best Possible Solution

Revise the PR so wrapping is driven by actual ingress/source provenance rather than reply-routing metadata; wrap real external channel body and thread-context text; preserve raw queue summaries; keep `heartbeat`, `cron-event`, `exec-event`, CLI, node, and other internal sources trusted; and add regression coverage for channel bodies, thread context, routed heartbeat/cron delivery, and internal-only scheduler prompts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Derive wrapping from the ingress provider:** `src/auto-reply/reply/get-reply-run.ts:267-268`
  - body: `OriginatingChannel` is reply-routing metadata, not proof that the current prompt body came from an external user. Heartbeat and cron runs can set it to the delivery channel while `Provider` remains `heartbeat` or `cron-event`, so this branch wraps trusted scheduler prompts as external attacker content and can distort scheduled automation. Use ingress/source trust metadata for wrapping and leave `OriginatingChannel` for routing.
  - confidence: 0.92
- **[P1] Keep internal event providers trusted:** `src/auto-reply/reply/inbound-text.ts:7-18`
  - body: `isExternalChannelProvider` returns true for every provider except `node`, `cli`, and `exec-event`, but current main emits internal scheduler prompts with `Provider: "heartbeat"` and `Provider: "cron-event"`. Internal-only heartbeat/cron prompts will therefore be fenced as untrusted channel messages even when no external user supplied the body. Prefer a positive channel-provider check or include all internal event providers.
  - confidence: 0.9
