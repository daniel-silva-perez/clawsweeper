## Summary

Keep this PR open for maintainer/security review. The hardening goal is valid because current main still sends channel bodies and thread context into the auto-reply prompt without external-content boundaries, but the proposed trust classification can misclassify internal heartbeat/cron prompts as external channel content.

## What This Changes

This PR wraps channel message bodies and thread-context bodies with external-content boundaries, adds a channel source label, and adds inbound-text tests for normalization, sanitization, provider classification, and wrapping.

## Best Possible Solution

Revise the PR so external-content wrapping is derived from the actual ingress/source trust class, not reply-routing metadata; keep wrapping for real external channel body and thread context; preserve raw queue summaries; explicitly classify heartbeat, cron-event, exec-event, node, CLI, and other internal sources as trusted; and add regression coverage for both external channel input and internal scheduled turns.

## Reproduction Assessment

Yes. Current-main source shows channel BodyForAgent and thread context are only sanitized, while the PR diff and existing heartbeat/cron tests show a high-confidence path where internal Provider values can coexist with external OriginatingChannel routing metadata.

## Solution Assessment

No. The PR is directionally useful, but the current implementation is not the best fix because it uses route metadata and an incomplete trusted-provider allowlist; the safer fix is an explicit ingress trust/source classification seam used only for prompt wrapping.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Derive wrapping from the ingress provider:** `src/auto-reply/reply/get-reply-run.ts:268-269`
  - body: `OriginatingChannel` is reply-routing metadata, not ingress trust metadata. Internal heartbeat/cron turns can carry an external channel for delivery while the prompt source remains trusted, so preferring this field wraps scheduler prompts as untrusted channel text and can degrade scheduled actions. Use the actual source provider or trust class for wrapping.
  - confidence: 0.9
- **[P1] Treat internal event providers as trusted:** `src/auto-reply/reply/inbound-text.ts:7`
  - body: `isExternalChannelProvider` returns true for every provider except `node`, `cli`, and `exec-event`, but current production paths use `heartbeat` and `cron-event` as internal prompt sources. Those internal turns would be fenced with external-content warnings even when no channel user message is involved.
  - confidence: 0.88
