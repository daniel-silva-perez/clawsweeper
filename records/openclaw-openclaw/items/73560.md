## Summary

Keep open. The goal is still useful, but current main does not expose `diagnostics.stuckSessionAbortMs`, and this PR's abort path would not reliably unblock the session lane because it clears only queued entries while leaving the active lane slot occupied; it also nests the abort threshold under warning emission, changing valid configured timing.

## What This Changes

The PR adds a `diagnostics.stuckSessionAbortMs` config/schema/help/reload surface plus diagnostics heartbeat logic and tests for aborting stuck sessions after a configurable threshold.

## Best Possible Solution

Rework the feature through the existing stuck-session recovery runtime: resolve the abort threshold independently from warning emission, pass an explicit active-abort signal only after that threshold, release lanes through the established abort/drain/reset path, make queued-message handling intentional, and align config docs, generated schema checks, and heartbeat integration coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Route aborts through the lane reset path:** `src/logging/diagnostic.ts:678-680`
  - body: `clearCommandLane` only rejects queued entries and does not release the active task slot. With `maxConcurrent=1`, a stuck active task can still hold the lane after diagnostics marks the session idle and zeroes queue depth, so later messages may remain blocked while diagnostics reports recovery.
  - confidence: 0.91
- **[P2] Evaluate the abort threshold independently:** `src/logging/diagnostic.ts:676`
  - body: The new abort check is inside the `ageMs > stuckSessionWarnMs` branch, so a valid abort threshold below the warning threshold cannot fire at the configured time. Either enforce `abort >= warn` in config resolution or evaluate abort timing separately from warning emission.
  - confidence: 0.87
