## Summary

Current main still has the stuck-session recovery gap, but this PR duplicates the existing open recovery track in #73243/#71127/#73510. The older #73243 PR is the stronger canonical implementation candidate: it predates this PR, explicitly closes #71127, and wires recovery through reply/embedded-run aborts, queue cleanup, and a typed `session.aborted` event. This PR's smaller diff only clears queued lane entries and resets diagnostic state, so it would likely not release an active stuck session and appears to miss the diagnostic event type update.

## Best Possible Solution

Close this PR as superseded by #73243 and continue the stuck-session recovery work there, with #71127 and #73510 left open until a recovery implementation lands. The canonical fix should abort active reply/embedded runs, clear queued work, emit a typed `session.aborted` diagnostic event, update config/docs/schema surfaces, and include regression coverage that proves a stuck active session lane is actually released.
