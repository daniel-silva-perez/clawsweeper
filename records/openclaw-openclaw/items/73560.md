## Summary

Keep this PR open. Current main still only warns on stuck processing sessions and does not implement `diagnostics.stuckSessionAbortMs` or automatic recovery. This PR is a plausible implementation candidate, but it needs review/fixes: the patch appears to emit an untyped `session.aborted` event, relies on `clearCommandLane()` which only rejects queued entries and does not clear the active stuck task, and overlaps with the more complete open #73243 plus a likely paired open issue by the same author (#70133).

## Best Possible Solution

Keep this PR open for maintainer review, but do not land it as-is. The best path is to choose one canonical stuck-session recovery implementation, likely by reviewing #73243 against #71127 and the same-author Feishu queue report #70133, then require a fix that aborts the active run or safely resets the lane, types/documents `session.aborted`, preserves threshold semantics, and adds regression tests for the recovery path.

## Risks / Open Questions

- Functional: the PR's recovery path may not release the active stuck lane if it only clears pending queue entries without aborting or generation-resetting the active task.

- Behavioral: aborting sessions from a diagnostics heartbeat is operationally sensitive and needs focused tests proving queued work, active run abortion, repeated heartbeat behavior, and threshold ordering.
