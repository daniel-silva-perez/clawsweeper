## Summary

Keep open. Current main still lacks the requested configurable abort threshold, and the PR is not safe to merge as-is because its abort path only clears queued entries while leaving the active lane slot blocked; it also makes abort timing depend on the warning threshold. The useful direction should be reworked against the existing stuck-session recovery runtime.

## What This Changes

The PR adds `diagnostics.stuckSessionAbortMs` config/schema/help/reload handling plus heartbeat logic and tests to clear a stuck session lane queue, reset diagnostic state, log, and emit `session.aborted` after a configurable threshold.

## Best Possible Solution

Implement the abort threshold as a policy gate on the existing stuck-session recovery runtime: resolve abort timing independently, pass an explicit active-abort signal only after that threshold, release lanes via the established abort/drain/reset path, and align generated config docs plus heartbeat/recovery integration tests.

## Reproduction Assessment

Yes. The queue contract is reproducible from current code and tests: one active task plus one queued task leaves `clearCommandLane` rejecting only the queued entry, while `resetCommandLane` is the path that releases the active slot; the threshold bug is reproducible by setting an abort threshold below the warn threshold and advancing the heartbeat before the warn threshold.

## Solution Assessment

No. The PR's direct heartbeat queue clearing bypasses the current recovery runtime and does not release active lane work; the safer solution is to extend `recoverStuckDiagnosticSession` with explicit abort-threshold semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Route aborts through the lane reset path:** `src/logging/diagnostic.ts:678-680`
  - body: `clearCommandLane` only rejects queued entries and leaves the active task slot occupied. With `maxConcurrent=1`, a stuck active task can continue to block the session lane after diagnostics marks the session idle, so later messages may remain wedged while recovery is reported as complete.
  - confidence: 0.91
- **[P2] Evaluate the abort threshold independently:** `src/logging/diagnostic.ts:676`
  - body: The abort check is nested under `ageMs > stuckSessionWarnMs`, so a valid configured abort threshold below the warning threshold cannot fire at the configured time. Either enforce the ordering during config resolution or check abort timing separately from warning emission.
  - confidence: 0.87
