## Summary

Keep open. Current main still handles Discord inbound text through Carbon `MessageCreateListener` only; it does not register `MessageUpdateListener`, expose `handleEdits` or `editDebounceMs` config/docs, or make Discord inbound replay edit-aware. The April 26 maintainer-side review remains accurate against current main `9bb1e59447a5fdf2c0164672df6c748188b26442`.

## Best Possible Solution

Keep this issue open as the canonical Discord edit-to-reprocess tracker. Implement the feature in the bundled Discord plugin with an opt-in Discord config/type/schema/docs surface, Carbon `MessageUpdateListener` registration, content-change filtering, per-message debounce, edit-aware replay semantics, and focused Discord monitor tests. Carry forward #48894's add-mention-on-edit acceptance criteria, treat #38505 as overlapping, and keep #53654's delete-to-cancel behavior separate unless maintainers explicitly choose to include it.

## Risks / Open Questions

- A naive edit handler could retrigger on embed-only updates, bot-authored edits, or rapid edit streams; the fix should filter content changes and debounce per message id.
- The current replay guard intentionally suppresses duplicate same-id deliveries, so edit support must add explicit edit-aware semantics without reintroducing duplicate create-message processing.
- Mention-gated guild behavior needs tests so edits that add a bot mention are accepted while ordinary unmentioned edits remain ignored when `requireMention` is true.
