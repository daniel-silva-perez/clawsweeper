## Summary

Keep open. Current main still leaves the core stale-hashed-chunk safety gap unresolved: the Gateway `update.run` path can invoke the package updater from the live Gateway process, the staged package swap replaces the active global package tree before restart scheduling, and release validation still treats the same self-swapped `ERR_MODULE_NOT_FOUND` class as acceptable after a nominally successful package update.

## What This Changes

The issue asks OpenClaw to make package auto-update safe when hashed `dist` chunks change so Telegram inbound and other runtime paths do not keep resolving stale chunk filenames until a manual restart.

## Best Possible Solution

Add a generic package-update safety boundary: Gateway/auto-update should delegate package tree replacement to a fresh external updater before mutating the active install, or fail closed before mutation when safe handoff is unavailable. Release validation should stop accepting stale self-swapped `ERR_MODULE_NOT_FOUND` as success unless the implementation guarantees the old process exits without loading more code.

## Reproduction Assessment

Yes. The issue and follow-up comment provide exact versions, npm-global install method, missing hashed chunk names, timestamps, and restart recovery; source inspection also shows a current path where package replacement can happen before the running process exits.

## Solution Assessment

No. Current main has useful partial hardening for shell-driven package updates, but the best maintainable fix is a generic update handoff or fail-closed boundary rather than Telegram-specific recovery or treating the stale import as an acceptable release-check outcome.
