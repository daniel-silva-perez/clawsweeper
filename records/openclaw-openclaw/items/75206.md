## Summary

Keep open. Current main has useful partial hardening for package updates, including staged npm installs, fresh-process CLI post-update work, a teardown guard for stale memory-runtime imports, and release tarball import-graph checks. It still does not prove the safety property reported here: the Gateway `update.run` path can call the package update runner from the live Gateway process, the package runner can swap the global package before the restart is scheduled, and the installed-package verifier still checks inventory membership rather than reusing the dist import-graph validation at install time.

## What This Changes

The issue requests that auto-update and supported package update flows avoid leaving a running Gateway or Telegram inbound path with old hashed ESM chunk references after the installed dist tree changes.

## Best Possible Solution

Add a generic package-update safety boundary: package-manager updates initiated from a running Gateway or auto-update should either hand off the swap to an external updater/fresh process before mutating the active dist tree, or fail closed when a safe restart cannot be guaranteed. Reuse the dist import-graph checker for staged installed packages where practical, and add focused regression coverage for stale conversation-runtime/inbound and CLI memory-state chunk classes.

## Reproduction Assessment

Yes. The report gives versions, install method, channel, concrete missing hashed chunk paths, timestamps, and restart recovery; current source also shows a plausible supported path where the live Gateway can mutate the package root before restart. The exact host sequence was not replayed because this was a read-only review.

## Solution Assessment

No. Current main contains partial mitigations, but the narrowest maintainable fix is to remove or fail closed the live-process package-swap path and verify staged dist import reachability before promotion; Telegram-specific recovery should be secondary to that generic update boundary.
