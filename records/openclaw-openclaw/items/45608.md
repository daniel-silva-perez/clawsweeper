## Summary

Keep open. Current main has reset-time preservation mitigations, including transcript archival and startup loading for slugged session-memory files, but it still does not run an outgoing agentic memory-flush turn before manual or stale session reset replaces the session.

## What This Changes

The issue requests running the same curated agentic memory flush used around compaction before `/new`, `/reset`, daily reset, and related session-reset teardown paths.

## Best Possible Solution

Keep this as the canonical maintainer design thread for reset-time curated memory persistence. The end state should be a bounded shared reset-flush path that can run the existing memory capability against the outgoing session before teardown when policy allows, while keeping session-memory as a raw audit fallback.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists by following `/new` or `/reset` through `initSessionState` and `maybeHandleResetCommand`, or stale daily/idle rollover through `evaluateSessionFreshness`; the outgoing session is replaced or archived without a call to `runMemoryFlushIfNeeded`.

## Solution Assessment

Unclear because there is no patch yet. Reusing the existing memory-flush primitive is plausible, but maintainers still need to choose blocking versus background execution, timeout/failure policy, config shape, and default coverage for daily and idle resets.
