## Summary

Keep this issue open. Current main has useful adjacent work: manual `/new` and `/reset` emit reset hooks, the bundled `session-memory` hook writes a mechanical transcript snapshot, stale daily resets now archive the old transcript, and reset/startup turns can preload recent daily memory. But the requested core behavior is still not implemented: there is no agentic pre-reset memory flush wired into `/new`, `/reset`, daily reset, or idle reset before the old session is destroyed, and daily/idle resets still do not emit the command hook path.

## Best Possible Solution

Keep the issue open and implement a core reset-teardown persistence path. The best direction is to run a bounded agentic memory flush against the previous session before explicit `/new`/`/reset` teardown and before stale daily/idle rollover, while preserving the current `session-memory` raw snapshot as a fallback. A low-risk companion path would copy or inject the last existing compaction summary across reset when present, since that covers the documented production case without another LLM call.

## Risks / Open Questions

- A pre-reset agentic turn would add latency and model/tool side effects to reset paths, so it needs explicit timeout, dedupe, empty-session, CLI, sandbox/read-only, and failure semantics.
- Daily and idle reset behavior is core session lifecycle behavior, not a ClawHub-only extension, because the old session is selected and destroyed inside core before external hooks can reliably run an agentic workspace turn.
