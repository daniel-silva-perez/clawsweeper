## Summary

Keep open. Current main has reset-time preservation mitigations, including session transcript archival and startup loading of slugged session-memory files, but it still does not run an agentic memory-flush turn against the outgoing session before manual `/new` or `/reset`, daily rollover, or idle rollover replaces that session.

## What This Changes

Request to run the existing silent agentic memory-flush mechanism before manual `/new`, `/reset`, daily reset, and idle reset destroy the outgoing session.

## Best Possible Solution

Keep this as the canonical maintainer design thread for reset-time curated memory persistence. The desired end state is a bounded shared reset-flush path that can run the existing memory capability against `previousSessionEntry` before teardown when policy allows, while keeping `session-memory` as raw audit fallback and documenting manual, daily, and idle behavior together.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: follow bare `/new` or `/reset` through `initSessionState` and `maybeHandleResetCommand`, or daily/idle rollover through `evaluateSessionFreshness`, and the outgoing session is replaced or archived without an agentic memory-flush call.

## Solution Assessment

Unclear because there is no patch yet. The requested direction is plausible and reuses the right primitive, but maintainers still need to choose blocking versus background execution, timeout/failure policy, config shape, and default coverage for daily and idle resets.

## Risks / Open Questions

- Reset-time agentic flush semantics need product/runtime decisions because blocking `/new` can add latency, while background flushing risks racing transcript archive and new-session startup context.
