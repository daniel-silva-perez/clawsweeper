## Summary

Keep open. Current main still has reset-time mitigations, but it does not run an agentic memory-flush turn against the outgoing session before manual `/new`/`/reset`, daily rollover, or idle rollover destroys the prior session. The remaining work is product-sensitive because several open PRs cover overlapping partial approaches, and maintainers should decide the reset semantics before another automated fix branch starts.

## What This Changes

Request to run a bounded silent agentic memory flush before manual and automatic session resets so durable memory can be curated before the old session is discarded.

## Best Possible Solution

Keep this as the canonical product/design thread for reset-time memory persistence. The maintainable end state is a shared, bounded reset-flush path that runs the existing memory capability against the outgoing session before teardown where appropriate, preserves `session-memory` as a raw fallback, and defines daily/idle rollover behavior consistently with startup context and plugin hooks.

## Reproduction Assessment

Yes. A source-level reproduction is clear on current main: follow manual `/new` or stale daily/idle rollover through `initSessionState` and command handling, and the old session is rotated/archived without any `runMemoryFlushIfNeeded` or equivalent agentic turn.

## Solution Assessment

Unclear because there is no proposed patch in this item. The requested direction is plausible, but the best implementation should be maintainer-reviewed and probably use a dedicated reset-flush helper rather than treating `session-memory` hook synthesis or startup-context preload as equivalent.
