## Summary

Keep open. Current main has reset-time mitigations, including archived transcripts, command hooks, slugged session-memory artifacts, and first-turn startup-context loading, but it still does not run an agentic memory-flush turn against the outgoing session before manual `/new`/`/reset`, daily rollover, or idle rollover destroys the prior session.

## What This Changes

The issue requests a silent agentic memory flush before `/new`, `/reset`, daily reset, and related reset rollovers, using the existing pre-compaction memory-flush mechanism rather than only mechanical session-memory dumps.

## Best Possible Solution

Keep this as the canonical maintainer design thread for reset-time memory persistence. The desired end state is a bounded, shared reset-flush path that can run the existing memory capability against `previousSessionEntry` before teardown where appropriate, preserves `session-memory` as an audit/raw fallback, and defines manual, daily, and idle rollover behavior consistently in code, tests, config help, and docs.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: follow bare `/new` or `/reset` through `initSessionState` and `maybeHandleResetCommand`, or stale daily/idle rollover through `evaluateSessionFreshness`, and the old session is rotated/archived without invoking `runMemoryFlushIfNeeded` against the outgoing transcript.

## Solution Assessment

Unclear because this issue is a feature/design request rather than a patch. The requested direction is plausible, but the safest implementation likely needs a dedicated reset-flush helper and maintainer decisions on latency, config shape, daily/idle policy, and overlap with existing hook/startup-context mitigations.
