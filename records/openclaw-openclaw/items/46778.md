## Summary

Keep open. Current main still keys Feishu inbound duplicate suppression to the platform `message_id`, and the latest release has the same id-only behavior, so the reported same-logical-message/different-id path remains a valid Feishu plugin bug rather than a stale or already-implemented item.

## What This Changes

The issue asks OpenClaw to suppress duplicate Feishu private-chat deliveries when the same logical message arrives with different `message_id` values.

## Best Possible Solution

Add a Feishu-local duplicate fingerprint that complements the existing `message_id` fast path, with focused tests for same-content/different-id duplicates and negative cases for distinct content, timestamp, or context, while leaving core channel/runtime contracts unchanged.

## Reproduction Assessment

Yes. A focused unit reproduction can enqueue two Feishu text events with the same account, sender, chat/thread, `create_time`, and content but different `message_id` values; source inspection shows both ids are treated as fresh today.

## Solution Assessment

Yes. A Feishu-owned fallback fingerprint is the narrowest maintainable solution because the observed failure is service-specific delivery behavior and the plugin already owns inbound event parsing, debounce, and persistent dedupe calls.

## Repair Work Prompt

Implement a focused fix for https://github.com/openclaw/openclaw/issues/46778. Current Feishu inbound dedupe is keyed only by `message_id`; when Feishu delivers the same logical private-chat text message with different ids, OpenClaw processes both. Add a Feishu-local fallback duplicate fingerprint that complements the existing id path. Keep the boundary inside `extensions/feishu`: do not change core channel/runtime contracts and do not add cross-channel policy. The fingerprint should be conservative enough to avoid false positives: include account/namespace, chat/thread or root context, sender open/user id, message type, normalized content hash, and `create_time` or a very tight time bucket when exact create time is absent. Apply it before duplicate logical events can reach dispatch, while preserving current same-`message_id` behavior and release-on-error behavior. Add regression tests for two p2p `im.message.receive_v1` text events with the same sender/chat/content/create_time but different `message_id` values dispatching once, and tests that different content or timestamp/context still dispatch. Update `CHANGELOG.md` with a single-line user-facing fix entry crediting `@kingcuty`. Do not broaden Feishu access policy, broadcast routing, streaming-card output, or core Plugin SDK behavior.
