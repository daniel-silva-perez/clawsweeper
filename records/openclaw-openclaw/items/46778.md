## Summary

Current main still deduplicates Feishu inbound messages by `message_id`, with batching/queueing around that key but no fallback using sender, timestamp, and content hash. The reported failure mode can still pass through when Feishu emits the same DM content with different message IDs, so this should remain open for an extension-owned fix.

## Best Possible Solution

Keep this issue open and fix it in the Feishu plugin. Add a secondary inbound dedupe key for text DMs, likely scoped by account, chat, sender, normalized content hash, and a narrow Feishu timestamp/window, while preserving the existing `message_id` replay protection. Add a regression test where two Feishu private-message events have identical sender/content/timestamp context but different `message_id` values and only one agent dispatch occurs.
