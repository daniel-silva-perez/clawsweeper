## Summary

Keep open. Current main still gates Feishu inbound delivery on `message_id`, so the reported same-message/different-`message_id` duplicate path remains plausible and narrow enough to fix in the Feishu plugin.

## What This Changes

Add a Feishu inbound fallback dedupe path for duplicate private-message deliveries where the same sender/content/timestamp arrives with different `message_id` values.

## Best Possible Solution

Implement the fallback inside the Feishu plugin, preserving the existing `message_id` fast path and suppressing only tightly matched duplicate deliveries with the same sender, chat/thread scope, normalized content, and either matching `create_time` or a very narrow receive window.

## Reproduction Assessment

Yes. A focused unit reproduction can enqueue two `im.message.receive_v1` text events with the same sender/chat/content/create_time but different `message_id` values; current main treats both ids as fresh.

## Solution Assessment

Yes. A Feishu-owned fallback fingerprint is the narrowest maintainable fix because the failure is service-specific delivery behavior, while the core plugin/runtime dedupe contracts already support owner-local handling.
