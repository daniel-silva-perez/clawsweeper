## Summary

Keep open. Current main still gates Feishu inbound delivery, debounce freshness, and final handler suppression on `message_id` only, so the reported same-message/different-`message_id` duplicate path remains valid and Feishu-owned.

## What This Changes

The issue requests a Feishu inbound dedupe fallback for duplicate private-chat messages that arrive with different `message_id` values.

## Best Possible Solution

Add the fallback inside the Feishu plugin while preserving the existing `message_id` fast path; the end state should suppress only tightly matched duplicate deliveries and leave core plugin/runtime dedupe contracts unchanged.

## Reproduction Assessment

Yes. A focused unit reproduction can enqueue two Feishu `im.message.receive_v1` text events with the same sender, chat/thread, content, and `create_time` but different `message_id` values; current main treats both ids as fresh.

## Solution Assessment

Yes. A Feishu-owned fallback fingerprint is the narrowest maintainable solution because the failure is service-specific delivery behavior and current core/plugin seams already allow owner-local handling.
