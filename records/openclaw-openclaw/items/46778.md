## Summary

Current main still deduplicates Feishu inbound messages by `message_id` only. The reported same-sender/same-content delivery with different `message_id` values remains a valid, narrow Feishu plugin bug and should stay open as a focused fix candidate.

## Best Possible Solution

Keep this issue open and implement a Feishu-owned fallback dedupe fingerprint for `im.message.receive_v1` deliveries that share sender, chat/thread context, create_time or a narrow arrival window, message type, and normalized content while carrying different `message_id` values. Preserve existing `message_id` dedupe and add regression coverage for both duplicate suppression and legitimate repeated messages.
