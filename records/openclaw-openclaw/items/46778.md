## Summary

Current main still deduplicates Feishu inbound messages by message_id only. The reported case is same sender/content/timestamp arriving with different message_id values, so this remains a valid, narrow Feishu-plugin bug and is a good focused fix-PR candidate.

## Best Possible Solution

Keep this issue open and implement a Feishu-owned fallback dedupe fingerprint for im.message.receive_v1 deliveries that have the same sender/chat/thread, create_time or narrow arrival window, and normalized content but different message_id values. Preserve the existing message_id dedupe path, keep the change inside extensions/feishu unless reusing an existing generic helper, and add regression coverage for same-content/different-id suppression plus a legitimate repeated-message case.
