## Summary

Keep open. Current main still has the reported auth-profile shadowing behavior: stored auth profiles are selected before env/config credentials in normal provider resolution, `models status` reports profiles as effective when any profile exists, and there is no bulk auth sync command or config-set warning. The related reconciliation PR was closed unmerged, so it is useful context but not an implemented fix.

## Best Possible Solution

Keep this issue open and implement a focused rotation path. The best product direction is probably an explicit `openclaw models auth sync`/`openclaw secrets` workflow that can update or migrate all configured agent auth profiles for a provider, preferably to a shared SecretRef/keyRef rather than duplicating plaintext, plus a targeted warning in `config set` or `models status` when auth profiles shadow a newly configured provider env/API key. The closed #57371 approach is useful prior art, but it should be reviewed against current auth precedence, all-agent store discovery, SecretRef semantics, and tests before landing.

## Risks / Open Questions

- Closing this now would leave users with the same multi-agent API key rotation gap: audit can detect shadowing, but routine `config set` or `models status` flows still do not guide or repair it.
- A direct plaintext auto-sync could spread rotated keys into every agent store; a safer implementation may need SecretRef/keyRef migration or an explicit sync command with clear scope and confirmation.
