## Summary

Keep this PR open because current main still lacks the requested `precedingText` and `messageId` fields, and the earlier feature issue was closed as superseded by this PR rather than implemented. The proposed branch is useful but should not merge as-is because it exposes assistant transcript text to ordinary `before_tool_call` hooks without the existing conversation-access opt-in, and it predates current main's hook-contract split.

## What This Changes

The PR adds optional `precedingText` and `messageId` to `before_tool_call`, derives them from the assistant tool-call message via `extensionContext`, forwards that context through wrappers, and adds tests and plugin docs.

## Best Possible Solution

Rework this into an additive hook-context update that preserves current main's hook contract files and client-tool behavior, gates assistant text/message IDs behind an explicit conversation-access policy or equivalent opt-in, and keeps the narrow tests for wrapped, abort-wrapped, and client-tool paths.

## Reproduction Assessment

Not applicable as a feature PR, but current-main verification is high confidence: source and docs inspection show the hook event and runtime payload still omit `precedingText` and `messageId`. The PR's own tests describe the desired assistant-message context path.

## Solution Assessment

No, not as currently proposed. The narrow API direction matches #12617, but the implementation should first resolve the conversation-access gate and rebase onto the current hook-type/client-tool code before it is a maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Gate assistant text behind conversation access:** `src/agents/pi-tools.before-tool-call.ts:280-281`
  - body: This adds `precedingText`/`messageId` to every `before_tool_call` event, but non-bundled `before_tool_call` hooks are not covered by `allowConversationAccess`. That means an existing third-party tool hook could start receiving assistant transcript text without the explicit operator opt-in required for other conversation hooks. Please gate these fields or add an opt-in contract before exposing them.
  - confidence: 0.91
- **[P2] Only return messageId after matching the tool call:** `src/agents/pi-tools.before-tool-call.ts:158-159`
  - body: `extractBeforeToolCallMessageContext` returns the leaf `messageId` even when the requested `toolCallId` is not found in that assistant message. In a stale or mismatched leaf, audit plugins would receive a message id that did not request the tool call. Track whether the target tool call was found and only set `messageId` for the matched message.
  - confidence: 0.82
