## Summary

Keep open. Current main still has a reachable Android/Termux sharp-unavailable path for image inputs that need optimization, resize, HEIC conversion, or PNG alpha resizing; the newer header-metadata fast path only avoids sharp for already-safe common images. The two existing ClawSweeper comments correctly identified a narrow graceful-fallback repair while leaving Termux-native rebuild automation as broader packaging work.

## What This Changes

The issue asks OpenClaw to handle Android/Termux sharp native-module failures after update, either by rebuilding with the Termux toolchain, adding a native-module repair command, or falling back to raw image input when sharp cannot load.

## Best Possible Solution

Add a bounded backend-unavailable fallback in shared image ingestion and sanitization: preserve original image bytes only when normalized MIME, byte size, and header-derived dimensions prove the input is already safe for provider submission; otherwise omit the image or return a clear backend-unavailable error. Keep Termux-specific native rebuild/update automation as a separate packaging decision.

## Reproduction Assessment

Yes for the current-main code path: make the bundled sharp import fail, then process a non-GIF web/local image with default optimization, a resize-required tool image, HEIC input, or PNG alpha resizing. The exact Android/Termux device failure is not locally reproduced here, but the reporter's error matches the reachable dependency-load path.

## Solution Assessment

Yes for the narrow graceful fallback: it is the smallest maintainable repair that keeps image handling useful without taking on platform-specific native rebuild policy. The Termux rebuild/update command is broader and should remain a separate maintainer-scoped packaging decision.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/47441 with a narrow graceful fallback for sharp-unavailable image handling. Current main can still fail on Android/Termux when `loadWebMedia` optimizes non-GIF images by default, when tool image blocks need resize/recompression, when HEIC conversion is needed, or when PNG alpha resizing reaches the sharp-backed media-understanding plugin. Add a bounded fallback in shared image ingestion/sanitization: if the sharp-backed backend is unavailable, preserve/pass through the original image only when normalized MIME is an allowed image type, byte length is within the existing cap, and header-derived dimensions prove it is within the existing pixel/dimension limits; otherwise omit the image or surface a clear backend-unavailable error. Do not add Termux-specific package-manager scripts, do not run node-gyp/NDK repair from `openclaw update`, do not change sharp dependency pins or lifecycle-script policy, and do not add Android-specific core IDs unless a generic helper needs platform context. Update focused tests for safe pass-through, unsafe omission/error, resize-required failure, web-media optimization fallback, and HEIC/no-header behavior. Add a changelog fix entry only if the final behavior is user-facing.
