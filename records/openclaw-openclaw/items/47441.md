## Summary

Keep this issue open. Current main still routes oversized image sanitization through sharp-backed media-understanding image ops, sharp 0.34.5 still has no android-arm64 optional runtime package, and the updater/doctor paths do not detect Termux or rebuild sharp with a Termux-native toolchain. The safest narrow repair is still a bounded raw-image fallback when the image backend is unavailable, not closing this as stale.

## What This Changes

The issue requests Android/Termux handling for sharp native image processing failures after OpenClaw updates, including a Termux rebuild utility or graceful raw-image fallback.

## Best Possible Solution

Keep this open and implement a focused image-path fallback first: when sharp-backed resizing cannot load on unsupported platforms, preserve the original canonical image block only if it is a supported image MIME and remains within the existing byte/pixel safety envelope; otherwise keep omitting unsafe images with a clearer reason. Leave a Termux-native rebuild utility or broader update repair policy as a separate maintainer decision.
