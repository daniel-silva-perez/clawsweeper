## Summary

Keep open. Current main and the latest release still have reachable sharp-unavailable image paths on Android/Termux: sharp remains the media-understanding image backend, Android sharp/libvips packages are absent from the lockfile, and resize/HEIC/PNG/web-media flows still call the backend when header-only fast paths are insufficient.

## What This Changes

The issue requests Android/Termux handling for sharp native-module failures after updates, with possible post-install rebuild tooling, a repair command, or a graceful raw-image fallback.

## Best Possible Solution

Add a bounded backend-unavailable fallback in shared image ingestion and optimization: preserve the original only when MIME type, byte size, and header-derived dimensions prove the image is already safe for provider limits; otherwise omit it or return a clear backend-unavailable error. Keep automatic Termux native rebuild/update tooling as a separate packaging decision.

## Reproduction Assessment

Yes for the current-main application path: force the bundled sharp import/load to fail, then process a non-GIF web/local image with default optimization, a resize-required image block, a HEIC input, or a PNG alpha resize path. The exact Android/Termux device setup was not live-tested here.

## Solution Assessment

Yes for a narrow graceful fallback: it is the smallest maintainable repair for the user-visible image failure without adding platform-specific native rebuild automation. A post-update Termux rebuild or fix-native-modules command may still be useful later, but it is broader packaging work.
