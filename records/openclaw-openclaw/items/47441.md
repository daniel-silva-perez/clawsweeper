## Summary

Keep open. Current main still has a reachable Android/Termux sharp native-module failure path: sharp remains the shipped image-processing backend, the lockfile still lacks Android sharp/libvips optional packages, and the update/doctor path stages runtime dependencies generically rather than repairing Termux toolchains. The newer header-metadata fast path narrows the blast radius for already-safe common image blocks, but it does not cover web-media optimization, resize-required images, HEIC conversion, or post-update native dependency repair.

## What This Changes

The issue requests Android/Termux handling for sharp native-module failures after updates, including rebuild tooling or a graceful image-processing fallback.

## Best Possible Solution

Add a bounded fallback in the shared image loading/sanitization surface: when the sharp-backed backend is unavailable, preserve the original only for image inputs whose MIME, byte size, and header-derived dimensions prove they are already safe; otherwise return a clear backend-unavailable omission/error. Keep Termux-specific native rebuild automation as a separate maintainer-scoped packaging decision.
