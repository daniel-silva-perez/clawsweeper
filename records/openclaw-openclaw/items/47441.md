## Summary

Keep open. Current main still has a reachable sharp-unavailable path on Android/Termux: safe common image headers can bypass sharp, but resizing, HEIC conversion, PNG alpha optimization, and default web-media optimization still load the sharp-backed media plugin, and runtime dependency repair does not provide a Termux-native sharp rebuild path.

## What This Changes

The issue requests Android/Termux handling for sharp native-module failures after OpenClaw updates, with suggested fixes including Termux rebuild support, a native-module repair command, or graceful raw-image fallback.

## Best Possible Solution

Add a bounded backend-unavailable fallback in shared image ingestion and sanitization: preserve the original image only when MIME, byte size, and header-derived dimensions prove it is already safe; otherwise omit the image or return a clear backend-unavailable error. Keep Termux-specific native rebuild automation as a separate packaging decision.
