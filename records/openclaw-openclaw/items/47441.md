## Summary

Keep this issue open. Current main still routes media attachment image operations through the bundled media-understanding plugin's `sharp` import, the lockfile still has only Sharp's linux/darwin/win32/wasm optional runtime packages rather than `android-arm64`, and `openclaw update` only has a Windows-specific install-script retry path. The report also includes a concrete Termux workaround and links the prior Android native-module issue #20349, so this is specific enough to remain actionable.

## Best Possible Solution

Keep the issue open. The most concrete path is a media-understanding/update fix owned by the sharp image-processing surface: detect Android/Termux, either use a supported non-sharp image backend or provide a documented `openclaw doctor --fix`/native-module repair that rebuilds sharp with the Termux toolchain, and add targeted tests for platform/env selection plus image-sanitization behavior. If Termux Gateway is intentionally unsupported, update the Android/install docs explicitly and close only after that product decision is made.
