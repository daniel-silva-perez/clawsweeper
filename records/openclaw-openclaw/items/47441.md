## Summary

Keep open. Current main still leaves the Android/Termux native-module problem unresolved: sharp remains in the media-understanding runtime, sharp 0.34.5 still has no android-arm64 optional package in the lockfile, resize-required image paths still call the sharp-backed media ops, and update/doctor only surface a generic native build hint rather than a Termux rebuild/fix path. Header-based metadata helps already-safe common images avoid sharp, but it does not close the reported update/native dependency failure.

## What This Changes

The issue asks OpenClaw to handle sharp image processing failures on Android/Termux after updates, either by rebuilding with the Termux toolchain, adding a native-module repair command, or falling back gracefully when sharp cannot load.

## Best Possible Solution

Implement a bounded image-path fallback first: when the sharp-backed backend is unavailable, preserve the canonical original image only when OpenClaw can prove the MIME, byte size, and dimensions are already safe; otherwise omit it with a clearer backend-unavailable reason. Treat a Termux-native rebuild utility or update-time native-module repair as a separate maintainer product decision.
