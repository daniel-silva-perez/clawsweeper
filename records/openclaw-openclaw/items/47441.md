## Summary

Keep open. Current main still has a reachable sharp-unavailable path for Android/Termux: the media-understanding runtime depends on sharp, sharp has no android-arm64 prebuilt in the locked dependency set, and image resize/optimization/HEIC paths still call the sharp-backed backend when sips is not selected. Prior ClawSweeper comments already converged on the same narrow fallback repair while keeping Termux-native rebuild automation separate.

## What This Changes

Make OpenClaw image handling or update repair resilient on Android/Termux when sharp lacks an android-arm64 prebuilt and source rebuild assumes an unavailable NDK cross-compiler.

## Best Possible Solution

Add a bounded backend-unavailable fallback in shared image ingestion/sanitization: when the sharp-backed backend cannot load, preserve the original only for normalized image MIME types whose byte size and header-derived dimensions are already within provider limits; otherwise omit the image or return a clear backend-unavailable error. Leave Termux-specific native rebuild tooling as a separate packaging decision.

## Reproduction Assessment

Yes for the current-main application path: make the bundled sharp import fail, then process a non-GIF web/local image that defaults to optimization, an image block that exceeds byte/dimension limits, HEIC input, or PNG alpha resizing. The exact Android/Termux device setup was not live-tested here.

## Solution Assessment

Yes for the narrow graceful fallback: it is the smallest maintainable fix for the user-visible image failure without taking on platform-specific native toolchain repair. A Termux rebuild command could still be considered later, but it should not block the shared image fallback.
