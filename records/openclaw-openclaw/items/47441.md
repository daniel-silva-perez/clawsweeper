## Summary

Keep open. Current main still has a reachable Android/Termux sharp native-module failure path: sharp remains the shipped media-understanding image backend, the lockfile still has no android-arm64 sharp/libvips package, and runtime-dependency repair installs generic packages with scripts disabled rather than doing a Termux-native rebuild. The header-metadata fast path preserves already-safe common images, but resize-required images, HEIC conversion, PNG alpha optimization, and web-media optimization still depend on the sharp-backed backend.

## What This Changes

The issue requests Android/Termux handling for sharp native-module failures after updates, either through Termux-aware rebuild/update repair, a fix-native-modules utility, or graceful fallback for image processing.

## Best Possible Solution

Implement a bounded backend-unavailable fallback in shared image loading and sanitization: preserve originals only when MIME, byte size, and header-derived dimensions prove they are already safe; otherwise produce a clear backend-unavailable omission/error. Treat Termux-native rebuild automation as a separate maintainer-scoped packaging design.
