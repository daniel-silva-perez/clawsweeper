## Summary

Keep open. Current main still has a reachable Android/Termux sharp failure path: sharp remains the shipped media-understanding image backend, the lockfile still has no Android sharp/libvips optional package, runtime dependency staging uses generic package-manager installs, and resize/optimization paths still load the sharp-backed backend when an image is not already safely pass-throughable.

## What This Changes

The issue requests a durable fix for Android/Termux sharp image-module failures after OpenClaw updates, including rebuild automation, a repair command, or graceful fallback to raw image inputs.

## Best Possible Solution

Add a bounded backend-unavailable fallback in shared image loading and sanitization: preserve original image inputs only when MIME, byte size, and header-derived dimensions prove they are already safe; otherwise omit or return a clear backend-unavailable error. Keep Termux-specific native rebuild automation as a separate packaging decision.
