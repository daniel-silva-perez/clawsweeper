## Summary

Keep open. Current main still has a reachable Android/Termux sharp-unavailable path: the bundled media-understanding runtime depends on sharp, the lockfile has linux/darwin/windows sharp optional packages but no Android sharp/libvips package, and ordinary web/local image optimization, HEIC conversion, and PNG resizing still call the sharp-backed image-ops backend outside the macOS sips path.

## What This Changes

The issue requests Android/Termux handling for sharp native-module failures after OpenClaw updates, including either a Termux-aware rebuild path, a native-module repair command, or a graceful raw-image fallback.

## Best Possible Solution

Add a bounded backend-unavailable fallback in shared image ingestion/optimization: when the sharp-backed backend cannot load, preserve the original only for normalized image MIME types whose byte size and header-derived dimensions are already within provider limits; otherwise omit the image or return a clear backend-unavailable error. Leave automatic Termux native rebuild/update tooling as a separate packaging decision.

## Reproduction Assessment

Yes for the current-main application path: force the bundled sharp import/load to fail, then process a non-GIF web/local image with default optimization, a HEIC image, or a PNG alpha resize path. The exact Android/Termux device setup was not live-tested here.

## Solution Assessment

Yes for the narrow graceful fallback, which is the smallest maintainable repair for the user-visible image failure without taking on platform-specific native rebuilds. The suggested post-update rebuild or fix-native-modules command may still be useful later, but it should be a separate packaging feature.
