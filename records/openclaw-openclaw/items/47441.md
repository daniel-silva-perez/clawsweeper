## Summary

Keep open. Current main still has the reported Android/Termux native-module gap: sharp remains the active image-processing backend for resize/convert/optimization paths, the lockfile still has no android-arm64 sharp/libvips package, and update/doctor repair remains a generic npm/runtime-deps path rather than a Termux-aware fix. The newer header-metadata fast path can preserve already-safe common image blocks, but it does not cover image-tool/loadWebMedia optimization, resize-required images, HEIC conversion, or the post-update native dependency failure.

## What This Changes

The issue requests Android/Termux handling for sharp failing after updates because sharp has no android-arm64 prebuilt, suggesting update-time rebuild repair, a native-module fix command, or a graceful raw-image fallback.

## Best Possible Solution

Add a bounded image fallback in the shared image loading/sanitization path: when the sharp-backed backend is unavailable, keep the original only for images that OpenClaw can prove are safe by MIME, size, and dimensions; otherwise return a clear backend-unavailable omission/error. Leave Termux-specific rebuild automation for a separate maintainer-scoped packaging decision.
