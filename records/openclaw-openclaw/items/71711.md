## Summary

Keep open. Current main still lacks a recurring cron expiration contract: the typed job shape, gateway protocol schema, CLI flags, docs, and scheduler result handling only model one-shot deletion and recurring rescheduling, so the requested `activeUntil`/`expiresAt` behavior remains real work.

## What This Changes

Add an optional per-job expiration for recurring cron schedules so `--every` and `--cron` jobs stop after a configured timestamp and either remove or disable themselves.

## Best Possible Solution

Add a backwards-compatible cron job lifecycle field in the core scheduler contract, validate it only for recurring schedules, enforce it before due execution and startup catch-up, persist the requested remove/disable outcome, surface it through CLI/RPC/agent tool/UI/docs, and cover both expiration actions with focused cron service and protocol tests.

## Reproduction Assessment

Not applicable as a runtime bug, but there is a high-confidence absence check: current main has no accepted cron expiration field or CLI flag, and the scheduler only auto-deletes successful `at` jobs while recurring jobs reschedule.

## Solution Assessment

Unclear pending maintainer API choice. The proposed direction belongs in core because cron job lifecycle is owned by the scheduler and persisted job contract, but maintainers should choose the public field names and default expiration action before implementation.
