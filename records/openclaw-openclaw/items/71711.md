## Summary

Keep open. Current main does not implement per-job expiration for recurring cron jobs: the cron job schema, CLI, docs, and scheduler currently support one-shot auto-delete via `deleteAfterRun`, while recurring `every`/`cron` jobs recompute `nextRunAtMs` after runs. The same-author related RFC #71712 is also open and depends on this concept for agent-created schedules, so this should stay open for maintainer/API design rather than cleanup closure.

## Best Possible Solution

Keep this open and decide the public cron lifecycle contract first: canonical field name, timestamp parsing rules, `onExpire` default/action set, audit event shape, and how it interacts with agent-facing scheduling in #71712. Then implement additively across cron types/schema/validators, store normalization, scheduler pre-run or next-run handling, CLI add/edit/list/show, agent cron tool, Control UI, tests, and public docs.

## Risks / Open Questions

- The proposed timestamp semantics need product/API choice: current docs treat offset-less `--at` timestamps as UTC unless `--tz` is provided, while the issue proposes local-time interpretation for timezone-naive `activeUntil`.
- The `activeUntil` versus `expiresAt` name and `onExpire: remove|disable` default would become public Gateway protocol and CLI contract, so maintainers should settle the shape before implementation.
- Expiration removal/disable should be designed with run history, task/audit records, manual `cron run`, scheduler reloads, and the related agent-facing scheduling RFC #71712 in mind.
