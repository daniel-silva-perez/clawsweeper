## Summary

Keep open. Current main still lacks a recurring cron expiration contract, and the request remains valid core scheduler work rather than a ClawHub/plugin item or duplicate cleanup. It should get maintainer API judgment before implementation because it changes persisted cron schema, gateway RPC, CLI, agent tool, Control UI, docs, audit behavior, and overlaps the related agent-scheduling RFC.

## What This Changes

Requests an optional recurring cron job expiration timestamp and expiration action, including CLI/list/audit behavior, while rejecting it for one-shot jobs.

## Best Possible Solution

Add a backwards-compatible recurring-job lifecycle field in the core cron contract, validate it only for recurring schedules, enforce it before due execution and startup/maintenance recomputation, persist remove/disable outcomes, emit/audit expiration, and expose the same semantics through RPC, CLI, agent tools, Control UI, and docs with focused service/protocol/UI tests.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. The absence is high-confidence because current main has no accepted activeUntil/expiresAt/onExpire field and recurring jobs continue by computing future nextRunAtMs values.

## Solution Assessment

Unclear pending maintainer API choice. A core scheduler field is the right layer for persisted cron lifecycle, but maintainers should choose the public contract and #71712 alignment before an automated implementation starts.
