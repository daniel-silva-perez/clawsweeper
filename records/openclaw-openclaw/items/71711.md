## Summary

Keep open. Current main still lacks recurring cron expiration: cron jobs only expose schedule, enabled, and deleteAfterRun lifecycle fields, and the scheduler only removes successful one-shot `at` jobs. The linked agent-facing scheduling RFC #71712 is still open and affects the API/security shape, so this should remain a maintainer-reviewed cron lifecycle item rather than an auto-close or narrow repair job.

## What This Changes

Add optional per-job `activeUntil` or `expiresAt` and `onExpire` behavior for recurring `--every` and `--cron` jobs, including scheduler cleanup, CLI/listing, audit, persistence, and client-surface support.

## Best Possible Solution

Define an additive cron lifecycle contract first, then implement it consistently across cron types/schema/normalization, scheduler expiration handling, CLI add/edit/list/show, agent cron tooling, Control UI, Swift protocol/macOS editor models, focused regression tests, and public docs while aligning the agent-scheduling security design with #71712.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high-confidence: targeted source/docs/client searches plus cron contract, Gateway schema, scheduler, CLI, UI, and Swift inspection show no activeUntil/expiresAt/onExpire support for recurring jobs today.

## Solution Assessment

Unclear until maintainers settle the API contract. Core support is the right area because this changes stored cron semantics and protocol/client surfaces, but field names, timestamp semantics, default expiration action, audit behavior, and #71712 interaction need product/security review first.
