## Summary

Keep open. Current main still lacks a recurring cron expiration contract: persisted job types, gateway schemas, CLI flags, agent tool schema, Control UI state, docs, and scheduler result handling only model one-shot cleanup plus recurring rescheduling. This is valid core scheduler work, but the public field names, default expiration action, and interaction with related #71712 need maintainer API judgment before an automated fix PR should start.

## What This Changes

Request adding optional `activeUntil`/`expiresAt` and `onExpire` lifecycle behavior for recurring `--every` and `--cron` jobs, including CLI/list/docs/audit support.

## Best Possible Solution

Add a backwards-compatible recurring-job lifecycle field in the core cron contract, validate it only for recurring schedules, enforce it before due execution and maintenance/startup recomputation, persist the selected remove/disable outcome, emit/audit expiration, and expose the same semantics through RPC, CLI, agent tools, Control UI, and docs with focused service/protocol/UI tests.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. The absence check is high-confidence because current main has no accepted cron expiration field or CLI flag, and recurring jobs continue by computing future `nextRunAtMs` values.

## Solution Assessment

Unclear pending maintainer API choice. The proposed direction belongs in core because cron job lifecycle is owned by the scheduler and persisted job contract, but maintainers should settle naming, default expiration behavior, and #71712 alignment first.
