## Summary

Keep open. Current main still does not support per-job expiration for recurring cron schedules, and the related agent-facing scheduling RFC remains open with provenance, quota, and permanent-job semantics that should shape this contract before implementation.

## What This Changes

Add an optional `activeUntil` or `expiresAt` deadline and expiration action for recurring `--every` and `--cron` jobs.

## Best Possible Solution

Define an additive cron lifecycle contract first, including field name, timestamp parsing rules, default expiration action, permanent-job semantics, audit behavior, and #71712 interaction; then implement it consistently across cron storage/types, Gateway validation, scheduler expiration handling, CLI, agent tool, Control UI, macOS protocol/editor models, tests, and public docs.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high-confidence: targeted source, protocol, CLI, UI, Swift, docs, and related-discussion checks show no recurring `activeUntil`/`expiresAt`/`onExpire` support today.

## Solution Assessment

Unclear for implementation until maintainers choose the public contract. Core is the right area because this changes persisted cron semantics and client protocol surfaces, but the safer path is maintainer API/security design before a code PR.
