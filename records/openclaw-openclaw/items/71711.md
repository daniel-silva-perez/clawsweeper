## Summary

Keep open. Current main still lacks a recurring cron expiration contract, and the request is concrete core scheduler work rather than a plugin-only idea. Because it changes persisted job shape, Gateway RPC schema, CLI/UI/API semantics, default expiration behavior, and it is linked to the open agent-facing scheduling RFC, the next step should be maintainer contract review before implementation.

## What This Changes

Add an optional per-job `activeUntil` or `expiresAt` field and `onExpire` behavior for recurring `--every` and `--cron` jobs so expired recurring jobs stop running and can remove or disable themselves.

## Best Possible Solution

Define a backward-compatible recurring-job lifecycle contract in the core cron model, validate it only for recurring schedules, enforce expiry before due execution and during maintenance/startup recomputation, persist the remove/disable result, emit/audit the expiration, and expose the same semantics through Gateway RPC, CLI add/edit/list/show, agent tools, Control UI, and public docs with focused scheduler/protocol/CLI/UI tests.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. The absence is high-confidence because current main has no accepted cron expiration field or CLI/UI/tool surface, and recurring jobs continue by computing future `nextRunAtMs` values.

## Solution Assessment

Unclear pending maintainer API choice. The direction belongs in core because cron job lifecycle is owned by the scheduler and persisted job contract, but maintainers should settle naming, default expiration action, audit semantics, and #71712 alignment before a fix PR starts.

## Risks / Open Questions

- The linked #71712 agent-facing scheduling RFC may require stricter or different semantics, such as making expiration mandatory only for agent-created recurring jobs.
