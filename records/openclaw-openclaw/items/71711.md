## Summary

Keep open. Current main still does not implement recurring cron expiration: the cron contract, Gateway schema, scheduler lifecycle, CLI, agent tool, Control UI, Swift model, and docs still only model one-shot cleanup through deleteAfterRun. Related #71712 is still an open design dependency for agent-facing scheduling, so this should remain a maintainer-reviewed cron lifecycle/API item.

## What This Changes

Add optional activeUntil/expiresAt and onExpire behavior for recurring every/cron jobs so they stop, remove, or disable after an expiration timestamp.

## Best Possible Solution

Define an additive cron lifecycle contract first, then implement it consistently across cron types/schema/validators, normalization and persistence, scheduler expiration handling, CLI add/edit/list/show, agent cron tool, Control UI/macOS protocol models, focused tests, and public docs.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high-confidence: targeted field search plus schema, CLI, UI, docs, and scheduler inspection show recurring jobs have no activeUntil/expiresAt/onExpire path today.

## Solution Assessment

Unclear until maintainers settle the public API contract. Core support is the right product area because this changes stored cron semantics and protocol/client surfaces, but the exact field names, timezone behavior, expiration action, and #71712 interaction need design first.
