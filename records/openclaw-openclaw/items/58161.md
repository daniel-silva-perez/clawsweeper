## Summary

Keep this issue open. Current main still exposes embedded Pi bundle MCP tools through a direct listTools/callTool client path with no elicitation capability declaration or request handler, while the existing elicitation bridge is scoped to the separate Codex app-server runtime and does not cover embedded Pi bundle MCP sessions.

## What This Changes

Request support for MCP elicitation in the embedded Pi bundle MCP client so plugin-backed MCP tools can ask for dynamic user approval or input during tool execution.

## Best Possible Solution

Keep the issue open and design a scoped embedded Pi elicitation bridge: declare the MCP client capability only when the session can answer it, register an `elicitation/create` handler, route approval-shaped prompts through the existing plugin approval request/waitDecision UX, fail closed for unsupported or non-interactive cases, and document the supported modes and security limits.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is available: inspect current main and see that embedded Pi bundle MCP creates a plain SDK client, materializes tools, and calls `session.client.callTool(...)` without an elicitation handler or approval bridge.

## Solution Assessment

Unclear. The requested direction is likely the right product substrate, but the exact implementation should be reviewed by maintainers because supporting MCP form/URL elicitation affects approval UX, sensitive-data boundaries, and headless runtime behavior.
