## Summary

Keep open. Current main still always adds Docker `no-new-privileges` to sandbox containers, has no accepted `noNewPrivileges` config field or fallback path, and the linked implementation PR remains open and unmerged.

## What This Changes

The issue requests a compatibility path for Docker hosts where sandbox containers fail to start when `--security-opt no-new-privileges` is applied, such as an explicit opt-out, fallback retry, or clearer diagnostic.

## Best Possible Solution

Land or replace PR #70229 with a secure default-on `sandbox.docker.noNewPrivileges` setting wired through config types, strict schema, config resolution, Docker create args, dangerous-config reporting, docs, and regression tests; close this issue only after that fix merges and ships.

## Reproduction Assessment

Yes. The reporter provided a minimal Docker command that isolates EPERM to `--security-opt no-new-privileges`, and current source inspection shows OpenClaw always applies that flag to sandbox containers.

## Solution Assessment

Yes. The narrow maintainable direction is an explicit default-on break-glass config knob with audit/docs coverage; a silent fallback retry would be riskier because it could weaken sandbox hardening without operator intent.
