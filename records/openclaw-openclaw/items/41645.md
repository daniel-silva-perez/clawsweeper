## Summary

Keep open. Current main has substantial session/provider usage accounting, daily totals, and Dashboard usage views, but it still does not provide the requested per-user input/output/total token aggregation, user filter/reporting surface, weekly/monthly user rollups, or quota/billing semantics.

## What This Changes

Add token usage accounting by user, including input/output/total tokens, day/week/month reporting, and `/usage` or Dashboard access for multi-user billing or quota management.

## Best Possible Solution

Define a core usage-accounting contract around a normalized principal key, extend `sessions.usage` shared result types with per-user and period rollups, add Dashboard filtering/export/reporting for users, and document how identity, privacy, and any quota enforcement work across channels.

## Reproduction Assessment

Not applicable as a feature request; the missing capability is high-confidence from source inspection. Current `sessions.usage`, its protocol schema, Dashboard query helpers, overview UI, and docs all lack a user aggregate or user filter.

## Solution Assessment

Unclear as proposed: adding raw `user_id` fields to logs would not be enough because OpenClaw needs a normalized cross-channel principal and access-controlled reporting. The maintainable path is to model that principal first, then expose typed aggregates and UI/docs around it.
