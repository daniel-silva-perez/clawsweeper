## Summary

Keep open. Current main has substantial token/cost usage tracking, including session logs, daily totals, and dashboard breakdowns by model, provider, agent, and channel, but it does not implement the requested per-user input/output/total token aggregation, user filter/reporting surface, weekly/monthly user rollups, or quota/billing management.

## Best Possible Solution

Keep this issue open and scope it as a core usage-accounting enhancement: define a normalized user/principal key from session origin metadata, extend `sessions.usage`/shared types with `byUser` and period rollups, add Dashboard filters/export for users, and document the privacy and quota semantics. If quota enforcement is included, add tests around cross-channel identity normalization and access control.

## Risks / Open Questions

- The main product decision is defining a stable cross-channel `user`/billing principal: Control UI browser identity, channel sender IDs, account IDs, group participants, and agent/session scopes do not all mean the same thing.
- Quota or billing enforcement would need privacy and access-control review so dashboard/API exports do not expose sender identifiers to callers who should only see aggregate usage.
