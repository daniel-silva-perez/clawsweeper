## Summary

Keep this PR open. Current main does not ship the proposed `privacy` config, PII redaction hooks, system-prompt masking, media attachment blocking, SDK privacy subpath, or AES-256-GCM session encryption. The PR overlaps with open privacy-filter work in #45783, but it is not fully superseded because it also proposes at-rest session encryption and media controls; the existing review discussion points to real blocker-level correctness/performance issues rather than an obsolete change.

## Best Possible Solution

Keep open for maintainer design review. Best path is to split or redesign: route the LLM-boundary PII filter through the canonical privacy-filter discussion in #45783, or replace both with one approved privacy plan that includes config/schema/docs, async AES-GCM handling, transcript read/write compatibility, media policy, system-prompt masking, SDK boundaries, and focused tests across CLI, embedded messaging channels, tool output, TTS, and session transcript consumers.
