## Summary

Keep this PR open. The PR targets a real reset/queue bug and moves in the right direction, but the current diff still leaves reset-triggered turns waiting behind same-session active work when the reply-run registry is active and the command lane has no depth.

## What This Changes

The PR adds reset-triggered handling to active-run queue policy, suppresses steer/followup handling for reset turns, clears and aborts the session lane for reset turns when lane depth exists, passes reset context into `runReplyAgent`, and adds queue-policy regression tests.

## Best Possible Solution

Update this PR or land a narrow equivalent so reset-triggered turns use interrupt semantics throughout active-run handling, independent of command-lane depth. Preserve heartbeat drops and non-reset queue behavior, abort the active reply or embedded run through the registry/lane path, and add integration coverage for reset turns during active same-session work.

## Reproduction Assessment

Yes. A focused source-level reproduction is to create a running same-session `ReplyOperation` with zero command-lane depth, configure `messages.queue.mode: steer`, and submit a reset-triggered turn; the PR makes policy return `run-now` but the prepared queue helper still waits instead of aborting because the effective queue mode remains `steer`.

## Solution Assessment

No. The proposed direction is good, but the abort semantics are still incomplete; reset should be passed as an interrupting condition into the wait/abort helper or converted to an effective interrupt mode before that helper runs.

- **[P2] Abort reset turns without lane depth:** `src/auto-reply/reply/get-reply-run.ts:820-823`
  - body: This reset abort path still depends on `laneSize > 0`.
