## Summary

Keep open. The PR targets a real queue/reset bug and current main still lacks reset-aware active-run queue policy, but the patch remains incomplete: reset-triggered turns only abort through the new early block when the embedded session lane has depth, while the later prepared-queue helper still receives the original steer/followup mode and waits instead of aborting.

## What This Changes

The PR adds resetTriggered-aware active-run queue-policy handling, suppresses steer/followup for reset turns, extends the lane clear/abort block for reset-triggered runs, passes reset context from the agent runner, and adds queue-policy tests.

## Best Possible Solution

Revise this PR, or land a narrow equivalent, so reset-triggered turns carry interrupt semantics through every active-run path independent of command-lane depth, while preserving heartbeat drops and normal non-reset queue modes. Add coverage for active reply registry state without embedded lane depth before using the PR to close #74093.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.79

Full review comments:

- **[P2] Abort reset runs even without lane depth:** `src/auto-reply/reply/get-reply-run.ts:823-825`
  - body: This reset abort path is still conditional on `laneSize > 0`. Same-session active work can be visible through the reply-run registry before the embedded session lane has depth; in that case the new reset policy returns `run-now`, but `resolvePreparedReplyQueueState` still receives the original steer/collect mode and only waits for the old run. Treat reset-triggered turns as interrupting in this path even when the lane is empty.
  - confidence: 0.79
