## Summary

Keep this PR open. Current main still does not implement the reset-triggered active-run bypass this PR targets: reset state is computed and passed into `runReplyAgent`, but the active-run queue policy still ignores it, so reset tails under `steer`, `followup`, `collect`, or `steer-backlog` can still be enqueued behind an active run. The PR is a focused implementation candidate for #74093, with no obvious supply-chain-sensitive diff, but it still needs maintainer review and regression coverage around lane clearing, active-run abort, and preserved non-reset queue behavior.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to land this focused core reset-queue fix or a narrow equivalent, with regression coverage proving reset-triggered `/new` and `/reset` turns bypass active-run steer/followup queues, clear queued work for the session lane, abort the active run, start the fresh session immediately, and preserve heartbeat plus non-reset queue-mode behavior.
