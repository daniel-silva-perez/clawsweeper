## Summary

Keep this issue open. Current main covers restored/persistent Feishu message-ID dedupe and in-flight same-ID claims, but the reported Feishu WebSocket replay path with fresh message IDs is still not guarded by a reconnect/startup replay window, create_time filtering before agent dispatch, or an explicit offline-message policy.

## Best Possible Solution

Fix this in the bundled Feishu plugin by keeping the current persistent and in-flight same-ID dedupe, then adding a conservative per-account WebSocket startup/reconnect replay guard for stale fresh-ID events. The guard should likely compare message.create_time with connection start and be controlled by an explicit offline-message policy/config. Add focused tests for stale fresh-ID replay suppression, fresh post-connect dispatch, same-ID dedupe, and debounced batches containing suppressed replay content.
