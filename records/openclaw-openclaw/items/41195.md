## Summary

Keep this issue open. Current main has persistent Feishu dedupe and early same-ID processing claims, but the reported fresh-ID WebSocket replay path is still not guarded: inbound Feishu messages are claimed by normalized `message_id` only, then debounced/dispatched without a connection-start replay window, `create_time` cutoff, or offline-message policy. The related discussion and prior partial fixes help explain the history, but they do not supersede this remaining bundled Feishu plugin bug.

## Best Possible Solution

Fix this in the bundled Feishu plugin. Keep the existing persistent same-ID dedupe, then add a conservative per-account WebSocket startup/reconnect replay guard that can suppress stale replayed inbound events even when Feishu assigns fresh `message_id` values, likely using `message.create_time` relative to connection start plus an explicit offline-message policy/config if user-visible behavior changes. Add focused tests for fresh-ID stale replays being skipped, fresh post-connect messages still dispatching, same-ID dedupe still working, and debounce batching not merging suppressed replay content.
