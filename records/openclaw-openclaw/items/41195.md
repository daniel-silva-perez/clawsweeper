## Summary

Keep this issue open. Current main restores and records Feishu dedupe state for the same normalized `message_id`, and it has early in-memory same-ID processing claims, but the reported fresh-ID WebSocket replay gap remains: there is no per-account reconnect/startup replay window, no stale `create_time` cutoff before dispatch, and no config/docs surface for ignoring queued/offline Feishu messages.

## Best Possible Solution

Fix this in the bundled Feishu plugin. Add a conservative per-account WebSocket startup/reconnect replay guard that can suppress stale replayed inbound events even when Feishu assigns fresh `message_id` values, likely based on `message.create_time` relative to connection start plus an explicit offline-message policy/config if user-visible behavior changes. Keep the existing persistent same-ID dedupe, and add focused tests for fresh-ID stale replays being skipped, fresh post-connect messages still dispatching, same-ID dedupe still working, and debounce batching not merging suppressed replay content.
