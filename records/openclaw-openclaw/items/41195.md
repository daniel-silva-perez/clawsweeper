## Summary

Keep this issue open. Current main restores and uses Feishu message-ID dedupe, including in-flight same-ID claims and persistent ID records, but the reported fresh-ID WebSocket replay path is still not guarded by a reconnect/startup replay window, message create-time filtering before agent dispatch, or an explicit offline-message policy.

## Best Possible Solution

Fix this in the bundled Feishu plugin by keeping the current persistent and in-flight same-ID dedupe, then adding a conservative per-account WebSocket startup/reconnect replay guard for stale fresh-ID events. The guard should likely compare message.create_time with connection start and be controlled by an explicit offline-message policy/config. Add focused tests for fresh-ID stale replay suppression, fresh post-connect dispatch, same-ID dedupe, and debounced batches containing suppressed replay content.
