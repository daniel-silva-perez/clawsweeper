## Summary

Keep open. Current main has persistent Feishu same-ID dedupe and in-flight same-ID claims, but the reported fresh-ID WebSocket replay path is still unguarded. The bundled Feishu channel receives WebSocket events, claims them by `message_id`, and dispatches without a connection-start replay window, `create_time` cutoff, or offline-message policy/config.

## Best Possible Solution

Fix this in the bundled Feishu plugin. Keep the persistent same-ID dedupe, then add a conservative per-account WebSocket startup/reconnect replay guard that suppresses stale replayed inbound events even when Feishu assigns fresh `message_id` values, likely using `message.create_time` relative to connection start plus an explicit offline-message policy/config if behavior changes. Add focused tests for fresh-ID stale replays being skipped, fresh post-connect messages still dispatching, same-ID dedupe still working, and debounce batching not merging suppressed replay content.
