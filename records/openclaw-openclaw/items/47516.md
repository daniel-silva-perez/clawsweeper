## Summary

Keep open. Current main still registers Discord reaction add/remove listeners during monitor startup without a config guard, so the title/root complaint remains true. Main now has handler-level early exits that avoid the expensive channel fetch for `reactionNotifications: "off"`, which likely mitigates part of the slow-listener symptom, but it does not implement the requested startup-level listener skip. The related PR #47588 attempted exactly that guard but is closed unmerged; #73133 addressed a separate fetch-before-mode-check path.

## Best Possible Solution

Implement a small startup helper in `extensions/discord/src/monitor/provider.startup.ts` that registers `DiscordReactionListener` and `DiscordReactionRemoveListener` only when at least one guild, DM, or group-DM path can produce reaction notifications. Add focused provider-startup tests for the skip path and for preserved registration when any guild uses `own`/`all`/`allowlist`, when no guild entries exist, or when DMs/group DMs remain enabled. Keep the existing handler-level early exits as defense in depth.

## Risks / Open Questions

- A startup-level skip must preserve DM and group-DM reaction notifications; skipping solely because all configured guilds are `off` would break DM-only or mixed setups where guild reactions are disabled but DM reactions are still valid.
- The current handler-level early exit may already reduce the observed slow REST-call warnings, so the remaining fix should be scoped to avoiding unnecessary listener registration and queue entry only when no configured reaction path can emit notifications.
