## Summary

Keep open. Current main still registers Discord reaction add/remove listeners unconditionally in `registerDiscordMonitorListeners`; the newer reaction-mode fast path only avoids work after the gateway event has already reached the listener. Related PR #47588 proposed this startup guard but was closed unmerged, and #73133 is a separate pre-fetch mitigation.

## Best Possible Solution

Implement a narrow Discord plugin startup guard in `registerDiscordMonitorListeners`: skip `DiscordReactionListener` and `DiscordReactionRemoveListener` only when no configured path can emit reaction notifications, with focused startup tests for skip/register cases and no core behavior changes.
