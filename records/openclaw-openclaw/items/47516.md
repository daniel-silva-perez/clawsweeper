## Summary

Keep open. Current main still registers Discord reaction add/remove listeners during monitor startup without checking whether any guild, DM, or group-DM path can emit reaction notifications; the related #73133 handler fast path reduces downstream REST work but does not remove the event-queue/listener-registration overhead reported here. The closed unmerged #47588 confirms a narrow source-level fix direction remains available.

## What This Changes

The issue asks to skip Discord reaction add/remove listener registration when every configured reaction-notification path is disabled.

## Best Possible Solution

Add a Discord plugin startup predicate in registerDiscordMonitorListeners that skips reaction add/remove listeners only when configured guild reaction modes cannot emit and DM/group-DM reaction handling is disabled, while keeping the handler-level fast paths as defense in depth.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: with guildEntries present and all reactionNotifications set to off, dmEnabled false, and groupDmEnabled false, current registerDiscordMonitorListeners still reaches the add/remove listener registration calls.

## Solution Assessment

Yes. The narrow maintainable solution is a local Discord startup guard with focused provider.startup tests; #73133's handler fast path is useful but does not address listener registration or event-queue overhead.

## Repair Work Prompt

Fix the Discord reaction listener startup overhead reported in openclaw/openclaw#47516. In the Discord plugin, add a narrow predicate around the reaction add/remove listener registration in extensions/discord/src/monitor/provider.startup.ts so the listeners are skipped only when no configured path can emit reaction notifications: configured guild reaction modes cannot emit, direct-message reactions are disabled, and group-DM reactions are disabled or otherwise cannot emit. Preserve listener registration when no guild entries exist but guild reactions may still default to own, when any guild uses own/all/allowlist, or when DM/group-DM reactions can still be authorized. Keep the existing handler fast paths in extensions/discord/src/monitor/listeners.reactions.ts. Add focused provider startup tests in extensions/discord/src/monitor/provider.startup.test.ts for skip and register cases, and adjust existing monitor tests only if needed. Credit the prior closed unmerged #47588 context in the PR body if a replacement PR is created. Do not move this behavior into core and do not change unrelated Discord auth, message routing, or reaction notification semantics.
