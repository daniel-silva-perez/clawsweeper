## Summary

Keep open. Current main still registers Discord reaction add/remove listeners unconditionally during monitor startup; the newer #73133 fix only skips work inside the listener before REST channel fetch and does not remove the event-queue/listener-registration overhead reported here.

## What This Changes

The issue requests skipping Discord reaction listener registration when every configured notification path has reactions disabled.

## Best Possible Solution

Add a narrow Discord plugin startup predicate in `registerDiscordMonitorListeners` that skips the reaction add/remove listeners only when no configured guild, DM, or group-DM path can emit reaction notifications, with focused startup tests for both skip and register cases.

## Reproduction Assessment

Yes. A read-only source reproduction is sufficient: with all guild entries set to `reactionNotifications: "off"` and DMs/group-DMs disabled, current `registerDiscordMonitorListeners` still reaches the unconditional registration calls at `provider.startup.ts:271` and `:275`.

## Solution Assessment

Yes. The best fix is a local Discord startup guard, not a core change; the existing handler fast path should stay as defense-in-depth for enabled listener paths and reconnect bursts.

## Repair Work Prompt

Implement a narrow Discord plugin startup guard for this issue. In `extensions/discord/src/monitor/provider.startup.ts`, avoid registering `DiscordReactionListener` and `DiscordReactionRemoveListener` only when no configured path can produce reaction notifications: configured guild entries are present and every entry has `reactionNotifications === "off"`, `dmEnabled` is false, and `groupDmEnabled` is false. Preserve current registration when guild entries are missing or empty, at least one guild is `own`/`all`/`allowlist`/undefined, `dmEnabled` is true, or `groupDmEnabled` is true. Add focused startup tests in `provider.startup.test.ts` for skip and register cases, adapting prior PR #47588 to the current `provider.startup.ts` split. Keep the change local to `extensions/discord`; do not change core/plugin SDK contracts or remove the existing handler fast path. Add a single-line changelog fix entry crediting @x4v13r1120 and, if source logic is reused, acknowledge #47588 source work appropriately.
