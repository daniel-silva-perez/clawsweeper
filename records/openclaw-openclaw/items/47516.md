## Summary

Keep open. Current main still registers Discord reaction add/remove listeners unconditionally in `registerDiscordMonitorListeners`; the newer reaction-mode fast path only avoids `fetchChannel` after the listener has already received the gateway event. Related PR #47588 proposed this exact startup guard but was closed unmerged, and the shipped/current changelog entry for #73133 covers a different pre-fetch mitigation.

## Best Possible Solution

Implement a narrow Discord plugin startup guard in `registerDiscordMonitorListeners`: skip `DiscordReactionListener` and `DiscordReactionRemoveListener` only when no configured path can emit reaction notifications, with focused startup tests for skip/register cases and no core behavior changes.
