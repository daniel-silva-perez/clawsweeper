## Summary

Keep open. Current main still only wires text, input_text, and image_url content into /v1/chat/completions, and this PR is a useful implementation candidate, but it is not merge-ready because the audio path can still silently drop malformed or failed audio and successful transcripts are not merged into the active current-message structure.

## What This Changes

The PR adds audio, file, and video_url content-block handling to /v1/chat/completions, plus chatCompletions audio/files config schema/types and focused multimodal unit tests.

## Best Possible Solution

Revise the PR so audio content is validated as request input, transcription failure returns a clear 400 instead of proceeding without speech content, and successful transcripts are assembled into the active user turn before prompt history/current-message formatting. Before landing, align the new user-facing config/API surface with docs, changelog, and config help/labels, then run the targeted gateway/config tests plus the changed gate.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: make transcribeFirstAudio return undefined for a request containing input_audio, or send a latest user turn with blank input_audio.data after prior chat history; the PR code either proceeds without a transcript or drops the malformed active turn.

## Solution Assessment

No. Reusing the existing STT and file-extraction pipelines is the right direction, but the current patch should fail closed for bad or untranscribed audio and should merge transcripts into the active user message before prompt assembly.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Reject audio when transcription produces no text:** `src/gateway/openai-http.ts:647-649`
  - body: transcribeFirstAudio returns undefined for provider/config/runtime failures, but this code only skips pushing a transcript and lets the request continue. For an audio-only turn, the gateway can dispatch a prompt containing only the generic audio placeholder and return 200 even though the user's speech was not included. Treat undefined as an invalid audio request when audio was supplied.
  - confidence: 0.91
- **[P2] Merge transcripts into the active user turn:** `src/gateway/openai-http.ts:1071`
  - body: At this point buildAgentPrompt has already formatted history and marked the current message as the media-only placeholder. Prepending the transcript to the whole prompt leaves the current-message block as "User attached 1 audio clip..." in multi-turn requests, so the actual speech is outside the active user turn. Add the transcript before prompt assembly or replace the active turn content with it.
  - confidence: 0.86
- **[P2] Reject empty input_audio blocks:** `src/gateway/openai-http.ts:358-360`
  - body: When input_audio.data is missing or blank, resolveInputAudioPart returns undefined and extractAudioParts silently drops the block. If that malformed audio block is the latest user turn after prior history, the prompt builder skips it and can treat an older user message as current instead of returning invalid_request_error.
  - confidence: 0.88
