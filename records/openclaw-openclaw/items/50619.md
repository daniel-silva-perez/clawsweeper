## Summary

Keep this issue open. Current main still persists `/model` switches and acknowledges them without comparing current session context usage against the target model budget or provider/account rate-limit risk; the existing safeguards are lower-level overflow handling that run later, not the requested switch-time warning.

## What This Changes

The issue requests a `/model` switch warning that estimates current session context and compares it with the target model’s context and rate-limit budget before switching.

## Best Possible Solution

Keep this open and narrow the implementation to a shared model-switch context-risk warning. Resolve the target model's effective `contextTokens`/`maxTokens`, compare it with fresh current session prompt usage when available, and append a clear warning to `/model` acknowledgements across WebChat/TUI and messaging channels when the session is near or above a conservative target threshold. Recommend `/compact` or `/new`; auto-compaction before switching should remain a separate maintainer product decision.
