## Summary

Keep this issue open. The report is recent, specific, and not covered by a clear shipped or current-main fix. Current main still appends cache-ttl transcript custom entries after eligible turns, still writes delivery-mirror assistant rows only after outbound delivery succeeds, and only filters delivery-mirror rows from provider replay. That explains why a later mirror row can have a cache-ttl parent, but it does not prove the underlying unsolicited Discord send is fixed or non-reproducible.

## Best Possible Solution

Keep the issue open and investigate the actual outbound source. The most useful maintainer path is a focused regression around an idle session whose last leaf is `openclaw.cache-ttl`: no cache-ttl pruning pass, replay load, status refresh, queue recovery, or transcript mirror append should call Discord outbound delivery without an explicit inbound/cron/heartbeat/tool trigger. If reproduced, fix the responsible owner path in outbound delivery, queue recovery, or the Discord/channel tool layer and add source/idempotency metadata that makes future delivery-mirror rows auditable.
