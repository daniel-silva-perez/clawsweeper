## Summary

Keep this issue open. Current main has since landed the blank-prompt guard from #74168, but that only narrows one possible trigger; the reported unsolicited Discord delivery is still not covered by a source-backed fix. Cache-ttl custom entries are pruning state, delivery-mirror rows still mean an outbound send succeeded, and the delivery queue still has intentional startup/reconnect replay behavior that could explain a delayed mirrored transcript entry without proving the report fixed or impossible.

## What This Changes

The issue asks OpenClaw to prevent any Discord outbound delivery from being emitted without a user, heartbeat, cron, or other explicit trigger after an idle cache-ttl transcript event.

## Best Possible Solution

Keep the issue open and investigate the outbound source with a focused regression around an idle session whose last leaf is `openclaw.cache-ttl`. The desired end state is that cache-ttl pruning, replay loading, queue recovery, transcript mirroring, and Discord delivery all carry enough source/idempotency metadata to prove no external send occurs without an explicit trigger, while preserving intentional crash recovery for fresh pending sends.

## Reproduction Assessment

No. The report contains credible transcript evidence, but it explicitly says the problem is not reliably reproducible and the checkout lacks the session file plus delivery logs needed to replay the exact sequence.

## Solution Assessment

Unclear. There is no proposed patch for this issue; the blank-prompt fix on current main is useful adjacent hardening, but the maintainable fix depends on proving whether the remaining source is delivery queue recovery, a Discord channel path, a tool-send path, or another background wake path.
