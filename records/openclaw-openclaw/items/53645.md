## Summary

Keep open. Current main still has the Feishu plugin action-layer gap described in the report: generic `send` actions can carry or receive a resolved `replyTo`, and the lower Feishu send helpers can send quoted replies, but the Feishu `send` action still forwards no reply target unless the action is `thread-reply`.

## What This Changes

The issue requests stable Feishu quoted replies when the message tool sends with `replyTo` after upgrading to OpenClaw 2026.3.23-2.

## Best Possible Solution

Fix the Feishu plugin action handler so ordinary `send` maps explicit and resolved reply targets into `replyToMessageId` for text/card sends and `replyToId` for media sends, while preserving `thread-reply` as the explicit `reply_in_thread` path.

## Repair Work Prompt

Fix the Feishu plugin action handler for this issue. Current main resolves `replyTo` before plugin dispatch and the lower Feishu outbound/send helpers already support reply targets, but `extensions/feishu/src/channel.ts` only forwards a reply target for `thread-reply`; ordinary `send` passes undefined to text/card/media helpers. Update the Feishu action layer so ordinary `send` maps explicit/resolved `replyTo` and supported aliases such as `reply_to`, `replyToId`, or `reply_to_id` into `replyToMessageId` for text/card sends and `replyToId` for media sends. Preserve `thread-reply` as the explicit `reply_in_thread` path requiring a message id, and do not change core outbound threading contracts. Add focused regressions in `extensions/feishu/src/channel.test.ts` for ordinary text, card/presentation, and media sends with reply targets. Add a user-facing changelog fix entry crediting @Kimin2011 and @Darrenpig if this behavior is changed. Validate with the focused Feishu channel test, targeted oxfmt check, and the changed gate through the usual OpenClaw Testbox path if promoted.
