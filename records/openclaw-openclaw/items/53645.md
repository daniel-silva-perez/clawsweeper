## Summary

Current main still has the Feishu ordinary-send reply target gap described by the reporter: core resolves `replyTo`, and the lower Feishu adapters can send quoted replies, but the Feishu plugin action handler only forwards a reply target for `thread-reply`. This should stay open and is narrow enough for a focused repair PR.

## What This Changes

The issue requests stable Feishu/Lark quoted replies when the message tool sends ordinary `send` actions with `replyTo` after upgrading to OpenClaw 2026.3.23-2.

## Best Possible Solution

Fix the Feishu plugin action layer so ordinary `send` preserves explicit quoted-reply targets for text, presentation/card, and media sends, while keeping `thread-reply` as the explicit `reply_in_thread` path that requires a message id.

## Repair Work Prompt

Fix the Feishu ordinary `send` action so explicit quoted-reply targets are preserved. Current main resolves `replyTo` in core, and the lower Feishu outbound adapter can send quoted replies, but `extensions/feishu/src/channel.ts` only maps `replyTo`/`reply_to`/`messageId` to `replyToMessageId` for `thread-reply`; ordinary `send` passes `undefined` to text, presentation/card, and media helpers. Update the Feishu plugin action handler and colocated tests so ordinary `send` with `replyTo`/`reply_to` forwards the target to `sendMessageFeishu`, `sendCardFeishu`, and `feishuOutbound.sendMedia` without setting `replyInThread`. Preserve `thread-reply` behavior as the explicit `reply_in_thread` path requiring a message id. Do not change core outbound threading semantics or lower adapter contracts unless a test proves a generic seam is required. Add a user-facing changelog fix entry with reporter credit, following the active version changelog format.
