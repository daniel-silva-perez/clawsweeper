## Summary

Current main still has the Feishu action-path gap described in the issue: generic `replyTo` resolution exists and the lower Feishu outbound adapter can send quoted replies, but ordinary Feishu `send` actions still pass no reply target to text, card, and media helpers. The issue remains valid and narrow enough for an implementation PR.

## What This Changes

The issue asks for Feishu message-tool `replyTo` quoted replies to display reliably after OpenClaw 2026.3.23-2, especially with WebSocket delivery and message-plus-card sends.

## Best Possible Solution

Fix the Feishu plugin action layer so ordinary `send` preserves explicit quoted-reply targets for text, presentation/card, and media sends, while keeping `thread-reply` as the explicit `reply_in_thread` path that requires a message id.

## Repair Work Prompt

Fix the Feishu quoted-reply action mapping bug tracked by this issue. Current main resolves `replyTo` in the shared message send path, and `extensions/feishu/src/outbound.ts` / `extensions/feishu/src/send.ts` can already deliver Feishu replies, but `extensions/feishu/src/channel.ts` only sets `replyToMessageId` for `thread-reply`; ordinary `send` drops explicit reply targets before text, presentation/card, and media sends. Implement the fix in the Feishu plugin action layer only. For `ctx.action === "send"`, resolve explicit `replyTo` / `reply_to` and direct `replyToId` / `reply_to_id` parameters into the reply target passed to `sendMessageFeishu` and `sendCardFeishu` as `replyToMessageId`, and to media sends as `replyToId`. Preserve `thread-reply` behavior as the explicit `reply_in_thread` path that requires a message id and sets `replyInThread: true`; ordinary `send` with `replyTo` should quote but not force `reply_in_thread`. Add focused regression coverage in `extensions/feishu/src/channel.test.ts` for ordinary text, presentation/card, and media sends with reply targets, plus existing no-reply-target behavior if needed. Add a user-facing changelog fix entry with `Thanks @Kimin2011` and, if mentioning corroboration, `Thanks @Darrenpig`; do not touch core outbound routing or lower Feishu adapter contracts unless a test proves it is necessary. Validate with `pnpm test extensions/feishu/src/channel.test.ts`, targeted `oxfmt --check` for touched files, and `pnpm check:changed` via the normal OpenClaw gate.
