## Summary

Keep open. Current main still exposes only whole-tool allow/deny policy: the typed config and Zod schema have no action-aware shape, the shared matcher and pipeline filter by tool name, and both `message` and `gateway` remain single multiplexed tools that dispatch on an `action` parameter only after policy admission. Related gateway-specific reports have already been closed back to this as the canonical tracker, so this remains valid maintainer design work.

## What This Changes

The issue requests config-level action-aware tool policy so operators can deny specific actions such as `message:send` while preserving read/search/react actions and approved workflow sends.

## Best Possible Solution

Keep this as the canonical core tracker and define a maintainer-approved, typed action-aware tool-policy contract with documented grammar, wildcard behavior, precedence, and regression coverage for `message` and `gateway` read/write splits before changing enforcement.
