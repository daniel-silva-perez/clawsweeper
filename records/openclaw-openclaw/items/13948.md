## Summary

Current main still lacks action-level tool allow/deny. The config contract remains whole-tool based, the matcher and pipeline evaluate only tool names, and the `message` and `gateway` tools still dispatch multiplexed `action` parameters after policy filtering. Related #46684 and #54260 were closed back to this issue as the canonical tracker, so this should stay open for maintainer design.

## Best Possible Solution

Keep this issue open as the canonical core tracker. The best path is a maintainer-approved, typed action-aware tool policy contract with documented grammar and wildcard behavior, clear precedence across existing policy scopes, enforcement before multiplexed actions execute, and regression coverage for denying `message:send` while allowing `message:read`, plus denying high-risk `gateway` actions while preserving targeted read-only actions.
