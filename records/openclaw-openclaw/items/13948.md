## Summary

Keep open. Current main still exposes whole-tool allow/deny policy only: the config contract has no action-aware shape, policy matching and inventory filtering operate on tool names, and both `message` and `gateway` dispatch on an `action` argument only after policy admission. Related gateway-specific reports are already closed back to this as the canonical tracker, so this remains valid core authorization design work.

## What This Changes

The issue requests config-level action-aware tool policy so operators can deny actions such as `message:send` while preserving read/search/react actions and approved workflow sends.

## Best Possible Solution

Keep this as the canonical core tracker and define a maintainer-approved, typed action-aware tool-policy contract with documented grammar, wildcard behavior, precedence, and regression coverage for `message` and `gateway` read/write splits before changing enforcement.
