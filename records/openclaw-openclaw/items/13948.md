## Summary

Keep open. Current main still lacks action-aware tool policy: config schemas expose only whole-tool allow/deny fields, policy filtering operates on tool names, and multiplexed tools such as message and gateway read their action argument only after policy admission. Existing maintainer and ClawSweeper comments also keep this as the canonical tracker for related closed gateway reports.

## What This Changes

Add action-level tool allow/deny policy so multiplexed tools like message and gateway can deny send or write actions while preserving read or lookup actions.

## Best Possible Solution

Keep this as the canonical core tracker and design a typed, documented action-aware tool-policy contract with explicit grammar, wildcard behavior, deny precedence, Gateway HTTP semantics, and regression coverage for message and gateway read/write splits.
