## Summary

Keep open. Current main still enforces tool policy at whole-tool granularity, while `message` and `gateway` remain multiplexed tools whose mutating/read-only operations are selected later via an `action` argument. The issue is also already treated in discussion as the canonical tracker for related closed gateway action-level requests.

## What This Changes

The issue requests action-level tool policy so operators can deny mutating actions such as `message:send` while keeping read-only actions on the same multiplexed tool available.

## Best Possible Solution

Keep this as the canonical core tracker and define a typed, documented action-aware tool-policy contract with clear grammar, wildcard behavior, precedence, and per-call enforcement before multiplexed tool actions execute, with regression coverage for `message` and `gateway` read/write splits.

## Reproduction Assessment

Yes. The static reproduction path is high-confidence: current config schema accepts only whole-tool policy fields, policy filtering receives only `tool.name`, and the `message`/`gateway` tools read `action` only inside their execution handlers after policy admission.

## Solution Assessment

Unclear. Action-level deny is the right capability direction, but the best concrete syntax and bypass/privilege semantics need maintainer design before choosing between `message:send`, a `denyActions` map, or another versioned contract.
