## Summary

Keep open. Current main still enforces tool policy at whole-tool granularity, while `message` and `gateway` remain multiplexed tools that dispatch mutating and read-only actions after name-level policy admission. The discussion and related closed reports already establish this issue as the canonical tracker for the remaining authorization-design work.

## What This Changes

Add action-level tool policy so operators can deny mutating actions such as `message:send` or `gateway:config.patch` while preserving read-only actions on the same tool.

## Best Possible Solution

Keep this as the canonical core tracker and define a maintainer-approved, typed, documented action-aware tool-policy contract that is enforced before multiplexed tool actions execute, with coverage for `message` and `gateway` read/write splits.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: configure-time policy accepts only whole-tool rules, admission receives only `tool.name`, and the affected tools read `action` only inside their execution handlers.

## Solution Assessment

Unclear. The requested capability is the right direction, but maintainers should choose the compatible syntax, precedence, and Lobster or privileged-invoke model before implementation.
