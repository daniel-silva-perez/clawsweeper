## Summary

Keep open. Current main still enforces tool policy at whole-tool granularity while `message` and `gateway` remain multiplexed tools whose specific actions are chosen after name-level policy admission; the related gateway-specific reports are already closed back to this issue as the canonical tracker.

## What This Changes

Request action-level tool allow/deny policy so operators can block mutating `message` and `gateway` actions while preserving read-only actions and approved workflow paths.

## Best Possible Solution

Keep this as the canonical core tracker and define a maintainer-approved, typed, documented action-aware tool-policy contract enforced before multiplexed tool actions execute, with coverage for `message` and `gateway` read/write splits.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: `tools.deny: ["message"]` removes the whole message tool, while `tools.deny: ["message:send"]` is only compared to the name `message` and does not block the send action; `denyActions` is not accepted by the strict schema.

## Solution Assessment

Unclear. The requested capability is the right direction, but the best implementation should be a generic action-policy contract with explicit precedence and privileged-workflow semantics, not a one-off special case or implicit bypass.
