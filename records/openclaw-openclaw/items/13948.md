## Summary

Keep open. Current main still enforces tool policy at whole-tool granularity, while `message` and `gateway` remain multiplexed tools whose mutating/read-only actions are chosen after name-level policy admission. The related gateway-specific reports were closed back to this issue as the canonical tracker, and the remaining work is a core authorization design decision rather than conservative cleanup.

## What This Changes

Request action-level tool allow/deny policy so operators can block mutating `message` and `gateway` actions while preserving read-only actions and approved workflow paths.

## Best Possible Solution

Keep this as the canonical core tracker and define a typed, documented action-aware tool-policy contract that is enforced before multiplexed tool actions execute, with regression coverage for `message` and `gateway` read/write splits.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: `tools.deny: ["message"]` removes the entire message tool, while `tools.deny: ["message:send"]` is only compared to the name `message` and does not block the send action; `denyActions` is not accepted by the strict schema. The same name-level pattern applies to `gateway` actions.

## Solution Assessment

Unclear. The requested capability is the right direction, but the best implementation should be a maintainer-approved generic action-policy contract, not a one-off special case or an implicit privileged workflow bypass.
