## Summary

Keep open. Current main still enforces tool policy at whole-tool granularity, and the maintainer discussion already treats this as the canonical tracker for action-aware authorization design across `message` and `gateway`.

## What This Changes

The issue requests config-level action-aware tool policy so mutating actions such as `message:send` or `gateway:config.patch` can be denied while read-only actions on the same tool stay available.

## Best Possible Solution

Keep this as the canonical core tracker and define a maintainer-approved, typed action-aware tool-policy contract that is documented, versioned, and enforced before multiplexed tool actions execute, with regression coverage for `message` and `gateway` read/write splits.

## Reproduction Assessment

Yes. The high-confidence static reproduction is that current config and schema only accept whole-tool policy fields, policy filtering receives only `tool.name`, and `message`/`gateway` read `action` inside their execution handlers after policy admission.

## Solution Assessment

Unclear. Action-level deny is the right capability direction, but the best concrete syntax and bypass model need maintainer product/security design before choosing `message:send`, a `denyActions` map, or a different compatible contract.
