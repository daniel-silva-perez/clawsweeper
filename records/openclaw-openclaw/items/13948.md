## Summary

Current main still only supports whole-tool allow/deny policy. The typed config and strict runtime schema expose `allow`, `alsoAllow`, `deny`, and `profile`, the matcher/pipeline filter by tool name before tool arguments are available, and both `message` and `gateway` remain multiplexed tools that dispatch on an `action` parameter after policy filtering. Related #46684 and #54260 were already closed back to this issue as the canonical tracker, so this should stay open for maintainer design.

## Best Possible Solution

Keep this issue open as the canonical core tracker. The best path is a maintainer-approved, typed action-aware tool policy contract with documented grammar and wildcard behavior, clear precedence across existing policy scopes, enforcement before multiplexed actions execute, and regression coverage for denying `message:send` while allowing read/search-style message actions plus denying high-risk `gateway` actions while preserving targeted read-only actions.
