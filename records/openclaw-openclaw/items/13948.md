## Summary

Keep open. Current main still enforces allow/deny at whole-tool granularity, and the discussion already treats this as the canonical tracker for action-level authorization on multiplexed tools such as `message` and `gateway`.

## What This Changes

Request support for action-level tool allow/deny rules so operators can block mutating actions like `message:send` or `gateway:config.patch` while preserving read-only actions on the same tool.

## Best Possible Solution

Define a maintainer-approved, typed action-aware tool-policy contract with documented syntax, wildcard behavior, and precedence, then enforce it per call before multiplexed tool actions execute, with coverage for `message` and `gateway` read/write splits.
