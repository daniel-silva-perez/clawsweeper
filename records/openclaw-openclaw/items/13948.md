## Summary

Keep open. Current main still enforces tool policy at whole-tool granularity, while the requested safety boundary is per action inside multiplexed tools such as `message` and `gateway`. Prior maintainer and ClawSweeper comments already treat this as the canonical tracker for the related gateway reports, so this is valid core authorization design work rather than cleanup-closeable stale work.

## What This Changes

Request support for action-level tool allow/deny rules, such as blocking `message` sends or mutating `gateway` actions while preserving read-only actions on the same tool.

## Best Possible Solution

Define a maintainer-approved, typed action-aware tool-policy contract with documented syntax and precedence, then enforce it per call for multiplexed tools before their mutating actions execute, with regression coverage for `message` and `gateway` read/write splits.
