## Summary

Keep open. Current main still supports only whole-tool allow/deny policy, while this issue requests action-aware enforcement for multiplexed tools such as message and gateway; the maintainer discussion also explicitly keeps this as the canonical tracker for the remaining core authorization design.

## What This Changes

The issue requests action-level tool policy so operators can deny specific actions such as message send or gateway config writes while preserving read-only actions on the same tool.

## Best Possible Solution

Keep this as the canonical core tracker and design a typed, documented action-aware tool-policy contract with explicit grammar, wildcard behavior, deny precedence, Gateway HTTP semantics, and regression coverage for message and gateway read/write splits.
