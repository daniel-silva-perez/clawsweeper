## Summary

Current main still does not implement action-level tool allow/deny. The config contract and schema remain whole-tool-name based, the policy matcher only receives a tool name, and `message`/`gateway` still dispatch multiplexed `action` parameters after the tool has already passed policy filtering. Related #46684 and #54260 were closed back toward this issue, so this should remain the canonical tracker.

## Best Possible Solution

Keep this open as the canonical core tracker. The best path is a maintainer-approved, typed action-aware tool policy contract with documented grammar and wildcard behavior, clear precedence across existing policy scopes, enforcement before multiplexed actions execute, and regression tests for cases like denying `message:send` while allowing `message:read`, and denying `gateway:config.patch` while allowing `gateway:config.schema.lookup`.
