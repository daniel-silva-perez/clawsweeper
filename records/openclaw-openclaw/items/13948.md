## Summary

Keep open. Current main still admits or removes tools at whole-tool granularity, while the requested split between actions such as `message:send` and `message:read` remains unimplemented and safety-sensitive.

## What This Changes

The issue requests action-level tool policy so operators can deny specific actions such as outbound message sends while preserving read-only or lower-risk actions on the same multiplexed tool.

## Best Possible Solution

Keep this as the canonical core tracker and define a maintainer-approved, typed action-aware tool-policy contract with documented grammar, wildcard behavior, precedence, Gateway HTTP semantics, and regression coverage for message and gateway read/write splits.
