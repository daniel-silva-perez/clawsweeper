## Summary

Current main still lacks action-level tool allow/deny. Tool policy remains a whole-tool-name contract, the runtime filters by `tool.name` before execution, and `message`/`gateway` still parse their `action` parameters inside the already-allowed multiplexed tool execution path. Related #46684 and #54260 were closed back toward this issue, so this should remain the canonical tracker.

## Best Possible Solution

Keep this open as the canonical core tracker. The best path is a maintainer-approved, typed action-aware tool policy contract with documented grammar, wildcard behavior, precedence across existing policy scopes, enforcement before multiplexed actions execute, and regression tests for cases like `message:send` denied while `message:read` remains available and `gateway:config.patch` denied while `gateway:config.schema.lookup` remains available.
