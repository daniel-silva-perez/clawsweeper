## Summary

Keep open. Current main still enforces tool policy at whole-tool granularity, while the `message` and `gateway` tools remain multiplexed surfaces whose mutating/read-only actions are selected from call arguments after name-level policy admission. The linked gateway-specific reports are already closed back to this issue as the canonical tracker, and the issue describes a real authorization-design gap rather than cleanup-close material.

## What This Changes

The issue requests action-level allow/deny policy so operators can block mutating actions such as `message:send` or `gateway:config.patch` while preserving read-only actions and approved workflow paths.

## Best Possible Solution

Keep this as the canonical core tracker and define a maintainer-approved, typed, documented action-aware tool-policy contract enforced before multiplexed tool actions execute, with regression coverage for `message` and `gateway` read/write splits.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: `tools.deny: ["message"]` removes the whole `message` tool, while `tools.deny: ["message:send"]` is only compared to the name `message` and does not block the send action; `denyActions` is not accepted by the strict schema.

## Solution Assessment

Unclear. The requested capability is the right direction, but the maintainable fix should be a generic action-policy contract with explicit precedence and privileged-workflow semantics, not a one-off `message` or `gateway` special case.
