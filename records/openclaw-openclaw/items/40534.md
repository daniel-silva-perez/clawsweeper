## Summary

Current main still has the cron announce reasoning-filter gap this PR targets. Main creates reasoning payload items, but the returned embedded payload object drops `isReasoning`, and the centralized cron outcome helpers still select payloads, fallback text, and post-error recovery without checking reasoning. The newer `finalAssistantVisibleText` mitigation helps only opt-in channels such as Telegram and Discord, so it does not fully supersede this PR, especially for the newer Matrix/Feishu report in #73186. Security review found the provided diff scoped to TypeScript payload/cron code and tests, with no CI, dependency, lockfile, package script, publishing, secret, or artifact-download changes.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to rebase it onto current main, propagate `isReasoning` from `buildEmbeddedRunPayloads`, include it in the cron delivery payload surface, filter it in summary/output/deliverable selection and first-text fallback inside `resolveCronPayloadOutcome`, exclude it from post-error recovery detection, update the `run-executor` path if needed, and add regression tests for mixed reasoning plus final text, reasoning-only, and error-then-reasoning-only cron announce cases.
