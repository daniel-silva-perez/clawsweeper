## Summary

Keep open. Current main still has only pre-delivery, adapter opt-in sanitization and the existing runtime-context scrubbers do not cover the reported Claude Code XML scaffolding. Discord, the reported channel, has no `sanitizeText` hook, and even `sanitizeForPlainText` would leave `<previous_response>` untouched while exposing `<system-reminder>` contents rather than stripping the internal block.

## Best Possible Solution

Keep this issue open. Add an allow-listed final-stage outbound sanitizer in the shared delivery path after `message_sending` and presentation normalization but before `sendPayload`, `sendFormattedText`, or chunked text sends. Seed it with `<system-reminder>` and `<previous_response>` block stripping, decide whether Discord should also opt into `sanitizeForPlainText`, and add regression coverage for Discord plus the shared outbound helper so both reported tag shapes are blocked.

## Risks / Open Questions

- An overbroad XML/HTML stripper could remove legitimate user text, code snippets, or angle-bracket prose; the safer fix is an allow-list for internal-only runtime tags plus focused tests.
- Only adding Discord `sanitizeText` would not protect all delivery paths after `message_sending` hooks; the durable fix belongs at the shared final outbound boundary.
