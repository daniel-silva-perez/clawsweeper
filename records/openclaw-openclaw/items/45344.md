## Summary

Keep this PR open. Current main at 891c7d9f1c1f5e0f52935fadfeacf695b0ef1eb8 still accepts explicit unknown agent selectors on the OpenAI-compatible chat and Responses HTTP paths instead of validating them against configured agents. The PR remains a focused implementation candidate, with remaining review work around explicit `main` compatibility and the `knownAgentIds` contract rather than obsolescence.

## Best Possible Solution

Keep this PR open for maintainer review or revision. The fix should validate explicit `x-openclaw-agent-id`, `x-openclaw-agent`, `openclaw:<agentId>`, `openclaw/<agentId>`, and `agent:<agentId>` selections against configured agents for both `/v1/chat/completions` and `/v1/responses`, return `400 invalid_request_error` for unknown explicit selectors, preserve implicit default routing and documented explicit `main`/`openclaw/default` behavior, clarify the validation helper contract, and retain regression tests for header and model-selector rejection.

## Risks / Open Questions

- Current main can still accept explicit unknown gateway agent selectors and continue with unknown agent-scoped routing instead of returning a stable 400 response.
- The PR should preserve documented explicit `main` / `openclaw/default` compatibility, especially in multi-agent configs where `main` may be the implicit fallback rather than an entry in `agents.list`.
- The PR should clarify whether omitted `knownAgentIds` means no validation or an invalid caller contract, so future gateway handlers do not get surprising behavior.
