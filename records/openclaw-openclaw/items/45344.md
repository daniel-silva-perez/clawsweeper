## Summary

Keep this PR open. Current main still normalizes explicit gateway agent selectors and uses them directly on the OpenAI-compatible chat and Responses HTTP paths, so the PR’s central security/behavior fix is not implemented or superseded. The candidate remains useful, but it should be rebased and revised to fit the current runtime config boundary and to resolve the explicit `main` compatibility and helper-contract review comments.

## Best Possible Solution

Keep this PR open for maintainer review or revision. The best path is to rebase onto current main and implement configured-agent validation in the current `http-utils`/gateway handler shape using `getRuntimeConfig()` plus `listAgentIds`/`resolveDefaultAgentId`, return `400 invalid_request_error` for unknown explicit `x-openclaw-agent-id`, `x-openclaw-agent`, `openclaw/<agentId>`, `openclaw:<agentId>`, and `agent:<agentId>` selectors on `/v1/chat/completions` and `/v1/responses`, preserve implicit default routing plus explicit `main`/`openclaw/default` compatibility, document or type the helper contract, and add regression tests for known, unknown, and default selectors.

## Risks / Open Questions

- Current main can still accept explicit unknown gateway agent selectors and proceed with unknown agent-scoped routing instead of returning a stable `400 invalid_request_error` on the chat and Responses compatibility endpoints.
- The open patch described in the PR imports `loadConfig()` in gateway HTTP request handling; current runtime guidance and current main prefer `getRuntimeConfig()` or a passed config snapshot, so the implementation should be rebased onto the current config boundary.
- A naive allowlist check can regress documented/default behavior if it rejects explicit `main` or `openclaw/default` when the configured default agent is not literally listed as `main`.
