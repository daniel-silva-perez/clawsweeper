## Summary

Keep this issue open. Current main already has primary-model, fallback, allowlist, `/model` switching, and configure-time multi-select infrastructure, but the onboarding path still prompts for one default model and only applies `applyPrimaryModel`. The requested first-run multi-model selection remains a real gap and needs maintainer product/UX direction before automation should change onboarding flow length and scope.

## Best Possible Solution

Keep this open for maintainer product direction. The narrow implementation path is to reuse `promptModelAllowlist`, `applyModelFallbacksFromSelection`, and `applyModelAllowlist` in `src/wizard/setup.ts` after provider auth/default-model selection, preserve the chosen primary model, add onboarding tests, and update onboarding docs. If maintainers want true multi-provider endpoint setup during first run, split that into a broader onboarding UX design instead of hiding it inside the allowlist/fallback step.
