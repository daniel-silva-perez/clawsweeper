## Summary

Keep open. Current main has manual configured fallbacks and OpenAI Codex forward-compatibility for specific GPT-5.4/GPT-5.5 refs, but it does not implement the requested provider-family wildcard/auto model pool such as `openai-codex/auto` or `openai-codex/*`. The remaining work belongs in core fallback policy with provider-plugin hints, not as a pure ClawHub/plugin close.

## Best Possible Solution

Add an explicit provider-scoped auto fallback design: parse/accept `provider/auto` or a documented config field, let provider plugins expose an ordered eligible model list or priority hints, and have the core fallback runner expand that into same-provider candidates before any configured cross-provider fallbacks. Document the behavior in model failover and provider docs, with tests that prove `openai-codex/auto` stays within OpenAI Codex unless the operator configured broader fallbacks.

## Risks / Open Questions

- Existing OpenAI Codex forward-compatibility already covers some renamed concrete models, so a future implementation should avoid duplicating that provider-specific catalog logic.
- Auto fallback must preserve the reporter's cost/provider boundary and should not silently cross into routers like `openrouter/auto` unless explicitly configured.
