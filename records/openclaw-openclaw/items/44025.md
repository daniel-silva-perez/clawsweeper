## Summary

Keep open. Current main has explicit model fallback chains and OpenAI Codex forward-compat entries for known current models, but it does not implement provider-scoped `auto` or wildcard fallback selection; the remaining request needs a maintainer decision on model ordering, cost/tier semantics, and whether this belongs in the core fallback contract or provider/plugin APIs.

## What This Changes

The issue requests provider-scoped wildcard or `auto` model fallback so a configured provider family can rotate through known models before falling back to cross-provider routing.

## Best Possible Solution

Define a generic provider-scoped fallback contract before implementation: keep explicit `fallbacks` as the predictable operator-controlled path, and, if maintainers want `provider/auto` or `provider/*`, add a documented provider/plugin API for ordered fallback candidates with capability, auth, and cost/tier filtering before wiring it into `runWithModelFallback`.

## Reproduction Assessment

Yes for the missing feature at source level: `openai-codex/auto` would parse as provider `openai-codex` with literal model `auto`, and the fallback planner only appends explicit configured fallback refs. I did not run live model calls because this review is read-only, but the code path is clear.

## Solution Assessment

Unclear as proposed. The user problem is real, but a literal wildcard/auto suffix is not obviously the safest contract until maintainers decide ordering, provider ownership, and how to avoid surprising cost/quality changes; explicit fallback lists are the current maintainable workaround.
