## Summary

Keep open. Current main still drops stability-bundle error message, stack, and cause, and the discussion identifies this PR as the maintainer-preferred base over #71352. The PR is not merge-ready because the current head copies raw error text into a shareable diagnostics surface without the existing support redaction model and serializes `cause` as a lossy string.

## What This Changes

This PR extends diagnostic stability bundle error metadata to include `message`, `stack`, and `cause`, adds a bundle string sanitizer, and updates serializer tests to expect error messages to be retained.

## Best Possible Solution

Keep this PR as the canonical implementation candidate, but revise it so stability bundles preserve useful error diagnostics through the same privacy-aware redaction approach used by support exports, with bounded `message` and `stack`, NUL stripping, structured single-level cause metadata, and focused regression tests for startup failures and imported support bundles.

## Reproduction Assessment

Yes. Current main has a source-level reproduction path: gateway startup failures pass the thrown error into `writeDiagnosticStabilityBundleForFailureSync()`, which calls a serializer that only returns `name` and `code`; the current tests also assert that thrown error messages are omitted.

## Solution Assessment

No, not as currently written. The PR is the right narrow area and likely the right base, but the safest solution needs support-style redaction and structured cause handling before the new fields are persisted or exported.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Redact error text before persisting diagnostics:** `src/logging/diagnostic-stability-bundle.ts:142-148`
  - body: This copies raw `message` and `stack` text into stability bundles, and those bundles are included in diagnostics exports that are documented as shareable. Home-path redaction and length caps do not catch tokens, prompts, chat text, IDs, or other secret-like values that can appear in validation errors or stacks, so this can leak private data in `stability/latest.json`. Please run these fields through the support redaction/privacy path before writing them and keep tests that prove secret-like error text is redacted.
  - confidence: 0.88
- **[P2] Preserve cause metadata instead of stringifying it:** `src/logging/diagnostic-stability-bundle.ts:150-152`
  - body: `String(error.cause)` loses the fields operators need from an `Error` cause, such as `name`, `message`, and `code`, and turns object causes into unhelpful text like `[object Object]`. Since the PR promises single-level cause preservation, serialize one bounded/redacted cause object and avoid recursion instead of storing a lossy string.
  - confidence: 0.78
