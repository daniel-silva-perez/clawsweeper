## Summary

Keep this PR open. Current main still defaults `tools.fs.workspaceOnly` to false/unrestricted when unset, documents workspace-only FS as optional hardening, and has no sensitive-path denylist implementation. The PR is a plausible security-hardening candidate, but it is not implemented on main and the submitted diff has unresolved security review gaps around the host read path and symlink-safe denylist enforcement.

## Best Possible Solution

Keep this PR open as a security-hardening implementation candidate, or replace it with a maintainer-owned patch. The best path is to make an explicit product/security decision on the default filesystem policy, then update `src/agents/tool-fs-policy.ts`, host read/write/edit wiring in `src/agents/pi-tools.ts` and `src/agents/pi-tools.read.ts`, config types/schema/help, public security docs, changelog, and regression tests together. If the denylist remains part of the design, enforce it for read/write/edit and check canonical real paths before file access. Coordinate the default-only overlap with #58947.

## Risks / Open Questions

- Changing the default from unrestricted to workspace-only is a compatibility and security-policy decision; config schema/help, docs, changelog, and regression tests need to move together.
- Merging the current PR as submitted could create false assurance because the host read tool remains able to read non-workspace paths when `tools.fs.workspaceOnly=false`.
- A sensitive-path denylist must be enforced consistently for read/write/edit and use symlink-safe canonicalization before file access; lexical prefix checks are not enough.
- Security/supply-chain pass: the provided PR file list is limited to source and tests, with no workflows, dependency files, lockfiles, install scripts, or publishing metadata changes, so the main concern is functional security correctness rather than third-party code execution.
