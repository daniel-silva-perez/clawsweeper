## Summary

Keep this PR open. Current main still does not implement the PR’s central security change: `tools.fs.workspaceOnly` remains opt-in/default-false, there is no agent file-tool sensitive-path denylist, and existing tests/docs still preserve or describe unrestricted host filesystem behavior when the flag is unset or false. The PR remains a plausible implementation candidate, but its submitted design still needs security fixes for read-path enforcement and symlink-safe path checks before merge.

## Best Possible Solution

Keep this PR open as the current implementation candidate, or replace it with a maintainer-owned security patch. The best path is to make an explicit product/security decision on the default filesystem policy, then update `src/agents/tool-fs-policy.ts`, host read/write/edit wiring in `src/agents/pi-tools.ts` and `src/agents/pi-tools.read.ts`, config types/schema/help, public security docs, changelog, and regression tests together. If a sensitive-path denylist remains part of the design, enforce it consistently for read/write/edit and use symlink-safe canonicalization before file access.

## Risks / Open Questions

- The PR changes a security and compatibility default; landing it needs coordinated config schema/help/docs/changelog and migration/product judgment, not only the small code diff.
- The submitted PR diff does not guard the standalone host read tool when `tools.fs.workspaceOnly=false`, leaving the credential-exfiltration path described in the PR motivation open.
- The proposed denylist uses lexical path checks in the provided diff/review context; without realpath/canonical checks before file access, symlinks can bypass sensitive-path blocking.
- Pattern-based sensitive-file denylisting can produce false positives or false assurance unless maintainers define the exact threat model and test read/write/edit behavior together.
