## Summary

Keep this PR open. Current main still preserves the legacy default where `tools.fs.workspaceOnly` is unset/default-false, and there is still no agent file-tool sensitive-path denylist. The submitted PR remains a plausible security hardening candidate, but the PR discussion correctly flags security gaps in the proposed diff: the host read tool is not denied sensitive paths when workspace-only mode is explicitly disabled, and the proposed path checks need symlink-safe canonicalization.

## Best Possible Solution

Keep this PR open as an implementation candidate, or replace it with a maintainer-owned security patch. The best path is an explicit product/security decision on the default filesystem policy, then coordinated changes to `src/agents/tool-fs-policy.ts`, host read/write/edit wiring in `src/agents/pi-tools.ts` and `src/agents/pi-tools.read.ts`, config types/schema/help/generated metadata, public security docs, changelog, and regression tests. If a sensitive-path denylist remains part of the design, enforce it consistently for read/write/edit and use symlink-safe canonicalization before file access.

## Risks / Open Questions

- Landing the PR as submitted would leave the main credential-exfiltration read path open for users who explicitly set `tools.fs.workspaceOnly=false`.
- A denylist that does not canonicalize paths with `realpath` before access can create false assurance because symlinked paths may bypass lexical checks.
- Changing this default is a security and compatibility decision that needs coordinated config schema/help, generated docs, public security docs, changelog, and regression tests.
- Filename-pattern denylisting can block legitimate files or miss secrets unless maintainers define the exact threat model and tests cover read/write/edit behavior together.
