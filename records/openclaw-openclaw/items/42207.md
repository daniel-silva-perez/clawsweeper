## Summary

Keep this PR open. Current main still defaults `tools.fs.workspaceOnly` to false/unrestricted behavior when unset, host read/write/edit paths still allow non-workspace host access when unrestricted, and current docs/config help still describe workspace-only FS as optional. The PR is a plausible security-hardening candidate, but the proposed diff also has unresolved security gaps called out in review, especially the unguarded host read path and symlink-safe denylist handling.

## Best Possible Solution

Keep this PR open as a security-hardening implementation candidate, or replace it with a maintainer-owned patch. The best path is to make an explicit product/security decision on the default filesystem policy, then update `src/agents/tool-fs-policy.ts`, host read/write/edit wiring in `src/agents/pi-tools.ts` and `src/agents/pi-tools.read.ts`, config types/schema/help, public security docs, changelog, and regression tests together. If a sensitive-path denylist remains part of the design, enforce it for read/write/edit and use symlink-safe canonicalization before file access. Coordinate with #58947 for the overlapping default-workspace-only request.

## Risks / Open Questions

- Changing the default from unrestricted to workspace-only is a compatibility and security-policy decision that needs maintainer review and aligned config, docs, changelog, and regression coverage.
- Merging the current PR as-is could create false assurance because the host read tool remains able to read sensitive host paths when `tools.fs.workspaceOnly=false`.
- A sensitive-path denylist must be enforced consistently for read/write/edit and made symlink-safe before file access; lexical path checks are not enough.
- The related open proposal #58947 tracks the default-workspace-only part of the same problem, but it does not by itself supersede this implementation candidate or cover the denylist design.
