## Summary

Keep this PR open. Current main still has the legacy unrestricted default for `tools.fs.workspaceOnly`, no sensitive-path denylist implementation, and tests/docs that preserve or describe the existing behavior. The PR is a plausible security-hardening candidate, but the submitted diff has unresolved security correctness gaps around host read enforcement and symlink-safe path checks.

## Best Possible Solution

Keep this PR open as a security-hardening implementation candidate, or replace it with a maintainer-owned patch. The best path is to make an explicit product/security decision on the default filesystem policy, then update `src/agents/tool-fs-policy.ts`, host read/write/edit wiring in `src/agents/pi-tools.ts` and `src/agents/pi-tools.read.ts`, config types/schema/help, public security docs, changelog, and regression tests together. If the denylist remains part of the design, enforce it consistently for read/write/edit and use symlink-safe canonicalization before file access. Coordinate the default-only overlap with #58947.

## Risks / Open Questions

- Changing the default from unrestricted to workspace-only is a compatibility and security-policy decision that needs coordinated schema/help/docs/changelog/tests.
- Merging the PR as submitted could create false assurance if the host read tool can still read non-workspace sensitive paths when `tools.fs.workspaceOnly=false`.
- A denylist based only on lexical path checks can be bypassed through symlinks unless canonical real paths are checked before file access.
