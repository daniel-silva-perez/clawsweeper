## Summary

Keep this draft PR open. Current main still lacks the requested SDK-facing `tools.invoke` RPC, so the PR is not obsolete; however the proposed SDK helper has a concrete tool-name override bug on a sensitive direct tool execution surface and the branch overlaps with open PR #74804, so it needs author/maintainer repair and API/security review before merge.

## What This Changes

The PR adds a Gateway `tools.invoke` RPC, protocol schemas/types, method discovery and scopes, a Gateway handler, shared HTTP/RPC direct-tool invocation core, SDK helper/types, generated Swift protocol params, docs, changelog, and tests for tool policy and approval behavior.

## Best Possible Solution

Keep the PR open, fix the SDK param builder so the explicit `oc.tools.invoke(name, params)` argument always wins or rejects a caller-supplied `name`, then compare this branch with #74804 and land one additive RPC implementation that preserves `operator.write`, admin-only owner treatment, Gateway allow/deny filtering, plugin approval semantics, generated protocol artifacts, and docs.

## Reproduction Assessment

Yes. On current main, calling `oc.tools.invoke("demo")` follows `ToolsNamespace.invoke()` and throws the unsupported Gateway API error, while the advertised methods and handler map omit `tools.invoke`. The PR bug is reproducible by passing a JavaScript params object with a hidden `name` field to the SDK helper.

## Solution Assessment

No, not as currently patched. The overall direction is appropriate and narrow for the member-authored request, but the SDK params construction must be corrected before this direct tool execution surface is safe to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve the explicit SDK tool name:** `packages/sdk/src/client.ts:185`
  - body: Because `gatewayParams` is spread after `{ name: toolName }`, any JavaScript caller or cast can pass `{ name: "other-tool" }` in the params object and silently invoke a different tool than the explicit first argument. On this direct tool execution surface, strip or reject `params.name`, or spread params first so the explicit `name` argument always wins.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff adds a sensitive direct tool invocation RPC and has a concrete SDK parameter-smuggling issue that can switch the invoked tool name before the request reaches Gateway policy.

Concerns:

- **[medium] SDK params can override tool name:** `packages/sdk/src/client.ts:185`
  - body: The new SDK helper spreads caller params after the trusted tool-name argument. If an app forwards a user-controlled params object, a hidden `name` field can switch the invoked tool while the call site appears to request another tool.
  - confidence: 0.88
