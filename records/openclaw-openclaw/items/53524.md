## Summary

Keep open. Current main still lacks the proposed Discord post-reset startup-window marker, and the PR branch is not ready to merge because it would leave one-shot reset-window gaps and regress the current active session-id resolution path. The earlier Greptile findings about the thread-only guard and disk-persist clear were addressed in the latest PR commit, but the remaining review findings are separate blockers.

## What This Changes

The PR adds a persisted `postRotationStartupUntilMs` session field and uses it to wrap Discord channel/thread post-reset queued owner messages with a startup-recovery steer in `agent-runner.ts`, `session.ts`, and session typing.

## Best Possible Solution

Consolidate this PR with #49001 into one durable Discord reset-window implementation that covers channel and thread resets, applies only to the first qualifying owner turn, clears after both normal and embedded-queue consumption paths, preserves active session-id resolution, and includes focused channel plus thread regression coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Clear the window only after an owner steer is consumed:** `src/auto-reply/reply/agent-runner.ts:854`
  - body: `startedRun` is set for any agent turn, but the startup steer only applies to owner Discord messages. If a non-owner channel message starts a run before the owner's first post-reset turn, this `finally` block clears `postRotationStartupUntilMs` and the owner turn never gets the recovery prompt. Gate clearing on actual steer consumption instead of any started run.
  - confidence: 0.88
- **[P2] Clear the window before returning from embedded steer enqueue:** `src/auto-reply/reply/agent-runner.ts:276`
  - body: When `queueEmbeddedPiMessage` succeeds, this early return exits before the new clear path runs. The marker remains active, so later owner messages inside the 30 second window can be wrapped again as the first post-reset message. Clear the marker on this successful enqueue path too.
  - confidence: 0.86
- **[P2] Preserve the active session id when queueing the steer prompt:** `src/auto-reply/reply/agent-runner.ts:276`
  - body: Current main resolves the live run id through `replyRunRegistry.resolveSessionId(sessionKey)` before queueing an embedded steer. This PR queues with `followupRun.run.sessionId`, which can target a stale pre-reset id after session rotation; keep the current active-id resolution while swapping in the steer prompt.
  - confidence: 0.74
