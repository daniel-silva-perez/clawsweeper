## Summary

Keep open. Current main still lacks the reset-startup window, while the PR branch has blocking one-shot correctness defects in the new reply path and overlaps the still-open #49001, so it is not ready to merge or close as implemented.

## What This Changes

The PR adds a persisted `postRotationStartupUntilMs` session field and uses it to wrap Discord channel/thread post-reset owner messages with a startup-recovery steer in `agent-runner.ts` and `session.ts`.

## Best Possible Solution

Consolidate this PR with #49001 into one Discord reset-window implementation that covers channel and thread resets, steers exactly the first qualifying owner turn, clears on both embedded and normal consumption paths, preserves active session-id resolution, and includes focused channel plus thread regression coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Clear the window only after an owner steer is consumed:** `src/auto-reply/reply/agent-runner.ts:850-857`
  - body: `startedRun` is set for any agent turn, but the new startup steer only applies to owner Discord messages. A non-owner channel message can start a run inside the reset window and this `finally` block then clears `postRotationStartupUntilMs` before the owner's first post-reset turn, leaving the channel race unresolved. Gate clearing on actual steer consumption.
  - confidence: 0.88
- **[P2] Clear the window before returning from embedded steer enqueue:** `src/auto-reply/reply/agent-runner.ts:273-277`
  - body: When `queueEmbeddedPiMessage` succeeds in streaming mode, this early return exits before the later clear path runs. The marker can stay active and wrap multiple owner messages inside the 30 second window instead of behaving as a one-shot reset recovery steer.
  - confidence: 0.86
- **[P2] Preserve the active session id when queueing the steer prompt:** `src/auto-reply/reply/agent-runner.ts:273`
  - body: Current main resolves the active run id through `replyRunRegistry.resolveSessionId(sessionKey)` before queueing an embedded steer. This new path sends the steer prompt to `followupRun.run.sessionId` directly, which can be stale after session rotation; keep the existing active-id resolution.
  - confidence: 0.74
