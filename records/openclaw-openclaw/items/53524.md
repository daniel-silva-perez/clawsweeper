## Summary

Keep open. Current main still lacks a Discord post-reset startup marker, and this PR remains relevant, but the branch should not merge as-is because its one-shot clearing behavior can still miss or repeat the intended owner-turn recovery path and it overlaps open #49001.

## What This Changes

The PR adds a persisted `postRotationStartupUntilMs` session window and uses it to steer Discord post-reset queued owner messages in `agent-runner.ts`, `session.ts`, and `SessionEntry` typing.

## Best Possible Solution

Consolidate this PR with #49001 into one durable Discord reset-window implementation that applies only to the first qualifying owner turn, clears after both normal and embedded-queue consumption paths, preserves current session-id resolution, and includes focused channel and thread reset regression coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Clear the window only after an owner steer is consumed:** `src/auto-reply/reply/agent-runner.ts:854`
  - body: `startedRun` is set for any run, but the startup steer only applies to owner Discord messages. If a non-owner channel message starts a run before the owner's first post-reset turn, this `finally` block clears `postRotationStartupUntilMs` and the owner turn never gets the recovery prompt. Gate the clear on actual steer consumption instead of any started run.
  - confidence: 0.88
- **[P2] Clear the window before returning from embedded steer enqueue:** `src/auto-reply/reply/agent-runner.ts:276`
  - body: When `queueEmbeddedPiMessage` succeeds, this early return exits before the new `finally` clear path is reached. The marker remains active, so subsequent owner messages inside the 30 second window can be wrapped again as the first post-reset message. Clear the marker on this successful enqueue path too.
  - confidence: 0.86
- **[P2] Preserve the active session id when queueing the steer prompt:** `src/auto-reply/reply/agent-runner.ts:276`
  - body: Current main resolves the active run id through `replyRunRegistry.resolveSessionId(sessionKey)` before queueing an embedded steer. This PR branch queues with `followupRun.run.sessionId`, which can target a stale pre-reset id after session rotation; keep the current active-id resolution while swapping in the steer prompt.
  - confidence: 0.74
