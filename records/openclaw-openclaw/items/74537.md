## Summary

Keep open. Current main appears to have fixed the earlier generic bare `/reset`/`/new` fallthrough into an empty provider call, but this report is narrower around lossless-claw/session-end lifecycle handling on an already-fresh session. Source still shows reset/session rollover emitting `session_end` lifecycle hooks without a transcript-history guard, so the plugin-facing empty-history path is not proven fixed.

## What This Changes

The issue requests that `/reset` and `/new` skip the session-end/wrap-up LLM call when the session is already fresh and has no message history.

## Best Possible Solution

Preserve the generic reset lifecycle contract while preventing empty transcript data from reaching any lifecycle LLM wrap-up path. The maintainable end state is either a core lifecycle signal/guard that makes empty reset sessions explicit and non-callable, or a lossless-claw-side no-op that reliably short-circuits before invoking the embedded agent when the transcript has no user/assistant messages.

## Reproduction Assessment

Yes. The issue gives a concrete Telegram/lossless-claw repro with logs, and current source inspection shows the ordinary command fallthrough is fixed while `session_end` lifecycle emission still has no empty-transcript guard. I did not live-run the external lossless-claw plugin in this read-only review.

## Solution Assessment

Unclear. The existing main-branch fix is the right solution for the older non-ACP empty-tail fallthrough, but it is not sufficient evidence for this plugin lifecycle report; a safer fix needs maintainer review of the session_end/plugin contract before suppressing lifecycle events or changing hook payload semantics.
