## Summary

Keep open. Current main has tests and command-handler code that acknowledge bare chat `/new` and `/reset` without the normal reply model path, but the reset/session lifecycle paths still pass empty prior transcripts into hooks and one gateway agent path intentionally converts a bare reset into a model prompt. The issue includes concrete Telegram/provider logs, and related PR #73371 appears to target the same remaining empty-call behavior, so this is a valid narrow bug for maintainer review rather than cleanup.

## What This Changes

Skip or acknowledge bare `/new` and `/reset` on already-fresh sessions without launching an empty-message LLM or lifecycle call that surfaces a provider 400.

## Best Possible Solution

Preserve the existing bare chat acknowledgement behavior, then make the reset lifecycle/gateway path stop only the empty-history model-backed work: either route the affected Telegram/native/non-ACP reset surface through the same acknowledgement contract or add an explicit empty-transcript lifecycle guard that does not remove useful reset hooks for plugins.

## Reproduction Assessment

Yes for triage: the issue provides concrete Telegram steps and provider log signatures, and current source shows empty reset lifecycle payloads can still reach hooks while one reset surface still continues into a model prompt. A live Telegram plus lossless-claw reproduction was not run during this read-only review.

## Solution Assessment

Unclear. The direction is right, but the best fix should be reviewed against #73371 and the plugin hook contract so it suppresses only empty-history LLM work rather than dropping reset/session lifecycle notifications wholesale.
