## Summary

Keep open. Current main has the bare `/reset` and `/new` acknowledgement behavior, but the empty-history reset lifecycle path is still present, so the reported lossless-claw/provider 400 path is not proven fixed.

## What This Changes

The issue asks OpenClaw to avoid model-backed reset/session lifecycle work when `/reset` or `/new` is invoked on an already-fresh session with no conversation history.

## Best Possible Solution

Keep the shipped bare-command acknowledgement behavior, then define an empty-transcript lifecycle contract so model-backed reset/session work no-ops on already-fresh conversations while non-LLM plugins can still observe lifecycle events where appropriate.

## Reproduction Assessment

Yes for triage: the issue gives concrete Telegram steps and provider/cooldown logs, and current source still shows empty reset lifecycle payloads can be emitted. I did not run a live Telegram plus lossless-claw reproduction because this review was read-only.

## Solution Assessment

Unclear. Skipping model-backed lifecycle work for empty histories is the right direction, but the best fix should be chosen at the plugin lifecycle contract boundary rather than silently dropping all reset or session_end notifications.
