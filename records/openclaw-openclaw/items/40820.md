## Summary

Keep open: the requested Telegram settings panel is still not implemented on current main, so the PR is not obsolete. The branch is not merge-ready because it adds the feature under the removed `src/telegram` tree while current Telegram runtime lives in the plugin, and it writes the legacy scalar streaming key instead of the canonical nested mode path.

## What This Changes

The PR adds a Telegram-only `/settings` inline keyboard panel for access-control and streaming settings, with callback parsing, config mutation/auth helpers, native command and callback handler changes, shared command registry provider filtering, and tests.

## Best Possible Solution

Port the panel into `extensions/telegram/src` using the current Telegram plugin command, menu, auth, and callback dispatch seams; preserve `nativeEnabled`, `configWrites`, DM-only, and command-auth parity; write canonical `streaming.mode` paths for default and named accounts; then add Telegram tests plus user-facing docs and changelog coverage before merge review.

## Reproduction Assessment

Yes for the merge-blocking integration defects: current-main static checks show no `src/telegram` tree or settings symbols, while the PR head adds `src/telegram/*` files and maps the streaming setting to scalar `streaming`. A live Telegram bot session is not needed to reproduce those code-path mismatches.

## Solution Assessment

No, this branch is not the best way to solve the issue as-is. The safer maintainable solution is a current-main port inside the Telegram plugin with canonical config paths and renewed auth review for the config-mutating callbacks.
