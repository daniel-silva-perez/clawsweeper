## Summary

Keep open: the feature is useful, but the PR is not merge-ready against current main because it targets the obsolete pre-plugin Telegram runtime path. Current main owns Telegram runtime under `extensions/telegram/src`, and the proposed settings modules and callback wiring are added under `src/telegram`, so the panel would not be integrated with the maintained command and callback paths.

## What This Changes

The PR adds a Telegram-only `/settings` inline keyboard panel for viewing and changing DM policy, group policy, streaming mode, and allowlist state, with new callback parsing, config-write logic, native command wiring, and tests.

## Best Possible Solution

Implement the panel inside `extensions/telegram/src` using the current plugin command, callback, auth, and config-write seams; keep it DM-only and native-command/config-write gated, write canonical nested streaming paths, and add plugin tests plus docs/changelog coverage before merge review.

## Reproduction Assessment

Yes. Static current-main checks reproduce the blocking integration issue: there is no `src/telegram` runtime tree or settings symbol on main, while the PR head adds and wires the feature under `src/telegram/*`.

## Solution Assessment

No. The feature direction is reasonable, but this branch is not the best implementation path as-is; the maintainable solution is a current-main port inside the Telegram plugin with canonical `streaming.mode` writes and renewed privileged-surface review.
