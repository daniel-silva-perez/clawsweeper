## Summary

Keep this issue open. Current main still exposes global session maintenance thresholds plus a separate cron-run retention reaper, but it does not implement ordered per-pattern or per-type retention rules for `session.maintenance`; the proposed `rules` array remains outside the strict config contract. The KeepMyClaw comment is useful backup-retention context, and the open per-agent override PR is adjacent, but neither replaces a core policy for selectively pruning entries in `sessions.json`.

## What This Changes

Add ordered session maintenance retention rules that match session-key patterns or session types and override `pruneAfter` before falling back to the global threshold.

## Best Possible Solution

Add an additive core `session.maintenance.rules` contract after maintainer design approval, with a documented matcher, ordered first-match retention, global `pruneAfter` fallback, strict schema/help/generated-doc coverage, and pruning call sites that preserve existing active/running-session and cron-specific protections.

## Reproduction Assessment

Not applicable as a runtime bug, but the missing feature is high-confidence by inspection: the strict config schema has no `rules` field and the pruning resolver/call sites use one global `pruneAfterMs` threshold.

## Solution Assessment

Unclear pending maintainer design. The requested direction is plausible, but the safest implementation should first settle the public matcher shape and deletion semantics rather than adding ad hoc string matching in the pruning loop.
