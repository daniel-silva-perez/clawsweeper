## Summary

Keep this issue open. Current main still implements `session.maintenance` as a global retention policy: the public type and strict schema expose no `rules`/`match` field, the resolver collapses retention to one `pruneAfterMs`, and save/load/cleanup paths pass that single threshold into uniform pruning. `cron.sessionRetention` is adjacent, but it only targets cron run-session keys and does not provide ordered per-pattern retention for subagents, Telegram topics, DMs, or arbitrary session keys. The KeepMyClaw comment is useful backup context, not a replacement for the requested core retention rule contract.

## Best Possible Solution

Keep the issue open for maintainer design/approval. If accepted, implement an additive `session.maintenance.rules` contract with ordered first-match retention by session key, global `pruneAfter` fallback, strict schema/type/help/docs coverage, and pruning paths that preserve active/running session protections and existing cron-specific behavior.
