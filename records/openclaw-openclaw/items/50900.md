## Summary

Keep this issue open. Current main still has one global session maintenance age/count policy plus a separate cron-run reaper; there is no ordered per-pattern or per-type retention-rule surface, and the strict config schema would reject the proposed `session.maintenance.rules` shape.

## What This Changes

Add ordered per-pattern or per-type `session.maintenance` retention rules so ephemeral session keys can expire sooner while unmatched sessions fall back to the global `pruneAfter`.

## Best Possible Solution

Add an additive core `session.maintenance.rules` contract after maintainer approval, with a documented matcher, ordered first-match retention, global `pruneAfter` fallback, strict schema/generated-doc/help/docs coverage, and pruning call sites that preserve active/running-session safeguards plus the existing cron-specific behavior.
