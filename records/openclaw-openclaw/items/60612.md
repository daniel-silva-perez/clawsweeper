## Summary

Keep open. Current main still warns for version-managed Gateway Node paths and only repairs to a supported system Node when one exists; it does not expose a persistent Node-path selector or scoped doctor acknowledgement for intentional NVM/ASDF use. The ASDF follow-up makes this a broader Gateway runtime policy gap, not only one stale NVM launchd case.

## What This Changes

The issue asks for a permanent Gateway Node path setting or a way to suppress doctor warnings when NVM/ASDF Node is intentionally used.

## Best Possible Solution

Add an explicit, documented Gateway service runtime policy that keeps the warning by default but lets intentional users either pin a supported Node executable through the install/repair path or acknowledge this specific doctor warning. The chosen design should cover NVM and ASDF consistently, preserve automatic migration to system Node when available, and document when version-manager use is safe to ignore versus fragile.

## Reproduction Assessment

Yes. A service command whose executable is under `~/.nvm` or `~/.asdf` reproduces the audit warnings in current main, and if no supported Homebrew/system Node candidate exists doctor has no migration path beyond printing the install-system-Node guidance.

## Solution Assessment

No current solution is complete. The wrapper option is useful for externally managed launch behavior, but the narrow maintainable fix should be an explicit runtime pin or explicit warning acknowledgement rather than relying on manual plist edits that doctor/install can overwrite.
