## Summary

Keep open. Current main still reports version-managed Gateway Node paths such as NVM/ASDF, and doctor only migrates away from them when a supported system Node is available. The related closed launchd/path-drift reports are adjacent, but they do not provide a first-class persistent Node-path setting or targeted warning suppression for intentional version-manager use.

## Best Possible Solution

Add an explicit, documented Gateway service runtime policy: either a supported persistent Node executable setting/CLI flag used by `gateway install` and doctor repair, or a scoped doctor acknowledgement such as an allow/suppress setting for intentional version-manager runtimes. The implementation should live in the Gateway service install/audit path, keep the warning by default, and document when the warning is safe to ignore versus when to migrate to Homebrew/system Node.

## Risks / Open Questions

- The Gateway may work correctly under NVM/ASDF today; the existing warning is about service fragility after version-manager updates, so any suppression should be explicit and documented rather than silently hidden.
- `--wrapper` can be a workaround for custom service launch behavior, but it is not documented as a persistent Node-path selector and would not answer the targeted suppression request cleanly.
