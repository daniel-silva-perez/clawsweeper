## Summary

Keep this PR open. Current main still has only the older 401-specific Telegram sendChatAction backoff/suspension state machine; it does not cool down transient network, 429, or 5xx sendChatAction failures. The PR is still a focused implementation candidate for the reported Telegram retry/log-spam problem, and the provided timeline shows related alternate PRs were closed while a maintainer pushed a narrow repair to this branch to keep the original contributor path canonical. I did not find supply-chain/security-sensitive drift in the PR diff, but there is one functional review point: the PR narrows the exported rate-limit helper used by send and webhook paths, so maintainers should confirm that broader behavior change is intentional or scope the narrowing to chat-action handling.

## Best Possible Solution

The best path is to either land the focused Telegram handler cooldown with its regression tests, or make a small follow-up adjustment so the rate-limit-helper narrowing is scoped to sendChatAction if maintainers want to preserve current send/webhook retry behavior.
