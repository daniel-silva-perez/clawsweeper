## Summary

Keep this PR open. Current main and the latest release still lack the transient Telegram sendChatAction cooldown, but the current PR diff has one blocking correctness issue: it narrows a shared Telegram rate-limit helper that is used outside chat-action handling.

## What This Changes

The PR adds Telegram sendChatAction transient network/rate-limit/5xx cooldown state and regression tests, injects runtime backoff helpers for tests, narrows Telegram rate-limit detection to structured error_code, and adds a changelog entry.

## Best Possible Solution

Land a version that keeps the dedicated sendChatAction transient cooldown and regression coverage while preserving the shared Telegram rate-limit helper behavior used by send and webhook retry paths. If chat-action needs stricter classification, make that strictness local to the sendChatAction handler instead of narrowing the exported helper.

## Reproduction Assessment

Yes. The original bug is reproducible with a unit-level sendChatAction handler that throws transient ETIMEDOUT/429/5xx errors; current main’s non-401 test already proves repeated transient failures are retried rather than cooled down. The PR regression is also reproducible by calling isTelegramRateLimitError on the current wrapped retry_after-without-error_code fixture.

## Solution Assessment

No, not as currently diffed. The sendChatAction cooldown is the right narrow area, but the shared rate-limit-helper narrowing is broader than the stated scope; preserving the helper and adding a chat-action-local stricter check is safer.
