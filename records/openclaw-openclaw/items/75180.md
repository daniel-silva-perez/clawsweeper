## Summary

Keep this PR open, but it still needs changes before merge. The updated head fixes the earlier public type placement, but CI now shows the re-exported `BlockReplyResult` is not locally bound in the pipeline and the widened callback return type was not propagated to the direct block-delivery handler; source review also shows the production callback still treats payload media as confirmation before queued dispatcher delivery is actually known.

## What This Changes

The PR adds a `BlockReplyResult` return contract for block replies, changes sent-media tracking in the block reply pipeline, wires dispatch-from-config to return sent media URLs, adds focused pipeline tests, and adds a changelog fix entry for the linked mixed text-plus-media Telegram bug.

## Best Possible Solution

Keep the delivery-aware direction, but make the callback result contract compile everywhere and only record media URLs when the block path has a real accepted or delivered result for those URLs. The final shape should preserve mixed text-plus-media attachments when block delivery did not confirm the media, while retaining media-only duplicate suppression and the changelog entry for the user-facing fix.

## Reproduction Assessment

Yes. The PR build failure is reproducible from current head CI annotations, and the original media bug has a high-confidence source path on current main: mixed block media is recorded as sent, then final payload media dedupe strips matching URLs.

## Solution Assessment

No. The proposed direction is maintainable, but this implementation is not yet the best fix because it does not compile and its production callback still derives confirmation from the normalized payload rather than the dispatcher delivery result.

Passing `params.opts.onBlockReply` into that helper now fails strict type checking, so the helper signatures need the same widened return contract or an adapter that intentionally discards the result.
  - confidence: 0.96
- **[P2] Return media confirmation only after accepted delivery:** `src/auto-reply/reply/dispatch-from-config.ts:1285-1287`
  - body: `confirmedMedia` is computed from `normalizedPayload` before the queued dispatcher path is known to accept or deliver the block reply, and the return value ignores `dispatcher.sendBlockReply(normalizedPayload)`. If that enqueue returns false, is cancelled by `beforeDeliver`, or later fails, the pipeline will still mark the media as sent and final dedupe can drop the attachment this PR is trying to preserve.
