## Summary

Keep this PR open. It targets a real Telegram mixed text-and-media delivery bug, but the patch uses text presence as a proxy for delivery state and can preserve final media after a successful mixed block send, causing the same attachment to be resent.

## What This Changes

The PR changes block reply sent-media tracking so only media-only block payloads are recorded for final-reply media dedupe, and updates the block pipeline unit expectation.

## Best Possible Solution

Make block media tracking delivery-aware: media actually delivered by the block path should still suppress final media, while media that only appeared beside previewed or text-only streamed content should survive into the final reply. Cover that contract in buildReplyPayloads and Telegram lane/media tests.

## Reproduction Assessment

Yes. A source-level reproduction is a successful block-streamed payload with both text and media followed by final payload construction for the same text+media reply; this PR leaves getSentMediaUrls empty for that successful mixed block send, so the final media can be delivered again.

## Solution Assessment

No. The current PR fixes the reported missing-attachment path by not tracking mixed media, but that is not the narrowest maintainable fix because it removes the duplicate guard for mixed media that was actually delivered.
