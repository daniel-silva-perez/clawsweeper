## Summary

Keep open. The latest head appears to address the earlier type-propagation blockers, but the production dispatch path still reports media as confirmed from the normalized payload rather than from accepted or completed block delivery, so the PR can still feed unsent media into final-reply dedupe.

## What This Changes

The PR widens block-reply callbacks to return sent-media confirmation, changes block reply media tracking, updates dispatch/delivery types, adds focused pipeline tests, and adds a changelog fix for the linked mixed text-plus-media reply bug.

## Best Possible Solution

Keep the delivery-aware direction, but only add media URLs to the block pipeline’s sent set when the channel callback can prove those URLs were delivered. For the queued dispatcher path, either return no sent-media confirmation and rely on the legacy media-only fallback, or extend the dispatcher to surface real post-delivery metadata, with a regression test for rejected or cancelled block media.

## Reproduction Assessment

Yes. The original bug has a source-level reproduction through mixed text-plus-media block streaming followed by final media dedupe, and the PR regression can be reproduced with a dispatch-from-config unit test where dispatcher.sendBlockReply returns false or delivery is cancelled while the callback still returns sentMediaUrls.

## Solution Assessment

No. The contract change is a maintainable shape, but this implementation is not yet the best fix because the production callback still derives confirmation from payload shape instead of accepted or completed delivery.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Only confirm media after block delivery is accepted:** `src/auto-reply/reply/dispatch-from-config.ts:1292-1294`
  - body: The callback computes `confirmedMedia` from `normalizedPayload` and returns it even when the queued dispatcher path does not accept or later deliver the block reply. `createBlockReplyPipeline` treats any returned `sentMediaUrls` as delivered and final payload filtering can then strip an attachment that never entered the dispatcher, so gate this result on real acceptance/delivery or return no confirmation from the queued path.
  - confidence: 0.88
