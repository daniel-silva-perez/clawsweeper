## Summary

Keep open. The linked bug and contribution are valid, but the current patch is not merge-ready because the media-only guard removes duplicate suppression for mixed text+media block payloads that were actually delivered, and the user-facing fix is missing the required changelog entry.

## What This Changes

The PR changes block reply sent-media tracking so mixed text+media block payloads no longer populate final-reply media dedupe state, while media-only block payloads still do.

## Best Possible Solution

Make block media tracking delivery-aware: media that was actually sent by the block path should still suppress repeated final media, while media that only appeared beside previewed or text-only streamed content should survive into the final reply. Add regression coverage across `buildReplyPayloads` and the Telegram lane/media delivery path, plus a single-line Unreleased Fixes changelog entry.

## Reproduction Assessment

Yes for the PR regression. A source-level reproduction is a successful block-streamed payload with both text and media followed by final payload construction for the same text+media reply; with the PR guard, `getSentMediaUrls()` stays empty and the final media can be delivered again.

## Solution Assessment

No. The proposed media-only guard fixes the reported drop for one path by not tracking mixed media, but the safer maintainable fix is to track based on whether block delivery actually sent the media, not whether the payload also had text.
