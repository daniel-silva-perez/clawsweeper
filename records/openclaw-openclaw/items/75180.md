## Summary

Keep open. The PR targets a real current-main bug, and the delivery-aware callback contract is the right direction, but the latest head still confirms media from payload content on the queued dispatcher path before actual delivery is accepted or completed, so final dedupe can still strip an unsent attachment.

## What This Changes

The PR widens block-reply callbacks to return confirmed sent media URLs, updates pipeline/dispatch/reply-delivery types, adds focused block-reply tests, and adds a changelog fix for mixed text-plus-media replies.

## Best Possible Solution

Keep the delivery-aware contract, but make confirmation semantically exact: direct channel sends should return sentMediaUrls only after the awaited send succeeds, and queued dispatcher sends should either return no confirmation or expose real post-acceptance/post-delivery media metadata before final dedupe trusts it.

## Reproduction Assessment

Yes. Current main has a source-level reproduction path: enqueue a mixed text-plus-media block payload, observe getSentMediaUrls include that media, then final payload dedupe strips the same media while preserving text. The remaining PR bug can be reproduced with a dispatch-from-config test where sendBlockReply returns false or beforeDeliver cancels a media block while the callback would otherwise return sentMediaUrls.

## Solution Assessment

No. The proposed API shape is maintainable, but deriving delivery confirmation from normalized payload content on the queued dispatcher path is not the best fix; the safer alternative is to omit queued-path confirmation until the dispatcher can report actual accepted or delivered media.
