## Summary

Keep open. The delivery-aware API direction is sound, but the latest PR head still reports media as delivered from payload shape on the queued dispatcher path, before the dispatcher has accepted, cancelled, failed, or completed channel delivery; that can preserve the same class of final-reply media loss for undelivered block media.

## What This Changes

The branch widens block-reply callbacks to return media-delivery confirmations, updates block media tracking and dispatch typing, adds focused pipeline tests, and adds a changelog fix entry for mixed text-plus-media replies.

## Best Possible Solution

Keep the delivery-aware contract, but make the confirmation semantically exact: direct channel handlers should return media URLs only after a successful awaited send, and queued dispatcher delivery should either expose real post-delivery media metadata or leave sent-media confirmation undefined so final dedupe does not strip unsent attachments.

## Reproduction Assessment

Yes. The original issue has a high-confidence source-level reproduction through mixed text-plus-MEDIA block streaming followed by final media dedupe; the remaining PR bug can be covered with a dispatch-from-config test where sendBlockReply returns false or beforeDeliver cancels a block with media while the callback would otherwise return sentMediaUrls.

## Solution Assessment

No. The proposed API is a good shape, but this implementation still derives delivery confirmation from normalized payload content on the queued path instead of from accepted or completed delivery.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Confirm media only after queued block delivery is accepted:** `src/auto-reply/reply/dispatch-from-config.ts:1294`
  - body: The callback returns sentMediaUrls based on normalizedPayload even on the dispatcher path. sendBlockReply can decline pre-queue, and beforeDeliver or deliver can cancel or fail later; createBlockReplyPipeline will still treat that media as delivered, so final dedupe can strip an attachment that never reached the channel. Gate this confirmation on real delivery/acceptance, or return no confirmation from the queued path.
