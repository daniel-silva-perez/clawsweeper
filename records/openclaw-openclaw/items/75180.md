## Summary

Keep open. The linked bug looks real and the PR targets the right area, but the proposed condition is too coarse: mixed text+media block payloads can already be delivered by the block path, and removing those URLs from block sent-media tracking can let the final payload resend the same attachment instead of preserving only media that was not actually sent.

## What This Changes

The PR changes the block reply pipeline to stop recording sent media URLs for mixed text+media block payloads and updates the existing sent-media pipeline test expectation.

## Best Possible Solution

Track whether media was actually delivered by the block path instead of using text presence as the proxy. Media already sent by successful block delivery should still suppress final media, while media that only appeared beside streamed/previewed text and was not delivered should survive into the final reply; add a regression through buildReplyPayloads or Telegram dispatch for the text plus MEDIA first-reply path.

## Reproduction Assessment

Yes. The source-level reproduction is a successful block-streamed payload with both text and media followed by the same final payload: current finalization preserves the mixed payload unless a text-only equivalent was sent, then relies on getSentMediaUrls to remove already-sent block media.

## Solution Assessment

No. The PR fixes the reported drop by stopping mixed media from entering sentMediaUrls, but that also removes the only later guard against duplicating mixed media that block delivery actually sent; a safer fix needs delivery-aware media tracking plus finalization coverage.
