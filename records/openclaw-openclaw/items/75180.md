## Summary

Keep this PR open, but it needs changes before merge. The underlying mixed text-plus-media block reply bug is still real on current main, but this PR head currently fails TypeScript because `BlockReplyResult` is used in the public options type without being defined or imported there, and the production callback still confirms media from payload shape before real delivery or dispatcher acceptance is known.

## What This Changes

The PR adds a block-reply callback result for sent media URLs, updates the block reply pipeline and dispatch-from-config path to use it, adds block reply pipeline tests, and adds a changelog fix entry for the linked Telegram mixed text-plus-media bug.

## Best Possible Solution

Keep the delivery-aware direction, but make the contract compile and mean what it says: define or import the result type from a safe leaf module, return sent media only after direct delivery succeeds or the dispatcher has a real accepted/delivered result, preserve final media when block delivery did not send it, and keep focused regression coverage plus the changelog entry.

## Reproduction Assessment

Yes. The PR build failure is reproducible from PR-head CI annotations and static inspection of the unresolved `BlockReplyResult` type. The original media bug also has a high-confidence source path through current main's block sent-media tracking followed by final payload media dedupe.

## Solution Assessment

No. The proposed direction is reasonable, but this implementation is not the best fix because it breaks the public type surface and still treats normalized payload content as delivery confirmation.
