## Summary

Current main still matches the reported LINE delivery bug: Markdown tables are extracted into Flex messages, text chunks consume the Reply API token, and generated rich/table messages are then forced through Push API with `allowReplyToken=false`. The directly related fix PR #65673 is closed and unmerged, so the issue remains valid and narrow enough for a repair PR.

## What This Changes

The issue asks for LINE auto-replies containing text plus generated table Flex messages to use the quota-free Reply API batch when reply slots remain, instead of dropping the table through a separate Push API call after push quota is exhausted.

## Best Possible Solution

Fix this in the LINE plugin delivery path. Extend `sendLineReplyChunks` or `deliverLineAutoReply` so text plus generated table/code Flex messages can be bundled into the same Reply API call when reply slots remain and there is no quick-reply ordering conflict, then push only overflow messages. Add regression coverage for text plus extracted Markdown table using one `replyMessageLine` call and no `pushMessagesLine`, plus overflow and quick-reply behavior.

## Risks / Open Questions

- A minimal `false` to `true` change after `sendLineReplyChunks` is not enough because the text reply call already marks the reply token used; the fix needs to bundle extras before consuming the token or return consumed-extra state.
- Quick replies have existing ordering behavior where rich messages are pushed before quick-reply text so quick replies remain visible; a fix should preserve that behavior or change it only with maintainer intent and tests.
- LINE Reply API batches are capped at five messages, so bundled Flex/media messages must fill only unused reply slots and push overflow messages.
