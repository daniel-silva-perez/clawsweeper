## Summary

Keep this issue open. Current main still matches the reported LINE failure path: Markdown tables are extracted into Flex messages, the text chunk is sent through the Reply API, and the generated rich/table message is then sent with `allowReplyToken=false`, forcing the Push API. The related fix PR #65673 is closed and unmerged, and current tests still assert the push-after-reply behavior.

## Best Possible Solution

Fix this in the LINE plugin delivery path. Extend the text reply helper or `deliverLineAutoReply` so text plus generated table/code Flex messages can be bundled into the same Reply API call when there are unused reply slots and no quick-reply ordering conflict, then push only overflow messages. Add a regression test that a text response with an extracted Markdown table uses one `replyMessageLine` call for both the text and Flex message and does not call `pushMessagesLine`; keep existing quick-reply ordering covered.

Current `deliverLineAutoReply` sends text via `sendLineReplyChunks`, then forces `richMessages`/`mediaMessages` through Push API with `allowReplyToken=false`; this drops table Flex messages when LINE Push API quota is exhausted. Push only overflow messages. Preserve the existing quick-reply ordering behavior unless a maintainer explicitly decides to change it. Do not move LINE behavior into core or change unrelated channel/plugin APIs. Add regression coverage showing a text response with an extracted Markdown table sends both the text and generated Flex message through one `replyMessageLine` call and does not call `pushMessagesLine`; also cover overflow and existing quick-reply behavior.
