## Summary

Keep open. Current main does not yet implement the PR's intended sudo rejection: systemd unit paths still resolve from the process home while sudo systemctl calls can target the sudo caller's user scope. The PR has useful direction and relevant reviewer discussion, but the latest diff has blocking test issues and an unrelated build sourcemap change, so it needs updates before merge rather than cleanup closure.

## What This Changes

The PR adds a Linux sudo guard to `openclaw gateway install`, adds post-install linger and service-not-running diagnostics, adds `failIfSudoInstall` tests, and enables node build sourcemaps.

## Best Possible Solution

Update the PR as a narrow current-main fix: reject sudo-to-root gateway user-service installs before writing a systemd user unit, keep `loginctl enable-linger` as the separate privileged step, update every affected test mock/assertion, remove the unrelated build config change, and preserve current systemctl user-scope hardening for restart/status paths.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: the PR's new shared test asserts a message that the new helper does not emit, and current `install.test.ts` mocks `./shared.js` without the newly imported export. A live Linux sudo/systemd run would be useful for product proof, but it is not needed to reproduce these patch failures.

## Solution Assessment

No. Rejecting sudo at the CLI boundary is a maintainable direction, but this patch is not the best current implementation because it is stale, leaves tests broken, prints custom-unit diagnostics for the wrong service name, and mixes in an unrelated build artifact change.
