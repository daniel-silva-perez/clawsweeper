## Summary

Current main still does not implement persistent multi-turn webhook hook sessions. Direct and mapped agent hooks can carry a stable `sessionKey`, but dispatch still builds an isolated cron-style job and `runCronIsolatedAgentTurn` still passes `forceNew: true` for isolated jobs, so `resolveCronSession` cannot reuse the existing transcript session id. The hook payload, mapping transform/config types, and strict schema still have no `sessionMode`, `reuseSession`, `targetExistingSession`, or equivalent control. Prior implementation attempts #15084, #27123, and #27193 are closed unmerged, while #42621 was closed as superseded by this issue rather than as an implementation.

## Best Possible Solution

Keep this issue open and implement a backward-compatible hook-session persistence option in core, most likely `sessionMode: "isolated" | "persistent"` for direct `/hooks/agent` payloads, hook mappings, and transform results. Default to `isolated`, preserve all current session-key security gates, and only pass `forceNew: false` for the explicit persistent hook path so stable `sessionKey` values reuse fresh existing session ids without changing isolated cron behavior.
