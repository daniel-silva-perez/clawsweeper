## Summary

Keep open. Current main still does not implement persistent multi-turn webhook hook sessions: direct and mapped hook agent dispatch can carry a stable sessionKey, but dispatch still wraps the turn as an isolated cron-style job and the runner still forces a fresh session id. The closed related PRs contain useful prior implementation attempts, especially #27193, but they were not merged and no equivalent option exists in the current hook payload, transform, mapping, or strict config schema.

## What This Changes

Webhook agent hooks should support multi-turn conversation reuse when a stable sessionKey is provided.

## Best Possible Solution

Implement an explicit backward-compatible persistent hook-session option in core: add sessionMode: "isolated" | "persistent" for direct /hooks/agent payloads, hook mappings, and transform results; default to isolated; preserve hook sessionKey security policy; and only pass forceNew: false for the explicit persistent hook-agent path so stable sessionKey values reuse fresh existing session ids without changing cron isolation.

## Repair Work Prompt

Fix persistent multi-turn webhook hook sessions for this issue. Current main lets direct and mapped hook agent dispatch carry a stable sessionKey, but dispatch still creates an isolated synthetic CronJob and runCronIsolatedAgentTurn still passes forceNew: true, so each invocation gets a fresh session id. Implement an explicit backward-compatible option, not automatic reuse: add sessionMode: "isolated" | "persistent" for direct /hooks/agent payloads, hook mappings, and transform results; default to isolated; thread the normalized mode through HookAgentPayload/HookAgentDispatchPayload, HookAction/HookTransformResult, mapping normalization/merge, request-handler replay scopes, and dispatchAgentHook. Preserve all existing hook security gates for sessionKey, allowedSessionKeyPrefixes, allowedAgentIds, token auth, external content wrapping, and idempotency. In src/gateway/server/hooks.ts, keep sessionTarget isolated by default but make the explicit persistent hook-agent path cause runCronIsolatedAgentTurn/resolveCronSession to reuse a fresh existing session id, for example by passing forceNew false only for persistent hook sessions. Do not change cron job isolation semantics or make stable keys automatically persistent. Update config types, strict zod schema, generated config schema/help/docs as required, and user docs at docs/automation/cron-jobs.md. Add focused tests for direct payload mode, mapping config mode, transform override mode, default isolation, forceNew behavior, and schema rejection/acceptance. Add a changelog entry in the active version because this is user-facing behavior/API; credit Thanks @marieldejesus12 and credit @ryanrhughes too if the implementation reuses prior PR ideas/code from #15084/#27123/#27193.
