## Summary

Keep open. Current main still routes direct and mapped hook-agent requests through the isolated cron-style runner, so stable hook session keys still reach session resolution with `forceNew: true` and cannot reuse an existing transcript session id. The related implementation PRs are useful prior art, especially #27193, but they are closed unmerged and do not supersede the remaining work.

## What This Changes

Request webhook hook-agent sessions with a consistent `sessionKey` to optionally reuse the existing session for multi-turn conversation history.

## Best Possible Solution

Add a documented, backward-compatible hook option such as `sessionMode: "isolated" | "persistent"` for direct `/hooks/agent` payloads, hook mappings, and transform results. Keep `isolated` as the default, propagate the mode through dispatch, and pass `forceNew: false` only for explicit persistent hook-agent runs with allowed stable session keys.

## Reproduction Assessment

Yes. A stable hook `sessionKey` goes through `dispatchAgentHook`, gets `sessionTarget: "isolated"`, and reaches `resolveCronSession` with `forceNew: true`; the current session tests assert that `forceNew: true` creates a new session id despite an existing entry.

## Solution Assessment

Yes. An explicit `sessionMode` opt-in is the narrowest maintainable fix because it enables the requested multi-turn hook behavior without weakening default isolation semantics for existing webhooks or cron-style runs.
