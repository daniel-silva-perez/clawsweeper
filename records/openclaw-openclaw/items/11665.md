## Summary

Keep open. Current main still lacks an explicit persistent hook-session mode: direct and mapped hook agent dispatch can carry a stable sessionKey, but the dispatch path still constructs an isolated cron-style job and the runner still forces a fresh session id for isolated jobs. The closed related PRs contain useful prior attempts, especially #27193, but they were not merged and no equivalent option exists in the current payload, mapping, transform, or strict config schema.

## What This Changes

Request to let webhook hook agent runs reuse an existing conversation when a stable sessionKey is supplied, preferably through a backward-compatible persistent session mode.

## Best Possible Solution

Add an explicit sessionMode: "isolated" | "persistent" hook option for direct /hooks/agent payloads, hook mappings, and transform results. Default to isolated, document the config and request behavior, preserve the hook sessionKey security policy, and pass forceNew: false only for the explicit persistent hook-agent path so stable sessionKey values reuse fresh existing session ids without changing cron isolation.
