## Summary

Keep open. Current main still routes direct and mapped hook-agent requests through an isolated cron-style run, so a stable hook `sessionKey` still reaches session resolution with `forceNew: true` and cannot reuse the existing transcript session id. The related implementation PRs are useful prior art, especially #27193, but they are closed unmerged and do not supersede the remaining work.

## What This Changes

Webhook hook-agent sessions should support multi-turn transcript reuse when callers provide a stable `sessionKey`.

## Best Possible Solution

Add a documented, backward-compatible hook option such as `sessionMode: "isolated" | "persistent"` for direct `/hooks/agent` payloads, hook mappings, and transform results. Keep `isolated` as the default, propagate the mode through dispatch, and pass `forceNew: false` only for explicit persistent hook-agent runs with allowed stable session keys.

## Reproduction Assessment

Yes. The source path is enough to reproduce at high confidence: a stable hook `sessionKey` goes through `dispatchAgentHook`, gets `sessionTarget: "isolated"`, and `resolveCronSession` is called with `forceNew: true`, which current tests assert creates a new session id despite an existing entry.

## Solution Assessment

Yes. An explicit `sessionMode` opt-in is the narrowest maintainable fix because it enables the requested multi-turn hook behavior without changing the default isolation semantics for existing webhooks or cron-style runs.
