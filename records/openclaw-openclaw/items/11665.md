## Summary

Keep open. Current main still routes direct and mapped hook-agent requests through an isolated cron-style job, and that path still forces a fresh session id even when the caller supplies the same stable hook session key. The reusable session helper exists only when `forceNew` is false, and the hook payload, mapping, transform, and strict config schema surfaces still do not expose an explicit persistent hook-session mode; the closed related PRs are useful prior art but were not merged.

## What This Changes

Allow webhook hook-agent requests with a stable `sessionKey` to opt into reusing the existing session transcript for multi-turn conversation history.

## Best Possible Solution

Add an explicit `sessionMode: "isolated" | "persistent"` option for direct `/hooks/agent` payloads, hook mappings, and transform results. Keep `isolated` as the default, update config types/schema/docs, and only let the explicit persistent hook-agent path call the session resolver with reuse enabled; do not broaden the change into general existing-main-session delivery or cross-channel binding unification.
