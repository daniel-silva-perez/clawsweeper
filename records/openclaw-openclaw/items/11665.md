## Summary

Current `main` still does not provide multi-turn webhook hook sessions from a stable `sessionKey`: direct and mapped hook-agent requests are dispatched as isolated cron-style runs, and isolated runs still force a fresh transcript session id. The prior related PRs are useful prior art, especially #27193, but they are closed unmerged and do not supersede the remaining work.

## What This Changes

Allow webhook hook-agent runs with a consistent `sessionKey` to opt into reusing an existing session id so multi-turn hook conversations can keep transcript history.

## Best Possible Solution

Add a documented, backward-compatible hook option such as `sessionMode: "isolated" | "persistent"` for direct `/hooks/agent` payloads, hook mappings, and transform results. Keep `isolated` as the default, propagate the mode through dispatch, and only pass `forceNew: false` for explicit persistent hook-agent runs whose stable session keys satisfy the existing hook session-key policy.

## Repair Work Prompt

Preserve current behavior by default: existing hooks must remain isolated and continue to force a fresh session id. When `sessionMode` is explicitly `persistent`, propagate that mode through the hook request handler and dispatch path so `runCronIsolatedAgentTurn` reaches `resolveCronSession` with `forceNew: false` for stable allowed hook session keys. Respect existing `hooks.allowRequestSessionKey` and `hooks.allowedSessionKeyPrefixes` policy; do not add broad target-existing-session behavior or weaken hook auth/session-key validation. Use #27193, #27123, and #15084 as prior art, but build against current `main`. Update config types, strict zod schema, hook normalization/merge logic, docs, generated config docs if required, and focused tests. Add a changelog entry only if repo policy treats this as user-facing behavior for the active release, with proper non-bot attribution.
