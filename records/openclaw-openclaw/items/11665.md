## Summary

Keep open. Current main still routes direct and mapped hook-agent requests through a synthetic isolated cron job, so a stable hook `sessionKey` still reaches the cron session resolver with `forceNew: true` and cannot reuse the existing transcript session. The closed related PRs are useful prior art but were not merged and do not supersede the remaining work.

## What This Changes

The issue requests stable webhook hook `sessionKey` values to reuse an existing transcript session for multi-turn hook conversations instead of minting a fresh isolated session every invocation.

## Best Possible Solution

Add a documented, backward-compatible hook option such as `sessionMode: "isolated" | "persistent"` for direct `/hooks/agent` payloads, hook mappings, and transform results. Keep `isolated` as the default, and only pass `forceNew: false` for the explicit persistent hook-agent path so stable hook session keys can reuse fresh existing session IDs without weakening cron isolation.

Preserve `isolated` as the default. Thread the mode through hook normalization, mapping resolution/merge, request dispatch, and `dispatchAgentHook` so only explicit persistent hook-agent runs call `runCronIsolatedAgentTurn` in a way that lets `resolveCronSession` reuse a fresh existing session id.
