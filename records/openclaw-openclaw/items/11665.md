## Summary

Keep open. Current main still hard-codes hook-agent dispatch as an isolated cron-style job, so stable hook `sessionKey` values still reach `resolveCronSession` with `forceNew: true` and cannot reuse the existing transcript session. The reuse helper exists, but the direct payload, mapping, transform, and strict config schema surfaces still lack an explicit persistent hook-session option; the related implementation PRs are closed unmerged prior art rather than an implemented fix.

## What This Changes

The issue requests multi-turn webhook hook sessions by reusing an existing session when a hook supplies the same stable `sessionKey`.

## Best Possible Solution

Add a documented, backward-compatible hook option such as `sessionMode: "isolated" | "persistent"` for direct `/hooks/agent` payloads, hook mappings, and transform results. Keep isolated behavior as the default, and only pass `forceNew: false` for the explicit persistent hook-agent path so stable hook session keys can reuse fresh existing session IDs without weakening cron isolation.
