## Summary

Keep open. Current main still sends direct and mapped hook-agent requests through the isolated cron-style runner, and that path still forces a fresh session id even when the caller supplies the same stable hook session key. The reusable session helper exists, but the hook payload, mapping, transform, and strict config schema surfaces still have no explicit persistent-session mode to reach it. The closed related PRs are useful prior art, especially #27193, but they were not merged and do not supersede the remaining work.

## What This Changes

The issue requests multi-turn webhook hook support so repeated hook-agent calls with a stable `sessionKey` can reuse the existing transcript session instead of creating a fresh isolated session each time.

## Best Possible Solution

Add an explicit `sessionMode: "isolated" | "persistent"` hook option for direct `/hooks/agent` payloads, hook mappings, and transform results. Keep `isolated` as the default, document the security model, and only pass `forceNew: false` for the explicit persistent hook-agent path so stable hook session keys can reuse fresh existing session ids without weakening cron isolation.
