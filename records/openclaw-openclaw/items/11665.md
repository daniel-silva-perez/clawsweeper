## Summary

Keep open. Current main still routes hook-agent requests through an isolated cron-style run, so a stable hook session key still reaches `resolveCronSession` with `forceNew: true` and cannot reuse the existing transcript session id. The recent maintainer and ClawSweeper comments remain consistent with the source, and the closed related PRs are useful prior art but were not merged.

## What This Changes

The issue requests persistent multi-turn webhook hook sessions when callers provide a stable hook session key.

## Best Possible Solution

Add a documented, backward-compatible hook option such as `sessionMode: "isolated" | "persistent"` for direct `/hooks/agent` payloads, hook mappings, and transform results. Default to isolated, preserve existing session-key allowlist behavior, and only let the explicit persistent hook-agent path call session resolution without `forceNew`.

## Reproduction Assessment

Yes. A direct or mapped hook with a stable `sessionKey` is normalized and dispatched as an isolated cron-style job, which passes `forceNew: true`; the existing session unit test confirms forced-new resolution creates a different session id despite a fresh stored entry.

## Solution Assessment

Yes. The explicit `sessionMode` opt-in is the narrowest maintainable direction because it enables the requested multi-turn webhook behavior without weakening default hook or cron isolation; automatic reuse by key shape would be less safe.
