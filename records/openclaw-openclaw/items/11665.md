## Summary

Current main still does not implement persistent multi-turn webhook hook sessions. Direct and mapped agent hooks can accept or derive a stable sessionKey, but dispatch still wraps them as isolated cron-style jobs, and the isolated runner still passes forceNew: true, so resolveCronSession mints a fresh sessionId instead of reusing the stored one. There is also no sessionMode/reuseSession/targetExistingSession field in hook payloads, mappings, transform results, or the strict hook mapping schema.

## Best Possible Solution

Keep this issue open and implement an explicit, backward-compatible persistent hook-session option in core. Add sessionMode: "isolated" | "persistent" for direct /hooks/agent payloads, hook mappings, and transform results; default to isolated; preserve all existing session-key and agent security gates; and only pass forceNew: false for the explicit persistent hook-agent path so stable sessionKey values reuse fresh existing session ids without changing cron isolation.
