## Summary

Keep open. Current main still routes direct and mapped hook-agent requests through a synthetic isolated cron job, so a stable hook session key still reaches session resolution with `forceNew: true` and cannot reuse an existing transcript session id. The closed related PRs are useful prior art, especially #27193, but they were not merged and do not supersede the remaining work.

## What This Changes

The issue requests multi-turn webhook hook sessions by reusing the existing transcript session id when a stable hook `sessionKey` is supplied.

## Best Possible Solution

Add a documented, backward-compatible hook option such as `sessionMode: "isolated" | "persistent"` for direct `/hooks/agent` payloads, hook mappings, and transform results. Default to isolated, propagate the explicit mode through dispatch, include it in idempotency scope, and set `forceNew: false` only for persistent hook-agent runs with allowed stable session keys.

## Reproduction Assessment

Yes. A direct or mapped hook with a stable `sessionKey` is normalized and dispatched as an isolated cron-style run, which passes `forceNew: true`; the existing session test confirms that forced-new resolution creates a different session id despite a fresh stored entry.

## Solution Assessment

Yes. The explicit `sessionMode` opt-in is the narrowest maintainable solution because it enables the requested multi-turn webhook behavior without weakening default hook or cron isolation; automatic reuse based only on key shape would be less safe.
