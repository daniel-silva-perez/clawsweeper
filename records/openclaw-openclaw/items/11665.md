## Summary

Keep open. Current main still routes direct and mapped hook-agent requests through isolated cron-style execution, so a stable hook `sessionKey` cannot reuse the existing transcript session. The reusable `resolveCronSession` branch exists, but the hook payload, mapping, transform, and strict config schema surfaces still lack an explicit persistent hook-session option; the related PRs are closed unmerged and remain useful prior art rather than an implemented fix.

## What This Changes

Allow hook/webhook agent invocations with a stable `sessionKey` to opt into reusing existing session IDs for multi-turn context instead of always creating fresh isolated sessions.

## Best Possible Solution

Add an explicit `sessionMode: "isolated" | "persistent"` option for direct `/hooks/agent` payloads, hook mappings, and transform results. Keep isolated behavior as the default, document the security model, and only pass `forceNew: false` for the explicit persistent hook-agent path so stable hook session keys can reuse fresh existing session IDs without weakening cron isolation.
