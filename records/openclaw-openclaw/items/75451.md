## Summary

Keep this PR open, but the patch is not mergeable as written. It adds a hidden process-wide plugin discovery metadata cache, which conflicts with the documented plugin cache boundary and regresses the current fresh-on-next-discovery behavior used by plugin mutation flows.

## What This Changes

The PR adds a module-level cache to `discoverOpenClawPlugins`, exports a cache-clearing helper, updates discovery tests to clear that state, and adds a performance changelog entry.

## Best Possible Solution

Keep `discoverOpenClawPlugins` fresh by default. If gateway startup is doing redundant work, pass the existing `PluginMetadataSnapshot`, `PluginLookUpTable`, or manifest registry through the hot call chain, or introduce a caller-owned short-lived snapshot with explicit invalidation rather than a hidden module cache.

## Reproduction Assessment

Yes. Current main already has a focused reproduction in `src/plugins/discovery.test.ts:1413`: delete a discovered plugin file and call discovery again with the same env; the PR changes that case to require manual cache clearing, which demonstrates the regression path.

## Solution Assessment

No. The PR optimizes the right hot path at the wrong layer; the maintainable fix is explicit metadata reuse owned by gateway/startup flows, not a process-global cache inside discovery.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep discovery fresh after inventory changes:** `src/plugins/discovery.ts:880-881`
  - body: This cache key only describes roots/env, so once a scan has run, later plugin installs, uninstalls, manifest edits, file removals, or permission changes under the same roots return the old `PluginDiscoveryResult`. Current main explicitly expects a removed plugin file to disappear on the next discovery call, and uninstall refreshes the persisted registry after deleting plugin files in the same process; with this return path that refresh can re-index stale plugins and skip discovery safety checks.
  - confidence: 0.91

## Security Review

Status: needs_attention

Summary: The diff introduces a cache around plugin code discovery and can skip permission and ownership checks on later reads.

Concerns:

- **[medium] Cache skips plugin path safety checks:** `src/plugins/discovery.ts:880`
  - body: Discovery performs world-writable and suspicious-ownership checks before accepting plugin code paths. Returning a cached result based only on roots/env means a later mode, owner, symlink, or file change under the same root can bypass those checks until manual cache clearing or process exit.
  - confidence: 0.84
