## Summary

Keep this issue open. Current main still has the reported Mattermost streaming contract mismatch: user docs advertise canonical preview streaming modes, but the Mattermost strict schema, public config type, generated config metadata, and inbound runtime still do not implement a Mattermost preview-mode branch.

## What This Changes

The issue reports that Mattermost documentation advertises `channels.mattermost.streaming` modes while the implementation only honors legacy block-streaming fields, causing draft-preview notification and visual-noise problems.

## Best Possible Solution

Keep this as the Mattermost streaming-contract tracker until PR #73252 or an equivalent Mattermost-local fix lands. The desired end state is root and per-account `streaming` schema/type/metadata support, preview-mode resolution in the inbound runtime, legacy `blockStreaming` compatibility, an `off` path that bypasses draft-preview callbacks/finalization, and focused schema/runtime/docs coverage; after that, evaluate whether final-notification behavior still needs a separate Mattermost UX option.

## Reproduction Assessment

Yes for the documented config mismatch: current main's docs advertise `channels.mattermost.streaming`, while the strict Mattermost schema/type/metadata omit it and the inbound monitor wires draft previews unconditionally. The notification symptom has reporter evidence, but no live Mattermost notification run was performed in this read-only review.

## Solution Assessment

Yes, the narrow maintainable direction is a Mattermost-local implementation of the documented canonical streaming mode contract, with PR #73252 as the visible candidate. A separate final-notification flag should be considered only after `streaming.mode` can actually disable or select the preview behavior.
