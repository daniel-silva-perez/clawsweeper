## Summary

Keep this issue open. Current main still has the Mattermost streaming contract mismatch: docs advertise `channels.mattermost.streaming` modes, while the strict Mattermost schema, public config type, generated metadata, and inbound runtime still omit a preview-mode implementation and wire draft previews unconditionally. Open PR #73252 remains the active narrow fix candidate, with adjacent PRs #73452 and #75256 also open, so this should not be closed or queued as duplicate repair work before maintainers land or replace that path.

## What This Changes

The issue asks OpenClaw to implement the documented Mattermost `channels.mattermost.streaming` modes and address draft-preview notification or visual-noise UX caused by final in-place edits.

## Best Possible Solution

Keep this as a Mattermost streaming-contract tracker until PR #73252 or an equivalent Mattermost-local fix lands. The desired end state is schema/type/metadata support for root and account `streaming`, preview-mode resolution through the inbound runtime, legacy `blockStreaming` compatibility, an `off` path that bypasses draft-preview callbacks/finalization, and focused schema/runtime/docs coverage; then the remaining final-notification UX can be evaluated against shipped mode behavior.

## Reproduction Assessment

Yes for the documented config mismatch: current main's docs advertise `channels.mattermost.streaming`, but the strict Mattermost schema/type/metadata omit it and the inbound monitor wires draft previews without a preview-mode branch. The notification symptom has reporter evidence, but no live Mattermost notification test was run in this read-only review.

## Solution Assessment

No current-main solution exists. The narrow maintainable path is a Mattermost-local implementation of documented `streaming.mode` behavior, with PR #73252 as the visible candidate; broader notification or block-splitting behavior should be layered only after the canonical config contract is real.
