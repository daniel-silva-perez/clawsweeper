## Summary

Keep open. Current main still has the Mattermost streaming contract mismatch: docs advertise `channels.mattermost.streaming` modes, but the Mattermost strict schema/type surface omits `streaming`, and inbound runtime still creates and wires draft-preview updates without resolving `off | partial | block | progress`. Open PR #73252 is the active related fix candidate, so this should stay open for maintainer review rather than be closed or duplicated.

## What This Changes

Mattermost should honor the documented `channels.mattermost.streaming` modes and improve the draft-preview/final-notification UX instead of silently dropping the key and always using the legacy draft-preview path.

## Best Possible Solution

Keep this as the Mattermost streaming-contract tracker until PR #73252 or an equivalent Mattermost-local fix lands. The end state should accept documented streaming config at root and account scope, reuse the shared preview-mode resolver, preserve legacy block config compatibility, disable draft-preview callbacks/finalization for `off`, define behavior for `partial | block | progress`, and add focused schema/runtime/docs coverage.

## Reproduction Assessment

Yes for the documented config mismatch: current main’s strict Mattermost schema/type/metadata omit `streaming` while docs advertise it, and the inbound monitor still wires draft-preview updates unconditionally. The notification symptom has credible reporter evidence, but I did not live-test Mattermost client notifications.

## Solution Assessment

No current-main solution exists yet. The best visible path is to review, land, or revise PR #73252 as the canonical `streaming.mode` fix, then decide whether the separate final-notification UX needs a follow-up.
