## Summary

Keep this open. Current main still has the Mattermost streaming contract mismatch: docs advertise `channels.mattermost.streaming` modes, but the Mattermost plugin schema/type/metadata still expose only legacy block-streaming controls, and the inbound runtime still wires draft-preview updates without resolving the documented mode. Open PR #73252 is already the active narrow fix candidate, so this should stay open for maintainer review rather than be closed or queued again.

## What This Changes

The issue asks Mattermost to honor the documented `streaming` configuration modes and resolve the draft-preview notification/noise behavior caused by the legacy preview streaming path.

## Best Possible Solution

Keep this as the canonical Mattermost streaming-contract tracker until PR #73252 or an equivalent Mattermost-local patch lands. The desired end state should accept documented `streaming.mode` config at the supported scopes, preserve legacy `blockStreaming` compatibility, gate draft-preview callbacks and finalization for `off`, define behavior for `partial`, `block`, and `progress`, and add focused schema/runtime/docs coverage.

## Reproduction Assessment

Yes for the config mismatch: current main’s strict Mattermost schema/type/metadata omit the documented `streaming` key while docs advertise it, and the runtime still wires draft-preview updates without a mode gate. The notification-specific symptom has credible reporter evidence but was not live-verified in a Mattermost client.

## Solution Assessment

No current-main solution exists yet. The best visible path is to review, land, or revise PR #73252 as the canonical `streaming.mode` fix, then decide whether final-notification behavior needs a separate follow-up.
