## Summary

Keep this PR open. Current main still lacks the proposed plugin-facing `api.resetSession()` SDK surface, and the earlier #43375 is closed unmerged, but this PR should not merge as-is because the diff removes the existing `sessions.reset` broadcast path for gateway subscribers.

## What This Changes

The PR adds an optional plugin-facing `api.resetSession(key, reason?)` method, result type and SDK export, registry wiring to the gateway reset service, reset-service snapshot/locking changes, generated SDK baseline updates, and focused tests.

## Best Possible Solution

Keep the PR active, repair the reset notification ownership so both gateway RPC resets and plugin-initiated resets emit exactly one `sessions.changed` event, document the new public SDK method and gating/result shape, then rerun the focused plugin/gateway tests plus the plugin SDK API check.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to compare current main’s `sessions.reset` handler, which broadcasts after `performGatewaySessionReset()`, with the PR diff that removes that broadcast while the reset service still does not emit a replacement lifecycle event.

## Solution Assessment

No, not as proposed. The plugin API direction is plausible and narrow, but the safest implementation must preserve current gateway reset subscriber behavior instead of moving only plugin-initiated resets onto a different event path.
