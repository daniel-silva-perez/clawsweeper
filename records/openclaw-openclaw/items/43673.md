## Summary

Keep open. Current main has meaningful adjacent support for isolated agents, per-agent tool/sandbox policy, group and sender-scoped group tool restrictions, and Gateway operator scopes, but it still does not provide the requested team scaffold command, declarative deployment manifest, or operator/session-scoped RBAC model. The related RFC and PRs are useful narrower slices, not a complete replacement for this umbrella request.

## What This Changes

The issue requests first-class org/team deployment support through workspace scaffolding, per-session RBAC-style tool policies, declarative deployment manifests, and official org deployment documentation.

## Best Possible Solution

Keep this as a maintainer-scoped umbrella or convert it into an RFC, then split accepted work into smaller issues for RBAC semantics, CLI/team scaffolding, deployment manifest schema, and org deployment documentation. The existing group/sender policy plumbing and open direct-session policy PR can be reviewed as incremental building blocks.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is source and docs inspection of CLI commands, config types, tool-policy resolution, group policy helpers, Gateway scopes, and security docs, which shows adjacent support but not the requested first-class org/team deployment model.

## Solution Assessment

Unclear that one implementation issue is the best way to solve this. The maintainable path is an explicit maintainer product/security design followed by narrower PRs, especially for any RBAC subset that must align with Gateway operator-scope and channel sender-identity semantics.
