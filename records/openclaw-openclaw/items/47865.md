## Summary

Keep open. Current main still routes main-session cron system events through the heartbeat path with `heartbeat.target = "last"` and does not consult `delivery.mode = "none"` before enqueueing the event or waking a user-deliverable heartbeat. The issue comments correctly identify two still-relevant paths: cron `systemEvent` responses and exec failure notifications.

## Best Possible Solution

Keep this issue open. The best fix is to define the intended policy for `sessionTarget: "main" + payload.kind: "systemEvent" + delivery.mode: "none"`, then either propagate an internal-only/no-user-delivery flag through the queued system event and heartbeat wake or update docs/UI to say main-session systemEvent responses may be visible. If fixing, add regression coverage for both the cron systemEvent path and the exec SIGTERM/notifyOnExit path; review #58893 as a possible implementation candidate but do not close this issue before an implementation lands.

- Maintainers need a product decision on whether WebChat transcript visibility counts as delivery; suppressing only outbound channel sends may still leave the reported UI leak.
