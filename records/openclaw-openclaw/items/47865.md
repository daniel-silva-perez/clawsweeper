## Summary

Keep open. Current main still routes `sessionTarget: "main"` cron `systemEvent` jobs through the normal heartbeat path, and `delivery.mode: "none"` is not consulted before enqueueing the system event or waking the last chat target. The open related PR is still draft, unmerged, and dirty, so it is not proof of an implemented fix.

## What This Changes

The issue reports that main-session cron `systemEvent` jobs with `delivery.mode: "none"` still produce user-visible chat messages, including scheduled reminder text and exec failure output.

## Best Possible Solution

Define and enforce a single delivery policy for `sessionTarget: "main"` plus `payload.kind: "systemEvent"`: either make explicit `delivery.mode: "none"` carry an internal-only/no-user-delivery flag through cron enqueueing and heartbeat delivery, or update validation/docs/tooling so users cannot expect main-session system events to be silent. Add regression coverage for main cron no-delivery behavior and the separate exec notification path, coordinating with #58893 and #73900.

## Reproduction Assessment

Yes. The issue's cron config maps directly to the current main-session cron path: source shows the system event is enqueued and heartbeat is woken with `target: "last"` despite `delivery.mode: "none"`; existing tests also assert that last-channel behavior.

## Solution Assessment

No for current main. The current implementation relies on heartbeat target resolution and prompt wording rather than enforcing the cron delivery policy at the cron-to-heartbeat boundary; a narrow maintainable fix should make the policy explicit in code and tests.
