## Summary

Keep open. Current main still routes main-session cron system events through the normal heartbeat path with `heartbeat.target = "last"`; `delivery.mode = "none"` resolves to `not-requested` bookkeeping, but does not suppress the enqueue or user-deliverable heartbeat. The related draft PR #58893 is unmerged, so it is an implementation candidate rather than a reason to close this issue.

## Best Possible Solution

Keep this issue open. The best fix is for maintainers to define the intended policy for `sessionTarget: "main" + payload.kind: "systemEvent" + delivery.mode: "none"`: either thread an internal-only/no-user-delivery flag through cron enqueueing and heartbeat wake/run, or update validation/docs/UI to make clear that main-session system events can still drive user-visible heartbeat replies. Add regression coverage for explicit main cron `delivery.mode: "none"` and for exec `notifyOnExit`/SIGTERM notification behavior; review #58893 as a candidate but do not close this issue before an implementation lands.
