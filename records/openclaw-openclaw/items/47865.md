## Summary

Keep open. Current main still routes main-session cron system events through the heartbeat path with `target: "last"`; `delivery.mode: "none"` is not consulted on that path, and tests currently assert the user-visible routing behavior instead of suppressing it.

## Best Possible Solution

Keep this issue open. The best fix is to define and implement the intended policy for `sessionTarget: "main" + payload.kind: "systemEvent" + delivery.mode: "none"`: propagate an internal-only/no-delivery flag through the queued system event and heartbeat wake, prevent user-visible outbound and WebChat transcript responses unless explicitly requested, and add regression tests for both cron systemEvent and exec completion/SIGTERM events. If maintainers instead want current behavior, update Control UI wording and cron docs to state that main-session system events can produce visible agent responses and that `none` only disables runner fallback delivery.
