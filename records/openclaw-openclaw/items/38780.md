## Summary

Keep this PR open. Current main does not implement the continuation feature, and the PR is still an active XL core behavior change with verified correctness and safety blockers in delegate-drain gating, TaskFlow-backed delegate lifecycle handling, and build warning suppression.

## What This Changes

The PR adds opt-in agent continuation tools and response-token fallbacks, continuation wake/delegate scheduling, context-pressure and compaction hooks, TaskFlow-backed delegate state, Gateway/client protocol fields, status/config/docs/generated baseline updates, and continuation-focused tests.

## Best Possible Solution

Keep the PR open for maintainer product and security review. If maintainers want this in core, the branch should be repaired in place or replaced with a narrower core design that only exposes continuation tools on runs that actually drain delegates, handles TaskFlow revision conflicts fail-safe, removes the lazy-import suppression by fixing the import boundary, and then validates the full changed surface before merge.

## Reproduction Assessment

Not applicable as a user bug reproduction. The blocking review findings are reproducible by static inspection of PR head 90db3699ccf3b6c7973dd3fdd9d489c8b507ff3f plus the current main TaskFlow update-result contract.

## Solution Assessment

No. The proposed direction may be useful, but this patch is not the best mergeable solution because it exposes continuation delegation when the drain flag is omitted, can replay consumed or canceled TaskFlow delegates after revision conflicts, and suppresses a build-time lazy-boundary warning instead of resolving the import graph.

  - confidence: 0.83

  - body: After `requestFlowCancel` succeeds, the status transition to `cancelled` also returns a result, but this code ignores it. A concurrent revision change can leave a cancel-requested flow with `status: "queued"`, and `listPendingFlows` will still make it eligible for dispatch.
  - confidence: 0.79

## Security Review

Status: needs_attention

Summary: Security review needs attention because the diff introduces autonomous scheduling/delegation paths while weakening delegate-drain, cancel, and build-boundary controls.

Concerns:

- **[medium] Continuation delegation is exposed outside draining runs:** `src/agents/openclaw-tools.ts:347`
  - body: At PR head, `continue_delegate` can be exposed when continuation is enabled and the drain flag is omitted, while embedded tool construction does not forward the drain flag. For an opt-in autonomous background-work feature, this broadens the tool surface beyond the expected scheduler/drain pipeline.
  - confidence: 0.86
- **[medium] TaskFlow delegate conflicts can replay canceled or consumed work:** `src/auto-reply/continuation-delegate-store-taskflow.ts:98`
  - body: The TaskFlow-backed delegate store ignores `applied:false` results from finish and cancel status updates. In conflict cases, queued records can remain dispatchable after they were logically consumed or canceled, which weakens operator cancellation and autonomous-work bounds.
  - confidence: 0.8
- **[low] Build warning suppression weakens lazy-boundary review:** `tsdown.config.ts:63`
  - body: The diff suppresses `INEFFECTIVE_DYNAMIC_IMPORT` for `pi-embedded.ts`. This is not a dependency or secret leak, but it weakens a build-time guard around a runtime code-execution boundary changed by the PR.
  - confidence: 0.72
