## Summary

Keep this PR open. The latest head fixes the earlier ClawSweeper logging-order concern by passing a `shouldLog` predicate, but the patch still treats `MESSAGE_DELETE_FORBIDDEN` as a benign no-op and suppresses the API error log for it. That can hide Telegram permission/configuration failures while the action runtime reports an overall successful tool result.

## What This Changes

The PR changes Telegram `deleteMessage` handling to return a warning for selected Bot API 400 errors, suppresses shared API error logging for those selected errors, surfaces `deleted: false` warnings from the Telegram action runtime, adds `send.ts` regression tests, and updates `CHANGELOG.md`.

## Best Possible Solution

Revise the PR so only true idempotent no-op delete states are fail-soft and no-log. Permission or authorization failures such as `MESSAGE_DELETE_FORBIDDEN` should either throw normally or be reported as a loud, non-success result. Keep the `shouldLog` suppression for true benign cases, add tests for forbidden failures and action-runtime warning behavior, and trim the changelog diff after rebasing.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Do not soft-fail forbidden delete errors:** `extensions/telegram/src/send.ts:1099`
  - body: `MESSAGE_DELETE_FORBIDDEN` can mean the bot lacks rights or is deleting in the wrong context, not that the message is already gone. Because this predicate also suppresses the shared API error log and the action runtime converts every soft result into `{ ok: true, deleted: false }`, permission/config regressions become quiet successful no-ops. Keep forbidden errors out of the benign/no-log classifier or handle them as a separate loud failure.
  - confidence: 0.88
