## Summary

Keep open. Current main still hardcodes the media store and sandbox staging cap at 5 MB, and the config schema/type/help surface still has no top-level media.maxBytes setting. The chat path now preflights sandboxed non-image attachments above that cap as a 4xx, but that only makes the rejection clearer; it does not make the limit user-configurable. The provided GitHub context also shows PR #40950 as an open, unmerged implementation candidate with closing syntax for this report, so the issue should remain open until that PR or an equivalent fix lands.

## Best Possible Solution

Keep this issue open and review/land PR #40950 or an equivalent narrow fix. The fix should add a bounded top-level media.maxBytes config field, thread it through the media store, media serving, sandbox staging, and chat preflight paths, preserve the 5 MB default when unset, and update schema/help/docs plus regression coverage for both default and configured limits. Broader image sanitization limits can remain tracked separately by #67031.
