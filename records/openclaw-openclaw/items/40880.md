## Summary

Keep open. Current main still hardcodes the 5 MB media-store and sandbox staging cap, the strict config schema still lacks a top-level or tools-level sandbox staging max-bytes setting, and open PR #40950 is the unmerged closing implementation path, so this issue should not be closed before that PR or an equivalent fix lands.

## What This Changes

The issue asks for the hardcoded 5 MB media-store and sandbox staging limit to become user-configurable so larger PDFs, presentations, images, and documents can reach sandboxed agents.

## Best Possible Solution

Land PR #40950 or an equivalent narrow fix that adds a bounded media staging/store max-bytes config with the current 5 MB default, threads the effective value through media storage, sandbox staging, and chat preflight, and updates schema/help/docs/tests without expanding into the broader image-limit work tracked separately by #67031.

## Reproduction Assessment

Yes. A sandboxed non-image attachment larger than 5 MB follows a clear current-main path: media storage and sandbox staging still use `MEDIA_MAX_BYTES`, chat preflight rejects `ref.sizeBytes > MEDIA_MAX_BYTES`, and existing tests assert both buffer and 6 MB sandbox PDF rejection behavior.

## Solution Assessment

Yes for keeping this open. The narrow maintainable fix is review and landing of #40950 or an equivalent implementation; closing this issue before that PR merges would drop the active regression tracker.
