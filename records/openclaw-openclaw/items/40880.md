## Summary

Keep open. Current main still has the hardcoded 5 MB media-store and sandbox staging cap, top-level `media.maxBytes` is not in the strict config schema, and the linked open implementation PR should be reviewed or landed before this issue closes.

## What This Changes

The issue requests making the hardcoded 5 MB sandbox/media-store staging limit configurable so larger files can be copied into sandboxed agents.

## Best Possible Solution

Land PR #40950 or an equivalent narrow fix that adds a bounded top-level `media.maxBytes` config with the current 5 MB default, threads the effective limit through media store, sandbox staging, and chat preflight, updates schema/help/docs, and keeps broader image-size configurability tracked separately by #67031.

## Reproduction Assessment

Yes. A sandboxed non-image attachment larger than 5 MB follows a clear current-main path: media store accepts only the fixed default, chat preflight compares against `MEDIA_MAX_BYTES`, and existing tests assert rejection for 5 MB plus one byte and a 6 MB sandbox PDF.

## Solution Assessment

Yes for keeping this issue open until the linked implementation lands. A narrow `media.maxBytes` config is the maintainable fix boundary; widening this into all image/provider limits should stay with the broader companion request.
