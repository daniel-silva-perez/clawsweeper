## Summary

Keep open. Current main has useful Baileys media hotfix machinery, but the reported packaged/runtime-deps path is still not fully covered: external staged roots can be accepted by dependency/version and entry-file checks without verifying that Baileys `messages-media.js` contains the encrypted-media hotfix.

## What This Changes

Requests that WhatsApp outbound media sends stop crashing the Gateway on Baileys encrypted temp-file `ENOENT` errors, while preserving proxy dispatcher behavior, improving timeout configurability, and making runtime-deps root selection safer.

## Best Possible Solution

Keep this open until OpenClaw either verifies and repairs the Baileys encrypted-media hotfix in every selected external runtime-deps root before WhatsApp loads, or upgrades to an upstream Baileys version that removes the race. The chosen fix should reject stale already-materialized unpatched roots and add coverage for packaged external install/reuse behavior.

## Reproduction Assessment

Yes. The issue provides a concrete outbound WhatsApp media CLI reproduction with ENOENT logs, and current source inspection confirms that media sends still flow into Baileys `sock.sendMessage`; I did not run a live linked-account send in this read-only sweep.

## Solution Assessment

No. The existing root/package patch and postinstall helper are useful, but they are not a complete solution because packaged external runtime-deps roots can be installed or reused without hotfix content verification.
