## Summary

Keep open. Current main contains a partial Baileys media hotfix in the root/package postinstall path and pnpm patch metadata, but the reported packaged runtime-deps path still installs `@whiskeysockets/baileys@7.0.0-rc.9` into external `plugin-runtime-deps` roots from a generated manifest without applying that hotfix or invalidating already-materialized unpatched roots.

## What This Changes

The issue asks OpenClaw to prevent WhatsApp outbound media sends from crashing the gateway on Baileys temporary encrypted media file `ENOENT` failures, while preserving proxy dispatcher handling and making media upload behavior more robust/configurable.

## Best Possible Solution

Keep this issue open until OpenClaw applies the Baileys encrypted-media hotfix to every bundled WhatsApp runtime dependency root that can be loaded, including external `OPENCLAW_PLUGIN_STAGE_DIR`/`~/.openclaw/plugin-runtime-deps` installs and previously materialized copies, with a regression test that proves stale unpatched Baileys trees are repaired or rejected before the gateway imports WhatsApp.

## Reproduction Assessment

Yes, with a linked WhatsApp account and packaged/staged runtime deps. The issue body gives concrete `openclaw message send --channel whatsapp --media /tmp/tiny.png --json` steps and ENOENT logs, and current source inspection confirms the OpenClaw media path delegates to Baileys' temp-file upload implementation.

## Solution Assessment

No. The current partial fix is directionally right for patched root dependencies, but the best fix needs to cover the external bundled runtime-deps install/repair path and staged-root completeness checks rather than relying only on root pnpm patches or package postinstall.
