## Summary

Keep open. Current main has useful Baileys media hotfix machinery, but it still does not fully cover the reported packaged/runtime-deps scenario: external staged roots can be selected or reused based on dependency/version and entry-file checks without verifying that Baileys `messages-media.js` contains the encrypted-media hotfix.

## What This Changes

The issue requests hardening WhatsApp outbound media upload so Baileys `*-enc` temp-file `ENOENT` failures cannot crash the Gateway, while preserving proxy dispatcher behavior, making upload timeout configurable, and clarifying runtime-deps root selection.

## Best Possible Solution

Keep this open until OpenClaw either verifies and repairs the Baileys encrypted-media hotfix in every selected external runtime-deps root before WhatsApp loads, or upgrades to an upstream Baileys version that removes the race. The fix should reject or repair stale unpatched roots and add coverage for packaged external install/reuse behavior.

## Reproduction Assessment

Yes. The issue gives concrete CLI reproduction steps and `ENOENT` logs, and current source inspection confirms outbound WhatsApp media still flows into Baileys `sock.sendMessage`; I did not perform a live linked-account send in this read-only sweep.

## Solution Assessment

No. The current postinstall/root-package hotfix is useful but incomplete because external staged runtime-deps roots can be installed or reused without hotfix content verification.
