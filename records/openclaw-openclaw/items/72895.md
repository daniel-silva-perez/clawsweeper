## Summary

Keep open. Current main still looks partial: WhatsApp media sends reach Baileys, and OpenClaw has a root/package-level Baileys media hotfix, but external packaged `plugin-runtime-deps` roots can still be reused by dependency spec/version without verifying that `messages-media.js` contains the hotfix. The fallback runtime-deps installer also installs normal npm specs with scripts ignored, so the reported staged-runtime crash path remains actionable.

## What This Changes

Prevent WhatsApp outbound media uploads from crashing the Gateway on Baileys encrypted temp-file ENOENTs while preserving media upload proxy behavior and runtime-deps consistency.

## Best Possible Solution

Keep the issue open until OpenClaw either verifies and repairs the Baileys encrypted-media hotfix in every external runtime-deps root before WhatsApp loads, or upgrades to an upstream Baileys version that removes the race. The fix should reject or refresh stale already-materialized unpatched roots and add regression coverage for the packaged external install/reuse path.

## Reproduction Assessment

Yes. The issue body gives a concrete linked-account reproduction with `openclaw message send --channel whatsapp --media /tmp/tiny.png --json` and ENOENT logs, and current source inspection confirms that outbound media still reaches Baileys' upload path through `sock.sendMessage`.

## Solution Assessment

No. The existing patch/postinstall direction is useful for root/package installs, but it is not the best complete fix because the packaged external runtime-deps path can install or reuse an unpatched Baileys tree without content verification.
