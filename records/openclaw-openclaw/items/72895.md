## Summary

Keep open. Current main has a partial Baileys media hotfix through the root pnpm patch/package postinstall path, and the existing ClawSweeper comment correctly identified the remaining gap: packaged external `plugin-runtime-deps` roots are still accepted by dependency spec/version materialization rather than by verifying the patched Baileys `messages-media.js` content.

## What This Changes

Ensure WhatsApp outbound media uploads cannot crash the Gateway on Baileys `*-enc` temp-file `ENOENT`s, while preserving the media-upload dispatcher workaround, improving timeout handling, and making staged runtime-deps selection/repair reliable.

## Best Possible Solution

Keep this open until OpenClaw applies or verifies the Baileys encrypted-media hotfix for every runtime dependency root that can be selected at Gateway startup, including existing external `plugin-runtime-deps/openclaw-*` roots, with regression coverage proving stale unpatched Baileys trees are repaired or rejected before the WhatsApp plugin imports.

## Reproduction Assessment

Yes. The issue gives concrete CLI steps and logs, and source inspection confirms the media path still reaches Baileys while external staged runtime-deps roots can be considered materialized without checking the hotfixed file contents.

## Solution Assessment

No. The full-buffer local workaround is not the best long-term shape because it can regress large-media memory use; the maintainable fix is to preserve the streaming Baileys flush-order patch and ensure that patch is propagated to, or enforced for, every packaged external runtime-deps root.
