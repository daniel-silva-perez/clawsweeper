## Summary

Keep open. Current main has useful Baileys media hotfix machinery, but the selected external bundled runtime-deps roots can still be accepted as materialized using only dependency specs, package versions, and entry files; they are not verified for the Baileys `messages-media.js` encrypted-media hotfix before WhatsApp loads.

## What This Changes

The issue asks OpenClaw to prevent WhatsApp outbound media sends from crashing the Gateway on Baileys encrypted temp-file `ENOENT` failures, preserve proxy dispatcher compatibility, make upload timeout configurable, and clarify stale runtime-deps root selection.

## Best Possible Solution

Keep this open until OpenClaw verifies and repairs the Baileys media hotfix in every selected external bundled runtime-deps root before WhatsApp runtime loading, or upgrades to an upstream Baileys version that removes the temp-file race. The fix should add tests for stale generated-manifest external roots and preserve the existing package-root hotfix behavior.

## Reproduction Assessment

Yes. The issue gives concrete CLI reproduction steps and crash logs, and current source confirms outbound WhatsApp media still flows through Baileys `sock.sendMessage`; I did not perform a live linked-account media send during this read-only review.

## Solution Assessment

No. The current package-root postinstall hotfix is useful but incomplete because external staged runtime-deps roots can be reused without verifying that Baileys `messages-media.js` contains the encrypted-media and dispatcher hotfixes.
