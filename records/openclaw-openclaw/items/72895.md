## Summary

Keep open. Current main has a Baileys media patch and postinstall hotfix machinery, but the reported packaged/runtime-deps scenario is still not fully covered: WhatsApp outbound media still delegates to Baileys, and staged runtime-deps roots can be reused based on dependency/version and entry-file checks without verifying that `@whiskeysockets/baileys/lib/Utils/messages-media.js` contains the encrypted-media hotfix. The existing ClawSweeper comment remains consistent with current main after the latest runtime-deps changes.

## What This Changes

The issue asks OpenClaw to prevent WhatsApp outbound media uploads from crashing the Gateway on Baileys encrypted temp-file `ENOENT` failures and to make the Baileys upload/runtime-deps fix reliable across staged roots.

## Best Possible Solution

Keep this open until OpenClaw either upgrades to a Baileys release that removes the encrypted temp-file race or verifies and repairs the existing Baileys `messages-media.js` hotfix in every staged runtime-deps root before WhatsApp loads. Add focused coverage for reusing a stale unpatched external root and for installing/repairing packaged WhatsApp runtime deps.

## Reproduction Assessment

Yes. The issue includes concrete CLI reproduction steps and `ENOENT` logs, and current source inspection confirms outbound WhatsApp media still flows through Baileys `sock.sendMessage`; I did not perform a live linked-account send in this read-only review.

## Solution Assessment

No. Current main's root/package patch and postinstall helper are useful, but they are not the complete fix for packaged staged roots; the reporter's full-buffer `readFile` workaround would avoid the stream race but could regress the large-media memory behavior the existing Baileys patch was designed to preserve.
