## Summary

Keep this open. Current main has bounded timeout compaction, profile rotation, model fallback, and stuck-session diagnostics, but it still allows the reported class of same-run retry churn: a timeout can compact and retry the same provider/model, assistant timeout handling can rotate profiles before model fallback, and stuck-session recovery does not abort active runs unless explicitly enabled.

## What This Changes

The issue requests a guard against one assistant turn repeatedly reissuing near-identical same-run provider requests while the session remains stuck in processing.

## Best Possible Solution

Keep this issue open and add a narrow regression plus runtime guard for unproductive same-run timeout churn. The guard should distinguish material compaction or real profile/model advancement from near-identical same-candidate retries, then either advance to the next fallback candidate or surface a clear timeout error after a small bounded budget, while preserving normal model-failover semantics and adding per-run diagnostics useful for future reports.

Current main still permits timeout-triggered compaction to retry the same provider/model, then assistant timeout failover/profile rotation/fallback can keep the same runId alive before a later fallback succeeds.
