## Summary

Keep open. Current main has substantial failover guardrails and diagnostics, but I could not verify that the reported same-turn same-candidate timeout churn is fully fixed. In fact, current behavior still intentionally permits some same-provider/model retries during timeout-triggered compaction and auth-profile rotation before model fallback, so closing this as implemented would be too aggressive.

## Best Possible Solution

Keep this issue open and add a focused regression/fix around timeout-driven same-candidate churn: simulate a single run with configured fallbacks, repeated no-chunk/timeouts, and near-unchanged prompt usage, then ensure OpenClaw either moves to the next fallback after a bounded same-provider/profile attempt budget or surfaces a clear timeout error. Preserve legitimate timeout compaction only when it materially reduces context and keep the existing model-fallback diagnostics.
