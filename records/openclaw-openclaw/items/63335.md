## Summary

Current main has useful guardrails for timeout compaction, auth-profile rotation, model fallback, and stuck-session diagnostics, but it does not prove the reported same-turn timeout churn is fixed. The embedded runner still retries the same provider/model after successful timeout-triggered compaction, configured-fallback runs bypass the newer same-model idle-timeout retry cap, and active stuck-session recovery is conservative rather than aborting the active attempt by default.

## Best Possible Solution

Keep this issue open and add a focused regression plus runtime guard for timeout-driven same-candidate churn. The best fix is to distinguish productive timeout compaction/profile rotation from unproductive near-identical prompt retries, then either advance to the next fallback candidate or surface a clear timeout error after a small bounded budget while preserving compaction when it materially reduces context.
