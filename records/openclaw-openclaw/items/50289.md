## Summary

Keep open. Current main still has repeated linear `ownerAllowFrom` normalization and lookup in the command authorization path, and open PR #73440 is the active implementation candidate with closing syntax, so closing this before merge would orphan the fix path.

## What This Changes

Improve command authorization performance for large `commands.ownerAllowFrom` allowlists by avoiding repeated linear normalization and lookup on inbound messages.

## Best Possible Solution

Review and land PR #73440, or an equivalent narrow command-auth fix, that caches or compiles normalized owner allowlists per raw array/provider/account/plugin while preserving config reload behavior and owner-only security semantics. Close this issue only after that fix merges and passes focused command-auth coverage plus the changed gate.

## Reproduction Assessment

Yes for the remaining code-level performance problem: configure a large `commands.ownerAllowFrom` list and send a normal inbound message; current main builds command context, resolves the owner allowlist twice, and performs linear membership checks. The historical config-reparse timing is less certain now because `loadConfig()` uses a pinned runtime snapshot.

## Solution Assessment

Yes directionally: a narrow cache or compiled lookup inside `src/auto-reply/command-auth.ts` is the maintainable fix, provided it keys on the inputs that affect channel normalization and invalidates naturally on config reload. A wildcard workaround or broad access-control redesign is not the best primary fix for this report.
