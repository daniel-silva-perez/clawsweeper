## Summary

Keep open. Current `main` still normalizes `commands.ownerAllowFrom` by walking every entry during command authorization and then does linear membership checks, while open PR #73440 is already the active fix candidate linked with GitHub closing semantics.

## What This Changes

The issue asks OpenClaw to avoid severe gateway latency when large `commands.ownerAllowFrom` lists are normalized and searched on the inbound message authorization path.

## Best Possible Solution

Review and land PR #73440, or an equivalent narrow command-auth change, that caches or compiles normalized owner allowlists per raw array/provider/account/plugin while preserving config reload behavior and owner-only authorization semantics.

## Reproduction Assessment

Yes. Configure a large `commands.ownerAllowFrom` list and send a normal inbound message; current main builds command context, resolves owner allowlists twice, and performs linear list membership checks on that path.

## Solution Assessment

Yes directionally. A narrow cache or compiled lookup inside `src/auto-reply/command-auth.ts` is the maintainable fix; wildcard owner access or broad access-control redesign would not be the right primary solution for this bug.
