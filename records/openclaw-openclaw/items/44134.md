## Summary

Keep this issue open. Current main still normalizes provider tool schemas during each embedded attempt and through provider runtime hooks without a normalized-catalog reuse layer. The Google Gemini CLI provider still uses the Gemini tool-compat hook, so the reported repeated schema work remains plausible even though current docs and OAuth setup now warn users about account restriction risk.

## Best Possible Solution

Implement a conservative normalized-tool-schema reuse layer in the runtime/tool-schema path. Cache only normalized schema/catalog data for unchanged tools plus provider/model/API/baseUrl context, rebuild attempt-local tool objects so execute callbacks stay fresh, invalidate on actual tool/context changes, and add focused regression coverage for unchanged reuse and changed-schema/context invalidation.

## Repair Work Prompt

Fix the repeated provider tool-schema normalization reported in openclaw/openclaw#44134. Current main still normalizes tools on each embedded attempt: `src/agents/pi-embedded-runner/run/attempt.ts` calls `normalizeAgentRuntimeTools`, which delegates through `src/agents/runtime-plan/tools.ts` and `src/agents/pi-embedded-runner/tool-schema-runtime.ts` into provider hooks; `src/plugin-sdk/provider-tools.ts` maps the full Gemini/OpenAI tool catalog, and `extensions/google/gemini-cli-provider.ts` uses the Gemini tool-compat family. Add a conservative reuse layer for normalized provider tool schemas/catalog data for unchanged tools plus provider/model/API/baseUrl context. Do not cache executable tool callbacks or per-attempt runtime context. Invalidate when tool names, parameter schemas, tool count/order, provider, modelId, modelApi, model/baseUrl, or filtered tool set changes; keep dynamic MCP/LSP/bundled tools correct. Add focused tests proving repeated unchanged normalization reuses cached schema work, changed schemas/context rerun normalization, and returned tools preserve the current attempt's execute functions. Keep Plugin SDK behavior backwards-compatible. Add a single-line changelog fix entry crediting @caimao9539 and @villavicencio if the fix is user-facing; avoid stronger Google policy claims than the user reports support. Do not change dependencies, provider auth, or Google OAuth warning text unless directly necessary.
