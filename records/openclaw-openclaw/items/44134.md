## Summary

Keep this issue open. Current main still normalizes provider tool schemas in the embedded attempt path and provider runtime path without a normalized-catalog cache. The Google Gemini CLI provider is still wired through the Gemini tool-compat hook, so the reported repeated schema work remains plausible on the affected provider family.

## Best Possible Solution

Implement a conservative normalized-tool-schema reuse layer in the runtime/tool-schema path. Cache normalized schema/catalog data for unchanged tool lists and provider/model/API context, invalidate on actual tool or context changes, and avoid retaining per-attempt closures. Add focused regression tests proving repeated attempts with unchanged tools do not rerun per-tool normalization, while changed schemas, provider context, and dynamic tools still update.

## Clownfish Work Prompt

Fix the repeated provider tool-schema normalization reported in openclaw/openclaw#44134. Current main still normalizes tools on each embedded attempt: `src/agents/pi-embedded-runner/run/attempt.ts` calls `normalizeAgentRuntimeTools`, which delegates through `src/agents/runtime-plan/tools.ts` and `src/agents/pi-embedded-runner/tool-schema-runtime.ts` into provider hooks; `src/plugin-sdk/provider-tools.ts` maps the full Gemini/OpenAI tool catalog, and `extensions/google/gemini-cli-provider.ts` uses the Gemini tool-compat family. Add focused tests proving repeated unchanged normalization reuses cached schema work, changed schemas/context rerun normalization, and returned tools preserve the current attempt's execute functions. Keep Plugin SDK behavior backwards-compatible.
