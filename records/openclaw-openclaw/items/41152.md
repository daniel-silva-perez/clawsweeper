## Summary

Keep open. Current main still sends ordinary gateway and node async exec approval completion through the compact notification formatter, so long multiline output is reduced to the 400-character tail and whitespace-normalized before the follow-up agent turn. The linked fix PR #41170 is closed and unmerged in the provided GitHub context, and the latest release tag still contains the compacting path.

## What This Changes

The issue reports that async exec approval completion delivered via Discord/gateway follow-up trims command output to the final roughly 400 characters and collapses multiline output into one line.

## Best Possible Solution

Reimplement the narrow behavior from the closed linked PR on current main: async approval completion should deliver the captured exec output with newlines preserved up to the existing capture cap, clearly mark capture-cap truncation, and leave ordinary background notification summaries compact.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: any approved async gateway or node exec completion with multiline output over 400 characters flows through the 400-character tail plus `normalizeNotifyOutput` before the agent follow-up prompt.

## Solution Assessment

Yes. The best fix is a scoped formatter split for approval completion output, with regression tests for multiline >400-character output and capture-cap truncation; changing the shared compact notify formatter directly would be broader than needed.
