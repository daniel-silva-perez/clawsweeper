## Summary

Keep open. Current main still follows the bug path this PR targets: Slack thread context hydrates root-message files into `threadStarterMedia`, and message preparation falls back to that media when a thread reply has no direct media. The proposed behavior is still useful, but the branch is now merge-conflicted against current main and needs rebase or replacement before merge.

## What This Changes

The PR changes Slack thread preparation so thread replies no longer hydrate media from the root thread message, while preserving direct media on the reply and adding targeted thread-context tests.

## Best Possible Solution

Land a narrow Slack-plugin fix that prevents root thread media from being delivered as fresh media on replies, keeps starter text and first-turn history context, preserves reply-owned media through `effectiveDirectMedia`, and adds focused regression coverage on current main.

## Reproduction Assessment

Yes. The high-confidence reproduction is a Slack root thread message with an image followed by a text-only thread reply: on current main, `effectiveDirectMedia` is null, `starter.files` are hydrated, and `prepareSlackMessage` falls back to `threadStarterMedia`.

## Solution Assessment

Yes in direction, but the source branch is stale. The maintainable fix is plugin-local: stop hydrating parent media for replies and remove the fallback in the current Slack prepare path, adapted to current main's lazy media imports and newer thread-context tests.
