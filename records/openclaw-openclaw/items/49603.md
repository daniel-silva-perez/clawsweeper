## Summary

Keep open. Current main still does not perform the requested boot-time cleanup for fresh self-owned session lock files: gateway startup calls `cleanStaleLockFiles()`, that helper only removes locks already classified as stale by PID/age parsing, and the self-owned orphan helper is only used during acquisition contention. The supplied GitHub context also shows open PR #49660 still targets this issue with closing syntax, so this issue should remain open until that PR is reviewed and landed or closed.

## Best Possible Solution

Keep this issue open and review PR #49660 against current main. The likely fix belongs in the startup cleanup path or shared `cleanStaleLockFiles()` contract: classify self-owned lock files with no matching in-memory `HELD_LOCKS` entry as startup orphans, surface/log a clear `orphan-self-pid` reason, preserve truly active in-process locks, and add focused regression coverage for both `cleanStaleLockFiles()` and the gateway startup sidecar path.
