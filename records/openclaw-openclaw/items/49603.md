## Summary

Keep open. Current main has partial session-lock recovery, but the gateway startup cleanup path still calls `cleanStaleLockFiles()` and that function only removes locks already marked stale by PID/age parsing. The self-owned orphan handling is used during lock acquisition contention, not during startup cleanup, and an open implementation PR still explicitly targets this issue with closing syntax.

## Best Possible Solution

Review PR #49660 against current main. The likely code path is a startup-specific extension to `cleanStaleLockFiles()` or a narrow gateway-boot sweep that treats self-owned lock files with no corresponding `HELD_LOCKS` entry as orphaned, records/logs `orphan-self-pid`, and preserves active in-process locks during ordinary acquisition. Add regression coverage for `cleanStaleLockFiles()` and the gateway startup sidecar path, then resolve the PR state before closing this issue.
