## Summary

Keep open. Current main still renders SOUL.md, IDENTITY.md, USER.md, and MEMORY.md-style bootstrap files as ordered but flat Project Context entries, exposes no configurable per-file elevation or exclusion surface, and the direct implementation PR is closed unmerged. The remaining blocker is maintainer product/API direction, not cleanup closure.

## What This Changes

The issue requests configurable elevation of selected context files such as SOUL.md, IDENTITY.md, USER.md, and MEMORY.md into dedicated prompt sections instead of rendering them as ordinary Project Context files.

## Best Possible Solution

Define a documented deterministic context-file elevation contract covering match semantics, ordering, exclusion, missing-file behavior, prompt-cache placement, and priority scoping, then implement it in the maintainer-approved surface with aligned config schema, tests, and public docs.

## Reproduction Assessment

Yes. A focused static reproduction is to pass SOUL.md, IDENTITY.md, USER.md, and MEMORY.md through buildAgentSystemPrompt; current main sorts them and emits them as `## <path>` under `# Project Context`, and existing tests assert the flat rendering model.

## Solution Assessment

Unclear. The requested behavior is narrow, but the best maintainable surface still needs maintainer direction: a typed core config could solve it directly, while a plugin/context-engine route would need a clear file-level elevation and exclusion contract.
