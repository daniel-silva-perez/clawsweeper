## Summary

Keep open. Current main still preserves ANSI-bearing log text in the dedicated Control UI Logs tab and renders it directly; the Overview tail has a local strip helper, but that does not cover the `logs.tail` parser/view path. ## What This Changes

The issue asks for Control UI gateway logs to stop showing raw ANSI escape sequences and instead render or normalize styled log text.

## Best Possible Solution

Land a focused Control UI logging fix that shares one ANSI display strategy across every `logs.tail` UI consumer, normalizes displayed message/subsystem/fallback text, keeps raw export behavior deliberate, and adds regression coverage for ANSI-bearing structured and plain log lines.

## Repair Work Prompt

Fix the Control UI ANSI log display bug tracked by this issue. Current main strips ANSI only in `ui/src/ui/views/overview-log-tail.ts`, while the dedicated Logs tab parses `logs.tail` lines in `ui/src/ui/controllers/logs.ts` and renders `entry.message ?? entry.raw` in `ui/src/ui/views/logs.ts` without ANSI normalization. Implement a shared UI-side ANSI display helper, apply it to parsed display fields (`message`, `subsystem`, and plain fallback lines), and reuse it from the Overview log tail instead of keeping a local helper. Preserve `raw` intentionally so export can continue returning raw JSONL/log lines unless maintainers decide otherwise. Add regression tests for ANSI-bearing JSON messages, ANSI in subsystem/context parsing, and plain text fallback lines. Use #64431 as prior art but make the patch fit current main and repo commands. Add a user-facing changelog Fixes entry crediting the reporter and relevant prior contributor if the changelog section is active. Do not change server-side redaction semantics, log file format, gateway auth/scopes, or unrelated Control UI layout.
