## Summary

Keep open. Current main still has the media URI fall-through this PR is trying to harden, but the proposed patch is not ready because its new early throw still rejects a valid whitespace-prefixed inbound media URI instead of letting the shared media-reference normalizer hydrate it, and the linked sandbox resolver work remains separately open in #74186.

## What This Changes

The PR adds `loadWebMediaInternal()` guards so unresolved `media://` and other scheme-bearing media references are not resolved under `workspaceDir`, plus regression tests for stranded URI error behavior.

## Best Possible Solution

Amend the PR so the loader normalizes and resolves valid inbound `media://` references before applying the stranded-URI not-found guard, keep the guard for truly unresolved scheme-bearing leftovers, add the missing changelog entry, and coordinate the closing claim with the separate sandbox resolver PR #74186.

## Reproduction Assessment

Yes. A focused unit test can create an inbound managed media file, call `loadWebMedia('  media://inbound/<id>', { workspaceDir })`, and observe that this PR's new guard returns `not-found` instead of hydrating the existing file.

## Solution Assessment

No. The PR is on the right loader boundary for defense-in-depth, but the best narrow fix is to normalize before resolver precheck and only fail fast after a normalized managed URI truly cannot resolve; #74186 remains the better place for the sandbox bridge path.
