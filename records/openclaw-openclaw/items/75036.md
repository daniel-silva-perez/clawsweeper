## Summary

Keep open: current main still has the reported terminal-lost cron recovery gap, and this PR is pointed at the right maintenance path, but the patch would leave recovered successful rows carrying the stale `backing session missing` error that status/task views still expose.

## What This Changes

The PR changes task-registry maintenance to recover terminal `lost` cron rows from durable cron run history when the lost error is empty or mentions `backing session missing`, and adds regression tests for recoverable and non-recoverable lost rows.

## Best Possible Solution

Update the PR so durable recovery of a false-positive lost cron row also clears the stale lost error when the durable terminal result has no error, add an assertion for that record shape, then land the narrow maintenance fix and resolve the linked cron audit reports according to their remaining scope.

## Reproduction Assessment

Yes. A high-confidence unit reproduction is to create a terminal `runtime: "cron"`, `status: "lost"`, `error: "backing session missing"` task with a matching durable `finished`/`ok` run log, run task maintenance, and inspect that the recovered row currently remains polluted with the old error.

## Solution Assessment

No as written. Reusing durable cron history is the right narrow approach, but the implementation must clear or replace the stale lost error during successful recovery so task/status surfaces do not continue showing the false-positive message.
