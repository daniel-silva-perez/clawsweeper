Keep open. Current main still skips terminal `lost` cron rows during durable cron recovery, while this PR now implements the narrow false-positive recovery path and addresses the stale-error issue raised in the earlier ClawSweeper review.

## What This Changes

The PR extends task-registry maintenance so selected terminal `lost` cron rows can recover from durable cron history, clears stale errors during successful recovery, and adds focused regression coverage.

## Best Possible Solution

Land this narrow PR after focused task-maintenance validation passes, then use the landed fix to resolve the linked cron false-positive reports while leaving any broader `stale_running` or timestamp cleanup scope in its own follow-up.

## Reproduction Assessment

Yes. A high-confidence unit repro is a `runtime: "cron"`, `status: "lost"`, `error: "backing session missing"` task whose `runId` matches a durable `finished`/`ok` cron run log; current main ignores it because it is not active, and the PR adds this case.

## Solution Assessment

Yes. Reusing existing durable cron history and constraining the terminal recovery to empty or backing-session-missing lost errors is the narrowest maintainable fix; the follow-up explicit error clearing closes the main correctness gap in the first revision.
