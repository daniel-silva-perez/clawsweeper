## Summary

Keep open. The PR targets a valid task-maintenance gap that is still present on current main, but the proposed patch leaves recovered `lost` cron rows with their stale `backing session missing` error when durable history says the run succeeded, so it needs a small corrective update before merge.

## What This Changes

The PR allows durable cron recovery for terminal `lost` cron task rows whose error is empty or mentions `backing session missing`, and adds regression tests for recoverable and non-recoverable lost rows.

## Best Possible Solution

Keep the durable-history recovery approach, but update the recovery path so successful `ok`/`skipped` terminal recovery clears the old lost-task error and add assertions that both inspection projection and applied maintenance no longer expose `backing session missing` after recovery.

## Reproduction Assessment

Yes. The linked reports provide a concrete live reproduction, and the test harness can reproduce it with a `lost` cron task whose `cron:<jobId>:<startedAt>` run log is `finished`/`ok`; current main cannot recover that terminal row.

## Solution Assessment

No, not as written. The chosen recovery boundary is narrow and maintainable, but the implementation should clear stale lost-row errors when durable history proves success before this PR lands.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Clear the stale lost error on recovery:** `src/tasks/task-registry.maintenance.ts:326`
  - body: When this branch lets a `lost` row enter durable recovery, successful run-log recovery returns no `error`, and `markTaskRecovered()` calls `markTaskTerminalById()` without an `error` field. That API merges the patch over the existing task, so the recovered record becomes `status: "succeeded"` while still carrying `error: "backing session missing"`, which remains visible in `tasks show --json` and human output. Clear the old lost error, and assert it in the new recovery test, when the durable result is `ok` or `skipped`.
  - confidence: 0.87
