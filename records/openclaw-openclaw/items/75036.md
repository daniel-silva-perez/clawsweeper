## Summary

Keep open. The PR targets a real cron task-maintenance gap, but the current patch would recover some terminal `lost` cron rows while leaving the stale `backing session missing` error on successful recovered rows, so status/task surfaces can still show the false-positive error after recovery.

## What This Changes

The PR changes task-registry maintenance to allow durable cron recovery for terminal `lost` cron rows with empty or `backing session missing` errors, and adds regression coverage for recoverable and non-recoverable lost rows.

## Best Possible Solution

Update the PR so durable recovery of a false-positive lost cron row clears the stale lost error when the durable terminal result has no error, add an assertion for that record shape, then run the targeted task-maintenance tests and land the narrow fix before resolving the linked cron audit reports by remaining scope.

## Reproduction Assessment

Yes. A high-confidence unit reproduction is to create a terminal `runtime: "cron"`, `status: "lost"`, `error: "backing session missing"` task with a matching durable `finished`/`ok` run log, run task maintenance, and inspect that the recovered row still carries the old error with this patch.

## Solution Assessment

No as written. Reusing durable cron history is the right narrow direction, but the implementation should clear stale lost errors during successful recovery so task/status surfaces stop reporting the false-positive failure.
