## Summary

Keep this PR open. Current main partially addresses the linked user-facing models.json/model-catalog hot-reload bug by clearing the model catalog cache on model-related hot reloads, with regression coverage. It does not implement this PR’s broader intended refactor: there is no config-derived cache registry on main, no gateway-wide invalidateConfigDerivedCaches call, and the context-window runtime cache is still only reset by test helpers rather than by hot reload. The PR also has an unresolved functional review concern about in-flight context-window loading repopulating stale cache entries, so it should be revised or consciously superseded by a narrower maintainer-owned follow-up rather than closed as implemented.

## Best Possible Solution

Keep the PR open for maintainer review, or supersede it with a focused follow-up that preserves the shipped model-catalog hot-reload behavior while adding a safe production invalidation path for context-window state. If the registry approach is kept, add a generation/cancellation guard for async context-window loading and regression coverage for hot reload during an in-flight load.
