## Summary

Current main still leaves the reported low-level Gateway `agent` RPC boundary unfixed: a canonical `agent:<agentId>:...` value supplied only in `params.to` is treated as a delivery/turn target, not as the selected session key. The agent command then derives direct-chat-shaped `to` values through normal target session resolution, which can collapse them to the agent main session and matches the reported WeChat callback misrouting.

## Best Possible Solution

Keep this issue open and land a narrow, plugin-agnostic Gateway fix: when `agent` RPC has no explicit `sessionKey` or `sessionId` and `to` is a canonical agent session key, either route the run to that session or reject it with a clear validation error. Preserve normal `to` delivery-target behavior for ordinary values and add regression coverage for the reported direct-session-key shape.
