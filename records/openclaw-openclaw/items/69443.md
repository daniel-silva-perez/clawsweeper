## Summary

Current main still appears to leave the reported Gateway `agent` RPC boundary unfixed: a canonical `agent:<agentId>:...` value supplied only in `params.to` is treated as a delivery target/source, not as the selected session key, so the run can still resolve through normal direct-chat session logic and collapse to the agent main session. The issue has no protected labels and no open PR found by public GitHub search, so it is a good narrow ClawSweeper repair candidate rather than a cleanup close.

## What This Changes

The issue asks for subagent `gateway call agent` callbacks with `to` set to a canonical WeChat session key to route to that session instead of collapsing to the main session.

## Best Possible Solution

Keep this issue open and land a narrow, plugin-agnostic Gateway fix: when `agent` RPC has no explicit `sessionKey` or `sessionId` and `to` is a well-formed canonical agent session key, route the run to that session or reject it with a clear validation error. Preserve normal `to` delivery-target behavior for ordinary values and add regression coverage for the direct-session-key shape reported here.

- A fix must not reinterpret ordinary channel destinations in `to` as session keys; only well-formed canonical `agent:<agentId>:...` values should get special handling, and only when explicit `sessionKey` and `sessionId` are absent.
