## Summary

Keep open. Current main still exposes the direct Moonshot provider as `moonshot`; `moonshotai` appears only as an external OpenRouter/pricing/catalog slug. Model refs split on the first slash, and there is no `moonshotai` to `moonshot` provider alias, so `moonshotai/kimi-k2.6` remains a real current-main mismatch rather than an implemented or stale report.

## Best Possible Solution

Keep the issue open and implement a backward-compatible direct-provider alias or canonicalization path: accept `moonshotai/<model>` as direct Moonshot while preserving existing `moonshot/<model>` configs, auth profiles, plugin id, and docs. Add regression coverage for model resolution, allowlist/fallback handling, onboarding/docs, and a clearer unknown-model hint if the alias is intentionally rejected.
