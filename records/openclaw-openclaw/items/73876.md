## Summary

Keep open. Current main still canonicalizes Moonshot as `moonshot` only, the reported `moonshotai/kimi-k2.6` direct form still has no provider alias path, and the open linked PR #74946 is an implementation candidate rather than a merged fix.

## What This Changes

Request to accept direct Moonshot model refs using the OpenRouter/MoonshotAI slug `moonshotai/...` while preserving existing `moonshot/...` configs.

## Best Possible Solution

Land a backward-compatible direct-provider alias for `moonshotai/<model>` that routes to canonical `moonshot` without changing existing plugin ids, auth profile keys, provider config keys, or documented canonical refs; review #74946 or an equivalent narrow patch first.

## Reproduction Assessment

Yes. Static inspection is enough to reproduce the resolver path: `moonshotai/kimi-k2.6` is parsed as provider `moonshotai`, current provider normalization does not map it to `moonshot`, and the Moonshot manifest exposes only `moonshot` catalog/provider entries.

## Solution Assessment

No for the issue's hard-rename suggestion. The maintainable solution is a compatibility alias that preserves canonical `moonshot`; #74946's manifest-alias approach looks directionally right but should be reviewed for all runtime resolution paths before merge.
