## Summary

Keep open. Current main still registers direct Moonshot as `moonshot` without a `moonshotai` direct-model alias, and open PR #74946 explicitly targets this issue with closing syntax, so this issue should stay open until that PR is reviewed and merged or replaced.

## What This Changes

Manual Moonshot direct model refs copied as `moonshotai/<model>` from OpenRouter should resolve to the bundled Moonshot provider instead of failing as an unknown model.

## Best Possible Solution

Review and land or replace #74946 with a narrow compatibility alias that preserves canonical `moonshot` ids while accepting `moonshotai/<model>` for direct Moonshot catalog/runtime resolution, with regression coverage for both listing/planning and actual model resolution.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: configure `moonshotai/kimi-k2.6`; current parsing treats `moonshotai` as the provider, and the Moonshot manifest only owns `moonshot`.

## Solution Assessment

Yes for the alias direction, no for the issue's suggested hard rename. The maintainable path is a compatibility alias through the existing manifest catalog alias seam or an equally narrow provider-id normalization path while keeping `moonshot` canonical.
