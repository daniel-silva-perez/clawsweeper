## Summary

Keep open. Current main still lacks a direct `moonshotai` alias for Moonshot models, and open PR #74946 explicitly targets this issue with closing syntax, so the issue should remain open until that PR is reviewed and merged or replaced.

## What This Changes

The issue asks for `moonshotai/<model>` direct Moonshot model refs copied from OpenRouter to resolve compatibly to the canonical `moonshot` provider instead of failing as an unknown model.

## Best Possible Solution

Review #74946 as the active fix candidate, preserving `moonshot` as the canonical provider while adding a backwards-compatible `moonshotai` direct-model alias with regression coverage for model catalog planning and runtime resolution.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: configure `moonshotai/kimi-k2.6`; current parsing treats `moonshotai` as the provider, and current Moonshot metadata has no direct provider or catalog alias for that id.

## Solution Assessment

Yes for the alias direction, no for the issue’s suggested hard rename. The narrow maintainable solution is to keep canonical `moonshot` and add a compatibility alias through the existing manifest catalog alias seam or an equally narrow provider-id normalization path with tests.
