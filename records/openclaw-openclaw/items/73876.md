## Summary

Keep open. Current main still keeps direct Moonshot under the canonical `moonshot` provider id and has no `moonshotai` direct-provider alias, while an open linked PR explicitly targets this issue and should be reviewed or landed before closing.

## What This Changes

Accept `moonshotai/<model>` as a direct Moonshot model reference, or otherwise align Moonshot direct-provider naming with OpenRouter/Moonshot AI slug expectations without breaking existing `moonshot/...` configs.

## Best Possible Solution

Review #74946 as the active fix candidate, preserving `moonshot` as the canonical provider while adding a backwards-compatible `moonshotai` direct-model alias with regression coverage for model catalog planning and runtime resolution.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: configure `moonshotai/kimi-k2.6`; current parsing treats `moonshotai` as the provider, and current Moonshot metadata has no direct provider or catalog alias for that id.

## Solution Assessment

The hard rename suggested in the issue is not the best fix because it would break existing `moonshot/...` users. The safer solution is a compatibility alias, ideally through the existing manifest catalog alias seam or an equally narrow provider-id normalization path with tests.
