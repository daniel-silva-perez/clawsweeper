## Summary

Keep open. Current main still registers direct Moonshot under `moonshot` with no `moonshotai` direct-model alias, and open PR #74946 is still the active unmerged fix candidate that explicitly targets this issue.

## What This Changes

The issue requests consistent direct Moonshot naming so `moonshotai/kimi-k2.6` resolves to the direct Moonshot API while `openrouter/moonshotai/kimi-k2.6` remains routed through OpenRouter.

## Best Possible Solution

Keep `moonshot` as the canonical provider id and land or revise a narrow compatibility alias so `moonshotai/<model>` resolves through the direct Moonshot catalog/runtime/auth path with regression coverage for both catalog planning and actual model selection.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: configure `moonshotai/kimi-k2.6`; current parsing treats `moonshotai` as the provider, provider normalization has no alias, and the Moonshot manifest only owns `moonshot`.

## Solution Assessment

Yes for the compatibility-alias direction, no for the issue's suggested hard rename. The existing manifest alias seam is the narrowest maintainable path if it is verified through runtime resolution and auth, while preserving existing `moonshot/...` configs.
