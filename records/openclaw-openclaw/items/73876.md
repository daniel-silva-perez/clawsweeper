## Summary

Keep open. Current main still exposes Moonshot direct models through the canonical `moonshot` provider id, while `moonshotai/<model>` is not recognized as a direct alias; an open linked PR with closing syntax specifically targets this issue, so this issue should stay open until that implementation is reviewed and merged or replaced.

## What This Changes

The issue asks for direct Moonshot model refs copied as `moonshotai/<model>` to resolve to the direct Moonshot provider while preserving existing `moonshot/<model>` configurations.

## Best Possible Solution

Review the active fix PR and land a backwards-compatible alias that lets `moonshotai/<model>` resolve to the direct Moonshot catalog/runtime path while keeping `moonshot` as the canonical provider id for existing configs, auth profiles, plugin metadata, and docs.

## Reproduction Assessment

Yes. Configure a model as `moonshotai/kimi-k2.6` on current main; the issue context and prior source review show the first path segment is treated as the provider id and no direct `moonshotai` provider or alias exists, producing the reported unknown-model path.

## Solution Assessment

Unclear until PR review. A hard provider rename is not the best solution because it risks breaking existing `moonshot/...` users; the maintainable path is a compatibility alias with regression coverage for both catalog planning and runtime resolution.
