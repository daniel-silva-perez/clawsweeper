## Summary

Keep this open. Current main still implements exec approvals as an allowlist-centered policy with `deny | allowlist | full` security modes and `off | on-miss | always` ask modes; there is no native denylist mode, denylist collection, `ask: "on-match"`, or deny-pattern evaluation before broad allow behavior. The SafeClaw and CRE links, closed PRs #43222/#60684/#72466, and open related tracker #41140 are useful prior art, but they do not replace a merged native OpenClaw policy change, and the latest maintainer discussion explicitly kept this issue open for follow-up.

## What This Changes

The issue requests exec-approvals denylist support so users can allow most commands while blocking or prompting on dangerous command/path/argument patterns.

## Best Possible Solution

Define an additive native exec approvals policy that evaluates deny rules before broad allow decisions, supports consistent command and argument matching across gateway and node `system.run` paths, exposes the policy in CLI and Control UI, documents the security model, and coordinates the overlap with #41140 rather than treating external plugins as the core fix.

## Reproduction Assessment

Not applicable as a runtime bug, but the missing feature has a high-confidence source reproduction path: current main has no accepted `denylist`/`on-match` values, no denylist config field, and no deny-match evaluation branch. I did not run tests because this review was required to remain read-only.

## Solution Assessment

Unclear that the exact proposed `security: "denylist"` plus `ask: "on-match"` shape is the best final API. The maintainable path is a maintainer-designed, additive policy update that preserves protocol compatibility and handles security ordering explicitly.
