## Summary

Keep this open. Current main still lacks native exec-approvals denylist support; the SafeClaw and CRE links plus closed PRs are useful prior art, but no merged core policy, protocol, CLI, UI, or docs implementation accepts a denylist mode or prompt-on-match behavior, and maintainer discussion has already kept the security design open.

## What This Changes

Add exec-approvals denylist support so users can allow most host commands while prompting or blocking specific dangerous command, path, or argument patterns.

## Best Possible Solution

Define an additive native exec approvals policy that evaluates deny rules before broad allow decisions, supports consistent command and argument matching across gateway and node `system.run` paths, exposes the policy in CLI and Control UI, documents the security model, and coordinates the narrower overlap with #41140.

## Reproduction Assessment

Yes. The missing feature is reproducible by source inspection on current main: proposed `denylist`, `security: "denylist"`, `ask: "on-match"`, and `argsMatch` are absent from policy types, normalization, protocol schema, CLI/UI, docs, and evaluation paths; I did not run tests because this review was required to remain read-only.

## Solution Assessment

Unclear that the exact proposed `security: "denylist"` plus `ask: "on-match"` API is the best final design. The safer path is a maintainer-reviewed additive policy with explicit deny-before-allow ordering and compatibility across config, protocol, node/gateway execution, UI, and docs.
