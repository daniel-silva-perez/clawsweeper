## Summary

Keep open. Current main still lacks native exec-approvals denylist mode, denylist storage, prompt-on-match policy, and deny-before-allow evaluation; the existing argument-pattern support is allowlist metadata and does not satisfy the requested denylist behavior. The SafeClaw/CRE links and closed PRs are useful prior art, but this remains a core host-command authorization design item.

## What This Changes

Add a native exec-approvals denylist policy that allows broad execution while prompting or blocking matched dangerous command/path/argument patterns.

## Best Possible Solution

Define an additive native exec approvals policy that evaluates deny rules before broad allow decisions, supports consistent command and argument matching across gateway and node system.run paths, exposes the policy through CLI and Control UI, documents the security model, and coordinates the overlap with #41140.

## Reproduction Assessment

Not applicable: this is a feature/policy request rather than a runtime bug. Source, schema, UI, and docs inspection provides a high-confidence current-main check that the requested denylist/on-match surface is absent.

## Solution Assessment

Unclear: native core support appears to be the right boundary for host exec authorization, but the exact config shape should be decided with maintainer security input before implementation. A safer path is an additive deny-before-allow policy design coordinated with #41140 rather than treating a plugin as the core fix.
