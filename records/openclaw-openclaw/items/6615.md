## Summary

Keep this open. Current main still does not provide native exec-approvals denylist support: the core policy model, normalizers, protocol schema, runtime config schema, CLI, Control UI, docs, and tests remain centered on `deny | allowlist | full` plus allowlist entries. The SafeClaw and CRE references in the discussion are useful prior art, and the closed implementation PRs are relevant design history, but they do not replace a merged native OpenClaw policy change; maintainer discussion has also already kept this security-sensitive design open.

## What This Changes

The issue requests native exec-approvals denylist support, including an allow-everything-except mode, argument/content matching, and prompt-on-match behavior for dangerous command patterns.

## Best Possible Solution

Define an additive native exec approvals policy that evaluates deny rules before broad allow decisions, supports consistent command and argument matching across gateway and node `system.run` paths, exposes configuration in CLI and Control UI, documents the security model, and coordinates the overlap with #41140 before implementation.

## Reproduction Assessment

Yes. The missing feature is reproducible by source inspection on current main: proposed `denylist`, `security: "denylist"`, `ask: "on-match"`, and `argsMatch` are absent from policy types, normalization, protocol schema, runtime config schema, CLI/UI, docs, and evaluation paths; I did not run tests because this review was required to remain read-only.

## Solution Assessment

Unclear that the exact proposed API is the best final design. The maintainable path is a maintainer-reviewed additive policy with explicit deny-before-allow ordering and compatibility across config, protocol, node/gateway execution, UI, and docs.
