## Summary

Keep this issue open. Current main still has native exec approvals modeled around `deny | allowlist | full`, allowlist entries, and `off | on-miss | always`; there is no denylist policy mode, denylist collection, `ask: "on-match"`, or deny-pattern evaluation path. The SafeClaw and CRE links are useful prior art, and the closed PRs show attempted implementations, but they do not replace a merged native policy change; the latest maintainer comment also explicitly kept this open for follow-up.

## What This Changes

Add native exec-approvals denylist support so operators can allow broad execution while prompting or blocking specific command, path, or argument patterns.

## Best Possible Solution

Define an additive native exec approvals policy that evaluates deny rules before broad allow decisions, supports consistent command and argument matching across gateway and node `system.run` paths, exposes the policy in CLI and Control UI, documents the security model, and resolves the overlap with #41140 in one coordinated design.
