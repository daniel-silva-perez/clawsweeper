## Summary

Keep this open. Current main still lacks native exec-approvals denylist support, including a denylist collection, `security: "denylist"`, `ask: "on-match"`, or deny-pattern evaluation before broad allow behavior; the linked SafeClaw/CRE examples and closed implementation PRs are useful prior art, but they do not replace a merged core policy change, and #41140 only covers part of the remaining command-content deny-pattern problem.

## What This Changes

Add native exec-approvals denylist support so operators can allow most host commands while blocking or prompting for specific command and argument patterns.

## Best Possible Solution

Define an additive native exec approvals policy that evaluates deny rules before broad allow decisions, supports command and argument matching consistently across gateway and node `system.run` paths, exposes the policy in CLI and Control UI, documents the security model, and coordinates the overlap with #41140 rather than treating external plugins as the core fix.

## Reproduction Assessment

Yes for the missing feature: current main can be checked by source inspection because `denylist`/`on-match` are not accepted policy values, no denylist field is in the approval file schema, and no deny-match branch exists in the policy path. I did not run tests because this review was required to remain read-only.

## Solution Assessment

Unclear that the exact proposed `security: "denylist"` plus `ask: "on-match"` API is the best final design. The safer path is a maintainer-reviewed, additive policy update with explicit deny-before-allow ordering and protocol/UI/docs compatibility.
