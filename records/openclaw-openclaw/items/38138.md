## Summary

Keep this issue open. Current main still resolves Discord agent routing from static route inputs before mention detection, and Discord mention handling still gates the already selected effective route instead of rerouting to a different mentioned agent. The closest related item in the provided context is broader content-based routing and is not a canonical replacement for this Discord mention override request.

## Best Possible Solution

Keep this issue open. Define the intended precedence for explicit Discord bot mentions and per-agent mentionPatterns versus runtime focus/configured bindings, then implement a Discord mention-based routing override before mention gating. Add a regression for a Discord thread inheriting a parentPeer binding while mentioning another configured agent, and update the Discord and multi-agent docs with the final routing order.

## Risks / Open Questions

- A mention override needs explicit precedence against /focus, runtime thread bindings, configured ACP bindings, static bindings, and parentPeer inheritance so it does not unexpectedly bypass intentional conversation routing.
- Native Discord mentions identify bot user IDs; mapping mentions of other bot identities to agent IDs may need account/identity rules beyond text mentionPatterns.
- Current docs imply per-agent mentionPatterns can help target agents in groups, while current Discord code only applies them to the already selected route.
