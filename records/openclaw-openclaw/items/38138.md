## Summary

Keep this issue open. Current main still resolves the Discord agent route from static route inputs before mention detection; a thread that inherits a parentPeer binding can still select the parent-channel agent, while explicit mentions and per-agent mentionPatterns are only evaluated afterward for gating the already selected effective route. Related #35821 attempted this but was closed unmerged, and broader #13304 is not a canonical replacement for this Discord mention override behavior.

## Best Possible Solution

Keep this open for maintainer product/contract review. The likely fix is a Discord-owned explicit-mention route selection path before mention gating, with a documented precedence rule for real bot mentions and per-agent mentionPatterns versus /focus runtime bindings, configured ACP/channel bindings, parentPeer inheritance, and route-cache behavior. Add regression coverage for a Discord thread inheriting a parentPeer binding while mentioning another configured agent, and update the Discord/channel routing/multi-agent docs once the contract is settled.
