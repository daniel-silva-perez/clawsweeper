## Summary

Keep open. Current main has a useful failure-class-aware fallback foundation and profile cooldown/disable state, but it still does not implement the full requested provider-level auth quarantine policy, invalid-request fail-fast behavior, or provider health/quarantine output in normal status.

## What This Changes

The issue requests failure-class-aware provider fallback that quarantines auth-broken providers, treats rate limits and network failures differently, fails fast on invalid requests, and exposes provider health in status.

## Best Possible Solution

Define a documented provider/profile health model that separates hard auth, transient refresh, rate-limit, network, billing, quota, schema, and invalid-request failures; apply it consistently in fallback selection; decide invalid-request fail-fast boundaries; and expose a redacted provider health snapshot in status while keeping deeper auth/probe details under models status.

## Reproduction Assessment

Yes for the current-main feature gap: source, docs, and tests show classification and cooldown skipping exist, but provider-level auth quarantine, invalid-request fail-fast behavior, and normal status provider health are absent. A live Anthropic OAuth failure is not needed to verify those missing behaviors.

## Solution Assessment

Unclear as an automated fix: the requested direction is reasonable, but the remaining policy choices are not just mechanical code changes. The safer path is maintainer agreement on hard-auth signals, quarantine windows, invalid-request semantics, and status redaction before implementation.
