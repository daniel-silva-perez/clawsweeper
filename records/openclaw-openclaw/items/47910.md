## Summary

Keep open. Current main has a failure-class-aware foundation and skips same-provider candidates once all profiles are already marked unavailable for auth/auth_permanent, but it still does not implement the full requested provider quarantine policy, invalid-request fail-fast behavior, or provider quarantine/health in the normal status surface.

## What This Changes

The issue requests failure-class-aware provider fallback that quarantines auth-broken providers, treats rate limits/network errors/invalid requests differently, and exposes provider health in status.

## Best Possible Solution

Define a documented provider/profile health model that separates hard auth failures from transient refresh, rate-limit, and network failures; apply that state consistently in fallback selection; decide the invalid-request fail-fast policy; and expose a redacted provider health snapshot in status while leaving deeper diagnostics in models status.
