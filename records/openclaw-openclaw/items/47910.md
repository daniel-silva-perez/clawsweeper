## Summary

Keep open. Current main has a useful failure-class-aware fallback foundation and skips same-provider candidates once auth/profile state is already unavailable, but it still does not implement the requested provider-level auth quarantine policy, invalid-request fail-fast behavior, network short-quarantine behavior, or provider health/quarantine output in normal status. Related #61328 covered foundation work only, while open #74909 and #64127 are adjacent policy work rather than a complete fix for this issue.

## What This Changes

The issue requests failure-class-aware provider fallback that quarantines auth-broken providers, treats rate limits and network failures differently, fails fast on invalid requests, and exposes provider health in status output.

## Best Possible Solution

Define a documented provider/profile health model that separates hard auth, transient refresh, rate-limit, network, billing/quota, schema, and invalid-request failures; use it consistently in fallback selection; decide invalid-request fail-fast boundaries; and expose a redacted provider health snapshot in status while keeping deeper auth/probe diagnostics in models status.

## Reproduction Assessment

Yes for the current-main feature gap: source, docs, and tests show classification and cooldown skipping exist, but provider-level auth quarantine, network short quarantine, invalid-request fail-fast, and normal status provider health are absent. A live Anthropic OAuth outage is not needed to verify those missing behaviors.

## Solution Assessment

Unclear as an automated fix: the requested direction is reasonable, but the remaining policy choices are not mechanical. The safer path is maintainer agreement on hard-auth signals, quarantine windows, invalid-request semantics, and status redaction before implementation.
