## Summary

Keep open. Current main has a useful failure-class-aware fallback foundation and same-provider skipping when all auth profiles are already unavailable for auth/auth_permanent, but it still does not implement the full requested provider-level auth quarantine policy, invalid-request fail-fast behavior, network short-quarantine policy, or provider health/quarantine output in normal status. Related #61328 closed the foundation work only, and no open PR found by exact reference directly closes this issue.

## What This Changes

Request provider fallback policy that classifies auth, rate-limit, network, and invalid-request failures, quarantines auth-broken providers, and exposes provider health in status.

## Best Possible Solution

Define a documented provider/profile health model that separates hard auth, transient refresh, rate-limit, network, billing/quota, schema, and invalid-request failures; apply that state consistently in fallback selection; decide invalid-request fail-fast boundaries; and expose a redacted provider health snapshot in normal status while keeping deeper auth/probe diagnostics under models status.

## Reproduction Assessment

Yes for the current-main feature gap: source, docs, and tests show classification and cooldown skipping exist, but provider-level auth quarantine, network short quarantine, invalid-request fail-fast, and normal status provider health are absent. A live Anthropic OAuth outage is not needed to verify those missing behaviors.

## Solution Assessment

Unclear as an automated fix: the requested direction is reasonable, but the remaining policy choices are not mechanical. The safer path is maintainer agreement on hard-auth signals, quarantine windows, invalid-request semantics, and status redaction before implementation.
