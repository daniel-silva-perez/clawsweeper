## Summary

Keep open. Current main has failure-class-aware fallback groundwork and skips same-provider candidates when all profiles are already unavailable for auth/auth_permanent, but it still does not implement the requested hard-auth provider quarantine policy, invalid-request fail-fast behavior, or provider quarantine/health in the normal status surface.

## What This Changes

Request provider fallback to classify failure classes so auth-broken providers are quarantined and skipped, rate limits and network failures get class-specific handling, invalid requests fail fast, and status exposes provider quarantine health.

## Best Possible Solution

Define a documented provider/profile health model that separates hard auth failures from transient refresh, rate-limit, network, billing, quota, and schema failures; apply that state consistently in fallback selection; decide invalid-request fail-fast semantics; and expose a redacted provider health snapshot in normal status while keeping detailed diagnostics under models status.
