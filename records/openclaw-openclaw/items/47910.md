## Summary

Keep open. Current main has a useful failure-class-aware foundation, but it still does not implement the full requested provider quarantine policy, invalid-request fail-fast behavior, or provider health/quarantine output in normal status.

## What This Changes

The issue requests provider fallback by failure class, including auth-broken provider quarantine, rate-limit retry handling, short network quarantine, invalid-request fail-fast behavior, and provider health in status output.

## Best Possible Solution

Define a documented provider/profile health model that separates hard auth, transient refresh, rate-limit, network, billing, quota, schema, and invalid-request failures; apply it consistently in fallback selection; decide invalid-request fail-fast semantics; and expose a redacted provider health snapshot in normal status while keeping deeper probe details under models status.

## Reproduction Assessment

Yes for the current-main feature gap: source, docs, and tests show classification and cooldown skipping exist, but provider-level auth quarantine, invalid-request fail-fast behavior, and normal status provider health are absent. I did not need a live Anthropic OAuth failure to verify the missing behavior.

## Solution Assessment

Unclear as an automated fix: the current foundation is a reasonable start, but the remaining policy choices need maintainer judgment before implementation. A safe solution should first specify hard-auth versus transient-refresh signals, quarantine windows, status redaction, and invalid-request fail-fast boundaries.
