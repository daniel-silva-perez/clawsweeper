## Summary

Keep this issue open. Current main has a real failure-class-aware fallback foundation, including auth/rate-limit classification and same-provider auth cooldown skipping, but it does not fully implement the requested hard-auth quarantine policy or provider-health status surface. In particular, `OAuth token refresh failed` is still treated as ordinary auth, ordinary auth cooldowns are short profile cooldowns rather than a 30-minute provider quarantine, and `openclaw status`/`/status` still lacks the requested provider health/quarantine snapshot.

## Best Possible Solution

Keep this issue open as the focused follow-up to #61328. The best path is a maintainer decision on hard-auth policy first, then a narrow implementation that distinguishes permanent re-auth-required OAuth failures from transient refresh timeout/contention, applies the intended provider/profile quarantine window, exposes secret-safe provider health/quarantine state in the intended `openclaw status` or `/status` surface, and updates model-failover/status docs plus focused tests. If maintainers decide the current short auth cooldown plus `models status --json` is the intended product behavior, document that policy and then close this issue as intentionally covered.

## Risks / Open Questions

- Changing OAuth refresh failures into longer quarantine without a maintainer policy could over-quarantine transient refresh timeouts or lock contention; the existing code already distinguishes `auth_refresh`, `refresh_timeout`, and `refresh_contention` for messaging, so the policy should use those distinctions deliberately.
- Adding provider health to status must avoid leaking credential material, profile IDs that reveal emails, or raw provider error bodies; the existing `models status` and doctor paths are safer starting points than raw logs.
