## Summary

Keep open. Current main has a useful failure-class-aware fallback foundation, including 401/429 classification, auth-profile cooldown/disabled state, and same-provider auth skipping once every profile for that provider is unusable. The requested end state is still not implemented: there is no fixed provider-level auth quarantine policy, invalid-request/format errors are still documented as failover-worthy, and normal status JSON does not expose provider health or quarantine state. Closed #61328 covered the foundation, not the remaining product behavior.

## What This Changes

Request failure-class-aware provider fallback that quarantines auth-broken providers, handles rate limits and network errors differently, fails fast on invalid requests, and exposes provider health in status.

## Best Possible Solution

Define a documented provider/profile health model that separates hard auth, transient OAuth refresh, rate-limit, overloaded, network, billing, schema, and invalid-request failures; apply it consistently in fallback selection; decide invalid-request fail-fast semantics; and expose a redacted provider health snapshot in normal status while keeping deeper auth/profile diagnostics under models status.
