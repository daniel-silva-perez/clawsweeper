## Summary

Keep open. Current main has failure classification and some cooldown-aware fallback behavior, but it still does not implement the requested provider-level auth quarantine, invalid-request fail-fast policy, network short-quarantine policy, or provider health/quarantine output in normal status.

## What This Changes

The issue requests failure-class-specific provider fallback behavior, including auth-broken provider quarantine, Retry-After-aware rate limiting, network short quarantine, invalid-request fail-fast handling, and provider health in status output.

## Best Possible Solution

Define a documented provider/profile health model that separates hard auth, transient refresh, rate-limit, network, billing/quota, schema, and invalid-request failures; apply that state consistently in fallback selection; and expose a redacted provider health snapshot in status while keeping deeper auth/probe diagnostics in models status.

## Reproduction Assessment

Yes for the current-main feature gap: source, docs, and tests show classification and cooldown skipping exist, but provider-level auth quarantine, network short quarantine, invalid-request fail-fast, and normal status provider health are absent. A live Anthropic OAuth outage is not needed to verify those missing behaviors.

## Solution Assessment

Unclear as an automated fix: the requested direction is reasonable, but the remaining policy choices are not mechanical. The safer path is maintainer agreement on hard-auth signals, quarantine windows, invalid-request semantics, and status redaction before implementation.
