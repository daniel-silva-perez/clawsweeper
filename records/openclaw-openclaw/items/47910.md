## Summary

Keep open. Current main has the failure-class fallback foundation and skips same-provider auth cooldowns, but it still does not fully implement the requested hard-auth quarantine policy or expose provider health/quarantine state through `openclaw status`/`/status`.

## What This Changes

The issue requests failure-class-specific provider fallback behavior that quarantines auth-broken providers, handles rate limits/network/invalid requests differently, and exposes provider health in status output.

## Best Possible Solution

Define and document the intended hard-auth policy, then implement a secret-safe provider/profile health snapshot that distinguishes re-auth-required failures from transient refresh contention/timeouts and applies the chosen quarantine window consistently across fallback and status surfaces.
