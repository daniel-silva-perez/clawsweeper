## Summary

Keep this issue open. Current main has a solid failure-class foundation and already skips providers whose profiles are unavailable for auth/auth_permanent reasons, which addresses part of the latency problem raised in the issue and comments. It does not yet implement the full requested behavior: generic 401/auth failures are still short cooldowns rather than a 30-minute auth-broken quarantine, invalid 400/422 request handling still feeds fallback in current policy, Retry-After-specific behavior was not evident in the model fallback path, and provider quarantine health is surfaced mainly through model-auth/doctor paths rather than the requested /status provider-health view.

## Best Possible Solution

Keep the existing failure-class foundation, then implement or explicitly reject the remaining policy details in a focused follow-up: decide which OAuth/401 signals become auth_permanent/quarantined, honor Retry-After for rate-limit windows where provider responses expose it, decide whether format/400 errors should fail fast or keep current fallback behavior, and surface provider/profile quarantine state in /status or document models status as the canonical health view.

- The requested operator-facing /status health snapshot is not clearly implemented; users may still need to know to run models status or doctor-auth.
