## Summary

Keep open. Current main has failure-class-aware fallback primitives and same-provider auth cooldown skips, but it still does not implement the requested hard-auth provider quarantine, invalid-request fail-fast policy, or provider health/quarantine state in the normal status surface.

## What This Changes

The issue requests failure-class-aware provider fallback that quarantines auth-broken providers, handles rate limits/network/invalid requests differently, and exposes provider health in status output.

## Best Possible Solution

Define a documented provider/profile health model that separates transient refresh failures from re-auth-required failures, applies the chosen quarantine window consistently in fallback selection, and exposes a redacted health snapshot through the intended status surface while keeping `models status` as the deeper auth diagnostic view.
