## Summary

Keep this issue open. Current main has a failure-class-aware fallback foundation and same-provider auth cooldown skipping, but it still does not fully implement the requested auth-broken provider quarantine policy or expose provider health/quarantine state through `openclaw status`/`/status`. The closed related item #61328 covers the broader foundation, not the remaining hard-auth quarantine and status-snapshot behavior requested here.

## What This Changes

Request provider fallback behavior that classifies failures, quarantines auth-broken providers, handles rate limits/network/invalid requests differently, and exposes provider health in status output.

## Best Possible Solution

Keep this issue open as the focused follow-up to #61328. The best path is a maintainer policy decision on hard-auth classification first, then a narrow implementation that distinguishes permanent re-auth-required OAuth/auth failures from transient refresh contention/timeouts, applies the intended provider/profile quarantine window, exposes secret-safe provider health/quarantine state in the chosen status surface, and updates model-failover/status docs plus focused tests. If maintainers decide the current short auth cooldown plus `openclaw models status --json` is the intended behavior, document that policy and then close this issue intentionally.
