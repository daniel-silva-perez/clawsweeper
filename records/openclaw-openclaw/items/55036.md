## Summary

Keep open. Current main still suppresses commentary-phase assistant output before it can reach Discord draft preview, partial reply, block reply, or final reply delivery paths, and current regression tests explicitly lock in that suppression. The closed related main-result latency report covers a different final-result timing problem, not this early-commentary durability request.

## What This Changes

Discord should commit early user-visible assistant commentary or status text immediately as its own visible message so it is not delayed or lost across gateway restarts.

## Best Possible Solution

Define an explicit, safe contract for user-visible status commentary, keep raw `commentary` suppressed, and route only approved status text through embedded subscribe plus Discord draft/dispatcher delivery with restart-sensitive regression coverage.
