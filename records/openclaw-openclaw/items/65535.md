## Summary

Keep open. Current main still lacks a blocking plugin interception point for user-visible streamed preview creates/edits or in-place final preview edits; normal outbound sends run `message_sending`, but the streaming preview paths can send or edit visible channel messages directly. The linked Discord before-deliver PR is useful related context but was closed unmerged, and the per-chat streaming-mode request is adjacent rather than a substitute for a content-filtering hook.

## What This Changes

The issue requests a blocking plugin hook or equivalent streaming override so plugins can inspect, modify, or cancel streamed outbound content before it becomes visible in chat.

## Best Possible Solution

Define a shared, documented blocking pre-delivery contract for streamed preview creates, preview edits, and final preview edits, then wire it through Telegram and the generic draft-preview finalizer with channel-specific tests. If maintainers also want an operations workaround, add scoped streaming-disable controls separately rather than treating them as the content-filtering fix.

## Reproduction Assessment

Yes. A high-confidence reproduction path is a Telegram plugin that registers `message_sending`, enables `channels.telegram.streaming.mode="partial"`, and observes that partial/final preview edits use `draftStream.update` and `editMessageTelegram` while bypassing `deliverReplies`, where the blocking hook currently runs.

## Solution Assessment

Unclear. The issue’s options are directionally valid, but a final-only `message_edit_sending` hook is not sufficient for the sensitive-data use case because partial previews can already expose content; the maintainable solution should cover every user-visible preview update with explicit metadata and cancellation semantics.

## Risks / Open Questions

- A final-only hook would not prevent sensitive content that appears in earlier partial preview edits from becoming visible first.
- Running `message_sending` on every preview update could add latency, rate-limit pressure, and repeated plugin side effects unless the contract is explicit and tested.
- The fix likely needs a shared contract across Telegram, Discord, Slack, Mattermost, and possibly Signal delivery paths so channel behavior does not diverge.
