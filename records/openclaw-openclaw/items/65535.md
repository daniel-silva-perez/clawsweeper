## Summary

Keep open. Current main still lacks a plugin-facing interception point for Telegram streaming preview edits/finals: normal final sends run `message_sending`, but preview finalization can edit/materialize the streamed message without running `message_sending`, and the existing test explicitly expects only the internal `message:sent` hook for that path. The later Signal comment is also plausible because Signal same-surface replies use a local delivery path that does not call the shared outbound hook pipeline.

## Best Possible Solution

Add an additive plugin-facing interception seam for streamed preview delivery, either by extending `message_sending` with `isStreamingEdit`/`isFinal` metadata or by adding a new hook, then wire it through Telegram preview final/edit/materialize paths and the Signal same-surface reply path. Include tests for cancel, content rewrite, final-preview `message_sent` behavior, and fallback normal delivery.
