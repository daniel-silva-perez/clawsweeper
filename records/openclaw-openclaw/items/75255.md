## Summary

Keep open. Current main has a bundled Xiaomi provider, but it only wires model catalog/auth usage and TTS; it does not declare Xiaomi webSearch config or append Xiaomi's provider-native web_search tool. The reporter's follow-up adds useful live Xiaomi API evidence, including the official endpoint/key distinction and response shape, so this is a concrete narrow provider-plugin feature rather than stale cleanup.

## What This Changes

Request support for Xiaomi MiMo's provider-native web_search tool by adding Xiaomi provider configuration and injecting a web_search tool into Xiaomi chat-completions requests when enabled.

## Best Possible Solution

Add a Xiaomi-owned provider-plugin implementation that declares plugin-scoped webSearch settings, gates the feature to supported Xiaomi OpenAI-compatible chat models and official endpoint/key behavior, appends the native web_search tool while preserving existing function tools, documents plugin activation and billing implications, and adds focused Xiaomi provider tests.

## Reproduction Assessment

Not applicable as a bug reproduction, but there is a high-confidence absence check: current main has no Xiaomi webSearch config schema and no Xiaomi runtime payload hook that can append the native web_search tool. The reporter also supplied a live working Xiaomi API example that defines expected request and response behavior.

## Solution Assessment

Yes, the narrow maintainable direction is provider-owned Xiaomi support using the existing manifest config and provider payload wrapping surface. A broader core abstraction should wait until the Xiaomi implementation proves the current hook is insufficient or multiple providers need the same generic native-tool contract.
