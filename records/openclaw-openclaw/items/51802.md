## Summary

Keep open. Current main still does not implement this PR's central behavior: the Chat New session button continues to send `/new` through the current-session reset path, and the session selector still lacks the PR's recent-first ordering and Lit `live(...)` binding. The PR remains a plausible implementation candidate, but it needs rebase and maintainer UX review before it can land.

## Best Possible Solution

Keep this PR open as potentially useful work. The best path is to rebase it onto current main, implement the independent Control UI new-session flow in the current chat/session-controls structure, add focused tests for unique key generation, `sessions.patch` creation, immediate session switching, recent-first selector ordering, and selector binding stability, then verify the Control UI/WebChat mutation boundary stays intact.
