## Summary

Keep open. Current main still preserves explicit inline dreaming mode but hardcodes light/REM inline output to `memory/{YYYY-MM-DD}.md`, and neither the memory-core manifest schema nor the host resolver exposes an inline path or target option. The request is valid, but it changes a memory-core storage/config contract that docs currently describe as internal, so it needs maintainer API/product judgment before an automated repair PR.

## What This Changes

The issue requests a configurable memory-core inline dreaming write destination so managed light/REM blocks can target Obsidian-style `memory/daily-memory/{date}.md` instead of hardcoded `memory/{date}.md`.

## Best Possible Solution

If maintainers want this workflow, add a memory-core-owned storage option that defaults to `memory/{date}.md`, validates a workspace-relative date template or explicit target enum, passes the resolved path through the host config to the inline writer, preserves the existing managed block markers, and documents the Obsidian daily-notes interaction.
