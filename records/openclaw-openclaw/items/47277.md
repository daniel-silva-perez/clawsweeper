## Summary

Keep this PR open. Current main still has shared memory discovery, indexing, and read behavior with no user-scoped identity parameter, no `memory.isolation` config contract, and no public docs/tests for per-user memory directories. The PR is still a plausible implementation candidate, but the discussion shows it is privacy-sensitive and needs maintainer review and rework before merge.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is a revised memory-core and memory-host-sdk design that enforces scoped identity consistently for builtin and QMD backends, memory_search, memory_get, memory writes/flushes, session-key/requesterSenderId derivation, path-safe user IDs, disabled-isolation compatibility, manager lifecycle, config schema/help/docs, and focused regression tests.

## Risks / Open Questions

- This PR changes a privacy boundary for multi-user memory. A partial implementation could create cross-user reads through memory_search, memory_get, QMD collections, shared root memory files, compaction flushes, or path normalization gaps.
- Default-on isolation is a config and behavior contract change; it needs schema/help/docs compatibility review and tests for disabled legacy behavior.
- Per-user manager/cache designs can create unbounded SQLite connections, file watchers, or timers unless the lifecycle is bounded or the manager design scopes per request.
- The provided diff does not appear to add CI workflow, dependency, lockfile, release, or package-publishing changes, so the main security concern is data isolation rather than supply-chain expansion.
