## Summary

Keep this PR open. Current main still does not implement the PR’s central behavior: unknown `openclaw.json` keys still flow into the strict Zod schema and become validation failures on the base read/startup path. The PR remains a plausible implementation candidate, with useful prior review fixes around union fail-closed behavior, legacy aliases, and warning sanitization, but it needs a current-main rebase or narrower rewrite plus security/config-safety review before merge.

## Best Possible Solution

Keep this PR open for maintainer follow-up. Rebase or rewrite it narrowly against current main, implementing read/startup tolerance for unknown config keys with sanitized warnings while preserving fail-closed behavior for invalid known fields, legacy keys, plugin schemas, and union/discriminator mismatches. Review `.secrets.baseline` separately, then run focused config validation/write/legacy tests plus the required changed gate for the final touched files.
