## Summary

Keep this PR open. Current main still does not implement the PR’s central behavior: unknown config keys reach strict Zod validation, base validation returns a fatal failure, plugin-aware validation cannot recover it, and startup still throws INVALID_CONFIG. The related predecessor and alternate PRs in the provided context are closed and unmerged, so they do not supersede this implementation candidate.

## Best Possible Solution

Keep this PR open for maintainer follow-up. Rebase or rewrite it narrowly against current main to add sanitized warning-based tolerance for unknown read/startup config keys while preserving fail-closed behavior for invalid known fields, legacy keys, plugin schemas, unsupported SecretRef policy issues, and union/discriminator mismatches. Keep doctor repair semantics distinct, review `.secrets.baseline` separately, then run focused config validation/write/legacy tests plus the required changed gate for the final touched files.
