## Summary

Keep open. Current main still does not implement the `</arg_value>` suffix stripping this PR is trying to add, and the linked bug #48780 remains the canonical user problem. The PR is still a plausible implementation candidate, but it needs a rebase/redesign because the proposed `normalizeToolParams()` integration point no longer exists on main and review comments identify real data-integrity gaps.

## Best Possible Solution

Keep the PR open, but require a rebase/redesign against current main: add a narrowly scoped sanitizer at the current tool-argument boundary for command/path-like fields, cover exec `command`, `workdir`, `host`, `security`, `ask`, and `node` intentionally, and add regression tests for Qwen/DashScope suffixes plus negative tests proving `content`, `oldText`, and `newText` are preserved when they legitimately end with XML-like text.

## Risks / Open Questions

- Closing this PR now would leave the linked Qwen/DashScope `</arg_value>` tool-argument corruption unimplemented on current main.
- Merging the PR as-is would risk silently truncating legitimate write/edit XML or text payloads because the proposed helper strips every string value in a record.
- A current-main fix needs to choose the right sanitization boundary now that `normalizeToolParams()` is gone, and should cover exec string parameters deliberately.
