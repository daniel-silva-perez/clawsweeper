## Summary

Keep this PR open. Current main still lacks the `</arg_value>` suffix cleanup this PR is trying to add, and the active read/write/edit and exec paths still pass those string arguments through without this sanitizer. The PR remains useful, but it needs a current-main rework because its `normalizeToolParams()` integration point is gone and the proposed record-wide stripping could corrupt legitimate write/edit payload text.

## Best Possible Solution

Keep this PR open for maintainer/contributor rework. The best fix is a narrowly scoped sanitizer at the active tool-argument boundary: cover read paths and intentional exec string params such as `command`, `workdir`, `host`, `security`, `ask`, and `node`, while preserving legitimate `content`, `oldText`, and `newText` values. Add regression tests for Qwen/DashScope suffixes plus negative tests for valid XML-like payload text, then close #48780 only after the fixed implementation lands.

## Risks / Open Questions

- Closing this PR now would leave the linked Qwen/DashScope `</arg_value>` tool-argument corruption unimplemented on current main.
- Merging the PR as-is risks data corruption for legitimate write/edit XML or text payloads because it strips every string value in a params record.
- The branch needs a current-main rework because `normalizeToolParams()` no longer exists in the active implementation.
- Security pass: the PR does not add workflows, dependencies, lifecycle hooks, package-resolution changes, or secret access. The security-sensitive surface is exec argument mutation, which should be narrowly scoped and tested so commands and policy fields are not changed incorrectly.
