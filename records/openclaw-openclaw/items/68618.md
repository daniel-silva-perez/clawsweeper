## Summary

Keep open. Current main still has only the contact-backed Feishu sender-name lookup and the `resolveSenderNames` opt-out; there is no schema, docs, or runtime path for an operator-provided open_id-to-display-name alias map. The adjacent #13751 and #73399 work overlaps sender-name resolution but does not supersede the explicit alias/override request.

## What This Changes

Request to add a Feishu sender-name alias mapping so configured open_id values can display custom names or fallbacks when API lookup is unavailable.

## Best Possible Solution

Add the feature, if accepted, inside the Feishu plugin rather than core: define a typed/account-aware alias map in the Feishu config schema, apply it in the sender-name resolver without affecting ID-based authorization, document the precedence, and cover DM/group fallback behavior with targeted Feishu tests.

## Reproduction Assessment

Not applicable as a runtime bug report; this is a feature request. Static verification is high-confidence: current main has no `senderNameMap`/alias field and the sender label path still depends on API resolution or falls back to the sender id.

## Solution Assessment

Unclear. A Feishu-owned config map is a plausible narrow addition, but it should be coordinated with the existing least-privilege sender-name work in #13751 and #73399 so aliasing, fallback order, cache scope, and attribution semantics land consistently.
