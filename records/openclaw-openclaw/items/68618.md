## Summary

Keep open: current main still lacks a Feishu sender alias map. The Feishu plugin only documents and implements `resolveSenderNames`, the resolver still depends on Feishu API lookups with permission-error fallback, and the adjacent open Feishu display-name PR does not add the requested `senderNameMap` surface.

## What This Changes

Request adding a Feishu channel configuration map from sender IDs to display aliases for API fallback or override in DMs and groups.

## Best Possible Solution

Keep this open and implement the feature in the Feishu plugin, not core: add a documented, per-account-capable alias map to the Feishu config contract, apply it only to display attribution, preserve ID-based authorization behavior, and cover API-success, API-failure, DM, group, and opt-out semantics in focused Feishu tests.

## Reproduction Assessment

Yes. The current source-level reproduction is high-confidence: `senderNameMap` is absent from the strict Feishu runtime schema and the inbound sender-name path has no alias lookup before falling back to the raw sender ID.

## Solution Assessment

Yes, broadly. A Feishu-owned config option is the narrow maintainable boundary, but maintainers should decide alias precedence and coordinate with the open Feishu sender-name fallback PR before implementation.
