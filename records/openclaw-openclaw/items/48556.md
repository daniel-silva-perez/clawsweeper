## Summary

Keep open. Current main still has not implemented this PR's intended change in the core proxy fetch wrappers: `makeProxyFetch()` and `resolveProxyFetchFromEnv()` pass the caller `init.body` straight into `undiciFetch` while only replacing the dispatcher. Current main does have FormData normalization in `runtime-fetch` and a Discord-specific workaround, but neither changes the public proxy helper used by media transcription and plugin SDK callers. The linked same-author issue #48554 is still open, so this PR remains a paired implementation candidate rather than cleanup to close.

## Best Possible Solution

Keep the PR open for maintainer review or replacement by a current-main patch. The best path is to rebase the fix, share or reuse the existing runtime FormData normalization where practical, apply it to both `makeProxyFetch()` and `resolveProxyFetchFromEnv()`, clear stale multipart headers, add targeted proxy-wrapper tests for global FormData conversion and undici FormData passthrough, and coordinate with #48554 and the overlapping #66243 work.

## Risks / Open Questions

- The fix touches network/proxy dispatch, so a revived patch should preserve current fetch-guard, dispatcher, and SSRF trust boundaries rather than duplicating divergent proxy behavior.
- The existing runtime FormData normalizer also clears stale `content-length` and `content-type`; a proxy-fetch fix that only swaps the body could still leave multipart header bugs.
- The submitted implementation has a review-noted filename edge case: `blobValue.name || undefined` drops an explicitly empty filename. That should be addressed if the patch is revived.
