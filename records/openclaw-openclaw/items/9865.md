## Summary

Keep open. Current main still lacks first-class Batch API routing for delayed/background LLM agent turns, and the existing issue discussion treats this as the canonical tracker while related closed reports are narrower context rather than shipped fixes.

## What This Changes

Request opt-in provider Batch API routing for non-real-time background agent work such as cron jobs, spawned subagents, scheduled maintenance, and heartbeat checks.

## Best Possible Solution

Keep this as the canonical product/API tracker for an additive delayed/batch agent-turn contract: public Plugin SDK/provider runtime lifecycle support, protocol/config opt-ins for cron, sessions.send, sessions_spawn/subagents, and heartbeat/background runs, plus provider implementations and docs for fallback, cancellation, accounting, provider eligibility, and delivery behavior.

## Reproduction Assessment

Not applicable as a feature request, but there is a high-confidence current-main verification path: inspect the strict cron/sessions/subagent schemas and provider runtime hooks, which show no batch opt-in or delayed-turn lifecycle.

## Solution Assessment

Unclear until maintainers choose the contract. The maintainable path is not a one-off Anthropic or OpenAI cron flag first; it should start with an additive provider/runtime and public Plugin SDK seam that provider plugins can implement consistently.
