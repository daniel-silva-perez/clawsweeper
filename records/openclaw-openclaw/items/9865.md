## Summary

Keep open. Current main still lacks first-class provider Batch API routing for delayed/background agent turns, and the issue discussion already treats this as the canonical tracker while the related closed reports are narrower duplicate context rather than shipped fixes.

## What This Changes

The issue requests opt-in use of provider Batch APIs for latency-tolerant background LLM work such as cron jobs, sub-agents, heartbeats, and scheduled maintenance.

## Best Possible Solution

Keep this as the canonical product/API tracker for an additive delayed/batch agent-turn contract: public SDK/provider runtime support, protocol/config opt-ins for cron, sessions_send, sessions_spawn/subagents, and heartbeat/background runs, plus OpenAI and Anthropic provider implementations and docs for fallback, cancellation, accounting, and delayed delivery.
