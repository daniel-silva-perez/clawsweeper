## Summary

Keep open. Current main still lacks first-class Batch API routing for delayed/background agent turns, and the issue discussion already treats this as the canonical tracker while #49326, #56126, and #70606 are narrower closed duplicate context rather than shipped fixes.

## What This Changes

The issue requests opt-in provider Batch API support for non-real-time background work such as cron jobs, sub-agents, scheduled maintenance, and heartbeat checks.

## Best Possible Solution

Keep this as the canonical product/API tracker for an additive delayed/batch agent-turn contract: public SDK and provider runtime lifecycle support, protocol/config opt-ins for cron, sessions_send, sessions_spawn/subagents, and heartbeat/background runs, plus provider implementations and docs for fallback, cancellation, accounting, provider eligibility, and delayed delivery.

## Reproduction Assessment

Not applicable as a feature request. There is a high-confidence current-main verification path: inspect the strict cron, agent, sessions_send, sessions_spawn schemas and provider runtime hooks, which show no batch opt-in or delayed-turn lifecycle.

## Solution Assessment

Unclear until maintainers choose the contract. The maintainable path is an additive provider/runtime and public Plugin SDK seam first, not a one-off OpenAI or Anthropic cron flag that would bypass shared cancellation, delivery, accounting, and fallback behavior.
