## Summary

Keep open. Current main still promotes configured `tools.fs` and `tools.exec` sections into profile-stage allowances, so restrictive profiles can receive filesystem/runtime tools contrary to the documented profile contract; the active related PR is still open and appears only partly aligned with the broader messaging/exec follow-up.

## What This Changes

The issue reports that restrictive tool profiles, especially `minimal` and `messaging`, still receive filesystem or runtime tools when `tools.fs` or `tools.exec` blocks are configured unless those groups are explicitly denied.

## Best Possible Solution

Make `tools.fs` and `tools.exec` blocks configure behavior only for tools already granted by the selected profile or explicit `allow`/`alsoAllow`, keep deny precedence, and add regression coverage for both `minimal` and `messaging` profiles before closing this issue.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence reproduction path: configure `tools.profile: "minimal"` or `"messaging"` with `tools.fs` or `tools.exec`, then `resolveEffectiveToolPolicy()` adds those tools to `profileAlsoAllow` before runtime profile filtering.

## Solution Assessment

Unclear for the active PR. Guarding `minimal` addresses the title, but the reporter and #47487 show the same policy class for `messaging`/exec, so the safer fix is to remove or profile-gate all implicit section grants while preserving explicit `alsoAllow`.
