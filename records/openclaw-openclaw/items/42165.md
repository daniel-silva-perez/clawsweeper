## Summary

Keep open. Current main still re-adds configured filesystem/runtime sections into the profile-stage allowlist, which contradicts the documented restrictive profile contract; the prior exact PR was closed unmerged and the active related PR has not landed.

## What This Changes

The issue reports that restrictive `tools.profile` values such as `minimal` and `messaging` still receive filesystem or runtime tools when `tools.fs` or `tools.exec` config sections are present.

## Best Possible Solution

Keep the issue open until the policy resolver treats `tools.fs` and `tools.exec` blocks as configuration for already-granted tools, preserves explicit `allow`/`alsoAllow` and deny precedence, and adds regression coverage for both `minimal` and `messaging` profiles.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence reproduction path: configure `tools.profile: "minimal"` or `"messaging"` with `tools.fs` or `tools.exec`, then `resolveEffectiveToolPolicy()` adds those tools to `profileAlsoAllow` before runtime filtering.

## Solution Assessment

No for current main. The narrow maintainable fix is to remove or profile-gate implicit section grants so section configuration cannot bypass the selected restrictive profile; #43061 should be reviewed against the broader messaging/exec case before this issue closes.
