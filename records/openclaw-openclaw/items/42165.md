## Summary

Current main still appears to contain the reported least-privilege regression. `tools.fs` and `tools.exec` section presence is converted into `profileAlsoAllow` without checking whether the effective profile is `minimal` or `messaging`, and current tests still assert filesystem tools are re-exposed under `messaging`. The issue should remain open while the implementation candidate is reviewed or split.

## Best Possible Solution

Keep this issue open. The best path is to fix `src/agents/pi-tools.policy.ts` so `tools.fs` and `tools.exec` configure already-granted tools but do not implicitly re-grant filesystem/runtime tools under restrictive profiles unless explicitly requested via `alsoAllow`, then update tests that currently assert the leaking behavior and add regression coverage for `minimal` and `messaging`. Review or split #43061 as the active implementation candidate.

## Risks / Open Questions

- This remains a least-privilege boundary bug: restrictive profiles can regain filesystem or runtime tools when config sections are present unless users explicitly deny the group.
- The open candidate PR bundles this fix with unrelated bugs, so maintainers should review, split, or land only the relevant tool-profile change with focused regression coverage.
- The fix needs to preserve explicit `alsoAllow` behavior and deny precedence while covering both global and per-agent `tools.fs` / `tools.exec` config under `minimal` and `messaging`.
