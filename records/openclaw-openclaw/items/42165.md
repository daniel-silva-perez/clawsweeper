## Summary

Keep open. Current main still promotes configured filesystem/runtime sections into the profile-stage allowlist, so restrictive profiles can receive tools outside the documented contract; an open PR already targets the issue and should be reviewed rather than queueing a separate repair job.

## What This Changes

The issue asks OpenClaw to keep `tools.profile: "minimal"` and other restrictive profiles from implicitly exposing filesystem or runtime tools solely because `tools.fs` or `tools.exec` config sections exist.

## Best Possible Solution

The end state should treat `tools.fs` and `tools.exec` blocks as configuration for tools that are already granted by the selected profile or explicit allowlist, preserve explicit `alsoAllow` and deny precedence, and add regression coverage for global and per-agent restrictive profiles.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence reproduction path: configure `tools.profile` as `minimal` or `messaging` with `tools.fs` or `tools.exec`, then `resolveEffectiveToolPolicy()` builds `profileAlsoAllow` and the runtime merges those entries before filtering.

## Solution Assessment

No for current main. The documented contract is stricter than the implementation; the maintainable fix is to stop section presence from acting as an implicit grant and review or replace #43061 if it does not cover the broader runtime/messaging case.
