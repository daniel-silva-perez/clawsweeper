## Summary

Keep open. Current main still widens restrictive tool profiles through implicit profile-stage grants from configured `tools.fs` and `tools.exec` blocks, which conflicts with the docs/catalog contract, and the active related PR is still unmerged.

## What This Changes

The issue reports that `tools.profile: "minimal"` and related restrictive profiles can expose filesystem or runtime tools unless users add explicit deny rules.

## Best Possible Solution

Make `tools.fs` and `tools.exec` sections configure behavior only for tools already granted by the effective profile or explicit allow/alsoAllow policy, preserve deny precedence, and add regression coverage for both `minimal` and `messaging`.

## Reproduction Assessment

Yes. Static inspection of current main gives a high-confidence reproduction path: configure a restrictive profile with `tools.fs` or `tools.exec`, then `resolveEffectiveToolPolicy()` adds those tools to `profileAlsoAllow` before profile filtering.

## Solution Assessment

Yes, the narrow maintainable fix is to remove or profile-gate implicit config-section expansion while keeping explicit `alsoAllow` working. The current open PR should be reviewed for full coverage or split/replaced if it only handles part of the profile surface.
