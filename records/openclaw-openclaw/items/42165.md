## Summary

Keep open. Current main still promotes configured filesystem/runtime sections into the profile-stage allowlist, so restrictive profiles can receive tools outside the documented contract; the related fix PR remains open and the earlier exact PR closed unmerged.

## What This Changes

The issue reports that restrictive `tools.profile` values expose filesystem or runtime tools when `tools.fs` or `tools.exec` sections are configured, even without an explicit allowlist grant.

## Best Possible Solution

The end state should treat `tools.fs` and `tools.exec` blocks as configuration for tools that are already granted by the selected profile or explicit allowlist, preserve explicit `alsoAllow` and deny precedence, and add regression coverage for global and per-agent `minimal`/`messaging` profiles with both filesystem and runtime sections.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence reproduction path: configure `tools.profile` as `minimal` or `messaging` with `tools.fs` or `tools.exec`, then `resolveEffectiveToolPolicy()` builds `profileAlsoAllow` and the runtime merges those entries before filtering.

## Solution Assessment

No for current main. The documented contract is stricter than the implementation; the maintainable fix is to stop section presence from acting as an implicit grant and review or replace #43061 if it does not cover the broader runtime/messaging case.
