## Summary

Current main still appears to violate the documented restrictive-profile contract: profile selection excludes filesystem/runtime tools, but the effective policy resolver still implicitly adds them from `tools.fs` and `tools.exec` config blocks before final filtering. The item should stay open while maintainers handle the already-open implementation candidate and related exec report.

## What This Changes

The issue reports that restrictive `tools.profile` values, especially `minimal` and later `messaging`, should not receive filesystem or runtime tools merely because corresponding config blocks exist.

## Best Possible Solution

Keep the issue open until the policy treats `tools.fs` and `tools.exec` blocks as configuration for already-granted tools, not as implicit grants for restrictive profiles; preserve explicit `alsoAllow` and deny precedence, add regression coverage for `minimal` and `messaging`, and review or split #43061 rather than duplicating the work.
