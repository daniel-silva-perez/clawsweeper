## Summary

Keep open because current main still lacks iMessage typing indicators and this PR is the live, unmerged implementation path. It is neither obsolete nor implemented on main, and unresolved review findings make conservative cleanup inappropriate.

## What This Changes

The PR adds an iMessage typing helper around `imsg typing` and rewires the iMessage monitor to use the typing-aware reply dispatcher during inbound reply generation.

## Best Possible Solution

Keep the legacy iMessage typing feature in this PR or a narrow replacement, but land it through the existing typing lifecycle with a reconciled imsg transport choice, preserved option/config precedence, serialized cleanup, and focused iMessage tests for target selection, typing modes, graceful degradation, and concurrency.

## Risks / Open Questions

- The submitted diff currently drops `opts.cliPath` and `opts.dbPath` precedence for typing calls, which can make typing use a different `imsg` binary or database than message delivery.
- The CLI-only implementation rationale conflicts with upstream `imsg` v0.5.0 exposing `typing.start` and `typing.stop` RPC methods, so the transport choice needs maintainer agreement.
- Unserialized start/stop subprocesses can leave a stale typing state or spawn overlapping commands, especially with SSH wrapper `cliPath` setups.
