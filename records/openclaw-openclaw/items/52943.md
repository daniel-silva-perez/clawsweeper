## Summary

Keep this PR open. Current main still lacks native imsg typing indicators, and the PR is a relevant implementation candidate, but the submitted patch is not merge-ready because it has concrete correctness defects, a merge conflict, and an unresolved transport choice now that upstream imsg v0.5.0 exposes both CLI and RPC typing support.

## What This Changes

The PR adds an iMessage typing helper around `imsg typing` and rewires the iMessage monitor to start and stop typing indicators during inbound reply generation.

## Best Possible Solution

Land iMessage typing support through this PR or a narrow replacement that uses the existing typing lifecycle, preserves option/config precedence, deliberately chooses the imsg transport, serializes start/stop cleanup, handles documented `dbPath` forms consistently with the RPC client, degrades cleanly on older imsg versions, and adds focused iMessage coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Remove the stale reply dispatcher import:** `extensions/imessage/src/monitor/monitor-provider.ts:37`
  - body: After this diff, `monitor-provider.ts` no longer uses `createReplyDispatcher`; leaving that import alongside the new typing dispatcher import will fail the repo's unused-import/typecheck gate. Remove the old import when switching this call site over.
  - confidence: 0.96
- **[P2] Reuse the resolved iMessage CLI and DB paths:** `extensions/imessage/src/monitor/monitor-provider.ts:412-413`
  - body: The new inner `cliPath` and `dbPath` bindings read only from account config, bypassing the outer variables that already include `opts.cliPath` and `opts.dbPath`. Programmatic callers and tests can then deliver replies through one imsg binary/database while typing indicators use another.
  - confidence: 0.94
- **[P2] Serialize typing subprocess calls:** `extensions/imessage/src/monitor/monitor-provider.ts:438`
  - body: `onReplyStart` fires `sendTyping(true)` without awaiting or gating the returned promise. Slow `imsg typing` calls, especially through SSH wrappers, can overlap with keepalives and the final stop, leaving a late start after cleanup.
  - confidence: 0.86
- **[P2] Resolve the DB path before spawning imsg typing:** `extensions/imessage/src/typing.ts:29`
  - body: The existing RPC client resolves `dbPath` with `resolveUserPath`, and the public config reference allows `~/Library/Messages/chat.db`. Passing that literal value directly as `--db` means no shell expands it, so typing can fail or use a different database while normal sends still work.
  - confidence: 0.82
