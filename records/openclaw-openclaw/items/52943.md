Keep this PR open. Current main still lacks iMessage typing indicators, and the PR is useful source work, but the submitted diff has unresolved correctness issues and a transport-choice conflict with upstream imsg v0.5.0 that need maintainer review before merge.

## What This Changes

The PR adds an iMessage typing helper around `imsg typing` and rewires the iMessage monitor to use the typing-aware reply dispatcher during inbound reply generation.

## Best Possible Solution

Land iMessage typing support through this PR or a narrow replacement that uses the existing typing lifecycle, preserves option/config precedence, chooses an imsg transport deliberately, serializes cleanup, degrades cleanly on older imsg versions, and adds focused iMessage coverage.

## Risks / Open Questions

- The submitted diff currently drops `opts.cliPath` and `opts.dbPath` precedence for typing calls, which can make typing use a different imsg binary or database than message delivery.
- The CLI-only implementation rationale conflicts with upstream imsg v0.5.0 exposing `typing.start` and `typing.stop` RPC methods, so the transport choice needs maintainer agreement.
- The PR fires typing start/stop subprocesses without awaiting or otherwise gating them, so slow wrappers can leave overlapping start/stop calls and stale typing state.
- The PR adds no focused iMessage tests for target selection, typing modes, graceful degradation, option precedence, or cleanup ordering.
