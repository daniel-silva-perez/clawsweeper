## Summary

Keep this PR open: current main still does not implement iMessage typing indicators, but the submitted patch has blocking correctness issues and needs maintainer review before merge.

## What This Changes

The PR adds an iMessage typing helper around `imsg typing` and rewires the iMessage monitor to start and stop typing indicators during inbound reply generation.

## Best Possible Solution

Land iMessage typing support through this PR or a narrow replacement after preserving option/config precedence, resolving configured paths consistently with the RPC client, serializing start/stop cleanup, deliberately choosing CLI versus RPC transport, degrading cleanly on older imsg versions, and adding focused iMessage coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Remove the stale reply dispatcher import:** `extensions/imessage/src/monitor/monitor-provider.ts:37`
  - body: After this diff switches the call site to `createReplyDispatcherWithTyping`, `createReplyDispatcher` is no longer used in this file. Leaving the old import will fail the repo's unused-import/typecheck gate.
  - confidence: 0.96
- **[P2] Reuse the resolved iMessage CLI and DB paths:** `extensions/imessage/src/monitor/monitor-provider.ts:412-413`
  - body: The new inner `cliPath` and `dbPath` bindings read only from account config, bypassing the outer values that already include `opts.cliPath` and `opts.dbPath`. Programmatic callers can then deliver replies through one imsg binary/database while typing indicators use another.
  - confidence: 0.94
- **[P2] Serialize typing subprocess calls:** `extensions/imessage/src/monitor/monitor-provider.ts:438`
  - body: `onReplyStart` fires `sendTyping(true)` without awaiting or gating the returned work. Slow `imsg typing` calls can overlap with keepalives and the final stop, leaving a late start after cleanup.
  - confidence: 0.86
- **[P2] Resolve DB paths before spawning typing:** `extensions/imessage/src/typing.ts:29`
  - body: The existing RPC client expands configured `dbPath` values with `resolveUserPath`, and docs allow `~/Library/Messages/chat.db`. Passing that literal value directly as `--db` means no shell expands it, so typing can fail or use a different database than normal sends.
  - confidence: 0.82
