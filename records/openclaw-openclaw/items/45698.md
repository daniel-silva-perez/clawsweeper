## Summary

Keep this open. Current main has backend channel-status fan-out, but the UI-side contributors described in the report are still present: the Channels tab can still await an 8s status request, browser RPCs still have no abort/stale-response hook, node polling still starts globally on connect, and session-change pushes still reload sessions immediately. Two open PRs already cover overlapping slices, so this should go to maintainer reconciliation rather than closure or another automated repair branch.

## What This Changes

The issue asks for the Control UI to stay responsive during long-lived dashboard sessions even when channel status probes, provider reconnects, or session-event churn are slow.

## Best Possible Solution

Land one reconciled Control UI responsiveness fix that scopes polling to active views, debounces or coalesces session reloads, ignores or aborts stale long-running tab requests, keeps slow channel status probes from blocking broader navigation, and adds regression coverage for slow status responses plus session-event churn.

## Reproduction Assessment

Unclear for a fully deterministic end-to-end reproduction: it requires a long-lived dashboard and a slow or reconnecting provider. The contributing source paths are high-confidence on current main, including the 8s channel status wait, global node polling, non-abortable browser RPCs, and immediate sessions.changed reload behavior.

## Solution Assessment

No, current main is not yet the best solution. The backend channel-status concurrency work helps, but the UI still lacks the narrower stale-request, active-view polling, and session-reload controls needed to resolve the reported progressive stuck behavior.
