## Summary

Keep this issue open. Current main still contains the reported Control UI responsiveness risks: the Channels refresh path waits up to 8s for `channels.status`, node polling still starts globally on connect with the original short polling intervals, `sessions.changed` still triggers full session reloads, and the relevant open PRs remain implementation candidates rather than proof of a merged fix.

## What This Changes

The issue requests that the Control UI remain responsive over long-lived sessions when channel status checks, websocket reconnects, polling, and session refreshes are slow or noisy.

## Best Possible Solution

Land a focused Control UI churn fix that scopes polling and refreshes to active views, debounces or coalesces session reloads, ignores or aborts stale long-running tab requests, preserves log tail behavior, and adds regression coverage for slow `channels.status` and stale in-flight refreshes.

## Reproduction Assessment

Unclear for a fully deterministic end-to-end reproduction because it needs a long-lived dashboard with a slow or unstable channel provider. The contributing paths are high-confidence from current source: the 8s channel status request, global node polling, and session reload-on-event behavior are still present.

## Solution Assessment

No, current main is not yet the best solution because it only shows partial backend mitigation for channel-status fan-in. The maintainable path is to reconcile the existing UI-focused PRs with explicit stale-request and slow-status regression tests before closing this report.
