## Summary

Current main has partial backend mitigation for `channels.status`, but the reported Control UI responsiveness problem is not fully resolved: the UI still requests `channels.status` with an 8s timeout, keeps aggressive/global polling paths, reloads sessions on every `sessions.changed`, and the two directly related implementation PRs remain open/unmerged.

## Best Possible Solution

Keep this open and let maintainers reconcile #45930 and #69454 into one accepted Control UI churn fix, or close those PRs with a replacement plan. The fix should scope polling and eager refreshes to active views, debounce or coalesce `sessions.changed` reloads, add stale-response or abort semantics for slow channel refreshes, preserve logs tail behavior, and add regression coverage for slow `channels.status` plus repeated session-change events before this report closes.
