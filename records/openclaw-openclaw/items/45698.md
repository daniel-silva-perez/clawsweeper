## Summary

Keep this issue open. Current main still contains the main Control UI churn paths described in the report: Channels still waits on an 8s `channels.status` request, stale generic gateway requests cannot be aborted, node polling still starts globally on connect, and `sessions.changed` still triggers session reloads without a dedicated debounce. The backend channel status path has partial concurrency mitigation, but the UI-side responsiveness work is not fully landed and overlapping PRs remain open.

## What This Changes

The issue asks for the Control UI to stay responsive over long-lived dashboard sessions even when channel status checks are slow and WebSocket reconnects or session events churn.

## Best Possible Solution

Land one reconciled Control UI churn fix that scopes polling to active views, debounces or coalesces session reloads, ignores or aborts stale long-running tab requests, keeps slow `channels.status` from blocking broader navigation, and adds regression coverage for slow status responses plus session-event churn.

## Reproduction Assessment

Unclear for a fully deterministic end-to-end reproduction: it needs a long-lived dashboard plus a slow or reconnecting provider. The contributing paths are high-confidence from current source and tests: 8s channel status waits, global node polling, non-abortable browser RPCs, and reload-on-`sessions.changed` behavior remain present.

## Solution Assessment

No, current main is not yet the best solution. The backend channel-status concurrency work is useful, but the UI still lacks the narrower stale-request, active-view polling, and session-reload controls needed to resolve the reported progressive stuck behavior.
