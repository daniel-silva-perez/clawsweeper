## Summary

Keep this issue open. Current main has a partial backend mitigation for channel status fan-out, but the Control UI still has the reported high-latency shape: Channels tab refreshes still issue an 8s `channels.status` probe, the browser RPC layer has no abort/timeout option for stale UI requests, and the aggressive polling intervals targeted by related open PR #45930 are still present on main.

## Best Possible Solution

Keep the issue open and use it as the canonical bug for Control UI progressive sluggishness. The best fix is a focused UI/backend validation path: land or replace #45930 if its polling changes still hold, add UI-side cancellation or stale-response guards for channel/status refreshes, consider shorter per-provider status timeouts or partial status rendering, and verify with a slow channel provider fixture plus the related large-session/reconnect scenarios.
