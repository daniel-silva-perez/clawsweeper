## Summary

Keep open. Current main still treats `gateway.nodes.denyCommands` as free-form strings in the config schema and the `openclaw config set` mutation path reaches the file write without command-registry validation; the existing security audit catches unknown or pattern-like entries only after persistence. The issue is also paired with open PR #50076 using closing syntax, with broader overlapping work in #50320, so it should remain open until one focused fix lands.

## What This Changes

Request write-time validation or warnings for invalid `gateway.nodes.denyCommands` entries when users mutate config, instead of only reporting them later in `openclaw security audit`.

## Best Possible Solution

Land a narrow post-mutation validator for config writes that touch `gateway.nodes.denyCommands`, `gateway.nodes.allowCommands`, or ancestor `gateway.nodes` objects. It should reuse the full built-in and dangerous node-command vocabulary plus same-update custom `allowCommands`, surface clear suggestions for unknown entries, preserve plugin/custom extensibility, and add focused config CLI tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: a typo such as `sendd` in `gateway.nodes.denyCommands` passes the current config schema and reaches `replaceConfigFile`, while the security audit helper would later flag the same value as unknown. I did not run a live config write because this review is read-only.

## Solution Assessment

Yes, the best fix belongs in the config mutation path rather than as a static schema enum, because command names can be runtime- or plugin-influenced. The direction in open PR #50076 matches that shape, but maintainers still need to review the warning/failure policy and overlap with #50320.
