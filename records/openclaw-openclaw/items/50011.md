## Summary

Keep open. Current main still accepts `gateway.nodes.denyCommands` as a free-form string array and `openclaw config set` writes the updated config without deny-command registry validation or warning. The security audit path does validate unknown/pattern-like deny entries, and current main has improved that audit logic to recognize valid dangerous commands, but the reported write-time feedback is still missing. The issue is also paired with open PR #50076 using closing syntax, so it should remain open until that PR or an equivalent focused fix lands.

## Best Possible Solution

Keep this issue open and review or land PR #50076, or replace it with an equivalent narrow fix. The fix should validate the post-change config when direct or ancestor `openclaw config set` writes touch `gateway.nodes.denyCommands`, warn on unknown or pattern-like entries, reuse the audit suggestion logic where practical, preserve valid dangerous command names, and allow custom commands introduced through `allowCommands` in the same update.
