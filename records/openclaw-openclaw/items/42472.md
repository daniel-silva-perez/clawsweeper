## Summary

Keep this issue open. Current main still preserves the reported mixed/trailing `NO_REPLY` behavior in the shared reply normalizer, outbound planning, and gateway history surfaces; only the cron announce path has a narrower scoped suppression fix. The issue is also paired with open implementation PRs, including one with closing syntax, so the next step is maintainer review/consolidation rather than closure or a new repair lane.

## What This Changes

The issue requests suppressing the entire model reply when the raw output contains or ends with `NO_REPLY`, instead of stripping only the token and delivering preceding text.

## Best Possible Solution

Land one canonical fix, or a replacement, that defines a shared silent-intent helper and applies it consistently across `normalizeReplyPayload`, reply directives/outbound delivery planning, gateway live/history projection, cron/subagent fallbacks, and runtime delivery classification, with tests for exact tokens, own-line/trailing tokens, inline false positives, media preservation, and streaming limits.
