## Summary

Keep this issue open. Current main still does not implement the requested shared suppression rule: the common reply normalizer strips a trailing mixed `NO_REPLY` token and returns the preceding text, while outbound and gateway-history tests still preserve mixed `NO_REPLY` payloads. Cron announce delivery has a narrower hardening path, but that does not cover the general reply/outbound/gateway surfaces, and open implementation PRs remain under review.

## What This Changes

The issue requests post-processing that treats model output containing or ending with `NO_REPLY` as a silent acknowledgement so prefixed internal text is not delivered to users.

## Best Possible Solution

Review and land one canonical fix, or a replacement, that adds a shared silent-intent helper and applies it consistently across `normalizeReplyPayload`, reply directive/outbound delivery planning, gateway live/history projection, cron/subagent fallbacks, and runtime delivery classification, with tests for own-line/trailing suppression, inline false positives, media preservation, and exact-token behavior.
