## Summary

Keep open. Current main still delivers the preamble before a trailing `NO_REPLY` through the shared reply normalizer and directive/outbound paths, while only cron has a narrower trailing-token suppression path. This issue is also paired with open implementation PRs using closing syntax, so it should stay open until maintainers consolidate and land one fix.

## What This Changes

Suppress the entire assistant reply when model output includes a trailing or own-line `NO_REPLY` after preamble text, preventing partial internal summaries from reaching users.

## Best Possible Solution

Define one shared silent-intent helper with explicit rules for exact tokens, JSON envelopes, glued leading tokens, own-line/trailing control tokens, and ordinary inline mentions, then reuse it across reply normalization, directive parsing, outbound delivery projection, gateway live/history projection, cron, and subagent follow-up paths.

## Reproduction Assessment

Yes. Current source and tests give a high-confidence reproduction: `normalizeReplyPayload({ text: "File's there. Not urgent.\n\nNO_REPLY" })` is still expected to return a non-null payload containing the preamble.

## Solution Assessment

Unclear as originally suggested. A raw `/NO_REPLY/` match is too broad for legitimate inline mentions; the safer solution is a shared helper with explicit trailing/own-line control-token semantics applied consistently across delivery paths.
