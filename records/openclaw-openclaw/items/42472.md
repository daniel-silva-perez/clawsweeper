## Summary

Keep open. Current main still strips a trailing `NO_REPLY` in the shared auto-reply normalizer and returns the preamble as deliverable text, and an existing regression test still asserts that behavior. Cron announce delivery has a narrower path-specific suppression, but the shared normalizer, directive parsing, and live/history projection surfaces are not consistently fixed; several open PRs already target overlapping versions of the same leak, so maintainers should consolidate those before this issue closes.

## What This Changes

The issue requests that model replies containing or ending with `NO_REPLY` suppress the entire visible message so pre-token summaries or internal text do not leak to users.

## Best Possible Solution

Adopt one shared silent-intent helper with explicit semantics for exact tokens, strict JSON envelopes, leading-token fragments, and agreed own-line/trailing-token control cases, then reuse it across normalization, directive parsing, outbound delivery projection, gateway live/history projection, cron, and subagent delivery while preserving ordinary inline mentions.

## Reproduction Assessment

Yes. Current source and tests give a high-confidence reproduction: `normalizeReplyPayload({ text: "File's there. Not urgent.\n\nNO_REPLY" })` is expected to return a non-null payload containing the preamble.

## Solution Assessment

Unclear as originally suggested. A raw `/NO_REPLY/` rule is too broad; the safer maintainable fix is a shared helper with explicit own-line/trailing-control semantics applied consistently across all delivery paths.
