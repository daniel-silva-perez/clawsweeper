## Summary

Keep open. Current main still strips a trailing `NO_REPLY` token and returns the preceding text in the shared auto-reply normalizer, and the existing regression test still codifies that leak-shaped behavior. There are overlapping open implementation PRs that reference this problem, so the issue should remain open until maintainers choose and land a consolidated fix.

## What This Changes

Request that replies containing a trailing or own-line `NO_REPLY` after model-generated preamble text be treated as fully silent instead of delivering the preamble.

## Best Possible Solution

Define one shared silent-intent helper for exact tokens, strict JSON envelopes, leading-token fragments, and agreed own-line/trailing-token semantics; apply it consistently across normalization, directive parsing, outbound delivery projection, gateway live/history projection, and cron/subagent delivery with focused regression tests and a small docs note for the `NO_REPLY` contract.

## Reproduction Assessment

Yes. Current source and tests provide a high-confidence reproduction: `normalizeReplyPayload({ text: "File's there. Not urgent.\n\nNO_REPLY" })` is expected to return a non-null payload containing the preamble.

## Solution Assessment

Yes, the requested shared post-processing direction is appropriate, but suppressing every occurrence of `NO_REPLY` would be too broad. The maintainable fix should use explicit own-line/trailing-token semantics and preserve non-control inline mentions.
