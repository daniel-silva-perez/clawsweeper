## Summary

Keep open. Current main still strips a trailing `NO_REPLY` token in the shared auto-reply normalizer and delivers the preceding text, while tests still assert that behavior; cron has a narrower path-specific suppression but the shared reply, directive, outbound, and live projection surfaces are not consistently covered. Open implementation PRs, including #42873 with closing syntax, already target this problem, so maintainers should consolidate or land one of those before this issue is closed.

## What This Changes

Request suppressing the entire reply when model output includes a trailing or own-line `NO_REPLY` after partial/internal text, instead of delivering the preamble with only the token removed.

## Best Possible Solution

Define one shared silent-intent helper for exact tokens, strict JSON envelopes, leading-token fragments, and agreed own-line/trailing-token semantics, then apply it consistently across normalization, directive parsing, outbound delivery projection, gateway live/history projection, cron, and subagent delivery while preserving ordinary inline mentions.

## Reproduction Assessment

Yes. The current source and tests give a high-confidence reproduction: `normalizeReplyPayload({ text: "File's there. Not urgent.\n\nNO_REPLY" })` is expected to return a non-null payload containing the preamble.

## Solution Assessment

Unclear as written. The shared post-processing direction is right, but a raw `/NO_REPLY/` suppression rule is too broad; the safer maintainable fix is explicit own-line/trailing-control semantics reused across all delivery paths.
