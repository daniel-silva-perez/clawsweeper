## Summary

Keep open. Current main still strips a trailing `NO_REPLY` token in the shared auto-reply normalizer and then delivers any preceding text, and gateway live/history projection paths still only suppress exact control replies or leading fragments. Cron announce delivery has a narrower trailing-token suppression fix, but the broader shared reply behavior requested here remains unresolved and is already paired with open implementation PRs.

## What This Changes

The issue requests suppressing an entire assistant reply when the raw output contains or ends with `NO_REPLY`, instead of stripping only the token and delivering preceding text.

## Best Possible Solution

Define one shared silent-intent helper for exact tokens, strict JSON envelopes, leading-token fragments, and agreed own-line/trailing-token semantics, then apply it consistently in reply normalization, directive parsing, outbound delivery projection, gateway live/history projection, cron/subagent delivery, and runtime delivery classification with focused regression tests and docs updates for the `NO_REPLY` contract.
