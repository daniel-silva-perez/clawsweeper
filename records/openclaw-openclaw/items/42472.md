## Summary

Keep this issue open. Current main still preserves mixed/trailing NO_REPLY text in the shared reply normalizer, outbound directive parsing, and gateway history expectations, while only narrower cron and silent-expected paths have been hardened. The issue is also paired with open implementation PRs, including one with closing syntax, so it needs maintainer consolidation rather than closure or a new repair job.

## What This Changes

The issue requests suppressing the entire user-visible reply when a model response contains or ends with NO_REPLY, instead of stripping only the token and delivering preceding text.

## Best Possible Solution

Land one canonical fix, or a consolidated replacement, that defines a shared silent-intent helper and applies it consistently across reply normalization, reply directives/outbound delivery planning, gateway live/history projection, cron/subagent delivery, and runtime delivery classification, with tests for exact tokens, own-line/trailing tokens, inline false positives, media preservation, and streaming limits.
