## Summary

Keep open. Current main still treats a trailing or own-line `NO_REPLY` in the shared auto-reply path as a token to strip, then delivers the preceding text; the issue is valid and already has overlapping open implementation PRs that should be reviewed together.

## What This Changes

Suppress the entire assistant reply when model output includes a trailing or own-line `NO_REPLY` signal instead of delivering the preamble text.

## Best Possible Solution

Define one shared silent-intent helper for exact tokens, strict JSON envelopes, leading-token fragments, and agreed own-line/trailing-token semantics; apply it consistently across normalization, directive parsing, outbound delivery projection, gateway live/history projection, and cron/subagent delivery with focused regression tests and a small docs note for the `NO_REPLY` contract.

## Reproduction Assessment

Yes. The current source and tests provide a high-confidence reproduction: `normalizeReplyPayload({ text: "File's there. Not urgent.\n\nNO_REPLY" })` is expected to return a non-null payload containing the preamble.

## Solution Assessment

Yes, the requested shared post-processing direction is appropriate, but suppressing every occurrence of `NO_REPLY` would be too broad. The maintainable fix should use a shared helper with explicit own-line/trailing-token semantics and preserve non-control inline mentions.
