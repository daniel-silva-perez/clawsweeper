## Summary

Keep this issue open. Current main appears to have fixed the Control UI model read-back regression, but the restart-sentinel channel notice still has no suppression surface and the BOOT.md silent-token behavior still needs maintainer direction.

## What This Changes

The issue asks for a suppressible restart-sentinel channel notification, correct Control UI model read-back after saving, and a non-conflicting BOOT.md silent-reply token contract.

## Best Possible Solution

Keep the issue open but narrow the remaining work: add a documented restart-sentinel notification policy/config key that suppresses only user-channel notices while preserving internal wake/continuation behavior, and separately decide whether BOOT.md should always use the canonical NO_REPLY token or gain a documented configurable silent-token contract.

## Reproduction Assessment

Yes. The restart-sentinel behavior is reproducible from current source and tests by scheduling a routed sentinel wake, which enqueues and delivers the notice; the BOOT.md behavior is reproducible by inspecting buildBootPrompt, which appends NO_REPLY after arbitrary BOOT.md content.

## Solution Assessment

No, closing this issue would not be the best outcome because two reported areas still need source-level follow-up. The Control UI model read-back portion appears solved, but the sentinel suppression and BOOT token contract should be handled deliberately rather than treated as implemented.
