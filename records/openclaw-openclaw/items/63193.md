## Risks / Open Questions

- The proposed workflow would build signed Android release artifacts in public GitHub Actions using repository secrets and contents: write, then clobber GitHub release assets; maintainers need to decide whether that belongs in public CI or a private release lane.
- The workflow downloads Android SDK command-line tools with curl/unzip and no checksum verification, so the supply-chain posture needs release/security review before merge.
- The proposed README publishes keystore secret names and Play upload-key reset flow while current release policy keeps credentials, signing, and recovery details maintainer-only; the public/private documentation boundary needs owner approval.
- The new android-release workflow is not covered by current release CODEOWNERS rules on main, so ownership and required review routing should be made explicit if this lane lands.
