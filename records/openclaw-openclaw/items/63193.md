## Risks / Open Questions

- The proposed workflow would build signed Android release artifacts in a public repository workflow using repository secrets and `contents: write`; maintainers need to decide whether this belongs in public CI or a private release lane.
- The workflow downloads Android SDK command-line tools with `curl` and unzips them without an explicit checksum verification step in the PR patch.
- `gh release upload --clobber` can overwrite existing release assets and checksum files for the supplied tag, so release managers should confirm or harden overwrite behavior before merge.
- The new Android release workflow path is not covered by the current release CODEOWNERS rules shown on main, so ownership/security routing should be added or otherwise confirmed.
