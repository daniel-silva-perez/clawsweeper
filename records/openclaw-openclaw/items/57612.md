## Summary

Keep this PR open. Current main still does not implement native DOCX text extraction for `input_file` or `read`: DOCX is only recognized as a MIME/storage type, the OpenResponses file allowlist and docs still list text/JSON/PDF only, the current extraction path is PDF-only, and the read wrapper has no DOCX buffer/extraction path. The PR discussion contains useful follow-up fixes and review hardening notes, so the best next step is maintainer review/adaptation rather than cleanup closure.

## Best Possible Solution

Keep this PR open for maintainer review. The best product/code path is to adapt the DOCX extractor into the existing `document-extract` plugin and generic document-extractor routing, add DOCX to the `input_file` defaults and public docs, wire `read` through a bounded DOCX-only extraction path, and preserve the PR’s regression coverage for paragraphs, tables, numeric entities, malformed archives, default MIME handling, `input_file`, and `read`. The related open PR #57600 is a narrower binary-office read guard and does not supersede native DOCX extraction.

## Risks / Open Questions

- Security review should stay focused on untrusted DOCX ZIP/XML parsing: compressed-file size alone is not enough, so uncompressed `word/document.xml` limits, malformed archive fallback, and output byte caps need to remain in the final design.
- The PR touches the high-use `read` path; if adapted, DOCX extraction should be gated narrowly so normal text/image reads keep adaptive paging, bounded I/O, and clean filesystem error behavior.
- The implementation likely needs rebasing/adaptation to the current plugin-owned `document-extract` architecture rather than adding a core-only `src/media` extractor that bypasses the document-extractor contract.
