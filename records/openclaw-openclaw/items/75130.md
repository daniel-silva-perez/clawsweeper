## Summary

Current main already closes the reported pre-ready WS dispatch window: gateway startup now rejects `connect` while sidecars are pending with a retryable startup-specific `UNAVAILABLE`, and clients retry instead of surfacing a terminal handshake failure. The reporter's #74864 cross-reference is useful startup-lifecycle context, but this specific WS pre-ready mechanism is covered by the startup-sidecars gate.

## What This Changes

The issue requests that gateway WS agent dispatch be gated until plugin tool registration and gateway readiness complete, avoiding pre-ready agent runs that see an empty tool registry.

## Best Possible Solution

Keep the current startup-sidecars handshake gate and client retry contract; stable users affected on `v2026.4.27` should move to a build containing `8d58ad4c15cd3723ececd26697b86339a5af2834` or newer rather than changing tool allowlist behavior.

## Reproduction Assessment

Yes. The issue gives a concrete restart-window log timeline, and current-main regression coverage exercises the same gate by forcing startup pending and verifying the WS client receives retryable startup `UNAVAILABLE` before any agent request can dispatch.

## Solution Assessment

Yes. Gating at WS `connect` is the narrowest maintainable fix for this report because it prevents all authenticated WS request dispatch during startup-sidecar readiness, while preserving normal client retry behavior instead of special-casing the tool registry error.
