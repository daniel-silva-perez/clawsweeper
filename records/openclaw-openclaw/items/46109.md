## Summary

Keep this open. Current main has substantial SecretRef runtime, CLI, config-write, redaction, and docs support, but the requested Control UI/config UX is still missing: structured SecretRefs are rendered as read-only blank generic placeholders, SecretInput unions are collapsed to the string editor path, and docs still describe form-mode limitations rather than source-aware config/env/unset badges or resolved status.

## What This Changes

The issue requests first-class .env/environment-backed secret configuration UX so secret fields can show config/env/unset source state, preserve env references, and avoid blank-looking Control UI fields.

## Best Possible Solution

Define a safe SecretRef metadata contract, then add a source-aware SecretInput renderer and supporting metadata flow that shows masked source labels, env variable identity where allowed, resolved/unresolved state, and explicit config/env/unset editing, with docs and tests aligned to the existing secrets CLI and config-write preflight behavior.

## Security Review

Status: needs_attention

Summary: No PR diff was reviewed, but this secret-management UX request needs focused security review before implementation.

Concerns:

- **[medium] Define safe SecretRef metadata disclosure:**
  - body: Showing env var names, provider aliases, config paths, or resolved/unresolved status can reveal deployment structure even when values stay masked. Maintainers should decide the safe disclosure boundary before exposing this metadata broadly in Control UI.
  - confidence: 0.87
- **[medium] Avoid transporting resolved secret values to the UI:**
  - body: The requested resolved/unresolved UX should use bounded metadata or existing preflight/status seams, not raw resolved secret material, so the browser does not receive more sensitive data while improving clarity.
  - confidence: 0.82
