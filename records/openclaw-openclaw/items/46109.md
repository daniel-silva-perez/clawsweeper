## Summary

Keep this issue open. Current main has strong SecretRef runtime, CLI, validation, and documentation foundations, but the requested first-class Control UI/config UX is still missing: env-backed SecretRefs render as read-only blank generic placeholders, SecretInput unions are collapsed to the string editor path, and the docs describe the limitation rather than source-aware config/env/unset controls.

## What This Changes

Request first-class .env/environment-backed secret configuration UX in CLI and Control UI, including masked source-aware fields, env reference preservation, and a Secrets/Environment helper.

## Best Possible Solution

Define a safe SecretRef metadata contract first, then implement a source-aware SecretInput renderer and supporting metadata flow that shows masked source labels, allowed env variable identity, resolved/unresolved status, and explicit config/env/unset editing, with Control UI, CLI/docs, and regression tests aligned.

## Reproduction Assessment

Not applicable as a runtime bug report, but the current-main UX gap is high-confidence. A config form containing an env SecretRef such as `channels.discord.token` currently renders as a read-only blank field with a generic Structured value placeholder, and the browser test asserts that behavior.

## Solution Assessment

Unclear until maintainers decide the security disclosure boundary. The maintainable solution is a bounded metadata contract plus UI/docs/tests, not a cosmetic placeholder change that guesses which secret source details are safe to expose.
