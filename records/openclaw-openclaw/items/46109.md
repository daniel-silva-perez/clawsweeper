## Summary

Keep this issue open. Current main has SecretRef runtime, CLI, redaction, restore, and docs foundations, but the requested source-aware Control UI/config UX is still missing: structured SecretRefs render as read-only empty inputs with a generic placeholder, and SecretInput unions still normalize to the plain string editor path.

## What This Changes

The issue requests first-class .env/SecretRef-backed secret configuration UX so CLI and Control UI surfaces show config/env/unset source state, preserve env references, and avoid blank-looking secret fields when runtime resolution works.

## Best Possible Solution

Define a redacted SecretRef metadata contract first, then implement a source-aware SecretInput renderer and aligned CLI/docs/tests that show masked source labels, safe env identity when allowed, resolved/unresolved state, and explicit config/env/unset editing without sending resolved secrets to the browser.

## Reproduction Assessment

Yes. The high-confidence reproduction path is the current Control UI form rendering path for an env SecretRef such as `channels.discord.token`, which existing code and tests show as a read-only empty input with a generic Structured SecretRef placeholder.

## Solution Assessment

Unclear until the disclosure boundary is set. The maintainable solution is a bounded metadata contract plus UI/docs/tests, not a placeholder-only change that guesses which secret-source details are safe to expose.
