## Summary

Keep this issue open. Current main has strong runtime and CLI SecretRef support, plus a partial Control UI safeguard that prevents SecretRef objects from being stringified or corrupted. It does not yet implement the requested first-class env-backed secret UX: source badges, env-var name display, config/env/unset state, Control UI switchers, or a dedicated resolved/unresolved secrets helper.

## Best Possible Solution

Keep the issue open and build a focused Control UI/CLI follow-up on top of the existing SecretRef contracts: render secret fields with `config` / `env` / `unset` source state, show safe metadata such as env var id and resolved/unresolved status, add controls to switch between inline config, SecretRef, and unset, and reuse the current `openclaw secrets` audit/configure target registry for the dedicated secrets/environment helper.

## Risks / Open Questions

- Any implementation should preserve the existing redaction and restore semantics so SecretRef ids and plaintext values are not leaked or flattened into config writes.
- The UI should distinguish source metadata from resolved secret values; showing env var names can be useful, but resolved secret payloads must remain masked.
