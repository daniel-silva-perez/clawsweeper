## Summary

Keep this open. Current main has substantial SecretRef runtime, CLI, docs, and Control UI corruption-guard work, but it still does not implement first-class env-backed secret UX in the Control UI: structured SecretRefs render as generic read-only redacted inputs, SecretInput unions are normalized to the string editor path, and there is no source badge, config/env/unset switcher, resolved/unresolved status, or dedicated Secrets/Environment helper in the UI.

## What This Changes

Request first-class `.env` / environment-backed secret config UX in CLI and Control UI, including source-aware secret fields, env/config/unset editing, resolved status, and safer migration guidance.

## Best Possible Solution

Keep this open for maintainer design and implementation on top of the existing SecretRef contracts: add a source-aware SecretInput renderer, safe metadata display, resolved/unresolved status, and explicit config/env/unset controls, then reuse the existing SecretRef target registry and `openclaw secrets` audit/configure model for any dedicated Secrets/Environment helper.
