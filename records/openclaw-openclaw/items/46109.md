## Summary

Keep this open. Current main has substantial SecretRef runtime, schema, redaction/restore, gateway, CLI, and docs support, but the requested first-class Control UI/config UX is still missing: structured SecretRefs render as read-only blank generic placeholders, SecretInput unions are normalized to the string editor path, and docs still describe the current form-mode limitation instead of source-aware config/env/unset badges or resolution status.

## What This Changes

The issue requests first-class `.env` and environment-backed secret configuration UX in CLI and Control UI, including source metadata, env reference preservation, a Secrets/Environment helper, migration guidance, and masked-but-visible secret state.

## Best Possible Solution

Define a safe SecretRef metadata contract first, then add a source-aware SecretInput renderer and supporting metadata flow that shows masked source labels, allowed env variable identity, resolved/unresolved state, and explicit config/env/unset editing, with docs and regression tests aligned to `openclaw secrets` and config-write preflight behavior.

## Reproduction Assessment

Not applicable as a feature request, but the current-main gap is high-confidence: static inspection and existing browser tests show structured SecretRefs still render as empty read-only generic placeholders rather than source-aware secret controls.

## Solution Assessment

Unclear until maintainer/security review decides the disclosure boundary. The maintainable path is a shared metadata contract plus UI/docs/tests, not a narrow cosmetic placeholder change that guesses what secret source details are safe to show.
