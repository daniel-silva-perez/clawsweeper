## Summary

Keep open. Current main has a substantial SecretRef runtime, CLI, redaction/restore, and preflight foundation, but the requested first-class Control UI/config UX is still not implemented: structured SecretRefs render as generic read-only placeholders, SecretInput schema analysis still collapses to string editing, and the docs describe those limitations rather than source-aware env/config/unset controls.

## What This Changes

The issue requests source-aware `.env` / environment-backed secret handling in the CLI and Control UI so secrets do not look unset when they are resolved from environment variables.

## Best Possible Solution

Add a source-aware SecretInput renderer and metadata flow on top of the existing SecretRef contracts, with masked display, safe source labels, env variable identity where appropriate, resolved/unresolved status, explicit config/env/unset editing, and aligned docs/tests for the current `openclaw secrets` and target-registry behavior.
