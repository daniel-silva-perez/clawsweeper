## Summary

Keep this PR open. Current main still lets schema-valid but syntactically invalid cron schedules reach `context.cron.add` / `context.cron.update`, where Croner parsing can throw through the gateway catch-all as `UNAVAILABLE`. The PR is a narrow implementation candidate for open bug #74066 and its diff does not introduce security-sensitive workflow, dependency, lockfile, publishing, permission, or secrets changes.

## Best Possible Solution

Review and land this PR or an equivalent gateway-boundary fix. The fix should validate cron schedules for `cron.add` and `cron.update` before mutating the cron service, defensively map Croner schedule parse failures to `INVALID_REQUEST`, preserve valid `cron`/`every`/`at` semantics, and keep focused coverage in `src/gateway/server-methods/cron.validation.test.ts`. After merge, #74066 can close through the PR.
