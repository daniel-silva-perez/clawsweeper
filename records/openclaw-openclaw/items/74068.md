## Summary

Keep open. Current main still maps any TypeError or RangeError thrown by cron.add/cron.update to INVALID_REQUEST, and the current tests only prove a plain Error passthrough on cron.add. This PR addresses a real remaining misclassification after #74193, but it should stay in normal maintainer review because the final guard needs an explicit decision on whether Croner message prefixes alone are sufficient or whether the TypeError/RangeError constructor check should remain part of the predicate.

## What This Changes

This PR narrows gateway cron.add/update parse-error handling from constructor-only TypeError/RangeError mapping to Croner-owned message-prefix matching and extends cron validation tests for non-Croner TypeError/RangeError passthrough.

## Best Possible Solution

Land a small gateway-side guard that classifies only Croner-owned cron pattern/date parse failures as INVALID_REQUEST while allowing non-Croner cron service failures to reach the existing gateway catch-all, with add/update regression tests covering both branches and the current croner error shape.
