## Summary

Keep open. Current main still maps any TypeError or RangeError from cron.add and cron.update to INVALID_REQUEST, so this PR's intended narrower Croner parse-error boundary is not implemented or superseded by the merged #74193 fix. The branch is narrow and relevant, but the final PR diff now checks Error plus CronPattern/CronDate prefix rather than the body's stated TypeError/RangeError plus prefix, so it needs normal maintainer review or revision rather than cleanup closure.

## What This Changes

This PR changes the gateway cron.add/update parse-error catch guards to use CronPattern/CronDate message-prefix matching and extends cron validation tests for non-Croner TypeError/RangeError passthrough.

## Best Possible Solution

Keep this PR open for normal maintainer review. The best path is a tiny gateway change that preserves #74193's INVALID_REQUEST handling for Croner parse failures while proving non-Croner TypeError/RangeError from cron.add and cron.update still escape to the gateway catch-all; maintainers should decide whether the final guard must require TypeError/RangeError in addition to the CronPattern/CronDate prefixes before landing.
