## Summary

Keep this PR open for normal maintainer review. The PR is a narrow, relevant fix for the disabled-job invalid cron edit path: current main still mutates cron service state before schedule parsing in `cron.update`, while the PR adds gateway-side cron schedule validation before `cron.add`/`cron.update` are called. I did not find a definite blocking bug in the patch, and the diff has no security-sensitive supply-chain or secret-handling changes.

## What This Changes

The PR imports cron schedule computation into the gateway cron handler, validates cron schedules before `cron.add` and `cron.update`, and adds two gateway validation tests proving invalid cron add/update requests do not call the cron service.

## Best Possible Solution

Land this gateway-boundary validation if maintainers accept the scope, then handle any broader service-level transactional hardening for pre-existing invalid persisted jobs as a separate, explicit follow-up if needed.
