## Summary

Keep open. Current main still lets reload/recompute apply `cron.retry.backoffMs` to recurring errored jobs, while the live recurring error path and public config contract keep that setting scoped to one-shot retries. This draft PR is a narrow, plausible fix for that inconsistency, and I found no blocking correctness or security issue in the two-file diff.

## What This Changes

The PR changes recurring-job recompute backoff to use the built-in recurring error backoff ladder instead of global one-shot retry config, and adds a focused regression test in `src/cron/service.jobs.test.ts`.

## Best Possible Solution

Land this focused fix, or an equivalent small patch, so recurring jobs keep their documented recurring backoff ladder across reload/recompute while `cron.retry` remains scoped to one-shot transient retries. Broader recurring transient-retry behavior should stay coordinated with #46341 and #46423.

## Reproduction Assessment

Yes. A deterministic unit-level reproduction is an enabled recurring `every` job with `lastStatus: "error"`, `lastRunAtMs`, `consecutiveErrors: 1`, and `cron.retry.backoffMs` set to one hour; current main recomputes the next run as last run plus one hour instead of the recurring 30-second backoff.

## Solution Assessment

Yes for the narrow bug. Removing the one-shot retry config from recurring recompute aligns reload behavior with `applyJobResult`, `isErrorBackoffPending`, and the public config/docs contract without changing one-shot retry semantics.
