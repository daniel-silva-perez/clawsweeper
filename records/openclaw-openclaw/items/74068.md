## Summary

Keep open. The PR targets a real current-main mismatch between reload/recompute backoff and the live recurring-job error path, but it is still draft and needs the required changelog entry before it is merge-ready.

## What This Changes

The PR changes cron recompute so recurring errored jobs ignore one-shot `cron.retry.backoffMs` after reload and adds a regression test for an errored recurring `every` job with a custom one-shot retry backoff.

## Best Possible Solution

Land the scoped service/test fix with a single-line changelog entry once the draft is ready and validation is green; keep the broader recurring transient-retry semantics tracked separately in #46341/#46423.

## Reproduction Assessment

Yes. On current main, construct an enabled recurring job with `lastStatus: "error"`, finite `lastRunAtMs`, `consecutiveErrors: 1`, and `cronConfig.retry.backoffMs: [3600000]`, then call `recomputeNextRuns`; the recompute path uses the one-shot hour-long delay instead of the fixed 30s recurring backoff.

## Solution Assessment

Yes for the code path, with one process gap. Removing the custom retry schedule from recurring recompute is the narrowest maintainable fix because it aligns reload behavior with the live recurring timer path and documented one-shot-only `cron.retry` contract.
