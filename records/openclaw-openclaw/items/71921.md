## Summary

Keep open. Current main still preserves stale explicit Bedrock catalog rows without `input`, and the image capability gate still treats missing `input` as no image support. The discussion also has open PR #72404 as the implementation candidate, so this issue should remain open until that PR or a replacement lands.

## What This Changes

The issue requests that Bedrock-hosted Claude Opus/Sonnet 4.x model metadata be treated as image-capable so image attachments are not blocked as unsupported.

## Best Possible Solution

Review and land PR #72404 or a provider-owned replacement that refreshes or inherits missing Bedrock `input` capability metadata for stale explicit/static rows after discovery succeeds, while preserving intentional explicit text-only opt-outs and covering foundation, regional, and global Claude ids.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a non-empty `models.providers.amazon-bedrock.models` list containing a Claude 4.x row with no `input`: current merge code preserves that row and `modelSupportsImages()` returns false for missing `input`.

## Solution Assessment

No for the original registry-only framing. Updating a few built-in registry entries would not repair already-persisted static catalogs; the maintainable fix should address stale explicit catalog metadata or replacement when discovery succeeds.
