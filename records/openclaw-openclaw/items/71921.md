## Summary

Keep open. Current main still preserves the stale Bedrock catalog path: discovery can produce image-capable Claude entries, but non-empty explicit/static Bedrock model lists and explicit-only rows without `input` remain unchanged, so the image gate still treats those rows as text-only. The issue is also paired with open PR #72404 using closing syntax, so it should remain open until that PR or a replacement lands.

## What This Changes

The issue requests that Bedrock Claude 4.x catalog entries expose image input capability so OpenClaw does not block image attachments.

## Best Possible Solution

Land PR #72404 or a provider-owned replacement that refreshes or inherits missing Bedrock `input` capability metadata for stale explicit/static rows after discovery succeeds, while preserving intentional explicit text-only opt-outs and covering foundation, regional, and global Claude ids.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a non-empty `models.providers.amazon-bedrock.models` list containing a Claude 4.x row with no `input`: current merge code preserves that row and the image gate returns false for missing `input`.

## Solution Assessment

No for the original registry-only framing. Marking specific catalog entries is only a partial fix; the discussion and code show the durable failure is stale explicit catalog metadata, and #72404 is a plausible narrow implementation candidate for that path.
