## Summary

Keep open. Current main still has the stale Bedrock catalog path described in the report: successful discovery can produce image-capable Bedrock Claude entries, but non-empty explicit/static Bedrock model lists and explicit-only rows without `input` are still preserved, so the image gate continues to treat those models as text-only. An open linked PR with closing syntax is already the implementation candidate, so this issue should not be closed before that PR or a replacement lands.

## What This Changes

Mark Bedrock-hosted Claude Opus/Sonnet 4.x model entries as image-capable, including stale static or explicit catalogs that currently cause image attachments to be rejected.

## Best Possible Solution

Land PR #72404 or a narrower replacement that refreshes missing/stale `input` capability metadata for Bedrock Claude explicit/static rows after successful discovery, while preserving intentional explicit opt-outs and adding regression coverage for foundation, regional, and global inference profile ids.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a non-empty `models.providers.amazon-bedrock.models` list containing a Claude 4.x row with no `input`: current merge code preserves that explicit row, and the image gate returns false when `input` is missing.

## Solution Assessment

Unclear. The open PR is a plausible narrow fix for the immediate stale-row symptom, but the most maintainable solution may be provider-owned refresh of Bedrock capability metadata plus coverage for the discovery/runtime-dependency race that seeds stale catalogs.
