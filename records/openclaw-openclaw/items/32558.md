## Summary

Keep this PR open. Current main still does not implement the Microsoft Teams Graph upload-session fallback for large OneDrive or SharePoint uploads: both helpers still issue a single `/content` PUT, search found no `createUploadSession`/`nextExpectedRanges`/upload-session code in Teams, and the current tests only cover direct uploads. The PR remains a relevant implementation candidate, but it needs maintainer review and a rebase because current main has newer User-Agent/preconnect expectations and the PR review identified a real stall-count off-by-one.

## Best Possible Solution

Keep this PR open for Microsoft Teams maintainer review. Rebase it onto current main, preserve current User-Agent and test preconnect behavior, add the upload-session fallback to the current OneDrive and SharePoint helpers, fix the stall guard to match the intended limit, and land focused large-file regression tests for both storage paths.

## Risks / Open Questions

- The PR head predates current main’s `buildUserAgent()` and preconnect-aware test expectations, so a direct merge would risk regressing current Microsoft Graph request header coverage unless rebased carefully.
- The upload-session URL returned by Microsoft Graph is preauthenticated; the final implementation should continue omitting `Authorization` on chunk PUTs and must avoid logging or exposing that URL.
- The stall guard in the PR review allows one more stalled response than `UPLOAD_SESSION_MAX_STALLS` implies; that should be corrected before landing.
