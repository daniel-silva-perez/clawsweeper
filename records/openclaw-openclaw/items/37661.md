## Summary

Keep open. Current main still lacks a generic assistant text repetition guard: tool-loop detection is tool-call scoped, assistant stream handling still emits any growing cleaned text, and block/outbound delivery dedupes exact payloads rather than detecting repeated appended phrases or enforcing a per-turn duplicate/message cap. The April 11 follow-up reports this as pure assistant text repetition on 2026.4.10, including normal main-session conversation, so the closed tool-loop and heartbeat-specific related issues do not supersede it.

## Best Possible Solution

Keep this issue open and implement a conservative assistant-output repetition guard in the streaming/final reply path, plus a shared per-turn outbound duplicate/message cap before channel delivery. Keep #34574 scoped to tool-call loops and treat closed #39536 as heartbeat-specific context rather than the generic fix.

## Risks / Open Questions

- A text repetition guard can false-positive on legitimate repeated logs, code blocks, tables, quoted text, or short acknowledgements unless thresholds and exclusions are conservative.
- A per-turn delivery cap must count logical assistant replies carefully so normal long-message chunking, media delivery, and block streaming are not truncated incorrectly.
- The live trigger is intermittent and provider/model-specific, so validation should rely on synthetic repeated-delta, repeated-final-text, and repeated-outbound fixtures rather than a live endpoint.
