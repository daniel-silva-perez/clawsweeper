## Summary

Keep this PR open. Current main still does not implement native iOS/macOS Adaptive Card rendering, but the proposed branch has concrete correctness and security blockers: valid numeric Adaptive Card widths fail decoding, hidden thinking can be parsed before the normal visibility filter, decoded column widths are ignored at render time, invalid image URLs drop content, and plugin/model-supplied URLs/images are handled without native-client policy gates.

## What This Changes

This PR adds Swift Adaptive Card models, marker parsing, and SwiftUI rendering under shared OpenClawChatUI, wires assistant messages to render `<!--adaptive-card-->` blocks inline, and adds an Adaptive Cards entry to the community plugin docs.

## Best Possible Solution

If maintainers want native iOS/macOS Adaptive Card rendering, rebase this branch, keep third-party plugin promotion in ClawHub unless explicitly approved, normalize schema-backed width unions, parse only visible assistant content, implement layout semantics, add URL/image policy gates, and cover the parser/rendering edge cases with focused OpenClawKit tests plus native chat smoke validation.

## Reproduction Assessment

Yes. A valid Adaptive Card with `Column.width: 2` reproduces the P1 parser failure because the branch models width as `String?`; a card marker inside `<think>` reproduces the visibility leak because the branch parses raw text before AssistantTextParser hides thinking.

## Solution Assessment

No. The direction may be viable after maintainer approval, but this diff is not the narrowest maintainable implementation yet because it mixes native rendering, plugin docs promotion, incomplete Adaptive Card layout semantics, and ungated URL/image handling.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Decode numeric Adaptive Card widths:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardModels.swift:47`
  - body: Adaptive Cards allow weighted numeric widths such as `"width": 2`, but this model declares `Column.width` as `String?` and table widths the same way. JSONDecoder treats those valid payloads as type mismatches, so `parseAdaptiveCardMarkers` returns nil and the card does not render.
  - confidence: 0.94
- **[P2] Filter hidden thinking before parsing cards:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/ChatMessageViews.swift:623`
  - body: This parses adaptive-card markers from raw `self.text` before the existing AssistantTextParser visibility policy runs. When thinking is hidden, a card embedded in a `<think>` block can still be decoded and rendered even though the normal assistant text path suppresses that content.
  - confidence: 0.9
- **[P2] Honor column widths during layout:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardView.swift:116`
  - body: `renderColumnSet` never reads the decoded `width` value and gives every column `.frame(maxWidth: .infinity)`. Cards using `auto`, `stretch`, or weighted widths render as equal columns, breaking common icon/content and weighted layouts.
  - confidence: 0.91
- **[P2] Render image fallback for invalid URLs:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardView.swift:138`
  - body: `renderImage` only renders inside the `URL(string:)` success branch. If the URL string cannot be parsed, the image element and its `altText` disappear instead of showing the same fallback label used for load failures.
  - confidence: 0.82
- **[P3] Encode unsupported action data explicitly:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardModels.swift:282`
  - body: `AnyCodableAction.encode(to:)` has no final fallback after the handled types. If an unsupported value reaches the wrapper through `init(_:)`, encoding completes without writing a value, which can silently corrupt action payloads.
  - confidence: 0.72

## Security Review

Status: needs_attention

Summary: Security review needs attention: the PR introduces native handling of plugin/model-supplied remote images and URL actions without explicit policy gates.

Concerns:

- **[medium] Gate Action.OpenUrl schemes before opening:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardView.swift:355`
  - body: `Action.OpenUrl` values come from assistant/plugin-emitted card JSON and are passed directly to SwiftUI `openURL`. Without a scheme allowlist or confirmation policy, cards can trigger arbitrary URL schemes or deep links from the native client.
  - confidence: 0.88
- **[medium] Constrain remote image fetching:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardView.swift:139`
  - body: `AsyncImage` fetches card image URLs directly from the native app. A card producer can cause the client to request arbitrary remote URLs, which has privacy and tracking implications and should be gated, proxied, or clearly documented before merge.
  - confidence: 0.86
