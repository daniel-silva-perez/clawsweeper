## Summary

Keep this PR open. Current main still does not implement native iOS/macOS Adaptive Card parsing or inline SwiftUI rendering, the existing plugin/channel presentation surfaces do not supersede a native Swift chat-bubble renderer, and the PR still needs maintainer product and security review before it can land.

## What This Changes

The PR adds Swift Adaptive Card models, marker parsing, and a SwiftUI renderer under shared OpenClawChatUI, wires assistant chat messages to render `<!--adaptive-card-->` blocks inline, and adds an Adaptive Cards entry to the community plugin docs.

## Best Possible Solution

Keep the PR open for maintainer product and security review. If native iOS/macOS Adaptive Card rendering is accepted, rebase it on current main, remove or split the community-docs listing, fix the Swift parser/layout/fallback/security issues, add focused Swift parser/view tests or app smoke coverage, and coordinate the platform decision with #42304.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Decode numeric column widths:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardModels.swift:47`
  - body: `Column.width` is modeled as `String?`, but the Adaptive Cards schema allows numbers for weighted column widths. A valid card with `"width": 2` will throw during `ColumnSet` decoding, causing `parseAdaptiveCardMarkers` to return nil and the chat to fall back to raw marker text instead of rendering the card.
  - confidence: 0.94
- **[P2] Filter hidden thinking before parsing cards:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/ChatMessageViews.swift:623`
  - body: This parses adaptive-card markers from raw `self.text` before applying `AssistantTextParser`. When thinking is hidden, a card inside a `<think>` block can still be decoded and rendered by `AdaptiveCardView`, leaking content that the existing assistant text path would suppress.
  - confidence: 0.9
- **[P2] Honor Column.width during layout:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardView.swift:116`
  - body: `renderColumnSet` gives every column `.frame(maxWidth: .infinity)` and never reads the decoded width value. Cards that use `auto`, `stretch`, or weighted widths render as equal columns, which breaks common icon/content and weighted layouts.
  - confidence: 0.91
- **[P2] Show image fallback for unparsable URLs:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardView.swift:138-139`
  - body: `renderImage` only enters the `AsyncImage` branch when `URL(string:)` succeeds. If the URL string cannot be parsed, the image element and its `altText` disappear completely instead of using the fallback label shown for load failures.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff introduces native remote image loading and URL opening from card JSON without a clear allowlist, consent, or privacy policy.

Concerns:

- **[medium] Gate Action.OpenUrl schemes before opening:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardView.swift:338`
  - body: `Action.OpenUrl` values come from assistant/plugin-emitted card JSON and are passed directly to `openURL`. Without an allowlist or explicit confirmation policy, cards can trigger arbitrary URL schemes or deep links from the native client.
  - confidence: 0.88
- **[medium] Constrain remote image fetching:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardView.swift:139`
  - body: `AsyncImage` fetches card image URLs directly from the native app. A card producer can cause the client to request arbitrary remote URLs, which has privacy/tracking implications and should be gated or documented before merge.
  - confidence: 0.86
