## Summary

Keep open. Current main still lacks native iOS/macOS Adaptive Card parsing or rendering, and the PR contains useful ecosystem context around the external `openclaw-adaptive-cards` plugin, but the branch is not merge-ready because the proposed renderer has concrete schema/layout/visibility bugs plus URL and remote-image policy concerns.

## What This Changes

This PR adds Swift Adaptive Card models, marker parsing, and SwiftUI rendering under shared OpenClawChatUI, wires assistant messages to render `<!--adaptive-card-->` blocks inline, and adds an Adaptive Cards entry to the community plugin docs.

## Best Possible Solution

If maintainers want native iOS/macOS Adaptive Card rendering, the branch should be rebased and narrowed around a schema-backed parser, visible-content-first extraction, faithful column/table layout, URL/image policy gates, and focused OpenClawKit plus native chat smoke coverage. Plugin promotion should stay aligned with ClawHub unless maintainers explicitly approve a docs listing.

## Reproduction Assessment

Yes. A valid Adaptive Card using `"width": 2` reproduces the parser failure because the branch decodes width as `String?`, and a card marker inside hidden `<think>` content reproduces the visibility leak because parsing runs on raw assistant text before `AssistantTextParser` filtering.

## Solution Assessment

No. The native rendering direction may be viable with maintainer approval, but this diff is not the best merge path yet because it mixes app rendering, plugin promotion, incomplete Adaptive Card layout semantics, missing tests, and ungated URL/image handling.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Decode numeric Adaptive Card widths:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardModels.swift:47-67`
  - body: Adaptive Cards allow weighted numeric widths such as `"width": 2`, but this model declares `Column.width` and table widths as `String?`. `JSONDecoder` treats those valid payloads as type mismatches, so otherwise valid cards can fail to parse instead of rendering inline.
  - confidence: 0.94
- **[P2] Filter hidden thinking before parsing cards:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/ChatMessageViews.swift:623`
  - body: This parses adaptive-card markers from raw `self.text` before the existing assistant text visibility policy runs. When thinking is hidden, a card embedded in a `<think>` block can still be decoded and rendered even though normal assistant text suppresses that content.
  - confidence: 0.9
- **[P2] Honor column widths during layout:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardView.swift:116`
  - body: `renderColumnSet` never reads the decoded `width` value and gives every column `.frame(maxWidth: .infinity)`. Cards using `auto`, `stretch`, or weighted widths render as equal columns, breaking common icon/content and weighted layouts.
  - confidence: 0.91
- **[P2] Render image fallback for invalid URLs:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardView.swift:138-139`
  - body: `renderImage` only renders inside the `URL(string:)` success branch. If the URL string cannot be parsed, the image element and its `altText` disappear instead of showing the same fallback label used for load failures.
  - confidence: 0.82
- **[P3] Encode unsupported action data explicitly:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardModels.swift:282`
  - body: `AnyCodableAction.encode(to:)` has no final fallback after the handled types. If an unsupported value reaches the wrapper through `init(_:)`, encoding completes without writing a value, which can silently corrupt action payloads.
  - confidence: 0.72

## Security Review

Status: needs_attention

Summary: The diff introduces native handling of plugin/model-supplied remote images and URL actions without explicit client policy gates.

Concerns:

- **[medium] Gate Action.OpenUrl schemes before opening:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardView.swift:355`
  - body: `Action.OpenUrl` values come from assistant/plugin-emitted card JSON and are passed directly to SwiftUI `openURL`. Without a scheme allowlist or confirmation policy, cards can trigger arbitrary URL schemes or deep links from the native client when tapped.
  - confidence: 0.88
- **[medium] Constrain remote image fetching:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardView.swift:139`
  - body: `AsyncImage` fetches card image URLs directly from the native app as soon as the card renders. A card producer can cause client-side requests to arbitrary remote URLs, which has privacy and tracking implications and should be gated, proxied, or documented before merge.
  - confidence: 0.86
