## Summary

Keep this PR open for maintainer review. Current main still lacks the proposed iOS/macOS Adaptive Card parser and SwiftUI renderer, and the external Adaptive Cards plugin gives a coherent producer for the proposed markers, but the branch still has correctness, product-scope, and native URL/image policy issues that should be resolved before merge.

## What This Changes

This PR adds Swift Adaptive Card models, marker parsing, and inline SwiftUI rendering to shared iOS/macOS chat UI, and adds an Adaptive Cards entry to the community plugin docs.

## Best Possible Solution

If maintainers want this native app feature, the mergeable path should be a narrower, rebased branch that decodes Adaptive Cards schema-compatible widths, filters visible content before marker parsing, preserves column/table layout semantics, gates URL schemes and remote image fetching, and adds focused OpenClawKit tests plus iOS/macOS smoke proof. Plugin promotion should stay aligned with ClawHub unless maintainers explicitly approve a docs listing.

## Reproduction Assessment

Yes. A valid Adaptive Card using a numeric weighted column width such as `"width": 2` exercises the branch's `String?` decoding failure, and a card marker inside `<think>...</think>` exercises the visibility leak caused by parsing raw assistant text first.

## Solution Assessment

No. The native rendering direction may be useful, especially with the linked external plugin producing markers, but this branch is not the best merge path yet because it bypasses existing visibility/security patterns, only partially implements Adaptive Card semantics, and mixes app rendering with plugin promotion.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Decode numeric Adaptive Card widths:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardModels.swift:47-67`
  - body: Adaptive Cards allow weighted numeric widths such as `"width": 2`, but the branch models column and table widths as `String?`. Swift Codable treats those valid payloads as type mismatches, so otherwise valid cards can fail to parse instead of rendering inline.
  - confidence: 0.94
- **[P2] Filter hidden thinking before parsing cards:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/ChatMessageViews.swift:623`
  - body: The branch parses adaptive-card markers from raw assistant text before the existing AssistantTextParser visibility policy runs. When thinking is hidden, a card embedded in `<think>` content can still be decoded and rendered even though normal assistant text suppresses that content.
  - confidence: 0.9
- **[P2] Honor column widths during layout:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardView.swift:116`
  - body: `renderColumnSet` gives every column an infinite-width frame and does not use the decoded width value. Cards using `auto`, `stretch`, or weighted widths render as equal columns, breaking common icon/content and weighted layouts.
  - confidence: 0.91
- **[P2] Render image fallback for invalid URLs:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardView.swift:138-156`
  - body: `renderImage` only renders inside the `URL(string:)` success branch. If the URL string cannot be parsed, the image element and its `altText` disappear instead of showing the same fallback label used for load failures.
  - confidence: 0.82
- **[P3] Encode unsupported action data explicitly:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardModels.swift:266-283`
  - body: `AnyCodableAction.encode(to:)` has no final fallback after the handled value types. If an unsupported value reaches the wrapper through `init(_:)`, encoding can complete without writing a value, which silently corrupts action payloads.
  - confidence: 0.72

## Security Review

Status: needs_attention

Summary: The diff introduces native handling of plugin/model-supplied remote images and URL actions without explicit client policy gates.

Concerns:

- **[medium] Gate Action.OpenUrl schemes before opening:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardView.swift:355`
  - body: `Action.OpenUrl` values come from assistant/plugin-emitted card JSON and are passed to SwiftUI `openURL`. Without an allowlist or confirmation policy, cards can trigger arbitrary URL schemes or deep links from the native client when tapped.
  - confidence: 0.88
- **[medium] Constrain remote image fetching:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardView.swift:139`
  - body: `AsyncImage` fetches card image URLs directly from the native app as soon as the card renders. A card producer can cause client-side requests to arbitrary remote URLs, which has privacy and tracking implications and should be gated, proxied, or documented before merge.
  - confidence: 0.86
