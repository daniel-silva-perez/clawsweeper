## Summary

Keep open. Current main still lacks the PR's native iOS/macOS chat Adaptive Card renderer, so the branch is not obsolete or implemented on main; however, the proposed patch has concrete correctness and security issues and needs maintainer product/security review before it can merge.

## What This Changes

The PR adds Swift Adaptive Card models, marker parsing, and a SwiftUI renderer under shared OpenClawChatUI, wires assistant chat text to render `<!--adaptive-card-->` blocks inline, and adds an Adaptive Cards entry to the community plugin docs.

## Best Possible Solution

If native iOS/macOS Adaptive Card rendering is accepted, rebase the branch on current main, keep plugin discovery/promotion in ClawHub unless maintainers explicitly want a docs entry, add focused Swift parser/rendering tests or app smoke coverage, and fix the numeric width, hidden-thinking, layout, image fallback, URL-scheme, and remote image policy issues before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Decode numeric column widths:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardModels.swift:47`
  - body: `Column.width` is modeled as `String?`, but Adaptive Cards allow weighted numeric widths such as `"width": 2`. Swift `Codable` treats that as a type mismatch, so an otherwise valid `ColumnSet` fails decoding and the chat falls back instead of rendering the card.
  - confidence: 0.94
- **[P2] Filter hidden thinking before parsing cards:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/ChatMessageViews.swift:623`
  - body: This parses adaptive-card markers from raw `self.text` before `AssistantTextParser` applies the hidden-thinking policy. When thinking is hidden, a card inside a `<think>` block can still be decoded and rendered, leaking content that the normal assistant text path suppresses.
  - confidence: 0.9
- **[P2] Honor column widths during layout:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardView.swift:116`
  - body: `renderColumnSet` never reads the decoded `width` value and gives every column `.frame(maxWidth: .infinity)`. Cards using `auto`, `stretch`, or weighted widths render as equal columns, breaking common icon/content and weighted layouts.
  - confidence: 0.91
- **[P2] Show fallback for invalid image URLs:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardView.swift:138-139`
  - body: `renderImage` only renders inside the `URL(string:)` success branch. If the URL string cannot be parsed, the image element and its `altText` disappear instead of showing the fallback label used for load failures.
  - confidence: 0.82
- **[P3] Encode unsupported action data explicitly:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardModels.swift:283`
  - body: `AnyCodableAction.encode(to:)` has no final fallback after the handled types. If an unsupported value reaches the wrapper through `init(_:)`, encoding completes without writing a value, which can silently corrupt action payloads.
  - confidence: 0.72

## Security Review

Status: needs_attention

Summary: Security review needs attention: the PR introduces native handling of plugin/model-supplied remote images and URL actions without an explicit allowlist, consent step, proxy, or privacy policy.

Concerns:

- **[medium] Gate Action.OpenUrl schemes before opening:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardView.swift:355`
  - body: `Action.OpenUrl` values come from assistant/plugin-emitted card JSON and are passed directly to SwiftUI `openURL`. Without a scheme allowlist or confirmation policy, cards can trigger arbitrary URL schemes or deep links from the native client.
  - confidence: 0.88
- **[medium] Constrain remote image fetching:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/AdaptiveCard/AdaptiveCardView.swift:139`
  - body: `AsyncImage` fetches card image URLs directly from the native app. A card producer can cause the client to request arbitrary remote URLs, which has privacy and tracking implications and should be gated, proxied, or clearly documented before merge.
  - confidence: 0.86
