## Summary

Keep this PR open. Current main still does not implement the PR’s intended graceful cron drain on hot reload: the reload path still calls synchronous `stop()`, the cron contract/class expose no `stopGraceful()`, and timer/manual run execution still has out-of-lock work that can persist later. The PR remains a plausible implementation candidate, but it needs maintainer review around bounded waits for long-lived runs.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to refresh it against current main, preserve regression coverage for timer ticks and `CommandLane.Cron` manual runs, add a bounded drain or explicit fallback for stuck/no-timeout runs, update the cron service contract and gateway reload tests, and then use the graceful stop path in `src/gateway/server-reload-handlers.ts` if maintainers want that stronger guarantee. If maintainers decide the shipped force-reload/jobs-state approach is the intended fix for #30098, document that decision and close this PR then.

## Risks / Open Questions

- Merging the PR as-is may make `cron.*` hot reload wait indefinitely on long-lived or stuck runs; current timeout policy returns `undefined` for `timeoutSeconds <= 0`.
