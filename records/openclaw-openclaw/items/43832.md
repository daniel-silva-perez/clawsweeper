## Summary

Keep open, but do not merge as-is. The PR is aimed at a real cron hot-reload race and the later updates address the previously flagged manual-run and timer-tick gaps, but the current patch can block cron config hot reload indefinitely and the branch is currently dirty against main with failing checks.

## What This Changes

The PR adds `CronService.stopGraceful()`, tracks in-flight cron runs, awaits graceful cron shutdown during gateway hot reload, and adds regression tests for locked operations, manual runs, timer ticks, and re-arm guards.

## Best Possible Solution

Land a rebased cron-owned fix that prevents old cron instances from writing after hot replacement while using a bounded drain/fallback consistent with gateway reload deferral, preserving existing no-timeout job semantics and updating the gateway reload test harness.

## Reproduction Assessment

Yes. A high-confidence reproduction path is a deferred timer tick or `enqueueRun` manual run that has left its initial locked persist, followed by a `cron.*` hot reload; current main stops only the timer while the old run can still complete and persist later.

## Solution Assessment

No as-is. Graceful draining is the right direction, but the current implementation needs a bounded wait or fallback so one long-lived/no-timeout cron run cannot wedge config reload indefinitely.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Bound the graceful-stop drain before hot reload:** `src/cron/service/ops.ts:174`
  - body: `stopGraceful()` awaits every pending run before the hot-reload path creates the replacement cron service. Because cron explicitly supports `timeoutSeconds <= 0` as no timeout, a stuck or long-lived agent turn can leave this promise unsettled forever and prevent the `cron.*` config change from applying. Add a deadline/fallback, likely tied to the gateway reload deferral timeout, before waiting on the drain.
  - confidence: 0.91
- **[P2] Update the cron reload test double:** `src/gateway/server-reload-handlers.ts:134`
  - body: The hot-reload path now calls `stopGraceful()`, but the existing `server.reload.test.ts` CronService mock only exposes `stop()` and still asserts that `stop()` was called. The cron-restart reload test will throw when this branch runs unless the mock and assertion are updated to the new graceful-stop method.
  - confidence: 0.88
