## Summary

Keep this PR open. Current main still does not implement the PR’s intended graceful cron drain on hot reload: cron reload still calls the synchronous stop path, the CronService contract has no stopGraceful API, and timer/manual runs still perform out-of-lock work that can persist later. Current main has adjacent forceReload/state-sidecar mitigations, but that is not the same lifecycle guarantee this PR proposes.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to refresh it against current main, preserve tests for timer ticks and CommandLane.Cron manual runs, add a bounded drain or explicit fallback for stuck/no-timeout runs, update the CronService contract and gateway reload tests, and then use the graceful stop path in src/gateway/server-reload-handlers.ts if maintainers want that stronger guarantee. If maintainers decide current forceReload/jobs-state behavior is the intended shipped fix for the linked cron overwrite report, document that decision and close this PR then.
