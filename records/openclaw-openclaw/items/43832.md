## Summary

Keep this PR open. Current main still does not implement the PR’s intended graceful cron drain on hot reload: the reload path still calls the synchronous cron stop, the public cron contract has no stopGraceful API, and timer/manual run paths still have out-of-lock work that can persist later. The current jobs-state/forceReload work is relevant mitigation for stale state, but it is not the same quiesce-before-replacement guarantee proposed here.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to refresh it against current main, preserve regression coverage for timer ticks and CommandLane.Cron manual runs, add a bounded drain or explicit fallback for stuck/no-timeout runs, update the CronService contract and gateway reload tests, and then use the graceful stop path in src/gateway/server-reload-handlers.ts if maintainers want that stronger guarantee. If maintainers decide the shipped jobs-state/forceReload behavior is the intended fix for #30098, document that decision and close this PR then.

- Current main's jobs-state split and forceReload behavior may already solve the original stale-overwrite symptom for many paths, so maintainers should decide whether the stronger quiesce-before-replacement guarantee is worth the reload-latency tradeoff.
