## Summary

Keep open. This is a fresh WSL2 2026.4.26 loopback WebSocket handshake regression with concrete logs, and current main does not provide enough source/test evidence to close it as implemented or superseded. The code has adjacent mitigations, but the server still binds before post-attach sidecars finish, readiness still distinguishes HTTP liveness from sidecar readiness, the literal 127.0.0.1 proxy path is not clearly covered by the open localhost/WSL proxy PR, and timeout tuning is still env-only rather than config-backed.

## Best Possible Solution

Keep this issue open as the canonical WSL 2026.4.26 loopback handshake regression until maintainers reproduce or disprove it on current main. The next useful path is to collect WSL startup trace output, /readyz eventLoop diagnostics, service proxy environment, and a literal-127.0.0.1 WebSocket probe, then either land a focused WSL/proxy fix, make handshake timeout configuration durable if that is the real blocker, or move any post-bind sidecar work that can starve the pre-auth challenge path off the startup hot path.
