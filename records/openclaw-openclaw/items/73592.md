## Summary

Keep open. This is a fresh 2026.4.26 WSL local-gateway regression report, not a stale or already-shipped cleanup candidate. Current main has related mitigations, but the exact failure remains plausible: literal 127.0.0.1 already takes the intended direct control-plane path, the configurable timeout PR is still open, and the server still binds before post-attach sidecars finish, so a sidecar/event-loop stall could still produce the reported WebSocket handshake timeout while HTTP bind/liveness looks healthy.

## Best Possible Solution

Keep this issue open as the canonical WSL 2026.4.26 loopback handshake regression until maintainers reproduce or disprove it on current main. The best path is to collect startup trace, /readyz event-loop diagnostics, service environment proxy variables, and a literal-127.0.0.1 WebSocket probe from WSL; then either land a focused fix for the WSL/proxy path, make timeout configuration durable if that is the real blocker, or move any post-bind sidecar work that can starve the pre-auth challenge path off the startup hot path.
