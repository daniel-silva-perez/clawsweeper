## Summary

Keep open. Current main still has no Gateway system-events recovery-notice policy, and the relevant recovery paths remain prompt-visible or channel-visible without a mode gate; the related WhatsApp banner filter and plugin-local recovery narrowing do not cover this general customer-facing recovery-notice problem.

## What This Changes

The issue requests a Gateway configuration policy, `gateway.systemEvents.recoveryNotice`, to control whether automatic config-recovery notices are injected into agent prompts, delivered out-of-band, logged only, or suppressed.

## Best Possible Solution

Add a Gateway-owned, documented config policy with a current-behavior default that gates config-recovery notices across both prompt-visible system events and out-of-band restart-sentinel channel delivery, while leaving actual config recovery and restart continuation delivery unchanged.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is available on current main: a whole-file recovery path calls `enqueueConfigRecoveryNotice`, and a `config-auto-recovery` restart sentinel flows through `scheduleRestartSentinelWake`, which unconditionally enqueues a system event and sends the outbound notice when routed.

## Solution Assessment

Yes. The proposed `gateway.systemEvents.recoveryNotice` knob is the narrow maintainable solution if it covers both recovery-notice implementations; adding more banner-specific prompt filters like #49579 would not solve the general operator-controlled policy need.

Implement a focused fix for https://github.com/openclaw/openclaw/issues/72043. Add a Gateway config policy `gateway.systemEvents.recoveryNotice` with modes `"user"` (default/current behavior), `"log"`, and `"off"`. Keep actual config recovery and restart continuation delivery unchanged. In `"user"`, preserve current prompt system-event and out-of-band restart-sentinel notice behavior. In `"log"`, suppress prompt-visible enqueueing and outbound channel delivery for recovery notices, but emit a single concise `gateway/restart-sentinel` or recovery-policy info log where appropriate. In `"off"`, suppress those notice surfaces without adding the log-only notice. Update the strict config schema, exported Gateway config types, schema help/generated config docs as required by repo tooling, focused tests for `config-recovery-notice` and `server-restart-sentinel`, Gateway configuration docs, and the active changelog with `Thanks @MaxK000`. Do not add channel-specific filters, do not change restart continuations, do not weaken last-known-good recovery, and do not move extension-owned behavior into core beyond the generic Gateway policy.
