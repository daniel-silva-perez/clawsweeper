## Summary

Keep open. Current main still has no operator-controlled Gateway recovery-notice policy, and the recovery notices remain visible through both the prompt system-event path and the restart-sentinel outbound channel path; the recent validation-detail work improved wording but did not add the requested suppression knob.

## What This Changes

Add a `gateway.systemEvents.recoveryNotice` config option to control whether last-known-good recovery notices are injected into prompts, delivered out of band, logged only, or suppressed.

## Best Possible Solution

Add a Gateway-owned, documented config policy with a current-behavior default that gates both prompt-visible recovery system events and out-of-band restart-sentinel recovery notices, while leaving last-known-good recovery and restart continuation delivery unchanged.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists on current main: trigger last-known-good config recovery and startup/reload queues a recovery system event; for routed `config-auto-recovery` restart sentinels, startup also sends the notice outbound.

## Solution Assessment

Yes. A `gateway.systemEvents.recoveryNotice` knob is the narrow maintainable solution because it centralizes operator policy for this recovery event class; channel-specific prompt filters like #49579 would not cover the out-of-band notice path.
