## Summary

Keep open. Current main still lacks an operator-controlled Gateway recovery-notice policy, and the recovery notice remains visible through both prompt system-event injection and restart-sentinel outbound delivery; recent recovery work improved validation detail and rollback scope but did not add the requested suppression knob.

## What This Changes

Request to add a `gateway.systemEvents.recoveryNotice` config mode that can keep the current recovery notice behavior, log it only, or suppress it for customer-facing bots.

## Best Possible Solution

Add a Gateway-owned, documented recovery-notice policy with a current-behavior default that gates prompt-visible config recovery system events and `config-auto-recovery` restart-sentinel outbound notices, while leaving the recovery operation and restart continuations unchanged.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists on current main: trigger last-known-good config recovery during startup/reload or process a `config-auto-recovery` restart sentinel, and current code queues the recovery text as a system event; routed restart sentinels also deliver the notice outbound.

## Solution Assessment

Yes. A default-preserving `gateway.systemEvents.recoveryNotice` policy is the narrow maintainable fix because it centralizes operator intent for this recovery event class; channel-specific prompt filters such as #49579 would not cover the outbound restart-sentinel notice path.
