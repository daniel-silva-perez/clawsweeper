## Summary

Keep open. Current main still has no operator-controlled Gateway recovery-notice policy, and both relevant delivery paths remain prompt-visible or channel-visible: config recovery queues a main-session system event, while restart-sentinel recovery enqueues a wake and sends an out-of-band notice when routing is available. The related #71289 recovery narrowing, #49579 WhatsApp banner filtering, and #39307 context-reset visibility request do not supersede this general config-policy request.

## What This Changes

Add a `gateway.systemEvents.recoveryNotice` config option to control whether last-known-good recovery notices are injected into the next prompt, delivered out-of-band, logged only, or suppressed.

## Best Possible Solution

Add a Gateway-owned, documented config policy with a current-behavior default that gates both prompt-visible recovery system events and out-of-band restart-sentinel recovery delivery, while leaving actual last-known-good recovery and restart continuation delivery unchanged.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists on current main: trigger last-known-good config recovery and the startup/reload path queues a recovery system event; for `config-auto-recovery` restart sentinels, the startup task also enqueues the notice and delivers it outbound when a route resolves.

## Solution Assessment

Yes. A `gateway.systemEvents.recoveryNotice` knob is the narrow maintainable solution because it centralizes operator policy for this event class; extending banner-specific prompt filters like #49579 would not cover the out-of-band recovery notice or future recovery wording.
