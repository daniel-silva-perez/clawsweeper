## Summary

Keep open. Current main still lacks a `gateway.systemEvents.recoveryNotice` policy, and both recovery-notice paths remain prompt-visible or channel-visible without an operator-controlled gate; the related plugin-local recovery narrowing in #71289 and WhatsApp-specific banner filter in #49579 do not cover this general customer-facing recovery-notice problem.

## What This Changes

Request to add a Gateway config knob that controls whether last-known-good recovery notices are injected into the next agent turn, delivered out-of-band to the channel, logged only, or suppressed.

## Best Possible Solution

Add a Gateway-owned, documented config policy with a current-behavior default that gates both prompt-visible recovery system events and out-of-band restart-sentinel recovery delivery, while leaving actual last-known-good recovery and restart continuation delivery unchanged.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists on current main: whole-file recovery calls `enqueueConfigRecoveryNotice`, and `config-auto-recovery` restart sentinel handling calls `enqueueRestartSentinelWake` plus `deliverRestartSentinelNotice`; both paths lack a config-mode check and system events are prepended into prompt bodies.

## Solution Assessment

Yes. The proposed `gateway.systemEvents.recoveryNotice` knob is the narrow maintainable direction if it covers both recovery-notice implementations; extending banner-specific filters like #49579 would not solve the general operator-controlled policy need.

Implement the Gateway recovery-notice policy requested by issue #72043. Add `gateway.systemEvents.recoveryNotice` with modes `"user"` (default/current behavior), `"log"`, and `"off"`. Gate both recovery notice surfaces: the prompt-visible `enqueueConfigRecoveryNotice` path used by startup/reload config recovery, and the restart-sentinel recovery path that calls `enqueueRestartSentinelWake` and `deliverRestartSentinelNotice`.
