## Summary

Keep this issue open. Current main still has no `gateway.systemEvents.recoveryNotice` config contract, the config auto-recovery restart sentinel still injects the hardcoded recovery text into the agent system-event queue and outbound channel notice, and the newer startup/reload `Config recovery warning` path is also prompt-visible without a suppression policy.

## What This Changes

Request adding a `gateway.systemEvents.recoveryNotice` config option to control whether last-known-good config recovery notices are user-facing, log-only, or suppressed.

## Best Possible Solution

Add a Gateway-owned, documented config policy with a current-behavior default that controls recovery-notice delivery across prompt-visible system events, out-of-band channel notices, and log-only/suppressed modes while leaving actual recovery and restart continuations unchanged.

## Reproduction Assessment

Yes. A code-level reproduction is clear: a `config-auto-recovery` restart sentinel with a session route flows through `loadRestartSentinelStartupTask`, which unconditionally enqueues the message as a system event and sends the outbound notice; drained system events are then prepended to the next agent prompt.

## Solution Assessment

Yes, with one scope refinement. The proposed `gateway.systemEvents.recoveryNotice` knob is a narrow maintainable fix if it covers both restart-sentinel recovery notices and `Config recovery warning` notices, rather than adding another banner-specific filter like #49579.
