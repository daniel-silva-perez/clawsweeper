## Summary

Keep this issue open. Current main still hardcodes config-recovery notices into prompt-visible system events and, for the restart-sentinel auto-recovery path, also delivers the same message out-of-band to the originating channel. The strict Gateway config schema and docs do not contain `gateway.systemEvents.recoveryNotice`, and related #71289, #49579, and #39307 are adjacent rather than superseding this operator-controlled policy request.

## Best Possible Solution

Keep this open and implement a focused Gateway config policy: add `gateway.systemEvents.recoveryNotice` with a current-behavior default, gate the prompt-visible system-event and out-of-band restart-sentinel recovery notice surfaces, preserve restart continuations and recovery mechanics, document the operator tradeoff, and add targeted tests for `user`, `log`, and `off`. The fix should also explicitly handle or document the existing `Config recovery warning` startup/reload enqueue path so the policy is not partial.
