Current main still strips MiniMax XML invocations instead of promoting them to structured tool calls, but this mixed-scope PR is not merge-ready because the MiniMax parser mishandles valid XML and the gateway half is stale against the current reload architecture and active browser-profile tracker.

Replace or rework this into a focused MiniMax parser change on current main that promotes XML invocations through the existing `toolUse` execution path with attribute-order-independent parsing and standard XML entity decoding. Leave the browser-profile restart semantics to the active reload-classification work tracked by #43803 unless a fresh gateway patch is intentionally scoped to that issue.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Parse XML attributes independent of order:** `src/agents/pi-embedded-utils.ts:35`
  - body: `parseMinimaxToolCallXml` only matches `<invoke>` and `<parameter>` when `name` is the first attribute. Valid XML such as `<invoke id="x" name="Bash">...` is skipped, after which the sanitizer strips the XML and silently drops the intended tool call.
  - confidence: 0.9
- **[P2] Decode XML entities before passing tool inputs:** `src/agents/pi-embedded-utils.ts:48-51`
  - body: Parameter text is copied directly from XML captures, so encoded values like `&amp;`, `&lt;`, or `&quot;` reach tools unchanged. Decode standard XML entities before building `toolUse.input` so commands and file contents match what the model encoded.
  - confidence: 0.88
