## Summary

Keep open: current main already covers the gateway config.patch restart behavior, but it still does not implement the MiniMax XML tool-call parsing/promotion that this PR is trying to add. The PR remains useful as source work, but its MiniMax parser has blocking correctness gaps, so it should not be closed as implemented or merged as-is.

## What This Changes

This PR combines MiniMax M2.5 XML tool-call parsing/promotion in the embedded agent path with a gateway config.patch hot-reload restart change, plus regression tests and prompt/tool wording updates.

## Best Possible Solution

Rebase or replace this PR with a focused MiniMax-only change that promotes XML invocations into the existing toolUse execution path using robust XML/attribute parsing and entity decoding, while leaving the gateway config.patch behavior to the current main implementation. If maintainers decide not to support legacy MiniMax XML tool calls, close the MiniMax work explicitly with that rationale.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Parse XML attributes independent of order:** `src/agents/pi-embedded-utils.ts:35`
  - body: `parseMinimaxToolCallXml` only matches `<invoke>` when `name` is the first attribute, so valid XML like `<invoke id="x" name="Bash">...` is skipped. Because the later sanitizer strips the XML, the intended tool call is silently lost; the parser should read attributes in any order for both invoke and parameter tags.
  - confidence: 0.9
- **[P2] Decode XML entities before passing tool inputs:** `src/agents/pi-embedded-utils.ts:48-51`
  - body: Parameter text is copied directly from XML captures, so valid encoded values such as `&amp;`, `&lt;`, or `&quot;` reach tools unchanged. That changes commands and file contents from what the model encoded in the XML, so decode standard XML entities before building the `toolUse.input`.
  - confidence: 0.88
