## Summary

Keep open. Current main has already superseded the gateway config.patch restart half through the newer shared config-write flow, but it still only strips MiniMax XML from assistant text and does not promote MiniMax invocations into toolUse blocks. The PR therefore still addresses a real missing behavior, but its proposed MiniMax parser has blocking XML correctness gaps before it is safe to merge or replace.

## What This Changes

The PR adds MiniMax M2.5 XML invocation parsing/promotion into structured toolUse blocks and changes config.patch restart scheduling, gateway tool/prompt wording, and regression tests so hot-reloadable config patches skip direct SIGUSR1 restart scheduling.

## Best Possible Solution

Rebase or replace this with a focused MiniMax-only change on current main that promotes XML invocations into the existing toolUse execution path using robust attribute parsing and XML entity decoding, while leaving config.patch restart behavior to the current shared gateway config-write implementation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Parse XML attributes independent of order:** `src/agents/pi-embedded-utils.ts:35`
  - body: `parseMinimaxToolCallXml` only matches `<invoke>` when `name` is the first attribute, so valid XML like `<invoke id="x" name="Bash">...` is skipped. The later sanitizer strips that XML, so the intended tool call is silently lost; parse attributes in any order for both invoke and parameter tags.
  - confidence: 0.9
- **[P2] Decode XML entities before passing tool inputs:** `src/agents/pi-embedded-utils.ts:48-51`
  - body: Parameter text is copied directly from XML captures, so valid encoded values such as `&amp;`, `&lt;`, or `&quot;` reach tools unchanged. That changes commands and file contents from what the model encoded in the XML; decode standard XML entities before building `toolUse.input`.
  - confidence: 0.88
