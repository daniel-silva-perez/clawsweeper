## Summary

Keep this PR open. Current main still lacks `channel:<id>` candidates in the Slack channel resolver, so the proposed production fix addresses a real allowlist miss, but the PR should not merge until the new regression fixtures use the current typed `enabled` channel-entry shape.

## What This Changes

The PR adds `channel:<id>` case variants to Slack channel config candidate matching and adds regression tests for uppercase and lowercase prefixed allowlist keys.

## Best Possible Solution

Land the Slack-owned resolver change after replacing the new test fixtures' `allow` properties with `enabled: true` or omitting the flag, keeping the patch limited to channel-key matching and focused regression coverage.

Yes. A source-level reproduction is enough: configure `resolveSlackChannelConfig` with a bare runtime channel ID and only a `channel:C...` config key; current main never supplies that prefixed key to the exact matcher, so allowlist enforcement can reject the room.

## Solution Assessment

Yes with a small correction. Adding prefixed candidates in `extensions/slack/src/monitor/channel-config.ts` is the narrowest maintainable fix; the safer test shape is to use the current typed `enabled` property instead of the legacy `allow` alias.
