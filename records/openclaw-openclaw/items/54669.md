## Summary

Keep this issue open. Current main still does not document the Chrome-on-Windows IPv6 CDP binding/`v4tov6` workaround in the WSL2 remote CDP guide, and browser status/doctor diagnostics still probe only the configured CDP endpoint with generic HTTP/WebSocket failure categories. The closed #68666 report corroborates the same failure mode but is not an open canonical tracker because it was marked duplicate of this issue.

## Best Possible Solution

Keep this issue open and make a narrow browser-plugin docs/diagnostics update. The docs change should add Windows-side checks for `curl http://127.0.0.1:9222/json/version`, `curl http://[::1]:9222/json/version`, `netstat -ano | findstr :9222`, and `v4tov6 ... connectaddress=::1` when Chrome listens only on IPv6 loopback; it should also mention the non-default `--user-data-dir` workaround if maintainers validate it. The diagnostic change should add a WSL2/Windows remote-CDP hint path that can distinguish configured IPv4 endpoint failure from an IPv6 loopback listener instead of reporting only generic CDP reachability failure.
