## Summary

Keep this issue open. Current main still lacks the requested Chrome-on-Windows IPv6 CDP documentation and diagnostic behavior for WSL2 remote browser profiles: the WSL2 troubleshooting guide does not mention `[::1]`, `netstat`, `v4tov6`, or the non-default `--user-data-dir` workaround, and browser status/diagnostics still probe only the configured CDP URL with generic HTTP/WebSocket failure categories. The related #68666 report is useful corroboration, but it is closed as a duplicate of this issue, so it is not a live canonical tracker for the remaining work.

## Best Possible Solution

Keep this issue open. The best maintainer path is a narrow browser-plugin docs and diagnostics update: add Chrome 136+/147+ Windows guidance to `docs/tools/browser-wsl2-windows-remote-cdp-troubleshooting.md` covering `curl http://127.0.0.1:9222/json/version`, `curl http://[::1]:9222/json/version`, `netstat -ano | findstr :9222`, `v4tov6 ... connectaddress=::1` portproxy when Chrome listens on IPv6 loopback, and the non-default `--user-data-dir` requirement if validated. Then extend browser doctor/status or CDP failure hints to recognize this WSL2/Windows remote-CDP failure mode instead of surfacing only generic reachability errors.
