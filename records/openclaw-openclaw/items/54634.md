## Summary

Keep open. Current main still does not implement the reported cross-HOME update safeguard. The adjacent config-clobber recovery and doctor multiple-state-dir warning help nearby failure modes, but the update/service refresh path still discovers and regenerates the gateway service from the updater process environment unless explicit OpenClaw path overrides are already present.

## Best Possible Solution

Keep this issue open and fix the Linux update/service refresh path. Before regenerating or restarting an installed gateway service, discover the previous unit/config root even when the updater runs under a different HOME, compare HOME, OPENCLAW_HOME, OPENCLAW_STATE_DIR, and OPENCLAW_CONFIG_PATH, then preserve the prior root when safe or emit an explicit migration warning/block. Add a regression test for a service installed with HOME=/home/openclaw and refreshed by an update process running with HOME=/root.

## Risks / Open Questions

- The current package update path can still miss an existing /home/openclaw systemd unit/config when invoked from HOME=/root and no explicit OPENCLAW_* path overrides are present.
- Config auto-restore and same-root backup recovery do not recover an old config that lives under a different HOME and was never read in the new root.
- The GHCR version-tag request in the discussion is useful rollback work, but it would not fix the core service/config-root split.
