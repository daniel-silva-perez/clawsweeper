## Summary

Keep open. Current main still lacks the reported cross-HOME update safeguard: update service discovery starts from the updater process environment, Linux user-unit lookup is HOME-keyed, and forced service reinstall deliberately drops prior HOME and OPENCLAW_* control roots. The existing diagnostics and related open PRs are useful adjacent work, but they do not yet preserve, migrate, warn, or block when a root-invoked update would abandon a dedicated-user config/state root.

## What This Changes

The issue asks OpenClaw update/service refresh to avoid silently abandoning existing config and state when the managed gateway service HOME changes during an update.

## Best Possible Solution

Add a Linux update/service-refresh guard that discovers the previously installed managed gateway service root before rewriting it, compares HOME, OPENCLAW_HOME, OPENCLAW_STATE_DIR, and OPENCLAW_CONFIG_PATH with the updater environment, and either safely preserves the prior root or emits a clear blocking migration warning. Cover the dedicated-user service refreshed from root case in update, daemon install, systemd, and doctor regression tests; keep GHCR version-tag rollback as separate work.
