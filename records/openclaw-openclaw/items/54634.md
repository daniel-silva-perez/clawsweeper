## Summary

Keep open. Current main has several adjacent update and service-env hardenings, but it still does not handle the reported cross-HOME update case: service discovery and refresh begin from the updater process environment, systemd unit lookup is HOME-keyed, and forced reinstall intentionally drops prior HOME and OPENCLAW_* control paths instead of preserving, migrating, warning, or blocking when they changed.

## What This Changes

The issue asks OpenClaw updates to preserve, migrate, or warn before abandoning an existing gateway config/state root when the managed service HOME changes from a dedicated user to root.

## Best Possible Solution

Add a Linux update/service-refresh guard that identifies the previously installed gateway service root before rewriting it, compares HOME, OPENCLAW_HOME, OPENCLAW_STATE_DIR, and OPENCLAW_CONFIG_PATH against the updater environment, and either preserves the prior root when safe or emits a clear migration/blocking warning. Cover it with a regression test for a service installed under HOME=/home/openclaw and refreshed by an updater running with HOME=/root; treat the GHCR version-tag request from the discussion as separate rollback/pinning work.
