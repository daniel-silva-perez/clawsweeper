## Summary

Keep open. Current main still classifies ordinary historical primary transcripts as orphan candidates solely because they are absent from the active `sessions.json` map; the interactive confirmation and archived-transcript discovery mitigations reduce damage but do not fix the classifier described here.

## What This Changes

The issue reports that `openclaw doctor --fix` treats valid historical session transcripts as orphan files and asks for classifier, prompt, restore, corrupt-store, and doctor-recommendation CI hardening.

## Best Possible Solution

Define real orphan semantics in the doctor state-integrity path first: preserve valid historical primary transcripts even when absent from the active session-key map, retain sidecar/archive exclusions, and add regression coverage. Then split prompt wording, restore support, corrupt-store diagnostics, and broad doctor-recommendation CI hardening into scoped follow-ups.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to create one active `sessions.json` entry plus an older valid `<sessionId>.jsonl` absent from the store; current `noteStateIntegrity` will mark that older primary transcript orphan because it is not in `referencedTranscriptPaths`.

## Solution Assessment

No. The current confirmation gate and `.deleted.*` discovery are useful mitigations, but they do not solve the false orphan definition; the narrow maintainable fix is to change the classifier and tests before expanding into restore UX or CI policy.
