## Summary

Keep open. Current main still classifies primary transcript files as orphan candidates solely because they are absent from the current `sessions.json` entries; the interactive-only confirmation and archived-transcript discovery work are useful mitigations, but they do not fix the underlying historical-transcript classification bug.

## What This Changes

The issue requests that doctor stop archiving valid historical primary session transcripts as orphans, improve warning/recovery behavior, surface corrupt session-store reads, and add stronger CI coverage around doctor recommendations.

## Best Possible Solution

Fix `src/commands/doctor-state-integrity.ts` so a valid historical primary transcript is not considered orphan solely because it is absent from the active session-key map. Add a regression test with an older valid transcript plus a newer current store entry, keep sidecar/archive exclusions, reword the prompt around discoverability/history impact, and split restore/corrupt-store/CI audit work if maintainers want smaller PRs.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is one active `sessions.json` entry plus an older valid `<id>.jsonl` in the same sessions directory; current `noteStateIntegrity` will place the older file in `orphanTranscriptPaths` because it is primary JSONL and absent from `referencedTranscriptPaths`.

## Solution Assessment

No. Current main's interactive-only confirmation and archived-file discovery are partial mitigations, but the narrow maintainable fix is to stop classifying valid historical transcripts as orphanable in the first place.
