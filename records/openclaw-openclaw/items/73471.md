## Summary

Keep open. Current main still defines orphan transcripts by absence from the active `sessions.json` map, so valid older primary transcripts can still be offered for `.deleted.<timestamp>` renaming; the recent interactive-confirmation and archived-discovery changes reduce damage but do not fix the classifier described here.

## What This Changes

The issue reports that `openclaw doctor --fix` can classify valid historical primary session transcripts as orphan files and rename them, hiding chat history, and asks for classifier, prompt, recovery, store-load, and CI hardening.

## Best Possible Solution

Define a real orphan transcript in the doctor state-integrity path instead of using absence from the active session-key map. A focused first fix should preserve valid historical primary transcripts, keep sidecar/archive exclusions, add regression coverage for older complete transcripts, and then split prompt wording, restore support, corrupt-store warnings, and doctor recommendation CI hardening into scoped follow-ups.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is clear: create a sessions directory with one active `sessions.json` entry and an older valid `<sessionId>.jsonl` absent from the store, then `noteStateIntegrity` will classify that older transcript as orphan because it is primary JSONL and missing from `referencedTranscriptPaths`.

## Solution Assessment

No. The current mitigation requiring interactive confirmation is safer than silent repair, but it does not solve the definition bug; the narrow maintainable solution is to change the classifier and tests first, then handle UX and CI hardening separately.
