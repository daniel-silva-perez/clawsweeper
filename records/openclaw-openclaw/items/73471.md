## Summary

Keep open. Current main still treats any primary session transcript not referenced by the active `sessions.json` map as an orphan candidate, and the prompt/test coverage still match the reported unsafe semantics; the newer interactive-only and archived-transcript indexing changes are partial mitigations, not a fix.

## What This Changes

The issue requests that doctor stop archiving valid historical primary session transcripts as orphans, improve warning/recovery behavior, warn on corrupt session-store reads, and add stronger CI coverage for doctor recommendations.

## Best Possible Solution

Fix the owner path in `src/commands/doctor-state-integrity.ts` so a valid historical primary transcript is not considered orphan solely because it is absent from the active session-key map. Add regression coverage with a real older transcript, keep the existing sidecar/archive exclusions, reword the prompt to state history/discoverability impact, and decide whether restore plus corrupt-store warnings belong in the same PR or split follow-ups.

## Reproduction Assessment

Yes. A read-only source trace is enough: create one active `sessions.json` entry plus an older valid `<id>.jsonl` in the sessions directory, and current `noteStateIntegrity` will classify the older file as orphan because it is primary JSONL but not in `referencedTranscriptPaths`.

## Solution Assessment

No. Current main's interactive-only confirmation and archived-file indexing reduce the worst silent-loss path, but the narrow maintainable solution is to stop classifying valid historical transcripts as orphanable in the first place.
