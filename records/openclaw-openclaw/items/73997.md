## Summary

Keep open. Current main still lacks the requested WhatsApp `dmPolicy: "observe"` behavior: the config contract rejects that enum value, blocked DMs are dropped without an observation store, and admitted DMs still resolve a route, record session metadata, and enter the agent reply turn. The existing delivery-suppression and `observeOnly` surfaces are not equivalent because they still run the agent turn or are scoped to other passive delivery flows.

## What This Changes

The issue requests a WhatsApp `dmPolicy: "observe"` mode that logs incoming DMs to a read-only, queryable observation store without creating or routing an agent session or allowing replies.

## Best Possible Solution

If maintainers approve the product and privacy model, add a WhatsApp-owned observe policy that is accepted by the shared config schema and docs, intercepts admitted direct messages after parsing/access normalization but before route/session resolution, appends bounded observation records, and exposes a guarded read tool with account, sender, time-range, retention, and redaction controls.

## Reproduction Assessment

Not applicable as a runtime bug report, but the missing feature has a high-confidence source verification path: `dmPolicy: "observe"` is absent from the type/schema/docs, blocked DMs are dropped, and admitted DMs route through `processMessage` into `runInboundReplyTurn`.

## Solution Assessment

Unclear until maintainer product/security review. The requested pre-routing observe mode is the narrow direction that matches the no-agent-turn requirement; `sendPolicy: "deny"`, `NO_REPLY`, plugin `message_received` hooks, and channel-turn `observeOnly` are not equivalent because they still expose the inbound message to the agent pipeline or only suppress delivery.

## Security Review

Status: needs_attention

Summary: No patch is under review, but the proposed feature would persist and expose third-party WhatsApp message content, so the design needs explicit privacy and access-control review.

Concerns:

- **[medium] Define observation log retention and access policy:**
  - body: The requested store would contain personal WhatsApp content, phone numbers, sender names, and account metadata. A safe implementation needs bounded retention, deletion behavior, redaction, account scoping, and a tool-access policy before exposing the data to agents.
  - confidence: 0.9
- **[medium] Avoid routing observed DMs through existing agent hooks:** `extensions/whatsapp/src/auto-reply/monitor/process-message.ts:435`
  - body: Using existing `message_received`, `sendPolicy`, or `observeOnly` paths would still run inside or near the agent/session pipeline. The design should mechanically prevent agent turns and reply paths for observed DMs rather than relying on model instructions or delivery suppression.
  - confidence: 0.86
