## Summary

Keep open. Current main still does not implement the strict WhatsApp observe mode requested here: `observe` is not an accepted `dmPolicy`, blocked DMs are discarded instead of persisted to an observation store, and admitted DMs still enter session recording and the inbound reply-turn path. The prior send-policy and hook-based closures cover reply suppression or post-routing observation, but they do not satisfy this issue's no-session/no-agent-turn requirement.

## What This Changes

The issue requests a WhatsApp `dmPolicy: "observe"` mode that records incoming DMs in a read-only queryable log without creating sessions, triggering agent turns, or exposing a reply path.

## Best Possible Solution

If maintainers approve the product and privacy model, add an explicit pre-routing observation path for WhatsApp DMs: accept `dmPolicy: "observe"` in the shared config contract, intercept inbound direct messages after normalization and access-policy resolution but before session/route resolution, append bounded observation records, and expose a guarded read surface with account, sender, time-range, retention, and redaction controls. Update WhatsApp docs, config metadata, and targeted tests with the same behavior.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for source verification: `dmPolicy: "observe"` is absent from the config type/schema/docs, blocked DMs return `null` before any store write, and admitted DMs call `runInboundReplyTurn`. A live WhatsApp account is not needed to confirm the current contract gap.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainer product/security review. A pre-routing observe branch is the right mechanical shape for the no-agent-turn requirement, but the observation store, agent read tool, retention, and access model should be approved before a fix PR is queued.
