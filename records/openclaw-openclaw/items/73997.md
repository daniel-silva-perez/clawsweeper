## Summary

Keep open. Current main still does not implement the requested strict WhatsApp observe mode: `dmPolicy` rejects `observe`, blocked DMs are dropped without a read-only observation store, and admitted DMs still enter session routing and `runInboundReplyTurn`. The older send-policy and hook-based resolutions cover delivery suppression, but not this issue's no-session/no-agent-turn requirement.

## What This Changes

The issue requests a WhatsApp `dmPolicy: "observe"` mode that logs incoming DMs to a read-only queryable store without creating or routing an agent session or allowing replies.

## Best Possible Solution

If maintainers approve the product and privacy model, add a WhatsApp-owned observe policy accepted by the shared config schema and docs, intercept admitted direct messages after parsing/access normalization but before route/session resolution, append bounded observation records, and expose a guarded read tool with account, sender, time-range, retention, and redaction controls.

## Reproduction Assessment

Not applicable as a runtime bug report, but there is a high-confidence source verification path: `dmPolicy: "observe"` is absent from type/schema/docs, blocked DMs are dropped, and admitted DMs route through `processMessage` into `runInboundReplyTurn`.

## Solution Assessment

Unclear until maintainer product/security review. A pre-routing WhatsApp observe branch matches the no-agent-turn requirement better than `sendPolicy: "deny"`, plugin hooks, or `observeOnly`, but the observation-store access and retention model needs approval first.
