## Summary

Keep open. Current main still does not implement the requested strict WhatsApp observe mode: `observe` is not accepted by the DM policy contract, blocked DMs are dropped instead of logged, and admitted DMs still resolve an agent route and enter `runInboundReplyTurn`. The earlier read-only/send-policy resolutions are useful context, but they suppress delivery after processing and do not satisfy this request's no-session/no-agent-turn boundary.

## What This Changes

Add a WhatsApp `dmPolicy: "observe"` mode that records inbound DMs in a read-only queryable log without creating sessions, running agents, or exposing reply paths.

## Best Possible Solution

If maintainers approve the product and privacy model, add a WhatsApp-owned observe policy accepted by the shared config type/schema and docs, branch after inbound normalization/access resolution but before route/session resolution, append bounded observation records, and expose a guarded read tool with account, sender, time-range, retention, and redaction controls.

## Reproduction Assessment

Not applicable as a runtime bug report, but there is a high-confidence source verification path: `dmPolicy: "observe"` is absent from the type/schema/docs, disallowed DMs are dropped, and admitted DMs route through `processMessage` into `runInboundReplyTurn`.

## Solution Assessment

Unclear until maintainer product/security review. A pre-routing observe branch is directionally the narrowest way to satisfy the no-agent-turn requirement; `sendPolicy: "deny"`, hooks, and instruction-only approaches are not equivalent because they still allow inbound agent processing.
