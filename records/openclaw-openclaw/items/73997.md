## Summary

Keep open. Current main still does not implement the requested WhatsApp DM observe policy: `dmPolicy` accepts only the existing four values, blocked DMs are dropped instead of persisted to a read-only observation store, and admitted DMs still resolve a session and enter the inbound reply turn path. The earlier ClawSweeper comment remains accurate because the central WhatsApp/config files are unchanged from its checked SHA to current main.

## What This Changes

Add a WhatsApp `dmPolicy: "observe"` mode that records inbound DMs for later read-only agent context without creating sessions, firing agent turns, or exposing a reply path.

## Best Possible Solution

If maintainers approve the product and privacy model, implement this as a WhatsApp-owned pre-routing observe policy accepted by the shared config contract, append bounded observation records before session resolution, and expose a guarded read tool with account, sender, time-range, retention, and redaction controls.

## Reproduction Assessment

Yes for source verification, not as a live runtime repro. On current main, `dmPolicy: "observe"` is rejected by the type/schema contract, blocked DMs return `null`, and allowed DMs flow through `runInboundReplyTurn`.

## Solution Assessment

Unclear pending maintainer product/security review. A WhatsApp pre-routing observe branch is closer to the requested mechanical safety than `sendPolicy: "deny"` or hooks, but the observation-store access model must be designed first.
