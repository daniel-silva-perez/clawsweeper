## Summary

Keep open. Current main still does not implement WhatsApp `dmPolicy: "observe"`: the shared config type/schema and WhatsApp docs only allow `pairing`, `allowlist`, `open`, and `disabled`; disabled/blocked DMs are dropped before any observation store; allowed DMs resolve an agent route and enter `processMessage`, which records session metadata and dispatches the agent reply pipeline. Existing `session.sendPolicy: "deny"` is not equivalent because it suppresses outbound delivery while explicitly still letting the agent process the inbound message. The related closed reports covered weaker receive-but-don't-auto-send behavior and do not supersede this stricter no-session/no-agent-turn request.

## Best Possible Solution

Keep this open for maintainer product/security review. The best path is likely a WhatsApp-owned `dmPolicy: "observe"` implementation that accepts the new config value, intercepts admitted direct messages after parsing/access normalization but before route/session resolution, writes bounded privacy-aware observation records, and exposes a guarded query tool only after maintainers approve the retention and access model. If the query/observation mechanism should be generic, add a documented plugin SDK seam first rather than hard-coding a late reply-suppression workaround.

## Risks / Open Questions

- A real implementation would store personal WhatsApp DM content, so retention, redaction, account scoping, encryption/storage location, and agent/tool access policy need maintainer review before a fix PR is safe.
- Implementing this as `sendPolicy: "deny"`, `NO_REPLY`, or a late `before_dispatch` hook would still create routed session context or expose inbound DM content to the agent pipeline, which misses the issue's mechanical-safety requirement.
- Adding `observe` to the shared `DmPolicy` enum without carefully scoping channel behavior could accidentally affect other channels that reuse the shared policy contract.
- Current plugin hooks can help observe already-routed messages, but blocked/disabled DMs are dropped before those hooks, so closing as a ClawHub-only/plugin task would leave a missing WhatsApp/core seam unresolved.
