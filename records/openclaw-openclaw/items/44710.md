## Summary

Keep open. Current main still does not implement this list-level plugin hook or an equivalent final-tool-list API, and the latest PR diff remains security-sensitive and not merge-ready because policy-hook failures still fail open and denied client tool names can affect bundled tool materialization.

## What This Changes

The PR adds a public `before_tools_resolve` plugin hook that filters the assembled tool list for normal runs and compaction using allow/deny results, plus hook runner support and tests.

## Best Possible Solution

If maintainers want this API, rebase it onto the current hook contract surface, define the hook in `src/plugins/hook-types.ts`, fail closed for policy-hook errors/timeouts, evaluate once over the final post-static tool list, ensure denied client tools cannot reserve or suppress bundled tools, and add docs plus regression tests for hook registration, failure behavior, allow intersection, client tools, bundled MCP/LSP tools, and compaction.

## Reproduction Assessment

Yes for the review findings: the source/diff trace shows the new hook is not added to the global fail-closed policy and that pre-hook client tool names are used for bundled reserved names. Runtime reproduction was not attempted because the review was read-only.

## Solution Assessment

No. The requested API is plausible, but the current PR is not the best merge shape until it is rebased to the current hook contract files and its access-control semantics fail closed without letting denied client tools affect later bundled tool registration.

## Security Review

Status: needs_attention

Summary: The diff introduces an access-control hook for model-visible tools, and the current patch can fail open on policy-hook errors and lets denied client tool names influence bundled tool registration.
