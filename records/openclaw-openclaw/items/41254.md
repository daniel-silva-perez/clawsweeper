## Summary

Keep open. Current main and the latest release still record delivered heartbeat dedupe metadata with a stale load/mutate/save snapshot, so the PR is not obsolete; however, the branch is stale because removing `saveSessionStore` no longer matches current main, where the same file still imports and calls it for other heartbeat writes.

## What This Changes

The branch changes `src/infra/heartbeat-runner.ts` to record delivered heartbeat dedupe metadata through `updateSessionStore` instead of raw `loadSessionStore`/`saveSessionStore`, and removes the `saveSessionStore` import from its original base.

## Best Possible Solution

Land a rebased, narrow heartbeat-runner fix that switches only the delivered heartbeat dedupe recording to `updateSessionStore`, preserves the remaining `saveSessionStore` import and call sites, and adds focused regression coverage for concurrent session-store writes during heartbeat delivery.

## Reproduction Assessment

Yes. The race is reproducible by source inspection: a concurrent writer between `loadSessionStore` and `saveSessionStore` can be overwritten by the stale snapshot, while `updateSessionStore` re-reads under the lock and current tests already prove that API serializes concurrent updates.

## Solution Assessment

No, not exactly as currently proposed against current main. The `updateSessionStore` change is the right narrow fix, but the stale import deletion must be adjusted and a heartbeat-specific race test should be considered before merge.
