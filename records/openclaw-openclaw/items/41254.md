## Summary

Keep this PR open. Current main still has the reported heartbeat dedupe race, and the proposed `updateSessionStore` direction is the right narrow fix, but the branch is stale against current main because removing `saveSessionStore` would break remaining call sites in `src/infra/heartbeat-runner.ts`.

## What This Changes

The PR changes `src/infra/heartbeat-runner.ts` to record delivered heartbeat dedupe metadata through `updateSessionStore` instead of a raw `loadSessionStore`/`saveSessionStore` snapshot, and removes the `saveSessionStore` import from its original base.

## Best Possible Solution

Land a rebased, narrow heartbeat-runner fix that switches only the delivered heartbeat dedupe metadata write to `updateSessionStore`, preserves remaining `saveSessionStore` usage, and adds focused regression coverage for concurrent session-store updates during heartbeat delivery if maintainers want a guard for this race.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a concurrent writer that updates the session store after the heartbeat reads it but before the heartbeat saves it can be overwritten by the stale snapshot; the existing `updateSessionStore` concurrency test shows the intended serialization primitive.

## Solution Assessment

No, not exactly as currently proposed against current main. The core `updateSessionStore` change is the right maintainable fix, but the stale import deletion must be adjusted before this can merge cleanly.
