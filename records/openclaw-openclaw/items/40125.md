## Summary

Current main still does not implement the central request: proactive, one-time context-pressure warnings in Telegram or other persistent auto-reply chat sessions before a context overflow. Main has related post-failure recovery, auto-reset paths, preflight compaction, `/context` reporting, and a Control UI notice, but those do not cover threshold warning/urgent messages sent into persistent chat before provider invocation.

## Best Possible Solution

Keep the issue open as a focused persistent-chat UX enhancement. Implement context-pressure evaluation in the auto-reply/session pipeline before provider invocation, persist per-session warning and urgent threshold state, and send concise `/reset` and `/new` guidance once per threshold for Telegram and similar persistent channels. Reuse existing `totalTokens`, `contextTokens`, preflight compaction, `/context`, and Control UI threshold logic where practical.
