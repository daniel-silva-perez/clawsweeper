## Summary

Keep open. Current main has preflight compaction, mid-turn context checks, post-overflow reset/recovery copy, `/context` reporting, optional compaction start/end notices, and a Control UI high-context notice, but it still lacks the central requested behavior: proactive, one-time context-pressure warning and urgent guidance delivered into persistent chat sessions before provider invocation.

## What This Changes

The issue requests proactive context-window pressure warnings in persistent chat sessions, plus explicit `/reset` or `/new` recovery guidance before model context overflow.

Implement this as a focused auto-reply/session UX feature: evaluate fresh context usage before provider invocation, persist warning and urgent threshold state on the session, and send concise persistent-chat guidance that names `/reset` and `/new` while reusing existing context-token and compaction primitives.

## Repair Work Prompt

Implement proactive context-pressure warnings for persistent chat sessions tracked by openclaw/openclaw#40125. Current main already has preflight compaction, mid-turn checks, `/context`, `/status`, post-overflow recovery, optional compaction notices, and a Control UI context notice, but Telegram and similar auto-reply sessions do not receive one-time warning/urgent guidance before provider invocation.

Expected boundary:
- Add a small helper in the auto-reply/session pipeline before provider invocation that evaluates fresh context usage, using existing `totalTokens`/`contextTokens` and the transcript fallback patterns already used by preflight compaction where practical.
- Persist per-session warning state so warning and urgent thresholds are emitted once per threshold and reset after `/reset`, `/new`, successful compaction that drops usage below threshold, or session replacement.
- Send concise chat-visible guidance for persistent channel sessions: the session is nearing the model context limit, continuing in the same chat keeps history/tool output in context, use `/reset` to clear this session or `/new` to start fresh. Mention `/compact` only if it is appropriate for that path.
- Avoid warnings from stale token snapshots, heartbeats, CLI-only paths, and non-chat/internal runs.
- Do not change provider catalogs, fallback model selection policy, dependencies, lockfiles, workflows, or channel-specific transport internals unless strictly needed.
- Add focused tests for warning threshold, urgent threshold, dedupe, stale-token suppression, and reset/new clearing behavior. Update docs if user-visible behavior changes; add a changelog entry only if repository policy expects one for this user-facing enhancement, with reporter credit.
