## Summary

Keep open. Current main still does not implement `model.tool`, and the proposed branch is not merge-ready because the effective tool model does not drive the full model-call pipeline, including tool enablement, provider transport, auth, provider-specific wrappers, diagnostics, and generated config docs.

## What This Changes

The branch adds a `model.tool` config field, resolver/schema support, and a Pi embedded runner stream wrapper/tests for routing tool-continuation turns to an alternate model.

## Best Possible Solution

Keep the PR open for maintainer architecture and security review. If split routing stays in core, the effective per-call model should drive the entire call path: tool availability, schema normalization, provider stream dispatch, auth profile/API key selection, provider-specific wrappers, transcript policy, context budgeting, fallbacks, diagnostics, trajectory metadata, generated config contracts, docs, and integration coverage. If maintainers prefer plugin routing, continue the design in #13225 before merging a core config key.

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete provider/auth boundary risk because tool-model turns can reuse primary-provider transport and credentials.

Concerns:

- **[medium] Tool turns can cross provider/auth boundaries:** `src/agents/pi-embedded-runner/run/attempt.ts:1287`
  - body: The added wrapper changes the model object after the stream function, provider-specific wrappers, and API-key resolution have already been built for the primary provider. In cross-provider configs, model input intended for the configured tool provider can be sent through primary-provider transport/auth behavior or fail under the wrong credential boundary.
