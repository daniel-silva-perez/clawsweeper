## Summary

Keep open. Current main does not already implement the PR’s chain-wide collector/runtime behavior, and the latest PR head still has a concrete security-sensitive regression: raw nested error messages can bypass the existing redaction boundary when they are reused for diagnostics or user-facing failure text.

## What This Changes

The PR adds `.error`/`.cause` chain message collection and uses it for context-overflow detection in model fallback, embedded overflow compaction, and auto-reply classification, with focused tests plus a MiniMax test ordering tweak.

## Best Possible Solution

Revise the branch so raw chain candidates are available only to classifiers, while any diagnostic, log, stored failure, reset-helper, or user-facing text is derived from `formatErrorMessage()` or an equivalent redaction path. Keep the useful cycle-safe `.error`/`.cause` detection and the billing/rate-limit precedence tests, and add a regression test for nested credentials in chain messages.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Redact chain messages before surfacing them:** `src/agents/failover-error.ts:214`
  - body: `collectErrorChainMessages()` returns direct nested `message` strings, and the new runtime call sites use those strings as overflow/error text. That bypasses the existing `formatErrorMessage()` redaction boundary, so an inner provider error containing `Authorization: Bearer ...`, an API key, or serialized request metadata can leak into logs or failure replies. Keep raw chain candidates separate from display text, or redact collected messages before returning them for those surfaces.
  - confidence: 0.91

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete secret-redaction regression by exposing raw nested error-chain messages outside the existing formatter/redaction path.

Concerns:

- **[high] Raw nested error messages can leak secrets:** `src/agents/failover-error.ts:214`
  - body: Provider and transport errors may include bearer tokens, API keys, or serialized request metadata in nested `.error`/`.cause.message` fields. The PR collects those raw strings and reuses them for runtime diagnostics and failure text instead of consistently passing display text through `formatErrorMessage()` or `redactSensitiveText()`.
  - confidence: 0.9
