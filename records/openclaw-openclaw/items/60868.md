## Summary

Keep open. Current main still does not implement the PR's chain-wide `.error`/`.cause` overflow detection at the affected runtime call sites, so the work is not obsolete; however, the latest patch remains incorrect because raw nested error messages can bypass the existing redaction boundary and reach logs or user-visible failure text.

## What This Changes

The PR adds a shared error-chain message collector and applies chain-aware context-overflow/failure classification across model fallback, embedded overflow compaction, and auto-reply handling, with tests plus small Discord/Gateway system-event and changed-lanes test edits.

## Best Possible Solution

Revise the branch so raw chain candidates are used only for classification, while all logged, reset-helper, stored failure, Control UI, and user-facing text is derived from `formatErrorMessage()`/`redactSensitiveText()` or an equivalent redacted display path. Preserve the chain-wide `.error`/`.cause` overflow detection, billing/rate-limit precedence, structured status/code hints, and add a nested-secret redaction regression test.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Redact chain messages before surfacing them:** `src/agents/failover-error.ts:214`
  - body: `collectErrorChainMessages()` returns raw nested `message` strings, and the new call sites reuse those strings as diagnostic/failure text. That bypasses the existing `formatErrorMessage()` redaction boundary, so an inner provider error containing an Authorization header, API key, or serialized request metadata can leak to logs or users. Keep raw chain candidates classifier-only, or redact them before producing display/log text.
  - confidence: 0.92

## Security Review

Status: needs_attention

Summary: The patch introduces a concrete redaction regression by allowing raw nested provider messages to reach logs or user-visible failure text.

Concerns:

- **[high] Raw nested error messages can leak secrets:** `src/agents/failover-error.ts:214`
  - body: Provider and transport errors may include bearer tokens, API keys, or serialized request metadata in nested `.error` or `.cause.message` fields. The PR collects those strings directly and uses them for runtime diagnostics and failure text instead of consistently passing display text through `formatErrorMessage()` or `redactSensitiveText()`.
  - confidence: 0.9
