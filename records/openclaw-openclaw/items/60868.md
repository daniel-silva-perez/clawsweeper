## Summary

Keep open. Current main does not contain the PR’s chain-message collector, so the intended bug fix is not already implemented, but the latest branch still introduces a blocking security regression by routing raw nested error messages into diagnostic and user-facing failure paths instead of the existing redacted formatter boundary.

## What This Changes

The PR adds a `.error`/`.cause` message collector and uses it for context-overflow and failure classification in model fallback, embedded overflow compaction, and auto-reply handling, with related tests plus a MiniMax test ordering tweak.

## Best Possible Solution

Revise the branch so raw chain candidates are used only for classification, while any log, stored metadata, reset-helper input, diagnostic text, or user-visible failure copy comes from `formatErrorMessage()` or an equivalent `redactSensitiveText()` path. Keep the useful chain-wide overflow detection and billing precedence tests, and add a regression test for nested credentials in `.error` / `.cause.message`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Redact collected chain messages before display:** `src/agents/failover-error.ts:218`
  - body: `collectErrorChainMessages()` returns direct nested `message` strings, and the new call sites reuse those strings as failure text for diagnostics, run metadata, reset helpers, and some user-facing branches. That bypasses the existing `formatErrorMessage()` redaction boundary, so an inner provider or transport error containing `Authorization: Bearer ...`, an API key, or serialized request metadata can leak. Keep raw candidates internal to classifiers or redact collected messages before they leave that boundary.
  - confidence: 0.92

## Security Review

Status: needs_attention

Summary: The PR introduces a concrete secret-redaction regression by exposing raw nested error-chain messages outside the existing formatter/redaction path.

Concerns:

- **[high] Raw nested error messages can leak secrets:** `src/agents/failover-error.ts:218`
  - body: Provider and transport wrappers may include bearer tokens, API keys, or request metadata in nested `.error.message` or `.cause.message` fields. The PR collects those raw strings and routes them into runtime diagnostics, stored metadata, and failure text instead of consistently using `formatErrorMessage()` or `redactSensitiveText()`.
  - confidence: 0.9
