## Summary

Keep this PR open. Current main still lacks the proposed chain-wide message collection at the model fallback, embedded overflow, and auto-reply call sites, so the change is not redundant; however, the latest patch should not merge yet because it introduces an unredacted error-message path for nested cause/error messages.

## What This Changes

The PR adds a shared error-chain message collector and applies chain-aware context-overflow classification to model fallback, embedded overflow compaction, and auto-reply failure classification, with focused tests plus a small MiniMax test ordering edit.

## Best Possible Solution

Revise the branch so chain-wide classification can inspect `.error` and `.cause` candidates without using raw collected messages for logs or user-facing text. Keep billing/rate-limit precedence, preserve structured status/code hints, add a redaction regression test for nested messages, and keep unrelated churn out of the final diff unless a maintainer wants it included.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Redact chain messages before surfacing them:** `src/agents/failover-error.ts:218`
  - body: `collectErrorChainMessages()` returns raw nested `message` strings, and the new call sites use those strings as `errorText`/`message` for diagnostics and failure replies. This bypasses the existing `formatErrorMessage()` redaction path, so an inner provider error containing `Authorization: Bearer ...`, an API key, or another secret can leak to logs or users. Redact collected messages before returning them, or keep raw candidates separate from redacted display text.
