## Summary

Keep open; current main does not contain the chain-wide overflow detection, and the latest PR head still needs revision because raw nested error messages can bypass the existing redaction boundary before merge.

## What This Changes

The PR adds `.error`/`.cause` message collection for agent context-overflow classification across model fallback, embedded runner overflow handling, and auto-reply failure classification, with regression tests plus small Discord/Gateway system-event trust and changed-lanes test updates.

## Best Possible Solution

Revise the branch so raw chain candidates are available only to classifiers, while logs, reset helpers, stored failure details, Control UI text, and user-facing replies are derived from `formatErrorMessage()` or an equivalent redaction path. Keep the cycle-safe `.error`/`.cause` detection and precedence regression tests, and split unrelated churn if maintainers do not want it in this fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Redact chain messages before surfacing them:** `src/agents/failover-error.ts:214`
  - body: `collectErrorChainMessages()` returns raw nested `message` values, and the new call sites turn those values into diagnostic/log and failure text. That bypasses the existing `formatErrorMessage()` redaction boundary, so a nested provider or transport error containing an Authorization header, API key, or serialized request metadata can leak. Keep raw chain candidates classifier-only, or redact collected messages before producing display/log text.
  - confidence: 0.92

## Security Review

Status: needs_attention

Summary: The PR introduces a concrete redaction regression by allowing raw nested provider error messages to reach logs or user-visible failure text.

Concerns:

- **[high] Raw nested error messages can leak secrets:** `src/agents/failover-error.ts:214`
  - body: Provider and transport errors may include bearer tokens, API keys, or serialized request metadata in nested `.error` or `.cause.message` fields. The PR collects those strings directly and reuses them for runtime diagnostics and failure text instead of consistently passing display text through `formatErrorMessage()` or `redactSensitiveText()`.
  - confidence: 0.9
