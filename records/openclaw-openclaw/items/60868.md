## Summary

Keep open. Current main still lacks the proposed chain-wide context-overflow detection, and the latest PR remains useful, but it is not mergeable because the embedded-runner overflow diagnostic path can still surface raw nested provider error messages into logs.

## What This Changes

The PR adds a chain-message collector and uses it to classify nested `.cause`/`.error` context-overflow signals across failover, model fallback, embedded overflow compaction, and auto-reply handling, with related tests plus small Discord/Gateway trust and changed-lanes test edits.

## Best Possible Solution

Keep the chain-wide context-overflow classifier, but separate raw classifier candidates from any surfaced diagnostic text. Logs, replies, stored failure text, and Control UI output should use `formatErrorMessage()`, the default redacted collector, or an equivalent redacted display value, with a regression test for a nested secret-bearing overflow message.

## Reproduction Assessment

Yes. A prompt error with an outer wrapper such as `request failed` and a nested cause containing both a secret and `context length exceeded` would be selected from the raw candidate list at `src/agents/pi-embedded-runner/run.ts:1296` and logged at `src/agents/pi-embedded-runner/run.ts:1329`. I did not run tests because this was a read-only sweep review.

## Solution Assessment

No. The implementation direction is good, but this revision is not the safest fix because it mixes raw classifier candidates with the diagnostic value that gets logged; the safer path is to inspect raw segments only for matching and then surface a redacted display string.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep raw chain messages out of overflow diagnostics:** `src/agents/pi-embedded-runner/run.ts:1296`
  - body: `collectErrorChainMessages(promptError, { redact: false })` can supply the selected `errorText`, which is logged below as `error=${errorText.slice(0, 200)}`. If a nested provider error contains an Authorization header, API key, or serialized request metadata plus an overflow hint, this bypasses the existing `formatErrorMessage()` redaction path and leaks the secret to diagnostics. Use raw chain messages only for matching, then log a redacted value.
  - confidence: 0.92

## Security Review

Status: needs_attention

Summary: The PR still has a concrete secret-redaction regression in the embedded-runner overflow diagnostic path.

Concerns:

- **[high] Raw nested error messages can leak to logs:** `src/agents/pi-embedded-runner/run.ts:1296`
  - body: The embedded runner opts into raw chain messages for prompt-error overflow detection, then propagates the matching raw segment into `contextOverflowError.text` and logs it. Provider and transport errors can include bearer tokens, API keys, or serialized request metadata in nested `.cause` or `.error.message` fields, so raw values should stay classifier-only or be redacted before surfacing.
