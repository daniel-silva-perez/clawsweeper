## Summary

Close as duplicate_or_superseded. The earlier open PR #73717 already tracks the same #73715 CLI bug with a narrower catalog-backed canonicalization path, while this PR’s unconditional lowercasing would regress documented case-sensitive model refs such as MiniMax.

## What This Changes

This PR lowercases CLI `--model` overrides in `resolveModelRefOverride` and `runModelRun`, then adds capability CLI tests asserting mixed-case model refs are passed to local dispatch in lowercase.

## Best Possible Solution

Use PR #73717, or an equivalent narrow fix, to canonicalize only unique case-insensitive catalog matches before local and gateway `infer model run` dispatch while preserving unmatched custom refs, case-sensitive provider model ids, and auth-profile suffix casing.

## Reproduction Assessment

Yes. There is a high-confidence source-level reproduction: current main forwards the raw mixed-case model override, while this PR lowercases it unconditionally, and the MiniMax docs/plugin constants prove mixed-case refs are valid and case-sensitive.

## Solution Assessment

No. Whole-string lowercasing is not the best fix because it solves the Anthropic case-only example by breaking documented mixed-case model ids; catalog-backed canonicalization with no-match preservation is the safer boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve case-sensitive model refs:** `src/cli/capability-cli.ts:643`
  - body: The new unconditional lowercasing changes every CLI override before dispatch. Please canonicalize only matched catalog entries and leave unmatched custom/profile-suffixed refs untouched.
