## Summary

Keep this open. Current main still has a plausible TUI shutdown bug: the TUI handles SIGTERM but not SIGHUP, the signal path only finishes after terminal drain/stop resolves successfully, and active status loader timers can outlive `tui.stop()` and keep the process alive.

## What This Changes

The issue requests that `openclaw tui --session <name>` exit when its terminal or gateway goes away, terminate cleanly on SIGTERM, and stop accumulating high-CPU/high-memory orphaned `openclaw-tui` processes.

## Best Possible Solution

Fix the TUI lifecycle itself: make shutdown idempotent, handle POSIX SIGHUP, always resolve/exit after bounded cleanup, stop loader/status timers, and tolerate closed terminal streams during drain/stop without changing gateway reconnect policy.

## Reproduction Assessment

Yes. The issue gives concrete macOS steps, and current main has matching source evidence: no TUI SIGHUP handler, a signal path that depends on successful terminal drain/stop, and active render/status intervals that are not cleaned up on exit.

## Solution Assessment

Yes. The narrow maintainable fix belongs in `src/tui/tui.ts` and focused shutdown tests, with dependency behavior accounted for; gateway restart logic and plugin loading should remain out of scope unless profiling later proves a second root cause.
