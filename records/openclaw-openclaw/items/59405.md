## Summary

Keep open. Current main shows Telegram now routes inbound messages through the shared channel turn and the agent runner has generic guards to preserve real tool results before synthesizing missing results, but I did not find high-confidence proof that the original Telegram plus `openai/gpt-5.1-codex` API-key path is fixed. The only discussion fix changes the provider/model to `openai-codex/gpt-5.4` via `/model`, which is a useful workaround but not evidence that the reported route was repaired.

## What This Changes

Ensure Telegram sessions deliver exec, grep, find, cat, and similar tool outputs back into agent context for the configured OpenAI Codex model the same way TUI sessions do.

## Best Possible Solution

Add or verify a regression path that drives a Telegram inbound turn for the configured agent/model, executes an exec/read-style tool, and proves the real tool result is present in the model continuation. If `openai/gpt-5.1-codex` is no longer a supported route for this behavior, make that explicit in model selection or docs and guide users to the supported `openai-codex/gpt-5.4` route.

## Reproduction Assessment

Yes, for a maintainer or reporter with Telegram/OpenAI credentials: the issue gives a concrete bound-bot, workspace-file, and prompt flow, plus a TUI control case. This review did not live-run that flow on current main, so current-main reproduction remains unverified.

## Solution Assessment

Unclear. Switching to `openai-codex/gpt-5.4` via `/model` is a useful workaround, but the narrow maintainable fix should either preserve real tool results for the reported Telegram route or clearly reject/document the unsupported provider/model combination.
