## Summary

Keep this PR open. Current main still has the reported Mattermost behavior: repeated `getBotUpdateAt`/REST health-check failures are logged and polling continues, but the WebSocket is not terminated, so `runWithReconnect` has no close/resolve signal to reconnect. The PR is a focused implementation candidate, not something already shipped on main. The provided review discussion is also useful: the counter-reset issue Greptile found appears addressed in the PR head patch, but the default-parameter/type-contract concern and missing regression tests still need maintainer review.

## Best Possible Solution

Keep this PR open as an active Mattermost bug-fix candidate. The best maintainer path is to review and land a polished version that removes the unsafe default parameter, adds focused tests for consecutive failures/reset/opt-out, and keeps the change scoped to `extensions/mattermost/src/mattermost/monitor-websocket.ts` unless broader Mattermost transport-liveness work is chosen separately.
