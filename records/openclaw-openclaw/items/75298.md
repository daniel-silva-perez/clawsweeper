## Summary

Keep this open. Current main has route-table idempotency and several recent responsiveness/performance mitigations, but I did not find source, tests, docs, or release evidence proving that repeated webhooks plugin `register(api)` entry under event-loop saturation is fixed or expected.

## What This Changes

The issue asks whether the webhooks plugin should re-enter `register()` during one gateway lifecycle under event-loop saturation and whether a guard should prevent repeated route registration.

## Best Possible Solution

Trace plugin activation and registry reloads during the reported saturation windows, then add a regression that a configured webhooks route registers once per explicit gateway/plugin lifecycle unless a real config write, gateway restart, or intentional plugin reload occurs.

## Reproduction Assessment

Unclear: the report has detailed timestamped operational evidence, but there is not yet a deterministic current-main command or test fixture that reproduces repeated webhooks `register()` calls under event-loop saturation.

## Solution Assessment

No: current main has useful mitigations and route idempotency, but closing would overfit adjacent fixes. The maintainable solution is to identify the re-entry caller and cover that lifecycle with a focused regression.
