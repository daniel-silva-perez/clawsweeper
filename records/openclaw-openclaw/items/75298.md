## Summary

Keep this open. Current main contains strong fixes for older plugin registry churn and duplicate HTTP-route replacement, but I did not find source, test, or release evidence proving that the fresh webhooks re-registration logs under event-loop saturation are fixed or expected.

## What This Changes

The issue reports repeated webhooks plugin route registration under event-loop saturation during cron catchup, Control UI polling, and memory-core dreaming.

## Best Possible Solution

Trace the actual plugin-load or reload caller during the reported saturation windows, then add a regression test that Control UI polling, cron catchup, and memory-core dreaming do not re-run the webhooks plugin registration unless an explicit gateway/plugin config reload occurs.
