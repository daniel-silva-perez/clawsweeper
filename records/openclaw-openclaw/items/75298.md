## Summary

Keep open: current main has adjacent fixes for stale HTTP-route registries and config-schema registry churn, but it does not prove this new webhooks report is resolved. The bundled webhooks plugin still performs route binding and emits the `registered route` log every time its `register(api)` path runs, while the report is from newer versions after the earlier registry fixes shipped.

## What This Changes

The issue reports repeated webhooks plugin route registration under event-loop saturation during cron catchup, Control UI polling, and memory-core dreaming.

## Best Possible Solution

Keep the issue open and make the webhooks route lifecycle explicitly full-runtime/idempotent: repeated discovery, status, schema, model-catalog, or Control UI reads should not emit new `registered route` logs or rebind active webhook routes, while normal gateway startup must still install the configured routes once.

## Reproduction Assessment

Unclear. The issue provides detailed operational logs from recent released versions, and static source inspection shows why any repeated `register(api)` invocation would produce the observed log, but there is not yet a minimized current-main command or test for the event-loop saturation trigger.

## Solution Assessment

Yes, the narrowest maintainable fix is likely in the webhooks plugin lifecycle guard and regression coverage, with generic plugin-runtime changes only if the failing test shows a shared registry/load contract bug.
