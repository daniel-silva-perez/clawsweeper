## Summary

Keep this open. Current main still has the older route-idempotency and registry-reuse fixes, but I found no current source, test, release, or history evidence that the newly reported repeated webhooks `register()` calls during event-loop saturation are fixed or expected.

## What This Changes

The issue reports repeated webhooks plugin route registration during cron catchup, Control UI polling, and memory-core dreaming when the Gateway event loop is saturated.

## Best Possible Solution

Trace plugin-load and registry-activation callers during liveness-warning windows, then prevent the webhooks plugin `register()` path from running again unless an explicit Gateway/plugin config reload or plugin reload occurs. Keep same-plugin route replacement idempotent and add regression coverage for Control UI polling, cron catchup, and memory-core dreaming not re-entering webhooks registration.

## Reproduction Assessment

Unclear: the report provides strong timestamped operational evidence and code inspection confirms the log source, but there is no deterministic current-main command or test fixture that reproduces the repeated registration calls.

## Solution Assessment

No: closing as already implemented or adding only a route-level guard would miss the lifecycle problem, because route replacement is already idempotent. The maintainable fix is to identify and guard the registry/load caller that re-runs plugin registration under saturation.
