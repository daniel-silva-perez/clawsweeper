## Summary

Keep open. Current main still lacks this PR's intended opt-in `OPENCLAW_INSTALL_BREW` Docker build/setup support, and brew-kind skills can still hit the `brew not installed` path when no brew executable resolves. The PR remains relevant to #14593, but it is not merge-ready as-is because it targets the old root setup script shape and has functional plus supply-chain issues that need maintainer follow-up.

## Best Possible Solution

Keep this PR open for maintainer follow-up, but do not merge it as-is. The best path is to choose the canonical fix for #14593: either a safe opt-in Linuxbrew Docker build/setup flow in current `Dockerfile` and `scripts/docker/setup.sh`, with conditional runtime environment handling, docs, regression coverage, and installer integrity policy; or a different shipped product path such as hiding brew-only installs in Linux containers or providing non-brew Linux installers. After one path lands on main, close or supersede the remaining PRs against that shipped implementation.

## Risks / Open Questions

- Merging this PR as-is would add unpinned remote installer execution during Docker builds, so maintainers should decide on pinning, checksum, trusted-source, or no-auto-installer policy first.
- The PR is stale against current main because Docker setup logic moved to `scripts/docker/setup.sh`; applying the old root `docker-setup.sh` edits would not correctly wire the current setup path.
- The Dockerfile changes described in the PR discussion have known functional blockers: unconditional Homebrew ENV/PATH pollution and fragile installer quoting.
- Closing this PR now would drop a relevant implementation candidate while #14593 and alternative fix PRs #55940 and #57553 remain open and unshipped.
