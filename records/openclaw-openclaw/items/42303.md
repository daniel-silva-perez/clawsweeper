## Summary

Keep this PR open. Current main still does not restore opt-in Linuxbrew support for Docker builds, and the linked Docker skill-install failure remains possible. The PR is a relevant implementation candidate, but it should not merge as-is because its Dockerfile path has functional and supply-chain concerns already identified in review.

## Best Possible Solution

Keep this PR open for maintainer follow-up, but do not merge it as-is. Choose a canonical fix for #14593: either revise this Docker build approach with safe opt-in Linuxbrew support, conditional runtime environment handling, docs, and a clear remote-installer integrity policy, or land a different product path such as Linux/container fallback behavior or hiding brew-only installers in containers. Once a canonical fix merges, close or supersede the remaining implementation PRs against that shipped path.

## Risks / Open Questions

- Merging this PR as-is would add an unpinned remote installer execution path during Docker builds, which needs explicit maintainer policy around trust, pinning, and integrity.
- The current PR's Dockerfile changes have concrete functional hazards already called out in review: non-brew images would get stale Homebrew environment variables, and the installer command is likely to fail because of shell quoting.
- The PR also adds an OPENCLAW_SKIP_BUILD recovery message that points users at an openclaw build command that the review context says does not exist.
