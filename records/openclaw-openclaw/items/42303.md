## Summary

Keep open. Current main still does not implement this PR's intended opt-in Linuxbrew Docker support, and the linked Docker skill-install failure remains current. The PR is a relevant implementation candidate, but it is dirty against main and has functional plus supply-chain review blockers that need maintainer follow-up before any merge.

## Best Possible Solution

Keep this PR open for maintainer follow-up, but do not merge it as-is. Choose a canonical fix for #14593: either a safe opt-in Linuxbrew Docker build/setup flow with conditional runtime environment handling, docs, regression coverage, and remote-installer integrity policy, or a different product path such as Linux-native installer metadata or Docker onboarding behavior that avoids brew-only installs when brew is unavailable. After that lands, close or supersede the remaining implementation PRs against the shipped path.

## Risks / Open Questions

- Merging this PR as-is would add an unpinned remote installer execution path during Docker builds; maintainers need an explicit trust, pinning, and integrity policy for that path.
- The Dockerfile diff has concrete functional hazards already identified in review: unconditional Homebrew environment pollution for non-brew images and fragile shell quoting around the downloaded installer.
- The PR is dirty against current main and targets an older Dockerfile shape, so any usable version needs a rebase onto the current multi-stage Dockerfile plus fresh Docker validation.
- Closing this PR now would discard a relevant implementation candidate before maintainers choose the canonical fix for the still-open Docker skill-install bug.
