## Summary

Keep open. Current main does not implement the requested TUI `/upload` command, and this PR is useful source work, but the latest branch is not merge-ready because it still has concrete pending-context and filesystem-read bugs in the proposed upload flow.

## What This Changes

The PR adds a TUI `/upload` slash command, a file-browser select-list component with theme/tests, and command-handler logic that reads selected local files and prepends their contents to the next chat message.

## Best Possible Solution

Implement TUI file context through bounded regular-file selection, preferably by extending `TuiBackend` to pass `chat.send` attachments so existing gateway validation/offload/limits apply; add targeted tests for send failure, empty or fully skipped replacement uploads, special-file filtering, size limits, and TUI docs.

## Reproduction Assessment

Yes. The remaining issues have high-confidence source-level reproductions: force `client.sendChat` to reject after selecting files to see queued context dropped, or attach one file and then confirm only skipped/unreadable files to see stale context remain queued.

## Solution Assessment

No. The branch is a reasonable feature direction, but as written it bypasses the existing attachment pipeline, reads selected local paths synchronously without regular-file or size validation, and mishandles pending context lifecycle on failure paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Retain pending upload context until send succeeds:** `src/tui/tui-command-handlers.ts:572`
  - body: `pendingFileContext` is cleared before `client.sendChat` runs. If the gateway/provider request rejects, the catch path only logs the failure, so retrying sends the typed text without the files the user selected. Clear the pending context after a successful send or restore it in the catch path.
  - confidence: 0.91
- **[P1] Clear stale upload context when no files attach:** `src/tui/tui-command-handlers.ts:275`
  - body: When a replacement `/upload` confirms only skipped binary or unreadable files, `blocks.length` stays zero and the old `pendingFileContext` is left intact. The next message can silently include a previous attachment the user was trying to replace. Clear pending context before processing the new selection or in an explicit empty-result branch.
  - confidence: 0.88
- **[P1] Validate selected paths before synchronous reads:** `src/tui/components/file-browser-select-list.ts:75-78`
  - body: The browser classifies every non-directory entry as a file and the handler then calls `readFileSync` on it. Selecting a FIFO, device node, or very large file can hang or exhaust memory in the TUI. Only regular files within a bounded size should be selectable/read.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The PR introduces local-file prompt ingestion and still has concrete privacy/availability concerns around stale queued file content and unbounded special-file reads.

Concerns:

- **[medium] Stale file context can leak into a later message:** `src/tui/tui-command-handlers.ts:275`
  - body: A failed replacement upload that attaches no readable files leaves the previous pending file context queued, so a later message can include old local file contents unexpectedly.
  - confidence: 0.88
- **[medium] Special files can hang or exhaust the TUI:** `src/tui/components/file-browser-select-list.ts:75`
  - body: Because non-directory filesystem entries are selectable and then read synchronously, device nodes, FIFOs, or very large files can block the process or consume unbounded memory.
  - confidence: 0.86
