## Summary

Keep open. Current main still lacks the requested TUI `/upload` feature, and the PR is useful, but the proposed patch still has merge-blocking context-lifecycle and local file-read safety bugs after the last review-fix commit.

## What This Changes

The PR adds a TUI `/upload` command, a file-browser select-list component, theme wiring, and tests so selected local file contents are queued as context for the next TUI message.

## Best Possible Solution

Keep the PR open and repair the branch before merge: use regular-file and size validation before reading, preserve queued upload context across failed sends, clear stale context when replacement uploads attach nothing, and add the required docs/changelog coverage without changing CLI mode or protocol contracts.

## Reproduction Assessment

Yes. The remaining bugs have high-confidence code-path reproductions: failed `sendChat` after `/upload` drops attachments, confirming only skipped/unreadable files leaves previous context queued, and selecting a huge or special file reaches `readFileSync` before validation.

## Solution Assessment

No. The feature direction fits the TUI, but this implementation is not the best merge path until file reads are bounded and the pending-context lifecycle is corrected; no message-protocol change appears necessary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Validate files before reading them:** `src/tui/tui-command-handlers.ts:257`
  - body: `readFileSync(filePath)` runs before any regular-file check or size cap. Because the browser can surface non-directory entries and users can select very large files, this can block the TUI or exhaust memory before binary detection gets a chance to skip anything.
  - confidence: 0.9
- **[P2] Move queued-context clearing after send succeeds:** `src/tui/tui-command-handlers.ts:572`
  - body: `pendingFileContext` is nulled before `client.sendChat` resolves. If the gateway/provider send rejects, the selected files are lost and retrying the same message sends only the typed text instead of the queued context.
  - confidence: 0.86
- **[P2] Clear stale context when an upload attaches no files:** `src/tui/tui-command-handlers.ts:278-282`
  - body: If a user has a previous upload queued and then confirms a replacement upload where every selected path is skipped or fails to read, this branch leaves the old `pendingFileContext` intact. The next message can silently send stale file contents.
  - confidence: 0.84
- **[P3] Add the required changelog entry:** `src/tui/commands.ts:117`
  - body: This is a user-facing `feat` that adds a new slash command, but the PR does not update `CHANGELOG.md`. OpenClaw policy requires a changelog entry for user-visible feature work, credited to the contributor.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The patch introduces a local file-read surface that needs bounded regular-file validation before merge.

Concerns:

- **[medium] Unbounded local file read:** `src/tui/tui-command-handlers.ts:257`
  - body: The upload handler reads each selected path synchronously before checking file type or size. A selected special file, FIFO, or very large file can block or exhaust the TUI process, so the handler should stat/open only regular files and enforce a size cap before full reads.
  - confidence: 0.87
