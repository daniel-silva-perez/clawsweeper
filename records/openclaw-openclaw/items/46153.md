## Summary

Keep open. Current main still generates the macOS Gateway LaunchAgent without `StandardInPath=/dev/null` or an equivalent stdin-closing wrapper, and both fresh install and restart-repair paths reuse that same plist builder. Related open issue #58414 overlaps on the missing-stdin theory, but it is broader and does not supersede this focused launchd startup hang.

## Best Possible Solution

Add a macOS LaunchAgent stdin mitigation in the shared launchd plist path, preferably `StandardInPath` set to `/dev/null`, and ensure fresh install plus restart/doctor repair rewrites produce it. Add focused launchd tests, update the user-facing changelog/docs if behavior changes, and verify with a real macOS LaunchAgent smoke where feasible.
