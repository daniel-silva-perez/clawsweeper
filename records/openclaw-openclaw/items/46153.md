## Summary

Keep open. Current main still generates the macOS Gateway LaunchAgent without `StandardInPath=/dev/null`, and both fresh install and restart-repair plist rewrites flow through the same renderer. The related open #58414 overlaps on the stdin theory, but it is broader and not a clear canonical replacement for this focused launchd startup hang.

## What This Changes

The issue asks OpenClaw to stop the macOS Gateway from hanging silently when started by launchd, likely by closing or redirecting stdin for generated LaunchAgent jobs.

## Best Possible Solution

Add `StandardInPath=/dev/null` to the central macOS Gateway LaunchAgent plist generation, or an exactly equivalent stdin-closing launch wrapper, so fresh installs and restart-repair rewrites both repair the service definition. Cover the generated plist and restart-rewrite path with focused tests, then add a user-facing changelog entry and verify on a real macOS LaunchAgent when available.

## Reproduction Assessment

Yes. The issue provides concrete macOS launchd reproduction steps, and this review statically confirmed current main still emits the LaunchAgent plist without the requested stdin binding; live launchd execution was not performed in this read-only pass.

## Solution Assessment

Yes. The narrowest maintainable fix is in the shared LaunchAgent plist builder because fresh install, macOS app install delegation, and restart repair all flow through that service definition; a docs-only workaround or one-off wrapper would leave regenerated plists broken.
