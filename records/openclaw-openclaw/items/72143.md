## Summary

Keep open. Current main still only partially addresses the report: successful code-0 direct fallback content is prefix-free, but denied and other non-success resume-failure fallbacks still expose the internal prefix, and the session resume path still performs a single `agent` gateway call with no retry/backoff before direct delivery.

## What This Changes

The issue asks for async exec approval follow-up delivery to retry session resume with short backoff, remove the internal fallback prefix from user-facing direct delivery, log sanitized resume failures internally, and preserve the command result/status content.

## Best Possible Solution

Land a small agents-focused fix: add bounded retry/backoff around the `agent` follow-up call, remove the resume-failure prefix from every direct fallback delivery path, and emit sanitized internal logging only after resume attempts are exhausted.

## Reproduction Assessment

Yes. A high-confidence unit-level reproduction exists by making `callGatewayTool` reject in the denied resume-failure path; the current test still expects the internal prefix in the Telegram message, and source inspection shows one resume attempt with no retry/backoff.

## Solution Assessment

Yes. The requested direction is the narrow maintainable fix: retry session resume briefly, silently direct-deliver sanitized result content only after retries fail, and log internal resume metadata without leaking raw command output or secrets.
