## Summary

Keep open. Current main still only suppresses the internal resume-failure prefix for successful `code 0` fallback content; denied and other non-success direct fallbacks can still expose it, and the session resume path still makes one `agent` gateway call with no retry/backoff before falling back. The narrow related PR was closed unmerged, and the broad open PR may overlap but has not landed on main for this behavior. https://github.com/openclaw/openclaw/issues/72143

## What This Changes

Requests bounded retry/backoff for exec approval follow-up session resume, prefix-free direct fallback delivery, and sanitized internal logging of resume failures.

## Best Possible Solution

Land a narrow agents change that wraps the follow-up `agent` gateway call in bounded retry/backoff, removes the resume-failure prefix from every direct fallback delivery path, and logs sanitized resume failure metadata only after retries are exhausted.

## Reproduction Assessment

Yes. A high-confidence unit reproduction exists by mocking `callGatewayTool` to reject in the denied approval follow-up path with a deliverable Telegram target; the current test still expects the internal prefix in the outbound message.

## Solution Assessment

Yes. The requested direction is the narrow maintainable fix: retry session resume briefly, then direct-deliver only sanitized result content and keep resume diagnostics internal. A broader delivery-policy rewrite is not required for this report.
