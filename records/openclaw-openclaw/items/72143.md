## Summary

Keep open. Current main only partially addresses the report: successful `code 0` resume-fallback deliveries no longer get the internal prefix, but denied/non-success fallback deliveries still do, session resume is still attempted only once with no backoff, and resume failures are not logged inside the resume-fallback path with the requested metadata.

## Best Possible Solution

Keep this issue open and land a narrow exec approval follow-up fix: retry the `agent` session follow-up call with short bounded backoff, silently direct-deliver sanitized fallback content without the internal resume-failure prefix for all result kinds, and log the exhausted resume failure internally with safe metadata. The existing partial success-only prefix suppression on current main should be preserved.
