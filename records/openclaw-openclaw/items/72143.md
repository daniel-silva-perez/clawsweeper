## Summary

Keep open. Current main only partially addresses the report: successful code-0 direct fallback deliveries no longer show the internal prefix, but denied and non-success fallback deliveries still do, session resume is attempted only once, and successful direct fallback after a resume failure is not logged with the requested diagnostic metadata.

## What This Changes

Request retry/backoff for exec approval follow-up session resume, removal of the internal fallback prefix from user-facing direct delivery, internal resume-failure logging, and preservation of command result content.

## Best Possible Solution

Land a small agents fix in `src/agents/bash-tools.exec-approval-followup.ts`: add bounded retry/backoff around the `agent` resume call, remove the resume-failure prefix from all direct fallback content, and emit a sanitized internal warning only after resume attempts are exhausted. Cover successful, denied, non-zero, retry-success, and retry-exhausted fallback cases in the colocated test, while keeping broader delivery-policy changes separate.

## Reproduction Assessment

Yes. A high-confidence unit-level reproduction exists: make `callGatewayTool` reject in the denied resume-failure path and the current test expects Telegram content to include the internal prefix. Source inspection also shows only one resume attempt and no retry/backoff code.

## Solution Assessment

No. Current main’s success-only suppression is not the best way to solve the broader issue because it leaves denied/non-success fallback messages prefixed and does not add retry/backoff or internal logging. A focused repair is safer than relying on the broad #68129 delivery refactor for this narrow behavior.

In `src/agents/bash-tools.exec-approval-followup.ts`, add a short bounded retry/backoff around the `callGatewayTool("agent", ...)` session-resume call, such as immediate/500ms/1500ms/3000ms or a similarly small schedule that tests can control.
