## Summary

Keep this issue open. Current main still leaves the central source-dependent drift in place: `sessions_send` injects requester/target A2A metadata through `extraSystemPrompt`, that text remains part of the system prompt sent as OpenAI Codex Responses `instructions`, and owner-only tools are still pruned from the exposed tool list for non-owner or unknown senders. The direct inbound metadata path and cache observability have improved, but they do not close this report.

## Best Possible Solution

Keep this issue open. Move `sessions_send` requester/target A2A metadata out of `extraSystemPrompt` and into the triggering user turn, provenance, or another non-prefix-critical structure. Then add a regression test that compares same-session OpenAI Codex Responses payloads with the same `prompt_cache_key` across direct and inter-session turns. Separately make an explicit product/security decision for owner-only tools: expose a stable schema with execution-time authorization denial, or document and test a deliberate cache-impact exception for source-based tool filtering.

## Risks / Open Questions

- Changing owner-only tool exposure is security-sensitive. A stable schema approach must still deny execution for non-owner and unknown senders, especially for `cron`, `gateway`, and `nodes`.
- Moving A2A metadata out of `extraSystemPrompt` must preserve requester/target provenance somewhere message-scoped or otherwise non-prefix-critical so inter-session behavior and auditability do not regress.
- Prompt-cache observability can surface drift, but it is not a substitute for a regression test that compares same-session OpenAI Codex Responses payloads across direct and inter-session turns.
