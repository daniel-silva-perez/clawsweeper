## Summary

Current main still leaves the central bug open. Direct inbound metadata is more cache-conscious than the original report described and prompt-cache observability now exists, but inter-session `sessions_send` still injects requester/target-specific A2A text through `extraSystemPrompt`, OpenAI Codex Responses still sends the stripped system prompt as top-level `instructions`, and owner-only tool filtering can still remove tools such as `cron` from the exposed schema for non-owner or unknown senders.

## What This Changes

The issue requests stable same-session OpenAI/Codex provider payload prefixes by moving source-dependent metadata out of `instructions`, preventing source-conditional tool schema drift, and adding guardrails for prompt-cache hygiene.

## Best Possible Solution

Keep this issue open. Move `sessions_send` requester/target A2A metadata out of `extraSystemPrompt` and into message-level provenance or another non-prefix-critical structure, then add a regression test that compares same-session OpenAI Codex Responses payload shape across direct and inter-session turns with the same `prompt_cache_key`. For owner-only tools, make an explicit security/product decision: either expose a stable schema with execution-time authorization denial, or document and test source-based tool filtering as a deliberate cache-impact exception.
