## Summary

Current main still leaves the central bug open. The direct inbound metadata path has cache-stability improvements and prompt-cache observability exists, but `sessions_send` still injects requester/target-specific A2A text through `extraSystemPrompt`, OpenAI Codex Responses still sends the stripped system prompt as `instructions`, and owner-only tools are still removed from the exposed tool set for non-owner or unknown senders.

## Best Possible Solution

Keep this issue open. Move `sessions_send` requester/target A2A metadata out of `extraSystemPrompt` and into the triggering user turn/provenance or another non-prefix-critical structure, then add a regression test comparing same-session OpenAI Codex Responses payloads across direct and inter-session turns with the same `prompt_cache_key`. Separately make an explicit product/security decision for owner-only tools: either expose a stable schema with execution-time authorization denial, or document and test a deliberate cache-impact exception for source-based tool filtering.

## Risks / Open Questions

- A stable exposed schema for owner-only tools is security-sensitive; any implementation must still deny execution for non-owner and unknown senders, especially for `cron`, `gateway`, and `nodes`.
- Moving A2A context out of `extraSystemPrompt` must preserve requester/target provenance in a message-level or provenance structure so inter-session behavior does not regress.
- Current prompt-cache observability can explain drops after the fact, but it does not prevent same-session instruction/tool drift before the OpenAI request is built.
