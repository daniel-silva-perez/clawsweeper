## Summary

Keep open. Current main still lacks a skill-specific runtime lifecycle hook or documented skill invocation event, and the enterprise permission-check comment makes this a security-sensitive plugin API design item rather than stale cleanup or ClawHub-only work.

## What This Changes

The issue requests before/after skill execution lifecycle hooks or equivalent skill invocation/success/failure events so plugins or scripts can log, analyze, monitor, or block skill execution.

## Best Possible Solution

Add an additive, documented plugin hook surface for explicit skill invocation first: a pre-invocation decision hook that can allow, block, or require approval with a reason, plus a post-invocation observation hook carrying bounded redacted metadata such as skill name, source command, agent/session/run context, dispatch mode, outcome, and failure summary. Treat model-selected skills separately unless the runtime can identify a concrete skill execution boundary.

## Reproduction Assessment

Not applicable as a runtime bug reproduction; this is a feature/API gap. Static inspection of the current hook catalog, hook runner, public hooks docs, inline skill command runtime, skill metadata, and focused tests gives high confidence that the requested boundary is absent.

## Solution Assessment

Unclear for the exact proposed `skill:beforeExecute`/`skill:afterExecute` script-hook shape. A typed plugin API with explicit blocking, approval, timeout, and redaction semantics is the safer maintainable direction, especially given the enterprise permission-check use case in the issue discussion.
