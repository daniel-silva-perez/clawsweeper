## Summary

Keep open. Current main still lacks a skill-specific execution lifecycle hook in the typed plugin hook catalog, internal hook event model, skill metadata, and explicit slash-skill invocation path; the discussion also adds an enterprise permission-check use case, so this is a security-sensitive plugin API design item rather than stale cleanup or ClawHub-only work.

## What This Changes

Request to add before/after skill execution hooks or equivalent skill invocation events so plugins or scripts can log, analyze, monitor, and enforce policy around skill runs.

## Best Possible Solution

Add an additive, documented typed plugin hook surface around explicit skill invocation: a pre-invocation decision hook that can allow, block, or require approval with a bounded reason, plus a post-invocation observation hook carrying redacted metadata such as skill name, invocation source, agent/session/run context, dispatch mode, outcome, duration, and failure summary. Keep script-style internal hooks secondary unless maintainers decide they need parity after the plugin contract is secure.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a runtime bug; this is a feature/API gap. Static inspection of current hook catalogs, docs, skill metadata, skill command dispatch, and related tests gives high confidence that the requested skill execution boundary is absent.

## Solution Assessment

Is this the best way to solve the issue? Unclear for the exact proposed JSON script-hook shape. A typed plugin API with explicit blocking, approval, timeout, ordering, and redaction semantics is the safer maintainable direction, especially for the enterprise permission-check use case raised in discussion.
