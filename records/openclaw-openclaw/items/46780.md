## Summary

Keep open. Current main still has tool, agent, message, session, gateway, install, and dispatch hooks, but it does not expose a skill execution lifecycle boundary for before/after invocation or success/failure observation. The permission-check use case raised in discussion makes this a plugin API/security design item rather than stale cleanup or ClawHub-only work.

## What This Changes

The issue requests before/after skill execution lifecycle hooks, or equivalent skill execution events/logging, so plugins can track, analyze, monitor, or gate skill runs.

## Best Possible Solution

Add an additive, documented typed plugin hook surface around explicit skill invocation: a pre-invocation decision hook that can allow, block, or require approval with bounded reason metadata, plus a post-invocation observation hook carrying redacted skill name, invocation source, agent/session/run context, dispatch mode, outcome, duration, and failure summary. Internal script hooks or raw execution logs should stay secondary unless maintainers decide they need parity after the plugin contract is secure.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a runtime bug; this is a feature/API gap. Static inspection of current hook catalogs, hook runners, internal hook types, skill metadata, docs, and explicit slash-skill dispatch gives high confidence that the requested skill execution lifecycle boundary is absent.

## Solution Assessment

Is this the best way to solve the issue? Unclear for the exact proposed JSON script-hook shape. A typed plugin API with explicit blocking, approval, timeout, ordering, and redaction semantics is the safer maintainable direction, especially for the enterprise permission-check use case raised in discussion.
