## Summary

Keep this open. Current main and v2026.4.27 still treat future-stamped configs as a warning for ordinary config load and snapshot reads, while hard refusal is limited to service/process mutation paths and `gateway --force`; the merged GPT-5.4 runtime hardening is adjacent but does not settle the downgrade/recovery policy reported here.

## What This Changes

The issue requests clearer handling when `openclaw.json` has a newer `meta.lastTouchedVersion` than the running binary because that was reported to silently drop Discord final replies for OpenAI-Codex/GPT-5.x models.

## Best Possible Solution

Define one supported future-config downgrade/recovery policy at the config and gateway boundary, apply it consistently to ordinary runtime startup as well as service/process mutations, document the recovery path, and add focused regression or manual proof for a future-stamped config with Discord/OpenAI-Codex final-message delivery.

## Reproduction Assessment

Unclear for the live Discord/OpenAI-Codex delivery symptom because no live gateway run was performed. There is a high-confidence source-level reproduction of the remaining boundary: a future-stamped config warns during normal load/snapshot reads and still returns a runtime config, while only service/force paths block.

## Solution Assessment

Unclear. The current action-scoped guard is a useful partial mitigation, but it is not a complete fix for the reported ordinary runtime failure mode; maintainers should choose the downgrade policy before changing load-time behavior.
