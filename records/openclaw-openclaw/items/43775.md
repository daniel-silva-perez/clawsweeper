## Summary

Keep this issue open. Current `main` still warns on a future `meta.lastTouchedVersion` and then continues normal runtime config materialization; the hard future-config refusal is scoped to service/process mutation paths, service-mode gateway startup, and `gateway --force`, not ordinary interactive gateway delivery. The related auto-downstamp PR was closed unmerged, and the later merged GPT/OpenAI-Codex hardening is adjacent but does not close this config-policy gap.

## What This Changes

Future-stamped `openclaw.json` configs may silently break Discord delivery for OpenAI-Codex/GPT-5 final messages, and the issue asks for refusal, repair/downstamp, or stronger startup warning/remediation.

## Best Possible Solution

Choose and implement a scoped future-config runtime policy for ordinary gateway starts that can deliver channel messages from an older binary, document the supported downgrade/recovery behavior, and add focused proof for future-stamped config plus Discord/OpenAI-Codex final-message delivery.
