## Summary

Keep this open. Current `main` still lets normal config loading continue after the newer-version warning, while the hard refusal is scoped to service/process mutations, service-mode gateway startup, and `gateway --force`; I did not find source, tests, or docs proving that ordinary Discord/OpenAI-Codex delivery is refused, repaired, downstamped, or otherwise made safe.

## What This Changes

The issue asks OpenClaw to make a future `meta.lastTouchedVersion` mismatch actionable because it can silently break Discord final-message delivery for `openai-codex-responses` models.

## Best Possible Solution

Add a scoped, documented future-config guard or repair path for gateway runtime starts that can deliver channel messages from an older binary, while keeping safe read-only config inspection available and covering the Discord/OpenAI-Codex final-message scenario with regression or manual proof.
