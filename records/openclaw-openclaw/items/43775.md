## Summary

Keep this open. Current main still treats a future `meta.lastTouchedVersion` as a warning on normal config load, while the hard refusal is scoped to service/process mutation paths, service-mode gateway startup, and `gateway --force`; the closed auto-downstamp PR did not merge, and the later GPT/OpenAI-Codex hardening is adjacent rather than a fix for the config policy gap.

## What This Changes

The issue asks OpenClaw to make a newer config version stamp fail loudly, repair/downstamp, or warn prominently because it can silently break Discord delivery for `openai-codex-responses` models.

## Best Possible Solution

Define one supported downgrade/recovery policy for ordinary gateway startup and channel delivery, implement it at the config/gateway boundary rather than in a provider-specific path, document the behavior, and add focused proof for future-stamped config plus Discord/OpenAI-Codex final-message delivery.
