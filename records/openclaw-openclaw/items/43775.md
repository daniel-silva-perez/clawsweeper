## Summary

Keep this open. Current main and the latest release still only partially handle future-stamped configs: ordinary config load and snapshot read warn, then continue with a runtime config, while hard refusal is limited to service/process mutations and `gateway --force`; the merged GPT-5 runtime hardening is adjacent but does not close the config downgrade policy gap.

## What This Changes

The issue reports that a future `meta.lastTouchedVersion` in `openclaw.json` can silently break Discord delivery for OpenAI-Codex GPT-5.x models and asks for a stronger downgrade or recovery policy.

## Best Possible Solution

Define one supported future-config downgrade/recovery policy at the config and gateway boundary, apply it consistently to ordinary runtime startup as well as service/process mutations, document the recovery path, and add focused regression or manual proof for a future-stamped config with Discord/OpenAI-Codex final-message delivery.

## Reproduction Assessment

Unclear for the live Discord/OpenAI-Codex delivery symptom because this review did not run a Discord gateway with GPT-5.x. There is a high-confidence source-level reproduction of the remaining boundary: a future-stamped config warns during ordinary load/snapshot read and still returns a runtime config, while only action-scoped callers block.

## Solution Assessment

Unclear. The current service/force guard is a useful partial mitigation, and the closed auto-downstamp PR is a plausible direction, but maintainers should choose the downgrade policy before changing ordinary runtime behavior.
