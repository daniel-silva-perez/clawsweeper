## Summary

Keep this open. Current main still has only partial future-config handling: service-mode startup, service mutations, and `gateway run --force` are guarded, but ordinary config load still logs the newer-version warning and returns a runtime config. The closed auto-downstamp PR directly targeted this report but did not merge, while the merged GPT-5.4 hardening is adjacent to the delivery symptom rather than a complete fix for the future-stamped config boundary.

## What This Changes

The issue asks OpenClaw to prevent a future-stamped `meta.lastTouchedVersion` from silently breaking Discord delivery for `openai-codex-responses` models, by hard failure, repair/downstamp, or stronger startup diagnostics.

## Best Possible Solution

Define one supported downgrade/recovery policy at the config and gateway boundary, document it, and add focused regression or manual proof for a future-stamped config plus Discord/OpenAI-Codex final-message delivery.

## Reproduction Assessment

Unclear for the original Discord/OpenAI-Codex delivery symptom. The issue gives a concrete config diff and model matrix, and source inspection gives a deterministic way to verify future-config guard behavior, but this read-only review did not run live Discord/OpenAI delivery.

## Solution Assessment

Unclear. The current service/force guard is a useful partial mitigation, but it is not yet the narrow complete solution for ordinary config load and runtime delivery; maintainers should choose the intended downgrade policy before a fix PR changes behavior.
