## Summary

Keep this open. Current main still only warns during ordinary config load when `meta.lastTouchedVersion` is newer, while the hard refusal is scoped to named service/process mutations such as service-mode startup and `gateway --force`; the linked auto-downstamp PR was closed unmerged, and the merged GPT-5.4 runtime hardening is adjacent but does not settle the future-config downgrade policy.

## What This Changes

Request to make a future `meta.lastTouchedVersion` config stamp fail clearly, repair/downstamp, or warn more prominently so OpenAI-Codex final Discord delivery cannot fail silently after downgrading OpenClaw.

## Best Possible Solution

Define one supported downgrade/recovery policy at the config and gateway boundary, document it, and add focused regression or manual proof for a future-stamped config plus Discord/OpenAI-Codex final-message delivery.
