## Summary

Keep this open. Current main still has only partial future-config handling: ordinary config load and snapshot read warn about a newer `meta.lastTouchedVersion`, then continue with a runtime config, while the hard refusal is limited to service/process mutations and `gateway --force`; the merged GPT-5 runtime hardening is adjacent but does not close this config downgrade boundary.

## What This Changes

The issue asks OpenClaw to make future-stamped config mismatches actionable so Discord delivery for OpenAI-Codex/GPT-5.x models does not silently fail when `meta.lastTouchedVersion` is newer than the running binary.

## Best Possible Solution

Define one supported downgrade/recovery policy at the config and gateway boundary, apply it consistently to ordinary runtime startup as well as service/process mutations, document the recovery path, and add focused regression or manual proof for a future-stamped config with Discord/OpenAI-Codex final-message delivery.

## Reproduction Assessment

Unclear for the live Discord/OpenAI-Codex delivery symptom because this review did not run a Discord gateway with GPT-5.x. There is a high-confidence source-level reproduction of the remaining boundary: a future-stamped config warns during ordinary load/snapshot read and still returns a runtime config, while only action-scoped callers block.

## Solution Assessment

Unclear. The current service/force guard is a useful partial mitigation, and the unmerged auto-downstamp PR is a plausible direction, but maintainers should choose the downgrade policy before changing ordinary runtime behavior.
