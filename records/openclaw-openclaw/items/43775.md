## Summary

Keep open. Current main still treats a future `meta.lastTouchedVersion` as a normal-load warning, while the hard refusal is scoped to service/destructive actions; the auto-downstamp PR was closed unmerged, and the later GPT/OpenAI-Codex hardening is adjacent rather than a config downgrade policy fix.

## What This Changes

The issue asks OpenClaw to make future-stamped config handling explicit enough that Discord delivery for OpenAI-Codex models cannot silently fail when an older binary reads a newer `openclaw.json`.

## Best Possible Solution

Define one supported downgrade/recovery policy at the config/gateway boundary, document it, and add focused regression or manual proof for a future-stamped config plus Discord/OpenAI-Codex final-message delivery.
