## Summary

Keep open. Current main still warns and continues on ordinary config load when `meta.lastTouchedVersion` is newer, while the hard refusal is scoped to service/process mutations such as service-mode startup and `gateway --force`. The closed auto-downstamp PR was not merged, and the merged GPT-5.4 hardening work is relevant to the symptom area but does not define the downgrade/future-config recovery policy requested here.

## What This Changes

The issue asks for a clear recovery policy when a newer `meta.lastTouchedVersion` stamp can silently break Discord delivery for OpenAI-Codex GPT-5.x models.

## Best Possible Solution

Define one supported downgrade/recovery policy at the config and gateway boundary, document it, and add focused regression or manual proof for a future-stamped config plus Discord/OpenAI-Codex final-message delivery.
