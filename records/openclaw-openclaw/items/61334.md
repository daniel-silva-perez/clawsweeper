## Summary

Keep open. The original 4x/second report was corrected by the reporter to a once-per-minute scheduler tick, and the remaining request is for wake-on-next-job scheduling. Current main still deliberately clamps future cron timer delays to 60 seconds and logs `cron: timer armed` on each re-arm, with regression coverage asserting that far-future schedules use a 60s cap. That means the refined issue is still current, not implemented or superseded by the related cron fixes listed in the context.

## Best Possible Solution

Keep this issue open and handle it as a core cron scheduler design/fix. The best path is to update `src/cron/service/timer.ts` so normal idle scheduling can sleep closer to `nextWakeAtMs` without losing the existing watchdog behavior for active runs, stuck markers, store maintenance, process pauses, and clock changes. Pair that with focused tests that replace or narrow the current far-future `60_000` expectation, and decide whether `cron: timer armed` should be less noisy or only emitted when timing materially changes.
