## Summary

Keep open: the original 4x/second claim was corrected by the reporter, but the refined report still matches current main. The scheduler computes the next due job, then deliberately clamps far-future waits to 60 seconds and logs `cron: timer armed`, with regression coverage that currently expects that behavior.

## What This Changes

The issue asks for the gateway cron scheduler to stop fixed 60-second polling/logging and sleep until the next due cron job.

## Best Possible Solution

Decide the scheduler policy explicitly, then separate due-job wake timing from maintenance/watchdog rechecks: use the true next job time for normal idle scheduling, keep bounded rechecks for running/stuck/reload/reaper recovery, and add regression tests for daily far-future jobs plus the existing hot-loop and long-running-job cases.

## Reproduction Assessment

Yes: current source and tests reproduce the refined report without live logs. A far-future or daily job has `nextRunAtMs` hours away, but `armTimer` still schedules a 60000 ms timeout and logs the timer as armed.

## Solution Assessment

Unclear because there is no patch yet. The requested wake-on-next-job direction is plausible, but the current 60-second cap is intentional scheduler hardening, so the maintainable fix should split normal scheduling from maintenance rechecks rather than simply removing the cap everywhere.
