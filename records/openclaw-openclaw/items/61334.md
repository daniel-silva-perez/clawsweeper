## Summary

Keep open. The reporter corrected the original 4x/second claim to once-per-minute ticks, but the refined wake-on-next-job request is still current: main deliberately caps future cron timer delays at 60 seconds and logs each re-arm, with regression coverage asserting that far-future schedules use the 60s cap.

## What This Changes

Request to replace the cron scheduler's fixed 60-second idle recheck/log cycle with wake-on-next-job timer arming for idle schedules.

## Best Possible Solution

Keep this open and decide the cron scheduler policy explicitly. A good implementation would let normal idle scheduling sleep closer to `nextWakeAtMs` while preserving separate watchdog or maintenance rechecks for active runs, stuck markers, external store changes, process pauses, and clock jumps, then update the current 60s-cap tests and log behavior accordingly.

## Reproduction Assessment

Yes. A high-confidence read-only reproduction exists: configure or inspect any far-future/daily cron job with `nextRunAtMs` hours away, and current `armTimer` still schedules `setTimeout` with `60_000`; the regression test asserts that exact delay.

## Solution Assessment

Unclear. Wake-on-next-job is a reasonable target for idle daily schedules, but the current 60s cap is deliberate recovery behavior, so the best fix should separate idle scheduling from watchdog/maintenance rechecks rather than removing the cap blindly.
