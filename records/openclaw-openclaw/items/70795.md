## Summary

Keep open. Current main still models filesystem tool policy as a single `workspaceOnly` boolean, and the direct-session `read` path is assembled around a workspace root plus a single-root guard rather than a shared approved-root policy. The report overlaps open security and filesystem-policy work, so it needs maintainer/security direction before implementation.

## What This Changes

The issue requests that direct-session `read` follow the same effective filesystem policy as the runtime, including approved non-workspace roots, instead of remaining pinned to the workspace root.

## Best Possible Solution

Define the intended filesystem policy model first, including approved roots, read-vs-write semantics, sandbox mount behavior, sensitive-path handling, and direct-session tool assembly. Then implement one shared contract with regression coverage across host tools, sandbox tools, image/PDF local paths, nodes paths, and docs.

## Reproduction Assessment

Yes for the source-level mismatch: current main shows direct `read` and related guards using a boolean workspace-root model rather than a shared approved-root policy. The exact macOS outer tool-host session should still be live-validated before changing access semantics.

## Solution Assessment

Unclear as an isolated code fix. The maintainable path is not only to relax `read`; it is to make filesystem access policy explicit and shared so broader read access does not undermine privacy or sensitive-path protections.
