## Summary

Keep open. Current main still treats filesystem access as a boolean workspace-only policy and the central read/sandbox guards are still single-rooted; the related sensitive-path PR is open, not merged, and this is security-sensitive enough to need maintainer policy before widening direct-session read access.

## What This Changes

The issue requests that the direct-session `read` tool honor the effective filesystem policy instead of remaining bound to the workspace root when approved non-workspace read access is configured.

## Best Possible Solution

Define one explicit filesystem access contract for read roots, sensitive-path denial, read-vs-write behavior, sandbox mounts, direct-session tool assembly, and per-agent privacy; then route read/write/edit/apply_patch, media loading, node output paths, sandbox bridges, and plugin tool context through that shared policy with regression coverage.

## Reproduction Assessment

Yes for the current source-level mismatch: current main still shows boolean-only filesystem policy and single-root read/guard wiring. The exact macOS direct-session failure should be live-validated before maintainers widen the policy.

## Solution Assessment

Unclear as an isolated code change. Relaxing `read` alone is not the best solution; the safer path is a shared filesystem policy contract with approved roots and deny rules first.
