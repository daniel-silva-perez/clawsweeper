## Summary

Keep open. Current main still has an all-or-nothing `tools.fs.workspaceOnly` model for direct filesystem tools, and the direct-session `read` path is not backed by a shared approved-root/read-vs-write policy. Related work exists, but it is still open and security-sensitive enough that this should stay with maintainers rather than be auto-closed or queued as an isolated fix PR.

## Best Possible Solution

Keep this issue open for maintainer/security triage. The best path is to define the intended filesystem policy model first, including approved roots, read-vs-write semantics, sandbox mount behavior, sensitive-path handling, and direct-session tool assembly, then implement one shared contract with regression coverage across direct host tools, sandbox tools, image/PDF local paths, nodes output paths, and docs.
