## Summary

Keep open. Current main still emits `session:patch` only from the privileged `sessions.patch` RPC path, while the reported `/model` and `session_status(model=...)` model-change paths mutate the session store directly and do not call the hook pipeline. The docs describe these paths as equivalent user session-selection writers, so the reporter's hook gap remains valid.

## What This Changes

The issue requests that session model changes made through `/model` or agent-internal paths such as `session_status(model=...)` emit `session:patch` with a model patch, or an equivalent model-changed hook event.

## Best Possible Solution

Add a shared session patch hook emission helper or shared model-patch path used by `sessions.patch`, directive-only `/model`, and `session_status(model=...)`. Keep the existing live-switch behavior, emit `patch: { model: ... }` only after successful persistence, and add focused regression coverage for both reported model-change paths.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: register a `session:patch` listener, then inspect the `/model` and `session_status(model=...)` paths on current main; both persist model changes directly without calling `triggerInternalHook`.

## Solution Assessment

Yes, Option A is the best direction, but it should be implemented as a small shared helper or patch pipeline rather than duplicating hook construction. A new `session:model_changed` event is not necessary unless maintainers want a separate additive hook surface after preserving `session:patch` consistency.
