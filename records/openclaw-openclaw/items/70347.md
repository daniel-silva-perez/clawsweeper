## Summary

Keep open. Current main still has the reported cron outer-timeout gap: the timeout wrapper aborts and rejects the run, but it does not emit a terminal lifecycle event with the active cron session identity, so the existing gateway session lifecycle persistence path is not driven in-band.

## Best Possible Solution

Keep this issue open and implement a cron-owned timeout finalization path. Preserve the active isolated cron run identity once the run session is prepared, and when the outer wall timer fires after execution start, publish a terminal lifecycle event with timeout metadata so the existing gateway lifecycle persistence path finalizes sessions.json immediately. Add focused regression coverage for a started isolated cron run timing out and the matching session row becoming terminal without waiting for watchdog cleanup.

## Clownfish Work Prompt

Fix the cron outer-timeout lifecycle finalization gap tracked by this issue. Current main's `executeJobCoreWithTimeout` in `src/cron/service/timer.ts` aborts and rejects on timeout, but does not emit a terminal lifecycle event with the active isolated cron session identity, so `sessions.json` is not finalized in-band.

Expected boundary:
- Keep the fix in the cron/gateway lifecycle path; do not add a bespoke direct sessions.json write if the existing lifecycle persistence path can be reused.
- Preserve the active isolated cron run identity once `runCronIsolatedAgentTurn` has prepared `runSessionId` / `runSessionKey`, for example with a narrow callback/seam from the isolated runner to the cron timeout wrapper or gateway cron bridge.
- When the outer timeout fires after execution start, publish a terminal lifecycle event for the active run with timeout metadata: `stream: "lifecycle"`, terminal phase (`error`, or `end` with `aborted: true` if that better matches existing timeout persistence), `error: "cron: job execution timed out"`, `errorKind: "timeout"`, `stopReason: "timeout"`, and the active session key.
- Ensure finalization is immediate; a plain lifecycle `error` event currently goes through retry grace in `src/gateway/server-chat.ts`, so either mark/bypass timeout errors appropriately or use the existing aborted/end timeout mapping.
- Preserve the existing deferred-start timeout behavior for isolated cron runs waiting on lanes.
- Add regression coverage that forces a started isolated cron run to time out and asserts both the cron result and the matching session row/lifecycle persistence become terminal without waiting for watchdog repair.
- Add a user-facing changelog fix entry if the repo's active changelog section expects one; credit the reporter as `Thanks @holdenjrussell`.

Do not change unrelated lane concurrency, provider fallback, queue timeout policy, or plugin/extension behavior.

## Risks / Open Questions

- A plain lifecycle `error` event may still be delayed by the gateway's lifecycle-error retry grace; the fix should ensure timeout finalization is synchronous, for example by using the established aborted/end timeout mapping or by explicitly bypassing grace for timeout-kind lifecycle errors.
- The timeout callback must avoid double-finalizing if the underlying runner later emits or returns a terminal result after the outer race has already timed out.
- The fix needs to preserve the existing behavior from #41783: isolated cron timeout should start when execution actually begins, not while the run is still waiting for its lane.
- A direct sessions.json write from cron would bypass the existing lifecycle semantics; the safer path is to emit or otherwise route through the gateway lifecycle persistence contract.
