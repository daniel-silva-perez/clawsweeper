## Summary

Keep open. Current main still has the reported cron outer-timeout lifecycle gap: the timeout path aborts and rejects, but it does not emit a terminal lifecycle event or carry the prepared isolated cron run session identity into in-band gateway session persistence. Related #67785 fixed nested lane scoping, and #41783 covers queue-timeout semantics; neither supersedes this cron session-finalization bug.

## Best Possible Solution

Implement a narrow cron-owned timeout finalization path. Preserve or expose the prepared isolated cron run session identity once execution starts, and when the outer wall timer fires after that point, emit one lifecycle error event with timeout metadata through the existing agent event bus so gateway lifecycle persistence finalizes the matching session row immediately. Add regression coverage for a started isolated cron run timing out and the session lifecycle being finalized without watchdog repair.

## Clownfish Work Prompt

Fix the cron outer-timeout lifecycle finalization gap for this issue. Current main starts the isolated cron execution timeout after execution begins, but when the outer timer fires it only aborts and rejects, so the active isolated cron run session key is lost before gateway lifecycle persistence can finalize sessions.json.

Expected boundary:
- In cron-owned code, preserve or expose the prepared isolated cron run identity (`runSessionId` and `runSessionKey`) once the run context is created and execution has started.
- When the outer cron wall timeout fires after execution start, emit exactly one terminal lifecycle event through the existing agent event path with `stream: "lifecycle"`, `phase: "error"`, `error: "cron: job execution timed out"`, and timeout-specific metadata such as `errorKind: "timeout"` and/or `stopReason: "timeout"`.
- Reuse the existing gateway lifecycle persistence path (`persistGatewaySessionLifecycleEvent`) rather than writing sessions.json directly from cron.
- Do not change #41783 queue-wait timeout semantics beyond what is required for this lifecycle event, and avoid duplicate terminal events if the underlying runner later returns after observing the abort.
