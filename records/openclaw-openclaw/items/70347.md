## Summary

Keep open. Current main has partial cron timeout cleanup, including deferred timeout start and forced embedded-run cleanup, but the timeout winner still returns a plain cron error without emitting a terminal lifecycle event or carrying session identity into the cron finished event, so the reported synchronous session-finalization gap remains.

## What This Changes

The issue requests that cron outer wall-clock timeouts for already-started isolated agent-turn runs emit a terminal lifecycle event so gateway session state finalizes synchronously.

## Best Possible Solution

Add a cron-owned terminal lifecycle emission for started timed-out isolated runs, routed through the existing agent event/session lifecycle pipeline with timeout metadata and the active run session key, while preserving the newer queue-wait timeout behavior from #41783.

## Reproduction Assessment

Yes. A focused unit harness can reproduce this by calling onExecutionStarted with a sessionId/sessionKey, making runIsolatedAgentJob never settle, advancing the cron timeout, and observing that cleanup runs and cron records an error without any lifecycle emission or session finalization assertion.

## Solution Assessment

Yes, the issue's proposed direction is the narrow maintainable fix: reuse the existing lifecycle persistence path instead of adding a second sessions.json write path. The implementation should be idempotent and should not broaden cron lane or queue semantics.
