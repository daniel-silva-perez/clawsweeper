## Summary

Keep this issue open. Current main has sub-agent sessions, per-agent/static sandbox and tool policy, and a plugin runtime helper for launching subagent runs, but it still does not implement a plugin-returned isolation directive, per-operation constrained/quarantined execution, or structured trust/provenance metadata for isolated results. This remains a core/plugin API and security design item, with related prerequisite hook-enrichment issues still open.

## Best Possible Solution

Keep the issue open as a core/plugin API design item. The best path is an additive hook/runtime capability that builds on existing before_tool_call, subagent, sandbox, and tool-policy infrastructure, but explicitly defines constrained execution, enforcement guarantees, and structured trust/provenance metadata. Coordinate it with the still-open related hook-enrichment proposals #48503, #48506, and #48509 before treating it as implementable by a plugin.

## Risks / Open Questions

- This is security-sensitive API design: hook shape, enforcement guarantees, tool/model/token/time constraints, auth/resource isolation, and result metadata semantics need explicit maintainer/security review.
- Closing now would conflate existing sub-agent delegation/session separation and static sandbox/tool policies with the requested per-operation quarantined execution and trust/provenance contract.
- A plugin-only implementation is not sufficient without new core semantics because plugins cannot currently force one tool operation into a constrained isolated execution path and receive standardized trust metadata back.
