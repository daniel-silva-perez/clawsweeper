## Summary

Keep this PR open for maintainer review. Current main now solves the main dropdown-noise problem with a view-based visible-catalog path, but it does not implement this PR's explicit `gateway.controlUi.modelSelector.filter` config, `_meta` response, or model-switch preflight semantics. The branch is also stale against the current `models.list` API and has blocking security/correctness issues if merged as proposed.

## What This Changes

The PR adds a Control UI model-selector filter config, extends gateway `models.list` with filter metadata, adds `sessions.patch` credential preflight, updates the chat model selector UI, and adds related tests.

## Best Possible Solution

Rework the idea against current main's `resolveVisibleModelCatalog` and provider-auth helpers if maintainers still want a configurable filter/preflight feature. The rework should preserve the existing `view` API, prevent clients from weakening server policy, use runtime-auth-equivalent credential checks, document the chosen product behavior, and add focused gateway/UI/session tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Honor the server-side filter config:** `src/gateway/server-methods/models.ts:36-39`
  - body: The PR lets request params override `gateway.controlUi.modelSelector.filter`, so a client can send `filter: "all"` and receive the full catalog even when the operator configured authenticated/configured visibility. That contradicts the server-side filtering/security contract described in the PR.
  - confidence: 0.88
- **[P2] Reuse runtime auth availability for filtering:** `src/agents/model-selection.ts:814-820`
  - body: `hasAuthForProvider` only checks profiles, env keys, and custom API keys. It misses supported runtime auth such as implicit Bedrock AWS SDK auth and synthetic local no-key providers, so authenticated filtering and the new preflight can hide or reject models that OpenClaw can actually run.
  - confidence: 0.9
- **[P2] Keep configured mode from widening to the full catalog:** `src/agents/model-selection.ts:934-936`
  - body: When `filter: "configured"` is explicitly selected but no configured keys are found, the PR returns the entire catalog. An explicit configured-only mode should not silently expose every built-in model when the configured set is empty.
  - confidence: 0.86
- **[P2] Enforce configured membership during model patches:** `src/gateway/sessions-patch.ts:410-411`
  - body: The `sessions.patch` preflight checks only provider auth for both authenticated and configured modes. In configured mode, a client can still submit any authenticated catalog model that is not referenced in agent config, bypassing the mode's stated model allowlist semantics.
  - confidence: 0.78

## Security Review
