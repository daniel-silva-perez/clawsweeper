## Summary

Keep open. Current main already has a newer `view`-based server-side picker path that addresses much of the reported dropdown problem, but this PR still proposes additional config, `_meta`, and `sessions.patch` behavior that is not implemented on main. The branch is also stale against that current path and contains auth/visibility regressions that should block merge until maintainer security/product review.

## What This Changes

The PR adds `gateway.controlUi.modelSelector.filter`, extends `models.list` with filter metadata, adds model-selection credential preflight in `sessions.patch`, updates the Control UI model selector state/status display, and adds related tests.

## Best Possible Solution

Rework the idea against current main's `resolveVisibleModelCatalog` and provider-auth helpers if maintainers still want a configurable filter/preflight feature. The rework should preserve the existing `view` contract, prevent clients from weakening server policy, use runtime-auth-equivalent checks, document the chosen product behavior, and add focused gateway/UI/session tests.

## Reproduction Assessment

Yes. The blocking PR findings are reproducible by static review of the diff against the current gateway/UI/model-auth path: request-level `filter` can override config, configured mode widens to the full catalog, and the new auth helper omits runtime auth sources current main already recognizes.

## Solution Assessment

No. This is not the best way to solve the issue as written; the safer direction is to integrate with the current `view: "configured"` model-list contract and shared runtime auth checker rather than adding a parallel client-selectable filter path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Honor the server-side filter config:** `src/gateway/server-methods/models.ts:36-39`
  - body: The handler lets request params override `gateway.controlUi.modelSelector.filter`, so a client can send `filter: "all"` and receive the full catalog even when the operator configured authenticated/configured visibility. That makes the visibility policy client-selectable instead of server-enforced.
  - confidence: 0.88
- **[P2] Reuse runtime auth availability for filtering:** `src/agents/model-selection.ts:814-820`
  - body: `hasAuthForProvider` checks only profiles, env keys, and custom API keys. It misses runtime-supported auth such as synthetic local providers and Bedrock AWS SDK auth, so authenticated filtering and the new preflight can hide or reject models OpenClaw can actually run.
  - confidence: 0.9
- **[P2] Keep configured mode from widening to the full catalog:** `src/agents/model-selection.ts:934-936`
  - body: When `filter: "configured"` is explicitly selected but no configured keys are found, this returns every catalog entry. A restrictive configured-only mode should not silently expose all built-in models when the configured set is empty.
  - confidence: 0.86
- **[P2] Enforce configured membership during model patches:** `src/gateway/sessions-patch.ts:410-411`
  - body: The `sessions.patch` preflight checks only provider auth for both authenticated and configured modes. In configured mode, a client can still submit any authenticated catalog model that is not referenced in agent config, bypassing the mode's stated allowlist semantics.
  - confidence: 0.78
