## Summary

Keep open. Current main still does not implement the PR’s central Telegram behavior: `/model` in a Telegram topic/thread is persisted only to the active session, with no parent-chat future-thread default state or Telegram-specific parent resolver. The PR contains relevant work, but the supplied review discussion and file list show unresolved correctness issues and unrelated memory/gateway/E2E tooling changes.

## Best Possible Solution

Keep this PR open, but ask for a scoped revision or replacement focused on Telegram thread model inheritance. The implementation should persist a same-chat parent future-thread default from Telegram `/model`, seed only newly created Telegram topic/DM-thread sessions from that default, cover both inline and directive-only command paths, handle main-scoped DM keys without cross-chat leakage, and split unrelated memory, gateway, recovery, and optional live E2E tooling into separate PRs.
