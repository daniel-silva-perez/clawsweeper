## Summary

Current main still has the ACP TUI lazy-init gap this PR targets, so the PR is not obsolete and should not be closed. The current reply-dispatch path returns unhandled when ACP metadata is missing, and the normal reply pipeline then falls through to embedded/default reply execution.

## Best Possible Solution

Keep this PR open as the active implementation candidate. The best path is a narrow lazy ACP initialization in `src/auto-reply/reply/dispatch-acp.ts` for sessions whose resolved agent has `runtime.type === "acp"`, with normalized `runtime.acp` fields, policy and empty-turn checks before runtime startup, unconditional re-resolve after init, focused regression tests for ACP and non-ACP agents, and explicit maintainer judgment on whether the CLI parity changes stay in the same PR.
