## Summary

Keep open. Current main still has the TUI ACP activation gap this PR is trying to fix, but the latest branch is not merge-ready: raw model runs can start ACP as a side effect, CLI ACP turns can use a stale pre-init prompt body, and first-turn inline image-only dispatch still falls back to embedded.

## What This Changes

The PR adds lazy ACP session initialization for agents configured with `runtime.type: "acp"` in TUI reply dispatch and CLI agent command execution, exposes runtime/default fields through `resolveAgentConfig`, adds regression tests, and updates the changelog.

## Best Possible Solution

Keep the lazy ACP initialization direction, but make it side-effect-safe and current-main-compatible: skip CLI lazy init for raw runs, recompute ACP prompt text after lazy re-resolve, treat inline images as non-empty dispatch input, rebase without dropping configured thinking catalog behavior, and add targeted tests for those paths.

## Reproduction Assessment

Yes. Current main can be source-reproduced by a TUI session whose ACP resolution is `kind: "none"`; the PR-introduced regressions have direct control-flow reproductions through raw model runs, internal-event CLI ACP turns, and first-turn inline image-only dispatch.

## Solution Assessment

No. Lazy initialization is the right narrow fix for the missing TUI path, but this implementation still starts ACP on a raw-run path and misses prompt/attachment edge cases that should be fixed before merge.
