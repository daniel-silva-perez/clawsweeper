## Summary

This PR should stay open. Current main still has the ACP lazy-init gap the PR is trying to address: the reply-dispatch ACP path returns unhandled when session metadata is missing, and the normal reply pipeline then continues into embedded/default reply execution. The PR is not obsolete or implemented on main.

## Best Possible Solution

Keep this PR open as the active implementation candidate. The best path is a narrow lazy ACP initialization in `src/auto-reply/reply/dispatch-acp.ts` for sessions whose resolved agent has `runtime.type === "acp"`, with normalized `runtime.acp` fields, policy and empty-turn checks before runtime startup, unconditional re-resolve after init, focused regression tests for ACP and non-ACP agents, and explicit maintainer judgment on whether the CLI parity changes stay in this PR.
