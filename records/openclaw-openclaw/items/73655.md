## Summary

Keep this issue open. The report is recent and specific, and current main still has the relevant runtime surfaces: deferred post-listen plugin reload, best-effort asynchronous cleanup of replaced plugin registries, optional plugin lifecycle cleanup hooks, and synchronous session-store I/O under the session write lock. Main has added event-loop health diagnostics after v2026.4.26, but that is observability, not a fix for the Manifest port retry, signal-listener accumulation, or session I/O starvation described here. The reporter’s note that #73653 is separate is consistent with the code: this is a runtime resource/lifecycle report, not the schema-validation regression tracked there.

## Best Possible Solution

Keep the issue open and treat it as a runtime lifecycle bug. The best path is to reproduce with the Manifest/clawrouter plugin enabled and a cron-driven session, then add targeted regression coverage around deferred plugin reload: no duplicate listen attempts, no growth in process signal listener counts across reloads, and bounded WS upgrade latency while session writes are active. The likely fix belongs in a combination of core plugin-resource cleanup semantics and session-store I/O behavior: make restart cleanup complete before replacement where necessary, centrally track or require disposal for plugin-owned sockets/listeners, and move hot session JSON/session transcript reads and writes off synchronous event-loop paths.
