## Summary

Keep this open. Current main has partial mitigations for startup/readiness and handshake budgets, but the reported gateway lifecycle problem is not fully fixed: session-store writes still run inside the session-store lock, durable atomic writes still await fsync, plugin replacement cleanup is hook-based rather than a general process/socket/signal cleanup registry, and the Manifest/clawrouter port-binding code was not present in the checked-in source I searched. The follow-up comments also add fresh Linux/ARM64 confirmations, including cases without the Manifest plugin, so this is still an active gateway lifecycle and persistence regression rather than an obsolete single-environment report.

## What This Changes

The issue reports a gateway restart/runtime leak triad involving Manifest port rebinding, process signal listener accumulation, and session persistence stalls that can starve local WebSocket handshakes.

## Best Possible Solution

Treat this as an active gateway lifecycle umbrella and split the implementation into staged, reviewed fixes: identify and make the Manifest/clawrouter listener idempotent, add a generic plugin-owned process/socket/signal cleanup contract or registry, and redesign session store/transcript persistence so slow durable writes do not hold latency-sensitive locks. Validate under restart/load with the related WebSocket-starvation and session-lock reports before closing.
