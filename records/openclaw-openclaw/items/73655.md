## Summary

Keep this open. Current main has added useful mitigations around gateway lock conflicts, handshake/readiness timing, runtime mirrors, and session-lock cleanup, but it still does not close the reported root causes: the session/transcript lock can still span long embedded attempts and durable writes, plugin replacement cleanup is still limited to registered host hooks, and the reported Manifest/clawrouter 2099/2100 listener implementation is not present in the checked-in source.

## What This Changes

The issue reports gateway WebSocket starvation after plugin/gateway restart from a Manifest port rebinding loop, accumulated process signal handlers, and long session JSONL/store lock holds.

## Best Possible Solution

Keep this as an active maintainer-owned gateway lifecycle/performance bug and split the final fix across the real owner surfaces: make the Manifest/clawrouter listener startup idempotent, add or enforce a generic cleanup/dedup contract for plugin-owned process resources, and redesign session persistence so slow durable writes or long embedded attempts cannot starve Gateway WebSocket handshakes.

## Reproduction Assessment

Unclear. The issue has concrete probes and logs for listener counts, same-PID port ownership, session-write-lock warnings, and WS handshake timeouts, plus independent symptom confirmations, but the full triad was described as emergent and was not deterministically reproduced against current main in this read-only review.

## Solution Assessment

No. Current main’s mitigations are useful but not the best complete solution because they mostly improve timeout/readiness behavior and supervisor recovery while leaving the likely root causes in plugin lifecycle cleanup and broad session persistence locks.
