## Summary

Keep this open. Current main has useful mitigations for client-side event-loop readiness, configurable handshake timeouts, startup retry responses, and session-lock watchdog cleanup, but the reported root causes are not closed: session-store updates still persist under the store lock, plugin replacement cleanup is still hook-based, and the Manifest/clawrouter listener implementation is not present in the checked-in source. Follow-up reports also confirm the same WebSocket starvation shape without Manifest, so this remains an active gateway lifecycle/performance bug.

## What This Changes

The issue requests fixes for gateway restart resource leaks and WebSocket handshake starvation involving Manifest port rebinding, process signal-listener accumulation, and slow session persistence locks.

## Best Possible Solution

Split this into maintainer-owned root-cause fixes: make the Manifest/clawrouter listener startup idempotent, provide or require a generic cleanup/dedup surface for plugin-owned process-level resources, and redesign session persistence so slow durable writes cannot hold latency-sensitive Gateway/session locks through long fsyncs while preserving the intended durability contract.

## Reproduction Assessment

Unclear for the full triad: the original report is emergent and was not live-reproduced in this read-only review, but the issue includes concrete probes for listener counts, same-PID port ownership, lock warnings, and WS timeout logs. Independent comments provide high-confidence symptom reproduction for the non-Manifest starvation path.

## Solution Assessment

No, current main is not the best complete solution because it mostly adds mitigations and diagnostics around startup/client timing while leaving the likely root causes in plugin lifecycle cleanup and session persistence. The safer path is staged subsystem fixes with explicit durability and plugin-lifecycle review.
