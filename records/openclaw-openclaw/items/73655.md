## Summary

Keep this open. Current main has useful mitigations for Gateway startup/readiness and stuck-session cleanup, but the reported root causes are not closed: session-store writes still persist under the store lock through durable fsync work, plugin replacement cleanup remains hook-based, and the reported Manifest/clawrouter listener implementation is not present in the checked-in source. Follow-up comments also confirm the same WebSocket starvation shape without Manifest, so this is still an active gateway lifecycle/performance bug rather than an obsolete report.

## What This Changes

The issue asks OpenClaw to fix a Gateway restart leak triad involving Manifest port rebinding, process signal-listener accumulation, and session persistence stalls that can starve local WebSocket handshakes.

## Best Possible Solution

Split this into maintainer-owned root-cause fixes while keeping the existing readiness mitigations: make the Manifest/clawrouter listener startup idempotent, provide a generic plugin lifecycle cleanup/dedup contract for process-level resources, and redesign session persistence so slow durable writes cannot block latency-sensitive Gateway handshakes or session lanes.

## Reproduction Assessment

Unclear for the full triad: the original report is emergent and was not live-reproduced in this read-only review, but it provides concrete probes for listener counts, same-PID port ownership, lock warnings, and WS timeout logs. Follow-up comments provide high-confidence symptom reproduction for the non-Manifest starvation path.

## Solution Assessment

No, current main is not the best complete solution because it mostly adds mitigations and diagnostics around startup/client timing while leaving the likely root causes in plugin lifecycle cleanup and session persistence. The safer path is staged subsystem fixes with explicit durability and plugin-lifecycle review.
