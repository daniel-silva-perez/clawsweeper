## Summary

Keep this open. Current main has useful mitigations around startup readiness, supervised EADDRINUSE exits, runtime-deps churn, and stale-session recovery, but the reported triad is not fully resolved: the Manifest/clawrouter listener source is not present in this checkout, session-store persistence still runs durable atomic writes while the session-store lock is held, and plugin restart cleanup remains limited to registered host-hook cleanup rather than arbitrary process listeners or sockets.

## What This Changes

The issue asks OpenClaw to fix a gateway restart/runtime regression where Manifest port rebinding, signal-handler accumulation, and slow session persistence combine to starve local WebSocket handshakes.

## Best Possible Solution

Split this into maintainer-owned root-cause fixes: make the Manifest/clawrouter listener startup idempotent, add or require a generic plugin lifecycle cleanup/dedup surface for process-level resources, and redesign session persistence so slow durable writes cannot block Gateway handshakes or session lanes while preserving the intended durability contract.

## Reproduction Assessment

Unclear for the full triad: the report provides concrete probes, logs, and independent confirmations, but there is no deterministic one-command reproduction. The session-lock/WS-starvation portion has stronger evidence than the Manifest-specific port loop because the Manifest listener code is not in this checkout.

## Solution Assessment

No, current main is not the best complete solution. The current changes are useful mitigations and diagnostics, but they do not remove all three reported root causes.
