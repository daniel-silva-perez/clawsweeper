## Summary

Keep this open. Current main contains useful mitigations for stuck-session recovery, handshake timeout tuning, client-side event-loop readiness, and bundled runtime-deps staging, but it does not close the reported triad: session-store writes still run inside the session-store lock through durable atomic fsync work, plugin replacement cleanup still depends on registered plugin host cleanup hooks, and the reported Manifest/clawrouter 2099/2100 listener bootstrap is not present in the checked-in source. The follow-up comments also add independent confirmations of the same WebSocket starvation shape without Manifest, so this is still an active gateway lifecycle/performance bug rather than a stale or implemented report.

## What This Changes

The issue requests fixes for gateway restart/resource leaks that compound into WebSocket handshake starvation, including Manifest port rebinding, process signal-listener accumulation, and long session JSON/session JSONL lock holds.

## Best Possible Solution

Split this into maintainer-owned fixes that preserve the current mitigations while removing the root causes: make the Manifest/clawrouter listener startup idempotent, add a generic plugin lifecycle cleanup/dedup surface for process-level resources, and redesign session persistence so slow durable writes cannot block latency-sensitive Gateway handshakes or session lanes.
