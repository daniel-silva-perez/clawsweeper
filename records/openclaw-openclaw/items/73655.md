## Summary

Keep this issue open. Current main still appears to have only a partial stuck-session mitigation that references this report; the remaining reported triad is not implemented or disproven: Manifest/clawrouter same-process EADDRINUSE retry behavior, process signal-handler accumulation across restart paths, and session-store durability work that can hold the session write lock while awaited fsync-style writes complete. The reporter's follow-up comments usefully narrow the scope and link downstream adapter retry mitigation in paperclipai/paperclip#4684, but they do not replace an upstream gateway/plugin/session fix.

## Best Possible Solution

Keep the issue open and treat it as a gateway runtime lifecycle bug with a partial stuck-session mitigation already on main. The best path is to reproduce with Manifest/clawrouter enabled under cron/session load, then split the fixes: make the Manifest bootstrap idempotent around already-owned ports, make plugin restart cleanup centrally disposable for process listeners and sockets, and bound or restructure session-store durability work so WebSocket upgrades are not starved. Add regression coverage for listener counts across reloads, duplicate listen attempts, and WS handshake latency while session writes are active.
