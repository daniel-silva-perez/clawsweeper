## Summary

Keep this open. Current main has partial mitigations for stale session-lane recovery and configurable handshake budgets, but it does not close the reported triad: the plugin reload cleanup still depends on registered lifecycle hooks, session-store writes still hold the store lock through durable atomic writes, the Manifest/clawrouter port-binding implementation was not found in checked-in source, and related #74323 reports the same WebSocket starvation shape on 2026.4.26 without the Manifest plugin.

## What This Changes

Fix a gateway restart/runtime resource leak triad causing WebSocket handshake starvation: Manifest/clawrouter port rebinding, process signal-listener accumulation, and long session-store lock holds during session persistence.

## Best Possible Solution

Keep this open as an active gateway lifecycle bug. The best maintainer path is to split or stage fixes for plugin port-binding idempotency, centralized/deduplicated plugin-owned process/signal/socket cleanup across reloads, and session persistence that avoids holding latency-sensitive locks through slow durable writes. Use #74323 as supporting evidence for the non-Manifest WebSocket starvation path and keep #73653 separate as the schema-validation regression it describes.
