## Summary

Keep this issue open. Current main has a partial Unreleased mitigation for stuck-session recovery that references this report, but it is scoped to stale session lanes after restart and does not resolve the full reported triad: Manifest/clawrouter same-process EADDRINUSE retry behavior, process signal-handler accumulation, and session-store work that can still hold the session write lock while doing synchronous reads and durable fsync writes. The reporter's later comments are useful evidence: they separate this from #73653, point to downstream adapter retry work in paperclipai/paperclip#4684, and narrow the remaining fixes to plugin bootstrap/lifecycle cleanup plus session durability architecture rather than a small already-shipped fix.

## Best Possible Solution

Keep the issue open and treat it as a runtime lifecycle bug with a partial stuck-session mitigation already on main. The best path is to reproduce with the Manifest/clawrouter plugin enabled under cron/session load, then split fixes if needed: make plugin restart cleanup complete and centrally disposable for process listeners/sockets, fix the Manifest bootstrap so it does not retry binding ports already owned by the gateway process, and redesign session persistence so durable reads/fsync work cannot hold session locks long enough to starve WebSocket upgrades. Add regression coverage for listener counts across reloads, no duplicate listen attempts, and bounded WS handshake latency while session writes are active.
