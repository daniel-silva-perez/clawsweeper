## Summary

Keep this open. Current main has additional mitigation around gateway readiness, stuck-session recovery, client-side event-loop waits, and supervised EADDRINUSE exits, but the root-cause triad is not fully fixed: durable session-store writes still run while the session-store lock is held, plugin replacement cleanup remains limited to registered host-hook cleanup, and the reported Manifest/clawrouter 2099/2100 listener implementation is not present in the checked-in source. The discussion also contains independent non-Manifest confirmations, so this remains active maintainer work rather than stale cleanup.

## What This Changes

The issue reports a gateway restart/runtime leak cluster involving Manifest port rebinding, process signal listener accumulation, and long session persistence lock holds that can starve WebSocket handshakes.

## Best Possible Solution

Split this into maintainer-owned root-cause fixes: make the Manifest/clawrouter listener startup idempotent, add or require a generic plugin lifecycle cleanup surface for process-level resources such as signal handlers and sockets, and redesign session persistence so slow durable writes cannot block latency-sensitive Gateway handshakes or session lanes while preserving the intended crash-durability contract.

## Reproduction Assessment

Unclear for the full triad: the report has concrete probes and logs for listener counts, same-PID port ownership, session-lock warnings, and WebSocket timeouts, but no deterministic one-command reproduction was run here. The non-Manifest WebSocket/session-lock symptom has stronger reproduction evidence from related comments and #74325/#13744-style reports.

## Solution Assessment

No, current main is not the best complete solution. The existing changes are useful mitigations and diagnostics, but the maintainable fix still needs root-cause work in Manifest listener ownership, plugin lifecycle cleanup, and session persistence lock scope.
