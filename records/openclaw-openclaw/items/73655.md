## Summary

Keep this open. Current main contains useful adjacent mitigations for supervised EADDRINUSE loops, startup diagnostics, runtime-dependency churn, and handshake timeout handling, but it does not close the reported root causes: session-store updates still hold the store/session lock while durable atomic writes fsync, plugin replacement cleanup remains limited to registered host-hook surfaces, and the reported Manifest/clawrouter 2099/2100 listener implementation is not present in this checkout.

## What This Changes

Reports an emergent Gateway restart leak triad where a Manifest plugin port retry loop, accumulating signal handlers, and slow session JSON persistence can starve local WebSocket handshakes while HTTP health checks stay green.

## Best Possible Solution

Split the work into maintainer-owned root-cause fixes: make the Manifest/clawrouter listener startup idempotent, add or require a generic plugin lifecycle cleanup/dedup surface for process-level resources, and redesign session persistence so slow durable writes cannot block latency-sensitive Gateway handshakes or session lanes while preserving the intended crash-durability contract.

## Reproduction Assessment

Unclear for the full triad. The issue gives concrete probes and logs for listener counts, same-PID port ownership, lock holds, and WS timeouts, plus related deterministic restart-starvation reports, but this read-only review did not run a live gateway-under-load reproduction against current main.

## Solution Assessment

No, current main is not the complete solution. The existing changes are useful mitigations around supervised restarts, startup diagnostics, and handshake handling, but the narrow maintainable fix still needs root-cause work across Manifest listener ownership, plugin lifecycle cleanup, and session-store lock scope.
