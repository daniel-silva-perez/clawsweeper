## Summary

Keep this PR open. Current main still does not implement the PR's intended hard approval boundary for destructive or mutating tool calls from external-content hook sessions. The existing runtime has prompt wrapping for hook content and a before_tool_call approval seam, but there is no production policy connecting external-content provenance to mutating tool approval. The provided PR diff is also not merge-ready as-is because its new guard is test-only/unused and its web_search warning behavior conflicts with current tests.

## Best Possible Solution

Keep this PR open for maintainer or contributor follow-up, or supersede it with a production-wired trusted before-tool-call policy. The best implementation should carry external-content provenance into runtime tool context, use or align with isMutatingToolCall, route required approvals through the existing plugin approval path, honor deliberate allowUnsafeExternalContent bypasses, and update tests/docs for any intentional web_search warning behavior change.

## Risks / Open Questions

- Merging the PR as-is would advertise a hard approval boundary while the new guard remains unused by production tool dispatch.
- The PR's destructive-tool regex is narrower than current action-aware mutation detection and could miss existing mutating surfaces.
- A raw hook:* session-key check can miss normalized or agent-scoped hook provenance unless the production fix carries explicit external-content provenance into the tool-call context.
- Changing web_search to always include the long SECURITY NOTICE contradicts current tests and should be handled as a deliberate behavior/docs update.
- No supply-chain or CI execution regression was visible in the provided diff; the main security risk is false assurance from an unenforced guard.
