## Summary

Keep this PR open. Current main still does not implement the PR's intended hard approval boundary for mutating/destructive tool calls from external-content hook sessions. Main has prompt wrapping for hook content and a before-tool-call approval seam, but no production policy connects external-content provenance to mandatory human approval.

## Best Possible Solution

Keep this PR open or supersede it with a maintainer-reviewed security fix wired into the production before-tool-call/trusted-policy path. The fix should carry external-content provenance into tool context, use `isMutatingToolCall`, route `requireApproval` through the existing plugin approval gateway, honor deliberate unsafe-content bypasses, and update tests/docs for any intentional `web_search` warning behavior change.

## Risks / Open Questions

- Closing this PR now would drop an unresolved security-hardening candidate; current main still relies on prompt wrapping, tool policy, exec approvals, sandboxing, and allowlists rather than a hook-provenance approval gate.
- Merging the PR as-is would create false assurance because the new guard is not wired into production tool dispatch.
- The PR's destructive-tool regex is narrower than current `isMutatingToolCall` behavior and could miss existing mutating surfaces.
- A raw `hook:*` session-key check is brittle against agent-scoped session keys unless production carries explicit external-content provenance into tool context.
- Changing `web_search` to always include the long warning conflicts with current tests and needs an intentional behavior/docs update.
