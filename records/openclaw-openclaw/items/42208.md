## Summary

Keep open. Current main still does not implement the PR's intended hard approval boundary for mutating/destructive tool calls from external-content hook sessions. The existing code wraps and warns on hook content, and the before-tool-call approval seam exists, but no production policy connects hook/external-content provenance to mandatory human approval.

## Best Possible Solution

Keep this PR open or supersede it with a production-wired trusted before-tool-call policy. The best implementation should carry external-content provenance into tool context, use or align with `isMutatingToolCall`, route required approval through the existing plugin approval path, honor deliberate `allowUnsafeExternalContent` bypasses, and update tests/docs for any intentional `web_search` warning behavior change.

## Risks / Open Questions

- Merging the PR as-is would advertise a hard security boundary while the new guard remains unused by production tool dispatch.
- The PR's destructive-tool regex is narrower than current `isMutatingToolCall` behavior and could miss existing mutating surfaces.
- A raw `hook:*` session-key check may be too brittle unless production tool context carries explicit external-content provenance through the runtime.
- Changing `web_search` to always include the long warning conflicts with current tests and should be handled as an intentional behavior/docs update.
