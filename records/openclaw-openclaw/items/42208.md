## Summary

Keep this PR open. Current main still does not implement the PR's intended hard approval policy for destructive tool calls from external-content hook sessions, and the submitted PR diff would not enforce it at runtime because the new guard is not wired into production tool dispatch.

## Best Possible Solution

Keep this PR open or supersede it with a maintainer-reviewed security fix wired into the production before_tool_call/trusted-policy path. The fix should carry external-content provenance into tool context, reuse isMutatingToolCall, route mandatory approval through the existing plugin approval gateway, honor deliberate unsafe-content bypasses, and update tests/docs for any intentional web_search warning behavior change.

## Risks / Open Questions

- Merging the PR as submitted would not enforce the promised hard approval boundary because the new guard is unused in production tool execution.
- The proposed destructive-tool regex is narrower than the current mutating-tool classifier and could miss existing mutating surfaces.
- A raw hook:* session-key policy would miss agent-scoped hook session keys unless external-content provenance is carried into before-tool-call context.
- Changing web_search to always include the long SECURITY NOTICE conflicts with current tests and needs an intentional behavior/docs update.
