## Summary

Keep this PR open. Current `main` still has prompt wrapping for external hook content and a generic before-tool-call approval seam, but it does not implement the PR's intended hard policy that mutating/destructive tool calls from external-content hook sessions must require human approval.

## Best Possible Solution

Keep this PR open or supersede it with a maintainer-reviewed security fix wired into the production before-tool-call/trusted-policy path. The fix should carry external-content provenance into tool context, use `isMutatingToolCall`, route mandatory approval through the existing plugin approval gateway, honor deliberate unsafe-content bypasses, and update tests/docs for any intentional `web_search` warning behavior change.

## Risks / Open Questions

- Closing this PR now would drop an unresolved security-hardening candidate for external-content hook sessions.
- Merging the PR as submitted would create false assurance because the new guard is not wired into production tool dispatch.
- The PR's destructive-tool regex is narrower than current `isMutatingToolCall` behavior and could miss existing mutating surfaces.
- A raw `hook:*` session-key check is brittle against agent-scoped session keys unless production carries explicit external-content provenance into tool context.
- Changing `web_search` to always include the long warning conflicts with current tests and needs an intentional behavior/docs update.
