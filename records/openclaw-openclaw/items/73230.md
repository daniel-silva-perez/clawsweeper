## Summary

Keep open. Current main still preserves cron `delivery.accountId` as an explicit delivery identity and gives it precedence over session and binding-derived account IDs, so a rotated external channel account can remain pinned to a stale value. The report is recent, concrete, and points at a generic cron delivery contract gap rather than a WeChat-specific core branch.

## What This Changes

Cron delivery should dynamically resolve or recover stale `delivery.accountId` values for channel plugins whose account IDs can rotate, such as `openclaw-weixin`.

## Best Possible Solution

Implement a channel-agnostic delivery-time account resolution rule: keep explicit account IDs when they still resolve, recover only when the stored account is stale and exactly one eligible account exists for the channel, and otherwise fail with a clear actionable error. Add focused cron tests and update docs without hardcoding `openclaw-weixin` policy into core.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: create an isolated cron delivery with a stale `delivery.accountId`; current `resolveDeliveryTarget` returns that stale value because explicit account IDs have highest precedence before dispatch passes the account into outbound delivery.

## Solution Assessment

Yes, if solved generically and conservatively. The best path is delivery-time account resolution with strict ambiguity handling, not a startup migration or a WeChat-specific branch.
