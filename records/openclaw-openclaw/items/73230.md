## Summary

Keep open. Current main still treats cron `delivery.accountId` as an explicit preserved value and gives it priority over session-derived and binding-derived accounts, so a rotated/stale account ID will continue through delivery resolution instead of using a dynamic or single-active-account fallback. The WeChat runtime is external, but the remaining gap is in OpenClaw's generic cron delivery target contract.

## What This Changes

The issue requests dynamic cron delivery account resolution so channel plugins with rotating account IDs can still receive scheduled delivery after relogin or Gateway restart.

## Best Possible Solution

Implement a channel-agnostic cron account-resolution rule that keeps explicit account IDs when they still resolve, but recovers or fails clearly when a stored account ID is stale. The end state should be documented, tested against single-account and multi-account cases, and avoid any `openclaw-weixin` special case in core.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: create an isolated cron announce target with a stale `delivery.accountId`, and current `resolveDeliveryTarget` returns that stale value because job account IDs outrank session and binding fallbacks. I did not run a live WeChat relogin test because this review is read-only and the static path already demonstrates the generic contract gap.

## Solution Assessment

Yes, if implemented generically and conservatively. The best direction is delivery-time account resolution with an explicit dynamic selector or an unambiguous single configured/enabled account fallback, plus a clear delivery-target error when resolution is unsafe.

## Repair Work Prompt

Fix the cron delivery stale-account gap for https://github.com/openclaw/openclaw/issues/73230. Observable bug: isolated cron jobs store a static `delivery.accountId`; for plugins whose configured account id rotates after login/restart, cron continues sending the stale id and delivery fails. Implement a generic OpenClaw cron/channel solution, not a WeChat-specific branch. Preserve explicit account IDs when they still resolve. Add either a documented explicit dynamic selector such as `accountId: "auto"`/`"active"`, or a delivery-time fallback for stale/missing explicit account IDs only when exactly one configured/enabled account for the resolved channel is safe to choose. If multiple accounts are configured/enabled or resolution is otherwise ambiguous, return a clear delivery-target error instead of guessing. Coordinate with related account-validation work in #69513/#69562 and account-inference discussion in #49507; do not replace those with a broad lint feature. Update protocol/schema/types, CLI/UI handling if a selector is introduced, docs, and focused tests. Changelog only if the change is user-visible; credit the original reporter `Thanks @sniu-git`. Do not hardcode `openclaw-weixin`, do not migrate existing cron files at startup as the primary fix, and do not silently switch identities in multi-account configs.
