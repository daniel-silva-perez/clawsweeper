## Summary

Keep open. Current main still preserves cron `delivery.accountId` as a static explicit account and forwards it into delivery, so a rotated WeChat/openclaw-weixin account can leave scheduled deliveries pinned to a stale account. The related merged spoof-guard work is relevant safety context, but it does not add dynamic account resolution or stale-account recovery.

## What This Changes

Add cron delivery support for resolving or recovering channel `accountId` at send time when a stored account ID has rotated or no longer matches the usable account for that channel.

## Best Possible Solution

Implement channel-agnostic delivery-time stale-account recovery: preserve explicit account IDs when they still resolve, recover only when the stored account is stale and exactly one eligible account exists for the channel, and otherwise fail with a clear diagnostic. Add focused cron resolver/dispatch tests and docs while preserving the existing accountId spoof guard.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: create an isolated cron announce delivery with a stale `delivery.accountId`; current `resolveCronDeliveryPlan` and `resolveDeliveryTarget` preserve that string, and `sendCronAnnouncePayloadStrict` forwards it to outbound delivery.

## Solution Assessment

Yes, a narrow generic runtime recovery is the best direction. It is safer than startup mutation and avoids a WeChat-specific core branch, provided it preserves strict ambiguity handling and the existing spoof guard.
