## Summary

Keep open. Current main still treats cron `delivery.accountId` as a static explicit identity and gives it precedence over session or binding-derived account IDs, so rotating channel account IDs can leave scheduled deliveries pinned to stale accounts. The related merged spoof-guard work is relevant to the safety boundary, but it does not implement dynamic or stale-account recovery.

## What This Changes

The issue requests dynamic cron delivery account resolution so scheduled channel sends do not stay pinned to stale account IDs after channel re-login or Gateway restart.

## Best Possible Solution

Implement channel-agnostic delivery-time account resolution: keep explicit account IDs when they still resolve, recover only when the stored account is stale and exactly one eligible configured or active account exists for the channel, and otherwise fail with a clear actionable error. Add focused cron tests and docs while preserving the existing accountId spoof guard.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: create an isolated cron delivery with a stale `delivery.accountId`; current `resolveDeliveryTarget` returns that explicit value before dispatch uses it.

## Solution Assessment

Yes, if solved generically and conservatively. Delivery-time stale-account recovery with strict ambiguity handling is narrower and safer than startup mutation or a WeChat-specific core branch.

## Repair Work Prompt

Fix cron delivery account resolution for stale explicit `delivery.accountId` values. Current main preserves `delivery.accountId` in `resolveCronDeliveryPlan` and gives it highest precedence in `resolveDeliveryTarget`, so rotating channel account IDs can break scheduled delivery. Implement a channel-agnostic delivery-time recovery rule: keep explicit account IDs when they still resolve; if the stored account is stale/missing and exactly one eligible configured or active account exists for the resolved channel, use that account; if zero or multiple eligible accounts exist, fail with a clear actionable error that points to updating the cron delivery account or logging in the channel. Do not hardcode `openclaw-weixin`, do not weaken the accountId spoof guard from #69940, and do not introduce cross-account fallback when ambiguous. Add focused tests for valid explicit account, stale single-account fallback, stale ambiguous-account failure, and no-account failure. Update cron docs to describe the recovery/ambiguity behavior. Add a `CHANGELOG.md` Fixes entry with `Thanks @sniu-git` if the change is user-facing.
