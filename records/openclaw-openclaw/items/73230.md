## Summary

Keep open. Current main still treats cron `delivery.accountId` as a literal explicit account id, gives it precedence over session and binding fallbacks, and has no documented or tested `auto`/`active` sentinel, stale-account repair, or startup warning. The WeChat plugin is external, but the failure sits on the generic cron delivery contract rather than a WeChat-only implementation detail.

## What This Changes

The issue requests dynamic cron delivery account resolution so jobs targeting channels with rotating account ids, such as `openclaw-weixin`, do not break after re-login or Gateway restart.

## Best Possible Solution

Add a generic cron account-resolution contract: an explicit dynamic mode such as `active` or a typed resolution option that resolves at delivery time only when the channel has exactly one safe active account, plus clear errors or doctor warnings for stale stored account ids. Keep the implementation channel-agnostic and avoid hardcoding `openclaw-weixin`.

## Reproduction Assessment

Yes. The current source and tests provide a high-confidence reproduction path: create an isolated cron job with `delivery.accountId` set, then change session or binding account context; `resolveDeliveryTarget` still returns the configured account id instead of resolving a current one.

## Solution Assessment

Unclear. The issue identifies a real gap, but the best fix needs maintainer choice between a sentinel, omitted-account fallback, stale-account repair, or a new plugin capability; a WeChat-specific core special case would not be the right direction.
