## Summary

Keep open. Current main still preserves cron `delivery.accountId` as an explicit value and gives it precedence during delivery target resolution, so a rotated channel-plugin account can remain pinned in stored cron config. The related WeChat false-success report was closed as external plugin behavior, but it explicitly leaves this generic core account-resolution gap as separate remaining work.

## What This Changes

Cron delivery should support dynamic or stale-aware account ID resolution for channel plugins such as `openclaw-weixin` whose account IDs can rotate after re-login or Gateway restart.

## Best Possible Solution

Implement a channel-agnostic delivery-time account-resolution contract: preserve explicit account IDs when they still resolve, recover only when a single safe active account is unambiguous, and otherwise fail with a clear delivery-target error plus documented operator guidance.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure isolated cron delivery with a stale `delivery.accountId`, then `resolveCronDeliveryPlan` and `resolveDeliveryTarget` preserve and prefer that stale value over session and binding fallbacks.

## Solution Assessment

Yes for the requested direction, if implemented generically. The maintainable fix is a typed cron/channel-plugin account-resolution path, not a WeChat-specific branch or an untyped magic string that could collide with a real account ID.

## Repair Work Prompt

Implement a conservative generic fix for https://github.com/openclaw/openclaw/issues/73230. Current main preserves stale `delivery.accountId` and gives it highest precedence during cron delivery target resolution. Add delivery-time account resolution without hardcoding `openclaw-weixin`: preserve an explicit accountId when it still resolves for the selected channel, but when it is absent or stale, recover only if the selected channel has exactly one safe configured/enabled/active account available through existing channel plugin config/status surfaces. If zero or multiple accounts are possible, return a clear delivery-target error and make sure failure notification/error state is visible. If existing plugin surfaces cannot express active account state, add the smallest typed SDK/channel-plugin seam needed rather than reading plugin-specific config. Avoid a raw magic string sentinel that could collide with a real account ID unless represented by an explicit schema/contract. Update cron delivery docs, add focused unit tests, and add a single changelog fix entry crediting `Thanks @sniu-git`. Do not change the external WeChat plugin and do not add WeChat-specific branches in core.
