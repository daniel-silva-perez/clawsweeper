## Summary

Keep open. Current main still treats cron `delivery.accountId` as a static explicit identity and gives it precedence over session- or binding-derived account IDs, so rotating channel account IDs can still leave scheduled deliveries pinned to stale accounts. The merged spoof-guard work in #69940 is relevant safety context, but it does not implement dynamic or stale-account recovery.

## What This Changes

Cron delivery should resolve or recover stale channel account IDs at delivery time for channels whose account IDs change across re-login or Gateway restart, such as `openclaw-weixin`.

## Best Possible Solution

Implement channel-agnostic delivery-time stale-account recovery: keep explicit account IDs when they still resolve, recover only when the stored account is stale and exactly one eligible configured or active account exists for the channel, and otherwise fail with a clear actionable error. Preserve the existing accountId spoof guard and document the behavior for cron delivery.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: create an isolated cron delivery with a stale `delivery.accountId`; current `resolveDeliveryTarget` returns that explicit value before dispatch sends it through `deliverOutboundPayloads`.

## Solution Assessment

Yes, if implemented generically and conservatively. Delivery-time stale-account recovery with strict ambiguity handling is narrower and safer than startup mutation or a WeChat-specific core branch.
