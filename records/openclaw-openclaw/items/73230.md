## Summary

Keep open. Current main still treats cron `delivery.accountId` as an explicit static identity and gives it precedence over session or binding-derived account IDs, so a channel account that rotates after re-login can leave scheduled delivery pinned to a stale account. The report is recent, reproducible from source inspection, and belongs in the generic cron delivery path rather than as a WeChat-specific core branch.

## What This Changes

Cron delivery should dynamically resolve or recover stale `delivery.accountId` values for channel plugins whose account IDs can rotate, such as `openclaw-weixin`.

## Best Possible Solution

Implement a channel-agnostic delivery-time account resolution rule: keep explicit account IDs when they still resolve, recover only when the stored account is stale and exactly one eligible configured or active account exists for the channel, and otherwise fail with a clear actionable error. Add focused cron tests and update cron docs without hardcoding `openclaw-weixin` policy into core.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: create an isolated cron delivery with a stale `delivery.accountId`; current `resolveDeliveryTarget` returns that stale value because explicit account IDs have highest precedence before outbound dispatch uses it.

## Solution Assessment

Yes, if solved generically and conservatively. Delivery-time account resolution with strict ambiguity handling is narrower and safer than a startup migration or a WeChat-specific branch.
