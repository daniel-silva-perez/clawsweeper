## Summary

Keep this issue open. Current main still treats cron `delivery.accountId` as a static literal with highest precedence, and I did not find an implemented `auto`/`active` sentinel, stale-account fallback, or startup/doctor warning for rotated channel account ids. The WeChat plugin is external, but the failing contract is generic cron delivery resolution in core, so this is not a clean ClawHub-only close.

## Best Possible Solution

Keep this issue open and implement an additive, channel-agnostic cron account-resolution path. The strongest direction is either a documented sentinel such as `delivery.accountId: "active"` or omitted-account fallback that resolves at delivery time only when exactly one configured/active account exists, plus clear errors or doctor/startup warnings for stale stored account ids. If plugin runtime state is required, add a typed SDK/plugin capability and keep WeChat-specific token handling in the external plugin.
