## Summary

Keep this PR open. Current main still has the default-only omitted-account routing behavior, still creates bare channel bindings without auto-wildcarding multi-account channels, and still lacks the PR's requested JSDoc/CLI-doc clarifications. The PR is useful, but the existing review feedback remains material: the behavior change needs to be scoped so channel-only unbind still works, and the docs need to match the actual routing implementation.

## Best Possible Solution

Keep the PR open and request a focused revision. Scope any automatic accountId "*" default to binding creation flows only, preserve channel-only unbind compatibility, correct the default-account wording against the routing implementation, update all affected examples consistently, and add targeted tests for multi-account bind/add behavior plus unbind compatibility. The narrower open Feishu docs PR #39389 can be reconciled after this broader core/docs direction is settled.

## Risks / Open Questions

- Applying the PR as written may change agents unbind semantics for existing channel-only bindings because bind, unbind, and add share parseBindingSpecs.
- The PR's defaultAccount wording can mislead users if it says omitted binding accountId follows channels.<channel>.defaultAccount while the routing implementation currently maps omitted accountId to the literal DEFAULT_ACCOUNT_ID bucket.
- The functional CLI behavior change needs targeted tests for multi-account auto-wildcard bind/add, explicit account preservation, single-account behavior, wildcard routing, and channel-only unbind compatibility.
- No supply-chain or CI security regression is visible in the PR file list; the main security concern is accidental routing/isolation broadening or failure from an untested binding-scope change.
