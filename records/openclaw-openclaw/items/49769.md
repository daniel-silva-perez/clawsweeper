## Summary

Keep open. Current main still does not implement this PR’s intended docs/type/CLI clarification, and the PR’s functional change still needs revision because it modifies shared bind/unbind parsing semantics rather than only binding creation.

## Best Possible Solution

Keep this PR open and request a focused revision: either make it docs/types-only with accurate routing wording, or scope any `accountId: "*"` default to binding creation paths only while preserving unbind compatibility. Add targeted tests for multi-account bind/add, explicit account preservation, single-account behavior, wildcard routing, and channel-only unbind compatibility, and update all examples consistently.

## Risks / Open Questions

- Functional risk in the PR as written: changing `resolveBindingAccountId` through shared `parseBindingSpecs` would affect both `agents bind` and `agents unbind`, so legacy channel-only bindings may become harder to remove by bare channel shorthand on multi-account channels.
- Accuracy risk in the PR docs/type wording: current routing maps omitted `accountId` to the literal default-account bucket rather than cleanly documenting all `channels.<channel>.defaultAccount` cases.
- Security review pass: the provided diff is limited to docs, a type annotation, and a CLI config-writing helper; no CI workflows, dependencies, lockfiles, install/release scripts, package publishing metadata, secrets handling, or downloaded artifacts are introduced.
