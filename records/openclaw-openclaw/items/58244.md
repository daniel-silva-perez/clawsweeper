## Summary

Keep open: current main still keeps idle command-queue lane entries indefinitely, and this PR targets a real leak path. The branch is useful, but it is stale against recent command-queue and cron-lane work and should be rebased or replaced with one small amendment for the current `cron-nested` lane.

## What This Changes

The PR adds pruning for idle default-concurrency command queue lanes after task completion, lane clearing, and all-lane reset, with regression tests and a test-only lane-count helper.

## Best Possible Solution

Land a narrow current-main fix that prunes only fully idle dynamic/default-concurrency lanes, preserves queued and active work, keeps configured concurrency, and treats every current `CommandLane` well-known lane, including `cron-nested`, as persistent.

## Reproduction Assessment

Yes. A focused unit check can enqueue completed work into many unique `session:*` or `auth-probe:*` lanes on current main and observe that `getCommandLaneSnapshots()` still contains the idle lanes because no production path deletes them.

## Solution Assessment

Yes, the proposed direction is the narrow maintainable fix for this leak. The implementation should be refreshed over current main and include `CommandLane.CronNested` in the persistent-lane coverage before merge.
