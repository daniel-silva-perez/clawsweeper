## Summary

Keep this PR open. Current main still has the command-queue lane Map lifetime behavior the PR targets: dynamic lanes are created on demand and remain in the module/global singleton Map after completion, clear, and reset paths. The PR is a focused implementation candidate, not superseded by current main, and the existing bot discussion gives useful review context rather than a close reason.

## Best Possible Solution

Keep this PR open and review it as the active implementation candidate. The best path is to update it against current main, preserve configured and well-known lanes including cron-nested where intended, add regression coverage for completion/clear/reset pruning and custom concurrency preservation, then run the targeted command-queue tests plus the changed gate before merge.
