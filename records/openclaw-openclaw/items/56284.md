## Summary

Keep open. Current main has useful restart-intent, drain, health-check, and restart-aborted session recovery pieces, but the Windows Scheduled Task restart path still calls the service-manager stop/run flow and explicitly terminates Gateway listeners with a very short grace window. The linked implementation PR is closed unmerged, so this still needs maintainer-owned Windows lifecycle follow-up rather than cleanup closure.

## What This Changes

Windows Scheduled Task `openclaw gateway restart` should drain active tasks, write restart recovery state, wait for health, and preserve or recover sessions instead of abruptly terminating the running Gateway.

## Best Possible Solution

Define one Windows Scheduled Task restart contract that coordinates with the running Gateway: write the appropriate restart/recovery state, let the process enter the existing drain path, avoid premature listener/taskkill cleanup, wait for healthy restart, and verify interrupted sessions are completed, resumed, or explicitly marked failed without silent loss.

## Reproduction Assessment

Unclear. The issue gives concrete Windows 11 Scheduled Task steps and current source shows a plausible forced-cleanup path, but a high-confidence reproduction still requires a live Windows run with active agent or cron work to observe drain and session recovery behavior.

## Solution Assessment

Unclear. Wiring the CLI directly to `deferGatewayRestartUntilIdle` and `writeRestartSentinel` may be useful context, but the safer design is likely to integrate the Scheduled Task service path with the existing gateway-owned restart intent, drain, sentinel, and health mechanisms rather than add a parallel Windows-only flow.
