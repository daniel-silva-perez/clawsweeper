## Summary

Keep open. Current main still has only partial restart-intent, drain, health-check, and main-session recovery wiring; the Windows Scheduled Task restart path still runs through `schtasks /End` plus listener termination, and there is no CLI/daemon restart-sentinel wiring or live Windows proof that active tasks drain before termination. The linked implementation PR is closed unmerged, so this is not safe to close as implemented.

## What This Changes

The issue requests that Windows Scheduled Task `openclaw gateway restart` wait for active tasks, preserve or recover session state, and return only after the restarted gateway is healthy.

## Best Possible Solution

Keep this issue open and define the Windows Scheduled Task restart contract explicitly: the CLI restart should coordinate with the running gateway to drain active tasks/runs, avoid duplicate `/Run` races, preserve recoverable session or continuation metadata, and wait for post-restart health. The implementation needs focused unit coverage around `runServiceRestart`, `restartScheduledTask`, restart intent consumption, and Windows handoff behavior, plus live Windows Scheduled Task proof before closure.

The direct Windows graceful-restart proposal in the closed PR is useful context, but current main already has restart-intent and drain machinery; the safer direction is to integrate the Scheduled Task CLI path with that gateway-managed drain/handoff instead of adding a parallel ad hoc restart flow.
