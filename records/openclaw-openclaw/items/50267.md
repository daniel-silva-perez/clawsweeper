## Summary

Keep this PR open. Current main still does not pass endpoint-aware OpenRouter attribution headers through the image-understanding direct `complete()` path, while the PR’s patch targets an old file path and uses a provider-only helper that bypasses current endpoint gating.

## Best Possible Solution

Rework this PR on current main to resolve image direct-completion headers with `resolveProviderRequestHeaders` or `resolveProviderRequestPolicyConfig` using `capability: "image"` and `transport: "media-understanding"`, pass those headers into `complete()`, and add focused tests for OpenRouter attribution plus custom-base-url non-injection. The broader direct-completion attempt in #57561 is closed unmerged, so this remains a useful fix path if updated.

## Risks / Open Questions

- The PR as written would apply enabled attribution based only on `model.provider`, which can send OpenAI/OpenAI-Codex hidden attribution headers to custom OpenAI-compatible image routes that current policy intentionally gates by endpoint.
- The PR targets a deleted/stale file path, so it needs a rebase to the current `src/media-understanding/image.ts` implementation.
- There is no current regression test proving OpenRouter image-tool requests get attribution headers while custom proxy routes do not.
- Security review: the diff is narrow and does not touch CI, dependencies, lockfiles, release scripts, or package metadata; the main security-sensitive concern is header injection policy, not supply-chain churn.
