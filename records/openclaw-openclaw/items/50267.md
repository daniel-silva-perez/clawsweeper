## Summary

Keep this PR open/reworkable. Current main still does not visibly pass endpoint-aware provider attribution headers through the image-understanding direct `complete()` path, so the intended OpenRouter free-tier behavior is not implemented on main. The PR is also stale against the moved current implementation and should be reworked to use the shared provider request policy rather than provider-only attribution lookup.

## Best Possible Solution

Rework this PR on current main to resolve direct image-completion headers through `resolveProviderRequestHeaders` or `resolveProviderRequestPolicyConfig` with `capability: "image"` and `transport: "media-understanding"`, pass those headers into `complete()`, and add focused tests for OpenRouter attribution plus custom-base-url non-injection. The broader direct-completion attempt in #57561 was closed unmerged, so this PR remains a useful place for the fix if it is updated.

## Risks / Open Questions

- The PR as written would apply attribution based only on `model.provider`, which can leak OpenAI/OpenAI-Codex hidden attribution headers to custom OpenAI-compatible image routes that current policy intentionally gates by endpoint.
- The PR targets an older file path that no longer exists on current main, so it needs a rebase and a different implementation point.
- There is no current regression test proving OpenRouter image-tool requests receive attribution headers while custom proxy routes do not.
