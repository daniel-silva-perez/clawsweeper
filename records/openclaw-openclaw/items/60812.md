## Summary

Keep this issue open. Current main has some shared agent-runner fallback text for provider/run failures and Telegram has channel-specific delivery-state fallback, but Discord, Slack, Signal, and WhatsApp still do not implement the requested parity path for delivery errors or no-delivered-output cases. The related same-author PR #60830 is still open and describes Telegram/agent work plus follow-up tracking here, not a merged implementation of this broader channel request.

## Best Possible Solution

Keep this issue open and implement a shared delivery-fallback helper or SDK-facing reply-runtime utility that Discord, Slack, Signal, WhatsApp, and LINE can use. It should track dispatcher delivery failures/skips, inspect queued/delivered/failed counts after dispatch settles, choose typed messages from the existing failover classification path, and add focused channel tests for provider error, empty response, delivery throw, and no-visible-output cases.
