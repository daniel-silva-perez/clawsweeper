## Summary

Keep this PR open. Current main still contains the exact `existsSync()` followed by `readFileSync()` patterns in the three security-sensitive paths named by the PR, so the PR is not obsolete or implemented on main. The proposed direction is plausible hardening, but it needs maintainer review around the intentional non-ENOENT behavior change and focused regression coverage before merge.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to rebase or reapply the focused hardening on current main, make the non-ENOENT behavior explicit in the PR description, and add targeted tests for ENOENT, parse errors, and non-ENOENT read failures across `device-auth-store`, `device-identity`, and `exec-approvals`. Maintainers should also decide whether the read path should use an existing verified-open helper rather than only collapsing `existsSync` plus `readFileSync`.

## Risks / Open Questions

- The PR has no regression tests in the provided test plan; tests should cover ENOENT-as-missing, non-ENOENT propagation for exec approvals/device auth, parse-error fallback, and device-identity regeneration semantics.
- The proposed `readFileSync` try/catch removes the existence-check race, but it is not a full symlink/hardlink/pinned-file hardening story. Maintainers should decide whether these state files need stronger `open`/`fstat`-style handling, especially since `src/infra/safe-open-sync.ts` already has a verified-open helper for stricter read boundaries.
- Changing exec approvals from broad fallback to throwing on EACCES/EMFILE/etc. is security-positive but user-visible; callers such as security audit, doctor security, and exec default resolution currently assume `loadExecApprovals()` returns defaults on read failure.
