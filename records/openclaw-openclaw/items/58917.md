## Summary

Keep open. Current main still does not display the response usage footer input side as prompt-token totals for cache-heavy runs, and #58875 was explicitly closed as superseded by this PR. The PR is useful but not merge-ready as-is: it is stale against current main, mixes unrelated delivery/test changes, and still has cache-accounting edge cases called out in review.

## What This Changes

The PR changes the response usage footer to display cached-token prompt totals, adds runReplyAgent regression coverage, and also carries unrelated Matrix, Telegram cron/outbound, Windows startup fallback, and provider fetch timeout changes.

## Best Possible Solution

Land a narrow footer-focused change that displays normalized prompt totals, preserves current cache and cost details, adds cached-token-heavy and double-count regression coverage, and splits unrelated delivery/test stabilization changes into separate PRs if they are still needed.

## Reproduction Assessment

Yes. A source-level reproduction is clear: on current main, passing `usage: { input: 7, cacheRead: 6993, output: 545 }` through the footer formatter labels input from `usage.input`, so the footer does not show `7.0k in` as requested.

## Solution Assessment

No. The core idea is right, but the current PR is not the best merge shape because it mixes unrelated changes and its cache-inclusive-input guard depends on optional `total`; a smaller formatter/test patch against current main is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.81

Full review comments:

- **[P2] Handle total-less cache-inclusive usage:** `src/auto-reply/reply/agent-runner-usage-line.ts:21-32`
  - body: `usage.total` is optional, so this guard only prevents double-counting when both `total` and `output` are present. If a provider or harness supplies `input` that already includes cached tokens plus `cacheRead` but omits `total`, the footer still displays `input + cacheRead + cacheWrite`; add a total-less regression or normalize that shape before formatting.
  - confidence: 0.82
- **[P2] Preserve the current cache breakdown suffix:** `src/auto-reply/reply/agent-runner-usage-line.ts:83`
  - body: Current main appends the explicit `cache ... cached / ... new` suffix when cache fields are present, but the PR branch's formatter return only emits input/output and estimated cost. Keep that cache detail while changing the input label, otherwise cache-heavy `/usage` footers lose the breakdown that current main already shows.
  - confidence: 0.76
