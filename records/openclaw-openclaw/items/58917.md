## Summary

Keep open. Current main still formats the response usage footer input label from raw `usage.input`, while the PR is trying to make cache-heavy turns show prompt-token totals in that main input position. The PR is useful prior work for #58875, but it is not implemented on main and the branch needs rework because it also carries unrelated Telegram/cron/provider-test changes plus review-noted correctness risks.

## Best Possible Solution

Keep this PR open as useful prior work, but require a focused rebase: change only the response usage footer path to display prompt-token totals for cached-token-heavy normalized usage, keep cost estimation on raw usage fields, add regression tests for the cache-heavy case and OpenAI-style no-double-count case, and split unrelated Telegram/cron/provider-test changes into separate PRs if still needed.
