## Summary

Keep open. Current main still has the Gemini CLI OAuth catch-only URL fallback, while this PR makes the URL visible before the best-effort browser launch, adds a focused regression test, has maintainer approval, and shows no blocking review or security findings.

## What This Changes

The PR prints the Gemini CLI OAuth authorization URL before attempting browser launch and adds a local-login test that asserts the URL is logged before `openUrl` runs.

## Best Possible Solution

Keep the fix in the Google plugin OAuth boundary: show the generated authorization URL before the browser opener runs, treat browser launch as best-effort after the URL is visible, and preserve the localhost callback/manual fallback behavior with the added ordering regression test.

## Reproduction Assessment

Yes. The linked Windows report gives concrete CLI reproduction steps, and current main statically reproduces the hidden-URL condition because `openUrl` can resolve false through the provider wrapper without entering the Google plugin catch block.

## Solution Assessment

Yes. Printing the URL before the best-effort opener is the narrowest maintainable fix; changing the core opener/provider auth contract to propagate a boolean could be useful later, but is broader than needed for this deadlock.
