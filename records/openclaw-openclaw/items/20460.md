## Summary

Current main still stores config backups beside the active config as `.bak` plus numbered `.bak.N` files. The requested dedicated `config-backup/` directory with datetime-based names is not implemented; the direct PR for this issue is closed unmerged, and the open related PRs cover narrower timestamp-suffix work rather than the full directory-plus-datetime request.

## Best Possible Solution

Keep this issue open as the broader config-backup ergonomics request unless maintainers explicitly consolidate it into #39923 or #33724. A complete fix should introduce a dedicated backup directory under the config directory, use deterministic datetime-based names, preserve bounded retention and owner-only permissions, handle timestamp collisions, keep or migrate legacy adjacent `.bak`/`.bak.N` files safely, update recovery/audit/doctor behavior, and add focused tests/docs.

## Repair Work Prompt

Implement the broader config-backup ergonomics request for this issue on current main. Move config backup files out of the active config directory root into a dedicated `config-backup/` directory under the config directory, and use deterministic UTC datetime-based names such as `openclaw.json.bak.2026-02-18T14-37-28.123Z` or an equivalent lexicographically sortable format. Preserve the current bounded retention intent, owner-only permissions, best-effort cleanup, and config write safety. Add collision handling for rapid writes, and define compatibility for existing adjacent `.bak`/`.bak.N` files so recovery, doctor, and audit paths do not silently lose usable backups. Update `maybeRecoverSuspiciousConfigRead` and sync recovery to restore from the newest valid backup in the new location while still considering legacy adjacent backups during migration. Update docs that mention `~/.openclaw/openclaw.json.bak`. Do not implement user-configurable backup paths or backup trigger policies; leave that broader customization to #33724. Coordinate with #39923, #39982, and #40392 because they cover narrower timestamp-only work in the same module. Changelog: add a user-facing fix/enhancement entry only if repo policy expects one for config backup behavior changes, with proper attribution to the reporter/contributors.
