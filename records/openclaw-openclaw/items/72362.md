Current main still has the reported misleading pinned-write helper path, and multiple open fixing PRs use closing syntax for this issue. It should remain open until maintainers choose and land one canonical fix.

## What This Changes

Request fs-safe write paths in Docker/slim environments without `python3` to either work without the Python helper or fail with an actionable missing-Python diagnostic and docs or doctor coverage.

## Best Possible Solution

Keep this issue open, choose a canonical fix from the linked PRs or an equivalent patch, and close only after main either removes the Python runtime dependency for write/copy paths or handles missing-helper spawn failures with clear diagnostics, regression tests, and Docker/docs/doctor follow-through if `python3` remains required.

Yes. The issue provides a direct Docker/Node reproduction, and current main statically still follows the same Linux write path: spawn the pinned Python helper, then normalize non-safe helper errors to the generic regular-file message.
