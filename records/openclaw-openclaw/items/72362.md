## Summary

Keep open. Current main still routes non-Windows fs-safe write and copy operations through the Python-backed pinned write helper and still normalizes helper spawn failures to the misleading regular-file error; the issue also has open closing PRs, so it should remain open until one canonical fix lands.

## What This Changes

The issue reports that Docker/slim installs without `python3` make fs-safe write consumers such as session-memory and workspace-only file tools fail with a misleading regular-file error.

## Best Possible Solution

Keep this issue open and land one canonical fs-safe fix that handles pinned write/copy helper spawn failures with regression coverage while preserving alias, symlink, hardlink, and post-write identity checks. If Docker images intentionally omit `python3`, the public Docker docs and diagnostics should also make that environment constraint explicit.

## Reproduction Assessment

Yes. The issue gives direct Docker and Node reproduction steps, and current main still has the deterministic source path that falls back to spawning literal `python3` and then maps that failure to the reported regular-file error.

## Solution Assessment

Unclear which open PR is the best final patch. A narrow write/copy spawn-failure fallback that mirrors the existing mkdir/remove fallback appears more complete than a message-only diagnostic, but the maintainer should choose among the existing candidates.
