## Summary

Keep this open. Current main still routes non-Windows fs-safe write and copy operations through the Python pinned-write helper and generic write-error normalization, so the reported Docker failure is still present; the issue is also already paired with open fixing PRs that maintainers should review instead of closing or duplicating.

## What This Changes

The issue asks OpenClaw to make fs-safe write/copy paths work, or at least fail with a clear diagnostic, when the Python pinned-write helper cannot spawn in slim Docker images without `python3`.

## Best Possible Solution

Land one canonical fix that handles pinned-write helper spawn failures for write and copy paths with regression tests, keeps the existing alias/symlink/hardlink boundary checks intact, and updates Docker docs or doctor diagnostics if `python3` remains an expected runtime dependency.

## Reproduction Assessment

Yes. The issue provides a direct Docker/Node reproduction, and current main still contains the Python-backed write/copy path plus generic normalization that explains the reported `spawn python3 ENOENT` being hidden behind the regular-file error.

## Solution Assessment

Unclear which open PR is the best final patch. A narrow fallback for helper spawn failures with write/copy tests appears more complete than a message-only fix, but maintainer review should choose among the existing candidates.
