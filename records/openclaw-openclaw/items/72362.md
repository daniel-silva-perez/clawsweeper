Keep this open. Current main still has the reported fs-safe failure mode, and three open unmerged PRs already reference this issue with proposed fixes, so the next step is maintainer review of the existing candidates rather than cleanup closure or another repair job.

## What This Changes

The issue requests that Python-backed fs-safe write/copy operations either work without python3 in slim Docker images or surface/document a clear missing-python dependency instead of the misleading regular-file error.

## Best Possible Solution

Land one canonical fs-safe fix that handles pinned write/copy helper spawn failures without weakening alias, symlink, hardlink, and post-write identity checks; add regression coverage for both write and copy paths, and update Docker docs or diagnostics if python3 remains an expected runtime dependency anywhere.

## Reproduction Assessment

Yes. The issue provides a direct Docker/Node reproduction, and current-main source inspection confirms the non-Windows write/copy path still depends on a Python helper and normalizes spawn failures to the reported regular-file error.

## Solution Assessment

Unclear which open PR is the final best patch. A focused spawn-failure fallback that mirrors the existing mkdir/remove behavior appears more complete than a message-only change because it restores writes in slim images while preserving security checks.
