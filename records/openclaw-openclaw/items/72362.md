## Summary

Keep this open. Current `main` still has the reported fs-safe failure shape: non-Windows write/copy operations go through the pinned Python write helper and generic error normalization, while the existing spawn-failure fallback is only implemented for mkdir/remove. The linked open PRs are the right place to resolve this rather than closing the report now.

## What This Changes

The issue asks OpenClaw to make fs-safe write/copy paths work or fail clearly in slim Docker images where `python3` is missing from `PATH`.

## Best Possible Solution

Land one canonical fix that handles pinned write/copy helper spawn failures on non-Windows, adds focused regression tests for write, copy, and non-spawn fail-closed behavior, and either removes the Docker runtime dependency on `python3` for this path or documents/diagnoses it clearly if it remains required elsewhere.

## Reproduction Assessment

Yes. The issue provides a direct Docker/Node reproduction, current main still contains the same write/copy helper path and generic normalization, and a local Node spawn probe confirms missing executables do not take the current helper fallback branch.

## Solution Assessment

Unclear which existing PR is the best final patch. A narrow write/copy spawn-failure fallback with fail-closed tests appears more complete than a message-only fix, but maintainers should review the active candidates and choose one canonical implementation.
