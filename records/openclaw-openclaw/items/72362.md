## Summary

Keep this open. Current main still sends non-Windows write and copy operations through the Python-backed pinned write helper and still maps helper failures to the misleading regular-file error; the issue is also already paired with open fixing PRs that use closing syntax, so it should remain open until one canonical fix lands or those PRs are closed.

## What This Changes

The issue requests that fs-safe writes/copies and consumers such as session-memory and workspace-only file tools work or fail clearly in slim Docker images without `python3` instead of surfacing a misleading regular-file error.

## Best Possible Solution

Land one canonical fix for pinned write/copy helper spawn failures, with regression coverage for write and copy behavior while preserving alias, symlink, hardlink, and post-write identity checks. If any Docker path still expects `python3`, document it in the public Docker install flow or surface it through doctor/diagnostics.

## Reproduction Assessment

Yes. The issue gives a direct Docker/Node reproduction, and current main still contains the deterministic source path that falls back to spawning literal `python3` and then normalizes that failure to the reported regular-file error.

## Solution Assessment

Unclear which open PR is the best final patch. A narrow spawn-failure fallback for write/copy paths with regression tests appears to solve the user-visible failure more completely than a message-only diagnostic, but maintainers should choose among the existing candidates.
