## Summary

Keep this open. Current `main` still sends non-Windows write and copy operations through the Python-backed pinned write helper and still normalizes helper failures to the misleading regular-file error, and the issue is already paired with open fixing PRs that use closing syntax.

## What This Changes

The issue asks OpenClaw to stop failing or at least clearly diagnose fs-safe writes in slim Docker images where `python3` is missing from `PATH`.

## Best Possible Solution

Land one canonical fix for pinned write/copy helper spawn failures, with regression tests for write and copy paths, while preserving alias, symlink, hardlink, and post-write identity checks. If `python3` remains expected in any Docker path, update Docker docs or doctor diagnostics to make that dependency explicit.

## Reproduction Assessment

Yes. The issue provides a direct Docker/Node reproduction, and current `main` still contains the Python-backed write/copy path plus generic normalization that explains the reported `spawn python3 ENOENT` being hidden behind the regular-file error.

## Solution Assessment

Unclear which open PR is the best final patch. A narrow fallback for helper spawn failures with write/copy regression tests appears more complete than a message-only diagnostic fix, but maintainer review should choose among the existing candidates.
