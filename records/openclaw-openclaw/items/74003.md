## Summary

Keep this open. Current main still matches the reported bug: `before_tool_call` plugin approvals are requested without turn-source routing metadata, while the gateway already supports that metadata and fails closed with `decision: null` when no approval route is available. The linked fix PR #74150 is closed and unmerged, and its params-sourced route approach left a trust-boundary concern unresolved.

## What This Changes

The issue requests forwarding trusted originating-channel route metadata into `plugin.approval.request` for `before_tool_call` plugin approval prompts.

## Best Possible Solution

Thread trusted turn-source metadata through the agent hook context, populate it from the same runtime route inputs used by exec approvals, forward it from `requestPluginToolApproval` to `plugin.approval.request`, and add regression coverage for routed and no-route plugin approvals.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: a `before_tool_call` hook returning `requireApproval` currently sends `plugin.approval.request` without route fields, causing the gateway no-route branch to return `decision: null` and the agent to surface the reported error.

## Solution Assessment

Yes, if implemented through trusted runtime context. The issue's original `args.params` idea and the closed PR's `baseParams` approach are not the best fix because those values can originate from model/tool input.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/74003. Current main's `before_tool_call` approval path calls `plugin.approval.request` without route metadata, so channel-originated plugin approvals can fail with `Plugin approval unavailable (no approval route)`. Implement the fix by adding trusted turn-source fields to `HookContext`, populating them from runtime route options in `createOpenClawCodingTools` using the same trusted route semantics as exec approvals (`messageProvider`, `currentChannelId`/established target, `agentAccountId`, `currentThreadTs`), and forwarding only those context fields to `requestPluginToolApproval`. Do not read `turnSource*` from model-controlled tool params, `baseParams`, or hook-returned params. Add regression tests proving routed plugin approvals include the route fields and no-route requests still fail closed. Include a user-facing `CHANGELOG.md` fix entry thanking @choury. Keep the change scoped to the agent hook/approval path; do not change gateway approval policy or plugin approval schemas unless tests prove an additive contract gap.
