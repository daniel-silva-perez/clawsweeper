## Summary

Current main still matches the reported routing bug: before_tool_call plugin approvals call `plugin.approval.request` without trusted turn-source metadata, while the gateway already depends on those fields to keep same-chat approval requests routable. The linked fix PR is closed and unmerged, and the reporter's correction about avoiding `args.params` is valid, so this should stay open as a narrow repair candidate.

## What This Changes

The issue asks OpenClaw to forward trusted originating-channel route metadata for before_tool_call plugin approval requests so approval prompts can reach the channel/chat that triggered the tool call.

## Best Possible Solution

Add a runtime-only route context seam for before_tool_call approvals, populate it from the trusted message-channel/account/target/thread values already available to the agent run, forward those fields to `plugin.approval.request`, and keep the gateway no-route behavior fail-closed.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: a channel-originated before_tool_call `requireApproval` currently calls `plugin.approval.request` without route fields, the gateway no-route path returns `decision: null`, and the agent surfaces the reported error.

## Solution Assessment

Yes, but only through trusted runtime context. The originally suggested `args.params`/`baseParams` direction is not the best fix because those values are tool-call input rather than a trusted route source.
