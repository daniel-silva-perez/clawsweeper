## Summary

Current main still matches the reported bug: the before_tool_call approval caller does not pass any turn-source routing fields to plugin.approval.request, while the gateway already depends on those fields to keep routed approvals pending instead of returning the reported no-route failure. The related #74150 fix PR is closed and unmerged, so it does not supersede this issue.

## What This Changes

The issue requests that before_tool_call plugin approval requests include originating turn-source routing metadata so channel approval prompts can be delivered.

## Best Possible Solution

Implement a narrow agent-side propagation fix: add trusted turn-source fields to the before_tool_call HookContext from createOpenClawCodingTools runtime options, forward them to plugin.approval.request, and add regression tests proving routed plugin approvals include the four route fields while no-route requests still fail closed.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: trigger a before_tool_call requireApproval from a channel-backed run, and current main sends plugin.approval.request without route fields, which the gateway no-route path turns into decision:null and the reported error.

## Solution Assessment

Yes, if implemented through trusted hook context. The original args.params/baseParams direction is not the best fix because those values are tool/model inputs; the maintainable fix is to carry messageProvider/message target metadata from the agent runtime into HookContext and forward it from there.
