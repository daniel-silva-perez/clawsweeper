## Summary

Current main still matches the reported approval-routing bug: before_tool_call approval requests are sent without trusted turn-source metadata even though the gateway already accepts and relies on those fields to keep routed approvals pending. The related fix PR #74150 is closed and unmerged, so this should remain open and is a narrow repair candidate.

## What This Changes

The issue asks OpenClaw to forward trusted turn-source routing metadata from before_tool_call plugin approval requests so the originating chat channel can receive the approval prompt.

## Best Possible Solution

Add runtime-only turn-source fields to HookContext, populate them from the same trusted channel/account/thread values used by exec approvals, forward them to plugin.approval.request, and cover both routed and no-route plugin approval cases with focused tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: a before_tool_call requireApproval path calls plugin.approval.request without route fields, the gateway no-route branch returns decision:null, and the helper converts that to the reported error.

## Solution Assessment

Yes, but the issue's original args.params direction should not be used. The narrow maintainable fix is trusted runtime context propagation through the tool wrapper and approval helper.

## Repair Work Prompt

Fix the before_tool_call plugin approval routing bug tracked at https://github.com/openclaw/openclaw/issues/74003. Current main sends plugin.approval.request without turn-source route metadata, so channel-backed plugin approvals can fail with 'Plugin approval unavailable (no approval route)'. Implement a narrow trusted-runtime fix: add turnSourceChannel, turnSourceTo, turnSourceAccountId, and turnSourceThreadId to the before_tool_call HookContext or equivalent runtime-only seam; populate them from createOpenClawCodingTools runtime route options using the same semantics as exec approval routing; forward those fields to plugin.approval.request. Do not read these values from args.params, baseParams, model-generated tool input, or plugin-controlled data. Add regression tests proving routed before_tool_call requireApproval sends the route fields, no-route requests still fail closed, and createOpenClawCodingTools passes the trusted route context into wrapped tools. Preserve existing plugin approval gateway behavior and avoid unrelated refactors. Add a CHANGELOG.md fix entry credited to @choury if the repair changes user-facing behavior.
