## Summary

Current main still matches the reported bug: before_tool_call plugin approvals are requested without trusted turn-source routing metadata, while the gateway already requires that route to keep same-chat/plugin approval requests pending. The linked repair PR is closed and unmerged, and the reporter's correction that model/tool params are not a trusted route source is important, so this should stay open as a narrow repair candidate.

## What This Changes

The issue reports that before_tool_call requireApproval plugin approvals do not receive originating channel/account/target routing metadata, causing plugin.approval.request to fail with no approval route.

## Best Possible Solution

Add trusted runtime route context for before_tool_call approvals, populate it from createOpenClawCodingTools' already-resolved channel/account/target/thread options, forward those fields to plugin.approval.request, and keep the gateway's no-route path fail-closed with regression tests for routed and unrouted plugin approvals.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: a channel-originated before_tool_call requireApproval call reaches requestPluginToolApproval without turnSource* fields, and the gateway no-route branch returns decision:null, which the agent maps to the reported error.

## Solution Assessment

Yes, if the fix uses trusted runtime context. The args.params/baseParams approach from the original proposal is not the best solution because those values are model/tool-call inputs rather than an authorization route source.
