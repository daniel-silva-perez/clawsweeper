## Summary

Keep this open. Current main still omits trusted turn-source routing metadata from before_tool_call plugin approval requests, the gateway still returns the reported no-route null decision when no route is present, and the only linked repair PR is closed unmerged after review identified an unsafe model/tool-params route source.

## What This Changes

The issue requests that before_tool_call plugin approvals carry trusted turn-source route metadata into plugin.approval.request so approval prompts can be routed back to the originating chat or channel.

## Best Possible Solution

Add trusted route context for before_tool_call approvals, populate it from createOpenClawCodingTools' already-resolved channel/account/target/thread inputs, forward only those trusted fields to plugin.approval.request, and keep the gateway's no-route path fail-closed with routed and unrouted regression coverage.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: a channel-originated before_tool_call requireApproval call reaches requestPluginToolApproval without turnSource* fields, then the gateway no-route branch returns decision:null and the agent maps that to the reported error.

## Solution Assessment

Yes, the best direction is a trusted HookContext or equivalent runtime-context plumb. The original args.params/baseParams approach is not the best fix because those values are model/tool-call boundary data and could spoof approval routing.
