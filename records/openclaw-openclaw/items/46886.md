## Summary

Keep open. Current main still drops cron wake targeting at multiple boundaries: the cron tool forwards only mode/text, the Gateway wake handler narrows to mode/text, the protocol type does not declare target fields, and CronService.wake queues an unscoped system event and heartbeat. The existing lower Gateway cron adapter already has agentId/sessionKey resolution when those options are supplied, and open PR #70268 is adjacent hook-routing work rather than a fix for this direct cron tool/Gateway wake path.

## What This Changes

The issue requests optional agentId/sessionKey support for the cron tool wake action so direct wake events can target a specific agent or session instead of falling back to the default main agent.

## Best Possible Solution

Add an additive target contract for direct wake calls and thread optional agentId/sessionKey through the cron tool, Gateway WakeParams/handler, CronServiceContract, CronService.wake, ops.wakeNow, and timer.wake, reusing the existing Gateway cron target resolver while preserving default no-target behavior.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: call the cron tool wake action with agentId/sessionKey and current main will drop those fields before enqueueing an unscoped system event and heartbeat.

## Solution Assessment

Yes. The narrowest maintainable solution is targeted plumbing through the existing wake contracts and resolver, plus regression tests at the tool, protocol, Gateway handler, adapter, and CronService boundaries.
