## Summary

Current main still does not implement the requested cron wake targeting. The agent-facing `cron` tool wake action sends only `mode` and `text`, the Gateway wake handler narrows the request to those fields, and `CronService.wake` enqueues/wakes without `agentId` or `sessionKey`. Normal scheduled main-session cron execution and the lower Gateway cron adapter already have target-routing seams, so this remains a focused plumbing bug rather than a missing product concept.

## Best Possible Solution

Keep this issue open and land a focused fix that threads optional `agentId` and `sessionKey` through the cron tool wake schema/execution path, Gateway wake params/handler, `CronServiceContract`/`CronService.wake`, and the timer wake implementation, reusing the existing Gateway cron target-resolution adapter and adding regression tests for both the agent-facing tool and direct Gateway wake method.
