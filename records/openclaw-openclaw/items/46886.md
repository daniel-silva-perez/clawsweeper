## Summary

Keep open. Current main still does not support targeted `cron` wake calls: the agent tool exposes/sends only `mode` and `text`, the gateway wake method forwards only those fields, and `CronService.wake()` enqueues and requests heartbeat with no `agentId` or `sessionKey`. Normal scheduled cron job execution is already target-aware, so the reported gap remains specific and actionable.

## Best Possible Solution

Keep this issue open and implement a narrow targeted cron wake path: add optional `agentId` and `sessionKey` to the wake tool/protocol/service contract, thread them through `cronHandlers.wake`, `CronService.wake`, `ops.wakeNow`, and `timer.wake`, and resolve one canonical target for both `enqueueSystemEvent` and `requestHeartbeatNow`. Add focused regression tests for agentId-only, sessionKey-only, both fields, blank/invalid target handling, and the existing untargeted fallback.
