## Summary

Keep open. Current main still drops wake target fields on the agent-facing cron wake path: the tool sends only mode/text, the Gateway handler forwards only mode/text, and CronService.wake enqueues/wakes without target options. The lower Gateway cron adapter already supports agentId/sessionKey once they are supplied, so this remains a narrow plumbing bug.

## What This Changes

The issue asks for the cron tool's wake action to accept optional agentId/sessionKey and route wake events to the requested agent/session instead of always falling back to the default agent.

## Best Possible Solution

Land a focused additive fix that threads optional agentId and sessionKey through the cron tool wake schema/execution path, Gateway wake params/handler, CronServiceContract/CronService.wake/ops/timer, and reuses the existing Gateway cron target resolver with regression coverage.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: call the cron tool wake action with agentId/sessionKey and the current path drops those fields before CronService.wake, which then queues an unscoped system event and heartbeat.

## Solution Assessment

Yes. The narrow maintainable solution is to plumb optional target fields through the existing wake path and reuse the lower Gateway cron routing adapter, while preserving default-agent behavior when no target is supplied.
