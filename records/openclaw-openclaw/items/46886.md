## Summary

Keep open. Current main still drops direct cron wake targeting before the request reaches cron execution, even though scheduled cron jobs and the lower Gateway cron adapter already have targeted agent/session plumbing.

## What This Changes

The issue asks for the cron wake action to accept optional agentId and sessionKey so direct wake events can target a non-default agent/session.

## Best Possible Solution

Add an additive target contract for direct wake calls and thread optional agentId/sessionKey through the cron tool, Gateway WakeParams/handler, CronServiceContract, CronService.wake, ops.wakeNow, and timer.wake, reusing the existing Gateway cron target resolver while preserving default no-target behavior.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: call the cron tool wake action with agentId/sessionKey and current main drops those fields before enqueueing an unscoped system event and heartbeat.

## Solution Assessment

Yes. The narrowest maintainable fix is targeted plumbing through the existing wake contracts and resolver, with regression tests at the tool, protocol, Gateway handler, adapter, and CronService boundaries.
