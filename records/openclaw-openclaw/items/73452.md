## Summary

Keep this PR open. Current main still hardcodes `disableBlockStreaming: true` on the Mattermost draft-preview reply path, while the sibling Mattermost reply paths and slash path derive the option from `account.blockStreaming`; the PR is a focused implementation candidate for the Mattermost streaming-config part of the linked bug, not work already implemented on main.

## What This Changes

The PR changes the Mattermost draft-preview dispatch path to derive `replyOptions.disableBlockStreaming` from `account.blockStreaming` instead of hardcoding it to `true`.

## Best Possible Solution

Land the one-line Mattermost fix after normal maintainer review, ideally with a focused assertion in `extensions/mattermost/src/mattermost/monitor.test.ts` or the existing inbound monitor harness that pins `account.blockStreaming: true` to `disableBlockStreaming: false` on the draft-preview path. Keep the broader linked agent-to-agent routing report open for separate trust-model and routing design work.

## Reproduction Assessment

Yes. On current main, the draft-preview inbound path can be reproduced by inspecting or mocking `dispatchReplyFromConfig`: it always receives `disableBlockStreaming: true` at `extensions/mattermost/src/mattermost/monitor.ts:1788`, which core resolves to block streaming off regardless of `account.blockStreaming`.

## Solution Assessment

Yes, with a validation caveat. The proposed code change is the narrowest maintainable fix for the Mattermost streaming-config mismatch because it copies the already-used sibling mapping; a regression assertion and maintainer confirmation of preview-vs-block behavior would make it merge-ready.
