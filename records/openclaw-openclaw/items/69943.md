## Summary

Keep this open. Current main still persists session-memory transcript text without stripping chat-template tokens, raw tool-call XML, role-marker lines, or NO_REPLY-only content, and two open unmerged PRs already reference this issue with closing syntax, so those PRs are implementation candidates rather than a close reason.

## What This Changes

The issue requests sanitizing the bundled session-memory hook so malformed raw model output is not persisted into memory files and re-injected across later `/new` or `/reset` sessions.

## Best Possible Solution

Land a narrow session-memory fix that sanitizes persisted transcript content at the transcript/read and/or handler/write boundary, with tests proving raw model special tokens, tool-call XML, bare role markers, and NO_REPLY-only malformed turns cannot be saved and re-injected. Keep the provider parser race out of scope, and handle broader historical-context framing in the adjacent tracker if needed.

## Reproduction Assessment

Yes. A high-confidence synthetic reproduction is available by putting `<|im_end|>`, raw `<tool_call>...</tool_call>`, or bare role markers in an assistant message in a session JSONL file; current `getRecentSessionContent` will return that text unchanged and the handler will write it under `## Conversation Summary`.

## Solution Assessment

Yes. Sanitizing at the session-memory persistence boundary is the narrowest maintainable fix for this issue because the hook is the component that converts prior transcript text into future prompt context; the open PRs should be reviewed for the exact artifact coverage and tests.
