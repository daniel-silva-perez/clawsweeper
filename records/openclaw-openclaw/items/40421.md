## Summary

Keep this PR open. Current main still does not implement Telegram bilingual waiting tips or the proposed `messages.waitingTips` config, so the contribution is not obsolete or implemented; however, the branch is stale, targets a removed Telegram path, and contains blocking correctness issues that need author or maintainer follow-up before merge.

## What This Changes

The PR adds a waiting-tip controller, `messages.waitingTips` config/schema/help labels, unit tests, and Telegram dispatch hooks to send and later delete a bilingual tip while the assistant is thinking.

## Best Possible Solution

Rebase the useful idea onto the current bundled Telegram plugin layout, decide whether OpenClaw wants the full bilingual-tip feature or a narrower processing indicator, keep any generic controller behind a documented channel/plugin-safe seam, fix the send/cancel cleanup race, remove or safely implement custom tip loading, and add focused Telegram dispatch coverage for fast streaming, cleanup, failures, topics, and superseded turns.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve the tip handle when cleanup races send:** `src/channels/waiting-tips.ts:212-216`
  - body: `cleanup()` calls `cancel()` before awaiting an in-flight `sendTip()`. Because the send path only stores `handle` when `!cancelled`, a Telegram send that finishes during cleanup can post a tip but leave no handle for deletion, so the temporary message stays in chat despite `deleteAfterReply` being enabled.
  - confidence: 0.92
- **[P2] Remove or implement customTipsPath before exposing it:** `src/config/schema.help.ts:1449-1450`
  - body: The patch documents and validates `messages.waitingTips.customTipsPath`, but no code reads that path or passes loaded tips into `createWaitingTipController`. Operators could configure the field and see no effect, so either wire the loader safely or drop the field until it exists.
  - confidence: 0.9
