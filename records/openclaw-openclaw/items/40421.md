## Summary

Keep open because current main still does not implement the waiting-tip behavior, and the proposed patch is stale relative to the current Telegram plugin layout and has blocking review findings in its new lifecycle/config code.

## What This Changes

The PR adds a configurable bilingual waiting-tip controller, shared messages config/schema/help labels, tests, and Telegram dispatch hooks intended to send and later delete a tip while the assistant is thinking.

## Best Possible Solution

Rebase the useful idea onto the current bundled Telegram plugin, decide the intended opt-in UX/config shape, keep any shared helper behind a plugin-safe seam or local plugin helper, fix the send/cleanup race, remove or safely implement custom tip loading, and add focused Telegram dispatch/config/docs coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve the tip handle when cleanup races send:** `src/channels/waiting-tips.ts:212-216`
  - body: `cleanup()` calls `cancel()` before awaiting an in-flight `sendTip()`. If Telegram sends the tip after cancellation, the `!cancelled` guard prevents storing the message id, so `deleteAfterReply` cannot delete the visible tip.
  - confidence: 0.92
- **[P2] Remove or implement customTipsPath before exposing it:** `src/config/schema.help.ts:1449-1450`
  - body: The patch documents and validates `messages.waitingTips.customTipsPath`, but no code reads that path or passes loaded tips into `createWaitingTipController`, so operators can configure it and see no effect.
  - confidence: 0.9
