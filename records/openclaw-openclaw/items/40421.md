## Summary

Keep open. Current main does not implement the waiting-tip feature, and the PR is not merge-ready because it targets the removed legacy Telegram path, has a send/delete race, and exposes a custom tips config option that the runtime never honors. The related #6946 is still open and narrower, so this needs maintainer product direction rather than cleanup closure.

## What This Changes

The PR adds a shared waiting-tip controller, tests, messages.waitingTips config/schema/help/labels, and Telegram dispatch wiring to show and delete bilingual tips while replies are being generated.

## Best Possible Solution

Decide the Telegram waiting-feedback UX against #6946, then port any accepted implementation into extensions/telegram with honest typed config, race-safe send/delete cleanup, focused dispatch/config regression coverage, and docs if user-facing configuration ships.

## Reproduction Assessment

Yes. The main mismatch is reproducible by checking that src/telegram is absent and the active dispatcher is extensions/telegram/src/bot-message-dispatch.ts; the cleanup race is reproducible by letting sendTip resolve after cancel/cleanup starts, which leaves no stored handle in the PR code.

## Solution Assessment

No. The current PR is useful source work but is not the best merge path as-is; the safer path is a maintainer-approved Telegram plugin implementation that either narrows to #6946 or deliberately ships bilingual tips with complete config/runtime behavior.
