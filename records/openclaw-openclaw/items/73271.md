## Summary

Close. Current main already persists gateway-owned session state and transcripts by default, reuses fresh stored session IDs, and includes restart-aborted main-session recovery. The broader arbitrary task checkpoint/reporting gap remains tracked separately by #42816, so this session-persistence request is implemented well enough without adding a new per-agent opt-in switch.

## What This Changes

Request optional per-agent session persistence so conversation history, decisions, and in-progress session context are saved and restored after Gateway restarts.

## Best Possible Solution

Keep the existing default Gateway-owned session store and transcript persistence. Use #42816 as the canonical place for any future durable task checkpoint/report loop, rather than adding a redundant `agents.<id>.sessionPersistence` boolean for normal session history.

## Reproduction Assessment

Yes for the claimed behavior check: inspect `sessions.json`, the per-session JSONL transcript, and the next run’s reused session key before and after a Gateway restart or via `openclaw sessions`. Current source/docs/tests point to the reported stateless behavior not reproducing on main for normal session history.

## Solution Assessment

Yes. Default persistence of session store and transcripts is narrower and more maintainable than adding an opt-in config flag; the only meaningful remaining scope is task-level checkpointing, which belongs in #42816.
