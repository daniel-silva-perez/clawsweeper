## Summary

Keep this PR open, but it should not merge as-is. Current main still lacks the proposed rescue watchdog cron payload and runner, so the work is not obsolete; however the branch is large, security-sensitive, and currently misses current-main launchd restart behavior that clears persisted disabled state before restarting.

## What This Changes

The PR branch adds LaunchAgent/service hardening, abortable managed-service restart paths, a new isolated `rescueWatchdog` cron payload and runner, gateway cron wiring, probe/client adjustments, and focused tests for the rescue watchdog flow.

## Best Possible Solution

Rebase the implementation onto current main, preserve existing launchd restart semantics, and land either a reviewed core slice or smaller maintainer-owned slices that add the `rescueWatchdog` payload contract, bounded runner, gateway cron integration, service-repair support, and focused tests. Coordinate the remaining notification and broader self-healing product questions with #53684 and #55347 rather than folding everything into this branch.

## Reproduction Assessment

Yes for the concrete review finding: current main has source and test coverage requiring `launchctl enable` before launchd restart, while the PR head restart path lacks that call. For the overall feature request, reproduction is not applicable; the current-main gap is confirmed by source inspection of the cron payload, schema, executor, and gateway cron wiring.

## Solution Assessment

No, not as-is. The core cron/service approach is plausible for the requested rescue watchdog, but the branch needs rebase and correction to preserve current launchd restart behavior before it can be considered the maintainable solution.
