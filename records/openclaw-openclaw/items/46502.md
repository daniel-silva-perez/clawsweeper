## Summary

Keep this PR open. Current main still lacks the rescueWatchdog cron payload and runner, so the feature is not implemented or obsolete, but the branch is stale against current launchd restart behavior and would drop the current enable-before-kickstart restart semantic.

## What This Changes

The branch adds LaunchAgent and managed-service hardening, abortable restart paths, a new isolated rescueWatchdog cron payload and runner, gateway cron wiring, gateway probe/client adjustments, and focused tests for the watchdog flow.

## Best Possible Solution

Rebase the implementation onto current main, preserve current launchd restart semantics including launchctl enable before direct kickstart, and then land a reviewed core slice or smaller maintainer-owned slices for the rescueWatchdog payload contract, bounded runner, gateway cron integration, service repair support, and tests. Coordinate broader notification and recovery workflow scope with #53684, #53716, #55347, and the external watchdog alternative in #40791.

Yes for the concrete review finding: current main source and tests require launchctl enable before kickstart, while the PR branch goes straight to kickstart on the direct restart path. ## Solution Assessment

No, not as-is. The isolated cron/service approach is plausible for the requested watchdog, but the branch must be rebased and must preserve current launchd restart behavior before it is maintainable.
