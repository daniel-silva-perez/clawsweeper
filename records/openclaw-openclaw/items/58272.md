## Summary

Keep open. Current main still lacks source or test evidence that the read tool can recover when a model-provided path has inserted spaces at CJK/ASCII boundaries; the pinned upstream resolver handles several macOS path variants, but not the reported `测试 test .pdf` to `测试test.pdf` mutation. The report is recent, specific, and narrow enough for a focused repair.

## What This Changes

The issue reports ENOENT failures when reading macOS paths whose directory or filename mixes CJK and ASCII characters, especially when the tool-call path gains spaces at the CJK/ASCII boundary.

## Best Possible Solution

Add a bounded read-tool fallback that, after the exact path and existing upstream macOS variants fail, retries only CJK/ASCII-boundary whitespace-compacted candidates that exist under the same workspace/sandbox guard. Cover both mixed-directory and mixed-filename cases with regression tests and avoid changing write/edit/exec semantics.

## Reproduction Assessment

Yes for the observable failure shape: if the model/tool call passes a path like `测试 test .pdf` while the filesystem contains `测试test.pdf`, current main forwards that string to the base read resolver and the pinned resolver does not try the compacted CJK/ASCII-boundary variant. A live macOS run was not performed in this read-only review.

## Solution Assessment

No proposed patch is present, but the likely best fix is narrow: repair only read-path resolution after an exact miss, with existing workspace/sandbox checks still enforced. Closing as stale or treating this solely as model behavior would leave a concrete, user-visible ENOENT recovery gap.
