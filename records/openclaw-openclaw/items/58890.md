## Summary

Keep open. Current main has adjacent macOS LaunchAgent supervision hardening, but it does not address this exact auto-update handoff path: the gateway-owned auto-update still uses piped output, `updateCommand` still refreshes the service environment before spawning the detached restart helper, and `gateway install --force` still reaches a LaunchAgent `bootout` before bootstrap.

## What This Changes

The issue asks OpenClaw to make macOS auto-update survive a gateway-owned update subprocess when launchd bootout removes the parent gateway before restart handoff completes.

## Best Possible Solution

Make the package auto-update path establish an independent restart/recovery handoff before any service refresh can boot out the running LaunchAgent, and consider detaching or otherwise hardening gateway-owned auto-update subprocess output so parent death cannot crash the child. Preserve the existing split-brain restart repair behavior and cover the exact ordering with regression tests.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction path remains: gateway startup runs `runAutoUpdateCommand`, which pipes output through `runCommandWithTimeout`; the update command then runs `refreshGatewayServiceEnv()` before `runRestartScript()`, and the refresh reaches LaunchAgent `bootout`.

## Solution Assessment

No. The current main solution is only adjacent: #72616 improves status/doctor/restart recovery for unloaded LaunchAgents, but the auto-update path can still boot out the parent before the detached update restart helper is spawned.
