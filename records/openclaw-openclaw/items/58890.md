## Summary

Keep open. Current main has related LaunchAgent supervision hardening, but it does not close the specific auto-update handoff bug: the auto-update child is still launched through the generic piped command runner, the update path still refreshes service metadata before launching the prepared restart script, and macOS `gateway install --force` still activates with `bootout` before `bootstrap`.

## What This Changes

The issue reports that macOS auto-update can crash before restart handoff when the update subprocess is launched from the gateway with piped stdout/stderr and then `gateway install --force` boots out the parent LaunchAgent.

## Best Possible Solution

Make the auto-update restart handoff independent before any launchd operation can remove the current service, either by staging service metadata without activation in the update path or by launching a detached recovery helper before `gateway install --force` can call `bootout`.

## Reproduction Assessment

Yes. The issue has a high-confidence source-level reproduction path: run a macOS LaunchAgent-managed gateway with auto-update due, and the current code still executes the updater as a piped child and refreshes service metadata through `gateway install --force` before launching the restart script.

## Solution Assessment

No. Current main's split-brain LaunchAgent repair is useful related hardening, but it is not the narrowest complete fix for this issue because the unsafe update ordering and install activation path remain.
