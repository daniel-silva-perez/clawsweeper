## Summary

Keep open. Current main has important mitigations around macOS LaunchAgent recovery and package-update safety, but it does not yet implement the specific safe auto-update handoff described here. The Gateway auto-updater still launches `openclaw update --yes --channel ... --json` through the generic command runner, stdout/stderr are still piped, the update path still refreshes service metadata before spawning the detached restart script, and `gateway install --force` still reaches a LaunchAgent `bootout` path. PR #72616 helps with installed-but-unloaded LaunchAgent recovery, but its changelog entry only refs this issue and fixes a narrower split-brain supervision family.

## Best Possible Solution

Keep this issue open until the LaunchAgent auto-update path has an explicit safe design: either run the updater/restart handoff from a detached supervisor with ignored or durable-log stdio before any service `bootout`, or deliberately disable managed-service package auto-updates with clear user-facing status and docs. Add a regression test that covers a LaunchAgent-marked auto-update invocation, not only manual `openclaw update` behavior.

## Risks / Open Questions

- Recent LaunchAgent recovery work and the package-update-in-service refusal reduce outage risk, so a future fix should preserve those protections instead of re-enabling unsafe in-service package replacement.
- The docs still describe a Gateway core auto-updater for package installs; if the intended policy is to fail fast under LaunchAgent rather than auto-apply there, the product behavior and docs need to say that explicitly.
