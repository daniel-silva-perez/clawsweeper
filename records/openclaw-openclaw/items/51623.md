## Summary

Keep this PR open. Current main still does not include the proposed pending followup queue persistence/replay surface, the paired tracker #51620 remains open, and this PR is an implementation candidate rather than obsolete. A security review also finds the proposed replay path needs maintainer attention before merge because user-originated queued text would be injected as trusted system-event content.

## Best Possible Solution

Keep this PR open for maintainer review or replacement. The best path is a durable pending-inbound/followup recovery design that persists typed snapshots before shutdown and drain-window rejection, consumes idempotently on startup, replays through normal untrusted inbound handling or trusted:false system events, avoids lossy short TTLs for valid queued work, and covers the remaining #51620 phases with focused tests. If #51803 or another branch becomes the selected implementation, close this PR only after that replacement is merged or maintainers explicitly choose the canonical path.
