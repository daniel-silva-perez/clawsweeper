## Summary

Keep open. Current main still does not implement the PR's intended pending followup queue persistence/replay behavior, the paired tracker #51620 remains open, and the PR is an implementation candidate rather than obsolete. The supplied PR context also shows unresolved functional and security-review concerns, especially replaying user-originated queued text through trusted system-event semantics.

## Best Possible Solution

Keep this PR open for maintainer review or replacement. The best path is a durable pending-inbound/followup recovery design tied to the existing queue, session-delivery, or inbound routing contracts: persist typed snapshots before shutdown and drain-window rejection, consume idempotently, replay through normal untrusted inbound handling or trusted:false system events with quoted user content, avoid a lossy short TTL for valid queued work, and cover the remaining #51620 phases with focused tests. If #51803 or another branch becomes the canonical implementation, close this PR only after that replacement is selected or merged.
