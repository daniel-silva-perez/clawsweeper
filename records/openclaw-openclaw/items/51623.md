## Summary

Keep open. Current main still does not persist or replay pending followup queues across gateway restart, and the paired upstream issue #51620 remains open. This PR is still an implementation candidate, but it needs maintainer review because the proposed diff has unresolved durability and security-review concerns.

## Best Possible Solution

Keep this PR open for maintainer review or replacement. The best path is a durable pending-inbound/followup recovery design tied to the existing queue, session-delivery, or inbound routing contracts, with idempotent snapshot consumption, replayed user text treated as untrusted content, no hard short TTL for valid queued work, and explicit coverage for the remaining #51620 phases.

## Risks / Open Questions

- Merging the PR unchanged could promote replayed user message text into trusted System: prompt lines unless recovered content is sanitized/quoted and enqueued with trusted:false.
- The proposed five-minute staleness guard can still lose valid queued user work after slow manual downtime or repeated failed restarts, which conflicts with the restart-survival goal.
- The in-process restart path can preserve some global queue state; a durable snapshot design needs idempotent consume/clear behavior to avoid duplicate processing or duplicate recovery notices.
