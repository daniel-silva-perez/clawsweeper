## Summary

Keep this PR open. Current main still lacks durable pending followup queue persistence/replay, the paired reliability tracker remains open, and the PR is a relevant implementation candidate, but the proposed patch should not merge as-is because it elevates user-originated queued text into trusted system events and has restart idempotency and expiry issues.

## What This Changes

The PR adds pending followup queue persistence to `state/pending-messages.json`, startup replay through system events, shutdown/startup hooks, and unit/integration proof coverage.

## Best Possible Solution

Keep a durable pending-inbound/followup recovery design in core, but route replay through normal untrusted inbound handling or explicit `trusted: false` system events, make snapshot consumption idempotent across process and in-process restarts, avoid lossy short TTLs for valid queued work, and let maintainers choose between this PR, #51803, or a replacement that also plans the remaining #51620 phases.

## Reproduction Assessment

Yes. The current gap is reproducible by queuing or sending inbound work near `openclaw gateway restart`; source inspection shows current main keeps followup queues in memory, clears gateway state on shutdown, and rejects new enqueues after `markGatewayDraining()` without persistence. The PR-specific trust and duplicate-replay problems are also reproducible by reading the changed replay call and the current in-process restart fallback path.

## Solution Assessment

No. The PR is a useful Phase 1 direction, but the safest maintainable solution needs untrusted/idempotent replay semantics and a less lossy persistence policy before merge; otherwise it partially fixes message loss while introducing prompt-trust and duplicate-delivery regressions.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Mark replayed prompts as untrusted:** `src/gateway/server-pending-messages.ts:44-47`
  - body: `item.prompt` is channel user text, but this call omits `trusted: false`; `enqueueSystemEvent` defaults to trusted and the next turn renders the content as `System:`. A queued message that contains instructions can be elevated across restart, so replay should go through normal inbound handling or enqueue an explicitly untrusted, quoted event.
  - confidence: 0.94
- **[P1] Clear persisted queues after a successful snapshot:** `src/gateway/server-close.ts:132`
  - body: After writing the snapshot, the global `FOLLOWUP_QUEUES` map is left intact. Current gateway restarts can fall back to an in-process restart, so the old queue can continue to exist while startup also replays the just-written file, producing duplicate recovery for the same pending messages.
  - confidence: 0.86
- **[P2] Do not expire valid restart snapshots after five minutes:** `src/auto-reply/reply/queue/persist.ts:110-116`
  - body: The loader discards every snapshot older than five minutes, which reintroduces message loss for slow reboots, manual downtime, or repeated failed restarts. Since this feature is meant to preserve queued messages across restarts, use explicit operator cleanup or a much safer policy instead of a fixed short TTL.
  - confidence: 0.84
- **[P3] Validate persistedAt before using it:** `src/auto-reply/reply/queue/persist.ts:105-112`
  - body: The parsed file is cast to `PersistedQueueFile`, but `persistedAt` is not checked. Missing or non-numeric values make `Date.now() - parsed.persistedAt` evaluate to `NaN`, so the current staleness check is bypassed for malformed files.
  - confidence: 0.78
