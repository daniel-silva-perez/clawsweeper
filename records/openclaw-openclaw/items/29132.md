## Summary

Keep open. Current main has a trusted bundled-plugin tool-result middleware path that can run before model-visible tool output, but the public plugin hook surface still does not expose `tool_result_before_model`, `transform_tool_result`, `before_context_send`, or another external post-execution/pre-model mutation hook. The Agent14 comment corroborates production need and a workaround, but it does not remove the SDK/API gap.

## Best Possible Solution

Keep this issue open until maintainers either land a documented, trust-gated public post-execution/pre-model tool-result mutation hook or explicitly decide the capability must remain bundled-only. A complete design should define ordering relative to `before_tool_call`, bundled `registerAgentToolResultMiddleware`, `after_tool_call`, `tool_result_persist`, and `before_message_write`; specify sync/async and failure behavior; coordinate with #62123 and #72733; update SDK docs/types; and add Pi/Codex runtime tests proving the model receives only the sanitized result.

## Risks / Open Questions

- External plugin access to same-turn model-visible tool-result rewriting is a security and trust boundary; ordering, allowed plugin origins, sync versus async behavior, failure semantics, and auditability need maintainer policy.
- There is meaningful overlap with #62123 and #72733, so landing a narrow sanitizer surface without coordination could create competing APIs for the same security-sensitive workflow.
- Closing this now would leave external plugin authors with only bundled middleware, `before_tool_call`, `after_tool_call`, and `tool_result_persist`, none of which provide the requested public deterministic post-execution/pre-model result replacement hook.
