## Summary

Keep open. Current main still does not expose `tool_result_before_model`, `transform_tool_result`, `before_context_send`, or an equivalent public post-execution/pre-model tool-result mutation hook. The closest implementation is internal runtime-neutral tool-result middleware, but it remains bundled-only, so the external plugin API gap described here is still real. The Agent14 comment usefully confirms a production workaround, but it is a workaround rather than an implemented OpenClaw public contract.

## Best Possible Solution

Keep this issue open until maintainers design and land a documented, trust-gated public post-execution/pre-model tool-result mutation hook, or explicitly decide that same-turn model-visible tool-result rewriting must remain bundled-only. The design should coordinate with #62123 and #72733 and define sync/async behavior, ordering relative to `before_tool_call`, `after_tool_call`, bundled middleware, `tool_result_persist`, and `before_message_write`, plus failure/timeout behavior, SDK types/docs, and Pi/Codex tests.

## Risks / Open Questions

- This is a security/trust boundary: a public hook that rewrites model-visible tool output needs explicit ordering, failure, timeout, provenance, and trust semantics.
- There is overlap with #62123 and #72733; landing one narrow hook without coordination could create competing security APIs or unclear guidance for plugin authors.
- A complete implementation must cover both Pi and Codex dynamic-tool paths, while docs currently state Codex-native tool result rewriting is outside `tool_result_persist`.
