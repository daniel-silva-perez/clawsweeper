## Summary

Keep this PR open, but do not merge it as-is. Current main and the latest release do not already provide the native assistant-message copy affordance, so the contribution is still relevant; the remaining blocker is that the proposed button copies raw assistant text even when the native renderer hides thinking/trace segments from the visible UI. Earlier review discussion shows the timer, hit-testing, iOS pasteboard, and tool-result scope issues were addressed in the current diff, leaving the hidden-trace clipboard mismatch as the concrete blocker.

## What This Changes

The PR adds a SwiftUI copy button to native OpenClawChatUI assistant messages, using platform pasteboards, macOS hover visibility, iOS visibility, and copied-state icon feedback.

## Best Possible Solution

Keep the native copy affordance, but compute its clipboard payload from the same visible assistant segment path used for rendering, respecting showsAssistantTrace, and add focused OpenClawChatUI coverage for trace-hidden and trace-visible copy payload behavior.

## Reproduction Assessment

Yes. A code-level reproduction is high confidence: with assistant text like `<think>secret</think><final>answer</final>` and `showsAssistantTrace == false`, current rendering displays only the response segment while the PR passes the raw primary text to the copy button.

## Solution Assessment

No. The feature direction is reasonable, but the proposed implementation should copy rendered/visible assistant content rather than raw payload text; a shared visible-copy-text helper or equivalent segment join is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

- **[P1] Copy only visible assistant text:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/ChatMessageViews.swift:231`
