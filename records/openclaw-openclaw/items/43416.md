## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks the native SwiftUI assistant-message copy affordance, so the contribution is not obsolete; however, the PR copies raw assistant `primaryText` while current rendering can hide thinking/trace segments, creating a clipboard privacy mismatch.

## What This Changes

The PR adds a shared SwiftUI copy button to assistant message bubbles in `OpenClawChatUI`, with macOS hover visibility, macOS/iOS pasteboard writes, and copied-state icon feedback.

## Best Possible Solution

Keep the native copy affordance, but compute the clipboard payload from the same visible assistant segment path used by rendering, respecting `showsAssistantTrace`, then add focused OpenClawChatUI coverage for trace-off and trace-on copy payload behavior.

## Reproduction Assessment

Yes. A code-level reproduction is high confidence: with assistant text like `<think>secret</think><final>answer</final>` and `showsAssistantTrace == false`, current rendering displays only the response segment while the PR passes the raw primary text to the copy button.

## Solution Assessment

No. The feature direction is reasonable, but the proposed implementation should copy rendered/visible assistant content rather than raw payload text; a shared visible-copy-text helper or equivalent segment join is the safer narrow fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Copy only rendered assistant text:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/ChatMessageViews.swift:231`
  - body: `CopyMessageButton` receives raw `primaryText`, but assistant rendering uses `AssistantTextParser.segments(..., includeThinking: showsAssistantTrace)`. With traces hidden, a message like `<think>secret</think><final>answer</final>` displays only the answer while this button copies the hidden trace and tags to the clipboard, so feed the button the same visible text the UI renders.
  - confidence: 0.94
