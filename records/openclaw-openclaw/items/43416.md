## Summary

Keep open. Current main still lacks this native assistant-message copy affordance, and the PR is useful, but the latest diff has a blocking clipboard privacy bug: it copies raw assistant payload text even when the renderer hides thinking/trace segments. Earlier review discussion resolved timer, hit-testing, iOS pasteboard, touch availability, and assistant-role scoping concerns; the remaining visible-text mismatch needs revision before merge.

## What This Changes

The PR adds a shared native SwiftUI copy button to assistant message bubbles in `ChatMessageViews.swift`, with macOS hover behavior, platform pasteboard writes, and copied-state icon feedback.

## Best Possible Solution

Keep the native copy affordance, but compute the clipboard payload from the same `AssistantTextParser` visible segment path used for rendering, respecting `showsAssistantTrace`, then add focused OpenClawChatUI coverage for trace-off and trace-on copy payload behavior.

## Reproduction Assessment

Yes. A focused code-level reproduction is high confidence: use assistant text like `<think>secret</think><final>answer</final>` with `showsAssistantTrace == false`; current rendering displays only the response segment, while the PR passes the raw `primaryText` into the copy button.

## Solution Assessment

No. The feature direction is reasonable, but the current implementation is not the best fix because it copies raw payload text; the safer narrow fix is a shared visible-copy-text helper or equivalent segment join used by the button and covered by Swift tests.
