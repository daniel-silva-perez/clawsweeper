## Summary

Keep this PR open, but do not merge the current patch. The native copy affordance is still useful and is not implemented on current main, and the latest diff appears to have addressed earlier timer, hit-testing, iOS pasteboard, and tool-role concerns; the remaining blocker is that the button copies the raw assistant payload instead of the visible assistant text, which can place hidden trace/thinking content on the clipboard.

## What This Changes

The PR modifies `apps/shared/OpenClawKit/Sources/OpenClawChatUI/ChatMessageViews.swift` to add assistant-message hover state plus a `CopyMessageButton` that writes text to `NSPasteboard` or `UIPasteboard` with temporary copied-state feedback.

## Best Possible Solution

Keep the native copy affordance, but derive the clipboard payload from the same visible assistant segments used for rendering, respecting `showsAssistantTrace`, and add focused OpenClawChatUI coverage for hidden-trace and visible-trace copy text.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is an assistant message like `<think>secret</think><final>answer</final>` with `showsAssistantTrace == false`: current rendering displays only the response segment, while the PR passes the raw primary text to the pasteboard button.

## Solution Assessment

No. The feature direction is reasonable, but the implementation should copy rendered/visible assistant content rather than raw payload text; a shared helper that joins `AssistantTextParser.segments(from:includeThinking:)` output is the safer maintainable path.
