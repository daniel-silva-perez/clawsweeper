## Summary

Keep this PR open. The feature is useful and not already implemented on current main, but the proposed patch still has a blocking clipboard payload issue: it copies raw assistant `primaryText` instead of the rendered text path that hides thinking segments when traces are off. Earlier review discussion shows the timer, hit-testing, iOS pasteboard, and tool-result scope issues were addressed; the remaining raw-payload copy finding should be fixed before merge.

## What This Changes

The PR adds hover state and a native assistant-message copy button in `ChatMessageViews.swift`, writing message text to `NSPasteboard` or `UIPasteboard` with temporary copied-state feedback.

## Best Possible Solution

Revise the copy feature so the clipboard payload is derived from the same `AssistantTextParser` segment path used for rendering, honoring `showsAssistantTrace` and joining only the currently visible assistant text, then add focused Swift coverage for that helper and validate the shared native chat UI on macOS and iOS.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Copy the rendered assistant text:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/ChatMessageViews.swift:231`
  - body: `CopyMessageButton` receives raw `primaryText`, but assistant rendering goes through `AssistantTextParser.segments(..., includeThinking: showsAssistantTrace)` and can hide thinking segments when traces are off. For tagged assistant output, the UI can show only the final response while the copy action places hidden reasoning and tags on the clipboard, so the copy payload should be built from the same visible segments as the renderer.
  - confidence: 0.94
