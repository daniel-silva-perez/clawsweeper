## Summary

Keep this PR open. Current main still lacks the requested native Swift assistant-message copy affordance, but the proposed patch should not merge as-is because the clipboard payload is the raw message text rather than the same assistant text that is rendered after trace/thinking filtering.

## What This Changes

The PR adds a native Swift copy button to assistant messages in OpenClawChatUI, with platform pasteboard writes and temporary copied-state feedback.

## Best Possible Solution

Revise the PR so the copy payload is derived from the same AssistantTextParser segment path used for rendering, joined as the currently visible assistant text for the active trace setting, with focused helper/parser coverage and native macOS/iOS verification before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Copy the rendered assistant text:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/ChatMessageViews.swift:231`
  - body: This passes raw `primaryText` into the copy button, while rendering calls `AssistantTextParser.segments(..., includeThinking: showsAssistantTrace)` and hides thinking segments when traces are off. For a tagged assistant response, copy would put hidden reasoning/tags on the clipboard even though the UI shows only the final text, so the payload should be built from the same visible segments as the renderer.
  - confidence: 0.94

## Security Review
