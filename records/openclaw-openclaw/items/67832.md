## Summary

Keep open. Current main already has the direct HTTP `/tools/invoke` endpoint and the `sessions_spawn` agent tool, but it still does not provide the narrowed user-facing CLI path requested in the reporter's follow-up; none of the provided related items supersedes that remaining CLI work.

## What This Changes

The issue requests a direct shell/API interface for spawning subagent sessions, later narrowed to a CLI command such as `openclaw sessions spawn` or a generic `openclaw tools invoke`.

## Best Possible Solution

Keep the issue open and settle a conservative CLI design that reuses the existing Gateway `/tools/invoke` contract, preserves Gateway auth and tool policy, keeps dangerous session orchestration opt-in by default, enforces child-session limits, and updates CLI docs plus targeted command and Gateway-policy tests.
