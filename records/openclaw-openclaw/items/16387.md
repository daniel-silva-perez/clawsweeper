## Summary

Keep this issue open. Current main still treats custom provider headers as literal strings, the Fireworks plugin does not add session-affinity behavior, and the current docs still describe provider headers as static. The related implementation PRs are closed unmerged, while the latest maintainer comment scopes the remaining work as a narrow runtime interpolation feature.

## What This Changes

The issue requests runtime interpolation for custom LLM provider headers so values like session, agent, or run identifiers can be injected into upstream provider requests.

## Best Possible Solution

Add a narrow documented resolver at model-request dispatch for configured provider headers and provider request headers, supporting only explicit placeholders such as {{session.key}}, {{agent.id}}, and {{run.id}} while keeping existing static-header, auth, attribution, and provider-plugin behavior intact.
