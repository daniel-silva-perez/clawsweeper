## Summary

Keep open. Current main and the latest release still treat configured model-provider headers as literal static strings, the Fireworks plugin does not add session-affinity routing, and the closed related PRs were not merged; the remaining work is a narrow core/runtime feature because OpenClaw owns the requested session, agent, and run identity.

## What This Changes

The issue requests dynamic interpolation for custom LLM provider HTTP headers so configured values can include session, agent, or run identifiers for provider-side cache affinity and tracing.

## Best Possible Solution

Land a small documented dispatch-time resolver for configured model-provider headers that supports only explicit placeholders such as {{session.key}}, {{agent.id}}, and {{run.id}} for both provider-level headers and request.headers, while preserving static header behavior, SecretRef resolution, auth injection, attribution headers, retry behavior, and provider-owned stream/transport hooks.

## Reproduction Assessment

Yes. Configure a custom provider header such as `x-session-affinity: {{session.key}}`; current main sanitizes and merges that value as a literal string, and the runtime wrapper path does not carry sessionKey or runId for interpolation.

## Solution Assessment

Yes. A generic OpenClaw resolver at model-request dispatch is the narrowest maintainable solution because OpenClaw owns the session/run identity and already centralizes configured header merging; a Fireworks-only plugin change would not solve the LiteLLM/Langfuse custom-provider case.
