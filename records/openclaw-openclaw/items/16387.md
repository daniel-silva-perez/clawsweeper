## Summary

Keep this issue open. Current main still supports static configured provider headers but does not resolve runtime placeholders such as session, agent, or run ids for `models.providers.*.headers` or `request.headers`; the related implementation PRs are closed unmerged, and the latest maintainer discussion already frames this as a narrow runtime feature gap.

## What This Changes

Request dynamic interpolation for custom LLM provider headers so runtime values like session key, agent id, and run id can be sent to Fireworks, LiteLLM, Langfuse, or similar upstreams.

## Best Possible Solution

Land a narrow documented runtime resolver for configured model-provider headers at model-request dispatch, covering both provider-level and request-level headers with explicit placeholders while preserving existing static-header behavior and provider-owned hooks.
