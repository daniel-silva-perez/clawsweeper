## Summary

Keep open. Current main still treats configured model-provider headers as literal strings, the Fireworks plugin does not add session-affinity routing, docs still describe headers as static, and the related dynamic-templating PRs were closed unmerged; the remaining work is narrow enough for a focused fix PR.

## What This Changes

Request dynamic interpolation for custom model-provider headers so values such as `{{session.key}}`, `{{agent.id}}`, and `{{run.id}}` can be applied per model request.

## Best Possible Solution

Land a documented dispatch-time resolver for configured model-provider headers that supports only explicit placeholders for session, agent, and run identity across provider-level headers and `request.headers`, while keeping existing static headers and provider-owned hooks unchanged.

## Reproduction Assessment

Yes. Configure a custom provider header like `x-session-affinity: {{session.key}}`; current main sanitizes and merges that value as a literal string, and the runtime path does not provide `sessionKey` or `runId` for interpolation.

## Solution Assessment

Yes. A generic OpenClaw resolver at model-request dispatch is the narrowest maintainable direction because OpenClaw owns the requested session/run identity and already centralizes configured header merging; a Fireworks-only plugin change would not solve the LiteLLM/Langfuse custom-provider case.
