## Summary

Keep open. Current main still has static-only configured provider headers, the Fireworks provider does not add session-affinity routing, and the related implementation attempts are closed unmerged while the latest maintainer discussion scopes this as a narrow runtime feature gap.

## What This Changes

Add dynamic interpolation for configured custom LLM provider headers so values like `{{session.key}}`, `{{agent.id}}`, and `{{run.id}}` can be resolved per model request for Fireworks/Langfuse-style routing.

## Best Possible Solution

Land a small documented dispatch-time resolver for configured provider headers that supports only explicit placeholders and works for both provider-level and request-level headers without hardcoding Fireworks or Langfuse behavior.

## Reproduction Assessment

Yes. A source-level reproduction is to configure a header such as `x-session-affinity: {{session.key}}`; current main sanitizes and merges that value as a literal string, and the runtime wrapper path does not carry `sessionKey` or `runId` for interpolation.

## Solution Assessment

Yes. A generic resolver at model-request dispatch is the narrowest maintainable path because OpenClaw owns the session/run identity and already centralizes configured header merging; a Fireworks-only plugin patch would not solve the LiteLLM/Langfuse custom-provider case.
