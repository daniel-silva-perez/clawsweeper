## Summary

Keep this issue open. Current main still forwards custom provider headers as static strings and has no documented or tested resolver for `{{session.key}}`, `{{agent.id}}`, or `{{run.id}}` in `models.providers.*.headers` or `models.providers.*.request.headers`; the Fireworks and LiteLLM/Langfuse use cases remain valid, and the two implementation PRs referenced from the issue are closed unmerged.

## Best Possible Solution

Implement a narrow, documented runtime header-template resolver for `models.providers.*.headers` and `models.providers.*.request.headers`. Resolve explicit placeholders such as `{{session.key}}`, `{{agent.id}}`, and `{{run.id}}` at request dispatch for main turns and compaction, while preserving static headers, SecretRef handling, auth injection, header precedence, retries, and existing provider-owned hooks. Keep this issue open until that lands or a maintainer chooses a different product boundary.

## Repair Work Prompt

Implement dynamic provider-header templating for this issue. Current main forwards `models.providers.*.headers` and `models.providers.*.request.headers` as static strings. Add a narrow runtime resolver for explicit placeholders `{{session.key}}`, `{{agent.id}}`, and `{{run.id}}` so Fireworks `x-session-affinity` and LiteLLM/Langfuse headers can be configured per session/run.

Expected boundary:
