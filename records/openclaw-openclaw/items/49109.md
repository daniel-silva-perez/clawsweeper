## Summary

Keep open. The central cron JSON/quiet-helper behavior is still missing on current main, and the gateway retry half is only partially covered by the separate open #54475. This PR remains useful as source context, but it is not mergeable as-is because the branch is stale against current CLI/gateway seams and includes unrelated security-sensitive, generated, build, plugin, status, and channel-test churn.

## What This Changes

The PR attempts to quiet cron helper status checks, add bounded CLI gateway transport retries, and adds tests, while also carrying broad unrelated changes across status startup, SSRF/network code, plugin registry/generated config, build entries, and channel tests.

## Best Possible Solution

Keep the useful intent, but split the implementation. Review or land #54475, or an equivalent focused gateway fix, for transient pre-hello close handling. Then apply the cron helper change as a small current-main PR in the cron CLI helper and tests: skip the helper in JSON mode, call it quietly/probe-style outside JSON mode, and force `expectFinal: false`. Drop or separately review the unrelated SSRF, status startup, plugin registry, generated config, build, and channel-test changes.

## Reproduction Assessment

Yes. There is a high-confidence code-level reproduction path: current main writes cron add JSON and then calls a helper that unconditionally performs `cron.status`, and the gateway call path still treats pre-settlement close as terminal. I did not run tests because this was a read-only review.

## Solution Assessment

No. The intended cron helper behavior is a good narrow fix, and #54475 is a better-focused gateway path, but this PR is not the best way to solve the issue because its branch is stale and mixes unrelated security-sensitive and cross-plugin changes into a CLI hotfix.
