## Summary

Keep open. Current main has some context-overflow improvements, including a failover guard that rethrows likely context-overflow errors instead of silently trying fallback models, and the UI can show a high-context compact action when usage is fresh. But the broader reported failure mode is not fully solved: preflight compaction still skips heartbeat turns, skips ordinary token-based proactive compaction when persisted token counts are fresh, and computes the token threshold from `reserveTokensFloor` rather than the effective `reserveTokens` setting. Related #66830 and PR #69379 cover only the reserve-token asymmetry, not the heartbeat/fresh-token/model-switch UX gaps discussed here.

## Best Possible Solution

Keep this issue open, or split it into canonical narrower bugs only after maintainers decide the tracking shape. The implementation path should make proactive compaction evaluate fresh persisted token counts directly, decide heartbeat compaction versus isolated heartbeat sessions as an explicit policy, align preflight/memoryFlush thresholds with the effective reserve `max(reserveTokens, reserveTokensFloor)`, and add regression coverage for long active sessions, heartbeat-only sessions, context-overflow failover handling, and model-switch recovery messaging.
