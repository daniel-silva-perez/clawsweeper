## Summary

Keep open. The reported data-loss bug is still present on current main and the PR targets the right predicate, but the patch should not merge as-is because the new archive/reset path can recreate transcript files without the existing private mode, and the new fixture setup is not portable to Windows CI.

## What This Changes

The PR narrows session initialization resets to header-only transcripts, archives before clearing, adds regression coverage for mid-turn restart transcripts, and adds a shell verification wrapper.

## Best Possible Solution

Amend the existing PR so the hasAnyMessage guard remains, the archive/reset branch preserves or recreates the transcript with 0o600 permissions, the fixtures use os.tmpdir() with path.join(), and the verification wrapper uses the repo test entrypoint if it stays.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve transcript file mode after archiving:** `src/agents/pi-embedded-runner/session-manager-init.ts:56`
  - body: The new archive step renames the existing private transcript before this write runs. Because the original path no longer exists, fs.writeFile(..., "utf-8") creates a replacement using default process permissions instead of the 0o600 mode used by transcript creation paths, so later transcript content can land in a less-private file. Recreate the file with mode 0o600 or restore the archived file's mode.
  - confidence: 0.87
- **[P2] Use the OS temp directory for fixtures:** `src/agents/pi-embedded-runner/session-manager-init.test.ts:10-11`
  - body: The new test hardcodes /tmp as the mkdtemp parent and then concatenates paths. Windows CI may not have that parent, so the suite can fail before it exercises the session-preservation behavior. Use os.tmpdir() and path.join() for the prefix and session file path.
  - confidence: 0.91
- **[P3] Use the repo test wrapper in the verification script:** `scripts/verify-session-init-fix.sh:13`
  - body: The added helper invokes pnpm vitest run directly, bypassing the repo's pnpm test/scripts wrapper that applies local Vitest scheduling and process handling. Use pnpm test with the target file so the script matches the documented validation path.
  - confidence: 0.76
