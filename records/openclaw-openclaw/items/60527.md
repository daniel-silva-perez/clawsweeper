## Summary

Keep open. Current main already fixes the concrete Telegram/private-proxy failure through the merged allowPrivateProxy path, but it does not implement this PR's broader behavior for public explicit proxies without allowPrivateProxy. That remaining behavior is a security-sensitive proxy trust contract decision, so this should stay with maintainers rather than be closed as implemented or duplicate cleanup.

## What This Changes

The PR strips hostnameAllowlist from the SSRF policy before validating explicit proxy hostnames and adds regression tests for private proxies, public proxies, and the private-proxy blocking invariant.

## Best Possible Solution

Keep the shipped private-proxy fix, then make an explicit maintainer decision on the remaining public explicit-proxy contract. If all explicit proxy URLs are considered operator-controlled, rebase this behavior onto current main with the public proxy regression test; otherwise document and test that allowPrivateProxy is the trust bit required to ignore target hostname allowlists for proxy host validation.

## Reproduction Assessment

Yes for the remaining behavior. The PR's public-proxy scenario still follows current main's default explicit-proxy branch, which preserves policy.hostnameAllowlist and therefore fails in resolveHostnamePolicyChecks before fetch is called.

## Solution Assessment

Unclear. Stripping a target URL allowlist from proxy hostname validation is the right shape for operator-trusted proxies, but applying it to every explicit proxy without allowPrivateProxy is broader than the shipped fix and should be confirmed against the security model.
