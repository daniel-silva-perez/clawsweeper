## Summary

Keep open. Current main fixes the Telegram/private-proxy regression path, but it does not fully implement this PR's intended behavior: explicit public proxies are still validated with the target URL hostnameAllowlist unless allowPrivateProxy is true. The remaining policy choice is security-sensitive and should get maintainer review rather than cleanup closure.

## Best Possible Solution

Review or adapt this PR instead of closing it. Maintainers should either land the proxy-specific policy change with tests for both private and public explicit proxies, or document and test that allowPrivateProxy is the required trust bit for any proxy host outside the target hostnameAllowlist.
