## Summary

Keep this issue open. Current main still fixes only part of the report: zero-byte audio is skipped before transcription and Telegram media fetch failures return before dispatch, but the shared buffer store used by Telegram inbound media still writes directly to the final path, so an ENOSPC/failed-write partial artifact can still be left behind.

## Best Possible Solution

Keep this issue open and finish the shared media-store fix: make `writeSavedMediaBuffer` write to a same-directory temporary file, close/fsync when feasible, rename into place only after the full write succeeds, and remove the temp file on failure. Add a focused regression test in `src/media/store.test.ts` that simulates an ENOSPC or mid-write failure and proves no final-path inbound media artifact remains.
