## Summary

Keep this issue open. Current main still fixes only part of the report: zero-byte audio is skipped before transcription and Telegram media fetch failures stop dispatch, but inbound media buffers are still written directly to the final media-store path, so the ENOSPC partial/zero-byte artifact class remains open.

## Best Possible Solution

Keep this issue open and finish it in the shared media store: make `writeSavedMediaBuffer` write to a same-directory temp path, fsync/close when feasible, rename into place only after the full write succeeds, and remove the temp file on failure. Add a focused regression test in `src/media/store.test.ts` that simulates an ENOSPC/final-write failure and proves no final-path media artifact is left behind. Keep the existing tiny-audio transcription guard and consider clearer operator-facing logging for Telegram `ENOSPC` media-store failures.

- Closing now would leave no tracker for making inbound Telegram media persistence atomic under ENOSPC or failed writes.
- A failed `fs.writeFile` after open/truncate can still leave a final-path partial or zero-byte artifact for later readers.
- No live ENOSPC reproduction was run because this review was read-only; the keep-open decision is based on current source and test coverage.
