## Summary

Keep open, but do not merge as-is. The cache-growth bug is real on current main, but this PR mixes the useful cache caps with unrelated web_search/provider credential rewrites and tool schema validation changes that would regress existing behavior.

## What This Changes

The PR adds oldest-entry caps to several agent-runtime cache maps and also changes agent tool schemas while replacing the current web_search provider-registry path with inline provider handling.

## Best Possible Solution

Land a narrow cache-only implementation on current main that caps the intended Maps with oldest-entry eviction only for new-key insertions, preserves web_search provider/SecretRef contracts and existing tool schemas, and adds focused regression tests for cap and refresh semantics.

## Reproduction Assessment

Yes. Current main has module-level Maps in the reported cache paths and the inspected write sites do not enforce max size; repeated distinct session keys, workspace paths, cache-trace paths, or model ids would grow those Maps without bound.

## Solution Assessment

No. The cache-cap direction is appropriate, but this PR is not the best implementation because it bundles unrelated provider, credential, and schema behavior changes; a cache-only replacement is the safer maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Route web_search through the provider registry:** `src/agents/tools/web-search.ts:25`
  - body: The PR replaces the registry-backed web_search tool with a hard-coded provider list. Current main resolves bundled and runtime providers through resolveWebSearchDefinition()/runWebSearch(), so custom runtime providers and bundled providers outside the inline list can stop working or fall back incorrectly.
  - confidence: 0.93
- **[P1] Use plugin-owned SecretRefs for web_search keys:** `src/agents/tools/web-search.ts:553-563`
  - body: The new credential resolvers read legacy tool config or env vars instead of plugin-owned plugins.entries.<plugin>.config.webSearch.* paths and resolved SecretRefs. Existing users with valid plugin config can see missing-key failures despite configured credentials.
  - confidence: 0.9
- **[P2] Evict workspace cache only for new keys:** `src/agents/workspace.ts:92`
  - body: The eviction call runs before every workspaceFileCache.set, including refreshes for a file path already in the map. At capacity, refreshing an existing file evicts an unrelated oldest entry and then overwrites the same key, reducing effective capacity and causing avoidable disk churn.
  - confidence: 0.88
- **[P2] Preserve sub-second node clip durations:** `src/agents/tools/nodes-tool.ts:165-170`
  - body: Adding minimum: 1000 to durationMs rejects 0-999 ms values that current main accepts up to the existing 300000 ms maximum. That is an unrelated behavior change for camera clips and contradicts the cache-only scope.
  - confidence: 0.82
- **[P3] Apply the model cache cap consistently:** `src/agents/context.ts:52-58`
  - body: Only the discovered-model path enforces the new model-cache cap, while applyConfiguredContextWindows writes to the same cache without a guard. A large configured model list can push the map past the cap and later cause discovered inserts to evict configured entries.
  - confidence: 0.76
