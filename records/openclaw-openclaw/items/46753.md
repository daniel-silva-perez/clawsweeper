## Summary

Keep open. Current main still leaves the four targeted agent/runtime caches uncapped, so the central memory-growth fix is still useful, but this PR should not merge as-is because it bundles unrelated tool-schema changes and a large web_search rewrite that bypasses the current plugin/runtime provider and SecretRef architecture.

## What This Changes

The PR caps four agent cache maps and also changes many agent tool schemas while replacing web_search with a large inline provider implementation.

## Best Possible Solution

Land a narrow cache-only implementation on current main that caps the four intended maps with oldest-entry eviction only when inserting new keys, keeps the provider/runtime web_search architecture unchanged, and adds focused regression coverage around cache size and refresh behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep web_search on the provider registry:** `src/agents/tools/web-search.ts:25`
  - body: The PR replaces the registry-backed `web_search` tool with a hard-coded provider set. Current main resolves bundled, runtime, keyless, and custom providers through `resolveWebSearchDefinition()`/`runWebSearch()`; this list omits supported providers such as Firecrawl, Tavily, Exa, DuckDuckGo, Ollama, SearXNG, and custom runtime providers, so configured setups would stop working or fall back incorrectly.
  - confidence: 0.93
- **[P1] Resolve provider keys through plugin-owned SecretRefs:** `src/agents/tools/web-search.ts:553-563`
  - body: The new credential resolvers read legacy tool config fields or env vars, but current setup stores bundled provider credentials under `plugins.entries.<plugin>.config.webSearch.*` and supports SecretRefs resolved through runtime snapshots. Users with those existing configs would get missing-key errors even though their providers are configured.
  - confidence: 0.9
- **[P2] Evict workspace cache only for new keys:** `src/agents/workspace.ts:92`
  - body: This eviction call runs before every `workspaceFileCache.set`, including stale-refresh writes for a file path already in the map. At capacity, refreshing an existing file evicts an unrelated oldest entry and then overwrites the same key, shrinking effective cache capacity and causing avoidable disk churn.
  - confidence: 0.88
- **[P2] Preserve sub-second node clip durations:** `src/agents/tools/nodes-tool.ts:165-170`
  - body: Adding `minimum: 1000` to `durationMs` rejects values from 0 to 999 ms that the existing schema accepted up to the 300000 ms maximum. That is an unrelated behavior change for camera clips and contradicts the PR's cache-only/no-behavior-change scope.
  - confidence: 0.82
- **[P3] Apply the model cache cap consistently:** `src/agents/context.ts:52-58`
  - body: Only the discovered-model path enforces the new cap, while `applyConfiguredContextWindows` writes to the same cache without a guard. A large configured model list can push the map past the cap, then the next discovered insert can evict configured entries; configured values should either be sticky or use the same documented policy.
  - confidence: 0.76
