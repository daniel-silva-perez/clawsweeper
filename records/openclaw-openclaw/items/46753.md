## Summary

Keep open. The cache-cap bug is still present on current main, but this PR should not merge as-is because it mixes the useful cache work with an unrelated registry-bypassing web_search rewrite, credential-routing changes, and schema behavior changes that would regress existing provider/plugin setups.

## What This Changes

The PR adds oldest-entry caps to four agent runtime cache Maps and also changes many agent tool schemas while replacing the registry-backed web_search tool with inline provider clients.

## Best Possible Solution

Land a narrow cache-only implementation on current main that caps the four intended Maps with oldest-entry eviction only for new-key insertions, preserves the existing web_search provider/SecretRef contracts and tool schemas, and adds focused regression tests for cache size and refresh semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Route web_search through the provider registry:** `src/agents/tools/web-search.ts:25`
  - body: The PR replaces the registry-backed web_search tool with a hard-coded provider list. Current main resolves bundled and runtime providers through resolveWebSearchDefinition()/runWebSearch(), so custom runtime providers and bundled providers outside the inline list can stop working or fall back incorrectly.
  - confidence: 0.93
- **[P1] Use plugin-owned SecretRefs for web_search keys:** `src/agents/tools/web-search.ts:553-563`
  - body: The new credential resolvers read legacy tool config or env vars instead of the plugin-owned plugins.entries.<plugin>.config.webSearch.* paths and resolved SecretRefs. Existing users with provider credentials stored in plugin config would see missing-key failures despite valid configuration.
  - confidence: 0.9
- **[P2] Evict workspace cache only for new keys:** `src/agents/workspace.ts:92`
  - body: The eviction call runs before every workspaceFileCache.set, including refreshes for a file path already in the map. At capacity, refreshing an existing file evicts an unrelated oldest entry and then overwrites the same key, reducing effective capacity and causing avoidable disk churn.
  - confidence: 0.88
- **[P2] Preserve sub-second node clip durations:** `src/agents/tools/nodes-tool.ts:165-170`
  - body: Adding minimum: 1000 to durationMs rejects 0-999 ms values that the current schema accepts up to the existing 300000 ms maximum. That is an unrelated behavior change for camera clips and contradicts the PR's cache-only scope.
  - confidence: 0.82
- **[P3] Apply the model cache cap consistently:** `src/agents/context.ts:52-58`
  - body: Only the discovered-model path enforces the new model-cache cap, while applyConfiguredContextWindows writes to the same cache without a guard. A large configured model list can push the map past the cap and later cause discovered inserts to evict configured entries.
  - confidence: 0.76
