## Summary

Keep this PR open. Current main still appends `.openclaw` for `OPENCLAW_HOME` default state/config paths, while this PR’s patch only changes `resolveConfigDir` and would return every explicit `OPENCLAW_HOME` directly. That leaves the linked open bug unsolved on main and keeps the PR unsafe as-is because it can split config consumers from state/config I/O and move trusted `.env` and plugin roots for service-home users.

## Best Possible Solution

Keep this PR open for revision or maintainer consolidation with #45765. The best fix is a narrow compatibility change that updates `resolveStateDir` and `resolveConfigDir` together, avoids double nesting only when explicit `OPENCLAW_HOME` already names a known state directory such as `.openclaw`, preserves general service-home behavior, removes the dead existence check from this PR, and adds regression tests for both fixed and preserved paths.

## Risks / Open Questions

- Merging this PR as-is would change `OPENCLAW_HOME=/srv/openclaw-home` defaults from `/srv/openclaw-home/.openclaw` to `/srv/openclaw-home` for `resolveConfigDir` consumers, contrary to current docs and tests.
- Because config I/O still uses `resolveStateDir`, this PR can split `openclaw.json` from `.env`, plugin roots, media, cron, and other `resolveConfigDir` consumers.
- The PR lacks regression tests for both the intended `.openclaw/.openclaw` fix and the preserved service-home compatibility behavior.
- The trusted global `.env` and global plugin root locations would move for explicit `OPENCLAW_HOME` users, which is security-sensitive even without supply-chain file changes.
