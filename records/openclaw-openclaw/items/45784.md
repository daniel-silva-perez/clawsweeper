## Summary

Keep this PR open. Current main still does not implement the intended nested `.openclaw/.openclaw` fix, and this PR remains unsafe as-is because it changes only `resolveConfigDir` while current config I/O still routes the default config path through `resolveStateDir`. The linked bug is real and still needs a narrow compatibility fix, but this PR would also move trusted config-dir consumers for all explicit `OPENCLAW_HOME` users.

## Best Possible Solution

Keep this PR open for revision or maintainer consolidation with #45765. The best fix is a narrow shared path rule used by both `resolveStateDir` and `resolveConfigDir`: avoid double nesting only when explicit `OPENCLAW_HOME` already names a known state directory such as `.openclaw`, preserve general `OPENCLAW_HOME` service-home behavior, remove the dead `existsSync` branch from this PR, and add regression tests for both fixed and preserved paths.

## Risks / Open Questions

- Merging this PR as-is would change `resolveConfigDir` defaults for service-style `OPENCLAW_HOME=/srv/openclaw-home` from `/srv/openclaw-home/.openclaw` to `/srv/openclaw-home`, contrary to current docs and tests.
- Because primary config I/O still uses `resolveStateDir`, the PR can split `openclaw.json` from `.env`, plugin roots, media, hooks, skills, and other `resolveConfigDir` consumers.
- The patch lacks regression tests for both the intended `OPENCLAW_HOME` ending in `.openclaw` fix and the preserved general service-home behavior.
- The path move affects trusted global `.env` and plugin-extension roots, which is security-sensitive even though the diff has no supply-chain or workflow changes.
