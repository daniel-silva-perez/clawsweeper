## Summary

Keep this PR open. Current main still treats `OPENCLAW_HOME` as a replacement home root and appends `.openclaw` for default state/config paths, so the intended nesting fix is not implemented. The PR also remains unsafe as-is because it only changes `resolveConfigDir`, returns every explicit `OPENCLAW_HOME` directly, and would diverge from `resolveStateDir` plus documented/tested behavior. Related #45765 is still the canonical open bug; #45793 and #46016 were alternative unmerged attempts.

## Best Possible Solution

Keep this PR open for author revision or maintainer consolidation with #45765. The best fix is a narrow compatibility change that updates `resolveStateDir` and `resolveConfigDir` together, preserves `OPENCLAW_HOME` as a replacement home for normal service directories, avoids double nesting only when the explicit value already names a known state directory such as `.openclaw`, removes the dead existence check, and adds regression tests for both fixed and preserved behaviors.

## Risks / Open Questions

- Merging this PR as-is would silently move `resolveConfigDir` consumers for `OPENCLAW_HOME=/srv/openclaw-home` from `/srv/openclaw-home/.openclaw` to `/srv/openclaw-home`, contrary to current docs and tests.
- Because primary config I/O still goes through `resolveStateDir`, the PR can split `openclaw.json` from global `.env` and plugin root resolution.
- The diff has no regression tests for either the intended `OPENCLAW_HOME=~/.openclaw` fix or the preserved `/srv/openclaw-home/.openclaw` compatibility behavior.
- The trusted global `.env` and global plugin root locations would change for all explicit `OPENCLAW_HOME` users, which is a security-sensitive path-resolution change even though the PR does not touch supply-chain files.
