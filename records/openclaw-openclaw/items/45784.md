## Summary

Keep this PR open. Current main still appends `.openclaw` under `OPENCLAW_HOME` for default state/config paths, so the PRâ€™s intended nesting fix is not implemented. The PR also remains unsafe as-is because it only changes `resolveConfigDir`, returns every explicit `OPENCLAW_HOME` directly, and would diverge from current `resolveStateDir`, docs, and tests. Related #45765 remains the canonical open bug; #45793 and #46016 were alternative unmerged attempts.

## Best Possible Solution

Keep this PR open for author revision or maintainer consolidation with #45765. The best fix is a narrow compatibility change that updates `resolveStateDir` and `resolveConfigDir` together, avoids double nesting only when explicit `OPENCLAW_HOME` already names a known state directory such as `.openclaw`, preserves service-home behavior for general paths, removes the dead existence check, and adds regression tests for fixed and preserved behaviors.

## Risks / Open Questions

- Merging this PR as-is would silently move `resolveConfigDir` consumers for `OPENCLAW_HOME=/srv/openclaw-home` from `/srv/openclaw-home/.openclaw` to `/srv/openclaw-home`, contrary to current docs and tests.
- Because primary config I/O still goes through `resolveStateDir`, this PR can split `openclaw.json` from global `.env` and plugin root resolution.
- The PR lacks regression tests for both the intended `OPENCLAW_HOME=~/.openclaw` fix and the preserved `/srv/openclaw-home/.openclaw` compatibility behavior.
- The trusted global `.env` and global plugin root locations would change for all explicit `OPENCLAW_HOME` users, which is security-sensitive even though the PR does not touch supply-chain files.
