## Summary

Keep open. Current main still intentionally keeps group/channel sessions separate from direct-message sessions, and the existing `dmScope`/`identityLinks` controls apply to direct-message peers rather than the issue’s requested per-user group-to-DM merge. The issue discussion also identifies privacy filtering requirements that are not implemented: only the linked user's group messages and bot replies should merge, while other group participants must not leak into that user's DM context.

## Best Possible Solution

Keep this open as a core/session-routing design item. The best path is an additive, opt-in routing model that can map an authorized sender's group and DM turns to a shared canonical session while filtering out other group participants, preserving reply routing to the originating chat, and updating Telegram/Discord plus shared routing tests and docs. It should be coordinated with the existing shared group/channel session tracker so the routing contract is generic rather than channel-specific.

## Risks / Open Questions

- A naive implementation could leak other group participants' messages into a private DM session; the issue comments explicitly call out this privacy boundary.
- Changing this behavior affects a documented safety boundary where group sessions are isolated and can run with different sandbox/tool posture from DMs.
- The work overlaps with open shared-session routing discussions such as #19929, so maintainers should coordinate the design rather than add a one-channel special case.
