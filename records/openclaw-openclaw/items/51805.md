## Summary

Keep open. Current main still keeps direct-message identity sharing separate from group/channel session routing: `identityLinks` is only applied inside direct-peer routing, group/channel peers keep `agent:<agentId>:<channel>:<kind>:<peerId>` keys, and docs/tests still describe group/channel sessions as isolated from DMs. The related #19929 is broader channel/thread grouping work, but this issue has a distinct per-user group-to-DM privacy-filtering requirement.

## Best Possible Solution

Keep this open as a core session-routing design item. The best path is an additive, opt-in generic routing model that maps an authorized sender's group and DM turns to a shared canonical session while filtering out other group participants, preserving reply delivery to the originating chat, and updating config schema, docs, Telegram/Discord routing, and shared routing tests. Coordinate with #19929 so the shared-session contract is generic rather than a one-channel special case.

## Risks / Open Questions

- A naive shared key could leak other group participants' messages into a private DM session, which the issue discussion explicitly calls out.
- Changing this behavior crosses an intentional safety boundary where group/channel sessions are non-main and can use different sandbox/tool posture from DMs.
- Reply routing and last-route handling would need to preserve delivery to the originating group or DM even when multiple origins share one canonical session.
- The design overlaps with broader open shared-session routing work in #19929, but this issue is not fully covered by that tracker because it needs per-identity group-to-DM filtering.
