## Summary

Keep this PR open. Current main still fetches only the first Microsoft Teams Graph replies page in `fetchThreadReplies`, while the PR targets a real, narrow gap: long Teams channel threads can lose newer reply context before `[Thread history]` is injected into the agent prompt. The PR is not implemented on main, is not superseded by the supplied related items, and should receive normal MS Teams maintainer review rather than cleanup closure.

## Best Possible Solution

Keep this PR open for MS Teams maintainer review. The best path is to land a narrow Teams-plugin fix that follows Graph `@odata.nextLink` with a hard cap, preserves useful partial results on later-page failures, returns a bounded recent context window only if the ordering assumption is validated, and adds deterministic unit coverage in `extensions/msteams/src/graph-thread.test.ts`. No core API or docs change appears required unless maintainers want to document the long-thread behavior.
