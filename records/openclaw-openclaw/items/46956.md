## Summary

Keep open. Current main still does not implement time-bounded exec approval trust windows, and this PR remains substantive security-sensitive feature work rather than cleanup. The proposed patch is not ready as-is: it introduces a command-approval bypass path with unresolved authorization, deny-policy, localhost-auth, and audit semantics issues, and it overlaps with other open trust-window proposals that need maintainer consolidation.

## What This Changes

The PR adds CLI `approvals trust`, `untrust`, and `trust-status` commands, Gateway RPC/protocol methods, in-memory trust-window state, JSONL audit logging, Swift protocol model updates, and exec/gateway/CLI tests.

## Best Possible Solution

Consolidate the open trust-window proposals into one maintainer-approved design, then land a single implementation that enforces non-agent/operator authorization on the Gateway, preserves or explicitly redefines `security: "deny"`, records all commands that started under a trust window, keeps CLI/Gateway/protocol/Swift/docs in sync, and includes focused security tests for the privilege boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Enforce agent-origin checks server-side:** `src/gateway/server-methods/exec-approvals.ts:135-136`
  - body: The trust RPC currently allows any CLI client with device identity before checking whether the request originated from an agent-run CLI. The only non-agent guard in the PR is a client-side environment-variable check, but current main has no producer for `OPENCLAW_SESSION_KEY` or `OPENCLAW_AGENT_ID`, so an agent can spawn the CLI and reach this new privilege-escalation RPC as an ordinary CLI client. This boundary needs to be enforced from authenticated Gateway/session metadata.
  - confidence: 0.84
- **[P1] Keep localhost shared-auth trust grants usable:** `src/gateway/server-methods/exec-approvals.ts:142`
  - body: This branch allows token/password trust operations only when `isLoopbackAddress(client?.clientIp)` is true, but the WebSocket handler intentionally stores `undefined` for local/loopback `reportedClientIp`. A local CLI using the shared-auth path without device identity, which the PR says is supported, will therefore be rejected by the new gate.
  - confidence: 0.88
- **[P1] Preserve deny policies as a hard ceiling:** `src/infra/exec-approvals.ts:626`
  - body: The trust override promotes a resolved agent with `security: "deny"` to the trust window's `security: "full"`, and the host context then bypasses the normal `minSecurity` cap. Current docs and mainline code treat `deny` as blocking host exec, so this lets a trust grant override an explicit deny policy unless maintainers intentionally redefine that contract and document it.
  - confidence: 0.86
- **[P2] Record commands that started under trust:** `src/agents/bash-tools.exec.ts:570-573`
  - body: The audit append is gated on the trust window being active both at command start and at completion. A command that starts during the trust window but completes after expiry or a concurrent revoke is omitted from the audit and revoke summary, even though it ran under bypassed approval policy.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff introduces a security-sensitive exec approval bypass feature and has concrete authorization, deny-policy, and audit-accounting concerns that should block merge until resolved.

Concerns:

- **[high] Agent-spawned CLI can self-grant trust:** `src/gateway/server-methods/exec-approvals.ts:135`
  - body: The server-side trust gate accepts CLI device identity while the non-agent restriction is enforced only by client-controlled environment variables that current main does not set for exec-spawned CLI processes. That is not a reliable security boundary for a feature that disables approval prompts.
  - confidence: 0.84
- **[high] Trust windows override explicit exec deny policy:** `src/infra/exec-approvals.ts:626`
  - body: The resolver promotes `security: "deny"` to trust-window `security: "full"`, contradicting the current hard-deny behavior and allowing a trust grant to bypass a configured block.
  - confidence: 0.86
- **[medium] Audit can miss trusted commands after expiry or revoke:** `src/agents/bash-tools.exec.ts:570`
  - body: Commands that start while trusted but finish after the window is no longer active are dropped from the JSONL audit trail, weakening the feature's accountability mitigation.
  - confidence: 0.9
