## Summary

Keep open. The trust-window feature is not implemented on current main, and this PR still has blocking security and correctness concerns in the proposed Gateway authorization, hard-deny semantics, and audit accounting; the related open proposals make this a maintainer consolidation item rather than conservative cleanup.

## What This Changes

This PR adds CLI trust, untrust, and trust-status commands, Gateway RPC/protocol and Swift model support, an in-memory trust-window exec approval override, JSONL trust audit logging, exec-host audit integration, and focused tests.

## Best Possible Solution

Consolidate the trust-window work into one maintainer-approved design that uses server-verifiable operator/session metadata, preserves explicit hard-deny policy or intentionally redefines it in code and docs, records every command that began under trust, and updates CLI, Gateway protocol, Swift clients, docs, and tests as one coherent security feature.

## Reproduction Assessment

Yes. The main blocking issues have high-confidence static reproduction paths: simulate a CLI Gateway client with device identity and no server-side agent-origin metadata, resolve a deny-configured agent under an active trust override, and run a command that starts under trust but finishes after revoke or expiry.

## Solution Assessment

No. The current PR direction is useful, but it is not the best mergeable solution until the Gateway owns the non-agent authorization boundary, localhost shared auth matches the advertised behavior, hard-deny semantics are settled, and audit accounting follows command start time.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Enforce non-agent trust grants on the Gateway:** `src/gateway/server-methods/exec-approvals.ts:129-130`
  - body: The trust RPC accepts a CLI client with device identity before proving the request did not originate from an agent-run CLI. The only non-agent guard is the CLI-side env check, but current main has no producer for those env vars, so this privilege-escalation boundary needs server-side authenticated origin or session metadata.
  - confidence: 0.84
- **[P1] Keep localhost shared auth usable for trust:** `src/gateway/server-methods/exec-approvals.ts:136`
  - body: The PR says loopback token/password auth is supported, but this branch requires `isLoopbackAddress(client?.clientIp)`. Current WebSocket handling intentionally stores `undefined` for local reported client IPs, so a local CLI using shared auth without device identity will be rejected.
  - confidence: 0.88
- **[P1] Preserve deny policies as a hard ceiling:** `src/infra/exec-approvals.ts:627-630`
  - body: When a trust window is active, the resolver promotes a resolved agent with `security: "deny"` to `"full"`. Current docs and host-context code treat deny as blocking host exec, so this silently lets a trust grant bypass an explicit deny policy unless maintainers intentionally redefine that contract.
  - confidence: 0.86
- **[P2] Record commands that started under trust:** `src/agents/bash-tools.exec.ts:570-573`
  - body: The audit append is gated on the trust window being active both at command start and at completion. A command that starts during the window but finishes after expiry or revoke is omitted from the audit and revoke summary even though it ran under bypassed approval policy.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: Security review needs attention because the patch introduces a security-sensitive exec approval bypass with concrete authorization, hard-deny, and audit-accounting gaps.

Concerns:

- **[high] Agent-spawned CLI can self-grant trust:** `src/gateway/server-methods/exec-approvals.ts:129`
  - body: The server-side trust gate accepts CLI device identity while the non-agent restriction is enforced only by client-side environment variables that current main does not set for exec-spawned CLI processes. That is not a reliable boundary for a feature that disables approval prompts.
  - confidence: 0.84
- **[high] Trust windows override explicit exec deny policy:** `src/infra/exec-approvals.ts:627`
  - body: The resolver promotes `security: "deny"` to trust-window `security: "full"`, contradicting current hard-deny behavior and allowing a trust grant to bypass a configured host-exec block.
  - confidence: 0.86
- **[medium] Audit can miss trusted commands after expiry or revoke:** `src/agents/bash-tools.exec.ts:570`
  - body: Commands that start while trusted but finish after the window is no longer active are dropped from the JSONL audit trail, weakening the feature's accountability mitigation.
  - confidence: 0.9
