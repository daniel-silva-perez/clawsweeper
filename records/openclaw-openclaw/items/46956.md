## Summary

Keep open. The feature is not implemented on current main, and the PR is a plausible answer to approval fatigue, but the proposed patch is not mergeable as-is because it adds a security-sensitive exec-approval bypass with concrete authorization, local-auth, hard-deny, and audit-accounting issues. The related open trust-window proposals also need maintainer consolidation before choosing one design.

## What This Changes

The PR adds CLI trust, untrust, and trust-status commands, Gateway RPC/protocol and Swift model support, in-memory trust-window resolution, JSONL trust audit logging, exec-host integration, and focused tests.

## Best Possible Solution

Consolidate the trust-window work into one maintainer-approved design that enforces non-agent operator authorization on the Gateway, preserves or deliberately redefines hard-deny semantics in docs and code, records every command that began under trust, and updates CLI, Gateway protocol, Swift clients, docs, and tests together.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Enforce non-agent trust grants on the Gateway:** `src/gateway/server-methods/exec-approvals.ts:129-130`
  - body: The trust RPC accepts a CLI client with device identity before proving the request did not originate from an agent-run CLI. The only non-agent guard is the CLI-side env check, but current main does not set those env vars for exec-spawned CLI processes, so this privilege-escalation boundary needs server-side authenticated origin/session metadata.
  - confidence: 0.84
- **[P1] Keep localhost shared-auth trust grants usable:** `src/gateway/server-methods/exec-approvals.ts:136`
  - body: The PR says loopback token/password auth is supported, but this branch requires `isLoopbackAddress(client?.clientIp)`. The WebSocket handler intentionally stores `undefined` for local reported client IPs, so a local CLI using shared auth without device identity will be rejected.
  - confidence: 0.88
- **[P1] Preserve deny policies as a hard ceiling:** `src/infra/exec-approvals.ts:627`
  - body: When a trust window is active, the resolver promotes a resolved agent with `security: "deny"` to `"full"`. Current docs and host-context code treat deny as blocking host exec, so this silently lets a trust grant bypass an explicit deny policy unless maintainers intentionally redefine that contract.
  - confidence: 0.86
- **[P2] Record commands that started under trust:** `src/agents/bash-tools.exec.ts:570-573`
  - body: The audit append is gated on the trust window being active both at command start and at completion. A command that starts during the window but finishes after expiry or revoke is omitted from the audit and revoke summary even though it ran under bypassed approval policy.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The patch introduces a security-sensitive exec approval bypass and has concrete authorization, hard-deny, and audit-accounting concerns that should block merge until resolved.

Concerns:

- **[high] Agent-spawned CLI can self-grant trust:** `src/gateway/server-methods/exec-approvals.ts:129`
  - body: The server-side trust gate accepts CLI device identity while the non-agent restriction is enforced only by client-side environment variables that current main does not set for exec-spawned CLI processes. That is not a reliable boundary for a feature that disables approval prompts.
  - confidence: 0.84
- **[high] Trust windows override explicit exec deny policy:** `src/infra/exec-approvals.ts:627`
  - body: The resolver promotes `security: "deny"` to trust-window `security: "full"`, contradicting current hard-deny behavior and allowing a trust grant to bypass a configured host-exec block.
  - confidence: 0.86
- **[medium] Audit can miss trusted commands after expiry or revoke:** `src/agents/bash-tools.exec.ts:570`
  - body: Commands that start while trusted but finish after the window is no longer active are dropped from the JSONL audit trail, weakening the feature's accountability mitigation.
  - confidence: 0.9
