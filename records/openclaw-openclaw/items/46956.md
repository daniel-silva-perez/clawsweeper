## Summary

Keep open. Current main still does not implement trust windows, and this PR is an XL security-sensitive exec approval bypass with blocking authorization, hard-deny policy, and audit-accounting issues that should be consolidated with the overlapping open trust-window proposals before merge.

## What This Changes

The PR adds CLI trust, untrust, and trust-status commands, Gateway RPC/protocol and Swift model support, an in-memory trust-window exec approval override, JSONL trust audit logging, exec-host audit integration, and focused tests.

## Best Possible Solution

Consolidate the trust-window work into one maintainer-approved design that uses server-verifiable operator/session metadata, preserves explicit deny policies unless deliberately redefined in code and docs, records every command that began under trust, and updates CLI, Gateway protocol, Swift clients, docs, and tests as one coherent security feature.

## Reproduction Assessment

Yes. The blocking issues have high-confidence static reproduction paths: compare the PR diff against current CLI Gateway identity defaults, current local WebSocket client-IP clearing, current security=deny docs/enforcement, and the PR audit append predicate that requires trust to remain active at command completion.

## Solution Assessment

No. The feature direction is useful, but this PR is not the best mergeable solution until the Gateway owns the non-agent authorization boundary, localhost shared auth matches the advertised behavior, hard-deny semantics are settled, and audit accounting follows command start time.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Enforce non-agent trust grants on the Gateway:** `src/gateway/server-methods/exec-approvals.ts:127-128`
  - body: The trust RPC accepts a CLI client with device identity before proving the request did not originate from an agent-run CLI. The non-agent guard is client-side env checking, while current CLI Gateway calls identify as CLI and load device identity by default, so this privilege-escalation boundary needs server-verifiable origin/session metadata.
  - confidence: 0.84
- **[P1] Keep localhost shared auth usable for trust:** `src/gateway/server-methods/exec-approvals.ts:134`
  - body: The PR says loopback token/password auth is supported, but this branch requires `isLoopbackAddress(client?.clientIp)`. Current WebSocket handling stores `undefined` for local reported client IPs, so local CLI shared-auth calls without device identity will be rejected.
  - confidence: 0.88
- **[P1] Preserve deny policies as a hard ceiling:** `src/infra/exec-approvals.ts:624-625`
  - body: When a trust window is active, the resolver promotes a resolved agent policy to `security: "full"` and `ask: "off"`. Current docs and host-context code treat `security=deny` as blocking host exec, so this silently bypasses an explicit deny policy unless maintainers intentionally redesign that contract.
  - confidence: 0.86
- **[P2] Record commands that start under trust:** `src/agents/bash-tools.exec.ts:570`
  - body: The audit append is gated on the trust window being active both at command start and at completion. A command that starts during the window but finishes after expiry or revoke is omitted from the audit and revoke summary even though it ran under bypassed approval policy.
  - confidence: 0.9
- **[P3] Evict expired trust windows from the cache:** `src/infra/exec-approvals.ts:198-199`
  - body: Expired entries remain in the module-level trust window cache unless explicitly revoked or the gateway restarts. A long-running gateway with many transient agent IDs can accumulate stale trust-window entries for the process lifetime.
  - confidence: 0.72

## Security Review

Status: needs_attention

Summary: Security review needs attention because the patch introduces a security-sensitive exec approval bypass with concrete authorization, hard-deny, and audit-accounting gaps.

Concerns:

- **[high] Agent-spawned CLI can self-grant trust:** `src/gateway/server-methods/exec-approvals.ts:127`
  - body: The server-side trust gate accepts CLI device identity while the non-agent restriction is enforced only by client-side environment variables. That is not a reliable boundary for a feature that disables approval prompts.
  - confidence: 0.84

  - body: The resolver promotes a trusted agent to security full, contradicting current hard-deny behavior and allowing a trust grant to bypass a configured host-exec block.
  - confidence: 0.86
