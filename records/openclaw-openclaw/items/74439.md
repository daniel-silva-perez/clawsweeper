## Summary

Keep open. Current main has substantial SecretRef support for env/file/exec-backed credential references, runtime snapshots, audit/configure/apply/reload flows, and supported credential-surface docs, but it does not implement the requested first-party central vault with per-key ACLs, usage accounting, TTL/expiry policy, rotation commands, or access audit log. This is a broad security/product architecture request rather than a conservative close candidate.

## What This Changes

Request a centralized OpenClaw API key and secret vault with logical references, runtime resolution, migration from existing credential locations, per-key ACLs, usage tracking, rotation/expiry support, audit logging, and CLI management.

## Best Possible Solution

Define the desired secret-management architecture explicitly: either extend the existing SecretRef system with a first-party vault provider and clear ACL/audit/rotation semantics, or document that OpenClaw delegates those controls to external vaults via file/exec providers. Any implementation should preserve current SecretRef compatibility and update the credential-surface docs, CLI reference, gateway runtime tests, and security threat model.
