## Summary

Keep open. The report is fresh and concrete, and current main still lacks direct timing proof that the macOS direct-install WebChat and Telegram pre-dispatch delay before LM Studio starts work is fixed. Adjacent provider-discovery and runtime-deps work on main addresses plausible latency sources, but it does not prove this exact per-message delay is gone.

## What This Changes

The issue reports a 10-15 second per-message delay before normal WebChat/UI and Telegram messages are dispatched from OpenClaw to an LM Studio OpenAI-compatible local endpoint on macOS direct install.

## Best Possible Solution

Keep the issue open until current main or the next release is tested in the reported macOS direct-install LM Studio setup. If the delay remains, capture timestamps at WebChat send/ACK, Telegram receipt/debounce flush, dispatch entry, onAgentRunStart, first LM Studio HTTP request, and any runtime-deps child-process activity, then fix the narrow layer identified by that timing data.

## Reproduction Assessment

No. The reporter gives clear 2026.4.26 steps, but there is no high-confidence reproduction or non-reproduction against current main with both WebChat and Telegram on macOS direct install using LM Studio.

## Solution Assessment

Unclear. The current adjacent fixes are plausible mitigations, but the best fix depends on timing proof that separates channel ingress, queueing, provider discovery, runtime-deps work, and first model HTTP dispatch.
