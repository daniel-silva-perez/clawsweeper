## Summary

Keep open. Current main still does not automatically recover an untracked, unhealthy OpenClaw-managed Chrome/CDP listener holding the configured loopback port. The code can reuse healthy CDP, restart a tracked child process, and clean stale Chromium Singleton locks, but it still treats an untracked occupied CDP port as a refusal path rather than verifying ownership and terminating the stale Chrome process. The workaround comments, including ClawAid/manual port cleanup, are useful recovery guidance but do not replace a core browser-plugin fix.

## Best Possible Solution

Keep this issue open and implement browser-plugin-owned stale managed Chrome recovery. Before managed launch, probe the loopback CDP endpoint with a short timeout; if the port is occupied but unhealthy, inspect the listener, verify it is an OpenClaw-managed Chrome/Chromium process for that profile, terminate it, wait for the port to clear, and relaunch. Add regression coverage for unhealthy owned listener cleanup, healthy pre-existing browser reuse, attachOnly/remote refusal, unrelated listener refusal, tracked restart behavior, and launch cooldown behavior.
