## Summary

Keep open. Current main still does not recover the reported stale managed Chrome/CDP-port case: it probes CDP health, clears stale Chromium Singleton lock files, and restarts tracked child processes, but an untracked OpenClaw-managed Chrome/Chromium process that remains bound to port 18800 while unhealthy still causes launch failure or a "not by openclaw" refusal. The related attempted fix in #41758 is closed unmerged.

## Best Possible Solution

Keep this issue open and implement owner-scoped recovery in the bundled browser plugin. Before launching a local managed profile, probe the loopback CDP endpoint with a short timeout; when the port is occupied but unhealthy, inspect the listener and terminate only OpenClaw-owned Chrome/Chromium processes for that profile, wait for the port to clear, and launch fresh. Add regression tests for unhealthy owned listener cleanup, healthy pre-existing browser reuse, attachOnly/remote refusal, and unrelated listener refusal.

## Risks / Open Questions

- A fix must not kill arbitrary services on the same loopback port; it should terminate only Chrome/Chromium processes that match the expected remote-debugging port and the OpenClaw-managed browser user data directory.
- The report uses "zombie" colloquially, so the robust trigger should be failed CDP health/readiness plus ownership checks, not RSS size or Unix zombie-state heuristics.
- Cross-platform listener inspection and termination need focused tests for macOS, Linux, and Windows, plus attachOnly/remote refusal and healthy existing-browser reuse.
