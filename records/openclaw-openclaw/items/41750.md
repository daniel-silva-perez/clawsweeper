## Summary

Keep open. Current main still does not automatically recover the reported stale OpenClaw-managed Chrome/CDP-port case. The browser plugin probes CDP health and can restart a tracked child process, but an untracked Chrome/Chromium process still holding port 18800 is treated as a busy or not-owned port rather than being owner-checked and terminated.

## Best Possible Solution

Keep this issue open and implement browser-plugin-owned recovery for stale managed Chrome listeners. Before managed launch, probe the loopback CDP endpoint with a short timeout; when the port is occupied but unhealthy, inspect the listener, verify it is an OpenClaw-managed Chrome/Chromium process for that profile, terminate it, wait for the port to clear, and relaunch. Add regression tests for unhealthy owned listener cleanup, healthy pre-existing browser reuse, attachOnly/remote refusal, unrelated listener refusal, and launch cooldown behavior.

## Risks / Open Questions

- A future fix must not kill arbitrary services on the same loopback port; it should terminate only Chrome/Chromium processes that match the expected OpenClaw-managed remote-debugging port and user-data directory.
- The report uses 'zombie' colloquially, so the implementation should key off failed CDP health/readiness plus ownership checks rather than RSS size or Unix zombie-state heuristics.
- Cross-platform listener inspection and termination need focused coverage for macOS, Linux, and Windows, plus attachOnly/remote refusal and healthy existing-browser reuse.
