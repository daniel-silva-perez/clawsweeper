## Summary

Current main still leaves this managed-browser recovery bug open. OpenClaw can reuse a healthy pre-existing loopback CDP endpoint, restart a tracked child Chrome process, and clear stale Chromium Singleton lock files, but it still does not owner-check and terminate an untracked unhealthy OpenClaw-managed Chrome process holding the CDP port. The ClawAid/manual cleanup comments are useful operational workarounds, and related PR #41758 was an attempted fix, but that PR is closed unmerged and the behavior is not shipped on main.

## Best Possible Solution

Keep this issue open. Implement browser-plugin-owned stale managed Chrome recovery: before managed launch, probe the loopback CDP endpoint with a short timeout; if the port is occupied but unhealthy, inspect the listener, verify it is a Chrome/Chromium process with the expected --remote-debugging-port and OpenClaw-managed --user-data-dir for that profile, terminate it gracefully with forced escalation if needed, wait for the port to clear, and relaunch. Preserve attachOnly/remote external-CDP refusal behavior and add regression coverage for unhealthy owned listener cleanup, healthy pre-existing browser reuse, unrelated listener refusal, tracked restart, and launch cooldown behavior.

## Risks / Open Questions

- Closing this now would lose a still-current managed browser recovery bug that can leave cron or scheduled browser workflows stuck until manual port cleanup.
- A future fix must avoid killing arbitrary loopback services; it should terminate only Chrome/Chromium processes matching the expected OpenClaw-managed CDP port and profile user-data directory.
- The report uses 'zombie' colloquially, so the implementation should key off CDP health/readiness plus ownership checks rather than RSS size or Unix zombie-state heuristics.
