## Summary

Current main still leaves this managed-browser recovery bug open. The browser plugin can reuse a healthy pre-existing loopback CDP endpoint, restart an in-memory tracked Chrome child, and clear stale Chromium Singleton lock files, but it still refuses or fails before owner-checking an untracked unhealthy Chrome process that holds the CDP port. The ClawAid/manual cleanup comments are useful operational workarounds, and related PR #41758 was an attempted fix, but that PR is closed unmerged and the behavior is not present on main.

## Best Possible Solution

Keep this issue open. Implement browser-plugin-owned stale managed Chrome recovery before managed launch: probe the loopback CDP endpoint with a short timeout; if the port is occupied but unhealthy, inspect the listener, verify it is Chrome/Chromium with the expected --remote-debugging-port and OpenClaw-managed --user-data-dir for that profile, terminate it with graceful-to-forced escalation, wait for the port to clear, and relaunch. Preserve attachOnly and remote external-CDP refusal behavior, and add regression coverage for owned unhealthy listener cleanup, healthy pre-existing browser reuse, unrelated listener refusal, tracked restart, and launch cooldown behavior.

## Risks / Open Questions

- Closing this now would lose a still-current managed browser recovery bug that can leave scheduled browser workflows stuck until manual port cleanup.
- A future fix must avoid killing arbitrary loopback services; it should terminate only Chrome/Chromium processes matching the expected OpenClaw-managed CDP port and profile user-data-dir.
- The report uses 'zombie' colloquially, so the implementation should key off CDP health/readiness plus ownership checks rather than RSS size or Unix zombie-state heuristics.
