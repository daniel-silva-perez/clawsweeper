## Summary

Keep open. Current main still lacks the targeted obsolete-key cleanup, and the PR appears to address the linked crash path, including the per-agent gap raised in review. The remaining issue I found is a non-blocking test-quality problem: one added test calls an invalid `memorySearch.limit` key valid, so the coverage should be tightened before merge.

## What This Changes

The PR adds a legacy config migration and tests to remove obsolete `chunkSize`, `chunkOverlap`, and root-level `maxResults` keys from `agents.defaults.memorySearch` and `agents.list[].memorySearch`.

## Best Possible Solution

Keep the linked bug and this PR open, land a narrow runtime/doctor migration for these retired memorySearch keys, and tighten the test to use a real schema-valid survivor or add explicit validation-path coverage. This should stay scoped to known retired keys, not become a generic unknown-key stripping change.

## Reproduction Assessment

Yes. A config containing `agents.defaults.memorySearch.chunkSize` or `agents.list[0].memorySearch.maxResults` follows the read-time migration path and then reaches the strict `MemorySearchSchema`; current main does not strip those keys, so validation can reject the config.

## Solution Assessment

Yes, with a small test correction. The PR’s scoped strip-after-move migration is narrower and safer than changing global config strictness or validation order, and it covers the per-agent override gap raised during review.
