## Summary

Keep open. Current main still implements the bundled session-memory hook as command-only persistence for `/new` and `/reset`; idle/daily rollover, gateway shutdown, message-count checkpoint triggering, trigger configuration, and concise summary output remain absent. The related idle/daily PRs are open partial implementations, so this broader tracker still needs maintainer consolidation.

## What This Changes

The issue requests automatic session-memory checkpoints on idle, shutdown, and message-count triggers, plus optional summary output instead of raw recent transcript persistence.

## Best Possible Solution

Keep this as the broader session-memory automation tracker. The maintainable end state should consolidate the daily/idle rollover path, then add a documented trigger configuration surface for shutdown-safe checkpoints, periodic message-count checkpoints, optional concise summaries, and focused regression tests while preserving existing `/new` and `/reset` behavior.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: enable session-memory and exercise idle/daily rollover, gateway shutdown, or ordinary message flow; current main only registers and accepts `command:new` and `command:reset`, and tests assert non-command events do not create memory output.

## Solution Assessment

Unclear as a single implementation. The requested direction is valid, but the best fix should first settle the overlapping idle/daily PR path and then separately design shutdown durability, message-count checkpoint state, and summary-output behavior.
