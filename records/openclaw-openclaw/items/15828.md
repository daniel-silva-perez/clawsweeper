## Summary

Keep open. Current main still saves the bundled session-memory hook only for explicit `/new` and `/reset` command events; idle/daily rollover, gateway shutdown, message-count checkpoints, trigger configuration, and concise summary output remain unimplemented, while the related PRs are only partial open implementations.

## What This Changes

Request to add automatic session-memory checkpoints for idle, shutdown, periodic message-count, and `/new` triggers, plus optional LLM-generated concise summaries.

## Best Possible Solution

Keep this as the broader session-memory automation tracker. The maintainable end state should consolidate the daily/idle rollover PR path, then add a documented trigger configuration surface for shutdown-safe checkpoints, periodic message-count checkpoints, optional concise summaries, and focused regression tests while preserving existing `/new` and `/reset` behavior.

Yes. The source-level reproduction is high confidence: enable session-memory and trigger daily/idle rollover, gateway shutdown, or ordinary message flow; current main only registers and accepts `command:new` and `command:reset`, and tests explicitly skip non-command events.

## Solution Assessment

Unclear as a single implementation. The requested direction is reasonable, but the best fix should first settle the overlapping daily/idle PR path and then separately design shutdown durability, message-count state, and summary-output semantics.
