## Summary

Keep open. Current main still saves bundled session-memory only for explicit `/new` and `/reset` command events; the requested idle, shutdown, periodic message-count, trigger-config, and concise-summary behavior remains unimplemented, and the related daily/idle PRs are only partial overlapping work.

## What This Changes

The issue requests automatic session-memory checkpointing on idle, shutdown, and periodic message-count triggers, with configurable trigger options and optional concise LLM summaries.

## Best Possible Solution

Keep this as the broader session-memory automation tracker. The end state should consolidate the daily/idle rollover work, then add a documented trigger configuration surface for shutdown-safe checkpoints, periodic message-count checkpoints, optional concise summaries, and focused regression tests while preserving existing `/new` and `/reset` behavior.

Yes. The source-level reproduction is high confidence: enable session-memory and trigger daily/idle rollover, gateway shutdown, or ordinary message flow; current main only registers and accepts `command:new` and `command:reset`, with tests explicitly skipping non-command events.
