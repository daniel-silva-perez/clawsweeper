## Summary

Keep open. Current main still implements bundled session-memory persistence as command-only `/new` and `/reset` handling; the requested idle/shutdown/message-count trigger configuration and concise summary-output mode are still missing, and the related daily/idle PRs remain partial open implementation context rather than a merged fix.

## What This Changes

Request automatic session-memory checkpoints on idle, shutdown, and periodic message counts, with configurable triggers and optional concise LLM summaries.

## Best Possible Solution

Keep this as the broader session-memory automation tracker. The desired end state is a documented trigger configuration surface that preserves existing `/new` and `/reset` behavior while adding maintained daily/idle, shutdown-safe, message-count, and optional summary-output paths with focused regression coverage.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: enable session-memory, then exercise stale daily/idle rollover, gateway shutdown, or normal message flow; current main only registers and accepts `command:new` and `command:reset`, and tests assert non-command events do not write memory.

## Solution Assessment

Unclear as a single one-shot change. The requested direction is valid, but the maintainable fix should first reconcile the overlapping daily/idle PRs, then separately design shutdown durability, message-count state, and summary-output behavior.
