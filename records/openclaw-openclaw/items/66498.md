## Summary

Keep open. Current main still rejects Gemini `web_search` freshness/date filters before constructing a Gemini request, still sends an empty Google Search grounding tool, and the Gemini docs still say freshness is unsupported. The request is narrow and actionable in the bundled Google plugin because Google's current API reference documents `GoogleSearch.timeRangeFilter` for Search grounding. citeturn1search2

## What This Changes

The issue requests making Gemini `web_search` respect freshness/date parameters by passing them to Google Search grounding instead of returning unsupported-filter errors.

## Best Possible Solution

Keep this issue open until the Google plugin converts supported OpenClaw freshness/date inputs into Gemini Google Search grounding time ranges, includes the normalized time range in caching, and updates focused tests plus Gemini/web-search docs. The fix should stay in the bundled Google web-search provider rather than broadening into the larger native-search passthrough work tracked by #17925.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: call the Gemini `web_search` execution path with a non-empty `freshness`, `date_after`, or `date_before` value and current main returns the shared unsupported-filter response before credential resolution or any Gemini request is built.

## Solution Assessment

Yes. The narrowest maintainable solution is to update the Google plugin/runtime to map existing shared freshness/date inputs to Gemini's `GoogleSearch.timeRangeFilter`, while leaving unrelated filters like country/language/domain unsupported unless a separate provider API supports them.
