## Summary

Keep this open: current main still has the reported Anthropic Messages tool-use unsafe-integer path. The runtime now uses OpenClaw's boundary-aware Anthropic transport rather than only the old SDK MessageStream path, but that transport still accumulates raw `input_json_delta` text and parses it through `parseStreamingJson`, whose pinned implementation reaches `JSON.parse`/`partial-json` without the Ollama unsafe-integer guard.

## What This Changes

The issue asks OpenClaw to preserve large integer tool-use input arguments in the Anthropic Messages streaming path instead of rounding them through JavaScript number parsing.

## Best Possible Solution

Add a neutral unsafe-integer-preserving JSON parse wrapper for raw streamed tool argument JSON, use it before Anthropic `input_json_delta` text reaches `parseStreamingJson`/`JSON.parse`, and add Anthropic regression coverage that proves Discord-sized bare numeric IDs are preserved as exact strings while safe integers remain numbers.

## Reproduction Assessment

Yes. Feeding an Anthropic `tool_use` block whose accumulated `input_json_delta` is `{"to":1481220477346119781}` through the current parse path will round the value because the path reaches `JSON.parse`; the same primitive was confirmed with a read-only Node probe.

## Solution Assessment

Yes, the narrow maintainable fix is to guard the raw accumulated Anthropic tool-input JSON before any parser converts numeric literals to JavaScript Numbers. Monkey-patching vendored dependencies is less maintainable than a shared or Anthropic-local wrapper covered by transport tests.
