## Summary

Keep this issue open. The exact Anthropic SDK accumulator path described in the report has been replaced on current main by OpenClaw’s boundary-aware Anthropic transport, but the observable bug remains: streamed Anthropic `input_json_delta` tool arguments are still parsed through a `JSON.parse`-based parser that rounds unsafe integer literals before tool execution.

## Best Possible Solution

Keep the issue open and implement a shared provider-stream JSON parser that quotes unsafe integer literals before `JSON.parse`, reusing or extracting the proven Ollama guard rather than leaving it extension-local. Apply it to Anthropic `input_json_delta` parsing, consider the OpenAI transport call sites that also use `parseStreamingJson`, and add a regression in `src/agents/anthropic-transport-stream.test.ts` where streamed `tool_use.input` contains a bare Discord-snowflake-sized integer and `toolCall.arguments` preserves it exactly as a string.

## Evidence

- **Current Anthropic tool input path parses accumulated input JSON without an unsafe-integer guard:** `content_block_delta` appends `delta.partial_json` to `block.partialJson`, assigns `block.arguments = parseStreamingJson(block.partialJson)`, and repeats the parse at `content_block_stop`. No `quoteUnsafeIntegerLiterals` or equivalent guard is present on this path.
  - file: [src/agents/anthropic-transport-stream.ts:1018](https://github.com/openclaw/openclaw/blob/0f3a9d812b40cbc8e545abcb80a1f12a93946cf6/src/agents/anthropic-transport-stream.ts#L1018)
  - command: `nl -ba src/agents/anthropic-transport-stream.ts | sed -n '948,1022p;1058,1070p'`
  - sha: [0f3a9d812b40](https://github.com/openclaw/openclaw/commit/0f3a9d812b40cbc8e545abcb80a1f12a93946cf6)
- **The parser dependency rounds complete valid JSON before any partial-parser fallback:** The locked `@mariozechner/pi-ai@0.70.5` implementation of `parseStreamingJson` calls `parseJsonWithRepair`, which first calls `JSON.parse(json)`. For complete valid tool input JSON, the fallback partial parser is not reached, so bare integers above `Number.MAX_SAFE_INTEGER` are converted to imprecise JavaScript numbers.
  - command: `curl -fsSL https://registry.npmjs.org/@mariozechner/pi-ai/-/pi-ai-0.70.5.tgz | tar -xOzf - package/dist/utils/json-parse.js | nl -ba | sed -n '50,112p'`
- **The reported numeric example still loses precision through the parser primitive used by the Anthropic path:** A direct Node check with the issue’s example shows `JSON.parse('{"to":1481220477346119781}')` produces `1481220477346119700` and `Number.isSafeInteger` is false.
  - command: `node -e 'const raw = "{\"to\":1481220477346119781}"; const parsed = JSON.parse(raw); console.log(JSON.stringify({raw, parsed: parsed.to, asString: String(parsed.to), safe: Number.isSafeInteger(parsed.to)}));'`

## Risks / Open Questions

- Fixing this in `parseStreamingJson` may affect OpenAI transport paths too, because they also import the same parser; that is likely desirable but should be covered with focused provider-stream tests.
- Partial streaming previews should keep returning useful partial objects while final complete JSON preserves unsafe integers as exact strings.
