## Summary

Current main still has the reported Anthropic Messages tool-use precision-loss path: streamed `input_json_delta` fragments are accumulated into `partialJson` and parsed through `@mariozechner/pi-ai` `parseStreamingJson`, whose pinned implementation reaches `JSON.parse` and `partial-json` without OpenClaw's Ollama unsafe-integer guard. No linked open PR or current-main implementation appears to supersede this issue.

## Best Possible Solution

Keep this issue open and fix the active Anthropic Messages transport parser so accumulated raw `input_json_delta` JSON is parsed with an unsafe-integer-preserving guard before any `JSON.parse` or `partial-json` path can round bare large integers. Prefer a neutral boundary-safe helper if it can also serve Ollama without crossing plugin boundaries; otherwise add a narrow Anthropic-local wrapper and regression coverage.

## Repair Work Prompt

Fix the Anthropic Messages tool-use unsafe-integer precision-loss bug tracked by https://github.com/openclaw/openclaw/issues/47229. Current `src/agents/anthropic-transport-stream.ts` accumulates `input_json_delta` into `block.partialJson` and calls `parseStreamingJson`, which reaches `JSON.parse`/`partial-json` and rounds bare integers larger than `Number.MAX_SAFE_INTEGER`. Add an unsafe-integer-preserving parse path for Anthropic tool-use input before any parser can round it. Add regression coverage that streams a tool_use input such as `{"to":1481220477346119781}` and asserts the final tool-call argument is the exact string `"1481220477346119781"`.
