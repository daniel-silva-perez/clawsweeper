## Summary

Keep this open. Current main still routes Anthropic Messages streamed tool-use input through `parseStreamingJson`, whose pinned dependency implementation uses `JSON.parse` for complete JSON, so Discord snowflake-sized bare numeric arguments can still be rounded before OpenClaw executes the tool.

## What This Changes

The issue asks OpenClaw to preserve large integer precision for Anthropic streamed `tool_use.input` arguments, matching the prior Ollama unsafe-integer guard behavior.

## Best Possible Solution

Add a core Anthropic transport parse helper that preserves unsafe integer literals as strings before any numeric coercion occurs, with focused mocked SSE regression coverage for `input_json_delta` and no dependency monkey-patching or core imports from the Ollama plugin.

## Reproduction Assessment

Yes. A mocked Anthropic SSE stream with a `tool_use` block and `input_json_delta` containing `{"to":1481220477346119781}` should reach the current parser path, and the parser-level Node probe confirms the rounded value.

## Solution Assessment

Yes. A guard at the Anthropic streamed tool-argument parse boundary is the narrow maintainable fix; monkey-patching dependency internals or post-processing already-rounded numbers would be more fragile.
