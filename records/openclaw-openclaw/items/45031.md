## Summary

Keep open. Current main has registry-side ClawHub scan summaries and a built-in scanner for the separate skill dependency-install path, but native ClawHub skill install/update still downloads and copies skill folders without a local scan or policy gate. The AgentShield and ClawGuard comments are useful candidate backend context, but this is security-sensitive and overlaps open protected security work, so it needs maintainer policy rather than stale closure or an automated repair PR.

## What This Changes

Request built-in security scanning and policy enforcement for skills installed or updated through ClawHub, manual skill loading, and ClawHub publishing.

## Best Possible Solution

Define a first-class OpenClaw skill supply-chain policy for native ClawHub install and update: local scan execution before copy/activation, scan-on-update, critical-finding blocking, explicit operator overrides, provenance/version evidence, CLI/Gateway/UI result surfacing, and docs/threat-model alignment. AgentShield and ClawGuard can remain candidate scanner backends or plugin integrations unless maintainers choose one as the default.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists by tracing `openclaw skills install <slug>` or Gateway `skills.install` with `source: "clawhub"` into `src/agents/skills-clawhub.ts`; current main downloads, extracts, and installs the archive without invoking `scanSkillInstallSource`, and `openclaw skills update` reuses the same forced install path.

## Solution Assessment

Unclear for an AgentShield-only integration. The narrow maintainable direction is a backend-agnostic core scan/policy seam for native ClawHub install/update, with third-party scanners and ClawHub registry scan data plugged in only after maintainers settle the default policy model.
