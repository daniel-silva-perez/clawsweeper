## Summary

Keep this PR open. Current main still has no `safetyPosture` config surface, so the feature is not obsolete, but the PR is not merge-ready because it accepts security-posture fields while leaving key advertised effects unenforced and its runtime hook is stale relative to the current config materialization flow.

## What This Changes

The PR adds a top-level `safetyPosture` config type/schema, generated schema metadata, runtime preset expansion for sandbox/session settings, tests, and minor extension-test helper import rewrites.

## Best Possible Solution

Keep the PR as an active implementation candidate, but require revision before merge: rebase onto current main, integrate preset expansion into `materializeRuntimeConfig`, translate profiles into existing `tools.profile`/deny-list policy or introduce a real typed profile contract, make `secureDmMode` independently effective, either map memory posture to concrete memory config or remove that promise, and update docs/schema/tests around effective sandbox, session, tool, and memory behavior.

## Reproduction Assessment

Yes. Static reproduction is high confidence: current main's strict schema has no `safetyPosture` key, and the PR patch shows accepted `agentProfile`, `secureDmMode`, and memory preset fields that are not wired into the runtime policies they advertise.

## Solution Assessment

No, not as currently implemented. Additive safety presets are a reasonable direction, but the PR needs to map them onto current runtime config and tool/memory/session policy rather than storing private no-op state.

  - confidence: 0.9

  - body: The strict preset defines `longTermMemoryEnabled: false`, but runtime expansion explicitly skips mapping `MEMORY_PRESETS`. That makes strict/public posture appear to disable long-term memory while no memory config changes occur.
  - confidence: 0.86
