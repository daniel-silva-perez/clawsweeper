## Summary

Keep this PR open: current main still drops OpenAI TTS speed directives, and the PR is a focused provider-owned fix for that remaining bug. It should not be closed as implemented because the fix has not landed, and it still needs maintainer rebase/check follow-up before merge.

## What This Changes

The PR adds OpenAI-owned `speed`/`openai_speed`/`openaispeed` TTS directive parsing with range validation, focused OpenAI speech-provider tests, and a changelog entry.

## Best Possible Solution

Land a rebased version of the provider-owned OpenAI parser fix with the focused regression tests, keep generic directive routing in core unchanged, and track telephony override behavior separately if maintainers want directives to affect telephony synthesis too.

## Reproduction Assessment

Yes. On current main, the OpenAI provider parser returns unhandled for `key: "speed"`, and `parseTtsDirectives` only forwards provider overrides after a handled parser result; no live OpenAI API call is needed to reproduce the parser drop.

## Solution Assessment

Yes for regular OpenAI speech synthesis. The provider-local parser case is the narrowest maintainable solution because TTS speed ranges are provider-specific; the safer alternative for telephony is a separate contract-aware follow-up rather than expanding this parser-only PR silently.
