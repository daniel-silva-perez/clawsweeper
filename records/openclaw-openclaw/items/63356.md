## Summary

Keep this PR open. Current main still normalizes Signal shared allowlist formatting and shared DM-policy entries through phone-only E.164 paths, while the Signal docs and pairing flow support `uuid:<id>` senders. The PR is still a focused Signal-owned fix with regression coverage and no observed supply-chain or CI/dependency surface in its provided diff.

## Best Possible Solution

Rebase and review this PR, then land it or an equivalent Signal-owned normalizer used by both `signalConfigAdapter.formatAllowFrom` and `signalSecurityAdapter.normalizeDmEntry`. Keep regression coverage for `uuid:`, `signal:uuid:`, wildcard preservation, invalid group/username rejection, and E.164 phone normalization; either canonicalize compact UUIDs to the runtime form or explicitly defer that edge in review.
