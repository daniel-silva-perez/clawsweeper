## Summary

Keep open. Current main still lacks the proposed watchdog command and `startup-error.txt` writer, and the PR branch still has blocking package/runtime/auth and stale-diagnostic bugs before it can merge. The feature also overlaps broader watchdog product work in #46502 and #36314, so maintainer direction is needed.

## What This Changes

The PR adds a foreground `openclaw gateway watchdog` command backed by a Bash health/restart script and adds startup-error file persistence for fatal CLI/Gateway startup failures.

## Best Possible Solution

Land only a rebased and reviewed implementation whose watchdog boundary is clear. The final shape should package any runtime assets, use the current stability/fatal-hook architecture, honor documented auth defaults, clear diagnostics when startup actually succeeds, validate CLI input before spawning shell code, handle child-process errors, document the public command, and include focused CLI/infra/script tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Ship the watchdog script with the packaged CLI:** `src/cli/gateway-cli/run.ts:573`
  - body: The command resolves `scripts/openclaw-watchdog.sh` at runtime, but the package `files` whitelist does not include that new script. Installed package users would get a `gateway watchdog` command that points at a missing file unless the script is moved into a shipped runtime asset or explicitly packaged.
  - confidence: 0.94
- **[P1] Read the watchdog token from the environment default:** `scripts/openclaw-watchdog.sh:22`
  - body: The CLI advertises `OPENCLAW_GATEWAY_TOKEN` as the default and forwards it to the child environment, but the script initializes `TOKEN` to an empty string unless `--token` is passed. Token-protected gateways using env auth will be treated as unhealthy and may enter restart/manual-intervention flows incorrectly.
  - confidence: 0.88
- **[P2] Clear startup errors after startup actually succeeds:** `src/cli/gateway-cli/run.ts:463-466`
  - body: This cleanup runs only after `await startLoop()` returns, but the Gateway loop is designed to stay running until shutdown or restart handling. On normal successful startup the block is not reached, leaving stale `startup-error.txt` content visible after recovery.
  - confidence: 0.88
- **[P2] Reject invalid watchdog numeric options before spawning:** `src/cli/gateway-cli/run.ts:585-590`
  - body: Both numeric parsers can return `NaN`, and those values are forwarded into shell arithmetic. Inputs like `--interval foo` or `--max-attempts foo` should fail with a Commander validation error instead of starting a script that aborts with an opaque shell error.
  - confidence: 0.9
- **[P2] Handle child process spawn errors with an error listener:** `src/cli/gateway-cli/run.ts:615`
  - body: Wrapping `spawn("bash", ...)` in `try/catch` does not handle normal spawn failures such as `ENOENT`; Node emits those on the returned `ChildProcess` as an `error` event. Without an `error` listener, missing Bash can become an uncaught exception instead of a controlled CLI error.
  - confidence: 0.86
