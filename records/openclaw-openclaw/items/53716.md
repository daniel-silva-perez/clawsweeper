## Summary

Keep open. Current main still does not provide the proposed `openclaw gateway watchdog` command, so the PR is not obsolete, but the branch remains not merge-ready because the packaged CLI would not ship the watchdog script, token-protected setups can be misdetected as down, startup-error cleanup is unreachable after successful long-running startup, invalid numeric options reach shell arithmetic, and spawn failures are not handled as controlled CLI errors.

## What This Changes

The PR adds startup-error file persistence for fatal CLI/Gateway startup failures and registers an `openclaw gateway watchdog` subcommand that spawns a Bash health-check, restart, logging, and desktop-notification script.

## Best Possible Solution

Land a rebased and reviewed implementation only after the watchdog boundary is clear. The final shape should either split startup-diagnostic persistence from a packaged first-class watchdog command, or be superseded by the broader rescue-watchdog service; it should use the current stability/fatal-hook architecture, ship runtime assets, honor env-token auth, validate CLI input before shell execution, handle child-process failures, document the public command, and include focused CLI/infra/script tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Ship the watchdog script with the packaged CLI:** `src/cli/gateway-cli/run.ts:573`
  - body: The new command resolves `scripts/openclaw-watchdog.sh` at runtime, but the package file list does not include that script. Installed package users would get a `gateway watchdog` command that points at a missing file, so the feature is unusable outside a source checkout unless the script is packaged or moved into a shipped runtime asset.
  - confidence: 0.94
- **[P1] Initialize the token from OPENCLAW_GATEWAY_TOKEN:** `scripts/openclaw-watchdog.sh:22`
  - body: The CLI advertises `OPENCLAW_GATEWAY_TOKEN` as the default and forwards it to the child environment, but the script resets `TOKEN` to an empty string unless `--token` is passed. Token-protected gateways that rely on the env default will return auth failures and the watchdog can incorrectly enter restart/manual-intervention flows.
  - confidence: 0.91
- **[P2] Clear startup errors after startup actually succeeds:** `src/cli/gateway-cli/run.ts:463-466`
  - body: This cleanup runs only after `await startLoop()` returns, but the Gateway loop is designed to stay running until shutdown or exit. On normal successful startup this block never executes, leaving stale `startup-error.txt` content visible after the Gateway has recovered.
  - confidence: 0.88
- **[P2] Reject invalid watchdog numeric options before spawning:** `src/cli/gateway-cli/run.ts:583-590`
  - body: Both numeric parsers can return `NaN` (`parseInt("foo", 10)` and `Math.max(1, NaN)`), and that value is forwarded into shell arithmetic. The script then aborts with a shell arithmetic error instead of a clear Commander validation failure for invalid `--interval` or `--max-attempts` input.
  - confidence: 0.9
- **[P2] Handle watchdog spawn errors explicitly:** `src/cli/gateway-cli/run.ts:615-623`
  - body: Wrapping `spawn("bash", ...)` in try/catch does not handle normal spawn failures such as `ENOENT`; Node emits those on the returned `ChildProcess` as an `error` event. Without an `error` listener, a missing Bash runtime can become an uncaught exception instead of a controlled CLI error.
  - confidence: 0.86
