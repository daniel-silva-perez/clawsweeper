## Summary

Keep open. Current main has Gateway stability diagnostics and fatal-error hook infrastructure, but it does not implement this PR's watchdog command/script or fully supersede the recovery/notification behavior. The branch still has merge-blocking functional issues and overlaps with the broader open rescue-watchdog design thread, so it needs maintainer direction rather than conservative cleanup closure.

## What This Changes

The PR adds a `openclaw gateway watchdog` subcommand backed by a Bash watchdog script, plus startup-error file persistence for fatal Gateway/CLI startup failures.

## Best Possible Solution

Keep the PR open for maintainer review and rebase direction. The final shape should either split out a narrow startup-diagnostics improvement that fits the current stability/fatal-hook architecture, or supersede the watchdog portion with the broader rescue-watchdog service. Any public watchdog command should be packaged, documented, validate CLI input before shell execution, honor documented auth defaults, and include focused CLI/script tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Ship the watchdog script with the packaged CLI:** `src/cli/gateway-cli/run.ts:573`
  - body: The command resolves `scripts/openclaw-watchdog.sh` at runtime, but the package `files` whitelist does not include that script. Installed npm users would get a `gateway watchdog` command that points at a missing file unless the script is packaged or moved into a shipped runtime asset.
  - confidence: 0.94
- **[P2] Clear startup errors after startup actually succeeds:** `src/cli/gateway-cli/run.ts:463-466`
  - body: The cleanup runs after `await startLoop()`, but the Gateway loop is long-running and normally does not resolve after a successful bind. A stale `startup-error.txt` can therefore survive normal recovery and keep reporting an obsolete startup failure.
  - confidence: 0.88
- **[P2] Reject invalid watchdog numeric options before spawning:** `src/cli/gateway-cli/run.ts:585-590`
  - body: The option parsers can return `NaN` for inputs such as `--interval foo` or `--max-attempts foo`, and that value is forwarded into shell arithmetic. The command should fail fast with a Commander validation error instead of starting a script that aborts with an opaque shell error.
  - confidence: 0.9
- **[P2] Handle child process spawn errors explicitly:** `src/cli/gateway-cli/run.ts:615-623`
  - body: Wrapping `spawn("bash", ...)` in `try/catch` does not catch normal spawn failures such as `ENOENT`; Node emits those on the child `error` event. Add an error listener so missing Bash or execution failures become controlled CLI errors.
  - confidence: 0.86
- **[P1] Read the watchdog token from the environment default:** `scripts/openclaw-watchdog.sh:22`
  - body: The CLI advertises `OPENCLAW_GATEWAY_TOKEN` as the default token source and forwards it in the child environment, but the script initializes `TOKEN` to an empty string and only updates it from `--token`. Token-protected Gateway setups that rely on the env var will be treated as unhealthy and can enter restart flows incorrectly.
  - confidence: 0.82
