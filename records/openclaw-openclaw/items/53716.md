## Summary

Keep open. Current main has related Gateway stability-bundle diagnostics, but it still does not provide the proposed `gateway watchdog` command or packaged external watchdog. The PR is useful source work, but it is not merge-ready because the branch still has concrete CLI/script correctness issues and overlaps the broader open rescue-watchdog design in #46502.

## What This Changes

The PR adds startup-error file persistence for fatal CLI/Gateway startup failures and registers a `openclaw gateway watchdog` subcommand that spawns a Bash script to poll Gateway health, restart the Gateway, log events, and send desktop notifications.

## Best Possible Solution

Land a rebased and reviewed implementation only after the watchdog boundary is clear. The final shape should either split startup-diagnostic persistence from a packaged first-class watchdog command, or be superseded by the broader rescue-watchdog service. It should use the current stability/fatal-hook architecture, package any runtime assets, validate CLI input before shell execution, document the user-facing command, and include focused CLI/infra/script tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Ship the watchdog script with the packaged CLI:** `src/cli/gateway-cli/run.ts:573`
  - body: The new command resolves `scripts/openclaw-watchdog.sh` at runtime, but the package file list does not include that script. Installed package users would get a `gateway watchdog` command that points at a missing file, so the feature is unusable outside a source checkout unless the script is packaged or moved into a shipped runtime asset.
  - confidence: 0.94
- **[P2] Clear startup errors after startup actually succeeds:** `src/cli/gateway-cli/run.ts:463-466`
  - body: This cleanup runs only after `await startLoop()` returns, but the Gateway loop is designed to stay running until shutdown or exit. On normal successful startup this block never executes, leaving stale `startup-error.txt` content visible after the Gateway has recovered.
  - confidence: 0.88
- **[P2] Reject invalid watchdog numeric options before spawning:** `src/cli/gateway-cli/run.ts:583-590`
  - body: Both numeric parsers can return `NaN` (`parseInt("foo", 10)` and `Math.max(1, NaN)`), and that value is forwarded into shell arithmetic. The script then aborts with a shell arithmetic error instead of a clear Commander validation failure for invalid `--interval` or `--max-attempts` input.
  - confidence: 0.9
- **[P2] Handle child process spawn errors with an error listener:** `src/cli/gateway-cli/run.ts:615-623`
  - body: Wrapping `spawn("bash", ...)` in try/catch does not handle normal spawn failures such as `ENOENT`; Node emits those on the returned `ChildProcess` as an `error` event. Without an `error` listener, a missing Bash runtime can become an uncaught exception instead of a controlled CLI error.
  - confidence: 0.86
