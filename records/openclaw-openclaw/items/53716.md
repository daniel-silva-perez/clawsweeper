## Summary

Keep open. Current main still lacks the watchdog command and the PR's startup-error file surface, so the work is not obsolete, but the branch has blocking package/runtime defects and overlaps broader rescue-watchdog work that needs maintainer direction before merge.

## What This Changes

The PR adds startup-error file persistence for fatal CLI/Gateway startup failures and registers a foreground `openclaw gateway watchdog` command backed by a Bash health-check, restart, log, and notification script.

## Best Possible Solution

Land only a rebased implementation with a clear watchdog boundary: package any runtime asset or move it into shipped code, validate CLI input before spawning shell code, clear diagnostics at actual successful startup, document the public command, and decide whether the broader watchdog behavior should instead follow the rescue-watchdog service thread.

## Reproduction Assessment

Yes. The missing current-main behavior is reproducible by inspecting the registered Gateway CLI surface and searching for watchdog/startup-error symbols; the PR defects are reproducible from the diff and package whitelist without running mutating commands.

## Solution Assessment

No. The PR is not the best merge shape as-is because the packaged CLI would miss the script and the diagnostics cleanup is placed after a long-running loop; a revised or superseding implementation should align with current stability/fatal-hook architecture and the broader rescue-watchdog plan.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Ship the watchdog script with the packaged CLI:** `src/cli/gateway-cli/run.ts:573`
  - body: The command resolves `scripts/openclaw-watchdog.sh` at runtime, but the package file list does not include that script. Installed package users would get a `gateway watchdog` command that points at a missing file unless the script is moved into a shipped runtime asset or explicitly packaged.
  - confidence: 0.94
- **[P2] Clear startup errors after startup actually succeeds:** `src/cli/gateway-cli/run.ts:463-466`
  - body: This cleanup runs only after `await startLoop()` returns, but the Gateway loop is designed to stay running until shutdown or restart handling. On normal successful startup this block is not reached, leaving stale `startup-error.txt` content visible after recovery.
  - confidence: 0.88
- **[P2] Reject invalid watchdog numeric options before spawning:** `src/cli/gateway-cli/run.ts:583-590`
  - body: Both numeric parsers can return `NaN`, and those values are forwarded into shell arithmetic. Inputs like `--interval foo` or `--max-attempts foo` should fail with a Commander validation error instead of starting a script that aborts with an opaque shell error.
  - confidence: 0.9
- **[P2] Handle child process spawn errors explicitly:** `src/cli/gateway-cli/run.ts:615-623`
  - body: Wrapping `spawn("bash", ...)` in `try/catch` does not handle normal spawn failures such as `ENOENT`; Node emits those on the returned `ChildProcess` as an `error` event. Without an `error` listener, missing Bash can become an uncaught exception instead of a controlled CLI error.
  - confidence: 0.86
