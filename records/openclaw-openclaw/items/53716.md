## Summary

Keep open. Current main does not implement the proposed Gateway watchdog or startup-error file, and the PR branch still has unresolved packaging, auth/env, numeric validation, docs/tests, and security-sensitive command-execution review concerns.

## What This Changes

The PR adds startup error persistence for fatal CLI/Gateway failures plus a Bash Gateway health watchdog launched by a new `openclaw gateway watchdog` CLI subcommand.

## Best Possible Solution

Land a rebased and reviewed implementation only after the watchdog boundary is clear: either split startup-error persistence from a packaged first-class watchdog command, or supersede it with the broader rescue-watchdog design. The final implementation should handle auth/env defaults, numeric validation, package inclusion or dist assets, correct startup-error cleanup, docs, and focused CLI/infra tests.

## Risks / Open Questions

- Merging the PR as-is could expose a `gateway watchdog` command that fails in installed packages because the runtime script is not included or moved into built assets.
- The watchdog option parsers still accept non-numeric interval/max-attempt values, which can propagate `NaN` into shell arithmetic and terminate the watchdog instead of producing a clear CLI error.
- The Bash script initializes `TOKEN` independently from `OPENCLAW_GATEWAY_TOKEN`, so env-token deployments can be treated as unhealthy and restarted unnecessarily.
- The branch adds shell execution, restart behavior, desktop notifications, and token-bearing HTTP probes; those are appropriate but security-sensitive runtime behavior that needs maintainer review and focused tests.
- The scope overlaps the still-open #46502 rescue-watchdog design, so maintainers should decide whether to split, narrow, or supersede this approach.
