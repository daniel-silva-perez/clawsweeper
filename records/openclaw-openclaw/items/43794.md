## Summary

Keep open. Current main still treats SecretRefs and plaintext scrubbing as opt-in mitigations, while the threat model continues to mark token theft from config storage as high residual risk and recommends token encryption at rest; the related auth-store sealing PR is open and not merged.

## What This Changes

Add encrypted-at-rest handling for sensitive OpenClaw credential/config fields, with keychain or master-password support and backward-compatible migration.

## Best Possible Solution

Define and ship a security-owned credential-storage policy end to end: either transparent encrypted-at-rest storage for OpenClaw-owned credential stores with migration, recovery, rotation, docs, and tests, or an explicit supported posture that mandates SecretRefs/external secret managers for at-rest secrecy and updates the threat model accordingly.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a runtime bug, but yes for current behavior: docs and source on current main show plaintext tokens/API keys remain supported, persisted as JSON, and detected by `openclaw secrets audit`.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the credential-storage contract; the request identifies the right risk, but the fix should be security-designed rather than a narrow automated patch because key management and compatibility are central.
