## Summary

Keep open. Current main has meaningful mitigations around SecretRefs, secure file modes, audit, and one-way plaintext scrubbing, but it still does not provide transparent encryption at rest for OpenClaw config or auth-profile credentials. Plaintext credential storage remains documented, accepted by the runtime contract, and used by default onboarding, so this is a security-owner design item rather than cleanup-closeable work.

## What This Changes

The issue asks OpenClaw to encrypt persisted config and auth credentials at rest, using a system keychain or master password while preserving backward compatibility.

## Best Possible Solution

Define and ship a security-owned credential-storage policy end to end: either implement real encrypted-at-rest storage for persisted OpenClaw credentials with migration, audit, recovery, docs, and token rotation, or explicitly narrow the supported security posture to SecretRef-backed external secret managers and update the threat model accordingly.
