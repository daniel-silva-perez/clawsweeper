## Summary

Keep open. Current main still supports plaintext credential persistence and treats SecretRefs, audit, scrub, and file permissions as mitigations rather than transparent encryption at rest; the threat model still marks token theft from config storage as high residual risk, and the related auth-store sealing PR is open/unmerged.

## What This Changes

Request to encrypt sensitive OpenClaw config and auth credentials at rest using a system keychain or master password while preserving backward compatibility.

## Best Possible Solution

Define and ship a security-owned credential-storage policy end to end: either transparent encrypted-at-rest storage for OpenClaw-owned credential files with backward-compatible migration, recovery, rotation, docs, and tests, or an explicit supported posture that mandates SecretRefs/external secret managers for at-rest secrecy and updates the threat model accordingly.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a runtime bug, but yes for current behavior: current docs and source show plaintext keys/tokens remain supported, persisted as JSON, and reported by `openclaw secrets audit`.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the credential-storage contract; the request identifies a real risk, but the fix should be security-designed rather than attempted as a narrow automated patch.
