## Summary

Keep open. Current main has meaningful mitigations around SecretRefs, secure file modes, audit, and one-way plaintext scrubbing, but it still does not implement transparent encryption at rest for OpenClaw config or auth-profile credentials. Plaintext storage remains documented, accepted by the runtime contract, and used by default onboarding, so this remains a security-owner design item rather than a cleanup close.

## What This Changes

Add encrypted-at-rest storage for sensitive config and auth-profile credentials, using a system keychain or master password while preserving backward compatibility.

## Best Possible Solution

Define a security-owned credential-storage policy and ship it consistently: either add real encrypted-at-rest storage for persisted OpenClaw credentials, with migration, audit, recovery, docs, and token rotation, or explicitly narrow the product commitment to SecretRef-backed external secret managers and update the threat model accordingly.
