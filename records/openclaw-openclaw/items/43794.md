## Summary

Keep open. Current main has important mitigations around SecretRefs, secure JSON file modes, audit, and one-way plaintext scrubbing, but it still does not implement transparent encryption at rest for OpenClaw config or auth-profile credentials. Plaintext credential storage remains a supported and documented default, so this needs security-owner design rather than cleanup closure.

## What This Changes

Add encrypted-at-rest storage for sensitive OpenClaw config and auth credentials, using a system keychain or master password while preserving backward compatibility.

## Best Possible Solution

Define and ship a security-owned credential-storage policy end to end: either implement encrypted-at-rest persisted credentials with migration, audit, recovery, docs, and token rotation, or explicitly narrow the supported posture to SecretRef-backed external secret managers and update the threat model accordingly.

## Reproduction Assessment

Not applicable as a runtime bug reproduction. The current posture is high-confidence from static docs and source: plaintext remains documented, accepted by the auth-profile types, and written through JSON persistence paths without encryption.

## Solution Assessment

Unclear until maintainers decide the credential-storage contract. SecretRefs are a useful mitigation, but they are opt-in and do not satisfy transparent encryption at rest; a keychain/master-password design needs security-owner approval before implementation.
