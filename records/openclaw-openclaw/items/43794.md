## Summary

Keep open. Current main has substantial SecretRef, audit, and file-permission mitigation, but it still does not provide transparent config/auth-profile encryption-at-rest with a system keychain or master password. The threat model continues to list token theft from config storage as a high residual risk, and current onboarding/docs still allow or default to plaintext credential storage.

## What This Changes

The issue requests encryption-at-rest for sensitive OpenClaw configuration credentials, using a system keychain or master password while preserving backward compatibility.

## Best Possible Solution

Define and ship a security-owned credential storage design that either adds real encrypted-at-rest storage for persisted OpenClaw credentials or explicitly narrows the product commitment to SecretRef-backed external secret managers, with migration, audit, docs, and rotation semantics aligned.
