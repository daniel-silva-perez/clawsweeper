## Summary

Keep this issue open. Current main still exposes `maxMissedJobsPerRestart` and `missedJobStaggerMs` only as internal `CronServiceDeps` fields; they are not public `cron` config, not accepted by the strict config schema, not present in generated config metadata/docs, and not wired from `openclaw.json` into gateway cron startup catch-up. The related implementation PR #42069 is closed and unmerged, and the related skipMissedRuns issue #48345 is also closed, so there is no active canonical replacement in the provided context.

## Best Possible Solution

Implement a narrow public cron catch-up configuration surface. Add either `maxMissedJobsPerRestart` plus `missedJobStaggerMs`, or a clearer global catch-up policy such as `skipMissedRuns`, to `CronConfig`, zod validation, generated schema/help/labels, gateway passthrough into `CronServiceDeps`, and the public cron/config docs. Preserve existing defaults for compatibility, define disabled semantics explicitly, and add focused tests for config acceptance, gateway wiring, startup catch-up limits, stagger timing, and skipped/deferred catch-up behavior. Consider per-job control only after maintainers decide whether announcement jobs and monitoring jobs need different policy.

## Risks / Open Questions

- The exact public semantics still need maintainer judgment: expose the two existing numeric knobs as requested, add a clearer global catch-up policy such as `skipMissedRuns`, or support per-job catch-up behavior later.
- Changing default catch-up behavior could regress users who rely on missed-job recovery from the earlier wake-from-sleep feature, so compatibility defaults and `0`/disabled semantics need to be explicit.
- The current hardcoded stale-delivery guard can reduce very late direct announcements, but it runs after the agent work and does not prevent provider/API load during startup catch-up.
