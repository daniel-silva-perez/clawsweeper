## Summary

Keep this issue open. Current main still does not expose `maxMissedJobsPerRestart` or `missedJobStaggerMs` as public cron configuration: the public `CronConfig` type and strict zod schema omit both keys, generated config schema metadata still disallows unknown cron properties, and gateway cron startup still passes only the existing cron config fields into `CronService`. The related implementation PR #42069 is closed and unmerged, and the related `skipMissedRuns` issue #48345 is also closed, so there is no active canonical replacement to close this against.

## Best Possible Solution

Implement a narrow, documented cron catch-up configuration surface. Add the selected option set to `CronConfig`, zod validation, generated schema/help metadata, gateway `CronService` wiring, and public cron/config docs; preserve existing defaults, define `0` or disabled behavior clearly, and add focused tests for config acceptance, gateway passthrough, startup catch-up limits, stagger timing, and skipped/deferred catch-up behavior.

## Risks / Open Questions

- The right public shape still needs maintainer product judgment: expose the two internal numeric knobs exactly, add a clearer global `skipMissedRuns` policy, or later support per-job catch-up control.
- Changing catch-up behavior can affect users who rely on missed-job recovery introduced for wake-from-sleep scenarios, so compatibility defaults and disabled semantics need to be explicit.
- A hardcoded stale-delivery guard may reduce very late direct announcements, but it does not replace a startup catch-up policy because provider work can still be launched before delivery suppression.
