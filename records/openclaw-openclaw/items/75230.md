## Summary

Keep open. Current `main` does not implement the PR's intended payment-credential redaction: default patterns still cover generic secrets only, and structured tool payloads are redacted by value patterns rather than by sensitive field names. The proposed patch is narrow, security-reducing, and I did not find a blocking correctness or supply-chain issue in the diff.

## What This Changes

The branch extends logging/tool-payload redaction for payment credential field names, adds key-aware structured tool-result masking, and updates focused tests, logging docs, and the changelog.

## Best Possible Solution

Land a narrow key-scoped redaction improvement with focused tests and docs, keeping broad card-number scanning out of generic logs and leaving the larger exec stdout/stderr secret-redaction issue tracked separately by #71211.

## Reproduction Assessment

Yes. A focused reproduction is to pass a tool result with `details.structuredContent.cardNumber` or `sharedPaymentToken` through `sanitizeToolResult`; current main only redacts by value pattern, while the PR adds key-aware masking and a regression test for that shape.

## Solution Assessment

Yes. The proposed direction is the narrow maintainable fix for this gap: extend default key patterns for textual surfaces and add key-aware masking only for structured sensitive fields, rather than scanning every arbitrary card-like number.
