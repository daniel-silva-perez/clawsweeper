## Summary

Keep open. Current main still only partially satisfies the request: owning context engines get an internal loop-time assemble path, but non-owning engines stay on the generic guard and there is still no public per-LLM-call full-message rewrite hook. The linked implementation PR is closed and unmerged, so this remains an API/product decision rather than maintainer cleanup.

## What This Changes

The issue requests a per-LLM-call context assembly or public context-transform hook so context engines/plugins can rewrite the full message array during multi-call agent loops.

## Best Possible Solution

Define a stable, documented per-model-call context rewrite contract for the plugin/context-engine API, align the docs with the actual semantics, and add regression coverage for multi-tool-call loops across owning engines, non-owning engines, and any public hook surface.

## Reproduction Assessment

Yes. A code-level reproduction is clear from current main: configure a non-owning context engine and run a multi-call tool loop, and assemble is only reached on the pre-prompt path while no public transform_context hook exists; I did not run a live LLM session because this was a read-only review.

## Solution Assessment

Unclear. An additive hook like the related PR proposed is probably safer than changing assemble frequency for all engines, but the hook ordering and trust model need maintainer API review before implementation.
