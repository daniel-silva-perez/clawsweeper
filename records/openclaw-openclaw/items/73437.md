## Summary

Keep open. Current main still only partially covers loop-time context assembly: owning context engines get an internal loop wrapper, but there is no public per-LLM-call full-message rewrite hook and non-owning engines still use the generic guard plus the single pre-prompt assembly path. The related transform_context PR was closed unmerged, so this remains a maintainer API/product decision rather than cleanup that can close.

## What This Changes

Request a per-LLM-call context assembly or transform_context hook so context engines and plugins can rewrite the full message history during multi-call agentic tool loops.

## Best Possible Solution

Define a stable, documented per-model-call context rewrite contract for the plugin/context-engine API, preserve existing ContextEngine compatibility, and add regression coverage for multi-tool-call loops across owning engines, non-owning engines, and the public hook surface.

## Reproduction Assessment

Yes for the code-level behavior: current main shows attempt-level assembly before the prompt run, an internal loop wrapper only for ownsCompaction engines, and no public transform_context hook. A live multi-call LLM run would be useful end-to-end proof, but the source and focused tests are enough to reproduce the API gap.

## Solution Assessment

Unclear. The additive hook direction from the related PR looks safer than changing assemble frequency because it preserves existing ContextEngine semantics, but hook ordering, trust boundaries, and budget-enforcement guarantees need maintainer API review before implementation.
