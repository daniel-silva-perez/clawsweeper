## Summary

Keep open. Current main still only partially addresses loop-time context assembly: owning context engines get an internal loop wrapper, but there is no public per-LLM-call full-message rewrite hook and non-owning engines still use the generic guard plus the single pre-prompt assembly path. The related implementation PR was closed unmerged, so the remaining work is an API/product decision rather than cleanup that can be closed.

## What This Changes

The issue requests a per-LLM-call context transformation or ContextEngine assembly path so plugins or context engines can rewrite the full message history during multi-call agent loops.

## Best Possible Solution

Define a stable, documented per-model-call context rewrite contract for the plugin/context-engine API, preserve existing ContextEngine compatibility, and add regression coverage for multi-tool-call loops across owning engines, non-owning engines, and the public hook surface.

## Reproduction Assessment

Yes for the code-level behavior: current main shows attempt-level assembly before the prompt run, an internal loop wrapper only for ownsCompaction engines, and no public transform_context hook. I did not run a live multi-tool LLM session because this was a read-only review, but the source and focused tests are enough to reproduce the API gap.

## Solution Assessment

Unclear. The additive hook direction from the related PR looks safer than changing assemble frequency because it preserves existing ContextEngine semantics, but the hook ordering, trust model, and budget-enforcement guarantees need maintainer API review before implementation.
