## Summary

Keep open. Current main still only partially addresses loop-time context assembly: owning context engines get an internal loop wrapper, but non-owning engines still use the generic guard plus the single pre-prompt assembly path, and there is no public per-LLM-call full-message rewrite hook. The related implementation PR is closed and unmerged, so this remains an API/product decision rather than cleanup that can be closed.

## What This Changes

The issue requests per-LLM-call context assembly or a public context-transform hook so context engines/plugins can rewrite the full message array during multi-call agent loops.

## Best Possible Solution

Define and document a stable per-model-call context rewrite contract for the plugin/context-engine API, align the context-engine docs with actual owning and non-owning semantics, and add regression coverage for multi-tool-call loops across owning engines, non-owning engines, and any public hook surface.

## Reproduction Assessment

Yes. A code-level reproduction is clear from current main: configure a non-owning context engine and run a multi-call tool loop, and assemble is only reached on the pre-prompt path while no public transform_context hook exists. I did not run a live LLM session because this was a read-only review, but the source and focused tests are enough to reproduce the API gap.

## Solution Assessment

Unclear. The additive hook direction from the related PR looks safer than changing assemble frequency for every context engine because it preserves existing ContextEngine compatibility, but hook ordering and prompt-content trust policy need maintainer API review first.
