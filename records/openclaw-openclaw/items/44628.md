## Summary

Keep open. Current main still does not implement this PR's intended managed Linux systemd scope selection, and the PR is not merge-ready because it updates the lower-level restart helper instead of the managed service lifecycle path, prefers enabled-only units before active-only units, and carries an unrelated Docker mount-validation change. The related #44772 discussion is useful prior art for the broader lifecycle surface, but that PR is closed unmerged.

## What This Changes

The PR removes Docker setup whitespace validation and adds user/system `systemctl is-enabled` and `is-active` probing in `triggerOpenClawRestart()` to choose a restart scope and include it in restart details.

## Best Possible Solution

Land a focused systemd lifecycle fix in the daemon service adapter: restart/stop/status should detect or fall back to system scope only under well-classified missing-user-unit conditions, report the chosen scope, preserve user-scope permission/bus failures, and add regression tests for system-scope restart and runtime/status. The Docker whitespace behavior should remain out of this PR.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to mock the managed systemd path so `systemctl --user restart/show` reports a missing unit while system scope succeeds; current main still calls only the user-scope managed path for restart and runtime/status.

## Solution Assessment

No. This PR is not the best fix as written because it updates `triggerOpenClawRestart()` rather than the managed service lifecycle path, chooses enabled-only scopes before active-only scopes, and mixes in an unrelated Docker validation change; a narrower `src/daemon/systemd.ts` fix with tests is safer.

  - body: This unrelated change removes the early whitespace rejection even though downstream mount validation still rejects spaces and Compose volume substitutions are not consistently quoted. That turns a clear validation error into broken or ambiguous Docker volume parsing.
