## Summary

Keep this PR open, but it needs changes before merge. Current main still has the stale token display paths, and the PR addresses the right surface, but its TUI guard only updates non-null totals, so stale-marker-only rows and fresh null resets can leave an old footer token count visible.

## What This Changes

The PR threads `totalTokensFresh` through TUI and Control UI session row handling and changes session token display paths to avoid showing totals marked stale.

## Best Possible Solution

Land a narrow display fix on current main: TUI should carry `totalTokensFresh`, clear the cached footer total whenever a row or patch marks totals stale, still allow fresh `totalTokens: null` to clear state, and Control UI session formatting should return `n/a` for stale rows with regression tests for both surfaces.

## Reproduction Assessment

Yes: seed TUI state with a numeric `sessionInfo.totalTokens`, then apply a session row or patch with `totalTokensFresh: false` and no fresh total; the old footer value remains. The Control UI formatter path is directly checkable with a `GatewaySessionRow` that has `totalTokensFresh: false` and a numeric `totalTokens`.

## Solution Assessment

No: the PR direction is right, but the TUI implementation is not the best fix because freshness is checked only after requiring a non-null total. The safer approach handles `totalTokensFresh === false` as a clear signal first, then applies fresh numeric or null totals normally.
