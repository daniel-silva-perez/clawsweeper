## Summary

Keep this PR open. Current main still does not implement Android inline Adaptive Card rendering, and the PR remains a real Android client implementation candidate tied to the existing openclaw-adaptive-cards plugin. It needs maintainer product/security review rather than cleanup closure, especially because same-author paired client work in #42350 is still open.

## Best Possible Solution

Keep the PR open for maintainer product and security review. If native Android rich chat is desired, rework it around the canonical MessagePresentation or a documented client-renderer extension path, preserve ordered fallback text and multiple cards, apply template data, add defensive render limits and URL scheme policy, wire or disable actions intentionally, and add focused Android tests. If maintainers decide Adaptive Cards should stay outside core, close this together with #42350 and document the missing client-renderer API needed by the openclaw-adaptive-cards plugin.

## Risks / Open Questions

- Malformed or model-generated card payloads could crash Android chat rendering unless depth, numeric bounds, card count, and template-data handling are defensive.
- Action.OpenUrl needs explicit Android URL policy, and submit/execute actions need real routing or visibly disabled affordances so chat does not show dead interactive controls.
- The marker-based transport should be reconciled with the documented MessagePresentation path or a clear client/plugin renderer extension point before merging.
