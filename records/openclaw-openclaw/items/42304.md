## Summary

Keep this PR open. Current main still does not implement Android inline Adaptive Card rendering: Android chat text goes directly through markdown rendering and non-text parts are limited to base64 images. The PR remains a real Android client implementation candidate tied to the already-posted openclaw-adaptive-cards plugin, and the same-author paired iOS/macOS PR #42350 is still open. It needs maintainer product and security review rather than conservative cleanup closure.

## Best Possible Solution

Keep the PR open for maintainer product and security review. If native Android rich chat is desired, rework the implementation around MessagePresentation or a documented client-renderer extension path, preserve ordered fallback text and multiple cards, apply template data, add defensive render limits and URL scheme policy, wire or deliberately disable actions, and add focused Android parser/render tests. If maintainers decide Adaptive Cards should stay outside core, close this together with #42350 and document the missing client-renderer API needed by the openclaw-adaptive-cards plugin.

## Risks / Open Questions

- Malformed or model-generated card payloads can still crash Android rendering unless depth, card-count, numeric, and size limits are enforced defensively.
- Action.OpenUrl needs an explicit Android URL policy, and submit/execute actions need real routing or intentionally disabled/non-interactive UI so chat does not show dead controls.
- The marker-based transport should be reconciled with MessagePresentation or a documented client-renderer extension path before this becomes a core Android feature.
