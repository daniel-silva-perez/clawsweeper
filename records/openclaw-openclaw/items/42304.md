## Summary

Keep this PR open. Current main still does not implement Android inline Adaptive Card rendering, the Android chat path remains text/base64-media only, and the PR is paired with the same author’s still-open iOS/macOS companion #42350. The proposed diff also still deserves product/security review because it would render model/plugin-supplied card JSON inside the Android client.

## Best Possible Solution

Keep the PR open for maintainer product and security review. If native Android rich cards are desired, rework it around the current MessagePresentation direction or a documented Android client-renderer extension path, preserve ordered mixed text/card content, support multiple cards and template data, add defensive render limits and URL/action policy, and add focused Android parser/render tests. If maintainers decide this belongs outside core, close it together with the companion client PRs only after documenting the missing client-renderer API needed by the external openclaw-adaptive-cards plugin.
