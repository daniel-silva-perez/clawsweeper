## Summary

Keep this PR open. Current main still does not implement Android inline Adaptive Card rendering: Android chat renders text through ChatMarkdown and non-text displayable parts through base64 image handling, with no Adaptive Card parser/composable or client-renderer API. The PR is also paired with the same author’s still-open iOS/macOS companion #42350 and has unresolved product/security review questions around rendering model/plugin-supplied card JSON in the Android client.

## Best Possible Solution

Keep the PR open for maintainer product and security review. If native Android rich cards are desired, rework the implementation around the current MessagePresentation direction or a documented Android client-renderer extension path, preserve ordered mixed text/card content, support multiple cards and template data, add defensive render limits and URL policy, wire or explicitly disable actions, and add focused Android parser/render tests. If maintainers decide this belongs outside core, close it together with the same-author companion after documenting the missing client-renderer API needed by the external openclaw-adaptive-cards plugin.
