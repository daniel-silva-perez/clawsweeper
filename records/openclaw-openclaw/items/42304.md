## Summary

Keep this PR open. Current main still does not implement Android inline Adaptive Card rendering: Android chat renders text through Markdown and non-text parts as base64 images, with no Adaptive Card parser/composable or structured card model. The PR is also paired with the same-author open iOS/macOS renderer PR #42350, so it needs maintainer product and security review rather than cleanup closure.

## Best Possible Solution

Keep the PR open for maintainer product and security review. If native Android rich chat is desired, rework it around MessagePresentation or a documented client-renderer extension path, preserve ordered fallback text, multiple cards, and template data, add defensive render limits and URL policy, wire or explicitly disable actions, and add focused Android parser/render tests. If maintainers decide Adaptive Cards should stay outside core, close this together with #42350 after documenting the missing client-renderer API needed by the openclaw-adaptive-cards plugin.

## Risks / Open Questions

- The PR diff renders model/plugin-supplied JSON and clickable controls in the Android client, so it needs defensive depth/size/count/numeric limits, URL scheme policy, and clear action routing or deliberately disabled controls before landing.
- The marker-based transport should preserve ordered mixed text/card content, multiple card blocks, and template data; otherwise native rendering can reorder or drop user-visible content.
- There is a product-boundary question: OpenClaw’s current direction favors optional plugins, but Android native client rendering may require a documented client-renderer extension path before the external adaptive-card plugin can provide this without core app changes.
