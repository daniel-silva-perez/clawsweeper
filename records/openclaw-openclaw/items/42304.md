## Summary

Keep this PR open. Current main still renders Android chat text through Markdown and non-text content as base64 attachments only; there is no Android Adaptive Card parser, renderer, presentation field, or test coverage. The PR is a real implementation candidate tied to the existing openclaw-adaptive-cards plugin, but it is not implemented on main, not superseded by a merged Android path, and its paired iOS/macOS PR #42350 remains open.

## Best Possible Solution

Keep the PR open for maintainer product and security review. If native Android rich chat is desired, rework it around the canonical MessagePresentation/client-rendering path with ordered fallback text, multiple-card handling, template data application, defensive render limits, URL scheme allowlisting, real action routing or disabled affordances, and focused Android tests. If maintainers decide Adaptive Cards should stay outside core, close this Android PR together with the paired iOS/macOS PR and document the missing client-renderer extension API needed by the existing openclaw-adaptive-cards plugin.

## Risks / Open Questions

- Malformed or model-generated card payloads can become an Android chat crash surface unless recursion depth, numeric bounds, card count, template data, and fallback ordering are handled defensively.
- Action.OpenUrl needs explicit URL scheme policy or equivalent Android app handling; submit/execute actions need real routing or visibly disabled controls so users do not get dead interactive UI.
- The marker-based transport bypasses the documented MessagePresentation contract, so maintainers should decide whether Android native rendering belongs in core client UI or needs a client/plugin renderer extension point first.
- Closing this PR alone would split a same-author cross-platform feature pair while #42350 remains open.
