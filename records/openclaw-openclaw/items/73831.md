## Summary

Keep open. This is a fresh, detailed Windows/Node 24 report against 2026.4.26 and 2026.4.23. Current release/main already contain the obvious HTTP/1-only mitigations for Telegram and guarded model fetch dispatchers, so the remaining reported LLM timeout is not proven obsolete and should be investigated as a still-active Windows/provider-transport failure path.

## Best Possible Solution

Keep this open as an active Windows network regression. Next maintainer pass should capture the reporter's resolved provider/model API for `openai/claude-opus-4-7`, verify whether that run uses `createBoundaryAwareStreamFnForModel` and `buildGuardedModelFetch`, and reproduce on Windows Node 24 with Telegram polling plus a real LLM call. If reproduced, fix the remaining bypass or add an explicit provider-transport family/dispatcher override with regression coverage for the concrete route.
