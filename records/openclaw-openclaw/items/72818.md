## Summary

Keep this PR open, but do not merge it as-is. Current main still has the configured-bundled-channel startup gap, yet the proposed fallback starts raw `cfg.channels` entries through direct bundled lookup without reapplying plugin enablement/deny policy, which can bypass explicit plugin disablement.

## What This Changes

The PR changes channel startup to merge loaded registry plugins with configured channel ids resolved by `getChannelPlugin()`, adds `context.assembled` trigger data to diagnostic stability output, adds QA-channel thread roundtrip coverage, and includes small test/format cleanup.

## Best Possible Solution

Revise the PR so startup adds only policy-approved configured channel owners, preferably by reusing the manifest/startup activation plan or an equivalent policy-checked helper, then direct-lookup those allowed channel ids and keep the bundled startup runtime fast path intact.

## Reproduction Assessment

Yes. A high-confidence static reproduction is a config with a bundled channel under `channels` plus `plugins.enabled=false` or a deny/disabled entry: the PR’s raw `Object.keys(cfg.channels)` loop can resolve the bundled plugin through `getChannelPlugin()` even though current startup policy would exclude it.

## Solution Assessment

No. The direction is right, but the raw config-key enumeration is not the best fix because it bypasses existing plugin policy; the safer implementation should derive candidates from policy-approved configured channel/plugin ownership before using bundled fallback lookup.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Gate fallback channel startup through plugin policy:** `src/gateway/server-channels.ts:692-696`
  - body: This loop starts every raw configured channel id via `getChannelPlugin()`, whose bundled fallback is not policy-aware. When the active registry is empty because `plugins.enabled=false`, a plugin is denied, or its entry is disabled, this still resolves a bundled channel and can start plugin code despite explicit disablement. Derive the fallback candidates from the existing policy-checked startup/configured-channel owner set before direct lookup.
  - confidence: 0.88
