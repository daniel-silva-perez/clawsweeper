## Summary

Keep open: the PR targets a real channel startup gap, but the current diff still adds a policy bypass in the fallback path, so it should not merge as-is.

## What This Changes

The PR merges loaded channel plugins with configured channel ids resolved through bundled fallback lookup, adds a sanitized context trigger field to diagnostic stability output, and adds QA-channel thread roundtrip coverage plus small test formatting cleanup.

## Best Possible Solution

Revise the PR so fallback startup candidates come from the same policy-approved startup/deferred channel owner plan that builds the registry, then use direct bundled lookup only after that approval. Add focused coverage for `plugins.enabled=false`, denylisted owners, disabled plugin entries, and the intended configured-channel startup case.

## Reproduction Assessment

Yes. A high-confidence static reproduction is a config with a bundled channel under `channels` plus `plugins.enabled=false`, `plugins.deny`, or `plugins.entries.<id>.enabled=false`; the PR loop still resolves the channel via `getChannelPlugin()`.

## Solution Assessment

No. The intended registry fallback is useful, but raw `cfg.channels` enumeration is not the best fix because it bypasses existing plugin activation policy; derive the fallback candidates from the policy-checked startup plan instead.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Gate fallback channel startup through plugin policy:** `src/gateway/server-channels.ts:692-696`
