## Summary

Keep open: the PR targets a real startup gap, but the current diff is not mergeable because its fallback path can start bundled channel plugins that the plugin activation policy intentionally kept out of the loaded registry.

## What This Changes

The PR adds configured-channel fallback enumeration in `startChannels()`, records `context.assembled` trigger data in diagnostic stability output, adds QA-channel thread roundtrip coverage, and includes minor test/format cleanup.

## Best Possible Solution

Revise the PR so fallback startup candidates come from the existing manifest/startup activation plan or an equivalent policy-checked owner set, then resolve only those approved configured channel owners through the bundled lookup. Add regression coverage for `plugins.enabled=false`, denylist, disabled plugin entry, and `channels.<id>.enabled=false`, while keeping the QA-channel thread coverage if it still proves the intended path.

## Reproduction Assessment

Yes. A static reproduction is enough: current `main` only queues `listChannelPlugins()`, while the PR's new raw `cfg.channels` loop can resolve bundled channels through `getChannelPlugin()` even when the active registry is empty because plugin policy disabled or denied them.

## Solution Assessment

No. The direction is useful, but raw config-key enumeration is not the best fix because it bypasses the activation policy; the safer solution is to reuse the policy-approved configured channel/plugin owner set before direct bundled lookup.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Gate fallback channel startup through plugin policy:** `src/gateway/server-channels.ts:690`
  - body: This loop starts every raw configured channel id via `getChannelPlugin()`, whose bundled fallback is not policy-aware. When the active registry is empty because `plugins.enabled=false`, a plugin is denylisted, or its plugin entry is disabled, this can still resolve and start bundled channel code despite explicit disablement. Derive fallback candidates from the policy-checked startup plan before direct lookup.
  - confidence: 0.88
