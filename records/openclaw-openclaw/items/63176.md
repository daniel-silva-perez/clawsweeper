## Summary

Keep open. Current main still does not expose `api.runtime.acp.spawn()` / `api.runtime.acp.prompt()` or the proposed `plugins.allowAcpSpawn` config key. Existing ACP support covers `/acp`, `sessions_spawn({ runtime: "acp" })`, and ACP backend registration SDK subpaths, but not the plugin runtime helper this PR is adding. This remains a plausible implementation candidate, with security/API review needed before merge.

## Best Possible Solution

Keep this PR open as the implementation candidate. Best path is to rebase on current main, remove the already-on-main Discord cherry-pick if redundant, complete maintainer security/API review for the privileged ACP plugin capability, add/update plugin runtime and config docs, then validate the targeted runtime/config tests plus the appropriate changed gate. Close only after an equivalent merged change provides `api.runtime.acp.spawn()`, `api.runtime.acp.prompt()`, and the opt-in config contract on main.

## Risks / Open Questions

- Security/API surface: exposing ACP spawn/prompt lets trusted plugins start host-side ACP coding agents and gateway agent calls. The opt-in gate, per-call config check, sandbox policy interaction, and operator documentation need maintainer review before merge.
- Branch hygiene: the PR includes a Discord thread-binding normalization cherry-pick that current main already appears to have, so the branch should be refreshed and the already-shipped unrelated surface dropped if it is no longer part of the ACP runtime change.
- Docs/contract gap: because this is a public plugin runtime capability plus a new config key, merge should include plugin runtime docs, config reference/help alignment, mocks, and contract tests, not just source wiring.
