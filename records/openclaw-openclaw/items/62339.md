## Summary

Keep open. The current source supports the maintainer comment: there is not clear evidence that normal runtime silently mutates configured WhatsApp `allowFrom`, but current `main` still has an actionable WhatsApp automation-recipient bug where pairing-store approvals can become heartbeat/cron delivery recipients.

## Best Possible Solution

Keep this issue open and fix it in the WhatsApp-owned automation delivery path: in strict allowlist mode, heartbeat/cron delivery should use explicit `--to` only when authorized, the active session recipient only when it is in configured `allowFrom`, or the configured `allowFrom` list itself. Pairing-store approvals should remain DM access state unless the user explicitly opts into using them for automation delivery. Update the heartbeat/cron tests to assert that store-only WhatsApp recipients are not selected for automated sends, and improve status/log labeling so pairing-store recipients cannot be mistaken for configured `allowFrom` mutations.
