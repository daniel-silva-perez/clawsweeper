## Summary

Keep this issue open. Current main does not show a normal WhatsApp setup or doctor path that silently appends arbitrary numbers into configured allowFrom, but it still has the risky outbound automation behavior identified in the maintainer comment: heartbeat/cron recipient resolution merges configured allowFrom with the persisted pairing allow-store and can deliver to a pairing-store-only number.

## What This Changes

The issue reports that an unknown WhatsApp phone number appeared in allowFrom and received an automated Eye Rest Reminder despite not being explicitly authorized by the user.

## Best Possible Solution

Fix this in the WhatsApp plugin by separating explicit automation delivery authorization from pairing-store DM approvals: in allowlist-style automation, heartbeat/cron recipients should come from explicit --to, an active session recipient that is also configured in allowFrom, or configured allowFrom itself. Pairing-store-only entries should not become automatic outbound recipients unless a user opts into that behavior, and status/log/docs wording should distinguish configured allowFrom from pairing-store approvals.

## Reproduction Assessment

Yes for the automated delivery path: the focused unit-test shape in heartbeat-recipients.test.ts reproduces a pairing-store-only number being included in heartbeat recipients. No high-confidence reproduction exists for literal config mutation because the issue provides no config-before/config-after artifact or logs showing a write to openclaw.json.

## Solution Assessment

Yes, the best fix direction is the maintainer-described WhatsApp-plugin-owned change, not a broad core migration or startup config repair. The current code already keeps inbound allowlist mode from reading pairing approvals, so the narrowest maintainable fix is to align outbound automation recipient selection with that authorization boundary.
