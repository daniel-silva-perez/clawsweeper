## Summary

Keep open. Current main still allows persisted cron jobs with neither `id` nor legacy `jobId` to load without a stable identity, and the timer outcome path can still match `undefined` against the first id-less job. The issue is also paired with open PR #72853, so it should remain open until that implementation candidate is reviewed, merged, or closed.

## What This Changes

Persisted cron jobs that omit both `id` and legacy `jobId` should get stable identities so runtime state and outcomes cannot collide on `undefined`.

## Best Possible Solution

Land one canonical cron fix, preferably through PR #72853 or an equivalent maintainer-approved patch, that assigns stable ids for missing or blank persisted job identities during store load/hydration, persists the canonical shape, rejects missing job ids in timer/state-application paths, preserves doctor repair behavior, and adds regression coverage for multiple id-less persisted jobs plus sidecar state merge.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to load two persisted cron jobs with neither `id` nor `jobId`, let a due-job outcome carry `jobId: undefined`, and observe `applyOutcomeToStoredJob()` matching the first job whose `id` is also undefined.

## Solution Assessment

Yes. Load-time identity canonicalization plus defensive timer/state guards is the narrow maintainable solution; doctor-only repair is useful but insufficient because the Gateway can still load and execute malformed rows before doctor is run.
