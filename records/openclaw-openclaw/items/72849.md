## Summary

Keep open. Current main still leaves missing cron job ids unbackfilled and still has undefined-id state/outcome lookup paths, and the issue is already paired with open PR #72853 using closing syntax, so maintainers should review/land or close that PR rather than close this issue now.

## What This Changes

The issue requests that cron jobs missing an `id` get stable identities and that runtime state/outcome handling refuse undefined job ids instead of collapsing state into the first id-less job.

## Best Possible Solution

Land one canonical cron fix that assigns stable ids for missing or blank persisted job identities during load/hydration, persists the canonical shape, refuses undefined job ids in timer/state application paths, and covers the regression with focused store/timer tests plus the small docs note already proposed in the linked PR.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to load multiple persisted jobs with no `id`, let one run produce an outcome with an undefined job id, and observe that `applyOutcomeToStoredJob()` can match the first undefined-id entry; current code inspection shows that path remains live.

## Solution Assessment

Yes. The narrow maintainable fix is load-time identity canonicalization plus defensive timer/state guards; the open PR appears aimed at that boundary, so review/landing that work is preferable to creating a separate duplicate repair lane.
