## Summary

Keep open. Current main still loads persisted cron jobs with neither `id` nor legacy `jobId` without assigning a stable identity, and the outcome path can still match `undefined` against the first id-less job; #72853 is open and explicitly paired as the implementation candidate.

## What This Changes

The issue reports that hand-authored or migrated cron jobs missing both `id` and legacy `jobId` can have runtime state and outcomes attributed to the first id-less job.

## Best Possible Solution

Land one canonical cron fix, preferably through #72853 or an equivalent maintainer-approved patch, that assigns and persists stable ids for missing or blank persisted job identities during store load/hydration, rejects missing job ids in timer/state-application paths, preserves existing doctor repair behavior, and adds regression coverage for multiple id-less persisted jobs plus sidecar state merge.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to load two persisted cron jobs with neither `id` nor `jobId`, let a due-job outcome carry `jobId: undefined`, and observe `applyOutcomeToStoredJob()` matching the first job whose `id` is also undefined.

## Solution Assessment

Yes. Load-time identity canonicalization plus defensive timer/state guards is the narrow maintainable solution; doctor-only repair is useful but insufficient because the Gateway can still load and execute malformed rows before doctor is run.
