## Summary

Keep open. Current main still has the reported cron missing-id collision surface: persisted jobs can be normalized without an `id`, runtime sidecar state is still merged by `job.id`, and timer outcome application still finds by `entry.id === result.jobId` without first rejecting a missing result id. The issue is also paired with open PR #72853 using closing syntax, so maintainers should review, land, or close that PR rather than closing this issue now.

## What This Changes

The issue asks OpenClaw to backfill stable ids for hand-authored or migrated cron jobs missing `id` and to guard cron runtime state/outcome paths against undefined-id find collisions.

## Best Possible Solution

Land one canonical cron fix that assigns stable ids for missing or blank persisted job identities during load/hydration, persists the canonical shape, refuses undefined job ids in timer/state application paths, and covers the regression with focused store/timer tests plus the proposed user-facing docs note.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to load multiple persisted cron jobs without `id`, let one job produce an outcome with an undefined job id, and observe that `applyOutcomeToStoredJob()` can match the first undefined-id entry; current code inspection shows that path remains live.

## Solution Assessment

Yes. The narrow maintainable solution is load-time identity canonicalization plus defensive timer/state guards; because open PR #72853 is already tied to this issue with closing syntax, maintainers should review that implementation rather than start a duplicate fix lane.
