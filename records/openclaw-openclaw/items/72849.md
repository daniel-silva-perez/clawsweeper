## Summary

Keep open. Current main still allows persisted cron jobs to load without a stable `id`, the timer outcome path can still match an undefined result id against the first undefined-id job, and the issue is already paired with open PR #72853 using closing syntax, so it should wait for that PR to be reviewed, landed, or closed.

## What This Changes

The issue requests load-time stable id backfill for hand-authored or migrated cron jobs missing `id`, plus defensive guards so runtime state cannot be applied through undefined job ids.

## Best Possible Solution

Land one canonical cron fix that assigns stable ids for missing or blank persisted job identities during store load/hydration, persists the canonical shape, rejects missing job ids in timer and state-application paths, preserves existing doctor repair behavior, and adds focused regression coverage for multiple id-less persisted jobs plus sidecar state merge.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to load two persisted cron jobs with neither `id` nor `jobId`, let the due-job path carry `id: undefined` into a run outcome, and observe `applyOutcomeToStoredJob()` matching the first undefined-id job.

## Solution Assessment

Yes. Load-time identity canonicalization plus defensive timer/state guards is the narrow maintainable fix; the existing doctor repair is useful but insufficient because runtime can still load and execute malformed rows before doctor is run.
