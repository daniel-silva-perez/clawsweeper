## Summary

Keep open: this PR addresses a real current-main ACPX/Codex auth gap linked to #73910, but the proposed copy fallback can persist Codex credentials under OpenClaw state where default backups include them. The branch also needs a rebase before normal merge handling.

## What This Changes

The PR bridges host Codex `auth.json` into the ACPX-managed isolated Codex home, updates ACP docs and the changelog, adds ACPX regression tests, and adjusts one Codex plugin test type.

## Best Possible Solution

Keep the host-auth reuse direction, but make the bridge credential-safe before merge: prefer symlink-only behavior or a location/path that backup creation explicitly excludes, with regression coverage proving copied Codex auth cannot enter default archives. After that, rebase and rerun the focused ACPX tests plus the changed gate.

## Reproduction Assessment

Yes. Current main can be inspected against #73910: the Codex ACP wrapper forces an isolated `CODEX_HOME`, current preparation writes only generated config, and the linked issue shows direct ACPX with normal Codex auth works while isolated auth fails.

## Solution Assessment

No. Bridging only `auth.json` while keeping Codex config hooks isolated is the right narrow fix, but this PR’s copy fallback is not the safest implementation because it can persist credential material in a recursively backed-up state directory.
