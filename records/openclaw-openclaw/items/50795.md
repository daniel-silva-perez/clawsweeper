## Summary

Keep open: current main still zeroes every assistant usage snapshot on non-retry compaction end, while the replay sanitizer already has boundary-aware logic and tests that preserve fresh post-compaction usage. The linked fix PR is closed unmerged, and the merged provider-usage PR explicitly left this helper out of scope, so the runtime bug remains actionable.

## What This Changes

The issue requests preserving post-compaction assistant usage so the TUI/context status counter shows fresh token usage instead of 0 after compaction.

## Best Possible Solution

Patch the embedded subscribe compaction handler so it finds the latest compaction summary and zeroes only assistant usage snapshots before that boundary, preserving fresh post-compaction usage. Add focused subscribe-handler regression coverage for stale-before/fresh-after messages and avoid unrelated UI, provider-usage, replay-sanitizer, or session-store rewrites unless a tiny shared helper falls out naturally.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to pass a session containing a compactionSummary followed by an assistant message with usage through a non-retry compaction_end path; current main reaches the helper and zeroes every assistant usage regardless of boundary.

## Solution Assessment

Yes. The narrow maintainable fix is to apply the existing timestamp/index boundary pattern to the live subscribe handler; a UI-only or provider-only change would not stop the live session mutation that erases fresh usage.

Fix the context usage zeroing bug described by this issue. In `src/agents/pi-embedded-subscribe.handlers.compaction.ts`, change `clearStaleAssistantUsageOnSessionMessages()` so it identifies the latest `compactionSummary` in `ctx.params.session.messages` and only replaces assistant `usage` snapshots that are stale before that compaction boundary. Preserve fresh assistant usage after the latest compaction. Use the existing replay sanitizer pattern in `src/agents/pi-embedded-runner/replay-history.ts` as the behavioral model, including timestamp handling and an index-order fallback for legacy messages without timestamps. Add focused regression coverage for the subscribe handler/session subscription path that includes both stale pre-compaction usage and fresh post-compaction usage. Do not change provider usage normalization, TUI display formatting, session-store token persistence, or replay sanitizer behavior unless extracting a tiny shared helper is clearly cleaner. Changelog is not required unless the maintainer wants a user-facing bugfix entry.
