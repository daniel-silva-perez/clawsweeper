## Summary

Keep this PR open. Current main still lacks the two intended behaviors, and the PR patch remains useful but not merge-ready because its mismatch regex is too broad and its locale startup gate can race between persisted/navigator locale and settings locale.

## What This Changes

This PR adds a Control UI i18n readiness promise and render gate, plus a tool_use/tool_result mismatch classifier and tests for user-facing /new recovery text.

## Best Possible Solution

Update or replace the PR with two focused fixes: make Control UI startup wait for the initial locale and then apply the settings locale deterministically before first render, and add a narrowly scoped tool_result/tool_use mismatch classifier only for actual provider pairing rejections, with regression tests for both the positive and false-positive cases.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Tighten the tool_use mismatch regex:** `src/agents/pi-embedded-helpers/errors.ts:839`
  - body: The new `tool_use_id` branch matches any provider message containing that field, so generic schema errors such as required or invalid `tool_use_id` would be shown as session corruption with `/new` advice. Require tool_result plus missing/corresponding context so unrelated validation errors still surface accurately.
  - confidence: 0.88
- **[P2] Apply settings locale after initial i18n readiness:** `ui/src/ui/app.ts:119-133`
  - body: The patch starts `i18n.ready` and the settings `setLocale()` concurrently. If stored/navigator locale and settings locale differ, whichever async locale load finishes last wins, so the app can render in the wrong language; wait for `i18n.ready`, then apply the settings locale before setting `localeReady`.
  - confidence: 0.84
