## Summary

Keep this PR open. Current main still does not implement the Control UI i18n readiness/gated first-render behavior, and the PR branch has two concrete correctness defects already visible in the diff: an over-broad tool mismatch classifier and nondeterministic locale precedence.

## What This Changes

The PR adds awaitable Control UI i18n startup readiness, gates initial Lit rendering until locale loading settles, and adds user-facing recovery text plus tests for tool_use/tool_result mismatch errors.

## Best Possible Solution

Ship a focused update or replacement that makes startup locale readiness awaitable, applies persisted or navigator locale first, then applies `settings.locale` as the authoritative override before the first render. Keep tool pairing repair as the primary loop fix; if user-facing fallback text remains, match only real tool_result/tool_use pairing rejections and add false-positive coverage.

## Reproduction Assessment

Yes. For the PR-introduced defects, source inspection is enough: the regex matches any string containing `tool_use_id`, and `Promise.all([i18n.ready, settingsLocale])` lets two locale setters race when the locale sources disagree.

## Solution Assessment

No. The direction is useful, but the implementation should serialize locale precedence and narrow the tool mismatch classifier; it also needs to be rebased onto current main's sanitizer split rather than adding broader logic back into `errors.ts`.
