## Summary

Keep this PR open, but do not merge it as written. Current main still lacks the i18n render-readiness behavior this PR is trying to add, but the PR introduces a locale precedence race, an over-broad tool mismatch classifier, and needs rebasing onto the current sanitizer split.

## What This Changes

The PR adds awaitable Control UI i18n startup readiness, gates initial Lit rendering until locale loading, and adds tool_use/tool_result mismatch recovery text with tests.

## Best Possible Solution

Ship a focused update or replacement that exposes initial i18n readiness, waits for persisted or navigator locale loading, then applies `settings.locale` authoritatively before first render. Keep transcript repair as the primary fix for tool pairing loops; if a user-facing fallback remains, match only real provider pairing rejections and add false-positive coverage.

## Reproduction Assessment

Yes. The PR-introduced defects reproduce from the diff itself: the regex matches any string containing `tool_use_id`, and `Promise.all([i18n.ready, settingsLocale])` allows nondeterministic locale precedence when the two sources disagree.

## Solution Assessment

No. The i18n direction is useful, but the implementation should serialize locale precedence; the tool mismatch fallback should be narrower and rebased into the current sanitizer layout, especially because main already repairs the linked loop at replay time.
