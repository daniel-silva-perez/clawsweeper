## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks the Control UI first-render locale readiness fix, while the linked tool mismatch loop is already addressed by replay-history repair; the branch adds two definite regressions that need a narrow update or replacement.

## What This Changes

The PR adds awaitable Control UI i18n startup readiness, gates initial Lit rendering on locale load, and adds user-facing tool_use/tool_result mismatch recovery text with tests.

## Best Possible Solution

Ship a focused update or replacement that exposes initial i18n readiness, waits for persisted or navigator locale loading, then applies `settings.locale` authoritatively before first render. Keep the current transcript repair policy unchanged; if mismatch fallback text remains, match only actual provider pairing rejections and add false-positive coverage.

## Reproduction Assessment

Yes for the PR-introduced defects: the regex false positive follows directly from the standalone `tool_use_id` branch, and the locale race follows from concurrent `Promise.all([i18n.ready, settingsLocale])`. I did not run tests because this review was explicitly read-only.

## Solution Assessment

No, this PR is not the best way to solve the issue as written. The i18n direction is useful but must serialize locale precedence, and the tool mismatch fallback should be narrow and defensive because main already fixes the linked loop at replay repair time.
