## Summary

Keep this PR open, but it still needs changes before merge. Current main does not yet gate the first Control UI render on completed locale loading, and the submitted branch has two concrete correctness bugs already noted in review: an over-broad tool mismatch regex and nondeterministic locale precedence.

## What This Changes

The PR adds awaitable Control UI i18n startup readiness, delays initial app rendering until locale loading completes, and adds tool_use/tool_result mismatch recovery messaging plus focused tests.

## Best Possible Solution

Ship a focused rebased update or replacement that makes startup locale loading awaitable, applies persisted or navigator locale first, then applies settings.locale as the authoritative override before the first render. Keep the tool mismatch fallback narrow to real tool_result/tool_use pairing rejections, add false-positive coverage, and fit the sanitizer-side change into the current split sanitizer module.

## Reproduction Assessment

Yes for the PR-introduced defects: source inspection shows the regex matches any string containing tool_use_id, and the Promise.all startup pattern lets competing locale setters resolve in either order. The underlying UI flash is also supported by current main's fire-and-forget locale load before immediate render.

## Solution Assessment

No. The direction is maintainable, but this branch should not merge until locale precedence is serialized, the mismatch classifier is narrowed, and the code is rebased onto current main's separated sanitizer module.
