## Summary

Keep open. Current main and `v2026.4.27` already fix the original doctor false-positive by excluding `.trajectory.jsonl` from primary transcript classification, but this PR now proposes broader context-aware `.trajectory` semantics and doctor coverage. It should not merge as-is because the exported/no-context helpers would again count `.trajectory.jsonl` as primary and usage-counted, and the branch also mixes in node exec code-execution changes with failing check-runs.

## What This Changes

The PR makes `.trajectory.jsonl` classification depend on sibling primary transcript presence across session artifact helpers, doctor, usage, disk-budget, and memory-host scans/tests, and also changes node exec explicit-workdir preparation behavior.

## Best Possible Solution

Land or replace this with a narrowed session-artifact change that either keeps no-context helpers conservative or introduces a required directory-context API for `.trajectory` disambiguation, adds doctor/usage/disk-budget/memory-host coverage, and splits or drops the node exec changes unless they are reviewed as their own PR.

## Reproduction Assessment

Yes. The original bug is reproduced by the reported `doctor` orphan scan path on older builds and is now fixed on current main; the PR-specific regression is directly shown by its tests asserting no-context helpers count `abc.trajectory.jsonl` as primary/usage-counted.

## Solution Assessment

No. The context-aware idea addresses valid `.trajectory` session IDs, but making context optional on exported helpers is not the safest contract; a required-context helper or conservative no-context default is the narrower maintainable fix.
