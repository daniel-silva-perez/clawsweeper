## Summary

Keep this PR open: the underlying Mattermost routing bug is still present on current main and in the latest release, but the PR branch still has a concrete message-loss regression for blank `channel_type` events that the API fallback can recover, plus a public `ChatType` contract question that needs maintainer review.

## What This Changes

The PR changes Mattermost nullish channel-type mapping to `unknown`, adds fail-closed guards in Mattermost post/button/reaction/session routing, updates related Mattermost tests and auth types, and widens shared chat-type/routing-adjacent typing.

## Best Possible Solution

Keep the PR open, fix the blank-event fallback by carrying a trusted non-empty event-or-API channel type through `handlePost`, decide whether `unknown` belongs in public `ChatType` or should remain Mattermost-local, then land this PR or an equivalent fail-closed fix with regression coverage so #57565 can rebase cleanly.

## Reproduction Assessment

Yes. Current main can be verified with the pure mapper tests (`undefined` still maps to `channel`), and the PR regression is statically reproducible with `payload.data.channel_type = ""` plus a valid `resolveChannelInfo(...).type`.

## Solution Assessment

No, not as currently patched. The fail-closed direction is right, but the branch should use the API fallback for blank strings and settle the public `ChatType` contract before merge.
