## Summary

Keep this PR open. Current main still does not implement the PR's intended Docker-daemon-unavailable handling, and the linked bug remains open. The PR is a plausible fix candidate, but it needs maintainer review because its current approach changes Docker image pull behavior and lacks a focused regression test for mode-off startup paths.

## Best Possible Solution

Keep this PR open for maintainer review. The best fix is to verify the exact cron/heartbeat/sub-agent startup path that reaches Docker while the effective sandbox mode is off, then either prevent Docker probing before Docker is touched or add a narrow shared Docker-daemon-unavailable classifier with tests. Add regression coverage proving mode-off startup does not require Docker, while sandbox-enabled sessions still fail clearly. Separately decide whether automatic Docker pulls are acceptable or should remain an explicit user action.

## Risks / Open Questions

- The PR currently changes sandbox image resolution from explicit user setup to automatic `docker pull`/tag behavior, including configured custom image names; maintainers should explicitly approve or remove that network and supply-chain behavior.
- The proposed daemon-unavailable classifier includes broad substrings such as `no such file or directory` and `connection refused`, which could hide unrelated Docker failures on sandbox-enabled paths.
- The PR/context does not show a focused regression test for `sandbox.mode: "off"` cron, heartbeat, or sub-agent startup with Docker unavailable.
