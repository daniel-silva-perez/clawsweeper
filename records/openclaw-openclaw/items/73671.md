## Summary

Keep this PR open. Current main still has unguarded Docker image-inspect paths for sandbox container, browser sandbox, and doctor image checks, while the linked bug #73586 remains open. This PR is an active implementation candidate, but it still needs maintainer review to ensure sandbox.mode="off" avoids Docker entirely without masking failures when sandboxing is actually required.

## Best Possible Solution

Keep this PR open for maintainer review and iteration. The best fix is to ensure `sandbox.mode: "off"` cannot reach Docker at all, or to add a narrow shared Docker-daemon-unavailable classifier at the image-probe helpers without masking unrelated Docker failures. Add focused regression coverage for default/custom sandbox images, browser image setup, doctor behavior, and the cron/heartbeat/sub-agent mode-off path tied to #73586.

## Risks / Open Questions

- The proposed defensive helper must not turn sandbox-required Docker failures into a silent success or any host-execution fallback; sandbox-enabled modes should still fail closed with a clear error when Docker is required but unavailable.
- If the real bug path is a mode-off path reaching Docker setup, returning early from image inspection alone may only move the failure to a later Docker command such as `docker create`; the stronger product fix is for mode-off sessions to avoid Docker entirely.
- Regression coverage should include both default and custom sandbox images, browser image setup, doctor behavior, and the cron/heartbeat/sub-agent mode-off path from #73586.
