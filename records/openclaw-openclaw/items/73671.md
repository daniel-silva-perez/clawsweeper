## Summary

Keep this PR open. Current main at 6d323ee still lacks Docker-daemon-unavailable handling in the sandbox image probes, and the PR is an open implementation candidate for still-open bug #73586 rather than cleanup-close material.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to verify the exact mode-off caller that reaches Docker, then either avoid the Docker probe entirely when the effective runtime is unsandboxed or add a narrow shared Docker-daemon-unavailable classifier. Add regression coverage for mode-off cron/heartbeat/sub-agent startup with Docker unavailable, plus coverage that sandbox-enabled sessions still fail clearly when Docker is actually required.

## Risks / Open Questions

- The PR’s proposed daemon-unavailable classifier uses broad substrings such as `no such file or directory` and `connection refused`, which could mask unrelated Docker CLI failures if applied outside the exact daemon-connectivity cases.
- The proposed logic is duplicated across Docker, browser, and doctor paths in the PR context; a shared narrow classifier would reduce drift.
- Security review pass: the supplied PR file list only touches sandbox Docker/browser/doctor code and does not add dependencies, workflow changes, package metadata, lockfiles, or downloaded code execution paths.
