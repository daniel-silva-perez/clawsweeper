## Summary

Keep this PR open. Current main at 5de06ac00e7af56d82cbf32f29b733b782ce1dad still does not implement the PR's intended Docker-daemon-unavailable handling, and the linked bug #73586 remains open. The PR is a plausible implementation candidate, but it needs maintainer review around the exact mode-off call path, regression coverage, and the security/behavior impact of automatic Docker pulls.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to verify the exact cron/heartbeat/sub-agent startup path that reaches Docker while `sandbox.mode` is effectively `off`, then either stop that probe before Docker is touched or add a narrow shared Docker-daemon-unavailable classifier with tests. Add regression coverage proving mode-off startup does not require Docker, while sandbox-enabled sessions still fail clearly. Separately decide whether automatic pulling of default or custom sandbox images is acceptable or should remain an explicit user action.

## Risks / Open Questions

- The best fix may be to prevent Docker probing entirely when effective sandbox mode is off, instead of broadly suppressing Docker errors inside shared image helpers used by sandbox-enabled paths.
- The PR's classifier includes broad substrings such as `no such file or directory` and `connection refused`, which could hide unrelated Docker failures or weaken diagnostics for sandbox-enabled sessions.
- The PR changes missing-image behavior by auto-pulling both `debian:bookworm-slim` and configured custom sandbox images; that deserves explicit maintainer approval because it changes network and supply-chain behavior.
- The PR currently changes only production files and does not add regression coverage for a mode-off cron/heartbeat/sub-agent startup path with Docker unavailable.
