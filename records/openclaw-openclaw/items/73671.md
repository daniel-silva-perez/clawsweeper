## Summary

Keep open. Current main does not implement this PR’s Docker-daemon-unavailable handling, and the linked bug #73586 is still open. This PR is an implementation candidate, but it needs maintainer review and regression coverage rather than cleanup closure.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to verify the separate call path that reaches Docker while the effective sandbox mode is off, then fix that caller or add narrowly scoped daemon-unavailable handling with tests for both mode-off no-Docker behavior and sandbox-enabled clear failure behavior.

## Risks / Open Questions

- The PR patch shown in the provided context returns from image checks when the Docker daemon is unavailable; for sandbox-enabled sessions this may move the failure to a later `docker create`/`docker start` call instead of preserving the current high-signal image/daemon error.
- The substring classifier in the PR context is broad (`no such file or directory`, `connection refused`, `dial unix`) and should be scoped so unrelated Docker CLI failures are not silently treated as a missing daemon.
- No regression test is shown for the exact reported path: sandbox mode off, daemon unavailable, cron/heartbeat/sub-agent startup should make no Docker probe and should not burn fallback model attempts.
