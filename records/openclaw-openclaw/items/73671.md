## Summary

Keep this PR open. Current `origin/main` at `4b69dc6228c716663b4cd1d31384a1116e3892cd` still has the Docker image probe behavior this PR is trying to change, and the linked bug #73586 is still an open implementation target in the supplied GitHub context. The PR is plausible but needs maintainer review, regression tests, and a tighter security/behavior review around Docker error classification and custom image pulls.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to verify the exact cron/heartbeat/sub-agent path that reaches Docker when effective sandbox mode is off, fix that path or add a narrow shared Docker-daemon-unavailable classifier, and add regression coverage showing mode-off startup does not require Docker while sandbox-enabled sessions still fail clearly. Separately decide whether automatic pulling of custom sandbox images is acceptable or should remain an explicit user action.

## Risks / Open Questions

- The best fix may be to prevent Docker probing entirely when effective sandbox mode is off, rather than broadly suppressing Docker errors in shared image helpers.
- The PR's broad error substrings such as `no such file or directory` and `connection refused` could hide unrelated Docker failures and weaken diagnostics for sandbox-enabled sessions.
- The latest PR commit appears to auto-pull configured custom sandbox images when missing; that is a supply-chain and behavior change beyond graceful handling of a stopped Docker daemon.
