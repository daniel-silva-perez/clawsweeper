## Summary

Keep this PR open. Current main still lacks the PR's intended Docker-daemon-unavailable handling, the linked bug remains open, and the PR is an active implementation candidate rather than obsolete. The provided diff also still needs maintainer review because it changes Docker image resolution behavior and appears to introduce an import/export gap for the shared daemon classifier.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to verify the exact startup path that reaches Docker while sandbox mode is off, then either prevent Docker probing before Docker is touched or add a narrow shared Docker-daemon-unavailable classifier with regression tests for mode-off cron, heartbeat, and sub-agent sessions. The PR should also avoid changing default-image pull/substitution policy unless maintainers explicitly approve that product and supply-chain behavior.

## Risks / Open Questions

- The provided PR diff changes Docker image resolution by adding pull/tag behavior around debian:bookworm-slim, which conflicts with current docs/tests requiring the OpenClaw sandbox image to be built explicitly.
- The proposed daemon-unavailable classifier includes broad substrings such as `no such file or directory` and `connection refused`, which could hide unrelated Docker failures on sandbox-enabled paths unless narrowed and tested.
- The provided changed-file list does not include src/agents/sandbox.ts, but doctor-sandbox.ts imports the new helper from the sandbox barrel; current main's barrel does not export that helper, so the PR appears to need a build/typecheck fix.
- The PR context does not show focused regression coverage for the cron, heartbeat, or sub-agent startup path with sandbox mode off and Docker unavailable.
