## Summary

Keep this PR open. Current main still does not implement the PR's intended Docker-daemon-unavailable handling, and the linked bug #73586 remains open. The PR is a plausible fix candidate, but it needs maintainer review because the current diff broadens Docker pull/tag behavior, lacks focused regression coverage, and appears to have an export/typecheck gap in the doctor import path.

## Best Possible Solution

Keep this PR open for maintainer review. The best fix is to trace the exact mode-off cron/heartbeat/sub-agent path and prevent Docker probing before Docker is touched. If defensive Docker error classification is still needed, add a narrow shared helper, export it correctly, and cover daemon-unavailable output plus mode-off startup behavior with tests. Separately decide whether automatic Docker pulls/tags belong here or should remain explicit user setup.

## Risks / Open Questions

- The PR changes sandbox image resolution from explicit user setup to automatic `docker pull` and `docker tag`, including configured custom image names; that needs explicit maintainer approval.
- The proposed classifier includes broad substrings such as `no such file or directory` and `connection refused`, which could hide unrelated Docker errors on sandbox-enabled paths if not scoped carefully.
- The PR context and diff show no focused regression test proving `sandbox.mode: "off"` cron, heartbeat, or sub-agent startup avoids Docker while sandbox-enabled sessions still fail clearly.
- The PR head appears to have a type/export issue for `isDockerDaemonUnavailable` through `src/agents/sandbox.ts`, so it is not ready to close or merge as implemented.
