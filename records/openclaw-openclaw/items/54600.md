## Summary

Keep open. Current main still does not extract object-shaped cron `agentTurn.payload.message` values before isolated execution, so the PR targets a real remaining bug, but the branch adds a fallback debug log that can expose serialized prompt content and should not merge as-is.

## What This Changes

The PR changes `src/cron/service/timer.ts` to unwrap object-shaped cron `agentTurn.payload.message` values from `text`, `message`, or `content`, with a JSON stringify fallback, before calling the isolated cron runner.

## Best Possible Solution

Land a tested cron fix that preserves the public `message: string` contract: repair or normalize malformed stored payloads before execution where appropriate, and keep any defensive extraction at the isolated execution boundary narrow and free of prompt-content logging.

## Reproduction Assessment

Yes. A malformed persisted/runtime `agentTurn` payload with an object-valued `message` reproduces the observed failure path because current main forwards that value unchanged and the isolated runner interpolates it into the model prompt; the linked report provides a manual cron add/run scenario, though the exact source of the object shape still needs confirmation.

## Solution Assessment

No, not as written. Defensive extraction is a reasonable part of the fix, but logging serialized prompt content and omitting focused regression coverage make this PR unsafe to merge without revision; logging only shape metadata or keys would be the safer fallback.
