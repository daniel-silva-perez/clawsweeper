## Summary

Keep this PR open for maintainer review, but do not merge it as-is. Current main still uses a strict core plugin-entry schema, and the documented/runtime contract routes plugin-owned settings through `plugins.entries.<id>.config`, validates that nested payload against the plugin manifest `configSchema`, and passes only that validated nested config to plugin APIs. The PR’s broad top-level passthrough would accept misplaced plugin settings without making them part of the validated runtime config path.

## Best Possible Solution

Request changes and keep this PR open for maintainer review. The best fix is to preserve the explicit core plugin-entry shape, keep plugin-owned settings under `plugins.entries.<id>.config`, and add regression coverage for base validation, manifest `configSchema` validation/defaulting, normalization, and `api.pluginConfig` runtime delivery. If maintainers want legacy top-level compatibility, implement an explicit migration or normalization path with warnings instead of raw passthrough.

## Risks / Open Questions

- Merging the PR as-is could turn actionable `Unrecognized keys` failures into accepted but ignored plugin settings, because normalization and runtime delivery still use only `entry.config`.
- Broad `.passthrough()` expands the core config surface outside plugin-owned `configSchema` validation, which can hide unsupported fields, typos, or misplaced credentials instead of producing a clear validation error.
- No supply-chain or CI execution regression is visible from the provided PR diff; the main risk is config validation and secret/config hygiene.
