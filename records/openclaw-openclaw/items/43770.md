## Summary

Keep open for maintainer handling, but the patch should not merge as-is. Current main has not implemented the PR's requested top-level plugin-entry passthrough behavior, and the proposed change would accept provider-specific keys outside the nested `config` path that current validation, normalization, and runtime delivery actually use.

## What This Changes

The PR changes `PluginEntrySchema` from `.strict()` to `.passthrough()` and adds a regression test asserting top-level plugin-entry keys such as `mode`, `userId`, `autoRecall`, and `oss` are accepted and preserved.

## Best Possible Solution

Keep the plugin-entry envelope explicit and route plugin-owned settings through `plugins.entries.<id>.config`, where manifest validation, defaults, runtime `api.pluginConfig` delivery, and secret handling stay aligned. If legacy top-level compatibility is wanted, add a deliberate migration or normalization path with warnings and tests instead of raw passthrough.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep plugin-owned keys under config:** `src/config/zod-schema.ts:179`
  - body: The new `.passthrough()` accepts keys like `plugins.entries.openclaw-mem0.mode`, but current validation and runtime still consume only `entry.config`: plugin-aware validation validates `entry?.config ?? {}`, normalization copies only the known envelope plus `config`, and the loader passes only the validated nested config to `api.pluginConfig`. This turns a loud schema error into a config that validates but is not applied.
  - confidence: 0.91

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete config hygiene concern by accepting arbitrary unvalidated plugin-entry fields outside the schema-backed plugin config path.

Concerns:

- **[medium] Avoid accepting unvalidated plugin-entry fields:** `src/config/zod-schema.ts:179`
  - body: `.passthrough()` would preserve arbitrary top-level fields under `plugins.entries.<id>` even though manifest validation and runtime delivery consume nested `config`. Misplaced credentials or provider settings could be accepted outside the plugin schema and SecretInput-oriented config path instead of being rejected or migrated deliberately.
  - confidence: 0.82
