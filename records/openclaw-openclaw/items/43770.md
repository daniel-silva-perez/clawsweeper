## Summary

Keep open, but the patch should not merge as written. Current main treats `plugins.entries.<id>` as a strict core-owned envelope and routes plugin-owned settings through nested `config`; accepting arbitrary top-level fields would make misplaced plugin settings validate while still bypassing manifest validation, normalization, and runtime delivery.

## What This Changes

The PR changes `PluginEntrySchema` from `.strict()` to `.passthrough()` and adds a regression test asserting extra top-level plugin-entry keys are accepted and preserved.

## Best Possible Solution

Keep the explicit plugin-entry envelope and route plugin-owned settings through `plugins.entries.<id>.config`, with regression coverage for base validation, manifest `configSchema` validation/defaulting, normalization, and runtime `api.pluginConfig` delivery. If compatibility for legacy top-level keys is desired, add a deliberate migration or normalization path with warnings rather than raw passthrough.

## Review Findings

  - body: Changing this schema to `.passthrough()` makes top-level plugin settings validate, but the rest of the pipeline still validates, normalizes, and passes only `entry.config` to plugins.
