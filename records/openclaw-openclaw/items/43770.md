## Summary

Keep this PR open for maintainer review, but the patch should not merge as-is. Current main still has a strict base plugin-entry schema, while the validated and runtime plugin config contract is the nested `plugins.entries.<id>.config` payload; the PR's broad top-level passthrough would accept misplaced provider settings without making them part of manifest validation, normalization, or `api.pluginConfig` delivery.

## What This Changes

The PR changes `PluginEntrySchema` from `.strict()` to `.passthrough()` and adds a regression test asserting extra top-level plugin-entry keys are accepted and preserved.

## Best Possible Solution

Keep the explicit plugin-entry envelope and route plugin-owned settings through `plugins.entries.<id>.config`, with regression coverage for base validation, manifest `configSchema` validation/defaulting, normalization, and runtime `api.pluginConfig` delivery. If maintainers want compatibility for legacy top-level plugin keys, add an explicit migration or normalization path with warnings rather than raw passthrough.

## Review Findings

- **[P1] Keep plugin-owned keys under config:** `src/config/zod-schema.ts:179`
  - body: Changing this schema to `.passthrough()` makes top-level plugin settings validate, but the rest of the pipeline still validates, normalizes, and passes only `entry.config` to plugins.
