## Summary

Keep open for maintainer follow-up; the patch should not merge as-is because it makes top-level plugin-owned fields pass base validation while the current validation, normalization, and runtime delivery paths still consume only nested `config`. That would turn a loud configuration error into accepted-but-ignored settings and bypass the schema-backed plugin config path.

## What This Changes

The PR changes `PluginEntrySchema` from `.strict()` to `.passthrough()` and adds a regression test asserting extra top-level plugin-entry keys such as `mode`, `userId`, `autoRecall`, and `oss` are accepted and preserved.

## Best Possible Solution

Keep the plugin-entry envelope explicit and route plugin-owned settings through `plugins.entries.<id>.config`, where manifest `configSchema` validation, defaulting, normalization, runtime `api.pluginConfig` delivery, and secret handling can stay aligned. If compatibility for legacy top-level keys is desired, add a deliberate typed migration or normalization path with warnings and tests instead of raw passthrough.

## Review Findings

This turns a previously loud config error into an accepted value that the plugin never receives.

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete config hygiene concern by accepting arbitrary unvalidated plugin-entry fields outside the schema-backed plugin config path.

Concerns:

- **[medium] Avoid accepting unvalidated plugin-entry fields:** `src/config/zod-schema.ts:179`
  - body: `.passthrough()` would preserve arbitrary top-level fields under `plugins.entries.<id>` even though manifest validation and runtime delivery consume nested `config`. Misplaced credentials or provider settings could be accepted outside the plugin schema and SecretInput-oriented config path instead of being rejected or migrated deliberately.
