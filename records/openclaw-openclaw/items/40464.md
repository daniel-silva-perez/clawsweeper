## Summary

Keep this PR open for maintainer review. The underlying config data-integrity concern is real, and current main has guard/recovery coverage for the catastrophic wipe, but it does not contain the PR's queued fresh-read owner-secret transaction; at the same time, the PR branch is stale enough that applying it as-is would regress current config-write contracts.

## What This Changes

The PR adds a module-level config write queue, changes ownerDisplaySecret auto-persistence to re-read and write inside the queued critical section, and adds queue/runtime-snapshot regression tests.

## Best Possible Solution

Implement the ownerDisplaySecret auto-persist as a current-main transaction: serialize the relevant write, read a fresh write snapshot inside that critical section, merge only the generated commands.ownerDisplaySecret, and preserve current config write safety, runtime refresh/finalization, plugin migration, skip/log flags, and write-listener notification behavior.
