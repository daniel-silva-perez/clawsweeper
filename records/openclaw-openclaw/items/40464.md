## Summary

Keep this PR open: the startup config-wipe bug is real and current main has shipped clobber recovery, but current main does not contain this branch's serialization/fresh-read approach, and the branch would regress newer config write contracts if merged as-is.

## What This Changes

The PR adds a module-level config write queue, changes ownerDisplaySecret auto-persistence to re-read and write inside that queued section, and adds config write queue/runtime snapshot regression tests.

## Best Possible Solution

The desired end state is one coherent config persistence path where destructive-write protection, runtime snapshot finalization, write notifications, and any optional queued fresh-read behavior all remain covered by regression tests.

## Reproduction Assessment

Yes. The linked report gives a concrete populated-config restart/truncation scenario, and this PR adds targeted queue and ownerDisplaySecret race coverage; in this read-only pass I verified the relevant source paths and diff contracts rather than executing tests.

## Solution Assessment

No as-is. A queue plus fresh read may still be a viable hardening direction, but this branch needs to be reworked against the current config writer contract before it is a maintainable fix.
