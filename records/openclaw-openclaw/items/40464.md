## Summary

Keep this PR open. Current main has added destructive-write rejection, rejected-payload artifacts, and last-known-good recovery for severe config clobbers, but it still does not implement the PR’s central serialized/fresh-read ownerDisplaySecret persistence path. The proposed change is not obsolete, though it needs a current-main rebase or narrower rewrite before it can land safely.

## Best Possible Solution

Keep this PR open for maintainer review, but ask for a current-main rebase or narrow rewrite. The best path is a per-config-path ownerDisplaySecret persistence transaction: serialize that write, read the fresh source snapshot inside the lock with readConfigFileSnapshotForWrite, merge only commands.ownerDisplaySecret, preserve retry behavior, and retain current destructive-write, recovery, runtime snapshot, env-ref, and plugin-install contracts. Add regression coverage for the concurrent startup write scenario described by #40410.
