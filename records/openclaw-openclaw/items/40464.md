## Summary

Keep open: this is a plausible fix for a real config data-loss report, but the branch is stale against current main's config writer and would drop current writer contracts if rebased as-is. Current main has destructive-clobber mitigations and docs, but not this queue/fresh-read implementation, so it is neither safe to close as implemented nor safe to merge unchanged.

## What This Changes

The PR adds a module-level config write queue, changes ownerDisplaySecret auto-persistence to re-read and write inside the queued section, and adds queued-write/runtime-snapshot regression tests.

## Best Possible Solution

Keep one coherent config persistence path: if queueing/fresh-read behavior is accepted, layer it around the current module-level writer so destructive-write protection, runtime snapshot finalization, canonical source notifications, skip/log flags, persisted hashes, and ownerDisplaySecret race coverage all stay intact.

## Reproduction Assessment

Yes. The linked config-wipe report gives concrete restart/truncation steps, and this PR adds targeted ownerDisplaySecret and queued-write race tests; this pass verified the source and diff paths rather than executing tests.

## Solution Assessment

No as-is. A queue plus ownerDisplaySecret fresh-read may be a maintainable direction, but the current branch must be rebased and reworked to preserve current main's full config-write contract.
