## Summary

Keep this PR open. Current main has strong guardrails for the catastrophic config-wipe symptom, but it still does not implement the PR's central serialized, fresh-read ownerDisplaySecret persistence path. The remaining lost-update surface is still plausible because current main persists a full config snapshot produced during load rather than rereading and merging only the generated secret inside a write transaction.

## Best Possible Solution

Keep this PR open for maintainer review, but require a current-main rebase or narrow rewrite. The best implementation is a per-config-path ownerDisplaySecret auto-persist transaction: serialize the relevant write, read the fresh source snapshot inside the lock with `readConfigFileSnapshotForWrite`, merge only the generated secret, preserve retry behavior, and keep current destructive-write rejection, last-known-good recovery, env-ref restoration, runtime snapshot refresh, and plugin install migration contracts. Add regression coverage proving concurrent startup writes survive while the generated owner secret is persisted.

## Risks / Open Questions

- Do not land the old branch unchanged: current main has destructive-write rejection, last-known-good recovery, env-ref restoration, runtime snapshot refresh, and plugin install migration behavior that a rebase must preserve.
- Config persistence and `commands.ownerDisplaySecret` affect data integrity and stable owner hashing, so the final fix needs targeted regression coverage for concurrent startup writes surviving while the generated owner secret is persisted.
- The PR's broad queue approach may be more invasive than needed; a narrower per-config-path ownerDisplaySecret transaction may reduce runtime snapshot and profile-switching risk.
