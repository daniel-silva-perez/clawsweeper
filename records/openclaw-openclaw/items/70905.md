## Summary

Keep open. Current main still does not expose a stable `agentId` on the plugin-facing outbound context, even though the outbound delivery path already has session or mirror agent ids available for adjacent media scoping. The related implementation PR was closed unmerged, so there is no shipped or superseding fix to close against.

## What This Changes

Request to add an optional `agentId` to `ChannelOutboundContext` and populate it for channel outbound sends so plugins can attribute messages to the producing agent.

## Best Possible Solution

Add a backward-compatible optional `agentId?: string` to the channel outbound context, thread it from session first and mirror second into every adapter context path, omit the key when no id exists, document the plugin-facing contract, and add focused tests for session, mirror fallback, and absent-id behavior.

## Reproduction Assessment

Yes. A read-only source trace on current main shows `OutboundSessionContext` and delivery mirror data can carry `agentId`, while `ChannelOutboundContext`, `ChannelHandlerParams`, and `createChannelOutboundContextBase` still omit it before adapter send methods run.

## Solution Assessment

Unclear until maintainers approve the public SDK contract, but the proposed technical direction is the narrowest maintainable fix: keep stable routing metadata separate from display-only `identity` and thread the already-available session or mirror id into outbound adapter contexts.
