## Summary

Keep open. Current main still does not expose a stable agent id on the plugin-facing outbound context: upstream session and mirror metadata carry `agentId`, and delivery already uses it for media access, but the adapter context type and builder drop it before `sendText` / `sendMedia` calls. The direct implementation PR was closed unmerged, so it does not supersede the request.

## What This Changes

Expose an optional stable `agentId` on `ChannelOutboundContext` so channel plugins can attribute outbound sends to the producing agent.

## Best Possible Solution

Add a backward-compatible optional `agentId?: string` to the channel outbound context, thread it from existing session or mirror metadata with explicit precedence, omit the property when no agent is in scope, document the attribution-only contract, and cover session, mirror fallback, and no-id sends in focused outbound delivery tests.

## Reproduction Assessment

Yes. A read-only source trace on current main is enough: session and mirror metadata can contain `agentId`, but the plugin-facing outbound type, handler params, and context builder do not pass it to adapter calls.

## Solution Assessment

Yes, with maintainer approval. Keeping stable agent routing metadata separate from display-only `identity` is the narrowest maintainable shape; the safer implementation should also document the trust boundary and avoid emitting `agentId: undefined`.
