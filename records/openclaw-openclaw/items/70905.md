## Summary

Keep open. Current main still does not expose a stable agent id on the plugin-facing outbound context, while the upstream session and mirror metadata already carry one; the only direct implementation PR was closed unmerged and did not supersede the request.

## What This Changes

Add an optional stable `agentId` to `ChannelOutboundContext` and thread it through outbound delivery so channel plugins can attribute sends to the producing agent.

## Best Possible Solution

Add a backward-compatible optional `agentId?: string` to the outbound channel context, thread it from trusted session or mirror metadata with clear session-over-mirror precedence, omit the property when absent, document the attribution-only contract, and add focused regression coverage for session, mirror fallback, and no-id sends.

## Reproduction Assessment

Yes. A read-only source trace on current main is enough: session and mirror contexts can contain `agentId`, but the exported outbound adapter context, handler params, and context builder do not carry it into plugin adapter calls.

## Solution Assessment

Yes, with maintainer approval. Keeping stable agent routing metadata separate from display-only `identity` is the narrowest maintainable shape, but the implementation must address absent-key behavior and the gateway trust boundary raised on the closed PR.
