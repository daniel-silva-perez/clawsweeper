## Summary

Keep open. Current main still lacks the requested plugin-facing `agentId` on `ChannelOutboundContext`, even though the outbound delivery path already has session and mirror agent metadata available. The linked implementation PR #73072 was closed unmerged, so it does not supersede this still-valid feature request.

## What This Changes

Expose an optional stable agent id on outbound channel plugin send contexts so plugins can attribute outbound messages to the agent that produced them.

## Best Possible Solution

Add a backward-compatible optional `agentId?: string` to the outbound channel context and internal handler params, populate it from session agent id first and mirror agent id second, omit the key when no agent is in scope, document the attribution-only contract, and cover session, mirror fallback, and no-id cases in delivery tests plus SDK baseline checks.

## Reproduction Assessment

Yes. A read-only source trace on current main reproduces the gap: `session.agentId` and `mirror.agentId` are available upstream, but the public outbound adapter context type and builder do not expose them to channel plugins.

## Solution Assessment

Yes, with maintainer approval. Keeping stable agent routing metadata separate from display-only `identity` is the narrowest maintainable approach, provided the patch documents the trust boundary and absent-key semantics.
