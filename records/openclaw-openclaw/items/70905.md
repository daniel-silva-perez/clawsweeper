## Summary

Keep open. Current main still lacks the requested plugin-facing `agentId` on outbound channel adapter contexts, while the upstream outbound path already carries agent identity metadata and the related implementation PR was closed unmerged rather than superseding the request.

## What This Changes

Expose an optional stable `agentId` on `ChannelOutboundContext` and thread existing session or mirror agent identity metadata into channel outbound sends.

## Best Possible Solution

Add a backward-compatible optional `agentId?: string` to the channel outbound context, populate it from session agent id first and mirror agent id second, omit it when no agent is in scope, document the attribution-only contract, and cover session, mirror fallback, and no-id delivery cases.

## Reproduction Assessment

Yes. A read-only source trace on current main reproduces the gap: session and mirror metadata can contain `agentId`, but the public outbound context type, handler params, and context builder do not carry it into adapter calls.

## Solution Assessment

Yes, with maintainer approval. Keeping stable agent routing metadata separate from display-only `identity` is the narrowest maintainable shape, provided the patch clarifies trust-boundary semantics and absent-key behavior.
