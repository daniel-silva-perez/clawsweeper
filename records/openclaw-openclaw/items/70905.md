## Summary

Keep open. Current main still lacks `agentId` on the plugin-facing outbound context and does not pass the already-available session or mirror agent id into adapter calls. The linked implementation PR was closed unmerged, so there is no implemented or superseding fix to close against.

## What This Changes

Add optional `agentId` metadata to `ChannelOutboundContext` so channel plugins can attribute outbound sends to the producing agent.

## Best Possible Solution

Add a backward-compatible optional `agentId?: string` to the public channel outbound context, thread it from existing session or mirror metadata into every outbound adapter context path, document the plugin-facing contract, and cover session, mirror fallback, and absent-id behavior with focused tests.

## Reproduction Assessment

Yes. A read-only source trace against current main shows `OutboundSessionContext` and `DeliveryMirror` can carry `agentId`, while `ChannelOutboundContext`, `ChannelHandlerParams`, and `createChannelOutboundContextBase` omit it before adapter send methods run.

## Solution Assessment

Yes, with maintainer approval. Keeping stable routing metadata separate from display-only `identity` is the narrowest maintainable direction; a future patch should conditionally omit the key when no id exists and avoid changing gateway authorization semantics as a side effect.
