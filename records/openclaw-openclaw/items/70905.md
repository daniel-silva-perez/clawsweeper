## Summary

Keep open. Current main still exposes `ChannelOutboundContext` without `agentId` and builds outbound adapter contexts without forwarding the `agentId` that is already present on session and mirror inputs. The linked implementation PR #73072 was closed unmerged, so there is no shipped or superseding fix to close against.

## What This Changes

Request adding optional `agentId` metadata to `ChannelOutboundContext` so outbound channel plugin send calls can attribute messages to the active agent.

## Best Possible Solution

Add a backwards-compatible optional `agentId?: string` to the public channel outbound context, thread it from existing session or mirror metadata into all outbound adapter context shapes, document the plugin-facing contract, and lock the behavior with focused delivery tests for session, mirror fallback, and absence cases.

## Reproduction Assessment

Yes. A read-only source trace shows `buildOutboundSessionContext` and `DeliveryMirror` can carry `agentId`, while `ChannelOutboundContext` and `createChannelOutboundContextBase` omit it before invoking adapter send methods.

## Solution Assessment

Yes. The proposed optional context field is the narrowest maintainable fix because it keeps stable routing metadata separate from display-only `identity` fields and avoids racy plugin-local workarounds.
