## Summary

Keep open. Current main still lacks the requested plugin-facing `agentId` on `ChannelOutboundContext`, while the outbound delivery path already has session and mirror agent metadata in scope. The related implementation PR #73072 was closed unmerged, so it does not supersede this issue, and the remaining decision is a public plugin SDK contract/trust-boundary approval rather than a conservative close.

## What This Changes

Add an optional stable `agentId` to `ChannelOutboundContext` so channel plugins can attribute outbound sends to the producing agent without abusing display-only identity fields.

## Best Possible Solution

Add a backward-compatible optional `agentId?: string` to the outbound channel context and internal handler params, populate it from session agent id first and mirror agent id second, omit the key when no agent is in scope, document the attribution-only contract, and cover session, mirror fallback, and no-id cases in delivery tests plus SDK baseline checks.

## Reproduction Assessment

Yes. A read-only source trace on current main reproduces the gap: `session.agentId` and `mirror.agentId` are available upstream, but the public outbound adapter context type and builder do not expose them to channel plugins.

## Solution Assessment

Yes, with maintainer sign-off. Keeping stable agent routing metadata separate from display-only `identity` is the narrowest maintainable direction, provided the implementation documents trust semantics and avoids adding a present-but-undefined key when no agent is in scope.
