## Summary

Keep open. Current main still lacks `agentId` on the plugin-facing outbound context even though session and mirror metadata already carry it upstream, and the only direct implementation PR was closed unmerged rather than superseding this request.

## What This Changes

Add optional `agentId` metadata to `ChannelOutboundContext` so channel plugins can attribute outbound sends to the producing agent.

## Best Possible Solution

Expose a backward-compatible optional `agentId?: string` on the channel outbound context, thread it from existing session or mirror metadata with clear precedence, omit the key when absent, document the attribution-only contract, and add focused regression coverage for session, mirror fallback, and no-id sends.

## Reproduction Assessment

Yes. A read-only source trace on current main is enough: session and mirror contexts can contain `agentId`, but the plugin-facing outbound type, handler params, and context builder do not carry it into adapter calls.

## Solution Assessment

Yes, with maintainer approval. Keeping stable agent routing metadata separate from display-only `identity` is the narrowest maintainable shape, but the implementation should address absent-key behavior and the gateway trust boundary called out on the closed PR.
