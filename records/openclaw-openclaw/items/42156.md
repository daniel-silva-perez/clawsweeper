## Summary

Keep this issue open. Current main still answers Telegram callback queries inside the downstream callback handler, while grammY sequentialize is registered earlier. The later approval-button lane mitigation only helps parsed approval callbacks; ordinary inline-button callbacks still share the normal chat/topic lane and can wait behind an active turn.

## Best Possible Solution

Keep this open and implement the Telegram plugin fix on current main: add pre-sequentialize middleware in extensions/telegram/src/bot-core.ts that immediately answers callbackQuery updates and catches/logs API failures, remove or avoid redundant handler-level acknowledgement in extensions/telegram/src/bot-handlers.runtime.ts, and add a regression test that drives the middleware order with a busy same-chat/topic turn plus ordinary callback data to prove answerCallbackQuery fires before the queued callback action is released. The closed contributor PRs #42125, #42181, and #42516 are good implementation references but need a fresh maintainer-ready patch.
