## Summary

Keep this issue open. Current main still acknowledges Telegram callback queries inside the downstream `callback_query` handler, after `@grammyjs/runner` sequentialize has already queued same-key updates. The newer approval-lane mitigation only moves parsed approval callbacks to a separate lane; ordinary inline-button callback data still uses the normal chat/topic lane and can wait behind an active agent turn. The contributor PRs #42125, #42181, and #42516 are useful implementation references, but they are closed unmerged and their pre-sequentialize acknowledgement approach is not present on main.

## Best Possible Solution

Keep this issue open and implement the Telegram plugin fix on current main: add pre-sequentialize middleware in `extensions/telegram/src/bot-core.ts` that immediately answers `callbackQuery` updates and catches/logs API failures, remove or avoid redundant handler-level acknowledgement in `extensions/telegram/src/bot-handlers.runtime.ts`, and add a regression test that drives middleware order with a busy same-chat/topic turn plus ordinary callback data to prove `answerCallbackQuery` fires before the queued callback action is released. The closed contributor PRs #42125, #42181, and #42516 are good references for the intended shape, but a fresh maintainer-ready patch is still needed.
