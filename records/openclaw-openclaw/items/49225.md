## Summary

Keep this issue open. Current main still has independent WhatsApp reaction, text-send, and quoted-reply paths, and the auto-reply path still does not surface a delivery-state reconciliation signal when model/session text exists but provider delivery fails or is not confirmed.

## Best Possible Solution

Keep the issue open. The best fix is a WhatsApp-owned delivery-state record around `dispatchWhatsAppBufferedReply`/`deliverWebReply` and the direct send/reaction APIs: capture model-produced text, queued delivery kind, provider dispatch invocation, provider message id/error per mode, and whether transcript/session text was recorded. Surface a clear operator/debug warning when text is produced but no WhatsApp provider delivery is invoked or accepted, and add regression coverage where ack reaction succeeds while plain text or quoted reply delivery fails.
