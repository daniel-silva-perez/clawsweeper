## Summary

Keep open. Current main still uses the monolithic locked sandbox registry, so the PR's performance direction is not already implemented; however, the branch is not mergeable because it drops same-entry coordination and can resurrect a runtime entry after recreate/prune/remove. The earlier ClawSweeper comment already identified this blocking race, and the latest PR head still has the same behavior.

## What This Changes

The PR replaces sandbox container and browser monolithic registry JSON files with per-entry files, adds single-entry lookup for Docker ensure, migrates legacy files, and rewrites registry tests for sharded storage.

## Best Possible Solution

Keep the sharded-storage direction only if the branch preserves same-entry update/remove ordering with a per-entry lock, tombstone, or compare-and-swap style protocol; propagate non-ENOENT filesystem errors; retain migration behavior; and restore focused race tests for both container and browser registries.

## Reproduction Assessment

Yes for the PR regression. Current main already has focused tests that gate an update write, race removeRegistryEntry/removeBrowserRegistryEntry, and assert the entry remains deleted; the PR removes that coverage while changing removal to an uncoordinated file delete.

## Solution Assessment

No. Sharding disjoint entries is a plausible performance direction, but dropping all coordination for mutations of the same runtime name is not the narrowest maintainable fix; per-entry coordination would keep the performance win without weakening remove semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Preserve same-entry remove/update ordering:** `src/agents/sandbox/registry.ts:223`
  - body: Current main serializes registry mutations and tests that a remove starting after an in-flight update leaves the entry deleted. Here removeRegistryEntry becomes a plain file delete while updateRegistry can still finish a writeJsonAtomic rename after the remove, so recreate/prune can re-add stale state for the same runtime. Keep per-entry coordination or equivalent tombstone/compare semantics for same-name mutations.
  - confidence: 0.89
- **[P3] Propagate registry removal failures:** `src/agents/sandbox/registry.ts:112-116`
  - body: fs.rm(..., { force: true }) already ignores missing files, but this catch also hides EACCES, ENOTDIR, and other I/O failures. openclaw sandbox recreate can report success after the runtime was removed while the registry file remains, so later list/prune sees stale state. Only ignore expected missing or concurrent-remove cases.
  - confidence: 0.82
