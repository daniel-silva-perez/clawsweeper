## Summary

Keep this PR open, but do not merge it as-is. The latest PR head still removes the same-entry mutation ordering that current main relies on and hides unexpected registry deletion failures; the existing ClawSweeper bot comment already flagged these issues, and the current diff still contains them after the later merge from main.

## What This Changes

The PR shards sandbox container and browser registries from monolithic JSON files into per-entry files, adds single-entry container lookup for Docker ensure, migrates legacy files, and updates sandbox registry tests.

## Best Possible Solution

Keep the sharded-registry direction only if it preserves same-container and same-browser mutation ordering with a per-entry lock, tombstone, compare-and-swap, or equivalent protocol, restores focused race coverage, and narrows deletion error suppression to expected missing-file cases.

## Reproduction Assessment

Yes for the blocking PR regression. Current main has focused tests that delay a registry update, race a removal, and assert the entry stays deleted; the PR removes that ordering guarantee and no longer carries equivalent race coverage.

## Solution Assessment

No. Per-entry sharding may be the right performance direction, but this implementation is not the safest maintainable fix until same-entry ordering and unexpected delete-error propagation are preserved.

- **[P2] Preserve same-entry remove/update ordering:** `src/agents/sandbox/registry.ts:219`
  - body: Current main serializes registry mutations and tests that a delayed update cannot resurrect an entry removed after that update started. This write can read the old per-entry file and later recreate it after removeRegistryEntry deletes it, leaving stale container or browser state after recreate/prune; add per-entry locking, tombstones, compare-and-swap, or equivalent ordering and restore the race tests.
  - confidence: 0.9

  - body: fs.rm(..., { force: true }) already ignores missing files, but this catch also hides EACCES, ENOTDIR, and other I/O failures. Sandbox recreate/prune can report success while the registry file remains, so only suppress ENOENT or another deliberately handled concurrent-remove case.
  - confidence: 0.82
