## Summary

Keep open. The sharded registry direction may be useful, but this PR is not mergeable as-is: it removes current same-name registry mutation ordering and suppresses unexpected deletion errors, both of which can leave stale sandbox runtime state behind. The existing ClawSweeper comment already flagged these issues, and this review confirmed them against current main and PR head.

## What This Changes

The PR changes sandbox container and browser registries from monolithic JSON files to per-entry JSON files, adds single-entry lookup for Docker ensure, migrates legacy registry files, and rewrites related registry tests.

## Best Possible Solution

Keep the sharded-registry approach only after preserving same-container and same-browser mutation ordering with a per-entry lock, tombstone, compare-and-swap, or equivalent protocol; propagate unexpected delete failures; and retain focused regression tests for update/remove races and deletion error handling.

## Reproduction Assessment

Yes for the PR regression. Current main has focused tests that delay a registry update, race a removal, and assert the entry stays deleted; the PR head's read-then-write per-entry path can write after the delete and recreate stale state.

## Solution Assessment

No. Per-entry sharding is a plausible performance fix, but dropping same-name coordination and hiding delete I/O failures is not the narrowest maintainable solution; per-entry coordination preserves the performance goal without weakening recreate/prune semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve same-entry remove/update ordering:** `src/agents/sandbox/registry.ts:219`
  - body: Current main serializes registry mutations and tests that a delayed update cannot resurrect an entry removed after that update started. This write can read the old entry and later write it after removeRegistryEntry deletes the per-entry file, so sandbox recreate/prune can leave stale container or browser registry state. Add per-entry locking, tombstones, compare-and-swap, or equivalent ordering and restore the race tests.
  - confidence: 0.9
- **[P3] Propagate unexpected registry removal errors:** `src/agents/sandbox/registry.ts:112-116`
  - body: fs.rm(..., { force: true }) already ignores missing paths, but this catch also hides EACCES, ENOTDIR, and other I/O failures. Sandbox recreate/prune can report runtime removal success while the registry file remains, so only suppress ENOENT or an explicitly handled concurrent-remove case.
  - confidence: 0.82
