## Summary

Keep open. The sharded registry direction is plausible, but the current PR head still drops same-entry mutation ordering that current main explicitly protects, and it hides unexpected registry delete failures. This should be revised on the contributor branch before merge rather than closed or replaced.

## What This Changes

This PR shards sandbox container and browser registries from monolithic JSON files into per-entry JSON files, adds single-entry lookup for Docker ensure, migrates legacy registry files, and updates related tests.

## Best Possible Solution

Keep the sharded-registry design only if it preserves same-container and same-browser mutation ordering with a per-entry lock, tombstone, compare-and-swap, or equivalent protocol, and only suppresses expected missing-file deletion cases while preserving focused race and delete-error regression coverage.

## Reproduction Assessment

Yes for the PR regression. Current main has focused tests that gate a registry write, race a removal, and assert the removed entry stays deleted; the PR head removes that protection and its read-then-write per-entry path can write after delete.

## Solution Assessment

No. Per-entry sharding can be the right performance direction, but the current implementation is not the narrowest maintainable fix because it weakens recreate/prune semantics and hides I/O failures.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve same-entry remove/update ordering:** `src/agents/sandbox/registry.ts:219`
  - body: Current main serializes registry mutations and tests that a delayed update cannot resurrect an entry removed after that update started. This write can read the old entry and later recreate the per-entry file after removeRegistryEntry deletes it, leaving stale container or browser state after recreate/prune. Add per-entry locking, tombstones, compare-and-swap, or equivalent ordering and restore the race tests.
  - confidence: 0.9
- **[P3] Propagate unexpected registry removal errors:** `src/agents/sandbox/registry.ts:112-116`
  - body: fs.rm(..., { force: true }) already ignores missing files, but this catch also hides EACCES, ENOTDIR, and other I/O failures. Sandbox recreate/prune can report success while the registry file remains, so only suppress ENOENT or a deliberately handled concurrent-remove case.
  - confidence: 0.82
