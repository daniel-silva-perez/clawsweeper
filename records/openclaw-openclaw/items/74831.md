## Summary

Keep open. The performance direction is plausible, but the patch weakens sandbox registry correctness: current main serializes registry mutations and tests remove/update races, while the PR removes same-entry coordination, hides unexpected delete failures, and uses raw registry names as filesystem path components.

## What This Changes

This PR shards sandbox container and browser registries into per-entry JSON files, adds single-entry container lookup, migrates legacy monolithic registry files, and updates Docker ensure-path tests.

## Best Possible Solution

Keep the sharded-registry approach only if it preserves per-entry mutation ordering, confines entry filenames with an encoded or hashed basename, propagates unexpected delete failures, and restores focused regression tests around same-name races and deletion error handling.

## Reproduction Assessment

Yes for the primary regression. Current main has focused tests that race a delayed registry update against removal and assert the entry stays deleted; the PR removes that coverage and its read-then-write per-file path can write after the delete.

## Solution Assessment

No. Per-entry sharding is a reasonable performance direction, but the maintainable fix needs per-entry locking, tombstones, compare-and-swap, or an equivalent protocol plus path-safe filenames, instead of dropping the current ordering guarantees.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve same-entry remove/update ordering:** `src/agents/sandbox/registry.ts:219`
  - body: Current main serializes registry mutations and tests that a delayed update cannot resurrect an entry removed after that update started. This write can read the old entry and later write it after removeRegistryEntry deletes the per-entry file, so sandbox recreate/prune can leave stale container or browser registry state. Add per-entry locking, tombstones, compare-and-swap, or equivalent ordering and restore the race tests.
  - confidence: 0.9
- **[P2] Encode registry entry names before using them as paths:** `src/agents/sandbox/registry.ts:82-83`
  - body: containerName is a registry/backend runtime string, not a filename-safe key. Joining it directly into the sharded path lets names with slashes or `..` escape the registry directory during migration, update, read, or remove. Use an encoded or hashed filename and keep the original name inside the JSON entry.
  - confidence: 0.78
- **[P3] Propagate unexpected registry removal errors:** `src/agents/sandbox/registry.ts:112-116`
  - body: `fs.rm(..., { force: true })` already ignores missing files, but this catch also hides EACCES, ENOTDIR, and other I/O failures. Sandbox recreate/prune can report runtime removal success while the registry file remains, so only suppress ENOENT or an explicitly handled concurrent-remove case.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The PR introduces one concrete local filesystem safety concern in the new sharded registry path construction.

Concerns:

- **[medium] Constrain sharded registry filenames:** `src/agents/sandbox/registry.ts:82`
  - body: The new entryFilePath helper uses the registry entry name directly in path.join, while migration and updates write through that path. A malformed legacy registry entry or backend runtime id containing path separators can escape the intended registry directory; encode/hash names or validate a basename before filesystem operations.
  - confidence: 0.78
