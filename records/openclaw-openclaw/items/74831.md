## Summary

Keep open. This PR is not obsolete, but the current branch introduces a same-name registry update/remove race and hides unexpected registry deletion failures; the existing ClawSweeper review comment is supported by current-main tests, current-main locking behavior, and the PR head source.

## What This Changes

The PR changes sandbox container and browser registry storage from monolithic JSON files to per-entry JSON files, adds single-entry lookup for Docker ensure, migrates legacy files, and updates related tests.

## Best Possible Solution

Keep the sharded registry direction only if same-runtime mutations retain ordering through a per-entry lock, tombstone, compare-and-swap, or equivalent protocol, and keep focused regression tests for container and browser update/remove races plus strict error propagation on deletion.

## Reproduction Assessment

Yes for the PR regression. Current main has focused tests that gate an update, race a remove, and assert the entry remains deleted; applying that scenario to the PR head's uncoordinated per-entry write/remove path can resurrect stale registry state.

## Solution Assessment

No. Per-entry sharding is a plausible performance approach, but dropping same-name mutation coordination is not the narrowest maintainable fix; per-entry coordination preserves the performance goal without weakening recreate/prune semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve same-entry remove/update ordering:** `src/agents/sandbox/registry.ts:219`
  - body: Current main serializes registry mutations and tests that a remove starting after an in-flight update leaves the entry deleted. Here an update can read the entry and later finish writeJsonAtomic after removeRegistryEntry deletes the file, so recreate or prune can leave stale runtime state for the same container name. Keep per-entry coordination or equivalent tombstone/compare semantics for same-name mutations.
  - confidence: 0.9
- **[P3] Propagate registry removal failures:** `src/agents/sandbox/registry.ts:112-116`
  - body: fs.rm(..., { force: true }) already ignores missing files, but this catch also hides EACCES, ENOTDIR, and other I/O failures. Sandbox recreate/prune can report success after the runtime is removed while the registry file remains, so later list/prune sees stale state. Only ignore expected missing or concurrent-remove cases.
  - confidence: 0.82
