## Summary

Keep open. The branch takes a plausible performance direction, but the proposed patch regresses current registry safety guarantees and introduces an unsafe path derivation for sharded entry files, so it should not be merged as-is.

## What This Changes

The PR changes sandbox container and browser registries from monolithic JSON files to per-entry JSON files, adds single-entry lookup for Docker ensure, migrates legacy registry files, and updates related tests.

## Best Possible Solution

Keep the sharded-registry design only if it preserves per-runtime mutation ordering, uses a safe filename encoding or hash independent of the public runtime ID, propagates unexpected filesystem errors, and restores focused regression tests for same-entry update/remove races plus plugin-runtime ID edge cases.

## Reproduction Assessment

Yes for the PR regressions. Current main has focused registry race tests, and the PR head removes that protection while the new read-then-write per-entry path can recreate a file after removeRegistryEntry deletes it; the unsafe filename path is directly reproducible by a sandbox backend runtime ID or migrated entry containing path separators.

## Solution Assessment

No. Per-entry sharding can be a maintainable performance fix, but this implementation is not the best version because it drops same-entry ordering, hides delete failures, and derives filesystem paths from unconstrained runtime IDs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Encode registry entry filenames:** `src/agents/sandbox/registry.ts:83`
  - body: Current main stores sandbox runtime IDs as JSON values, and the plugin SDK exposes `runtimeId` as an unconstrained string. Using `containerName` directly as a path segment means a backend runtime ID or migrated legacy entry containing `/` or `..` can escape the sharded directory or fail with missing parent dirs; encode or hash the filename and keep the original name inside the JSON entry.
  - confidence: 0.78
- **[P2] Preserve same-entry remove/update ordering:** `src/agents/sandbox/registry.ts:219`
  - body: Current main locks registry mutations and tests that a delayed update cannot resurrect an entry removed after that update started. This write can read the old entry and later recreate the per-entry file after removeRegistryEntry deletes it, leaving stale sandbox or browser state after recreate/prune; add per-entry locking, tombstones, compare-and-swap, or equivalent ordering and restore the race tests.
  - confidence: 0.9
- **[P3] Propagate unexpected registry removal errors:** `src/agents/sandbox/registry.ts:112-116`
  - body: `fs.rm(..., { force: true })` already ignores missing files, but this catch also hides EACCES, ENOTDIR, and other I/O failures. Sandbox recreate/prune can report success while the registry file remains, so only suppress expected missing-file or deliberately handled concurrent-remove cases.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff has one concrete local filesystem security concern: sharded registry paths are derived from an unconstrained runtime/container name.

Concerns:

- **[medium] Unsafe registry file path derivation:** `src/agents/sandbox/registry.ts:83`
  - body: `entryFilePath` joins the registry directory with `containerName` directly, while sandbox backend `runtimeId` is a public plugin SDK string. A plugin backend or migrated legacy entry containing path separators can cause reads/writes/deletes outside the intended shard directory or fail unpredictably; use a safe filename encoding or hash.
  - confidence: 0.78
