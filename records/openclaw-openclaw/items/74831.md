## Summary

Keep open. The sharded registry direction is plausible, but the current PR regresses same-entry update/remove ordering that current main explicitly serializes and tests, and it suppresses unexpected registry delete failures. This should be revised on the PR branch before merge rather than closed or replaced.

## What This Changes

The PR changes sandbox container/browser registries from monolithic JSON files to per-entry JSON files, adds single-entry registry lookup for Docker ensure, migrates legacy registry files, and updates related sandbox tests.

## Best Possible Solution

Keep the per-entry sharding approach only after adding same-name mutation ordering, such as a per-entry lock, tombstone, or compare-and-swap protocol, and after restoring focused race coverage. Delete handling should ignore only expected missing-file cases and propagate other I/O failures.

## Reproduction Assessment

Yes. Current main has focused registry tests that model the delayed update racing a removal and assert the entry stays deleted; the PR head removes that coverage and its read-then-write path can recreate the deleted file.

## Solution Assessment

No. Sharding can address cross-session contention, but this implementation is not the narrowest maintainable fix until it preserves same-entry semantics and reports unexpected deletion errors.

- **[P2] Preserve same-entry remove/update ordering:** `src/agents/sandbox/registry.ts:219`
  - body: This write can read an existing per-entry file, race with removeRegistryEntry, and then write the stale entry after the remove.   - confidence: 0.9
- **[P3] Propagate unexpected registry removal errors:** `src/agents/sandbox/registry.ts:112-116`
  - body: fs.rm(..., { force: true }) already ignores missing paths, but this catch also hides EACCES, ENOTDIR, and other I/O failures. Sandbox recreate/prune can report success while the registry file remains, so only suppress ENOENT or an explicitly handled concurrent missing-file case.
  - confidence: 0.82
