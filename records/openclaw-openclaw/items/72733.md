## Summary

Keep this PR open. Current main does not already contain the proposed reversible guardrail, and the branch is a broad security-sensitive core change with concrete merge blockers in the submitted implementation plus unresolved API/product-boundary questions raised by the related open guardrail reports.

## What This Changes

The PR branch adds an opt-in reversible sensitive-data guardrail that scans user prompts with a local OpenAI-compatible model or regex fallback, replaces values with `[VAULT_N]` tokens before model transit, restores tokens for assistant/tool paths, and adds config, UI metadata, docs, and a vault unit test.

## Best Possible Solution

Define the durable security boundary first, then land either a reviewed generic pre-model guardrail contract or a narrowly hardened core implementation with full-block secret coverage, no raw secret/error logging, an explicit vault lifecycle and storage policy, and focused regression tests across prompt transit, replay/history, tool execution, and final delivery.

## Reproduction Assessment

Yes for the concrete review findings: a PEM private key passed through the regex fallback would only replace the BEGIN header, and the plaintext vault write plus raw response-body logging are directly visible in the PR head. I did not run tests because this review was required to stay read-only.

## Solution Assessment

No. The current PR is not the best solution yet because it embeds security policy and reversible secret persistence directly into the embedded runner without enough at-rest handling, fallback coverage, validation, or core-vs-plugin boundary clarity.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Redact entire private key blocks:** `src/security/patterns.ts:43-45`
  - body: The fallback patterns only match the `-----BEGIN ... PRIVATE KEY-----` header. Since `TokenVault.redact()` replaces exactly the matched text, the private-key body and footer remain in the prompt sent to the cloud model even though the guardrail claims private keys are protected. Match the full PEM/OpenSSH/PGP block or drop this coverage claim until it is safe.
  - confidence: 0.92
- **[P2] Avoid persisting vault secrets in plaintext:** `src/agents/pi-embedded-runner/run.ts:923-926`
  - body: This writes the reversible token map, including raw passwords, API keys, and connection strings, to a `${sessionFile}.vault.json` sidecar and never cleans it up. That creates a new long-lived secret store outside the existing SecretRef/credential handling; use an explicit protected state model with cleanup, or keep the vault in memory when persistence is not accepted.
  - confidence: 0.9
- **[P2] Do not log local model error bodies:** `src/security/scanner.ts:120-122`
  - body: On a non-2xx sanitizer response, this logs the raw response body. Some OpenAI-compatible servers include request details in error responses, so a failed sanitizer call can print the sensitive user prompt to Gateway logs. Log status/provider metadata only, or pass the body through existing redaction before logging.
  - confidence: 0.83
