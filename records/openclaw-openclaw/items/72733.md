## Summary

Keep this PR open. Current main does not already contain the Security Guardrail surface, and the branch is useful source work for several related security-redaction requests, but latest head still has a concrete security regression in regex-only private-key handling plus unresolved secret-config and vault-storage design questions that need maintainer/security review before merge.

## What This Changes

The PR adds an opt-in reversible sensitive-data guardrail that scans user prompts with a local OpenAI-compatible model or regex fallback, replaces detected values with `[VAULT_N]` tokens before model transit, restores tokens for assistant and tool paths, and adds config, UI metadata, docs, and a vault test.

## Best Possible Solution

Keep the useful implementation work open, but require a hardened design before merge: fix multiline fallback scanning, use the existing SecretInput/sensitive config contract for the sanitizer credential, decide the vault storage lifecycle and protection model, add focused tests for regex/local-model redaction plus built-in/client tool restoration, and align the architecture with the related pre-model guardrail interface discussion.

## Reproduction Assessment

Yes. The regex-only private-key leak is reproducible from source inspection: `scanSensitive` splits text by newline and then applies full-block private-key regexes per line, so a normal PEM/OpenSSH/PGP key cannot be matched as claimed.

## Solution Assessment

No. The current patch is not yet the best way to solve the issue because it embeds a security policy and reversible secret store directly in the runner while still missing multiline fallback coverage and existing secret-config semantics; a reviewed guardrail contract plus hardened implementation would be safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Scan the whole prompt for multiline secrets:** `src/security/scanner.ts:97-100`
  - body: The full PEM/OpenSSH/PGP regexes can never match a normal multiline key because `scanSensitive` splits the text on `\n` and runs each pattern against one line. In regex-only mode, private key material will still be sent to the cloud despite the documented coverage; scan the whole prompt for block patterns or handle multiline blocks before per-line matching.
  - confidence: 0.93
- **[P2] Store the local model API key as a SecretInput:** `src/config/zod-schema.ts:1029`
  - body: `localApiKey` is sent as an Authorization bearer token, but the schema only accepts a plain string and does not register the field as sensitive. That prevents the existing SecretRef flow used by other `apiKey` fields and can expose a real sanitizer credential in config/UI/schema surfaces; use the repo's SecretInput pattern and update the aligned types/generated schema.
  - confidence: 0.86
