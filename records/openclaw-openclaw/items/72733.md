## Summary

Keep open. Current main does not implement the proposed guardrail, so this PR is not obsolete, but the branch is a broad security-sensitive core prompt path change with concrete blockers in the regex fallback, reversible vault persistence, and sanitizer error logging. The author’s linked related items show real demand, especially #72741 for a standard guardrail interface, but they also reinforce that this needs maintainer/security design review before merge.

## What This Changes

This PR adds an opt-in reversible sensitive-data guardrail that scans user prompts with a local OpenAI-compatible model or regex fallback, replaces secrets with `[VAULT_N]` tokens before model transit, restores tokens for replies and tool calls, and adds config, UI metadata, docs, tests, and a persisted session vault.

## Best Possible Solution

Keep the PR open for maintainer and security review, but require a hardened design before merge: full-block secret detection or reuse of existing redaction helpers, no raw secret/error logging, an explicit at-rest vault lifecycle and permissions model, focused tests for prompt redaction plus built-in/client tool restoration, and a product decision on whether this belongs in core or behind a generic pre-model guardrail/plugin contract.

## Reproduction Assessment

Yes for the concrete review findings: a PEM private key passed through regex fallback would match only the header, and the plaintext vault write plus raw error-body logging are directly visible in the PR head. I did not run tests or a live agent turn because this review was required to stay read-only.

## Solution Assessment

No: this is not the best mergeable solution yet because it embeds security policy and reversible secret storage directly into the embedded runner without enough at-rest handling, fallback coverage, or plugin/API boundary clarity. A safer path is a reviewed pre-model guardrail contract plus a hardened implementation that can be tested independently.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Redact entire private key blocks:** `src/security/patterns.ts:43-45`
  - body: The regex fallback claims private-key coverage, but these expressions match only the `-----BEGIN ... PRIVATE KEY-----` header. Since `TokenVault.redact()` replaces exactly the matched text, the key body and footer remain in the prompt sent to the cloud model when regex fallback is used. Match the full PEM/OpenSSH/PGP block or remove this coverage claim until it is safe.
  - confidence: 0.92
- **[P2] Avoid persisting vault secrets in plaintext:** `src/agents/pi-embedded-runner/run.ts:925`
  - body: This writes the reversible `[VAULT_N]` map, including raw passwords, API keys, and connection strings, to a `${sessionFile}.vault.json` sidecar without cleanup or integration with existing protected state/SecretRef handling. That creates a new long-lived secret store for the exact values the guardrail is meant to keep contained.
  - confidence: 0.9
- **[P2] Do not log local model error bodies:** `src/security/scanner.ts:121`
  - body: On sanitizer HTTP failure this logs the raw response body. Some OpenAI-compatible servers echo request messages or validation details in error responses, so a failed sanitizer call can print the sensitive user prompt to Gateway logs. Log status/provider metadata only, or pass the body through established redaction first.
  - confidence: 0.83
