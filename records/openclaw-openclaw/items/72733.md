## Summary

Keep open. Current main does not implement the reversible guardrail, and the PR is a broad security-sensitive core change with concrete blocking concerns in the proposed patch. The latest branch appears to address the earlier bot comments about raw extracted-value logging and vault counter initialization, but the regex fallback can still leak private-key bodies, the reversible vault is persisted as plaintext, and failure logging can expose local-model response bodies.

## What This Changes

The PR adds an opt-in reversible sensitive-data guardrail that scans user prompts with a local OpenAI-compatible model or regex fallback, replaces secrets with `[VAULT_N]` tokens before model transit, restores tokens for replies/tool calls, and adds config, UI metadata, docs, and a persisted session vault.

## Best Possible Solution

Keep the PR open for maintainer and security review, but require a hardened design before merge: full-block secret detection, no raw secret/error logging, an explicit at-rest vault lifecycle and permissions model, focused tests for prompt redaction plus built-in/client tool restoration, and a product decision on whether this should be a generic pre-model guardrail API or an external plugin built on a new seam.

## Reproduction Assessment

Yes for the concrete review findings: a full PEM private key passed through `scanSensitive()` would only match the BEGIN header, and the plaintext vault write plus raw error-body logging are directly visible in the PR diff. I did not run tests or a live agent turn because the review was required to stay read-only.

## Solution Assessment

No: the current patch is not the best way to solve the issue yet because it adds security policy and reversible secret storage directly into the embedded runner without enough at-rest handling, fallback coverage, or plugin/API boundary clarity. A safer direction is a reviewed pre-model guardrail contract plus a hardened implementation that can be tested independently.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Redact entire private key blocks:** `src/security/patterns.ts:43-45`
  - body: The fallback patterns only match the `-----BEGIN ... PRIVATE KEY-----` header. Since `TokenVault.redact()` replaces exactly the matched text, the private-key body and footer remain in the prompt sent to the cloud model even though the guardrail claims private keys are protected. Match the full PEM/OpenSSH/PGP block or drop this coverage claim until it is safe.
  - confidence: 0.92
- **[P2] Avoid persisting vault secrets in plaintext:** `src/agents/pi-embedded-runner/run.ts:925`
  - body: This writes the reversible token map, including raw passwords, API keys, and connection strings, to a `${sessionFile}.vault.json` sidecar and never cleans it up. That creates a new long-lived secret store outside the existing SecretRef/credential handling; use an explicit protected state model with cleanup, or keep the vault in memory when persistence is not accepted.
  - confidence: 0.9
- **[P2] Do not log local model error bodies:** `src/security/scanner.ts:121`
  - body: On a non-2xx sanitizer response, this logs the raw response body. Some OpenAI-compatible servers include request details in error responses, so a failed sanitizer call can print the sensitive user prompt to Gateway logs. Log status/provider metadata only, or pass the body through existing redaction before logging.
  - confidence: 0.83
