## Summary

Keep open. Current main does not contain this guardrail surface, and the latest PR head addresses several earlier review findings, but the branch still has source-proven security blockers in the regex fallback and unresolved reversible-secret storage design concerns that need maintainer/security review.

## What This Changes

The PR adds an opt-in reversible sensitive-data guardrail that scans user prompts with a local OpenAI-compatible model or regex fallback, replaces detected secrets with `[VAULT_N]` tokens before model transit, restores tokens for assistant/tool paths, and adds config, UI metadata, docs, and a vault unit test.

## Best Possible Solution

Keep the PR open, but require a hardened security design before merge: regex fallback must actually cover multi-line secrets, reversible vault persistence needs an explicit lifecycle or protected storage model, and maintainers should decide whether this belongs as a generic pre-model guardrail contract rather than embedded-runner-specific policy code.

## Reproduction Assessment

Yes. A source-level reproduction is clear: pass a normal multi-line PEM private key through `scanSensitive`; because the function splits on `\n` before applying the full-block regexes, the regex fallback cannot produce a complete private-key finding. I did not run tests because this review was required to stay read-only.

## Solution Assessment

No. The current PR is not the best solution yet because it adds security policy and durable reversible secret storage directly into the embedded runner while leaving concrete fallback coverage and vault lifecycle gaps; a reviewed pre-model guardrail contract with a hardened implementation would be safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Scan private key blocks before line splitting:** `src/security/patterns.ts:97-109`
  - body: The new private-key regexes match full PEM/OpenSSH/PGP blocks, but `scanSensitive` splits the prompt by newline and then runs every regex on each individual line. A normal multi-line private key therefore yields no finding in regex-only mode, so the key body is sent to the cloud model despite the guardrail claiming private-key coverage. Run multi-line block patterns against the full text before line splitting, or otherwise scan with block-aware logic.
  - confidence: 0.94
- **[P2] Avoid leaving reversible vault files in session storage:** `src/agents/pi-embedded-runner/run.ts:923-929`
  - body: This writes the token-to-secret map as raw JSON beside the session file. `0600` helps with cross-user reads, but the file still survives session end, backups, and support collection outside the existing SecretRef/audit/scrub lifecycle. Keep the vault in memory unless persistence has a reviewed lifecycle, or integrate it with protected storage, cleanup, and audit handling.
  - confidence: 0.82
