## Summary

Keep open. Current main still exposes raw managed dreaming blocks through QMD daily-memory reads, so this PR targets a real memory-isolation gap, but the branch still has merge blockers around canonical path detection, Plugin SDK API baseline drift, and unintended ordinary-content formatting rewrites.

## What This Changes

The branch adds a public `stripDreamingManagedBlocks()` markdown helper, applies it to QMD daily-memory full and partial read paths, and adds helper/read-path tests.

## Best Possible Solution

Land a corrected memory-core read-path filter that classifies daily files from the resolved canonical workspace-relative path, strips only managed Light/REM blocks from both full and paginated reads, preserves ordinary daily-note formatting when no managed block is removed, keeps dreaming internals reading raw files, and updates the Plugin SDK API baseline if the helper remains public.

## Reproduction Assessment

Yes. On current main, a daily file containing `<!-- openclaw:dreaming:light:start -->` is returned raw by `QmdMemoryManager.readFile()` for both full and partial reads; on the PR branch, the remaining bypass is a normalized-equivalent request such as `memory/./2026-04-18.md` resolving to the same file while failing the raw `relPath` regex.

## Solution Assessment

No. The direction is the right boundary, but the implementation should base daily-file detection on the resolved canonical path, avoid rewriting blank lines when no managed block was stripped, and include the generated Plugin SDK baseline update before merge.
