## Summary

Keep open. Current main still does not recognize `prompt_cache_hit_tokens`, and this PR is an open implementation candidate for the linked MiniMax compaction bug rather than obsolete or superseded. The central `normalizeUsage` boundary is the right place for this kind of provider usage alias, but the field source should be confirmed with a raw MiniMax fixture before merge because the official MiniMax docs checked describe cache usage through fields current main already handles.

## What This Changes

The PR updates `src/agents/usage.ts` so `prompt_cache_hit_tokens` is accepted on usage payloads, extracted as `cacheRead`, and treated as an OpenAI-style prompt-total cache field.

## Best Possible Solution

Keep the PR open and either land a tested central usage-normalization fix for the real MiniMax raw payload shape, or adjust the patch if live/raw evidence shows the bug is in a different usage conversion layer.

## Reproduction Assessment

Unclear for end-to-end reproduction from this checkout: the linked issue provides user steps and logs, but no raw MiniMax usage JSON fixture is available here. A high-confidence unit reproduction would be a `normalizeUsage` test using the exact raw MiniMax response fields from the failing session.

## Solution Assessment

Unclear as-is. Extending `normalizeUsage` is the narrow maintainable boundary if `prompt_cache_hit_tokens` is truly emitted with inclusive `prompt_tokens`, but the best merge path should include a source-backed regression fixture for that exact provider payload.
