## Summary

Keep open. Current main only partially addresses archive accounting: broad `usage.cost` scans include reset/deleted archive transcripts, but per-session usage, filters, detail views, exports, and overview cards still derive totals from `sessions.usage`, which resolves each session id to a single transcript and prefers the active `.jsonl` when it exists. The issue discussion also has a concrete active-plus-reset repro, a local workaround, and a closed unmerged source PR (#46605), so this remains a valid narrow fix candidate.

## What This Changes

The issue requests that cost and usage reporting include archived `.jsonl.reset.*` transcript files created by `/new` or `/reset`, not only active `.jsonl` session files.

## Best Possible Solution

Keep the archive-aware global scan, then add a shared per-session transcript-segment resolver that returns every usage-counted segment for a session id: active `.jsonl` plus `.jsonl.reset.*` and `.jsonl.deleted.*`. Use that segment list for per-session summaries, `sessions.usage`, timeseries, logs, exports, and dashboard totals so all usage surfaces agree without counting backup or checkpoint artifacts.

## Reproduction Assessment

Yes. The issue gives a concrete active-plus-reset file pair and costs, and current source statically reproduces the bug because `loadSessionCostSummary({ sessionId })` resolves one active transcript and scans only that file.

## Solution Assessment

Yes, the narrow maintainable path is a shared resolver/aggregation fix in `src/infra/session-cost-usage.ts`, with gateway/UI behavior following from the existing API calls. A UI-only adjustment would leave RPC totals, exports, timeseries, and logs inconsistent.
