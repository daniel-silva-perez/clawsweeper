## Summary

Keep open. Current main still launches the Google Gemini CLI backend without a noninteractive edit/write approval mode, and the regression test now pins that `--skip-trust`-only argv shape. Upstream Gemini CLI policy docs/source still make the reported behavior plausible in headless `--prompt` runs: write tools default to approval, and approval prompts are denied/non-available without an interactive user. The linked fix PR #68366 remains closed unmerged, so there is no implemented-on-main close reason.

## What This Changes

Fix `google-gemini-cli` OAuth-backed agent sessions so noninteractive Gemini CLI runs can write or edit workspace files such as `IDENTITY.md` and `SOUL.md`.

## Best Possible Solution

Make an explicit maintainer decision for Gemini CLI headless approval behavior, then update the Google plugin backend defaults, resume defaults, docs, and regression coverage together, followed by a live OAuth smoke that edits a disposable `IDENTITY.md` while keeping the existing OAuth caution visible.

## Reproduction Assessment

Yes. The current default argv plus upstream Gemini CLI policy docs provide a high-confidence reproduction path for a headless `gemini --prompt` run where write tools require approval and are unavailable/denied without a TTY; a live OAuth smoke would provide final end-to-end proof.

## Solution Assessment

Unclear. PR #68366's `--yolo` direction likely fixes the observable write failure, but it auto-approves all Gemini CLI tools, so the safer maintainable fix may be `auto_edit` or a policy-mapped approval mode chosen by maintainers.
