## Summary

Keep open. Current main still carries `memorySearch.outputDimensionality` through resolved config and provider requests, but the built-in local GGUF provider ignores it and returns the full normalized `node-llama-cpp` vector; the local provider cache key also omits dimensionality, so changing the setting would not reliably trigger cache separation or a full reindex.

## What This Changes

The issue requests `memorySearch.outputDimensionality` support for local GGUF embedding models, with truncation before storage and reindexing when the configured dimension changes; query-instruction support is mentioned as a bonus.

## Best Possible Solution

Implement local GGUF dimensionality truncation in the local embedding provider, include the configured dimension in local provider runtime/cache identity so dimension changes trigger the existing full-reindex path, add focused unit tests for query and batch vectors, and document local support. Treat configurable query instructions as follow-up product/API work unless maintainers explicitly expand the scope.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to instantiate `createLocalEmbeddingProvider` with the existing mocked `node-llama-cpp` test harness, a known vector, and `outputDimensionality: 2`; current code returns the full normalized vector because it never reads that option.

## Solution Assessment

Yes for the main requested fix. The narrow maintainable solution is local-provider truncation plus cache/reindex identity and tests; adding a new `queryInstruction` config in the same change would broaden the public config surface unnecessarily.
