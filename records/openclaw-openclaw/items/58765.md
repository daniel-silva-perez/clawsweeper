## Summary

Keep open. Current main still accepts and forwards `memorySearch.outputDimensionality`, but the bundled local GGUF provider ignores it and returns the full normalized node-llama-cpp vector; the local provider cache key also omits dimensions, so a future dimension change would not trigger the existing full-reindex path. The stale label does not change the result because the issue is recent and still actionable.

## What This Changes

Support truncating local GGUF memory embedding vectors via `memorySearch.outputDimensionality`, with optional follow-up support for configurable query instructions.

## Best Possible Solution

Implement local GGUF truncation in the memory/local embedding path, apply it consistently to query and batch vectors before final normalization, include the selected dimension in local provider cache/reindex identity, and document the local setting. Treat configurable query instructions as a follow-up unless the maintainer wants that bundled into the same memory-search config change.

## Reproduction Assessment

Yes. A unit-level reproduction can mock node-llama-cpp to return a 4-element vector, call `createLocalEmbeddingProvider` with `outputDimensionality: 2`, and observe that current `embedQuery`/`embedBatch` still return all 4 normalized dimensions because the option is never read.

## Solution Assessment

Yes for the truncation portion: the narrowest maintainable fix is provider-local truncation plus a local provider cache-key update, using the existing full-reindex provider-key mechanism. The query-instruction bonus is less settled and should be scoped separately unless product owners explicitly include it.
