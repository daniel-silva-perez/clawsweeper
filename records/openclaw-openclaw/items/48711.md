## Summary

Keep this issue open. Current main has meaningful memory improvements, including memory files, memory_search/memory_get, compaction memory flush, and optional Active Memory, but it still does not close the reported default reliability gap for project/task/focus recall across realistic conversations. Active Memory is opt-in and eligibility-gated, there is no canonical PROJECTS.md/TASKS.md open-task contract in the documented workspace model, and the OpenAI prompt cache path still forwards the raw session id as prompt_cache_key while explicit session ids are accepted without a provider-limit validation step.

## Best Possible Solution

Keep this open and split or implement the remaining work in narrower owner-owned pieces: add a repo-contained memory/task recall gate, define or document a canonical project/task/focus workspace-state pattern if that is a supported product contract, harden default recall ordering toward current canonical state over stale semantic memory, ensure open-task queries exclude completed work by default where OpenClaw owns that task surface, and validate or normalize provider prompt cache/session keys with a clear error instead of letting long ids surface as context_overflow.
