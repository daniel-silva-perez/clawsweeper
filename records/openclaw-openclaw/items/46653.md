## Summary

Keep this PR open. Current main still lacks a managed `openclaw tunnel` command, so the feature is not obsolete, but the proposed branch is stale against the current descriptor-based CLI registration path and has concrete process-control blockers that should be fixed before merge.

## What This Changes

The PR adds a top-level `openclaw tunnel up/down/status` CLI, persistent SSH tunnel PID state, CLI registration, and a new `docs/cli/tunnel.md` reference page.

## Best Possible Solution

Refresh the feature onto current main, register `tunnel` through the descriptor catalog and current lazy registration helpers, reuse or extend the existing SSH tunnel readiness logic, persist enough process identity to stop only OpenClaw-started tunnels, and add focused CLI lifecycle tests plus docs under the current docs navigation structure.

## Reproduction Assessment

Yes. Current-main absence is reproducible by inspecting the CLI descriptor catalog, docs nav, and remote access docs; the PR bugs are reproducible from the code paths for fresh missing state directories, pre-bound local ports, and stale PID reuse.

## Solution Assessment

No. A first-class managed tunnel CLI is a reasonable product direction, but this patch is not the safest implementation until it is rebased onto the current CLI architecture and hardens process ownership/readiness handling.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Register tunnel through the current CLI catalog:** `src/cli/program/register.subclis.ts:168-176`
  - body: Current main no longer keeps the top-level command list in this file; root commands are discovered from `subcli-descriptors.ts` and loaded through `register.subclis-core.ts`. Please rebase and add the descriptor/spec there, otherwise the command will not be wired into the maintained help/completion path.
  - confidence: 0.87
- **[P1] Verify PID ownership before stopping tunnel processes:** `src/cli/tunnel-cli.ts:291`
  - body: `tunnel down` signals the numeric PID from the state file without proving it is still the SSH tunnel started by OpenClaw. If the original tunnel exits and the OS reuses that PID, this can terminate an unrelated same-user process; persist and verify ownership metadata before SIGTERM/SIGKILL.
  - confidence: 0.94
- **[P1] Reject occupied ports before reporting tunnel success:** `src/cli/tunnel-cli.ts:222-224`
  - body: The readiness loop treats any listener on `--port` as proof that the new SSH forward is active. If another process already owns the port, SSH can still be alive briefly before `ExitOnForwardFailure` exits, causing stale state and a false success message.
  - confidence: 0.92
- **[P2] Create the state directory before writing tunnel state:** `src/cli/tunnel-cli.ts:39`
  - body: `writeTunnelState` writes to `STATE_DIR/tunnel.pid.json` without creating the directory. On a fresh profile or new `OPENCLAW_STATE_DIR`, `tunnel up` can start the detached SSH process and then throw `ENOENT`, leaving an unmanaged tunnel.
  - confidence: 0.95
- **[P3] Surface unexpected port-probe errors:** `src/cli/tunnel-cli.ts:62-64`
  - body: `isPortBound` collapses every listen error other than `EADDRINUSE` into `false`, so OS errors look like a free port and later become a generic timeout. Propagate unexpected errors or reuse the existing connect-based readiness helper.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff adds detached SSH process management and currently stops stored PIDs without verifying that they still belong to an OpenClaw-started tunnel.

Concerns:

- **[medium] Unverified PID signaling can stop unrelated processes:** `src/cli/tunnel-cli.ts:291`
  - body: The PID file stores only a numeric PID, and `tunnel down` sends SIGTERM/SIGKILL through `killPid` without checking process ownership, command identity, process start time, or another durable marker. PID reuse can make this terminate an unrelated same-user process.
  - confidence: 0.93
