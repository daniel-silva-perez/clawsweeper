## Summary

Keep this PR open, but it is not merge-ready. Current main still lacks a managed `openclaw tunnel` command, while the PR branch adds a useful feature with concrete lifecycle and security bugs plus a stale CLI registration shape that needs refresh onto current main.

## What This Changes

This PR adds a top-level `openclaw tunnel up/down/status` CLI, detached `/usr/bin/ssh` process management backed by `~/.openclaw/tunnel.pid.json`, and a new CLI docs page/nav entry.

## Best Possible Solution

Refresh the feature onto current main, register it through the descriptor-based CLI catalog, reuse or extend the existing SSH tunnel and port-readiness helpers, persist enough ownership metadata to stop only OpenClaw-started SSH processes, and add focused lifecycle tests plus public CLI docs.

## Reproduction Assessment

Yes. The main-branch absence is reproducible by inspecting the CLI descriptor catalog and docs search, and the PR bugs have code-level reproduction paths: fresh missing `STATE_DIR`, pre-bound local port during startup, and a stale numeric PID with no ownership metadata.

## Solution Assessment

No. A first-class tunnel CLI is a reasonable direction, but this implementation should not merge as-is; the safer path is to build on the maintained SSH helper/readiness contracts and add ownership verification before persistent process control.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Verify PID ownership before stopping tunnel processes:** `src/cli/tunnel-cli.ts:291`
  - body: `tunnel down` signals the numeric PID from the state file without proving it still belongs to the SSH tunnel started by OpenClaw. If the original tunnel exits and the OS reuses that PID, this can terminate an unrelated same-user process; persist and verify ownership metadata before SIGTERM/SIGKILL.
  - confidence: 0.94
- **[P1] Reject occupied ports before reporting tunnel success:** `src/cli/tunnel-cli.ts:222-224`
  - body: The readiness loop treats any listener on `--port` as proof that the new SSH forward is active. If another process already owns the port, SSH can still be alive briefly before `ExitOnForwardFailure` exits, causing stale state and a false success message.
  - confidence: 0.92
- **[P2] Create the state directory before writing tunnel state:** `src/cli/tunnel-cli.ts:38-39`
  - body: `writeTunnelState` writes to `STATE_DIR/tunnel.pid.json` without creating the directory. On a fresh profile or new `OPENCLAW_STATE_DIR`, `tunnel up` can start the detached SSH process and then throw `ENOENT`, leaving an unmanaged tunnel.
  - confidence: 0.95
- **[P3] Surface unexpected port-probe errors:** `src/cli/tunnel-cli.ts:62-64`
  - body: `isPortBound` collapses every listen error other than `EADDRINUSE` into `false`, so OS errors look like a free port and later become a generic timeout. Propagate unexpected errors or reuse the existing connect-based readiness helper.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff adds detached SSH process management and currently stops stored PIDs without verifying they still belong to an OpenClaw-started tunnel.

Concerns:

- **[medium] Unverified PID signaling can stop unrelated processes:** `src/cli/tunnel-cli.ts:291`
  - body: The PID file stores only a numeric PID, and `tunnel down` sends SIGTERM/SIGKILL through `killPid` without checking process ownership, command identity, start time, or another durable marker. PID reuse can make this terminate an unrelated same-user process.
  - confidence: 0.93
