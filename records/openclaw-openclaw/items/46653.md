## Summary

Keep this PR open. Current main still lacks a managed persistent SSH tunnel CLI, so the feature is not obsolete, but the proposed patch is not merge-ready because it targets an older CLI registration shape and has concrete process-management, readiness, and state-file bugs already raised in review discussion and confirmed against the diff.

## What This Changes

This PR adds a top-level `openclaw tunnel up/down/status` CLI, a new tunnel docs page/nav entry, and detached `/usr/bin/ssh` process management backed by `~/.openclaw/tunnel.pid.json` state.

## Best Possible Solution

Refresh the feature onto current main, wire the command through the descriptor-based CLI catalog/registration path, reuse or extend the existing SSH tunnel and port helpers for parsing/readiness/error handling, store enough ownership metadata to manage only OpenClaw-started SSH processes, and add focused lifecycle tests plus the public CLI docs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Verify PID ownership before stopping tunnel processes:** `src/cli/tunnel-cli.ts:265`
  - body: `tunnel down` sends SIGTERM/SIGKILL to the numeric PID from the state file. If the original SSH process exits and the OS reuses that PID, this can terminate an unrelated same-user process, while `status` will also treat the stale PID as active. Store and verify ownership metadata before signaling.
  - confidence: 0.94
- **[P1] Reject occupied ports before reporting success:** `src/cli/tunnel-cli.ts:203-205`
  - body: The startup loop treats any bound local port as proof that the new SSH tunnel is ready. If another process already owns `--port`, SSH can still be briefly alive before `ExitOnForwardFailure` exits, so this writes stale state and reports a tunnel that was never established.
  - confidence: 0.92
- **[P2] Create the state directory before writing tunnel state:** `src/cli/tunnel-cli.ts:34`
  - body: `writeTunnelState` writes to `STATE_DIR/tunnel.pid.json` without ensuring the directory exists. On a fresh profile or new `OPENCLAW_STATE_DIR`, `tunnel up` can start the detached SSH process and then throw `ENOENT`, leaving an unmanaged tunnel.
  - confidence: 0.95
- **[P3] Surface unexpected port-probe errors:** `src/cli/tunnel-cli.ts:55`
  - body: `isPortBound` collapses every non-`EADDRINUSE` listen error into `false`, making OS errors look like a free port and leading to a generic timeout. Propagate unexpected errors or reuse the existing connect-based readiness helper so users see the real failure.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff adds detached SSH process management and currently signals stored PIDs without proving they still belong to the OpenClaw-started tunnel.

Concerns:

- **[medium] Unverified PID signaling can kill unrelated processes:** `src/cli/tunnel-cli.ts:265`
  - body: The PID file stores only a numeric PID, and `tunnel down` signals that PID without verifying it is still the SSH tunnel started by OpenClaw. PID reuse can make this terminate an unrelated same-user process, so persistent state needs ownership verification before SIGTERM/SIGKILL.
  - confidence: 0.93
