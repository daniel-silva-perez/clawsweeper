## Summary

Keep this PR open. Current main still lacks the proposed persistent `openclaw tunnel up/down/status` command and CLI docs, but the branch is not correct as-is: it targets an older CLI registration shape and introduces blocking detached-process/state-management bugs.

## What This Changes

The PR adds a new `openclaw tunnel up/down/status` CLI, a docs page/nav entry, and detached SSH process plus PID-file state management for remote gateway port forwards.

## Best Possible Solution

Refresh the feature onto the current descriptor-based CLI registration, reuse or extend the existing SSH tunnel helper for target parsing/readiness, persist enough process metadata to safely manage only the tunnel process, and add focused tests plus docs for the persistent tunnel lifecycle.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Verify PID ownership before signaling stored PIDs:** `src/cli/tunnel-cli.ts:265`
  - body: `tunnel down` sends SIGTERM/SIGKILL to whatever process currently has the stored numeric PID. If the detached SSH process exited and the OS reused that PID, this can kill an unrelated user process; `status` also treats that reused PID as an active tunnel. Store and verify ownership metadata before signaling.
  - confidence: 0.94
- **[P1] Require the requested port to be free before success:** `src/cli/tunnel-cli.ts:203-205`
  - body: The startup loop treats any bound local port as proof that the new SSH tunnel is ready. If another process already owns the port, the SSH child can still be briefly alive before `ExitOnForwardFailure` exits, so the command reports success and writes a stale PID. Check availability before spawning or race listener readiness against the child exit using the existing helper semantics.
  - confidence: 0.92
- **[P2] Create the state directory before writing tunnel state:** `src/cli/tunnel-cli.ts:34`
  - body: On a fresh profile or new `OPENCLAW_STATE_DIR`, `writeTunnelState` writes to a directory that may not exist. Because the SSH child has already been detached, the write can throw after the tunnel starts, leaving a running tunnel that `tunnel down/status` cannot manage.
  - confidence: 0.95
- **[P3] Propagate unexpected port-probe errors:** `src/cli/tunnel-cli.ts:54-55`
  - body: `isPortBound` converts every non-`EADDRINUSE` listen error into `false`, making OS errors indistinguishable from a free port. For cases such as permission failures or unexpected bind errors, the command should surface the real error instead of timing out or reporting a generic tunnel failure.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff adds a detached SSH/process-signaling surface and has a concrete local process-safety issue before merge.

Concerns:

- **[medium] Unverified PID signaling can kill unrelated processes:** `src/cli/tunnel-cli.ts:265`
  - body: The PID file stores only a numeric PID, and `tunnel down` signals that PID without proving it still belongs to the SSH tunnel started by OpenClaw. PID reuse can make this terminate an unrelated same-user process, so the persistent state needs ownership verification before SIGTERM/SIGKILL.
  - confidence: 0.93
