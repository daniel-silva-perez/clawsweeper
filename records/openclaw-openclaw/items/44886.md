## Summary

Keep open. Current main still exposes Telegram per-DM tool policy config and applies direct config for DM prompts/access, but mounted tool filtering still flows through the group/channel policy resolver and direct session keys do not resolve to a direct tool policy. Related PR #53530 remains open and unmerged, so it does not prove the issue is fixed on main.

## What This Changes

The issue asks OpenClaw to either reliably enforce Telegram `channels.telegram.accounts.<account>.direct.<peer>.tools` as per-DM tool policy or clarify that the accepted config surface is not supported.

## Best Possible Solution

Adapt Telegram's legacy `direct[peer] ?? direct["*"]` surface onto the shared direct-session tool-policy resolver, preserving explicit-over-wildcard precedence, account scoping, sender overrides, DM topic/thread fallback, normal DM runs, and native-command target sessions. Add focused Telegram regression tests and document the behavior if this config remains supported.
