## Summary

Keep open. Current main still accepts Telegram per-DM tool policy in config and applies the same direct peer config to prompt context, but the mounted-tool filtering path still resolves only group/channel policy; the overlapping direct-session policy PR is open and unmerged, so this should remain a maintainer-tracked authorization fix rather than be closed early.

## What This Changes

The issue asks whether Telegram `direct.<peer>.tools` is intended to enforce per-DM-user tool permissions, and if so to fix that enforcement path or clarify the schema/docs.

## Best Possible Solution

Route Telegram `direct[peer] ?? direct["*"]` policy through a shared direct-session mounted-tool resolver with account scoping, explicit-over-wildcard precedence, `toolsBySender`, thread/parent fallback where appropriate, and fail-closed filtering. If Telegram DM tool policy is not intended, remove or deprecate the schema/type surface and document the supported alternative.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: `agent:main:telegram:direct:<peer>` is rejected by the group/channel-only session parser before mounted-tool policy lookup, while the same direct peer config is accepted and consumed for prompt overrides.

## Solution Assessment

Yes for triage. Keeping this open while maintainers review, land, split, or replace #53530 is safer than closing it or starting a parallel automated repair for an authorization-sensitive policy path.
