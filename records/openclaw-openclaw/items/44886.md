## Summary

Keep open. Current main still accepts Telegram DM-level tool policy in schema/types and uses the same direct config for prompt overrides, but mounted tool filtering still derives policy only from group/channel contexts, so Telegram `direct.<peer>.tools` is not reliably enforced. The related open PR #53530 remains the right implementation candidate to review rather than treating this as implemented.

## What This Changes

The issue asks whether Telegram `direct.<peer>.tools` and `toolsBySender` should enforce per-DM tool policy, or whether the accepted config surface should be clarified or removed.

## Best Possible Solution

Route Telegram's `direct[peer] ?? direct["*"]` config through a shared direct-session mounted-tool resolver, preserving account scoping, explicit-over-wildcard precedence, `toolsBySender`, thread/parent fallback, and fail-closed filtering. If maintainers decide DM tool policy is not supported, remove or deprecate the schema/type surface and document the supported alternative.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: a Telegram direct `tools` config is accepted, but `agent:<id>:telegram:direct:<peer>` is rejected by the group/channel-only policy parser before mounted-tool lookup.

## Solution Assessment

Yes, if DM tool policy is intended. The narrow maintainable fix is to adapt the existing direct config surface into the shared tool-policy resolver and cover Telegram direct explicit/wildcard/sender cases, preferably by reviewing or replacing PR #53530.
