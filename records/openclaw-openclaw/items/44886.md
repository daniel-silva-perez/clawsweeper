## Summary

Keep open. Current main still accepts and advertises Telegram direct-level tool policy config and applies direct config to DM prompt/access handling, but the actual tool mounting path still resolves only group/channel policy; direct session keys are rejected before policy lookup. Related PR #53530 is useful context, but it remains open and unmerged, so it does not prove the issue is fixed on main.

## What This Changes

The issue asks whether Telegram `direct.<peer>.tools` and `toolsBySender` should enforce per-DM-user tool policy, and requests either a runtime fix or schema/docs clarification.

## Best Possible Solution

Adapt Telegram's legacy `direct[peer] ?? direct["*"]` surface onto a shared direct-session tool-policy resolver, preserving explicit-over-wildcard precedence, account scoping, sender overrides, DM topic/thread fallback, native-command target sessions, and normal DM runs. Add focused regression tests and document the supported behavior if the config remains accepted.
