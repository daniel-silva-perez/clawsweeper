## Summary

Keep open. Current main still accepts and advertises Telegram direct-level tool policy config, and Telegram DM prompt handling consumes the same direct config, but the mounted-tool path still uses the group/channel policy resolver and rejects direct-session keys before policy lookup. Related PR #53530 is useful implementation context, but in the provided context it is open and unmerged, so it does not prove this is fixed on main.

## What This Changes

The issue asks whether `channels.telegram.accounts.<account>.direct.<peer>.tools` is intended to enforce per-DM tool permissions, and requests either runtime enforcement or schema/docs clarification.

## Best Possible Solution

Adapt Telegram's `direct[peer] ?? direct["*"]` surface onto a shared direct-session tool-policy resolver, preserving account scoping, explicit-over-wildcard precedence, `toolsBySender`, DM topic/thread fallback, native-command target sessions, and normal DM runs. Add focused regression coverage and update docs if the config remains supported.
