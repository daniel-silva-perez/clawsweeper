## Summary

Keep open. Current main still exposes Telegram direct-message tool policy in schema/types and applies the same direct config to prompts, but the mounted-tool policy path remains group/channel-oriented and rejects `direct` session keys before policy lookup. Related PR #53530 is useful context, but it is open and unmerged in the provided GitHub context, so it does not make current main implemented.

## What This Changes

The issue asks whether Telegram `direct.<peer>.tools` and `toolsBySender` should enforce per-DM tool policy, or whether the accepted config/schema/docs should be narrowed to avoid implying support.

## Best Possible Solution

Route Telegram's `direct[peer] ?? direct["*"]` config surface through a shared direct-session mounted-tool resolver, preserving account scoping, explicit-over-wildcard precedence, `toolsBySender`, DM topic/thread fallback, and normal fail-closed mounted-tool filtering. If maintainers decide DM tool policy is not supported, remove or deprecate the schema/type surface and document the supported alternative instead.
