## Summary

Keep open. Current main still exposes Telegram DM-level tool policy in config, but the mounted-tool path only resolves group/channel policy, so a Telegram direct session can miss the accepted `direct.<peer>.tools` enforcement. The related implementation PR is still open and should be handled by maintainers rather than closing this issue early.

## What This Changes

The issue requests confirmation and repair or clarification for Telegram `direct.<peer>.tools` and `toolsBySender` as per-DM mounted-tool policy.

## Best Possible Solution

Route Telegram `direct[peer] ?? direct["*"]` policy through a shared direct-session mounted-tool resolver with account scoping, explicit-over-wildcard precedence, `toolsBySender`, thread or parent fallback where appropriate, and fail-closed filtering. If Telegram DM tool policy is not intended, remove or deprecate the schema/type surface and document the supported alternative.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: `agent:main:telegram:direct:<peer>` is rejected by the group/channel-only session parser before mounted-tool policy lookup, while the same direct config is accepted and used for prompt overrides.

## Solution Assessment

Yes for triage. Keeping this open while maintainers review, land, split, or replace #53530 is safer than closing it or starting a parallel automated repair for an authorization-sensitive policy path.
