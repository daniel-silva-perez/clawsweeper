## Summary

Keep open. Current main still accepts Telegram DM-level `tools` and `toolsBySender` config, and the same resolved direct config is used for DM prompt overrides, but mounted tool filtering still routes through group/channel-only policy resolution. The related open PR #53530 remains the implementation candidate to review rather than treating this as fixed.

## What This Changes

The issue asks whether Telegram `channels.telegram.accounts.<account>.direct.<peer>.tools` is intended to enforce per-DM tool permissions, and requests either runtime enforcement or schema/docs clarification.

## Best Possible Solution

Route Telegram `direct[peer] ?? direct["*"]` policy through a shared direct-session mounted-tool resolver with account scoping, explicit-over-wildcard precedence, `toolsBySender`, thread/parent fallback, and fail-closed filtering. If maintainers decide DM tool policy is not supported, remove or deprecate the schema/type surface and document the supported alternative.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: a Telegram direct session key such as `agent:main:telegram:direct:<peer>` is rejected by the group/channel-only policy parser before mounted-tool lookup, while the same direct config is accepted and used for prompt overrides.

## Solution Assessment

Yes, keeping this open is the right outcome. The narrow maintainable fix is to adapt the existing direct config surface into the shared tool-policy resolver and review or replace PR #53530 rather than create a parallel fix lane.
