## Summary

Keep open. Current main still accepts and advertises Telegram direct-level tool policy config and applies direct config to DM prompt handling, but the mounted-tool path still resolves only group/channel policy; direct-session keys are rejected before policy lookup. Related PR #53530 is useful implementation context, but it is open and unmerged in the provided context, so it does not prove the issue is fixed on main.

## What This Changes

The issue requests confirmation and enforcement, or schema/docs clarification, for Telegram per-DM `direct.<peer>.tools` tool permissions.

## Best Possible Solution

Adapt Telegram's `direct[peer] ?? direct["*"]` surface onto a shared direct-session tool-policy resolver, preserving account scoping, explicit-over-wildcard precedence, `toolsBySender`, DM topic/thread fallback, native-command target sessions, and normal DM runs. Add focused regression coverage and update docs if the config remains supported.
