## Summary

Keep this issue open. Current main still documents and implements context overflow as active-model compaction/retry plus a blocked overflow result after recovery is exhausted; the model fallback path explicitly excludes context-overflow errors instead of selecting a larger fallback model. Related closed issues cover overloaded/server-error fallback, output-token freezes, graceful overflow messaging, or fallback candidate minimum context checks, but not the requested smaller-primary to larger-context fallback routing.

## Best Possible Solution

Keep this issue open until maintainers either explicitly decide and document that context overflow must remain compaction-only, or implement a guarded fallback path that runs only after compaction and oversized-tool-result truncation are exhausted, filters candidates by sufficient context headroom, preserves replay safety, logs the fallback event, and adds regression coverage for smaller-primary to larger-fallback behavior.

## Risks / Open Questions

- Retrying a context-overflow turn on a different model needs replay-safety gating so tool side effects, outbound channel messages, cron adds, and block replies are not duplicated.
- A correct implementation needs trustworthy context-window metadata for the active model and every configured fallback candidate, including provider runtime caps such as contextTokens versus native contextWindow.
- The current docs intentionally describe context overflow as compaction/retry-only, so changing behavior would require docs and tests to avoid surprising users who rely on strict overflow handling.
