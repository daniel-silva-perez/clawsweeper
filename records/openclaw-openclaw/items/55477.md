## Summary

Keep open. Current main does not already implement the requested LiteLLM session metadata export, and the proposed PR is not merge-ready because it writes to `options.metadata` on a path that the LiteLLM/OpenAI-completions transport does not serialize into the request payload, while also introducing an ungated outbound path for stable session/channel identifiers.

## What This Changes

The PR adds a `runEmbeddedAttempt` stream wrapper that writes `session_key`, `message_channel`, and `context_limit` into outbound model-call `options.metadata` when those run parameters are present.

## Best Possible Solution

Add a narrow, tested LiteLLM/OpenAI-completions metadata path that serializes only approved string fields into the actual chat-completions request payload, preserves existing caller/provider metadata, gates the behavior by LiteLLM provider or explicit config, and documents the privacy semantics for exported session identifiers.

## Reproduction Assessment

Yes. A high-confidence static reproduction is available: trace the PR wrapper into the current LiteLLM route (`litellm` -> `openai-completions` -> `buildOpenAICompletionsParams`) and observe that `options.metadata` is never copied into the request payload sent to `chat.completions.create`.

## Solution Assessment

No. The runner-level wrapper is not the best fix because it is both too broad for privacy and too shallow for the LiteLLM payload path; a provider-gated transport metadata or payload hook implementation is safer and more likely to work.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Propagate metadata into the LiteLLM payload:** `src/agents/pi-embedded-runner/run/attempt.ts:2458-2462`
  - body: This wrapper only writes to `options.metadata`, but LiteLLM is registered as `openai-completions` and that transport builds chat-completion params without reading `options.metadata`. As written, LiteLLM callbacks still will not receive these fields.
  - confidence: 0.91
- **[P2] Gate session metadata before sending it to providers:** `src/agents/pi-embedded-runner/run/attempt.ts:2447-2451`
  - body: The wrapper is installed for any provider whenever session fields are present. Since `session_key` can encode platform and channel routing identifiers, this needs a LiteLLM-only or explicit opt-in boundary before arbitrary provider streams can receive it.
  - confidence: 0.84
