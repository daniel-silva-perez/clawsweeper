## Summary

Keep this PR open. Current main does not implement the requested LiteLLM request metadata stamping, and the PR is not superseded by the unrelated provided title-search match. It still needs maintainer review because the current OpenAI-compatible LiteLLM path appears not to consume `options.metadata`, so the PR’s wrapper alone may not be sufficient on main.

## Best Possible Solution

Keep this PR open, but do not merge it as-is. The best path is to update it against current main with a tested metadata seam that reaches the LiteLLM/OpenAI-compatible request body, uses `contextTokenBudget != null`, preserves existing metadata safely, and includes a maintainer security/product decision on exposing session/channel identifiers to configured model proxies.

## Risks / Open Questions

- The feature intentionally sends session/channel metadata to the configured LiteLLM proxy; maintainers should explicitly accept and document that disclosure before landing.
- The PR lacks regression tests for metadata merging, absent-field behavior, and the `contextTokenBudget != null` edge noted in review.
