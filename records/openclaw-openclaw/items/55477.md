## Summary

Keep open. The PR addresses a real observability gap, but the proposed wrapper is not sufficient on current main because the LiteLLM LLM route goes through OpenAI-completions transport code that does not serialize `options.metadata` into the outbound request payload. The diff also needs a privacy boundary before shipping because it can attach stable session/channel identifiers to any provider stream function that honors metadata.

## What This Changes

The PR wraps `runEmbeddedAttempt`'s stream function to add `session_key`, `message_channel`, and `context_limit` into `options.metadata` for outbound model calls.

## Best Possible Solution

Implement this as a tested LiteLLM/OpenAI-completions request-payload path, with an explicit provider/config boundary for session metadata and preservation of existing caller metadata.

Full review comments:

- **[P2] Propagate metadata into the LiteLLM request payload:** `src/agents/pi-embedded-runner/run/attempt.ts:2458-2462`
  - body: This wrapper only writes `options.metadata`, but the current LiteLLM LLM route uses the OpenAI-completions transport, and `buildOpenAICompletionsParams` does not copy `options.metadata` into the request params. LiteLLM callbacks still will not see these fields unless the OpenAI-completions/LiteLLM payload path is updated.

  - body: The wrapper runs for every provider when session fields are present. Since `session_key` can encode channel-specific routing identifiers, please restrict this to LiteLLM or an explicit opt-in before passing it to arbitrary stream functions that may serialize metadata.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff has no dependency or workflow supply-chain changes, but it introduces an ungated outbound metadata path for stable session/channel identifiers.

- **[medium] Ungated session metadata leaves OpenClaw:** `src/agents/pi-embedded-runner/run/attempt.ts:2447`

Any provider/custom stream that serializes metadata would receive that routing identifier without a LiteLLM-only or explicit opt-in boundary.
