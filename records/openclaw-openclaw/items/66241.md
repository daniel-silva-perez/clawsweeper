## Summary

Keep open. Current main still loses Discord component and modal registry entries across gateway restarts, and this PR remains a useful implementation candidate, but the branch is not the best/safest path as-is because it writes a new Discord-local JSON registry under a raw home-cache path instead of the active OpenClaw state directory or the now-available bundled plugin state store.

## What This Changes

The PR adds lazy JSON persistence, validation, pruning, best-effort writes, and registry tests for Discord component/modal entries so interactions can survive gateway restarts.

## Best Possible Solution

Rework this PR so Discord component and modal entries persist through the active OpenClaw state system, preferably via api.runtime.state.openKeyedStore for bundled plugins or, if a local file remains necessary, via resolveStateDir plus the existing lock and atomic JSON helpers. The end state should preserve route overrides and allowedUsers, validate persisted records before load, prune expiry, bound growth, and cover restart, malformed-file, expired-entry, authorization, and routing regressions.

## Reproduction Assessment

Yes. The source-level reproduction is clear: register a Discord component, restart the gateway so the process-global Maps are empty, then click the existing Discord button; current main resolves null and the handler replies that the component/form has expired.

## Solution Assessment

No. The PR addresses the right bug, but the storage path is not the narrowest maintainable solution now that OpenClaw has state-dir based Discord persistence patterns and a bundled plugin keyed store suitable for restart-safe registries.
