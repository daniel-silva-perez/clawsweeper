## Summary

Keep open. Current main still stores Discord component and modal registries only in process-global Maps, so the restart-survival behavior proposed here is not implemented. The related #70626 draft helper may be the right direction for the storage primitive, but it explicitly has no Discord consumer migration, so it does not supersede this PR yet.

## Best Possible Solution

Keep this PR open and either revise it to use the existing state-dir/atomic JSON/locking patterns, or rebase it onto the persistent keyed store helper if #70626 lands. The final Discord migration should preserve per-component allowlists and route override fields, validate persisted records before loading, prune TTL-expired entries, and include focused restart round-trip tests.

## Risks / Open Questions

- The PR is still useful because the bug is real on current main, but accepting it as-is could regress Discord authorization or routing if persisted entries lose `allowedUsers`, `sessionKey`, `agentId`, or `accountId`.
- Disk persistence should follow the existing OpenClaw state-dir, atomic JSON, and locking conventions to avoid state drift and restart-loss edge cases.
- Closing this PR now would lose an active Discord-specific implementation candidate before the generic persistent-store work has a Discord consumer migration.
