## Summary

Keep open. Current main still does not implement this PR's central behavior: Discord component and modal entries are stored only in process-global maps, and a registry miss still returns the user-facing expired-component response. The PR remains a useful implementation candidate, but it needs maintainer review because the proposed disk persistence touches security-sensitive Discord routing and allowlist metadata and has unresolved feedback around state-dir storage, atomic/locked persistence, route overrides, and allowlists.

## Best Possible Solution

Keep this PR open for maintainer follow-up. Rework the Discord registry persistence against the existing state-dir, atomic JSON, and locking patterns, or rebase it onto the generic persistent keyed store if #70626 lands first. The final version should preserve route override and allowlist semantics, validate persisted records before loading, prune expired entries, and add focused restart round-trip, malformed-file, and expired-prune tests.

## Risks / Open Questions

- Current main still loses Discord component and modal registry state on gateway restart, so closing this PR would discard an active implementation candidate before the behavior is fixed.
- The proposed persistence path is security-sensitive: mishandling `allowedUsers`, `sessionKey`, `agentId`, or `accountId` could make Discord component authorization or routing change after a gateway restart.
- The final implementation should follow existing OpenClaw state-dir, JSON validation, file-locking, and atomic-write conventions to avoid state drift and corrupted restart data.
