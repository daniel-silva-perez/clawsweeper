## Summary

Keep open. Current main still treats no-approval host exec as the trusted-operator default unless the operator opts into stricter policy, and the code/docs do not provide the native hard/default dangerous-operation gate requested in the discussion. Existing advisory prompt guardrails, sandboxing, allowlists, and approval modes are useful controls, but they remain opt-in or model-guidance layers rather than a default destructive-command classifier for cases like Windows PowerShell `Remove-Item`.

## Best Possible Solution

Keep this issue open until maintainers either implement a native hard/default dangerous-operation gate in core exec/tool policy or make an explicit product decision that the current opt-in exec approvals, sandboxing, and tool-policy model is the final stance. A concrete path should cover default safe mode, Windows PowerShell destructive-command classification such as `Remove-Item`, approval UX, sandbox interaction, and regression tests/docs.
