## Summary

Keep this issue open. Current main has strong fail-closed mitigations for the reported silent config clobber path: source/runtime projection preserves source-only keys, destructive size-drop writes are rejected, plugin installs add the installed plugin to restrictive `plugins.allow`, and invalid config blocks normal plugin install. However, two requested policy changes remain intentionally unresolved: the root config schema is still strict and rejects unknown top-level keys, and bundled/plugin CLI command surfaces still honor restrictive `plugins.allow`. Those are config-contract and plugin-security decisions, not conservative cleanup-close material.

## Best Possible Solution

Keep this open for maintainer judgment. The best path is to split the remaining decisions: either keep strict root validation and add regression coverage proving unknown top-level config fails closed without writing during plugin install/startup paths, or deliberately change the root config contract to preserve unknown top-level keys with tests, docs, and doctor/migration policy. Separately decide whether bundled plugin CLI commands should continue honoring `plugins.allow` or receive a narrow OpenClaw-owned exemption.
