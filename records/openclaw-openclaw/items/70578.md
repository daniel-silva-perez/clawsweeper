## Summary

Keep this open. Current main mitigates the reported silent config-clobber path with source-preserving writes, destructive-write rejection, and plugin-install allowlist updates, but the remaining root-schema passthrough request and bundled plugin CLI allowlist behavior are still unresolved config-contract and security-policy decisions.

## What This Changes

The issue requests that OpenClaw preserve unknown top-level config fields across config/plugin write round-trips and reconsider how restrictive `plugins.allow` applies to bundled CLI command surfaces.

## Best Possible Solution

Maintainers should choose the intended config contract explicitly: either keep strict root validation with stronger fail-closed regression coverage and diagnostics for config/plugin writes, or preserve unknown top-level keys as opaque data with tests, docs, and migration/doctor policy. The bundled command allowlist question should be decided separately as a narrow plugin-security policy change.

## Reproduction Assessment

Yes. The exact historical byte-drop chronology was not rerun in this read-only review, but current-main source and tests give a high-confidence focused reproduction of the remaining behavior: unknown top-level config is still rejected by the strict root schema, while the prior projection/drop and destructive-write paths are now preserved or rejected before commit.

## Solution Assessment

Unclear. The suggested `.passthrough()` direction would address unknown-key preservation, but it is not obviously the best fix without maintainer agreement on typo/stale-key handling; the bundled command exemption is also a security-policy decision rather than a purely mechanical bug fix.
