## Summary

Keep open. Current main has several adjacent compaction-timeout and replay-safety improvements, but I did not find a complete fix for the core user-visible bug: a successful message-tool send can still be re-executed after an in-attempt compaction retry, especially on direct channel plugins such as Feishu. The issue is also recent and has cross-channel reproduction data, so this is not a stale low-signal report.

## Best Possible Solution

Keep this issue open and fix it in the embedded runner/message-tool boundary: persist a stable per-run/per-tool-call send identity or committed-send cache across compaction retries, and short-circuit duplicate message-tool sends after retry for both direct plugin delivery and gateway delivery. Add a regression test that simulates a message tool send followed by compaction_end(willRetry: true) and verifies Feishu/Telegram-style direct sends are not repeated; separately keep the existing compaction-timeout lock/grace behavior covered.

## Evidence

- **Issue discussion shows cross-channel current impact:** The provided discussion includes the original Telegram duplicate-send/compaction-timeout report plus a 2026-04-03 Feishu reproduction where compaction succeeded and retry replayed a message-tool media send. That makes the remaining problem broader than Telegram delivery recovery.
- **Compaction retry still resets in-memory send tracking:** On compaction_end with willRetry, current main calls resetForCompactionRetry(), which records a deterministic side effect flag but clears messagingToolSentTexts, messagingToolSentTargets, and messagingToolSentMediaUrls. That preserves terminal replay metadata, but it does not itself prevent the retried model/tool loop from sending again.
