## Summary

Keep this issue open. Current main still lets the Control UI Sessions tab consume broad Gateway session discovery, including disk-discovered agent stores, and the browser restore path still accepts explicit unknown-agent session keys. Related cleanup and deleted-agent targeting guards mitigate parts of the fresh-delete path, but they do not close the UI display/restore boundary reported here.

## Best Possible Solution

Keep the issue open. Add a Control UI-specific configured-agent boundary: either filter the Sessions tab against agents.list or add an additive sessions.list option for configured-agent UI rows while preserving broad Gateway/ACP discovery. Also normalize or reject restored agent:<id>:* keys when <id> is no longer configured, falling back to the Gateway main session key, with regression tests for the Sessions tab and URL/browser-storage restore path.
