## Summary

Keep open. Current main still lets the Control UI consume broad Gateway session discovery and render disk-discovered agent stores, while URL/storage handling still preserves explicit non-alias agent-scoped session keys. Recent deletion-time purge and resolve guards mitigate targeting deleted-agent sessions, but they do not resolve the listing and rehydration behavior reported here.

## What This Changes

The issue asks for the Control UI Sessions tab and restored chat session selection to ignore sessions belonging to agents no longer registered in config, matching the CLI configured-agent view.

## Best Possible Solution

Keep broad Gateway/ACP discovery available for recovery and non-UI consumers, but make the default Control UI Sessions tab show configured-agent sessions only, or add an explicit configured-only sessions.list mode for the UI. Validate restored agent-scoped URL/storage selections against configured agents and fall back to the current main session key, with regression coverage for listing and restore paths.

## Reproduction Assessment

Yes. Current-main source and tests give a high-confidence reproduction path: seed a disk-only agents/<id>/sessions/sessions.json store while config lists only main, then sessions.list uses the combined Gateway store and the Control UI renders those returned rows.

## Solution Assessment

Unclear for automation. Removing disk discovery from sessions.list, as the closed linked PR proposed, is not clearly the best fix because Gateway/ACP discovery is documented and tested as broader; a safer direction is an additive configured-only UI/API mode plus URL/storage fallback validation.
