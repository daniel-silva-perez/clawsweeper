## Summary

Keep open. Current main still lets the Control UI consume broad Gateway session discovery and render disk-discovered agent stores, while URL/storage handling still preserves explicit agent-scoped session keys for unconfigured agents. The merged deletion/session-resolution guards mitigate sending to deleted-agent sessions, but they do not resolve the listing or browser-state rehydration behavior reported here.

## What This Changes

The issue requests that the Control UI stop listing or restoring sessions for agents no longer registered in OpenClaw config, matching the `openclaw sessions` CLI behavior after agent deletion.

## Best Possible Solution

Keep broad Gateway/ACP disk discovery available for recovery and non-UI clients, but make the default Control UI Sessions tab use configured-agent sessions only or request an explicit configured-only mode. Validate restored `agent:<id>:*` URL/storage selections against configured agents and fall back to the current main session key, with regression coverage for both listing and restore paths.

## Reproduction Assessment

Yes. Source and tests give a high-confidence reproduction path: leave a disk-only `agents/<id>/sessions/sessions.json` store while config lists only another agent, then `sessions.list` combines that store and the Control UI renders the returned row. The URL restore path is also characterized by tests that preserve explicit `agent:<id>:main` session parameters.

## Solution Assessment

Unclear for the closed proposed PR. Removing disk discovery from `sessions.list` conflicts with documented and tested Gateway/ACP behavior, so the safer maintainable fix is an additive configured-only UI/API mode plus restored-session validation.
