## Summary

Keep open. Current main still matches the reported failure mode: Control UI loads the broad Gateway `sessions.list` result and renders those rows, while the Gateway combined store intentionally includes disk-discovered agent session stores. The later deleted-agent guards reduce send/resolve hazards, but they do not hide disk-only agent sessions from the Sessions tab or validate restored `agent:<id>:*` URL/storage selections against configured agents.

## What This Changes

The issue asks for the Control UI Sessions tab to stop displaying or restoring sessions from agent namespaces that are no longer registered in configuration.

## Best Possible Solution

Keep broad Gateway/ACP disk discovery available for recovery and non-UI clients, but add a clear configured-agent boundary for the default Control UI Sessions tab, either through a new `sessions.list` mode or UI-side filtering backed by configured agent data. Also validate restored `agent:<id>:*` URL/storage selections against configured agents and fall back to the current main session key when the agent is unknown.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to leave a disk-only `agents/<id>/sessions/sessions.json` store while config lists only another agent; current Gateway listing loads the combined discovered store and the Control UI renders those returned rows. The restore variant is also reproducible from code paths that apply URL and persisted scoped session keys literally.

## Solution Assessment

Unclear. The closed #41687 approach of removing disk discovery globally is not the best proven fix because current docs and tests intentionally depend on broad Gateway/ACP discovery; a scoped configured-agent UI/API boundary plus restore validation is the narrower maintainable direction.
