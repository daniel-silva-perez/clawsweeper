## Summary

Keep open. Current main still lets the Control UI consume broad Gateway `sessions.list` results from a combined store that includes disk-discovered agent stores, and browser URL/storage paths still preserve explicit agent-scoped session keys. The merged deletion/session-resolution guards mitigate sending to deleted-agent sessions, but they do not resolve the listing or rehydration behavior described in the report and follow-up discussion.

## What This Changes

The issue asks for Control UI Sessions to hide sessions belonging to deleted or unregistered agents and avoid restoring stale agent-scoped session keys, keeping UI behavior consistent with `openclaw sessions`.

## Best Possible Solution

Keep broad Gateway/ACP disk discovery available for recovery and non-UI clients, but make the default Control UI Sessions tab use configured-agent sessions only or request an explicit configured-only mode. Validate restored `agent:<id>:*` URL/storage selections against configured agents and fall back to the current main session key, with regression coverage for both listing and restore paths.

## Reproduction Assessment

Yes. A high-confidence source/test reproduction is to leave a disk-only `agents/<id>/sessions/sessions.json` store while config lists only another agent; `loadCombinedSessionStoreForGateway` includes that store, `sessions.list` returns it, and the Control UI renders the returned row. The URL/browser restore variant is also characterized by current code/tests that preserve explicit `agent:<id>:main` selections.

## Solution Assessment

Unclear for the original closed PR. Removing disk discovery from the broad Gateway list conflicts with current docs and tests, so the safer maintainable solution is an additive configured-only UI/API boundary plus restored-session validation.
