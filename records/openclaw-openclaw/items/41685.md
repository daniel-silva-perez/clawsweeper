## Summary

Keep this issue open. Current main has mitigations for freshly deleted agents and runtime targeting, but the Control UI Sessions tab still consumes broad Gateway session discovery and the browser restore path still preserves explicit unknown-agent session keys instead of validating them against configured agents.

## Best Possible Solution

Keep the issue open. Add a Control UI-specific configured-agent boundary: either filter the Sessions tab against agents.list or add an additive sessions.list option intended for UI rows while preserving broad Gateway/ACP discovery. Also normalize or reject restored agent:<id>:* session keys when that agent is no longer configured, falling back to the Gateway main session default, with regression tests for both the Sessions tab and URL/browser-storage restore path.
