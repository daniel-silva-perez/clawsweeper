## Summary

Keep this open. Current main still exposes broad Gateway session discovery to the Control UI Sessions tab, and the URL/browser-storage restore path still accepts explicit unknown-agent session keys. Related cleanup for deleted agent directories helps one path, but it does not close the Control UI configured-agent boundary reported here.

## Best Possible Solution

Keep the issue open. Add a Control UI-specific configured-agent boundary: either filter the Sessions tab rows against agents.list or add an additive sessions.list UI/configuredOnly option while preserving broad Gateway/ACP disk discovery. Also normalize or reject restored agent:<id>:* keys when <id> is no longer configured, falling back to the Gateway main session key, with regression coverage for both the Sessions tab and URL/browser-storage restore path.

## Risks / Open Questions

- Filtering sessions.list globally could regress intended Gateway/ACP discovery of disk-only or retired stores; current tests explicitly protect that broader behavior.
- Leaving the UI restore path unchanged means a stale URL or browser storage value like agent:<old-id>:main can still select an unconfigured agent namespace before the UI falls back or validates it.
- A partial directory-cleanup fix can reduce fresh delete repros but does not solve historical stores or browser state already persisted on user machines.
