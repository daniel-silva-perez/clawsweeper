## Summary

Keep this issue open. Current main has added useful adjacent safeguards for deleted-agent targeting, but the Control UI Sessions tab still asks the broad Gateway `sessions.list` surface, and that surface still merges disk-discovered agent session stores. The follow-up concern from the issue discussion also remains: explicit unknown-agent session keys from the URL/browser state are preserved rather than normalized back to the current default session.

## Best Possible Solution

Keep this issue open. Add an explicit split between broad Gateway/ACP session discovery and the Control UI Sessions tab: either add a `sessions.list` option/default suitable for configured-agent-only UI rows or filter in the UI using `agents.list`, while preserving the broad API for ACP/disk-only callers. Also normalize or reject restored `agent:<id>:*` URL/browser session keys when `<id>` is no longer configured, falling back to `sessionDefaults.mainSessionKey`, with regression tests for both the Sessions tab and restore path.
