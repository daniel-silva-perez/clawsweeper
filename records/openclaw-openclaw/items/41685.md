## Summary

Keep this issue open. Current main has adjacent deleted-agent cleanup and runtime guards, but the Control UI Sessions tab still renders the broad Gateway `sessions.list` result, and that Gateway path intentionally includes disk-discovered agent stores. The restore path also still preserves explicit `agent:<id>:*` URL/browser-storage selections except for narrow token-without-session and main-alias normalization cases.

## Best Possible Solution

Keep this issue open. Add a Control UI-specific configured-agent boundary: filter the Sessions tab against `agents.list`, or add an additive `sessions.list` option intended for UI rows while preserving broad Gateway/ACP discovery. Also normalize or reject restored `agent:<id>:*` session keys when `<id>` is not configured, falling back to `sessionDefaults.mainSessionKey`, with regression tests for both the Sessions tab and restore path.
