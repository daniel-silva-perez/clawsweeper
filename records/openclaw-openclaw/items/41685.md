## Summary

Keep open. Current main still exposes disk-discovered agent session stores through the Gateway `sessions.list` path that the Control UI consumes, and the UI still preserves explicit agent-scoped session selections from URL/browser state. The deletion/session-resolution guards merged later mitigate sending to deleted-agent sessions, but they do not settle the listing or restore behavior described in the issue and follow-up comments.

## What This Changes

The issue asks for the Control UI to stop displaying or restoring sessions for deleted/unregistered agents and to align default session visibility with configured agents/CLI behavior.

## Best Possible Solution

Keep broad Gateway/ACP disk discovery available for recovery and non-UI clients, but add a clear configured-agent boundary for the default Control UI Sessions tab and validate restored `agent:<id>:*` URL/storage selections against configured agents, falling back to the current main session key when the agent is unknown.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to leave a disk-only `agents/<id>/sessions/sessions.json` store while config lists only another agent; current `sessions.list` loads the combined discovered store and the Control UI renders returned rows. The explicit restore variant is also reproducible from code/tests that preserve URL and stored `agent:<id>:main` selections.

## Solution Assessment

Unclear. The original closed PR direction of removing broad disk discovery globally is not the safest solution because current docs and tests intentionally rely on that behavior; an additive configured-only UI/API boundary plus restored-session validation is the narrower maintainable path.
