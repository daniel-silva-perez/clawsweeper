## Summary

Keep this issue open. Current main has adjacent deleted-agent cleanup and guards, but the Control UI Sessions tab still consumes the broad Gateway `sessions.list` result, and that Gateway path still intentionally includes disk-discovered agent session stores. The broader restore path from the issue discussion also remains: explicit URL/browser session keys for unconfigured agents are still preserved unless they are only aliases of the current default session.

## Best Possible Solution

Keep this issue open. Add a Control UI-specific configured-agent boundary: either filter the Sessions tab against `agents.list` or add an additive `sessions.list` option/default intended for UI rows, while preserving broad Gateway/ACP disk-only discovery. Also reject or normalize restored `agent:<id>:*` URL/browser-storage session keys when `<id>` is no longer configured, falling back to `sessionDefaults.mainSessionKey`, with regression coverage for both the Sessions tab and restore path.
