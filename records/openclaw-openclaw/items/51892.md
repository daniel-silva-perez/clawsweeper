## Summary

Keep this issue open. Current main partially addressed the visible-tag symptom by stripping inline reply directives and forwarding sanitized reply metadata as `reply_to`, but I could not verify the native iMessage threading request as implemented: OpenClaw does not send `reply_to_guid`, and the public imsg RPC source/docs I checked do not show `reply_to` or `reply_to_guid` as supported `send` parameters.

## Best Possible Solution

Keep this issue open and resolve it by verifying the actual imsg native-reply send contract first, then updating `extensions/imessage/src/send.ts` to send the supported field (`reply_to`, `reply_to_guid`, or another documented name) with a focused fake-client regression test that asserts the RPC params and stripped delivered text. If public imsg still lacks native reply send support, the best path is to coordinate or pin the required imsg release/API before changing OpenClaw docs and closing this.

- If OpenClaw intentionally targets a private or newer imsg build where `reply_to` is a supported native-threading send field, that contract is not visible in the checked public imsg source/docs and should be documented or pinned before closing this issue.
