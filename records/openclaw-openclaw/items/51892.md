## Summary

Keep this issue open. Current main still does not implement the requested native iMessage reply threading contract: OpenClaw forwards sanitized reply metadata as `reply_to`, not `reply_to_guid`, and the public imsg v0.5.0/main send handler does not parse either reply field for outbound sends. The merged #39512 work and current code appear to fix the visible leaked `[[reply_to:...]]` tag symptom, but they do not prove native iMessage threaded replies. The closed #51966 PR proposed the requested `reply_to_guid` field but was not merged, and the linked Keep My Claw comment is useful debugging context rather than a code fix.

## Best Possible Solution

Keep this issue open and resolve it by first verifying the actual imsg native-reply send contract. If imsg supports native threading, update `extensions/imessage/src/send.ts` to send the documented field and add a fake-client regression test asserting both the RPC params and stripped delivered text. If public imsg does not support outbound native replies yet, coordinate or pin the required imsg release/API before changing OpenClaw docs or closing this.
