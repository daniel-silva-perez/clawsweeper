## Summary

Keep open for maintainer review. This is an active contributor PR paired with open issue #74141 and overlapping open PR #72306; current main already resolves stored requester sessions for `sessionKey="current"`, while the linked live log shows the constructed requester key can also be missing from the store, which this branch still appears not to handle.

## What This Changes

The PR adds a second `sessionKey="current"` fallback in `session-status-tool.ts` for channel-style requesters plus a regression test for a stored channel-plugin requester session.

## Best Possible Solution

Review #74141, #72306, and this PR together, then land one canonical fix that preserves literal `current` key/sessionId precedence and existing visibility checks while deciding explicitly whether `session_status` should synthesize a requester-scoped status entry when the session store row is absent.

## Reproduction Assessment

Yes for a focused unit-level reproduction of the resolver class: a channel-style requester with an empty session store and `sessionKey="current"` follows the current-main path to an unknown-session error. The live channel-plugin replay itself was not rerun in this read-only review.

## Solution Assessment

No. The branch’s source fallback still depends on `resolveSessionEntry` finding the requester key in the store, which current main already handles for stored requester sessions and which does not cover the second missing-requester-key failure shown in #74141.
