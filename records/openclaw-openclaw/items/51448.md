## Summary

Keep this PR open. The reported forwarding gap still exists on current main, but the submitted patch is not merge-ready because it uses a stream option type that does not include `toolChoice`, resolves aliases after precedence has already been flattened, and is stale against the current extra-params test layout.

## What This Changes

This PR forwards configured `tool_choice`/`toolChoice` agent model params into embedded runner stream options and adds a focused regression capture for that path.

## Best Possible Solution

Keep the narrow user-visible fix, but port it to current main as a typed extra-param bridge: resolve `tool_choice`/`toolChoice` with the same ordered-source precedence used by other aliases, validate or warn on invalid values without dropping supported provider object forms, forward through a stream option type that explicitly includes provider-owned `toolChoice`, and place regression coverage in the current extra-params test files.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure `agents.defaults.models["openai/gpt-5.4"].params.tool_choice = "required"` and follow `applyExtraParamsToAgent()` into `createStreamFnWithExtraParams()`; current main never adds `toolChoice` to the underlying stream options even though the OpenAI transport consumes it when present.

## Solution Assessment

No, this PR is not the best way to solve the issue as submitted. The intent is correct, but the implementation should preserve alias/source precedence, use a valid option type for provider-specific `toolChoice`, and move the regression into the current test layout before merge.
