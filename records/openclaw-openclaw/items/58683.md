## Summary

Keep open: current main still lacks heartbeat time-of-day scheduling, and the branch is useful source work for the linked feature requests, but the proposed runner logic still misses fallback-gap transitions and needs a rebase onto the current heartbeat phase scheduler before merge.

## What This Changes

The PR adds `heartbeat.schedule[]` config, schema/help/docs, schedule resolution helpers, runner scheduling changes, and tests for time-of-day heartbeat intervals.

## Best Possible Solution

Land a rebased core heartbeat implementation that keeps the current phase-based scheduler, adds `heartbeat.schedule[]` as a validated config/docs surface, recalculates from the effective interval on every tick and boundary including fallback gaps, and ships targeted scheduler/schema/docs coverage plus a changelog entry.

## Reproduction Assessment

Yes for the blocking PR bug: configure a long scheduled window that ends into a gap with a shorter fallback `every`; at the boundary `resolveScheduleIntervalMs(...)` returns `null`, so the PR's tick-time recalculation leaves `nextDueMs` at the old long interval. Current-main reproduction of the feature itself is not applicable because `heartbeat.schedule` is absent.

## Solution Assessment

No: the feature direction is appropriate for core heartbeat behavior, but this patch is not yet the best implementation because fallback-gap transitions are wrong and the branch needs adaptation to the current scheduler files. The safer fix is to compute `effectiveInterval = resolveScheduleIntervalMs(...) ?? agent.intervalMs` wherever due time is recalculated.
