## Summary

Keep this PR open. Current main does not implement heartbeat time-of-day schedules, and the feature matches the linked requests, but the proposed runner change still mishandles transitions from a scheduled window into an unscheduled gap because it only recalculates `nextDueMs` when a schedule entry currently matches.

## What This Changes

The PR branch adds a `heartbeat.schedule[]` config contract, validation/help/schema/docs, schedule resolution helpers, runner boundary wakeups, and tests for time-window matching plus one scheduled-interval runner case.

## Best Possible Solution

Land a rebased or replacement implementation that keeps the `heartbeat.schedule[]` MVP, preserves current stable phase scheduling, and recalculates the effective interval as `matching schedule interval ?? fallback every` at boundary wakes, with focused tests for scheduled-to-fallback, fallback-to-scheduled, shorter-to-longer, and longer-to-shorter transitions.

## Reproduction Assessment

Yes. A high-confidence reproduction path is a fake-timer runner test with fallback `every: "30m"`, a `23:00-08:00` schedule entry at `2h`, and a last run shortly before `08:00`; at the boundary the PR leaves the old `2h` due time because `resolveScheduleIntervalMs` returns `null`.

## Solution Assessment

No. The config shape and helper direction are reasonable, but the runner fix is incomplete; the narrow maintainable solution is to compute the effective interval from the schedule match or fallback `agent.intervalMs` whenever schedule boundaries are used.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Recompute fallback cadence at schedule gaps:** `src/infra/heartbeat-runner.ts:1152`
  - body: When a boundary wake lands outside all schedule windows, `resolveScheduleIntervalMs(...)` returns `null`, so this guard skips recalculating `nextDueMs` and the agent keeps the previous window's due time. For example, a `2h` overnight window ending at `08:00` with fallback `every: "30m"` can wait until the old `09:50` due time instead of using the fallback cadence. Recompute from the effective interval (`scheduleMs ?? agent.intervalMs`) when `lastRunMs` exists.
  - confidence: 0.9
