## Summary

Keep open. Current main still lacks the acpx orphan-process cleanup, and this active PR is the right general owner boundary, but the proposed patch still has a blocking startup ownership bug that would leave generated wrapper orphans behind for custom acpx state directories.

## What This Changes

The PR adds an acpx process reaper, wires pid-based cleanup into cancel/close and startup, hardens generated Codex/Claude ACP wrappers, and updates tests, docs, and the changelog.

## Best Possible Solution

Keep the cleanup in the acpx plugin: use acpx's recorded pid and agentCommand for cancel/close cleanup, reap startup orphans only from OpenClaw's generated wrapper root and staged plugin-runtime-deps commands, and keep unsupported process-listing platforms quiet or explicitly skipped before normal CI/review.

## Reproduction Assessment

Yes. Current main reproduces the cleanup gap by source inspection: cancel/close do not use pid metadata, startup has no stale reaper, and docs still prescribe manual stale process cleanup. The PR-specific bug is reproducible with a custom acpx config stateDir because wrappers are generated under ctx.stateDir/acpx while startup filters with pluginConfig.stateDir.

## Solution Assessment

No, not as-is. The acpx plugin is the right owner boundary and the acpx 0.6.1 record contract supports pid-based cleanup, but the branch should match the generated wrapper directory for startup reaping and avoid noisy unsupported ps scans before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Match the generated wrapper dir during startup reaping:** `extensions/acpx/src/service.ts:222`
  - body: Generated wrappers are written under `ctx.stateDir/acpx`, but startup reaping filters wrapper ownership with `pluginConfig.stateDir`. When users configure the acpx plugin `stateDir`, the scan no longer recognizes OpenClaw's generated wrapper orphans, so they survive gateway restart. Pass the generated wrapper root, or return it from the auth-bridge preparation step, instead of using the session store state directory for this filter.
  - confidence: 0.89
- **[P3] Skip unsupported process scans quietly:** `extensions/acpx/src/service.ts:221-222`
  - body: Service startup now calls the stale reaper unconditionally, while the default lister shells out to `ps -axo`. On Windows or minimal hosts without `ps`, this logs a warning on every acpx service start even though PPID=1 orphan reaping is Unix-specific. Guard the scan by platform/availability or make unsupported listing a quiet no-op.
  - confidence: 0.82
