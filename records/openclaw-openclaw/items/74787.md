## Summary

Keep open. This is an active implementation PR for a real acpx cleanup gap on current main, but the latest head still has a correctness bug: startup reaping filters by the configurable acpx session state directory while generated wrappers live under the plugin service state directory. It also still makes unsupported Unix process scans noisy on hosts without `ps`.

## What This Changes

The PR adds acpx process reaping on startup, cancel, and close, hardens generated Codex/Claude ACP wrappers to signal child process groups, and updates acpx tests, docs, and the changelog.

## Best Possible Solution

Keep the cleanup inside the acpx plugin boundary, pass the generated wrapper root derived from ctx.stateDir into startup reaping, make unsupported process enumeration a quiet no-op, and keep the tests, docs, and changelog aligned with that behavior.

## Reproduction Assessment

Yes. Current main reproduces the original cleanup gap by source inspection, and the PR-specific startup bug reproduces with a custom acpx stateDir because wrappers are generated under ctx.stateDir/acpx while startup reaping filters by pluginConfig.stateDir.

## Solution Assessment

No, not as-is. The PR is directionally correct and uses the right owner boundary, but the startup ownership root and unsupported-host behavior need correction before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use the generated wrapper root for startup reaping:** `extensions/acpx/src/service.ts:222`
  - body: The wrappers are written under `ctx.stateDir/acpx`, but this filters startup orphan reaping by `pluginConfig.stateDir`. With a custom acpx session-store `stateDir`, stale generated wrappers no longer match the ownership path and remain orphaned. Pass `ctx.stateDir` or the generated wrapper root to the reaper instead.
  - confidence: 0.9
- **[P3] Skip unsupported startup process scans quietly:** `extensions/acpx/src/service.ts:220-232`
  - body: Startup calls the stale-process reaper unconditionally, and the default lister shells out to `ps -axo`. On Windows or minimal hosts without `ps`, every acpx service start logs a warning even though PPID=1 orphan scanning is Unix-specific. Guard this path or make unsupported listing return an empty result without warning.
  - confidence: 0.82
