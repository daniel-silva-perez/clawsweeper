## Summary

Keep open. The PR is aimed at a real acpx orphan-process gap on current main, but the latest head still has a source-level startup reaper mismatch that skips generated wrappers when the configured acpx state directory differs from the plugin service state directory, plus smaller best-effort cleanup gaps.

## What This Changes

The PR adds acpx-owned stale process reaping for generated Codex/Claude ACP wrappers and plugin-runtime-deps process trees, cleans recorded harness trees on cancel/close, hardens generated wrapper signal handling, and updates tests, docs, and the changelog.

## Best Possible Solution

Keep the cleanup inside the acpx plugin boundary, pass the generated wrapper root or ctx.stateDir into startup reaping, add a direct recorded-root kill fallback when process listing is unavailable, quiet unsupported startup scans, and cover the split stateDir/list-failure cases with focused tests before normal review and CI.

## Reproduction Assessment

Yes. A source-level reproduction is clear: configure acpx with a custom plugin stateDir, start the service so wrappers are generated under ctx.stateDir/acpx, then leave a stale PPID=1 generated wrapper there; the PR head filters startup reaping against pluginConfig.stateDir/acpx and skips it.

## Solution Assessment

No, not as-is. The owner boundary and cleanup direction are maintainable, but the current diff should use the actual generated wrapper state root and preserve a recorded-root fallback before it reliably solves the orphan cleanup problem.

  - body: `resolveCleanupContextFromRecord` turns process-list failures into an empty list, and this line passes that empty snapshot to cleanup. The cleanup can verify `rootCommand`, but `collectProcessTree([])` returns no root, so cancel/close sends no signal despite having an owned recorded PID.
  - confidence: 0.84
- **[P3] Skip unsupported startup process scans quietly:** `extensions/acpx/src/service.ts:221-222`
  - body: Startup always invokes the Unix `ps` scanner through the reaper and logs a warning on failure. On Windows or hosts without that scan, every acpx startup can emit an avoidable best-effort cleanup warning; guard unsupported platforms or make unsupported listing return an empty result.
  - confidence: 0.79
