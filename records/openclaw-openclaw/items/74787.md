## Summary

Keep open. Current main still lacks acpx process-tree cleanup, but this active PR should not merge as-is because startup reaping filters generated wrappers with the configurable acpx session-store directory instead of the service wrapper directory.

## What This Changes

The branch adds acpx stale-process reaping, pid-record cleanup on cancel/close, generated Codex/Claude ACP wrapper signal handling, tests, docs, and a changelog entry.

## Best Possible Solution

Keep the cleanup in the acpx plugin boundary, pass the generated wrapper root/service state directory into startup ownership checks, make unsupported process enumeration a quiet no-op, and rerun focused acpx tests plus the changed gate before maintainer review.

## Reproduction Assessment

Yes. Source inspection of current main shows no startup reaper and no pid-based cleanup on cancel/close, while the PR-specific regression follows from using pluginConfig.stateDir even though wrappers are generated under ctx.stateDir/acpx.

## Solution Assessment

No, not as-is. The implementation direction is appropriate, but startup reaping must use the generated wrapper state root and avoid warning on hosts where the Unix ps scan is unsupported.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use the generated wrapper state dir for startup reaping:** `extensions/acpx/src/service.ts:222`
  - body: The wrapper is generated under `ctx.stateDir/acpx`, but startup reaping filters wrapper ownership with `pluginConfig.stateDir`. With a custom acpx session-store `stateDir`, stale OpenClaw-generated wrappers no longer match and remain orphaned. Pass the service state dir or generated wrapper root here instead.
  - confidence: 0.9
- **[P3] Skip unsupported startup process scans quietly:** `extensions/acpx/src/service.ts:221-231`
  - body: Startup calls the stale-process reaper unconditionally, and the default lister shells out to `ps -axo`. On Windows or minimal hosts without `ps`, every acpx service start logs a warning even though PPID=1 orphan scanning is Unix-specific. Guard this path or return an empty process list without warning when unsupported.
  - confidence: 0.82
