## Summary

Keep open. The PR addresses a real acpx orphan-process problem and the live v2026.4.29 evidence supports the direction, but latest head still misses generated wrappers when acpx uses a custom plugin state directory and still has best-effort cleanup gaps on process-list failures or unsupported startup scans.

## What This Changes

The PR adds an acpx process reaper, wires startup/cancel/close cleanup for OpenClaw-owned Codex ACP process trees, hardens generated Codex/Claude ACP wrappers, and updates tests, docs, and the changelog.

## Best Possible Solution

Keep the fix inside the acpx plugin, but pass the generated wrapper root used by prepareAcpxCodexAuthConfig into startup reaping, preserve a recorded-root kill fallback when process enumeration fails, and make unsupported startup scans quiet best-effort behavior before merge.

## Reproduction Assessment

Yes. A source-level repro is clear: configure acpx with a custom plugin stateDir, leave a stale generated wrapper under ctx.stateDir/acpx, and latest head filters against pluginConfig.stateDir so the startup reaper skips it; process-list failure during cancel/close is another direct source-level path.

## Solution Assessment

No, not as-is. The proposed acpx-local architecture is the right narrow place for this behavior, but the wrapper-root mismatch and missing recorded-root fallback need repair before this is the best maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Use the generated wrapper root for startup reaping:** `extensions/acpx/src/service.ts:221-222`
  - body: Startup generates the Codex/Claude ACP wrappers under `ctx.stateDir/acpx`, but this call filters stale-wrapper ownership with `pluginConfig.stateDir`. With a custom acpx `stateDir`, stale generated wrappers under the service state directory are skipped; pass `ctx.stateDir` or an explicit generated wrapper root instead.
  - confidence: 0.94
- **[P3] Kill the recorded root when process listing fails:** `extensions/acpx/src/runtime.ts:531`
  - body: When process listing throws, `context.processes` is empty and this passes that empty list into cleanup. The cleanup helper can validate the recorded command, but it collects an empty tree and sends no signal to the recorded PID, so cancel/close cleanup is lost on hosts where listing is unavailable or incomplete.
  - confidence: 0.88
- **[P3] Avoid warning on unsupported startup scans:** `extensions/acpx/src/service.ts:221-231`
  - body: Startup now always invokes the Unix `ps` scanner and logs any failure as a warning. On Windows or hosts without that scanner, every acpx startup can warn for a best-effort cleanup path; gate unsupported platforms or make unsupported listing return an empty result.
  - confidence: 0.8
