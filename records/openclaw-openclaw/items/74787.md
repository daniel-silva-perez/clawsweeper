## Summary

Keep open. Current main still lacks pid/process-tree cleanup for acpx cancel/close and still documents manual stale process cleanup, while this active contributor PR implements the right acpx-owned direction but still needs maintainer review, green CI, and a small portability decision.

## What This Changes

The PR adds an acpx process reaper, wires startup plus cancel/close cleanup into the bundled acpx runtime, hardens generated Codex/Claude ACP wrappers, and updates tests, docs, and the changelog.

## Best Possible Solution

Land a scoped acpx-plugin fix after maintainer review and green validation, keeping ownership checks limited to OpenClaw-generated wrappers and staged plugin-runtime-deps commands, and making unsupported process-listing platforms quiet or explicitly skipped.

## Reproduction Assessment

Yes. Current main reproduces the gap by inspection: acpx `cancel`/`close` do not inspect stored pid metadata, and the docs still prescribe manual stale acpx process cleanup for stalled completed sessions.

## Solution Assessment

Mostly yes. The acpx plugin is the right owner boundary and the pinned acpx 0.6.1 record contract provides the stored `agentCommand` and `pid` fields the PR uses; the safer adjustment is to guard or quiet the Unix startup reaper on unsupported hosts.

## Review Findings

Overall correctness: patch is correct

Overall confidence: 0.84

Full review comments:

- **[P3] Skip the Unix reaper on unsupported hosts:** `extensions/acpx/src/service.ts:220-223`
  - body: The startup path now calls the stale-process reaper unconditionally. That helper shells out to `ps -axo ...`, so Windows or other hosts without `ps` will hit this catch and log a warning on every acpx service start even though PPID=1 orphan reaping is Unix-specific. Guard this path or make the helper a quiet no-op when process listing is unsupported.
  - confidence: 0.82
