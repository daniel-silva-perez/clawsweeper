## Summary

Keep open: the cleanup direction fits the acpx plugin boundary, but the latest head still has a blocking startup-root mismatch and a narrower fallback gap already identified in review.

## What This Changes

The PR adds an acpx process reaper, wires cleanup on startup/cancel/close, hardens generated Codex/Claude ACP wrappers, and updates tests, docs, and the changelog.

## Best Possible Solution

Keep the cleanup inside the acpx plugin, use the generated wrapper root for startup reaping, preserve direct recorded-root cleanup when process listing is unavailable, and make unsupported startup scans a quiet best-effort path with regression coverage.

## Reproduction Assessment

Yes. Configure acpx with a custom plugin stateDir while service wrappers are generated under ctx.stateDir/acpx, then leave a stale PPID=1 generated wrapper there; latest head filters against pluginConfig.stateDir/acpx and misses it. A second source-level path is process-list failure during cancel/close, which leaves an empty snapshot and signals no recorded PID.

## Solution Assessment

No, not as-is. The proposed architecture is the right narrow place for this behavior, but the implementation needs the wrapper-root and recorded-root fallback fixes before it reliably solves orphan cleanup.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Use the generated wrapper root for startup reaping:** `extensions/acpx/src/service.ts:222`
  - body: Startup generates Codex/Claude ACP wrappers under `ctx.stateDir/acpx`, but this filters stale-wrapper ownership with `pluginConfig.stateDir`. With a custom acpx `stateDir`, stale generated wrappers under the service state directory are skipped; pass `ctx.stateDir` or an explicit generated wrapper root instead.
  - confidence: 0.94
- **[P3] Keep a recorded-root kill fallback when ps fails:** `extensions/acpx/src/runtime.ts:510-512`
  - body: When process listing throws, this stores an empty snapshot. `cleanupOpenClawOwnedAcpxProcessTree` can validate the recorded `rootCommand`, but it walks an empty tree and sends no signal to the recorded PID, so cancel/close cleanup is lost on hosts where listing is unavailable.
  - confidence: 0.86
- **[P3] Avoid warning on unsupported startup scans:** `extensions/acpx/src/service.ts:221-222`
  - body: Startup now always invokes the Unix `ps` scanner and logs failures as warnings. On Windows or hosts without that scanner, every acpx startup can warn for a best-effort cleanup path; gate unsupported platforms or make unsupported listing return an empty result.
  - confidence: 0.8
