## Summary

Keep open. Current main still lacks acpx-owned process-tree cleanup, and this open PR is a scoped implementation for that gap; the diff fits the acpx plugin boundary, with one non-blocking portability finding around unconditional Unix `ps` startup reaping.

## What This Changes

The PR adds acpx process reaping for OpenClaw-generated Codex ACP wrappers and plugin-runtime-deps processes, wires cleanup into startup and cancel/close, hardens generated wrappers, and updates tests, docs, and the changelog.

## Best Possible Solution

Land a scoped acpx-plugin fix after maintainer review and final CI, keeping ownership checks limited to OpenClaw-generated wrappers and staged plugin-runtime-deps commands, and making unsupported process-listing platforms quiet or explicitly skipped.

## Reproduction Assessment

Yes. Current main reproduces the gap by inspection: acpx `cancel`/`close` do not load stored pid metadata or clean process trees, startup has no stale orphan reaper, and the docs still prescribe manual stale acpx process cleanup. I did not run live process tests because this review is read-only.

## Solution Assessment

Mostly yes. The acpx plugin is the right owner boundary, and the pinned acpx 0.6.1 `SessionRecord` contract exposes the `agentCommand` and optional `pid` fields the PR uses; the safer adjustment is to guard or quiet the Unix startup reaper on unsupported hosts.

## Review Findings

Overall correctness: patch is correct

Overall confidence: 0.84

Full review comments:

- **[P3] Skip the Unix reaper on unsupported hosts:** `extensions/acpx/src/service.ts:219-221`
  - body: The startup path now calls the stale-process reaper unconditionally, and that helper defaults to `ps -axo ...`. On Windows or minimal hosts without `ps`, this catch logs a warning on every acpx service start even though PPID=1 orphan reaping is Unix-specific. Guard this path or make unsupported process listing a quiet no-op.
  - confidence: 0.82
