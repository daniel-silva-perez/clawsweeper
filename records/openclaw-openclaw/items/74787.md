## Summary

Keep open. Current main still has the acpx stale-process gap documented as manual cleanup and runtime cancel/close still delegate without recorded process-tree cleanup, while this active contributor PR is a plausible scoped implementation that needs maintainer review and CI rather than cleanup closure.

## What This Changes

The PR adds an acpx process reaper, wires startup plus cancel/close cleanup into the bundled acpx runtime, hardens generated Codex/Claude ACP wrappers, and updates tests, docs, and the changelog.

## Best Possible Solution

Land a scoped acpx-plugin fix after maintainer review and green validation, keeping ownership checks limited to OpenClaw-generated wrappers and staged plugin-runtime-deps commands, with cross-platform startup behavior kept quiet and the docs/changelog retained.

## Reproduction Assessment

Yes. Current main reproduces the gap by inspection: the docs still prescribe manual stale acpx process cleanup, and runtime `cancel`/`close` do not inspect the stored acpx pid or process tree.

## Solution Assessment

Mostly yes. The acpx plugin is the right owner boundary, and the pinned acpx contract provides the stored `agentCommand`/`pid` data the patch uses; the main safer adjustment is making the startup reaper no-op or debug-only on hosts where Unix `ps` is unavailable.

## Review Findings

Overall correctness: patch is correct

Overall confidence: 0.84

Full review comments:

- **[P3] Skip the Unix reaper on non-Unix hosts:** `extensions/acpx/src/service.ts:220-223`
  - body: The startup path now calls the stale-process reaper unconditionally. That helper shells out to `ps -axo ...`, so Windows or other hosts without `ps` will hit this catch and log a warning on every acpx service start even though PPID=1 orphan reaping is Unix-specific. Guard this path or make the helper a quiet no-op when process listing is unsupported.
  - confidence: 0.82
