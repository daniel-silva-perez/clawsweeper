## Summary

Keep open. The PR addresses a real acpx process cleanup gap, but the latest branch still has a startup ownership-root bug that would leave generated wrappers unreaped when the acpx session-store stateDir is customized; there are also low-priority best-effort cleanup issues on hosts without Unix `ps`.

## What This Changes

The PR adds an acpx-owned process reaper, generated Codex/Claude ACP wrapper signal handling, runtime cancel/close cleanup, startup orphan reaping, focused tests, ACP docs, and a changelog entry.

## Best Possible Solution

Keep the cleanup inside the acpx plugin boundary, but derive startup reaping from the generated wrapper root used by `prepareAcpxCodexAuthConfig`, make unsupported process enumeration a quiet no-op, and cover the custom acpx stateDir case with focused tests before merge.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence repro: configure a custom acpx plugin `stateDir`, then stale generated wrappers under `ctx.stateDir/acpx` do not match the PR head's startup ownership filter using `pluginConfig.stateDir`.

## Solution Assessment

No, not as-is. The owner boundary and cleanup direction are right, but the startup ownership root and unsupported-host process-listing behavior should be corrected before this is the narrowest maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Use the generated wrapper root for startup reaping:** `extensions/acpx/src/service.ts:222`
  - body: The reaper filters wrapper commands under the `stateDir` passed here, but the service generates Codex/Claude ACP wrappers under `ctx.stateDir/acpx` before this call. If acpx `stateDir` is customized, stale generated wrappers live outside `pluginConfig.stateDir/acpx` and will not be reaped on startup; pass `ctx.stateDir` or a dedicated wrapper root instead.
  - confidence: 0.92
- **[P3] Kill the recorded root when listing fails:** `extensions/acpx/src/runtime.ts:531`
  - body: When process listing throws, `resolveCleanupContextFromRecord` returns an empty process list, and this call passes that empty list as authoritative. `cleanupOpenClawOwnedAcpxProcessTree` then validates the owned root command but collects no root process, so cancel/close sends no signal despite having the recorded pid; include a synthetic root or add a direct-root fallback.
  - confidence: 0.82
- **[P3] Skip unsupported startup process scans quietly:** `extensions/acpx/src/service.ts:220-232`
  - body: Startup invokes the default reaper even on hosts where `ps -axo` is unavailable, then logs a warning for the failed best-effort PPID=1 scan. Guard this Unix-specific startup path or make unsupported process listing return an empty result without warning.
  - confidence: 0.8
