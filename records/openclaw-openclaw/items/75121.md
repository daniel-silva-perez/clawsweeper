## Summary

Keep open. Current main still has the Telegram forwarded-burst attribution gap, and this active PR appears to address it with a narrow typed-metadata change plus regression coverage; I did not find a blocking correctness or security issue in the diff.

## What This Changes

The PR updates Telegram forwarded-message handling so multi-message forwarded bursts carry per-item untrusted structured attribution while single and text-fragment forwards keep clean agent bodies and typed forwarded fields, with regression tests.

## Best Possible Solution

Land the typed metadata approach after normal maintainer review: keep single-forward attribution on ForwardedFrom* fields, represent multi-message forwarded bursts as per-item UntrustedStructuredContext entries, preserve group visibility filtering, and keep the text-fragment regression coverage.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to send two forwarded Telegram updates close enough to enter the 80 ms forward debounce lane; current main joins the body into one synthetic message without per-item forwarded attribution.

## Solution Assessment

Yes. The PR appears to be the narrow maintainable fix because it reuses normalizeForwardedContext, carries batch attribution through the existing untrusted JSON prompt path, and scopes forward-metadata stripping to the multi-message batch case.
