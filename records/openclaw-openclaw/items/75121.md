## Summary

Keep open. The latest draft head fixes the earlier forwarded-batch visibility leak by filtering batch metadata through the existing Telegram supplemental-context policy, but the new `body_line` accounting is still wrong for multiline forwarded items, so the branch needs revision before merge.

## What This Changes

The PR updates Telegram inbound handling to strip inherited forward metadata from synthetic burst messages, attach per-item forwarded structured context with visibility filtering, and add regression tests for single, self, batched, mixed-media, and prompt-rendered forwarded messages.

## Best Possible Solution

Finish this PR on its current branch by keeping the structured forwarded-context design, fixing line-number accounting for multiline and blank-only forwarded text, and landing only after the Telegram targeted tests plus the changed gate pass.

## Reproduction Assessment

Yes. A static reproduction is a forwarded burst where the first item text is `line one\nline two` and the second item is `next`; the joined `BodyForAgent` places the second item on line 3, but the PR records `body_line: 2`.

## Solution Assessment

No, not yet. The typed structured-context direction and visibility filtering are the right narrow approach, but the current line calculation must count actual emitted body lines and skip blank-only text before it is merge-ready.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Count forwarded text lines when assigning body_line:** `extensions/telegram/src/bot-handlers.runtime.ts:318`
  - body: The batch metadata advances `nextBodyLine` only once per text-bearing forwarded item. If an item contains embedded newlines, it contributes multiple lines to the joined `BodyForAgent`, so every following `body_line` points at the wrong prompt line. Count the actual emitted lines, and avoid assigning a line for blank-only text.
  - confidence: 0.87
