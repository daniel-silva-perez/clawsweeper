## Summary

Keep open. Current main still exposes cron as Gateway-owned stored jobs with optional agent binding, while the per-agent config type, strict runtime schema, scheduler startup path, and user docs do not define or load an `agents.list[].cron` contract. The request is coherent, but it needs maintainer product/API approval before implementation because it changes scheduler source-of-truth, precedence, reload, and autonomous execution semantics.

## What This Changes

Add an `agents.list[].cron` configuration field for agent-owned scheduled tasks that are loaded by the Gateway scheduler with namespacing, inheritance, backwards compatibility, and stored-job precedence.

## Best Possible Solution

If maintainers accept the feature, add an additive core config and scheduler contract: typed per-agent cron config, strict schema/generated schema/help updates, deterministic compilation into scheduler jobs on startup and reload, documented ID namespacing and stored-job precedence, and tests for validation, merge behavior, duplicates, and agent-setting inheritance.

## Reproduction Assessment

Not applicable as a feature request. Source and docs inspection provides a high-confidence current-main check: `agents.list[].cron` is not defined by the strict agent config schema and is not loaded by the Gateway scheduler.

## Solution Assessment

Unclear pending maintainer product review. The proposed direction is coherent, but the maintainable solution belongs in core config/scheduler semantics rather than a plugin-only path because precedence, reload, and scheduler merge behavior are central contracts.
