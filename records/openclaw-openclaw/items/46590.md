## Summary

Keep open. Current main still exposes cron as Gateway-owned stored jobs with optional `agentId` binding, while per-agent configuration has no typed, strict, documented, or startup-loaded `cron` field. This remains coherent core product/API work rather than a closeable stale or duplicate item.

## What This Changes

The issue requests adding a declarative `cron` array under each `agents.list[]` entry so agent-owned scheduled tasks can be loaded into the Gateway scheduler with inherited agent settings and defined precedence over stored jobs.

## Best Possible Solution

If maintainers accept the direction, add an additive core scheduler/config contract: typed `agents.list[].cron` entries, strict runtime and generated schema/help support, deterministic compilation into scheduler jobs on startup and reload, documented namespacing and stored-job precedence, and focused tests for validation, duplicate handling, inheritance, and delivery behavior.

## Reproduction Assessment

Not applicable as a feature request. Source and docs inspection gives a high-confidence current-main check: `agents.list[].cron` is not defined by the strict agent config schema and is not loaded by the Gateway scheduler.

## Solution Assessment

Unclear pending maintainer product review. The requested direction is coherent, but the narrowest maintainable solution needs core config and scheduler semantics rather than a small automated repair because precedence, reload, and autonomous execution safety are central contracts.
