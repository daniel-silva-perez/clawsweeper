## Summary

Keep open. Current main still treats cron as a Gateway-owned scheduler backed by a top-level store, while per-agent config remains a strict typed contract with heartbeat/tools/runtime-style overrides but no `cron` field or startup compiler for `agents.list[].cron`. This is coherent core config/API work and needs maintainer product/security review rather than stale cleanup or automated closure.

## What This Changes

Add declarative `cron` entries to `agents.list[]` so scheduled tasks can be owned by individual agent configuration and loaded into the Gateway scheduler.

## Best Possible Solution

If maintainers accept the direction, add an additive core contract for agent-owned schedules: typed `agents.list[].cron` entries, strict runtime and generated schema/help support, deterministic compilation into scheduler jobs on startup and config reload, documented namespacing and `jobs.json` precedence, and focused tests for validation, duplicate handling, inheritance, delivery, and migration/doctor behavior.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high-confidence: inspect `AgentConfig`, `AgentEntrySchema`, Gateway cron startup, cron store types, and public cron/config docs to see that `agents.list[].cron` is not defined or loaded.

## Solution Assessment

Unclear pending maintainer product review. The requested direction is coherent, but the best maintainable solution is a deliberate core config and scheduler contract, not a small automated patch, because declarative scheduled execution changes authority, validation, reload, and precedence semantics.
