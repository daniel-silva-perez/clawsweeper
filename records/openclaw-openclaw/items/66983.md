## Summary

Keep this issue open. Current main still lacks an explicit web/browser node platform and still keeps browser-origin or device-less node-role sessions outside the existing Control UI auth bypasses, so the requested browser-tab A2UI canvas node support is not implemented and #68497 only covers a narrower action-return bridge.

## What This Changes

The issue requests first-class browser-tab A2UI canvas nodes with explicit web platform command policy, secure node-role auth/pairing behind reverse proxies, and documentation for the supported web canvas-node pattern.

## Best Possible Solution

Define an explicit web canvas-node contract in the Gateway: a web platform policy with narrow canvas/A2UI commands, a secure browser-node enrollment/auth model for reverse-proxied deployments, regression coverage for Origin and node-role auth behavior, and public docs for the supported browser canvas-node setup.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: role: "node" with shared auth but no device identity evaluates to reject-device-required, and browser-origin node auto-approval is explicitly rejected in the current tests.

## Solution Assessment

No, not as a blanket extension of dangerouslyDisableDeviceAuth or shared-token trust. The maintainable path is a deliberate web platform plus scoped browser-node pairing/bootstrap design reviewed against the gateway auth model.
