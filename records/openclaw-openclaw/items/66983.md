## Summary

Keep open. Current main still does not provide first-class web canvas node support: `web` is not a recognized node platform, Linux still lacks canvas defaults, and the only explicit device-auth bypass remains scoped to Control UI operator sessions. The related open browser action-bridge issue overlaps with web canvas usability, but it does not cover the broader node registration, platform allowlist, and node pairing/auth work requested here.

## Best Possible Solution

Keep this open as the canonical design/implementation request for web browser tabs acting as A2UI canvas node surfaces. A good fix would add an explicit `web` node platform policy, define the allowed canvas command set, and add a carefully scoped node pairing/auth path for trusted reverse-proxied browser nodes, with tests covering browser-origin rejection, proxy trust, token/device-token semantics, and command allowlist behavior. Coordinate with #68497 for the separate web action bridge.
