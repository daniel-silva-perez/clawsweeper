## Summary

Keep this PR open, but do not merge it as-is. The reported shared-schema mismatch is real for common web-search providers, but this patch fixes it by lowering the single agent-facing schema below the documented/runtime limits for provider-specific paths such as Tavily and Exa.

## What This Changes

The PR changes the generic `web_search` count schema maximum from `20` to `MAX_SEARCH_COUNT` and adds a regression test asserting the generic tool schema matches that shared constant.

## Best Possible Solution

Make the agent-facing `web_search` count contract provider-aware through lightweight provider metadata, or use another generic schema strategy that does not over-promise common providers while still preserving Tavily and Exa limits. Add regression coverage for the common 10-result providers and at least one higher-limit provider.

## Reproduction Assessment

Yes. Static source inspection is enough: current main shows the generic schema/runtime mismatch, and the PR patch can be checked against Tavily and Exa provider schemas, runtimes, and docs without live API calls.

## Solution Assessment

No. Replacing the generic maximum with `MAX_SEARCH_COUNT` is too broad because it fixes common providers by under-advertising configured providers whose current contracts intentionally allow more than 10 results.
