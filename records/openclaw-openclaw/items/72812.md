## Summary

Keep open. Current main still uses the shared Codex app-server client for spawned runs and clears that singleton on startup errors, so the PR targets a real failure mode from #72574. The branch should not merge as-is because the new isolated-client path can still leak helper app-server clients on startup timeout races and on `turn/start` failures.

## What This Changes

The PR changes Codex app-server runs to use an isolated client when `spawnedBy` is present, limits shared-client teardown to non-isolated startup failures, closes isolated clients on some cleanup paths, and adds regression tests, with unrelated formatting-only edits in several other files.

## Best Possible Solution

Finish the Codex app-server isolated-client lifecycle so `spawnedBy` runs never clear the shared singleton and every failure after isolated client creation, including timeout races and `turn/start` errors, deterministically closes the isolated client. Keep non-spawned shared-client behavior unchanged and add focused regression tests for those cleanup paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Close isolated clients when startup races time out:** `extensions/codex/src/app-server/run-attempt.ts:465`
  - body: When `withCodexStartupTimeout` rejects after `createIsolatedCodexAppServerClient` has already returned, this catch only aborts the helper signal. The startup operation can continue through `ensureCodexComputerUse` or `startOrResumeThread` and succeed without any later owner, leaving the isolated app-server process running after the spawned run has failed. Track the created isolated client and close it when the timeout or abort path wins.
  - confidence: 0.88
- **[P2] Close isolated clients on turn/start failures:** `extensions/codex/src/app-server/run-attempt.ts:1033-1035`
  - body: The added `client.close()` runs only in the final completion cleanup, but the `turn/start` failure catch above throws before that cleanup is installed. A spawned run that reaches thread-ready and then gets an auth/error response from `turn/start` will still leak its isolated app-server client. Close isolated clients in that failure path too.
  - confidence: 0.9
