## Summary

Keep open. Current main still exposes provider `length` signals in several transports, but the main interactive runner does not implement the requested Phase 1 recovery: OpenAI WebSocket response conversion still derives `stopReason` from content shape, the WebSocket stream collapses non-toolUse completions to `stop`, incomplete-turn fallback does not treat `length` as a truncation recovery case, and the live stream wrappers sanitize/repair tool calls without refusing suspect length-terminated tool execution. The issue discussion also includes a recent confirmed main-session tool-parameter truncation reproduction, while open PR #66146 is only a max-token forwarding mitigation and does not add the runtime guard requested here.

## What This Changes

Request to detect provider output truncation in main interactive agent sessions, prevent suspect truncated tool calls from executing, and prompt the model to recover or continue.

## Best Possible Solution

Add a focused main-session truncation guard that carries or observes provider `length` through assistant-message conversion, detects length-terminated turns with tool-call activity before dispatch, blocks malformed or suspect truncated calls, and appends concise model-facing recovery or continuation instructions. Leave retry budgets and circuit breakers for a separate design pass.
