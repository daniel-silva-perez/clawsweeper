## Summary

Keep open. Current main still exposes provider length/max-token signals and has adjacent main-runner repair/fallback code, but it does not implement the requested Phase 1 main-session recovery: preserving or consuming `length` through all conversion paths, refusing suspect length-terminated tool execution before dispatch, and injecting a model-facing recovery or continuation prompt. The linked open PR #66146 only forwards model max-token limits for OpenAI-compatible transports and is partial mitigation, not the runtime guard requested here.

## What This Changes

Request detection and recovery for `stopReason: "length"` in main interactive agent sessions, especially before executing potentially truncated tool calls.

## Best Possible Solution

Add a focused Phase 1 runtime recovery path for main sessions: carry or observe provider `length` stop reasons through assistant-message conversion, detect length-terminated turns with tool-call activity before dispatch, avoid executing suspect truncated calls, and inject a concise model-facing recovery or continuation message. Leave retry-budget and circuit-breaker policy for a later design issue.
