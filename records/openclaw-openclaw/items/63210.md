## Summary

Keep open. Current main still exposes provider length/max-token signals in transport paths, but the main interactive embedded runner does not consume that signal before tool dispatch, block suspect truncated tool calls, or inject a model-facing recovery prompt. The confirmed issue discussion and related reports show real main-session truncation cases, while open PR #66146 only reduces one provider-side cause by forwarding max token limits.

## What This Changes

The issue requests detection and model-facing recovery for provider output truncation in main interactive agent sessions, especially before executing potentially truncated tool calls.

## Best Possible Solution

Implement a focused Phase 1 main-session recovery path: preserve or consume provider `length` stop reasons through the stream/result path, detect length-terminated turns with tool-call activity before dispatch, refuse suspect truncated tool execution, and inject a concise model-facing recovery or continuation message. Keep the broader retry-budget and circuit-breaker configuration as a separate follow-up unless maintainers explicitly expand scope.
