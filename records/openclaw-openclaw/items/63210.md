## Summary

Keep open. Current main still has the same product gap: provider `length` signals exist in several transports, but the main interactive runner does not consistently preserve/consume them before tool dispatch or inject a model-facing recovery turn for length-truncated tool calls.

## What This Changes

The issue requests main interactive agent-session detection and recovery when provider output stops with `stopReason: "length"`, especially to avoid executing truncated tool-call arguments.

## Best Possible Solution

Add a focused Phase 1 main-session recovery path: preserve provider `length` through message conversion, detect length-terminated turns with tool-call activity before dispatch, refuse suspect truncated tool execution, and inject a concise continuation/retry instruction for the model. Keep retry budgets and circuit breakers as a later design follow-up.

## Reproduction Assessment

Yes. The issue has a confirmed production example, and a deterministic repro can synthesize an OpenAI WebSocket `response.completed` with `status: "incomplete"` plus function-call output, or an Anthropic/OpenAI transport message with `stopReason: "length"`, then assert the main runner does not execute the suspect tool call and adds recovery feedback.

## Solution Assessment

Yes for a narrow Phase 1 fix. The best first repair is runtime-level detection and model-facing recovery before tool dispatch; forwarding larger `maxTokens` values, as in #66146, reduces some provider defaults but does not solve truncation that still occurs.
