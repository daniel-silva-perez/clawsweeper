## Summary

Keep open. Current main still leaves configured placeholder paths literal, so this PR addresses a real logging gap, but the latest patch still has timezone-dependent regression tests and leaves daemon status using the unresolved path; it needs author or maintainer follow-up rather than closure or a replacement job.

## What This Changes

The PR resolves `YYYY-MM-DD` placeholders in configured logging file paths, updates rolling log-path resolution, and adds logger settings and log-tail tests.

## Best Possible Solution

Land a focused version that uses one shared host-local date-placeholder resolver for logger settings, file writes, tailing, and status output, with tests that either use timezone-safe local instants or derive expected filenames from the same formatter contract.

## Reproduction Assessment

Yes. Current main is reproducible by source inspection: configured `logging.file` is returned unchanged. The PR test failures are reproducible with timezone probes showing the fixed UTC instants resolve to the previous local date in western timezones.

## Solution Assessment

No, not as written. The implementation direction is narrow and appropriate, but the resolver should be shared with status output and the tests should avoid hard-coded UTC instants that imply a different host-local date.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use a local-safe date in the log-tail test:** `src/logging/log-tail.test.ts:60-63`
  - body: The logger formats dates with local getters. In `America/Los_Angeles`, this fixed UTC instant resolves to 2026-04-26, but the test writes and expects `openclaw-2026-04-27.log`, so it fails outside UTC-like zones.
  - confidence: 0.92
- **[P2] Use a local-safe date in the settings test:** `src/logging/logger-settings.test.ts:54-61`
  - body: This test has the same host-local date contract. In zones such as `Pacific/Honolulu`, `2026-03-09T08:30:00.000Z` formats as 2026-03-08 while the expected filename is hard-coded to March 9.
  - confidence: 0.88
- **[P3] Resolve placeholders in status output too:** `src/logging/logger.ts:478`
  - body: This only resolves placeholders inside the logger. Daemon status still uses `resolveConfiguredLogFilePath(cliCfg)`, which returns `config.logging.file` unchanged, so a placeholder config would show `YYYY-MM-DD` in status while the logger writes to the dated path.
  - confidence: 0.84
