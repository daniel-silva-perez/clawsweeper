## Summary

Keep open. Current main and the latest release still leave configured `logging.file` placeholders literal, so this PR is addressing a real gap, but the current branch still has timezone-dependent regression tests and does not share the new resolver with daemon status output.

## What This Changes

The PR adds `YYYY-MM-DD` placeholder resolution for configured logging file paths in `src/logging/logger.ts` and adds logger settings plus log-tail regression tests.

## Best Possible Solution

Land a focused shared logging-path resolver, likely in `src/logging/log-file-path.ts` or a nearby logging helper, and use it consistently for logger writes, log tailing, and daemon status while keeping tests pinned to host-local dates that are stable in every timezone.

## Reproduction Assessment

Yes. Current main reproduces by source inspection because configured `logging.file` is returned unchanged, and the PR's remaining test bugs reproduce by evaluating the frozen UTC instants under western timezones with Node local date getters.

## Solution Assessment

No, not as written. The runtime direction is narrow and appropriate, but the tests should use local-safe instants or derive expectations from the same local-date contract, and the placeholder resolver should be shared with status output before merge.
