## Summary

Keep open. Current main still leaves configured `logging.file` paths literal, so the reported gap is real; the PR is useful but not merge-ready because two new tests are not safe under the documented host-local date contract and the daemon status surface would still report the unresolved placeholder.

## What This Changes

The PR resolves `YYYY-MM-DD` placeholders in configured logging file paths inside the logger and adds logger settings and log-tail regression tests.

## Best Possible Solution

Land a focused version that centralizes dated log-file path resolution for logger settings, file writes, tailing, and status output, with tests that derive expectations from the same host-local date contract.

## Reproduction Assessment

Yes. Source inspection on current main shows configured `logging.file` is returned unchanged, and the timezone probes give a concrete way to reproduce the PR test failures without mutating the checkout.

## Solution Assessment

No, not as written. The core approach is narrow and appropriate, but the resolver should be shared across logging status surfaces and the new tests should use local-safe fake dates or derive the expected date from the same formatter contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use a host-local date in the log-tail test:** `src/logging/log-tail.test.ts:60-63`
  - body: The logger formats dates with local getters. With this fixed UTC instant, hosts such as `America/Los_Angeles` resolve the date as 2026-04-26, but the test writes and expects `openclaw-2026-04-27.log`, so it fails outside UTC-like zones.
  - confidence: 0.92
- **[P3] Use a host-local date in the settings test:** `src/logging/logger-settings.test.ts:54-61`
  - body: This test has the same local-date contract. `2026-03-09T08:30:00.000Z` becomes 2026-03-08 in zones such as `Pacific/Honolulu`, while the expected filename is hard-coded to March 9.
  - confidence: 0.84
- **[P3] Share placeholder resolution with status output:** `src/logging/logger.ts:690-697`
  - body: The new resolver is private to `logger.ts`, but daemon status still calls the separate `resolveConfiguredLogFilePath(cliCfg)` helper, which returns configured `logging.file` unchanged. With a placeholder config, status would still show the literal token.
  - confidence: 0.84
