## Summary

Keep open. Current main still treats a configured `logging.file` containing `YYYY-MM-DD` as a literal path, while the docs publish that pattern as a user-facing logging configuration. The PR addresses a real gap, but the current patch still needs a timezone-portable test fix before merge and should either share placeholder resolution with daemon status or get an explicit maintainer decision to leave status display as follow-up.

## What This Changes

The PR resolves `YYYY-MM-DD` placeholders in configured logger file paths and adds regression coverage for resolved logger settings and log tailing.

## Best Possible Solution

Update the PR to use local-safe fake dates in the new placeholder tests, then land the focused placeholder support with either shared configured-log-file resolution for status output or a clear maintainer decision to keep status display as a follow-up. The final validation should cover the logging tests under a non-UTC timezone plus the normal changed gate.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use local-safe dates in placeholder tests:** `src/logging/log-tail.test.ts:60-63`
  - body: The logger formats log dates with local `getFullYear()`/`getMonth()`/`getDate()`, and the docs say log filenames use the gateway host's local timezone. Freezing this test at `2026-04-27T02:03:04.000Z` makes `TZ=America/Los_Angeles` resolve `2026-04-26`, while the test writes and expects `openclaw-2026-04-27.log`, so it can fail on non-UTC hosts. Construct the fake time with local components or use a local-noon instant before asserting the filename.
  - confidence: 0.92
- **[P3] Share placeholder resolution with status output:** `src/logging/logger.ts:687-697`
  - body: The new resolver stays private to `logger.ts`, but daemon status still reports `resolveConfiguredLogFilePath(cliCfg)`, whose helper returns `config.logging.file` unchanged. With a `openclaw-YYYY-MM-DD.log` config, file logging writes to the dated path while status can keep showing the placeholder literal. Share the resolver with `src/logging/log-file-path.ts` or make the status-display boundary explicit.
  - confidence: 0.82
