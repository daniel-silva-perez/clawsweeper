## Summary

Keep open. Current main still leaves a configured `logging.file` placeholder unresolved for resolved settings and log tail, while the PR implements the focused behavior and tests it; however the new tests freeze UTC instants while the implementation intentionally formats host-local dates, so they can fail outside UTC before merge.

## What This Changes

The PR adds `YYYY-MM-DD` placeholder resolution for `logging.file` in logger path handling and adds regression coverage for resolved settings and log tail.

## Best Possible Solution

Fix the new tests to use local-safe dates, then land the focused placeholder support with either shared path-resolution for daemon status or an explicit maintainer decision to leave status display as follow-up.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use local-safe dates in placeholder tests:** `src/logging/log-tail.test.ts:58-61`
  - body: The logger formats dates with local `getFullYear()`/`getMonth()`/`getDate()`, and the docs say log dates use the gateway host's local timezone. Freezing this test at `2026-04-27T02:03:04.000Z` makes hosts west of UTC resolve `2026-04-26`, so the test can fail while expecting `openclaw-2026-04-27.log`. Construct the date with local components or choose a local-noon timestamp before asserting the filename.
  - confidence: 0.91
- **[P3] Share placeholder resolution with status output:** `src/logging/logger.ts:690-697`
  - body: The new resolver stays private to `logger.ts`, but daemon status still reports `resolveConfiguredLogFilePath(cliCfg)`, whose helper returns `config.logging.file` unchanged. If placeholder support is meant to cover user-visible log-file paths, share this resolver with `src/logging/log-file-path.ts` so status does not keep showing `openclaw-YYYY-MM-DD.log`.
  - confidence: 0.82
