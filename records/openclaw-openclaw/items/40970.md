## Summary

Keep this PR open. Current main still does not implement the PR head’s intended shared `YYYY-MM-DD` substitution for `logging.file`: file writes get partial rolling-path behavior, but resolved settings, log tailing, daemon/status output, and other consumers can still see or read the literal placeholder path.

## Best Possible Solution

Keep this PR open and review or land an equivalent focused fix. The best path is a shared logging-file resolver used consistently by resolved settings, file transport, `logs.tail`, startup output, daemon status, and channel-log helpers, with deterministic fake-timer tests and docs/config wording that clearly defines the supported `YYYY-MM-DD` placeholder semantics.
