## Summary

Keep open. Current main still returns configured `logging.file` paths literally, so the user-visible gap is real, but the latest PR head still has deterministic timezone bugs in its new tests and leaves daemon status reporting inconsistent with the resolved file path.

## What This Changes

The PR adds `YYYY-MM-DD` substitution for configured logging file paths in `src/logging/logger.ts` and adds logger settings and log-tail regression tests.

## Best Possible Solution

Land a focused version that moves date-placeholder resolution into a shared logging path helper used by logger settings/writes, log tailing, and daemon status, while keeping the host-local date contract and making the new tests timezone-safe.

## Reproduction Assessment

Yes. Current main reproduces by source inspection because configured `logging.file` is returned unchanged, and the PR test failures are reproducible with timezone probes that show the frozen UTC instants resolve to the previous local date in western timezones.

## Solution Assessment

No, not as written. The implementation direction is narrow and appropriate, but the resolver should be shared with status output and the tests should use local-safe instants or derive expected filenames from the same local-date formatter contract.
