## Summary

Keep open. Current main still leaves configured `logging.file` placeholders literal, so the PR addresses a real gap, but the latest PR head still has timezone-dependent tests that fail outside UTC-like zones and should be fixed before merge.

## What This Changes

The PR adds `YYYY-MM-DD` substitution for configured logging file paths, updates rolling log path re-resolution, and adds logger settings plus log-tail regression tests.

## Best Possible Solution

Land a focused logging-path helper that resolves the host-local `YYYY-MM-DD` placeholder consistently for logger writes, log tailing, and status output, with timezone-safe tests that use local-noon instants or derive expectations from the same local-date contract.

## Reproduction Assessment

Yes. Current main reproduces by source inspection because configured `logging.file` is returned unchanged, and the PR test failures follow from fixed UTC instants combined with local `Date` getters in western timezones.

## Solution Assessment

No, not as written. The implementation direction is narrow and appropriate, but the new tests should be timezone-safe and the placeholder resolver should be shared with status output before merge.
