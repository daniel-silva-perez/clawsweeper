## Summary

Keep this PR open. Current main still has the reported memory keyword snippet behavior, and the PR direction is correct, but the branch edits the legacy `src/memory/manager-search.ts` path while the active implementation now lives under `extensions/memory-core/src/memory/manager-search.ts`.

## What This Changes

The PR branch changes legacy memory keyword search to return SQLite FTS5 `snippet()` output with clamped token count and visible ellipsis markers instead of truncating the full FTS row text from the beginning.

## Best Possible Solution

Port the snippet behavior into `extensions/memory-core/src/memory/manager-search.ts`, clamp FTS5 snippet tokens to the documented 1..64 range, use visible omission markers, preserve fallback lexical ranking and LIKE/trigram fallback behavior, and add a regression test where the match occurs deep in a large chunk.

## Reproduction Assessment

Yes. The linked bug gives a concrete CLI path, and current-main source confirms MATCH-backed keyword rows still select full `text` and truncate from the beginning, which hides a term located beyond the first 700 characters.

## Solution Assessment

No, not in its current form. The proposed SQLite `snippet()` approach is the right narrow fix, but it must be rebased or replaced against the active memory-core plugin path and covered by a deep-match regression test before merge.
