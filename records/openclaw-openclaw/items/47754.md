## Summary

Keep this PR open, but do not merge it as-is. The bug remains on current main in the active memory-core search path, while this PR changes a legacy path that no longer exists on main, so the implementation idea is useful but needs to be ported and tested against the current plugin.

## What This Changes

The PR branch changes legacy memory keyword search to return SQLite FTS5 snippet output with clamped token count and visible ellipsis markers instead of truncating the full FTS row text from the beginning.

## Best Possible Solution

Port the FTS5 snippet behavior into extensions/memory-core/src/memory/manager-search.ts, clamp snippet tokens to SQLite's 1..64 range, use visible omission markers, preserve LIKE/trigram fallback and fallback ranking behavior, and add a regression test where the matched term appears past the old 700-character prefix window.

## Reproduction Assessment

Yes. The linked bug gives a concrete CLI path, and current-main source confirms MATCH-backed keyword rows still return truncateUtf16Safe(row.text, 700), so an indexed chunk with a unique term after the first 700 characters will reproduce the missing-snippet behavior.

## Solution Assessment

No. The SQLite snippet approach is the right narrow fix, but this PR applies it to the wrong file for current main; the maintainable solution is the same behavior ported to memory-core with a targeted regression test.
