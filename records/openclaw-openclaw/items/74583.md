## Summary

Current main now handles the reported local-loopback `openclaw logs --follow` failure mode by falling back to the configured gateway file log when the implicit loopback RPC closes or times out, so the command should keep following instead of exiting with the gateway-not-reachable message shown in the report. The fix is on main only; no release tag contains the proof commit yet.

## What This Changes

The issue asks for `openclaw logs --follow` to keep live monitoring connected or recover from websocket/log-tail hiccups instead of exiting with a gateway-not-reachable error while the gateway remains healthy.

## Best Possible Solution

Keep the current main fallback for local operator monitoring, and let #56475 continue separately if maintainers want a broader streamed/persistent websocket follow mode for remote or higher-throughput cases.
