## Summary

Keep open: current main still emits raw ClawHub 429 failures without an unauthenticated-login hint, and the issue is paired with open implementation PR #56394, so it should remain open until that PR is reviewed, merged, replaced, or closed.

## What This Changes

Request adding a login/authentication hint when ClawHub skill search or install receives an unauthenticated HTTP 429 rate-limit response.

## Best Possible Solution

Add the hint in the shared ClawHub request/error boundary by tracking whether a request used an auth token and appending the login guidance only for unauthenticated 429 responses across JSON and archive-download paths, with regression coverage for unauthenticated 429, authenticated 429, and non-429 failures.

## Reproduction Assessment

Yes. A mocked `fetchImpl` returning HTTP 429 through `searchClawHubSkills`, `downloadClawHubPackageArchive`, or `downloadClawHubSkillArchive` will still produce the raw `ClawHub ... failed (429): Rate limit exceeded` message on current main.

## Solution Assessment

Yes. The requested shared ClawHub infra fix is the narrowest maintainable approach because it covers CLI and gateway skill/plugin paths without changing authenticated rate-limit behavior.
