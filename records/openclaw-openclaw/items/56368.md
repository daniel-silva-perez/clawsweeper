## Summary

Keep open: current main still surfaces raw ClawHub 429 failures without unauthenticated login guidance, and the provided GitHub context shows an open implementation PR with closing syntax, so this issue should remain open until that PR is reviewed, merged, replaced, or closed.

## What This Changes

The issue requests a ClawHub login tip on unauthenticated HTTP 429 rate-limit failures from `openclaw skills search` and `openclaw skills install`.

## Best Possible Solution

Add the hint in the shared ClawHub request/error boundary by tracking whether a request used an auth token and appending login guidance only for unauthenticated 429 responses across JSON and archive-download paths, with regression coverage for unauthenticated 429, authenticated 429, and non-429 failures.

## Reproduction Assessment

Yes. A mocked `fetchImpl` returning HTTP 429 through `searchClawHubSkills`, `downloadClawHubPackageArchive`, or `downloadClawHubSkillArchive` should still produce the raw `ClawHub ... failed (429): Rate limit exceeded` message on current main.

## Solution Assessment

Yes. The shared ClawHub infra fix is the narrowest maintainable direction because CLI and gateway skill/plugin callers share this request/error path; maintainer action should focus on reviewing, landing, replacing, or closing PR #56394.
