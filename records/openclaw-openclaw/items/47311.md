## Summary

Keep open. Current main still lacks the requested WebChat server-side dictation path: the composer has attachments and realtime Talk, but no MediaRecorder one-shot capture flow or authenticated Gateway method that transcribes audio through `tools.media.audio` and returns text to the draft.

## What This Changes

Request server-side speech-to-text for WebChat voice input by recording browser audio and transcribing it through configured Gateway media audio backends.

## Best Possible Solution

Implement a focused Control UI/WebChat batch dictation flow: capture completed microphone audio with `getUserMedia` plus `MediaRecorder`, upload it through an authenticated and size-bounded Gateway method, transcribe with the existing `tools.media.audio` media-understanding path, and append the transcript to the composer draft without starting an agent run or conflating it with realtime Talk.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The high-confidence verification path is current-main source inspection: no server-side dictation control or Gateway transcription method exists, and the UI test asserts the old `Voice input` control is absent.

## Solution Assessment

Yes, the requested direction is the narrow maintainable solution. A dedicated batch dictation method is safer than reusing `chat.send` because it can return transcript text to the draft while preserving authenticated upload limits and the existing `tools.media.audio` provider/CLI configuration.
