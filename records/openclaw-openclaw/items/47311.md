## Summary

Keep open: current main still lacks the requested WebChat server-side STT dictation path. The chat UI can attach files and run realtime Talk, and the Gateway/media-understanding stack can transcribe audio attachments, but there is no MediaRecorder microphone capture flow or Gateway method that returns a transcript to the draft without starting a chat run.

## What This Changes

Request a WebChat voice-input option that records browser microphone audio and sends it to Gateway for configured `tools.media.audio` transcription instead of relying on browser Web Speech APIs.

## Best Possible Solution

Implement a focused Control UI/WebChat batch dictation flow: capture microphone audio with `getUserMedia` plus `MediaRecorder`, send the completed recording through an authenticated bounded Gateway method, transcribe it via existing `tools.media.audio` media-understanding plumbing, and append the returned text to the draft while keeping realtime Talk and generic file attachments separate.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The high-confidence verification path is current-main inspection: no `MediaRecorder` path exists in the relevant UI/Gateway surface, and the UI test explicitly asserts the old `Voice input` control is absent.

## Solution Assessment

Yes, the requested direction is the narrow maintainable solution. A dedicated batch dictation method is safer than repurposing `chat.send`, because it can return transcript text without starting an agent run while still reusing existing audio transcription limits and provider/CLI configuration.
