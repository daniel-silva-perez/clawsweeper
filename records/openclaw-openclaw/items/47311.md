## Summary

Current main still does not implement the requested WebChat batch server-side STT dictation path. The UI supports file attachments and realtime Talk, and Gateway/media-understanding can transcribe audio files through `tools.media.audio`, but there is no `MediaRecorder` microphone recorder that sends audio to Gateway and inserts the returned transcript into the draft.

## What This Changes

Add a WebChat voice-input path that records browser microphone audio and sends it to Gateway for server-side `tools.media.audio` transcription instead of relying on browser Web Speech APIs.

## Best Possible Solution

Implement a focused Control UI/WebChat batch dictation path: record microphone audio with `getUserMedia` plus `MediaRecorder`, send the completed recording through an authenticated additive Gateway method, transcribe it via existing `tools.media.audio`/media-understanding plumbing, and append the returned text to the chat draft while keeping realtime Talk and generic file attachments separate.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The high-confidence verification path is current-main code inspection: the composer/test surface lacks a dictation control, targeted search finds no `MediaRecorder`, and the only microphone capture paths are realtime Talk transports.

## Solution Assessment

Yes, the requested direction is the best narrow solution. A dedicated batch dictation RPC is safer and clearer than repurposing `chat.send`, because it can return transcript text without starting an agent run while still reusing existing audio/media safeguards.
