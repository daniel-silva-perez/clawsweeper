## Summary

Keep this issue open. Current main still routes normal Feishu inbound messages through a per-chat sequential promise queue before the shared gateway queue policy sees them, while `collect` only batches when a follow-up arrives while the gateway run is still active. The prior discussion correctly distinguishes #70133 as related but a different failure mode, and the newer #67793 is a broader open report rather than a better canonical replacement for this detailed Feishu root-cause issue.

## Best Possible Solution

Fix the Feishu inbound dispatch path so follow-up-eligible messages for the same chat can reach `dispatchReplyFromConfig` while the first run is still active, without breaking the intentional separate lanes for abort and `/btw` or the existing dedupe/debounce behavior. Add a Feishu regression test that simulates an active run with `messages.queue.mode = "collect"` and verifies multiple later same-chat messages become one collected follow-up turn; coordinate with #70133 only if the queue refactor can safely address both failure modes.
