## Summary

Keep this issue open. Current main still serializes normal Feishu inbound messages through a same-chat promise queue before the gateway collect/follow-up policy can observe an active run, so `messages.queue.mode = "collect"` can still degrade into separate `run-now` turns. The related open reports are useful context, but #70133 covers a hung/stuck queue and #67793 is broader rather than a better canonical replacement for this focused root-cause report.

## Best Possible Solution

Fix the Feishu inbound dispatch path so normal same-chat messages can reach `dispatchReplyFromConfig` while the first run is still active, allowing `collect` to enqueue and batch them as one follow-up turn. Preserve the existing abort and `/btw` out-of-band lanes, keep Feishu dedupe/debounce behavior intact, and add a Feishu regression test that simulates an active run with `messages.queue.mode = "collect"` and multiple later same-chat messages.
