## Summary

Keep this open. Current main still lacks a SQLite/FTS-backed wiki search cache; memory-wiki search remains digest-assisted Markdown reads, and the shared memory bridge delegates wiki hits back through the same local wiki search path. The request is recent, concrete, extension-owned, and narrow enough for a focused implementation PR.

## What This Changes

Request adding an internal SQLite FTS5 index for memory-wiki `wiki_search`, refreshed by compile/apply/write paths and used before falling back to the existing file-based search.

## Best Possible Solution

Add the index inside the memory-wiki plugin as a read-optimized cache under the wiki cache area, refreshed by compile/apply/ingest, with Markdown and compiled digests remaining the source of truth and the current local search retained as fallback.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/performance request. The missing capability is high-confidence from current source because `wiki_search` uses digest-assisted Markdown reads and memory-wiki config exposes no SQLite/FTS index path.

## Solution Assessment

Yes, the requested direction is the narrow maintainable path if implemented inside memory-wiki. Expanding `memory/main.sqlite` or moving wiki-specific ranking into core memory search would be broader than needed.
