## Summary

Keep this issue open. Current main has a memory-wiki search path that uses compiled digest files as a candidate shortcut and then reads Markdown pages, but it does not have the requested SQLite/FTS-backed wiki_search cache; the report is concrete, recent, extension-owned, and narrow enough to be a repair candidate.

## What This Changes

The issue requests a memory-wiki-owned SQLite FTS5 index for wiki_search that is refreshed by wiki compile/apply, returns ranked results, and falls back to file scanning when unavailable or stale.

## Best Possible Solution

Add the SQLite/FTS cache inside the memory-wiki plugin as a read-optimized vault cache under the wiki-owned cache directory, with Markdown and compiled digests remaining the source of truth and the existing local search path preserved as fallback.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature/performance request. The missing capability is verifiable from current-main source because wiki_search uses digest-assisted Markdown reads and no SQLite/FTS wiki index exists.

## Solution Assessment

Yes, the requested direction is a good narrow solution if implemented as a memory-wiki-owned cache with graceful fallback. It should not expand memory/main.sqlite or move wiki-specific ranking into core memory search.

## Risks / Open Questions

- node:sqlite and FTS5 availability varies across Node builds, so a repair must keep the current file-scan/digest path as a reliable fallback rather than making wiki_search depend on FTS5.
- External edits to Markdown files can bypass wiki_apply, so the index needs compile-time rebuild/staleness detection or mtime/content-hash checks to avoid stale ranked results.
- The reported performance gain is plausible from the implementation shape, but a large-vault benchmark would still be needed to quantify latency/token savings.
