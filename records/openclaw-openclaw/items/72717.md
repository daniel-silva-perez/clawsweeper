## Summary

Keep this issue open. Current main still implements memory-wiki search with compiled JSON/JSONL digest candidates plus Markdown reads, while the requested SQLite/FTS-backed wiki cache is not present; the report is recent, concrete, plugin-owned, and narrow enough for a focused repair PR.

## What This Changes

Request a SQLite FTS5-backed index for memory-wiki `wiki_search`, refreshed by wiki compile/apply writes and used before falling back to file scanning.

## Best Possible Solution

Add a memory-wiki-owned SQLite/FTS cache under the wiki cache area, refreshed by compile/apply/ingest paths, with Markdown and compiled digests remaining the source of truth and the current local search retained as fallback.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/performance request. The missing capability is high-confidence from current source because `wiki_search` uses digest-assisted Markdown reads and memory-wiki config has no SQLite/FTS index option.

## Solution Assessment

Yes, the requested direction is the narrow maintainable path if implemented inside memory-wiki as a read-optimized cache. Expanding `memory/main.sqlite` or moving wiki-specific ranking into core memory search would be broader than the report requires.
