## Summary

Keep this issue open. Current main has useful partial safeguards for plugin hook errors, selected hook timeouts, plugin init failure isolation, and a few subsystem-specific breakers, but it still lacks the requested framework-level plugin circuit breaker with rolling budgets, lock attribution, persisted open/half-open/forced-open state, operator status/reset surfaces, and per-plugin fail modes. The related #55253 is still open and narrower, and the latest discussion adds concrete production reliability context rather than making the request stale.

## Best Possible Solution

Keep this open as core plugin/gateway runtime stability work. The best path is an additive plugin-runtime breaker around hook and plugin-owned runtime execution: record structured plugin id, hook name, session id, timing, timeout, error, and lock-overrun telemetry; maintain rolling failure budgets; persist open/half-open/forced-open state; expose status, metrics, and operator reset controls; support per-plugin fail modes; and add tests proving one failing plugin degrades without blocking unrelated plugins. Coordinate with #55253 if maintainers choose one canonical tracker.
