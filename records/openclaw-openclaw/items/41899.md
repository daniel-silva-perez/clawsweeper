## Summary

Keep this issue open. Current main has partial safeguards for plugin startup failures, static hook timeouts, session-lock cleanup, and a few owner-specific circuit breakers, but it still does not implement the requested framework-level plugin circuit breaker with rolling plugin budgets, lock attribution, persisted open/half-open/forced-open state, operator reset/status surfaces, or per-plugin fail modes.

## Best Possible Solution

Keep this open as plugin/gateway runtime stability work. The likely path is an additive core plugin-runtime breaker around hook and plugin-owned runtime execution: record plugin id, hook name, session id, timing, timeout, error, and lock-overrun telemetry; maintain rolling failure budgets; persist open/half-open/forced-open state; expose status, metrics, and operator reset controls; support per-plugin fail modes; and add tests proving one failing plugin degrades without blocking unrelated plugins. Coordinate with #55253 if maintainers want a single canonical tracker.
