## Summary

Keep open. Current main has partial safeguards such as hook timeouts, fail-open/fail-closed hook policy, invalid plugin config isolation, session-lock cleanup, and gateway diagnostics, but it still lacks the requested framework-level plugin circuit breaker with rolling plugin budgets, lock attribution, persisted open/half-open/forced-open state, operator reset/status surfaces, and per-plugin fail modes. Related #55253 and the newer deployment comment reinforce that this remains active gateway/plugin-runtime reliability work.

## Best Possible Solution

Keep this open as the broad plugin/gateway runtime circuit-breaker tracker unless maintainers explicitly make #55253 canonical. The best implementation belongs around core plugin hook/runtime execution and session-lock telemetry: record plugin id, hook name, session id, timing, timeout, error, and lock-overrun signals; maintain rolling budgets; persist closed/open/half-open/forced-open state; expose status, metrics, and reset controls; support per-plugin fail modes; and test that one failing plugin degrades without blocking unrelated plugins.

## Risks / Open Questions

- Closing this now would lose a concrete core runtime reliability request that remains broader than current timeout, diagnostics, invalid-config quarantine, and subsystem-specific breaker safeguards.
- A future implementation must preserve security-sensitive hook policy, especially before_tool_call fail-closed behavior, so degraded plugins are not accidentally skipped fail-open.
- Breaker state, metrics, reset controls, and per-plugin degraded behavior need explicit docs/config/tests to avoid silently disabling critical plugins or hiding operational failures.
