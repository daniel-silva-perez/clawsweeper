## Summary

Keep open. Current main has useful resilience pieces, including hook timeouts, static fail-open/fail-closed hook policy, invalid plugin config isolation, lock cleanup, and an Active Memory-local timeout breaker, but it still lacks the requested framework-level plugin circuit breaker with rolling budgets, plugin/hook lock attribution, persisted open/half-open/forced-open state, status/reset controls, and per-plugin degraded fail modes.

## What This Changes

The issue requests a framework-level plugin circuit breaker that wraps plugin hook execution, attributes lock ownership to plugins and hooks, tracks rolling failures and overruns, persists breaker state, skips only degraded plugin hooks, and exposes status/reset/fail-mode controls.

## Best Possible Solution

Use this as the broad gateway/plugin-runtime breaker tracker unless maintainers make #55253 canonical. The end state should be a host-side breaker around plugin hook execution and session-lock telemetry, with plugin/hook/session attribution, rolling error and latency budgets, persisted closed/open/half-open/forced-open state, status and reset controls, per-plugin fail modes, and regression tests proving one failing plugin degrades without blocking unrelated plugins.

## Reproduction Assessment

Not applicable as a conventional bug reproduction; this is a feature request. The implementation gap is high-confidence from static inspection of current hook execution, lock ownership, docs, and tests.

## Solution Assessment

Yes at the architecture boundary: this needs core runtime participation because external plugins cannot reliably gate all plugin hooks or attribute host session locks to active plugin handlers. The safer implementation path is phased: telemetry and lock attribution first, passive evaluation second, then active breaker behavior behind a gateway flag before enabling defaults.
