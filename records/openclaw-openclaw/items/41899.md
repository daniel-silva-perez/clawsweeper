## Summary

Keep this issue open. Current main has partial safeguards, including static hook timeouts, fail-open/fail-closed hook policy, invalid plugin config quarantine, session-lock watchdog cleanup, tool-loop breakers, and plugin-specific breakers, but it still lacks the requested framework-level plugin circuit breaker with rolling plugin budgets, lock attribution, persisted open/half-open/forced-open state, operator reset/status surfaces, and per-plugin fail modes. The newer deployment comment and related open #55253 reinforce that this is still active gateway/plugin-runtime reliability work.

## Best Possible Solution

Keep this open as plugin/gateway runtime stability work. The likely implementation belongs around core plugin hook/runtime execution: record plugin id, hook name, session id, timing, timeout, error, and lock-overrun telemetry; maintain rolling budgets; persist open/half-open/forced-open state; expose status, metrics, and operator reset controls; support per-plugin fail modes; and add tests proving one failing plugin degrades without blocking unrelated plugins. Coordinate with #55253 if maintainers want one canonical tracker.
