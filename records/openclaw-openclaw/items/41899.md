## Summary

Keep this issue open. Current main has partial safeguards for plugin hook errors, one modifying-hook timeout, session-lock cleanup, invalid plugin config isolation, and subsystem-specific circuit breakers, but it still lacks the requested plugin-scoped circuit breaker with rolling budgets, lock attribution, persisted open/half-open/forced-open state, status visibility, reset controls, and per-plugin fail modes. Related #55253 is still open and narrower, so it does not supersede this broader framework request.

## Best Possible Solution

Keep this open as core plugin/gateway stability work. The best path is an additive plugin-runtime breaker around hook and plugin-owned runtime execution, with structured plugin id plus hook/session telemetry, lock-overrun attribution, rolling failure and timeout budgets, persisted breaker state, status/diagnostics output, operator reset and forced-open controls, and tests proving one failing plugin degrades without blocking unrelated plugins. Coordinate with #55253 if maintainers want one canonical tracker.
