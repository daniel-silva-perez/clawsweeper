## Summary

Keep this issue open. Current main has useful partial resilience work for plugin hook errors, selected hook timeouts, plugin init failure isolation, and session-lock cleanup, but it still lacks the requested framework-level plugin circuit breaker with rolling budgets, plugin/hook/session lock attribution, persisted open/half-open/forced-open state, status visibility, reset controls, and per-plugin fail modes. The related #55253 remains open and narrower, so it does not supersede this broader design request.

## Best Possible Solution

Keep this open as core plugin/gateway stability work. The best path is an additive plugin-runtime breaker around hook and plugin-owned runtime execution: record structured plugin id, hook name, session id, timing, timeout, and lock-overrun telemetry; maintain rolling failure budgets; persist open/half-open/forced-open state; expose status/diagnostics and operator reset controls; support per-plugin fail modes; and add tests proving one failing plugin degrades without blocking unrelated plugins. Coordinate with #55253 only if maintainers choose one canonical tracker.
