## Summary

Keep open. Current main has related knobs, but it does not implement the requested raw tailnet/IP Control UI mode with no token and no HTTPS. `gateway.auth.mode: "none"` disables shared-secret auth only where the Gateway is still on loopback; non-loopback binds without token/password are explicitly refused. The documented tokenless path is Tailscale Serve over HTTPS, while direct tailnet bind is still documented with token auth.

## Best Possible Solution

Leave this open for maintainer product/security judgment. A safe implementation would need an explicit, narrowly named trusted-tailnet/private-ingress mode with startup warnings, origin handling, docs, audit coverage, and tests proving exactly which Control UI, WebSocket, and HTTP API surfaces are allowed without shared-secret auth. If maintainers do not want non-loopback no-auth, close manually as a policy decision and point users to Tailscale Serve or trusted-proxy auth.

## Risks / Open Questions

- The request is security-sensitive because it combines disabling shared-secret auth, HTTPS/device identity, pairing, and possibly HTTP API auth on an admin surface.
- Current policy intentionally treats tokenless direct non-loopback Gateway exposure differently from tokenless Tailscale Serve identity-header auth, so implementing this would need explicit maintainer product/security approval rather than cleanup triage.
