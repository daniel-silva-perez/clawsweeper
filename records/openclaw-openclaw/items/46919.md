## Summary

Keep open. Current main still does not implement the requested direct tailnet/IP Control UI mode with no token, no HTTPS, and no pairing. Main has narrower alternatives: loopback-only `gateway.auth.mode: "none"`, tokenless Tailscale Serve identity-header auth, and trusted-proxy auth, but non-loopback no-auth binds are still refused and direct tailnet docs still require token auth.

## Best Possible Solution

Keep this open for maintainer product/security judgment. A safe implementation would need a narrowly named trusted-tailnet/private-ingress mode, explicit warnings, origin and HTTP API scope handling, docs, and tests proving exactly which Control UI, WebSocket, and HTTP surfaces are allowed without shared-secret auth. If maintainers reject direct non-loopback no-auth as policy, close manually with that rationale and point users to Tailscale Serve identity-header auth or trusted-proxy auth.

## Risks / Open Questions

- The requested behavior would deliberately remove shared-secret auth, HTTPS/device identity, and pairing from an admin surface, so it needs explicit product/security approval rather than cleanup closure.
- Current docs mention private-ingress `gateway.auth.mode="none"` for some HTTP surfaces, while startup still refuses direct non-loopback no-auth binds; any implementation should reconcile that boundary clearly.
- Adding a dashboard token URL config would need to preserve existing protections for SecretRef-managed tokens and avoid increasing token leakage through logs, clipboard history, browser launch arguments, or query strings.
