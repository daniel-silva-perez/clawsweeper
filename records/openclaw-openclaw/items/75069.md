## Summary

Keep open. Current main contains useful partial mitigations for hardlink reuse, dist-mirror memoization, and already-materialized runtime-deps fast paths, but the issue’s broader Gateway readiness and event-loop blocking problem is still live in source and maintainer discussion explicitly leaves it unresolved.

## What This Changes

The issue reports that bundled plugin runtime-deps and mirror preparation run synchronously during Gateway startup and agent/plugin activation, causing long event-loop stalls in packaged installs.

## Best Possible Solution

Keep this issue open until PR #75183 or an equivalent staged fix is resolved, then verify with packaged/global-install profiling that Gateway HTTP readiness and first-agent event-loop lag are bounded for bundled runtime-deps and mirror preparation.

## Reproduction Assessment

Yes. The issue body gives a versioned npm-global reproduction, CPU attribution, and standalone mirror benchmarks, and NikolaFC added an acpx-only startup-readiness repro; current main still contains the source paths those reproductions target.

## Solution Assessment

Unclear. Current main is a partial mitigation, while the remaining maintainable direction needs profiled maintainer choice between readiness deferral/caching and a broader async or cooperative mirror-preparation rewrite.
