## Summary

Keep open. Current main has partial mitigations for repeated bundled runtime dist mirroring, but the remaining runtime-root preparation path is still synchronous, still uses synchronous filesystem walking/fingerprinting/locking, and the hardlink freshness check still matches a weakness described in the report.

## What This Changes

The issue asks OpenClaw to prevent bundled plugin runtime mirror preparation from blocking pi-agent and Gateway invocations by repeatedly walking and materializing packaged plugin runtime roots synchronously.

## Best Possible Solution

Fix this in staged, profile-backed patches: preserve the process-level dist-root memoization, fix hardlink reuse with inode identity, avoid unnecessary capability registry loads during tool construction, and move any remaining large mirror/fingerprint work off the agent invocation path or make it cooperative with event-loop yields.

## Reproduction Assessment

Yes for the released regression: the report provides version, install method, lag output, CPU stack attribution, and a direct bench plan. For current main, this review confirms the remaining synchronous hot-path code but did not execute a live Gateway or bench profile.

## Solution Assessment

Unclear. The proposed full async migration would address event-loop starvation, but it crosses the current synchronous plugin loader contract; a safer maintainable path is to land narrower dedupe/hardlink/provider-load fixes first, profile, then widen to async/cooperative mirroring where needed.
