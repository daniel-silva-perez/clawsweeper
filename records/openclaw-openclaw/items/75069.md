## Summary

Keep open. Current main has partial mitigations for the bundled runtime mirror regression, but the issue is not fully implemented: runtime root preparation, mirror refresh/fingerprinting, runtime-deps lock waiting, and some package-manager staging paths still contain synchronous work, and maintainer comments in the thread explicitly identify remaining startup-readiness/event-loop follow-up.

## What This Changes

Fix a regression where bundled plugin runtime mirror and runtime-deps preparation can run synchronously and repeatedly on pi-agent and Gateway startup paths, blocking the main event loop for long periods.

## Best Possible Solution

Keep the issue open and finish this as a staged, profile-backed runtime-loading effort: preserve the landed hardlink/dist-mirror cache fixes, then move or defer readiness-critical runtime-deps staging and make the remaining mirror/fingerprint work async or cooperative with clear event-loop lag targets.

## Reproduction Assessment

Yes for the released regression: the issue and follow-up comment provide versioned packaged-install steps, lag output, stack attribution, and an acpx-only HTTP-readiness repro. For current main, the same packaged-build probes should be rerun because the code has partial mitigations but still contains synchronous hot-path work.

## Solution Assessment

Unclear. A full async migration would address starvation, but it cuts across existing synchronous loader contracts; the safer maintainer path is staged mitigation with before/after profiles, starting with Gateway readiness deferral/caching and remaining repeated mirror/fingerprint work.
