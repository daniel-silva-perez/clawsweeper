## Summary

Keep open. Current main contains partial mitigations for the bundled runtime mirror path, but the issue discussion and maintainer comments explicitly identify remaining synchronous runtime-deps/mirror and gateway-readiness work, and an unmerged closing PR already targets the remaining fix.

## What This Changes

The issue reports that bundled plugin runtime dependency mirroring/staging runs synchronously during gateway and pi-agent paths, causing long event-loop stalls after the 2026.4.22+ runtime-deps changes.

## Best Possible Solution

Keep this issue open and use #75183 as the active implementation path: review whether it bounds packaged gateway readiness and first-agent runtime-deps work, then land it or replace it with a narrower profiled fix that preserves the partial mirror-cache and inode-hardlink repairs.

## Reproduction Assessment

Yes for the released regression: the report includes versioned packaged-install steps, lag summaries, CPU-profile attribution, and NikolaFC added a smaller acpx-only gateway readiness repro. For current main, the reproduction should be rerun against a packaged build because source inspection shows remaining synchronous code but not the post-mitigation timing.

## Solution Assessment

Unclear. The full async migration suggested in the issue would address event-loop starvation, but the currently safest path appears to be the staged approach in #75183 or an equivalent profiled fix that avoids duplicate staging and keeps heavy runtime-deps work out of read-only and readiness-critical paths.
