## Summary

Keep open. Current main has meaningful mitigations for the reported regression, including the inode-based hardlink freshness check and narrower provider-policy/runtime-deps behavior, but it has not fully solved the observable problem: bundled runtime dependency staging, mirror refresh, fingerprinting, and lock waiting are still synchronous, and Gateway startup still loads enabled startup plugins before HTTP bind. NikolaFC's follow-up also adds a concrete startup-readiness reproduction in the same bundled runtime-deps/mirror family.

## What This Changes

The issue asks to prevent bundled plugin runtime mirror and dependency preparation from synchronously blocking Gateway readiness and pi-agent invocation paths.

## Best Possible Solution

Keep the issue open and finish the fix as a staged, profile-backed runtime-loading effort: preserve the current dist-mirror cache and inode hardlink check, keep capability/provider lookups narrow, then move package-manager staging and large mirror/fingerprint work off Gateway HTTP readiness and pi-agent synchronous execution, or make that work async/cooperative with clear event-loop lag bounds.

## Reproduction Assessment

Yes for the released regression: the issue and follow-up comment include versioned packaged-install setups, timing tables, lag output, stack attribution, and a minimal acpx-only Gateway readiness repro. For current main, the reproduction should be rerun against a packaged build because source inspection shows remaining sync work but not the exact post-mitigation timing.

## Solution Assessment

Unclear. The suggested full async migration would address event-loop starvation, but it crosses the synchronous plugin-loader/runtime-root contract; a safer path is staged mitigation with profile proof, starting from readiness-critical package-manager staging and any remaining repeated mirror/fingerprint work.
