## Summary

Keep open. Current main has partial mitigations for the bundled runtime mirror regression, but source inspection still shows synchronous runtime-deps, mirror, fingerprint, and lock work on the same startup/runtime preparation family, and the issue is already paired with open PR #75183 as the active implementation candidate.

## What This Changes

The issue reports that bundled plugin runtime mirror and dependency preparation synchronously repeats during pi-agent and gateway startup paths, blocking the gateway event loop for tens of seconds on packaged installs.

## Best Possible Solution

Keep the issue open until PR #75183 or a successor lands a profile-backed fix that removes repeated runtime-deps/mirror work from read-only and pi-agent hot paths, avoids per-plugin sync repair during Gateway startup, and proves bounded event-loop/readiness behavior for the reported packaged-install scenarios.

## Reproduction Assessment

Yes for the released regression: the report and follow-up comment include versioned packaged-install setups, timing tables, lag output, stack attribution, and an acpx-only Gateway readiness repro. For current main, the exact residual timing still needs a packaged-build rerun because source inspection shows remaining sync work but not its post-mitigation duration.

## Solution Assessment

Unclear. The full async migration suggested in the report may be the eventual best fix, but the safer maintainer path is to review the staged runtime-deps/mirror PR, validate it against the acpx and first-agent-run profiles, and then decide whether a larger async contract change is still needed.
