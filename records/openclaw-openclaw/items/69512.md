## Summary

Keep this issue open. Current main still maps Claude CLI permission mode only for OpenClaw YOLO exec policy and does not forward or enforce `exec-approvals.json` allowlist entries for Claude Code-internal Bash execution; the related Claude CLI policy work explicitly stopped short of host exec approval enforcement.

## What This Changes

Request forwarding per-agent OpenClaw exec allowlist policy into `claude-cli` backend sessions so Claude Code-internal shell execution follows the same allow/deny semantics as native OpenClaw exec.

## Best Possible Solution

Design a deny-by-default Claude CLI policy bridge that preserves OpenClaw allowlist semantics for allowed and denied shell commands, preferably through a Claude-supported permission hook or prompt-tool mechanism rather than lossy string-only translation, then document unsupported cases and add regression tests for per-agent allowlist allow and deny paths.

## Reproduction Assessment

Yes. The issue has concrete live steps, and current source-level checks reproduce the core mismatch: native gateway exec evaluates `exec-approvals.json`, while `resolveCliBackendConfig("claude-cli")` under allowlist policy leaves Claude permissions to Claude Code rather than forwarding OpenClaw allowlist entries.

## Solution Assessment

Unclear. Forwarding the policy is the right product direction if Claude CLI remains a primary runtime, but the exact mechanism needs maintainer/security design because a naïve `--allowedTools Bash(...)` translation could weaken OpenClaw's stricter allowlist semantics.
