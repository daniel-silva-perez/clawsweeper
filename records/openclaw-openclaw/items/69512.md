## Summary

Keep this open. Current main has Claude CLI permission-mode normalization and a live stdio runner, but I did not find any path that translates or enforces `exec-approvals.json` allowlist entries for Claude Code's internal shell/tool execution; the linked Codex resume sandbox fix is a different backend and does not cover this surface.

## What This Changes

Forward OpenClaw exec approval allowlists into `claude-cli` backend sessions so Claude Code internal exec follows the same allow/deny policy as OpenClaw native exec.

## Best Possible Solution

Keep the issue open for a security/design decision, then implement a Claude CLI-owned permission bridge that either maps OpenClaw's effective allowlist/deny policy into a supported Claude Code permission mechanism or clearly diagnoses `allowlist` as unsupported for that backend. The implementation should cover global and per-agent exec policy, fresh and resumed/live Claude sessions, docs, and regression tests for the resolved argv/policy behavior.

## Reproduction Assessment

Yes. The issue gives concrete config and commands, and current source provides a high-confidence static reproduction of the missing bridge: the Claude CLI path builds external CLI args/env without reading or enforcing `exec-approvals.json`; I did not run Claude Code live.

## Solution Assessment

Unclear. Forwarding the policy is the right end state, but the exact mechanism needs maintainer review against Claude Code's current permission API; a safer alternative may be to fail closed or warn for `claude-cli` plus `security: "allowlist"` until a real bridge exists.
