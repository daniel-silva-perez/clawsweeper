## Summary

Keep this issue open for now. Current main and the v2026.4.27 release tree appear to implement configurable image-tool timeouts, but the same-author fix PR #56454 is still open and explicitly claims to fix this issue, so conservative cleanup should reconcile that PR before closing the issue.

## What This Changes

The issue requests making the image tool timeout configurable because fixed OpenClaw-side timeout enforcement can abort slow image-analysis provider calls.

## Best Possible Solution

Reconcile #56454 against the released implementation: either close the PR as superseded or land any remaining useful compatibility/docs pieces, then close this issue as implemented once the issue/PR pair is no longer inconsistent.

## Reproduction Assessment

Yes. The original failure shape has a clear path: configure a slow image-capable provider, run the `image` tool with a request exceeding the configured/default timeout, and observe the shared runtime abort path; current main changes the timeout source so the fixed 30-second cutoff no longer matches the implementation.

## Solution Assessment

Yes. Using the existing media-understanding config surface, `tools.media.image.timeoutSeconds` plus per-model overrides, is a narrower and more consistent fix than adding a separate `agents.defaults.imageTimeoutSeconds` key.
