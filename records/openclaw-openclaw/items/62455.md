## Summary

Keep open. Current main still treats nested Discord channel `agentId` as an unrecognized key, while the supported route is top-level `bindings[]`. The existing doctor repair path can strip the unknown key generically, but that loses the intended per-channel route instead of migrating it or giving targeted guidance.

## What This Changes

The issue requests support, repair, or actionable guidance for Discord channel-level `agentId` entries that currently invalidate gateway config reloads.

## Best Possible Solution

Keep channel-local `agentId` out of the Discord runtime schema unless maintainers explicitly want that alias, and add a Discord-owned doctor compatibility path that safely migrates it to top-level route bindings or emits a targeted `bindings[]` migration hint.

## Reproduction Assessment

Yes. Current main can reproduce the validation failure by parsing the issue's config shape: the strict Discord guild channel schema lacks `agentId`, and gateway startup/reload still treat that invalid snapshot as invalid unless recovery applies.

## Solution Assessment

No. Generic unknown-key stripping is not the best fix because it can make the file valid while discarding the user's intended per-channel agent route; a targeted migration or explicit binding guidance is narrower and more maintainable.
