## Summary

Keep this PR open. Current main still has related usage infrastructure, Gateway RPCs, Control UI usage loading, chat `/usage cost`, and the narrower `openclaw gateway usage-cost`, but it does not implement the PR's intended top-level `openclaw usage` command with custom ranges, session ranking, by-model summaries, and JSON output.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is to rebase it onto current main and implement `openclaw usage` as a thin, tested CLI wrapper over the existing `usage.cost` and `sessions.usage` APIs, with Commander option tests for conflicting date flags, local/UTC semantics, complete aggregation before display limiting, shared token formatting, and JSON output. If maintainers decide `openclaw gateway usage-cost` plus the Control UI is sufficient, that should be an explicit product decision or a canonical follow-up rather than an automatic cleanup close.

## Risks / Open Questions

- As submitted, custom range totals can be wrong outside UTC because date-only `--from` and local-time `--to` parsing are mixed.
- As submitted, `--days` can silently override more explicit date range flags, which can produce surprising summaries.
- The submitted session/model breakdown can exclude older high-cost in-range sessions if it applies `--limit` before complete aggregation.
- Landing the PR without refit could create a second CLI aggregation path that drifts from the existing `usage.cost` and `sessions.usage` behavior.
