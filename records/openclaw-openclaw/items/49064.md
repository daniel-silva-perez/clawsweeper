## Summary

Keep this PR open. Current main has the underlying usage/cost RPCs, a Control UI usage view, chat `/usage cost`, and the narrower `openclaw gateway usage-cost` command, but it still does not implement the PR's intended top-level `openclaw usage` command with custom date ranges, session ranking, by-model summaries, and JSON output. The PR remains a plausible implementation candidate, but it needs rebase/refit to current usage APIs and should address the existing review blockers before maintainer consideration.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is to rebase it onto current main and implement `openclaw usage` as a thin CLI wrapper over `usage.cost` and `sessions.usage`, with production-module tests for Commander options, date conflict handling, local/UTC semantics, complete aggregation before display limiting, shared token formatting, and JSON output. If maintainers decide `openclaw gateway usage-cost` is sufficient, that should be made as an explicit product decision or tracked in a canonical follow-up, not handled as an automatic cleanup close.

## Risks / Open Questions

- As submitted, custom date ranges can be wrong outside UTC because the PR parses `--from` as a date-only UTC timestamp while `--to` is parsed as local end-of-day.
- The PR lets `--days` silently override more explicit range flags, which can produce surprising summaries.
- The proposed session/model breakdown path appears to use the display limit as an input scan limit, so older high-cost sessions in the selected range can be excluded from totals.
- Landing this as-is could create a second aggregation path that diverges from the existing `usage.cost` and `sessions.usage` Gateway behavior.
