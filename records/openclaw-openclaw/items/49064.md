## Summary

Keep this PR open. Current main still does not provide the proposed top-level `openclaw usage` command with custom date ranges, session rankings, by-model summaries, and JSON output. Main has related usage infrastructure, Control UI usage views, chat `/usage cost`, and `openclaw gateway usage-cost`, but those are narrower surfaces and do not make this CLI-first PR obsolete.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is to rebase it onto current main and make `openclaw usage` a thin, tested CLI surface over the existing `usage.cost` and `sessions.usage` infrastructure, with explicit date-option conflict handling, consistent timezone semantics, shared `formatTokenCount`/cost formatting, complete aggregation before display limiting, and JSON output. If maintainers decide the existing Control UI plus `openclaw gateway usage-cost` is enough, that should be recorded as an explicit product decision or canonical follow-up, not an automatic cleanup close.

## Risks / Open Questions

- Closing this PR would drop a still-relevant CLI surface while related cost/usage requests, including #13219, remain open or only partially satisfied by other surfaces.
- Landing the PR as submitted risks incorrect custom date boundaries outside UTC and surprising behavior when users combine `--days` with explicit date flags.
- Ranking output should aggregate over the full requested range before applying display limits; the current gateway API also has a limit-before-aggregate behavior that a CLI wrapper needs to account for.
- The test file in the submitted diff reimplements helper logic instead of exercising production helpers, so it may not catch regressions in the real command implementation.
- No unexplained security-sensitive or supply-chain changes were visible in the provided PR file list.
