## Summary

Keep this PR open. Current main still does not implement the PR’s intended top-level `openclaw usage` command with custom ranges, session ranking, by-model summaries, and JSON output. Main has strong underlying usage infrastructure, a Control UI Usage view, chat `/usage cost`, and the narrower `openclaw gateway usage-cost`, but those do not make the CLI-first contribution obsolete.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is to rebase it onto current main and implement `openclaw usage` as a thin, tested CLI surface over the existing `usage.cost` and `sessions.usage` infrastructure, with explicit date-option conflict handling, consistent timezone semantics, shared `formatTokenCount`/cost formatting, complete aggregation before display limiting, and JSON output. If maintainers decide `openclaw gateway usage-cost` plus the Control UI is enough, that should be an explicit product decision or a canonical follow-up, not an automatic cleanup close.

## Risks / Open Questions

- As submitted, custom range parsing can produce wrong boundaries outside UTC, and conflicting date flags can silently change the requested range.
- As submitted, session/model rankings can be wrong if `--limit` is used as a scan limit before complete aggregation.
- Current `sessions.usage` also aggregates over `limitedEntries`, so a CLI wrapper should either extend that API or request enough data and apply display limits after aggregation.
- Landing the PR without refit could create a second CLI aggregation path that drifts from the existing Gateway RPC and Control UI behavior.
