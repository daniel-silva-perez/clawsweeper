## Summary

Keep open for maintainer review. The PR targets a real current-main gateway startup regression, but the proposed resolver guard should not merge as-is because it also suppresses the existing inactive SecretRef warning path for disabled `web_fetch` surfaces with plugin-scoped credentials; overlapping #74921 appears to preserve that contract more carefully.

## What This Changes

The PR changes secrets fast-path detection, web-fetch provider-resolution branching, and regression tests so `tools.web.fetch.enabled: false` skips bundled/plugin web-fetch provider discovery.

## Best Possible Solution

Land one selected fix that makes `tools.web.fetch.enabled: false` a reliable startup escape hatch for credential-free disabled fetch config while preserving inactive SecretRef diagnostics for credential-bearing plugin or legacy fetch config; then close #74896 and close the duplicate implementation PR.

## Reproduction Assessment

Yes. #74896 provides a concrete config and startup trace, and current main source shows that a disabled `tools.web.fetch` key still bypasses the fast path and reaches awaited web-fetch provider discovery before gateway bind.

## Solution Assessment

No for this exact patch as written. The narrow startup fix is directionally right, but the resolver guard should distinguish credential-free disabled fetch config from disabled fetch surfaces that still need inactive SecretRef warnings, or adopt the safer shape from #74921.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve inactive SecretRef warnings for disabled fetch:** `src/secrets/runtime-web-tools.ts:685`
  - body: With `tools.web.fetch.enabled: false` and `plugins.entries.firecrawl.config.webFetch.apiKey` configured as a SecretRef, this guard skips the fetch resolver entirely. The existing disabled-fetch test expected `SECRETS_REF_IGNORED_INACTIVE_SURFACE`; after this patch no provider list is loaded, so `pushInactiveProviderCredentialWarnings()` never runs and operators get no warning that the configured SecretRef is ignored. Skip discovery only for credential-free disabled surfaces, or emit the warning without full discovery.
  - confidence: 0.88
