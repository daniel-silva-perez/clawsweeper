## Summary

Keep open. The PR targets a real current-main startup path, but the proposed guard is incomplete: disabled `tools.web.fetch` can still trigger web-fetch provider discovery when plugin-scoped `webFetch` config is present, which is exactly the class of slow pre-bind work this fix is meant to avoid.

## What This Changes

The PR changes secrets runtime fast-path detection and runtime web-fetch resolution so `tools.web.fetch.enabled: false` skips web-fetch provider discovery, with regression tests for the disabled-fetch path.

## Best Possible Solution

Update the guard so an explicitly disabled top-level web_fetch policy suppresses web-fetch provider discovery even when stale or plugin-scoped `webFetch` config exists, while preserving active provider SecretRef resolution when fetch is enabled or not explicitly disabled.

## Reproduction Assessment

Yes. #74896 provides concrete config, version, startup trace boundary, and a removal workaround, and current main’s source explains the pre-bind path from secrets fast-path bypass into provider discovery.

## Solution Assessment

No, not yet. The direction is narrow and in the right module, but the resolver-level guard should apply to plugin-scoped webFetch config as well when `tools.web.fetch.enabled === false`.
