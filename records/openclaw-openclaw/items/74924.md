## Summary

Keep open for maintainer review. Current main still takes the slow secrets/runtime web-fetch discovery path whenever `tools.web.fetch` is present, while the latest PR head now appears to address the earlier plugin-scoped `webFetch` edge by guarding both top-level fetch config and plugin webFetch config when fetch is explicitly disabled. I did not find a blocking correctness or security issue in the latest diff, but maintainers still need to choose between this PR and the overlapping #74921 and verify exact-head CI.

## What This Changes

The PR changes secrets runtime fast-path detection, runtime web-fetch provider resolution, and regression tests so `tools.web.fetch.enabled: false` skips web-fetch provider discovery, including plugin-scoped `webFetch` config.

## Best Possible Solution

Land one narrow fix that makes `tools.web.fetch.enabled: false` a reliable gateway-startup escape hatch by skipping web-fetch provider discovery before bind, while keeping active SecretRef resolution intact when fetch is enabled or not explicitly disabled; then close the remaining duplicate PR and the linked bug after merge.

## Solution Assessment

Yes for the reported disabled-fetch hang. The latest PR keeps the change in the secrets runtime/web-tools boundary and, after the second commit, suppresses plugin-scoped webFetch provider discovery when the top-level fetch surface is explicitly disabled.
