## Summary

Keep this PR open for maintainer review. Current main still routes an explicitly disabled `tools.web.fetch` surface into the secrets slow path and web-fetch provider discovery, while this PR is an active implementation with clean exact-head checks and an overlapping open alternative in #74921 that needs a maintainer choice.

## What This Changes

The PR changes secrets runtime fast-path detection, runtime web-fetch provider resolution, and regression tests so `tools.web.fetch.enabled: false` skips web-fetch provider discovery.

## Best Possible Solution

Land one selected fix that makes `tools.web.fetch.enabled: false` a reliable gateway-startup escape hatch before provider discovery, preserves the intended SecretRef inactive-surface behavior, then close #74896 and close the duplicate implementation PR.

## Reproduction Assessment

Yes. #74896 gives a concrete Linux/Docker startup trace and disabled-fetch config, and current main source shows that config still bypasses the secrets fast path and awaits web-fetch provider discovery before the gateway binds.
