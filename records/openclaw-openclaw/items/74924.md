## Summary

Keep this PR open for maintainer review. Current main still has the disabled `tools.web.fetch` startup-discovery bug, and this PR is a plausible narrow fix, but it overlaps open #74921 and changes the disabled-fetch inactive SecretRef diagnostic behavior enough that a maintainer should choose the intended semantics before merge.

## What This Changes

The PR changes secrets runtime fast-path detection, web-fetch provider resolution, and regression tests so `tools.web.fetch.enabled: false` skips web-fetch provider discovery.

## Best Possible Solution

Land one selected fix that makes `tools.web.fetch.enabled: false` a reliable gateway-startup escape hatch before provider discovery, while preserving the intended SecretRef inactive-surface diagnostics contract; then close #74896 and close the duplicate implementation PR.

## Reproduction Assessment

Yes. #74896 gives a concrete Linux/Docker startup trace and config workaround, and current main source shows the disabled fetch key still routes secrets activation into web-fetch provider discovery before gateway bind.

## Solution Assessment

Unclear whether this exact PR is the best final shape. The patch is a reasonable narrow fetch-only fix for the reported hang, but #74921 covers a broader disabled web-tool surface and credential-bearing diagnostic semantics that need maintainer choice.
