## Summary

Keep this PR open for maintainer review, but do not merge it as-is. Current main still has the disabled-fetch startup path described by #74896, and this PR is a narrow implementation candidate, but its resolver guard is too broad because it drops the existing inactive SecretRef warning behavior for plugin-scoped web-fetch credentials; it also overlaps with #74921, so maintainers should choose or fold the fixes.

## What This Changes

The PR changes secrets runtime fast-path detection, runtime web-fetch provider resolution, and regression tests so `tools.web.fetch.enabled: false` skips web-fetch provider discovery.

## Best Possible Solution

Land one selected fix that makes credential-free `tools.web.fetch.enabled: false` skip web-fetch provider discovery during gateway startup, while preserving inactive SecretRef diagnostics for credential-bearing plugin-scoped fetch config; then close #74896 and close or supersede the overlapping implementation PR.

## Reproduction Assessment

Yes. #74896 provides a concrete Linux/Docker startup trace and disabled-fetch config, and current main source still shows that config bypassing the secrets fast path and awaiting web-fetch provider discovery before the gateway reaches later startup markers.

## Solution Assessment

No for this exact patch as currently shaped. The fast-path direction is right, but the resolver-level guard should be narrower so plugin-scoped or credential-bearing disabled fetch config can still emit inactive SecretRef warnings without resolving secrets.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve inactive fetch SecretRef warnings:** `src/secrets/runtime-web-tools.ts:683-686`
  - body: When `tools.web.fetch.enabled` is false but plugin-scoped `webFetch.apiKey` is still configured as a SecretRef, current main still discovers provider metadata enough to emit `SECRETS_REF_IGNORED_INACTIVE_SURFACE` without resolving the secret. This guard skips the whole fetch branch whenever top-level fetch is disabled, including the `hasPluginWebFetchConfig` case, and the test removes that warning assertion, so users lose the diagnostic for an ignored SecretRef. Keep the startup fast path for credential-free disabled fetch, but preserve the credential-bearing plugin-config warning path.
  - confidence: 0.86
