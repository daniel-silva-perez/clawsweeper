## Summary

Keep this PR open. Current main has some shared env-proxy machinery, including protocol-aware proxy selection and NO_PROXY fallback, but it still does not implement the PR's central user-facing change: a default-off `tools.web.fetch.useTrustedEnvProxy` config option wired into built-in `web_fetch`. Main also still lacks the PR's trusted-env-proxy hostname-policy hardening before `EnvHttpProxyAgent` dispatch.

## Best Possible Solution

Keep this PR open as the canonical implementation candidate for proxy-only `web_fetch` support. The best path is a maintainer review of the latest branch, preserving strict default behavior while landing a default-off `tools.web.fetch.useTrustedEnvProxy` config field wired from schema/docs through `createWebFetchTool` into `fetchWithWebToolsNetworkGuard`, with hostname-policy checks before trusted proxy dispatch and regression tests for default strict mode, matching proxy env vars, no matching proxy, and NO_PROXY fallback.

- This PR changes a security-sensitive `web_fetch` network boundary: with the opt-in enabled, local post-DNS IP pinning is delegated to an operator-controlled proxy, so maintainers should explicitly review the trust model, naming, docs, and tests before merge.
- The provided PR file list touches source, config schema, generated config metadata, docs, and tests. It does not show workflow, dependency, lockfile, install/build/release script, or package publishing changes, so the security review focus is SSRF/network behavior rather than supply-chain drift.
- Firecrawl/provider fallback behavior is intentionally left as a scope boundary in the latest author comment; maintainers should decide whether provider API calls should share this opt-in or remain a follow-up.
