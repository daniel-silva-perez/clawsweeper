## Summary

Keep this PR open. Current main includes the shared trusted-env-proxy guard machinery and protocol-aware proxy matching, but it still does not implement this PR's central user-facing change: an explicit `web_fetch` config opt-in that passes trusted env proxy mode into the untrusted `web_fetch` path. The current tests and docs still describe/preserve strict DNS pinning for normal `web_fetch` even when `HTTP_PROXY` is configured. Related cleanup also points here rather than replacing it: #63565 and #71056 were closed as superseded by this active PR, while #59007 only merged the narrower DNS-before-proxy ordering fix.

## Best Possible Solution

Keep this PR open as the canonical implementation/review thread for proxy-only `web_fetch` support. The best path is to land an explicit, default-off config field wired through `createWebFetchTool` into `fetchWithWebToolsNetworkGuard`, keep protocol-aware proxy and `NO_PROXY` fallback behavior aligned with Undici, preserve pre-dispatch hostname policy checks, update public docs/config schema, and retain focused regression tests for default strict mode, opt-in trusted proxy mode, ALL_PROXY-only fallback, and NO_PROXY fallback.

- The current PR diff in the provided context focuses on built-in `web_fetch`; maintainers should decide whether provider fallback paths such as Firecrawl also need trusted env proxy behavior in proxy-only deployments.
