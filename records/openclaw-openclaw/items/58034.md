## Summary

Current main does not implement this PR's central user-facing change. The shared trusted-env-proxy guard machinery exists, and main has protocol-aware proxy matching, but `web_fetch` still does not expose or pass a default-off `tools.web.fetch.useTrustedEnvProxy` opt-in. Main also still lacks this PR's trusted-env-proxy hostname-policy precheck and the PR's documented config/schema surface, so this PR should remain open for maintainer review.

## Best Possible Solution

Keep this PR open as the canonical implementation and review thread for proxy-only `web_fetch` support. The best path is to land a default-off `tools.web.fetch.useTrustedEnvProxy` config field wired through `createWebFetchTool` and `runWebFetch` into `fetchWithWebToolsNetworkGuard`, keep shared proxy matching aligned with Undici HTTP(S) proxy and NO_PROXY semantics, preserve pre-dispatch hostname policy checks, update public docs/config schema, and retain focused regression tests for strict default mode, explicit trusted-proxy mode, ALL_PROXY-only non-activation, and NO_PROXY fallback.

## Risks / Open Questions

- This PR changes a security-sensitive network boundary: when enabled, local post-DNS IP pinning is delegated to an operator-controlled HTTP(S) proxy. Maintainers should review the setting name, default-off behavior, hostname policy checks, NO_PROXY fallback, and documentation before merging.
- The PR touches generated config baseline files plus schema/docs; those should be regenerated and checked in CI or Testbox before landing. This read-only review did not run write-capable generation or broad test gates.
- The current PR focuses on built-in `web_fetch`; maintainers should decide whether provider fallback paths such as Firecrawl need separate proxy-only deployment behavior.
