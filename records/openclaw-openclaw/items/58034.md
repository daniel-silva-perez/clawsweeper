## Summary

Keep open for maintainer review. Current main still lacks the explicit `web_fetch` trusted-env-proxy opt-in and docs, so the PR is not superseded by #59007; however, the branch is currently dirty against main, has failing remote checks, and needs a rebase that preserves newer web_fetch SSRF policy fields before it is mergeable.

## What This Changes

The PR adds a default-off `tools.web.fetch.useTrustedEnvProxy` config/docs surface, threads it into `web_fetch`, adds hostname-policy checks before trusted env-proxy dispatch, aligns NO_PROXY handling with Undici, and expands regression coverage plus generated config metadata.

## Best Possible Solution

Rebase the branch onto current main, preserve both existing fake-IP SSRF policy flags while adding `useTrustedEnvProxy`, keep the opt-in default-off with hostname policy and NO_PROXY fallback tests, then rerun focused web_fetch/fetch-guard/config-doc checks plus the normal changed gate before merge.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes: run `web_fetch` in a proxy-only environment where local UDP DNS is blocked, set HTTP_PROXY/HTTPS_PROXY, and compare current main's strict DNS-pinned failure with the PR's explicit `tools.web.fetch.useTrustedEnvProxy: true` path; the PR discussion also includes independent proxy-only sandbox verification.

## Solution Assessment

Is this the best way to solve the issue? Not as-is, because the branch is dirty and predates current main's IPv6 fake-IP policy work; the design direction is narrow and maintainable if rebased cleanly with those current config/schema/runtime fields preserved.
