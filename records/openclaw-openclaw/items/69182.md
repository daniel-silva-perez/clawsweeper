## Summary

Keep open. Current main still runs Active Memory as a bounded but blocking `before_prompt_build` recall pass, so eligible turns can still wait on the configured recall budget before first model output. Recent fixes added hard timeout behavior, setup grace handling, partial timeout recovery, recall-tool wiring, and a circuit breaker after repeated timeouts, but none implements non-blocking/background recall or lowers/auto-tunes the default latency budget.

## What This Changes

Request that Active Memory avoid delaying chat replies by the full configured recall timeout, either by becoming non-blocking, failing open faster, or using lower/auto-tuned latency defaults.

## Best Possible Solution

Keep this open as the focused Active Memory first-output latency follow-up. Maintainers should choose the intended product direction: explicit non-blocking/background recall, lower or auto-tuned timeout defaults, stronger fail-open behavior, or clearer low-latency presets/docs while preserving the current blocking mode for users who want it. The implementation should stay in `extensions/active-memory` unless the chosen design exposes a genuinely generic plugin hook capability.
