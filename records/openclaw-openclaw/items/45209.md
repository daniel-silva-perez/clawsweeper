## Summary

Keep open. Current main and the latest release still lack the Feishu Bitable tool-category toggle, and the PR is the right Feishu-owned shape, but the branch is not merge-ready because adding the new required config key leaves typed Feishu tool-config mocks incomplete and it still needs focused coverage plus public docs/changelog follow-through.

## What This Changes

The PR adds a Feishu `tools.bitable` boolean to the config type, schema, defaults, account merge helper, and Bitable tool registration so `feishu_bitable_*` tools stay enabled by default but can be skipped when disabled.

## Best Possible Solution

Land or replace this with a Feishu-plugin-owned opt-out toggle that defaults Bitable tools on, skips Bitable registration only when all enabled Feishu accounts disable it, and includes focused schema/default/registration tests, updated typed mocks, public Feishu docs, and a credited changelog entry.

## Reproduction Assessment

Yes. Source inspection on current main is enough: the Feishu config surfaces omit `bitable`, and Bitable registration proceeds after enabled-account discovery with no category guard. A focused runtime reproduction would configure `channels.feishu.tools.bitable: false` and observe that `feishu_bitable_*` tools are still registered on current main.

## Solution Assessment

Mostly yes. Mirroring the existing Feishu tool-category toggle pattern inside the Feishu plugin is the narrowest maintainable solution; the safer merge path is to keep the PR's scope but repair typed mocks, add disabled-registration coverage, and document the new config key before landing.
