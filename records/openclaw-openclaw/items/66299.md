## Summary

Keep this open. Current main still returns the explicit-token failure before attempting the bound-thread ACP resolver, so the reported `/acp close <non-session-token>` path inside a bound thread remains broken. The closed sibling PR #66338 contains useful prior work but did not land, and current tests cover bound-thread close only when no bad token is present.

## What This Changes

Fix ACP target resolution so an unresolved explicit/native token can fall through to the current bound-thread ACP session before failing.

## Best Possible Solution

Land a focused resolver change in `targets.ts`: keep successful explicit target resolution first, try `resolveBoundAcpThreadSessionKey` when token resolution returns null, preserve the explicit bad-token error when no bound thread resolves, and add regression coverage for both the bound-thread bad-token success path and the no-binding bad-token error path.

## Repair Work Prompt

Fix the ACP target resolver bug from this issue. In `src/auto-reply/reply/commands-acp/targets.ts`, update `resolveAcpTargetSessionKey` so a non-empty token still resolves by key/sessionId/label first, but if `resolveSessionKeyByToken` returns null it tries `resolveBoundAcpThreadSessionKey(params.commandParams)` before returning `Unable to resolve session target: <token>`. Preserve the explicit bad-token error when no bound thread target resolves, and do not silently route a bad explicit token to the requester-session fallback unless maintainers explicitly choose that behavior. Add focused regression coverage in `src/auto-reply/reply/commands-acp.test.ts`: `/acp close <non-session-token>` inside a mocked bound Discord thread should close the bound ACP session and unbind it, while the same bad token without a bound thread should keep the existing warning/error and should not close/unbind. Use the existing helpers around `mockBoundThreadSession`, `runThreadAcpCommand`, `hoisted.callGatewayMock`, and the close tests near the current bound-thread close coverage. Credit #66338 as prior useful source work in the PR body if creating a replacement PR. Changelog is probably not needed unless maintainers want user-facing ACP bugfix notes.
