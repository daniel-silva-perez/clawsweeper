## Summary

Keep this open. Current main still returns the explicit unresolved-token error before reaching the ACP bound-thread resolver, the latest release has the same resolver shape, and the issue discussion has converged on a narrow Plan A fix rather than a close reason.

## What This Changes

The issue asks ACP session-targeting commands to fall through to current thread binding when an explicit token is present but cannot be resolved as a session key, id, or label.

## Best Possible Solution

Land the Plan A resolver behavior: successful explicit token resolution remains first, unresolved explicit tokens try resolveBoundAcpThreadSessionKey next, the explicit bad-token diagnostic is preserved when no bound thread target resolves, and regression coverage proves both the bound-thread success path and no-binding error path.

## Repair Work Prompt

Fix the ACP target resolver bug reported in openclaw/openclaw#66299. In `src/auto-reply/reply/commands-acp/targets.ts`, keep successful explicit token resolution first, but when `resolveSessionKeyByToken(token)` returns null, fall through to `resolveBoundAcpThreadSessionKey(params.commandParams)` before returning the explicit bad-token error. Preserve the `Unable to resolve session target: ${token}` diagnostic when a token was provided and neither a bound thread nor requester fallback resolves. Do not add a config flag or broaden ACP/plugin APIs. Add focused Vitest coverage in `src/auto-reply/reply/commands-acp.test.ts`, anchored near the existing bound-thread close tests, for `/acp close <non-session-token>` inside a bound thread calling `closeSession` and `unbind`; also cover the no-binding bad-token error path if not already covered. Add a user-facing changelog fix entry under the active release section with `Thanks @kindomLee` and appropriate additional credits if desired. Use #66338 only as prior context; its old shape dropped the bad-token diagnostic and should not be copied blindly.
