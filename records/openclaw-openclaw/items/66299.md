## Repair Work Prompt

Fix the ACP target resolver bug described in openclaw/openclaw#66299. In `src/auto-reply/reply/commands-acp/targets.ts`, keep successful explicit token resolution first, but when `resolveSessionKeyByToken(token)` returns null, fall through to `resolveBoundAcpThreadSessionKey(params.commandParams)` before returning the explicit `Unable to resolve session target: ${token}` error. Preserve the existing bad-token diagnostic when no bound thread target resolves; do not add a config flag and do not change ACP dispatch policy.

Add regression coverage in `src/auto-reply/reply/commands-acp.test.ts` near the existing bound-thread close tests. Cover `/acp close <non-session-token>` inside a bound thread where `sessions.resolve` does not produce a key, and assert `closeSession` and `unbind` run for the bound ACP session. Also cover the no-binding bad-token case so the warning remains `Unable to resolve session target: <token>` and close/unbind do not run.

Because this is user-facing, add a single-line `CHANGELOG.md` entry under `## Unreleased` / `### Fixes` with attribution, e.g. thanking @kindomLee and the relevant follow-up contributors. Do not touch unrelated ACP runtime behavior, Discord command registration, or dependency/config surfaces.
