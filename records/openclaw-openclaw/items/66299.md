## Summary

Keep this issue open. Current main still returns the explicit bad-token error before trying the thread-bound ACP resolver, so the reported `/acp close <non-session-token>` path inside a bound thread remains broken. The related fix PR #66338 was closed unmerged, and current tests cover bound-thread ACP actions only when no bad target token is supplied.

## Best Possible Solution

Keep this issue open and land a focused resolver fix in `src/auto-reply/reply/commands-acp/targets.ts`: when explicit token resolution returns null, try `resolveBoundAcpThreadSessionKey` before returning the explicit-token error. Preserve the existing bad-token diagnostic when no bound thread target resolves, and add regression coverage for `/acp close <non-session-token>` inside a bound thread plus the no-binding bad-token case.

## Evidence

- **Resolver still short-circuits before bound-thread fallback:** `resolveAcpTargetSessionKey` normalizes a non-empty token, calls `resolveSessionKeyByToken`, and immediately returns `Unable to resolve session target: ${token}` when it cannot resolve. The call to `resolveBoundAcpThreadSessionKey(params.commandParams)` is below that return, so it is unreachable for the reported bad-token-in-thread case.
