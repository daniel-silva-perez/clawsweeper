## Summary

Keep open. Current main still sends all generated ACP runtime config options when a backend exposes `session/set_config_option` but does not advertise `configOptionKeys`, which is exactly the behavior this PR is trying to change for Claude ACP. The PR is not obsolete, but it needs maintainer review because it lacks regression tests and its patch also changes the advertised-key path from throwing on unsupported keys to silently filtering them.

## Best Possible Solution

Keep this PR open for normal maintainer review. The best fix is a tested ACP runtime-control change that gracefully avoids unsupported generic config IDs for strict adapters, preserves or deliberately revises the current capability contract, and adds regression coverage for both undefined `configOptionKeys` and advertised-key filtering. Maintainers should also decide whether Claude `thinking` should map to adapter-specific `effort` instead of being dropped.

## Risks / Open Questions

- The PR has no regression test for a backend with undefined `configOptionKeys`, so the intended behavior is not locked down.
- The patch changes behavior for advertised key lists too: unsupported generated keys are silently filtered instead of raising `ACP_BACKEND_UNSUPPORTED_CONTROL`, which may or may not be the intended contract.
- Skipping all config options when keys are undefined avoids Claude failures but also drops useful `model` or `thinking` controls unless a handle-specific capability/key discovery path is added later.
- The current runtime type documentation says undefined/empty `configOptionKeys` is permissive; implementation, docs, and tests need to be aligned if this PR’s semantics are accepted.
