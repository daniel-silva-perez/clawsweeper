## Summary

Keep open. Current main still has the reported ACP runtime-control behavior, and this PR is a relevant implementation candidate for the linked Claude ACP failure, but the patch should not merge as-is because it silently drops requested options when a backend does advertise a strict key list.

## What This Changes

The PR changes ACP runtime control setup to send session config options only when the backend advertises matching config option keys, skipping all config options when no key list is advertised.

## Best Possible Solution

Land a tested ACP runtime-control fix that avoids sending generic OpenClaw config IDs to strict adapters like Claude ACP, while preserving a clear failure or warning when a backend advertises a strict key list and a requested runtime option is unsupported. The contract docs and tests should be updated with the intended meaning of missing `configOptionKeys`.

## Reproduction Assessment

Yes. The current-main code path can be reproduced with a runtime that supports `session/set_config_option` and returns no `configOptionKeys`, plus persisted runtime options for `model`, `thinking`, `approval_policy`, or `timeout`; the manager will call `setConfigOption` for each. The linked Claude ACP issue adds live adapter steps, though I did not run live ACP during this read-only review.

## Solution Assessment

No. The PR targets the right area, but the implementation is not the best fix because it changes the advertised-key path from explicit rejection to silent filtering and leaves the undefined-key capability contract and tests unresolved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve errors for advertised-key mismatches:** `src/acp/control-plane/manager.runtime-controls.ts:99-108`
  - body: When a backend advertises a non-empty `configOptionKeys` list, this patch now filters unsupported requested keys instead of raising `ACP_BACKEND_UNSUPPORTED_CONTROL`. That means a configured `approval_policy`, `timeout`, or similar control can be ignored while the session proceeds as if the runtime accepted it. Keep the clear failure or add an explicit warning for non-empty advertised lists, and limit the graceful skip to the no-key-list case if that contract is approved.
  - confidence: 0.92

  - confidence: 0.8

- The PR conflicts with the current `configOptionKeys` contract that treats undefined/empty key lists as permissive, so maintainers should explicitly decide whether that contract is changing.
- The patch has no regression coverage for a backend that exposes `session/set_config_option` but omits `configOptionKeys`.
