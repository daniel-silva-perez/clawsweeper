## Summary

Keep open. Current main and the latest release still use a direct rename-based SQLite atomic reindex swap in memory-core, the issue has multiple Windows reproduction reports, and the provided context shows open PR #71611 as the unmerged closing implementation candidate.

## What This Changes

Fix Windows memory search/index sync failures when memory-core's SQLite atomic reindex swap hits EBUSY/EPERM/EACCES file-lock errors.

## Best Possible Solution

Keep this issue open and use #71611, or a revised replacement, to fix memory-core's atomic reindex move with bounded Windows lock-error handling, careful temp/backup cleanup, sidecar preservation, and regression coverage for EBUSY/EPERM/EACCES.

## Reproduction Assessment

Yes. The issue body and later comments give concrete Windows command sequences and logs, and current main still contains the same rename path that would throw on EBUSY; this review did not run Windows locally, but the source and discussion line up closely.

## Solution Assessment

Unclear for the exact PR implementation until it is reviewed, but the fix belongs in memory-core's atomic reindex path. Reusing the existing retry/backoff pattern is plausible, with extra care for SQLite sidecars, cleanup, and cases where a live gateway holds the file open.
