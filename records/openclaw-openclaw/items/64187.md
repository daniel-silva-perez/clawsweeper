## Summary

Keep open. Current main and the latest release still use a direct rename-based SQLite atomic reindex swap in memory-core, while the issue has multiple Windows reproduction reports and an open linked fix PR that has not merged.

## What This Changes

The issue reports that Windows memory search/index sync can fail with EBUSY during memory-core's SQLite atomic reindex swap and asks for graceful retry or recovery around locked-file rename operations.

## Best Possible Solution

Review, land, or revise PR #71611 so memory-core's atomic reindex move has bounded Windows lock-error handling for the main database and sidecars, keeps rollback semantics intact, cleans stale temp/backup files safely, and adds regression coverage for EBUSY/EPERM/EACCES.

## Reproduction Assessment

Yes. The issue and comments give concrete Windows commands and logs, and current main still contains the direct fs.rename() path that would throw on EBUSY during the same swap sequence.

## Solution Assessment

Unclear for the exact PR implementation until it is reviewed, but the repair belongs in memory-core's atomic reindex helper rather than user config. The existing cron-store retry pattern is a reasonable starting point if adapted carefully for SQLite sidecars and rollback cleanup.
