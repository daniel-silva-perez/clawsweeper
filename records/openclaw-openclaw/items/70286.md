## Summary

Keep open. Current `main` still returns `queuedFinal: false` for a plugin-bound `inbound_claim` handled outcome, Telegram still sends the silent-reply fallback when that false signal combines with no immediate delivery, and open PR #70288 still references this issue with closing syntax, so this should not be closed before the PR or an equivalent fix lands.

## What This Changes

The issue requests that plugin-owned async inbound claims returning `{ handled: true }` suppress Telegram's silent-reply fallback until the plugin has a chance to send its response.

## Best Possible Solution

Land PR #70288 after updating it, or land an equivalent maintainer patch, so a plugin-owned async inbound claim creates a durable response-committed signal that Telegram respects without changing missing-handler, declined, or error fallback behavior. Add coverage at the direct dispatch, wrapper/finalizer, and Telegram fallback levels.

## Reproduction Assessment

Yes. A plugin-owned binding whose `inbound_claim` returns `{ handled: true }` currently yields `queuedFinal: false` and no immediate delivery, which satisfies Telegram's silent fallback condition before the async plugin response arrives.

## Solution Assessment

No, current `main` is not solved. The proposed `queuedFinal: true` direction is plausible, but the maintainable fix should also handle `finalizeDispatchResult` so a zero-count async claim is not normalized back to false.
