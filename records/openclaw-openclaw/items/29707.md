## Summary

Keep this issue open. Current main still explicitly excludes direct Amazon Nova Bedrock model IDs from cache-retention eligibility and then forces those requests through the Bedrock no-cache wrapper, while current AWS Bedrock docs now describe Nova-compatible prompt caching on `system` and `messages` with `cachePoint` checkpoints.

## What This Changes

Request enabling native prompt caching for Amazon Nova models on Amazon Bedrock by allowing supported Nova calls to use cache retention and Nova-compatible Bedrock cache markers.

## Best Possible Solution

Implement the remaining behavior inside the bundled Amazon Bedrock provider and its provider-owned cache semantics: recognize AWS-supported Nova IDs, pass explicit cache retention where valid, emit Bedrock/Nova-compatible cache checkpoints only in supported request locations, keep unsupported non-Anthropic Bedrock models on the no-cache path, and update tests, docs, and changelog together.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: the unit-tested direct Nova path resolves cache retention to `undefined`, then the provider wrapper forwards `cacheRetention: "none"` for `amazon.nova-micro-v1:0`.

## Solution Assessment

Yes. A focused provider-owned repair is the best path because the gap is specific to Amazon Bedrock model-family semantics and AWS prompt-cache rules; moving Nova IDs or TTL policy into generic core would be broader than needed.
