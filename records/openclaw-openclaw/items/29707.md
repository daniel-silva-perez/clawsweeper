## Summary

Keep this issue open. Current main still drops explicit `cacheRetention` for direct Amazon Nova Bedrock IDs, wraps direct Nova streams with `cacheRetention: "none"`, has tests that assert that behavior, and documents Nova/Mistral cache settings as forced off. AWS Bedrock’s current prompt-caching docs list Amazon Nova models as supported on `system` and `messages`, so this remains a valid bundled amazon-bedrock provider gap.

## Best Possible Solution

Implement a focused bundled amazon-bedrock plugin/runtime fix: recognize supported Amazon Nova direct and region-prefixed Bedrock model IDs as cache-eligible, preserve explicit `cacheRetention` for Nova, avoid sending unsupported 1-hour TTL to Nova, inject Bedrock/Nova-compatible `cachePoint` blocks on system and the last user message, keep unsupported non-Anthropic Bedrock models forced to no-cache, and update tests, prompt-caching docs, and changelog.

## Clownfish Work Prompt

Implement a focused fix for native Amazon Nova prompt caching on Bedrock. Current main drops explicit `cacheRetention` for direct Nova IDs and wraps direct Nova streams with `cacheRetention: "none"`. Add a Nova-family eligibility path for supported direct and region-prefixed Bedrock model IDs such as `amazon.nova-micro-v1:0`, `amazon.nova-lite-v1:0`, `amazon.nova-pro-v1:0`, `amazon.nova-premier-v1:0`, `amazon.nova-2-lite-v1:0`, and region/system prefixes like `us.amazon.nova-*`, `eu.amazon.nova-*`, or `global.amazon.nova-*` if those are valid in existing Bedrock model-ref handling. Preserve the existing no-cache wrapper for unsupported non-Anthropic/non-Nova Bedrock models such as Mistral.

Expected implementation boundary: `extensions/amazon-bedrock/register.sync.runtime.ts` should recognize Nova as cache-eligible and inject Bedrock Converse `cachePoint` blocks on `system` and the last user message using the existing payload-patch pattern where appropriate. `src/agents/pi-embedded-runner/prompt-cache-retention.ts` and/or `src/agents/pi-embedded-runner/anthropic-family-cache-semantics.ts` should preserve explicit `cacheRetention` for Nova Bedrock models without treating Nova as Anthropic. Do not send `ttl: "1h"` to Nova unless the AWS docs/source currently prove Nova supports 1-hour TTL; if `cacheRetention: "long"` is configured for Nova, clamp to the default 5-minute Bedrock cache point or document/test the chosen conservative behavior. Do not move provider-specific Nova policy into generic core except for a small generic seam if truly needed.

Update tests in `extensions/amazon-bedrock/index.test.ts` and `src/agents/pi-embedded-runner/extra-params.cache-retention-default.test.ts` to cover direct Nova no longer forced to none, explicit retention preserved, cachePoint injection on system/messages, long-retention TTL safety, region-prefixed Nova IDs, and non-Nova non-Anthropic models still forced to no-cache. Update `docs/reference/prompt-caching.md` and add a user-facing changelog entry with reporter/contributor credit. Do not change unrelated provider caching behavior, dependency versions, or bundled provider discovery semantics.

## Risks / Open Questions

- Nova currently appears to support the default 5-minute Bedrock cache, not the 1-hour TTL used by `makeCachePoint` for `cacheRetention: "long"`; a fix should clamp or map long retention conservatively unless AWS docs/source prove 1-hour Nova support.
- A blanket removal of `createBedrockNoCacheWrapper` would likely regress unsupported non-Anthropic Bedrock models and revive #20815-style extraneous `cachePoint` failures.
- Direct Nova support should use an explicit Nova allowlist/family check and Bedrock-compatible `cachePoint` injection on `system` and `messages`, while preserving existing Claude and application-inference-profile behavior.
