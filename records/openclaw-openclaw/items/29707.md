## Summary

Keep this issue open. Current main still treats direct Amazon Nova Bedrock model IDs as non-cache-eligible, wraps them with `cacheRetention: "none"`, has tests that assert that behavior, and documents Nova/Mistral cache settings as forced off. AWS Bedrock now documents Amazon Nova prompt caching and `cachePoint` usage for Nova, so this remains a valid bundled amazon-bedrock provider gap. ([docs.aws.amazon.com](https://docs.aws.amazon.com/bedrock/latest/userguide/prompt-caching.html))

## Best Possible Solution

Implement a focused bundled `amazon-bedrock` plugin/runtime fix: recognize supported Amazon Nova direct and region/system-prefixed Bedrock model IDs as cache-eligible, preserve explicit `cacheRetention`, inject Bedrock/Nova-compatible `cachePoint` blocks on the system prompt and last user message, avoid unsupported Nova 1-hour TTL, keep unsupported non-Anthropic Bedrock models on `cacheRetention: "none"`, and update tests, prompt-caching docs, and changelog.

## Repair Work Prompt

Implement native prompt-caching support for supported Amazon Nova models in the bundled `amazon-bedrock` provider. Preserve explicit `cacheRetention` for direct and region/system-prefixed Nova IDs such as `amazon.nova-micro-v1:0`, `amazon.nova-lite-v1:0`, `amazon.nova-pro-v1:0`, `amazon.nova-premier-v1:0`, `amazon.nova-2-lite-v1:0`, and corresponding `us.`/`global.` style Bedrock refs. Do not make a broad core migration and do not change unrelated providers.

Expected boundary:
- Add a provider-family helper or equivalent targeted logic so `resolveCacheRetention` keeps explicit `none`/`short`/`long` for supported Nova Bedrock IDs, while unsupported non-Anthropic Bedrock models still resolve to no cache.
- Update `extensions/amazon-bedrock/register.sync.runtime.ts` so direct Nova IDs are not sent through `createBedrockNoCacheWrapper` when caching is explicitly enabled.
- Inject Bedrock/Nova-compatible `cachePoint` blocks for Nova on the system prompt and last user message when `cacheRetention` is active, reusing the existing Bedrock payload patch pattern where appropriate.
- Avoid unsupported Nova 1-hour TTL. Treat `long` conservatively for Nova, such as using the default 5-minute cache point without `ttl: "1h"`, and document that behavior.
- Preserve the protective no-cache behavior for non-cache-capable non-Anthropic Bedrock models and keep #20815’s class of request rejection from regressing.
- Update `extensions/amazon-bedrock/index.test.ts`, `src/agents/pi-embedded-runner/extra-params.cache-retention-default.test.ts`, prompt-caching docs, and `CHANGELOG.md` with a user-facing fix entry and proper attribution to the reporter.

Do not add dependencies, do not touch package manager metadata, and do not broaden provider APIs unless a narrow helper is necessary.
