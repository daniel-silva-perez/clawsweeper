## Summary

Keep this open. Current main still blocks direct Amazon Nova Bedrock model IDs from cache-retention eligibility and forces non-Anthropic Bedrock streams through `cacheRetention: "none"`, while current AWS Bedrock documentation supports explicit Nova prompt caching with `cachePoint` checkpoints in `system` and `messages`.

## What This Changes

Request native Amazon Nova prompt caching in the bundled Amazon Bedrock provider by allowing Nova `cacheRetention` handling and Bedrock-compatible `cachePoint` checkpoints instead of forcing no-cache behavior.

## Best Possible Solution

Add Nova-specific prompt-cache support inside the bundled Amazon Bedrock provider: recognize supported direct, regional, and global Nova model IDs; preserve valid explicit cache retention; inject only Bedrock/Nova-compatible `cachePoint` checkpoints in supported fields; keep unsupported non-Anthropic Bedrock models on the no-cache path; and update tests, docs, and changelog together.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: `resolveCacheRetention({ cacheRetention: "long" }, "amazon-bedrock", "openai-completions", "amazon.nova-micro-v1:0")` returns `undefined`, and the Amazon Bedrock plugin wraps direct Nova IDs with the no-cache wrapper.

## Solution Assessment

Yes. A focused Amazon Bedrock provider/plugin repair is the narrowest maintainable solution because the remaining gap is provider-family semantics and Bedrock cache-point placement, not a broad core cache contract change.

Update focused tests for `resolveCacheRetention`, Amazon Bedrock stream wrapping/cache-point injection, config pruning defaults if defaults change, public docs, and changelog with a real credited username, not bot credits.
