## Summary

Keep this issue open. Current main still blocks direct Amazon Nova Bedrock models from explicit cache-retention handling and wraps them with `cacheRetention: "none"`, while current AWS Bedrock docs describe Nova-compatible prompt caching and recommend explicit prompt caching for cost savings. 

## What This Changes

The issue requests native prompt-cache support for Amazon Nova models on Amazon Bedrock by allowing cache-retention settings and using Nova-compatible cache checkpoints instead of the current non-Anthropic no-cache behavior.

## Best Possible Solution

Add provider-owned Nova prompt-cache support in the bundled Amazon Bedrock plugin: recognize AWS-supported Nova IDs, preserve explicit cache retention where valid, emit only Bedrock/Nova-compatible `cachePoint` checkpoints in supported request locations, keep unsupported non-Anthropic Bedrock models on the no-cache path, and update tests, docs, and changelog together.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: `resolveCacheRetention({ cacheRetention: "long" }, "amazon-bedrock", "openai-completions", "amazon.nova-micro-v1:0")` resolves to `undefined`, and the provider wrapper for the same direct Nova ID forwards `cacheRetention: "none"`.

## Solution Assessment

Yes. A focused provider-owned repair is the best path because the gap is specific to Amazon Bedrock model-family semantics and AWS prompt-cache rules; moving Nova IDs or TTL policy into generic core would be broader than needed.

## Repair Work Prompt

Implement native prompt-cache support for Amazon Nova models in the bundled Amazon Bedrock provider. Observable current behavior: direct Nova IDs such as `amazon.nova-micro-v1:0` are excluded by `resolveCacheRetention`, then `extensions/amazon-bedrock/register.sync.runtime.ts` routes them through `createBedrockNoCacheWrapper`, which forces `cacheRetention: "none"`. AWS Bedrock docs now document Nova-compatible prompt caching with `cachePoint` and recommend explicit prompt caching for Nova cost savings.

Expected boundary: keep the fix provider-owned in `extensions/amazon-bedrock` plus the narrow shared cache-retention helper path as needed. Recognize AWS-supported Nova model IDs, including direct `amazon.nova-*`, region-prefixed/system-profile refs such as `us.amazon.nova-*` or `global.amazon.nova-*` if AWS supports them, and any safely resolvable profile form. Pass explicit `cacheRetention` where valid. Inject only Bedrock/Nova-compatible `cachePoint` checkpoints in supported locations (`system` and `messages` unless current AWS docs prove more). Keep unsupported non-Anthropic Bedrock models on the no-cache path.

Before coding, verify current AWS Bedrock prompt-caching docs and the pinned pi-ai Bedrock provider source for `@mariozechner/pi-ai@0.70.6`. Do not blindly reuse the Claude `cacheRetention: "long"` to `ttl: "1h"` mapping for Nova; current AWS docs limit 1-hour TTL to specific Claude models unless newer docs say otherwise. Do not change unrelated provider families, OpenClaw core prompt-cache policy, AWS credentials/auth behavior, or dependency versions.

Update focused tests in the Amazon Bedrock plugin and cache-retention helper tests to cover direct Nova, unsupported non-Anthropic Bedrock models, explicit `none`, and any TTL behavior. Update `docs/reference/prompt-caching.md`, `docs/providers/bedrock.md` or `docs/providers/anthropic.md` where the Bedrock notes currently say non-Anthropic models are forced off. Add a user-facing changelog entry with proper contributor attribution.
