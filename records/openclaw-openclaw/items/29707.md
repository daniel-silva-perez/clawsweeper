## Summary

Keep this issue open. Current main still excludes direct Amazon Nova Bedrock model IDs from cache-retention eligibility and forces those direct models through the Bedrock no-cache wrapper, while current AWS Bedrock docs list Nova prompt caching support and recommend explicit prompt caching for cost savings. 

## What This Changes

Request native prompt-caching support for Amazon Nova on Bedrock by allowing valid cache retention and using Nova-compatible Bedrock cache checkpoints instead of disabling caching.

## Best Possible Solution

Add provider-owned Nova prompt-cache support in the bundled Amazon Bedrock plugin: recognize supported direct, regional, global, and application-profile Nova model IDs; preserve explicit `none`; emit Bedrock/Nova-compatible `cachePoint` checkpoints only in supported fields; avoid unsupported 1-hour TTL for Nova unless AWS documents it; keep other unsupported non-Anthropic Bedrock models on the no-cache path; and update tests, public docs, and changelog together.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: the cache-retention helper returns `undefined` for a direct `amazon.nova-micro-v1:0` Bedrock model, and the provider wrapper then forces direct non-Anthropic Bedrock streams through `cacheRetention: "none"`.

## Solution Assessment

Yes, with one correction to the proposed shape. A focused Amazon Bedrock provider repair is the narrowest maintainable path, but the implementation should treat Nova TTL semantics separately rather than passing `long` through as Claude-style `ttl: "1h"`.
