## Summary

Keep open. The underlying bug is still real on current main, but this PR is not merge-ready: current verification and the active Anthropic transport still use the Anthropic API-key path, and the proposed fix does not consistently persist or route Bearer-mode auth through the current transport contract.

## What This Changes

This PR changes custom Anthropic-compatible provider runtime and onboarding paths so providers with `authHeader: true` use `Authorization: Bearer <apiKey>` instead of `x-api-key`, with focused tests including MiniMax-style cases.

## Best Possible Solution

Rework the fix on current main around the active Anthropic transport and onboarding config helpers: `authHeader: true` should make custom `anthropic-messages` runtime requests Authorization-only without losing the credential, verification should use the same decision, inferred Bearer mode should be saved or preserved, and native Anthropic/OAuth/default behavior should remain unchanged with focused tests.

## Reproduction Assessment

Yes. Configure a custom provider with `api: "anthropic-messages"` and `authHeader: true`, then inspect or capture runtime/onboarding requests; current main still builds `x-api-key` in both the verification helper and the active Anthropic transport path.

## Solution Assessment

No. The direction is right, but the proposed branch is stale and solves the problem at the wrong layer; the safer fix belongs in the current transport contract plus the saved custom-provider config path.
