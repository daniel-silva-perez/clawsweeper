## Summary

Keep this PR open. Current main still does not implement the full intended behavior: Anthropic custom-provider verification always uses `x-api-key`, and the runtime support only injects `Authorization` without preventing the Anthropic transport from also building `x-api-key` from the API key. The PR is stale against newer provider-request/auth code, but the underlying bug is still real and focused.

## Best Possible Solution

Rebase this PR onto current main and narrow it around the remaining auth contract: make Anthropic-compatible custom-provider verification and runtime transport consistently honor `authHeader: true`, remove or suppress `x-api-key` in that opted-in path, preserve native Anthropic defaults and explicit false overrides, and add focused tests for onboarding verification plus the Anthropic runtime transport.
