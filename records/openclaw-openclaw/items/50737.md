## Summary

Keep open. The reported `authHeader` bug is still real on current main, and this PR has useful source work, but the branch is stale against the active boundary-aware Anthropic transport and still leaves verification, saved config, and runtime credential serialization out of sync.

## What This Changes

The PR adds runtime and onboarding changes so custom `anthropic-messages` providers with `authHeader: true` use `Authorization: Bearer <apiKey>` instead of `x-api-key`, with focused tests for custom and MiniMax-style endpoints.

## Best Possible Solution

Rework this on current main around the active transport contract: make custom `anthropic-messages` providers with `authHeader: true` send Authorization-only in `src/agents/anthropic-transport-stream.ts`, make custom endpoint verification use the same decision, persist or preserve `authHeader` when onboarding infers it, and keep native Anthropic/OAuth/default and explicit-false behavior unchanged with focused tests.

## Reproduction Assessment

Yes. A high-confidence reproduction can be built with a custom provider configured as `api: "anthropic-messages"` and `authHeader: true`; current-main inspection shows verification still sends `x-api-key`, and the active runtime transport still builds `x-api-key` from `apiKey`.

## Solution Assessment

No, not as-is. The direction is right, but the implementation needs to be rebased and moved into the current active Anthropic transport plus the saved custom-provider config path, rather than relying on the stale extra-params wrapper alone.
