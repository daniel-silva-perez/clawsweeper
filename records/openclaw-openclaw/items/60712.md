## Summary

Keep this issue open. Current main still resolves a matched binding whose agent ID is not in agents.list through pickFirstExistingAgentId, which falls back to the configured default agent, and Telegram inbound routing uses that resolver. The existing related reports in the provided context cover account-scoped routing, hook events, model fallback chains, and direct hook agent IDs; they do not supersede this specific stale binding-to-default-agent privacy behavior.

## Best Possible Solution

Keep the issue open and fix the routing contract: when a configured binding matches but its agentId is not a configured agent, OpenClaw should fail closed or drop with an explicit log/diagnostic instead of silently using the default agent. Add config validation or doctor coverage for bindings[].agentId, targeted resolveAgentRoute tests for stale binding IDs, and Telegram coverage for the reported channel path.
