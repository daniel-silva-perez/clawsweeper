## Summary

Keep open. Current `main` still resolves a matched route binding whose `agentId` is absent from `agents.list` through the same default-agent fallback path, and Telegram inbound routing still enters that resolver. The related #44375 covers default-agent selection when no explicit binding handles traffic; it does not supersede this stale matched-binding privacy behavior.

## What This Changes

The issue asks OpenClaw to stop silently routing matched bindings with missing agent IDs to the default/main agent and instead fail explicitly or use an explicit catch-all.

## Best Possible Solution

Fail closed for matched bindings whose target agent is absent, while preserving the existing default-agent fallback only for true no-match routing. Pair the behavior change with config validation or doctor diagnostics for `bindings[].agentId`, and add focused core routing plus Telegram regression coverage.

## Reproduction Assessment

Yes. A source-level reproduction is clear: configure `agents.list` with `main` as the default, add a matching Telegram or core route binding whose `agentId` is absent, and `resolveAgentRoute` will return the default agent through `pickFirstExistingAgentId`.

## Solution Assessment

Unclear as a product contract until maintainers choose fail-closed versus configurable catch-all, but the narrowest maintainable default is to fail closed for stale matched bindings and leave the normal no-match default route intact.
