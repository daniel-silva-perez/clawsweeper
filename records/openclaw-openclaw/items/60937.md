## Summary

Keep this PR open, but it is not ready as written. Current main still does not apply baseline headers on the exported standalone `startCanvasHost` HTTP path, so the PR is not obsolete; however, adding `X-Frame-Options: DENY` conflicts with the current canvas/A2UI embedding model and needs removal or explicit maintainer policy signoff before merge.

## What This Changes

The PR adds baseline response headers to the standalone canvas host server and adds a canvas-host test asserting those headers on 200 and 404 responses.

## Best Possible Solution

Add standalone-server hardening with a canvas-safe header set: apply `X-Content-Type-Options` and `Referrer-Policy` before routing for success, not-found, and error responses, add focused tests, and leave frame/CSP restrictions out unless maintainers explicitly decide to block canvas iframe embedding.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to start `startCanvasHost` on current main and fetch a served canvas path plus a missing path; the handler does not set the proposed headers before routing. I did not run live tests because this review was required to remain read-only.

## Solution Assessment

No, not as proposed. The narrow hardening is good, but the `X-Frame-Options: DENY` part conflicts with the existing embeddable canvas/A2UI policy; the safer fix is a canvas-specific baseline that omits frame denial or gets explicit maintainer approval for that policy change.
