## Summary

Keep this PR open. Current main already applies baseline headers on the documented Gateway canvas HTTP surface, but it does not implement the PR's standalone `startCanvasHost` hardening. The PR remains a focused security-hardening candidate, with one security/compatibility point for maintainer review around `X-Frame-Options: DENY`.

## Best Possible Solution

Keep the PR open as a candidate. The best path is to rebase it, add standalone-server coverage for success and error/not-found responses, and choose a canvas-specific header set that matches current policy: at minimum `X-Content-Type-Options` and `Referrer-Policy`, with explicit maintainer confirmation before adding `X-Frame-Options: DENY`.
