## Summary

Keep open. Current main only removed the model call for ordinary bare chat resets; it still has no public reset-greeting policy, and the reply-producing and gateway agent reset paths still build a hardcoded greeting prompt and dispatch an agent run using the normal request/session model selection.

## What This Changes

Request configurable bare `/new` and `/reset` greeting behavior, either by selecting a cheaper greeting model or suppressing the greeting model turn entirely.

## Best Possible Solution

Define one typed reset-greeting policy that covers chat acknowledgements, reply-producing reset turns, and gateway agent resets, then wire it through schema, exported types, generated help/docs, and regression tests so users can suppress the model-run greeting or select an explicit cheaper greeting model without changing `/new <message>` and `/reset <message>` behavior.

## Reproduction Assessment

Yes for the observable gap. Current source and tests show bare text-channel resets acknowledge without invoking the model, while gateway and reply-producing reset paths still send the hardcoded greeting prompt into an agent run.

## Solution Assessment

Unclear as proposed. The requested options target the right cost problem, but the maintainable fix should be a single maintainer-chosen reset-greeting policy across reset surfaces rather than separate ad hoc keys that only cover one path.
