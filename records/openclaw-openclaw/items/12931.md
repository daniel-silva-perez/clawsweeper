## Summary

Keep this issue open. Current main partially addresses the cost problem for bare chat `/new` and `/reset` by returning static acknowledgements without invoking the model, but gateway/CLI-style `agent` resets still build and dispatch the reset greeting prompt, and the strict config surface still has no `resetGreeting`, `resetGreetingModel`, or equivalent policy/model override.

## Best Possible Solution

Keep this issue open for maintainer product follow-up. The best path is to decide one typed reset-greeting policy for chat and gateway agent reset paths, probably alongside the open reset-prompt issues, then implement schema/types/help/docs/tests so users can either suppress the model-run greeting or route it through an explicit cheaper model without changing normal reset semantics.
