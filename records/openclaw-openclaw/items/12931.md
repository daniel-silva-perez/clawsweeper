## Summary

Current main only partially addresses the request. Bare chat `/new` and `/reset` now return static acknowledgements without a model run, but reset turns that still produce an agent reply and gateway `agent` resets continue to build the hardcoded reset greeting prompt and dispatch an agent run. The strict config surface exposes `agents.defaults.startupContext` but no `resetGreeting`, `resetGreetingModel`, `resetPrompt`, or equivalent policy key, and an owner comment already kept this open for coordination with #45501 and #51586.

## What This Changes

Request to add configuration for `/new` and `/reset` greeting behavior so users can suppress model-generated reset greetings or route them through a cheaper model.

## Best Possible Solution

Define one typed reset-greeting policy that covers chat acknowledgements, reply-producing reset turns, and gateway agent resets; then wire it through schema/types/help/docs/tests so users can suppress the model-run greeting or choose an explicit cheaper model without changing normal reset semantics.
