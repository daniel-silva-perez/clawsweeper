## Summary

Keep open. Current main has removed the ordinary bare chat `/new` and `/reset` model call, but the remaining reset-greeting policy is still not implemented: reply-producing reset paths and gateway agent resets still build the hardcoded greeting prompt and run the agent, and the strict config surface has no public key to suppress that greeting or choose a cheaper greeting model.

## What This Changes

Request adding configuration to choose a cheaper model for, or suppress, the model-generated greeting after bare `/new` or `/reset`.

## Best Possible Solution

Define one typed reset-greeting policy that covers chat acknowledgements, reply-producing reset turns, and gateway agent resets, then wire it through schema, exported types, generated help/docs, and regression tests so users can either suppress the model-run greeting or select an explicit cheaper greeting model without changing `/new <message>` and `/reset <message>` behavior.

## Reproduction Assessment

Yes for the observable gap. Current source and tests show bare text-channel resets acknowledge without invoking the model, while the gateway agent path for bare `/new` still sends the reset greeting prompt into an agent run.

## Solution Assessment

Unclear as proposed. The requested options address the right user problem, but the maintainable fix should be one maintainer-chosen reset-greeting policy across reset surfaces rather than separate ad hoc keys that cover only one path.
