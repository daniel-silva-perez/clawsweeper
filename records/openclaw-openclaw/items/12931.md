## Summary

Keep this open. Current main only solves the ordinary bare text-channel reset cost by returning a static acknowledgement; the shared reset prompt, reply-run path, gateway agent reset path, and strict config schema still do not provide the requested greeting suppression or cheaper greeting-model policy.

## What This Changes

The issue requests configurable control over bare `/new` and `/reset` greetings, either by selecting a cheaper greeting model or suppressing the greeting turn entirely.

## Best Possible Solution

Add a typed reset-greeting policy that covers chat acknowledgements, reply-producing reset turns, and gateway agent resets, then wire it through config schema, exported types, help text, docs, and regression tests without changing `/new <message>` or `/reset <message>` behavior.
