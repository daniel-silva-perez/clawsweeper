## Summary

Keep this issue open. Current main has implemented the most important chat-facing suppression path: bare chat `/new` and `/reset` now return static acknowledgements and tests assert no embedded model run. However, the gateway `agent` RPC/CLI-style path still turns a bare reset into the Session Startup greeting prompt and dispatches an agent run, and the typed `session` config schema still has no `resetGreeting` or `resetGreetingModel` option. This is now partially addressed, but not complete enough for conservative cleanup closure.

## Best Possible Solution

Keep this issue open but narrow it to the remaining product decision: either declare the current chat-only static acknowledgement sufficient and document that gateway `agent` reset runs are intentionally different, or finish the feature by adding a typed reset-greeting policy/model override and applying it consistently to the gateway `agent` path as well as chat reset handling.
