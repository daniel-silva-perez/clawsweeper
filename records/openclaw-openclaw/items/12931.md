## Summary

Keep open. Current main only solves the ordinary bare text-command cost path: bare chat `/new` and `/reset` stop with a static acknowledgement, but gateway agent resets and reply-producing reset paths still build the hardcoded reset greeting prompt and dispatch a model run, and no public config key exists to suppress that greeting or use a cheaper reset-greeting model.

## What This Changes

The issue requests a configurable reset greeting policy so bare `/new` or `/reset` can either use a cheaper model for the greeting turn or suppress the greeting/model invocation entirely.

## Best Possible Solution

Define one typed reset-greeting policy that covers chat acknowledgements, reply-producing reset turns, and gateway agent resets, then wire it through schema/types/help/docs/tests so users can suppress the model-run greeting or choose an explicit cheaper model without changing `/new <message>` and `/reset <message>` behavior.

## Reproduction Assessment

Yes for the observable gap. Current source and tests show bare text-channel resets no longer invoke the model, while the gateway agent test path for bare `/new` still sends the reset greeting prompt into an agent run.

## Solution Assessment

Unclear as proposed. The requested options describe the right user problem, but the maintainable fix should be a single maintainer-chosen reset-greeting policy rather than separate ad hoc keys that only cover one surface.
