## Summary

Keep open. Current main only resolves the ordinary bare chat reset cost by acknowledging `/new` and `/reset` without a model call; the reply-producing and gateway agent reset paths still build the hardcoded greeting prompt and run the agent with normal request/session model selection, and no public reset-greeting suppression or cheaper-model policy exists.

## What This Changes

The issue requests a configurable `/new` and `/reset` greeting policy that can either use a cheaper model for the reset greeting turn or suppress that model turn entirely.

## Best Possible Solution

Define one typed reset-greeting policy across chat acknowledgements, reply-producing reset turns, and gateway agent resets, then wire it through strict config schema, exported types, generated help/docs, and regression tests so users can suppress the model-run greeting or select an explicit cheaper greeting model without changing `/new <message>` and `/reset <message>` behavior.

## Reproduction Assessment

Yes. Source and existing tests provide a high-confidence reproduction path: bare text-channel resets acknowledge without invoking the model, while gateway and reply-producing reset paths still feed the hardcoded reset greeting prompt into an agent run.

## Solution Assessment

Unclear as proposed. The requested options address the token-cost problem, but the maintainable solution should be a single maintainer-chosen reset-greeting policy rather than ad hoc keys that only cover one reset path.
