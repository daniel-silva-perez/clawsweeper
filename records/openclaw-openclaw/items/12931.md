## Summary

Keep this open. Current main partially reduces the original token-cost problem for ordinary bare text-channel `/new` and `/reset`, but the requested public reset-greeting policy is still missing: reply-producing reset paths and the gateway agent reset path still construct a hardcoded greeting prompt and send it through an agent run, and the strict config schema only exposes startup-context preload controls, not greeting suppression or a cheaper greeting model.

## What This Changes

Requested configurable behavior for bare `/new` and `/reset` greetings, either by selecting a cheaper greeting model or suppressing the greeting/model run entirely.

## Best Possible Solution

Define one typed reset-greeting policy that covers static chat acknowledgements, reply-producing reset turns, and gateway agent resets, then wire it through config schema, exported types, help text, public docs, and regression tests while preserving `/new <message>` and `/reset <message>` behavior.
