## Summary

Keep open. Current main still has the reported `sessions_send(timeoutSeconds=0)` A2A announce path using a hardcoded 30s announce timeout, with no `sessions_send` tool parameter or config key for extending it. Adjacent sub-agent announce delivery has a configurable/default 120s timeout, but that is a separate `sessions_spawn`/subagent path and does not cover this issue’s `sessions_send` flow.

## Best Possible Solution

Implement a `sessions_send` A2A announce timeout policy in `src/agents/tools/sessions-send-tool.ts` and `src/agents/tools/sessions-send-tool.a2a.ts`: either add an explicit `announceTimeoutSeconds` tool parameter or a documented config knob, make `timeoutSeconds=0` use a durable/generous default instead of 30s, and add regression tests proving delayed target replies still announce back without changing the separate subagent announce config semantics.

## Risks / Open Questions

- A fix should distinguish the `sessions_send` peer A2A announce path from the already-configurable `sessions_spawn` subagent announce path so existing subagent delivery behavior is not regressed.
- For nonzero `timeoutSeconds`, the current A2A helper and `runAgentStep` also cap waits at 60s in places, so simply adding a new config key may not fully solve long queued-session replies unless those caps are reviewed deliberately.
