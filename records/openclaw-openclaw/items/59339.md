## Summary

Keep open. Current main still contains the hardcoded `sessions_send` A2A timeout behavior described in the report, and the related open PR/issue overlap with the topic but do not fully supersede this request.

## What This Changes

The issue requests configurable A2A announce behavior for `sessions_send`, especially for `timeoutSeconds=0`, because the current fixed timeout can drop delayed replies.

## Best Possible Solution

Make the `sessions_send` A2A timeout budget explicit and documented, remove the internal 60s caps that defeat larger budgets, and add regression coverage for `timeoutSeconds=0` with delayed target replies.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is `sessions_send(timeoutSeconds=0)` with a target run that completes after 30 seconds; the current code starts the A2A follow-up but gives it only a 30,000 ms announce budget.

## Solution Assessment

Yes, the requested configurability is the right direction, but the safest implementation should fix the `sessions_send` A2A path directly rather than relying on the separate sub-agent announce timeout setting.
