## Summary

Keep open. Current main still gives `sessions_send(timeoutSeconds=0)` a hardcoded 30s A2A announce budget, and the A2A wait path still caps longer budgets at 60s, so the reported fire-and-forget reply loss remains plausible and source-backed. The related open PR #40961 is useful context but only addresses the 60s cap/sub-agent default and does not remove the 30s fire-and-forget budget.

## What This Changes

The issue requests making the `sessions_send` A2A announce timeout configurable or tied to the tool timeout so asynchronous replies are not dropped after the current hardcoded timeout.

## Best Possible Solution

Land a narrow sessions_send A2A timeout fix: honor `timeoutSeconds` end-to-end when it is positive, use a documented configurable or generous default for fire-and-forget mode, remove internal caps that make longer budgets ineffective, and add deterministic regression coverage plus docs/changelog for the user-facing timeout behavior.

## Reproduction Assessment

Yes. The source path is a high-confidence reproduction: `sessions_send(timeoutSeconds=0)` sets the A2A announce budget to 30,000 ms, then returns accepted while the background announce flow gives up if the target run is still pending after that budget.

## Solution Assessment

No. Current main is not the best solution because sub-agent announce configurability does not cover this `sessions_send` path, and the related PR only partially addresses the cap without changing the fire-and-forget default.
