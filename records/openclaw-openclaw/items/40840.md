## Summary

Keep this open. Current main still has the lifecycle gap described in the report: `sessions.delete` removes the live store row after cleanup, but the normal agent-run finalizer still writes `store[sessionKey] = merged` from a pre-run session snapshot without checking whether that key was deleted in the meantime.

## What This Changes

The issue asks for `sessions.delete` to either provide a reliable hard-kill/delete boundary after normal agent runs, fail when that boundary cannot be guaranteed, or document weaker cleanup semantics.

## Best Possible Solution

Add a deletion fence for successful `sessions.delete`, preferably in the post-run session-store updater: when a run started from an existing session key and the current on-disk store no longer contains that key at finalization time, skip or fail the stale write instead of recreating the row. Cover it with a focused regression test that simulates a stale pre-run snapshot, deletes the key, runs the finalizer, and verifies the key stays deleted; if maintainers intentionally want weaker semantics, document that explicitly and keep a separate hard-kill primitive tracked.
