## Summary

Keep open. Current `main` still has the post-run session-store finalizer that can recreate a session key after `sessions.delete` removes it, and there is no merged fix, linked PR, or canonical superseding issue for this narrower durable-delete race.

## What This Changes

The issue asks for durable session termination semantics so `sessions.delete` cannot return success and then have the same session key immediately reappear after a normal agent run.

## Best Possible Solution

Make `sessions.delete` durable for an existing key by adding a post-run finalizer guard: if the caller's snapshot contained the key but the locked current store no longer does, skip the finalizer write and leave the key absent. Preserve current behavior for normal updates while the row still exists and for first-time session creation paths that are not racing a delete.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to start a normal agent run with a loaded `sessionStore` snapshot, delete that same key through `sessions.delete`, then let `updateSessionStoreAfterAgentRun` run; current `main` merges against an undefined locked-store entry and writes the key back.

## Solution Assessment

Yes. The narrowest maintainable fix is a durable-delete guard in `src/agents/command/session-store.ts` with regression coverage; a broader native `sessions_kill` API remains a separate product/API discussion.
