## Summary

Keep open. Current main still has the stale post-run session-store write path that can recreate a key after `sessions.delete` removes it, and the broader closed session-kill discussions do not supersede this narrower runtime bug.

## What This Changes

The issue asks for `sessions.delete` to durably remove a session after normal agent runs, fail when it cannot, or document that it is not a hard-kill boundary.

## Best Possible Solution

Make deletion durable for existing session keys by having the post-run session-store finalizer skip its write when the caller snapshot contained the key but the current locked store no longer does, while preserving normal creation for brand-new keys and current merge behavior when the row still exists.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is: resolve an existing session so the agent run holds a `sessionStore` snapshot, delete that key through `sessions.delete`, then let `updateSessionStoreAfterAgentRun` run; current code merges against `undefined` and writes the same key back.

## Solution Assessment

Yes. The narrowest maintainable fix is a durable-delete guard in `src/agents/command/session-store.ts`; a broader native `sessions_kill` API can remain a separate product discussion after this stale finalizer bug is fixed.
