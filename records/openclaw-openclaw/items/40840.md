## Summary

Keep open. Current main still has the stale post-run session-store finalizer that can recreate a deleted session key, and I did not find current source, tests, docs, changelog, or related-item evidence proving durable `sessions.delete` semantics have shipped. The closed `sessions_kill` feature discussion is broader and does not supersede this narrower runtime race.

## What This Changes

The issue asks for `sessions.delete` or an equivalent session-kill primitive to be a durable boundary so a normal agent-run finalizer cannot immediately re-register the same `sessionKey` after deletion.

## Best Possible Solution

Make deletion durable for existing session keys by adding a post-run finalizer guard: when the caller snapshot contained the key but the current locked store no longer does, skip the write and keep the key absent. Preserve current merge behavior when the row still exists and normal first-time creation when the run is creating a new key.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is: resolve a session so the agent run holds a pre-run `sessionStore` snapshot, delete that key through `sessions.delete`, then let `updateSessionStoreAfterAgentRun` run; current main merges against `undefined` and writes the key back.

## Solution Assessment

Yes. The narrowest maintainable fix is a durable-delete guard in `src/agents/command/session-store.ts` with focused regression coverage; a broader native `sessions_kill` API can remain a separate product discussion.

## Repair Work Prompt

Fix openclaw/openclaw#40840. Current main can resurrect a session row after `sessions.delete` because `updateSessionStoreAfterAgentRun` builds `next` from a pre-run `sessionStore` snapshot, then unconditionally writes `store[sessionKey] = mergeSessionEntry(store[sessionKey], next)` even when the locked current store no longer has that key. Implement a narrow durable-delete guard in `src/agents/command/session-store.ts`: if the caller snapshot contained `sessionKey` but the current locked store no longer does, skip re-creating it and keep the in-memory snapshot consistent with the absence. Preserve normal creation for brand-new keys where the caller snapshot did not contain the key, and preserve current merge behavior when the locked store still has the row. Add focused regression coverage in `src/agents/command/session-store.test.ts` for stale snapshot + deleted disk row, and add a gateway-level lifecycle test under `src/gateway/server-methods/` if feasible to show `sessions.delete` remains durable across the post-run finalizer. Do not add a broad `sessions_kill` API, do not change the main-session delete restriction, and do not weaken ACP metadata preservation. Add a single-line changelog fix entry under the active version with `Thanks @AL-knows`.
