## Summary

Keep open. Current main still does not provide a guaranteed hard-kill/delete boundary for normal `agent` lifecycle writers: `sessions.delete` removes the current row after cleanup, but there is no deletion tombstone/generation/write fence, and the normal agent-run finalizer can still write the same session key from a stale in-memory session snapshot. The issue is a concrete core Gateway/session-store semantic gap, not ClawHub scope and not covered by the closed broader `sessions_kill` request.

## Best Possible Solution

Keep this open and fix it in core Gateway/session-store semantics. Add a regression test that starts from a normal `agent` run with a stale session-store snapshot, calls `sessions.delete`, then proves the late `updateSessionStoreAfterAgentRun` cannot resurrect the deleted key. A robust fix should add a per-session deletion generation/tombstone/write fence for in-flight writers, or make `sessions.delete` fail until all tracked normal agent finalizers for that key can no longer persist state. Then document whether `sessions.delete` is a hard-kill boundary or introduce a separate native `sessions.kill` primitive with that guarantee.
