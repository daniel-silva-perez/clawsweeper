## Summary

Keep this open. Current main still has the reported lifecycle gap: `sessions.delete` cleans runtime state and deletes the current row, but the normal agent-run finalizer can persist from a stale in-memory session-store snapshot after that delete and recreate the same key. The newer HTTP kill route is subagent-oriented and does not change the `sessions.delete` finalizer/write-fence semantics.

## What This Changes

The issue asks OpenClaw to make `sessions.delete` a reliable hard-kill/delete boundary, add a native session kill primitive, or explicitly document that delete is only best-effort maintenance when normal agent runs are in flight.

## Best Possible Solution

Make successful `sessions.delete` fence off stale finalizer writes for the deleted key, preferably with a generation/tombstone check or an update-path guard, and add regression coverage for delete racing a normal agent-run finalizer. If maintainers want delete to remain maintenance-only instead, the protocol/docs should say that clearly and a separate native hard-kill primitive should cover integrator termination controls.

## Repair Work Prompt

Fix the stale-finalizer race reported here: after a normal agent run has resolved a session and holds an in-memory `sessionStore` snapshot, a successful `sessions.delete` must not allow the later `updateSessionStoreAfterAgentRun` finalizer to recreate the deleted key. Implement the narrowest core write fence/generation/tombstone or update-path guard needed for this behavior. Preserve legitimate future creation of the same key by a new inbound/run after delete, preserve existing main-session rejection, ACP/subagent cleanup, lifecycle hooks, transcript archiving, and session maintenance behavior. Do not add a new `sessions.kill` RPC unless a maintainer explicitly asks. Add regression coverage that deletes a key between session resolution and `updateSessionStoreAfterAgentRun`, then asserts the stale finalizer does not restore it; also cover that a genuinely fresh post-delete run can create a new entry if the chosen guard could affect that path. Update protocol/docs only if the final semantics need user-facing clarification. Add a changelog fix entry because this is user-facing integrator behavior, crediting `Thanks @AL-knows`.
