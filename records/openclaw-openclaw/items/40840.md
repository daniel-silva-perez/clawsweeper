## Summary

Keep open. Current main still has the durable-delete race described in the report: a normal agent run keeps a pre-delete session-store snapshot and the post-run finalizer can write the same key back after `sessions.delete` removed it, with no merged fix or canonical open replacement found for this narrower behavior.

## What This Changes

The issue asks for `sessions.delete` to either be a durable hard-kill/delete boundary after normal agent runs or to document that it is only best-effort store cleanup.

## Best Possible Solution

Add a durable-delete guard to the post-run session-store finalizer: when the caller's pre-run snapshot contained the key but the locked current store no longer does, skip the finalizer write and keep the key absent. Preserve current first-time session creation and normal update behavior, and cover the race with focused regression tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to resolve a normal agent run with an existing `sessionStore` snapshot, delete that same key through `sessions.delete`, then let `updateSessionStoreAfterAgentRun` execute; current main merges against a missing locked-store entry and writes the key back.

## Solution Assessment

Yes. The narrowest maintainable fix is a guard in `src/agents/command/session-store.ts` with focused tests; adding a broad `sessions_kill` API can remain a separate product/API discussion.
