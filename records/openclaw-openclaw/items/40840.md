## Summary

Keep this open. Current main still has the reported lifecycle gap: a normal agent run resolves and holds a pre-run session-store snapshot, `sessions.delete` can remove that row successfully, and the post-run finalizer can write the same key back because it unconditionally assigns into the locked store without checking whether the current row was deleted meanwhile.

## What This Changes

The issue requests hard session termination semantics, or at least durable same-key deletion semantics, for `sessions.delete` after normal agent runs touch a session.

## Best Possible Solution

Make deletion durable for an existing session key by having the post-run session-store finalizer treat a missing current locked-store row as a deleted session and skip the stale write, while preserving normal creation of brand-new sessions and existing merge behavior when the current row still exists.
