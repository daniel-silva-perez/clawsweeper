## Summary

Keep this open. Current main still has the lifecycle gap described in the report: `sessions.delete` cleans up active runtime state and deletes the current store row, but the normal agent-run finalizer can write a stale pre-run session snapshot back to the same key after the delete succeeds.

## What This Changes

The issue asks for `sessions.delete` to either be a reliable hard termination boundary after normal agent runs or be documented as best-effort session-store maintenance only.

## Best Possible Solution

Add a deletion fence for successful `sessions.delete`, preferably by having post-run session-store updates detect that a pre-existing session key was removed before finalization and skip or fail that stale write. Cover the behavior with a focused regression test that simulates a loaded pre-run store snapshot, deletes the same key, then runs the finalizer and verifies the key stays deleted. If maintainers intentionally want delete to remain maintenance-only, document that limitation and keep a separate hard-kill primitive tracked explicitly.
