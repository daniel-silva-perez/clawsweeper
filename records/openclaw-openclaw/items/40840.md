## Summary

Keep open. Current main still has the reported lifecycle gap: the normal agent-run finalizer can write a stale pre-run session snapshot back to `sessions.json` after `sessions.delete` has successfully removed that same key, and I did not find a deletion fence, tombstone, generation guard, or documentation that makes the weaker semantics explicit.

## What This Changes

The issue asks for `sessions.delete` to be a reliable hard-kill/delete boundary after normal agent runs, or for OpenClaw to document that it is only best-effort store cleanup.

## Best Possible Solution

Add a deletion fence to the session-store write path, preferably in `updateSessionStoreAfterAgentRun`: when the run started from an existing key and the current locked store no longer contains that key at finalization time, skip the stale write or return a structured no-op/failure instead of recreating the row. Keep the existing cleanup behavior, add regression coverage for the stale snapshot case, and document weaker semantics only if maintainers intentionally decline hard-delete behavior.

## Repair Work Prompt

Fix the stale session resurrection race reported in https://github.com/openclaw/openclaw/issues/40840. Current main lets `updateSessionStoreAfterAgentRun` recreate a deleted `sessionKey` because it merges from a pre-run `sessionStore` snapshot and unconditionally writes `store[sessionKey] = merged` after `sessions.delete` removed the key. Add a deletion fence at the narrowest safe boundary, preferably in `src/agents/command/session-store.ts`: if the run began from an existing session key and the current locked store no longer contains that key at finalization time, skip the stale write or return a structured no-op/failure instead of recreating the row. Preserve intended behavior for genuinely new sessions and for normal existing sessions, and do not introduce a new public hard-kill API unless the existing API surface requires it. Add focused regression coverage in `src/agents/command/session-store.test.ts` for the stale snapshot/delete case and, if useful, one Gateway lifecycle test in `src/gateway/server.sessions.delete-lifecycle.test.ts`. Keep existing cleanup behavior in `src/gateway/server-methods/sessions.ts` and `src/gateway/session-reset-service.ts`. Update `CHANGELOG.md` under the active fixes/changes section with a single-line user-facing fix entry crediting `Thanks @AL-knows`; do not touch unrelated snapshots/baselines.
