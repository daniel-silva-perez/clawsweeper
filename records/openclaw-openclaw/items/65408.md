## Summary

Keep this issue open. Current main still preserves the reported Control UI shared-token mismatch recovery path through cached device tokens, current docs and tests describe that as expected behavior, and the logout/bootstrap UX parts are not implemented as first-class Control UI session controls.

## What This Changes

The issue requests that Control UI shared-token rotation invalidate stale browser access, that the UI provide a logout action clearing persisted auth state, and that refresh show a bootstrap/restoring state instead of flashing the login form.

## Best Possible Solution

Define the shared-token rotation contract first, then enforce it consistently by either binding/revoking browser device tokens when the shared token changes or removing the shared-token mismatch fallback for Control UI shared-token sessions. Add a Control UI logout that clears the sessionStorage gateway token, persisted device auth entry, and active GatewayBrowserClient state, plus an explicit restoring state before deciding to show the login form.

## Reproduction Assessment

Yes. The current source and focused tests provide a high-confidence reproduction path: authenticate with a gateway token, retain a stored paired device token, rotate the shared token, and the client/server retry path can still authenticate through the cached device token.

## Solution Assessment

No current fix is present. The maintainable path needs a maintainer-approved auth contract before changing the fallback, because the existing behavior is documented and tested rather than an accidental one-line bug.
