## Summary

Keep open. Current main still has only partial internal `argPattern` support: it is enforced for Windows targets, but non-Windows path matches ignore it, and the public CLI/UI/docs still present manual allowlists as executable path or command-name patterns rather than a documented argument-level contract for the reported `node script args` use case.

## Best Possible Solution

Keep this issue open and implement a cross-platform, documented argument-level allowlist contract for exec approvals. Maintainers should choose the public field name and matching semantics, enforce it on macOS/Linux as well as Windows, and update schema validation, CLI, Control UI, docs, and regression tests around interpreter-script invocations such as `/opt/homebrew/bin/node /Users/me/.openclaw/mute_user.mjs * *`.

## Risks / Open Questions

- Host exec allowlist semantics are security-sensitive; adding argument patterns needs a deliberate contract for tokenization, glob-vs-regex behavior, path normalization, shell wrappers, and interpreter/script binding.
- The existing internal `argPattern` is regex-based and Windows-oriented, while the issue proposes a glob-style `argsPattern`; reusing or renaming it without compatibility design could either over-trust commands or break existing allow-always metadata.
