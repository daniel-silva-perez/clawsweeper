## Summary

Keep open. Current main still has only partial internal `argPattern` support: the matcher deliberately ignores it on non-Windows platforms, public docs describe manual allowlists as executable path or bare command-name patterns, and CLI/UI editors expose only a single executable pattern field. The related closed reports cover adjacent path/compound-command allowlist behavior, but they do not supersede this cross-platform argv-level interpreter/script allowlist request.

## What This Changes

Request to let exec approval allowlist entries constrain the argv after the executable, so a restricted agent can authorize one trusted interpreter/script invocation without allowlisting the whole interpreter.

## Best Possible Solution

Define and implement a cross-platform public argument-level allowlist contract for host exec approvals. The end state should cover macOS, Linux, and Windows matching; interpreter/script invocations such as `node mute_user.mjs <chat_id> <user_id>`; CLI and Control UI editing; protocol/schema compatibility; docs; and regression tests for both matching and bypass-resistant tokenization.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a pure feature request, but current behavior is high-confidence from source and docs: non-Windows `matchAllowlist` ignores `argPattern`, and public CLI/UI/docs expose executable-pattern allowlists only.

## Solution Assessment

Is this the best way to solve the issue? Unclear for the exact proposed `argsPattern` glob shape; the capability is valid, but maintainers should reconcile it with existing regex-based `argPattern` metadata and define a secure cross-platform argv contract before implementation.
