## Summary

Keep open. Current main still supports only per-provider model request policy, and the reported global/default provider proxy plus SSRF posture is not present in schema, exported types, docs, or the model runtime merge path. The request is valid but security-sensitive because a global `allowPrivateNetwork` default could broaden SSRF exceptions, so it needs maintainer product/security design before implementation.

## What This Changes

Request a global/default model-provider request policy so operators can configure proxy routing and `allowPrivateNetwork` once while preserving per-provider overrides.

## Best Possible Solution

Design a security-reviewed model-provider request-default layer, likely under `models`, that merges before per-provider `request` overrides, is visible in effective config, documents local/self-hosted opt-out behavior, and keeps proxy/TLS/auth secrets covered by existing config schema and redaction contracts.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure only a proposed global default such as `models.defaults.request` and omit `models.providers.<id>.request`; current strict schema has no such field, and runtime consumes only the selected provider's `request` block.

## Solution Assessment

Unclear. The requested direction is plausible and narrower than relying on the process-wide proxy path, but the safest shape and SSRF inheritance semantics need maintainer review before this should be implemented.
