## Summary

Keep this issue open. Current main still keys runtime auth snapshots through resolved auth-store paths, and the default auth path can follow `OPENCLAW_AGENT_DIR`; that means the requested agent and default runtime snapshot can collapse to the same key. MiniMax portal discovery still reads through `ensureAuthProfileStore(ctx.agentDir)`, and the direct fix PR #64320 was closed unmerged.

## What This Changes

The issue asks OpenClaw to resolve MiniMax OAuth/token auth from the selected agent’s `auth-profiles.json` instead of effectively using `agents/main` when a dashboard/session agent such as `kate` is active.

## Best Possible Solution

Land a focused auth-runtime fix that keys active auth snapshots by stable runtime identity, keeps fallback to the real default/main store explicit, and adds a regression covering distinct main and selected-agent `minimax-portal` profiles when `OPENCLAW_AGENT_DIR` points at the selected agent.

## Repair Work Prompt

Fix the agent-specific MiniMax auth runtime snapshot collision reported at https://github.com/openclaw/openclaw/issues/64274. Current main keys active auth profile runtime snapshots by resolved auth-store paths, so when `OPENCLAW_AGENT_DIR` points at a selected agent, the default and selected-agent snapshot identities can collapse and MiniMax portal auth may resolve from the wrong store. Use https://github.com/openclaw/openclaw/pull/64320 as source context, but rebase the idea onto current main rather than copying blindly.

Expected boundary: update the auth-profile runtime snapshot identity/resolution path so default/main and explicitly requested agent snapshots remain distinct, while preserving documented read-through fallback from selected agents to the real default/main store when no local profile exists. Do not copy OAuth credentials between agents, do not broaden OAuth portability, do not change MiniMax provider ids or model catalog semantics, and do not add core special cases for MiniMax beyond regression coverage.
