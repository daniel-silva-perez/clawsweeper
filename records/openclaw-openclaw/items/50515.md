## Summary

Keep this PR open. Current main still has the shared channel logging helpers interpolating log fields without sanitization, so the PR is not obsolete or implemented-on-main. The security goal is valid, but the submitted patch needs revision before merge because it leaves `channel` unsanitized, duplicates an existing stronger sanitizer, and can change runtime behavior for loosely typed plugin callers.

## Best Possible Solution

Keep this PR open and request a narrow revision: reuse `sanitizeForLog` from `src/terminal/ansi.ts`, sanitize every interpolated field including `channel`, `reason`, `target`, `action`, and `error`, coerce values with `String(...)` before sanitizing to preserve JS plugin compatibility, and add focused tests for channel injection, DEL/C1/ANSI stripping, and non-string runtime inputs.

## Risks / Open Questions

- Merging the PR as-is would still leave `params.channel` as a log-forging surface in all three helpers.
- Merging the PR as-is would introduce a second sanitizer with weaker/different behavior than `src/terminal/ansi.ts`, including no ANSI/OSC8/C1 coverage.
- Merging the PR as-is could throw for JavaScript plugin callers that pass non-string `target` or `reason` values that current main stringifies through template literals.
