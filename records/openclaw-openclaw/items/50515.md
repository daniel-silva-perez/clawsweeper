## Summary

Keep this PR open. Current main is still not fixed, and the PR addresses a real shared channel logging gap, but the patch is not merge-ready because it leaves an interpolated field unsanitized, duplicates a weaker sanitizer, and can regress loose JavaScript plugin callers.

## What This Changes

The PR adds a local channel log sanitizer, applies it to reason, target, and error fields in shared channel logging helpers, and adds focused channel logging sanitization tests.

## Best Possible Solution

Revise the PR to use the shared terminal log sanitizer through a small local wrapper that coerces values with `String(...)`, then sanitize every interpolated field in these helpers, including `channel`, `reason`, `target`, `action`, and `error`, with tests for channel injection, DEL/C1/ANSI stripping, and non-string JavaScript runtime inputs.

## Reproduction Assessment

Yes. By inspection, a PR-branch call such as `logInboundDrop({ channel: "x\nFAKE", reason: "drop", log })` would still emit a forged second line, and a loose JS caller passing a numeric `target` would hit `.replace` on a non-string value; I did not run tests because this review was required to be read-only.

## Solution Assessment

No, this is not the best way to solve the issue as-is. The narrow maintainable fix is to centralize on the existing `src/terminal/ansi.ts` sanitizer, preserve runtime stringification behavior, and cover all fields that enter the log message.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Sanitize the channel field before logging:** `src/channels/logging.ts:16`
  - body: The new sanitizer is never applied to `params.channel`, but that value is still interpolated into every helper output. A channel string containing a newline or other control character can still forge log entries after this patch, so sanitize a local `channel` value before building each message.
  - confidence: 0.93
- **[P2] Coerce values before calling replace:** `src/channels/logging.ts:6`
  - body: `sanitizeForLog` accepts a string and immediately calls `.replace`, while these helpers are public plugin SDK exports and JavaScript plugins can pass loose runtime values. Before this patch, template literals stringified those values; after it, a numeric or object `target`/`reason` can throw instead of logging, so coerce inside the sanitizer wrapper.
  - confidence: 0.88
- **[P2] Reuse the shared log sanitizer:** `src/channels/logging.ts:6`
  - body: This local sanitizer duplicates the existing terminal `sanitizeForLog` but does not strip the same log-control surface, including C1 controls and ANSI/OSC sequences. Reuse the shared helper so channel logs follow the repository-wide log-safety contract.
  - confidence: 0.84
