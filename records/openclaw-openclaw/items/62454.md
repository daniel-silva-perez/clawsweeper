## Summary

Keep this PR open rather than closing it as cleanup. Current main does not implement the PR's intended behavior: no-message final events still trigger the history reload fallback, session entry loading still deliberately chooses the freshest duplicate row, and the cron session reaper still only prunes by retention. The PR also needs rework because review comments and current tests show likely regressions in all three touched areas.

## Best Possible Solution

Keep the PR open only if the author or maintainer wants to rework it. The best path is to split the three fixes, add a reproduction-backed test for each observed bug, preserve the no-message final reload fallback while fixing the /new or /reset race at the command/session boundary, reconcile archived-session clicks without breaking freshest duplicate handling, and make cron recovery use live runtime ownership plus durable run/job state instead of a blanket updatedAt cutoff.
