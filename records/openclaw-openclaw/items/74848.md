## Summary

Keep open. Current main still limits macOS companion stale TLS pin auto-repair to loopback and `*.ts.net`, while the report is for a public `wss://` host; open PR #75228 explicitly targets the remaining case and is still unmerged, so this issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue requests fixing a macOS app remote-mode node reconnect loop where the app node fails with `cancelled` against a WSS gateway while `openclaw node run --tls` works in the same environment.

## Best Possible Solution

Review and either land PR #75228 or an equivalent macOS companion fix that repairs stale pins for explicitly opted-in, system-trusted public WSS hosts, keeps default behavior strict for arbitrary public hosts, preserves fail-closed handling for self-signed/private-CA mismatches, and ships focused Swift coverage plus docs/config alignment.

## Reproduction Assessment

No. The reporter logs and current source/tests provide a strong code-level reproduction hypothesis, but a high-confidence runtime repro needs the macOS app with an existing stale pin against a rotated, system-trusted, non-`.ts.net` WSS gateway.

## Solution Assessment

Unclear pending maintainer review. PR #75228 is a promising direction because it makes the public-host repair opt-in and default-off, which is safer than silently broadening all system-trusted hosts, but this remains a TLS trust-policy change.
