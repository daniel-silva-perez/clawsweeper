## Summary

Keep open. Current main still has a macOS-native node-only WSS session path that can plausibly fail with `connect failed ... cancelled` while the app control channel and CLI node continue to work, and the likely repair overlaps unresolved TLS pinning/trust policy work rather than a merged fix.

## What This Changes

The issue reports that the macOS app native node repeatedly disconnects with `cancelled` in remote direct WSS mode even though the app control channel and `openclaw node run --tls` work against the same gateway.

## Best Possible Solution

Diagnose and fix the native macOS app node WSS trust/reconnect path so it follows the documented remote TLS fingerprint and trust behavior, surfaces actionable stale-pin remediation when a stored fingerprint no longer matches, and keeps control-channel, native-node, and CLI-node behavior consistent without weakening gateway trust.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No. The logs plus current source give a strong code-level hypothesis, especially a stale stored WSS fingerprint mismatch, but a deterministic reproduction still needs the reporter's macOS pin state or a local macOS direct-WSS setup with a rotated certificate.

## Solution Assessment

Is this the best way to solve the issue? Unclear. The narrow code surface is likely `GatewayTLSPinningSession`, `MacNodeModeCoordinator.buildSessionBox`, and remote TLS config plumbing, but maintainers should first choose the expected first-enrollment and stale-fingerprint policy because this is security-sensitive.
