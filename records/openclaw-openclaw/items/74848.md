## Summary

Keep open. Current main still leaves the reported public `wss://` macOS app-node stale TLS pin path unhandled, and open PR #75228 is an unmerged closing PR for the same behavior, so this issue should stay open pending review or rejection of that PR.

## What This Changes

The issue reports that the macOS app's native node repeatedly disconnects with a `cancelled` WSS gateway error in remote direct mode while `openclaw node run --tls` works against the same gateway.

## Best Possible Solution

Review and either land PR #75228 or an equivalent macOS fix that repairs stale pins for system-trusted public WSS hosts, keeps untrusted/self-signed/private-CA mismatches fail-closed, and preserves focused Swift coverage for the policy boundary.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No. The logs, current-main policy, and tests provide a strong reproduction hypothesis, but a high-confidence live repro still needs the reporter's stale pin state or a controlled macOS WSS gateway certificate rotation.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers review the TLS trust boundary in #75228. The proposed direction is narrow and well matched to the app-vs-CLI difference if `systemTrustOk` is accepted as the repair gate.
