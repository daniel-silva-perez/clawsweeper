## Summary

Keep open. Current main still only auto-repairs the macOS native node stale TLS pin path for loopback and `*.ts.net`, while the reported public `wss://` host remains a plausible failure case; open PR #75228 explicitly targets this issue but is unmerged, so cleanup should not close the issue ahead of that PR.

## What This Changes

The issue asks OpenClaw to fix a macOS app remote-mode native node reconnect loop where the app control channel and CLI node connect to the same WSS gateway but the macOS app node repeatedly fails with `cancelled`.

## Best Possible Solution

Review and either land PR #75228 or an equivalent macOS fix that repairs stale pins for system-trusted public WSS hosts, preserves strict behavior for self-signed/private-CA gateways, and includes focused Swift coverage plus a macOS remote WSS smoke comparison against the CLI node path.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No. The logs and current-main tests give a strong reproduction hypothesis, but a deterministic repro needs the reporter's stale pin state or a local macOS WSS gateway with a rotated system-trusted certificate on a non-`*.ts.net` host.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers review #75228's trust-policy boundary. The candidate direction is narrow and well-targeted if `systemTrustOk` is accepted as the auto-repair gate, but this should not bypass security review of macOS TLS pinning behavior.
