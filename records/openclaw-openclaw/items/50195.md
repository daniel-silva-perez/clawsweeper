## Summary

This issue should stay open. Current main still lacks a WhatsApp pre-built `linkPreview` send surface, an opt-in `generateHighQualityLinkPreview` config key, and a batch `send` Gateway RPC.

## Best Possible Solution

Keep this issue open, but split the work. First add a typed WhatsApp pre-built `linkPreview` pass-through through the outbound/Gateway send surface with focused tests and docs. Separately decide whether automatic `generateHighQualityLinkPreview` should be an opt-in config because of SSRF risk. Treat batch send as a Gateway protocol design item with per-message idempotency and partial-result semantics rather than folding it into the preview pass-through PR.

## Risks / Open Questions

- Automatic high-quality link previews require OpenClaw/Baileys to fetch remote URLs, so the config design needs explicit opt-in semantics and SSRF/network-fetch documentation.
- Pre-built preview metadata should be treated as untrusted display metadata and should not enter agent context or bypass existing outbound sanitization expectations.
- A batch Gateway method needs product/API decisions for idempotency, per-message partial results, sequencing, retry/rate-limit behavior, and transcript mirroring before an automated implementation is safe.
