## Summary

Keep this issue open. Current main still lacks a WhatsApp pre-built `linkPreview` pass-through, has no opt-in WhatsApp `generateHighQualityLinkPreview` config/socket setting, and exposes only the single-message Gateway `send` RPC. The related attempts do not resolve the current behavior: #43397 is closed unmerged, #18414 was merged and then reverted, and #6722 was closed as superseded by this broader tracker.

## What This Changes

Request WhatsApp support for caller-supplied link-preview payloads in outbound sends, an opt-in high-quality preview config, and a batch send Gateway RPC.

## Best Possible Solution

Split the issue into staged work: first add a typed WhatsApp-owned pre-built `linkPreview` pass-through through the outbound/Gateway send path with tests and docs, then handle automatic preview fetching and batch Gateway sending as separate design decisions with explicit security and protocol semantics.

## Risks / Open Questions

- Enabling Baileys automatic high-quality previews can make server-side URL fetches, so SSRF and URL-exfiltration policy need explicit maintainer review before adding an opt-in config.
- A batch Gateway send method needs protocol design for per-message idempotency, partial failures, rate limiting/backoff, transcript mirroring, and result ordering.
- The pre-built preview pass-through should be scoped to WhatsApp-owned outbound payloads unless maintainers intentionally expand the generic channel payload contract.
