## Summary

Keep this open. Current main still lacks the requested WhatsApp pre-built link preview pass-through, the proposed WhatsApp high-quality preview opt-in, and a batch Gateway send RPC. The request is valid, but it combines a narrow WhatsApp plumbing change with SSRF-sensitive automatic URL fetching and a new protocol surface, so maintainer split/prioritization is needed before repair automation.

## What This Changes

Add WhatsApp pre-built `linkPreview` pass-through, optional automatic high-quality preview generation, and a batch send Gateway RPC for bulk outbound messages.

## Best Possible Solution

Split the work into staged maintainable changes: first add a WhatsApp-owned typed pre-built `linkPreview` pass-through through the outbound and Gateway send path with focused tests and docs, then separately design automatic preview fetching and batch Gateway sending with explicit security and protocol semantics.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: current main rejects extra Gateway `send` fields through `SendParamsSchema`, builds WhatsApp text payloads as `{ text }`, has no WhatsApp preview-generation config/socket option, and advertises no `send.batch` method.

## Solution Assessment

No, not as one combined automated change. The pre-built preview pass-through is a narrow extension-owned fix, but automatic URL fetching and batch Gateway sending are separate security/protocol decisions that should be split and reviewed by maintainers.

## Risks / Open Questions

- Automatic high-quality preview fetching would make the Gateway fetch URLs from outbound message text and needs explicit SSRF, DNS/private-network, redirect, timeout, response-size, proxy, and privacy/logging design before enablement.
- A `send.batch` Gateway method needs protocol semantics for per-message idempotency, partial failure, ordering, rate limiting, session mirroring, and result shape before implementation.
- The pre-built `linkPreview` pass-through is narrower, but it still needs a typed schema/validation boundary so arbitrary caller objects are not blindly bridged into Baileys.
