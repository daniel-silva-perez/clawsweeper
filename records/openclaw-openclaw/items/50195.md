## Summary

Keep this issue open. Current main still lacks a WhatsApp pre-built `linkPreview` pass-through, has no opt-in WhatsApp `generateHighQualityLinkPreview` config/socket setting, and exposes only the single-message Gateway `send` RPC, while related implementation attempts are closed, reverted, or superseded.

## What This Changes

Request WhatsApp support for caller-supplied link-preview payloads in outbound sends, an opt-in high-quality preview config, and a batch send Gateway RPC.

## Best Possible Solution

Keep this issue open but split the work. First add a typed WhatsApp pre-built `linkPreview` pass-through through the outbound/Gateway send surface with focused tests and docs. Separately decide whether automatic `generateHighQualityLinkPreview` should be an opt-in config because of SSRF risk. Treat batch send as a distinct Gateway protocol design item with per-message idempotency and partial-result semantics.

## Risks / Open Questions

- A batch Gateway send method needs protocol design for per-message idempotency, partial failures, rate limiting/backoff, transcript mirroring, and result ordering.
- The pre-built preview pass-through should be scoped to WhatsApp-owned outbound payloads unless a generic channel payload contract is intentionally expanded.
