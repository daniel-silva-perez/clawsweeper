## Summary

Keep this issue open. Current main still defaults memory to the builtin backend and the builtin forced/full reindex path creates a fresh random temp SQLite database, builds the whole index, then swaps it into place only after success. I found progress reporting and embedding-cache seeding, but no durable checkpoint cursor, reusable temp DB discovery, resume-on-restart config, incomplete-generation status, or interrupted-reindex regression coverage.

## Best Possible Solution

Keep this issue open until `memory-core` has a durable full-reindex checkpoint/resume design for the builtin backend. A good fix should persist generation/checkpoint state or safely reuse/discard incomplete temp DBs, resume from committed file/chunk progress after restart, prevent search/status from treating incomplete generations as clean results, expose config/status/docs if new controls are added, and add regression coverage for interrupted forced/full reindex behavior. Keep #70567 separate unless maintainers intentionally create a broader memory indexing reliability tracker.
