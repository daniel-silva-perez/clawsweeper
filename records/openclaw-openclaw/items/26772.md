## Summary

Keep this issue open. Current main still uses the builtin SQLite memory backend by default and forced/full builtin reindex still builds a fresh random temp database that is swapped only after the full build succeeds; there is no durable checkpoint cursor, reusable temp DB discovery, resume-on-restart setting, or incomplete-generation status/search contract. The existing maintainer/Codex comment on the issue also explicitly kept it open, and the related reports are adjacent reliability bugs rather than a canonical replacement for crash checkpoint/resume.

## What This Changes

The issue requests durable checkpoint/resume support for long-running memory indexing so interrupted forced/full reindexes can resume instead of losing all temp-database progress.

## Best Possible Solution

Add a memory-core design for resumable builtin full reindexing that persists generation/checkpoint state or safely reuses incomplete temp databases, resumes from committed file/chunk progress after restart, keeps status/search aware of incomplete generations, and updates config/docs/tests only for the controls that are actually supported.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: use the default builtin backend, start `openclaw memory index --force`, interrupt it mid-build, and the next run enters `runSafeReindex` with a new random temp database and no checkpoint/temp discovery path.

## Solution Assessment

Unclear for the issue’s proposed options. The best implementation should be chosen by maintainers because the safe design has to balance resumability with the existing atomic-swap invariant and clean search/status semantics.
