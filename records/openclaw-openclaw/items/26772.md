## Summary

Keep this issue open. Current main still defaults memory search to the builtin SQLite backend, and forced/full builtin reindexes still build a fresh random temp database before a final atomic swap. I found progress reporting, WAL checkpointing, and embedding-cache seeding, but no durable full-reindex checkpoint cursor, reusable temp database discovery, resume-on-restart config, status surface for incomplete generations, or regression coverage for interrupted full-reindex resume. The Apr 26 maintainer/Codex comment already reached the same conclusion and correctly treats #70567 as adjacent rather than canonical.

## Best Possible Solution

Keep this issue open until memory-core has a durable full-reindex checkpoint/resume design. A good fix should persist generation/checkpoint state or safely reuse/discard incomplete temp DBs, resume from committed file/chunk progress after process restart, prevent search from treating incomplete generations as clean results, expose status/config/docs if new controls are added, and add regression coverage for interrupted full reindex behavior. Keep #70567 separate unless a broader memory reliability project intentionally covers both transient embedding-failure dirty-state reporting and process-crash resume.
