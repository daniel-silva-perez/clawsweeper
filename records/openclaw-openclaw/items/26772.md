## Summary

Keep this issue open. Current main still uses the builtin SQLite memory backend by default, forced/full reindex still builds a fresh random temp database and swaps it only after the full build succeeds, and there is still no durable checkpoint cursor, reusable temp DB discovery, resume-on-restart config, incomplete-generation status, or interrupted-reindex regression coverage.

## Best Possible Solution

Keep this issue open until the builtin memory-core full-reindex path has a durable checkpoint/resume design. A good fix should persist generation/checkpoint state or safely reuse/discard incomplete temp DBs, resume from committed file/chunk progress after restart, keep search/status from treating incomplete generations as clean, expose config/status/docs if new controls are added, and add regression coverage for interrupted forced/full reindex behavior. Keep adjacent reports such as #56815 and #70567 separate unless maintainers intentionally create a broader memory indexing reliability tracker.

## Risks / Open Questions

- A fix needs to preserve the existing safe atomic-swap invariant so an incomplete generation cannot replace a known-good index or be treated as clean search/status data.
- Embedding-cache seeding reduces repeated embedding work from the committed main DB, but it does not discover or resume abandoned random temp DB progress after a process crash.
- Related retry work for transient embedding failures may reduce how often long reindexes abort, but it does not by itself provide deterministic checkpoint/resume after process death.
