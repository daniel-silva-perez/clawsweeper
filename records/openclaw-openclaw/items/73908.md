## Summary

Current main partially addresses the manifest-provider startup side: primary model prewarm is now backgrounded and scoped to the resolved primary provider. This issue should stay open because the external CLI credential side still appears valid: generic auth-store overlay still iterates Codex CLI, Claude CLI, and MiniMax CLI credential readers without a config/plugin/auth-profile scope, and the Claude reader can call macOS Keychain by default. Gateway auth status also calls the generic auth-store path without disabling that overlay, so an opencode-only setup can still hit unrelated external CLI credential discovery.

## Best Possible Solution

Keep this open and narrow the remaining work to provider-scoped external CLI credential discovery. Preserve the current scoped model-prewarm implementation, then make external CLI overlays honor configured provider/plugin/auth-profile eligibility so Claude CLI Keychain is not read for an opencode-only setup. Add regression coverage for restrictive plugins.allow plus auth.profiles/model defaults that exclude Claude/Anthropic, and cover gateway models.authStatus so TUI startup does not reintroduce the Keychain read.
