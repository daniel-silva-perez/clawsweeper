## Summary

Keep open. Current main still lacks the requested Mantle discovery opt-out and the linked bug remains valid, but this branch should be revised before merge because the guard is tied to registration-time plugin config instead of the current catalog config and the user-facing config path still needs regression coverage and docs.

## What This Changes

This PR adds an `amazon-bedrock-mantle` `config.discovery.enabled` manifest field with UI hints and returns `null` from the Mantle catalog handler when registration-time plugin config sets discovery to false.

## Best Possible Solution

Update this PR, or an equivalent replacement, to mirror the current provider pattern: add the Mantle manifest schema/UI hints and public docs, resolve current plugin config from `ctx.config`, pass typed config into `resolveImplicitMantleProvider`, gate before IAM token generation, and add focused false/true/default discovery tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Resolve the catalog gate from runtime config:** `extensions/amazon-bedrock-mantle/register.sync.runtime.ts:23-25`
  - body: The new guard reads `api.pluginConfig` captured at registration time, but catalog hooks receive the current runtime config in `ctx.config`. After a config reload or any catalog run against a different config object, `discovery.enabled: false` can be ignored and this path will still call `resolveImplicitMantleProvider`, reaching IAM token generation. Mirror the sibling Bedrock provider by resolving the plugin config from `ctx.config` and passing it into the resolver.
  - confidence: 0.84
