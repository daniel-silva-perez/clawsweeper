## Summary

Keep open. Current main still lacks the requested numeric-bound guidance, but this PR’s implementation is not ready to merge because the new formatter would also rewrite current non-numeric length/size errors and strict positive-number errors with misleading “allowed value” text; the referenced CI run is also still failing and needs maintainer triage after the code fix.

## What This Changes

The PR adds a config-validation helper that appends min/max hints to mapped Zod `too_big`/`too_small` issues and adds regression coverage for `session.agentToAgent.maxPingPongTurns`.

## Best Possible Solution

Keep the central mapper approach, but make the bound hinting follow Zod’s issue contract: only use numeric value wording for numeric origins, respect `inclusive:false` for strict bounds, leave length/count errors alone or give them origin-specific wording, and cover those cases in the focused validation test file before rerunning the changed gate.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: `session.agentToAgent.maxPingPongTurns: 10` exercises the missing max hint, while applying the PR helper to current `.positive()` and string/array `.max()` schemas reproduces the misleading-message regressions. Tests were not run because this review is read-only.

## Solution Assessment

No. The patch is aimed at the right mapper, but it is not the best safe fix until it uses Zod’s `origin` and `inclusive` fields so existing config validation errors stay accurate.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Limit bound hints to numeric Zod issues:** `src/config/validation.ts:495-498`
  - body: This helper runs for every Zod `too_big`/`too_small` issue, but current config schemas include string and array bounds such as `ui.assistant.name` and secret-provider argument arrays. Those failures would now say `maximum allowed value is N` even though the bound is a length or item count, making the validation guidance wrong. Check `record.origin` before adding value-bound wording, or use origin-specific text.
  - confidence: 0.9
- **[P2] Respect exclusive lower bounds:** `src/config/validation.ts:502-505`
  - body: Several current config fields use `.positive()`, and Zod implements that as `greater_than 0` with `inclusive:false`. With this change, setting one of those fields to `0` would report `minimum allowed value is 0` even though `0` is rejected. Use `record.inclusive` to choose strict `greater than`/`less than` wording before appending the hint.
  - confidence: 0.88
