## Summary

Keep open. The PR targets a valid config UX gap and current main still lacks numeric-bound enrichment, but the proposed mapper change should not merge as-is because it treats every Zod size/range issue as an inclusive numeric value constraint, which can misstate existing positive-number and string/array constraints. The current CI run for the PR head is also failed, so merge readiness is not established.

## What This Changes

The PR adds a central helper in `src/config/validation.ts` that appends maximum/minimum hints for Zod `too_big` and `too_small` config issues, plus a regression test for `session.agentToAgent.maxPingPongTurns`.

## Best Possible Solution

Keep the central mapper approach, but revise it to respect Zod metadata: only use numeric value wording for numeric origins, handle `inclusive: false` with strict inequality wording, and avoid value-bound hints for string/array/set/file length or size issues unless those messages are explicitly phrased as length/item/size limits. Add regression coverage for numeric max, numeric min, exclusive positive/greater-than behavior, and at least one non-numeric size constraint that should not receive a numeric value hint.

## Reproduction Assessment

Yes. Current main directly reproduces the linked user problem through `validateConfigObjectRaw({ session: { agentToAgent: { maxPingPongTurns: 10 }}})`, and the PR finding is reproducible from code by applying the new helper to existing `.positive()` and string `.max()` schemas.

## Solution Assessment

No, not as currently proposed. The central mapper is the right maintainable location, but the implementation needs to preserve Zod's `origin` and `inclusive` semantics before it can be the best fix.
