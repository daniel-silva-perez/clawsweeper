## Summary

Keep this PR open, but do not merge it as-is. The timeout-validation direction is useful, but the branch is stale against current main: it misses the active gateway-rpc runtime module and would change the current channels status probe default.

## What This Changes

The PR adds stricter CLI timeout parsing helpers, wires them into gateway, gateway-rpc, devices, and channels status timeout paths, and adds focused regression tests.

## Best Possible Solution

Land one rebased timeout-validation fix that rejects explicit invalid timeout values at the active gateway-facing CLI boundaries, applies validation in src/cli/gateway-rpc.runtime.ts, preserves existing defaults including gateway-rpc 30s and channels status probe 30s, and keeps focused tests for both Commander parsing and direct helper calls.

## Reproduction Assessment

Yes. A high-confidence static reproduction on current main is callGatewayFromCli('health', { timeout: 'nope', json: true }), which routes through gateway-rpc.runtime.ts and forwards Number('nope') as NaN; channelsStatusCommand({ probe: true }) without timeout is the default-preservation check.

## Solution Assessment

No. The proposed validation approach is the right product direction, but this branch is no longer the narrowest maintainable fix until it is rebased onto the runtime split and preserves the current channels status probe fallback.
