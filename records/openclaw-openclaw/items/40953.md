## Summary

Keep this PR open. It targets a real gateway-facing CLI timeout validation bug that still exists on current main, but the branch is stale/dirty and has blocking correctness issues against current main: the active gateway-rpc runtime path remains unvalidated after the runtime split, and the channels status change would shorten the probe default from 30s to 10s.

## What This Changes

The PR adds stricter shared timeout parsing helpers, wires them into gateway, gateway-rpc, devices, and channels status CLI timeout handling, and adds focused regression tests.

## Best Possible Solution

Land one rebased canonical timeout-validation PR that rejects explicit invalid timeout values at gateway-facing CLI boundaries, applies validation in src/cli/gateway-rpc.runtime.ts, preserves existing defaults including gateway-rpc 30s and channels status probe 30s, and keeps focused tests for Commander parsing plus direct helper calls.

## Reproduction Assessment

Yes. A high-confidence static reproduction on current main is callGatewayFromCli('health', { timeout: 'nope', json: true }), which delegates to gateway-rpc.runtime.ts and forwards NaN before callGateway falls back; channelsStatusCommand({ probe: true }) without timeout is the default-preservation check.

## Solution Assessment

No. The validation direction is correct, but this branch is no longer the best merge shape until it is rebased onto the runtime split and preserves current probe defaults; a canonical rebased PR should also account for the overlapping timeout PRs.
