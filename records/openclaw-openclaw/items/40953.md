## Summary

Keep open. Current main still has the loose timeout coercion the PR is trying to fix, but this branch is stale against main and has blocking correctness issues: it edits the old gateway-rpc call site rather than the active lazy runtime module, and it does not preserve the current channels status probe timeout default.

## What This Changes

The PR adds stricter CLI timeout parsing helpers, wires them into gateway, gateway-rpc, devices, and channels status timeout paths, and adds focused regression tests.

## Best Possible Solution

Land one rebased timeout-validation fix that rejects explicit invalid timeout values at the active gateway-facing CLI boundaries, applies validation in current runtime modules, preserves existing defaults including gateway-rpc 30s and channels status probe 30s, and keeps focused tests for both Commander parsing and direct-call helpers.

## Reproduction Assessment

Yes. The current-main reproduction path is a direct call through callGatewayFromCli with an invalid timeout, which reaches gateway-rpc.runtime.ts and forwards Number(...) output; channelsStatusCommand({ probe: true }) without timeout also has a clear static check for the 30s default that the rebased patch must preserve.

## Solution Assessment

No, not as proposed. The validation direction is sound, but the branch must be rebased onto the lazy runtime split and current channels status default before it is the narrowest maintainable fix.
