## Summary

Keep this PR open. Current main still has the gateway-facing CLI timeout validation gap the PR targets: gateway call, gateway-rpc, devices, and channels status paths register string `--timeout` options and still convert them with raw `Number(...)` before calling the gateway. The PR is stale against the current gateway-rpc runtime split and overlaps with narrower open timeout-validation work, but the central fix is not implemented on main and remains a plausible focused bug fix. Security review found no unexplained supply-chain, secrets, CI, dependency, or code-execution surface changes in the provided PR diff; it touches CLI parsing/call sites and tests only.

## Best Possible Solution

Keep this PR open for maintainer review or contributor rebase. The best path is to adapt the strict shared timeout parser to the current gateway-rpc runtime split, apply it to the gateway-facing call sites that still use `Number(...)`, preserve intended defaults, add focused regression tests for invalid input and valid passthrough, and coordinate with #54646/#59397 so duplicate timeout fixes converge.
