## Summary

Keep open. The PR targets a real current-main bug: gateway-facing CLI timeout options still pass unchecked `Number(...)` results into gateway calls. It is not merge-ready as-is because the branch predates the current `gateway-rpc` runtime split and would need a rebase that preserves current defaults, especially the `channels status --probe` fallback.

## What This Changes

The PR adds strict CLI timeout parsing helpers, wires them into gateway, gateway-rpc, devices, and channels status timeout paths, and adds focused regression tests.

## Best Possible Solution

Land one coordinated timeout-validation fix that keeps a shared strict parser, applies it to the active gateway-rpc runtime plus gateway call, devices, and channels status paths, preserves each command's current default timeout, and either folds or explicitly supersedes the narrower overlapping timeout PRs after review.
