## Summary

Keep this PR open, but do not merge it as-is. The reported timeout-validation bug is still present on current main, but the branch is dirty against main and its patch misses the active gateway-rpc runtime split while also changing the current channels probe default from 30s to 10s.

## What This Changes

The PR adds strict shared CLI timeout parsing helpers, wires them into gateway, gateway-rpc, devices, and channels status timeout handling, and adds focused regression tests.

## Best Possible Solution

Land one rebased canonical timeout-validation change that rejects explicit invalid timeout values at gateway-facing CLI boundaries, validates the active src/cli/gateway-rpc.runtime.ts path, preserves the current gateway-rpc 30s and channels probe 30s defaults, and keeps focused tests for Commander parsing plus direct helper calls.

## Reproduction Assessment

Yes. A high-confidence static reproduction on current main is callGatewayFromCli("health", { timeout: "nope", json: true }), which delegates to gateway-rpc.runtime.ts and forwards NaN before callGateway falls back; channelsStatusCommand({ probe: true }) without timeout is the default-preservation check.

## Solution Assessment

No. The validation direction is correct, but this branch is no longer the best merge shape until it is rebased onto the runtime split and preserves current probe defaults; maintainers should also account for the overlapping timeout PRs.
