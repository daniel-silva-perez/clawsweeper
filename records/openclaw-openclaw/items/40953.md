## Summary

Keep this PR open. Current main still has the gateway-facing timeout validation gap the PR targets: the gateway call, gateway-rpc runtime, devices CLI, and channels status paths still accept string `--timeout` options and forward raw `Number(...)` results into gateway calls. The PR is useful source work, but it needs rebase/adaptation to the current gateway-rpc runtime split and coordination with overlapping timeout-validation PRs.

## What This Changes

The PR adds strict CLI timeout parsing helpers, wires them into gateway, gateway-rpc, devices, and channels status call sites, and adds regression tests for invalid and valid timeout inputs.

## Best Possible Solution

Land one coordinated timeout-validation fix that adds a strict shared CLI timeout parser, applies it to the current gateway-rpc runtime split plus gateway call, devices, and channels status, preserves intended defaults for each command path, and includes focused regression tests for invalid input and valid passthrough. The narrower overlapping PRs should either be folded into that fix or clearly scoped so the behavior stays consistent.
