## Summary

Keep open. Current main does not yet contain the PR behavior, the linked Slack bug has source-backed reproduction and live logs for the finite reconnect cap, and the proposed one-line change matches the reconnect loop's existing unlimited-attempt convention while leaving separate transient reconnect-window issues for follow-up.

## What This Changes

The PR changes Slack Socket Mode reconnect policy from 12 attempts to unlimited by setting `SLACK_SOCKET_RECONNECT_POLICY.maxAttempts` to `0`.

## Best Possible Solution

Review and land this narrow policy change if maintainers agree Slack should reconnect indefinitely by default, preferably with focused Slack monitor coverage for the `maxAttempts: 0` path. Track reconnect-window delivery loss and upstream Slack SDK pong-stall behavior separately so this PR stays scoped to the permanent dead-connection failure.

## Reproduction Assessment

Yes. The linked bug discussion and current source give a high-confidence path: repeated Slack Socket Mode disconnect or start-failure events increment `reconnectAttempts` until the hardcoded cap throws, after which the Slack provider exits while the gateway remains otherwise usable.

## Solution Assessment

Yes for the permanent reconnect exhaustion bug. The patch uses the narrowest existing seam because the provider already treats `maxAttempts <= 0` as unlimited and still fails fast for known permanent auth errors; it is not intended to solve transient message loss during reconnect windows.
