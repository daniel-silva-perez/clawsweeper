## Summary

Keep open for maintainer review. The PR is a narrow one-line fix for a real Slack Socket Mode reconnect exhaustion path that is still present on current main, and I did not find a blocking correctness issue in the patch; it should not be closed because it is not merged and it is the open implementation path for the linked Slack bug.

## What This Changes

The PR changes Slack Socket Mode reconnect policy from 12 attempts to unlimited retries by setting `SLACK_SOCKET_RECONNECT_POLICY.maxAttempts` to `0`.

## Best Possible Solution

Land this narrow policy change if maintainers agree Slack should keep reconnecting indefinitely by default, ideally with focused coverage for the unlimited retry path. Track upstream Slack SDK pong-stall behavior and message loss during reconnect windows separately so this PR stays scoped to the permanent cap exhaustion failure.

## Reproduction Assessment

Yes. Repeated Slack Socket Mode disconnect or start-failure events on current main increment `reconnectAttempts` until the hardcoded `maxAttempts: 12` guard throws, and the linked bug discussion includes live `retry 1/12` logs matching that path.

## Solution Assessment

Yes for the permanent reconnect exhaustion bug. The patch uses the existing provider seam because current code already treats `maxAttempts <= 0` as unlimited and still fails fast for known permanent Slack auth errors; adding a new config surface would be broader than needed for this PR.
