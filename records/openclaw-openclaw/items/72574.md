Current main still has the reported failure boundary: Codex app-server attempts start through the shared client factory and the startup catch clears the shared singleton on error, while the linked open PR is the active implementation candidate and uses closing syntax for this issue.

## What This Changes

The issue asks for invalid or expired bearer-token failures from helper Codex app-server spawn runs to fail gracefully without disconnecting or destabilizing the active OpenClaw gateway/session.

## Best Possible Solution

Review and land PR #72812 or an equivalent narrow fix that gates isolated Codex app-server startup on real spawned/helper lineage, keeps ordinary runs on the shared client, and preserves explicit cleanup for startup failure, timeout, abort, turn-start failure, and completion paths.

## Solution Assessment

Yes. The linked PR's direction is the narrow maintainable fix because per-agent Codex state isolation on current main does not isolate helper runs within the same shared client lifecycle; spawnedBy-gated isolated startup addresses the reported boundary directly.
