## Summary

Keep open. Current main still starts Codex app-server attempts through the shared client path and clears that shared client on startup failure, while PR #72812 is open, unmerged, and explicitly proposes the narrow isolation fix for this issue.

## What This Changes

The issue reports that a spawned Codex helper run with an invalid bearer token returns 401 and then destabilizes the active gateway/session until the gateway is restarted.

## Best Possible Solution

Review and land PR #72812, or apply the same narrow Codex plugin change: spawned/helper attempts should use an isolated app-server client, avoid shared-client teardown on isolated auth/start failures, close isolated clients explicitly, and add regression coverage for that boundary.

## Reproduction Assessment

Yes. The issue provides concrete user steps, and current main gives a source-level reproduction path: a runCodexAppServerAttempt call with spawnedBy and a startup/auth failure still reaches the catch block that clears the shared Codex client.

## Solution Assessment

Yes. The narrow maintainable fix is to use existing spawnedBy metadata and the existing isolated-client seam in the Codex app-server harness; changing gateway lifecycle behavior would be broader than necessary.
