## Summary

Keep open. Current main still routes Codex app-server startup through the shared client and clears that shared client on startup failure, while open PR #72812 explicitly references this issue with closing syntax and proposes the narrow isolation fix.

## What This Changes

The issue requests that invalid or expired Codex bearer-token failures from `btw spawn` fail locally without disconnecting or destabilizing the active gateway/session.

## Best Possible Solution

Review and either land PR #72812 or apply the same narrow Codex plugin fix: spawned/helper runs should use an isolated app-server client, shared-client teardown should remain limited to non-isolated attempts, isolated clients should be explicitly closed, and regression coverage should protect startup/auth failure isolation.

## Reproduction Assessment

Yes. The issue provides concrete user steps, and current main gives a focused unit reproduction path: run `runCodexAppServerAttempt` with `params.spawnedBy` and a startup/auth failure, then assert the shared client is not cleared and only the child attempt fails.

## Solution Assessment

Yes. The narrow maintainable fix is to key off existing `spawnedBy` metadata and the existing isolated-client seam; changing gateway lifecycle or global auth behavior would be broader than necessary.
