## Summary

Keep open. Current main still routes Codex app-server startup through the shared client factory and clears that shared client on startup failure, while open PR #72812 explicitly references this issue with closing syntax and remains unmerged. Per policy, the issue should stay open until the paired fix PR lands or is replaced.

## What This Changes

Invalid or expired bearer tokens during `btw spawn codex 5.5` should fail locally without disconnecting the active gateway/session or requiring a manual gateway restart.

## Best Possible Solution

Review and either land PR #72812 or apply the same narrow Codex plugin fix: spawned/helper Codex app-server runs should use an isolated client, shared-client teardown should remain limited to non-isolated attempts, isolated clients should be explicitly closed, and regression coverage should protect auth/startup failure isolation.

## Reproduction Assessment

Yes. The issue provides concrete user steps, and current main gives a focused source-level reproduction path: run `runCodexAppServerAttempt` with `params.spawnedBy` and a startup/auth failure, then verify the shared app-server client is not cleared and only the child attempt fails.

## Solution Assessment

Yes. The narrow maintainable solution is to use the existing `spawnedBy` metadata and existing isolated-client seam; changing gateway lifecycle or global auth handling would be broader than necessary.
