## Summary

Keep open. Current main still has the reported failure-isolation risk, and PR #72812 is open, unmerged, and explicitly attempts to fix this issue with closing syntax, so the issue should remain open until that PR is reviewed and merged or closed.

## What This Changes

Invalid or expired bearer-token failures during `btw spawn codex 5.5` should fail locally without disconnecting the active gateway/session or requiring a gateway restart.

## Best Possible Solution

Review and either land PR #72812 or apply the same narrow Codex plugin fix: use an isolated app-server client for spawned/helper attempts, keep shared-client teardown limited to non-isolated attempts, explicitly close isolated clients, and retain regression coverage for startup/auth failures.

## Reproduction Assessment

Yes. The issue provides concrete user steps, and current-main code gives a focused unit reproduction path: run a Codex app-server attempt with `params.spawnedBy` and a startup/auth failure, then assert the shared client is not cleared and only the child attempt fails.

## Solution Assessment

Yes. The linked PR direction is the narrow maintainable solution because current main already has both `spawnedBy` metadata and an isolated-client seam; changing gateway lifecycle or global auth behavior would be broader than necessary.
