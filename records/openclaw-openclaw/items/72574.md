## Summary

Keep open. Current main still routes Codex app-server startup through the shared client and clears that shared client on startup/auth failure, while the linked open PR #72812 is the active implementation candidate for isolating spawned/helper runs.

## What This Changes

The issue reports that `btw spawn` with an invalid or expired Codex bearer token returns 401 and also destabilizes the active gateway/session instead of failing only the helper run.

## Best Possible Solution

Review and either land PR #72812 or apply the same narrow Codex plugin fix: spawned/helper app-server attempts should use an isolated client, shared-client teardown should remain limited to non-isolated runs, and regression coverage should prove auth/startup failures do not clear the live shared client.

## Reproduction Assessment

Yes. The issue gives concrete user steps, and current main has a focused source-level reproduction path: a spawned Codex app-server attempt that fails during startup/auth still goes through the shared client and the startup catch clears the shared singleton.

## Solution Assessment

Yes. The linked PR direction is the narrow maintainable fix because current main already has both `spawnedBy` metadata and an isolated-client seam; changing gateway lifecycle globally would be broader than necessary.
