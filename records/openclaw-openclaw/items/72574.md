## Summary

Keep open. Current main still starts spawned/helper Codex app-server attempts through the shared client and still clears that shared client on startup failure, while PR #72812 is open and unmerged with closing syntax for this issue; the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Invalid or expired bearer tokens during helper Codex spawning should fail locally without disconnecting or destabilizing the active OpenClaw gateway/session.

## Best Possible Solution

Review and land PR #72812, or apply the same narrow Codex plugin fix: spawned/helper runs should use an isolated app-server client, startup failures in that isolated path should not clear the shared singleton, isolated clients should be closed explicitly, and regression coverage should protect auth-failure isolation.

## Reproduction Assessment

Yes. The issue provides concrete user steps and logs, and current main gives a focused source-level reproduction: invoke runCodexAppServerAttempt with params.spawnedBy and a startup/auth failure, then assert the shared Codex app-server client is not cleared.

## Solution Assessment

Yes. The narrowest maintainable solution is to key off existing spawnedBy metadata and the existing isolated-client seam; changing gateway lifecycle or global auth behavior would be broader than necessary.
