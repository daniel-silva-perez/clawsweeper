## Summary

Keep open. Current main still routes Codex app-server harness startup through the default shared client factory and clears the shared client on startup failure, while PR #72812 is open, unmerged, and explicitly uses closing syntax for this issue. Repository policy says the issue should stay open until that implementation PR is reviewed and merged or closed.

## What This Changes

Running `btw spawn codex 5.5` with an invalid or expired bearer token should fail locally without disconnecting or destabilizing the active OpenClaw gateway/session.

## Best Possible Solution

Review and either land PR #72812 or apply the same narrow fix in the Codex plugin: choose an isolated app-server client for spawned/helper runs, keep shared-client teardown limited to non-isolated runs, explicitly close isolated clients during cleanup, and preserve regression coverage for invalid-auth startup failures.

## Reproduction Assessment

Yes. The issue provides concrete steps, and current-main inspection gives a focused unit reproduction path: run a Codex app-server attempt with `params.spawnedBy` and a startup/auth failure, then assert the shared client is not cleared and only the child attempt fails.

## Solution Assessment

Yes. The linked PR direction is the narrow maintainable solution because it uses existing spawned-run metadata and the existing isolated client seam, instead of changing gateway lifecycle or global auth behavior.
