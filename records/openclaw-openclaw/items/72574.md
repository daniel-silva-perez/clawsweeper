## Summary

Keep this open. The current main path still uses the shared Codex app-server startup client and clears the shared singleton on startup-path errors, while PR #72812 explicitly references this issue with closing syntax and remains the active unmerged fix vehicle.

## What This Changes

The issue reports that `btw spawn codex 5.5` with an invalid or expired bearer token returns a 401 and then disconnects the active gateway/session until the gateway is restarted.

## Best Possible Solution

Codex helper/spawned app-server startup should use an isolated client so auth failures fail only the child run, while ordinary Codex runs preserve the shared-client lifecycle and explicit cleanup on startup, timeout, abort, turn-start failure, and completion paths.

## Reproduction Assessment

Yes. The issue includes concrete steps and logs, and current main has a source-level reproduction path where `thread/start` auth failure propagates through shared app-server startup cleanup.

## Solution Assessment

Yes. The narrow maintainable direction is to gate isolated startup on real helper/spawned lineage rather than changing the broader shared-client contract for ordinary Codex runs.
