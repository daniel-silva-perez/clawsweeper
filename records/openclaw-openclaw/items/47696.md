## Summary

Keep open. Current main still does not implement temporary `/trust` or `/untrust` exec escalation, and this PR is useful source work, but it changes command-execution security semantics, is stale against the current command registry/handler layout, overlaps broader open trust-window PRs, and still needs maintainer/security design review before merge.

## What This Changes

The PR adds `/trust [minutes]` and `/untrust` slash commands, in-memory session-scoped trust-window state, an exec-security resolver hook, unit tests, slash-command docs, and a changelog entry.

## Best Possible Solution

Consolidate this with the broader trust-window work into one explicit trust-window authority that revalidates at execution time, has clear authorization boundaries, discloses active trust in status/help surfaces, handles queued runs intentionally, and includes focused docs and tests for the final chosen surface.

## Reproduction Assessment

Yes. Current-main absence is reproducible by inspecting the command registry, handler list, docs, and exec override resolver; the PR's queued-run issue is reproducible from the current queue path that stores `execOverrides` at enqueue time and replays them unchanged later.

## Solution Assessment

No. The feature direction is plausible, but implementing the trust window as a parse-time wrapper around `execOverrides.security` is not the safest maintainable design for time-bounded security; the enforcement should be checked at execution time and reflected in user-visible status.

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Revalidate trust before queued runs execute:** `src/auto-reply/reply/get-reply-directives.ts:75`
  - body: This snapshots `security=full` into `execOverrides` when the message is parsed. Queued/follow-up runs later replay `run.execOverrides` unchanged, so a message submitted near expiry can still execute with full exec security after the trust window has elapsed. The trust window needs to be rechecked at execution time or represented as an expiring authority instead of a stored security value.
  - confidence: 0.88
- **[P2] Disclose active trust in exec status:** `src/auto-reply/reply/get-reply-directives.ts:75`
  - body: This new resolver can make effective exec security `full`, but `/exec` status still reports only persisted/config-derived defaults. During an active trust window, users can see `security=deny` or `allowlist` while execution is actually trusted, so status should include the active trust override or remaining window.
  - confidence: 0.82
