## Summary

Keep open. Current main still does not implement the requested trust commands, and this PR remains an active implementation proposal, but the proposed patch introduces a security-sensitive exec elevation path with concrete queue-expiry and status-disclosure issues and overlaps broader open trust-window designs.

## What This Changes

The PR adds `/trust [minutes]` and `/untrust` chat commands, in-memory session-scoped trust-window state, an exec-security override in reply directive resolution, command registry/docs/changelog updates, and unit tests.

## Best Possible Solution

Consolidate the trust-window work into one explicit authority that is checked at execution time, handles queued and delayed runs intentionally, discloses active trust in status/help surfaces, keeps authorization boundaries clear, and lands with scoped docs, changelog, and tests for the chosen user surface.

## Reproduction Assessment

Yes. Current-main absence is reproducible by inspecting the command registry, handler list, docs, and exec override resolver; the PR's queued-run issue is reproducible from the diff plus the current queue path that stores and replays `execOverrides` later.

## Solution Assessment

No. The feature direction is plausible, but resolving trust into parse-time `execOverrides.security` is not the safest maintainable way to implement time-bounded security; the trust window should be revalidated at execution time and reflected in user-visible status.

Queued and follow-up runs later replay the stored `run.execOverrides`, so a run accepted near expiry can execute with `security=full` after the window elapsed. Represent trust as an expiring authority or recheck it at execution time.

Operators can be told `security=deny` or `allowlist` while trust is active, so status should include the override and remaining time.
