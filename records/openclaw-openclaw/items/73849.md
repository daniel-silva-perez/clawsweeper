## Summary

Keep this PR open, but do not merge it as-is. Current main still has not adopted the proposed `SuccessfulExit=false` launchd policy, and the PR conflicts with the current supervised restart/update paths that exit cleanly and rely on `KeepAlive=true`; it also carries unrelated sandbox Docker changes with a concrete runtime and supply-chain concern.

## What This Changes

The PR changes the macOS LaunchAgent plist, launchd audit check, and launchd test from boolean `KeepAlive=true` to `KeepAlive` with `SuccessfulExit=false`, while also modifying sandbox Docker/browser/doctor behavior for Docker daemon and image availability.

## Best Possible Solution

Keep the launchd problem open and replace this branch with a focused daemon-only fix that preserves explicit `openclaw gateway restart` and update restarts while improving crash-loop debuggability, likely by coordinating plist policy with supervised restart handoff or by adding a targeted backoff/disable-after-repeated-failures path. The sandbox Docker changes should stay out of this PR and continue honoring the no-silent-Debian-substitution contract.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: under launchd, current supervised restart and update paths exit with status 0 and rely on `KeepAlive=true`; applying `SuccessfulExit=false` would stop launchd from relaunching those clean exits. The Docker regression is also reproducible by missing `openclaw-sandbox:bookworm-slim` with Docker available, where the PR pulls and tags plain Debian instead of failing fast.

## Solution Assessment

No. The plist primitive is real, but using `SuccessfulExit=false` alone is not the best solution because it changes clean-exit behavior without updating the restart/update flow that depends on clean exits, and the PR also carries unrelated sandbox behavior. A safer solution is a focused launchd lifecycle change with tests for both crash-loop debugging and intentional restarts.
