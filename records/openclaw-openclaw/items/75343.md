## Summary

Keep this PR open. The reported Windows LSP `.cmd` shim failure is real on current main, and wiring the existing Windows spawn resolver into the LSP process path is the right direction, but the patch is not merge-ready because the resolver does not use the same merged environment that the child process receives and the user-facing fix lacks a changelog entry.

## What This Changes

The PR changes `spawnLspServerProcess` to resolve LSP server commands through `resolveWindowsSpawnProgram`, materialize the invocation argv, and pass the resolved `shell`/`windowsHide` options into `child_process.spawn`.

## Best Possible Solution

Update the PR so `spawnLspServerProcess` computes the merged child environment once, passes that same env into `resolveWindowsSpawnProgram`, and then uses it for `spawn`; add focused coverage for a Windows `.cmd` LSP shim and a single-line `CHANGELOG.md` fix entry.

## Reproduction Assessment

Yes. Current main statically reproduces the reported path because LSP servers are launched by direct `spawn(config.command, ...)`, while the repository's Windows spawn helper and tests show npm `.cmd` shims need explicit resolution on Windows.

## Solution Assessment

No, not as written. The selected resolver is the right shared seam, but the PR must use the same launch environment for resolution and spawning before it is the narrow maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Resolve LSP commands with the launch environment:** `src/agents/pi-bundle-lsp-runtime.ts:72-75`
  - body: `resolveWindowsSpawnProgram` defaults to `process.env`, but this spawn later runs with `{ ...process.env, ...config.env }`. If an LSP config adds or overrides `PATH`/`PATHEXT` to point at its server shim, the resolver will not see it and Windows can still hit the original direct-spawn failure or resolve a different parent PATH shim. Build the merged env once and pass it to both the resolver and `spawn`.
  - confidence: 0.88
- **[P3] Add the Windows LSP changelog entry:** `src/agents/pi-bundle-lsp-runtime.ts:76-84`
  - body: This is a user-facing `fix(lsp)` change for Windows plugin/bundle LSP startup, but the diff does not add a `CHANGELOG.md` entry. OpenClaw policy requires user-facing fixes to be listed under the active Fixes section, so the release notes would miss this repair.
  - confidence: 0.91

## Security Review

Status: needs_attention

Summary: The diff touches a command-execution path and currently can resolve the LSP executable against a different environment than the one passed to the spawned child.

Concerns:

- **[medium] Environment mismatch can select the wrong shim:** `src/agents/pi-bundle-lsp-runtime.ts:72`
  - body: Because the resolver defaults to `process.env` while the spawned LSP process receives the merged launch env, a config-pinned Windows `PATH` can be ignored and a different parent PATH command can be selected. Use one merged env for both lookup and spawn.
  - confidence: 0.82
