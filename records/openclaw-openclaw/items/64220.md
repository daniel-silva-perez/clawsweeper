## Summary

Keep open. Current main has partial cleanup for session switches, but the reported stale Control UI status gap remains: active/retrying compaction still has no age-bound cleanup, terminal chat/reconnect/clear-history/send-failure paths clear run or stream state without clearing compaction/fallback status, and the existing tests do not cover a missed compaction end after the reply is done. The related #39908 symptom was broader and is already closed, but this narrower frontend stale-state diagnosis is still actionable.

## What This Changes

The issue requests that the Control UI stop showing stale `Compacting context...` status when no compaction is actually running.

## Best Possible Solution

Control UI should have one shared cleanup path for compaction and fallback indicators: active/retrying compaction should be age-bounded, terminal chat events and reconnect/clear-history/send-failure paths should invoke the same cleanup, and tests should cover a missed compaction `end` after a final reply plus reconnect and clear-history recovery.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is: process a compaction `start` event, miss the corresponding `end`, then process an owned final chat event; current code clears `chatRunId` and `chatStream` but leaves active `compactionStatus`, so the renderer continues to show `Compacting context...`.

## Solution Assessment

Yes, the best fix is a narrow UI state-machine repair rather than a backend compaction change: add timer-aware cleanup in the Control UI and call it from all terminal/reset/reconnect paths while preserving the existing run/session filtering.
