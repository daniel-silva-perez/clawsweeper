## Summary

Keep this issue open. Current main still hard-blocks normal OpenAI Codex browser OAuth when the speculative TLS preflight classifies an error as a certificate failure, and the in-tree regression test preserves that exact behavior. The current docs offer device-code login as a workaround, and adjacent OpenAI Codex auth/profile/runtime fixes do not prove the reported macOS custom-CA browser-OAuth and runtime-fetch failure is fixed.

## What This Changes

The issue reports that openai-codex browser OAuth and runtime requests fail on macOS with TLS/fetch errors despite a working custom CA setup.

## Best Possible Solution

Keep the issue open until OpenAI Codex browser OAuth and runtime transport share a clear custom-CA/proxy/env story. The desired end state is either an advisory TLS preflight with actionable diagnostics before upstream browser OAuth starts, or a targeted repair that proves effective `NODE_EXTRA_CA_CERTS`/macOS trust handling and Codex runtime fetch dispatching work end to end, backed by regression coverage.
