## Summary

Keep this open. Current main still hard-blocks normal OpenAI Codex browser OAuth when the TLS preflight classifies a failure as certificate-related, and the current regression test explicitly preserves that behavior. A newer runtime transport fix is useful context for the fetch-failure side, but it does not fix the reported OAuth precheck blocker or prove macOS custom-CA runtime behavior end to end.

## What This Changes

The issue reports that OpenAI Codex OAuth and runtime requests should work when the operator has a valid custom CA setup via Node TLS environment variables.

## Best Possible Solution

Keep the issue open until the OpenAI Codex owner path has one coherent custom-CA/proxy/env story across browser OAuth preflight and runtime requests. The likely end state is either an advisory preflight with actionable diagnostics or targeted custom-CA validation/repair, backed by regression coverage for certificate-class preflight behavior and Codex runtime transport behavior.

## Reproduction Assessment

Partially yes. The OAuth precheck blocker is reproducible from the current unit test that mocks `UNABLE_TO_GET_ISSUER_CERT_LOCALLY` and verifies OAuth never starts; the exact macOS custom-CA runtime fetch failure still needs live or targeted transport reproduction.

## Solution Assessment

No. The current implementation is not the best solution for the reported behavior because it hard-blocks browser OAuth on a speculative TLS preflight while the docs only offer device-code as a workaround; a narrower maintainable fix needs owner-approved preflight policy plus runtime custom-CA proof.
