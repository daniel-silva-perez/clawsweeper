## Summary

Keep open. Current main still treats inbound audio as a media-understanding/STT transcription surface and forwards only images as native prompt media to the embedded agent. The related open PR #48900 is a plausible implementation candidate, but it has not merged and the requested opt-in raw audio forwarding for WhatsApp/Telegram voice notes remains missing.

## Best Possible Solution

Keep this issue open and review or evolve PR #48900 into a shared implementation: add an opt-in native-audio config, capability detection for audio-capable primary models, secure audio staging and prompt content-part forwarding, provider compatibility checks, STT fallback, and transcript/hash/duration metadata so WhatsApp and Telegram voice notes can reach omni-modal models without losing auditability.

## Risks / Open Questions

- Closing now would strand an unimplemented user-facing feature: WhatsApp/Telegram voice notes still reach the agent as STT text, not raw audio content for omni-modal models.
- The existing model/config metadata can mention audio input, which may make native audio look supported even though the primary run pipeline lacks an audio content-part path.
- A correct implementation needs payload limits, secure staging/cleanup, provider content-block compatibility, STT fallback, and audit/continuity metadata such as hash, duration, transcript, or model interpretation.
