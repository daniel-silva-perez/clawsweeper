## Summary

Current main still does not implement this PR’s intended direct tool-result reply behavior. The embedded Pi subscription path has normal tool-result output, media buffering, and after_tool_call hooks, but no directReply marker contract, no direct-reply helpers, and no wiring that lets a tool-end handler emit a block reply and abort the active run before the follow-up model pass. The PR remains a live implementation candidate that needs maintainer product/security review rather than cleanup closure.

## Best Possible Solution

Keep this PR open for maintainer review. If OpenClaw wants the capability, land it as a generic, typed tool-result contract with explicit trust policy, abort reason typing, media filtering through the existing block-reply path, focused embedded-run regression tests proving one direct reply and no follow-up model inference, and docs/changelog cleanup. If maintainers prefer a different product direction, close only after that decision is explicit.

## Risks / Open Questions

- Security/product boundary remains the main review point: directReply would let successful tool output bypass final LLM synthesis and deliver text/media directly to the user, so maintainers should decide whether it is allowed for all tools, trusted tools only, or a declared capability.
- Media delivery needs explicit regression coverage against the existing filterToolResultMediaUrls local-media trust checks before merge.
- The contract should be durable and typed across OpenClaw/Pi tool-result shapes rather than an incidental extra property that only one handler understands.
