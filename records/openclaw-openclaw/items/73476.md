## Summary

Keep this PR open. Current main does not implement the requested direct-reply seam, but the latest patch still has a blocking delivery-policy bug: it can abort the agent run as a successful direct reply even when current source-reply or silent-turn policy suppresses the block delivery.

## What This Changes

The PR adds an optional `directReply` flag and helper constructors for tool results, wires a run-abort callback through embedded agent subscription, emits direct tool output as block replies, treats `direct_reply` aborts as terminal success, adds tests and a changelog entry, and includes parity-gate/QA/test stabilization changes.

## Best Possible Solution

Land a narrow, typed direct-reply contract only after it is policy-aware: the direct path should know whether source delivery is allowed or actually queued, abort only after allowed/successful delivery, fall back to normal inference or message-tool guidance when delivery is suppressed, add regression coverage for `message_tool_only` and `silentExpected`, and split unrelated workflow/QA stabilization from the feature branch.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a successful tool result with `{ directReply: true, content: [{ type: "text", text: "done" }] }` during a default group/channel or explicit `message_tool_only` turn; current delivery wrappers drop the block reply while this PR still aborts further inference as `direct_reply`.

## Solution Assessment

No. The direct-reply seam may be the right product direction for external-agent tools, but this implementation is not the safest merge path until delivery policy or delivery success is threaded into the abort decision and unrelated workflow changes are separated.
