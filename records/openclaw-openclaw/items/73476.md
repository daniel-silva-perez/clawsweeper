## Summary

Keep this PR open. Current main still does not implement the proposed directReply tool-result contract, direct-reply helpers, or the abort/emit wiring needed for a tool result to bypass the follow-up model inference. The PR remains a live implementation candidate, and the latest supplied PR context shows the author addressed the earlier bot P2 feedback, but the behavior still needs maintainer product/security review before merge.

## Best Possible Solution

Keep this PR open for maintainer review. If OpenClaw wants this capability, land it as a generic typed tool-result contract with an explicit trust policy, typed abort reason, media filtering through the existing block-reply path, docs/changelog cleanup, and an embedded-run regression test proving the direct reply is delivered once while the follow-up model inference is skipped. If maintainers prefer ACP/subagent/external-harness delivery instead, close only after that product decision is explicit.

## Risks / Open Questions

- Security/trust boundary: this PR would let successful tool output bypass final LLM synthesis and send text/media directly to users. Maintainers should decide whether that is allowed for all tools, trusted tools only, or an explicit capability.
- Media delivery needs explicit review. Current main has filterToolResultMediaUrls for local-media trust checks, and the supplied PR diff appears to call it, but direct block delivery should be tested against local-media and external/MCP provenance restrictions.
- Contract durability: the proposed marker should become a typed, documented OpenClaw tool-result contract rather than an incidental extra property on an upstream Pi result shape.
- PR hygiene still needs maintainer review, especially the changelog placement/wording and attribution rules before merge.
