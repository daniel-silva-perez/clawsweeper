## Summary

Keep this PR open. Current main still lacks a direct-reply tool-result contract, so the proposed feature remains relevant, but the current patch should not merge as-is because the new direct block-reply path bypasses existing silent-turn visibility guards and needs maintainer review for the tool trust model.

## What This Changes

The PR adds a `directReply` marker and helper functions for tool results, wires an abort callback through the embedded PI subscription path, emits tool text/media directly as block replies, treats `direct_reply` aborts as successful, updates tests/changelog, and adds a Node 24 workflow env override to the parity gate.

## Best Possible Solution

Land a typed direct-delivery contract only after it preserves the existing silent/group visibility rules, defines direct-reply eligibility for core and plugin tools, keeps media filtering and redaction behavior intact, documents the helper/API surface, and adds tests proving both visible and silent-turn behavior.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: the PR diff adds an unconditional `emitBlockReply`/`abortRun` direct-reply branch, while current main suppresses visible assistant and block-reply delivery under `silentExpected`.

## Solution Assessment

No. The feature direction may be useful, but this patch is not the best form until the direct-reply branch receives the same visibility policy inputs as normal delivery and the trust boundary is documented and tested.
