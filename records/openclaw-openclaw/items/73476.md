## Summary

Keep this PR open. The direct-reply seam is a plausible core feature and current main does not already implement it, but the latest PR still aborts the run after a block-reply callback that current delivery policy can intentionally suppress, so it can end group/channel or silent turns with no visible response; the branch also mixes in unrelated workflow and test-stabilization changes that need maintainer review.

## What This Changes

The PR adds a `directReply` tool-result flag, helper constructors, embedded-runner subscription wiring, direct block-reply emission with run abort, tests, a changelog entry, and unrelated parity-gate/qa-lab/test assertion updates.

## Best Possible Solution

Land a narrow, typed direct-reply contract only after it honors source-reply visibility and silent-turn policy, aborts only when direct delivery is allowed or known to have happened, adds focused regression coverage for suppressed-delivery cases, documents the public helper/API surface if it is meant for plugins, and splits unrelated CI/test stabilization out of the feature branch.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a successful tool result with `{ directReply: true, content: [{ type: "text", text: "done" }] }` in a group/channel turn using `message_tool_only` or a `silentExpected` run; current delivery wrappers drop the block reply while the PR branch still aborts further inference.

## Solution Assessment

No. The direct-reply idea may be the right seam for external-agent tools, but this implementation is not the safest merge path until delivery policy is threaded into the branch and unrelated workflow/test changes are separated.
