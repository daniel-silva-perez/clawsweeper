## Summary

Keep this PR open. Current main does not implement the proposed direct tool-result delivery contract, and the PR remains a relevant implementation candidate, but the latest diff still has a blocking test compile error plus an unresolved runner lifecycle path where `direct_reply` aborts would be reported as prompt errors instead of intentional completion.

## What This Changes

The PR adds an optional `directReply` flag and helper constructors for tool results, wires an abort callback through the embedded subscription path, emits direct text/media block replies from successful tool results, updates related tests, and adjusts several test/CI assertions plus the changelog.

## Best Possible Solution

Land this only if maintainers want a documented, typed direct-delivery contract: keep sanitized text and existing media filtering, define the trust/eligibility policy for plugin and MCP tools, handle `direct_reply` as intentional successful completion in runner telemetry/fallback, repair the kitchen-sink test binding, and split or clearly justify unrelated test assertion churn.
