## Summary

Keep this PR open. Current main does not implement a direct-reply tool-result contract, so the proposal is still relevant, but the current patch should not merge as-is because the new direct block-reply path bypasses the existing silent-turn visibility guards and needs maintainer review for the tool trust model.

## What This Changes

The PR adds a `directReply` marker and helper functions for tool results, wires `abortRun` through the embedded PI subscription path, emits tool text/media directly as block replies, treats `direct_reply` aborts as successful, updates tests/changelog, and adds a Node 24 workflow env override to the parity gate.

## Best Possible Solution

Land a typed direct-delivery contract only after it preserves the existing silent/group visibility rules, defines direct-reply eligibility for core and plugin tools, keeps media filtering and redaction behavior intact, documents the helper/API surface, and adds tests proving both visible and silent-turn behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Respect silent turns before direct replies:** `src/agents/pi-embedded-subscribe.handlers.tools.ts:1196-1201`
  - body: The new direct-reply branch emits a block reply from a tool result without checking `silentExpected`. Normal assistant/block delivery suppresses visible output for silent or group-tool-only turns, so a tool returning `directReply: true` can now post text/media visibly and abort the turn where the existing policy would keep output private. Thread the silent visibility policy into this handler before emitting or aborting.
  - confidence: 0.88
