## Summary

Current main still does not implement this PR’s intended direct tool-result reply behavior. The PR remains the active implementation candidate: the shipped embedded Pi tool-end path emits normal tool output/media, fires after_tool_call, and then lets the agent loop continue, but there is no directReply/direct_reply contract, no helper constructors, and no abortRun/emitBlockReply seam exposed to the tool-completion handler. The provided PR discussion is also consistent with keeping it open: Greptile’s P2 review comments appear addressed by later PR commits, while the remaining question is maintainer product/security review of whether successful tool output may bypass final LLM synthesis.

## Best Possible Solution

Keep this PR open for maintainer review. If OpenClaw wants the capability, land it as a generic, typed tool-result contract with explicit trust policy, typed abort reason, media filtering through the existing safe block-reply path, focused embedded-run tests proving direct delivery without a follow-up model pass, and docs/changelog cleanup. If maintainers prefer a different product direction, close only after that decision is explicit.

## Risks / Open Questions

- Direct replies would let successful tool output bypass final LLM synthesis and deliver text/media directly to the user; maintainers should decide whether this is allowed for all tools, trusted tools only, or an explicit declared capability.
- The contract should be durable and typed across OpenClaw/Pi tool-result shapes and middleware/serialization boundaries, not only an incidental marker understood by one handler.
- Media delivery should stay routed through existing filterToolResultMediaUrls/local-media trust checks and have focused regression coverage proving one direct user reply and no follow-up model inference.
