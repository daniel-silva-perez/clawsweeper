## Summary

Current main still does not implement this PR's intended direct tool-result reply behavior. The embedded Pi tool-end path emits ordinary tool result output, runs after_tool_call hooks, and then returns without any directReply/direct_reply contract, direct reply helper, or abortRun seam available to the tool handler. The PR remains the active implementation candidate, with the main remaining question being maintainer product/security review of whether successful tool output may bypass the final LLM synthesis pass.

## Best Possible Solution

Keep this PR open for maintainer review. If OpenClaw wants this capability, land it as a generic, typed tool-result contract with an explicit trust policy, safe media filtering through the existing block-reply path, a typed abort reason, focused embedded-run tests proving direct delivery without a follow-up model pass, and docs/changelog cleanup. If maintainers prefer a different product direction, close the PR only after that decision is explicit.
