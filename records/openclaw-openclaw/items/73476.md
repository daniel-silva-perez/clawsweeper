## Summary

Keep this PR open. Current main does not implement a `directReply` tool-result contract, and the contributor’s external-agent/sandbox use case is concrete, but the latest patch still needs maintainer review because it can emit a visible direct block reply without honoring existing silent/group visibility policy and it changes the tool-result trust boundary.

## What This Changes

The PR adds optional `directReply` tool-result helpers, wires `abortRun` through the embedded PI subscription and run attempt path, emits successful tool text/media directly as block replies, treats `direct_reply` aborts as successful, adds tests, and updates changelog/test assertions.

## Best Possible Solution

Land a typed direct-delivery contract only if maintainers accept the product and security model: preserve `silentExpected`/group visibility behavior, define which tools or plugin classes may use the marker, keep existing text sanitization and media filtering, document the helper/API, and prove end-to-end that the turn completes without a follow-up model pass or error telemetry.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Respect silent turns before emitting direct replies:** `src/agents/pi-embedded-subscribe.handlers.tools.ts:1196-1201`
  - body: The new direct-reply branch calls `emitBlockReply` without checking `silentExpected`. Existing block streaming suppresses visible output in silent/group-tool-only turns, so a tool result with `directReply: true` can post visibly where normal final answers are meant to stay private and then abort the turn. Thread the visibility policy into this handler and skip or require the existing message tool in those turns.
  - confidence: 0.87
