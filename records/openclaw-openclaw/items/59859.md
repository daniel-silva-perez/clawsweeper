## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve non-cron schedules when editing jobs:** `apps/linux/src/section_cron.c:293-296`
  - body: The edit save path always serializes `patch.schedule` as `{ kind: "cron", expr: ... }`. Current cron jobs can use `kind: "every"` or `kind: "at"`, so saving one of those jobs after changing unrelated fields can fail validation or silently change when it runs. Carry the original schedule variant through the dialog or block editing unsupported schedule types.
  - confidence: 0.9
- **[P2] Preserve session-scoped cron targets:** `apps/linux/src/test_seams.c:42-47`
  - body: Unknown persisted targets, including valid `session:<id>` cron targets, are mapped to index 0 and saved back as `isolated`. Editing a session-scoped job can therefore retarget it away from the intended session even when the user only changes another field.
  - confidence: 0.86
- **[P2] Send supported verbose levels:** `apps/linux/src/section_sessions.c:176-178`
  - body: The Verbose dropdown sends `none`/`low`/`high`, but `sessions.patch` accepts `off`/`on`/`full`. These updates are rejected by the server, so users cannot change session verbose behavior from the Linux app.
  - confidence: 0.88
- **[P2] Resolve local session logs from session IDs:** `apps/linux/src/section_sessions.c:265-266`
  - body: The Log action builds `<storeDir>/<session-key>/transcript.jsonl`, but the gateway stores and resolves transcripts by `sessionId` or `sessionFile` under the sessions directory. This usually misses existing local transcripts and falls back to the dashboard, breaking offline or no-dashboard access.
  - confidence: 0.87
- **[P2] Bracket IPv6 hosts before composing URLs:** `apps/linux/src/gateway_config.c:1034-1040`
  - body: The HTTP and WebSocket URL builders interpolate `config->host` directly into `scheme://host:port`. For an IPv6 literal such as `::1`, that produces malformed URLs like `http://::1:18789`, breaking health checks, WebSocket connection, and dashboard handoff on valid IPv6 setups.
  - confidence: 0.84
- **[P2] Re-enable the probe button after a successful refresh:** `apps/linux/src/section_channels.c:87-93`
  - body: `Probe All Channels` disables the clicked button, but the success path only updates status and starts a refresh; it never re-enables the header button, which is outside the rebuilt list. After one successful probe, the control stays disabled until the whole section is rebuilt.
  - confidence: 0.82
