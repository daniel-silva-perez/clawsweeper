## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve schedule variants when editing cron jobs:** `apps/linux/src/section_cron.c:270-275`
  - body: The edit save path always sets `schedule_kind` to `"cron"` and passes the textbox as `expr`. Current gateway jobs may use `kind: "every"` or `kind: "at"`, so saving an existing non-cron job after changing unrelated fields can fail validation or change when it runs. Carry the original schedule variant through the dialog or block editing unsupported schedule types.
  - confidence: 0.9
- **[P2] Resolve session logs from session transcript candidates:** `apps/linux/src/section_sessions.c:265-266`
  - body: `on_session_log` probes `<storeDir>/<session-key>/transcript.jsonl`, but the gateway resolves transcripts by `sessionId` and optional `sessionFile` under the sessions directory. Normal local transcripts will be missed, forcing a dashboard fallback and breaking offline or no-dashboard access.
  - confidence: 0.87
- **[P2] Bracket IPv6 literals in gateway URLs:** `apps/linux/src/gateway_config.c:1032-1040`
  - body: The URL builders interpolate `config->host` directly into `scheme://host:port`. For an IPv6 literal such as `::1`, this produces malformed URLs like `http://::1:18789`, breaking health checks, WebSocket connection, and dashboard handoff on valid IPv6/custom-bind setups.
  - confidence: 0.84
- **[P2] Resolve cron logs from session transcript candidates:** `apps/linux/src/section_cron.c:186-190`
  - body: The cron Log action derives a local path under `<dirname(cronStore)>/sessions/<sessionKey>/transcript.jsonl`, but cron status points at the cron store while session transcripts live under the agent sessions layout and are resolved from session IDs/files. This usually misses existing local transcripts before falling back to the dashboard.
  - confidence: 0.82
