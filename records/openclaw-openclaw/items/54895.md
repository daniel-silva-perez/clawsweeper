## Summary

Keep open. Current main still lacks inter-session progress streaming for long-running `sessions_send` runs: the tool has only fire-and-forget or synchronous wait behavior, wait expiry is still surfaced as `status: "timeout"`, A2A announce is completion-only, and the gateway has no agent-callable run-event read/subscription API. Adjacent Task Flow and ACP parent-streaming support cover different spawn/task flows, not observing a peer run initiated through `sessions_send`.

## What This Changes

Request an inter-session progress stream or subscription for long-running `sessions_send` runs so callers can distinguish accepted delivery from wait timeout and observe tool, assistant, and lifecycle progress.

## Best Possible Solution

Add an additive, permission-checked core run-event observation surface, likely backed by bounded per-run event buffering, then expose it through either a `sessions_subscribe` tool or an explicit `sessions_send` progress option. The implementation should also separate accepted delivery from wait timeout and add regression coverage for long-running sends, event isolation, ACP/Codex behavior, and Telegram/topic-bound delivery.
