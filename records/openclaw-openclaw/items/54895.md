## Summary

Current main still does not provide inter-session progress streaming for `sessions_send`. The tool can return `accepted` for fire-and-forget or synchronously wait and return `timeout`, but it has no stream/subscription option, no `sessions_subscribe` tool, no `agent.events` gateway read API, and the A2A announce path remains completion-only. Adjacent ACP `sessions_spawn streamTo="parent"` support is useful, but it only applies to ACP spawn flows and does not let a caller observe a long-running run initiated via `sessions_send`.

## Best Possible Solution

Keep this issue open. The best path is an additive, permission-checked core run-event observation surface, likely `agent.events` with bounded per-run event buffering, plus either a `sessions_subscribe` tool or an explicit `sessions_send` progress option. `sessions_send` should also distinguish accepted delivery from wait timeout, and coverage should include long-running ACP/Codex sends, event visibility, routing isolation, A2A final announce behavior, and Telegram/topic delivery.

## Risks / Open Questions

- A future run-event observation API needs explicit authorization so one session cannot inspect unrelated runs.
- Forwarding tool args, tool results, or assistant deltas could expose sensitive intermediate data unless payloads are gated, summarized, or redacted.
- Messaging-channel progress forwarding needs throttling and delivery policy, especially for Telegram topics and other rate-limited channels.
