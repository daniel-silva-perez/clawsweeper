## Summary

Keep this issue open. Current main still emits the reported TUI pairing-required hint without the concrete approval command, and PR #73208 is an open, unmerged implementation that targets the same issue with closing syntax.

## What This Changes

The issue asks for the TUI pairing-required disconnect message to include `openclaw devices approve <requestId>` after telling users to run `openclaw devices list`.

## Best Possible Solution

The desired end state is for the TUI disconnect hint to match the documented devices flow by naming both `openclaw devices list` and `openclaw devices approve <requestId>`, preferably substituting the parsed request ID when the gateway close reason includes one.

## Reproduction Assessment

Yes. The current-main helper `resolveGatewayDisconnectState("gateway closed (1008): pairing required")` deterministically returns a hint that lacks `openclaw devices approve <requestId>`, and the existing unit test only locks the list command.

## Solution Assessment

Yes, the narrow TUI copy and test change in PR #73208 is the right fix boundary for the reported UX gap. A slightly better refinement would reuse `readConnectPairingRequiredMessage` so the hint can show the actual sanitized request ID when available.
