## Summary

Keep open. Current main does not implement `executeTool` on `OpenClawPluginApi`, and this PR is still an active implementation proposal rather than obsolete shipped work. It needs rebase plus security/contract fixes before it should be considered merge-ready.

## Best Possible Solution

Keep this PR open as the active proposal, but require maintainer design review before merge. If OpenClaw wants this core SDK seam, revise it to resolve tools by actual factory output names, preserve own-plugin isolation, enforce optional-tool allowlists, account for `before_tool_call`/trusted policy/owner-only behavior or clearly document the narrower trust boundary, add focused registry/API contract tests, update plugin SDK docs, and rebase on current main.

## Risks / Open Questions

- The proposed direct execution path is security-sensitive: it must either intentionally share the normal tool policy/hook/audit path or document and enforce a narrower trust model that maintainers accept.
- The PR’s described `entry.names` lookup can miss factory-registered tools that only reveal their names after resolving the factory.
- Optional plugin tools should not become callable through the plugin API when the user has not enabled them via the existing allowlist policy.
- The PR appears stale against current main in test helper layout and needs rebase plus focused contract tests/docs for a public SDK API addition.
