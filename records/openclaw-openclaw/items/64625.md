## Summary

Keep open. Current main still has the Slack custom-invoke path described in the report: core supplies agent-scoped media access on the channel action context, but the bundled Slack wrapper forwards only raw `toolContext` when `options.invoke` is present, so `upload-file` can still reach the Slack runtime without the local roots needed for agent workspace files. The related fix PR is closed and unmerged.

## What This Changes

Slack `message` tool `upload-file` should preserve agent-scoped `mediaLocalRoots` and `mediaReadFile` when the bundled Slack plugin routes through its custom invoke callback.

## Best Possible Solution

Fix the bundled Slack plugin by constructing one augmented action context from `toolContext` plus top-level `ctx.mediaLocalRoots` and `ctx.mediaReadFile`, then pass that same trusted context through both the custom invoke and fallback runtime paths with focused regression coverage for `upload-file`.

## Reproduction Assessment

Yes. A source-level reproduction is to call `createSlackActions` with `options.invoke` and a `ChannelMessageActionContext` containing `mediaLocalRoots`/`mediaReadFile`; current code forwards only `ctx.toolContext`, so the invoked Slack upload path lacks the agent-scoped roots. The reporter's live gateway confirmation aligns with the current code path.

## Solution Assessment

Yes. The narrow maintainable fix is to augment the Slack action context at the wrapper boundary before invoking either runtime path; broad core changes or relaxing the local-media guard would be less safe.
