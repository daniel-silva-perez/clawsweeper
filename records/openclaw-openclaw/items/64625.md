## Summary

Keep open. Current main still drops agent-scoped Slack media access on the bundled custom invoke path: core supplies mediaLocalRoots/mediaReadFile on the top-level channel action context, but createSlackActions forwards only the raw toolContext when options.invoke is present. The closed #64632 PR targeted the same defect but was not merged, and the current Slack tests still do not cover the custom-invoke upload-file bridge.

## What This Changes

Ensure Slack upload-file message actions preserve agent-scoped mediaLocalRoots and mediaReadFile across the bundled custom invoke path.

## Best Possible Solution

Fix this in the bundled Slack plugin by augmenting the action context once from toolContext plus ctx.mediaLocalRoots/mediaReadFile and passing that same context through both custom invoke and fallback runtime paths, with focused regression coverage for upload-file through createSlackActions(options.invoke).
