## Summary

Keep this issue open. Current main still does not expose `sessionModelOverride`, `sessionProviderOverride`, or `modelOverrideSource` on `PluginHookAgentContext` for `before_model_resolve`; the hook receives prompt/attachment input and can return overrides, but cannot tell from its hook context whether the current provider/model came from a user-pinned session override versus auto fallback state.

## Best Possible Solution

Keep this issue open and implement an additive plugin SDK contract: plumb the prepared session provider/model override values, plus `modelOverrideSource` or an explicit user-pinned flag, into `PluginHookAgentContext` before `before_model_resolve` runs. Add focused hook wiring tests around the embedded runner setup and update `docs/plugins/hooks.md` so routing plugins can yield when the user has pinned a session model.

## Risks / Open Questions

- Closing now would leave routing plugins without a stable hook-context signal for user-pinned `/model` choices, so they may still overwrite explicit user selections or keep relying on session-store reads during `before_model_resolve`.
- Adding only raw `sessionModelOverride` and `sessionProviderOverride` may be incomplete because current behavior also depends on `modelOverrideSource` or an equivalent user-pinned boolean to distinguish user overrides from auto fallback state.
