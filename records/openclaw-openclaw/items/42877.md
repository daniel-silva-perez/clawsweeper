## Summary

Keep this issue open. Current main still exposes memory-core as recall/read tooling plus read/injection limits, not the requested built-in bounded memory CRUD/write surface with hard per-store character budgets, over-limit rejection, substring replace/remove, duplicate handling, or credential screening. The KinthAI comment reinforces the same memory-bloat problem, and the related draft PR #70540 is an add-only first slice rather than a full close-out for this design.

## What This Changes

Request a built-in bounded memory tool for add/replace/remove operations with configurable hard character limits, duplicate detection, usage feedback, snapshot injection, and optional credential-pattern screening.

## Best Possible Solution

Keep this open for memory-core product/API design. The maintainable end state is a model-facing memory write surface in the default memory plugin with config-backed per-store character budgets, atomic add/list/replace/remove behavior, duplicate handling, credential-pattern screening, and over-limit responses that provide current usage plus entries for consolidation.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/API request. Current-main verification is high-confidence because the registered tools, schemas, docs, tool catalog, and gateway allowlist all expose only memory_search/memory_get and read/injection limits.

## Solution Assessment

No proposed patch is present on this issue, and the related draft PR is only a partial add-only step. The best path is a memory-core-owned CRUD/budget design with tests and docs, coordinated with #70540 rather than closing this issue early.
