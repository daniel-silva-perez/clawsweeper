## Summary

Keep this issue open. Current main still intentionally blocks ACP spawns from sandboxed requester sessions, documents that ACP runs on the host rather than inside the OpenClaw sandbox, and has tests asserting the block. The related closed issue is the predecessor for the same design problem, not a shipped implementation or active canonical replacement.

## Best Possible Solution

Keep this open for maintainer/product design. The likely path is a narrow core ACP/sandbox boundary change: either an auditable host-side ACP bridge/proxy, ACP sessions launched in their own sandbox, or explicit support for running ACP inside the requester sandbox. The implementation should update `src/agents/acp-spawn.ts`, ACP availability/tool advertising, `/acp` command handling, config/schema/docs, and the existing sandbox-block tests to prove the new opt-in mode preserves the sandbox boundary.

## Risks / Open Questions

- Any implementation crosses the sandbox-to-host boundary and needs an explicit security model, opt-in operator controls, audit logging, and tests that prove sandboxed sessions cannot get general host execution through the ACP bridge.
- Closing this as stale, duplicate, or ClawHub-only would lose a real current limitation that is still documented, enforced in core, and covered by regression tests.
