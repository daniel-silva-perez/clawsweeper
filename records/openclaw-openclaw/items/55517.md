## Summary

Keep open: current `main` still routes plugin runtime subagent runs through direct Gateway `agent` dispatch without the registry metadata that drives the completion announce flow. The report is valid, but the next step is an explicit SDK/API contract decision because changing default auto-delivery semantics could affect existing plugin behavior.

## What This Changes

The issue requests a supported path for child runs started by plugin `api.runtime.subagent.run()` inside `before_dispatch` to invoke `subagent_delivery_target` and deliver the final result back to the external requester.

## Best Possible Solution

Define an explicit plugin-runtime deferred completion contract, then either implement it by registering plugin-created child runs with the existing `registerSubagentRun(...)` and `runSubagentAnnounceFlow(...)` machinery, or document `api.runtime.subagent.run()` as manual-only with plugin-owned polling/delivery. The implementation should include SDK docs and focused tests for a `before_dispatch` plugin run that returns immediately while the child completes later.
