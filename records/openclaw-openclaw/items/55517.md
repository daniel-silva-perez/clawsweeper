## Summary

Keep open: current main still routes `api.runtime.subagent.run()` through the Gateway `agent` method without creating the subagent registry record that drives completion announce delivery, so the reported `subagent_delivery_target` and final external-delivery gap remains valid. The open question is SDK/API semantics, not repro quality.

## What This Changes

The issue requests a supported deferred completion-delivery path for plugin subagent runs started from `before_dispatch`, including `subagent_delivery_target` resolution back to the original external requester.

## Best Possible Solution

Define an explicit plugin-runtime deferred completion contract, then either add a deliberate opt-in/default registration path that reuses the existing subagent registry and announce machinery, or document plugin-created runs as manual polling/delivery only. The chosen path should include SDK docs and a regression test for a `before_dispatch` hook that returns immediately while the child finishes later.
