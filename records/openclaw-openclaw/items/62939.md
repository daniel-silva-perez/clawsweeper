## Summary

Keep open. Current main has useful partial prompt-injection hardening for external web/file/hook/media content and inbound system-marker spoofing, but it still does not implement the requested shared trust-boundary contract for non-owner current message bodies, generic tool results, or stable system-prompt guidance. The linked narrow PR is closed unmerged, and the broader CaMeL PR remains open rather than shipped.

## What This Changes

The issue requests structural prompt-injection defenses that mark externally sourced tool results, fetched content, file/API content, and non-owner messages as data rather than executable instructions at model-context boundaries.

## Best Possible Solution

Implement or explicitly supersede this with a core-owned prompt-injection boundary design: cache-stable guidance, tested non-owner current-message marking, shared classification/wrapping for open-world tool results, delimiter escaping, and provider-shape regression coverage across OpenAI, Anthropic, and Google while preserving the existing external-content marker sanitizer.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: a non-owner prompt is assembled as queuedBody, senderIsOwner is passed to policy, transformTransportMessages returns user messages unchanged, and provider transports emit that text as ordinary user content; no live model behavior is needed to verify the missing boundary.

## Solution Assessment

Unclear that the exact proposed XML tags are the best final form. The maintainable fix should either reuse the existing randomized external-content marker/sanitizer pattern or introduce a rigorously escaped shared boundary, and it should be coordinated with the open CaMeL work rather than added as an ad hoc transport patch.
