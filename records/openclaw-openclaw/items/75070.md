## Summary

Keep open. Current main still sends a bare `/new` or `/reset` startup prompt that tells the model to run Session Startup and greet, and may prepend runtime-loaded daily memory, but it does not yet include the requested anti-fabrication/verification guardrail. The issue is also explicitly paired with open PR #75127 using closing syntax, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Harden bare session reset behavior so the model does not fabricate calendar, email, JSON, or other user-provided-looking data after `/new` or `/reset`.

## Best Possible Solution

Review and land PR #75127, or an equivalent narrow fix, so the reset prompt and reset-prompt tests explicitly forbid presenting invented data as real and require tool verification before acting on calendar, email, task, or structured-data facts.

## Reproduction Assessment

Unclear. There is a high-confidence static reproduction path showing current main sends the reset startup prompt and optional daily memory on bare `/new` or `/reset`, but the actual fabricated-data response depends on the reported model/provider and lacks transcript evidence.

## Solution Assessment

Yes. A narrow prompt-contract guardrail with direct prompt regression tests is the best repo-level fix; broader runtime, routing, permissions, or provider changes are not supported by the evidence here.
