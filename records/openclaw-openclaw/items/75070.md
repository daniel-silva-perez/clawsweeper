## Summary

Keep open. Current main still has reset/startup paths that can send a model-run bootstrap prompt and optional startup memory after a bare reset, and the prompt does not explicitly forbid inventing calendar/email/JSON facts or acting on inferred data. Chat slash resets are already acknowledged without invoking the model, so the remaining concern is the model-run reset/bootstrap surface rather than a fully implemented fix.

## What This Changes

Prevent session-reset/bootstrap turns from hallucinating fabricated user data, calendar events, or JSON and acting on it when no real user-provided payload exists.

## Best Possible Solution

Keep the issue open and harden the reset/startup model-run contract so reset-generated prompts treat workspace/bootstrap/memory as context only, never as user-provided data, verify real-world facts with tools before acting, and produce only a short greeting or no-op unless the user supplied an explicit reset-tail task.

## Reproduction Assessment

Unclear. There is a high-confidence static path to exercise the reset startup prompt and startup-memory prelude on current main, but the fabricated JSON/calendar output is model/provider-specific and the report did not include a transcript or live credentials.

## Solution Assessment

Unclear. The existing no-model acknowledgement is already the best behavior for bare chat reset commands, but the model-run reset prompt still needs a narrow guardrail; changing broader reset semantics should remain a maintainer product decision.
