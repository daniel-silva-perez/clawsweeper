## Summary

Keep open. Current main still has a model-run reset path that can send a fresh-session startup prompt, optionally with runtime-loaded daily memory, but the reset prompt and prelude do not explicitly say that the reset turn contains no user-provided calendar/email/JSON data or require verification before acting on real-world facts. The remaining work is narrow and testable in the reset prompt/prelude and gateway reset coverage.

## What This Changes

The issue requests a reset-session guardrail so empty or bootstrap reset turns produce a brief greeting or no-op instead of fabricated calendar/email/JSON data presented as user input.

## Best Possible Solution

Harden the reset/startup model-run contract so bootstrap files, daily memory, and workspace context are clearly context only, not user-provided facts; the model should greet briefly or stay quiet unless reset-tail or pending BOOTSTRAP work requires action, and it should verify calendar/email/task facts with tools before acting.

## Reproduction Assessment

Unclear. There is a high-confidence static path on current main showing bare model-run resets can send the startup prompt plus optional daily memory, but the fabricated JSON/calendar output itself is provider-specific and the issue does not include a transcript or live-test evidence.

## Solution Assessment

Yes. A narrow guardrail in the reset prompt and startup prelude with regression tests is the best repo-level fix; the reporter's AGENTS.md workaround is useful evidence but should not be required in every workspace.
