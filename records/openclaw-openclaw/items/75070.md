## Summary

Keep open. Current main still routes bare `/new` and `/reset` runs through a reset startup prompt that lacks the requested anti-fabrication and verification wording, and #75127 is open with closing syntax for this issue, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Prevent session reset/bootstrap turns from fabricating calendar, email, JSON, or other structured data as if it were user-provided context and acting on it without verification.

## Best Possible Solution

Land a narrow reset prompt guardrail shared by normal, bootstrap-pending, and limited-bootstrap reset modes, with focused prompt assertions and any needed gateway/auto-reply coverage, then close this as implemented after merge.

Unclear. There is a high-confidence static path showing bare `/new` and `/reset` can send the current startup prompt plus runtime-loaded memory, but the fabricated calendar/JSON output requires provider-specific live behavior and the report includes no transcript or logs.

## Solution Assessment

Yes, the linked prompt-level guardrail is the narrowest maintainable repo-level fix for this report. Broader runtime or tool-permission changes would be over-scoped unless the model still acts on invented data after the reset prompt contract is hardened.
