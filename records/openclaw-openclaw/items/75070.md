## Summary

Keep open. Current main still routes bare session resets through a startup/greeting prompt that can include runtime startup memory, but that prompt does not explicitly forbid invented user data or require verification before acting on calendar/email/JSON-like facts. A linked open PR already targets the narrow guardrail, so this issue should remain open until that PR lands or is replaced.

## What This Changes

The issue requests a reset-startup guardrail so bare `/new` or `/reset` turns do not let the model present invented calendar, email, or JSON data as user-provided facts.

## Best Possible Solution

Review and land PR #75127, or replace it with an equivalent narrow patch that adds a shared reset-startup anti-fabrication and verification guardrail plus prompt-construction regression tests. The issue should close only after that fix is merged.
