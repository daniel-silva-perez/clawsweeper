## Summary

Keep this issue open. Current main still has no file-based `memory_search` fallback when `node:sqlite` cannot load: the builtin manager creation fails through the SQLite opener, the tool converts that into an unavailable payload, and the existing LIKE fallback only runs after a SQLite/FTS table already exists. The related open #69199 tracks only the narrower error-message UX fix, not the capability-preserving file scan requested here.

## What This Changes

The issue requests a non-SQLite keyword fallback for `memory_search` over `MEMORY.md` and `memory/*.md` when `node:sqlite` cannot initialize.

## Best Possible Solution

Add a memory-core owned fallback that activates only when the builtin SQLite manager cannot initialize because `node:sqlite` is unavailable, scans the same safe memory-file surface, returns bounded keyword-ranked snippets, and marks the response/debug metadata as degraded file keyword search.

## Reproduction Assessment

Yes. The code path is high-confidence from static inspection: a missing `node:sqlite` throw in `openMemoryDatabaseAtPath` is caught as manager creation failure and becomes an unavailable `memory_search` response, with no file scan in between.

## Solution Assessment

Yes, the requested file-based keyword fallback is the narrowest maintainable direction if it stays inside the memory-core plugin and reuses existing memory file discovery/chunking helpers. Replacing the SQLite backend or adding startup migrations would be broader than needed for this issue.
