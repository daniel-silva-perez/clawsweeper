## Summary

Keep open. Current main still requires `node:sqlite` before the builtin memory manager can search files, and the `memory_search` tool returns an unavailable empty result when manager creation fails. The existing fallback code only handles SQLite/FTS query failures after SQLite is already available, while #69199 is scoped to the narrower error-message fix for #59457.

## What This Changes

Add a fallback path so `memory_search` scans `MEMORY.md` and `memory/*.md` with keyword matching when `node:sqlite` cannot initialize.

## Best Possible Solution

Implement a bounded memory-core fallback that activates only when the memory manager is unavailable because SQLite cannot load, scans allowed markdown memory files, returns clearly labeled fallback keyword results, and preserves existing indexed, QMD, wiki, and session-search semantics.

## Reproduction Assessment

Yes. A high-confidence reproduction is to run `memory_search` in a Node runtime where `require("node:sqlite")` fails, or mock `requireNodeSqlite()` to throw: current main propagates that into a manager error and returns `results: []` instead of reading memory files.

## Solution Assessment

Yes. The requested file-scan fallback is the narrowest maintainable solution for preserving basic recall when SQLite itself is unavailable; the separate error-message PR is useful but does not restore search capability.
