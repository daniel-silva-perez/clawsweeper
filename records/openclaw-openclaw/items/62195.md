## Summary

This PR is still relevant because current main does not implement cron or heartbeat preHook support, but the proposed patch should not be merged as-is. The diff adds a new gateway-host process execution path and still accepts non-absolute executable names, and the heartbeat integration appears to miss configured heartbeat preHooks for cron-triggered immediate heartbeat runs.

## What This Changes

The PR adds optional cron-job and heartbeat `preHook` configuration that runs a configured executable before agent execution, with protocol/config/tool/schema/docs/tests updates and skip/error exit-code handling.

## Best Possible Solution

Keep the PR open and require changes before merge: enforce the documented absolute executable contract at schema/normalization time, consider routing preHook execution through existing exec approval policy or documenting an explicit admin-only bypass, and add coverage for configured heartbeat preHooks during cron-triggered immediate heartbeat runs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Require absolute preHook executables:** `src/cron/pre-hook.ts:66-70`
  - body: The docs and tool schema describe `preHook.file` as an absolute path, but the normalizer accepts any non-empty string and `runPreHook` passes it straight to `execFile`. With `shell: false`, bare names such as `node` are still resolved from PATH, so a relative/PATH-controlled executable can run under the gateway process instead of the intended fixed script. Please reject non-absolute paths before persisting or spawning the hook.
  - confidence: 0.91
- **[P2] Preserve configured heartbeat preHooks for cron wakes:** `src/infra/heartbeat-runner.ts:870-881`
  - body: This gate reads only `heartbeat?.preHook`, but the cron `wakeMode: "now"` path calls `runHeartbeatOnce` with an override object containing only `{ target: "last" }`. Because that override replaces the resolved heartbeat config, an operator-configured `agents.defaults.heartbeat.preHook` is skipped for immediate cron-triggered heartbeat turns, even though the PR adds cleanup for preHook-blocked cron wake results. Merge overrides with the resolved heartbeat config or pass through the effective preHook.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The patch introduces gateway-host executable execution and still needs path and policy hardening before merge.

Concerns:

- **[high] Relative/PATH preHook execution:** `src/cron/pre-hook.ts:66`
  - body: `preHook.file` is accepted as any non-empty string and then executed with `execFile`; bare executable names are resolved through PATH even with `shell: false`. This contradicts the documented absolute-path contract and can execute an unintended binary when PATH or the working directory is influenced.
  - confidence: 0.9
- **[medium] Exec approval policy bypass risk:** `src/cron/pre-hook.ts:66`
  - body: The new runner calls `child_process.execFile` directly rather than the existing exec/system.run approval path. Since cron jobs can be persisted through the admin cron API/tool surface, deployments that deny or gate `exec` could still gain scheduled gateway-host command execution via preHook unless this is explicitly scoped, documented, and tested as an admin-only bypass.
  - confidence: 0.74
