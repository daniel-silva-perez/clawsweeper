## Summary

Keep open: current main still lacks the requested cron/heartbeat preHook feature, and this PR is the active implementation path, but the proposed patch still has blocking execution-policy issues before it can merge.

## What This Changes

The PR adds optional cron-job and heartbeat `preHook` configuration with protocol/config/tool/schema/docs/test updates, a direct executable runner, and skip/error exit-code handling.

## Best Possible Solution

Land a hardened version of this PR that rejects non-absolute or non-canonical executable paths before persistence, preserves the existing exec approval/tool-policy guarantees for scheduled host execution, and keeps the cron plus heartbeat behavior covered by focused tests and user-facing docs.

## Reproduction Assessment

Yes. The PATH-resolution issue has a focused proof: the PR accepts a non-empty `file`, and Node `execFile` with `shell:false` still resolved a bare `node` executable from PATH in local inspection.

## Solution Assessment

No, not as-is. The feature direction is useful, but direct `execFile` with non-absolute paths and no exec-policy integration is not the narrowest maintainable security boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Reject non-absolute preHook files:** `src/cron/normalize.ts:506-509`
  - body: The docs/tool schema describe `preHook.file` as an absolute executable path, but the normalizer stores any non-empty string and `runPreHook` executes it. With `shell: false`, bare names still resolve through PATH, so values like `node` or a relative path can run an unintended binary. Reject non-absolute paths before persisting or spawning the hook.
  - confidence: 0.9
- **[P1] Route preHook through exec policy:** `src/cron/pre-hook.ts:66-70`
  - body: This adds a scheduled gateway-host execution path that calls `child_process.execFile` directly. Operators can deny or gate host exec through the existing exec approval/tool-policy boundary, but a cron preHook created through the admin cron API/tool would execute later without that approval path. Route the hook through the existing exec/system.run policy, or make the bypass an explicit maintainer-approved admin-only contract with docs and tests.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The patch introduces gateway-host executable execution and needs path validation plus an explicit exec-approval policy boundary before merge.

Concerns:

- **[high] Relative or PATH-resolved preHook execution:** `src/cron/pre-hook.ts:66`
  - body: `preHook.file` is accepted as any non-empty string and then executed with `execFile`; bare executable names are resolved through PATH even with `shell: false`, which contradicts the documented absolute-path contract and can run an unintended binary.
  - confidence: 0.9
- **[medium] Exec approval policy bypass:** `src/cron/pre-hook.ts:66`
  - body: The new runner executes configured files directly instead of using the existing exec approval/system.run boundary, so deployments that deny or gate host exec could still gain scheduled gateway-host command execution through cron preHook unless this is explicitly scoped and reviewed.
  - confidence: 0.78
