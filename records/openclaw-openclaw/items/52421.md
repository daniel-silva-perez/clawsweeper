## Summary

Current main has substantially improved tool_use/tool_result pairing repair, so the compaction root described in the discussion is partly addressed. This issue should stay open because the reported user-visible failure is still actionable: JSON-wrapped Anthropic invalid_request_error payloads can still be formatted as `LLM request rejected: <raw provider message>`, and there is no specific handling or regression test for the `unexpected tool_use_id` case.

## Best Possible Solution

Keep this issue open. Preserve the existing transcript repair work, then add explicit handling for Anthropic `unexpected tool_use_id` / orphaned `tool_result` validation failures in the provider error path: re-sanitize replay history and retry once when feasible, otherwise surface a friendly session-recovery message. Add focused tests in the error formatter/failover path and, if retry is implemented, in the embedded runner path.
