## Summary

Current main has substantially improved tool_use/tool_result pairing repair, including replay and compaction paths, but the reported user-visible failure is still actionable: JSON-wrapped Anthropic invalid_request_error payloads can still be formatted as `LLM request rejected: <raw provider message>`, and there is no specific current-main handler or regression coverage for `unexpected tool_use_id` recovery.

## Best Possible Solution

Keep this issue open. Preserve the existing transcript repair work, then add a narrow classifier for Anthropic `unexpected tool_use_id` / orphaned `tool_result` validation failures in the provider error path. When safe, re-sanitize replay history and retry once; otherwise surface a friendly session-recovery message. Add focused tests for `formatAssistantErrorText` and, if retry is implemented, the embedded runner/failover path.
