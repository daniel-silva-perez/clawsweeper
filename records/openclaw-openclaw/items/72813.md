## Summary

Keep open. Current main still lacks stale-runtime recovery on the five lazy-loader sites this PR targets, and the general direction matches the shared reply-pipeline failure from #68466. The patch is not merge-ready because its staged-media block is based on older code and drops the current `ctx.MediaStaged` guard added for `chat.send` prestaging.

## What This Changes

The PR wraps five remaining `get-reply` lazy-loader call sites with stale-runtime recovery, resets cached import promises on failure, pins those modules as stable `tsdown` entries, and adds regression coverage for the recovery paths.

## Best Possible Solution

Update the PR against current main, keep the stale-runtime recovery and stable chunk entries, but wrap the existing staged-media block without removing `!ctx.MediaStaged`; add a focused regression proving pre-staged media does not call `stageSandboxMedia`.

## Reproduction Assessment

Yes. The stale-runtime user problem is reproducible by the #68466 path for dynamic-import `ERR_MODULE_NOT_FOUND` after an in-place package upgrade, and the PR regression is reproducible statically by comparing current main's `!ctx.MediaStaged` guard with the PR branch's line 638 condition.

## Solution Assessment

No, not as submitted. Extending stale-runtime recovery to the remaining lazy loaders is the right narrow direction, but the staged-media catch must preserve the current-main prestaging guard before the patch is safe to merge.
