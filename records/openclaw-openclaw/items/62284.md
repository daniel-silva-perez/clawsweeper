## Summary

Keep open. Current main still has no timed exec approval decision, no `approvedUntil` or trust-window contract in the approvals schema, and no `/approve 30m` parsing; the exact same-author PR was closed unmerged, while the newer trust-window PRs are related but open and security-sensitive alternatives that need maintainer design choice.

## What This Changes

The issue requests time-bounded exec approvals so users can approve a command pattern for a finite duration such as 30 minutes or 1 hour, with chat, CLI, storage, and expiry handling.

## Best Possible Solution

Define one canonical timed exec approval model before merging code: either a per-command-pattern timed approval integrated into the existing approval resolver, or an explicit trust-window model with clear scope, authorization, audit, expiry, and restart semantics. The chosen path should update the protocol/schema, `/approve` handling, Control UI, approvals CLI, docs, and focused regression tests, and should supersede the competing open PRs once maintainers pick the design.

## Reproduction Assessment

Not applicable as a runtime bug, but yes for source-level verification. Current main can be checked by reading the approval decision union, `/approve` parser, protocol schema, CLI renderer, and UI actions, all of which lack any duration/timed approval path.

## Solution Assessment

Unclear. The requested timed approval direction addresses the approval-fatigue gap, but storing `approvedUntil` in the approvals file may not be the safest authority for a non-persistent command-execution privilege; maintainers should choose between a command-pattern timed entry and the broader in-memory trust-window proposals before implementation.

## Security Review

Status: needs_attention

Summary: This issue is security-sensitive because it would allow commands to bypass repeated approval prompts for a time window.
