## Summary

Keep open. Current main has an opt-in tool-loop detector, but it is disabled by default, keys off same tool plus same params, and only blocks critical no-progress cases; it does not provide the requested content-aware write/edit dedupe or a general per-turn tool-call cap for standard agent runs.

## What This Changes

Request for agent safeguards that detect duplicate file writes by content hash and enforce run-level limits on total or repeated tool executions.

## Best Possible Solution

Keep the issue open and design this as an extension of the existing `tools.loopDetection` and file-tool wrappers: track recent write/edit content fingerprints per run, return a clear unchanged/duplicate result to the model, and add a bounded run-level tool-call limit with documented defaults and tests for versioned duplicate filenames.

## Reproduction Assessment

Yes for the missing safeguards: current code shows write/edit operations do not hash or skip duplicate content, and existing loop detection is disabled by default and same-params based. The exact historical 382-file run was not replayed because the report only includes a file listing, not a runnable session trace.

## Solution Assessment

Unclear. The requested direction is sound, but the proposed same-target content hash check alone would not stop new versioned filenames with identical content; a better solution should detect repeated write content across recent tool calls and add a general run cap through the existing tool-loop configuration surface.
