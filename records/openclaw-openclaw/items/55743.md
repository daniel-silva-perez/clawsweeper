## Summary

Keep this issue open. Current main still renders assistant shell snippets as generic sanitized markdown code blocks with Copy-only behavior; the existing approval UI only resolves already-pending exec/plugin approval events and does not create a command card or launch an assistant-suggested command with cwd, host, session, and risk metadata.

## What This Changes

Request Control UI command cards for assistant-suggested shell snippets with Copy, Approve and Run, target metadata, risk labels, and streamed output back into chat.

## Best Possible Solution

Implement a narrow structured command-card path in Control UI only when the assistant output includes an exact command plus cwd, host/node, session, and execution target metadata; route execution through the existing exec approval and node `systemRunPlan` mechanisms, preserve Copy, stream results into chat, and document the security model.

## Reproduction Assessment

Yes. A source-level reproduction is clear: an assistant message containing a shell fenced code block flows through sanitized markdown rendering, produces only `.code-block-copy`, and the click handler only copies to the clipboard.

## Solution Assessment

Unclear until maintainers settle the security/product contract. The requested UX belongs in the Control UI surface, but the maintainable solution should reuse existing exec approval machinery rather than adding a direct chat-to-shell shortcut.

## Security Review

Status: needs_attention

Summary: The requested feature is security-sensitive because it would let the browser Control UI initiate local or node-host shell execution from assistant text.

Concerns:

- **[high] Bind approved runs to structured execution metadata:**
  - body: An implementation should not execute arbitrary markdown code blocks directly. It needs an exact command, cwd, host/node, session, and approval record binding so the executed payload cannot drift from what the user approved.
  - confidence: 0.86
- **[medium] Avoid optimistic risk labels from plain text parsing:**
  - body: Read-only/destructive/network/config labels could mislead users if inferred only from shell text. The initial design should be conservative and route uncertain commands through the strongest approval path.
  - confidence: 0.78
