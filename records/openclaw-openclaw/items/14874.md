## Summary

Keep open. Current main does not implement optional SIE verification in the skill loader: skill loading reads and parses `SKILL.md`, applies path/size/config eligibility checks, and has no signed-envelope, trusted-issuer, warn/strict, or invalid-signature decision path. The related implementation PR #14893 was closed unmerged, and the request is security-sensitive enough to need maintainer design before an automated fix PR.

## Best Possible Solution

Keep the issue open for maintainer security design. A good implementation would add a default-off, backwards-compatible TS skill-loader/config contract for SIE verification, deterministic reason codes, warn/strict behavior, tests around unsigned/valid/invalid cases, and docs explaining how it relates to ClawHub scans, skill install/update flows, and any lockfile/hash integrity work.

## Risks / Open Questions

- This is security-sensitive loader trust policy: strict mode could reject existing third-party or workspace skills, so defaults, migration behavior, and operator UX need maintainer design.
- A plugin-only implementation is likely insufficient without a core load-time hook or loader change; the current `before_install` hook covers install scans, not every skill load/prompt inclusion path.
- The closed related PR #14893 proposed a standalone Python integration surface rather than integration with the current TypeScript loader/config contract, so reusing it directly may not fit current architecture.
- Related integrity work such as #56256 may overlap conceptually; maintainers should align signing, hashes, ClawHub scan metadata, and update/install behavior before choosing a final design.
