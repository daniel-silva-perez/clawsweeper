## Summary

Keep this open. Current main still routes Telegram outbound presentation and action sends through inline keyboard-only helpers, and neither the message tool schema nor the Telegram docs expose native reply keyboards, keyboard removal, resize_keyboard, or one_time_keyboard. The related issues cover inline buttons and callback behavior, so they do not supersede this separate capability gap.

## What This Changes

Request first-class Telegram native reply keyboard support through the agent/message send surface, including keyboard rows/buttons, keyboard removal, resize_keyboard, and one_time_keyboard.

## Best Possible Solution

Add an additive Telegram plugin-owned reply_markup path that supports typed reply keyboard rows, keyboard removal, resize_keyboard, and one_time_keyboard through the message tool/action runtime, with docs and tests proving existing inline keyboard behavior remains unchanged.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: search for ReplyKeyboardMarkup/ReplyKeyboardRemove returns no Telegram runtime/schema/docs support, while the send and action paths visibly normalize only inline buttons into reply_markup.

## Solution Assessment

Yes for the narrow direction. The best fix is Telegram-owned, additive schema/runtime/docs support; it should avoid changing the generic presentation contract unless maintainers deliberately want portable reply-keyboard semantics.
