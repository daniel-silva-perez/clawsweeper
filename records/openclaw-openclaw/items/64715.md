## Summary

Keep this issue open. Current main still routes Telegram outbound presentation and action sends through inline-keyboard-only code, while Telegram and grammY expose reply keyboard and keyboard-removal reply_markup types as separate native surfaces.

## What This Changes

Add Telegram-native reply keyboard rows, keyboard removal, resize_keyboard, and one_time_keyboard support to the agent/message outbound send surface.

## Best Possible Solution

Implement an additive Telegram-owned reply_markup path that supports reply keyboard rows and keyboard removal through the message tool/action runtime, preserves existing inline keyboard behavior, and documents when to use inline buttons versus reply keyboards.
