## Summary

Keep open. Current main still lacks the reported replay-loop fix, the default tool-loop detector remains opt-in for normal repeated calls, and two open closing PRs are active implementation candidates, so closing this issue before one canonical fix lands would be premature.

## What This Changes

Prevent the embedded runtime from automatically replaying a prior failed tool call in Qwen/OpenAI-completions agent runs, especially after failed `exec` calls in Feishu sessions.

## Best Possible Solution

Review and land one canonical fix that disables the inner pi-coding-agent retry path for OpenClaw embedded runs without persisting user settings, preserves failed tool-result/error context across the next model turn, and adds focused regression coverage for failed `exec` followed by a changed or no-op assistant action; keep any broader reasoning-loop guard as an explicit maintainer decision.

## Reproduction Assessment

Unclear. The issue provides a concrete Qwen/Feishu/failed-`exec` loop pattern and dependency inspection confirms a plausible retry path, but this read-only review did not run the live provider/channel path and current main has no regression test for that exact sequence.

## Solution Assessment

Unclear. The narrow in-memory retry disable proposed by the linked PRs appears to be the smallest maintainable fix boundary, while the broader thinking-repetition hint needs maintainer review to avoid changing normal replay semantics too widely.
