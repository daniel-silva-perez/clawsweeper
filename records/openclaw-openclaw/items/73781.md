## Summary

Keep open. Current main still leaves the embedded Pi settings manager without any retry disable, the pinned pi-coding-agent package defaults auto-retry on and removes the failed assistant turn before calling `continue()`, and the default OpenClaw tool-loop detector remains opt-in. The two issue-closing PRs are still open and unmerged, so this issue should remain open until maintainers land or reject one canonical fix.

## What This Changes

Prevent the runtime from replaying a previous failed tool call in embedded agent runs, especially the reported Qwen/Feishu failed-`exec` loop after upgrading to OpenClaw 2026.4.25.

## Best Possible Solution

Land one canonical fix that disables the inner pi-coding-agent auto-retry for embedded OpenClaw runs while preserving OpenClaw’s own retry/failover layer, with regression coverage for a failed tool result followed by a changed or final next assistant action. Treat thinking-repetition hints as a separate follow-up only if the replay still reproduces after the nested retry is removed.

## Reproduction Assessment

Unclear. The issue gives a concrete Qwen/Feishu failed-`exec` replay pattern, and dependency source confirms a plausible replay mechanism, but this review did not run the live provider/channel path or a targeted embedded-run reproduction.

## Solution Assessment

Unclear. The narrow retry-disable proposal in #74434 appears to match the confirmed dependency behavior most directly, while #74503’s broader thinking-repetition guard may need separate maintainer judgment after the nested retry fix is evaluated.
