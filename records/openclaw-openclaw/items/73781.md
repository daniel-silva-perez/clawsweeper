## Summary

Keep open. Current main still does not contain the focused auto-retry/replay fix described by the linked implementation candidates, and the issue is explicitly paired with open PRs that use closing syntax, so it should remain open until a canonical fix lands or maintainers close the candidate PRs.

## What This Changes

The issue reports a 2026.4.25 regression where Qwen/Feishu agent runs replay identical failed `exec` tool calls at runtime instead of letting the model choose a new action.

## Best Possible Solution

Review the linked fix PRs and land one canonical change that prevents embedded Pi retry/replay from hiding the failed tool result from the next model turn, with a regression test that simulates failed `exec` output followed by a changed or no-op next assistant action. Keep default loop-detection policy decisions separate unless maintainers explicitly want a broader guardrail change.

## Reproduction Assessment

Unclear. The issue gives a concrete Qwen/Feishu/failed-`exec` replay pattern, but this read-only pass did not run a live provider/channel reproduction and current tests do not cover that exact failed-tool replay sequence.

## Solution Assessment

Unclear. Disabling the inner SDK auto-retry looks like the narrowest proposed root-cause fix, while the broader thinking-repetition guard may be useful but should not substitute for preserving failure context; current main is not sufficient because its generic loop guard is default-off.
