## Summary

Keep open. Current main still prepares embedded Pi settings without disabling the inner Pi retry path, default tool-loop detection remains opt-in, and open closing PRs #74434 and #74503 are active implementation candidates for this reported replay loop.

## What This Changes

Prevent the runtime from replaying a previous failed tool call after a tool failure, especially in Qwen/Feishu workflows upgraded around 2026.4.25.

## Best Possible Solution

Land one canonical fix that preserves failed tool-result context across the next model turn, likely by disabling the inner Pi retry layer in the prepared settings manager and adding a regression test for failed `exec` followed by a changed or no-op next assistant action; treat broader thinking-repetition guards as a separate maintainer choice.

## Reproduction Assessment

Unclear. The issue gives a concrete Qwen/Feishu/failed-`exec` replay pattern and linked PRs describe a plausible root cause, but this read-only review did not run the live provider/channel path and current tests do not cover that exact sequence.

## Solution Assessment

Unclear. The focused auto-retry disable in #74434 looks like the narrowest current proposal, while #74503 adds a broader guard; current main's default-off loop detection is not enough by itself.
