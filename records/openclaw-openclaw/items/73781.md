## Summary

Keep open. This is a fresh regression report against 2026.4.25 with a concrete Qwen/Feishu/exec loop pattern, and current main does not provide enough proof that the reported runtime replay loop is fixed. Main has opt-in loop detection and some replay-safety guards, but no focused regression coverage for a failed `exec` result followed by the runtime re-emitting the same prior tool call on Qwen/Feishu. Related ClawSweeper context also points to unresolved overlapping loop work rather than a canonical merged fix.

## Best Possible Solution

Keep the issue open and reproduce against current main with a minimal Qwen/Feishu or OpenAI-compatible mocked transcript: failed `exec` result, assistant reasoning, then verify the runtime does not inject the previous tool call unless the model actually emits it. If reproduced, fix the embedded runner/provider replay path or default guard behavior in the owning module, and add a focused regression test plus docs/config alignment if loop detection defaults change.
