## Summary

Keep open. Current main still lacks the focused embedded Pi auto-retry/replay fix, and this issue is paired with open, unmerged implementation PRs that use closing syntax, so it should remain open until a canonical fix lands or maintainers close those candidates.

## What This Changes

The issue reports that after a failed tool call, the agent runtime can replay the prior tool call instead of letting the model choose a new action, causing repeated loops on Qwen/Feishu workflows.

## Best Possible Solution

Land one canonical fix that disables embedded Pi SDK auto-retry through a session-scoped in-memory settings override, not a persistent user settings write, and add regression coverage for a failed `exec`/tool result followed by a changed or no-op next assistant action. Treat the broader repeated-thinking hint as a separate maintainer choice unless it is needed to close the same root cause.

## Reproduction Assessment

Unclear. The report gives a concrete Qwen/Feishu failed-`exec` replay pattern and source/dependency inspection shows a plausible default inner retry path, but this pass did not run a live provider/channel reproduction and current tests do not cover that exact sequence.

## Solution Assessment

Unclear for the open candidates, and no for current main. The narrow session-scoped Pi retry-disable approach appears to be the most maintainable first fix, while current main still leaves SDK auto-retry enabled and default loop detection off.
