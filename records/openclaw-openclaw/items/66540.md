## Summary

Keep open: current main still treats `replyToCurrent: false` as an implicit-threading veto, while ordinary directive parsing and streaming parsing can still produce that false value for non-tagged text. The related fix PRs cited in the discussion are closed unmerged, and the latest release tag still contains the same behavior, so this remains a narrow actionable bug rather than cleanup that can close.

## What This Changes

The issue requests that default `replyToCurrent: false` values from directive parsing stop blocking implicit `replyToId` threading for queued or follow-up replies.

## Best Possible Solution

Represent explicit no-threading separately from parser-default `replyToCurrent: false`, let ordinary parsed or streamed payloads receive implicit current-message threading, carry the correct origin/current message id through queued follow-up delivery, and lock the behavior with focused regression tests for default-false payloads plus command replies.
