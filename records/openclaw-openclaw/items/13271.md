## Summary

Keep open. Current main still has adjacent loop detection, channel/group allow-deny tool policy, plugin before-tool hooks, and explicit `sessions_spawn`, but it does not implement a channel-scoped max-per-turn tool-call budget or the requested `warn`, `block`, and `auto-delegate` overflow behavior. The discussion already treats this as unresolved maintainer design work, and automatic delegation is security-sensitive enough to need explicit product and authority-boundary decisions.

## What This Changes

Add per-channel tool-call budgets with configurable `warn`, `block`, and `auto-delegate` overflow handling so constrained channels can cap inline tool use.

## Best Possible Solution

Define a generic scoped tool-call budget model that resolves alongside effective tool policy. Ship deterministic run-scoped `warn` and `block` enforcement in the trusted before-tool path first, then design `auto-delegate` separately around `sessions_spawn` authority inheritance, child-run tracking, and completion handoff.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The high-confidence current-main check is source, schema, docs, and test inspection showing that the requested config keys and enforcement paths are absent.

## Solution Assessment

Unclear. A per-channel budget is a plausible direction, but combining warning, blocking, and policy-triggered subagent creation into one first change is not the safest maintainable path; deterministic budget enforcement should precede automatic delegation.
