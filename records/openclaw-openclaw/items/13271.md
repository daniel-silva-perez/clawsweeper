## Summary

Keep open. Current main has explicit sub-agent spawning and opt-in loop detection, but it does not provide a per-channel max-per-turn tool budget, `onExceed` policy, or gateway-level auto-delegation when a channel-specific budget is exceeded. Related #9912 still tracks global maxToolCalls, while this issue remains distinct because it asks for channel-scoped enforcement plus overflow behavior.

## Best Possible Solution

Keep this open and design a generic core runtime policy for tool budgets, probably sharing the existing tool-call interception path but adding explicit per-run counters, channel/session scoping, and config precedence. Align it with #9912 first; implement `block`/`warn` before treating `auto-delegate` as safe, because automatic handoff needs a well-defined child-task prompt and result path.

## Risks / Open Questions

- Auto-delegation after a denied tool call needs a reliable way to describe the remaining work; the current `before_tool_call` surface sees the next tool name/params plus run/session context, not the model's full intended task plan.
- Per-channel budgets overlap with #9912, existing `tools.loopDetection`, per-agent tool policy, and channel/group tool policy; the config precedence needs maintainer design before implementation.
- Hard-blocking tool calls can affect provider transcript validity and tool-result replay, so block/warn behavior needs tests across the embedded runner and channel-routed sessions.
- Automatic sub-agent spawning could increase gateway resource pressure unless concurrency, child limits, timeouts, and delivery failure behavior are part of the design.
