## Summary

Keep open. Current main has a code-fence-aware sanitizer for assistant-visible `<relevant-memories>` blocks, but the streaming outbound block-reply path still does not consistently apply that sanitizer before channel delivery, and the linked closing PR is still open and unmerged in the provided GitHub context.

## What This Changes

The issue asks OpenClaw to strip assistant-reflected `<relevant-memories>` blocks before sending outbound channel messages such as Telegram, Discord, Slack, or iMessage replies.

## Best Possible Solution

Review and adjust the linked PR so the existing code-fence-aware assistant-visible sanitizer is applied at every assistant-origin outbound delivery boundary, including user-facing sanitizer output, block replies, final block replies, and partial/preview updates.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to feed assistant text containing `<relevant-memories>...</relevant-memories>` through the non-phase `text_end` block-reply path and observe that `emitBlockChunk` only strips think/final/tool scaffolding before `emitBlockReply`.

## Solution Assessment

Yes for the requested direction: reusing the shared code-fence-aware sanitizer at outbound delivery boundaries is the narrow maintainable fix. No for closing this issue today, because the linked closing PR is open and the inspected current-main streaming paths still need coverage.
