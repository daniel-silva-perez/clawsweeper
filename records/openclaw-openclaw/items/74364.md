## Summary

Keep open. Current main has the shared sanitizer needed to remove `<relevant-memories>` blocks, but several outbound assistant delivery paths still bypass it, and an open linked PR is already the implementation candidate rather than a reason to close the issue before merge.

## What This Changes

The issue requests stripping reflected `<relevant-memories>` blocks from assistant responses before delivery through outbound messaging channels such as Telegram and Discord.

## Best Possible Solution

Keep this issue open until PR #74436 or a replacement lands. The end state should apply the existing code-fence-aware assistant-visible sanitizer across normal outbound text, partial preview edits, and text_end/block replies, with focused regressions for shared agent delivery and Telegram-visible streaming.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists by passing a reflected `<relevant-memories>` assistant chunk through the current partial-preview or text_end block delivery path, which uses `stripBlockTags`/reply-directive parsing rather than the shared memory-aware sanitizer.

## Solution Assessment

No current-main fix is complete. The narrow maintainable direction is to reuse the existing assistant-visible sanitizer at outbound assistant delivery boundaries instead of adding a separate ad hoc regex per channel.
