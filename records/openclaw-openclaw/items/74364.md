## Summary

Keep open. Current main still has the shared memory-scaffolding sanitizer, but outbound channel delivery paths can still bypass it, and the linked open PR should be reviewed or expanded before this issue closes.

## What This Changes

The issue requests stripping reflected `<relevant-memories>` blocks from assistant responses before delivery through messaging channels such as Telegram, Discord, Slack, and iMessage.

## Best Possible Solution

Apply the existing code-fence-aware assistant-visible sanitizer consistently at every assistant-origin outbound delivery boundary, including final replies, block replies, text_end streaming chunks, and preview partials, with focused regressions for shared delivery and Telegram preview behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to feed an assistant text_delta/text_end or outbound normalized reply containing a `<relevant-memories>` block through the current channel delivery paths and observe that the narrower strip chains do not remove that block.

## Solution Assessment

Unclear. Reusing the shared assistant-visible sanitizer is the right direction, but the open PR's stated scope omits the streaming preview/text_end paths that current main still exposes, so the safest fix likely needs broader delivery-boundary coverage.
