## Summary

Keep open. Current main has a shared, code-fence-aware sanitizer for `<relevant-memories>`, but the outbound channel streaming/block delivery paths still do not consistently apply it; #74436 is an open, unmerged closing PR and should be reviewed rather than closing this issue early.

## What This Changes

Request to strip memory plugin `<relevant-memories>` scaffolding from assistant text before it is delivered through outbound messaging channels.

## Best Possible Solution

Apply the existing assistant-visible sanitizer at every assistant-origin outbound delivery boundary, including `sanitizeUserFacingText`, block chunk replies, final block replies, and preview/partial reply updates, with regressions proving code-fence preservation and Telegram/default streaming behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to feed assistant text containing `<relevant-memories>...</relevant-memories>` through `sanitizeUserFacingText`, `emitBlockChunk`, or the `onPartialReply` text update path and observe that current main does not apply the relevant-memory sanitizer there.

## Solution Assessment

Yes for the issue direction, but unclear for the linked PR as currently scoped. The maintainable fix is to reuse `sanitizeAssistantVisibleText` at outbound assistant delivery boundaries; a one-boundary fix that excludes preview/text_end streaming may leave visible leaks.
