## Summary

Keep open. Current main has a code-fence-aware sanitizer for assistant-visible `<relevant-memories>` blocks, but the relevant outbound channel boundaries still do not consistently apply it, and the open unmerged closing PR should be reviewed rather than closing this issue early.

## What This Changes

Strip `<relevant-memories>` blocks from assistant responses before delivery to messaging channels such as Telegram, Discord, Slack, and iMessage.

## Best Possible Solution

Apply the existing code-fence-aware assistant-visible sanitizer at every assistant-origin outbound delivery boundary, including user-facing sanitizer output, block replies, final block replies, and preview/partial reply updates, with focused regressions for code-fence preservation and Telegram/default streaming behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to feed assistant text containing `<relevant-memories>...</relevant-memories>` through `sanitizeUserFacingText`, `emitBlockChunk`, or the partial preview update path and observe that current main does not consistently apply the relevant-memory sanitizer there.

## Solution Assessment

Yes for the issue direction, but incomplete for the currently linked PR as described. The narrow maintainable fix is to reuse `sanitizeAssistantVisibleText` or its relevant-memory stage at outbound assistant delivery boundaries rather than adding a raw regex that can strip code examples.
