## Summary

Keep this PR open. Current main still has session-lock doctor coverage but no gateway-lock doctor inspection or repair path, and the linked product gap remains open. The PR is a plausible implementation candidate, but it is stale against the current doctor health contribution flow and has unresolved review findings that would need revision before merge.

## Best Possible Solution

Keep this PR open and revise it rather than closing it. The best path is to add a `doctor:gateway-locks` health contribution in `src/flows/doctor-health-contributions.ts`, expose or reuse the same owner-status semantics used by `acquireGatewayLock`, ensure repair never removes a confirmed-live lock, add stale-without-repair and repair-mode tests, and update the gateway doctor docs after the behavior is safe.

## Risks / Open Questions

- Merging as submitted could let `openclaw doctor --fix` delete an active gateway lock after the age threshold, allowing a second gateway instance to start while the first is still running.
- PID-only inspection would miss recycled-PID cases that current gateway startup already handles with Linux start-time and argv checks.
- The branch is stale against the current doctor health contribution architecture and needs to be rewired through `src/flows/doctor-health-contributions.ts`.
- The submitted tests do not cover the stale-without-repair user-facing warning/hint path.
