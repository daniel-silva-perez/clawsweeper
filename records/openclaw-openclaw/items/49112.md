## Summary

Keep this PR open. Current main still does not implement doctor coverage for gateway lock inspection or repair, so the PR is not obsolete. It remains a focused implementation candidate for open stale-gateway-lock doctor coverage, but the submitted diff needs revision before merge because its inspection/removal semantics diverge from the existing runtime gateway-lock owner checks.

## Best Possible Solution

Keep this PR open and revise it. The best path is to add a gateway-lock doctor contribution and focused tests, but base inspection and removal on the same owner-status logic used by `acquireGatewayLock`, so live owners are preserved, PID reuse is detected, stale-without-repair output is covered, and stale-with-repair removal is safe. After revision, run the targeted doctor/gateway-lock tests and the appropriate changed gate.
