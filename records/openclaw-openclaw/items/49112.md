## Summary

Keep this PR open. Current main still lacks `openclaw doctor` gateway-lock inspection/repair, and the linked product gap in #49036 remains. The PR is a plausible implementation candidate, but it needs revision before merge because the submitted inspection semantics are weaker than the gateway startup lock owner checks and can remove a live gateway lock in repair mode.

## Best Possible Solution

Keep this PR open and revise it. The best path is to add a gateway-lock doctor health contribution in `src/flows/doctor-health-contributions.ts`, reuse or expose the same owner-status semantics used by `acquireGatewayLock`, ensure repair never removes a confirmed-live lock, add stale-without-repair and stale-with-repair tests, and update the doctor/gateway docs once the behavior is safe.

## Risks / Open Questions

- Merging as submitted could let `openclaw doctor --fix` delete an active gateway lock purely because it is older than the stale threshold, allowing a second gateway instance to start.
- PID-only inspection would miss recycled-PID cases that current gateway startup already detects with Linux start-time and argv checks.
- The branch is stale against the current doctor health contribution flow; it wires directly into an older `src/commands/doctor.ts` shape.
