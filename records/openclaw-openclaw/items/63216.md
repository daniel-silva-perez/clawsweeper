## Summary

Keep open. Current main has partial mitigations for small context-window reserve floors and aggregate tool-result truncation, but the reported remaining behavior is still present: the hard-reset copy can still fall back to the reserveTokensFloor hint, overflow recovery still retries compaction up to three times, tool-result recovery can still report an empty session, and post-compaction AGENTS.md reinjection remains enabled by default. The related reinjection-loop issue #66126 is open, so this should stay available for maintainer follow-up rather than being swept closed.

## What This Changes

Investigate and improve repeated context-overflow reset loops where high reserveTokensFloor settings, retry compaction, tool-result truncation, and bootstrap context reinjection combine to keep a session failing.

## Best Possible Solution

Keep this issue open. The best path is a maintainer-scoped fix that makes hard-reset guidance aware of the effective reserve floor, configured contextWindow/contextTokens, and already-high reserve settings, then explicitly decides how retry/post-compaction paths should suppress, shrink, dedupe, or mark bootstrap reinjection. Keep #66126 as the narrower retry reinjection loop and #55366 as the hard-reset state-loss UX follow-up unless maintainers intentionally merge those scopes.
