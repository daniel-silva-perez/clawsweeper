## Summary

Keep open. Current main has partial mitigations for context-window/reserve-floor sizing and aggregate tool-result truncation, but it still preserves the reported remaining behavior: the reset fallback can still point users at reserveTokensFloor when that is already high, the overflow path is still designed to retry compaction up to three times, and docs/code still allow post-compaction/bootstrap context reinjection on retry-like paths.

## Best Possible Solution

Keep this open and scope the remaining work explicitly: make context-overflow reset guidance model/config-aware when reserveTokensFloor is already high or contextWindow/contextTokens is the likely bottleneck, then decide whether repeated overflow retries should suppress or shrink bootstrap/post-compaction reinjection. Coordinate the reinjection portion with #66126 and keep the hard-reset state-loss UX separate under #55366 unless a maintainer wants a larger recovery redesign.
