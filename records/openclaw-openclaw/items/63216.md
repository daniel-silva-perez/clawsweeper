## Summary

Keep open. Current main has new partial overflow mitigations, including pre-prompt and optional mid-turn prechecks, but the reported recovery shape still exists: context-overflow results can still trigger a session reset with a generic reserveTokensFloor fallback, retry/compaction remains bounded, post-compaction bootstrap reinjection remains default behavior, and empty-session tool-result truncation is still a possible diagnostic path.

## What This Changes

The issue requests investigation of repeated context-overflow hard resets where retry/compaction paths may rebuild bootstrap context and give misleading reserveTokensFloor guidance even with a high reserveTokensFloor.

## Best Possible Solution

Define a maintainer-approved overflow-recovery contract that separates model context-window misconfiguration from reserve-floor tuning, makes reset guidance budget-aware, and gives retry/post-compaction paths explicit rules for suppressing, shrinking, deduping, or marking bootstrap reinjection before falling back to a hard reset.

## Reproduction Assessment

Yes for the source-level recovery shape: force an embedded context-overflow meta error or oversized prompt path and the auto-reply layer still resets the session and builds the fallback recovery text. The exact Basecamp/Voltti session is not reproducible from this checkout alone.

## Solution Assessment

No. The current precheck and tool-result truncation changes are useful mitigations, but they do not fully solve the broad issue because reset UX, generic reserveTokensFloor guidance, and bootstrap reinjection policy still need a coherent product-level recovery contract.
