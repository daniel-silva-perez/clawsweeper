## Summary

Keep open. Current main still has the reported overflow-recovery shape: hard-reset guidance can still fall back to a generic reserveTokensFloor hint, overflow recovery still performs bounded retry/compaction loops, tool-result truncation can still report an empty session, and post-compaction bootstrap reinjection remains the default path. The narrower retry-reinjection report #66126 is open but does not cover the whole remaining scope here.

## What This Changes

Investigate and improve context-overflow recovery so high reserve floors, repeated compaction/truncation failures, and bootstrap reinjection do not keep causing hard resets on the same session key.

## Best Possible Solution

Define a maintainer-approved overflow-recovery contract that separates model-budget problems from reserve-floor tuning, makes reset guidance budget-aware, and gives retry/post-compaction paths explicit rules for suppressing, shrinking, deduping, or marking bootstrap reinjection.
