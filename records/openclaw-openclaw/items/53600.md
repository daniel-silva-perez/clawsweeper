## Summary

Keep this issue open. Current main has useful generic docs and diagnostics for VPS startup, prompt/context mechanics, memory search, compaction, queues, subagents, Telegram delivery, hooks, and OTEL export, but it still does not provide the requested systematic native-VPS reasoning-latency analysis or instrumentation plan with measured or estimated contribution by layer. The prior Codex review reached the same conclusion, and related #54470 is a different native Windows Gmail/gcloud crash, not a duplicate or superseding tracker.

## Best Possible Solution

Keep this open and scope it into a bounded native-Linux gateway performance investigation. Add or document a trace/runbook that captures bootstrap/context assembly timing, token estimation and truncation, memory query embedding/search and recall, compaction trigger and duration, queue waits, subagent spawn and announce-back, Telegram preview/final delivery, hook execution, auth profile decisions, and heap/RSS/GC. Publish quick-win VPS config guidance in the Linux server/performance docs, then split measured hot spots into focused optimization issues.
