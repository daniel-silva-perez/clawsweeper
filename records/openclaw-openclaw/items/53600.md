## Summary

Keep open. Current main has generic diagnostics, context, memory, compaction, concurrency, VPS, and Telegram tuning documentation, but it still does not contain the requested systematic native-Linux VPS reasoning-latency analysis or instrumentation plan with layer-by-layer measured or estimated latency contributions.

## What This Changes

The issue asks for a documented analysis or instrumentation plan for reasoning slowness on a constrained native Hetzner VPS setup, covering context assembly, memory search, compaction, subagents, Telegram delivery, hooks, auth fallback, and Node/OS overhead.

## Best Possible Solution

Scope this into a bounded native-Linux gateway performance investigation: publish a runbook or doc that captures context assembly duration, token estimation/truncation, memory query embedding/search, compaction trigger/duration, queue wait, subagent spawn/announce, Telegram preview/final delivery, hook execution, auth decisions, and heap/RSS/GC, then split any measured hot spots into focused optimization issues.
