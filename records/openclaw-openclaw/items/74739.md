## Summary

Keep open. The PR addresses a real package-update gap, but the proposed restart-verification child process is capped at 70s even though its own stale-PID recovery can legitimately spend longer than that on restart and health waits, so merging as-is can create false failed updates.

## What This Changes

The PR modifies the update CLI to delegate package-update restart verification to the freshly installed OpenClaw entrypoint via a hidden restart-verification mode and updates CLI tests around restart/version/plugin-error verification.

## Best Possible Solution

Revise the PR so delegated restart verification gets a budget that covers the composed recovery path, preferably derived from the update timeout or explicit child-step budgets, and add coverage for stale-PID cleanup before re-running the focused CLI update tests and changed gate.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Raise delegated restart verification timeout budget:** `src/cli/update-cli/update-command.ts:554`
  - body: The parent limits the child `openclaw update` restart verifier to `RESTART_VERIFY_TIMEOUT_MS` (70s), but the verifier can enter stale-PID recovery and then spend up to 60s running `gateway restart` plus another 60s waiting for health. The parent can kill a valid recovery path early and report a package update failure even though the gateway would recover shortly after.
