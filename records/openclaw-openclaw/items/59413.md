## Summary

Keep this open. Current main has model fallback, auth-profile rotation, model-scoped rate-limit cooldowns, and configurable same-provider profile-rotation limits, but it still does not expose the requested per-fallback-candidate or provider/model retry-before-fallback budget needed for pool/proxy providers.

## What This Changes

The issue requests configurable retry counts at the fallback candidate or provider/model route level so pool-based proxy providers can retry before moving to the next model fallback.

## Best Possible Solution

Add a backwards-compatible route-level retry-before-fallback policy, likely under provider/model configuration or an extensible fallback-entry shape, with runtime support, schema/docs updates, and tests proving pool/proxy 502 or timeout retries do not break existing string fallback lists or fail-fast behavior for direct providers.

## Reproduction Assessment

Yes for the missing capability: current main can be checked deterministically by configuring a fallback chain and mocking the first candidate to fail, which shows the fallback runner advances after one candidate attempt. A full production reproduction of the reported TicketPro pool hit rate would require that external provider.

## Solution Assessment

No current solution is sufficient. Existing `auth.cooldowns.*ProfileRotations` controls same-provider auth-profile rotation and cooldown behavior, but it is not the requested per-candidate or provider/model retry budget; the safer next step is a maintainer API decision before implementation.
