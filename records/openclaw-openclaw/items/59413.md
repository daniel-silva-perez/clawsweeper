## Summary

Keep open. Current main has global auth cooldown and same-provider profile-rotation controls, plus cooldown probes, but it still does not expose a per-fallback-candidate or provider/model retry-before-fallback budget; the related PR discussion also identified that remaining granularity gap after PR #55118 was closed unmerged.

## What This Changes

The issue requests configurable retry counts for individual fallback candidates or provider/model routes so pool-based proxy providers can retry the same candidate before moving to the next fallback.

## Best Possible Solution

Add a documented provider/model-scoped retry-before-fallback policy that composes with existing auth-profile rotation, cooldown, and probe behavior, with schema/types/help/docs/tests updated together; only add per-agent or per-candidate overrides if maintainers explicitly choose that extra granularity.

## Reproduction Assessment

Yes for the missing OpenClaw surface: current-main schema rejects per-candidate fallback objects and the fallback loop runs each model candidate once. The external proxy pool failure itself would still need provider credentials or a log-replay harness for live confirmation.

## Solution Assessment

Unclear until maintainers choose the config contract. A provider/model-scoped retry policy looks safer than ad hoc candidate objects because current config already treats provider/model as an operational unit, but it should be designed against the existing auth cooldown/profile-rotation semantics.
