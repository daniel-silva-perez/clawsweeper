## Summary

Keep this issue open. Current main has improved model failover and global auth-profile rotation controls, but it still does not provide per-fallback-candidate or provider/model retry budgets for a single proxy provider that internally rotates keys. The related PR #55118 was closed unmerged, and the discussion explicitly narrowed the remaining ask to provider/model or candidate-level retry granularity.

## Best Possible Solution

Keep this issue open and implement a conservative core failover config for provider/model-route or fallback-candidate retry budgets, with schema/types/docs and tests. The design should likely build on the existing `auth.cooldowns.*` profile-rotation controls but add route-level retry-before-fallback for proxy/pool providers, including reason filtering for timeout/5xx/rate-limit cases and clear bounds to avoid duplicate side effects.

## Risks / Open Questions

- Closing this now would lose a still-valid operator use case: a single proxy provider/auth profile with internal key-pool rotation cannot be tuned separately from direct providers.
- Any implementation should avoid replaying non-idempotent multi-step agent work; retry-before-fallback should be bounded and scoped to replay-safe provider request failures or carefully classified pre-side-effect attempts.
