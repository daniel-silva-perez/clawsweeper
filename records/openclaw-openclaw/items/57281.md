## Summary

Keep open. Current main still performs the potentially locked auth-profile failure write before advancing to the next profile in both the prompt-side and assistant-side failover paths, and the linked implementation PR that says it fixes this issue is still open and unmerged. The right cleanup is to review/land or close that PR, not close this issue first.

## Best Possible Solution

Keep this issue open and review PR #57283 as the implementation candidate. The likely fix belongs in the embedded runner failover paths: advance the auth profile before awaiting or asynchronously scheduling cooldown marking, and add focused regression coverage that proves rate_limit failover rotates without waiting on the auth-store lock while still recording cooldown eventually.
