## Summary

Keep this issue open. Current main still does not expose `costCurrency` or any equivalent pricing-currency contract; pricing remains modeled, stored, documented, and rendered as USD/dollar values across config, runtime usage estimation, session APIs, chat/CLI/status output, Control UI, and macOS.

## Best Possible Solution

Keep this issue open for a real core pricing-currency design. Maintainers should choose the contract boundary first, such as explicit USD-only storage plus display conversion, or per-provider/per-model/global currency metadata. Then update config schema/types/generated help, pricing lookup and aggregation, transcript/session/API fields, CLI/chat/status output, Control UI/macOS displays, docs, and tests, including mixed-currency behavior.

## Risks / Open Questions

- A display-only symbol option could mislead users when a session aggregates providers priced in different currencies unless the contract defines split, conversion, suppression, or explicit labeling behavior.
- Existing persisted/API names such as `estimatedCostUsd`, `costUsd`, `formatUsd`, and `openclaw_model_cost_usd_total` mean this needs compatibility planning beyond a narrow formatter rename.
