## Summary

Keep this issue open. Current `main` still has no `costCurrency` or equivalent currency metadata, and the cost pipeline remains USD-shaped from config schema/types through runtime formatting, session/API fields, CLI/chat/status output, Control UI/macOS formatting, telemetry, and docs. The prior ClawSweeper comment is still accurate against current main `bdfb408ce6eeb724a638c106104dbbcae50b612c`.

## Best Possible Solution

Keep this open for a core pricing-currency design. Maintainers should decide whether OpenClaw stores USD-only estimates, per-provider/per-model currency metadata, a global display currency, conversion, or mixed-currency summaries, then update config schema/types/generated help, pricing lookup and aggregation, transcript/session/API fields, CLI/chat/status output, Control UI/macOS displays, telemetry/docs, and focused tests.

## Risks / Open Questions

- A display-only symbol option could mislead users when one session aggregates providers or models priced in different currencies.
- A proper fix likely touches compatibility-sensitive public names such as `estimatedCostUsd`, `costUsd`, `formatUsd`, `openclaw_model_cost_usd_total`, and `openclaw.cost.usd`, so it needs an explicit migration/backward-compatibility design.
