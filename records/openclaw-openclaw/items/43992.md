## Summary

Keep open. Current main still has the reported gap, and the PR remains useful, but it is not merge-ready after the current subagent run-manager refactor: gateway `agent.wait` still drops the stored result, `run-wait` drops it before subagent completion code can use it, and the branch fallback is wired to the old inline registry path while also missing normal `{ payloads, meta }` result envelopes. The author did address the earlier fallback-overwrite concern raised in review.

## What This Changes

The PR threads `agent.wait` result payloads through gateway dedupe/wait responses, uses wait-result text as a subagent announce/completion fallback when history is behind, and adds gateway plus subagent regression tests.

## Best Possible Solution

Land a current-main implementation that preserves the structured gateway result through dedupe and every `agent.wait` response, carries it through the shared run-wait layer, parses normal result envelopes, and uses the extracted text only as a fallback when chat-history freezing has not already captured a reply.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Parse agent result payload envelopes:** `src/agents/subagent-announce.ts:345-347`
  - body: `extractAgentWaitResultText` passes `wait.result` straight to the text extractor, but normal gateway agent runs store an envelope shaped like `{ payloads, meta }`. Because the helper does not look inside `payloads`, ordinary completions such as `{ payloads: [{ text: "..." }] }` still produce no fallback text when history has not caught up.
  - confidence: 0.9
- **[P1] Port fallback to the run manager:** `src/agents/subagent-registry.ts:1259-1264`
  - body: This wires the completion fallback into the old inline registry wait path. Current main performs subagent completion waits through `subagent-registry-run-manager.ts` and `waitForAgentRun`, whose result type still drops the result, so the real completion path will not receive the fallback after rebase.
  - confidence: 0.87
- **[P2] Wait for the result-bearing dedupe snapshot:** `src/gateway/server-methods/agent.ts:772-775`
  - body: When the lifecycle promise wins the race, this response path aborts the dedupe wait and returns a lifecycle snapshot that never carries `result`. Agent execution can emit lifecycle end before the gateway writes the final dedupe payload, so live `agent.wait` callers can still get `result: undefined`.
  - confidence: 0.84
