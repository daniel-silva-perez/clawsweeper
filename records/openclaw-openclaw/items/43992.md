## Summary

Keep open. The bug the PR targets is real on current main, but the branch is not merge-ready: it misses normal result envelopes, does not port the fallback into the current subagent run-manager path, and can still return a lifecycle snapshot before the dedupe result payload exists.

## What This Changes

The PR threads `agent.wait` result payloads through gateway dedupe snapshots and responses, uses that text as a subagent announce/registry fallback, and adds gateway plus subagent regression tests.

## Best Possible Solution

Land a current-main implementation that carries `agent.wait` result payloads end to end, parses normal `{ payloads, meta }` result envelopes, waits or merges carefully when lifecycle and dedupe terminal signals arrive in different orders, and freezes subagent result text only as a fallback before completion delivery.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main stores `payload.result` but drops it from dedupe snapshots, `agent.wait` responses, `waitForAgentRun`, and the active subagent run-manager completion path.

## Solution Assessment

No. The PR is directionally useful, but it is not the best current fix because it targets an obsolete registry wait path, misses the normal result envelope shape, and does not guarantee a result-bearing snapshot when lifecycle wins the live wait race.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Parse result payload envelopes:** `src/agents/subagent-announce.ts:345-347`
  - body: `extractAgentWaitResultText` only checks direct scalar fields and chat-style arrays. Normal gateway agent results are envelopes like `{ payloads: [{ text: "..." }], meta }`, so the fallback still drops the completion text when history has not caught up.
  - confidence: 0.91
- **[P1] Port fallback into the run manager:** `src/agents/subagent-registry.ts:1259-1264`
  - body: This fallback is added to the older inline registry wait path, but current main completes subagent waits through `createSubagentRunManager` and `waitForAgentRun`. After rebase, the active completion path still will not receive or freeze the fallback text.
  - confidence: 0.9
- **[P2] Wait for result-bearing snapshots:** `src/gateway/server-methods/agent.ts:772-775`
  - body: The live `agent.wait` path returns the first terminal snapshot and aborts the other waiter. If lifecycle wins before the gateway dedupe entry writes `payload.result`, the added `result` field can still be `undefined`, so callers miss the completion payload.
  - confidence: 0.85
