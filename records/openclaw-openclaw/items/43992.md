## Summary

Keep open. The PR addresses a real current-main gap, but the branch is not merge-ready: it does not decode the normal `{ payloads, meta }` agent result envelope, it patches an obsolete registry wait path instead of the active run manager, and the live gateway wait race can still return before the result-bearing dedupe snapshot is available.

## What This Changes

The PR threads `agent.wait` result payloads through gateway dedupe snapshots/responses, reuses that text as a subagent announce/registry fallback, and adds gateway plus subagent regression tests.

## Best Possible Solution

Land a current-main implementation that defines `result` in the `agent.wait` path end to end, preserves existing wait metadata, extracts text from normal `{ payloads, meta }` envelopes, waits or merges carefully when lifecycle and dedupe finish in different orders, and fills `frozenResultText` only as a fallback before completion delivery.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main stores `payload.result`, then drops it from the dedupe snapshot, both `agent.wait` response paths, `waitForAgentRun`, and the active subagent run-manager completion path.

## Solution Assessment

No. The PR is directionally useful but not the best current fix because it targets an obsolete registry wait path, misses normal result envelopes, and still permits a lifecycle-first live wait to return without the dedupe result.

  - confidence: 0.9

  - body: The live `agent.wait` path returns the first terminal snapshot and aborts the other waiter. If lifecycle wins before the gateway dedupe entry writes `payload.result`, the added `result` field is still `undefined`, so callers can miss the completion payload.
  - confidence: 0.85
