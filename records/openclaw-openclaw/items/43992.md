## Summary

Keep open. Current main still stores agent command results but drops them from the terminal wait snapshot, the `agent.wait` response, and the shared run-wait helper; the PR addresses the right area but needs rework because it targets an older registry path, misses normal `{ payloads, meta }` result envelopes, and can still return a lifecycle-only snapshot before the result-bearing dedupe entry is available.

## What This Changes

This PR threads `agent.wait` result payloads through gateway dedupe/wait responses, uses wait-result text as a fallback for subagent announce/completion flows, and adds gateway plus subagent regression tests.

## Best Possible Solution

Land a current-main implementation that defines `result` in the `agent.wait` contract end to end, extracts reply text from structured result envelopes, and fills `frozenResultText` only as a fallback before completion delivery without provider-specific core behavior.

  - confidence: 0.9

  - body: The live `agent.wait` path returns the first terminal snapshot and aborts the other waiter. If lifecycle wins first, it has no `result`, while the result-bearing dedupe entry is written afterward, so callers can still receive `result: undefined`.
  - confidence: 0.85
