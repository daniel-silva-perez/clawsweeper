## Summary

Keep open. The PR addresses a real current-main gap, but it is not merge-ready: current main still drops the stored `agent.wait` result before it reaches subagent completion, while the PR branch targets an older registry wait path, misses the normal `{ payloads, meta }` result envelope, and can still return a lifecycle snapshot before the result-bearing dedupe snapshot is available.

## What This Changes

The PR threads `agent.wait` result payloads through gateway dedupe responses, uses that text as a subagent completion fallback when chat history is late, and adds gateway plus subagent regression tests.

## Best Possible Solution

Land a current-main implementation that defines `result` in the `agent.wait` contract end to end, extracts reply text from structured `{ payloads, meta }` result envelopes, and fills `frozenResultText` only as a fallback before completion delivery without provider-specific core behavior.

## Reproduction Assessment

Yes. The reproduction path is source-level and high confidence: current main stores `payload.result`, then omits it from the dedupe snapshot, `agent.wait` response, `waitForAgentRun`, and the active subagent run-manager completion path.

## Solution Assessment

No. The PR is directionally useful, but it is not the best current fix because it targets an obsolete registry wait path, does not parse normal result envelopes, and leaves a live wait race that can still return without a result.
