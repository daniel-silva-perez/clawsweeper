## Summary

Keep open. Current main still does not expose the terminal agent result through the full `agent.wait` path, and this PR remains useful source work, but the branch is not merge-ready because it misses current run-manager plumbing, normal result-envelope parsing, and a live wait race. The author addressed the earlier fallback-overwrite concern, so the remaining work is narrower than the original review discussion.

## What This Changes

The PR threads `agent.wait` result payloads through gateway dedupe/wait responses, uses extracted wait result text as a subagent announce/completion fallback, and adds gateway plus subagent regression coverage.

## Best Possible Solution

Port the behavior onto current main by preserving the structured result in gateway dedupe snapshots and all `agent.wait` responses, carrying it through `run-wait` into the current subagent run manager, extracting text from normal `{ payloads, meta }` result envelopes, and using that text only as a fallback when chat-history freezing has not already captured a reply.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Parse agent result payload envelopes:** `src/agents/subagent-announce.ts:345-347`
  - body: `extractAgentWaitResultText` passes `wait.result` directly to the text extractor, but normal gateway agent runs return an envelope shaped like `{ payloads, meta }`. Since that helper does not look inside `payloads`, ordinary completions such as `{ payloads: [{ text: "..." }] }` still produce no fallback text when history has not caught up.
  - confidence: 0.9
- **[P1] Port fallback to the run manager:** `src/agents/subagent-registry.ts:1251-1264`
  - body: This wires the completion fallback into the old inline registry wait path. Current main now performs subagent completion waits in `subagent-registry-run-manager.ts` through `waitForAgentRun`, whose type still drops the result, so the real completion path will not receive the fallback after rebase.
  - confidence: 0.87
- **[P2] Wait for the result-bearing dedupe snapshot:** `src/gateway/server-methods/agent.ts:772-775`
  - body: When the lifecycle promise wins the race, this response path aborts the dedupe wait and returns a lifecycle snapshot that never carries `result`. Agent execution can emit lifecycle end before the gateway writes the final dedupe payload, so live `agent.wait` callers can still get `result: undefined`.
  - confidence: 0.84
