## Summary

Keep open. Current main still leaves loop detection disabled by default and generic_repeat warning-only, so the reliability problem is not obsolete; however, this PR should not merge as-is because it changes documented exec-approval layering, can drop provider-specific tool-call state during error stripping, and adds hard generic-repeat blocking based on aggregate history rather than a tight loop.

## What This Changes

The PR enables tool-loop detection by default with lower thresholds, adds generic_repeat critical blocking, changes exec host ask merging, strips trailing infrastructure-error turns before session sanitization, and adds SANDBOX_POLICY.json to the agent file whitelist.

## Best Possible Solution

Land a narrower, maintainer-approved fix that keeps host exec approvals as stricter-of-two, implements any generic_repeat hard stop using consecutive/no-progress evidence, updates the loop-detection docs/tests if default-on is accepted, and moves queue-flood cleanup plus sandbox-policy file delivery into separate reviewed PRs.

## Reproduction Assessment

Yes. Current-main source and docs reproduce the reported loop-detection gap, and the proposed regressions are source-reproducible by comparing tools.exec.ask=off with host ask=always and by passing an errored assistant turn containing toolUse or functionCall content through the new predicate.

## Solution Assessment

No. The PR tackles a real reliability issue, but the current patch is not the best maintainable solution because it weakens documented approval layering and blocks generic repeats from aggregate history rather than a proven loop streak; a narrower detector change plus separate approval/session-cleanup work is safer.

  - confidence: 0.9
- **[P2] Block only consecutive generic repeats:** `src/agents/tool-loop-detection.ts:488-492`
  - body: The new critical branch uses the total count of matching tool+args anywhere in the 50-entry history. A healthy session that revisits the same command intermittently can hit the hard block without being stuck; use a consecutive/no-progress streak or keep generic_repeat warning-only until there is loop-tail evidence.
  - confidence: 0.86
