## Summary

Keep open. Current main still has evidence of the same split source-of-truth class: Control UI chat/context warning data comes from Gateway `sessions.list`, which can backfill fresh transcript usage, while `openclaw status` builds its session rows from the session store only and explicitly preserves stale token values. The related PR #48364 was closed unmerged and targeted a browser-control `/status` route, not the Gateway/Control UI session-usage mismatch reported here.

## Best Possible Solution

Keep the issue open and fix the shared session-usage source. The best path is to make `openclaw status` use the same Gateway session row resolver as `sessions.list`/`loadGatewaySessionRow`, or extract a single read-only usage resolver used by both CLI status and Control UI. Add a regression test with stale store `totalTokensFresh: false` plus fresher transcript usage to prove both surfaces report the same token count, and separately verify pending device pairing still flows through `device.pair.list`/local loopback fallback.
