## Summary

Keep open. Current main still has the split source-of-truth described in the report and in Ryce's comment: the Control UI chat context notice is fed by Gateway session rows that can backfill fresher transcript usage, while `openclaw status` and the Gateway `status` RPC still build session rows directly from the session store and preserve stale token totals. The related #48364 was closed unmerged and scoped to a browser-server `/status` route, so it does not resolve the Dashboard/CLI session-usage mismatch.

## Best Possible Solution

Keep this open and fix the shared session-usage source. The strongest path is to make `src/commands/status.summary.ts` use the same read-only resolver as `src/gateway/session-utils.ts` (`loadGatewaySessionRow`/`listSessionsFromStore`) or extract a shared session-usage resolver used by both CLI/Gateway status and Control UI. Add a regression test with stale store usage plus fresher transcript usage proving `openclaw status --json` and `sessions.list` report the same token count, and separately recheck the pending-device path through `device.pair.list` plus the local loopback fallback.
