## Summary

Keep this PR open for maintainer security follow-up. Current main still lacks the requested gateway-layer outbound secret sanitization, but the proposed patch is not merge-ready because it sanitizes too early, only covers `payload.text`, uses an incomplete credential index, and advertises an opt-out that is not in the config contract.

## What This Changes

The PR adds an outbound secret-sanitization helper and tests, then wires delivery normalization to replace detected `payload.text` secrets with named `key(...)` handles or `[REDACTED:high-entropy]`, with a `messages.sanitizeSecrets` opt-out helper.

## Best Possible Solution

Land a revised gateway final-send sanitizer that runs after hooks, presentation rendering, payload normalization, and channel-specific rendering; recursively sanitizes all outbound user-visible string surfaces; derives named handles from canonical SecretRef/sensitive config metadata and runtime auth-profile resolution; and updates config schema, exported types, generated help, docs, and focused regression tests.

## Reproduction Assessment

Yes. The review findings are reproducible by source inspection: the PR diff inserts redaction before `message_sending`, only under `sanitizedPayload.text`, while current delivery code can rewrite content later and send structured payload fields without another sanitizer.

## Solution Assessment

No, not as-is. A gateway-layer final-send boundary is the right direction, but the current patch should be revised to use the canonical sensitive/SecretRef surfaces and to sanitize after every mutation path before transport delivery.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Re-sanitize after message_sending hooks:** `src/infra/outbound/deliver.ts:625`
  - body: The new secret pass is wired into payload normalization, but `message_sending` hooks run later and can replace `payload.text` before the final `sendPayload`/`sendFormattedText`/`sendText` calls. A hook that appends or substitutes a secret therefore bypasses this security boundary entirely, so the sanitizer needs to run again at the final effective-payload send point.
  - confidence: 0.93
- **[P2] Sanitize structured payload strings too:** `src/infra/outbound/deliver.ts:328-329`
  - body: This only calls `sanitizeSecrets` for `sanitizedPayload.text`, leaving `presentation`, `interactive`, and `channelData` strings untouched even though those payloads can be delivered through `sendPayload`. Secrets in button labels/values, select placeholders/options, presentation text, or channel-specific data will still be sent verbatim.
  - confidence: 0.9
- **[P1] Build the index from canonical secret surfaces:** `src/infra/outbound/sanitize-secrets.ts:82-108`
  - body: `buildCredentialIndex` only walks `cfg.auth` and top-level `cfg.channels` string properties. Many real sensitive config values live under registered SecretRef/sensitive paths such as model/tool/talk provider keys, gateway auth, provider headers, and nested channel account fields, so low-entropy configured secrets on those surfaces will miss the named-handle path and may bypass redaction.
  - confidence: 0.88
- **[P1] Tokenize quoted and structured values before matching:** `src/infra/outbound/sanitize-secrets.ts:151`
  - body: The tokenizer splits on whitespace, commas, semicolons, pipes, and `=`, but keeps quotes, braces, and colons attached. Common JSON/YAML/code output such as `{"token":"ghp_..."}` or `password: "shared-password-0000"` will not exact-match configured credentials or known prefixes, so those values can bypass the sanitizer.
  - confidence: 0.86
- **[P2] Add the sanitizeSecrets option to the config contract:** `src/infra/outbound/sanitize-secrets.ts:216-219`
  - body: The helper reads `messages.sanitizeSecrets`, but the strict `MessagesSchema` and `MessagesConfig` do not declare that field. Users cannot set the advertised opt-out in `openclaw.json` without schema validation rejecting it, so the config surface needs schema, type, docs, and generated-help coverage.
  - confidence: 0.91

## Security Review

Status: needs_attention

Summary: The PR changes an outbound security boundary, and the current diff leaves concrete bypasses that need maintainer security review before merge.

Concerns:

- **[high] Hook rewrites bypass the sanitizer:** `src/infra/outbound/deliver.ts:625`
  - body: Because redaction is applied before `message_sending` hooks, hook-supplied replacement content can introduce a secret after the only sanitizer pass and still reach channel delivery.
  - confidence: 0.93
- **[medium] Structured outbound payloads bypass redaction:** `src/infra/outbound/deliver.ts:328`
  - body: The sanitizer only rewrites `payload.text`, while structured payload fields can carry user-visible strings through `sendPayload`; this leaves a practical exfiltration path for buttons, presentations, and channel-specific payloads.
  - confidence: 0.9
- **[medium] Incomplete credential index misses configured secrets:** `src/infra/outbound/sanitize-secrets.ts:82`
  - body: The ad hoc index omits many registered sensitive/SecretRef config surfaces, so known low-entropy secrets outside top-level auth/channel strings may not receive named handles and may not be caught by entropy rules.
  - confidence: 0.88
