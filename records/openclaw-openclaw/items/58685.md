## Summary

Keep open. The PR is useful, but the patch does not fix the common Claude missing-scope path because current provider usage code returns an error snapshot instead of throwing; the new warning only runs in the catch block. The classifier also hides raw Claude 403 details too broadly, and the user-facing CLI change still needs a changelog entry.

## What This Changes

The PR adds structured warnings for `openclaw channels list` usage-load exceptions, adds focused command tests, updates channels CLI docs, and includes small docs/test ordering cleanups.

## Best Possible Solution

Handle usage failure messaging in one shared CLI formatting path that classifies both thrown load failures and resolved `ProviderUsageSnapshot.error` entries before `formatUsageReportLines`, preserves raw diagnostics unless the error explicitly signals `user:profile`, and keeps the separate JSON-output behavior tracked by #59472 out of this PR unless intentionally addressed.

## Reproduction Assessment

Yes. A mocked Claude OAuth 403 body containing `scope requirement user:profile` and no valid Claude web session makes `fetchClaudeUsage` return a snapshot with `error: "HTTP 403: missing scope requirement user:profile"`; `channelsListCommand` then formats that snapshot instead of entering the PR's catch block.

## Solution Assessment

No. The catch-only warning is not the narrowest correct solution because the real Claude scope path commonly resolves successfully with an error snapshot; the safer fix is to classify explicit scope errors at the snapshot-formatting boundary while keeping raw diagnostics for other 403s.
