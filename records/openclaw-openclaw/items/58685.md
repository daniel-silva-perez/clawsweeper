## Summary

Keep this PR open, but do not merge it as-is. Current main still has the raw `channels list` usage-error behavior the PR is trying to improve, and the PR is useful direction, but the patch misses the common Claude scope-failure path because that failure is returned as a normal usage snapshot error rather than thrown. It also hides raw diagnostics for any Claude 403, even when the 403 is not clearly the `user:profile` scope problem.

## What This Changes

The PR adds warning-oriented error handling and tests for `openclaw channels list` usage snapshot failures, updates the channels CLI docs, and includes small MiniMax test-order and video-generation docs cleanup.

## Best Possible Solution

Land a narrow current-main fix that classifies both thrown usage-load failures and resolved `ProviderUsageSnapshot.error` entries before formatting, shows Claude-specific guidance only for explicit `user:profile`/scope signals, preserves raw diagnostics for ambiguous 403s, and updates the focused CLI docs/tests around that behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is available: mock a Claude 403 `scope requirement user:profile` usage response, observe that `fetchClaudeUsage` returns an error snapshot, and then `channelsListCommand` formats that snapshot through `formatUsageReportLines` instead of entering the new catch block.

## Solution Assessment

No. The PR's direction is right, but catch-only handling is not the best fix because the known Claude scope path normally resolves successfully with an error snapshot; the safer solution is to classify snapshot errors before rendering and reserve Claude-specific suppression for explicit scope signals.
