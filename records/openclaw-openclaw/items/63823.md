## Summary

Keep open. Current main still lacks the MiniMax manifest `providerAuthAliases` mapping, while MiniMax runtime still declares the CN hook aliases and the current auth resolver only maps auth aliases from manifest metadata. The linked open fix PR #72798 remains the active implementation path, so closing this issue before that PR or an equivalent patch lands would be premature.

## What This Changes

The issue requests adding MiniMax provider auth alias mappings for `minimax-cn` and `minimax-portal-cn` to the MiniMax plugin manifest so CN endpoint credentials resolve through the manifest-backed provider auth alias system.

## Best Possible Solution

Review and land #72798, or apply an equivalent maintainer patch that adds `providerAuthAliases` for `minimax-cn` and `minimax-portal-cn` in `extensions/minimax/openclaw.plugin.json` with focused MiniMax manifest regression coverage. No core auth resolver change appears necessary.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: the MiniMax manifest has no `providerAuthAliases`, so `resolveProviderIdForAuth("minimax-cn")` falls back to `minimax-cn` while the CN auth choices are owned by `minimax` and `minimax-portal`.

## Solution Assessment

Yes. The proposed manifest alias map plus focused regression test is the narrowest maintainable fix because the core resolver is already designed around manifest-backed aliases and sibling providers use the same pattern.
