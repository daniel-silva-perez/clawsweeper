## Summary

Keep open. Current main still lacks the requested Slack RTS/action-token path: no `assistant.search.context` call, no inbound `action_token` extraction or propagation, and no Slack `search` message action. The open user-token `search.messages` work is related, but it does not satisfy bot-token RTS via assistant action tokens.

## What This Changes

Add Slack Real-Time Search support by extracting Slack assistant action tokens from inbound events and exposing `assistant.search.context` as a message or dedicated search action.

## Best Possible Solution

Implement a design-reviewed Slack plugin RTS path: capture the current Slack `action_token` from authorized `message`/`app_mention` payloads, scope it to the originating account/team/user/channel/thread/request, expose a typed read-only `assistant.search.context` action, document required AI/search scopes and privacy limits, and keep it distinct from user-token `search.messages` unless maintainers intentionally merge the two designs.

## Reproduction Assessment

Not applicable as a runtime bug report, but there is a high-confidence inspection path for the missing feature. Current-main symbol search plus Slack action discovery, dispatch, runtime, inbound event types, docs, and read-path checks all show no RTS/action-token implementation.

## Solution Assessment

Unclear until maintainers settle the security and product contract. The requested direction is plausible, but the safest implementation should follow Slack’s current `message`/`app_mention` action-token contract and not fold this blindly into the adjacent user-token `search.messages` work.
