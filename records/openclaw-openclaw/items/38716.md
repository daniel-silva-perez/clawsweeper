## Summary

Keep open. Current main still does not implement Slack RTS via assistant action tokens: the Slack plugin has no action-token capture path, no `assistant.search.context` call, no Slack `search` message action advertisement or dispatch, and the existing read action is still conversation history/replies. The related `search.messages` work in #65691 and #54224 is adjacent user-token search/retrieval work, not the requested assistant action-token RTS behavior.

## What This Changes

Request support for Slack Real-Time Search by extracting assistant event action tokens and exposing a Slack search action backed by `assistant.search.context`.

## Best Possible Solution

Implement the remaining behavior inside the bundled Slack plugin with an explicit RTS contract: capture the inbound Slack action token from the authorized assistant/message event, keep it scoped to the current request context, expose a typed read-only `assistant.search.context` action, document required Slack AI/search scopes and privacy limits, and keep it distinct from user-token `search.messages` unless maintainers deliberately consolidate them.

Status: needs_attention

  - body: Slack RTS can return broad workspace context depending on scopes and channel visibility, so maintainers should define consent, result sharing, logging, and rate-limit behavior before exposing it to agents.
