## Summary

Keep open. Current main still lacks the requested Slack RTS path: no `assistant.search.context` call, no captured Slack `action_token`, and no Slack `search` message action. The related #65691 work is adjacent user-token `search.messages` support, but it does not satisfy event-scoped assistant action-token RTS behavior.

## What This Changes

Request Slack plugin support for extracting assistant action tokens from Slack assistant/message events and exposing workspace search backed by `assistant.search.context`.

## Best Possible Solution

Implement a design-reviewed Slack plugin RTS path that captures the event-scoped `assistant_thread.action_token`, confines it to the originating Slack team/user/channel/thread/request context, exposes a typed read-only `assistant.search.context` action, and documents required AI/search scopes plus privacy and retention limits.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a runtime bug, but yes for feature absence: current-main source/docs inspection shows no RTS symbols, no action-token field in Slack event types, and no Slack search action dispatch path.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers settle the product and security contract; the requested direction is plausible, but it should be implemented as a scoped Slack plugin RTS feature rather than folded blindly into user-token `search.messages` work.
