## Summary

Keep open. Current main still lacks the requested Slack RTS/action-token path: there is no `assistant.search.context` implementation, no `action_token` field captured from Slack inbound events, and no Slack `search` message action. The related open Slack `search.messages` PR is adjacent user-token search work, but it does not satisfy bot-token RTS via event-scoped assistant action tokens.

## What This Changes

Add Slack plugin support for extracting Slack assistant/event `action_token` values and exposing workspace search through `assistant.search.context` as a message action or dedicated tool.

## Best Possible Solution

Implement the remaining work in the bundled Slack plugin behind an explicit RTS contract: capture the ephemeral Slack action token from authorized inbound `message`/`app_mention` payloads, scope it to the originating Slack account/team/user/channel/thread/request, expose a typed read-only `assistant.search.context` action, document required AI/search scopes and privacy limits, and keep it distinct from user-token `search.messages` unless maintainers deliberately consolidate the two designs.

## Reproduction Assessment

Not applicable as a runtime bug report, but there is a high-confidence inspection path for the missing feature. Current-main source search plus Slack action discovery, dispatch, runtime, inbound event types, docs, and the related PR body all show no RTS/action-token implementation.

## Solution Assessment

Unclear until maintainers settle the product and security contract. The requested direction is plausible, but the safest solution should be a design-reviewed Slack plugin RTS path using event-scoped action tokens rather than folding it blindly into adjacent user-token `search.messages` work.
