## Summary

Keep open. Current main still lacks the requested Slack RTS action-token search path: the Slack plugin does not capture Slack `action_token`, does not call `assistant.search.context`, and does not advertise or dispatch a Slack `search` message action. The related open Slack `search.messages` work is adjacent user-token search, but it does not satisfy event-scoped RTS behavior.

## What This Changes

Request to add Slack Real-Time Search support by capturing Slack assistant action tokens from inbound events and exposing an RTS-backed search action to agents.

## Best Possible Solution

Implement this in the bundled Slack plugin after design review: capture the event-scoped Slack action token from the inbound Slack event that authorized the request, keep it confined to that team/user/channel/thread/request context, expose a typed read-only RTS action that calls `assistant.search.context`, and update Slack setup docs for current granular RTS scopes and privacy limits.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a runtime regression, but yes for feature absence: current-main source/docs inspection shows no RTS method call, no action-token field in Slack event handling, and no Slack search action dispatch path.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainer product/security review settles the contract; the requested direction is plausible, but it should be implemented as a scoped Slack plugin RTS feature rather than folded blindly into user-token `search.messages` work.
