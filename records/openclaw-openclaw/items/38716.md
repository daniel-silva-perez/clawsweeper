## Summary

Keep open. Current main still lacks the requested Slack RTS/action-token path: there is no action-token capture from inbound Slack events, no `assistant.search.context` call, and no Slack `search` message action in discovery, dispatch, or runtime. The related `search.messages` work in #65691 and #54224 is adjacent user-token search/retrieval work, not the bot-token RTS flow requested here.

## What This Changes

Add Slack Real-Time Search support by capturing Slack assistant action tokens from inbound events and exposing `assistant.search.context` search to agents.

## Best Possible Solution

Implement the remaining work inside the bundled Slack plugin with an explicit RTS contract: capture the inbound Slack action token from authorized message/app_mention events, keep it scoped to the current request/thread/account context, expose a typed read-only `assistant.search.context` action, document required AI/search scopes and privacy limits, and deliberately decide how it coexists with user-token `search.messages`.

## Security Review

Status: needs_attention

Summary: No patch was reviewed, but the requested feature is security-sensitive because it would handle ephemeral Slack action tokens and workspace-wide search results.

Concerns:

- **[medium] Scope action tokens to the originating Slack context:**
  - body: An RTS implementation must not persist or reuse `action_token` values outside the Slack event, user, thread, team/account, and request context that authorized the search.
  - confidence: 0.88
- **[medium] Define workspace search privacy boundaries:**
  - body: Slack RTS can return broad workspace context depending on scopes and channel visibility, so maintainers should define consent, result sharing, logging, redaction, and rate-limit behavior before exposing it to agents.
  - confidence: 0.84
