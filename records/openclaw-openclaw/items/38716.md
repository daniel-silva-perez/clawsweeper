## Summary

Keep open. Current main still lacks the Slack RTS action-token path, and the adjacent user-token `search.messages` work does not satisfy the requested bot-token `assistant.search.context` flow.

## What This Changes

The issue requests Slack RTS support by capturing Slack assistant action tokens from inbound events and exposing `assistant.search.context` as a message search capability.

## Best Possible Solution

Implement this in the bundled Slack plugin with an explicit RTS contract: capture the ephemeral Slack `action_token` from authorized inbound `message` or `app_mention` events, keep it scoped to the originating team/account/channel/thread/request, expose a typed read-only `assistant.search.context` action, document required AI/search scopes and privacy limits, and deliberately keep or merge it with user-token search only after maintainer review.

## Security Review

Status: needs_attention

Summary: No patch was reviewed, but the requested feature is security-sensitive because it would introduce ephemeral Slack action-token handling and workspace-wide search results.

Concerns:

- **[medium] Scope action tokens to the originating Slack context:**
  - body: An RTS implementation must not persist or reuse `action_token` values outside the Slack event, user, thread, team/account, and request context that authorized the search.
  - confidence: 0.9
- **[medium] Define workspace search privacy boundaries:**
  - body: Slack RTS can expose broad workspace context depending on scopes and channel visibility, so maintainers should define consent, result sharing, logging, redaction, and rate-limit behavior before exposing it to agents.
  - confidence: 0.86
