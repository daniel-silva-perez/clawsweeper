## Summary

Keep open. Current main still lacks Slack RTS/action-token search: the Slack plugin does not type or carry `action_token`, does not call `assistant.search.context`, does not advertise or dispatch a Slack `search` message action, and its implemented read path is still Slack conversation history/replies rather than Real-Time Search. The related user-token `search.messages` work is adjacent but does not satisfy the assistant action-token RTS request.

## What This Changes

Add Slack Real-Time Search support by extracting Slack assistant action tokens from inbound events and exposing `assistant.search.context` through a Slack message action or dedicated tool.

## Best Possible Solution

Implement the remaining behavior inside the bundled Slack plugin: type and capture the inbound RTS action token from authorized Slack events, pass it only through the current message/assistant context, expose a typed read-only `assistant.search.context` action, document the required AI/search scopes and privacy boundaries, and keep the user-token `search.messages` path distinct unless maintainers intentionally consolidate them.

  - body: An RTS implementation must avoid persisting or reusing `action_token` values outside the user-initiated Slack event, thread, account, and requester context that authorized the search.
  - confidence: 0.88
- **[medium] Define workspace search privacy boundaries:**
  - body: Slack RTS can return broad workspace context depending on scopes and channel visibility, so maintainers should define consent, result sharing, rate-limit, logging, and scope behavior before exposing it to agents.
