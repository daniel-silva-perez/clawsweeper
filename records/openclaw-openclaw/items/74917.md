## Summary

Keep open. Current main still has the reported budget mismatch in the non-context-engine tool-result guard, and the issue is already paired with open closing PRs that should be reviewed or reconciled before the issue is closed.

## What This Changes

The issue asks OpenClaw to make the tool-result context guard use the resolved effective context budget instead of falling back to a lower model.contextWindow value.

## Best Possible Solution

Review and land #75092, #74934, or an equivalent narrow fix that passes contextTokenBudget into the non-context-engine tool-result guard before the existing fallbacks, with regression coverage for a resolved budget that differs from model.contextWindow.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. A focused source-level reproduction can construct an embedded attempt where contextTokenBudget differs from model.contextWindow and assert the current non-context-engine guard is installed with the lower model.contextWindow value.

## Solution Assessment

Is this the best way to solve the issue? Yes for the current resolver contract. Passing the already-resolved contextTokenBudget into the guard is the narrowest maintainable fix; metadata validation is a broader policy alternative.
