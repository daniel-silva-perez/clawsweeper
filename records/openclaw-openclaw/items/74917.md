## Summary

Keep open. Current `main` still has the mismatch reported here: the embedded runner resolves and forwards `ctxInfo.tokens` as the effective context budget, but the non-context-engine tool-result guard installs with a budget derived from `params.model.contextWindow`/`maxTokens`, so `contextTokens > contextWindow` can still enforce the smaller guard threshold.

## What This Changes

The issue asks OpenClaw to make the embedded runner tool-result guard use the same resolved effective context budget as the rest of the runtime when `contextTokens` and `contextWindow` diverge.

## Best Possible Solution

Make the non-context-engine tool-result guard share the same resolved budget used by the rest of the embedded runner, or explicitly reject/normalize inconsistent model metadata so every runtime context limit path agrees before a session starts.

## Reproduction Assessment

Yes. A focused source-level reproduction can set a model with `contextWindow` lower than the resolved `contextTokenBudget` and assert the attempt guard is installed with the lower value today; the relevant call path is visible without needing live provider credentials.

## Solution Assessment

Yes, with one product-policy caveat. If OpenClaw continues treating `contextTokens` as the effective runtime budget, passing `params.contextTokenBudget` into `installToolResultContextGuard` is the narrowest fix; if larger-than-native budgets should be invalid, the resolver should clamp or reject them and tests should assert that invariant instead.

In the Pi embedded runner non-context-engine path, ensure `installToolResultContextGuard` uses the resolved effective context budget (`params.contextTokenBudget`, i.e. `ctxInfo.tokens`) instead of falling back first to `params.model.contextWindow`. Preserve a defensive fallback to `params.model.contextWindow ?? params.model.maxTokens ?? DEFAULT_CONTEXT_TOKENS` only when the resolved budget is absent or invalid. Add focused regression coverage using a model whose `contextWindow` is lower than the attempt `contextTokenBudget` and assert the guard receives the resolved larger budget. Do not change provider catalogs, release metadata, dependency versions, or unrelated context-engine behavior. No changelog entry is needed unless the final behavior changes user-visible config semantics beyond bug consistency.
