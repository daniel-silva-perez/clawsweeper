## Summary

Keep open. Current main still has the reported budget mismatch in the non-context-engine tool-result guard, and the issue is already linked to open closing PRs that should be reviewed or consolidated rather than pre-closing the report.

## What This Changes

The issue asks OpenClaw to make the tool-result context guard use the resolved effective runtime context budget instead of falling back to the model's native `contextWindow` when the two values differ.

## Best Possible Solution

Review and land #75092, #74934, or an equivalent narrow fix that passes the resolved `contextTokenBudget` into the non-context-engine guard while preserving existing fallbacks and adding focused regression coverage for a budget that differs from `model.contextWindow`.

## Reproduction Assessment

Yes. A source-level reproduction can run the embedded attempt path without a context engine using `contextTokenBudget` larger than `model.contextWindow` and assert that current main installs the guard with the smaller `model.contextWindow` value.

## Solution Assessment

Yes, if OpenClaw continues to treat `contextTokens` as the effective runtime cap. Passing `params.contextTokenBudget` ahead of the existing fallback chain is the narrowest maintainable fix; metadata validation is a separate policy choice.
