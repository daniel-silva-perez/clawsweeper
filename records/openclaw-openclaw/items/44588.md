## Summary

Keep this PR open. Current main still omits cache read/write token counts from isolated cron run-log telemetry, while the supplied patch is a narrow additive forwarding change with no definite correctness or security issue found in review.

## What This Changes

The PR updates `src/cron/isolated-agent/run.ts` to copy positive `usage.cacheRead` and `usage.cacheWrite` values into cron telemetry as `cache_read_tokens` and `cache_write_tokens`.

## Best Possible Solution

Land this PR or an equivalent one-file fix, ideally with a focused regression test on the isolated cron runner or run-log write path that proves positive cache read/write counts appear in emitted cron telemetry. No docs or changelog entry looks necessary unless maintainers want a user-facing cost-accounting note.

## Reproduction Assessment

Yes. A high-confidence source reproduction is available: provide `finalRunResult.meta.agentMeta.usage` with positive `cacheRead` or `cacheWrite`, and current `finalizeCronRun` will enter the usage block but return telemetry without `cache_read_tokens` or `cache_write_tokens`.

## Solution Assessment

Yes. The proposed fix is the narrowest maintainable solution for the observed gap because the type/schema/reader/cost-estimator already support these fields; the missing step is only forwarding them from normalized isolated-run usage into `telemetryUsage`.
