## Summary

Keep this PR open. Current main still has the cron run-log telemetry gap: isolated cron stores cacheRead/cacheWrite on the session entry, and cost accounting can price those fields, but the CronRunTelemetry.usage object that is handed to the cron JSONL run log still only includes input/output/total tokens. The supplied PR remains a narrow one-file implementation candidate, and the diff does not show security-sensitive or unrelated changes.

## Best Possible Solution

Review and land this PR or an equivalent one-file fix that forwards positive usage.cacheRead and usage.cacheWrite into CronRunTelemetry.usage as cache_read_tokens and cache_write_tokens. Add a focused regression test around isolated cron result/run-log emission so the reader's existing cache-field support is paired with producer coverage. No docs or changelog entry looks necessary unless maintainers want a user-facing cost-accounting note.
