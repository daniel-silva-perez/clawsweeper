## Summary

Keep open. Current main still returns `0` and negative integers from `normalizeTelegramReplyToMessageId`, and the latest release has the same behavior, so this PR's validation tightening is not implemented. The diff is narrow, consistent with the existing Telegram reply-id normalizer, and the existing Greptile review also found it safe with focused coverage.

## What This Changes

This PR tightens Telegram `replyToMessageId` normalization to drop zero and negative numeric or string values and adds focused unit coverage while leaving thread-id parsing unchanged.

## Best Possible Solution

Land the narrow Telegram plugin validation fix: require positive integers for reply-to message ids in the shared normalizer, preserve the existing `parseTelegramThreadId` semantics for negative group/thread keys, and keep regression tests focused on numeric/string zero, negative, positive, non-finite, and non-numeric inputs.
