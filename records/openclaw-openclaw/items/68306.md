Keep open. Current main still allows zero and negative Telegram reply-to message IDs through the shared normalizer, and the PR is a narrow validation tightening with focused tests. I found no blocking functional or security review finding; it needs normal maintainer review and CI before merge.

## What This Changes

This PR updates Telegram reply-to message ID normalization to reject zero and negative numeric or string values and adds focused outbound-parameter tests while preserving thread ID parsing.

## Best Possible Solution

Land the narrow Telegram plugin validation fix: require reply-to message IDs to be positive integers in the shared normalizer, keep negative group/thread ID parsing unchanged, and keep the regression coverage focused on reply-id normalization rather than broader Telegram send behavior.
