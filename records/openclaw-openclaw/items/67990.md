## Summary

Keep open. Current main and the v2026.4.27 release include a useful partial fix for installed plugin node_modules bloat, but the broader backup design is still missing configurable exclusions, general recursive pruning, live-file retry/skip handling on main, service-aware quiesce, and restore/import/full-mode surfaces. Related #72251 is still an open partial PR for one live-backup race and does not supersede this proposal.

## What This Changes

The issue proposes a broader backup architecture with configurable exclusion rules, service-aware archiving, symlink-aware traversal, tar.gz output, config-only/full modes, and restore-oriented behavior.

## Best Possible Solution

Keep this as the broader backup design tracker and split the work into narrower maintainer-approved tasks: a documented exclusion policy surface, general archive filtering, live-file resilience, an explicit quiesce mode if desired, and restore/import disaster-recovery behavior coordinated with #13616, #54242, #67417, and #72251.

## Reproduction Assessment

Not applicable as a single reproduction because this is a design proposal. The related race reports have concrete repro paths, and source/docs inspection gives high-confidence proof that the requested configurable exclusions, service lifecycle, retry/skip behavior, and restore surface are still absent from current main.

## Solution Assessment

No for implementing this as one automated change; the direction is valid, but the maintainable path is to preserve the shipped backup primitives and split the remaining design into focused, reviewed slices.
