## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks the proposed deictic resolver, chat-shaping helper, and memory-core thread-first guidance, while the PR head still has several prompt-behavior regressions that should be fixed before maintainer review resumes.

## What This Changes

The PR adds deictic-context and chat-shaping prompt helpers for auto-reply follow-ups, updates memory-core recall guidance to prefer sessions_history/thread context, adds targeted tests, and updates a Discord component test import.

## Best Possible Solution

Land a revised thread-aware follow-up implementation that keeps nearby context primary, narrows resolver/chat-shaping triggers to true short follow-ups, treats all thread/reply content as untrusted, and covers Slack/Discord/thread-starter edge cases with focused tests.

## Reproduction Assessment

Yes. The problematic paths are pure predicates or static prompt assembly: examples like a detailed question ending in `?`, a group first-thread reply with only `threadStarterBody`, and Slack thread history with a trailing `[slack message id: ...]` line follow directly from the PR head source.

## Solution Assessment

No. The overall direction is plausible, but the current patch is not the narrowest maintainable solution because it over-applies shaping, misses valid thread-starter context, and can extract transport metadata as the referent; those should be fixed before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Gate punctuation-only shaping on short follow-ups:** `src/auto-reply/reply/chat-shaping.ts:25`
  - body: The punctuation pattern makes any message ending in `?` or `!` apply the chat-shaping note after only the 1200-character cap. Detailed questions that are not short conversational follow-ups can therefore get persona/style guidance, which is broader than the PR's stated scope.
  - confidence: 0.86
- **[P2] Honor thread starters in the group shaping guard:** `src/auto-reply/reply/chat-shaping.ts:66`
  - body: For group/channel sessions, this early return ignores `threadStarterBody`, even though the later nearby-context check includes it. First replies in a thread that only have starter context will skip chat shaping despite having the nearby context the helper is meant to use.
  - confidence: 0.82
- **[P2] Select semantic thread text before metadata:** `src/auto-reply/reply/deictic-context.ts:109`
  - body: `pushCandidate` always keeps the last non-empty line, but Slack thread history appends a `[slack message id: ...]` metadata line after each message. The resolver can present that ID line as the referent instead of the message content, degrading the new deictic behavior.
  - confidence: 0.9
- **[P2] Require nearby context for bare pronoun deictics:** `src/auto-reply/reply/deictic-context.ts:26`
  - body: The first English direct-reference pattern matches ordinary short sentences starting with `it`, `this`, or `that` even when no reply or thread context exists. Messages like `It still fails after reboot.` will get a resolver note with no nearby candidates, biasing the reply toward unnecessary ambiguity handling.
  - confidence: 0.84
- **[P3] Remove the unbounded json matcher:** `src/auto-reply/reply/chat-shaping.ts:13`
  - body: The extra `/json/u` matcher duplicates the word-boundary JSON check but also matches substrings like `jsonl` and `jsonify`. That can suppress chat shaping for messages that are not actually structured JSON-output requests.
  - confidence: 0.8
