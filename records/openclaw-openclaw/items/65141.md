## Summary

Keep open. Current main still exposes both `accounts.default` and a named iMessage account as startup targets, and the gateway starts one iMessage monitor per returned account id; open PRs #65194 and #65383 already target this exact duplicate-watcher issue, so maintainers should review/reconcile those rather than close the issue early.

## What This Changes

The issue requests preventing duplicate iMessage replies when `channels.imessage.accounts` contains both a named account and `default` that resolve to the same local `imsg` backend.

## Best Possible Solution

Land one narrow iMessage-plugin fix, or an equivalent owner-maintained change, that suppresses or deduplicates a fallback-only `default` watcher when a named account already covers the same local `cliPath`/`dbPath` backend, while keeping intentionally distinct default backends and adding regression coverage for both cases.

## Reproduction Assessment

Yes. The reporter supplied concrete process and message-history evidence, and current source gives a deterministic static reproduction path: the reported config lists both account ids, gateway starts each, and each iMessage account subscribes to `watch.subscribe`.

## Solution Assessment

Yes, the best direction is a scoped iMessage account-enumeration/startup fix with tests. A broad shared account-helper change would risk changing valid `accounts.default` behavior for other channels.
