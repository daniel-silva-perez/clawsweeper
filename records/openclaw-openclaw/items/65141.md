## Summary

Keep open. Current main still exposes both `accounts.default` and a named iMessage account as startup targets, and each returned iMessage account can start its own `watch.subscribe` monitor. The linked fix paths #65194 and #65383 remain open and unmerged, so the right maintainer action is to review or reconcile those PRs rather than close this issue now.

## What This Changes

Prevent duplicate iMessage replies when a fallback-only `channels.imessage.accounts.default` entry coexists with a named account for the same local Messages backend.

## Best Possible Solution

Land a narrow iMessage-plugin fix that treats fallback-only `accounts.default` as non-startable when a named account already covers the same local backend, preserves distinct default backends, and adds regression coverage for the reported default-plus-named config shape.
