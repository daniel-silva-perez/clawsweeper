## Summary

Keep open. Current main still exposes both `accounts.default` and a named iMessage account as startup targets, and each enabled/configured target can start its own `imsg` watcher. The provided context also shows open fix paths #65194 and #65383, so this issue should remain open until maintainers choose, land, or close those PRs.

## What This Changes

Avoid duplicate iMessage watcher and reply handling when `channels.imessage.accounts` contains both `default` and a named account that resolve to the same local Messages backend.

## Best Possible Solution

Land a narrow iMessage-plugin fix that treats a fallback-only `accounts.default` as non-startable when a named account already covers the same local backend, while preserving distinct default backends and adding regression coverage for the reported config shape.
