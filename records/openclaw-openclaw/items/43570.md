## Summary

Keep open. Current main still has the Discord thread-reply attachment alias gap: the Discord handler maps only `media`, while the send action accepts `media`, `path`, and `filePath`; the linked open PR with closing syntax remains the implementation candidate.

## What This Changes

The issue requests that Discord `thread-reply` accept `filePath` and `path` attachment parameters with the same precedence used by `send`.

## Best Possible Solution

Land #43574 or an equivalent Discord-owned patch that maps `thread-reply` attachments with `media` precedence followed by `path` and `filePath`, with regression tests for both aliases and precedence.

## Reproduction Assessment

Yes. A high-confidence source trace reproduces the bug: `runMessageAction` sends non-`send` actions through the plugin path, shared hydration does not normalize `thread-reply`, and the Discord handler drops `filePath`/`path` by reading only `media`.

## Solution Assessment

Yes. The proposed Discord handler alias chain is the narrowest maintainable fix because runtime attachment delivery already works once `mediaUrl` is populated; tests should cover `filePath`, `path`, and `media` precedence.
