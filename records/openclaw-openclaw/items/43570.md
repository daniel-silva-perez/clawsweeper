## Summary

Keep open. Current main still has the Discord thread-reply attachment alias gap, and the linked open PR #43574 remains the focused implementation path, so this issue should not be closed before that PR lands or is explicitly replaced.

## What This Changes

Discord thread-reply should honor `media`, `path`, and `filePath` for single attachments instead of returning success without uploading when `path` or `filePath` is used.

## Best Possible Solution

Land #43574 or an equivalent Discord-owned patch that maps thread-reply attachments with `media` first, then `path`, then `filePath`, and add focused regression coverage for alias handling and precedence.

## Reproduction Assessment

Yes. A high-confidence source trace reproduces the bug: `thread-reply` bypasses core send handling, shared hydration does not normalize its aliases, and the Discord handler only reads `media`.

## Solution Assessment

Yes. The narrowest maintainable fix is in the Discord thread-reply handler because the downstream runtime already sends the attachment when `mediaUrl` is populated.
