## Summary

Keep open. Current main still has the Discord `thread-reply` attachment alias gap: the handler reads only `media`, while Discord `send` and the user-facing Discord docs support `media`, `path`, and `filePath`; the focused open PR #43574 still appears to be the right review/landing path rather than closing this issue early.

## What This Changes

The issue requests that Discord `thread-reply` attachment handling accept `media`, `path`, and `filePath` with the same fallback behavior as Discord `send`.

## Best Possible Solution

Land #43574 or an equivalent Discord-owned patch that maps `thread-reply` attachments as `media` first, then `path`, then `filePath`, with focused regression coverage for alias handling and precedence.

## Reproduction Assessment

Yes. A high-confidence source trace reproduces the issue: `thread-reply` reaches the Discord plugin handler, shared hydration does not rewrite its aliases, and the handler only reads `media`.

## Solution Assessment

Yes. The narrowest maintainable fix is in the Discord `thread-reply` handler because the runtime already sends an attachment when `mediaUrl` is populated.
