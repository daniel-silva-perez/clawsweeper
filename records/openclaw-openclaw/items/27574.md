## Summary

Current main still does not implement a live browser preview panel in the Control UI/Dashboard. The bundled browser plugin exposes backend/API/CLI primitives for tabs, screenshots, console/errors, and actions, but there is no Control UI browser route, render branch, live pixel stream, URL/title display, console-error panel, or handoff controls. The related #63926 remains open and narrower around visible browser management/login handoff, so this issue should stay open as the broader remote browser preview tracker.

## Best Possible Solution

Keep this issue open as the broad remote browser-preview/dashboard tracker. The best path is a maintainer-reviewed design for an admin-only Control UI browser surface that reuses browser.request and browser-control primitives where safe, or an explicit Control UI panel contribution API followed by a browser plugin implementation. Coordinate with #63926, but do not treat that narrower issue as a complete replacement.

## Risks / Open Questions

- A live browser preview would expose sensitive page pixels, URLs/titles, console errors, and potentially authenticated session context in an admin UI, so it needs explicit product/security design rather than an automatic narrow fix.
- The current browser.request primitive is admin-scoped and request/response based; building a live 1-2 FPS stream or interactive handoff panel may require new UI state, polling/streaming/backpressure, media retention, and permission boundaries.
- The generic plugin Control UI descriptor seam exists, but current UI does not consume it as a full panel renderer, so a plugin-only implementation may need a new or expanded Control UI contribution API.
- The open #63926 is related but narrower around visible browser management/login handoff; closing this broader live-preview tracker now would lose the remote debugging/real-time preview requirements.
