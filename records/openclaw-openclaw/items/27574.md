## Summary

Current main still does not implement the requested live browser preview panel in the Control UI/Dashboard. The browser plugin exposes backend/API/CLI primitives for tabs, screenshots, console/errors, and actions, but there is no Control UI browser route, render branch, live pixel stream, URL/title display, console-error panel, or handoff controls. The related #63926 remains open and narrower around visible browser management/login handoff, so this issue should stay open as the broader remote browser preview tracker.

## Best Possible Solution

Keep this issue open as the broad remote browser-preview/dashboard tracker. The best path is a maintainer-reviewed design for an admin-only Control UI browser surface that reuses browser.request and browser-control primitives where safe, or an explicit Control UI panel contribution API followed by a browser plugin implementation. Coordinate with #63926, but do not treat that narrower issue as a complete replacement.

## Risks / Open Questions

- A live browser preview can expose page pixels, URLs/titles, console errors, cookies/session context, private intranet content, and possibly interactive browser control, so it needs explicit operator-admin authorization, origin checks, session/tab scoping, and rate limiting.
- The current plugin Control UI descriptor seam appears limited to session/tool/run/settings surfaces and is not consumed by the UI, so a plugin-owned browser panel may require a new or expanded Control UI contribution contract before this can live outside core cleanly.
- Related #63926 overlaps on visible browser management and login handoff. Implementation should coordinate with it instead of creating two divergent browser surfaces.
