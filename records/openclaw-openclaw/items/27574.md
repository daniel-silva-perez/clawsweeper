## Summary

Current main still does not implement the requested Control UI/Dashboard browser preview panel. The browser plugin exposes backend/API/CLI primitives, and sandboxed browser mode has a noVNC observer URL, but neither provides a dashboard panel with live browser pixels, URL/title, console errors, or handoff controls. Related #63926 remains narrower browser-management/login-handoff work, not a complete duplicate.

## Best Possible Solution

Keep this issue open as the broad remote browser-preview/dashboard tracker. The best path is a maintainer-reviewed design for an admin-only Control UI browser surface that reuses browser.request and browser-control primitives where safe, or an explicit Control UI panel contribution API followed by a browser plugin implementation. Coordinate with #63926, but do not treat that narrower issue as a full replacement.

## Risks / Open Questions

- A live browser preview can expose page pixels, URLs/titles, console/page errors, cookies/session context, and possibly interactive browser control, so it needs explicit operator-admin authorization, origin checks, rate limiting, and session/tab scoping.
- The current plugin Control UI descriptor API does not appear to support a full plugin-owned dashboard route/panel, so a plugin-based implementation may require a new or expanded UI contribution contract.
- Related #63926 overlaps on browser visibility/login handoff, so implementation should coordinate rather than create two divergent browser surfaces.
