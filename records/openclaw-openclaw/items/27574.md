## Summary

Current main still does not implement the requested live browser preview panel in the Control UI/Dashboard. The browser plugin has backend primitives for tabs, screenshots, console/errors, navigation, and browser.request, but there is no Control UI browser route, tab, render branch, live image stream, or panel UI. The related #63926 remains open and narrower around visible browser management/login handoff, so this issue should stay open as the broader remote browser preview tracker.

## Best Possible Solution

Keep this issue open as the broad remote browser-preview/dashboard tracker. The best path is maintainer-reviewed design for an admin-only Control UI browser surface that reuses browser.request and browser-control primitives where safe, or an explicit Control UI panel contribution API followed by a browser plugin implementation. Coordinate with #63926, but do not treat that narrower login-handoff/browser-management issue as a complete replacement.

## Risks / Open Questions

- A live remote browser panel can expose logged-in browser state, URLs, page pixels, console errors, and screenshots; it needs explicit admin-only scope, device-auth behavior, retention/caching rules, and remote-access review.
- The current browser primitives are enough for screenshots and scripted debugging, but users on remote VPS/container installs still lack real-time visibility until a product/UI design lands.
- Treating this as a simple external plugin may be premature because the existing Control UI descriptor surface is metadata-oriented and may not be enough for a first-class live panel without a new explicit UI contribution seam.
