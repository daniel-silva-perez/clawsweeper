## Summary

Keep open: current main and the latest shipped release still have the reported resolve-before-init path, and #73904 is an open closing PR for this exact sessions_spawn failure. Per policy, this issue should remain open until that PR or an equivalent fix lands; #75129 can cover the adjacent cli-compaction site separately.

## What This Changes

The issue requests initializing built-in context engines before native subagent spawn preparation resolves the configured context engine on cliBackend-only cold starts.

## Best Possible Solution

Merge #73904 or an equivalent narrow fix that calls the built-in context-engine initializer at the native subagent spawn preparation boundary before resolving the engine, with the regression test for cold registry ordering; handle the cli-compaction variant in #75129 as a separate follow-up.

## Reproduction Assessment

Yes. A cold cliBackend-only process can reach sessions_spawn through spawnSubagentDirect without prior PI-runner initialization; current main then resolves the default legacy context engine before registration and reaches the explicit unregistered-engine throw path.

## Solution Assessment

Yes. Initializing with ensureContextEnginesInitialized at the spawn-side resolution boundary is the narrow maintainable fix for this issue; making legacy registration a top-level import side effect would be broader and riskier.
