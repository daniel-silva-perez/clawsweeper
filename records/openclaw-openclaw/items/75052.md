## Summary

Keep open. Current main still has the reported data-loss path: state/config-only uninstall removes the whole state directory recursively, while default and configured workspaces can live under that same `.openclaw` tree. The issue also has two open linked closing PRs, so it should remain open until maintainers choose, consolidate, and land one canonical fix.

## What This Changes

The issue asks `openclaw uninstall` to preserve workspaces under `.openclaw` when state/config deletion is selected but workspace deletion is not.

## Best Possible Solution

Make `openclaw uninstall` honor state and workspace deletion scopes independently. When a configured workspace is inside the state directory and workspace deletion is not selected, preserve that workspace through selective cleanup or stop with a clear warning, then add focused regression coverage and update the uninstall docs before landing one canonical PR.

## Reproduction Assessment

Yes. A destructive live uninstall was not run, but the current-main source gives a high-confidence reproduction path: a workspace under `.openclaw` is nested inside the state directory, and state-only uninstall recursively removes that parent directory.

## Solution Assessment

Yes, a narrow uninstall cleanup guard with regression tests is the best maintainable direction. The next step is to review or consolidate the already-open fix PRs rather than create another repair branch.
