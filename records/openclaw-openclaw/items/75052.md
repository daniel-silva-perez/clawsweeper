## Summary

Keep open. Current main still has the reported state-only uninstall data-loss path: state cleanup recursively removes the whole state directory, while default workspaces can live under that same `.openclaw` tree, and the workspace scope is only handled afterward. Two open, unmerged closing PRs already target this issue, so the best cleanup outcome is maintainer review or consolidation of those PRs rather than closing the issue early.

## What This Changes

Uninstall should preserve workspaces located under `.openclaw` when state/config deletion is selected but workspace deletion is not.

## Best Possible Solution

Make `openclaw uninstall` honor state and workspace deletion scopes independently. When a configured workspace is inside the state directory and workspace deletion is not selected, preserve that workspace through selective cleanup or stop with a clear warning, then land focused regression coverage and docs updates through one canonical PR.

## Reproduction Assessment

Yes. A destructive live uninstall was not run, but the current-main source gives a high-confidence reproduction path: a workspace under `.openclaw` is nested inside the state directory, and state-only uninstall recursively removes that parent directory before workspace scope is considered.

## Solution Assessment

Yes. A narrow uninstall cleanup guard with regression coverage is the best maintainable direction; the immediate path should be to review or consolidate the already-open fix PRs rather than create another repair branch.
