## Summary

Keep open. Current main still allows state-only uninstall to recursively remove the parent state directory before workspace deletion scope is considered, and default or configured workspaces can live inside that same state tree. Two open closing PRs already target this bug, so the issue should remain open until maintainers choose and land one canonical fix.

## What This Changes

The issue asks `openclaw uninstall` to preserve workspace files under `.openclaw` when state/config deletion is selected but workspace deletion is not selected.

## Best Possible Solution

Make uninstall honor state and workspace deletion scopes independently. If a workspace to preserve is nested under the state directory, either selectively clean state around it or stop with a clear warning, then add focused regression coverage and update uninstall docs before landing one canonical PR.

## Reproduction Assessment

Yes. A destructive live uninstall was not run, but current-main source gives a high-confidence reproduction path: default workspaces resolve under `.openclaw`, `openclaw uninstall --state` recursively removes the state directory, and the workspace removal branch is skipped when workspace scope is unchecked.

## Solution Assessment

Yes. A narrow cleanup/uninstall guard with regression tests is the best maintainable direction; the immediate path should be review or consolidation of #75061 and #75062 rather than opening another repair branch.
