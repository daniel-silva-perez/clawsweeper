## Summary

Keep open. Current main still has the reported data-loss path: state/config-only uninstall recursively removes the state directory, while default workspaces normally live under that same `.openclaw` tree. The issue now has two open linked fix PRs, so the right cleanup path is to review and land one canonical fix rather than close the issue or dispatch another repair lane.

## What This Changes

The issue reports that uninstalling with state/config deletion selected can delete workspace data nested under `.openclaw` even when workspace deletion is not selected.

## Best Possible Solution

Make `openclaw uninstall` honor state and workspace deletion choices independently: when state deletion would cover a configured workspace and workspace deletion is not selected, preserve that workspace with selective cleanup or stop with a clear warning. Review and land one canonical fix from #75061 or #75062, with docs and regression tests for nested workspaces under the state directory.

## Reproduction Assessment

Yes. A destructive live uninstall was not run in this read-only review, but current-main source gives a high-confidence reproduction path: default workspaces are under `.openclaw`, and state-only uninstall recursively removes that parent directory.

## Solution Assessment

Yes. A narrow uninstall cleanup fix with regression coverage is the best way to solve this; changing default workspace locations or treating it as broad migration work would be less direct, and opening another automated repair is unnecessary while two linked fix PRs are already open.
