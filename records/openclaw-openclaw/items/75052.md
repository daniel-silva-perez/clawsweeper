## Summary

Keep open. Current main still has the reported data-loss path: state/config uninstall recursively removes the state directory, while default workspaces live under that same `.openclaw` tree. Two open PRs already reference this issue as their fix target, so maintainers should review and consolidate those rather than close the issue or dispatch another repair lane.

## What This Changes

The issue asks OpenClaw uninstall to preserve workspaces nested under `.openclaw` when state/config deletion is selected but workspace deletion is not selected.

## Best Possible Solution

Make `openclaw uninstall` honor state/config and workspace deletion choices independently: when state deletion would cover a configured workspace and workspace deletion is not selected, preserve the workspace through selective cleanup or stop with a clear warning, with regression tests and docs for nested workspaces under the state directory.

## Reproduction Assessment

Yes. A destructive live uninstall was not run in this read-only review, but current-main source gives a high-confidence reproduction path: default workspaces are inside `.openclaw`, and state-only uninstall recursively removes that parent directory.

## Solution Assessment

Yes. The best path is a narrow uninstall cleanup fix with regression coverage; while PR #75061 and PR #75062 are open, the issue should remain open until one canonical fix lands.
