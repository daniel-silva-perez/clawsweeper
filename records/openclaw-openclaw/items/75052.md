## Summary

Keep open. Current main still has the reported data-loss path: state-only uninstall recursively removes the state directory, while the default workspace normally lives under that same `.openclaw` tree. The issue is also explicitly paired with two open fix PRs, so it should remain open until maintainers review and land or close the canonical fix.

## What This Changes

The issue asks for uninstall to preserve workspace files when state/config deletion is selected but workspace deletion is not selected, including workspaces nested under `.openclaw`.

## Best Possible Solution

Review #75061 and #75062, choose one canonical implementation, and land a narrow uninstall cleanup fix that preserves configured workspaces nested under the state directory unless the workspace scope is selected. The landed fix should include regression coverage for state-only uninstall plus docs that warn about manual `rm -rf ~/.openclaw` behavior.

## Reproduction Assessment

Yes. A destructive live uninstall was not run in this read-only review, but current source gives a high-confidence reproduction path: the default workspace is under `.openclaw`, and state-only uninstall recursively removes that parent directory.

## Solution Assessment

Yes. A narrow uninstall cleanup change with regression tests is the best way to solve the issue; changing default workspace locations or opening another automated repair branch would be less direct while two linked fix PRs are already open.
