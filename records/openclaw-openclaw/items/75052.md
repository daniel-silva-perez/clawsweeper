## Summary

Keep open. Current main still has the reported data-loss path: state/config-only uninstall recursively removes the state directory, while the default workspace normally lives under that same `.openclaw` tree. Two open PRs already reference this issue as fixes, so maintainers should review and land one canonical fix rather than close the issue or dispatch another repair lane.

## What This Changes

The issue asks OpenClaw uninstall to preserve workspace directories under `.openclaw` when state/config deletion is selected but workspace deletion is not selected.

## Best Possible Solution

Make `openclaw uninstall` honor state and workspace deletion choices independently: when a configured workspace is inside the state directory and workspace deletion is not selected, preserve it with selective cleanup or stop with a clear warning. Land one canonical fix from #75061 or #75062 with regression coverage and uninstall docs for nested workspaces.

## Reproduction Assessment

Yes. A destructive live uninstall was not run in this read-only review, but current-main source gives a high-confidence reproduction path: the default workspace is inside `.openclaw`, and state-only uninstall recursively removes that parent directory.

## Solution Assessment

Yes. A narrow uninstall cleanup fix with regression tests is the best direction; changing default workspace locations or opening another automated repair branch would be less direct while two linked fix PRs are already open.
