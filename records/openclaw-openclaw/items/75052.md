## Summary

Keep open. Current `main` still has the state-only uninstall path that can remove a nested default workspace, and two open implementation PRs already explicitly target this issue, so the issue should remain open until one fix is reviewed and merged.

## What This Changes

The issue reports that `openclaw uninstall` deletes workspaces nested under `.openclaw` when state/config deletion is selected but workspace deletion is not selected.

## Best Possible Solution

Review #75061 and #75062, choose the safer narrow implementation, and land a fix that preserves configured nested workspaces during state-only uninstall or aborts with a clear warning when selective cleanup cannot be made robust. The landed change should update dry-run behavior, docs, and regression tests for the state/workspace scope contract.

## Reproduction Assessment

Yes. A destructive uninstall was not run in this read-only review, but source review gives a high-confidence reproduction path: default workspaces resolve under `.openclaw`, while state-only uninstall recursively removes the parent state directory.

## Solution Assessment

Yes. The best fix is a narrow change in the uninstall cleanup path that makes state and workspace deletion choices independent, backed by containment/realpath regression tests and docs updates; changing workspace defaults would be broader and riskier.
