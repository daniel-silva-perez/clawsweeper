## Summary

Keep open. Current main still has the reported data-loss path: selecting state/config removal without workspace removal calls the shared state cleanup helper, and that helper removes the entire state directory even though the default workspace lives under the same `.openclaw` tree.

## What This Changes

The issue asks OpenClaw uninstall to preserve workspace files when state/config deletion is selected but workspace deletion is not selected.

## Best Possible Solution

Make uninstall honor the independent workspace selection. When state/config deletion is selected without workspace deletion, configured workspace directories inside the state directory should be preserved, or the command should stop with a clear warning before removing state; docs and dry-run output should match that behavior.

## Reproduction Assessment

Yes. A destructive live uninstall was not run in this read-only review, but the current source path is a high-confidence reproduction: default workspace resolution nests the workspace under `.openclaw`, and state-only uninstall removes that parent directory recursively.

## Solution Assessment

Yes, a narrow CLI cleanup fix is the best path. The safest product behavior is to preserve configured nested workspaces during state-only uninstall; a warning-only abort is a fallback if selective deletion cannot be made robust.
