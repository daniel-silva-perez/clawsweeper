## Summary

Keep open. Current main still has the reported data-loss path: state-only uninstall recursively removes the state directory, while the default workspace lives under that same `.openclaw` tree. Two open PRs already reference this issue with closing syntax, so the right maintainer action is to review and consolidate one canonical fix rather than close the issue or dispatch another repair lane.

## What This Changes

The issue requests that uninstall preserve workspaces under `.openclaw` when state/config deletion is selected but workspace deletion is not selected.

## Best Possible Solution

Make `openclaw uninstall` honor state and workspace deletion choices independently: when state deletion would cover a configured workspace and workspace deletion is not selected, preserve that workspace with selective cleanup or stop with a clear warning. Land one canonical fix from #75061 or #75062 with docs and regression tests for nested workspaces under the state directory.

## Reproduction Assessment

Yes. A destructive live uninstall was not run in this read-only review, but current-main source gives a high-confidence reproduction path: default workspaces are under `.openclaw`, and state-only uninstall recursively removes that parent directory.

## Solution Assessment

Yes. A narrow uninstall cleanup fix with regression coverage is the best way to solve this; changing default workspace locations would be broader and opening another automated repair is unnecessary while two linked fix PRs are already open.
