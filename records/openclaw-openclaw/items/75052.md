## Summary

Keep open. Current main still has the reported data-loss path: state/config-only uninstall recursively removes the state directory, and default workspaces live under that same `.openclaw` tree. The issue is also linked by two open closing PRs, so it should stay open until maintainers choose and land one canonical fix.

## What This Changes

The issue requests that uninstall preserve workspace files under `.openclaw` when state/config deletion is selected but workspace deletion is not selected.

## Best Possible Solution

Make `openclaw uninstall` honor state and workspace deletion scopes independently: when a configured workspace is inside the state directory and workspace deletion is not selected, preserve that workspace through selective cleanup or stop with a clear warning. Land one canonical fix from #75061 or #75062 with regression coverage and docs for nested workspaces.

## Reproduction Assessment

Yes. A destructive live uninstall was not run in this read-only review, but current-main source gives a high-confidence reproduction path: default workspaces are nested under `.openclaw`, and state-only uninstall recursively removes that parent directory.

## Solution Assessment

Yes. A narrow uninstall cleanup guard with regression tests is the best way to solve this; changing default workspace locations or dispatching another repair branch would be less direct while #75061 and #75062 are already open.
