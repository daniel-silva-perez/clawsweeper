## Summary

Current main and the shipped v2026.4.27 release address the reported Anthropic assistant-prefill failure at the provider request boundary: Anthropic-family replay policies enable strict turn validation, and outbound replay now strips stale trailing assistant turns before the request reaches the provider. The cron main-session systemEvent path still matches the report, but the LLM-bound Anthropic payload no longer has to end with the stale assistant turn that caused the 400.

## What This Changes

Cron main-session systemEvent wakeups should not send Anthropic a conversation ending in an assistant turn when the active session's latest transcript entry is an assistant response.

## Best Possible Solution

Keep the provider-boundary replay repair as the canonical fix, because it covers cron systemEvents and other system-event/context-engine paths without adding cron-specific transcript mutations or user-visible placeholder turns.

## Reproduction Assessment

Yes. The report gives a deterministic cron main-session path, and current test coverage reproduces the failing provider-boundary shape by sending an Anthropic replay context ending in an assistant turn and asserting it is stripped before the provider call.

## Solution Assessment

Yes. Fixing this at the Anthropic outbound replay boundary is narrower and more general than a cron-only placeholder user turn, because every systemEvent or context-engine short-circuit path that reaches Anthropic gets the same final-turn protection.
