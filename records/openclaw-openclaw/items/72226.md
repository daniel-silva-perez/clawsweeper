## Summary

Current main already covers the reported cron systemEvent/Anthropic failure at the provider boundary: trailing plain assistant replay turns are stripped before Anthropic Messages transport validation, and the fix shipped in v2026.4.27. The existing discussion correctly identified the cron-to-heartbeat path, but the landed fix is broader than a cron-only placeholder user turn and protects other system-event/heartbeat paths too.

## What This Changes

Main-session cron systemEvents should not fail Anthropic provider calls when the target session transcript currently ends with an assistant turn.

## Best Possible Solution

Keep the provider-boundary sanitizer as the canonical fix because it covers the cron systemEvent case and the related heartbeat, bridge, and context-engine paths without adding cron-specific transcript sentinels.

## Reproduction Assessment

Yes. The report gives a concrete cron main-session reproduction, and current-main regression coverage exercises the failing provider-boundary shape by sending an assistant-ending replay array through the Anthropic wrapper and asserting the outbound messages no longer end with assistant.

## Solution Assessment

Yes. The landed provider-boundary strip is narrower and more maintainable than adding cron-only placeholder user turns because it prevents the same Anthropic 400 for all stale assistant-tail replay paths while preserving valid assistant tool-use continuations.
