## Summary

Keep open. Current main still leaves a credible MiniMax Anthropic-compatible usage propagation gap: the MiniMax provider defaults to `anthropic-messages`, but the Anthropic transport only copies a narrow native Anthropic usage shape from `message_start` and `message_delta`; downstream session and Usage page code can only aggregate nonzero usage after that transport snapshot exists.

## What This Changes

The issue reports that MiniMax CN Anthropic-compatible runs persist zero token usage, causing the Control UI Usage page to show message counts but `tokens: 0`.

## Best Possible Solution

Fix the provider/transport-side usage capture so MiniMax Anthropic-compatible streaming responses produce a nonzero assistant usage snapshot before session persistence, with regression coverage for the MiniMax event shape. Keep the Control UI transcript-backed instead of inferring token totals from message counts or provider identity.

## Reproduction Assessment

Yes. The issue gives a concrete release, endpoint, raw nonzero provider usage, and persisted JSONL zero-usage symptom; current source tracing shows how missed transport usage becomes zero persisted usage while message counts still aggregate.

## Solution Assessment

Yes. The narrow maintainable fix is at the Anthropic transport or immediate usage-capture boundary, using the existing normalized usage aliases and adding MiniMax-shaped coverage; a UI-only change would hide the symptom while leaving `/usage`, session JSONL, status, and cost summaries wrong.
