## Summary

Keep this PR open because current main still lacks Telegram `poll_answer` session routing and the PR targets the right Telegram plugin surface, but the current head should not merge as-is due to a prompt-trust regression and a non-idempotent send failure mode. The earlier CODEOWNERS overwrite review appears resolved in the live diff; the current blocking concern is in the new poll event path.

## What This Changes

The PR adds a persistent Telegram poll registry, records non-anonymous poll metadata on send, registers `poll_answer` updates, routes poll vote events into agent sessions, and adds Telegram tests.

## Best Possible Solution

Keep the Telegram-plugin registry design, but revise the branch so poll vote notifications preserve the untrusted-external-content boundary and registry persistence is best-effort after a successful Telegram send. After that, review the focused Telegram tests and changed gate for merge.

## Reproduction Assessment

Yes. Current main can send a public Telegram poll and return a `pollId`, but source inspection shows no `bot.on("poll_answer")` handler or persistent poll registry; the PR-head findings reproduce by reading the exact enqueue and post-send registry-write lines.

## Solution Assessment

No, not yet. The owner boundary and registry-backed routing approach are maintainable, but the implementation must mark vote-derived system events untrusted and avoid failing an already-sent poll when local registry persistence fails.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Mark poll vote events untrusted:** `extensions/telegram/src/bot-handlers.runtime.ts:1098-1101`
  - body: The new handler builds `text` from Telegram user identity plus stored poll content, then enqueues it without `trusted: false`. Since system events default to trusted `System:` prompt framing, vote-derived external text can be promoted into trusted context; pass `trusted: false` and cover that in the routing test.
  - confidence: 0.86
- **[P2] Keep poll sends successful when registry writes fail:** `extensions/telegram/src/action-runtime.ts:459-467`
  - body: This registry write runs after `sendPollTelegram` has already created the Telegram poll. If the local state write fails, the action throws even though the poll was sent, so callers can report failure or retry a non-idempotent send; catch and log registry persistence failures as best-effort before returning success.
  - confidence: 0.82
