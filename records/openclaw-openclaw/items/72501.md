## Summary

Keep open. Current main still lacks Telegram `poll_answer` session routing, so the PR addresses a real gap, and the latest diff has removed the earlier CODEOWNERS overwrite noted by Greptile. It should not merge yet because the new handler promotes Telegram-derived vote text into a trusted system event and the registry write can fail a poll action after Telegram has already sent the non-idempotent poll.

## What This Changes

The PR adds a Telegram poll registry, records public poll metadata on send, registers and handles `poll_answer` updates, routes vote events into session system events, and adds Telegram tests.

## Best Possible Solution

Revise this PR in the Telegram plugin: keep the registry-backed `poll_answer` routing design, but mark vote-derived system events untrusted or otherwise structure/sanitize external text, make registry persistence best-effort after a successful poll send, and keep the auth/session-routing tests focused on public poll votes.

## Reproduction Assessment

Yes. On current main, a public Telegram poll can return a `pollId`, and upstream Telegram can deliver `poll_answer`, but there is no registered handler or registry lookup; for the PR findings, source inspection shows vote text reaches `enqueueSystemEvent` as trusted and registry writes are awaited after the poll send.

## Solution Assessment

No, not yet. The PR chooses the right owner boundary and architecture for routing votes, but the implementation should preserve the existing untrusted-external-content boundary and avoid failing an already-sent poll when best-effort registry persistence fails.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Mark poll vote events untrusted:** `extensions/telegram/src/bot-handlers.runtime.ts:1098-1101`
  - body: The new handler builds `text` from Telegram user data plus poll question/options, then enqueues it without `trusted: false`. Since formatted system events default to the trusted `System:` prefix, a voter-controlled name or poll text can be promoted into trusted prompt context; enqueue these vote notifications as untrusted and cover that in the routing test.
  - confidence: 0.86
- **[P2] Keep poll sends successful when registry writes fail:** `extensions/telegram/src/action-runtime.ts:459-467`
  - body: This write runs after `sendPollTelegram` has already sent the poll. If the local state write fails, the action throws even though Telegram already created the poll, so callers can report failure or retry a non-idempotent send; catch and log registry persistence failures as best-effort before returning success.
  - confidence: 0.82
