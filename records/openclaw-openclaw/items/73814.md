## Summary

Keep open. Current main still documents the reported `curl ... | bash` one-liner, the issue is paired with open closing PR #73830, and the review did not find enough proof that the exact WSL2 stdin-corruption hang is already resolved. The installer has current safeguards that route prompts through `/dev/tty`, but a docs-only `bash -s --` change needs review because `bash -s --` still executes the script from stdin.

## What This Changes

The issue asks OpenClaw to prevent the documented piped installer command from hanging or truncating the installer script when interactive installer steps read from stdin.

## Best Possible Solution

Keep this issue open until PR #73830 is reviewed and either replaced or landed. The durable end state should make the public install path safe for interactive installer behavior, with docs and installer smoke coverage aligned; the separate Node auto-installer failure should remain tracked by #73837.

## Reproduction Assessment

Unclear for the full WSL2 hang because this read-only pass did not run the mutating installer. The report gives a concrete command and environment, current docs still publish that command, and a minimal bash-pipe check confirms the stdin-consumption mechanism is plausible.

## Solution Assessment

Unclear/no for the linked docs-only direction as stated. `bash -s --` is useful for option passing but still leaves the script on stdin, so the safer fix is to ensure all interactive subprocesses read from `/dev/tty` or document an invocation that separates the script stream from prompt input.
