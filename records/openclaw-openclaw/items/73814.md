## Summary

Keep open. Current main still documents the reported piped installer path and still has an installer onboarding path that checks for a TTY but invokes `openclaw onboard` without redirecting stdin to `/dev/tty`; the issue also has open PR #73830 linked with closing syntax, so it should remain open until that PR or a replacement actually lands.

## What This Changes

The issue asks OpenClaw to make `curl -fsSL https://openclaw.ai/install.sh | bash` installs robust against interactive installer/onboarding code consuming the piped script stream and mangling `install.sh`.

## Best Possible Solution

Make the installer itself robust under piped execution by ensuring every interactive prompt and child process launched during install reads from `/dev/tty` or is skipped/noninteractive, then align public install docs and add a focused `install.sh` regression for piped stdin behavior.

## Reproduction Assessment

Unclear. The issue provides a concrete WSL2 command and logs, and current main shows a plausible stdin-inheritance path, but this read-only review did not run the mutating installer end to end.

## Solution Assessment

No for a docs-only command change as the complete solution. The narrower maintainable fix is installer-level TTY isolation for onboarding/prompt subprocesses, with docs updated only to reflect the supported invocation.
