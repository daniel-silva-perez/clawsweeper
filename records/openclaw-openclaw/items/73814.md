## Summary

Keep open. Current main still publishes the reported piped installer command, the issue is paired with open closing PR #73830, and the read-only review did not prove the WSL2 stdin-corruption hang is fixed end to end.

## What This Changes

The issue reports that the documented `curl ... | bash` installer command can hang on WSL2 Ubuntu and corrupt the script stream when interactive installer prompts consume stdin.

## Best Possible Solution

Keep this open until #73830 is reviewed and either landed with a correct fix or replaced. The desired end state is a public install path that cannot let interactive prompts consume the script stream, with docs and installer smoke coverage matching that behavior while #73837 tracks the separate Node auto-installer failure.

## Reproduction Assessment

Unclear. The report gives a concrete command and WSL2 Ubuntu environment, and current docs still publish that command, but this read-only pass did not run the mutating installer; a minimal shell check does confirm the stdin-consumption mechanism is plausible.

## Solution Assessment

Unclear/no for the linked docs-only direction. `bash -s --` is useful for option passing but does not isolate stdin by itself, so the safer solution is to ensure every interactive subprocess reads from `/dev/tty` or to document an invocation that separates script input from prompt input.
