## Summary

Keep open. Current main still publishes the reported `curl ... | bash` install path, the issue is paired with open closing PR #73830, and the read-only review did not prove the exact WSL2 stdin-corruption hang is fixed. The installer now has useful `/dev/tty` safeguards, but the proposed docs-only `bash -s --` direction does not by itself isolate script stdin from interactive reads.

## What This Changes

The issue reports that the documented `curl -fsSL https://openclaw.ai/install.sh | bash` installer path can hang and execute a truncated `warn_shell_path_missing_dir` function name on WSL2 when something consumes the piped script stream.

## Best Possible Solution

Keep this open until #73830 is reviewed and either landed, closed, or replaced. The durable end state should make the public install path safe for interactive installer behavior by redirecting interactive subprocesses to `/dev/tty` or noninteractive subprocesses away from the script stream, with docs and installer smoke coverage aligned; the Node auto-installer failure should stay in #73837.

## Reproduction Assessment

Unclear. The report gives a concrete WSL2 command and the shell-stream corruption mechanism is easy to demonstrate, but this read-only pass did not run the mutating installer against WSL2/current main.

## Solution Assessment

No/unclear. A docs-only `bash -s --` change is not enough by itself because the script still arrives on stdin; the safer solution is to audit and redirect installer child-process stdin, or document an invocation that separates the script stream from prompt input.
