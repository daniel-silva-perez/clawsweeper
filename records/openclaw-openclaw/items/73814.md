## Summary

Keep open: current main still exposes the reported piped install path, the installer still has at least one onboarding handoff that can inherit the script stream, and open PR #73830 is already linked as the active fix candidate.

## What This Changes

The issue reports that the documented `curl ... install.sh | bash` installer path can hang on WSL2 Ubuntu and corrupt the bash script stream when interactive work reads from stdin.

## Best Possible Solution

Make the installer itself robust under piped execution by ensuring every interactive prompt or child process that can run during install reads from `/dev/tty` or is skipped/noninteractive, then align the public install docs and add a focused `install.sh` regression for piped stdin behavior.

## Reproduction Assessment

Unclear. The issue includes a concrete WSL2 command and the stdin-consumption mechanism is reproducible with a minimal `bash -s --` pipe check, but this read-only pass did not run the full mutating installer end to end.

## Solution Assessment

No for a docs-only `bash -s --` change as the complete fix. The narrower maintainable solution is installer-level tty isolation for onboarding/prompt subprocesses, with docs updated only to reflect the supported invocation.
