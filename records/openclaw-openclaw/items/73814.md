## Summary

Keep open. The report is concrete, current main still publishes the piped installer command, and an open unmerged closing PR already targets this issue, so the issue should not be closed before that PR is reviewed or replaced.

## What This Changes

The issue asks OpenClaw to prevent the documented `curl ... | bash` installer path from hanging or corrupting the script stream when interactive installer prompts run under WSL2/Linux.

## Best Possible Solution

Keep this open until PR #73830 is reviewed and either landed with a verified installer/docs fix or closed in favor of an installer-level TTY/stdin hardening change. The desired end state is a public install path and installer prompt/onboard flow that cannot consume the script stream on WSL/Linux, while #73837 remains the separate Node auto-installer failure report.

## Reproduction Assessment

Unclear. The issue gives a concrete WSL2 Ubuntu 24.04 command and error, and current docs still expose that path, but this read-only review did not run the mutating installer end to end.

## Solution Assessment

Unclear/no for the docs-only direction. The safer solution is to verify installer-level `/dev/tty` isolation for every interactive subprocess and then align the public docs, rather than relying on `bash -s --` alone.
