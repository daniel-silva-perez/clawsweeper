## Summary

Keep open. The Feishu `NO_REPLY` leak is still a real gap on current `main`, but this PR is not landable as-is: the submitted patch uses an unsupported root SDK import on current main and still misses media, mixed-payload, and streaming-card paths. The newer open #56167 is useful consolidation context, but it has not merged and does not prove the reply-dispatcher side is fully covered.

## What This Changes

This PR adds Feishu-local silent-reply checks to suppress text-only `NO_REPLY` before `sendText` and before text-only reply-dispatcher delivery.

## Best Possible Solution

Land one Feishu-owned fix that suppresses exact silent tokens at every Feishu API boundary, keeps media delivery intact, preserves streaming closeout behavior, imports helpers from a supported narrow SDK subpath, and adds focused outbound plus reply-dispatcher regression tests.

## Reproduction Assessment

Yes. The code paths are inspectable without live Feishu: direct `feishuOutbound.sendText({ text: "NO_REPLY" })`, `sendMedia({ text: "NO_REPLY", mediaUrl })`, and reply-dispatcher payloads with `NO_REPLY` all reach Feishu send helpers on current main unless a higher-level normalizer happens to intercept them.

## Solution Assessment

No. The direction is valid, but this patch is not the best fix because it handles only text-only paths, uses an unsupported SDK import on current main, and can skip streaming-card closeout; a small Feishu-local boundary guard plus tests is the safer route.

  - confidence: 0.86
- **[P2] Suppress silent text even when media is present:** `extensions/feishu/src/reply-dispatcher.ts:259-261`
  - body: The `!hasMedia` condition means a payload with media plus `text === "NO_REPLY"` skips this guard and still enters the text-send block before media delivery. Treat the silent token as non-deliverable text while still allowing the media path to run.
  - confidence: 0.91

  - confidence: 0.89
