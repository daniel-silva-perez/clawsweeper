## Summary

Keep open. Current main still routes default scheduled heartbeats through the main session and only filters no-op HEARTBEAT_OK pairs from replay; meaningful sent/tool heartbeat turns remain append-only and can still be visible to later heartbeat prompts. The discussion’s local sent-path prune workaround is not present on current main, so the reported effective 2x interval is not resolved.

## What This Changes

The issue requests that scheduled heartbeat cycles independently evaluate each configured interval instead of letting a prior successful heartbeat make the next scheduled run short-circuit as HEARTBEAT_OK.

## Best Possible Solution

Keep this as a core heartbeat bug and settle the context contract explicitly: scheduled heartbeats should either default to an isolated/fresh run context or replay should exclude prior heartbeat-generated sent/tool turns while preserving HEARTBEAT.md/workspace injection, delivery routing, dedupe state, and append-only transcript auditability. Add regression coverage proving a meaningful sent/tool heartbeat cannot make the next scheduled heartbeat falsely short-circuit to HEARTBEAT_OK.

## Reproduction Assessment

Yes for the source-level stale-context condition: seed a main-session transcript with a meaningful heartbeat/tool turn, run the next default heartbeat replay path, and verify that current filtering leaves that prior heartbeat activity available to the prompt. The exact sent -> ok-token alternation needs a live model or mocked decision layer, but the reporter's logs and local patch report make the bug path credible.

## Solution Assessment

No current-main solution is in place. The present ack-pair filter is too narrow for the reported case because it preserves meaningful heartbeat/tool turns; a safer fix needs a maintainer choice between heartbeat isolation and targeted heartbeat-turn replay filtering.
