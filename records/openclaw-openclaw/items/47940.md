## Summary

Keep open. Current main still runs scheduled heartbeats in the main session by default, keeps heartbeat transcripts append-only, and filters only no-op HEARTBEAT_OK prompt/ack pairs before replay; meaningful sent/tool heartbeat turns can still remain visible to the next scheduled tick. The append-only cleanup fixed unsafe transcript truncation, but it does not prove the reported effective 2x exploration interval is resolved.

## What This Changes

The issue reports that scheduled heartbeat runs alternate between real sent work and silent HEARTBEAT_OK responses, making the effective exploration cadence twice the configured heartbeat interval.

## Best Possible Solution

Keep this as a core heartbeat replay-contract bug. Scheduled heartbeat prompts should not treat previous heartbeat-generated sent/tool work as fresh main-session context, while preserving append-only transcript audit, HEARTBEAT.md/workspace injection, delivery routing, dedupe state, and system-event handling; the likely direction is targeted heartbeat-turn replay filtering or explicit default isolation, not restoring fs.truncate-based pruning.

## Reproduction Assessment

Yes for the source-level path: default scheduled heartbeat uses the main session, current replay filtering removes only no-op HEARTBEAT_OK pairs, and meaningful sent/tool heartbeat turns are preserved. The exact alternating cadence still needs a live model or mocked decision-layer regression, but the reporter's logs and local patch notes make the behavior credible.

## Solution Assessment

No current-main solution is in place. The append-only/no-op-pair filter is a good safety improvement over transcript truncation, but it does not stop meaningful sent/tool heartbeat context from influencing the next tick; a targeted replay contract is the safer alternative.
