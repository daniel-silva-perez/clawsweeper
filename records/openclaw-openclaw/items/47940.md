## Summary

Keep open. Current main still defaults scheduled heartbeat runs to the main session, keeps heartbeat transcripts append-only, and filters only no-op HEARTBEAT_OK prompt/ack pairs before replay, so meaningful sent/tool heartbeat turns can still influence the next scheduled tick. The merged append-only cleanup fixed transcript truncation races, but it is too narrow to prove the reported 2x effective interval is resolved.

## What This Changes

The issue reports that scheduled heartbeat runs alternate between real sent work and silent HEARTBEAT_OK responses, making the effective exploration interval twice the configured heartbeat interval.

## Best Possible Solution

Keep this as a core heartbeat bug and make the replay contract explicit: scheduled heartbeat prompts should not treat previous heartbeat-generated sent/tool work as fresh main-session context, while preserving append-only transcript audit, HEARTBEAT.md/workspace injection, delivery routing, dedupe state, and system-event handling. The safest direction is likely context-replay filtering or explicit default isolation, not restoring fs.truncate-based pruning.

## Reproduction Assessment

Yes for the source-level reproduction path: default scheduled heartbeat uses the main session, current replay filtering only removes no-op HEARTBEAT_OK pairs, and meaningful sent/tool heartbeat turns are preserved. The exact alternating cadence still needs a live model or mocked decision-layer regression, but the reporter's logs and local patch notes make the observed behavior credible.

## Solution Assessment

No current-main solution is in place. The current append-only/no-op-pair filter is a good safety improvement over transcript truncation, but it does not solve meaningful sent/tool heartbeat context influencing the next tick; a targeted replay contract is the safer alternative.
