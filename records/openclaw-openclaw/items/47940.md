## Summary

Keep open. Current main still runs non-isolated heartbeats in the main session by default, only filters prior HEARTBEAT_OK no-op pairs from replay, and explicitly keeps meaningful heartbeat/tool turns append-only. That leaves the reported sent -> ok-token alternating context-pollution class unresolved for default `heartbeat.every` users.

## Best Possible Solution

Keep this as an OpenClaw core heartbeat bug. Maintainers should choose a durable policy: either make scheduled heartbeats isolated by default, or keep append-only transcripts while filtering prior heartbeat-generated sent/tool turns out of the next heartbeat replay. The fix should preserve HEARTBEAT.md/workspace injection, delivery routing, last-heartbeat dedupe state, and add regression coverage proving a meaningful sent/tool heartbeat cannot cause the next scheduled tick to short-circuit to HEARTBEAT_OK.
