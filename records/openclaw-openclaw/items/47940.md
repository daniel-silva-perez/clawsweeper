## Summary

Keep open. Current main still defaults scheduled heartbeats to the main session, keeps heartbeat transcripts append-only, and only filters no-op HEARTBEAT_OK pairs before replay; meaningful sent/tool heartbeat turns can still influence the next scheduled heartbeat, so the reported 2x effective interval is not proven fixed.

## What This Changes

The issue requests that scheduled heartbeats evaluate each configured interval independently instead of alternating between meaningful sent cycles and silent HEARTBEAT_OK cycles.

## Best Possible Solution

Keep this as a core heartbeat bug and settle the replay contract explicitly: scheduled heartbeat prompts should not treat prior heartbeat-generated sent/tool work as fresh user-session context, while preserving HEARTBEAT.md/workspace injection, delivery routing, dedupe state, system-event handling, and append-only transcript auditability. Add regression coverage proving a meaningful sent/tool heartbeat cannot cause the next scheduled tick to falsely short-circuit to HEARTBEAT_OK.

## Reproduction Assessment

Yes for the source-level stale-context path: seed the main-session transcript with a meaningful heartbeat/tool turn and run the next default heartbeat replay path; current filtering leaves that prior activity available to the prompt. The exact alternating cadence needs a live model or mocked decision layer, but the reporter's logs and local patch report make the observed behavior credible.

## Solution Assessment

No current-main solution is in place. The present ack-pair filter is too narrow for this report because it preserves meaningful heartbeat/tool turns; the safer fix needs a maintainer choice between heartbeat isolation and targeted heartbeat-turn replay filtering.
