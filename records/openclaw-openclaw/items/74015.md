## Summary

Keep open. Current main still implements `llm_input` and `llm_output` as observation-only void hooks, and the runner/harness call sites do not await or consume mutation results. Existing prompt hooks can add context or replace system prompt text, but they still do not expose a writable user prompt, history, or assistant-output contract; the related draft #74483 also explicitly says it does not implement this request.

## What This Changes

Add writable Plugin SDK support so trusted plugins can modify LLM input and output payloads, including prompt, system prompt, history messages, and assistant text, instead of only observing them.

## Best Possible Solution

Keep this open for maintainer Plugin SDK design: add an additive, versioned mutating pre-provider and post-output contract, or a deliberately designed extension of the existing LLM hooks, with explicit conversation/prompt-injection permissions, deterministic handler ordering, SDK type/API docs, contract tests, and coverage across embedded runner, Codex/CLI harness helpers, and public docs.

## Reproduction Assessment

Not applicable as a feature request, but the current-main verification path is high confidence: inspect the SDK hook types, hook runner, embedded runner, harness helpers, and docs. They consistently show observe-only LLM hooks with no writable result contract.

## Solution Assessment

Unclear for implementation because there is no patch yet. The requested direction is valid, but the maintainable solution should be a designed and permissioned SDK contract rather than silently making the current fire-and-forget observation hooks mutable.
