## Summary

Keep open. Current main still treats `llm_input` and `llm_output` as observation-only Plugin SDK hooks, and the runner/harness call sites do not await or consume prompt, history, system-prompt, or assistant-output mutations.

## What This Changes

Request to let existing `llm_input` and `llm_output` Plugin SDK hooks optionally return writable prompt/history/system-prompt and assistant-output mutations for privacy redaction and restoration workflows.

## Best Possible Solution

Design an additive mutable pre-provider and post-output Plugin SDK contract, with explicit raw-conversation and prompt-mutation permissions, deterministic sequential merge semantics, embedded runner support, Codex/CLI harness support, SDK baseline updates, docs, and focused hook contract tests.

## Reproduction Assessment

Not applicable as a feature request, but the current-main verification path is high-confidence: inspect the hook types, runner methods, embedded runner, harness helpers, and docs listed in the evidence. They consistently show observation-only LLM hooks with no writable result contract.

## Solution Assessment

Unclear whether returning mutation results from the existing hooks is the best final API. The maintainable path is a maintainer-designed additive or versioned contract that preserves existing observation semantics and defines permissions and merge behavior before implementation.
