## Summary

Keep open. Current main still exposes `llm_input` and `llm_output` as observation-only hooks and the PI, CLI, and Codex harness call sites do not await or consume mutation results. The related draft #74483 is useful SDK context, but it explicitly says it does not implement writable prompt mutation, so this remains a real public SDK/security design item.

## What This Changes

Add a public Plugin SDK contract that lets LLM input/output hooks return prompt, system prompt, history, or assistant-output mutations for privacy transforms before and after model calls.

## Best Possible Solution

Design an additive, permissioned Plugin SDK surface for mutating pre-provider input and post-output content, with deterministic handler ordering, failure behavior, docs, API baseline updates, and contract coverage across PI, CLI, and Codex harnesses.

## Reproduction Assessment

Not applicable as a feature request, but the current-main verification path is high confidence: inspect the SDK types, hook runner, PI runner, shared harness helpers, Codex harness, tests, and docs. They consistently show observation-only LLM hooks with no writable result contract.

## Solution Assessment

Unclear for implementation because there is no patch yet. The requested direction is valid, but the maintainable solution should be a designed public SDK/security contract rather than silently making the existing fire-and-forget observation hooks mutable.
