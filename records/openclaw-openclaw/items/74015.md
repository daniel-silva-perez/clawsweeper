## Summary

Keep this issue open. Current main still exposes `llm_input` and `llm_output` as observation-only Plugin SDK hooks: the public handler map returns `Promise<void> | void`, the runner delegates both through the parallel void hook path, and embedded plus harness call sites do not await or consume prompt/history/output mutations. Existing prompt hooks can add context or system-prompt text, but they still cannot replace the original user prompt, history messages, or assistant output for the privacy/desensitization use case.

## Best Possible Solution

Keep open for maintainer Plugin SDK design. The best path is an additive, versioned mutable pre-provider and post-output contract, or a deliberately designed result extension to the existing hooks, with explicit conversation/prompt-mutation permissions, deterministic merge behavior, and coverage across the embedded runner, Codex/app-server harness helpers, CLI harness, SDK exports/baselines, docs, and hook contract tests.
