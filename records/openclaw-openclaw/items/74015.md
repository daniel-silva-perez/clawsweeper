## Summary

Keep open. Current main still exposes `llm_input` and `llm_output` as observation-only hooks and every inspected runner path fires them without awaiting or consuming mutation results. The request is a real Plugin SDK gap, but it changes a public raw-conversation security boundary, so it needs maintainer API/security design rather than conservative cleanup closure.

## What This Changes

The issue requests writable Plugin SDK LLM input/output hooks so plugins can modify user prompts, system prompts, history, and assistant text around model calls.

## Best Possible Solution

Add an additive, versioned mutating pre-provider and post-output Plugin SDK contract, or a deliberately designed extension of the existing LLM hooks, with explicit raw-content permissions, deterministic ordering, docs, SDK API checks, and contract tests across Pi embedded, CLI harness, and Codex app-server paths.

## Reproduction Assessment

Not applicable as a feature request, but the verification path is high confidence: inspect the current SDK hook types, hook runner, Pi embedded runner, harness helpers, Codex app-server call sites, and docs. They consistently show observe-only LLM hooks with no writable prompt/history/output result contract.

## Solution Assessment

Unclear because there is no proposed patch. The requested direction is valid, but the maintainable solution should be a designed and permissioned SDK contract rather than silently making today’s fire-and-forget observation hooks mutable.
