## Summary

Keep open. Current main has useful redaction and sanitization surfaces, but they are narrower than the requested automatic output sanitization for agent responses/tool/file-read output. The strongest related implementation candidate is open PR #45783, which explicitly lists this issue as related, but it is unmerged and needs maintainer security/product review before this issue can close.

## Best Possible Solution

Keep this issue open as the product/security tracker. Review #45783 as the current implementation candidate; if accepted, land or split it into a smaller design that covers final replies, streamed block replies, tool/file-read output, config schema, docs, and regression tests. If maintainers decide this belongs in plugins instead, document the supported `message_sending` and tool-result hook path and close only after any missing extension API is identified or added.

## Risks / Open Questions

- Treating current `logging.redactSensitive` as the requested feature would overstate protection for actual channel-delivered replies and streamed block replies.
- Open PR #45783 is large and security-sensitive; even if it is the right direction, it needs review for outbound/inbound LLM handling, mapping storage, false positives, ReDoS safety, and whether it covers channel output delivery.
- The existing plugin hooks make an external sanitizer possible, but no bundled/default sanitizer was found on current main.
