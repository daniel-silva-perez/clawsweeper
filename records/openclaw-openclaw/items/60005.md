## Summary

Keep open. Current main still exposes MCP tools only, while this PR adds useful resource support but is not safe to merge as-is because it reintroduces direct plugin-tool execution and exposes config through ad hoc redaction.

## What This Changes

Adds MCP resources for channel version/status/conversations and plugin tool/config inspection, wires resource list/read handlers into the plugin-tools server, and expands plugin-tools tests.

## Best Possible Solution

Rebase the resource feature onto the current MCP helper structure, add resource support without bypassing createPluginToolsMcpHandlers, serialize config resources through shared redaction, and add regression tests for arbitrary secret-like MCP server keys plus approval/owner-only enforcement.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Use shared redaction for MCP config resources:** `src/mcp/plugin-resources.ts:105-143`
  - body: `openclaw://config/mcp` clones each server config and masks only `headers`, `env`, and selected URL pieces before returning the rest of the object. MCP server entries allow arbitrary keys, so fields such as `apiKey`, `authToken`, or `clientSecret` would be exposed to any MCP client reading this resource; route this payload through the existing config redaction helper instead.
  - confidence: 0.92
- **[P2] Preserve the current plugin-tool handler seam:** `src/mcp/plugin-tools-serve.ts:77-86`
  - body: The branch's plugin-tools server still builds its own tool map and calls `tool.execute` directly. Current main routes this server through `createPluginToolsMcpHandlers`, which filters `ownerOnly` tools and enforces `before_tool_call` approvals, so the resource rebase needs to keep that helper instead of restoring the older direct execution path.
  - confidence: 0.86
