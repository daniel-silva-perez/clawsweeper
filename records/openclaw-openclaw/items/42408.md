## Summary

Keep open. Current main still has only include-side `memorySearch.extraPaths` support; it does not provide `excludePaths`/glob exclusions for builtin memory indexing, and the user-facing docs still do not cover benchmark/queryset contamination or multi-workspace path hygiene beyond basic `extraPaths` behavior.

## Best Possible Solution

Keep this issue open. Add a documented `memorySearch` exclusion mechanism for builtin memory indexing and shared config, then wire it through schema/types/help/docs, `resolveMemorySearchConfig`, file discovery/read access, watcher setup, reindex scope hashing, and tests proving benchmark/queryset Markdown under `extraPaths` can be excluded. Also expand public memory docs with absolute vs workspace-relative `extraPaths`, `memory status --deep`, reindexing after path changes, broad-path noise, and benchmark/evaluation contamination guidance.

## Risks / Open Questions

- Closing now would drop a concrete bundled memory-core config/docs gap: broad `extraPaths` can still index benchmark or evaluation Markdown and affect local/hybrid retrieval quality.
- QMD collection `pattern` support is not a complete substitute because the report targets shared `memorySearch.extraPaths` and the builtin local/hybrid memory path.
- Any eventual fix needs schema/types/help/docs, discovery/read/watch filtering, reindex scope hashing, and regression tests so exclusions affect both indexing and stale index invalidation.
