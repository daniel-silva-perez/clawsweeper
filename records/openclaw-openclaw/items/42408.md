## Summary

Keep open. Current main still lacks the requested memory indexing exclusion mechanism and docs for benchmark/queryset contamination. Existing diagnostics do report missing extra paths, but the core request remains: `memorySearch.extraPaths` is include-only, memory-core still recursively indexes eligible files under those paths, and user-facing docs do not describe excluding benchmark/evaluation files.

## Best Possible Solution

Keep open. Add a documented `memorySearch` exclusion mechanism for the bundled memory-core path and shared config: schema/types/help/docs, `resolveMemorySearchConfig`, file discovery/read access, watcher setup, reindex scope hashing, and regression tests proving benchmark/queryset Markdown under `extraPaths` can be excluded. Update memory docs with absolute vs workspace-relative `extraPaths`, `memory status --deep`, reindexing after path changes, broad-path noise, and benchmark/evaluation contamination guidance.

## Risks / Open Questions

- Closing now would drop a concrete bundled memory-core config/docs gap that can still affect local/hybrid retrieval when broad `extraPaths` include benchmark or evaluation Markdown.
- QMD `extraCollections.pattern` is not a complete substitute for this request because the report targets local/hybrid builtin memory search and shared `memorySearch.extraPaths`.
- The implementation should avoid treating the existing missing-path status warning as sufficient; the remaining work is exclusion support plus explicit path hygiene and contamination guidance.
