## Summary

Keep open. Current main still resolves an unset memory sync watcher to true and the default memory_search manager path can still create long-lived builtin or QMD chokidar watchers; the related watcher-freshness issue and ENOSPC PR are adjacent, not the requested gateway default policy change.

## What This Changes

Default `memorySearch.sync.watch` to false for headless gateway mode while preserving explicit `sync.watch` overrides and CLI/desktop defaults.

## Best Possible Solution

Add a runtime-aware memory sync default that resolves unset `sync.watch` to false only for the intended headless gateway manager context, while explicit true/false config remains authoritative and CLI/desktop behavior stays covered by tests and docs.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to leave `sync.watch` unset, resolve memory sync config for an enabled agent, then follow a gateway memory_search call into the default manager path where builtin and QMD managers can start chokidar watchers because the resolved value is true.

## Solution Assessment

Unclear as originally phrased. Keying only on raw `gateway.mode` may be too broad; the safer fix is to thread an explicit headless/default-policy context into sync resolution or manager creation so local interactive freshness is not changed by accident.
