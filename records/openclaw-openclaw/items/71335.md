## Summary

Keep this issue open. Current main still resolves an unset memory sync watcher to true and the normal gateway memory_search path can still create long-lived builtin or QMD chokidar watchers; the related watcher-freshness and ENOSPC work is adjacent, not the requested gateway default policy change.

## What This Changes

Request to make `memorySearch.sync.watch` default to false for headless gateway deployments while preserving explicit overrides and the existing CLI/desktop freshness default.

## Best Possible Solution

Add a runtime-aware memory sync default that resolves unset `sync.watch` to false only in the intended headless gateway manager context, while explicit true/false config remains authoritative and CLI/desktop behavior stays covered by tests and help text.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to leave `sync.watch` unset, resolve memory sync config for an enabled agent, then follow a gateway memory_search call into the default manager path where builtin and QMD managers start chokidar watchers because the resolved value is true.

## Solution Assessment

Unclear as originally phrased. Keying only on raw `gateway.mode` may be too broad; the safer implementation is to thread an explicit headless/default-policy context into memory sync resolution or manager creation so local interactive gateway use keeps near-real-time freshness.
