## Summary

Keep this issue open. Current main still resolves unset memory sync watching to true without any gateway/headless input, and the normal memory search tool path can create default builtin or QMD managers that start chokidar watchers when that resolved value is true. The closed related #64549 is an adjacent auto-indexing bug, not a replacement for this default-policy request.

## What This Changes

The issue requests that unset `memorySearch.sync.watch` default to false for headless gateway mode while preserving explicit user overrides and the existing desktop/CLI default behavior.

## Best Possible Solution

Add a runtime-aware memory sync default so unset `sync.watch` resolves to false only for the intended headless gateway-created manager context, while explicit `true` and `false` overrides remain authoritative and CLI/status/desktop freshness behavior stays covered by tests and docs.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: leave `sync.watch` unset, resolve memory sync config for an enabled agent, and follow the default memory search manager path to builtin or QMD watcher creation; a live FD-count test would only quantify the same behavior.

## Solution Assessment

Unclear as originally phrased if it keys only on `gateway.mode`. The safer fix is a backwards-compatible runtime/headless context threaded into memory sync resolution or manager creation, not a broad config-only default change that could affect local desktop gateway flows.

Current `resolveMemorySearchSyncConfig` defaults unset `sync.watch` to true everywhere, and default gateway memory search managers can create chokidar watchers for builtin and QMD memory. Implement a backwards-compatible runtime/headless context so unset `sync.watch` defaults to false only for the intended headless gateway-created manager context. Preserve explicit `sync.watch: true` and `sync.watch: false`; preserve CLI/status transient no-watch behavior; do not disable desktop/interactive freshness by broadly keying only on `gateway.mode=local` without a precise runtime signal. Cover builtin and QMD manager paths plus resolver behavior. Update schema help/docs only if the visible default semantics change. Related but separate context: https://github.com/openclaw/openclaw/issues/64549 was about auto-indexing when watch is true and should not be folded into this fix. Follow OpenClaw extension/core boundaries; keep new plugin runtime parameters optional/backwards-compatible for third-party memory plugins; do not change unrelated memory indexing policy.
