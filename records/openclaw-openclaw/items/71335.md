## Summary

Keep this issue open. Current main still resolves an unset memory sync watcher to true with no gateway/headless context, and the normal memory_search path can create full builtin or QMD managers that start chokidar watchers. The closed related #64549 is adjacent watcher freshness behavior, not this gateway default policy change.

## What This Changes

Default `agents.defaults.memorySearch.sync.watch` to false for headless gateway mode while preserving explicit watch overrides and the current CLI/desktop default behavior.

## Best Possible Solution

Add a runtime-aware memory sync default so unset `sync.watch` resolves false only for headless gateway memory manager creation, while explicit true/false overrides remain authoritative and CLI/status/desktop freshness stays covered by tests and schema/docs.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is available: leave `sync.watch` unset, resolve memory sync config, then follow a normal gateway `memory_search` call into a full builtin or QMD manager that starts chokidar because the resolved watch value is true.

## Solution Assessment

Unclear as originally phrased if implemented by broadly checking `gateway.mode`. The narrow maintainable solution is to thread an explicit headless/default-policy context into sync resolution or manager creation so only gateway memory tool managers get the false default.
