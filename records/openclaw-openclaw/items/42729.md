## Summary

Keep open. Current main still lacks the PR's stale already-executed slot guard in the cron runnable path, while startup catch-up still runs against persisted due timestamps before full recomputation. The PR remains a focused implementation candidate for the duplicate restart execution bug confirmed in the discussion.

## Best Possible Solution

Review and land this PR, or port the same narrow strict stale-slot guard into current main's `isRunnableJob` startup catch-up path with the regression test. The implementation should skip already-executed stale persisted slots while preserving genuine missed cron slots and same-millisecond retry slots.
