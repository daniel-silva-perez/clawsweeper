## Summary

Keep this PR open. Current main still lacks the PR's stale already-executed slot guard in the cron runnable path, and startup catch-up still evaluates persisted due timestamps before the later full recompute. The PR remains a focused candidate fix with a matching regression test and no visible security-sensitive drift.

## Best Possible Solution

Review and land this PR, or port the same strict already-executed stale-slot guard into current main's isRunnableJob/startup catch-up path with the regression test. Keep the guard narrow so genuine missed cron slots still catch up and same-millisecond retry slots are not suppressed.
