## Summary

Keep open. The linked doctor false-positive report is still a real product issue on current main, but this PR is not merge-ready: the cleanup-hint refactor regresses status output, drops manual legacy cleanup guidance, and emits user-scope commands for system-scope services; the SecretRef part also changes exported strict helper behavior in a security-sensitive config path.

## What This Changes

The PR changes doctor gateway cleanup hints to render from detected non-legacy extra services, wraps doctor execution in a process-wide SecretRef leniency env flag, and updates generated plugin SDK API baselines for the new SecretRef option.

## Best Possible Solution

Revise the fix on current main so doctor and status both render cleanup hints from actual detected service metadata, preserve manual guidance for unremoved legacy services, honor user versus system service scope, and keep exported SecretRef helpers strict while doctor uses explicit inspect/read-only resolution paths for warnings.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Pass detected services to status hint rendering:** `src/daemon/inspect.ts:29-30`
  - body: `renderGatewayServiceCleanupHints()` now returns `[]` when called without services, but the status printer still calls it with no arguments after reporting `extraServices`. That would make `openclaw status --deep` report extra gateway-like services without cleanup hints.
  - confidence: 0.93
- **[P2] Keep manual hints for unresolved legacy services:** `src/commands/doctor-gateway-services.ts:441-442`
  - body: Filtering out every legacy service before rendering hints removes the fallback commands when legacy cleanup is skipped by policy, declined, fails, or is system-scoped. Doctor still reports those services but no longer gives the user actionable cleanup guidance.
  - confidence: 0.88
- **[P2] Use system launchctl commands for system plists:** `src/daemon/inspect.ts:39-43`
  - body: For deep macOS findings from `/Library/LaunchDaemons`, `svc.scope` can be `system`, but this helper always emits `launchctl bootout gui/$UID/...`. That targets the user GUI domain instead of unloading the detected system service.
  - confidence: 0.9
- **[P2] Use systemd system commands for system units:** `src/daemon/inspect.ts:47`
  - body: For deep Linux findings from systemd system directories, `svc.scope` can be `system`, but this helper always emits `systemctl --user`. That targets the user service manager and leaves the detected system unit installed.
  - confidence: 0.9

## Security Review
