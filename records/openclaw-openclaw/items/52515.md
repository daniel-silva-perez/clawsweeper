## Summary

Keep open. Current main still has the active-profile cleanup-hint behavior and the linked user problem remains valid, but this branch is not merge-ready because it misses a status caller, drops some fallback legacy-service guidance, emits user-scope cleanup commands for system-scope services, and weakens a public SecretRef helper path.

## What This Changes

The PR changes doctor gateway cleanup hints to render from detected extra services, wraps doctor execution in a process-wide env flag that downgrades unresolved exec SecretRefs to warnings, and updates generated plugin SDK API baselines.

## Best Possible Solution

Revise the fix on current main so cleanup hints are rendered from actual detected service metadata in both doctor and status, user versus system service scope is reflected in commands, legacy services that were not actually removed still get manual guidance, and doctor-only SecretRef tolerance uses explicit inspect/read-only paths without weakening exported strict helpers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Pass detected services into status cleanup hints:** `src/daemon/inspect.ts:29-30`
  - body: `renderGatewayServiceCleanupHints()` now returns `[]` when no service list is supplied, but `status.print.ts` still calls it with no arguments after reporting `extraServices`. That regresses `openclaw status --deep` by showing extra gateway-like services without cleanup guidance.
  - confidence: 0.93
- **[P2] Preserve manual hints for unremoved legacy services:** `src/commands/doctor-gateway-services.ts:441-442`
  - body: Filtering every `svc.legacy` entry before rendering hints removes the only manual cleanup guidance when legacy cleanup is skipped by policy, declined, fails, or is system-scoped. In those paths doctor still reports the legacy service but no longer tells the user how to remove it.
  - confidence: 0.88
- **[P2] Honor service scope in cleanup commands:** `src/daemon/inspect.ts:38-46`
  - body: The new helper receives service metadata but always emits user-manager commands such as `launchctl bootout gui/$UID/...` and `systemctl --user ...`. For system LaunchDaemons or systemd units found by `doctor --deep`, those commands target the wrong manager and leave the detected service installed.
  - confidence: 0.9
