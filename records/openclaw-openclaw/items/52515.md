## Summary

This PR should stay open because current `main` still has the reported cleanup-hint problem, but the proposed patch is not ready to merge. The branch regresses the shared status cleanup-hint path, drops fallback guidance for some legacy-service cases, and changes exported SecretRef behavior in a way that needs maintainer/secops review.

## What This Changes

The PR changes doctor gateway cleanup hints to render from detected extra services, wraps doctor execution in an env flag that downgrades unresolved `exec` SecretRef errors to warnings, and updates generated plugin SDK API baselines for the changed SecretRef helper signature.

## Best Possible Solution

Revise the patch so both doctor and status render cleanup hints from actual detected service metadata, cleanup commands honor user versus system service scope, unremoved legacy services still get manual guidance, and doctor-specific SecretRef tolerance stays on an explicit inspection path while exported runtime/plugin helpers remain strict.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Pass detected services into status cleanup hints:** `src/daemon/inspect.ts:29-30`
  - body: `renderGatewayServiceCleanupHints()` now returns an empty list unless a service list is supplied, but the status printer still calls it with no arguments after reporting `extraServices`. That regresses `openclaw status --deep` by showing extra gateway-like services without the cleanup guidance that existed before this patch.
  - confidence: 0.93
- **[P2] Keep manual hints for unremoved legacy services:** `src/commands/doctor-gateway-services.ts:441-442`
  - body: Filtering every `svc.legacy` entry before rendering hints removes the only manual cleanup guidance when legacy cleanup is skipped by policy, declined, fails, or is system-scoped. In those paths doctor still reports the legacy service but no longer tells the user how to remove it.
  - confidence: 0.88
- **[P2] Honor service scope in cleanup commands:** `src/daemon/inspect.ts:38-46`
  - body: The new helper receives `svc.scope` but always emits user-manager commands such as `launchctl bootout gui/$UID/...` and `systemctl --user ...`. For services detected under system launchd/systemd locations, those commands target the wrong manager and leave the detected service installed.
  - confidence: 0.9

## Security Review

Status: needs_attention

Concerns:
