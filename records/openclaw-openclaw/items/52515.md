## Summary

Keep open. This PR targets a real doctor/gateway false-positive problem, and linked issue #52510 remains open, but the branch still has blocking correctness findings around status cleanup hints, legacy fallback guidance, service scope handling, and a security-sensitive SecretRef API change that needs maintainer/secops review before merge.

## What This Changes

The PR changes doctor gateway cleanup hints to render from detected extra services, wraps doctor execution in a process-wide env flag that downgrades unresolved exec SecretRefs to warnings, and updates generated plugin SDK API baselines.

## Best Possible Solution

Revise the fix on current main so cleanup hints are rendered from actual detected service metadata in both doctor and status, commands honor user versus system service scope, unremoved legacy services keep actionable manual guidance, and doctor SecretRef warnings use explicit inspect/read-only paths while exported strict helpers remain strict for runtime and plugin callers.

  - body: `renderGatewayServiceCleanupHints()` now returns `[]` when called without a service list, but the current status printer still calls it with no arguments after printing `extraServices`. - **[P2] Preserve manual hints for unremoved legacy services:** `src/commands/doctor-gateway-services.ts:441-442`
  - body: Filtering out every `svc.legacy` before rendering hints removes fallback commands when legacy cleanup is skipped, declined, fails, or is system-scoped. Doctor still reports the legacy service in those paths, but no longer gives the user actionable cleanup guidance.
  - confidence: 0.88
- **[P2] Honor service scope in cleanup commands:** `src/daemon/inspect.ts:38-46`
  - body: The helper receives `svc.scope` but always emits user-domain `launchctl` and `systemctl --user` commands. For `doctor --deep` findings from system LaunchDaemons or systemd directories, those commands target the wrong manager and leave the detected service installed.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: Needs attention: the diff does not add dependencies or workflows, but it changes exported SecretRef resolution behavior through a process-wide doctor env flag.

Concerns:

- **[medium] Avoid process-wide leniency for exported SecretRef helpers:** `src/config/types.secrets.ts:143`
  - body: The PR lets exported strict SecretRef helpers return successfully for unresolved `exec` refs whenever `OPENCLAW_ALLOW_EXEC_SECRETREF_FAILURES` is set. Because `normalizeResolvedSecretInputString` is exposed through the plugin SDK and used by channels/providers, leniency can leak beyond explicit doctor inspection paths and mask credential failures.
  - confidence: 0.78
