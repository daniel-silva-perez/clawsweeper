## Summary

Keep this PR open. The linked bug remains relevant, but the branch is not ready to merge: current main has not implemented the cleanup-hint fix, the PR still has concrete regressions in the cleanup-hint paths, and the SecretRef change touches security-sensitive exported behavior that needs maintainer/secops review.

## What This Changes

The PR changes doctor gateway cleanup hints to render from detected extra services, wraps doctor execution in an env flag that downgrades unresolved exec SecretRefs to warnings, and updates generated plugin SDK API baselines.

## Best Possible Solution

Revise and rebase the fix onto the current doctor health flow so doctor and status both render cleanup hints from actual detected service metadata, cleanup commands honor user versus system service scope, unremoved legacy services still get actionable manual guidance, and doctor SecretRef warnings use an explicit read-only inspection path while exported strict helpers remain strict for runtime and plugin callers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep status cleanup hints wired to detected services:** `src/daemon/inspect.ts:28-30`
  - body: `renderGatewayServiceCleanupHints()` now returns `[]` when no service list is passed, but the status printer still calls it with no arguments after reporting `extraServices`. That makes `openclaw status --deep` report extra gateway-like services without the cleanup hints users currently get for that troubleshooting path.
  - confidence: 0.93
- **[P2] Preserve manual hints for unremoved legacy services:** `src/commands/doctor-gateway-services.ts:441-442`
  - body: Filtering out every legacy service before rendering hints removes fallback cleanup guidance when legacy cleanup is skipped by policy, declined, fails, or is system-scoped. In those cases doctor still reports the service but no longer gives a manual command to remove it.
  - confidence: 0.88
- **[P2] Honor service scope in cleanup commands:** `src/daemon/inspect.ts:38-46`
  - body: The helper receives `svc.scope`, but the new commands always target the user launchd/systemd manager. Deep scan findings from `/Library/LaunchDaemons` or system systemd directories are marked `scope: "system"`, so these hints target the wrong service manager and leave the detected system service installed.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff does not add dependencies or workflows, but it changes exported SecretRef resolution through a process-wide doctor leniency flag.

Concerns:

- **[medium] Avoid process-wide leniency for strict SecretRef helpers:** `src/config/types.secrets.ts:143`
  - body: The PR lets strict exported SecretRef helpers return successfully for unresolved `exec` refs whenever `OPENCLAW_ALLOW_EXEC_SECRETREF_FAILURES` is set. Because these helpers are exported through the plugin SDK and used outside doctor, that flag can mask credential failures beyond an explicit doctor inspection path.
  - confidence: 0.78
