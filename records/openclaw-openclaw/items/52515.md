## Summary

Keep open. The linked doctor false-positive is real and current main still renders cleanup hints from the active profile once any extra gateway-like service is reported, but this PR introduces regressions in the status path, legacy-service fallback guidance, service-scope cleanup commands, and security-sensitive SecretRef behavior.

## What This Changes

The PR changes doctor cleanup hints to render from detected extra services, wraps doctor execution in a process env flag that downgrades unresolved exec SecretRefs to warnings, and updates generated plugin SDK API baselines for the new SecretRef helper parameter.

## Best Possible Solution

Revise and rebase the fix so doctor and status both render cleanup hints from the actual detected service metadata, scope cleanup commands correctly for user versus system launchd/systemd services, preserve fallback manual guidance for legacy services when automatic cleanup is skipped or fails, and use explicit read-only SecretRef inspection/gateway resolution paths while keeping exported strict helpers strict.

## Reproduction Assessment

Yes for the patch regressions: current main shows both doctor and status call renderGatewayServiceCleanupHints() without detected-service arguments, while the PR changes that helper to return an empty list in that case. The linked bug is also reproducible in principle whenever an extra active gateway-like service is reported, because current main's cleanup hints are still derived from the active profile.

## Solution Assessment

No. The PR targets a valid problem, but this is not the best fix because it only updates the doctor call site, drops legacy fallback hints, ignores service scope in generated commands, and handles doctor SecretRef tolerance through process-wide exported-helper leniency instead of a narrow inspection path.

  - confidence: 0.9
