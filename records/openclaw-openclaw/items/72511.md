## Summary

Keep this PR open for maintainer review. Current main still lacks the replay idempotency guard needed for the linked overflow-recovery duplication report, but the proposed patch would also collapse legitimate same-role, same-content transcript messages during any rewrite, so it is not safe to merge as-is.

## What This Changes

The PR adds SHA-256-based identity tracking to transcript-rewrite suffix replay so duplicate message, compaction, custom, and custom_message branch entries are skipped, with regression coverage for the reported duplicate replay cases.

## Best Possible Solution

Keep the PR open, but revise the replay idempotency rule so it fixes the #66443 pollution pattern without making transcript rewrites lossy for valid repeated turns. A safer fix should either narrow content-only suppression to known replay artifacts or include enough stable message metadata to preserve distinct logical entries, then keep #69208 open for the broader duplicate-transcript program.

## Reproduction Assessment

Yes. A unit-level `SessionManager.inMemory()` setup with repeated `role=user` suffix entries after the first replacement exercises the reported current-main duplication path; for the PR regression, place two legitimate identical messages or same-content tool results with different call ids in the replayed suffix and the new key will skip the later one.

## Solution Assessment

No. The PR is aimed at the right boundary, but role-plus-content dedupe for every message entry is broader than the maintainable fix; the narrower solution is to preserve generic rewrite losslessness while suppressing only entries that are proven replay pollution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve repeated message entries in the dedupe key:** `src/agents/pi-embedded-runner/transcript-rewrite.ts:41`
  - body: The message identity key is only `role + content`, and the replay loop applies it to every message entry. Any rewrite that starts before two legitimate same-role/same-content turns, or two tool results with the same content but different call ids, will skip the later entry and silently delete transcript history. Please include distinguishing message metadata in the generic key or restrict content-only suppression to a narrowly identified polluted-replay case.
  - confidence: 0.86
