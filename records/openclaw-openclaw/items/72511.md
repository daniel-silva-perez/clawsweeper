## Summary

Keep this PR open for revision: it targets a real transcript-growth bug, but the proposed message identity is broad enough to make generic transcript rewrites lossy by skipping legitimate repeated messages or same-text tool results. Current main still replays the suffix unconditionally, so this is useful work, but the head is not merge-ready.

## What This Changes

The PR adds SHA-256 identity tracking to transcript rewrite suffix replay and regression tests for duplicate user, compaction, custom, and custom_message entries.

## Best Possible Solution

Revise the replay idempotency rule so it fixes the #66443 polluted-suffix cases without changing the losslessness contract for ordinary transcript rewrites. The maintainable end state is a narrowly scoped dedupe invariant with metadata-aware message identity, plus regression coverage for legitimate repeated messages and same-text tool results.

## Reproduction Assessment

Yes. A focused SessionManager.inMemory setup can append two distinct same-role/same-content entries after a rewrite target; with this PR's key, the later entry is skipped during suffix replay. The original bug also has a concrete #66443 overflow-recovery capture with repeated role=user content hashes.

## Solution Assessment

No. The PR is aimed at the right helper, but role-plus-content dedupe for every message entry is broader than the safest fix; the alternative is to include stable per-message metadata or restrict content-only suppression to entries proven to be replay pollution.
