## Summary

Keep this issue open. Current `main` already exposes channel metadata in normal inbound/agent-run paths, but the slash-command prompt reconstruction path still drops runtime channel metadata: `resolveCommandsSystemPromptBundle()` passes `host/os/arch` as `"unknown"` and omits `channel`, `capabilities`, `shell`, and channel action metadata. The linked fix PR #21271 is closed unmerged, so this remains a narrow valid fix candidate.

## Best Possible Solution

Keep this issue open and implement the missing runtime metadata in `resolveCommandsSystemPromptBundle()` only. Reuse or extract the embedded-runner runtime prompt pattern so `/context` and `/export-session` prompt reconstruction includes normalized channel, account-aware capabilities, shell, host/OS/arch, and supported channel actions where available. Treat selective reply routing as a separate follow-up product/API decision.

## Clownfish Work Prompt

Fix the remaining channel-awareness gap for this issue. Current `main` already includes channel metadata in normal inbound and embedded-runner prompts, but `src/auto-reply/reply/commands-system-prompt.ts` rebuilds slash-command prompts for `/context` and `/export-session` with `host/os/arch: "unknown"` and without `channel`, `capabilities`, `shell`, or `channelActions`.

Scope the PR to `resolveCommandsSystemPromptBundle()` and its tests. Derive a normalized runtime channel from the command/context (`ctx.OriginatingChannel` and existing command/provider fallbacks), pass real host/OS/arch/shell values, resolve account-aware capabilities using `params.ctx.AccountId`, include message-tool capabilities without duplicating capability ids, and include supported channel actions where the current command context can support them. Reuse or extract the existing embedded-runner pattern from `src/agents/pi-embedded-runner/run/attempt.ts` if that keeps behavior aligned, but do not change ordinary agent-run prompt generation unless a pure helper extraction requires it.

Add focused coverage in `src/auto-reply/reply/commands-system-prompt.test.ts` proving the bundle passes channel, capabilities, shell, host/OS/arch, and channel actions to `buildSystemPromptParams()`, including an account-specific capabilities case. Existing inbound metadata behavior should remain unchanged. Do not implement selective cross-channel reply routing in this PR; that remains a separate product/API decision.

Related refs: https://github.com/openclaw/openclaw/issues/20837 and closed-unmerged PR https://github.com/openclaw/openclaw/pull/21271. Changelog likely not needed for this internal prompt-diagnostics fix unless maintainers want the user-visible `/context`/`/export-session` improvement called out.
