## Summary

Keep open. The feature request is valid, but this PR is not merge-ready on current main: it targets old memory files that are no longer the active implementation, and the active schema, resolver, SDK contract, remote request parser, and OpenAI plugin/batch paths still do not carry a configurable encoding format end to end.

## What This Changes

The PR adds an optional `memorySearch.remote.encodingFormat` setting intended to send `encoding_format` on remote embedding requests and decode base64 embedding responses.

## Best Possible Solution

Rework the feature onto the current memory-host SDK and provider-plugin architecture after maintainers decide the provider compatibility policy. A complete change should update schema, public config types, labels/help/docs, resolved memory settings, plugin SDK provider options, OpenAI-compatible direct and batch request bodies, robust base64 float32 decoding, and focused tests for config propagation plus direct and batch response parsing.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: configure `agents.defaults.memorySearch.remote.encodingFormat` and trace schema, `resolveMemorySearchSettings`, `MemoryEmbeddingProviderCreateOptions`, and the active OpenAI direct/batch paths; the key is either rejected on main or not carried through the active provider paths in this PR.

## Solution Assessment

No. The direction is plausible, but this PR is not the best current fix because it is based on retired `src/memory/*` implementation files and misses the active package-backed SDK and bundled OpenAI plugin paths.
