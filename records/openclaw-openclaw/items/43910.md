## Summary

Keep this PR open. Current main still does not implement configurable `memorySearch.remote.encodingFormat`, and the PR remains a plausible feature candidate, but it is stale against the current memory-host-sdk/provider-plugin layout and needs a rebase plus broader propagation before it can merge.

## Best Possible Solution

Keep this PR open for maintainer review, but require a rebase onto current main. The complete implementation should add `encodingFormat` to the strict config schema, exported config types, labels/help/docs, `ResolvedMemorySearchConfig`, `MemoryEmbeddingProviderCreateOptions`, provider-owned embedding clients, direct and batch OpenAI-compatible request builders, response parsers, parser tests for `float` and `base64`, and cache-key/reindex behavior where vector encoding changes stored data. Maintainers should also decide whether the narrower #41670 compatibility fix should land separately or be folded into this broader configurable feature.

## Risks / Open Questions

- Merging the PR as-is would likely miss the active memory-host-sdk and bundled provider-plugin paths on current main, leaving the advertised config incomplete or ineffective.
- A complete `base64` implementation should validate decoded embeddings: malformed base64, byte lengths not divisible by four, non-finite floats, empty vectors, and dimension mismatches should fail clearly instead of corrupting search vectors.
- If encoding format can affect stored vectors or provider behavior, the provider runtime cache key/reindex behavior should be reviewed so existing memory indexes are not silently reused with incompatible embedding semantics.
- The related open PR #41670 covers a narrower compatibility fix by sending `encoding_format: "float"`; it does not fully supersede this configurable `float`/`base64` feature.
