## Summary

Keep open. Current main still does not implement configurable `memorySearch.remote.encodingFormat`, and the PR is not merge-ready because the config value is not preserved through the current resolved memory settings path. The branch is useful context, including follow-up work that addressed the earlier base64 decode concern, but it needs a maintainer-guided rebase/port against the current memory host SDK and OpenAI plugin architecture; related #41670 is still open and unmerged rather than a superseding shipped fix.

## What This Changes

The PR adds an optional `memorySearch.remote.encodingFormat` setting, sends `encoding_format` on remote embedding requests, decodes base64 embedding responses in direct and batch memory paths, and updates config labels/types.

## Best Possible Solution

Port the feature onto the current memory host SDK and provider plugin architecture: add schema/type/help/generated config coverage, preserve the setting through resolved memory config, send `encoding_format` consistently in direct and OpenAI-compatible batch requests only where the provider supports it, decode and validate base64 float32 responses in shared parsers, document the provider compatibility caveat, and cover omitted, `float`, `base64`, invalid config, direct, and batch cases with focused tests.
