## Summary

Keep open. The contribution addresses useful memory embedding compatibility work and follow-up commits did add base64 decode paths after the initial review comment, but the PR is still not merge-ready: the new config key is not propagated through resolved memory settings, the branch is dirty against current main, and the active implementation has moved into the memory host SDK plus provider plugins.

## What This Changes

The PR adds an optional `memorySearch.remote.encodingFormat` config surface, sends `encoding_format` on embedding requests, and adds direct and batch base64 response decoding in the older `src/memory` embedding paths.

## Best Possible Solution

Port the feature onto the current memory host SDK and provider plugin architecture only after maintainers choose the provider policy: carry the setting through resolved memory config, add schema/type/help/label coverage, send `encoding_format` only on supported provider paths, decode and validate base64 float32 responses in shared parsers, document compatibility caveats, and cover omitted, `float`, `base64`, invalid config, direct, and batch behavior with focused tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: add `agents.*.memorySearch.remote.encodingFormat`, then follow `resolveMemorySearchSettings`, which currently rebuilds `remote` without that key before provider creation.

## Solution Assessment

No, not as proposed. The user-facing direction is plausible, but the patch needs a rebase onto the active SDK/plugin paths and a maintainer decision on whether this should be configurable, always `float`, provider-gated, or omitted for compatibility.
