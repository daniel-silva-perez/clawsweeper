## Summary

Keep open. The feature request is real, but current main still lacks a `memorySearch.remote.encodingFormat` contract and the PR is stale against the active package-backed memory SDK and OpenAI plugin paths. The proposed patch would not reliably make the config reach active embedding requests without a rebase and maintainer policy decision, especially given the overlapping open PR #41670.

## What This Changes

The PR adds an optional `memorySearch.remote.encodingFormat` config/type/label surface, sends `encoding_format` on the older remote embedding request path, and attempts base64 response decoding in older direct and batch memory embedding helpers.

## Best Possible Solution

Rework the feature onto current main after maintainers choose the provider policy. A complete implementation should document where `encodingFormat` is supported, carry it through schema/types/help/labels, resolved memory settings, provider options, cache identity, the package-backed SDK, bundled OpenAI direct and batch paths, and robust base64 float32 decoding with focused tests.

Yes. A high-confidence static reproduction exists: configure `agents.defaults.memorySearch.remote.encodingFormat` and trace schema/resolution/provider creation; current main rejects or lacks the key, and the proposed PR still does not carry it through the active package-backed memory SDK and OpenAI plugin paths.

## Solution Assessment

No. The direction is plausible, but this PR is not the best current fix until it is rebased onto the active memory-host SDK/provider-plugin architecture and reconciled with provider compatibility policy and #41670.
