## Summary

Keep open. Current main still has no `memorySearch.remote.encodingFormat` config contract and no remote memory embedding request/response support for configurable `encoding_format`; the PR is useful source work, but it is not merge-ready because the normal config resolver would still drop the new key and current main has since moved the touched memory code into the memory host SDK and provider plugins.

## What This Changes

The PR adds an optional `memorySearch.remote.encodingFormat` config key, sends `encoding_format` on remote embedding requests, and decodes base64 embedding responses in direct and batch paths.

## Best Possible Solution

Implement this against the current memory host SDK and provider plugin architecture: add the config schema/type/help/generated entries, carry `encodingFormat` through resolved memory settings, send `encoding_format` only when explicitly configured, decode and validate base64 float32 responses in shared direct and batch parsers, and add focused tests for omitted, `float`, `base64`, invalid, and batch cases. Then decide whether that current-main implementation supersedes the narrower #41670.
