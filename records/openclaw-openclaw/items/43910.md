## Summary

Keep open. Current main still does not implement the requested `memorySearch.remote.encodingFormat` surface, and this PR is not ready to merge because the new config key is not preserved through the resolved memory settings path. The branch also overlaps with open PR #41670 and is stale against the current memory host SDK/OpenAI plugin architecture, so maintainers should choose the provider policy and rebase direction before promoting repair work.

## What This Changes

The PR adds an optional `memorySearch.remote.encodingFormat` setting, sends `encoding_format` on remote embedding requests, and attempts to decode base64 embedding responses in direct and batch paths.

## Best Possible Solution

Port the feature onto the current memory host SDK and OpenAI provider plugin architecture: add schema/type/help/label coverage, preserve the setting through resolved memory config, send `encoding_format` only on provider paths where maintainers decide it is supported, decode and validate base64 float32 responses in shared parsers, document the provider compatibility caveat, and cover omitted, `float`, `base64`, invalid config, direct, and batch cases with focused tests.

## Reproduction Assessment

Yes. The blocking issue has a high-confidence static reproduction path: add `agents.*.memorySearch.remote.encodingFormat` and follow `resolveMemorySearchSettings`, which currently rebuilds `remote` without that key before provider creation.

## Solution Assessment

No, not as proposed. The user-facing direction is plausible, but the patch is not the narrowest maintainable solution until it is rebased onto the active SDK/plugin paths and reconciled with the current provider compatibility policy.
