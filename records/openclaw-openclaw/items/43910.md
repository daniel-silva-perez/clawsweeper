## Summary

Keep open. Current main still does not implement a `memorySearch.remote.encodingFormat` config contract, and the PR branch is not merge-ready because it targets removed legacy `src/memory/*` embedding files while current main routes memory embeddings through `packages/memory-host-sdk` and the OpenAI provider plugin. The related open PR #41670 tracks a narrower always-`float` compatibility fix, so maintainers should decide the product contract before asking for a rebase or replacement.

## What This Changes

The PR adds an optional `memorySearch.remote.encodingFormat` config value and threads `float`/`base64` handling through legacy remote embedding request, response decoding, and batch-output code.

## Best Possible Solution

Maintainers should first decide whether OpenClaw wants an always-float compatibility fix, a user-configurable float/base64 memory-search option, or both. If configurable encoding is accepted, the implementation should be rebuilt on current main across the strict config schema, exported config/provider types, resolved memory config, active memory-host SDK request/parser/batch helpers, OpenAI plugin direct and batch paths, docs/help metadata, cache-key behavior if response format can affect stored vectors, and targeted tests.
