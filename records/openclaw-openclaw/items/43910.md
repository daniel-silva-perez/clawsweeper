## Summary

Keep this PR open. Current main still does not implement configurable `memorySearch.remote.encodingFormat`, and the PR remains a plausible feature candidate, but it is stale against the current memory-host-sdk/provider-plugin layout and should not merge as-is.

## Best Possible Solution

Keep this PR open for maintainer review, but require a rebase onto current main. The complete path should add `encodingFormat` to the config schema, exported config types, help/labels/docs, `ResolvedMemorySearchConfig`, `MemoryEmbeddingProviderCreateOptions`, provider-owned embedding context, direct and batch OpenAI-compatible request builders, cache-key/reindex behavior where vector encoding affects stored data, and parser tests for validated `float` and `base64` responses. Maintainers should also decide how this broader feature relates to the narrower #41670 compatibility fix.

## Risks / Open Questions

- Merging the PR as-is would likely miss the active memory-host-sdk and provider-plugin paths, making the advertised config incomplete or ineffective on current main.
- A complete base64 implementation must validate remote provider responses: malformed base64, byte lengths not divisible by four, non-finite floats, empty vectors, and dimension mismatches should fail clearly instead of corrupting stored/search vectors.
- The narrower open #41670 may solve a compatibility subset by always sending `encoding_format: "float"`, but it does not supersede the broader configurable `float`/`base64` feature requested here.
