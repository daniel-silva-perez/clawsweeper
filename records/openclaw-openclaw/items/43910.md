## Summary

Keep this PR open for maintainer review, but do not merge it as-is. Current main still does not expose configurable `memorySearch.remote.encodingFormat`, while this branch is stale against the current memory-host-sdk and provider-plugin layout and misses the active request/parse paths.

## What This Changes

The PR adds an optional `memorySearch.remote.encodingFormat` config value for `float` or `base64` and threads it through old remote embedding request and batch decoding paths.

## Best Possible Solution

Rebase or replace this branch against current main only after maintainers decide the product contract. A complete implementation would align schema, exported types, generated config/help/docs, resolved memory config, provider create options, OpenAI-compatible direct and batch request builders, and validated float/base64 response parsing tests across the active memory-host-sdk and provider-plugin paths, while coordinating with #41670.

  - body: Adding `encodingFormat` to the schema/type surface is not enough: the current config resolver rebuilds `remote` without this field, and provider creation reads resolved memory settings. Without updating those active direct and batch paths, the advertised config will miss the code that now sends embedding requests.
  - confidence: 0.9
- **[P2] Validate decoded embedding payloads:** `src/memory/embeddings-remote-fetch.ts:3-6`
  - body: The new base64 decoder accepts whatever `Buffer.from(..., "base64")` returns and then casts fallback values as `number[]`. Malformed byte lengths, non-finite floats, or unexpected strings should fail clearly before vectors reach storage/search math; current adjacent LanceDB code already does this kind of validation.
  - confidence: 0.78
