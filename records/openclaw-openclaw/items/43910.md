## Summary

Keep open. The feature request is real, but the PR is not merge-ready against current main: the requested config surface is still missing, the branch targets retired `src/memory/*` files, and the active implementation now lives in `packages/memory-host-sdk` plus provider plugins. The remaining decision also overlaps with #41670 and the current documented compatibility policy around omitting `encoding_format`.

## What This Changes

This PR adds optional `memorySearch.remote.encodingFormat` support for float/base64 embedding requests and response decoding across config, remote embedding, and batch paths in the old `src/memory` implementation.

## Best Possible Solution

Rework the feature, if maintainers want it, on the current memory-host SDK and provider-plugin architecture. A complete design should align schema, public config types, labels/help/docs, resolved memory settings, provider option contracts, OpenAI-compatible direct and batch request bodies, robust base64 float32 decoding, cache/reindex implications, and focused tests for config propagation plus direct and batch parsing.
