## Summary

Keep open. Current main still has the reported runtime gap: `skipBootstrap` is honored when creating the workspace, but the shared runtime bootstrap resolver used by CLI and embedded agent prompt construction does not check it. The PR targets the right shared path and I did not find a blocking correctness or security issue, but it broadens the public config contract without a matching docs/schema/test update.

## What This Changes

The PR adds an early `agents.defaults.skipBootstrap === true` guard in `src/agents/bootstrap-files.ts` so runtime bootstrap file resolution returns no files before system-prompt injection.

## Best Possible Solution

Accept a shared-runtime fix if maintainers want `skipBootstrap` to mean no workspace bootstrap creation and no runtime injection; pair it with a small resolver or prompt-report regression test and docs/schema wording so users can distinguish it from `contextInjection`.

## Reproduction Assessment

Yes. A deterministic code-path reproduction is available: create or keep workspace bootstrap files, set `agents.defaults.skipBootstrap: true`, then run a CLI or embedded agent turn; current main still resolves those files through `resolveBootstrapContextForRun` and records them in `systemPromptReport`.

## Solution Assessment

Mostly yes. The shared resolver is the narrowest code point that feeds both CLI and embedded runtime injection, but the best mergeable version should also align the documented config contract or intentionally steer users to `contextInjection: "never"` if `skipBootstrap` is meant to remain creation-only.
