## Summary

Keep open. Current main still advertises shared managed skills from outside the agent workspace while `tools.fs.workspaceOnly` wraps the `read` tool with a workspace-root guard, and the config surface has no `allowRead` or equivalent read-only exception. The sandbox skill-sync path helps sandbox workspaces, but it is scoped to sandbox workspace setup and does not solve plain `tools.fs.workspaceOnly: true` agents reading `~/.openclaw/skills` paths.

## What This Changes

The issue requests that agents using `tools.fs.workspaceOnly: true` can still read prompt-advertised platform skill files without manually copying skills into every workspace.

## Best Possible Solution

Keep the issue open for a maintainer-approved design that lets workspace-only agents read only the skill files they are meant to see, either by using validated read-only skill roots or by presenting workspace-scoped synced skill paths, with schema, docs, and regression tests aligned to the chosen security model.

## Reproduction Assessment

Yes. The code-based path is high confidence: configure a visible managed skill under `~/.openclaw/skills`, enable `tools.fs.workspaceOnly: true`, observe the prompt advertise that outside-workspace `SKILL.md`, then `read` rejects it through the workspace-root guard.

## Solution Assessment

Unclear. The suggested `allowRead` or auto-exempt design would address the user-visible failure, but the narrowest maintainable fix needs maintainer security judgment so the workspace-only read boundary is not silently weakened.
