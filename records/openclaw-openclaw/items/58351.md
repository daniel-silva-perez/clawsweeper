## Summary

Keep open. Current main has a partial sandbox-specific mitigation that syncs skills into read-only sandbox workspaces, but the reported `tools.fs.workspaceOnly: true` behavior still appears current for normal host workspace-only runs: managed skills are loaded from `~/.openclaw/skills`, prompt paths are rendered from those skill file paths, and the `read` tool is still wrapped with a workspace-root guard when `workspaceOnly` is enabled.

## Best Possible Solution

Keep this issue open and fix the remaining host `tools.fs.workspaceOnly` path mismatch. A conservative implementation would either add a typed read-only allowlist for platform-managed skill roots or rewrite/sync prompt-facing skill paths whenever the effective filesystem policy would block the listed `SKILL.md`; it should include a regression test with a managed skill under `~/.openclaw/skills` and `tools.fs.workspaceOnly: true`.

## Risks / Open Questions

- Current main may be better than the original report for sandboxed read-only workspaces because eligible skills are copied into the sandbox workspace, but that is narrower than the reported `tools.fs.workspaceOnly` host read-path problem.
- Closing now could hide a real isolation/usability bug for users who rely on shared managed skills plus workspace-only filesystem tools without a read-only sandbox mirror.
