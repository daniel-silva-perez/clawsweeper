## Summary

Current main does not implement the requested WhatsApp per-contact messaging policy. The WhatsApp config schema exposes root/account-level `dmPolicy`, `allowFrom`, `groupPolicy`, `dms`, and `direct`, but no `contacts` map or `allowInbound`/`allowOutbound`/`allowAutonomousReply` fields. Inbound admission, outbound target resolution, and auto-reply dispatch still rely on the existing global/account allowlist and prompt surfaces, so the reported safety use case remains valid and actionable in the bundled WhatsApp plugin.

## Best Possible Solution

Keep this issue open. The likely best implementation is WhatsApp-owned: add a strict root/account `contacts` config schema, a helper such as `extensions/whatsapp/src/contact-policy.ts` to normalize E.164 keys and field-merge account overrides, then enforce it in inbound access control, explicit outbound target resolution, and auto-reply delivery before `dispatchWhatsAppBufferedReply`, with config/docs/tests covering `allowInbound`, `allowOutbound`, and `allowAutonomousReply`.

## Risks / Open Questions

- This is a real safety/product design request, not a stale support report; closing it now would lose a concrete bundled WhatsApp policy gap.
- Implementation needs careful merge semantics across root and account config so field-level account overrides do not accidentally replace the whole contact policy object.
