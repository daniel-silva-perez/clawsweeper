## Summary

Keep open. Current main still lacks a WhatsApp per-contact policy surface: direct contact config is prompt-only, strict validation has no inbound/outbound/autonomous direction fields, inbound admission is still governed by dmPolicy/allowFrom/pairing, accepted DMs continue into read/enqueue/session/reply handling, and outbound authorization reuses allowFrom rather than a separate explicit outbound permission.

## What This Changes

The issue requests root-level and account-level WhatsApp per-contact policy rules for allowInbound, allowOutbound, and allowAutonomousReply with account-level field overrides.

## Best Possible Solution

Design this as a WhatsApp-owned plugin/config change: add a typed per-contact policy surface, merge root and account policy fields predictably, enforce allowInbound before read receipts/hooks/enqueue/session/reply side effects, enforce allowOutbound in outbound target resolution, and enforce allowAutonomousReply on reply/heartbeat/autonomous delivery paths with focused docs and tests.

## Reproduction Assessment

Not applicable as a feature request, but the gap is source-verifiable. Static inspection of the current schema, docs, inbound admission path, reply dispatch path, and outbound resolver shows the requested policy cannot be expressed today.

## Solution Assessment

Unclear until a maintainer chooses the public config shape. The maintainable direction is a hard WhatsApp access-control policy in config/schema/runtime, not memory, prompts, or a silent-agent workaround.
