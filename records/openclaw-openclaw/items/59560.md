## Summary

Keep open. Current main still lacks a WhatsApp per-contact policy surface, and the newer related #73740 was closed as the narrower duplicate of this broader tracker. The code and docs still expose coarse `dmPolicy`/`allowFrom` access plus prompt/history maps, so the requested inbound, outbound, and autonomous-reply controls remain real work.

## What This Changes

The issue requests root-level and account-level WhatsApp per-contact policy rules for `allowInbound`, `allowOutbound`, and `allowAutonomousReply`, with account-level field overrides.

## Best Possible Solution

Design this as a WhatsApp-owned plugin/config change with a typed per-contact policy model, predictable root/account merging, enforcement before inbound side effects, outbound-target checks for explicit sends, autonomous-reply delivery guards, docs, generated config metadata, and focused tests.

## Reproduction Assessment

Not applicable as a feature request, but there is a high-confidence static verification path. Current schema and routing code show no per-contact inbound/outbound/autonomous policy, and allowed inbound DMs proceed into normal handling.

## Solution Assessment

Unclear until a maintainer chooses the public config shape. The maintainable direction is a hard WhatsApp access-control policy in config/schema/runtime, not memory, prompts, or the dummy/silent-agent workaround described in the related discussion.
