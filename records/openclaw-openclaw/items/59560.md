## Summary

Keep open. Current main still lacks the requested WhatsApp per-contact policy surface: the schema, docs, inbound gate, outbound target resolver, and auto-reply delivery path all remain driven by global/account `dmPolicy`, `allowFrom`, `direct`, and `dms` settings. Related #73740 overlaps on an outbound-only subset, but it remains open and is narrower than this request.

## What This Changes

Add root- and account-scoped WhatsApp per-contact policy controls for inbound admission, explicit outbound sends, and autonomous replies, with account-level per-field overrides.

## Best Possible Solution

Implement this in the WhatsApp plugin, not core: add a typed contact-policy config/schema, normalize E.164 keys, merge root and account policy per field, enforce `allowInbound` before enqueue/session handling, enforce `allowOutbound` in outbound target resolution, enforce `allowAutonomousReply` before reply delivery, and document/test precedence and the #73740 outbound-only subset.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: the proposed config keys are absent from the strict WhatsApp schema, and the current runtime only consults `dmPolicy`, `allowFrom`, and pairing for the relevant inbound/outbound paths.

## Solution Assessment

Unclear pending maintainer confirmation. The requested WhatsApp-owned policy shape is plausible and narrower than a core change, but the public field names, defaults, and account-level merge semantics need product/API approval before coding.
