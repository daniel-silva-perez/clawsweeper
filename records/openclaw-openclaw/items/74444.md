## Summary

Keep open. Current main still lacks the requested Jina task configuration, and the PR is a useful implementation candidate for the open related feature request, but the proposed patch is not ready because the main OpenAI inline embedding path still would not send `task` and the config schema artifacts are not kept in sync.

## What This Changes

The PR adds optional `queryTask` and `documentTask` memory-search config fields, attempts to propagate them through memory embedding provider options/cache/docs, and updates OpenAI-compatible embedding request handling and documentation.

## Best Possible Solution

Land a rebased, tested PR that preserves current `input_type` behavior while adding task configuration through config types, strict schema, help/labels/generated schema, resolved memory settings, memory-core provider requests, OpenAI inline and batch request bodies, cache identity, and docs verified against current Jina model/API documentation.

## Reproduction Assessment

Yes. Configure an OpenAI memory embedding provider with `queryTask`/`documentTask` and exercise `createOpenAiEmbeddingProvider().provider.embedQuery()` or non-batch `embedBatch()`; static inspection shows the resulting `fetchRemoteEmbeddingVectors` body would not include `task`.

## Solution Assessment

No. The typed `queryTask`/`documentTask` direction is a maintainable shape, but this patch is incomplete until the inline OpenAI body, generated config schema/help, current-main `input_type` merge, docs, and tests are fixed together.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Forward task in inline OpenAI embeddings:** `extensions/openai/embedding-provider.ts:16-17`
  - body: The patch copies `queryTask` and `documentTask` onto the OpenAI client, but the inline provider still builds request bodies with only `model`, `input`, and optional `input_type`. Memory search queries always use `embedQuery`, and non-batch indexing uses `embedBatch`, so `queryTask` would never reach Jina and `documentTask` would only work in the separate batch adapter. Add task selection to this body path and cover query/document tests.
  - confidence: 0.94
- **[P2] Regenerate the public config schema artifacts:** `src/config/zod-schema.agent-runtime.ts:691-692`
  - body: Adding strict `memorySearch` keys changes the computed config schema, but the PR does not update `schema.help.ts`, `schema.labels.ts`, `schema.base.generated.ts`, or the generated docs baseline. The existing generated-schema check compares the computed payload to the committed generated file, so this config surface will drift and the UI/help schema will not expose the new fields.
  - confidence: 0.9
- **[P2] Correct the Jina v5 docs examples:** `docs/providers/openai.md:207-209`
  - body: The new docs section is titled for Jina v5 but names `jina-embeddings-v3`/`v4` and uses v3/v4-style `retrieval.query` and `retrieval.passage` examples. Official v5 pages identify the v5 text small/nano models and their adapter names separately, so the published docs would guide users to the wrong values for the advertised feature.
  - confidence: 0.82
