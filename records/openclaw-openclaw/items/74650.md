## Summary

Keep this open. Current main still leaves the `openclaw acp` GatewayClient on the default device identity path, while adjacent fixes only covered backend/native approval clients, so the reported ACP/ACPX approval loop remains a plausible source-level bug rather than an implemented fix.

## What This Changes

Fix ACP runtime and direct ACPX `openclaw` launches so approving one ACP scope request allows initialization instead of repeating scope-upgrade approvals.

## Best Possible Solution

Add a narrow, shared loopback shared-auth device-identity omission path for trusted local backend bridges that includes `openclaw acp`, while preserving device identity requirements for remote gateways, browser/control-ui clients, unauthenticated clients, and device-token-only clients.

## Reproduction Assessment

Unclear: the report has credible symptoms but only one reproduction step and no logs. The source-backed repro path is to seed a lower-scope paired device identity, run `openclaw acp` or `acpx openclaw` against a loopback token/password gateway, and verify it still requests a scope upgrade before the bridge initializes.

## Solution Assessment

Yes, the best repair direction is narrow: extend or extract the existing loopback shared-auth exemption pattern to the ACP bridge and cover it with focused tests. A broad startup migration or relaxed remote pairing policy would be riskier than needed.

Observable bug: after updating to 2026.4.26, ACP runtime launches and direct `acpx openclaw` can get stuck in repeated `scope upgrade pending approval` loops because the `openclaw acp` bridge attaches the persisted default device identity when connecting to a local token/password-backed gateway. Expected boundary: keep the fix limited to trusted local loopback shared-auth ACP bridge clients. Add or extend an integration/poc test if practical to seed a lower-scope paired identity and prove `openclaw acp` no longer creates a scope-upgrade request. Add a user-facing `CHANGELOG.md` fix entry with reporter credit if available; do not change ACPX dependency versions or install policy.
