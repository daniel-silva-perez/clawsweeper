## Summary

Keep this open. Current main has adjacent fixes for stale gateway pairing baselines, including loopback shared-auth exemptions for backend RPC calls and native approval clients, but the ACP bridge path still constructs a GatewayClient without that exemption, so the reported ACP/ACPX approval loop remains plausible and not fully covered by current tests.

## What This Changes

The issue reports ACP runtime and direct ACPX OpenClaw launches getting stuck in repeated gateway scope-upgrade approval after updating to OpenClaw 2026.4.26.

## Best Possible Solution

Unify the local loopback shared-auth device-identity bypass across gateway-backed local clients that are acting as trusted backend bridges, including `openclaw acp`, while preserving device identity requirements for remote gateways and unauthenticated/device-token-only clients.

## Reproduction Assessment

Unclear. The report gives strong observed symptoms but only one setup step; the source-backed reproduction path is to seed a lower-scope paired default device identity, then start `openclaw acp` against a loopback token/password gateway and verify it still requests a scope upgrade.

## Solution Assessment

No. The current-main fix in #74472 is a good adjacent pattern, but it stops at native approval clients; the narrower complete solution is to extend or extract that loopback shared-auth exemption for the ACP bridge path with focused regression coverage.
