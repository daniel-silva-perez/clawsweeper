## Summary

Keep open. Current main still has the ACP bridge connecting as a CLI client with the gateway client’s default operator-admin scope request, while the gateway scope-upgrade path remains an explicit pairing approval flow; I did not find a current-main fix or regression coverage for the reported ACP/acpx approval loop clearing after UI approval.

## What This Changes

The issue reports ACP runtime and direct ACPX OpenClaw launches on 2026.4.26 getting stuck in repeated gateway scope-upgrade approval requests after the operator approves them.

## Best Possible Solution

Add a focused regression for the ACP/acpx OpenClaw bridge after a scope-upgrade approval, then fix the narrowest ACP bridge, gateway-client scope selection, or pairing-token handoff path that prevents the approved request from being honored, while preserving manual approval for broader operator scopes.

## Reproduction Assessment

Unclear. The report gives concrete failing surfaces and request IDs, and current source explains a plausible scope-upgrade path, but the provided steps and existing tests do not yet give a high-confidence end-to-end reproduction of approval not sticking for ACP/acpx.

## Solution Assessment

Unclear. The likely fix should be narrow, but choosing between changing ACP bridge client identity/scopes and changing pairing approval retry behavior needs maintainer/security judgment because this is an operator-scope authorization boundary.
