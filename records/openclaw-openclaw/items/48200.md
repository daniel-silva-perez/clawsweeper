## Summary

Keep this PR open. Current main still lacks the proposed configured own-account iMessage handle guard, but the submitted patch is not merge-ready because it adds a public strict config key without updating generated config metadata, UI hints, or user-facing docs.

## What This Changes

The PR adds an optional iMessage `accountHandle` config field, startup allowlist validation, per-message own-sender drops, loop-rate accounting for those drops, and focused iMessage monitor tests.

## Best Possible Solution

Refresh this iMessage-plugin-owned fix so the guard stays optional and backward-compatible, then expose the new config key consistently across schema, exported types, generated metadata, UI hints, docs, and focused regression tests for normalization, access-ordering, and loop-rate behavior.

## Reproduction Assessment

Yes. A high-confidence unit-level reproduction is an iMessage inbound payload with `sender` equal to the local account handle, `is_from_me: false`, `dmPolicy: "open"`, and no echo-cache match; current main has no configured own-handle check before access routing.

## Solution Assessment

No, not as submitted. The account-handle guard is a narrow plausible fix, but the mergeable form needs config metadata/docs/UI follow-through and a branch refresh against current main.
