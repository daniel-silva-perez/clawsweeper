## Summary

Keep this PR open. Current main still lacks the requested iMessage own-account handle guard, so the work is not obsolete, but the submitted patch is incomplete for merge because it adds a public strict config key without updating generated channel metadata, UI hints, or user-facing docs, and the branch needs refresh against current main.

## What This Changes

The PR adds an optional iMessage `accountHandle` config field, startup allowlist validation, per-message own-sender drops, loop-rate accounting for those drops, and focused iMessage monitor tests.

## Best Possible Solution

Refresh and review an iMessage-plugin-owned implementation that keeps the guard optional and backward-compatible, exposes the new config key consistently across schema, exported types, generated metadata, UI hints, and docs, and preserves regression coverage for normalization, ordering before access routing, loop-rate behavior, and unchanged behavior when unset.

## Reproduction Assessment

Yes. A high-confidence unit-level reproduction is an iMessage inbound payload whose `sender` is the local account handle and `is_from_me` is false; current main has no configured own-handle check before `dmPolicy`/access routing, so that payload can dispatch unless another echo/reflection guard happens to match.

## Solution Assessment

No, not as submitted. The account-handle guard is a plausible narrow iMessage-owned fix, but adding a strict public config key without metadata/docs/UI follow-through and branch refresh is not the best mergeable form.
