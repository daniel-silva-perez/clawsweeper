## Summary

Keep this PR open. Current main still lacks the PR's optional iMessage `accountHandle` config field, startup allowlist validation, pre-policy own-account sender drop, and account-handle loop-rate reason; the existing self-chat, echo-cache, from-me, and reflected-content guards are adjacent but do not implement this proposed protection.

## What This Changes

The PR adds an optional `channels.imessage.accountHandle` config field plus startup allowlist validation, per-message sender-handle filtering, loop-rate integration, and focused iMessage monitor tests to prevent own-account echo loops.

## Best Possible Solution

Keep this PR open and review or refresh it against current main. The best implementation remains iMessage-plugin-owned: add an optional own-account handle to the iMessage config type/schema and public config metadata/docs, validate it against the effective config/store allowlists at monitor startup, pass a normalized value into inbound decision processing, drop matching senders before `dmPolicy`/allowlist routing, feed that drop into self-chat/loop-rate protection, and keep focused regression tests for normalization, ordering, and unchanged behavior when the field is absent.
