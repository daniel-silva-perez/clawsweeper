## Summary

Keep this issue open. Current main still receives Discord reaction add/remove events through dedicated Discord reaction listeners, but the path only enqueues session-scoped system events. The public plugin hook names, internal hook event docs/types, and tests still expose message hooks without a reaction hook surface. Related Discord reaction hook reports #52658 and #60805 were closed toward this canonical gap, while #17840 separately tracks reaction-triggered agent turns.

## Best Possible Solution

Keep this issue open as the canonical Discord reaction hook/API tracker. The best implementation is an additive, documented reaction event surface that fires after existing Discord reaction authorization/filtering and route resolution, carries structured reaction metadata, updates plugin and internal hook docs/types/tests, and leaves reaction-triggered agent turns to #17840.
