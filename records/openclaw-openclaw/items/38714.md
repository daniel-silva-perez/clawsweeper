## Summary

Keep open. Current main still ingests Discord reaction add/remove events and stores them as session-scoped system events, but it does not expose a reaction-specific internal hook or typed plugin hook, and the message_received bridge still lives in the normal inbound message dispatch path. Related #11747 tracks wake/auto-response behavior, while #52658 and #60805 are overlapping closed reports rather than a shipped implementation.

## Best Possible Solution

Keep this issue open until maintainers define the reaction event contract. The likely implementation belongs in the Discord reaction handler plus the internal/plugin hook surfaces, with a structured payload for action, emoji, reactor, target message, account/guild/channel/thread/session metadata, and removal state. Add Discord listener tests, hook contract tests, docs, and keep wake/auto-response behavior coordinated separately with #11747.
