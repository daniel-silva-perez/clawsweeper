## Summary

Keep this PR open: current main still lacks the requested Telegram `/new` and `/reset` control-lane bypass, but the PR’s reset predicate is over-broad and would also route `/reset soft` around the normal per-topic lane. That subcommand is documented and implemented as an in-place session reset, so it should remain serialized with the active turn.

## What This Changes

This PR factors Telegram command-lane resolution and routes registered `/new` and `/reset` commands to `telegram:<chatId>:control`, with sequential-key tests for bare `/new`, bare `/reset`, and bot-username suffixed `/new`.

## Best Possible Solution

Keep the PR open and narrow the resolver so bare `/new` and hard `/reset` can bypass a busy Telegram topic while `/reset soft` remains serialized with the current session; add both sequential-key unit coverage and a busy-lane Telegram middleware regression.

## Reproduction Assessment

Yes. A source-level reproduction is clear: current main returns the normal `telegram:<chatId>` key for `/new` and `/reset`, while the PR would return `telegram:<chatId>:control`; the PR bug follows by passing `/reset soft`, which resolves to the registered `/reset` command because reset accepts arguments.

## Solution Assessment

No. The requested direction is reasonable, but the proposed predicate is not the best implementation because it matches all reset subcommands; the safer fix is to classify only hard session rollover commands as control-lane traffic and explicitly keep soft reset on the regular lane.
