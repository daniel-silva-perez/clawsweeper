## Summary

Keep open. Current main already covers the image-side resolver path, but PDF execution still uses the registry-only lookup that ignores `models.providers` overrides; this PR addresses real remaining behavior but overlaps the narrower open #65746 and still has a PDF correctness issue in its proposed guard.

## What This Changes

The PR exports `buildUnknownModelError`, removes the shared registry-only media resolver, routes image/PDF model lookup through `resolveModelWithRegistry`, and adds image/PDF regression tests for configured provider and fallback behavior.

## Best Possible Solution

Preserve the current image implementation on main, then land one canonical PDF resolver path that applies configured provider baseUrl/API/header overrides for registry-backed models and config-defined custom providers while preserving text-only PDF extraction fallback for registry, config, and plugin-dynamic models.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Let plugin-resolved PDF text models use fallback:** `src/agents/tools/pdf-tool.ts:240`
  - body: `isKnownTextOnlyModel` only accepts text-only models declared in config or found in the registry. `resolveModelWithRegistry` can also return a plugin-dynamic text-only model, and this new guard now throws before the existing text-extraction fallback runs, so PDFs with images plus extractable text can fail instead of returning a text-only summary. Use the resolved model capability to distinguish real text-only models from synthetic unresolved fallbacks.
  - confidence: 0.84
