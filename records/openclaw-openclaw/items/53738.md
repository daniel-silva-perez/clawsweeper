## Summary

Keep open: the underlying PDF resolver problem is still real on current main, but this PR is not ready to merge because it is dirty against main, overlaps the narrower open #65746, and introduces a PDF fallback regression for plugin-dynamic text-only models.

## What This Changes

The branch exports the unknown-model error helper, removes the shared registry-only media resolver, routes image and PDF model lookup through `resolveModelWithRegistry`, and adds image/PDF regression tests for configured provider and fallback behavior.

## Best Possible Solution

Preserve the current image implementation on main, then land one canonical PDF resolver path that applies configured provider overrides for registry-backed models and config-defined custom providers while preserving text-only extraction fallback for registry, config, and plugin-dynamic models.

## Review Findings

Overall correctness: patch is incorrect

Full review comments:

- **[P2] Let plugin-resolved PDF text models use fallback:** `src/agents/tools/pdf-tool.ts:240`
  - body: `isKnownTextOnlyModel` only accepts text-only models declared in config or found in the registry. `resolveModelWithRegistry` can also return a plugin-dynamic text-only model, and this new guard now throws before the existing text-extraction fallback runs, so PDFs with images plus extractable text can fail instead of returning a text-only summary. Use the resolved model capability to distinguish real text-only models from synthetic unresolved fallbacks.
