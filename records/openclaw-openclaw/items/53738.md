## Summary

Keep open: current main still has the PDF custom-provider resolver gap, and this PR remains useful but should not merge as-is because the PDF guard can skip the documented text fallback for plugin-dynamic text-only models while the same area also overlaps open #65746.

## What This Changes

The PR exports the unknown-model error helper, removes the shared registry-only media resolver, routes image and PDF model lookup through `resolveModelWithRegistry`, and adds image/PDF regression tests for configured provider and fallback behavior.

## Best Possible Solution

Land one canonical PDF model-resolution path that uses config-aware provider/model matching for registry-backed and inline custom provider models, keeps the current image implementation on main, and preserves PDF extraction fallback for resolved text-only models regardless of whether they come from registry, config, or plugin-dynamic resolution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Let plugin-resolved PDF text models use fallback:** `src/agents/tools/pdf-tool.ts:240`
  - body: `isKnownTextOnlyModel` only checks configured models and registry entries. When `resolveModelWithRegistry` returns a plugin-dynamic text-only model, this new throw runs before the existing PDF text-extraction fallback, so PDFs with images plus extractable text can fail instead of returning a text-only summary. Treat the resolved model's own text-only capability as sufficient while still rejecting synthesized unresolved fallbacks.
  - confidence: 0.84
