## Summary

Keep open. Current main already covers the image-side resolver behavior, but the PDF execution path still uses the registry-only lookup, and this PR’s current PDF guard can reject plugin-dynamic text-only models before the existing extraction fallback. The remaining PDF bug overlaps open #65746, and the branch is currently unmergeable, so this needs maintainer choice rather than cleanup closure.

## What This Changes

The PR exports the unknown-model error helper, removes the shared registry-only media resolver, routes image and PDF model lookup through `resolveModelWithRegistry`, and adds image/PDF regression tests for configured-provider and fallback behavior.

## Best Possible Solution

Preserve the current image implementation on main, then land one canonical PDF resolver path that honors configured provider overrides for registry-backed, config-defined, and plugin-resolved models while preserving native PDF handling and text-extraction fallback semantics.

## Reproduction Assessment

Yes. #65742 gives concrete config steps and real-runtime evidence for `local-openai/gpt-5.4-mini`, and current main still has the registry-only PDF execution call that explains the `Unknown model` failure. The PR finding is reproducible by a plugin-dynamic text-only model with a PDF containing images plus extractable text.

## Solution Assessment

No. The direction is valid, but this branch is stale/unmergeable and the added PDF guard is not the narrowest maintainable fix because it ignores plugin-dynamic resolved model capability; a rebased PDF-focused resolver change or a corrected resolved-model-source check is safer.
