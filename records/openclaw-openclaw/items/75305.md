## Summary

Keep open. Current main still routes the pre-compaction memory flush as a runtime-only turn with an empty visible prompt, and existing tests encode that empty `transcriptPrompt`/blank prompt behavior. The related reports show pre-compaction memory flush exists and prior token-threshold issues were handled, but they do not cover this Anthropic prompt-serialization regression.

## What This Changes

The issue requests fixing pre-compaction memory flush so Anthropic receives provider-valid non-empty user content instead of resetting Telegram sessions during auto-compaction.

## Best Possible Solution

Keep the memory-flush instructions hidden/silent, but make the provider-facing turn non-empty and valid for Anthropic, then add regression coverage that exercises the memory-flush handoff through the embedded attempt path.

## Reproduction Assessment

Yes for the OpenClaw-side failure path: a memory flush call currently produces `transcriptPrompt: ""`, `resolveRuntimeContextPromptParts` turns that into `prompt: ""`, and the runtime-only branch submits it. The exact live Anthropic 400 should be confirmed by a targeted provider/request-shape regression test or live probe when implementing.

## Solution Assessment

No; the current runtime-only blank prompt is not the best provider-facing shape because some providers reject empty user content. A safer fix is a non-empty housekeeping prompt or sentinel that preserves hidden memory instructions and silent transcript behavior.
