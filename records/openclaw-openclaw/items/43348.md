## Summary

Keep this PR open. Current main still does not implement the proposed `acp_send` agent tool, `sessions_cancel` agent tool, or `sessions.cancel` Gateway RPC. Main has adjacent ACP/session controls (`sessions_send`, `sessions.abort`, `/acp cancel`, and `AcpSessionManager.cancelSession()`), but those do not fully replace the PR’s intended agent-facing surfaces. The PR also remains authorization/delivery sensitive based on the provided review comments and latest diff shape.

## Best Possible Solution

Keep this PR open as a rework candidate. Rebase it onto current main, decide whether a separate ACP-only `acp_send` tool is still needed beyond current `sessions_send` behavior, and if agent-facing ACP cancellation is still desired, implement it through the existing session-resolution and ACP-control model with visibility/ownership checks, ACP-only target validation, protocol/tool-catalog/docs updates, and focused regression tests.

## Risks / Open Questions

- As proposed, an agent-visible cancel path must not accept a raw learned session key without reusing session resolution, visibility, ownership, sandbox, and A2A policy checks.
- `acp_send` needs ACP-only target validation; sending fire-and-forget to a normal/subagent session can accept work without a callback delivery path and lose output.
- Adding `sessions.cancel` beside `sessions.abort`, `/acp cancel`, task cancellation, session reset/delete cleanup, and ACP runtime cancellation could fragment cancellation semantics unless protocol docs, scopes, catalog entries, and regression tests are aligned.
