## Summary

Keep this PR open. Current main still lacks the proposed `acp_send` tool, `sessions_cancel` tool, and `sessions.cancel` Gateway RPC. Main has adjacent ACP/session controls (`sessions_send`, `sessions.steer`, `sessions.abort`, `/acp cancel`, and `AcpSessionManager.cancelSession()`), but those do not implement the PR’s intended agent-facing ACP send/cancel surfaces. The PR also needs security-sensitive review before merge because it adds cross-session send/cancel control paths.

## Best Possible Solution

Keep this PR open as a rework candidate. Rebase it onto current main, decide whether a separate ACP-only `acp_send` is still needed now that `sessions_send` has parent-owned ACP handling, and if agent-facing ACP cancellation is still desired, implement it through the existing session-resolution and visibility model with ACP-only target validation, protocol/tool-catalog/docs updates, and regression tests covering authorization and delivery behavior. Resolve the security-sensitive review comments before merge.

## Risks / Open Questions

- `sessions_cancel` is a cross-session control surface; the PR needs session resolution, visibility, ownership, sandbox, and A2A policy checks before exposing cancellation to normal agent tools.
- `acp_send` should reject non-ACP targets before fire-and-forget dispatch, otherwise normal/subagent targets can accept work without an ACP callback delivery path.
- Adding `sessions.cancel` beside `sessions.abort`, `sessions.steer`, `/acp cancel`, and ACP runtime cancellation can fragment cancellation semantics unless protocol docs, tool catalog entries, and focused tests are aligned.
