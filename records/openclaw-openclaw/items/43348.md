## Summary

Current main still does not implement the proposed `acp_send` agent tool, `sessions_cancel` agent tool, or `sessions.cancel` Gateway RPC. Main has partial overlap through `sessions_send`, `sessions.abort`, `/acp cancel`, and ACP task cancellation, but that does not make the PR obsolete. The PR remains a plausible implementation candidate, but it needs rework around authorization, ACP-only target validation, and API semantics before merge.

## Best Possible Solution

Keep this PR open as a rework candidate. Rebase onto current main, decide whether a separate ACP-only `acp_send` tool is still needed beyond the current `sessions_send` parent-owned ACP skip, and if an agent cancellation tool is desired, implement it through the existing `sessions.abort`/ACP cancellation model or a clearly documented new RPC with session resolution, visibility and ownership checks, ACP-only target validation, protocol/docs/tool-catalog updates, and focused regression tests.

## Risks / Open Questions

- The proposed `sessions_cancel` path is security-sensitive: as shown in the PR diff and review comments, it can cancel an ACP turn by raw session key unless it reuses current session resolution, visibility, ownership, and A2A policy checks.
- The proposed `acp_send` path can silently drop output for non-ACP targets because it dispatches fire-and-forget while relying on ACP callback delivery; it should fail fast unless the resolved target is an ACP session with a callback path.
- Adding `sessions.cancel` beside shipped `sessions.abort`, `/acp cancel`, task cancellation, and fast abort paths could fragment cancellation semantics unless naming, scope, docs, and behavior are reconciled.
- Security/supply-chain pass found no dependency, lockfile, CI workflow, package-publishing, lifecycle-script, or downloaded-artifact changes in the provided PR file list; the main concern is Gateway/session-control authorization.
