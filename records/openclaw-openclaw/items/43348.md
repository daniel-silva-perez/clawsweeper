## Summary

Keep open. Current main at 189c91eae6e1f4f280179961bca2c4153cf42e86 still does not implement this PR's intended agent-facing `acp_send` tool, `sessions_cancel` tool, or `sessions.cancel` Gateway RPC. Main has related ACP/session control paths (`sessions_send` fire-and-forget, owned ACP-child A2A skipping, `sessions.steer`, `sessions.abort`, `/acp cancel`, task cancellation, and `AcpSessionManager.cancelSession()`), but those do not fully replace the proposed surfaces. The PR also remains security-sensitive based on the provided review discussion: cancellation needs session visibility/ownership checks, and ACP fire-and-forget sending needs ACP-only target validation before merge.

## Best Possible Solution

Keep this PR open as a rework candidate. Rebase it onto current main, decide whether a separate `acp_send` tool is still needed beyond current `sessions_send` behavior, and if agent-facing ACP cancellation is still desired, implement it through the existing `sessions.abort`/ACP cancellation model or a clearly documented new RPC with session resolution, visibility/ownership checks, ACP-only target validation, protocol/tool-catalog/docs updates, and focused regression tests.

## Risks / Open Questions

- The proposed `sessions_cancel` path is authorization-sensitive: an agent-visible tool must reuse session resolution, visibility, ownership, and A2A policy checks before it can cancel work by session key.
- The proposed `acp_send` path must validate that the resolved target is an ACP session before fire-and-forget dispatch; otherwise normal/subagent targets can accept a run whose reply has no ACP callback path.
- Adding `sessions.cancel` beside current `sessions.abort`, `sessions.steer`, `/acp cancel`, task cancellation, and ACP runtime cancellation could fragment cancellation semantics unless protocol docs, scopes, tool catalog entries, and regression tests are reconciled.
