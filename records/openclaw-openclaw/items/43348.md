## Summary

Keep this PR open. Current main still does not provide the proposed `acp_send` agent tool, `sessions_cancel` agent tool, or `sessions.cancel` Gateway RPC. Main has adjacent ACP/session behavior (`sessions_send`, `sessions.steer`, `sessions.abort`, `/acp cancel`, and `AcpSessionManager.cancelSession()`), but those do not fully replace the PRâ€™s intended agent-facing surfaces. The PR also remains security-sensitive because its cancel/send paths affect cross-session control and delivery semantics.

## Best Possible Solution

Keep this PR open as a rework candidate. Rebase it onto current main, decide whether a separate ACP-only `acp_send` is still needed now that `sessions_send` has parent-owned ACP handling, and if agent-facing ACP cancellation is still desired, implement it through the existing session-resolution and visibility model with ACP-only target validation, protocol/tool-catalog/docs updates, and focused regression tests. The security-sensitive review comments should be resolved before merge.

## Risks / Open Questions

- `sessions_cancel` is a cross-session control surface; the PR needs session resolution, visibility, ownership, sandbox, and A2A policy checks before exposing cancellation to normal agent tools.
- `acp_send` should reject non-ACP targets before fire-and-forget dispatch, otherwise output from normal/subagent targets can be accepted without an ACP callback path.
- Adding `sessions.cancel` alongside `sessions.abort`, `sessions.steer`, `/acp cancel`, and ACP runtime cancellation can fragment cancellation semantics unless protocol docs, tool catalog entries, and tests are aligned.
