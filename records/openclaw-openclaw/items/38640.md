## Summary

Keep this PR open. Current main still does not implement the proposed prompt-level LLM retry wrapper with exponential backoff; it has adjacent failover, auth-profile rotation, SDK retry-wait caps, and cron/channel retry infrastructure, but not bounded same-call prompt retry for transient LLM errors.

## Best Possible Solution

Keep this PR open for maintainer design judgment. The best path is to fold the useful pieces into the configurable LLM/provider retry direction tracked by #49376/#61443, or rework this PR at the chosen runner/provider boundary with abortable backoff, shared failover-status parsing, explicit interaction with SDK retries and model fallback, and focused tests for retry classification, retry-after parsing, cancellation, and token/fallback behavior.

## Risks / Open Questions

- This PR overlaps with #49376 and #61443; maintainers should choose one canonical retry boundary before merging competing designs.
- Prompt-level retries can interact badly with provider SDK retries, auth-profile cooldowns, model fallback, token accounting, and run cancellation.
- The proposed PR has review feedback around abortable retry sleeps; without that, cancellation or timeout could still allow later provider calls.
