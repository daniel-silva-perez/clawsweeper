## Summary

Keep open. The cron diff preview addresses a real gap that is not present on current main, but the PR is not merge-ready because it regresses existing `cron edit --tools` parsing, has at least one inaccurate preview path, and carries an unrelated stale Control UI change.

## What This Changes

The PR adds a stderr before/after diff preview for `cron edit` with cron preview tests, while also changing cron `--tools` parsing and modifying Control UI pairing-required error formatting.

## Best Possible Solution

Keep the cron-only diff preview idea, but rework the branch so it reuses the shared cron tools parser, aligns preview output with `applyJobPatch` including validation-before-clear cases, and drops unrelated Control UI changes before maintainer review.

## Reproduction Assessment

Yes. The tools regression is reproducible from the existing current-main unit scenario for `cron edit --tools "exec read write"`, and the delivery-preview mismatch is provable by reading the `applyJobPatch` validation order. I did not run tests because this review was explicitly read-only.

## Solution Assessment

No as-is. The UX direction is good, but the maintainable fix should preserve existing CLI parsing contracts and either reuse service patch semantics or simulate them closely enough that preview output cannot contradict the actual update path.

## Review Findings

Full review comments:

- **[P2] Reuse the shared tools parser:** `src/cli/cron-cli/register.cron-edit.ts:448`
  - body: This comma-only split regresses the existing `cron edit --tools "exec read write"` contract: current main uses `parseCronToolsAllow`, which splits on comma or whitespace and drops empty token lists. With this change, quoted PowerShell-style input becomes one tool id, and separator-only input can persist `toolsAllow: []`, changing the job to allow no tools. Please keep using the shared parser for edit as well as add.
  - confidence: 0.94
- **[P2] Honor the failureDestination guard in the preview:** `src/cli/cron-cli/register.cron-edit.ts:232-240`
  - body: This block says non-webhook delivery will be cleared whenever the effective session is `main`, but `applyJobPatch` first throws if that delivery has `failureDestination`. For `cron edit <id> --session main` on such a job, the preview reports a clear that never persists because the update is rejected. Please gate this preview branch on the same guard or simulate the service update order.
  - confidence: 0.87
- **[P2] Drop the stale Control UI formatter change:** `ui/src/ui/connect-error.ts:66`
  - body: This cron PR also changes Control UI pairing-required formatting, but current main already has a richer protocol-backed formatter that preserves role/scope upgrade details. Bringing this older local formatter forward would regress that current behavior and mixes an unrelated UI change into the cron edit feature.
  - confidence: 0.78
