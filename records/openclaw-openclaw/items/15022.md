## Summary

Keep open. Current main still does not implement opt-in coalescing/final-only delivery for assistant text blocks interleaved with tool calls: final payload assembly preserves multiple assistant text entries, dispatch sends each final reply separately, and the block reply path still flushes before tool execution. The closed prior PR #16275 remains useful prior art but was not merged.

## Best Possible Solution

Keep this issue open as the focused core reply-pipeline feature. The best implementation is an opt-in coalescing or final-only delivery mode that covers both final embedded-run assistantTexts and block-streamed text around sequential and parallel tool execution, likely using hold/defer semantics similar to #16275, plus config schema/docs and regression tests for text -> tool -> text -> tool -> text turns.
