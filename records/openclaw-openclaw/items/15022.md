## Summary

Keep open. Current main still does not implement the requested opt-in coalescing/final-only delivery for assistant text blocks interleaved with tool calls. The related PR #16275 remains useful prior art, but it was closed unmerged and its hold/resume API is absent from the current reply pipeline.

## Best Possible Solution

Keep this issue open as the focused core reply-pipeline feature. The best implementation is an opt-in coalescing or final-only delivery mode that covers both final embedded-run assistantTexts and block-streamed text around sequential and parallel tool execution, likely using hold/defer semantics similar to PR #16275, plus config schema/docs and regression tests for text -> tool -> text -> tool -> text turns.
