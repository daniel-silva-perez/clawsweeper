## Summary

Keep open. Current main still preserves tool-interleaved assistant text as multiple outbound replies: final payload assembly keeps each assistant text block separate, dispatch sends each reply item separately, and the block-streamed path still flushes before tool execution. The closed related PR #16275 is useful prior art but was not merged.

## Best Possible Solution

Keep this issue open as the focused core reply-pipeline feature. The best implementation is an opt-in coalescing or final-only delivery mode that covers both final embedded-run assistantTexts and block-streamed text around sequential and parallel tool execution, likely using hold/defer semantics similar to #16275, plus config schema/docs and regression tests for text -> tool -> text -> tool -> text turns.
