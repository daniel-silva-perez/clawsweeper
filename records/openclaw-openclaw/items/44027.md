## Summary

Keep open. The underlying bug is still real on current main for ordinary group chats, and the DingTalk AI Card context shows why `kind="tool"` delivery matters, but this PR branch is stale against the current dispatch gate and would regress Slack non-DM suppression, Telegram forum-topic delivery, and native direct tool-summary delivery.

## What This Changes

The PR imports `normalizeVerboseLevel` and changes `dispatch-from-config.ts` so stored `/verbose on` or `/verbose full` can bypass group-chat tool-summary suppression while retaining native command-source suppression.

## Best Possible Solution

Rebase or replace the branch with a narrow change to the current shared gate: explicit stored verbose `on` or `full` should enable text tool summaries for ordinary non-Slack group chats, while preserving Slack non-DM suppression, Telegram forum-topic delivery, native direct delivery, exec approval delivery, media-only fallback, and preview-progress suppression.

## Reproduction Assessment

Yes. A focused unit test can seed a group session entry with `verboseLevel: "on"`, emit `opts.onToolResult({ text: "..." })`, and assert `dispatcher.sendToolResult` receives the text payload; current main already has the negative group-suppression test but lacks this positive case.

## Solution Assessment

No. The current PR condition is not the best way to solve it because it replaces current main's richer surface gate with group/native-only logic; the safer fix is to layer verbose opt-in into the existing gate and add focused regression coverage.
