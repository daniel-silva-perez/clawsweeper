## Summary

Keep open. The reported behavior is still a valid gap on current main, but this PR is stale against the current dispatch gate and would regress existing Slack non-DM suppression, Telegram forum-topic delivery, and native direct tool-summary delivery if merged as written. The DingTalk AI Card cross-reference supports the user need for `kind="tool"` delivery once a channel plugin handles tool results.

## What This Changes

The PR reads the stored `verboseLevel` in `dispatchReplyFromConfig` and lets `/verbose on` or `/verbose full` override group-chat tool-summary suppression while leaving native command-source sessions suppressed.

## Best Possible Solution

Layer explicit current session verbose `on` or `full` into the current `shouldSendVerboseProgressMessages`/tool-summary gate for ordinary non-Slack group chats, while preserving Slack non-DM suppression, Telegram forum-topic delivery, native direct delivery, exec approval/media-only fallback, preview-progress suppression, and focused regression coverage.

## Reproduction Assessment

Yes. A focused unit test can seed a normal group session entry with `verboseLevel: "on"`, emit `opts.onToolResult({ text: "..." })`, and assert `dispatcher.sendToolResult` receives the text payload; current main already has the negative group-suppression test but not this positive verbose override case.

## Solution Assessment

No. This PR is not the best way to solve the issue because it rebuilds the delivery decision from only group/native flags; the safer fix is to extend the current shared gate instead of replacing its Slack, forum, native, media, exec-approval, and preview-suppression behavior.
