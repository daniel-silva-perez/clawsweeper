## Summary

Keep open. The underlying bug still appears valid, and the linked DingTalk AI Card work confirms channel plugins need `kind="tool"` delivery, but this branch should not merge as-is because its stale group/native-only gate would regress current main's Slack non-DM suppression, Telegram forum-topic exception, and native direct tool-summary delivery.

## What This Changes

The PR imports `normalizeVerboseLevel` and changes `dispatch-from-config.ts` so stored `/verbose on` or `/verbose full` can allow group-chat tool-summary delivery while preserving the default group suppression.

## Best Possible Solution

Layer explicit stored verbose `on` or `full` onto current main's `shouldSendVerboseProgressMessages` gate so ordinary non-Slack group chats can receive text tool summaries after user opt-in, while preserving Slack non-DM suppression, Telegram forum-topic delivery, native direct delivery, exec approvals, media-only fallback, and preview tool-progress suppression.

## Reproduction Assessment

Yes. A focused unit test can set a group `MsgContext`, seed the session entry with `verboseLevel: "on"`, emit `opts.onToolResult({ text: "..." })`, and assert `dispatcher.sendToolResult` receives the text payload; current main lacks that positive case and still has the suppression path.

## Solution Assessment

No. The branch's current condition is not the best fix because it replaces current main's richer surface gate with a group/native-only gate; the safer fix is to extend the current gate with a narrow stored-verbose override and add regression tests for Slack non-DM, Telegram forum, native direct, group off, and group on/full behavior.
