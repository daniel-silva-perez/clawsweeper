## Summary

Keep open. Current main still remaps tar entry paths during backup creation but does not rewrite or avoid hardlink link targets, and backup verification still validates only entry paths. The reported macOS extraction failure remains a plausible, narrow backup correctness bug with clear implementation and test surfaces.

## What This Changes

The issue asks for backup archives to avoid or reject hardlink targets containing `..` so broad macOS tar extraction does not fail after `backup verify` succeeds.

## Best Possible Solution

Keep this open until backup creation emits restore-safe hardlink entries after archive path remapping and backup verification rejects unsafe tar link targets before reporting success. Treat configurable excludes and broader backup redesign as separate tracks.

## Reproduction Assessment

Yes. The issue includes a real macOS restore drill, and current source plus the pinned `tar` 7.5.13 behavior show a focused hardlink fixture can produce a separate `linkpath` containing `..` while `backup verify` still only inspects entry paths.

## Solution Assessment

No current solution is in place. The narrow maintainable fix is to rewrite hardlink `linkpath` values into archive-root-safe targets or avoid hardlink entries, then teach `backup verify` to inspect tar link targets with regression coverage.
