## Summary

Keep open. Current main still remaps tar entry paths during backup creation but does not remap tar hardlink targets, and backup verification still validates only archive entry paths. The reported “archive verifies but macOS broad extraction rejects a hardlink target containing '..'” remains plausible and actionable.

## Best Possible Solution

Implement a focused backup fix: make `openclaw backup create` emit restore-safe hardlink entries by rewriting hardlink targets into the archive namespace or dereferencing hardlinks when target rewriting is brittle, and extend `openclaw backup verify` to reject unsafe hardlink targets. Keep broader exclude-pattern and redesign work in #40786 and #67990.

## Clownfish Work Prompt

Fix the backup hardlink-target safety gap reported in this issue. Current `openclaw backup create` rewrites tar `entry.path` but not hardlink `entry.linkpath`, so archives can pass `openclaw backup verify` while macOS `bsdtar` rejects broad extraction with `Path contains '..'`.

Expected boundary:
- In `src/infra/backup-create.ts`, make hardlink entries restore-safe. Prefer rewriting hardlink targets into the same archive namespace as their target entry when the target is within planned backup sources; if that is too brittle, dereference hardlinks into regular file entries in a clear, tested way.
- In `src/commands/backup-verify.ts`, validate hardlink targets for tar `Link` entries and reject traversal, absolute, backslash, outside-root, or missing/unsafe targets as appropriate. Do not treat ordinary symlink payload link text as equivalent unless the implementation intentionally supports that with tests.
- Add focused tests that create real hardlinked payload files, inspect tar `entry.type`/`entry.linkpath`, and prove `backup verify` rejects an unsafe hardlink target fixture.
- Update docs only if user-facing verify semantics change, and add a changelog fix entry if this changes shipped behavior. Credit `Thanks @amzzzzzzz` if adding a changelog entry.

Do not implement general exclude rules, a restore command, service-aware backup redesign, dependency version changes, or broader backup source policy changes. Keep #40786 and #67990 out of scope.
