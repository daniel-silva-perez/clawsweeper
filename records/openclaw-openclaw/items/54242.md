## Summary

Keep open. Current main still has the reported backup hardlink-target gap: archive creation remaps tar entry names but not tar link targets, and verification validates only entry names. The issue is recent, concrete, not protected, and not covered by an open PR or a broader related backup proposal.

## What This Changes

The issue asks for `openclaw backup create` and `openclaw backup verify` to handle unsafe hardlink targets so archives do not verify successfully and then fail broad extraction on macOS tar.

## Best Possible Solution

Keep this open until backup creation emits restore-safe hardlink entries and backup verification validates tar link targets, with regression coverage for a hardlinked payload whose filesystem-relative link target would otherwise contain `..`. Keep configurable excludes and broader backup redesign in their existing follow-up threads.

## Repair Work Prompt

Fix #54242. Current main remaps tar entry paths for backup archives but leaves hardlink `linkpath` untouched, so macOS bsdtar can reject broad extraction when a hardlink target contains `..` even though `openclaw backup verify` passes. Implement a focused core backup fix: in `createBackupArchive`, either rewrite hardlink targets into the same archive namespace after path remapping or dereference hardlinks when target rewriting is brittle; in `backupVerifyCommand`, read `ReadEntry.linkpath` for link entries and reject absolute, backslash, traversal, or archive-root-escaping targets. Add regression tests for create+verify with hardlinked files whose original filesystem-relative target would contain `..`, plus a malformed archive whose unsafe linkpath verify rejects. Keep configurable excludes and backup restore/snapshot redesign out of scope. Do not change the `tar` dependency, archive root naming, absolute path encoding contract, or unrelated backup feature behavior. Update docs/changelog only for the user-facing backup safety fix; the changelog entry should credit `Thanks @amzzzzzzz`.
