## Summary

Keep open. Current main has the backup create/verify surface and a partial mitigation for installed plugin dependency trees, but it still does not normalize or validate hardlink link targets. The linked exclude-pattern issue is related but not a substitute for restore-safe archive entries, and the closed symlink duplicate issue is adjacent rather than canonical for this hardlink target bug.

## Best Possible Solution

Keep this issue open and fix the backup archive contract directly: either dereference/disable hardlink entries during backup creation or rewrite hardlink `linkpath` values into archive-root-safe targets, then extend `backup verify` to inspect `ReadEntry.linkpath` for hardlink entries and reject absolute, backslash, `.`/`..`, or outside-root targets. Add regression coverage with hardlinked files under a workspace dependency-tree shape. Treat #40786 as a useful size/noise follow-up, not the core safety fix.

## Evidence

- **Backup creation remaps entry paths only:** `createBackupArchive` calls `tar.c` with `portable: true` and `preservePaths: true`; its `onWriteEntry` callback rewrites `entry.path` through `remapArchiveEntryPath`, but there is no corresponding rewrite or validation of `entry.linkpath` for hardlink entries.
  - file: [src/infra/backup-create.ts:374](https://github.com/openclaw/openclaw/blob/189c91eae6e1f4f280179961bca2c4153cf42e86/src/infra/backup-create.ts#L374)
  - command: `nl -ba src/infra/backup-create.ts | sed -n '268,390p'`
  - sha: [189c91eae6e1](https://github.com/openclaw/openclaw/commit/189c91eae6e1f4f280179961bca2c4153cf42e86)
- **Verifier ignores link targets:** `backupVerifyCommand` builds its entry list from `entry.path` only and normalizes those paths; `ReadEntry.linkpath` is not collected, normalized, or checked, so a hardlink target containing `..` can pass verification if the entry path itself is safe.
  - file: [src/commands/backup-verify.ts:169](https://github.com/openclaw/openclaw/blob/189c91eae6e1f4f280179961bca2c4153cf42e86/src/commands/backup-verify.ts#L169)
  - command: `nl -ba src/commands/backup-verify.ts | sed -n '169,307p'`

- **Current docs describe path-only verification:** The backup docs say verification rejects traversal-style archive paths and checks manifest payloads. They do not claim restore-safe hardlink target validation, and the dependency-tree skip is limited to installed plugin `extensions/*/node_modules`, not arbitrary workspace `node_modules` like the reported source checkout path.
  - file: [docs/cli/backup.md:30](https://github.com/openclaw/openclaw/blob/189c91eae6e1f4f280179961bca2c4153cf42e86/docs/cli/backup.md#L30)

Upstream `node-tar` v7.5.13 sets hardlink `linkpath` with `path.relative(this.cwd, linkpath)` and writes that field into the header after `onWriteEntry`, which matches the reported `../...` hardlink target shape unless OpenClaw rewrites or disables hardlink entries.

## Risks / Open Questions

- Current `openclaw backup verify` can still overstate restore confidence for archives containing hardlink entries whose link targets are unsafe for common extractors.
- The installed-plugin `node_modules` skip reduces one source of backup bloat, but it does not address dependency trees inside configured workspaces, which is where the reporter found the offending esbuild hardlink.
