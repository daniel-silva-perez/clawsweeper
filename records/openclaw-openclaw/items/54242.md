## Summary

Keep open. Current main still remaps tar entry paths during backup creation but does not remap or dereference tar hardlink targets, and backup verification still validates only archive entry paths. The reported macOS bsdtar failure remains actionable, with no open PR found that references this issue and no related item that supersedes the narrow hardlink-target gap.

## What This Changes

Make backup archives and verification safe when tar hardlink targets contain traversal segments that common extraction tools reject.

## Best Possible Solution

Implement a focused backup fix: make `openclaw backup create` emit restore-safe hardlink entries by rewriting hardlink targets into the archive namespace or dereferencing hardlinks when target rewriting is brittle, and extend `openclaw backup verify` to reject unsafe hardlink targets. Keep configurable exclusions and broader backup redesign in #40786 and #67990.

## Repair Work Prompt

Fix the backup hardlink-target safety gap tracked by https://github.com/openclaw/openclaw/issues/54242. Current `openclaw backup create` remaps tar `entry.path` but leaves hardlink `entry.linkpath` untouched, so `tar` 7.5.13 can emit relative hardlink targets containing `..`; `openclaw backup verify` then validates only `entry.path` and can pass an archive that macOS bsdtar rejects on broad extraction. Implement a focused fix in the backup create/verify surface: either rewrite hardlink targets into the archive namespace when correctness is clear, or dereference hardlinks / emit file entries when link target rewriting is brittle. Extend verification to inspect `ReadEntry.linkpath` for hardlink entries and reject absolute, backslash, or traversal-style targets that common tar extractors reject. Add tests that create actual hardlinked payload files or otherwise exercise `entry.linkpath`, including a failing unsafe-target verify case and a create+verify restore-safe archive case. Do not broaden this into configurable exclude patterns or the full backup redesign; leave #40786 and #67990 separate. Add a user-facing changelog fix entry under the active version with `Thanks @amzzzzzzz`.
