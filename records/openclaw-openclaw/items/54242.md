## Summary

Keep open. Current main still remaps tar entry paths during backup creation but leaves hardlink targets untouched, while backup verification still validates only archive entry paths. The reported macOS broad-extraction failure remains a plausible, narrow backup correctness bug with clear implementation and regression-test surfaces.

## What This Changes

Ensure backup archives and verification handle hardlink link targets containing traversal segments so `openclaw backup create --verify` cannot pass an archive that macOS `bsdtar` rejects during broad extraction.

## Best Possible Solution

Keep this open until backup creation emits restore-safe hardlink entries after archive path remapping and backup verification rejects unsafe tar link targets before reporting success. Keep configurable excludes and broader backup redesign in their existing separate tracks.

## Reproduction Assessment

Yes. The issue includes a real macOS restore drill, and current source plus the pinned `tar` behavior show a focused hardlink fixture can produce a separate `linkpath` containing `..` while `backup verify` still inspects only entry paths.

## Solution Assessment

No current solution is in place. The narrow maintainable fix is to rewrite hardlink `linkpath` values into archive-root-safe targets or avoid hardlink entries, then teach `backup verify` to inspect tar link targets with regression coverage.
