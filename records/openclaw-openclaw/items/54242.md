## Summary

Keep open. Current main still remaps tar entry names during backup creation but does not rewrite or avoid tar hardlink targets, and backup verification still validates only entry paths. The related symlink duplicate-path fix and plugin dependency exclusion reduce nearby backup archive problems, but they do not cover the reported hardlink link-target safety gap.

## What This Changes

The issue requests restore-safe backup archive creation and verification when tar hardlink targets contain traversal-style `..` segments that macOS `bsdtar` rejects during broad extraction.

## Best Possible Solution

Update backup creation so hardlinked files are represented in a restore-safe way after path remapping, and update backup verification to reject unsafe tar link targets. Add focused regression coverage for hardlinked payloads whose generated link target would otherwise contain `..`, while leaving configurable excludes and broader backup redesign to their separate tracks.

## Repair Work Prompt

Fix the backup hardlink restore-safety bug described in this issue. Current `src/infra/backup-create.ts` remaps tar `entry.path` but leaves hardlink `entry.linkpath` untouched, and `src/commands/backup-verify.ts` validates only `ReadEntry.path`. Implement restore-safe hardlink handling: either rewrite hardlink targets to the matching remapped archive path or fall back to ordinary file entries when a hardlink target would contain traversal after remapping. Also make `backup verify` inspect tar link targets for `Link` entries and reject absolute, backslash, empty, or traversal-style targets that common extractors would reject. Keep the fix scoped to backup create/verify; do not implement configurable excludes or the service-aware backup redesign, and do not change the `tar` dependency. Add regression tests with a hardlinked payload whose current generated link target would contain `..`, plus a verify fixture with an unsafe hardlink `linkpath`. Update docs/changelog only if user-facing CLI behavior or messages change, and credit the reporter `@amzzzzzzz` in any changelog entry per repo policy.
