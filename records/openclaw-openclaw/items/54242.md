## Summary

Keep open. Current main still rewrites backup tar entry names without handling hardlink targets, and `backup verify` still validates only entry paths, so the reported macOS broad-extraction failure remains a plausible narrow backup correctness bug.

## What This Changes

Ensure backup archives do not verify successfully when they contain hardlink targets with `..` that common macOS tar extraction rejects.

## Best Possible Solution

Keep the issue open until backup creation either dereferences unsafe hardlinks or rewrites hardlink targets into restore-safe archive-relative paths, and backup verification rejects unsafe tar link targets with focused regression coverage. Broader configurable exclusions should stay in #40786/#67990 rather than being treated as the core fix here.

## Repair Work Prompt

Fix the backup hardlink target safety bug reported in https://github.com/openclaw/openclaw/issues/54242. In current main, `createBackupArchive` rewrites tar `entry.path` but not hardlink `entry.linkpath`, and `backupVerifyCommand` validates only entry names. Add a focused fix so backup archives either avoid unsafe hardlink entries by dereferencing them or rewrite hardlink targets into restore-safe archive-relative targets after path remapping. Also make `backup verify` inspect tar `ReadEntry.linkpath` for link entries and reject traversal/absolute/backslash-unsafe targets that common extractors would reject. Add regression coverage for a hardlinked payload that would otherwise produce a `linkpath` containing `..` while entry paths are safe. Keep broad configurable excludes and service-aware backup redesign out of scope; those remain in #40786 and #67990. Add a single `CHANGELOG.md` Unreleased Fixes entry crediting the reporter, and do not change dependency versions or package manager patches.
