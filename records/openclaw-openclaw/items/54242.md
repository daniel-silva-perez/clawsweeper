## Summary

Keep open. Current main still remaps tar entry names during backup creation but does not rewrite tar hardlink targets, and backup verification still validates only entry paths. The related symlink de-duplication fix and plugin dependency exclusion reduce some archive-shape problems, but they do not cover the reported hardlink `linkpath` safety gap.

## What This Changes

The issue asks backup creation or verification to handle tar hardlink targets containing `..` so archives that verify successfully can still be broadly extracted on macOS bsdtar.

## Best Possible Solution

Keep this open until backup creation emits restore-safe hardlink entries and backup verification rejects unsafe tar link targets, with regression coverage for a hardlinked payload whose generated link target would otherwise contain `..`. Keep configurable excludes in #40786 as a separate feature track.

## Repair Work Prompt

Fix the backup hardlink-target safety gap reported in openclaw/openclaw#54242. Current `createBackupArchive` remaps tar `entry.path` but leaves hardlink `entry.linkpath` untouched, and `backupVerifyCommand` validates only entry paths. Add a narrow fix so `openclaw backup create` emits restore-safe hardlink entries and `openclaw backup verify` rejects unsafe tar link targets containing traversal, absolute paths, or backslashes. Prefer a conservative implementation that either dereferences hardlinks or rewrites hardlink targets to valid archive-root-relative targets; do not add general exclude-pattern support, which belongs to #40786. Add regression coverage for a hardlinked workspace/dependency-like payload where the generated tar link target would otherwise contain `..`, and cover verify rejecting a crafted archive with an unsafe hardlink target. Update user-facing backup docs/changelog only if the CLI behavior or guarantee changes. Do not broaden backup restore scope.
