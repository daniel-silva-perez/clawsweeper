## Summary

Keep open. Current main still remaps tar entry paths during backup creation but does not rewrite or avoid tar hardlink targets, and backup verification still validates only entry paths, so the reported macOS broad-extraction failure remains plausible and uncovered.

## What This Changes

Ensure backup archives and verification handle tar hardlink targets with traversal segments so macOS bsdtar broad extraction does not fail after OpenClaw verification succeeds.

## Best Possible Solution

Keep this open until backup creation emits restore-safe hardlink entries after OpenClaw's archive path remapping and backup verification rejects unsafe tar link targets before reporting success. Treat configurable excludes and broader backup redesign as separate tracks.

Yes. The issue includes a real macOS restore drill, and current source plus `tar` 7.5.13 behavior show a focused synthetic fixture with hardlinked files can reproduce a `linkpath` containing `..` without needing the reporter's large archive.

## Solution Assessment

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/54242. Current `openclaw backup create` remaps tar entry paths but leaves tar hardlink `linkpath` values untouched, so archives can pass `backup verify` and still fail broad extraction on macOS bsdtar when a hardlink target contains `..`. Keep the fix scoped to backup create/verify safety: either rewrite hardlink link targets into restore-safe archive-relative targets after OpenClaw path remapping, or avoid hardlink entries by dereferencing/copying file contents when necessary. Also update `backup verify` so it inspects tar link targets and rejects traversal/absolute/backslash-unsafe link targets before reporting success. Add focused regression coverage for a hardlinked payload that would otherwise produce a `..` link target. Do not implement configurable excludes, a broad backup redesign, or a full restore command in this PR. Do not change tar dependency versions or package manager overrides. Add a changelog fix entry crediting `@amzzzzzzz` if behavior changes are user-visible.
