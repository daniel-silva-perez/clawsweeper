## Summary

Keep open. Current main still remaps backup tar entry paths without handling tar hardlink targets, and backup verification still validates only entry paths, so the reported macOS broad-extraction failure remains a plausible narrow backup correctness bug.

## What This Changes

The issue requests that `openclaw backup create` and/or `openclaw backup verify` handle tar hardlink targets containing `..` so archives that verify successfully also extract reliably with common macOS tar tooling.

## Best Possible Solution

Keep this open until backup creation emits restore-safe hardlink entries after archive path remapping, or avoids hardlink entries for backup payloads, and `backup verify` rejects unsafe tar link targets with focused regression coverage. Broader configurable excludes should remain in their separate design tracks.

## Reproduction Assessment

Yes. The issue includes real restore-drill evidence, and current source plus the pinned `tar` behavior support a focused fixture with two hardlinked files where the generated hardlink `linkpath` contains `..` while verification still inspects only entry paths.

## Solution Assessment

No current solution is in place. The narrow maintainable fix is to rewrite or dereference unsafe hardlink targets during archive creation and validate `ReadEntry.linkpath` during verification, rather than relying on broad excludes or a full backup redesign.

## Repair Work Prompt

Fix the backup hardlink target safety gap reported in https://github.com/openclaw/openclaw/issues/54242. Current `createBackupArchive` remaps tar entry paths but leaves tar `linkpath` untouched, and `backupVerifyCommand` validates only entry paths. Add a narrow fix so created backups do not contain hardlink targets with traversal-style `..` after archive path remapping, either by rewriting hardlink link targets into archive-root-safe paths or by avoiding hardlink entries for backup payloads when safe. Also teach backup verification to inspect tar `ReadEntry.linkpath` for hardlink entries and reject unsafe link targets before reporting success. Add focused regression coverage with a hardlinked payload whose generated tar link target would otherwise contain `..`. Keep broader configurable exclude behavior in #40786/#67990 out of scope, do not change unrelated backup defaults, and avoid core-wide tar/archive refactors unless needed for the focused fix. Changelog: add a user-facing fix entry with reporter credit if the behavior changes are user-visible.
