## Summary

Keep this PR open for maintainer review. Current main still returns `false` for malformed matched session-kill paths, while the PR merge ref makes that path a deterministic `400 invalid_request_error` before auth or session lookup; I found no actionable correctness or security findings.

## What This Changes

The branch updates the gateway session-kill HTTP path parser to distinguish non-matches from malformed matched session keys and adds a regression test for `POST /sessions/%zz/kill`.

## Best Possible Solution

Land the narrow handler and regression-test change after normal maintainer review, preserving valid session kill auth, ownership, kill, and RPC semantics while making malformed route-shaped requests fail at the API boundary.

## Reproduction Assessment

Yes. On current main, `/sessions/%zz/kill` matches the session-kill route shape, `decodeURIComponent()` throws in `resolveSessionKeyFromPath()`, and the handler returns `false`, so the request can continue to later HTTP stages.

## Solution Assessment

Yes. The discriminated path-resolution result is the narrowest maintainable fix, aligns session kill with the session-history invalid-request behavior, and avoids auth/session/kill side effects for malformed keys.
