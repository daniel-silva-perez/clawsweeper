## Summary

Keep open. Current main still closes the long-lived WhatsApp inbound listener by calling the raw WebSocket close path, and the shared WhatsApp close helper also still does the same. The shipped Baileys socket API for the pinned dependency exposes `end(error)` and its implementation performs the broader teardown described in the report, so the requested fix has not landed. The linked fix PR was closed unmerged, and existing tests still assert `ws.close()` rather than the Baileys `end()` path.

## Best Possible Solution

Implement the teardown in the WhatsApp plugin by widening the socket close helper type to include `end(error?: Error)`, preferring `sock.end(new Error('OpenClaw listener close'))` when available and falling back to `sock.ws?.close?.()` for compatibility. Update the inbound listener and shared `closeWaSocket` path coherently, then adjust the WhatsApp mocks/tests to assert `end()` for Baileys-capable sockets plus fallback behavior for older shapes. A smaller follow-up can separately audit the short-lived login/QR raw-close paths.
