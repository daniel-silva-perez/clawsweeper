## Summary

Keep this issue open. Current main still closes the long-lived WhatsApp inbound listener through `sock.ws?.close?.()` instead of the Baileys `sock.end(error)` teardown path, and the shared WhatsApp close helper still has the same raw-close behavior. The same-author fix PR was closed unmerged, and current tests still assert raw WebSocket closing rather than Baileys cleanup.

## Best Possible Solution

Keep this issue open and implement the teardown in the WhatsApp plugin. Widen the close helper socket type to include `end(error?: Error)`, prefer `sock.end(new Error('OpenClaw listener close'))` when available, fall back to `sock.ws?.close?.()` for compatibility, and apply the same policy coherently to the inbound listener and shared `closeWaSocket` path. Update the existing WhatsApp teardown tests to assert `end()` for Baileys-capable sockets plus fallback behavior for older shapes; separately audit the short-lived login/QR raw-close sites if needed.
