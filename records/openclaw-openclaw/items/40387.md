## Summary

Keep this PR open. Current main still has no EasyRunner guide or navigation entry, and provider-specific hosting docs are an established OpenClaw surface. The PR is docs-only from a supply-chain perspective, but it still needs rebase plus docs/security review before merge.

## Best Possible Solution

Keep the PR open and ask for a revised, rebased docs contribution. The best path is to place or link EasyRunner under the current Install > Hosting surface, align the Podman/Caddy example with current gateway bind/auth/allowedOrigins behavior, remove normal-use dangerouslyDisableDeviceAuth guidance, document secret handling and image rebuild/update steps, and have maintainers review the EasyRunner-specific flow for operational accuracy before merge.

## Risks / Open Questions

- The proposed guide normalizes gateway.controlUi.dangerouslyDisableDeviceAuth for ordinary reverse-proxy use, conflicting with current break-glass security guidance.
- The proposed deployment flow may not match current Docker image behavior because the image does not appear to auto-create the config described by the guide.
- The proposed update flow omits rebuilding localhost/openclaw:latest before redeploying, so users may keep running the old image.
- The PR is merge-conflicting/dirty against the current GitHub base and its navigation placement predates the current Install > Hosting structure.
- No repository supply-chain regression was found in the PR file list; the remaining risk is operator guidance quality and security accuracy.
