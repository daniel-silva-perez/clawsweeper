## Summary

Keep open. Current main has real chat responsiveness work, including optimistic user turns, a pending stream indicator, non-blocking `chat.send`, and duplicate-send coverage, but the broader request is still not implemented because config/settings paths remain coarse and sequential and end-to-end UI-to-gateway-to-provider timing attribution is still missing.

## What This Changes

Request to improve gateway configuration and Web UI chat responsiveness with immediate visual feedback, non-blocking UI behavior, and latency instrumentation across UI, gateway, provider, and model phases.

## Best Possible Solution

Keep this as an umbrella UX/performance tracker or split it into scoped follow-ups: add client-to-gateway-to-provider timing attribution, preserve the existing optimistic/non-blocking chat path, isolate slow config/status panels, add per-control pending states and stale-response guards, and cross-link the narrower Control UI slowness and composer-status work.
