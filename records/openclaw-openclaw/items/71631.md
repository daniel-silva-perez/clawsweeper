## Summary

Keep open. Current main implements meaningful chat responsiveness pieces, but the broader request is still only partially satisfied: config/settings flows remain coarse and sequential, slow-RPC isolation is incomplete, and end-to-end UI-to-gateway-to-provider timing attribution is not present.

## What This Changes

The issue requests broader Web UI responsiveness improvements across gateway configuration, chat acknowledgement and streaming feedback, slow-RPC isolation, and latency instrumentation.

## Best Possible Solution

Keep this as an umbrella tracker or split it into narrower follow-ups: timing attribution, config/status panel isolation, per-control pending states, stale-response guards, and preservation of the existing optimistic/non-blocking chat path.
