## Summary

Keep open. Current main has meaningful Web UI chat responsiveness work, including optimistic user turns, a pending stream/reading indicator, non-blocking `chat.send`, and tests around duplicate in-flight sends, but the broader request remains unresolved because gateway/config UI operations still use coarse loading/saving/applying paths and the requested end-to-end UI-to-gateway-to-provider timing attribution is not implemented. The related open trackers also show overlapping remaining work rather than a merged canonical fix.

## What This Changes

Request to improve gateway configuration and Web UI chat responsiveness with immediate visual feedback, non-blocking UI behavior, and latency instrumentation across UI, gateway, provider, and model phases.

## Best Possible Solution

Keep this as an umbrella UX/performance tracker or split it into scoped follow-ups: add client-to-gateway-to-provider timing attribution, preserve the existing optimistic/non-blocking chat path, isolate slow config/status panels, add per-control pending states and stale-response guards, and cross-link the narrower Control UI slowness and composer-status work.
