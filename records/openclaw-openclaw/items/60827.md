## Summary

Keep open. Current main still supports Docker sandbox resource limits only when operators configure them; it does not provide default PID, memory, or CPU caps for new sandbox containers. Related PR #44098 is useful prior art for PID limiting, but it is open, unmerged, and narrower than this issue because it does not cover memory or CPU defaults.

## What This Changes

Request default Docker sandbox resource limits for new containers so PID, memory, and CPU caps apply unless operators override them.

## Best Possible Solution

Have maintainers choose explicit Docker sandbox default resource values, then implement them in the core Docker sandbox config resolution path while preserving global and per-agent overrides. The implementation should update schema/help/docs/changelog as needed and add tests proving default flags are emitted and explicit overrides still win.

## Reproduction Assessment

Yes. The code path is reproducible by resolving Docker sandbox config with no resource-limit settings and observing that the resulting config leaves `pidsLimit`, `memory`, `memorySwap`, and `cpus` undefined, so Docker create args omit those flags.

## Solution Assessment

Unclear until maintainers choose the default budget. The narrow maintainable code boundary is `resolveSandboxDockerConfig` plus Docker create-args tests and docs; a docs-only change or the pids-only related PR would not solve the broader issue.
