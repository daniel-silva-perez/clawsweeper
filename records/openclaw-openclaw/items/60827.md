## Summary

Keep this issue open. Current main supports operator-configured Docker sandbox resource limits, but it still does not provide default `pidsLimit`, `memory`, or `cpus` values for new sandbox containers. The related PR #44098 is useful prior art for the PID-limit part, but it is only partial and remains open, so it does not implement or supersede this broader CPU/memory default request.

## Best Possible Solution

Keep the issue open and resolve it in the core Docker sandbox config path if maintainers agree on product defaults. The likely implementation is to add Docker-backend defaults for `agents.defaults.sandbox.docker.pidsLimit`, `memory`, and `cpus` in `resolveSandboxDockerConfig`, preserve global and per-agent overrides, update schema/help/docs/changelog, and add tests proving new containers receive the default Docker flags while explicit overrides still win.

## Risks / Open Questions

- Choosing default CPU and memory limits is a behavior change: too-low defaults could break heavier agent workloads, browser-backed tasks, package installs, or custom sandbox images.
- A pids-only implementation would reduce fork-bomb risk but leave the reported RAM/CPU exhaustion path open; the issue likely needs either a broader implementation or an explicit maintainer decision to split the work.
