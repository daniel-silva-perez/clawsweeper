## Summary

Keep this issue open. Current main still supports explicitly configured Docker sandbox resource limits, but it does not provide default `pidsLimit`, `memory`, `memorySwap`, or `cpus` values for new sandbox containers. The related PR #44098 remains useful prior art for a PID-limit default only; it is not merged and does not cover the broader CPU/memory default request.

## Best Possible Solution

Keep this open for maintainer product/security judgment. If accepted, implement Docker-backend defaults in `resolveSandboxDockerConfig` for `pidsLimit`, `memory`, and `cpus` or a clearly documented subset, preserve explicit global/per-agent override precedence, update schema/help/docs/changelog, and add tests proving unset config receives the default Docker flags while explicit overrides still win.

## Risks / Open Questions

- Choosing default CPU and memory limits is a behavior change; too-low defaults could break heavier agent workloads, browser-backed tasks, package installs, or custom sandbox images.
- A PID-only fix would reduce fork-bomb risk but leave the reported RAM/CPU exhaustion path open.
