## Summary

Keep open for maintainer judgment. Current main shipped the default Docker image portion of this problem in #73362, but it does not fully implement the PR's remote-shell/custom-image missing-Python guidance, and the PR branch is stale against the current local mutation helper contract.

## What This Changes

The PR adds a sandbox mutation error helper, wraps local and remote sandbox fs-bridge mutation execution to convert missing-`python3` failures into actionable guidance, and adds a local bridge unit test.

## Best Possible Solution

Keep the shipped #73362 default-image behavior. If maintainers still want this PR's remaining DX improvement, rebase it into a narrow current-main patch that centralizes missing-interpreter guidance for the current Docker helper and remote-shell mutation path, preserves the original error as `cause`, and adds symmetric local and remote tests.

## Reproduction Assessment

Yes. The current-main remote-shell path is directly reproducible with a mocked `runRemoteShellScript` throwing `python3: not found` during `createRemoteShellSandboxFsBridge().writeFile()`, while the default Docker image path is covered by current `ensureDockerImage()` tests.

## Solution Assessment

No, not as-is. The proposed local detector targets the obsolete `python3 /dev/fd/3` helper signature, so the safer solution is a rebase against the current `python_cmd` helper contract plus a remote-shell wrapper and tests.
