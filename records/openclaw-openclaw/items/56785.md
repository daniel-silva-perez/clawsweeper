## Summary

Keep open: current main fixed the default Docker-image root cause through the related sandbox/Docker work, but it still lacks this PR's remaining remote/custom-runtime missing-Python guidance, and the proposed local detector is stale against the current helper launch contract.

## What This Changes

The PR adds a sandbox mutation error helper, wraps local and remote filesystem mutation execution to translate missing-`python3` failures into actionable guidance, and adds a local bridge unit test.

## Best Possible Solution

Keep the shipped default-image behavior from #73362. If maintainers still want this DX improvement, rebase it into a narrow patch that detects missing Python against the current pinned-helper contract, adds the same friendly mapping for the remote-shell mutation path, preserves the original error as `cause`, and covers both local and remote bridge paths with focused tests.

## Reproduction Assessment

Yes. A high-confidence unit reproduction is to mock `runRemoteShellScript` or the Docker backend to throw `moltbot-sandbox-fs: 2: python3: not found` during `writeFile`; source inspection shows current main has no remote catch and no generic mapper, while the default-image case is covered by `ensureDockerImage()` tests.

## Solution Assessment

No, not as-is. The PR's local wrapper targets the obsolete `python3 /dev/fd/3` local launch string, so the maintainable fix is to rebase against the current `python_cmd` helper contract and keep the wrapper scoped to mutation-helper execution.
