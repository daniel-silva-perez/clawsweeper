## Summary

Keep open. Current main and v2026.4.27 already shipped the linked Docker default-image/root-cause fix via #73362, but current main still lacks the PR's remote-shell missing-Python guidance and the PR's local detector is stale against the current pinned mutation helper, so the branch remains useful only after a maintainer-directed rebase/adaptation.

## What This Changes

The PR adds a sandbox mutation error helper and wraps local and remote fs-bridge mutation execution so `python3: not found` failures surface actionable sandbox image guidance, with one local bridge unit test.

## Best Possible Solution

Keep the shipped #73362 default-image behavior, then either close this PR as superseded with maintainer judgment or ask for a narrow rebase that centralizes missing-interpreter guidance for the current Docker/custom-image and remote-shell mutation paths, preserves the original error as `cause`, and adds symmetric targeted tests.

## Reproduction Assessment

Yes. The original Docker default-image path is covered by current `ensureDockerImage()` tests, and the remaining remote-shell case is reproducible by making `runRemoteShellScript` fail with `python3: not found` during `createRemoteShellSandboxFsBridge().writeFile()`.

## Solution Assessment

No, not as-is. The PR's local guard keys on the obsolete `python3 /dev/fd/3` launch string, while current main uses a `python_cmd` wrapper and a different missing-interpreter message; a rebased solution should target the current helper contract and the still-direct remote-shell launch.
