## Summary

Keep open. Current main still lacks the requested WebChat/Control UI inline-button and callback path, and the related closed reports do not supersede it; the remaining work crosses public Gateway protocol, config schema, Control UI rendering/history, and security-sensitive callback/webhook semantics.

## What This Changes

Request WebChat/Control UI support for inline message controls, including a capability config path, chat event payload, callback RPC, Control UI rendering, and optional webhook routing.

## Best Possible Solution

Add an additive WebChat interactive-control path based on the shared `presentation` contract: preserve sanitized final-only button/select blocks through chat events and history, render them in Control UI with clicked/disabled state, and add a scoped in-session callback RPC. Treat webhook routing as a separately hardened follow-up unless maintainers explicitly include it in the first design.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: the config schema rejects the requested capability path, Gateway chat schemas and method registration lack callback/interactive fields, and the Control UI normalizer/renderer has no button item or click path.

## Solution Assessment

No. The requested outcome is valid, but the literal Telegram-style `buttons` field plus `callbackWebhookUrl` is not the narrowest first contract; shared `presentation` plus scoped in-session callback routing is the safer maintainable path.

## Security Review

Status: needs_attention

Summary: The issue requests browser callback and optional webhook behavior that affects Gateway trust boundaries and egress, so security design is required before implementation.

Concerns:

- **[medium] Define callback RPC trust boundaries:** `src/gateway/method-scopes.ts:147`
  - body: A WebChat callback RPC would turn authenticated browser clicks into session input. The contract needs method scoping, session and message ownership checks, callback value validation, replay or disabled-state behavior, and audit semantics before shipping.
  - confidence: 0.88
- **[medium] Harden webhook egress separately:**
  - body: The proposed callback webhook would let UI clicks trigger Gateway-originated HTTP requests. That needs explicit SSRF and egress policy, authentication, retry behavior, and observability rather than being bundled into the first UI-button change by default.
  - confidence: 0.82
