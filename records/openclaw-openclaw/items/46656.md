## Summary

Keep open. Current main still does not implement the requested WebChat/Control UI inline-button path: the WebSocket chat schema has no button or presentation payload, no `chat.callback` RPC is registered or scoped, WebChat config only accepts `chatHistoryMaxChars`, final chat projection only sends display messages, and the Control UI message model/render path has no interactive button item. The existing closed #54720 is a narrower duplicate, while closed #46841 is Telegram-specific webhook routing, so this issue remains the active WebChat tracker.

## What This Changes

Add WebChat and Control UI support for inline button payloads, callback routing, and capability configuration comparable to the existing rich-message/presentation model used by channel plugins.

## Best Possible Solution

Implement an additive WebChat interactive-control path based on the shared `presentation` model: preserve sanitized final-only button/select blocks through chat events and history, render them in Control UI with clear clicked/disabled state, and add a scoped `chat.callback` RPC that initially routes values back into the same session. Keep webhook routing as a separately hardened follow-up unless maintainers explicitly include it in the first design.
