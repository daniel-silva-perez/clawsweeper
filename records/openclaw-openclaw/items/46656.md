## Summary

Keep open. Current `main` still lacks the requested WebChat/Control UI inline-button path: there is no button/presentation payload in the WebSocket chat schema, no `chat.callback` method registration or scope, no WebChat inline-button config, no final-chat projection for interactive metadata, and no Control UI message item/render path for buttons. The existing discussion correctly treats closed #54720 as a narrower duplicate and closed #46841 as Telegram-specific webhook routing, so this remains the canonical WebChat tracker.

## What This Changes

Add WebChat/Control UI inline button support for chat messages, including a capability/config path, final chat event payloads, callback RPC handling, and Control UI rendering/click behavior.

## Best Possible Solution

Implement an additive WebChat interactive-control path based on the shared `presentation` contract: preserve sanitized final-only button blocks through chat events and history, render them in Control UI with clicked/disabled state, and add a scoped `chat.callback` RPC that initially routes selected values back into the same session. Keep webhook routing as a separate hardened design decision.
