## Summary

Keep open. Current main still lacks the requested WebChat/Control UI inline-button path: there is no button or presentation field in the WebSocket chat schema, no `chat.callback` method registration or scope, no WebChat inline-button config, and no Control UI message item/render branch for clickable controls. The related closed items do not supersede this WebChat work.

## What This Changes

Add WebChat/Control UI inline button support by carrying interactive payloads through chat events/history, adding callback handling and config, and rendering clickable controls in the built-in UI.

## Best Possible Solution

Implement an additive WebChat interactive-control path based on the shared `presentation` contract: preserve sanitized final-only button/select blocks through chat events and history, render them in Control UI with clicked/disabled state, and add a scoped `chat.callback` RPC that initially routes selected values back into the same session. Treat webhook routing as a separate hardened follow-up unless maintainers explicitly include it in the first design.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence on current main: `gateway.webchat.capabilities.inlineButtons` is rejected by the strict schema, `chat.send`/`chat` events have no interactive fields, and the Control UI normalizer/renderer has no button item or callback request path.

## Solution Assessment

No, the exact legacy `buttons` plus webhook proposal is not the best first contract. The maintainable path should use the documented shared `presentation` payload and a scoped in-session callback RPC first, with webhook egress handled as a separate security-reviewed design.

## Security Review

Status: needs_attention

Summary: No PR diff was reviewed, but the requested callback/webhook feature is security-sensitive and needs maintainer trust-boundary review before implementation.

Concerns:

- **[medium] Define callback routing trust boundaries:**
  - body: A WebChat callback RPC would turn authenticated browser clicks into agent/session input. The contract should bind callbacks to session and message ownership, validate callback values, define replay or disabled-state behavior, and preserve operator scopes before shipping.
  - confidence: 0.88
- **[medium] Harden webhook egress separately:**
  - body: The proposed `callbackWebhookUrl` would let UI clicks trigger Gateway-originated HTTP requests. That path needs explicit SSRF/egress policy, authentication, retry, and audit semantics, so it should not be bundled into a first UI-button implementation without design approval.
  - confidence: 0.82
