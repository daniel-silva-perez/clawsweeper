## Summary

Keep open. Current main still lacks the requested WebChat/Control UI inline-button path, and the related closed items do not supersede it: #54720 is a narrower closed duplicate and #46841 is Telegram-specific webhook routing. The request is valid, but first implementation needs maintainer contract and security review because it crosses Gateway protocol, config schema, Control UI rendering/history, callback authorization, and optional webhook egress.

## What This Changes

Add full inline button support for WebChat/Control UI, including capability config, chat event payloads, a `chat.callback` RPC, Control UI rendering, and optional callback webhook routing.

## Best Possible Solution

Implement an additive WebChat interactive-control path based on the shared `presentation` contract: preserve sanitized final-only button/select blocks through chat events and history, render them in Control UI with clicked/disabled state, and add a scoped `chat.callback` RPC that initially routes selected values back into the same session. Keep webhook routing as a separate hardened follow-up unless maintainers explicitly include it in the first design.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: the config schema rejects the requested capability path, Gateway chat schemas and method registration lack callback/interactive fields, and the Control UI normalizer/renderer has no button content or click path.

## Solution Assessment

No. The requested outcome is valid, but the literal Telegram-style `buttons` field plus `callbackWebhookUrl` proposal is not the narrowest maintainable first contract; the safer path is shared `presentation` plus a scoped in-session callback, with webhook egress reviewed separately.

## Security Review

Status: needs_attention

Summary: No PR diff was reviewed, but the requested WebChat callback and webhook behavior is security-sensitive and needs a maintainer trust-boundary design before implementation.

Concerns:

- **[medium] Define callback RPC trust boundaries:** `src/gateway/method-scopes.ts:147`
  - body: A WebChat callback RPC would turn authenticated browser clicks into agent/session input. The contract needs method scoping, session and message ownership checks, callback value validation, replay/disabled-state behavior, and audit semantics before shipping.
  - confidence: 0.88
- **[medium] Harden webhook egress separately:**
  - body: The proposed callback webhook would let UI clicks trigger Gateway-originated HTTP requests. That needs explicit SSRF/egress policy, authentication, retries, and observability, so it should not be bundled into a first UI-button implementation without security review.
  - confidence: 0.82
