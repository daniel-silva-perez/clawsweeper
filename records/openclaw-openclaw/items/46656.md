## Summary

Keep open. Current main still does not provide the requested WebChat/Control UI inline-button path: the WebSocket chat schemas remain text/history oriented, no `chat.callback` method is registered or scoped, WebChat config exposes only history sizing, and the Control UI chat model/rendering still has no interactive button item.

## What This Changes

Add WebChat/Control UI inline-button support with a capability config path, final chat payload controls, callback handling, and built-in Control UI rendering.

## Best Possible Solution

Define an additive WebChat interactive-control contract around the existing shared `presentation` model: preserve sanitized final-only button/select blocks through chat events and history, render them in Control UI with clear clicked/disabled state, and add a scoped callback path that initially routes values back into the same session. Treat webhook routing as a separate hardened follow-up unless maintainers explicitly include it in the first design.

## Security Review

Status: needs_attention

Summary: No PR diff was reviewed, but the requested WebChat callback and webhook design is security-sensitive and needs maintainer trust-boundary review before implementation.

Concerns:

- **[medium] Define callback routing trust boundaries:**
  - body: A WebChat callback RPC would convert authenticated browser clicks into agent input. The contract should bind callbacks to session and message ownership, validate callback values, define replay/disabled-state behavior, and preserve operator scopes before shipping.
  - confidence: 0.88
- **[medium] Harden webhook egress separately:**
  - body: The proposed `callbackWebhookUrl` would let UI clicks trigger Gateway-originated HTTP requests. That path needs explicit SSRF/egress policy, authentication, retry, and audit semantics, so it should not be bundled into a first UI button implementation without design approval.
  - confidence: 0.82
