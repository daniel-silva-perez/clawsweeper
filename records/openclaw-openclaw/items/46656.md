## Summary

Keep this issue open. Current main still lacks the requested WebChat/Control UI inline-button contract: WebChat config has no inline-buttons capability, the Gateway chat protocol and method list have no `buttons` payload or `chat.callback` RPC, final WebChat events project only a display message, and the Control UI normalized chat model/rendering path has no interactive button item. The existing shared `presentation` contract is the right direction, but WebChat/Control UI has not yet been wired to render or route those controls. The narrower #54720 was already closed as duplicate/superseded in favor of this older, more detailed tracker.

## Best Possible Solution

Keep this issue open as the canonical tracker. The best implementation path is an additive WebChat/Control UI contract built around the existing semantic `presentation` model: expose a deliberate WebChat/Control UI capability config, carry final presentation button blocks through Gateway chat events/history in a typed way, render them in Control UI with stable disabled/clicked states, and add a bounded `chat.callback` route that initially injects callback values into the session. Treat webhook/zero-token callback routing as an explicit security/product decision, possibly a later plugin/ClawHub follow-up.
