## Summary

Keep open. Current main still lacks the WebChat/Control UI inline-button path, and the remaining work crosses protocol, config, UI rendering, and callback security boundaries rather than being a stale or already-implemented request.

## What This Changes

The issue requests WebChat/Control UI inline button support across config, WebSocket chat payloads, callback RPC handling, and built-in UI rendering.

## Best Possible Solution

Implement an additive WebChat interactive-control path using the shared `presentation` model: preserve sanitized final-only button blocks through chat events and history, render them in Control UI with clicked/disabled state, and add a scoped `chat.callback` RPC that initially routes selected values back into the same session. Treat webhook routing as a separately hardened follow-up.
