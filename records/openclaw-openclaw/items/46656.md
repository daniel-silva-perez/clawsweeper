## Summary

Keep open. Current main still lacks the requested WebChat/Control UI inline-button path: the Gateway chat schema has no button or presentation payload, no `chat.callback` RPC is registered or scoped, WebChat config has no inline-button capability, final WebChat projection only carries text/media display content, and Control UI chat content types do not include interactive controls.

## What This Changes

Add WebChat/Control UI inline button support, including a capability config path, final chat payloads that preserve buttons, a callback RPC, and Control UI rendering/click handling.

## Best Possible Solution

Implement an additive WebChat interactive-control path based on the shared `presentation` contract: preserve sanitized final-only button/select blocks through chat events and history, render them in Control UI with clicked/disabled state, and add a scoped `chat.callback` RPC that routes selected values back into the same session. Treat webhook routing as a separate hardened follow-up unless maintainers explicitly include it in the first design.

## Security Review

Status: needs_attention

Summary: No PR diff was reviewed, but the requested callback/webhook design is security-sensitive and needs maintainer trust-boundary review before implementation.

Concerns:

- **[medium] Define callback routing trust boundaries:**
  - body: A WebChat callback RPC would convert authenticated browser clicks into agent input. The contract should bind callbacks to session and message ownership, validate callback values, define replay/disabled-state semantics, and preserve operator scopes before shipping.
  - confidence: 0.88
- **[medium] Harden webhook egress separately:**
  - body: The proposed `callbackWebhookUrl` would let UI clicks trigger Gateway-originated HTTP requests. That path needs explicit SSRF/egress policy, auth, retry, and audit semantics, so it should not be bundled into a first UI button implementation without design approval.
  - confidence: 0.82
