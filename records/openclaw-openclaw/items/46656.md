## Summary

Keep open. Current main still lacks the requested WebChat/Control UI inline-button path, and the related closed items do not supersede it: #54720 is a narrower duplicate and #46841 is Telegram-specific webhook routing. The request is valid, but the first implementation needs maintainer contract and security review before automation starts.

## What This Changes

Add full inline button support to WebChat/Control UI, including capability config, chat event payloads, callback handling, optional webhook routing, and built-in UI rendering.

## Best Possible Solution

Implement an additive WebChat interactive-control path based on the shared `presentation` contract: preserve sanitized final-only button blocks through chat events and history, render them in Control UI with clicked/disabled state, and add a scoped in-session callback RPC. Keep webhook routing as a separate hardened follow-up unless maintainers explicitly include it in the first design.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence on current main: the WebChat config schema rejects the requested capability path, the Gateway chat schemas and method registry have no callback or interactive payload, and the Control UI renderer has no button content item or click path.

## Solution Assessment

No. The requested outcome is valid, but the literal Telegram-style `buttons` field plus `callbackWebhookUrl` proposal is not the safest first contract; the narrow maintainable path should use shared `presentation` and a scoped callback first, with webhook egress reviewed separately.

## Security Review

Status: needs_attention

Summary: No PR diff was reviewed, but the requested callback and webhook behavior is security-sensitive and needs a maintainer trust-boundary design before implementation.

Concerns:

- **[medium] Define callback RPC trust boundaries:** `src/gateway/method-scopes.ts:147`
  - body: A WebChat callback RPC would convert authenticated browser clicks into agent/session input. The contract needs method scoping, session and message ownership checks, callback value validation, replay/disabled-state behavior, and audit semantics before shipping.
  - confidence: 0.88
- **[medium] Harden webhook egress separately:**
  - body: The proposed callback webhook would let UI clicks trigger Gateway-originated HTTP requests. That needs explicit SSRF/egress policy, authentication, retries, and observability, so it should not be bundled into a first UI-button implementation without security review.
  - confidence: 0.82

## Risks / Open Questions

- The public WebChat wire contract is not defined yet: maintainers need to choose shared `presentation` blocks versus a legacy Telegram-shaped `buttons` field and decide how history persistence should work.

- The proposed webhook route would create Gateway-originated HTTP egress from UI clicks and needs separate SSRF, authentication, retry, and audit policy before implementation.
