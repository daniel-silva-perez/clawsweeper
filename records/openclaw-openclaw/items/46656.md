## Summary

Keep open. Current main still lacks the requested WebChat/Control UI inline-button path: the WebSocket chat schemas have no button or presentation payload, there is no `chat.callback` method registration or scope, WebChat config only exposes history truncation, final chat broadcasts project only the display message, and the Control UI renderer has no interactive button item. Closed #54720 is a narrower duplicate of this request, while closed #46841 is Telegram-specific webhook routing and does not supersede the WebChat protocol/UI work.

## What This Changes

Add WebChat/Control UI support for inline message controls, including a config capability, final chat event payload, callback RPC, and browser rendering/click handling.

## Best Possible Solution

Implement an additive WebChat interactive-control path based on the shared `presentation` contract: preserve sanitized final-only button blocks through chat events and history, render them in Control UI with clicked/disabled state, and add a scoped in-session callback RPC. Treat webhook routing as a separate hardened follow-up unless maintainers explicitly include it in the first design.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence on current main: `gateway.webchat.capabilities.inlineButtons` is rejected by the strict schema, `chat.send`/`chat` events have no interactive fields, no `chat.callback` RPC is registered or scoped, and the Control UI normalizer/renderer has no button item or click path.

## Solution Assessment

No, the exact legacy `buttons` plus webhook proposal is not the best first contract. The narrow maintainable path should use the documented shared `presentation` payload and a scoped in-session callback first, with webhook egress handled as a separate security-reviewed design.

## Security Review

Status: needs_attention

Summary: No PR diff was reviewed, but the requested callback/webhook feature is security-sensitive and needs maintainer trust-boundary design before implementation.

Concerns:

- **[medium] Define WebChat callback trust boundaries:**
  - body: A WebChat callback RPC would turn authenticated browser clicks into agent/session input. The contract should bind callbacks to session and message ownership, validate callback values, define replay or disabled-state behavior, and preserve operator scopes before shipping.
  - confidence: 0.88
- **[medium] Harden webhook egress separately:**
  - body: The proposed `callbackWebhookUrl` would let UI clicks trigger Gateway-originated HTTP requests. That needs explicit SSRF/egress policy, authentication, retry, and audit semantics, so it should not be bundled into a first UI-button implementation without design approval.
  - confidence: 0.82
