## Summary

Keep this issue open. Current main still accepts Google Chat `replyToMode: "off"`, but the Google Chat monitor, reply delivery helper, and routed outbound adapter still pass thread identifiers into `sendGoogleChatMessage`; the direct open fix PR #67055 is useful context but is unmerged and does not cover every current send path.

## What This Changes

Google Chat should suppress implicit thread replies when `channels.googlechat.replyToMode` is set to `"off"`.

## Best Possible Solution

Keep the issue open until a Google Chat fix lands on main that gates implicit thread propagation on the resolved account `replyToMode` across inbound context, typing placeholders, natural text/media reply delivery, and routed outbound `sendText`/`sendMedia`, while preserving intentional explicit thread targets and adding focused regression tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction is an inbound Google Chat event with `message.thread.name` and `channels.googlechat.replyToMode: "off"`: current main still passes that thread through the typing path and reply delivery, and routed outbound sends independently pass `threadId ?? replyToId` to the API sender.

## Solution Assessment

No. Current main is not a solution, and the direct open monitor-focused PR is incomplete for routed outbound sends; the safer fix is a narrow cross-path Google Chat boundary change with tests for monitor typing, natural text/media replies, and routed outbound text/media.
