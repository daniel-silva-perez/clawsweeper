## Summary

Keep this issue open. Current main still accepts Google Chat `replyToMode`, but the Google Chat send paths still propagate thread IDs independently of that mode: inbound monitor context and typing placeholders use `message.thread.name`, routed outbound text/media use `threadId ?? replyToId`, and the API sender threads any request that receives a `thread` parameter. The related PR #67055 remains useful context, but main does not yet contain a complete fix for the reported behavior.

## Best Possible Solution

Keep this issue open until a Google Chat fix lands on main. The fix should gate implicit Google Chat thread propagation on the resolved account `replyToMode` across inbound context, typing placeholder creation, monitor text/media delivery, and routed outbound `sendText`/`sendMedia`, while preserving intentionally explicit thread targets. PR #67055 or an equivalent maintainer patch is useful implementation context, but the final fix should include focused regression coverage for off-mode typing, text, media, and routed outbound sends.
