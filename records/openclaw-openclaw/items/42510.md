## Summary

Keep open. Current main still accepts `channels.googlechat.replyToMode: "off"`, but Google Chat continues to propagate thread identifiers through the inbound typing-placeholder path and routed outbound text/media path without a Google Chat transport guard. The related open PR #67055 remains an implementation candidate, not shipped behavior.

## Best Possible Solution

Keep this issue open until a Google Chat fix lands on main. The fix should gate implicit Google Chat thread propagation on the resolved `replyToMode` across inbound context, typing placeholder creation, final text/media delivery, and routed outbound `sendText`/`sendMedia`, with focused regression tests for off-mode typing, text, media, and routed outbound paths. PR #67055 or an equivalent maintainer patch is the likely review path.

## Risks / Open Questions

- Closing this now would leave Google Chat users with threaded typing-placeholder replies despite `replyToMode: "off"`.
- A monitor-only fix could miss routed outbound `sendText` and `sendMedia`; an outbound-only fix could miss the typing-placeholder update path.
- The eventual fix should distinguish implicit inbound thread propagation from deliberate explicit thread targets so it does not break intentional Google Chat thread sends.
