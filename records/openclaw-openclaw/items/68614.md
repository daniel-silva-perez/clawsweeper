## Summary

Keep open. Current main still lacks the PR's deferred-response timeout behavior for Google Chat, and the branch is useful but not merge-ready because it leaves the timeout armed when dispatch exits without a delivery callback. The branch is also dirty against current main after the Google Chat reply-delivery split and turn-kernel migration.

## What This Changes

The PR modifies Google Chat reply handling to start a 25-second timeout after sending the typing placeholder, post a fallback message, delete the placeholder, and send the eventual agent reply as a new message if the deferred update window expires.

## Best Possible Solution

Port the useful timeout behavior to current main's split Google Chat code: keep the existing typingIndicator default and reaction fallback, start a bounded timer only after a typing message is created, pass an expired-deferred flag into the delivery helper, reuse the existing send/update fallback helper logic, and always clear the timer when the turn dispatch exits without delivery.

## Reproduction Assessment

Yes. A focused unit reproduction can fake timers, create a message-mode typing placeholder, make dispatch return without invoking `deliver`, advance 25 seconds, and observe the PR's spurious fallback; current main also lacks any timer path for long-running delivery before the first reply callback.

## Solution Assessment

No. The PR is directionally right, but the proposed patch is not the best current solution because it targets the pre-refactor file shape and misses timeout cleanup around dispatch completion; a narrow port to `monitor.ts` plus `monitor-reply-delivery.ts` with a `try/finally` cleanup is safer.
