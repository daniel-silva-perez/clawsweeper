## Summary

Keep this issue open. Current main has partial active-registry reuse for gateway-bindable plugin tools, but the reported provider-auth hot path still exists: repeated tool creation can call `resolveEnvApiKey`, which rebuilds manifest-backed provider candidate/evidence maps, and those maps still force plugin-registry snapshot derivation with `preferPersisted: false`. The issue also has a concrete production profile fingerprint plus an independent macOS data point in the comments.

## What This Changes

The issue reports a 2026.4.27 gateway performance regression where repeated tool creation synchronously rebuilds plugin registry metadata and provider auth env-var maps, causing multi-second event-loop stalls.

## Best Possible Solution

Fix the hot path with request- or active-runtime-scoped reuse: tool construction should reuse the active gateway plugin registry when appropriate and should compute provider auth candidate/evidence maps once per stable config/workspace/env snapshot, then thread those maps through repeated `resolveEnvApiKey`/tool model checks without adding a broad stale metadata cache.

## Reproduction Assessment

Yes. The issue gives a high-confidence reproduction path and profile fingerprint against 2026.4.27, and the current-main source still contains the same repeated lookup shape; I did not run a live gateway load replay in this read-only review.

## Solution Assessment

Unclear because there is no proposed patch. The reporter's suggested instrumentation and memoization direction is appropriate, but the maintainable fix should be narrow and respect the plugin metadata freshness contract rather than caching all plugin discovery globally.
