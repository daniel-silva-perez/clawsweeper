## Repair Work Prompt

Fix the provider-auth/tool-construction hot path reported in https://github.com/openclaw/openclaw/issues/74860. Current main still lets repeated tool creation call resolveEnvApiKey without injected candidate/evidence maps, causing manifest-backed provider auth resolution through loadPluginManifestRegistryForPluginRegistry({ preferPersisted: false }) and derived plugin registry work. Implement a narrow active-runtime/request-scoped reuse path: compute provider auth candidate/evidence maps once per stable config/workspace/env snapshot used by createOpenClawTools/createOpenClawCodingTools/media tool capability checks, thread those maps through hasAuthForProvider/resolveEnvApiKey-style checks, and preserve plugin metadata freshness/trust semantics. Do not add a broad persistent discovery cache, do not weaken workspace plugin trust gating, and do not duplicate #73039's media/web factory memoization; coordinate with it if it has landed. Add focused regression tests proving repeated provider auth checks do not reload the manifest registry, plus coverage for workspace plugin trust and active gateway plugin registry reuse. Add a user-facing changelog fix entry with proper attribution if behavior changes.
