## Summary

Keep open. Current main has Telegram voice-note transcription and voice-note TTS delivery, but it still lacks the requested voice-specific priority lane, streaming STT/TTS pipeline, concurrency model, result/preset caching, and stage-boundary timing snapshots.

## Best Possible Solution

Keep this issue open as the tracking item for Telegram voice pipeline performance. A good implementation should start with stage timing/state snapshots and benchmarks, then add a voice-aware lane or priority policy plus provider-backed streaming/caching only where Telegram, media-understanding, and speech-core contracts can support it without breaking ordered chat behavior.

- The latency targets need a concrete benchmark setup and provider matrix; source inspection cannot prove <3s STT or <2s TTS across configured providers.
- A voice bypass or concurrent lane could reorder per-chat Telegram turns, command handling, approvals, or reply delivery unless the sequencing contract is designed carefully.
- Streaming STT/TTS support varies by provider, so exposing it beyond Telegram may require an additive SDK/provider contract rather than a Telegram-only shortcut.
