## Summary

Keep open. Current main has Telegram voice-note support, preflight batch audio transcription, and native voice-note TTS delivery, but it does not implement the requested Telegram voice-processing optimization work: voice-specific priority/bypass queueing, streaming transcription for Telegram attachments, concurrent transcription/TTS staging, transcription/TTS result caching, or stage-boundary timing snapshots.

## Best Possible Solution

Keep this issue open as the tracking item for Telegram voice pipeline performance. A good implementation would stay mostly in the Telegram plugin and shared media/TTS seams: add stage timing/state snapshots, benchmark the current batch path, then introduce a voice-aware lane/priority policy and provider-backed streaming or concurrency only where the SDK contracts can support it without breaking existing channel behavior.
