## Summary

Keep open. The linked bug is valid and this PR targets the right subscription lifecycle seam, but the current patch emits the unsubscribe fallback outside the existing serialized session-event queue, so it can finalize the chat before an already-scheduled async `agent_end` path finishes. The diff also lacks the required changelog entry for a user-facing fix.

## What This Changes

The PR shares terminal lifecycle idempotency through embedded subscription state, emits a synthetic terminal lifecycle event from `unsubscribe()` when `agent_end` was missed, and adds unsubscribe fallback regression tests.

## Best Possible Solution

Keep the backend subscription-level fallback, but route it through the same serialized event ordering as normal `agent_end` handling or explicitly track `agent_end` as seen/in-flight so unsubscribe only synthesizes a terminal lifecycle when the terminal event was genuinely missed. Add focused regression coverage for pending async message flush plus unsubscribe, and add the active-version changelog entry for the user-facing Control UI fix.

## Reproduction Assessment

Yes. The original bug has a high-confidence code-level path: timeout calls fire-and-forget `activeSession.abort()`, cleanup calls `unsubscribe()`, and current main only finalizes chat on lifecycle `end` or `error`. The PR regression can be reproduced with a stub session that queues an async `message_end` or `agent_end`, then calls `unsubscribe()` before that promise settles.

## Solution Assessment

No, not in its current form. The subscription seam is the right place to guarantee a terminal lifecycle event, but the fallback must preserve the existing event queue semantics instead of emitting directly from teardown.
