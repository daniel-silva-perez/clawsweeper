## Summary

Keep this issue open. Current main still lacks first-class `--thread-id` support on `openclaw cron add/create` and `openclaw cron edit`, even though `openclaw message send` exposes the flag and the cron delivery/protocol path can already represent Telegram topic delivery. The provided GitHub context also shows #60890 as an open, unmerged PR with closing syntax for this issue, so the maintainer path is to review, land, or close that implementation candidate rather than closing this issue now.

## Best Possible Solution

Keep this issue open while maintainers review #60890. The best product/code path is a focused cron CLI change that adds `--thread-id` to `cron add/create` and `cron edit`, composes or stores Telegram topic delivery through the existing `delivery.threadId`/`:topic:` contract, validates invalid non-Telegram or blank combinations early, preserves existing manually encoded `:topic:` targets, and adds regression coverage in `src/cli/cron-cli.test.ts`.
