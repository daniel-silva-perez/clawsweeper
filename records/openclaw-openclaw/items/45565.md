## Summary

Keep open. Current main still does not implement a configurable target such as `messages.lifecycleTarget`, `messages.internalWarningTarget`, or category-based `messages.messageRouting` for gateway lifecycle/tool/internal warnings. Main now has `messages.suppressToolErrors` plus heartbeat-scoped `suppressToolErrorWarnings`, but delivery still follows the active/originating reply route rather than a dedicated warning channel. Related #39406 overlaps on suppression policy but does not supersede this dedicated-channel routing request; #28637 is closed stale, not a merged implementation.

## Best Possible Solution

Keep open for maintainer design review, then implement a generic internal notification routing policy rather than a Discord-only special case. The likely product direction is a documented `messages` config surface with safe defaults, optional suppression, category routing for lifecycle/tool/system notices, target validation through existing outbound routing contracts, and tests for active-channel fallback, dedicated-channel delivery, and mutating tool safety.
