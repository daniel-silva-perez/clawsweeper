## Summary

Keep this issue open. Current main has a partial `messages.suppressToolErrors` schema/runtime path, but it does not implement the requested dedicated-channel routing for lifecycle/internal warnings, and tests still preserve visible mutating tool failure warnings such as edit/write failures even when `messages.suppressToolErrors` is enabled. The discussion usefully broadened the request from gateway lifecycle probes to a unified internal warning routing policy; that broader product/config work remains open.

## Best Possible Solution

Keep this issue open and treat it as the broader core config/design task for internal warning delivery. The likely fix belongs in the shared agent/gateway reply-routing path: add a documented `messages` routing policy such as a catch-all `internalWarningTarget` plus optional per-category overrides, define how it composes with `messages.suppressToolErrors`, heartbeat `suppressToolErrorWarnings`, cron timeout notifications, and channel target syntax, then cover lifecycle, non-mutating tool, mutating tool, and cron-warning cases with tests.

## Risks / Open Questions

- The existing `messages.suppressToolErrors` key can look like a fix at first glance, but it is not equivalent to the requested dedicated-channel routing and intentionally does not hide mutating edit/write failures in normal chat.
- A real implementation needs a clear target grammar and delivery policy so internal warnings can be routed or suppressed without hiding safety-critical mutating-tool failures unexpectedly.
