## Summary

Keep open. Current main still has no configurable dedicated target for gateway lifecycle, tool-error, or internal warning payloads: the schema only exposes suppression, warning generation still produces normal error reply payloads, and dispatch still delivers through the current or originating conversation route. The discussion broadened into category-aware routing and suppression, so this needs maintainer design rather than stale cleanup.

## What This Changes

Request a `messages` config option to route or suppress gateway lifecycle, tool-error, and internal warning notifications to a dedicated channel instead of the active conversation.

## Best Possible Solution

Add a generic core message-routing policy for internal warnings under `messages.*`, with strict schema/types/help/docs, validated outbound target resolution through existing channel routing contracts, backwards-compatible active-channel fallback, and regression coverage for lifecycle/tool/system categories plus existing suppression behavior.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a feature request, but yes for source-level verification: current schema rejects the proposed keys, and warning payloads are still emitted as normal reply payloads routed through origin/current delivery.

## Solution Assessment

Is this the best way to solve the issue? Unclear for the exact config shape because no patch or maintainer decision exists; the maintainable direction is a shared, category-aware routing policy with validated targets, not a Discord-only special case or blanket suppression default.
