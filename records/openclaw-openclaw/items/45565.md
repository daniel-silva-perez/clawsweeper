## Summary

Keep open. Current main still lacks a configurable dedicated target for gateway lifecycle, tool-error, or internal warning notifications; the only normal-chat config found is the narrower `messages.suppressToolErrors` suppression boolean, and reply delivery still follows the active/originating conversation route.

## What This Changes

The issue requests a configurable `messages` routing policy so gateway lifecycle, tool-error, and other internal warning notifications can go to a dedicated channel or be suppressed instead of appearing in the active conversation channel.

## Best Possible Solution

Keep this open for maintainer design review, then implement a generic internal notification routing policy rather than a Discord-only special case. The end state should be a documented `messages` config surface with safe defaults, optional suppression, category-aware lifecycle/tool/system routing, target validation through existing outbound routing contracts, and regression coverage for active-channel fallback, dedicated-channel delivery, and mutating-tool safety.

## Reproduction Assessment

Not applicable as a feature request, but there is a high-confidence source-level check for the gap: the proposed routing keys are absent from strict config schema, and warning payloads are still emitted as normal reply payloads through the current/originating route.

## Solution Assessment

No proposed patch is present. The best direction is a shared core warning-routing policy with validated outbound targets and docs/tests, not a Discord-only workaround or a broad suppression toggle that could hide important mutating failures.
