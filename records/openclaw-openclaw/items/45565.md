## Summary

Keep open. Current main only has partial suppression for some tool-error warnings and still lacks a configurable dedicated target for gateway lifecycle, internal tool-warning, or category-based warning delivery; mutating edit/write failures can still be emitted as normal error payloads through the active/originating conversation route.

## What This Changes

Add configuration to route gateway lifecycle and internal tool-warning messages to a dedicated channel, or suppress them, instead of posting them into the active conversation channel.

## Best Possible Solution

Add a generic core message-delivery policy for internal warning categories under the public config contract, with strict schema/types/help/docs, validated target resolution through the existing channel-route layer, backward-compatible active-channel fallback, and regression coverage for lifecycle, tool-error, and system-notice delivery alongside existing suppression behavior.

## Reproduction Assessment

Not applicable as a feature request, but the source-level verification is high-confidence: current main has no proposed routing keys, the strict schema would reject them, and unresolved mutating tool warnings still become normal `isError` reply payloads.

## Solution Assessment

Unclear for the exact key names because maintainers have not chosen the public API. The maintainable direction is a shared category-aware routing policy with validated targets, not a Discord-only special case or broad silent suppression by default.
