## Summary

Keep open. Current main still has the silent implicit-resume behavior for `openclaw agent --agent <id>` without a real `--session-id`: the CLI has no fresh-session flag, both local and Gateway paths can reuse a fresh stored agent main session, and docs/tests do not expose a fresh/no-resume UX. The adjacent open PR #72932 is relevant context for session routing but is unmerged and does not supersede this issue's observable-resume/fresh-session request.

## What This Changes

The issue asks for `openclaw agent --agent <id>` runs without an explicit session id to avoid or clearly surface implicit reuse of the stored agent main session.

## Best Possible Solution

Preserve resume-by-default compatibility, add `--new-session` with `--fresh` as an alias, plumb a force-fresh field through CLI, Gateway protocol, and local embedded execution, bypass stored-session reuse when set, expose implicit resume through stderr diagnostics plus JSON metadata, and update focused docs/tests/changelog.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: seed or create a fresh `agent:<id>:main` session store entry, then invoke `openclaw agent --agent <id> -m ...` without a real `--session-id`; both local and Gateway resolution paths select the stored session id. I did not run the live CLI because this review was required to keep the checkout read-only.

## Solution Assessment

Yes. The best near-term fix is the reporter's backward-compatible Option B: keep existing resume semantics for scripts, add an explicit fresh-session control, and make implicit resume observable. Flipping the default to fresh would be a larger compatibility decision.
