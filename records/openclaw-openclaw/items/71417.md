## Summary

Keep open. Current main still has the reported CLI/session UX gap: `openclaw agent --agent <id>` has no fresh-session flag, both local and Gateway paths can reuse a fresh stored agent main session, and docs/tests still describe or assert the resume behavior without a visible no-resume control. The reporter's follow-up comments add useful concrete shape and a second reproduction, while adjacent PR #72932 appears related but not merged and not sufficient for the requested fresh-session UX.

## What This Changes

The issue requests a way for `openclaw agent --agent <id>` to avoid or visibly disclose implicit reuse of the most recent agent main session when no `--session-id` is passed.

## Best Possible Solution

Preserve resume-by-default compatibility, add an explicit `--new-session` flag with `--fresh` as an alias, plumb an additive force-fresh field through CLI, Gateway protocol, and local embedded options, bypass stored-session reuse only when requested, and surface implicit resume with stderr diagnostics plus structured JSON metadata and docs/tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: seed or create a fresh `agent:<id>:main` session entry, run `openclaw agent --agent <id> -m ...` without a real `--session-id`, and the local/Gateway paths select the stored session id. I did not run the live CLI because this review was required to keep the checkout read-only.

## Solution Assessment

Yes. The reporter's backward-compatible Option B is the narrowest maintainable fix: add a fresh-session knob and make resume observable while preserving workflows that intentionally rely on agent main-session continuity. Flipping the default to fresh should be a separate product/deprecation decision.
