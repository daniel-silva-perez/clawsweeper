## Summary

Keep open. Current main still has the reported CLI/session UX gap: `openclaw agent --agent <id>` without explicit fresh intent still targets the persisted agent main session, can reuse its fresh stored `sessionId`, and exposes no `--new-session`/`--fresh` control, no resume banner, and no structured resumed metadata. The related open PR #72932 is adjacent to the `--session-id` routing problem, but it is not merged and does not replace the fresh-session/observable-resume request here.

## What This Changes

The issue asks for `openclaw agent` to avoid or make visible silent reuse of the last/main agent session when no `--session-id` or fresh-session flag is supplied.

## Best Possible Solution

Preserve resume-by-default compatibility, add an explicit `--new-session` flag with `--fresh` as an alias, plumb a force-fresh field through both Gateway and `--local` paths, and make implicit resume observable with stderr diagnostics plus structured JSON metadata and docs.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: seed a fresh `agent:<id>:main` session entry, run `openclaw agent --agent <id> -m ...` without session controls, and the current local/Gateway logic selects the stored `sessionId`; I did not run the live CLI because this review was read-only.

## Solution Assessment

Yes. The reporter's Option B is the narrowest maintainable direction: preserve existing scripts that rely on resume-by-default while adding an explicit fresh-session knob and visible resume diagnostics instead of flipping the default immediately.
