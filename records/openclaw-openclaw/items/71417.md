## Summary

Keep open. Current main still lacks a fresh-session flag, force-new protocol field, resume banner, or structured resumed metadata, and both local embedded and Gateway paths still reuse a fresh stored agent main session when the caller omits `--session-id`. Open PR #72932 addresses adjacent `--session-id` routing, but its diff does not implement this issue's fresh-session opt-out or resume visibility.

## What This Changes

`openclaw agent --agent <id>` without `--session-id` or `--channel` should make implicit session resume visible and provide a way to force a fresh session.

## Best Possible Solution

Keep the existing resume-by-default behavior for compatibility, but add an explicit fresh-session control and make implicit resume observable in both human and JSON/scripted output across Gateway and `--local` paths.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to seed a fresh `agent:<id>:main` session entry, invoke `openclaw agent --agent <id> -m ...` without `--session-id`, and observe current local/Gateway resolution reuse the stored `sessionId`; I did not run the live CLI because this read-only review must not mutate session state.

## Solution Assessment

Yes. A backward-compatible `--new-session`/`--fresh` flag plus stderr resume banner and structured resumed metadata is the narrowest maintainable fix; changing the default to fresh would be more disruptive for scripts that intentionally rely on continuation.
