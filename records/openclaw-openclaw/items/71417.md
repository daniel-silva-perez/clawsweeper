## Summary

Keep open. Current main still resumes a fresh stored agent main session when `openclaw agent --agent <id>` is invoked without a real session id, and the CLI/Gateway surface still lacks an explicit fresh-session control or observable resume banner/metadata. The adjacent open PR #72932 addresses the separate `--session-id` routing bug, but it is not merged and does not replace the fresh/no-resume UX requested here.

## What This Changes

The issue requests that `openclaw agent` stop silently reusing the last persisted agent session, either by defaulting to fresh sessions or by adding an explicit fresh-session flag and visible resume diagnostics.

## Best Possible Solution

Preserve resume-by-default compatibility, add `--new-session` with `--fresh` as an alias, plumb a force-fresh field through CLI, Gateway protocol, and local embedded agent options, bypass stored-session reuse when set, and make implicit resume observable through stderr diagnostics plus structured JSON metadata and docs.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: create or seed a fresh `agent:<id>:main` store entry, invoke `openclaw agent --agent <id> -m ...` without a real `--session-id`, and both local and Gateway paths select the stored session id. I did not run the live CLI because this review was required to keep the checkout read-only.

## Solution Assessment

Yes. The reporter's Option B is the best near-term fix: it avoids breaking scripts that rely on main-session resume while adding an explicit fresh-session knob and making implicit resume visible. Flipping the default to fresh can remain a later deprecation or major-version decision.
