## Summary

Current main still has the reported behavior: `openclaw agent --agent <id>` without a real `--session-id` resolves to the agent main session, reuses a fresh stored `sessionId`, and omitted delivery channel still falls back through `last`. Related explicit `--session-id` fixes/tests do not add a fresh-session flag, resume banner, or resumed metadata for this default-resume path.

## Best Possible Solution

Keep this issue open. Implement a backward-compatible fresh-session path: add `--new-session` with optional `--fresh` or `--no-resume` alias, plumb a force-new flag through `AgentCliOpts`, `AgentParamsSchema`, Gateway session resolution, and local embedded `resolveSession`; when an existing session is reused, emit a clear stderr resume banner and structured result metadata such as `resumed: true` plus the reused `sessionId`; update `docs/cli/agent.md` and `docs/tools/agent-send.md`; add focused tests for omitted session, blank `--session-id`, explicit existing `--session-id`, and fresh-session controls across Gateway and `--local`.

## Risks / Open Questions

- Changing the default to always create a fresh session could break scripts and multi-turn workflows that intentionally rely on automatic agent main session continuity.
- Leaving the current behavior silent can append unrelated old context to one-shot tests, increasing token usage and producing confusing replies.
- A resume banner must be emitted on stderr, especially with `--json`, and should be covered in both Gateway and `--local` paths.
