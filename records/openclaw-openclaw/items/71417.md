## Summary

Keep open. Current main still preserves the reported silent-resume behavior for `openclaw agent --agent <id>` when the caller omits `--session-id`: there is no fresh-session flag, no force-new Gateway field, no resume banner, and no structured resumed metadata. The reporter’s follow-up comments add a concrete VPS reproduction, and related PR #72932 is adjacent but does not implement this issue’s fresh-session opt-out or resume visibility.

## What This Changes

`openclaw agent --agent <id>` without `--session-id` or `--channel` should make implicit session resume visible and provide a way to force a fresh session.

## Best Possible Solution

Preserve resume-by-default for compatibility, but add an explicit fresh-session control and make implicit resume observable in both human stderr output and structured JSON metadata across Gateway and `--local` paths.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to seed a fresh `agent:<id>:main` session entry, invoke `openclaw agent --agent <id> -m ...` without `--session-id`, and observe current local and Gateway resolution reuse the stored `sessionId`; I did not run the live CLI because this review is read-only and would mutate session state.

## Solution Assessment

Yes. A backward-compatible `--new-session` or `--fresh` flag plus a stderr resume banner and structured resumed metadata is the narrowest maintainable fix; changing the default to fresh would be more disruptive for scripts that intentionally rely on continuation.
