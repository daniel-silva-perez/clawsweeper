## Summary

Keep open. Current main still lacks a fresh-session flag, resume banner, or structured resumed metadata, and the checked code continues to reuse fresh stored agent-main sessions while omitted delivery channel input falls back through `last`; the reporter also supplied concrete follow-up reproductions after the earlier review.

## What This Changes

Request a safer `openclaw agent` session UX so omitted `--session-id`/`--channel` does not silently resume stale agent-main context without a fresh-session control or clear resume visibility.

## Best Possible Solution

Add a backward-compatible fresh-session control and make implicit resume observable: support `--new-session` with `--fresh` as an alias, plumb a force-new flag through CLI, Gateway schema/handler, and local embedded session resolution, emit stderr resume text when reusing an existing session, include structured resumed/session metadata in results, and document the behavior for scripted users.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to seed a fresh `agent:<id>:main` entry, invoke `openclaw agent --agent <id> -m ...` without `--session-id` or `--channel`, and observe current local/Gateway resolution reuse `entry.sessionId`; I did not run it live because this review was read-only and a live CLI run would mutate session state.

## Solution Assessment

Yes, the backward-compatible flag plus resume visibility is the narrowest maintainable fix visible from the current code and discussion. Defaulting every `openclaw agent` run to fresh would solve surprise at the cost of breaking intentional multi-turn scripts, so it should not be the first repair path.
