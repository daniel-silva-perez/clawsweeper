## Summary

Keep this PR open. Current main still does not implement the proposed non-interactive onboarding flags for trusted-proxy gateway auth, trusted proxy IPs, trusted-proxy allowUsers, or Control UI allowed origins. The PR is also paired by the same author with open issue #73639, so it should be reviewed or replaced as the implementation candidate rather than closed by cleanup.

## Best Possible Solution

Keep this PR open, or replace it with an updated focused PR, to add non-interactive onboarding support for `gateway.auth.mode="trusted-proxy"`, `gateway.trustedProxies`, `gateway.auth.trustedProxy.userHeader`, `requiredHeaders`, `allowUsers`, an explicit `allowLoopback` decision, and `gateway.controlUi.allowedOrigins`. Tie it to #73639, coordinate wording with #69066 where relevant, and include targeted tests/docs for the trusted-proxy security model.

## Risks / Open Questions

- The submitted patch, as shown in the provided diff, appears to spread existing `gateway.auth` when writing trusted-proxy mode; final review should ensure token/password fields are cleared so startup does not reject mixed trusted-proxy/token config.
- The PR docs example includes `127.0.0.1` in trusted proxies, but current runtime requires explicit `gateway.auth.trustedProxy.allowLoopback = true` for loopback-source trusted-proxy auth. That needs a deliberate product/security decision and matching flag/docs behavior.
- Because this is a non-interactive path for gateway auth, maintainers should verify proxy IP parsing, header normalization, requiredHeaders semantics, allowUsers behavior, Control UI allowed-origin handling, and trusted-proxy warnings before landing.
