Keep this PR open. Current main still does not implement the proposed non-interactive onboarding flags for trusted-proxy gateway auth, and the PR is paired by the same author with open feature issue #73639. The useful maintainer path is to review, update, land, or replace this implementation candidate rather than close it as cleanup.

## Best Possible Solution

Keep this PR open, or replace it with an updated focused PR, to add non-interactive onboarding support for `gateway.auth.mode="trusted-proxy"`, `gateway.trustedProxies`, `gateway.auth.trustedProxy.userHeader`, `requiredHeaders`, `allowUsers`, an explicit `allowLoopback` decision, and `gateway.controlUi.allowedOrigins`. Tie the implementation to #73639, coordinate security wording with #69066 where relevant, and include targeted tests/docs for the trusted-proxy security model.

## Risks / Open Questions

- The proposed implementation needs review to ensure switching to trusted-proxy clears any existing `gateway.auth.token`; current startup rejects trusted-proxy mode when a shared token is also configured.
- The proposed docs example includes `127.0.0.1` as a trusted proxy but the diff does not add an explicit `allowLoopback` flag; current runtime rejects loopback-source trusted-proxy auth unless `gateway.auth.trustedProxy.allowLoopback = true`.
- Because this is a non-interactive auth path, maintainers should verify parsing and validation for proxy IPs, header names, requiredHeaders, allowUsers, Control UI allowed origins, and failure-mode tests before landing.
