## Summary

Keep this PR open. Current main still does not implement the proposed non-interactive onboarding flags for trusted-proxy gateway auth, trusted proxy IPs, trusted-proxy allowUsers, allowLoopback, or Control UI allowed origins. The PR is also paired by the same author with open issue #73639 and was explicitly described as addressing it, so the useful maintainer path is to review, update, land, or replace the implementation candidate rather than close it as cleanup.

## Best Possible Solution

Keep this PR open, or replace it with an updated focused PR, to add non-interactive onboarding support for `gateway.auth.mode="trusted-proxy"`, `gateway.trustedProxies`, `gateway.auth.trustedProxy.userHeader`, `requiredHeaders`, `allowUsers`, an explicit `allowLoopback` decision, and `gateway.controlUi.allowedOrigins`. Tie the implementation to #73639, coordinate security wording with #69066 where relevant, and include targeted tests/docs for the trusted-proxy security model.

## Risks / Open Questions

- The submitted approach needs final review to ensure switching to trusted-proxy clears any existing `gateway.auth.token` and inappropriate password state, because current startup rejects mixed trusted-proxy/token config and the docs call mixed token config unsafe.
- The proposed docs example includes loopback proxy IPs, but current runtime rejects loopback-source trusted-proxy requests unless `gateway.auth.trustedProxy.allowLoopback = true`; the final design needs an explicit flag or a deliberate omission with matching docs.
- Because this is a non-interactive auth configuration path, maintainers should verify IP parsing, header normalization, requiredHeaders semantics, allowUsers behavior, Control UI allowed-origin handling, and tests for failure modes before landing.
