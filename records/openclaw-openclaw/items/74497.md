## Summary

Keep open. Current main still has Telegram group ingress, but the reporter's migration log maps to a no-op branch for unconfigured old group IDs, directory groups are still config-backed, and several commenters report group-only failure after normal Telegram setup steps. This remains actionable, but the fix needs maintainer policy because observed/migrated group handling is authorization-sensitive.

## What This Changes

Telegram group chats should be discovered and ingested/responded to by the OpenClaw Telegram bot when direct messages work normally.

## Best Possible Solution

Define Telegram's intended behavior for newly observed and migrated groups, then implement it without granting reply authority from observation alone. A good end state would either keep directories config-backed with clearer diagnostics for the new `-100...` ID, or add a read-only observed-groups surface plus explicit authorization steps for replies.

## Reproduction Assessment

Yes. The issue provides concrete live steps and logs, and the reported migration/no-config messages map directly to the current migration handler and helper behavior on main.

## Solution Assessment

Unclear. Auto-discovering and enabling Telegram groups is not obviously the safest fix; the maintainable solution needs an explicit policy for observed/migrated groups and sender authorization before code changes.
