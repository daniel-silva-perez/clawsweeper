## Summary

Keep open. Current main still matches the reported Telegram group-only failure path: group migration is logged but does not create or authorize a new group entry when the old ID was never configured, group discovery remains config-backed, and multiple issue comments report DMs working while groups fail after normal Telegram setup steps. This is actionable, but the fix changes group discovery and reply authorization semantics, so it needs maintainer policy before an automated repair branch is safe.

## What This Changes

Report asks Telegram group chats to be discovered by `openclaw directory groups list --channel telegram` and to ingest/respond to group messages after normal Telegram bot setup, while direct messages already work.

## Best Possible Solution

Define the intended Telegram behavior for newly observed and migrated groups, then implement it inside the Telegram plugin without granting reply authority from observation alone. A maintainable end state would either keep directories config-backed with clearer diagnostics for the new supergroup ID, or add a read-only observed-groups/doctor surface plus explicit config steps before replies are enabled.

## Reproduction Assessment

Yes. The issue gives concrete live setup steps and logs, and those logs map directly to the current migration handler and helper behavior on main.

## Solution Assessment

Unclear. Automatically discovering and enabling Telegram groups from observed traffic is not clearly the safest solution; the narrowest maintainable fix needs an explicit policy for observed/migrated groups and sender authorization.
