## Summary

Keep open. Current main still matches the reported Telegram group-only failure shape: migration events are logged but only migrate pre-existing old-ID config, directory group listing remains config-backed, and group ingress is still gated by explicit group/sender policy. The issue has recent live reproduction from multiple users, and the remaining fix would decide authorization semantics for observed or migrated groups, so it should get maintainer policy review before automation changes behavior.

## What This Changes

The issue asks for Telegram group chats to be discovered, ingested, and replied to after normal bot setup, including newly created groups that migrate to supergroups, while direct messages already work.

## Best Possible Solution

Define the intended Telegram behavior for newly observed and group-to-supergroup-migrated chats, then implement it inside the Telegram plugin while preserving explicit group and sender authorization. A maintainable end state would either provide clearer diagnostics/doctor guidance for the new supergroup ID or add a read-only observed-groups surface, with replies still requiring explicit config or an approved policy.

## Reproduction Assessment

Yes. The issue gives concrete live steps and logs, and the reported migration/no-config branch maps directly to current source; code-level checks also show why directory listing remains empty and why group ingress can be blocked while DMs work.

## Solution Assessment

Unclear. Automatically discovering and enabling Telegram groups from observed traffic is not clearly the safest fix; the best solution needs maintainer policy on whether observation should create directory entries, config suggestions, or reply authorization.
