## Summary

Keep open. Current main does handle Telegram group and supergroup message updates, but the reported directory expectation is not implemented because Telegram group directory listing is still config-backed, and the reporter's follow-up points at group-to-supergroup migration where current code only rewrites an exact configured old group ID. The remaining behavior needs maintainer judgment and redacted current config/log evidence before it can be closed or automated.

## What This Changes

The issue asks OpenClaw to discover, ingest, and respond to Telegram group chats through the Telegram bot, including showing groups in `openclaw directory groups list --channel telegram`.

## Best Possible Solution

Define Telegram group behavior explicitly: keep directory groups as configured IDs and improve docs/doctor/status around migration IDs, or add a read-only observed-groups surface that never grants access by itself. Then reproduce the migration path with redacted config and add targeted tests for the chosen behavior.
