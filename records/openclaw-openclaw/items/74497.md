## Summary

Keep open. Current main supports Telegram group/supergroup ingress, but the reported directory expectation is still not implemented because Telegram group directory listing is config-backed only. The reporter's follow-up narrows the response symptom toward group-to-supergroup migration or post-migration policy/config resolution, and current code only migrates an explicitly configured old group ID, so this is not safe to close as implemented.

## What This Changes

Telegram group chats are reported as not discovered by `openclaw directory groups list --channel telegram` and not triggering responses even though direct Telegram messages work.

## Best Possible Solution

Define the intended Telegram group behavior explicitly: keep directory groups as configured IDs and improve docs/doctor/status around post-migration IDs, or add a read-only observed-groups surface that never grants group access by itself. Then reproduce the post-migration response path with redacted config and add targeted coverage for the chosen behavior.
