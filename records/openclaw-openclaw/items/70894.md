## Summary

Keep open. Current main and the latest release still route approved webhook agent calls through isolated cron with `forceNew: true`, so the reported session continuity regression is still present; open PR #70895 is the active paired fix candidate and should be reviewed or replaced before this issue closes.

## What This Changes

Webhook `/hooks/agent` runs should reuse fresh sessions for policy-approved stable `sessionKey` values instead of forcing every isolated hook invocation onto a new session.

## Best Possible Solution

Land a narrow provenance-aware fix that keeps scheduled isolated cron runs fresh while allowing approved hook/webhook-sourced isolated runs with stable session keys to reuse fresh sessions, with regression coverage for both paths.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: send repeated approved `/hooks/agent` requests with the same stable session key, and the current path forwards that key into an isolated cron job that always calls `resolveCronSession` with `forceNew: true`.

## Solution Assessment

Yes. The narrow maintainable solution is to compute `forceNew` from true scheduled isolated-cron semantics rather than from `sessionTarget` alone; PR #70895 appears to target that boundary but still needs maintainer review and validation.
