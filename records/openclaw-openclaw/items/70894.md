## Summary

Keep open. Current main still routes approved `/hooks/agent` calls into an isolated cron job and passes `forceNew: true` solely because `sessionTarget` is `"isolated"`, so stable, policy-allowed webhook `sessionKey` values still cannot preserve conversation state. The issue also has open PR #70895 linked with closing syntax, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Webhook-triggered agent runs should reuse fresh sessions for the same approved `sessionKey` instead of always starting a new conversation.

## Best Possible Solution

Review and land PR #70895 or a narrow equivalent that derives `forceNew` from hook/webhook provenance instead of `sessionTarget` alone, while keeping normal scheduled isolated cron runs fresh and adding regression coverage for both paths.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: repeated approved `/hooks/agent` requests with the same stable `sessionKey` reach `runCronIsolatedAgentTurn`, and current main still passes `forceNew: true` before `resolveCronSession` can reuse a fresh entry.

## Solution Assessment

Yes. The narrow maintainable fix is to allow only hook/webhook-sourced isolated runs with approved stable keys to reuse fresh sessions, rather than making all isolated cron runs reusable or weakening hook session-key policy.
