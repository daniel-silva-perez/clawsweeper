## Summary

Current main still appears to reproduce the reported webhook session-continuity bug: `/hooks/agent` dispatches through an isolated cron-style run, the isolated runner still passes `forceNew: true` for every `sessionTarget: "isolated"`, and the session resolver only reuses prior history when `forceNew` is false. The issue is also paired with open PR #70895 using closing syntax, so the right cleanup outcome is to keep this issue open while maintainers review or replace that narrow fix PR.

## Best Possible Solution

Keep this issue open and review/land PR #70895 or a narrow equivalent. The fix should compute `forceNew` from true scheduled isolated-cron semantics rather than every isolated agent turn, allow policy-approved hook/webhook-sourced isolated turns to reuse fresh existing sessions, and keep regression coverage proving ordinary isolated cron jobs still force fresh sessions.
