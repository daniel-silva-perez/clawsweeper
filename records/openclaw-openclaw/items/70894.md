## Summary

Keep open. Current main still routes approved `/hooks/agent` calls through the isolated cron runner with `forceNew: true`, so stable session keys cannot preserve webhook conversation state; open PR #70895 is already paired with this issue and should be reviewed, landed, or replaced before closure.

## What This Changes

The issue requests that repeated webhook-triggered agent runs with the same policy-approved `sessionKey` preserve conversation history instead of always starting a fresh isolated session.

## Best Possible Solution

Review and land PR #70895 or a narrow equivalent that derives `forceNew` from hook/webhook provenance for approved stable keys, while preserving fresh sessions for normal isolated cron jobs and adding regression coverage for both paths.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: repeated approved `/hooks/agent` requests with the same stable `sessionKey` flow into an isolated cron job that still calls `resolveCronSession` with `forceNew: true`.

## Solution Assessment

Yes. The narrow maintainable fix is provenance-aware `forceNew` handling for hook/webhook-sourced isolated runs, not making all isolated cron jobs reusable or changing hook session-key policy broadly.
