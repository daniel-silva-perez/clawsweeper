## Summary

Keep open. Current `main` still forces fresh sessions for webhook-triggered isolated agent runs even after hook policy accepts a stable request `sessionKey`, and open PR #70895 is already paired with this issue using closing syntax, so this should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue asks for webhook-triggered agent runs with approved stable `sessionKey` values to preserve conversation context instead of always starting a fresh isolated session.

## Best Possible Solution

Review and land PR #70895, or an equivalent narrow change, so webhook-sourced isolated runs with approved stable session keys can reuse fresh sessions while ordinary scheduled isolated cron runs continue to start fresh sessions; include a focused regression test for the webhook path.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: repeated approved `/hooks/agent` calls with the same `sessionKey` flow through hook dispatch into an isolated cron job, and current `main` still passes `forceNew: true` before `resolveCronSession` can reuse the stored session.

## Solution Assessment

Yes. The best fix direction is a narrow `forceNew` decision based on both isolated cron target and hook/webhook provenance, rather than making all isolated cron jobs reusable or loosening hook session-key policy.
