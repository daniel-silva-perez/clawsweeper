## Summary

Keep https://github.com/openclaw/openclaw/issues/70894 open. Current main and latest release still accept policy-approved webhook session keys but route `/hooks/agent` through the isolated cron runner with `forceNew: true`, which bypasses the fresh-session reuse branch; the issue is also paired with open PR #70895 using closing syntax, so it should remain open until that fix or a narrow equivalent lands.

## What This Changes

The issue reports that repeated `/hooks/agent` calls with the same allowed `sessionKey` lose webhook conversation history because isolated agent runs always force a new session.

## Best Possible Solution

Review and land PR #70895 or a narrow equivalent that computes `forceNew` from true scheduled isolated-cron semantics, allows approved webhook/hook runs with stable session keys to reuse fresh sessions, and adds regression coverage for both webhook reuse and ordinary isolated cron freshness.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: direct `/hooks/agent` forwards an approved stable session key into an isolated cron job, and the runner unconditionally passes `forceNew: true`, which makes `resolveCronSession` allocate a new `sessionId` even when a fresh entry exists.

## Solution Assessment

Yes, the narrowest maintainable direction is to adjust the isolated-run `forceNew` decision based on hook provenance while keeping normal scheduled isolated cron runs fresh. The linked PR appears to target that boundary, but it still needs maintainer review and validation before the issue closes.
