## Summary

Keep open. Current main still forces new sessions for `/hooks/agent` isolated runs even when the request session key is accepted by hook policy, and the issue is paired with open, unmerged PR #70895 using closing syntax, so this should remain open until that PR or an equivalent fix lands.

## What This Changes

Webhook-triggered agent runs should preserve conversation context when repeated calls provide the same policy-approved stable `sessionKey`.

## Best Possible Solution

Review and land PR #70895, or an equivalent narrow change, so webhook-sourced isolated runs with approved stable session keys can reuse fresh sessions while ordinary scheduled isolated cron jobs continue to start fresh sessions.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: repeated approved `/hooks/agent` calls with the same `sessionKey` flow into an isolated cron job, and current main still passes `forceNew: true` before `resolveCronSession` can reuse the stored session.

## Solution Assessment

Yes. The narrow maintainable fix is to derive `forceNew` from both isolated cron target and hook/webhook provenance, rather than making all isolated cron jobs reusable or weakening hook session-key policy.
