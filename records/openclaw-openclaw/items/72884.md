## Summary

Keep open. Current main still leaves actionable task-maintenance gaps: terminal `lost` cron rows are excluded from durable cron recovery, and the newer stale CLI/subagent evidence matches a current path where a persisted child-session row can be treated as liveness proof after the live run context is gone. Open PR #75036 targets only the cron portion and does not supersede the full issue.

## What This Changes

The issue asks for `openclaw tasks maintenance --apply` to repair or prune `lost` backing-session-missing task records and stale running task records, with optional cleanup for benign timestamp inconsistencies.

## Best Possible Solution

Keep this issue open as the canonical tracker, land or replace the focused cron recovery fix from #75036, then add a separate narrow fix so stale CLI tasks owned by historical subagent session rows are not treated as live after their run context is gone. Split timestamp clamping unless maintainers explicitly want maintenance to rewrite clock-skew fields.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is clear: create a terminal `lost` cron task with durable run history and a stale running CLI task whose only remaining backing evidence is an old subagent child-session row, then observe current main leaves those rows unrepaired.

## Solution Assessment

No. Current main is not the best solution because it only covers active cron durable recovery and chat-backed CLI rows; the safer repair path is a focused terminal-lost cron fix plus a separate CLI liveness predicate and regression test.
