## Summary

Keep open. Current main still rejects terminal `lost` cron rows before durable cron recovery, and the related fix is open and unmerged in PR #75036; the broader `stale_running` and timestamp cleanup parts are not fully settled on main.

## What This Changes

Requests that `openclaw tasks maintenance --apply` recover or prune `lost` backing-session-missing task records, handle `stale_running` records, and optionally clamp benign timestamp skew.

## Best Possible Solution

Land the narrow terminal lost cron recovery once validated, then either keep this issue open for the remaining stale-running/timestamp policy or split those parts into focused follow-ups with explicit maintenance semantics.

## Reproduction Assessment

Yes for the central lost-row behavior: create a cron task already marked `lost` with `error: "backing session missing"` and a matching durable finished cron run log; current main skips recovery because the task is not active. The exact `stale_running` live records need more backing-state detail to reproduce confidently.

## Solution Assessment

No, current main is not the best solution because it leaves terminal lost cron rows unrecovered. PR #75036 appears to be the right narrow direction for that gap, while generic stale-running cleanup and timestamp clamping should be handled deliberately rather than folded in implicitly.
