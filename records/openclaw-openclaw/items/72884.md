## Summary

Keep this issue open. Current main has maintenance paths for active stale tasks and ordinary terminal pruning, but it still does not recover terminal `lost` cron rows from durable cron history, and the open related PR that targets that gap is unmerged and has a concrete review finding.

## What This Changes

The issue requests that `openclaw tasks maintenance --apply` clean up `lost` backing-session-missing records, `stale_running` records, and benign timestamp skew warnings instead of reporting no maintenance work.

## Best Possible Solution

Update and land the narrow terminal-lost cron recovery fix so false-positive `backing session missing` rows recover from durable cron history and clear stale errors, then decide whether offline CLI stale-running cleanup and small timestamp-skew clamping belong in the same maintenance policy or a follow-up issue.

## Reproduction Assessment

Yes. A high-confidence unit reproduction is a terminal `runtime: "cron"`, `status: "lost"`, `error: "backing session missing"` task with a matching durable `finished`/`ok` cron run log; current main skips recovery because the durable cron recovery path only accepts active tasks.

## Solution Assessment

No, current main is not the best solution for the reported lost-row behavior. The open PR's durable-history approach is the right narrow direction, but it needs to clear stale lost errors and may not cover the broader stale-running/timestamp-cleanup scope without maintainer judgment.
