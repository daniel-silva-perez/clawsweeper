## Summary

Keep open. Current main still leaves actionable task-maintenance gaps: terminal lost cron rows cannot use durable cron recovery, and the newly supplied stale CLI evidence matches a current liveness path that can treat historical subagent session rows as backing proof.

## What This Changes

The issue asks for `openclaw tasks maintenance --apply` to repair or prune lost backing-session-missing task records and stale_running task records instead of reporting no changes.

## Best Possible Solution

Keep this open until the terminal lost cron recovery is merged or otherwise implemented, then add a focused fix for stale subagent-owned CLI task liveness and decide whether timestamp clamping should be split into its own explicit maintenance policy.

## Reproduction Assessment

Yes. A source-level reproduction is clear: model a terminal `lost` cron task with durable run history and observe recovery is skipped, and model an old running `cli` task with an `agent:...:subagent:...` child session row but no active run context and observe current main treats the session row as backing proof.

## Solution Assessment

No. Current main is not the best solution because it still leaves both the terminal lost cron recovery gap and the stale subagent-owned CLI liveness gap; PR #75036 appears to be a narrow candidate for only the cron portion.
