## Summary

Keep open. Current main still has actionable task-maintenance gaps: terminal `lost` cron rows are excluded from durable cron recovery, and stale CLI tasks with old subagent child-session rows can still be treated as backed even after the live run context is gone. The related open PR #75036 addresses only the cron portion, so it does not supersede the full issue.

## What This Changes

Requests that `openclaw tasks maintenance --apply` repair or prune `lost` backing-session-missing task records and `stale_running` task records, with optional timestamp cleanup for benign clock skew.

## Best Possible Solution

Keep this as the canonical tracker until current main repairs both core paths: terminal backing-session-missing cron rows should be recovered or retired using durable cron evidence, and subagent-owned CLI rows without a live run context should not stay running solely because a historical child-session row exists. Treat timestamp correction as a separate follow-up unless maintainers explicitly want maintenance to rewrite clock-skew fields.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is available: seed a terminal `lost` cron task with durable run evidence and a stale CLI task whose only remaining backing evidence is an `agent:...:subagent:...` child-session row, then inspect that current main leaves those rows unrepaired by the existing maintenance predicates.

## Solution Assessment

No. Current main is not the best solution because it covers active cron recovery and chat-backed CLI false-liveness, but not terminal lost cron recovery or subagent-owned stale CLI rows; #75036 is a useful partial cron fix, with the CLI liveness predicate needing a separate focused change.
