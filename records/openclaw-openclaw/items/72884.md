## Summary

Keep this issue open. Current main partially improves terminal cleanup stamping/pruning, but durable cron recovery still skips terminal `lost` cron rows because it only accepts active tasks, and offline CLI maintenance still cannot treat stale cron `running` rows as lost when the Gateway is the authoritative cron runtime. Open PR #75036 targets the central terminal-lost cron recovery gap but is unmerged, so this is not ready for cleanup closure.

## What This Changes

The issue asks for `openclaw tasks maintenance --apply` to clean or recover `lost` backing-session-missing task records and `stale_running` task records, with possible auto-correction for benign timestamp skew.

## Best Possible Solution

Land a narrow terminal-lost cron recovery fix that recovers false-positive `backing session missing` cron rows from durable run history and clears stale lost errors on success, then explicitly decide whether offline CLI stale-running cleanup and small timestamp-skew clamping should be included or tracked as separate maintenance-policy follow-ups.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a cron task already marked `lost` with `error: "backing session missing"` and a matching durable finished cron run log: current main skips recovery because the durable cron recovery guard requires `isActiveTask(task)`. A stale running cron row can also remain unchanged under offline CLI maintenance when the CLI is not the authoritative cron runtime and no durable terminal result is available.

## Solution Assessment

No, current main is not the best final solution for the reported behavior. The open PR’s durable-history recovery direction is the right narrow fix for terminal lost cron rows, but it needs the stale-error cleanup issue addressed and does not settle the broader stale-running or timestamp-clamping policy by itself.
