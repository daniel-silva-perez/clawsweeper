## Summary

Keep open. Current main still leaves actionable task-maintenance gaps: terminal `lost` cron rows are excluded from durable cron recovery, and stale CLI tasks backed only by historical subagent session rows can still look live after the actual run context is gone. The open related PR only targets the cron portion, so it does not supersede the full issue.

## What This Changes

The issue asks `openclaw tasks maintenance --apply` to repair or prune backing-session-missing `lost` task records and stale running task records, with optional correction for benign timestamp skew.

## Best Possible Solution

Keep this issue as the canonical tracker until the terminal lost cron recovery lands or is replaced, then add a separate focused fix so stale CLI tasks owned by historical subagent child-session rows are not treated as live after the run context is gone. Handle timestamp clamping separately if maintainers want maintenance to rewrite clock-skew fields.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is clear: create a terminal `lost` cron task with matching durable run history and a stale running CLI task whose only remaining backing evidence is an old `agent:...:subagent:...` child-session row, then observe current main leaves those rows unrepaired.

## Solution Assessment

No. Current main is not the best solution because it covers active cron durable recovery and chat-backed CLI rows, but not terminal lost cron recovery or stale CLI rows backed only by subagent session-store history; the safer direction is a focused cron fix plus a separate liveness predicate regression test.
