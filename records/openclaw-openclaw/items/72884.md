## Summary

Keep open. Current main still leaves the reported maintenance family unresolved: terminal `lost` cron rows are excluded from durable cron recovery, and the newer discussion identifies a separate stale CLI false-liveness path where a persisted child session row can keep an old `running` task alive after the live run context is gone.

## What This Changes

Request that `openclaw tasks maintenance --apply` repair or prune `lost` backing-session-missing and `stale_running` task records, with possible auto-correction for benign timestamp skew.

## Best Possible Solution

Keep this issue open as the canonical task-maintenance cleanup thread for now: resolve the terminal lost cron recovery in PR #75036 or an equivalent patch, then add a focused CLI liveness fix for stale non-chat/subagent-owned CLI rows whose live run context is gone, while splitting timestamp clamping into a deliberate follow-up if maintainers want that behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists by creating a terminal `lost` cron task with matching durable finished run history, or a stale `running` non-chat CLI task with no live run context but a persisted child session row; current main skips recovery or keeps the row live respectively.

## Solution Assessment

No. Current main is not the best solution because it only handles nearby active cron/chat-backed CLI cases; PR #75036 appears to be the right narrow fix for terminal lost cron rows, but it does not cover the newly documented stale non-chat CLI false-liveness path.
