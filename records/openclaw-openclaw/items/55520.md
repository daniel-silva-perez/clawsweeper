## Summary

Keep open. Current main still deterministically treats non-empty OpenAI-compatible `delta.content` as visible text before looking at reasoning fields, and still maps bare `reasoning_content`/`reasoning`/`reasoning_text` to `thinking`. Recent related work covers OpenRouter typed `reasoning_details`, Z.AI request-side thinking controls, Qwen request payload flags, and reasoning-only retry recovery, but it does not add provider-scoped Bailian/Z.AI output normalization for the mirrored cases reported here.

## What This Changes

The issue requests stable streaming assembly for OpenAI-compatible providers where `content` and `reasoning_content` can be misclassified, including a fallback for reasoning-only visible answers without leaking true thinking.

## Best Possible Solution

Add provider-owned, explicitly documented stream normalization or compatibility metadata for known providers/models whose reasoning field can carry the final user-visible answer, with regression fixtures for the reported mirrored samples and default guards that keep true reasoning private.

## Reproduction Assessment

Yes for the parser-level behavior: a chunk with only `delta.reasoning_content` will be classified as `thinking`, and a chunk with non-empty `delta.content` will be classified as visible text. The live-provider race/nondeterminism still needs redacted fixtures or live confirmation.

## Solution Assessment

No, the issue's proposed global fallback is not the safest final fix because it can leak real chain-of-thought. The maintainable solution is provider/model-scoped normalization with tests and explicit privacy policy.
