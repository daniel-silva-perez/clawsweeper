## Summary

Keep open. Current main still does not implement the reported provider-aware fallback/reclassification. The generic OpenAI-compatible stream parser treats `delta.content` as visible text and bare `reasoning_content`/`reasoning`/`reasoning_text` as `thinking`; only typed OpenRouter `reasoning_details` entries can be surfaced as visible text. The existing reasoning-only recovery retries with a visible-answer continuation or returns an incomplete-turn error, which mitigates empty replies but does not convert an answer-like bare reasoning field into text or classify provider-specific internal-looking `content` back into thinking.

## Best Possible Solution

Keep this issue open and implement a provider-aware stream-classification fix with regression coverage for both mirrored samples: answer-like bare `reasoning_content` with empty `content`, and provider output where visible `content` is known to be internal reasoning. The safest path is explicit provider/plugin compat metadata or typed upstream fields, not a global fallback that exposes all reasoning fields.

## Risks / Open Questions

- Closing now would hide a still-current loss path for OpenAI-compatible providers that return user-visible answer text only through bare reasoning fields.
- A blanket fallback from `reasoning_content` to visible text could expose private chain-of-thought for providers where that field is truly internal reasoning.
- The inverse sample, where `delta.content` appears to contain internal thinking, needs provider-specific evidence or metadata; the generic parser cannot safely infer that from text shape alone.
