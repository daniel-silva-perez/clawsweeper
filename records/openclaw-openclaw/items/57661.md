## Summary

Keep open. This is a recent sessions_send timeout report with a confirming comment, and current main does not clearly implement a fix for the one-to-many or non-existent-target timeout behavior. The earlier related report #55315 was closed for delivery/persistence, but this issue is about timeout behavior; the related #64622 PR is still an implementation candidate, not shipped code.

## Best Possible Solution

Keep this issue open and use it as the user-facing bug report while maintainers review, revise, or replace #64622. The concrete path is to add targeted regression coverage for `sessions_send` to missing key-shaped targets and one-to-many A2A sends, then implement scoped timeout/target-resolution hardening in `src/agents/tools/sessions-send-tool.ts`, `src/agents/tools/sessions-resolution.ts`, and the gateway agent/session persistence path as needed.

## Evidence

- **Issue report and confirmation:** The issue reports frequent `session_send` timeouts in a one-to-many agent-to-agent development-team workflow on v2026.3.28 Docker with `ark-code-latest`; a later commenter independently reported timeouts when sending to a non-existent session.
- **Current sessions_send still uses caller/default wait timeouts:** `sessions_send` derives `timeoutSeconds` directly from the tool input or a 30 second default, then converts it to `timeoutMs`; there is no current A2A-specific timeout floor or hardened timeout resolver on main.
