## Summary

Keep this issue open. The report is recent, has concrete logs for Kimi/Moonshot failing with the Anthropic stream-order error, and current main does not show a source-backed fix that relaxes or repairs that event ordering for the Kimi Anthropic-compatible stream path. Current main does include related improvements: Kimi Coding now has a bundled provider, defaults thinking off, preserves Kimi-specific tool behavior, and sanitizes the raw stream-order error text. Those are helpful, but they do not prove the reported Kimi run is usable again.

## Best Possible Solution

Keep this issue open and fix it in the provider/transport boundary. The best path is a focused Kimi/Moonshot regression test that simulates the reported out-of-order Anthropic stream behavior, then either add a Kimi-owned stream compatibility shim/retry or move Kimi Coding to a provider path that avoids the strict Anthropic SDK event-order failure. The shipped error sanitization should remain, but it is not a substitute for making `kimi/kimi-code` or the affected Moonshot/Kimi refs usable.
