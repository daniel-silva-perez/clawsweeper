## Summary

Keep this issue open. The report is recent and concrete, with repeated Kimi/Moonshot failures showing the Anthropic stream-order error and follow-up comments confirming similar Kimi failures plus a manual config workaround. Current main has adjacent Kimi/Moonshot improvements and sanitizes the raw error, but I did not find source, tests, or docs proving that the Kimi Coding Anthropic-compatible stream path now recovers from out-of-order `message_start`/`message_stop` events.

## Best Possible Solution

Keep this issue open and fix it in the provider/transport boundary. Add a focused Kimi Coding regression test that simulates the reported out-of-order Anthropic stream behavior, then either add a Kimi-owned compatibility shim/retry or route Kimi Coding through a provider path that avoids the strict Anthropic SDK event-order failure. Keep the current sanitized error message, but treat it as UX hardening rather than the functional fix.
