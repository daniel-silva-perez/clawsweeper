## Summary

Keep open: current main still classifies `browser.profiles.*` under the browser plugin's broad restart rule, so `config.patch` can still schedule a direct SIGUSR1 restart in hot mode and the file watcher can still queue a restart in hybrid mode. The generic unconditional-restart path has improved, but this browser-profile-specific gap remains valid.

## What This Changes

The issue requests that `config.patch` changes to hot-reloadable browser profile paths avoid SIGUSR1/full gateway restarts while preserving restarts for browser settings that truly require process reinitialization.

## Best Possible Solution

Add granular browser plugin reload metadata or planner behavior so profile CRUD, `browser.profiles.*`, and `browser.defaultProfile` avoid full gateway restarts where the browser runtime can refresh them, while preserving restart-required behavior for plugin registration and security-sensitive browser runtime settings. Back it with reload-plan, `config.patch`, and browser profile refresh tests, then update the browser docs to describe the narrower restart boundary.
