## Summary

Keep open: current main still routes `browser.profiles.*` through the browser plugin's broad `browser` restart prefix, so the reload planner marks those paths restart-required and `config.patch` can still schedule SIGUSR1 in hot mode or queue a restart in hybrid mode. The related generic unconditional-restart issue is fixed, but this browser-profile-specific gap remains the canonical open tracker.

## What This Changes

The issue asks for `config.patch` to avoid SIGUSR1/full gateway restarts when all changed paths, especially `browser.profiles.*`, are hot-reloadable or no-op browser profile changes.

## Best Possible Solution

Add granular browser plugin reload metadata so profile CRUD and dynamic profile selection paths avoid full gateway restarts, while preserving restart-required behavior for browser settings that actually affect managed browser services or security-sensitive runtime state. Back it with reload-plan and `config.patch` regression tests, and keep #43803 as the canonical tracker until that lands.

## Repair Work Prompt

Current main gates `config.patch` restarts through the reload planner, but the browser plugin still registers `restartPrefixes: ["browser"]`, so `browser.profiles.*` and likely `browser.defaultProfile` still plan as restart-required and can schedule SIGUSR1 in hot mode or queue a restart in hybrid mode.
