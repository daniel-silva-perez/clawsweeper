## Summary

Keep open: current main still reproduces the core runtime bug. The old unconditional restart path has been narrowed, but `browser.profiles.*` still matches the browser plugin's broad `browser` restart metadata, so `config.patch` in hot mode can schedule direct SIGUSR1 and the file-watcher path can still queue a gateway restart in hybrid/restart modes. The focused implementation PR is closed unmerged, and the narrower duplicate issue is also closed, leaving this issue as the right tracker.

## What This Changes

The issue asks OpenClaw to avoid SIGUSR1/full gateway restarts when `config.patch` changes browser profile paths that the browser runtime can refresh without restarting.

## Best Possible Solution

Express browser profile/default-profile reloadability through plugin-owned reload metadata or a specificity-aware planner, while keeping broader browser runtime settings restart-required. The fix should add focused regression tests for `browser.profiles.*` through both `config.patch` and watcher planning, and update user-facing docs/changelog if the restart behavior changes.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a `config.patch` that changes `browser.profiles.<name>.cdpUrl` while the browser plugin is active: the planner matches the broad `browser` restart rule, which can drive direct SIGUSR1 in hot mode and watcher restart in hybrid/restart modes.

## Solution Assessment

Yes, the best fix direction is narrow reload classification, not a blanket removal of browser restart behavior. A plugin-owned granular prefix or specificity-aware reload planner keeps core extension-agnostic and preserves restart semantics for browser settings that still require service re-registration.
