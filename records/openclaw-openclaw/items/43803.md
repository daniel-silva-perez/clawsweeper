## Summary

Current main still leaves the specific browser-profile restart bug unresolved. The older unconditional config.patch restart behavior has been narrowed, but `browser.profiles.*` still matches the browser plugin’s broad restart prefix, so profile-only changes are still planned as gateway restarts rather than hot/no-op reloads.

## Best Possible Solution

Keep this issue open as the canonical tracker. Implement granular browser reload registration so safe profile-scoped paths such as `browser.profiles.<name>.cdpUrl` and likely `browser.defaultProfile` hot-reload or no-op, while preserving restart/reconcile behavior for service-affecting browser settings. Add reload-plan and config.patch regression coverage for hot and hybrid modes, then update the gateway/browser docs if the shipped contract changes.
