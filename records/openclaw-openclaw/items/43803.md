## Summary

Keep open: current main still routes `browser.profiles.*` through the browser plugin's broad `browser` restart rule. That means `config.patch` in hot mode can still schedule direct SIGUSR1 when the reload plan says restart, and the config watcher can still queue a full restart in hybrid/restart modes. The focused implementation PR #45624 is closed unmerged, and the narrower browser-rule issue #64204 is also closed, leaving this issue as the right open tracker.

## What This Changes

`config.patch` should avoid SIGUSR1/full gateway restarts when changed paths such as `browser.profiles.*` can be handled by hot reload.

## Best Possible Solution

Add granular browser reload metadata or a generic specificity-aware reload planner so `browser.profiles.*` and likely `browser.defaultProfile` avoid full gateway restarts when the browser runtime can refresh them, while broader browser runtime settings still require restart. The fix should cover both the `config.patch` direct restart path and the file-watcher reload path, with focused regression tests and a changelog entry if user-facing behavior changes.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is `config.patch` changing `browser.profiles.<name>.cdpUrl` while the browser plugin is active: the planner matches the broad `browser` restart rule, then hot-mode `config.patch` schedules SIGUSR1 and hybrid/restart watcher handling queues a full restart.

## Solution Assessment

No. Current main's plan-driven restart path is narrower than the old unconditional restart behavior, but it still treats profile-only browser changes as restart-required; the maintainable fix is to express browser profile/default-profile reloadability without hardcoding bundled browser IDs in core.
