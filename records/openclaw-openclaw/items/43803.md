## Summary

Keep open: current main still preserves the reported browser-profile restart path. The direct config.patch restart is reload-plan-driven now, but browser.profiles.* still matches the browser plugin's broad browser restart prefix, so hot mode still reaches the SIGUSR1 scheduling branch.

## What This Changes

The issue asks for config.patch to skip SIGUSR1 restarts when changed paths such as browser.profiles.* are browser profile changes that the runtime can hot-reload.

## Best Possible Solution

Make browser profile/default-profile reloadability explicit through plugin-owned granular reload metadata or a generic specificity-aware reload planner, keep structural browser settings restart-required, and add regression coverage for browser.profiles.* in both reload planning and config.patch direct-restart scheduling.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a config.patch that changes browser.profiles.<name>.cdpUrl with gateway.reload.mode set to hot: current main classifies the path under the broad browser restart prefix and schedules direct SIGUSR1.

## Solution Assessment

Yes in direction, but the safest implementation is narrower than a hard-coded core prefix list. The maintainable fix should refine plugin-owned reload metadata or planner specificity so profile-only browser writes are no-op/hot while structural browser config still restarts.
