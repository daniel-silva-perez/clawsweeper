## Summary

Keep open: current main still classifies browser profile config changes as restart-required. `config.patch` now consults the reload plan, but `browser.profiles.*` still inherits the browser plugin's broad `browser` restart prefix, so the specific reported restart/SIGUSR1 behavior remains unresolved.

## What This Changes

The issue asks for `config.patch` to avoid SIGUSR1/full gateway restarts when only safe browser profile config paths such as `browser.profiles.<name>.cdpUrl` change.

## Best Possible Solution

Implement a narrow reload contract for browser profile paths that are safe to apply without a full gateway restart, backed by reload-plan and `config.patch` regression tests, while preserving restart/reconcile behavior for browser settings that affect managed processes or security policy.

## Repair Work Prompt

Fix the browser-profile `config.patch` restart bug tracked by https://github.com/openclaw/openclaw/issues/43803. Current main still has `extensions/browser/plugin-registration.ts` declaring only `restartPrefixes: ["browser"]`, and `src/gateway/config-reload-plan.ts` matches plugin restart prefixes before hot/no-op prefixes, so `browser.profiles.<name>.cdpUrl` remains restart-required and `config.patch` can still schedule SIGUSR1 in hot mode. Implement a narrow fix: allow safe browser profile-scoped changes, at minimum `browser.profiles.<name>.cdpUrl` and likely `browser.defaultProfile` or profile metadata that runtime reads dynamically, to avoid full gateway restart. Do not make all `browser.*` hot/no-op. Preserve restart/reconcile behavior for browser executable paths, SSRF policy, launch/service settings, and other security-sensitive or process-affecting browser config. Check whether rule precedence must prefer more-specific plugin prefixes over broad plugin restart prefixes; add tests that would fail if `{ restartPrefixes: ["browser"], noopPrefixes/hotPrefixes: ["browser.profiles"] }` still resolves to restart. Add focused regression coverage in `src/gateway/config-reload.test.ts` and handler-level `config.patch` coverage for hot and hybrid modes so browser profile patches do not call `scheduleGatewaySigusr1Restart()` or queue a full restart when the plan is safe. Keep `config.apply` behavior unchanged unless the tests prove the same helper is intentionally shared. Update browser/gateway docs only if the public reload contract changes. Add a single user-facing changelog fix entry if repo policy requires it, crediting the reporter or prior contributor, not bots.
