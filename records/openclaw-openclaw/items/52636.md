## Summary

Keep open. Current main already contains the `config.openFile` hardening slice, but the PR is still broad, security-sensitive, and not safely mergeable: the branch targets an obsolete browser path and has concrete defects in loopback throttling, webhook verification, and Shield runtime state handling.

## What This Changes

The PR adds a new `src/security/shield` runtime subsystem and modifies gateway config file opening, auth rate limiting, default HTTP security headers, and browser Playwright evaluate filtering.

## Best Possible Solution

Split the work into focused, separately reviewable security changes. Keep the shipped `config.openFile` hardening as the canonical solution for that slice, move any browser evaluate restriction to the active browser plugin path, and evaluate loopback throttling, global header policy, and any Shield-style runtime subsystem against OpenClaw’s threat model with targeted tests and maintainer/secops review.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Move evaluate hardening to the active browser plugin path:** `src/browser/pw-tools-core.interactions.ts:305-307`
  - body: Current main no longer has `src/browser`; the active `evaluateViaPlaywright()` implementation lives under `extensions/browser/src/browser`. This branch adds the claimed critical filter to the obsolete path, so after rebase it either conflicts or leaves the real browser evaluate surface unchanged.
  - confidence: 0.88
- **[P1] Enforce the loopback 10x threshold in the lockout path:** `src/gateway/auth-rate-limit.ts:214-215`
  - body: `recordFailure()` still sets the normal `maxAttempts` lockout for loopback clients, but `check()` returns from `isExempt()` before honoring that lock while the count is below `maxAttempts * 10`. Later failures stop being recorded while `lockedUntil` is active, so the advertised 10x ceiling is never reached.
  - confidence: 0.9
- **[P2] Reject malformed webhook signatures without throwing:** `src/security/shield/webhook-dispatch.ts:63`
  - body: `verifyWebhookSignature()` compares hex string lengths before decoding, then passes decoded buffers to `timingSafeEqual()`. A same-length non-hex signature can decode to a different buffer length and throw instead of failing closed with `false`.
  - confidence: 0.86
- **[P2] Refresh escalation state when circuits recover:** `src/security/shield/gateway-shield.ts:174-175`
  - body: `recordSuccess()` updates the circuit map but never recomputes `this.escalation`. Once a circuit closes, `getStatus()` and `getSummary()` can keep reporting an escalated pause state until another failure opens a circuit.
  - confidence: 0.84
- **[P2] Collapse health metrics by function before storing scores:** `src/security/shield/gateway-shield.ts:286-287`
  - body: `aggregateByFunctionAndMinute()` returns one bucket per function and minute, but this loop stores each bucket by function name only. If buckets arrive out of chronological order, an older minute can overwrite newer health and make throttling depend on insertion order.
  - confidence: 0.78
- **[P2] Drop stale health scores when metrics leave the window:** `src/security/shield/gateway-shield.ts:301`
  - body: `computeHealthScores()` filters to recent metrics but only updates functions present in the new aggregate. Functions with no recent metrics keep old `CRITICAL` or `CIRCUIT_OPEN` scores, so `shouldThrottle()` can keep throttling recovered or idle functions.
  - confidence: 0.82

## Security Review

Summary: The branch changes gateway/browser security policy and adds a runtime Shield subsystem, but several claimed protections are stale, bypassable, or can fail open/incorrectly.

  - confidence: 0.86
- **[high] Loopback throttling remains bypassable:** `src/gateway/auth-rate-limit.ts:214`
  - body: The proposed loopback 10x cap is undercut by the normal lockout path, which stops additional failure recording before the 10x attempt threshold is reached.
  - confidence: 0.9
- **[medium] Malformed webhook signatures can throw:** `src/security/shield/webhook-dispatch.ts:63`
  - body: Webhook signature verification can throw for same-length malformed hex input instead of returning `false`, creating an exception path where security verification should fail closed.
  - confidence: 0.86
- **[medium] Shield runtime state can remain stale:** `src/security/shield/gateway-shield.ts:174`
  - body: Escalation and health-score state can persist after recovery or after metrics age out, which is risky for a runtime subsystem that may pause or throttle gateway behavior.
  - confidence: 0.82
