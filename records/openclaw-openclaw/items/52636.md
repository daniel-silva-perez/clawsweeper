## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Move evaluate hardening to the active browser path:** `src/browser/pw-tools-core.interactions.ts:305-307`
  - body: Current main no longer has `src/browser`; `evaluateViaPlaywright()` lives under `extensions/browser/src/browser`. This filter is added to a stale path, so after rebase it either conflicts or leaves the active browser evaluate surface unchanged.
  - confidence: 0.88
- **[P1] Enforce the loopback cap in the lockout path:** `src/gateway/auth-rate-limit.ts:214-215`
  - body: `recordFailure()` still locks loopback clients at the normal `maxAttempts`, but `check()` returns from the loopback exemption path until the count reaches `maxAttempts * 10`. Because later failures are ignored while locked, the intended 10x ceiling is never reached.
  - confidence: 0.9
- **[P2] Reject malformed webhook signatures without throwing:** `src/security/shield/webhook-dispatch.ts:63`
  - body: This compares hex string lengths before decoding, then passes decoded buffers to `timingSafeEqual()`. A same-length non-hex signature can decode to a different buffer length and throw instead of failing closed with `false`.
  - confidence: 0.86
- **[P2] Refresh escalation state when circuits recover:** `src/security/shield/gateway-shield.ts:174-175`
  - body: `recordSuccess()` updates the circuit map but never recomputes `this.escalation`. Once a circuit closes, status APIs can keep reporting an escalated pause state until another failure opens a circuit.
  - confidence: 0.84
- **[P2] Collapse health buckets before storing scores:** `src/security/shield/gateway-shield.ts:286-287`
  - body: `aggregateByFunctionAndMinute()` returns one bucket per function and minute, but this stores each bucket by function name only. If buckets are not chronological, an older minute can overwrite newer health and make throttling depend on insertion order.
  - confidence: 0.78
- **[P2] Expire health scores when metrics age out:** `src/security/shield/gateway-shield.ts:301`
  - body: `computeHealthScores()` filters to recent metrics but only updates functions present in the new aggregate. Functions with no recent metrics keep old `CRITICAL` or `CIRCUIT_OPEN` scores, so idle or recovered functions can remain throttled from stale state.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff changes gateway/browser security policy and adds a runtime Shield subsystem, but several claimed protections are stale, bypassable, or fail incorrectly.

Concerns:

- **[high] Browser evaluate hardening targets an obsolete path:** `src/browser/pw-tools-core.interactions.ts:305`
  - body: The claimed browser evaluate mitigation is added under `src/browser`, while current main’s active browser plugin implementation lives under `extensions/browser`. Until rebased and moved, the active evaluate surface is not hardened by this change.
  - confidence: 0.86
- **[high] Loopback throttling remains bypassable:** `src/gateway/auth-rate-limit.ts:214`
  - body: The proposed loopback 10x cap is undercut by the normal lockout path, which stops additional failure recording before the 10x attempt threshold is reached.
  - confidence: 0.9
- **[medium] Malformed webhook signatures can throw:** `src/security/shield/webhook-dispatch.ts:63`
  - body: Webhook signature verification can throw for same-length malformed hex input instead of returning `false`, creating an exception path where security verification should fail closed.
  - confidence: 0.86
- **[medium] Shield runtime state can remain stale:** `src/security/shield/gateway-shield.ts:174`
  - body: Escalation and health-score state can persist after recovery or after metrics age out, which is risky for a runtime subsystem that may pause or throttle gateway behavior.
  - confidence: 0.82
