## Summary

Keep open. Current main independently covers the `config.openFile` shell-injection slice, but it does not implement this PR's broader Shield runtime layer, finite loopback throttling, browser-evaluate blocklist, or common COOP/CORP/X-Frame header changes. The PR is also broad and security-sensitive, with review comments still pointing at proposed Shield/runtime correctness issues, so it needs maintainer/secops review and likely splitting rather than conservative cleanup closure.

## Best Possible Solution

Keep this PR open for explicit maintainer and secops review, but do not land it as one broad bundle. The best path is to split the work: keep the already-shipped `config.openFile` hardening as the canonical fix for that slice, separately evaluate loopback throttling, browser evaluate restrictions, and header policy against OpenClaw's trust model, and require a dedicated design/security review plus targeted tests before adding any Shield-style runtime subsystem.
