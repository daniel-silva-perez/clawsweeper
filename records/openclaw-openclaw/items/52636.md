## Summary

Keep open. Current main only covers the `config.openFile` hardening slice; the remaining branch is broad, unmergeable against main, and still has concrete security/correctness blockers that require maintainer and secops review before any merge path.

## What This Changes

The PR adds a new `src/security/shield` runtime subsystem and modifies gateway config file opening, auth rate limiting, default HTTP security headers, and browser Playwright evaluate filtering.

## Best Possible Solution

Split the work into focused security PRs: keep the shipped `config.openFile` hardening as canonical, move any browser evaluate mitigation into `extensions/browser`, and evaluate loopback throttling, headers, and any Shield runtime subsystem separately against OpenClaw's threat model with targeted tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Enforce the loopback 10x threshold in the lockout path:** `src/gateway/auth-rate-limit.ts:214-215`
  - body: `recordFailure()` still sets the normal `maxAttempts` lockout for loopback clients, while `check()` allows loopback until `maxAttempts * 10`. Once the normal lockout is set, further failures are ignored, so localhost failures never reach the intended cap.
  - confidence: 0.9
- **[P2] Move evaluate hardening to the active browser path:** `src/browser/pw-tools-core.interactions.ts:305-307`
  - body: Current main no longer has `src/browser`; `evaluateViaPlaywright()` lives under `extensions/browser/src/browser`. Adding the blocked-pattern filter here either conflicts on rebase or leaves the active browser evaluate surface unchanged.
  - confidence: 0.88
- **[P2] Reject malformed webhook signatures without throwing:** `src/security/shield/webhook-dispatch.ts:63`
  - body: `verifyWebhookSignature()` compares hex string lengths before decoding, then passes decoded buffers to `timingSafeEqual()`. A same-length non-hex signature can decode to a different buffer length and throw instead of failing closed with `false`.
  - confidence: 0.86
- **[P2] Refresh escalation state when circuits recover:** `src/security/shield/gateway-shield.ts:174-175`
  - body: `recordSuccess()` updates the circuit state but never recomputes `this.escalation`. If a previously open circuit closes, `getStatus()` and `getSummary()` can keep reporting `PARTIAL_PAUSE` or `FULL_PAUSE` until another circuit opens.
  - confidence: 0.84
- **[P2] Collapse health metrics by function before storing scores:** `src/security/shield/gateway-shield.ts:287`
  - body: `aggregateByFunctionAndMinute()` returns one bucket per function and minute, but this loop stores each bucket under only the function name. If buckets are not chronological, an older minute can overwrite newer health and make throttling depend on insertion order.
  - confidence: 0.78
- **[P2] Drop stale health scores when metrics leave the window:** `src/security/shield/gateway-shield.ts:301`
  - body: `computeHealthScores()` filters to recent metrics but only updates functions present in the new aggregate. Functions with no recent metrics keep prior `CRITICAL` or `CIRCUIT_OPEN` scores, so idle or recovered functions can remain throttled from stale state.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff changes gateway/browser security policy and adds Kairos-attributed Shield runtime code, but several claimed protections are stale, bypassable, or fail incorrectly.

Concerns:

- **[high] Browser evaluate hardening targets an obsolete path:** `src/browser/pw-tools-core.interactions.ts:305`
  - body: The claimed evaluate mitigation is added under `src/browser`, while current main's active browser implementation lives under `extensions/browser`; until rebased and moved, the active evaluate surface is not hardened.
  - confidence: 0.86
- **[high] Loopback throttling remains bypassable:** `src/gateway/auth-rate-limit.ts:214`
  - body: The proposed loopback 10x cap is undercut by the normal lockout path, which stops additional failure recording before the 10x attempt threshold is reached.
  - confidence: 0.9
- **[medium] Malformed webhook signatures can throw:** `src/security/shield/webhook-dispatch.ts:63`
  - body: Webhook signature verification can throw for same-length malformed hex input instead of returning `false`, creating an exception path where security verification should fail closed.
  - confidence: 0.86
- **[medium] Shield runtime state can remain stale:** `src/security/shield/gateway-shield.ts:174`
  - body: Escalation and health-score state can persist after recovery or after metrics age out, which is risky for a runtime subsystem that may pause or throttle gateway behavior.
  - confidence: 0.82
- **[low] Kairos-attributed code needs provenance review:** `src/security/shield/SHIELD.md:4`
  - body: The new Shield files state they are based on the Kairos Shield Protocol and by Kairos Lab, but the PR does not add local provenance or license evidence for the adapted code.
  - confidence: 0.76
