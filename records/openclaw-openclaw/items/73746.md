## Summary

Keep open. Current main still lacks the richer task states, so the PR is not obsolete, but latest PR head is not ready to merge because the new statuses are added to the type model while `normalizeTaskStatus` still drops them to `queued`; the user-facing docs and changelog also have not been updated.

## What This Changes

The PR adds `awaiting_approval` and `waiting_external` task statuses plus lifecycle/operational formatting across the task registry, Task Flow projection, audit/maintenance, CLI and auto-reply rendering, and focused tests.

## Best Possible Solution

Make the intermediate statuses first-class anywhere task status values are parsed, normalized, displayed, audited, documented, and announced, while preserving the Task Flow compatibility projection to `running`; then land this PR or an equivalent replacement after focused task tests, docs checks, changelog audit, and the changed gate pass.

## Reproduction Assessment

Yes. On PR head, create or update a task with `status: "awaiting_approval"` or `status: "waiting_external"`; that path reaches `normalizeTaskStatus`, which still accepts only the old status set and falls back to `queued`.

## Solution Assessment

No, not in the current form. The product direction is maintainable, but the safer implementation is to centralize or update status normalization, add create/update-path regression coverage, and align docs and changelog before merge.
