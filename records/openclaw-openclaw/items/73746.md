## Summary

Keep open. Current main still has the old task status model, and this PR is the active implementation path, but PR head still has a blocking normalization defect that can silently turn the new active statuses back into `queued`; CI for the head SHA is also not green.

## What This Changes

The PR branch adds intermediate active task statuses, propagates active-state handling through task registry, Task Flow, CLI, and auto-reply surfaces, and adds lifecycle/operational formatting with focused tests.

## Best Possible Solution

Make the intermediate statuses first-class everywhere status values are parsed, normalized, displayed, audited, and documented, while preserving the Task Flow compatibility projection to `running`; then land this PR or an equivalent patch after focused task tests and the changed gate pass.

## Reproduction Assessment

Yes. On PR head, any path that passes `awaiting_approval` or `waiting_external` through `createTaskRecord` or run-scoped status updates reaches `normalizeTaskStatus`, which still maps unrecognized values to `queued`; source inspection gives a high-confidence reproduction path without mutating the checkout.

## Solution Assessment

No, not in the current form. The implementation direction is maintainable, and the Greptile transition comments appear addressed, but the status normalizer and user-facing docs must be updated before this is the narrowest safe solution.
