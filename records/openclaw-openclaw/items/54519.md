## Summary

Keep open. Current main still lacks a hard per-account read-only channel mode: shared channel config has no receive-only/outbound-deny field, WhatsApp config/schema still has no `readOnly` key, gateway send and poll paths still call outbound delivery directly, and the WhatsApp transport still has message, poll, reaction, read receipt, typing, and presence send paths without an account-level deny gate. The related #71382 PR is useful implementation context, but it is open and WhatsApp-scoped, so it should not close this broader cross-channel request yet.

## What This Changes

Requested a per-account channel configuration option that allows inbound messages while blocking all outbound delivery at the gateway or transport level.

## Best Possible Solution

Define a maintainer-approved channel/account outbound policy in shared config, schema/help, docs, and UI metadata, then enforce it before every outbound surface that can leave the gateway or a channel transport. Coordinate #71382 as the WhatsApp subset, but close this issue only after the generic policy lands or maintainers explicitly split the remaining scope into narrower tracked work.
