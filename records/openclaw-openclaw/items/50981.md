## Summary

Keep this PR open. Current main still has the timeout propagation gaps the PR targets, so it is not obsolete or implemented on main. However, the proposed patch is not safe to close as shipped or merge as-is because it forwards string timeouts into a numeric gateway contract, changes fire-and-forget runtime budget semantics, and leaves unresolved budget alignment issues for session locks and A2A helper waits.

## Best Possible Solution

Keep the PR open for maintainer revision or replacement. A mergeable fix should send numeric `agent.timeout` values only when the caller is actually waiting, preserve current fire-and-forget target runtime semantics unless maintainers intentionally change them, count session-lock wait against the effective run/caller budget, align A2A target and wait budgets, and add regression coverage for gateway validation, `timeoutSeconds: 0`, A2A helper waits, and contended session locks.

## Risks / Open Questions

- Closing this PR as implemented would hide a real remaining bug: current main still does not propagate `sessions_send` caller timeout into the target run or session lock wait.
- Merging this PR as-is risks breaking real gateway paths because it sends `agent.timeout` as a string while the gateway schema requires an integer.
- Merging this PR as-is risks changing `timeoutSeconds: 0` from non-blocking caller behavior into a hard 30-second target run cap.
- The lock-wait change needs a remaining-budget design; otherwise lock contention can still make the caller's timeout feel ignored.
- The A2A helper path needs aligned target-run and wait budgets, especially when `timeoutMs` exceeds the existing 60-second wait cap.
