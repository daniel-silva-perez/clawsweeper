## Summary

Keep this PR open. Current main still has the timeout propagation gaps the PR targets: `sessions_send` and the A2A helper start nested `agent` runs without forwarding the caller timeout, and the embedded runner still acquires the session write lock without `timeoutMs`, so the lock helper can fall back to its 10s wait. The PR is not obsolete, but the provided diff/review discussion shows it is not mergeable as-is because it sends string `agent.timeout` values into an integer gateway contract, changes fire-and-forget runtime budget semantics, and leaves lock/wait budget alignment unresolved.

## Best Possible Solution

Keep the PR open for maintainer revision or replacement. A mergeable fix should forward numeric `agent.timeout` only when the caller is actually waiting, preserve fire-and-forget target runtime semantics unless maintainers intentionally change them, account for session-lock wait inside the effective run/caller budget, align A2A target and wait budgets, and add regression coverage for gateway validation, `timeoutSeconds: 0`, contended session locks, and A2A helper waits.

## Risks / Open Questions

- Closing this PR as implemented would hide a real remaining bug: current main still does not propagate the caller timeout into nested target runs or session lock waits.
- Merging the PR as-is risks breaking real gateway calls because it sends `agent.timeout` as a string while the protocol requires an integer.
- Merging the PR as-is risks changing `timeoutSeconds: 0` from non-blocking caller behavior into a short target run cap.
- The lock-wait fix needs a remaining-budget design so lock contention does not consume the caller's whole wait and then still allow another full model run.
- The A2A helper path needs target-run and wait budgets aligned, especially above the existing 60s wait cap.
