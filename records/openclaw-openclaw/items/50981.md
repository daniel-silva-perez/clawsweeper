## Summary

Keep this PR open. Current main still has the timeout propagation gaps this PR targets: `sessions_send` and the A2A helper start nested `agent` runs without forwarding the caller timeout, and the embedded runner still waits for the session write lock without `timeoutMs`. The PR is not obsolete, but the provided diff is not merge-ready because review comments identify protocol validation, fire-and-forget semantics, and budget-accounting issues.

## Best Possible Solution

Keep the PR open for maintainer revision or replacement. A mergeable fix should forward numeric `agent.timeout` only when appropriate, preserve `timeoutSeconds: 0` target-run semantics unless maintainers intentionally change them, account for session-lock wait inside the effective run/caller budget, align A2A target and wait budgets, and add regression coverage for gateway validation, fire-and-forget sends, contended session locks, and A2A helper waits.

## Risks / Open Questions

- Closing this PR as implemented would hide a real remaining bug in current main.
- Merging the current diff as-is would likely make real nested `agent` calls fail gateway validation because it sends `timeout` as a string while the protocol requires an integer.
- Merging the current diff as-is would change `timeoutSeconds: 0` fire-and-forget target runtime semantics by forcing a short target timeout.
- Passing the full run timeout into session-lock acquisition without a remaining-budget design can still let lock contention consume the caller wait and then allow more model runtime.
- The A2A helper still needs target-run and wait budgets aligned, especially above the existing 60-second wait cap.
