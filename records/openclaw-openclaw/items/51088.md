## Summary

Keep open. Current main supports iMessage acknowledgements through the recommended BlueBubbles channel and exposes generic inbound hooks, but the still-documented legacy `channels.imessage`/`imsg` path requested here has no channel-owned read-receipt or tapback acknowledgement support; because that path is deprecated but still documented, the remaining decision is maintainer product direction rather than conservative cleanup closure.

## What This Changes

The issue requests native acknowledgement support for the iMessage channel, either through read receipts, tapback acknowledgements wired to `messages.ackReactionScope`, or an inbound hook that avoids silent queued-message delivery.

## Best Possible Solution

Make the supported iMessage acknowledgement contract explicit. If legacy `channels.imessage` remains supported, add channel-owned acknowledgement settings, schema/docs, capabilities, inbound behavior, and focused tests; if BlueBubbles is the intended answer, document that read receipts and tapback acknowledgements require `channels.bluebubbles` and explicitly scope legacy `imsg` out.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to configure legacy `channels.imessage` with top-level `messages.ackReactionScope` and trace inbound handling: the legacy config/schema has no channel-owned ack/read field and the monitor dispatch path contains no acknowledgement call.

## Solution Assessment

Unclear. Native acknowledgement in deprecated `imsg` may be the right fix only if maintainers want to keep expanding that path; otherwise the safer solution is to document BlueBubbles as the iMessage acknowledgement path and explicitly leave legacy `imsg` without this feature.
