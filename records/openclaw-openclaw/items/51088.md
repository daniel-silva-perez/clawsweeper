## Summary

Keep open. Current main still does not provide native acknowledgement for the legacy iMessage channel: the legacy plugin advertises chat/media only, its config type and strict schema omit ack/read-receipt settings, and the inbound monitor records and dispatches accepted messages without sending a tapback or read receipt. BlueBubbles does support read receipts and tapbacks, but it is documented and implemented as the recommended separate iMessage path rather than a fix for the legacy imsg channel.

## What This Changes

The issue requests native iMessage message acknowledgement through read receipts or tapback-style acknowledgement so senders can tell inbound messages were received while the agent is processing.

## Best Possible Solution

Decide the supported iMessage path first. If legacy `channels.imessage` remains supported, add channel-owned acknowledgement after routing/access checks with config, schema, docs, and direct/group tests; if BlueBubbles is the intended answer, document that read receipts and tapbacks require BlueBubbles and explicitly scope legacy imsg out.
