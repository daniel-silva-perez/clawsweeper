## Summary

Keep open. Current main supports iMessage acknowledgements through the recommended BlueBubbles path and now defaults queued messages to steering, but the still-documented legacy `channels.imessage` / `imsg` path has no native acknowledgement config, capability, schema support, or inbound tapback/read-receipt behavior. Because that legacy channel is explicitly deprecated, the remaining question is maintainer product direction rather than a conservative cleanup close.

## What This Changes

The issue requests native iMessage acknowledgement support, such as read receipts or tapback acknowledgements, so senders can tell inbound messages reached the agent while processing is queued.

## Best Possible Solution

Make the supported iMessage direction explicit. If legacy `channels.imessage` remains supported, add channel-owned acknowledgement with config/schema/docs and direct/group tests; if BlueBubbles is the intended answer, document that read receipts and tapback acknowledgements require `channels.bluebubbles` and explicitly scope legacy `imsg` out.

## Risks / Open Questions

- Legacy `imsg` is documented as deprecated and may be removed, so adding acknowledgement support there could conflict with the intended BlueBubbles migration path.
- The acknowledgement mechanism needs maintainer UX/security judgment: tapbacks add conversation noise, read-receipt activation can steal focus and require Accessibility/Automation permissions, and current repo code does not establish a reliable native `imsg` read/tapback API.
- The generic `message_received` hook can observe inbound messages, but it does not provide channel-owned sender-visible read/tapback semantics for legacy iMessage.
