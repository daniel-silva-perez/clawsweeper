## Summary

Keep open. Current main still does not implement native acknowledgement for the legacy iMessage channel: the legacy channel advertises no reaction/read capability, its typed config/schema lacks ack or read-receipt settings, and the inbound monitor records and dispatches messages without any tapback or read-receipt action. BlueBubbles does support read receipts and tapbacks, but that is a separate recommended iMessage path and does not by itself close the requested legacy imsg behavior.

## What This Changes

Request native acknowledgement for the legacy iMessage channel via read receipts or tapback-style ack so senders can see that queued inbound messages were received.

## Best Possible Solution

Keep this issue open for maintainer product decision. If legacy channels.imessage remains supported, add channel-owned acknowledgement after access/mention gating and before dispatch, with typed config/schema/docs, echo/reflection safeguards, and direct/group tests. If BlueBubbles is the intended supported path for iMessage read receipts and tapbacks, document that explicitly on https://docs.openclaw.ai/channels/imessage and close only after that decision is made.
