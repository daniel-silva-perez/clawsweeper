## Summary

Keep open. Current main still lacks native acknowledgement for the legacy iMessage channel: the bundled imsg plugin has no reaction/read-receipt capability, typed config/schema do not expose ack/read options, and the inbound monitor records and dispatches accepted messages without sending a tapback or read receipt. BlueBubbles does support read receipts and tapbacks, but it is a separate recommended iMessage path rather than an implementation of this legacy imsg request.

## What This Changes

Request native acknowledgement for inbound iMessage messages, via ackReactionScope tapbacks/read receipts or a message:inbound hook, so senders can tell queued messages were received.

## Best Possible Solution

Decide the supported iMessage path first. If legacy channels.imessage remains supported, add channel-owned acknowledgement after routing/access checks with config/schema/docs and direct/group tests; if BlueBubbles is the intended answer, document that read receipts and tapbacks require BlueBubbles and explicitly scope legacy imsg out.
