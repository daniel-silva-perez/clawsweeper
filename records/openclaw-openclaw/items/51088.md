## Summary

Keep open. Current main supports read receipts and configured ack tapbacks for the recommended BlueBubbles iMessage path, but not for the legacy `imessage` channel named by the report. The legacy iMessage config/schema/capabilities and inbound monitor still have no `ackReactionScope`, tapback, or read-receipt handling, so the reported gap remains for users on `channels.imessage`.

## Best Possible Solution

If maintainers still want the legacy `imessage` channel to grow features, implement this in `extensions/imessage`: add supported config/schema for ack/read behavior, trigger a native read receipt or tapback before dispatch/queue processing according to `messages.ackReactionScope`, and add direct/group scope tests plus remove-after-reply behavior if tapbacks are used. If legacy `imsg` should stay frozen, the better product path is to document that acknowledgment support is available through BlueBubbles and close only after explicit maintainer triage, not as already implemented for `channels.imessage`.
