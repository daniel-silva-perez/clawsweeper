## Summary

Keep open. Current main supports iMessage acknowledgements through the recommended BlueBubbles channel and exposes generic inbound hooks, but the still-documented legacy `channels.imessage`/`imsg` path requested here has no channel-owned read-receipt or tapback acknowledgement support. Because that legacy path is deprecated but still documented, the remaining decision is maintainer product direction rather than conservative cleanup closure.

## What This Changes

Add native acknowledgement for inbound iMessage messages, via read receipts or tapback-style reactions, or expose a hook that lets users implement that behavior without polling.

## Best Possible Solution

Make the supported iMessage acknowledgement contract explicit. If legacy `channels.imessage` remains supported, add channel-owned acknowledgement settings, schema/docs, capabilities, inbound behavior, and focused tests; if BlueBubbles is the intended answer, document that read receipts and tapback acknowledgements require `channels.bluebubbles` and explicitly scope legacy `imsg` out.
