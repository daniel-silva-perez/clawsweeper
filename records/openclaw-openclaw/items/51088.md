## Summary

Keep open. Current main has BlueBubbles support for iMessage read receipts/tapbacks and a generic inbound message hook, but the legacy `channels.imessage`/`imsg` path described by this issue still has no native acknowledgement setting, capability, schema support, or inbound tapback/read-receipt behavior. Because legacy imsg is documented as deprecated but still supported, the remaining decision is maintainer product direction rather than an automatic close or repair.

## What This Changes

Request native iMessage inbound acknowledgement through read receipts or tapbacks via `ackReactionScope`, or an inbound message hook that can trigger custom acknowledgement without polling.

## Best Possible Solution

Make the supported iMessage direction explicit. If legacy `channels.imessage` remains supported, add channel-owned acknowledgement with config/schema/docs plus direct/group tests; if BlueBubbles is the intended answer, document that read receipts and tapback acknowledgements require `channels.bluebubbles` and explicitly scope legacy imsg out.

## Risks / Open Questions

- Legacy `imsg` is documented as legacy and may be removed, so adding native acknowledgement there could conflict with the intended BlueBubbles migration path.
- The acknowledgement mechanism needs product/UX/security judgment: tapbacks create conversation noise, read-receipt activation can steal focus and require Accessibility/Automation permissions, and native macOS API availability is not established in current code.
- The existing generic `message_received` hook may be enough for some custom workarounds, but it is fire-and-forget and does not provide channel-owned read/tapback semantics for legacy iMessage.
