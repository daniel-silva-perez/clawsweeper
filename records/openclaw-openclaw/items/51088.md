## Summary

Keep open. Current main covers iMessage acknowledgements through the recommended BlueBubbles channel and now has generic inbound message hooks, but the still-documented legacy `channels.imessage`/`imsg` path requested here has no channel-owned read receipt or tapback acknowledgement support. Because that legacy path is deprecated but not removed, this is a maintainer product-direction decision rather than a conservative cleanup close.

## What This Changes

Request native acknowledgement for iMessage inbound messages, via read receipts, tapback-backed `ackReactionScope`, or an inbound hook, so senders can tell queued messages were received.

## Best Possible Solution

Make the supported iMessage acknowledgement contract explicit. If legacy `channels.imessage` remains supported, add channel-owned acknowledgement with config/schema/docs, capabilities, inbound behavior, and tests; if BlueBubbles is the intended answer, document that read receipts and tapback acknowledgements require `channels.bluebubbles` and scope legacy `imsg` out.

## Risks / Open Questions

- Legacy `imsg` is documented as legacy and may be removed, so adding acknowledgement support there could conflict with the intended BlueBubbles migration path.
- The acknowledgement mechanism needs product, UX, and privacy judgment: tapbacks add conversation noise, read-receipt activation can steal focus and requires macOS permissions, and a native non-focus-stealing API is not established in the current code.
- The generic `message_received` hook may be sufficient for some custom scripts, but it is fire-and-forget and does not provide channel-owned read receipt or tapback semantics for legacy iMessage.
