## Summary

Keep this PR open. Current main still treats any non-template `session.store` as one default-agent store, so the #54435 behavior is not implemented; the PR targets the right narrow bug and includes useful before/after proof, but it is not merge-ready because the implementation has moved and the branch still has correctness issues around resolved literal paths and agent-id selection.

## What This Changes

The PR changes gateway session-store combination so literal `agents/<id>/sessions/sessions.json` store paths discover sibling per-agent stores scoped to that agents root, with regression tests for the literal-path case.

## Best Possible Solution

Adapt the fix to the current `src/config/sessions/combined-store-gateway.ts` path: literal store paths that resolve to `agents/<id>/sessions/sessions.json` should discover sibling stores under the same agents root, preserve the resolved literal path for downstream lookups, derive the target agent id from the on-disk path, and keep the old single-store behavior for non-agent-layout literal paths.

Use the agent id encoded in the store path or discovered directory for literal-layout targets.
