## Summary

Keep this PR open. Current main still short-circuits literal non-template `session.store` paths before multi-agent discovery, so the linked `sessions.list` bug remains real; the PR direction is useful, but the branch needs adaptation to the current `src/config/sessions` split and two correctness fixes before merge.

## What This Changes

The PR changes gateway session-store aggregation so literal `agents/<id>/sessions/sessions.json` store paths discover sibling stores under the same agents root and adds regression tests for the literal-path case.

## Best Possible Solution

Adapt the fix to current main in `src/config/sessions/combined-store-gateway.ts` and, if needed, `src/config/sessions/targets.ts`: literal paths matching `agents/<id>/sessions/sessions.json` should discover sibling stores under that same agents root, preserve single-store behavior for non-agent-layout literals, return a resolved literal store path for downstream lookups, and derive literal-layout agent ids from the disk path when canonicalizing legacy keys.

## Reproduction Assessment

Yes. The linked issue and PR comments give an API-backed count mismatch, and current main source shows the reproducible path: configure `session.store` as a literal `.../agents/main/sessions/sessions.json` with a sibling agent store, then `sessions.list` loads only the literal main store.

## Solution Assessment

No. The proposed direction is the narrow maintainable fix, but this branch is not the best merge candidate until it is adapted to the current config/session modules and the resolved-path plus agent-id canonicalization issues are fixed.
