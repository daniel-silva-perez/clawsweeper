## Summary

Current main still matches the reported Google Chat provider-boundary bug. `sendGoogleChatMessage` accepts any truthy `thread` string, writes it directly to `body.thread.name`, and enables `messageReplyOption`, while several outbound paths still pass raw `threadId`, `replyTo`, or `replyToId` values through. Related `replyToMode: "off"` work remains open in #42510 and PR #67055, and the exact validation PR #46519 was closed unmerged, so this issue should not be closed yet.

## Best Possible Solution

Keep this issue open until the Google Chat plugin has a shared thread-routing helper used by every outbound path. It should preserve valid `spaces/{space}/threads/{thread}` resources, omit or convert raw message/session/action IDs before API construction, honor `replyToMode: "off"`, and add regression tests for text, media, message actions, attached-results sends, monitor reply delivery, and streamed/block reply paths. #67055 may be part of the solution if it is expanded or paired with provider-boundary validation.
