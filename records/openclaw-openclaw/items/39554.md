## Summary

Keep this issue open. Current main still accepts arbitrary Google Chat thread strings at the API boundary and several outbound paths still pass raw `threadId`, `replyTo`, `replyToId`, or `[[reply_to_current]]`-derived message ids through to `body.thread.name`, so the reported invalid thread resource-name failure is not fully fixed.

## Best Possible Solution

Keep this issue open until the Google Chat plugin has a shared plugin-owned thread-routing helper used by monitor delivery, block/stream replies, attached-results sends, and message actions. The helper should preserve valid `spaces/{space}/threads/{thread}` resources, prefer the inbound thread resource over `message.name` for explicit current-message replies where possible, omit invalid raw ids before constructing the Chat API request, honor `replyToMode: "off"`, and add regression tests for direct replies, streamed/block replies, media, attached results, and action sends.
