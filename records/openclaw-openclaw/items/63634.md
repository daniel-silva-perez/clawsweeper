## Summary

Keep open. Current main still uses the strict SDK `listTools()` path, so the linked typeless `inputSchema` failure is not implemented on main, and this PR remains useful. The latest patch is still not merge-ready because its raw `client.request()` replacement bypasses MCP SDK 1.29.0 tool metadata caching used by later `callTool()` validation and task-required checks.

## What This Changes

This PR changes bundled MCP tool discovery to parse `tools/list` with a lenient schema, backfill missing root `inputSchema.type`, skip explicit non-object root schemas with a warning, and adds regression tests plus a changelog entry.

## Best Possible Solution

Rework bundle MCP discovery so typeless root object schemas are accepted and normalized while preserving the SDK 1.29.0 semantics that `listTools()` provides for output-schema validation and task-required tools. Add regression coverage for typeless discovery, explicit non-object root skipping, and a call path where a listed tool has `outputSchema` or `execution.taskSupport: "required"`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve MCP SDK tool metadata caching:** `src/agents/pi-bundle-mcp-runtime.ts:170-173`
  - body: Replacing `client.listTools()` with raw `client.request()` skips SDK 1.29.0's `cacheToolMetadata(result.tools)` call. `callTool()` later uses that cache to reject required-task tools and validate `outputSchema` structured output, so tools using those fields will now run through the normal call path and return unvalidated results. Please preserve or reimplement that metadata path when making `tools/list` parsing lenient.
  - confidence: 0.9
