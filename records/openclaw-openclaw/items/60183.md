## Summary

Keep this PR open for normal maintainer review. Current main still computes TaskFlow audit freshness from the flow record only, so the false-positive `stale_running` case the PR targets remains plausible, and the PR’s source/test change is narrow with no blocking correctness or security finding.

## What This Changes

This PR updates TaskFlow audit freshness to consider the latest linked task activity timestamp and adds regression coverage for active managed flows.

## Best Possible Solution

Review and land the narrow TaskFlow audit freshness fix if maintainers agree that recent linked task activity is valid evidence of flow progress; otherwise adjust the helper to count only active linked tasks before merging.

## Reproduction Assessment

Yes. On current main, a running managed flow with stale `updatedAt` and a linked running task whose `lastEventAt` is within the 30-minute freshness window still uses the stale flow timestamp, so `stale_running` can be emitted even while the linked child is active.

## Solution Assessment

Yes, with one semantic note. Taking the maximum of the flow timestamp and linked task activity is the narrowest maintainable fix for the false positive, but maintainers should decide whether terminal linked task activity should count or whether the helper should filter to active tasks only.
