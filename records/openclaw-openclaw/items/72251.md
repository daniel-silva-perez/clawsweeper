## Summary

Keep open. Current main still lacks the backup EOF retry and volatile-file skip behavior, and this PR addresses a valid live-backup failure class, but the patch introduces a concrete `--json` stdout regression that should block merge until fixed.

## What This Changes

The PR adds a backup volatile-file filter, tar EOF retry/backoff with cleanup and enriched errors, `skippedVolatileCount` reporting, command logger plumbing, and focused backup retry/filter tests.

## Best Possible Solution

Revise this PR so non-fatal retry/skip notices do not contaminate machine-readable JSON output, then document the default volatile-skip semantics and merge it only after maintainers accept the backup-content policy and coordinate the separate ENOENT/exclusion-design follow-ups.

## Reproduction Assessment

Yes. The original backup failure has a high-confidence reproduction path from #72249 and upstream tar source behavior; the PR-introduced regression is reproducible by running backup create with --json when a volatile skip or retry notice is emitted.

## Solution Assessment

No, not as currently written. The retry-plus-filter approach is a plausible narrow fix for the EOF race, but the command logging path should suppress stdout notices in JSON mode or route them away from the JSON stream before this is the best mergeable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep retry notices out of JSON output:** `src/commands/backup.ts:25`
  - body: When `opts.json` is true, this still injects `runtime.log` as the archive logger. `createBackupArchive` logs retry and volatile-skip notices before `writeRuntimeJson`, and `runtime.log` writes to stdout, so any skip or retry prefixes human text before the JSON payload and makes `openclaw backup create --json` unparsable. Suppress the default logger in JSON mode or route these notices somewhere that cannot contaminate stdout JSON.
  - confidence: 0.91
