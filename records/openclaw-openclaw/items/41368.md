## Summary

Keep open. Current main still does not expose the proposed Feishu Bitable delete tools, and the narrower open PR #40697 plus issue #55654 do not cover batch record deletion or field deletion. The patch is a focused Feishu plugin change with no supply-chain-sensitive diff, but it still needs maintainer review for destructive tool wording, response shape polish, schema guidance, and targeted coverage before merge.

## What This Changes

This PR adds Feishu Bitable record deletion, batch record deletion, and field deletion helpers, TypeBox schemas, and tool registrations in `extensions/feishu/src/bitable.ts`.

## Best Possible Solution

Keep this PR open for normal maintainer review, rebase it onto current main, preserve the broader batch-record and field-delete scope if maintainers want more than #40697, tighten the model-facing destructive-action wording and parameter hints, return the SDK response details for batch deletes, and add focused Feishu Bitable tool tests before merge.

## Review Findings

Overall correctness: patch is correct

Overall confidence: 0.82

Full review comments:

- **[P3] Restore the token lookup hints on delete schemas:** `extensions/feishu/src/bitable.ts:588-610`
  - body: The three new delete schemas use bare `"Bitable app token"` and `"Table ID"` descriptions, unlike the existing Bitable schemas that tell the model to use `feishu_bitable_get_meta` and the URL `?table=` value. These are destructive tools, so keeping the established lookup hints helps avoid calls against the wrong table.
  - confidence: 0.83
- **[P3] Return the batch delete response records:** `extensions/feishu/src/bitable.ts:473`
  - body: `batchDeleteRecords` reports `count: recordIds.length`, which is only the input size. The Lark SDK returns `data.records` with per-record delete results, so callers lose the API-confirmed IDs and can get a misleading confirmation if the response shape changes or includes partial details.
  - confidence: 0.74
