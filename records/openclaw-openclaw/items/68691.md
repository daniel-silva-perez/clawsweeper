## Summary

Keep open. Current main still creates Docker sandbox containers without `--init`, the standard sandbox command keeps `sleep infinity` as the long-running process, and the related open PR #74083 is a focused implementation candidate but has not merged and still needs maintainer/secops rollout review.

## What This Changes

The issue reports unreaped zombie processes adopted by PID 1 inside Docker sandbox sessions, causing zombie accumulation and possible process-limit exhaustion.

## Best Possible Solution

Add Docker `--init` through the shared sandbox create-args helper with focused regression coverage, then make an explicit rollout decision for already-created containers under secops review.

## Reproduction Assessment

Yes for the configuration defect: current main can be inspected to see Docker sandbox containers are created without `--init` and launched with `sleep infinity` as the long-running process. A live workload stress test would strengthen the accumulation proof but is not needed to show the missing reaper setup remains.

## Solution Assessment

Yes, the related PR's direction is the narrowest maintainable fix for new Docker sandbox containers because it uses Docker's built-in init/reaper at the shared create-args boundary. It should not close this issue until the PR lands and maintainers settle the existing-container rollout path.
