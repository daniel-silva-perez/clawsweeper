## Summary

Keep this open. Current main still creates Docker sandbox containers without an init/reaper and launches the sandbox command as `sleep infinity`, which matches the reporter's later observation that PID 1 is not reaping orphaned zombies; the open related PR is a plausible focused fix but is not merged.

## What This Changes

The issue asks OpenClaw to stop Docker sandbox sessions from accumulating orphaned zombie processes under PID 1 until they threaten the container PID limit.

## Best Possible Solution

New Docker sandbox and sandbox-browser containers should run with a real PID 1 reaper, preferably Docker `--init` or an equivalent init wrapper, with a regression test on `buildSandboxCreateArgs` and an upgrade path for already-running containers that need recreation.

## Reproduction Assessment

Yes: the report includes PPID=1 zombie process snapshots and a follow-up identifying sandbox PID 1 as `sleep infinity`; current main constructs that same Docker container command without `--init`. A before/after stress run would still be useful for merge validation.

## Solution Assessment

Yes for this orphaned-PID1 Docker sandbox path: adding an init/reaper at container creation is the narrow maintainable fix. The remaining implementation detail is ensuring existing long-lived containers are recreated or clearly reported as needing recreation.
