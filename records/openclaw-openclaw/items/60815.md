## Summary

Keep open. Current main has already moved parts of the model override/fallback design forward, but it does not clearly implement every central behavior proposed here, especially the PR's uniquely-bound outbound delivery fallback, and the draft branch mixes runtime fixes with legal/product launch docs. The patch also introduces at least one concrete regression in CLI banner config handling, so it should not be merged as-is.

## What This Changes

This draft PR changes session model override persistence, Discord gateway lifecycle handling, outbound delivery fallback logic, agent directive defaults, gateway health/probe/connect timing, Apple gateway connection tests, README/package discoverability copy, and adds several legal/privacy/launch documents.

## Best Possible Solution

Split the branch into narrow maintainer-reviewable pieces: keep any still-needed runtime hardening aligned with current main's model failover and delivery APIs, preserve existing documented CLI config behavior, and handle the legal/product launch documents in a separate owner-reviewed docs/legal change.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Restore config-backed banner tagline mode:** `src/cli/banner-config-lite.ts:13`
  - body: The patch changes `readCliBannerTaglineMode` to read only `OPENCLAW_CLI_BANNER_TAGLINE_MODE`. The CLI schema and docs expose `cli.banner.taglineMode`, so users who set the banner mode in config would lose that behavior and still see the default/random banner. Keep the config read path or migrate the schema/docs/tests with the behavior change.
  - confidence: 0.87
- **[P3] Pass proxy trust settings into readiness probes:** `src/gateway/server-http.ts:786-788`
  - body: The patch handles all gateway probes before loading config and passes empty trusted proxy settings into `handleGatewayProbeRequest`. That makes `/ready` ignore configured `gateway.trustedProxies` / `allowRealIpFallback` when deciding whether readiness details are safe to reveal behind a local reverse proxy. Keep readiness probes on the config-backed path or load the proxy settings before calling the probe handler.
  - confidence: 0.72
