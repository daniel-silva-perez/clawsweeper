## Summary

Keep open. The target bug is plausible on current main because exec approval followups still wait for a final gateway agent result with an explicit 60s timeout, but this patch drops the existing direct fallback path when the gateway accepts the resumed run and later reports a terminal dispatch/runtime error.

## What This Changes

This PR changes exec approval session followups to return after the gateway accepts the resumed agent request and adds regression assertions for accepted-only behavior.

## Best Possible Solution

Keep the PR open and revise the fix so the background followup no longer blocks on the 60s final wait while still preserving an observable terminal-error path that can trigger direct fallback delivery when the resumed run fails after acceptance.

## Reproduction Assessment

Yes. A focused unit-level reproduction can mock the gateway call returning an accepted ack while the gateway code shows a later terminal error is possible; with this patch, sendExecApprovalFollowup returns true before fallback can run.

## Solution Assessment

No. Switching only to accepted-only semantics is too narrow because it fixes the apparent freeze by discarding the failure signal that currently drives fallback delivery; a safer fix needs a non-blocking terminal observer, agent.wait path, or timeout adjustment that still handles post-accept errors.
