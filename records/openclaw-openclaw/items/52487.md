## Summary

Keep open. The underlying Windows restart race still exists on current main, and this contributor PR targets the right helper, but the proposed PowerShell probe should not merge as-is because it is not cmd-safe for custom/profile-derived task names and runs with user PowerShell profiles enabled.

## What This Changes

The PR adds a PowerShell `Get-ScheduledTask` running-state check to the generated Windows scheduled-task self-restart helper before it retries `schtasks /Run`.

## Best Possible Solution

Keep the narrow Windows restart-race fix, but revise the generated helper to use a profile-free, noninteractive PowerShell state check and pass or validate the task name in a cmd-safe way consistent with the update restart helper. Add focused tests for guard ordering, default and custom task names, unsafe/custom task names, and cleanup behavior before merging.

## Reproduction Assessment

Yes. The issue has a high-confidence code-level reproduction path: on Windows, `openclaw gateway restart` can trigger CLI `/End` then `/Run` while the shutdown self-restart helper independently retries `/Run`; current main still contains both paths.

## Solution Assessment

No, not as currently proposed. The direction is narrow and appropriate, but the implementation should harden the PowerShell probe against cmd parsing/profile side effects before it is the best maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Quote or validate the task name before the probe:** `src/infra/windows-task-restart.ts:38`
  - body: The new probe embeds `taskName` directly in a `.cmd` `powershell -Command` line and only doubles PowerShell single quotes. Values from `OPENCLAW_WINDOWS_TASK_NAME` or profile-derived names containing `%`, `&`, or `"` can be expanded or break the outer batch command before PowerShell sees them, so the guard can miss an already-running task or execute unintended batch text. Pass the value through a cmd-safe assignment or validate it like the update restart helper.
  - confidence: 0.88
- **[P2] Run the probe without PowerShell profiles:** `src/infra/windows-task-restart.ts:38`
  - body: `powershell -Command` can load user or system profiles before `Get-ScheduledTask`. Profile output containing `Running` can make `findstr` take the cleanup path when the task is not running, and slow or interactive profile code can block the helper before it reaches `/Run`. Add `-NoProfile` and preferably `-NonInteractive` for this generated restart probe.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff introduces a PowerShell-in-batch execution path whose task-name interpolation is not cmd-safe under crafted local task/profile names.

Concerns:

- **[low] Unescaped task name in generated batch command:** `src/infra/windows-task-restart.ts:38`

A crafted local `OPENCLAW_WINDOWS_TASK_NAME` or profile-derived task name could alter the generated batch command when the restart helper runs, so the probe should pass the value through a safe assignment or reject unsafe names before embedding it.
