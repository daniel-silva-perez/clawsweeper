## Summary

Keep this issue open. Current main still has the reported Google Live browser Talk failure path: the Control UI uses the Google Live browser WebSocket transport for `json-pcm-websocket`, but that transport parses every inbound frame with `JSON.parse(String(data))`, drops parse failures, only leaves `Connecting Talk...` after `setupComplete`, and has no Blob/ArrayBuffer/TextDecoder handling. The reporter's follow-up identifying binary WebSocket frames containing JSON matches the current source and is not covered by the narrower #73460/#73466 interruption cleanup.

## Best Possible Solution

Keep this issue open and fix the Google Live browser Talk transport in `ui/src/ui/chat/realtime-talk-google-live.ts`: decode Blob/ArrayBuffer/ArrayBufferView frames as UTF-8 JSON before parsing, set an appropriate WebSocket binary mode, keep closed-state guards around async message handling, surface protocol/parse errors, and add focused UI transport tests for binary setup/serverContent/toolCall frames, `Connecting Talk...` to `Talk live`, playback scheduling, interruption/stop cleanup, and the post-failure `chat.send` wedge path. Coordinate with #73460/#73466 for scheduled-source cleanup, but do not treat that narrower interruption work as superseding this issue.
