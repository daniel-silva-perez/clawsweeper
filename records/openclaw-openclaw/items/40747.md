## Summary

Keep this PR open. Current main still lacks no-anchor Feishu document insertion, but the proposed patch should not merge as-is because it exposes `index` as an unconstrained number even though it is a 0-based child position.

## What This Changes

The PR makes Feishu `feishu_doc` insert accept optional `after_block_id`, adds an optional root-child `index`, and forwards that index to Descendant insertion when no anchor block is provided.

## Best Possible Solution

Land the Feishu plugin feature after constraining `index` to a non-negative integer, preserving anchored insert behavior, adding focused regression coverage for single and batched root-index inserts, updating the bundled Feishu document skill guidance, and adding a changelog entry that credits the contributor.

## Reproduction Assessment

Yes. Static current-main verification is enough: the schema requires `after_block_id`, and the runtime always resolves an anchor block before insertion, so a no-anchor prepend request has no supported path.

## Solution Assessment

No as proposed. Exposing Feishu Descendant API root insertion through the Feishu plugin is the right narrow boundary, but the schema should reject fractional and negative indexes and the change needs tests/docs/changelog before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Constrain insert index to non-negative integers:** `extensions/feishu/src/doc-schema.ts:43-48`
  - body: `index` is a 0-based child position, but `Type.Number()` allows fractional and negative values through tool validation and leaves users to hit Feishu API errors later. Use `Type.Integer({ minimum: 0 })`, matching the existing `upload_image` positional schema.
