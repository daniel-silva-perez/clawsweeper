## Summary

Keep this PR open. Current main still lacks the cron read-model fields and richer queued manual-run acknowledgment proposed here, and the PR is not superseded by a shipped implementation. Main does preserve the previous completed status while a run is active, but consumers still only get stored `runningAtMs` plus the old `lastStatus` fields, not the explicit `currentStatus` / `lastCompleted*` read model or in-progress ack metadata.

## Best Possible Solution

Keep this PR open for normal maintainer review. The best path is to land a reviewed variant that adds separate cron read-state/read-job types, projects both `list()` and `listPage()` through that read model, keeps derived fields out of writable cron update state, updates gateway/client/docs/tests, and resolves or documents the queued-versus-persisted semantics of the `cron.run` in-progress acknowledgment.

## Risks / Open Questions

- The proposed `cron.run` acknowledgment reports a running-style status before the queued command necessarily persists `runningAtMs`; maintainers should decide whether that means accepted/in-flight or persisted running state.
- Gateway protocol/read schema changes are additive, but generated clients, UI types, docs, and tests need to stay synchronized before merge.
- Derived read-only fields must remain excluded from `CronJobPatchSchema` so clients cannot persist bogus running/completed status.
