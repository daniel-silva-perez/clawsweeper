## Summary

Keep open. Current main has partial safeguards, but the reported daemon/runtime mismatch still exists for model-assisted shell or direct CLI config edits: setup validates env-backed SecretRefs against the current process, direct `openclaw config set` writes can persist them without a daemon-runtime preflight, and gateway startup still fails fast when an active `models.providers.*.apiKey` SecretRef cannot resolve.

## Best Possible Solution

Keep this issue open and implement a daemon-aware guard for active provider/model SecretRefs created by setup or direct CLI config changes. The safest product path is to preflight against the environment the gateway service will actually use, offer to persist the key in `~/.openclaw/.env` or another configured SecretRef provider, and block or roll back changes that would make startup fail. A narrower fallback would make secondary providers non-fatal only when they are provably inactive, but current code treats enabled `models.providers.*.apiKey` refs as active.
