## Summary

Keep open. Current main has useful guardrails for some paths, but the core setup/runtime mismatch still exists: interactive provider setup can validate an env SecretRef against the current shell, normal non-dry-run CLI config writes can persist refs without the same activation preflight, and enabled model provider keys still remain startup-active and fail-fast when unresolved in the daemon runtime.

## What This Changes

The issue asks OpenClaw to prevent model-assisted Moonshot/Kimi provider setup from persisting a daemon-unresolvable env SecretRef that can crash-loop the gateway.

## Best Possible Solution

Add a daemon-aware boundary for provider credential changes: route assistant/TUI/config mutations through gateway-side activation validation or an OpenClaw-managed daemon-readable secret store, and make failed secondary provider additions atomic by blocking, disabling, or rolling back the change before it can break startup.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to persist an enabled models.providers.moonshot.apiKey env SecretRef while MOONSHOT_API_KEY is absent from the daemon runtime; current collection marks it active and startup activation throws the required-secrets failure.

## Solution Assessment

Unclear. The current RPC preflight and daemon install env propagation are good partial fixes, but they do not cover post-install shell/model-assisted writes or settle the startup-fatal policy for newly added secondary providers.
