## Summary

Keep open. Current main has partial guardrails, but the reported setup/runtime mismatch remains: some gateway-mediated writes now reject unresolved active SecretRefs and daemon install can persist env refs visible during install, yet interactive/provider setup and normal non-dry-run CLI config writes can still persist env-backed provider refs that only resolve in the current shell, while enabled model provider keys remain startup-active and unresolved refs still fail gateway startup.

## What This Changes

The issue reports that model-assisted Moonshot/Kimi setup can persist an env-backed provider API key reference that the systemd gateway daemon cannot resolve, causing the gateway to restart-loop even though the primary model remains valid.

## Best Possible Solution

Add a daemon-aware activation boundary for provider credential changes: assistant/TUI/CLI model-provider mutations should either write secrets through an OpenClaw-managed daemon-visible store or validate against the gateway service runtime before commit, and failed secondary provider additions should be blocked, disabled, or rolled back before they can break startup.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to persist an enabled `models.providers.moonshot.apiKey` env SecretRef while `MOONSHOT_API_KEY` is absent from the daemon runtime; current collection treats it as active and startup activation throws the required-secrets failure.

## Solution Assessment

Not applicable as a patch, and the current partial mitigations are not sufficient. RPC preflight and install-time env propagation help some paths, but the durable fix needs a maintainer decision on daemon secret persistence and active-provider startup semantics.
