## Summary

Keep open. Current main has useful partial guardrails for Gateway RPC config writes and secret reload, but the reported post-install setup/runtime mismatch remains: interactive setup and non-dry-run CLI config writes can still persist an env-backed provider SecretRef based on the shell environment, while daemon startup still treats enabled model provider API keys as required secrets and fails when the daemon cannot resolve them.

## What This Changes

The issue asks OpenClaw to prevent model-assisted Moonshot/Kimi setup from saving an env SecretRef that is visible only to an interactive shell and later crash-looping the gateway daemon.

## Best Possible Solution

Add a daemon-aware boundary for post-install provider SecretRef changes: route assistant/TUI/config mutations through daemon-side validation or OpenClaw-managed daemon-readable secret storage, and block or roll back activation when an enabled secondary provider's SecretRef cannot resolve in the actual gateway runtime environment.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: persist models.providers.moonshot.apiKey as an env SecretRef with MOONSHOT_API_KEY absent from the daemon environment; current runtime collection marks it active and startup activation throws the same missing-env startup failure described in the report.

## Solution Assessment

Unclear. The current Gateway RPC preflight is a good partial fix, but it is not the complete solution because direct CLI/model-assisted shell writes and startup fatal policy still leave the daemon crash-loop path available.
