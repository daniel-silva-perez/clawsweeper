## Summary

Keep open. Current main has partial guardrails for Gateway RPC config writes and daemon install-time env propagation, but the reported post-install setup/runtime mismatch remains: interactive setup and non-dry-run CLI config writes can still persist env-backed model provider SecretRefs based on the shell environment, while gateway startup still treats enabled provider API keys as required and fails when the daemon cannot resolve them.

## What This Changes

The issue asks OpenClaw to prevent model-assisted Moonshot/Kimi setup from saving a daemon-unresolvable provider SecretRef that can make the gateway fail on restart.

## Best Possible Solution

Add a daemon-aware boundary for post-install provider SecretRef changes: route assistant/TUI/config mutations through daemon-side validation or OpenClaw-managed daemon-readable secret storage, and block or roll back activation when an enabled secondary provider SecretRef cannot resolve in the actual gateway runtime environment.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: persist models.providers.moonshot.apiKey as an env SecretRef with MOONSHOT_API_KEY absent from the daemon environment; current runtime collection marks the provider key active and startup activation throws the missing-secret startup failure described in the report.

## Solution Assessment

Unclear. The current RPC preflight and daemon install propagation are good partial fixes, but they do not fully solve post-install shell/model-assisted writes or the startup-fatal policy for enabled secondary providers.
