## Summary

Keep this issue open. Current main has partial guardrails for Gateway RPC config writes and install-time service environment capture, but the reported post-install setup/runtime mismatch remains: model-assisted shell or direct CLI config writes can still persist an env-backed provider SecretRef that resolves in the interactive shell but not in the already-installed daemon environment, and Gateway startup still treats enabled provider API key refs as required startup secrets.

## Best Possible Solution

Keep this issue open for maintainer design on the daemon-aware SecretRef boundary. A good fix should either preflight active provider SecretRefs against the actual gateway service environment before saving or activating post-install config changes, or offer a safe OpenClaw-managed daemon-readable secret storage path. If a secondary provider cannot resolve its secret, the product should block or roll back the change rather than leaving the Gateway in a startup crash loop.
