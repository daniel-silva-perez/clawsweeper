## Summary

Current main still should not close this issue. OpenClaw has partial guardrails: Gateway RPC config writes now preflight active SecretRefs in the running gateway process, and daemon install can copy env-backed config SecretRefs into the generated service environment. But the reported post-install shell/session versus systemd daemon mismatch remains for model-assisted shell or direct CLI config writes: non-dry-run `openclaw config set` can still persist an env-backed provider SecretRef without resolving it against the already-installed daemon environment, and startup still treats enabled provider API-key SecretRefs as required startup secrets.

## Best Possible Solution

Keep this open for maintainer design on the daemon-aware SecretRef boundary. A good fix should block or roll back post-install provider/model config mutations when the active SecretRefs are not resolvable by the actual gateway service environment, or route setup into an OpenClaw-managed daemon-readable secret store such as the durable state `.env` path. Startup should remain fail-closed for truly required secrets, but adding a secondary provider should not leave the whole gateway in a crash loop.
