## Summary

Keep open. Current main has fallback lifecycle tracking, verbose-mode notices, Control UI lifecycle events, and /status visibility, but it still does not implement default-on activation and recovery notifications in the affected chat/session when verbose mode is off or expose a dedicated disable/tuning config.

## What This Changes

The issue requests default-on model fallback activation and recovery notifications delivered to the affected session/topic, with a config switch to disable or tune them.

## Best Possible Solution

Define a product-owned fallback notification policy that sends one-time activation and recovery notices to the affected conversation by default, redacts or normalizes sensitive failure reasons, documents throttling and recovery semantics, and exposes a stable config switch to disable or tune the behavior.

## Reproduction Assessment

Yes. The current e2e fixture provides a high-confidence reproduction path: simulate a fallback transition with verbose mode disabled and observe lifecycle events without activation or recovery notice text in the reply payload; I inspected that test path rather than executing tests because this review is read-only.

## Solution Assessment

Unclear. The requested default-on direction may be the right reliability behavior, but the exact implementation needs maintainer decisions on wording, redaction, throttling, recovery truth, and config shape before an automated narrow patch is safe.
