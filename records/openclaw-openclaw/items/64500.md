## Summary

Keep open. Current main still has the reported pair-aware enforcement gap: the global breaker counts the current tool/signature, ping-pong critical escalation still requires stable per-side result hashes, and the before-tool hook only blocks the current invocation.

## What This Changes

The issue requests pair-aware tool-loop circuit breaking so alternating tool pairs such as memory_search/memory_get or exec/read cannot continue after one side trips the breaker.

## Best Possible Solution

Implement pair-aware loop protection in the agent core: track canonical pair signatures, let sustained alternating pairs escalate under the existing thresholds even when per-side result hashes vary, and persist a scoped block or suppression for both signatures when a pair breaker trips. Preserve run scoping and existing config semantics unless maintainers intentionally add a new threshold, and cover variable-result memory_search/memory_get plus exec/read alternation in regression tests.

## Reproduction Assessment

Yes. A deterministic reproduction can be built by recording alternating A/B tool outcomes with changing result hashes until the critical threshold and then calling detectToolCallLoop; current tests already show that case remains warning-only instead of blocking.

## Solution Assessment

Yes, the requested pair-aware direction is the right fix boundary; the tools.deny workaround removes useful tools globally and the current per-tool breaker does not stop the alternating pair. The safest implementation is a narrowly scoped pair-signature guard with tests rather than changing unrelated defaults or broad tool policy behavior.
