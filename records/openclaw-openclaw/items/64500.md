## Summary

Current main still leaves the reported pair-aware loop-breaker gap open: the global breaker is scoped to the current tool/signature, ping-pong blocking still requires stable per-side result hashes, and the before-tool-call hook blocks only the current invocation.

## Best Possible Solution

Keep this issue open. The best fix belongs in the agent tool-loop guard and before-tool-call enforcement path: make ping-pong protection pair-aware by adding a pair-scoped threshold or allowing sustained A/B alternation to escalate without identical per-side result hashes, then persist short-lived session-level blocks for both tool signatures when a critical pair breaker trips. Add regression coverage for variable-result memory_search <-> memory_get alternation and for blocking both sides of the offending pair; update docs/config only if a new user-facing threshold or behavior is introduced.
