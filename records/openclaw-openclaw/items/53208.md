## Summary

Current main still lacks the reported WhatsApp trusted-contact natural-language relay path. Same-provider cross-context sends are allowed by default, and WhatsApp outbound delivery can send to explicit E.164/JID targets, but WhatsApp still does not expose a contact-name or alias resolver for a prompt target like “Jack”. The prior ClawSweeper comment correctly kept this open; related closed WhatsApp reports cover adjacent setup, E.164 normalization, and outbound send behavior, not name-based trusted-contact relay from another DM.

## Best Possible Solution

Keep this issue open. Add a WhatsApp-owned regression where an authorized direct sender asks to relay to another approved direct contact by name, then implement explicit alias/contact metadata resolution in the WhatsApp plugin by populating directory name/handle fields or adding targetResolver.resolveTarget. Core cross-context policy does not appear to need a change unless that work exposes a missing generic plugin API.
