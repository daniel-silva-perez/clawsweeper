## Summary

Current main still does not implement the reported WhatsApp trusted-contact natural-language relay path. Same-provider cross-context sends are allowed by default, and explicit WhatsApp outbound targets are supported, but a prompt target like “Jack” still has no WhatsApp contact-name/alias resolver or directory metadata path to become Jack’s E.164/JID target. The prior ClawSweeper comment correctly kept this open; the related closed WhatsApp items cover adjacent setup, target normalization, and outbound send behavior rather than trusted-contact relay by name from another DM.

## Best Possible Solution

Keep this issue open. The best path is a WhatsApp-owned regression where an authorized direct sender asks to relay to another approved direct contact by name, followed by a WhatsApp plugin implementation that exposes trustworthy alias/contact metadata in directory entries or implements targetResolver.resolveTarget for configured contact aliases. Core cross-context policy does not appear to need a change unless that work exposes a missing generic plugin API.
