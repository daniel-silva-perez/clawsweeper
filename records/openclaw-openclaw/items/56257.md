## Summary

Keep open. Current main does not implement declarative plugin runtime capabilities, so this PR is not obsolete, but the branch should not merge as-is: the new audit check can miss installed legacy plugins, can warn on disabled plugins, and the public capability contract needs maintainer security/product review before it becomes a manifest surface.

## What This Changes

The PR adds a `capabilities` field and normalizers to plugin manifests, adds a legacy-unrestricted plugin audit warning, and introduces an RFC for phased runtime capability and budget enforcement.

## Best Possible Solution

Keep the PR open for maintainer security/design review, then rework it on current main by adding the manifest capability contract, public docs, and tests while implementing the legacy audit in the current plugin trust audit module using absolute manifest paths and the enabled plugin set.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: install a plugin under `<stateDir>/extensions/some-plugin/openclaw.plugin.json` while running from another cwd and the PR's `loadPluginManifest('some-plugin')` lookup misses it; a config with plugins disabled also shows the raw `pluginDirs` loop would warn on plugins that are not loaded.

## Solution Assessment

No. The declaration-only direction may be reasonable, but this PR is not the best current implementation until it is rebased onto the current audit split, scopes findings to enabled plugins, uses the installed extension paths, and gets explicit review for the new security contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Join plugin dirs before loading manifests:** `src/security/audit-extra.async.ts:833`
  - body: `pluginDirs` are entry names from `listInstalledPluginDirs`, while `loadPluginManifest` resolves `<rootDir>/openclaw.plugin.json`. Passing the bare name makes the audit look under the process cwd instead of `<stateDir>/extensions/<plugin>`, so installed legacy plugins are usually skipped and the new `plugins.capabilities.legacy_unrestricted` finding never fires.
  - confidence: 0.94
- **[P2] Scope legacy capability warnings to enabled plugins:** `src/security/audit-extra.async.ts:831`
  - body: This loop scans every discovered directory, including plugins disabled globally, denied, or marked `enabled: false`. The emitted finding says those plugins "run without declared capabilities", which creates false-positive security warnings for plugins that are not loaded; reuse the enabled plugin set or relabel this as installed inventory.
  - confidence: 0.88
