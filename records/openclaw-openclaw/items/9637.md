## Summary

Current main still does not implement a TUI plain-text/accessibility mode. The proposed `ui.accessibility.plainText` key is absent from config types, strict zod validation, generated help/labels, and public TUI docs, while TUI/tool render paths still emit emoji and Unicode glyphs unconditionally. Related PR #9685 was closed unmerged, so this remains actionable core TUI/config work.

## Best Possible Solution

Keep this issue open and implement it as core TUI/config accessibility work. Add a documented `ui.accessibility.plainText` or equivalent key to `OpenClawConfig`, zod validation, generated labels/help/schema, and public TUI/config docs; load it once at TUI startup; thread it through `ChatLog`, `ToolExecutionComponent`, selector/theme rendering, settings surfaces, waiting/status/tool summary formatting, and any pi-tui wrapper seams; provide ASCII alternatives for emoji and Unicode glyphs; and add focused config validation plus TUI rendering tests. Treat closed PR #9685 as useful prior art, not completed work.
