## Summary

Keep open: current main still applies symmetric jitter after deriving a `retryAfterMs`-based delay and then clamps only to `minDelayMs`, so the reported lower-bound violation remains. The issue is also paired with open closing PR #68543, so the safe maintainer path is to review or replace that PR rather than close the issue early.

## What This Changes

The issue requests that `retryAsync` preserve a server-supplied `retryAfterMs` value as a lower bound when jitter is enabled.

## Best Possible Solution

Land #68543 or a narrow replacement that guarantees `retryAsync` never schedules below a finite `retryAfterMs` after jitter, adds deterministic regression coverage for retry-after plus jitter, and explicitly decides whether the standalone memory-host SDK retry helper should be updated in the same change.

## Reproduction Assessment

Yes. The current source gives a high-confidence static reproduction: `retryAfterMs: 1000`, `jitter: 0.5`, and `generateSecureFraction() === 0` produce 500 ms before the final clamp, which only restores `minDelayMs`.

## Solution Assessment

Unclear for the exact implementation choice. The requested invariant is correct, but maintainers should choose between positive-only jitter and a post-jitter retry-after floor while preserving existing exponential-backoff behavior for callers without `retryAfterMs`.
