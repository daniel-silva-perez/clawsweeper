## Summary

Current main still has the reported Retry-After lower-bound bug, and the issue is already paired with an open focused fix PR, so it should stay open until that PR or an equivalent fix lands.

## What This Changes

Change retryAsync so jitter cannot schedule below a finite server-supplied retryAfterMs value when honoring Retry-After semantics.

## Best Possible Solution

Review and land PR #68543, or an equivalent narrow fix, so core retryAsync preserves Retry-After as a lower bound while keeping symmetric jitter for ordinary exponential backoff and adding a deterministic regression test.

## Reproduction Assessment

Yes. The current-main source provides a deterministic reproduction: mock generateSecureFraction() to 0, call retryAsync with retryAfterMs: () => 1000 and jitter: 0.5, and onRetry will observe a 500ms delay before any fix.

## Solution Assessment

Yes. A Retry-After-aware jitter mode or final Retry-After floor in src/infra/retry.ts is the narrow owner-boundary fix; provider-specific work is not needed unless the mirrored memory-host helper is intentionally shipped and should align.
