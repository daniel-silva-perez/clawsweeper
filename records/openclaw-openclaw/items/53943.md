## Summary

Keep this open. Current main still forwards Slack `read` requests by channel and optional `threadId` only; it does not accept, pass through, or filter by `messageId`, so the specific thread-reply lookup described here remains unresolved.

## What This Changes

Support Slack message `read` requests that fetch or filter a specific `messageId`, including replies addressed by `messageId` plus `threadId`.

## Best Possible Solution

Add exact-message read support in the Slack plugin path: expose/describe `messageId` for read, forward it through the Slack dispatcher and runtime, make `readSlackMessages` return only the requested timestamp for both threaded and channel reads, preserve existing `threadId` behavior, and cover the URL timestamp normalization path if that feeds message parameters.
