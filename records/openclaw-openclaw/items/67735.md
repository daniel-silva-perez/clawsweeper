## Summary

Keep this issue open. Current main still defaults WebChat/Control UI and TUI empty session selection to the selected agent's main session, while identityLinks canonicalization only runs when a concrete direct peer id is present. A focused open PR (#61145) appears to address the same product gap with an opt-in peer default, but it is unmerged and needs normal maintainer review rather than cleanup closure.

## What This Changes

Request WebChat/Control UI support, or clear documentation, for binding a configured WebChat identity to the same canonical direct session as a linked Telegram DM under session.identityLinks.

## Best Possible Solution

Review #61145 as the current focused implementation candidate. The desired end state is either an explicit opt-in WebChat/Control UI session mode or configured peer identity that resolves through the existing dmScope and identityLinks rules, or a deliberate docs update stating that WebChat remains operator/main-session scoped unless a user manually selects another session.
