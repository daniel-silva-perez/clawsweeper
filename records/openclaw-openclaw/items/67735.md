## Summary

Keep this issue open. Current main still treats WebChat/Control UI as a selected-session/internal surface that defaults to the agent main session, while `session.identityLinks` canonicalization only runs when routing has a concrete direct peer id. The open #61145 PR is a focused implementation candidate, but it is unmerged and the latest issue comment adds a named Telegram-account variant that still needs maintainer product/API review.

## What This Changes

The issue asks for WebChat/Control UI to support a canonical direct-human session mode so configured `session.identityLinks` can unify WebChat and Telegram DM continuity instead of defaulting WebChat to `agent:main:main`.

## Best Possible Solution

Review #61145 as the current focused implementation candidate, and decide whether its opt-in WebChat/TUI default peer mode should also cover the named-account Telegram behavior reported in the latest comment. The desired end state is either a documented, explicit WebChat identity/session mode that resolves through the existing `dmScope` and `identityLinks` rules, or a deliberate docs clarification that WebChat remains operator/main-session scoped unless the user manually switches sessions.
