## Summary

Keep open. Current main still has the two-path shape described by the reporter: ACP `mode: "session"` spawns deliver inline to the bound thread, while the ACP `sessions_spawn` wrapper also registers the run with `expectsCompletionMessage` enabled, so the subagent completion/announce lifecycle can still deliver a second message. The related closed duplicate-final-fallback report (#55568) does not cover this ACP inline-plus-announce registry path.

## Best Possible Solution

Keep the issue open and fix ACP `sessions_spawn` tracking so thread-bound `mode: "session"` runs that deliver inline do not also request completion announce delivery. The narrow product/code direction is to mirror the native subagent guard: when ACP session-mode has an inline bound delivery plan, register the tracked run with `expectsCompletionMessage: false` or otherwise suppress the completion announce, while preserving run-mode and `streamTo: "parent"` behavior. Add regression coverage for Discord ACP session-mode showing inline delivery remains enabled but no second announce is dispatched on initial and follow-up turns.
