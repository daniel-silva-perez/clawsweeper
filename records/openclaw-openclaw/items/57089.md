## Summary

Keep open. Current main still has the silent failure behavior, so the PR is addressing a real linked bug, but the branch is stale against the current queued writer implementation and should not merge as-is because it would drop current file-write hardening and the `flush()` API.

## What This Changes

The PR changes `QueuedFileWriter` to count consecutive write failures, emit threshold/periodic warnings, expose a queue-drain helper, and adds queued-writer tests plus diagnostic writer mock updates.

## Best Possible Solution

Port the warning counter onto current main's `safeAppendFile` queue path, keep `flush()` as the completion API, preserve the existing permissions/symlink/hardlink/max-size protections, and add focused tests for threshold, periodic re-warning, reset-after-success, and continued hardening behavior.

## Reproduction Assessment

Yes. Current main can be checked by creating a directory at the intended log file path, calling `write()` three times, awaiting `flush()`, and observing that the current `.catch(() => undefined)` path emits no `console.warn`.

## Solution Assessment

No, not as this PR currently stands. The warning schedule discussed in the PR is reasonable, but the branch should be rebased or replaced so it wraps the current hardened writer instead of restoring the older direct-append/drain implementation.
