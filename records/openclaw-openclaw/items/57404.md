## Summary

Keep this issue open. Current main still has no `agent` WebSocket lifecycle event with `phase: "usage"`; usage is persisted into session state and exposed through session/RPC/status/OpenAI-compatible surfaces, but not as the requested passive per-run lifecycle event. The directly related implementation PR #51317 is closed unmerged, and the production incident comment is relevant supporting context rather than evidence that current main fixed the gap.

## Best Possible Solution

Keep this issue open and either revive/supersede PR #51317 or land a fresh additive protocol change: use a shared helper to emit one normalized post-run `stream: "lifecycle"`, `phase: "usage"` agent event after usage is known, including provider/model, accumulated usage, lastCallUsage, duration, run/session identity, and focused Gateway protocol docs/tests covering WebSocket broadcast behavior plus the independent run paths.

## Risks / Open Questions

- Event shape still needs maintainer protocol design because `AgentEventSchema.data` is generic while lifecycle consumers depend on phase semantics.
- Several run paths emit or persist lifecycle/usage independently, so a fix should avoid drift across embedded, CLI, messaging, follow-up, cron, memory, and ACP paths.
- Current `sessions.changed` and `session.message` usage snapshots may help some observers, but they do not provide the requested per-run lifecycle usage contract, `lastCallUsage`, or duration on the `agent` lifecycle stream.
