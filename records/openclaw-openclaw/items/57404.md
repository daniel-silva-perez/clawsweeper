## Summary

Keep open. The request is specifically for passive WebSocket lifecycle visibility of per-run usage, and current main still does not emit a lifecycle `phase: "usage"` event or any `stream: "usage"` agent event. Usage is persisted to session state, exposed through RPC/CLI/UI/diagnostic surfaces, and useful related context exists in the Bridge/Paperclip discussion, but PR #51317 was closed unmerged and no merged current-main replacement covers the requested WS lifecycle stream behavior.

## Best Possible Solution

Keep this issue open and either revive/supersede PR #51317 or land a fresh additive protocol change: emit one normalized post-run lifecycle usage event after usage is known, include provider/model, accumulated usage, lastCallUsage, duration, and session/run identity, then update Gateway protocol docs and focused tests for the WS broadcast path plus each covered run path.

## Risks / Open Questions

- The event shape still needs maintainer design because current `AgentEventSchema` leaves `data` untyped while lifecycle consumers already rely on specific phase semantics.
- Adding this broadly should avoid five independent ad hoc emitters; a shared helper would reduce drift across primary agent, auto-reply, followup, memory, cron, and ACP paths.
