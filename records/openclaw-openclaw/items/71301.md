## Summary

Current main still only has partial support: docs are packaged and exposed to agents as a local path in prompt guidance, but there is no first-class local-first docs search/get/API retrieval surface. The reporter's follow-up adds useful deployment evidence that a scraped local knowledge layer reduced support load and misconfiguration, so this should remain open for maintainer product/API design rather than be closed as implemented or moved to ClawHub.

## What This Changes

Request to ship a version-matched local docs bundle with every OpenClaw release and expose native agent-facing docs retrieval for onboarding, config, API, and release-specific answers.

## Best Possible Solution

Define a version-coupled docs manifest/search index in release packaging, expose a maintained local-first docs search/get/API lookup surface to agents, keep live Mintlify lookup as an explicit fallback, and add focused tests/docs that distinguish packaged docs discovery from native retrieval.

## Reproduction Assessment

Not applicable: this is a feature/product request rather than a runtime bug. The missing capability is high-confidence from source inspection of packaged docs, docs path discovery, prompt guidance, gateway tool actions, gateway RPC inventory, and CLI docs behavior on current main.

## Solution Assessment

Unclear: the product direction is well supported by the onboarding and docs-drift evidence, but the exact agent-facing API and package/index format still need maintainer choice. A local-first manifest/index plus native retrieval API is safer than relying only on prompt guidance, generic file reads, or the live Mintlify MCP endpoint.
