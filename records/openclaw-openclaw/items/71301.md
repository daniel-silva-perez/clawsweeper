## Summary

Keep open. Current main has partial support through packaged docs, local docs path discovery, prompt guidance, and `config.schema.lookup`, but the central request remains missing: there is no native local-first docs search/get/API retrieval surface for agents, and `openclaw docs` still calls the live Mintlify MCP search endpoint.

## What This Changes

Request to ship version-matched bundled OpenClaw docs with releases and expose native local-first docs retrieval tools for agent-guided onboarding.

## Best Possible Solution

Define a version-coupled docs manifest/search index as part of release packaging, expose local-first docs search/get lookup through a maintained agent-facing runtime surface, keep live Mintlify search as an explicit fallback, and add tests/docs that distinguish packaged docs discovery from native retrieval.

## Reproduction Assessment

Not applicable: this is a feature/product request rather than a runtime bug. The missing capability is high-confidence by source inspection of the docs path resolver, system prompt, gateway tool actions, gateway RPC inventory, CLI docs implementation, and latest release contents.

## Solution Assessment

Unclear: the product direction is plausible and directly addresses onboarding/docs drift, but the best concrete interface still needs maintainer choice. A local-first manifest/index plus native retrieval API is safer than relying only on prompt guidance, generic file reads, or the live docs MCP endpoint.
