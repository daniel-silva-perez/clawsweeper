## Summary

Keep open. Current main still has only partial support: docs are packaged, local docs paths are discovered, the system prompt points agents at local docs, and `config.schema.lookup` covers config fields, but there is still no native, version-matched, local-first docs retrieval layer for agents. The related live-doc drift issue #48920 remains complementary evidence that installed-vs-latest semantics still need a deliberate product/API contract.

## What This Changes

Request to ship a version-matched OpenClaw docs bundle with releases and expose local-first docs search/get/schema/API retrieval to agents for onboarding and configuration guidance.

## Best Possible Solution

Define and ship a release-packaged docs manifest/search index, expose local-first docs search/get/schema/API lookup through a maintained agent-facing surface, keep live Mintlify as an explicit latest-docs fallback, and add docs/tests that distinguish packaged docs discovery from native retrieval.

## Reproduction Assessment

Not applicable as a runtime reproduction because this is a feature request. Source inspection gives a high-confidence way to verify the gap across package allowlisting, docs path resolution, system prompt guidance, Gateway tool actions, Gateway RPC inventory, and live-backed CLI docs search.

## Solution Assessment

Unclear until maintainers choose the package/index contract and agent-facing API. The local-first retrieval direction is the right shape, but it needs a deliberate core contract rather than ad hoc prompt guidance or live-doc search.
