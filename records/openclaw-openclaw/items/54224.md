## Summary

Keep this open. Current main satisfies the history and thread-read parts through Slack's existing `read`/`readMessages` path with user-token-preferred read behavior, but Slack message search is still not implemented or advertised. The related open PR #65691 is the active implementation candidate for the remaining search work, so this should stay open until that PR is reviewed and either landed or closed.

## What This Changes

The issue requests read-only Slack message retrieval actions for history, search, and thread reads that work with user-token read-only workflows without enabling writes.

## Best Possible Solution

End state should be a Slack-owned `search` message action backed by Slack `search.messages`, advertised only for usable user-token read credentials, wired through the existing generic message-action contract and CLI, documented in the Slack channel docs, and covered by focused discovery, dispatch, runtime, and formatting tests. The existing `read` action can remain the channel history and thread history solution.
