## Summary

Keep this PR open. Current main still lacks per-agent dreaming control, and the PR is a relevant implementation candidate, but the latest diff still has a blocking all-disabled fallback bug and is paired with the same-author open feature issue #67413 for maintainer scope review.

## What This Changes

This PR adds `agents.list[].dreaming.enabled` to config types, strict Zod schema, generated base schema, docs, and memory dreaming workspace filtering, with a mixed enabled/disabled unit test.

## Best Possible Solution

Update this PR, or land a successor, so the config contract and docs stay aligned while workspace resolution treats 'no configured agents' differently from 'all configured agents opted out', then cover mixed, all-disabled, and schema-validation cases. Per-agent schedules can be split into a follow-up if maintainers want this PR scoped to enable/disable only.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a config with one valid agent, or all valid agents, setting `dreaming.enabled: false`; the PR guard skips those entries, then the existing empty-list fallback re-adds the default agent unless the implementation tracks that agents were explicitly configured.

## Solution Assessment

No. The direction is maintainable, but the implementation is not yet the best fix because it filters disabled entries before fallback without preserving the explicit opt-out state; the safer approach is to fall back only when there were no configured valid agents, not when every configured agent disabled dreaming.
