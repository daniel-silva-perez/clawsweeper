## Summary

Keep this open. Current main has adjacent model failover, timeout-triggered compaction, memory flush, session-memory, dreaming, and session transcript indexing, but it still does not implement the requested consecutive-timeout pending session capture and later recovery into daily memory. The linked #46930 is a separate gateway log-level issue, not a canonical tracker for this memory recovery work.

## Best Possible Solution

Keep this issue open and implement a generic timeout recovery path in the session/model-fallback lifecycle. When repeated timeout exhaustion leaves user turns unprocessed, persist a redacted pending transcript artifact with stable session/message metadata, then process it after model availability returns through a bounded memory-core or daily-memory maintenance job with dedupe, retention, cleanup, docs, and regression tests.

## Risks / Open Questions

- Persisting failed-session transcripts during provider outages could expose private messages unless artifacts are redacted, access-controlled, and retained for a bounded time.
- Recovery could duplicate memory entries unless it uses stable session/message identifiers and idempotent writes.
- The target store needs product clarity: daily notes, MEMORY.md, dreaming state, and searchable session transcripts have different semantics.
- A recovery worker should avoid depending blindly on the same unavailable model path; it needs bounded backoff and fallback selection.
