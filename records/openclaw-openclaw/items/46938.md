## Summary

Current main has adjacent memory mechanisms, but it still does not implement the requested consecutive-timeout pending session capture and recovery flow. Memory flush is pre-compaction and model-dependent, the session-memory hook only runs on /new or /reset, and dreaming/session transcript ingestion writes different stores with different semantics.

## Best Possible Solution

Keep this issue open and implement a generic timeout recovery path in the session/model-fallback lifecycle. When repeated timeout exhaustion leaves user turns unprocessed, persist a redacted pending transcript artifact with stable session/message metadata, then process it after model availability returns through a bounded memory-core or daily-memory maintenance job with dedupe, retention, cleanup, docs, and regression tests.

## Risks / Open Questions

- Persisting failed-session transcripts during provider outages could expose private messages unless recovery artifacts are redacted, access-controlled, and retained for a bounded time.
- A recovery worker could duplicate memory entries if timeout retries and later recovery do not use stable session/message IDs for idempotency.
- The target memory surface needs product clarity: daily notes, MEMORY.md, dreaming short-term state, and session-memory files each have different semantics.
- Recovery should avoid depending on the same unavailable model path without bounded backoff or fallback selection.
