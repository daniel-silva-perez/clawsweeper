## Summary

Keep open. Current main still lacks the requested target-channel account binding: Feishu tools can use an explicit tool `accountId`, a same-id contextual `agentAccountId`, or `channels.feishu.defaultAccount`, but there is no per-agent cross-channel binding for a DingTalk-originated agent whose DingTalk account id differs from the intended Feishu account id.

## What This Changes

The issue requests a per-agent cross-channel account binding so agents invoked from one channel can select the correct accountId when calling tools for another channel.

## Best Possible Solution

Add a generic, documented target-channel account resolver keyed by active agentId plus target channel. Feishu should preserve explicit tool accountId first, then use the agent's bound Feishu account, then valid same-id contextual routing, then channels.feishu.defaultAccount/runtime fallback; update config types/schema/help/docs and add DingTalk-to-Feishu regression coverage.
