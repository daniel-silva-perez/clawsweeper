## Summary

Keep open. Current main still lacks the requested per-agent target-channel account binding: Feishu tools can use an explicit tool accountId, a same-id contextual accountId from the active channel, or the Feishu default account, but there is no generic agent-to-target-channel account map for the reported DingTalk-to-Feishu isolation case.

## What This Changes

The issue asks for per-agent cross-channel account binding so an agent reached through one channel can select the correct accountId when calling tools for another channel.

## Best Possible Solution

Add a generic, documented target-channel account resolution contract, likely agents.list[].channelBindings or an equivalent resolver keyed by active agentId plus target channel. Feishu tool account resolution should preserve explicit tool accountId first, then use the agent's bound Feishu account, then same-id contextual routing when valid, then channels.feishu.defaultAccount/runtime fallback. Add config schema/types/help/docs and a regression test for a DingTalk-originated agent whose DingTalk account id differs from the intended Feishu account id.
