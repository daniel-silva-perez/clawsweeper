## Summary

Keep this issue open. Current main still does not visibly render read-tool image result blocks in the Control UI tool-output path: the read/image helpers produce top-level image blocks with `data` and `mimeType`, while the chat renderer only extracts URL/source-style images, tool-message images are hidden behind the collapsed disclosure, and live tool-result events reduce structured output to text or sanitized metadata. The open related PR #52025 is the direct implementation candidate, so the right cleanup action is maintainer review or a focused replacement, not closing this report.

## What This Changes

The issue requests that Control UI chat visibly render images returned from tool output, especially read-tool image results, as inline previews or obvious clickable thumbnails.

## Best Possible Solution

Land #52025 or a focused replacement that converts direct read-tool image blocks into safe renderable data-image previews, preserves enough structured image information in live tool streams, surfaces a visible preview or thumbnail for tool results without requiring users to hunt inside collapsed details, and adds targeted Control UI regression tests for read-tool image results.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: render a tool-result message whose content includes `{ type: "image", data: "...", mimeType: "image/png" }`; current extraction does not create a visible image, and the live stream path strips or flattens that structured result before rendering.

## Solution Assessment

Unclear until #52025 is reviewed against current main. The best direction is a narrow Control UI fix plus live-stream handling and tests, not a broader inline-media feature program or a remote-image policy change.
