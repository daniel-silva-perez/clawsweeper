## Summary

Keep this issue open. Current main still does not render the read/tool-result image shape inline in the Control UI: read-tool helpers return top-level `{ type: "image", data, mimeType }` blocks, but the chat extractor only recognizes `source`/`url` image shapes, and the live tool stream still flattens structured tool results to text. The related open PR #52025 remains the direct implementation candidate, so this needs maintainer review rather than cleanup closure.

## What This Changes

The issue reports that Control UI chat should show images returned by tool output inline or through an obvious preview/thumbnail affordance.

## Best Possible Solution

Control UI should safely render direct tool-result image blocks as visible previews, keep enough structured image information through live tool-result updates, preserve existing CSP/local-media protections, and add targeted regression coverage for read-tool image results in both history and live stream paths.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: render a tool-result message whose content includes `{ type: "image", data: "...", mimeType: "image/png" }`; current extraction does not create a visible image, and live tool-result events flatten the structured result to text.

## Solution Assessment

Unclear until #52025 is reviewed against current main. The best direction is a narrow Control UI/tool-stream fix with regression tests, not a broad inline-media feature program or a remote-image policy change.
