## Summary

Keep this issue open. Current main still misses the reported Control UI path: read-tool image results are structured as top-level image blocks with `data` and `mimeType`, while the chat renderer only extracts source/url-style image blocks, tool-message image rendering remains hidden behind the collapsed tool disclosure, and live tool output still flattens structured results to text. The open related PR #52025 is the direct implementation candidate, so the issue should stay open until that PR or a focused replacement lands.

## What This Changes

The issue requests that Control UI chat render images returned from tool output inline, or at least expose an obvious clickable preview or thumbnail.

## Best Possible Solution

Land #52025 or a focused replacement that converts direct read-tool image blocks into safe renderable data images, surfaces a visible preview or thumbnail for tool results without requiring users to hunt inside collapsed details, preserves structured image blocks in live tool streams, and adds focused Control UI regression tests for read-tool image results.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: render a tool-result message whose content includes `{ type: "image", data: "...", mimeType: "image/png" }` and the current extractor will not produce a visible image; the live stream path additionally drops that structured block to text.

## Solution Assessment

Unclear until #52025 is reviewed against current main. The best direction is a narrow Control UI fix plus live-stream handling and tests, not a broader media feature program or remote-image policy change.
