## Summary

Keep open. Current main still parses and honors `[[reply_to_current]]`, still prompts agents to prefer it, and only has global/group visible-reply controls rather than the requested per-agent directive guard plus per-agent automatic-send suppression. The narrower same-author directive issue remains open, while the auto-send half still needs a coherent maintainer-owned config decision.

## What This Changes

This issue requests per-agent config to disable processing of `[[reply_to_current]]` and suppress automatic channel replies so visible output is produced only by explicit message-tool sends.

## Best Possible Solution

Define one coherent agent-scoped outbound policy contract: a directive policy that can reject or rewrite `reply_to_current` while preserving explicit `reply_to:<id>` and message-tool threading, plus clear layering for per-agent visible delivery over the existing global/group visible-reply settings. Then update config types, schema/help/docs, and focused parser, reply plumbing, delivery, and config coverage.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists by parsing assistant text containing `[[reply_to_current]]` with a current message id and then running reply threading; the current code maps it to that message id. Direct chats also remain automatic unless the global `messages.visibleReplies` workaround is configured.

## Solution Assessment

Unclear. The requested runtime guard is a valid direction, but the exact public config shape needs maintainer design because `agents.list[].autoReply` or a directive policy could conflict with the existing global/group visible-reply controls.
