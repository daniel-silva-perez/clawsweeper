## Summary

Keep open. Current main partially addresses auto-send with global visible-reply controls, but it still lacks the requested per-agent runtime guard: `[[reply_to_current]]` is actively parsed and encouraged, and there is no `agents.list[]` directive or auto-reply setting.

## What This Changes

Add optional per-agent runtime configuration to disable `[[reply_to_current]]` and suppress automatic assistant text delivery so visible output must use explicit message-tool sends.

## Best Possible Solution

Define one coherent config contract: extend the existing visible-reply/source-delivery policy to support per-agent scope, and add a separate per-agent directive policy that blocks or rewrites `reply_to_current` while leaving explicit `reply_to:<id>` and message-tool threading intact. Update schema/help/docs and tests around directive parsing, prompt generation, streaming/block replies, final delivery, and Feishu-style explicit message-tool threading.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: `[[reply_to_current]]` currently resolves to `currentMessageId` through directive parsing and reply threading tests, and direct chats remain automatic unless the global `messages.visibleReplies` workaround is set.

## Solution Assessment

Unclear. The requested direction is valid, but the best implementation should be chosen by maintainers because adding `agents.list[].autoReply` beside `messages.visibleReplies` could create overlapping public config semantics.
