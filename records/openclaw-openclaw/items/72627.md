## Summary

Keep open. Current main still accepts and honors `[[reply_to_current]]`, still prompts agents to prefer it, and has only global/group visible-reply controls rather than the requested per-agent directive guard plus per-agent automatic-send suppression. The same-author narrower directive-only issue is also still open, so this needs maintainer product/API follow-up rather than cleanup closure.

## What This Changes

Request per-agent configuration to disable `[[reply_to_current]]` runtime directive handling and suppress automatic assistant-text delivery so only explicit message tool sends produce visible replies.

## Best Possible Solution

Define one coherent agent-scoped outbound policy contract: a directive policy that can reject or rewrite `reply_to_current` while preserving explicit `reply_to:<id>` and message-tool threading, plus clear layering for per-agent visible delivery over the existing global/group visible-reply settings. Then update config types, strict schema/help/docs, prompt guidance, parser/reply plumbing, delivery tests, and config docs together.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists by parsing assistant text containing `[[reply_to_current]]` with a current message id and then applying reply threading; current main maps it to that message id. Direct chats also remain automatic unless `messages.visibleReplies: "message_tool"` is configured globally.

## Solution Assessment

Unclear. The requested runtime guard is a valid direction, but the exact public config shape needs maintainer design because per-agent `autoReply` or directive policy can conflict with existing global/group visible-reply controls and native-command delivery exceptions.
