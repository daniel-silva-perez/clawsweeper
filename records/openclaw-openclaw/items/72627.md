## Summary

Keep open. Current main still accepts and honors `[[reply_to_current]]`, still teaches agents to prefer it, and only offers global/group visible-reply controls rather than the requested per-agent directive guard plus per-agent automatic-send suppression.

## What This Changes

Add per-agent configuration to disable the `[[reply_to_current]]` reply directive and optionally suppress automatic assistant text delivery so only explicit `message` tool sends are visible.

## Best Possible Solution

Define one coherent agent-scoped outbound policy contract that can reject or rewrite `reply_to_current`, preserve explicit `reply_to:<id>` and message-tool sends, and clearly layer over existing visible-reply controls; then update config types, strict schema/help/docs, prompt guidance, parser/reply plumbing, and regression tests together.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: assistant text containing `[[reply_to_current]]` is parsed as `replyToCurrent` and mapped to the current message id, and direct chats still deliver automatic final text unless the global `messages.visibleReplies: "message_tool"` mode is configured.

## Solution Assessment

Unclear. The runtime guard is a plausible direction, but the exact public API should be designed by maintainers because per-agent directive policy and per-agent auto-send suppression overlap existing global/group visible-reply controls and native-command exceptions.
