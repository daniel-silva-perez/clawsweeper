## Summary

Keep open. Current main still parses and honors `[[reply_to_current]]`, still prompts agents to prefer it, and only offers global/group visible-reply controls rather than the requested per-agent directive guard plus per-agent automatic-send suppression. The linked narrower directive issue is still open, so this broader report remains useful for maintainer coordination.

## What This Changes

The issue requests per-agent runtime controls to disable the `[[reply_to_current]]` directive and suppress automatic visible replies so agents can reply only through explicit message tool calls.

## Best Possible Solution

Define one coherent public contract for agent-scoped outbound policy: a directive policy that can reject or rewrite `reply_to_current` while preserving explicit `reply_to:<id>` and message-tool threading, plus a clear layering rule for per-agent visible delivery over the existing global/group `messages.visibleReplies` model. Then update config types/schema/help/docs and add focused parser, reply plumbing, delivery, and config tests.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction path exists: parse assistant text containing `[[reply_to_current]]` with a current message id, then run reply threading and observe that it becomes a reply to the current message; direct chats also remain automatic unless the global `messages.visibleReplies` setting is changed.

## Solution Assessment

Unclear. The requested runtime guard is a valid direction, but the exact public config shape needs maintainer design because `agents.list[].autoReply` or directive policy could conflict with the existing global/group visible-reply controls.
