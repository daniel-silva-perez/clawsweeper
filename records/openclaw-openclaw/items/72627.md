## Summary

Keep open. Current main still accepts and honors `[[reply_to_current]]`, still tells agents to prefer it, and only has global/group visible-reply controls rather than the requested per-agent directive guard plus per-agent automatic-send suppression. The prior ClawSweeper review and the same-author narrower #73272 both point to unresolved product/API work, not cleanup closure.

## What This Changes

The issue requests per-agent runtime configuration to disable `[[reply_to_current]]` processing and suppress automatic visible assistant replies so agents must use explicit message-tool sends.

## Best Possible Solution

Define one coherent agent-scoped outbound policy contract that can reject or rewrite `reply_to_current` while preserving explicit `reply_to:<id>` and message-tool threading, and specify how per-agent visible delivery layers over the existing global/group `messages.visibleReplies` settings. Then update config types, strict schema/help/docs, prompt guidance, parser/reply plumbing, delivery tests, and operator warning feedback together.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists by parsing assistant text containing `[[reply_to_current]]` with a current message id, then applying reply threading; current main maps it to that message id. Direct chats also remain automatic unless `messages.visibleReplies: "message_tool"` is set globally.

## Solution Assessment

Unclear. A runtime-level guard is a plausible direction, but the requested config shape needs maintainer design because per-agent directive policy and per-agent automatic-send suppression must layer cleanly with existing visible-reply settings, native command behavior, and message-tool availability fallback.
