## Summary

This should stay open. Current main has streaming chunking, exact duplicate suppression, tool-call loop detection, and abort plumbing, but I did not find a core assistant-output repetition circuit breaker that halts a repeating model stream, alerts the user, and waits for confirmation.

## What This Changes

Add a core daemon stream-level circuit breaker that detects repeated assistant output, halts the LLM stream, alerts the user, and requires confirmation before continuing or aborting.

## Best Possible Solution

Keep this open for maintainer product/design review. The best end state is an opt-in core assistant-output loop guard near the embedded stream subscription or stream wrapper layer, with conservative phrase detection, provider-safe abort/cancel behavior, one clear user-facing alert, config/docs/tests, and explicit semantics for whether confirmation retries, cancels, or starts a follow-up turn.

## Reproduction Assessment

Not applicable as a normal bug reproduction; this is a feature request. Source inspection gives a high-confidence absence check: repeated assistant deltas would flow through the existing streaming/chunking path without any repetition threshold or halt-and-confirm flow.

## Solution Assessment

Unclear until maintainers choose the product semantics. The requested core stream guard is the right general area, but the pause/resume confirmation design should likely be specified before an implementation PR because provider streams are usually abortable rather than resumable.
