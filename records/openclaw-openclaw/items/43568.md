## Summary

Keep open. Current main still supports process-level proxy routing and model-provider request proxy overrides, but it does not expose or propagate an auth-profile-scoped proxy setting. The auth profile credential shape, config schema, and resolved auth result have no transport field, and runtime auth plugin request overrides explicitly reject proxy/TLS transport settings, so this is not already implemented or plugin-only today.

## What This Changes

The issue requests an optional `proxyUrl` on auth profiles so different provider profiles can route API requests through distinct HTTP proxies.

## Best Possible Solution

Design a first-class profile-scoped transport setting or equivalent routing seam, add schema/types/docs with redaction and SecretRef policy, preserve existing provider-level request proxy behavior, and route the selected profile's transport metadata into model request construction with focused tests for multiple Anthropic/OpenAI profiles using distinct proxies.
