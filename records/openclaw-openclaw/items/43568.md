## Summary

Keep this issue open. Current main has process-wide proxy routing and provider-config-scoped `models.providers.*.request.proxy`, but it still does not implement the requested auth-profile-scoped `proxyUrl` or an equivalent profile transport seam. The prior ClawSweeper comment correctly identified this gap, and the current checkout still matches that conclusion.

## Best Possible Solution

Keep the issue open. The best maintainer path is to design a documented profile-scoped transport setting or equivalent core/plugin seam, flow the selected profile's transport through auth resolution into model request construction, preserve existing provider-level `models.providers.*.request.proxy`, update schema/help/docs/redaction, and add tests for multiple Anthropic/OpenAI profiles under one provider using distinct proxies.

## Risks / Open Questions

- A profile-scoped proxy URL can contain credentials and changes outbound provider routing, so any implementation needs validation, redaction, SecretRef/security design, and regression coverage.
- Using separate custom provider IDs with `models.providers.*.request.proxy` may be a workaround, but it is not equivalent to multiple auth profiles under one provider using distinct proxies.
- Closing this as ClawHub/plugin work would be premature because current runtime auth hooks intentionally reject proxy/TLS transport overrides.
