## Summary

Keep open. Current main has process-level proxy routing and model-provider request proxy overrides, but it still has no auth-profile-scoped proxy metadata or resolved-auth transport path. This is core auth/provider plumbing rather than a ClawHub-only plugin because profile selection and provider request construction are owned by the core auth/model path, and the remaining design is security-sensitive.

## What This Changes

Add optional per-auth-profile proxy routing so individual provider credentials can send model requests through different proxy URLs.

## Best Possible Solution

Define a first-class auth-profile transport policy with secret-aware proxy credential handling, validation, redaction, copy/inheritance semantics, and explicit precedence, then thread the selected profile's resolved transport policy into provider request construction while preserving existing process-level and request-scoped proxy behavior.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Current-main verification is high confidence: static inspection shows no auth-profile proxy fields, no resolved-auth transport field, and existing proxy handling only on provider request config.

## Solution Assessment

No, adding a raw proxyUrl directly to credential records is not yet the best complete solution. The safer path is a designed profile transport policy that composes with SecretRef/redaction and existing egress precedence before implementation.
