## Summary

Keep open. Current main still has process-level proxy routing and model-provider request proxy overrides, but it does not expose auth-profile-scoped proxy metadata or carry selected-profile transport policy through resolved auth. The April 27 ClawSweeper review remains accurate against the current main SHA, and this is security-sensitive enough to need maintainer design review before implementation.

## What This Changes

The issue requests optional per-auth-profile proxy routing, such as a `proxyUrl` field, so different provider credentials can use different outbound proxies from one OpenClaw instance.

## Best Possible Solution

Define a first-class auth-profile transport policy that is secret-aware, validates proxy URLs, documents precedence, preserves redaction and copy semantics, and threads the selected profile's resolved policy into provider request construction without weakening existing process-level and request-scoped proxy behavior.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Static verification is high confidence because the credential types, strict config schema, resolved-auth type, runtime-auth sanitization, docs, and request-proxy tests show no profile-scoped proxy channel on current main.

## Solution Assessment

No, adding a raw `proxyUrl` directly to credential records is not yet the best complete solution. The safer path is a designed profile transport policy that composes with SecretRef/redaction and existing egress precedence before implementation.
