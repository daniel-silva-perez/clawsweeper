## Summary

Keep open. Current main still has process-level proxy routing and model-provider request proxy overrides, but auth profiles do not carry profile-scoped transport metadata and resolved model auth has no way to pass a selected profile proxy into provider request construction. The remaining work is security-sensitive because proxy URLs may include credentials and change provider egress routing.

## What This Changes

The issue requests an optional auth-profile `proxyUrl` field so each model-provider credential profile can route API requests through its own outbound proxy.

## Best Possible Solution

Define a first-class profile transport policy with secret-aware proxy credential handling, validation, redaction, and explicit precedence, then thread the selected auth profile's resolved transport policy into model-provider request construction while preserving existing process-level and request-scoped proxy behavior.
