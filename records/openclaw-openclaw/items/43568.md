## Summary

Keep open. Current main has process-level proxy routing and model-provider request proxy overrides, but it still has no auth-profile-scoped transport field or resolved-auth path that can carry selected-profile proxy policy into provider requests. Because proxy URLs can contain credentials and change egress routing, this needs maintainer design review before implementation.

## What This Changes

Add optional per-auth-profile proxy routing so different provider credentials can send model-provider API requests through different proxy URLs.

## Best Possible Solution

Define a first-class auth-profile transport policy that is secret-aware, validates proxy URLs, documents egress precedence, preserves redaction and agent-copy semantics, and then threads the selected profile's resolved policy into provider request construction without weakening existing process-level and provider request proxy behavior.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Static verification is high confidence because current credential types, strict config schema, resolved-auth type, runtime-auth sanitization, docs, and request-proxy tests show no profile-scoped proxy channel on current main.

## Solution Assessment

No, adding a raw proxyUrl directly to credential records is not yet the best complete solution. The safer path is a designed profile transport policy that composes with secrets/redaction and existing egress precedence before implementation.
