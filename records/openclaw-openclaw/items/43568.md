## Summary

Keep open. Current main still has process-wide proxy support and provider-config-scoped request proxy support, but it does not expose an auth-profile-scoped proxy setting or flow profile transport metadata through auth resolution into model request construction. The prior ClawSweeper review remains accurate against the current checkout.

## What This Changes

The issue requests an optional per-auth-profile `proxyUrl` field so different provider credentials can route model requests through different proxies within one OpenClaw instance.

## Best Possible Solution

Design a first-class profile-scoped transport setting or equivalent core seam, document it, preserve existing provider-level request proxy behavior, route the selected profile’s transport into model request construction, and cover multiple Anthropic/OpenAI profiles using distinct proxies with schema, redaction, and runtime tests.

## Risks / Open Questions

- A profile-scoped proxy URL can contain credentials and affects outbound provider routing, so any implementation needs explicit validation, SecretRef/redaction behavior, and log/runtime-state hygiene.
- Provider-level `models.providers.*.request.proxy` can be a workaround for separate provider IDs, but it is not equivalent to multiple auth profiles under the same provider each using distinct proxies.
