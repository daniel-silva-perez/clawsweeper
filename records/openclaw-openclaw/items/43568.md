## Summary

Keep this issue open. Current main has process-wide proxy routing and model-provider-scoped `models.providers.*.request.proxy`, but it still does not implement the requested auth-profile-scoped `proxyUrl` or an equivalent profile transport seam. The prior ClawSweeper comment identified the same gap, and the current checkout still matches that conclusion.

## Best Possible Solution

Keep the issue open. The best maintainer path is to design a documented profile-scoped transport setting or equivalent core/plugin seam, flow the selected profile's transport through auth resolution into model request construction, preserve existing `models.providers.*.request.proxy`, update schema/help/docs/redaction/SecretRef handling, and add regression tests for multiple Anthropic/OpenAI profiles under one provider using distinct proxies.

## Risks / Open Questions

- A future implementation must treat profile proxy URLs as sensitive because they may embed proxy credentials; schema help, redaction, SecretRef policy, and logs need explicit coverage.
- Per-profile egress routing interacts with auth-profile rotation, cooldown, fallback, runtime auth refresh, and provider plugins; tests should cover multiple profiles under one provider using distinct proxies.
- Using separate custom provider IDs with `models.providers.*.request.proxy` is a possible workaround, but it is not equivalent to the requested multiple auth profiles under one provider.
