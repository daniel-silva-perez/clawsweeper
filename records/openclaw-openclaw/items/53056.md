## Summary

Keep open: current `main` still sends Slack-bound text through formatters that encode raw ampersands to `&amp;`, and the latest release snapshot contains the same behavior. The linked fix PRs were closed unmerged, and the discussion still needs a live Slack rendering decision that preserves exactly-one entity escaping and does not accidentally activate literal entity text as Slack mentions or tokens.

## What This Changes

Slack outbound messages containing literal ampersands should render as `&` instead of visible `&#38;` without weakening Slack mrkdwn escaping.

## Best Possible Solution

Keep the report open until Slack rendering is verified directly, then update the Slack-owned formatter, send, preview, slash/interactions, and tests so outbound text gets one safe entity-escaping layer while preserving intentional Slack angle tokens and literal-entity safety.
