## Summary

Keep open: current `main` and the latest release still encode Slack outbound ampersands before sending, while the related fix PRs are closed unmerged and the discussion identifies a real exactly-one-escaping and mention-safety question that needs live Slack verification.

## What This Changes

The issue reports that Slack outbound messages containing ampersands render as `&#38;` instead of `&`, likely due to double entity encoding in the Slack send and formatting pipeline.

## Best Possible Solution

Keep the report open until Slack rendering is verified directly, then update the Slack-owned formatter, send, preview, and slash-response paths to produce exactly one safe entity-escaping layer while preserving intentional Slack angle tokens and literal entity safety.
