## Summary

This should stay open because current main and the latest release still route Slack outbound text through mrkdwn escaping that converts `&` to `&amp;`, while the linked fix attempts were closed unmerged and the discussion explicitly calls for direct Slack API verification before changing the escaping contract.

## What This Changes

Slack outbound messages containing ampersands should display `&` in Slack instead of a double-encoded entity such as `&#38;`.

## Best Possible Solution

Keep this open until Slack behavior is verified directly, then update the Slack-owned formatter/send paths to produce exactly one safe escaping pass for user-visible text while preserving Slack angle-bracket tokens, literal mention safety, and regression coverage for raw and already-escaped ampersands.
