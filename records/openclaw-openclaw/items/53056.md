## Summary

Keep this open. Current main still routes normal Slack outbound text through Slack mrkdwn helpers that escape `&` to `&amp;`, the tests explicitly assert that behavior, and the two linked fix PRs were closed unmerged. The later discussion also makes the fix direction non-trivial because removing ampersand escaping outright may conflict with Slack mrkdwn control-character handling.

## What This Changes

The issue reports that Slack outbound messages can display literal `&#38;` instead of `&`, apparently from double encoding ampersands in Slack mrkdwn send paths.

## Best Possible Solution

Preserve Slack's required mrkdwn control-character safety while making outbound Slack text idempotent across normal sends, slash responses, preview final edits, and tool-progress labels, with regression tests for raw `&`, existing entities such as `&amp;`, and mention-like text.
