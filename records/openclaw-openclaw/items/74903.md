## Summary

Keep open. Current main still has the reported mismatch: the generic tool-only group prompt steers visible replies to `message(action=send)`, while Feishu only applies `reply_in_thread` semantics for explicit `thread-reply` or the automatic reply dispatcher path. The issue is fresh, unpaired with an open fix PR in the provided context, and narrow enough for a focused Feishu/plugin-threading repair.

## What This Changes

The issue requests Feishu `group_topic` sessions to send visible assistant replies back into the originating topic/thread instead of posting them as normal top-level group messages.

## Best Possible Solution

Add a Feishu-owned topic-aware tool-delivery path so same-conversation visible replies from `group_topic` and `group_topic_sender` sessions carry the topic root into Feishu `reply_in_thread` semantics, with targeted tests proving normal groups still use the existing top-level behavior.

## Reproduction Assessment

Yes. A high-confidence source reproduction is available: use a Feishu group topic session with default tool-only group delivery, observe the injected prompt tells the model to call `message(action=send)`, and verify Feishu `send` does not set `replyInThread`.

## Solution Assessment

Yes, the maintainable fix should stay at the Feishu/plugin threading boundary rather than changing the generic group prompt for every channel. A prompt hint can help, but the robust behavior is for Feishu same-conversation topic sends to resolve to the topic root/thread transport.
