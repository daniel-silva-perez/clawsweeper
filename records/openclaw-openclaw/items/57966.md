## Summary

Keep this PR open, but do not merge it as-is. Current main still has lowercase-only alias lookup and exact configured-model inference, so the intended alias-normalization and alias-hint behavior is not implemented; however, the latest PR patch introduces concrete model-routing and error-output regressions and is stale against the current split resolver.

## What This Changes

The PR changes model alias resolution to fold spaces and underscores to hyphens, infer providers from configured model allowlists, return configured model IDs from inference, append available aliases to disallowed-model errors, update session identity resolution, and add model-selection tests.

## Best Possible Solution

Keep this PR open as a candidate, but require a rebase onto `src/agents/model-selection-shared.ts` and a narrower implementation that preserves exact model-id precedence, treats normalized collisions as ambiguous, returns matched configured model IDs only when unambiguous, sanitizes alias hints/log text, and documents/tests the chosen separator policy.

## Reproduction Assessment

Yes. The PR patch itself includes enough code and tests to reproduce the main regression: with `anthropic/my_model` allowlisted and alias `my-model -> openai/gpt-4o`, folding `_` to `-` makes `/model my_model` route through the alias instead of the exact model ID.

## Solution Assessment

No. The requested direction is plausible, but this PR is not the best implementation because it applies fuzzy alias matching before preserving exact model IDs, exposes raw aliases in errors, and no longer targets the current split resolver.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve exact model IDs before fuzzy aliases:** `src/agents/model-selection.ts:72`
  - body: Folding `_` and spaces to `-` in `normalizeAliasKey` lets a bare exact model ID such as `my_model` match an alias like `my-model` first. With `anthropic/my_model` allowlisted and `my-model` pointing at another provider, `/model my_model` silently switches providers; the PR even adds a test expecting that behavior. Check exact configured/default-provider model IDs before accepting fuzzy alias matches.
  - confidence: 0.92
- **[P2] Escape aliases before adding error hints:** `src/agents/model-selection.ts:768`
  - body: The new error hint joins configured alias strings directly into `model not allowed` errors. Aliases come from config, so control characters, newlines, or ANSI escapes can be reflected into terminal or UI output; sanitize or escape each alias before joining.
  - confidence: 0.88
- **[P2] Reject same-provider normalized model collisions:** `src/agents/model-selection.ts:218`
  - body: `inferUniqueProviderFromConfiguredModels` records only providers and overwrites `matchedModelId` for each match. If one provider configures both `gpt_5.4` and `gpt-5.4`, input like `gpt 5.4` resolves to whichever entry is visited last instead of reporting an ambiguous match.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The PR introduces a concrete output-sanitization issue by appending raw configured alias strings to user-visible disallowed-model errors.

Concerns:

- **[medium] Raw aliases are reflected in error hints:** `src/agents/model-selection.ts:768`
  - body: Configured aliases can contain newlines, control characters, or ANSI escapes, and the PR returns them via `aliases.join(", ")` in `model not allowed` errors. That can forge terminal/chat output unless each alias is sanitized or escaped first.
  - confidence: 0.88
