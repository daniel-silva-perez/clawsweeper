## Summary

Keep this open. Current main still has a closed cron payload contract for `systemEvent` and `agentTurn`; there is no first-class `exec` or `lobster` cron kind, no `condition`/`onCondition` handoff contract, and the CLI/UI/docs still describe agent-mediated isolated jobs. #18160 is related but narrower direct-exec work, not a full duplicate of the Lobster/conditional-agent handoff request.

## Best Possible Solution

Keep this open for maintainer design. The best path is either an additive cron executor contract for `exec` and Lobster payloads with bounded condition evaluation and conditional `agentTurn` handoff, or an explicit plugin/ClawHub plan that first adds any missing cron executor API seam. Coordinate with #18160 as the narrower direct-exec subset, but keep this issue tracking the broader deterministic pipeline plus conditional-agent workflow.

## Risks / Open Questions

- A native scheduled `exec`/Lobster path would broaden command-execution behavior in a background scheduler, so it needs explicit maintainer decisions on approvals, sandboxing, timeouts, output caps, env/secrets exposure, logging, and failure delivery.
- Closing as a duplicate of #18160 would lose the Lobster pipeline and conditional `onCondition` handoff requirements; #18160 is only the direct-exec subset.
- A ClawHub/plugin implementation may be possible, but current exposed cron plugin types are observational/control-plane oriented and still model cron payloads around text/system jobs, so maintainers need to decide whether a plugin API seam is sufficient before implementation.
