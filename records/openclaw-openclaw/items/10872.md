## Summary

Current main still only forwards iMessage reply context when `imsg` supplies already-resolved `reply_to_*` fields; it does not resolve GUID-only inline reply metadata from `chat.db`. The issue has recent maintainer/user context and open PR #42683 remains the active implementation path, so this should stay open for maintainer review rather than be closed or sent to a duplicate repair lane.

## What This Changes

Request that inbound iMessage inline replies include the replied-to message id, body, and sender in the context passed to the agent.

## Best Possible Solution

Ship an iMessage-owned monitor change that resolves reliable inline-reply origin metadata into `ReplyToId`, `ReplyToBody`, and `ReplyToSender` before dispatch, using the existing `dbPath` surface where needed and covering GUID-only replies plus lookup failure modes in focused tests.
