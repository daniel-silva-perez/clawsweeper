## Summary

Keep this open. Current main still only forwards iMessage reply context when the incoming RPC payload already contains resolved `reply_to_id`, `reply_to_text`, and `reply_to_sender`; it does not model `thread_originator_guid`/`reply_to_guid` or enrich GUID-only inline replies from `chat.db`. Open PR #42683 remains the closest implementation candidate, but it is open and unmerged.

## What This Changes

Request iMessage inbound messages include native inline reply/thread context by resolving reliable reply-origin metadata into `ReplyToId`, `ReplyToBody`, and `ReplyToSender` when `imsg` does not emit resolved `reply_to_*` fields.

## Best Possible Solution

Keep this issue open until the bundled legacy iMessage plugin resolves reliable reply-origin metadata into the existing inbound `ReplyToId`, `ReplyToBody`, and `ReplyToSender` fields. The concrete path is to review, port, or supersede PR #42683 in `extensions/imessage/src/monitor/*`, prefer reliable origin metadata such as `thread_originator_guid`, use a narrow read-only `chat.db` lookup only when appropriate, fail closed on permission/schema/locking errors, and add regression tests for GUID-only inline replies.
