## Summary

Keep this open. Current main still forwards iMessage reply context only when the incoming payload already has resolved `reply_to_*` fields; it does not accept `thread_originator_guid`/`reply_to_guid` or enrich GUID-only inline replies from `chat.db`. Recent maintainer and user comments identify the same remaining gap, and open PR #42683 is the active implementation path rather than a reason to close this issue.

## What This Changes

Request iMessage inbound handling to include inline reply/thread context by resolving iMessage reply metadata into `ReplyToId`, `ReplyToBody`, and `ReplyToSender` before agent dispatch.

## Best Possible Solution

Land or replace the existing iMessage-focused fix so the bundled iMessage plugin resolves reliable inline-reply origin metadata into `ReplyToId`, `ReplyToBody`, and `ReplyToSender` before dispatch, using the existing `dbPath` surface for any read-only fallback and covering GUID-only replies plus lookup failure modes in focused tests.
