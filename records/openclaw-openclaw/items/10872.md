## Summary

Keep this issue open. Current main and the latest release still only forward iMessage reply context when the incoming imsg RPC payload already includes resolved `reply_to_id`, `reply_to_text`, and `reply_to_sender`; there is still no current-main handling for GUID-only inline reply metadata such as `thread_originator_guid` or a read-only `chat.db` enrichment path. Maintainer discussion already points to open PR #42683 as the closest implementation candidate, so this should not be closed or sent to a new repair lane before that PR is reviewed, ported, or superseded.

## What This Changes

The issue requests inbound iMessage inline replies to include the original replied-to message context so the agent can understand which prior message the user is answering.

## Best Possible Solution

Land or replace the existing iMessage-focused fix so the bundled legacy iMessage plugin resolves reliable inline-reply origin metadata into the existing `ReplyToId`, `ReplyToBody`, and `ReplyToSender` fields before dispatch. Keep the change inside `extensions/imessage/src/monitor/*`, prefer `thread_originator_guid` when available, use a narrow read-only `chat.db` lookup through the existing `dbPath` surface only when needed, and add regression tests for GUID-only inline replies plus failure modes.
