## Summary

Current main still does not resolve GUID-only iMessage inline replies into inbound reply context. The iMessage monitor handles already-resolved `reply_to_*` fields, but the parser/model still lack `thread_originator_guid`/`reply_to_guid` handling and the monitor does not enrich from `chat.db`; the issue also has a maintainer keep-open signal and open PR #42683 remains the active implementation path.

## What This Changes

Request inbound iMessage messages to include native inline reply/thread context so the agent can see the message being replied to.

## Best Possible Solution

Land or replace the existing iMessage-focused fix so the bundled iMessage plugin resolves GUID-only inline replies into `ReplyToId`, `ReplyToBody`, and `ReplyToSender` before dispatch, using the existing `dbPath` surface for any read-only fallback and covering lookup failure modes in focused tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an imsg RPC notification with only GUID reply metadata and no resolved `reply_to_text`; current main parses only resolved `reply_to_*` fields, then `describeReplyContext` returns null because no reply body exists.

## Solution Assessment

Yes for triage. Keeping this open for the existing PR path is the best current handling; the narrow maintainable fix belongs in `extensions/imessage/src/monitor/*`, while a duplicate automated repair PR would add churn while PR #42683 is active.
