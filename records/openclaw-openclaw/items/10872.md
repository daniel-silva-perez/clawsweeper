## Summary

Current main still only forwards iMessage reply context when the incoming payload already contains resolved `reply_to_*` fields; it does not handle GUID-only inline reply metadata or enrich from `chat.db`. The issue remains valid, and open PR #42683 is the active implementation path, so this should stay open for maintainer review rather than closure or a separate repair lane.

## What This Changes

The issue requests that inbound iMessage inline replies include the replied-to message id, sender, and body in the context passed to the agent.

## Best Possible Solution

Land or replace the existing iMessage-focused fix so the bundled iMessage plugin enriches GUID-only inline replies into `ReplyToId`, `ReplyToBody`, and `ReplyToSender` before dispatch, using the existing `dbPath` surface for any read-only fallback and covering lookup failure modes in focused tests.
