## Summary

Current main still does not resolve iMessage inline reply/thread context when imsg omits resolved reply_to_* fields. The active iMessage plugin accepts and forwards already-resolved reply_to_id/reply_to_text/reply_to_sender, but it does not model thread_originator_guid/reply_to_guid or enrich from chat.db before inbound dispatch. Open PR #42683 remains the closest implementation candidate, but it is open/unmerged and appears to need porting to the active extensions/imessage path.

## Best Possible Solution

Keep this issue open until the bundled legacy iMessage plugin resolves imsg/chat.db reply-origin metadata into the existing inbound ReplyToId, ReplyToBody, and ReplyToSender fields. The likely implementation belongs in extensions/imessage/src/monitor/*: enrich missing reply context before inboundDebouncer.enqueue, prefer reliable origin GUID metadata, use read-only local chat.db lookup only when appropriate, fail closed on permission/schema/locking errors, avoid unsafe remote assumptions, and add regression tests for GUID-only inline replies. Review, port, or supersede PR #42683 rather than queueing a parallel fix.
