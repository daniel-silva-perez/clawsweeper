## Summary

Keep this issue open. Current main still only forwards iMessage reply context when resolved `reply_to_*` fields are already present; it does not model or resolve GUID-only inline reply metadata before dispatch, and the discussion already points to open PR #42683 as the active implementation path.

## What This Changes

Request that inbound iMessage messages include inline reply/thread context by resolving the replied-to message into `ReplyToId`, `ReplyToBody`, and `ReplyToSender` before agent dispatch.

## Best Possible Solution

Land or replace the existing iMessage-focused fix so the bundled iMessage plugin resolves GUID-only inline replies into `ReplyToId`, `ReplyToBody`, and `ReplyToSender` before dispatch, using the existing `dbPath` surface for any read-only fallback and adding focused regression coverage for lookup success and failure modes.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an imsg RPC notification with reply GUID metadata but no resolved `reply_to_text`; current main parses only resolved `reply_to_*` fields and then `describeReplyContext` returns null because no reply body exists.

## Solution Assessment

Yes for triage. Keeping this open for the existing PR path is the narrowest maintainable handling right now; a separate automated repair PR would duplicate #42683 while the lookup-key and chat.db assumptions still need review.
