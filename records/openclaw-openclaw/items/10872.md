## Summary

Keep this open. Current main still only forwards iMessage reply context when the RPC payload already contains resolved `reply_to_*` fields; it does not model GUID-only inline reply metadata or enrich from `chat.db`. The issue also has a maintainer keep-open signal and open PR #42683 already targets the same gap, so this needs maintainer review rather than closure or a duplicate repair lane.

## What This Changes

Incoming iMessage inline replies should include the quoted/thread-origin message context so the agent can see what prior message the user replied to.

## Best Possible Solution

Land or replace the existing iMessage-focused fix so the bundled iMessage plugin resolves GUID-only inline replies into `ReplyToId`, `ReplyToBody`, and `ReplyToSender` before dispatch, using the existing `dbPath` surface for any read-only fallback and covering lookup failure modes in focused tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an imsg RPC notification with only GUID reply metadata and no resolved `reply_to_text`: current main parses only resolved `reply_to_*` fields, then `describeReplyContext` returns null because no reply body exists.

## Solution Assessment

Yes for triage: keeping this open for the existing PR path is the best current handling. The narrow maintainable fix belongs in `extensions/imessage/src/monitor/*`; a core hook or duplicate automated repair PR would add churn while PR #42683 is active.
