## Summary

Keep this open. Current main still forwards iMessage reply context only when the RPC payload already includes resolved `reply_to_*` fields; it does not accept GUID-only inline reply metadata or enrich from `chat.db`. The issue remains valid, and open PR #42683 is already the active implementation path for the same gap.

## What This Changes

Resolve inbound iMessage inline replies so reply/thread metadata is included in the agent-visible `ReplyToId`, `ReplyToBody`, and `ReplyToSender` context.

## Best Possible Solution

Land or replace the existing iMessage-focused fix so the bundled iMessage plugin enriches GUID-only inline replies into `ReplyToId`, `ReplyToBody`, and `ReplyToSender` before dispatch, using the existing `dbPath` surface for any read-only fallback and covering lookup failure modes in focused tests.
