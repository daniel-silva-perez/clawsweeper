## Summary

Keep this open. Current main still only forwards iMessage reply context when the RPC payload already contains resolved `reply_to_*` fields; it does not model GUID-only inline reply metadata or enrich messages from `chat.db`. The issue discussion has a maintainer keep-open signal and open PR #42683 is already the active implementation path, so this should stay in maintainer review rather than be closed or queued as a separate repair lane.

## What This Changes

The issue requests that inbound iMessage inline replies include the original replied-to message context when delivered to the agent.

## Best Possible Solution

Land or replace the existing iMessage-focused fix so the bundled iMessage plugin enriches GUID-only inline replies into `ReplyToId`, `ReplyToBody`, and `ReplyToSender` before dispatch, using the existing `dbPath` surface for any read-only fallback and covering lookup failure modes in focused tests.
