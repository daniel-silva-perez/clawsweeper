## Summary

Current main still does not resolve iMessage inline reply/thread context when imsg omits resolved reply_to_* fields. The active iMessage plugin only accepts already-resolved reply_to_id/reply_to_text/reply_to_sender, does not model thread_originator_guid or reply_to_guid, and enqueues parsed RPC notifications without chat.db enrichment. Open PR #42683 is still the closest implementation candidate, but it is open, unmerged, and targets the old src/imessage path rather than the active extensions/imessage path.

## Best Possible Solution

Keep this issue open until the bundled legacy iMessage plugin resolves reliable reply-origin metadata into the existing inbound ReplyToId, ReplyToBody, and ReplyToSender fields. The best path is to review, port, or supersede PR #42683 in extensions/imessage/src/monitor/*, prefer thread_originator_guid or other reliable origin metadata, use a read-only local chat.db lookup only when appropriate, fail closed on permission/schema/locking errors, and add regression tests for GUID-only inline replies.
