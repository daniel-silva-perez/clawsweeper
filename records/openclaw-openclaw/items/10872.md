## Summary

Current main still only forwards iMessage reply context when the imsg payload already includes resolved reply_to_id, reply_to_text, and reply_to_sender. It does not model or resolve thread_originator_guid / reply_to_guid, and there is no chat.db fallback before dispatch. The linked open PR #42683 remains a useful implementation candidate, but it is not merged or shipped, so this issue should stay open.

## Best Possible Solution

Keep this issue open until the bundled legacy iMessage plugin resolves imsg/chat.db reply-origin metadata into the existing inbound ReplyToId, ReplyToBody, and ReplyToSender fields. The likely path is in extensions/imessage/src/monitor/*: enrich missing reply context before inboundDebouncer.enqueue, prefer reliable origin GUID metadata, use read-only local chat.db lookup only when appropriate, fail closed on permission/schema/locking errors, skip unsafe remote assumptions, and add regression tests for GUID-only inline replies. PR #42683 can be reviewed, ported, or superseded by a maintainer-approved replacement.
