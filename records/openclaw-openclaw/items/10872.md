## Summary

Keep this issue open. Current main still only forwards iMessage reply context when the incoming payload already contains resolved `reply_to_*` fields; it does not handle GUID-only inline reply metadata or enrich from `chat.db`. The discussion and open PR #42683 are still the active path for this gap, so this should go to maintainer review rather than closure or a new repair lane.

## What This Changes

The issue requests that inbound iMessage inline replies include the replied-to message context in the agent-visible payload.

## Best Possible Solution

Land or replace the existing iMessage-focused fix so the bundled iMessage plugin resolves reliable inline-reply origin metadata into `ReplyToId`, `ReplyToBody`, and `ReplyToSender` before dispatch. Keep the change in `extensions/imessage/src/monitor/*`, use the existing `dbPath` surface for any read-only fallback, and add regression coverage for GUID-only replies and failure modes.
