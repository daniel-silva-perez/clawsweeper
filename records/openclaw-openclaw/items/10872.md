## Summary

Keep this open. Current main still forwards iMessage reply context only when the incoming payload already contains resolved `reply_to_*` fields; it does not model GUID-only inline reply metadata or enrich from `chat.db`. The issue also has a maintainer keep-open signal and open PR #42683 already targets this gap, so the right path is maintainer review of that PR rather than closure or a separate repair lane.

## What This Changes

The issue requests iMessage inbound inline replies to include the original replied-to message context so agents can understand what the user is responding to.

## Best Possible Solution

Land or replace the existing iMessage-focused fix so the bundled iMessage plugin enriches GUID-only inline replies into `ReplyToId`, `ReplyToBody`, and `ReplyToSender` before dispatch, using the existing `dbPath` surface for any read-only fallback and covering lookup failure modes in focused tests.

## Reproduction Assessment

Yes. A focused source-level reproduction is an imsg RPC notification with reply GUID metadata but no resolved `reply_to_text`: current main parses only the resolved `reply_to_*` fields and `describeReplyContext` returns null without a reply body.

## Solution Assessment

Yes, keeping this open for the existing PR path is the best current handling. The narrow maintainable fix belongs in `extensions/imessage/src/monitor/*`; a separate core hook or duplicate automated repair PR would add churn while PR #42683 is already active.
