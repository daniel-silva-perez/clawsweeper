## Summary

Keep open. Current main still sends WhatsApp read receipts before accepted inbound messages reach enrichment, inbound dedupe commit, debounce flush, or `onMessage`, and the WhatsApp inbound dedupe path is memory-only rather than a durable replay boundary. Ryce's KeepMyClaw comment is useful complementary recovery context, but it does not replace a bundled WhatsApp plugin ordering/durability fix.

## What This Changes

The issue reports that WhatsApp inbound messages can be acknowledged before OpenClaw has durably stored or processed them, causing silent message loss if the gateway crashes.

## Best Possible Solution

Implement the fix in the bundled WhatsApp plugin: delay `readMessages` until accepted inbound turns have either been durably stored for replay or have successfully reached `onMessage` and dedupe finalization, while preserving privacy behavior for blocked senders, self-chat mode, disabled read receipts, append history, media failures, debounce batching, and retryable errors. Update focused tests and WhatsApp docs so the new ordering is executable and user-visible.
