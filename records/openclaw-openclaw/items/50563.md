## Summary

Current main still has the reported WhatsApp ack-before-process path: read receipts are sent immediately after normalization and before append filtering, enrichment, inbound dedupe claim/commit, debounced `onMessage` delivery, or any durable inbound replay boundary. Ryce's KeepMyClaw comment is useful complementary crash-state recovery context, but it does not replace a bundled WhatsApp plugin ordering or persistence fix.

## What This Changes

The issue requests preventing WhatsApp inbound message loss by persisting or processing accepted messages before sending WhatsApp read receipts.

## Best Possible Solution

Keep the issue open and fix it in the bundled WhatsApp plugin by delaying `readMessages` until accepted inbound turns are durably stored for replay or have successfully reached `onMessage` and inbound dedupe finalization, with docs and focused tests covering the new ordering.

## Repair Work Prompt

Fix the WhatsApp ack-before-process message-loss issue. Current `extensions/whatsapp/src/inbound/monitor.ts` calls `maybeMarkInboundAsRead` before append filtering, enrichment, inbound dedupe claim/commit, debounce flush, and `onMessage`. Implement a plugin-local fix so accepted `notify` inbound messages are marked read only after they are durably stored for replay or after `onMessage` succeeds and inbound dedupe is finalized. Keep blocked senders and self-chat from sending read receipts, preserve `sendReadReceipts: false`, and handle retryable inbound errors so redelivery remains possible. Be careful with media/enrichment failures, debounce batching, group participant read receipt payloads, and append history: stale append cleanup may keep explicit mark-read behavior only if tests/docs make clear it is not acknowledging a turn that should be processed. Prefer changes in the WhatsApp plugin; do not introduce core owner-specific behavior or broad plugin API changes unless a generic seam is clearly necessary. Update focused tests and the WhatsApp docs, and add a changelog fix entry with reporter credit if behavior changes are user-facing.
