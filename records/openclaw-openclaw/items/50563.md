## Summary

Keep open. Current main still sends WhatsApp read receipts before append filtering, enrichment, inbound dedupe claim/commit, and debounced `onMessage` delivery, so the reported ack-before-process crash window remains. Ryce's KeepMyClaw comment is useful complementary crash-state recovery context, but it does not replace a bundled WhatsApp plugin ordering or durability fix.

## What This Changes

The issue asks OpenClaw to prevent WhatsApp inbound message loss by avoiding read-receipt acknowledgment before the message is durably stored or successfully processed.

## Best Possible Solution

Fix the bundled WhatsApp plugin so accepted inbound messages are either durably recorded for replay before `readMessages` or only marked read after successful `onMessage` delivery and inbound dedupe finalization, with focused tests and docs describing the new acknowledgment boundary.

## Repair Work Prompt

Fix the WhatsApp ack-before-process crash window reported in https://github.com/openclaw/openclaw/issues/50563. Current `extensions/whatsapp/src/inbound/monitor.ts` calls `maybeMarkInboundAsRead` before append filtering, enrichment, inbound dedupe claim/commit, and `onMessage` delivery. Implement a plugin-local fix that either persists accepted inbound messages for replay before `sock.readMessages` or delays read receipts until successful `onMessage` delivery and inbound dedupe finalization. Keep the change scoped to the WhatsApp plugin and existing SDK seams; do not add core owner-specific policy or change Baileys dependency versions. Preserve privacy behavior for unauthorized senders, self-chat mode, and `sendReadReceipts: false`. Decide and test the stale `append` behavior explicitly because current tests expect stale append history to be marked read while skipped for auto-reply. Update focused WhatsApp tests to prove read receipts are not sent before the chosen durable/success boundary, including failure/retry behavior. Update `docs/channels/whatsapp.md` if the user-visible read-receipt timing changes.
