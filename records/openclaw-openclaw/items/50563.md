## Summary

Keep open. Current main still acknowledges accepted WhatsApp inbound messages before there is durable inbound storage or replay: the WhatsApp listener marks messages read before enqueueing them, ack reactions are sent before reply dispatch, and the inbound debounce/dedupe path is in-process rather than a WAL. Existing durable recovery covers outbound/session delivery, not inbound WhatsApp message replay. Ryce’s KeepMyClaw comment is useful state-recovery context, but it does not replace an OpenClaw-side persist-before-ack path for accepted inbound messages.

## Best Possible Solution

Keep this issue open and implement a durable inbound acceptance journal for WhatsApp, or a generic plugin SDK inbound-WAL seam if multiple channels need it. The journal should persist the normalized accepted inbound message before read receipts/ack reactions, mark it processed only after the agent turn is durably queued or completed, and replay unprocessed entries on gateway restart. Coordinate with the broader persistent-queue discussion in #40079 if maintainers want one canonical reliability effort.

## Risks / Open Questions

- A gateway process exit between WhatsApp `readMessages` or ack reaction and the later in-memory debounce/agent dispatch can still drop an accepted inbound message without OpenClaw replaying it.
- The existing outbound/session delivery queues may give a false sense of coverage for this report because they do not journal inbound WhatsApp messages before acknowledgment.
