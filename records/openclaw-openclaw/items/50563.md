## Summary

Keep open. Current main still sends WhatsApp read receipts before the accepted inbound message reaches enrichment, inbound dedupe claim/commit, debounced `onMessage` delivery, or any durable replay boundary, so the reported crash-loss window remains. Ryce's KeepMyClaw comment is useful complementary gateway-state recovery context, but it does not replace a bundled WhatsApp plugin ordering or durability fix.

## What This Changes

The issue reports that WhatsApp inbound messages can be marked read before they are durably recorded or processed, causing silent message loss if the gateway crashes mid-processing.

## Best Possible Solution

Fix the bundled WhatsApp plugin so accepted inbound messages are either durably recorded for replay before `readMessages` or only marked read after successful `onMessage` delivery and inbound dedupe finalization, with docs and focused tests covering the new acknowledgment boundary.
