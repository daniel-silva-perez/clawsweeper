## Summary

Keep open. Current main still sends WhatsApp read receipts before durable inbound storage, inbound dedupe commit, or successful handler delivery, so the reported crash-loss window remains present. Ryce's KeepMyClaw comment is useful complementary recovery context for gateway state snapshots, but it does not replace fixing the bundled WhatsApp plugin's acknowledgment boundary.

## What This Changes

The issue requests preventing WhatsApp inbound message loss by using WAL, persist-before-ack, or delayed read receipts instead of acknowledging before processing completes.

## Best Possible Solution

Fix this in the bundled WhatsApp plugin by moving read receipts after accepted inbound turns are durably recorded for replay or after successful `onMessage` delivery plus inbound dedupe finalization, with focused tests proving no accepted message is marked read before the chosen durable/processed boundary.

## Reproduction Assessment

Yes. A default accepted WhatsApp `notify` upsert reproduces the ordering statically: `readMessages` is awaited before enrichment, dedupe claim, enqueue, and debounced `onMessage`, so a crash in that interval matches the reported loss window.

## Solution Assessment

Yes, the requested direction is right, but the narrowest maintainable fix is plugin-owned delayed acknowledgment or plugin-owned persistent inbound replay rather than a broad core migration. A full WAL can be considered if delayed read receipts are not enough for media/history edge cases.
