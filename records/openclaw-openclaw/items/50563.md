## Summary

Keep open. Current main still sends WhatsApp read receipts before durable inbound storage, session recording, successful `onMessage` delivery, or inbound dedupe commit, so the reported crash-loss window remains present. Ryce's KeepMyClaw comment is useful complementary recovery context for gateway state snapshots, but it does not replace fixing the bundled WhatsApp plugin's acknowledgment boundary.

## What This Changes

The issue requests preventing WhatsApp inbound message loss when the gateway crashes after acknowledging a message but before processing it.

## Best Possible Solution

Fix this in the bundled WhatsApp plugin by moving `readMessages` after accepted inbound turns are durably recorded for replay or after successful `onMessage` delivery plus inbound dedupe finalization, with focused tests proving no accepted message is marked read before the chosen durable or processed boundary.

## Reproduction Assessment

Yes. A default accepted WhatsApp `notify` upsert follows a static path where `readMessages` is awaited before enrichment, dedupe claim, debouncer enqueue, `onMessage`, and session recording; a crash in that interval matches the reported loss window.

## Solution Assessment

Yes, the requested persist-before-ack or delayed-ack direction is appropriate. The narrowest maintainable fix appears plugin-owned delayed read receipts or plugin-owned persistent inbound replay, not a broad core migration unless multiple channel plugins need the same durable inbound seam.
