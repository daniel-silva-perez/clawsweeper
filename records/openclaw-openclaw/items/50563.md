## Summary

Current main still has the reported WhatsApp crash-loss window: accepted inbound messages are marked read before enrichment, durable storage, dedupe commit, enqueue completion, or agent processing. The existing durable queue evidence I found is outbound/session-delivery oriented, not an inbound WhatsApp WAL. Ryce's KeepMyClaw comment is useful complementary recovery context, but it does not replace a core persist-before-read-receipt boundary.

## Best Possible Solution

Keep this issue open and design a WhatsApp inbound durability boundary: persist accepted inbound turns before sending read receipts, or delay read receipts until processing reaches a durable/replayable point. The design should cover notify and append upserts, debounce, dedupe claim/release/commit, media metadata, ack reactions, docs, and focused WhatsApp monitor tests. Treat KeepMyClaw-style snapshots as complementary recovery context, not a substitute for core read-receipt ordering.
