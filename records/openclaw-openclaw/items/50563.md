## Summary

Keep open. Current main still sends the WhatsApp read receipt before append filtering, enrichment, inbound dedupe claim/commit, debounced `onMessage` delivery, or any durable replay boundary. Ryce's KeepMyClaw comment is useful complementary gateway-state recovery context, but it does not replace a bundled WhatsApp plugin ordering or durability fix.

## What This Changes

The issue requests preventing WhatsApp inbound message loss by moving read receipts behind durable storage or successful processing, for example via WAL, persist-before-ack, or delayed ack.

## Best Possible Solution

Fix the bundled WhatsApp plugin so accepted inbound messages are either durably recorded for replay before `readMessages` or only marked read after successful `onMessage` delivery and inbound dedupe finalization, with tests and docs covering the new acknowledgment boundary.

## Repair Work Prompt

Fix the WhatsApp ack-before-process message-loss window reported in this issue. Current `extensions/whatsapp/src/inbound/monitor.ts` calls `maybeMarkInboundAsRead` before append filtering, enrichment, `claimRecentInboundMessage`, `enqueueInboundMessage`, debounced `onMessage`, and dedupe commit; `extensions/whatsapp/src/inbound/dedupe.ts` currently uses memory-only `createClaimableDedupe`. Make a narrow bundled WhatsApp plugin fix: either durably record accepted inbound messages for replay before sending `sock.readMessages`, or delay `sock.readMessages` until the accepted message has successfully reached `onMessage` and inbound dedupe finalization. Keep WhatsApp-specific policy in the WhatsApp plugin unless a minimal generic SDK seam is truly required. Update tests that currently assert eager read receipts and add regression coverage that read receipts are not sent before the chosen durability/processing boundary, including failure behavior. Update `docs/channels/whatsapp.md` and `CHANGELOG.md` if user-facing behavior changes; any changelog entry should be one line and credit `Thanks @coywolffuturist`. Do not change unrelated channel behavior or broad gateway queue architecture.
