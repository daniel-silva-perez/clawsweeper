## Summary

Keep open. Current main still marks accepted WhatsApp inbound messages as read before enrichment, dedupe commit, debounced dispatch, or agent processing, and I did not find an inbound WhatsApp WAL/replay boundary. Ryce's KeepMyClaw comment is useful complementary crash-recovery context, but it does not replace fixing the bundled WhatsApp read-receipt ordering.

## What This Changes

The issue requests a WhatsApp inbound durability boundary so accepted messages are durably stored or processed before the gateway sends a read/ack signal that can make crash-time loss silent.

## Best Possible Solution

Keep this issue open and fix the bundled WhatsApp plugin by adding an inbound durability boundary: either persist accepted inbound turns before sending read receipts, or delay read receipts until `onMessage` reaches a durable/replayable or completed processing point. Preserve current privacy behavior for blocked/self-chat messages, update WhatsApp docs, and add focused monitor tests for notify, append, debounce, dedupe, media metadata, and failure paths. Treat KeepMyClaw snapshots as complementary recovery context, not a substitute for core read-receipt ordering.

## Repair Work Prompt

Fix the WhatsApp inbound ack-before-process crash-loss window tracked by this issue. In the bundled WhatsApp plugin, accepted inbound messages must not send WhatsApp read receipts before the turn is either durably persisted for replay or `onMessage` has reached a completed/durable processing point. Prefer a narrow delayed-read-receipt implementation if it satisfies the issue without introducing a new broad storage subsystem; otherwise add a small WhatsApp-owned inbound WAL under the existing OpenClaw state conventions. Preserve blocked-sender and self-chat behavior, preserve `sendReadReceipts: false`, handle notify and append upserts intentionally, and make debounce batching mark each accepted message only after the batch is safe. Keep ack reactions explicit: either leave them pre-reply with docs/tests explaining they are not read receipts, or move them behind the same boundary if required by the implementation. Do not change unrelated channels, outbound delivery queue behavior, provider routing, or global core startup migration behavior. Update `docs/channels/whatsapp.md` if visible receipt timing changes. Add focused tests for crash-window ordering/failure behavior, dedupe retry/release/commit, append upserts, and media metadata preservation. Changelog only if maintainers expect this user-facing WhatsApp behavior change to ship in the active release notes.
