## Summary

Keep open. Current main still sends WhatsApp read receipts before the accepted inbound message reaches enrichment, dedupe commit, debounced dispatch, or agent processing, and I did not find a durable inbound WhatsApp replay/WAL boundary. Ryce's KeepMyClaw comment is useful complementary recovery context, but it does not replace fixing the bundled WhatsApp plugin's read-receipt ordering.

## What This Changes

Add a WhatsApp inbound durability or processing boundary so accepted messages are stored or handled before the gateway sends read receipts/acknowledgment.

## Best Possible Solution

Implement the fix in the bundled WhatsApp plugin by delaying read receipts until accepted inbound turns have reached a durable/replayable point or completed `onMessage`, with explicit handling for append-history, blocked senders, self-chat privacy, media enrichment failures, debounce batching, and dedupe release/commit semantics. Update WhatsApp docs and focused monitor tests so the new ordering is executable and user-visible behavior is documented.
