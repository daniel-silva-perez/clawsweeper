## Summary

Keep this issue open. Current main still has no reusable, platform-agnostic `@openclaw/gateway-client` workspace package or equivalent public SDK; it has a Node-oriented internal `GatewayClient`, a separate browser `GatewayBrowserClient`, protocol docs, and a plugin-runtime re-export of the internal client. The Brani5 reference implementation in the discussion is useful context, but it is external and does not satisfy the requested OpenClaw package/API design.

## Best Possible Solution

Keep this issue open for maintainer product/API design. The best path is an official gateway client package or explicitly scoped public SDK surface that shares generated protocol types, separates transport/storage/signing adapters, documents Node/browser/RN support boundaries, incorporates the current protocol docs and external reference implementation as input, and then migrates or parity-tests CLI and Control UI against the shared core.
