## Summary

Keep open. Current main now has useful App SDK progress, but it does not fully satisfy this public gateway-client request: the SDK package is still private/node-built, its transport wraps the internal Node `GatewayClient`, and the browser Control UI still carries its own separate client implementation. This remains a maintainer API/product-scope decision rather than conservative cleanup.

## What This Changes

Request an official reusable gateway WebSocket client SDK package for third-party consumers, with shared protocol handling and eventual CLI/Control UI reuse.

## Best Possible Solution

Turn the current App SDK work into an explicit supported public contract: either publish and document `@openclaw/sdk` as the gateway client surface, or split a lower-level adapterized gateway client package from it; then define Node/browser/RN support boundaries, keep protocol schema/types in lockstep, and add parity coverage for the existing Node and Control UI clients.

## Reproduction Assessment

Not applicable as a feature/API request. The current-main check is source-based: the SDK exists, but the package metadata and transport implementation show the requested public, platform-agnostic gateway-client surface is not complete.

## Solution Assessment

Unclear. The current `@openclaw/sdk` direction is a strong start for Node/TypeScript consumers, but closing this as solved would skip the package publication, low-level generated client, platform adapter, and Control UI drift concerns raised in the request.
