## Summary

Keep this issue open. Current main still lacks a reusable, platform-agnostic `@openclaw/gateway-client` package or equivalent public SDK: the workspace has no gateway-client package, the root export map exposes only `plugin-sdk/gateway-runtime`, the Node `GatewayClient` remains coupled to Node/ws/filesystem device auth, and Control UI still carries a separate browser client implementation. The Brani5 reference client linked in discussion is useful prior art, but it is external and does not satisfy the requested OpenClaw-supported package/API surface.

## Best Possible Solution

Keep this issue open for maintainer product/API design. The strongest path is an official gateway client package or clearly scoped public SDK surface that shares the generated protocol contract, separates transport/storage/signing adapters, documents Node/browser/RN support boundaries, incorporates the current protocol docs and Brani5 reference implementation as input, and then migrates or parity-tests the CLI and Control UI clients against the shared core.
