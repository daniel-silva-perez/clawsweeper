## Summary

Keep open. Current main still does not provide the requested reusable, platform-agnostic `@openclaw/gateway-client` workspace package. The repo has a Node/host-oriented `GatewayClient`, a separate browser `GatewayBrowserClient`, protocol docs, and a plugin SDK re-export, but the central request for an official consumable SDK package and shared implementation remains unresolved. The linked external reference implementation is useful context, not an implementation in this repository.

## Best Possible Solution

Keep this issue open for maintainer product/API design. The best path is a small official gateway client package or explicitly scoped public SDK surface that shares protocol types, separates transport/storage/signing adapters, documents Node/browser/RN support boundaries, and then lets CLI/UI migrate or at least parity-test against the shared core.

## Risks / Open Questions

- Closing now would drop a valid API/product request while third-party clients still need to choose between internal source imports, protocol docs, or external reverse-engineered implementations.
- A future SDK needs careful security review around device identity storage, device-token persistence, trusted retry endpoints, nonce signing, and the internal-only direct-loopback `gateway-client` backend exception.
