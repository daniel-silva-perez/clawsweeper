## Summary

Keep open. The current Control UI source still has a tool-boundary reset gap that matches the report: chat sends track a client-generated run id, while tool events are documented in the same file as using the server engine run id, yet the stream-to-segment flush still requires those ids to match. Multiple issue comments also confirm the client-only duplicate/cumulative rendering shape on 2026.4.24 and v2026.4.26.

## Best Possible Solution

Fix the Control UI live stream/tool-boundary handling so a non-empty assistant `chatStream` is committed as an independent stream segment and cleared when the first tool event for the visible active session arrives, even when the tool event run id is the server engine id rather than the client `chat.send` idempotency key. Add focused tests for text -> tool -> text -> tool turns and keep gateway transcript persistence unchanged.
