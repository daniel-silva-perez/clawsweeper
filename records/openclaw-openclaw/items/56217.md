## Summary

Keep this issue open. Current main still has startup SecretRef resolution fail fast without retry/backoff, the docs and tests still describe that fatal startup behavior, and open PR #56499 already references this issue with closing syntax, so cleanup should wait for that PR or an equivalent maintainer-reviewed fix.

## What This Changes

The issue requests backoff, circuit breaking, batching, or rate-limit-aware handling so repeated startup failures in exec-backed secret providers do not exhaust 1Password service account limits under a process-manager restart loop.

## Best Possible Solution

Review and either land PR #56499 or replace it with an equivalent narrow gateway startup fix that retries transient SecretRef provider failures in-process without weakening reload last-known-good behavior. Keep the broader external-provider batching and memoization work on #72278 rather than folding that whole design into the startup backoff fix.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to make `prepareRuntimeSecretsSnapshot`, or an active exec SecretRef inside it, fail during `activateRuntimeSecrets` with `reason: "startup"`; current source and tests show this rejects with the fatal startup message that a process manager can restart repeatedly.

## Solution Assessment

Unclear. Startup exponential backoff is a reasonable narrow first fix for the crash-loop amplification, but the full requested solution also needs a broader supported external-secret-provider model for batching and memoization across vault backends.
