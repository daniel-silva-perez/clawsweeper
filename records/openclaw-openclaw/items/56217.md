## Summary

Keep this issue open. Current main still fails gateway startup immediately when active SecretRefs cannot be resolved, and open PR #56499 already references this issue with closing syntax, so cleanup should wait for that PR or an equivalent fix to merge.

## What This Changes

The issue asks the gateway SecretRef startup path to avoid crash-looping external exec providers such as 1Password by adding backoff, circuit breaking, batching, or rate-limit-aware behavior.

## Best Possible Solution

Review and either land PR #56499 or replace it with an equivalent narrow gateway startup fix that retries transient SecretRef provider failures in-process without weakening reload last-known-good behavior. Keep the broader external-provider batching/memoization work in #72278 as a separate design track.

## Reproduction Assessment

Yes. The current-main reproduction path is to make prepareSecretsRuntimeSnapshot, or an active exec SecretRef inside it, fail during activateRuntimeSecrets(..., { reason: "startup" }); code and tests show that startup throws immediately, matching the launchd crash-loop trigger.

## Solution Assessment

Unclear. A narrow startup backoff is a reasonable first fix for transient failures, but it does not solve the full rate-limit problem when many configured exec providers still each invoke an external backend; the durable solution likely combines startup backoff with the batched provider model tracked in #72278.
