## Summary

Keep this issue open. Current main still treats startup SecretRef resolution failures as fatal, the docs and tests still encode startup fail-fast behavior, and open PR #56499 already references this issue with closing syntax, so cleanup should wait for that PR or an equivalent maintainer-reviewed fix.

## What This Changes

The issue requests retry/backoff, circuit-breaker, batching, and rate-limit-aware handling for gateway SecretRef provider failures so launchd restarts do not repeatedly exhaust 1Password service-account quotas.

## Best Possible Solution

Review and either land PR #56499 or replace it with an equivalent narrow gateway startup fix that retries transient SecretRef provider failures in-process without weakening reload last-known-good behavior. Keep the broader external-provider batching and memoization work tracked separately in #72278.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to make prepareRuntimeSecretsSnapshot, or an active exec SecretRef inside it, fail during activateRuntimeSecrets with reason "startup"; current source and tests show that this rejects with the fatal startup message that a process manager can restart repeatedly.

## Solution Assessment

Unclear. Startup exponential backoff is a reasonable narrow first fix, but the full requested solution also needs a broader supported external-secret-provider model for batching and memoization across vault backends.
