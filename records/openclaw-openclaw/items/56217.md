## Summary

Keep open. Current main still resolves active SecretRefs once during gateway startup and turns any provider failure into a fatal startup error without a retry/backoff/circuit-breaker path, and open PR #56499 already references this issue with closing syntax, so cleanup should wait for that PR or an equivalent maintainer-reviewed fix.

## What This Changes

The issue asks OpenClaw to prevent gateway SecretRef provider failures, especially 1Password `op read` failures under launchd restarts, from amplifying into external service-account rate-limit exhaustion.

## Best Possible Solution

Review and either land PR #56499 or replace it with an equivalent narrow gateway startup fix that retries transient SecretRef provider failures in-process without weakening reload last-known-good behavior. Keep broader external-provider batching and memoization work on #72278 unless maintainers decide to fold that product design into the same effort.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to make `prepareRuntimeSecretsSnapshot`, or an active exec SecretRef inside it, fail while `activateRuntimeSecrets` runs with `reason: "startup"`; current source and tests show that rejects with the fatal startup message that a supervisor can restart repeatedly.

## Solution Assessment

Unclear. Startup exponential backoff is a reasonable narrow first fix for crash-loop amplification, but the full requested outcome also needs a broader external-secret-provider batching/memoization model to reduce per-secret backend calls across vaults.
