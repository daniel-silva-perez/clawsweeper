## Summary

Keep open. Current main still handles the relevant Slack channel denial path as a dropped inbound message with `logVerbose` output rather than the requested warning-level, structured, deduped observability. An open linked PR, #58640, uses closing syntax for this issue and is the right place for maintainer review before the issue is closed.

## What This Changes

The issue requests warning-level structured logging when a resolved channel route explicitly denies a message via `allow: false` or its current canonical `enabled: false` equivalent.

## Best Possible Solution

Land or replace the linked Slack fix with a channel-owned warning path for explicit policy denials that includes channel, account, route key, and a stable reason such as `channel_not_allowed`, with a small rate-limit or dedupe guard and a regression test around the denied channel message path.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: configure a Slack room entry as denied (`enabled: false`, or legacy `allow: false` after doctor migration), send an inbound room message, and current main returns null after verbose-only drop logs.

## Solution Assessment

Yes in direction, but the implementation should cover the explicit channel-denial branch with warning-level structured metadata rather than only changing a generic final drop string. If the scope stays Slack-only, #58640 is the natural review target; if maintainers want all channels, use a shared channel logging helper.
