## Summary

Keep open. Current main still drops explicitly denied Slack channel messages after verbose-only logging, and Discord has a similar verbose-only explicit-deny path. An open PR already references this issue with closing syntax, so the issue should remain open until that PR or a replacement fix merges.

## What This Changes

The issue requests a structured warning when the gateway drops a resolved channel message because channel configuration explicitly denies it with allow: false or its current enabled: false equivalent.

## Best Possible Solution

Keep the issue open until #58640 or a replacement PR merges. The end state should be plugin-owned structured warning logs for explicit channel-deny decisions, with a stable reason such as channel_not_allowed and lightweight dedupe/rate limiting, without changing fail-closed routing semantics.

## Reproduction Assessment

Yes. A source-level reproduction is clear: configure Slack channels with an explicit disabled entry, send a channel message, and current main returns false from isChannelAllowed before prepareSlackMessage drops the message with only verbose logging.

## Solution Assessment

Unclear. Warning at the channel plugin denial boundary is the right direction, but a Slack-only change may not fully satisfy the broader gateway/channel request; the safer fix should cover the explicit-deny paths that share this behavior while preserving plugin ownership.
