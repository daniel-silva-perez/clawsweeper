## Summary

Keep open. The PR targets a real streaming reliability area and current main has adjacent fixes for cumulative/snapshot merge behavior, but this branch should not merge as-is because empty `replace` payloads can erase the stored assistant buffer; current main also moved the merge helper into `live-chat-projector.ts`, so the branch needs a focused rebase before maintainers can decide whether explicit Gateway replace plumbing is still needed.

## What This Changes

The PR adds Gateway assistant-stream `replace` handling, bypasses the chat delta throttle for replacements, and adds a regression test plus changelog entry for overlapping stream duplication.

## Best Possible Solution

Land a rebased patch on current main's live-chat projection flow that treats `replace` as an explicit full textual snapshot only when the payload has meaningful text, preserves existing buffers for empty/media-only replacements, keeps final-flush semantics intact, and covers both overlapping vLLM snapshots and empty replacement events.

## Reproduction Assessment

Yes for the PR defect: send a normal assistant text delta, then a `replace: true` assistant event with `text: ""`; the branch bypasses the empty guard and clears the raw buffer. The original vLLM symptom also has a plausible path, and current main includes tests for full `text_end` snapshot repeats.

## Solution Assessment

No. The explicit replace direction is reasonable, but this branch is not the safest implementation because it clears buffers on empty replacements and is stale against current main's `live-chat-projector.ts` split; a safer fix would consume `replace` in the current Gateway path with an empty-payload guard and targeted regression tests.

## Review Findings

- **[P1] Ignore empty replacements before clearing the buffer:** `src/gateway/server-chat.ts:113`
  - body: A `replace: true` event with `text: ""` now bypasses the empty-merge guard and writes an empty assistant buffer. Preserve the previous text or ignore empty replacement payloads instead.
