## Summary

Keep this PR open for maintainer/security review. Current main still lacks quoted HTTP client config/header redaction in the shared default redactor, and `formatErrorMessage` delegates to that redactor after assembling cause-chain messages, so the PR targets a real leak class; I found no blocking correctness or supply-chain issue in the three-file diff.

## What This Changes

This PR expands shared logging redaction patterns for quoted HTTP client config/header fields and adds regression tests for `redactSensitiveText` plus `formatErrorMessage` cause-chain output.

## Best Possible Solution

Review and land this narrow shared-redaction update if targeted tests and formatting pass, while keeping the broader exec-output and `/config show` secret-exposure work on their separate tracks.

## Reproduction Assessment

Yes. A code-level reproduction is an Error cause message like `request config: { appSecret: '...', headers: { authorization: 'Bearer ...' } }` passed through `formatErrorMessage`; current main assembles that text and applies default redaction patterns that do not match those quoted object fields.

## Solution Assessment

Yes for the narrow HTTP-client diagnostics gap. Extending the shared default redaction patterns is the narrowest maintainable fix because the logging, tool-payload, and error-formatting surfaces already depend on `redactSensitiveText`; it is not a complete fix for the related broader issues.
