## Summary

Keep open. Current main still has the quoted HTTP-client redaction gap, and this focused contributor PR appears correct and security-positive, but shared secret-redaction changes should receive maintainer/security review and validation before merge.

## What This Changes

This PR extends shared logging redaction patterns and adds regression tests for HTTP client config/header secrets in `redactSensitiveText` and `formatErrorMessage` cause-chain output.

## Best Possible Solution

Review and land the narrow shared redaction update after maintainer/security approval and targeted validation, while keeping the broader exec-output and `/config show` secret-exposure work on their separate tracks.

## Reproduction Assessment

Yes. A code-level reproduction is an Error cause message like `request config: { appSecret: '...', headers: { authorization: 'Bearer ...' } }` passed through `formatErrorMessage`; current main assembles that cause-chain text and lacks default patterns for those quoted object fields.

## Solution Assessment

Yes for the narrow HTTP-client diagnostics gap. Extending shared default redaction patterns is the narrowest maintainable fix because logging, tool-payload, and error-formatting surfaces already depend on `redactSensitiveText`; it is not a complete fix for the broader related issues.
