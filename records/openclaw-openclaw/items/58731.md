## Summary

Keep this PR open, but it is not merge-ready. Current main still keeps Slack thread participation in a process-local cache, so the restart-survival problem remains valid; however this PR now overlaps the merged plugin state-store foundation and open persistence PR #33845, and the proposed patch has concrete correctness issues in the new Slack fallback and mention-gating API behavior.

## What This Changes

The PR adds a Slack JSON thread-participation store persisted beside the session store, wires Slack prepare/dispatch to read and write it, adds thread_ts fallback/ambiguity handling and diagnostic logging, expands mention-gating types, adds tests, and includes an unrelated Dockerfile runtime CLI assertion.

## Best Possible Solution

Keep the Slack restart-survival fix alive, but consolidate it around one current design: preserve the in-memory hot path, add durable Slack participation state through the merged plugin state-store API or another maintainer-selected store, add a cold-start regression test, and remove or prove any Slack `conversations.replies` fallback before shipping it.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to record Slack thread participation, restart the gateway or clear the in-memory cache, then process an unmentioned reply in a `requireMention` Slack channel; current main only checks `hasSlackThreadParticipation`, so the continuation signal is gone. The PR findings are also directly reproducible with focused unit tests for `threadContinuation: true` and unresolved missing-`thread_ts` replies.

## Solution Assessment

No. The PR targets a real bug, but this is not the best form to merge because it adds a custom JSON persistence layer after the plugin state-store foundation landed, accepts a `threadContinuation` API that is ignored, and adds an unverified Slack API fallback that likely cannot recover a parent thread timestamp from a child reply timestamp.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Wire threadContinuation into mention decisions:** `src/channels/mention-gating.ts:14`
  - body: The new `threadContinuation` parameter is documented as contributing to `effectiveWasMentioned`, but both deprecated resolver entry points ignore it and the core decision still only uses explicit mentions, implicit mention kinds, and command bypass. A caller passing `threadContinuation: true` will still be skipped under `requireMention`, so either wire it into the decision or remove the new API field.
  - confidence: 0.89
- **[P2] Remove the reply-ts conversations.replies fallback:** `extensions/slack/src/monitor/thread-resolution.ts:57-60`
  - body: This fallback calls `conversations.replies` with the child reply's own timestamp, but that method retrieves a thread by its thread/root `ts`. In the exact case this code handles, the parent/root timestamp is unknown, so the fallback normally returns `thread_not_found` and burns an extra Slack Web API call without recovering `thread_ts`.
  - confidence: 0.87
- **[P2] Make ambiguous replies affect routing:** `extensions/slack/src/monitor/thread-resolution.ts:181-184`
  - body: The resolver now returns `_ambiguousThreadReply: true` for unresolved replies, but the downstream prepare/routing path still keys off `thread_ts` and does not consult this marker. That means an unresolved Slack reply can still be treated as a root message despite the new type comment saying downstream should not do so; route or drop these events explicitly, and keep the marker through cached-null hits.
  - confidence: 0.8
