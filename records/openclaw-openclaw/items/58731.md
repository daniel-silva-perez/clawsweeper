## Summary

Keep open. Current main still loses Slack thread-participation state on restart, but this branch is not merge-ready: its ambiguous-thread marker is not consumed downstream, the new fallback cache path can race under duplicate handlers, and related persistence work in #33845 and the merged #74190 state-store foundation need maintainer direction before choosing the final implementation.

## What This Changes

The PR adds a Slack-specific JSON thread-participation store, wires persistent participation checks into Slack prepare/dispatch, adds a `conversations.replies` fallback for missing `thread_ts`, adds diagnostic logging, extends mention-gating types, and includes related tests plus an unrelated Dockerfile test.

## Best Possible Solution

Keep the restart-surviving Slack thread-participation behavior, but land it through the Slack-owned path that maintainers choose after #74190: either migrate the in-memory hot path to `api.runtime.state.openKeyedStore` with fire-and-forget durable writes, or use the narrower hardened cache design from #33845. The final patch should also make unresolved `parent_user_id` messages consistently non-root, cover concurrent duplicate Slack events, and keep unrelated Dockerfile coverage out of the Slack PR.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: in a `requireMention` Slack channel, a bot reply records participation only in `sent-thread-cache.ts`; after restart that cache is empty, and `prepare.ts` only checks that in-memory cache before skipping a non-mentioned thread follow-up.

## Solution Assessment

No. The PR targets the right Slack behavior, but this exact patch is not the best merge path because it adds a custom synchronous JSON store after main gained a plugin state-store API, and its thread-resolution fallback/ambiguous-marker behavior is incomplete under retries and unresolved replies.

## Review Findings

Overall confidence: 0.86

Full review comments:

- **[P2] Honor ambiguous thread replies downstream:** `extensions/slack/src/monitor/thread-resolution.ts:181-184`
  - body: The resolver now returns `_ambiguousThreadReply` when both history and replies fail, but no downstream code reads that flag. Since `resolveSlackThreadContext` still treats messages without `thread_ts` as non-thread replies, the first unresolved thread reply can still flow through root-message routing/gating despite the comment saying it should not be treated as a genuine root message.
  - confidence: 0.91
- **[P2] Keep fallback resolution inside the shared inflight result:** `extensions/slack/src/monitor/thread-resolution.ts:155-167`
  - body: Only the `conversations.history` call is shared through `inflight`; after it resolves empty, duplicate handlers run `conversations.replies` independently and each writes the cache. A later transient fallback failure can overwrite an earlier successful thread resolution with `null`, so wrap fallback plus cache write in the shared promise or cache a full resolved/ambiguous result.
  - confidence: 0.86
- **[P3] Wire or remove threadContinuation:** `src/channels/mention-gating.ts:14-31`
  - body: The new `threadContinuation` fields are added to the deprecated mention-gate parameter types with comments claiming they contribute to `effectiveWasMentioned`, but the gate implementation never reads them. Either remove the fields or include them in the legacy normalization/core decision so callers are not given a no-op option.
  - confidence: 0.88
