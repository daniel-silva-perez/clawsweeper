## Summary

Keep open. Current main does not implement this PR’s central behavior: Slack thread participation is still a 24-hour in-memory cache, dispatch only records to that cache, and mention gating only consults that cache. The bounded thread_ts fallback, ambiguous-thread marker, persisted participation store, and separate threadContinuation signal from the PR are also absent on main. Related PRs #33845 and #70626 are open, not merged, so they are candidates for the canonical path rather than shipped replacements.

## Best Possible Solution

Keep this PR open for maintainer/contributor follow-up. The best path is to choose a canonical persistence approach, likely by finishing the generic persistent keyed store in #70626 or the focused Slack persistence work in #33845, then wire Slack thread participation through that path with a cold-start restart regression test. If this PR continues, it should fix or remove the questionable conversations.replies fallback, preserve ambiguous-thread semantics consistently, and drop the unrelated Docker assertion into a separate change.

## Risks / Open Questions

- The PR’s conversations.replies fallback has unresolved review comments saying it likely passes the reply ts where Slack expects the parent/root ts, which could make the fallback inert while consuming rate-limit budget.
- The persistence design writes a new JSON store from Slack runtime code; it should be reconciled with the open generic persistent keyed store work in #70626 or the narrower Slack persistence PR #33845 before choosing a canonical implementation.
- The PR includes an unrelated Docker runtime-image assertion in src/dockerfile.test.ts, which should be split or justified so a Slack behavior change does not carry unrelated infrastructure surface.
