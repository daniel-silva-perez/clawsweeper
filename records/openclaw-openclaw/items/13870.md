## Summary

Keep open. Current main at 5f13af6 still lacks the requested durable, editable paired-device naming workflow: it has client-supplied displayName metadata and displays that value when present, but there is no separate operator-owned label field, paired-device rename RPC/schema/handler, CLI rename command, or Control UI rename action. The related #23916 remains broader metadata display work, and closed #57424 was folded into this distinguishability cluster rather than superseding the rename request.

## What This Changes

The issue requests human-friendly, user-editable names for paired devices in the web UI and `openclaw devices list` instead of relying on long cryptographic device ids.

## Best Possible Solution

Add an additive paired-device label workflow: persist an operator-owned label separate from client displayName, define display precedence as operator label then client displayName then raw deviceId, expose a typed Gateway method for renaming paired devices with authorization, add CLI and Control UI rename actions, and document the behavior while keeping broader automatic metadata display in #23916 separate.
