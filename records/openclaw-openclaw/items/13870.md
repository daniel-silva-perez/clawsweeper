## Summary

Keep open. Current main still only carries client-supplied paired-device display metadata and does not provide durable operator-owned labels, a paired-device rename protocol/handler, or CLI/Control UI actions to set a friendly name; related open work covers richer display metadata rather than this editable naming workflow.

## What This Changes

Request to let users assign and display durable human-friendly names for paired devices in `openclaw devices list` and the Control UI.

## Best Possible Solution

Add an additive paired-device label workflow: persist an operator-owned label separate from client metadata, define display precedence as operator label, then client `displayName`, then raw `deviceId`, expose a `device.pair.rename` Gateway method with matching authz, add CLI and Control UI rename actions, and document the behavior while keeping broader automatic metadata display under #23916.
