## Summary

Keep open. Current main still lacks the requested end-to-end paired-device naming workflow: it stores and displays client-supplied `displayName` metadata, but there is no durable operator-owned label, no paired-device rename protocol or server method, and no CLI or Control UI rename action.

## What This Changes

Allow users to assign durable human-friendly labels to paired devices and show those labels in `openclaw devices list` and the Control UI Devices/Nodes panel.

## Best Possible Solution

Add an additive paired-device label workflow with durable operator-owned label storage, explicit display precedence over client metadata, a `device.pair.rename` Gateway method, CLI and Control UI rename actions, docs, focused tests, and no expansion into the broader metadata-display work tracked by #23916.

## Repair Work Prompt

Implement the paired-device human-friendly naming workflow for this issue. Add a durable operator-owned paired-device label separate from reconnect/client metadata, with display precedence such as operator label first, then client displayName, then deviceId. Add an additive `device.pair.rename` Gateway protocol schema/type/validator, method-list entry, method-scope entry, and server handler. The handler should enforce the same self/admin paired-device management boundary used by remove/rotate/revoke: non-admin paired-device-token callers may rename only their own device, while admin callers may rename any paired device. Update `openclaw devices` with a `rename --device <id> --name <label>` command and make list output use the resolved friendly label. Update the Control UI devices controller, Nodes view types, and paired-device UI with a rename action. Update `docs/cli/devices.md`, `docs/gateway/protocol.md`, and the changelog with appropriate user-facing credit, likely `Thanks @koome23` and optionally `Thanks @shokk` for the related comment.
