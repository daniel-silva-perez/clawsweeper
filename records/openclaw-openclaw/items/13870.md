## Summary

Keep open. Current main still lacks a durable, operator-owned paired-device label workflow: paired devices only have client-supplied displayName metadata, and there is no rename protocol method, server handler, CLI command, or Control UI action. The related #23916 tracks broader metadata display and does not supersede the editable naming request.

## What This Changes

Allow users to assign and display human-friendly labels for paired devices in the CLI and Control UI instead of relying on long device IDs.

## Best Possible Solution

Add an additive paired-device label feature: persist an operator-owned label separate from client displayName, define display precedence as label then displayName then deviceId, expose a typed Gateway rename method with authorization, add CLI and Control UI rename actions, and document the behavior.

## Reproduction Assessment

Not applicable as a runtime reproduction: this is a feature request. Current-main source inspection gives a high-confidence gap because the storage, protocol, server methods, CLI, and Control UI surfaces all lack the requested editable label workflow.

## Solution Assessment

Yes. A separate operator-owned label is the narrowest maintainable solution because client displayName is reconnect metadata and can be overwritten by the device; reusing it would not satisfy the durable rename requirement.
