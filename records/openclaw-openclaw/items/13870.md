## Summary

Keep open. Current main still lacks the requested durable, operator-owned paired-device naming workflow: paired records only have client-supplied display metadata, the Gateway protocol and handlers expose list/approve/reject/remove/token actions but no rename action, and both CLI and Control UI render existing metadata without a way to set a label. The maintainer and ClawSweeper comments in the discussion correctly keep this separate from the broader metadata-display request in #23916.

## What This Changes

Request to let users assign and display human-friendly names for paired devices in `openclaw devices list` and the Control UI, ideally through durable paired-device storage and a CLI rename command.

## Best Possible Solution

Add an additive paired-device label workflow: persist an operator-owned label separate from client metadata, define display precedence over client `displayName` and raw `deviceId`, expose a `device.pair.rename` Gateway method with matching authz, add CLI and Control UI rename actions, update docs/changelog, and keep broader automatic metadata display under #23916.
