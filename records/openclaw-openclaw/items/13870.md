## Summary

Keep open. Current main still lacks a durable operator-owned paired-device label workflow: paired devices only expose client-supplied displayName metadata, and there is no paired-device rename method, server handler, CLI command, Control UI action, or docs entry. The related #23916 remains broader metadata-display work and does not supersede this editable naming request.

## What This Changes

Allow users to assign and view durable human-friendly names for paired devices in the web UI and `openclaw devices list`.

## Best Possible Solution

Add an additive paired-device label feature: persist an operator-owned label separate from client displayName, define display precedence as label then displayName then deviceId, expose a typed Gateway rename method with authorization, add CLI and Control UI rename actions, and document the behavior.

## Reproduction Assessment

Not applicable as a runtime reproduction: this is a feature request. Current-main source inspection gives a high-confidence gap because storage, protocol, server methods, CLI, Control UI, docs, and tests all lack the requested editable label workflow.

## Solution Assessment

Yes. A separate operator-owned label is the narrowest maintainable solution because client displayName is reconnect metadata and can be overwritten by the device; reusing node.rename would also target the wrong pairing surface.
