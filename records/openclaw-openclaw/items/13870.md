## Summary

Keep open: current main still exposes only client-supplied paired-device metadata and does not provide a durable operator-owned name/label workflow for paired devices. The maintainer comment and related #23916/#57424 context support keeping this as the canonical editable naming request rather than closing it as broader metadata display work.

## What This Changes

Request to let operators assign and see human-friendly names for paired devices in `openclaw devices list` and the Control UI.

## Best Possible Solution

Add an additive paired-device label feature: persist an operator-owned label separate from client `displayName`, define display precedence as label then client displayName then deviceId, expose a typed Gateway rename method with authorization, add CLI and Control UI rename actions, and document the behavior.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. Current-main source inspection gives a high-confidence gap because storage, protocol, server methods, CLI, Control UI, docs, and tests all lack the editable paired-device label workflow.

## Solution Assessment

Yes: a separate operator-owned label is the narrowest maintainable direction because client `displayName` is reconnect metadata and can be overwritten, while `node.rename` targets a different pairing store.
