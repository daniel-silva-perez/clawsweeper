## Summary

Keep open with high confidence. Current main still only stores and displays client-supplied `displayName` metadata or the device ID, and it lacks a durable operator-owned label, paired-device rename RPC/schema/handler, CLI command, Control UI action, and docs. The related context supports keeping this as the canonical paired-device naming request: #23916 is broader metadata display, and #57424 was closed into this area rather than replacing the rename workflow.

## What This Changes

Add user-set human-friendly labels for paired devices so the CLI and Control UI can identify devices without relying on long cryptographic IDs or reconnect metadata.

## Best Possible Solution

Add an additive paired-device label workflow with durable operator-owned label storage, explicit display precedence over client metadata, a `device.pair.rename` Gateway method, CLI and Control UI rename actions, docs, and focused tests while leaving `node.rename` and broader metadata display as separate surfaces.
