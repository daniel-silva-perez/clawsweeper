## What This Changes

The PR adds a `config explain` CLI command, attaches provenance metadata to config snapshots, exports provenance helpers/types, and documents the new command.

## Best Possible Solution

Keep the PR open for author or maintainer follow-up: rebase it onto current main, complete provenance for every snapshot return path, make unapplied include status neutral or reason-aware, add focused config/CLI tests, and update the docs against the current `config patch` page before product/API review.

## Reproduction Assessment

Yes for the review findings: code inspection shows schema-invalid snapshots return without `provenance`, and `configExplainCommand` renders all unapplied entries as `(not found)`. I did not run live commands because this review had to keep the checkout read-only.

## Solution Assessment

No, the proposed patch is not the best mergeable solution as-is. The direction is reasonable, but the maintainable fix should cover all snapshot branches, avoid misleading include status text, and be rebased onto the current config CLI/docs surface.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Populate provenance on invalid config snapshots:** `src/config/io.ts:2320-2325`
  - body: In the schema-validation failure branch, this snapshot is returned without `provenance`. Since `configExplainCommand` falls back to empty provenance when the field is missing, a config that parses and resolves includes/env but fails schema validation loses the loaded-entry and key-source data that this command is meant to expose during troubleshooting. Build provenance here the same way the valid snapshot path does.
  - confidence: 0.88
- **[P2] Use neutral status text for unapplied includes:** `src/commands/config-explain.ts:36`
  - body: The renderer labels every `applied: false` entry as `(not found)`, but the provenance builder also marks circular include chains and failed ancestors as unapplied. Those are not missing-file cases, so `config explain` can point operators at the wrong root cause; render a neutral `(not applied)` status or carry the actual failure reason.
  - confidence: 0.86
