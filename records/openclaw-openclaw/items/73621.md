## Summary

Current main still has the reported Control UI failure mode. Busy follow-up sends are stored only in the in-memory `chatQueue`, both session-switch paths clear that queue and the active run id, and queue flushing only drains that same in-memory array after the current view becomes idle. The related open reports #45952 and #51549 cover reconnect/refresh durability, but this session-switch trigger remains a distinct, actionable bug.

## Best Possible Solution

Keep this issue open. The conservative fix is to make queued Control UI messages session-scoped, preserve the original session queue when navigating away, and flush it when that session's active run becomes idle. A broader durable design could share work with #45952 and #51549 through client persistence or gateway ACK/queue semantics, but this issue should at least get a focused regression test for switch-away/switch-back with a pending queued follow-up.
