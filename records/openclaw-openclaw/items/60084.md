## Summary

Keep this issue open. Current main still hard-wires builtin memory and task registries to file-backed `node:sqlite` `DatabaseSync`, only exposes `sqlite` as the memory store driver, and has no `better-sqlite3`, env-var driver, runtime lock probe, or in-memory/file-copy fallback for the reported ARM64 container `SQLITE_BUSY` path. Related memory fallback and FTS5 reports cover narrower memory-search degradation, not this broader file-backed SQLite/storage resilience issue that also affects task registries.

## Best Possible Solution

Keep the issue open as core storage resilience work. The best path is a shared SQLite storage seam or runtime lock probe used by builtin memory and task registries, with either automatic fallback or an explicit supported driver setting, regression coverage for `DatabaseSync` construction/first-statement `SQLITE_BUSY` failures, and aligned config/docs.
