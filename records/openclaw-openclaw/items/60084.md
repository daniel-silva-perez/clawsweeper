## Summary

Keep this issue open. Current main still has multiple file-backed SQLite paths that directly use `node:sqlite` `DatabaseSync`, the public memory config still exposes only the SQLite store driver, and no merged code provides the requested driver selection, runtime lock probe, or file-copy/in-memory fallback for the reported ARM64 container `SQLITE_BUSY` failure mode.

## What This Changes

Request supported SQLite storage resilience for ARM64 containers so memory, task, and other file-backed SQLite stores can recover from platform locking failures instead of hard-failing with `SQLITE_BUSY`.

## Best Possible Solution

Keep this as core storage resilience work. The desired end state is a shared SQLite storage/driver seam or runtime file-lock probe used by builtin memory, task registries, plugin state, and other core file-backed stores, with a documented fallback policy and regression coverage for first-statement `SQLITE_BUSY` failures.

## Reproduction Assessment

Unclear. The reporter provided a concrete ARM64 container repro, but this read-only review did not have that ARM64/Kubernetes environment; current main still exercises the same file-backed `DatabaseSync` paths that would hit the reported failure.

## Solution Assessment

Unclear. A memory-only fallback is not the best complete fix because task, Task Flow, plugin state, and proxy capture also use file-backed `node:sqlite`; the safer direction is a shared storage seam or lock-probe policy before choosing a native dependency or in-memory persistence fallback.
