Keep open. Current main no longer shows the original hardcoded Feishu tool-call log text, but the embedded tool lifecycle logger is still static under `agent/embedded` and does not receive the originating channel context already available upstream.

## Clownfish Work Prompt

Fix the current embedded tool lifecycle log attribution bug for this issue. Current main no longer has the old `feishu: tool call` string, but `subscribeEmbeddedPiSession` still uses a static `agent/embedded` logger and `handleToolExecutionStart` logs through that context, while `runEmbeddedAttempt` already has `messageChannel`/`messageProvider` upstream. Implement a narrow, generic fix: add channel/provider origin to `SubscribeEmbeddedPiSessionParams` or an equivalent logger-selection seam, pass `params.messageChannel ?? params.messageProvider` from `src/agents/pi-embedded-runner/run/attempt.ts` into `buildEmbeddedSubscriptionParams`, derive `gateway/channels/<channel>` only when a normalized channel value is available, and fall back to `agent/embedded`. Do not hard-code bundled channel IDs or resurrect raw `tool call params=...` logging. Preserve existing sanitized tool-event behavior and redaction. Add focused regression coverage proving a Telegram or Slack-origin tool start log uses the channel-derived subsystem and is not attributed to Feishu, plus existing redaction tests still pass. Changelog is likely not needed unless maintainers want a user-facing logging fix note.
