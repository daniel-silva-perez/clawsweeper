## Summary

Keep open. Current main no longer shows the old Feishu-specific `tool call` wording, but the underlying attribution problem remains: embedded tool lifecycle logs still inherit a static `agent/embedded` logger because the subscription layer is not given the originating message channel.

## What This Changes

Tool-call lifecycle logs should be attributed to the originating message channel instead of a hardcoded or static subsystem.

## Best Possible Solution

Add a typed channel-origin or logger seam from `runEmbeddedAttempt` into `subscribeEmbeddedPiSession`, derive a safe `gateway/channels/<channel>` logger for message-channel runs, keep `agent/embedded` as the fallback, and cover tool start/end attribution with focused regression tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an embedded run with `messageChannel: "telegram"` followed by a tool start/end event: the runner has the channel, but subscription construction drops it and the tool handler logs through the static embedded logger.

## Solution Assessment

Yes. The narrow maintainable solution is to pass channel origin or an already-derived logger through the existing runner-to-subscriber seam; a channel-specific workaround or broad logging rewrite would be less targeted.
