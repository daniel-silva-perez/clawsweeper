## Summary

Keep open. Current main no longer has the old Feishu-specific `tool call` strings, but the reported attribution problem is still present: embedded tool lifecycle logs are emitted through a static `agent/embedded` logger, while the runner has channel identity available and does not pass it into the subscription layer.

## What This Changes

The issue requests that tool-call lifecycle logs be tagged with the originating channel subsystem instead of a static or unrelated subsystem.

## Best Possible Solution

Tool lifecycle logging should use a normalized originating channel subsystem logger for message-channel runs, fall back to `agent/embedded` for non-channel/internal runs, and have focused regression coverage for start/end attribution without expanding raw tool logging.

## Repair Work Prompt

Fix the tool lifecycle log attribution bug tracked by https://github.com/openclaw/openclaw/issues/50565. Current main no longer contains the old Feishu-specific `tool call` strings, but `subscribeEmbeddedPiSession` still installs a static `agent/embedded` logger and `runEmbeddedAttempt` does not pass `messageChannel`/`messageProvider` into the subscription layer. Add a typed channel-origin or logger seam to `SubscribeEmbeddedPiSessionParams` and `buildEmbeddedSubscriptionParams`, pass `params.messageChannel ?? params.messageProvider` from `runEmbeddedAttempt`, and derive a logger with `createSubsystemLogger("gateway/channels/<normalized-channel>")` only after using the repo's message-channel normalization. Keep `agent/embedded` for missing, internal, or invalid channel origin. Do not change channel plugin behavior, plugin hook payloads, or add raw tool args/results/exec command text to debug logs. Add focused regression coverage proving Telegram/Slack-style channel origin reaches tool start/end debug logging and that channel-less runs keep the fallback. Add a single-line changelog fix entry in the active version crediting `Thanks @xucheng` because this is user-visible log behavior.
