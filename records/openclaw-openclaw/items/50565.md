## Summary

Keep open. Current main no longer contains the old Feishu-specific `tool call` wording, but the underlying channel-attribution bug remains: embedded tool lifecycle logs still inherit a static `agent/embedded` logger because the subscription layer is not given the originating message channel or provider.

## What This Changes

The issue requests that tool call lifecycle logs be attributed to the originating channel subsystem instead of a hardcoded or static subsystem.

## Best Possible Solution

Add a typed channel-origin or logger seam from `runEmbeddedAttempt` into `subscribeEmbeddedPiSession`, derive a safe `gateway/channels/<channel>` logger for message-channel runs, keep `agent/embedded` as the fallback for channel-less runs, and cover tool start/end attribution with focused regression tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path is a message-channel embedded run with `messageChannel` set, followed by a tool start/end event: current main has the channel on `RunEmbeddedPiAgentParams` but drops it before subscriber construction, so the lifecycle log cannot use the channel subsystem.

## Solution Assessment

Yes. The narrow maintainable fix is to pass channel origin or an already-derived logger through the existing runner-to-subscriber seam; a channel-specific workaround or broad logging rewrite would be less targeted.
