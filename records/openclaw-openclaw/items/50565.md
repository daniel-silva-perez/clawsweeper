## Summary

Keep open. Current main no longer shows the old literal `feishu: tool call` strings, but the remaining tool lifecycle log path is still not tagged with the originating channel. Tool start/end logs use the generic `agent/embedded` subsystem and omit channel metadata, while the run parameters already carry `messageProvider`/`messageChannel` that could be used to fix attribution.

## Best Possible Solution

Keep this issue open and fix the embedded tool lifecycle logging path: carry the originating channel into `SubscribeEmbeddedPiSessionParams` or derive it from the existing run channel/session context, then emit tool start/end/fail logs through a channel-scoped subsystem such as `gateway/channels/telegram` or include equivalent structured channel metadata. Add a regression test that simulates at least Telegram and Feishu runs and proves tool lifecycle logs are not all attributed to Feishu or the generic embedded-agent subsystem.
