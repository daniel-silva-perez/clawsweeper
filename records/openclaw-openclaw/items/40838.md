## Summary

Keep open. Current main still exposes Telegram sender name, id, and username to the agent context, but it does not expose `msg.from.is_bot` or an equivalent bot/source classification field; related #50482 is adjacent message metadata work and does not supersede this sender classification request.

## What This Changes

Expose Telegram sender/source metadata, especially `msg.from.is_bot`, to the agent context so agents can route bot and user messages differently.

## Best Possible Solution

Add a narrow Telegram-owned sender classification field, populated from `msg.from.is_bot`, and document it as untrusted provider metadata for agent routing. Keep the existing sender identity fields intact and avoid exposing the raw Telegram user object unless maintainers intentionally define a bounded structured contract.

## Reproduction Assessment

Not applicable as a runtime bug report, but the gap has a high-confidence verification path: construct a Telegram inbound message with `from.is_bot` true or false and inspect the reply payload. Current code and tests show only `SenderName`, `SenderId`, and `SenderUsername` are surfaced.

## Solution Assessment

Yes, the best path is a small Telegram/shared-context addition with focused tests and docs. A raw `msg.from` export would be broader and harder to treat safely than a stable boolean or bounded structured metadata field.
