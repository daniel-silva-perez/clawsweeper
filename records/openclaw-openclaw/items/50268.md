## Summary

Keep open. Current main still has the provider-auth ordering bug: configure requests default-model preservation, but real provider auth methods can merge a config patch that writes `agents.defaults.model.primary` before the preservation helper reads the previous primary. The merged #70793 work documents and partly covers the intended contract, but xAI/OpenRouter-style provider patches can still clobber an existing Bedrock or other primary.

## What This Changes

The issue reports that adding a second provider through onboard/configure can overwrite an existing `agents.defaults.model.primary` and leave the previous provider unusable without manual config repair.

## Best Possible Solution

Carry the pre-auth primary model through the provider-auth application path and base preservation decisions on that original value, while still allowing explicit default-setting flows and first-run/no-existing-primary setup to choose the new provider default. Add regression coverage with a provider auth method whose config patch includes `agents.defaults.model.primary`, preferably modeled after xAI/OpenRouter behavior.

## Repair Work Prompt

Fix the provider-auth default-model preservation bug tracked by https://github.com/openclaw/openclaw/issues/50268. Current `openclaw configure` asks `applyAuthChoice` to preserve an existing primary, but `runProviderPluginAuthMethod` merges provider `configPatch` before preservation reads the previous primary, so xAI/OpenRouter-style patches that include `agents.defaults.model.primary` can replace Bedrock or another existing primary. Implement a narrow fix, likely in `src/plugins/provider-auth-choice.ts`, that bases preservation on the original pre-auth config while still allowing explicit default-setting flows such as `openclaw models auth login --provider <id> --set-default` and first-run/no-existing-primary onboarding to set the new provider default. Add regression coverage with a plugin/provider auth method whose `configPatch` includes `agents.defaults.model.primary` and returns `defaultModel`; use xAI/OpenRouter-style behavior where practical. Do not change provider SDK contracts except for a backward-compatible internal seam if needed, and do not weaken explicit `--set-default` semantics. Add a single-line changelog fix entry under the active version thanking @rixcorp-oc. Validate with the targeted provider-auth/configure tests and `pnpm check:changed` through the normal Testbox gate if promoted by a maintainer.
