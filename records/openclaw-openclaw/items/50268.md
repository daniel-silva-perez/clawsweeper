## Summary

Keep this open. Current main still has the provider-auth default-model clobber path: provider `configPatch` output can write a new `agents.defaults.model.primary` before the preservation helper reads the prior primary, so configure/onboard can preserve the wrong value. PR #70793 improved the configure contract and the Keep My Claw comment is a useful recovery workaround, but neither fully fixes the OpenClaw source behavior for xAI/OpenRouter-style provider patches.

## What This Changes

Provider setup through onboard/configure should add a second provider without silently overwriting an existing `agents.defaults.model.primary`, except for initial setup or explicit default-model changes.

## Best Possible Solution

Preserve the pre-auth primary model through the provider-auth application path before merging provider-owned patches, then only replace it when the caller explicitly requested a default-model change. Add real-provider coverage for xAI/OpenRouter-style `applyConfig` patches and classic onboarding reruns, while keeping `openclaw models auth login --provider ... --set-default` and `openclaw models set` as intentional overwrite paths.

## Repair Work Prompt

Fix the provider-auth default-model preservation bug from this issue. Current `main` can merge a provider `configPatch` that already writes `agents.defaults.model.primary` before the preservation helper reads the previous primary, so adding xAI/OpenRouter-style providers through configure/onboard can preserve the wrong value. Keep the fix generic in the provider-auth application path: merge provider config, auth profiles, aliases, allowlist entries, and plugin enablement, but preserve the original pre-auth `agents.defaults.model.primary` when `preserveExistingDefaultModel` is true unless the caller explicitly requested a default-model overwrite. Cover provider patches that contain a primary model, especially xAI/OpenRouter-style `applyConfig` helpers, and cover classic onboarding reruns with an existing primary if that path still opts into setting defaults. Do not break `openclaw models auth login --provider <id> --set-default`, `openclaw models set`, initial setup with no primary, or provider allowlist merging. Add or update a user-facing changelog fix entry with `Thanks @rixcorp-oc` and credit PR #70793 context if the final change relies on it.
