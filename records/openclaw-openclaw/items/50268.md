## Summary

Keep this open. Current main still has a provider-default clobber path for real API-key providers whose `configPatch` already writes `agents.defaults.model.primary`: the patch is merged before the preservation helper checks the previous primary, and classic onboarding still opts into setting the default without preservation. PR #70793 improved the configure contract, and the Keep My Claw comment is a useful recovery workaround, but neither fully fixes the OpenClaw provider-auth behavior.

## What This Changes

The issue asks OpenClaw to avoid silently changing an existing configured primary model when a user adds or reconfigures another model provider through onboard/configure.

## Best Possible Solution

Fix the generic provider-auth application path so provider setup can merge provider config, auth profiles, aliases, and allowlist entries while restoring the original pre-auth `agents.defaults.model.primary` unless the caller explicitly requested a default-model change. Cover xAI/OpenRouter-style patches and classic onboarding reruns, while keeping `openclaw models auth login --provider ... --set-default` and `openclaw models set` overwrites intact.
