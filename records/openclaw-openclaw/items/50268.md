## Summary

Keep open. Current main still has a plausible xAI/OpenRouter-style path where provider auth applies an extension config patch that sets `agents.defaults.model.primary` before the default-model preservation logic runs. The existing Keep My Claw comment is a useful external recovery workaround, but it does not fix the OpenClaw config mutation bug.

## Best Possible Solution

Keep this issue open and fix the auth-choice flow so provider config patches cannot overwrite an existing primary before the default-model decision runs. A conservative implementation would apply provider/catalog/auth-profile config separately, then set or preserve `agents.defaults.model.primary` based on the original config and an explicit user/defaulting policy. Add a regression for adding xAI or OpenRouter when `amazon-bedrock/...` is already primary, and align the configure/onboard docs with the shipped behavior.

## Risks / Open Questions

- The current docs claim configure preserves an existing `agents.defaults.model.primary`, but the real provider patch ordering appears to undercut that guarantee for helpers that include a primary in `applyConfig`.
- Changing this should preserve first-run onboarding behavior while separating provider/catalog/auth-profile setup from the explicit default-model decision.
- Regression coverage should use a real bundled provider helper such as xAI or OpenRouter plus an existing Bedrock primary, not only a mocked provider that omits `configPatch`.
