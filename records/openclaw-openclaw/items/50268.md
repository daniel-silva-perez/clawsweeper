## Summary

Keep this issue open. Current main still has a real provider-default clobber path: provider API-key auth can return a `configPatch` that already contains `agents.defaults.model.primary`, and the generic auth flow merges that patch before the later `preserveExistingDefaultModel` logic inspects the existing primary. Configure opts into preservation but can still see the provider's newly written primary as the baseline, while classic onboarding still calls `applyAuthChoice` without the preservation flag at all. The Keep My Claw comment is a useful restore workaround, but it does not fix the OpenClaw provider auth behavior.

## What This Changes

The issue asks for provider onboarding/configure to stop overwriting an existing `agents.defaults.model.primary` when adding a secondary provider unless the user explicitly opts in.

## Best Possible Solution

Keep this issue open and fix the generic provider auth flow so provider setup can add provider config, auth profiles, aliases, and allowlist entries without clobbering an existing `agents.defaults.model.primary` unless the caller explicitly chooses to change the default. The repair should compare against the original pre-auth config, handle real xAI/OpenRouter-style `configPatch` values that already contain primary, apply the same protection to classic onboarding reruns, and keep explicit set-default commands working.

## Repair Work Prompt

Fix the provider default-model clobber tracked by openclaw/openclaw#50268. Current main merges provider auth `configPatch` before default-model preservation, so xAI/OpenRouter-style `applyConfig` patches that already contain `agents.defaults.model.primary` can overwrite an existing Bedrock/Anthropic/etc. primary before `preserveExistingDefaultModel` runs. Implement the fix generically in the provider auth flow, not with provider-specific IDs. Expected boundary: provider setup should still add provider config, auth profiles, aliases, and allowlist entries, but must preserve the original pre-auth `agents.defaults.model.primary` when the caller asks to preserve or when classic onboarding is adding a provider to an existing config without an explicit default-change choice. Keep explicit default-changing commands such as `openclaw models auth login --provider <id> --set-default` and `openclaw models set <model>` overwriting as documented. Cover real clobbering patch shapes, especially xAI/OpenRouter-style `applyConfig` that writes primary inside `configPatch`, and classic onboarding reruns. Do not change dependency versions, provider IDs, release metadata, or unrelated auth behavior. Add a single-line user-facing changelog fix under `CHANGELOG.md` Unreleased Fixes with `Thanks @rixcorp-oc`.
