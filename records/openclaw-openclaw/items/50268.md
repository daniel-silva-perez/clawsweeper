## Summary

Keep open. Current main still has a source-level clobber path: provider auth can merge a patch that already rewrites `agents.defaults.model.primary`, and the preservation helper then reads that mutated value as the previous primary. The recovery service mentioned in the discussion is useful context, and PR #70793 fixed an adjacent preservation contract, but neither removes this remaining OpenClaw config mutation bug.

## What This Changes

The issue requests that onboard/configure preserve an existing default model when adding a second provider, unless the user is doing initial setup or explicitly opts into changing the default.

## Best Possible Solution

Carry the pre-auth primary model through the provider-auth application path and use that original value for preservation decisions. The provider-owned patch should still add auth, provider config, aliases, and allowlist entries, but it should not change an existing primary unless the flow is fresh setup or an explicit default-setting command.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is available: start with a config whose primary is a Bedrock model, run the configure/provider-auth path with `preserveExistingDefaultModel: true`, and use an xAI/OpenRouter-style auth method that returns both a primary-setting `configPatch` and `defaultModel`; current code will treat the patched provider primary as the previous primary.

## Solution Assessment

Yes, a narrow code fix is the best path. The safer alternative to broad prompting is to make the existing preservation option read from the pre-auth config, then add targeted regression tests for provider patches that already contain `agents.defaults.model.primary`.
