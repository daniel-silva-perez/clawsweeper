## Summary

Keep open. Current main still has the source-level clobber path: real provider auth methods can return a `configPatch` that already rewrites `agents.defaults.model.primary`, and the preservation helper then reads that mutated value as the previous primary. PR #70793 fixed an adjacent preservation contract, and the Keep My Claw comment is a useful recovery workaround, but neither removes this OpenClaw config mutation bug.

## What This Changes

The issue reports that adding a second provider through onboard/configure can overwrite an existing `agents.defaults.model.primary` instead of preserving the current primary.

## Best Possible Solution

Carry the pre-auth primary model through the provider-auth application path and use that original value for preservation decisions. Provider-owned patches should still add auth, provider config, aliases, and allowlist entries, but they should not change an existing primary unless setup is fresh or the caller is an explicit default-setting command.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is available: start with a config whose primary is a Bedrock model, run the configure/provider-auth path with `preserveExistingDefaultModel: true`, and use an xAI/OpenRouter-style auth method that returns both a primary-setting `configPatch` and `defaultModel`; current code treats the patched provider primary as the previous primary.

## Solution Assessment

Yes, a narrow code fix is the best path. The safer solution is to make the existing preservation option read from the pre-auth config and add targeted regression tests for provider patches that already contain `agents.defaults.model.primary`, without changing explicit `--set-default` behavior.

## Repair Work Prompt

Fix the provider-auth default-model preservation bug tracked by https://github.com/openclaw/openclaw/issues/50268. Current `openclaw configure` passes `preserveExistingDefaultModel: true`, but `runProviderPluginAuthMethod` merges provider `configPatch` before `applyDefaultModelFromAuthChoice` reads the previous primary. xAI/OpenRouter-style API-key auth methods return both a primary-setting `configPatch` and `defaultModel`, so the preservation helper sees the new provider primary instead of the pre-auth Bedrock/current primary. Implement the narrow fix in the provider-auth application path: carry the pre-auth model/default primary through and use it for preservation decisions, or restore it before applying the default-model helper when preservation is requested. Keep explicit default-setting flows such as `models auth login --provider ... --set-default` overwriting as before. Do not move provider-owned xAI/OpenRouter behavior into core beyond the generic seam needed for preservation. Add regression coverage using a realistic provider auth method whose config patch contains `agents.defaults.model.primary` plus `defaultModel`; cover both preserve and explicit overwrite behavior. Update changelog if user-facing CLI bug fixes require it, crediting @rixcorp-oc and the PR author, never @codex.
