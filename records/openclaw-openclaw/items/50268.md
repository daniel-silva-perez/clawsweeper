## Summary

Keep open. Current main still merges provider-owned `configPatch` output before it tries to preserve the existing default model, so a real xAI/OpenRouter-style patch that already contains `agents.defaults.model.primary` can become the value that preservation keeps. Merged #70793 established the intended configure contract, but it does not close this provider-patch ordering gap.

## What This Changes

Prevent adding another provider through onboarding/configure from silently overwriting an existing `agents.defaults.model.primary` unless the user explicitly chooses a new default.

## Best Possible Solution

Carry the original pre-auth default model through the provider-auth application path and base preservation decisions on that original value, while still allowing explicit default-setting flows to overwrite it. Add regression coverage using a provider patch that includes `agents.defaults.model.primary`, preferably modeled after xAI/OpenRouter behavior.

## Repair Work Prompt

Fix the provider-auth default-model preservation bug tracked by openclaw/openclaw#50268. Current `openclaw configure` opts into `preserveExistingDefaultModel: true`, but provider auth methods can return a `configPatch` that already contains `agents.defaults.model.primary`; `runProviderPluginAuthMethod` merges that patch before `applyDefaultModelFromAuthChoice` reads the previous primary, so xAI/OpenRouter-style patches can still clobber an existing Bedrock/Anthropic/etc. primary. Keep the fix in the generic provider-auth path unless a smaller generic helper is cleaner. Preserve explicit default-setting behavior: `setDefaultModel: true` without preservation and explicit model commands must still overwrite, and `setDefaultModel: false` agent override behavior must continue restoring the prior config primary. Do not remove provider-owned model/provider patch merging. Add regression coverage for a provider auth patch that includes `agents.defaults.model.primary` plus `defaultModel`, and cover configure preserving an existing primary while still adding the new provider model/allowlist entry. Include xAI/OpenRouter-style coverage where practical. Update docs only if the public contract changes; otherwise no docs change is needed. If adding a user-facing changelog entry, put it in the active Fixes section and credit `Thanks @rixcorp-oc`.
