## Summary

Keep this issue open. Current main has DeepSeek/Venice-specific `reasoning_content` replay handling and generic stream capture, but I did not find an equivalent Qianfan or `baiduqianfancodingplan/qianfan-code-latest` fix. The related #71150 work is the same bug class, not a Qianfan implementation.

## Best Possible Solution

Keep this issue open and fix the provider-owned replay/payload path for Qianfan-like OpenAI-compatible thinking models. Either add a Qianfan-specific wrapper matching the DeepSeek/Venice `reasoning_content` placeholder behavior, or expose a documented compat option for custom providers that require `reasoning_content` on replayed assistant tool-call messages. Add focused regression coverage for Qianfan/custom OpenAI-compatible tool-call replay and update Qianfan docs if the old Coding Plan route is retired or renamed.
