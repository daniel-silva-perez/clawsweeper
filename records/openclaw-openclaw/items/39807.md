## Summary

Current main still does not give inline `models.providers.<id>.apiKey` credentials a durable auth-profile or provider cooldown identity. Billing failure tracking and later fallback skipping remain auth-profile based, while the related synthetic-profile PR #39836 is closed and unmerged in the provided context.

## Best Possible Solution

Keep this issue open and implement a stable cooldown identity for inline `models.providers.<id>.apiKey` credentials. The most direct fix is to return a synthetic profile id or provider-level cooldown key for inline API-key resolution, persist billing/auth failure state for that identity, and add regression coverage proving later turns skip the disabled inline provider and proceed to configured fallbacks.
