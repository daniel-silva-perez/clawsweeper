## Summary

Keep open. Current main still routes inline `models.providers.<id>.apiKey` credentials outside the durable auth-profile cooldown identity: inline config API keys resolve without `profileId`, the embedded runner skips failure recording when no profile id exists, and fallback pre-skipping only reasons over auth-profile ids. The related synthetic-profile PR #39836 was closed unmerged, so it does not supersede this issue.

## What This Changes

The issue asks OpenClaw to apply billing-error cooldown/backoff and fallback skipping to providers configured only with inline `models.providers.<id>.apiKey` credentials.

## Best Possible Solution

Add a generic, provider-agnostic cooldown identity for inline API-key credentials, then persist billing/auth failure state against that identity so later turns skip the disabled inline primary and walk configured fallbacks. The implementation should avoid MiniMax-specific logic and avoid storing plaintext inline keys in auth-profile state.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure a provider with only `models.providers.<id>.apiKey`, no auth profile, then a billing-classified failure has no `profileId` to record and no later profile-order entry for fallback skipping.

## Solution Assessment

Not applicable to a current patch. The maintainable direction is a stable inline-provider cooldown identity, either through a non-secret synthetic profile id or an equivalent provider-level usage key that is consumed by both failure recording and fallback pre-skip logic.
