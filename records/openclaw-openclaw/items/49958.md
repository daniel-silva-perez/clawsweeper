## Summary

Keep this PR open. Current main still does not implement the intended Feishu no-allowFrom group command fix, and the PR remains a plausible candidate for #49915. It should not be closed as obsolete, but it also should not merge as-is because the proposed shared fallback broadens command authorization beyond the Feishu group case and leaves the root SDK CJS mirror out of sync.

## Best Possible Solution

Keep this PR open for maintainer/contributor revision. The best path is a narrower fix that authorizes Feishu group commands when the group is already admitted and no sender allowFrom restriction is configured, while preserving open-DM non-authorization, keeping the root SDK CJS mirror aligned or generated, and adding direct Feishu group regression coverage plus DM negative coverage.

## Risks / Open Questions

- The PR's unconditional all-unconfigured allow fallback would also apply to shared DM command-gating callers, potentially authorizing control commands in open DMs that current tests intentionally keep non-authorized without explicit allowlists.
- The PR updates the TypeScript helper but not the mirrored root SDK CJS implementation, creating divergent command authorization behavior for plugins importing the root SDK entrypoint.
- Closing this PR now would drop an active implementation candidate for the still-open Feishu group command bug in #49915; current main still reproduces the central no-allowFrom group command denial.
