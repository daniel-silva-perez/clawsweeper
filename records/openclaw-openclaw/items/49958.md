## Summary

Keep this PR open. Current main still has the reported Feishu group command behavior: `useAccessGroups=true` denies when every authorizer is unconfigured, and the Feishu group path still passes an unconfigured authorizer when no `allowFrom` exists. The PR is a plausible implementation candidate for #49915, but it is not safe to close or merge as-is because the unconditional fallback would also affect DM command authorization paths and the root SDK CJS shim remains separately mirrored.

## Best Possible Solution

Keep this PR open as the active candidate for the linked Feishu group command bug, but require a narrower revision before merge: scope the no-`allowFrom` fallback to the intended group command flow or add an explicit command-gating mode for that case, preserve the open-DM no-allowlist contract, keep the root SDK CJS shim or generation path aligned, and add targeted tests for Feishu group commands plus DM non-authorization.

## Risks / Open Questions

- The PR's unconditional all-unconfigured allow fallback can authorize command handling in open DM flows that current tests intentionally keep non-authorized without explicit allowlists.
- The PR updates the TypeScript helper but not the root SDK CJS shim, so plugin SDK entrypoints could disagree after merge.
- The current Feishu test uses a mocked resolver return value, so the PR needs a direct regression test that proves the real Feishu group command path is authorized only for the intended no-`allowFrom` group case.
