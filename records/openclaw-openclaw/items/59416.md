## Summary

Keep this issue open. Current main appears to have addressed much of the display inconsistency by routing `/think`, session rows, and WebUI/TUI selectors through provider-owned thinking profiles, but the requested GitHub Copilot xhigh expansion is not fully present: the Copilot provider still does not advertise xhigh for `gpt-5.4-mini` or `gpt-5-mini`.

## What This Changes

The issue requests consistent thinking-level display/options across `/think`, `/status`, TUI/WebUI, plus expanded xhigh support for GitHub Copilot models such as `gpt-5.4-mini` and `gpt-5-mini`.

## Best Possible Solution

Keep the provider-profile architecture for consistency, then have a maintainer confirm the GitHub Copilot xhigh model matrix. If the mini models are confirmed to support xhigh, update the Copilot provider profile and focused tests so `/think`, `sessions.patch`, `/status`, TUI, and WebUI all inherit the same supported level set; otherwise document why those models intentionally remain high-only.

## Reproduction Assessment

Yes for the remaining source-level gap: on current main, inspect `extensions/github-copilot/index.ts` and see that `gpt-5.4-mini` and `gpt-5-mini` are absent from the Copilot xhigh allowlist. The original display inconsistency is now better covered by session row, slash-command, and WebUI source paths rather than a live Windows reproduction in this read-only review.

## Solution Assessment

Unclear for the remaining model-support request. The current provider-profile approach is the right maintainable shape for display consistency, but simply adding Copilot mini ids is only the best fix if maintainers confirm those Copilot routes support xhigh.
