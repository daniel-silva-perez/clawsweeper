## Summary

Keep open. Current main largely fixes the display-consistency side by deriving `/think`, session rows, and the Web UI picker from provider/model thinking profiles, but the requested GitHub Copilot mini-model `xhigh` support is still not implemented. The related closed items only cover stale preservation semantics or a different startup downgrade path, so a meaningful part of this issue remains.

## What This Changes

The issue reports inconsistent thinking-level display across `/think`, `/status`, TUI, and Web UI, plus missing `xhigh` support for additional GitHub Copilot models such as `gpt-5.4-mini` and `gpt-5-mini`.

## Best Possible Solution

Keep the profile-driven display behavior on main, use the narrower startup-profile issue for any remaining gateway startup work, and resolve the Copilot mini-model question inside the GitHub Copilot provider with capability proof plus focused profile tests. If those Copilot models do not actually support xhigh, document that boundary and close a narrower follow-up rather than broadening core policy.

## Reproduction Assessment

Yes for the remaining repository-level gap: current main lists the requested Copilot mini models but excludes them from the provider xhigh profile. A full live reproduction of Copilot accepting or rejecting xhigh for those models was not attempted.

## Solution Assessment

No patch is under review. The best fix, if provider capability is confirmed, is a narrow GitHub Copilot provider/profile update with tests; a core hardcoded allowlist would be weaker because thinking support is now provider-owned.
