## Summary

Keep this PR open. Current main and the latest release still hard-disable the Android Voice Tab manual mic path when `SpeechRecognizer.isRecognitionAvailable(context)` is false, and there is no configured OEM speech-recognizer fallback yet. The PR addresses a real remaining gap with concrete Xiaomi validation, but it needs revision before merge because it targets the old package path and would remove the hard stop without a no-backend retry safety path.

## Best Possible Solution

Keep this PR open for revision. The best path is to rebase it onto `apps/android/app/src/main/java/ai/openclaw/app/voice/MicCaptureManager.kt`, keep the generic configured-service approach using `Settings.Secure.getString(..., "voice_recognition_service")`, `ComponentName.unflattenFromString`, and `SpeechRecognizer.createSpeechRecognizer(context, component)`, and add a no-backend safety path that disables manual mic or avoids retrying when no usable recognizer exists. Add focused Android/Robolectric coverage around the creation seam if practical, or retain the Xiaomi manual proof plus a clear reason a platform test is not reliable.

## Risks / Open Questions

- Merging the PR as-is can regress unsupported Android devices: without a valid OEM service, the fallback default recognizer may produce `ERROR_CLIENT` or similar errors that current code retries every 600 ms instead of disabling the mic.
- The PR targets the old `ai.openclaw.android.voice` package path; current main uses `ai.openclaw.app.voice`, so it must be rebased before it can apply cleanly.
- There is no focused `MicCaptureManager` recognizer fallback regression test, so this path should get a small seam/Robolectric test if feasible, or clearly documented manual device proof if platform mocking is too brittle.
