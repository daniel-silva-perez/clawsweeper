## Summary

Current main still does not restore the reported cross-machine Chrome-extension relay workflow. The browser plugin intentionally keeps the extension relay lifecycle as a no-op, the supported browser profile drivers still exclude `extension`, profile creation rejects the legacy driver, doctor migrates old extension-relay config to `existing-session`, and docs point users to host-local Chrome MCP, node-host browser proxy, or raw remote CDP instead of a one-click Chrome extension relay. The issue is recent, concrete, confirmed by other users, and includes useful prior art in the linked ClawHosters relay, so it should remain open for maintainer product/security judgment rather than cleanup closure.

## Best Possible Solution

Keep this issue open for maintainer product and security judgment. The likely paths are either an explicitly opt-in, hardened extension relay in the browser plugin with schema/docs/tests/security review, or a narrow public browser profile-driver/transport API so the linked ClawHosters relay or a ClawHub plugin can integrate cleanly without private browser internals.

## Risks / Open Questions

- Restoring this relay would reintroduce a remote browser-control surface and needs explicit maintainer product/security review for authentication, origin validation, token handling, consent, public exposure, and operator threat model.
- Closing as ClawHub-only would be premature: a full replacement browser plugin may be possible, but current main does not provide a shipped or documented way to add an `extension` browser profile transport to the bundled browser tool.
- Current node-host, Chrome MCP, and remote CDP paths cover some remote-browser cases, but they do not solve the reported one-click Chrome extension install flow for non-technical users on managed remote OpenClaw instances.
