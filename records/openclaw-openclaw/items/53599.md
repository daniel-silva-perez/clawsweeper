## Summary

Keep open. Current main still removes the Chrome extension relay path, rejects the legacy extension driver, and documents host-local Chrome MCP, raw CDP, or node-host proxy alternatives; the reporter's managed-hosting workflow with `Yixn/clawhosters-relay` and the corroborating cross-machine comments remain unserved.

## What This Changes

The issue asks to restore or provide an optional plugin/gateway replacement for the removed Chrome extension browser relay so remote OpenClaw instances can control a user's local Chrome across machines without client-side Node.js or special Chrome flags.

## Best Possible Solution

Keep this open until maintainers choose a deliberate browser strategy: either a hardened opt-in Chrome extension relay owned by the browser plugin, or an additive public browser profile-driver/transport API that lets an external plugin implement the relay without private internals, with schema, docs, tests, and a threat model.
