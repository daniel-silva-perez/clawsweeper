## Summary

Keep open. Current main still intentionally removes the Chrome extension relay path, rejects the legacy profile driver, and documents Chrome MCP existing-session as host-local or node-host based rather than the one-click cross-machine browser relay flow described by the reporter and confirmed in comments.

## What This Changes

The issue asks OpenClaw to restore or replace the removed Chrome extension browser relay so remote hosted instances can control a user’s local Chrome without a local Node/CLI setup or remote CDP flags.

## Best Possible Solution

Keep this issue open until maintainers choose a deliberate path: either an opt-in hardened extension relay owned by the browser plugin, or an additive public browser profile-driver/transport API that lets a ClawHub/plugin implementation wire a relay without private internals. The chosen path should include schema/docs/tests plus an explicit threat model for remote browser control.
