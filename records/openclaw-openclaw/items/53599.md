## Summary

Current main still does not restore the reported cross-machine Chrome extension relay workflow. The browser plugin keeps the extension relay lifecycle as an intentional no-op, the public browser profile contract excludes `driver: "extension"`, profile creation rejects that driver, doctor migrates old relay config to Chrome MCP `existing-session`, and docs point remote users to node host proxy or raw remote CDP instead of a one-click Chrome extension relay. The issue is recent, concrete, confirmed by other users, and includes useful external prior art in the ClawHosters relay, so it should stay open for maintainer product/security judgment.

## Best Possible Solution

Keep this issue open for maintainer product and security judgment. The likely paths are either an explicitly opt-in, hardened extension relay in the browser plugin with schema/docs/tests/security review, or a narrow public browser profile-driver/transport API so the linked ClawHosters relay or a ClawHub plugin can integrate cleanly without private browser internals.

## Risks / Open Questions

- Restoring this would reintroduce a remote browser-control surface and needs explicit maintainer security review for authentication, origin validation, token handling, consent, public exposure, and operator threat model.
- Closing as ClawHub-only would be premature because current main does not provide a documented way for a third-party plugin to add an `extension` browser profile transport to the bundled browser tool; a full replacement plugin is not the same product path.
- The current node-host, Chrome MCP, and raw remote CDP paths cover some remote-browser cases, but they do not solve the reported one-click Chrome extension install flow for non-technical users on managed remote OpenClaw instances.
