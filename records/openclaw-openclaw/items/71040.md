## Summary

Keep this PR open. Current main still has the cron active-job asymmetry the PR targets: scheduled due runs and direct executeJob runs update the active-job set, but startup catch-up and manual run paths still execute without marking the job active, while task maintenance continues to treat that in-memory active signal as the live cron ownership check before falling back to terminal durable-history recovery.

## What This Changes

The PR adds active-job mark/clear instrumentation around startup catch-up and manual cron runs and adds a symmetry regression test for those execution paths.

## Best Possible Solution

Land a rebased version of the narrow active-job symmetry fix for startup catch-up and manual cron runs, keeping durable restart/recovery policy separate under the broader cron reconciliation work.

## Reproduction Assessment

Yes. Current main gives a high-confidence static reproduction: a startup catch-up or manual cron run can create a runtime='cron' task with lastEventAt pinned to startedAt, run longer than the 5-minute grace window, and still have isCronJobActive(jobId) return false because those producer paths never mark active.

## Solution Assessment

Yes. Mirroring markCronJobActive/clearCronJobActive around the two missing producer paths is the narrowest maintainable fix for the live in-process ownership signal; durable terminal recovery and restart recovery remain separate concerns.
