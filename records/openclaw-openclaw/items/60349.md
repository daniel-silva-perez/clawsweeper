## Summary

Keep open. Current main does not implement `OPENCLAW_SQLITE_MMAP`, `storage.sqliteMmap`, or any `PRAGMA mmap_size` handling at the SQLite open sites, and the PR contains a focused production NFS/SIGBUS report. The PR should be revised rather than closed because review comments identified real correctness and contract gaps, including the WAL `-shm` limitation and generated schema drift.

## Best Possible Solution

Keep this PR open as a plausible fix candidate, but require a revision that implements a coherent SQLite storage policy across current WAL-backed stores, documents or addresses the WAL `-shm` limitation, updates config types/schema/generated artifacts and public docs, keeps plugin SDK boundary/export files aligned, and adds focused tests for env precedence, config validation, and the applied PRAGMAs.

## Risks / Open Questions

- Closing this PR would drop a concrete production crash report while main still lacks any equivalent mitigation.
- `PRAGMA mmap_size=0` alone likely does not solve the stated WAL `-shm` mmap crash mode on NFS; the fix needs a clearer storage policy, exclusive-locking option, or explicit limitation.
- Adding a root `storage.sqliteMmap` config key requires schema/generated-doc alignment and tests so CLI schema lookup and Control UI consumers do not drift.
- The current PR should audit all current SQLite/WAL open sites, including proxy capture, rather than assuming the original four sites are exhaustive.
