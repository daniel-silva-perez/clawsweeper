## Summary

Keep open, but do not merge as-is. Current main does not already provide the requested SQLite mmap toggle, so the reported NFS crash mode remains actionable; however, the PR's mitigation is incomplete because `PRAGMA mmap_size = 0` does not remove WAL shared-memory mapping, and the new config key also lacks generated schema artifacts. ## What This Changes

The PR adds an env/config switch for SQLite memory-mapped I/O, exposes a new `openclaw/plugin-sdk/infra-sqlite` helper, and calls the helper from task and memory SQLite connection sites.

## Best Possible Solution

Rework the change as an explicit SQLite storage compatibility mode in the central WAL helper, with clear semantics for NFS-backed deployments. If the goal is to avoid WAL shared-memory mmap, apply the needed locking or journal-mode policy before first WAL access, document the concurrency tradeoff, and regenerate/test the public config and SDK contract artifacts.

## Reproduction Assessment

Yes for the PR defect: the diff applies only `PRAGMA mmap_size = 0`, while current and PR-touched connection sites use WAL. Upstream SQLite docs show WAL shared memory is avoided only with exclusive locking before first WAL access, so the proposed control can be verified as incomplete without a live NFS mount.

## Solution Assessment

No. The requested direction is useful, but the PR's current approach is not the narrowest safe solution because it exposes a NFS-oriented knob that does not fully disable the WAL shared-memory path; the safer fix belongs in the shared SQLite/WAL setup with generated schema and contract coverage.
