## Summary

Keep open. Current main still matches the reported Paperclip failure: Gateway `agent` params are validated against a closed schema with no root-level `paperclip` metadata field, and the linked fix PR is still open rather than merged. Duplicate reports have already been closed in favor of this issue plus the dedicated PR, so this should remain the canonical tracker until an accepted protocol/schema fix lands.

## What This Changes

Paperclip's `openclaw_gateway` adapter needs its root-level `paperclip` run metadata accepted by OpenClaw Gateway `agent` params or moved to a documented supported metadata path.

## Best Possible Solution

Land one additive Gateway protocol compatibility fix for adapter metadata in `agent` params, with regression coverage and any needed protocol docs, then close this issue and consolidate the remaining duplicate Paperclip reports. A generic adapter metadata namespace is the more extensible endpoint unless maintainers explicitly choose the Paperclip-specific shim for compatibility.

## Reproduction Assessment

Yes. A minimal `agent` request containing `message`, `sessionKey`, `idempotencyKey`, and a root-level `paperclip` object follows the reported payload and will fail current `validateAgentParams` because the schema is closed and omits that field.

## Solution Assessment

Unclear. The linked PR's Paperclip-specific field is a narrow fix for the observable break, but a generic adapter metadata namespace may be the better long-term protocol surface for third-party adapters.
