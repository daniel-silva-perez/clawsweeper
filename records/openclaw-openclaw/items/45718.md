## Summary

Keep open. Current main still serializes the bulky session-entry fields through the normal session-store write path, and open PR #48425 already uses closing syntax for this issue, so the issue should remain open until that PR or a scoped replacement lands.

## What This Changes

The issue reports unbounded `sessions.json` growth because `skillsSnapshot` and `systemPromptReport` are persisted across repeated agent runs and stale-session rollover.

## Best Possible Solution

Review and land #48425, or replace it with an equivalent scoped serializer that strips only transient `SessionEntry.skillsSnapshot` from disk, keeps disk-budget sizing aligned with the exact persisted shape, and deliberately defines how much `systemPromptReport` must remain persisted for warning dedupe.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: populate a session entry with `skillsSnapshot` or `systemPromptReport` and save it through `saveSessionStore`; current main serializes the full entry into `sessions.json`.

## Solution Assessment

Unclear. A shared session-store serialization helper is the narrowest maintainable boundary for `skillsSnapshot`, but dropping all `systemPromptReport` conflicts with the existing warning-dedupe persistence test, so that part needs maintainer judgment.
