## Summary

Keep open. Current main still serializes the full session store, still has `skillsSnapshot` and `systemPromptReport` as session-entry fields, and the linked implementation PR remains open/unmerged. The skills path now has version/filter checks, but there is no merged serialization filter that removes the large transient data from `sessions.json`.

## What This Changes

The issue requests preventing `sessions.json` bloat by avoiding redundant persistence of `skillsSnapshot` and `systemPromptReport` across agent runs.

## Best Possible Solution

Review and either land #48425 or replace it with an equivalent narrow fix that centralizes session-store disk serialization, applies the same filter to disk-budget measurement, adds focused save/load tests, and explicitly decides whether `systemPromptReport` should be preserved in full or reduced to the warning-dedupe state it needs.

## Reproduction Assessment

Yes. A high-confidence source reproduction path exists: run an agent session that produces `systemPromptReport` and/or rebuilds skills, then inspect `sessions.json`; current main’s save path writes the whole `SessionEntry`, including these fields.

## Solution Assessment

Unclear. The linked PR’s shared serializer direction is maintainable for `skillsSnapshot`, but preserving full `systemPromptReport` may only partially solve the issue unless maintainers confirm that full report retention is intentional.
