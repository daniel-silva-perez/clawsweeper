## Summary

Keep open. Current main still serializes the full session store, still keeps both reported fields on `SessionEntry`, and an open implementation PR already uses closing syntax for this issue, so closing now would race the active review path.

## What This Changes

The issue requests preventing `sessions.json` bloat by avoiding redundant persisted `skillsSnapshot` and `systemPromptReport` data across agent runs.

## Best Possible Solution

Land #48425 or an equivalent narrow replacement that centralizes session-store disk serialization, aligns disk-budget measurement with the write path, and explicitly preserves only the `systemPromptReport` state needed for warning dedupe if the full report is not meant to live in `sessions.json`.

## Reproduction Assessment

Yes. A high-confidence source reproduction path exists: run an agent path that populates `skillsSnapshot` or `systemPromptReport`, then inspect `sessions.json`; current main’s save path writes the entire `SessionEntry` object, including those fields.

## Solution Assessment

Unclear. The linked PR’s shared serialization-filter direction looks like the right narrow boundary for `skillsSnapshot`, but maintainers still need to decide whether full `systemPromptReport` persistence is intentional or should be reduced to dedupe metadata.
