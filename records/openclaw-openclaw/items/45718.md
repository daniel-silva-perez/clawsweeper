## Summary

Keep open. Current main still persists the reported bulky session fields through the full session-store write path, and an open implementation PR already uses closing syntax for this issue, so the issue should remain open until that PR or a narrow replacement is resolved.

## What This Changes

The issue requests reducing `sessions.json` bloat caused by persisted `skillsSnapshot` and `systemPromptReport` data across agent runs.

## Best Possible Solution

Land #48425 after review, or replace it with an equivalent scoped serializer that strips only top-level transient `SessionEntry.skillsSnapshot`, keeps disk-budget measurement aligned with the exact write shape, and deliberately defines the persisted `systemPromptReport` subset needed for restart-safe warning dedupe.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: populate a session entry with `skillsSnapshot` or `systemPromptReport` and save it through `saveSessionStore`; current main serializes the full entry into `sessions.json`.

## Solution Assessment

Unclear. A shared session-store serialization helper is the right narrow boundary for `skillsSnapshot`, but the best `systemPromptReport` treatment depends on maintainer judgment because current tests preserve it for warning dedupe.
