## Summary

Keep open. Current main still lacks a plugin scaffold command, so the PR is not obsolete, but the patch should not merge as-is because its new validator rejects scoped plugin ids that current install code accepts and covers with tests; the branch is also currently unmergeable against its recorded base.

## What This Changes

This PR adds an `openclaw plugins init [directory]` CLI subcommand that scaffolds `openclaw.plugin.json`, `index.ts`, and `package.json`, with options, validation, overwrite handling, and unit tests.

## Best Possible Solution

Land a rebased version of this feature that shares or exactly mirrors `src/plugins/install-paths.ts` plugin-id validation, adds a scoped-id regression test for `@scope/name`, updates the plugin CLI and authoring docs, and aligns the generated template with the current documented plugin entry and activation conventions.

## Reproduction Assessment

Yes. The PR head can be checked statically: `src/cli/plugins-init.ts` rejects any `/` in an id, while current main's `resolvePluginInstallDir("@scope/name", ...)` test proves scoped ids are accepted elsewhere.

## Solution Assessment

No, not as proposed. A plugin init command is a reasonable narrow DX addition, but the maintainable implementation should reuse the existing install-path validator and ship docs/template updates with the new public CLI surface.
