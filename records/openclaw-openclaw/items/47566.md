## Summary

Keep open. Current main still has no `openclaw plugins init` command, and the PR is a plausible additive DX feature, but the proposed implementation is not merge-ready because its new validator rejects scoped plugin ids that the existing install path accepts and tests cover; the branch is also currently dirty against main.

## What This Changes

This PR adds an `openclaw plugins init [directory]` CLI subcommand that scaffolds `openclaw.plugin.json`, `index.ts`, and `package.json` with options, validation, overwrite handling, and unit tests.

## Best Possible Solution

Land a rebased version that reuses or directly shares `src/plugins/install-paths.ts` plugin-id validation, adds a scoped-id regression test for `@scope/name`, and updates the plugin CLI and authoring docs so the generated template follows current `definePluginEntry` and activation guidance.

## Reproduction Assessment

Yes. The issue is reproducible by static inspection: the PR rejects any `/` in `src/cli/plugins-init.ts`, while current main accepts and tests `@scope/name` through `resolvePluginInstallDir`.

## Solution Assessment

No. The feature direction is reasonable, but the maintainable solution should share the existing plugin install validator instead of introducing a stricter, divergent CLI-only validator.
