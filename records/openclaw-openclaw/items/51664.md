## Summary

Keep this issue open. Current main and the latest release still wire the WebChat code-block Copy button directly to `navigator.clipboard.writeText` and ignore failures, with no Edge/clipboard fallback and no regression test for rejected clipboard writes. The related fallback PRs #51746 and #52010 were closed unmerged, so they are useful context but not an implemented fix.

## Best Possible Solution

Keep the issue open and implement a shared Control UI clipboard helper for WebChat copy actions. The helper should try `navigator.clipboard.writeText`, handle missing or rejected Clipboard API calls, fall back to a browser-supported copy path such as a temporary textarea plus `document.execCommand('copy')` where appropriate, and expose success/failure state. Wire code-block Copy through it and add focused UI tests for rejected Clipboard API calls and fallback behavior; then do a manual Edge-on-Windows smoke check.
