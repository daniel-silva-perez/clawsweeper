## Summary

Keep open. Current main partially addresses the report by threading gateway runtime snapshots into health refreshes, so fresh health snapshots can inherit Telegram running/lastStartAt/mode. The tokenSource mismatch is still present in the source path: health builds the Telegram summary from a runtime-only snapshot, the runtime snapshot does not carry tokenSource, and the shared token summary defaults a missing tokenSource to "none". channels.status still uses the fuller per-account snapshot builder, which can include credential source metadata from the resolved Telegram account. There is also an open external implementation candidate with closing syntax, so this issue should stay open until that fix is merged, superseded, or explicitly closed by maintainers.

## Best Possible Solution

Keep this issue open. The best fix is to make health build channel account summaries through the same runtime-plus-resolved-account snapshot contract used by channels.status, or otherwise carry safe credential-source metadata such as Telegram tokenSource into the health snapshot. Then add a focused regression test that compares health and channels.status for a configured running Telegram account, including running, lastStartAt, mode, and tokenSource, and resolve or supersede the open implementation candidate.
