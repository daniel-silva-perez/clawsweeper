## Summary

Keep this issue open. Current main still routes bundled LiteLLM through `openai-completions`, cache-retention resolution still requires `anthropic-messages` for non-Bedrock custom providers, and Anthropic `cache_control` payload injection is still OpenRouter-gated. Related PRs either closed unmerged or only fixed the custom `anthropic-messages` path, so the reported default LiteLLM route remains unimplemented.

## Best Possible Solution

Implement a narrow LiteLLM Anthropic prompt-cache pass-through: when explicit `cacheRetention`/`cacheControlTtl` is configured for a bundled LiteLLM `openai-completions` Claude/Anthropic-looking model and retention is not `none`, inject Anthropic `cache_control` markers into the OpenAI-compatible payload. Keep the behavior opt-in, avoid broadening OpenRouter/custom-proxy behavior, add targeted regression tests, and update the LiteLLM docs/changelog.

## Clownfish Work Prompt

Fix the valid bug from this issue: explicit `agents.defaults.models["litellm/<model>"].params.cacheRetention` / legacy `cacheControlTtl` is ignored on the bundled LiteLLM `openai-completions` route for Anthropic/Claude-backed models, so no Anthropic `cache_control` markers reach LiteLLM. Implement a conservative opt-in fix in the LiteLLM/provider-wrapper path or shared proxy wrapper helpers. Activate only for provider `litellm`, API `openai-completions`, explicit cache config, retention other than `none`, and Claude/Anthropic-looking LiteLLM model IDs such as `claude-*`, `anthropic/claude*`, or `anthropic.claude*`. Reuse `applyAnthropicEphemeralCacheControlMarkers`/payload patch helpers where possible. Do not default caching for LiteLLM, do not send markers to non-Claude LiteLLM models, do not broaden OpenRouter custom-proxy behavior, and do not try to solve OpenRouter cacheRetention follow-up #42961 in this PR. Add targeted regression tests for configured `long`/`short`, `none`, no config, non-Anthropic LiteLLM models, and existing OpenRouter behavior. Update `docs/providers/litellm.md` to replace the current no-prompt-cache-hints note if support is added, and add a user-facing changelog fix entry with reporter/prior-contributor attribution as appropriate.
