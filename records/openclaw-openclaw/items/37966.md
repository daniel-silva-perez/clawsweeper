## Summary

Current main still does not honor explicit `cacheRetention` for the bundled LiteLLM `openai-completions` route. LiteLLM still defaults to the OpenAI-compatible proxy path, the cache-retention resolver only accepts explicit custom-provider caching on `anthropic-messages`, and Anthropic `cache_control` payload injection remains OpenRouter-gated. The issue is still valid and is a narrow fix-PR candidate; the previously linked PR #38221 is closed unmerged and only partially covered the needed payload-level behavior.

## Best Possible Solution

Keep this issue open and implement a narrow LiteLLM Anthropic prompt-cache pass-through: when explicit `cacheRetention` or `cacheControlTtl` is configured for a bundled LiteLLM `openai-completions` Claude/Anthropic-looking model and retention is not `none`, inject Anthropic `cache_control` markers into the OpenAI-compatible payload. Keep it opt-in, do not broaden OpenRouter custom-proxy behavior, add regression tests for the bundled LiteLLM route and negative cases, and update LiteLLM docs/changelog.
