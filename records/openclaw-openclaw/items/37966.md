## Summary

Current main still does not honor explicit cacheRetention for the bundled LiteLLM OpenAI-compatible Anthropic route. The prior PR #38221 is closed unmerged and only addressed resolver pass-through, while current main still lacks LiteLLM payload-level Anthropic cache_control injection for the default openai-completions path.

## Best Possible Solution

Keep this issue open and implement a narrow LiteLLM Anthropic prompt-cache pass-through: when a bundled LiteLLM openai-completions Claude/Anthropic-looking model has explicit cacheRetention or cacheControlTtl and retention is not none, inject Anthropic cache_control markers into system/developer prompt blocks before the LiteLLM proxy request. Preserve existing OpenRouter gating, add LiteLLM positive and negative regression tests, and update LiteLLM prompt-caching docs plus the changelog.

## Risks / Open Questions

- A fix should stay opt-in and scoped to LiteLLM Anthropic-looking OpenAI-compatible routes; injecting Anthropic cache_control into arbitrary OpenAI-compatible proxies could break non-Anthropic backends.
- LiteLLM model IDs can be local aliases like claude-opus-4-6 rather than anthropic/... refs, so model-family detection needs tests for bundled defaults and negative cases.
- If cacheRetention: "long" is expected to map to a 1-hour Anthropic TTL through LiteLLM, the current ephemeral-only helper may need a small typed extension rather than reusing OpenRouter’s short-cache wrapper unchanged.
