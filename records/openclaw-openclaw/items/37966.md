## Summary

Current main still does not honor explicit `cacheRetention` for the bundled LiteLLM `openai-completions` route. LiteLLM defaults to an OpenAI-compatible proxy route, cache-retention eligibility still excludes that route, and Anthropic `cache_control` payload injection remains OpenRouter-gated. The issue should stay open and is a good focused fix-PR candidate.

## Best Possible Solution

Keep this issue open and implement a narrow LiteLLM Anthropic prompt-cache pass-through: when explicit `cacheRetention` or `cacheControlTtl` is configured for a bundled LiteLLM `openai-completions` Claude/Anthropic-looking model and retention is not `none`, inject Anthropic `cache_control` markers into the OpenAI-compatible payload. Keep it opt-in, do not broaden OpenRouter custom-proxy behavior, add regression tests for the bundled LiteLLM route, and update LiteLLM docs/changelog.
