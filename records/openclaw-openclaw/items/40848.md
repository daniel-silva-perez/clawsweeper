## Summary

Keep this issue open. Current main still does not expose Telegram Bot API message `entities` or `caption_entities` through the shared `message` tool, Telegram plugin schema contribution, CLI send flags, Telegram action runtime, or outbound `sendMessageTelegram` path, so custom emoji delivery via `custom_emoji` entities remains an actionable bundled Telegram plugin gap.

## Best Possible Solution

Keep this open and implement it in the bundled Telegram plugin. Add Telegram-owned `entities` and likely `caption_entities` schema fields for the message tool, validate a narrow Telegram `MessageEntity` shape, thread the values through `extensions/telegram/src/action-runtime.ts` into `TelegramSendOpts`, and send `entities`/`caption_entities` instead of HTML `parse_mode` on compatible text and caption payloads. Add focused regression coverage for schema exposure, runtime forwarding, and Bot API params for a `custom_emoji` entity.

## Evidence

  - command: `web.run search_query site:core.telegram.org/bots/api sendMessage entities MessageEntity custom_emoji custom_emoji_id`

## Risks / Open Questions

- Implementation needs Telegram-specific validation for `MessageEntity`, especially UTF-16 offsets/lengths and `custom_emoji_id`.
- The send path should define precedence between explicit `entities`/`caption_entities` and the existing HTML `parse_mode` fallback, including chunked text, media captions, follow-up text after caption splitting, and parse-error retry behavior.
