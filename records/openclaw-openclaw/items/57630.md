## Summary

Keep open. Current main partially covers this PR by supporting `appendContext`, but it still does not implement the requested `prompt` replacement field in the public hook result contract, merge path, prompt resolver, embedded runner, CLI runner, docs, or focused tests. The PR discussion is useful: the author addressed the review concerns about priority ordering and empty-string prompt replacement, so the remaining maintainer path is a rebase/successor patch rather than cleanup closure.

## Best Possible Solution

Keep this PR open and ask for a rebase or successor patch against current main. The concrete path is to add `prompt?: string` to `PluginHookBeforePromptBuildResult` and `PLUGIN_PROMPT_MUTATION_RESULT_FIELDS`, merge it with higher-priority-wins semantics, propagate it through `resolvePromptBuildHookResult`, apply it before `prependContext`/`appendContext` in both embedded and CLI prompt assembly, update current English docs, and add focused regression tests for replacement behavior and safety-sensitive empty-string redaction.
