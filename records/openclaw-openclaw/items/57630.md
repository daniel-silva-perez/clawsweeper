## Summary

Keep open. Current main already has the `appendContext` half of the request, but it still does not expose or consume a `prompt` replacement field, and the proposed patch is stale against the current hook type/docs layout and misses important runner behavior.

## What This Changes

The PR extends plugin prompt-build hook result types, merge logic, embedded prompt assembly, and docs with `appendContext` and full user-prompt replacement via `prompt`.

## Best Possible Solution

Land a rebased or successor patch that adds `prompt?: string` to the current public hook result type and mutation field list, merges it with higher-priority-wins semantics, propagates it through the shared prompt resolver, applies it consistently in embedded and CLI prompt assembly before prepend/append wrapping, preserves bootstrap truncation warnings, updates current English docs only, and adds focused tests for replacement, empty-string redaction, legacy hook compatibility, CLI parity, and bootstrap-warning ordering.

## Reproduction Assessment

Yes. Source inspection on current main is enough to reproduce the missing capability: a plugin returning `{ prompt: "redacted" }` has no public result type field, no merge/resolver path, and no embedded or CLI consumption path; a focused regression test could assert that `before_prompt_build` replacement changes the final prompt in both runners.

## Solution Assessment

No. The requested API direction is plausible and motivated by the GuardClaw/EdgeClaw redaction use case in the PR body, but this branch is not the best mergeable form because it only applies replacement in the embedded runner, can drop bootstrap truncation warnings, and targets outdated files.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Apply prompt replacement in the CLI runner:** `src/agents/pi-embedded-runner/run/attempt.prompt-helpers.ts:75`
  - body: Adding `prompt` to the shared prompt-build result makes it a public hook contract, but the patch only consumes it in `runEmbeddedAttempt`. CLI-backed agents still build from the original `params.prompt` and then apply only prepend/append context, so a redaction plugin can leave the unredacted user text on CLI paths. Consume `hookResult.prompt` in CLI preparation before wrapping it with prepend/append context.
  - confidence: 0.9
- **[P2] Preserve bootstrap warnings after prompt replacement:** `src/agents/pi-embedded-runner/run/attempt.ts:1410-1411`
  - body: This replacement runs after `effectivePrompt` has already been annotated with bootstrap truncation warnings. When a hook returns `prompt`, the warning block is discarded, so the model loses the signal that injected workspace context is partial. Apply replacement before warning injection or re-append the warning after replacement.
  - confidence: 0.87
