## Summary

Keep open. Current main still has the reported telemetry gap: the reply guard suppresses the note only when `successfulCronAdds` is nonzero or an enabled same-session cron already exists, while `successfulCronAdds` is still incremented only for the native `cron` tool path. A successful `openclaw cron add --at ...` executed through `exec`/`bash` reaches Gateway `cron.add` through the CLI but does not update that counter, so the misleading note remains possible; the direct fix PR #53024 is closed unmerged and #47589 is adjacent regex work rather than this cron-success path.

## What This Changes

The issue requests that successful one-time cron reminders scheduled through `openclaw cron add --at` stop receiving the visible `I did not schedule a reminder` note.

## Best Possible Solution

Implement a narrow scheduling-signal fix so successful current-turn reminders are recognized consistently across the native cron tool and the assistant shell/exec CLI path, with regression tests for successful, failed, and unrelated shell commands plus the final reply behavior. Keep broader note stripping, regex tightening, opt-out config, and plugin-SDK scheduling signals separate.

## Reproduction Assessment

Yes. A high-confidence code reproduction is visible: a successful `exec`/`bash` command that runs `openclaw cron add --at ...` can create a cron job through Gateway `cron.add`, but the embedded-run counter remains zero, so a reminder-like final reply still satisfies the guard-note append condition.

## Solution Assessment

No, current main is not the best solution for this issue because it only recognizes native `cron` tool success. The narrow maintainable fix is to record successful current-turn cron scheduling for the CLI-through-exec path without changing unrelated reminder regex or Telegram delivery behavior.
