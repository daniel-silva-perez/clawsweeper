## Summary

Current main still has the reported failure mode: the visible reminder note is appended whenever the reply contains a reminder commitment and `successfulCronAdds` is zero, while the only current counter increment is for the native `cron` tool path. A successful `openclaw cron add --at ...` run through exec/bash can still create the job through the Gateway CLI path without updating that run-local counter. The direct fix PR (#53024) is closed and unmerged, so this issue remains actionable.

## Best Possible Solution

Keep this issue open. Implement #53024 or an equivalent current-main fix that treats successful `openclaw cron add --at ...` executions through exec/bash as successful reminder scheduling for the current turn, or ensure assistant-created reminders consistently use the native cron tool. Add regression coverage for successful exec/bash cron-add suppression, failed exec/bash cron-add still warning, and the Telegram delivery path not receiving the false note after successful scheduling.

## Risks / Open Questions

- The same-session cron-store fallback can mask some reproductions when an enabled matching job already exists, so regression tests should isolate a newly created exec/bash one-shot reminder with no prior session cron job.
- A broad suppression based only on command text could hide genuine unsupported reminder promises; a fix should be tied to successful CLI scheduling telemetry or should steer reminder creation through the native cron tool.
- Related external-scheduler work such as #71170 does not directly solve this issue because this report is about OpenClaw’s own CLI cron add path invoked via exec/bash.
