## Summary

Keep open. Current main still leaves the reported shell/Telegram cron-add path possible: the reply guard suppresses the note only when `successfulCronAdds` is nonzero or a same-session cron already exists, while the counter is still incremented only for the native `cron` tool path. The direct fix PR #53024 is closed unmerged, and the related open reminder-guard work addresses adjacent false positives rather than this exact successful `openclaw cron add --at` telemetry gap.

## What This Changes

The issue asks OpenClaw to stop appending the unscheduled-reminder note after a reminder was successfully created via `openclaw cron add --at`.

## Best Possible Solution

Implement a narrow scheduling-signal fix so successful current-turn reminders are recognized consistently across the native cron tool and the assistant shell/exec CLI path, with regression tests for successful, failed, and unrelated shell commands plus the final reply behavior. Keep broader note stripping, opt-out config, regex tightening, and plugin-SDK scheduling signals as separate maintainer-reviewed work.
