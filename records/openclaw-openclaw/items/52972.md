## Summary

Keep open. Current main still leaves the reported failure mode in place: the reply guard appends the visible unscheduled-reminder note when `successfulCronAdds` is zero and no same-session cron job is found, while the run-local counter is still updated only for the native `cron` tool path. A successful `openclaw cron add --at ...` executed through `exec` or `bash` can create a real Gateway cron job without setting that counter, and the direct fix PR #53024 is closed unmerged.

## What This Changes

The issue asks OpenClaw to stop appending the “I did not schedule a reminder” note after a successful one-time reminder is scheduled via `openclaw cron add --at`, especially from Telegram.

## Best Possible Solution

Track successful current-turn reminder scheduling consistently across the native `cron` tool and the CLI-through-exec/bash scheduling path, or route assistant-created reminders through the native cron tool, with regression coverage for successful exec/bash scheduling, failed exec/bash scheduling, unrelated exec commands, and the final user-visible reply.

## Repair Work Prompt

Fix the reminder-guard false note for successful `openclaw cron add --at ...` scheduling performed through `exec`/`bash`. Current main only increments `successfulCronAdds` for the native `cron` tool path in `src/agents/pi-embedded-subscribe.handlers.tools.ts`, while `openclaw cron add` creates jobs via Gateway `cron.add` from `src/cli/cron-cli/register.cron-add.ts`. Implement a narrow current-turn success signal for the CLI exec/bash path, or otherwise route assistant-created reminders through the native cron tool, so the final reply guard in `src/auto-reply/reply/agent-runner.ts` does not append `UNSCHEDULED_REMINDER_NOTE` after a real reminder was scheduled. Cover at least: successful exec/bash `openclaw cron add` increments/suppresses; failed exec/bash cron add does not; unrelated exec commands do not; native `cron` tool behavior remains unchanged; final reply text is unchanged when scheduling succeeded. Use PR #53024 as useful prior art but re-evaluate the command detection carefully so echoed text or failed commands are not counted. Do not strip the note globally, do not add a broad public config opt-out for this narrow bug, and do not combine the separate memory false-positive regex work from #47586/#47589 unless required. Add a user-facing changelog fix entry with the reporter/contributor credit expected by repo policy.
