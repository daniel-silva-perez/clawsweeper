## Summary

Keep open. Current main still has the reported failure mode: the reminder guard suppresses the note only when `successfulCronAdds` is nonzero or an existing same-session cron job is found, while `successfulCronAdds` is still incremented only for the native `cron` tool path. A successful `openclaw cron add --at ...` invoked through exec/bash can create the job through the CLI/Gateway RPC path without updating that run-local counter, and the direct fix PR #53024 is closed and unmerged.

## What This Changes

The issue requests that OpenClaw stop appending the “I did not schedule a reminder” note after a reminder is successfully created with `openclaw cron add --at`, reported from a Telegram session.

## Best Possible Solution

Track successful current-turn reminder scheduling consistently across the native `cron` tool and OpenClaw CLI exec/bash scheduling path, or route assistant-created reminders through the native cron tool, with regression coverage for successful exec/bash scheduling, failed exec/bash scheduling, and Telegram-visible delivery.

Treat #53024 as useful source work, but adapt it to current main. Count only successful exec/bash scheduling attempts; do not suppress the note for failed commands, approval-pending/unavailable commands, unrelated exec commands, or broad reminder-like text without a real scheduling signal. Prefer a robust helper around the existing exec tool start args/result shape rather than scattering regex logic. Do not change cron storage semantics, Telegram routing, or the same-session cron fallback policy except as needed for this current-turn success signal. Add regression coverage in `src/agents/pi-embedded-subscribe.handlers.tools.test.ts` for successful exec/bash `openclaw cron add`, unrelated exec, and failed exec cron add; add or extend auto-reply coverage if needed to prove the note is suppressed after the successful signal.
