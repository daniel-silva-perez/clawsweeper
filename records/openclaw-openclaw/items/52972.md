## Summary

Keep open. Current main still has the reported failure mode: reminder-note suppression depends on `successfulCronAdds` or an existing same-session cron job, while `successfulCronAdds` is still incremented only for the native `cron` tool path; a successful `openclaw cron add --at ...` run through exec/bash can create the job without updating that run-local signal, and the direct fix PR #53024 is closed unmerged.

## What This Changes

The issue asks OpenClaw to stop appending the “I did not schedule a reminder” note after a reminder is successfully created with `openclaw cron add --at`, reported from a Telegram session.

## Best Possible Solution

Track successful current-turn reminder scheduling consistently across the native `cron` tool and the OpenClaw CLI exec/bash scheduling path, or steer assistant-created reminders through the native cron tool, with regression coverage for successful exec/bash scheduling, failed exec/bash scheduling, unrelated exec commands, and Telegram-visible delivery.

## Repair Work Prompt

Fix the current-main reminder guard bug tracked by this issue: when an embedded agent successfully schedules a reminder by running `openclaw cron add --at ...` through exec/bash, the user-visible reply must not get the `Note: I did not schedule a reminder in this turn...` suffix. Start from the current `successfulCronAdds` flow in `src/agents/pi-embedded-subscribe.handlers.tools.ts` and `src/auto-reply/reply/agent-runner.ts`. Treat #53024 as useful source work, but adapt it to current main. Count only successful exec/bash scheduling attempts; do not suppress the note for failed commands, approval-pending/unavailable commands, unrelated exec commands, or broad reminder-like text without a real scheduling signal. Prefer a robust helper around the existing exec tool start args/result shape rather than scattering regex logic. Do not change cron storage semantics, Telegram routing, or the same-session cron fallback policy except as needed for this current-turn success signal. Add regression coverage in `src/agents/pi-embedded-subscribe.handlers.tools.test.ts` for successful exec/bash `openclaw cron add`, unrelated exec, and failed exec cron add; add or extend auto-reply coverage if needed to prove the note is suppressed after the successful signal. Add a single-line user-facing changelog fix entry in the active version with `Thanks @vitobotta`; credit @mvanhorn too if reusing #53024 materially.
