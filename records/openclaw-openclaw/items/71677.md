## Summary

Current main still lacks the Feishu instant callback-response path requested here: `card.action.trigger` is registered and parsed, but the handler falls through without returning a card or toast payload, while the action implementation remains a `Promise<void>` path that awaits command dispatch. The related #71670 Schema 2.0 parsing fix is present, but it does not supersede this UI-latency enhancement.

## What This Changes

Add Feishu WebSocket callback-response support so interactive-card button clicks can immediately return a transitional disabled-card/toast response instead of waiting for agent processing.

## Best Possible Solution

Implement this as a Feishu-plugin-owned, typed, backward-compatible instant callback-response path: preserve validation and replay dedupe, return a valid transitional disabled-card/toast response for eligible structured buttons immediately, dispatch the synthetic command asynchronously, and document any new `ocf1` disable-state field if added.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to invoke the registered `card.action.trigger` handler with a valid structured quick-action payload and observe that it resolves without returning a callback payload while the action function remains `Promise<void>`.

## Solution Assessment

Yes. The best direction is the requested plugin-local callback-response path, using the Feishu/Lark SDK response contract; a core API change is not indicated unless implementation proves a missing generic plugin seam.
