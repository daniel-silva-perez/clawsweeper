## Summary

Keep open. Current main still treats Feishu card-action callbacks as a void path: the handler returns no callback payload, the action implementation awaits synthetic dispatch for quick/legacy paths, and there is no typed disable-state or transitional card response contract yet. Adjacent PR #71787 is useful Feishu card-action work, but it closes #71670 and explicitly does not change outbound card schema or routing, so it does not supersede this instant-update request.

## What This Changes

Request Feishu `card.action.trigger` handlers return an immediate disabled-button/toast callback response while continuing command dispatch asynchronously.

## Best Possible Solution

Implement this inside the bundled Feishu plugin: add a typed opt-in disable-state or transitional-response contract to the structured card interaction envelope, return a valid immediate Feishu callback response from the `card.action.trigger` handler, continue command dispatch asynchronously, and cover WebSocket/webhook callbacks plus quick-action, approval-card, and duplicate-token behavior. Keep the core extension-agnostic unless a generic plugin API gap is proven.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: invoke the registered `card.action.trigger` path for a valid quick-action card and the handler resolves `undefined` after `handleFeishuCardAction`, so the SDK has no result to encode into the WebSocket callback response.

## Solution Assessment

Yes, the requested direction is the right owner boundary, but the implementation should be a validated Feishu-plugin contract rather than an ad hoc return object. The safest fix is to preserve current dedupe and permission behavior while adding a typed transitional callback response path with focused tests.
