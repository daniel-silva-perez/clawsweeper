## Summary

Keep open. Current main still routes Feishu `card.action.trigger` through a handler that resolves without returning a callback payload, while the quick/confirm action path is typed as `Promise<void>` and awaits synthetic command dispatch. The related Schema 2.0 parsing work addresses malformed callback acceptance, not the instant callback response requested here.

## What This Changes

Support instant Feishu interactive-card callback responses that immediately transition or disable clicked buttons while the synthetic command continues asynchronously.

## Best Possible Solution

Implement a Feishu-plugin-owned instant callback response path for eligible card actions: validate and claim the token first, return a typed transitional card/toast through the SDK response path, dispatch the synthetic command asynchronously, and document any backward-compatible `ocf1` disable-state field.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. A static reproduction is to invoke the registered `card.action.trigger` handler with a valid structured quick-action event and observe that it resolves without a callback response while `handleFeishuCardAction` awaits the synthetic message path.

## Solution Assessment

Is this the best way to solve the issue? Yes, if the change stays inside the Feishu plugin and uses the existing SDK callback-response return path. The safer implementation is opt-in typed transitional rendering that preserves malformed-payload rejection, replay-token dedupe, and command authorization boundaries.
