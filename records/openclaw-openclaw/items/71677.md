## Summary

Keep open. Current main still routes Feishu `card.action.trigger` callbacks through a void handler and a `Promise<void>` action path, so the SDK has no card/toast payload to return immediately; the already-shipped #71670 work fixed Schema 2.0 parsing only and does not supersede this instant-update feature.

## What This Changes

Support immediate Feishu card-action callback responses that disable or transition clicked card buttons while the agent command continues asynchronously.

## Best Possible Solution

Implement a plugin-owned, typed instant callback response path for Feishu card actions: preserve current validation and replay dedupe, return a valid transitional card/toast response immediately for eligible structured buttons, continue synthetic command dispatch asynchronously, and keep any new `ocf1` disable-state contract backward-compatible and documented.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. A static reproduction on current main is to invoke the registered `card.action.trigger` handler with a valid structured quick-action event and observe that it resolves without returning any callback payload while `handleFeishuCardAction` awaits the synthetic message path.

## Solution Assessment

Is this the best way to solve the issue? Yes, if it stays inside the Feishu plugin and uses the Feishu callback-response contract rather than a core migration. The safer implementation is a typed opt-in transitional response that preserves token dedupe, malformed-payload rejection, and command authorization behavior.
