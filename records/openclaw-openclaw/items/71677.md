## Summary

Keep open. Current main accepts and routes Feishu card-action callbacks, including the adjacent Schema 2.0 parsing path, but it still does not return an immediate callback payload for a disabled/loading card state; the open related PR is parser-focused and does not supersede this UX request.

## What This Changes

Add Feishu/Lark `card.action.trigger` instant callback responses so Schema 2.0 card buttons can enter a disabled or transitional state immediately instead of waiting for agent processing.

## Best Possible Solution

Implement a Feishu-plugin-owned instant callback response for eligible structured quick and approval-confirm actions: validate and claim the callback token first, launch the synthetic command path in the background with token completion in its `finally` path, and immediately return a typed disabled/loading card or toast through the existing SDK return path. If an `ocf1` disable-state field is added, make it backward-compatible, documented, and covered by Feishu tests without moving Feishu-specific behavior into core.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to invoke the registered `card.action.trigger` handler with a valid structured quick-action event and observe that it resolves without any callback response while `handleFeishuCardAction` awaits the synthetic command path.

## Solution Assessment

Yes. The issue's proposed handler-return path matches the locked Lark SDK behavior; the safest implementation is a narrow Feishu plugin change that preserves callback validation, replay-token dedupe, wrong-user/wrong-conversation checks, stale checks, and command authorization.
