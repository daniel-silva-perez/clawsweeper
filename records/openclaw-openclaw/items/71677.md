## Summary

Keep open. Current main still parses Feishu card-action callbacks and deduplicates repeated clicks, but it does not return an immediate WebSocket callback payload, so the requested instant disabled/transitional card UX remains unimplemented.

## What This Changes

Add Feishu card.action.trigger support for returning an immediate transitional disabled-button card or toast while command processing continues asynchronously.

## Best Possible Solution

Implement this inside the Feishu plugin: validate and claim the callback token first, return a typed transitional disabled/loading card or toast through the existing SDK return path, continue command dispatch asynchronously, and complete or release the token in the background path without moving Feishu-specific behavior into core.

## Reproduction Assessment

Yes. A static reproduction is to invoke the registered card.action.trigger handler with a valid structured quick-action event and observe that it resolves without any callback response while handleFeishuCardAction awaits the synthetic command path.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix if it stays Feishu-plugin-owned and preserves token dedupe, wrong-user/wrong-conversation checks, stale checks, and command authorization. A backward-compatible ocf1 disable-state field is reasonable if the renderer and decoder validate it explicitly.

## Repair Work Prompt

Fix Feishu card.action.trigger instant feedback for structured quick and approval-confirm card buttons. Keep the change inside the Feishu plugin. The handler should validate/parse the callback, claim the callback token before any response, return a Feishu-compatible transitional disabled/loading card or toast through the existing SDK return path, and run the synthetic command dispatch asynchronously with token completion/release handled in the background. Preserve wrong-user, wrong-conversation, stale, malformed, duplicate-token, and command authorization behavior. If adding an ocf1 disable-state field such as ds, make it backward-compatible, typed, validated, rendered only by Feishu card helpers, and covered by tests. Do not move Feishu-specific behavior into core. Update docs/changelog only if a user-facing config or card contract changes.
