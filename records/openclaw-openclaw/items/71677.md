## Summary

Keep open. Current main still accepts Feishu card-action callbacks but does not return an immediate callback payload, so the requested instant disabled/transitional card state remains unimplemented. The existing Schema 2.0 parser fixes and duplicate-token dedupe are useful adjacent work, but they do not solve the 3-5 second no-feedback window described here.

## What This Changes

The issue requests instant Feishu card callback responses so interactive card buttons can show disabled or transitional feedback immediately after a click.

## Best Possible Solution

Implement a Feishu-plugin-owned instant callback response path for eligible structured quick and approval-confirm actions: validate and claim the callback token first, return a typed transitional disabled/loading card or toast through the existing SDK return path, keep the command dispatch running asynchronously with token completion in its background finally path, and document any backward-compatible ocf1 disable-state field if added.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. A static reproduction is to invoke the registered card.action.trigger handler with a valid structured quick-action event and observe that it resolves without a callback response while handleFeishuCardAction awaits the synthetic command path.

## Solution Assessment

Is this the best way to solve the issue? Yes, if the change stays inside the Feishu plugin and preserves callback validation, wrong-user/wrong-conversation checks, stale checks, replay-token dedupe, and command authorization. The maintainable version is an opt-in typed transitional response rather than moving Feishu-specific behavior into core.
