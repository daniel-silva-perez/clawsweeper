## Summary

Keep open. Current main still routes Feishu `card.action.trigger` through a handler that resolves without returning a callback response, while the quick-action path is typed as `Promise<void>` and awaits synthetic command dispatch; the already closed #71670 parsing fix does not implement instant card updates.

## What This Changes

Add instant Feishu interactive-card callback responses so button clicks can immediately show disabled/transitional UI while OpenClaw continues command processing asynchronously.

## Best Possible Solution

Implement this inside the Feishu plugin: keep the existing callback validation, replay-token dedupe, and command authorization path, return a valid immediate Feishu callback response for eligible card actions, and dispatch the synthetic command asynchronously without adding core-specific Feishu behavior.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to invoke the registered `card.action.trigger` handler on current main with a valid structured quick-action event and observe that it resolves without returning a callback payload while `handleFeishuCardAction` awaits the synthetic message path.

## Solution Assessment

Yes, the requested direction is the narrowest maintainable shape if it stays Feishu-plugin-owned and uses the existing SDK callback-response plumbing. The safer implementation is typed and opt-in/configurable, preserving malformed-payload rejection, token dedupe, and command authorization boundaries.
