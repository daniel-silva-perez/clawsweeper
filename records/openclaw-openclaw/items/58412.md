## Summary

Current main already resolves the underlying Kimi Coding `anthropic-messages` stream-order failure by routing those runs through OpenClaw's per-request Anthropic SSE transport instead of shared Anthropic SDK `MessageStream` state. The PR is now obsolete and should not be merged as-is because its subscribe-layer workaround has blocking ordering hazards.

## What This Changes

The PR adds subscribe-session state to synthesize a `message_end` when an assistant `message_start` arrives before the previous end event, filters later stale end events, resets that state during compaction retry, and adds focused tests.

## Best Possible Solution

Close this PR and keep the shipped transport-boundary fix. Fresh Kimi Coding regressions should be reopened as new reports against the current Anthropic transport path rather than merged through this subscribe-layer workaround.

## Reproduction Assessment

Yes. The original failure path is documented in #57753/#58358 as Kimi Coding via `anthropic-messages` on v2026.3.28 hitting the Anthropic SDK `MessageStream` event-order error; current main has source and unit-test coverage for the missing-`message_stop` SSE shape through OpenClaw's own transport.

## Solution Assessment

No for this PR, yes for current main. Handling the symptom after malformed assistant events reach `subscribeEmbeddedPiSession` is less maintainable and introduces ordering bugs; removing the shared SDK stream state at the Anthropic transport boundary is the narrower durable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Await the synthetic close before resetting state:** `src/agents/pi-embedded-subscribe.handlers.messages.ts:223-225`
  - body: `handleMessageEnd()` can return a promise when message-end block flushing is async. This path fires it with `void` and immediately resets for the next message, so the synthetic close can finalize after the new message starts and clear the new message's buffers or `inAssistantMessage` state mid-stream.
  - confidence: 0.9
- **[P1] Correlate stale ends without text-only matching:** `src/agents/pi-embedded-subscribe.handlers.messages.ts:500-504`
  - body: The stale-end filter records only normalized visible text and later drops the first real `message_end` with the same text. If overlapping messages share text, or the newer message ends before the delayed stale one, this can skip the valid current end and process the stale one instead, losing or reordering output.
  - confidence: 0.86
