## Summary

Keep this PR open. Current main does not implement the KimiCodingPlan malformed-stream workaround: assistant `message_start` still unconditionally resets the current assistant-message buffers without tracking whether a prior assistant message was open, and there is no `inAssistantMessage` or stale synthetic-close state on main. The PR remains a relevant implementation candidate for open bug #58358, but it needs rework before merge because the current patch shown in the provided review context has unresolved ordering/state risks around asynchronous synthetic `message_end` handling and stale-end correlation.

## Best Possible Solution

Keep the PR open as a focused candidate for bug #58358, but ask the author or maintainer to rework the implementation so malformed Kimi/Anthropic event ordering is handled in the stream subscription/event chain without asynchronous state races. A good fix should add regression coverage for `message_start` before prior `message_end`, delayed stale real `message_end`, same-text overlapping messages, non-text/media-only messages, and async `onBlockReplyFlush`/`message_end` flushing. If the fix belongs in the Kimi provider wrapper instead of generic agent subscription, it should still preserve the core stream contract and avoid provider-specific ids in core unless the seam is generic.

## Risks / Open Questions
