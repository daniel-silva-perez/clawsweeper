## Summary

Keep this issue open. Current main still has only the existing public queue modes and drains queued followups via debounce/quiet-time; the internal subagent announce queue has useful busy-parent deferral prior art, but it is not an operator-selectable `messages.queue` mode for ordinary chat/status completion delivery.

## What This Changes

The issue requests a public readiness-aware queue collection mode that buffers status/completion messages until the session is ready, then flushes or coalesces them with clear starvation protection.

## Best Possible Solution

Keep the issue open until OpenClaw has a documented public queue option that waits on a generic session/run readiness signal, coalesces ordinary queued notices only when delivery is safe, and defines a clear fallback so buffered updates cannot starve indefinitely.
