## Summary

Keep this issue open. The report is fresh, concrete, and still lacks current-main live proof on Raspberry Pi arm64; the only Discord change since the prior review is slash-command deploy rate-limit logging, which is adjacent startup noise handling rather than evidence that the reported gateway CPU wedge is fixed.

## What This Changes

The issue reports that enabling the Discord channel on Raspberry Pi arm64 with OpenClaw 2026.4.26 wedges the gateway at about 100% CPU, while disabling Discord restores gateway health.

## Best Possible Solution

Validate current main or the next build on the reported Raspberry Pi arm64 setup with a real Discord bot token. If it still wedges, keep the fix inside the Discord plugin startup/lifecycle/voice surface, add targeted regression coverage for the narrowed failure mode, and document channels.discord.voice.enabled=false only as a confirmed workaround if live testing proves it avoids the spin.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the reported 2026.4.26 release because the issue gives a concrete Pi arm64, Node, Discord-token startup path and a Discord-disabled recovery check; no for current main because that live hardware and credential path was not exercised here.

## Solution Assessment

Is this the best way to solve the issue? Unclear. The existing lifecycle guards and new slash-deploy rate-limit handling are in adjacent startup areas but are not proven fixes for the Pi CPU wedge; the safer path is live validation first, then a narrow Discord plugin fix if the wedge remains.
