## Summary

Keep this issue open. The report is fresh, concrete, and still describes a plausible Discord startup/runtime regression on Raspberry Pi arm64; current main has adjacent Discord lifecycle guards and plugin runtime-dependency fixes, but there is no source, test, or live Pi proof that the reported 100% CPU gateway wedge is fixed.

## What This Changes

Investigate a regression where enabling the Discord provider on Raspberry Pi arm64 causes the gateway to spin near 100% CPU and makes gateway/channel status probes time out after startup.

## Best Possible Solution

Validate current main or the next build on the reported Raspberry Pi arm64 setup with Discord enabled and a real bot token. If the wedge still occurs, keep the fix in the narrow owner surface indicated by the live evidence, add a targeted regression test for that failure mode, and document any confirmed workaround such as disabling Discord voice only after it is proven to change the CPU/probe behavior.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the reported 2026.4.26 path because the issue gives specific platform, token, startup, probe, and Discord-disabled recovery steps; no for current main because this review did not exercise the real Raspberry Pi arm64 plus Discord-token runtime.

## Solution Assessment

Is this the best way to solve the issue? Unclear because there is no proposed patch and the current adjacent fixes are not proven against the observed CPU wedge. The safest path is live validation first, then a narrow Discord plugin or plugin-runtime-deps fix based on the confirmed failure path.
