## Summary

Keep this issue open. The report is fresh, concrete, and Discord-specific; current main has substantial related gateway changes, including removal of the Carbon dependency and internal reconnect/heartbeat handling, but there is still no direct proof that the Raspberry Pi arm64 100% CPU startup wedge is fixed with a real Discord bot token.

## What This Changes

Investigate a Discord startup regression where enabling the Discord channel on Raspberry Pi arm64 wedges the gateway near 100% CPU and makes gateway status/probe calls time out.

## Best Possible Solution

Validate current main or the next candidate build on Raspberry Pi arm64 with a real Discord bot token. If the wedge persists, keep the fix inside the bundled Discord plugin startup/gateway/voice lifecycle, add focused lifecycle tests, and document a voice-disabled workaround only if live validation proves it avoids the spin.

## Reproduction Assessment

Yes, the report gives a concrete reproduction path with release, host architecture, Node version, Discord token setup, CPU/status checks, and a Discord-disabled control case. Current-main reproduction is still unverified because this read-only review did not have the required Raspberry Pi and bot-token environment.

## Solution Assessment

Unclear. Current main contains the narrowest plausible direction by replacing Carbon gateway behavior with owned internal gateway/timer handling, but the best fix cannot be confirmed until the reported hardware and default Discord configuration are live-validated.
