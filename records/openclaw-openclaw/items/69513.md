## Summary

Keep open. Current main still validates cron announce channel selection but does not validate explicit primary or failure-destination account IDs against configured channel accounts before persistence, and there is open candidate PR context for the same bug.

## What This Changes

The issue requests creation-time validation so `openclaw cron create --announce --account <id>` rejects unknown channel account IDs instead of storing them.

## Best Possible Solution

Land one focused gateway-side validation change that rejects unknown explicit announce account IDs for both `cron.add` and `cron.update`, including failure destinations, backed by regression tests and a user-facing changelog entry; then close this issue as implemented.

## Reproduction Assessment

Yes. The issue gives concrete CLI steps, and current-main source inspection shows the same path still forwards `--account`, shape-checks it as a string, and reaches persistence after channel-only validation.

## Solution Assessment

Yes. Fail-fast validation in the gateway cron add/update handlers is the narrowest maintainable boundary because it covers CLI and RPC callers before persistence while leaving delivery execution and implicit account resolution unchanged.
