## Summary

Keep open. Current main still validates cron announce delivery channel selection but does not validate explicit delivery.accountId or delivery.failureDestination.accountId against configured channel accounts before cron.add/cron.update persistence. A closed unmerged PR (#69518) and an open paired PR in iamlukethedev/openclaw#2 both confirm the intended fix boundary, so the best maintainer path is to review/port/land or close that candidate work rather than closing this issue now.

## Best Possible Solution

Keep this issue open and route it to the existing candidate fix work: validate explicit announce delivery.accountId and delivery.failureDestination.accountId in the gateway cron.add/cron.update validation path, using the configured account list/resolution semantics for the resolved channel, with focused tests in src/gateway/server.cron.test.ts. Maintainers should review, port, land, or close the open paired PR before queueing duplicate work.
