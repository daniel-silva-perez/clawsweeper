## Summary

Keep this issue open. Current main still has the reported LiteLLM/OpenAI MCP schema path exposed: bundled MCP tools keep raw MCP `inputSchema` values, those tools are appended after provider schema normalization, and the LiteLLM provider still has no OpenAI-family `normalizeToolSchemas` hook. The existing native OpenAI strict-compat helpers do not cover the LiteLLM proxy route and the non-strict OpenAI transport path still does not recursively collapse nested array-valued `type` fields such as `["string", "null"]`.

## Best Possible Solution

Keep this issue open. The maintainer path is to review #66415 or a superseding PR, but require coverage that feeds an MS365-like bundled MCP schema with nested array-valued `type` fields through the LiteLLM/OpenAI completions request serialization path. The fix likely needs post-materialization provider-family normalization or transport-level schema rewriting plus diagnostics that identify the offending tool/schema path without changing non-OpenAI behavior.
