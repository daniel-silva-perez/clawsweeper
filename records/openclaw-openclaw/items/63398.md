## Summary

Keep this issue open. Current main still exposes the reported LiteLLM-to-OpenAI MCP schema path: bundled MCP tools keep raw MCP `inputSchema`, those tools are appended after the provider normalization pass, LiteLLM registers no tool-schema normalization hook, and the OpenAI compat helpers are limited to native OpenAI/Codex routes rather than LiteLLM proxy routing. The related PR #66415 is still an open implementation candidate in the provided context, not a shipped fix.

## Best Possible Solution

Keep this issue open and review #66415 or a superseding PR, but require coverage that sends an MS365-like bundled MCP schema through the LiteLLM/OpenAI completions serialization path. The likely fix is post-materialization provider-family normalization or transport-level schema rewriting, plus diagnostics that identify the offending tool/schema path without changing non-OpenAI LiteLLM behavior.
