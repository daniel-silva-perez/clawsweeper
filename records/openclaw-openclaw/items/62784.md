## Summary

Keep this issue open. Current main has partial Slack Socket Mode mitigations, including a 15s pong timeout, configurable ping/pong options, disabled Slack SDK auto-reconnect, and OpenClaw-managed reconnect backoff, but it still starts all configured Slack accounts in the same channel startup wave and has no Slack-specific startup staggering or bounded account-start concurrency.

## What This Changes

The issue reports that 8 or more concurrent Slack Socket Mode accounts can create synchronized pong timeouts, reconnect storms, and gateway termination, and asks OpenClaw to survive this with staggered startup, safer reconnect behavior, or connection pooling.

## Best Possible Solution

Add a narrow Slack Socket Mode startup staggering or per-account concurrency limit that spreads initial WebSocket opens for multi-account Slack configs while preserving the existing 15s timeout tuning, OpenClaw-owned reconnect backoff, and fail-fast handling for non-recoverable auth errors.

## Reproduction Assessment

Yes for the missing implementation path: current source deterministically starts all Slack accounts for the channel via Promise.all, and each account opens its own Slack monitor. The exact SIGKILL outcome still needs live Slack credentials or a focused simulated multi-account startup test.

## Solution Assessment

Yes for a narrow Slack-owned startup jitter or account-start throttling fix. Connection pooling is broader product work, and reconnect backoff already exists for post-disconnect recovery, so the remaining maintainable fix is to avoid the synchronized initial Socket Mode connection burst.
