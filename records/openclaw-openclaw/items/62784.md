## Summary

Keep this issue open. Current main has useful partial mitigations for Slack Socket Mode pong timeouts: a 15s default client ping timeout, account-level timeout config, Slack SDK auto-reconnect disabled, and OpenClaw-managed reconnect backoff with jitter. But the central multi-account startup problem is still not implemented: gateway startup still fans out all accounts for a channel with `Promise.all`, and I found no Slack-specific startup staggering or bounded account-start concurrency.

## Best Possible Solution

Keep the issue open and implement a Slack-owned or generic account-start throttling path that staggers Socket Mode startup for multiple Slack accounts without weakening existing timeout tuning or OpenClaw-managed reconnect backoff. The likely patch should add a focused regression test with several configured Slack accounts proving account monitor startup is staggered or concurrency-limited, plus docs/config only if the delay becomes user-tunable.

## Risks / Open Questions

- I did not run a live 8+ Slack-app reproduction; the keep-open decision rests on current source showing the requested startup staggering remains absent, while existing code only covers timeout tuning and reconnect backoff after startup/disconnect events.
- Current timeout and reconnect changes may reduce the observed crash frequency on newer releases, so the eventual fix should first reproduce or simulate the synchronized multi-account startup burst before adding a broad gateway-level policy.
