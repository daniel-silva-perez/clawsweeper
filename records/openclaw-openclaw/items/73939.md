## Summary

Keep this open. Current main still rejects direct custom top-level fields under skill entries, and the related implementation PR was closed unmerged; the remaining question is whether maintainers want to loosen the public config contract or keep the documented nested `config` shape.

## What This Changes

The issue requests that custom fields such as `api_endpoint` and `user_api_key` be accepted directly under `skills.entries.<skill>` during config patch and Gateway validation.

## Best Possible Solution

Align the public skill-entry contract end to end: either intentionally support direct custom fields with schema, generated schema, types, tests, docs, and redaction metadata updated together, or keep the stricter contract and answer the report by documenting `skills.entries.<skill>.config` plus the plugin-owned Tavily path.

## Reproduction Assessment

Yes. The current-main schema/test path reproduces the validation rejection: `OpenClawSchema.safeParse` is expected to fail for an unknown top-level `skills.entries.custom-skill.url`, and `config.patch` runs the same validation path before writing config.

## Solution Assessment

Unclear. Accepting arbitrary direct fields would address the reported shape, but it conflicts with current docs/tests and needs a conscious secret-redaction and typo-detection contract; the safest fix depends on maintainer product judgment.
