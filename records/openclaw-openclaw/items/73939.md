## Summary

Keep this issue open. Current main still rejects unknown direct fields under `skills.entries.<skill>`, the latest release tag in this checkout has the same behavior, and the related implementation PR #74024 was closed unmerged. The remaining question is a public config-contract decision, not a cleanup-safe close.

## What This Changes

The issue requests that custom per-skill fields such as `api_endpoint` and `user_api_key` be accepted under `skills.entries.<skill-name>` instead of being rejected during config validation and reload.

## Best Possible Solution

Make an explicit maintainer decision on the skill-entry config contract. If direct custom fields are accepted, update the Zod schema, generated schema, config tests, docs, and sensitive-field metadata together; if the stricter contract stays, keep custom data under `skills.entries.<skill>.config` and route Tavily users to `plugins.entries.tavily.config.webSearch.*`.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists through `OpenClawSchema.safeParse` with an unknown direct field under `skills.entries.custom-skill`; current tests encode that rejection, and `config.patch` uses the same validation path before writing.

## Solution Assessment

Unclear. Allowing arbitrary top-level keys, as #74024 attempted, would solve the reported shape but changes the public config contract; the safer path needs maintainer confirmation plus schema, docs, tests, and secret-handling updates.
