## Summary

Keep this issue open. Current main no longer appears to have the exact write-then-runtime-rollback mismatch described: `config.patch` validates the merged config before writing, and both runtime and generated schemas reject unknown direct `skills.entries.<skill>` fields. However, the requested direct custom fields such as `api_endpoint` and `user_api_key` under `skills.entries.<skill>` are still not implemented; current main only supports custom per-skill payloads under `skills.entries.<skill>.config`, while Tavily is plugin-owned under `plugins.entries.tavily.config.webSearch.*`.

## Best Possible Solution

Have a maintainer decide the skills config contract explicitly. The conservative path is to keep direct `skills.entries.<skill>` fields strict, document `skills.entries.<skill>.config` consistently, and route Tavily users to `plugins.entries.tavily.config.webSearch.*`. If direct custom fields should be supported for compatibility, add a deliberate schema/runtime/UI/redaction design rather than broad arbitrary top-level acceptance.
