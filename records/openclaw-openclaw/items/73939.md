## Summary

Keep this issue open. Current main still rejects custom top-level fields under `skills.entries.<skill>`, the linked implementation PR #74024 is closed unmerged, and the current docs intentionally route custom per-skill data to nested `config` while Tavily uses plugin-owned config. The remaining decision is whether OpenClaw should loosen the public skill-entry config contract or keep the stricter shape and improve guidance.

## What This Changes

The issue asks OpenClaw to accept and persist skill-specific custom fields directly under `skills.entries.<skill-name>` during config validation and Gateway reload.

## Best Possible Solution

Make an explicit config-contract decision. If direct custom skill-entry fields are accepted, update `SkillEntrySchema`, generated schema output, focused config tests, docs, and secret redaction metadata together; if the stricter contract is preferred, keep custom data under `skills.entries.<skill>.config` and route Tavily users to `plugins.entries.tavily.config.webSearch.*`.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists through `OpenClawSchema.safeParse` with an unknown direct field under `skills.entries.custom-skill`; the current test suite encodes that rejection, and `config.patch` invokes the same validation before writing.

## Solution Assessment

Unclear. Accepting arbitrary top-level keys would solve the reported custom-skill shape, but it changes the public config contract; the safer maintainable path needs maintainer confirmation plus schema, docs, tests, and secret-handling updates.
