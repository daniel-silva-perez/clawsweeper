## Summary

Keep this issue open. Current main still rejects unknown direct fields under `skills.entries.<skill>`, the docs/tests now encode nested `config` as the custom-data location, and the linked implementation PR #74024 is closed unmerged, so this needs an explicit maintainer config-contract decision rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to let user-defined skill entry settings such as `api_endpoint` and `user_api_key` survive `config.patch`/reload instead of being rejected by runtime config validation.

## Best Possible Solution

Make the skill-entry config contract explicit. If direct custom fields are part of the public API, update `SkillEntrySchema`, generated schema, config tests, docs, and secret-redaction behavior together; otherwise keep custom data under `skills.entries.<skill>.config` and improve guidance/error messaging for users coming from older examples or external skills.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists by parsing a config with an unknown direct field under `skills.entries.custom-skill`; the current test suite encodes that rejection, and `config.patch` uses the same validation path before writing.

## Solution Assessment

Unclear. Allowing arbitrary top-level fields would solve the reported custom-shape failure, but it broadens the public config contract and secret-handling surface; the safer path is maintainer approval of that contract before an automated fix PR changes schema behavior.
