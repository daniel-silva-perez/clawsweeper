## Summary

Keep this issue open. Current main still rejects unknown top-level fields under `skills.entries.<skill>` through the strict runtime schema and generated JSON schema, while the docs route custom per-skill data to nested `config`; the linked implementation PR #74024 is closed unmerged, so the remaining question is an explicit config-contract decision.

## What This Changes

Allow custom per-skill fields such as `api_endpoint` and `user_api_key` directly under `skills.entries.<skill-name>` to persist through Gateway config validation and reload.

## Best Possible Solution

Make an explicit maintainer decision for the skills config contract. If direct top-level custom fields are allowed, update the Zod schema, generated schema, config types, focused validation tests, docs, and sensitive-field handling together; if the stricter contract is intentional, keep custom skill data under `skills.entries.<skill>.config` and route Tavily users to the plugin config path.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: `OpenClawSchema.safeParse` rejects an unknown direct field under `skills.entries.custom-skill`, the current test asserts that rejection, and Gateway `config.patch` invokes the same validation path.

## Solution Assessment

Unclear. Allowing top-level custom fields would satisfy the report, but it changes the public config shape; the safer maintainable path needs maintainer confirmation before automation changes schema behavior.
