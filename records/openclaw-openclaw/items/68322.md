## Summary

Keep open. Current main still fetches the Kilocode model catalog without auth, and this active PR is useful, but the proposed patch only forwards ambient process environment credentials; the provider catalog flow already has a context credential resolver for auth profiles and discovery keys, so the PR should be adjusted or explicitly accepted as a Kiloclaw-only env-scope fix before merge.

## What This Changes

The PR builds Kilocode model-discovery request headers from trimmed `KILOCODE_API_KEY` and `KILOCODE_ORGANIZATION_ID`, adds fetch-header tests, and updates the raw-fetch allowlist line.

## Best Possible Solution

Use the plugin catalog context to pass the resolved Kilocode discovery credential into model discovery, while keeping the env-header behavior and tests, then land after confirming the Kilo Gateway header contract and running the targeted plugin test plus changed-file gate.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.83

Full review comments:

- **[P2] Use the catalog credential resolver for discovery auth:** `extensions/kilocode/provider-models.ts:131-133`
  - body: The new auth header reads only `process.env`, but model discovery is normally called through `ProviderCatalogContext`, where config and auth-profile credentials are exposed as `discoveryApiKey` without necessarily mutating `process.env`. With a Kilocode key stored in an auth profile or threaded discovery env, this still fetches the catalog unauthenticated and BYOK models can remain hidden. Please pass the resolved discovery key from the catalog context into `discoverKilocodeModels` and cover that path in tests.
  - confidence: 0.82
