## Summary

Keep open. Current main has the session-store cache/load portion of the structuredClone memory issue fixed, but the auth-profile cache, runtime snapshot, and external OAuth overlay paths still clone through structuredClone, and maintainer discussion explicitly kept the item open for that remaining work plus benchmark/soak confirmation.

## What This Changes

The issue requests replacing structuredClone-based cloning in session-store and auth-profile hot paths with JSON-shaped cloning or serialized-store reuse to reduce native-memory growth and Gateway OOM risk.

## Best Possible Solution

Finish or explicitly defer the auth-profile clone cleanup in the auth-profile owner area, preserve JSON-shaped credential semantics, add mutation-isolation/no-structuredClone coverage for persisted stores and runtime OAuth snapshots, then validate with the reported large-store benchmark or a production-shaped Gateway soak. Broader Gateway memory accumulation should stay on narrower trackers such as #54155, #51264, and #63998.

## Reproduction Assessment

Yes for the reported class of issue: the discussion includes production OOM evidence plus a 500 sessions x 10KB x 2000 clone-cycle benchmark, and current source inspection confirms remaining auth-profile structuredClone paths. I did not rerun the benchmark in this read-only review.

Yes, keeping this open is the best current resolution. Current main is only a partial fix, and the remaining credential-sensitive auth-profile work plus the open implementation candidate need maintainer triage and benchmark/soak confirmation before closure.
