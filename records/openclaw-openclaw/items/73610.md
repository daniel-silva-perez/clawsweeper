## Summary

Keep open. Current main still gates Telegram polling startup on a successful `deleteWebhook` cleanup and sends recoverable cleanup failures through setup backoff/retry before the grammY runner starts. The later transport-timeout fallback that closed #73255 may mitigate some route-timeout cases, but it does not implement this issue's requested empty-webhook continuation; open PR #47384 is still the narrow implementation candidate for that behavior.

## Best Possible Solution

Keep this issue open and use it to drive the Telegram plugin fix. The best path is to review/rebase #47384 or land an equivalent owner-module change in `extensions/telegram/src/polling-session.ts`: after a recoverable `deleteWebhook` cleanup failure, call `getWebhookInfo`, mark cleanup satisfied and start polling only when the returned webhook URL is empty, preserve retries for active webhooks or recoverable confirmation failures, and add/update the Telegram regression tests and docs if a user-facing option is introduced.
