## Summary

Current main still has the reported channel-visible recovery gap. Discord persists enough route metadata to know where the interrupted turn came from, and restart-sentinel delivery can post routed notices, but ordinary restart-aborted main-session recovery still resumes internally with `deliver: false` or marks sessions failed without posting a same-channel recovery message. Related open issue #38836 shows the same class of problem for Feishu, so this should stay open for a generic fix rather than a Discord-only cleanup.

## What This Changes

Request to automatically resume or send a visible same-channel recovery notice when a gateway restart aborts an in-flight Discord-backed turn.

## Best Possible Solution

Implement a generic post-restart channel-visible interruption recovery path: detect recently abandoned channel-originated main sessions after startup, resolve the persisted delivery context or last route, send a same-channel recovery notice when safe, and only auto-resume when the transcript tail is explicitly resumable. Keep #38836 in scope as the same generic bug class.
