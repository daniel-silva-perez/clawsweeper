## Summary

Keep this issue open. Current main has partial restart hardening, but it still does not guarantee a Discord-visible recovery message when an in-flight Discord turn is interrupted by an unplanned gateway restart. The implemented recovery path is session/internal-first and the planned restart-sentinel path only runs when a sentinel file was written before restart.

## Best Possible Solution

Keep the issue open and implement a shared post-restart interruption recovery path: detect abandoned channel-originated running/aborted sessions on startup, use the persisted delivery context/last route to post a visible recovery notice to Discord, and only auto-resume when the transcript tail is proven safe. Reuse the existing restart-sentinel outbound delivery machinery where possible, and keep related Feishu recovery in view so the core seam is not Discord-only.
