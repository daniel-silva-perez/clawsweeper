## Summary

Keep open. Current main still has the reported Discord-visible recovery gap: Discord records enough route metadata for the originating channel, but the abort path returns without posting any recovery notice; main-session restart recovery only resumes internally with `deliver: false`; and the restart-sentinel notice path only runs when a sentinel file exists. Related open issue #38836 reports the same class of user-visible restart loss for Feishu, so the right fix should be generic rather than Discord-only.

## Best Possible Solution

Implement a generic post-restart, channel-visible interruption recovery path. On startup, detect recently abandoned channel-originated running or restart-aborted main sessions, resolve the persisted delivery context or last route, post a same-channel recovery notice when safe, and only auto-resume when the transcript tail is explicitly resumable. Reuse the restart-sentinel outbound machinery where practical and keep the related Feishu report #38836 in scope.
