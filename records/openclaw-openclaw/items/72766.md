## Summary

Keep this PR open. Current main still has the heartbeat stale-snapshot writes the PR targets, and the proposed change is a narrow reuse of the existing session-store update helper with no blocking review finding found in the diff.

## What This Changes

The PR replaces two heartbeat session-store load/mutate/save blocks in `src/infra/heartbeat-runner.ts` with `updateSessionStore` for task timestamp and heartbeat dedupe writes.

## Best Possible Solution

Land a rebased version of this focused helper-based fix, preferably with one regression test that interleaves a heartbeat store write with a session metadata update and proves metadata survives; then treat #41254 as superseded for the dedupe-only slice.

## Reproduction Assessment

Yes. The code-level reproduction is to have heartbeat read a session-store snapshot, let a concurrent inbound update modify the same session through the locked store path, then let heartbeat save its stale object and observe fields such as `messageCount`, `lastMessageAt`, route metadata, or `updatedAt` being lost.

## Solution Assessment

Yes. Reusing `updateSessionStore` is the narrowest maintainable fix because it applies the existing lock and re-read contract to the two unsafe heartbeat writes without introducing a new concurrency mechanism.
