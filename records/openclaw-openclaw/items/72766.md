## Summary

Keep this PR open. Current main still has the reported stale load-mutate-save pattern in both heartbeat task timestamp persistence and heartbeat dedupe persistence, while the PR is a narrow replacement with the existing lock-and-reread helper; I did not find a blocking correctness or security issue in the patch.

## What This Changes

The PR changes `src/infra/heartbeat-runner.ts` so heartbeat task timestamp updates and last-delivered heartbeat dedupe fields are written through `updateSessionStore` instead of separate `loadSessionStore` and `saveSessionStore` calls.

## Best Possible Solution

Land a rebased version of this narrow helper-based fix, ideally with one focused regression test that interleaves a heartbeat store write with session metadata mutation and proves metadata survives; coordinate the overlapping dedupe concern with #41254 rather than carrying two divergent fixes.

## Reproduction Assessment

Yes. The code-level reproduction is to let heartbeat hold or use an older loaded session-store object, perform a concurrent metadata update through the locked store path, then let heartbeat save its stale object and observe fields such as `messageCount`, `lastMessageAt`, or route metadata being lost.

## Solution Assessment

Yes. Reusing `updateSessionStore` is the narrowest maintainable fix because it applies the existing session-store lock and re-read contract at the two heartbeat writes without adding a new concurrency mechanism.
