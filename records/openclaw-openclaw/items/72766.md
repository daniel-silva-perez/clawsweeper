## Summary

Keep this PR open. Current `main` still has the two heartbeat load/mutate/save session-store writes, while the PR’s helper-based change matches the existing locked `updateSessionStore` contract and I found no blocking review or security findings.

## What This Changes

The PR changes `src/infra/heartbeat-runner.ts` so heartbeat task timestamp updates and delivered-heartbeat dedupe fields are written through `updateSessionStore` instead of separate `loadSessionStore` and `saveSessionStore` calls.

## Best Possible Solution

Land a rebased version of this narrow helper-based fix, ideally with one focused regression test proving concurrent session metadata survives heartbeat task and dedupe writes; coordinate the overlapping dedupe-only PR #41254 during review.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to let heartbeat save an older loaded session-store object after a concurrent locked metadata update, then observe fields such as `messageCount`, `lastMessageAt`, or route metadata being lost.

## Solution Assessment

Yes. Reusing `updateSessionStore` at the two heartbeat write sites is the narrowest maintainable fix because it applies the existing lock and re-read contract without adding a new concurrency mechanism.
