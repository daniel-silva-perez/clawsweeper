## Summary

Keep open. The PR is aimed at a real gap in current main, but the latest patch is not mergeable as-is: the earlier mixed-message ordering bug was addressed, yet the full-drop path still persists a synthetic toolResult with no preceding assistant tool call, and it ignores the existing synthetic-result opt-out used by provider policy.

## What This Changes

This PR adds dropped-tool-call diagnostics/reporting and changes the session write guard/tests to log dropped assistant tool calls and persist synthetic error toolResult entries when an assistant tool-call turn is fully dropped.

## Best Possible Solution

Keep the structured drop report and observability work, but make unavailable-tool feedback provider-safe: do not persist orphan toolResult entries, respect transcriptPolicy.allowSyntheticToolResults, add replay-level regression coverage, and cap any logged rejected names.

## Reproduction Assessment

Yes. A high-confidence reproduction is an assistant message containing only a malformed or non-allowlisted tool call; the PR persists a standalone toolResult, while current replay repair drops free-floating toolResult entries before model context. A second focused reproduction is the PR's allowSyntheticToolResults=false case, where the new test still expects a synthetic error result.

## Solution Assessment

No. The patch fixes the earlier mixed valid/invalid ordering issue by skipping synthetic errors for partial drops, but the full-drop path is still an orphan toolResult and bypasses the existing synthetic-result policy. Logging/report export is a good direction; model feedback needs a different provider-safe surface.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Avoid persisting orphan synthetic tool results:** `src/agents/session-tool-result-guard.ts:429-446`
  - body: When the sanitized assistant turn is fully dropped, this branch persists only a toolResult for detail.id. There is no preceding assistant tool call for that id; current replay repair drops free-floating toolResult entries before model context, so the advertised feedback will not reach the next model turn and raw history contains an invalid span. Use a provider-safe feedback shape instead of an orphan toolResult.
  - confidence: 0.9
- **[P2] Respect synthetic-result opt-out:** `src/agents/session-tool-result-guard.ts:429-446`
  - body: The new path calls persistToolResult(... isSynthetic: true) without checking allowSyntheticToolResults. In real runs that flag comes from transcriptPolicy.allowSyntheticToolResults, which is false for providers that should not receive synthetic results; the new test even expects a synthetic error with the flag disabled. Gate this path or split the option explicitly.
  - confidence: 0.86
- **[P3] Cap raw tool names before logging:** `src/agents/session-tool-result-guard.ts:416-420`
  - body: droppedToolCallDetails keeps the rejected raw name, and guardedAppend interpolates it into a WARN line. Names rejected for length or invalid tokens are model-controlled text, so a malformed stream can write very long or newline-bearing values to logs. Cap and sanitize the display value before logging or storing details.
  - confidence: 0.78
