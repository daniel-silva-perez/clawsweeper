## Summary

Keep this open. Current main still exposes only static DM/group access policies for BlueBubbles and the shared channel access helper; existing message hooks observe or claim messages after channel access has admitted the turn, so they do not provide the requested pre-allowlist dynamic authorization path. This is also not a clean ClawHub-only close because the missing piece appears to be a generic, security-sensitive channel authorization seam.

## What This Changes

Request dynamic DM and group allowlist authorization for BlueBubbles/iMessage, including resolver or pre-allowlist hook support and optional unknown-sender routing instead of static config edits.

## Best Possible Solution

Define an additive generic channel authorization contract that runs before static allowlist denial, keeps current pairing/static allowlist defaults, fails closed on resolver errors, records denial reasons safely, and then adopt it in BlueBubbles with docs and focused tests.

## Reproduction Assessment

Yes. Current-main source shows that a BlueBubbles sender absent from allowFrom/groupAllowFrom is blocked or put through pairing before reply dispatch, and the config schema has no dynamic policy or resolver field to exercise.

## Solution Assessment

Unclear. The reported HTTP resolver and authorization hook are plausible, but the maintainable fix should first settle the generic channel/plugin authorization API and its security semantics rather than adding a one-off BlueBubbles resolver.
