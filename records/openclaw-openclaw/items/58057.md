## Summary

Keep this open. Current main still has static DM/group access policies, and BlueBubbles drops non-allowlisted senders before routing or message hooks can decide anything. The request is also security-sensitive product/API work, so it needs maintainer design rather than an automated cleanup close.

## What This Changes

Request to add dynamic DM/group identity authorization for allowlists, including a resolver or pre-allowlist hook and optional unknown-sender routing for BlueBubbles/iMessage-style deployments.

## Best Possible Solution

Design a typed pre-admission channel authorization seam, or a deliberately narrow BlueBubbles resolver if maintainers want to scope the first step, with bounded cache/timeout semantics, audit logging, docs for phone/email/chat identifiers, and tests for allow, deny, timeout, and unknown-sender routing behavior.

## Reproduction Assessment

Yes. A high-confidence reproduction exists from current code and tests: BlueBubbles dmPolicy=allowlist with a missing or nonmatching allowFrom accepts the webhook but does not dispatch or create a pairing request, and the config schema has no dynamic policy value.

## Solution Assessment

Unclear because there is no patch. The HTTP resolver and authorization hook are plausible directions, but the best maintainable solution should be chosen as a security/API design with fail-closed behavior rather than added as an ad hoc channel bypass.

- Treating existing group session bindings as authorization could admit groups accidentally unless the binding/auth contract is explicitly designed.
