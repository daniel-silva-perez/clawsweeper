## Summary

Keep this open. Current main still has only static DM/group policies for BlueBubbles and the shared access helper, and the available hook surfaces do not provide the requested pre-allowlist dynamic authorization path. The existing discussion already includes a ClawSweeper keep-open review, and the current d117ed main tree still supports that conclusion.

## What This Changes

The issue requests dynamic DM/group authorization for allowlists, via an HTTP resolver or pre-allowlist authorization hook, plus optional unknown-sender routing for BlueBubbles/iMessage deployments.

## Best Possible Solution

Define an additive, generic pre-admission channel authorization contract with fail-closed resolver semantics, bounded caching, safe denial logging, and backward-compatible defaults, then adopt it in BlueBubbles with schema, docs, and focused access-control tests.

## Reproduction Assessment

Yes. A source-level reproduction is clear: configure BlueBubbles with dmPolicy=allowlist and a sender absent from allowFrom, then the current monitor path returns before dispatch; the existing tests cover that blocked path.

## Solution Assessment

Unclear. The proposed HTTP resolver and authorization hook are plausible, but the best maintainable fix should settle the generic channel/plugin authorization contract before adding a BlueBubbles-specific dynamic resolver.
