## Summary

Keep this issue open. Current main still does not implement `dmPolicy: "dynamic"`, `groupPolicy: "dynamic"`, a pre-allowlist authorization hook, or an unknown-sender routing path. Existing BlueBubbles/iMessage behavior supports static allowlists and DM pairing, but allowlist mode remains static and group authorization remains explicit. This is not a clean ClawHub close because the useful plugin path likely requires a core/plugin authorization seam before a plugin can decide access before channel drops.

## Best Possible Solution

Keep the issue open for maintainer product/API design. A good path would add a typed channel/plugin authorization seam, or a BlueBubbles-owned dynamic identity resolver if maintainers want to scope it narrowly, that runs before DM/group allowlist drops with bounded caching, timeout/error semantics, tests for allow/deny/timeout, docs for phone/email/chat identifiers, and a clear decision on whether group session bindings can satisfy group access.

## Risks / Open Questions

- The requested behavior is security-sensitive inbound authorization; any implementation should fail closed on resolver errors/timeouts, bound cache TTLs, and make audit logging explicit.
- A BlueBubbles-only resolver could solve the reported deployment, but a generic hook/API affects all channels and needs maintainer product judgment to avoid bypassing safe defaults.
