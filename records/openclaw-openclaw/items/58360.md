## Summary

Keep this PR open. Current main has since shipped call-level `model_call_started` / `model_call_ended` plugin hooks, but they are intentionally metadata-only and do not expose the per-call request/response payload surface that this PR proposes with `before_model_call` / `after_model_call`. The PR is not already implemented on main; it likely needs a rebase/redesign against the newer metadata-only hook surface and a security pass for raw conversation payload access.

## Best Possible Solution

Keep the PR open only as a useful implementation candidate, not as already shipped. The best path is to rebase/redesign it on top of current `model_call_started` / `model_call_ended`: either extend those hooks with an explicitly gated raw-payload mode, or add separate raw-payload hooks with `allowConversationAccess` enforcement, public docs, loader tests, and embedded-runner tests proving provider-bound request/response alignment and error/abort behavior.

## Risks / Open Questions

- Security review: the PR intentionally expands plugin data access to raw model-call payloads. Current main gates raw conversation hooks through `CONVERSATION_HOOK_NAMES` and `plugins.entries.<id>.hooks.allowConversationAccess`; any revived version should add the new raw-payload hooks to that policy, docs, and loader tests before merge.
- API drift: current main now has `model_call_started` / `model_call_ended` metadata hooks, so landing the older `before_model_call` / `after_model_call` API as-is may create overlapping hook names and unclear semantics unless maintainers deliberately want both layers.
- Correctness risk from the PR discussion: review comments already identified hard edges around provider-bound payload alignment, retry-unique call IDs, synthetic responses, synchronous errors, and `.result()` early-exit behavior. Several were addressed in the PR thread, but the remaining boundaries need re-evaluation against current main before merge.
