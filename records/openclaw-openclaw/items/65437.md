## Summary

Keep open. Current main still has the Slack room/group early-drop path: Slack command authorization is built from Slack account/channel allowlists only, so a sender listed only in `tools.elevated.allowFrom.slack` can be blocked before elevated permission resolution runs. The related Discord and tiered-auth items are useful context, but they do not supersede this Slack-specific bug.

## What This Changes

The issue requests that Slack `/elevated` authorization honor `tools.elevated.allowFrom.slack` without requiring a separate Slack command allowlist, or provide an implemented and documented Slack fallback.

## Best Possible Solution

Fix the Slack room/group `/elevated` path narrowly so senders listed in `tools.elevated.allowFrom.slack` can reach elevated permission resolution without widening access to other Slack control commands. Add a Slack regression test for a channel/group sender listed only under the elevated allowlist, and update the public elevated or Slack docs to describe the resulting behavior.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: with Slack access groups enabled, no Slack account/channel command allowlist, and a room/group message containing `/elevated on`, `hasControlCommand` is true, `commandGate.shouldBlock` is true, and `prepareSlackMessage` returns `null` before `resolveElevatedPermissions`.

## Solution Assessment

Unclear until a maintainer chooses the seam. The safest direction is an elevated-specific Slack authorizer or fallback that only affects `/elevated`, not a general addition of the elevated allowlist to every Slack control command gate.
