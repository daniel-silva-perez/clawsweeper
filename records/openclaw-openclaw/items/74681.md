## Summary

Keep open. Current main still defers non-main isolated `agentTurn` cron timeout arming until the runner reports execution start, while the issue requests a hard dispatch-time wall-clock deadline. The existing #41783 regression coverage intentionally protects the conflicting lane-wait behavior, so this needs maintainer contract judgment rather than cleanup closure.

## What This Changes

The issue requests cron `agentTurn` `timeoutSeconds` enforcement as a fixed wall-clock deadline from job start, independent of isolated runner callbacks or async work.

## Best Possible Solution

Define the cron timeout contract explicitly, then update implementation, tests, and docs together. A maintainable path is likely split total-deadline and execution-budget semantics, or a clearly named queue/preparation budget, so long pre-execution delays are bounded without regressing #41783.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an isolated `agentTurn` fake-timer path where `runIsolatedAgentJob` waits longer than `timeoutSeconds` before calling `onExecutionStarted`; the existing #41783 regression test demonstrates the abort signal remains un-aborted during that wait.

## Solution Assessment

No, not as a direct automated change to start every isolated `agentTurn` timeout at dispatch. That would match this report narrowly but contradict the currently tested #41783 lane-wait semantics, so the best fix needs an explicit maintainer decision on split or renamed budgets.
