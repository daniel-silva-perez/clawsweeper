## Summary

Keep open. Current main still defers isolated non-main `agentTurn` timeout arming until `onExecutionStarted`, while docs/changelog/tests preserve the #41783 execution-lane semantics, so the requested dispatch/start-time hard deadline remains unimplemented and needs maintainer contract judgment before a safe fix.

## What This Changes

Request hard wall-time enforcement for isolated cron `agentTurn` `timeoutSeconds` from job start or dispatch, independent of runner callbacks or async operations.

## Best Possible Solution

Define the cron timeout contract explicitly, likely by separating an absolute job/run deadline from the existing execution-lane budget or by adding a named queue/preparation budget. Then update the cron wrapper, runner callback tests, docs, and changelog together so #41783 remains intentional while unbounded pre-execution delays are capped.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an isolated `agentTurn` fake-timer path where `runIsolatedAgentJob` delays `onExecutionStarted` beyond `timeoutSeconds`; the existing #41783 regression already demonstrates the abort signal remains un-aborted during that wait.

## Solution Assessment

No, not as a direct automated patch that starts every isolated `agentTurn` timeout at dispatch. That would address this report narrowly but conflicts with the current #41783 execution-timeout contract, so the safer solution needs an explicit maintainer decision on split or renamed budgets.
