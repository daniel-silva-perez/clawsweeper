## Summary

Keep open. Current main has the merged static-prompt split for volatile inbound metadata, but the group-channel failure mode reported here still appears present: `groupIntro` is injected only on first/reintro turns and is still included in the hashable `extraSystemPromptStatic` value used for CLI session reuse.

## Best Possible Solution

Keep this issue open and fix the group-channel variant in the CLI session hash path. The narrow implementation should exclude `groupIntro` from `extraSystemPromptStatic` or otherwise make the hash source stable across first and follow-up group turns while leaving the actual prompt injection unchanged, then add a regression test for first-turn to second-turn group reuse.

## Clownfish Work Prompt

Fix the group-channel CLI session reset bug described in this issue. Current main already split dynamic inbound metadata from `extraSystemPromptStatic`, but `src/auto-reply/reply/get-reply-run.ts` still includes `groupIntro` in `extraSystemPromptStaticParts` even though `groupIntro` is only injected on first turn or explicit group reintro. Make the CLI session reuse hash source stable across first and follow-up group turns, at minimum by excluding `groupIntro` from `extraSystemPromptStaticParts`. Do not remove `groupIntro` from the actual `extraSystemPrompt` sent to the model and do not change group reply activation behavior. Add a regression test that models a group/channel first turn followed by a second turn and proves the hash/static prompt or session reuse no longer changes solely because `groupIntro` dropped out. Keep the fix focused; do not rewrite MCP resume hashing or auth epoch invalidation unless the narrow test requires a small compatibility seam. Add a single-line changelog fix entry with `Thanks @Zordon7-pixel` unless maintainers decide this is internal-only.
