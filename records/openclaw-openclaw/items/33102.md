## Summary

Current main still does not implement a config-backed default for `openclaw tui --deliver`. The CLI declares `--deliver` with a hard false default and passes `Boolean(opts.deliver)`, `runTui` still resolves `deliverDefault` as `opts.deliver ?? false`, the send path uses that fixed value, and the config type/schema/docs surfaces still have no `tui.deliver` or TUI `--no-deliver` support. The related implementation PRs #33198 and #33208 are closed unmerged; #66991/#69747 addressed the stuck waiting/no active run symptom, not the requested config default.

## Best Possible Solution

Keep this open and implement a narrow TUI config contract: add documented `tui.deliver?: boolean`, make omitted CLI input use `config.tui?.deliver ?? false`, preserve `--deliver` as a true override, add or verify `--no-deliver` as a false override, update generated config/docs surfaces, and add focused CLI/runtime/send-path tests. Do not flip the global default when the config key is absent.
