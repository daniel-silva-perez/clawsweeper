## Summary

Current main still does not implement a config-backed default for `openclaw tui --deliver`. The CLI defines `--deliver` as an explicit boolean defaulting to false, passes `Boolean(opts.deliver)` into `runTui`, `runTui` falls back to `false`, and the send path uses that fixed value. The config type, strict zod schema, generated help search, and public TUI docs still lack a `tui.deliver` config key. Related PRs #33198 and #33208 are closed unmerged; #66991 is closed as implemented for the stuck-spinner lifecycle bug but explicitly leaves this persisted-default request open.

## Best Possible Solution

Keep this issue open and implement a focused config contract for the TUI delivery default. Add a documented `tui.deliver` boolean to `OpenClawConfig`, strict zod schema, generated config help/schema output, and user docs. Resolve precedence explicitly: `--deliver` forces true, `--no-deliver` forces false, and omitted CLI input uses `tui.deliver ?? false`. Add focused tests for config default, CLI override true, CLI override false, missing-config fallback, and any docs-promised `/deliver` or settings behavior that is kept.
