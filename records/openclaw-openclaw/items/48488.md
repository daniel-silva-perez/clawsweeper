## Summary

Keep open. Current main still has no task-level timeout in the lane queue, and the newer stuck-session recovery only covers stale bookkeeping cases, not an active hung lane task. The same author has an open implementation PR, #48690, with closing syntax for this issue, so the issue should stay open until that PR or a maintainer equivalent lands.

## Best Possible Solution

Keep the issue open and review #48690 or a maintainer replacement that adds a configurable task-level timeout/recovery path for active lane tasks, with regression tests proving a never-settling task rejects, clears `activeTaskIds`, and allows queued session/cron work to drain. The current stuck-session diagnostic recovery can stay as complementary protection, but it does not replace the requested queue-level guard.
