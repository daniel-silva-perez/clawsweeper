## Summary

Keep this issue open. Current main still has no active task-level timeout in the command lane queue: `drainLane()` awaits `entry.task()` directly, `enqueueCommandInLane()` has no `taskTimeoutMs` option, and stuck-session diagnostics intentionally skip active lane tasks. The provided GitHub context also shows open PR #48690 with closing syntax for this issue, so cleanup policy should leave this issue open until that PR or a maintainer replacement lands.

## Best Possible Solution

Keep this issue open and review PR #48690 or land a maintainer-authored equivalent that adds a configurable active-task timeout or recovery path in `src/process/command-queue.ts`, with tests proving a never-settling task rejects, clears the active slot, notifies waiters, and lets queued session and cron work drain.

## Risks / Open Questions

- A queue-level timeout can release the lane while the original task promise continues running in the background, so late resolution, cancellation, logging, and duplicate side effects need careful review.
- Default timeout values need maintainer judgment across session turns, cron work, compaction, and slow provider calls.
- The current stuck-session recovery is complementary but not a substitute for this issue because it intentionally leaves active lane tasks alone.
