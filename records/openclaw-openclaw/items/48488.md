## Summary

Keep open. Current main still awaits active lane tasks without a task-level timeout, and the existing stuck-session recovery explicitly leaves active lane tasks warning-only. The reporter has an open implementation PR, #48690, with closing syntax for this issue, so cleanup policy should leave this issue open until that PR or a maintainer replacement lands.

## Best Possible Solution

Keep this issue open and review #48690 or a maintainer equivalent that adds a configurable task-level timeout/recovery path for active lane tasks, with regression coverage proving a never-settling task rejects, clears `activeTaskIds`, and allows queued session and cron work to drain. The existing stuck-session diagnostic recovery can stay as a separate stale-bookkeeping safeguard.

## Risks / Open Questions

- A queue-level timeout can release the lane while the original hung promise continues running in the background, so cancellation, logging, and duplicate side-effect behavior need careful review.
- The existing stuck-session recovery is complementary but not a substitute for the requested queue-level guard because it intentionally skips active lane tasks.
- Default timeout values need maintainer judgment for long-running compaction, cron, provider, and session work so legitimate slow tasks are not cut off unexpectedly.
