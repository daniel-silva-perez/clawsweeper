## Summary

Keep open. Current main still has no MS Teams live voice path, and the draft branch is not merge-ready: prior maintainer feedback already asks for a narrower split, and the diff has concrete security, dependency, and credential-handling blockers.

## What This Changes

The PR adds MS Teams voice capability tiers, TS gRPC/audio orchestration, a Windows .NET Graph media worker scaffold, manifest/config/schema updates, and a new voice documentation page.

## Best Possible Solution

Split the work into a maintainer-reviewed sequence: first land documented capability tiers and config/manifest contracts with no live media path, then add a pinned and build-proven Windows worker with a secure credential/TLS model, and only then wire the TS audio pipeline with compliance tests and live proof.

## Reproduction Assessment

Not applicable as a feature PR rather than a bug report. Source inspection gives high confidence that current main lacks the requested Teams live voice capability, but the branch does not provide high-confidence runtime proof because the critical worker, mock integration, live E2E, and RSC permission checks are unchecked.

## Solution Assessment

No. The proposed all-in-one branch is not the best way to solve this; the safer path is a small contract/config/docs PR first, followed by separately validated worker and audio-pipeline PRs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep app secrets out of plaintext worker RPC:** `extensions/msteams/src/voice/worker-bridge.ts:113`
  - body: `JoinMeeting` sends the Teams app secret to the worker, but the bridge is created with `createInsecure()` and `workerAddress` can point at a separate Windows host. That exposes the Entra client secret over plaintext gRPC; the worker should own/resolve credentials locally or the bridge must use an authenticated encrypted channel before this can land.
  - confidence: 0.93
- **[P2] Use the Teams credential resolver for voice joins:** `extensions/msteams/src/voice/manager.ts:141-145`
  - body: This branch derives `appSecret` only when `appPassword` is already a raw string, so normal SecretInput refs, env fallback, and federated configurations that current msteams supports produce an empty secret for live voice joins. Reuse the existing credential resolution path or move credential ownership to the worker.
  - confidence: 0.88
- **[P2] Declare the gRPC runtime dependencies:** `extensions/msteams/src/voice/worker-bridge.ts:34-37`
  - body: The new bridge dynamically imports `@grpc/grpc-js` and `@grpc/proto-loader`, but the msteams package manifest is not updated to depend on them. A packaged plugin or staged bundled runtime can fail module resolution as soon as voice is enabled.
  - confidence: 0.86
- **[P2] Pin media-worker package versions:** `extensions/msteams/media-worker/TeamsMediaWorker.csproj:11-15`
  - body: The new .NET worker uses wildcard `PackageReference` versions such as `1.2.*`, `2.*`, and `3.*`. That makes `dotnet build` non-deterministic and weakens supply-chain review for a Windows media process handling meeting audio; pin exact versions or add an approved lockfile strategy.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff introduces concrete security and supply-chain concerns around credential transport and dependency pinning.

Concerns:

- **[high] Plaintext gRPC carries Teams app secret:** `extensions/msteams/src/voice/worker-bridge.ts:113`
  - body: The bridge uses insecure gRPC credentials while the join request includes the Teams app secret for a configurable Windows worker address. This can expose an Entra client secret outside the local process boundary.
  - confidence: 0.93
- **[medium] Floating .NET package versions:** `extensions/msteams/media-worker/TeamsMediaWorker.csproj:11`
  - body: The media worker uses wildcard package versions without a lockfile, so future restore results can change under the same source commit in a security-sensitive audio worker.
  - confidence: 0.9
