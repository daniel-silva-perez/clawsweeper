## Summary

Keep open. Current main does not implement Teams live voice, and this draft PR still contains merge-blocking security, dependency, and media-pipeline issues that match the prior maintainer review saying the work should be split and proven before landing.

## What This Changes

This draft PR adds Microsoft Teams live voice documentation, manifest/config/schema types, a TypeScript voice manager and gRPC bridge, and a new Windows .NET media-worker scaffold.

## Best Possible Solution

Split this into a small owner-reviewed first PR for documented capability tiers, config schema, and no-op capability negotiation, then land separate authenticated worker, build-pinned .NET project, compliance-gated audio, and live/RSC proof PRs.

## Reproduction Assessment

Yes for the review blockers: static inspection reproduces missing gRPC dependency declarations, insecure bridge setup, direct file-byte playback into a PCM API, and floating NuGet refs. No live Teams E2E reproduction is available from the supplied context, and the PR test plan leaves that proof unchecked.

## Solution Assessment

No. The proposed all-in-one implementation is not the best way to land this feature because it combines public SDK/config changes, external Windows worker infrastructure, compliance policy, and realtime media behavior before the dependency, security, and permission contracts are proven.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Authenticate the worker bridge before sending secrets:** `extensions/msteams/src/voice/worker-bridge.ts:113`
  - body: `connect()` creates an insecure gRPC channel, while `JoinMeetingRequest` carries `appSecret` and Program.cs listens on all interfaces. With the documented open worker port, a network peer can observe or invoke call/audio control; require TLS/mTLS or a scoped auth token and avoid sending the app secret over this bridge.
  - confidence: 0.92
- **[P2] Declare the gRPC client dependencies:** `extensions/msteams/src/voice/worker-bridge.ts:27-35`
  - body: The new bridge imports `@grpc/grpc-js` and `@grpc/proto-loader`, but the PR does not add them to the Teams plugin package or lockfile. Typecheck/package builds will fail, and `voice.enabled` will throw before capability negotiation can run.
  - confidence: 0.95
- **[P2] Convert TTS output before streaming it as PCM:** `extensions/msteams/src/voice/manager.ts:461-465`
  - body: `textToSpeech()` returns an audio file path in the provider's output format, but this code reads those bytes and sends them to a worker API documented as raw 16 kHz mono PCM. Playback will send compressed/container bytes as PCM unless this uses telephony PCM output or an explicit transcode step.
  - confidence: 0.88
- **[P2] Open a send-capable audio socket for playback:** `extensions/msteams/media-worker/CallHandler.cs:95-99`
  - body: The worker creates the call audio socket with `StreamDirection.Recvonly`, then later uses the same socket for `AudioPlayback.Send()`. A receive-only media stream cannot satisfy the advertised listen-and-speak live voice path; use a send/receive configuration or a separate send-capable socket.
  - confidence: 0.82
- **[P2] Pin the media worker package versions:** `extensions/msteams/media-worker/TeamsMediaWorker.csproj:10-14`
  - body: The new .NET project restores wildcard package ranges such as `1.2.*`, `2.*`, and `3.*`. That makes builds nondeterministic and can silently pull different Graph/gRPC code; pin exact versions and commit the expected restore inputs before landing a worker scaffold.
  - confidence: 0.9

## Security Review

Status: needs_attention
