## Summary

Keep open. Current main still has only adjacent generic tool-loop detection; it is disabled by default, hashes raw tool name plus full params, and warning-level results are diagnostic-only rather than a model-visible read-specific reread nudge.

## What This Changes

The issue requests a warning-only same-turn reread guard for the first-class `read` tool to reduce token and quota burn from repeated file rereads.

## Best Possible Solution

Add a read-aware core guard that normalizes per-run read file identities, detects repeated same-file and small-file-set rereads without nearby progress, returns a model-visible soft nudge, emits diagnostics, and documents/tests the warning-only behavior without changing exec policy by default.

## Reproduction Assessment

Yes for the missing product behavior: current source and tests show repeated `read` calls only hit the generic exact-params detector when opt-in loop detection is enabled, and warnings are diagnostics-only. The concrete live UI transcript is not in the repo, so runtime validation should use synthetic focused tests that model the reported reread pattern.

## Solution Assessment

Yes, a narrow read-aware guard in the core agent tool loop is the best fit because the requested failure mode is in the first-class `read` path, not an optional plugin surface. Reusing the existing diagnostic/history plumbing is preferable, but the read identity and model-visible warning need dedicated logic.
