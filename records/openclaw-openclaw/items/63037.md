## Summary

Keep this PR open. The timeout direction is still relevant, but the branch is not ready to land: current main still has a no-deadline helper wait, the PR does not reach the primary controller-based Gateway startup path, and the direct monitor timeout path can reject before the new socket is owned or closed. Current main's Baileys `connectTimeoutMs` support reduces normal handshake hangs, but it is not the same as an OpenClaw-side deadline for the no-event helper case.

## What This Changes

The PR adds an optional `timeoutMs` parameter to `waitForWaConnection`, adds timeout/listener-cleanup tests, and passes a 60s timeout from the legacy direct WhatsApp inbox monitor.

## Best Possible Solution

Revise the branch into a narrow WhatsApp connection-lifecycle fix: keep QR login waits unbounded unless the login layer supplies its own deadline, add an opt-in initial connection deadline for monitor startup, route it through `WhatsAppConnectionController.openConnection` and the direct monitor path, close sockets on timeout before retry, align the runtime wrapper if the helper parameter is exposed, and cover helper cleanup plus controller/direct-monitor timeout cleanup in tests.

The main monitor loop can still hang in the no-event case this PR is meant to fix.
