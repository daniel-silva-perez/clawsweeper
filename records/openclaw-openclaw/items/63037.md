## Summary

Keep this PR open. Current main still has the unbounded WhatsApp connection wait, and the PR is relevant, but it is not yet sufficient to close or land as-is because the primary Gateway controller startup path still awaits the helper without a deadline and the direct monitor timeout path would need socket cleanup on timeout.

## Best Possible Solution

Keep this PR open and request a focused update: add an opt-in initial WhatsApp connection timeout to waitForWaConnection, plumb it through WhatsAppConnectionController.openConnection and the Gateway monitorWebChannel startup path, pass it from the legacy direct monitor path, close timed-out sockets before retrying, preserve unbounded QR login waits unless the login layer supplies its own deadline, update the runtime wrapper type if the optional parameter is part of the exposed helper contract, and keep regression coverage for helper timeout/listener cleanup plus controller/direct-monitor socket cleanup.

## Risks / Open Questions

- Landing the PR as-is would likely leave the primary controller-based WhatsApp Gateway startup path able to hang indefinitely before listener ownership, heartbeat, or watchdog setup.
- The direct monitor timeout path added by the PR should close the just-created socket before rethrowing; otherwise repeated retries can orphan sockets before attachWebInboxToSocket owns them.
- If the helper timeout is an exposed runtime helper contract, the runtime wrapper type needs to accept the optional timeout parameter as well.
