## Summary

Keep this PR open. Current main still has the unbounded WhatsApp connection wait, so the PR targets a real remaining failure mode, but the submitted diff is not enough to close or land as-is because it only wires the timeout into the legacy direct monitor path and leaves the primary controller-based Gateway startup path awaiting the helper without a deadline.

## Best Possible Solution

Keep this PR open and request a focused update: add an opt-in initial WhatsApp connection timeout to waitForWaConnection, plumb it through WhatsAppConnectionController.openConnection and the Gateway monitorWebChannel startup path, pass it from the legacy direct monitor path, close timed-out sockets before retrying, preserve unbounded QR login waits unless the login layer supplies its own deadline, align the runtime wrapper type if the optional parameter is exposed, and keep regression coverage for helper timeout/listener cleanup plus controller/direct-monitor socket cleanup.

## Risks / Open Questions

- Landing the PR as-is would likely leave the primary controller-based WhatsApp Gateway startup path able to hang indefinitely before listener ownership, heartbeat, or watchdog setup.
- The direct monitor timeout path added by the PR should close the just-created socket before rethrowing; otherwise repeated retries can orphan sockets before attachWebInboxToSocket owns them.
- If the helper timeout is part of the exposed runtime helper contract, the runtime wrapper type should accept the optional timeout parameter too.
