## Summary

Keep this PR open. The reported unbounded WhatsApp connection wait is still present on current main and the PR is useful source work; the latest branch also addressed the earlier QR-login default-timeout regression by making the helper timeout opt-in. It is not correct as-is because it only applies the timeout to the legacy direct monitor path, leaving the primary Gateway controller startup path unbounded, and the direct timeout path needs to close the newly created socket before retrying.

## What This Changes

The PR adds an optional `timeoutMs` parameter to `waitForWaConnection`, adds timeout/listener-cleanup tests, and passes a 60-second timeout at the legacy `monitorWebInbox` callsite.

## Best Possible Solution

Land a focused WhatsApp startup-timeout fix that keeps QR login waits unbounded unless the login layer supplies its own deadline, applies an initial connection deadline to both controller-based Gateway startup and legacy direct monitor startup, and guarantees the just-created socket is closed on timeout before reconnect retry.

## Reproduction Assessment

Yes. A focused source-level reproduction is to call `waitForWaConnection` with an EventEmitter-backed socket and never emit `connection.update` with `open` or `close`; on current main that promise remains pending, and the controller path awaits that helper before installing any later watchdog.

## Solution Assessment

No. The helper-level opt-in timeout is a reasonable primitive, but this PR is not the best complete fix until the timeout is plumbed through the controller startup path and the direct monitor timeout path closes the socket it created.

Wrap the wait in a try/catch and close the newly created socket before rethrowing to avoid accumulating orphaned WhatsApp sockets across retries.
  - confidence: 0.91
