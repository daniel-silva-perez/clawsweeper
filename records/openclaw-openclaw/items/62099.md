## Summary

Keep open. Current main still has the request-visible cascade path, and the provided GitHub context shows an open, unmerged PR explicitly paired with this issue, so the issue should stay open until that PR or an equivalent fix lands.

## What This Changes

Make Windows auth-profile bookkeeping write failures non-fatal so EPERM on auth-profiles.json cannot cascade through model fallback and wedge the gateway.

## Best Possible Solution

Keep this open until #67077 or an equivalent focused patch makes last-good, usage, failure, and cooldown bookkeeping persistence failures warn-only while preserving fatal behavior for credential and OAuth token saves, with regression coverage for both lock-guarded and direct fallback save paths.

## Reproduction Assessment

Yes for the request-level cascade: current source shows successful embedded runs awaiting auth-profile bookkeeping, lock-path persistence errors converting to null, and direct fallback saves that can still throw. The underlying Windows ReadOnly attribute race itself was not live-reproduced in this read-only review.

## Solution Assessment

Yes, the narrow maintainable solution is to catch and warn-log persistence failures inside the bookkeeping mark* functions rather than weakening saveAuthProfileStore globally. Broader Windows atomic-writer hardening can remain separate because OAuth and credential writes still need hard failure semantics.
