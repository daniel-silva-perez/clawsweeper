## Summary

Current main still does not implement this PR's intended Slack Agents & Assistants DM-root thread preservation for non-`all` reply modes, so the PR is not obsolete. The branch remains directionally useful, but it should be revised before merge because the proposed guard uses raw `channel_type` instead of OpenClaw's normalized DM classification and the PR includes an unrelated agents test-harness change.

## Best Possible Solution

Keep open and revise. Preserve Slack Agents & Assistants DM root thread context using the normalized direct-message classification already computed in the Slack message pipeline, only set `MessageThreadId` for DM assistant self-thread roots, keep non-DM self-thread roots from forcing threaded tool output, add focused coverage for raw `im`, D-prefix DMs with missing or wrong `channel_type`, channel/group self-thread roots, and concurrent assistant-DM roots, and split the unrelated agents harness change into a separate PR if still needed.

## Risks / Open Questions

- The proposed raw `channel_type === "im"` guard can miss D-prefixed Slack DMs when Slack omits or misreports `channel_type`, while current OpenClaw intentionally normalizes those events as direct messages.
- Any overly broad `MessageThreadId` assignment for all `thread_ts === ts` events would regress channel/group behavior because Slack tool threading forces effective reply mode to `all` when `MessageThreadId` is present.
- The unrelated agents overflow-compaction harness edit increases review scope for a Slack-only fix and should be split unless it is needed by a separate PR.
