## Summary

Keep open. Current main still does not handle the exact Feishu fallback shape from the report: interactive-card extraction is flat and ignores nested `elements` arrays, while the post fallback parser requires a `content` array and does not reinterpret top-level `title` plus `elements`. The issue also has an open same-author PR with closing syntax, so it should remain open until that PR or an equivalent patch is reviewed and landed.

## What This Changes

The issue requests Feishu `message(action=read)` support for interactive-card fallback payloads that contain top-level `title` plus nested `elements` arrays with `text`, `a`, and media tags.

## Best Possible Solution

Update the Feishu plugin parser to extract title/header text and recursively flatten or otherwise normalize interactive fallback `elements` arrays, preferably reusing the post-element rendering behavior where appropriate, with regression coverage for the reported top-level `title` plus nested `elements` payload.

## Reproduction Assessment

Yes. A focused `getMessageFeishu` unit test can mock `client.im.message.get` with the issue's `msg_type: "interactive"` body content; by current source inspection it falls through to `[Interactive Card]` because nested arrays are ignored and `parsePostContent` rejects payloads without `content`.

## Solution Assessment

Yes, the best path is a Feishu-plugin-local parser/test patch such as #60383 or an equivalent maintainer revision. No core API or cross-plugin change appears necessary for this owner-specific parsing gap.
