## Summary

Keep open. Current main still has the reported failure shape: steer attempts are gated on streaming state, active non-streaming embedded runs reject queued steering, active `steer` falls back to followup, and Discord same-session inbound work is still serialized before reply handling can attempt steering. Related PR #73365 is open and maintainer-labeled, so this should wait for review, landing, or replacement rather than close early.

## What This Changes

The issue requests that `messages.queue.mode: "steer"` inject eligible inbound messages into an active Pi run instead of silently degrading to followup behavior during active tool execution.

## Best Possible Solution

Land or replace the focused lifecycle-aware steering fix so eligible same-session messages reach the embedded steering queue while a run is active, rejected steering attempts log actionable diagnostics, deliberate followup fallback remains intact, and docs/tests consistently describe the actual delivery boundary.
