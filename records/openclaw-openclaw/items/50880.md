## Summary

Keep open. Current main still has the reported steer-to-followup shape for active same-session messages: the reply path only attempts steering when `isStreaming` is true, the embedded Pi helper rejects active non-streaming handles, the active-run policy maps `steer` to followup, and Discord inbound jobs are serialized per routed session key before reply handling. Related PR #73365 is open and unmerged, so this should wait for maintainer review or an equivalent fix.

## What This Changes

The issue asks for `messages.queue.mode: "steer"` to inject Discord/main-session messages into the active run at the documented steering boundary and to log rejected steer-to-followup fallbacks.

## Best Possible Solution

Implement lifecycle-aware active-run steering so eligible same-session messages can reach the embedded steering queue while the run is active, emit actionable diagnostics when steering is rejected, preserve deliberate followup fallback, and keep the Discord/session queue behavior covered by focused tests.
