## Summary

Keep this issue open. Current main still does not satisfy the documented steer behavior for active non-streaming runs: steer injection is still gated on `isStreaming`, the embedded helper still rejects active handles that are not streaming, active `steer` still resolves to followup, and Discord inbound work still serializes same-key messages before reply handling can attempt steering. The focused maintainer PR #73365 remains open and unmerged, while the earlier Telegram-specific PR #60604 was closed without merge.

## What This Changes

The issue asks for `messages.queue.mode: "steer"` to inject inbound messages into the active agent run at the documented boundary, with diagnostics when steering falls back to followup.

## Best Possible Solution

Ship a lifecycle-aware steer path that lets eligible same-session messages reach the embedded steering queue while a run is active, emits useful diagnostics for rejected steer attempts, preserves deliberate followup fallback, and updates tests/docs so the injection boundary is consistent.
