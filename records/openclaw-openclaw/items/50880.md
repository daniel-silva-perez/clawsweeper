## Summary

Keep this issue open. Current main still documents `steer` as immediate active-run injection, but the implementation only attempts steering when the reply path reports `isStreaming`, `queueEmbeddedPiMessage` rejects active embedded handles that are not streaming, and active `steer` mode still falls back to `enqueue-followup` when the steer attempt is skipped or rejected. Discord also still serializes same-session inbound jobs before the reply pipeline, so the reported channel path remains plausible. Open PR #73365 is the right current follow-up thread, but it is unmerged and does not prove current main is fixed.

## Best Possible Solution

Keep this issue open and route it to #73365 or a narrow replacement. The best fix should make active steer injection lifecycle-aware in core, add diagnostic coverage for rejected steer attempts, and regression-test the Discord same-session inbound path so `messages.queue.mode: "steer"` can reach `queueEmbeddedPiMessage` while a run is active. After the fix merges and ships, close this issue as implemented with release provenance.
