## Summary

Keep this issue open. Current main still has the reported steer-to-followup behavior: the documented steer contract remains current-run injection, but the reply runner only attempts injection behind an `isStreaming` gate, the embedded queue helper rejects active non-streaming handles, and the active-run queue policy still maps active `steer` runs to followup. The supplied GitHub context also shows focused maintainer PR #73365 is open and unmerged, so this is not implemented on main.

## What This Changes

The issue reports that `messages.queue.mode: "steer"` falls back to followup behavior instead of injecting inbound messages into active agent turns at tool-call boundaries.

## Best Possible Solution

Review and land #73365 or a narrow replacement that makes steer injection lifecycle-aware for active non-streaming runs, adds diagnostics for rejected steer attempts, and covers the same-session Discord path so inbound messages can reach the embedded steering queue while a run is active.
