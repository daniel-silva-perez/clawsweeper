## Summary

Keep open. Current main still has the reported steer-to-followup behavior shape: active steering is gated on `isStreaming`, active embedded Pi handles reject non-streaming steering, active `steer` maps to followup, and Discord same-session inbound work is serialized before reply handling can attempt steering. Related PR #73365 is open and unmerged, so this should wait for maintainer review or an equivalent fix.

## What This Changes

The issue asks for `messages.queue.mode: "steer"` to inject eligible inbound messages into the active agent run instead of silently falling back to followup delivery during tool execution.

## Best Possible Solution

Implement lifecycle-aware active-run steering so eligible same-session messages can reach the embedded steering queue while the run is active, emit actionable diagnostics when steering is rejected, preserve deliberate followup fallback, and cover the Discord/session queue path with focused tests.

## Reproduction Assessment

Yes. The report has concrete Discord/config steps, and current main provides a source-level reproduction path: an active same-session `steer` message that is non-streaming or serialized by the same session key falls through to followup rather than successful steering.

## Solution Assessment

Yes, the best path is to review and land #73365 or an equivalent narrow replacement that changes the active/non-streaming steering gate and diagnostics without broad unrelated queue churn.
