## Summary

Keep this open. Current main still has the reported fallback path: the documented `steer` mode says inbound messages should enter the active run, but the reply runner and embedded-run queue helper still require `isStreaming`, active `steer` still resolves to `enqueue-followup`, and the Discord inbound worker still serializes work by session key before message processing reaches the reply queue policy. The related fix PR is open and unmerged, so current main is not fixed.

## What This Changes

The issue requests that `messages.queue.mode: "steer"` inject inbound messages into active agent runs at tool-call/model boundaries instead of silently behaving like followup.

## Best Possible Solution

Review and land #73365 or a narrow replacement that makes steer injection lifecycle-aware for active non-streaming runs, adds diagnostics for rejected steer attempts, and covers the Discord same-session inbound path so messages can reach `queueEmbeddedPiMessage` while a run is active.
