## Summary

Keep this issue open. Current main still contains the Nostr restart-loop root causes described by the reporter: the bundled Nostr bus passes an array-wrapped DM filter to `nostr-tools.subscribeMany`, the gateway `startAccount` promise resolves immediately after startup so the channel supervisor treats it as an exit and schedules restarts, and cleanup closes only the subscription handle rather than the pool relay connections. The related implementation PR #42832 is still open and unmerged, so the bug is not implemented on main yet.

## Best Possible Solution

Keep this issue open as the active Nostr restart-loop tracker. Review and either refresh/land PR #42832 or make an equivalent current-main patch that passes a plain DM `Filter` object to `SimplePool.subscribeMany`, keeps `startNostrGatewayAccount` pending until abort, closes subscription and pool resources on cleanup, and adds regression tests for the serialized REQ filter shape plus gateway lifecycle behavior.
