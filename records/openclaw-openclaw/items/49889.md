## Summary

Keep this issue open. Current main has internal Telegram lane result kinds and an internal-only preview-finalized message:sent bridge, but it still does not expose a clear public/log/operator delivery outcome taxonomy for sent, preview-finalized, preview-retained, hook-canceled, transport-failed, and intentional skipped paths.

## Best Possible Solution

Keep this open and implement an additive Telegram delivery-observability contract: expose a structured final delivery outcome for fresh sends, preview-finalized edits, preview-retained cases, hook-canceled skips, transport failures, and intentional skips; emit it consistently through internal and plugin hooks or a companion delivery event; add dispatch/lane tests for each path; document the fields in Telegram docs and hook docs.
