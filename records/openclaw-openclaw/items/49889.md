## Summary

Keep open. Current main still exposes Telegram partial-stream finalization mostly as local dispatch state plus an internal-only preview-finalized hook bridge; the public hook/log/docs surfaces still do not provide the requested delivery outcome taxonomy, and hook-canceled plus preview-retained paths remain hard to distinguish operationally.

## What This Changes

The issue requests clear Telegram partial-stream final delivery observability for sent, preview-finalized, preview-retained, hook-canceled, transport-failed, and skipped outcomes.

## Best Possible Solution

Implement a backwards-compatible Telegram delivery-observability contract: carry a structured final delivery outcome through the lane result, canonical/internal/plugin message-sent surfaces or a companion delivery event, emit it for fresh sends, preview-finalized edits, retained previews, hook cancellations, transport failures, and intentional skips, then document the fields in Telegram and hooks docs.

## Reproduction Assessment

Yes. A source-level reproduction is available on current main: partial-stream finalization can return preview-finalized or preview-retained from the lane deliverer, but the emitted hook context only exposes success/error/messageId and hook cancellation simply continues without a final outcome event.

## Solution Assessment

Yes, the narrow maintainable direction is an additive outcome field or companion delivery event rather than replacing partial streaming or forcing fresh final messages. The existing internal lane result is a useful implementation hook, but it is not sufficient until operators and hooks can observe it consistently.
