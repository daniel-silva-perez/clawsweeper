## Summary

Keep open. Current main has partial managed-inbound media support, but the normal agent image-tool path still passes a workspaceDir and can rewrite a media://inbound reference into a workspace path before loadWebMedia can hydrate it. PR #63497 remains open and unmerged with closing syntax for this issue, so the issue should stay open until that fix or an equivalent lands.

## What This Changes

Agent-session image tool calls should preserve inbound image data so configured image models receive the image instead of reporting that no image was uploaded.

## Best Possible Solution

Land PR #63497 or an equivalent narrow patch that preserves media://inbound refs through image-tool workspace normalization, lets the shared media loader resolve them, and adds regression coverage for createOpenClawTools/createImageTool with workspaceDir before closing with fix provenance.

## Reproduction Assessment

Yes. The source-level reproduction is createImageTool on the normal agent path with workspaceDir plus image media://inbound/<id>; current code routes that ref through resolve(workspaceDir, normalizedRef), and #74123 reports the resulting malformed workspace media:/inbound path.

## Solution Assessment

Yes. The narrow maintainable solution is to keep canonical inbound media refs intact until loadWebMedia/resolveInboundMediaReference handles them, with a regression test for the workspaceDir path; broad channel or provider changes are not needed.
