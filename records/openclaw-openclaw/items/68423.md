## Summary

Keep open. Current main still restores from `openclaw.json.bak` after parsing it without checking for redacted secret placeholders, and the linked PR that explicitly closes this issue is still open/unmerged in the provided GitHub context.

## Best Possible Solution

Keep this issue open and review PR #68462, or replace it with an equivalent maintainer-owned patch. The minimum fix should validate parsed `.bak` contents before both async and sync restore copies and add regression coverage for polluted `.bak` refusal. Backup-creation validation, deeper rotation policy, and backup-aware `openclaw config validate` behavior can be included if maintainers want that broader scope or split into follow-ups.
