## Summary

Keep open. Current main still captures `reasoning_content`/`reasoning`/`reasoning_text` from OpenAI-compatible streams as assistant thinking metadata, but the replay stripping policy only turns on for Gemma 4 OpenAI-compatible model ids. Qwen/oMLX-style custom `openai-completions` histories remain uncovered, and the related fix PRs in the discussion are closed and unmerged.

## What This Changes

The issue requests that OpenClaw stop replaying Qwen/oMLX `reasoning_content` from prior assistant turns into later OpenAI-compatible request history.

## Best Possible Solution

Land a narrow replay-policy fix for Qwen/oMLX-style OpenAI-compatible histories: strip or downgrade historical assistant thinking blocks with `thinkingSignature` values such as `reasoning_content`, `reasoning`, and `reasoning_text` before follow-up requests for affected local/self-hosted Qwen paths, while preserving provider-owned replay contracts and same-turn tool-call continuations. Add focused policy/replay tests, update transcript-hygiene docs, and add a changelog fix entry crediting the reporter.
