## Summary

Keep open. Current main has reasoning-history stripping machinery, but the active policy still covers Gemma 4 and selected provider-owned paths rather than the reported Qwen/oMLX-style OpenAI-compatible replay path. ## What This Changes

The issue asks OpenClaw to stop replaying Qwen/oMLX `reasoning_content` thinking metadata into subsequent OpenAI-compatible conversation-history requests when that history breaks strict provider parsing.

## Best Possible Solution

Add a targeted replay-history policy for affected Qwen/local self-hosted OpenAI-compatible backends that removes or downgrades historical provider reasoning thinking blocks before follow-up requests, while preserving providers that explicitly require reasoning replay and current same-turn tool-call continuations.

## Repair Work Prompt

Fix the Qwen/oMLX-style OpenAI-compatible reasoning replay bug tracked by https://github.com/openclaw/openclaw/issues/46637. Current main captures `reasoning_content` as assistant thinking history, but only Gemma 4 OpenAI-compatible replay enables `dropReasoningFromHistory`, so Qwen/local self-hosted OpenAI-compatible follow-up turns can still replay prior reasoning into request history. Implement a targeted policy or provider-family hook that strips or downgrades historical provider reasoning blocks for affected Qwen/local OpenAI-compatible backends, while preserving current same-turn tool-call continuations and provider-owned contracts that intentionally replay `reasoning_content` such as DeepSeek V4 and opt-in Z.AI preserved thinking. Do not make a broad low-level JSON escaping-only fix unless tests prove the serializer is the actual failure. Add focused tests for `reasoning_content`/`reasoning_text` replay policy, update transcript hygiene/provider docs if behavior changes, and add a user-facing changelog fix entry with proper attribution.
