## Summary

Keep open. Current main has the machinery to drop replayed reasoning, but the active OpenAI-compatible policy only enables it for Gemma 4; the reported Qwen/oMLX-style path can still carry `reasoning_content` thinking blocks into follow-up requests, and the related fix PRs are closed unmerged.

## What This Changes

The issue reports that Qwen/oMLX `reasoning_content` from a first OpenAI-compatible response is replayed in later conversation history and causes subsequent turns to fail with oMLX JSON parse errors.

## Best Possible Solution

Historical Qwen/local OpenAI-compatible provider reasoning blocks should be removed or downgraded before follow-up replay to strict backends, while preserving same-turn tool continuations and providers that explicitly require replayed reasoning content.

## Reproduction Assessment

Yes for the source-level condition: construct a prior assistant thinking block with `thinkingSignature: "reasoning_content"`, sanitize for a non-Gemma `openai-completions` Qwen/oMLX model, and observe that the current policy does not enable reasoning stripping. A live oMLX 422 was not rerun during this read-only review.

## Solution Assessment

Yes: the narrow maintainable fix is targeted replay policy/hygiene for affected Qwen-style local OpenAI-compatible backends. The code points more strongly to replaying provider reasoning metadata than to a low-level `JSON.stringify` escaping bug, so blanket control-character rewriting is the weaker direction.
