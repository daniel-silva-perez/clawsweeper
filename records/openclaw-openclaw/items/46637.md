## Summary

Keep open. Current main has the replay-history mechanism needed to strip historical reasoning, but the active OpenAI-compatible policy only enables it for Gemma 4. The reported Qwen/oMLX-style path can still preserve `reasoning_content` thinking blocks across follow-up turns, and the related fix attempts are closed and unmerged.

## What This Changes

The issue asks OpenClaw to stop replaying Qwen/oMLX OpenAI-compatible `reasoning_content` into later conversation-history requests that strict local backends reject.

## Best Possible Solution

Add targeted OpenAI-compatible replay hygiene for affected Qwen/local self-hosted backends so prior-turn provider reasoning blocks are removed or downgraded before follow-up requests, while preserving same-turn tool-call continuations and provider contracts that explicitly require replayed reasoning.

## Reproduction Assessment

Yes for the source-level replay condition: construct history with a prior assistant thinking block using `thinkingSignature: "reasoning_content"`, then sanitize for a non-Gemma `openai-completions` Qwen/oMLX model and observe that current policy does not enable reasoning stripping. A live 422 requires an oMLX server, which was not exercised during this read-only review.

## Solution Assessment

Yes, a targeted replay-policy fix is the maintainable direction. The source points more strongly to provider reasoning replay compatibility than to a low-level `JSON.stringify` escaping bug, so blanket control-character rewriting is a weaker fix.
