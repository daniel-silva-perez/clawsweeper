## Summary

Keep open. Current main still has replay-history stripping, but the default OpenAI-compatible policy only enables `dropReasoningFromHistory` for Gemma 4, and the Qwen/oMLX-style path can still preserve same-model `reasoning_content` thinking blocks across turns. The linked repair PRs are closed unmerged, and a GitHub search found no open PR currently tied to this issue.

## What This Changes

The issue asks OpenClaw to prevent Qwen/oMLX `reasoning_content` from being replayed in conversation history in a way that breaks follow-up OpenAI-compatible requests.

## Best Possible Solution

Add targeted replay-history hygiene for Qwen/oMLX-style OpenAI-compatible reasoning fields so prior-turn provider reasoning blocks are removed or downgraded before follow-up requests, while preserving same-turn tool-call continuations and explicit provider contracts that require reasoning replay. Update focused tests and transcript hygiene docs/changelog if behavior changes.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to put a prior assistant thinking block with `thinkingSignature: "reasoning_content"` into same-model `openai-completions` history for a non-Gemma Qwen/oMLX model and observe that current policy does not enable `dropReasoningFromHistory`; the issue also gives a concrete two-turn Telegram/oMLX flow.

## Solution Assessment

Yes for the repair direction. The narrow maintainable fix is provider/replay-policy hygiene for affected Qwen-style OpenAI-compatible history, not a generic JSON escaping patch or unconditional stripping across all providers.
