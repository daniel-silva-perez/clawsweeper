## Summary

Keep open. Current main records `reasoning_content` from OpenAI-compatible streams as assistant thinking metadata and the pi-ai OpenAI-completions converter replays signed thinking back as an assistant `reasoning_content` field. The new `dropReasoningFromHistory` seam only applies to Gemma 4 OpenAI-compatible model ids, so Qwen/oMLX-style `openai-completions` sessions remain uncovered. The related fix PRs for this report are closed and unmerged, and I found no open PR with closing syntax for this issue.

## Best Possible Solution

Land a narrow replay-policy fix for Qwen/oMLX-style OpenAI-compatible histories: strip or downgrade historical assistant thinking blocks with `thinkingSignature` values such as `reasoning_content`, `reasoning`, and `reasoning_text` before follow-up requests for affected Qwen local/self-hosted paths, while preserving existing provider-owned replay contracts for DeepSeek, Z.AI, Venice, OpenAI Responses `rs_*` items, and current tool-call continuations. Add focused policy and `sanitizeSessionHistory` tests plus a transcript-hygiene docs note.
