## Summary

Keep open. Current main still lacks the queue-first/idempotent completion delivery and iMessage internal-context hardening this PR targets, and related reporter context remains active; however, the PR head still has two blocking correctness issues around persisted-claim cleanup and iMessage code-fence chunking.

## What This Changes

The PR adds persisted subagent delivery claim/user-payload state, changes completion dispatch toward queue-first delivery, and updates iMessage outbound sanitization, chunking, send behavior, and focused tests.

## Best Possible Solution

Keep the hardened-delivery direction, but first release or avoid persisted claims on all early exits, make code-fence normalization operate on the complete iMessage payload rather than independently per chunk, reconcile the behavior with the active missing-payload reports, then run the targeted subagent announce/registry and iMessage monitor/send tests before maintainer merge review.

## Reproduction Assessment

Yes for the PR-head defects: source inspection shows a fresh delivery claim can be persisted before an already-aborted return that bypasses cleanup, and iMessage text is normalized before chunking and again during each chunk send. The original production leak/duplicate-delivery scenario was not live-reproduced in this read-only pass.

## Solution Assessment

No. The overall queue-first, persisted-payload direction looks appropriate for the current-main gap, but the current implementation is not the best fix until claim cleanup covers early aborts and iMessage fence closure is owned by a single full-message normalization boundary.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Release the claim on early abort:** `src/agents/subagent-announce-dispatch.ts:185`
  - body: `beginPersistedAnnounceDeliveryClaim` can persist a fresh `claimed` state before the aborted-signal return below it. Since that return happens before the `try/finally`, the claim is not deleted or finalized, and later retries for the same announce can be rejected as `delivery-already-in-flight` until the lease expires.
  - confidence: 0.92
- **[P2] Avoid normalizing each chunk independently:** `extensions/imessage/src/send.ts:146`
  - body: Monitor delivery already normalizes the full reply before chunking, but this send path normalizes each chunk again. If a long fenced code block is split across chunks, an intermediate chunk can have an odd fence count and this call appends a closing fence, corrupting the code formatting delivered to iMessage.
  - confidence: 0.86
