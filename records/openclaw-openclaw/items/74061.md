## Summary

Keep open. Current main still contains the reported sandboxed final-reply failure mode: with a sandbox root present, absolute host-local `MEDIA:` paths are forced through sandbox resolution, rejected before the existing agent-scoped outbound media policy can stage them, and converted to the visible `Media failed` fallback. The discussion's cron data point is relevant and appears adjacent, but it travels a separate direct-delivery path that also lacks sandbox-to-host media bridging.

## What This Changes

The issue asks OpenClaw to restore Telegram delivery for local `MEDIA:` file paths in final replies, with a related cron announce local-media case noted in comments.

## Best Possible Solution

Fix the sandboxed normal-reply media path narrowly: after sandbox mapping fails, allow only managed media or paths that pass the existing agent-scoped outbound media policy for the active workspace/session, then stage through the existing outbound attachment path. Add positive coverage for the reported sandboxed Telegram-style final reply and negative coverage for unrelated host paths; explicitly decide whether cron direct delivery should use the same bridge or be tracked separately.

## Reproduction Assessment

Yes. Current source and checked-in tests give a high-confidence reproduction path: with `sandboxRoot` present, an absolute host-local `MEDIA:` path is rejected by sandbox mapping, dropped, and surfaced as `⚠️ Media failed.`.

## Solution Assessment

No. Current main is not the best solution because it keeps a blanket sandbox-first rejection for host-absolute media paths that may already be valid under the agent-scoped outbound media policy; the safer fix is a policy-checked fallback, not an unrestricted host-path allowlist.
