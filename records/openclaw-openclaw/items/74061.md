## Summary

Keep open. Current main still preserves the reported sandboxed final-reply failure mode: host-absolute local MEDIA paths are resolved through the sandbox mapper first, dropped when they are outside the sandbox root, and can be converted into the visible media-failure fallback before Telegram gets a usable upload path.

## What This Changes

The issue asks OpenClaw to restore Telegram delivery for final replies that contain local MEDIA paths produced by agent workflows, especially host-local workspace files from sandboxed runs.

## Best Possible Solution

Add a narrow, policy-checked bridge for sandboxed final-reply media: after sandbox mapping fails, allow only managed media or files that pass the active agent/session outbound media policy, then stage through the existing outbound attachment path. Add positive coverage for the reported Telegram-style final reply and negative coverage for unrelated host paths, and explicitly decide whether cron-owned announce delivery uses the same bridge.

## Reproduction Assessment

Yes. The current source and checked-in tests provide a high-confidence reproduction path: with sandboxRoot present, an absolute host-local MEDIA path is rejected by sandbox mapping, dropped, and converted to the Media failed fallback.

## Solution Assessment

No. Current main is not the best solution because it keeps a blanket sandbox-first rejection for host-absolute paths that may already be valid under the agent-scoped outbound media policy; the safer fix is a policy-checked fallback, not an unrestricted allowlist.
