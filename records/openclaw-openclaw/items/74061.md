## Summary

Keep open. Current main still drops the reported sandboxed Telegram final-reply media path before the agent-scoped outbound media staging path can read it, and the newer cron comment points to the same sandbox-to-host translation gap on another delivery surface.

## What This Changes

Telegram final replies should deliver local media referenced by `<final>MEDIA:` from an agent workspace path instead of sending `Media failed`.

## Best Possible Solution

Keep this issue open and fix the sandboxed normal-reply media path narrowly: after sandbox mapping fails, allow only managed media or paths that pass the existing agent-scoped outbound media policy for the active workspace/session, then stage through the existing outbound attachment path. Add positive coverage for a sandboxed Telegram-style final reply with a host absolute agent-workspace path and negative coverage for unrelated host paths; decide whether the cron-owned path should share the same fix or get a narrower follow-up.

## Reproduction Assessment

Yes. The code path and existing tests provide a high-confidence reproduction: with a sandbox root present, an absolute host-local workspace `MEDIA:` path is resolved as a sandbox path, rejected as outside the sandbox root, dropped, and converted to `⚠️ Media failed.`

## Solution Assessment

No. Current main is not the best solution because it preserves a blanket sandbox-first rejection for host absolute paths that may already be valid under the agent-scoped outbound media policy; the safer fix is a narrow policy-checked fallback, not an unrestricted host-path allowlist.
