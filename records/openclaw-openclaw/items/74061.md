## Summary

Keep open. Current main still has the reported failure mode for sandboxed final replies: a host-absolute local MEDIA path is resolved through sandbox mapping first, rejected when it is outside the sandbox root, and then converted into the visible media-failure fallback. The Telegram adapter can load trusted local media roots, so the remaining gap is upstream sandbox/host media normalization rather than Telegram multipart upload itself.

## What This Changes

The issue requests that Telegram outbound delivery send local files referenced by final-reply MEDIA paths instead of replacing them with “Media failed.”

## Best Possible Solution

Add a narrow, policy-checked fallback for sandboxed normal final replies: after sandbox mapping fails, allow only managed media or paths that pass the active agent/session outbound media policy, then stage through the existing outbound attachment path. Preserve negative coverage for unrelated host paths, and make an explicit decision on wiring the same sandbox bridge into cron-owned announce delivery.

## Reproduction Assessment

Yes. Current source and checked-in tests provide a high-confidence reproduction path: with sandboxRoot present, an absolute host-local MEDIA path is rejected as outside the sandbox root, dropped, and converted to the warning text.

## Solution Assessment

No. Current main is not the best solution because it keeps a blanket sandbox-first rejection for host-absolute paths that may already be valid under the agent-scoped outbound media policy; the safer repair is a policy-checked fallback, not an unrestricted host-path allowlist.
