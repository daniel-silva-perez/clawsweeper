## Summary

Keep this issue open. Current main still has a plausible sandboxed normal-reply failure path that matches the reported Telegram `⚠️ Media failed.` result: final/block/tool replies are normalized before channel delivery, but when a sandbox workspace exists, `createReplyMediaPathNormalizer()` forces absolute local media through `resolveSandboxedMediaSource()` first. That resolver rejects host absolute paths outside the sandbox root before the normal agent-scoped outbound media access path can stage them. Existing tests cover non-sandbox absolute workspace media as working, and explicitly cover sandboxed absolute host-local paths being dropped; they do not cover the reported sandbox-plus-absolute-`~/.openclaw/workspace/...` case.

## Best Possible Solution

Keep this issue open and investigate the sandboxed normal-reply media path. The likely product fix is a narrow change in reply-media normalization: when a sandbox root exists, allow absolute host-local `MEDIA:` paths only if they are already managed media or pass the existing agent-scoped outbound media access policy for the active workspace/session, then stage them through `resolveOutboundAttachmentFromUrl`; otherwise keep the current sandbox-root rejection. Add regression coverage for a sandboxed Telegram-style final reply with an absolute `/home/openclaw/.openclaw/workspace/...` media path, plus negative tests for unrelated host paths.
