## Summary

Current main still lacks explicit `@agent` or `agent:` targeting for `/new` and `/reset`. The commands accept only generic input, and Discord/Telegram native command paths target the already-routed channel/topic/binding session rather than resolving a requested agent. The newer Telegram production comment reinforces the remaining bug; related #58110 is broader UI work, not a superseding command-targeting issue.

## Best Possible Solution

Keep this issue open and implement a focused native-command targeting feature. Add a structured target option/parser for `/new` and `/reset` on Discord and Telegram, resolve it against configured agents, identities, mention patterns, bindings, and channel authorization, fail closed on ambiguous or unauthorized targets, and add regression tests for shared channels, topic/thread bindings, ACP-bound sessions, blocked channels, no-target defaults, and both Discord and Telegram syntax. Treat #58110 as adjacent broader UI work, not the canonical tracker for this native command bug.

## Risks / Open Questions

- The no-target behavior needs explicit product direction: fail closed, prompt for a target, use the routed session only, or support an explicit all-agents action are materially different semantics.
- A fix must preserve per-channel, per-topic, per-account, and per-agent authorization so a target argument cannot reset a session the sender could not otherwise control.
- Existing workarounds such as per-agent bots, channel/topic bindings, and mention patterns reduce ambiguity but do not solve the native `/new` and `/reset` targeting UX in shared multi-agent channels.
