## Summary

Keep open. Current main still lacks explicit `@agent` or `agent:` targeting for `/new` and `/reset`; Discord and Telegram native commands route to the already-effective channel/topic/binding target, and extra command text is treated as normal command input or reset follow-up text. The newer Telegram production comment strengthens the report rather than making it stale, and the related #58110 is a broader multi-agent chat UI request, not a replacement for this native command targeting bug.

## Best Possible Solution

Implement this as a focused native-command targeting feature. Add an optional agent target for `/new` and `/reset` in Discord and Telegram command paths, resolve it against configured agents, identities, mention patterns, bindings, and channel authorization, set `CommandTargetSessionKey` only for the selected authorized session, define a conservative no-target policy, and add regression tests for shared channels, bound ACP/thread sessions, blocked channels, unauthorized targets, ambiguous targets, and both Discord and Telegram syntax.

## Risks / Open Questions

- The no-target default still needs product direction: fail closed, prompt/select, reset active participants, highest-context session, or reset all are materially different behaviors.
- Any fix must preserve channel/guild/topic authorization and binding checks so a target argument cannot reset an agent/session the sender is not allowed to control.
- Prefix-routing and channel-isolation workarounds from the discussion are useful operational context, but they do not solve the native command UX or least-surprise problem in core.
