## Summary

Keep this PR open. Current main still intentionally shows catalog-friendly model labels while preserving qualified values underneath, and the PR is a product-choice change that is not obsolete. The patch also leaves a blocking test coverage gap for the helper whose behavior it changes, so it should not merge as-is.

## What This Changes

The PR changes the Control UI chat model picker to display exact provider/model refs for option and default labels instead of catalog display names, with select-state tests updated for that label behavior.

## Best Possible Solution

Choose the Control UI label policy explicitly. If exact visible refs are the desired UX, update all affected helper and select-state tests together and remove the now-unused display lookup path; if friendly catalog labels remain intended, close this PR by product decision and keep the existing alias/name behavior.

## Reproduction Assessment

Yes for the behavioral delta. A source-level reproduction is the existing aliased or duplicate-name catalog test cases: current main resolves qualified values but returns friendly labels such as `Kimi K2.5 (NVIDIA)` and `Claude Sonnet · openrouter` instead of exact provider/model refs.

## Solution Assessment

No as-is. The direction is narrow, but the patch changes the helper label contract without updating `chat-model-ref.test.ts`, leaves dead lookup plumbing, and still needs maintainer product confirmation before replacing the current friendly-label UX.
