## Summary

Keep open. Current main still has generic inbound audio transcription through media understanding, but the requested BlueBubbles-owned pre-dispatch enricher is not implemented: there is no BlueBubbles audio-transcript endpoint client, no `channels.bluebubbles.inboundAudioEnricher` config, no UTI-aware transcript rewrite before dispatch, and no scoped public `message_transcribed` hook.

## What This Changes

Add a BlueBubbles inbound audio enricher that fetches Apple's BlueBubbles audio transcript before agent dispatch, rewrites placeholder audio bodies, emits a scoped transcription hook, and exposes opt-out config.

## Best Possible Solution

Implement this as BlueBubbles plugin-owned behavior only after maintainer approval of the config and hook contract: add a guarded client method for the BlueBubbles transcript endpoint, detect audio by MIME and UTI before dispatch, rewrite empty/audio-placeholder bodies from the server transcript, keep transcript contents out of logs, and document/test the opt-out defaults and scoped event payload.

## Reproduction Assessment

Not applicable as a feature request, but the source-level verification is high confidence: current main can be checked by sending an attachment-only BlueBubbles audio webhook through the existing code path and observing that it builds `<media:audio>` plus media paths, not a BlueBubbles transcript rewrite.

## Solution Assessment

Unclear until maintainers approve the hook/config shape. The generic `tools.media.audio` path is not the best answer for this request because it uses configured STT providers later in the shared reply pipeline instead of BlueBubbles' on-device transcript endpoint before dispatch.
