## Summary

Keep open. Current main still lacks outbound `toolContext` for forwarded exec-approval deliveries, and this PR head is not mergeable as-is: the new test uses missing imports, the new context field does not reach channel handler creation, and queued delivery replay would drop the context.

## What This Changes

The PR adds target-derived `toolContext` plumbing for forwarded exec-approval notices, extends outbound channel context typing, and adds regression coverage for forwarded request/resolved deliveries.

## Best Possible Solution

Land an adapted fix on current main that makes target-derived channel identity part of the typed outbound delivery contract, reaches real channel adapter contexts, survives delivery queue replay, and keeps focused regression coverage for immediate and recovered forwarded approval notices.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: in a multi-channel setup, forwarded approval delivery can resolve the correct target but current main sends no `toolContext`, while message-action channel selection relies on `toolContext.currentChannelProvider` before throwing the existing multi-channel error.

## Solution Assessment

No. The PR's direction is right, but the current branch is incomplete because it only adds part of the context plumbing and does not persist it through the write-ahead queue/recovery path.
