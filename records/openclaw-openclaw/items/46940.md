## Summary

Keep this PR open. Current main still lacks the requested target-derived exec-approval forwarding context, and the PR remains directionally useful but is incomplete: its outbound context does not reliably reach adapter calls or queued replay after retry/recovery.

## What This Changes

The PR adds a `toolContext` field to outbound delivery plumbing, derives `currentChannelId`/`currentChannelProvider` from exec-approval forward targets, and adds regression coverage for forwarded approval notices.

## Best Possible Solution

Adapt the fix against current main so target-derived channel identity is part of the typed outbound delivery contract, reaches real channel adapter contexts, and survives queued retry/recovery, with regression tests for both immediate forwarded approval sends and queue replay.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Propagate toolContext into channel handler creation:** `src/infra/outbound/deliver.ts:191`
  - body: The PR adds `toolContext` to delivery params and the outbound context builder, but the handler creation path also needs to pass `toolContext: params.toolContext`. Without that, adapter `sendText`/`sendPayload` contexts still receive `ctx.toolContext` as `undefined`, so the multi-channel approval routing failure can remain.
  - confidence: 0.88
- **[P2] Persist toolContext for queued delivery replay:** `src/infra/exec-approval-forwarder.ts:348-351`
  - body: Forwarded deliveries are written to the delivery queue before send, and recovery rebuilds deliver params from `QueuedDeliveryPayload`. Since the PR does not add `toolContext` to the queued payload, enqueue write, and recovery replay path, a retry can resend the approval notice without `currentChannelProvider/currentChannelId`.
  - confidence: 0.9
