## Summary

Keep open. Current main still drops the target-derived channel identity before outbound forwarding, and this PR is useful prior art but is not mergeable as written because the new context is not fully typed, propagated, or preserved through queued replay.

## What This Changes

The PR adds target-derived channel tool context to forwarded exec-approval deliveries, extends outbound delivery context typing, and adds regression coverage for forwarded request and resolved notices.

## Best Possible Solution

Preserve target-derived channel identity as a typed outbound delivery contract from exec-approval forwarding through immediate adapter sends and queued recovery replay, with regression tests covering both direct forwarded notices and recovered queued notices.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: current main can resolve a forwarded approval target but sends no `toolContext`, while message-action channel selection relies on `toolContext.currentChannelProvider` before throwing the multi-channel channel-required error.

## Solution Assessment

No. The direction is right, but this PR is not the best mergeable fix until it is rebased/adapted to current main and carries `toolContext` through the adapter context, delivery queue storage, and recovery replay.
