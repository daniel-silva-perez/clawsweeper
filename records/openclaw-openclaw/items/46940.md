## Summary

Keep open. Current main still lacks the target-derived outbound tool context needed for multi-channel forwarded exec-approval notices, and this PR remains useful source work, but the current head is not mergeable because it leaves compile-time and delivery-path gaps that prior review comments only partly addressed.

## What This Changes

The branch adds target-derived `toolContext` to forwarded exec-approval deliveries, extends outbound channel context typing, and adds regression assertions for forwarded request and resolved notices.

## Best Possible Solution

Land an adapted current-main fix that makes target-derived channel identity part of the typed outbound delivery contract, passes it into real channel adapter contexts, persists and replays it through the delivery queue, and covers immediate plus recovered forwarded approval notices.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: forwarded approval delivery resolves the correct target but current main sends no `toolContext`, while message-action channel selection depends on `toolContext.currentChannelProvider` before throwing the multi-channel channel-required error.

## Solution Assessment

No. The PR's direction is right, but this branch is not the best mergeable fix until it imports the new type correctly, forwards `toolContext` through handler creation, and preserves it through queued delivery replay.
