## Summary

Keep open. The PR targets a real multi-channel exec-approval forwarding bug, but the branch is stale and incomplete: it adds target-derived `toolContext` at the forwarder boundary while missing required type/test imports, missing propagation into the outbound channel handler, and missing queue/recovery persistence for retry replay.

## What This Changes

The PR adds target-derived channel tool context to forwarded exec-approval deliveries, extends outbound adapter delivery context typing, and adds regression coverage for forwarded pending and resolved notices.

## Best Possible Solution

Preserve target-derived channel identity as a typed outbound delivery contract from exec-approval forwarding through adapter sends and queued recovery replay, with focused regression tests for live forwarded pending/resolved notices and recovered queued notices on current main.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: current main can resolve a forwarded approval target, but sends no `toolContext`, while message-action channel selection relies on `toolContext.currentChannelProvider` before throwing the multi-channel channel-required error.

## Solution Assessment

No. The PR direction is right, but this is not the best mergeable fix until it imports/types the new contract correctly, passes `toolContext` into `createChannelHandler`, and persists/replays it through the delivery queue.
