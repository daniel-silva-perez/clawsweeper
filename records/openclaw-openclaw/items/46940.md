## Summary

Keep open. The PR targets a real current-main gap in forwarded exec-approval routing, but the branch is not mergeable: it is stale against current outbound type layout, misses required imports, does not actually pass the new context into channel handler creation, and would still drop the context on queued delivery replay.

## What This Changes

The PR adds target-derived current channel/provider context to forwarded exec-approval deliveries, extends outbound delivery context typing, and adds regression assertions for forwarded request and resolved notices.

## Best Possible Solution

Land an adapted current-main fix that treats target-derived channel identity as part of the typed outbound delivery contract, exposes it to real channel adapters, stores it in the write-ahead delivery queue, replays it during recovery, and covers both immediate and recovered forwarded approval notices.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: current main can resolve a forwarded approval target but sends no `toolContext`, while message-action channel selection falls back to `toolContext.currentChannelProvider` before throwing the multi-channel channel-required error.

## Solution Assessment

No. The proposed direction is right, but this PR is not the best mergeable fix until it is rebased/adapted to current main, imports the new types/helpers correctly, forwards the context into handler creation, and preserves it through queued replay.
