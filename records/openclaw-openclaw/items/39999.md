## Summary

Keep open. Current main has related hardening for staged global package updates, verification, swap rollback, and post-update doctor execution, but it still does not implement this PR's intended previous-version reinstall when the post-update doctor fails. The submitted PR remains plausible core update-safety work, but it needs rebasing onto the shared package update helper and should address the existing guard concern before merge.

## Best Possible Solution

Keep this PR open for contributor or maintainer revision. The best implementation is to move rollback support into the shared package update flow, carry the known `beforeVersion`, attempt `packageName@beforeVersion` rollback only after a successful live replacement/direct install followed by post-update doctor failure, rerun doctor when the restored entrypoint exists, preserve the overall error result, and add focused CLI/helper tests for rollback and no-rollback cases.

## Risks / Open Questions

- Merging the submitted patch as-is could run an extra global package-manager install even when the original update failed and the live install was not replaced, which can obscure the root cause and repeat the same registry/network failure path.
- The current PR has no focused regression coverage for doctor-failure rollback, no-rollback when the install step fails, or rollback doctor rerun behavior.
- Because the package update flow is now centralized in `src/infra/package-update-steps.ts`, implementing rollback only in older CLI-local code would miss or diverge from the Gateway update path.
