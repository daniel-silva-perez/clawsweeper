## Summary

Keep open. Current main has related global npm safety work, especially staged install verification before swapping the package tree, but it still does not implement this PR's intended rollback to the previous global package version when post-update doctor fails. The PR remains plausible core update-safety work, but it needs to be rebased onto the current shared package update helper and should address the existing guard concern.

## Best Possible Solution

Keep the PR open and revise it against the shared package update helper. The best implementation is to carry the known beforeVersion into package update handling, attempt packageName@beforeVersion rollback only after the live install was actually replaced or the final direct package install succeeded and a later verification/post-update doctor step failed, rerun doctor when the restored entrypoint exists, preserve the overall error result, and add focused CLI/helper tests for the rollback and no-rollback cases.

## Risks / Open Questions

- The submitted patch targets an older inline CLI flow; current main centralizes package update behavior in src/infra/package-update-steps.ts, so a direct merge would miss the shared helper design and likely the gateway update path.
- Rollback is a security- and supply-chain-sensitive extra package-manager execution. It should only run after the final package install/swap actually succeeded and later verification or doctor failed, so network, registry, or version-resolution failures do not trigger a redundant reinstall.
- The current PR lacks focused regression tests for doctor failure rollback, verification failure behavior, rollback doctor rerun, and failed-install-with-no-rollback.
