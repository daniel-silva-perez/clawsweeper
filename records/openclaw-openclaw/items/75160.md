## Summary

Current main already covers this PR's intended behavior through a generic Gateway tool-resolution fallback: known catalog tool IDs are resolved core-only first, then retried with plugin tools only when the requested tool is absent. That covers plugin-backed memory slot tools while preserving the core-tool protection for ordinary built-in tools, and the same implementation is present in the v2026.4.27 release tag.

## What This Changes

The PR changes Gateway HTTP `/tools/invoke` so `memory_search` and `memory_get` do not disable plugin tool resolution, and adds regression tests for memory-slot and ordinary core-tool invokes.

## Best Possible Solution

Close this PR without merging the alternate memory-name exception, keep the shipped generic fallback, and consider a separate test-only follow-up only if maintainers want explicit memory-slot regression coverage on top of the existing behavior.

## Reproduction Assessment

Yes. The behavior can be verified by the current source path: `memory_search`/`memory_get` are known catalog IDs implemented by the memory plugin, and `/tools/invoke` retries known missing tools with plugin resolution enabled; I did not run tests because this was a read-only review.

## Solution Assessment

No for merging this PR as-is. Current main's generic fallback is the better maintained solution now because it handles plugin-backed catalog tools without hard-coding memory tool names while still preferring ordinary built-in tools.
