## Summary

Keep this PR open. Current main still uses the numbered config backup ring, the linked feature issue remains open, and this PR is still a live implementation candidate rather than obsolete or implemented. It needs maintainer consolidation with the parallel open implementation and a decision on legacy backup migration and timestamp collision handling before landing.

## Best Possible Solution

Keep this PR open for maintainer review alongside #39923 and #39982. The best path is to consolidate one datetime-backup implementation in `src/config/backup-rotation.ts`, preserve or migrate legacy `.bak.N` files until retention is safe, make timestamp destinations collision-safe, keep bounded retention and permission hardening, update `src/config/config.backup-rotation.test.ts`, and add user-facing docs or a changelog entry only if the behavior ships.

## Risks / Open Questions

- Merging this PR as-is may delete existing legacy `.bak.N` rollback points on the first config write after upgrade.
- The PR still relies on millisecond timestamps without a collision-safe destination fallback; same-millisecond writes can lose one backup slot on POSIX or silently fail on Windows.
- There is another open implementation candidate for #39923, so maintainers should avoid parallel divergence by selecting or consolidating one approach.
