## Summary

Keep open. Current main still uses the numbered `.bak.1`-`.bak.4` config backup ring, while this PR remains a live implementation candidate for the same-author feature issue #39923 and overlaps with the still-open implementation PR #39982. It also still needs maintainer consolidation around migration safety and timestamp collision behavior.

## Best Possible Solution

Keep this PR open for maintainer review alongside #39923 and #39982. The best path is one consolidated datetime-backup implementation in `src/config/backup-rotation.ts` with collision-safe destinations, bounded retention, owner-only permission hardening, and a migration policy that preserves or safely converts legacy `.bak.N` recovery points until equivalent retention exists; update `src/config/config.backup-rotation.test.ts` and add docs or changelog only if the behavior ships.

## Risks / Open Questions

- The PR design, as described in the body and review comments, can delete legacy `.bak.N` rollback points on the first config write after upgrade before the datetime backup set is populated.
- The timestamp destination still has a same-millisecond collision concern: repeated rotations targeting the same suffix can lose one historical backup slot depending on platform rename semantics.
- There is a parallel open implementation candidate in #39982, so maintainers should consolidate behavior, migration policy, and tests before landing either PR.
