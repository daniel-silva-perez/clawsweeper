## Summary

Keep open. Current main still uses the numbered config backup ring, and this PR remains a live implementation candidate for the same-author open feature issue #39923. There is also a parallel open implementation candidate in #39982, so the right maintainer action is consolidation/review rather than closing this PR as implemented or superseded.

## Best Possible Solution

Keep this PR open for maintainer review alongside #39923 and #39982. The best path is to land one consolidated datetime-backup implementation in `src/config/backup-rotation.ts` that uses collision-safe timestamp destinations, preserves or safely migrates legacy `.bak.N` recovery points until retention remains safe, keeps bounded retention and permission hardening, updates `src/config/config.backup-rotation.test.ts`, and adds user-facing docs or a changelog entry only if the behavior ships.

## Risks / Open Questions

- Merging this PR as described can delete existing legacy `.bak.N` rollback points on the first config write after upgrade, reducing recovery history before the datetime set is populated.
- The PR discussion shows same-millisecond timestamp collisions were documented, but the implementation candidate may still lose one backup slot if two rotations target the same timestamped filename.
- There is a second open implementation candidate for the same linked feature, so maintainers should consolidate behavior, tests, and migration policy before landing either PR.
