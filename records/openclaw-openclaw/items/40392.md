## Summary

Keep this PR open. Current main still uses the numbered `.bak.1` through `.bak.4` config backup ring, the linked same-author issue #39923 is still open, and parallel PR #39982 is still an open implementation candidate rather than a merged replacement.

## Best Possible Solution

Keep this PR open for maintainer review alongside #39923 and #39982. The best path is one consolidated datetime-backup implementation in `src/config/backup-rotation.ts` with collision-safe destinations, bounded retention, owner-only permission hardening, and a migration policy that preserves or safely converts legacy `.bak.N` recovery points until equivalent datetime retention exists; update `src/config/config.backup-rotation.test.ts` and add docs or changelog only if the behavior ships.

## Risks / Open Questions

- The PR still needs maintainer resolution for migration behavior: deleting legacy `.bak.N` files on first upgrade can reduce available rollback points before the datetime set is populated.
- Timestamp-only destinations still need collision-safe behavior or explicit product acceptance, because same-millisecond rotations can clobber or skip one historical backup depending on platform rename semantics.
- There is a parallel open implementation candidate in #39982, so maintainers should consolidate behavior and tests before landing either PR.
