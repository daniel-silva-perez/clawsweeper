## Summary

Keep open. Current main still lacks this capability, and the force-pushed PR addresses the earlier hook-rewrite, explicit read-failure result, and network-policy exposure problems, but the current patch still has concrete authorization gaps in session-store error handling, path matching, and Bash command allowlist semantics.

## What This Changes

The PR adds an optional `envelope` object to Gateway `sessions.create` and `sessions.patch`, persists it on `SessionEntry`, enforces tool/path/command constraints in the before-tool-call path, and adds protocol docs plus create/patch/envelope tests.

## Best Possible Solution

Keep the PR open and require the envelope implementation to fail closed on actual store read/parse uncertainty, compare path rules against canonicalized tool-access paths, and make Bash command allowlists structurally enforceable before exposing this as a Gateway runtime policy surface.

## Reproduction Assessment

Yes. The remaining issues have source-level reproductions: a corrupt/unreadable session store is swallowed by `loadSessionStore`, raw `..` path strings can differ from the eventual filesystem path, and a command containing an allowed substring can pass the Bash allowlist.

## Solution Assessment

No. The PR is moving in the right direction and narrowed the network field appropriately, but the current implementation is not the safest maintainable solution for an authorization boundary until these bypass/fail-open paths are fixed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Fail closed on swallowed store read failures:** `src/agents/session-runtime-envelope.ts:115`
  - body: `loadSessionStore` catches read and parse failures and returns an empty store, so this lookup returns `{ ok: true, envelope: undefined }` instead of blocking. For a session that had a persisted envelope, a transient unreadable or corrupt store disables the policy and lets the tool run unrestricted; use a strict read path or have the loader surface lookup uncertainty here.
  - confidence: 0.9
- **[P1] Canonicalize paths before matching envelope globs:** `src/agents/session-runtime-envelope.ts:143-151`
  - body: The evaluator matches raw parameter strings against `allowedPaths` and `deniedPaths`, but filesystem tools will resolve equivalent paths such as `src/../secrets/token.txt` before access. A denied or allowed-path rule can therefore be bypassed by spelling the same target differently; normalize/resolve collected paths using the same base semantics as the tool before comparing.
  - confidence: 0.86
- **[P1] Avoid substring command allowlists:** `src/agents/session-runtime-envelope.ts:102`
  - body: Using `command.includes(pattern)` means an allowlist entry like `pnpm test` also permits `curl https://host && pnpm test` or `pnpm test && curl https://host`. Since this field is presented as an executable command constraint, require anchored/tokenized matches or explicit regex rules that cannot be satisfied by embedding an allowed fragment in a larger command.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff adds a new authorization boundary and still has concrete fail-open and bypass paths in store lookup, path matching, and Bash command matching.

Concerns:

- **[high] Envelope lookup can fail open:** `src/agents/session-runtime-envelope.ts:115`
  - body: Because the underlying session store loader swallows read/parse failures, `readSessionRuntimeEnvelope` can treat an unreadable envelope store as no policy and allow the tool call.
  - confidence: 0.9
- **[high] Path rules are matched before canonicalization:** `src/agents/session-runtime-envelope.ts:143`
  - body: Raw path-string glob matching does not prove the eventual filesystem target is inside the allowed set or outside denied paths, so traversal spellings can bypass the declarative path policy.
  - confidence: 0.86
- **[high] Command allowlist accepts embedded fragments:** `src/agents/session-runtime-envelope.ts:102`
  - body: Substring matching lets an allowed command fragment appear inside a larger command with additional side effects, weakening the Bash envelope into a hint rather than an enforceable command boundary.
  - confidence: 0.88
