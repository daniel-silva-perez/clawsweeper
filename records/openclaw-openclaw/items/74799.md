## Summary

Keep this PR open. The latest branch usefully implements the per-session envelope requested by the paired feature issue, and it appears to have fixed the earlier single-ampersand command allowlist bypass, but the current diff still has a blocking availability regression: a missing session store is treated as an envelope lookup failure and would block all tool calls for fresh/default sessions that do not have any envelope.

## What This Changes

The PR adds an optional Gateway session `envelope` field to `sessions.create` and `sessions.patch`, persists it on `SessionEntry`, enforces tool/path/command constraints in the before-tool-call path, and documents/tests the new protocol surface.

## Best Possible Solution

Keep the branch direction, but change envelope loading so an absent store or missing session entry means no persisted envelope, while malformed/unreadable data for a session that should carry an envelope still fails closed. Add a regression test for a missing session store with a normal tool call, then rerun the focused envelope, before-tool-call, Gateway session, typecheck, and changed-gate checks before maintainer/security review.

## Reproduction Assessment

Yes. The code path is reproducible by invoking any tool with a `ctx.sessionKey` while the resolved session store file does not exist; PR head returns a session-envelope failure from `readSessionStoreStrict`, while current main's session loading treats the missing store as an empty store.

## Solution Assessment

No. The proposed feature shape is reasonable, but the current lookup semantics are too broad for an authorization boundary because they fail closed even when no envelope exists; the safer narrow fix is to reuse current missing-store/no-entry behavior and only enforce when a persisted envelope is actually present or an existing envelope cannot be safely read.

## Review Findings

  - body: The new strict store read returns a blocking failure whenever the resolved session store file is absent.
