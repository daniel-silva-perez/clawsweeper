## What This Changes

The PR adds an optional session `envelope` to Gateway `sessions.create` and `sessions.patch`, persists it on session entries, enforces tool/path/command constraints through the before-tool-call path, and updates protocol docs and focused tests.

## Best Possible Solution

Land the additive session-envelope implementation after owner/security review, a required Unreleased changelog entry, and refreshed validation; keep network constraints and any external authorization callback as explicit follow-up work rather than adding hook-level URL scanning here.

## Reproduction Assessment

Yes. Current main’s Gateway schemas reject unknown `envelope` fields because `sessions.create` and `sessions.patch` omit that field under `additionalProperties: false`; the PR adds focused create/patch/enforcement tests, though they were not run in this read-only review.

## Solution Assessment

Yes, with merge prerequisites. Persisting a declarative session envelope and enforcing it in the existing before-tool-call path is the narrow core seam for tool/path/command constraints, while deferring network controls to sandbox/proxy enforcement is the safer boundary.
