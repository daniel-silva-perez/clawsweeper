## Summary

Keep this PR open. The updated branch implements the intended session-envelope shape and addresses several earlier review concerns, but it still introduces a security-sensitive command-allowlist bypass: a plain allowlist entry can be followed by a single `&` shell operator and an additional command. The PR also references the still-open paired feature issue #74632, so the right path is author fixes plus maintainer/security review, not cleanup closure.

## What This Changes

The PR adds a Gateway `envelope` field to `sessions.create` and `sessions.patch`, persists it on `SessionEntry`, enforces tool/path/command checks in the before-tool-call path, and updates protocol docs plus focused tests.

## Best Possible Solution

Keep the PR open and tighten the command envelope before merge: reject all shell control operators for plain prefix allowlists, including a lone `&`, or replace the string suffix check with the existing shell-command analysis/parsing path. Add a regression test for `pnpm test & curl https://example.com` and rerun the focused envelope, before-tool-call, Gateway session, typecheck, and changed gates before maintainer/security review.

## Reproduction Assessment

Yes. With `bashCommandAllowlist: ["pnpm test"]`, the PR source allows `pnpm test & curl https://example.com` because the command starts with the allowed prefix, has a whitespace boundary, and the suffix does not match `SHELL_CONTROL_PATTERN`; current Bash execution then sends the command through a shell.

## Solution Assessment

No. The PR is directionally useful and already narrows the network-scope issue, but the current command allowlist is not yet the best maintainable solution for an authorization boundary because it relies on incomplete shell metacharacter filtering.
