## Summary

Keep open. Current main still prints a bare dashboard URL from `gateway status`, so the requested behavior is not implemented there, but this PR should not merge as-is because it regresses the current no-token-in-logs invariant and misses current TLS/disabled-Control-UI behavior.

## What This Changes

The PR adds a shared Control UI access URL resolver, wires `gateway status` to use token-fragment dashboard URLs with JSON redaction and SecretRef hints, refactors `openclaw dashboard` to share that resolver, and adds focused tests plus a changelog entry.

## Best Possible Solution

Rework this on top of current main with a shared resolver that preserves `openclaw dashboard` clean logging, threads `tlsEnabled` through both dashboard and status URL generation, skips token/SecretRef resolution when Control UI is disabled, and adds tests for status parity, JSON redaction, TLS, disabled UI, and no token fragments in ordinary command logs.

## Reproduction Assessment

Yes. The PR diff plus current tests give a high-confidence reproduction path: a plain `gateway.auth.token` makes `access.dashboardUrl` include `#token=...`, and the PR then logs that value in `dashboardCommand`, contradicting the current `dashboard.links.test.ts` no-log assertion.

## Solution Assessment

No. The shared-resolver direction is reasonable, but the proposed implementation is not the narrowest maintainable fix until it preserves current secret-exposure and TLS contracts and avoids resolving dashboard access data when the dashboard is disabled.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep dashboard logs free of token fragments:** `src/commands/dashboard.ts:20-21`
  - body: Current main deliberately logs only the clean Control UI URL while sending the tokenized URL to clipboard/browser, and the regression test rejects any `#token=` in `runtime.log`. This refactor makes `dashboardUrl` come from `access.dashboardUrl`, so plain/env token auth would be printed to CLI output and captured logs.
  - confidence: 0.92
- **[P2] Pass TLS state into Control UI access links:** `src/commands/control-ui-access-url.ts:35-40`
  - body: Current main threads `gateway.tls.enabled` into dashboard/status link rendering so TLS gateways get `https://` dashboard URLs. The new resolver calls `resolveControlUiLinks` without `tlsEnabled`, so the shared dashboard/status URL will fall back to `http://` even when the gateway is serving TLS.
  - confidence: 0.88
- **[P2] Skip access URL resolution when the dashboard is disabled:** `src/cli/daemon-cli/status.gather.ts:333-337`
  - body: `gateway status` already prints `Dashboard: disabled` when `gateway.controlUi.enabled === false`, but this code resolves the dashboard access URL unconditionally before printing. That can invoke SecretRef/token providers and add latency or failures for a URL that will not be used.
  - confidence: 0.78
