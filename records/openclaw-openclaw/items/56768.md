## Summary

Keep open. Current main still prints a bare dashboard URL from `gateway status`, so the requested parity behavior is not implemented there, but the PR should not merge as-is because it regresses the current no-token-in-logs invariant and misses current TLS and disabled-Control-UI behavior.

## What This Changes

The PR adds a shared Control UI access URL resolver, wires `gateway status` to print token-fragment dashboard URLs with JSON redaction and SecretRef hints, refactors `openclaw dashboard` to share the resolver, and adds focused tests plus a changelog entry.

## Best Possible Solution

Rework this on top of current main with a shared resolver that preserves clean `openclaw dashboard` logging, threads `tlsEnabled` through dashboard and status URL generation, skips token/SecretRef resolution when Control UI is disabled, and adds coverage for status parity, JSON redaction, TLS, disabled UI, and no token fragments in ordinary command logs. A maintainer should explicitly decide whether human `gateway status` may ever print token-bearing URLs.

## Reproduction Assessment

Yes. The PR diff plus current tests give a high-confidence reproduction path: a plain `gateway.auth.token` makes `access.dashboardUrl` include `#token=...`, and the PR then logs that value in `dashboardCommand`, contradicting the current no-log regression test.

## Solution Assessment

No. The shared-resolver direction is reasonable, but this implementation is not the narrowest maintainable fix until it preserves the current secret-exposure, TLS, and disabled-dashboard contracts.

  - confidence: 0.88

  - body: `gateway status` already prints `Dashboard: disabled` when `gateway.controlUi.enabled === false`, but this code resolves the dashboard access URL unconditionally before printing. That can invoke SecretRef/token providers and add latency or failures for a URL that will not be used.
  - confidence: 0.78
