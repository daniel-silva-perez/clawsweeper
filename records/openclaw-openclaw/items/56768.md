## Summary

Keep open. Current main still prints a bare dashboard URL from `gateway status`, so the PR is not obsolete, but the proposed patch has blocking regressions around token logging, TLS dashboard links, and disabled-Control-UI handling. The review discussion already resolved the runtime-port and SecretRef hint gaps, but the later security and behavior findings still need maintainer attention before merge.

## What This Changes

The PR adds a shared Control UI access URL resolver, wires `openclaw gateway status` to emit token-aware dashboard URLs with JSON redaction and SecretRef hints, refactors `openclaw dashboard` to use that resolver, and adds focused tests plus a changelog entry.

## Best Possible Solution

Rework the shared resolver so status/dashboard URL generation is centralized without changing the existing secret-exposure boundaries: keep ordinary dashboard logs clean, thread TLS state into every rendered Control UI link, avoid token/SecretRef resolution when Control UI is disabled, and add focused tests for status parity, JSON redaction, TLS links, disabled UI, and no token fragments in dashboard logs.

## Reproduction Assessment

Yes. For the PR defects, the path is deterministic: with a plain gateway token, the proposed helper returns `dashboardUrl` with `#token=...`, and the PR logs that value from `dashboardCommand`; the existing dashboard test is the current-main guardrail that would catch the regression.

## Solution Assessment

No. The shared-resolver direction is reasonable, but this implementation is not the best fix until it preserves the current token-log, TLS, and disabled-dashboard contracts and gets maintainer/secops agreement on human status token output.

  - body: Current main logs only the clean Control UI URL while sending the tokenized URL to clipboard/browser, and the regression test rejects any `#token=` in `runtime.log`. This refactor logs `access.dashboardUrl`, so plain/env token auth would be emitted to CLI output and captured logs.

  - body: Current main threads `gateway.tls.enabled` into dashboard/status link rendering so TLS gateways get `https://` dashboard URLs. The new resolver calls `resolveControlUiLinks` without `tlsEnabled`, so shared dashboard/status URLs fall back to `http://` even when the gateway is serving TLS.
  - confidence: 0.88
- **[P2] Skip access URL resolution when dashboard is disabled:** `src/cli/daemon-cli/status.gather.ts:333-337`
  - body: `gateway status` already prints `Dashboard: disabled` when `gateway.controlUi.enabled === false`, but this code resolves dashboard access data unconditionally before printing. That can invoke SecretRef/token providers and add latency or failures for a URL that will not be used.
  - confidence: 0.78
