## Summary

Keep this issue open. Current main still has the reported `gh-issues` `user-invocable` frontmatter key, the skill remains a single 34,800-byte file with no `references/` split, and the skill-creator validator still rejects frontmatter keys outside its allowlist. The description has been tightened since the report, but the central validation and size issues remain unresolved.

## Best Possible Solution

Keep this issue open and fix it with a focused bundled-skill cleanup. The immediate fix is to make the frontmatter contract consistent, most likely by removing the redundant `user-invocable: true` from `skills/gh-issues/SKILL.md` because runtime already defaults it to true, or by deliberately updating the validator and skill-creator guidance to accept the runtime-documented optional keys. The follow-up should split bulky orchestration details into `references/` files and keep only concise triggers and operating constraints in the top-level `SKILL.md`.
