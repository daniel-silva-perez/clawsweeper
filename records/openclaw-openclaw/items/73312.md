## Summary

Keep open. The PR targets a valid Matrix MessagePresentation feature, and the latest head fixes earlier version-order and mediaUrls review comments, but it is still not safe to merge because the new presentation hooks are only added to the Matrix runtime adapter and are not exposed through the registered Matrix channel facade that core delivery actually calls. The new sendPayload path also forwards arbitrary Matrix extraContent into event content, which is security-sensitive once that hook is reachable.

## What This Changes

The PR adds Matrix runtime MessagePresentation rendering, Matrix extraContent forwarding for com.openclaw.presentation, media URL handling tests, and a new Matrix presentation metadata docs page.

## Best Possible Solution

Revise this PR so the Matrix channel facade forwards presentationCapabilities, renderPresentation, and sendPayload, or extend createRuntimeOutboundDelegates to support those hooks; add a delivery or channel-facade regression test proving MessagePresentation reaches Matrix events; validate and whitelist only com.openclaw.presentation metadata for this sendPayload path; and wire the docs/changelog surfaces required for a user-facing Matrix feature.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to trace normal Matrix delivery through src/infra/outbound/deliver.ts into the registered outbound handler from extensions/matrix/src/channel.ts: the PR adds hooks on matrixOutbound, but the registered facade still exposes only sendText, sendMedia, and sendPoll.

## Solution Assessment

No. The runtime renderer and fallback direction are reasonable, but this is not the best mergeable solution until the facade/core delivery path is wired and the new Matrix metadata forwarding contract is constrained to validated presentation content.
