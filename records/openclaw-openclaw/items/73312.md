## Summary

Keep this PR open. Current main does not expose Matrix MessagePresentation metadata: the Matrix outbound adapter and plugin facade still only provide text/media/poll delivery, while the generic channel contract already has the renderPresentation/sendPayload seams this work is trying to use. The PR is not obsolete, but it needs maintainer review/revision before merge because the submitted diff appears to miss the Matrix channel facade and has credible payload/media and extraContent merge risks from the existing review comments and current code.

## Best Possible Solution

Keep this PR open and ask for revisions: wire Matrix presentation support through the public channel outbound facade, declare the appropriate presentation capabilities, preserve mediaUrls using the shared reply-payload helpers or equivalent sequencing, keep the schema version authoritative, and constrain or sanitize Matrix extraContent so only the intended com.openclaw.presentation metadata is attached without overriding core Matrix event fields.

## Risks / Open Questions

- The submitted diff appears incomplete unless it also updates the Matrix channel facade or runtime delegate helper so core can call renderPresentation and sendPayload.
- The sendPayload mediaUrls concern is credible: current generic helpers resolve payload.mediaUrls and mediaUrl together, while the proposed Matrix path described in the review context only forwards the single mediaUrl field.
- Forwarding arbitrary channelData.matrix.extraContent needs careful validation or namespacing because Matrix send currently merges extraContent after base event content, so unsafe keys could override body, msgtype, relation, or mention fields.
- Security/supply-chain review found no dependency, lockfile, workflow, package publishing, script, generated, or vendored-file changes in the provided PR file list; the main security-sensitive area is Matrix event-content forwarding semantics.
