## Summary

Keep open. Current main still does not expose Matrix MessagePresentation metadata, and this active contributor PR is directionally useful, but the latest branch is not merge-ready because normal delivery still cannot reach the new Matrix presentation hooks and the new payload path broadens Matrix event-content forwarding.

## What This Changes

The PR adds Matrix outbound MessagePresentation rendering under `com.openclaw.presentation`, a `sendPayload` path for Matrix metadata/media sends, outbound tests, and a Matrix presentation metadata docs page.

## Best Possible Solution

Revise this PR so the registered Matrix channel facade forwards presentation capabilities, `renderPresentation`, and `sendPayload` to the runtime adapter; validate and forward only the versioned `com.openclaw.presentation` metadata shape; preserve existing reply-to fanout semantics for multi-media payloads; and add targeted Matrix facade/core delivery coverage plus the existing outbound adapter tests.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to follow `deliverOutboundPayloads` into the registered `matrixPlugin.outbound`: the PR adds hooks to the runtime adapter, but the facade that core loads still lacks them, so normal delivery cannot attach Matrix presentation metadata.

## Solution Assessment

No. The runtime renderer and fallback text approach are the right direction, but the current PR is not the narrowest maintainable solution until it wires the facade/core path and constrains Matrix-specific event metadata.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Forward presentation hooks through the Matrix facade:** `extensions/matrix/src/outbound.ts:75-84`
  - body: These hooks are only added to `matrixOutbound`, but core delivery reads `renderPresentation` and `sendPayload` from the registered Matrix channel facade. Since `extensions/matrix/src/channel.ts` still delegates only text, media, and poll methods, normal Matrix replies will never attach `com.openclaw.presentation`; forward these hooks from the facade or extend the shared delegate helper and cover that path.
  - confidence: 0.94
- **[P2] Whitelist Matrix presentation extra content:** `extensions/matrix/src/outbound.ts:65-67`
  - body: `sendPayload` forwards every key from `payload.channelData.matrix.extraContent`, and Matrix sending merges those keys after generated event content. That lets a structured payload override `body`, `msgtype`, relations, or mentions instead of only adding `com.openclaw.presentation`; validate and forward just the presentation metadata this feature owns.
  - confidence: 0.89
- **[P2] Preserve reply-to fanout for media payloads:** `extensions/matrix/src/outbound.ts:105-117`
  - body: The new media loop reuses the same resolved `replyToId` for every media URL and ignores `replyToMode`/`replyToIdSource`. Existing media delivery consumes single-use reply context per media unit, so Matrix `replyToMode: first` presentation sends with multiple attachments would reply every media event instead of only the first; use the shared reply-to fanout helper or equivalent.
  - confidence: 0.82

## Security Review
