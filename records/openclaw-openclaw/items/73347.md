## Summary

Keep this PR open. Current main still reaches the wizard health check with the setup token for the preliminary reachability probe, but then calls healthCommand without a config pinned to that token, so the linked token-drift bug remains possible. The PR is still a focused implementation candidate for open bug #72203.

## Best Possible Solution

Review and land this PR or an equivalent narrow patch that builds a health-check config from nextConfig and pins gateway.auth.token to the setup session token before calling healthCommand. Keep the focused regression test that asserts healthCommand receives the setup token when the saved config token differs, then close #72203 after the fix merges.
