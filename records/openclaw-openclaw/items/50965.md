## Summary

Keep this PR open. Current main still has the cron-remove false-success behavior, and the PR remains a focused but incomplete implementation candidate. The service-layer change in the PR moves in the right direction, and the existing review discussion correctly identifies the remaining CLI exit-status gap, but current main does not already implement the intended fix well enough to close this PR as obsolete.

## Best Possible Solution

Keep this PR open for follow-up, or replace it with a complete fix: make cron service removal return a typed not-found failure without persisting or rearming on a no-op, align `CronRemoveResult` and gateway consumers, make `cron rm/remove/delete` exit non-zero when `ok` or `removed` is false, and add focused service plus CLI regression tests. If name-based removal is desired, implement exact-name resolution deliberately rather than relying on silent id mismatch behavior.

## Risks / Open Questions

- Merging this PR as-is would improve the service payload but still leave `openclaw cron rm/remove/delete` with a false-success exit status for missing jobs.
- Changing the service payload without updating the exported result type, gateway consumers, CLI behavior, and regression tests could leave cron removal semantics inconsistent across surfaces.
- The original user-facing scenario may involve passing a job name where the remove command expects an id; maintainers should decide whether remove should resolve exact names like `cron show` or fail clearly with a non-zero status.
