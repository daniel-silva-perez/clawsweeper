## Summary

Keep open. Current main still lacks the requested staged, provider-generic model migration workflow, and the issue already has maintainer keep-open guidance. Existing code supports immediate primary changes, manual fallback edits, OpenRouter-specific scan/probe behavior, and runtime failover, but not candidate staging, verification-before-promotion, old-primary preservation, or rollback state.

## What This Changes

Request a safer model-switching UX that stages a new model as fallback, verifies it, promotes it to primary, and supports rollback if the new model breaks.

## Best Possible Solution

Add a maintainer-designed `openclaw models` migration flow that stages any provider/model candidate, verifies it against the intended agent surfaces before promotion, preserves the old primary in the fallback chain, stores rollback state, and documents/tests the config transitions.

## Reproduction Assessment

Not applicable as a feature request rather than a bug report. Source and docs inspection provide a high-confidence current-main check: the requested test/migrate/promote/rollback workflow is not present.

## Solution Assessment

Unclear until maintainers define the product contract. The narrowest maintainable direction is a generic `openclaw models` migration workflow built around existing set, fallback, status/probe, scan, and runtime failover primitives, not a small rollback bolt-on to `models set`.
