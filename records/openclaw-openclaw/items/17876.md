## Summary

Keep open. Current main still lacks the requested staged, provider-generic model migration workflow, and the issue discussion already includes explicit maintainer keep-open guidance. The existing implementation covers immediate default-model updates, fallback list editing, OpenRouter-specific scan/probe application, auth probes, and runtime failover, but not candidate staging, verification-before-promotion, old-primary preservation, or rollback state.

## What This Changes

Request a staged model migration UX that can add a candidate as a fallback, verify it, promote it to primary, and roll back to the prior primary if needed.

## Best Possible Solution

Add a maintainer-designed `openclaw models` migration flow that stages any provider/model candidate, verifies it against the intended agent surfaces before promotion, preserves the old primary in the fallback chain, stores rollback state, and documents/tests the resulting config transitions while coordinating the fallback-chain verification work tracked by #6599.

## Reproduction Assessment

Not applicable as a feature request rather than a bug report. Source and docs inspection provide a high-confidence current-main check: the requested test/migrate/promote/rollback workflow is not present.

## Solution Assessment

Unclear until maintainers define the product contract. The narrow maintainable direction is a generic migration workflow built around existing set, fallback, status/probe, scan, and runtime failover primitives, not an OpenRouter-only scan extension or rollback-only tweak to `models set`.
