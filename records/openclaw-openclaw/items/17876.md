## Summary

Keep open. Current main still has immediate default-model setting, manual fallback editing, OpenRouter-specific scan/probe support, and runtime failover, but it does not provide the requested staged, provider-generic model migration flow with verification-before-promotion and rollback. The issue discussion also includes explicit maintainer keep-open guidance, and related #6599 remains open for the narrower fallback-chain test command rather than covering the whole migration workflow.

## What This Changes

Request a staged model migration UX that adds a candidate as fallback, verifies it, promotes it to primary while preserving the old primary as fallback, and supports rollback.

## Best Possible Solution

Add a maintainer-designed `openclaw models` migration workflow that stages any provider/model candidate, verifies it against the intended agent surfaces before promotion, preserves the previous primary in the fallback chain, stores rollback state, and documents/tests the resulting config transitions while coordinating with the narrower fallback-chain verification work tracked by #6599.

## Reproduction Assessment

Not applicable as a feature request rather than a bug report. Source and docs inspection provide a high-confidence current-main check: the requested test/migrate/stage/promote/rollback workflow is not present.

## Solution Assessment

Unclear until maintainers define the product contract. The maintainable direction is a provider-generic migration workflow built around existing set, fallback, status/probe, scan, and runtime failover primitives, not an OpenRouter-only scan extension or rollback-only tweak to `models set`.
