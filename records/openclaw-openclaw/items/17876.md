## Summary

Keep open. Current main still lacks the requested staged, provider-generic model migration workflow, and the issue discussion already includes maintainer keep-open guidance. The existing code supports immediate primary swaps, manual fallback-list edits, OpenRouter-specific scan/probe behavior, and runtime failover, but not candidate staging, verification-before-promotion, old-primary preservation, or rollback state.

## What This Changes

The issue requests a staged model migration workflow for adding, testing, promoting, and rolling back model changes instead of immediately swapping the primary model.

## Best Possible Solution

Add a maintainer-designed `openclaw models` migration flow that stages any provider/model candidate, verifies it before promotion against the intended agent surfaces, preserves the old primary in the fallback chain, stores rollback state, and documents/tests the config transitions.

## Reproduction Assessment

Not applicable as a feature request rather than a bug report. Source and docs inspection provide a high-confidence current-main check: the requested test/migrate/promote/rollback workflow is not present.

## Solution Assessment

Unclear until maintainers define the product contract. The narrowest maintainable direction is not to bolt rollback onto `models set`, but to design a generic `openclaw models` migration workflow around existing set, fallback, status/probe, scan, and runtime failover primitives.
