## Summary

Keep open. Current main has adjacent diagnostic stuck-session warnings, conservative stale-lane release, manual session reset/delete/abort tools, and restart recovery for a narrow lock-marked case, but it still does not implement the requested session-health watchdog that notices repeated inbound messages without a successful reply and automatically clears or prunes only the affected channel session. The open related report #59878 covers a narrower stale-running abort/restart gap and should be coordinated with this work, not used to close it.

## What This Changes

The issue requests a gateway session-health watchdog that logs and automatically clears, prunes, or restarts only the stuck channel session after prolonged unresponsiveness despite incoming messages.

## Best Possible Solution

Add an explicit session-health recovery design that persists enough per-session state to compare inbound activity with successful visible replies, distinguishes slow active work from orphaned or silently failing sessions, logs or notifies before recovery, and invokes the existing targeted reset/prune primitive for only the affected channel session. The implementation should preserve routing metadata where possible and align with the narrower stale-running recovery tracked by #59878.

## Reproduction Assessment

Unclear. The incident report and corroborating comment are credible, and static inspection shows the missing mechanism, but a high-confidence live reproduction would need a captured corrupted session or a focused harness that simulates repeated inbound turns with no successful assistant final and verifies that current main leaves the session intact.

## Solution Assessment

Unclear. A health watchdog is the right product direction, but automatic clearing is not yet proven to be the safest default; maintainers should first choose the threshold, prune-versus-reset behavior, active-work handling, notification policy, and how it composes with existing diagnostic recovery.
