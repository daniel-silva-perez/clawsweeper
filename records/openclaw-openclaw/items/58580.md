## Summary

Keep this issue open. Current main has useful adjacent work, but it does not implement the requested session health check: there is no per-session tracking of received-but-unanswered messages, no configurable `sessionHealth` policy, and no automatic targeted reset/prune when a session keeps failing after inbound traffic. Startup recovery only handles marked `running` main sessions tied to stale transcript locks, and channel health monitoring restarts channel transports rather than clearing a corrupted conversation session. The related #59878 remains open for a narrower stale-running-lane case and does not supersede this broader auto-clear request.

## Best Possible Solution

Keep this issue open and implement a generic gateway/session health supervisor: persist per-session inbound attempt timestamps, last successful reply/delivery timestamps, consecutive failed or unanswered counts, and active-run state; expose a conservative configurable policy; log before action; then call the existing targeted reset/prune path for only the affected session when the threshold is exceeded. Related #59878 should continue to cover stale persisted `running` rows, while this issue should track the broader auto-clear-after-unresponsiveness behavior.

## Risks / Open Questions

- A future implementation needs careful false-positive handling so genuinely slow active runs are not reset while still in flight.
- The design should distinguish session corruption from channel transport failures; current channel health restart logic is adjacent but should not be treated as the session-level fix.
