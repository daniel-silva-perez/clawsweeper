## Summary

Keep open. Current main has adjacent stuck-session diagnostics and queue/lane recovery, but it does not implement the requested persisted-session auto-clear that would rotate the affected channel session, clear corrupted model/token state, and start fresh after repeated unanswered inbound messages.

## What This Changes

Add a gateway session health check that detects prolonged unresponsive sessions, logs a warning, and automatically resets only the stuck channel session.

## Best Possible Solution

Add an explicit, configurable gateway session-health policy that tracks inbound attempts and successful outbound replies per session, warns first, then uses the existing reset service to rotate only the affected session after active work is safely aborted, drained, or proven absent. It should preserve delivery metadata, clear auto-pinned model/auth/runtime counters, document the policy, and include focused tests for false-positive slow runs, queued messages, and corrupted persisted state.

## Reproduction Assessment

No: the original incident does not include exact corrupted session state, provider errors, logs, or versions for an end-to-end replay. Source inspection does give high confidence that current main lacks the requested persisted-session auto-clear path.

## Solution Assessment

Unclear: the requested direction is reasonable, but destructive automatic reset needs maintainer-approved threshold/action semantics. The narrowest maintainable implementation would reuse `performGatewaySessionReset` behind a configurable health policy rather than expanding the existing diagnostics lane reset blindly.
