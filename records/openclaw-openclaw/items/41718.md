## Summary

Keep open. Current `main` still clears the Gateway Token when the Overview WebSocket URL input changes, so this PR's stated UI problem is not implemented on main. The PR also should not land as-is because preserving the current token while changing `gatewayUrl` can carry a gateway-scoped credential to a different endpoint and persist it under the edited URL scope. The unrelated memory-lancedb retry hunk is already covered more completely on current `main`.

## Best Possible Solution

Keep this PR open for maintainer rework or replacement in the #41545 fix cluster. The best path is to drop the already-superseded memory-lancedb hunk, then fix the Overview editing UX without breaking per-gateway credential isolation: clear or reload the token when the normalized endpoint actually changes, or require explicit confirmation before reusing a token across gateways. Add focused regression coverage for both in-progress URL editing ergonomics and cross-gateway token isolation, and coordinate with open PRs #41546 and #42001.

## Risks / Open Questions

- Landing the UI hunk as-is can reuse a token entered for one gateway endpoint against another edited `gatewayUrl` when the user clicks Connect.
- Because settings changes are saved immediately, preserving the token while typing a different URL can persist the old token under intermediate or new gateway URL scopes in sessionStorage.
- The PR mixes the UI credential-handling change with an unrelated memory-lancedb retry hunk that current `main` already covers, making review and landing harder than necessary.
- The current checkout has shallow/grafted snapshot history, so owner provenance is less precise than a full upstream history search.
