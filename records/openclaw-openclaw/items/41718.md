## Summary

Keep this PR open. Current main still has the reported Overview behavior, but the PR should not be closed as implemented or landed as-is: its UI hunk preserves a gateway token across endpoint edits, and its separate memory-lancedb hunk is unrelated to the stated UI fix and already covered more completely on main.

## Best Possible Solution

Keep this PR open as a rework candidate for #41545. The best path is to split or drop the memory-lancedb hunk, then implement a Control UI fix that avoids clearing the token on harmless in-progress URL edits while still clearing, reloading, or explicitly confirming credentials when the normalized selected gateway actually changes. Add focused regression coverage for both the editing ergonomics and per-gateway token isolation, and coordinate with the overlapping open PRs #41546 and #42001.

## Risks / Open Questions

- Landing the UI hunk as-is can send a token intended for one gateway to a different edited WebSocket endpoint on Connect.
- Landing the UI hunk as-is can persist the old token under newly typed gateway URL scopes, weakening the documented per-gateway token boundary.
- The PR bundles an unrelated memory-lancedb retry change, and current main already handles that retry path more completely.
- There are overlapping open implementation candidates for the same linked bug, so maintainers should choose or request one canonical security-aware fix rather than merging duplicate partial fixes.
