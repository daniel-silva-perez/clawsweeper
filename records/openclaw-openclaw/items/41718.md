## Summary

Keep this PR open. Current main still clears the Gateway Token during Overview WebSocket URL edits, so the intended UI bug is not implemented on main. The PR should not land as-is because it preserves a token across gateway endpoint changes, which can send and persist credentials under a different gateway URL, and its unrelated memory-lancedb retry hunk is already covered more completely on current main.

## Best Possible Solution

Keep this PR open as a rework candidate for #41545. The best path is to split or drop the memory-lancedb hunk, then implement a security-aware Control UI fix that avoids clearing the token on harmless in-progress edits while still clearing, reloading, or confirming credentials when the normalized selected gateway actually changes. Add focused regression coverage for both the editing ergonomics and the per-gateway token isolation boundary, and coordinate with the overlapping open PRs #41546 and #42001.

## Risks / Open Questions

- Landing the UI hunk as-is can send a token intended for one gateway to a different edited WebSocket endpoint on Connect.
- Landing the UI hunk as-is can persist the old token under newly typed gateway URL scopes, weakening the documented per-gateway token boundary.
- The PR bundles an unrelated memory-lancedb change that current main already handles more completely, increasing review and merge risk without helping the linked UI bug.
- There are overlapping open implementation candidates for #41545, so maintainers should choose or request one canonical rework instead of landing duplicate fixes.
