## Summary

Current main still has the reported Overview behavior, so this PR is not obsolete or implemented. It should stay open as a rework candidate, but the submitted UI change is not safe to land as-is because it preserves a gateway token while the endpoint is edited, conflicting with the documented/tested per-gateway token boundary. The PR also bundles an unrelated memory-lancedb retry hunk that current main now handles more completely.

## Best Possible Solution

Keep this PR open for rework against #41545. The best path is to split or drop the memory-lancedb hunk, then implement a Control UI fix that preserves editing ergonomics without carrying credentials across a real normalized gateway change: clear, reload the token for the selected gateway scope, or require explicit confirmation when the endpoint changes. Add focused regression coverage for both in-progress URL editing and per-gateway token isolation, and coordinate with the overlapping open candidates #41546 and #42001.

## Risks / Open Questions

- Landing the UI hunk as-is can send a token intended for one gateway to a different edited WebSocket endpoint on the next Connect.
- Landing the UI hunk as-is can persist the old token under newly typed gateway URL scopes, weakening the documented per-gateway token boundary.
- The PR bundles an unrelated memory-lancedb retry change that current main already covers with a broader loader/init retry path and tests.
