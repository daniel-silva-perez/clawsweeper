## Summary

Keep this PR open for maintainer follow-up. Current main still has the token-clearing WebSocket URL input behavior, so the intended UI bug fix has not shipped. The PR is not ready to close as implemented or land as-is because its UI hunk preserves credentials across gateway URL scopes, and its unrelated memory-lancedb hunk is already handled more completely on main.

## Best Possible Solution

Keep this PR open only as an implementation candidate and require rework before landing: split or drop the memory-lancedb hunk, fix URL-editing ergonomics without carrying credentials across gateway scopes, and add focused regression coverage. A good product direction is a draft URL or confirmation flow that avoids clearing the token on harmless in-progress edits but clears or reloads the token when the normalized selected gateway actually changes, coordinated with #41545 and the other open implementation candidates.

## Risks / Open Questions

- Security: preserving props.settings.token while gatewayUrl changes can send a token intended for one gateway to another WebSocket endpoint on Connect.
- Storage regression: preserving settings.token during gatewayUrl edits can persist the old token under newly typed gateway URL scopes, weakening per-gateway token isolation.
- Scope risk: the PR bundles an unrelated memory-lancedb retry change even though current main already has broader retry handling and coverage for that area.
