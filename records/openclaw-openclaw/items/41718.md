## Summary

Keep this PR open for maintainer follow-up. Current main still has the WebSocket URL edit behavior the PR is trying to improve, so the PR is not obsolete or implemented. It should not land as-is because the UI hunk preserves a gateway token across edited gateway URLs, while the unrelated memory-lancedb hunk is already covered more completely on main.

## Best Possible Solution

Keep this PR open only as a rework candidate for #41545. The best path is to split or drop the memory-lancedb hunk, then implement a security-aware UI fix that avoids clearing the token on harmless in-progress edits while still clearing, reloading, or confirming credentials when the normalized selected gateway actually changes. Add focused Control UI regression coverage for both the editing ergonomics and the per-gateway token isolation boundary.

## Risks / Open Questions

- Landing the UI hunk as-is can send a token intended for one gateway to a different edited WebSocket endpoint on Connect.
- Landing the UI hunk as-is can persist the old token under newly typed gateway URL scopes, weakening the documented per-gateway token boundary.
- The PR bundles an unrelated memory-lancedb retry change that current main already handles more completely, increasing review and merge risk without helping the linked UI bug.
