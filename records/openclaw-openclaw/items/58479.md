## Summary

Keep open. Current main contains adjacent approval-flow fixes and tests, and the latest reporter repro no longer shows the original “approved but exec never resumes” failure. However, a maintainer explicitly kept this issue open pending a fresh current-main or next-release repro, and the latest repro was on 2026.4.24 rather than current main. The remaining observed `allow-always` behavior used `tools.exec.strictInlineEval=true`, where docs say inline eval should keep prompting.

## Best Possible Solution

Keep this issue open for one focused current-main or v2026.4.26+ verification pass: test `allow-once`, non-inline `allow-always` durability, and strict inline-eval prompting separately on the affected surface. If only strict inline-eval re-prompts, close this as implemented/expected and point to the docs; if WebChat policy still blocks exec before approval, handle that in #72858; if approval resolves but the waiting exec still does not resume, add a targeted regression test around that exact surface path.
