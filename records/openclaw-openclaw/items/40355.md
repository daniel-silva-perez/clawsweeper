## Summary

Keep open. Current main still lacks the requested generic doctor warning for configured default and per-agent model refs, and the proposed patch remains useful but has blocking correctness issues around catalog provenance and runtime-equivalent model resolution, plus a low-severity terminal-output hygiene issue.

## What This Changes

This PR adds a config-validation helper, tests, and a doctor health contribution to warn when `agents.defaults.model.primary` or `agents.list[].model` is absent from the loaded model catalog.

## Best Possible Solution

Land a focused doctor or shared validation path that resolves configured model refs with the same alias/provider semantics as runtime paths, distinguishes provider/runtime/static catalog evidence from user-declared config rows when warning about unsupported models, sanitizes all displayed warning text, and coordinates semantics with the related config-set warning work in #47318.

## Reproduction Assessment

Yes. The PR findings reproduce by deterministic source inspection: `loadModelCatalog({ config })` appends configured provider rows, current runtime resolution uses an alias index, and the PR diff sends unsanitized config-derived strings to `note()`. No live provider call is required.

## Solution Assessment

No. Warning through doctor is a reasonable direction, but this implementation is not the best fix until it matches runtime alias resolution, avoids treating user-declared catalog rows as provider proof, and sanitizes terminal output.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Validate against non-config catalog rows:** `src/flows/doctor-health-contributions.ts:407`
  - body: The new doctor check passes `loadModelCatalog({ config: ctx.cfg })` into the warning helper, but that catalog appends `models.providers.*.models` from the same config. In the linked reproduction, the bogus model is first added there, so this can treat it as present and emit no warning even though runtime/provider calls can still fail.
  - confidence: 0.88
- **[P2] Resolve aliases before checking primaries:** `src/config/validation.ts:1341`
  - body: `collectModelConfigWarnings` calls `resolveModelRefFromString` without `cfg` or an `aliasIndex`, so a valid alias such as `fast` is parsed as `anthropic/fast` and warned as missing. Runtime and model-picker paths resolve configured aliases first, so doctor would produce false positives for supported configs.
  - confidence: 0.9
- **[P3] Sanitize model warning text before printing:** `src/flows/doctor-health-contributions.ts:411`
  - body: The warning message includes model identifiers derived from local config and is passed straight to `note()`. A model id containing ANSI or control characters can spoof doctor output; route the path and message through the existing terminal sanitizer before display.
  - confidence: 0.82
