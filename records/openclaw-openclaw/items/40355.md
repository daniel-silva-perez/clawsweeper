## Summary

Keep open. Current main still lacks the intended generic doctor/config validation for default and per-agent primary model refs, and this contributor PR is the active implementation path. The proposed patch should not merge as-is because its catalog source can mask the configured-provider reproduction, and its resolver/output handling diverges from current model-selection and terminal-safety contracts.

## What This Changes

The PR adds a config-validation helper, a doctor health contribution, and tests to warn when `agents.defaults.model.primary` or `agents.list[].model` points to a model missing from the loaded catalog.

## Best Possible Solution

Land a focused doctor or shared validation path that resolves model refs the same way runtime/model-picker paths do, distinguishes provider/runtime/static catalog rows from user-declared config rows when warning about unsupported models, keeps catalog inspection read-only where possible, sanitizes terminal output, and covers aliases plus the `models.providers` misconfiguration from the linked report. Leave broader fallback-user-notification behavior to its separate tracking item.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Validate against catalog rows that do not come from config:** `src/flows/doctor-health-contributions.ts:407`
  - body: The new doctor check passes `loadModelCatalog({ config: ctx.cfg })` into the warning helper, but that catalog appends `models.providers.*.models` from the same config. In the linked #39811 reproduction, the bogus model is first added under `models.providers.openai.models`, so it will be present in this catalog and no warning is emitted even though runtime/provider calls can still fail. Use a catalog/source that distinguishes provider-discovered rows from user-declared config rows, or carry origin metadata into the check.
  - confidence: 0.88
- **[P2] Resolve aliases before checking configured primaries:** `src/config/validation.ts:1341`
  - body: `collectModelConfigWarnings` calls `resolveModelRefFromString` without `cfg` or an alias index, so a valid configured alias such as `fast` is parsed as `anthropic/fast` and warned as missing. Runtime and model-picker paths build `agents.defaults.models` aliases before resolving, so doctor would produce false positives for supported configs.
  - confidence: 0.9
- **[P3] Sanitize model warning text before printing it:** `src/flows/doctor-health-contributions.ts:411`
  - body: The warning message includes model identifiers derived from local config and is passed straight to `note()`. A model id containing ANSI or control characters can spoof doctor output; route the path/message through the existing terminal sanitizers before display.
  - confidence: 0.82
