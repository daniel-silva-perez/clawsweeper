## Summary

Keep open. Current main still lacks a generic doctor/config warning for invalid default or per-agent model primaries, so the contribution targets a real gap, but the current PR branch has merge-blocking correctness issues around catalog source semantics and alias resolution.

## What This Changes

The PR adds a config validation helper, a doctor health contribution, and tests to warn when configured default or per-agent primary model refs are missing from the loaded model catalog.

## Best Possible Solution

Land a focused doctor/config diagnostic that resolves model refs the same way runtime and model-picker paths do, distinguishes provider/runtime/static catalog evidence from user-declared config rows, sanitizes terminal output, and covers the linked invalid-model configuration path without changing runtime failover behavior in this PR.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Validate against provider-backed catalog entries:** `src/flows/doctor-health-contributions.ts:407`
  - body: The new doctor check passes `loadModelCatalog({ config: ctx.cfg })`, but that catalog appends rows from `models.providers.*.models` in the same config. In the linked misconfiguration, the bogus model is declared there, so this can treat it as present and emit no warning. Use source/origin metadata or a provider/runtime/static-only catalog for this diagnostic.
  - confidence: 0.88
- **[P2] Resolve configured aliases before checking primaries:** `src/config/validation.ts:1342`
  - body: `collectModelConfigWarnings` calls `resolveModelRefFromString` without `cfg` or an alias index, so a valid alias such as `fast` from `agents.defaults.models` is parsed as `anthropic/fast` and warned as missing. Doctor should match runtime/model-picker resolution before checking the catalog.
  - confidence: 0.9
- **[P3] Sanitize model warning text before printing it:** `src/flows/doctor-health-contributions.ts:410`
  - body: The warning path and message include model identifiers derived from local config and are passed directly to `note()`. A model id with ANSI/control characters can spoof or obscure doctor output; route this through the existing terminal sanitizer or config issue formatter.
  - confidence: 0.82
