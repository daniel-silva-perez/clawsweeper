## Summary

Keep open. The underlying UX problem is still present on current main, and this PR is useful source work, but the current PR head is not merge-ready because its doctor check can miss the linked configured-provider reproduction, can warn on valid aliases, and prints config-derived warning text without terminal sanitization.

## What This Changes

The PR adds a config-validation helper, tests, and a doctor health contribution to warn when configured default or per-agent primary model refs are absent from the loaded model catalog.

## Best Possible Solution

Land a focused doctor or shared validation path that resolves model refs the same way runtime/model-picker paths do, distinguishes provider/runtime/static catalog evidence from user-declared config rows, sanitizes all displayed warning text, and covers the #39811 configured-provider case plus alias regression tests. Coordinate the surface with #47318 so config-set warnings and doctor warnings share semantics rather than diverging.

## Reproduction Assessment

Yes. The PR findings reproduce by inspection against current main and the PR diff: loadModelCatalog({ config }) appends configured provider rows, and resolveModelRefFromString only honors aliases when given an alias index. No live provider call is required for these deterministic contract checks.

## Solution Assessment

No. The warning-through-doctor direction is appropriate, but this implementation is not the narrowest maintainable fix until it matches runtime alias resolution, avoids trusting user-declared catalog rows as provider proof, and sanitizes terminal output.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Validate against catalog rows that do not come from config:** `src/flows/doctor-health-contributions.ts:407`
  - body: The new doctor check passes `loadModelCatalog({ config: ctx.cfg })` into the warning helper, but that catalog appends `models.providers.*.models` from the same config. In the linked #39811 reproduction, the bogus model is first added there, so this can treat it as present and emit no warning even though runtime/provider calls can still fail.
  - confidence: 0.88
- **[P2] Resolve aliases before checking configured primaries:** `src/config/validation.ts:1344`
  - body: `collectModelConfigWarnings` calls `resolveModelRefFromString` without `cfg` or an alias index, so a valid configured alias such as `fast` is parsed as `anthropic/fast` and warned as missing. Runtime and model-picker paths resolve configured aliases first, so doctor would produce false positives for supported configs.
  - confidence: 0.9
- **[P3] Sanitize model warning text before printing it:** `src/flows/doctor-health-contributions.ts:411`
  - body: The warning message includes model identifiers derived from local config and is passed straight to `note()`. A model id containing ANSI or control characters can spoof doctor output; route the path and message through the existing terminal sanitizer before display.
  - confidence: 0.82
