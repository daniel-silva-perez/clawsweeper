## Summary

Keep open. Current main still does not provide the PR's intended generic doctor warning for `agents.defaults.model` or `agents.list[].model`, but the proposed patch has concrete correctness issues around catalog source semantics and alias resolution, plus a small terminal-output sanitization issue.

## What This Changes

The PR adds a `collectModelConfigWarnings` helper, a `doctor:model-config` health contribution, and tests to warn when configured default or per-agent primary model refs are missing from the loaded model catalog.

## Best Possible Solution

Land a focused validation path that resolves configured model refs with runtime-equivalent alias/provider semantics, distinguishes provider/runtime/static catalog evidence from user-declared config rows, reports through doctor with sanitized terminal output, and leaves broader fallback-notification behavior to its own tracked work.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Validate against non-config catalog rows:** `src/flows/doctor-health-contributions.ts:407`
  - body: The new doctor check passes `loadModelCatalog({ config: ctx.cfg })`, but that catalog appends `models.providers.*.models` from the same config. In linked #39811, the bogus model is first declared there, so this check can treat the bad ref as catalog-present and emit no warning. Use catalog origin/source metadata or a provider/runtime/static-only catalog for this diagnostic.
  - confidence: 0.88
- **[P2] Resolve aliases before warning about primary models:** `src/config/validation.ts:1341`
  - body: `collectModelConfigWarnings` resolves strings without `cfg` or an alias index, so a valid configured alias such as `fast` is parsed as `anthropic/fast` and warned as missing. Doctor should match runtime/model-picker semantics by resolving through `agents.defaults.models` aliases before checking the catalog.
  - confidence: 0.9
- **[P3] Sanitize model warnings before printing:** `src/flows/doctor-health-contributions.ts:411`
  - body: The warning path and message include model refs derived from local config and are passed directly to `note()`. A model id containing ANSI or control characters can spoof or obscure doctor output, so route the rendered strings through the existing terminal sanitizer before display.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The patch does not add dependencies, workflows, or package-resolution changes, but it prints config-derived model warning text without terminal sanitization.

Concerns:

- **[low] Unsanitized doctor warning output:** `src/flows/doctor-health-contributions.ts:411`
  - body: Model ids and warning paths from local config are interpolated into `note()` output without `sanitizeTerminalText`, allowing control characters in config to spoof or obscure doctor output. This is local-config scoped but should be fixed before merge.
