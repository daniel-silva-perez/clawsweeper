## Summary

Keep open. The PR targets a real gap that is still present on current main, but the current diff is not merge-ready because it validates against a catalog that can include the same user-declared config rows, misses runtime alias semantics, and prints config-derived warning text without terminal sanitization.

## What This Changes

The branch adds a config-validation helper, tests, and a doctor health contribution to warn when configured default or per-agent model primaries are missing from the loaded model catalog.

## Best Possible Solution

Land a focused doctor validation path that resolves model refs with the same alias/provider semantics as runtime model selection, distinguishes provider/runtime/static catalog evidence from user-declared config rows, sanitizes displayed warning text, and adds regression coverage for aliases plus the models.providers misconfiguration from #39811.

## Reproduction Assessment

Yes. The current problems reproduce by deterministic source inspection: current main lacks the generic doctor check, the PR loads a catalog that appends config-declared provider rows, runtime resolution uses an alias index, and the PR prints config-derived warning text directly.

## Solution Assessment

No. Warning through doctor is a reasonable direction, but this implementation is not the best fix until it matches runtime alias resolution, avoids self-validating against config-provided catalog rows, and sanitizes warning output.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Validate against catalog rows that do not come from config:** `src/flows/doctor-health-contributions.ts:407`
  - body: The new doctor check passes `loadModelCatalog({ config: ctx.cfg })` into the warning helper, but that catalog appends `models.providers.*.models` from the same config. In the linked #39811 shape, a bogus model first added there can appear present and suppress the warning even though provider/runtime calls still fail.
  - confidence: 0.88
- **[P2] Resolve aliases before checking configured primaries:** `src/config/validation.ts:1341`
  - body: `collectModelConfigWarnings` calls `resolveModelRefFromString` without `cfg` or an alias index, so a valid configured alias such as `fast` is parsed as a default-provider model and warned as missing. Runtime and model-picker paths resolve `agents.defaults.models` aliases first, so doctor would produce false positives for supported configs.
  - confidence: 0.9
- **[P3] Sanitize model warning text before printing it:** `src/flows/doctor-health-contributions.ts:411`
  - body: The warning message includes model identifiers derived from local config and is passed straight to `note()`. A model id containing ANSI or control characters can spoof doctor output; route the path and message through the existing terminal sanitizer before display.
  - confidence: 0.82
