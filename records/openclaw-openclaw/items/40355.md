## Summary

Keep open. Current main still lacks the generic doctor/config warning this PR is trying to add, but the proposed patch does not yet match current model resolution and catalog semantics, and it introduces unsanitized config-derived terminal output.

## What This Changes

The PR adds a config-validation helper, a doctor health contribution, and tests to warn when default or per-agent primary model refs are absent from the loaded model catalog.

## Best Possible Solution

Land a focused doctor/config diagnostic that resolves model refs through the same alias-aware path as runtime/model-picker code, validates against provider/static/runtime catalog evidence rather than user-declared config rows, sanitizes warning output, and adds regression coverage for #39811 plus alias-based configs. Coordinate with #47318 and #53590 so config-set and post-update warnings do not diverge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Validate against catalog rows that do not come from config:** `src/flows/doctor-health-contributions.ts:407`
  - body: The new doctor check passes loadModelCatalog({ config: ctx.cfg }) into the warning helper, but that catalog appends models.providers.*.models from the same config. In the linked #39811 reproduction, the bogus model is first added there, so this can treat it as present and emit no warning even though provider/runtime calls can still fail.
  - confidence: 0.88
- **[P2] Resolve aliases before checking configured primaries:** `src/config/validation.ts:1341`
  - body: collectModelConfigWarnings calls resolveModelRefFromString without cfg or an aliasIndex, so a valid configured alias such as fast is parsed under the default provider and warned as missing. Doctor should use the same alias-aware resolution path as runtime/model-picker behavior before checking catalog membership.
  - confidence: 0.9
- **[P3] Sanitize model warning text before printing it:** `src/flows/doctor-health-contributions.ts:411`
  - body: The warning path and message include model identifiers derived from local config and are passed directly to note(). Route them through sanitizeTerminalText or the existing config issue formatter so control characters cannot spoof or obscure doctor output.
  - confidence: 0.82
