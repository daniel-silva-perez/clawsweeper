## Summary

Keep open. Current main still separates embedded-run `livenessState` from user-facing reply payloads: the state is present in run metadata, lifecycle events, and `agent.wait`, but `ReplyPayload`, runtime payloads, outbound JSON projection, and channel/CLI text rendering still do not carry a stable blocked/abandoned signal in `payloads[]`.

## What This Changes

Request to surface embedded-run `meta.livenessState` for blocked or abandoned exits in final user-facing reply payloads so CLI and channel renderers do not silently drop the terminal state.

## Best Possible Solution

Add a generic payload surfacing layer for terminal embedded runs: when final `meta.livenessState` is `blocked` or `abandoned`, the final visible payload should carry a stable user-readable signal, with tests proving both the strict-agentic blocked path and an abandoned incomplete-turn path reach at least CLI/auto-reply delivery. A future structured field can be added separately if maintainers want a channel-aware envelope contract.
