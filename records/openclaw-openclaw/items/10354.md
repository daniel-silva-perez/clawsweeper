## Summary

Keep open. Current main still exposes the shared message tool `channel` parameter as an undescribed optional string, and the latest release has the same gap. The prompt and runtime fallback paths help adjacent routing failures, but they do not add the requested agent-visible schema guidance, and both related implementation PRs are closed unmerged.

## What This Changes

The issue requests adding clearer schema metadata, and possibly enum guidance, for the message tool `channel` parameter so agents pass a provider or channel plugin id rather than a destination conversation id.

## Best Possible Solution

Add a narrow core schema update that keeps `channel` optional, describes it as a messaging provider or channel plugin id such as `slack` or `discord` rather than a destination target, and covers that metadata with a focused message-tool schema test. Defer enum support unless it can be generated plugin-aware without blocking third-party channels.
