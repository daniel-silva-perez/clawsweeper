## Summary

Keep open. Current main still exposes the shared message tool `channel` parameter as an undescribed optional string, and the latest release has the same gap. Runtime fallback handles some bad explicit channel values when a valid current provider exists, but it does not provide the requested agent-visible schema guidance and related implementation PRs are closed unmerged.

## What This Changes

The issue requests adding a description, and possibly a plugin-aware enum, to the shared message tool `channel` parameter so agents pass provider/plugin ids rather than destination conversation ids.

## Best Possible Solution

Land a narrow core schema/test update that keeps `channel` optional, adds TypeBox description text explaining it expects a messaging provider or channel plugin id such as `slack` or `discord` rather than a destination target, and defers enum support unless it can be generated from plugin-aware channel discovery.

## Reproduction Assessment

Yes. A high-confidence reproduction is to inspect or instantiate the current message tool schema and check `parameters.properties.channel`; current main and `v2026.4.27` expose it as an optional string without description or enum.

## Solution Assessment

Yes. The narrowest maintainable fix is a schema-description assertion in `message-tool.test.ts` plus the TypeBox description in `message-tool.ts`; enum support should only be added if it is dynamic and plugin-aware.
