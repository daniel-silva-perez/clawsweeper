## Summary

Keep open. Current main and the latest stable release still expose the shared message tool `channel` parameter as an undescribed optional string, while the existing prompt guidance and runtime fallback do not add the requested agent-visible schema metadata. The linked implementation PRs are closed unmerged, so this remains a valid narrow repair.

## What This Changes

Add guidance, and possibly a plugin-aware enum, to the message tool `channel` schema so agents pass a provider or channel plugin id rather than a destination conversation/channel/user id.

## Best Possible Solution

Land a narrow core update that keeps `channel` optional, adds TypeBox description text explaining that it expects a messaging provider or channel plugin id such as `slack` or `discord` rather than a destination id, and adds a focused schema assertion. Defer enum support unless it is dynamically derived from registered channel/plugin values.

## Reproduction Assessment

Yes. A high-confidence reproduction is source or schema inspection on current main: `createMessageTool()` uses `buildRoutingSchema()`, whose `channel` property has no description or enum metadata.

## Solution Assessment

Yes for the description portion of the request. The narrowest maintainable fix is schema metadata plus a focused test; a hard-coded enum is not the best path unless it is generated from plugin-aware channel values.
