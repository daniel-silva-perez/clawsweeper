## Summary

Keep open. Current main still defines the shared message tool `channel` parameter as a bare optional string with no schema description or enum, while the related implementation PRs are closed unmerged; this remains a valid narrow schema/test repair.

## What This Changes

The issue requests adding schema guidance, and possibly enum support, for the message tool `channel` parameter so agents pass a provider or channel plugin id instead of a destination conversation/channel/user id.

## Best Possible Solution

Land a narrow core update that keeps `channel` optional, adds a TypeBox description explaining that it expects a messaging provider or channel plugin id such as `slack` or `discord`, and adds a focused schema assertion. Treat enum support as a follow-up unless it can be derived from the same plugin-aware channel values used at runtime.

## Reproduction Assessment

Yes. A high-confidence reproduction is source/schema inspection on current main: `createMessageTool()` exposes `buildRoutingSchema()`, and its `channel` property has no `description` or `enum` metadata.

## Solution Assessment

Yes for the description portion. The narrowest maintainable fix is schema metadata plus a focused test; a hard-coded static enum is not the best path unless it is derived from registered channel/plugin values.
