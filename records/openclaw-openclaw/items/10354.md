## Summary

Keep open. Current main and the latest stable tag still expose the shared message tool `channel` parameter as an optional string without schema guidance, while the linked implementation PRs are closed unmerged; this remains a narrow, valid repair.

## What This Changes

Add agent-visible schema metadata to the message tool `channel` parameter clarifying that it expects a messaging provider or channel plugin id, not a destination channel/conversation/user id, with enum support only if plugin-aware.

## Best Possible Solution

Land a narrow core schema/test update that keeps `channel` optional, adds a TypeBox description saying it expects a messaging provider or channel plugin id such as `slack` or `discord` rather than a destination id, and adds a focused assertion. Treat enum support as a follow-up unless it is derived from plugin-aware channel values.

## Reproduction Assessment

Yes. A high-confidence reproduction is source/schema inspection on current main: `createMessageTool()` exposes `buildRoutingSchema()`, whose `channel` property has no description or enum metadata.

## Solution Assessment

Yes for the description portion. The narrowest maintainable fix is schema metadata plus a focused test; a hard-coded static enum is not the best path unless it is derived from registered channel/plugin values.
