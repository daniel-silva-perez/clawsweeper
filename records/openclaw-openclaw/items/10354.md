## Summary

Keep open. Current main still exposes the shared message tool `channel` parameter as an undescribed optional string, so the requested schema guidance is not implemented; runtime prompt text and fallback routing help adjacent cases but do not fix the tool schema metadata, and the two related implementation PRs are closed unmerged.

## What This Changes

Add tool-schema guidance, and possibly a dynamic enum, so the message tool `channel` parameter communicates that it expects a messaging provider or channel plugin id rather than a destination conversation/channel id.

## Best Possible Solution

The message tool should keep `channel` optional, describe it as the messaging provider or channel plugin id such as `slack` or `discord`, clearly distinguish it from destination targets, and add a focused schema assertion; enum support should only be added if it can include dynamic plugin ids without blocking third-party channels.

Do not change outbound routing behavior, provider normalization, or Slack-specific target parsing for this narrow fix. No changelog entry is expected for a schema/test-only metadata repair unless the maintainer wants a user-facing note. Related refs: this issue, #10364, #13584, #10668, #4751.
