## Summary

Keep open. Current main still exposes the shared message tool's `channel` parameter as a bare optional string, so the requested schema guidance is not implemented. Prompt guidance and runtime fallback reduce some failures, but they do not add the requested tool-schema metadata, and the two linked implementation PRs are closed unmerged.

## What This Changes

The issue requests clearer schema metadata for the message tool's optional `channel` parameter so agents pass a provider/plugin id like `slack` or `discord`, not a destination channel/user id.

## Best Possible Solution

Land a narrow core update that keeps `channel` optional, adds a TypeBox description explaining it expects a messaging provider or channel plugin id rather than a destination conversation/channel/user id, and adds a focused schema assertion. Defer enum support unless it can be generated from the registered/configured channel values without excluding plugins.
