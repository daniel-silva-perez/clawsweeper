## Summary

Keep open. Current main and the latest stable release still expose the shared message tool `channel` parameter as an undescribed optional string, while the existing prompt guidance and runtime fallback do not provide the requested agent-visible schema metadata. The two linked implementation PRs are closed unmerged, so this remains a narrow valid repair candidate.

## What This Changes

The issue requests adding a description, and possibly a safe enum, to the shared message tool `channel` parameter so agents pass provider or channel plugin ids instead of destination conversation ids.

## Best Possible Solution

Land a narrow core update that keeps `channel` optional, adds TypeBox description text explaining that it expects a messaging provider or channel plugin id such as `slack` or `discord` rather than a destination channel/user/conversation id, and adds a focused schema assertion. Defer enum support unless it is generated from the same plugin-aware channel discovery surface used at runtime.

## Reproduction Assessment

Yes. A high-confidence reproduction is to inspect the current `createMessageTool()` parameter schema or the source that builds it; current main and `v2026.4.27` both expose `channel` as an optional string without description or enum metadata.

## Solution Assessment

Yes for the description portion of the request. The narrowest maintainable fix is a schema-description assertion in `message-tool.test.ts` plus the TypeBox description in `message-tool.ts`; a static enum is not the best fix unless it is derived from plugin-aware channel values.
