Keep open. Current main still exposes the shared message tool's channel parameter as a bare optional string, so the requested tool-schema guidance is not implemented. Runtime prompt/fallback paths reduce some failures, but they do not add the requested schema metadata, and the two linked implementation PRs are closed unmerged.

## What This Changes

Add tool-schema metadata to the shared message tool's optional channel parameter so agents understand it expects a messaging provider or channel plugin id, not the destination conversation id.

## Best Possible Solution

Land a narrow core update that keeps channel optional, adds a TypeBox description explaining it expects a messaging provider or channel plugin id such as slack or discord rather than a destination conversation/channel/user id, and adds a focused schema assertion. Defer enum support unless it is dynamically derived from configured or registered channel/plugin ids.
