## Summary

Keep open. The branch still contains a useful memory-search direction, but the webhooks half has been superseded on main and the proposed memory-search change is stale: it passes `cfg` to the registered-only provider lookup instead of using the cfg-aware runtime resolver.

## What This Changes

The PR forwards active config through memory-search provider, fallback, and multimodal lookup, adds regression tests, and changes webhooks registration/config handling for inline-secret versus SecretRef routes.

## Best Possible Solution

Rebase and narrow the branch to the memory-search fix: import the cfg-aware memory embedding provider runtime resolver, pass the active config through primary, fallback, and multimodal validation lookup, preserve targeted non-activating capability loading, and leave the current request-time webhooks SecretRef design from #67941 intact.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Use the cfg-aware memory provider resolver:** `src/agents/memory-search.ts:157`
  - body: The patch calls `getMemoryEmbeddingProvider(provider, cfg)`, but this file still imports from `../plugins/memory-embedding-providers.js`, whose lookup is registered-only and takes only an id. That either fails type checking or still bypasses the runtime capability resolver the new test mocks; import/use `../plugins/memory-embedding-provider-runtime.js` for this path.
  - confidence: 0.93
- **[P2] Keep SecretRef route registration synchronous:** `extensions/webhooks/index.ts:59-66`
  - body: When `resolveWebhooksPluginConfigSync` returns null for a SecretRef-backed route, this promise callback registers routes after `register()` has returned. The current loader closes the guarded registration API in `finally`, so those `registerHttpRoute` calls are ignored and SecretRef routes disappear; keep route registration synchronous and resolve secrets at request time.
  - confidence: 0.92
