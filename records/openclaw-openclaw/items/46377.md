## Summary

Keep open. The PR addresses a real current-main security gap in native Anthropic/Gemini PDF calls, but the branch is not merge-ready because its strict guarded-fetch path conflicts with the proxy-aware provider HTTP helper and is stale against current timeout behavior.

## What This Changes

The branch replaces raw native Anthropic/Gemini PDF fetches with strict guarded fetches, moves Gemini auth from the query string to `x-goog-api-key`, and releases guarded-fetch resources.

## Best Possible Solution

Revise the native Anthropic and Gemini PDF calls to use the shared provider HTTP helper, or an equivalent helper, while preserving SSRF blocking, proxy-mode selection, provider transport normalization, the 120s timeout, audit context, release cleanup, and Gemini header-based authentication; add focused regression tests for private-network blocking, proxy-mode selection, timeout preservation, and absence of the Gemini API key in the URL.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve provider proxy routing in guarded PDF fetches:** `src/agents/tools/pdf-native-providers.ts:69`
  - body: The new `withStrictGuardedFetchMode` path forces strict pinned-dispatch behavior instead of the shared provider HTTP helper. In HTTP(S)_PROXY-only deployments, native Anthropic/Gemini PDF calls can fail or bypass operator proxy routing while other provider calls still work through the proxy, so this needs to preserve the provider helper's proxy-selection semantics.
  - confidence: 0.9
- **[P2] Keep the native PDF request timeout on guarded fetches:** `src/agents/tools/pdf-native-providers.ts:69`
  - body: Current main gives these native provider calls a 120s timeout and tests assert an `AbortSignal`. The guarded fetch call added here passes neither `timeoutMs` nor `signal`, so rebasing this patch as written can reintroduce indefinite native PDF hangs that current main already fixed.
  - confidence: 0.86

## Security Review
