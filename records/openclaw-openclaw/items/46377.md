## Summary

Keep open. Current main still has the Anthropic/Gemini native PDF SSRF and Gemini API-key-in-query behavior, but this PR is stale against later provider transport, timeout, and Google endpoint-normalization work, so it needs a maintainer/security rework before merge.

## What This Changes

The PR changes native Anthropic and Gemini PDF HTTP calls to use strict SSRF-guarded fetches, moves Gemini authentication from the URL query string to an `x-goog-api-key` header, and releases guarded dispatchers after response handling.

## Best Possible Solution

Rework the PR on current main so both native PDF provider posts use the shared guarded provider HTTP path, or an equivalent wrapper, while preserving the 120s timeout, Google transport normalization, env-proxy/dispatcher policy, Gemini header authentication, response release cleanup, provider-specific audit context, and focused tests for private-network blocking and key placement.

## Reproduction Assessment

Yes. A focused unit test can mock `fetch`, call `geminiAnalyzePdf`, and observe the API key in the current-main URL; a similar test can pass a private `baseUrl` to either native provider and see current main build a raw credentialed fetch instead of blocking before dispatch.

## Solution Assessment

No. The security direction is correct, but the proposed implementation is not the best current-main fix because direct strict guarded fetch bypasses the provider HTTP transport contract and does not preserve later timeout and Google normalization behavior.
