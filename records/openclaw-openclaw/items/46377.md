## Summary

Keep open. Current main still has the native PDF SSRF and Gemini key-in-query problem, but this PR is stale and should not merge as-is because it bypasses the proxy-aware provider HTTP path, drops current timeout behavior, and omits current Google transport normalization.

## What This Changes

The PR replaces raw native Anthropic/Gemini PDF requests with strict SSRF-guarded fetches, moves Gemini auth from the URL query string to `x-goog-api-key`, and adds guarded-fetch release cleanup.

## Best Possible Solution

Rebase or replace the PR with a current-main implementation that routes native Anthropic and Gemini PDF requests through the shared provider HTTP helper, or an equivalent helper, while preserving SSRF blocking, proxy-mode selection, provider transport normalization, the 120s timeout, audit context, release cleanup, and Gemini header-based authentication; add focused regression tests for private-network blocking, proxy-mode behavior, timeout preservation, transport normalization, and absence of the Gemini API key in the URL.

## Reproduction Assessment

Yes. The current-main reproduction is code-level: configure a native PDF model with a custom `baseUrl`, then the PDF tool passes that URL into raw provider fetches; the PR findings are also reproducible from the PR head because strict guarded fetch is used without timeout or shared provider normalization.

## Solution Assessment

No. The PR has the right security direction, but the implementation is not the best current fix because it bypasses the established proxy-aware provider helper and loses current-main behavior that provider requests already rely on.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve provider proxy routing in guarded PDF fetches:** `src/agents/tools/pdf-native-providers.ts:69`
  - body: The patch calls `withStrictGuardedFetchMode` directly instead of the shared provider HTTP helper. In HTTP(S)_PROXY-only or centrally filtered deployments, these native PDF calls can fail or bypass the operator-managed provider egress path while other provider requests still route through the helper.
  - confidence: 0.9
- **[P2] Keep the native PDF request timeout:** `src/agents/tools/pdf-native-providers.ts:69`
  - body: Current main gives these native provider calls a 120s timeout and tests assert an `AbortSignal`. The guarded fetch calls added here pass neither `timeoutMs` nor `signal`, so rebasing this patch as written can reintroduce unbounded Anthropic/Gemini PDF requests.
  - confidence: 0.88
- **[P2] Preserve Google native transport normalization:** `src/agents/tools/pdf-native-providers.ts:154-157`
  - body: The PR head hand-normalizes Gemini `baseUrl` and no longer calls the current Google transport-normalization hook. That can build invalid native Gemini URLs for configs such as `https://generativelanguage.googleapis.com/v1beta/openai`, which current main normalizes back to the native `/v1beta` root before appending `generateContent`.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff addresses a real SSRF/key-leak surface but introduces a concrete provider-egress concern by bypassing the proxy-aware provider HTTP helper.

Concerns:

- **[medium] Strict guarded fetch can bypass provider proxy routing:** `src/agents/tools/pdf-native-providers.ts:69`
  - body: Using `withStrictGuardedFetchMode` directly skips the shared provider HTTP helper that conditionally uses trusted env-proxy mode. That can route these credentialed provider requests outside operator-managed HTTP(S)_PROXY egress controls or break proxy-only deployments, weakening the network security boundary.
  - confidence: 0.89
