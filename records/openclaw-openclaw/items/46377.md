## Summary

Keep open. Current main still has the reported native PDF provider exposure, but this PR is not merge-ready because the branch is stale against later main work and its direct strict guarded-fetch calls would drop current timeout and provider transport/proxy handling that the PDF path now has or should preserve.

## What This Changes

The PR changes native Anthropic and Gemini PDF HTTP calls to use `fetchWithSsrFGuard` in strict mode, moves Gemini authentication from the URL query string to an `x-goog-api-key` header, and releases guarded dispatchers after response handling.

## Best Possible Solution

Rework the PR on current main so both native PDF provider posts go through the shared guarded provider HTTP path or an equivalent wrapper that preserves the 120s timeout, proxy/dispatcher policy, Google transport normalization, SSRF blocking, Gemini header auth, response release cleanup, and focused tests for private-network blocking and key placement.

## Reproduction Assessment

Yes. A focused unit test can stub `fetch`, call `geminiAnalyzePdf`, and observe that current main puts the key in the URL; similarly, passing a private `baseUrl` to either native provider currently reaches raw `fetch` instead of being blocked before dispatch.

## Solution Assessment

No. The security direction is right, but the PR’s current implementation is not the narrowest maintainable fix because it bypasses existing provider HTTP transport handling and does not preserve later timeout and Google normalization behavior.
