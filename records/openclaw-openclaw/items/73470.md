## Summary

Keep open. This is a recent title-only feature request, and current main does not provide high-confidence evidence for closing it as implemented, duplicate, or out of scope. Main has related soft guidance and read-tool paging, but I did not find a mandatory completeness-scan guardrail that broadly covers read/search operations. The provided related item #73431 is a separate live Discord read/search regression, not a merged canonical replacement for this request.

## Best Possible Solution

Keep this issue open for maintainer product clarification. Define which read/search surfaces are in scope and what must happen when results are truncated, partial, empty, or suspiciously narrow. If this is a core agent guardrail, implement runtime-visible metadata plus finalization checks that require a follow-up scan or an explicit partial/blocked answer, with tests for read, search, and at least one channel or memory/web surface. If the intent is specifically Discord message read/search, route that work through #73431 and the relevant hardening PR rather than closing this broader request early.
