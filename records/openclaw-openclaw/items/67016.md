## Summary

Keep open. Current main still exposes only partial Dreams visibility: enabled state, aggregate signal/promoted counts, phase next-run timing, and diary content are present, but last run time, run outcome/partial failure, processed trace count, and diary write/workspace status are not exposed to the Dreams UI. The closed cross-reference #74929 is useful context and appears separately addressed by the managed cron using `delivery.mode: "none"`, but that reliability fix does not solve this observability request.

## What This Changes

Expose dreaming run metadata in the Control UI, including last run time, items processed, diary update/workspace status, run outcome, and next scheduled run.

## Best Possible Solution

Add a memory-core-owned Dreams run summary, expose it through `doctor.memory.status` or a sibling doctor method, normalize it in the Control UI controller, render it in the Dreams tab, and update focused backend/UI/memory-core tests plus the Dreaming docs and changelog.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to inspect `doctor.memory.status`, `loadDreamingStatus`, `renderDreaming`, and the memory-core sweep path on current main; the requested UI-visible fields are not present in the payload, controller state, props, or view.

## Solution Assessment

Yes. The best direction is a memory-core-owned run summary surfaced through the existing Dreams status path; relying only on generic cron last-run state would miss phase, diary, and multi-workspace partial-failure semantics.
