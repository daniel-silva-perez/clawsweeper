## Summary

Current main still resolves unset or explicit heartbeat target `none` to `channel: "none"` with `reason: "target-none"`, and the heartbeat runner forwards that reason into the persisted `last-heartbeat` event. The docs confirm this is intentional no-delivery behavior, but the user-facing reason remains misleading. The related rename PR #46870 is closed unmerged, and no open PR referencing this issue was found.

## Best Possible Solution

Keep this issue open and land a narrow heartbeat DX fix. Make the no-delivery-by-configuration path explicit, either by renaming this skipped reason to `delivery-disabled` or by adding target metadata / CLI display text that says delivery is disabled because `target=none`. The fix should not change heartbeat scheduling, routing, prompt execution, or the default `target: "none"` behavior.
