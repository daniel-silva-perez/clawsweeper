## Summary

Keep this issue open. Current `main` still runs the secrets-in-config audit on the resolved runtime config, so a `${VAR}` token/password that resolves to a real secret will still bypass `looksLikeEnvRef()` and emit the reported findings. The adjacent config write-back preservation work does not fix this audit collector.

## Best Possible Solution

Keep the issue open and fix the audit path. The likely fix is to thread the pre-substitution authored config, or an explicit env-ref path map derived from the parsed config file, into `collectSecretsInConfigFindings`; then test both `gateway.auth.password` and `hooks.token` with authored `${VAR}` values resolving to non-empty secrets, plus literal plaintext values that should still warn.
