## Summary

Keep this issue open. Current main still runs the secrets-in-config security audit check on the resolved runtime config, so authored `${VAR}` values for `hooks.token` and `gateway.auth.password` can still be reported as stored-in-config secrets after substitution.

## What This Changes

The issue asks for `openclaw security audit --deep` to suppress `config.secrets.hooks_token_in_config` and `config.secrets.gateway_password_in_config` when those config fields are authored as documented `${ENV_VAR}` references.

## Best Possible Solution

Keep this open and implement a source-aware secrets-in-config audit path. The fix should consult authored config text, parsed config, or an env-ref path map for the two affected fields, suppress the finding only for real authored env references backed outside the config file, and keep warnings for literal plaintext.

## Reproduction Assessment

Yes. The current code path is enough to reproduce logically: author `hooks.token` or `gateway.auth.password` as `${VAR}`, provide a non-empty env value, let config loading resolve it, then `collectSecretsInConfigFindings` sees the resolved secret rather than the sentinel and emits the finding. I did not run the command because this review is read-only.

## Solution Assessment

No for the minimal suggested implementation as written. The requested direction is correct, but simply threading `context.sourceConfig` is not sufficient on current main because that value is already post-substitution; the safer fix should use authored parsed config or an explicit env-ref path map.
