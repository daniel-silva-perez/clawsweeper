## Summary

Keep this PR open. Current main already has the Telegram media send pipeline and generic message-tool aliases, but the direct Telegram action runtime still only reads `mediaUrl` and `media`, so the PR is not obsolete. The latest provided PR context also shows the earlier precedence and PR-only docs problems were addressed, leaving a narrow implementation candidate with one small docs follow-up.

## What This Changes

The PR adds `filePath` and `path` fallbacks to Telegram `sendMessage` media resolution and tests that those aliases map into the existing `mediaUrl` send path, including `asDocument` behavior.

## Best Possible Solution

Land the focused alias mapping in the Telegram action runtime, preserve existing `mediaUrl` then `media` precedence, keep the tests for `filePath`, `path`, and `asDocument`, and add a small public Telegram docs update for the supported aliases and existing local media-root boundary.

## Reproduction Assessment

Yes. A focused unit check against current main can call `handleTelegramAction` with `action: "sendMessage"` and only `filePath` or `path`; the current runtime does not read those keys, while the PR adds exactly those fallback cases.

## Solution Assessment

Yes, with a small docs follow-up. Mapping the aliases into the existing `mediaUrl` send pipeline is the narrowest maintainable fix because Telegram already has guarded media loading and the shared message tool already exposes these parameter names.
