## Summary

Keep this PR open. Current main still uses browser-visible `token=` assistant-media URLs, while the latest PR head implements the scoped-ticket flow and has fixed the earlier missing changelog entry; the remaining blockers are branch mergeability, validation, and maintainer acceptance of the retained legacy query-token compatibility.

## What This Changes

The PR adds short-lived HMAC-scoped assistant media tickets in the Gateway, updates Control UI rendering to fetch metadata with Authorization headers and render `mediaTicket=` URLs, adds gateway/UI tests, and adds a changelog entry.

## Best Possible Solution

Update the branch against current main, run the targeted gateway/UI tests plus the changed gate through the repo-approved path, and land the scoped-ticket flow if maintainers accept keeping legacy query-token support for compatibility.

## Reproduction Assessment

Yes. Current main can be reproduced by rendering a local assistant attachment with `assistantAttachmentAuthToken`, which produces a `token=` assistant-media URL; the gateway tests also show query-token media fetches are accepted. PR head changes that same path to Authorization-probed metadata plus `mediaTicket=` render URLs.

## Solution Assessment

Yes, with merge-readiness caveats. Source-scoped, short-lived HMAC tickets are the narrowest maintainable way to preserve native browser media rendering without putting long-lived credentials in URLs; cookie auth or blob URLs would be broader changes.
