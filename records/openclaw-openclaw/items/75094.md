## Summary

Keep this PR open for normal maintainer review. Current main still emits browser-visible `token=` assistant-media URLs, while the PR implements the linked scoped-ticket direction and I did not find a concrete blocking bug in the diff; merge readiness still depends on final CI and maintainer judgment on legacy query-token compatibility.

## What This Changes

The PR adds Gateway-minted short-lived HMAC assistant-media tickets, serves media bytes with `mediaTicket=`, and updates Control UI chat rendering/tests to probe metadata with Authorization headers instead of rendering credential-bearing URLs.

## Best Possible Solution

Land the scoped-ticket architecture once CI is green and maintainers are satisfied with the compatibility boundary, preserving native browser rendering while keeping long-lived operator credentials out of rendered Control UI media URLs.

## Reproduction Assessment

Yes. On current main, a local assistant attachment with `assistantAttachmentAuthToken` produces an assistant-media URL containing `token=`, and the PR diff changes the metadata/auth/rendering path to return and use `mediaTicket=` instead.

## Solution Assessment

Yes, with normal maintainer review. Source-scoped, short-lived HMAC tickets are the narrowest maintainable option named by #70830 and avoid the broader cookie/session or blob-URL tradeoffs.
