## Summary

Keep open. Current main still lacks the prerequisite per-session activity RPC/WS state-change contract from #39127, and the existing diagnostics/stuck-session machinery is adjacent rather than the requested opt-in session activity watchdog with deduped operator alerts and optional tokenized summaries.

## Best Possible Solution

Keep this open and sequence it after #39127, or after an equivalent per-session activity/attention API with WebSocket state-change events lands. Then implement an opt-in gateway watchdog that consumes that contract, emits no-token structured alerts with configurable thresholds and dedupe/cooldowns, exposes safe UI/log/webhook/plugin delivery seams, gates optional tokenized summaries behind explicit config and rate limits, and adds focused protocol/runtime/docs tests. Treat #39305 and #73327 as adjacent inputs rather than replacements for this broader watchdog request.
