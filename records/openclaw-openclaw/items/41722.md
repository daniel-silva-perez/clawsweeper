## Summary

Keep this issue open. Current main still sends the macOS Skills settings install action through the gateway-backed `skills.install` dependency-installer path, still prefers Homebrew when `skills.install.preferBrew` is enabled and `brew` is present, and still reports selected installer failures through the generic `UNAVAILABLE` / `Install failed` path. I found no current classifier, remediation, or safe alternate-installer retry for the reported macOS 26 CLT/Homebrew failure.

## Best Possible Solution

Fix the gateway-backed skill dependency installer path: add a conservative classifier for Homebrew/Xcode CLT unsupported-macOS failures, return a targeted remediation message, avoid selecting or advertising brew when the host toolchain is known unusable, and only retry a viable alternate installer spec when metadata and policy make that safe. Add focused tests for the classifier, Homebrew selection, and fallback behavior.
