## Summary

Keep this issue open. Current main still routes the macOS Skills settings install action through gateway-backed `skills.install`, still prefers Homebrew when `brew` exists, and still reports failed installer commands with generic `Install failed (...)` output. I did not find a current classifier/remediation path for the macOS 26 CLT/Homebrew failure or a retry to alternate installer specs after a selected brew install fails.

## Best Possible Solution

Fix the gateway-backed skill dependency installer path: detect Homebrew/Xcode CLT unsupported-macOS failures, return a targeted remediation message, avoid selecting brew when the host toolchain is known unusable, retry a viable alternate installer spec when metadata provides one and policy allows it, and add focused tests for the classifier plus installer-selection/fallback behavior.
