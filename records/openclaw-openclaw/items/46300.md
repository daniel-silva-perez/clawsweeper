## Summary

Keep this PR open. Current main still has the reported read-only behavior for `workspaceAccess: "none"`, but the proposed patch is incomplete because it updates the Docker/local bridge path without the SSH remote bridge and leaves public compaction reference docs inconsistent with the changed memory-flush gate.

## What This Changes

The PR changes sandbox workspace mount writability, local FS bridge write gating, sandbox config hashing, memory-flush gating, concepts docs, and related tests so `workspaceAccess: "none"` is treated as writable inside the isolated sandbox workspace.

## Best Possible Solution

Rebase and land a corrected sandbox-permission patch that applies the isolated-writable `none` semantics across Docker/browser mounts, local and remote FS bridges, config-hash recreation, memory flush, public docs, and regression tests while preserving `ro` as read-only and keeping the agent workspace hidden.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

  - body: The patch changes the local FS bridge write gate to allow every mode except `ro`, but SSH-backed sandboxes still mark the workspace writable only for `rw` and `ensureWritable` rejects every non-`rw` sandbox. Writes through the remote bridge will still fail for `workspaceAccess: "none"`, leaving the bug unfixed for that backend.
  - confidence: 0.91

  - confidence: 0.84
