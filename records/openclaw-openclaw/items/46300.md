## Summary

Keep open. The PR addresses a real current-main sandbox write bug, but the patch is still incomplete for the remote sandbox filesystem bridge and would leave the public compaction reference docs inconsistent with the changed memory-flush behavior.

## What This Changes

The PR changes sandbox workspace mount permissions, local filesystem bridge writability, config-hash recreation, memory-flush gating, docs, and tests so `workspaceAccess: "none"` is treated as writable inside the isolated sandbox workspace.

## Best Possible Solution

Land a corrected sandbox-permission patch that applies isolated-writable `none` semantics consistently across Docker/browser mounts, local and remote filesystem bridges, config-hash recreation, memory flush, and public docs, with `ro` remaining the only read-only mode and the agent workspace still hidden for `none`.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: `workspaceAccess: "none"` maps to an isolated sandbox workspace, but the Docker mount helper, local bridge, remote bridge, and tests still treat it as read-only or non-writable.

## Solution Assessment

No. The PR is directionally right for the Docker/browser/local bridge path, but it is not the best complete fix yet because remote sandbox writes and the reference compaction docs remain inconsistent with the new contract.
