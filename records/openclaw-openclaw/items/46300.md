## Summary

Keep open. The PR addresses a valid current-main sandbox bug, but the latest branch is not merge-ready: it is dirty against current main, leaves the remote SSH/OpenShell filesystem bridge on the old `rw`-only write rule, and does not update the reference compaction docs that would contradict the changed memory-flush gate.

## What This Changes

The PR changes sandbox mount permissions, filesystem bridge writability, config-hash recreation, memory-flush gating, docs, and tests so `workspaceAccess: "none"` becomes writable inside the isolated sandbox workspace instead of read-only.

## Best Possible Solution

Land a corrected sandbox-permission patch that keeps the host agent workspace hidden for `none`, makes the per-scope sandbox workspace writable across Docker/browser/local and SSH/OpenShell remote filesystem bridges, aligns memory-flush behavior, updates both concept and reference docs, and proves the boundary with targeted tests plus Docker and remote sandbox smokes.

## Reproduction Assessment

Yes. Current main has a high-confidence static reproduction: `workspaceAccess: "none"` selects the isolated sandbox workspace, but Docker mount generation, FS mount metadata, local write guards, remote write guards, and tests still treat it as read-only; related #37634 provides the matching user-visible reproduction path.

## Solution Assessment

No. The PR is directionally right for the local Docker/browser and memory-flush path, but it is not the best complete fix yet because the remote bridge and reference docs remain inconsistent and the branch is currently unmergeable against main.
