## Summary

Keep this PR open. Current main still has the `workspaceAccess: "none"` write gap, and the PR is aimed at the right product boundary, but the proposed patch is not complete because SSH-backed remote sandboxes still enforce the old `rw`-only write rule and the compaction reference docs would contradict the changed memory-flush behavior.

## What This Changes

The PR changes sandbox workspace mount writability, FS bridge write gating, config hashing, memory-flush gating, concepts docs, and related tests so `workspaceAccess: "none"` is treated as writable inside the isolated sandbox workspace.

## Best Possible Solution

Rebase and land a corrected sandbox-permission patch that applies isolated-writable `none` semantics across Docker/browser mounts, local and remote FS bridges, config-hash recreation, memory flush, public docs, and regression tests while preserving `ro` as read-only and keeping the agent workspace hidden.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: `workspaceAccess: "none"` makes `/workspace` read-only in the Docker mount helper and leaves both local and remote FS bridge write gates in the `rw`-only bucket, so writes to the sandbox workspace are rejected by the exact code paths shown above.

## Solution Assessment

No. The PR is directionally correct for the local Docker/browser/local bridge path, but it is not the best complete fix yet because the remote FS bridge and reference docs are left inconsistent with the new writable-`none` contract.
