## Summary

Keep this PR open, but do not merge it as-is. The intended behavior is still valid on current main, and the branch addresses much of the local Docker/browser path, but it leaves the remote sandbox filesystem bridge inconsistent and would make the public compaction reference contradict the new memory-flush behavior.

## What This Changes

The PR changes sandbox mount permissions, filesystem bridge writability, config-hash recreation, memory-flush gating, docs, and tests so `workspaceAccess: "none"` is writable inside the isolated sandbox workspace instead of read-only.

## Best Possible Solution

Land a corrected sandbox-permission patch that keeps the host agent workspace hidden for `none`, makes the per-scope sandbox workspace writable across Docker, browser, local FS bridge, and SSH/OpenShell remote FS bridge paths, aligns memory-flush behavior and both public docs pages, and proves the boundary with targeted unit tests plus Docker and remote sandbox smokes.

## Reproduction Assessment

Yes. Current main has a high-confidence static reproduction: `workspaceAccess: "none"` selects the isolated sandbox workspace, but Docker mount generation, FS mount metadata, local write guards, remote write guards, and tests still treat it as read-only; related #37634 gives the matching user-visible reproduction path.

## Solution Assessment

No. The PR is directionally right for the local Docker/browser and memory-flush path, but it is not the best complete fix yet because the remote bridge and reference docs remain inconsistent and the branch is currently unmergeable against main.
