## Summary

Keep this PR open. Current main still has the reported writable-sandbox gap, and the PR is aimed at a real product boundary, but the patch is incomplete because the remote sandbox FS bridge and public compaction reference docs remain inconsistent with the proposed writable-`none` behavior.

## What This Changes

The PR changes sandbox workspace and browser mount permissions, local FS bridge write gates, sandbox config hashing, memory-flush gating, concepts docs, and related tests so `workspaceAccess: "none"` is treated as writable inside the isolated sandbox workspace.

## Best Possible Solution

Update and land a corrected sandbox-permission patch that applies isolated-writable `none` semantics across Docker/browser mounts, local and remote FS bridges, config-hash recreation, memory flush, public docs, and regression tests while preserving `ro` as read-only and keeping the agent workspace hidden.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: `workspaceAccess: "none"` makes `/workspace` read-only in the Docker mount helper and leaves local and remote FS bridge write gates in the `rw`-only bucket, so writes to the sandbox workspace are rejected by the implicated code paths.

## Solution Assessment

No. The PR is directionally correct for the local Docker/browser/local bridge path, but it is not the best complete fix yet because the remote FS bridge and reference docs remain inconsistent with the new writable-`none` contract.
