## Summary

Keep open. The PR targets the right Docker sandbox create path and Docker documents `--init` as the reaper mechanism, but the proposed patch is incomplete for already-created sandboxes because the reuse hash/epoch does not change; this needs a rollout decision plus normal secops/CODEOWNERS review before merge.

## What This Changes

The PR adds Docker `--init` to sandbox container creation and unit coverage asserting the flag appears after `create`.

## Best Possible Solution

Land the `--init` create-args change with focused unit coverage, but pair it with an explicit rollout mechanism: either update a sandbox create-args/config epoch so affected existing containers are recreated through the existing hash path, or make an intentional documented release note requiring `openclaw sandbox recreate` for already-running sandboxes.

## Reproduction Assessment

Yes for the source-level behavior: current main builds Docker sandbox containers without `--init` and launches `sleep infinity` as the long-running command, matching the linked report's PID-1 observation. I did not run a live zombie workload in this read-only review, but the missing-flag path and Docker `--init` contract are high-confidence.

## Solution Assessment

No, not as written. Adding Docker `--init` is the right mechanism for newly created containers, but the PR does not yet address existing container reuse through `openclaw.configHash`, so it is not the narrowest complete fix for the reported long-running sandbox problem.
