## Summary

Keep this PR open. Current main still lacks Brazilian WhatsApp ninth-digit equivalence for direct targets, so the linked #20187 behavior is not implemented; however, the branch is not merge-ready because it is based on older WhatsApp paths and rewrites explicit chat targets to the allowlist's canonical variant, which can misroute reactions and per-peer session state.

## What This Changes

The PR adds Brazilian WhatsApp ninth-digit direct-target equivalence across normalization, outbound allowlist resolution, inbound/command/self-chat authorization, main-route refresh, heartbeat recipient matching, SDK exports, and regression tests.

## Best Possible Solution

Port the fix into the current `extensions/whatsapp` plugin layout as a narrow direct-target equivalence helper. Use it for authorization checks only, keep the explicit normalized chat target for delivery, reactions, quote lookup, and per-peer session routing, and add focused regression tests for outbound authorization, inbound allowlist/self-chat, command authorization, main last-route refresh, heartbeat recipient matching, and action/session routing.

## Reproduction Assessment

Yes. A static reproduction path is clear: configure `allowFrom` with `+553598627740` and send or receive from `+5535998627740`; current main uses exact `includes`/`Set.has`/`===` checks, while the PR branch authorizes the pair but then returns the allowlist variant to downstream callers.

## Solution Assessment

No. The equivalence direction is useful, but returning the allowlist canonical target and widening the public SDK are not the narrowest maintainable fix; authorization equivalence should be separated from delivery/session target selection and kept inside the WhatsApp plugin unless a public API need is proven.
