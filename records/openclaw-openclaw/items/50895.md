## Summary

Keep this PR open. Current main still lacks Brazilian WhatsApp ninth-digit equivalence, and the PR addresses a real linked delivery/access-control bug from #20187, but the patch needs to be rebased/ported to the current bundled WhatsApp plugin layout and adjusted so authorization equivalence does not rewrite the actual chat target used for reactions and session routing.

## Best Possible Solution

Keep the PR open and ask for a rebase/port onto current main. The revised implementation should add a narrow WhatsApp BR direct-target equivalence helper in the bundled plugin surface, use it for allowlist, command, self-chat, main-last-route, and heartbeat authorization checks, preserve the original explicit chat target for reactions, quote lookup, outbound delivery, and per-peer session routing, and add focused regression tests under the current `extensions/whatsapp` paths.

## Risks / Open Questions

- The current PR branch is not already implemented on main, so closing would drop a still-valid fix candidate for #20187.
- The fix is authorization-sensitive: BR equivalence must stay narrowly scoped to direct Brazilian mobile-number variants and must not broaden group JIDs, hosted/LID handling, or unrelated countries.
- Returning the allowlist canonical form as the delivery target can misroute reactions or per-peer sessions unless authorization matching is kept separate from the explicit chat target.
- The patch was written against older paths and SDK export assumptions; it needs a rebase/port to the current `extensions/whatsapp` plugin layout and current public contract boundaries.
