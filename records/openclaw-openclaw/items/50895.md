## Summary

Keep this PR open. Current main still lacks Brazilian WhatsApp ninth-digit equivalence for direct targets, so the linked #20187 user problem is not implemented on main; the PR remains a relevant fix candidate but needs to be ported to the current bundled WhatsApp plugin layout and revised so authorization matching does not rewrite delivery/reaction/session targets.

## Best Possible Solution

Keep the PR open and ask for a rebase/port onto current main. The revised patch should add a narrow WhatsApp direct-target equivalence helper in the bundled plugin surface, use it for allowlist, command, self-chat, main-last-route, and heartbeat authorization checks, preserve the original explicit chat target for reactions, quote lookup, outbound delivery, and per-peer session routing, and add focused regression tests under the current `extensions/whatsapp` paths. Avoid expanding the public SDK unless there is a clear third-party plugin API need.

## Risks / Open Questions

- The PR's current approach of returning the allowlist canonical target can authorize correctly but send reactions or route per-peer sessions through the wrong chat identity.
- The equivalence is authorization-sensitive and should remain narrowly scoped to Brazilian direct mobile-number variants, excluding groups, hosted/LID forms, and unrelated countries.
- The PR was written partly against older `src/whatsapp` and `src/channels/plugins` paths; current main has moved this surface under `extensions/whatsapp`, so a direct close or merge would lose needed layout/boundary work.
