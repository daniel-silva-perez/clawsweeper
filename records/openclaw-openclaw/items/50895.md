## Summary

Keep this PR open. Current main still lacks Brazilian WhatsApp ninth-digit equivalence, and the PR addresses the same user-facing delivery/access-control problem described in #20187, but it needs to be rebased/ported onto the current bundled WhatsApp plugin layout and adjusted so authorization equivalence does not rewrite the actual chat target used for reactions and per-peer routing.

## Best Possible Solution

Keep the PR open and request a rebase/port onto current main. The revised patch should add a narrow WhatsApp BR direct-target equivalence helper in the bundled plugin surface, use it for allowlist, command, self-chat, main-last-route, and heartbeat authorization checks, preserve the original explicit chat target for reactions, quote lookup, outbound delivery, and per-peer session routing, and add focused regression tests under the current `extensions/whatsapp` paths.

## Risks / Open Questions

- Closing this PR would drop a still-relevant fix candidate for the linked Brazilian WhatsApp delivery/access-control bug.
- The equivalence is authorization-sensitive and should stay narrowly scoped to Brazilian direct mobile-number variants, without broadening group JIDs, hosted/LID handling, or unrelated countries.
- Rewriting the delivery target to the allowlist canonical form can misroute reactions or per-peer sessions; authorization matching should be kept separate from the target used for delivery/session routing.
- The PR was written against older source/SDK paths; a rebase should align it with the current `extensions/whatsapp` bundled plugin boundary and avoid unnecessary public SDK expansion.
