## Summary

Keep open. Current main still retains the reported Telegram typing outage surface: the generic keepalive loop has no tick-level failure breaker, Telegram typing can still use the normal outbound retry profile, and the dedicated sendChatAction handler only suspends repeated 401 errors. The discussion narrowed the issue correctly toward retry cost plus guard coverage, and overlapping open work needs maintainer scope triage before another repair branch is queued.

## What This Changes

The issue asks OpenClaw to prevent Telegram typing/sendChatAction network failures from repeatedly retrying during outages and making the gateway unresponsive.

## Best Possible Solution

Land one canonical runtime fix that keeps typing indicators best-effort: bound repeated keepalive failures, make Telegram typing fail cheaply or enter cooldown without changing normal message delivery retries, preserve the existing 401 suspension behavior, and add regression coverage for shared typing lifecycle behavior plus Telegram-specific sendChatAction retry/cooldown behavior.

## Reproduction Assessment

Yes. The issue gives concrete network-outage steps and production log signatures, and current main still contains source paths that can repeatedly invoke expensive Telegram typing failures under outage conditions.

## Solution Assessment

Yes. The best direction is a combined bounded typing keepalive failure policy plus Telegram-specific best-effort retry/cooldown behavior; the immediate path is to review or expand the existing related work rather than spawn another duplicate fix first.
