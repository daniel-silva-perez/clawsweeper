## Summary

Keep open. Current main and the latest stable tag still retain the reported Telegram typing outage surface: the generic keepalive loop has no tick-level failure breaker, Telegram typing still inherits the normal outbound retry profile unless callers override it, and non-401 sendChatAction failures are not covered by the dedicated 401 suspension handler. The issue discussion correctly narrowed the root cause to retry cost plus bypass/guard coverage, and related open work needs maintainer scope triage before spawning another duplicate repair branch.

## What This Changes

The issue asks OpenClaw to keep Telegram typing/sendChatAction network failures from repeatedly retrying during outages and making the gateway unresponsive.

## Best Possible Solution

Land one canonical runtime fix that keeps typing indicators best-effort: bound repeated keepalive failures, make Telegram typing fail cheaply or enter cooldown without changing normal message delivery retries, preserve the existing 401 suspension behavior, and add regression coverage for both shared typing lifecycle behavior and Telegram-specific sendChatAction retry/cooldown behavior.

## Reproduction Assessment

Yes. The issue gives concrete network-outage steps and production log signatures, and current main still contains the source path that can repeatedly invoke expensive Telegram typing failures under outage conditions.

## Solution Assessment

Yes, the best direction is to combine bounded typing keepalive failure handling with a Telegram-specific best-effort retry/cooldown policy. The immediate maintainer path should be to review or expand #55886 together with #56096, not create another duplicate fix PR first.
