## Summary

Keep this issue open. Current `main` and the latest release still show the reported Telegram typing failure path: the generic keepalive loop has no tick error breaker, the auto-reply typing controller can rethrow keepalive failures without a max failure threshold, and `sendTypingTelegram` still inherits the normal channel retry runner unless a caller supplies cheaper retry options. Related PRs #45836 and #47018 were closed unmerged, so they do not supersede the remaining work.

## What This Changes

The issue requests bounded failure handling for Telegram typing keepalive network failures so cosmetic typing calls cannot stall or crash the gateway.

## Best Possible Solution

Land one focused bug-fix PR that bounds typing keepalive failures and gives Telegram typing a cheap best-effort retry posture, while preserving normal Telegram message delivery retries, existing typing modes, and the Telegram 401 `sendChatAction` backoff.

## Reproduction Assessment

Yes. The issue gives concrete steps and production log signatures, and current-main source still shows the same unbounded error/retry path for Telegram typing failures under network outage conditions.

## Solution Assessment

Yes. The narrowest maintainable solution is a small runtime fix in the typing lifecycle/controller plus the Telegram typing send path; disabling typing or lowering global Telegram retry config is only a workaround because it changes broader behavior.
