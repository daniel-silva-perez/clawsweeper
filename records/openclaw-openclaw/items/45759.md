## Summary

Keep this issue open. Current main and the latest release still show the Telegram typing failure surface: the generic keepalive loop has no tick-level breaker, the auto-reply typing controller can rethrow keepalive failures without a max-failure threshold, and `sendTypingTelegram` still inherits normal outbound retry settings unless callers override them. Related fix attempts #45836, #47018, #55838, and #56153 are closed unmerged, while open PR #55886 overlaps the transient `sendChatAction` portion but does not yet prove this issue is implemented on main.

## What This Changes

The issue requests bounded, best-effort Telegram typing behavior so network failures cannot keep retrying typing indicators until the gateway becomes unresponsive.

## Best Possible Solution

Review or expand the active Telegram cooldown work so a single focused fix bounds typing keepalive failures, makes Telegram typing best-effort/cheap to fail, preserves normal message delivery retries and existing 401 `sendChatAction` suspension behavior, and adds regression tests for both the generic typing lifecycle and Telegram-specific retry path.

## Reproduction Assessment

Yes. The issue provides concrete steps and production log signatures, and current main still contains the same source-level failure path for repeated Telegram typing failures during network outages.

## Solution Assessment

Yes, the narrow maintainable direction is to bound typing keepalive failures and make Telegram typing retries cosmetic rather than inheriting normal delivery retry posture. The safer immediate path is maintainer review of #55886 against this issue and #56096 before opening a separate repair PR.
