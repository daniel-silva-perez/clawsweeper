## Summary

Keep this issue open. Current main still lacks the two fixes agreed in the issue discussion: `createTypingKeepaliveLoop` has no catch/circuit-breaker around failed ticks, and `sendTypingTelegram` still uses the normal Telegram retry path rather than a cheap best-effort retry profile for cosmetic typing actions. The existing higher-level typing guard and the Telegram 401-only sendChatAction backoff are useful but do not fully cover the reported network-failure/direct-caller path.

## Best Possible Solution

Keep the issue open and land a focused fix in the Telegram typing path: add regression coverage for failed keepalive ticks, add a generic `createTypingKeepaliveLoop` circuit breaker or equivalent failure containment, make `sendTypingTelegram` best-effort with a short/one-attempt retry profile, and audit the direct Telegram typing call sites so cosmetic typing cannot block real delivery work.
