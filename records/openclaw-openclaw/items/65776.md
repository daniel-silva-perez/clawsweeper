## Summary

Keep open. Current main still has only configured/static remote MCP HTTP headers: config and schema expose `mcp.servers.<name>.headers` as a scalar record, the HTTP resolver converts that record once, the transport constructors pass those static headers into MCP SDK `requestInit`/`eventSourceInit`, and the session runtime reuses the connected client/transport without a request-scoped header resolver or session/runtime-context hook. Related #65590 covers static header forwarding, while #62134 covers OAuth/authProvider work; neither covers arbitrary dynamic per-request tenant/user/correlation headers.

## Best Possible Solution

Keep this issue open for maintainer design. The best implementation path is a typed request-scoped header resolver for remote MCP transports, or a public plugin/runtime API that can safely supply one: thread session/runtime context into MCP request handling, merge dynamic headers after static config with clear precedence and redaction, cover both streamable-http and SSE, document the security model, and add focused tests for tenant/user/token-refresh/correlation-header behavior. Coordinate with #62134 so OAuth/authProvider support is not conflated with arbitrary dynamic headers.
