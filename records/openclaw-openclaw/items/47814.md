## Summary

Keep open. Current main still does not contain the named-DM session API or store selector this PR proposes, so the PR is not obsolete; however, the PR head has concrete P2 correctness defects around malformed key parsing, malformed persisted state, chat-type policy classification, and changelog history rewrites, and it is stale against the current session-chat-type layout.

## What This Changes

The PR adds named-DM session key helpers and tests, an optional activeNamedSession SessionEntry field, gateway helpers, session documentation, and changelog entries without wiring routing behavior.

## Best Possible Solution

Keep the work open for maintainer review or a narrow rebased replacement. If named-DM session switching is still wanted, land the infrastructure with strict raw key parsing, fail-closed persisted-state handling, explicit dm-named direct classification in the current chat-type and send-policy paths, focused malformed-key and policy-collision tests, contributor docs that distinguish primitives from routed behavior, and one credited changelog entry in the active section only.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Reject collapsed empty segments in named-DM parsing:** `src/sessions/session-key-utils.ts:180-184`
  - body: parseNamedDmSessionKey relies on parseAgentSessionKey, which drops empty ':' segments before this parser validates the dm-named shape. A malformed key like agent:main:dm-named:123::work can therefore be accepted as peer 123 and name work; validate the raw segments or explicitly reject empties before returning a parsed named-DM key.
  - confidence: 0.92
- **[P2] Fail closed for malformed stored active names:** `src/gateway/session-utils.ts:862-870`
  - body: activeNamedSession is persisted store data, but getActiveNamedSessionKey passes the stored name directly into buildNamedDmSessionKey. If an existing or manually edited row contains ':', active-session lookup throws instead of returning null; validate the stored name and ignore or repair malformed values before building the key.
  - confidence: 0.9
- **[P2] Prioritize dm-named before token chat-type checks:** `src/sessions/session-key-utils.ts:50`
  - body: Adding dm-named to the direct branch after group and channel checks still misclassifies valid named DM keys whose user-chosen name is group or channel. Since send policy can fall back to key-derived chat type, those DMs can pick up group/channel rules unless dm-named is recognized before arbitrary name tokens are scanned.
  - confidence: 0.91
- **[P2] Keep the changelog entry in the active section only:** `CHANGELOG.md:109`
  - body: The branch repeats the named-DM changelog bullet in an older release section. That rewrites release history and can confuse release-note tooling; keep one credited entry under Unreleased or the active release section only.
  - confidence: 0.96
- **[P3] Correct the helper nullability docs:** `docs/concepts/session.md:79`
  - body: The docs say all three helpers accept null or undefined and fail safely, but buildNamedDmSessionKey requires a params object and immediately reads params.agentId. Either narrow that sentence to the parser/predicate helpers or change the implementation and types to match the documented contract.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff adds no dependency, CI, secret, or package execution surface, but the named-DM chat-type classification can apply group/channel send-policy rules to user-named direct-message sessions.

Concerns:

- **[medium] Named-DM names can collide with policy chat-type tokens:** `src/sessions/session-key-utils.ts:50`
  - body: Because dm-named is recognized only after group and channel token checks, a direct-message session named group or channel can be classified as that chat type when send policy falls back to key-derived classification. That can apply the wrong outbound allow/deny policy to a DM session.
