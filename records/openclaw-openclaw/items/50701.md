## Summary

Keep open. Current main has transcript-backed global usage counting and archive discovery, but the WebUI/session usage surface still has no logical-session family contract, no `includeHistorical` or grouping option, and no UI distinction between current instance and historical lineage. Related archive-recovery work is useful context, but it does not cover the usage lineage rollup requested here.

## What This Changes

Add transcript-backed logical-session lineage usage rollups and WebUI controls so usage history remains continuous across updates, restarts, resets, and sessionId rebinding.

## Best Possible Solution

Design an additive usage-history contract that keeps current-instance behavior intact while adding a stable logical family key, transcript-backed rollups across current and archived/rebound instances, explicit date ranges, and a WebUI toggle that clearly shows what sessions are included.

## Reproduction Assessment

Yes, at source level. Current main can be inspected to show that `sessions.usage` has no family/grouping parameter and per-session summaries resolve a single transcript file, while tests show archived usage data exists for global counting.

## Solution Assessment

Unclear until maintainer product/API review. The requested transcript-backed lineage direction is sound, but the maintainable fix should be an additive family/index API plus UI affordance rather than a one-off restore script or external snapshot workaround.

## Risks / Open Questions

- The right stable family key is a product/API decision: over-merging transcripts could double-count or merge unrelated channel/session surfaces, while under-merging would preserve the current fragmented UX.
- Lineage aggregation may need an indexed summary or cache to avoid expensive full transcript rescans for long-range WebUI loads.
- This should be coordinated with #45003, #60409, and related archive-history work so session history recovery and usage rollups share compatible archive semantics.
