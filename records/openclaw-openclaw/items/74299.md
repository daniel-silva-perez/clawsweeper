## Summary

Keep open. Current main already avoids the exact reported connected log string and only marks Telegram polling connected after a successful getUpdates call, but the requested operator diagnostic remains incomplete: doctor/status do not currently raise a Telegram-specific warning for a running account with no polling or webhook activity, and webhook mode does not appear to publish comparable connected/activity runtime status.

## What This Changes

Request to make Telegram channel health and logs distinguish valid initialization from active polling or webhook delivery, and to add doctor diagnostics for false-positive connected state.

## Best Possible Solution

Keep this open for a focused Telegram diagnostics fix: publish explicit runtime activity for polling and webhook modes, then surface a doctor/channel-status warning when Telegram is running but has not proven polling or webhook activity after startup. Leave bidirectional startup self-messaging as a separate maintainer product decision.
