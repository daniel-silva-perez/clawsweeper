## Summary

Keep this PR open, but it is not merge-ready. Current main still only supports Synology Chat direct messages, while the PR is a useful plugin-owned direction for group/channel support; however, the current branch introduces a channel media URL validation bypass, misses per-channel outbound URL routing, leaves a shared test mock out of sync, and is currently dirty with failing checks.

## What This Changes

The PR adds Synology Chat group/channel support across the bundled plugin and docs, including group config fields, group access and mention handling, channel reply helpers, group session routing, outbound channel sends, and tests.

## Best Possible Solution

Finish the Synology Chat plugin-owned implementation by preserving the existing media URL safety contract for channel sends, resolving channel-specific incoming URLs consistently for inbound and proactive sends, normalizing group allowlist entries at the config boundary, repairing the mock/test surface, then rebase and validate the Synology Chat plugin lanes before merging.

## Reproduction Assessment

Yes. The main-branch absence is reproducible from source/docs, and the PR defects are reproducible by static inspection plus targeted cases: channel media send with a blocked/private file URL, proactive sends to a per-channel override, and loading channel integration tests with the shared client mock.

## Solution Assessment

No as submitted. The feature belongs in the Synology Chat plugin rather than core, but the current patch needs the security and routing fixes above before it is the narrowest maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Validate channel media URLs before sending:** `extensions/synology-chat/src/client.ts:177-180`
  - body: `sendFileUrlToChannel` builds a `file_url` payload directly from `fileUrl`, while the existing DM media path calls `assertSafeWebhookFileUrl` before forwarding. Since `sendMedia` routes `channel:` targets through this helper, private or non-HTTP URLs can bypass the existing Synology file URL guard for group media; reuse the same validation before `doPost`.
  - confidence: 0.94
- **[P2] Resolve per-channel URLs for proactive channel sends:** `extensions/synology-chat/src/channel.ts:354-355`
  - body: The new config type and inbound reply path support `channels.*.incomingUrl`, but the outbound `channel:`/`group:` branches always use `requireIncomingUrl(account)`. In a shared-account setup, proactive sends to a specific channel can post to the account default webhook instead of that channel's override.
  - confidence: 0.88
- **[P2] Add the channel file helper to the shared mock:** `extensions/synology-chat/src/channel.test-mocks.ts:76-80`
  - body: `channel.ts` now imports `sendFileUrlToChannel`, but this shared `./client.js` mock does not export it. Integration tests that import the real plugin with this mock can fail during module loading with a missing mocked export before assertions run.
  - confidence: 0.9
- **[P2] Normalize group allowlist entries before matching:** `extensions/synology-chat/src/group-access.ts:67-87`
  - body: `groupAllowFrom` and `channels.*.allowFrom` are passthrough config, but array entries are kept as-is and then `id.toLowerCase()` is called. Numeric Synology IDs in JSON config can therefore throw a TypeError and turn an authorized group webhook into a 500; coerce entries to strings before wildcard and equality checks.
  - confidence: 0.84
