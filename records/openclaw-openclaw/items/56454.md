## Summary

Current main already implements the PR's user-facing goal: the explicit image tool no longer hardcodes a 30s timeout and instead uses the shipped media-understanding timeout configuration. The implementation landed through the canonical `tools.media.image.timeoutSeconds` / per-model `timeoutSeconds` surface, so adding a second `agents.defaults.imageTimeoutSeconds` key is no longer needed.

## What This Changes

The PR adds an `agents.defaults.imageTimeoutSeconds` config field, generated schema/docs metadata, and tests, then wires the explicit image tool to pass that timeout to image provider calls.

## Best Possible Solution

Keep the shipped `tools.media` timeout surface as the single operator-facing media-understanding configuration path, and use the already-released implementation rather than adding a parallel agent-defaults timeout key.

## Reproduction Assessment

Yes. The old source-level reproduction was any slow image provider exceeding the hardcoded 30s tool timeout; current main shows that path now resolves a configurable media timeout and focused tests verify propagation to provider calls.

## Solution Assessment

No, merging this PR is no longer the best path. The best solution has already shipped through the broader media-understanding config surface, which also supports per-model timeouts and avoids adding a second overlapping agent default.
