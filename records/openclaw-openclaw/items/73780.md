## Summary

Keep open. Current main still has a narrow Telegram delivery bug: provider rate-limit payloads marked as errors are delivered as normal Telegram replies, while Telegram `errorPolicy` suppression is only applied to Telegram delivery failures after a send attempt fails.

## What This Changes

The issue reports repeated Telegram-visible API rate limit errors when OpenClaw is used through the Telegram plugin.

## Best Possible Solution

Add a Telegram pre-send policy for provider/agent error payloads: classify rate-limit or overload `isError` replies before delivery, apply account/group/topic `errorPolicy` and cooldown by account/chat/thread, keep normal successful replies intact, and align docs with the typed config contract.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to dispatch repeated Telegram `ReplyPayload` values with `isError: true` and rate-limit text; current main routes them through `deliverReplies` instead of coalescing or suppressing them.

## Solution Assessment

Yes. The narrowest maintainable fix is at the Telegram/agent delivery boundary, because provider cooldown already exists and the remaining observable bug is repeated Telegram delivery of successful provider-error replies.
