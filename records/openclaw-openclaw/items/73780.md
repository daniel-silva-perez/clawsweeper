## Summary

Keep open. Current main still forwards provider error payloads through Telegram as normal outbound replies, and Telegram `errorPolicy` is only applied after Telegram delivery failures, so the OpenClaw-visible spam-control part of the report remains unresolved even though the reporter’s Z.ai high-concurrency block is external.

## What This Changes

The issue asks OpenClaw to stop repeatedly surfacing Telegram replies for provider API rate-limit or high-concurrency errors, especially from Z.ai/GLM usage through Telegram.

## Best Possible Solution

Add Telegram pre-send handling for provider/agent error payloads: classify rate-limit and overload `isError` replies before delivery, apply the existing account/group/topic `errorPolicy` and per account/chat/thread cooldown, keep normal successful replies unchanged, and align docs/tests with the actual config contract.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to deliver repeated Telegram `ReplyPayload` values with `isError: true` and rate-limit text, or use the existing mid-turn 429 runner path; current main still sends those payloads instead of suppressing or coalescing them before Telegram delivery.

## Solution Assessment

Yes. The narrowest maintainable fix is at the Telegram agent-delivery boundary, because provider fallback/cooldown already mitigates upstream retry storms while the remaining user-visible bug is repeated Telegram delivery of successful error replies.

## Repair Work Prompt

Fix the Telegram-visible repeated provider rate-limit error behavior tracked at https://github.com/openclaw/openclaw/issues/73780. Implement a narrow Telegram pre-send policy for provider/agent error payloads: classify rate-limit/overload `isError` replies before delivery, apply existing account/group/topic `errorPolicy` and `errorCooldownMs` per account/chat/thread, preserve normal non-error replies and successful turns, and avoid changing provider fallback semantics. Update Telegram dispatch/error-policy tests and docs so the documented values/defaults match the actual config contract. Add a user-facing `CHANGELOG.md` Fixes entry crediting @xace1825 unless a maintainer says otherwise.
