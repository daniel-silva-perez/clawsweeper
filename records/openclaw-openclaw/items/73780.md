## Summary

Keep open. Current main still leaves the Telegram-visible part of the report unresolved: provider rate-limit payloads marked `isError` are sent as Telegram replies, while Telegram `errorPolicy` suppression only runs after Telegram delivery failures rather than before successfully sending provider-error text.

## What This Changes

The issue asks OpenClaw to reduce repeated Telegram-visible API rate-limit error replies when a provider such as Z.ai/GLM-5 is rate-limited.

## Best Possible Solution

Add a Telegram pre-send policy for provider/agent error payloads: classify rate-limit or overload `isError` replies before delivery, apply account/group/topic `errorPolicy` and per account/chat/thread cooldown, preserve normal successful replies, and update docs/tests to match the actual config contract.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to dispatch repeated Telegram `ReplyPayload` values with `isError: true` and rate-limit text, or use the existing mid-turn 429 runner path; current main will still deliver those payloads rather than coalescing them.

## Solution Assessment

Yes. The narrowest maintainable fix is at the Telegram/agent delivery boundary, not deeper provider fallback logic, because provider cooldown already exists while the remaining user-visible behavior is repeated Telegram delivery of successful error replies.
