## Summary

Keep open. Current main has provider-side cooldown/fallback handling, but the Telegram-facing problem remains: rate-limit error payloads can still be delivered as normal Telegram replies, and Telegram errorPolicy is only applied after Telegram delivery failures rather than before sending provider error replies.

## What This Changes

The issue requests reducing repeated Telegram-visible API rate-limit error messages when a provider such as Z.ai returns rate-limit failures.

## Best Possible Solution

Handle this at the Telegram/agent delivery boundary: classify provider rate-limit error payloads before sending, apply a per account/chat/thread cooldown or silent policy using the existing Telegram error-policy helper, preserve normal successful replies, and align docs/tests with the actual always/once/silent config contract.

## Reproduction Assessment

Yes. A focused code-level reproduction is to feed Telegram dispatch repeated ReplyPayload values with isError true and rate-limit text, or use the existing mid-turn 429 runner path that produces "API rate limit reached"; current main will still deliver those payloads instead of coalescing them.

## Solution Assessment

Yes, the narrowest maintainable solution is a Telegram pre-send policy for provider error payloads, not more provider fallback logic. Provider cooldown already exists and addresses upstream retry storms, while the remaining regression is visible message delivery behavior.
