## Summary

Keep open. Current main still leaves the OpenClaw-controlled Telegram spam surface unresolved: upstream Z.ai rate-limit behavior may be external, but Telegram still forwards repeated provider/agent `isError` payloads instead of applying the configured error policy before delivery.

## What This Changes

The issue requests suppression or cooldown handling for repeated Telegram-visible API rate-limit errors, especially Z.ai/GLM high-concurrency blocks.

## Best Possible Solution

Add Telegram pre-send handling for provider/agent error payloads: classify rate-limit and overload `isError` replies before delivery, apply account/group/topic `errorPolicy` and per account/chat/thread cooldown, preserve normal successful replies, and align docs/tests with the actual config contract.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to dispatch repeated Telegram `ReplyPayload` values with `isError: true` and rate-limit text; current main sends them through `deliverReplies` instead of suppressing or coalescing them.

## Solution Assessment

Yes. The narrowest maintainable solution is at the Telegram delivery boundary, because provider cooldown/fallback already exists and the remaining observable bug is repeated Telegram delivery of successful provider-error replies.
