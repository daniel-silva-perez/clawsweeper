## Summary

Keep this issue open. Current main has important partial fixes for commentary-phase/final_answer separation and group/channel message-tool-only defaults, but the broader direct-chat leak remains plausible and recently reproduced on stable. Open implementation PRs also still use closing syntax for this issue, so the issue should remain open until maintainers review and land or reject those candidates.

## Best Possible Solution

Keep this issue open as the canonical product bug. Maintainers should review the linked open implementation PRs and choose a direct-chat-safe policy, such as explicit sourceReplyDeliveryMode/message-tool-only for DMs, configurable final-tag enforcement, or another final-only delivery guard. Add regression coverage for unphased assistant text before and between tool calls in both streaming/block reply and final reply assembly paths.

## Risks / Open Questions

- Changing direct chats to message-tool-only or final-only delivery could break deployments that rely on automatic bare-text DM replies; this likely needs an opt-in, migration path, or explicit maintainer product decision.
- Follow-up work should preserve the existing commentary/final_answer safeguards and group/channel message-tool-only behavior while addressing ordinary unphased assistant text before or between tool calls.
- Several open PRs are implementation candidates with different policy choices, so maintainers should review or close those before resolving the canonical issue.
