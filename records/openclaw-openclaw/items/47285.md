## Summary

Keep this PR open. The requested Azure embedding support is still missing on current main, but this patch is not ready to merge because it only changes header/query defaults and does not handle Azure deployment-scoped embedding URLs, supported Azure host suffixes, or configurable API versions.

## What This Changes

The PR changes the memory-lancedb direct OpenAI-compatible embedding client to detect `.openai.azure.com` base URLs and add Azure `api-key` and `api-version=2024-02-01` request defaults.

## Best Possible Solution

Add Azure-aware memory embedding request shaping in the memory-lancedb/OpenAI embedding path: parse recognized Azure hostnames, send Azure `api-key` auth without an invalid OpenAI bearer credential, normalize resource-root and `/openai/v1` inputs to `/openai/deployments/{deployment}/embeddings`, expose API-version configuration through runtime config, manifest/UI hints, docs, and tests, and preserve generic OpenAI-compatible proxy behavior for non-Azure endpoints.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to configure `memory-lancedb` with `embedding.provider: "openai"`, a plugin `apiKey`, and an Azure resource `baseUrl`; current main posts to `/embeddings`, and this PR would still not build Azure's deployment-scoped URL or expose API-version configuration.

## Solution Assessment

No. The patch is a useful start, but the maintainable fix should reuse the repo's parsed-host Azure pattern, avoid OpenAI bearer auth for Azure API-key calls, build deployment-scoped URLs, and add a supported API-version surface instead of hard-coding a query default in the constructor.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Build deployment-scoped Azure embedding URLs:** `extensions/memory-lancedb/index.ts:176-180`
  - body: The Azure branch leaves `baseURL` unchanged, so `embed()` still posts to `/embeddings`. A user who points `baseUrl` at an Azure resource root, as the PR describes, will miss Azure's required `/openai/deployments/{deployment-id}/embeddings` path and fail at runtime.
  - confidence: 0.91
- **[P2] Suppress bearer auth for Azure embeddings:** `extensions/memory-lancedb/index.ts:176-179`
  - body: The Azure branch still constructs the normal OpenAI client with `apiKey`. In `openai@6.34.0`, that client adds `Authorization: Bearer <key>` before applying default headers, so adding `api-key` does not remove the non-Azure bearer auth shape from Azure requests.
  - confidence: 0.87
- **[P2] Make the Azure API version configurable:** `extensions/memory-lancedb/index.ts:179-180`
  - body: The patch hard-codes `api-version=2024-02-01`, but memory-lancedb config and manifest schema do not accept an API-version field. Users who need a different Azure-supported version would have no supported override path.
  - confidence: 0.9
- **[P2] Parse all supported Azure host suffixes:** `extensions/memory-lancedb/index.ts:173`
  - body: Checking `baseUrl?.includes(".openai.azure.com")` misses the repo's existing `.services.ai.azure.com` and `.cognitiveservices.azure.com` Azure host handling, and it can match text outside the parsed hostname. Use URL hostname parsing like the existing OpenAI provider path.
  - confidence: 0.84
