## Summary

Keep this PR open. Current main still lacks native Azure OpenAI request shaping for memory-lancedb embeddings, and the submitted patch remains useful but incomplete for the current config, schema, and Azure host coverage.

## What This Changes

The PR changes the memory-lancedb OpenAI-compatible embedding client to detect Azure OpenAI base URLs and add Azure-style `api-key` headers plus a hardcoded `api-version` query parameter.

## Best Possible Solution

Add Azure-aware embedding transport for memory-lancedb across the direct OpenAI-compatible and provider-backed paths, reuse parsed-host Azure detection, expose an optional `embedding.apiVersion` through runtime config, manifest UI hints, and docs, and add focused Azure/non-Azure request-shaping tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Expose the Azure API version instead of hardcoding it:** `extensions/memory-lancedb/index.ts:179`
  - body: The patch adds a fixed `api-version=2024-02-01`, but memory-lancedb config and the manifest do not accept any override. Azure deployments that require a newer or tenant-specific API version would have no supported config path, so embedding calls can fail until users fork the plugin.
  - confidence: 0.88
- **[P2] Detect Azure endpoints by hostname and include Foundry hosts:** `extensions/memory-lancedb/index.ts:173`
  - body: The new substring check only recognizes `.openai.azure.com` and misses `*.services.ai.azure.com`, which OpenClaw already classifies as Azure during onboarding. Those Foundry endpoints would still be sent OpenAI-style auth without `api-version`, leaving native Azure support broken for valid Azure URLs.
  - confidence: 0.9
