## Summary

Keep this PR open. It targets a real memory-lancedb gap, and current main still has no Azure-aware direct embedding request path, but the proposed patch is not merge-ready because it only adds a narrow hostname check, an extra `api-key` header, and a hard-coded API version while leaving the request on the wrong Azure URL shape and without a supported config surface.

## What This Changes

The PR changes `extensions/memory-lancedb/index.ts` so the direct OpenAI-compatible embedding client detects `.openai.azure.com` base URLs and adds an Azure `api-key` header plus `api-version=2024-02-01` default query.

## Best Possible Solution

Add Azure-aware memory embedding request shaping that parses recognized Azure hostnames, sends Azure `api-key` auth without an invalid OpenAI bearer header, normalizes resource-root and `/openai/v1` base URLs to deployment-scoped `/openai/deployments/{deployment}/embeddings` requests, exposes API-version configuration through runtime config, manifest/UI hints, docs, and tests, and preserves OpenAI-compatible proxy behavior for non-Azure endpoints.

## Reproduction Assessment

Yes. A code-level reproduction is clear: configure `memory-lancedb` with `embedding.provider: "openai"`, a plugin `apiKey`, and an Azure resource `baseUrl`; current main selects the direct client and posts to `/embeddings`, while this PR still would not build Azure's deployment-scoped embeddings URL or expose API-version configuration.

## Solution Assessment

No. The patch is a useful start, but the narrow maintainable fix should reuse the repo's parsed-host Azure pattern, clear or avoid OpenAI bearer auth, build the deployment-scoped URL, and add a supported API-version surface instead of hard-coding a query default in the constructor.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Suppress bearer auth for Azure embeddings:** `extensions/memory-lancedb/index.ts:174-179`
  - body: The Azure branch still constructs the normal OpenAI client with `apiKey`. In `openai@6.34.0`, that client adds `Authorization: Bearer <key>` before `defaultHeaders`, so adding `api-key` does not remove the invalid bearer auth shape for Azure requests.
  - confidence: 0.91
- **[P2] Build deployment-scoped Azure embedding URLs:** `extensions/memory-lancedb/index.ts:176-180`
  - body: Leaving `baseURL` unchanged means the existing embed path posts to `/embeddings`. A user who points `baseUrl` at an Azure resource root as described by the PR will miss Azure's `/openai/deployments/{deployment}/embeddings` path and fail at runtime.
  - confidence: 0.9
- **[P2] Make the Azure API version configurable:** `extensions/memory-lancedb/index.ts:179-180`
  - body: The patch hard-codes `api-version=2024-02-01`, but the runtime config and manifest do not accept an API-version field. Users who need a different Azure-supported version would have no supported override path.
  - confidence: 0.89
- **[P2] Parse all supported Azure host suffixes:** `extensions/memory-lancedb/index.ts:173`
  - body: Checking `baseUrl?.includes(".openai.azure.com")` misses the repo's recognized `.services.ai.azure.com` and `.cognitiveservices.azure.com` Azure hosts, and it can match text outside the parsed hostname. Use URL hostname parsing like the existing OpenAI provider path.
  - confidence: 0.86
