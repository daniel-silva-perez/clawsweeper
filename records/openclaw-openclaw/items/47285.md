## Summary

Keep open. Current main has not implemented native Azure embeddings for memory-lancedb, and this PR is useful source work but not merge-ready: it leaves Azure auth, URL shape, endpoint detection, API-version configuration, and provider-backed embedding behavior inconsistent with existing Azure OpenAI surfaces.

## What This Changes

The PR changes `extensions/memory-lancedb/index.ts` to detect `.openai.azure.com` base URLs and add Azure-specific `api-key` and hard-coded `api-version` defaults to the direct OpenAI-compatible embedding client.

## Best Possible Solution

Add Azure-aware memory embedding request shaping that parses hostnames, supports the repo’s Azure suffix list, sends `api-key` without bearer auth, normalizes Azure embedding URLs, exposes an optional config/manifest/docs API-version setting, and covers both direct and provider-backed memory embeddings with focused tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Remove bearer auth on Azure embedding requests:** `extensions/memory-lancedb/index.ts:174-179`
  - body: The Azure branch still passes `apiKey` into the generic OpenAI client and only adds `defaultHeaders: { "api-key": apiKey }`. In openai-node v6.34.0 the client still builds `Authorization: Bearer <apiKey>`, so Azure requests can carry the wrong auth shape instead of matching the existing Azure paths that omit Authorization.
  - confidence: 0.87
- **[P2] Normalize Azure embedding URLs before posting:** `extensions/memory-lancedb/index.ts:176-179`
  - body: This PR says users can point `baseUrl` at an Azure resource, but the client leaves `baseURL` unchanged and later posts to `/embeddings`. With a resource-root URL that produces `https://<resource>.openai.azure.com/embeddings?...`, while existing Azure surfaces normalize to Azure `/openai/v1` or deployment-scoped paths.
  - confidence: 0.78
- **[P2] Expose the Azure API version through config:** `extensions/memory-lancedb/index.ts:179`
  - body: The patch hard-codes `api-version=2024-02-01`, while `MemoryConfig.embedding`, the runtime key allowlist, and the plugin manifest have no `apiVersion` field. Azure deployments that require another supported version would still have no config path short of forking the plugin.
  - confidence: 0.9
- **[P2] Detect Azure endpoints by hostname and shared suffixes:** `extensions/memory-lancedb/index.ts:173`
  - body: `baseUrl?.includes(".openai.azure.com")` misses Azure host families already handled elsewhere, including `*.services.ai.azure.com` and `*.cognitiveservices.azure.com`, and it is not constrained to the parsed hostname. Those valid endpoints would still use non-Azure request shaping.
  - confidence: 0.88
