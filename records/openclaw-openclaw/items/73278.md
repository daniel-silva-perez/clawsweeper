## Summary

Keep open. Current main still has the two prompt sources that can steer an agent toward a manual Discord message-tool reply, and the remaining fix is already represented by open closing PRs; current main's newer message-tool-only default changes the exact default reproduction but does not resolve the prompt/product mismatch.

## What This Changes

The issue requests preventing Discord `autoThread` channel mentions from producing both a parent-channel `message.send` reply and a normal final response in the created thread.

## Best Possible Solution

Review and consolidate one canonical prompt/skill change that clearly separates normal current-conversation replies from explicit Discord messaging actions, while preserving the existing autoThread delivery mechanics and the newer source-reply visibility policy.

## Reproduction Assessment

Yes, with current main under the compatibility setting `messages.groupChat.visibleReplies: "automatic"`; the broad prompt inputs can still steer a parent-channel `message.send` while autoThread automatic delivery targets the created thread. The exact default 2026.4.26 reproduction no longer matches current defaults because guild/channel final replies are now message-tool-only by default.

## Solution Assessment

Yes as a direction: clarifying the Codex runtime guidance and/or Discord skill is narrower than changing autoThread routing, since the delivery plan already targets the created thread. The maintainer choice is which of the open PRs provides the best wording and test coverage.
