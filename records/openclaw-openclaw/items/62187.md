## Summary

Keep open. Current main still lacks the requested Feishu Bitable list-records filter/sort/field-selection parameters, so the PR is not obsolete, but the proposed patch is not ready as-is because the new tool schema advertises the structured sort shape used by the Bitable search contract while the implementation forwards the value unchanged to the list-records query parameter.

## What This Changes

The PR adds optional `filter`, `sort`, `field_names`, and `automatic_fields` parameters to `feishu_bitable_list_records` and forwards them to the Lark SDK list-records call.

## Best Possible Solution

Expose the server-side list-records controls in the Feishu plugin with dependency-correct tool schema text and focused tool-harness coverage; if maintainers want object-style sort/filter inputs, translate them through the correct Bitable API instead of forwarding them raw to `list`.

## Reproduction Assessment

Yes. A static reproduction is enough for the PR finding: inspect the PR schema at `extensions/feishu/src/bitable.ts:496` and the pinned Lark SDK types showing `list.params.sort` as a string while structured sort objects belong to the search contract.

## Solution Assessment

No. The pass-through direction is narrow and useful, but this branch currently teaches callers a sort shape that does not match the endpoint it calls; correcting the schema or translating the input is the safer solution before merge.

## Review Findings

Overall correctness: patch is incorrect

- **[P2] Advertise a valid list-records sort shape:** `extensions/feishu/src/bitable.ts:496`
  - body: The tool schema tells callers to pass `sort` as `[{"field_name":"Created Time","desc":true}]`, but this implementation forwards the value unchanged to `appTableRecord.list`.
