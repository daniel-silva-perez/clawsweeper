## Summary

Keep open. The issue remains a valid, recently reconfirmed Windows Docker gateway regression: v2026.4.27 and current main contain useful adjacent mitigations for runtime-deps storage, stale locks, Slack timeout tuning, and event-loop diagnostics, but the latest reporter data still shows a 103-second event-loop block with Slack/session fallout after the named-volume mitigation. Current main still has unbounded normal HTTP stage dispatch and unbounded WebSocket RPC handler completion, so there is not enough evidence to close this as implemented or duplicate-only.

## What This Changes

Investigate and fix a Windows 11 Docker Desktop gateway regression where HTTP routes, internal WebSocket RPC, Slack Socket Mode, and queued sessions stall behind event-loop or dispatch starvation after startup.

## Best Possible Solution

Keep this as an active Windows Docker gateway reliability investigation. Reproduce with the named runtime-deps volume layout, capture startup trace plus a CPU profile around the liveness warning, identify the synchronous or never-resolving stack, then land a narrow root-cause fix with coverage for normal HTTP routes, WebSocket RPCs, and Slack Socket Mode behavior under startup/runtime stalls.

## Reproduction Assessment

Yes. The reporter provided deterministic Windows 11 + Docker Desktop steps and reconfirmed the remaining failure on v2026.4.27 with event-loop delay metrics after removing the runtime-deps bind-mount variable; this review did not live-run that Windows setup.

## Solution Assessment

Unclear. Generic per-route or per-RPC timeouts may improve failure responses, but the best fix should remove or offload the event-loop blocking work that starves Slack and sessions rather than only masking it with client-side waits or diagnostics.
