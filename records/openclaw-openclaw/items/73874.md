## Summary

Keep open. Current main includes useful adjacent mitigations for Docker runtime-deps storage, stale locks, Slack ping/pong tuning, and event-loop diagnostics, but the latest reporter update still shows the core Windows Docker event-loop stall on v2026.4.27 and the current HTTP/WS dispatch paths still have no server-side bound for normal request stages or gateway RPC handlers.

## What This Changes

The issue asks maintainers to investigate and fix a Windows Docker Desktop gateway regression where HTTP/WS dispatch can accept connections but hang or starve after startup, breaking Slack and internal RPC delivery.

## Best Possible Solution

Keep this as an active Windows Docker gateway regression investigation. Maintainers should reproduce with the named runtime-deps volume layout, capture startup trace and CPU/profile data around the liveness warning window, identify the blocking stack or plugin/channel path, then land a narrow root-cause fix that preserves /healthz and /readyz behavior while restoring normal HTTP routes, WebSocket RPCs, and Slack Socket Mode delivery under startup load.

## Reproduction Assessment

Yes. The reporter provided concrete Windows 11 + Docker Desktop steps and later reconfirmed the remaining v2026.4.27 failure with named-volume runtime deps plus event-loop delay metrics; this review did not independently live-run that Windows setup.

## Solution Assessment

Unclear. Current main's named-volume storage, stale-lock hardening, Slack ping/pong tuning, readiness waits, and diagnostics are good mitigations, but the best final fix should identify and remove or offload the blocking work rather than only adding broader client-side waits or route-level timeouts.
