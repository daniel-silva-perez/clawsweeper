Keep open. Current main contains useful adjacent mitigations for probe ordering, Docker runtime-deps storage, stale runtime-deps locks, and event-loop diagnostics, but the remaining Windows Docker Desktop event-loop/dispatcher stall is not source-proven fixed. The reporter’s latest v2026.4.27 retest specifically says the named-volume runtime-deps fix helped startup while a 103-second event-loop delay, Slack disconnects, and stuck sessions still reproduced.

## What This Changes

The issue reports a Windows Docker Desktop gateway regression where HTTP/WS dispatch accepts connections but stalls after readiness, with Slack/session fallout and related runtime-deps, permissions, and migration symptoms.

## Best Possible Solution

Keep this as an active gateway/Docker regression investigation. The desired end state is a root-cause fix that removes or offloads the blocking startup/request work on Windows Docker Desktop, while preserving the named-volume runtime-deps improvements and keeping separate symptoms tracked separately.
