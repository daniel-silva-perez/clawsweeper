## Summary

Keep open. Current main has useful adjacent mitigations for Docker runtime-deps storage, stale runtime-deps locks, Slack ping/pong tuning, event-loop diagnostics, and client-side readiness waits, but it does not prove the remaining Windows Docker dispatcher/event-loop stall is fixed. The latest reporter update still shows v2026.4.27 blocking the event loop for 103 seconds after the named-volume mitigation, and the core HTTP/WS dispatch paths still await handlers without a server-side bound for normal routes or RPC handlers.

## What This Changes

The issue reports a Windows 11 Docker Desktop gateway regression where HTTP and WebSocket dispatch can stall after readiness, disrupting Slack Socket Mode and leaving sessions stuck despite the listener accepting connections.

## Best Possible Solution

Keep this as an active Windows Docker gateway regression investigation. Reproduce with the named runtime-deps volume layout, capture startup trace/timeline plus a CPU profile around the liveness warning window, identify the blocking stack or plugin/channel path, then land a narrow root-cause fix with coverage for normal HTTP routes, WebSocket RPCs, and Slack Socket Mode behavior under stalled startup work.

## Reproduction Assessment

Yes. The reporter provided concrete Windows 11 + Docker Desktop steps and later reconfirmed the remaining failure on v2026.4.27 with event-loop delay metrics after removing the runtime-deps bind-mount variable; this review did not live-run that Windows setup.

## Solution Assessment

Unclear. Current main's diagnostics, named-volume runtime-deps storage, Slack ping/pong tuning, and client readiness wait are useful mitigations, but the best final fix should remove or offload the blocking event-loop work rather than only increasing client-side waits or probe visibility.
