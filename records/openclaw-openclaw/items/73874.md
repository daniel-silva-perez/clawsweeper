## Summary

Keep open. Current main has several adjacent mitigations for Docker runtime-deps storage, stale locks, Slack timeouts, event-loop diagnostics, and client-side readiness waits, but the active reporter update still shows the core Windows Docker event-loop stall on v2026.4.27 and the current dispatch paths do not prove normal HTTP routes, WebSocket RPC handlers, or Slack delivery can survive the reported blocking stack.

## What This Changes

The issue reports a Windows Docker gateway regression where HTTP and WebSocket dispatch can stall after startup, causing loopback probes, Gateway RPCs, and Slack delivery to hang despite the listener being bound.

## Best Possible Solution

Keep this as an active Windows Docker gateway regression investigation. Maintainers should reproduce with the named runtime-deps volume layout, capture startup trace plus event-loop diagnostics and a CPU profile during the liveness warning window, identify the synchronous or long-running handler stack, then land a narrow fix that restores normal HTTP routes, WebSocket RPCs, and Slack Socket Mode delivery under startup load.

## Reproduction Assessment

Yes. The reporter provided concrete Windows 11 + Docker Desktop steps and later reconfirmed the remaining failure on v2026.4.27 with named-volume runtime deps plus event-loop delay metrics; this review did not independently live-run that Windows setup.

## Solution Assessment

Unclear. Current main's named-volume storage, stale-lock hardening, Slack timeout tuning, liveness/readiness diagnostics, and client-side event-loop readiness waits are useful mitigations, but the best final fix should identify and remove or offload the blocking work rather than only adding broader waits or route-level timeouts.
