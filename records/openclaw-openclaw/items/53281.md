## Summary

Keep open. Current main still lacks this PR's caller-facing `attachAs.mountPath` diagnostic; it only treats the value as a sanitized prompt hint while materializing files under the internal attachment directory. The PR is narrow and security-neutral, but it needs maintainer review/rebase because it changes user-visible tool result wording and its docs hunk targets an older docs placement.

## What This Changes

This PR adds a note to accepted native subagent spawns when attachments and `attachAs.mountPath` are provided, passes the mount path through once, and updates focused tests plus session-tool documentation.

## Best Possible Solution

Keep the hint-only attachment materialization contract. If maintainers want the diagnostic, land a rebased narrow version that appends it only when a sanitized mount-path hint is actually used with attachments, updates the current docs surfaces, and preserves focused tests without implementing custom mounts.
