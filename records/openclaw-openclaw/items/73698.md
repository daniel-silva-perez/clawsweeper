## Summary

Keep this issue open. Current main strongly supports the maintainer explanation that `dmPolicy: "open"` plus `allowFrom: ["*"]` intentionally allowed an unknown Telegram user to message the bot, and docs/audit already warn about public DMs and shared DM sessions. But the report is same-day, security-impacting, already has maintainer follow-up/assignment, and still needs sanitized Telegram update/log evidence before maintainers can close it as expected behavior versus a Telegram routing/session presentation bug.

## Best Possible Solution

Keep this open for maintainer/security follow-up. Ask the reporter for sanitized Telegram update payloads, gateway logs for 2026-04-28 23:00-23:15 GMT+7, full redacted Telegram/session/tool-policy config including `session.dmScope`, session key shown in logs/UI, and whether Telegram Business or bridge features were involved. If the evidence confirms expected public-DM plus shared-main-session behavior, close later with guidance to use `dmPolicy: "allowlist"`, explicit numeric `allowFrom`, `session.dmScope: "per-channel-peer"`, stricter tool policy/exec approvals, and `openclaw security audit`; if logs show cross-sender routing under isolated DM scope, fix the Telegram routing/session path.
