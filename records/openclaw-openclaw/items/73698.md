## Summary

Keep this issue open. The reported `dmPolicy: "open"` / `allowFrom: ["*"]` setup explains why an unknown Telegram sender could reach the bot, and current docs plus audit warnings cover that dangerous configuration. However this is a same-day security-impacting Telegram report with maintainer follow-up already active, and the remaining question about forwarded context/shared session presentation still needs sanitized raw Telegram updates or logs before maintainers can confidently close it as expected behavior.

## Best Possible Solution

Keep this open for maintainer/security follow-up. Ask for sanitized Telegram update payloads for the affected messages, gateway logs for the incident window, the effective `session.dmScope`, the OpenClaw session key where the messages appeared, redacted tool/exec approval policy, and whether Telegram Business or a chat bridge was involved. If the evidence shows expected `dmPolicy: "open"` plus shared main-session behavior, close later with allowlist, `session.dmScope: "per-channel-peer"`, and tool-policy guidance; if it shows cross-sender routing under an isolated scope, fix or link the canonical Telegram routing/session bug.
