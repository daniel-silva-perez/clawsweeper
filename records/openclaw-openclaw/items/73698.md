## Summary

Keep this issue open. The reporter's `dmPolicy: "open"` / `allowFrom: ["*"]` config explains why an unknown Telegram sender could reach the bot, and current docs/audit already warn about open DMs plus shared DM sessions. However this is a same-day security-impacting Telegram report, maintainer follow-up is active, and the only newly found related main-branch fix references this issue without fully closing the Telegram incident or being in a shipped release.

## Best Possible Solution

Keep this open for maintainer/security follow-up. Ask for sanitized Telegram update payloads, gateway logs for 2026-04-28 23:00-23:15 GMT+7, the effective `session.dmScope`, redacted tool/exec policy, the session key shown in logs/UI, and whether Telegram Business or chat-bridge features were involved. If the evidence shows expected `dmPolicy: "open"` plus shared main-session behavior, close later with allowlist, `session.dmScope: "per-channel-peer"`, and tool-policy guidance; if it shows cross-sender routing under isolated scope, fix the Telegram routing/session path or link the canonical fix after it ships.
