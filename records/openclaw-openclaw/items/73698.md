## Summary

Keep this issue open. Current main explains why the reporter's original `dmPolicy: "open"` plus `allowFrom: ["*"]` configuration allowed an unknown Telegram sender to reach the bot, and it has warnings plus related routing/provenance hardening. It does not yet prove whether the reported forwarded-context/session presentation was expected shared-main-session behavior, Telegram forwarding metadata, a bridge/business-chat edge case, or a Telegram routing bug.

## Best Possible Solution

Keep this open for maintainer/security follow-up. Ask for sanitized Telegram update payloads for the affected messages, gateway logs for the incident window, effective `session.dmScope`, the session key/transcript where the messages appeared, redacted exec/tool approval policy, and whether Telegram Business, forwarding, or a chat bridge was involved. If the evidence shows expected `dmPolicy: "open"` plus shared `session.dmScope: "main"`, close later with allowlist, `per-channel-peer`, and tool-policy guidance; if it shows cross-sender routing under an isolated scope, fix the Telegram routing/session path in the Telegram plugin or link the canonical routing bug.
