## Summary

Keep this issue open. Current main explains the likely mechanism for an unknown Telegram sender reaching the bot: the reported `dmPolicy: "open"` plus `allowFrom: ["*"]` intentionally admits all Telegram DMs, and the default DM session scope shares direct messages in the main session. However, this is a same-day security-impacting Telegram incident with maintainer follow-up/assignment in the provided thread data, and the issue still needs sanitized Telegram update/log evidence before maintainers can confidently rule out a Telegram routing or session-presentation bug.

## Best Possible Solution

Keep this open for maintainer/security follow-up. Ask the reporter for sanitized Telegram update payloads for message IDs 1401, 1403, and 1409; gateway logs for 2026-04-28 23:00-23:15 GMT+7; the effective `session.dmScope`; redacted exec/tool approval policy; and whether Telegram Business, chat-bridge, DM topics, or forwarded-message flows were involved. If the evidence matches expected `dmPolicy: "open"` plus shared `session.dmScope: "main"`, close later with guidance to use `allowlist`, `session.dmScope: "per-channel-peer"`, and stricter exec approvals/sandboxing. If it shows cross-sender routing under isolated scope, fix the Telegram routing/session path or link the canonical fix after it ships.
