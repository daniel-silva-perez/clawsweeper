## Summary

Keep open. Current main still accepts arbitrary `tools.web.search.provider` strings at config-validation time, while runtime only warns and falls back when the configured provider is unknown; the manifest contract already exposes plugin-owned `webSearchProviders`, so this remains a real, narrow validation gap.

## What This Changes

Validate `tools.web.search.provider` against built-in and plugin-manifest web search provider IDs while preserving compatibility for stale or missing plugins.

## Best Possible Solution

Add a manifest-backed validation step in the plugin-aware config validation path that derives known provider IDs from `contracts.webSearchProviders`, accepts bundled and installed plugin providers, rejects clear unknown typos with allowed-value context, and preserves warning-only handling when stale plugin evidence exists.
