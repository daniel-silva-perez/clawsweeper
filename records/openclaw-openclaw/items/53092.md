## Summary

Keep open. Current main still accepts arbitrary `tools.web.search.provider` strings during config validation, while runtime only warns and falls back when the configured provider is unknown; the manifest contract already exposes plugin-owned `webSearchProviders`, so the reported validation gap remains real and narrow.

## What This Changes

Request that `tools.web.search.provider` validation derive allowed provider IDs from built-in and plugin-registered web search providers instead of accepting any string.

## Best Possible Solution

Add a manifest-backed provider validation step in the plugin-aware config validation path that derives known search provider IDs from bundled and installed plugin metadata, reports clear unknown-provider errors with allowed values for likely typos, and preserves a documented warning path for stale plugin-owned config where startup resilience is intended.
