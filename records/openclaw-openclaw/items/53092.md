## Summary

Keep open. Current main still does not validate `tools.web.search.provider` against built-in plus extension-registered web search provider IDs. The manifest contract exists and bundled provider manifests declare IDs, but the base/generated config schema still accepts any string, the plugin-aware config validation path does not check that field, and runtime behavior still treats unknown configured providers as warning-plus-auto-detect fallback rather than a config validation failure. The provided related items do not identify a canonical duplicate or merged fix.

## Best Possible Solution

Keep this issue open. Add plugin-aware validation for `tools.web.search.provider` in the config validation path, deriving valid IDs from built-in and installed plugin manifest `contracts.webSearchProviders`. Preserve warning-only handling for stale missing-plugin compatibility, and add focused tests for built-in IDs, extension-registered IDs, invalid typos, and stale removed-plugin configs.
