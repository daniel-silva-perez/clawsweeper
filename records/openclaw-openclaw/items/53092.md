## Summary

Keep open. Current main still accepts any `tools.web.search.provider` string in the base config schema and handles unknown configured providers as warning-only runtime fallback rather than plugin-aware config validation. The prior ClawSweeper comment matches the current source state at `5828dcdb05aad587bff831467e7fa51f73312ff8`; I found no merged canonical fix or duplicate in the provided related-item context.

## Best Possible Solution

Keep this issue open. Add plugin-aware validation for `tools.web.search.provider` in the config validation path, deriving known IDs from bundled and installed plugin manifests via `contracts.webSearchProviders`. Reject clear unknown provider typos with an actionable config issue, preserve warning-only behavior for stale/missing-plugin compatibility, and add focused tests for built-in IDs, extension-registered IDs, invalid typos, and stale missing-plugin configs.
