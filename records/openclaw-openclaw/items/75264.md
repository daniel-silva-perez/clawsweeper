## Summary

Keep open. The report is recent, detailed, and still matches current `main`: claude-cli is configured as a live stdio backend with bundled MCP, OpenClaw excludes native file/exec tools from that MCP loopback, the no-output watchdog can produce a minute-scale timeout, and Telegram still sends a generic fallback after dispatch errors. I did not find a current-main source, test, docs, or history change after the report that fixes the wrapper-path `Read` stall or improves the external Telegram error.

## What This Changes

This issue asks OpenClaw to avoid a multi-minute claude-cli and Telegram stall when a wrapper path triggers a native `Read`, either by executing the approved wrapper or failing fast with a clear, sanitized tool error.

## Best Possible Solution

Keep this open until maintainers define the supported workflow and close the diagnostic gap: claude-cli native-tool stalls should fail fast with useful sanitized diagnostics, Telegram should surface a bounded actionable timeout/tool message instead of only a generic fallback, and wrapper execution should be routed or documented through the intended ACP or node-exec path without loosening file-read or exec approval boundaries.
