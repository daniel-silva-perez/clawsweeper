## Summary

Keep this issue open. The report is recent, detailed, and matches current-main surfaces where claude-cli uses bundle MCP, Claude live stdio can wait for a multi-minute no-output watchdog, and external chat failures default to generic text; I did not find current-main or release evidence that the wrapper-path Read stall is already fixed. The related #72338 gateway CPU-stall report is useful context but remains broader and open, so it should not supersede this more specific CLI-backend/Telegram failure.

## What This Changes

The issue reports that Telegram turns using the claude-cli Sonnet 4.6 backend stall for about three minutes during a Claude file Read against an approved wrapper path and only return a generic Telegram error.

## Best Possible Solution

Keep this open and split the investigation into two maintainable outcomes: make claude-cli bundle-MCP/native-tool stalls fail fast with useful diagnostics when Claude attempts an unsupported file read, and clarify or enforce the supported route for Telegram-to-wrapper execution, likely ACP or OpenClaw `exec host=node` rather than an ambiguous Claude native `Read` path.

## Solution Assessment

Unclear. There is no proposed patch, and the safest fix may be either a targeted claude-cli/MCP watchdog and diagnostic improvement or a docs/runtime guard that steers this workflow to ACP/node exec instead of allowing a silent Claude native `Read` stall.
