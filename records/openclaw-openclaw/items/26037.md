## Summary

Keep this issue open. Current main has partial Qwen/Bailian Coding Plan support and a Qwen `enable_thinking` stream wrapper, but the requested reasoning path is still incomplete for the Coding Plan models named by the reporter: `qwen3.5-plus`, `glm-5`, and `kimi-k2.5` are still cataloged as non-reasoning, the wrapper only patches models whose normalized catalog says `reasoning: true`, and the public custom-provider compat schema still does not accept a Qwen top-level thinking format.

## Best Possible Solution

Implement a focused Qwen/Bailian Coding Plan provider fix: verify the Coding Plan models that support DashScope `enable_thinking`, mark only those catalog entries or compat paths as thinking-capable, ensure OpenClaw thinking levels map to top-level `enable_thinking` for both bundled Qwen and documented custom-provider configs, update Qwen docs/config schema as needed, and add targeted regression tests for catalog metadata, payload construction, and config validation.
