## Summary

Keep this issue open. Current main has Qwen Coding Plan support and a Qwen-owned `enable_thinking` wrapper, but the requested Bailian/Coding Plan reasoning path is still incomplete for `qwen3.5-plus`, `glm-5`, and `kimi-k2.5`; those catalog entries remain non-reasoning, the wrapper is gated on `model.reasoning`, and the custom-provider `compat.thinkingFormat: "qwen"` workaround shown in Alibaba docs is not accepted by OpenClaw's current config schema.

## Best Possible Solution

Keep this issue open and implement a focused Qwen/Bailian Coding Plan fix: add model-specific thinking support for the Coding Plan models that Alibaba documents, map OpenClaw thinking controls to DashScope `enable_thinking`, avoid unsupported `thinking_budget` fields, decide whether the public config schema should accept the documented `qwen` thinking format for custom Bailian providers, update Qwen docs, and add regression tests for catalog metadata, payload construction, and config validation.
