## Summary

Keep this issue open. Current main has Qwen/Bailian Coding Plan auth and a Qwen `enable_thinking` stream wrapper, but the requested reasoning path for `qwen3.5-plus`, `glm-5`, and `kimi-k2.5` is still incomplete: those catalog entries are not reasoning-enabled, the wrapper only patches reasoning models, and the public compat schema still rejects the `qwen` thinking format shown in current Qwen/Bailian setup docs. 

## Best Possible Solution

Implement a focused Qwen/Bailian Coding Plan provider fix: verify the Coding Plan models that support DashScope `enable_thinking`, mark only those catalog entries or compat paths as thinking-capable, ensure OpenClaw thinking levels map to top-level `enable_thinking` for both bundled Qwen and the documented custom-provider path, update Qwen docs/config schema as needed, and add regression tests for catalog metadata, payload construction, and config validation.
