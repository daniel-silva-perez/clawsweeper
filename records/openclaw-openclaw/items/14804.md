## Summary

Keep open. Current main still lacks first-class ephemeral or tmpfs-backed sandbox workspace lifecycle support, and PR #73738 is an open, unmerged implementation candidate with closing syntax for this issue, so this should stay open for maintainer/secops review rather than conservative cleanup closure.

## What This Changes

Add an option for isolated sandbox sessions to use disposable or tmpfs-backed workspaces so each run starts clean and avoids persistent side effects.

## Best Possible Solution

Review and either land or close PR #73738 after confirming an opt-in sandbox workspace lifecycle contract with persistent as the backwards-compatible default, safe run-scoped cleanup on success and setup failure, aligned config/schema/generated metadata/docs/tests, and an explicit decision on tmpfs-backed workspace support.

## Reproduction Assessment

Yes for the current-main feature gap; this is not a runtime bug reproduction. The focused check is that workspaceLifecycle is absent from the config type and strict schema, while sandbox workspace resolution still derives stable per-scope directories and preserves existing workspace contents.

## Solution Assessment

Unclear until PR #73738 is reviewed. The maintainable direction is an opt-in lifecycle setting with persistent defaults and cleanup constrained to run-owned paths; the safest next step is maintainer/secops review of that linked PR rather than a separate cleanup action.
