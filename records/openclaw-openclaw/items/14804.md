## Summary

Keep open. Current main still does not expose a first-class ephemeral or tmpfs-backed sandbox workspace lifecycle, and the linked implementation candidate remains open and unmerged, so this needs maintainer and secops review rather than conservative cleanup closure.

## What This Changes

Request an opt-in ephemeral or tmpfs-backed workspace mode for isolated sandbox sessions so each run starts from a clean disposable filesystem.

## Best Possible Solution

Review and either land or close PR #73738 after confirming a backwards-compatible opt-in lifecycle contract, safe run-owned cleanup semantics, aligned config/schema/generated metadata/docs/tests, and an explicit decision on tmpfs-backed workspace support.

## Reproduction Assessment

Yes for the current-main feature gap; this is not a runtime bug reproduction. The focused check is that workspaceLifecycle is absent from the config type and strict schema while sandbox workspace resolution still derives stable per-scope directories that preserve existing contents.

## Solution Assessment

Unclear until PR #73738 is reviewed. The maintainable direction is an opt-in lifecycle setting with persistent defaults and cleanup constrained to run-owned paths; the safer next step is maintainer/secops review of that linked PR, not a separate cleanup action.
