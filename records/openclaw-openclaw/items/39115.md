## Summary

Keep this PR open. Current main still does not implement the requested global `agents.defaults.reasoningDefault` surface: the runtime defaults schema/type omit the key, generated config metadata only exposes per-agent `agents.list[].reasoningDefault`, and reply/status resolution paths still read session or per-agent reasoning defaults rather than the global defaults object. The PR is active, has recent contributor follow-up, and prior maintainer discussion says the intended behavior is still being evaluated.

## Best Possible Solution

Keep this PR open for maintainer review. A mergeable solution should settle the product/security policy, then implement `agents.defaults.reasoningDefault` consistently across strict schema, exported types, generated config metadata, docs, reply resolution, current-level status, direct status, and status text. It should preserve explicit `off` over model-derived reasoning defaults and include tests for directive, session, per-agent, global, model-default precedence, plus authorized and unauthorized sender behavior. If maintainers reject the feature, close it with that explicit product decision rather than as stale or implemented.

## Risks / Open Questions

- Security review should stay explicit: this PR changes reasoning visibility/default behavior in authorization-sensitive reply and status paths. Reviewers should verify unauthorized senders cannot enable reasoning with inline `/reasoning` and that configured defaults only apply in the intended owner/operator/authorized contexts.
- Product behavior is still unsettled by maintainer discussion. A global reasoning default can affect cost, latency, and reasoning exposure across channels, so it needs maintainer product judgment before merge or rejection.
- Config contract drift is easy here unless schema, exported types, generated schema/help/labels, docs, reply resolution, current-level status, direct status, and tests are updated together.
