## Summary

Keep this PR open. Current main still has the stale runtime-source projection bug, so the PR is not obsolete; however, the proposed fix is not merge-ready because it still falls back to the stale runtime source for invalid-but-readable config files and it rereads the config during the same write operation.

## What This Changes

The PR changes `src/config/io.ts` so runtime-derived config writeback prefers the latest valid existing on-disk snapshot as the projection base and adds a regression test for preserving disk-added `session.dmScope`.

## Best Possible Solution

Land a focused config-writer fix that chooses the latest existing on-disk resolved/source snapshot before computing the runtime patch, falls back only for truly missing or unreadable files, reuses that same snapshot for the inner write, and keeps the `session.dmScope` stale-snapshot regression test.

## Reproduction Assessment

Yes. A high-confidence reproduction is to set a runtime snapshot from an older source config, add `session.dmScope` directly on disk, then call `writeConfigFile(loadConfig())`; current main projects the write onto the stale source snapshot and the merge-patch code can delete the disk-only key.

## Solution Assessment

No, not as written. The PR is the right narrow area and includes the right main regression test, but the `latestSnapshot.valid` guard and second disk read leave correctness gaps in the same config-preservation contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Use readable snapshots even when validation fails:** `src/config/io.ts:1519-1523`
  - body: When the current config file exists but is validation-invalid, this branch falls back to `runtimeConfigSourceSnapshot`, so a runtime write can still overwrite disk-only keys that were added after the runtime snapshot. Use the readable snapshot's resolved/source payload as the projection base and reserve fallback for missing or unreadable files.
  - confidence: 0.86
- **[P3] Reuse the snapshot used for projection:** `src/config/io.ts:1518`
  - body: This added read computes `nextCfg`, but the inner writer then reads the config file again. If the file changes between those reads, `nextCfg` can be based on an older disk state while the inner merge uses the newer one, turning keys added in the gap into deletions. Thread this snapshot through the existing `baseSnapshot` path or otherwise keep the write based on one snapshot.
  - confidence: 0.72
