## Summary

Current main still contains the reported failure chain for legacy/manual shell profiles that run `openclaw gateway start`: the gateway exec PATH probe still invokes a login shell before the actual command, and `gateway start` still restarts an installed/loaded managed service instead of no-oping when it is already running. The linked #62086 discussion is useful context, but that closed report describes a different SIGTERM path where no shell-profile autostart was present, so it does not supersede this issue.

## Best Possible Solution

Keep this issue open. The conservative fix is to make `openclaw gateway start` idempotent for an already-running managed service and to prevent the exec login-shell PATH probe from triggering startup side effects, for example by marking the probe environment with an exec-specific guard or replacing it with a safer PATH discovery path. Add regression coverage for `host=gateway` PATH probing plus an installed/running service so a profile line invoking `openclaw gateway start` cannot restart the gateway during exec.
