## Summary

Keep open. Current main still has the reported failure chain: host exec performs a login-shell PATH probe before the actual command, and `openclaw gateway start` still maps to a managed-service restart. The later comment about SIGTERM without shell-profile autostart is useful context and appears to be a separate path tracked by #62086, but it does not invalidate this original profile-sourcing bug.

## What This Changes

The issue requests that exec tool invocations stop restarting the Gateway when login shell startup files invoke `openclaw gateway start`.

## Best Possible Solution

Keep this open and fix the source path narrowly: make `openclaw gateway start` idempotent for an already-running managed service, and make the host exec login-shell PATH probe unable to trigger gateway-start side effects, either by marking the probe environment or by replacing the probe with a safer PATH discovery path. Add regression tests for both the service-start no-op and the exec PATH probe environment.

## Reproduction Assessment

Yes. The source path is high-confidence: with a managed service active and a login profile invoking `openclaw gateway start`, `host=gateway` exec runs `shell -l -c env -0` before the command and can reach the restart-backed service start path. I did not run the live systemd mutation in this read-only review.

## Solution Assessment

No. Current main's `OPENCLAW_SHELL=exec` marker is applied only to the actual exec process, not to the login-shell PATH probe, and `gateway start` still restarts managed services. The safer fix is a small two-sided change: idempotent managed start plus a side-effect-safe PATH probe.
