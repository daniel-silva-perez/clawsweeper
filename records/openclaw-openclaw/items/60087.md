## Summary

Keep open. Current main still has the core failure chain: host exec performs a login-shell PATH probe before running the command, and `openclaw gateway start` still maps to a managed-service restart rather than an idempotent start. The normal exec command now uses a non-login shell and sets `OPENCLAW_SHELL=exec`, but that marker is applied after the separate login-shell PATH probe, so it does not close the profile-sourcing case reported here.

## Best Possible Solution

Keep this issue open and fix both sides conservatively: make `openclaw gateway start` idempotent for an already-running managed service, and prevent the host exec login-shell PATH probe from triggering startup side effects, for example by marking the probe environment or replacing it with a safer PATH discovery path. Add regression coverage around `host=gateway` PATH probing plus an installed/running service so a profile line invoking `openclaw gateway start` cannot restart the gateway during exec.
