## Summary

Keep this issue open. Current main still constructs the core tool list before applying `toolsAllow`, and `createOpenClawCodingTools()` has no runtime force-allow option that could restore `exec`, `read`, or `write` after a restrictive config allowlist removes them. The issue is also paired with open PR #60842, which uses closing syntax for this issue, so maintainers should review, land, replace, or close that implementation candidate rather than closing this report early.

## What This Changes

Embedded and isolated cron runs should let runtime `toolsAllow` re-expose explicitly requested core tools through restrictive config allowlists such as `tools.allow: ["group:openclaw"]`, while preserving stronger deny and runtime restrictions.

## Best Possible Solution

Keep this issue open and review #60842, or replace it with a narrow maintainer patch that forwards runtime `toolsAllow` into core tool construction as a force-allow for known core tools only. The end state should preserve deny precedence and runtime capability gates, then add focused regression coverage for restrictive `group:openclaw` config plus runtime `toolsAllow` in the tool-constructor and embedded-runner forwarding surfaces.

## Reproduction Assessment

Yes. The source path is deterministic: with `tools.allow: ["group:openclaw"]`, `exec`, `read`, and `write` are removed inside `createOpenClawCodingTools()` before `applyEmbeddedAttemptToolsAllow()` runs, so a runtime `toolsAllow` post-filter cannot recover them.

## Solution Assessment

Yes, the maintainable fix direction is to pass a narrow runtime force-allow into the construction/policy merge rather than relying on the final filter. A final post-filter should remain authoritative for trimming the run's tool set, but it is not sufficient to restore tools removed earlier.
