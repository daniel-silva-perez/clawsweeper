## Summary

Current main still does not implement Slack tool-level `assistant.threads.setStatus` updates. The Slack plugin sets the native assistant thread status to the static `"is typing..."` value at reply start and clears it on stop; tool-start events are available and already used for Slack reactions/draft-preview progress, but they do not call `setSlackThreadStatus` with tool-specific labels. The closed unmerged PR #33420 addressed this directly, while related open items #42555, #67129, and #73054 cover adjacent configurable typing/status or preview behavior and do not supersede this request.

## Best Possible Solution

Keep this issue open and implement the narrow Slack-owned wiring: add a Slack-local tool-name-to-human-label helper, call `ctx.setSlackThreadStatus` from Slack dispatch on tool `start` events when an assistant thread target exists, dedupe repeated labels, preserve start/clear behavior, and add focused Slack tests for the status call sequence. Keep #42555/#67129 as separate configurable status/loading-message product discussions and #73054 as separate draft-preview behavior.
