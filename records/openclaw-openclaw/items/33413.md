## Summary

Current main still does not implement Slack tool-level `assistant.threads.setStatus` updates. Slack dispatch sets the assistant thread status to the static `"is typing..."` at reply start and clears it at stop; tool-start events are available and used for reactions/draft preview progress, but not for assistant thread status labels. The direct implementation PR #33420 is closed unmerged, while adjacent open work #42555 and #73054 does not supersede this request.

## Best Possible Solution

Keep this issue open and implement the narrow Slack-owned wiring: add a small Slack-local tool-name-to-status-label helper, update `ctx.setSlackThreadStatus` from Slack dispatch on tool-start when an assistant thread target exists, keep the existing start/clear behavior, avoid update-phase spam, and add focused Slack coverage for the status call sequence. If maintainers decide that draft-preview progress or configurable loading messages are the intended UX instead, document that product decision and link a canonical superseding item before closing this issue.

## Clownfish Work Prompt

Implement Slack tool-level assistant thread status updates for this issue. Boundary: keep the fix Slack-owned if possible; do not add a new core/plugin seam unless the existing `onToolStart` callback is insufficient.

Expected behavior: when Slack is processing a reply in an assistant thread, keep the existing `"is typing..."` start status and final clear, but update `assistant.threads.setStatus` before each new tool starts with a human-readable label such as `exec -> "Running command..."`, `web_search -> "Searching the web..."`, `Read -> "Reading files..."`, and a reasonable fallback for unknown tools. Avoid Slack API spam: use only `payload.phase === "start"` or dedupe repeated labels, because current core emits `onToolStart` for both `start` and `update` phases.

Likely implementation: add a small helper in `extensions/slack/src/monitor/message-handler/dispatch.ts` near the Slack dispatch layer, then extend the existing `replyOptions.onToolStart` handler to call `ctx.setSlackThreadStatus({ channelId: message.channel, threadTs: statusThreadTs, status: label })` when a thread target exists and source replies are not tool-only. Preserve status reactions, draft-preview progress, typing reactions, and the final status clear.

Add or update focused coverage in `extensions/slack/src/monitor.tool-result.test.ts` so a simulated tool start produces a `setStatus` sequence containing `"is typing..."`, a tool-level label, and then `""`. Update `docs/channels/slack.md` if the user-visible behavior is documented. Add a changelog entry if maintainers expect one for this Slack UX behavior, crediting the reporter/contributor and not Codex.

Related refs: this issue, closed unmerged PR #33420, adjacent open PRs #42555 and #73054. Do not change Slack draft-preview semantics from #73054 and do not implement configurable persona/status messages from #42555 in this PR.
