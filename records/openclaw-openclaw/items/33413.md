## Summary

Keep this issue open. Current main still sets Slack assistant thread status to the static "is typing..." text at reply start and clears it at reply end. Tool-start events are available and Slack dispatch already handles them for status reactions and draft-preview progress, but it still does not map tool names to `assistant.threads.setStatus` labels. The direct implementation PR #33420 is closed unmerged; #42555, #67129, and #73054 are related Slack status/preview work, but none is a merged or canonical replacement for this requested assistant-thread status behavior.

## Best Possible Solution

Implement the narrow Slack-owned wiring: keep core generic, add a small Slack-local tool-name-to-status-label helper, and update `ctx.setSlackThreadStatus` from Slack dispatch on tool start when an assistant thread target exists. Keep the existing start/clear behavior, avoid spamming Slack on update phases, and add focused Slack coverage asserting the status sequence includes a tool-level label before the final clear. If maintainers instead choose draft-preview progress or configurable loading messages as the intended UX, document that product decision and link a canonical superseding item before closing this issue.
