## Summary

Keep this PR open. Current main still has the Control UI basePath favicon problem: the served HTML keeps root-relative icon hrefs, basePath routing ignores root icon requests, and the index HTML serving helper does not rewrite icon links. The PR’s intended fix is narrow and the author addressed the earlier `$` replacement review comment, but the branch is stale against current CSP-aware index serving and needs a rebase that preserves inline script hash CSP headers before merge.

## What This Changes

The PR adds a Control UI HTML-serving helper that prepends the configured basePath to favicon and apple-touch-icon hrefs, plus gateway tests for basePath rewrite and no-basePath behavior.

## Best Possible Solution

Rebase the PR onto current main so favicon/apple-touch-icon href rewriting happens in the final `serveResolvedIndexHtml` response body while preserving `computeInlineScriptHashes` and `buildControlUiCspHeader`. Keep focused coverage for direct index serving, SPA fallback, basePath and no-basePath cases, untouched non-icon hrefs, and CSP hash preservation.

## Reproduction Assessment

Yes. The current-main source gives a high-confidence reproduction path: configure a Control UI basePath, open the prefixed UI, and the browser resolves the root-relative icon hrefs outside that prefix while the router ignores those root icon paths.

## Solution Assessment

No, not as currently proposed against current main. The favicon rewrite is the right narrow direction, but the branch predates the CSP-aware HTML helper and should be rebased so the new rewrite wraps the final body without removing CSP hash handling.
