## Summary

Keep this issue open. Current main still renders the macOS Talk Mode surface as the generic orb and only plumbs Talk voice settings plus `ui.seamColor`, while the resolved assistant identity/avatar contracts already exist in gateway code and are not consumed by the macOS overlay.

## What This Changes

Request to make the macOS Talk Mode overlay render the configured assistant avatar when available, while preserving the existing animated voice-state visuals and orb fallback.

## Best Possible Solution

Add a narrow macOS Talk Mode identity layer: resolve the active session or default agent identity through the existing `agent.identity.get` and avatar route contracts, store only the render-safe avatar state needed by the overlay, render a circular cropped avatar inside the existing rings/arcs when available, and fall back to the current orb on missing identity, unsupported avatar type, failed image loading, or auth failure.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: configure `ui.assistant.avatar`, enable macOS Talk Mode, and the overlay still has no code path that fetches identity or renders an avatar, so it will display the generic orb. I did not run a live macOS UI session because this review is read-only.

## Solution Assessment

Yes, the requested direction is the right narrow product fix, but it should reuse the existing gateway identity/avatar contracts rather than adding ad hoc `ui.assistant.avatar` parsing directly to the overlay. Keeping the old orb as the failure fallback preserves current behavior.
