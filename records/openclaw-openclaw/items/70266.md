## Summary

Keep this issue open. Current main still renders the macOS Talk Mode overlay as the generic Circle-based orb, while the existing assistant identity/avatar gateway contract is available but not consumed by the macOS overlay; the reporter's follow-up correctly distinguishes this from Control UI chat identity work.

## What This Changes

Request to make the macOS Talk Mode overlay render the configured assistant avatar when available, while preserving the existing animated voice-state rings and falling back to the default orb.

## Best Possible Solution

Add a narrow macOS Talk Mode identity path: fetch the active or default assistant identity through the existing agent.identity.get/avatar-source contract, cache only render-safe avatar state needed by the overlay, render a circular cropped avatar inside the current rings/arcs when available, and keep the current orb as the fallback for missing identity, unsupported avatar type, image-load failure, or authorization failure.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: configure ui.assistant.avatar, open macOS Talk Mode, and current main has no code path that fetches identity or renders an avatar in TalkOverlayView/TalkOrbView, so the generic orb remains.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix, but it should consume agent.identity.get and the avatar route/source metadata rather than parsing ui.assistant.avatar directly in Swift. Preserving the existing orb fallback keeps the change backward-compatible.
