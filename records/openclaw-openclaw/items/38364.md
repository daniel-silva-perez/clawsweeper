## Summary

Current main still does not implement the central request: structured reply-to metadata is not accepted by the Control UI `chat.send` path, `chat.send`/`sessions.send` schemas are closed without reply fields, and the Gateway does not map any parent/reply target into the existing agent `MsgContext.ReplyTo*` fields. The no-output side is partially mitigated by current `NO_REPLY` filtering, but that does not close the reply metadata gap.

## Best Possible Solution

Keep this open and implement one focused additive change: add validated reply metadata to `chat.send` and likely `sessions.send`, capture a selected parent/reply target in Control UI, map it in the Gateway into existing `MsgContext.ReplyTo*` fields with bounded/sanitized parent content, and add focused protocol, Gateway, and UI tests while preserving current silent-output filtering.

Implement a focused fix for the open Control UI reply metadata issue. Add an additive, bounded reply metadata object to `chat.send` and likely `sessions.send` rather than changing existing fields. Add focused tests for protocol rejection/acceptance, Gateway `MsgContext` mapping, `sessions.send` forwarding, and Control UI request payload/state behavior.
