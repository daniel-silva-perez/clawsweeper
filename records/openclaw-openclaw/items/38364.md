## Summary

Current main still does not implement the central reply-metadata request. `chat.send` and `sessions.send` are closed protocol schemas with no reply-target fields, `sessions.send` forwards no reply metadata into `chat.send`, Gateway builds `MsgContext` without populating the existing `ReplyTo*` fields, and Control UI send state/request has no outbound selected reply target. The no-output side has current `NO_REPLY` suppression coverage, but that does not resolve the missing structured reply metadata path.

## Best Possible Solution

Keep this open and implement a focused additive change: add validated reply metadata to `chat.send` and likely `sessions.send`, capture a selected parent/reply target in the Control UI composer, map that metadata in Gateway `chat.send` into the existing `MsgContext.ReplyTo*` fields with bounded sanitized quoted content, update protocol/docs/tests, and preserve current silent-output filtering.
