## Summary

Keep open. Current main still lacks REST-native session reset/status/compact/model/reasoning endpoints for `/v1/chat/completions` integrations. The Gateway RPC/session logic exists, but the HTTP router and public HTTP docs still expose only chat/models/embeddings/responses, tools invoke, and adjacent session kill/history helpers.

## Best Possible Solution

Keep this open as the specific Gateway HTTP session-management tracker. The best implementation is a focused authenticated REST surface for reset/status/compact/model/reasoning that maps onto existing `sessions.resolve`/`sessions.list`, `sessions.reset`, `sessions.compact`, and `sessions.patch` semantics, preserves admin/operator method scopes, adds route/request/response tests, and updates the public HTTP docs. Do not re-enable slash-command interception on `/v1/chat/completions`; coordinate read-only status/query shape with #47826.

- There is overlap with broader HTTP query work in #47826, so naming and response shapes should be coordinated rather than growing two inconsistent REST surfaces.
