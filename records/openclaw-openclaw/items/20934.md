## Summary

Current main still does not expose REST-native session reset/status/compact/model/reasoning endpoints for HTTP `/v1/chat/completions` integrations. The Gateway already has underlying session RPC/control-plane methods, and there are adjacent HTTP session helpers for kill/history, but the public HTTP router and OpenAI-compatible HTTP docs still omit the requested `/v1/sessions/*` management surface. The related direct implementation PR #23941 was closed unmerged, so this issue remains valid as the canonical REST session-management tracker.

## Best Possible Solution

Keep this issue open as the canonical tracker for REST-native session management on the Gateway HTTP API. The best implementation path is a focused, authenticated HTTP API that maps REST requests onto the existing `sessions.reset`, `sessions.resolve`/`sessions.list`, `sessions.compact`, and `sessions.patch` semantics, with route predicates, bearer/scoped auth behavior, request/response schemas, tests, and public docs updated together. The closed #23941 PR is useful design context but should not be treated as shipped or merged.

## Clownfish Work Prompt

Implement REST-native session management endpoints for HTTP integrations tracked by this issue. Add authenticated Gateway HTTP routes for `POST /v1/sessions/reset`, `POST /v1/sessions/compact`, a status/resolve endpoint that can query a session by key, and model/reasoning updates via either dedicated endpoints or a clearly documented patch route that maps to existing `sessions.patch` semantics. Reuse existing session business logic in `src/gateway/server-methods/sessions.ts` and protocol schemas; do not duplicate reset/compact/patch behavior. Keep command interception hardening unchanged: do not make `/new`, `/status`, or `/compact` text commands execute through `/v1/chat/completions`. Use existing bearer/scoped auth helpers and HTTP patterns from `src/gateway/server-http.ts`, `src/gateway/session-kill-http.ts`, `src/gateway/sessions-history-http.ts`, and `src/gateway/method-scopes.ts`. Address lessons from closed #23941: avoid broad prefix matching such as `/v1/sessionsXYZ`, do not pass unsupported `key` fields into strict `sessions.list` params, and validate reset `reason` against the existing `"new" | "reset"` contract. Add focused unit/integration tests for route matching, method validation, auth/scope failures, successful reset/compact/status/patch behavior, and unknown session behavior. Update https://docs.openclaw.ai/gateway/openai-http-api source docs under `docs/gateway/openai-http-api.md`. Add a changelog entry only if maintainers consider the new HTTP API user-facing for the active release, with proper attribution to the original reporter and relevant contributors. Do not add bundled plugin-specific behavior or relax Gateway auth.
