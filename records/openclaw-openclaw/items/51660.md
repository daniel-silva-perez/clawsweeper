## Summary

Keep open. Current main still has no supported umask, file-mode, dir-mode, or filePermissions config for agent-created user-facing files. The existing surfaces cover exec routing/security/path controls and filesystem workspace containment, while host write/edit/apply_patch/sandbox paths still inherit process defaults or use fixed hard-coded modes. This is core work because it spans built-in exec, filesystem tools, apply_patch, and the sandbox bridge rather than an optional plugin-only integration.

## Best Possible Solution

Add a core, documented, per-agent-aware file creation policy for user-facing workspace output, likely under tools.fs or a narrow tools.fileCreation block. Validate octal umask/file/dir values, apply them only to write/edit/apply_patch/sandbox bridge and exec-created user-output files where feasible, and keep sensitive OpenClaw-managed state on existing restrictive permission paths. Include regression tests for host filesystem tools, sandbox bridge writes, apply_patch, and exec-created files under a controlled umask, plus docs that distinguish this from config/credential permission hardening.

## Risks / Open Questions

- A naive global process.umask change could weaken sensitive OpenClaw-managed files such as config, credentials, transcripts, auth profiles, or plugin state that intentionally use restrictive 0600/0700 behavior.

- The config needs tight octal validation and clear docs distinguishing user-facing workspace output from sensitive OpenClaw state permission hardening.
