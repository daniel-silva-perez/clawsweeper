## Summary

Keep open. Current main still has no supported umask, file-mode, dir-mode, or filePermissions config for agent-created user-facing files. The existing tool config covers exec routing/security/path controls and filesystem workspace containment, while write/edit/apply_patch/sandbox paths either use process defaults or fixed hard-coded modes. The issue is also not a ClawHub fit because the requested behavior spans core built-in exec, filesystem tools, sandbox bridging, and apply_patch rather than an optional standalone plugin.

## Best Possible Solution

Add a core, documented, per-agent-aware config surface for user-facing workspace output permissions, likely under tools.fs or a narrow tools.fileCreation block. Validate octal values, apply them only to user-facing file/directory creation by write/edit/apply_patch/sandbox bridge and exec runtime where feasible, and keep sensitive OpenClaw-managed files on their existing restrictive 0600/0700 paths. Include regression tests for host filesystem tools, sandbox filesystem tools, apply_patch, and an exec-created file under a controlled umask, plus docs that distinguish this from config/credential permission hardening.

## Risks / Open Questions

- A naive global umask change could weaken sensitive OpenClaw state, credentials, configs, transcripts, or plugin state that intentionally use 0600/0700.
- The implementation needs consistent behavior across host write/edit, apply_patch, sandbox fs bridge, exec-created files, and node-host execution; partial coverage would leave the Docker/NAS use case unreliable.
