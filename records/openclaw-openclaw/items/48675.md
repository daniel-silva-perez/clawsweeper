## Summary

Keep open. Current main still does not pre-create `memory/YYYY-MM-DD.md` before gateway or CLI agent runs, so the intended PR behavior is not implemented on main. The linked ENOENT path is partially mitigated by startup-context loading and `memory_get` returning empty text for missing memory files, but raw workspace reads can still surface ENOENT. The PR remains a plausible implementation candidate, but it needs a current-main rebase and design cleanup before merge.

## Best Possible Solution

Keep this PR open for maintainer follow-up, but do not merge it as submitted. Rebase or replace it with a current-main fix that wires the helper into `getReplyFromConfig` and `src/agents/agent-command.ts`, preserves existing daily memory contents, treats concurrent creation as success, avoids unconditional writes when memory flush/startup memory behavior is disabled or the workspace is read-only, and removes the unrelated manifest-registry test change.

## Risks / Open Questions

- Merging the submitted branch as-is would miss the active CLI entrypoint because current main uses `src/agents/agent-command.ts`, while the PR modifies the now-reexported `src/commands/agent.ts`.
- The proposed helper opens the target in append/write mode even when the daily file already exists, so read-only workspaces or disabled memory-flush configurations could fail runs with `EACCES`/`EROFS` unless the design is gated or changed to treat existing files as success without requiring write access.
- The PR includes an unrelated `src/plugins/manifest-registry.test.ts` change, which should be dropped or split before merge.
- No supply-chain or secrets regression is visible in the provided PR diff, but startup file creation is a behavior change and should be covered by focused tests for fresh workspaces, existing files, disabled memory flush, read-only workspaces, and concurrent creation.
