## Summary

Keep this PR open: current main still does not pre-create the daily memory file before gateway or CLI agent runs, so the underlying contribution remains relevant. The branch is not merge-ready because it targets the old CLI command file, does not use the current plugin-owned memory flush plan, and can introduce startup writes even when memory flush is disabled or the workspace should be treated as non-writable.

## What This Changes

The PR adds an `ensureMemoryFlushTarget` helper that touches `memory/YYYY-MM-DD.md`, wires it into gateway and CLI agent startup, adds memory-flush tests, and also adds an unrelated manifest-registry duplicate-path test.

## Best Possible Solution

Implement a current-main, plan-aware pre-touch that derives the target from `resolveMemoryFlushPlan`, no-ops when no flush plan exists, avoids blocking startup when the target already exists or is concurrently created, respects disabled memory flush and non-writable workspace policy, wires both `getReplyFromConfig` and `src/agents/agent-command.ts`, and keeps unrelated plugin-registry coverage out of the PR.

## Reproduction Assessment

Yes. Source inspection on current main shows fresh workspaces are created before agent runs, but no daily memory file is created; a raw agent read of `memory/YYYY-MM-DD.md` can still match the ENOENT scenario reported in #48599.

## Solution Assessment

No. The PR is directionally useful, but the current patch is not the best implementation because it targets an obsolete CLI file and performs an unconditional write instead of using the current memory-core flush plan and writable-policy checks.
