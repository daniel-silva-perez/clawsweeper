## Summary

Keep open. Current main still routes `sessions_spawn` through one fixed `subagent` queue lane: the tool schema has no `lane` input, neither spawn path forwards one, native and ACP dispatch hard-code `AGENT_LANE_SUBAGENT`, and config/docs still expose only the scalar sub-agent concurrency cap. The underlying Gateway queue already supports string lanes, so this is a bounded feature gap rather than a stale or out-of-scope request.

## What This Changes

Request adding an optional `sessions_spawn.lane` parameter and per-lane sub-agent concurrency configuration so independent sub-agent work can use separate queue lanes instead of the global `subagent` lane.

## Best Possible Solution

Implement a backward-compatible `sessions_spawn.lane` API that defaults to `subagent`, validates lane names, forwards the resolved lane through native and ACP spawn paths, applies per-lane sub-agent concurrency during Gateway startup and config reload, and updates schema/help/docs/changelog plus regression tests without changing existing callers.

## Reproduction Assessment

Yes. The feature gap is reproducible by static inspection on current main: no `lane` schema field exists, no execution path reads or forwards a lane, and both native and ACP child dispatches send the fixed `subagent` lane.

## Solution Assessment

Yes. An additive lane parameter is the narrowest maintainable direction because the Gateway `agent` protocol and command queue already accept named lanes; the fix should reuse that infrastructure rather than introducing a separate scheduler.
