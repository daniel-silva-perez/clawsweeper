## Summary

Keep open. Current main still exposes `sessions_spawn` through one fixed `subagent` queue lane: the tool has no caller-selectable `lane` input, native and ACP child dispatch both hard-code `AGENT_LANE_SUBAGENT`, and config/docs/tests still describe one scalar subagent concurrency cap. The related open per-agent lane PR is adjacent Gateway infrastructure, but it does not implement this sub-agent API gap.

## What This Changes

The issue requests an optional `sessions_spawn.lane` parameter plus per-lane subagent concurrency configuration so independent sub-agent work can run on separate queue lanes.

## Best Possible Solution

Add an additive `sessions_spawn.lane` API that validates a bounded lane name, defaults omitted calls to `subagent`, threads the resolved lane through both native and ACP child dispatch, applies compatible per-lane subagent concurrency config, and updates docs plus regression tests without changing existing callers.
