## Summary

Keep open. Current main still has a single fixed `subagent` lane for `sessions_spawn`: the tool schema does not accept a caller-selected lane, both native and ACP dispatch paths hard-code `AGENT_LANE_SUBAGENT`, and config/docs/tests still describe one scalar subagent concurrency cap. The related per-agent lane PR targeted inbound agent runs, was closed unmerged, and does not supersede this request.

## What This Changes

Add an optional `lane` parameter to `sessions_spawn` plus per-lane subagent concurrency configuration so independent sub-agent flows can avoid one global queue bottleneck.

## Best Possible Solution

Implement a backward-compatible `sessions_spawn.lane` API that defaults omitted or blank values to `subagent`, validates lane names, forwards the resolved lane through native and ACP spawn dispatch, applies compatible per-lane subagent concurrency at Gateway startup and reload, and updates user-facing docs plus regression tests.
