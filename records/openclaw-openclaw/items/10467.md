## Summary

Keep open. Current main still lacks caller-selected `sessions_spawn` lanes: the tool schema has no `lane` field, the execute path does not read or forward one, native and ACP dispatch still hard-code the `subagent` lane, and config/docs/tests still model sub-agent concurrency as one scalar lane cap. The related closed per-agent lane PR targeted inbound agent runs, not sub-agent spawn lanes.

## What This Changes

The issue requests an optional `sessions_spawn.lane` parameter plus per-lane sub-agent concurrency configuration so independent sub-agent classes can run on separate queue lanes instead of sharing the global `subagent` lane.

## Best Possible Solution

Implement a backward-compatible `sessions_spawn.lane` API that defaults to `subagent`, validates lane names, forwards the resolved lane through both native and ACP spawn paths, applies per-lane sub-agent concurrency at Gateway startup and reload, and updates schema, docs, changelog, and regression tests while preserving all existing callers.

## Reproduction Assessment

Yes. The feature gap is reproducible by static inspection on current main: no `lane` schema field exists, no execution path reads or forwards a lane, and both native and ACP child dispatches send the fixed `subagent` lane.

## Solution Assessment

Yes. The requested additive lane parameter is the narrow maintainable solution because the command queue already supports arbitrary named lanes; the repair should reuse that infrastructure instead of introducing a separate scheduler.
