## Summary

Keep open. Current `main` still implements `sessions_spawn` with a single fixed `subagent` queue lane, with no caller-selectable `lane` input, no lane forwarding through native or ACP spawn paths, and only scalar subagent concurrency configuration. The recent maintainer-side review remains accurate, and the adjacent open per-agent lane PR targets inbound agent lanes rather than this sub-agent API gap.

## What This Changes

Add multi-lane concurrency support for sub-agents by allowing `sessions_spawn` callers to choose a named lane and configure per-lane subagent concurrency while preserving the default `subagent` lane.

## Best Possible Solution

Add an additive `sessions_spawn.lane` API that validates a bounded lane name, defaults omitted calls to `subagent`, threads the resolved lane through both native and ACP child dispatch, applies compatible per-lane subagent concurrency config, and updates docs plus regression tests without changing existing callers.
