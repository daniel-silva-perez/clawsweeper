## Summary

Keep this issue open. Current main still exposes `sessions_spawn` as a single global `subagent` lane: the tool schema has no caller-selectable `lane`, native and ACP spawn dispatch both hard-code `AGENT_LANE_SUBAGENT`, and config/docs/tests still describe one scalar `agents.defaults.subagents.maxConcurrent` cap. The April 26 maintainer-side review comment remains accurate against current main, and the added user feedback reinforces that this is still an active workflow bottleneck.

## What This Changes

Add optional named queue lanes for `sessions_spawn` sub-agents, with per-lane concurrency configuration and a default fallback to the existing `subagent` lane.

## Best Possible Solution

Add an additive `sessions_spawn.lane` API that defaults to the existing `subagent` behavior, validates lane names, threads the resolved lane through native and ACP child dispatch, and exposes compatible per-lane subagent concurrency configuration with updated docs and regression tests.
