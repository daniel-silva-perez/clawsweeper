## Summary

Keep open. Current main still lacks caller-selected `sessions_spawn` lanes: the tool schema and execution path do not read or forward a `lane` parameter, native and ACP dispatch remain pinned to `subagent`, and config/docs/tests still describe one scalar subagent concurrency cap. The related per-agent lane PR was for inbound agent runs, was closed unmerged, and does not supersede this subagent-specific request.

## What This Changes

The issue requests an optional `lane` parameter for `sessions_spawn`, independent subagent queue lanes, per-lane concurrency configuration, and default compatibility with the existing `subagent` lane.

## Best Possible Solution

Implement a backward-compatible `sessions_spawn.lane` API that defaults omitted or blank values to `subagent`, validates lane names, forwards the resolved lane through both native and ACP spawn paths, applies compatible per-lane subagent concurrency at Gateway startup and reload, and updates user-facing docs plus regression tests.

## Reproduction Assessment

Yes. The feature gap is reproducible by static inspection on current main: `sessions_spawn` has no `lane` schema field, no execution-time lane read, and both dispatch paths still send the fixed `subagent` lane.

## Solution Assessment

Yes. The narrow maintainable solution is to reuse the existing arbitrary named command-lane infrastructure, add a validated additive parameter and config resolver, and preserve the current `subagent` default for all existing callers.
