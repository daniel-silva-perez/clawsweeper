## Summary

Keep open. Current main still lacks caller-selected `sessions_spawn` lanes: the tool schema has no `lane` parameter, the execute path does not read or forward one, native and ACP child runs are still dispatched on the fixed `subagent` lane, and config/docs/tests still model subagent concurrency as one scalar lane cap. The related per-agent lane PR was closed unmerged and explicitly targeted inbound agent runs, not subagent spawn lanes.

## What This Changes

Add an optional `sessions_spawn.lane` parameter with per-lane subagent concurrency so independent subagent work can run in separate queue lanes while preserving the default `subagent` behavior.

## Best Possible Solution

Implement a backward-compatible `sessions_spawn.lane` API that defaults to `subagent`, validates lane names, forwards the resolved lane through native and ACP spawn paths, applies per-lane subagent concurrency at Gateway startup and reload, and updates docs plus regression tests without changing existing callers.

## Reproduction Assessment

Yes. The feature gap is reproducible by static inspection on current main: no `lane` schema field is exposed, no execution path reads or forwards a lane, and both native and ACP child dispatches send the fixed `subagent` lane.

## Solution Assessment

Yes. Reusing the existing arbitrary named command-lane infrastructure is the narrow maintainable path; the fix should be additive API/config plumbing with the current scalar `subagent` default preserved.
