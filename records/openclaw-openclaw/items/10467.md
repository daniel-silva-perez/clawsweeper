## Summary

Keep open. Current main still has one fixed `subagent` execution lane for `sessions_spawn`: the tool schema has no caller-selectable lane, native and ACP spawn paths both hard-code the subagent lane, and config/docs/tests still describe one scalar subagent concurrency cap. The related per-agent lane PR is not a substitute because it targeted inbound agent runs and was closed unmerged.

## What This Changes

Add an optional `lane` parameter to `sessions_spawn`, with per-lane subagent concurrency configuration and a default fallback to the existing `subagent` lane.

## Best Possible Solution

Implement a backward-compatible `sessions_spawn.lane` API that validates a bounded lane name, defaults omitted or blank values to `subagent`, threads the resolved lane through both native and ACP spawn dispatch, adds compatible per-lane subagent concurrency configuration, and updates docs/tests while preserving existing callers.
