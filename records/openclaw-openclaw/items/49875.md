## Summary

Keep this PR open. Current main still does not implement the PR's intended broadcast `runInboundClaim` dispatch path: regular messages fire `message_received`, then go straight to processing/fast-abort/agent dispatch, while the only dispatch caller remains the targeted plugin-bound path. The PR is relevant implementation work, but it is incomplete against current main because tests still encode no-broadcast behavior and the proposed insertion has ordering/diagnostic issues already called out in review.

## Best Possible Solution

Refresh this PR against current main and keep it focused on the core dispatch wiring: add broadcast `runInboundClaim` for regular non-plugin-bound messages in an order that preserves `message_received`, fast-abort, and plugin-bound fallback semantics; call `markIdle` and record/commit the same diagnostics/dedupe state as other handled early returns; update dispatch tests for claimed, unclaimed, plugin-bound handled/fallback, and abort-message behavior.

## Risks / Open Questions

- Adding the broadcast claim exactly as proposed can let a global `inbound_claim` handler swallow stop/abort messages before the existing fast-abort branch runs.
- Broadcasting after plugin-bound `missing_plugin` or `no_handler` fallback can let an unrelated plugin claim a turn that current behavior routes back through normal OpenClaw handling after notifying the user.
- The current dispatch test suite would fail or lose coverage unless the old no-broadcast assertion is replaced with focused claimed, unclaimed, abort-order, and plugin-bound fallback cases.
