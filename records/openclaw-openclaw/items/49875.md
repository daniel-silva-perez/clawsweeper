## Summary

Keep this PR open. Current main still does not wire the broadcast `runInboundClaim` call into regular dispatch; only plugin-owned bindings invoke the targeted inbound-claim path, and tests still assert the old no-broadcast behavior. The PR is useful but incomplete because its current diff also needs test updates and ordering/diagnostic fixes before merge.

## Best Possible Solution

Keep this PR open as useful but incomplete implementation work. The best path is to refresh it against current main, add broadcast `runInboundClaim` for regular non-plugin-bound messages, preserve plugin-owned binding and fast-abort precedence, call `markIdle` on claimed early return, and replace the old no-broadcast assertion with targeted claimed/unclaimed regression tests.

## Risks / Open Questions

- Functional merge risk: adding broadcast claim before fast-abort could let a broad `inbound_claim` handler swallow documented stop/abort messages unless abort precedence is preserved and tested.
- Functional merge risk: broadcasting after a plugin-owned missing-plugin/no-handler fallback could let an unrelated plugin claim a turn that current behavior promises to route back to OpenClaw.
- Test risk: current main still has the old no-broadcast assertion and lacks focused regular-message claimed/unclaimed dispatch-path regression tests.
