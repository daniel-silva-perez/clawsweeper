## Summary

Keep this PR open. Current main still does not wire the broadcast `runInboundClaim` hook into the regular dispatch path; the only dispatch caller is the targeted plugin-bound path. The PR is useful implementation work, but it is not ready to close or merge as-is because current tests still encode the old no-broadcast behavior and the proposed insertion needs ordering and diagnostic fixes.

## Best Possible Solution

Keep this PR open as useful but incomplete. Refresh it against current main, wire broadcast `runInboundClaim` for regular non-plugin-bound messages, preserve plugin-owned binding and fast-abort precedence, call `markIdle` on the claimed early return, and update dispatch tests to cover claimed, unclaimed, plugin-bound fallback, and abort-order behavior.

## Risks / Open Questions

- Adding the broadcast claim exactly where the PR patch places it can let a global `inbound_claim` handler swallow documented stop/abort messages before the fast-abort branch runs.
- Broadcasting after plugin-bound `missing_plugin` or `no_handler` fallback could let an unrelated plugin claim a turn that current tests expect to route back through normal OpenClaw handling.
- The existing no-broadcast dispatch test must be replaced with focused claimed/unclaimed regression tests; otherwise this PR remains blocked by current test expectations.
