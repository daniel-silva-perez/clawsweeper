## Summary

Keep open. Current main still defaults the signed macOS companion Sparkle updater into automatic checks and downloads through a single toggle, and the config-clobber path still records/audits suspicious update-channel-only config loss without restoring or returning the last known good config. The reporter also confirmed the suggested recovery workaround did not help.

## What This Changes

The issue asks OpenClaw to prevent the macOS companion app from automatically applying incompatible beta updates and to preserve or recover the user's OpenClaw configuration after companion app reinstall/update flows.

## Best Possible Solution

Keep this open until the macOS companion separates Sparkle check/download/install policy, defaults automatic downloads conservatively, requires explicit beta-channel opt-in, and changes `OpenClawConfigFile` so an update-channel-only clobber is preserved for audit but the active config is restored or safely merged from the last known good state.

## Reproduction Assessment

Yes. Current main has a focused reproduction in `OpenClawConfigFileTests`: write a full config with `gateway.mode`, replace it with an update-channel-only root, and `loadDict()` returns a config with no gateway while only audit artifacts are produced. The full March appcast beta incident was not live-reproduced, but the updater code path that can automatically check and download remains present.

## Solution Assessment

No. The current implementation is not the best fix because it treats suspicious config loss as audit-only and uses one app setting to force both Sparkle checks and downloads; the safer path is conservative updater defaults plus active last-known-good config recovery.
