## Summary

Keep open. Current main still lacks Linux desktop notification delivery for the core `system.notify` node surface: Linux is allowlisted, but the headless node host does not advertise or dispatch the command, and the CLI/docs still frame notifications as macOS app/node behavior. The related closed #24616 improved the rejection error, but did not add Linux notification delivery.

## What This Changes

The issue requests Linux desktop notification support for OpenClaw headless Linux nodes, preferably via `notify-send`/libnotify or D-Bus with graceful no-display degradation.

## Best Possible Solution

Add a focused core Linux headless node-host `system.notify` implementation with desktop-session/backend detection, safe helper invocation, structured unavailable results for missing display or backend, aligned CLI/docs/changelog, and targeted tests while preserving existing macOS, iOS, Android, and Windows behavior.

## Reproduction Assessment

Yes. Current-main source inspection gives a high-confidence reproduction path: a Linux headless node host never declares `system.notify`, and the Gateway requires declared commands before forwarding, so `openclaw nodes notify` cannot reach any Linux notification backend.

## Solution Assessment

Yes. A narrow core node-host repair is the best path because `system.notify` is a reserved core command; a plugin cannot provide the same command surface without a new core API or policy change.

## Repair Work Prompt

Fix Linux desktop notification support for the core `system.notify` node command on headless Linux node hosts. Treat https://github.com/openclaw/openclaw/issues/14079 as the canonical issue and #24616 as related diagnostics-only context. Implement this in core node-host code, not as a plugin, because `system.notify` is reserved by the registry. Expected boundary: advertise `system.notify` from Linux-capable headless node hosts, add a Linux handler that gracefully returns structured unavailable results when no desktop session or notification backend is available, prefer safe argv-based `notify-send` invocation without shell interpolation, and avoid changing macOS, iOS, Android, Windows, `system.run`, or exec approval behavior. Update CLI/help/docs to remove the mac-only Linux gap, add focused tests for advertisement and invocation behavior, and add a user-facing changelog entry crediting @holistic-medical-supply-base. Do not add broad D-Bus or dunst-specific behavior unless it remains small and well-tested; keep fallback scope conservative.
