## Summary

Keep open. Current main at 5f13af6b68603c49367e1540212779d1dcafafb2 still does not implement Linux desktop notification delivery for the core `system.notify` node surface: Linux is allowlisted by policy, but the headless node host neither advertises nor handles the command, and the CLI/docs still frame notification delivery as macOS app behavior.

## What This Changes

The issue requests Linux desktop notification support for `system.notify`, primarily via `notify-send`/libnotify with graceful degradation on headless Linux hosts.

## Best Possible Solution

Add a focused Linux headless node-host `system.notify` implementation with desktop-session/backend detection, safe helper invocation, structured unavailable results for missing display or backend, and docs/CLI/tests aligned with the new supported surface while preserving current macOS, iOS, Android, and Windows behavior.

## Repair Work Prompt

Implement Linux desktop notification support for the existing core node `system.notify` surface on the headless node host. Scope the fix to Linux node-host delivery: advertise/handle `system.notify` from the node host when the implementation can return a real result or structured unavailable result, use `notify-send` as the initial backend with title/body/sound/priority/delivery passed safely as argv or typed fields, never shell-interpolate user-controlled notification text, and gracefully return an unavailable-style result when no `DISPLAY`/`WAYLAND_DISPLAY` or notification backend is available. Preserve existing macOS app, iOS, Android, Windows, `system.run`, and plugin node-host behavior. Update CLI copy, nodes/node docs, and changelog with user-facing attribution to the reporter; do not add broad D-Bus or `dunstify` fallback support unless it stays small and tested.
