## Summary

Keep open. Current main still lacks Linux desktop notification delivery for the core `system.notify` node surface: Linux is allowlisted, but the headless node host does not advertise or dispatch the command, and docs/CLI still describe notification delivery as macOS app behavior.

## What This Changes

The issue requests Linux desktop notification support for OpenClaw nodes, preferably via `notify-send` or D-Bus with graceful degradation on headless hosts.

## Best Possible Solution

Add a focused Linux headless node-host `system.notify` implementation with desktop-session/backend detection, safe helper invocation, structured unavailable results for missing display or backend, and aligned CLI/docs/tests while preserving existing macOS, iOS, Android, and Windows behavior.

## Reproduction Assessment

Yes. Current-main source inspection gives a high-confidence reproduction path: a Linux headless node host never declares `system.notify`, and the Gateway requires declared commands before invocation, so `openclaw nodes notify` cannot reach a Linux notification backend.

## Solution Assessment

Yes. A narrow core node-host repair is the best path because `system.notify` is a reserved core command; a plugin cannot provide the same command surface without a core API or policy change.

Fix Linux desktop notifications for the core `system.notify` node surface tracked by https://github.com/openclaw/openclaw/issues/14079. Implement a focused Linux headless node-host path: advertise `system.notify` from the headless node host when appropriate, dispatch it in the node-host invoke flow, support a safe primary backend such as `notify-send` without shell interpolation, detect missing desktop sessions/backends and return structured unavailable-style errors, and keep macOS, iOS, Android, and Windows behavior unchanged.
