## Summary

Keep open. Current main still does not implement Linux desktop notification delivery for the core `system.notify` node surface: Linux policy allowlists the command, but the headless node host does not advertise or dispatch it, and current docs/CLI still describe notification delivery as macOS node behavior. The related closed #24616 addressed the rejection/error message, not Linux notification delivery.

## What This Changes

Add Linux desktop notification support for `openclaw nodes notify` / `system.notify`, with desktop-session detection and graceful fallback when Linux notification backends are unavailable.

## Best Possible Solution

Add a focused core Linux headless node-host `system.notify` implementation with desktop-session/backend detection, safe helper invocation, structured unavailable results for headless or missing-backend cases, aligned CLI/docs/changelog text, and targeted tests while leaving macOS, iOS, Android, and Windows behavior unchanged.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a Linux headless node host never declares `system.notify`, and the Gateway requires declared commands before forwarding, so `openclaw nodes notify` cannot reach a Linux notification backend.

## Solution Assessment

Yes. A narrow core node-host repair is the best path because the exact `system.notify` command is reserved by core; a plugin cannot fill this command surface without a new core seam or policy change.

## Repair Work Prompt

Fix Linux desktop notification support for the core `system.notify` / `openclaw nodes notify` surface from https://github.com/openclaw/openclaw/issues/14079. Current main allowlists `system.notify` for Linux but the headless node host does not advertise or dispatch it, and plugins cannot register the exact command because it is core-reserved. Implement a focused core Linux headless node-host path: advertise `system.notify` on Linux when appropriate, dispatch it safely, detect desktop sessions (`DISPLAY` or `WAYLAND_DISPLAY`), use `notify-send` as the initial helper if available, pass title/body as argv values without shell execution, and return structured unavailable/invalid results for headless sessions or missing backends. Keep macOS, iOS, Android, Windows, `system.run`, exec approvals, browser proxy, and plugin node-host behavior unchanged. Update CLI text and docs so Linux support is accurately described. Add or update targeted tests for advertisement, invocation success/unavailable paths, policy behavior, and CLI coverage. Add a single-line user-facing changelog entry with reporter credit, likely `Thanks @holistic-medical-supply-base`; reference #14079 and mention #24616 only if useful for context. Do not implement broad D-Bus or `dunstify` fallback unless the first implementation remains small and well-tested.
