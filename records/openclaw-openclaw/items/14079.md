## Summary

Keep open. Current main still lacks Linux desktop notification delivery for the core `system.notify` node surface: the Linux policy allowlists the command, but the headless node host does not advertise or dispatch it, and the CLI/docs still frame notification delivery as macOS node behavior. The related closed #24616 addressed the rejection/error experience rather than adding Linux notification support.

## What This Changes

Request Linux desktop notification support for OpenClaw nodes, primarily via `notify-send` or libnotify with graceful degradation on headless Linux systems.

## Best Possible Solution

Add a focused core Linux headless node-host `system.notify` implementation with desktop-session/backend detection, safe helper invocation, structured unavailable results for missing display or backend, aligned CLI/docs/changelog text, and targeted tests while preserving current macOS, iOS, Android, and Windows behavior.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence reproduction path: a Linux headless node host never declares `system.notify`, and the Gateway requires declared commands before forwarding, so `openclaw nodes notify` cannot reach any Linux notification backend.

## Solution Assessment

Yes. A narrow core node-host repair is the best path because `system.notify` is a reserved core command; a plugin cannot provide the same command surface without a new core API or policy change.

## Repair Work Prompt

Implement Linux desktop notification support for the core `system.notify` node-host command requested by https://github.com/openclaw/openclaw/issues/14079. Keep the fix focused on the headless node host path: advertise `system.notify` on Linux only when appropriate or advertise with structured unavailable handling, dispatch it in `src/node-host/invoke.ts`, and add a small `src/node-host/invoke-system-notify.ts` helper if that matches local patterns. Use `notify-send` as the first backend, detect headless sessions with missing `DISPLAY` and `WAYLAND_DISPLAY`, return structured unavailable/invalid-request style failures for no display or missing helper, and never shell-interpolate title/body/sound/priority/delivery values. Do not change macOS companion-app, iOS, Android, or Windows notification behavior. Update CLI/docs wording so Linux headless nodes are no longer described as mac-only once supported, and add a user-facing changelog entry with proper reporter credit. Related #24616 is only the prior error-message bug and should not be treated as the implementation. Validate with the targeted node-host, CLI, gateway policy, formatter, and changed-gate commands listed in `workValidation`; run the broad changed gate in Testbox before handoff if promoted.
