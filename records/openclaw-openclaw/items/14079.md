## Summary

Keep open. Current main still lacks Linux desktop notification delivery for the reserved core `system.notify` node command: Linux is allowed at the gateway policy layer, but the headless node host does not advertise or dispatch the command, the CLI/docs still present notification delivery as macOS/non-headless behavior, and plugin registration cannot fill the exact reserved command gap.

## What This Changes

Add Linux desktop notification support for OpenClaw nodes, primarily via `notify-send`/libnotify with graceful handling for headless Linux sessions.

## Best Possible Solution

Implement a focused core Linux headless node-host `system.notify` path with desktop-session detection, safe backend invocation, structured unavailable results for missing display/backend, docs and CLI updates, and tests proving advertisement plus invocation behavior without changing macOS, iOS, or Android notification behavior.

## Repair Work Prompt

Fix Linux desktop notifications for the core headless node host. Current main allows `system.notify` for Linux in gateway policy, but `openclaw node run` does not advertise or dispatch it, so `openclaw nodes notify` cannot deliver on Linux. Add a focused Linux `system.notify` implementation owned by the node host: advertise `NODE_SYSTEM_NOTIFY_COMMAND` when appropriate, dispatch it in `src/node-host/invoke.ts`, implement a small `src/node-host/invoke-system-notify.ts` helper, detect desktop availability via `DISPLAY` or `WAYLAND_DISPLAY`, prefer `notify-send` with argv-only spawning, and return structured unavailable/error results when no desktop session or backend exists. Do not use shell interpolation. Keep macOS, iOS, Android, Windows, exec approvals, and `system.run` behavior unchanged. Do not implement broad D-Bus or `dunstify` fallback unless the existing code structure makes it very small and testable. Update CLI/docs to remove the mac-only limitation for Linux headless nodes, add a user-facing changelog entry with proper attribution, and add focused tests for advertisement, invocation, env sanitization, CLI coverage, and gateway policy interactions. Related refs: this issue and closed #24616 for the prior error-experience distinction.
