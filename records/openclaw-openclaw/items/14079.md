## Summary

Keep open. Current main still does not provide Linux desktop notification delivery for the core `system.notify` node surface: the Linux policy allowlists the command, but the headless node host does not advertise or dispatch it, and the CLI/docs still describe notification delivery as macOS app/node behavior. The related closed #24616 addressed the rejection message, not Linux notification delivery.

## What This Changes

Request Linux desktop notification support for OpenClaw node notifications using a desktop-session-aware backend such as `notify-send` or D-Bus with graceful headless degradation.

## Best Possible Solution

Add a focused core Linux headless node-host `system.notify` implementation with desktop-session/backend detection, safe helper invocation, structured unavailable results for missing display or backend, aligned CLI/docs/changelog, and targeted tests while preserving existing macOS, iOS, Android, and Windows behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a Linux headless node host never declares `system.notify`, and the Gateway requires declared commands before forwarding, so `openclaw nodes notify` cannot reach a Linux notification backend.

## Solution Assessment

Yes. A narrow core node-host repair is the best path because `system.notify` is a reserved core command; a plugin cannot provide the same command surface without a new core API or policy change.
