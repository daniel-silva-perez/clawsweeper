## Summary

Keep this PR open. Current main does not implement the PR’s central behavior: there is no `openclaw auth rotate` CLI, no `gateway.auth.tokenExpiryHours` config/schema surface, and no token-expiry warning state in the shared Gateway HTTP auth path. The PR also still needs maintainer review for security-sensitive token-write semantics before it could be merged.

## Best Possible Solution

Keep the PR open as a feature candidate, but require a current-main rework before merge: add `openclaw auth rotate` through the current CLI registration system, add `gateway.auth.tokenExpiryHours` to config types/schema/generated docs/tests if the product decision still stands, emit expiry warnings from the shared Gateway auth path, and make rotation fail or require explicit opt-in for non-token mode, SecretRef/env-managed tokens, and running-gateway restart/reload timing. Drop unrelated generated baseline drift and add targeted tests for the token-source and auth-mode guardrails.

## Risks / Open Questions

- The PR touches authentication token generation and config persistence; merging as-is could silently change `gateway.auth.mode` or move env/SecretRef-managed secrets into plaintext config if the branch still matches the provided diff/review comments.
- The rotate command’s user-visible success semantics need to account for gateway restart or reload timing so scripts do not immediately switch to a token that the running gateway has not accepted yet.
- The expiry warning should be wired through the shared current-main HTTP auth path, not a stale or partial helper path, or the new config field may not affect normal gateway requests.
