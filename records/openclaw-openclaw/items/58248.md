## Summary

Keep this PR open for maintainer review. Current main does not already implement the new gateway token-expiry config or `openclaw auth rotate`, but the proposed patch still changes auth/token handling in ways that are not safe to merge without revision: it can silently rewrite the gateway auth mode, mishandles env-managed token sources, and wires the expiry warning into a stale/non-shared HTTP auth path.

## What This Changes

The PR adds a `gateway.auth.tokenExpiryHours` config surface, an in-process gateway token-expiry warning, an `openclaw auth rotate` CLI command, tests, and generated config metadata updates.

## Best Possible Solution

Revise the PR or build a replacement against current main: keep the new CLI and expiry warning only if rotation refuses or explicitly handles non-token modes, SecretRef/env-managed tokens, env-only token sources, restart activation timing, and the current shared HTTP auth path, with targeted tests for each auth source and mode.

## Reproduction Assessment

Yes. The main failures are reproducible from code inspection: a config with `gateway.auth.mode: "password"` or `"trusted-proxy"` would be rewritten by the PR's `mode: "token"` assignment, and a file-authored `${OPENCLAW_GATEWAY_TOKEN}` is hidden by current snapshot substitution semantics before the PR's SecretRef guard runs.

## Solution Assessment

No. The product direction is plausible, but this patch is not the narrowest maintainable solution because it writes auth policy and secret sources too broadly and places warning logic outside the current shared HTTP auth flow.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Preserve the configured gateway auth mode:** `src/commands/auth-rotate.ts:44`
  - body: The rotate command always writes `mode: "token"` after generating a token. On configs intentionally using `password`, `none`, or `trusted-proxy`, running this helper silently changes the gateway auth policy on disk instead of refusing or preserving the existing mode.
  - confidence: 0.93
- **[P1] Reject env-managed gateway tokens before writing config:** `src/commands/auth-rotate.ts:24-28`
  - body: The SecretRef guard checks `snapshot.sourceConfig`, but current main records that snapshot after `${ENV}` substitution. A file-authored `${OPENCLAW_GATEWAY_TOKEN}` or an env-only `OPENCLAW_GATEWAY_TOKEN` setup can therefore be replaced with a new literal `gateway.auth.token`, changing the secret source of truth.
  - confidence: 0.9
- **[P2] Emit expiry warnings from the shared HTTP auth flow:** `src/gateway/http-auth-helpers.ts:32-41`
  - body: The warning is added only to the bearer helper shown in this diff. Current main routes normal HTTP endpoints through `checkGatewayHttpRequestAuth` / `authorizeGatewayHttpRequestOrReply` in `http-auth-utils.ts`, so common routes such as OpenAI-compatible HTTP, tools invoke, and session history may never trigger the new warning.
  - confidence: 0.86
- **[P2] Tell callers when the rotated token becomes active:** `src/commands/auth-rotate.ts:57-58`
  - body: The command writes config and immediately prints the new token, but current gateway config reload treats `gateway.*` changes as restart-required. A running gateway can keep accepting the old in-memory token until restart/reload completes, so scripts switching immediately to the printed token can fail authorization.
  - confidence: 0.82
- **[P3] Remove the CLI-local expiry state reset:** `src/commands/auth-rotate.ts:56`
  - body: `setGatewayTokenIssuedAtNow()` updates module-local state in the short-lived CLI process, while expiry checks run in the gateway process. This does not reset the running gateway's warning clock and can mislead future maintainers about the command's runtime effect.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The PR does not add a third-party supply-chain path, but it does change gateway token persistence and auth policy with concrete secret-handling regressions.

Concerns:

- **[medium] Env-managed token can be persisted as plaintext:** `src/commands/auth-rotate.ts:24`
  - body: Because the command does not reliably detect file-authored env templates or env-only gateway token sources, it can write a new literal `gateway.auth.token` into config and make future startups prefer that config token over the env-managed token.
  - confidence: 0.9
- **[medium] Auth mode can be silently rewritten:** `src/commands/auth-rotate.ts:44`
  - body: Writing `mode: "token"` during rotation changes password, none, or trusted-proxy deployments without an explicit operator decision, which alters the gateway auth boundary and can break or weaken an intended deployment model.
  - confidence: 0.88
