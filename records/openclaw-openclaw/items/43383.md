## Summary

Keep open. Current main still reuses the host-built skills snapshot prompt after switching non-rw sandbox execution to the sandbox workspace, so the reported unreadable skill locations can still be exposed; the open #38789 PR remains the focused root-fix candidate.

## What This Changes

The issue asks OpenClaw to expose sandbox-readable skill locations in sandboxed non-main sessions instead of host-only managed skill paths that fail the sandbox file guard.

## Best Possible Solution

Review and land #38789 or an equivalent root fix that rebuilds or remaps `<available_skills>` from sandbox-readable skill entries whenever a non-rw sandbox attempt or compaction run receives a host-built snapshot, while preserving skill filters, exposure metadata, environment overrides, and normal snapshot reuse outside isolated sandboxes.

## Reproduction Assessment

Yes. The high-confidence source path is a host-built `skillsSnapshot` with prompt locations from the agent workspace, followed by a non-rw sandbox attempt or compaction run where `resolvedSkills` prevents live sandbox entry loading and the saved snapshot prompt is returned unchanged.

## Solution Assessment

Yes. The best fix is a root runtime/prompt-resolution change that refreshes or remaps the snapshot for isolated sandbox workspaces; prompt-only hardening such as #74161 is useful defense-in-depth but cannot repair a stale location already stored in the snapshot.
