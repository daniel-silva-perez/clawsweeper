## Summary

Keep open. Current main still builds the skills snapshot from the agent workspace, switches non-rw sandbox execution to the sandbox workspace, and then reuses the original snapshot prompt unchanged, so host-only skill paths can still be exposed. Open PR #38789 is the focused root-fix path; #74161 is related prompt hardening but does not remap stale paths.

## What This Changes

The issue reports that Docker-based non-main sandbox sessions can emit host-only skill paths in `<available_skills>` even though eligible skills are mirrored into the sandbox workspace.

## Best Possible Solution

Review and land #38789 or an equivalent root fix that rebuilds or remaps available-skills prompts from sandbox-readable skill entries whenever a non-rw sandbox attempt or compaction run receives a host-built snapshot, while preserving skill filters and exposure metadata.

## Reproduction Assessment

Yes. The source-backed reproduction path is a host-built skills snapshot with `resolvedSkills`, followed by a non-rw sandbox attempt or compaction run; current main switches execution to the sandbox workspace but still returns the host snapshot prompt.

## Solution Assessment

Yes. Keeping this open for the root implementation path is the narrowest maintainable outcome; prompt wording like #74161 is useful hardening, but it cannot fix a stale `<location>` value by itself.
