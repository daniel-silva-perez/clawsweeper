## Summary

Keep open. Current `main` still reuses a host-built skills snapshot prompt after switching non-rw sandbox execution to the sandbox workspace, so the reported host-only skill path exposure can still occur. The open #38789 PR remains the focused implementation path; #74161 is related prompt hardening but does not remap a stale `<location>` value.

## What This Changes

The issue reports that sandboxed non-main sessions expose host-only `~/.openclaw/skills/.../SKILL.md` paths in available-skills prompts even though skills are mirrored into the sandbox workspace.

## Best Possible Solution

Review and land #38789 or an equivalent root fix that rebuilds or remaps available-skills prompts from sandbox-readable skill entries whenever a non-rw sandbox attempt or compaction run receives a host-built snapshot, while preserving skill filters, exposure metadata, and normal non-sandbox snapshot reuse.

## Reproduction Assessment

Yes. The source-backed reproduction path is a host-built `skillsSnapshot` with `resolvedSkills`, followed by a non-rw sandbox attempt or compaction run; current `main` switches execution to the sandbox workspace but still returns the original snapshot prompt.

## Solution Assessment

Yes. Keeping this open for the root implementation path is the narrowest maintainable outcome; prompt wording like #74161 is useful hardening, but it cannot fix a stale host-only `<location>` value by itself.
