## Summary

Keep open. Current main still builds and persists the skills snapshot from the host workspace, then non-rw sandbox attempt and compaction runs switch to the sandbox workspace while reusing that host-built snapshot prompt and skipping live skill loading, so the reported host-only skill path exposure can still occur. Open #38789 remains the focused root-fix path; #74161 is related prompt hardening, not the path remap/rebuild fix.

## What This Changes

The issue reports that Docker non-main sandbox sessions expose host-style skill paths in `<available_skills>`, causing sandbox-rooted `read` calls to fail even though eligible skills were mirrored into the sandbox workspace.

## Best Possible Solution

Land a root fix that rebuilds or remaps available-skills prompt locations from sandbox-readable skill entries whenever a non-rw sandbox run receives a host-built snapshot. The fix should cover both attempt and compaction paths while preserving normal snapshot reuse, skill filters, exposure metadata, remote eligibility notes, and skill env override behavior.
