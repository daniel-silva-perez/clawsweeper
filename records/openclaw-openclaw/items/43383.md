## Summary

Keep open. Current main still has the reported mismatch: non-rw sandbox setup mirrors eligible skills into the sandbox workspace, but attempt and compaction runs can reuse a host-built skills snapshot prompt, so `<available_skills>` can still point at host-only paths.

## What This Changes

The issue reports that Docker non-main sandbox sessions expose host-only managed skill paths in `<available_skills>` even though those skills are synced into the sandbox workspace.

## Best Possible Solution

Land or replace a root fix that rebuilds or remaps `<available_skills>` from sandbox-readable skill entries whenever a non-rw sandbox attempt or compaction run receives a host-built snapshot. Prompt wording that tells the model to use the exact `<location>` value is useful hardening, but the prompt locations themselves must be correct.

## Reproduction Assessment

Yes. The high-confidence source path is a managed or host skill captured in a skills snapshot, followed by a non-main sandbox run with `workspaceAccess: "none"` or `"ro"`; current main switches execution to the sandbox workspace but can still emit the host-built snapshot prompt.

## Solution Assessment

Yes, keeping this open for the root implementation path is the right outcome. The focused path is #38789 or an equivalent fix that rebuilds/remaps prompt entries for sandbox-readable locations; #74161 alone is not enough because it can still instruct the model to use a stale host path exactly.
