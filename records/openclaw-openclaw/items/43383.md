## Summary

Keep open. Current main still reuses a host-built skills snapshot after switching non-rw sandbox execution to the sandbox workspace, so the reported host-only skill path exposure can still occur. The open #38789 PR remains the focused root-fix path; #74161 is only prompt hardening and does not remap a stale `<location>` value.

## What This Changes

The issue requests sandboxed non-main sessions to expose sandbox-readable skill paths instead of host-only `~/.openclaw/skills/...` paths in `<available_skills>`.

## Best Possible Solution

Review and land #38789 or an equivalent root fix that rebuilds or remaps `<available_skills>` from sandbox-readable skill entries whenever a non-rw sandbox attempt or compaction run receives a host-built snapshot, while preserving skill filters, exposure metadata, environment overrides, and normal snapshot reuse outside isolated sandboxes.

## Reproduction Assessment

Yes. The source-backed reproduction path is a host-built `skillsSnapshot` containing managed skill locations, followed by a non-rw sandbox attempt or compaction run; current main switches execution to the sandbox workspace but still returns the original snapshot prompt.

## Solution Assessment

Yes, keeping this open for the root implementation path is the best current outcome. Prompt-only hardening such as #74161 may reduce path guessing, but it cannot fix a stale host-only `<location>` already present in the saved snapshot.
