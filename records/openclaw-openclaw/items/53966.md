## Summary

Keep open. The user problem is valid and this PR improves Slack SDK error detail extraction, including the earlier top-level retryAfter concern, but the current patch replaces redacting error formatting at multiple Slack log/status paths with a helper that returns raw message and metadata text, so it should not merge yet.

## What This Changes

The PR adds a Slack-local formatSlackError helper with unit tests and routes selected Slack API error catch/logging paths through it to include SDK code, scope, retry, and HTTP status metadata.

## Best Possible Solution

Land a Slack-plugin-local formatter that appends structured Slack SDK fields and then redacts the final formatted string via the plugin-safe logging SDK export, with a regression test for Slack-shaped token data. Keep broader delivery-queue error serialization work tracked separately through #62391 or a generic core serialization seam.

## Reproduction Assessment

Yes for the introduced PR bug: the diff is enough to reproduce by passing a Slack-shaped Error whose message or metadata contains a token-shaped string into formatSlackError and observing raw output. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No. A Slack-local formatter is the right narrow direction, but the implementation should preserve the existing redaction contract by wrapping the assembled output with redactSensitiveText and adding a focused regression test.
