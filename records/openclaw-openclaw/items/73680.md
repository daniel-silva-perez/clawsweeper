## Summary

Keep this issue open. Current main still has the reported contract mismatch: strict per-agent config validation omits `verboseDefault` and `elevatedDefault`, while global defaults, types/runtime readers, generated schema metadata, and docs show adjacent support; open PR #74643 is the active closing implementation and should be reviewed or landed before this issue closes.

## What This Changes

The issue requests that `agents.list[].verboseDefault` and `agents.list[].elevatedDefault` be accepted as per-agent config overrides, or that the surrounding per-agent-facing runtime and type surfaces be removed for consistency.

## Best Possible Solution

Land PR #74643 or an equivalent narrow fix that makes the contract explicit across `AgentEntrySchema`, public config types, resolved agent config, generated schema/help metadata, docs, and focused regression tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: `agents.list` is validated through strict `AgentEntrySchema`, and that schema omits the two reported keys, so `config set` or config validation will reject them as unrecognized. I did not run the CLI because this review is read-only.

## Solution Assessment

Yes. The linked PR's direction is the smallest maintainable fix for the current contract: accept the two fields where the rest of the surface already expects them, document precedence, and add parser-level regression tests rather than adding a competing repair branch.
