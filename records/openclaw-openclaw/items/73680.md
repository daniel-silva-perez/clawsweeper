## Summary

Keep this issue open. Current main still has the reported config contract mismatch: `agents.list[]` is validated through a strict `AgentEntrySchema` that declares adjacent per-agent defaults but not `verboseDefault` or `elevatedDefault`, while global defaults, generated schema metadata, docs, and runtime readers expose related verbose/elevated default behavior. Two open, unmerged PRs already reference this issue with closing syntax, so the issue should remain open until one implementation lands or is explicitly superseded.

## What This Changes

Per-agent `verboseDefault` and `elevatedDefault` config keys should be accepted, or the schema, runtime, types, and docs should be made consistent about those keys being global-only.

## Best Possible Solution

Land one focused implementation that aligns `AgentEntrySchema`, `AgentConfig`/resolved agent config typing, schema labels/help, generated schema metadata, public docs, and parser-level regression tests for valid and invalid per-agent verbose/elevated defaults.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: `agents.list` validates through strict `AgentEntrySchema`, and that schema omits both reported keys, so config validation rejects them as unknown. I did not run the CLI because this review was constrained to read-only inspection, but the validation path is direct.

## Solution Assessment

Yes. Accepting these two keys on per-agent entries and updating the type/docs/generated/test surface is the narrowest maintainable fix because adjacent per-agent defaults already exist and runtime readers already consume the related values. Making the keys global-only would require a broader removal of existing runtime/type surface.
