## Summary

Keep this issue open. Current main still has the reported contract mismatch: `agents.list[]` is validated through a strict `AgentEntrySchema` that lacks `verboseDefault` and `elevatedDefault`, while the defaults schema and runtime/default-resolution paths still expose adjacent default behavior. Two open, unmerged PRs already reference this issue with closing syntax, so the issue should remain open until one implementation lands or is explicitly superseded.

## What This Changes

The issue requests accepting `verboseDefault` and `elevatedDefault` on per-agent `agents.list[]` entries, or else removing/documenting the unsupported per-agent surface consistently.

## Best Possible Solution

Land one focused implementation, likely PR #74643 if it remains the more complete branch, after reviewing/deduping against PR #73696. The end state should align `AgentEntrySchema`, config types, generated schema/help metadata, docs, and focused validation tests so per-agent verbose/elevated defaults are either accepted consistently or intentionally removed from all per-agent runtime surfaces.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: `agents.list` is validated by strict `AgentEntrySchema`, and that schema does not declare the two reported keys while declaring adjacent per-agent defaults. I did not run `openclaw config set` because this review was constrained to read-only inspection, but the validation path is direct.

## Solution Assessment

Yes. Adding the two fields to the per-agent schema and aligning generated docs/types/tests is the narrowest maintainable fix because the product already supports adjacent per-agent defaults and the runtime already reads at least `verboseDefault` per agent. The alternative of making these global-only would require removing existing runtime/type surface and is a broader product change.
