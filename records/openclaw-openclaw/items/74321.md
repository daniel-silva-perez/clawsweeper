## Summary

Keep open. Current main has partial mitigations, including bounded recovery and Telegram chunking, but it still records first live-send permanent failures as retryable queue failures and the shared permanent-error classifier still misses the reported message-too-long, 401, and generic 404 classes. The follow-up discussion also raises a real maintainer choice between a contained pattern patch and a channel-owned classifier seam.

## What This Changes

The issue asks outbound delivery recovery to classify permanent HTTP/provider delivery failures and stop retrying them, with Telegram long-message handling through splitting or dead-lettering.

## Best Possible Solution

Add a high-confidence permanent delivery error classifier that is applied on both first live-send failure and recovery replay, preferably through a channel-owned seam if maintainers approve that boundary. Permanent entries should move directly to failed/, transient errors should keep bounded retry/backoff, Telegram chunking should remain in place, and regression tests should cover message-too-long, auth, not-found, and ambiguous-400 behavior.
