## Summary

Keep open. Current main still leaves non-abort live delivery failures as retryable queue entries, and the recovery classifier does not match the reported permanent classes. The related fix PR and referenced commit are closed/unmerged and diverged from main, so they do not implement the behavior on current main.

## What This Changes

The issue asks outbound delivery to classify permanent HTTP/provider failures such as Telegram too-long, auth, and not-found errors and stop retrying them as stale queued messages.

## Best Possible Solution

Implement permanent delivery-error classification for both first live-send failures and recovery replay, while preserving bounded retry/backoff for transient errors. If maintainers approve a channel-owned classifier seam, put provider-specific rules there; otherwise a narrow shared classifier patch should cover high-confidence cases without treating every 400 as permanent.

## Reproduction Assessment

Yes. A focused harness can enqueue or live-send a delivery whose send function throws `400: Bad Request: message is too long`; current main records it through `failDelivery` and the classifier returns false, leaving it retryable. The issue also includes a concrete Telegram queue entry with the same failure mode.

## Solution Assessment

Unclear for the exact implementation path. The requested behavior is correct, but the discussion raises a real boundary choice between a small shared-regex fix and a channel-owned plugin/API seam; either solution must cover both live send and recovery paths.
