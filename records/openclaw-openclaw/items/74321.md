## Summary

Keep open. The checked-out current main still treats the reported permanent delivery failures as retryable in both the live-send failure path and the recovery replay path, although Telegram text chunking now reduces one trigger for oversized messages. The linked attempted fix was closed unmerged and the newer related issue is broader, so neither safely closes this issue.

## What This Changes

The issue asks OpenClaw to classify high-confidence permanent outbound delivery failures such as Telegram message-too-long, auth, and not-found errors, move them out of the retry queue, and preserve transient retry behavior.

## Best Possible Solution

Implement conservative permanent-error classification for high-confidence non-transient delivery failures at the outbound queue boundary, covering both live send failures and recovery replay, while preserving bounded retry/backoff for transient failures and keeping Telegram chunking intact. A follow-up can move channel-specific policy into a plugin-owned seam if maintainers want that architecture cleanup.

## Reproduction Assessment

Yes. The current-main static path is high-confidence: a Telegram `400: Bad Request: message is too long` string does not match `isPermanentDeliveryError`, so recovery calls `failDelivery`, and the live send catch path also calls `failDelivery` for non-abort errors.

## Solution Assessment

Yes for a narrow repair, not for closing. The best immediate fix is to classify only high-confidence permanent delivery failures in both live and recovery paths, add regression tests, and avoid treating every 400 as permanent; a larger channel-owned seam is a maintainability option rather than a prerequisite for the bug fix.
