## Summary

Keep open. Current main still misses the reported permanent-error classes in both the live send failure path and the recovery classifier, so bounded retries and Telegram chunking do not fully solve the phantom replay problem.

## What This Changes

Requests outbound delivery to classify permanent HTTP/client errors such as Telegram message-too-long, auth failures, and not-found responses, then halt retries or move entries to failed instead of backing off indefinitely.

## Best Possible Solution

Add a maintainable permanent-delivery-error classification path that runs on both first live send failure and recovery replay, with Telegram-owned handling for message-too-long/auth/not-found cases, preserved transient retry/backoff behavior, and regression tests for both the classifier and queue cleanup behavior.

## Reproduction Assessment

Yes. A queued delivery whose send function throws `Call to 'sendMessage' failed! (400: Bad Request: message is too long)` follows the current unclassified error path into `failDelivery`, and the same non-abort live send failure is recorded with `failDelivery` before any permanent classification.

## Solution Assessment

Unclear pending maintainer direction. Extending the shared regex list would fix the observed cases quickly, but a channel-owned classifier seam is the cleaner long-term boundary because the shared core classifier already contains Telegram- and Matrix-specific strings.
