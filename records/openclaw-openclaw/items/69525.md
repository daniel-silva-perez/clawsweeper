## Summary

Current main still has the reported accountId propagation path: `runMessageAction` falls back to `input.defaultAccountId`, the allowed cross-provider policy returns without clearing it, and the send path forwards that accountId to the target plugin. Related PR #69529 targets the bug but remains open and bundles unrelated changes, so this issue should stay open.

## Best Possible Solution

Keep this issue open and land a narrow outbound fix: in `runMessageAction`, preserve an explicitly supplied `accountId`, keep same-provider defaults and agent binding behavior, but suppress `input.defaultAccountId` when cross-provider sends are allowed and `toolContext.currentChannelProvider` differs from the resolved target channel. Add a regression test in the message-action runner plugin-dispatch/context tests. Review or split #69529 rather than closing this before a merged fix.
