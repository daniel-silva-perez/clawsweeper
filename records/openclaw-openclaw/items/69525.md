## Summary

Current main still appears to have the reported accountId leak: `runMessageAction` resolves `accountId` from `input.defaultAccountId` before cross-provider policy, the enabled cross-provider path returns without clearing it, and outbound plugin dispatch passes that accountId through. Open PR #69529 proposes the exact suppression and a regression test, but it is still open and also bundles unrelated changes, so this issue should stay open until a narrow fix lands.

## Best Possible Solution

Keep this issue open and land a narrow fix in the outbound message-action path: preserve an explicitly provided `accountId`, suppress only the implicit `input.defaultAccountId` when `toolContext.currentChannelProvider` differs from the target channel and `allowAcrossProviders` is enabled, and add a focused regression test in the current message-action runner test split. Review or split #69529 rather than closing this before a merged fix.
