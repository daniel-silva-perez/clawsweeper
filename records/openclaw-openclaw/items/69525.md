## Summary

Current main still has the reported cross-provider accountId propagation path: `runMessageAction` falls back to the source session `defaultAccountId`, allowed cross-provider policy returns without clearing it, and plugin dispatch forwards that accountId to the target channel. Related PR #69529 remains open and targets this bug, so this issue should stay open until that work is reviewed, narrowed, and merged.

## What This Changes

Ensure cross-provider message sends do not pass the source session's implicit `defaultAccountId` into the target provider plugin while preserving explicit account selection and same-provider defaults.

## Best Possible Solution

Keep this issue open and review or split PR #69529. The narrow fix should live in `runMessageAction`: preserve explicit `accountId`, keep same-provider defaults and agent binding behavior, but suppress `input.defaultAccountId` when cross-provider sends are allowed and `toolContext.currentChannelProvider` differs from the resolved target channel. Add a regression test in the message-action runner plugin-dispatch or context tests.
