## Summary

Keep open. Current main still has the Mattermost `updateLastRoute` write guarded to direct messages only, so the proposed change is not implemented on main, but the PR is a draft that the author explicitly marked incomplete and later abandoned; it should be handled with the open Mattermost thread-routing cluster rather than merged or auto-closed as cleanup.

## What This Changes

The PR removes Mattermost's direct-message guard around inbound `updateLastRoute` recording and adds `threadId: effectiveReplyToId` to the persisted delivery context.

## Best Possible Solution

Keep #45082 as the canonical bug and either land a complete Mattermost-local fix covering stored delivery context, heartbeat routing for explicitly thread-scoped sessions, and the current-thread cron/tool gap with focused tests, or close this abandoned draft after a replacement/canonical path is chosen.

## Reproduction Assessment

Yes for a high-confidence source-level reproduction: with `replyToMode: "all"`, Mattermost computes a thread-scoped session key, but current main does not call `updateLastRoute` for non-direct inbound posts. I did not run a live Mattermost server in this read-only review.

## Solution Assessment

No. Persisting `deliveryContext.threadId` for all Mattermost session types is likely a useful prerequisite, but it is not the complete fix because heartbeat delivery and the cron tool session-target gap remain separate paths.
