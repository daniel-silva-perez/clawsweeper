## Summary

Keep this PR open for maintainer review. Current main still lacks iMessage default/named watcher dedupe, but this branch is not correct as-is because it scopes startup dedupe through the plugin-wide enabled predicate and also includes an unrelated outbound target resolution change with concrete regression risk.

## What This Changes

The PR adds iMessage watcher-key dedupe tests and a new `shouldStartIMessageAccount` gate wired into the iMessage plugin config, plus an outbound target early return when no implicit target, channel config, or active plugin is present.

## Best Possible Solution

Land one iMessage-owned startup dedupe that suppresses only duplicate monitors for the same effective `cliPath`/`dbPath` backend after enabled and configured checks, without changing global account enabled semantics or shared outbound target resolution. Cover the reported duplicate default/named config, disabled named accounts, unconfigured placeholders, and distinct backend cases, then close #65141 after the selected PR lands.

## Reproduction Assessment

Yes. The source-level path is high-confidence: a config with `channels.imessage.accounts.default` plus a named account exposes both ids, gateway startup starts each id independently, and the reporter in #65141 supplied process/message evidence showing two `imsg rpc` watchers and duplicate replies.

## Solution Assessment

No. The PR moves toward the right iMessage-owned fix, but the dedupe belongs on the watcher startup path or a startup-specific eligibility seam rather than `config.isEnabled`, and the unrelated outbound resolver change should be removed or handled separately.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Scope watcher dedupe to startup:** `extensions/imessage/src/shared.ts:88`
  - body: Wiring this into `config.isEnabled` changes the meaning of enabled for every iMessage account consumer, including runtime snapshots, `channels.status`, channel inspection, and channel selection. With `default` shadowed by a named watcher, those surfaces can report or treat the default account as disabled even though the channel is running, so this should be a startup-only gate.
  - confidence: 0.9
- **[P2] Preserve plugin target resolution:** `src/infra/outbound/targets.ts:70-82`
  - body: This early return runs before `resolveOutboundChannelPlugin`, so implicit deliveries with no `channels.<channel>` config cannot bootstrap or consult a channel plugin that could resolve `defaultTo` or `allowFrom`. That can turn previously resolvable delivery targets into `Missing target` failures outside the iMessage watcher fix.
  - confidence: 0.86
