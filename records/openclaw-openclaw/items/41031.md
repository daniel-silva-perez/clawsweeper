## Summary

Keep this PR open, but do not merge the current branch as-is. Current main still has the reported behavior gap because heartbeat session selection falls back to the main session when `heartbeat.session` is unset, while delivery is resolved later from `target` and `to`; the PR touches the right seam, but its async conversion needs a rebase update for newer synchronous callers and the semantics need maintainer confirmation against the published heartbeat docs.

## What This Changes

The PR changes heartbeat session resolution to derive the run session from explicit `heartbeat.target` plus `heartbeat.to` through outbound session routing, and adds a Telegram regression test for that case.

## Best Possible Solution

Land an updated implementation only if maintainers accept target-derived run-session semantics: derive the run session from explicit non-history `target` plus `to` only when no forced session or `heartbeat.session` is set, preserve global/subagent/`last`/`none`/isolated behavior, add regression coverage for Telegram and isolated/current-main callers, and update the heartbeat docs to match the accepted contract.
