## Summary

Keep open. Current main still chooses the heartbeat run session before resolving explicit heartbeat delivery, so `heartbeat.target` plus `heartbeat.to` can deliver to Telegram while the model run remains bound to the main session. The PR is still a focused implementation candidate, with one product/docs question around whether `target` should change run context or remain delivery-only.

## Best Possible Solution

Keep this PR open and review/adapt it on current main. The best implementation path is to derive the heartbeat run session from the resolved explicit delivery target via `resolveHeartbeatDeliveryTarget` plus `resolveOutboundSessionRoute` only when there is no forced `sessionKey` or configured `heartbeat.session`, while preserving existing fallback behavior and updating docs/tests if maintainers accept the target-derived session semantics.

## Risks / Open Questions

- Maintainers should confirm the intended semantics because current docs describe `target`/`to` as delivery routing while `session` controls run context.
- Changing the preflight session affects transcript selection, queue lane selection, event inspection/draining, dedupe metadata, and task timestamp writes; the final patch should preserve fallbacks for `target: last`, `target: none`, global scope, subagent keys, forced session keys, explicit `heartbeat.session`, account IDs, direct policy, and thread IDs.
- Security review did not find supply-chain-sensitive files in the proposed diff. The main review concern is functional: invoking provider route derivation during heartbeat preflight should not bypass existing routing, allowlist, direct-message, or same-agent restrictions.
