## Summary

Keep this PR open. Current main still does not derive the heartbeat run session from an explicit `heartbeat.target` plus `heartbeat.to`; the PR remains a focused implementation candidate, but it needs maintainer confirmation because current docs describe `target`/`to` as delivery-only.

## Best Possible Solution

Keep this PR open and review/adapt it on current main. The best implementation is to derive the heartbeat run session through `resolveHeartbeatDeliveryTarget` plus `resolveOutboundSessionRoute` only when no forced session or explicit `heartbeat.session` is present and both `heartbeat.target` and `heartbeat.to` name an explicit non-history target. Preserve the existing fallbacks, add the explicit Telegram group regression test, and update `docs/gateway/heartbeat.md` if maintainers accept target-derived run-session semantics.

## Risks / Open Questions

- The PR changes heartbeat transcript/session selection semantics for explicit delivery targets, which has privacy and context implications because the model would see a different session history.
- The final implementation must preserve existing behavior for global scope, forced session keys, explicit `heartbeat.session`, subagent guards, `target: last`, `target: none`, account IDs, direct-policy checks, and isolated heartbeat sessions.
- Docs currently describe `target`/`to` as delivery-only, so accepting the behavior change should include a heartbeat docs update.
