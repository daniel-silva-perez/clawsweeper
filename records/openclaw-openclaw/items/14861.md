## Summary

Keep open. Current main still does not provide the requested structured Gateway boot outcome summary, and the concrete Gmail watcher case remains silent when `hooks.enabled` is true but `hooks.gmail.account` is absent because the startup path never calls the lifecycle wrapper in that state.

## What This Changes

The issue requests a Gateway startup summary that reports which subsystems, hooks, and sidecars started, skipped, or failed, including visible Gmail watcher non-start reasons.

## Best Possible Solution

Add a structured, redacted Gateway startup outcome collector emitted after sidecars settle, covering Gmail watcher, internal hooks, channels, plugin services, memory scheduling, recovery sidecars, and `gateway_start` hooks while preserving existing readiness and failure behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: with `hooks.enabled=true` but no `hooks.gmail.account`, `startGatewaySidecars` bypasses the Gmail lifecycle wrapper, and the lifecycle wrapper/test path still suppresses expected skip reasons when invoked.

## Solution Assessment

Yes. The narrow maintainable fix is to collect and log startup outcomes around the existing sidecar flow rather than changing hook/channel startup semantics or making optional subsystem skips fatal.
