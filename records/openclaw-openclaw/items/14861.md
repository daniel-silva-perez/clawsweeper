## Summary

Keep open. Current main still lacks the requested structured Gateway boot outcome summary, and the concrete Gmail watcher gap remains: the Gateway only calls the Gmail lifecycle wrapper when an account is already configured, while the wrapper still suppresses expected skip reasons when it is called.

## What This Changes

Request to add Gateway boot observability that reports which startup subsystems and hooks started, skipped, or failed, using Gmail watcher skip reasons as the concrete example.

## Best Possible Solution

Add one structured, redacted Gateway startup outcome collector emitted after sidecars settle, covering Gmail watcher, internal hooks, channels, plugin services, memory scheduling, recovery sidecars, and gateway_start hooks without turning optional skips into fatal startup errors.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: with hooks enabled but hooks.gmail.account absent, startGatewaySidecars never calls the Gmail lifecycle wrapper, and the wrapper/test path still suppresses expected skip reasons when invoked.

## Solution Assessment

Yes. The narrow maintainable fix is to collect and log startup outcomes around the existing sidecar flow rather than changing hook/channel startup semantics or making optional subsystem skips fatal.
