## Summary

Keep open. Current `main` still has timing traces, readiness gating, generic `gateway ready` logging, Gmail watcher warnings for some failures, and a plugin-init failure warning, but it does not collect or emit the requested structured Gateway boot outcome summary. The concrete Gmail watcher gap remains: startup skips the lifecycle wrapper unless `hooks.gmail.account` is already set, and the wrapper still suppresses the missing-account reason when called.

## What This Changes

The issue requests a Gateway boot observability summary that reports which subsystems, hooks, and sidecars started, skipped, or failed, including Gmail watcher non-start reasons.

## Best Possible Solution

Add one concise, structured, redacted Gateway boot outcome collector after sidecars settle, covering Gmail watcher, internal hooks, channels, plugin services, memory scheduling, recovery sidecars, and `gateway_start` hooks while preserving current readiness and failure-swallowing behavior.
