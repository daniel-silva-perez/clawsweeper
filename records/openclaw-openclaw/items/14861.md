## Summary

Keep open. Current main still has startup trace timings, readiness probes, plugin/service failure logs, and targeted Gmail watcher lifecycle logs, but it does not emit the requested aggregate Gateway boot summary with per-subsystem started/skipped/failed outcomes. The specific Gmail missing-account path is still silent during normal gateway startup.

## Best Possible Solution

Keep this open and implement a structured Gateway startup outcome collector around `startGatewaySidecars`/post-attach startup. It should record started/skipped/failed outcomes for Gmail watcher, internal hooks, channels, plugin services, memory, recovery/restart sidecars, and related startup tasks, then emit one concise normal boot summary after sidecars settle. Include regression coverage for `hooks.enabled=true` with missing `hooks.gmail.account` and for a representative aggregate summary.

## Clownfish Work Prompt

Implement the startup observability request from openclaw/openclaw#14861. Add a structured Gateway boot outcome collector around `startGatewaySidecars` and post-attach sidecar startup. Record safe started/skipped/failed outcomes for the existing startup phases: session lock cleanup, Gmail watcher, Gmail model validation, internal hooks, channels, plugin services, memory backend scheduling, restart sentinel, subagent recovery, main-session recovery, and any existing startup task helper that already reports skipped/failed results. Emit one concise normal startup summary after sidecars settle, before or near the existing `gateway ready` log. Preserve current startup behavior; do not make expected disabled/skipped components fatal. Specifically cover `hooks.enabled=true` with missing `hooks.gmail.account`: the summary should surface `gmail-watcher: skipped/not started (no gmail account configured)` or equivalent safe wording. Keep log output redacted and operational: no secrets, tokens, account identifiers, hostnames, webhook payloads, or raw plugin errors beyond existing safe error strings. Do not introduce bundled plugin ids into core. Add focused tests in `src/gateway/server-startup-post-attach.test.ts` and, if needed, `src/hooks/gmail-watcher-lifecycle.test.ts` or a small new helper test. Update user-facing docs only if a new documented log surface or flag is added. Because this is user-facing observability, add a single-line changelog entry under the active version with `Thanks @slucasmyer` if the repo's current changelog policy requires one for this change.
