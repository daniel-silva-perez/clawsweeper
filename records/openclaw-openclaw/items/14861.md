## Summary

Keep open. Current main still lacks the requested structured Gateway boot outcome summary, and the concrete Gmail watcher observability gap remains: startup bypasses the Gmail lifecycle wrapper unless an account is configured, while the wrapper still suppresses expected skip reasons when it is called.

## What This Changes

Request to add Gateway startup observability that reports which subsystems, hooks, and sidecars started, skipped, or failed, with Gmail watcher missing-account and missing-gog cases as concrete examples.

## Best Possible Solution

Add one structured, redacted Gateway boot outcome collector after sidecars settle, covering Gmail watcher, internal hooks, channels, plugin services, memory scheduling, recovery sidecars, and gateway_start hooks while preserving current readiness, skip, and failure-swallowing semantics.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: with hooks enabled but hooks.gmail.account absent, startGatewaySidecars never calls the Gmail lifecycle wrapper, and the wrapper/test path still suppresses expected skip reasons when invoked.

## Solution Assessment

Yes. The narrow maintainable solution is to collect and log startup outcomes around the existing startup sidecar flow rather than making optional subsystem skips fatal or changing hook/channel startup semantics.
