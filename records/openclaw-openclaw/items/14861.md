## Summary

Keep open. Current main still has adjacent startup observability, but not the requested structured Gateway boot outcome summary. The concrete Gmail watcher example remains valid: normal startup only calls the Gmail lifecycle logger when an account is already configured, and the lifecycle logger suppresses the missing-account skip reason when it is called. Existing readiness, startup trace, and plugin failure logs do not collect started/skipped/failed outcomes across Gateway sidecars and hooks.

## What This Changes

Request a Gateway boot summary that reports which sidecars, hooks, and subsystems started, skipped, or failed, with Gmail watcher skip reasons as the concrete example.

## Best Possible Solution

Add one concise, structured, redacted Gateway boot outcome collector after sidecars settle, covering observable startup outcomes for Gmail watcher, internal hooks, channels, plugin services, memory scheduling, recovery sidecars, and gateway_start hooks while preserving current readiness and failure-swallowing behavior.
