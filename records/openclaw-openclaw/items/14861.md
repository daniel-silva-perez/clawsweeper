## Summary

Current main still lacks the requested aggregate Gateway boot summary. Startup now has timing traces, readiness gating, and targeted sidecar logs, but it does not collect and emit per-subsystem started/skipped/failed outcomes, and the Gmail missing-account case remains silent in normal gateway startup.

## Best Possible Solution

Implement a structured, redacted Gateway startup outcome collector around startGatewaySidecars and post-attach startup. It should record safe started/skipped/failed outcomes for Gmail watcher, internal hooks, channels, plugin services, memory scheduling, restart/session recovery sidecars, and gateway_start hooks, then emit one concise normal boot summary after sidecars settle, with regression coverage for hooks.enabled=true and missing hooks.gmail.account.

## Repair Work Prompt

Implement the startup observability requested by this issue. Add a structured, redacted Gateway startup outcome collector around `startGatewaySidecars` and post-attach startup. It should record safe started/skipped/failed outcomes for Gmail watcher, internal hooks, channels, plugin services, memory scheduling, restart/session recovery sidecars, and gateway_start hooks where practical, then emit one concise normal boot summary after sidecars settle. Preserve existing readiness semantics and targeted warning/error logs. Do not log secrets, tokens, raw credential refs, full config objects, or sensitive account details. Add regression coverage for `hooks.enabled=true` with missing `hooks.gmail.account` so the boot summary surfaces that Gmail watcher was skipped/not started with a safe reason. Add/update focused tests around `src/gateway/server-startup-post-attach.ts`, `src/hooks/gmail-watcher-lifecycle.ts`, and any startup log helper you introduce. If the behavior is user-facing, add a single-line changelog entry in the active version section with `Thanks @slucasmyer`.
