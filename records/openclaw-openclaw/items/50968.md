## Summary

Keep this PR open. Current main still has the reported static LiteLLM and Copilot Proxy model limits, but the branch is stale against the current plugin-owned onboarding layout and the PR head still has a correctness bug that can treat fallback values as authoritative discovered limits.

## What This Changes

The PR adds LiteLLM-compatible `/model/info` probing during onboarding so Copilot Proxy and LiteLLM model configs can use discovered `max_input_tokens` and `max_output_tokens` instead of static 128k/8192 defaults.

## Best Possible Solution

Refresh this as a provider-owned implementation in the current `extensions/litellm` and `extensions/copilot-proxy` onboarding paths, requiring complete authoritative token metadata before overwriting saved limits and preserving existing tuned values when probing fails or returns incomplete data.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure a LiteLLM or Copilot Proxy model whose `/v1/model/info` reports a larger `max_input_tokens`, run onboarding, and inspect that current main still writes 128000/8192 from the static model builders.

## Solution Assessment

No. The intended direction is right, but this branch is not the best current solution because part of it targets deleted legacy command files and the remaining LiteLLM logic can mark partial metadata as discovered.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

- **[P2] Require complete token metadata before patching limits:** `src/commands/onboard-auth.config-litellm.ts:88-96`
