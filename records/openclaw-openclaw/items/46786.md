## Summary

Keep open. Current main still matches the reported routing chain: allowed elevated sessions without an explicit directive, persisted session override, or configured `agents.defaults.elevatedDefault` fall back to `on`; omitted `exec` calls inherit that active elevated default; and elevated target resolution routes to `gateway` unless the selected target is already `node`. The latest release tag still contains the same contract, and the related default-off PR is closed unmerged, so this needs maintainer product/security judgment rather than stale cleanup.

## What This Changes

The issue asks for `tools.elevated.enabled` to preserve sandbox exec defaults for non-elevated calls and make host execution opt-in via an explicit elevated request or approved session mode.

## Best Possible Solution

Maintainers should explicitly choose the elevated-mode security contract. If sandbox-default opt-in elevation is desired, update default resolution, directive persistence/status, exec inheritance, prompt wording, docs, and regression tests together; if default-on host execution is intentional, keep the implementation but make the exec docs and security guidance state the tradeoff clearly.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: an allowed sandboxed session with no explicit elevated default resolves to `on`, omitted `exec` calls inherit that elevated state, and elevated target resolution sends execution to `gateway`. I did not run a live Discord/Docker session because this was a read-only review, but current code and tests match the reported path.

## Solution Assessment

Unclear. The requested default-off or ask-as-opt-in behavior is a plausible safer direction, but it conflicts with current docs and tests that define default-on elevated mode and treat `ask` as an alias for `on`; maintainer product/security approval should decide the contract before code changes.
