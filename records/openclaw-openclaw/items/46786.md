## Summary

Keep open. Current main still has the behavior reported: when elevated is allowed and no session or agent default is set, the reply path resolves the elevated level to `on`; the exec tool treats `on` as an elevated request by default; and the exec target resolver sends elevated sandboxed execution to `gateway`. The linked fix PR was closed unmerged, and current docs/config reference still document `ask` as equivalent to `on` and `elevatedDefault` as defaulting to `on`, so this needs maintainer/product resolution rather than cleanup closure.

## Best Possible Solution

Keep this issue open for maintainer resolution. If the intended contract is sandbox-by-default with opt-in elevation, change the implicit elevated fallback and persistence defaults to `off`, ensure omitted `exec.elevated` keeps `host=auto` resolving to sandbox, define whether `ask` is a session-default mode or per-call approval mode, add regression tests for `tools.elevated.enabled` without `elevatedDefault`, and update the elevated/config docs. If maintainers intentionally want the current default-on behavior, they should make that product decision explicitly and close manually with that rationale.
