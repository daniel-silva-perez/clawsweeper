## Summary

Current main still has the reported elevated-exec routing contract: when elevated is allowed and no directive/session/config default is set, reply resolution falls back to elevated level `on`; the exec tool treats `on` like approval-gated elevated execution for calls that omit `elevated: false`; and elevated requests resolve `host=auto` to `gateway` unless the target is already `node`. The docs now describe `/elevated ask` as an alias for `on` and `agents.defaults.elevatedDefault` as defaulting to `on`, so the requested sandbox-default, per-call opt-in behavior remains unresolved and needs maintainer product/security review rather than cleanup closure.

## Best Possible Solution

Keep this issue open for a maintainer decision on the elevated-mode contract. If maintainers choose the reporter's safer opt-in behavior, update reply default resolution, directive persistence/status text, exec elevated-mode handling, system prompt copy, public docs, and focused regression tests for sandboxed sessions with `tools.exec.host=auto|sandbox` and explicit per-call `elevated` values. If the current default-on behavior is intentional, document that decision clearly and close only after maintainer confirmation.
