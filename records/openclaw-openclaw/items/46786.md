## Summary

Keep open. Current main and the latest release still preserve the reported routing chain: allowed elevated sessions without an explicit directive, persisted session level, or configured elevated default fall back to `on`; omitted `exec` calls inherit that active elevated state; and elevated target resolution routes to `gateway` unless the selected target is already `node`. This is a sandbox-boundary/security contract question, and the linked default-off PR was closed unmerged, so it needs maintainer product/security judgment rather than cleanup.

## What This Changes

Request that enabling `tools.elevated.enabled` not change default sandbox exec routing, with host elevation becoming opt-in per call or explicit per-session/default configuration.

## Best Possible Solution

Maintainers should explicitly choose the elevated-mode security contract. If sandbox-default opt-in elevation is desired, update default resolution, directive persistence/status, exec inheritance, prompt wording, docs, and regression tests together; if default-on host execution is intentional, keep the implementation but make the exec docs and security guidance state the tradeoff clearly.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: a sandboxed, allowed elevated session with no explicit elevated default resolves to `on`; an omitted `exec` call inherits that elevated state; and elevated `auto` routing resolves to `gateway`. I did not run a live Discord/Docker session because this review was read-only.

## Solution Assessment

Unclear. The requested default-off or ask-as-opt-in behavior is a plausible safer direction, but it conflicts with current docs and tests that define default-on elevated mode and treat `ask` as equivalent to `on`; maintainer product/security approval should decide the contract before code changes.
