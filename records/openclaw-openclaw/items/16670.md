## Summary

Keep open. Current main still documents and diagnoses memory embedding readiness, but the setup/onboarding/configure flows do not provide a dedicated memorySearch provider, model, API-key, or SecretRef setup step, and the mandatory behavior needs maintainer product and credential-UX direction.

## What This Changes

The issue requests making memory/embedding setup a prominent mandatory onboarding step that configures `memorySearch`, captures the needed embedding credential, and verifies the embedding endpoint.

## Best Possible Solution

Add a maintainer-approved memory setup path in CLI onboarding and `openclaw configure` that configures `agents.defaults.memorySearch` provider/model plus API-key or SecretRef input, verifies readiness with a controlled embedding probe or `openclaw memory status --deep`, and updates setup/configure/onboarding/memory docs for API-key, Codex OAuth, and daemon-environment behavior.

## Risks / Open Questions

- The remaining product decision is whether memory setup should be mandatory, prominently optional, or diagnostic-only.
- Credential UX is security-sensitive because embedding setup may involve plaintext API keys, SecretRefs, daemon environment visibility, and the Codex OAuth versus API-key distinction.
- A live embedding probe during onboarding could add network dependency, provider cost, latency, or failure modes unless designed carefully.
- Making memory setup mandatory could block users who intentionally rely on local/QMD/keyword-only or later-configured memory paths.
