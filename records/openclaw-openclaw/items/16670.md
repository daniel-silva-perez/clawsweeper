## Summary

Keep this issue open. Current main still has memorySearch documentation, doctor diagnostics, and `openclaw memory status --deep`, but the onboarding/setup and configure flows still do not offer a dedicated memory embedding provider/model/API-key or SecretRef setup step. The request also asks whether this should be mandatory, which is a maintainer product decision rather than a safe automatic cleanup close.

## Best Possible Solution

Keep this issue open for maintainer decision on whether memory embedding setup should be mandatory, prominently optional, or diagnostic-only. If approved, implement a focused setup/configure memory path that writes agents.defaults.memorySearch provider/model and API-key or SecretRef config, verifies readiness with an embedding probe or `openclaw memory status --deep`, and updates the public onboarding/configure/memory docs for API-key, Codex OAuth, and daemon environment behavior.

## Risks / Open Questions

- Making memory setup mandatory could block users who intentionally rely on auto-detection, keyless/local providers, QMD, or diagnostic-only setup, so the mandatory-vs-prominent-optional choice needs maintainer product judgment.
- Any implementation that prompts for embedding API keys must preserve the existing SecretRef and daemon environment semantics so LaunchAgent/systemd gateways can resolve credentials without leaking plaintext into service metadata.
- Related open PRs #44323 and #55684 improve validation/remediation around memorySearch, but they do not add the requested onboarding/configure setup step, so closing this issue now would lose the remaining product gap.
