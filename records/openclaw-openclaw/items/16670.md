## Summary

Keep open. Current main documents and diagnoses memorySearch readiness, but the setup/onboarding/configure flows still do not offer a dedicated memory embedding provider/model/API-key or SecretRef setup step. The remaining mandatory-versus-prominent-optional behavior needs maintainer product and credential-policy judgment.

## What This Changes

The issue asks for onboarding to include a prominent or mandatory memorySearch embedding provider/model/API-key setup and verification step.

## Best Possible Solution

Add a maintainer-approved memory setup path in CLI onboarding and openclaw configure that configures agents.defaults.memorySearch provider/model plus API-key or SecretRef input, verifies readiness through a controlled embedding probe or memory status --deep, and updates setup/configure/onboarding/memory docs for API-key, Codex OAuth, and daemon-environment behavior.

## Reproduction Assessment

Yes. Static inspection on current main reproduces the gap: setup only writes workspace/gateway defaults, the local wizard has no memory step, and configure has no memory/memorySearch section or dispatch path.

## Solution Assessment

Unclear. Adding a memory setup path is the right area, but making it mandatory may not be the narrowest maintainable UX without maintainer direction; a prominent optional SecretRef-aware step with diagnostics may be safer.
