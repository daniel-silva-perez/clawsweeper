## Summary

Keep open. Current main still has memorySearch docs, doctor guidance, and `openclaw memory status --deep`, but the setup/onboarding/configure flows do not provide a dedicated memory embedding provider/model/API-key or SecretRef setup step. The issue discussion and prior maintainer review comments support the gap, and the requested mandatory onboarding behavior needs maintainer UX and credential-policy judgment before implementation.

## What This Changes

Request that first-run setup/onboarding/configure prominently, possibly mandatorily, configure `agents.defaults.memorySearch` provider/model plus embedding credentials so semantic memory works across sessions.

## Best Possible Solution

Add a maintainer-approved memory setup path in CLI onboarding and `openclaw configure` that configures `agents.defaults.memorySearch` provider/model and API-key or SecretRef input, verifies readiness through a controlled embedding probe or `openclaw memory status --deep`, and updates setup/configure/onboarding/memory docs for API-key, Codex OAuth, and daemon-environment behavior.

## Reproduction Assessment

Yes. The gap is reproducible by static inspection on current main: setup only writes workspace/gateway defaults, the wizard flow has no memory step, and configure has no `memory`/`memorySearch` section or dispatch path.

## Solution Assessment

Unclear. The requested direction is valid, but making the step mandatory may not be the best maintainable UX without maintainer guidance; a prominently optional setup path with SecretRef-aware credential handling and clear diagnostics may be safer.
