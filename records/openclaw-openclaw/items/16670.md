## Summary

Keep open. Current main has memory-search docs, doctor diagnostics, and `openclaw memory status --deep`, but the setup, onboarding, and configure flows still do not provide a dedicated memory embedding provider/model/API-key or SecretRef setup step; the requested mandatory behavior also needs maintainer product and credential-policy judgment.

## What This Changes

Requests a prominent mandatory onboarding/setup step that configures `agents.defaults.memorySearch`, captures embedding credentials, and verifies embedding readiness.

## Best Possible Solution

Add a maintainer-approved memory setup path in CLI onboarding and `openclaw configure` that writes `agents.defaults.memorySearch` provider/model plus API-key or SecretRef configuration, verifies readiness through a controlled probe or `openclaw memory status --deep`, and updates setup/configure/onboarding/memory docs for API-key, Codex OAuth, and daemon-environment behavior.

## Reproduction Assessment

Yes. Static inspection on current main reproduces the gap: setup only writes workspace/gateway defaults, the local wizard has no memory step, and configure has no `memory`/`memorySearch` section or dispatch path.

## Solution Assessment

Unclear. Adding a memory setup path is the right area, but making it mandatory may not be the narrowest maintainable UX without maintainer direction; a prominent optional SecretRef-aware step with diagnostics may be safer.
