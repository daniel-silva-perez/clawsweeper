## Summary

Keep open. Current main still lacks a dedicated memory embedding provider/model/API-key or SecretRef setup path in `openclaw setup`, CLI onboarding, and `openclaw configure`; the existing memory docs, doctor checks, and `openclaw memory status --deep` are diagnostic/manual paths rather than onboarding setup. Making the step mandatory still needs maintainer product and credential-policy judgment.

## What This Changes

Request to add a prominent mandatory onboarding step that configures `agents.defaults.memorySearch`, captures the required embedding credential, and verifies the embedding endpoint.

## Best Possible Solution

Add a maintainer-approved memory setup path in CLI onboarding and `openclaw configure` that configures `agents.defaults.memorySearch` provider/model plus API-key or SecretRef input, verifies readiness through a controlled probe or `openclaw memory status --deep`, and updates setup/configure/onboarding/memory docs for API-key, Codex OAuth, and daemon-environment behavior.

## Reproduction Assessment

Yes. Static inspection on current main reproduces the gap: setup only writes workspace/gateway defaults, the onboarding wizard has no memory step, and configure has no memory section or dispatch path.

## Solution Assessment

Unclear. A dedicated memory setup path is the right area, but making it mandatory may not be the narrowest maintainable UX without maintainer direction; a prominent SecretRef-aware optional step with diagnostics may be safer.
