## Summary

Keep open. Current main already covers the rapid short-message batching part through `messages.inbound`, and the related #51361/#46445 cleanup context supports that. This issue is broader, though: it also asks for semantic long-message multi-topic handling and adaptive behavior, and current code/docs/schema expose only timing-based text debounce, not topic detection, splitting, or a generic semantic preprocessing hook.

## Best Possible Solution

Keep this issue open, but narrow the remaining scope around semantic preprocessing beyond the shipped debounce layer. The best path is not to reimplement `messages.inbound`; it is to decide whether OpenClaw should add a gateway-level input preprocessor or plugin/SDK hook for long-message topic detection, optional split-vs-unified response behavior, adaptive windows, and clear media/control-command semantics, with focused docs and tests.

## Risks / Open Questions

- Closing this as implemented would drop the still-missing long-message multi-topic handling and adaptive-window parts of the request.
- A future implementation needs careful product/API design because splitting a user message into multiple turns can affect transcript semantics, command handling, reply threading, latency, cost, and privacy.
