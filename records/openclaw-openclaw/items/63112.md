## Summary

Keep open. Current main still lacks the warning, and this PR is the active contributor implementation for open #63107; it should proceed through normal maintainer review and validation rather than ClawSweeper closure or a replacement repair branch.

## What This Changes

The PR adds a shared shell-command heuristic, non-blocking stderr warnings in cron add/edit for main-session systemEvent payloads that look like shell commands, and focused CLI/shared tests.

## Best Possible Solution

Land a narrow cron CLI fix that warns before the gateway RPC for both add and edit, covers existing-main edits when --session is omitted, uses the normalized effective session target, reuses paginated existing-job lookup, and preserves current cron runtime semantics.

## Review Findings

Overall correctness: patch is correct

Overall confidence: 0.84

Full review comments:

- **[P3] Paginate edit-session lookup before skipping warnings:** `src/cli/cron-cli/register.cron-edit.ts:326-328`
  - body: When `--session` is omitted, this lookup checks only one `cron.list` page. The gateway caps list pages at 200 jobs, so editing a main-session job beyond the first page can still skip the new warning. Reuse the existing paginated edit lookup before deciding the effective session.
  - confidence: 0.86
- **[P3] Use the normalized session target for edit warnings:** `src/cli/cron-cli/register.cron-edit.ts:318`
  - body: The edit path normalizes accepted values like `--session MAIN` to `main` for the patch, but this guard compares the raw option to `"main"`. That accepted casing updates a main-session systemEvent job without the new warning; base the guard on the normalized `sessionTarget`.
  - confidence: 0.78
