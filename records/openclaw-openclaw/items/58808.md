## Summary

Keep open. Current main still has the reported sender-context gap for channel agent-tool factories, and the PR appears to be a narrow additive fix, but it touches a public plugin/channel contract and needs maintainer review plus focused validation before merge.

## What This Changes

The PR adds optional `requesterSenderId` and `senderIsOwner` params to `ChannelAgentToolFactory`/`listChannelAgentTools` and forwards the existing `createOpenClawCodingTools` sender options into channel-owned agent-tool factories.

## Best Possible Solution

Land this PR or an equivalent narrow additive patch after maintainer review, with a regression test for trusted sender-context propagation and any SDK docs/changelog updates maintainers require; keep the DMWork-specific access policy in the plugin rather than hardcoding it in core.

## Reproduction Assessment

Yes. On current main, a channel plugin factory reached through `listChannelAgentTools` can observe only `cfg`, while the adjacent core `message` tool receives `requesterSenderId` and `senderIsOwner` from the same `createOpenClawCodingTools` options.

## Solution Assessment

Yes, the proposed direction is the narrowest maintainable core change: widen the generic channel agent-tool factory params and forward existing server-injected context. The safer completion is to add regression coverage and let maintainers decide public SDK docs/changelog treatment.
