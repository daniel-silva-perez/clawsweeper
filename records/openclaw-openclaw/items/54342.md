## Summary

Keep this issue open. Current main has adjacent improvements for ACP task records, `sessions_send`, and hook delivery suppression via `deliver:false`, but the central hook/headless persistent ACP workflow is still blocked: `sessions_spawn(runtime:"acp", mode:"session")` still requires `thread:true`, the accepted spawn result still does not expose ACP backend/agent resume identity, and `/hooks/agent` still routes through isolated embedded-agent execution rather than ACP `runTurn` dispatch.

## Best Possible Solution

Keep this issue open and narrow the implementation to a conservative headless ACP persistence contract: add an explicit unbound/headless persistent ACP mode or narrowly relax the `mode:"session"` thread guard once lifecycle cleanup is defined, expose ACP backend/agent session identity through spawn/status, add regression coverage for channel-less hook-triggered persistent ACP spawns, and decide whether hook requests targeting known ACP session keys should dispatch to `acpManager.runTurn` or whether `sessions_send` is the documented follow-up path.

## Risks / Open Questions

- Closing now would leave hook-triggered/headless ACP workflows without a supported persistent `mode:"session"` path for resume or multi-turn continuity.
- A future fix should not simply remove the thread guard without defining lifecycle, cleanup, ownership, visibility, and timeout semantics for unbound persistent ACP sessions.
- Exposing backend ACP resume identifiers or adding hook-to-ACP dispatch should preserve existing ACP policy checks and session visibility boundaries.
