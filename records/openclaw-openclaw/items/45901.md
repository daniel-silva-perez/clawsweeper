## Summary

Keep open. Current main still creates managed session directories with plain recursive mkdir in the store, transcript, setup/onboarding, model-probe, gateway agent creation, and lock paths, so the security hardening is still useful. This PR has addressed much of the earlier symlink/custom-path review history, but the branch is not landable as-is because it is stale against current main and drops the compaction-checkpoint session ID validation now present in `src/config/sessions/paths.ts`.

## What This Changes

The PR adds private-permission helpers for managed session directories, routes setup/onboarding/agent creation/store/transcript/lock/probe flows through them, and adds permission, symlink, custom-root, and custom-path coverage.

## Best Possible Solution

Rebase or update this PR, keep the hardening centralized around managed session directories, preserve plain mkdir behavior for truly caller-supplied custom paths, carry forward the current compaction-checkpoint session ID guard and test, then land after maintainer security review and targeted session/setup/gateway validation.

## Reproduction Assessment

Yes. On POSIX, current main still has managed session paths that call recursive mkdir without an explicit private mode, and Node documents the mkdir mode default as `0o777`; a fresh setup or session-store/transcript write under a permissive umask can reproduce the reported `0755` directory mode.

## Solution Assessment

Yes as a product/code direction, but not as a ready-to-merge branch. A centralized managed-session-directory helper is the narrow maintainable fix, while this PR first needs to be updated so it does not regress current session ID validation.
