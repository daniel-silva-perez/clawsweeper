## Summary

Keep open. Current main still lacks compaction lifecycle event emission from embedded-runner overflow and timeout recovery, so this PR is not obsolete or implemented. The PR targets a real Control UI observability gap, but should be revised before merge to align completion semantics and avoid incorrect memory compaction accounting.

## Best Possible Solution

Keep this PR open as a valid implementation candidate. Best path: revise `src/agents/pi-embedded-runner/run.ts` so overflow and timeout recovery emit compaction lifecycle events consistent with `handleCompactionEnd`, update memory accounting to only count completed compactions, and add targeted runner recovery tests for both event sinks before merge.

## Risks / Open Questions

- The PR computes `completed` as `ok && compacted`, while the canonical subscribe handler uses result presence and abort state; that semantic mismatch should be resolved before merge.
- Because `agent-runner-memory.ts` treats any compaction end event as completed, failed recovery end events could incorrectly increment memory compaction state unless the consumer checks `completed === true` or the emission contract avoids failed end events.
- Focused regression coverage is still missing for both global `emitAgentEvent` and `params.onAgentEvent` compaction start/end payloads across successful and failed overflow/timeout recovery paths.
