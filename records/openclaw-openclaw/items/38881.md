## Summary

Keep this PR open. Current main still supports `agents.defaults.imageModel` for media-understanding and image/PDF tool fallback, and WebChat now offloads images from text-only sessions into `ctx.MediaPaths`, but the final agent reply path still passes the normal selected provider/model into `runPreparedReply` and uses the regular model fallback chain. The PR is therefore not obsolete, but it still needs maintainer review and changes before merge because the autoswitch must be coordinated with media-understanding, attachment typing, alias/fallback resolution, and explicit override semantics.

## Best Possible Solution

Keep this PR open for maintainer review or author follow-up. The best fix should live in the reply/model-selection path: detect normalized current-turn image attachments, resolve `agents.defaults.imageModel.primary` and fallbacks through the real alias/catalog helpers, switch the final `runPreparedReply` model only when the active model is positively known not to accept images, respect heartbeat/session/channel/user overrides, coordinate with media-understanding to avoid duplicate analysis unless intended, and add focused regression tests proving the final agent run receives the configured vision model.

## Risks / Open Questions

- Merging the PR as-is could make one image turn do both media-understanding and a final vision-model run unless the switch is moved before media-understanding or media-understanding is skipped/re-evaluated after switching.
- The PR’s proposed image detection must use normalized current-turn image metadata, not generic `MediaPath`/`MediaPaths` alone, or non-image attachments and URL-only image inputs can be mishandled.
- Autoswitch should require positive evidence that the active model cannot accept images; treating unknown catalog entries as non-vision can silently reroute custom/new models to another provider, with cost and privacy implications.
- Alias, primary/fallback-only imageModel configs, heartbeat/session/channel overrides, and normal model fallback state need one coherent policy before this is safe to merge.
