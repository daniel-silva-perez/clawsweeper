## Summary

Keep open: the PR targets a real bug that is still present on current main and the latest release, but the proposed path construction still fails PDF.js validation on Windows and the changelog has a stray marker.

## What This Changes

The branch resolves PDF.js standard font assets from the installed `pdfjs-dist` package root, passes `standardFontDataUrl` into the bundled `document-extract` PDF loader, adds a legacy type shim, a mock regression test, and a changelog entry.

## Best Possible Solution

Revise this PR or land #74355/equivalent so the plugin passes a plain filesystem path ending in literal `/`, keeps the type surface aligned with PDF.js, includes real standard-font PDF coverage, removes the changelog marker, and then lets the linked bug close through the merged PR.

## Reproduction Assessment

Yes. The original bug is reproducible with the fallback extractor on a standard-font PDF while current main omits `standardFontDataUrl`; the PR-specific regression is reproducible by evaluating the branch on Windows where `path.sep` is `\` but PDF.js requires `/`.

## Solution Assessment

No as written. Resolving from `pdfjs-dist/package.json` is the right direction, but the PR should append a literal `/`, remove the changelog marker, and ideally use the real-PDF coverage from the overlapping candidate.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Use a forward slash for the PDF.js font path:** `extensions/document-extract/document-extractor.ts:83`
