## What This Changes

Recover gateway startup and `doctor --repair` when interrupted bundled plugin runtime dependency staging leaves a corrupt `node_modules` tree that passes the current package sentinel checks.

## Best Possible Solution

Make the bundled runtime dependency scan/repair path distinguish a fully materialized install root from a partially written package tree, then cover both gateway startup and doctor repair with focused incomplete-install tests.

## Reproduction Assessment

Yes, there is a high-confidence unit-level reproduction path: build a packaged plugin root where the required dependency has a satisfying `node_modules/<dep>/package.json` but the install root lacks a completed materialization signal or complete package contents, then observe `scanBundledPluginRuntimeDeps` report no missing deps and both gateway and doctor return before repair. I did not run tests because this review was required to stay read-only.

## Solution Assessment

Unclear until #75310 is reviewed, because the requested direction is narrow and appropriate, but the exact completion signal must match current main's generated install-manifest behavior rather than the legacy `.openclaw-runtime-deps.json` cleanup path. The safest fix stays inside bundled-runtime-deps detection plus the two early-return call sites, without broad package-manager or directory-wipe changes.
