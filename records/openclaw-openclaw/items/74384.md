## Summary

Keep open. Current main still has a credible OpenClaw tool-policy path for the reported 4.26 behavior: required plugin tools are materialized, then the default coding profile can filter third-party tools down to the two built-in memory names. The related hook-access workaround does not cover this tool-visibility contract.

## What This Changes

The issue asks OpenClaw to restore agent visibility for all required tools registered by a third-party plugin through `api.registerTool()` after the 4.26 plugin SDK/tool-policy changes.

## Best Possible Solution

Define and document the intended contract between required plugin tools and tool profiles. A durable fix should either add a safe slot/plugin-owned policy mechanism for active memory plugins, or make the opt-in path and diagnostics explicit, with regression coverage for a memory plugin that registers more than `memory_search` and `memory_get`.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the core mechanism: a synthetic enabled memory plugin registering `memory_search`, `memory_get`, and another required tool under `tools.profile: "coding"` should exercise `createOpenClawCodingTools()` and the policy pipeline. The exact third-party Windows setup has not been live-verified here.

## Solution Assessment

Is this the best way to solve the issue? Unclear because there is no proposed patch and the product/security contract is the open decision. A naive expose-all fix is likely too broad; a maintainer-chosen plugin/memory policy or explicit opt-in path is safer.
