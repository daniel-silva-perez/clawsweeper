## Summary

Keep open. Current main still has a plausible core/tool-policy path for the reported 4.26 symptom: plugin tools are materialized, then filtered through the active tool profile/allowlist, and the default coding memory group only includes `memory_search` and `memory_get`. The related #73748 workaround addresses conversation hook access, but this report is about agent-visible tools registered through `api.registerTool()`.

## What This Changes

The issue reports that after upgrading to OpenClaw 4.26, a third-party memory plugin using `api.registerTool()` exposes only `memory_search` and `memory_get` to the agent instead of its full registered tool set.

## Best Possible Solution

Define the intended contract for required plugin tools under tool profiles, especially for active memory-slot plugins. A durable fix should either classify slot-owned memory plugin tools through an explicit, safe policy/group mechanism or provide clear doctor/docs guidance for opting into the full plugin tool set, then add regression coverage for a memory plugin that registers more than `memory_search` and `memory_get`.
