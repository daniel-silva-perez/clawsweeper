## Summary

Keep open. Current main still has a credible OpenClaw-side policy path for the report: the related conversation-hook workaround from #73748 addresses capture hooks, but required plugin tools can still materialize and then be removed by the default coding profile unless they are explicitly granted.

## What This Changes

The issue reports that after upgrading to OpenClaw 4.26, a third-party memory plugin's required `api.registerTool()` tools are no longer all exposed to the agent, leaving only `memory_search` and `memory_get` visible.

## Best Possible Solution

Define the intended contract between required third-party plugin tools and restrictive tool profiles, then align code, docs, diagnostics, and tests. A durable outcome should either add a maintainer-approved policy slot for active memory plugins or make the `tools.alsoAllow`/plugin opt-in path explicit enough that plugin authors and users can configure it safely.

## Reproduction Assessment

Yes for the core mechanism: enable a synthetic memory-like plugin that registers `memory_search`, `memory_get`, and one additional non-optional tool, then run with `tools.profile: "coding"` through `createOpenClawCodingTools` and the policy pipeline. The exact external Windows setup was not live-verified here.

## Solution Assessment

Unclear because there is no proposed patch and the open question is the intended product/security contract. A broad expose-all-required-plugin-tools change is likely too risky; a maintainer-approved memory/plugin policy or explicit opt-in path is safer.
