## Summary

Keep open. Current main still has a plausible OpenClaw tool-policy regression path for the reported 4.26 symptom: plugin tools are materialized, but the active profile allowlist can filter third-party required tools unless their names or plugin id are explicitly allowed. This is recent, reproducible from the current code path, and not superseded by the related conversation-hook workaround.

## What This Changes

The issue asks OpenClaw to restore agent visibility for all required tools registered by a third-party plugin through `api.registerTool()` after the 4.26 plugin SDK/tool-policy changes.

## Best Possible Solution

Define the intended contract between required plugin tools and tool profiles. A durable fix should either add a safe slot/plugin-owned policy mechanism for active memory plugins, or provide clear opt-in diagnostics/docs for exposing the full plugin tool set, with regression coverage for a memory plugin that registers more than `memory_search` and `memory_get`.

## Reproduction Assessment

Yes for the core mechanism: a synthetic enabled memory plugin that registers `memory_search`, `memory_get`, and another required tool under `tools.profile: "coding"` should reproduce the two-tool visibility pattern through `createOpenClawCodingTools()` and `applyToolPolicyPipeline()`. The exact third-party Windows setup was not live-verified in this read-only review.

## Solution Assessment

No automated code fix should be queued yet. Exposing every required third-party tool by default may be too broad; the safer direction is a maintainer-chosen policy contract or explicit opt-in path for active memory plugin tools.
