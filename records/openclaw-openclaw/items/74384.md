## Summary

Keep open. Current main still has a credible OpenClaw policy path for the reported behavior: required third-party plugin tools are loaded, but the default coding profile can filter them back down to the explicitly profiled memory tool names. This is not resolved by the related conversation-hook workaround.

## What This Changes

The issue reports that a third-party memory plugin's required api.registerTool() tools are no longer exposed to the agent after upgrading to OpenClaw 4.26, leaving only memory_search and memory_get available.

## Best Possible Solution

Define the intended contract between required plugin tools and restrictive tool profiles, then align code, docs, diagnostics, and tests. A durable outcome should either provide a maintainer-approved policy slot for active memory plugins or make the tools.alsoAllow/plugin opt-in path explicit enough that plugin authors and users can configure it safely.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the core mechanism: enable a synthetic memory-like plugin that registers memory_search, memory_get, and one additional required tool, then run with tools.profile: "coding" through createOpenClawCodingTools and the policy pipeline. The exact external Windows installation was not live-verified here.

## Solution Assessment

Is this the best way to solve the issue? Unclear because there is no proposed patch and the open question is the intended product/security contract. A broad expose-all-required-plugin-tools change is likely too risky; a maintainer-approved memory/plugin policy or explicit opt-in path is safer.
