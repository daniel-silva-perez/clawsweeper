## Summary

Keep open. Current main still has a plausible core policy path for the reported 4.26 symptom: required plugin tools are materialized, but the default coding profile filters the final tool list through `group:memory`, which only includes `memory_search` and `memory_get`; the post-report docs update explains `alsoAllow` for restrictive profiles but does not resolve the plugin SDK contract mismatch.

## What This Changes

The issue reports that a third-party memory plugin’s required tools registered with `api.registerTool()` are no longer exposed to the agent after upgrading to OpenClaw 4.26, leaving only `memory_search` and `memory_get` visible.

## Best Possible Solution

Define the intended contract between required plugin tools and tool profiles, especially for active memory-slot plugins. A durable fix should either give memory-slot plugins an explicit safe group/policy mechanism for required tools or make the opt-in path clear through doctor/docs, with regression coverage for a memory plugin registering more than `memory_search` and `memory_get`.

## Reproduction Assessment

Yes for the core policy path: a non-optional plugin tool set containing `memory_search`, `memory_get`, and additional memory/share/skill tools under the default coding profile will be materialized and then filtered through the profile allowlist. The exact reporter environment was not live-tested.

## Solution Assessment

Unclear. Exposing all required plugin tools by default would solve the degraded-plugin symptom, but it may not be the safest maintainable contract for mutating/share/publish capabilities; an explicit memory-slot policy or documented opt-in may be safer.
