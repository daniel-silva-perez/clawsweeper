## Summary

Keep open. Current main still has the reported custom-provider onboarding gap: custom OpenAI-compatible setup verifies and persists Chat Completions, while OpenAI Responses support exists in the config/runtime surface and docs only through manual configuration or Azure-specific onboarding paths.

## What This Changes

The issue requests that custom OpenAI-compatible model onboarding work for a provider whose `/v1/models` endpoint succeeds but whose verification fails because OpenClaw probes Chat Completions instead of Responses.

## Best Possible Solution

Add first-class custom-provider onboarding support for OpenAI Responses endpoints: interactive selection or detection, non-interactive flag parsing, verification against `/responses`, persistence as `models.providers.<id>.api: "openai-responses"`, and aligned docs/tests while preserving the existing Chat Completions default for compatibility.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: choose custom OpenAI-compatible onboarding and OpenClaw builds a `/chat/completions` probe, then persists `api: "openai-completions"`; there is no custom Responses option to select or parse.

## Solution Assessment

Yes. A narrow onboarding/config/docs/test change is the best path because the runtime and config schema already support `openai-responses`; a broader core provider rewrite is unnecessary for the reported failure.
