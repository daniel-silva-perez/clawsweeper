## Summary

This should stay open. Current main still exposes manual `openai-responses` custom-provider support, but the custom-provider onboarding and non-interactive flags still model OpenAI-compatible custom endpoints as Chat Completions only, so a Responses-only endpoint can still fail verification with the reported status path.

## What This Changes

The issue reports that custom model verification fails with status 503 for an OpenAI-compatible endpoint that lists models successfully and asks for parity with the successful Anthropic-compatible setup path.

## Best Possible Solution

Add first-class custom-provider onboarding support for Responses API endpoints while preserving the existing Chat Completions default and the manual config path, so users can select or detect Responses support and persist `models.providers.<id>.api: "openai-responses"`.

## Repair Work Prompt

Fix the custom-provider onboarding gap from https://github.com/openclaw/openclaw/issues/44870. Current main supports manual custom providers with `api: "openai-responses"`, but interactive and non-interactive onboarding only let users choose `openai` -> `openai-completions` or `anthropic` -> `anthropic-messages`, and the non-Azure OpenAI verification probe posts to `/chat/completions`. Preserve the current Chat Completions default for `openai`, keep Anthropic behavior unchanged, and do not add bundled-provider-specific IDs or hidden core migrations. Update interactive prompts, non-interactive flags/types/help, parser/config application, and focused tests. Update public docs where the custom-provider flags are documented.
