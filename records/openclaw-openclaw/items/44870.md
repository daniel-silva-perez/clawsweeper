## Summary

Keep open. Current main still has the reported custom-provider onboarding gap: ordinary custom OpenAI-compatible setup verifies and persists Chat Completions, while OpenAI Responses support exists only for manual config and Azure OpenAI-specific onboarding paths.

## What This Changes

The issue asks for custom OpenAI-compatible model verification to work for a Responses-style endpoint instead of failing with status 503 during onboarding.

## Best Possible Solution

Add first-class custom-provider onboarding support for OpenAI Responses endpoints: interactive choice or detection, non-interactive flag parsing, verification against `/responses`, persistence as `models.providers.<id>.api: "openai-responses"`, and aligned docs/tests while preserving the existing Chat Completions default for compatibility.
