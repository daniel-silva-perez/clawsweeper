## Summary

Keep open. Current main still supports `openai-responses` in the runtime and manual config, but custom-provider onboarding only offers OpenAI-compatible Chat Completions or Anthropic-compatible setup, so a Responses-only OpenAI-compatible endpoint can still fail verification with the reported status path.

## What This Changes

Support validating and configuring custom OpenAI-compatible endpoints that expose the Responses API instead of Chat Completions during onboarding.

## Best Possible Solution

Add first-class custom-provider onboarding support for OpenAI Responses endpoints: interactive selection or detection, non-interactive flag parsing, verification against `/responses`, config persistence as `models.providers.<id>.api: "openai-responses"`, and aligned docs/tests while preserving the existing Chat Completions default.
