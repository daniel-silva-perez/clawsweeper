## Summary

Keep open. Current main still leaves the reported custom-provider path unresolved: OpenClaw can run manually configured `openai-responses` providers, but custom onboarding only accepts OpenAI Chat Completions or Anthropic Messages compatibility and verifies ordinary OpenAI-compatible endpoints with `/chat/completions`, so a Responses-only endpoint can still fail with the reported 503 path.

## What This Changes

The issue asks for custom OpenAI-compatible model setup to validate successfully when the endpoint supports OpenAI Responses rather than Chat Completions.

## Best Possible Solution

Add first-class custom-provider onboarding support for OpenAI Responses endpoints: interactive selection or detection, non-interactive flag parsing, verification against `/responses`, config persistence as `models.providers.<id>.api: "openai-responses"`, and aligned docs/tests while preserving the existing Chat Completions default.
