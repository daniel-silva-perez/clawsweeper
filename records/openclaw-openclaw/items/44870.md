## Summary

Keep open. Current main still lacks first-class custom-provider onboarding for ordinary OpenAI Responses endpoints: custom OpenAI-compatible setup verifies and persists Chat Completions, while `openai-responses` is supported only through manual config or Azure-specific handling.

## What This Changes

The issue asks for custom OpenAI-compatible provider verification/setup to work for the reporter's Responses-style backend instead of failing with HTTP 503 during the Chat Completions probe.

## Best Possible Solution

Custom provider onboarding should expose Chat Completions and Responses as distinct OpenAI-compatible API choices, verify against the matching endpoint, persist the matching `ModelApi`, and document the choice while preserving the current Chat Completions default.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: choose custom OpenAI-compatible onboarding for a Responses-only `/v1` backend and OpenClaw builds a `/chat/completions` probe, then persists `openai-completions` with no custom Responses option to select or parse.

## Solution Assessment

Yes. The narrowest maintainable fix is to extend onboarding/config parsing, verification, tests, and docs for `openai-responses`; the runtime and config schema already support that adapter, so a broader provider runtime rewrite is unnecessary.
