## Summary

Keep this PR open, but do not merge it as-is. The initialization loader direction matches the linked TUI hatch bug, but the latest PR head still unconditionally overwrites reconnect recovery state with `initializing`, which can regress active streaming status after reconnect.

## What This Changes

The PR adds an `initializing` TUI activity status, treats it as a busy status in the TUI renderer, sets it during post-connect `refreshAgents()` and `loadHistory()`, and adds helper coverage for busy-status classification.

## Best Possible Solution

Keep the initialization loader behavior, but make the status transition respect active reconnect state: only show `initializing` for cold-start or no-active-run initialization, never clear an active busy state to `idle`, and add a focused regression test for reconnecting during a tracked streamed run.

## Reproduction Assessment

Yes. The original issue reproduces through `openclaw onboard` -> Terminal hatch -> local TUI connect, where current main can show `local ready | idle` during `refreshAgents()` and `loadHistory()`; the PR regression follows a reconnect with an active tracked run through `reconnectStreamingWatchdog()` to `streaming`, then the new unconditional assignment to `initializing`.

## Solution Assessment

No, not as currently patched. The new busy helper and initialization status are a good narrow fix for the hatch gap, but the `onConnected` status transition must be guarded so it does not clobber active reconnect streaming state.
