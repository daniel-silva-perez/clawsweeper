## Summary

Keep this PR open for maintainer review. Current main still shows the TUI as idle during post-connect hatch initialization, while the PR is a narrow status-rendering fix that reuses the existing busy loader and now preserves reconnect streaming state; I found no blocking correctness or security issue in the diff.

## What This Changes

This PR adds an exported `initializing` TUI activity status and busy-status helper, applies it during post-connect `refreshAgents()`/`loadHistory()` initialization with a guard for existing busy states, and adds focused TUI helper tests.

## Best Possible Solution

Land the guarded initialization-loader change after normal maintainer review and validation, keeping the fix in the TUI status path while preserving reconnect streaming invariants.

## Reproduction Assessment

Yes. Current main can be traced through `openclaw onboard` -> Terminal hatch -> local TUI connect: `onConnected` runs `refreshAgents()` and `loadHistory()` while `activityStatus` remains `idle`, and the busy renderer has no initialization state.

## Solution Assessment

Yes. The proposed approach is the narrowest maintainable fix I found: reuse the existing busy renderer, add a named initialization status, and skip the transition when another busy state such as reconnect `streaming` is already active.
