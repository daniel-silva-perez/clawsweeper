## Summary

Keep open. Current main has several runtime-deps mitigations, but the lock-removal path still trusts an alive PID when start-time evidence cannot be read, and the tests preserve that behavior. Gateway startup still performs plugin bootstrap before HTTP binding, so a runtime-deps stall can still keep `/healthz` unavailable; open PR #74950 remains the targeted follow-up for the Docker legacy-lock case.

## What This Changes

Docker/Unraid gateway startup should become healthy after 2026.4.24+ upgrades, and `openclaw doctor --fix` should recover or fail cleanly instead of hanging.

## Best Possible Solution

Keep this issue open until #74950 or an equivalent stale Docker runtime-deps lock fix lands, then validate a Docker/Unraid or focused Docker stale-lock scenario where the gateway binds `/healthz` and `openclaw doctor --fix` exits cleanly. Any remaining plugin-loader hot-loop behavior should stay tracked under the narrower related issue #73532.

## Reproduction Assessment

Yes for a focused reproduction of the remaining lock edge: current source returns `false` for an alive PID when start-time evidence cannot be read, and the test suite asserts that behavior. The full Docker/Unraid container scenario was not live-run here.

## Solution Assessment

No, current main is not yet the complete solution. The recent runtime-deps materialization fast path helps the hot-loop side, but the Docker stale-lock fallback still needs #74950 or an equivalent narrow fix plus Docker/Unraid validation.
