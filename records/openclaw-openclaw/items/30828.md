## Summary

Current main still does not implement the requested generic model rate limiter. OpenClaw has adjacent reactive 429/Retry-After handling, rate-limit classification, auth-profile cooldown/rotation, model fallback, and provider quota/status display, but there is no documented config/schema or shared request-path scheduler for per-model tokens-per-interval or requests-per-interval budgets. The OpenAI `x-ratelimit-*` header suggestion in the discussion also remains observation-only, not enforced scheduling behavior.

## Best Possible Solution

Keep this issue open for a core model request-path design: add documented per-provider/per-model, and likely per-auth-profile, request and token budget schema with interval windows; enforce it before shared model calls; reconcile actual usage after responses/streams; optionally ingest provider headers such as OpenAI `x-ratelimit-*`; and add focused tests/docs covering scheduler, fallback, retry, streaming, and auth-profile interactions.

## Risks / Open Questions

- A naive transport-only implementation could conflict with streaming reconciliation, auth-profile cooldowns, model fallback, retries, and usage accounting unless the scheduler owns both preflight estimates and post-response correction.
