## Summary

Keep this PR open. Current main still does not implement the PR’s intended `openclaw system event --agent-id/--session-key` targeting, and the PR remains a plausible focused implementation candidate. The existing review discussion is useful: the session-key-only path needs a small but real routing fix before merge so enqueue and heartbeat target the same agent/session.

## Best Possible Solution

Refresh this PR or land an equivalent focused implementation: add the CLI flags and docs, type and normalize `agentId` and `sessionKey` consistently, thread both through the wake RPC and cron wake service, and use one shared target resolver so system-event enqueue and heartbeat wake agree when only `sessionKey` is supplied. Add regression coverage for `--agent-id`, `--session-key` alone, both together, blank/invalid values, and the gateway/cron target-resolution path.

## Risks / Open Questions

- Merging the PR as shown could keep the reviewed session-key-only bug: event enqueue may resolve to the default agent while heartbeat wake resolves to the agent derived from the session key.
- The PR schema uses plain `Type.String()` for `sessionKey`, so direct RPC callers could send an empty string that validates but has no effect after later falsy checks.
- The protocol/docs contract needs alignment because current main allows opaque extra wake properties but does not document or validate these target fields as first-class wake parameters.
