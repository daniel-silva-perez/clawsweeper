## Summary

Keep this PR open. Current main still lacks the requested `openclaw system event` targeting flags and still drops target fields through the wake handler, but the proposed patch has a real session-key-only routing bug that should be fixed before merge.

## What This Changes

The PR adds `--agent-id` and `--session-key` to `openclaw system event` and threads those optional target fields through wake validation, gateway cron handling, CronService wake, timer enqueue/heartbeat wake, and CLI tests.

## Best Possible Solution

Refresh this PR or land an equivalent focused implementation that adds the CLI flags, updates the public CLI docs, validates `agentId` and `sessionKey` consistently, and routes enqueue plus heartbeat wake through one shared target resolver so `--session-key` alone targets the same agent/session end to end.

## Reproduction Assessment

Yes. The session-key-only bug can be reproduced by code inspection: the PR forwards only `sessionKey` into `enqueueSystemEvent`, while current `buildGatewayCronService` derives enqueue agent only from `opts.agentId` and heartbeat wake can derive from `sessionKey`.

## Solution Assessment

No. The requested direction is good, but the proposed patch is not the best merge shape until target resolution is centralized and the schema/docs/tests cover `--session-key` alone, both flags together, and blank target values.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Resolve the session key target before enqueueing:** `src/cron/service/timer.ts:1244-1246`
  - body: The new `--session-key`-only flow forwards `sessionKey` to `enqueueSystemEvent` without deriving `agentId`. In `buildGatewayCronService`, enqueue still resolves the agent from `opts?.agentId`, so `agent:coding:...` can queue the event under the default agent while heartbeat wake targets `coding`. Use one shared target resolver for both enqueue and wake.
  - confidence: 0.91
- **[P3] Reject blank session keys in the wake schema:** `src/gateway/protocol/schema/agent.ts:138`
  - body: `sessionKey` becomes a typed wake parameter but accepts `""`; direct RPC callers can send a value that validates and is later dropped by the falsy guards. Use the same non-empty validation as `agentId` while keeping CLI whitespace normalization.
  - confidence: 0.79
