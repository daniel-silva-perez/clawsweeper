## Summary

Keep open. Current main still lacks the Windows suffixless sqlite-vec retry, while the PR is a narrow fix for a real memory vector-loading failure and has no blocking review or security findings from this read-only pass.

## What This Changes

The PR adds a Windows-only sqlite-vec fallback in both memory host SDK loaders that retries bundled `.dll` loads without the suffix, returns the effective loaded path, and preserves both errors on dual failure.

## Best Possible Solution

Land this PR or an equivalent narrow fix after Windows runtime validation, keeping the two memory-host SDK copies synchronized and preserving the returned effective extension path for later reloads.

## Reproduction Assessment

Yes. The current-main code path is clear: bundled sqlite-vec loading calls `sqliteVec.load(db)` once with the package-resolved `vec0.dll` path and then degrades to unavailable on failure; the linked Windows reports provide the user-facing FTS-only symptom.

## Solution Assessment

Yes. The PR's suffixless retry is narrow, aligns with SQLite's documented extension-loading convention, and avoids changing user-supplied override paths; the safer next step is validation, not a different design.
