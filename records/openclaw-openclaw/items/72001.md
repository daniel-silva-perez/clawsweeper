## Summary

Keep open. Current main still has the reported Gmail wizard/session-policy gap, but this PR is not merge-ready because the prefix-only change makes startup validation pass while mapped Gmail delivery still fails without `hooks.allowRequestSessionKey=true`, and it can invalidate supported configs where a static Gmail mapping shadows the preset.

## What This Changes

The PR adds a helper for merging `hooks.allowedSessionKeyPrefixes`, has Gmail setup write `"hook:"` alongside the Gmail preset, adds Gmail/gateway tests, and appends a changelog fix entry.

## Best Possible Solution

Update the wizard so it writes a complete, internally consistent Gmail hook session policy for the effective mapping it installs: templated Gmail routing should set `hooks.allowRequestSessionKey=true` plus a bounded Gmail/hook prefix allowlist, while configs that shadow the preset with static mappings or use non-`hook:` default session keys should remain loadable. Add tests for startup validation, mapped `/hooks/gmail` dispatch, and the static-shadow/default-session compatibility case.

## Reproduction Assessment

Yes. The source path on current main is enough to reproduce the original startup failure: the wizard writes the Gmail preset, the preset has a templated session key, and `resolveHooksConfig` requires prefixes. The PR regressions are also reproducible by code inspection against the dispatch gate and existing static-shadow/default-session tests; tests were not run because this review had to stay read-only.

## Solution Assessment

No. Prefix seeding is necessary for the startup validator failure, but it is not sufficient for Gmail delivery and is too broad for existing configs where the templated Gmail preset is not the effective mapping. A safer fix should update the full session policy conditionally or choose a static mapping.
