## Summary

Keep this PR open. Current main still has the Gmail wizard bug the PR targets, but the PR's prefix-only patch is incomplete against current Gateway hook policy: templated Gmail mapping session keys also require `hooks.allowRequestSessionKey=true`, and adding only `hook:` can invalidate existing non-`hook:` default-session configurations. The diff is narrow and does not add supply-chain or CI execution risk, but it does touch security-sensitive session routing semantics and needs maintainer review before merge.

## Best Possible Solution

Keep this PR open and revise it before merge. The final patch should make `openclaw webhooks gmail setup` emit a config that passes both `resolveHooksConfig` and actual `/hooks/gmail` mapped dispatch, preserves existing prefixes and valid `defaultSessionKey` setups, and makes an explicit policy choice: either opt into template-derived Gmail session keys with `allowRequestSessionKey=true` plus constrained prefixes, or install a static Gmail mapping/session policy when request-derived session keys remain disabled. Add focused regression coverage for fresh wizard output, existing-prefix preservation, non-`hook:` default-session configs, static Gmail shadow mappings, and mapped Gmail dispatch behavior.
