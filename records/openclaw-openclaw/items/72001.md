## Summary

Keep this PR open. Current main still writes the Gmail preset from `openclaw webhooks gmail setup` without writing `hooks.allowedSessionKeyPrefixes`, while the built-in Gmail preset still has a templated `sessionKey` that the Gateway validator rejects without prefixes. The PR is not obsolete, but review should address the existing Codex review edge case and the runtime/docs requirement that template-driven mapping session keys also need `hooks.allowRequestSessionKey=true` or a static mapping override.

## Best Possible Solution

Keep this PR open and revise it before merge. The final patch should make the Gmail wizard emit a config that passes both `resolveHooksConfig` and actual mapped Gmail dispatch: preserve existing prefixes, do not invalidate an existing `defaultSessionKey`, and either opt into `allowRequestSessionKey` with a narrow Gmail prefix or write a static Gmail mapping/session policy when keeping request/template session keys disabled. Add regression coverage for the plain wizard output, existing prefix preservation, non-hook default-session configs, and mapped Gmail dispatch behavior.

## Risks / Open Questions

- The proposed PR only writes `hooks.allowedSessionKeyPrefixes`; current runtime and docs also require `hooks.allowRequestSessionKey=true` for template-rendered mapping session keys, so Gmail delivery could still fail after startup unless the wizard chooses a secure static-session alternative or opts in explicitly.
- Unconditionally adding only `hook:` can make an existing non-hook `hooks.defaultSessionKey` invalid, especially in configs where a static mapping already shadows the Gmail preset and currently loads.
- This is a security-sensitive config surface: hook token holders and Gmail payload-derived templates can influence session routing. The final fix should keep prefixes narrow and avoid broadening access beyond the Gmail namespace or preserved existing defaults.
