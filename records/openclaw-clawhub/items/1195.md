## Summary

Keep open. This is an open ClawHub issue, and current main still leaves the named owner-publisher skill digest helper able to execute repeated `.paginate()` calls inside one internal mutation invocation. The scheduling split mitigates the original inline trigger shape for small owner catalogs, but it does not fully address multi-page owner/publisher result sets. This is a narrow fix-PR candidate, preferably consolidated with #1182 and sibling reports.

## Best Possible Solution

Keep this issue open unless maintainers consolidate it under #1182. The code fix should make owner-publisher package and skill digest sync cursor-batched: each internal mutation should accept a cursor, perform at most one `.paginate()` call, process that page, and schedule the next invocation with `continueCursor` when `isDone` is false. Add focused tests for the multi-page skill and package paths.

## Clownfish Work Prompt

Fix the publish/import failure caused by owner-publisher digest sync running multiple paginated queries in one Convex mutation. In `convex/functions.ts`, change both `syncPackageSearchDigestsForOwnerPublisherIdInternal` and `syncSkillSearchDigestsForOwnerPublisherIdInternal` so each invocation accepts an optional cursor and performs at most one `.paginate()` call for its table, processes that page, and schedules itself with `continueCursor` when `!isDone`. Update `scheduleOwnerPublisherDigestSync` to start both jobs without a cursor. Preserve existing digest contents, owner resolution, and missing-table behavior. Add focused Vitest coverage in `convex/functions.test.ts` for an `isDone: false` first page for both the skill and package owner-publisher paths, asserting a follow-up scheduler job is enqueued and no second paginate happens in the same invocation. Do not change public publish API/CLI semantics, schema, or unrelated search/list behavior. No changelog entry is expected unless maintainers prefer one for backend publish reliability.
