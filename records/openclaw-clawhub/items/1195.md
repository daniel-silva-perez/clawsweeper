## Summary

Keep this issue open. Repository policy for openclaw/clawhub says this cleanup pass should not auto-close issues, and current main still leaves the named skill digest sync helper as a same-invocation pagination loop. The scheduling split on publisher changes is a useful mitigation, but it does not fully implement the cursor-batched continuation needed for owners with more than one page of skills.

## Best Possible Solution

Keep this issue open, or have maintainers explicitly consolidate it under #1182. The robust fix should make owner-publisher package and skill digest sync cursor-batched: each internal mutation should accept a cursor, call .paginate() at most once, process that page, and schedule the next invocation with continueCursor. Add regression coverage for an isDone: false skill page.
