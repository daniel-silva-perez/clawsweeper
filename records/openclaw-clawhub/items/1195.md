## Summary

Keep this issue open. ClawHub cleanup policy does not auto-close open issues in this pass, and current main still leaves the reported owner-publisher digest sync path able to call `.paginate()` more than once inside one Convex internal mutation when an owner has more than one page of skills or packages. The scheduler split reduces the original inline trigger blast radius, but it does not fully fix the Convex single-paginated-query limit.

## Best Possible Solution

Keep this issue open unless maintainers consolidate it under #1182. The code fix should make owner-publisher package and skill digest sync cursor-batched: each internal mutation should accept an optional cursor, perform at most one `.paginate()` call, process that page, and schedule itself with `continueCursor` when `isDone` is false. Add focused tests for both the skill and package multi-page paths.

## Clownfish Work Prompt

Fix the Convex multiple-paginated-query failure reported by the owner-publisher digest sync issues. In `convex/functions.ts`, update `syncPackageSearchDigestsForOwnerPublisherIdInternal` and `syncSkillSearchDigestsForOwnerPublisherIdInternal` so each internal mutation invocation accepts an optional cursor/batch size, performs at most one `.paginate()` call, processes that page, and schedules itself with `continueCursor` when `isDone` is false. Keep `scheduleOwnerPublisherDigestSync` as the initial trigger scheduler. Do not replace pagination with unbounded `.collect()` or broaden the publish flow. Add focused tests in `convex/functions.test.ts` for both package and skill owner-publisher helpers: simulate an `isDone: false` first page, assert only one paginate call in that invocation, assert a continuation job is scheduled with the cursor, and assert the final page does not schedule another job. Inspect sibling digest pagination loops for the same risk, but keep the PR focused unless a directly shared helper makes the fix cleaner. Mention related issues #1171, #1182, #1195, #1197, and sibling reports in the PR summary; no changelog entry is needed unless the repository has an active bugfix changelog convention for ClawHub backend fixes.
