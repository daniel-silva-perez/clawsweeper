## Summary

Keep open. Current main still contains the Convex failure shape: the owner-publisher skill and package digest sync helpers can call `.paginate()` repeatedly inside one internal mutation, and the internal mutations do not accept continuation cursors. The related cluster, especially #1182, remains open, and ClawHub cleanup policy keeps issues open unless a maintainer consolidates them or a fix lands.

## What This Changes

The issue reports that `clawhub publish` fails because `syncSkillSearchDigestsForOwnerPublisherId` can trigger Convex's multiple-paginated-query limit during owner-publisher digest sync.

## Best Possible Solution

Keep this issue open unless maintainers explicitly consolidate it under #1182. The concrete fix should make owner-publisher package and skill digest sync cursor-batched: each internal mutation should accept an optional cursor, perform at most one `.paginate()` call, process that page, and schedule itself with `continueCursor` when `isDone` is false. Add focused tests for both skill and package multi-page paths.

## Repair Work Prompt

Fix the owner-publisher digest sync pagination bug reported by this issue and the related cluster (#1182, #1171, #1197). In `convex/functions.ts`, change both owner-publisher digest sync paths so each internal mutation invocation performs at most one `.paginate()` call: accept an optional `cursor`, process one page for packages or skills, and when `isDone` is false schedule the same internal mutation with `ownerPublisherId` and `continueCursor`. Preserve the existing split between package and skill background jobs, keep indexed `by_owner_publisher` queries, and do not introduce full table scans. Add focused tests in `convex/functions.test.ts` that simulate multi-page package and skill owner-publisher results, assert only one page is fetched per invocation, and assert continuation scheduling with the returned cursor. Keep the PR scoped to this Convex digest/publish failure; do not refactor unrelated catalog or publish behavior.
