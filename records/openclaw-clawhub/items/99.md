## Summary

Keep open. This is a ClawHub issue, and the repository cleanup policy says issues should remain open unless maintainers decide otherwise. Current main does not prove the reported browser/token login path is fixed: the CLI still verifies `/api/v1/whoami` before persisting the token, auth failures still surface as `Unauthorized`, related Unauthorized reports remain open, and the current production redirect behavior differs from the older workaround theory enough that maintainer reproduction is needed.

## What This Changes

The issue reports that `clawhub login` browser auth and `clawhub login --token <fresh-token>` can complete on the web side but fail in the CLI with `Unauthorized` instead of storing a valid token.

## Best Possible Solution

The desired end state is a confirmed production auth path where fresh web-created and browser-created CLI tokens verify consistently against the canonical host, the CLI persists the correct registry, and troubleshooting docs/tests cover any required host or token-state behavior. Maintainers should either consolidate the related Unauthorized reports under one canonical tracker or land one focused fix after reproduction.
