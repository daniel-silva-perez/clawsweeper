## Summary

Keep open. This is a ClawHub issue, and the repository cleanup policy says issues should remain open for maintainer judgment. Current main does not prove the reported login/token verification path is fixed: the CLI still verifies `/api/v1/whoami` before persisting the token, auth failures still surface as plain `Unauthorized`, the related redirect-focused PR was closed unmerged, and live host behavior no longer matches the original workaround closely enough to close this as resolved.

## What This Changes

The issue reports that `clawhub login` completes browser authentication but the CLI then fails token verification with `Unauthorized` instead of storing the token.

## Best Possible Solution

The desired end state is a confirmed production auth path where fresh browser-created and settings-created CLI tokens verify against the canonical host, the CLI persists the correct registry, and troubleshooting docs/tests cover any host or token-state requirements. Maintainers should either consolidate the related Unauthorized reports under one canonical tracker or land one focused fix after reproduction.
