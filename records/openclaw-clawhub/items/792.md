## Summary

Keep this issue open. Current main still does not inject a post-install/update system event into an active OpenClaw session after `clawhub install` or `clawhub update`; the related skill-page prompt flow is explicit, copy-only UX rather than automatic handoff or execution. For openclaw/clawhub, the supplied policy also says issues should remain open unless maintainer-handled, with implemented-on-main auto-close reserved for PRs.

## Best Possible Solution

Keep this open for maintainer product/API triage. The best path is to define an explicit, user-consented ClawHub-to-OpenClaw session notification contract, decide whether automatic post-install/update signaling is allowed, then implement the accepted behavior with CLI/session tests, docs, and security-boundary coverage. Maintainers may also decide the current copyable prompt UX is the intended boundary, but that should be an explicit product decision.
