## Summary

Keep this issue open. Current main still implements `clawhub install` and `clawhub update` as filesystem/download/update flows with success output, plus an explicit copyable OpenClaw prompt surface on skill pages; it does not inject a post-install/update system event into an active OpenClaw session. Repository policy for openclaw/clawhub also says issues should remain open unless maintainer-handled, with implemented-on-main auto-close reserved for PRs.

## Best Possible Solution

Keep this open for maintainer product/API triage. The best path is to define an explicit, user-consented ClawHub-to-OpenClaw session notification contract, decide whether automatic post-install/update signaling is allowed, then implement the accepted behavior with CLI/session tests, docs, and security-boundary coverage. Maintainers may also decide the current copyable prompt UX is the intended boundary, but that should be an explicit product decision.
