## Summary

Keep this issue open. The repository policy for openclaw/clawhub keeps issues open, and current main still does not implement post-install/update active-session event injection. The CLI install/update paths resolve, download/extract, write origin/lock metadata, and report success; the web UX provides an explicit copyable OpenClaw prompt, and the approved design explicitly excludes direct handoff or automatic execution into OpenClaw.

## Best Possible Solution

Keep this issue open for maintainer product/API triage. The best path is to define an explicit, user-consented ClawHub-to-OpenClaw session notification contract, then implement post-success install/update signaling with tests and docs for install, update, non-interactive CLI use, and security boundaries. Maintainers may also decide the current explicit Copy Prompt UX is the intended boundary, but that should be a deliberate product decision.

## Risks / Open Questions

- Automatically prompting an active agent to read and act on newly installed third-party SKILL.md content crosses a consent and security boundary around setup commands, config edits, plugin activation, and service restarts.
- The current Copy Prompt UX partially helps guided setup, but treating it as equivalent to post-install/update event injection would close over a real product/API decision.
- A CLI-only hint would not satisfy the requested active-session injection unless OpenClaw exposes and accepts a safe host/session notification contract.
