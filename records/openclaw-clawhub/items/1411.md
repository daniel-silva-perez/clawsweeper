## Summary

Keep open: current main does not implement owner-editable skill summaries, and this contributor PR is still an open, unmerged implementation with blocking functional and moderation defects. It is also paired with the open same-author request #1345, so it should stay open for maintainer review rather than cleanup closure.

## What This Changes

The PR adds a public `skills.updateSummary` mutation, detail-page state wiring, inline header edit controls, and CSS so authorized users can edit a skill summary without publishing a new version.

## Best Possible Solution

Land a revised implementation that treats summary editing as a validated metadata update: authorize the same owner, publisher manager, and moderator actors exposed by the UI; define how edits interact with future publishes; and keep detail, search digest, API, audit, and moderation state consistent.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep moderation state in sync with edited summaries:** `convex/skills.ts:4814-4818`
  - body: The publish path derives moderation flags from the summary that gets persisted, but this new mutation patches `summary` and `updatedAt` only. That lets an owner change public searchable text after the latest version was scanned while `moderationFlags`, `isSuspicious`, and moderation summary fields still describe the old text, so route this through the same moderation or manual-override workflow.
  - confidence: 0.9
- **[P1] Authorize publisher-owned skill managers:** `convex/skills.ts:4808-4812`
  - body: The edit button is shown through `canManage`, which includes members of `skill.ownerPublisherId`, but `updateSummary` only allows the direct `ownerUserId` or moderators. Publisher members can open the editor and will always fail on save, so align the backend authorization with publisher ownership or narrow the UI gate.
  - confidence: 0.93
- **[P2] Move summary edit form out of the paragraph:** `src/components/SkillHeader.tsx:211-214`
  - body: When edit mode is active, this renders a `<form>` and its block children inside `<p className="section-subtitle">`. Browsers implicitly close the paragraph before the form, which can cause hydration and layout issues and leave the edit controls outside the intended subtitle styling; use a non-paragraph wrapper.
  - confidence: 0.95
- **[P2] Cap editable summary length:** `convex/skills.ts:4800-4803`
  - body: The new public mutation accepts any Convex string and writes it into the skill document and search digest. A client can save a near-document-limit summary that bloats list/detail/API responses, so add a server-side max length and a matching textarea `maxLength`.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The PR introduces a public metadata edit path that can change searchable summary text without rerunning or updating moderation state.

Concerns:

- **[medium] Edited summaries bypass moderation recomputation:** `convex/skills.ts:4814`
  - body: `updateSummary` changes public summary text but does not recompute moderation flags or `isSuspicious`, even though publish-time moderation derives flags from the persisted summary. This can leave public safety labels stale after an owner edit.
  - confidence: 0.9
- **[low] Summary writes are unbounded:** `convex/skills.ts:4800`
  - body: The public mutation accepts an unrestricted string and stores it on public list/detail/API surfaces. A server-side length cap is needed to limit response bloat and document-size pressure.
  - confidence: 0.78
