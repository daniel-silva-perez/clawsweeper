## Summary

Keep this PR open. Current main does not implement owner-editable skill summaries: the detail header still renders the summary read-only, there is no `api.skills.updateSummary` wiring, and repository docs/current publish code still model the summary as publish/frontmatter-derived metadata. The PR remains a useful implementation attempt for related #1345, but the proposed branch needs maintainer/contributor revision around authorization, input validation, valid JSX, tests, and the security/moderation semantics of changing a public summary after publish.

## Best Possible Solution

Keep this PR open for contributor or maintainer revision. A mergeable version should add a validated owner-facing summary update mutation, align authorization with the owner/publisher-member/moderator model, trim and cap the summary, preserve trigger-driven `skillSearchDigest` sync, define how post-publish summary edits affect moderation/security state, render valid JSX, and add focused Convex/frontend tests. It should coordinate with related #1345 rather than closing that user request prematurely.

## Risks / Open Questions

- Closing this PR would discard the only open implementation path tied to related #1345 while current main still has no owner-facing summary edit workflow.
- The proposed branch exposes a new public mutation for user-controlled public summary text without an explicit length cap or clear normalization policy.
- The proposed UI uses `canManage`, which includes publisher members, while the proposed backend mutation only allows `ownerUserId` or moderators, so publisher-owned skills would show an editor that fails on save.
- Changing a summary after publish affects security/moderation signals because summary text feeds static scan/moderation flag inputs; the PR needs a clear policy for re-evaluation, rescans, manual overrides, and suspicious-state updates.
- The proposed header edit form is rendered inside a paragraph in the provided diff, which is invalid HTML and can cause layout or hydration problems.
