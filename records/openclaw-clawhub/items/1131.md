## Summary

Keep open. The repository cleanup policy keeps ClawHub issues open, the report is paired with open implementation PRs, and current main still lacks the requested declared security-tool category/context mechanism; the existing mitigation only suppresses a narrow allowlisted static code when both external scanners are clean.

## What This Changes

The issue requests a declared security-tool category system so static moderation findings and LLM evaluation can distinguish coherent security tooling behavior from suspicious behavior.

## Best Possible Solution

Resolve this through maintainer review of PR #1132 and PR #1133, or a replacement design that parses declared security-tool metadata, contextualizes only allowlisted suspicious reason codes, preserves malicious and non-contextual safety floors, passes category context and false-category guidance to the LLM evaluator, documents the behavior, and covers moderation-hold recovery with tests.

## Security Review

Status: needs_attention

Summary: This issue is security-sensitive because the proposed category system would intentionally change moderation outcomes.

Concerns:

- **[medium] Bound category-based suppression carefully:**
  - body: A self-declared security category could become a moderation bypass if it suppresses broad suspicious findings. Any implementation should contextualize only allowlisted suspicious codes, keep malicious codes and incoherent false-category evidence escalated, and include bypass-focused tests.
