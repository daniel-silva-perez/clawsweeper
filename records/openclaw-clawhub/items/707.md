## Summary

This issue should stay open. Current main still does not document, normalize, scan, expose, or render skill lifecycle `hooks`; the only first-class `hooks` field I found is for package/plugin capability summaries, not skill frontmatter. The request is also security-sensitive and needs maintainer product/security policy before an autonomous fix PR.

## Best Possible Solution

Keep this issue open for maintainer security/product review. The best path is to define an accepted skill hook metadata contract, document it in `docs/skill-format.md`, extend schema/parser/API/UI surfaces to expose hook declarations, add static and LLM scanner coverage for hook commands/events/target files, and decide install/inspect warnings or restrictions for high-privilege hook types before shipping.

## Risks / Open Questions

- The requested change touches a security-sensitive execution surface: lifecycle hooks can run commands automatically and may block or influence tool use, so hook type/event policy needs maintainer judgment before implementation.
- There is a compatibility decision to make between Claude Code’s top-level `hooks` frontmatter and ClawHub’s current normalized `metadata.openclaw`/Clawdis model.
- Existing raw frontmatter may be stored on versions, but current public APIs and UI intentionally expose only selected parsed metadata, so surfacing hooks requires a deliberate API/UI contract rather than only a docs change.
- Issue #129 remains open for broader malicious-skill persistence concerns; this issue is narrower and should not be closed as a duplicate without maintainer confirmation.
