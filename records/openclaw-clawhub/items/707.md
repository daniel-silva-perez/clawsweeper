## Summary

Keep this issue open. Current main still does not document or type a skill `hooks` field, does not parse/promote hook metadata, does not perform hook-specific validation or moderation, and does not expose hooks in the public API or skill detail UI. The related #129 remains useful security context, but it is broader and open rather than a hook-specific superseding tracker.

## Best Possible Solution

Keep this issue open. The implementation should add a typed hooks declaration to the skill metadata contract and docs, parse hooks from the intended SKILL.md frontmatter locations, validate lifecycle event names, hook types, matchers, timeout values, and referenced command files, add hook-aware moderation reason codes and tests, expose hook declarations through the public API, and surface them prominently on skill detail/listing UI before install. Treat #129 as related security context, not as the hook-specific tracker.

## Risks / Open Questions

- Hooks are security-sensitive because they can run automatic code on lifecycle events and may block tool/session behavior; a docs-only change would leave the main risk unresolved.
- Current main may preserve raw SKILL.md frontmatter containing hooks, but it does not create structured hook metadata for registry review, API consumers, or UI disclosure.
- Maintainers still need product/security policy decisions for event names, blocking events such as PreToolUse, hook types such as command/agent/http, and how strict command-target validation should be.
