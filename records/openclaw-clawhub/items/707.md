## Summary

Keep this issue open. The repository policy keeps ClawHub issues out of auto-close scope, and current main still does not implement the requested hooks field contract, parser promotion, hook-aware moderation, API exposure, or skill detail UI disclosure. The linked #129 is relevant security background, but it is broader and remains open rather than superseding this hook-specific work.

## Best Possible Solution

Keep this issue open. The implementation path should add a typed hooks declaration to the skill metadata contract and public skill-format docs; parse hooks from both metadata.openclaw/clawdis and any intended top-level syntax; validate lifecycle event names, hook types, command targets, and missing files; add hook-aware moderation reason codes for privileged or blocking events; expose hooks through the public API; surface them clearly on skill detail pages; and add focused parser, moderation, API, and UI tests. Treat #129 as related security context, not as the hook-specific tracker.

## Risks / Open Questions

- Lifecycle hooks are security-sensitive because they can run automatic code and may block tool/session behavior, so a docs-only change would leave the main risk unresolved.
- If skills already publish hooks in SKILL.md frontmatter, current main will preserve the raw file but will not expose structured hook metadata for registry review or user disclosure.
- Hook event and type policy needs maintainer judgment, especially for blocking PreToolUse hooks and powerful command/agent hook types.
