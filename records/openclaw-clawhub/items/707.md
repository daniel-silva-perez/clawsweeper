## Summary

Keep this issue open. Under the provided openclaw/clawhub review policy, issues should not be auto-closed, and current main still does not document, type, parse, validate, scan, expose, or render skill `hooks` as a first-class skill-format field. The broader #129 security report is relevant context, but it does not supersede this hook-specific RFC.

## Best Possible Solution

Keep the issue open. The likely implementation path is to add a typed `hooks` declaration to the skill metadata contract and skill-format docs, parse it from `SKILL.md` frontmatter, add static checks for lifecycle event names, hook types, command target existence, privileged/blocking events, and missing files, then expose the declarations through the public API and skill detail UI with focused tests. Keep #129 as related broader security context rather than the canonical tracker for this hook-specific RFC.

## Risks / Open Questions

- Closing this now would lose a concrete, hook-specific security and transparency task that remains unimplemented on current main.
- Future implementation should distinguish package/plugin capability hooks from Claude Code skill frontmatter hooks so the two metadata surfaces do not become conflated.
- Because hooks can represent automatic code execution, the eventual implementation needs explicit security review for event names, hook types, command target existence, blocking behavior, and install-time disclosure.
