## Summary

Keep this issue open. Current main still formats `clawhub sync` update lines with a semver-bumped publish target, while `inspect` prints the actual registry latest version. The ClawHub policy also only permits auto-closing pull requests, not issues.

## Best Possible Solution

Keep this issue open and fix the CLI reporting path. The narrow fix should make `clawhub sync` stop presenting the semver-bumped publish target as an available update version, either by changing the update line wording or by separating registry latest from proposed publish version. Add a focused sync test for `match: null` plus `latestVersion` to lock the display semantics.

## Clownfish Work Prompt

Fix the misleading `clawhub sync` version display for this issue. Current behavior: when `/api/v1/resolve` returns `match: null` and `latestVersion: { version: "1.0.0" }`, sync classifies the skill as `update` and formats `UPDATE 1.0.0 -> 1.0.1`, which users read as a registry update even though `1.0.1` is only the proposed publish version for changed local content. Keep the publish workflow intact, but change the sync report/prompt wording or data model so it clearly separates actual registry latest from proposed publish version. Do not touch unrelated dashboard deployment-drift code. Add or update focused tests in the CLI sync test suite for the unmatched-fingerprint/latest-version case, including the displayed line. Run the package CLI test for sync and the CLI typecheck/build validation. No changelog is required unless the repo has a current unreleased changes file for CLI fixes.
