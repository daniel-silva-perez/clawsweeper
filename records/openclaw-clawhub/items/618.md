## Summary

Keep open. Current main still has the reported CLI presentation bug: unmatched local skill content is classified as an update and the sync report displays the semver-bumped publish target as if it were an available registry version. The related dashboard drift issue is a separate UI path and does not supersede this CLI report.

## What This Changes

The issue asks for `clawhub sync` to show actual registry latest versions instead of higher semver-bumped values for installed skills that are already at the registry latest.

## Best Possible Solution

Update the CLI sync report and interactive upload-selection label so unmatched local content is described as local changes relative to the registry latest, with any bumped publish version clearly labeled as a proposed upload version. Add regression coverage for the `latestVersion` present plus `match: null` case and leave the separate dashboard drift warning to #1180.

## Repair Work Prompt

Fix the CLI sync reporting bug where a local skill with `latestVersion` present and `match: null` is displayed as `UPDATE <latest> -> <semver.inc(latest, bump)>`, making the proposed publish version look like the registry latest. Update the sync report and interactive multiselect label to describe this as local changes/no matching published fingerprint relative to the registry latest; if the bumped version is shown, label it clearly as the proposed upload/publish version. Keep upload behavior intact: `resolvePublishMeta` should still bump the version for actual uploads, and do not change install/update behavior or the dashboard deployment drift warning. Add regression coverage for `/api/v1/resolve` returning `{ match: null, latestVersion: { version: "1.0.0" } }` in both printed sync output and the selection label. Related #1180 is separate and should not be folded into this fix. No changelog entry is expected for this narrow CLI output correction unless maintainers request one.
