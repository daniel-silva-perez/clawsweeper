## Summary

Keep open. Current main still formats unmatched local skill content as an update from the registry latest version to a semver-bumped publish target, so `clawhub sync` can display versions that do not exist while `clawhub inspect` shows the actual latest registry version. The related dashboard drift warning is a separate path and does not supersede this CLI reporting bug.

## What This Changes

The issue asks for `clawhub sync` to report actual latest registry versions for installed skills instead of higher semver-bumped versions that do not exist.

## Best Possible Solution

Land a focused CLI reporting fix: keep `clawhub sync` as an upload-local-skills workflow, but when resolve returns an actual `latestVersion` with `match: null`, make the report and selection label distinguish the registry latest version from the proposed publish version. Add tests for the unmatched-latest case and leave the separate dashboard drift warning to #1180.

## Repair Work Prompt

Fix the `clawhub sync` version display bug from this issue. When a local skill resolves to `{ latestVersion: { version }, match: null }`, the sync planning output and interactive selection label must not present the semver-bumped publish target as if it were an existing registry latest version. Preserve the upload workflow and publish semantics: `resolvePublishMeta` may still publish `semver.inc(latest, bump)` for changed local content. Adjust the reporting helper/output wording so users can distinguish actual registry latest from proposed publish version. Add regression tests for dry-run/planning output and helper/selection label behavior for the unmatched-latest case. Do not change the registry resolve API, fingerprint matching, install/update commands, telemetry, package publishing, or the unrelated dashboard drift warning. No changelog file is required unless the repo already has a current CLI changelog pattern for this class of fix; include the behavior change and commands run in the PR summary.
