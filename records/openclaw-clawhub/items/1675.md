## Summary

Keep open. This is a ClawHub issue, and the provided repository policy only allows auto-closing ClawHub pull requests that are certainly implemented on main. Substantively, current main still lacks the requested existing-skill transfer-to-org path: transfer records and handlers are user-recipient based, org publishers are excluded from the recipient resolver, and the CLI/API contract still uses `toUserHandle`. Web publish has partial org owner selection, but that does not re-home an already personal-owned skill while preserving stats and canonical URLs.

## Best Possible Solution

Keep this issue open and implement publisher-target skill ownership transfer in core. The likely path is to migrate `skillOwnershipTransfers` from user recipients to source/target publisher IDs, resolve org publisher handles in the HTTP API and CLI, require owner/admin permission on the target publisher to accept, update existing skill and alias ownership without losing stats or URLs, and add regression coverage for user-to-org, org-to-user, and org-to-org transfers. If maintainers choose publish-as-org or re-home-on-next-publish instead, the public HTTP API and CLI need `ownerHandle` support plus an explicit safe re-home flow for existing personal-owned skills.
