## Summary

Keep this issue open. Repository policy for openclaw/clawhub keeps issues open, and current main still has the reported product gap: skill transfers are modeled as user-to-user, org publishers are not valid transfer recipients, and the existing publish-as-org path does not provide a no-loss re-home flow for an already personal-owned skill.

## Best Possible Solution

Keep this issue open for core implementation. The likely product/code path is to migrate skill transfers from user-recipient records to publisher-target records, resolve org handles as valid destinations, authorize acceptance by the agreed org roles, update skills and slug aliases without losing stats or redirects, keep compatibility for existing user-to-user transfers where needed, and add coverage for user-to-org, org-to-user, and org-to-org transfers. The existing publish-as-org flow can remain, but it does not replace a safe re-home flow for already published personal-owned skills.
