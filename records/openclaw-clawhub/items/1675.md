## Summary

Keep open. This is a ClawHub issue, and the repository policy for this review keeps issues open unless maintainers handle them manually. Substantively, current main still does not provide the reported existing-skill transfer-to-org path: transfers remain user-recipient based, org publisher handles are excluded as recipients, and the CLI/API transfer contract still uses `toUserHandle`. Current main has partial publisher/org support for ownership and web publish selection, but not the consolidation path needed to move an existing personal-owned skill to an org while preserving stats and canonical URLs.

## Best Possible Solution

Keep this issue open and implement publisher-target skill ownership transfer in core. The likely path is to migrate `skillOwnershipTransfers` from user recipients to source/target publisher IDs, resolve org publisher handles in the API and CLI, require owner/admin permission on the target publisher to accept, update existing skill ownership without losing stats or URLs, and add coverage for user-to-org, org-to-user, and org-to-org transfers. If maintainers prefer publish-as-org/re-home-on-next-publish instead, the public HTTP API and CLI need `ownerHandle` support plus an explicit safe re-home path for existing personal-owned skills.
