## Summary

Keep open. This is a ClawHub issue, and the repository policy for this review says ClawHub issues should not be auto-closed. Current main also still has a plausible implementation path for the reported stale duplicate attribution: the public skill detail query can return `forkOf` metadata for a referenced skill document without checking whether that duplicate target is soft-deleted or otherwise non-public.

## Best Possible Solution

Keep this issue open for maintainer handling. The likely fix is to suppress `forkOf` and canonical duplicate metadata in the public skill detail response when the referenced duplicate target is soft-deleted, removed, hidden, owner-hidden, or otherwise non-public, then add a regression test for that case. Maintainers can also use the existing moderator duplicate-clear path or a targeted data cleanup to remove the stale `apexsearch` reference from the affected production skill.
