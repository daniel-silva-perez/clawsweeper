## Summary

Keep this issue open. The repository-specific policy keeps ClawHub issues open, and current main still does not explicitly support `.ps1` uploads: shared text-file allowlists omit PowerShell extensions, web and server publish paths reject files outside that allowlist unless a text MIME type is supplied, and the CLI/package file discovery path uses the same extension gate. Related #897 remains open and describes the same user problem plus a workaround of putting the PowerShell script into `SKILL.md`.

## Best Possible Solution

Keep this issue open and implement explicit, security-aware PowerShell publish support: add intended PowerShell extensions such as `.ps1` and possibly `.psm1`/`.psd1` to the shared text-file allowlists, normalize their content type as text, update web upload and server publish tests, update CLI file discovery tests, document the supported extensions, and extend static moderation scanning for PowerShell-specific risky execution and obfuscation patterns. Consider using #897 as the linked companion report rather than closing either issue automatically under the current ClawHub issue policy.

## Risks / Open Questions

- PowerShell scripts are executable code. Adding `.ps1` support should be paired with explicit moderation/security scanning patterns and tests so PowerShell payloads receive scrutiny comparable to shell, JavaScript, and Python files.
- There are parallel schema/CLI allowlist surfaces (`packages/schema` and `packages/clawhub/src/schema`, plus package file discovery). A fix should update all publish paths and tests together to avoid web/server/CLI drift.
