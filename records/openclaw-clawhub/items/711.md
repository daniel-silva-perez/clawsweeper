## Summary

Current main does not definitely implement this PR’s intended change. The fallback search path now reads lightweight skillSearchDigest rows instead of full skill documents, but lexicalFallbackSkills still uses the active updated/created digest indexes and filters nonSuspiciousOnly in JavaScript instead of selecting the non-suspicious indexes at the database level.

## Best Possible Solution

Keep this PR open. The best path is to rebase or rewrite it against current main so lexicalFallbackSkills uses skillSearchDigest non-suspicious indexes for both updated and created scan windows when nonSuspiciousOnly is set, while preserving digest-based hydration, exact slug handling, capability/highlight filtering, and current search pagination semantics unless maintainers intentionally accept the broader client-side pagination change. Add tests for non-suspicious index selection and for the default path still allowing suspicious rows when the flag is unset.

## Risks / Open Questions

- The PR is based on an older fallback shape: current main uses skillSearchDigest and scans both updated and created windows, so the index change needs a rebase rather than a direct stale merge.
- The client-side search pagination change is broader than the PR title and was debated in review; maintainers should decide whether it belongs with this performance fix or should be split.
- Because isSuspicious is optional in the digest schema, a rebased non-suspicious index read should account for backfill/rollout semantics so older undefined values are not accidentally dropped if any remain.
