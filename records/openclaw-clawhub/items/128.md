## Summary

Current main still lacks the requested `--dry-run` mode for `clawhub update`. The update command only registers `--all`, `--version`, and `--force`; `cmdUpdate` only accepts those options and still proceeds to remove/install files and write lock/origin metadata. Existing dry-run behavior applies to `sync` and `package publish`, so the related package publish dry-run work is precedent rather than a fix for this issue.

## Best Possible Solution

Keep this issue open and implement a narrow CLI enhancement: add `--dry-run` to `clawhub update`, produce a preview for single-skill and `--all` updates that separates would-update, already-up-to-date, skipped/blocked, and local-change cases, and add tests proving dry-run performs registry/version resolution but never removes files, downloads zips, writes skill origin metadata, or writes the lockfile. Use existing `sync` and `package publish` dry-run patterns as precedent, but keep behavior scoped to the skill update command.

## Clownfish Work Prompt

Implement `--dry-run` for `clawhub update` in the ClawHub CLI. Add the Commander flag to `packages/clawhub/src/cli.ts`, extend `cmdUpdate` option typing/logic in `packages/clawhub/src/cli/commands/skills.ts`, and add focused tests in `packages/clawhub/src/cli/commands/skills.test.ts`. The dry-run path should support both a single slug and `--all`, resolve the same registry/latest-version information needed to preview work, print a useful plan that distinguishes would-update and already-up-to-date skills, and preserve existing handling for malware-blocked/suspicious/local-change cases. It must not call `rm`, `downloadZip`, `extractZipToDir`, `writeSkillOrigin`, or `writeLockfile`, and must not mutate the in-memory lock in a way that later gets persisted. Use existing `sync --dry-run` and `package publish --dry-run` as style references, but do not change those commands. Update CLI docs/README if the new flag is user-facing, and add an Unreleased changelog entry following the existing convention.
