## Summary

Current main still does not implement `clawhub update --dry-run`: the CLI registers only `--all`, `--version`, and `--force`, `cmdUpdate` has no dry-run option, and the normal update path still performs filesystem/download/origin/lockfile writes. Repository policy for ClawHub issues also says to keep issues open, so this remains a narrow valid enhancement.

## What This Changes

Add a `--dry-run` flag to `clawhub update` so users can preview single-skill or `--all` updates without mutating installed skills or lockfiles.

## Best Possible Solution

Keep this issue open and implement a focused CLI enhancement: add `--dry-run` to `clawhub update`, reuse the existing update resolution checks, print a useful preview for single-skill and `--all` flows, and guarantee dry-run mode performs no filesystem, origin metadata, download/extract, or lockfile mutation. Use the existing sync and package publish dry-run paths as precedents without changing those commands.

## Repair Work Prompt

Implement the feature requested by this issue: add a `--dry-run` flag to `clawhub update` for both `clawhub update <slug> --dry-run` and `clawhub update --all --dry-run`. Keep the scope limited to the ClawHub CLI update command. Reuse the existing update resolution and moderation checks where reasonable, but guarantee dry-run mode performs no mutations: no lock-entry normalization, no `rm`, no zip download/extraction, no origin metadata writes, and no final `writeLockfile`. Print a useful preview that distinguishes skills that would update from skills already up to date, and handle explicit `--version`, suspicious/malware-blocked skills, local-change/no-match cases, empty installs, and mixed `--all` output. Add focused unit tests that assert the dry-run path avoids filesystem/download/origin/lockfile writes. Update CLI docs/README/help examples for the new flag. Do not change package publish dry-run or sync dry-run behavior, and do not modify release workflow or package publishing metadata.
