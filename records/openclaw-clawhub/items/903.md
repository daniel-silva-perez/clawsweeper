## Summary

Keep open. Current main does not implement the PR's intended performance changes, and the proposed patch still needs revision before merge because the sharded consume path can weaken the aggregate rate-limit cap and the countPublicSkills tests remain aligned with the old live fallback behavior.

## What This Changes

The PR removes live global-stats recount paths and the deprecated manual stats mutation, updates the seeding docs to call the action reconciler, and changes API rate limiting from a single per-window counter to sharded counters with aggregate read status.

## Best Possible Solution

Land a revised, maintainer-approved patch that removes the live full-table global-stats scans, deletes the deprecated mutation and stale docs reference, updates tests for action-based reconciliation, and reduces rate-limit write contention without silently weakening protected endpoint caps. If it resolves the user-facing throttling symptom, link the outcome back to #654.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Preserve the aggregate rate-limit cap:** `convex/rateLimits.ts:69-70`
  - body: The status query reads all shards, but this mutation then chooses one shard and increments only that shard. When many requests see allowed=true near the cap, writes to different shards will not conflict, so the window can admit more than args.limit requests. Keep a hard per-window reservation/cap or explicitly make the API policy a soft limit before using this on protected routes.
  - confidence: 0.87
- **[P2] Update countPublicSkills fallback tests:** `convex/skills.ts:2753`
  - body: countPublicSkills now returns 0 when globalStats is missing, but the existing tests still assert the live-count fallback returns counted skills in that case. As-is, bun run test should fail once this branch is tested; update those assertions or add coverage for the action reconciliation path.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The PR changes an API abuse-control path and may weaken hard rate-limit enforcement under concurrent sharded writes.

Concerns:

- **[high] Sharded writes can bypass the aggregate cap:** `convex/rateLimits.ts:69`
  - body: Because the mutation writes only one random shard after a separate aggregate status query, concurrent requests near the limit can increment different shards without conflicting and admit more requests than args.limit. That matters for protected write/read API routes that rely on the limiter as an abuse-control boundary.
  - confidence: 0.86
