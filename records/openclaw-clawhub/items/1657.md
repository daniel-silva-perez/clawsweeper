## Summary

Keep open. Current main does not implement dependency-registry/phantom-package scanning, and the latest PR head still has blocking review findings in generated Convex API types, unresolved registry failure handling, and retry coverage. Because this is security-sensitive supply-chain moderation work, it needs maintainer/security review rather than cleanup closure or an automated replacement lane.

## What This Changes

The PR adds a Convex dependency-registry scanner for bundled dependency files, persists scan/cache state, schedules scans after publish, injects a DEP_NOT_FOUND static-scan finding, and adjusts security-scan line rendering.

## Best Possible Solution

Land this PR or a successor after regenerating committed Convex API types, representing unresolved registry lookups as retryable state, making dependency-registry retry status queryable without a fixed newest-500 scan, and coordinating the final moderation behavior with the broader trust remediation item.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Regenerate Convex API types for the new module:** `convex/lib/skillPublish.ts:336-338`
  - body: The branch adds `convex/depRegistryScan.ts` and schedules `internal.depRegistryScan.checkDependencyRegistries`, but the committed generated API file still lacks a `depRegistryScan` module entry. The Convex/deploy typecheck will reject this reference until codegen is rerun and the generated file is committed.
  - confidence: 0.94
- **[P1] Mark partial registry failures as retryable:** `convex/depRegistryScan.ts:508-518`
  - body: When a registry lookup exhausts retries, this code drops that dependency from `results` and only marks the scan as `error` after three consecutive failures. One or two unresolved lookups can therefore be persisted as `clean`, preventing a later retry even though some packages were never checked.
  - confidence: 0.91
- **[P2] Index retryable dependency scans:** `convex/depRegistryScan.ts:640-643`
  - body: This retry query only inspects the newest 500 `skillVersions` and filters `depRegistryAnalysis.status` in memory. Older error-state versions outside that window will never be scheduled for retry, so failed scans can remain stuck indefinitely.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The PR adds useful supply-chain scanning, but the current branch can record an incomplete registry check as clean after partial lookup failures.

Concerns:

- **[high] Incomplete registry checks can be marked clean:** `convex/depRegistryScan.ts:508`
  - body: Retry-exhausted lookups are omitted from `results`; unless three failures are consecutive, the scan can still store `status: "clean"`. That creates a false-negative path in the new dependency-confusion detector and prevents future retries for unchecked packages.
  - confidence: 0.91
