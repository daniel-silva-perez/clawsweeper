## Summary

Keep open. Current main does not implement phantom dependency detection, and this open security-sensitive PR still needs branch fixes and maintainer review before it can land.

## What This Changes

The PR adds a Convex dependency-registry scanner for bundled dependency files, stores scan/cache state, schedules scans after skill publish, injects a DEP_NOT_FOUND static-scan finding, and adjusts security-scan line rendering.

## Best Possible Solution

Land this PR or a successor after regenerating committed Convex API types, fixing unresolved lookup handling, making retry state queryable without a bounded in-memory scan, and completing maintainer/security review against the broader dependency-confusion remediation plan.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Regenerate Convex API types for the new module:** `convex/lib/skillPublish.ts:336-338`
  - body: The branch adds `convex/depRegistryScan.ts` and references `internal.depRegistryScan`, but the committed generated API types in the PR head still do not include a `depRegistryScan` module entry. With the generated types committed this way, TypeScript will reject these references during the Convex/deploy typecheck. Run Convex codegen and commit the updated `_generated/api.d.ts` entry with the new module.
  - confidence: 0.94
- **[P1] Mark partial registry failures as retryable:** `convex/depRegistryScan.ts:508-518`
  - body: When a registry lookup exhausts retries, this code drops the dependency from `results` and only sets `abortedDueToErrors` after three consecutive failures. If one or two failures are interleaved with successful lookups, the scan can still write `status = "clean"` even though some dependencies were never checked, and the early skip then prevents a later retry. Track any unresolved lookup and return `error` unless confirmed missing packages already make the scan suspicious.
  - confidence: 0.91
- **[P2] Index retryable dependency scans:** `convex/depRegistryScan.ts:640-643`
  - body: This retry query only inspects the newest 500 `skillVersions` and filters `depRegistryAnalysis.status` in memory. Older error-state versions outside that window will never be scheduled for retry, so failed scans can remain stuck indefinitely. Store a denormalized status field and query it with an index or use cursor-based pagination that cannot strand older errors.
  - confidence: 0.86
