## Summary

Keep open. This is an open ClawHub issue, and the repository-specific cleanup policy for this pass only permits auto-closing pull requests that are certainly implemented on current main. The code now has partial support for parsing and storing `metadata.openclaw`, but the static scanner still does not directly recognize nested `metadata.openclaw.requires.env`, so the reported mismatch is not fully resolved from source evidence alone.

## Best Possible Solution

Keep this issue open for maintainer handling. The likely fix is to make the static scanner consume normalized parsed metadata, or explicitly descend into `metadata.openclaw`/`metadata.clawdis`/`metadata.clawdbot` when collecting declared env vars and install specs, add regression tests for nested `metadata.openclaw.requires.env` plus install declarations, then backfill or rescan the affected hosted skill version and verify the public registry/scanner output no longer reports the mismatch.
