## Summary

Keep this issue open. Current main still has a broad Crypto capability matcher that treats standalone `swap` as Crypto, publish/backfill paths still derive persisted capability tags from uploaded text files, and #1857 is the open draft implementation candidate for this false-positive path. ClawHub policy also keeps issues open unless the linked implementation has landed.

## Best Possible Solution

Keep this issue open and review or land #1857, or an equivalent maintainer patch: tighten Crypto matching so generic CSS/web text like `display=swap` does not trigger Crypto, preserve positive detection for real DeFi/token swap language, add regression tests, and decide whether already-published capability tags need a targeted backfill or manual correction. A small UI/API/docs improvement exposing matched capability evidence would also help future reporters diagnose labels without guessing.
