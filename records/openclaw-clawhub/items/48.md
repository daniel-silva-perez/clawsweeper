## Summary

Keep open. Repository policy for this ClawHub review does not auto-close issues, and current main still contains a credible implementation path for the reported `SKILL.md required` failure: `skill publish .` resolves the user path under the resolved global workdir, which can be a configured Clawdbot/OpenClaw workspace rather than the shell directory that contains `SKILL.md`. Related #1509 is open and reports the same user-visible failure, so both should be handled by one focused CLI fix.

## What This Changes

The issue requests fixing CLI skill publishing so an existing `SKILL.md` in the requested skill directory is found instead of failing with `SKILL.md required`.

## Best Possible Solution

Skill publish should treat user-entered relative source paths as relative to the caller's current directory when that resolves to a skill folder, while preserving absolute paths and existing configured-workdir behavior for install/update/sync flows. Add focused regression coverage for a cwd skill folder with a different configured Clawdbot/OpenClaw workdir.

## Repair Work Prompt

Fix the CLI skill publish path-resolution bug reported here and in #1509. Current `cmdPublish` resolves relative source paths against `opts.workdir`, so `clawhub skill publish .` can inspect a configured Clawdbot/OpenClaw workspace instead of the caller's skill directory containing `SKILL.md`. Implement a focused fix so user-entered relative source paths resolve against the process cwd when that cwd/path is the requested skill folder, while preserving absolute paths and existing install/update/sync behavior. Be careful that `cmdSync` passes already-resolved skill folders into `cmdPublish`; those sync-driven publishes must continue to work. Add a regression test for publishing `.` from a cwd skill folder while the configured/global workdir points somewhere else. Do not change the registry API, package publish semantics, auth flow, or unrelated workdir behavior. Include an appropriate CLI changelog/release-note entry only if this repo has an established place for CLI user-facing fixes.
