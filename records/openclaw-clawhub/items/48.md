## Summary

Keep open. This is a ClawHub issue, and the repository policy for this review pass does not auto-close issues; independently, current main still has a credible path to the reported failure because relative publish paths are resolved against the resolved global workdir rather than necessarily against the caller's skill directory. ## What This Changes

The issue reports that CLI publish fails with `SKILL.md required` even when `SKILL.md` exists in the directory passed as the publish source.

## Best Possible Solution

Skill publish should resolve user-entered relative source paths against the caller's current directory when that resolves to a skill folder, while preserving absolute paths and existing configured-workdir behavior for install, update, and sync flows; add focused regression coverage for a cwd skill folder with a different configured Clawdbot/OpenClaw workdir.

## Repair Work Prompt

Fix the CLI publish path bug where `clawhub skill publish .` or the legacy `clawhub publish .` can throw `SKILL.md required` even though `SKILL.md` exists in the caller's current directory. Treat user-entered relative publish source paths as relative to `process.cwd()` when that resolves to an existing skill folder, while preserving absolute paths and existing configured-workdir behavior for install/update/sync flows. Add a regression test where the process cwd contains `SKILL.md` but `opts.workdir` points elsewhere, and ensure sync-driven publishes that pass resolved folders still work. Related refs: openclaw/clawhub#48 and openclaw/clawhub#1509. Likely files: `packages/clawhub/src/cli/commands/publish.ts`, `packages/clawhub/src/cli/commands/publish.test.ts`, possibly `packages/clawhub/src/cli.ts` or shared CLI path helpers. Do not broaden this into registry publishing, auth, moderation, or package/plugin publish behavior. No changelog entry is required unless the repo has an active unreleased CLI changelog section for bug fixes.
