## Summary

Keep open. ClawHub cleanup policy does not auto-close open issues, and current main still has the plausible publish failure path: a relative publish source is resolved against the global workdir, which may be a configured Clawdbot/OpenClaw workspace instead of the caller's current skill directory. Related #1509 appears to cover the same user-visible bug and should be fixed with this issue.

## What This Changes

The issue requests a CLI publish fix for `SKILL.md required` being reported even when the publish directory contains a valid `SKILL.md`.

## Best Possible Solution

Keep this issue open and land a focused CLI fix shared with #1509. Relative source paths for `clawhub skill publish .` and legacy `clawhub publish .` should resolve against the caller's cwd when that cwd contains the requested source/SKILL.md, while preserving absolute paths and configured-workspace behavior for install/sync flows. Add regression coverage where the configured OpenClaw/Clawdbot workdir differs from cwd and the cwd skill folder contains `SKILL.md`.

## Repair Work Prompt

Fix the ClawHub CLI publish path resolution bug reported here and in #1509. Current behavior resolves `clawhub skill publish .` / legacy `clawhub publish .` relative to `opts.workdir`, which can be a configured Clawdbot/OpenClaw workspace rather than the caller's cwd containing `SKILL.md`. Implement a focused rule so relative publish source paths resolve against the caller's cwd when that cwd contains the requested source/SKILL.md, while preserving absolute paths, explicit `--workdir` behavior where intentional, and configured-workspace behavior for install/update/sync flows. Add regression coverage where the configured workdir differs from process cwd and the cwd skill folder contains `SKILL.md`. Also preserve `cmdSync` behavior, since sync passes `skill.folder` into `cmdPublish`. Keep the change scoped to CLI publish path resolution and tests; do not alter package publishing, registry API payloads, or broad workdir semantics outside what is needed for this bug. No changelog is required unless the repository convention expects one for CLI bug fixes.
