## Summary

Keep open. ClawHub cleanup policy does not auto-close open issues, and current main still has the plausible publish failure path: a relative publish folder is resolved against the global workdir, which may be redirected to a configured OpenClaw/Clawdbot workspace instead of the caller's current skill directory. Related #1509 appears to cover the same user-visible bug and should be handled with the same focused CLI fix.

## What This Changes

The issue requests fixing a CLI publish failure where `SKILL.md required` is thrown even though a valid `SKILL.md` exists in the target skill directory.

## Best Possible Solution

Keep this issue open and land a focused CLI fix shared with #1509. Relative source paths for `clawhub skill publish .` and legacy `clawhub publish .` should resolve against the caller's cwd when that cwd contains the requested source, while preserving absolute paths and configured-workspace behavior for install/sync flows. Add regression coverage where the configured OpenClaw/Clawdbot workdir differs from cwd and the cwd skill folder contains `SKILL.md`.

## Repair Work Prompt

Fix the CLI publish source path bug reported here and in #1509. Current `cmdPublish` resolves `folderArg` against `opts.workdir`, so `clawhub skill publish .` and legacy `clawhub publish .` can inspect a configured OpenClaw/Clawdbot workspace instead of the caller's current skill folder and throw `SKILL.md required`. Implement a conservative publish-only source path resolver: absolute paths must keep working; relative publish paths should prefer `process.cwd()` when that candidate exists, with a fallback that preserves existing `opts.workdir` behavior where appropriate. Do not change install/update/list/sync global workdir semantics. Preserve `cmdSync` behavior because it passes absolute `skill.folder` values to `cmdPublish`. Add regression coverage where cwd contains `SKILL.md`, configured/global `opts.workdir` points to another folder, and `cmdPublish(opts, '.', ...)` uploads the cwd skill rather than throwing. Also cover the legacy alias path indirectly if command-level tests exist; otherwise unit coverage on `cmdPublish` is acceptable. No release metadata or broad refactors unless maintainers request them.
