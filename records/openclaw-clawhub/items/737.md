## Repair Work Prompt

Fix the stale duplicate/fork relation display for public skill detail pages. Current production shows @nexecuteinc/cmdbcompass as `duplicate of @nexecuteinc/cmdb-compass` even though the old slug is not public. In current main, `convex/skills.ts#getBySlug` returns `forkOf` and `canonical` payloads whenever the related docs exist, without checking whether the related skill and owner are public. Make public `getBySlug` relation hydration only include `forkOf` / `canonical` when the related skill passes `isPublicSkillDoc` and its owner can be publicized; otherwise return null for that relation. Preserve staff detail/management behavior and legitimate public duplicate/fork labels. Do not attempt production data repair, do not broaden moderation policy, and do not change unrelated skill publishing or duplicate-management workflows. Add focused regression coverage for a soft-deleted or hidden related target and, if useful, a UI assertion that no duplicate label renders when the backend relation is null. Mention in the PR summary that staff still need to clear the persisted association for the specific skill if they want the internal relation removed.
