## Summary

Keep this issue open. Current main still does not contain GitHub Issue Forms, `.github/ISSUE_TEMPLATE/config.yml`, or broad support-intake labels for the requested ClawHub workflows. The only related implementation on main is the narrower merged rescan-guidance path from #1878, which handles explicit rescan/re-evaluation requests rather than structured intake for auth, publish, search, ownership, false-positive appeals, or general feature reports. Repository policy for openclaw/clawhub also says issues should remain open unless they are PRs certainly implemented on main.

## Best Possible Solution

Keep this issue open for maintainer/product follow-up. The best path is to approve the support categories and label names, add `.github/ISSUE_TEMPLATE` YAML forms plus `config.yml`, use form-level static labels where possible, link forms to maintainer-approved canonical threads, and only add a small `issues: write` triage workflow if GitHub issue-form labels are not enough.
