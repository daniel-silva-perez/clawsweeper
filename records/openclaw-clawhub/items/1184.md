## Summary

Keep this issue open. Current main still leaves the owner-publisher package and skill digest sync helpers able to run multiple `.paginate()` calls inside one internal mutation invocation. The existing scheduler split mitigates the package+skill same-trigger path, but it does not fix multi-page owner catalogs, and the related publish/import reports remain open.

## Best Possible Solution

Keep this issue open and land one focused Convex fix for the duplicate publish/import pagination cluster. The durable implementation is to make owner-publisher package and skill digest sync process one indexed page per scheduled internal mutation, pass `cursor` and `batchSize` through internal args, reschedule while `isDone` is false, and add regression tests proving the multi-page path never performs more than one `.paginate()` call in one invocation. After that merges and deploys, maintainers can consolidate the duplicate reports against the canonical fix.

In `convex/functions.ts`, change `syncPackageSearchDigestsForOwnerPublisherIdInternal` and `syncSkillSearchDigestsForOwnerPublisherIdInternal` so each invocation processes at most one indexed `.paginate()` page for the given `ownerPublisherId`, then uses `ctx.scheduler.runAfter(0, sameInternalFunction, { ownerPublisherId, cursor: page.continueCursor, batchSize })` while `!page.isDone`. Add focused Vitest coverage in `convex/functions.test.ts` for package and skill multi-page behavior: the first invocation should call `.paginate()` once, sync that page, and schedule the continuation with `continueCursor`; a done page should not reschedule. Preserve the existing scheduling/no-scheduler tests. Run codegen if internal function arg typing requires generated API changes; commit generated files only if they change.
