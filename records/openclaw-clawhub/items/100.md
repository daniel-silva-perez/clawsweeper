## Summary

Keep open. Current main still treats `https://clawhub.ai` as the canonical CLI registry/site host, while a live dummy-token probe no longer reproduces the originally reported non-www to www redirect. Because this is a ClawHub issue, the cleanup policy keeps issues open, and the remaining question needs maintainer verification with a real production token plus a decision about related #99.

## What This Changes

The issue reports that authenticated CLI requests fail when the default registry uses `https://clawhub.ai` because a redirect to `https://www.clawhub.ai` strips the bearer token.

## Best Possible Solution

Maintainers should verify the current published CLI against production with a real token. If the default non-www flow works, close or consolidate this with #99; if a real-token redirect still occurs, make one consistent canonical-domain fix across CLI defaults, well-known discovery, docs/tests, and hosting configuration.
