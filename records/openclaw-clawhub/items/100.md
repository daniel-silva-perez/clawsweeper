## Summary

Keep open. Current main and the live well-known discovery still use `https://clawhub.ai` as canonical, while a dummy-token probe no longer reproduces the originally reported non-www to www redirect. Because this is a ClawHub issue, the repo policy keeps it open, and the remaining question needs maintainer verification with a real production token plus a decision about whether to consolidate with related #99.

## What This Changes

Ensure CLI authentication succeeds with the default ClawHub registry without losing the Authorization header across `clawhub.ai` and `www.clawhub.ai` redirects.

## Best Possible Solution

Maintainers should verify the current published CLI against production with a real token. If the default non-www flow works, close or consolidate this with #99; if a real-token redirect still occurs, make one narrow canonical-domain fix across CLI defaults, well-known discovery, tests, docs, and hosting configuration.
