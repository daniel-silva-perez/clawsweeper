## Summary

Keep open. This is an issue, and the supplied ClawHub cleanup policy says issues should not be auto-closed by this workflow. Current main still uses the apex host for CLI defaults and discovery metadata; live header checks now show www redirecting back to the apex host, so the right next step is maintainer verification with a fresh production token rather than a blind www-default change.

## Best Possible Solution

Keep this issue open for maintainer handling. A maintainer should verify `clawhub login --token <fresh production token>` and `clawhub whoami` against the default registry, then compare explicit `CLAWHUB_REGISTRY=https://clawhub.ai` and `CLAWHUB_REGISTRY=https://www.clawhub.ai`. If auth still fails, fix production routing so authenticated API requests never cross origins, or intentionally switch the canonical host across CLI defaults, discovery metadata, docs, tests, and release notes together. Coordinate any remaining Unauthorized-login work with #99 and do not revive the closed www-default PR without rechecking current production routing.
