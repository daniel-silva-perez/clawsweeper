## Summary

Keep open. Current main still intentionally uses the non-www canonical host, and a live dummy-token probe no longer reproduces the reported `clawhub.ai` to `www.clawhub.ai` redirect. Because this is an issue in `openclaw/clawhub`, and because the remaining question requires a real production token plus domain-config verification, this should stay with maintainers rather than be auto-closed.

## What This Changes

The issue reports CLI token authentication failing because requests to the non-www registry allegedly redirect to `www.clawhub.ai`, stripping `Authorization`, and suggests changing the default registry or redirect behavior.

## Best Possible Solution

The desired end state is a verified canonical-domain auth flow: either maintainers confirm current non-www production works with a real token and close or consolidate this with #99, or they make one narrow domain/auth fix across CLI defaults, well-known discovery, tests, docs, and hosting configuration.
