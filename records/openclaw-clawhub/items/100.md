## Summary

Keep open. Current main and live discovery still treat `https://clawhub.ai` as canonical, while a dummy-token probe no longer reproduces the reported non-www to www redirect; the remaining auth question needs maintainer verification with a real production token and may overlap broader Unauthorized tracking in #99.

## What This Changes

The issue reports that CLI token authentication fails when the default `https://clawhub.ai` registry redirects to `https://www.clawhub.ai` and strips the Authorization header.

## Best Possible Solution

Maintainers should verify the current published CLI against production with a real token. If the default non-www flow works, close or consolidate this with #99; if a real-token redirect still occurs, make one consistent canonical-domain fix across CLI defaults, well-known discovery, docs/tests, and hosting configuration.
