## Summary

Keep open. This is an issue, and the openclaw/clawhub cleanup policy says issues are not auto-closed. Current main still defaults and documents the CLI registry/site as https://clawhub.ai, the committed discovery document still returns apex URLs, and the token login/whoami path still verifies through GET /api/v1/whoami with a bearer token. The previously linked default-to-www PR was closed unmerged, while #99 remains open for a broader Unauthorized login report. A live invalid-token header check no longer reproduces the originally reported apex-to-www redirect, but without a valid production API token that is not enough to prove the authenticated CLI flow is fixed.

## Best Possible Solution

Keep this issue open for maintainer handling. Maintainers should verify the live token flow with a fresh API token against https://clawhub.ai. If it still fails, fix production routing so authenticated API requests are not cross-origin redirected, or intentionally switch CLI/discovery/docs/tests to the chosen canonical host. If production is now verified fixed, a maintainer can manually close this as obsolete or cannot-reproduce under the repository policy and consider consolidating any remaining Unauthorized-login work with #99.
