## Summary

Keep open. Current main and live discovery still use the non-www canonical host, while a dummy-token production probe no longer reproduces the reported non-www to www redirect. Because this is a ClawHub issue, the repository policy does not auto-close it, and the remaining verification requires a real production token plus maintainer judgment about related #99.

## What This Changes

The issue reports CLI authentication failing because requests to `https://clawhub.ai` allegedly redirect to `https://www.clawhub.ai` and lose the bearer Authorization header.

## Best Possible Solution

Maintainers should verify the current published CLI against production with a real token. If the non-www default flow works, this can close or consolidate with #99; if it fails, make one consistent canonical-domain fix across hosting configuration, CLI defaults, well-known discovery, docs, and focused CLI/auth tests.
