## Summary

Keep this issue open. The ClawHub cleanup policy only allows auto-closing PRs, not issues, and current main still has the reported failure path: a non-trusted clean update to an existing active skill can write the skill back to hidden pending scan, while public search/API/CLI inspect paths filter hidden skills out. PR #136 mitigated owner visibility, but PR #274, which directly proposed preserving active moderation status on updates, was closed unmerged.

## Best Possible Solution

Keep this issue open for maintainer decision. If active skills should stay public during clean update scans, update insertVersion/publishVersionForUser so existing active skills preserve active moderation status unless the new version is static-malicious, quality-quarantined, blocked, removed, or the publisher is under moderation; add regression tests for non-trusted active-skill updates and owner/public visibility. If quarantine-on-update is intentional, document that policy and align CLI, web UI, API responses, dashboard status, and production repair workflows so successful publish does not look like data loss.
