## Summary

Keep this PR open for maintainer/security review: current main still lacks the moderation-hold lift path, and the latest PR patch remains an active security-sensitive implementation with blocking restore and audit correctness issues.

## What This Changes

The PR adds public and internal Convex admin mutations to clear user moderation holds, batch-restore eligible owned skills, add tests, and document the admin recovery flow.

## Best Possible Solution

Land a maintainer-reviewed implementation that clears user hold fields, restores only skills actually eligible for hold recovery, keeps malicious/manual/quality/soft-deleted/unresolved-scan states protected, records accurate cumulative audit data, and documents the admin Convex CLI path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Treat not_found VT results as unresolved:** `convex/skills.ts:4147`
  - body: The restore path only keeps missing, `pending`, or `loading` VT results in the scan queue. Current moderation logic maps `vtAnalysis.status === "not_found"` to `scanner.vt.pending`, so a lifted skill with `not_found` becomes `active/restored.moderation_lift` and drops out of scanning. Include `not_found` or reuse the existing pending/final-status helper before making the skill public.
  - confidence: 0.92
- **[P1] Preserve pre-hold moderation locks:** `convex/skills.ts:4128-4129`
  - body: This restore treats the post-hold `user.moderation` reason as proof that the skill is safe to reactivate. The hold batch rewrites every non-malicious owned skill to `user.moderation`, including rows already hidden for independent reasons such as `quality.low`, so lifting the hold can publish content that was quarantined before the user hold. Store the prior moderation state or skip rows that were already hidden for another reason.
  - confidence: 0.82
- **[P2] Report all restored skills in the lift audit:** `convex/users.ts:813`
  - body: The caller records only the first batch's `restoredCount`; when `scheduledSkills` is true, continuation batches can restore additional skills without updating the `user.moderation.lift` metadata or returned count. Large false-positive recoveries will therefore have incomplete audit data, which undermines the recovery trail this PR is adding.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: No supply-chain changes were found, but the moderation restore semantics can expose hidden or unresolved content and need security review before merge.

Concerns:

- **[high] Unresolved VT results can be published:** `convex/skills.ts:4147`
  - body: The PR treats `not_found` VT status as safe to restore even though current moderation classifies it as pending, allowing a skill to become public and leave the scan queue.
  - confidence: 0.92
- **[high] Independent moderation decisions can be cleared:** `convex/skills.ts:4128`
  - body: Because the existing hold batch overwrites hidden non-malicious skills with `user.moderation`, the new restore can make previously manual- or quality-hidden skills active.
  - confidence: 0.82
- **[medium] Recovery audit can undercount batch restores:** `convex/users.ts:813`
  - body: Only the first batch count is written to the lift audit log, so large recoveries can omit later restored skills from the security trail.
  - confidence: 0.86
