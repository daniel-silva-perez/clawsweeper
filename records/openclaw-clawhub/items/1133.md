## Summary

Keep this PR open. Current main does not implement the moderation-hold lift API/docs surface, and the related false-positive reports support the need for a recovery path, but the proposed patch still has security-sensitive restore semantics that can publish unresolved or independently hidden content and can undercount multi-batch recovery audits.

## What This Changes

The PR adds public and internal Convex admin mutations to clear user moderation holds, batch-restore eligible owned skills, write audit logs, add tests, and document the admin CLI recovery flow.

## Best Possible Solution

Land a maintainer-reviewed implementation that adds an admin-only hold-lift operation, clears user hold fields, restores only skills genuinely hidden by that hold, preserves malicious, manual/quality, soft-deleted, and unresolved-scan states, records complete recovery audit data across batches, and documents the Convex CLI workflow.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Treat not_found VT results as unresolved:** `convex/skills.ts:4147-4148`
  - body: The restore path only keeps missing, pending, or loading VT states in the scan queue. Main maps vtAnalysis.status === "not_found" to scanner.vt.pending and the queue retries non-final VT states, so a lifted not_found skill becomes active/restored.moderation_lift and drops out of scanning. Include not_found or reuse the existing final-status logic before making the skill public.
  - confidence: 0.92
- **[P1] Preserve pre-hold moderation locks:** `convex/skills.ts:4124-4128`
  - body: This predicate treats post-hold user.moderation as proof that a skill can be reactivated, but the existing hold batch overwrites every non-malicious owned skill with user.moderation, including rows already hidden for quality or manual review. Store the prior moderation state or avoid restoring rows that were locked before the hold.
  - confidence: 0.82
- **[P2] Report all restored skills in the lift audit:** `convex/users.ts:813`
  - body: The caller records only the first batch's restoredCount. When scheduledSkills is true, continuation batches can restore additional rows without updating the returned count or user.moderation.lift metadata, leaving large recoveries with an incomplete audit trail.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: No supply-chain changes were found, but the moderation restore path can expose unresolved or independently hidden content and needs security review before merge.

Concerns:

- **[high] Unresolved VT results can be published:** `convex/skills.ts:4147`
  - body: The PR treats not_found VT status as safe to restore even though current moderation classifies it as pending, allowing a skill to become public and leave the scan queue.
  - confidence: 0.92
- **[high] Independent moderation decisions can be cleared:** `convex/skills.ts:4128`
  - body: Because the hold batch overwrites hidden non-malicious skills with user.moderation, the new restore path can make previously manual- or quality-hidden skills active.
  - confidence: 0.82
- **[medium] Recovery audit can undercount batch restores:** `convex/users.ts:813`
  - body: Only the first batch count is written to the lift audit log, so large recoveries can omit later restored skills from the security trail.
  - confidence: 0.86
