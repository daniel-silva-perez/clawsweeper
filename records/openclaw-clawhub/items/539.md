## Summary

Keep this issue open. Current main has real mitigations for authenticated/API-token callers and retry-aware 429 handling, but anonymous search/install/download paths still use per-IP buckets and can still fail for VPS/shared-egress users. The ClawHub cleanup policy also says issues stay open; only clearly implemented PRs are eligible for auto-close.

## Best Possible Solution

Keep this issue open for maintainer triage and consolidate it with #390 and #1410 if desired. The best path is to choose a concrete backend/product direction: make CLI authentication the default onboarding path, adjust anonymous shared-IP limits, verify production client-IP forwarding, improve 429 diagnostics across install paths, or design official mirror/offline registry support.

## Risks / Open Questions

- Anonymous VPS/shared-egress users can still hit per-IP read or download limits during search, metadata lookup, package file reads, and install/download flows.
- If production proxy headers are absent or not trusted, anonymous callers can collapse into one shared proxy IP or ip:unknown bucket.
- CLI authentication is a practical workaround, but it remains optional for search/install and does not fully solve anonymous plugin install paths.
- Changing anonymous limits, allowlisting, mirror behavior, or default-login onboarding has abuse, security, and service-capacity tradeoffs that need maintainer judgment.
- There are overlapping open reports (#390 and #1410), so follow-up should avoid fragmenting one rate-limit problem across several partial fixes.
