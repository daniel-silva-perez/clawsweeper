## Summary

Keep this issue open. Current main has meaningful mitigations for authenticated CLI/API-token users and retry-aware 429 handling, but the reported shared-IP failure mode is still present for anonymous search, metadata, file, and download/install paths. The ClawHub cleanup policy for this review also says issues should stay open; only clearly implemented PRs are eligible for auto-close.

## Best Possible Solution

Keep this issue open for maintainer triage and consolidate it with #390 and #1410 if desired. The best path is to choose a concrete product/backend direction: make CLI authentication the default onboarding path, reshape anonymous shared-IP limits, verify production client-IP forwarding, improve 429 diagnostics across all install paths, or design official mirror/offline registry support.

## Risks / Open Questions

- Anonymous VPS/shared-egress users can still hit per-IP read or download limits during search, metadata lookup, package file reads, and install/download flows.
- If production proxy headers are absent or not trusted, anonymous callers can collapse into one shared proxy IP or ip:unknown bucket.
- CLI authentication is a practical workaround, but it remains optional for search/install and does not cover anonymous OpenClaw plugin install paths.
- Changing anonymous limits, allowlisting, or mirror behavior has abuse and service-capacity tradeoffs that need maintainer product/security judgment.
- There are overlapping open reports (#390 and #1410), so a fix should avoid fragmenting follow-up work across multiple partially overlapping issues.
