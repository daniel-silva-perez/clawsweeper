## Summary

Keep this issue open. Current main has useful mitigations for authenticated API-token users and retry-aware 429 handling, but anonymous search/package/download flows are still IP-bucketed and can still fail for VPS/shared-egress users. The repository policy for openclaw/clawhub also says issues should stay open under this cleanup workflow; only clearly implemented PRs are eligible for auto-close.

## Best Possible Solution

Keep this issue open for maintainer triage. The best path is to consolidate with #390 and #1410 if desired, then choose a concrete backend/product fix: make CLI authentication the default onboarding path, raise or reshape anonymous shared-IP limits, verify production client-IP forwarding, improve 429 diagnostics across all install paths, or design official mirror/offline registry support.

## Risks / Open Questions

- Anonymous VPS/shared-egress users can still hit per-IP read or download limits during search, metadata lookup, file fetches, and install/download flows.
- If production proxy headers are absent or not trusted, anonymous callers can collapse into a shared proxy IP or ip:unknown bucket.
- CLI authentication is a practical workaround, but it is still optional and does not solve anonymous install paths or OpenClaw plugin installs that do not send a token.
- Closing this now would conflict with the ClawHub issue-review policy and could hide remaining work already reflected in #390 and #1410.
