## Summary

Keep open. ClawHub cleanup policy does not auto-close issues, and current main still contains a plausible dashboard-to-detail path for the reported state: owned packages link to plugin detail, and a package detail 404/null response renders the exact “Plugin not found” UI. The affected package name, publisher/account, signed-in state, and API response are still needed before this can be classified as a code bug versus data, auth, visibility, publish-state, or scan-state behavior.

## What This Changes

A published plugin shown to its publisher should be viewable instead of showing “Plugin not found.”

## Best Possible Solution

Keep the issue open until a maintainer obtains the affected package name or canonical URL, publisher/account, signed-in state, browser/network details, and /api/v1/packages/<name> status/body. If reproduced as a code path, update the package detail API and /plugins/$name UI to distinguish owner-visible private, pending, blocked, auth-failed, and fetch-failed states from truly missing packages while preserving public 404 behavior for unreadable packages.
