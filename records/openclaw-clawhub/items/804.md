## Summary

Keep this issue open. Current main still leaves the reported owner/slug ambiguity unresolved in the CLI search/install contract, and the ClawHub cleanup policy says issues should not be auto-closed. The web skill detail page now partially mitigates the problem with owner-qualified prompt text, but visible CLI install commands, CLI install input validation, and the public v1 search API remain bare-slug based.

## Best Possible Solution

Keep this issue open and implement a scoped owner/slug identity design: expose ownerHandle or owner id in the v1 search API and CLI schema, show owner or owner/slug in `clawhub search`, decide whether `clawhub install` should accept owner-qualified targets while preserving slug-only compatibility, resolve owner-qualified input without writing unsafe nested paths, and align the skill detail command panel, docs, tests, and download/install naming with that contract.

## Risks / Open Questions

- Changing install input from bare slug to owner/slug will need a careful compatibility plan so existing slug-only installs, lockfile entries, and destination paths keep working.
- The current slash rejection protects filesystem writes from nested paths and traversal; any owner-qualified install support should resolve identity separately from the local install directory name.
- Adding owner identity to /api/v1/search changes the public API/CLI schema and should be documented and covered by tests.
