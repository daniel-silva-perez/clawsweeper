## Summary

Keep this issue open. The ClawHub cleanup policy says open issues should not be auto-closed, and current main only partially addresses the reported owner/slug confusion: web pages are more owner-aware, but CLI search, the public search API/schema, CLI install targets, docs, and download filenames are still slug-only in the paths that caused the original mistake.

## Best Possible Solution

Keep this issue open and implement a scoped identity clarification: expose ownerHandle or owner in /api/v1/search, update the CLI search schema and output to show owner/slug or an owner column, decide whether `clawhub install` should accept owner-qualified targets while preserving slug-only compatibility and filesystem safety, then align web install commands, download filenames, and docs with that decision.

## Risks / Open Questions

- Closing now would leave the original duplicate-slug confusion unresolved for CLI users, especially when search results contain multiple skills with the same or similar slug.
- Owner-qualified installs need a backwards-compatible design for local directory names, lockfile keys, origin metadata, update/uninstall behavior, and path traversal protections.
- Adding owner fields to public search should be coordinated across the Convex API response, shared schemas, CLI output, docs, and older-client compatibility.
