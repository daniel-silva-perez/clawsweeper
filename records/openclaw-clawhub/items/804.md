## Summary

Current main still leaves the reported owner/slug ambiguity unresolved, and repository policy for openclaw/clawhub says issues should stay open unless they fall under a non-implemented close reason. CLI search output still shows only the bare slug, CLI install still validates a bare slug and rejects owner-qualified input, the v1 search response omits owner identity even though backend search results can contain it, and the web skill page only partially mitigates this by using owner-qualified prompt labels while visible install commands remain bare-slug based.

## Best Possible Solution

Keep this issue open and implement a scoped owner/slug identity design: expose owner handle or owner id in the v1 skill search API and CLI schema, display owner or owner/slug in `clawhub search`, decide whether `clawhub install` should accept owner/slug while preserving slug-only compatibility, and align download filenames, docs, and skill-page install commands with that contract.
