## Summary

Keep open. Current main still has the reported owner/slug ambiguity in the CLI and public v1 search API; the web skill page has a partial owner-qualified prompt surface, but `clawhub search` remains ownerless and `clawhub install` still only accepts bare slugs. ClawHub policy also says issues should stay open unless maintainers handle them explicitly.

## What This Changes

The issue asks for clearer owner-qualified skill identification in ClawHub search/install flows so users can distinguish skills with the same or similar bare slug.

## Best Possible Solution

Keep this issue open and land a narrow owner-disambiguation PR: expose owner identity in the v1 search response, update both search response schemas, print owner or owner/slug in `clawhub search`, and refresh CLI/API docs and tests. Treat owner-qualified install locators as separate product/security design unless maintainers explicitly scope that into the same work.

## Repair Work Prompt

Implement a narrow owner-disambiguation fix for ClawHub skill search. Expose owner identity from the existing internal `SkillSearchEntry.ownerHandle`/`owner` data in `/api/v1/search`, update both `packages/schema/src/schemas.ts` and `packages/clawhub/src/schema/schemas.ts`, and update `clawhub search` to display an owner or owner-qualified locator so users can distinguish skills with the same or similar bare slug. Refresh CLI/API docs and focused tests. Do not relax `normalizeSkillSlugOrFail`, do not implement `clawhub install owner/slug`, and do not change install directory layout/API lookup semantics in this PR unless maintainers explicitly decide that scope. Preserve backward-compatible additive API behavior and handle null/missing owner handles gracefully.
