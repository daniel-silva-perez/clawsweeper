## Summary

Keep open. This is an open ClawHub issue, and the supplied repository policy only allows auto-closing pull requests that are certainly implemented on current main. The underlying request is also only partially addressed: the web UI now has owner-aware surfaces, but CLI search, the public search API, CLI install, docs, and download filenames remain slug-only in the paths that caused the confusion.

## Best Possible Solution

Keep this issue open and implement a scoped identity clarification: expose ownerHandle or owner in /api/v1/search, update the CLI search schema and output to show owner/slug or a separate owner column, decide whether clawhub install should accept owner-qualified targets while preserving slug-only compatibility, and align the web install commands, docs, and download filenames with that decision.

## Risks / Open Questions

- Owner-qualified installs need a backwards-compatible design for install target directories, lockfile keys, origin metadata, update, uninstall, and existing slug-only workflows.
- Adding owner data to public search should be coordinated across the HTTP API response, CLI schema, CLI output, docs, and older-client compatibility.
- The current slug sanitizer intentionally blocks slashes for filesystem safety, so accepting owner/slug should preserve path traversal protections rather than relaxing the existing guard broadly.
