## Summary

Keep open. Current main still leaves the reported CLI discovery ambiguity in place: skill search results do not expose owner identity through the public v1 response or CLI output, and install still accepts only bare slugs. The web skill-detail prompt surface now partially mitigates owner-qualified identity for copied prompts, but it does not solve the CLI search/install confusion described by the reporter.

## What This Changes

The issue asks ClawHub to make skill identity clearer when multiple skills share similar bare slugs, preferably with owner-qualified identifiers or at least owner information in search and install guidance.

## Best Possible Solution

Expose owner identity in the public v1 skill search response, update the shared and CLI-local schemas, and render the owner in `clawhub search` so users can distinguish entries like `steipete/weather` from other `weather`-named skills while keeping owner-qualified install syntax as a separate product/security decision.

## Repair Work Prompt

Implement a narrow owner-disambiguation fix for the ClawHub skill search workflow. Add owner identity to the public v1 skill search response from `searchSkillsV1Handler`, update both `packages/schema/src/schemas.ts` and `packages/clawhub/src/schema/schemas.ts`, and render that owner information in `clawhub search` output so users can distinguish results such as `steipete/weather` from other `weather` skills. Prefer an additive nullable `ownerHandle` field or an existing public owner shape consistent with nearby API responses. Update HTTP API docs and focused tests. Keep `clawhub install owner/slug`, download filename behavior, and filesystem slug normalization out of scope unless maintainers explicitly request that broader design. Do not change unrelated package/plugin search behavior.
