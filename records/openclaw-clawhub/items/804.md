## Summary

Keep open. Current main still has the reported owner/slug ambiguity in the CLI and public v1 search API: web skill pages now have a partial owner-qualified prompt surface, but `clawhub search` remains ownerless and `clawhub install` still accepts only bare slugs. Repository policy also says ClawHub issues should stay open unless maintainers handle them explicitly.

## What This Changes

Request to make ClawHub skill search/install less ambiguous when multiple owners publish similar slugs, for example by showing owner identity or supporting owner-qualified locators.

## Best Possible Solution

Keep this issue open and land a narrow owner-disambiguation PR: expose owner identity in the v1 search response, update both search response schemas, print owner or owner/slug in `clawhub search`, and refresh CLI/API docs and tests. Treat owner-qualified install locators as separate product/security design unless maintainers explicitly scope that into the same work.

## Repair Work Prompt

Implement a narrow owner-disambiguation fix for this issue. On current main, `clawhub search` prints only bare slugs and `/api/v1/search` omits owner identity even though internal search entries carry `ownerHandle` and `owner`. Add an additive nullable owner field to the public v1 search response, preferably `ownerHandle: string | null` and only a larger owner object if that matches existing public schema conventions. Update both schema copies and any committed schema dist artifacts required by the repo. Update `clawhub search` output to show the owner clearly, such as `@owner/slug` or an Owner column, with a clean fallback when owner is missing. Update CLI/API docs and focused tests. Do not loosen `normalizeSkillSlugOrFail`, do not add owner-qualified install/update/uninstall parsing, and do not change filesystem target naming or lockfile/origin key behavior in this PR; that is a separate maintainer-scoped design.
