## Summary

Keep open. Current main still leaves the reported owner/slug ambiguity unresolved in the CLI and public v1 skill search contract, and the repository policy for this sweep keeps ClawHub issues open. The web skill page has a partial owner-qualified prompt path, but CLI search, CLI install, the v1 search schema, docs, and tests still use bare slugs for the main install/search flow.

## Best Possible Solution

Keep this issue open and implement a staged owner-identity fix. The lowest-risk first PR is to expose owner identity in the v1 skill search response/schema, show owner or owner/slug in `clawhub search`, and update CLI/API docs and tests. A broader maintainer-owned follow-up can decide whether `clawhub install` should accept `@owner/name` or `owner/slug`, while preserving bare-slug compatibility and avoiding unsafe filesystem paths.

Scope:

- Update `docs/http-api.md` and `docs/cli.md` to document the owner field/search output.

Do not implement owner-qualified install in this PR unless maintainers explicitly approve it. Keep existing bare-slug clients compatible.

## Risks / Open Questions

- Owner-qualified install input intersects with filesystem paths. The current slash rejection prevents nested writes, so any future `owner/slug` or `@owner/name` support must resolve to a safe canonical slug or directory key before extraction, origin writes, and lockfile writes.
- Adding owner fields to the public v1 search response should be backward compatible for existing clients and mirrored in both schema packages if both remain published/consumed.
- A UI-only or prompt-only mitigation would leave the original CLI confusion partly unresolved if `clawhub search`, docs, and install/download naming stay bare-slug-only.
