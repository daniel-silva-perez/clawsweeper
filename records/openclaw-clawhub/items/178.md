## Summary

Keep open. The cleanup policy for ClawHub issues does not allow auto-closing this issue, and current main still appears to have the reported mismatch: local update hashing can include generated root `_meta.json` from downloaded skills while server resolve compares against publish-time/version-file fingerprints that do not include that generated metadata.

## What This Changes

The issue requests that `clawhub update` recognize freshly installed or force-updated skills as up to date instead of warning `local changes (no match)` when the registry fingerprint should match.

## Best Possible Solution

Align downloaded/local and published/resolve fingerprint inputs so generated ClawHub metadata does not create false local-change warnings, while preserving integrity coverage for authored files and adding regression tests for install or force-update followed by update reporting up to date.

## Repair Work Prompt

Fix the false `local changes (no match)` warning for `clawhub update` after a fresh install or immediate `--force` update. Current main downloads skill ZIPs with generated root `_meta.json`; local `listTextFiles` includes root JSON in the update fingerprint, while publish/resolve fingerprints are computed from authored version files. Implement a narrow alignment: exclude only ClawHub-generated root `_meta.json` from managed local update/sync fingerprinting or otherwise make download/package/resolve compare the same logical file set. Preserve normal authored JSON files in fingerprints and do not broaden ignore behavior. Coordinate the fix with related #169 and #937, and note #416 as a closed duplicate/reference. Add regression coverage that install or force-update followed by update resolves as up to date/no prompt, plus unit coverage that authored JSON remains hashed while generated root `_meta.json` does not cause a mismatch. Do not change unrelated package download behavior, security moderation, dependency sources, or release scripts.
