## Summary

Keep open. This is an open ClawHub issue, and the repository-specific cleanup policy only permits auto-closing ClawHub pull requests that are certainly implemented on current main. Substantively, current main still has the reported mismatch path: downloaded skill ZIPs add generated `_meta.json`, local fingerprinting includes JSON files unless ignored, and server resolve compares against publish-time file fingerprints that do not include generated download metadata.

## Best Possible Solution

Keep this issue open for maintainer follow-up. The best fix is to make client and server fingerprint inputs match, most likely by excluding generated ClawHub metadata such as `_meta.json` from managed-install fingerprints or changing the download/package metadata behavior, then adding a regression test for install or force-update followed immediately by update reporting up to date. Maintainers may also consolidate this with open related #169 if they want one canonical tracker.

## Risks / Open Questions

- Users may still see false `local changes (no match)` warnings immediately after a clean install or force update when generated `_meta.json` is present locally but absent from registry fingerprints.
- Repeated false warnings can push users toward routine `--force` updates, weakening the local modification protection that update is intended to provide.
- I did not find a focused regression test proving a freshly installed or force-updated managed skill immediately resolves as up to date when generated download metadata is present.
