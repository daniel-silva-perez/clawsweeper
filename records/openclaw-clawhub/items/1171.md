## Summary

Keep this issue open. ClawHub cleanup policy keeps open issues open, and current main still has a concrete residual Convex pagination failure path: publisher digest sync is scheduled into separate internal mutations, but each scheduled mutation can still call `.paginate()` repeatedly for multi-page owners.

## What This Changes

Fix GitHub import and publish failures caused by Convex's multiple-paginated-query error during owner-publisher digest sync.

## Best Possible Solution

Keep the issue open and land one focused backend fix in `convex/functions.ts`: make owner-publisher package and skill digest rebuilds cursor-batched internal mutations where each Convex function invocation performs at most one `.paginate()` call, processes that page, and schedules the next cursor until complete. Add regression tests for multi-page owner-publisher package and skill sync, then consolidate sibling reports after the fix ships.

## Repair Work Prompt

Fix the ClawHub publish/import Convex pagination failure for owner-publisher digest sync. Current main schedules package and skill sync separately, but `syncPackageSearchDigestsForOwnerPublisherId` and `syncSkillSearchDigestsForOwnerPublisherId` still loop over `.paginate()` inside one internal mutation invocation. Refactor the owner-publisher package and skill digest rebuilds so each Convex mutation invocation performs at most one `.paginate()` call, processes that page, and schedules itself with the continuation cursor until complete. Cover both packages and skills. Add focused regression tests that simulate multi-page owner-publisher sync and assert cursor continuation is scheduled rather than a second paginate call in the same invocation. Do not replace this with unbounded `.collect()`, do not introduce `.filter()` table scans, and avoid unrelated CLI/UI changes. No changelog is expected unless the repo has an established entry point for this backend bugfix.
