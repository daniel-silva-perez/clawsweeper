## Summary

Keep open. This is an open ClawHub issue, and the repository-specific cleanup policy only allows auto-closing ClawHub pull requests that are definitely implemented on current main. Current source also still has the reported behavior path: the owner/slug route can SSR a 200 shell with null metadata when `skills.getBySlug` misses, while the public API returns 404 for the same missing public skill state. The hosted affected URL still reproduces, so this remains production-data or route-behavior triage rather than conservative cleanup.

## Best Possible Solution

Keep this issue open for maintainer production triage. Inspect the `mastrophot/near-phishing-detector` production skill row, owner and slug alias mapping, `latestVersionId`, `latestVersionSummary`, search digest, moderation/security scan state, and SSR/API cache behavior. If the skill should be public, repair or reindex the production record and add focused regression coverage for the public detail API plus owner/slug SSR loader. If it should remain unavailable, change the owner/slug route to render an explicit not-found or moderated-state page instead of a 200 shell with null metadata.
