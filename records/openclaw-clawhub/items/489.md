## Summary

Keep open. This is a ClawHub issue, and the repository cleanup policy only permits auto-closing pull requests that current main definitely implements. The reported production inconsistency still reproduces, and current main still has a route/loader path that can return an owner/slug SSR shell with null metadata when the public skill lookup has no public result.

## What This Changes

The report asks for `mastrophot/near-phishing-detector` to expose normal public skill metadata instead of resolving to a null-metadata SSR page while the API returns not found.

## Best Possible Solution

Resolve the production inconsistency explicitly: if the skill should be public, repair or reindex the production record so the public API and owner/slug SSR loader return populated metadata; if it should remain unavailable, make the owner/slug route render a clear not-found or moderated-state response instead of a successful null-metadata shell, with regression coverage for both API and SSR behavior.
