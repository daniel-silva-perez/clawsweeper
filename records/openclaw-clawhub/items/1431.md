## Summary

Keep this issue open. Current main still has the reported behavior: the publish UI can submit an organization owner for an existing skill, but the backend rejects an existing skill whose `ownerPublisherId` differs from the selected publisher as a slug collision. The alternate transfer path is still user-to-user only, and open PR #1635 is the active implementation candidate for the broader ownership-transfer design.

## Best Possible Solution

Keep this issue open and route it with #1569 and open PR #1635. The best fix is an authorized publisher-change or ownership-transfer path that handles org targets consistently across `skills.ownerPublisherId`, `skillSlugAliases`, web publish/edit behavior, HTTP API, CLI, docs, and regression tests.
