## Summary

Keep this issue open. Current `main` still matches the reported behavior: the publish UI lets a user select an organization owner for an existing skill, but the backend rejects an existing skill whose `ownerPublisherId` differs from the selected publisher as a slug collision. The alternate transfer path is still user-to-user only, and related org-transfer PRs remain unmerged or open for maintainer review.

## Best Possible Solution

Keep this issue open and route it to maintainer review alongside #1569 and open PR #1635. The best fix is an authorized publisher-change path, either in the publish/version flow or through org-aware ownership transfers, that consistently updates `skills.ownerPublisherId`, `skillSlugAliases`, web UI behavior, HTTP API, CLI, docs, and regression tests.
