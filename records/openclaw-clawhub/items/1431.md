## Summary

Keep open. Current main still matches the reported product gap, and the ClawHub cleanup policy for this pass keeps issues open. The publish UI can submit an organization publisher for an existing skill, but the backend still treats a changed `ownerPublisherId` as a slug collision; the separate transfer path remains user-to-user/personal-publisher based.

## What This Changes

The issue asks for an existing skill owner to move a skill from a personal publisher to an organization publisher without getting a slug-collision error.

## Best Possible Solution

Land a maintainer-approved publisher-target ownership-transfer design that updates the skill row, slug aliases, public URL/API behavior, CLI behavior, docs, and regression tests consistently. If republish should not move ownership, the existing-skill publish UI should stop offering owner changes and route users to the supported transfer flow.
