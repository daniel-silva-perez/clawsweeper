## Summary

Keep this issue open. Current main records download and star events, but the user-visible skill counters still depend on a delayed skill-document sync path, so the reported “metrics not updating” symptom remains plausible. Also, the repository policy for openclaw/clawhub says issues are not eligible for automatic close; only PRs that are certainly implemented on main should be proposed for close.

## Best Possible Solution

Keep this issue open for maintainer triage. Verify production cron health and event backlog, then either make the public counters read from a fresher source, shorten or trigger the skill-document stat sync with reactive-load safeguards, or document the expected delay and hourly download dedupe behavior if delayed counters are intentional.

## Risks / Open Questions

- Visible skill counters can still lag recorded events until the 6-hour skill-doc-stat-sync cron runs, so users may reasonably experience the metrics as broken or stale.
- Successful downloads can miss a metric if the best-effort recording path fails or no user/IP identity is available.
- Repeated downloads by the same API-token user or IP within one hour are intentionally deduped, which can look like a non-updating download counter.
- Production cron health and any skillStatEvents backlog were not verifiable from the read-only source review.
