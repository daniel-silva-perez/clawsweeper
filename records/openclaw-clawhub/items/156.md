## Summary

Keep this issue open. Current main does record download and star events, but the public counters still read skill-document fields that are only synced from events every 6 hours, so the user-visible report remains plausible. The related PR that attempted to reduce stat latency was closed unmerged, and the supplied ClawHub cleanup policy says issues are not eligible for automatic closure here.

## Best Possible Solution

Keep this issue open for maintainer triage. Verify production end to end, then choose a focused fix: make public counters read a fresher stat source, run skill-doc-stat-sync more frequently with acceptable reactive-load tradeoffs, document the expected delay and hourly download dedupe behavior, or open a narrower follow-up for counter freshness if that is the real remaining product gap.

## Risks / Open Questions

- Users can still perceive downloads and stars as broken because public counters are backed by skill document fields synced on the 6-hour skill-doc-stat-sync schedule.
- Repeated downloads by the same API-token user or IP within the same hour intentionally do not increment due to downloadDedupes.
- The download metric path is best-effort and catches metric errors, so a successful download can miss a metric event if that internal mutation path fails.
- Production may need operational verification for stat-event backlog, cron health, and star-count drift before deciding whether to change freshness guarantees.
