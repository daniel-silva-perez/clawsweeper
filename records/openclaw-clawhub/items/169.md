## Summary

Keep open. ClawHub policy keeps issues open, and current main still has a credible clean-install fingerprint mismatch path: the CLI hashes installed files and refuses when the registry cannot resolve that hash, while downloaded skill zips can include generated root metadata that is not part of the published-version fingerprint contract.

## What This Changes

Request that `clawhub update` recognize freshly installed or force-updated skills as clean instead of reporting `local changes (no match)` for every installed skill.

## Best Possible Solution

Define one canonical skill-content fingerprint contract shared by publish, download, install, update, sync, and resolve. Fresh installs and force-updates should immediately resolve to their installed version, while real local edits still block non-forced updates.

Fix the ClawHub CLI/backend fingerprint mismatch where `clawhub update` reports `local changes (no match)` immediately after `clawhub install` or `clawhub update --force`. Define a canonical skill-content fingerprint contract shared by publish, download, install, update, sync, and resolve. Keep the fix narrow: fresh installs and force-updates must resolve to the installed version, but real user edits must still block non-forced updates. Pay special attention to generated root `_meta.json` from skill downloads versus published `version.files`; do not broadly ignore all JSON or all metadata.
