## Summary

Keep open. ClawHub policy keeps issues open, and current main still has a credible clean-install fingerprint mismatch path: the CLI hashes installed files and refuses when the registry cannot resolve that hash, while downloaded skill zips can include generated root metadata that is not part of the published-version fingerprint contract.

## What This Changes

The issue reports that `clawhub update` reports `local changes (no match)` for freshly installed or force-updated skills instead of recognizing clean installed content.

## Best Possible Solution

Define one canonical skill-content fingerprint contract shared by publish, download, install, update, sync, and resolve. Fresh installs and force-updates should immediately resolve to their installed version, while real local edits still block non-forced updates.

## Repair Work Prompt

Fix the false `local changes (no match)` behavior for freshly installed or force-updated ClawHub skills. Define one canonical fingerprint contract across CLI install/update/sync and backend publish/download/resolve paths so clean installed content resolves to the installed version, while real local edits still block non-forced updates. Pay special attention to generated root `_meta.json`, `.clawhub/origin.json`, indexed `skillVersionFingerprints`, and recomputation from published `version.files`. Do not broadly ignore all JSON or user-editable metadata. Add regression coverage for a fresh install or force-update followed immediately by `clawhub update`/`update --all`, plus a negative case showing an actual local edit still requires `--force`. Related reports: #178 and #937. No changelog entry unless the repository convention requires one for CLI bug fixes.
