## Summary

Keep open. The ClawHub cleanup policy keeps issues open, and current main still has the credible false-positive path described in the report: update hashes installed files, resolve-by-hash only recognizes published version fingerprints, and downloaded installs can include generated root metadata that is not part of the publish-time fingerprint. Related open reports #178 and #937 describe the same failure cluster, so this should stay available for a focused fix rather than be closed as stale or implemented.

## What This Changes

The issue asks for `clawhub update` to recognize freshly installed or force-updated skills as clean instead of reporting `local changes (no match)` for every skill.

## Best Possible Solution

Define one canonical skill-content fingerprint contract shared by publish, download, install, update, sync, and resolve. A freshly installed or force-updated skill should immediately resolve to its installed version, while real local edits should still block non-forced updates; add regression tests and a maintenance/backfill step if existing registry fingerprints need repair.

## Repair Work Prompt

Fix the `clawhub update` false `local changes (no match)` behavior for clean installs and force-updates. Define and implement one canonical skill-content fingerprint contract shared by publish, download, install, update, sync, and resolve. The fix should make a fresh install followed by `clawhub update <slug>` report up to date or update cleanly, and a force-update followed by a normal update remain clean. Preserve local tamper/edit detection: do not broadly ignore all JSON, all metadata, or arbitrary generated-looking files. Investigate whether the correct boundary is excluding download-generated root `_meta.json` from installed-content fingerprints, avoiding extraction of registry metadata into installed skill folders, storing a trusted installed-content baseline in `.clawhub/origin.json`, repairing backend fingerprint rows, or a combination. Add regression tests for fresh install/update and force-update/update, plus backend resolve/download fingerprint consistency. If existing production fingerprint rows can be stale, add or update a cursor-based maintenance/backfill path. Add an Unreleased changelog fix entry in `CHANGELOG.md`. Do not change unrelated CLI behavior, moderation checks, package download behavior, or broad text-file inclusion semantics. Related refs: #178, #937.
