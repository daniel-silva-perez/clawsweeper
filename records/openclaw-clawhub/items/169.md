Keep this issue open and land one focused PR that defines a single canonical fingerprint for installed skill contents. The likely fix boundary is to ignore only generated root _meta.json metadata in local and downloaded-zip fingerprint calculations, or otherwise stop generated metadata from entering installed skill content, while preserving detection for real user-authored JSON edits. Add regressions proving a fresh install or force-update followed immediately by update is clean, and that a real local edit is still detected. Coordinate the fix with #178 and #937.

## Clownfish Work Prompt

Fix the ClawHub CLI update fingerprint false positive tracked by openclaw/clawhub#169, coordinating with #178 and #937. Current main can install or force-update a skill and then immediately report `local changes (no match)` because generated root `_meta.json` from skill downloads can be included in local/downloaded-zip fingerprints while `/api/v1/resolve` compares against stored published version files/fingerprints. Define one canonical fingerprint for installed skill contents. Prefer a narrowly scoped approach that excludes only generated root `_meta.json` from local and downloaded-zip fingerprint calculations, or otherwise prevents generated metadata from becoming installed skill content, while preserving hashing for user-authored JSON and preserving detection of real local edits. Do not weaken malware/suspicious moderation checks, lockfile/origin writes, registry auth handling, package download behavior, or unrelated plugin/package surfaces. Add regression tests for: fresh install then update is clean/up to date; force-update then update is clean/up to date; editing SKILL.md or another real text file still reports local changes; sync fallback does not diverge from update. If Convex download/fingerprint code changes, include backend tests for skill zip/download behavior and stored fingerprint compatibility. Include no unrelated refactors or lockfile churn.
