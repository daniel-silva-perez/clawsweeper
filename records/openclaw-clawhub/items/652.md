## Summary

Keep this issue open. It is an issue, and this ClawHub cleanup run only auto-closes PRs that are definitely implemented on current main. A source review also confirms current main still does not pass GitHub identity into skill backup commits: publish-time and bulk backup paths carry owner handles only, and both GitHub create-commit payloads omit `author` and `committer` metadata.

## Best Possible Solution

Keep this issue open. The best path is a maintainer decision on attribution semantics, then a future-only implementation that carries a privacy-safe GitHub author object through publish-time and bulk backup paths, adds it to both skill backup create-commit payloads, and covers the emitted payloads with focused tests. Existing backup history should stay untouched unless maintainers explicitly choose otherwise.

## Risks / Open Questions

- A fix needs a privacy-safe Git author email strategy, likely GitHub noreply addresses derived from immutable provider account ids rather than private account emails.
- Maintainers should decide author versus committer semantics: the publishing user can be the author while the GitHub App/bot remains the committer.
- Org/shared-owner and bulk-sync cases need explicit policy: attribute to the publishing actor, the version creator, the owner publisher, or keep bot attribution when identity is ambiguous.
- Existing bot-authored backup history should not be rewritten as part of this issue without a separate maintainer decision.
