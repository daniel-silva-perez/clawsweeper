## Summary

Keep this issue open. Current main still does not pass a GitHub user author identity into skill backup commits, and ClawHub cleanup policy keeps issues open unless a merged PR has certainly implemented the requested change.

## Best Possible Solution

Keep the issue open for maintainer attribution policy, then implement a future-only fix that builds a privacy-safe GitHub author object from existing GitHub identity data, passes it through publish-time and bulk backup paths, and adds it to both skill and metadata create-commit payloads. Existing backup history should stay untouched unless maintainers explicitly choose a rewrite/backfill.

## Risks / Open Questions

- A future implementation must not use private user email addresses for Git authoring; prefer a GitHub noreply-style address derived from immutable providerAccountId plus current login.
- Maintainers should decide whether the publishing actor, version creator, owner user, or bot should be the Git author, especially for org/shared-owner and bulk-sync cases.
- The GitHub App/bot should probably remain the committer even if the publishing user becomes the author, to keep the write authority clear.
