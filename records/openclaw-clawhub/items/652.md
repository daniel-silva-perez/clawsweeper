## Summary

Keep open. This is a valid ClawHub issue, and current main still creates skill backup commits without threading any Git author or committer identity into the GitHub commit payloads. The affected code path is clear, but the remaining decision is attribution and privacy policy, especially how to represent personal versus organization owners and which email address is safe to publish in Git history.

## What This Changes

Request that skill addition/update commits mirrored to the skills repository use the owner’s GitHub author details instead of the ClawHub bot/App identity.

## Best Possible Solution

Define a maintainer-approved attribution policy, then pass a privacy-safe Git author object through publish-time and bulk backup paths into both skill and metadata Git commit payloads, with explicit bot fallback behavior and coverage for personal owners, org/publisher ownership, missing identity, and bulk sync.

## Risks / Open Questions

- Maintainers still need to choose whether Git author attribution should represent the publishing actor, the personal owner, an organization/publisher owner, or a bot fallback for bulk sync and restore paths.
- Using stored user email directly as the Git author email could expose private addresses; a safe implementation likely needs a GitHub noreply-style address or another approved public identity source.
- ownerHandle alone is mutable and insufficient as a Git author identity, especially for renamed users and organization publishers.
