## Summary

Keep this issue open. The ClawHub cleanup policy for this repository keeps issues open, and current main still does not implement GitHub user attribution for skill backup commits: the publish and bulk sync paths pass only an owner handle, while the GitHub create-commit payloads omit `author` and `committer` metadata.

## Best Possible Solution

Keep this issue open and implement future-only commit attribution in the GitHub backup path. Carry the publishing user's GitHub identity through publish-time backups and bulk sync, derive a privacy-safe Git author object, decide whether committer stays as the app/bot, include the metadata in both create-commit payloads, and add focused tests for the emitted GitHub payloads.

## Risks / Open Questions

- Implementing this needs a privacy-safe Git author email strategy, likely GitHub noreply addresses rather than private account emails.
- Maintainers should choose explicit semantics for author versus committer; the publishing user can be the author while the GitHub App or bot remains the committer.
- Existing bot-authored backup history should not be rewritten without a separate maintainer decision.
