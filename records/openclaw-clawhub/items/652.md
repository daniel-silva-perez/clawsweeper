## Summary

Keep open. ClawHub cleanup policy keeps issues open, and the current `main` code still creates skill backup commits without passing a Git author or committer identity. The prior Codex comments are consistent with the current source: the implementation path is clear, but maintainers need to decide attribution semantics and privacy-safe email handling before this should become an automated fix.

## What This Changes

Request skill addition and update commits mirrored to the skills repository to use the publishing owner’s GitHub identity as the Git commit author instead of the ClawHub bot identity.

## Best Possible Solution

Define a maintainer-approved attribution policy, then thread a privacy-safe Git author object through publish-time and bulk backup paths into both skill and metadata Git commit payloads, with tests for personal owners, org/publisher ownership, missing identity, bulk sync, and bot fallback behavior.

## Risks / Open Questions

- Maintainers still need to choose whether Git author attribution should represent the publishing actor, the personal owner, an organization/publisher owner, or a bot fallback for bulk sync and restore paths.
- Using stored OAuth email directly as a Git author email could expose private addresses; a safe implementation likely needs a GitHub noreply-style address, explicit consent, or another maintainer-approved source.
- Because the backup repository is written through a GitHub App token, any author/committer change should be checked against GitHub API behavior and the desired audit trail.
