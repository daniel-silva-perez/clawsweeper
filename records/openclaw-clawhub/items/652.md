## Summary

Keep open. This is a valid ClawHub issue and current main still creates skill backup commits without passing a Git author or committer identity, so the mirrored commits will continue to use the GitHub App/bot identity. The remaining work needs maintainer policy on attribution semantics and privacy-safe author email handling before implementation.

## What This Changes

Request skill addition/update backup commits in the skills repository use the owner’s GitHub account details as the Git commit author instead of the ClawHub bot identity.

## Best Possible Solution

Define the attribution policy first, then thread a privacy-safe GitHub author object through publish-time and bulk backup paths into both skill and metadata Git commit payloads, with explicit bot fallback behavior and tests for personal owners, org/publisher ownership, missing identity, and bulk sync.

## Risks / Open Questions

- Maintainers still need to choose whether Git author attribution should use the publishing actor, the personal owner, an organization/publisher owner, or a bot fallback for bulk sync and restore paths.
- Using stored user email directly as a Git author email could expose private addresses; a safe implementation likely needs a GitHub noreply-style address or another maintainer-approved source derived from GitHub identity.
- ownerHandle is not sufficient as an author identity because it can represent a publisher/org handle or fallback value rather than the authenticated GitHub user who should be attributed.
