## Summary

Keep this issue open. The ClawHub cleanup policy for this review says issues are not auto-closed, and current main still does not carry the publishing user's GitHub identity into skill backup commit author metadata. The prior ClawSweeper comment is still accurate against current main SHA 4e13e729fb07bc8cf35e58e554ef557fca0b3713.

## Best Possible Solution

Keep this issue open and implement future-only commit attribution in the GitHub skill backup path. Carry the publishing user's GitHub identity through both publish-time and bulk sync backups, construct a privacy-safe Git author object, include author and deliberately chosen committer metadata in the GitHub create-commit payloads, and add focused tests asserting the payload includes the intended author data.

## Risks / Open Questions

- Author attribution should avoid leaking private email addresses; a GitHub numeric-id noreply address or another explicitly privacy-safe author email is likely needed.
- Maintainers should decide whether the publishing user should be author only while the GitHub App remains committer, or whether committer metadata should also change.
- Existing bot-authored backup history should remain unchanged unless maintainers separately choose a history rewrite plan.
