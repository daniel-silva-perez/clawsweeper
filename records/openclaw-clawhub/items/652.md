## Summary

Keep open. This is a valid ClawHub issue and current main still creates GitHub backup commits without any user author or committer identity; repository cleanup policy also keeps ClawHub issues open. The implementation surface is clear, but the remaining decision is maintainer policy for attribution and privacy-safe Git author email handling.

## What This Changes

Use the skill owner’s GitHub account details as the author for skill addition/update commits in the skills backup repository instead of the bot identity.

## Best Possible Solution

Define a maintainer-approved attribution policy, then pass a privacy-safe Git author object through publish-time and bulk backup paths into both skill and metadata Git commit payloads, with explicit bot fallback behavior and tests for personal owners, org/publisher ownership, missing identity, and bulk sync.

## Risks / Open Questions

- Maintainers still need to decide whether Git author attribution should represent the publishing actor, a personal owner, an organization/publisher owner, or a bot fallback for bulk sync and restore-style paths.
- Using stored user email directly as a Git author email could expose private addresses in public Git history; a safe implementation likely needs a GitHub noreply-style address or another approved source.
- ownerHandle alone is not a stable author identity because handles can be missing, renamed, or represent publishers/orgs rather than the user who performed the publish.
