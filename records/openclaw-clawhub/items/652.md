## Summary

Keep open. Repository policy for openclaw/clawhub does not auto-close issues, and current main still does not pass a Git author or committer identity into the skill backup commits. The remaining work is real, but it needs a maintainer attribution/privacy decision before implementation.

## What This Changes

The issue asks for openclaw/skills skill addition and update commits to use the owner’s GitHub identity as the Git author instead of the ClawHub bot identity.

## Best Possible Solution

Define a maintainer-approved attribution policy, then thread a privacy-safe GitHub author object through both publish-time and bulk backup paths into both skill and metadata Git commit payloads, with clear bot fallback behavior and tests covering personal, org, missing-identity, and bulk-sync cases.

## Risks / Open Questions

- Using users.email directly as the Git author email could expose private email addresses; a fix needs a maintainer-approved privacy-safe source, likely GitHub noreply-style attribution derived from immutable providerAccountId plus current login.
- Maintainers still need to choose whether commits should attribute the publishing actor, the owning personal publisher, an organization owner/member, or a bot fallback for org and bulk-backup cases.
- Existing openclaw/skills history should remain future-only unless maintainers explicitly decide to rewrite or backfill historical commits.
