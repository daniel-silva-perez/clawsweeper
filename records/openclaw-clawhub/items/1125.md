## Summary

Keep this PR open. Current main still does not implement install-time OS requirement checks, so the PR is not obsolete or implemented-on-main. The contribution remains relevant, but the existing discussion points to real changes needed before merge: optional metadata parsing, OS vocabulary normalization, requested-version metadata handling, tests, and explicit `--force` policy.

## Best Possible Solution

Keep the PR open and revise it. The maintainable path is to parse `metadata` as optional, normalize host platform aliases to the documented `metadata.os` vocabulary, validate requirements against the resolved version when `--version` is used, decide whether `--force` should bypass OS mismatch separately from suspicious-skill handling, and add focused `cmdInstall` tests for allowed, rejected, forced, missing-metadata, and version-specific metadata cases.

## Risks / Open Questions

- Merging the PR as-is could reject supported macOS or Windows installs if raw Node platform values such as `darwin` or `win32` are compared against documented values such as `macos` or `windows`.
- Making `metadata` required in the CLI response schema could break installs for responses that omit that field.
- Checking latest-version metadata before resolving `--version` can allow or reject the wrong version when OS requirements differ by release.
- The proposed `--force` bypass for OS mismatch may be acceptable, but it needs explicit maintainer policy and tests.
