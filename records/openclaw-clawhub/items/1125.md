## Summary

Keep this PR open. Current main does not already implement install-time OS compatibility checks, and the proposed patch still has correctness issues around OS alias matching, version-specific metadata, and response-schema compatibility.

## What This Changes

The PR adds CLI parsing of skill platform metadata, rejects `clawhub install` when the host OS is not listed in `metadata.os`, and expands `--force` help text to mention bypassing OS mismatch checks.

## Best Possible Solution

Revise the PR so install compatibility uses the same normalized OS vocabulary as the docs/UI, validates platform metadata for the selected version, preserves backward-compatible parsing when metadata is absent, and adds focused CLI tests for compatible, incompatible, alias, `--version`, and `--force` cases.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Normalize OS aliases before rejecting installs:** `packages/clawhub/src/cli/commands/skills.ts:100`
  - body: The new guard compares `metadata.os` directly with `platform()`, but documented skill metadata uses values like `macos` and `windows` while Node reports `darwin` and `win32`. On those hosts, supported skills can be rejected unless both sides are normalized before the membership check.
  - confidence: 0.94
- **[P2] Validate OS metadata for the resolved version:** `packages/clawhub/src/cli/commands/skills.ts:100`
  - body: The check runs before `versionFlag` is resolved and uses the skill detail response, whose metadata comes from the latest version. Installing an older version after the latest release changes its OS list can reject a compatible version or allow an incompatible one; validate metadata for the selected version instead.
  - confidence: 0.88
- **[P2] Keep metadata optional in the response schema:** `packages/clawhub/src/schema/schemas.ts:203-206`
  - body: Adding `metadata` as a required field makes the CLI reject registries or older API responses that omit this newly consumed key, even when no OS restriction exists. The install code already treats `metadata` as optional, so the schema should allow the field to be absent as well as null.
  - confidence: 0.78
