## Summary

Keep this PR open. Current main still does not implement install-time OS compatibility checks, so the PR is not obsolete, but the supplied diff is not ready to treat as implemented because it compares raw Node platform values to documented metadata values, checks latest-version metadata before resolving requested versions, and makes CLI response metadata required.

## Best Possible Solution

Keep the PR open and revise it. The maintainable path is to parse platform metadata as optional and nullable, normalize host platform aliases to the documented metadata vocabulary, validate metadata for the resolved version when --version is used, decide whether --force may bypass OS mismatch, and add focused cmdInstall tests for those compatibility paths.

## Risks / Open Questions

- Merging the supplied diff as-is can incorrectly reject macOS and Windows users because Node reports darwin/win32 while ClawHub metadata is documented as macos/windows-style values.
- Checking latest-version metadata before resolving --version can allow or reject the wrong release when OS requirements differ by version.
- Making metadata required in ApiV1SkillResponseSchema could break installs when an API response omits the field; it should be optional and nullable.
- Whether --force should bypass OS incompatibility is a product/safety policy decision that should be explicit and tested.
- There are no focused install tests in the inspected current-main cmdInstall test block for allowed, rejected, forced, missing-metadata, or requested-version OS compatibility paths.
