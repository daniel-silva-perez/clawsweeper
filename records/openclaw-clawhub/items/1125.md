## Summary

Keep this PR open. Current main still does not implement the intended OS compatibility check during `clawhub install`, and the PR remains a plausible contribution but needs revision for schema optionality, OS vocabulary normalization, version-specific metadata, and focused tests. Security review found no unexplained workflow, dependency, lockfile, secret, or downloaded-artifact changes in the PR diff, but the changed install-flow semantics around `--force` still need maintainer product judgment.

## Best Possible Solution

Keep the PR open and revise it. The maintainable path is to parse `metadata` as optional in the CLI schema, normalize host platform aliases to the documented `metadata.os` vocabulary, validate OS requirements against the resolved version when `--version` is used, decide whether `--force` should bypass OS mismatch independently from suspicious-skill handling, and add focused `cmdInstall` tests for allowed, rejected, forced, missing-metadata, and version-specific metadata cases.

## Risks / Open Questions

- Merging the PR as-is could reject supported macOS or Windows installs if raw Node platform values like `darwin` or `win32` are compared to documented metadata values like `macos` or `windows`.
- Making `metadata` required in the CLI response schema could break install flows for responses or fixtures that omit the field.
- Checking the latest-version metadata before resolving `--version` can allow or reject the wrong install when an older requested version has different OS requirements.
- The PR changes `--force` to bypass OS mismatch; that policy may be acceptable, but it needs explicit maintainer decision and tests because it changes install safety semantics.
