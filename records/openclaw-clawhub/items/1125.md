## Summary

Keep this PR open. Current main does not implement install-time OS requirement checks, and the PR still targets a real missing behavior. The contribution needs revision before merge because the current diff validates raw Node platform names against documented metadata values, checks latest-version metadata before resolving an explicit version, makes CLI response metadata required, and changes install safety behavior around --force without tests or maintainer policy confirmation.

## Best Possible Solution

Keep the PR open and revise it. The maintainable path is to parse metadata as optional and nullable, normalize host platform aliases to the documented metadata.os vocabulary, validate compatibility against the resolved version when --version is used, decide explicitly whether --force should bypass OS mismatches, and add focused cmdInstall tests for allowed, rejected, forced, missing-metadata, and version-specific metadata cases.

## Risks / Open Questions

- Merging the PR as-is could reject supported macOS or Windows installs because Node reports darwin and win32 while documented skill metadata uses macos and windows-style values.
- Checking latest skill metadata before resolving --version can allow or reject the wrong version when OS requirements differ across releases.
- Making metadata required in ApiV1SkillResponseSchema would be a backward-compatibility risk for API responses that omit the field.
- Changing --force to bypass OS mismatch is an install-safety policy decision and should be covered by explicit tests and maintainer agreement.
