## Summary

Keep this PR open. Current main still does not implement install-time OS requirement checks, and the PR’s intended behavior is not obsolete or already shipped. The contribution remains relevant, but it needs revision before merge because the current approach should normalize platform aliases, validate the resolved version’s metadata, make API metadata parsing optional/nullable, clarify `--force` policy, and add focused install tests.

## Best Possible Solution

Keep the PR open and revise it. The maintainable path is to parse skill response metadata as optional and nullable, normalize host platform aliases to the documented `metadata.os` vocabulary, validate OS compatibility against the resolved version when `--version` is used, decide explicitly whether `--force` should bypass OS mismatches, and add focused `cmdInstall` tests for allowed, rejected, forced, missing-metadata, and version-specific metadata cases.

## Risks / Open Questions

- Merging the PR as-is could reject supported macOS or Windows users because Node reports `darwin` and `win32`, while skill metadata is documented as `macos` and `windows`/`linux` style values.
- Checking latest skill metadata before resolving `--version` can allow or reject the wrong version when OS requirements differ across releases.
- Making `metadata` required in the CLI response schema would be a backward-compatibility risk for responses that omit the field; it should be optional and nullable.
- Expanding `--force` to bypass OS mismatch changes user-safety policy beyond overwrite/suspicious-skill handling and needs explicit maintainer approval.
