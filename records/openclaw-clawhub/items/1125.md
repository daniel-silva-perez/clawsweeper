## Summary

Keep open. Current main does not implement install-time OS compatibility checks, and the PR still has blocking correctness issues around OS alias normalization, requested-version metadata, and schema compatibility.

## What This Changes

The PR adds CLI parsing for skill metadata, checks `metadata.os` during `clawhub install`, and expands the `--force` help text to describe bypassing OS mismatch checks.

## Best Possible Solution

Revise this PR so install compatibility checks use optional/nullable metadata, normalize documented OS names against Node platform values, validate the resolved version's metadata, define the intended `--force` semantics, and add focused `cmdInstall` coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Normalize platform aliases before rejecting installs:** `packages/clawhub/src/cli/commands/skills.ts:101`
  - body: This compares `metadata.os` directly to `platform()`, but documented skill metadata uses values like `macos` and `windows` while Node reports `darwin` and `win32`. Supported macOS and Windows skills would be rejected unless both sides are normalized first.
  - confidence: 0.95
- **[P2] Validate metadata for the requested version:** `packages/clawhub/src/cli/commands/skills.ts:101`
  - body: The guard runs before `versionFlag` is resolved, and the skill detail endpoint exposes metadata from the latest version. Installing an older version can therefore be allowed or rejected based on the wrong release's OS requirements.
  - confidence: 0.9
- **[P2] Keep metadata optional in the CLI schema:** `packages/clawhub/src/schema/schemas.ts:203-206`
  - body: The new schema requires the `metadata` key, so an otherwise valid response that omits metadata will fail parsing before install can proceed. Match the optional/nullable handling used for fields that may be absent.
  - confidence: 0.86
