## Summary

Keep this PR open. Current main still does not implement install-time OS requirement checks, and the PR is not obsolete or already shipped. The contribution remains relevant, but it still needs revision before merge because the current diff compares raw Node platform values against the documented metadata vocabulary, checks latest-version metadata before resolving requested versions, and lacks focused install tests.

## Best Possible Solution

Keep the PR open and revise it. The maintainable path is to parse `metadata` as optional, normalize host platform aliases to the documented `metadata.os` vocabulary, validate against the resolved version when `--version` is used, decide whether `--force` should bypass OS mismatch separately from suspicious-skill handling, and add focused `cmdInstall` tests for allowed, rejected, forced, missing-metadata, and version-specific metadata cases.

## Risks / Open Questions

- Merging the PR as-is could incorrectly reject macOS or Windows users because Node reports `darwin` and `win32`, while skill metadata is documented as `macos` and `windows`/`linux` style values.
- Checking the latest skill metadata before resolving `--version` can allow or reject the wrong version when OS requirements differ across releases.
- Making `metadata` required in the CLI response schema would be a compatibility risk for responses that omit the field; it should be optional and nullable.
- The proposed `--force` behavior for OS mismatches needs explicit maintainer policy and tests, since it would expand `--force` beyond overwrite/suspicious-skill behavior.
