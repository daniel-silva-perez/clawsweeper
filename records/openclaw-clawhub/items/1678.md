## Summary

Keep open. The repository cleanup policy keeps ClawHub issues open, and current main still appears to leave the reported package publish path unresolved: `--source-path` is applied after GitHub shorthand sources are fetched, while package owner selection remains explicit through `--owner` rather than inferred from the package scope or source repository.

## What This Changes

The issue requests a working `clawhub package publish` flow for org-owned packages located in monorepo subdirectories, covering both org namespace selection and `owner/repo@ref --source-path` handling.

## Best Possible Solution

Provide a reliable monorepo publish path by applying a validated `--source-path` to GitHub shorthand/URL sources before files are fetched and inspected, then either clearly document explicit `--owner <org>` as the org-publish mechanism or implement a maintainer-approved, permission-checked owner inference rule.

## Repair Work Prompt

Fix the package publish monorepo GitHub source-path bug without changing repository ownership semantics. Apply a validated `--source-path` to GitHub shorthand/URL sources before `fetchGitHubSource` downloads/extracts files, so `clawhub package publish owner/repo@ref --source-path plugins/foo` inspects `plugins/foo/package.json` and `plugins/foo/openclaw.plugin.json`. Preserve source attribution in the publish payload. Add regression coverage around GitHub shorthand plus `--source-path`, preferably in `packages/clawhub/src/cli/commands/packages.test.ts` or the existing GitHub source tests with mocked archive fetches. Also update CLI docs/help examples to show org monorepo publishing with explicit `--owner <org>` if appropriate. Do not infer org ownership from scoped package names or `--source-repo` in this repair PR unless a maintainer explicitly expands the task. Do not touch unrelated package publishing behavior, trusted publishing, or Convex schema.
