## Summary

Keep open. This is a ClawHub issue, and the repository-specific cleanup policy for this run permits auto-close only for pull requests that current main certainly implements. Current main has narrowed the declared-env-var part of this false-positive pattern, but it still preserves file-read plus network-send as `suspicious.potential_exfiltration`, still renders the suspicious banner when `moderationInfo.isSuspicious` is true, and does not prove the two named production listings have been reviewed and cleared.

## Best Possible Solution

Keep this issue open for maintainer moderation review. A maintainer should inspect the two ClawHub listings and upstream CLI repositories, then use the existing manual override path if the API-key and user-file behavior is expected. If ClawHub should reduce future API-client false positives, track a narrower scanner-design change around declared input/config path semantics rather than broadly suppressing file-read plus network-send findings.
