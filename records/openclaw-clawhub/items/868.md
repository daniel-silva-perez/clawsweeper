## Summary

Keep this issue open. Current main still has the reported failure shape: sync publishes selected skills sequentially, `cmdPublish` rethrows publish failures, and the backend still returns the reported slug-conflict message for another owner's skill, so one conflict can abort the batch.

## What This Changes

The issue requests that `clawhub sync` handle a slug conflict for one local skill without preventing other selected skills from updating.

## Best Possible Solution

Implement a narrow CLI sync change that recognizes the backend slug-taken publish error for a selected skill, records that skill as failed or skipped with the existing owner URL when present, continues publishing the remaining selected skills, and ends with a clear partial-success summary while preserving backend slug uniqueness.

## Repair Work Prompt

Fix `clawhub sync` for the reported slug-conflict batch abort. In `packages/clawhub/src/cli/commands/sync.ts`, catch publish failures per selected skill, detect only the known backend message shape `Slug is already taken. Choose a different slug.` optionally followed by `Existing skill: /owner/slug`, record that skill as failed or skipped, continue publishing the rest, and print a clear final summary showing uploaded count and slug-conflict failures with the existing owner URL when present. Do not change backend slug uniqueness, owner-scoped publishing, the `/api/v1/resolve` contract, or general `cmdPublish` behavior unless a tiny helper is needed. Do not hide unexpected publish errors; they should still abort or be surfaced according to existing semantics. Add focused Vitest coverage in `packages/clawhub/src/cli/commands/sync.test.ts` proving one slug-taken publish failure does not prevent a later selected skill from publishing and that the final output is not a plain full-success message. Related refs: openclaw/clawhub#868 and the existing Codex review comment on this issue.
