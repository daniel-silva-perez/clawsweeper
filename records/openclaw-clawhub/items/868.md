## Summary

Keep open. Current main still has the reported failure path: `sync` classifies changed local skills as publish candidates, calls `cmdPublish` without a per-skill error boundary, and `cmdPublish` rethrows the backend slug-conflict error, so one conflicting skill can abort the remaining batch. This is also an issue in `openclaw/clawhub`, where the cleanup policy keeps issues open rather than auto-closing them.

## What This Changes

The issue asks for `clawhub sync` to avoid failing the whole sync batch when one selected skill publish hits a backend slug conflict.

## Best Possible Solution

Add focused CLI sync conflict handling: detect the known slug-taken publish error for one selected skill, record that skill as failed or skipped with the existing owner URL when available, continue publishing later selected skills, print an accurate partial-success summary, and cover the continuation behavior in the sync test suite while preserving backend ownership rules.

## Repair Work Prompt

Scope the change to CLI batch behavior: detect the backend message shape `Slug is already taken. Choose a different slug.` and preserve any `Existing skill: /owner/slug` detail, record that selected skill as failed or skipped, continue with later selected skills, and replace the final `Uploaded N skill(s).` message with an accurate full/partial result. Unknown publish errors should remain visible and should not be silently swallowed. Do not relax backend slug uniqueness or ownership checks. Add a regression test in `packages/clawhub/src/cli/commands/sync.test.ts` where `mockCmdPublish` rejects for one selected skill with the slug-taken message and verify a later selected skill is still published and the output reports the conflict/partial result. Keep changes focused; no changelog is required unless maintainers request one.
