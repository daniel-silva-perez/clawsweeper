## Summary

Keep this issue open. Current main has the right account model for GitHub username changes by using the immutable GitHub numeric ID and syncing mutable profile fields after sign-in, but the reported failure is an OAuth verifier/session mismatch before login completes plus a production relink request. That cannot be proven fixed from source alone, and the openclaw/clawhub cleanup policy only allows auto-closing PRs that are certainly implemented on main, not issues.

## What This Changes

The issue requests recovery for a user who cannot log in with GitHub after changing their GitHub username and seeing a `convexAuthOAuthVerifier` token mismatch.

## Best Possible Solution

Keep the stable GitHub numeric-ID identity model, but provide a maintainer-supported recovery path for renamed-account OAuth failures: inspect the affected production `users` and `authAccounts` rows, verify the GitHub numeric `providerAccountId`, clear or repair stale/duplicate/deactivated identity state if needed, and add focused callback/verifier coverage if a code defect is found.
