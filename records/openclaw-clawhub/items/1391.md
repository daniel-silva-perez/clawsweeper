## Summary

Keep open. This is a ClawHub issue, and the repository policy for this review lane keeps issues open; substantively, current main still only partially addresses the artifact false positives because package UI/CLI paths ignore `.git/`, while skill web/API/GitHub publish paths and package multipart API paths still use macOS-junk-only filtering and do not cover `__pycache__/`, `.pyc`, or `.pyo`.

## What This Changes

Request to avoid false-positive suspicious scan findings from standard Git and Python generated artifacts by excluding or better distinguishing `.git/`, `__pycache__/`, `.pyc`, and `.pyo` during publish/security scanning.

## Best Possible Solution

Keep this issue open and, after maintainer/security sign-off, land a narrow shared publish-artifact cleanup helper used consistently before storage where possible and before static/LLM/VirusTotal scanner input across web skill upload, API JSON/multipart skill publish, GitHub import, CLI skill publish, package upload, and package multipart publish. Cover `.git/`, `__pycache__/`, `.pyc`, and `.pyo`, add focused tests for each affected path, and document the cleanup contract if users will see it.
