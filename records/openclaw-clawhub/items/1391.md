## Summary

Keep open. This is an open ClawHub issue, which this cleanup lane should not auto-close, and the current source still only partially handles the reported artifact false positives: package and CLI publish paths ignore some VCS/dotfile artifacts, but web upload, API multipart publish, backend publish filtering, and GitHub import still rely on macOS-junk-only filtering before storage and scan input.

## What This Changes

The issue requests that skill publishing exclude or correctly handle standard `.git/` and Python `__pycache__/` artifacts so clean skill projects are not flagged as suspicious.

## Best Possible Solution

Introduce a small shared publish-artifact cleanup helper used by all skill publish entry points before storage and static/LLM moderation, covering `.git/`, `__pycache__/`, `.pyc`, and `.pyo` while preserving normal source-file scanning and existing macOS junk handling.
