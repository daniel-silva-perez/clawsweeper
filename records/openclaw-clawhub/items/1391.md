## Summary

Keep open. This is an open ClawHub issue, and the repository-specific cleanup policy only permits auto-closing ClawHub pull requests that are certainly implemented on current main. Substantively, current main still lacks a shared artifact exclusion policy for `.git/`, `__pycache__/`, `*.pyc`, and `*.pyo` across skill/package upload, API normalization, persistence, static scan, and VirusTotal payload construction.

## Best Possible Solution

Keep this issue open. Implement a shared publish artifact exclusion helper used by CLI skill/package collection, web folder/archive expansion, HTTP multipart and JSON publish normalization, server-side release persistence, static scan input construction, and VirusTotal zip construction. Include `.git/`, `__pycache__/`, `*.pyc`, and `*.pyo`; preserve `.clawhubignore` behavior; add focused tests for CLI, web, API, multipart, JSON, static-scan, and VirusTotal payload paths; document `.clawhubignore` or `.gitignore` as the interim workaround where applicable.

## Risks / Open Questions

- Direct JSON/API and multipart package publish paths can still persist artifact paths before static scan and VirusTotal archive construction.
- The `.git/` behavior is inconsistent: CLI and some web package helpers filter it, but server-side package normalization does not enforce it for all clients.
- Python cache and bytecode artifacts are not covered by a built-in shared ignore policy, so the reported false-positive class can recur.
