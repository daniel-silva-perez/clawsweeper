## Summary

Keep open. Repository policy for openclaw/clawhub keeps issues open, and current main still does not implement a shared artifact exclusion policy for the reported `.git/`, `__pycache__/`, `*.pyc`, and `*.pyo` false-positive class. There is partial `.git/` filtering in CLI and package-upload client code, but web skill upload, server-side publish normalization, static scan input construction, and VirusTotal ZIP construction can still receive publish files that survive earlier path-specific filtering.

## Best Possible Solution

Keep this issue open. Implement a shared artifact exclusion helper used by CLI skill collection, web folder/archive expansion, HTTP/API publish normalization, server-side skill/package publish persistence, static scan input construction, LLM package evaluation, and VirusTotal ZIP construction. Exclude `.git/`, `__pycache__/`, `*.pyc`, and `*.pyo`; preserve `.clawhubignore` and documented `.gitignore` behavior; add focused tests across CLI, web skill upload, package upload, API/server publish, static scan, and VT payload paths; document the interim ignore-file workaround where applicable.

## Risks / Open Questions

- False-positive scan inputs can still reach current main through web skill upload, direct/API skill publish, and server-side package publish paths.

- A fix should avoid weakening security scanning of real source files while excluding standard generated VCS/cache artifacts consistently before persistence, static scanning, LLM evaluation, and VT ZIP construction.
