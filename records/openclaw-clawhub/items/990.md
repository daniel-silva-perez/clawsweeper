## Summary

Keep open. This is an open ClawHub issue, and the repository-specific cleanup policy for this review says ClawHub issues should remain open unless they are PRs already implemented on main. Substantively, current main exposes suspicious/moderation state through the v1 JSON API and docs, but the requested static skill-page metadata path is still missing.

## Best Possible Solution

Keep this issue open for maintainer product handling. A concrete path is to pass public moderationInfo from the skill page loader into the route head metadata path, extend buildSkillMeta or the route head output with stable clawhub:suspicious metadata plus a public verdict/reason summary where appropriate, add route/head tests for flagged and clean skill pages, and document when agents should use static metadata versus the existing v1 detail and moderation APIs.

## Risks / Open Questions

- Agents or tools that consume only static skill-page HTML still lack a stable head metadata field for suspicious status.
- The existing v1 API moderation object should be preserved while adding static metadata, because API consumers may already depend on moderation.isSuspicious and the dedicated moderation endpoint.
- The public detail API includes moderation only for flagged skills or owner-visible cases, so any static metadata design should define whether clean public pages emit an explicit false value or omit the tag.
