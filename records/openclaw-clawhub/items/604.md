## Summary

Keep this open. ClawHub cleanup policy only permits automated closure of PRs that are certainly implemented on current main, and this item is an issue. Source review also supports a real account-recovery/admin-data problem: inactive/deactivated users are hidden from public profile lookup, deleted/banned users are blocked during OAuth, published skills can remain public through publisher/skill records, and current admin restore/reclaim endpoints still depend on the same active handle lookup. The related recovery PR #1133 remains open and is not present on current main.

## Best Possible Solution

Keep this open for maintainer support. A maintainer with production access should inspect samiralibabic’s user/auth/publisher/skill records, determine whether the account is deleted, deactivated, banned, duplicated by OAuth, orphaned behind a publisher, or under moderation, then restore or relink ownership and clean duplicate/orphaned entries as appropriate. The durable code path is to land PR #1133 or an equivalent audited admin recovery flow that can operate when active handle lookup fails and can safely restore only eligible false-positive skills.
