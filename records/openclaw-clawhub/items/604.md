## Summary

Keep this open. The repository policy supplied for openclaw/clawhub only permits automated closure of PRs that are certainly implemented on current main, and this item is an issue. Source review also supports a real account-recovery/admin-data problem: public profile lookup hides inactive users, OAuth blocks deleted/deactivated/banned users, skills can still exist under owner/publisher records, and the current admin restore/reclaim HTTP endpoints still begin with the same active public handle lookup. The related recovery PR #1133 is open and its proposed liftModerationHold path is not present on current main.

## Best Possible Solution

Keep this open for maintainer support. A maintainer with production access should inspect samiralibabic’s user/auth/publisher/skill records, determine whether the account is deleted, deactivated, banned, duplicated by OAuth, orphaned behind a publisher, or under moderation, then restore or relink ownership and clean accidental duplicate/orphaned entries as appropriate. The durable code path is to review and land PR #1133 or an equivalent audited admin recovery flow that can operate when active handle lookup fails and can safely restore only eligible false-positive skills.
