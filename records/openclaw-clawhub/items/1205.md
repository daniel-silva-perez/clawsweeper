## Summary

Keep open. Although the reporter confirmed their immediate production publish recovered, current main still has the same Convex failure shape: both owner-publisher digest helpers can call `.paginate()` repeatedly inside one internal mutation, and the scheduled internal mutations do not accept continuation cursors. This is also a ClawHub issue, and the repository policy for this cleanup keeps issues open unless a fix has landed and maintainers consolidate them.

## What This Changes

The issue reports that `clawhub publish` can fail when owner-publisher search digest sync runs multiple Convex paginated queries inside one backend function invocation.

## Best Possible Solution

Land one focused Convex backend fix that makes the owner-publisher package and skill digest internal mutations process one indexed page per invocation, accept cursor and optional batch-size arguments, self-schedule continuation when more rows remain, and add multi-page regression tests before consolidating the related publish/import reports.

## Repair Work Prompt

Fix the Convex owner-publisher digest sync pagination bug reported by this issue and related reports #1170, #1171, #1175, #1176, #1182, #1184, #1195, #1197, and #1201. In `convex/functions.ts`, change `syncPackageSearchDigestsForOwnerPublisherIdInternal` and `syncSkillSearchDigestsForOwnerPublisherIdInternal` so each invocation performs at most one `.paginate()` call, accepts cursor and optional batch-size args, processes that page, and uses `ctx.scheduler.runAfter(0, ...)` to schedule the same internal mutation with the continuation cursor when `isDone` is false. Preserve the existing publisher-trigger split that schedules package and skill digest rebuilds separately. Keep indexed queries; do not replace pagination with unbounded `.collect()`. Add focused multi-page tests in `convex/functions.test.ts` for both package and skill digest helpers proving the first invocation schedules continuation and does not call paginate more than once. Update generated Convex files only if required by codegen. Do not change unrelated publish, auth, package parsing, or UI behavior. No changelog entry is needed unless the repo convention requires one for backend bug fixes.
