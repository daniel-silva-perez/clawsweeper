## Summary

Keep open. ClawHub cleanup policy keeps issues open, and current main still contains the backend pagination hazard reported here: the owner-publisher skill and package digest rebuild helpers can call `.paginate()` repeatedly within one internal mutation, while the scheduled mutation APIs do not accept continuation cursors.

## What This Changes

The issue reports a `clawhub publish` failure caused by `syncSkillSearchDigestsForOwnerPublisherId` running multiple Convex paginated queries during owner-publisher digest sync.

## Best Possible Solution

Implement cursor-based, one-page-per-invocation owner-publisher digest rebuilds for both package and skill search digests, with continuation scheduling when `isDone` is false and tests that simulate multi-page rebuilds.

## Repair Work Prompt

Fix the owner-publisher digest sync pagination bug reported across the publish/import pagination cluster. In `convex/functions.ts`, change the owner-publisher package and skill digest rebuilds so each internal mutation performs at most one `.paginate()` call, accepts cursor/batch-size continuation arguments, and self-schedules the next page when `isDone` is false. Preserve the existing trigger behavior that schedules package and skill digest rebuilds separately, but pass continuation state through the internal mutation args. Add focused tests in `convex/functions.test.ts` that simulate multi-page package and skill rebuilds and assert continuation scheduling instead of a loop that paginates twice in one invocation. Do not replace pagination with unbounded `.collect()`, do not broaden CLI behavior unless required by the backend contract, and keep the change scoped to the Convex publish/search digest path. Related refs: #1170, #1171, #1175, #1176, #1182, #1184, #1195, #1197, #1201, and this issue.
