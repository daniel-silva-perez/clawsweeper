## Summary

Keep this issue open. The requester is asking maintainers to hard-delete a specific production registry skill and its version history after an accidental config upload. Current main already has admin-only hard-delete machinery, while owner-facing CLI/API delete is intentionally soft-delete; the remaining work is maintainer/operator verification and production data action, not conservative repository cleanup. The ClawHub review policy also says issues should stay open unless maintainers handle them manually.

## Best Possible Solution

Keep this issue open for maintainer/operator handling: verify ownership of fm7077/easyimage-uploader, advise the requester to rotate any exposed credentials from the uploaded config, run the existing admin hard-delete flow if appropriate, and close manually with the action taken. If maintainers want owners to request or perform irreversible hard-deletes themselves, that should be designed as a separate product/security flow rather than handled by this cleanup pass.
