## Summary

Keep open. Current main still lacks a deterministic CLI/API path for exact-slug or slug-prefix registry enumeration; the existing CLI search remains relevance-ranked and limit-sliced, while explore/list remains bounded browsing without prefix/exact filtering. This is an open ClawHub issue, so the cleanup policy should not auto-close it.

## What This Changes

The issue requests a deterministic CLI discovery path for exact-slug and slug-prefix skill enumeration instead of relying only on relevance-ranked search.

## Best Possible Solution

Keep this issue open and implement a narrow indexed discovery path: exact slug lookup can reuse the existing skill detail path, while slug-prefix enumeration should use an indexed, cursor-paginated backend query exposed through conservative CLI flags or a hub-list mode with JSON support. Preserve current ranked search behavior and update CLI/API docs plus focused tests.

## Repair Work Prompt

Implement deterministic registry discovery for ClawHub CLI skill search/listing. Add conservative exact and prefix discovery support to the existing skill CLI surface, preferably `search --exact <slug>` and `search --prefix <prefix>` or an equivalent hub-list mode that matches the current command style. Preserve existing ranked `search <query...>` behavior when no exact/prefix mode is requested. Exact slug lookup may reuse the existing `/api/v1/skills/{slug}` detail path. Prefix enumeration must be backed by an indexed Convex query with cursor pagination; do not use `.filter()` or scan the full registry. Consider adding a lightweight active slug range index on `skillSearchDigest` or another existing lightweight public projection. Return stable machine-readable JSON for JSON mode and sensible text output for terminal listing. Update response schemas, CLI/API docs, and focused tests. Do not change package discovery behavior or unrelated plugin/package APIs. No changelog entry is required unless the repo has an active unreleased CLI changelog section.
