## Summary

Keep this issue open. Current main still matches the reported bug: CLI publish discovers files through extension-only filtering before reading file bytes, `.tsv` is absent from both text allowlists, and archive/GitHub import paths also drop extensionless entries before content inspection.

## What This Changes

Skill publish should include `.tsv` files and extensionless text executables, warn when unsupported files are skipped, and continue rejecting binaries.

## Best Possible Solution

Use one conservative text-file classification policy across skill publish entry points: add `.tsv` to the allowlists, classify extensionless text files by content without accepting binaries, surface skipped-file warnings to authors, and cover CLI, archive/web, GitHub import, and server validation with regression tests.

## Repair Work Prompt

Fix skill publish file classification for extensionless text files and `.tsv` files. Observable bug: `clawhub publish` silently omits extensionless text executables such as `bin/gurkerl` / `bin/openclaw-kraken` and `.tsv` files such as `config/endpoints.tsv`, producing successful but broken installs. Expected boundary: update shared and bundled text-file allowlists to include `.tsv`; add a conservative text-content fallback for extensionless files in CLI file discovery and archive/web/GitHub import paths; emit clear CLI publish warnings for skipped unsupported files; keep binary files rejected by server validation. Do not broaden package/plugin binary archive behavior or accept arbitrary binaries as skill files. Add focused regression tests for extensionless text scripts, `.tsv` files, binary rejection, and warning behavior. Include a short changelog/release-note entry only if this repo has an existing appropriate unreleased notes location.
