## Summary

Keep open. This is an open ClawHub issue, and the repository-specific cleanup policy only allows auto-closing pull requests that are certainly implemented on current main. Source review also does not prove the reported hosted skill is unstuck: current main has VirusTotal polling, stale handling, and generic owner-visible pending messaging, but no per-item queue/heartbeat/retry status surface, and skill-family VirusTotal results can still remain pending for undetected-only engine stats.

## Best Possible Solution

Keep this issue open for maintainer follow-up. Maintainers should verify the affected production skill/version record, repair or resync it if still pending, and decide whether to add an owner-visible scan status endpoint/UI showing queue position or age, last check or worker heartbeat, retry timing, and state transitions. If undetected-only VirusTotal engine results should clear skill scans when OpenClaw, LLM, and static signals are clean, that policy should be added to the skill scan path with focused tests instead of relying on the package-only fallback.
