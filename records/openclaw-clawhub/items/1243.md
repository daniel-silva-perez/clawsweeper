## Summary

Keep this open. The ClawHub cleanup policy for this pass keeps issues open, and current main still has a plausible search-recall gap: skill tags are stored in the lightweight search digest, but both vector-hydrated exact matching and lexical fallback compare only display name, slug, and summary. Related #1254 remains open as the broader direct lexical recall issue.

## What This Changes

The issue requests that a published skill with explicit `phone` and `voice` tags and matching title/description text surface when users search for those terms.

## Best Possible Solution

Keep this open and land one focused search-recall PR, ideally coordinated with #1254, that treats normalized skill tag names and capability-tag-like fields as first-class lexical candidates while preserving bounded Convex reads and current moderation filters. Add regression coverage for `phone` and `voice` tag queries.

## Repair Work Prompt

Fix ClawHub skill search recall for tag terms reported here and coordinate with related #1254. Current main stores `tags` and `capabilityTags` on `skillSearchDigest`, but `searchSkills` and `lexicalFallbackSkills` only pass display name, slug, and summary into `matchesExactTokens`. Implement bounded lexical matching so normalized skill tag names such as `phone`, `voice`, `ai-phone`, and `phone-screening` can make a skill a candidate when users search those terms. Preserve existing `nonSuspiciousOnly`, `highlightedOnly`, and `capabilityTag` behavior; do not broaden moderation visibility. Do not scan full skill documents or unbounded tables. Prefer reusing the lightweight digest path or adding a purpose-built indexed/lightweight term path if needed; if schema changes are required, follow the repo's Convex migration/backfill rules and update generated files in the fix PR. Add regression tests in `convex/search.test.ts` for `phone` and `voice` tag queries, including hyphenated tags tokenizing to bare words, and keep the change focused to search recall rather than general ranking redesign.
