## Summary

Keep open. ClawHub cleanup policy keeps issues open, and current main still has a credible search recall gap: tag data is present in lightweight search records, but both vector-hydrated exact matching and lexical fallback match only display name, slug, and summary, so `phone` and `voice` tag queries can still miss relevant skills.

## What This Changes

The issue asks for skill search to surface `batthis/amber-voice-assistant` for `phone` and `voice` queries based on explicit tags and prominent skill metadata.

## Best Possible Solution

Land a bounded tag-aware search recall fix, preferably coordinated with #1254, that treats normalized skill tag names and capability tags as lexical candidates while preserving existing moderation filters. Add regression coverage for `phone` and `voice` tag queries, and leave the suspicious-label decision to #1326.

## Repair Work Prompt

Fix this issue by making ClawHub skill search tag-aware for lexical recall. In `convex/search.ts`, include normalized skill tag names (`Object.keys(skill.tags)`) and `capabilityTags` in the candidate text used by vector-hydrated exact matching and `lexicalFallbackSkills`, so queries like `phone` and `voice` match explicit tags and hyphenated tags such as `ai-phone` or `phone-screening`. Preserve exact-slug injection, scoring order, `highlightedOnly`, `capabilityTag`, and `nonSuspiciousOnly` behavior. Do not change moderation policy, scanner verdict handling, publication flow, or suspicious-flag clearing; leave that to #1326. Do not add full `skills` table scans or unbounded digest scans; use existing hydrated/digest data or, if a broader indexed approach is needed for #1254, add normalized indexed fields with a cursor-based backfill. Add focused regression tests for tag-only `phone` and `voice` matches in both vector-hydrated and lexical fallback paths. Coordinate with #1254 so this does not become a one-off special case.
