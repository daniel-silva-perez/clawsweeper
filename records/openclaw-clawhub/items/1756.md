## Summary

Keep open. This is a ClawHub issue, and the cleanup policy only permits auto-closing ClawHub pull requests that are certainly implemented on main. Current main also still has a plausible implementation path for the reported behavior: the /skills UI replaces the whole search result set after scroll/load-more increases the requested limit, while backend relevance search still derives candidate expansion, fallback size, and final slicing from that requested limit.

## Best Possible Solution

Keep this issue open and fix the search pagination/ranking contract. The best implementation is to make relevance search stable across load-more requests, either by using a server-side cursor over a fixed ranked result set or by decoupling candidate expansion from the display limit so a 50-result request cannot discover higher-scoring entries above the already-rendered 25. Add regression coverage comparing the top results for the initial 25-result request against a larger request for the same query.

## Clownfish Work Prompt

Fix the /skills relevance search reorder bug described in this issue. Current behavior: the skills browse page requests 25 search results, then scroll/load-more increases the same search to 50 and replaces the whole result set; backend search derives candidate expansion/fallback/slicing from the requested limit, so higher-scoring skills can appear above already-rendered results after the scroll cycle. Implement a narrow fix so the top 25 relevance results for a query are stable when requesting a larger limit for the same query. Prefer a backend-centered fix that decouples candidate discovery/scoring from display limit, or a stable cursor/ranked-set approach if that is cleaner in the existing Convex architecture. Add regression tests that fail against the current 25-vs-50 instability and pass after the fix. Also keep the existing load-more behavior working.
