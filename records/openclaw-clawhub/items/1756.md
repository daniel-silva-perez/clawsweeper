## What This Changes

Fix `/skills` relevance search so the initial results for a query do not reorder or gain higher-relevance entries only after a scroll-triggered load-more cycle.

## Best Possible Solution

Make relevance search pagination stable so the first rendered page for a query remains a prefix of later expanded results. A durable fix should decouple recall/candidate expansion from the display limit or page over a stable ranked result set, with regression coverage for 25-result initial search versus 50-result load-more search.

## Repair Work Prompt

Fix the `/skills` relevance search pagination instability reported in this issue. Current main calls `api.search.searchSkills` with an expanding `limit` from `src/routes/skills/-useSkillsBrowseModel.ts`, replaces the whole `searchResults` array, and `convex/search.ts` can build a different ranked pool for `limit: 50` than for `limit: 25`. Implement a focused fix so the initial relevance results for a query remain a stable prefix after load-more, preferably by decoupling recall/candidate expansion from display limit or by paging over a stable ranked result set. Add regression coverage showing that a 25-result initial relevance request and the subsequent 50-result request do not insert higher-ranked items ahead of the original first page. Keep the change scoped to skills search behavior and do not alter unrelated browse sorts, moderation filters, or ClawHub publishing flows. Related refs: this issue, #1256, #1254. Changelog: no changelog required unless maintainers request one for user-facing search fixes.
