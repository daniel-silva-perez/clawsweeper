## Summary

Keep open. ClawHub policy keeps issues open, and current main still has a plausible scroll-triggered reorder path: the UI replaces the search result array after load-more increases the requested limit, while backend relevance search changes its candidate expansion and stopping behavior based on that same limit. Existing tests verify the larger request is made, but do not lock top-result stability across limits.

## What This Changes

Make /skills relevance search keep the same top-ranked results across initial render and subsequent scroll/load-more requests so high-relevance skills do not appear only after a scroll cycle.

## Best Possible Solution

Keep this open and fix the search pagination/ranking contract. The best path is to compute a stable ranked pool per query/filter independent of the displayed page size, or serve a cursor over a fixed ranked result set, then add regression coverage proving the top 25 for a query are identical for 25-result and 50-result requests.

## Repair Work Prompt

Fix the /skills relevance search stability bug reported here. Current main requests `searchSkills` with `limit: 25` initially, then scroll/load-more increases the limit to 50 and replaces the entire result set; backend relevance search also changes recall/candidate behavior based on that limit. Implement a stable pagination/ranking contract so the top 25 results for the same query/filter/sort are identical whether the caller asks for 25 or 50 results. Prefer computing a stable ranked pool independent of display limit or returning/carrying a cursor over a fixed ranked set. Keep the change focused to search relevance pagination; do not rewrite unrelated browse sorting, package search, auth, publishing, or UI styling. Add regression coverage in backend search tests for top-N stability across limits and UI coverage that load-more does not reorder already-visible relevance results. Preserve Convex query/read limits and avoid broad candidate scans.
