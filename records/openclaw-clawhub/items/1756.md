## Summary

Keep open. ClawHub cleanup policy does not auto-close issues, and current main still has a credible path for the reported scroll-triggered reorder: the UI increases the search limit on load-more and replaces the whole result set, while backend relevance recall and fallback sizing depend on that same limit. Related #1254 and #1256 cover direct lexical prefix recall, but they do not by themselves prove top-N search stability across larger result requests.

## What This Changes

The issue asks for `/skills` relevance search to keep initially visible high-relevance results stable instead of inserting different higher-ranked skills above the first page after a scroll-triggered load-more cycle.

## Best Possible Solution

Establish a stable relevance pagination contract for skills search: the first page should be computed from a candidate set that does not change when the UI asks for more visible results, or the backend should expose cursor-style paging over a stable ranked set. Add regression coverage proving the top N results for a query are unchanged between the initial page and a larger loaded page.

## Repair Work Prompt

Fix the `/skills` relevance search pagination stability bug reported in openclaw/clawhub#1756. The observable problem is that a search such as `image` can show one top ordering for the initial 25-result request, then after scroll-triggered load-more the larger request can insert newly discovered higher-ranked results above the original first page. Keep the fix focused on the skills search ranking/pagination contract, not a full search rewrite. Likely approaches: make backend candidate recall/fallback independent of the requested display limit for the first ranked window, or introduce stable paging/cursor behavior over a ranked set. Preserve existing exact slug, vector, lexical fallback, highlighted/nonSuspicious/capability filters, and public API response shape. Coordinate with #1254 and #1256 so direct lexical-prefix recall work is not duplicated or broken. Add regression coverage proving that the top 25 results for a query are unchanged when the same query is requested with a larger limit, and add/adjust UI coverage if the frontend contract changes. Do not change unrelated ranking weights, skill stat migration behavior, auth, release workflows, or package/plugin search surfaces.
