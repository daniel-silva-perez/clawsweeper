## Repair Work Prompt

Implement a focused false-positive appeal affordance for suspicious skill pages. Use a safe private default, preferably a `mailto:security@openclaw.ai` link, with a prefilled subject/body containing only safe context: owner handle, slug, latest version, canonical ClawHub URL, scanner statuses, and links to the skill/security detail pages. Do not embed long scanner findings, secrets, uploaded file contents, or raw evidence blobs. Keep this distinct from the existing generic report dialog and from rescan/staff override flows. Likely approach: add a small helper for building the appeal href, pass it from `SkillDetailPage` into `SkillHeader` and/or `DetailSecuritySummary`, and render it only for manageable suspicious skills. Avoid changing moderation/scanner logic, report counters, Convex schema, or plugin/package behavior unless the component API requires an explicitly gated optional prop. Add or update tests for the suspicious owner banner/security summary and verify that generic report/rescan behavior remains unchanged. Changelog: add a short user-facing note only if this repo's current changelog convention expects one for small ClawHub UI improvements; otherwise leave changelog untouched.
