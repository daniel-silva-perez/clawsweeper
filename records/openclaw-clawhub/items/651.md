## Summary

Keep this issue open. The repository policy for openclaw/clawhub says issues should not be auto-closed, and current main still routes `search` and token login transport failures through a generic `formatError(error)` path that can surface a bare `fetch failed`. Proxy support and troubleshooting docs exist, but there is still a narrow, useful CLI diagnostics fix here.

Keep the issue open for maintainer follow-up. A focused fix should improve CLI transport diagnostics in the HTTP/error formatting path while preserving API behavior and redacting secrets.

## Clownfish Work Prompt

Implement a focused CLI diagnostics improvement for bare `fetch failed` and related transport errors in the ClawHub CLI. Keep API behavior, schemas, command semantics, and retry behavior unchanged. Preferred boundary: add transport-aware formatting in `packages/clawhub/src/cli/ui.ts` and/or a small helper near `packages/clawhub/src/http.ts` that unwraps safe `Error.cause` details from Node/Undici/curl failures and adds actionable context for DNS, TLS/certificate, proxy, timeout, connection reset/refused, and offline/network failures. The output must redact bearer tokens, authorization headers, token-like query params, and proxy credentials embedded in URLs. It may point users to the existing troubleshooting docs when proxy/TLS/network hints are detected. Add focused Vitest coverage for `TypeError('fetch failed')` with nested causes such as `ENOTFOUND`, TLS/certificate errors, proxy/connection failures, timeout/abort, non-Error throws, and secret redaction. Do not change search ranking/results, login token persistence, API routes, or package publishing behavior. Update docs only if the new CLI message references a troubleshooting path or adds new user-facing advice.
