## Summary

Keep open. Current main includes proxy support and troubleshooting docs, but the reported user problem remains actionable because CLI transport failures still flow through a generic formatter as raw `Error.message`; both search and token login use that path. ClawHub policy also says issues should not be auto-closed, and this is a narrow CLI diagnostics repair candidate.

## What This Changes

The issue reports ClawHub CLI commands returning the low-level `fetch failed` error, with follow-up comments confirming similar failures for search and token login.

## Best Possible Solution

Add centralized, redacted CLI transport diagnostics at the shared HTTP/error formatting boundary so DNS, TLS/certificate, proxy, timeout/abort, and curl failures produce actionable guidance while preserving existing HTTP semantics.

## Repair Work Prompt

Implement a focused CLI diagnostics fix for the current issue: `clawhub search` and `clawhub login --token` can still show raw `fetch failed` or curl transport messages. Add centralized, redacted transport-error formatting at the shared HTTP/error boundary rather than per-command special cases. Classify common DNS, TLS/certificate, proxy, timeout/abort, and curl failures into actionable messages; keep existing HTTP status, retry/rate-limit, timeout, and successful request behavior unchanged. Build on the existing proxy support from openclaw/clawhub#363. Do not change registry API semantics, package publishing behavior, auth token storage, or unrelated command output. Add focused tests for formatting/classification and command-visible output where practical, including redaction of Authorization headers, API tokens, token query parameters, and proxy credentials. Update troubleshooting docs only if the new CLI guidance changes what users should do.
