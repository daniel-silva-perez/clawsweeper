## Summary

Keep open. Current main has proxy support and troubleshooting docs, but the shared CLI error path still surfaces transport failures as raw messages; both `search` and token login use that path, matching the issue discussion. This remains a narrow CLI diagnostics/UX bug rather than a close candidate for a ClawHub issue.

## What This Changes

The issue reports that `clawhub search "web"` fails with `fetch failed` and needs actionable CLI guidance for that transport failure.

## Best Possible Solution

Add focused CLI transport diagnostics in the shared HTTP/error formatting path so DNS, TLS/certificate, proxy, timeout, abort, and curl failures produce actionable redacted messages while preserving existing HTTP semantics.

## Repair Work Prompt

Implement shared ClawHub CLI transport diagnostics for raw `fetch failed` and curl transport failures. Focus on `packages/clawhub/src/http.ts` and/or `packages/clawhub/src/cli/ui.ts` so all CLI commands benefit, especially `search` and token login. Classify common Node/Undici and Bun curl failures for DNS (`ENOTFOUND`, `EAI_AGAIN`), connection refused/reset, TLS/certificate errors, timeout/abort, and likely proxy misconfiguration. Return actionable messages that point users toward proxy env vars/troubleshooting when relevant. Redact Authorization headers, bearer/API tokens, `token=` query params, and credentials embedded in proxy URLs or env values. Preserve successful request behavior, HTTP status errors, 401/404/429 retry/rate-limit hints, retry counts, timeout values, Bun curl use, and proxy dispatcher behavior. Add focused unit tests for the formatter/HTTP path; do not change package publishing, auth token storage, or registry API contracts.
