## Summary

Keep this issue open. It reports a real CLI `fetch failed` path with multiple independent confirmations, and the ClawHub repository policy for this review says issues should not be auto-closed. Current main has related proxy and Node HTTP hardening, but the CLI can still surface opaque network failures and the report lacks environment data needed to prove the problem is fixed.

## Best Possible Solution

Keep this issue open and ask for a focused reproduction: CLI version, install method, `node -v`, macOS version, proxy/VPN/corporate TLS details, `HTTPS_PROXY`/`HTTP_PROXY`/`NO_PROXY` values, and whether `curl -i https://clawhub.ai/api/v1/search?q=test` succeeds from the same machine. If reproducible, improve `packages/clawhub/src/http.ts` or CLI error handling to unwrap fetch causes and point users to proxy/TLS diagnostics instead of only printing `fetch failed`.
