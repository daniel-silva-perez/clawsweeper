## Summary

Keep open. ClawHub review policy keeps issues open, and current main still has a real CLI diagnostics gap: search and token login use the shared HTTP path, but transport failures can still surface as raw `fetch failed`/curl messages instead of actionable DNS, TLS, proxy, timeout, or network guidance.

## What This Changes

The issue reports `clawhub search "web"` failing with the unhelpful CLI error `fetch failed`, with later comments showing the same symptom for search and token login.

## Best Possible Solution

Keep the issue open and land a focused CLI diagnostics improvement in the shared CLI HTTP/error path so DNS, TLS, proxy, timeout, and curl failures produce actionable, redacted messages while preserving existing successful HTTP, status, retry, and rate-limit behavior.

## Repair Work Prompt

Implement a focused CLI diagnostics improvement for raw transport failures reported here. Expected boundary: shared CLI HTTP/error handling in `packages/clawhub`, likely `packages/clawhub/src/http.ts`, `packages/clawhub/src/cli/ui.ts`, and tests. Detect or format actionable messages for DNS lookup failures, connection refused/reset, TLS/certificate failures, proxy configuration failures, timeout/AbortError, and Bun curl failures. Redact Authorization headers, `clh_` tokens, token query params, and credentials embedded in proxy URLs or environment variables. Preserve successful request behavior, HTTP status errors, 429 retry/rate-limit messaging, retry behavior, registry URL composition, and normal CLI output. Add focused unit tests for Node fetch errors, timeout/cause handling, curl failure formatting, and redaction. Update `docs/troubleshooting.md` or `docs/cli.md` only if the new behavior needs user-facing documentation, and add a concise Unreleased changelog entry for the CLI error-message change.
