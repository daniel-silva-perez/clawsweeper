## Summary

Keep open. This issue is still actionable: current main has proxy support and troubleshooting docs, but the shared CLI error path still surfaces transport failures as raw `Error.message`, so `search` and token login can still show unhelpful `fetch failed` or curl output instead of actionable DNS, TLS, proxy, timeout, or redacted credential guidance.

## What This Changes

Users report `clawhub search` and `clawhub login --token` failing with a raw `fetch failed` CLI error.

## Best Possible Solution

Add focused CLI transport diagnostics in the shared HTTP/error formatting path so DNS, TLS, proxy, timeout, and curl failures produce actionable, redacted messages while preserving existing HTTP status, retry, rate-limit, and successful response behavior.

## Repair Work Prompt

Fix the ClawHub CLI diagnostics gap for raw transport failures reported here. Add a focused improvement in the shared CLI HTTP/error path so `clawhub search`, `clawhub login --token`, and other `apiRequest` users turn DNS, TLS/certificate, proxy, timeout, connection refused/reset, and Bun curl failures into actionable user-facing messages. Preserve existing successful request behavior, HTTP status handling, retry/rate-limit messages, schema parsing, and command output. Redact Authorization headers, `clh_...`/API tokens, token query params, and credentials in proxy URLs or environment variables. Prefer a shared helper in `packages/clawhub/src/http.ts` and/or `packages/clawhub/src/cli/ui.ts` with unit coverage in `packages/clawhub/src/http.test.ts`, `packages/clawhub/src/http.bun.test.ts`, and `packages/clawhub/src/cli/ui.test.ts`; add command-level tests only if needed to prove search/login use the shared formatted result. Do not change registry discovery semantics or add network calls beyond existing request paths. Update `docs/troubleshooting.md` only if the emitted guidance changes materially.
