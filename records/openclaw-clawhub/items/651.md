## Summary

Keep open. Current main has proxy support and troubleshooting docs, but the reported CLI failure path still surfaces low-level transport errors through the generic formatter, so users can still see unhelpful `fetch failed` output from search and token login.

## What This Changes

The issue reports `clawhub search "web"` failing with a raw `fetch failed` CLI error.

## Best Possible Solution

Add centralized, redacted CLI transport diagnostics at the shared HTTP/error formatting boundary so DNS, TLS/certificate, proxy, timeout/abort, and curl failures produce actionable guidance while preserving existing HTTP semantics.

## Repair Work Prompt

Fix the ClawHub CLI transport-error UX for the issue where `clawhub search "web"` and `clawhub login --token ...` can print only `fetch failed` or raw curl output. Implement the fix in the shared CLI HTTP/error boundary, likely `packages/clawhub/src/http.ts` and/or `packages/clawhub/src/cli/ui.ts`, with focused tests. Classify common DNS, TLS/certificate, proxy, timeout/abort, and curl failures into actionable messages, but do not assume every failure is proxy-related. Redact Authorization headers, API tokens, token query parameters, and credentials embedded in proxy URLs or environment variables. Preserve existing HTTP status behavior, retry/rate-limit messages, successful requests, timeout semantics, and Bun curl fallback behavior. Related context: #363 already added proxy env support and docs, so this PR should build diagnostics on top of that rather than replacing proxy support. Add a concise `CHANGELOG.md` fix entry if user-visible CLI output changes.
