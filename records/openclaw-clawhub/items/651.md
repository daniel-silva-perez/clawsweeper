## Summary

Keep open. This is a ClawHub issue, and the repository-specific review policy says this workflow should keep issues open; only PRs that current main definitely implements are eligible for auto-close. Source review also shows the user-facing `fetch failed` path is still actionable: proxy support and docs exist, but CLI commands still pass low-level transport errors through `formatError`, which returns only `error.message`.

## Best Possible Solution

Keep this issue open for maintainer follow-up. Ask affected users to retest with the latest `clawhub` CLI and provide CLI version, install method, `node -v`, runtime, macOS version, proxy/VPN/corporate TLS details, `HTTPS_PROXY`/`HTTP_PROXY`/`NO_PROXY` values, and whether `curl -i https://clawhub.ai/api/v1/search?q=test` works from the same machine. If reproducible, improve `packages/clawhub/src/http.ts` and `packages/clawhub/src/cli/ui.ts` to unwrap fetch causes and print actionable proxy/TLS/DNS diagnostics with regression tests.
