## Summary

Keep this issue open. The public profile route still calls users:getByHandle, and current main still resolves handles with Convex .unique() over non-unique handle indexes, so duplicate or conflicting production identity rows can still plausibly produce the reported server-error class. Current main has related publisher and digest-sync improvements, and #1179 remains the broader first-time-publisher cluster, but this issue needs maintainer production-data verification rather than auto-close.

## What This Changes

The issue reports that the public profile URL for a GitHub-linked handle returns a Convex users:getByHandle server error, likely because another GitHub or publisher identity is using the same name.

## Best Possible Solution

Keep the issue open and have a maintainer verify the production Convex error and rows for the reported handle. The best fix is either a targeted data cleanup/merge for duplicate identity or publisher rows, or a small backend change that makes public handle lookup duplicate-tolerant in a deterministic, privacy-safe way, with regression tests for duplicate users.handle and publishers.by_handle cases and coordination with #1179.
