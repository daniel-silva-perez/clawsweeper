## Summary

Keep open. ClawHub cleanup policy does not auto-close issues, and current main still does not prove the reported production GitHub OAuth/session-loss behavior is fixed. The code has OAuth callback handling and clearer user-facing auth errors, but it still intentionally blocks deleted/deactivated users, requires an active app user document before showing signed-in UI, and related OAuth-loop reports remain open.

## What This Changes

The issue reports that GitHub OAuth authorization succeeds but the callback returns users to ClawHub without an active session or persisted login state.

## Best Possible Solution

Treat this as part of a canonical OAuth/session-loop investigation. The end state should clearly separate active-account login failures from intentionally blocked deleted/deactivated accounts, restore reliable session creation and user bootstrap for active accounts, and provide a clear terminal error or support/recovery path when an account is blocked by policy.
