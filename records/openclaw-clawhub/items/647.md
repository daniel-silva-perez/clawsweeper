## Summary

Keep open. This is an open ClawHub issue, and the repository cleanup policy only permits conservative auto-close for pull requests that current main certainly implements. Current main has OAuth callback handling and clearer auth error paths, but it does not prove the reported production GitHub OAuth/session-loss behavior is fixed; active-user resolution can still produce logged-out UI, deleted/deactivated accounts are intentionally rejected, and several related OAuth/session-loop reports remain open.

## What This Changes

The issue reports that GitHub OAuth authorization succeeds but ClawHub does not create or reflect an active signed-in session after redirect.

## Best Possible Solution

Treat this as part of a canonical OAuth/session-loop investigation: active accounts should reliably establish a session and bootstrap a user/publisher, deleted/deactivated accounts should get a clear terminal error or maintainer-approved recovery path, and callback/user-resolution/UI-state regressions should be covered by targeted tests.
