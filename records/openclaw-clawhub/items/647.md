## Summary

Keep open. This is an open ClawHub issue, and the provided repository policy only permits conservative auto-close for pull requests that current main certainly implements. Current main has OAuth callback handling and clearer auth errors, but it does not prove the reported production session-loss behavior is fixed, and related OAuth/session-loop reports remain open.

## What This Changes

The issue reports GitHub OAuth authorization completing successfully while ClawHub fails to establish or display an authenticated session afterward.

## Best Possible Solution

Treat this as part of a canonical OAuth/session-loop investigation. The desired end state is reliable session creation and user bootstrap for active accounts, a clearly separated terminal error or support/recovery path for intentionally blocked deleted/deactivated accounts, and regression coverage around callback handling, user resolution, and signed-in UI state.
