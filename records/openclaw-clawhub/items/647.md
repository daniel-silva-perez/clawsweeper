## Summary

Keep open. This is an open ClawHub issue, and the repository cleanup policy only permits conservative auto-close for PRs that current main certainly implements. Current main has OAuth callback handling and clearer auth errors, but it does not prove the reported production GitHub OAuth/session-loss behavior is fixed; deleted/deactivated accounts are still intentionally blocked, signed-in UI depends on resolving an active app user, and several related OAuth/session-loop reports remain open.

## What This Changes

The issue reports GitHub OAuth authorization completing successfully but ClawHub failing to persist an authenticated session, leaving the user logged out and causing repeated login attempts.

## Best Possible Solution

Treat this as part of a canonical OAuth/session-loop investigation. The desired end state is reliable session creation and user bootstrap for active accounts, a clearly separated terminal error or support/recovery path for intentionally blocked deleted/deactivated accounts, and regression coverage around callback handling, user resolution, and signed-in UI state.
