## Summary

Keep open. This is an open ClawHub issue, and the cleanup policy only permits conservative auto-close for pull requests that current main certainly implements. Current main has OAuth callback handling and clearer auth error paths, but it does not prove the reported production GitHub OAuth/session-loss behavior is fixed; deleted/deactivated accounts are intentionally rejected, signed-in UI depends on resolving an active app user, and related OAuth/session-loop reports remain open.

## What This Changes

The issue reports successful GitHub OAuth authorization returning to ClawHub without creating or reflecting an active session, leaving the user logged out after callback.

## Best Possible Solution

Treat this as part of a canonical OAuth/session-loop investigation: active accounts should reliably establish a session and bootstrap a user/publisher, deleted or deactivated accounts should get a clear terminal error or maintainer-approved recovery path, and callback/user-resolution/UI-state regressions should be covered by targeted tests.
