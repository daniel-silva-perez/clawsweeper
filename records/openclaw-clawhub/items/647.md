## Summary

Keep this open. Current main has frontend OAuth callback handling and user-facing auth-error plumbing, but the source and tests do not prove the reported production GitHub OAuth/session-loss symptom is fixed. The backend still deliberately rejects deleted/deactivated users during OAuth completion, active-user lookup can still resolve to no app user, related OAuth-loop reports remain open, and the ClawHub cleanup policy says open issues should stay open unless maintainers handle them manually.

## Best Possible Solution

Keep this open and route it into a canonical OAuth/session-loop investigation. Maintainers should correlate production Convex and Vercel/frontend logs for failed GitHub OAuth callbacks, verify AUTH_GITHUB_ID/AUTH_GITHUB_SECRET, callback URLs, CONVEX_SITE_URL, VITE_CONVEX_URL, VITE_CONVEX_SITE_URL, and cookie behavior for clawhub.ai, reproduce clean-browser and deleted/deactivated-account cases, then decide whether deleted-account users get purge, restore, relink, re-registration, or a clearer terminal error/support path.
