## Summary

Keep open. This is an open ClawHub issue, and the repository cleanup policy only allows conservative auto-close for PRs that current main certainly implements. Current main has OAuth callback handling and clearer auth errors, but it does not prove the reported production GitHub OAuth/session-loss behavior is fixed; deleted/deactivated users are still intentionally blocked, active UI depends on a resolved active user document, and related OAuth/session-loop reports remain open.

## What This Changes

The issue reports that GitHub OAuth authorization succeeds on clawhub.ai but the callback does not create or persist an active session, leaving the user logged out after repeated login attempts.

## Best Possible Solution

Treat this as part of a canonical OAuth/session-loop investigation. The desired end state is reliable session creation for active accounts, a clearly separated terminal error or support path for intentionally blocked deleted/deactivated accounts, and regression coverage around callback handling, user bootstrap, and signed-in UI state.
