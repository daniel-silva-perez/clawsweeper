## Summary

Keep this open. ClawHub cleanup policy does not allow auto-closing open issues here, and current main still does not prove the reported production GitHub OAuth/session-loss behavior is fixed. Main has callback handling and user-facing auth-error plumbing, but OAuth completion still intentionally rejects deleted/deactivated users, active-user resolution can return no app user, the signed-in UI still requires both Convex auth and a resolved `users.me` document, and several closely related OAuth-loop reports remain open.

## What This Changes

The issue reports that GitHub OAuth authorization succeeds, but ClawHub does not persist an authenticated session or update logged-in UI state after redirect.

## Best Possible Solution

Keep this open and route it into a canonical OAuth/session-loop investigation. Maintainers should correlate production Convex and frontend logs, verify GitHub OAuth callback URLs plus `AUTH_GITHUB_ID`, `AUTH_GITHUB_SECRET`, `CONVEX_SITE_URL`, `VITE_CONVEX_URL`, `VITE_CONVEX_SITE_URL`, and cookie behavior for `clawhub.ai`, then distinguish clean active-account failures from deleted/deactivated-account cases. The eventual fix should either restore reliable session creation/profile bootstrap for active accounts or provide a clear terminal error/support/recovery path for accounts that are intentionally blocked.
