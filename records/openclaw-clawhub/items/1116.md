## Summary

Keep this issue open. Current main still intentionally blocks OAuth completion for deleted/deactivated ClawHub users, and the repository cleanup policy for ClawHub issues does not allow auto-closing issues. The remaining work is a maintainer account-lifecycle and support decision: allow re-registration, provide an admin reset/purge path, or keep deletion irreversible with clearer UX.

## What This Changes

The issue requests that ClawHub allow a user who deleted their account to sign back in or re-register with the same GitHub identity instead of hanging in the OAuth loading/unauthorized state.

## Best Possible Solution

Keep this open for maintainer triage across the deleted-account login-loop reports. The desired end state should explicitly define whether deleted GitHub identities are permanently blocked, eligible for verified admin reset/purge, or allowed through a deliberate re-registration flow, then update the Convex auth/account lifecycle code, tests, docs, and user-facing error handling to match that policy.

## Risks / Open Questions

- Changing this behavior affects privacy/account-deletion semantics because current docs and UI promise permanent, non-recoverable deletion.
- A recovery or re-registration path must distinguish user-initiated deletion from bans and malware autobans so account lifecycle changes do not bypass moderation protections.
- Manual production cleanup requires identity verification and careful handling of Convex Auth account bindings, purged profile data, personal publishers, public skills, audit logs, and revoked API tokens.
- Several related open reports describe the same symptom, so maintainers should choose one canonical account-lifecycle direction before implementation.
