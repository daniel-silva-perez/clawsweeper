## Summary

Keep this issue open. The ClawHub cleanup policy for this run keeps issues open, and current main still only has per-version soft-delete data/read behavior, not an owner-facing way to hide, delete, restore, or unhide one published skill version. The related open security request #600 reinforces that this remains an active product need.

## Best Possible Solution

Keep this issue open and implement a first-class per-version lifecycle workflow. The likely path is an authenticated API/mutation that resolves a skill by slug plus version, authorizes owners/org members/moderators, patches `skillVersions.softDeletedAt` with audit logging, handles latest-version and tag constraints explicitly, documents the API, adds CLI commands such as `clawhub version hide/delete/unhide/undelete`, and covers public history/detail/file/download/resolve behavior plus owner/staff access in focused tests.
