## Clownfish Work Prompt

Fix owner-facing diagnostics for hidden or quality-held skills in `clawhub inspect` for openclaw/clawhub. The observable bug is that a publisher whose skill is hidden by the quality gate can see the skill disappear from public search but normal `clawhub inspect <slug>` does not show structured moderation status/reason/recovery guidance.

Expected boundary:
- Preserve public search filtering and moderation behavior. Do not make hidden or `quality.low` skills public.
- Teach `clawhub inspect` to surface owner-authenticated moderation diagnostics when available, preferably by calling `/api/v1/skills/<slug>/moderation` with the existing auth token after the normal detail request succeeds with moderation info or fails with an owner-visible hidden-skill status.
- Print clear human output for status/verdict, `legacyReason` such as `quality.low`, reason codes/summary when present, and recovery guidance. Include the same data in `--json` output.
- Handle current text 403/410/423 hidden-skill responses from `/api/v1/skills/<slug>` without losing useful messages or breaking public/not-found behavior.
- Add focused tests for hidden `quality.low` owner diagnostics and existing inspect behavior. Do not attempt to fix stale rendered SKILL.md/tags here; coordinate that under #1407.
