## Summary

Keep open. ClawHub cleanup policy does not auto-close issues, and current main still leaves the documented nested `metadata.openclaw.requires.env` shape invisible to static moderation even though docs, parsing, and the LLM prompt path understand it.

## What This Changes

The issue requests registry/static moderation extraction of `metadata.openclaw.requires.env`, `primaryEnv`, and related runtime metadata from published skill files so valid skills are not falsely flagged suspicious.

## Best Possible Solution

Make static publish and static rescan moderation consume the same normalized OpenClaw/Clawdis runtime metadata used by parsing/display/LLM context, add regression coverage for the inline JSON `metadata.openclaw.requires.env` shape, then rescan or manually review the affected DeepRead versions while leaving broader `metadata.json` ingestion to #1053.

## Repair Work Prompt

Fix the static moderation false positive where documented nested `metadata.openclaw.requires.env` and `metadata.openclaw.primaryEnv` declarations are not treated as declared env vars. Keep the change narrow: either pass normalized `clawdis`/OpenClaw metadata into `runStaticPublishScan` for publish and static rescan, or make the static env collector descend through documented aliases (`metadata.openclaw`, `metadata.clawdis`, `metadata.clawdbot`) without changing unrelated metadata ingestion. Add a regression test using the inline JSON frontmatter shape from this issue with `DEEPREAD_API_KEY` and a code file that reads `process.env.DEEPREAD_API_KEY` before `fetch`. Preserve tests that flag undeclared env access, broad `process.env` access, hardcoded secret docs, and file-read plus network-send exfiltration. Do not implement package-level `metadata.json` ingestion here; leave that to #1053. After the code fix, note that maintainers still need to rescan or manually review existing affected DeepRead versions. No changelog entry is required unless the repo has an unreleased user-facing bugfix section for ClawHub scanner changes.
