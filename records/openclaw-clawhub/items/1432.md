## Summary

Keep open. This is an openclaw/clawhub issue, and the repository-specific cleanup policy only permits auto-closing pull requests that are definitely implemented on main. Substantively, current main still does not prove the reported false-positive path is fixed end to end: publish parsing normalizes `metadata.openclaw` into `parsed.clawdis`, and the LLM prompt can use that normalized data, but the static moderation scanner still collects declared environment variables only from direct `frontmatter`/`metadata` fields rather than from `metadata.openclaw` or `parsed.clawdis`.

## Best Possible Solution

Keep this issue open for maintainer handling. The likely fix belongs in the skill publish/moderation pipeline: make static moderation consume normalized `parsed.clawdis`, or explicitly descend into `metadata.openclaw`/`metadata.clawdis`/`metadata.clawdbot` when collecting declared env vars; add a regression test using the exact inline `metadata: {"openclaw":{"requires":{"env":["DEEPREAD_API_KEY"]},"primaryEnv":"DEEPREAD_API_KEY"}}` shape; then rescan/backfill affected versions and clear confirmed false positives through the existing moderation path.
