## Summary

Keep open. ClawHub cleanup policy keeps issues open, and the reported false-positive path still appears valid: current main documents and parses nested `metadata.openclaw.requires.env`, but the static moderation env collector used for publish and rescan still only sees direct env declaration fields. The affected live DeepRead skills also still report suspicious moderation status while their hosted `SKILL.md` files declare `DEEPREAD_API_KEY` in the documented nested shape.

## What This Changes

Registry/static moderation should extract `openclaw.requires.env` and `primaryEnv` from published skill metadata so correctly declared env-backed skills are not flagged suspicious.

## Best Possible Solution

Feed the normalized runtime metadata used by parsing/display/LLM context into static publish and static rescan moderation, add regression coverage for the inline JSON `metadata.openclaw.requires.env` shape, then rescan or manually review the affected DeepRead versions while leaving broader package metadata ingestion to #1053.

## Repair Work Prompt

Fix the false-positive path where skills declaring required env vars under documented nested `metadata.openclaw.requires.env` are treated as undeclared by static publish/rescan moderation. Current main parses nested OpenClaw metadata in `parseClawdisMetadata`, but `runStaticPublishScan`/`runStaticModerationScan` only collect direct `requires`, `env`, `envVars`, and `primaryEnv` fields. Update the static moderation input or env collector so publish and rescan consume the same normalized runtime metadata used by display/LLM context. Add regression coverage for an inline JSON frontmatter example matching this report (`metadata: {"openclaw":{"requires":{"env":["DEEPREAD_API_KEY"]},"primaryEnv":"DEEPREAD_API_KEY"}}`) with code that reads `process.env.DEEPREAD_API_KEY` and calls `fetch`, and assert it is not flagged as `suspicious.env_credential_access`. Keep or add negative tests proving undeclared env access and broad `process.env` access remain suspicious. Do not broaden package `metadata.json` ingestion beyond #1053, do not weaken exposed-secret/dangerous-exec/broad-env checks, and do not mutate live moderation records except through existing rescan/admin surfaces. Include a short changelog/PR note that existing affected versions need rescan or manual review after deploy.
