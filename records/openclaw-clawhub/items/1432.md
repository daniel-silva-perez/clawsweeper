## Summary

Keep open. The ClawHub cleanup policy for this pass does not allow auto-closing issues, and current main still has the substantive static moderation gap: documented `metadata.openclaw.requires.env` is parsed into normalized Clawdis metadata, but the static publish scan only passes raw frontmatter/metadata to an env collector that reads direct env fields and misses nested `metadata.openclaw` aliases.

## Best Possible Solution

Keep this issue open and land a narrow backend fix. Static moderation should consume normalized `parsed.clawdis` or explicitly collect env declarations from `metadata.openclaw`, `metadata.clawdis`, and `metadata.clawdbot`; if skill-level `metadata.json` or `package.json` env declarations are intended to be supported, parse and document those shapes as part of the same cluster with #1053. Add a regression test for the inline JSON frontmatter shape from this report, then rescan or manually review affected published versions.

## Clownfish Work Prompt

Fix the static moderation env-declaration mismatch for skill publishes. Current docs support `SKILL.md` frontmatter `metadata.openclaw.requires.env` and `primaryEnv`; `publishVersionForUser` parses that into `clawdis`, but `runStaticPublishScan` only receives raw `frontmatter`/`metadata`, and `collectDeclaredEnvNames` only reads direct env fields. Update the backend so static moderation recognizes declared env vars from normalized Clawdis metadata and/or nested `metadata.openclaw`, `metadata.clawdis`, and `metadata.clawdbot`. Add a regression test using the report's inline JSON frontmatter shape: `metadata: {"openclaw":{"requires":{"env":["DEEPREAD_API_KEY"]},"primaryEnv":"DEEPREAD_API_KEY","homepage":"https://www.deepread.tech"}}` plus code that reads `process.env.DEEPREAD_API_KEY` and sends it to the intended API; assert it does not produce `suspicious.env_credential_access`. Preserve the existing undeclared-env and broad-env suspicious behavior. If supporting `metadata.json` or `package.json` env declarations for skills is in scope, parse the supported shapes from uploaded text files conservatively and document the supported shape; do not change the CLI payload contract unless it is truly necessary. Do not run production rescans in the PR; note that affected published versions need a rescan/backfill or moderator review after deployment.
