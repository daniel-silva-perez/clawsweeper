## Summary

Keep this issue open. The ClawHub cleanup policy for this pass does not auto-close issues, and the current evidence still leaves a real production auth question: current main and live discovery use the canonical non-www registry, the CLI sends Bearer auth to `/api/v1/whoami`, and the backend token lookup path is indexed and coherent, but I could not verify a fresh production UI-created token end to end. The earlier www workaround reports and unmerged PRs are useful history, but production now redirects `www.clawhub.ai` back to `clawhub.ai`, so they are not enough to call the token-login failure fixed.

## Best Possible Solution

Keep this issue open for maintainer verification with a fresh production UI token. If current published CLI token login, browser login, whoami, and a minimal publish/sync smoke path all succeed against the default `https://clawhub.ai` registry, close manually with that production evidence and leave the canonical non-www defaults alone.
