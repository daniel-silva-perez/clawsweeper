## Summary

Keep this issue open. Current main partially mitigates the original inline publisher-trigger failure by scheduling package and skill digest rebuilds separately, but both owner-publisher digest helpers still loop over `.paginate()` inside one Convex mutation invocation, so large publisher result sets can still match the reported Convex single-pagination failure. The closed, unmerged #1183 is useful prior context, but a safer fix should use cursor-based self-scheduling rather than unbounded `.collect()`.

## Best Possible Solution

Keep this issue open and land one focused Convex backend PR for the publish-pagination cluster: make the owner-publisher package and skill digest rebuilds process at most one paginated page per internal mutation invocation, carry the cursor in validated args, and schedule the continuation when `isDone` is false. Add regression coverage that a multi-page owner rebuild schedules a continuation instead of calling `.paginate()` twice in one invocation, then deploy and verify a real updated-skill publish succeeds before consolidating the duplicate reports.
