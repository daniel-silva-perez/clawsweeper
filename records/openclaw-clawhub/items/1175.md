## Summary

Keep this issue open. Repository policy for this ClawHub cleanup pass only permits auto-closing pull requests that are certainly implemented on current main, and this item is an issue. Current main mitigates the original inline publisher-trigger failure by scheduling package and skill digest rebuilds into separate internal mutations, but the owner-publisher digest helpers still loop over `.paginate()` inside one Convex mutation invocation, so a larger owner can still hit the same Convex single-pagination limit.

## Best Possible Solution

Keep this issue open and land one focused Convex backend PR for the publish-pagination cluster: make the owner-publisher package and skill digest rebuilds cursor-based, processing at most one paginated page per mutation invocation and scheduling the continuation when needed. After that lands and is manually deployed to production, maintainers should verify a real updated-skill publish succeeds and then consolidate/close the related duplicate reports.

## Clownfish Work Prompt

Fix the Convex multiple-paginated-query risk in the publisher digest sync path used by publish/`ensurePersonalPublisherForUser`.

Current main already schedules package and skill owner-publisher digest rebuilds separately, but `convex/functions.ts` still has `syncPackageSearchDigestsForOwnerPublisherId` and `syncSkillSearchDigestsForOwnerPublisherId` loop over `.paginate()` until done inside one internal mutation invocation. Change this to a bounded cursor-based pattern: each scheduled internal mutation should perform at most one `.paginate()` call for its owner publisher, process that page, and if `isDone` is false, use `ctx.scheduler.runAfter(0, sameInternalMutation, { ownerPublisherId, cursor: continueCursor })` to continue. Keep the existing `by_owner_publisher` indexed queries and missing-table handling. Do not replace the loops with unbounded `.collect()` or scan tables.

Audit the sibling pagination loops in `convex/functions.ts`, especially `syncPackageSearchDigestsForOwnerUserId`, and include the same one-page scheduling pattern if it can be reached from a trigger/mutation path with the same Convex limit. Keep the PR focused to digest sync behavior.

Add/adjust tests in `convex/functions.test.ts` proving the owner-publisher package and skill sync paths issue only one `paginate` call per invocation, schedule continuations with the next cursor, and do not schedule when the first page is done. Update existing scheduling tests as needed. Regenerate Convex generated files only if function references or args require it. No changelog entry is required unless the repo has a current changelog convention for backend bug fixes.
