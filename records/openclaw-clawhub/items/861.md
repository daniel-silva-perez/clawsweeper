## Summary

This should stay open. Current main still has a plausible CLI login diagnostics gap: browser auth can return a token, then the required `/api/v1/whoami` verification fetch can fail and be reported as only the generic top-level `fetch failed` message. ClawHub cleanup policy also says issues should remain open unless a maintainer handles them, while this one has a narrow code/docs/test repair path.

## What This Changes

The issue asks for the CLI login flow to handle or explain a `fetch failed` error after `npx clawhub login` opens the browser auth URL.

## Best Possible Solution

Keep browser-token verification intact, but improve the login verification failure path so users get sanitized network-cause details plus login-specific proxy and `clawhub login --token` fallback guidance, backed by focused regression tests and troubleshooting docs.

## Repair Work Prompt

Fix the CLI login diagnostics gap reported in https://github.com/openclaw/clawhub/issues/861. When browser auth succeeds but the subsequent `/api/v1/whoami` token verification fetch fails, keep verification mandatory and do not write the token, but surface a clearer sanitized error message that includes safe network-cause details and login-specific remediation. Include proxy guidance and `clawhub login --token` fallback guidance. Do not print API tokens, Authorization headers, callback URLs with sensitive query data, or raw request objects. Add focused tests around login verification fetch failure and any shared error formatting/helper behavior, and update troubleshooting docs. Avoid broad CLI refactors or changes to auth token storage semantics.
