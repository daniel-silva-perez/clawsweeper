## Summary

This should stay open: current main still has an actionable CLI login diagnostics gap where browser auth can hand a token back to the CLI, but the subsequent `/api/v1/whoami` verification fetch can fail with only the generic top-level `fetch failed` message. The prior ClawSweeper comments are consistent with the current source, docs, tests, and path history.

## What This Changes

The issue reports that `npx clawhub login` opens the browser auth URL but then fails in the CLI with a generic `fetch failed` error.

## Best Possible Solution

Keep browser-token verification intact, but make login verification failures produce sanitized network-cause details plus login-specific proxy and `clawhub login --token` fallback guidance, backed by focused regression tests and troubleshooting docs.
