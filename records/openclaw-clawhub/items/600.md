## Summary

Keep this issue open. Current main has a `skillVersions.softDeletedAt` field and public read/download paths honor it, but I found no owner/staff HTTP or documented workflow that hides or deletes only `tencent-doc-update-watch@0.1.0`; the exposed delete/undelete flow is skill-level. The related general feature request #905 is still open, and this incident also needs private maintainer/security handling for caches and download/install records.

## Best Possible Solution

Keep this open for maintainer/security follow-up. The immediate path is to privately identify the `skillVersions` row for `tencent-doc-update-watch@0.1.0`, mark or remove only that version, verify `/versions/0.1.0`, `/file?version=0.1.0`, and `/download?version=0.1.0` are blocked, review any cache/storage purge options, and answer the reporter privately. Longer term, implement the owner/staff single-version withdrawal workflow tracked by #905 with audit logging and clear limits for latest-version handling.
