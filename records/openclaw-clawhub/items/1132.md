## Summary

Keep open. Current main does not implement the PR's security-category moderation flow: static scan findings are still converted directly into reason codes, `verdictFromCodes()` still escalates any non-empty code list to suspicious, and the LLM prompt/docs have no category context. The PR also changes a moderation trust boundary and adds an unrelated npm lockfile, so it needs maintainer/security review rather than cleanup closure.

## What This Changes

This PR adds a publisher-declared `category: security` moderation context that rewrites selected static scan reason codes to `info.security_context.*`, passes category context to the LLM evaluator, updates docs/tests, and adds a `package-lock.json`.

## Best Possible Solution

If maintainers want this core feature, rebase it onto the current moderation engine, settle the security policy for category-based contextualization, remove or explicitly justify the npm lockfile, and land it with focused moderation tests plus documentation that matches the final trust model.
