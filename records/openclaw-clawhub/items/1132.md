## Summary

Keep open. Current main does not implement the PR's moderation-category behavior, and the remaining decision is security/product review rather than conservative cleanup: the PR would change how publisher-declared metadata affects static scanner verdicts, has unresolved fail-open versus LLM-gated policy discussion, and includes an unrelated npm lockfile artifact in a Bun repo.

## What This Changes

This PR adds a publisher-declared `category: security` moderation context that rewrites selected `suspicious.*` static scan reason codes to `info.security_context.*`, passes category context to the LLM evaluator, updates moderation docs/tests, and adds a `package-lock.json`.

## Best Possible Solution

Keep this PR open for maintainer/security review. If maintainers want the feature, rebase it onto current main, align it with moderation engine `v2.4.2`, decide the fail-open versus explicitly LLM-gated posture for security-category contextualization, remove or justify `package-lock.json`, and refresh tests/docs while coordinating with #1131 and #1133.
