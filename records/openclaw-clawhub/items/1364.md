## Summary

Keep open. Current main still reproduces the reported serialization boundary bug: the transfer command passes a JSON string as `body`, and both HTTP transports serialize POST bodies again before sending them, so the server receives a JSON string instead of an object with `toUserHandle`.

## What This Changes

Requests that `clawhub transfer request <slug> <handle>` map the positional handle to the API `toUserHandle` payload field instead of failing with `toUserHandle required`.

## Best Possible Solution

Land a focused CLI fix that passes plain object bodies from transfer, rename, and merge command handlers to `apiRequest`, updates command tests to assert object bodies, and keeps the HTTP helper responsible for JSON serialization.

## Repair Work Prompt

Fix the CLI JSON body serialization bug reported for `clawhub transfer request <slug> <handle>`. In current main, `cmdTransferRequest` passes `JSON.stringify(...)` as `body`, while `apiRequest` and the Bun curl path serialize POST bodies themselves, causing the server to receive a double-encoded JSON string and return `toUserHandle required`.

Expected boundary: command handlers pass plain object bodies to `apiRequest`; `apiRequest` remains the single JSON serialization boundary. Do not change the server to accept double-encoded strings and do not refactor unrelated HTTP behavior.

Implement the same object-body fix for adjacent ownership commands that share the pattern: `cmdRenameSkill` and `cmdMergeSkill`. Update `packages/clawhub/src/cli/commands/transfer.test.ts` and `packages/clawhub/src/cli/commands/ownership.test.ts` so they assert object bodies rather than serialized JSON strings. Add a concise `CHANGELOG.md` Unreleased Fixes entry for CLI ownership command request bodies.

Related ref: https://github.com/openclaw/clawhub/issues/1364
