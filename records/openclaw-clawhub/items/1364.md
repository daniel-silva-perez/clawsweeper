## Summary

Keep this issue open. Current main still contains the reported `transfer request` body serialization bug: the positional handle is mapped to `toUserHandle`, but the CLI passes a JSON string into an HTTP helper that serializes bodies again before sending them.

## Best Possible Solution

Land a focused CLI fix: pass a plain object body from `cmdTransferRequest` to `apiRequest`, update the transfer test to assert an object payload, and audit the adjacent rename/merge ownership commands that use the same pre-serialized body pattern. Keep the HTTP helper's single-serialization contract intact.

## Clownfish Work Prompt

Fix the CLI JSON double-serialization bug reported in openclaw/clawhub#1364. In `packages/clawhub/src/cli/commands/transfer.ts`, change `cmdTransferRequest` so the POST body passed to `apiRequest` is a plain object `{ toUserHandle: toHandle, message: options.message }`, not `JSON.stringify(...)`. Keep the HTTP helper serialization contract in `packages/clawhub/src/http.ts` unchanged: it should serialize plain bodies once. Update `packages/clawhub/src/cli/commands/transfer.test.ts` so the transfer request test asserts the mocked request body equals the object payload, not a string containing text. Audit and fix the adjacent pre-serialized `apiRequest` bodies in `packages/clawhub/src/cli/commands/ownership.ts` for rename and merge, and update `packages/clawhub/src/cli/commands/ownership.test.ts` to assert object bodies. Do not broaden the change into unrelated HTTP refactors, auth behavior, API schema changes, or release automation. No changelog is expected unless the repo already has an unreleased CLI notes file being maintained for this release.
