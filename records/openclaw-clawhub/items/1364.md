## Summary

This issue should stay open. Current main still pre-serializes the `clawhub transfer request` POST body before handing it to an HTTP helper that serializes POST bodies itself, so the API can receive a JSON string instead of an object and return `toUserHandle required`. The bug is narrow and has a clear fix path, including adjacent ownership commands that use the same pattern.

## Best Possible Solution

Land a focused CLI fix that passes plain object bodies from `cmdTransferRequest` to `apiRequest`, updates the transfer test to assert the object payload, and fixes the adjacent ownership rename and merge command bodies/tests that use the same pattern. Keep the HTTP helper as the single JSON serialization boundary and do not change the API contract.

## Clownfish Work Prompt

Fix the ClawHub CLI double JSON serialization bug reported in the transfer request workflow. In `packages/clawhub/src/cli/commands/transfer.ts`, change `cmdTransferRequest` to pass a plain object body `{ toUserHandle: toHandle, message: options.message }` to `apiRequest`, not a pre-serialized string. Audit and fix the adjacent ownership commands in `packages/clawhub/src/cli/commands/ownership.ts` so `cmdRenameSkill` and `cmdMergeSkill` also pass plain object bodies for `newSlug` and `targetSlug`. Update `packages/clawhub/src/cli/commands/transfer.test.ts` and `packages/clawhub/src/cli/commands/ownership.test.ts` to assert object bodies rather than string contents. Keep `packages/clawhub/src/http.ts` as the single JSON serialization boundary for both Node and Bun/curl transports; do not change API endpoint paths, server parsing, auth behavior, or CLI command names. No docs change is expected unless command behavior or usage text changes.
