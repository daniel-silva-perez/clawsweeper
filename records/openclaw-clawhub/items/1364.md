## Summary

Keep open. Current main still has the reported CLI serialization bug: the transfer command passes a pre-serialized JSON string, then both HTTP transport paths serialize POST bodies again, so the API receives a JSON string rather than an object with `toUserHandle`.

## What This Changes

The issue asks for `clawhub transfer request <slug> <handle>` to send the positional handle as the `toUserHandle` API field instead of failing with `toUserHandle required`.

## Best Possible Solution

Land a focused CLI fix that passes plain object bodies from transfer, rename, and merge command handlers to `apiRequest`, updates command tests to assert object bodies, and leaves the server API expecting normal JSON objects.

## Repair Work Prompt

Fix the CLI JSON body boundary for skill ownership operations reported in this issue. `apiRequest` already serializes POST `body` values, so update `packages/clawhub/src/cli/commands/transfer.ts` to pass a plain object with `toUserHandle` and `message`, and update `packages/clawhub/src/cli/commands/ownership.ts` so rename and merge pass plain objects for `newSlug` and `targetSlug`. Update command tests to assert object bodies instead of serialized JSON strings. Do not change the Convex API to accept double-encoded JSON strings, and avoid broad changes to the HTTP helper unless a test exposes a real contract gap. No changelog entry is expected for this narrow CLI bug fix unless the repository has a local convention requiring one.
