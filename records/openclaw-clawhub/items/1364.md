## Summary

Keep open. Current main still contains the reported CLI serialization bug: the transfer command passes a pre-serialized JSON string as `body`, then both HTTP transport paths serialize request bodies again, so the API receives a JSON string instead of an object with `toUserHandle`.

## What This Changes

The issue requests that `clawhub transfer request <slug> <handle>` send the positional handle as the API payload field `toUserHandle` instead of failing with `toUserHandle required`.

## Best Possible Solution

Land a focused CLI fix that passes plain object bodies from transfer, rename, and merge command handlers into `apiRequest`, updates command tests to assert object bodies, and keeps the server API expecting normal JSON objects.

## Repair Work Prompt

Fix the CLI request-body serialization bug reported in https://github.com/openclaw/clawhub/issues/1364. `apiRequest` is the JSON serialization boundary, so update `packages/clawhub/src/cli/commands/transfer.ts` to pass a plain object body for transfer requests instead of `JSON.stringify(...)`. Also fix the same command-side serialization pattern in `packages/clawhub/src/cli/commands/ownership.ts` for rename and merge. Update `packages/clawhub/src/cli/commands/transfer.test.ts` and `packages/clawhub/src/cli/commands/ownership.test.ts` so they assert object bodies, and keep the existing `packages/clawhub/src/http.ts` behavior that serializes POST bodies for both Node and Bun transports. Do not change the server API to accept double-encoded JSON strings. No changelog entry is required unless the repo has an unreleased CLI changelog section for user-visible bug fixes.
