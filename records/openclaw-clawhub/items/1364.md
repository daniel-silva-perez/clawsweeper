## Summary

Keep this issue open. Current main still passes a pre-serialized JSON string from `cmdTransferRequest` into `apiRequest`, and both Node and Bun HTTP paths serialize POST bodies again before the server reads `parsed.payload.toUserHandle`. The reporter’s root-cause comment is corroborated by current source, and the fix is narrow enough for a focused repair PR.

## What This Changes

The issue requests that `clawhub transfer request <slug> <handle>` send the positional handle as the API payload field `toUserHandle` instead of failing with `toUserHandle required`.

## Best Possible Solution

Land a focused CLI fix that passes plain object bodies from `cmdTransferRequest`, `cmdRenameSkill`, and `cmdMergeSkill` to `apiRequest`; update the matching command tests to assert object payloads; keep the HTTP helper as the single JSON serialization boundary and leave the server API contract unchanged.

## Repair Work Prompt

Fix the CLI double-serialization bug reported for `clawhub transfer request`. Current main passes `JSON.stringify(...)` from `cmdTransferRequest`, `cmdRenameSkill`, and `cmdMergeSkill` into `apiRequest`, while `apiRequest` is already responsible for JSON-serializing POST bodies in both Node and Bun/curl runtimes. Change those command callers to pass plain object bodies (`{ toUserHandle, message }`, `{ newSlug }`, `{ targetSlug }`) and update the matching command tests to assert object payloads instead of string contents. Keep the server API contract unchanged, keep `packages/clawhub/src/http.ts` as the single serialization boundary, and do not broaden `parseJsonPayload` to accept double-encoded strings. No changelog entry is expected unless the repository’s CLI release convention requires one for this narrow bug fix.
