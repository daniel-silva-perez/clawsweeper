## Summary

Keep open. This is a ClawHub issue, and the repository-specific cleanup policy only permits auto-closing pull requests that are certainly implemented on current main. Substantively, current main’s `clawhub install <slug>` path appears to perform single-skill work, while full local folder scanning is the `clawhub sync` path; maintainer triage is still useful to confirm which flow the reporter observed.

## Best Possible Solution

Keep this issue open for maintainer triage. If the report targets current `clawhub install`, add a focused regression test proving install only touches the requested slug and does not call scan helpers. If the slow path is actually `clawhub sync`, an older CLI, or downstream OpenClaw installation, rescope the issue to that flow and consider an incremental install-state update path while preserving full scans for manual sync and repair.
