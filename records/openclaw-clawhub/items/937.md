## Summary

Keep open. Current main still has a credible false-positive path: skill downloads inject generated root `_meta.json`, the CLI extracts and hashes root JSON files, and backend resolve compares against publish-time fingerprints built from stored version files that do not include the generated download metadata. Repository policy also keeps ClawHub issues open unless they are PRs already implemented on main.

## What This Changes

The issue reports that `clawhub update --all` falsely reports local changes immediately after a force update because installed skill fingerprints do not match registry resolution.

## Best Possible Solution

Land a focused CLI/backend integrity fix so install/update/download/resolve compare the same canonical published text file set, excluding only generated install/download metadata such as root `_meta.json` or avoiding that metadata in installed skill directories, while preserving fingerprint coverage for real published files.

## Repair Work Prompt

Fix the ClawHub CLI false `local changes (no match)` warning after fresh install or `clawhub update --all --force`. Current main downloads/extracts generated root `_meta.json`, local fingerprinting includes root JSON text files, and backend resolve compares against publish-time `version.files` fingerprints that do not include generated download metadata. Implement one focused fix so local installed skills, downloaded zips, and registry resolve use the same canonical published file set. Exclude only generated install/download metadata such as root `_meta.json`, or stop injecting/extracting it for skill installs. Do not broadly ignore user-authored JSON or weaken fingerprint coverage for real published text files. Add regression tests for a force-update followed immediately by normal update reporting up to date, and tests proving real published JSON/tampered files still affect fingerprints. Include an Unreleased changelog fix entry if the repo expects user-visible CLI fixes there. Related refs: #937, #169, #178.
