## Repair Work Prompt

Fix the false `local changes (no match)` warning after a freshly installed or force-updated skill. Current main hashes local text files for `clawhub update`, downloads can inject/extract generated root `_meta.json`, and backend resolve compares against publish-time `version.files` fingerprints that exclude that generated metadata. Implement one canonical fix for this cluster (#169, #178, and this issue): ensure local directory hashing, downloaded zip hashing, and registry resolve compare the same canonical published file set. Prefer excluding only generated install/download metadata such as root `_meta.json` from local and zip fingerprinting, or stop injecting/extracting that metadata for skill installs. Do not broadly ignore user-authored JSON files; published JSON must still affect fingerprints. Add focused regression tests for fresh install/force-update followed by normal update, plus zip/local hashing tests showing generated `_meta.json` does not cause a mismatch while real file edits still do. Keep changes scoped to ClawHub CLI/backend integrity comparison; do not change unrelated package/plugin download behavior or release metadata.
