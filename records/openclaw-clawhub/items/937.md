## Summary

Keep open. ClawHub review policy keeps issues open, and current main still has a credible source-level false-positive path: the CLI hashes extracted local text files, skill downloads inject and extract generated `_meta.json`, and registry resolve compares against published version fingerprints that do not include that generated file. Related open reports #169 and #178 describe the same update-integrity failure and should be handled by one fix.

## What This Changes

The issue reports that `clawhub update --all` flags freshly force-updated skills as `local changes (no match)` because local integrity fingerprints diverge from registry fingerprints.

## Best Possible Solution

Keep this issue open and land one focused CLI/backend integrity fix for this cluster: compare the same canonical published file set on local, downloaded zip, and registry resolve paths; exclude only generated install/download metadata such as root `_meta.json` or stop injecting/extracting it for skill installs; preserve fingerprint coverage for real published text files; add a regression test for force-update followed immediately by normal update.

## Repair Work Prompt

Fix the ClawHub CLI false `local changes (no match)` update cluster covered by this issue plus #169 and #178. Current main hashes local extracted text files, but skill downloads add generated root `_meta.json` while registry resolve compares against published `version.files`, so a fresh install or force update can immediately fail fingerprint matching. Implement one canonical file-set policy across local hashing, downloaded zips, publish fingerprints, and resolve: prefer stopping skill install/update downloads from extracting generated `_meta.json`, or otherwise exclude only generated install/download metadata from local update fingerprints. Do not broadly ignore user-authored JSON files; published JSON content such as `public.json` must still affect integrity fingerprints. Preserve zip path-safety behavior and existing package download behavior. Add regression coverage for install/force-update followed immediately by normal update reporting up to date, for generated metadata not affecting local match, and for a real published text/JSON edit still producing local-change detection. Do not change unrelated CLI commands or release metadata.
