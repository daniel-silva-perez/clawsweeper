## Summary

Keep open. Current main still contains the intentional deployment-drift alert path when frontend and Convex build SHAs are both present and unequal, and the ClawHub cleanup policy says issues should stay open for maintainer handling. A live static-asset check currently did not expose `VITE_APP_BUILD_SHA`, so the exact production state may have shifted, but that makes this deployment-stamping/product-policy triage rather than an auto-close candidate.

## What This Changes

Investigate and stop the dashboard from showing a persistent deployment version drift warning when the frontend build SHA and Convex `APP_BUILD_SHA` differ even though the dashboard otherwise works.

## Best Possible Solution

Maintain a clear deployment contract: both deployables should carry the same main SHA for coordinated production deploys, and the banner policy should only alert when that drift is actionable. If maintainers want partial deploys to be normal, adjust the banner severity/copy and update the drift tests and deploy docs together; keep #618 separate unless a broader version-display policy is later identified.
