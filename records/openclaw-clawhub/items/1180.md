## Summary

Keep open. Current main still intentionally shows the dashboard deployment-drift alert when frontend and Convex SHAs are both present and unequal, and the ClawHub review policy for this run keeps issues open rather than auto-closing them. The remaining question is operational/product policy: whether production stamping is wrong, or whether partial deploy drift should be treated differently.

## What This Changes

The issue asks to resolve a persistent dashboard deployment-drift warning where the frontend build SHA and Convex-reported backend SHA differ while the dashboard otherwise works.

## Best Possible Solution

Keep a clear production deployment contract: coordinated production deploys should stamp the same main SHA into both Convex and the frontend, and the dashboard should alert only when that drift is actionable. If partial deploy drift is expected, update the banner severity/copy, tests, and deploy docs together; if the frontend stamp is missing accidentally, fix the Vercel/deploy configuration instead.
