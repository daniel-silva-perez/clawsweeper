## Summary

Keep this issue open. ClawHub cleanup policy keeps open issues in maintainer hands, and current main still preserves the reported split: authenticated downloads use a per-user bucket, while anonymous downloads use an IP bucket that can fall back to `ip:unknown` when no trusted client IP header is available. The source, docs, and tests make the false-positive path plausible, but the actual fix needs production proxy/header and rate-limit log review.

## Best Possible Solution

Keep this open for maintainer and ops follow-up. Inspect production `rate_limit_denied` logs for `/api/v1/packages/*/download` and `/api/v1/download`, especially `kind`, auth state, `ipSource`, `hasClientIp`, and bucket identity. If anonymous traffic is collapsing to `ip:unknown` or a shared proxy IP, fix the deployed proxy/header configuration or adjust anonymous download bucket behavior with an abuse-protection review. Track the OpenClaw install-client retry-header display gap separately if that client still hides `Retry-After` and rate-limit metadata.
