## Summary

Keep this issue open. The current main branch still has the reported split: valid Bearer-token downloads use a per-user bucket and bypass the anonymous IP bucket, while anonymous downloads use an IP-derived bucket that falls back to `ip:unknown` when no trusted client IP header is available. The false-positive path is plausible, but resolving it requires production proxy/header and rate-limit log review plus abuse-protection judgment. Also, the ClawHub cleanup policy for this repository only permits automated closes for PRs that are certainly implemented on main, not open issues like this one.

## Best Possible Solution

Keep this open for maintainer and ops follow-up. Inspect production `rate_limit_denied` logs for `/api/v1/packages/*/download` and `/api/v1/download`, especially `kind`, auth state, `ipSource`, `hasClientIp`, and bucket identity. If anonymous traffic is collapsing to `ip:unknown` or a shared proxy IP, fix deployed proxy/header configuration or adjust anonymous download bucketing with an abuse-protection review. Track the OpenClaw install-client retry-header display gap separately if that client still hides `Retry-After` and rate-limit metadata.
