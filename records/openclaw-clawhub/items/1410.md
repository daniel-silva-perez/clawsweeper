## Summary

Keep this issue open. Current main still has the reported anonymous/authenticated split: authenticated package or skill downloads use a per-user rate bucket, while anonymous downloads use an IP bucket that can fall back to ip:unknown when no trusted client-IP header is available. The source tree supports the plausibility of the report, but confirming the live false positive needs production rate-limit logs and proxy/header review, and repository cleanup policy for ClawHub issues does not allow an automated close here.

## What This Changes

The issue reports that anonymous ClawHub plugin installs can immediately hit a 429 download rate limit while the same install succeeds after authentication, suggesting anonymous IP bucketing may collapse to a shared or unknown bucket.

## Best Possible Solution

Keep this issue open for maintainer and ops follow-up. Inspect production rate_limit_denied logs for /api/v1/packages/*/download and /api/v1/download, including kind, auth state, ipSource, hasClientIp, and effective bucket identity. If anonymous traffic is collapsing to ip:unknown or a shared proxy IP, fix deployed proxy/header configuration or adjust anonymous download bucketing after abuse-protection review; track any OpenClaw install-client retry-header display gap separately.
