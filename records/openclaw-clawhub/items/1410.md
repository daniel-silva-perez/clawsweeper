## Summary

Keep open. This is an open ClawHub issue, and the repository policy for this cleanup pass only allows auto-closing PRs that are already implemented on main; the current code still has the reported split where authenticated downloads use a user bucket while anonymous downloads can fall back into a shared `ip:unknown` bucket if no trusted client IP is available.

## What This Changes

The issue reports anonymous plugin package downloads returning immediate 429 responses while the same install succeeds after ClawHub login, suggesting anonymous client-IP attribution or bucketing is misconfigured in production.

## Best Possible Solution

Keep the issue open until maintainers inspect production rate-limit logs and proxy/header configuration, then either correct real client-IP attribution or intentionally adjust anonymous download bucketing with abuse-protection review. Any client-side retry-header display gap should be tracked in the repository that owns the OpenClaw install command if it is outside ClawHub.
