## Summary

Keep open. The cleanup policy for ClawHub issues does not allow an automated close here, and current main still has the reported anonymous/authenticated split: authenticated downloads use a user bucket, while anonymous downloads use an IP bucket that can fall back to `ip:unknown` when no trusted client IP is available. The remaining decision needs production proxy/header and rate-limit log review.

## What This Changes

The issue reports anonymous plugin/package installs returning immediate 429s on a quiet VPS while the same install succeeds after ClawHub login, suggesting anonymous client IP attribution or bucketing may be wrong.

## Best Possible Solution

Keep the issue open until maintainers inspect production rate-limit logs and proxy/header configuration, then either correct real-client-IP attribution or deliberately adjust anonymous download bucketing with an abuse-protection review. Track any remaining install-client retry-hint gap separately if it lives outside ClawHub.
