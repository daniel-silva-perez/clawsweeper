## Summary

Keep open. This is an open ClawHub issue, and the repository-specific cleanup policy only allows auto-closing PRs that are certainly implemented on main. Current main still blocks same-GitHub sign-in after account deletion, and related #504 remains open for the same account lifecycle problem.

## Best Possible Solution

Keep this issue open and triage it with #504. Maintainers should choose the product policy: either keep deletion permanently irreversible with clearer login/support messaging, or implement a safe same-GitHub re-registration/relink flow that retires the old auth binding without restoring banned accounts. Add regression coverage for delete-account followed by same-GitHub sign-in once the policy is chosen.

## Risks / Open Questions

- A same-GitHub re-registration flow must preserve the distinction between self-deleted users and banned or auto-banned users so OAuth rebinding cannot bypass moderation.
- Production accounts that already have stale GitHub auth bindings to deactivated user rows may need a support/admin recovery path even if product policy stays irreversible.
- Clearer frontend messaging reduces the silent loop, but it does not implement the reporter's requested re-signup behavior.
