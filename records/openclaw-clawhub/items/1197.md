## Summary

Keep open. This is an open ClawHub issue, and the repository cleanup policy for this pass keeps issues open. Substantively, current main still has the reported Convex failure shape: first publish reaches personal publisher setup, publisher writes schedule owner-publisher digest sync, and the owner-publisher package/skill digest helpers can still call `.paginate()` multiple times inside one internal mutation invocation.

## What This Changes

The issue reports first-publish failures caused by Convex's single-paginated-query limit in owner-publisher skill search digest synchronization.

## Best Possible Solution

Land one narrow backend fix for the publish-pagination cluster: make the owner-publisher package and skill digest sync internal mutations process at most one paginated page per invocation, pass continuation cursors through self-scheduled follow-up mutations, preserve indexed `by_owner_publisher` reads, and add regression tests for multi-page package and skill owner-publisher digest sync. After the fix is merged and deployed, consolidate duplicate reports such as #1195 and #1171.

## Repair Work Prompt

Fix the ClawHub publish-pagination cluster reported by this issue and related reports #1195 and #1171. In `convex/functions.ts`, convert `syncPackageSearchDigestsForOwnerPublisherId` and `syncSkillSearchDigestsForOwnerPublisherId` plus their internal mutations to process one `.paginate()` page per invocation and self-schedule continuation work with a cursor when `isDone` is false. Preserve indexed `by_owner_publisher` reads; do not replace them with `.filter()` or table scans. Keep the existing publisher trigger behavior that schedules package and skill rebuilds separately. Add focused regression coverage in `convex/functions.test.ts` for multi-page owner-publisher package and skill sync: simulate a first page with `isDone: false`, assert only one paginate call happens in that invocation, assert a continuation mutation is scheduled with the continue cursor, and verify the next invocation uses that cursor. Do not change CLI/API publish semantics, auth, publisher ownership rules, or unrelated digest behavior. No changelog is required unless the repository convention expects one for backend bug fixes.
