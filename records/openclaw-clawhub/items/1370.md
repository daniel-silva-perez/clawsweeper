## Summary

Keep this issue open. The ClawHub cleanup policy for this review keeps issues open, and current main still only partially covers plugin package lifecycle: package data and an authenticated DELETE API path exist, but package restore/unhide, version de-list controls, CLI lifecycle commands, owner-facing UI controls, and docs are still missing.

## Best Possible Solution

Keep this issue open and complete plugin package lifecycle support end to end: decide package-level versus release-level semantics, add package undelete/unhide backend and HTTP API paths, add CLI commands such as `clawhub package delete|undelete` with hide/unhide aliases, expose owner-facing controls on package/plugin pages or an owner dashboard, keep withdrawn packages discoverable to owners for restore, and update CLI/API docs with the rollback workflow for plugin publishers.

## Risks / Open Questions

- The existing package DELETE endpoint can withdraw a package, but without a documented self-service restore path owners may still need maintainer or database intervention after a temporary rollback.
- Version-level de-list and restore semantics remain unresolved, especially around latestReleaseId, tags, and avoiding accidental re-exposure of a broken release.
- Owner-facing package lifecycle authorization needs coverage for owner, publisher membership, moderator/admin, scoped package names, and non-owner denial before UI or CLI exposure.
