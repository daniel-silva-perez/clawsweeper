## Summary

Keep open. Current main still only partially addresses package lifecycle: packages and package releases can be soft-deleted through an authenticated DELETE API path, but there is no package undelete/unhide API, no owner-facing CLI lifecycle command, no owner UI hide/delete/restore workflow, and the docs still present delete/undelete/hide/unhide as skill-only. The cleanup policy for openclaw/clawhub also keeps issues open rather than auto-closing them.

## Best Possible Solution

Keep this issue open and complete package lifecycle support end to end: decide package-level versus version-level semantics, add package undelete/unhide backend and HTTP API paths, add CLI commands such as `clawhub package delete|undelete` with hide/unhide aliases, expose owner-facing controls on the package dashboard/detail surface, keep soft-deleted packages visible in the owner restore path, and update CLI/API docs with the rollback workflow for plugin publishers.

## Risks / Open Questions

- The existing package DELETE endpoint can withdraw a package but has no documented self-service restore counterpart, so owners may still need maintainer or database intervention after a temporary withdrawal.
- Version-level de-list/restore semantics remain unresolved; restoring a package may need to restore latest release and tags carefully without re-exposing a known-broken release.
- Package lifecycle authorization should be tested for owner, publisher membership, moderator/admin, scoped package names, and non-owner denial before exposing owner-facing controls.
