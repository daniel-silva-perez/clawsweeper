## Summary

Keep this issue open. It is an issue in openclaw/clawhub, and the provided cleanup policy says ClawHub issues should not be auto-closed. Current main also does not fully implement the requested owner-facing plugin package lifecycle: package soft-delete storage and a DELETE API exist, but owner-facing restore/undelete, hide/unhide aliases, CLI commands, dashboard controls, docs, and per-version package de-list workflows are still missing.

## Best Possible Solution

Keep this issue open and finish plugin package lifecycle support end to end: define package-level versus release-level semantics, add package restore/undelete and optional package-release de-list backend/API paths, expose owner-facing CLI commands and dashboard controls, include a deleted/hidden package restore view, and document the rollback workflow for plugin publishers. Coordinate per-version behavior with open #905 and validate against cleanup cases like #1744.

## Risks / Open Questions

- The existing package DELETE endpoint can withdraw a bad package, but without an owner-facing restore/undelete path, publishers may still need maintainer or database intervention afterward.
- Per-version package de-list semantics remain unresolved around latestReleaseId, tags, install resolution, and avoiding accidental re-exposure of a bad release.
- Because dashboard owner package lists filter out soft-deleted packages, a package can disappear from the publisher UI before there is a matching self-service recovery workflow.
