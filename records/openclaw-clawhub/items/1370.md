## Summary

Keep open. This is a ClawHub issue, and the repository cleanup policy for this review keeps issues open. Current main only partially addresses the request: package soft-delete data and an authenticated DELETE API path exist, but owner-facing restore/unhide, version de-list, CLI lifecycle commands, site controls, and docs are still missing.

## Best Possible Solution

Keep this issue open and finish plugin package lifecycle support end to end: define package-level versus release-level semantics, add package restore/unhide and optional release de-list backend/API paths, expose CLI commands such as `clawhub package delete|undelete` with hide/unhide aliases, add owner-facing site controls including a way to see withdrawn packages for restore, and document the rollback workflow for plugin publishers. Coordinate any per-version design with the related open per-version lifecycle request #905.

## Risks / Open Questions

- The existing package DELETE endpoint can withdraw a broken package, but without a restore/unhide path owners may still need maintainer or database intervention afterward.
- Version-level de-list semantics remain unresolved, especially around latestReleaseId, tags, and avoiding accidental re-exposure of a bad release.
- Owner-facing authorization and UX still need coverage for package owners, publisher memberships, moderators/admins, scoped package names, and non-owner denial.
