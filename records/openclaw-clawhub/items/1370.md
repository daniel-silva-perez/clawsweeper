## Summary

Keep this issue open. For openclaw/clawhub, the cleanup policy only proposes auto-closing PRs that are certainly implemented on main, and this is an issue. Current main also still only partially covers the requested plugin package lifecycle: package and release soft-delete storage plus an authenticated package DELETE API exist, but owner-facing package restore/unhide, CLI lifecycle commands, dashboard controls, docs, and per-version de-list workflows are still missing.

## Best Possible Solution

Keep this issue open and finish plugin package lifecycle support end to end: define package-level versus package-release-level semantics, add package restore/unhide and optional release de-list backend/API paths, expose owner-facing CLI commands such as `clawhub package delete|undelete` with hide/unhide aliases, add dashboard controls plus a deleted/hidden package view for restore, and document the rollback workflow for plugin publishers. Coordinate per-version behavior with related open issue #905 and validate it against real cleanup cases like #1744.

## Risks / Open Questions

- The existing package DELETE endpoint can withdraw a broken package, but without an owner-facing restore/unhide path, publishers may still need maintainer or database intervention afterward.
- Per-version de-list semantics remain unresolved around latestReleaseId, tags, install resolution, and avoiding accidental re-exposure of a bad release.
- Dashboard filtering hides soft-deleted packages from owner package lists, which works for public suppression but makes self-service recovery harder unless a separate deleted/hidden view is added.
