## Summary

Keep open. The ClawHub cleanup policy keeps issues open, and current main still leaves the reported failure mode reachable: anonymous installs consume an IP or `unknown` download bucket, while `clawhub install` only authenticates when a token is already saved. The login/token, `pnpm dlx`, VPN-node, and skills.sh workarounds in the discussion are useful, but related #1410 and #349 remain open for the underlying anonymous-install and rate-limit policy work.

## What This Changes

The issue asks ClawHub to stop or better explain `Rate limit exceeded` failures when installing skills with `npx clawhub@latest install <skill>`.

## Best Possible Solution

Keep this issue open while maintainers consolidate it with #1410 for anonymous-install attribution and #349 for rate-limit policy/visibility. The desired end state is reliable anonymous bucket attribution in production, a clear product decision on whether installs should prompt for or require login before rate-limited downloads, and install-specific 429 guidance that points users to `clawhub login` or `clawhub login --token ...` when authentication would move them to a user bucket.
