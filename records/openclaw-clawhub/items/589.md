## Summary

Keep open. The ClawHub cleanup policy keeps issues open, and current main still leaves the reported install failure mode reachable: anonymous installs can consume an IP or `unknown` download bucket, while `clawhub install` only sends authentication when a token is already saved. The login/token, `pnpm dlx`, VPN-node, and skills.sh workarounds in the discussion are useful, but the underlying install/rate-limit decision still needs maintainer handling with related #1410 and #349.

## What This Changes

The issue reports `npx clawhub@latest install <skill>` failing with `Rate limit exceeded` across networks and asks for the install/rate-limit behavior or guidance to be fixed.

## Best Possible Solution

Keep this issue open while maintainers consolidate it with #1410 for anonymous-install attribution and #349 for rate-limit policy/visibility. The desired end state is reliable anonymous bucket attribution in production, a clear product decision on whether installs should prompt for or require login before rate-limited downloads, and install-specific 429 guidance that points users to `clawhub login` or `clawhub login --token ...` when authentication would move them to a user bucket.
