## Summary

Keep open. The repository policy keeps ClawHub issues open, and current main still leaves the reported failure mode reachable: anonymous installs consume an IP or `unknown` download bucket, while CLI install only sends authentication when a token is already saved. The discussion’s login/token, `pnpm dlx`, VPN-node, and skills.sh workarounds are useful, but they do not remove the underlying production attribution, rate-limit policy, and install UX question.

## What This Changes

The issue requests fixing `npx clawhub@latest install <skill>` returning `Rate limit exceeded` across networks, or at least improving install/login guidance for affected users.

## Best Possible Solution

Keep this issue open while maintainers consolidate it with #1410 for anonymous-install attribution and #349 for rate-limit policy/visibility. The desired end state is reliable anonymous bucket attribution in production, a clear product decision on whether installs should prompt for or require login before rate-limited downloads, and install-specific 429 guidance that points users to `clawhub login` or `clawhub login --token ...` when authentication would move them to a user bucket.
