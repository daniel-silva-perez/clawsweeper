## Clownfish Work Prompt

Implement a focused fix for the VirusTotal Code Insight false-positive display gap. Current main already suppresses uncorroborated VT Code Insight suspicious results in backend moderation when `scanner === 'code_insight'`, ClawScan/static scans are clean, and VT engineStats have 0 malicious and 0 suspicious hits. However, UI and v1 API display paths still read raw `vtAnalysis.status` or `vtAnalysis.verdict` and can show `VirusTotal: Suspicious` for clean-engine Code Insight analyses.

Expected boundary:
- Add or extract a shared normalization helper for VT display/security status. It should treat Code Insight-only `suspicious` as clean/undetected only when engineStats malicious and suspicious are both 0 and corroborating ClawScan/static signals are clean or absent per the existing backend rule.
- Preserve `suspicious`/`malicious` for any VT AV engine suspicious/malicious count, malicious VT status, non-clean ClawScan/static signal, or insufficient engineStats.
- Keep Code Insight text visible as context where useful; do not erase `analysis` or external report links.
- Apply the normalized status consistently to `SecurityScanResults` badges/panel rows/static guidance, `DetailSecuritySummary`, `SecurityScannerPage`, and `convex/httpApiV1/skillsV1.ts` security snapshots for `/scan` and `/versions/:version`.
- Consider whether existing persisted banner state needs reconciliation or whether the current backend clearing paths are sufficient; do not add broad migrations unless needed.
- Do not change unrelated scanner policy, package security behavior, release workflows, or dependency metadata.

Add regression tests for the Code Insight-only 0-engine-hit case in the UI and v1 API, and keep existing tests that prove AV-engine suspicious hits remain suspicious. No changelog entry is required unless the repo already expects one for this bug class.
