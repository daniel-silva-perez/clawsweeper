## Summary

Keep open. Current main has a backend moderation suppression rule for uncorroborated VirusTotal Code Insight suspicious results, but visible skill/security UI and the v1 security snapshot still read raw `vtAnalysis.status` or `verdict`; ClawHub cleanup policy also keeps issues open unless they are implemented PRs.

## What This Changes

The issue asks ClawHub to show a clean or undetected VirusTotal badge and remove the warning banner when VirusTotal has zero malicious or suspicious engine detections.

## Best Possible Solution

Add one shared VirusTotal display/status normalization helper used by skill UI badges, scanner detail pages, and v1 security snapshots. It should reuse the backend Code Insight suppression semantics, keep Code Insight text visible as context, and leave stronger warnings intact whenever another signal corroborates risk.

## Repair Work Prompt

Fix the VirusTotal zero-detection Code Insight false-positive display/API gap. Add a shared normalization path for skill VirusTotal display/security status that reuses the current backend suppression semantics: when `vtAnalysis.scanner === 'code_insight'`, VT status/verdict is suspicious, malicious and suspicious AV engine counts are both zero, ClawScan is clean/benign, and static scan is clean or absent, display/API normalized VT status should be clean or undetected rather than suspicious. Keep Code Insight analysis text visible as context. Do not suppress malicious VT results, non-Code Insight suspicious results, suspicious/malicious AV engine counts, suspicious static scan findings, suspicious/malicious ClawScan verdicts, manual moderation, or blocked malware flags. Likely update `src/components/SkillSecurityScanResults.tsx`, `src/components/DetailSecuritySummary.tsx`, `src/components/SecurityScannerPage.tsx`, `convex/httpApiV1/skillsV1.ts`, and focused tests. Do not change VirusTotal ingestion or broad moderation policy. Add regression coverage for a Code Insight suspicious result with `engineStats: { malicious: 0, suspicious: 0 }` becoming clean in UI/API display, and a corroborated suspicious engine count remaining suspicious. No changelog is required unless maintainers prefer user-visible release notes for the false-positive fix.
