## Summary

Keep open. Current main still has only token-paste and browser loopback CLI login, no device-code CLI flag or server/verification endpoints, and #1867 is an open unmerged implementation PR that references this issue with closing syntax.

## What This Changes

Request GitHub-style device-flow authentication for `clawhub login` so headless agents can obtain and store ClawHub API tokens without a localhost browser callback.

## Best Possible Solution

Keep this issue open and use #1867 as the current implementation candidate or replacement starting point. A complete fix should add CLI `--device` support, route constants, server-side device-code issuance and polling endpoints, an authenticated verification page, secure API-token creation/storage, rate limiting, and tests for success, pending, slow_down/backoff, expiry, denial, replay, and persistence.

- #1867 is open and described as client-side implementation with server-side requirements, so merging only the client path would not solve the reported headless-auth problem.
- The current manual token-paste workaround still requires an already browser-authenticated user to create or obtain the first token.
