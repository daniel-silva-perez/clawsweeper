## Summary

Keep this PR open. Current main still scans full `skills` documents, joins version and owner documents, and returns version files eagerly, so the central digest-based performance change is not implemented on main. The PR branch also introduces blocking owner-correctness problems around handle-less users and stale/deactivated owner metadata, while the later fast-forward retry portion is already present on main.

## What This Changes

This PR changes GitHub backup sync to paginate `skillSearchDigest`, lazily fetch version files after a GitHub version mismatch, remove the missing-owner stat, update backup pagination tests, and add retry handling for GitHub backup push conflicts.

## Best Possible Solution

Land a revised digest-based scan that keeps the scan phase lightweight, preserves the live owner-id fallback for handle-less owners, validates current owner eligibility before backup writes, fetches version files only after a version mismatch, and keeps the fast-forward retry behavior already present on main.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve owner fallback for handle-less users:** `convex/githubBackups.ts:75`
  - body: The digest writer stores `ownerHandle: ''` when a visible owner has no handle, while current main falls back to `owner._id`. This guard skips those rows entirely, so public skills from handle-less owners stop being mirrored. Resolve a fallback owner path instead of treating the empty string as ineligible.
  - confidence: 0.94
- **[P1] Revalidate owner state before trusting digest rows:** `convex/githubBackups.ts:72-75`
  - body: The previous scan read the live owner and excluded deleted or deactivated users. This replacement only trusts `skillSearchDigest`, while user deletion/deactivation does not directly refresh skill digests by owner user id, so stale rows can still be backed up and then pruned repeatedly. Add live owner validation or guarantee digest refresh for those owner-state changes before enqueueing backups.
  - confidence: 0.88
- **[P2] Resolve the current owner path before backup lookup:** `convex/githubBackups.ts:84`
  - body: Using `digest.ownerHandle` directly can send lookups and backup writes to a stale owner path after a handle change for legacy or owner-user-only skills whose digest was not refreshed. Since prune compares against the live owner path, this can create backup churn. Resolve the current owner handle or fallback before calling GitHub.
  - confidence: 0.8
