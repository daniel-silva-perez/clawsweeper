## Summary

Keep this PR open. Current main does not implement the PR’s primary digest-based GitHub backup pagination: `getGitHubBackupPageInternal` still paginates full `skills` rows, joins `skillVersions` and `users`, and returns version `files` during page construction. Main does contain the later GitHub fast-forward retry behavior, but that only covers one part of the PR and does not satisfy the performance change in the title/body.

## Best Possible Solution

Keep this PR open for maintainer/contributor follow-up. The best path is to revise the digest-based scan so it keeps the intended low-byte `skillSearchDigest` pagination, preserves the old eligibility semantics, handles empty owner handles with the user-id fallback, revalidates or refreshes live owner state before choosing backup paths, fetches version files lazily only after the GitHub version mismatch check, and retains the fast-forward retry behavior already present on main.

## Risks / Open Questions

- Closing this PR as implemented would leave the main performance issue unresolved because current main still reads full skill docs plus version and owner docs during backup page construction.
- Merging the PR as described needs care around owner semantics: trusting digest owner metadata alone can skip handle-less owners, write backups under stale renamed handles, or briefly mirror deactivated/deleted-owner content before prune removes it.
- Security/supply-chain pass: the provided PR file set touches Convex backup code and GitHub API write paths, but does not add dependencies, CI workflow changes, install scripts, lockfile changes, package publishing metadata, downloaded artifacts, or broader secret permissions. The security-relevant concern is correctness of what content gets mirrored to GitHub.
