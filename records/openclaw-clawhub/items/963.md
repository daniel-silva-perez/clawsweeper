## Summary

Keep open. Current main still lacks the central digest-based backup scan, so this PR is not obsolete; however, the proposed patch should not merge unchanged because it drops existing owner fallback and live owner validation in a GitHub backup path.

## What This Changes

This PR changes GitHub backup sync to scan `skillSearchDigest`, lazily fetch version files after a GitHub version mismatch, remove the missing-owner stat, and add fast-forward retry handling for GitHub backup pushes.

## Best Possible Solution

Land a revised digest-based scan that keeps the scan phase lightweight, preserves owner-id fallback for handle-less owners, validates current owner eligibility before backup writes, fetches version files only after a version mismatch, and retains the fast-forward retry behavior already on main.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve owner fallback for handle-less users:** `convex/githubBackups.ts:75`
  - body: The digest writer stores `ownerHandle: ''` when a visible owner has no handle, while current main falls back to `owner._id` before backing up. This new guard skips those rows entirely, so public skills from handle-less owners stop being mirrored. Resolve a fallback owner path instead of treating the empty string as ineligible.
  - confidence: 0.94
- **[P1] Revalidate owner state before trusting digest rows:** `convex/githubBackups.ts:72-75`
  - body: The previous scan read the live owner and excluded deleted or deactivated users. This replacement only trusts `skillSearchDigest`, while user deactivation/deletion does not directly refresh skill digests by owner user, so stale rows can be backed up and then pruned repeatedly. Add live owner validation or guarantee digest refresh for those owner-state changes before enqueueing backups.
  - confidence: 0.88
- **[P2] Resolve the current owner path before backup lookup:** `convex/githubBackups.ts:84`
  - body: Using `digest.ownerHandle` directly can send lookups and backup writes to a stale owner path after a handle change for legacy or ownerUserId-only skills whose digest was not refreshed. Since prune compares against the live owner path, this can create backup churn. Resolve the current owner handle or fallback before calling GitHub.
  - confidence: 0.8
