## Summary

Keep this PR open. Current main still does not implement the central digest-based GitHub backup pagination or lazy version-file loading proposed here: the backup page query still paginates full `skills` documents, joins `skillVersions` and `users` per row, and passes `version.files` eagerly into the Node backup action. The PR also remains a normal open implementation branch with review concerns around owner-handle fallback and stale owner state, so it needs contributor/maintainer follow-up rather than cleanup closure.

## What This Changes

This PR changes GitHub backup sync to paginate `skillSearchDigest`, remove eager owner/version joins during the scan phase, lazily fetch version files only when a backup is needed, remove the dead missing-owner stat, and add fast-forward retry handling for backup Git pushes.

## Best Possible Solution

Keep this PR open for contributor or maintainer follow-up. The best path is to revise the digest-based backup scan so `getGitHubBackupPageInternal` pages `skillSearchDigest`, fetches version files lazily only after a GitHub version mismatch, preserves the current owner-id fallback for owners without handles, and either revalidates live owner state or reliably refreshes skill digest owner metadata when users rename, deactivate, or delete accounts. Keep the already-present fast-forward retry behavior on main.
