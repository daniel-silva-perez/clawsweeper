## Summary

Keep open. Current main still does not implement the PR's digest-based scan and lazy version-file fetch, so this cannot be closed as already implemented; the PR branch also needs revision for owner fallback and stale-owner correctness in the GitHub backup write path.

## What This Changes

This PR changes GitHub backup sync to scan `skillSearchDigest`, lazily fetch version files after a GitHub version mismatch, remove the missing-owner stat, update backup pagination tests, and add fast-forward retry handling for GitHub backup pushes.

## Best Possible Solution

Land a revised digest-based scan that keeps the scan phase lightweight, preserves owner-id fallback for handle-less owners, validates current owner eligibility before backup writes or guarantees digest refresh for owner state changes, resolves the current backup owner path, fetches version files only after a version mismatch, and retains the fast-forward retry behavior already on main.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve owner fallback for handle-less users:** `convex/githubBackups.ts:75`
  - body: The digest writer can store `ownerHandle: ''` for visible owners without handles, while current main falls back to `owner._id`. This guard skips those rows entirely, so public skills from those owners stop being mirrored. Resolve a fallback owner path instead of treating an empty string as ineligible.
  - confidence: 0.94
- **[P1] Revalidate owners before enqueueing backups:** `convex/githubBackups.ts:72-75`
  - body: The previous scan read the live owner and excluded deleted or deactivated users. This replacement trusts `skillSearchDigest`, but user deletion/deactivation does not directly refresh skill digests by owner user id, so stale rows can still be backed up before cleanup deletes them again.
  - confidence: 0.88
- **[P2] Resolve the current owner path before backup lookup:** `convex/githubBackups.ts:82-84`
  - body: Using `digest.ownerHandle` directly can send GitHub lookups and writes to a stale owner path after a handle change for legacy or ownerUserId-only skills whose digest was not refreshed. Since prune compares against the live owner path, this can create repeated recreate/delete churn.
  - confidence: 0.8

## Security Review

Revalidate owner state or guarantee digest refresh on user state changes before this lands.
