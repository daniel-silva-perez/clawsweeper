## Summary

Keep this PR open. Current main has the fast-forward retry portion in the GitHub backup helper, but it does not implement the PR’s central digest-based backup scan: `getGitHubBackupPageInternal` still paginates `skills`, joins `skillVersions` and `users` per row, and returns version files eagerly. The PR also has unresolved review concerns around owner handles and stale owner status if backup sync trusts `skillSearchDigest` owner fields without a fallback or live revalidation.

## What This Changes

This PR changes GitHub backup sync to paginate `skillSearchDigest`, lazily fetch version files only when a backup is needed, remove the missing-owner stat, update tests for digest pagination, and add retry handling for GitHub backup push conflicts.

## Best Possible Solution

Keep this PR open for maintainer or contributor follow-up. The best path is to land a digest-based backup scan only after it preserves the current owner-id fallback, handles stale cursor migration visibly, fetches version files lazily after a GitHub version mismatch, and either revalidates live owner state or guarantees `skillSearchDigest` owner metadata is refreshed when users rename, deactivate, or delete accounts. The fast-forward retry behavior already present on main should be retained.
