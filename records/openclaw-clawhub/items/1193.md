## Summary

Keep open. Current main still allows the reported API-token identity path to return a missing handle as `unknown`, and publishing still depends on personal-publisher resolution that may require production account repair. The related digest pagination path has a background scheduling improvement, but that does not prove the reporter's null/wrong handle and missing publisher state are fixed.

## What This Changes

The issue asks ClawHub to make CLI login, `whoami`, and publishing resolve the reporter's GitHub handle and personal publisher after OAuth/API-token login.

## Best Possible Solution

ClawHub should guarantee that an authenticated API-token user has a canonical GitHub handle and active personal publisher before login, `whoami`, or publish can appear successful, then repair existing bad production rows and add regressions for null/wrong-handle users across CLI and web publish flows.
