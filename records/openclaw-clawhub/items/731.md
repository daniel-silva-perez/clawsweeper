## Summary

This issue should stay open. Current main has made scanner findings more structured across the backend, API, UI, and CLI-facing publish flow, but it still does not fully satisfy the requested actionable scanner contract: remediation guidance is not modeled, per-finding blocking versus informational semantics are not exposed, and there is no local `clawhub scan <path> --format json` workflow.

## Best Possible Solution

Keep this issue open for maintainer-owned scanner transparency work: define a stable finding contract covering rule/category, severity, file and line range, redacted evidence policy, remediation guidance, and blocking versus informational semantics; then wire it consistently through Convex schema/API, owner/staff redaction, the skill UI, CLI publish and local scan UX, docs, and tests. Related #990 should remain a narrower follow-up for static/API suspicious-status visibility, not a substitute for this broader remediation contract.
