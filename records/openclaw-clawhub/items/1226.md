## Clownfish Work Prompt

Implement a focused CLI fix for this issue: `clawhub package publish . --family code-plugin` can resolve `.` against the resolved global CLI workdir instead of the caller's plugin repository. When cwd is a plugin repo without a ClawHub marker and a Clawdbot/OpenClaw default workspace exists, the command may inspect the wrong folder and throw `openclaw.plugin.json required` even though the manifest exists in cwd.

Expected boundary:
- Fix local package publish source resolution so relative local sources, especially `.`, inspect the caller's current directory by default.
- Preserve explicit `--workdir` behavior and existing GitHub source behavior (`owner/repo`, `owner/repo@ref`, GitHub URLs, existing local `owner/repo` directories).
- Do not combine this with plugin search/install/list catalog commands.
- Add regression tests for cwd/global-workdir mismatch and for the existing GitHub shorthand/local path behavior.
- Update docs only if the semantics need clarification; no changelog entry unless the repo convention requires one.
