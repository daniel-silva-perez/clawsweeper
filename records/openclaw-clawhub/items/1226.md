## Summary

Keep this issue open. The repository-specific policy for openclaw/clawhub only permits auto-closing pull requests that are certainly implemented on main, and this item is an issue. Substantively, current main still leaves a plausible local path-resolution failure for `clawhub package publish .`: the publish source is resolved against the resolved ClawHub workdir, while that workdir can fall back to configured Clawdbot/OpenClaw state instead of the caller's plugin repository. The reporter's pinned repo commit does contain `openclaw.plugin.json`, so the false missing-manifest report remains credible; once the manifest path is fixed, the pinned package may also need the newer required OpenClaw compatibility metadata.

## Best Possible Solution

Keep this issue open for maintainer triage. Add a regression test for `clawhub package publish . --family code-plugin` from a plugin repository while the configured global workdir points elsewhere, then either resolve explicit local publish paths relative to the caller's cwd or clearly document and enforce the required `--workdir` behavior. After that, surface the newer required compatibility metadata as a distinct validation error, and track the plugin search/install/list CLI request separately if maintainers want that terminal workflow.
