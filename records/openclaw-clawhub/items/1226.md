## Summary

Keep open. This is a ClawHub issue, and the supplied cleanup policy keeps issues open; more importantly, current main still has a plausible source-level path to the reported false manifest error because local package sources are resolved from the CLI global workdir before publish validation.

## What This Changes

The issue reports that `clawhub package publish . --family code-plugin` can reject a plugin repository with `openclaw.plugin.json` present by raising `openclaw.plugin.json required`.

## Best Possible Solution

Update the CLI so local package publish sources resolve against the caller's intended package directory when no explicit workdir override is supplied, while preserving explicit workdir/env behavior and GitHub source handling. Add regression coverage for the configured-workspace versus plugin-cwd case so the next validation error is about actual package metadata, not a missing manifest.

## Repair Work Prompt

Fix the ClawHub CLI package publish path-resolution bug reported here: when a user runs `clawhub package publish . --family code-plugin` from a plugin directory, the local source should resolve to the caller's intended package directory unless an explicit `--workdir`/env override intentionally changes that behavior. Preserve GitHub source parsing/fetching, existing shorthand precedence, explicit workdir/env behavior, and code-plugin compatibility metadata validation. Add regression coverage for a plugin cwd with `openclaw.plugin.json` present while the configured Clawdbot/OpenClaw default workspace points elsewhere; the command should no longer fail with `openclaw.plugin.json required` for the plugin repo. Coordinate with related publish-manifest issues #48 and #1509 if the same source-resolution helper covers skills. Do not broaden into plugin browse/install commands or org namespace inference. Changelog expectation: no broad release-note churn unless the repo has a targeted CLI changelog entry for this bugfix; update docs only if the intended semantics change.
