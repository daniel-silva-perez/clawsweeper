## Summary

Keep open. This is an open ClawHub issue, and the cleanup policy only allows auto-closing ClawHub PRs that are certainly implemented on current main. Substantively, current main still leaves the reported failure path plausible: package publish resolves relative local sources against the resolved CLI workdir, which can be a configured Clawdbot/OpenClaw workspace instead of the caller's plugin repository, so the manifest check can still inspect the wrong folder. The reported GuardClaw commit does have `openclaw.plugin.json` at the repo root, although current main may next surface newer compatibility-metadata validation after the path bug is fixed.

## What This Changes

Fix `clawhub package publish . --family code-plugin` so a valid root `openclaw.plugin.json` in the caller's plugin repository is found instead of failing manifest validation.

## Best Possible Solution

Keep this issue open and land a focused CLI fix for local package publish source resolution. `clawhub package publish . --family code-plugin` should inspect the caller's plugin repository when the default/global workdir points at a configured Clawdbot/OpenClaw workspace, while preserving explicit workdir behavior and existing local/GitHub source precedence. Add a regression test for the cwd-vs-configured-workdir case and ship it in the next CLI release.
