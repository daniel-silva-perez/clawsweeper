## Summary

Keep open. This is an open ClawHub issue, and this cleanup pass only auto-closes ClawHub PRs that are certainly implemented on main. Current main still leaves the reported failure plausible: local `package publish .` sources are resolved under the CLI's resolved workdir, which can differ from the caller's plugin repository, and the reported GuardClaw commit does contain a root `openclaw.plugin.json`.

## What This Changes

The issue asks for `clawhub package publish . --family code-plugin` to accept a valid code-plugin repository whose root contains `openclaw.plugin.json` instead of failing with `openclaw.plugin.json required`.

## Best Possible Solution

Implement a focused CLI fix so local package publish sources resolve against the caller's intended package directory when no explicit workdir override is supplied, while preserving explicit workdir/env behavior and GitHub source precedence. Add regression coverage for the configured-workspace versus plugin-cwd case and keep the compatibility-metadata validation as the next clear error for packages that still lack required OpenClaw metadata.

## Repair Work Prompt

Implement a narrow fix in the package publish flow so local relative package sources resolve against the caller's intended package directory when no explicit global workdir/env override is supplied, while preserving absolute paths, explicit `--workdir` / `CLAWHUB_WORKDIR` behavior, GitHub shorthand/URL precedence, and existing local-directory-over-GitHub-shorthand behavior. Add regression coverage for running from a plugin cwd while the resolved default workspace points elsewhere. Also ensure the reported package would advance to the existing compatibility metadata validation rather than being treated as a successful publish without required `openclaw.compat.pluginApi` / `openclaw.build.openclawVersion`.
