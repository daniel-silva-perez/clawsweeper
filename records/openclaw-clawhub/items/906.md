## Summary

Keep this issue open. The ClawHub cleanup policy for this repository keeps issues open, and current main does not prove the reported production Chrome GitHub OAuth callback/session failure is fixed. The code has callback handling and better error surfacing, but the tests mock the critical session-establishment step and the upload page still returns to the signed-out gate when auth state is not established.

## Best Possible Solution

Keep this issue open and triage it with #647 and #1552 as the likely generic OAuth callback/session cluster, while keeping account-deletion-specific reports such as #1244 separate until account state is confirmed. Maintainers should reproduce the production GitHub OAuth flow in Chrome, collect HAR/cookie/storage state and Convex auth logs, determine whether the PKCE verifier, callback exchange, session cookie, or user mapping is failing, then add an end-to-end or smoke path that verifies callback session creation and recovers the sign-in UI after auth failure.
