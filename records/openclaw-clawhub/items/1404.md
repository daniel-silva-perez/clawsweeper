## Summary

Keep open. This is an open ClawHub issue, and the repository-specific cleanup policy only permits auto-closing PRs that are certainly implemented on main. Current main also leaves both reported concerns actionable: the skill header still uses compact unlabeled indicators, and search has exact full-slug handling plus bounded lexical fallback without reporter-specific `yescan` regression coverage.

## Best Possible Solution

Keep this issue open for maintainer triage. The practical path is to add explicit labels, tooltips, or accessible text for the skill header version and download/package indicators, then investigate the specific `yescan` recall failure against production data: skill visibility, owner state, moderation flags, search digest row, embedding map, deployment version, and fallback candidate coverage. Add a focused regression test for a short slug-token query such as `yescan` matching a multi-token slug like `yescan-scan-universal`.
