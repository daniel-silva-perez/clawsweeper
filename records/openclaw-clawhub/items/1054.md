## Summary

Keep open. This is an open ClawHub issue, and the target-repository cleanup policy keeps issues open unless a maintainer resolves them manually. Current main also shows the symptom is tied to an intentional account-lifecycle boundary: deleted accounts are deactivated/purged, OAuth reauth is blocked, and active-session helpers intentionally return no user for inactive accounts, so the remaining work is maintainer policy/support/UX rather than conservative cleanup closure.

## What This Changes

After a ClawHub account is deleted, GitHub OAuth appears to complete but no authenticated user session/profile is created, leaving the UI logged out.

## Best Possible Solution

Maintainers should choose and document the intended post-deletion lifecycle, then make the implementation consistent: keep permanent blocking with reliable support UX, provide a verified administrative purge/re-registration process, or design a deliberate tombstone flow with tests for OAuth callback, active-session queries, account deletion, and UI error display.

## Security Review

Status: needs_attention

Summary: Account deletion and re-registration semantics are security- and privacy-sensitive, so this should not be changed automatically from the report alone.

Concerns:

- **[medium] Preserve irreversible deletion semantics:** `docs/security.md:96`
  - body: The documented policy says user-initiated deletion is irreversible and cannot be restored by logging in again. Recreating sessions or accounts for the same deleted identity without a maintainer-approved lifecycle could weaken privacy and account-recovery expectations.
  - confidence: 0.9
- **[medium] Avoid automatic account resurrection:** `convex/auth.ts:42`
  - body: The OAuth callback currently blocks deactivated users before normal post-login work. Reversing that behavior or auto-purging records needs explicit review because it changes who can regain access after deletion.
  - confidence: 0.86
