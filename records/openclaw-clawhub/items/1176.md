## Summary

Keep open. ClawHub cleanup policy keeps issues open, and current main still leaves a real Convex pagination failure shape: publisher-triggered package and skill digest rebuilds are scheduled separately now, but each owner-publisher digest helper can still run repeated `.paginate()` calls inside one internal mutation for owners with more than one page of packages or skills.

## Best Possible Solution

Keep this issue open and land one focused backend PR for the related pagination cluster: process at most one owner-publisher package or skill page per internal mutation invocation, then reschedule the same internal mutation with the continuation cursor until complete. Preserve `by_owner_publisher` indexed reads, avoid unbounded `.collect()`, and add tests for multi-page self-rescheduling.

## Clownfish Work Prompt

Fix the Convex owner-publisher digest pagination bug reported by this issue and the related publish/sync cluster. Current main already schedules package and skill owner-publisher digest rebuilds separately, but `syncPackageSearchDigestsForOwnerPublisherId` and `syncSkillSearchDigestsForOwnerPublisherId` still loop over `.paginate({ cursor, numItems: 100 })` inside a single internal mutation invocation. Change the internal mutations to process one page per invocation and reschedule themselves with the continuation cursor until `page.isDone` is true. Preserve the `by_owner_publisher` indexed reads, keep bounded page sizes, and do not replace the queries with unbounded `.collect()`. Add focused tests in `convex/functions.test.ts` for package and skill multi-page continuation scheduling, including assertions that one invocation performs only one paginated read and schedules the next cursor. Update generated Convex API files with `bunx convex codegen` if the internal mutation args change. No changelog entry is expected for this narrow backend fix; the PR summary should mention the related issue cluster and list validation commands run.
