## Summary

Keep open. Repository policy for this ClawHub cleanup only permits auto-closing pull requests that are certainly implemented on main, and this item is an issue. Substantively, current main still lets test/spec/fixture contents feed both LLM and static moderation, while non-forced CLI install/update flows still block or skip suspicious skills. Related PR #273 shipped a partial false-positive reduction, but the targeted test-file exclusion PR #270 was closed unmerged.

## Best Possible Solution

Keep this issue open as the product/security tracking item. Define how ClawHub scanners should treat security-skill tests and fixtures, add regression coverage for .test.*, .spec.*, fixture, mock, and test-directory handling across both LLM and static scan paths, preserve scanning for runtime files, and improve review/override behavior so automated installs and updates are not unnecessarily blocked by known false positives.

## Risks / Open Questions

- Ignoring all test-looking paths outright could become a scanner bypass if malicious runtime code is hidden under test/spec/fixture names, so the eventual fix needs explicit semantics rather than a broad blind spot.
- The false-positive surface spans LLM evaluation, deterministic static scanning, VirusTotal aggregation, manual moderation overrides, and CLI install/update gating; fixing only one path may leave the user-visible problem unresolved.
- Related reports remain open, including #288 for behavioral security signals and #348 for flagged/unreachable skills, so closing this issue would remove the direct tracking item for security-skill false positives.
