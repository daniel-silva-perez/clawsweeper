## Summary

Keep open. This is an open ClawHub issue, and the repository cleanup policy for this pass only permits auto-closing ClawHub PRs that current main certainly implements. Substantively, current main still has no explicit test/spec/fixture/mock semantics in the LLM or static scan paths, while CLI install/update still blocks or skips suspicious skills unless the user confirms or passes --force; related PR #270 was closed unmerged and merged PR #273 only addressed a narrower keyword false-positive class.

## What This Changes

The issue asks ClawHub to remove or revise the security audit flow so security skills with malicious-looking test strings are not flagged and skipped by OpenClaw bots.

## Best Possible Solution

Keep this open for product/security maintainer judgment. The best path is to define explicit scanner semantics for test/spec/fixture/mock/example content, apply that policy consistently across LLM and deterministic static moderation, preserve scanning for runtime files and any install/hook-executed files, and ensure trusted/manual false-positive clearance updates the moderation state consumed by API and CLI install/update paths.
