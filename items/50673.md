## Close Comment

Closing after Codex review because I could not reproduce a current-main bug from this fallback swap. On `main`, the compact workflow resolves the effective compaction target before safeguard setup in `src/agents/pi-embedded-runner/compact.ts:326`, and that resolved model is what gets stored in safeguard runtime by `src/agents/pi-embedded-runner/extensions.ts:100`. In the safeguard itself, `src/agents/pi-hooks/compaction-safeguard.ts:767` documents that `ctx.model` is undefined in the `compact.ts` path, so the current `ctx.model ?? runtime?.model` logic already falls through to `runtime.model` there. In the other production call site, `src/agents/pi-embedded-runner/run/attempt.ts:1117` passes the active attempt model into the same runtime slot, so reordering the `??` chain does not change behavior on current `main`. Main also now resolves `agents.defaults.compaction.model` earlier for compaction runtime/context-engine flows in `src/agents/pi-embedded-runner/compaction-runtime-context.ts:31` (covered by `src/agents/pi-embedded-runner/run/attempt.test.ts:2983`). If there is still a concrete failing path on a newer checkout, please open a fresh issue with the exact compaction entrypoint and model transition sequence.
