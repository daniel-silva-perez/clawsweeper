## Summary

Keep #70007 open. Current main has partial non-video attachment storage for Control UI/chat.send, but it still does not provide a documented or working custom-client voice ingress path that routes audio attachments through server-side audio understanding/transcription. `/v1/responses` still has no `input_audio`, `input_file` defaults exclude audio MIME types, and the chat.send audio/file path stores non-image attachments as media refs without putting those offloaded media paths into the dispatch context that triggers media understanding.

## Best Possible Solution

Keep the issue open for maintainer API/product design. The best path is to choose one supported contract, likely typed `chat.send` audio attachments or an authenticated upload/media-ref flow, then wire offloaded audio into MediaPath/MediaType so `tools.media.audio` runs, and document whether `/v1/responses` will support `input_audio` or intentionally remains text/image/file-only.

## Risks / Open Questions

- Current Control UI already allows audio files as non-video attachments for display/storage, so the eventual fix should preserve that behavior while adding explicit transcription semantics.
- The API choice affects public gateway protocol and docs; the safest direction is additive schema/docs plus tests rather than relying on the existing unknown attachment shape.
