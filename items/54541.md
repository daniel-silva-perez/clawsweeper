## Summary

Current `main` does not auto-run arbitrary skill-side scripts during discovery/load. Skill loading reads and parses `SKILL.md`, ClawHub install copies a skill folder into `workspace/skills`, and executable install actions only run through explicit `metadata.openclaw.install` handling. The issue’s reported `main()` auto-execution path does not match current OpenClaw behavior.

## Close Comment

Thanks for the report. Codex reviewed current `main` at `8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf` and I could not reproduce the claimed automatic execution path.

On current OpenClaw, skill discovery/load reads and parses `SKILL.md` metadata (`src/agents/skills/local-loader.ts:44`) and the runtime instructs the model to read a selected `SKILL.md` on demand (`src/agents/system-prompt.ts:162`, `docs/concepts/context.md:119`). The ClawHub install flow downloads and copies a skill folder into `workspace/skills` after checking for `SKILL.md` (`src/agents/skills-clawhub.ts:245`), rather than executing arbitrary `scripts/*.py` files during load. Executable install actions only run through the separate explicit `skills.install` / `metadata.openclaw.install` path (`src/agents/skills-install.ts:423`, `docs/cli/skills.md:44`), and that path is guarded by built-in dangerous-code blocking by default (`src/plugins/install-security-scan.runtime.ts:503`, `docs/platforms/mac/skills.md:19`).

Because the reported "normal skill warmup/load process automatically runs `scripts/polymarket.py`" behavior does not match current repo behavior, I’m closing this Codex review as not reproducible in `openclaw/openclaw`. If you can still reproduce an actual automatic execution path on current `main`, please open a fresh issue with the exact skill contents, the install path used, and logs showing the command OpenClaw invoked.
