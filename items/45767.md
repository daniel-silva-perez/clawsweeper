## Summary

Keep open. Current `main` still appears to have the reported behavior: Discord `lastInboundAt`/`lastOutboundAt` are ultimately sourced from a process-local activity map, so they can reset across a gateway restart, and Discord `connected` is still derived from runtime gateway readiness rather than a stronger durable/probe-backed truth signal. Docs now tell operators to use `openclaw channels status --probe` for live health, but the issue’s requested status semantics are not implemented directly in `channels status --json`.

## Evidence

- **Channel activity is process-local only:** `recordChannelActivity` and `getChannelActivity` use an in-memory `Map` with no persistence layer, so activity timestamps naturally disappear on process restart.
  - file: [src/infra/channel-activity.ts:9](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/src/infra/channel-activity.ts#L9)

- **Status snapshots still backfill from the ephemeral activity map:** `channels.status` reads `getChannelActivity(...)` and only fills `snapshot.lastInboundAt` / `snapshot.lastOutboundAt` from that map when the snapshot fields are empty.
  - file: [src/gateway/server-methods/channels.ts:240](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/src/gateway/server-methods/channels.ts#L240)

  - file: [src/plugin-sdk/status-helpers.ts:184](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/src/plugin-sdk/status-helpers.ts#L184)
