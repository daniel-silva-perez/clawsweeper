## Summary

Keep PR #39726 open. Current main still preserves `notifyOnExitEmptySuccess` as explicit opt-in only, so chat-context background execs that exit 0 with no output still do not get the default completion notification this PR is trying to add. The PR remains focused and useful: the discussion shows the contributor addressed the two concrete review concerns by preserving explicit `false` opt-out behavior and limiting the default to real reply-capable channels rather than generic provider labels.

## Best Possible Solution

Keep tracking PR #39726 or rework it on current main: default `notifyOnExitEmptySuccess` from `resolveGatewayMessageChannel(messageProvider)` only when the config value is unset, preserve explicit `false`, keep generic provider labels silent, add the regression cases from the PR, and update the user-facing docs/changelog if maintainers accept the product behavior.
