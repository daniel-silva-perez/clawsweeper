## Summary

Keep #71555 open. Current main still routes embedded runs through `SessionManager.open(params.sessionFile)` before prompt assembly, and the pinned `@mariozechner/pi-coding-agent` version still has the double full-file load pattern described in the report. Existing truncation and compaction helpers run after, or by using, an opened `SessionManager`, so they do not eliminate the initial large-transcript load/crash risk.

## Best Possible Solution

Keep the issue open and fix the embedded session-open path. The likely maintainer direction is either to upstream/fork a `SessionManager.open` API that parses the header once and reuses the parsed entries, or add an OpenClaw adapter that reads the header cheaply and avoids duplicate whole-file parsing before context rebuild. Add focused regression coverage with a large JSONL fixture or mocked `SessionManager` loader counts, and consider whether pre-run repair/truncation can stream or bound reads for oversized transcripts.
