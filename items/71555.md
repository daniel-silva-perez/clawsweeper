## Summary

Keep open. The report is recent and concrete, and current main still has the implicated dependency behavior: OpenClaw depends on `@mariozechner/pi-coding-agent@0.70.2`, whose `SessionManager.open()` reads/parses the whole JSONL once to get the header and then constructs a `SessionManager` that calls `setSessionFile()` and reads/parses the same file again. OpenClaw's embedded attempt path still calls `prewarmSessionFile()` and then `SessionManager.open()` before `createAgentSession()`, and upstream `createAgentSession()` immediately calls `sessionManager.buildSessionContext()`. The current OpenClaw cache helper only warms 4 KB into the OS page cache; it does not reuse parsed entries or a `SessionManager` instance. The related stale-lock cleanup item #52929 explains the observed restart recovery symptom, but it does not supersede this crash/memory-pressure path.

## Best Possible Solution

Keep #71555 open. The best fix is to remove the double full-file parse before embedded context rebuild, either by updating/fixing `pi-coding-agent` so `SessionManager.open()` does header-only cwd discovery or passes the first parsed result into the constructed manager, or by adding an OpenClaw-side adapter that avoids reopening large transcripts. Add a focused regression that exercises a large JSONL session and proves embedded startup/tool-result truncation does not read/parse the full file twice or crash before returning a controlled error.
