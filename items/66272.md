## Summary

Keep open. Current main still has the reported behavior: `gateway.trusted_proxies_missing` is emitted specifically for `gateway.bind: "loopback"`, and the current regression test still expects that warning. The PR is not ready to close as implemented or obsolete, and the review discussion raises a real remaining design issue: a correct fix should classify effective exposure, not just flip the raw bind-mode comparison to `bind !== "loopback"`.

## Best Possible Solution

Keep this PR open and request a revised implementation that bases `gateway.trusted_proxies_missing` on effective gateway exposure rather than only the raw `gateway.bind` string. The fix should stop warning for definitely local-only loopback-resolving binds, still warn for genuinely exposed reverse-proxy cases, keep adjacent gateway audit checks coherent, and add regression coverage for plain loopback, `lan`, custom loopback, and exposed custom/auto behavior.
