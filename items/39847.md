## Summary

Keep #39847 open. Current main still sends outbound reply payload text through directive parsing and delivery normalization without applying `stripInboundMetadata()`, so the reported Discord/channel metadata echo gap remains unresolved in the shared outbound pipeline. Related shipped work strips copied metadata in Pi/user-facing replay paths, but it does not cover `normalizeReplyPayloadsForDelivery()` or `deliverOutboundPayloads()`.

## Best Possible Solution

Keep #39847 as the canonical bug until maintainers land a source-level fix in the shared outbound normalization path. The likely fix is to apply `stripInboundMetadata()` before reply directive parsing or before final delivery projection in `src/infra/outbound/payloads.ts`, then add focused regression tests for echoed `Conversation info`, `Sender`, trailing `Untrusted context`, and MEDIA/directive preservation. After one PR lands, close or supersede the duplicate open fix PRs (#39894, #50520, #51653).
