## Summary

Current `main` does not reproduce the reported CWE-78 claim. The Signal daemon still calls `spawn(opts.cliPath, args)` in `extensions/signal/src/daemon.ts`, but it does so without `shell: true`, so metacharacter payloads are treated as literal executable names and fail with `ENOENT` instead of executing injected commands. The configured `cliPath` is also already constrained by `ExecutableTokenSchema`, which delegates to `isSafeExecutableValue()` and rejects the same metacharacters called out in the PR.

## Evidence

- **Signal daemon uses direct spawn, not shell execution:** The only Signal daemon spawn site on current `main` is `spawn(opts.cliPath, args, { stdio: ["ignore", "pipe", "pipe"] })`. There is no `shell: true` or shell wrapper here.
  - file: [extensions/signal/src/daemon.ts:91](https://github.com/openclaw/openclaw/blob/4630ce3d9e2f089062ccd386b0ae241325961d1a/extensions/signal/src/daemon.ts#L91)

- **Config already validates `channels.signal.cliPath`:** Signal config still binds `cliPath` to `ExecutableTokenSchema.optional()`, so normal config loading rejects unsafe executable tokens before runtime.
  - file: [src/config/zod-schema.providers-core.ts:1072](https://github.com/openclaw/openclaw/blob/4630ce3d9e2f089062ccd386b0ae241325961d1a/src/config/zod-schema.providers-core.ts#L1072)

## Risks / Open Questions

- If this call site ever switches to `shell: true` or to a shell wrapper, the current reproduction result would no longer apply.
- There is still a small UX gap around invalid/untrimmed executable paths causing spawn failures instead of a friendlier validation error, but that is not command injection.

## Close Comment

Closing after a Codex review because I could not reproduce the reported command-injection behavior on current `main`.

Concrete evidence:
- The current Signal daemon path is `extensions/signal/src/daemon.ts:91`, and it calls `spawn(opts.cliPath, args, ...)` directly, without `shell: true`.
- Signal config already validates `channels.signal.cliPath` through `ExecutableTokenSchema` in `src/config/zod-schema.providers-core.ts:1072`, backed by `isSafeExecutableValue()` in `src/infra/exec-safety.ts:16`.
- Reproducing the payloads from this PR (`signal-cli; whoami`, `$()`, backticks, pipes, and a leading `-`) against Node's direct `spawn()` resulted in `ENOENT` for each case, meaning Node treated them as literal executable names rather than executing injected shell syntax.

Given that, this does not currently demonstrate a reproducible CWE-78 issue on `main`, so I’m closing it as `cannot_reproduce`. If there is a concrete path where `cliPath` reaches a shell-backed execution path, or a reproduction that shows code execution on current `main`, please open a fresh report with that exact call path and runtime context.
