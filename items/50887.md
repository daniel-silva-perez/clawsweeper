## Summary

Keep open. Current `main` does not give strong evidence that #50887 is already fixed or obsolete. The Anthropic-compatible SSE parser on `main` is UTF-8 boundary-safe and a targeted split-byte reproduction of that parser produced intact Chinese text, so the originally suspected provider-stream assembly path is not proven. But there is also no clear shipped fix for the issue on `main` or in `v2026.4.22`, and the provided issue context references follow-up fix work that is still open. That is not enough to close conservatively.

## Evidence

- **Anthropic SSE parsing is boundary-safe on main:** `src/agents/anthropic-transport-stream.ts:437-465` reads `ReadableStream<Uint8Array>` with `TextDecoder().decode(value, { stream: true })`, which is the correct pattern for UTF-8 across chunk boundaries.
  - file: [src/agents/anthropic-transport-stream.ts:437](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/src/agents/anthropic-transport-stream.ts#L437)
  - command: `nl -ba src/agents/anthropic-transport-stream.ts | sed -n '420,490p'`
  - sha: [ebd7f19a3b20](https://github.com/openclaw/openclaw/commit/ebd7f19a3b20001960474ec9ba9cc424e45f7b23)
- **Targeted split-byte reproduction did not corrupt Chinese text:** A read-only `node --input-type=module -e ...` reproduction using the same `TextDecoder(..., { stream: true })` assembly pattern emitted `["特油盘点 张建洋 3月18日"]` even when the UTF-8 bytes were split across many chunk boundaries.
  - command: `node --input-type=module -e '...TextDecoder stream reproduction...'`
- **No clear shipped fix in suspected alternate chunk-decoding area:** `src/process/supervisor/adapters/child.ts:111-120` on current `main` still decodes per-chunk with `chunk.toString()`. The same lines are present in `v2026.4.22`, so there is no obvious release-tagged fix there.
  - file: [src/process/supervisor/adapters/child.ts:111](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/src/process/supervisor/adapters/child.ts#L111)
  - command: `nl -ba src/process/supervisor/adapters/child.ts | sed -n '1,220p' ; git show v2026.4.22:src/process/supervisor/adapters/child.ts | nl -ba | sed -n '100,130p'`
  - sha: [ebd7f19a3b20](https://github.com/openclaw/openclaw/commit/ebd7f19a3b20001960474ec9ba9cc424e45f7b23)
- **Issue context still points to unresolved follow-up work:** The provided GitHub context for #50887 cross-references closed PR #50909 and open PR #51303 (`fix: use StringDecoder for UTF-8 boundary-safe chunk decoding in child adapter`), which is not consistent with a high-confidence maintainer close on current `main`.

## Risks / Open Questions

- The real corruption point may be later in the webchat/gateway delivery path rather than the Anthropic SSE parser I checked, and I did not run an end-to-end live reproduction in this read-only review.
- The `child.ts` per-chunk decoding may be unrelated to this specific webchat bug; it only shows that there is no clear, shipped UTF-8-boundary fix to point to.
