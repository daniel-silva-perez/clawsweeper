## Summary

Keep #48745 open. Current main still sends WhatsApp document media without an explicit filename as `file` on the direct inbound send API path, and the existing test still asserts that behavior. The PR is stale against moved WhatsApp plugin paths, but the reported UX bug remains actionable and can be ported to the current `extensions/whatsapp` implementation.

## Best Possible Solution

Keep the PR open and ask for a small rebase/port: use the plugin-safe `openclaw/plugin-sdk/media-mime` `extensionForMime()` export in `extensions/whatsapp/src/inbound/send-api.ts`, update `extensions/whatsapp/src/inbound/send-api.test.ts` to expect `file.pdf` and a dynamic case such as `file.csv`, and consider centralizing document filename normalization in `extensions/whatsapp/src/outbound-media-contract.ts` so auto-reply and outbound sends do not drift.
