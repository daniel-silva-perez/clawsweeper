## Summary

Keep open. Current main still implements `memory-lancedb` autoCapture as a narrow heuristic `agent_end` hook, not broad per-session or per-turn capture. It only scans successful runs, only user-role text, requires hard-coded trigger patterns, skips several classes of text, dedupes by vector search, and stores at most the first three capturable messages from the hook payload. The adjacent doctor warning also still has a plausible root cause: doctor depends on an active memory runtime, while `memory-lancedb` registers tools/hooks/services but no memory capability/runtime on current main.

## Best Possible Solution

Keep this issue open. Maintainers should define the intended `memory-lancedb` autoCapture contract, then either implement robust current-turn/session capture with poisoning safeguards, duplicate/history cursor handling, retry/logging semantics, and regression tests, or document that autoCapture is intentionally heuristic-only. The adjacent doctor/status false-negative should be fixed by registering a memory capability/runtime for `memory-lancedb` if it is meant to satisfy the active memory plugin contract; #42083 and #65066 should be reviewed as partial implementation candidates rather than treated as shipped fixes.
