## Summary

Keep this issue open. The cron-lane half is still present on current main: cron isolated runs enter with `lane: "cron"`, cron-owned embedded work is still routed to the shared `nested` lane, and only `cron`, `main`, and `subagent` lane concurrency are configured while unconfigured lanes default to `maxConcurrent: 1`. The open #54314 PR is the best active vehicle for Bug A, but it has not landed and it does not fully cover the separate subagent cleanup packaging claim.

## Best Possible Solution

Keep this issue open. Review, rebase, and land #54314 or an equivalent dedicated cron-owned inner lane fix so `cron.maxConcurrentRuns` applies to both outer cron scheduling and cron-owned embedded work without widening unrelated shared `nested` behavior. Then verify the subagent cleanup/finalize packaging claim against an actual package artifact and either fix it or split it into a narrower issue with exact missing-module proof.
