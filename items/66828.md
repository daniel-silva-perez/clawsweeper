## Summary

Keep this issue open. The cron-lane regression still exists on current main: cron isolated jobs enter with `lane: "cron"`, cron-owned embedded work is still mapped to the shared `nested` lane, and `nested` is still unconfigured so it keeps the command queue default of `maxConcurrent: 1`. The subagent packaging half appears partly addressed by the later stable `subagent-registry.runtime` entry and changelog entry, but Bug A remains unresolved and the open #54314 PR is still only an implementation candidate.

## Best Possible Solution

Keep this issue open. Review, rebase, and land #54314 or an equivalent dedicated cron-owned inner lane fix that applies `cron.maxConcurrentRuns` to both outer cron execution and cron-owned embedded work without widening unrelated shared `nested` behavior. Then verify the subagent cleanup/finalize packaging claim against a real package artifact; if anything remains after the shipped `subagent-registry.runtime` fix, split that into a narrower issue with exact missing-module and cleanup-failure evidence.

## Risks / Open Questions

- Closing now would leave the current main cron path still serializing cron-owned embedded work on the shared `nested` lane when `cron.maxConcurrentRuns` is greater than 1.
- A broad fix that simply raises shared `nested` lane concurrency could change unrelated nested agent flows; the safer direction is a dedicated cron-owned inner lane as proposed in #54314.
- The subagent packaging/runtime-shim half appears partly shipped via #66189 in 2026.4.14, but if failures persist they should be verified against an actual installable package artifact or split into a narrower follow-up.
