## Summary

Keep open. Current main still does not apply config-backed gateway probe auth in restart health checks, and the rate-limit fallback behavior on main is not the same as this PR's immediate fallback request: main defaults to one same-provider auth-profile rotation before cross-provider fallback, with a config knob for zero rotations. The PR is stale against newer failover-policy refactors and should be rebased or split, but the underlying restart-health work remains actionable.

## Best Possible Solution

Keep the PR or a replacement open, but split the work if needed: implement restart health with the same config/SecretRef-aware probe-auth path used by `gateway status`, including warning handling, then make an explicit maintainer decision on whether rate-limit fallback should remain the current configurable one-rotation default or change to immediate fallback when a model fallback is configured.
