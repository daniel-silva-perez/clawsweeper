## Summary

Keep open. Current `main` still lacks the requested hardening in the relevant paths: `src/channels/typing-lifecycle.ts` has no error-aware circuit breaker, `src/auto-reply/reply/typing.ts` still rethrows typing-start failures from keepalive ticks, and `extensions/telegram/src/send.ts` still routes `sendTypingTelegram()` through the default channel retry runner instead of a cheap typing-specific retry policy.

## Evidence

- **Keepalive loop still has no catch/breaker:** `createTypingKeepaliveLoop()` only guards `tickInFlight`; it awaits `params.onTick()` in a `try/finally` and restarts on `setInterval`, with no consecutive-error tracking or stop-on-failure logic.
  - file: [src/channels/typing-lifecycle.ts:17](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/src/channels/typing-lifecycle.ts#L17)

- **Auto-reply typing path still rethrows tick errors:** `createTypingController()` builds a start guard with `rethrowOnError: true`, then wires `triggerTyping` into `createTypingKeepaliveLoop()`. A failing `onReplyStart` therefore still rejects out of the keepalive tick instead of tripping a local breaker.

## Risks / Open Questions

- The normal Telegram reply-dispatch path does have higher-level safeguards in `src/channels/typing.ts`, so the original issue title overstates the generic keepalive seam. That said, current `main` still retains the expensive Telegram typing retry policy and at least one keepalive path that rethrows failures, so closing would be premature.
