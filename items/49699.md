## Summary

Keep PR #49699 open for maintainer judgment rather than auto-closing. Current main has strong adjacent device-token hardening, including caller-scope containment and generic public denial messages, but it does not implement this PR's requested structured error-object contract for `ensureDeviceToken`, `rotateDeviceToken`, or `revokeDeviceToken`. The PR head also has serious review concerns in the provided discussion, so the best path is not to merge it as-is; if maintainers want this capability, it should be redesigned as an additive/internal typed-denial shape that preserves existing public RPC behavior.

## Best Possible Solution

Keep this PR open for explicit maintainer judgment. If the structured-error idea is desired, rework it as an additive typed-denial model that preserves the existing `ok/reason` contract for internal callers, keeps gateway RPC client responses generic, adds detailed server-side logs only where safe, and includes focused tests for scope escalation, missing device/role, and non-enumerating public errors.
