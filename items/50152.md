## Summary

Keep open. Current main has adjacent string-content normalization, but it is gated behind `compat.requiresStringContent`; the bundled NVIDIA provider and generic OpenAI-compatible detection do not enable that flag, so the exact out-of-box NVIDIA failure described by #50152/#50107 is not proven fixed on main. The PR also has review concerns that make it a candidate for maintainer-directed narrowing rather than conservative auto-close.

## Best Possible Solution

Keep the PR open for maintainer judgment, but do not merge it blindly. The best path is to reuse the existing `flattenCompletionMessagesToStringContent` behavior and either set `compat.requiresStringContent: true` for bundled NVIDIA models, add a provider-owned compat contribution for known strict string-only backends, or deliberately document that users must opt in. Add NVIDIA/default-provider regression coverage that reproduces #50107 without requiring manual compat configuration, and preserve current safeguards for mixed content, metadata-bearing text blocks, and newline separation.

## Risks / Open Questions

- Adjacent ClawSweeper reports closed similar PRs as implemented via the opt-in `compat.requiresStringContent` path, but this PR and linked #50107 target NVIDIA/default strict-provider behavior where current main does not enable that flag automatically.
- A universal normalizer for every `openai-completions` provider could be broader than the apparent current product direction; the safer fix may be provider-owned compat metadata or a targeted compatibility contribution for NVIDIA and other known string-only backends.
