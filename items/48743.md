## Summary

The request is an optional persona/policy-enforcement feature that fits the existing plugin/hook extension surface better than OpenClaw core. Core already injects the relevant identity/persona workspace files, and the remaining validation/logging/alerting behavior is already expressible through plugin hooks.

## Evidence

- **Core already injects persona/identity files:** OpenClaw documents `SOUL.md`, `IDENTITY.md`, and `USER.md` as workspace bootstrap files that are injected into agent context on new sessions.
  - file: [docs/concepts/agent.md:27](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/docs/concepts/agent.md#L27)

## Close Comment

Closing this as a Codex review under `clawhub`. The core part of the request is already covered by existing workspace bootstrap behavior: `SOUL.md`, `IDENTITY.md`, and `USER.md` are injected from the agent workspace (`docs/concepts/agent.md:27`), and reinjection timing is already configurable with `agents.defaults.contextInjection` (`docs/gateway/config-agents.md:70`). The rest of the proposal is custom persona/policy enforcement, and OpenClaw already exposes plugin hooks for prompt injection plus outbound/LLM interception (`before_prompt_build`, `before_agent_reply`, `message_sending`, `llm_output`, `agent_end`) via `docs/concepts/agent-loop.md:93`, `src/plugins/hook-before-agent-start.types.ts:21`, and `src/plugins/hook-message.types.ts:45`. Per `VISION.md:54`, optional capability should usually ship as plugins, and `VISION.md:68` says new skills should go to ClawHub first. This makes the request a better fit for a ClawHub plugin/skill than a new core `agents.defaults.identity` feature.
