## Summary

Keep open. Current main still has the PR's headline plugin loader fail-closed behavior, so this is not implemented or superseded. However, the PR should be reworked before merge because it bundles unrelated fixes and the proposed runtime `throwOnLoadError: false` change has fail-open and stale-cache risks.

## Best Possible Solution

Keep this PR open for maintainer review, but require a narrower rework: split unrelated fixes into separate PRs; address plugin-local config invalidity in the config validation/control-plane path; keep production runtime fail-closed or add explicit required-plugin/degraded-health semantics; avoid caching failed graceful-load registries; and add regression coverage for gateway boot, CLI bootstrap, plugin status/doctor diagnostics, and the linked #70371 scenario.

## Risks / Open Questions

- Merging the PR as-is could fail open for runtime plugin loads, leaving required/security-sensitive plugins unavailable while the process continues.
- Changing runtime loads to non-throwing without cache invalidation can cache a failed registry and skip retry until restart or explicit cache reset.
- The branch bundles unrelated QQBot, media reply, and Vertex metadata changes with the plugin loader fix, including a P1 media-drop review finding.
- The proposed two-line loader change may not fully solve #70371 because normal `loadConfig()` still throws on plugin-local schema validation failures.
