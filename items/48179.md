## Summary

Keep open. Current `main` does not implement the requested Telegram voice-pipeline optimization. Voice/media updates still run on the normal Telegram sequentialized lane, media is downloaded inline before enqueue, and audio preflight transcription is awaited synchronously during inbound body resolution. I found no Telegram-specific implementation for a priority voice queue, streaming transcription, parallel TTS/transcription, or transcription/TTS caching.

## Evidence
