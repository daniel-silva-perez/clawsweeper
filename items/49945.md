## Summary

Keep PR #49945 open. Current main still has the underlying TUI `/new` regression: `/new` only switches the local TUI session to a fresh `tui-<uuid>` key, while `/reset` is the only path that calls gateway `sessions.reset` and emits `command:new`/`command:reset` hook events. The PR is therefore not obsolete, but its current patch also has unresolved review concerns around hook session identity/source context and blank persisted `tui-*` sessions, so it needs maintainer/contributor follow-up rather than conservative auto-close.

## Best Possible Solution

Keep the PR open or ask the contributor for revisions. The best fix should restore hook-visible TUI `/new` behavior while preserving per-client `tui-<uuid>` isolation from #39217, pass hooks the session that is actually ending, avoid attributing archived/logged data to the wrong fresh key, and avoid cluttering session history with empty `tui-*` rows. The likely implementation area remains `src/tui/tui-command-handlers.ts`, `src/tui/gateway-chat.ts`, gateway `sessions.reset` protocol/server handling if a source-key seam is needed, plus regression coverage in the TUI command-handler and gateway session hook tests.
