## Summary

Keep open. Current main has meaningful Discord delivery hardening: transient-send retry, write-ahead delivery queueing, failure logging, and low-cardinality delivery diagnostics. But the specific user-facing reliability request is not fully implemented: final reply delivery errors are caught inside the reply dispatcher, reported to `onError`, and not propagated back to `processDiscordMessage`, so the Discord message path can still finish without a visible fallback notification or a message-id audit trail for the failed final response.

## Best Possible Solution

Keep this Discord-specific bug open or fold it into an explicit maintainer-owned delivery-failure follow-up. The best fix is to make reply delivery failure observable to `processDiscordMessage`: propagate dispatcher failed counts or structured delivery results, mark status/error state accurately, log successful Discord message IDs in a bounded audit event, and send a best-effort visible fallback when final delivery cannot be confirmed. Coordinate with the broader open delivery-failure-notification work (#46830) without closing this incident until Discord final replies have that end-to-end behavior.
