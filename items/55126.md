## Summary

Current main still drops `memorySearch.outputDimensionality` for the bundled OpenAI memory embedding path. The resolved config and provider-create contract carry the field, but the OpenAI client type and shared remote embedding request body do not include dimensions, and the OpenAI batch path builds requests with only `model` and `input`. Related PR #55507 is the right follow-up thread but remains open, so the issue should stay open until that or an equivalent fix lands.

## Best Possible Solution

Keep #55126 open and land PR #55507 or an equivalent focused fix: add OpenAI dimensions/outputDimensionality plumbing to the OpenAI embedding client, validate supported `text-embedding-3-*` dimensions, include `dimensions` in inline and batch OpenAI embedding request bodies, update cache key data if dimension affects provider identity, and add regression tests plus a docs note for OpenAI memory embeddings.

## Risks / Open Questions

- A fix must handle both inline OpenAI embedding calls and the OpenAI batch embedding runtime; fixing only one path would leave inconsistent vector dimensions.
- Docs currently document `outputDimensionality` for Gemini and Bedrock but not OpenAI, so OpenAI Matryoshka support should include a memory config docs update when implemented.
