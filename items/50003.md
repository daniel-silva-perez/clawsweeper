## Summary

Current `main` does not implement this PR's non-interactive GitHub Copilot token onboarding. The PR is stale/conflicted, but the requested behavior is still missing, so there is not strong evidence to close it.

## Evidence

- **No CLI flag metadata for Copilot onboarding:** `register.onboard.ts` only registers plugin-owned provider auth flags returned by `resolveManifestProviderOnboardAuthFlags()`. That resolver skips any auth choice missing `optionKey`, `cliFlag`, or `cliOption`. The current GitHub Copilot manifest defines only the interactive choice metadata and has none of those fields, so `--github-copilot-token` is not exposed on `main`.
  - file: [extensions/github-copilot/openclaw.plugin.json:11](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/extensions/github-copilot/openclaw.plugin.json#L11)
  - command: `nl -ba extensions/github-copilot/openclaw.plugin.json | sed -n '1,80p'; nl -ba src/plugins/provider-auth-choices.ts | sed -n '240,247p'; nl -ba src/cli/program/register.onboard.ts | sed -n '50,105p'`
  - sha: [45146913007d](https://github.com/openclaw/openclaw/commit/45146913007da723b4f06aba6a66cf4a695974c0)
- **Copilot provider still lacks non-interactive setup hook:** The bundled GitHub Copilot provider registers only `run` for the `device` auth method, with an explicit TTY requirement. The non-interactive onboarding path errors when `method.runNonInteractive` is absent. On current `main`, choosing `github-copilot` in `openclaw onboard --non-interactive` still routes to that failure path rather than token import.
  - file: [extensions/github-copilot/index.ts:41](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/extensions/github-copilot/index.ts#L41)
