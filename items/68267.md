## Summary

Keep open. Current `main` does not implement optional `runId` filtering for `chat.history`, and the request belongs in the Gateway protocol/server surface rather than ClawHub. The PR discussion shows the author addressed several prior review findings, but the latest review still raises an actionable token-boundary issue for arbitrary `NonEmptyString` run IDs, so this should be revised and reviewed rather than auto-closed.

## Best Possible Solution

Keep this PR open and have maintainers review a revised version that adds `runId` to the Gateway protocol schema, applies filtering before history budget/sanitization, tests the production path including punctuation/symbol/Unicode collision cases, and updates protocol docs if the new parameter is accepted.
