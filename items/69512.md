## Summary

Keep this issue open. Current main has partial Claude CLI permission-mode handling, but it still does not forward or enforce OpenClaw `exec-approvals.json` allowlists for Claude Code internal Bash/tool execution. The related #67666 PR was a Codex CLI resumed-session sandbox fix, not a Claude CLI allowlist-forwarding fix.

## Best Possible Solution

Keep the issue open. The best path is an Anthropic-owned `claude-cli` permission bridge, with a generic CLI-backend/native-permission seam only if needed, that maps Claude Bash permission requests through the same OpenClaw exec approval evaluation used by native exec. It should add regression coverage for allowed and denied commands under per-agent `tools.exec.security: "allowlist"`, and docs should state the Claude CLI parity behavior and any known limitations.

## Risks / Open Questions

- Current main reduces one unsafe default by avoiding `bypassPermissions` under non-YOLO policy, but that still leaves Claude's permission layer separate from OpenClaw's exec allowlist semantics.
- A correct fix is security-sensitive: naive raw Claude args may not preserve OpenClaw shell parsing, safe-bin, strict inline-eval, ask-fallback, per-agent allowlist, and audit behavior.
