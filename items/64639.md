## Summary

Keep this issue open. Current main still has no source-level config to disable the personal `~/.agents/skills` source; it is loaded and watched as a built-in source, while existing allowlists, per-skill disables, and prompt limits only mitigate visibility after discovery or affect narrower scopes.

## Best Possible Solution

Keep this issue open and implement a narrow core skills-loader config surface that disables `agents-skills-personal` before both `loadSkillEntries` discovery and `ensureSkillsWatcher` path registration. Update config types, strict Zod schema, generated schema/help metadata, docs at `docs/tools/skills-config.md` and `docs/tools/skills.md`, and focused regression tests proving the personal source is skipped while existing precedence, bundled allowlists, per-agent skill filters, per-skill `enabled: false`, prompt limits, and plugin/extra-dir behavior continue to work.

## Risks / Open Questions

- The public config shape needs maintainer judgment: a typed source list is flexible, while an explicit `personal` scan boolean may be simpler but less extensible.
- A post-load filter would not satisfy the issue because it would still scan and watch `~/.agents/skills`; the fix should skip both discovery and watcher registration for disabled sources.
- Status reports, snapshots, slash-command discovery, and future plugin skill-source UI should all agree on whether a disabled source is absent or merely hidden.
