## Summary

Keep this PR open. Current `main` still lacks the `quota_exhausted` failover reason, still classifies periodic/daily/monthly quota exhaustion as `rate_limit`, and still sends only `billing` and `auth_permanent` through the long disabled lane. The PR is also an active implementation candidate for the already-closed linked issue, with recent author follow-up splitting unrelated status-provider/TUI work into #72557 rather than abandoning this scope.

## Best Possible Solution

Keep the PR open for focused maintainer review. The best path is to either land a narrowed quota-exhaustion circuit breaker with tests for periodic quota classification, 429 payload precedence, failure-reason scoring, disabled-window routing, and probe throttling, or have maintainers make an explicit product decision to keep periodic quota exhaustion in the existing rate_limit lane and close the PR with that rationale. #72557 should remain the separate track for the TUI/plugin status-provider API split.
