## Summary

Current main treats `xai/grok-4-fast-reasoning` and `xai/grok-4-1-fast-reasoning` as legacy compatibility aliases, not distinct catalog entries, and the docs now explicitly say so. The issue's requested fallback outcome (clarify docs/config acceptance when those ids are not separately selectable) is already implemented and shipped.

## Evidence

- **Legacy xAI reasoning ids are normalized to canonical ids:** `normalizeNativeXaiModelId()` rewrites `grok-4-fast-reasoning` -> `grok-4-fast` and `grok-4-1-fast-reasoning` -> `grok-4-1-fast`, so config/model resolution intentionally collapses those older ids.
  - file: [src/plugin-sdk/provider-model-id-normalize.ts:29](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/src/plugin-sdk/provider-model-id-normalize.ts#L29)
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
- **The bundled xAI plugin applies that normalization at runtime:** The xAI provider registers `normalizeModelId: ({ modelId }) => normalizeXaiModelId(modelId)`, so resolved model identity uses the canonical id before catalog/UI surfaces see it.
  - file: [extensions/xai/index.ts:197](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/extensions/xai/index.ts#L197)
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
- **The bundled xAI catalog exposes canonical ids, not separate legacy reasoning aliases:** The built-in catalog includes `grok-4-fast`, `grok-4-fast-non-reasoning`, `grok-4-1-fast`, and `grok-4-1-fast-non-reasoning`; there is no separate `*-fast-reasoning` row to list in `models list` or the picker.
  - file: [extensions/xai/model-definitions.ts:112](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/extensions/xai/model-definitions.ts#L112)
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
- **Docs explicitly clarify the legacy-alias behavior:** The xAI provider docs now list the canonical built-in catalog and separately document `grok-4-fast-reasoning` and `grok-4-1-fast-reasoning` as legacy aliases that normalize to canonical ids. This matches the issue's requested clarification path if those ids are not distinct/selectable.
  - file: [docs/providers/xai.md:45](https://github.com/openclaw/openclaw/blob/9b18ae877018a34933e27bb7b2986cb97e9602aa/docs/providers/xai.md#L45)
  - sha: [9b18ae877018](https://github.com/openclaw/openclaw/commit/9b18ae877018a34933e27bb7b2986cb97e9602aa)
- **Control UI picker is catalog-backed:** The Control UI fetches `models.list` from the gateway and builds picker options directly from that catalog, so once the provider canonicalizes legacy ids, the picker intentionally shows only the canonical entries.
  - file: [ui/src/ui/controllers/models.ts:11](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/ui/src/ui/controllers/models.ts#L11)
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)

## Risks / Open Questions

- If upstream xAI reintroduces distinct `*-reasoning` ids with different behavior, the alias mapping and docs would need to change.
- This closure assumes the issue is satisfied by the documented clarification path, since current main still does not expose those legacy ids as separate selectable catalog rows.

## Close Comment

This is implemented on current OpenClaw main, so I would close this.

- Codex review of current `main` found that OpenClaw intentionally normalizes `xai/grok-4-fast-reasoning` to `xai/grok-4-fast` and `xai/grok-4-1-fast-reasoning` to `xai/grok-4-1-fast` in `src/plugin-sdk/provider-model-id-normalize.ts:29`.
- The bundled xAI plugin wires that normalization into runtime model resolution in `extensions/xai/index.ts:197`, so `openclaw models list` and other catalog-backed surfaces see the canonical ids.
- The bundled xAI catalog in `extensions/xai/model-definitions.ts:112` exposes the canonical rows (`grok-4-fast`, `grok-4-fast-non-reasoning`, `grok-4-1-fast`, `grok-4-1-fast-non-reasoning`) rather than separate legacy `*-fast-reasoning` rows.
- The docs now explicitly clarify this behavior in `docs/providers/xai.md:45` and `docs/providers/xai.md:107`, where those ids are documented as legacy compatibility aliases that normalize to canonical ids.
- That clarification is present in the shipped `v2026.4.23` release (`9b18ae877018a34933e27bb7b2986cb97e9602aa`), so the current main/released behavior matches the documented contract.
