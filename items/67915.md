## Summary

Cannot close this issue in the cleanup pass because it has an open same-author implementation PR linked with closing syntax. Current main also still appears to use an empty `localMediaPreviewRoots` array as a hard client-side deny before the authoritative server metadata request, so the reported behavior is not clearly shipped as fixed.

## Best Possible Solution

Keep this issue open while PR #67916 remains open. Maintainers should review, adjust, and either land that PR or close it in favor of an equivalent fix that skips the client-side allowed-folder hard deny while bootstrap roots are still unloaded and relies on the existing server metadata endpoint for the authoritative decision.
