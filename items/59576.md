## Summary

Keep open. This is a recent non-maintainer issue with no protected labels, and the current Mattermost code still appears to process inbound media directly from the WebSocket post's `file_ids` without the proposed REST re-fetch fallback. The report is concrete, has a corroborating commenter for large files, and the linked PR is closed without clear evidence that the fix landed on current main.

## Evidence

- **Issue is recent and actionable:** Issue #59576 was opened 2026-04-02 with concrete reproduction steps for Mattermost attachments, an explained race around empty WebSocket `file_ids`, and a proposed REST re-fetch fix. It is not older than 60 days and therefore does not qualify for stale insufficient info.
- **Current monitor still uses WebSocket file_ids directly:** The current Mattermost monitor builds pending body text from `post.file_ids` and calls `resolveMattermostMedia(post.file_ids)` before constructing the inbound payload. I did not find a re-fetch path in this flow in the inspected current main source.

  - sha: [d6ef1fcf24ee](https://github.com/openclaw/openclaw/commit/d6ef1fcf24eea3a5985d653a3baf400e32aa1b75)
- **Client helper lacks the requested post re-fetch helper:** The current Mattermost client source includes helpers such as user/channel fetches, post creation, post update, file upload, and DM channel retry, but no visible `fetchMattermostPost()` / `GET /api/v4/posts/{post_id}` helper in the inspected current main source.
