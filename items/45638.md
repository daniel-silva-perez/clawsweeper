## Summary

Current `main` still appears to have the reported Telegram callback bug. The callback handler passes `callback.id` as `messageIdOverride`, and that override directly becomes `MessageSid`, while abort-cutoff logic only treats digit-only IDs as numeric message IDs. I did not find evidence in the checked-out `main` that this was fixed or covered by a regression test.

## Evidence

- **Callback handler still uses callback query id:** `processMessage(...)` in the Telegram callback path still passes `messageIdOverride: callback.id` instead of the callback message's `message_id`.

## Risks / Open Questions

- Telegram inline-button callbacks can be associated with a synthetic non-message `MessageSid`, which can break session continuity or per-message bookkeeping.
- Abort-cutoff comparisons may not correctly suppress older callback-driven work when the current message sid is a callback UUID instead of the Telegram message id.
- There does not appear to be a regression test on current `main` that would catch this exact callback `messageIdOverride` mixup.
