## Summary

Keep open. Current main implements important parts of the request, including atomic JSON writes, same-provider/profile refresh serialization, refresh-token-reused recovery, and bootstrap-only Codex CLI import. The remaining hardening is still actionable: OAuth-producing login/write paths still flow through the unlocked auth-profile upsert helper, and duplicate-enrollment protection is still keyed to provider/profileId rather than a stable enrollment identity or documented single-writer boundary.

## Best Possible Solution

Keep this open and narrow it to the remaining hardening: route OAuth-producing auth writes through updateAuthProfileStoreWithLock or upsertAuthProfileWithLock, or make the default OAuth upsert path lock-safe without breaking the public SDK contract; add practical duplicate-enrollment detection within one store where feasible; and document the operator boundary for copied auth-profiles.json files, multiple hosts/stores, and Codex CLI coexistence. Preserve the already-shipped atomic writes, same-profile refresh serialization, refresh-token-reused recovery, and Codex bootstrap-only behavior.
