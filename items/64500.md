## Summary

Keep this issue open. Current main still treats the global circuit breaker as a current-tool/current-signature no-progress breaker, while critical ping-pong blocking still requires stable result hashes on both sides. The reported memory_search <-> memory_get variable-result loop remains plausible and is not covered by pair-scoped blocking or session-level pair suppression. Related #16583 and #62971 cover adjacent loop guards, and #63411 is broader streaming-loop work, but none supersede this pair-aware tool-loop gap.

## Best Possible Solution

Keep this issue open and fix the loop guard in the agent tool-loop layer. A strong maintainer path is to make ping-pong protection pair-aware: either allow sustained A/B alternation to escalate without identical per-side result hashes, add a pair-scoped threshold before the per-tool global breaker, or persist short-lived session-level blocks for both tools/signatures when a critical ping-pong or pair breaker trips. Add regression coverage for variable-result memory_search <-> memory_get alternation and for blocking both sides of the offending pair.
