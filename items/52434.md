## Summary

Keep open. Current main still does not start a typing indicator for tool-dispatched skill commands: `handleInlineActions` resolves `dispatch.kind === "tool"` and directly awaits `tool.execute()` without calling `typing.startTypingLoop()`. The submitted PR targets a real remaining gap, but it should not be closed or landed as-is because review comments identified two real follow-up requirements: typing-start failures must not abort tool execution, and the fix must respect typing suppression / `typingMode: "never"`.

## Best Possible Solution

Keep the PR open but require a revised implementation on current main: start typing for tool-dispatched skill execution only when typing is allowed by the same effective policy used by normal runs, isolate typing-start errors so tool execution still proceeds, and add focused inline-action tests for successful typing start, cleanup, non-fatal typing failure, and suppressed/never typing mode. If that requires threading resolved typing policy into `handleInlineActions`, do that explicitly rather than relying on an unconditional controller call.

## Risks / Open Questions

- The PR's one-line implementation can make a typing-indicator failure prevent the skill tool from running, because `startTypingLoop()` can rethrow and the proposed call sits in the same `try` as `tool.execute()`.
- An unconditional inline `startTypingLoop()` can ignore `typingMode: "never"` or `opts.suppressTyping`, because the normal run typing policy is not yet applied when `handleInlineActions` handles a tool-dispatched skill.
- No current test asserts typing behavior for tool-dispatched skill commands, so a safe fix needs focused regression coverage for start, cleanup, typing failure, and suppressed typing.
