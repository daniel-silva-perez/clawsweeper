## Summary

Keep open. Current main has partial mitigation, but it does not implement the requested Slack config to exclude history images, and new thread sessions can still hydrate root-message files into media context. The report is still actionable even though older session images are now pruned after a few completed turns.

## Evidence

- **No Slack config knob for excluding history images:** The Slack config surface exposes thread history scope/limits and media size caps, but no `historyImages`, `maxSizeBytes`, or equivalent flag to skip images when assembling Slack history context.
  - file: [src/config/types.slack.ts:92](https://github.com/openclaw/openclaw/blob/3361593442e295e9bb1f55cbfb38861ea9413e45/src/config/types.slack.ts#L92)

  - sha: [3361593442e2](https://github.com/openclaw/openclaw/commit/3361593442e295e9bb1f55cbfb38861ea9413e45)
- **Docs likewise document history limits and media caps only:** Slack docs document `thread.initialHistoryLimit`, `historyLimit`, `dmHistoryLimit`, and `mediaMaxMb`, but nothing that disables images specifically for fetched history context.
  - file: [docs/channels/slack.md:518](https://github.com/openclaw/openclaw/blob/3361593442e295e9bb1f55cbfb38861ea9413e45/docs/channels/slack.md#L518)

## Risks / Open Questions

- The issue body overstates current behavior for fetched thread replies: main already uses placeholder text for historical file-only replies rather than hydrating each reply attachment as media.
- The remaining gap is narrower but real: root thread starter files can still be injected into a new session, and there is still no user-facing way to opt out of that Slack history-media behavior.
