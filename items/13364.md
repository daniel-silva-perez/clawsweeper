## Summary

Keep open. Current `main` still does not expose tool-call events through the internal `HOOK.md` system. Internal hooks are limited to `command`/`session`/`agent`/`gateway`/`message`, and the internal hook runtime has no return channel for blocking or mutating tool calls. `before_tool_call`/`after_tool_call` remain plugin-SDK-only surfaces.

## Evidence

- **Internal hook type surface excludes tool calls:** `src/hooks/internal-hook-types.ts:1` defines `InternalHookEventType` as only `command | session | agent | gateway | message`. The same file defines `InternalHookHandler` as `(event) => Promise<void> | void`, so internal hooks cannot return block/param-mutation results.
  - file: [src/hooks/internal-hook-types.ts:1](https://github.com/openclaw/openclaw/blob/3361593442e295e9bb1f55cbfb38861ea9413e45/src/hooks/internal-hook-types.ts#L1)
  - sha: [3361593442e2](https://github.com/openclaw/openclaw/commit/3361593442e295e9bb1f55cbfb38861ea9413e45)
- **Internal hook dispatcher ignores handler results:** `src/hooks/internal-hooks.ts:286` awaits each handler and discards its return value. That means the internal hook runtime currently has no mechanism to stop execution, rewrite params, or propagate `after_tool_call` payloads the way the issue requests.
  - file: [src/hooks/internal-hooks.ts:286](https://github.com/openclaw/openclaw/blob/3361593442e295e9bb1f55cbfb38861ea9413e45/src/hooks/internal-hooks.ts#L286)
  - sha: [3361593442e2](https://github.com/openclaw/openclaw/commit/3361593442e295e9bb1f55cbfb38861ea9413e45)
- **Current internal-hook bridge covers message events, not tool events:** `src/auto-reply/reply/dispatch-from-config.ts:586` explicitly bridges `message_received` into the `HOOK.md` discovery system. A repo search over `createInternalHookEvent(` / `triggerInternalHook(` call sites showed internal-hook producers for command/session/agent/gateway/message flows only, with no tool-call producer on current `main`.
  - file: [src/auto-reply/reply/dispatch-from-config.ts:586](https://github.com/openclaw/openclaw/blob/3361593442e295e9bb1f55cbfb38861ea9413e45/src/auto-reply/reply/dispatch-from-config.ts#L586)
  - command: `rg -n "triggerInternalHook\(|createInternalHookEvent\(" src/agents src/gateway src/auto-reply src/infra`
  - sha: [3361593442e2](https://github.com/openclaw/openclaw/commit/3361593442e295e9bb1f55cbfb38861ea9413e45)
- **Docs still describe tool interception as plugin-hook-only:** `docs/automation/hooks.md:36` lists the available internal hook events and does not include any `tool_call:*` event. Later, `docs/automation/hooks.md:208` says tool-call interception is provided by Plugin SDK hooks and points readers to plugin-hook docs for `before_tool_call`.
  - file: [docs/automation/hooks.md:36](https://github.com/openclaw/openclaw/blob/3361593442e295e9bb1f55cbfb38861ea9413e45/docs/automation/hooks.md#L36)
  - sha: [3361593442e2](https://github.com/openclaw/openclaw/commit/3361593442e295e9bb1f55cbfb38861ea9413e45)
- **Tool-call hooks exist only in the plugin hook runner:** `src/plugins/hook-types.ts:71` includes `before_tool_call` and `after_tool_call`, and `src/plugins/hooks.ts:781` implements them with plugin-specific merging/blocking semantics. That separation is still intact on current `main`; there is no equivalent internal-hook API.
  - file: [src/plugins/hooks.ts:781](https://github.com/openclaw/openclaw/blob/3361593442e295e9bb1f55cbfb38861ea9413e45/src/plugins/hooks.ts#L781)
  - sha: [3361593442e2](https://github.com/openclaw/openclaw/commit/3361593442e295e9bb1f55cbfb38861ea9413e45)
