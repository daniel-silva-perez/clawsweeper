## Summary

Keep open. Current `main` still implements `openclaw nodes list` as a pairing-list view, not a merged live-node view, so the PR's reported inconsistency remains plausible and is not already fixed on `main`. The command starts from `node.pair.list` and only filters that paired set, which means connected nodes missing from pairing data are still omitted. `node.list` already exists as the merged known-node source, but `nodes list` does not use it as its source of truth.

## Evidence

- **`nodes list` still only seeds from `node.pair.list`:** `src/cli/nodes-cli/register.status.ts` defines `nodes list` as "List pending and paired nodes", calls `node.pair.list`, and builds `filteredPaired = paired.filter(...)`. Even when `--connected`/`--last-connected` are used, `node.list` is only consulted as a lookup map for already-paired rows, not as an additional source of rows.
  - file: [src/cli/nodes-cli/register.status.ts:317](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/src/cli/nodes-cli/register.status.ts#L317)
  - command: `nl -ba src/cli/nodes-cli/register.status.ts | sed -n '300,420p'`
  - sha: [ebd7f19a3b20](https://github.com/openclaw/openclaw/commit/ebd7f19a3b20001960474ec9ba9cc424e45f7b23)
- **Current docs still describe `nodes list` as pending/paired only:** The docs say: "`nodes list` prints pending/paired tables. Paired rows include the most recent connect age (Last Connect)." There is no mention of connected-only rows being included in the main list output.
  - file: [docs/cli/nodes.md:38](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/docs/cli/nodes.md#L38)

- **`node.list` is the merged known-node source on current main:** Gateway `node.list` already merges paired devices, paired nodes, and live connected nodes via `createKnownNodeCatalog(...)` with `connectedNodes: context.nodeRegistry.listConnected()`. That confirms there is a separate richer source available, but `nodes list` is not using it as the primary dataset.
  - file: [src/gateway/server-methods/nodes.ts:697](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/src/gateway/server-methods/nodes.ts#L697)

So a connected node can exist in `node.list` even when absent from pairing data, which is exactly the case this PR is trying to surface in CLI output.
