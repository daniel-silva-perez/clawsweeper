## Summary

Current `main` already blocks this failure path in two places: Telegram media filenames are sanitized before they become saved media paths, and `[media attached: ...]` note rendering strips Unicode control characters before prompt text can reach CLI spawn args. The same implementation is present in the shipped `v2026.4.22` release.

## Close Comment

Closing from a Codex review because current `main` already implements the needed guardrails for this path. Telegram attachment filenames flow through `saveMediaBuffer()`, where the saved media ID/path is built from a sanitized filename (`src/media/store.ts:56`, `src/media/store.ts:423`), and the Telegram channel then uses that saved path for `MediaPath`/`MediaPaths` (`extensions/telegram/src/bot/delivery.resolve-media.ts:404`, `extensions/telegram/src/bot-message-context.session.ts:366`). Separately, the prompt-side `[media attached: ...]` formatter strips Unicode control characters, including NUL, before anything can reach CLI spawn args (`src/auto-reply/media-note.ts:4`). I also verified the same implementation is present in the shipped `v2026.4.22` release. If you can still reproduce this on `v2026.4.22` or newer, please open a new issue with exact repro steps and fresh logs.
