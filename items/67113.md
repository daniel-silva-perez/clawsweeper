## Summary

Current main partially addresses the reported timeout/config surface, but the central ARM/WSL embed-loop complaint is not resolved. QMD still runs forced boot embedding by default, still intentionally runs embeddings in `searchMode: "search"`, and tests assert that behavior. The related #59234 is open but narrower, focused on the boot vector/status probe rather than this issue’s full boot plus periodic `qmd embed` behavior.

## Best Possible Solution

Keep this issue open for QMD memory-core work. The strongest path is to define a clear QMD embed contract, then update `extensions/memory-core/src/memory/qmd-manager.ts`, config docs, and regression tests so constrained ARM/WSL hosts can run BM25-only without triggering llama.cpp/Vulkan embedding work. If maintainers want to preserve warm vectors by default, add an explicit config switch for embedding/vector maintenance and make `embedInterval: "0"` suppress all periodic embed paths consistently, while cross-linking #59234 for the narrower boot vector/status probe.

## Risks / Open Questions

- Maintainers need to settle the product contract: whether `searchMode: "search"` means BM25-only operationally, or only selects the query-time retrieval command while embeddings stay warm in the background.
- A fix that suppresses embeddings in search mode could affect users who rely on current vectors after switching later to hybrid/vector modes, so an explicit embed/vector enable flag may be safer than overloading `searchMode`.
