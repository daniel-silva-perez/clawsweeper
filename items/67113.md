## Summary

Do not close this issue yet. Current main partially addresses the timeout/config surface by using `memory.qmd.update.embedTimeoutMs` for `qmd embed` and documenting `update.embedInterval`, but it still intentionally runs `qmd embed` in `searchMode: "search"` and still forces a boot embed through the boot update path. The linked #59234 is related and open, but it covers the boot vector/status probe, not the full periodic/boot embed timeout loop reported here.

## Best Possible Solution

Keep this issue open for maintainer/product resolution in the QMD memory backend. The likely path is to decide whether `searchMode: "search"` should suppress boot and maintenance embedding on constrained hosts, or add a separate explicit config such as a vector/embed enable flag. In the same change, clarify docs around `memory.qmd.update.embedTimeoutMs` versus `memory.qmd.limits.timeoutMs`, and cross-link #59234 for the related boot vector/status probe behavior.
