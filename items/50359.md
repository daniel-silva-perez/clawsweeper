## Summary

The PR should not be closed as obsolete. Current main has the newer pending-tool-media queue and structured `details.media` path, but it still does not recover an image media path from the originating `tool_execution_start` args when the tool result contains image content with no `MEDIA:` token and no `details.path`. The current regression test even asserts that this shape returns no media paths, which is the gap this PR is trying to cover.

## Best Possible Solution

Keep the PR open and adapt it to current main: either add a start-args-aware fallback to the current media artifact extraction path or enrich OpenClaw-owned read image results with structured media metadata, then add focused tests for `path`, `file_path`, and `filePath` through `handleToolExecutionStart` -> `handleToolExecutionEnd` and the pending media merge path.
