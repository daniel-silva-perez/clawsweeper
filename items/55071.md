## Summary

Current main does not implement skill-driven plugin context visibility. Skill metadata parsing and docs only support existing fields like `requires`, plugin tool visibility is still driven by static tool policy/allowlists plus `optional: true`, and the context-engine API receives the already-selected tool names after tool construction rather than controlling which plugin tool schemas are exposed. Related context-engine/dynamic-pruning items cover message-context assembly, not this narrower tool-schema gating request.

## Best Possible Solution

Keep this open for maintainer design. A good implementation would add a typed `metadata.openclaw.activates.plugins` contract, document it, and introduce a request-time tool visibility plan before `createOpenClawCodingTools()`/session creation. That plan should define how skills become “matched” before the model call, merge activated plugin ids into optional plugin tool visibility without bypassing `tools.deny`, and add focused tests for parser metadata, skill snapshot/activation behavior, plugin id activation, deny precedence, and context/reporting output.

## Risks / Open Questions

- Implementing this needs a clear pre-model activation source, because current skills are listed in the prompt and read on demand; there is not yet a built-in semantic skill matcher that can safely decide `activates.plugins` before the model call.
- The design must preserve existing security policy: `tools.deny` should still win, static profiles/allowlists must remain predictable, and prompt-cache determinism may be affected by per-turn tool-set changes.
