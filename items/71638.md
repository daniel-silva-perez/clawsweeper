## Summary

Keep #71638 open. Current main still does not surface Telegram forward, via-bot, or reply provenance on MessageReceivedHookEvent.context.metadata. The shared hook mappers only expose generic routing/sender metadata, the Telegram hook emit path passes only those generic fields, and Telegram reply/forward data is normalized only for agent/session context. A full source search found no via_bot/viaBot handling.

## Best Possible Solution

Keep the issue open and implement a Telegram-owned additive fix. Derive bounded metadata.forward, metadata.viaBot, and metadata.replyTo from inbound Telegram Message objects, thread that provider metadata through the canonical received-hook mapper path and the direct Telegram mention-skip hook path, and add focused tests for toInternalMessageReceivedContext/toPluginMessageReceivedEvent plus the Telegram emit path. Reuse existing reply/forward normalization where it preserves privacy semantics, and add explicit via_bot normalization because current main has no via-bot handling.
