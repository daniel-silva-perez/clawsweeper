## Summary

Keep open. Current `main` still contains the stale-timestamp guard in `appendCronStyleCurrentTimeLine`, so heartbeat/main-session cron prompts can still preserve an old `Current time:` line instead of refreshing it.

## Evidence

- **Stale guard still present on main:** `appendCronStyleCurrentTimeLine` still returns early when the prompt already contains `"Current time:"`, which preserves existing text instead of replacing it with a fresh timestamp.
  - file: [src/agents/current-time.ts:35](https://github.com/openclaw/openclaw/blob/40be5ad58187fd20b8defe5fc30e649190c131a8/src/agents/current-time.ts#L35)
  - command: `nl -ba src/agents/current-time.ts | sed -n '1,120p'`
  - sha: [40be5ad58187](https://github.com/openclaw/openclaw/commit/40be5ad58187fd20b8defe5fc30e649190c131a8)
- **Heartbeat path still uses the stale helper:** `runHeartbeatOnce` builds the outbound body with `appendCronStyleCurrentTimeLine(prompt, cfg, startedAt)`. `startedAt` is fresh per run, but the helper discards it whenever `prompt` already includes `Current time:`.
  - file: [src/infra/heartbeat-runner.ts:974](https://github.com/openclaw/openclaw/blob/40be5ad58187fd20b8defe5fc30e649190c131a8/src/infra/heartbeat-runner.ts#L974)
  - command: `nl -ba src/infra/heartbeat-runner.ts | sed -n '950,990p'`
  - sha: [40be5ad58187](https://github.com/openclaw/openclaw/commit/40be5ad58187fd20b8defe5fc30e649190c131a8)
- **Repository already documents this exact failure mode elsewhere:** `post-compaction-context.ts` has an explicit comment that AGENTS.md content may contain `"Current time:"` and therefore must not gate on that substring, which matches the issue's reported root cause. The shared helper used by heartbeat has not been updated accordingly.
  - file: [src/auto-reply/reply/post-compaction-context.ts:136](https://github.com/openclaw/openclaw/blob/40be5ad58187fd20b8defe5fc30e649190c131a8/src/auto-reply/reply/post-compaction-context.ts#L136)
