## Summary

Current main still forces every `/hooks/agent` dispatch through an isolated cron job, and isolated jobs still call `resolveCronSession` with `forceNew: true`. That bypasses the existing fresh-session reuse path even when the webhook request supplied an allowed stable `sessionKey`, so #70894 remains a real open bug rather than cleanup-closeable.

## Best Possible Solution

Keep #70894 open and review/land a focused fix like #70895: preserve the documented fresh-session behavior for ordinary isolated cron jobs, but pass `forceNew: false` for trusted webhook-sourced isolated runs when an allowed stable `sessionKey` or hook external-content source is present. Add a regression test at the `runCronIsolatedAgentTurn` or gateway hook boundary proving repeated `/hooks/agent` calls with the same allowed `sessionKey` reuse the current sessionId, and update hook docs if `sessionKey` should be listed as a supported field.
