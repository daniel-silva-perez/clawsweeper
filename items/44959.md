## Summary

Keep open. Current `main` does not implement the Ollama-side promotion of Qwen-style embedded `<tool_call>...<function=...>` XML into executable `toolCall` blocks, so this PR's core request is not already fixed on `main`.

## Evidence

- **Ollama adapter still gates tool use only on native `message.tool_calls`:** `buildAssistantMessage()` in `extensions/ollama/src/stream.ts:523` only appends `toolCall` blocks from `response.message.tool_calls` and sets `stopReason` to `"toolUse"` only when that native array is present. Text content is added as plain text and never parsed for embedded XML tool calls.
  - file: [extensions/ollama/src/stream.ts:523](https://github.com/openclaw/openclaw/blob/4630ce3d9e2f089062ccd386b0ae241325961d1a/extensions/ollama/src/stream.ts#L523)

- **Current agent utilities do not contain the proposed Qwen embedded-tool parser:** `src/agents/pi-embedded-utils.ts:14-18` exports only existing text-sanitization helpers; there is no `parseQwenEmbeddedToolCalls` or equivalent parser on `main` in this module.
  - file: [src/agents/pi-embedded-utils.ts:14](https://github.com/openclaw/openclaw/blob/4630ce3d9e2f089062ccd386b0ae241325961d1a/src/agents/pi-embedded-utils.ts#L14)

## Risks / Open Questions

- The PR targets an older path (`src/agents/ollama-stream.ts`) while current `main` keeps Ollama runtime logic in `extensions/ollama/src/stream.ts`, so the branch likely needs a rebase/port rather than a straight merge.
- Current `main` already strips Qwen-style tool XML from visible text, so user-facing leakage is partly addressed; the remaining missing behavior is execution-path detection and promotion into structured tool calls.
