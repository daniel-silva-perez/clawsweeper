## Summary

Keep open. Current `main` still has the reported `llm-task` model-prefix bug, and there is not strong end-to-end evidence that the Ollama empty-output path is fully fixed in the plugin itself.

## Evidence

- **`llm-task` still concatenates provider + model verbatim:** `toModelKey()` returns `${provider}/${model}` with no normalization, and `execute()` passes the raw `params.model` through. A caller that supplies `provider: "anthropic"` and `model: "anthropic/claude-sonnet-4-6"` would still produce a doubled key like `anthropic/anthropic/claude-sonnet-4-6`.
  - file: [extensions/llm-task/src/llm-task-tool.ts:32](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/extensions/llm-task/src/llm-task-tool.ts#L32)
  - command: `nl -ba extensions/llm-task/src/llm-task-tool.ts | sed -n '32,39p;106,135p'`
  - sha: [45146913007d](https://github.com/openclaw/openclaw/commit/45146913007da723b4f06aba6a66cf4a695974c0)
- **Raw model override is forwarded unchanged into the embedded runner:** `runEmbeddedPiAgent` receives `provider` and `model` exactly as resolved above; there is no stripping of an already-qualified `provider/model` input before the runner call.
