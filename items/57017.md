## Summary

Keep #57017 open. Current main still drains pending writes and then reads the full cron JSONL file into memory in both per-job and all-jobs run-history paths before parsing, filtering, sorting, and slicing. The write path prunes after append, but the read path has no stat/read cap or read-time prune. The related PR #57088 is still open and unmerged, while PR #57913 was closed unmerged after the reporter noted the read-time prune race concern.

## Best Possible Solution

Keep the issue open until main ships a core fix. Prefer a bounded, non-mutating read in src/cron/run-log.ts using fs.open/read from the tail for descending/default history pages, with careful handling for filters/sort direction; otherwise, if maintainers accept mutating reads, thread a best-effort prune through the existing writesByPath serialization and runtime cron.runLog options. Cover readCronRunLogEntriesPage, readCronRunLogEntriesPageAll, gateway cron.runs, oversized files, prune/read failures, and concurrent append behavior. PR #57088 is the current related implementation attempt and should be reviewed or reworked rather than closing this bug now.
