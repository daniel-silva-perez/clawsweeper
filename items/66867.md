## Summary

Current main does not contain the idempotent repair described in the issue discussion. `tryRepairWritableBits` still returns `false` when a directory is already secure, so the reported invalid-then-already-tightened fallback path can still reach the existing `Unsafe fallback OpenClaw temp dir` throw. No open PR referencing this issue was found in the live public search.

## Best Possible Solution

Keep this issue open and fix it in `src/infra/tmp-openclaw-dir.ts` by making `tryRepairWritableBits` idempotent for already-secure owned directories, or by adding a narrow lock around the trust/repair path. Add regression coverage for both preferred and fallback paths where the first state check returns `invalid` and the repair stat sees `0o700`, then run the relevant changed test/typecheck lane.
