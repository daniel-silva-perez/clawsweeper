## Summary

Keep this issue open. Current main still has the reported temp-dir repair behavior: `tryRepairWritableBits` returns `false` when the repair-time stat already shows no group/other write bits, and the fallback invalid path can still throw `Unsafe fallback OpenClaw temp dir`. An open implementation candidate exists in #67108, but it is unmerged and should be reviewed/refined before this issue closes.

## Best Possible Solution

Keep this issue open. The best maintainer path is to review, refine, and land a narrow fix in `src/infra/tmp-openclaw-dir.ts` that treats expected concurrent permission repair as success by re-checking `resolveDirState`, including the already-secure owned-directory branch, then add regression coverage for preferred and fallback invalid-to-`0o700` sequences. #67108 is the current implementation candidate, but it should merge or be superseded before this issue closes.
