## Summary

Keep open. Current main still rejects `auth-profiles.json` entries whose normalized `type` is `aws-sdk` as `invalid_type`; the Bedrock provider-level `auth: "aws-sdk"` path exists, but the specific persisted marker regression from #69708 is not implemented. The linked fix PR #69753 was closed unmerged, and current main has no SDK-marker skip behavior or regression coverage.

## Best Possible Solution

Keep this issue open and land a narrow parser/test fix: treat `type` or legacy `mode` value `aws-sdk` as a non-secret SDK-managed marker during auth-profile store load, skip it without logging `invalid_type`, preserve warnings for real invalid entries, and keep actual Bedrock credential resolution on the existing provider-level `auth: "aws-sdk"` AWS SDK default-chain path.

## Risks / Open Questions

- The parser is shared by persisted and legacy auth-store loading, so a fix should preserve warnings for genuinely invalid entries while handling `aws-sdk` as a non-secret SDK-managed marker.
- Adding `aws-sdk` as a real `AuthProfileCredential` variant would affect downstream credential-state and OAuth switches; the narrower marker-skip approach from the linked PR is the lower-risk path.
