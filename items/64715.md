## Summary

Keep this issue open. Current main still exposes Telegram inline keyboards through the shared message/presentation send path, but does not expose Telegram native reply keyboards, ReplyKeyboardRemove, resize_keyboard, or one_time_keyboard semantics. The related closed issues are about inline buttons/callback handling, not this separate reply-keyboard capability.

## Best Possible Solution

Keep the issue open and implement native Telegram reply keyboard support in the Telegram plugin send/action path. A conservative path is to add a typed Telegram-owned reply_markup union for inline keyboards, reply keyboard rows/buttons, ReplyKeyboardRemove, resize_keyboard, and one_time_keyboard; thread it through extensions/telegram/src/button-types.ts, extensions/telegram/src/send.ts, extensions/telegram/src/action-runtime.ts, outbound delivery, tests, and Telegram/message-presentation docs. If maintainers want this portable across channels, define a generic quick-reply or persistent-keyboard presentation contract first, then map Telegram to native ReplyKeyboardMarkup.
