## Summary

Keep this issue open. Current main still models plugin slots as bare string IDs throughout the public config type, runtime schema, generated schema, normalization, activation, install/write, update, uninstall, tests, and docs. The requested `{ owner, claimed_at, claimed_by_version }` read/write shape plus takeover invariant remains a real core plugin/config contract change, not a duplicate, stale report, or ClawHub-only feature.

## Best Possible Solution

Keep the RFC open for maintainer design review. A concrete path would add a backwards-compatible slot ownership union, normalize legacy string slots to an owner object on read, update install/register/write paths to persist canonical ownership metadata, add explicit conflict and force/allow-override semantics for slot takeover, and cover validation, generated schema/help, docs, activation, install/update/uninstall, default slots, and `"none"` behavior with focused tests.

## Risks / Open Questions

- This is a public config and plugin contract change, so maintainers still need to choose the exact schema shape, migration order, provenance fields, and takeover semantics.
- Any implementation must preserve legacy bare-string configs, `"none"` disable behavior, inferred defaults such as `memory-core` and `legacy`, generated schema/help alignment, and install/update/uninstall compatibility.
- The existing `plugins install --force` behavior is about reinstalling/overwriting install targets, not slot takeover authorization, so slot override semantics need a deliberate design rather than assuming current force handling applies.
