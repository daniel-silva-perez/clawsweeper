## Summary

Keep #70823 open. Current main still models plugin slots as bare string IDs, and the RFC asks for a core config/schema/runtime invariant change with provenance, not a ClawHub-only plugin idea or already-shipped behavior.

## Best Possible Solution

Keep the RFC open for maintainer product/API judgment. If accepted, the best path is a backward-compatible slot value contract, likely `string | { owner: string; claimed_at?: string; claimed_by_version?: string }`, with read normalization, schema/type updates, install/update/uninstall write-path changes, explicit override semantics, and focused tests/docs across the plugin slot surfaces.
