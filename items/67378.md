## Summary

Keep this issue open. Current main still routes proactive WhatsApp direct sends through `toWhatsappJid(to)`, which converts phone-number targets to `digits@s.whatsapp.net`; the existing LID mapping support is only on the inbound LID-to-phone path, not the outbound phone-to-LID send path described by the reporter.

## Best Possible Solution

Keep the issue open and fix it in the WhatsApp plugin’s outbound path. The likely product fix is to add an account-aware outbound recipient resolver near `createWebSendApi` / active listener setup that can resolve phone-number targets to known LID JIDs using Baileys `signalRepository.lidMapping` and/or persisted authDir mapping files, then cover proactive text, media, poll, presence, and reaction paths with focused regression tests. The inbound `resolveJidToE164` helper and tests are useful reference material, but they do not currently solve outbound routing.
