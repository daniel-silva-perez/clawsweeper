## Summary

Keep open. Current `main` does not already contain the three session-management fixes, and the PR as submitted is not conservative to close out because review evidence shows at least two likely regressions plus unrelated script additions mixed into the session fix set.

## Evidence

- **Current UI still reloads on every final event, including no-message finals:** `shouldReloadHistoryForFinalEvent` is still a plain `payload.state === "final"` check, and the existing test suite explicitly expects a final event with no `message` payload to reload history.
  - file: [ui/src/ui/chat-event-reload.ts:3](https://github.com/openclaw/openclaw/blob/ecac696643c604c6d854341bfe6424bfa31423e9/ui/src/ui/chat-event-reload.ts#L3)
  - command: `nl -ba ui/src/ui/chat-event-reload.ts | sed -n '1,40p'; nl -ba ui/src/ui/chat-event-reload.test.ts | sed -n '12,28p'`
  - sha: [ecac696643c6](https://github.com/openclaw/openclaw/commit/ecac696643c604c6d854341bfe6424bfa31423e9)
- **No-message final reloads are still part of the current other-run flow:** `handleChatEvent` returns `"final"` for a final event from another run when there is no usable final assistant message, and `app-gateway` reloads history for that terminal state. That matches the Codex review comment that suppressing all no-message finals would hide persisted messages from other runs.
  - file: [ui/src/ui/controllers/chat.ts:437](https://github.com/openclaw/openclaw/blob/ecac696643c604c6d854341bfe6424bfa31423e9/ui/src/ui/controllers/chat.ts#L437)

## Risks / Open Questions

- Merging the chat-event change as written would likely suppress legitimate history reloads for final events from other runs that intentionally arrive without a message payload.
- Merging the session-utils change as written would likely break the existing freshest-duplicate selection contract that current tests enforce.
- Closing this PR would discard still-unimplemented cron orphan recovery work even though `main` does not yet contain that behavior.
