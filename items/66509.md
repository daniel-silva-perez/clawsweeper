## Summary

Keep this issue open. Current main has Telegram preview tool-progress support, but the QQBot gateway source that matches the reported compiled snippet still buffers `kind: "tool"` text, waits on a 60s tool-only fallback, and clears that fallback when a block/final response arrives. The provided related items do not supersede this QQBot-specific streaming gap.

## Best Possible Solution

Keep open for a QQBot-specific fix. The likely path is to deliver visible progress for QQBot when `streaming.mode: "partial"` is active, either via an immediate progress send or a QQBot preview/edit path, while preserving existing media fallback behavior and tool-summary suppression rules. Add a focused regression test in `extensions/qqbot/src/engine/gateway/outbound-dispatch.test.ts` that emits a `kind: "tool"` text delivery followed by a block/final reply and proves the progress text is visible instead of being cleared with the timeout. If a new config key is added, update the QQBot schema, manifest metadata, and docs.
