## Summary

Current `main` already contains shipped cron cleanup work for orphaned cron-side worker/process lifecycles. The v2026.4.22 changelog explicitly calls out cron/MCP cleanup for isolated cron runs and orphaned stdio wrapper/server processes, and the current code now tears down bundled MCP runtimes and browser/session resources in cron-run cleanup paths.

## Close Comment

Closing as `implemented_on_main` after Codex review. Current `main` already includes shipped cron cleanup for this class of leak: `CHANGELOG.md:209-210` under `2026.4.22` documents cron/MCP cleanup for isolated cron run ends and orphaned wrapper/server processes, `src/cron/isolated-agent/run-executor.ts:165` enables end-of-run cleanup for isolated cron jobs, `src/agents/pi-embedded-runner/run.ts:2254` retires bundled MCP runtimes in a `finally` block, and `src/gateway/server-cron.ts:333` also performs cron-session browser cleanup. There is also cleanup regression coverage in `scripts/e2e/cron-mcp-cleanup-docker-client.ts:120`, which waits for the cron-managed probe PID to exit after the run finishes. If you still see lingering workers on `v2026.4.22` or newer, please open a fresh issue with the exact version, how the process was launched, and `ps`/log output from the stuck PID so it can be distinguished from any remaining environment-specific process leak.
