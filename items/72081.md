## Summary

Keep #72081 open. Current main intentionally does not print `#token=...` in `openclaw dashboard` output, while still sending a tokenized URL to clipboard/browser for plaintext gateway tokens. That security behavior is covered by tests, but the issue is still actionable because the reported WSL/headless-style workflow can leave the user with only a non-tokenized printed URL, and the UI still tells users that `openclaw dashboard --no-open` gives a tokenized URL. Related #50614/#50615 make clear that printing bearer URLs is security-sensitive, so the right fix is a safe UX/product decision, not simply restoring token fragments in logs.

## Best Possible Solution

Keep the default no-token-in-terminal behavior, but add a safe, explicit path for headless or clipboard-unavailable users to authenticate the dashboard, such as clearer guidance to retrieve/paste `gateway.auth.token`, a deliberately named opt-in command/flag that avoids accidental log exposure, or UI/docs copy that no longer promises `openclaw dashboard --no-open` will print a tokenized URL. The implementation likely belongs in `src/commands/dashboard.ts`, its tests, and the dashboard/Control UI guidance surfaces.
