## Summary

Keep open. The issue is recent, concrete, and current main still has the reported CLI/Gateway and embedded-lock risk shape: `openclaw agent` waits for a final Gateway response with `(timeout + 30)s`, ignores the fast `accepted` ack, falls back to embedded mode on Gateway failure, and the embedded runner still acquires the session write lock with a 10s default acquisition timeout. The docs mention fallback but do not document a daemon-vs-embedded same-state limitation.

## Best Possible Solution

Keep this open as a real Gateway/CLI integration bug. The best fix is to clarify the CLI/Gateway contract so scripted `openclaw agent` either completes from the fast accepted `runId` plus an explicit `agent.wait` flow, or waits in a way that cannot falsely fall back to same-state embedded mode while the daemon is healthy. Embedded fallback should also gain a daemon/state-dir guard, separate state-dir support, or a clear refusal/error message, with docs under `docs/cli/agent.md` and `docs/tools/agent-send.md` plus regression coverage for the final-wait timeout and session-lock fallback behavior.
