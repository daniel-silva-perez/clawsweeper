## Summary

Keep open. Current main does not implement the proposed instruction-file write guard: write/edit tools are constructed with workspace boundary wrappers only, and apply_patch still applies add/update/delete/move hunks without protected-basename checks. The PR is a plausible security hardening candidate, has targeted tests in its diff, and the provided discussion shows the author followed up on review findings for apply_patch, path aliases, and symlink handling. It is not implemented on main and is not clearly superseded by the related closed, unmerged memory-write PR.

## Best Possible Solution

Keep this PR open for maintainer review or replacement with an equivalent core implementation. The best path is to decide the intended approval/product boundary, then land a guard in the agent mutation paths (`src/agents/pi-tools.ts`, `src/agents/pi-tools.read.ts`, and `src/agents/apply-patch.ts`) with regression tests covering write, edit, apply_patch, path normalization, and alias/canonical-target cases.
