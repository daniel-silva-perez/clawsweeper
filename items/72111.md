## Summary

Keep open. Issue #72111 describes a distinct current-main gap: OpenClaw supports streamable-http only as a URL-backed HTTP client transport, while the reported mixed `command` + `url` + `transport: "streamable-http"` config is resolved as stdio first and can plausibly time out exactly as reported. This is not covered by the related Accept-header issue #66940, which the reporter says is already resolved for pre-existing HTTP servers.

## Best Possible Solution

Keep the issue open for maintainer/product handling. The best fix is to choose and document one behavior: either implement a supported process-managed streamable-http mode that starts `command`, waits for the configured `url` to become ready, connects with `StreamableHTTPClientTransport`, and owns shutdown; or reject mixed `command`+`url`/HTTP transport configs early with a clear diagnostic and align docs/tests. In either case, add focused coverage around `mcp.servers` configs that include both process fields and HTTP transport fields.
