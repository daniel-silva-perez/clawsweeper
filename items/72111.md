## Summary

Keep open. Current main still treats the issue's mixed `command` + `url` + `transport: "streamable-http"` server config as stdio because stdio resolution wins first, so the reported 30000ms timeout remains plausible. The closed Accept-header issue #66940 does not supersede this one because #72111 is about managed child-process lifecycle or missing validation for a mixed transport shape.

## Best Possible Solution

Keep #72111 open. Maintainers should either add an explicit managed streamable-http child-process mode with readiness probing, HTTP client connection, lifecycle cleanup, and regression tests, or reject mixed `command` + `url`/HTTP transport configs early with a clear diagnostic and docs/test alignment. The current silent stdio precedence should not remain the user-visible timeout path.
