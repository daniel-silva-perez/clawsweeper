## Summary

Keep #52176 open. Current main still has no `OPENCLAW_INSTALL_UV` Docker build arg, while OpenClaw already supports uv-backed skill installers that require `uv` to be present. The PR addresses a real Docker deployment gap, but it is not ready to close or treat as implemented because review comments identify unresolved supply-chain/reproducibility and runtime PATH concerns. The posted `OPENCLAW_DOCKER_APT_PACKAGES` workaround is useful discussion, but it is not a documented supported uv install surface and does not supersede a proper Dockerfile/setup/docs change.

## Best Possible Solution

Keep the PR open and request a safer, complete Docker implementation: add a documented optional uv install surface only if it pins/verifies the uv artifact, installs both `uv` and `uvx`, ensures uv-installed tool binaries are usable by the `node` runtime user, wires the setup script/docs, and adds targeted Dockerfile tests. If maintainers decide uv should remain custom-image territory, close only after that explicit product decision, not as an automated cleanup.

## Risks / Open Questions

- The submitted patch should not merge as-is if it continues to execute an unpinned remote installer script without checksum/signature verification.

- A complete implementation should update Docker setup/docs and add a Dockerfile regression test so the build arg remains wired through supported deployment paths.
