## Summary

Keep this PR open. Current main still has the active-job asymmetry described in the PR: scheduled due runs and executeJob mark cron jobs active, but startup catch-up and manual run execution do not, while task-registry maintenance still relies on isCronJobActive for authoritative cron tasks after the 5-minute grace window. The PR remains a focused implementation candidate with relevant regression coverage, not cleanup to close.

## Best Possible Solution

Review and land this PR, or request targeted adjustments if maintainer review finds an issue. The product-correct path is to mirror markCronJobActive/clearCronJobActive around startup catch-up and manual cron runs, and keep the new symmetry regression tests so task-registry maintenance does not depend on a missing active-job signal.
