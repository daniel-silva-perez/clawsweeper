## Summary

Keep #49063 open. Current main still does not implement the PR's central behavior: Telegram native command handlers only read `ctx.message`, so a `channel_post` update has no native-command message path on main. The PR is not obsolete, but it also has unresolved authorization/session concerns from review comments, so the best maintainer cleanup is to keep it open for a narrower, security-preserving rework rather than auto-close it.

## Best Possible Solution

Keep #49063 open, rebase it onto current main, and rework the native Telegram command path to use a channel-post-aware message source while preserving existing access semantics: explicit chat allow when required, `commands.allowFrom` precedence, `sender_chat`-based identity where applicable, per-group/topic policy overrides, and callback/session routing. Add focused regression tests for allowed channel posts, denied channel posts, `commands.allowFrom`, wildcard-vs-explicit groups, and the relevant follow-up issue #49062 semantics.
