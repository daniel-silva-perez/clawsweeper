## Summary

Current main still contains the reported cross-agent requester-origin behavior. `resolveRequesterOriginForChild` resolves a bound account for `targetAgentId` when `targetAgentId !== requesterAgentId`, then uses that value ahead of the caller's `requesterAccountId`; the completion delivery path still feeds that origin into the bound-delivery router for the child session. The linked fix PR #70607 is relevant, but the provided context says it is still open and the current checkout has not adopted its proposed condition change.

## Best Possible Solution

Keep #70574 open. The maintainer path is to land an equivalent of PR #70607: preserve the parent's `requesterAccountId` for cross-agent `sessions_spawn`, keep any same-agent binding behavior intentional, and add focused regression coverage in `src/agents/spawn-requester-origin.test.ts` for Agent A -> Agent B plus nested `streamTo="parent"` completion routing before allowing the PR to close the issue.

The return value then takes precedence over `params.requesterAccountId` in the normalized requester origin.
