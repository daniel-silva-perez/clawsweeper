## Summary

Keep open. Current `main` still routes `sessions.list` through combined session-store discovery that only includes validated on-disk `sessions.json` files, and the synthetic `agent:main:main` fallback discussed in PR #46133 is not present on `main`. The issue is therefore still plausible for fresh/template-path setups where the main store file does not exist yet.

## Evidence

- **`sessions.list` depends entirely on combined-store loading:** The gateway RPC handler loads `loadCombinedSessionStoreForGateway(cfg)` and immediately passes that store to `listSessionsFromStore(...)`; there is no separate fallback that would inject `agent:main:main` for listing.
  - file: [src/gateway/server-methods/sessions.ts:579](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/src/gateway/server-methods/sessions.ts#L579)
  - command: `nl -ba src/gateway/server-methods/sessions.ts | sed -n '579,590p'`
  - sha: [f03252aaf9cc](https://github.com/openclaw/openclaw/commit/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022)
- **Current combined-store loader has no synthetic main-session injection:** `loadCombinedSessionStoreForGateway` merges entries from the loaded stores and returns them directly. In the current `main` file, the single-store and multi-store branches both return the merged map without any `ensureMainSessionKey`-style fallback.
  - file: [src/gateway/session-utils.ts:1011](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/src/gateway/session-utils.ts#L1011)
