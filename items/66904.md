## Summary

Keep open. Current main still does not capture Baileys `messaging-history.set` events or expose reconnect-history sync configuration, and the existing WhatsApp reconnect path intentionally suppresses stale append/offline history instead of adding it to agent-visible history.

## Best Possible Solution

Keep this open and implement it in the WhatsApp plugin runtime: add a typed `messaging-history.set` handler, introduce explicit opt-in reconnect-history configuration, normalize and dedupe synced messages through the existing inbound/history surfaces, and populate DM/group history or session context without accidentally auto-replying to stale backlog unless a separate opt-in says to. Update WhatsApp config types, zod schema, docs, generated config metadata as needed, and regression tests for sync-type filtering, stale backlog behavior, dedupe, and `historyLimit`/`dmHistoryLimit` interactions. Also account for Baileys `syncFullHistory`/`shouldSyncHistoryMessage` behavior and the prior on-demand limitations noted in #56088.
