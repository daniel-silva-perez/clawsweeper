## Summary

Keep #72157 open. Current main still has a browser Control UI Enter handler that only checks composition and connection before calling `onSend`; it does not gate on `props.sending`, `props.stream`, or the computed busy state. The related ClawSweeper items cover different WebChat duplicate-message paths or Enter-key preferences and do not supersede this fresh keyboard-path report.

## Best Possible Solution

Keep the issue open and fix the Control UI composer path. The safest product direction is to centralize keyboard and button send eligibility, add a regression test for rapid Enter while `sending` is true, and explicitly decide whether Enter should queue during an active stream or require the visible Queue button.
