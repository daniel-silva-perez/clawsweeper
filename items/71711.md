## Summary

Keep #71711 open. Current main has a real core cron gap: recurring `every`/`cron` jobs have no `activeUntil`/`expiresAt` field, no CLI/API/UI input for it, and no scheduler path that removes or disables recurring jobs after a deadline. The related #71712 RFC depends on this concept but does not supersede it.

## Best Possible Solution

Keep this open and implement it in core cron: extend the persisted cron job contract with optional `activeUntil` and `onExpire`, validate it through CLI/gateway/tool/UI paths, reject it for `at` jobs, check expiration before recurring execution and during recompute/list maintenance, remove or disable as configured, emit an auditable `cron-expired` event, and update docs/tests for `every` and `cron` schedules.

## Risks / Open Questions

- Expiration behavior needs a clear product decision for manual `cron run`, missed jobs after gateway downtime, and whether `disable` or `remove` should be default.
- Deleting recurring jobs by default should be paired with durable run/audit history so operators can see why a job disappeared.
