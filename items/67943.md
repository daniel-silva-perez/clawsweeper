## Summary

This issue should stay open. It is paired with open PR #67946 by the same author using closing syntax, and current `main` still contains the stale-lineage paths described in the report.

## Best Possible Solution

Keep this issue open while #67946 remains open. Maintainers should review, update, and either land that PR or replace it with an equivalent fix that clears persisted subagent lineage for ordinary top-level session turns while preserving lineage for real spawned sessions and adding regression coverage.

## Evidence

- **Open paired implementation PR:** The provided GitHub context shows PR #67946 is open, unmerged, by the same author, cross-referenced from this issue, and its body says it fixes this issue. Per cleanup rules, the issue should stay open while that implementation candidate is still open.
- **Gateway reuses stored spawnedBy for requested sessions:** For a requested session, the gateway derives `spawnedByValue` from `entry?.spawnedBy`, persists it back into the session entry, and passes it into the dispatched agent run. That matches the reported stale persisted lineage path for ordinary later turns.

- **Agent command falls back to stored spawnedBy:** The agent command path still computes `spawnedBy` as `normalizedSpawned.spawnedBy ?? sessionEntry?.spawnedBy`, so a normal run without live spawned context can still inherit stored lineage from the session entry.
  - file: [src/agents/agent-command.ts:903](https://github.com/openclaw/openclaw/blob/7c0fdae9b95bfcc667af3ad341b63ecc194464e4/src/agents/agent-command.ts#L903)
