## Summary

Close as `clawhub`. Current main keeps internal workspace hooks limited to command/message/session/gateway events, while the existing plugin hook surface already exposes tool lifecycle interception and approval gating. The request is useful, but it fits a plugin/ClawHub implementation rather than expanding OpenClaw core internal hooks.

## Risks / Open Questions

- Users who specifically want file-based internal hook scripts, not plugins, still do not have a first-class tool lifecycle event surface there.
- If maintainers later decide internal hooks should gain parity with plugin hooks for operator scripting, this issue may need to be reopened or superseded.

## Close Comment

Closing after Codex review. Current `main` keeps internal workspace hooks scoped to command/message/session/gateway lifecycle events (`docs/automation/hooks.md:11`, `docs/automation/hooks.md:34`) and sends deeper execution interception through the plugin surface instead. That plugin surface already exposes `before_tool_call`, `after_tool_call`, and `tool_result_persist` with the data this request asks for (`src/plugins/hook-types.ts:306`, `src/plugins/hooks.ts:777`), and it already supports approval gating via `requireApproval` (`docs/plugins/building-plugins.md:184`).

Given `VISION.md:52` (`core stays lean; optional capability should usually ship as plugins`), this fits a plugin/ClawHub solution rather than a core internal-hook expansion. The reported use cases here—audit logging, HITL approval, execution-aware memory/learning—map to that existing plugin path.
