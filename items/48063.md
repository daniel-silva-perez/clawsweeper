## Summary

Current main already implements the forced-drain restart handoff: after restart drain/timeout it always runs the post-close respawn path, and macOS launchd-supervised gateways now use the supervised restart path instead of detached self-respawn. The changelog shows this shipped in v2026.4.5.

## Close Comment

Closing from a Codex review because current `main` already implements this restart path. The gateway restart loop now continues into the post-close respawn handoff even when the drain window times out (`src/cli/gateway-cli/run-loop.ts:164`), and that handoff performs a full-process restart after releasing the gateway lock (`src/cli/gateway-cli/run-loop.ts:82`). On macOS, launchd-supervised gateways are explicitly detected via `LAUNCH_JOB_LABEL` / `LAUNCH_JOB_NAME` / `XPC_SERVICE_NAME` / `OPENCLAW_LAUNCHD_LABEL` (`src/infra/supervisor-markers.ts:1`) and use the supervised restart path instead of detached self-respawn (`src/infra/process-respawn.ts:20`). The regression is covered by restart-timeout and launchd restart tests in `src/cli/gateway-cli/run-loop.test.ts:229`. The changelog shows the macOS restart fix shipped in `v2026.4.5` (`CHANGELOG.md:1269`). If you still see a fresh repro on `v2026.4.5` or newer, please open a new issue with current logs from that version.
