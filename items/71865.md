## Summary

Keep open. Current main still has a raw byte-size drop guard that blocks config rewrites when canonical two-space JSON is less than half the previous file size, and `models auth login` still routes its auth metadata/default-model update through that guarded config write path. The issue is recent, non-maintainer-authored, has no protected labels, and the related ClawSweeper items are adjacent auth/clobber reports rather than a canonical fix for this PowerShell/BOM formatting case.

## Best Possible Solution

Keep the issue open and fix the config write guard. The best path is to add a semantic/normalized comparison for parseable existing config files before treating `size-drop:` as blocking, with a regression test that uses BOM/PowerShell-style verbose JSON and exercises the `models auth login` config-update path. The auth flow should also avoid leaving auth profile store changes half-applied when the follow-up config metadata write is rejected.
