## Summary

Keep this PR open. Current main still applies symmetric jitter after selecting a finite retryAfterMs, so an adversarial jitter sample can schedule a retry below the server-supplied lower bound. The PR is a focused implementation candidate for the paired open issue #68541, while the related #68600 is closed and unmerged.

## Best Possible Solution

Review and land a focused retryAsync fix that preserves server-supplied Retry-After as a lower bound whenever the configured maxDelayMs can honor it, with regression coverage for adversarial jitter, retryAfterMs == maxDelayMs, retryAfterMs > maxDelayMs, and non-integer retryAfterMs. After that merges, the paired issue #68541 can close as implemented; #68600 is closed and unmerged, so it does not supersede this PR.

## Risks / Open Questions

- The remaining maintainer judgment is the maxDelayMs policy: when retryAfterMs exceeds the local cap, the PR chooses symmetric jitter because the server hint is already unsatisfiable under the cap. That trade-off should be reviewed explicitly before landing.
