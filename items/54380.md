## Summary

Keep open. Current `main` still does not classify `application/msword` or `application/x-cfb` as binary in the media-understanding path, so there is no strong basis to close this PR as already implemented or obsolete. The repo also already recognizes those MIME shapes elsewhere, which makes the gap plausible rather than incoherent.

## Evidence

- **Current main binary MIME classifier omits legacy Office/container MIME types:** `isBinaryMediaMime` on current `main` marks images/audio/video, octet-stream, zip/gzip/rar/7z, `+zip`, and most `application/vnd.*` as binary, but not `application/msword` or `application/x-cfb`. Those attachments therefore fall through to later text-heuristic logic instead of being rejected up front.
  - file: [src/media-understanding/apply.ts:309](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/media-understanding/apply.ts#L309)
  - command: `nl -ba src/media-understanding/apply.ts | sed -n '300,340p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Text heuristics run after the binary-MIME gate:** In `extractFileBlocks`, the code checks `isBinaryMediaMime(normalizedRawMime)` before `looksLikeUtf8Text` / text coercion. Because the legacy Office MIME types are not in the binary list, they can continue into the text-detection path on current `main`.
  - file: [src/media-understanding/apply.ts:386](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/media-understanding/apply.ts#L386)
