## Summary

Keep PR #60485 open. Current main still rejects Feishu webhook requests with missing Lark signature headers before parsing `url_verification` challenge payloads, and the current e2e suite explicitly expects an unsigned challenge to return `401 Invalid signature`. The PR targets a real remaining Feishu/Lark webhook setup bug and adds plaintext/encrypted unsigned challenge coverage, though it needs rebasing against current main and should add the negative wrong-token test noted in review.

## Best Possible Solution

Keep the PR open as a valid bug-fix candidate. Rebase it onto current main, adapt the token comparison to the current `safeEqualSecret` helper/imports, keep non-challenge events behind the normal Lark signature check, and add the missing wrong-token negative test before merge consideration.
