## Summary

Current main still has the reported NO_REPLY leak shape: silent detection is exact-only, mixed-content normalization strips a trailing token but keeps any remaining preamble, and tests still assert that substantive text ending in NO_REPLY is delivered. Two open, unmerged PRs (#66755 and #69394) explicitly reference this issue as implementation candidates, so the issue should stay open until maintainers choose and merge a fix.

## Best Possible Solution

Keep this issue open and review the active implementation candidates #66755 and #69394. Maintainers should decide the silent-token policy, add regression coverage around src/auto-reply/tokens.ts and src/auto-reply/reply/normalize-reply.ts for bare think, <think>, and trailing-token cases, merge the selected fix, and track any streaming-buffering gap separately if it remains.

## Risks / Open Questions

- Maintainers still need to choose the intended policy: narrowly suppress reasoning-marker-wrapped silent replies, or broadly suppress any payload with a trailing silent token. The broader policy may intentionally change older #30916 behavior for substantive text followed by NO_REPLY.
- Even after a final-payload normalization fix, streaming/block delivery may need separate buffering if reasoning text is emitted and delivered before the final NO_REPLY token arrives.
