## Summary

Keep this issue open. Current main separates the TUI client identity `openclaw-tui` from the actual agent session key, and Ollama native tool transport has received several fixes since the reported 2026.4.12 build, but I did not find a direct fix, regression test, release note, or live reproduction proof showing that the specific `openclaw-tui` session-id failure is resolved. The issue is recent and detailed, so it should remain actionable rather than be closed as stale or implemented.

## Best Possible Solution

Keep the issue open and add a focused regression for the TUI UI-client path with `--deliver`, native Ollama tool calls, and the canonical agent session key. If the model is being shown or encouraged to pass `openclaw-tui` as a session reference, fix the prompt/session metadata or session-reference tools so tool calls use `current`, `main`, or the canonical `agent:main:main` session key instead of the TUI client id. If the failure is transport-specific, cover it in the Ollama provider tests with mocked native tool-call responses.
