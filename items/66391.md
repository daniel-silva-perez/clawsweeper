## Summary

Keep this issue open. Current main still separates the TUI client identity `openclaw-tui` from the canonical session key, and Ollama native tool transport has related fixes, but I did not find a direct fix, regression test, release note, or live proof that the reported `openclaw-tui` session-id failure is resolved.

## Best Possible Solution

Keep the issue open and add a focused regression for `openclaw tui --deliver` with the UI client identity `openclaw-tui`, canonical session key `agent:main:main` or `current`, and a native Ollama tool-capable response that exercises a session-aware tool. If the model is being shown or encouraged to pass `openclaw-tui` as a session reference, fix the prompt/session metadata or resolver guidance so tools use `current`, `main`, or the canonical agent session key instead of the TUI client id.
