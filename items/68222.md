## Summary

Keep open. Current main still exposes the Kimi Coding provider path, still rewrites Kimi tagged tool-call text into structured tool calls, and still treats `sessions_yield` as an intentional turn-ending tool. Existing fixes cover related Kimi replay/tool-argument issues and subagent yield fallback, but I did not find source, tests, or release evidence that fixes unexpected Kimi `sessions_yield` calls or the reported need to manually send a follow-up message to resume. The related #71273 review also kept the broader Kimi Code tool-call loop family open.

## Best Possible Solution

Keep this issue open. Reproduce against current main with a captured Kimi Coding transcript around the first `sessions_yield` call, including outstanding subagent/run state. If Kimi calls `sessions_yield` with no pending child completion or repeats it after a yielded result, fix the Kimi provider stream/replay path or the session-yield guard and add a focused regression in `extensions/kimi-coding/stream.test.ts` plus embedded runner coverage. If the root cause is missing delivery of a real yielded subagent completion, extend the direct-fallback path and QA-lab coverage to the Kimi Coding flow.
