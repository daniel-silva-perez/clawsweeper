## Summary

Current `main` already implements the underlying workspace cleanup this issue asks for: workspace files are separated from config/credentials/session state, agents can have isolated workspaces and agent state directories, and shared skills already live in standardized shared roots.

## Evidence

- **Workspace is already separated from system/state files:** The workspace docs explicitly say the agent workspace is separate from `~/.openclaw/`, which stores config, credentials, and sessions, and list managed skills under `~/.openclaw/skills` as outside the workspace.

## Close Comment

Closing from a Codex review because the requested workspace cleanup is already implemented in current OpenClaw. The current docs already separate the agent workspace from config/credentials/session state (`docs/concepts/agent-workspace.md:12` and `docs/concepts/agent-workspace.md:126`), multi-agent mode already gives each agent its own workspace plus `agentDir` and session store (`docs/concepts/multi-agent.md:8` and `docs/concepts/multi-agent.md:53`), and the resolver code enforces per-agent workspace/state paths (`src/agents/agent-scope-config.ts:149`). Shared skill locations are also standardized today via `<workspace>/skills`, `<workspace>/.agents/skills`, `~/.agents/skills`, and `~/.openclaw/skills` (`docs/tools/skills.md:13`). The supported management surface for this is `openclaw agents add ... --workspace ... --agent-dir ...` rather than a separate `openclaw workspace` command family (`docs/cli/agents.md:20` and `docs/cli/agents.md:100`). I also verified the same behavior is present in the released `v2026.4.22` tree (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). If there is still a concrete gap beyond the current per-agent workspace and shared-skills model, please open a focused follow-up issue with the exact missing behavior.
