## Summary

Current `main` does not implement lightweight label-based subagent instruction files. `sessions_spawn` still accepts explicit spawn parameters (`task`, optional `label`, `agentId`, `model`, `thinking`, etc.), while persistent role-specific behavior remains tied to full `agents.list` agent profiles or general skills, not auto-loaded `subagents/<role>.md` files.

## Evidence

- **`sessions_spawn` schema has no subagent-instruction-file input or lookup path:** The tool schema exposes `task`, `label`, `agentId`, `model`, `thinking`, `cwd`, context/sandbox options, and inline attachments. The executor reads those fields directly; there is no parameter or resolver for workspace/user `subagents/*.md` files.
  - file: [src/agents/tools/sessions-spawn-tool.ts:100](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/agents/tools/sessions-spawn-tool.ts#L100)
  - command: `nl -ba src/agents/tools/sessions-spawn-tool.ts | sed -n '95,230p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Subagent prompt uses `task` as the role definition and treats `label` as metadata only:** `buildSubagentSystemPrompt()` says the child was spawned for a specific task, injects `task` into 'You were created to handle: ...', and only includes `label` later under 'Session Context'. There is no role-file body injection.
  - file: [src/agents/subagent-system-prompt.ts:4](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/agents/subagent-system-prompt.ts#L4)

No label-to-frontmatter mapping exists.
