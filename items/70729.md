## Summary

Current main does not yet surface trusted-proxy identity from `X-Forwarded-User` into webchat agent message context. The auth layer extracts the trusted-proxy user, but the WebSocket client state drops it, and `chat.send` builds `SenderId`/`SenderName` from the generic Control UI client identity instead of the authenticated proxy subject.

## Best Possible Solution

Keep the issue open. The likely core fix is to persist the trusted-proxy auth subject on the WebSocket connection after successful auth, then have webchat-originated `chat.send` populate agent-visible identity, with tests covering trusted-proxy connect plus a webchat send. Maintainers should decide whether to reuse `SenderId`/`SenderName` or add a distinct authenticated-user field.
