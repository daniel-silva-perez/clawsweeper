## Summary

Keep open. Current `main` still has the missing Mattermost thread-context wiring described in #45082: proactive `message` tool sends still go through the Mattermost plugin send action, which ignores `threadId`, and threaded Mattermost sessions still do not persist full last-route delivery context needed for cron/heartbeat wake replies to target the thread reliably.

## Evidence

- **Plugin send path still bypasses Mattermost thread fallback:** `executeSendAction()` prefers the channel plugin action path before the core outbound send path. Mattermost declares `send` support, so tool-driven sends hit `mattermostPlugin.actions.handleAction()` first.
  - file: [src/infra/outbound/outbound-send-service.ts:143](https://github.com/openclaw/openclaw/blob/4630ce3d9e2f089062ccd386b0ae241325961d1a/src/infra/outbound/outbound-send-service.ts#L143)
  - command: `nl -ba src/infra/outbound/outbound-send-service.ts | sed -n '142,170p'; nl -ba extensions/mattermost/src/channel.ts | sed -n '108,116p'`
  - sha: [4630ce3d9e2f](https://github.com/openclaw/openclaw/commit/4630ce3d9e2f089062ccd386b0ae241325961d1a)
- **Mattermost send action still ignores `threadId`:** The Mattermost `send` action only derives `replyToId` from `params.replyToId` or `params.replyTo`; it does not fall back to `params.threadId`. That means recovered thread context is not applied when the message tool uses the plugin send action.
  - file: [extensions/mattermost/src/channel.ts:189](https://github.com/openclaw/openclaw/blob/4630ce3d9e2f089062ccd386b0ae241325961d1a/extensions/mattermost/src/channel.ts#L189)
  - command: `nl -ba extensions/mattermost/src/channel.ts | sed -n '184,205p'`
  - sha: [4630ce3d9e2f](https://github.com/openclaw/openclaw/commit/4630ce3d9e2f089062ccd386b0ae241325961d1a)
- **Mattermost tool context still lacks thread-specific context:** When a channel plugin does not implement `threading.buildToolContext`, the generic tool context only carries `currentChannelId`, `currentChannelProvider`, and `currentMessageId`. Mattermost's threading adapter currently exposes `resolveReplyTransport` only, so the tool path does not get a Mattermost-specific current thread/root id to auto-apply.
  - file: [src/auto-reply/reply/agent-runner-utils.ts:113](https://github.com/openclaw/openclaw/blob/4630ce3d9e2f089062ccd386b0ae241325961d1a/src/auto-reply/reply/agent-runner-utils.ts#L113)
  - command: `nl -ba src/auto-reply/reply/agent-runner-utils.ts | sed -n '111,123p'; nl -ba extensions/mattermost/src/channel.ts | sed -n '412,431p'`
  - sha: [4630ce3d9e2f](https://github.com/openclaw/openclaw/commit/4630ce3d9e2f089062ccd386b0ae241325961d1a)
- **Threaded Mattermost sessions still do not persist full delivery context for later wakes:** In the Mattermost monitor, `updateLastRoute()` is only called inside the `kind === "direct"` branch. Channel/thread sessions record `MessageThreadId` in inbound context, but they do not persist `deliveryContext.to/threadId` via `updateLastRoute()`. Later heartbeat/cron routing reconstructs targets from `lastTo` or `deliveryContext.to`, not from `origin.to`, so thread-scoped wakes still lack the stored route they need.
  - file: [extensions/mattermost/src/mattermost/monitor.ts:1570](https://github.com/openclaw/openclaw/blob/4630ce3d9e2f089062ccd386b0ae241325961d1a/extensions/mattermost/src/mattermost/monitor.ts#L1570)
