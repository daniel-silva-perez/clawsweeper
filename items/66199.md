## Summary

Keep open. Current main has not shipped an iMessage change that routes inbound body text through the untrusted-content path, and this PR is not safe to merge as-is because it wraps only the legacy `Body` envelope while the active reply path prefers raw `BodyForAgent`/`BodyStripped`. The existing review comments identify the same bypass, so the useful outcome is a revision or replacement patch that protects the actual model-bound iMessage text path with focused tests.

## Best Possible Solution

Keep this PR open for revision or replace it with a maintainer-reviewed iMessage hardening patch. The fix should make an explicit product/security choice about primary DM text, then protect the actual model-bound path (`BodyForAgent`/`BodyStripped` or `UntrustedContext` as assembled into the prompt) while keeping command parsing raw and deterministic. Add focused iMessage tests for prompt content, command parsing, reply context, history, and echo behavior.

## Risks / Open Questions

- Merging this PR as-is could create a false sense of iMessage prompt-injection protection because the normal model-bound text remains raw.
- The patch introduces randomized `wrapExternalContent` markers into legacy envelope/history rendering without proving deterministic echo, history, or dedupe behavior remains unchanged.
- There is no current iMessage test asserting that inbound message text reaches the prompt behind external-content markers or an equivalent untrusted boundary.
