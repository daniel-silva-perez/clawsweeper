## Summary

Current `main` has turn-time cooldown probing and fallback-cleared notifications, but it does not implement the requested background health-probe auto-recovery with sticky existing sessions. The issue remains a plausible feature request rather than something already implemented or obsolete.

## Evidence

- **Per-run probe logic only:** `runWithModelFallback()` keeps probe state in-memory and probes only while evaluating fallback candidates for a run. The throttle and near-expiry logic are hard-coded here; this is not a background recovery loop or configurable recovery feature.
  - file: [src/agents/model-fallback.ts:537](https://github.com/openclaw/openclaw/blob/40be5ad58187fd20b8defe5fc30e649190c131a8/src/agents/model-fallback.ts#L537)
  - sha: [40be5ad58187](https://github.com/openclaw/openclaw/commit/40be5ad58187fd20b8defe5fc30e649190c131a8)
- **No requested config surface found:** Search for the proposed config keys returned no matches, so there is no evident `fallbackRecovery` / `probeIntervalMinutes` / `notifyOnRecovery` config surface on current `main`.
  - command: `rg -n "fallbackRecovery|probeIntervalMinutes|notifyOnRecovery" src docs`
  - sha: [40be5ad58187](https://github.com/openclaw/openclaw/commit/40be5ad58187fd20b8defe5fc30e649190c131a8)
- **Current behavior does auto-clear on later turns:** The reply runner emits a `fallback_cleared` lifecycle event and verbose notice when a later turn returns to the selected model, which shows the current design is per-turn recovery rather than a background health-probe workflow.
  - file: [src/auto-reply/reply/agent-runner.ts:1526](https://github.com/openclaw/openclaw/blob/40be5ad58187fd20b8defe5fc30e649190c131a8/src/auto-reply/reply/agent-runner.ts#L1526)
  - sha: [40be5ad58187](https://github.com/openclaw/openclaw/commit/40be5ad58187fd20b8defe5fc30e649190c131a8)
- **Existing test proves fallback clears on the next run:** The e2e test expects the first run to show `Model Fallback:` and the second run in the same session to show `Model Fallback cleared:`, confirming current main already retries and clears fallback on subsequent turns instead of keeping the session pinned until a separate recovery mechanism runs.
  - file: [src/auto-reply/reply/agent-runner.runreplyagent.e2e.test.ts:872](https://github.com/openclaw/openclaw/blob/40be5ad58187fd20b8defe5fc30e649190c131a8/src/auto-reply/reply/agent-runner.runreplyagent.e2e.test.ts#L872)
  - sha: [40be5ad58187](https://github.com/openclaw/openclaw/commit/40be5ad58187fd20b8defe5fc30e649190c131a8)
- **No sticky-session fallback selection in agent startup:** Agent startup begins from the configured default model and only pins a session when there is an explicit stored override (`modelOverride` / `providerOverride`). That is not the sticky-fallback-for-existing-sessions behavior requested here.
  - file: [src/agents/agent-command.ts:652](https://github.com/openclaw/openclaw/blob/40be5ad58187fd20b8defe5fc30e649190c131a8/src/agents/agent-command.ts#L652)
