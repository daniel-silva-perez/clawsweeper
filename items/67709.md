## Summary

Keep this issue open. Current main has a startup-gating mitigation for `chat.history` and `models.list`, but the reported same-duration stall still has a plausible current code path: both RPCs can await the same first-load model catalog promise, and the browser Control UI request layer has no per-RPC timeout. The known changelog fix for startup sidecar gating predates the reported 2026.4.14 build, so it is not enough evidence to close this as implemented.

## Best Possible Solution

Keep the issue open and fix the remaining startup responsiveness path in the gateway/UI boundary: make `chat.history` return promptly without waiting on full model catalog discovery when only a thinking default is missing, make Control UI model catalog loading backgrounded or timeout-bounded, and add a regression test with a deferred `loadGatewayModelCatalog()` proving `chat.history`/startup chat rendering stay responsive while `models.list` is still loading.
