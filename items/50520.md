## Summary

Keep open. Current main has newer user-facing reply sanitization that strips copied inbound metadata during normal reply normalization, but it does not implement this PR's shared outbound-payload safety net in `normalizeReplyPayloadsForDelivery` / `createOutboundPayloadPlanEntry`. The PR remains relevant, though it still needs maintainer judgment on sanitizer scope because the broad helper can strip metadata-like blocks beyond a leading echoed prompt prefix.

## Best Possible Solution

Keep #50520 open for maintainer refinement or consolidation with the duplicate outbound-metadata PRs. The best code path is a single canonical fix around `createOutboundPayloadPlanEntry()` before directive parsing, using either `stripLeadingInboundMetadata()` or a dedicated outbound echo sanitizer that preserves legitimate timestamp-only and mid-message example text, with regression tests for leading echoed `Conversation info` / `Sender` blocks, directive preservation, and non-prefix sentinel examples. After one fix lands, close the duplicate PRs and #39847 together.
