## Summary

Keep open. Current main already covers the config-method scope-enforcement portion through reserved `config.*` admin handling, but it still does not implement the PR's central config HMAC sidecar signing/verification or integrity-warning reload behavior. The PR is also not merge-ready because review comments identify build-breaking and security/upgrade correctness blockers.

## Best Possible Solution

Keep this PR open for maintainer/security review or request changes. A mergeable version should separate unrelated CodexBar onboarding from config security, preserve the existing reserved `config.*` admin-scope behavior, design a backwards-compatible HMAC bootstrap/migration path for existing configs, propagate signature-write failures consistently, use injected filesystem/homedir dependencies for testability, and add focused tests for first-upgrade, tamper detection, reload behavior, and write failure handling.

## Risks / Open Questions

- Do not merge this PR as submitted: provided review comments identify a syntax error and unused import that would break validation.
- The proposed HMAC enforcement is security-sensitive and currently appears to risk upgrade lockout for existing configs without `.sig` files.
- The proposed signature write path is described as best-effort while read-side enforcement can reject missing signatures, creating a latent self-inflicted startup failure after write I/O errors.
- The PR mixes unrelated CodexBar onboarding changes into a config security PR, broadening review scope and increasing the chance of an unintended product change.
