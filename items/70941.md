## Summary

Keep this issue open. Current main still routes `openclaw models set` through the generic config replacement path, while the config writer still treats a raw more-than-half size drop as blocking and throws `CONFIG_WRITE_REJECTED` unless a caller explicitly opts into destructive writes. The models command tests still mock the persistence layer, so they do not cover the reported failure path.

## Best Possible Solution

Keep this open. Add a regression test that exercises a real `openclaw models set` write against a valid shrink-prone config, then fix the models command/config mutation path or add a narrowly scoped config-writer policy that accepts legitimate `agents.defaults.model.primary` updates while preserving the size-drop guard for true clobbers. Coordinate with the similar open size-drop/auth-login report so the fix addresses the generic raw-byte-shrink false positive rather than only this command.
