## Summary

Current main still lacks the behavior this PR targets: `parseConfigValue` only recognizes plain integers/decimals, so finite exponent notation falls through as a string. The live PR diff is focused on the parser regex plus colocated unit coverage, and the earlier heartbeat review comments are stale against the final file list.

## Best Possible Solution

Keep this PR open for maintainer review. The likely best path is to land the focused exponent-number parsing change with the colocated regression tests if maintainers agree that unquoted scientific notation should be numeric; otherwise document that exponent-looking values must be quoted to remain strings.
