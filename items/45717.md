## Summary

Current `main` does not show a run-over-run session-bloat path for these fields. `systemPromptReport` is stored as a single overwritten field, `skillsSnapshot` is only rebuilt for a real session rollover or snapshot invalidation, and the session-store loader now returns cloned data so nested snapshot mutations do not leak across loads.

## Evidence

- **`systemPromptReport` is overwritten, not appended:** After each run, the session entry assigns `next.systemPromptReport = result.meta.systemPromptReport`; there is no array push or merge of prior reports.
  - file: [src/agents/command/session-store.ts:107](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/src/agents/command/session-store.ts#L107)

  - sha: [f03252aaf9cc](https://github.com/openclaw/openclaw/commit/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022)
- **`skillsSnapshot` refresh is gated:** The snapshot only rebuilds when missing, when the snapshot version changes, when the skill filter changes, or when `isNewSession` is true; otherwise the existing snapshot is reused.
  - file: [src/agents/agent-command.ts:578](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/src/agents/agent-command.ts#L578)

  - sha: [f03252aaf9cc](https://github.com/openclaw/openclaw/commit/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022)
- **Stale sessions are a rollover condition, not an every-run path:** `isNewSession` is computed from session freshness, and freshness only flips when the configured reset policy says the session is stale. That means a stale session rolls once into a new session id; it is not treated as new on every subsequent run.

  - file: [src/config/sessions/store-load.ts:161](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/src/config/sessions/store-load.ts#L161)
