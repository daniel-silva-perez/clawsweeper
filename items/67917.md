## Summary

Keep open. Current main has adjacent Codex proxy/runtime transport hardening, but it does not prove the reported macOS TLS precheck regression is fixed. The OAuth login path still runs a TLS preflight and aborts on `UNABLE_TO_GET_ISSUER_CERT_LOCALLY` before starting OAuth, and the regression tests explicitly preserve that fail-fast behavior. `NODE_EXTRA_CA_CERTS` support exists as process-startup environment handling, not as a targeted fix for the reported case where direct Node fetch works with a custom CA bundle but OpenClaw's Codex precheck/runtime does not.

## Best Possible Solution

Keep this issue open for a maintainer or contributor to reproduce on macOS/Homebrew with an explicit `NODE_EXTRA_CA_CERTS` bundle and then fix the OpenAI Codex OAuth preflight/runtime boundary if OpenClaw's effective process env or dispatcher differs from direct Node. A good fix would add targeted diagnostics for effective `NODE_EXTRA_CA_CERTS`/`NODE_USE_SYSTEM_CA`, cert bundle readability, and gateway/CLI startup inheritance, plus regression coverage around the Codex OAuth precheck and runtime fetch path.
