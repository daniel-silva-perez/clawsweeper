## Summary

Keep open. Current `main` still clears self-declared scopes for device-less shared-token WebSocket operator connects, so a LAN/token WebSocket client without device identity can still hit `missing scope: operator.write`. The only scope-restoration change on `main` is on HTTP/plugin-route bearer-auth surfaces, not the WebSocket connect path described here.

## Evidence

- **WS shared-token operators still lose scopes without device identity:** The default-token gateway suite explicitly expects `operator + valid shared token => connected with cleared scopes`, then expects a follow-up RPC to fail with `missing scope`. The same suite also asserts `hello-ok` reports `auth.scopes` as `[]` for device-less shared-token auth.
