## Summary

Keep open. Current main implements model-provider HTTP proxy transport overrides (`env-proxy` and `explicit-proxy`) across the main LLM paths, and that partial work is documented in the 2026.4.5 changelog. However, the issue title explicitly asks for SOCKS5/HTTP support, and the current model-provider proxy stack is still HTTP-proxy oriented: it wires Undici `ProxyAgent`/`EnvHttpProxyAgent`, documents `request.proxy` as an HTTP proxy override, ignores `ALL_PROXY` for env-proxy routing, and has no `Socks5ProxyAgent`/SOCKS model-provider implementation or regression test.

## Best Possible Solution

Keep the issue open as the canonical remaining model-provider SOCKS5 request. The likely implementation path is to extend the shared provider dispatcher/fetch stack to recognize `socks://` and `socks5://` explicit proxy URLs, instantiate Undici `Socks5ProxyAgent` or an equivalent dispatcher with the same SSRF/TLS policy guarantees, add model-provider regression tests for SOCKS5, and update the public config docs to state exactly which proxy protocols are supported.
