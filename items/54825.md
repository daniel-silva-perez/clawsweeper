## Summary

Keep #54825 open. Current main sanitizes raw provider error JSON and has partial Telegram delivery-error controls, but it still deliberately emits billing/provider failures as user-facing error reply payloads and WhatsApp has no channel-level suppress/custom/owner-only error policy.

## Best Possible Solution

Keep the issue open and implement a generic user-facing assistant/provider error policy at the reply payload or channel dispatch boundary. It should support per-channel/account/group overrides, at least suppress and custom-message modes, and preferably owner-only routing. Wire it through WhatsApp and Telegram config schemas/docs/tests while preserving raw-error sanitization as the safe default when replies remain enabled.

## Risks / Open Questions

- Suppressing provider errors globally could make shared channels appear silently broken, so the default policy and owner/admin visibility need an explicit product decision.
- Telegram already has a partial error-policy surface, so a fix should avoid regressing existing Telegram delivery-error cooldown/silent behavior while adding the missing provider-error and WhatsApp coverage.
