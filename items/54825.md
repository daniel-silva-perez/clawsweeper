## Summary

Keep this issue open. Current main still turns billing/provider fallback failures into normal user-visible final replies, WhatsApp has no suppress/custom/owner-only error policy surface, and Telegram's adjacent error policy is wired to dispatcher delivery errors rather than the provider-failure final-reply path described here.

## Best Possible Solution

Keep this issue open and implement a shared outbound provider/runtime error-reply policy before channel delivery. The likely shape is a generic final-error classification plus per-channel/account/group/topic overrides for normal reply, silent suppression, custom text, and owner-only routing, with WhatsApp and Telegram coverage, diagnostics preserved in logs/control UI, and focused tests for WhatsApp group suppression/custom text, Telegram group suppression, owner DM visibility, and unchanged tool-error behavior.

## Risks / Open Questions

- Telegram docs and code currently use different policy vocabulary in places (`reply`/`silent` in docs versus `always`/`once`/`silent` in types), so any implementation should reconcile the public config contract instead of adding another incompatible model.
