## Summary

Keep open. Current main still does not implement the PR's central behavior: health snapshots are still built from config/probe data without a runtime snapshot overlay, the gateway health RPC still serves/cache-refreshes through the old cache path, and Discord still withholds the connected status patch when the gateway plugin is not already marked connected. The PR discussion shows a concrete live repro and an addressed bot-review blocker, so this remains plausible maintainer backlog rather than cleanup-close material.

## Best Possible Solution

Continue tracking this PR or consolidate it with a maintainer-chosen canonical health-runtime PR. The product fix should make health snapshots use the same runtime-backed account snapshot construction as `channels.status`, preserve explicit probe intent, update gateway health cache/broadcast behavior without overwriting runtime fields, remove the Discord `gateway.isConnected` startup gate, and land regression coverage for both health snapshots and the Discord race.
