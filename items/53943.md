## Summary

Current main still does not implement Slack `message` read by `messageId`. The generic runner can preserve arbitrary action params, but the Slack read dispatcher, runtime handler, and `readSlackMessages` helper only wire `channelId`, pagination fields, and optional `threadId`; no exact message timestamp is accepted, forwarded, or filtered.

## Best Possible Solution

Keep this issue open. The fix should stay in the Slack plugin read path: accept `messageId` in the Slack read action mapping, pass it through the internal `readMessages` handler, extend `readSlackMessages` to return/filter the exact Slack timestamp for both `messageId + threadId` and messageId-only reads, and add regression tests for both cases. If Slack URL parsing feeds this tool path, cover `pMESSAGE_TS?thread_ts=THREAD_TS` normalization in the same change. No broad core change is clearly required from the current evidence.
