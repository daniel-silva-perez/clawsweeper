## Summary

Keep open. Current main still sends OpenShell exec command text into the shared SSH `/bin/sh -c` wrapper without a generated-command or malformed-shell preflight, and generic repeated same-tool/same-args calls remain warning-level until the later global no-progress breaker. The prior ClawSweeper comment and local related reports are still consistent with the current checkout; adjacent loop-detection issues/PRs do not supersede the OpenShell malformed-command hardening request.

## Best Possible Solution

Keep this issue open. The best maintainer path is a focused runtime hardening change: add a shared exec/OpenShell preflight before SSH command construction for obvious generated-command hazards and malformed shell structure, then update loop detection so repeated identical failed or no-progress generic calls can become a clear critical block at a configured threshold. Add regression coverage for OpenShell/shared SSH command construction and before-tool-call loop handling, and separately clean up or lint bundled help/skill examples that look copy-executable while containing placeholders.

## Risks / Open Questions

- A command preflight needs to avoid false positives for legitimate shell syntax such as redirection, comparisons, command substitution, or literal angle brackets in data.
- A generic repeat hard-abort should likely use failed/no-progress outcomes and configurable thresholds so legitimate repeated operations are not blocked.
