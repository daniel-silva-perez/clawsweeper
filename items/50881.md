## Summary

Keep open. Current `main` does not implement the PR's broader idempotent Slack reaction handling in the shared reaction helpers; it only swallows `already_reacted` / `no_reaction` in specific Slack status-reaction dispatch paths.

## Evidence

- **Shared Slack reaction helpers still throw on duplicate/no-op reactions:** `reactSlackMessage` and `removeSlackReaction` directly await `client.reactions.add/remove` with no `already_reacted` or `no_reaction` guard, so the helper-level behavior proposed by the PR is not present on `main`.
  - file: [extensions/slack/src/actions.ts:95](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/extensions/slack/src/actions.ts#L95)
  - command: `nl -ba extensions/slack/src/actions.ts | sed -n '95,121p'`
  - sha: [ebd7f19a3b20](https://github.com/openclaw/openclaw/commit/ebd7f19a3b20001960474ec9ba9cc424e45f7b23)
- **Bulk remove path still bypasses the helper-level no-op handling:** `removeOwnSlackReactions` still calls `client.reactions.remove` directly inside `Promise.all(...)` instead of delegating through `removeSlackReaction`, so the PR's race-safe `no_reaction` handling is also absent on `main`.
  - file: [extensions/slack/src/actions.ts:123](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/extensions/slack/src/actions.ts#L123)

- **Current main only handles these errors at one Slack dispatch seam:** The Slack monitor status adapter now catches `already_reacted` and `no_reaction` around status-reaction updates, which reduces noise for that path only; this is not a repo-wide replacement for the PR's helper changes.
