## Summary

Keep PR #51001 open. Current main has generic recurring error backoff, but it still uses the natural recurring schedule as a lower bound, so long-period daily/hourly jobs do not retry earlier than their next scheduled slot. The PR’s central feature, per-job fixed-delay retry via `--retry-count`/`--retry-delay` that can run before the next natural cadence slot, is not implemented on main, and the linked issue #49740 remains a real product gap rather than cleanup-only stale work.

## Best Possible Solution

Keep the PR or the linked issue #49740 active. The best path is a scoped scheduler/API design for per-job recurring retries: add validated retry policy fields to the cron contract and CLI, schedule retries before the natural next run only when useful, preserve cadence after retry success/exhaustion, keep existing retry-storm backoff protections, reject or explicitly define one-shot retry flags, and cover add/edit/persistence/restart cases with focused tests.
