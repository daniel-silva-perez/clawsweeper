## Summary

Keep open. The issue includes a useful, locally validated Linux/systemd prototype in the comments, but current main still implements `openclaw logs` as `logs.tail` over the gateway file-log path with a configured local file fallback. I did not find the requested active systemd journal preference, provenance output, service PID/ExecStart reporting, or mismatch warning in current source, tests, or docs.

## Best Possible Solution

Keep this issue open and implement the local Linux/systemd path in `src/cli/logs-cli.ts`: detect the active `openclaw-gateway.service`, prefer `journalctl --user _PID=<pid>` for local follow when available, print CLI/service/log-source provenance before tailing, warn on mismatches, retain RPC/file-log behavior for remote `--url` and Control UI surfaces, add focused CLI tests for pairing/scope fallback to journal, and update `docs/cli/logs.md` plus `docs/logging.md`.

## Risks / Open Questions

- The unresolved behavior can still mislead operators after side-by-side cutover because the CLI may tail an old configured file-log context instead of the active service journal.
- A fix needs to preserve explicit `--url` remote RPC behavior and Control UI `logs.tail` file-log behavior, while adding a local Linux/systemd journal path only where appropriate.
- The implementation should define a stable text and JSON provenance contract so scripts consuming `openclaw logs --json` are not broken unexpectedly.
