## Summary

Keep open. The issue discussion includes a validated local Linux/systemd prototype, but current main has not incorporated that behavior: the logs command still uses the gateway `logs.tail` file-log path and falls back to the configured local log file instead of the active systemd journal, and docs/tests still describe and assert file-tail behavior.

## Best Possible Solution

Keep this issue open and implement the Linux local-service path in the logs CLI: detect the active `openclaw-gateway.service`, prefer `journalctl --user _PID=<pid>` when appropriate, print provenance before tailing, warn on CLI/service/log-source mismatches, retain RPC behavior for remote targets, add focused tests for pairing/scope fallback to journal, and update `docs/cli/logs` plus logging docs.
