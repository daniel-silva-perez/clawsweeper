## Summary

Keep #72314 open. Current main still drops inbound dedupe entirely when `MessageSid` is absent, so the PR is aimed at a real remaining gap rather than obsolete or duplicate work. The submitted patch needs changes before merge because the review comment correctly identifies a provider-ID cache-key compatibility regression, but that is a reason to revise the PR, not close it.

## Best Possible Solution

Keep the PR open and ask for a narrow revision: preserve the raw provider `MessageSid` suffix for existing id-bearing messages, use a clearly namespaced `content:<hash>` suffix only for idless messages, remove the dead `!messageKey` guard, and add regression coverage for idless retries, distinct timestamps, and provider-ID key compatibility across shared module instances.
