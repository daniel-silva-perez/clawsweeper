## Summary

Keep this issue open. Current main only mitigates persisted running cron jobs on startup; the reported safety problem still exists across out-of-hours catch-up replay, in-flight cron cancellation, gateway shutdown semantics, and lack of an emergency outbound stop surface.

## Best Possible Solution

Keep this open and fix it in cron/gateway core. Add a service-level registry for active cron run AbortControllers, make cron.remove/disable and gateway shutdown abort matching active executions, add an explicit emergency stop or pause-all outbound path, and add regression coverage for an America/Guatemala out-of-hours reconnect plus deleting or aborting an active WhatsApp-style isolated delivery. If #55160 becomes the canonical maintenance-window/deferred-replay design, link this issue there but do not close it before the relevant safety behavior ships.
