## Summary

Keep open. Current `main` still routes background exec completion through `exec-event` heartbeats, and `runHeartbeatOnce()` still short-circuits to `disabled` before handling `reason: "exec-event"` when the agent heartbeat interval resolves false (for example `heartbeat.every: "0m"`). That matches the later issue discussion narrowing the failure to disabled-heartbeat setups, and I found no code or docs evidence that this path has been fixed on `main`.

## Evidence

- **`exec-event` wake is still blocked when heartbeat cadence is disabled:** `runHeartbeatOnce()` resolves the agent heartbeat config and returns `{ status: "skipped", reason: "disabled" }` before any exec-event preflight if heartbeats are globally disabled, disabled for the agent, or `resolveHeartbeatIntervalMs(...)` is falsy. That means `reason: "exec-event"` does not surface completion when `heartbeat.every` is `0m`.
  - file: [src/infra/heartbeat-runner.ts:743](https://github.com/openclaw/openclaw/blob/ecac696643c604c6d854341bfe6424bfa31423e9/src/infra/heartbeat-runner.ts#L743)
  - command: `nl -ba src/infra/heartbeat-runner.ts | sed -n '736,756p'`
  - sha: [ecac696643c6](https://github.com/openclaw/openclaw/commit/ecac696643c604c6d854341bfe6424bfa31423e9)
- **Background exec completion still depends on a heartbeat wake:** When a backgrounded exec exits, the runtime enqueues a system event and immediately calls `requestHeartbeatNow(scopedHeartbeatWakeOptions(sessionKey, { reason: "exec-event", coalesceMs: 0 }))`. There is no separate completion-delivery path here.
  - file: [src/agents/bash-tools.exec-runtime.ts:300](https://github.com/openclaw/openclaw/blob/ecac696643c604c6d854341bfe6424bfa31423e9/src/agents/bash-tools.exec-runtime.ts#L300)
  - command: `nl -ba src/agents/bash-tools.exec-runtime.ts | sed -n '294,308p'`
  - sha: [ecac696643c6](https://github.com/openclaw/openclaw/commit/ecac696643c604c6d854341bfe6424bfa31423e9)
- **Gateway node exec completion still depends on the same heartbeat wake:** `exec.finished` node events enqueue the system event, then call `requestHeartbeatNow(... reason: "exec-event" ...)` for delivery. This confirms the current main path still relies on heartbeat execution to surface completion.
  - file: [src/gateway/server-node-events.ts:687](https://github.com/openclaw/openclaw/blob/ecac696643c604c6d854341bfe6424bfa31423e9/src/gateway/server-node-events.ts#L687)
  - command: `nl -ba src/gateway/server-node-events.ts | sed -n '687,699p'`
  - sha: [ecac696643c6](https://github.com/openclaw/openclaw/commit/ecac696643c604c6d854341bfe6424bfa31423e9)
- **Current docs describe automatic completion wake as conditionally unavailable, not fixed:** The background-process docs say to rely on automatic completion wake only "when it is enabled" and instruct users to use `process` polling/logging when automatic completion wake is unavailable. The same page documents `notifyOnExit` as enqueueing a system event plus requesting a heartbeat.
  - file: [docs/gateway/background-process.md:32](https://github.com/openclaw/openclaw/blob/ecac696643c604c6d854341bfe6424bfa31423e9/docs/gateway/background-process.md#L32)
