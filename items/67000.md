## Summary

Keep this issue open. Current main still has active-memory latency mitigations such as lightweight bootstrap, query shaping, and identical-query result caching, but it does not implement warm standby or persistent embedded-agent session reuse for active-memory. Each uncached active-memory lookup still constructs a fresh sub-agent session id, usually a fresh temp transcript, and calls the embedded runner, which then performs the same queueing, workspace/plugin/model/auth/hook setup and creates a new Pi session stack for the attempt. The related #50963 is broader subagent session-reuse work, but the provided context does not show a maintainer decision that it supersedes this active-memory-specific request.

## Best Possible Solution

Keep the issue open and treat it as active-memory embedded-agent warm reuse work. A maintainer-quality fix likely belongs either in the active-memory plugin as a bounded per-parent-session memory sub-agent reuse pool, or in a generic embedded-runner/session API if multiple plugin-owned subagents need the same warm standby semantics. The fix should define invalidation, preserve isolation/security boundaries, and add tests proving repeated active-memory turns reuse the intended initialized state or skip the expensive setup path.
