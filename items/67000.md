## Summary

Keep this issue open. Current main has some active-memory latency mitigations, including lightweight bootstrap for the memory sub-agent and a short TTL cache for identical recall queries, but it still starts a fresh embedded agent attempt for each uncached active-memory lookup. The active-memory plugin creates a new sub-agent session id and usually a new temp transcript file per call, then calls `runEmbeddedPiAgent`; the runner still performs the same per-attempt workspace, plugin, model, auth, hook, resource-loader, session-manager, and `createAgentSession` setup. The related #50963 is broader native subagent session reuse work and was also kept open in the provided context, but it does not make this active-memory-specific request obsolete.

## Best Possible Solution

Keep the issue open and treat it as active-memory embedded-agent warm reuse work. A maintainer-quality fix likely belongs either in the active-memory plugin by reusing a bounded per-parent-session memory sub-agent session, or in a generic embedded-runner/session API if multiple plugin-owned subagents need the same warm standby semantics. The implementation should define invalidation on config/model/auth/tool/prompt changes, avoid preserving unsafe stale tool/runtime state, and add tests proving repeated active-memory turns skip the expensive setup path or reuse the intended session state.
