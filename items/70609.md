## Summary

Keep open. Current main does not implement the PR's outbound Hermes/generic metadata propagation, and the remaining question is a real product/API boundary decision rather than an auto-close case. The PR should not merge as-is because it mixes Hermes-specific hardcoded behavior with a generic outbound contract change and includes unresolved review concerns, but there is no strong implemented/duplicate/clawhub-only close reason yet.

## Best Possible Solution

Keep the PR open for maintainer judgment, but do not merge it as-is. The best path is to split the unrelated auth commit, remove Hermes-specific hardcoded topic/bot names from core, and decide whether OpenClaw wants a generic outbound metadata contract across sendMessage, gateway send, delivery queue storage/recovery, and ChannelOutboundContext. If maintainers decide Hermes is purely optional, move the Hermes behavior to a ClawHub plugin and open a narrow plugin-API issue only for any missing generic metadata seam.
