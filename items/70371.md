## Summary

Current main still has the reported product gap. Plugin-local config schema failures are converted into global config validation issues under `plugins.entries.<id>.config`; gateway startup skips last-known-good recovery for plugin-local invalidity and then throws `Invalid config`, so the worker does not degrade by disabling only the bad plugin. The loader can record plugin errors in non-strict paths, but strict loader paths and tests still throw on plugin errors, and current docs/tests explicitly preserve the plugin-local fail-loud behavior. Related PR #70394 is still only a proposed fix in the provided context, and current main has not incorporated its `throwOnLoadError` changes.

## Best Possible Solution

Keep #70371 open. The fix should be in the config validation/startup/doctor path, not only the plugin loader throw flag: when validation failures are scoped to `plugins.entries.<id>.config`, quarantine that plugin entry in-memory as disabled/error, skip its registration and runtime surfaces, expose the exact path/schema failure in `openclaw doctor`, and continue gateway/CLI startup for unrelated plugins, channels, and jobs. Preserve fail-closed behavior for root config, mixed config, legacy migration, and security-sensitive failures.
