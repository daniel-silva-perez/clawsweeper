## Summary

Keep open. Current main still only supports workspace selection at the agent level for normal route bindings; `bindings[].workspace` is absent from the route binding type/schema, resolved routes do not carry a workspace, and `getReplyFromConfig` still resolves the workspace from the selected agent. The PR is a plausible core routing/config feature rather than ClawHub work or an obsolete duplicate. The discussion also remains useful: the author addressed the initial case-sensitive peer-id review in the PR head, while later review comments identify remaining implementation issues around full binding scope and agent-id normalization that should be resolved before merge.

## Best Possible Solution

Keep this PR open as the implementation candidate. The best path is to revise it so the matched route binding carries an optional workspace through `ResolvedAgentRoute`, update the strict config schema/types/docs, and have reply execution consume that resolved workspace without re-scanning bindings or losing channel/account/team/guild/agent-id normalization semantics. Add focused routing/config/reply tests for peer, account/channel scoped, uppercase peer-id, and case-insensitive agent-id cases before merge.
