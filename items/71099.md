## Summary

Keep #71099 open. The issue is recent, has a maintainer comment explicitly saying not to close yet, and current main still lacks a source-backed fix or regression coverage for the reported U+25C6 Telegram delivery failure. The Telegram reply path still sends rendered HTML/plain text through grammY `sendMessage`; existing ByteString hardening is for secret/header inputs, not outbound Telegram response content.

## Best Possible Solution

Keep the issue open. Ask for the stack frame or a minimal current-main repro that sends a reply containing `◆` through Telegram, then add a focused regression test in the Telegram delivery/send path and fix the exact byte-only boundary, likely a header/token/custom-fetch parameter rather than blanket sanitization of message text.
