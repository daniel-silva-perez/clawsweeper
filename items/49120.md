## Summary

Keep PR #49120 open. Current main still lacks the requested pre-persistence SSRF rejection in `normalizeHttpWebhookUrl`; runtime cron webhook POSTs are guarded by `fetchWithSsrFGuard`, but dangerous literal hosts can still pass create/update URL normalization and be persisted.

## Best Possible Solution

Keep #49120 open and rebase or land an equivalent narrow patch: call `isBlockedHostnameOrIp(parsed.hostname)` inside `src/cron/webhook-url.ts` after protocol validation, preserve the existing runtime `fetchWithSsrFGuard`, and add focused cron webhook URL tests for private, loopback, link-local, metadata, `.local`/`.internal`, `0.0.0.0`, and IPv4-mapped IPv6 cases.

## Risks / Open Questions

- This is security-adjacent behavior: landing the idea should get maintainer review because it changes currently accepted cron webhook URLs that point at private/internal hosts.
- Legacy `cron.webhook` config validation still uses a generic HTTP URL schema; maintainers may want to decide whether static rejection should also happen at config-load time, not only when cron resolves a target through `normalizeHttpWebhookUrl`.
