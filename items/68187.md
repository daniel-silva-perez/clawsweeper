## Summary

Current main still appears to have the reported failure mode. OpenClaw creates and caches one downstream MCP client/transport per session runtime, defaults HTTP MCP configs to SSE, and tool execution forwards through the cached client without detecting `Session not found`, disposing the affected downstream session, rebuilding the transport, or retrying/recreating on the next call. The related ClawSweeper matches are stale-session issues in other surfaces, not a canonical fix for this SSE-backed MCP runtime path.

## Best Possible Solution

Keep #68187 open. The best fix belongs in the bundle MCP HTTP/SSE runtime: classify stale downstream-session failures such as `Session not found`, dispose the affected cached `BundleMcpSession` or session runtime, clear any stale catalog/client state, reconnect with a fresh transport, and add a regression test using a fake SSE MCP server that restarts and rejects the old session.

- Retry policy needs care for mutating tools; the safest fix should prove the stale session rejected the request before retrying, or invalidate and make the next call recreate cleanly.
