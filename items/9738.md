## Summary

Google Cloud Speech-to-Text/Text-to-Speech is better scoped as a community provider plugin, not a new OpenClaw core/bundled integration. Current main explicitly keeps optional vendor capabilities in plugins, exposes first-class speech/STT plugin hooks, and uses ClawHub/community plugins as the intended distribution path.

## Risks / Open Questions

- There may not be an existing ClawHub/npm plugin for Google Cloud Speech yet; closing here only says the right home is a plugin, not that the functionality already exists.
- If product direction later changes toward first-party bundled Google Cloud speech onboarding, this scope call could be revisited.

## Close Comment

Closing this as a Codex review because this request fits the community plugin lane, not OpenClaw core. Current main explicitly scopes optional vendor capabilities to plugins in `VISION.md:54`, uses ClawHub/community plugins as the canonical discovery path for new providers in `docs/plugins/community.md:9`, and already exposes first-class plugin hooks for this exact surface via `registerSpeechProvider`, `registerRealtimeTranscriptionProvider`, and `registerMediaUnderstandingProvider` in `docs/tools/plugin.md:335` and `docs/plugins/sdk-provider-plugins.md:482`. Main also already has a bundled `google` plugin for Gemini-based speech/media in `extensions/google/index.ts:109` and `docs/tools/tts.md:10`, so Google Cloud Speech-to-Text / Text-to-Speech is best added as its own plugin and distributed through ClawHub or npm rather than expanded in core.
