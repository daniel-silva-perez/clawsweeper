## Summary

Keep open. Current `main` still preserves the reported behavior: the Control UI connects with client mode `webchat`, `chat.send` explicitly refuses to inherit an external delivery route for `webchat` clients on channel-scoped sessions, and agent delivery still resolves an internal/webchat reply channel back to `webchat` instead of the session’s stored external channel.

## Evidence

- **Clean checkout preserved:** `git status --porcelain` returned no output, so the checkout stayed clean during review.
  - command: `git status --porcelain`
- **Control UI identifies as webchat:** `ui/src/ui/app-gateway.ts:279-286` creates the browser client with `clientName: "openclaw-control-ui"` but `mode: "webchat"`. That means Control UI requests hit the webchat-specific routing path, not the separate `ui` mode path.
  - file: [ui/src/ui/app-gateway.ts:279](https://github.com/openclaw/openclaw/blob/40be5ad58187fd20b8defe5fc30e649190c131a8/ui/src/ui/app-gateway.ts#L279)

  - sha: [40be5ad58187](https://github.com/openclaw/openclaw/commit/40be5ad58187fd20b8defe5fc30e649190c131a8)

For a Telegram-backed session opened in Control UI, this forces `OriginatingChannel` back to `webchat`.

  - file: [src/gateway/server-methods/chat.directive-tags.test.ts:1314](https://github.com/openclaw/openclaw/blob/40be5ad58187fd20b8defe5fc30e649190c131a8/src/gateway/server-methods/chat.directive-tags.test.ts#L1314)
