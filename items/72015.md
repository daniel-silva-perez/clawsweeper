## Summary

Keep open. Current main has the hard-deadline mitigation for Active Memory timeout overshoot, but the core reliability request remains: Active Memory is still a foreground `before_prompt_build` hook that waits up to the configured timeout, still inherits the current/session model when `config.model` is unset, and has no nonblocking/deadline/concurrency mode. QMD startup also still arms QMD managers for every configured memory-enabled agent and each manager can start its own default boot update; no `activeOnly`, boot allowlist, jitter, or global boot-update concurrency control is present.

## Best Possible Solution

Keep the issue open or split it into two canonical follow-ups. In `extensions/active-memory`, decide and implement a safer foreground contract: nonblocking/background recall, a short fail-open deadline separate from the full subagent timeout, per-agent/session recall concurrency, and explicit fast-model default or warning. In QMD startup, add a maintainer-approved control such as global boot concurrency, jitter, active-agent-only initialization, or an on-boot allowlist, with docs and regression tests covering multi-agent gateways.
