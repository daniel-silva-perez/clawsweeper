## Summary

Keep PR #71063 open. Current main still has the unguarded initial `child.stdin.write/end` path in `createChildAdapter`, current tests do not cover the non-writable stdin case, and the PR has a concrete repro plus a focused adapter/test diff. This is not implemented, duplicated, or ClawHub-scoped. One caveat for maintainer review: a read-only Node probe showed `stdin.writable` can still be `true` immediately before an async `EPIPE` when a shell wrapper execs a missing binary, so the final fix may need error handling around the initial write/end in addition to or instead of only checking `writable`.

## Best Possible Solution

Keep PR #71063 open. The best path is a focused fix in `src/process/supervisor/adapters/child.ts` that makes initial stdin write/end safe for closed or failing child stdin, plus regression coverage in `src/process/supervisor/adapters/child.test.ts` for non-writable/destroyed stdin and, if feasible, async `EPIPE` after a missing binary exits.
