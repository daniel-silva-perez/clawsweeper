## Summary

Keep this PR open. Current main still has the unprotected initial stdin write in the child-process supervisor path, and the Gemini CLI missing-binary route can still reach that code. The proposed patch is relevant and focused, but the one-line `stdin.writable` guard may not be sufficient because a shell-wrapped missing binary can report writable first and then emit an async `EPIPE` on stdin.

## Best Possible Solution

Keep this PR open for maintainer review or contributor amendment. The best fix belongs in `src/process/supervisor/adapters/child.ts`: protect the initial `params.input` write/end path from both already-non-writable stdin and async stdin `error`/`EPIPE` after a child exits early, then add regression coverage in `src/process/supervisor/adapters/child.test.ts` for both shapes while preserving normal writable stdin behavior.

## Risks / Open Questions

- The current PR patch models `stdin.writable === false`, but the observed shell-wrapped failure can be `writable === true` followed by async `EPIPE`; landing only the one-line guard may leave the gateway crash path open.
- The PR touches supervisor child-process stdin handling, so maintainer review should confirm normal stdin delivery for live CLI sessions and pipe-open/pipe-closed modes remains unchanged.
