## Summary

Keep this PR open for maintainer rework rather than closing it as implemented or duplicate. Current main still lacks the PR's status-probe timeout cap, and the Feishu startup latency area is still active, but this PR is not safe to merge as-is: its single-account fire-and-forget change breaks the gateway `startAccount` promise lifecycle that current code depends on. A related open PR, #63481, is the safer candidate for the multi-account startup probe-concurrency bug, while this PR's status-timeout piece would need a narrower, lifecycle-safe change.

## Best Possible Solution

Keep this PR open for maintainer judgment, but do not merge it as-is. Preserve the long-lived `monitorFeishuProvider({ accountId })` promise for gateway lifecycle tracking, let #63481 or an equivalent queue-based fix handle the multi-account startup probe concurrency tracked by #63475, and split or rework the setup/status probe timeout into a small lifecycle-safe change with focused tests.
