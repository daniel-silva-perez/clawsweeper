## Summary

Keep #70003 open. Current main has useful mitigations for long-running work, including `chat.abort`, `/stop`, configurable `interrupt`, soft `steer`, and hard `/redirect`/`sessions.steer`. But the central request is still not implemented: normal same-session follow-up messages are queued while a run is active, and embedded agent runs still serialize through a per-session command lane rather than supporting concurrent active turns during a blocking tool call.

## Best Possible Solution

Keep the issue open and narrow it to the remaining product gap: decide whether normal inbound follow-ups during an active run should preempt, soft-steer, hard-interrupt, or start a concurrent same-session turn. A likely implementation would add an explicit same-session active-run policy, wire it through `chat.send` and/or `sessions.steer`, make blocking tools observe cancellation promptly, and add a gateway/UI regression test where a long-running tool is active and a later user message is processed immediately according to that policy.
