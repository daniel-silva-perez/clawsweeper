## Summary

Keep this issue open. Current main has Feishu topic-session routing fixes, including use of `thread_id` as a session key when `root_id` is absent, but the outbound reply target still falls back to the inbound `message_id` rather than resolving the topic root message. That leaves the reported Feishu topic-group reply bug plausibly unresolved for the issue’s exact scenario.

## Best Possible Solution

Implement a Feishu-owned topic-root resolver in the Feishu plugin path: when a group topic event has `thread_id` but no `root_id`/`parent_id`, query the thread container for the earliest/root `om_*` message, cache the `thread_id` to root-message mapping, use that root as `replyToMessageId` and `rootId` for normal, streaming, media, and ACP-error reply paths, and add regression tests for `thread_id`-only topic-group events so session routing and outbound reply anchoring are both covered.
