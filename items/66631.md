## Summary

Keep this issue open. Current main has adjacent Feishu topic-session fixes, especially `thread_id` session routing, but it still does not resolve a topic container id to the original `om_*` root message before sending replies. For the reported `thread_id`-only topic event, the outbound reply target still falls through to the triggering message id, which is the behavior the reporter says creates a new topic.

## Best Possible Solution

Implement the fix in the Feishu plugin path. When a group topic event has `thread_id` but no usable `root_id`, `parent_id`, or `reply_target_message_id` in a topic/reply-in-thread context, resolve the thread container to the earliest/root `om_*` message, cache `thread_id -> rootMessageId`, and use that root as both `replyToMessageId` and `rootId` for normal replies, streaming cards, media replies, and ACP/init error notices. Add regression tests for `thread_id`-only topic-group events that assert outbound reply anchoring, not only session routing.
