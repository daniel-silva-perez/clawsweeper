## Summary

Current main does not satisfy the requested Codex-authenticated voice transcription path. The bundled Codex media-understanding provider is explicitly image-only, the Codex plugin manifest advertises only image media-understanding capability, and the shared audio runner requires a provider `transcribeAudio` hook before an audio attachment can be transcribed. The request is also recent, has no maintainer/protected close signal, and the provided related items are adjacent auth/media reports rather than a canonical duplicate for this exact Codex transcription gap.

## Best Possible Solution

Keep this issue open. The best path is to treat it as a real Codex plugin/media-understanding feature request: verify the Codex app-server or ChatGPT-authenticated transcription contract, then add an audio transcription implementation to the bundled `codex` media-understanding or realtime transcription provider. If the existing audio provider request contract cannot carry the selected Codex auth profile and agent context safely, add a narrow generic SDK/runtime extension first, then document the supported `tools.media.audio` configuration and add regression tests for Telegram/attachment audio routing through Codex without `OPENAI_API_KEY`.

## Risks / Open Questions

- The requested behavior depends on whether the Codex app-server or ChatGPT/Codex login path exposes a supported, stable audio transcription/realtime API and whether its quota semantics are intentionally independent from normal Codex text-message limits. That upstream contract should be verified before implementation.
- The existing `AudioTranscriptionRequest` contract is API-key oriented and lacks the profile/auth store/context fields that the Codex image path uses, so a robust fix may need a small generic media-understanding runtime contract extension rather than only adding code inside the Codex plugin.
