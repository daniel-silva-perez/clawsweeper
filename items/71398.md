## Summary

Current main still has the reported QQ Bot SecretRef gap. A literal `secretref:/...` clientSecret is treated as a normal non-empty string, QQ Bot can pass that value to the QQ token endpoint, and QQ Bot still lacks the channel secret contract/target-registry coverage needed for normal runtime materialization and SecretRef tooling.

## Best Possible Solution

Keep this issue open. The fix should live in the QQ Bot plugin plus the shared SecretRef policy surface: add a QQ Bot secret contract and public contract artifact covering `channels.qqbot.clientSecret` and `channels.qqbot.accounts.*.clientSecret`, wire runtime assignment/materialization, update the SecretRef credential surface/docs, and add regression tests for structured SecretRefs and the literal `secretref:/...` case. For the literal marker, either migrate/resolve it precisely if it is supported legacy syntax, or reject it during config validation with actionable `openclaw doctor --fix` or structured SecretRef guidance.
