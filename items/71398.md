## Summary

Keep issue #71398 open. Current main still does not resolve the reported `secretref:/...` string form for QQ Bot `clientSecret`; that value is accepted as a plain string and can reach the QQ token request path. The prior issue discussion also already reached the same keep-open conclusion, and I found no current-main fix, release note, or duplicate that closes the regression.

## Best Possible Solution

Keep the issue open. Maintainers should decide whether `secretref:/...` was a previously supported file-provider shorthand that needs compatibility migration/runtime support, or invalid syntax that should fail validation/doctor with clear remediation. The likely fix is shared SecretInput validation/migration plus QQ Bot regression coverage for both `channels.qqbot.clientSecret` and `channels.qqbot.accounts.*.clientSecret`, with docs aligned to the supported structured file SecretRef form.

## Evidence

- **Issue discussion:** The issue reports a recent v2026.4.21 to v2026.4.23 regression using `channels.qqbot.accounts.*.clientSecret: "secretref:/..."`; hardcoding the same secret works. The existing Codex review comment from steipete also explicitly kept it open after checking the same behavior.
- **QQ Bot config path:** QQ Bot resolves `accountConfig.clientSecret` through the platform adapter when the field is configured, otherwise falls back to `clientSecretFile` or the default-account `QQBOT_CLIENT_SECRET`; there is no QQ Bot-specific `secretref:/...` parser or migration here.
