## Summary

Current main already implements restart catch-up for recurring cron jobs: startup clears stale running markers, runs missed recurring jobs immediately, replays missed cron slots based on lastRunAtMs, and staggers deferred backlog jobs. The request's underlying behavior is shipped, although it landed as built-in restart recovery rather than an opt-in `catchUp` schedule flag.

## Risks / Open Questions

- The issue body specifically proposed an opt-in `catchUp` flag, but current main implements catch-up behavior unconditionally for recurring jobs rather than exposing that flag.
- Docs in `docs/automation/cron-jobs.md` describe persistence and scheduling but do not explicitly call out restart catch-up semantics, so user discoverability may still be imperfect.

## Close Comment

Closing from a Codex review: current `main` already implements cron restart catch-up for recurring jobs. On startup, the cron service clears stale running markers and calls restart catch-up before normal scheduling (`src/cron/service/ops.ts:93-121`). The catch-up path replays overdue recurring jobs and missed cron slots, and staggers deferred backlog jobs instead of skipping them (`src/cron/service/timer.ts:962-1118`). The behavior is covered by regression tests for overdue recurring jobs, missed cron slots, and interrupted recurring vs one-shot jobs (`src/cron/service.restart-catchup.test.ts:79-220`). This also appears in the changelog as shipped in `v2026.3.7` (`CHANGELOG.md:2982`) with a follow-up backlog-staggering improvement in `v2026.3.8` (`CHANGELOG.md:2852`). The one difference from this issue’s proposal is that main ships the recovery behavior directly rather than adding a per-job `catchUp` flag.
