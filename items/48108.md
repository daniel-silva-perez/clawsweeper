## Summary

Keep open. Current `main` reloads the sessions list on `sessions.changed`, but the chat session dropdown still deliberately injects the active `state.sessionKey` even when that key is missing from `sessions.list`. That means a deleted currently-selected session can still remain visible in the dropdown, which matches the reported stale-entry behavior rather than disproving it.

## Evidence

- **Dropdown still forces current session into options even when absent from backend list:** `resolveSessionOptionGroups()` builds options from `sessions?.sessions`, then unconditionally calls `addOption(sessionKey)`. If the selected session key was deleted from `sessions.json` and is no longer returned by `sessions.list`, the UI still adds it back into the dropdown.

  - sha: [b45329334965](https://github.com/openclaw/openclaw/commit/b453293349657dedd069d291fa5b078d910067b1)
- **Behavior is covered by a test:** The test suite explicitly preserves an active session that is absent from `sessions.list` (`omitSessionFromList: true`), showing this is intentional current behavior rather than an old bug that has already been removed.
  - file: [ui/src/ui/chat/session-controls.test.ts:249](https://github.com/openclaw/openclaw/blob/b453293349657dedd069d291fa5b078d910067b1/ui/src/ui/chat/session-controls.test.ts#L249)
