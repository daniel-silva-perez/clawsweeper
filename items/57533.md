## Summary

The proposal asks for a native core orchestration/safety framework, but current project direction explicitly keeps optional orchestration layers out of core and routes that kind of policy/automation into plugins/hooks/ClawHub. Main already provides the low-level primitives this proposal depends on: inter-session provenance, cross-agent/session gating, and synchronous hook points that can block or rewrite message persistence.

## Close Comment

Thanks for the detailed proposal. After a Codex review of current `main`, this looks better scoped as a ClawHub/plugin-layer effort rather than an OpenClaw core issue.

- `VISION.md:54` says optional capability should usually ship as plugins, and `VISION.md:69` says new skills should go to ClawHub first.
- `VISION.md:106` and `VISION.md:107` explicitly call out default agent-hierarchy frameworks and heavy orchestration layers as out of scope for core.
- The low-level primitives this proposal needs already exist in core: structured inter-session provenance in `src/agents/tools/sessions-send-tool.ts:283`, transcript provenance handling in `src/sessions/input-provenance.ts:12`, and transcript hygiene docs in `docs/reference/transcript-hygiene.md:80`.
- Cross-agent/session access is already policy-gated in core through `tools.agentToAgent` and session visibility controls (`src/agents/tools/sessions-access.ts:94`, `docs/gateway/config-tools.md:240`, `docs/gateway/config-tools.md:255`).
- The remaining firewall/audit/orchestration behavior fits the existing extension seams: hook-only plugins are supported (`docs/plugins/architecture.md:54`), and `before_message_write` hooks can block or rewrite message persistence (`src/plugins/hooks.ts:897`).

Closing on that basis: useful direction, but it belongs in a plugin/ClawHub implementation unless there is a narrower core primitive with a separate scope and maintainer buy-in.
