## Summary

Keep open. Current main does not implement the proposed `neverReply` config, and the current plugin hook surface is not yet a complete substitute because generic `inbound_claim` is not broadcast for unbound conversations. The PR is broad and needs maintainer architecture/security review, but it is not obsolete or safely closeable.

## Best Possible Solution

Keep this PR open for maintainer product judgment. The best path is either to review and narrow/land a shared core `neverReply` policy with strong cross-channel tests, or first land a generic plugin API extension for global pre-agent inbound claims and then move listen-only/passive memory behavior to a ClawHub plugin such as the linked `openclaw-listen-only` implementation.
