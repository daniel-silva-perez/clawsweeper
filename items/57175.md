## Summary

Keep open. Current main has adjacent mitigations and provenance plumbing, but it does not implement the requested native session-isolation audit surface, and the issue still has linked active work.

## Evidence

- **No native isolation-audit surface on main:** A repo search for `doctor --isolation`, `audit sessions`, `isolationAudit`, `ISOLATION_VIOLATION`, `session isolation audit`, `session leakage`, and `ghost context` returned no matches in `src/` or `docs/`. That is strong evidence the proposed native command/logging surface is not implemented.
  - command: `rg -n "doctor --isolation|audit sessions|isolationAudit|ISOLATION_VIOLATION|session isolation audit|session leakage|ghost context" src docs`
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
- **Existing audit is narrower than the request:** `openclaw security audit` currently warns about shared-DM setups and emits the multi-user trust-model heuristic, but the docs describe config/risk checks rather than transcript-level cross-session leakage detection or per-reply isolation violations.
  - file: [docs/cli/security.md:28](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/docs/cli/security.md#L28)
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
- **Core security model does not promise hostile multi-tenant isolation:** The security docs explicitly say OpenClaw is a personal-assistant trust model, not a hostile multi-tenant boundary, and recommend separate gateways / OS users / hosts for adversarial isolation. That makes the issue's compliance/privacy framing important, not obsolete.
  - file: [docs/gateway/security/index.md:9](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/docs/gateway/security/index.md#L9)
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
- **Main has provenance plumbing, not the requested audit consumer:** Current code normalizes `InputProvenance`, tags inter-session sends with `sourceSessionKey`/`sourceChannel`/`sourceTool`, and replays that provenance into inter-session prefixes. This is supporting infrastructure, but it is not the proposed audit hook or violation logger.
  - file: [src/sessions/input-provenance.ts:12](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/src/sessions/input-provenance.ts#L12)
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
- **Issue still has active related work:** The provided issue timeline links open PRs `#58972` and `#63629` plus related open issue `#57533`. For conservative maintainer cleanup, that is not a strong close candidate.
