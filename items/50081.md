## Summary

Keep open. Current `main` still leaves the Control UI / `chat.send` attachment path capable of passing inline HEIC/HEIF images through as `image/heic` or `image/heif` on the current run. The only explicit HEIC->JPEG normalization on `main` is in the Gateway `input_image` HTTP path, not the webchat attachment path described in this issue.

## Evidence

- **Webchat attachment parser preserves inline MIME/data:** `parseMessageWithAttachments()` derives `finalMime` from the provided/sniffed MIME and, when the image is not offloaded, pushes the original base64 plus that MIME directly into `images` with no HEIC normalization step.
  - file: [src/gateway/chat-attachments.ts:455](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/src/gateway/chat-attachments.ts#L455)
  - command: `nl -ba src/gateway/chat-attachments.ts | sed -n '330,455p'`
  - sha: [45146913007d](https://github.com/openclaw/openclaw/commit/45146913007da723b4f06aba6a66cf4a695974c0)
- **Control UI send path uses parsed attachment images for the live run:** `chat.send` assigns `parsed.images` from `parseMessageWithAttachments()` into `parsedImages`, and the embedded runner later calls `activeSession.prompt(..., { images: imageResult.images })` for the current run.
  - file: [src/gateway/server-methods/chat.ts:2218](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/src/gateway/server-methods/chat.ts#L2218)
