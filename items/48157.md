## Summary

Keep open. Current `main` has Windows update hardening for environment/setup issues, but I did not find a fix for the reported self-update file-lock/`EBUSY rename` class: the package update path still runs the global package-manager install from the currently running CLI process, and the fresh-process handoff only happens after the install step.

## Evidence

- **Package update still runs in-process:** `runPackageInstallUpdate` invokes the global package-manager install directly via `runUpdateStep({ name: "global update", argv: globalInstallArgs(...) })` from the active CLI process.
  - file: [src/cli/update-cli/update-command.ts:378](https://github.com/openclaw/openclaw/blob/b453293349657dedd069d291fa5b078d910067b1/src/cli/update-cli/update-command.ts#L378)

- **Fresh-process handoff is post-install only:** The respawn path (`continuePostCoreUpdateInFreshProcess`) launches a fresh `node ... update` process only for post-core follow-up work, and it is triggered after `const postUpdateRoot = result.root ?? root;` once the package install has already completed.
  - file: [src/cli/update-cli/update-command.ts:900](https://github.com/openclaw/openclaw/blob/b453293349657dedd069d291fa5b078d910067b1/src/cli/update-cli/update-command.ts#L900)
