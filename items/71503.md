## Summary

Keep PR #71503 open. Current main still uses the older LAN address selection: exact `en0`/`eth0` preference followed by the first external IPv4, while pairing setup and the bundled device-pair plugin still choose the first private IPv4 they see. The PR remains a focused, active fix for the linked Docker bridge advertisement bug, and its comments show the author addressed the CNI/default-route review concerns while respecting the plugin boundary.

## Best Possible Solution

Keep the PR open for maintainer review and either land it or land an equivalent fix that gives gateway status/control UI links, setup-code QR URLs, and the bundled device-pair plugin the same LAN IPv4 ordering: default/physical LAN first, non-deprioritized fallback next, and Docker/bridge/CNI interfaces only as last fallback. A later public SDK/helper seam could reduce duplication, but that should not block this focused bug fix.
