## Summary

Keep #39900 open. Current main still does not implement OpenAI-compatible TTS `extraBody` passthrough. The config layer now accepts arbitrary provider-specific TTS keys, but the bundled OpenAI speech provider only normalizes known fields and the request builder sends a fixed `/audio/speech` JSON body.

## Best Possible Solution

Implement a scoped `extraBody` / `extra_body` option in the bundled OpenAI speech provider: normalize it as a plain object, pass it through to `openaiTTS`, merge it into the `/audio/speech` body with a documented collision policy, add focused tests under `extensions/openai`, and document the field on https://docs.openclaw.ai/tools/tts. If maintainers prefer plugin-first for arbitrary custom TTS transports, record that product decision explicitly; current main does not yet satisfy this issue.

## Risks / Open Questions

- The issue body is partly outdated because current config validation accepts unknown provider-specific TTS keys; the remaining bug is request-body passthrough, not schema acceptance.
- Adding `extraBody` needs an explicit collision policy for built-in keys such as `model`, `voice`, `response_format`, and `speed`, plus tests for official OpenAI endpoints and custom `baseUrl` endpoints.
