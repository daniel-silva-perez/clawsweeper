## Summary

Current main does not fully resolve #62872. The merged related PR only skips A2A for parent-owned one-shot ACP children; persistent/normal peer sessions still enter the sessions_send A2A follow-up path, including bounded ping-pong and an announce step. There is now a global session.agentToAgent.maxPingPongTurns config, but the sessions_send API itself still has no per-call control and maxPingPongTurns=0 only disables reply-back chaining, not the announce flow the issue also calls out.

## Best Possible Solution

Keep #62872 open as the canonical broader sessions_send A2A issue. The likely fix is a context-aware A2A policy or per-call/API control that can suppress ping-pong and possibly announce injection for persistent peer-session sends, while preserving the already-shipped parent-owned one-shot ACP child skip and legitimate peer delivery cases. Add regression coverage for persistent/session-mode targets and update docs/config guidance with the final behavior.
