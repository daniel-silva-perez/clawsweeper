## Summary

Keep open. Current `main` does not implement Bedrock `requestMetadata` wiring from OpenClaw config: the Amazon Bedrock plugin schema only exposes `discovery` and `guardrail`, the Bedrock runtime wrapper only adds region/guardrail/cache behavior, and the generic agent extra-param wrapper forwards only a small fixed set of stream options rather than arbitrary Bedrock request fields.

## Evidence

- **Bedrock plugin config surface lacks requestMetadata:** `extensions/amazon-bedrock/openclaw.plugin.json` defines only `discovery` and `guardrail` under `configSchema.properties`; there is no provider/plugin config field for `requestMetadata`.
  - file: [extensions/amazon-bedrock/openclaw.plugin.json:8](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/extensions/amazon-bedrock/openclaw.plugin.json#L8)
  - command: `nl -ba extensions/amazon-bedrock/openclaw.plugin.json | sed -n '1,260p'`
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
- **Bedrock runtime wrapper does not inject requestMetadata:** `registerAmazonBedrockPlugin()` registers `wrapStreamFn`, but that hook only handles Bedrock region, guardrail injection, and prompt-cache behavior for Bedrock models/app inference profiles. There is no handling for `requestMetadata` from config or extra params.
  - file: [extensions/amazon-bedrock/register.sync.runtime.ts:325](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/extensions/amazon-bedrock/register.sync.runtime.ts#L325)
  - command: `nl -ba extensions/amazon-bedrock/register.sync.runtime.ts | sed -n '220,420p'`
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
- **Generic agent params do not pass arbitrary Bedrock request fields through:** `resolveExtraParams()` merges `agents.defaults.params`, per-model `params`, and `agents.list[].params`, but `createStreamFnWithExtraParams()` only forwards `temperature`, `maxTokens`, `transport`, `openaiWsWarmup`, `cachedContent`, and `cacheRetention`. Arbitrary keys like `requestMetadata` are not forwarded to the provider call.
  - file: [src/agents/pi-embedded-runner/extra-params.ts:66](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/src/agents/pi-embedded-runner/extra-params.ts#L66)
  - command: `nl -ba src/agents/pi-embedded-runner/extra-params.ts | sed -n '1,220p' && nl -ba src/agents/pi-embedded-runner/extra-params.ts | sed -n '279,349p'`
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
- **No current-main references to requestMetadata:** Repository search over `extensions/amazon-bedrock`, `src`, and `docs` returned no `requestMetadata` matches, which is consistent with the missing config/runtime wiring.
  - command: `rg -n "requestMetadata" extensions/amazon-bedrock src docs`
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)

## Risks / Open Questions

- `agents.*.params` is intentionally generic, so a future implementation could add Bedrock-specific pass-through without a new top-level schema key. On current `main`, though, the runtime forwarding path still drops `requestMetadata`.
- `gh` inspection was unavailable because `GH_TOKEN` was not set in this environment, but local code/docs inspection was sufficient for this review.
