## Summary

Keep open. Current `main` has partial restart mitigation, but it does not fully implement the issue’s requested gateway-wide session recovery/readiness behavior. The code now drains active work before restart, surfaces restart errors, resumes some subagent sessions, replays interrupted recurring cron jobs, and can continue one explicitly tagged session via a single restart sentinel. But there is still no general pre-restart manifest of all active sessions, no restart readiness gate on `openclaw gateway restart`, no drain-time message queue/replay path, and docs still acknowledge restart-loss cases for subagents.

## Evidence

- **Restart drain rejects new work instead of queueing/replaying it:** `run-loop` marks the gateway draining during restart, and `command-queue` rejects new enqueues with `GatewayDrainingError` rather than buffering them for replay. That conflicts with the issue’s requested drain-aware message queuing.

- **Drain path explicitly rejects enqueues:** `markGatewayDraining()` flips a global flag, and `enqueueCommandInLane()` immediately rejects with `GatewayDrainingError`; `resetAllLanes()` only preserves already-queued entries. There is no manifest/replay path for inbound messages received during drain.
  - file: [src/process/command-queue.ts:228](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/src/process/command-queue.ts#L228)
