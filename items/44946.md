## Summary

Current `main` already implements the requested configurability and no longer treats quiet WhatsApp inbound silence as stale-socket evidence.

## Close Comment

Closing from a Codex review: current `main` already covers this request. The stale threshold is configurable through `gateway.channelStaleEventThresholdMinutes` in the config schema (`src/config/zod-schema.ts:747`) and is wired into the gateway health monitor (`src/gateway/server-runtime-services.ts:36`). In addition, stale-socket detection now relies on provider-proven transport activity and explicitly ignores stale app-event timestamps when no transport-liveness signal exists (`src/gateway/channel-health-policy.ts:117`, `src/gateway/channel-health-policy.test.ts:139`). WhatsApp’s current account snapshot reports inbound/event timestamps but not `lastTransportActivityAt` (`extensions/whatsapp/src/channel.ts:243`), so the overnight false-restart pattern described here does not match current `main`. I could not confirm a released tag for this from the available history, so `fixedRelease` is left unset.
