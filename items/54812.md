## Summary

Keep open. Current main has channel-level and multi-account `configWrites` gates, but it does not implement the requested per-context controls such as `dmConfigWrites` or `groups.<id>.configWrites`. The enforcement path, public SDK helper, docs, and strict channel schemas are all scoped to channel/account config writes, not per DM/group/conversation type.

## Best Possible Solution

Keep the issue open and design an additive, backward-compatible config-write policy that can evaluate channel, account, conversation kind, and conversation id. The likely path is to extend the shared config-write SDK/core helpers, pass conversation context from inbound command handling, add strict schema/docs support for per-DM/per-group overrides where channel configs already model those contexts, and add contract tests proving channel-level defaults, account overrides, and context overrides compose correctly.

## Risks / Open Questions

- Current main has partial account-level support, so any implementation should preserve existing `channels.<provider>.configWrites` and `channels.<provider>.accounts.<id>.configWrites` semantics while adding narrower context overrides.
- A good design needs a shared origin context contract for plugin channels so Telegram, Signal, WhatsApp, and future channels can evaluate DM/group policy consistently without channel-specific bypasses.
