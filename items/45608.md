## Summary

Keep open. Current `main` still does not run the existing agentic memory flush before session teardown on `/new`, `/reset`, daily reset, or idle reset. The only production caller of `runMemoryFlushIfNeeded()` is the normal reply/compaction pipeline. Reset flows rotate/archive the session and emit hooks, but they do not invoke the agentic flush. Related reset-memory visibility improved on `main` because startup context now reads date-prefixed slugged memory artifacts, so the #45607-style filename mismatch is no longer the blocker here.

## Evidence

- **Agentic memory flush is still only wired into the active reply pipeline:** `runMemoryFlushIfNeeded()` is called from `agent-runner.ts` right after preflight compaction and before the normal follow-up turn. I did not find any production reset/session-rollover caller.
  - file: [src/auto-reply/reply/agent-runner.ts:1121](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/src/auto-reply/reply/agent-runner.ts#L1121)
  - command: `rg -n "runMemoryFlushIfNeeded\(" src`
- **Session rollover on manual and stale resets creates a new session and archives the old transcript without invoking the agentic flush:** `initSessionState()` captures `previousSessionEntry` for explicit resets and stale daily/idle resets, then creates the new session entry, archives the old transcript, and emits `session_end`/`session_start` hooks. There is no call to `runMemoryFlushIfNeeded()` anywhere in this reset path.
  - file: [src/auto-reply/reply/session.ts:455](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/src/auto-reply/reply/session.ts#L455)
  - command: `nl -ba src/auto-reply/reply/session.ts | sed -n '430,870p'`
- **Gateway reset service also lacks a pre-reset agentic flush:** `performGatewaySessionReset()` updates the session store, emits `before_reset`, archives transcripts, and emits lifecycle hooks. The `before_reset` hook is fire-and-forget and receives transcript messages, but this service never runs the core silent memory-flush turn.
  - file: [src/gateway/session-reset-service.ts:423](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/src/gateway/session-reset-service.ts#L423)
  - command: `nl -ba src/gateway/session-reset-service.ts | sed -n '420,680p'`
- **Existing reset preservation remains mechanical/plugin-based:** The bundled `session-memory` hook still handles only explicit `command:new` / `command:reset` events and writes `memory/YYYY-MM-DD-<slug>.md` from recent transcript content. It is not the same as the core agentic flush path that runs a silent tool-enabled turn.
  - file: [src/hooks/bundled/session-memory/handler.ts:53](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/src/hooks/bundled/session-memory/handler.ts#L53)
  - command: `nl -ba src/hooks/bundled/session-memory/handler.ts | sed -n '45,210p'`
- **The related slugged-file visibility problem is already addressed on main:** Startup context now scans `memory/YYYY-MM-DD-*.md` alongside canonical daily files, and tests explicitly cover loading slugged reset-memory artifacts. That means the remaining gap is the missing pre-reset agentic flush itself, not the old filename visibility mismatch.
  - file: [src/auto-reply/reply/startup-context.ts:233](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/src/auto-reply/reply/startup-context.ts#L233)
  - command: `nl -ba src/auto-reply/reply/startup-context.ts | sed -n '228,310p'; nl -ba src/auto-reply/reply/startup-context.test.ts | sed -n '52,70p'`

## Risks / Open Questions

- I did not run a live reproduction because the review was read-only, but the current production call graph is clear enough to show the requested behavior is not implemented.
- There are newer lifecycle hooks (`before_reset`, `session_end`, `session_start`) that partially improve reset extensibility, so someone could mistake that for a fix even though the core agentic flush requested here still does not run.
