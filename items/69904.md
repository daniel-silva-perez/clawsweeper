## Summary

Keep PR #69904 open. Current main still does not implement the proposed stale transient async notice policy, and the PR is a plausible focused fix. It should not be closed as obsolete, but it also needs revision before merge because review identified a grounded regression risk around non-completion exec-event notices.

## Best Possible Solution

Keep the PR open and revise it rather than close it. The best path is to add the stale-drop policy on main's system-event queue, pass the latest session activity timestamp into reply and heartbeat drains, mark only transient success/completion-style events as drop-on-session-advance, preserve durable failures/timeouts/blocked and non-completion exec lifecycle notices, and add regression tests for stale success drops, mixed stale+cron queues, and `Exec started`/`Exec denied` retention.
