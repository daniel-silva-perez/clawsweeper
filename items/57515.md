## Summary

Keep open. Current main dedupes top-level backup sources and has regression coverage for a symlinked workspace path, but there is no direct fix or test for symlink descendants inside an included workspace tree. The reported `--verify` failure remains plausible on current `main`, so there is not strong enough evidence to close conservatively.

## Evidence

- **Top-level backup sources are canonicalized and deduped:** `resolveBackupPlanFromPaths` canonicalizes candidate paths, sorts by canonical path, and skips covered workspace entries when they live under the state dir. That addresses duplicate top-level assets, not archive traversal inside a workspace tree.
  - file: [src/commands/backup-shared.ts:160](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/src/commands/backup-shared.ts#L160)
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
- **Existing regression only covers symlinked workspace roots:** The only symlink-specific backup regression asserts that a workspace path which is itself a symlink does not duplicate the state backup entry. It does not cover a symlink located inside the workspace contents.
  - file: [src/commands/backup.test.ts:121](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/src/commands/backup.test.ts#L121)
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
- **Archive creation still hands whole source roots to tar without explicit descendant-symlink filtering:** `createBackupArchive` passes each included asset root directly to `tar.c(...)` and only rewrites entry paths. There is no code here that explicitly excludes, dereferences, or dedupes symlink descendants inside an archived tree.
  - file: [src/infra/backup-create.ts:353](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/src/infra/backup-create.ts#L353)
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
- **Verify still hard-fails on any duplicate normalized archive path:** `backupVerifyCommand` rejects the archive as soon as two entries normalize to the same path, which matches the reported failure mode if tar emits duplicates for a workspace tree with symlink aliases.
  - file: [src/commands/backup-verify.ts:295](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/src/commands/backup-verify.ts#L295)
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
- **No backup code/test change after the latest release:** Diffing `v2026.4.23..HEAD` for the relevant backup files showed no code or test changes beyond docs, so there is no newer main-branch fix to point to.
  - command: `git diff --stat v2026.4.23..HEAD -- src/commands/backup-shared.ts src/commands/backup.test.ts src/infra/backup-create.ts src/commands/backup-verify.ts docs/cli/backup.md`

## Risks / Open Questions

- The current evidence shows protection against duplicate top-level backup assets, but not against duplicate tar entries caused by symlink descendants within a workspace tree.
- Closing as implemented_on_main would overclaim: the code and tests inspected do not directly cover the reporter's exact reproduction path.
- The checkout is clean and unchanged; this review did not run a write-producing reproduction or test lane.
