## Summary

Keep #71382 open. Current main does not implement a WhatsApp transport-level read-only mode: the WhatsApp config type/schema/account resolver lack `readOnly`, and send/inbound paths still perform outbound sends, presence updates, read receipts, and pairing replies. The existing `sendPolicy` surface is session-layer delivery control, not the hard WhatsApp transport gate requested by this PR. The PR discussion also contains actionable follow-ups that still need branch-owner or maintainer changes before merge.

## Best Possible Solution

Keep the PR open or replace it with a maintainer-owned patch that adds channel/account `readOnly` config, schema, metadata, account resolution, and transport-level WhatsApp enforcement. The implementation should block text/media sends, polls, reactions, typing/presence, read receipts, listener send APIs, and pairing replies while preserving inbound ingestion and pairing request creation, with focused regression tests for those paths.
