## Summary

Keep this PR open. Current main still lacks a WhatsApp channel/account readOnly config surface and still has WhatsApp transport paths that can send presence, read receipts, pairing replies, messages, reactions, polls, and typing indicators without a transport-level read-only gate. The PR is a plausible implementation candidate for the linked WhatsApp read-only request and broader #54519, but the review discussion identifies concrete fixes needed before merge.

## Best Possible Solution

Keep this PR open and finish the WhatsApp transport-level read-only implementation here or in a direct replacement patch. The shipped fix should add channel/account readOnly config and generated metadata, resolve it per account, pass it into the WhatsApp inbound monitor, hard-gate outbound messages/media/polls/reactions/typing/presence/read receipts/inbound send closures, and preserve pairing request upserts while suppressing WhatsApp replies. Then run the WhatsApp extension typecheck/tests before merge.
