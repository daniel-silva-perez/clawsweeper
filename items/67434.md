## Summary

Keep open. Current main addresses part of the operator confusion: the docs now describe strict-agentic auto-exposure, `tools.experimental.planTool=false` overrides it, and `update_plan` now returns compact empty content. But current source and tests do not prove a fix for the reported runtime bug: hundreds of consecutive `update_plan`-only calls before the next real tool call.

## Best Possible Solution

Keep this issue open as the focused tracker for excessive consecutive `update_plan` bursts. The likely fix belongs in the embedded Pi runtime/tool-loop layer: treat pure `update_plan` tool-only activity as non-progress even without visible assistant prose, cap consecutive plan-tool-only calls or turns before retry/blocking, and add regression coverage for many consecutive `update_plan` calls before a real tool call. Keep the current docs and `planTool: false` mitigation because they address the operator confusion but not the runtime protection gap.
