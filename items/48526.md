## Summary

Keep open. Current `main` still preserves provider billing-disable state across onboarding/re-auth in the onboarding path, and same-provider Anthropic fallback candidates are still skipped during a billing disable window. There is a newer manual recovery path via `openclaw models auth login --provider <provider>`, but that is not what `openclaw onboard` uses, so the reported bug remains valid on `4013c658537ee2051c44186f197b0710e84c9049`.

## Evidence

- **Onboarding still uses provider auth flow without clearing stale lockouts:** Interactive onboarding calls `applyAuthChoice(...)`, which routes into `runProviderPluginAuthMethod(...)`. That function executes the provider auth method directly and does not clear `usageStats` / billing-disable state first.
  - file: [src/wizard/setup.ts:561](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/src/wizard/setup.ts#L561)

  - file: [src/agents/auth-profiles/profiles.ts:60](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/src/agents/auth-profiles/profiles.ts#L60)
