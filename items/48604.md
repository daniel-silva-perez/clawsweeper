## Summary

Keep open. Current `main` does not implement per-agent `contextTokens` overrides: `agents.list[]` cannot declare the field, the context-window guard only reads `agents.defaults.contextTokens`, and the embedded runner compaction/extension budget paths still have no agent-specific context-window input.

## Evidence

- **Agent schema lacks `agents.list[].contextTokens`:** `AgentEntrySchema` on current `main` defines per-agent fields like `model`, `skills`, `contextLimits`, and `runtime`, but there is no `contextTokens` property, so the requested config key is not accepted for agent entries.
  - file: [src/config/zod-schema.agent-runtime.ts:812](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/src/config/zod-schema.agent-runtime.ts#L812)
  - command: `nl -ba src/config/zod-schema.agent-runtime.ts | sed -n '812,930p'`
  - sha: [4013c658537e](https://github.com/openclaw/openclaw/commit/4013c658537ee2051c44186f197b0710e84c9049)
- **Context-window resolution only applies the global default cap:** `resolveContextWindowInfo` takes `cfg`, `provider`, `modelId`, model context metadata, and `defaultTokens`, then applies only `cfg?.agents?.defaults?.contextTokens`. There is no `agentId` parameter or lookup into `agents.list`.
  - file: [src/agents/context-window-guard.ts:23](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/src/agents/context-window-guard.ts#L23)

- **Current tests cover only `agents.defaults.contextTokens`:** The existing guard tests verify the global `agents.defaults.contextTokens` cap and the non-override case, but there is no coverage for per-agent `agents.list[].contextTokens` behavior on `main`.
  - file: [src/agents/context-window-guard.test.ts:150](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/src/agents/context-window-guard.test.ts#L150)

## Risks / Open Questions

- If this PR were closed as obsolete, OpenClaw would still lack any supported way to set different runtime context caps per agent when multiple agents share provider/model config but need different effective windows.
- The current runner layout suggests an implementation must update more than one call site; otherwise per-agent caps would be applied inconsistently between normal runs and compaction/pruning paths.
