## Summary

Keep PR #63981 open. Current main still awaits gateway startup directly with no startup deadline, and the launchd service is still RunAtLoad/KeepAlive, so the reported sleep/wake startup hang is not implemented away. The PR is a plausible focused fix with relevant tests, but it likely needs revision for the latest review concern around late cleanup close failures.

## Best Possible Solution

Keep the PR open and rebase or replace it with an equivalent fix on current main: add a bounded startup deadline in `src/cli/gateway-cli/run-loop.ts`, preserve first-start propagation so supervised installs restart, handle non-initial timeout cleanup without silently leaking late servers, and add focused `run-loop.test.ts` coverage for first-start timeout plus late-resolving startup cleanup.

## Risks / Open Questions

- A naive startup `Promise.race` can leave `params.start()` running after timeout; the final implementation should either cancel startup or make late-resolution cleanup observable enough to avoid leaked listeners.
- The chosen deadline and cleanup behavior should preserve the existing first-start non-zero exit path while keeping in-process restart failures manageable and diagnosable.
