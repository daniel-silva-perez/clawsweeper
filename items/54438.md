## Summary

Keep open. Current `main` does not implement a cron `exec` payload or model-free scheduled shell execution. Cron still only supports `systemEvent` and `agentTurn`, and isolated runs go through the normal model-backed agent execution path.

## Evidence

- **Cron payload type only has two variants:** `CronPayload` is defined as `systemEvent | agentTurn`; there is no `exec` payload kind in the core type contract.
  - file: [src/cron/types.ts:112](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/cron/types.ts#L112)
  - command: `nl -ba src/cron/types.ts | sed -n '100,155p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Gateway protocol schema rejects any third payload kind:** `CronPayloadSchema` accepts only `{ kind: "systemEvent" }` and `{ kind: "agentTurn" }`, so `cron.add`/`cron.update` have no wire-format support for `exec`.
  - file: [src/gateway/protocol/schema/cron.ts:130](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/gateway/protocol/schema/cron.ts#L130)
  - command: `nl -ba src/gateway/protocol/schema/cron.ts | sed -n '100,145p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Job validation hard-codes systemEvent vs agentTurn:** `assertSupportedJobSpec(...)` throws unless `main` jobs use `systemEvent` and isolated/current/custom-session jobs use `agentTurn`.
  - file: [src/cron/service/jobs.ts:156](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/cron/service/jobs.ts#L156)
  - command: `nl -ba src/cron/service/jobs.ts | sed -n '150,230p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Isolated cron execution still runs through model-backed agent code:** The isolated cron executor wraps work in `runWithModelFallback(...)` and then calls `runCliAgent(...)` or `runEmbeddedPiAgent(...)`; there is no direct shell-command execution branch.
  - file: [src/cron/isolated-agent/run-executor.ts:109](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/cron/isolated-agent/run-executor.ts#L109)
  - command: `nl -ba src/cron/isolated-agent/run-executor.ts | sed -n '1,220p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **User docs still describe only system-event and agent-turn cron styles:** Docs say main-session jobs enqueue a system event and isolated jobs run a dedicated agent turn; payload options are prompt/model/tool controls, not raw command execution.
  - file: [docs/automation/cron-jobs.md:89](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/docs/automation/cron-jobs.md#L89)
  - command: `nl -ba docs/automation/cron-jobs.md | sed -n '86,110p'`

## Risks / Open Questions

- There are partial workarounds such as external OS cron plus hooks/webhooks, but those are not the requested first-class `payload.kind="exec"` feature inside OpenClaw cron.
- If maintainers consider scheduled host exec out of scope for security/product reasons, that scope decision is not documented in the issue thread or encoded as an existing close-worthy implementation on `main`.
