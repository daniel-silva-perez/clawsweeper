## Summary

This PR should stay open as an active implementation candidate for a real webchat Enter-to-send bug, but it needs maintainer review rather than cleanup closure. Current main still has an Enter key path that can call onSend after only checking connection, while the PR proposes a busy guard and tests. The review also found a functional concern: current main intentionally supports queueing while a run is active, so the PR's stream-based isBusy guard may suppress existing queue behavior and its tests do not cover two Enter keydowns in the same render frame.

## Best Possible Solution

Keep this PR open for normal maintainer review. The likely product path is to fix Enter-to-send duplicate submission while preserving intentional queue behavior during active runs, with a regression test that covers the actual rapid double-Enter timing. If the current PR is revised accordingly and CI passes, landing it should close the linked webchat bug; otherwise request changes rather than closing as obsolete.
