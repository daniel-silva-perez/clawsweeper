## Summary

Keep open. Current main still does not make memory/embedding setup part of onboarding, so the feature request is not implemented. The current product does mitigate the original impact with auto-detected memory providers, FTS-only fallback, and doctor readiness checks, but those are adjacent improvements rather than the requested mandatory onboarding step.

## Evidence

- **Onboarding docs still list no memory step:** The current CLI onboarding guide enumerates only Model/Auth, Workspace, Gateway, Channels, Daemon, Health check, and Skills. There is no Memory/Embedding setup step.
  - file: [docs/start/wizard.md:63](https://github.com/openclaw/openclaw/blob/a59d1bd46d9fc0bd2d9194b3f17db2561efc51e3/docs/start/wizard.md#L63)

## Risks / Open Questions

- Docs could theoretically lag runtime behavior, but the onboarding entrypoint code search also found no memory/embedding integration.
- The original report's 'silently fails' claim is partly outdated on current main because memory has auto-detection and FTS-only fallback, so the remaining gap is product/onboarding UX rather than a confirmed hard failure in all setups.
- The macOS LaunchAgent environment-key propagation concern from comments was not reproduced locally in this read-only review.
