## Summary

Keep open. Current `main` at `45146913007da723b4f06aba6a66cf4a695974c0` still does not validate `gateway.nodes.denyCommands` during `openclaw config set`/`unset`. The schema accepts arbitrary strings, runtime enforcement only removes exact known command IDs, and the only current mitigation on `main` is a post-write `security audit` warning for ineffective deny entries.

## Evidence

- **`config set` has no deny-command validation hook:** `runConfigSet` clones the resolved config, applies writes, optionally does dry-run schema/resolvability checks, and then calls `replaceConfigFile(...)`. There is no call to a deny-command validator anywhere in this path.
  - file: [src/cli/config-cli.ts:1199](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/src/cli/config-cli.ts#L1199)
  - command: `nl -ba src/cli/config-cli.ts | sed -n '1170,1365p'`
  - sha: [45146913007d](https://github.com/openclaw/openclaw/commit/45146913007da723b4f06aba6a66cf4a695974c0)
- **`config unset` also writes without revalidating deny entries:** `runConfigUnset` removes the path and immediately calls `replaceConfigFile(...)`; it does not revalidate `gateway.nodes.denyCommands` after `gateway.nodes.allowCommands` or parent-object changes.
  - file: [src/cli/config-cli.ts:1393](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/src/cli/config-cli.ts#L1393)
  - command: `sed -n '1393,1495p' src/cli/config-cli.ts`
  - sha: [45146913007d](https://github.com/openclaw/openclaw/commit/45146913007da723b4f06aba6a66cf4a695974c0)
- **Schema still allows any string array for `denyCommands`:** The gateway config schema defines `gateway.nodes.denyCommands` only as `z.array(z.string()).optional()`. There is no enum/known-command refinement here, so typos like `sendd` still pass config validation.
  - file: [src/config/zod-schema.ts:869](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/src/config/zod-schema.ts#L869)
  - command: `nl -ba src/config/zod-schema.ts | sed -n '860,895p'`
  - sha: [45146913007d](https://github.com/openclaw/openclaw/commit/45146913007da723b4f06aba6a66cf4a695974c0)
- **Runtime policy ignores unknown deny entries:** `resolveNodeCommandAllowlist` builds the allowlist, then deletes each trimmed `denyCommands` entry by exact string match. Unknown entries simply do nothing at runtime, which is the bug this PR is trying to prevent.
  - file: [src/gateway/node-command-policy.ts:178](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/src/gateway/node-command-policy.ts#L178)
  - command: `nl -ba src/gateway/node-command-policy.ts | sed -n '55,195p'`
  - sha: [45146913007d](https://github.com/openclaw/openclaw/commit/45146913007da723b4f06aba6a66cf4a695974c0)
- **Current main only adds a post-write audit warning:** Main now contains `collectNodeDenyCommandPatternFindings`, which warns about ineffective `gateway.nodes.denyCommands` entries, and docs list `gateway.nodes.deny_commands_ineffective` as an audit finding. That is detection after the fact, not the pre-write validation this PR implements.
  - file: [src/security/audit-extra.sync.ts:856](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/src/security/audit-extra.sync.ts#L856)

## Risks / Open Questions

- Without this PR or an equivalent change, `openclaw config set gateway.nodes.denyCommands '["sendd"]'` can still be written successfully and remain ineffective until an operator later runs `openclaw security audit`.
- `config unset gateway.nodes.allowCommands` can still leave previously-valid custom deny entries ineffective because the unset path does not revalidate command names.
