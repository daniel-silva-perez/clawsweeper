## Summary

Keep #70050 open. Current main still runs load-time session-store maintenance synchronously when `session.maintenance.mode` resolves to `enforce` and the store exceeds `maxEntries`; the pruning loop and full sort are still on the `loadSessionStore` path. The later shipped/config-related work only makes load-time maintenance honor configured settings, not non-blocking or deferred, and tests still assert load-time prune/cap behavior.

## Best Possible Solution

Keep the issue open and fix the startup path in core: move oversized session-store pruning/capping out of synchronous `loadSessionStore` startup reads, or make it bounded/chunked/background after Gateway is responsive. Preserve an explicit debug opt-out through `session.maintenance.mode: "warn"`, update docs/schema help if needed, and add a regression test that an oversized store does not block startup/chat history readiness while maintenance completes safely.
