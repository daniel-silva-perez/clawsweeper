## Summary

Keep open. Current main still restarts the macOS LaunchAgent through `launchctl kickstart -k` rather than the requested bootout/wait/bootstrap two-phase ordering. The existing stale-port cleanup is a related mitigation, but the normal restart test explicitly asserts no bootout/bootstrap path, and PR #43961 is closed unmerged.

## Best Possible Solution

Keep this issue open until `src/daemon/launchd.ts` either implements a conservative two-phase macOS restart or maintainers explicitly choose and document an equivalent launchd-safe contract. A robust fix should capture the previous PID and identity, prevent launchd from starting the replacement before the old process/port is confirmed gone, then bootstrap/kickstart cleanly, while preserving the detached handoff path and adding ordering-focused tests.

## Evidence

- **Current restart path still uses kickstart-first ordering:** `restartLaunchAgent` performs optional stale-port cleanup, enables the service, then calls `launchctl kickstart -k`. `bootstrap` is only reached after that kickstart reports a not-loaded service, so current main does not implement the requested `bootout` -> wait-for-old-PID-gone -> `bootstrap + kickstart` sequence.

## Risks / Open Questions

- Changing this path is product-sensitive because `bootout` can leave a per-user LaunchAgent unloaded if bootstrap later fails, and the existing detached self-restart path depends on not killing the caller before the restart command returns.
