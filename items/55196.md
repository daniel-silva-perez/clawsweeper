## Summary

Keep this issue open. Current main still lacks a `brave.http` diagnostics flag or Brave-specific request/status/timing/cache diagnostics, while the existing diagnostics API and Telegram pattern make the request actionable inside the bundled Brave plugin without a new core API. The existing ClawSweeper comment reached the same conclusion on an earlier main SHA, and the current main SHA still has the same gap.

## Best Possible Solution

Keep this issue open and implement the feature in `extensions/brave`: pass full config or precomputed diagnostics state into the Brave runtime, gate logs on `brave.http` via `openclaw/plugin-sdk/diagnostic-runtime`, use a Brave subsystem diagnostics logger, log sanitized outbound URL/query/params plus response status and duration, log cache hit/miss around the existing cache path, add focused Brave provider tests, and document the flag in `docs/diagnostics/flags.md` and `docs/tools/brave-search.md`. No new core API appears necessary.
