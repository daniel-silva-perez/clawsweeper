## Summary

Keep open. Current main still lacks the requested `brave.http` diagnostics flag: Brave search builds outbound Brave API requests and uses the shared cache, but there is no Brave-specific diagnostics gate or logging analogous to Telegram’s `telegram.http` path. The existing plugin SDK already exposes diagnostic flag helpers, so the best path is a small Brave plugin change rather than closing as external/ClawHub work.

## Best Possible Solution

Keep #55196 open and implement it inside the bundled Brave plugin. Add a `brave.http` diagnostics gate using `openclaw/plugin-sdk/diagnostic-runtime`, log redacted outbound URL/query/params, response status and elapsed time around `withTrustedWebSearchEndpoint`, and cache hit/miss around `readCachedSearchPayload`/`writeCachedSearchPayload`; then add focused Brave provider tests and update `docs/diagnostics/flags` or `docs/tools/brave-search` to document the new flag.
