## Summary

Keep this PR open. Current main still reads only one resolved transcript file for `chat.history`, so the underlying gateway history truncation across compaction checkpoints is not implemented on main. The PR is a plausible core fix candidate with production-derived evidence, but it should not be closed or merged as-is because the latest review context still identifies unresolved correctness and path-containment issues.

## Best Possible Solution

Keep this PR open for maintainer review and request changes rather than close it. The best path is a chain-aware `readSessionMessages` implementation in `src/gateway/session-utils.fs.ts` that uses stored checkpoint references, constrains resolved transcript paths to the session store root, avoids double-counting across `firstKeptEntryId` boundaries, defines reset/restore epoch behavior, and adds regression tests for the main-session chain plus reset/restore and path-containment cases. Coordinate the fallback path with #60409 instead of treating that PR as a replacement.

## Risks / Open Questions

- The PR’s latest reviewed approach reportedly trusts `compactionCheckpoints[*].preCompaction.sessionFile` and `entry.sessionFile` after `path.resolve` and existence checks; without a store-root containment check, corrupted or stale session metadata could cause `chat.history` to ingest unrelated files.
- A full-file concatenation strategy can duplicate retained tail entries because compaction records `firstKeptEntryId` and successor transcripts preserve unsummarized post-boundary content.
- Reset and restore currently preserve checkpoint arrays, so chain replay needs explicit epoch semantics before it can be safe for `/reset` and `sessions.compaction.restore` flows.
- The PR stacks near #60409’s reset-archive fallback work, so the eventual dispatcher should keep the two paths additive without replaying reset archives into normal compaction-chain history.
