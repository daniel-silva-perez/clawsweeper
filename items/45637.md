## Summary

Keep open. Current `main` does not implement a generic `models.providers.*.timeoutMs` override, so the requested feature is still missing. The issue’s specific `hardcoded 30s` framing looks obsolete on current `main`, but there is still a real provider-scoped timeout gap.

## Evidence

- **Model provider config type has no timeout field:** `ModelProviderConfig` includes `baseUrl`, `apiKey`, `auth`, `api`, `injectNumCtxForOpenAICompat`, `headers`, `authHeader`, `request`, and `models`, with no `timeoutMs` entry.

## Risks / Open Questions

- The report’s original reproduction points at an outdated timeout explanation, so maintainers may want to retitle or reframe the issue around the still-open provider-scoped timeout feature gap.
- There is still an unresolved design question between a provider-level overall run budget and a true transport-only HTTP timeout; current code uses the resolved run timeout to tune the embedded attempt HTTP runtime.
