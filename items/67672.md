## Summary

Keep open. Current main still does not contain the PR's explicit awaited Slack SocketModeClient disconnect, and the reported leak path remains plausible during stale-socket health-monitor restarts. The supplied PR discussion shows the earlier risky `stopAccount` hook and redundant cast feedback were addressed, leaving a narrow Slack-owned cleanup change that should get maintainer review rather than cleanup closure.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to land a narrow Slack-owned cleanup that awaits the underlying SocketModeClient disconnect before `app.stop()`, preserves the removal of the unnecessary `stopAccount` hook, and adds a small regression test for disconnect-before-stop/awaited cleanup so stale-socket restart cycles cannot accumulate old Slack WebSocket clients.

## Risks / Open Questions

- The PR still lacks a focused regression test proving the underlying SocketModeClient `disconnect()` is called and awaited before `app.stop()` completes; maintainers should request or add one before landing if feasible.
- The production leak was not live-reproduced in this read-only review, so the strongest proof is the author's production evidence plus the current code and locked dependency semantics.
- The proposed helper relies on the existing private `app.receiver.client` shape. Current Slack code already relies on that shape for shutdown state and socket event waiting, and the locked Bolt version exposes it, but it remains a dependency-coupled seam.
