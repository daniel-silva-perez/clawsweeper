## Summary

Keep open. Current main still has the reported cleanup gap: the Feishu upload helpers accept string paths, create fs.ReadStream objects, and call the Feishu SDK without a try/finally destroy path if the SDK call rejects. The high-level sendMediaFeishu path currently passes Buffers, which lowers the common blast radius, but it does not make the lower-level string-path helper behavior implemented or superseded. Existing tests do not cover this failure cleanup path, and the PR discussion supports the root cause while noting only a style issue that the PR author later addressed.

## Best Possible Solution

Keep the PR open and either land a cleaned-up variant or ask the contributor to add regression tests. The best code path is to make uploadImageFeishu and uploadFileFeishu clean up only streams they create from string paths on both success and failure, while preserving Buffer behavior. If string-path inputs are no longer an intended internal contract, remove that support explicitly instead of leaving an untested partial branch.
