## Summary

Keep open. Current main has overlapping cron timeout and task-ledger recovery hardening, but it does not implement this PR's distinct `runTimeoutSeconds` cron payload/API or the unresponsive-runner wall-clock test path. The PR also has a concrete production repro and supportive related discussion, so it remains a plausible focused reliability change that needs maintainer review and rebase rather than automated cleanup closure.

## Best Possible Solution

Keep this PR open for maintainer review. The next best step is to rebase it onto current main, preserve existing timeout-start and durable-recovery behavior, then decide the product/API contract for a separate `runTimeoutSeconds` field. If accepted, update types, normalization, docs/schema as needed, and add regression coverage for both terminal task-ledger closure and an isolated runner that ignores the inner abort.
