## Summary

Current main already defers pending delivery recovery until after startup sidecars run, and channel startup is part of those sidecars. That satisfies the issue’s concrete fallback request to start channels before delivery recovery, even though startup order is not exposed as a general user-configurable setting.

## Close Comment

Closing from a Codex review: current OpenClaw already sequences startup so channel startup happens before pending delivery recovery. In `src/gateway/server-startup-post-attach.ts:250`, startup sidecars run `startChannels()`, and in `src/gateway/server-runtime-services.ts:97` plus `src/gateway/server.impl.ts:849`, heartbeat/cron/delivery recovery are only activated after post-attach sidecars finish. There is also coverage in `src/gateway/server-runtime-services.test.ts:64` for this ordering. The same sequencing is present in the shipped `v2026.4.22` release, so the concrete startup race described here appears to already be addressed on current main/release. The broader idea of fully user-configurable startup ordering is not implemented, but the issue body’s minimum requested behavior is already in place.
