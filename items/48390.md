## Summary

Current `main` appears to have addressed the specific stale-socket heuristic described in the report, but it still stops WhatsApp monitoring on 440 conflicts and still has generic auto-restart paths that can restart a conflict-stopped channel. The evidence is not strong enough to conclude that #48390 is fully fixed on `main`, so this should stay open.

## Evidence

- **440 conflict is still terminal inside the WhatsApp monitor:** `isNonRetryableWebCloseStatus()` still treats status 440 as non-retryable, and the monitor logs a session-conflict error, shuts down, and breaks out of the loop. The regression test also still expects 440 to stop without retrying.
  - file: [extensions/whatsapp/src/auto-reply/monitor.ts:40](https://github.com/openclaw/openclaw/blob/b453293349657dedd069d291fa5b078d910067b1/extensions/whatsapp/src/auto-reply/monitor.ts#L40)
  - command: `nl -ba extensions/whatsapp/src/auto-reply/monitor.ts | sed -n '1,560p'`
  - sha: [b45329334965](https://github.com/openclaw/openclaw/commit/b453293349657dedd069d291fa5b078d910067b1)
- **The exact stale-socket heuristic from the report was changed:** Gateway health now bases stale-socket recovery on `lastTransportActivityAt`, not ordinary app-event freshness, and explicitly skips connected channels that do not report transport liveness. That means the March 2026 "30 minutes of no inbound events => stale-socket restart" path is no longer the same on current `main`.
  - file: [src/gateway/channel-health-policy.ts:117](https://github.com/openclaw/openclaw/blob/b453293349657dedd069d291fa5b078d910067b1/src/gateway/channel-health-policy.ts#L117)
  - command: `nl -ba src/gateway/channel-health-policy.ts | sed -n '1,260p'`
  - sha: [b45329334965](https://github.com/openclaw/openclaw/commit/b453293349657dedd069d291fa5b078d910067b1)
- **Exited channel tasks are still auto-restarted generically:** When a channel task exits, `server-channels` marks it stopped and, unless it was manually stopped, schedules automatic restart attempts with backoff up to `MAX_RESTART_ATTEMPTS`. There is no WhatsApp-conflict-specific opt-out in this path.
  - file: [src/gateway/server-channels.ts:437](https://github.com/openclaw/openclaw/blob/b453293349657dedd069d291fa5b078d910067b1/src/gateway/server-channels.ts#L437)
  - command: `nl -ba src/gateway/server-channels.ts | sed -n '420,620p'`
  - sha: [b45329334965](https://github.com/openclaw/openclaw/commit/b453293349657dedd069d291fa5b078d910067b1)
- **Health monitor still restarts stopped channels and resets restart attempts:** `evaluateChannelHealth()` still marks `running: false` as unhealthy, and `channel-health-monitor` responds by stopping, resetting restart attempts, and starting the channel again. There is no special handling for WhatsApp `healthState: "conflict"` here, so a persistent 440 conflict can still be re-armed by generic restart logic.
  - file: [src/gateway/channel-health-monitor.ts:160](https://github.com/openclaw/openclaw/blob/b453293349657dedd069d291fa5b078d910067b1/src/gateway/channel-health-monitor.ts#L160)
