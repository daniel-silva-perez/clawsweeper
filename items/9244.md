## Close Comment

Closing after Codex review. This proposal fits OpenClaw's plugin/provider ecosystem better than OpenClaw core. Current main's scope docs say core stays lean and optional capability should usually ship as plugins, with new skills published to ClawHub first (`VISION.md:54-70`). The plugin docs explicitly route 'LLM, proxy, or custom endpoint' work to provider plugins (`docs/plugins/building-plugins.md:16-19,33-35`), and the provider-plugin guide says proxy/custom-LLM integrations belong there, with native daemon/thread/tool-event behavior handled by an agent harness instead of core (`docs/plugins/sdk-provider-plugins.md:6-8,21-25`). OpenClaw already supports external gateway patterns like LiteLLM for cost tracking and model routing (`docs/providers/litellm.md:5-18`) and ships gateway-style provider integrations such as `cloudflare-ai-gateway` and `vercel-ai-gateway` (`docs/tools/plugin.md:98-106`). Given that documented boundary, this is better pursued as a ClawHub/community provider plugin or external gateway project rather than a core feature request.
