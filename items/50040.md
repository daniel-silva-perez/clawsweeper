## Summary

Keep open. Current `main` has meaningful Telegram mitigations for outbound send failures, but it does not implement the issue’s core runtime-recovery gap: pending outbound deliveries are only recovered on gateway startup, and the Telegram polling stall/restart path does not trigger a reconnect drain or continuous recovery worker.

## Evidence

  - file: [src/gateway/server-runtime-services.ts:55](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/src/gateway/server-runtime-services.ts#L55)

## Risks / Open Questions

- Ambiguous non-idempotent `sendMessage` failures during Telegram recovery windows can still leave replies truncated or stuck until a full gateway restart, because Telegram has no reconnect-time pending-delivery drain.
- Closing this as fixed would overstate what `main` currently implements: there are send-layer mitigations, but not the continuous/runtime outbound recovery described in the report.
