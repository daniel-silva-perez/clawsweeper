## Summary

Keep open. Current main still only partially addresses the report: OpenAI Chat Completions image budgets and `agents.defaults.imageMaxDimensionPx` exist, but there is still no shared configurable image limit contract for sanitizer bytes, sharp input pixels, generic image media caps, sandbox/media-store staging, gateway WebSocket attachments, or ACP attachment reads.

## Best Possible Solution

Keep this issue open as the canonical broad tracking item. The best implementation is a backwards-compatible, documented image limit contract that threads configured byte, input-pixel, dimension, and per-request total limits through the agent sanitizer, media/image-ops, generic inbound media, media-store/sandbox staging, gateway WebSocket attachments, ACP attachments, and OpenAI-compatible HTTP paths, while preserving existing narrower knobs as aliases or scoped overrides. Review #40950 only as the narrower sandbox/media-store piece.

## Risks / Open Questions

- Closing now would leave several user-visible image size limits unsupported by config despite the documented partial knobs.
- The eventual fix needs resource and abuse bounds because raising image byte and pixel limits affects memory, disk, network, and provider request costs.
- The schema design needs to reconcile existing partial settings such as `agents.defaults.imageMaxDimensionPx`, `tools.media.image.maxBytes`, and `gateway.http.endpoints.chatCompletions.images.*` without breaking current configs.
