## Summary

Keep open. Current main only partially addresses the report: `agents.defaults.imageMaxDimensionPx` is documented/configurable and OpenAI Chat Completions now has image byte budget config, but the broader requested image limit contract is still missing across tool-result sanitization, sharp input pixel handling, generic inbound media, sandbox staging, and Control UI/WebSocket attachment paths.

## Best Possible Solution

Keep this issue open as the canonical broad tracking item. The best fix is a backwards-compatible, documented image limit contract that threads through the agent tool-image sanitizer, media/image-ops input pixel guard, generic inbound media limits, sandbox/media-store staging, and gateway attachment/OpenAI-compatible paths, while preserving existing knobs such as `agents.defaults.imageMaxDimensionPx` and the current gateway Chat Completions image settings as aliases or scoped overrides where appropriate. #40950 can still be reviewed as the narrower sandbox piece.

## Risks / Open Questions

- This is a real cross-layer config gap, not just stale discussion; closing now would leave operators with no supported way to raise several current hardcoded limits.
- The desired schema needs product/API design because there are existing narrower knobs (`agents.defaults.imageMaxDimensionPx`, `gateway.http.endpoints.chatCompletions.images.maxBytes`, `tools.media.image.maxBytes`) that should be reconciled without breaking existing configs.
- Raising image limits globally has resource and abuse implications, so defaults and maximum allowed values should be bounded and documented.
