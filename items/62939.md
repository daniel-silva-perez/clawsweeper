## Summary

Keep open. Current main has meaningful partial prompt-injection hardening for source-specific external content, web fetch/search results, inbound metadata/history, and owner-based tool policy, but it still does not implement the central request: shared trust-boundary marking for non-owner current user messages and generic tool results, plus a stable system-prompt anchor for those wrappers. The linked implementation PR #62973 was closed unmerged.

## Best Possible Solution

Keep this issue open and implement a core-owned, tested trust-boundary strategy: add stable system-prompt guidance before the cache boundary, wrap or otherwise label non-owner current message bodies at prompt assembly, classify and wrap open-world tool results at the shared transport/provider boundary, preserve existing external-content marker sanitization, and add regression tests across owner/non-owner ingress and major provider payload shapes.

## Risks / Open Questions

- Security-relevant behavior remains partially covered by source-specific wrappers and policy gates rather than a shared trust-boundary contract for all current non-owner messages and open-world tool results.
- A future fix must preserve the existing `EXTERNAL_UNTRUSTED_CONTENT` marker sanitizer and special-token stripping instead of replacing it with weaker XML-only wrapping.
- A transport-level wrapper needs careful tests for provider payload shape, double-wrapping, prompt-cache stability, owner vs non-owner behavior, and web_fetch/web_search/x_search/external MCP paths.
