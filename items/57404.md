## Summary

Keep open. Current `main` still does not emit a per-run lifecycle usage event on the WebSocket stream. The lifecycle contract remains `start|end|error`, the terminal lifecycle path fires before session usage is persisted in the primary agent flow, and the WS session snapshots only expose aggregate session fields rather than the requested per-run `usage`/`lastCallUsage` payload.

## Evidence

- **Lifecycle contract still excludes a usage phase:** The current agent-loop docs define lifecycle events only as `phase: "start" | "end" | "error"`, and `agent.wait` still waits only for end/error.
  - file: [docs/concepts/agent-loop.md:41](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/docs/concepts/agent-loop.md#L41)
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
- **Gateway lifecycle parser only recognizes start/end/error:** `src/gateway/session-lifecycle-state.ts` hard-codes `type LifecyclePhase = "start" | "end" | "error"` and rejects any other phase in `resolveLifecyclePhase()`.
  - file: [src/gateway/session-lifecycle-state.ts:6](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/src/gateway/session-lifecycle-state.ts#L6)
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
- **Primary agent path emits lifecycle end before persisting run usage:** `agent-command.ts` emits the terminal lifecycle event at lines 943-953, then only afterward updates the session store with token/model fields at lines 1048-1063. That means the lifecycle event itself still does not carry the per-run usage payload, and the end-of-run WS path is not sourced from freshly persisted usage.
  - file: [src/agents/agent-command.ts:943](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/src/agents/agent-command.ts#L943)
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
- **WS lifecycle/session snapshots omit the requested per-run fields:** The session snapshot builder used for WS broadcasts includes aggregate session fields like `inputTokens`, `outputTokens`, `totalTokens`, `estimatedCostUsd`, `modelProvider`, and `model`, but not per-run `usage`, `lastCallUsage`, `cacheRead`, or `cacheWrite` payloads on lifecycle events.
  - file: [src/gateway/server-chat.ts:553](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/src/gateway/server-chat.ts#L553)
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
- **Closest existing WS surface is transcript/session metadata, not lifecycle usage:** Tests show `session.message` / `sessions.changed` transcript events can include live aggregate usage metadata, but that is a different surface from the requested lifecycle usage event and does not prove `lastCallUsage` or a dedicated lifecycle usage payload.
  - file: [src/gateway/session-message-events.test.ts:292](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/src/gateway/session-message-events.test.ts#L292)
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)

## Risks / Open Questions

- There is partial overlap from `session.message` / `sessions.changed` transcript updates carrying aggregate totals, so some downstream observers may already get limited usage data. That does not implement the requested per-run lifecycle usage event shape.
- `gh` inspection was unavailable in this checkout because `GH_TOKEN` was not set, so this review relies on current `main` code/docs and the provided GitHub context rather than PR metadata.
