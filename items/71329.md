## Summary

Keep #71329 open. Current main still gates Slack lifecycle status reactions through the ack-reaction eligibility path, so `messages.statusReactions.enabled: true` does not independently activate the controller when `ackReaction` is absent or the ack scope blocks it. The PR is a focused Slack plugin fix, with one review-noted test assertion gap to address before merge.

## Best Possible Solution

Keep the PR open and either merge or adapt its focused Slack prepare change: treat `messages.statusReactions.enabled === true` as an independent opt-in, preserve `enabled: false` and the existing ack-scope compatibility path, and add regression tests for explicit enable without `ackReaction` plus explicit disable with the plain ack reaction still firing.
