## Summary

Keep this PR open. Current main still requires the Slack ack-reaction gate before lifecycle status reactions can start, so `messages.statusReactions.enabled: true` does not independently activate the existing controller. The PR is a focused implementation candidate, with one small test assertion gap to address before merge and no security-sensitive changes visible in the provided diff.

## Best Possible Solution

Keep this PR open and review or adapt its narrow Slack change: let `messages.statusReactions.enabled === true` activate lifecycle reactions independently of the ack-reaction gate, preserve `enabled: false` as an explicit disable, keep the existing plain ack-reaction fallback behavior, and add the missing assertion in the explicit-disable test before merge.
