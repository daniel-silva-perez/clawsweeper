## Summary

Current `main` does not appear to implement #6662. Subagent completion announces are still injected into the parent session as inter-session input on a `role: "user"` message path, and the Control UI only renders Tool Output styling for explicit tool roles/fields (`tool`, `toolResult`, `toolCallId`, `toolName`, or tool content blocks). I did not find a provenance-based remap for `sourceTool: "subagent_announce"`, so the reported UX confusion remains plausible on current `main`.

## Evidence

- **Subagent announce still enters the parent session as inter-session input:** `deliverSubagentAnnouncement` calls the gateway `agent` method with `inputProvenance.kind = "inter_session"` and `sourceTool = "subagent_announce"`, which is the same wake/injection path the issue describes.
  - file: `src/agents/subagent-announce-delivery.ts:547`
  - command: `nl -ba src/agents/subagent-announce-delivery.ts | sed -n '540,556p'`
  - sha: `178a314a4c1cfa98ac0e8545707162427e38f9c5`
- **Provenance is only attached to user-role messages:** `applyInputProvenanceToUserMessage` explicitly returns early unless `message.role === "user"`, then adds `provenance` without changing the role. That means an inter-session subagent wake remains a user message in transcript/history storage.
  - file: `src/sessions/input-provenance.ts:43`
  - command: `nl -ba src/sessions/input-provenance.ts | sed -n '43,60p'`
  - sha: `178a314a4c1cfa98ac0e8545707162427e38f9c5`
- **chat.history does not remap provenance-based wakes into tool results:** `sanitizeChatHistoryMessage` treats a message as tool-shaped only when it already has a tool role or tool metadata (`toolName`, `toolCallId`, etc.). There is no provenance-based branch for `inter_session` or `subagent_announce`.
  - file: `src/gateway/server-methods/chat.ts:1045`
  - command: `nl -ba src/gateway/server-methods/chat.ts | sed -n '1045,1068p'`
  - sha: `178a314a4c1cfa98ac0e8545707162427e38f9c5`
- **Control UI normalization still leaves plain user messages as user messages:** `normalizeMessage` upgrades messages to `toolResult` only when tool ids/names/content blocks are present; it does not inspect `provenance`. `normalizeRoleForGrouping` also preserves `user` as `user`.
