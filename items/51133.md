## Summary

Keep open. The issue is recent, externally authored, unprotected, and has a concrete Feishu/MiniMax reproduction plus a follow-up comment identifying `sessions_list` current-session semantics as the suspected loop trigger. Current main has relevant session-tool visibility and opt-in loop detection, but I did not find a shipped default fix that would prevent repeated successful `message` sends paired with `sessions_list` results, nor a targeted regression test for this Feishu/MiniMax current-session loop.

## Best Possible Solution

Keep this open and fix it with a targeted regression around Feishu-style inbound sessions using MiniMax/tool calls. The best product/code path is to make `sessions_list` unambiguous for the current session, for example by always surfacing the current session or explicitly marking that an empty list does not mean the active session was lost, and to add a default-safe guard for repeated side-effecting `message` sends in the same run when paired with no-progress session-list results. Cover the fix in `src/agents/tools/sessions-list-tool.test.ts` or the nearest embedded-run/message-tool test, plus a Feishu session-key fixture if the channel route is part of the failure.
