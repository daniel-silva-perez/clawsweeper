## Summary

Keep open. Current main does not implement the proposed command-lane circuit breaker or the prerequisite maxExecutionMs timeout, and the PR still represents plausible core queue hardening. It is not merge-ready as-is because review comments and the patch show correctness blockers around uncancelled timed-out tasks and an advertised but unimplemented priority option.

## Best Possible Solution

Keep this PR open for maintainer review, but require a revision before merge: land or fold the timeout prerequisite with cancellable/abort-aware semantics, remove or implement the priority option, keep the circuit-breaker API focused on depth/wait thresholds, and add regression tests for depth, wait-time, retryAfterMs, default main-lane behavior, and no overlap after timeout.

## Risks / Open Questions

- Merging the PR as-is could violate lane serialization because the timeout path rejects and frees the lane slot without cancelling the underlying task, allowing subsequent work to overlap with still-running command work.
- The PR advertises priority?: number as queue ordering but does not store or apply it, creating a misleading public contract on enqueueCommandInLane.
- The default main-lane breaker thresholds need maintainer review against current session/global lane behavior so legitimate bursts are not rejected unexpectedly.
