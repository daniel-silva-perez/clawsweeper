## Summary

Keep #71449 open. Current main still lacks the PR's central behavior: the gateway startup log still says generic `ready (...)`, there is no `sidecars settled` channel-health summary, Telegram polling still awaits/retries webhook cleanup before polling, and Feishu startup probing still lacks the PR's shorter explicit probe timeout/transient socket handling. Related #70728 remains the broader open issue, but it has not shipped a superseding implementation.

## Best Possible Solution

Keep #71449 open for maintainer review. The best path is to preserve the legacy `ready (` marker while clarifying it means the gateway is listening, add a deterministic `sidecars settled` summary from the existing channel runtime snapshot/health policy, and review the Telegram/Feishu startup hardening as channel-owned work tied to the broader #70728 startup UX issue. If maintainers dislike the breadth, split the PR rather than auto-closing it.
