## Summary

Keep open. Current `main` still does not implement BlueBubbles auto-TTS voice-message delivery end to end: the shared TTS layer does not treat `bluebubbles` as a native voice-note channel, and both active BlueBubbles outbound media paths call `sendBlueBubblesMedia(...)` without forwarding `audioAsVoice`/`asVoice`, even though the lower attachment layer still supports voice memos.

## Evidence

- **Shared TTS layer excludes BlueBubbles from native voice-note routing:** `supportsNativeVoiceNoteTts()` only includes `telegram`, `feishu`, `whatsapp`, `matrix`, and `discord`. The auto-TTS payload sets `audioAsVoice` only when that helper returns true, so BlueBubbles auto-TTS does not get marked as voice output on current `main`.
  - file: [extensions/speech-core/src/tts.ts:594](https://github.com/openclaw/openclaw/blob/a59d1bd46d9fc0bd2d9194b3f17db2561efc51e3/extensions/speech-core/src/tts.ts#L594)
  - command: `nl -ba extensions/speech-core/src/tts.ts | sed -n '588,598p'; nl -ba extensions/speech-core/src/tts.ts | sed -n '1172,1180p'`
  - sha: [a59d1bd46d9f](https://github.com/openclaw/openclaw/commit/a59d1bd46d9fc0bd2d9194b3f17db2561efc51e3)
- **Core outbound delivery already forwards `audioAsVoice`:** Current generic outbound delivery passes `audioAsVoice` through `sendOverrides` and then into `handler.sendMedia(...)`, so the missing behavior is not a solved core close-out; the receiving channel still has to consume that flag.
  - file: [src/infra/outbound/deliver.ts:872](https://github.com/openclaw/openclaw/blob/a59d1bd46d9fc0bd2d9194b3f17db2561efc51e3/src/infra/outbound/deliver.ts#L872)
  - command: `nl -ba src/infra/outbound/deliver.ts | sed -n '850,900p'; nl -ba src/infra/outbound/deliver.ts | sed -n '959,977p'`
  - sha: [a59d1bd46d9f](https://github.com/openclaw/openclaw/commit/a59d1bd46d9fc0bd2d9194b3f17db2561efc51e3)
- **BlueBubbles plugin outbound media path ignores the forwarded voice flag:** The BlueBubbles channel plugin's `sendMedia` handler destructures `cfg`, `to`, `text`, `mediaUrl`, `accountId`, and `replyToId`, but does not read `ctx.audioAsVoice` and does not pass `asVoice` into `sendBlueBubblesMedia(...)`.
  - file: [extensions/bluebubbles/src/channel.ts:331](https://github.com/openclaw/openclaw/blob/a59d1bd46d9fc0bd2d9194b3f17db2561efc51e3/extensions/bluebubbles/src/channel.ts#L331)
  - command: `nl -ba extensions/bluebubbles/src/channel.ts | sed -n '302,352p'`
  - sha: [a59d1bd46d9f](https://github.com/openclaw/openclaw/commit/a59d1bd46d9fc0bd2d9194b3f17db2561efc51e3)
- **BlueBubbles monitor auto-reply media path also omits `asVoice`:** The live BlueBubbles monitor-processing reply path resolves media from the reply payload and calls `sendBlueBubblesMedia(...)` with `mediaUrl`, `caption`, `replyToId`, and `accountId`, but never forwards `payload.audioAsVoice` as `asVoice`. That is the path closest to the reported auto-TTS behavior.
  - file: [extensions/bluebubbles/src/monitor-processing.ts:1656](https://github.com/openclaw/openclaw/blob/a59d1bd46d9fc0bd2d9194b3f17db2561efc51e3/extensions/bluebubbles/src/monitor-processing.ts#L1656)

- **Lower BlueBubbles attachment layer still supports voice memos when `asVoice` is provided:** `sendBlueBubblesAttachment()` still has explicit voice-memo handling and sets `isAudioMessage=true` when `asVoice` is true. This confirms the capability exists at the bottom layer, but current `main` does not wire auto-TTS into it.
  - file: [extensions/bluebubbles/src/attachments.ts:170](https://github.com/openclaw/openclaw/blob/a59d1bd46d9fc0bd2d9194b3f17db2561efc51e3/extensions/bluebubbles/src/attachments.ts#L170)
