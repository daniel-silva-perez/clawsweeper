## Summary

Keep open. The report is fresh, concrete, unprotected, and matches a still-plausible current-main gap: cron systemEvent heartbeats can become runtime-only prompts with an empty submitted prompt, while the current Anthropic replay validator repairs tool/result ordering and consecutive user turns but does not ensure an assistant-tailed transcript is followed by a user turn before provider submission.

## Best Possible Solution

Keep #72226 open and fix it in the embedded runner/runtime-context path, not by closing as already implemented. The likely maintainer path is to make runtime-only cron systemEvent runs submit a non-empty user turn, or otherwise make Anthropic-bound provider payloads end with a user message, then add a regression test with an assistant-tailed session transcript plus a main-session cron systemEvent using the direct anthropic provider policy.
