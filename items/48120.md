## Summary

Current `main` already matches plugin HTTP routes case-insensitively at gateway dispatch, so the proposed LINE-specific `.toLowerCase()` route registration change is unnecessary and the reported path-case 404 does not reproduce from the current code.

## Close Comment

Thanks for the PR. Codex review against current `main` (`b4532933`) could not reproduce the reported case-only route mismatch. Plugin HTTP route dispatch already canonicalizes both registered routes and incoming request paths to lowercase in `src/gateway/security-path.ts:34` and `src/gateway/server/plugins-http/route-match.ts:15`, and the current tests explicitly cover mixed-case requests such as `/API/demo` matching a registered `/api/demo` route in `src/gateway/server/plugins-http.test.ts:259` and `src/gateway/server/plugins-http.test.ts:409`. The LINE monitor on `main` still registers `normalizePluginHttpPath(webhookPath, "/line/webhook")` directly in `extensions/line/src/monitor.ts:305`, so the proposed extra `.toLowerCase()` is not needed for the current gateway matcher. Closing this as a Codex review because the reported path-casing 404 does not reproduce on current `main`. If there is still a reproducible LINE webhook 404 on a current build, please open a fresh issue with the exact config, request path, and logs so it can be traced to the remaining cause.
