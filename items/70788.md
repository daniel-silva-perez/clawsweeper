## Summary

Keep open. Current main still uses the reported Startup-folder `.cmd` launcher behavior: `buildStartupLauncherScript()` emits `start "" /min cmd.exe /d /c ...`, and the existing tests assert that exact command. I found no `OpenClaw Gateway.vbs`, `buildStartupVbsLauncher`, or startup-path `wscript.exe //nologo` implementation on main. The related closed #57682 covered a different Windows restart/update path and does not supersede this Startup-folder login item issue.

## Best Possible Solution

Implement this in the Windows daemon fallback path rather than closing it: update `src/daemon/schtasks.ts` so the Startup-folder `.cmd` remains at the existing detector path but delegates to a generated silent `.vbs` launcher, write the `.vbs` next to the `.cmd` during fallback install, remove it during uninstall/repair cleanup, and update the existing startup fallback tests to assert the new files and command shape. Keep #57682 separate because it covered the update/restart handoff, not this login Startup-folder launcher.
