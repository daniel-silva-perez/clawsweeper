## Summary

Keep open. Current main has several restart-resilience pieces, but it does not implement this PR's core durable pending-inbound capture/replay store. The PR may be stale and merge-conflicted, but the remaining product problem is real and not fully solved by main's current command-queue drain, restart-continuation queue, or stale transcript-lock recovery.

## Best Possible Solution

Keep the PR open for maintainer judgment or contributor rework. The best path is to rebase onto current main and turn the durable inbound-loss problem into a narrow design that fits existing command-queue, plugin/channel inbound, session-delivery, and main-session recovery boundaries, with explicit untrusted replay semantics and regression tests for restart/drain message loss. If maintainers prefer the newer SQLite/message-store direction, link that canonical design and close only after that replacement is accepted or implemented.
