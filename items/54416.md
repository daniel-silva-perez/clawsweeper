## Summary

Keep this issue open. Current main still has documented and tested no-approval host exec behavior for gateway/node (`security="full"`, `ask="off"`), and I did not find a command-specific runtime guard that blocks model-issued `openclaw config set`, `openclaw models set`, or `openclaw gateway restart` shell commands. Prompt guidance and the guarded first-class gateway config tool are useful hardening, but they do not enforce the confirmation boundary for the exec path described in this issue.

## Best Possible Solution

Keep this open for a maintainer/security fix. Add a hard runtime guard at the agent exec boundary that blocks or requires explicit user approval for model-initiated OpenClaw self-management and config mutations such as `openclaw config set`, `openclaw models set`, `openclaw gateway restart`, update/uninstall equivalents, and related lifecycle commands unless tied to a user-approved config-change action. Add regression coverage for ambiguous model-switch prompts that may show example commands but must not execute them automatically, and coordinate the broader destructive-command policy with #65307.
