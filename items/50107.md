## Summary

Keep open. Current `main` has only an opt-in compatibility path for string-only OpenAI-compatible backends; it does not show a bundled NVIDIA-specific fix for the reported default `api: "openai-completions"` path.

## Evidence

- **Bundled NVIDIA catalog still lacks string-content compat:** The NVIDIA provider on current `main` still declares `api: "openai-completions"` and its built-in model entries do not set any `compat.requiresStringContent` override.
  - file: [extensions/nvidia/provider-catalog.ts:13](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/extensions/nvidia/provider-catalog.ts#L13)
  - sha: [45146913007d](https://github.com/openclaw/openclaw/commit/45146913007da723b4f06aba6a66cf4a695974c0)
- **OpenAI-completions flattening is gated by an opt-in flag:** `buildOpenAICompletionsParams()` only flattens message content when `compat.requiresStringContent` is truthy; otherwise it sends the converted message array unchanged.
  - file: [src/agents/openai-transport-stream.ts:1553](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/src/agents/openai-transport-stream.ts#L1553)
  - sha: [45146913007d](https://github.com/openclaw/openclaw/commit/45146913007da723b4f06aba6a66cf4a695974c0)
- **Embedded runner wrapper is also opt-in only:** The stream wrapper for OpenAI completions applies string-content flattening only when `model.compat.requiresStringContent === true`.
  - file: [src/agents/pi-embedded-runner/openai-stream-wrappers.ts:218](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/src/agents/pi-embedded-runner/openai-stream-wrappers.ts#L218)
  - sha: [45146913007d](https://github.com/openclaw/openclaw/commit/45146913007da723b4f06aba6a66cf4a695974c0)
- **Tests cover the workaround, not NVIDIA-by-default behavior:** Current tests verify flattening only for a model explicitly configured with `compat.requiresStringContent: true`, which supports the manual workaround but does not prove the NVIDIA provider is fixed out of the box.
  - file: [src/agents/openai-transport-stream.test.ts:1576](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/src/agents/openai-transport-stream.test.ts#L1576)
  - sha: [45146913007d](https://github.com/openclaw/openclaw/commit/45146913007da723b4f06aba6a66cf4a695974c0)
- **Current NVIDIA docs still present plain OpenAI-completions config as sufficient:** The NVIDIA docs show a config example with only `baseUrl` and `api: "openai-completions"`, and say OpenAI-compatible tooling should work out of the box. There is no NVIDIA-specific `requiresStringContent` guidance there.
  - file: [docs/providers/nvidia.md:39](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/docs/providers/nvidia.md#L39)
  - sha: [45146913007d](https://github.com/openclaw/openclaw/commit/45146913007da723b4f06aba6a66cf4a695974c0)
- **Repo-wide search found no NVIDIA-specific hookup for the compat flag:** A focused search for `requiresStringContent` found only the generic transport/wrapper/docs/test references, and none under `extensions/nvidia`, which is consistent with the bug still being unresolved for the bundled NVIDIA provider path.
  - command: `rg -n "requiresStringContent" extensions/nvidia src/agents src/plugins docs/providers/nvidia.md docs/help/troubleshooting.md docs/gateway/troubleshooting.md docs/gateway/local-models.md`
  - sha: [45146913007d](https://github.com/openclaw/openclaw/commit/45146913007da723b4f06aba6a66cf4a695974c0)

## Risks / Open Questions

- This review is based on read-only code/docs inspection; I did not run a live NVIDIA reproduction because the task forbids mutating the checkout and no provider credentials were available.
- There is a documented/manual workaround via `models.providers.<provider>.models[].compat.requiresStringContent: true`, but the issue reports the default bundled NVIDIA path, and that default still is not evidenced as fixed on `main`.
