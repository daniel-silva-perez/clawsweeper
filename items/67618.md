## Summary

Keep this issue open. Current main still contains the exact ambiguous TUI pairing hint reported by the user, and the matching TUI test only asserts that the message mentions `openclaw devices list`, not the actual approval command. Other surfaces have better pairing guidance, so this is a narrow UX cleanup rather than a duplicate or already-implemented core fix.

## Best Possible Solution

Update `src/tui/tui.ts` so `resolveGatewayDisconnectState` names `openclaw devices approve <requestId>` after `openclaw devices list`; parse the request ID from the close reason when present and otherwise show the placeholder. Tighten `src/tui/tui.test.ts` to assert both the listing command and the approval command.
