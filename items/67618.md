## Summary

Keep this issue open. Current main still emits the same ambiguous TUI pairing guidance reported in the issue, and the test only requires the message to mention `openclaw devices list`. The approval command exists and is documented elsewhere, and gateway close reasons can already carry a sanitized request ID, so this remains a narrow TUI copy/test cleanup rather than implemented work.

## Best Possible Solution

Update `src/tui/tui.ts` so `resolveGatewayDisconnectState` includes the concrete `openclaw devices approve <requestId>` command after `openclaw devices list`. Prefer reusing the existing pairing-message parser/sanitization from `src/gateway/protocol/connect-error-details.ts` when a request ID is present in the close reason, and fall back to the placeholder when it is not. Tighten `src/tui/tui.test.ts` to assert both the list command and approval command.
