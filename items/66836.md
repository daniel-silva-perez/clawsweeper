## Summary

Keep this PR open. Current main still lacks a configurable sandbox initialization timeout and cooperative startup cancellation, and this PR is a concrete implementation candidate rather than obsolete cleanup. The PR still needs maintainer review because the current diff leaves abort propagation incomplete and includes a sandbox browser security-epoch regression.

## Best Possible Solution

Keep this PR open as the active implementation candidate. The best maintainer path is to request a revised patch that adds `agents.defaults.sandbox.initTimeoutMs` to the public config types, strict schema, generated schema/help/docs as appropriate; wraps `resolveSandboxContext` in a bounded init timeout; makes startup cancellation effective through docker, ssh, openshell, prune, browser image/network/container setup, CDP polling, and Docker port reads; preserves the current browser security hash epoch; and keeps focused timeout/abort regression tests.

## Risks / Open Questions

- PR head makes `abortSignal` optional and the diff does not update `createDockerSandboxBackend` to pass it into `ensureSandboxContainer`, so backend startup can still continue after the timeout.
- PR head rolls `SANDBOX_BROWSER_SECURITY_HASH_EPOCH` back from the current security marker, which could let older sandbox browser containers be treated as current instead of forcing the intended refresh.
- The PR changes an OpenShell e2e availability guard in the same patch, which is not central to the sandbox timeout fix and should be checked for test reliability before landing.
