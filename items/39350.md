## Summary

Keep PR #39350 open. Current main still has the gateway final-message gap for a run that ends after buffering only a partial NO_REPLY lead fragment, but the PR is not safe to merge unchanged because its prefix guard would also drop a legitimate uppercase "NO" final reply.

## Best Possible Solution

Keep the PR open and ask for a revised gateway final-path fix. The best direction is to carry provenance that the buffered text came from a suppressed control-token lead fragment during streaming, then suppress only that final payload while preserving intentional uppercase "NO" replies. Add regression coverage for lifecycle end after only "NO"/"NO_" is buffered and for an intentional final uppercase "NO" that remains visible.
