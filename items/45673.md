## Summary

Keep open. Current `main` implements the basic precedence change for Feishu tools, but it does not include the PR's later review fixes that scope inherited account routing to Feishu-origin sessions and cover those cases in tests. The PR still contains unique behavior not present on `main`.

## Evidence

- **Current resolver still ignores channel origin:** `resolveFeishuToolAccount` only considers `executeParams.accountId`, then `defaultAccountId`, then configured `defaultAccount`. There is no `messageChannel` input or Feishu-origin guard, so any session whose `agentAccountId` happens to match a configured Feishu account will override `channels.feishu.defaultAccount` regardless of source channel.
  - file: [extensions/feishu/src/tool-account.ts:15](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/extensions/feishu/src/tool-account.ts#L15)
  - command: `nl -ba extensions/feishu/src/tool-account.ts | sed -n '1,70p'`
  - sha: [f03252aaf9cc](https://github.com/openclaw/openclaw/commit/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022)
- **Feishu tools still pass only agentAccountId into account resolution:** The Feishu tool factories on `main` capture `ctx.agentAccountId` and pass it as `defaultAccountId`, but do not thread `ctx.messageChannel` into account resolution. This is true in wiki, drive, perm, bitable, and doc tools.
  - file: [extensions/feishu/src/wiki.ts:176](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/extensions/feishu/src/wiki.ts#L176)
  - command: `nl -ba extensions/feishu/src/wiki.ts | sed -n '174,190p'; nl -ba extensions/feishu/src/drive.ts | sed -n '754,770p'; nl -ba extensions/feishu/src/perm.ts | sed -n '135,150p'; nl -ba extensions/feishu/src/bitable.ts | sed -n '556,583p'; nl -ba extensions/feishu/src/docx.ts | sed -n '1406,1438p'`
  - sha: [f03252aaf9cc](https://github.com/openclaw/openclaw/commit/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022)
- **Tool runtime context carries channel and account independently:** Plugin tool context includes both `messageChannel` and `agentAccountId` as separate fields, so cross-channel account-id collisions are a real input shape. Current Feishu routing ignores the channel field even though the runtime provides it.
  - file: [src/agents/openclaw-tools.plugin-context.ts:42](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/src/agents/openclaw-tools.plugin-context.ts#L42)
  - command: `nl -ba src/agents/openclaw-tools.plugin-context.ts | sed -n '42,65p'`
  - sha: [f03252aaf9cc](https://github.com/openclaw/openclaw/commit/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022)
- **Gateway/session inputs are not normalized to Feishu-specific semantics:** The HTTP tool surface reads `x-openclaw-account-id` with `normalizeOptionalString`, not Feishu account normalization, and forwards it as generic tool context. Current resolver then does a raw `includes` check against Feishu account ids, so the PR's normalization and channel-scoping changes are not already present on `main`.
  - file: [src/gateway/tools-invoke-http.ts:217](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/src/gateway/tools-invoke-http.ts#L217)

  - file: [extensions/feishu/src/tool-factory-test-harness.ts:3](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/extensions/feishu/src/tool-factory-test-harness.ts#L3)

- Current main also lacks the PR's added regression coverage for channel-scoped routing and normalized inherited account ids.
