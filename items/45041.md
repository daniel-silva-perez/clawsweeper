## Summary

Keep open: current `main` suppresses explicit reasoning payloads and strips some structured scaffolding, but it does not implement the requested configurable output-level filter for freeform meta-planning text in assistant replies.

## Evidence

- **No config surface for reply output filters:** Agent config exposes `thinkingDefault` and `reasoningDefault`, and agent defaults expose `heartbeat.includeReasoning`, but there is no `outputFilters` or similar outbound reply-filter setting in the current config types.
  - file: [src/config/types.agents.ts:81](https://github.com/openclaw/openclaw/blob/4630ce3d9e2f089062ccd386b0ae241325961d1a/src/config/types.agents.ts#L81)
  - sha: [4630ce3d9e2f](https://github.com/openclaw/openclaw/commit/4630ce3d9e2f089062ccd386b0ae241325961d1a)
- **Defaults schema still only covers reasoning visibility, not text filtering:** The agent-defaults type includes `thinkingDefault` and `heartbeat.includeReasoning`; there is no generic assistant-output regex/filter config here either.
  - file: [src/config/types.agent-defaults.ts:297](https://github.com/openclaw/openclaw/blob/4630ce3d9e2f089062ccd386b0ae241325961d1a/src/config/types.agent-defaults.ts#L297)
  - sha: [4630ce3d9e2f](https://github.com/openclaw/openclaw/commit/4630ce3d9e2f089062ccd386b0ae241325961d1a)
- **Repository search found no `outputFilters` implementation:** A repo-wide search for `outputFilters` returned no matches, which is strong evidence that the proposed config surface was not added on `main`.
  - command: `rg -n "outputFilters" -S .`
  - sha: [4630ce3d9e2f](https://github.com/openclaw/openclaw/commit/4630ce3d9e2f089062ccd386b0ae241325961d1a)
- **Current shared sanitizer only strips tagged reasoning and structured scaffolding:** The shared visible-text pipeline removes `<think>/<final>` tags, model special tokens, memory tags, and tool-call scaffolding. The reasoning-tag matcher is limited to explicit tag forms, not freeform prose like `Let me think...`.
  - file: [src/shared/text/reasoning-tags.ts:5](https://github.com/openclaw/openclaw/blob/4630ce3d9e2f089062ccd386b0ae241325961d1a/src/shared/text/reasoning-tags.ts#L5)
  - sha: [4630ce3d9e2f](https://github.com/openclaw/openclaw/commit/4630ce3d9e2f089062ccd386b0ae241325961d1a)
- **Outbound suppression is exact/tag-based, not a configurable meta-planning filter:** Core outbound suppression only drops payloads marked `isReasoning`, while Discord skips `Reasoning:`-prefixed stream text and iMessage strips known scaffolding markers. That does not satisfy the issue's request for opt-in regex/pattern filtering of normal assistant output.
  - file: [src/auto-reply/reply/reply-payloads-base.ts:82](https://github.com/openclaw/openclaw/blob/4630ce3d9e2f089062ccd386b0ae241325961d1a/src/auto-reply/reply/reply-payloads-base.ts#L82)
  - sha: [4630ce3d9e2f](https://github.com/openclaw/openclaw/commit/4630ce3d9e2f089062ccd386b0ae241325961d1a)

## Risks / Open Questions

- Users can still get freeform meta-planning leakage in normal assistant text even with `/reasoning off`, because current guards focus on explicit reasoning payloads/tags and known scaffolding formats.
- There are some channel- and provider-specific mitigations on `main`, so future cleanup should avoid closing this as implemented unless a generic configurable outbound filter is actually added.
