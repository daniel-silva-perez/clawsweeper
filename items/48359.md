## Summary

Current `main` routes bound thread follow-ups to the ACP session key stored in the conversation binding, not to a derived `agent:...:acp:...:thread:...` session entry. The reported missing-`acp` metadata path does not match the current routing/binding implementation.

## Evidence

- **Bound conversations retarget to the binding target session:** `initSessionState` resolves the current conversation binding and rewrites `SessionKey` to `binding.targetSessionKey` before session state is loaded, so follow-up thread messages use the bound ACP session key.

## Close Comment

Closing after a Codex review of current `main` (`b4532933`). I could not reproduce the reported routing path in the current implementation: bound thread/conversation follow-ups are retargeted to the binding’s `targetSessionKey` before session init (`src/auto-reply/reply/session.ts:192`), and ACP thread binds store the ACP session key itself as that target (`src/agents/acp-spawn.ts:871`). The current tests also assert that thread-bound ACP commands/status resolve the bound ACP session key rather than a derived `:thread:` key (`src/auto-reply/reply/commands-acp.test.ts:627`, `src/auto-reply/reply/commands-acp.test.ts:1671`) and that bound conversations route to the bound ACP session (`src/auto-reply/reply/session.test.ts:1200`). Based on current `main`, this issue is no longer reproducible as filed.
