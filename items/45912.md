## Summary

Keep open. Current `main` does not already implement cross-root skill symlinks, and the proposed PR shape appears to introduce a real regression: a bundled skill could be loaded through a non-bundled root and bypass `skills.allowBundled` because bundled gating is source-based.

## Evidence

- **Current main still rejects cross-root skill symlinks:** `resolveContainedSkillPath` only accepts a candidate when its realpath stays inside that root's realpath; otherwise it warns and returns null. There is no secondary check for 'inside another configured skill root' on `main`.
  - file: [src/agents/skills/workspace.ts:268](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/src/agents/skills/workspace.ts#L268)
  - command: `nl -ba src/agents/skills/workspace.ts | sed -n '260,590p'`
  - sha: [f03252aaf9cc](https://github.com/openclaw/openclaw/commit/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022)
- **Loaded skill source is inherited from the caller root:** The local skill loader stamps `skill.source` and `sourceInfo.source` directly from `params.source`. A skill discovered via the workspace loader is therefore classified as `openclaw-workspace`, not by its resolved realpath.
  - file: [src/agents/skills/local-loader.ts:77](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/src/agents/skills/local-loader.ts#L77)
  - command: `nl -ba src/agents/skills/local-loader.ts | sed -n '55,90p'`
  - sha: [f03252aaf9cc](https://github.com/openclaw/openclaw/commit/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022)
- **Bundled allowlist enforcement is source-based:** `isBundledSkillAllowed` only applies the allowlist when `resolveSkillSource(entry.skill)` is `openclaw-bundled`; all other sources return true immediately.
  - file: [src/agents/skills/config.ts:52](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/src/agents/skills/config.ts#L52)
  - command: `nl -ba src/agents/skills/config.ts | sed -n '45,95p'`
  - sha: [f03252aaf9cc](https://github.com/openclaw/openclaw/commit/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022)
- **Existing test proves same-name workspace skills bypass bundled allowlist:** The status test asserts that a skill named like a bundled skill but sourced as `openclaw-workspace` is not treated as bundled, is not blocked by the allowlist, and remains eligible.
  - file: [src/agents/skills.buildworkspaceskillstatus.test.ts:142](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/src/agents/skills.buildworkspaceskillstatus.test.ts#L142)

## Risks / Open Questions

- Merging the PR as described can reintroduce bundled skills through workspace/managed/extra/project roots and silently bypass `skills.allowBundled`.
- Closing this PR would be incorrect because `main` still lacks the requested cross-root symlink behavior; the item remains unresolved, just not safe in its current form.
