## Summary

Keep open. Current main still has the reported fail-open path for completion-mode subagent announces when no usable result payload is available: missing captured output is stored as null, cleanup passes no reply/fallback into the announce flow, the announce flow retries history reads and then still builds a delivered completion event with `(no output)`. A current regression test explicitly asserts that behavior. Related PR #68464 appears to be the right direction but is open/unmerged in the provided GitHub context, so it does not close this issue.

## Best Possible Solution

Keep #69767 open and link it to #68464 rather than closing it. The best fix is to make completion-mode announce delivery fail closed: persist a user-safe delivery payload or explicit sanitized fallback before marking delivery complete, return false and keep cleanup retryable when expectsCompletionMessage is true and no reply/child findings/fallback exists, and add regression coverage that no agent call is made instead of delivering `(no output)`. #68464 should be reused or rebased if its delivery-claim/payload design still applies.
