## Summary

Keep this issue open. Current main still has a hardcoded 5,000,000 byte upload-ingestion cap for Control UI/WebChat attachments, and the `gateway.controlUi` config contract has no `maxUploadSizeMb`, `mediaLimits`, or equivalent field. The issue is recent and has an active contributor proposal, with no matching open PR found by issue-number or `maxUploadSizeMb` searches.

## Best Possible Solution

Implement a documented upload policy instead of closing this issue. A conservative first step is a default-preserving `gateway.controlUi.maxUploadSizeMb` or shared chat attachment limit, wired through config types, zod schema, generated schema/help metadata, server-side `chat.send` attachment parsing, Control UI runtime config and prevalidation, and docs. Tests should cover default 5 MB rejection, raised-limit acceptance, oversized rejection, and the WebSocket/base64 ceiling behavior.

## Risks / Open Questions

- A fix that only raises the parser cap may still fail for larger raw files because Control UI sends base64 over a 25 MiB WebSocket frame limit.
- The product/API choice remains whether the knob should be `gateway.controlUi.maxUploadSizeMb`, per-kind `gateway.controlUi.mediaLimits`, or a shared Gateway `chat.send` attachment policy.
