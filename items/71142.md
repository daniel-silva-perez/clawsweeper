## Summary

Keep this issue open. Current main still has a hardcoded 5 MB Control UI/WebChat attachment ingestion limit, no `gateway.controlUi` upload-size config contract, no strict-schema support for the proposed keys, and no configured browser-side prevalidation. Related #44814/#44757 is useful prior context, but it is an older open PR/closed issue pair using a different `gateway.webui.attachmentMaxBytes` surface and does not close or fully cover this issue's current Control UI API/docs/client-prevalidation request.

## Best Possible Solution

Keep this issue open for maintainer product/API direction. The conservative implementation path is a default-preserving configurable Control UI/chat attachment policy, wired through config types, strict zod schema, generated schema/help metadata, `chat.send` attachment parsing, Control UI bootstrap/runtime prevalidation, docs, and focused tests for default 5 MB rejection, raised-limit acceptance, oversized rejection, and the WebSocket/base64 ceiling. Related #44814 can be mined for intent but should be superseded or rebased into the current `gateway.controlUi` design rather than treated as having resolved this request.

## Risks / Open Questions

- A server-only cap increase can still fail for larger raw files because Control UI sends base64 over the Gateway WebSocket, whose documented post-auth max payload is 25 MB.
- There is a product/API choice still open between a simple `gateway.controlUi.maxUploadSizeMb`, a shared chat attachment policy, or per-kind limits; landing the wrong config surface could create avoidable compatibility debt.
