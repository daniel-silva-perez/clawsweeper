## Summary

Keep open. Current main still lacks a first-class tool-plugin setup hook for global setup/onboarding. The generic plugin-config prompt overlaps a small part of the UX, but it only considers enabled/default plugins and explicitly skips sensitive fields, so it does not solve credentialed tool-only plugin onboarding.

## Best Possible Solution

Keep this open as a plugin-SDK/core setup design item. The best path is an additive manifest-first tool-plugin setup descriptor plus a setup-only API/helper that can prompt for SecretInput-backed credentials, write through documented plugin-owned config paths, and include contract tests and public docs alongside the existing provider and channel setup surfaces.

## Risks / Open Questions

- The current generic plugin-config wizard partially overlaps the desired setup experience, so a future implementation should decide whether to extend that path or add a separate descriptor-backed tool-plugin setup step.
- Credential storage still needs maintainer design: tool plugins generally need `plugins.entries.<id>`/plugin-owned config with SecretRef support, while existing provider helpers write provider auth profiles.
