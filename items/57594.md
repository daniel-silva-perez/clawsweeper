## Summary

Keep #57594 open. Current main still routes `openclaw message send --channel zalo --target ...` through generic target resolution, while the Zalo plugin only marks numeric-looking values as native chat IDs. A valid non-numeric Zalo `chat_id` can still fall through directory lookup and produce the reported `Unknown target ... Hint: <chatId>` error before the Zalo send layer is reached.

## Best Possible Solution

Keep the issue open and fix it in the bundled Zalo plugin by changing Zalo target semantics so valid Bot API `chat_id` strings, including hex-like non-numeric values after `zalo:`/`zl:` normalization, are treated as explicit native targets. Add focused regression coverage for `openclaw message send --channel zalo --target 3becaa50ae12474c1e03`-style values and update the Zalo docs/examples away from numeric-only target guidance if upstream `chat_id` is the supported proactive-send identifier.
