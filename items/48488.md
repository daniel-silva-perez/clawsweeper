## Summary

Current `main` still lacks a task-level timeout or automatic stale-task recovery in the lane queue, so the reported permanent lane-blocking failure mode remains plausible and is not implemented on `main`. Keep open.

## Evidence

- **Queue still awaits task completion with no timeout wrapper:** `drainLane()` adds the task ID, then does `const result = await entry.task();` and only calls `completeTask()`/`pump()` after the promise settles. A never-settling task still blocks the lane indefinitely.
  - file: [src/process/command-queue.ts:185](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/src/process/command-queue.ts#L185)
  - command: `nl -ba src/process/command-queue.ts | sed -n '150,260p'`
  - sha: [4013c658537e](https://github.com/openclaw/openclaw/commit/4013c658537ee2051c44186f197b0710e84c9049)
- **Enqueue API still exposes no task-timeout option:** `enqueueCommandInLane()` options only include `warnAfterMs` and `onWait`; there is no `taskTimeoutMs` or equivalent escape hatch for lane tasks.
  - file: [src/process/command-queue.ts:242](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/src/process/command-queue.ts#L242)

  - file: [src/process/command-queue.ts:327](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/src/process/command-queue.ts#L327)

  - file: [src/process/command-queue.test.ts:338](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/src/process/command-queue.test.ts#L338)
