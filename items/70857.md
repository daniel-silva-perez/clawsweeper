## Summary

Keep #70857 open. Current main still has source behavior matching the reported `sessions.json.lock` warning: the session-store lock defaults to a 10s timeout, derives a 15s max hold, holds the file lock across the whole async store task, and the watchdog force-releases over-held locks with the same warning shape. Startup/doctor cleanup only covers transcript `.jsonl.lock` files, not `sessions.json.lock`, and related #71903 is still open and only partially overlaps transcript lock safety.

## Best Possible Solution

Keep #70857 open as a live Windows/session-store lock regression. The maintainer path should profile Windows startup and reply latency, narrow or make safer the `sessions.json` store-lock critical section, prevent watchdog-forced release of active store work, decide whether startup/doctor should inspect `sessions.json.lock`, and coordinate with #71903/#64362 without treating them as a complete fix for this issue.
