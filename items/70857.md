## Summary

Current main still contains the reported long-held session-store lock path: the main mutation helpers synchronously reread `sessions.json` with `skipCache: true` while holding the store lock, and the exact watchdog warning remains active. The related open PR #68554 is still the direct implementation candidate; #71903 is adjacent lock hardening and does not close this report by itself.

## Best Possible Solution

Keep this issue open and review or land #68554, or an equivalent maintainer-owned fix that removes redundant `skipCache` disk rereads from session-store mutation helpers while preserving cache coherency and serialized writes. Update the cache-bypass regression expectation, add focused performance or regression coverage where feasible, and verify startup/RPC responsiveness with a large session store before closing. Treat #71903 as related lock hardening, not the direct fix for this report.
