## Summary

Keep PR #49579 open. Current main still does not suppress routine WhatsApp gateway banners during system-event prompt assembly, so the central issue is not implemented or superseded. However, the submitted PR should not merge unchanged because its broad `whatsapp gateway disconnected` prefix filter would also suppress actionable disconnect states that are currently only queued as the generic disconnect system event.

## Best Possible Solution

Keep the PR open, but revise it before merge: suppress the routine connected banner and only suppress disconnect events known to be low-signal retry flaps, or change the WhatsApp monitor to enqueue richer typed/differentiated system events so prompt assembly can preserve logged-out, conflict, and max-attempt-stop recovery signals. Add regression tests for both suppression and preservation paths.
