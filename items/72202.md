## Summary

Keep #72202 open. It is a fresh, specific Windows report against v2026.4.24, and current main does not provide enough proof to close it as implemented, duplicate, or stale. The OpenAI image provider has adjacent route-selection logic, but there is no source/test evidence covering the exact reported case: `OPENAI_API_KEY` supplied through `openclaw.json` env, no `openai-codex` OAuth profile, gateway restart on Windows, and `openai/gpt-image-2` still taking the Codex Responses route.

## Best Possible Solution

Keep #72202 open. Add a focused regression test for `openclaw.json`/`cfg.env.vars.OPENAI_API_KEY` with an empty auth store and no `openai-codex` profile, asserting `openai/gpt-image-2` calls `/v1/images/generations` and does not resolve or call the Codex Responses route. Then investigate whether Windows gateway restart or runtime snapshot loading can initialize image provider auth before config env is applied. Track the provider-config persistence part separately under #44786 unless the same root cause is proven.
