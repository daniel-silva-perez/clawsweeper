## Summary

Current main should not be closed as implemented. The Control UI now has image-only drag/drop, paste, and file-picker attachment support, but #54854 asks for the broader old local desktop workflow where dropping a local file produces a usable local path/reference. Current UI and Gateway code still filter to image attachments and do not insert or send arbitrary local file paths/references. Related reports about image upload are therefore adjacent but not canonical for this remaining request.

## Best Possible Solution

Keep #54854 open and scope it as a Control UI/Gateway local-file workflow request. The likely maintainer path is to design a safe local desktop mechanism: accept non-image drops in the composer, produce either a scoped local path/reference or a managed Gateway attachment reference, extend the `chat.send` attachment contract beyond image base64, add Gateway validation/local-root safeguards, and cover image, PDF/text, unsupported-file, and remote-browser cases in UI and Gateway tests. Treat the `openclaw-weixin` load failure as a separate plugin compatibility/support thread if it still reproduces.
