## Summary

Current `main` has partial mitigations for large tool payloads, but it does not implement the requested `exec`-level streaming or file-backed argument surface. The issue body's specific `>10KB` / JSON-RPC diagnosis does not match current code paths, yet the enhancement itself is still coherent and core-scope, so there is not strong evidence to close it.

## Evidence

- **`exec` still has no stdin/file argument API:** The core `exec` schema only exposes `command`, `workdir`, `env`, timing, PTY, host, security, ask, and node fields. There is no `stdin`, `stdinFile`, or equivalent file-backed argument field on `exec`.
  - file: [src/agents/bash-tools.schemas.ts:3](https://github.com/openclaw/openclaw/blob/754acc4478fe361db1e5303fd60f589d52beea7a/src/agents/bash-tools.schemas.ts#L3)
  - command: `nl -ba src/agents/bash-tools.schemas.ts | sed -n '1,24p;51,66p'`
  - sha: [754acc4478fe](https://github.com/openclaw/openclaw/commit/754acc4478fe361db1e5303fd60f589d52beea7a)
- **There is an indirect stdin workaround via `process`:** OpenClaw already supports background-session stdin writes through `process.write` and `process.paste`, and the exec docs show `process` paste as the supported way to send multiline content after starting a command.
  - file: [src/agents/bash-tools.process.ts:435](https://github.com/openclaw/openclaw/blob/754acc4478fe361db1e5303fd60f589d52beea7a/src/agents/bash-tools.process.ts#L435)
  - command: `nl -ba src/agents/bash-tools.process.ts | sed -n '432,514p'`

- **Docs expose the same workaround:** The exec docs explicitly document `process` paste for multiline input, which means the repo already expects large content to be fed through a running process rather than a new `exec` argument surface.
  - file: [docs/tools/exec.md:235](https://github.com/openclaw/openclaw/blob/754acc4478fe361db1e5303fd60f589d52beea7a/docs/tools/exec.md#L235)
