## Summary

Current `main` still routes local CLI Gateway RPCs to `127.0.0.1` regardless of `gateway.bind`, so the reported `bind=tailnet` management-command failure is not implemented/fixed on `main` and should remain open.

## Evidence

- **Local CLI connection details still hardcode loopback:** `buildGatewayConnectionDetailsWithResolvers()` derives `localUrl` as `${scheme}://127.0.0.1:${localPort}` and uses it whenever there is no explicit URL override or `gateway.remote.url`, while only recording `Bind: ${bindMode}` as metadata.
  - file: [src/gateway/connection-details.ts:41](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/src/gateway/connection-details.ts#L41)
  - command: `nl -ba src/gateway/connection-details.ts | sed -n '1,220p'`
  - sha: [ebd7f19a3b20](https://github.com/openclaw/openclaw/commit/ebd7f19a3b20001960474ec9ba9cc424e45f7b23)
- **All CLI RPC commands use that connection-details path:** `callGatewayWithScopes()` builds connection details and then connects to `connectionDetails.url`; `callGatewayFromCliRuntime()` is the shared path used by cron management commands such as `cron.list` and `cron.add`.
  - file: [src/gateway/call.ts:573](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/src/gateway/call.ts#L573)

- **Current tests explicitly expect tailnet local calls to use loopback:** `call.test.ts` has active test cases for `gateway.bind: "tailnet"` that assert the URL is `ws://127.0.0.1:18800` / `wss://127.0.0.1:18800`, confirming this is still the intended current behavior on `main`.
  - file: [src/gateway/call.test.ts:226](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/src/gateway/call.test.ts#L226)
