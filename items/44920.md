## Summary

Current main already covers the reported stale-`runningAtMs` restart failure and the related `delivery.mode: "none"` behavior. Startup now clears persisted running markers and replays interrupted recurring jobs, `mode:none` no longer triggers outbound delivery just because a channel/target is present, and cron update patches are normalized before validation so nullish delivery fields are stripped instead of hard-failing schema validation.

## Close Comment

Closing from a Codex review because current `main` already implements this well enough. The restart path in `src/cron/service/ops.ts:87` clears persisted `runningAtMs` markers on startup, and `src/cron/service.restart-catchup.test.ts:118` covers the exact stale-marker-after-restart case by asserting the marker is cleared and the recurring job replays immediately. This implementation is present in the shipped `v2026.4.22` release (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). If someone can still reproduce this on `v2026.4.22` or newer, a fresh issue with current logs and the exact stored cron payload/state would be the right follow-up.
