## Summary

Keep #70586 open. Current main still has a real Slack message-event liveness gap for paths that return before `createSlackMessageHandler`, but the submitted PR is not safe to close as implemented or merge as-is because it appears to double-update liveness for normal message and channel `app_mention` flows that already call `trackEvent`.

## Best Possible Solution

Keep the PR open and revise it narrowly: thread `trackEvent` into `registerSlackMessageEvents`, but only call it for accepted message-event branches that do not reach `createSlackMessageHandler`, or otherwise centralize tracking so each inbound event updates liveness once. Add regression tests for subtype system events, DM-skipped mentions, mismatched-event drops, and no double tracking for normal messages/app mentions.
