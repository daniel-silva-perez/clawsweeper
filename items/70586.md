## Summary

Keep this PR open. Current main still has a Slack message-event activity gap for accepted registrar branches that return before `createSlackMessageHandler`, but this PR tracks too early and would duplicate or over-count events; its stale-socket rationale is also obsolete because current health checks use transport activity, not `lastEventAt`.

## Best Possible Solution

Request a narrower revision: thread `trackEvent` into `registerSlackMessageEvents`, but invoke it only on accepted message-event branches that return before `handleSlackMessage` and genuinely represent inbound app activity. Keep `createSlackMessageHandler` as the single tracking point for normal handled messages and app mentions, and add tests proving handled events track once, accepted early-return branches track once, and mismatched, duplicate, invalid, unauthorized, or self-generated events do not track. Reframe the PR around app-activity timestamp completeness rather than stale socket liveness.
