## Summary

Current main still does not provide a live provider/model-attempt signal that the TUI footer consumes during model fallback. The fallback runner iterates candidates and records/logs attempts, but it does not emit a per-attempt agent event. The TUI footer still renders from sessionInfo, which is refreshed from session/list state, and the agent-event handler ignores fallback/model identity streams. Existing fallback lifecycle events are post-success observability for channel replies, not a real-time status-bar update path. The related items in the provided context are adjacent stale/model-display reports, but none is a canonical merged fix for this provider-fallback TUI case.

## Best Possible Solution

Keep this open. Implement a narrow, typed live fallback-attempt status path: emit an agent event before each actually attempted fallback candidate with runId, sessionKey, provider, model, attempt, and total; make the TUI handle that stream with session/run filtering before the existing known-run drop when needed; update state.sessionInfo.modelProvider/model and the footer immediately; and add regression coverage for the fallback emission path plus tui-event-handlers state updates. Reconcile the model-failover docs if the chosen implementation changes the documented persisted-override behavior.
