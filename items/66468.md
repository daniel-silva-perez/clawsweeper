## Summary

Keep open. Current main still lacks a real-time model-attempt signal from the fallback loop, and the TUI still updates the footer from session-list/session-info state rather than a fallback-attempt event. The provided related reports are adjacent stale/status-bar items, but none is a canonical merged fix for this provider-fallback case.

## Best Possible Solution

Implement a narrow live fallback-attempt status path: emit a typed agent event from the fallback runner or its caller before each actually attempted candidate, include runId and sessionKey, update the TUI model footer from that event with session/run filtering, and add regression tests for `runWithModelFallback`/agent-command event emission plus `tui-event-handlers` state updates.

## Evidence
