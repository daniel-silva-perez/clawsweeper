## Summary

Current main still has a plausible stale-context path after compaction. The shared status formatter starts from the persisted session total and only lets transcript usage replace it when the transcript-derived candidate is larger, so a smaller post-compaction prompt size cannot lower the displayed Context line. The related closed status reports cover sparse/zero/missing usage fallback, but not this post-compaction decrease case.

## Best Possible Solution

Keep the issue open. The best fix is to make successful manual and auto-compaction update the session usage snapshot to the post-compaction prompt size, or explicitly mark stored totals stale/compacted so `buildStatusMessage()` can prefer the latest post-compaction transcript usage even when it is smaller. Add regression coverage for `/compact`, auto-compaction reconcile, `/status`, and `session_status` showing `Context` from the current post-compaction state while preserving the pre-compaction peak on the Compactions line.

## Evidence
