## Summary

Keep open. Current main has not implemented this behavior: the CLI runner still resolves a no-output timeout for normal CLI spawns and passes it to the supervisor even when prompt input is sent through stdin. The PR is a plausible focused fix, but it is not superseded, and its current diff still needs maintainer review for the empty-stdin edge case, timeout messaging, and regression/live validation.

## Best Possible Solution

Keep this PR open and review it as an implementation candidate. The best path is to update the condition to reflect stdin mode/presence precisely, avoid misleading timeout text when the watchdog is disabled, add a focused CLI-runner regression test showing `noOutputTimeoutMs` is omitted for stdin-fed closed-stdin spawns while `timeoutMs` remains set, and separately decide whether Claude live-session no-output behavior needs the same product change.

## Risks / Open Questions

- The proposed truthiness check can miss an empty-string stdin payload even though stdin mode is active and closed.
- The proposed `?? 0` fallback can produce confusing no-output timeout messages if an unexpected branch is reached.
- Disabling the no-output watchdog for stdin-fed CLI runs may let a truly silent hung backend consume the full overall timeout, so the accepted behavior should be covered by a regression test and, ideally, one live/manual CLI check.
- The bundled Claude CLI now has a separate live-session path with its own no-output timer, so maintainers should decide whether the central Claude Code use case needs a companion change there or whether this PR is only for non-live/custom stdin backends.
