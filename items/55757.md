## Summary

Keep this PR open. Current main still computes and passes a no-output watchdog timeout into normal CLI child spawns even when the prompt is sent through stdin, so the change is not implemented or obsolete. The PR is a plausible focused fix, but the current diff still needs review for the stdin-mode condition, timeout messaging, regression coverage, and Claude live-session behavior.

## Best Possible Solution

Keep this PR open as an implementation candidate. The best path is to revise the condition to key off raw stdin presence or explicit stdin mode, avoid misleading timeout text when the watchdog is disabled, add a focused CLI-runner regression test proving `noOutputTimeoutMs` is omitted for stdin-fed closed-stdin child spawns while `timeoutMs` remains set, and have maintainers decide whether the Claude live-session path needs the same product behavior.

## Risks / Open Questions

- The proposed `stdinPayload` truthiness check can miss empty-string stdin mode even though stdin is closed immediately.
- The proposed `?? 0` fallback can produce misleading no-output timeout messages if an unexpected branch is reached.
- Disabling the no-output watchdog for stdin-fed child spawns makes silent hung backends rely on the overall timeout, so this needs a regression test and ideally manual/live validation.
- The Claude Code example in the PR may now route through the separate `claude-stdio` live-session path, which has its own no-output timer behavior.
