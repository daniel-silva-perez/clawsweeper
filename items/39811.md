## Summary

Keep #39811 open. Current main has partial mitigations for the silent-fallback symptom, but it still accepts arbitrary configured model strings and `openclaw config validate` remains schema-focused rather than catalog/upstream-model-aware.

## Best Possible Solution

Keep the issue open. The best path is to add catalog-aware validation for configured primary/fallback/model-provider refs in the config or doctor surface, with clear warning/error semantics for dynamic and custom providers, and separately ensure model fallback events are surfaced to users through an appropriate channel/status path rather than only subsystem logs.

## Risks / Open Questions

- Strict upstream validation needs a provider policy for dynamic catalogs, custom/self-hosted providers, and intentionally user-defined model IDs so valid operator configurations are not rejected.
- The issue now combines two related asks: config-time model validation and user-visible fallback notification. Maintainers may want to split the remaining work, but #39811 is still a valid canonical tracker for the unresolved config-validation half.
