## Summary

Keep open. Current main partially supports remote Ollama discovery when an explicit provider uses `models: []`, but the reported schema/docs contradiction still exists: `models.providers.*.models` is required by the runtime schema and generated config schema, while the Ollama custom-base-url docs still show a provider block without `models`. The related PRs that would have made `models` optional were closed unmerged, so there is no canonical shipped fix for the specific validation issue.

## Best Possible Solution

Keep the issue open and resolve it with a narrow config-contract fix: either make `ModelProviderConfig.models` and `ModelProviderSchema.models` optional, regenerate/update the config schema and docs, and add a regression test for `models.providers.ollama` with `baseUrl` and no `models`; or explicitly document `models: []` as the supported remote autodiscovery sentinel and update the custom-base-url example so the documented config validates. The cleaner product direction is optional `models`, because the existing Ollama discovery code already handles `undefined` and empty arrays as no explicit models.

## Risks / Open Questions

- Current main has a workaround with `models: []`, so the eventual maintainer fix may be schema/docs alignment rather than a larger discovery rewrite.
- `openclaw models list --local` also filters by loopback, `0.0.0.0`, `::1`, or `.local` hosts, so a LAN IP such as `192.168.x.x` may still be excluded from `--local` output even after schema alignment.
