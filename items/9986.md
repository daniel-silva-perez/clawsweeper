## Summary

Current `main` does not implement context-overflow-triggered model fallback. The runtime explicitly keeps context overflow on the compaction/retry path, rethrows overflow errors out of the model-fallback loop, and eventually surfaces a `context_overflow` error instead of advancing to fallback candidates.

## Evidence

- **Docs explicitly exclude overflow from model fallback:** `docs/concepts/model-failover.md` says model fallback advances on auth/rate-limit/timeout/billing paths, but 'does not continue on ... context overflow errors that should stay inside compaction/retry logic.'
  - file: [docs/concepts/model-failover.md:213](https://github.com/openclaw/openclaw/blob/754acc4478fe361db1e5303fd60f589d52beea7a/docs/concepts/model-failover.md#L213)
  - command: `nl -ba docs/concepts/model-failover.md | sed -n '200,280p'`
  - sha: [754acc4478fe](https://github.com/openclaw/openclaw/commit/754acc4478fe361db1e5303fd60f589d52beea7a)
- **Fallback loop rethrows context overflow instead of trying next candidate:** `src/agents/model-fallback.ts` comments that context overflow should be handled by inner compaction/retry logic, then immediately rethrows when `isLikelyContextOverflowError(errMessage)` is true.
  - file: [src/agents/model-fallback.ts:840](https://github.com/openclaw/openclaw/blob/754acc4478fe361db1e5303fd60f589d52beea7a/src/agents/model-fallback.ts#L840)
