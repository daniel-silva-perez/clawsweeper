## Summary

Current main still matches the reported failure mode. The stop-timeout branch records `running: true` and returns before clearing `store.tasks` / `store.aborts`, while `startChannelInternal` still silently returns when a task entry exists. The current unit test also still asserts that a second start is blocked after a stop timeout. The related fix PR #71456 is closed/unmerged in the provided context and its expected test change is absent from current main.

## Best Possible Solution

Keep #71412 open and land PR #71456 or an equivalent focused patch. The fix should clear stale `store.aborts` and `store.tasks` entries on stop timeout, mark the runtime stopped instead of running, guard late zombie-task status writes so they cannot clobber a fresh task, and update the regression test to assert that the next start registers a new account task. Treat the Telegram `openclaw doctor` liveness improvement as a separate follow-up unless maintainers explicitly want it bundled.
