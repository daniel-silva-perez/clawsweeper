## Summary

Current main still contains the reported stop-timeout failure mode. The timeout branch returns before clearing the stale task/abort entries, marks the runtime as running, and the start path still no-ops when a task entry is present. The existing regression test also still asserts that a second start is blocked after a stop timeout. The related fix PR #71456 is closed and unmerged in the provided GitHub context.

## Best Possible Solution

Keep this issue open and land PR #71456 or an equivalent core gateway fix. On the stop-timeout path, clear the stale store.aborts and store.tasks entries before returning, mark the runtime stopped instead of running, and update the regression test to prove that a subsequent start registers a fresh task. Treat any openclaw doctor liveness improvement as a separate follow-up unless maintainers choose to bundle it.
