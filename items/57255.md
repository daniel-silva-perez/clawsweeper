## Summary

Current `main` no longer silently lets built-in tools shadow caller-provided `/v1/responses` tools. It now detects name collisions before session creation and returns a 400 `invalid_request_error`, so the PR's reported failure mode does not reproduce as described.

## Close Comment

Closing this PR because the reported silent collision behavior does not reproduce on current `main`.

- Codex review: `src/agents/pi-embedded-runner/run/attempt.ts:1168` now preflights caller-provided tool names against built-in and reserved Pi tool names, and rejects conflicts before session creation.
- `src/agents/pi-tool-definition-adapter.ts:172` normalizes and detects both built-in collisions and duplicate client tool names.
- `src/gateway/openresponses-http.ts:783` turns that path into a `400 invalid_request_error` with `invalid tool configuration`, instead of silently letting a built-in tool win.
- `src/gateway/openresponses-http.test.ts:328` covers that response behavior.
- The same guard logic is already present in the latest release tag, `v2026.4.23`.

That means the original bug this PR addresses is obsolete on current `main`. If transparent auto-prefixing is still desired, that should be proposed as a fresh behavior change rather than kept open as a required bug fix.
