## Summary

Current main still has the memory-lancedb behavior this PR targets: the visible `memory_forget` candidate list truncates IDs to 8 characters while direct deletion validates a full UUID. The PR is therefore not obsolete, but maintainers should resolve the security/product tradeoff raised in review and remove the unrelated generated baseline churn before merge.

## Best Possible Solution

Keep this PR open as the active implementation candidate. Maintainers should either land the focused memory-lancedb fix with its regression test, or revise it to use non-authoritative handles or explicit confirmation for candidate deletion; before merge, remove or justify the unrelated generated docs baseline change.

## Risks / Open Questions

- Showing full UUIDs in visible tool output fixes ambiguity but may increase the blast radius of prompt-injection or compromised-model paths that can issue follow-up `memory_forget` calls. Ephemeral handles or explicit confirmation may be safer.
- The PR includes unrelated generated `docs/.generated/plugin-sdk-api-baseline.sha256` churn for a memory-lancedb behavior change; that should be removed or justified before merge.
