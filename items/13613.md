## Summary

Keep open. Current `main` still models cron as gateway-local state and execution, with only process-local locking/active-job tracking. I found no distributed cron leader election, lock backend, or HA coordination config on `main`.

## Evidence

- **Cron is documented as gateway-local with local state files:** The cron docs say cron runs inside the Gateway process and persists definitions/state in local files (`~/.openclaw/cron/jobs.json` and `jobs-state.json`), which matches single-gateway/local ownership rather than shared HA coordination.
  - file: [docs/automation/cron-jobs.md:36](https://github.com/openclaw/openclaw/blob/3361593442e295e9bb1f55cbfb38861ea9413e45/docs/automation/cron-jobs.md#L36)
  - command: `nl -ba docs/automation/cron-jobs.md | sed -n '34,60p'`
  - sha: [3361593442e2](https://github.com/openclaw/openclaw/commit/3361593442e295e9bb1f55cbfb38861ea9413e45)
- **Cron config has no coordination backend settings:** `CronConfig` exposes `enabled`, `store`, `maxConcurrentRuns`, retry/webhook/session retention/run-log/failure settings, but nothing for Redis/etcd/DynamoDB, leader election, leases, or distributed locks.
  - file: [src/config/types.cron.ts:30](https://github.com/openclaw/openclaw/blob/3361593442e295e9bb1f55cbfb38861ea9413e45/src/config/types.cron.ts#L30)
  - command: `nl -ba src/config/types.cron.ts | sed -n '25,70p'`
  - sha: [3361593442e2](https://github.com/openclaw/openclaw/commit/3361593442e295e9bb1f55cbfb38861ea9413e45)
- **Scheduler lock is process-local only:** The cron service lock is an in-memory `Map<string, Promise<void>>` keyed by store path. That serializes operations within one process, not across multiple gateways.
  - file: [src/cron/service/locked.ts:3](https://github.com/openclaw/openclaw/blob/3361593442e295e9bb1f55cbfb38861ea9413e45/src/cron/service/locked.ts#L3)
  - command: `nl -ba src/cron/service/locked.ts | sed -n '1,80p'`
  - sha: [3361593442e2](https://github.com/openclaw/openclaw/commit/3361593442e295e9bb1f55cbfb38861ea9413e45)
- **Active-job tracking is process-global, not distributed:** Cron active jobs are tracked in a global singleton `Set<string>`, again indicating only in-process coordination.
  - file: [src/cron/active-jobs.ts:7](https://github.com/openclaw/openclaw/blob/3361593442e295e9bb1f55cbfb38861ea9413e45/src/cron/active-jobs.ts#L7)
  - command: `nl -ba src/cron/active-jobs.ts | sed -n '1,120p'`
  - sha: [3361593442e2](https://github.com/openclaw/openclaw/commit/3361593442e295e9bb1f55cbfb38861ea9413e45)
- **Multi-gateway is allowed, but only by bypassing the single-gateway lock:** Gateway startup can skip the gateway lock when `OPENCLAW_ALLOW_MULTI_GATEWAY=1`, and the discovery docs explicitly say isolated multi-gateway setups are possible. I found no matching cron-side HA coordination added alongside that capability.
  - file: [src/infra/gateway-lock.ts:246](https://github.com/openclaw/openclaw/blob/3361593442e295e9bb1f55cbfb38861ea9413e45/src/infra/gateway-lock.ts#L246)
