## Summary

Keep open. Current main still resolves `skills info` with an exact `name` or `skillKey` comparison only, so the PR's case-insensitive and separator-normalized lookup behavior is not implemented on main. The linked report is plausible because skill status entries carry separate `name` and `skillKey` fields, and the prior PR #38701 was closed unmerged and explicitly superseded by this PR rather than serving as a canonical shipped fix.

## Best Possible Solution

Keep this PR open and refresh it against current main. The best implementation path is to add a small resolver in `src/cli/skills-cli.format.ts` that preserves exact lookup first, then allows unique case-insensitive and separator-normalized matches with ambiguity guards, trims/sanitizes not-found output, and adds focused regression tests in `src/cli/skills-cli.test.ts`.
