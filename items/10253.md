## Summary

Current `main` does not implement the requested hook/webhook session lifecycle policy. Hook-triggered agent runs still persist under their hook session keys, while the new retention path only targets `agent:<id>:cron:<jobId>:run:<uuid>` entries. Generic `session.maintenance` exists, but it is a coarse global store policy (default `30d` / `500` entries), not hook-specific cleanup, delayed archival, transcript-retention control, or reuse of `agents.defaults.subagents.archiveAfterMinutes`.

## Evidence

- **Hook dispatch passes through the requested hook session key:** Gateway hook agent dispatch builds an isolated cron-style job but forwards `value.sessionKey` directly into `runCronIsolatedAgentTurn`, so hook-triggered runs stay bound to the caller/mapping session key.
  - file: [src/gateway/server/hooks.ts:43](https://github.com/openclaw/openclaw/blob/754acc4478fe361db1e5303fd60f589d52beea7a/src/gateway/server/hooks.ts#L43)
  - command: `nl -ba src/gateway/server/hooks.ts | sed -n '35,90p'`
  - sha: [754acc4478fe](https://github.com/openclaw/openclaw/commit/754acc4478fe361db1e5303fd60f589d52beea7a)
- **Non-cron hook keys do not become ephemeral `:run:` sessions:** `runCronIsolatedAgentTurn` uses the incoming session key unless absent, and only appends `:run:<sessionId>` when the base key starts with `cron:`. For hook keys like `hook:gmail:...`, `runSessionKey` stays the stable agent-scoped hook key.
  - file: [src/cron/isolated-agent/run.ts:459](https://github.com/openclaw/openclaw/blob/754acc4478fe361db1e5303fd60f589d52beea7a/src/cron/isolated-agent/run.ts#L459)
  - command: `nl -ba src/cron/isolated-agent/run.ts | sed -n '455,505p'`
  - sha: [754acc4478fe](https://github.com/openclaw/openclaw/commit/754acc4478fe361db1e5303fd60f589d52beea7a)
- **The dedicated reaper only recognizes cron run-session keys:** `isCronRunSessionKey` matches only agent-scoped keys whose suffix is `cron:<jobId>:run:<uuid>`, and `sweepCronRunSessions` prunes only entries passing that check.
  - file: [src/sessions/session-key-utils.ts:50](https://github.com/openclaw/openclaw/blob/754acc4478fe361db1e5303fd60f589d52beea7a/src/sessions/session-key-utils.ts#L50)
  - command: `nl -ba src/sessions/session-key-utils.ts | sed -n '1,70p'`
  - sha: [754acc4478fe](https://github.com/openclaw/openclaw/commit/754acc4478fe361db1e5303fd60f589d52beea7a)
- **Cron retention is documented as cron-run-only:** The user-facing docs say `cron.sessionRetention` prunes isolated run-session entries. They do not document hook/webhook session cleanup or a hook-specific archival policy.
  - file: [docs/automation/cron-jobs.md:386](https://github.com/openclaw/openclaw/blob/754acc4478fe361db1e5303fd60f589d52beea7a/docs/automation/cron-jobs.md#L386)

  - sha: [754acc4478fe](https://github.com/openclaw/openclaw/commit/754acc4478fe361db1e5303fd60f589d52beea7a)
- **Existing session maintenance is global, coarse, and not hook-specific:** The built-in session maintenance defaults are a global `30d` stale cutoff and `500` entry cap. That is not equivalent to on-complete or after-minutes hook cleanup, transcript retention options, or subagent-style archival behavior.
  - file: [src/config/sessions/store-maintenance.ts:12](https://github.com/openclaw/openclaw/blob/754acc4478fe361db1e5303fd60f589d52beea7a/src/config/sessions/store-maintenance.ts#L12)
