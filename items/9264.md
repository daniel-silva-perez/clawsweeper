## Summary

Current main treats cross-session/channel recall as a memory/plugin concern, not a core session-merging feature. Core intentionally keeps group chats isolated, while the documented Honcho memory plugin already provides cross-session and cross-channel user memory plus search/modeling.

## Risks / Open Questions

- This closes a real UX need, but only because current architecture routes it to plugins rather than core; users who want zero-setup built-in behavior may still be dissatisfied.
- Honcho is an external plugin path, so operators with stricter local-only/privacy requirements may still want a different memory plugin implementation later.

## Close Comment

Closing after a Codex review. This is a useful idea, but on current `main` it fits the memory/plugin layer rather than OpenClaw core session routing. `VISION.md:52` says optional capability should usually ship as plugins and that memory is a dedicated plugin slot; `docs/concepts/memory-honcho.md:5` already documents a plugin path that provides persistent memory across sessions and channels with user modeling/search. Core routing is still intentionally conservative: `docs/concepts/session.md:16` keeps group chats isolated, and `src/routing/session-key.ts:142` only applies identity linking to direct-message peers. Current main also already exposes explicit cross-session recall/send surfaces in `docs/concepts/session-tool.md:15`, and the shipped core session-linking feature in `CHANGELOG.md:6110` is DM-only. Given that architecture, this does not look like a core-maintainer cleanup item for `main`; it belongs as a plugin/ClawHub-style capability instead.
