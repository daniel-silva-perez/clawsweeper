## Summary

Keep #71330 open. Current main still hardcodes memory-core short-term/deep promotion writes to workspace `MEMORY.md`, and the public config schema, CLI docs, and tests all reflect that behavior. The related memory/vault issues are broader adjacent proposals, not a canonical replacement for this narrow configurable promotion-target request.

## Best Possible Solution

Keep this issue open and implement an additive memory-core promotion target config with `MEMORY.md` as the default. The change should wire scheduled dreaming and `memory promote --apply` through the same resolver, create missing parent files safely, preserve promotion marker de-duplication, document how targets under `memory/` or extra paths remain searchable, and add source/docs/tests for the new config surface.
