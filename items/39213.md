## Summary

Keep PR #39213 open. Current main still rotates Telegram answer previews across assistant-message boundaries in `streamMode: "partial"`, lacks the PR's `revive()` / `shouldSplitPreviewMessages` / refinalization mechanism, and tests still assert new Telegram messages in partial mode. The PR also has an explicit maintainer hold asking for rebase/alignment with #38906 and unresolved boundary-regression work.

## Best Possible Solution

Keep #39213 open as the active Telegram partial-preview coalescing work, or replace it with a rebased maintainer-owned PR. The implementation should live under `extensions/telegram`, explicitly account for #38906/current lane lifecycle behavior, add a safe stream-revival/refinalization path for partial mode, preserve block-mode rotation, and cover early-partial and media-only boundary cases with focused regression tests.
