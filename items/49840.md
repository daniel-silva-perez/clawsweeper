## Summary

Keep open. Current main still has no `maxCostPerRun` heartbeat or embedded-run cost cap, and the PR is a focused attempt to address the open #49823/#3181 cost-control problem. It is not closeable as implemented or obsolete, but it also needs maintainer/code-review follow-up because the PR discussion identifies unresolved correctness issues around transcript-aware estimation and effective model resolution.

## Best Possible Solution

Keep the PR or its linked issue active for maintainer follow-up. The best path is to implement a real per-run cap in the embedded heartbeat dispatch path using the same effective model and session context that `getReplyFromConfig()` will use, with schema/docs/tests updated and the existing review comments resolved. If maintainers prefer not to revive this PR, #49823 should remain the canonical tracking issue rather than closing the capability as already handled.
