## Summary

Current `main` does not implement the requested safeguard. Android notification forwarding still forwards any non-OpenClaw package unless the operator manually configures the app-level package filter, and the gateway still turns `notifications.changed` into a separate node-scoped heartbeat/session path. That means the duplicate WhatsApp-via-Baileys plus WhatsApp-via-node-notification routing conflict described in #48516 remains plausible on current `main`, so there is not strong evidence to close this conservatively.

## Evidence

- **Android listener still forwards notifications by package policy, with no built-in native-channel exclusions:** `DeviceNotificationListenerService.onNotificationPosted()` only skips the app's own package, then forwards when `policy.allowsPackage(normalizedPackage)` passes. There is no hardcoded exclusion for WhatsApp/Telegram/Discord/Signal; filtering is entirely operator-configured.
