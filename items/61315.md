## Summary

Keep #61315 open. Current main still has the behavior the PR targets: `sessions.usage` aggregates only the limited returned set, channel/agent rankings are cost-first, Top Channels/Agents still show cost as the primary value, and the response contract has no full-match overview scope. The PR is plausible but needs revision because its review thread includes unresolved limit, scope, and fallback concerns.

## Best Possible Solution

Keep the PR open and ask for a rebase plus focused fixes: add shared cost/token ranking semantics with regression tests, make no-filter `sessions.usage` totals/aggregates and UI insight denominators use a consistent full matched-session scope, preserve returned-session `limit` exactly, and retain a backward-compatible limit warning fallback when `overview` is absent.
