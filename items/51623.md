## Summary

Keep open. Current main still does not implement the PR’s pending followup-queue persistence/replay, and the PR itself has unresolved review concerns around duplicate replay, TTL-based loss, and trusted system-event replay of user text.

## Best Possible Solution

Keep the PR open for maintainer judgment or replace it with a narrower maintainer-clean implementation: persist queued followups and drain-window inbound context through a durable queue, replay via the normal inbound/followup dispatch path with routing/auth semantics preserved, mark replayed user content untrusted, avoid fixed short TTL loss, handle in-process restarts without duplication, and add focused regression tests.

## Risks / Open Questions

- Queued followup items can still be lost on a true process restart because current main has no disk-backed followup snapshot or replay path.
- Merging this PR as-is could introduce duplicate recovery on in-process restarts and trust-boundary issues if user text is replayed as trusted system context.
- The later restart-continuation/session-delivery work is adjacent but narrower, so closing this PR as implemented or superseded would hide remaining generic inbound/followup restart-loss work.
