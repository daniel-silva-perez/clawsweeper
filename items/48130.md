## Summary

Keep open. Current `main` still has the broken unquoted Windows executable regex in `extensions/browser/src/browser/chrome.executables.ts`, so the PR is not fully implemented on `main`. One hunk is now stale because the quoted-path regex is already fixed on `main`, but the unquoted-path fix is still missing and there is no Windows default-browser regression test covering this path.

## Evidence

- **Current main still has broken unquoted regex:** `extractWindowsExecutablePath()` on current `main` still uses `const unquoted = command.match(/([^\\s]+\\.exe)/i);`, which excludes backslashes and cannot capture a full Windows path.
  - file: [extensions/browser/src/browser/chrome.executables.ts:451](https://github.com/openclaw/openclaw/blob/b453293349657dedd069d291fa5b078d910067b1/extensions/browser/src/browser/chrome.executables.ts#L451)
  - command: `nl -ba extensions/browser/src/browser/chrome.executables.ts | sed -n '446,452p'`
  - sha: [b45329334965](https://github.com/openclaw/openclaw/commit/b453293349657dedd069d291fa5b078d910067b1)
- **Buggy regex is on the active Windows default-browser path:** `detectDefaultChromiumExecutableWindows()` expands the registry command and immediately feeds it into `extractWindowsExecutablePath()`, so the bad unquoted regex directly affects Windows default-browser detection.
  - file: [extensions/browser/src/browser/chrome.executables.ts:299](https://github.com/openclaw/openclaw/blob/b453293349657dedd069d291fa5b078d910067b1/extensions/browser/src/browser/chrome.executables.ts#L299)
  - command: `nl -ba extensions/browser/src/browser/chrome.executables.ts | sed -n '299,318p'`
  - sha: [b45329334965](https://github.com/openclaw/openclaw/commit/b453293349657dedd069d291fa5b078d910067b1)
- **Local reproduction shows current regex returns only basename:** For sample `C:\\ProgramFiles\\Google\\Chrome\\Application\\chrome.exe --single-argument %1`, the current regex returns `chrome.exe`, while the PR's proposed `/(\\S+\\.exe)/i` returns the full executable path.
  - command: `node - <<'NODE'
const current = /([^\\s]+\.exe)/i;
const fixed = /(\S+\.exe)/i;
const sample = 'C:\\ProgramFiles\\Google\\Chrome\\Application\\chrome.exe --single-argument %1';
console.log({sample, current: sample.match(current)?.[1] ?? null, fixed: sample.match(fixed)?.[1] ?? null});
NODE`
- **PR is only partially obsolete:** Current `main` already has the quoted-path form `const quoted = command.match(/"([^"]+\\.exe)"/i);`, so that part of the PR is already present, but the unquoted fix is not.
  - file: [extensions/browser/src/browser/chrome.executables.ts:447](https://github.com/openclaw/openclaw/blob/b453293349657dedd069d291fa5b078d910067b1/extensions/browser/src/browser/chrome.executables.ts#L447)
  - command: `nl -ba extensions/browser/src/browser/chrome.executables.ts | sed -n '446,451p'`
  - sha: [b45329334965](https://github.com/openclaw/openclaw/commit/b453293349657dedd069d291fa5b078d910067b1)
- **No Windows regression test covers this path:** `chrome.default-browser.test.ts` currently exercises only macOS default-browser detection cases; there is no Windows default-browser test guarding `extractWindowsExecutablePath()` behavior.
  - file: [extensions/browser/src/browser/chrome.default-browser.test.ts:36](https://github.com/openclaw/openclaw/blob/b453293349657dedd069d291fa5b078d910067b1/extensions/browser/src/browser/chrome.default-browser.test.ts#L36)

## Risks / Open Questions

- The PR needs rebasing because one of its two regex changes is already present on `main`.
- The bug may affect only unquoted Windows registry command strings; quoted command strings are already handled on `main`.
- There is still no targeted Windows regression test for this code path, so even a correct fix could regress again without added coverage.
