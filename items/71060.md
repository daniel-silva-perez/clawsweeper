## Summary

Keep open. Current main has strong hardening for normal macOS `gateway restart`, but the exact `stopLaunchAgent` behavior cited by this new report still exists: it waits 10 x 100ms and then falls back to `bootout`, leaving the LaunchAgent unloaded. Because the issue is recent, detailed, and claims an observed restart outage on 2026.4.22, this should not be auto-closed as already implemented without maintainer investigation into whether a restart path can still reach the stop/degraded bootout behavior.

## Best Possible Solution

Keep #71060 open and narrow the product decision. First verify whether `openclaw gateway restart` from a gateway-routed exec/session can still miss the launchd self-restart handoff or otherwise reach the stop/degraded bootout path. If the remaining issue is only `gateway stop`, decide whether degraded stop should re-bootstrap the disabled LaunchAgent, extend the stop wait to the gateway drain timeout, or keep the current hard-stop semantics but document the behavior. Add targeted launchd unit tests for busy drain >1s and for restart invoked from gateway-served command contexts.
