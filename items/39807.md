## Summary

Keep open. Current main still resolves explicit inline `models.providers.<id>.apiKey` credentials without a `profileId`, and the auth-profile billing disable/backoff path still requires an existing profile id/store entry before it records `disabledUntil`. The related PR #39836 proposed the right synthetic-profile direction but is closed unmerged and its behavior is not present in the checked-out main.

## Best Possible Solution

Keep #39807 open and implement a maintained version of the synthetic inline-profile/provider-level cooldown fix. The likely code path is `src/agents/model-auth.ts` for assigning a stable cooldown key to inline `models.providers.<id>.apiKey`, plus `src/agents/auth-profiles/usage.ts` or an equivalent provider-level tracker so billing failures persist `disabledUntil` and model fallback can skip the provider. Add regression coverage showing inline-key 402 billing failures are recorded and skipped on later runs while fallback providers remain eligible.
