## Summary

Keep #7679 open. Current main still defaults Telegram DMs to pairing, documents pairing as the default, validates allowlist mode only when allowFrom is populated, and issues pairing challenges to unknown DM senders. The setup flow supports manually entering numeric allowFrom IDs, but it does not implement token-verified owner ID capture during onboarding.

## Best Possible Solution

Keep #7679 open. The maintainer path should add a Telegram onboarding flow that generates a short verification token, asks the intended owner to send that token to the bot, captures that sender's numeric Telegram user ID, writes dmPolicy="allowlist" plus allowFrom for the correct account, and updates schema/runtime/docs/tests with a backcompat story for existing pairing-mode configurations.

## Risks / Open Questions

- A schema-only default change would be incomplete because allowlist mode needs a populated allowFrom list.
- Blindly capturing the first inbound Telegram message would preserve the race condition raised in the issue discussion; onboarding needs token or equivalent sender verification.
- Existing pairing-mode installs need an explicit compatibility path so users are not unexpectedly locked out.
