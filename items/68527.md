## Summary

Keep open. Current main has important MCP leak mitigations: session runtimes are reused per session/config, stale and in-flight runtimes are disposed, stdio MCP process trees are killed on close, and related #68660 was closed after those cleanup fixes. But this issue is broader and still has unimplemented core asks: no per-MCP failed-start backoff/circuit breaker, no `openclaw mcp set` startup preflight, no MCP child/RSS accounting in status, and no default `MemoryHigh`/`MemoryMax`/`TasksMax`/`OOMScoreAdjust` resource guardrails in the canonical systemd unit.

## Best Possible Solution

Keep this issue open as the canonical hardening follow-up. Build on the already-shipped cleanup from related #68660 by adding per-server MCP startup health state with exponential backoff/circuit-open behavior reset by config fingerprint changes, adding a safe `openclaw mcp set` preflight path or explicit opt-in mode, exposing MCP child/RSS diagnostics in status, and making a maintainer product decision on default Linux systemd resource controls versus documented operator overrides.

## Risks / Open Questions

- A misconfigured MCP server can still be retried by future new sessions or scheduled runs without a cross-session failed-start cooldown or circuit breaker keyed by server/config fingerprint.
- Concurrent agents can still launch failing MCP startup attempts at the same time, even though current cleanup should prevent the old orphan accumulation pattern.
- Default Linux installs still rely on operator tuning for memory/task ceilings; the canonical OpenClaw systemd unit does not bound gateway memory or task count by default.
