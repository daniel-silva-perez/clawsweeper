## Summary

Keep #72062 open. Current main has partial script-path handling for exec approvals, but it does not implement the requested per-agent wrapper/script-path deny policy. Tool allow/deny still filters by tool name, the typed config/schema has no `deny_wrappers` equivalent, and exec allowlist matching can still satisfy a wrapper call through an allowlisted shell executable before the script-path match is considered.

## Best Possible Solution

Implement a core-owned per-agent exec script-path deny policy, preferably under `agents.list[].tools.exec` or another config shape consistent with existing camelCase config naming. Reuse the existing shell wrapper script candidate resolver, apply the deny check before approval prompting and before executable allowlist success, make deny win over allowlist/full where intended, add tests for `bash`/`sh` allowlisted plus denied script paths, and document it on the public exec approvals / multi-agent tool-policy pages. A plugin hook can be a temporary local workaround, but it should not close this core security-policy request.
