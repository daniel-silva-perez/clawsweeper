## Summary

Keep open. Current `main` still appears to serialize Discord inbound messages by session key before they reach the steer path, so a second message for the same session cannot hit `runReplyAgent` while the first run is actively streaming. The only steer injection path still requires `shouldSteer && isStreaming`, and the fallback policy for an already-active run remains `enqueue-followup`. Main also contains a docs-only steering update rather than a channel-path fix, and the steering docs are still inconsistent across pages.

## Evidence

  - file: [extensions/discord/src/monitor/inbound-worker.ts:207](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/extensions/discord/src/monitor/inbound-worker.ts#L207)
