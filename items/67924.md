## Summary

Keep this PR open. Current main and the latest release still have the reported exec preflight false-positive behavior, and the PR is a plausible implementation candidate for the open bug it references. It should not be auto-closed as implemented or superseded, but it does need review changes because the proposed JS string/comment handling has an unresolved security-regression concern.

## Best Possible Solution

Keep this PR open and require a safer revision, or replace it with a maintainer-owned implementation that suppresses $VAR matches only in well-understood non-executable string/comment ranges while preserving rejection for payload = $DM_JSON-style code. Add regression tests for the reported Python docstring/f-string/help text cases, Node string-literal cases, and the review-comment counterexample where JS comments or regex literals must not hide a real shell-token leak. Once a corrected PR merges, #67621 can close through the PR's closing syntax.
