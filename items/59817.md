## Summary

Keep open. The report is concrete, recent, externally authored, and current main still appears to depend on Discord channel-info classification for DM routing. I did not find strong evidence that main now has the requested degraded-mode fallback or more aggressive canonicalization.

## Evidence

- **Issue provides concrete reproduction evidence:** The issue reports OpenClaw 2026.4.1 with dmPolicy=pairing and session.dmScope=per-channel-peer, showing the same Discord DM split between direct and channel session keys during Discord gateway/network failures. The sole comment also treats it as a Discord provider session-management bug, though its workaround is not source-backed enough to close the issue.
- **Current main still gates DM classification on channel info:** In current main, Discord preflight resolves channelInfo, then sets isDirectMessage only with channelInfo?.type === ChannelType.DM and isGroupDm only with channelInfo?.type === ChannelType.GroupDM. That is the brittle path described in the issue when channel-info lookup returns null/undefined under network failure.
  - file: [extensions/discord/src/monitor/message-handler.preflight.ts](https://github.com/openclaw/openclaw/blob/d6ef1fcf24eea3a5985d653a3baf400e32aa1b75/extensions/discord/src/monitor/message-handler.preflight.ts)
  - command: `web/raw inspection of current main`
  - sha: [d6ef1fcf24ee](https://github.com/openclaw/openclaw/commit/d6ef1fcf24eea3a5985d653a3baf400e32aa1b75)
- **Routing still receives the possibly false DM flags:** The same preflight path passes isDirectMessage/isGroupDm into buildDiscordRoutePeer with directUserId=author.id and conversationId=messageChannelId. If channelInfo is missing, this path has no visible guild_id-absent fallback before route resolution, so the issue's channel-session fallback remains plausible.
  - file: [extensions/discord/src/monitor/message-handler.preflight.ts](https://github.com/openclaw/openclaw/blob/d6ef1fcf24eea3a5985d653a3baf400e32aa1b75/extensions/discord/src/monitor/message-handler.preflight.ts)
  - command: `web/raw inspection of current main`
  - sha: [d6ef1fcf24ee](https://github.com/openclaw/openclaw/commit/d6ef1fcf24eea3a5985d653a3baf400e32aa1b75)
- **Explicit session-key normalization remains direct-context limited:** normalizeExplicitDiscordSessionKey only rewrites discord:channel:<id> to discord:direct:<id> when ctx.ChatType normalizes to direct, matching the issue's observation that normalization cannot repair a message already classified as channel.
  - file: [extensions/discord/src/session-key-normalization.ts:371](https://github.com/openclaw/openclaw/blob/d6ef1fcf24eea3a5985d653a3baf400e32aa1b75/extensions/discord/src/session-key-normalization.ts#L371)
