## Summary

Keep open. Current main still classifies tool-result failure only from `details.status === "error" | "timeout"`, while bundled MCP materialization only writes `details.status` for `isError === true`. That means the PR's reported status-less MCP content-error path is not implemented on main. The PR discussion also shows a real maintainer-review question around the exact heuristic for `Error:` text versus false positives, so the best cleanup path is review/revise/land or explicitly reject this PR, not auto-close it.

## Best Possible Solution

Keep the PR open for maintainer review. The likely path is to decide the product contract for MCP results that omit or contradict `isError`, then either land a revised classifier/materialization change with focused `isToolResultError` and handler tests, or close the PR after a maintainer explicitly decides OpenClaw should trust MCP `isError` strictly and not add a content-prefix fallback.
