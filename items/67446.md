## Summary

Keep open. Current main still classifies tool-result failures only from `details.status`, while bundled MCP materialization only writes `details.status` for `isError === true`, so the status-less MCP content-error path is not implemented. The PR is focused and does not introduce supply-chain-sensitive changes, but its `Error:` heuristic and later `isError: false` handling still need maintainer product judgment before merge or rejection.

## Best Possible Solution

Keep this PR open for maintainer review. Decide the MCP error contract, then either land a revised implementation in `toAgentToolResult` and/or `isToolResultError` with focused tests for status-based errors, status-less MCP content errors, `after_tool_call` error routing, and false-positive cases, or close the PR after explicitly deciding OpenClaw should trust MCP `isError` strictly.
