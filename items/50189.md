## Summary

Keep open. Current `main` does not implement the requested normalization, and the proposed `delivered=true => ok` rule is too broad for OpenClaw's current cron semantics because fatal error payloads can still be delivered successfully.

## Evidence

- **Current main still persists raw cron run status:** `applyJobResult()` writes `result.status` directly into `lastRunStatus`/`lastStatus`, derives `lastErrorReason` from raw `result.status`, and uses raw status for consecutive-error tracking, one-shot disable/delete, and recurring backoff branches. The requested behavior is not present on `main`.
  - file: [src/cron/service/timer.ts:418](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/src/cron/service/timer.ts#L418)
  - command: `nl -ba src/cron/service/timer.ts | sed -n '380,520p'`
  - sha: [45146913007d](https://github.com/openclaw/openclaw/commit/45146913007da723b4f06aba6a66cf4a695974c0)
- **Delivery is already modeled separately from execution outcome:** `resolveDeliveryState()` records `lastDeliveryStatus` from `delivered`, independent of run status. `CronJobState` also keeps `lastRunStatus` and `lastDeliveryStatus` as separate fields, so a safe fix has to preserve that distinction rather than treating delivery as a blanket success signal.
  - file: [src/cron/service/timer.ts:257](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/src/cron/service/timer.ts#L257)
  - command: `nl -ba src/cron/service/timer.ts | sed -n '248,278p'`
  - sha: [45146913007d](https://github.com/openclaw/openclaw/commit/45146913007da723b4f06aba6a66cf4a695974c0)
- **Delivered payloads can still be fatal errors:** `pickLastDeliverablePayload()` falls back to an error payload when no non-error deliverable payload exists, so `delivered=true` does not necessarily mean the cron job succeeded in the user-facing sense.
  - file: [src/cron/isolated-agent/helpers.ts:96](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/src/cron/isolated-agent/helpers.ts#L96)
  - command: `nl -ba src/cron/isolated-agent/helpers.ts | sed -n '1,140p'`
  - sha: [45146913007d](https://github.com/openclaw/openclaw/commit/45146913007da723b4f06aba6a66cf4a695974c0)
- **Isolated cron runs intentionally keep fatal payloads as status=error:** `resolveRunOutcome()` returns `status: "error"` whenever `hasFatalErrorPayload` is true, while still threading through delivery metadata. That makes the PR's proposed `result.status === "error" && result.delivered === true ? "ok" : result.status` rewrite semantically unsafe on current main.
  - file: [src/cron/isolated-agent/run.ts:822](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/src/cron/isolated-agent/run.ts#L822)
  - command: `nl -ba src/cron/isolated-agent/run.ts | sed -n '820,940p'`
  - sha: [45146913007d](https://github.com/openclaw/openclaw/commit/45146913007da723b4f06aba6a66cf4a695974c0)
- **User-visible cron notifications still key off event status:** Finished events emit raw `result.status`, and gateway cron failure notifications trigger on `evt.status === "error"`. A fix limited to persisted job state would leave other user-visible paths inconsistent.
  - file: [src/cron/service/timer.ts:1388](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/src/cron/service/timer.ts#L1388)

## Risks / Open Questions

- Closing this PR would discard a still-relevant bug report area: `main` still records raw error status even when delivery metadata is tracked separately.
- Merging the PR as described would risk masking genuine fatal isolated-agent failures whenever the only delivered payload is itself an error payload.
- A state-only normalization would likely leave finished-event/webhook/failure-destination behavior inconsistent with persisted cron state.
