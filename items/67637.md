## Summary

Keep open. Current main still does not expose tool-call traces on QaBusMessage or through the QA bus message lifecycle. The mock providers and several scenarios now use /debug/requests with plannedToolName/plannedToolArgs as adjacent evidence, but that is mock-provider scoped and not the requested generalized QaBusMessage.toolCalls surface populated from actual agent/tool execution logs. The referenced parity-scenario PR #67540 is closed unmerged, so it is not a current-main implementation or a reason to close this issue.

## Best Possible Solution

Keep this issue open. If maintainers want generalized scenario assertions, implement a backwards-compatible QA trace surface: either an optional structured tool-call trace on QaBusMessage or a dedicated scenario assertion API, populated from actual agent/tool execution logs rather than only mock-provider planning, carried through create/clone/snapshot/read/search/poll paths, and covered by focused qa-channel/qa-lab tests plus at least one scenario that fails when the agent fabricates success without calling the expected tool.
