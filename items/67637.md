## Summary

Keep open. Current main still does not expose tool-call traces on `QaBusMessage`; the QA bus protocol, message creation, and cloning/snapshot paths only carry text, attachments, reactions, and message metadata. There is useful adjacent mock-provider evidence via `/debug/requests` and `plannedToolName`, and current scenarios use that for some parity assertions, but that is not the requested generalized `QaBusMessage.toolCalls` surface populated from agent/tool execution logs.

## Best Possible Solution

Keep this issue open and implement a backwards-compatible QA trace surface if maintainers want generalized scenario assertions: add an optional structured tool-call trace to the QA bus/runtime contract or a dedicated scenario assertion helper, populate it from actual agent/tool execution logs rather than only mock-provider request planning, clone/poll/search/read it through qa-lab, and add focused qa-channel/qa-lab tests plus one scenario that proves text-only hallucinated success no longer passes.
