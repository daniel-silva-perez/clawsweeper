## Summary

Keep open. Current main still loads dotenv values literally: `.env` files are parsed with `dotenv.parse`, normalized, and then assigned directly to `process.env` without an expansion pass for `$VAR` or `${VAR}` aliases. The related PR #38254 is closed unmerged in the provided GitHub context, so it does not supersede the issue.

## Best Possible Solution

Implement dotenv-level expansion in `src/infra/dotenv.ts` before assigning values to `process.env`, supporting `${VAR}` and `$VAR` aliases from the existing process environment plus earlier loaded dotenv entries while preserving current precedence and workspace security blocklists. Add focused regression tests in `src/infra/dotenv.test.ts` for shell-backed aliases, same-file aliases, chained references, missing references, workspace `.env`, global/state-dir fallback, and gateway.env fallback behavior. Reuse or re-review PR #38254 as prior work rather than closing this issue.
