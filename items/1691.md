## Summary

Current `main` already covers this request: proxy-like OpenAI-compatible Responses routes strip `prompt_cache_key`/`prompt_cache_retention` by default, and there is a model-level `compat.supportsPromptCacheKey` override in the config surface.

## Close Comment

Closing as `implemented_on_main` after a ClawSweeper/Codex review. Current `main` (`178a314a4c1cfa98ac0e8545707162427e38f9c5`) already handles this in two ways: proxy-style OpenAI-compatible Responses routes strip prompt-cache hints by default (`src/agents/provider-attribution.ts:632`, `src/agents/openai-responses-payload-policy.ts:138`), and there is a model-level compat override at `models.providers.<provider>.models[].compat.supportsPromptCacheKey` (`src/config/types.models.ts:37`, `src/config/zod-schema.core.ts:187`). The runtime path is covered by tests (`src/agents/provider-attribution.test.ts:623`, `src/agents/pi-embedded-runner-extraparams.test.ts:3356`), and the docs now state that proxy-style routes, including vLLM, do not send prompt-cache hints (`docs/concepts/model-providers.md:612`, `docs/providers/vllm.md:115`). The separate vLLM 500 mentioned in the thread may still have been another compatibility issue, but the specific `prompt_cache_key` opt-out request is already addressed on current `main`.
