## Summary

Keep open. Current main still contains the reported channel stop-timeout behavior: timeout leaves `running: true`, skips `store.aborts`/`store.tasks` cleanup, and the existing test still asserts that a restart attempt is blocked after timeout. The linked fix PR #70056 is open and unmerged, so the issue should remain open until that PR or an equivalent fix lands.

## Best Possible Solution

Review and land PR #70056 or an equivalent gateway fix: on stop timeout, clear the retained abort/task entries, mark the account `running: false`, set an appropriate stop timestamp/error, and update `server-channels.test.ts` so a subsequent `startChannel` succeeds after timeout. Close #70024 from the merged fix.
