## Summary

Keep open. Current `main` does not implement the requested channel-registration suppression, and the PR as described has unresolved correctness risks: it would suppress warnings based on `channels` ownership rather than plugin identity, and the reviewed implementation can silently drop later candidates from the manifest registry.

## Evidence

- **Current main only suppresses duplicates for the same physical plugin:** `loadPluginManifestRegistry` skips duplicate warnings only when two candidates share the same `rootDir` or resolve to the same real path. There is no channel-registration exception on current `main`.
  - file: [src/plugins/manifest-registry.ts:625](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/src/plugins/manifest-registry.ts#L625)
  - command: `nl -ba src/plugins/manifest-registry.ts | sed -n '625,647p'`
  - sha: [f03252aaf9cc](https://github.com/openclaw/openclaw/commit/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022)
- **Distinct duplicate ids are still expected to warn and collapse to one winner:** The current tests assert that two different plugin roots with the same id produce one duplicate warning and only one surviving registry entry.
  - file: [src/plugins/manifest-registry.test.ts:313](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/src/plugins/manifest-registry.test.ts#L313)
  - command: `nl -ba src/plugins/manifest-registry.test.ts | sed -n '313,369p'`
  - sha: [f03252aaf9cc](https://github.com/openclaw/openclaw/commit/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022)
- **Channel plugins normally use plugin id == channel id:** The manifest contract says `channels` are ids owned by the plugin, and bundled channel manifests like Telegram use `id: "telegram"` with `channels: ["telegram"]`. That means a blanket 'declared channel matches plugin id' suppression would also match genuine collisions with a distinct plugin using that same id.
  - file: [extensions/telegram/openclaw.plugin.json:2](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/extensions/telegram/openclaw.plugin.json#L2)
  - command: `nl -ba extensions/telegram/openclaw.plugin.json | sed -n '1,12p'`
  - sha: [f03252aaf9cc](https://github.com/openclaw/openclaw/commit/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022)
- **Loader skips candidates absent from manifest registry:** `loadOpenClawPlugins` builds `manifestByRoot` from `manifestRegistry.plugins` and then ignores any discovered candidate whose root is missing from that map. If a duplicate path is suppressed via an early `continue` without preserving a manifest record, that candidate disappears from later duplicate/override handling.
  - file: [src/plugins/loader.ts:2035](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/src/plugins/loader.ts#L2035)
  - command: `nl -ba src/plugins/loader.ts | sed -n '2035,2058p'`
  - sha: [f03252aaf9cc](https://github.com/openclaw/openclaw/commit/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022)
- **PR review threads identify unresolved logic flaws:** The provided review context calls out two unresolved problems on PR #45900: the channel-registration check is order-dependent, and the suppression path skips `records.push(...)`, silently dropping the later candidate from the registry.
  - sha: [02b9fed05d99](https://github.com/openclaw/openclaw/commit/02b9fed05d999780217733509c5a12ee6bed9d82)
