## Summary

Keep open for maintainer/security review. The PR is paired with open issue #72418 and has closing syntax, but its final diff appears to add `bootstrap-token` to the backend self-pairing bypass while leaving the reported shared-token/password bypass intact. Current docs also describe the shared-token backend exception as intentional, so the right outcome needs explicit maintainer security judgment rather than cleanup closure.

## Best Possible Solution

Keep this PR open. Maintainers should either revise it into a real fix for #72418, likely by removing or tightly narrowing the shared-token/password backend skip and adding regression coverage for a spoofed `gateway-client`/`backend` loopback connection with no registered device, or explicitly reject the linked report with a documented security rationale if the shared-token backend exception is intentionally accepted.
