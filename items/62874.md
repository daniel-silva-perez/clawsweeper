## Summary

Keep open. Current main has a released per-model workaround path via `params.extra_body` for OpenAI-compatible proxies, but it does not resolve the reported default `tool_choice: "auto"` behavior for vLLM + Qwen. The vLLM/OpenAI-compatible path still defaults tool requests to `auto`, and Qwen-style XML tool-call text is only stripped from visible assistant text rather than converted into executable tool calls.

## Best Possible Solution

Keep the issue open and handle it as a vLLM/Qwen compatibility task. The low-risk current path is to document the per-model workaround, for example `agents.defaults.models["vllm/<model-id>"].params.extra_body: { tool_choice: "required" }`, on the vLLM provider page. A stronger product fix would be a provider-owned vLLM/Qwen compatibility flag or model-family heuristic that can force `required` only for affected models after live validation, without changing the default OpenAI-compatible behavior for all proxy models.
