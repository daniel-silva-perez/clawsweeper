## Summary

Keep open. Current main still matches the reported vLLM/Qwen failure mode: proxy-like OpenAI-compatible completions with tools default to `tool_choice: "auto"`, the vLLM plugin does not add a Qwen-specific tool-choice compatibility policy, and the generic `extra_body` override is only a workaround rather than a scoped fix or documented vLLM/Qwen behavior.

## Best Possible Solution

Keep this issue open for maintainer work. The likely path is a vLLM/Qwen-scoped compatibility option, first-class per-model `params.toolChoice` forwarding, or explicit vLLM docs showing `agents.defaults.models["vllm/<model>"].params.extra_body: { tool_choice: "required" }`, with regression coverage proving the override reaches the local vLLM OpenAI-compatible payload without changing native OpenAI behavior globally.

## Risks / Open Questions

- Changing the global proxy/local default from `auto` to `required` could force tool use on turns where a model should answer normally, so a fix should be scoped to vLLM/Qwen, made explicit per model, or documented as an opt-in compatibility setting.
- No live vLLM plus Qwen server was exercised in this read-only review; implementation should include payload-builder regression coverage and ideally a captured or live vLLM verification path.
- The generic `extra_body` escape hatch can unblock affected users, but relying on it silently leaves discoverability and per-model policy unclear.
