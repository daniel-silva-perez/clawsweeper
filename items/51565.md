## Summary

Keep #51565 open. Current main still uses only `channels.whatsapp.ackReaction.emoji` for WhatsApp ack reactions, so partial WhatsApp ack configs cannot fall back to the responding agent identity emoji. The PR addresses a real remaining behavior gap, though it still needs normal review cleanup before merge.

## Best Possible Solution

Keep the PR open and finish it as a focused WhatsApp behavior fix: add a WhatsApp-specific ack emoji resolver that resolves account emoji, channel emoji, then responding agent identity emoji while skipping `messages.ackReaction`; wire `maybeSendAckReaction` through that helper; keep existing no-config and explicit-empty disable behavior; and update tests without lint suppressions.
