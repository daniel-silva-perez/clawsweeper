## Summary

Keep open. Current main does not implement the PR's central remaining behavior: a numeric Telegram chat ID stored in `delivery.channel` is still treated as the requested channel, while the PR discussion and linked #70758 identify that stored/misfiled shape as part of the bug. The PR also has an unresolved security-review concern about inferring `to` from `delivery.channel`, so it needs maintainer review rather than cleanup closure.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is a narrow cron delivery fix that accepts numeric Telegram IDs via `delivery.to` and deliberately handles or rejects legacy/misfiled `delivery.channel` values without bypassing allowlist semantics, with regression coverage in `src/cron/isolated-agent/delivery-target.test.ts`. The unrelated macOS formatting suppressions should be separated or justified before landing.
