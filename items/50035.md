## Summary

Current main no longer uses a launchctl kickstart/bootstrap handoff for config-triggered gateway restarts on macOS launchd. `config.patch`/`config.apply` now schedule a SIGUSR1 restart, the gateway run loop treats launchd as supervised, and the respawn path exits cleanly so launchd `KeepAlive` performs the restart instead of calling `kickstart -k` against a process that just died.

## Risks / Open Questions

- The fix is present on current `main` but I could not tie it to a tagged release newer than `v2026.4.22`, so affected released builds may still need an upgrade once a newer release ships.
- `src/infra/restart.ts` still contains launchctl `kickstart` logic for explicit restart paths, but the config-triggered macOS launchd flow described in this issue no longer goes through that code path.

## Close Comment

Closing from a Codex review because current `main` already avoids the reported macOS launchd race for config-triggered restarts. `config.patch` / `config.apply` now schedule a `SIGUSR1` restart (`src/gateway/server-methods/config.ts:592`), the gateway run loop handles that restart internally (`src/cli/gateway-cli/run-loop.ts:230`), and the macOS launchd respawn path exits cleanly with `{ mode: "supervised" }` so launchd `KeepAlive` performs the relaunch instead of calling `launchctl kickstart -k` against a process that just terminated (`src/infra/process-respawn.ts:30`). The regression tests also pin this behavior, including explicit assertions that the launchd path does not call `triggerOpenClawRestart()` (`src/infra/process-respawn.test.ts:88`). This behavior is present on `main` via commit `512d3d2287d509f75a32a2344f1b71b8fbfdb0c1`; I could not verify a released tag that already includes it, so `fixedRelease` is left unset.
