## Summary

Keep open. Current main has partial adjacent fixes for startup isolation, readiness probes, Telegram request timeouts, and Feishu sequential preflight, but it does not fully resolve this issue's central startup UX/recovery request: the generic `ready (...)` log still precedes channel/sidecar startup, there is no per-channel recovery timing summary, Telegram polling still waits/retries `deleteWebhook` before polling can begin, and the Feishu gateway per-account path can still run startup identity probes concurrently. The issue is recent, concrete, and not superseded by the closed adjacent startup issues.

## Best Possible Solution

Keep #70728 open as the combined Windows startup/channel-recovery UX issue unless maintainers create a narrower canonical replacement. The best path is to clarify the `ready (...)` log as gateway-listening readiness, add a deterministic post-sidecar/per-channel recovery summary, keep degraded channel recovery fault-isolated from core gateway usability, and finish channel-owned Telegram `deleteWebhook` and Feishu bot-identity startup hardening with regression coverage for the actual gateway paths.
