## Summary

Keep open. Current main still has two macOS LaunchAgent tracks: `ai.openclaw.mac` for the app login item and `ai.openclaw.gateway` for the gateway service. The app does try to attach to an existing gateway before enabling launchd, and Syu0’s workaround comment correctly points users away from terminal `openclaw doctor --fix`, but the CLI doctor/install repair paths still do not guard on the Mac app LaunchAgent before installing or repairing `ai.openclaw.gateway`. The related closed-unmerged #33348 and prior #42573 review reinforce that launchd ownership/classification remains a live product boundary rather than a closeable duplicate or shipped fix.

## Best Possible Solution

Keep this open and make the macOS launchd ownership boundary explicit. The conservative implementation path is for CLI doctor/gateway repair to detect the Mac app LaunchAgent (`ai.openclaw.mac`) and either skip reinstalling `ai.openclaw.gateway` with a clear app-specific note or coordinate through one canonical owner. Add regression coverage for doctor repair and launchd scanning, preserve the existing attach-existing behavior, and document when terminal `openclaw doctor --fix` is safe for Mac app users.
