## Summary

Keep this issue open. Current main has mitigations for cached device-token reuse and the local backend native-approval client path, plus a documented Telegram native-approval opt-out, but the CLI-mode path still defaults to the full operator scope set and still attaches a device identity for shared-token CLI calls. That means the reported CLI scope-upgrade/pending-approval behavior remains plausible on current main and should not be closed as implemented.

## Best Possible Solution

Keep the issue open and fix the CLI connection policy directly: route read-only CLI commands through least-privilege scopes, avoid attaching device identity for trusted local shared-token CLI probes where pairing is not needed, or make the CLI prefer an approved cached operator device token/scopes before requesting the full admin scope set. Preserve the current Telegram `execApprovals.enabled=false` opt-out and backend native-approval local self-call behavior, and add regression coverage for a paired device approved only for `operator.approvals`/`operator.read` running representative CLI commands without creating a full-scope pending upgrade.

## Risks / Open Questions

- A blanket server-side silent approval for CLI scope upgrades would weaken the pairing model; the better fix is to make CLI calls request least-privilege scopes per method or reliably reuse the approved stored device-token scopes when appropriate.
- The Telegram native-approval half appears partly mitigated on current main, so a fix should avoid regressing the backend `gateway-client` local self-call path while addressing the distinct CLI-mode behavior.
