## Summary

Keep #71429 open. Current main has related restart/reload hardening and durable queues for routed outbound/restart-continuation paths, but the normal same-surface Telegram auto-reply path still sends through the in-process dispatcher to Telegram Bot API delivery without a write-ahead durable queue. Typing/sendChatAction failures are logged/suppressed through typing guards, and final Telegram delivery failures are recorded as dispatcher delivery failures rather than converted into a visible hard turn failure with retry guidance. The maintainer discussion on the issue already called out this exact boundary and left the item open; current main at 6b6dcafcee9ca86385b7c1ee2442752314d50257 does not change that conclusion.

## Best Possible Solution

Keep #71429 open. Add a focused Telegram auto-reply/hot-reload regression that simulates a live turn, sendChatAction failure, and final Bot API delivery failure. Then either route normal Telegram final replies through an existing durable outbound/session-delivery queue before Bot API delivery, or deliberately fail the turn with visible Telegram/UI status and retry guidance.
