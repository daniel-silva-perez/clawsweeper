## Summary

Keep #69633 open. Current main does not implement a WebView2 `window.chrome.webview` bridge in the Control UI, and the linked openclaw-windows-node host-side PR is related but does not supersede the SPA-side work. Windows companion apps are explicitly planned in the project docs, so this needs maintainer product/API review and tests rather than conservative cleanup closure.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is to decide the Windows companion app bridge contract, then either finish this PR or replace it with a tested implementation integrated into the existing Control UI lifecycle and chat state. If maintainers want this to live outside core, the prerequisite would be a concrete UI/host extension API; current plugin surfaces do not obviously cover injecting WebView2 browser-global handling into the shipped SPA.
