## Summary

Current `main` only supports Slack `[[reply_to_current]]` when Slack reply threading is otherwise enabled. The requested behavior in #50031 is not fully implemented because Slack explicitly disables reply-tag forcing when `channels.slack.replyToMode` is `off` (the documented default), so the tag can still be stripped without creating a threaded reply.

## Evidence

- **Slack plugin deliberately disables explicit reply tags in off mode:** The Slack channel plugin sets `allowExplicitReplyTagsWhenOff: false`, overriding the generic reply-tag behavior used by other channels.
  - file: [extensions/slack/src/channel.ts:608](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/extensions/slack/src/channel.ts#L608)
  - command: `nl -ba extensions/slack/src/channel.ts | sed -n '603,620p'`
  - sha: [45146913007d](https://github.com/openclaw/openclaw/commit/45146913007da723b4f06aba6a66cf4a695974c0)
- **Delivery path ignores inline replyToId when Slack replyToMode is off:** Slack reply delivery drops `payload.replyToId` whenever `replyToMode === "off"`, so `[[reply_to_current]]` cannot force `thread_ts` in that mode.
  - file: [extensions/slack/src/monitor/replies.ts:39](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/extensions/slack/src/monitor/replies.ts#L39)
  - command: `nl -ba extensions/slack/src/monitor/replies.ts | sed -n '36,41p'`
  - sha: [45146913007d](https://github.com/openclaw/openclaw/commit/45146913007da723b4f06aba6a66cf4a695974c0)
- **Regression test confirms explicit replyToId is ignored in off mode:** Slack monitor tests assert that a forced reply id is ignored when `replyToMode` is `off`, and only threads when mode is `all`.
  - file: [extensions/slack/src/monitor.tool-result.test.ts:539](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/extensions/slack/src/monitor.tool-result.test.ts#L539)
  - command: `nl -ba extensions/slack/src/monitor.tool-result.test.ts | sed -n '539,577p'`
  - sha: [45146913007d](https://github.com/openclaw/openclaw/commit/45146913007da723b4f06aba6a66cf4a695974c0)
- **Docs document the same limitation:** Slack docs say manual reply tags are supported, but explicitly note that `replyToMode="off"` disables all Slack reply threading, including `[[reply_to_*]]` tags.
  - file: [docs/channels/slack.md:528](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/docs/channels/slack.md#L528)
  - command: `nl -ba docs/channels/slack.md | sed -n '528,533p'`
  - sha: [45146913007d](https://github.com/openclaw/openclaw/commit/45146913007da723b4f06aba6a66cf4a695974c0)

## Risks / Open Questions

- If maintainers consider 'works when Slack replyToMode is enabled' sufficient, they may treat this as already addressed. The issue text, though, asks for explicit `[[reply_to_current]]` support itself, and current `main` still intentionally blocks that in the default off-mode path.
