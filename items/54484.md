## Summary

Current `main` does not expose Telegram inbound local media paths in the visible message body. The path is still carried only in structured `MediaPath`/`MediaPaths` context fields, while body rendering stays on `<media:...>` placeholders or plain caption text. This issue should remain open.

## Evidence

- **Primary media rendering still resolves to placeholders only:** `resolveTelegramPrimaryMedia` maps Telegram photos, videos, audio, documents, and stickers to placeholder strings like `<media:image>` and `<media:document>`; it does not inject saved local file paths into the visible body.
  - file: [extensions/telegram/src/bot/body-helpers.ts:34](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/extensions/telegram/src/bot/body-helpers.ts#L34)
  - command: `nl -ba extensions/telegram/src/bot/body-helpers.ts | sed -n '30,64p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Visible body builder does not add local paths:** `resolveTelegramInboundBody` builds `rawBody` from caption/text/location, falls back to the placeholder, and only prepends a placeholder + `file_id` when `allMedia.length === 0`. When media was downloaded, no local path is appended to `bodyText`; for non-audio media with no text it even falls back to `<media:image>` generically.
  - file: [extensions/telegram/src/bot-message-context.body.ts:178](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/extensions/telegram/src/bot-message-context.body.ts#L178)
  - command: `nl -ba extensions/telegram/src/bot-message-context.body.ts | sed -n '178,239p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Local paths are still only in structured context fields:** The session payload sends `Body`/`BodyForAgent` separately from `MediaPath`/`MediaPaths`. Saved inbound file paths are attached to `MediaPath`, `MediaUrl`, `MediaPaths`, and `MediaUrls`, not merged into the visible body fields.
  - file: [extensions/telegram/src/bot-message-context.session.ts:322](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/extensions/telegram/src/bot-message-context.session.ts#L322)
  - command: `nl -ba extensions/telegram/src/bot-message-context.session.ts | sed -n '322,370p'`

- **Current tests assert placeholder-in-body and path-in-MediaPaths behavior:** The Telegram inbound media e2e test on current `main` expects `payload.Body` to contain `<media:image>` while the downloaded saved path is asserted only in `payload.MediaPaths`. That matches the reported bug rather than a fix.
