## Summary

Keep this PR open. Current main still treats global cron failure alerts as opt-in, and the public config docs still say `cron.failureAlert.enabled` defaults to `false`. The PR is a focused implementation candidate for changing that behavior, not obsolete or already implemented, but it needs maintainer product judgment and docs/config alignment before merge.

## Best Possible Solution

Review the behavioral default change on product grounds. If maintainers agree recurring cron failure alerts should be default-on, land a variant that preserves `failureAlert: false` and per-job override semantics, keeps the default-unset regression test, updates public config docs/generated config help as needed, and runs the changed cron validation lanes. If maintainers want opt-in behavior, close the PR with that rationale and consider adding a guardrail test or docs clarification for the existing default.
