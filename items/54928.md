## Summary

Keep this issue open. Current main still does not support `plugins.entries.exa.config.webSearch.baseUrl`: the Exa manifest schema only permits `apiKey`, runtime requests still use the fixed Exa endpoint, docs only document `apiKey`, and open PR #54939 is the implementation candidate that should be reviewed before this issue closes.

## Best Possible Solution

Keep this issue open and review PR #54939 as the implementation candidate. The likely fix belongs in the Exa plugin: add `webSearch.baseUrl` to `extensions/exa/openclaw.plugin.json` and UI hints, plumb it through the Exa runtime request path, include the resolved endpoint in cache-key behavior, document it on `https://docs.openclaw.ai/tools/exa-search` and `https://docs.openclaw.ai/tools/web`, and add Exa plugin coverage proving config validation accepts `baseUrl` and runtime requests use it. If maintainers decide Exa should intentionally not support custom endpoints, document the capability matrix and return provider-specific validation guidance instead, as suggested in the prior discussion.
