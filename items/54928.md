## Summary

Keep open. Current main still does not support `plugins.entries.exa.config.webSearch.baseUrl`: the Exa manifest schema is strict and only declares `apiKey`, the runtime posts to the fixed Exa endpoint, and the docs still document only the API key. The linked PR #54939 directly tracks the requested fix, but the provided context shows it is open and unmerged, so the issue remains a valid tracker until that work lands or is superseded by another merged patch.

## Best Possible Solution

Keep the issue open until the Exa provider actually ships `baseUrl` support or an explicit product decision documents that it is unsupported. The likely fix belongs in the Exa plugin: add `webSearch.baseUrl` to `extensions/exa/openclaw.plugin.json` and UI hints, resolve and normalize it in `extensions/exa/src/exa-web-search-provider.runtime.ts`, pass it into the request endpoint, add focused Exa tests for configured endpoint usage and default fallback, and update `docs/tools/exa-search.md` / web-search docs. If maintainers choose not to support it, add provider-aware schema/docs guidance so users get a clear Exa-specific error instead of a generic additional-properties failure.
