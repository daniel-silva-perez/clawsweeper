## Summary

Keep this issue open. Current main has partial mitigations for Active Memory, including a hard timeout deadline and toolsAllow forwarding, but the central reported behavior remains: eligible direct-chat turns still await Active Memory in the foreground before prompt build and can block until the configured timeout fires. Fresh issue discussion also reports that `openclaw memory index --force` can help but Active Memory remains slow, so the report is not obsolete.

## Best Possible Solution

Keep this issue open as the canonical Active Memory direct-chat latency/reliability tracker unless a maintainer intentionally splits it. A real fix should make Active Memory safe for primary chat by adding a non-blocking/background recall mode, a much smaller foreground fail-open budget with optional late memory update, or a prewarm/readiness/indexing gate so cold memory/index work is not paid inside the first user turn. Preserve the existing hard-deadline and toolsAllow mitigations, and document or fix the reindexing requirement if `openclaw memory index --force` is part of the operational answer.
