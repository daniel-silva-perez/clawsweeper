## Summary

Current main does not implement automatic forwarding of original `agent.request` attachments. The existing Gateway path parses attachments into agent image input, then delivery sends only the agent-produced reply payloads. A direct `send` RPC can deliver media when a caller supplies `mediaUrl`/`mediaUrls`, but that is a separate workflow and does not satisfy the requested `agent.request` auto-forward flag/fallback.

## Best Possible Solution

Keep this issue open. The best path is an additive Gateway protocol change for `agent.request` that preserves sanitized/original attachment delivery references when explicitly requested, then injects them into outbound reply payloads only under well-defined conditions such as `deliver: true`, no agent-supplied media, a deliverable target, and channel media support. Update `src/gateway/protocol/schema/agent.ts`, both `src/gateway/server-methods/agent.ts` and `src/gateway/server-node-events.ts`, `src/agents/command/delivery.ts` or a narrower outbound seam, plus protocol/docs and regression tests for text-only replies, no-reply runs, and agent-media override behavior.

## Risks / Open Questions

- This is a real core Gateway/protocol and outbound-delivery behavior change, not just a ClawHub plugin idea, because current `agent.request` drops the original attachment references before delivery.
- A maintainer should decide the security/product semantics before implementation: sender-controlled automatic media fanout may need an explicit opt-in flag, size/type limits, and clear precedence when the agent also returns media.
