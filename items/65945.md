## Summary

Current main still does not implement automatic forwarding of original Node `agent.request` attachments when `deliver: true`. The gateway accepts and parses attachments into agent image input, but the request path does not retain a delivery-ready attachment reference or auto-forward flag, and final delivery still sends only agent-produced reply payloads. Direct media sending and QQ media delivery exist, but they are separate paths and do not satisfy this requested fallback/opt-in behavior.

## Best Possible Solution

Keep this issue open. The best path is an additive Gateway protocol change for `agent.request`/`agent` that preserves sanitized delivery-ready attachment references only when explicitly requested, then injects them into outbound reply payloads under narrow conditions such as `deliver: true`, a resolved deliverable target, no agent-supplied media, and channel media support. Update `src/gateway/protocol/schema/agent.ts`, `src/gateway/server-node-events.ts`, `src/gateway/server-methods/agent.ts`, `src/agents/command/types.ts`, `src/agents/command/delivery.ts` or a narrower outbound seam, plus docs and regression tests for text-only replies, no-reply runs, agent-media override, and unsupported-channel behavior.

## Risks / Open Questions

- Automatic sender-controlled media fanout has security and product semantics that need maintainer design: explicit opt-in, size/type limits, target validation, and clear precedence when the agent also returns media.
- A plugin alone cannot implement this cleanly today because the node/gateway path consumes attachment data before outbound delivery sees it.
