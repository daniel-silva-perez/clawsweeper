## Summary

Keep open. Current main has adjacent observability support through diagnostics-otel, model-call events, and W3C traceparent propagation, but it does not implement the PR's requested configurable per-run/session provider headers (`diagnostics.sessionTracing`, `x-session-id`, `x-session-name`, or custom header names). Existing provider/static header surfaces are useful context, but they do not fully cover a global dynamic run/session header feature across LLM requests, so this still needs maintainer product/API judgment rather than cleanup closure.

## Best Possible Solution

Keep the PR open for maintainer product/API judgment. The best path is either to adapt the work into the current diagnostics/provider architecture with documented privacy semantics, or to add a narrower plugin/API extension that lets an external observability plugin inject dynamic run/session headers without hardcoding a one-off core config surface.

## Risks / Open Questions

- Adding a new core `diagnostics.sessionTracing` block may duplicate or conflict with the existing OpenTelemetry diagnostics model and provider plugin header APIs.
- The proposed fallback of sending session keys as headers can expose channel/user routing identifiers to third-party observability dashboards, so it needs explicit privacy/security review and documentation if accepted.
- Because main now has model-call diagnostics, traceparent propagation, and provider transport identity hooks, the PR likely needs a rebase and design pass to fit the current wrapper ordering and plugin boundary.
