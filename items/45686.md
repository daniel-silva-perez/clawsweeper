## Summary

Keep open. Current `main` has partial mitigations for compaction failures and timeout fallout, but it does not fully implement this issue’s requested circuit breaker. There is a configurable compaction-only model override, aggregate timeout handling, pre-compaction snapshot fallback, and session reset after compaction failure, but no automatic fallback model after repeated failures, no repeated-failure force-truncate path, and no gateway-status compaction health surface. The compaction/transport coupling also still exists in the unsubscribe path.

## Evidence

- **Compaction model is configurable, but only as a static override:** Docs and runtime config expose `agents.defaults.compaction.model` as an optional compaction-only model override. This is not the requested automatic fallback-after-failures behavior.
  - file: [docs/concepts/compaction.md:45](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/docs/concepts/compaction.md#L45)
  - command: `nl -ba docs/concepts/compaction.md | sed -n '38,110p'`
- **Compaction failure now resets the session instead of retrying forever:** Current auto-reply handling treats compaction/context-overflow failures as session-level failures and resets the conversation to a fresh session with a user-facing error. That is a mitigation, but it is not the requested fallback-model-plus-force-truncate circuit breaker.
  - file: [src/auto-reply/reply/agent-runner-execution.ts:1486](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/src/auto-reply/reply/agent-runner-execution.ts#L1486)
  - command: `nl -ba src/auto-reply/reply/agent-runner-execution.ts | sed -n '1360,1515p'`
- **Timeout fallout is mitigated, not eliminated:** The runner now bounds compaction retry waits with an aggregate timeout and falls back to a pre-compaction snapshot when compaction times out, which helps avoid getting stuck on the lane. That still does not move compaction off the agent lane or implement the requested background transport-safe architecture.
  - file: [src/agents/pi-embedded-runner/run/compaction-retry-aggregate-timeout.ts:1](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/src/agents/pi-embedded-runner/run/compaction-retry-aggregate-timeout.ts#L1)
  - command: `nl -ba src/agents/pi-embedded-runner/run/compaction-retry-aggregate-timeout.ts | sed -n '1,220p'`
- **Transport remains coupled to compaction teardown:** The embedded subscription still rejects pending waits with `AbortError("Unsubscribed during compaction")` and aborts in-flight compaction on unsubscribe, so the exact architectural coupling called out in the issue is still present in current code.
  - file: [src/agents/pi-embedded-subscribe.ts:816](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/src/agents/pi-embedded-subscribe.ts#L816)
  - command: `nl -ba src/agents/pi-embedded-subscribe.ts | sed -n '780,940p'`
- **Status surface is limited to session compaction count:** Current status output shows `🧹 Compactions: <count>` in session status, but there is no gateway-status compaction health surface for running/failed/stuck compaction state as requested.
  - file: [src/status/status-message.ts:712](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/src/status/status-message.ts#L712)
