## Summary

Keep open. Current `main` does not implement the PR's requested cleanup across the file-consent paths, and the PR itself appears stale against the current MSTeams file-consent architecture.

## Evidence

- **Current success path replaces card only when a stored activity id exists:** On accept+success, `handleMSTeamsFileConsentInvoke` builds a `FileInfoCard` and either sends it normally or calls `context.updateActivity(...)` when `pendingFile.consentCardActivityId` is present. There is no `deleteActivity` cleanup here.
  - file: [extensions/msteams/src/file-consent-invoke.ts:86](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/extensions/msteams/src/file-consent-invoke.ts#L86)
  - command: `nl -ba extensions/msteams/src/file-consent-invoke.ts | sed -n '1,280p'`
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
- **Decline and expired paths do not remove or replace the consent card:** Conversation mismatch returns after optionally sending the expired message, missing-pending-upload accept sends the expired message, and decline only removes the pending upload records. None of those paths call `updateActivity` or `deleteActivity`.
  - file: [extensions/msteams/src/file-consent-invoke.ts:51](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/extensions/msteams/src/file-consent-invoke.ts#L51)
