## Summary

Keep open. Current main still delegates ACP subprocess management to acpx@0.6.1, and that dependency still only keeps a bounded startup stderr buffer plus optional verbose mirroring to process.stderr. OpenClaw's acpx plugin config has no stderr log path, retention, or redacted persistent stderr capture. The issue is recent, concrete, and tied to existing ACPX debugging failures, so it remains actionable.

## Best Possible Solution

Keep this issue open. The best path is to implement bounded, redacted, per-process stderr teeing in the acpx AcpClient.start path, add configurable log directory and retention through the OpenClaw acpx plugin schema/manifest/docs, cover startup and post-initialize stderr behavior with tests, then bump extensions/acpx to the released acpx version containing the fix.

## Risks / Open Questions

- Persisting raw child stderr may capture credentials, environment diagnostics, or provider/auth details, so the implementation should align with existing redaction, retention, and file permission expectations.
- The subprocess stderr hook lives in the upstream acpx runtime package, so duplicating it only in OpenClaw's wrapper risks drift unless the change is made in openclaw/acpx and consumed by a pinned dependency bump.
