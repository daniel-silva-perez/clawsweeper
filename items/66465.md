## Summary

Keep open. Current main still does not persist ACPX child-process stderr after initialization. OpenClaw now delegates ACP subprocess ownership to the pinned `acpx@0.6.1` runtime, and that exact package still only keeps a bounded startup stderr buffer plus optional verbose mirroring to `process.stderr`. There is no OpenClaw config, manifest, docs, or current-main test surface for an ACPX stderr log directory or retention policy.

## Best Possible Solution

Keep this issue open. The best path is to implement bounded, redacted, per-process stderr teeing in the upstream `openclaw/acpx` `AcpClient.start` path, expose configurable log directory and retention through the OpenClaw ACPX plugin schema/manifest/docs, add tests for startup and post-initialize stderr capture, then bump `extensions/acpx` to the released `acpx` version containing the fix.

## Risks / Open Questions

- Persisting raw child stderr can capture API keys, OAuth diagnostics, environment details, provider auth paths, or private prompt/tool data; the implementation should define redaction, restrictive file permissions, retention, and discoverability before enabling it broadly.
- The subprocess stderr hook lives in the upstream `acpx` runtime package, and `extensions/acpx/AGENTS.md` says shared ACP runtime behavior belongs in `openclaw/acpx`; patching only OpenClaw's wrapper would risk drift now that current main delegates to `acpx/runtime`.
- The existing open PR #53997 overlaps conceptually with stderr artifacts but targets an older wrapper-owned implementation and is not merged, so it should not be treated as a current-main fix or canonical close reason without maintainer review.
