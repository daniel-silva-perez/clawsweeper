## Summary

Keep open. Current `main` still does not implement the PR's core sudo-install guard, and the Linux systemd install path still resolves the unit file from `HOME` while `systemctl` operations can target `SUDO_USER`, so the reported mismatch is not proven fixed on `main`.

## Evidence

- **CLI install path still has no sudo guard:** `runDaemonInstall()` only checks `failIfNixDaemonInstallMode()` and then proceeds. `src/cli/daemon-cli/shared.ts` contains no `failIfSudoInstall()` equivalent on current `main`.
  - file: [src/cli/daemon-cli/install.ts:64](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/src/cli/daemon-cli/install.ts#L64)
  - command: `nl -ba src/cli/daemon-cli/install.ts | sed -n '50,85p'; nl -ba src/cli/daemon-cli/shared.ts | sed -n '20,50p'`
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
- **Systemd unit path still comes from HOME:** `resolveSystemdUnitPathForName()` uses `resolveHomeDir(env)`, and `writeSystemdUnit()` writes directly to that path. Under a typical `sudo` environment this still points at `/root/...` rather than the invoking user’s home.
  - file: [src/daemon/systemd.ts:47](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/src/daemon/systemd.ts#L47)

- **systemctl calls still honor SUDO_USER separately:** `execSystemctlUser()` prefers `--machine <SUDO_USER>@ --user` when `SUDO_USER` is set, so current `main` still has a split between the file path source (`HOME`) and the service-control target (`SUDO_USER`).
  - file: [src/daemon/systemd.ts:333](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/src/daemon/systemd.ts#L333)

  - file: [docs/gateway/index.md:218](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/docs/gateway/index.md#L218)
