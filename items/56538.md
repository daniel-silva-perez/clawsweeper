## Summary

Keep PR #56538 open. Current main still does not apply Windows code-page decoding to the `runExec` and `runCommandWithTimeout` paths that the PR targets, and the docs still describe Windows exec Chinese mojibake as an active workaround/report path. The PR discussion is also substantive: it links the concrete bugs #56462/#50519, includes Windows reproduction/verification, and later review feedback says the main code-page concerns were addressed with only minor style issues remaining. This is not implemented on main, not superseded by the closed unmerged #51547, and not a conservative auto-close candidate.

## Best Possible Solution

Keep the PR open for maintainer review/rework. The best path is to land a focused Windows output-decoding fix that captures `execFile` output as buffers, decodes with the existing code-page helper, handles streaming paths with a persistent `TextDecoder` or buffered decode, preserves non-Chinese Windows code pages and UTF-8 fallback behavior, adds targeted tests for GBK and UTF-8 Windows cases, and removes unrelated ACPX symlink churn. If maintainers prefer a separate smaller PR, close #56538 only after that replacement lands or is clearly designated canonical.
