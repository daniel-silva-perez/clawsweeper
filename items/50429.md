## Summary

Keep this PR open. Current main still has the Feishu non-streaming card reply path sending structured cards without a text fallback, and the linked user problem remains plausible. The PR is not merge-ready as-is because review found remaining edge cases, but it contains relevant focused work rather than cleanup-only churn. The newer #70651 is related, but it targets outbound send paths and does not clearly supersede this reply-dispatcher fix.

## Best Possible Solution

Keep this PR open and ask for a revision that incorporates the useful fallback behavior while addressing the remaining review findings: only fall back for final/user-visible payloads, avoid duplicating already delivered chunks, consider matching the Feishu table-limit error specifically, and coordinate any shared error parsing with the related outbound fix in #70651.

## Risks / Open Questions

- Merging the PR as-is could resend already delivered card chunks as a full plain-text reply if a later chunk fails.
- Merging the PR as-is could allow block/preview payloads to be resent as ordinary Feishu text after a card failure.
- A narrower table-limit predicate or shared card-error helper may be needed so unrelated Feishu card failures still surface normally.
