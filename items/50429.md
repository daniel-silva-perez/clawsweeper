## Summary

Keep this PR open. Current main still lacks a Feishu non-streaming reply-card fallback, and the linked table-limit user problem is still plausible. The PR is focused and relevant, but it needs revision before merge because the current fallback catches all card errors, can resend already delivered chunks, and can turn block/preview payloads into ordinary text.

## Best Possible Solution

Keep this PR open and ask for a focused revision: limit fallback to final user-visible replies, avoid duplicating already delivered chunks by retrying only the failed/undelivered portion or only when no card chunk was sent, match Feishu table-limit errors explicitly, and coordinate any shared card-error helper with #70651 if that work lands first. Add regression tests for those cases before merge, then use it to resolve #43690.

## Risks / Open Questions

- Merging as-is can resend the whole reply as plain text after one or more card chunks were already delivered, duplicating content and mentions.
- Merging as-is can convert block/preview payloads into ordinary Feishu text after a card failure, which risks exposing non-final content before the final answer.
- The PR catches every structured-card failure instead of matching the Feishu table-limit error, which can hide unrelated card/auth/permission failures behind a text fallback.
- The current tests cover first-chunk card failure and success, but not later-chunk failure, block payloads, or a narrow table-limit predicate.
