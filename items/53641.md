## Summary

Keep open. Current main fixes the file/path Discord upload route, but the buffer-backed `message.send` case reported here still is not implemented or rejected: the public send schema advertises `buffer`/`filename`/`contentType`/`mimeType`, while the shared send runner and Discord adapters only forward URL/path media. Related #41779 and #44555 point to the same shared send-contract gap rather than a Discord multipart-only bug.

## Best Possible Solution

Keep this issue open and scope the fix to direct buffer attachments on ordinary `message.send`. The best implementation is to either hydrate and deliver `buffer` plus `filename` plus `contentType`/`mimeType` end to end through the shared send pipeline and Discord multipart upload path, with regression coverage proving a buffer-backed image reaches `files`, or to remove/fail-fast those fields from ordinary `send` if buffers are intentionally unsupported. Coordinate with #41779 and #44555 because the root mismatch is shared send-schema/runtime behavior.
