## Summary

Keep this issue open. Current main still has sender-scoped tool policy only on group/channel contexts, while direct-message sessions are parsed and routed as direct peers without a DM-level tool-policy resolver. Signal exposes `toolsBySender` only under `groups`, and Telegram's direct config type has `toolsBySender` but the active resolver still goes through group policy only.

## Best Possible Solution

Implement a generic direct-message sender tool-policy contract rather than a one-off channel workaround. The likely maintainer path is to add typed config/schema/docs for DM-level `toolsBySender`, resolve direct sender identity consistently for Signal and Telegram, feed that policy into both `createOpenClawCodingTools` and `applyFinalEffectiveToolPolicy`, and add regression tests proving one allowlisted DM sender can retain tools while another allowlisted sender is denied them.
