## Summary

Keep open. Current main still has a reconnect/listener-attach gap in the WhatsApp path, and I found no durable Baileys message-store replay layer that would prove messages arriving during that gap are recovered.

## Evidence

- **Reconnect path still waits for socket open before creating the listener:** `WhatsAppConnectionController.openConnection()` creates the socket, awaits `waitForWaConnection(sock)`, and only then calls `createListener(...)`. Messages delivered after the socket is open but before listener creation would have no `messages.upsert` handler attached yet.
  - file: [extensions/whatsapp/src/connection-controller.ts:355](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/extensions/whatsapp/src/connection-controller.ts#L355)
  - command: `nl -ba extensions/whatsapp/src/connection-controller.ts | sed -n '340,385p'`
  - sha: [ebd7f19a3b20](https://github.com/openclaw/openclaw/commit/ebd7f19a3b20001960474ec9ba9cc424e45f7b23)
- **Standalone inbox monitor has the same ordering:** `monitorWebInbox()` also awaits `waitForWaConnection(sock)` before calling `attachWebInboxToSocket(...)`. The actual `messages.upsert` and `connection.update` listeners are attached inside `attachWebInboxToSocket()`, after that wait completes.
  - file: [extensions/whatsapp/src/inbound/monitor.ts:699](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/extensions/whatsapp/src/inbound/monitor.ts#L699)

  - sha: [ebd7f19a3b20](https://github.com/openclaw/openclaw/commit/ebd7f19a3b20001960474ec9ba9cc424e45f7b23)
- **Tests currently lock in the open-before-listener ordering:** `connection-controller.test.ts` asserts the call order is `create` then `wait-for-connection`, and only afterwards does listener creation happen. That supports the reported race rather than disproving it.
  - file: [extensions/whatsapp/src/connection-controller.test.ts:78](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/extensions/whatsapp/src/connection-controller.test.ts#L78)

A repo search found no `makeInMemoryStore`, `store.bind`, `loadMessages`, or similar Baileys store/replay usage under `extensions/whatsapp`, so I found no evidence that messages arriving during the attach gap are recovered from a persistent store.
