## Summary

Keep open. Current `main` has SecretRef storage/runtime plumbing and plugin hook surfaces for custom blocking/redaction, but it does not implement the requested default masked-secrets model that prevents agents from accessing raw API keys via inherited runtime env or `~/.openclaw/.env`.

## Evidence

- **Global runtime secrets still load from `~/.openclaw/.env`:** `loadDotEnv()` loads the workspace `.env` and then the global fallback `~/.openclaw/.env` / `OPENCLAW_STATE_DIR/.env` into `process.env`, so gateway credentials can still exist as normal environment variables.
  - file: [src/infra/dotenv.ts:245](https://github.com/openclaw/openclaw/blob/754acc4478fe361db1e5303fd60f589d52beea7a/src/infra/dotenv.ts#L245)
  - command: `nl -ba src/infra/dotenv.ts | sed -n '240,320p'`
  - sha: [754acc4478fe](https://github.com/openclaw/openclaw/commit/754acc4478fe361db1e5303fd60f589d52beea7a)
- **Exec still inherits sanitized `process.env`:** `exec` builds host env from `process.env`, sanitizes only selected inherited keys, and then passes the resulting env through to host execution. There is no secret-broker substitution layer here.
  - file: [src/agents/bash-tools.exec.ts:1536](https://github.com/openclaw/openclaw/blob/754acc4478fe361db1e5303fd60f589d52beea7a/src/agents/bash-tools.exec.ts#L1536)

## Risks / Open Questions

- Community or third-party plugins may already mitigate some of this with hooks, but that does not satisfy the issue’s core request to protect OpenClaw-managed credentials by default.
- The issue bundles several subproblems: secret storage, exec-time exposure, transcript/output redaction, and command blocking. Future work may address those incrementally rather than via one single feature flag.
