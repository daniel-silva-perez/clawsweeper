## Summary

Current main still appears to have the reported Telegram local Bot API migration bug. `apiRoot` itself is implemented, but update offset state is scoped only by account/bot id, so a lower local Bot API update sequence can still be skipped before dispatch; `deleteWebhook` still only retries recoverable network-style errors, not Telegram 400 `Logged out` during migration cooldown.

## Best Possible Solution

Keep this issue open and fix it in the Telegram plugin. Persist a normalized apiRoot or endpoint fingerprint with the update-offset state, ignore or reset stored offsets when the endpoint changes, handle Telegram 400 `Logged out` during local Bot API migration without crash-looping, and add focused regression tests plus docs for local Bot API migration.
