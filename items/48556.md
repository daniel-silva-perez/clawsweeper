## Summary

Keep open. Current `main` does not implement this PR’s FormData normalization, and the affected proxy-upload path is still present for audio transcription when `HTTP_PROXY`/`HTTPS_PROXY` is configured.

## Evidence

- **Proxy wrapper still forwards init unchanged:** `makeProxyFetch()` and `resolveProxyFetchFromEnv()` still spread the caller’s `init` directly into `undiciFetch(...)` and only inject `dispatcher`; there is no FormData conversion step in current `main`. That means a global `FormData` body still reaches `undici.fetch` unchanged on the proxy path.
  - file: [src/infra/net/proxy-fetch.ts:25](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/src/infra/net/proxy-fetch.ts#L25)
  - command: `nl -ba src/infra/net/proxy-fetch.ts | sed -n '1,140p'`
  - sha: [4013c658537e](https://github.com/openclaw/openclaw/commit/4013c658537ee2051c44186f197b0710e84c9049)
- **Audio transcription still selects proxy fetch from env:** The media-understanding runner still resolves `fetchFn = resolveProxyFetchFromEnv()` before audio transcription requests, so proxy-configured transcription traffic still goes through the unfixed proxy wrapper.
  - file: [src/media-understanding/runner.entries.ts:573](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/src/media-understanding/runner.entries.ts#L573)
