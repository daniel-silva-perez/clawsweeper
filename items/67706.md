## Summary

Current main still does not emit `stream: "compaction"` events from the embedded runner's timeout/overflow recovery compaction paths, so this PR targets a real remaining gap. It should stay open, but the patch should be revised before merge because the current main still has at least one downstream consumer that treats any compaction `phase: "end"` as completed, and the PR emits failed end events too.

## Best Possible Solution

Keep this PR open as a valid implementation candidate. The best path is to update the recovery helper event emission on current main, align the end-event `completed` semantics with `handleCompactionEnd` or explicitly justify the difference, harden `agent-runner-memory.ts` to require `completed === true` before counting a compaction, and add focused tests in the overflow/timeout recovery harness for both global `emitAgentEvent` and `params.onAgentEvent` start/end payloads on success and failure.

## Risks / Open Questions

- Merging the PR exactly as-is can cause failed recovery compaction attempts to look completed to `agent-runner-memory.ts`, because that consumer still treats any `phase: "end"` as completed.
- The PR currently relies on existing overflow-compaction tests for behavior but does not add the narrow event-emission regression test that would lock in the reported UI/event-bus fix.
- The `completed` payload should be aligned with the canonical subscribe handler or documented as intentionally different before merge.
