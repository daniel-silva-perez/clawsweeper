## Summary

Keep this issue open. Current main still has the reported persistence shape: heartbeat turns can resolve and run on the heartbeat model, but the session usage persistence layer has no heartbeat signal and still writes the just-used provider/model into the persisted session entry fields that gateway/UI model resolution reads. The prior direct fix PR is closed unmerged, and the later cost snapshot work fixed `estimatedCostUsd` semantics without adding the missing heartbeat guard for `model` / `modelProvider`.

## Best Possible Solution

Keep this open and implement the narrow persistence fix in the auto-reply session usage path: thread `isHeartbeat` through `persistRunSessionUsage` / `persistSessionUsageUpdate` from both `runReplyAgent` and `createFollowupRunner`; when true, preserve `entry.model` and `entry.modelProvider` in both persistence branches while still recording usage, context, cost, and system prompt fields as intended. Add a focused regression test with default model != heartbeat model proving the heartbeat run uses the heartbeat model but the persisted session entry and `resolveSessionModelRef` display identity remain on the normal session model.
