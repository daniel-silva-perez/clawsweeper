## Summary

Keep open. Current main still has the reported write path: heartbeat runs resolve their heartbeat model correctly for the turn, but the shared session usage persistence code still writes the just-used `providerUsed`/`modelUsed` into `SessionEntry.modelProvider` and `SessionEntry.model`. The related fix PR #62987 is closed/unmerged in the provided context, and its proposed `isHeartbeat` persistence guard is not present on main.

## Best Possible Solution

Keep the issue open and implement the narrow persistence fix: thread `isHeartbeat` through `persistRunSessionUsage` and `persistSessionUsageUpdate` from both main and follow-up runner call sites; when true, preserve existing `entry.model`/`entry.modelProvider` and consider preserving CLI session binding fields as well. Add focused regression tests for heartbeat usage and no-usage persistence, plus a `sessions.list`/UI-facing row assertion for a default-model session after a heartbeat turn.

## Risks / Open Questions

- Default-model sessions without an explicit `modelOverride` can still display the heartbeat model after a heartbeat turn.
- Explicit `/model` sessions are partly protected by `modelOverride`, so the bug may be masked in some sessions while remaining present for default sessions.
- Heartbeat CLI session bindings may still overwrite provider session binding state because `applyCliSessionIdToSessionPatch` also lacks a heartbeat guard.
