## Summary

Keep open. Current `main` added a visible `sessions_send` announce path for ACP peer messaging, but it still routes that visible post through gateway `send`, which derives and persists an outbound Discord session entry instead of doing proxy-only delivery. The requested `visible:true` / `proxyOnly:true` / ACP-bound-thread exclusion behavior is not implemented on `main` yet.

## Evidence

- **No `visible` option on `sessions_send`:** `SessionsSendToolSchema` only exposes `sessionKey`, `label`, `agentId`, `message`, and `timeoutSeconds`; there is no `visible` flag matching option A from the issue.
  - file: [src/agents/tools/sessions-send-tool.ts:37](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/src/agents/tools/sessions-send-tool.ts#L37)
- **Visible follow-up goes through gateway `send`:** The new A2A follow-up path resolves an announce target and then calls gateway `send` with `to`, `accountId`, and `threadId` to post the visible message.
  - file: [src/agents/tools/sessions-send-tool.a2a.ts:131](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/src/agents/tools/sessions-send-tool.a2a.ts#L131)
- **Announce target reuses stored delivery route:** Announce target resolution pulls `channel`, `to`, `accountId`, and `threadId` from the target session’s stored delivery metadata, so the visible follow-up re-enters the normal outbound send pipeline rather than a proxy-only bypass.
  - file: [src/agents/tools/sessions-announce-target.ts:48](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/src/agents/tools/sessions-announce-target.ts#L48)
- **Gateway `send` still creates/persists outbound session routes:** Gateway `send` resolves an outbound session route and then calls `ensureOutboundSessionEntry` before delivery, so visible outbound posts are not side-effect-free.
  - file: [src/gateway/server-methods/send.ts:482](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/src/gateway/server-methods/send.ts#L482)
- **Outbound send writes session metadata for the derived session key:** `ensureOutboundSessionEntry` records session metadata via `recordSessionMetaFromInbound` for `route.sessionKey`, which is exactly the stub/session-pollution side effect the issue is asking to avoid.
  - file: [src/infra/outbound/outbound-session.ts:135](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/src/infra/outbound/outbound-session.ts#L135)
- **Discord threaded outbound routing still has a normal session key:** Discord outbound routing builds a normal `agent:<agentId>:discord:channel:<id>` session key and only carries `threadId` as route metadata; the docs now describe this as a visible A2A fallback, not as proxy-only delivery or an ACP-thread exclusion rule.
  - file: [extensions/discord/src/outbound-session-route.ts:30](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/extensions/discord/src/outbound-session-route.ts#L30)
- **Docs describe visible A2A announce, not the requested proxy-only feature:** Current ACP docs say `sessions_send` uses an A2A follow-up that delivers an announce to the visible channel/thread. They do not document `visible:true`, `proxyOnly:true`, or a rule preventing main-session route creation for ACP-bound Discord threads.
  - file: [docs/tools/acp-agents.md:349](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/docs/tools/acp-agents.md#L349)
