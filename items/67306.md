## Summary

Current main does not implement first-class JSON Schema support on the requested `openclaw agent` or `openclaw infer model run` CLI surfaces. There is a useful optional `llm-task` plugin that validates JSON output against a provided schema for workflow use, but it is not exposed as a core `agent`/`infer` CLI schema contract and does not define the requested repair/retry behavior. The cross-referenced #69030 is adjacent raw-inference work, not a duplicate or implementation of this request.

## Best Possible Solution

Keep this issue open. The best path is to implement a narrow provider-neutral schema contract on `openclaw infer model run` first, likely `--json-schema <path-or-inline>`, parse and validate the final JSON payload with AJV, return machine-readable validation/repair-exhaustion failures in `--json` mode, and add an additive gateway protocol field if the gateway transport is supported. After that is solid, decide whether `openclaw agent` should expose the same contract or delegate users to the existing optional `llm-task` plugin for workflow-only JSON tasks.

## Risks / Open Questions

- A schema-aware implementation needs a clear provider-neutral contract so native provider schema modes can be used opportunistically without leaking provider-specific behavior into core CLI semantics.
- The existing `llm-task` plugin provides useful AJV validation behavior to reuse or align with, but treating it as sufficient would leave the requested `agent`/`infer` CLI workflow unsolved.
