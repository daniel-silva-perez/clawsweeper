## Summary

Current main still does not implement first-class JSON Schema support on `openclaw infer model run` or `openclaw agent`. The existing `llm-task` plugin is useful related work for workflow JSON validation, but it is optional plugin/tool behavior, not the requested core CLI schema contract, and it throws on invalid JSON/schema instead of defining repair/retry semantics. The open #69030 cross-reference is adjacent raw-inference work, not a duplicate or implementation of this request.

## Best Possible Solution

Keep this issue open. The best path is to implement a narrow schema contract on `openclaw infer model run` first, likely `--json-schema <path-or-inline>`, validate final JSON with a shared AJV/schema-validation helper, return machine-readable validation and repair-exhaustion errors in `--json` mode, and add an additive gateway protocol field for the gateway transport. After that lands, maintainers can decide whether `openclaw agent` should expose the same contract or continue steering workflow-only use cases to the optional `llm-task` plugin.

## Risks / Open Questions

- A core implementation needs a provider-neutral CLI/output contract so provider-native schema modes can be used opportunistically without changing OpenClaw semantics by provider.
- Gateway support needs an additive protocol field and matching validator/docs/client updates because current `agent` params reject unknown fields.
- The existing `llm-task` AJV validation path is useful prior art, but reusing it directly would not address inline/file schema parsing, `--json` machine-readable failure shape, or repair-exhaustion behavior.
