## Summary

Keep #55099 open. Current main narrows part of the failure mode, but it does not contain a targeted fix or regression test for the reported macOS `/etc/hosts` case where a `fetch failed` aggregate includes loopback `ECONNREFUSED` plus IPv6 timeout, and the current fallback classifier still treats `ETIMEDOUT`/`UND_ERR_SOCKET` as sticky-IPv4 fallback triggers.

## Best Possible Solution

Keep the issue open and add a focused Telegram transport regression around a `fetch failed` aggregate containing IPv4 loopback `ECONNREFUSED` and IPv6 `ETIMEDOUT`/`UND_ERR_SOCKET` with `dnsResultOrder: "verbatim"`. The likely fix belongs in `extensions/telegram/src/fetch.ts`: classify intentional loopback IPv4 refusal differently so it does not arm or warn as sticky IPv4 fallback when the configured path is IPv6/verbatim, while preserving the existing sticky fallback for genuine broken-IPv6 hosts where IPv4 succeeds.
