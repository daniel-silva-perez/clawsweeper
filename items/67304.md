## Summary

Keep open. Current main still has no `OPENCLAW_JITI_CACHE_DIR` support, and the PR addresses a real core plugin-loader concern, not a ClawHub/external-only request. The PR should not be closed as obsolete, but it needs maintainer review and revision before merge because the proposed unset fallback would override Jiti's own `JITI_FS_CACHE=false` behavior.

## Best Possible Solution

Keep this PR open for maintainer/contributor revision. The best path is to add a narrow helper that returns the trimmed `OPENCLAW_JITI_CACHE_DIR` only when non-empty and otherwise omits `fsCache`/returns `undefined` so Jiti keeps honoring its own env defaults; wire that through the shared loader, root alias, and Matrix wrapper; add unit/contract coverage for unset, blank, custom path, and `JITI_FS_CACHE=false`; and add the required user-facing changelog/docs entry if maintainers accept the env var.

## Risks / Open Questions

- Merging the PR unchanged would make covered OpenClaw loaders ignore Jiti's own `JITI_FS_CACHE=false` or legacy `JITI_CACHE` env behavior when `OPENCLAW_JITI_CACHE_DIR` is unset.
- If the OpenClaw cache-dir override is read dynamically in tests or long-lived processes, the Jiti loader cache key needs to account for the effective `fsCache` value or callers need explicit cache isolation.
- A new user-facing env var needs focused tests and a changelog/docs decision before merge.
