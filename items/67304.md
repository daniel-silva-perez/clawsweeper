## Summary

Keep open. Current main still does not implement `OPENCLAW_JITI_CACHE_DIR`, and the PR targets a real core/plugin-loader behavior. It is not obsolete or ClawHub-only, but the current patch is not merge-ready because its unset fallback passes `fsCache: true` and would override Jiti's own `JITI_FS_CACHE=false` / `JITI_CACHE=false` behavior.

## Best Possible Solution

Keep this PR open for contributor revision and maintainer review. The best implementation is a narrow helper that returns the trimmed `OPENCLAW_JITI_CACHE_DIR` only when non-empty and otherwise omits `fsCache` or returns `undefined` so Jiti keeps honoring its own env defaults; wire that through the shared loader, root alias, and Matrix wrapper; add tests for unset, blank, custom path, and `JITI_FS_CACHE=false`; include cache-key isolation if the effective value can vary in-process; and add the required changelog/docs entry if maintainers accept the env var.

## Risks / Open Questions

- Merging the PR unchanged would cause covered OpenClaw Jiti loaders to ignore `JITI_FS_CACHE=false` or `JITI_CACHE=false` whenever `OPENCLAW_JITI_CACHE_DIR` is unset, because OpenClaw would pass explicit `fsCache: true`.
- If the OpenClaw cache-dir override is read dynamically in tests or long-lived processes, the Jiti loader cache key may need the effective `fsCache` value or another explicit cache-isolation mechanism.
- A new user-facing env var needs focused coverage and a changelog/docs decision before merge; otherwise the behavior is easy to regress and hard for operators to discover.
- Security review found no new dependencies, workflow changes, network calls, publishing metadata, or secret handling changes. The relevant security/ops surface is filesystem write behavior from the cache directory option.
