## Summary

Current `main` contains the service-env, config-write, and headless-systemd recovery changes that cover the 2026.3.28 Linux VPS outage pattern described in #57188.

## Evidence

- **Config write-back regression called out in the report is fixed on main:** Config persistence now writes `persistCandidate` instead of AJV-injected `validated.config`, explicitly avoiding the `#56772` schema-defaults regression; there is a dedicated regression test for that path.
  - file: [src/config/io.ts:1664](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/src/config/io.ts#L1664)
  - command: `nl -ba src/config/io.ts | sed -n '1658,1675p'; nl -ba src/config/io.write-config.test.ts | sed -n '1,40p'`
- **Update flow now surfaces gateway service refresh failures instead of hiding them:** `maybeRestartService()` always logs `refreshGatewayServiceEnv()` failures, including JSON mode, so update callers can detect and act on service-refresh breakage instead of silently wedging the gateway.
  - file: [src/cli/update-cli/update-command.ts:738](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/src/cli/update-cli/update-command.ts#L738)
  - command: `nl -ba src/cli/update-cli/update-command.ts | sed -n '736,760p'`
- **Forced reinstall now preserves PATH and safe custom service env:** Gateway install planning merges the managed PATH with preserved absolute segments from the existing service environment and drops unsafe carryover; regression coverage explicitly checks an npm/global-style custom PATH survives reinstall.
  - file: [src/commands/daemon-install-helpers.ts:104](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/src/commands/daemon-install-helpers.ts#L104)
  - command: `nl -ba src/commands/daemon-install-helpers.ts | sed -n '104,249p'; nl -ba src/commands/daemon-install-helpers.test.ts | sed -n '381,411p'`
- **Headless Linux systemd recovery is implemented and tested:** Systemd operations fall back from `systemctl --user` to `systemctl --machine <user>@ --user` when the user bus is unavailable, and CLI hints now tell operators to enable lingering and set `XDG_RUNTIME_DIR`; tests cover both behaviors.
  - file: [src/daemon/systemd.ts:367](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/src/daemon/systemd.ts#L367)

- **Changelog shows the Linux daemon/service fixes shipped after 2026.3.28:** `v2026.4.7` shipped the plugin initialization stack-overflow fix path referenced in the report, and `v2026.4.10` shipped the gateway-install PATH/custom-env preservation fix for forced reinstalls.
  - file: [CHANGELOG.md:1148](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/CHANGELOG.md#L1148)

## Risks / Open Questions

- Custom root-level systemd units are still a manual path; `openclaw gateway install` manages the user unit, so status/repair flows remain user-unit-centric unless the operator uses the documented system-unit path.

## Close Comment

Current `main` appears to cover the failure modes in this report well enough to close.

- Codex review found the `#56772` config-write regression fixed: config persistence now avoids writing AJV-injected schema defaults back to disk, with regression coverage in `src/config/io.ts:1664` and `src/config/io.write-config.test.ts:15`.
- The update path now always surfaces gateway service refresh failures instead of hiding them in JSON/update flows, in `src/cli/update-cli/update-command.ts:738`.
- Forced reinstall now preserves safe custom service env vars and merges prior `PATH` segments, which directly addresses the npm-global / service-PATH drift described here; see `src/commands/daemon-install-helpers.ts:104`, `src/commands/daemon-install-helpers.ts:242`, and the regression in `src/commands/daemon-install-helpers.test.ts:381`. This shipped in `v2026.4.10` (`CHANGELOG.md:930`).
- Headless Linux recovery is explicitly handled now: systemd operations fall back to `systemctl --machine <user>@ --user` when the user bus is unavailable, and the CLI prints the same `loginctl enable-linger` / `XDG_RUNTIME_DIR` guidance the reporter later confirmed as the workaround; see `src/daemon/systemd.ts:367` and `src/daemon/systemd-hints.ts:17`.
- The other suspected root cause from the report, plugin-init stack overflow during repair, is also called out as fixed in `v2026.4.7` at `CHANGELOG.md:1148`.
- Current docs also separate Linux user-unit installs from manual system-level units for always-on hosts, matching the root-level systemd setup described in the issue (`docs/gateway/index.md:216` and `docs/gateway/index.md:269`).
