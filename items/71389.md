## Summary

Keep #71389 open. Current main still loads `sharp` through the bundled `media-understanding-core` image ops path, and the central requested behavior is not implemented: there is no `OPENCLAW_IMAGE_BACKEND=disabled`/pass-through backend, no pre-import CPU capability gate, and `resizeToJpeg`, `convertHeicToJpeg`, and `resizeToPng` still delegate directly to the sharp-backed plugin. The related #47419 ClawSweeper report was a separate Docker image non-AVX2 release-pipeline regression fixed in v2026.3.22, not this runtime `sharp` load path.

## Best Possible Solution

Keep the issue open and fix the runtime image backend boundary. Add a Linux/x64 pre-import compatibility gate or isolated subprocess preflight before loading sharp, support an explicit `OPENCLAW_IMAGE_BACKEND=disabled`/`none` pass-through mode, and make `resizeToJpeg`, `convertHeicToJpeg`, and `resizeToPng` degrade to original-buffer passthrough or a clear safe error depending on the caller contract. Add regression tests with a mocked media-understanding-core sharp load failure proving the gateway/image paths continue without importing sharp on disabled or unsupported backends, plus docs for the supported runtime workaround.

## Risks / Open Questions
