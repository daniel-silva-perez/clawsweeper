## Summary

Keep PR #39653 open. Current main still contains the hostname fallback this PR removes, and multiple current tests still assert hostname `ws://` acceptance when `OPENCLAW_ALLOW_INSECURE_PRIVATE_WS=1` is set. The related issue #37177 was closed stale and prior PR #37883 was closed unmerged, so neither supersedes this active PR.

## Best Possible Solution

Keep PR #39653 open for maintainer security review. The best path is to merge or adapt the narrow validator change so `allowPrivateWs` accepts loopback/private IP literals but rejects unresolved hostnames, then update the current gateway net, client, connection-details, and onboarding tests to reflect that contract. If maintainers want private DNS hostnames, that should be tracked as a separate explicit allowlist or async DNS-resolution design rather than auto-closing this fix.
