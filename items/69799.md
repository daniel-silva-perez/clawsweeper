## Summary

Keep this issue open. Current main has meaningful mitigations: app-owned TCC surfaces for system.run/PeekabooBridge, install-cli local-prefix Node support, Homebrew stable-path handling, and optional LaunchAgent wrapper support. But the supported macOS Gateway path still does not ship or require an OpenClaw-owned bundled/dedicated Node runtime by default, so the core TCC boundary requested here remains unresolved.

## Best Possible Solution

Keep this as core macOS security and packaging architecture work. The best fix is to make the supported macOS Gateway/FDA runtime use a stable OpenClaw-owned identity by default, either a signed app-owned Gateway/runtime or a dedicated installed Node binary, with migration, doctor/onboarding, docs, and LaunchAgent reinstall behavior coordinated with #7227, #69561, and #22179. Treat install-cli local Node and `--wrapper` as mitigations until maintainers explicitly make one the supported TCC boundary.

## Risks / Open Questions

- Closing this as implemented would conflate optional local-prefix/wrapper mitigations with the requested default OpenClaw-owned TCC identity; npm/global external Node installs would remain exposed to the shared-node permission boundary described by the reporter.
- Related #7227 is still open and security-labeled in the provided context, so this overlaps broader macOS Accessibility hardening and should stay in maintainer-owned security/packaging scope rather than being treated as a plugin or docs-only cleanup.
- The docs now describe app-owned TCC for some macOS surfaces while the Gateway docs/code still use an external launchd CLI runtime; maintainers should clarify and align that boundary as part of the fix.
