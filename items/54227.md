## Summary

Keep this issue open. Current main still implements Discord component expiry per component id: build registers separate entries for sibling buttons, the registry consumes only the requested id, and the interaction handlers consume only the clicked component before dispatch. The closed broader #54233 does not supersede this focused remaining bug.

## Best Possible Solution

Keep this issue open as the focused tracker. Fix the Discord plugin by assigning a shared message or panel consumption group when component entries are registered, then invalidating sibling non-reusable entries in that group when an authorized component is consumed before callback dispatch. Add a regression test with two buttons in one non-reusable component message proving the second sibling click returns the expired response and does not dispatch, while `reusable:true` remains active.
