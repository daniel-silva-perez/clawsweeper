## Summary

Keep open. Current main still persists auto-enable-derived `enabled` flags during gateway startup and `doctor --fix`. The separate startup self-restart loop has evidence of a shipped mitigation, but the requested runtime-only auto-enable behavior is still not implemented.

## Best Possible Solution

Keep the issue open and narrow the fix to separating effective runtime activation from persisted config normalization: gateway startup should pass an auto-enabled runtime snapshot to plugin/channel loading without writing `openclaw.json`, and `doctor --fix` should report auto-enable reasoning without persisting derived `enabled` flags. Preserve the existing restart-loop hash guard and add regression tests that remove `channels.whatsapp.enabled`, `channels.telegram.enabled`, and `plugins.entries.openai.enabled`, then verify startup/doctor leave the source config unchanged while runtime activation still works.
