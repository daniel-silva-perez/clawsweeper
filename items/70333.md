## Summary

Keep open. Current main partially mitigates startup restart churn by suppressing the first watcher reread for startup internal writes, but the core reported behavior remains: plugin auto-enable still materializes `enabled: true` into the config object, gateway startup persists that config with `writeConfigFile`, and doctor can include the auto-enable mutation in the config it writes.

## Best Possible Solution

Keep the issue open and implement a focused change that separates runtime activation from persisted config repair. Gateway startup and doctor should be able to use the auto-enabled runtime config and reason diagnostics without writing auto-enable-derived `enabled` flags back to `openclaw.json`; add regression tests for startup and `doctor --fix` preserving a config that omits those legacy keys while still activating configured channels/providers in memory.
