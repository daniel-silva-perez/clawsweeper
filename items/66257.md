## Summary

Keep this PR open. Current main still does not contain the PR's narrower CLI behavior: `devices approve <requestId>` calls the gateway first, local approve only runs after a qualifying gateway pairing error, and the local fallback guard still does not reject explicit `--token` or `--password`. The PR is also still open, draft, and unmerged, with a focused two-file CLI/test diff rather than an obsolete or superseded change.

## Best Possible Solution

Keep this PR open as a focused implementation candidate. The likely maintainer path is to rebase it onto current main, preserve or adapt the explicit token/password no-fallback guard, decide whether local-first approval is still the intended behavior now that gateway-side silent local pairing exists, and add regression coverage for direct local approve plus unknown-local-request gateway fallback.
