## Summary

Keep PR #49310 open. Current main still lacks the PR's central behavior: the TUI has no busy `awaiting follow-up` state for `sessions_yield`, final chat events still resolve to `idle` unless they are errors, and history restore does not inspect the persisted `sessions_yield` context marker. The PR is a focused draft with concrete code/test changes rather than stale low-signal churn, and no protected/maintainer ownership signal requires auto-keeping for a different reason.

## Best Possible Solution

Keep the PR or a maintainer-owned replacement open and implement the narrow TUI behavior on main: add a typed `awaiting follow-up` busy state, detect the existing `sessions_yield` context marker on final assistant output, restore/clear that state during session history load, and cover both live-event and history-restore paths with focused TUI tests.

## Evidence
