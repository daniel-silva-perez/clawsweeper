## Summary

Keep open. The issue is recent, detailed, and reports a narrower post-login Discord ingest-lag failure after the already-shipped #68159 startup race fix. Current main has reconnect/status supervision and listener queue improvements, but I did not find a Discord RESUME catch-up/backfill path or evidence that current main fixes the reported 100-400s delay between Discord message creation and OpenClaw enqueueing.

## Best Possible Solution

Keep #71546 open as a focused Discord plugin reliability bug. The maintainer path should be to reproduce or instrument a single-account idle bot on current main, log Carbon close/RESUME/RESUMED/MESSAGE_CREATE timing against OpenClaw enqueue timestamps, and then either preserve listener delivery across Carbon internal reconnects or add a guarded, deduped Discord catch-up read since the last seen message id after successful RESUME/reconnect. Coordinate with #38596 only for shared health-monitor symptoms; do not close this as duplicate unless that issue explicitly becomes the canonical tracker for post-login ingest lag.

## Evidence
