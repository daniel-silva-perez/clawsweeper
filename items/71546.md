Keep #71546 open. It is a fresh, detailed Discord reliability report against 2026.4.23/current main, not a stale duplicate of the startup fixes. Current main contains the #68159-style gateway registration/IDENTIFY-race hardening, but I did not find a shipped fix for delayed MESSAGE_CREATE ingestion during Carbon RESUME/runtime reconnect windows or a defensive post-RESUME catch-up path.

## Best Possible Solution

Keep #71546 open and treat it as the canonical current-main Discord ingest-lag report. A maintainer fix should reproduce on current main/latest stable, instrument Carbon READY/RESUMED/MESSAGE_CREATE timing, verify listener lifetime across internal reconnects, and likely add a bounded Discord resume/catch-up path using the existing REST message read helper with last processed message IDs, plus focused tests for reconnect/RESUMED delivery and duplicate suppression.
