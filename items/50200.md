## Summary

Keep open. Current `main` still drops inbound Slack attachments when the event payload omits `url_private`/`url_private_download`, while a separate Slack action path already uses `files.info` to refresh file metadata. This PR addresses a real gap that is not implemented on `main` yet.

## Evidence

- **Inbound resolver still returns null when Slack omits private URLs:** `resolveSlackMedia()` only reads `file.url_private_download ?? file.url_private`; if neither is present it immediately returns `null`, so no download attempt is made for that file.
  - file: [extensions/slack/src/monitor/media.ts:242](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/extensions/slack/src/monitor/media.ts#L242)
  - command: `nl -ba extensions/slack/src/monitor/media.ts | sed -n '225,380p'`
  - sha: [45146913007d](https://github.com/openclaw/openclaw/commit/45146913007da723b4f06aba6a66cf4a695974c0)
- **Current tests codify the drop-on-missing-URL behavior:** `media.test.ts` explicitly expects `resolveSlackMedia()` to return `null` and perform no fetch when a file lacks a private URL, matching the reported failure mode rather than disproving it.
  - file: [extensions/slack/src/monitor/media.test.ts:364](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/extensions/slack/src/monitor/media.test.ts#L364)
  - command: `nl -ba extensions/slack/src/monitor/media.test.ts | sed -n '330,380p'`
  - sha: [45146913007d](https://github.com/openclaw/openclaw/commit/45146913007da723b4f06aba6a66cf4a695974c0)
- **Inbound message preparation does not pass a Slack client into media resolution:** `resolveSlackMessageContent()` calls `resolveSlackMedia()` and `resolveSlackAttachmentContent()` with only `files`, `token`, and `maxBytes`, so the inbound path has no way to call `files.info` today.
  - file: [extensions/slack/src/monitor/message-handler/prepare-content.ts:75](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/extensions/slack/src/monitor/message-handler/prepare-content.ts#L75)
  - command: `nl -ba extensions/slack/src/monitor/message-handler/prepare-content.ts | sed -n '68,110p'`
  - sha: [45146913007d](https://github.com/openclaw/openclaw/commit/45146913007da723b4f06aba6a66cf4a695974c0)
- **Thread-starter hydration has the same limitation:** When a thread starter has files, `prepare-thread-context.ts` also calls `resolveSlackMedia()` without any client object, so the same missing-URL drop applies there too.
  - file: [extensions/slack/src/monitor/message-handler/prepare-thread-context.ts:125](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/extensions/slack/src/monitor/message-handler/prepare-thread-context.ts#L125)
  - command: `nl -ba extensions/slack/src/monitor/message-handler/prepare-thread-context.ts | sed -n '118,140p'`

- **Another Slack code path already uses `files.info` as the fix pattern:** `downloadSlackFile()` fetches fresh file metadata via `client.files.info({ file: fileId })` before calling `resolveSlackMedia()`, which matches the PR's rationale that current inbound handling is missing an existing fallback pattern.
  - file: [extensions/slack/src/actions.ts:441](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/extensions/slack/src/actions.ts#L441)
