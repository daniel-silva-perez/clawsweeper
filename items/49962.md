## Summary

Keep open. Current `main` does not implement this PR’s behavior change: `resolveChannelGroupPolicy()` still turns any non-empty `groups` map into an allowlist, and the docs still describe that as the intended contract. This is not conservative cleanup-close material.

## Evidence

- **Current main still enables allowlist for any non-empty groups map:** `resolveChannelGroupPolicy()` on `main` still uses `const allowlistEnabled = groupPolicy === "allowlist" || hasGroups;`, so `groupPolicy: "open"` plus explicit `groups` entries still enables allowlist behavior and can reject unlisted groups.
  - file: [src/config/group-policy.ts:345](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/src/config/group-policy.ts#L345)

  - sha: [45146913007d](https://github.com/openclaw/openclaw/commit/45146913007da723b4f06aba6a66cf4a695974c0)
- **Docs still say configured groups act as an allowlist:** Telegram docs explicitly say that once `groups` is configured, it "acts as allowlist (explicit IDs or `"*"`)". That matches current code, not the PR’s proposed semantics.
  - file: [docs/channels/telegram.md:151](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/docs/channels/telegram.md#L151)
