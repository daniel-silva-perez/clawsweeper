## Summary

Keep PR #49962 open. Current main still has the reported behavior: `resolveChannelGroupPolicy()` enables the group allowlist whenever any `groups` entries exist, even when `groupPolicy` is explicitly `"open"`. The proposed fix and regression tests are not present on main, and the surrounding docs/types show this needs maintainer product judgment rather than conservative cleanup.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is to decide the intended contract for explicit `groupPolicy: "open"` with specific `groups` entries, then either merge the one-line policy fix with the added regression tests and align docs/security guidance, or explicitly reject it with a docs/type cleanup that makes the current allowlist semantics unambiguous.
