## Summary

Keep open. The report is recent, concrete, and still maps to current main: Discord channel sessions are serialized through per-session command lanes, and the queue only releases an active lane when the task promise settles. Current main has session-lock cleanup paths, but I did not find a stale-lane watchdog, auto-release policy, or `status --lanes` diagnostic surface that would resolve the reported stuck-lane symptom.

## Best Possible Solution

Keep this issue open and treat it as a core auto-reply/session-lane reliability bug, not a Discord-only plugin cleanup. The best fix is to instrument lane active-task state with run/session metadata, add a regression covering cron or heartbeat system-event completion on a Discord channel session key, ensure `runAgentTurn`/`runEmbeddedPiAgent` settles and releases the lane in every terminal branch, and add a read-only lane diagnostic surface such as `openclaw status --lanes`. A defensive auto-release should only be added once it can distinguish a stale bookkeeping entry from a genuinely active run.

## Risks / Open Questions

- An unconditional stale-lane auto-release could allow concurrent writes to the same session if a run is actually still active; any watchdog should be tied to explicit run/LLM/process liveness metadata.
- The report is intermittent and lacks a tight unit repro, so the first maintainer step should add instrumentation or a targeted regression harness around cron/heartbeat/system-event terminal paths before changing recovery policy.
