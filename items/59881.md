## Summary

Keep #59881 open. The issue is recent, has no protected-label/maintainer-author reason to exclude it, and current main still appears to construct plugin hook context `channelId` from the channel/provider routing hint rather than the actual conversation/channel identifier. That matches the reporter's complaint closely enough that this should not be conservatively closed.

## Evidence

- **Current hook context still uses provider/channel hint for channelId:** `runEmbeddedPiAgent` builds `hookCtx` with `messageProvider: params.messageProvider ?? undefined` and `channelId: params.messageChannel ?? params.messageProvider ?? undefined`, so when only the provider-style channel hint is available, `ctx.channelId` can still be `discord` rather than the Discord snowflake/conversation id.
  - file: [src/agents/pi-embedded-runner/run.ts:2835](https://github.com/openclaw/openclaw/blob/d6ef1fcf24eea3a5985d653a3baf400e32aa1b75/src/agents/pi-embedded-runner/run.ts#L2835)
  - command: `GitHub source view for current main; local exec was unavailable`
  - sha: [d6ef1fcf24ee](https://github.com/openclaw/openclaw/commit/d6ef1fcf24eea3a5985d653a3baf400e32aa1b75)
- **Run params distinguish routing channel from actual current channel id:** `RunEmbeddedPiAgentParams` has separate fields for `messageChannel`, `messageProvider`, and `currentChannelId`; `currentChannelId` is documented as the current channel ID for auto-threading, but it is not used in the hook context assignment above.
  - file: [src/agents/pi-embedded-runner/run/params.ts:575](https://github.com/openclaw/openclaw/blob/d6ef1fcf24eea3a5985d653a3baf400e32aa1b75/src/agents/pi-embedded-runner/run/params.ts#L575)
