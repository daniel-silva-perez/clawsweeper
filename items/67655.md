## Summary

Keep this PR open. The checked-out main still has the specific gap the PR targets: Windows shell-wrapper blocking in allowlist mode is limited to `cmdInvocation`, while PowerShell is recognized as a shell wrapper and can produce a shell payload with `cmdInvocation: false`. The PR is a focused implementation candidate, but the existing review comments identify small test-message regressions that should be fixed before landing.

## Best Possible Solution

Keep the PR open and ask for a small revision: make the guard block Windows shell wrappers based on `shellWrapperInvocation && isWindows` in allowlist mode, update denial guidance and all affected assertions to mention `powershell -Command`, add the PowerShell-wrapper regression, and either remove or document the now-nonblocking `cmdInvocation` parameter before maintainer review/landing.
