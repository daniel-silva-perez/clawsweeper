## Summary

Keep open. The current Telegram inbound body path still can emit `<media:image>` for a `msg.photo` solely from the Telegram message shape, without reclassifying from the downloaded media `contentType`. The referenced PR #69893 is closed/unmerged in the provided context, and current `main` still contains the hardcoded placeholder logic with no regression test for a non-image `allMedia` content type in `resolveTelegramInboundBody`.

## Best Possible Solution

Keep #69793 open and fix the Telegram inbound body builder to derive media-only placeholders from resolved `allMedia` content types using the existing MIME kind contract, while preserving sensible fallback behavior when downloads fail. Add focused regression tests for Telegram photo-shaped messages whose saved media is `application/octet-stream` or another document MIME, plus multi-image and mixed-media cases if the helper is generalized.

For a photo message with no caption and downloaded media whose `contentType` is non-image or unknown, the photo placeholder remains `<media:image>`.
