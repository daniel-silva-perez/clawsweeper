## Summary

Keep this issue open. Current main still retries overflow compaction by rebuilding the same prompt from the original run params and continuing the run loop after successful compaction. Recent cron/exec and silent-output fixes reduce some trigger paths, but they do not implement the durable turn/retry identity or reply-obligation ledger requested here.

## Best Possible Solution

Keep this issue open and fix the embedded runner/subscriber compaction retry contract: preserve a stable current-turn identity across compaction, carry retry attempt and original message metadata into the resumed attempt, prevent cron/system prompt reconstruction from being treated as a fresh user turn, and add regression coverage for a failing cron/tool cascade that compacts without re-executing as new input. Keep #59618 and #63216 cross-linked as related broader compaction-resume reports rather than using them to close this narrower bug.
