## Summary

Current main still does not implement manifest-level install-time `configPatch`. The manifest contract and loader omit the field, install result types do not carry it, plugin install persistence only enables/records install metadata, and the existing `configPatch` machinery remains provider-auth scoped. The related PR #43421 was closed unmerged in the provided context, so it does not supersede this issue.

## Best Possible Solution

Keep this issue open for a core plugin install/API decision. A complete implementation should add a typed manifest-level `configPatch` contract, parse and sanitize it from plugin metadata, propagate it through all install result paths, apply a validated merge during install persistence with explicit array semantics, show the applied diff or skip reason, and update tests plus public docs such as `https://docs.openclaw.ai/plugins/manifest` and `https://docs.openclaw.ai/tools/plugin`.

## Risks / Open Questions

- Manifest-declared config mutation is trust-sensitive and needs sanitization, prototype-pollution protection, validation before write, and clear diff/skip UX.
- Array semantics remain an unresolved product/API decision, especially additive lists such as `tools.alsoAllow` where replacement could clobber another plugin's entry.
- A complete fix needs coverage across local path, `--link`, archive, npm, ClawHub, marketplace, bundled fallback, invalid patch, and config include write-through paths.
