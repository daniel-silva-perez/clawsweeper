## Summary

Keep this PR open. Current main still has the Feishu thread-history media gap the PR targets: current inbound Feishu media is resolved through the existing media download path, but fetched thread-history messages are still rendered from parsed placeholder text and do not retain raw Feishu body content for later media-key extraction. The PR is a focused implementation candidate, not obsolete or superseded.

## Best Possible Solution

Keep this PR open as the implementation candidate. The best path is to finish the small review items, retain raw Feishu body content for thread-history records, resolve media for media-typed history messages before building ThreadHistoryBody, inline runner-compatible attachment hints, add a download-failure regression test, and then run the Feishu plugin test lane before maintainer merge.
