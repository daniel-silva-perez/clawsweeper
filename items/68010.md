## Summary

Keep open. Current main still eagerly initializes `PROVIDER_ENV_API_KEY_CANDIDATES`, which calls manifest-backed provider env-var resolution at module import time. OpenRouter's `OPENROUTER_API_KEY` is supplied through plugin manifest metadata, so the PR targets a real unresolved main-branch loading-order gap. The branch should be reviewed rather than closed, with the lazy-loading test fixed and the broader security hardening judged separately.

## Best Possible Solution

Keep this PR open and review it as the implementation candidate for the unresolved main-branch bug: narrow or approve the production change that makes `PROVIDER_ENV_API_KEY_CANDIDATES` delegate to the existing lazy `PROVIDER_AUTH_ENV_VAR_CANDIDATES`, add a corrected regression test that awaits module import before checking manifest-registry calls, and separately decide whether the security-hardening changes belong in this PR or a focused follow-up.
