## Summary

Keep this issue open. Current main still lacks Discord messageUpdate/messageDelete handling: the bundled Discord plugin only wires message-create, reaction, thread-update, and optional presence listeners, while the active inbound worker and replay paths do not maintain a Discord source-message-id to active run abort mapping. Related #38505 remains open and covers only the edit half, so it does not supersede this broader edit-to-reprocess plus delete-to-cancel request.

## Best Possible Solution

Implement this in the bundled Discord plugin. Add Carbon MessageUpdateListener and MessageDeleteListener handling in extensions/discord/src/monitor/listeners.ts, register them from extensions/discord/src/monitor/provider.startup.ts, route message edits through the existing preflight/message processing path with content-change and mention gating plus edit-aware replay keys, and track the triggering Discord message id to the accepted pending job or active run abort controller so delete events cancel only matching active work. Coordinate with #38505 for the edit-only half, but keep this broader tracker open until both edit-to-reprocess and delete-to-cancel ship.

## Risks / Open Questions

- Edit handling needs explicit replay/dedupe design; otherwise an edited Discord message with the same message id may be ignored as already processed.
- Delete-to-cancel needs a precise Discord source message id to active job/run abort link. Cancelling by session or channel alone could stop a newer unrelated run in the same Discord conversation.
- Carbon update/delete payloads may be partial or missing message content, so implementation should hydrate/validate payloads and gate on actual content changes and mentions before reprocessing.
