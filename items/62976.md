## Summary

Keep this issue open. Current main still treats invalid configured plugin entries as config validation failures, deliberately skips last-known-good recovery for plugin-entry-only invalidity, and then throws during gateway startup. Doctor repair still covers stale/unknown plugin references and bundled load-path aliases, not known third-party plugins with invalid config/runtime state. Related #70371 and #70394 overlap on graceful startup degradation, but they are open context and do not prove this issue's doctor/safe-mode recovery scope is implemented on main.

## Best Possible Solution

Keep this issue open and fix it in core. Add a narrow startup recovery path that quarantines or disables only the bad third-party plugin entry while keeping the gateway bootable, surface the failed plugin in status/plugins doctor, and extend openclaw doctor --fix or a plugin quarantine/safe-mode command to repair plugins.entries, plugins.allow, and related plugins.load.paths deterministically. Coordinate with #70371/#70394, but do not close this issue until the startup degradation plus doctor/safe-mode recovery behavior is merged with tests.
