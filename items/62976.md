## Summary

Keep open. Current main still intentionally treats plugin-local config validation failures under `plugins.entries.<id>...` as invalid startup config, skips last-known-good recovery for that class, and throws before gateway boot. Doctor has stale/missing plugin reference cleanup, but not the requested quarantine/safe-mode path for a known third-party plugin whose config/schema is invalid. Related local context for #70371 points to the same remaining graceful-degradation gap rather than a shipped fix.

## Best Possible Solution

Keep the issue open and implement a narrow plugin-local recovery path: either quarantine/disable only the bad third-party plugin entry during gateway startup, or add an explicit `openclaw doctor --fix` / plugin quarantine repair that can remove or disable broken `plugins.entries`, `plugins.allow`, and relevant `plugins.load.paths` entries without rolling back unrelated config. Add regression coverage for startup remaining bootable and doctor repairing the configured bad-plugin case.
