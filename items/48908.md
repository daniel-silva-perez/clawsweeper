## Summary

Keep PR #48908 open. The linked Telegram 421 problem still appears unsolved on current main, and the PR is not obsolete, but the proposed patch is incomplete because Telegram outbound sends use the strict safe-send retry path that bypasses the shared regex the PR changes.

## Best Possible Solution

Keep the PR open, but request a rework against current main: add a Telegram-owned 421/Misdirected Request classifier in `extensions/telegram/src/network-errors.ts`, include it in the strict non-idempotent send retry predicate in `extensions/telegram/src/send.ts`, and add focused regression tests for both the classifier and the actual Telegram send retry path. The shared retry-policy regex can be updated too, but it is not sufficient by itself.

## Evidence

A regex-only change in src/infra/retry-policy.ts would not affect this user-visible send path.

## Risks / Open Questions

- Merging this PR unchanged would likely add partial non-strict retry behavior while leaving the main outbound Telegram send path from #48892 unfixed.
- A broad retry classification for non-idempotent sends must stay careful to avoid duplicate Telegram deliveries; 421 should be handled as a narrow safe rejection rather than by enabling the generic regex fallback for all send errors.
