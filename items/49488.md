## Summary

Keep PR #49488 open. Current main has not implemented the requested behavior: the Android app still declares minSdk 31, docs still reference minSdk 31, and there are still API-level assumptions that matter if support is lowered to API 26. The PR is coherent, has user/reporter validation in the supplied discussion, and is not duplicated or superseded by a shipped main change. The best path is maintainer product judgment on whether older Android support should be a dedicated compatibility effort, not automated closure.

## Best Possible Solution

Keep the PR available for maintainer judgment or convert it into a dedicated Android API 26-30 support effort. The right implementation would lower `apps/android/app/build.gradle.kts` only after auditing and guarding all API 29/30/31+ Android calls, updating docs/README support statements, rebasing onto current main, and providing focused manual/API-level verification evidence.
