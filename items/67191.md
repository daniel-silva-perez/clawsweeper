## Summary

Current main has not implemented the requested delivery change. The boot-md startup hook still routes BOOT.md through `runBootOnce`, which still calls `agentCommand` with `deliver: false`, and the unit test explicitly asserts that behavior. The docs/prompt indicate user-visible boot messages should be sent through the `message` tool followed by `NO_REPLY`, so the remaining work needs a targeted maintainer fix or regression test rather than an auto-close.

## Best Possible Solution

Keep this issue open. The best path is a focused maintainer fix in the boot-md/gateway boot path: either add regression coverage proving `BOOT.md` message-tool sends deliver to the configured channel/target during gateway startup, or add an explicit supported delivery target for boot sessions while preserving the documented `NO_REPLY` behavior and avoiding duplicate final replies.
