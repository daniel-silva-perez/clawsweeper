## Summary

Keep open. Current main still allows a persisted session skill snapshot from the old workspace to remain authoritative after the configured workspace path changes, because the snapshot does not record its source workspace and the refresh logic does not compare against the current workspace.

## Best Possible Solution

Keep the issue open and implement workspace-aware skill snapshot invalidation. The most direct fix is to record a normalized workspace/source root on SkillSnapshot/SessionSkillSnapshot, rebuild when it differs from the current resolved workspaceDir, invalidate snapshots on agent workspace config changes, and add a regression test with a sessions.json snapshot containing old absolute SKILL.md paths plus a current config pointing at a new workspace.
