## Summary

Keep open. Current main still follows the reported Discord DM path: `resolveDiscordAutoThreadReplyPlan` builds `replyTarget` as `channel:<dmChannelId>`, DMs get no `autoThreadContext`, and `processDiscordMessage` assigns `ctx.To` from that `replyTarget` before computing the `user:<authorId>` DM conversation target. The existing tests cover `lastRoute.to` being user-addressed, but they do not assert message-handler `ctx.To`, and a related DM component test still expects `To` to be channel-shaped. The linked PR #68144 was a focused fix candidate, but the provided context shows it closed unmerged, so this issue still tracks real remaining work.

## Best Possible Solution

Land a narrow Discord message-handler fix that computes `dmConversationTarget` before `effectiveTo` and uses `autoThreadContext?.To ?? dmConversationTarget ?? replyTarget` for direct messages while leaving `deliverTarget` channel-based for the Discord API. Add a regression test that exposes and asserts `ctx.To`, `ctx.From`, and `ctx.ChatType` for DM inbound messages; then review whether Discord component DM contexts should follow the same `To=user:<id>` semantic or remain intentionally channel-addressed.
