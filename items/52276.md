## Summary

Keep PR #52276 open. Current main has route-aware suppression helpers for final reply payloads, but the embedded block-reply path targeted by this PR still suppresses duplicate assistant block replies by text alone, without checking provider/account/target/thread scope. The PR is not merge-ready as-is because the review thread reports a blocking undefined `resolveRouteMessageProvider` issue on the PR head, but the underlying core bug is still real and not superseded by a canonical merged item.

## Best Possible Solution

Keep the item open and refresh the route-scoped dedupe fix on current main. The fix should thread originating provider/target/account and current channel/thread context into the embedded subscriber, align send-target extraction with `message` tool normalization and plugin `extractToolSend` behavior, use `shouldSuppressMessagingToolReplies` for both `emitBlockChunk` and `message_end` block replies, and add subscriber-level regression tests for same-route, different-route, providerless, implicit current-route, and Telegram topic sends.
