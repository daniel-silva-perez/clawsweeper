## Summary

Current main does not fully solve this report. `gateway.auth.mode="none"` still only bypasses the shared-secret auth check; unauthenticated loopback probes deliberately omit device identity, and the WebSocket connect policy still rejects missing device identity with close 1008. The direct-local backend bypass that exists on main is scoped to shared token/password or device-token auth, not `authMethod === "none"`. The related #69066 service-identity RFC is relevant, but it is broad open design work rather than a shipped or maintainer-canonical replacement for this concrete bug.

## Best Possible Solution

Keep this issue open. The best fix is to make internal service identity explicit for direct-loopback Gateway-owned callers, or add a tightly scoped interim path for auth-none internal probe/followup/backend RPCs with regression tests for `probeGateway` without token/password and backend `callGateway` without shared auth. The fix should preserve existing rejection for remote, browser-origin, node-role, and non-internal clients, and should align with the open #69066 service-identity design rather than expanding the current shared-secret exception ad hoc.
