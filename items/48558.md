## Summary

Keep open. Current `main` has substantial plugin-based memory features, but it does not implement Anthropic's native `memory_20250818` tool type or any Anthropic-specific memory-tool transport/config path. The request is coherent, recent, and still plausibly actionable, so there is not strong evidence for a cleanup close reason.

## Evidence

- **Repo vision keeps memory as a plugin slot, not proof of implementation:** The project vision says optional capability should usually ship as plugins and that memory is a special plugin slot with multiple memory options. That shows existing scope/architecture, but it does not satisfy this Anthropic-native feature request.
  - file: [VISION.md:52](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/VISION.md#L52)

- **Current docs describe OpenClaw memory as file-backed workspace memory with OpenClaw-defined tools:** The memory docs describe cross-session memory through `MEMORY.md` and `memory/YYYY-MM-DD.md`, and document `memory_search` and `memory_get` as the active memory tools. This is related functionality, but not Anthropic's `memory_20250818` tool type.
  - file: [docs/concepts/memory.md:9](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/docs/concepts/memory.md#L9)

  - file: [extensions/memory-core/index.ts:24](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/extensions/memory-core/index.ts#L24)

- There is related demand in the issue timeline via the open cross-reference to #70049, so closing this now risks discarding still-active design discussion rather than cleaning up a resolved item.
