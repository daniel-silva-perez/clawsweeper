## Summary

Keep #71583 open. The report is new, concrete, and targets the bundled QQ Bot plugin. Current main still performs the QQ token POST to `https://bots.qq.com/app/getAppAccessToken` without an OpenClaw-level timeout/AbortSignal, while the gateway connection awaits that token before it can open the WebSocket. The existing QQ Bot status/docs cover generic credential checks but do not provide a live token probe or bots.qq.com egress/proxy/region diagnostics. The provided related items are other timeout classes, not a canonical QQ Bot duplicate or shipped fix.

## Best Possible Solution

Keep the issue open and handle it in the bundled QQ Bot plugin: add an explicit token-fetch timeout/AbortController with clear `bots.qq.com` diagnostics, surface token-fetch failures in `channels status --probe` or account status, and document QQ Bot egress/proxy/region/IPv4 troubleshooting. If investigation proves this is purely upstream QQ policy, close later with a documented workaround and evidence.
