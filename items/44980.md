## Summary

Keep open. Current main contains a related fix for Telegram media downloads through explicit proxies, but the issue is not fully resolved for the documented `channels.telegram.proxy` surface because the SSRF guard still rejects non-HTTP(S) explicit proxy URLs while the docs still advertise `socks5://...` Telegram proxies.

## Evidence

- **Telegram media downloads now use exported dispatcher attempts plus trusted explicit-proxy mode:** `downloadAndSaveTelegramFile()` passes `dispatcherAttempts` into `fetchRemoteMedia()`, and `fetchRemoteMedia()` switches to `withTrustedExplicitProxyGuardedFetchMode` when `trustExplicitProxyDns` is set. This addresses the original dispatcher-overwrite path for explicit HTTP proxies, but only as a partial fix.
  - file: [extensions/telegram/src/bot/delivery.resolve-media.ts:235](https://github.com/openclaw/openclaw/blob/40be5ad58187fd20b8defe5fc30e649190c131a8/extensions/telegram/src/bot/delivery.resolve-media.ts#L235)
  - command: `nl -ba extensions/telegram/src/bot/delivery.resolve-media.ts | sed -n '140,255p'`
  - sha: [40be5ad58187](https://github.com/openclaw/openclaw/commit/40be5ad58187fd20b8defe5fc30e649190c131a8)
- **Current SSRF guard still hard-rejects non-HTTP(S) explicit proxies:** `assertExplicitProxyAllowed()` throws `Explicit proxy URL must use http or https`, so a configured `socks5://...` Telegram proxy still cannot get through the guarded media-download path.
  - file: [src/infra/net/fetch-guard.ts:188](https://github.com/openclaw/openclaw/blob/40be5ad58187fd20b8defe5fc30e649190c131a8/src/infra/net/fetch-guard.ts#L188)
