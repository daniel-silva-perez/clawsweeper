## Summary

Current `main` already handles the reported Discord gateway shutdown race by routing `Max reconnect attempts` errors through a supervisor and suppressing late reconnect-exhausted emissions during teardown/dispose instead of letting them crash the process.

## Evidence

- **Reconnect exhaustion is classified and suppressed during teardown/dispose:** `classifyDiscordGatewayEvent()` maps `Max reconnect attempts` to `reconnect-exhausted`, and `createDiscordGatewaySupervisor()` logs and suppresses late gateway errors once the lifecycle is in `teardown` or `disposed`.

## Close Comment

This looks implemented on current `main`, so I'm closing it from this Codex review.

- `extensions/discord/src/monitor/gateway-supervisor.ts:89` classifies `Max reconnect attempts ...` as a `reconnect-exhausted` gateway event, and `extensions/discord/src/monitor/gateway-supervisor.ts:161` suppresses late gateway errors once the lifecycle is in `teardown` or `disposed`.
- `extensions/discord/src/monitor.gateway.ts:28` now keeps lifecycle handling attached until after `disconnect()`, explicitly covering late disconnect-time `error` emissions instead of letting them surface uncaught.
- `extensions/discord/src/monitor/provider.ts:1143` disconnects the startup gateway and `extensions/discord/src/monitor/provider.ts:1152` disposes the supervisor during startup cleanup, which matches the shutdown race described here.
- Regression coverage is present: `extensions/discord/src/monitor/provider.test.ts:281` emits `Max reconnect attempts (0) reached after code 1005` after cleanup and asserts that it does not throw, and `extensions/discord/src/monitor/gateway-supervisor.test.ts:107` covers the same late-error suppression after supervisor dispose.
- Codex could tie the current implementation to commit `d38ed0831d7372c8505dc944aea69b5b83da1435`. I could not tie it to a shipped release tag yet, so this appears fixed on `main` but not yet confirmed in a release.
