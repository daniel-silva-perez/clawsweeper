## Summary

Keep open. Current `main` still documents these bridge tools for generic MCP clients, but the tool implementations return only count/summary text in `content` and put the actionable payload in `structuredContent`, so the reported interoperability gap is still present in code.

## Evidence

- **Docs still promise generic MCP client support:** The MCP docs explicitly say generic MCP clients should use the standard bridge tools.
  - file: [docs/cli/mcp.md:75](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/docs/cli/mcp.md#L75)
  - command: `nl -ba docs/cli/mcp.md | sed -n '72,90p'`
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
- **Summary helper only emits counts in content:** `summarizeResult()` returns `content` as a single text block like `"<label>: <count>"`; it does not serialize the payload.
  - file: [src/mcp/channel-shared.ts:139](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/src/mcp/channel-shared.ts#L139)
  - command: `nl -ba src/mcp/channel-shared.ts | sed -n '130,150p'`
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
- **Channel MCP tools still place real data in structuredContent:** `conversations_list`, `messages_read`, `attachments_fetch`, `events_poll`, `events_wait`, `messages_send`, `permissions_list_open`, and `permissions_respond` all return payloads via `structuredContent`, while `content` is only a count/status string.
  - file: [src/mcp/channel-tools.ts:25](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/src/mcp/channel-tools.ts#L25)
  - command: `nl -ba src/mcp/channel-tools.ts | sed -n '20,190p'`
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
- **Existing fallback materializer does not help these tools:** The bundle MCP materializer only JSON-stringifies `structuredContent` when `content` is empty. Because the channel tools always set non-empty summary/status `content`, that fallback does not apply here.
  - file: [src/agents/pi-bundle-mcp-materialize.ts:21](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/src/agents/pi-bundle-mcp-materialize.ts#L21)
