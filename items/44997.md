## Summary

Keep open. Current `main` does not implement this PR's serialized read-modify-write flow. It mitigates silent lost updates by rejecting stale writes with optimistic hash checks, but concurrent `agents add` calls still race at the command level and one can fail with `ConfigMutationConflictError` instead of queuing behind a lock and succeeding.

## Evidence

- **Agents add still uses snapshot + base-hash replace:** `src/commands/agents.commands.add.ts:60` reads one config snapshot up front, stores `baseHash`, computes `nextConfig` from that stale snapshot, and then calls `replaceConfigFile` at `src/commands/agents.commands.add.ts:136` with that same `baseHash`. There is no retry loop or in-lock transform in this command.
  - file: [src/commands/agents.commands.add.ts:60](https://github.com/openclaw/openclaw/blob/40be5ad58187fd20b8defe5fc30e649190c131a8/src/commands/agents.commands.add.ts#L60)
  - command: `nl -ba src/commands/agents.commands.add.ts | sed -n '55,140p'`
  - sha: [40be5ad58187](https://github.com/openclaw/openclaw/commit/40be5ad58187fd20b8defe5fc30e649190c131a8)
- **Main now rejects stale config writes instead of serializing them:** `src/config/mutate.ts:38` defines `assertBaseHashMatches`, which throws `ConfigMutationConflictError` when the on-disk hash changed. `replaceConfigFile` at `src/config/mutate.ts:139` reads a fresh snapshot and calls that guard before writing. That prevents silent overwrite, but it is not the file-lock/update-transform behavior proposed by PR #44997.

## Risks / Open Questions

- The original silent overwrite appears partially mitigated on `main`, which could make the PR look redundant at a glance even though its stronger guarantee is not present.
- Concurrent `openclaw agents add` calls can still fail under contention because the command does not retry or recompute from fresh state after a hash conflict.
