## Summary

Keep #70472 open. Current main still has the reported failure shape: compaction checkpoint persistence only warns and returns when the active session entry is missing, normal session initialization mints a new session id when the key has no reusable entry, and the compaction-failure recovery path still rotates the active session mapping. The linked fix PR #70479 is open and unmerged in the provided context, so the issue remains the right tracker until that behavior is changed and shipped.

## Best Possible Solution

Keep #70472 open and use #70479, or an equivalent maintainer-reviewed patch, to preserve the active `agent:main:main` mapping on overflow/compaction failures. The fix should make missing active-session entries during checkpoint persistence an explicit invariant/recovery path, avoid automatic remap unless the user intentionally resets, and add WebChat direct regression coverage proving the session id remains stable after compaction/checkpoint failure. If a fallback reset is truly required, persist an auditable session-rotation event with old/new ids and surface it in WebChat/session history.
