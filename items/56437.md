## Summary

Keep #56437 open. Current main partially addresses the outage symptoms by routing fallback attempts through the normal agent runner with system prompt/bootstrap context and by surfacing active fallback state, but it still does not implement a native `fallbackRunbook` config field or preflight/doctor validation for configured fallback model IDs.

## Best Possible Solution

Keep the issue open and implement it as a core agent/runtime feature: add a documented `fallbackRunbook` field to the agent model config contract, inject it into embedded and CLI system prompts only when the active fallback candidate differs from the selected primary, and add doctor/config validation warnings for fallback model refs across defaults, per-agent overrides, and cron payload fallbacks. The existing fallback visibility and retry-context behavior should remain in place.
