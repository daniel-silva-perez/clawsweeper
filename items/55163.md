## Summary

Keep open. Current main has a later startup config hash guard that mitigates one plugin-auto-enable restart-loop path, but the specific #55163 request is still not implemented: `applyPluginAutoEnable` can still write `plugins.entries.<id>.enabled = true` for a non-built-in plugin that is already selected in `plugins.allow`. The related fix PRs #55167, #55189, and #55190 are closed unmerged in the provided context, and current tests do not cover the requested allowlisted-local-plugin skip.

## Best Possible Solution

Implement the narrow guard proposed in the issue lineage inside `materializePluginAutoEnableCandidatesInternal`: before `registerPluginEntry`, skip non-built-in plugin candidates that are already explicitly present in `plugins.allow`, while preserving built-in channel/provider auto-enable behavior. Add a regression test for a global/workspace/local manifest plugin already in `plugins.allow` with no `plugins.entries` entry, asserting no `plugins.entries` write and no repeated changes on a second pass; add doctor or gateway-startup coverage if feasible.
