## Summary

Keep open. Current main still models usage as per-session-instance data keyed by `sessionId`/current store entries rather than transcript-backed logical-session lineage, and the WebUI does not expose the requested lineage/history controls or longer range presets.

## Evidence

- **RPC schema has no lineage/family aggregation controls:** `sessions.usage` only accepts `key`, `startDate`, `endDate`, date interpretation, `limit`, and `includeContextWeight`. There is no `groupBy`, `includeHistorical`, family key, or custom-range API shape matching the issue request.
  - file: [src/gateway/protocol/schema/sessions.ts:285](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/src/gateway/protocol/schema/sessions.ts#L285)
  - command: `nl -ba src/gateway/protocol/schema/sessions.ts | sed -n '285,330p'`
  - sha: [ebd7f19a3b20](https://github.com/openclaw/openclaw/commit/ebd7f19a3b20001960474ec9ba9cc424e45f7b23)
- **Gateway usage handler resolves one current sessionId, not a logical lineage:** For a specific key, the handler resolves a single `sessionId` and one `sessionFile`. For list mode, it merges discovered transcripts with store entries by `sessionId`; unmatched transcripts are exposed as separate `agent:<agentId>:<sessionId>` rows.
  - file: [src/gateway/server-methods/usage.ts:409](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/src/gateway/server-methods/usage.ts#L409)

- **Transcript discovery deduplicates by sessionId only:** `discoverAllSessions()` scans transcript files, parses a `sessionId` from each filename, and keeps one discovered record per `sessionId`. It does not compute any stable logical-session family across rebinding or resets.
  - file: [src/infra/session-cost-usage.ts:457](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/src/infra/session-cost-usage.ts#L457)

It does not aggregate multiple historical transcript instances together.
