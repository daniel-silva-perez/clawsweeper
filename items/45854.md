## Summary

Keep open. Current `main` already includes the merged sub-items from the issue (`callLog.search`, `sms.search`, and per-device Android session isolation), but the remaining requested work (`device.control`, an Android ADB tool surface, and dynamic program execution) is not present in the current Android protocol, registry, dispatcher, or docs.

## Evidence

- **Android docs show the shipped command surface only:** The Android platform docs list `device.status`, `device.info`, `device.permissions`, `device.health`, `callLog.search`, and `sms.search` as the expanded Android command families. They do not list `device.control`, any ADB command family, or dynamic program execution.
  - file: [docs/platforms/android.md:181](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/docs/platforms/android.md#L181)
- **Protocol constants include sms/call log and only four device commands:** The Android protocol enum defines `sms.search` and the device commands `device.status`, `device.info`, `device.permissions`, and `device.health`. There is no `device.control` enum and no Android ADB/script execution command family.
  - file: [apps/android/app/src/main/java/ai/openclaw/app/protocol/OpenClawProtocolConstants.kt:54](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/apps/android/app/src/main/java/ai/openclaw/app/protocol/OpenClawProtocolConstants.kt#L54)
- **Registry and dispatcher only advertise/handle the shipped commands:** `InvokeCommandRegistry` advertises the four device commands plus `sms.search` and `callLog.search`, and `InvokeDispatcher` routes only those device commands through `DeviceHandler`. There is no dispatch path for `device.control`, ADB, or dynamic program execution.
  - file: [apps/android/app/src/main/java/ai/openclaw/app/node/InvokeCommandRegistry.kt:157](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/apps/android/app/src/main/java/ai/openclaw/app/node/InvokeCommandRegistry.kt#L157)
- **Per-device Android session isolation is implemented on main:** Android now builds a device-scoped main session key (`agent:<agentId>:node-<deviceId-prefix>`), and the chat session policy tests verify the active session follows that per-device main key when still on the default session.
  - file: [apps/android/app/src/main/java/ai/openclaw/app/SessionKey.kt:21](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/apps/android/app/src/main/java/ai/openclaw/app/SessionKey.kt#L21)
- **No code or docs match the remaining requested features:** A repository search for the unfinished items found no matches in the Android app, Android tests, or Android/node docs for `device.control`, `adb.`, `kotlin-script`, `script-runtime`, or `dynamic program`.
  - command: `rg -n "device\.control|adb\.|kotlin-script|script-runtime|dynamic program" apps/android/app/src/main apps/android/app/src/test docs/platforms/android.md docs/nodes/index.md docs/concepts/features.md`
