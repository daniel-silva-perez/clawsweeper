## Summary

Keep this PR open. Current main still emits `gateway.trusted_proxies_missing` as `warn` only for literal `gateway.bind="loopback"`, and the existing regression test and public audit catalog still lock in/document that behavior. The PR is not obsolete or superseded, but its current shape remains security-sensitive because it expands the change from one severity split into shared bind classification that can suppress critical audit findings for `auto`/`custom` cases that may resolve or fall back to `0.0.0.0`.

## Best Possible Solution

Keep this PR open for maintainer review or revision. The best path is to implement a conservative, source-backed effective-bind classification: downgrade only definitely local-only exposure, fail closed for ambiguous `auto` and fallback-prone `custom` binds, keep `trusted_proxies_missing`, `bind_no_auth`, `loopback_no_auth`, and Control UI origin checks semantically coherent, add regression coverage for loopback, lan, custom loopback, custom exposed/fallback, and auto/container behavior, and update the public audit-check docs if severity changes.

## Risks / Open Questions

- Leaving main unchanged keeps the default loopback Control UI audit path noisy for users affected by the linked report.
- Merging the current PR shape without tightening effective-bind classification could downgrade or suppress critical findings for `auto` or `custom` configurations that resolve or fall back to `0.0.0.0`.
- Same-host reverse proxy deployments make a blanket loopback downgrade a security/product severity-taxonomy decision rather than a purely mechanical false-positive fix.
