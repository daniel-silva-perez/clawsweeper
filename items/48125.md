## Summary

Current `main` already prevents malformed Feishu card-action webhook payloads from reaching the crashing path, so this PR is superseded by an upstream fix on `main`.

## Risks / Open Questions

- The protection currently relies on the production entrypoint validating payload shape before `handleFeishuCardAction` is called; direct internal callers still need to pass a valid `FeishuCardActionEvent`.
- I could not verify a shipped release tag for commit `972c7112a292f2776b8008b8c0e311e64569236f`; the provided latest release `v2026.4.22` predates that commit.

## Close Comment

Closing as implemented on `main` after Codex review. The current Feishu gateway path already rejects malformed card-action webhook payloads before `handleFeishuCardAction` runs: `extensions/feishu/src/monitor.account.ts:243` validates the payload shape, and `extensions/feishu/src/monitor.account.ts:580` logs `ignoring malformed card action payload` and returns early instead of dispatching into the handler. There is also regression coverage in `extensions/feishu/src/monitor.card-action.lifecycle.test-support.ts:226` confirming malformed card-action events are dropped without dispatch. I was able to trace this behavior to commit `972c7112a292f2776b8008b8c0e311e64569236f`, so this PR is superseded by what is already on `main`.
