## Summary

Keep open. Current main still lacks the requested inter-session progress stream for `sessions_send`: the tool defaults to a terminal-only 30s wait, reports `timeout` when `agent.wait` does not finish in time, and has no `sessions_subscribe`, `agent.events`, or `streamProgress` API. Existing websocket/session subscribers and Task Flow/background-task work are adjacent, but they do not give the calling agent live target-run tool/assistant events; `sessions_send` marks these runs as `inter_session`, and the gateway excludes those from background task tracking.

## Best Possible Solution

Keep the issue open. The maintainer path is to add a core/tool-facing progress API for inter-session runs, such as `sessions_subscribe` plus a bounded authorized `agent.events`/long-poll buffer, or extend `sessions_send` so accepted delivery and long-running progress are represented separately from terminal wait timeout. The implementation should include visibility/authorization checks, ACP/Telegram-topic regression coverage, and docs for how callers consume progress without requiring a websocket client.
