## Summary

Current main still does not implement the requested inter-session progress stream for `sessions_send`. The issue remains specific and actionable: `sessions_send` still supports only fire-and-forget or synchronous wait, wait expiry still returns `status: "timeout"`, A2A announce is final-only, and existing gateway event subscriptions are WebSocket/session-subscriber paths rather than an agent-callable run event stream.

## Best Possible Solution

Keep this issue open. Implement an additive core session-event observation path, likely a permission-checked `agent.events` long-poll/read API backed by bounded per-run buffering, plus either a `sessions_subscribe` tool or a `sessions_send` progress option. Also adjust `sessions_send` result semantics so successful delivery is not confused with wait timeout, and add regression coverage for long-running sends, event visibility, routing isolation, and Telegram/topic-bound delivery.
