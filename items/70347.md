## Summary

Keep open. Current main still has the specific gap reported in #70347: the cron outer wall-clock timeout aborts the isolated run and records the cron job failure, but I found no current-main path that emits or synthesizes a terminal lifecycle event at that timeout boundary so the gateway session store finalizes synchronously.

## Best Possible Solution

Implement a first-party cron timeout finalization path: when the outer timeout fires after an isolated cron session has started, preserve or resolve the active session key/run id, emit a terminal lifecycle `error` event with timeout metadata (`errorKind`/`stopReason`), and add regression coverage that verifies the matching `sessions.json` row becomes terminal during the same gateway flow without waiting for reapers/watchdogs. If direct emission from the timer cannot safely know the session identity, return a structured timeout result from the isolated-run layer and synthesize the lifecycle event in the cron result-finalization path.
