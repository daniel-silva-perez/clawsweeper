## Summary

Keep this issue open. The report is recent, concrete, and corroborated, and current main still has the same overflow-compaction-success retry path plus cleanup that depends on the retry attempt unwinding. Current timeout and diagnostic paths may mitigate some hangs, but source/tests do not prove recovery from the reported `processing` with `queueDepth=0` post-compaction lock.

## Best Possible Solution

Keep this issue open as the canonical tracker for the post-overflow-compaction retry/session-lock freeze. The best fix is an integration regression that exercises overflow compaction followed by retry success, retry error, and a hung/lost retry terminalization path, then asserts active-run cleanup, diagnostic state leaving `processing` or entering a bounded abort state, queue depth draining, and follow-up messages proceeding. If current timeout paths are intended to solve this, add source-backed and test-backed proof tying those paths to the reported diagnostics before closing.
