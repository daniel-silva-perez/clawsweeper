## Summary

Keep this issue open. Current main has partial QMD contention mitigations, but it does not close the reported gateway-vs-CLI runtime race. The CLI memory search path still creates and uses a local memory manager in the CLI process, while gateway startup/background memory work creates a separate manager in the gateway process. The QMD update queue is process-local, `qmd update` is not protected by a cross-process file lock, and current tests still encode `SQLITE_BUSY: database is locked` as a possible QMD search/read failure with fallback rather than as a prevented condition.

## Best Possible Solution

Keep this issue open and fix it in the QMD owner path. The likely product direction is a per-agent cross-process coordination seam for QMD maintenance/search, such as a shared file lock or lock-aware skip/wait policy around `qmd update` and foreground CLI search, plus a regression test that models a gateway background update and a separate CLI search contending for the same agent QMD SQLite index. The existing same-process queue, embed-only file lock, read-only `busy_timeout`, and builtin fallback should be preserved as mitigations, but they are not enough to close the reported runtime race.

## Evidence

- **Issue context:** The report is a recent external-author issue with no comments, no protected labels, and a concrete reproduction pattern: QMD backend, gateway running, daily memory files present, documented `memory status --fix` and `memory index --force` tried, then intermittent `qmd update retry ... database is locked` during ordinary `openclaw memory search`.
- **CLI search uses an in-process manager, not a gateway-coordinated search RPC:** `runMemorySearch` resolves config/secrets, builds a CLI session key, then calls `withMemoryManagerForAgent` and `manager.search(...)` directly. That means a foreground `openclaw memory search` has its own process-local QMD manager rather than sharing the running gateway manager's pending update state.
