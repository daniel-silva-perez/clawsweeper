## Summary

Keep this issue open. Current main still has mitigations for QMD SQLite contention, but not a source-backed fix for the reported gateway-vs-CLI runtime race. The CLI search path creates a local memory manager, gateway startup arms separate QMD background managers, QMD update serialization is process-local, search waits only on the same manager's pending update for 500 ms, and tests still treat SQLITE_BUSY as a possible QMD read/search failure with builtin fallback.

## Best Possible Solution

Keep this issue open and fix it in the memory-core QMD owner path. The likely maintainer direction is a per-agent cross-process coordination mechanism around QMD maintenance and foreground search, such as a shared update lock or lock-aware skip/wait policy, plus a regression test that models a gateway background QMD update and a separate CLI `memory search` contending for the same agent index. Preserve the existing same-process queue, read-only `busy_timeout`, embed lock, and builtin fallback as mitigations, but do not treat them as the complete fix.
