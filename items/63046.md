## Summary

Keep open. Current main still hardcodes the image tool timeout to 30 seconds in all three `runImagePrompt()` provider-call paths. The config contract and docs expose `tools.media.image.timeoutSeconds` and per-entry `tools.media.models[].timeoutSeconds`, and the shared media-understanding runner already uses `resolveTimeoutMs`; the PR direction is still relevant but should add regression coverage and clarify the config key mismatch from the linked issue.

## Best Possible Solution

Keep the PR open and revise it into a tested fix: compute one timeout in `runImagePrompt()` with `resolveTimeoutMs(effectiveCfg?.tools?.media?.image?.timeoutSeconds, 30)` or the maintainer-approved equivalent, pass it to all three image-description call paths, and add `image-tool.test.ts` coverage that asserts configured `tools.media.image.timeoutSeconds` is forwarded as milliseconds. Clarify #62944 to use the documented config key, or separately decide whether per-entry `tools.media.models[].timeoutSeconds` should also influence the explicit image tool.
