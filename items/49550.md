## Summary

Keep open. Current main already implements the safer raw JSON `api_error` portion of the MiniMax failover fix, including regression coverage for `unknown error, 520` and guards for billing/auth/non-transient format-like payloads. But current main still does not implement the PR's HTTP 520 status-code handling: `520` is absent from the transient HTTP status set and there is no `status === 520` timeout branch, so the PR is not fully obsolete or implemented on main.

## Best Possible Solution

Keep the PR open, but ask for a rebase onto current main and narrow the patch to the remaining HTTP 520 work: add 520 to the transient HTTP status handling and add a `status === 520` timeout branch, while preserving current main's safer `api_error` transient-signal, billing/auth, and format/schema guards. Add the 520 status tests to the existing `src/agents/pi-embedded-helpers.isbillingerrormessage.test.ts` coverage.
