## Summary

Keep this PR open. Current main still leaves the central logs.tail hardening gap for Basic auth headers, standalone JWTs, and Pomerium JWT assertion headers, and the linked bug remains open. The PR is a relevant implementation candidate, but it should be amended before merge because the Basic-auth regex can leave padding unredacted and the tests do not fully isolate the new JWT/default-redaction behavior.

## Best Possible Solution

Keep the PR open as the implementation candidate for the linked logs.tail redaction bug, but request a small amendment before merge: remove or replace the Basic-auth trailing word boundary, add focused regression tests for padded Basic auth, standalone JWTs without a Bearer prefix, x-pomerium-jwt-assertion, and the logs.tail/redactSensitiveLines path, then run the targeted logging tests.

## Risks / Open Questions

- Current main can still return unredacted Basic auth values, standalone JWTs, or Pomerium JWT assertion headers through logs.tail if those values reach file logs.
- The PR's Basic auth pattern uses a trailing word boundary after a base64 character class, which can leave trailing '=' padding outside the masked span.
- The PR's submitted JWT test uses an Authorization: Bearer prefix, so it can pass through the existing Bearer redaction path without proving standalone JWT redaction.
- The PR tests are added under the redactSensitiveLines describe block while calling redactSensitiveText, which weakens coverage organization and makes the intended behavior less clear.
