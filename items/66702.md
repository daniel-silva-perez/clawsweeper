## Summary

Keep open. Current main partially addresses the broader stale-image family via #71868 by scrubbing older replay media markers, but it does not implement this issue’s central requested behavior: when the current turn has media, vision input is not hard-scoped to current-turn media. `detectAndLoadPromptImages()` still scans the whole submitted prompt, hook-prepended context can still contain `[media attached: ...]` text inside memory/untrusted blocks, and `pruneProcessedHistoryImages()` intentionally preserves the three most recent completed turns and their image/media markers. The existing tests cover old-history pruning and recent-history preservation, not the reported exclusion of `<relevant-memories>` / `Untrusted context` blocks from image detection.

## Best Possible Solution

Keep this issue open and implement a focused embedded-runner/media fix. The likely path is to make prompt image hydration current-turn aware: when the current turn has explicit attachments, load only those current-turn refs unless the user asks to compare prior images; exclude known injected context blocks such as `<relevant-memories>` and `Untrusted context` from prompt-ref detection; and add regression tests covering stale media refs inside memory/untrusted blocks plus recent image-heavy session history.
