## Summary

Keep PR #50875 open. Current main still has only static `session.identityLinks` for DM identity unification and does not expose a `before_identity_resolve` hook, `resolveInboundPeerIdentity` SDK helper, or `resolvedPeerId` route input. This is a real core/plugin API gap rather than ClawHub-only work because external plugins cannot change session-key construction before routing without a core seam. The PR discussion also contains useful review follow-up: the author addressed the P1 canonical-ID normalization issues in commit 676886ac5072d291bf37c9afd2737300353f8314, while P2 gaps around blank hook results and secondary Slack/WhatsApp routing paths remain relevant before merge.

## Best Possible Solution

Keep the PR open, but require an updated implementation against current main. The best path is a core plugin-API addition: define `before_identity_resolve` in the current hook type/runner files, expose a narrow SDK helper, thread a `resolvedPeerId` or equivalent into route/session-key construction, and integrate every direct-message routing path that can create or reuse a session, including Slack DMs/slash/system-event paths, MS Teams DMs, WhatsApp direct and broadcast paths, and any other channel surfaces that use `identityLinks`. Before merge, add focused tests for precedence over `identityLinks`, blank-result fallback across multiple hooks, canonical IDs such as email/E.164/`main`, cache bypass, fail-open errors, and secondary routing consistency; update public session/plugin docs after the API shape is stable.

## Risks / Open Questions

- The PR is stale against current main: hook typing currently lives in `src/plugins/hook-types.ts`, while the provided PR diff modifies older hook/type locations, so a rebase will need careful conflict resolution.
- The remaining review comments are not cosmetic. Blank `canonicalPeerId` handling can block lower-priority resolvers, and secondary Slack/WhatsApp routing paths can split transcript/session state if they continue using raw peer IDs.
- A dynamic identity hook adds an awaited plugin seam to inbound direct-message routing; the final design should keep fail-open behavior, test hook ordering, and document timeout/caching expectations for plugin authors.
