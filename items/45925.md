## Summary

Keep open. Current main does not contain the proposed cron-session guard, and the PR’s described wake-layer change appears to drop the only targeting data that exec-triggered heartbeat wakes currently carry.

## Evidence

- **Main still preserves forced cron session keys in wake scoping:** `scopedHeartbeatWakeOptions` on current `main` returns `{ ...wakeOptions, sessionKey }` for any parseable agent session key. There is no cron-session exclusion here, so the requested behavior is not implemented on main.
  - file: [src/routing/session-key.ts:35](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/src/routing/session-key.ts#L35)
  - command: `nl -ba src/routing/session-key.ts | sed -n '28,45p'`
  - sha: [f03252aaf9cc](https://github.com/openclaw/openclaw/commit/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022)
- **Main still accepts forced cron session keys in heartbeat session resolution:** `resolveHeartbeatSession` only rejects forced subagent sessions. The forced-session path canonicalizes and returns any non-subagent agent session, which includes cron run sessions on current main.
  - file: [src/infra/heartbeat-runner.ts:273](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/src/infra/heartbeat-runner.ts#L273)
  - command: `nl -ba src/infra/heartbeat-runner.ts | sed -n '245,315p'`
  - sha: [f03252aaf9cc](https://github.com/openclaw/openclaw/commit/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022)
- **Unscoped wake requests fan out beyond one target:** In the heartbeat runner, a wake with `sessionKey` or `agentId` takes the targeted path, but without either the runner falls through to iterating all enabled agents. That means dropping the cron `sessionKey` without preserving `agentId` changes behavior materially.
  - file: [src/infra/heartbeat-runner.ts:1484](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/src/infra/heartbeat-runner.ts#L1484)
  - command: `nl -ba src/infra/heartbeat-runner.ts | sed -n '1465,1525p'`
  - sha: [f03252aaf9cc](https://github.com/openclaw/openclaw/commit/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022)
- **Exec-triggered heartbeat wakes currently carry only session scope:** The exec completion path calls `requestHeartbeatNow(scopedHeartbeatWakeOptions(sessionKey, ...))` and does not pass `agentId`. If the PR strips cron `sessionKey` at this layer, the wake becomes unscoped.
  - file: [src/agents/bash-tools.exec-runtime.ts:305](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/src/agents/bash-tools.exec-runtime.ts#L305)
  - command: `nl -ba src/agents/bash-tools.exec-runtime.ts | sed -n '300,308p'`
  - sha: [f03252aaf9cc](https://github.com/openclaw/openclaw/commit/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022)
- **Current tests do not cover the proposed cron fallback case on main:** The session-key tests cover `isCronSessionKey`, but there is no `scopedHeartbeatWakeOptions` cron regression test on current main, and the heartbeat runner tests around forced session overrides cover non-cron override sessions only.
  - file: [src/routing/session-key.test.ts:60](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/src/routing/session-key.test.ts#L60)

## Risks / Open Questions

- The underlying report still looks plausible on current main because both the wake-scoping helper and forced-session resolver still preserve cron run session keys.
- The PR, as described in the issue context, appears to introduce a new targeting regression for cron-originated exec wakes by removing `sessionKey` without preserving `agentId`.
