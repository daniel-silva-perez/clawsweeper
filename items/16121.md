## Summary

Keep open. Current `main` still escapes raw HTML in the normal Telegram markdown renderer, so `<tg-emoji emoji-id="...">` is not supported in ordinary bot replies/messages. OpenClaw does have lower-level HTML passthrough paths for Telegram sends, but the public message/action surfaces inspected here do not expose a way to use that for general bot output, and there is no markdown-level custom-emoji syntax on `main`.

## Evidence

- **Default Telegram renderer escapes raw HTML:** `markdownToTelegramHtml()` renders markdown through `escapeHtml`, and the Telegram formatting tests still assert that raw HTML like `<b>nope</b>` is escaped instead of preserved. That means a literal `<tg-emoji ...>` tag in normal markdown-mode output would be escaped, not sent as Telegram HTML.

## Risks / Open Questions

- There is partial adjacent support through explicit Telegram html-mode send paths, so an implementer will need to decide whether this issue should expose that capability broadly or only for specific tool/plugin surfaces.
- The request mixes three scopes: raw `<tg-emoji>` HTML passthrough, recognition of Telegram custom emoji identifiers, and a markdown-level authoring syntax. `main` does not clearly implement that full set today.
