## Summary

Keep PR #49396 open. Current main still computes Peak Error Hours by proportionally spreading each session's total errors and messages across its active hour slices, and the backend/API types do not expose the PR's proposed precise hourly or quarter-hour message counts. The PR is a plausible live fix for a core UI bug, but it still needs the maintainer-requested date/changelog follow-ups and the DST/per-bucket local-offset review concern addressed before merge.

## Best Possible Solution

Keep #49396 open and have the author update the patch: add the precise UTC quarter-hour message-count contract and regression tests, compute local-hour mapping from each bucket's own UTC datetime rather than a single current offset, align the `date` comment/semantics, and add the required changelog entry. After those changes, run the UI and infra changed lanes before merge.
