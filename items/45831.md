## Summary

Keep open. Current `main` still does not implement this PR’s two concrete hardening changes: `MediaAttachmentCache` reads local files through its own `readLocalBuffer()` path instead of `readLocalFileSafely()`, and temp attachment writes still omit an explicit `0o600` mode.

## Evidence

- **Attachment cache still uses the custom local-read helper:** `getBuffer()` calls `this.readLocalBuffer(...)` for local attachments rather than `readLocalFileSafely(...)`.
  - file: [src/media-understanding/attachments.cache.ts:115](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/src/media-understanding/attachments.cache.ts#L115)
  - command: `nl -ba src/media-understanding/attachments.cache.ts | sed -n '105,125p'`
  - sha: [f03252aaf9cc](https://github.com/openclaw/openclaw/commit/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022)
- **Current helper is not the same safe-read path the PR proposes:** `readLocalBuffer()` does open with `O_NOFOLLOW` and canonicalize the path, but it does not do the inode/path-identity checks that `openVerifiedLocalFile()` performs via `sameFileIdentity()` before and after resolving the real path.
  - file: [src/media-understanding/attachments.cache.ts:379](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/src/media-understanding/attachments.cache.ts#L379)

- **Safe-read helper has stronger integrity checks that are still unused here:** `openVerifiedLocalFile()` verifies the opened fd against both `lstat/stat(filePath)` and `stat(realPath)` using `sameFileIdentity()`, which is the extra protection the PR is trying to reuse through `readLocalFileSafely()`.
  - file: [src/infra/fs-safe.ts:161](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/src/infra/fs-safe.ts#L161)
  - command: `nl -ba src/infra/fs-safe.ts | sed -n '107,205p'`

  - file: [src/media-understanding/attachments.cache.ts:244](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/src/media-understanding/attachments.cache.ts#L244)
