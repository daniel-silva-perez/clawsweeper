## Summary

Keep open. Current main still only partially addresses stale deleted-agent state: agent deletion purges core config/session-store/files, but it does not clean cron jobs or plugin/context-engine-owned state, and there is no first-class agent reset command. The issue’s central multi-location cleanup/reset request remains actionable in core.

## Best Possible Solution

Keep this issue open. The best fix is a scoped core cleanup/repair path: extend `agents delete`/`agents.delete` or add `agents reset --clean-state` to purge core session-store/transcript state, remove/disable/report cron jobs whose `agentId` targets the deleted agent, emit appropriate session/plugin lifecycle cleanup for purged sessions, and add a versioned context-engine/plugin hook so Lossless Claw and other engines can clean agent/session-owned state themselves. Add regression coverage for CLI deletion, Gateway deletion, cron jobs targeting the deleted agent, and plugin/context-engine cleanup behavior.

## Risks / Open Questions

- The exact `TypeError: Cannot read properties of undefined (reading 'length')` was not reproduced in this read-only review; a fix should add a focused fixture or logs-backed repro before closing.
- Cron cleanup needs maintainer product judgment: deleting an agent could remove matching jobs, disable them, or fail/report actionable cleanup instead.
