## Summary

Keep open. Current `main` already uses user-scoped `systemctl`, so the title's narrow fix is outdated, but the underlying Ubuntu/headless install bug is still plausibly present: the unit file path is derived from the current HOME while `systemctl` under `sudo` is still redirected to `SUDO_USER` via `--machine ... --user`, and `XDG_RUNTIME_DIR` is still not injected.

## Evidence

- **User unit path still comes from resolved HOME:** `resolveSystemdUnitPathForName` writes the gateway unit under `<home>/.config/systemd/user/<name>.service`, so the install target depends on the current process home directory.
  - file: [src/daemon/systemd.ts:47](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/daemon/systemd.ts#L47)
  - command: `nl -ba src/daemon/systemd.ts | sed -n '40,48p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **`sudo` path still redirects `systemctl` to the invoking user:** `resolveSystemctlMachineScopeUser` prefers `SUDO_USER`, and `execSystemctlUser` immediately runs `systemctl --machine <user>@ --user ...` when `SUDO_USER` is non-root. That can diverge from the unit path above if the install ran as root.
  - file: [src/daemon/systemd.ts:333](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/daemon/systemd.ts#L333)

  - file: [src/daemon/systemd.ts:266](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/daemon/systemd.ts#L266)
