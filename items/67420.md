## Summary

Keep this PR open. Current main still lacks per-agent dreaming control, the PR is paired with the same-author open feature issue #67413, and the PR is a plausible implementation candidate. It still needs maintainer review and likely a fix for the all-agents-disabled fallback behavior before merge.

## Best Possible Solution

Continue reviewing this PR or a successor against #67413. The implementation should add agents.list[].dreaming to the exported config type, strict Zod schema, generated schema/help/docs surfaces, and memory dreaming workspace resolution; preserve default-agent fallback only when no agents are configured; skip explicitly disabled agents without falling back when all configured agents opt out; and add tests for mixed enabled/disabled agents, all-disabled agents, and config validation. The current diff has no obvious supply-chain or secrets regression, so the main blocker is functional correctness and maintainer product scope.

## Risks / Open Questions

- The current PR patch appears to skip disabled entries before the existing default-agent fallback. If every configured agent opts out, agentIds becomes empty and the helper can still fall back to the default agent, re-enabling dreaming unexpectedly. That needs an explicit all-disabled test and behavior decision before merge.
- The linked feature issue also asks for per-agent schedule support. Maintainers should decide whether this PR is intentionally limited to enable/disable control or should leave schedule support to a narrower follow-up.
