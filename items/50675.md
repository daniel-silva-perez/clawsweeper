## Summary

Current `main` does not implement this PR’s requested behavior. Plugin command handling still forces `shouldContinue: false`, the public SDK type does not expose a `shouldContinue` field on `PluginCommandResult`, and current docs still describe plugin commands as bypassing the LLM agent.

## Evidence

- **Plugin command handler still hardcodes terminal behavior:** `handlePluginCommand` executes the matched plugin command and always returns `{ shouldContinue: false, reply: result }`, so plugin commands cannot hand control back to the agent on current `main`.
  - file: [src/auto-reply/reply/commands-plugin.ts:59](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/src/auto-reply/reply/commands-plugin.ts#L59)
  - command: `nl -ba src/auto-reply/reply/commands-plugin.ts | sed -n '1,140p'`
  - sha: [ebd7f19a3b20](https://github.com/openclaw/openclaw/commit/ebd7f19a3b20001960474ec9ba9cc424e45f7b23)
- **Public SDK type still lacks `shouldContinue`:** `PluginCommandResult` is still just `ReplyPayload`, and the `registerCommand` JSDoc still says custom commands 'bypass the LLM agent'. That means the proposed API surface from the PR is not present on `main`.
  - file: [src/plugins/types.ts:1812](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/src/plugins/types.ts#L1812)

## Risks / Open Questions

- Closing this PR as implemented would be incorrect; the requested SDK/runtime behavior is absent from current `main`.
- Plugin authors still cannot build command flows that mutate agent state and then resume the agent in the same turn.
- Docs currently reinforce the old terminal-command model, so closing would also leave discoverability/documentation gaps unresolved.
