## Summary

Keep this issue open. Current main has simple Teams thread-session suffixing, but the specific malformed mixed-key failure is still plausible: `resolveAgentRoute()` caches the same route object returned to callers, and the Teams handler mutates `route.sessionKey` in place after adding the thread suffix. A later Teams message in another thread of the same channel can therefore start from a cached old-thread key and append the new thread id, producing the reported double `:thread:` shape.

## Best Possible Solution

Fix Teams thread routing so cached base routes remain immutable. The narrowest fix is to avoid mutating the object returned by `resolveAgentRoute()` in the Teams handler: compute a separate thread-qualified session key from a clean base route and pass that through context, session recording, dispatch, and logging. Also harden either the Teams helper or the shared thread helper against already thread-qualified bases, and add a regression test that sends two Teams channel replies in the same channel with different thread ids and asserts the second key is `base:thread:new`, never `base:thread:old:thread:new`.
