## Summary

Current main does not implement outbound/native WhatsApp location messages through the message tool. It only parses inbound channel locations. The PR is focused and still potentially useful, but the provided review discussion identifies unresolved P1 wiring gaps, so there is no conservative auto-close reason.

## Best Possible Solution

Keep the PR open for maintainer/product judgment. The best implementation path is to make native WhatsApp location sending a plugin-owned WhatsApp message action using describeMessageTool and handleAction, with only the minimal shared core/API seam needed for the action name if maintainers approve it. It should advertise the action in WhatsApp discovery, execute through the production gateway/runtime path, include schema validation for coordinate ranges, avoid raw coordinate logs, and add regression coverage for schema discovery, gateway dispatch, and Baileys payload construction.

## Risks / Open Questions

- Merging the PR as described would still risk a feature that normal configured agents cannot discover or deliver because of the discovery, production outbound, and gateway gaps called out in review.
- Precise coordinates are sensitive data; any implementation should avoid logging raw latitude/longitude, as noted in the PR review.
- The current action-name surface is a closed core union. If maintainers do not want a generic location action, the best path may require a deliberate plugin API extension rather than a one-off core action.
