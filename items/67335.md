## Summary

Keep open. Current main has meaningful macOS launchd hardening for normal restart and installed-but-unloaded recovery, but the newer split-brain symptom from the discussion is not fully reconciled: when the LaunchAgent is not loaded but a verified gateway listener is still alive, `gateway restart` takes an unmanaged SIGUSR1 path and does not restore launchd supervision, and `status --deep` can still report mixed service/RPC/listener signals.

## Best Possible Solution

Keep this issue open and narrow it around launchd/listener reconciliation. The best fix is in the gateway daemon lifecycle/status surface: when the LaunchAgent plist exists but launchd is not loaded and a verified gateway listener is alive, `openclaw gateway restart` should restore supervision deterministically, or stop/adopt the unmanaged listener before re-bootstrap, and `gateway status --deep` should print one coherent recovery path. Add a macOS launchd integration test for not-loaded + RPC OK + listener on 127.0.0.1:18789, then verify restart ends with `launchctl print gui/$UID/ai.openclaw.gateway` loaded and the gateway healthy.

## Evidence

That matches the fresh comment's split-brain state but leaves the LaunchAgent unrepaired on that path.
