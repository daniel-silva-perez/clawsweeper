## Summary

Keep #71744 open. Current main has useful model-fallback diagnostics in structured subsystem logs and FallbackSummaryError attempts, but the requested runtime-event/trajectory source of truth is still missing. The fallback runner does not emit a `fallback_step` or equivalent runtime trajectory event carrying primary failure detail through a later fallback failure. Related #71710 explains the incident's host pressure root cause, and #62684 is a different closed Ollama timeout/no-provider-logs report, not a canonical replacement for this observability gap.

## Best Possible Solution

Keep the issue open and implement a core fallback-step observability path. A good fix would plumb an optional runtime event sink through `runWithModelFallback` or the reply runner, emit one event per transition with requested/from/to provider-model, reason/detail/status/code, chain position, and final outcome, write it into OpenClaw trajectory/runtime JSONL, and add tests for primary failure followed by fallback failure so both root causes remain visible.
