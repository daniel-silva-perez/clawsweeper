## Summary

Keep open. Current `main` still appears to reject chained forms like `cd <dir> && python script.py` / `cd <dir> && node script.js`: script-target extraction only handles direct interpreter argv, and the fallback preflight still throws the exact `complex interpreter invocation detected` error for complex shell syntax with interpreter+script hints.

## Evidence

- **Direct script extraction is limited to direct interpreter argv:** `extractScriptTargetFromCommand()` tokenizes the whole command once and only returns a target when `extractInterpreterScriptTargetFromArgv()` sees `python...` or `node` as the executable. A command starting with `cd` will return `null` here rather than extracting the later script step.
  - file: [src/agents/bash-tools.exec.ts:356](https://github.com/openclaw/openclaw/blob/ecac696643c604c6d854341bfe6424bfa31423e9/src/agents/bash-tools.exec.ts#L356)
  - command: `nl -ba src/agents/bash-tools.exec.ts | sed -n '356,428p'`
  - sha: [ecac696643c6](https://github.com/openclaw/openclaw/commit/ecac696643c604c6d854341bfe6424bfa31423e9)
- **Fallback preflight still throws the reported error:** When no direct script target is found, `validateScriptFileForShellBleed()` calls `shouldFailClosedInterpreterPreflight()` and throws `exec preflight: complex interpreter invocation detected; refusing to run without script preflight validation. Use a direct `python <file>.py` or `node <file>.js` command.` when complex syntax and interpreter/script hints are present.
  - file: [src/agents/bash-tools.exec.ts:941](https://github.com/openclaw/openclaw/blob/ecac696643c604c6d854341bfe6424bfa31423e9/src/agents/bash-tools.exec.ts#L941)

  - sha: [ecac696643c6](https://github.com/openclaw/openclaw/commit/ecac696643c604c6d854341bfe6424bfa31423e9)
- **Current tests codify fail-closed behavior for multi-step interpreter shells:** The script-preflight tests explicitly expect rejection for shell-wrapped or multi-step interpreter invocations such as `bash -c "python bad.py"`, `env bash -c "python bad.py"`, and control-flow forms, confirming that current `main` intentionally still blocks non-direct interpreter/script execution patterns rather than allowing them.
  - file: [src/agents/bash-tools.exec.script-preflight.test.ts:452](https://github.com/openclaw/openclaw/blob/ecac696643c604c6d854341bfe6424bfa31423e9/src/agents/bash-tools.exec.script-preflight.test.ts#L452)

- **Issue discussion shows multiple matching reports:** The provided issue comments report the same behavior for `cd /path && node script.js`, note that `cd` never matches the binary allowlist because it is a shell builtin, and another commenter says it started after version `2026.4.10 (44e5b62)`. That discussion is consistent with the current code path still rejecting chained interpreter commands.
