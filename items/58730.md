## Summary

Keep open. Current main has substantial related work, including optional sandbox backends, exec approvals, session persistence, and auto-compaction, but the issue still contains plausible unresolved core requests, especially the specific host exec OS-level sandbox/fallback model and per-skill permission semantics. The related sandbox-exec PR is still open, so this is not a high-confidence conservative close.

## Evidence

- **Current sandboxing is optional, not a complete replacement for the requested default host exec isolation:** The public sandboxing docs say OpenClaw can run tool execution including exec in sandbox backends, but also explicitly say sandboxing is optional and tools run on the host when sandboxing is off. This supports that main has related functionality, but does not prove the requested bwrap/sandbox-exec style host exec isolation/fallback is fully implemented.
  - file: [docs/gateway/sandboxing.md:93](https://github.com/openclaw/openclaw/blob/d6ef1fcf24eea3a5985d653a3baf400e32aa1b75/docs/gateway/sandboxing.md#L93)
  - sha: [d6ef1fcf24ee](https://github.com/openclaw/openclaw/commit/d6ef1fcf24eea3a5985d653a3baf400e32aa1b75)
- **Main has a substantial exec approval model:** Exec approvals are documented as a host guardrail where policy, allowlist, and optional user approval must agree; effective policy is the stricter of tools.exec and host approvals defaults. This addresses much of the safety concern but is not exactly the proposed per-skill automatic/askOnce/askAlways/deny model.
  - file: [docs/tools/exec-approvals.md:251](https://github.com/openclaw/openclaw/blob/d6ef1fcf24eea3a5985d653a3baf400e32aa1b75/docs/tools/exec-approvals.md#L251)
  - sha: [d6ef1fcf24ee](https://github.com/openclaw/openclaw/commit/d6ef1fcf24eea3a5985d653a3baf400e32aa1b75)
- **Current exec approval knobs differ from the requested model:** The current documented knobs are security deny/allowlist/full, ask off/on-miss/always, and askFallback deny/allowlist/full. That is close to parts of the request, but not a direct implementation of the issue's proposed layered per-skill permission field.
  - file: [docs/tools/exec-approvals.md:390](https://github.com/openclaw/openclaw/blob/d6ef1fcf24eea3a5985d653a3baf400e32aa1b75/docs/tools/exec-approvals.md#L390)
  - sha: [d6ef1fcf24ee](https://github.com/openclaw/openclaw/commit/d6ef1fcf24eea3a5985d653a3baf400e32aa1b75)
- **Session persistence and compaction pieces appear implemented:** The session management docs describe persisted sessions.json plus JSONL transcripts, persistent compaction entries, and auto-compaction on overflow or threshold maintenance. This supports the issue comment's note that those portions already have current infrastructure.
  - file: [docs/reference/session-management-compaction.md:120](https://github.com/openclaw/openclaw/blob/d6ef1fcf24eea3a5985d653a3baf400e32aa1b75/docs/reference/session-management-compaction.md#L120)

An open related PR is strong evidence that the exact sandbox-exec proposal has not been conclusively landed or superseded by maintainer decision in the issue discussion provided.
