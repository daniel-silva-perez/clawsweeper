## Summary

Keep open. Current main still has the reported channel auto-reply tool-policy gap, and the latest PR is a plausible security-hardening candidate, but it still needs maintainer/security review for scope, memory-flush behavior, and runtime owner-only enforcement. The Greptile provider-specific-deny replacement concern appears addressed by the later PR commit, but subsequent review comments and current source inspection show unresolved regressions if the broad default deny lands as-is.

## Best Possible Solution

Keep this PR open for maintainer/security review. Revise the implementation so dangerous filesystem and exec-capable tools are denied for untrusted external channel auto-reply contexts, while internal webchat, heartbeat/cron/exec-event runs, and memory-flush read/write behavior are explicitly preserved. If exec owner-only hardening remains in scope, enforce it through the runtime owner-only policy or exec tool metadata, and review the final shape together with #64718.

## Risks / Open Questions

- Closing this PR now would leave current main's external-channel dangerous-tool exposure unchanged.
- Merging the latest diff as-is may remove `exec`, `read`, `write`, and edit tools from trusted internal providers such as webchat and system-event runs, outside the stated untrusted-channel threat model.
- The PR's `exec` owner-only defense-in-depth can create a false sense of enforcement unless it is wired through `src/agents/tool-policy.ts` or the exec tool's runtime metadata.
- Channel memory-flush runs depend on the `read`/`write` tool subset; a broad message-provider deny can prevent persistence unless the final policy explicitly preserves that path.
- This PR is paired with open same-author PR #64718, so maintainer review should consider both patches as one threat-model cleanup.
