## Summary

Keep open. Current `main` still has the original stale-subsystem-logger behavior, and the proposed PR has unresolved review findings on both correctness and hot-path performance, so there is not strong evidence for a conservative cleanup close.

## Evidence

- **Current main still caches subsystem file loggers indefinitely:** `createSubsystemLogger()` stores `fileLogger` in a closure and only initializes it when null before writing. There is no refresh path when the parent logger is rebuilt, so an existing subsystem child logger can outlive a date-rolled parent logger.
  - file: [src/logging/subsystem.ts:327](https://github.com/openclaw/openclaw/blob/ecac696643c604c6d854341bfe6424bfa31423e9/src/logging/subsystem.ts#L327)
  - command: `nl -ba src/logging/subsystem.ts | sed -n '325,354p'`
  - sha: [ecac696643c6](https://github.com/openclaw/openclaw/commit/ecac696643c604c6d854341bfe6424bfa31423e9)
- **Parent logger can rebuild on rolling-path change:** `resolveSettings()` defaults the file target to `defaultRollingPathForToday()`, and `getLogger()` rebuilds the cached parent logger whenever resolved settings change. That means a date roll changes the parent logger independently of any already-cached subsystem child.
  - file: [src/logging/logger.ts:237](https://github.com/openclaw/openclaw/blob/ecac696643c604c6d854341bfe6424bfa31423e9/src/logging/logger.ts#L237)
  - command: `nl -ba src/logging/logger.ts | sed -n '89,129p'; nl -ba src/logging/logger.ts | sed -n '237,245p'; nl -ba src/logging/logger.ts | sed -n '337,339p'`
  - sha: [ecac696643c6](https://github.com/openclaw/openclaw/commit/ecac696643c604c6d854341bfe6424bfa31423e9)
- **Main branch has no generation/refresh state for subsystem child loggers:** `loggingState` on current `main` has cached parent logger/settings fields but no generation counter or equivalent mechanism to invalidate cached subsystem file loggers after a parent rebuild.
  - file: [src/logging/state.ts:1](https://github.com/openclaw/openclaw/blob/ecac696643c604c6d854341bfe6424bfa31423e9/src/logging/state.ts#L1)

The added staleness/regeneration tests described in PR #62458 are not present on `main`, so there is no current regression test proving the bug is fixed elsewhere.
