## Summary

Keep #70296 open. Current main appears to have improved the Control UI model read-back path, but the umbrella issue still includes unresolved restart-sentinel channel-notification behavior: config/apply, update, and /restart still write sentinel payloads with doctor hints, startup still delivers the formatted restart notice to the originating channel when routing is available, and I found no config schema/key for suppressing those channel notices. BOOT.md also still uses the single supported silent token `NO_REPLY`; `NO_REPLY_NEEDED` is not recognized in current docs/source, so that part needs maintainer clarification or migration guidance rather than auto-close.

## Best Possible Solution

Keep the issue open or split it into narrower follow-ups. The best core fix is to add an explicit, documented restart-sentinel channel-notice policy in config and thread it through sentinel formatting/delivery before `deliverRestartSentinelNotice`, with tests for WhatsApp/Telegram-style routed restarts. For BOOT.md, either document/migrate `NO_REPLY_NEEDED` to the current `NO_REPLY` token or add an intentional compatibility alias if maintainers still consider that token public. The Control UI model display path can likely be marked as addressed separately once a targeted regression test covers object-form `agents.defaults.model.primary` and per-agent `agents.list[].model.primary` read-back.
