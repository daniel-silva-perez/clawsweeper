## Summary

Keep open. Current `main` does not contain the proposed worker-mode entry point, and the PR still has unresolved correctness risks around worker startup order and container-targeted execution, so there is not strong evidence to close it conservatively.

## Evidence

- **Feature is absent on current main:** A read-only search of `src/cli` and `src/infra` found no `worker-mode` or `agent-process-factory` implementation on `main`, so this cannot be closed as already implemented.
  - command: `find src/cli src/infra -maxdepth 2 \( -name '*worker*' -o -name '*agent-process*' \) -print`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **CLI still routes container targets before any worker-specific handling:** `runCli()` calls `maybeRunCliInContainer(originalArgv)` and returns before later startup stages. The PR's proposed `--mode=worker` hook would need to coexist with this ordering, but current main has no such branch.
  - file: [src/cli/run-main.ts:150](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/cli/run-main.ts#L150)
  - command: `nl -ba src/cli/run-main.ts | sed -n '150,210p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Container exec path does not forward worker env:** The container handoff only injects `OPENCLAW_CONTAINER_HINT` and `OPENCLAW_CLI_CONTAINER_BYPASS`, then clears `OPENCLAW_CONTAINER`. It does not forward the worker env vars described in the PR (`OPENCLAW_TEAM_NAME`, `OPENCLAW_MEMBER_NAME`, `OPENCLAW_ROLE`, `OPENCLAW_CONFIG_PATH`, `OPENCLAW_NOTIFY_PORT`), so container-targeted worker startup would still be broken.
  - file: [src/cli/container-target.ts:126](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/cli/container-target.ts#L126)
  - command: `nl -ba src/cli/container-target.ts | sed -n '126,162p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Existing plugin registration happens later than the PR's early return point:** On current main, plugin-related CLI registration happens during program build and subcommand registration. In the PR context, `src/cli/worker-mode.ts` starts with `let _factory: WorkerFactory | null = null`, and `run-main.ts` adds an early return immediately after `assertSupportedRuntime()`. That means the proposed worker path would run before the normal plugin-loading path that could populate the factory.
  - file: [src/cli/program/build-program.ts:9](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/cli/program/build-program.ts#L9)
  - command: `nl -ba src/cli/program/build-program.ts | sed -n '1,29p'; nl -ba src/cli/program/register.subclis-core.ts | sed -n '28,40p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)

## Risks / Open Questions

- Closing this PR would drop work that is still not present on `main` and therefore is not cleanup-safe under the conservative bar.
- Merging the PR as described still appears risky because worker startup depends on initialization and environment propagation paths that are not yet convincingly wired end-to-end.
