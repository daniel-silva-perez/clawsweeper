## Summary

Keep open. Current main still has the reported Feishu write behavior: `clearDocumentContent` performs one non-paginated `documentBlock.list` call, deletes only that returned range, and `feishu_doc` write still routes through that clear path before inserting replacement content. The EronFan comment and external fork PR are useful prior art, but the pagination fix is not present in this repository's current main.

## Best Possible Solution

Keep the issue open and land a focused Feishu plugin fix in `extensions/feishu/src/docx.ts`: paginate `client.docx.documentBlock.list` inside `clearDocumentContent` until no next page token remains, delete the full top-level range, and add a regression test where `feishu_doc` write sees multiple list pages and clears all old top-level blocks before inserting replacement content. The EronFan comment and external fork PR can be used as prior art, but the fix still needs to land in `openclaw/openclaw` main.
