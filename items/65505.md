## Summary

Current main has adjacent timeout/dedupe hardening, but it does not implement the requested phase-aware timeout attribution. The linked PR #65511 was closed unmerged, and current code still exposes only generic `agent.wait` timeout snapshots while auth-profile timeout handling intentionally avoids cooldown rather than gating cooldown on provider-started state.

## Best Possible Solution

Keep this open. The best fix is an additive embedded-run timing model that records queue wait, session-lock wait, preflight, provider start/first token/provider elapsed, post-turn cleanup, and gateway-draining state; carries a `timeoutPhase`/reason through lifecycle, dedupe, and `agent.wait`; and only marks auth-profile timeout failures when a provider request actually started. Add focused regression tests for queued/preflight no-cooldown, provider-started timeout cooldown/rotation, and terminal dedupe preference during post-turn cleanup.

- Cooldown changes need tests that distinguish queued/preflight/post-turn timeouts from provider-started timeouts to avoid reintroducing auth-profile poisoning.
