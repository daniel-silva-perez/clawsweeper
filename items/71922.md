## Summary

Keep open. Current main still implements the behavior described by the issue: model fallback/provider runtime classification collapses an empty signal, the explicit "unknown error (no error details in response)" payload, and the final unmatched catchall to `unknown`. The public failover reason type and cooldown policy also do not include `empty_response`, `no_error_details`, or `unclassified`, so the requested observable split is not shipped.

## Best Possible Solution

Keep the issue open and implement the split in the core model fallback/failover path: extend the failover/provider failure reason contracts with `empty_response`, `no_error_details`, and `unclassified`; update the classifier branches; decide transient cooldown/probe behavior for each new reason; preserve useful error previews for unmatched raw messages; and add focused regression tests for the three collapsed paths plus affected cron/log/fallback surfaces if their schemas expose the reason set.
