## Summary

Current main has adjacent rate-limit failover improvements, but it does not implement the requested dynamic model-tier switching policy. Model config is still a static primary/fallback list, runtime behavior is ordered reactive failover, and docs/tests cover cooldowns and fallback probes rather than health-aware round-robin, downgrade thresholds, proactive rate-limit headers, or gradual upgrade-back behavior.

## Best Possible Solution

Keep this issue open for maintainer product/design work in the core model routing and auth-profile fallback layer. A good implementation would define a documented policy such as failover versus health-aware/round-robin routing, add per-provider/model/profile health state with mark-success/mark-failure recovery semantics, expose observable downgrade and upgrade-back events, update config schema/docs, and add regression tests. The aashari fork is useful prior art but should not be treated as upstream implementation.
