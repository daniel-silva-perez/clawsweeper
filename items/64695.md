## Summary

Keep this issue open. Current main still uses SQLite-only default stores for the task registry and task-flow registry, and `requireNodeSqlite()` still throws the reported missing-`node:sqlite` error. The scheduled sweep rejection path is now guarded, which may reduce the unhandled-rejection crash symptom, but there is still no runtime availability probe, JSON fallback, or replacement backend, so the reported Homebrew runtime still cannot restore or persist task state reliably.

## Best Possible Solution

Keep this issue open. Implement runtime-safe storage selection for both the task registry and task-flow registry: detect unavailable `node:sqlite` before default store use, choose a maintainer-approved durable fallback or dependency, preserve existing SQLite behavior when available, and add regression tests that simulate missing `node:sqlite` across restore, update/upsert, and stale-task maintenance. Reuse or supersede the closed #64722 work only after reviewing its durability and multi-process behavior.

## Risks / Open Questions

- Closing this now would leave users on Node runtimes without `node:sqlite` unable to restore or persist task and task-flow registry state through the default stores.
- A fallback backend needs maintainer review for durability, locking, and overlapping gateway process semantics; the closed #64722 proposal is useful prior art but not current shipped behavior.
- The scheduled sweep rejection guard may reduce process exits while still allowing repeated persistence failures, so diagnostics and regression coverage should cover restore, upsert/update, and stale-task maintenance paths.
