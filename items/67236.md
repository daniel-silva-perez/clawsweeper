## Summary

Keep this issue open. Current main has a QQ Bot self-echo suppression fix, but I did not find a same-message inbound dedupe path for repeated QQ gateway pushes. The dispatcher preserves QQ event ids as `messageId`, yet the connection and per-peer queue still forward/enqueue every message event without checking whether that `messageId` was already processed recently.

## Best Possible Solution

Implement this in the QQ Bot plugin owner module, not core: add a per-account short-TTL inbound dedupe claim before slash-command handling or queue enqueue, keyed by `accountId`, QQ event surface/peer, and `messageId` with a safe fallback only when the platform id is missing. Cover C2C, group, guild, and DM events with regression tests that replay the same `messageId` 2-3 times and assert only one agent dispatch while distinct messages still process.
