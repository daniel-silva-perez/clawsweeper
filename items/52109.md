## Summary

Keep open. Current main does not implement PR #52109's requested behavior for successful `every` schedules: `applyJobResult` still applies the minimum refire floor only to `cron` schedules and assigns `naturalNext` directly for `every` schedules. The current `every` next-run computation can still return `endedAt` when execution time exceeds the interval, so the tight-refire scenario described by the PR remains plausible. Adjacent shipped fixes cover cron-expression spin loops (#17821) and unresolved cron next-run values (#66019), but not this `every` overrun guard, and the PR's added regression test file is not present on main.

## Best Possible Solution

Keep the PR open and have cron owners either merge a narrowed version or request changes. The best path is a small `applyJobResult` change for successful enabled `every` jobs that prevents immediate refire when execution overruns the interval, plus regression tests for the overrun case, malformed/unresolved `everyMs`, sub-2s intervals, and `preserveSchedule` exact-boundary behavior.

That leaves `every` jobs without the PR's safety floor.
