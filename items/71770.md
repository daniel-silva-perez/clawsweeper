## Summary

Keep #71770 open. Current main has ordered model fallback, auth-profile cooldown rotation, and provider usage/quota reporting, but I did not find a supported config or runtime path for proactive token/cost-threshold model failover, reverse failback, or user-visible threshold-switch notifications.

## Best Possible Solution

Keep the issue open for product/design work in the model request routing path. A strong implementation would add a documented config surface for provider/model token or cost thresholds, evaluate usage snapshots and estimated turn cost before selecting each candidate, integrate cleanly with runWithModelFallback without weakening existing cooldown/error fallback, emit user-visible lifecycle notifications when threshold switching occurs, define reverse-failover behavior when quota windows reset, and add focused docs/tests around providers with and without usage APIs.

## Risks / Open Questions

- Provider usage data is uneven across providers, so a real implementation needs explicit behavior for providers without quota APIs or only post-response token usage.
- Automatic switching can unexpectedly increase cost or change model behavior unless thresholds, fallback order, reverse-failover semantics, and notifications are explicit and auditable.
- This overlaps adjacent budget/rate-limit work, so maintainers should avoid a one-off path that conflicts with existing cooldown/error fallback semantics.
