## Summary

Current main does not implement the reported split-provider auth behavior. Runtime auth still follows auth-profile order unless the provider entry explicitly declares `auth: "api-key"`, and even then `models.providers.*.apiKey` is treated as a literal credential or SecretRef, not an auth-profile id. Session auto-selection and status labels still derive the active profile from auth order/profile order, so the reported wrong-profile symptom remains actionable.

## Best Possible Solution

Keep this issue open. Maintainers should define an explicit split-provider auth contract, such as a dedicated provider-entry auth profile field or documented profile-reference semantics, then update runtime auth resolution, session auto-selection, status labels, docs, and focused regression tests for multiple provider entries sharing one upstream auth provider.

## Risks / Open Questions
