## Summary

Keep open. Current main still resolves normal auth-profile order before provider config apiKey on the reported runtime/status paths, and there is no implementation or regression coverage for per-provider-entry auth-profile pinning for split OpenRouter-style entries.

## Best Possible Solution

Keep this issue open. The maintainer path should define an explicit split-provider auth contract, preferably a dedicated provider-entry auth profile field or clearly documented apiKey profile-reference semantics, then update runtime auth resolution, session auto-selection, status labels, docs, and focused regression tests for multiple provider entries sharing one upstream provider/auth family.

## Risks / Open Questions

- Changing this by simply overloading apiKey to mean both literal credentials and auth-profile ids could conflict with the documented credential contract and accidentally send a profile id as a bearer token.
- A runtime-only fix would leave /status and session auto-selection misleading unless resolveSessionAuthProfileOverride and resolveModelAuthLabel use the same provider-entry-aware selection rule.
