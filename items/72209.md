## Summary

Keep this PR open. Current main still does not contain the passive session recovery prompt discipline, recovery bundle/task ledger state, or checkpoint writes for update_plan, sessions_yield, and sessions_spawn. The related open draft PR #72436 is adjacent recovery logging work, not a canonical replacement for this PR's agent prompt/checkpoint behavior.

## Best Possible Solution

Keep this PR open for maintainer review rather than conservative cleanup. The best path is to rebase onto current main, address the redaction and synchronous-I/O review comments, keep recovery strictly passive and informational, prove approvals/auth/permissions are never inherited, reconcile the design with #72436 and existing restart/orphan recovery, and then land or close based on maintainer product/security judgment.

## Risks / Open Questions

- The PR persists local recovery summaries and identifiers, so sensitive-data redaction, file permissions, retention, and user-visible privacy expectations need maintainer review before merge.
- Greptile's review concerns remain material: the broad bare-keyword redaction pattern can corrupt legitimate free-text recovery context, and synchronous fs writes on plan/yield/spawn paths can add latency.
- The design should be reconciled with existing active restart/orphan recovery and the adjacent open draft PR #72436 so OpenClaw does not end up with competing recovery ledgers or incompatible semantics.
- Prompt text that injects recovered context into future sessions needs careful review to preserve the PR's stated invariant that recovered context is data, not instruction or authorization.
