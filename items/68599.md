## Summary

Keep this PR open. Current main still does not implement the proposed `[context-overflow-hint]`, and the user problem is real enough: `toolResultMaxChars` exists and is used for live tool-result truncation, but the overflow path only logs diagnostics and recovery attempts. The PR is a plausible small fix, but its latest diff still appears to key the hint off the cumulative `overflowCompactionAttempts` counter, matching the review concern that it can mislabel a later independent overflow as an immediate post-compaction retry.

## Best Possible Solution

Keep this PR open and request a focused revision: emit the hint only when the overflow is the immediate retry after an overflow compaction in the same recovery sequence, preserve the global retry-budget counter, and add a regression test proving the hint does not fire for a later independent overflow after an intervening successful/non-overflow iteration. The docs/changelog direction and lack of supply-chain-sensitive changes look fine once the runtime condition is corrected.

## Risks / Open Questions

- The PR's proposed `overflowCompactionAttempts > 0` gate can produce a misleading hint on a later independent overflow because the counter is cumulative for the run, not scoped to an immediate compact-then-retry-then-overflow sequence.
