## Summary

Keep #71066 open. Current main has Telegram polling watchdog and transport-rebuild hardening, including the #71502 wedged-runner baseline, but the issue discussion explicitly distinguishes this report from the generic watchdog bug: it is still about a region/network-specific sticky IPv4 fallback loop where the Telegram API is reachable outside the long-running gateway process. There is no source, test, release, or maintainer-comment evidence that current main has resolved that specific failure mode.

## Best Possible Solution

Keep the issue open and have the reporter retest current main after #71502 with `OPENCLAW_DIAG=telegram` around a full stall/rebuild cycle plus an in-process probe using OpenClaw's `resolveTelegramFetch`. If the probe succeeds while gateway polling still wedges, the next fix should target Telegram polling-session lifecycle/runner ownership; if the probe fails in-process, target the Telegram fetch/dispatcher fallback path and add diagnostics for dispatcher construction/reuse and the full undici error chain.
