## Summary

Keep this issue open. Current main has Telegram polling watchdog and transport-rebuild hardening, but the exact sticky IPv4 fallback path reported here still exists, maintainers already said local healthy probes do not disprove the affected India/macOS report, and there is no current-main retest from the affected host showing the failure is gone.

## Best Possible Solution

Keep this issue open and ask for a current-main retest on the affected host with Telegram diagnostics through a full stall/rebuild cycle. If an in-gateway probe using `resolveTelegramFetch` fails, focus the fix in `extensions/telegram/src/fetch.ts` around fallback diagnostics, dispatcher lifecycle, and error-chain logging. If the probe succeeds while gateway polling still wedges, continue in `extensions/telegram/src/polling-session.ts` around runner ownership and restart behavior, using #71502 as the baseline.
