## Summary

Keep #71848 open. The report is recent, concrete, and overlaps known macOS launchd gateway lifecycle gaps that are not fully resolved on current main. Current code still embeds TMPDIR into LaunchAgent environments, tests explicitly pin that behavior, and the service audit does not detect a stale or session-scoped TMPDIR. Related launchd startup/respawn issues #46153 and #53475 are still open, while prior implemented fixes mainly cover KeepAlive/restart repair paths rather than this SIGABRT/TMPDIR/TCC/memory-pressure incident.

## Best Possible Solution

Keep #71848 open and investigate it as a macOS LaunchAgent reliability bug. A good maintainer path is to add or run a macOS launchd repro that captures the generated plist, launchctl state, gateway log, health binding result, memory use, and Node version; then decide whether to stop freezing session TMPDIR into plists, add doctor/service-audit repair for stale TMPDIR, add StandardInPath=/dev/null or other launchd startup hardening, and document the exact Full Disk Access/TCC requirement for the process context that actually runs the gateway.
