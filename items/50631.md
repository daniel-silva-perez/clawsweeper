## Summary

Keep #50631 open. Current main still does not reject `gateway.tailscale.mode="serve"` with `gateway.auth.mode="none"`: runtime validation only rejects Funnel with non-password auth and non-loopback Tailscale binds, while `authorizeGatewayConnect` still accepts `auth.mode === "none"` unconditionally. Existing tests even cover a Serve+none runtime-config path that resolves successfully. The PR is not obsolete; the review discussion suggests the fix may need to cover install/configuration flow as well as startup validation.

## Best Possible Solution

Keep the PR open for maintainer/security review. The best fix is to reject or normalize `tailscale.mode="serve"` with `gateway.auth.mode="none"` at the shared gateway config validation boundary, cover startup and install/onboarding flows, and add regression tests proving Serve+none is refused while Serve+token/password remains valid.
