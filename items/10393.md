## Summary

Current `main` already provides both requested capabilities for `memory_search`: provider-less local keyword/BM25 fallback and a built-in local embedding provider that does not require an external API.

## Evidence

- **Docs describe local embeddings without API:** `memory_search` docs explicitly say to use `provider: "local"` for local embeddings with no API key.
  - file: [docs/concepts/memory-search.md:32](https://github.com/openclaw/openclaw/blob/754acc4478fe361db1e5303fd60f589d52beea7a/docs/concepts/memory-search.md#L32)

## Risks / Open Questions

- The issue body mentions ONNX/all-MiniLM-L6-v2 as one possible implementation; that exact backend is not what ships today. The close rationale is that the requested no-external-API capability is already covered by the existing local provider plus FTS fallback.

## Close Comment

Closing as implemented on current `main` after Codex review. OpenClaw already supports local, no-external-API memory search in two ways: `docs/concepts/memory-search.md:32` documents `provider: "local"`, `src/memory-host-sdk/host/embeddings.ts:23` implements the local embedding provider via `node-llama-cpp`, and `extensions/memory-core/src/memory/manager.ts:341` has an explicit FTS-only/BM25 search path when no embedding provider is available, including keyword expansion fallback. The no-provider path is also covered by `extensions/memory-core/src/memory/manager.fts-only-reindex.test.ts:10` and related assertions. If you meant a new specific backend such as ONNX/all-MiniLM-L6-v2 rather than the already-shipped local provider, please open a narrower follow-up issue for that backend and expected configuration UX.
