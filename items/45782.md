## Summary

Keep open. Current main does not already contain this PR, and the submitted patch is not merge-ready: the new logging calls proposed in ff921b0 would route `{ error }` into `logWarn(message, runtime)`, but the logger expects a full `RuntimeEnv` and dereferences `runtime.log`, which would throw in the very cleanup/fallback paths this PR is trying to harden. The TLS part of the patch also leaves the chmod failures at `info`, matching the unresolved review concern.

## Evidence

- **Logger API requires a real runtime, not `{ error }`:** `logWarn` takes `runtime: RuntimeEnv = defaultRuntime`, and `logWithSubsystem` calls `params.runtime[params.runtimeMethod](...)`. `RuntimeEnv` requires `log`, `error`, and `exit`, so passing `{ error }` would fail when `logWarn` tries to call `runtime.log(...)`. This matches the review comments on PR commit `ff921b0ad71cd52ee6a033c9284891b32ce16c54`.
  - file: [src/logger.ts:21](https://github.com/openclaw/openclaw/blob/ff921b0ad71cd52ee6a033c9284891b32ce16c54/src/logger.ts#L21)
  - command: `nl -ba src/logger.ts | sed -n '16,60p'; nl -ba src/runtime.ts | sed -n '1,20p'`
  - sha: [ff921b0ad71c](https://github.com/openclaw/openclaw/commit/ff921b0ad71cd52ee6a033c9284891b32ce16c54)
- **Malformed allow-from JSON must remain non-throwing:** Current main's sync allow-from reader returns `{ entries: [], exists: true }` on JSON parse failure, preserving the existing fallback behavior. The PR adds logging in this path, but with the invalid logger runtime noted above that would make malformed JSON fatal instead of non-throwing.
  - file: [src/pairing/allow-from-store-file.ts:263](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/src/pairing/allow-from-store-file.ts#L263)
