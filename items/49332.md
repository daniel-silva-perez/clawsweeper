## Summary

Current main does not implement the PR's runId idempotency guard or an equivalent fix. `handleChatEvent` still appends a terminal `final` payload and clears `chatRunId`, so a retransmitted final with the same `runId` can pass once `chatRunId` is null. The PR should stay open for maintainer handling, but it should not be merged as-is because the provided diff also bundles unrelated Discord, Vertex, sandbox, config, and AGENTS changes and the review discussion flags correctness gaps in the proposed dedupe approach.

## Best Possible Solution

Keep the item open, but ask for or prepare a focused UI-only replacement: add explicit terminal runId idempotency state (or another durable idempotency key) to the chat state contract, apply it before appending final/aborted assistant messages in both own-run and different-run paths, clear it only on real session/agent switches, and add regression tests that retransmit the same final event after `chatRunId` has been cleared. The unrelated PR changes should be split out or dropped.
