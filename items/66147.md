## Summary

Keep this PR open. Current main still lacks `model.tool`, split model routing, and an equivalent plugin hook for per-tool-continuation model swaps. The PR is a plausible implementation candidate, but the review discussion and current runner code show unresolved correctness/security-sensitive routing issues around primary-model-bound tool enablement, transport, auth, transcript policy, and provider wrappers.

## Best Possible Solution

Keep this PR open for maintainer review or author revision. The best path is to decide whether split model routing is a core config contract or a new plugin/API seam, then rework the implementation so the effective per-call model controls tool availability, provider transport, auth profile resolution, transcript policy, provider-specific stream wrappers, context budgeting, fallback behavior, diagnostics, config docs, generated baselines, and tests. If maintainers prefer the plugin/API route, use #13225 as the design discussion instead of closing this PR as already handled.

## Risks / Open Questions

- Merging the PR as written may not satisfy the advertised local-primary plus Anthropic-tool workflow because the first tool-emitting turn still depends on primary-model tool support.
- Cross-provider routing could reuse primary-provider transport, auth profile, transcript policy, and stream wrappers for tool-model turns, which is both a correctness issue and an auth/credential boundary risk.
- The config schema change would need generated config docs/baselines and public docs updates before landing.
- The PR test plan still lacks manual integration proof and full repo validation in the provided context.
