## Summary

Keep open. Current main still rejects hosted LID targets in the shared WhatsApp target normalizer, while the runtime JID conversion path already recognizes `@hosted.lid`; PR #70803 remains a focused, plausible fix with matching regression tests in the supplied PR patch.

## Best Possible Solution

Keep PR #70803 open for maintainer review. If the intended product behavior is that hosted LID explicit targets should normalize like existing `@lid` targets, merge or reimplement the PR's narrow regex change and add the lower/uppercase hosted-LID regression assertions in `extensions/whatsapp/src/resolve-target.test.ts`.

## Evidence

- **Current normalizer still excludes hosted LID:** On current main, `WHATSAPP_LID_RE` only matches `^(\d+)@lid$`, and both `isWhatsAppUserTarget` and `extractUserJidPhone` depend on that regex. `123456789@hosted.lid` therefore falls through as a candidate containing `@` and normalizes to `null`.
