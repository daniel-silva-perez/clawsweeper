## Summary

Keep this issue open. Current main still accepts Telegram `direct.<peer>.tools` in schema/types and uses Telegram direct config for DM prompt/access behavior, but the actual tool-policy enforcement path remains group/channel-oriented and resolves `channels.telegram.groups`, not `channels.telegram.direct`. The related #53530 is useful shared direct-session work, but it remains an open PR and the provided discussion explicitly says it does not by itself adapt Telegram's legacy `direct.*` surface.

## Best Possible Solution

Keep this issue open and implement the Telegram adapter on top of the shared direct-session policy work: parse trusted Telegram direct-session/account context, resolve `channels.telegram.accounts.<account>.direct[peer] ?? direct["*"]`, apply `tools` and `toolsBySender` with explicit-over-wildcard precedence, and add regression coverage through the mounted agent tool-policy path for Telegram DM and native-command runs. #53530 can remain related shared resolver work, but it is not canonical closure until Telegram `direct.*` enforcement is covered on main.
