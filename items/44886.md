## Summary

Keep this issue open. Current main still accepts Telegram `direct.<peer>.tools` in schema/types and resolves Telegram `direct` config for DM runtime context, but the tool-policy enforcement path still resolves group/channel policy only. The related #53530 is useful shared direct-session work, but it remains open and the issue discussion explicitly says it does not by itself adapt Telegram's legacy `direct.*` surface.

## Best Possible Solution

Keep this issue open and implement the Telegram adaptation on top of the shared direct-session policy work: parse trusted Telegram direct-session/account context, resolve `channels.telegram.accounts.<account>.direct[peer] ?? direct["*"]`, apply `tools` and `toolsBySender` with explicit-over-wildcard precedence, and add regression coverage through the agent tool-policy path for Telegram DM runs. #53530 can remain related shared resolver work, but it should not be treated as canonical closure until Telegram `direct.*` enforcement is covered.
