## Summary

Keep open. Current `main` parses `envelope.editMessage.dataMessage`, but it still treats the edit as a fresh inbound message and does not carry the original `targetSentTimestamp`, mark the transcript as edited, or suppress a new response trigger.

## Evidence

- **Edit envelope type drops correlation metadata:** `SignalEnvelope.editMessage` only exposes `dataMessage`; it does not include `targetSentTimestamp`, so the handler has no typed way to match an edit back to the original message.
  - file: [extensions/signal/src/monitor/event-handler.types.ts:18](https://github.com/openclaw/openclaw/blob/754acc4478fe361db1e5303fd60f589d52beea7a/extensions/signal/src/monitor/event-handler.types.ts#L18)
  - command: `nl -ba extensions/signal/src/monitor/event-handler.types.ts | sed -n '12,24p'`
  - sha: [754acc4478fe](https://github.com/openclaw/openclaw/commit/754acc4478fe361db1e5303fd60f589d52beea7a)
- **Edit payload is consumed as ordinary message content:** The inbound handler sets `dataMessage = envelope.dataMessage ?? envelope.editMessage?.dataMessage` and then proceeds through the normal text path. There is no edit-specific branch, marker, or skip logic here.
  - file: [extensions/signal/src/monitor/event-handler.ts:526](https://github.com/openclaw/openclaw/blob/754acc4478fe361db1e5303fd60f589d52beea7a/extensions/signal/src/monitor/event-handler.ts#L526)
  - command: `nl -ba extensions/signal/src/monitor/event-handler.ts | sed -n '522,538p'`
  - sha: [754acc4478fe](https://github.com/openclaw/openclaw/commit/754acc4478fe361db1e5303fd60f589d52beea7a)
- **Edited messages still enqueue as new inbound events:** The handler derives `messageId` from the new envelope timestamp and enqueues the message normally. That preserves the edit as a separate inbound event instead of replacing or linking to the original message.
  - file: [extensions/signal/src/monitor/event-handler.ts:855](https://github.com/openclaw/openclaw/blob/754acc4478fe361db1e5303fd60f589d52beea7a/extensions/signal/src/monitor/event-handler.ts#L855)
  - command: `nl -ba extensions/signal/src/monitor/event-handler.ts | sed -n '823,878p'`
  - sha: [754acc4478fe](https://github.com/openclaw/openclaw/commit/754acc4478fe361db1e5303fd60f589d52beea7a)
- **Current code only uses `targetSentTimestamp` for reactions:** Reaction handling explicitly uses `reaction.targetSentTimestamp` as the referenced message id, which highlights that equivalent edit correlation logic is still missing for `editMessage` envelopes.
  - file: [extensions/signal/src/monitor/event-handler.ts:455](https://github.com/openclaw/openclaw/blob/754acc4478fe361db1e5303fd60f589d52beea7a/extensions/signal/src/monitor/event-handler.ts#L455)

  - sha: [754acc4478fe](https://github.com/openclaw/openclaw/commit/754acc4478fe361db1e5303fd60f589d52beea7a)
- **No edit-message regression tests found:** A search for `editMessage` in Signal test files returned no matches, so there is no current test coverage proving transcript replacement, `[edited]` marking, or retrigger suppression.
