## Summary

Current `main` does not reproduce the claimed CWE-78 shell-injection boundary. The parsed argv from systemd/schtasks/process inspection is used for metadata, port extraction, gateway-process matching, or direct `spawn(executable, args)` execution without a shell. The proposed `isSafeExecutableValue` gate would also reject legitimate Windows executable paths containing characters like `&` in directory names.

## Evidence

- **Systemd parsing is read-back metadata, not shell execution:** `readSystemdServiceExecStart` reads `ExecStart`, parses it with `parseSystemdExecStart`, and returns `programArguments` inside a `GatewayServiceCommandConfig`; there is no shell execution on this path.
  - file: [src/daemon/systemd.ts:73](https://github.com/openclaw/openclaw/blob/40be5ad58187fd20b8defe5fc30e649190c131a8/src/daemon/systemd.ts#L73)
  - command: `nl -ba src/daemon/systemd.ts | sed -n '70,130p'`
  - sha: [40be5ad58187](https://github.com/openclaw/openclaw/commit/40be5ad58187fd20b8defe5fc30e649190c131a8)
- **Daemon lifecycle only uses parsed service args to derive the port:** `resolveGatewayLifecyclePort` reads the service command and passes `command?.programArguments` into `parsePortFromArgs`; this is configuration/status logic, not command execution.
  - file: [src/cli/daemon-cli/lifecycle.ts:63](https://github.com/openclaw/openclaw/blob/40be5ad58187fd20b8defe5fc30e649190c131a8/src/cli/daemon-cli/lifecycle.ts#L63)
  - command: `nl -ba src/cli/daemon-cli/lifecycle.ts | sed -n '63,79p'`
  - sha: [40be5ad58187](https://github.com/openclaw/openclaw/commit/40be5ad58187fd20b8defe5fc30e649190c131a8)
- **Scheduled-task fallback executes argv directly, not through a shell:** When a parsed scheduled-task command is reused, the code does `spawn(executable, args, ...)`. That does not treat `;`, `&`, `|`, `$`, or backticks as shell metacharacters. The only shell-based fallback is `cmd.exe /d /c <scriptPath>` when no parsed argv is available.
  - file: [src/daemon/schtasks.ts:316](https://github.com/openclaw/openclaw/blob/40be5ad58187fd20b8defe5fc30e649190c131a8/src/daemon/schtasks.ts#L316)

- **Windows PID inspection only parses argv for gateway matching:** Windows process args are read with PowerShell/WMIC, parsed into argv, then consumed by `findVerifiedGatewayListenerPidsOnPortSync` and `signalVerifiedGatewayPidSync` for `isGatewayArgv(...)` checks before signaling. This path filters processes; it does not execute parsed strings.
  - file: [src/infra/windows-port-pids.ts:108](https://github.com/openclaw/openclaw/blob/40be5ad58187fd20b8defe5fc30e649190c131a8/src/infra/windows-port-pids.ts#L108)

## Risks / Open Questions

- If maintainers still want extra defense-in-depth for tampered service files, that should be framed as service-config integrity hardening rather than CWE-78 shell injection.
- Any future hardening on parsed executable paths needs a policy that preserves valid Windows filesystem names instead of reusing `isSafeExecutableValue` unchanged.

## Close Comment

Thanks for the PR. I reviewed this with Codex against current `main` (`40be5ad58187fd20b8defe5fc30e649190c131a8`) and I could not reproduce the reported CWE-78 boundary. On current `main`, the parsed argv from service files/process inspection is used for metadata, port derivation, or gateway-process matching (`src/daemon/systemd.ts:73`, `src/cli/daemon-cli/lifecycle.ts:63`, `src/infra/gateway-processes.ts:35`), and the one place that reuses parsed scheduled-task argv executes it with `spawn(executable, args)` rather than a shell (`src/daemon/schtasks.ts:316`). That means payloads like `; rm -rf /` are parsed as argv text, not executed as shell syntax. I’m also not comfortable landing the proposed `isSafeExecutableValue` check here because it would reject legitimate Windows executable paths containing characters like `&` in directory names (`src/infra/exec-safety.ts:1`). If you have a concrete current-main reproduction that reaches a real shell execution boundary, please open a fresh issue/PR with that exact path.
