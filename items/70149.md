## Summary

Keep open. Current main still routes WhatsApp login through the ambient `proxy-agent` path that #70149 reports as failing, while the issue specifically says an explicit `HttpsProxyAgent` works for the same WSS endpoint. The earlier #61825 close covered adding ambient proxy support, but this newer report is about that ambient implementation still timing out.

## Best Possible Solution

Keep #70149 open. Reproduce the reported WSS login path against current main, then add a focused WhatsApp WebSocket proxy resolver if it holds: resolve the concrete env proxy URL with `NO_PROXY` semantics, use explicit `HttpsProxyAgent` for HTTP(S) proxies on the Baileys `agent` path, preserve fallback for unsupported schemes, and add a regression test that distinguishes `ProxyAgent` from `HttpsProxyAgent` or exercises a mocked CONNECT/ws handshake.
