## Summary

Keep PR #71820 open. Current main still only hydrates inbound BlueBubbles reply context from webhook fields and the process-local reply cache; it does not have the proposed opt-in API fallback for cache misses. The related #43764/#65447 fix is shipped, but it covers outbound Private API status refresh for reply threading/effects, not inbound recovery of missing reply body/sender context.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is to revise the fallback so cache misses use the existing guarded BlueBubbles client/request path for `GET /api/v1/message/{guid}`, add the config surface without breaking channel-level defaults for multi-account configs, keep the behavior opt-in unless maintainers choose otherwise, and retain focused tests for cache misses, failures, in-flight dedupe, SSRF policy, and config inheritance.
