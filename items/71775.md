## Summary

Keep open. The issue is fresh, non-maintainer-authored, and has no protected labels, but it maps to real core-owned session lifecycle behavior: current main defaults to a daily reset at 4:00 AM local gateway time and will create a new session on the next user message after that boundary. That matches the reported 11 PM to 5 AM WeChat gap, but without config, session-store timestamps, logs, or WeChat plugin/session-key details, there is not enough evidence to close it as fixed, unreproducible, external, duplicate, or stale.

## Best Possible Solution

Keep the issue open for maintainer triage. Ask the reporter for session.reset, session.resetByChannel, session.dmScope, WeChat plugin version, /status before and after the overnight gap, the affected session key, redacted sessions.json timestamps (sessionStartedAt, lastInteractionAt, updatedAt), and gateway logs around 4:00-5:00 AM local gateway time. If the default daily reset explains it, the likely product fix is clearer reset notice/docs/config guidance for channel users; if not, add a focused regression test around WeChat/openclaw-weixin session-key stability or core direct-session reuse.
