## Summary

Keep #66712 open. Current main still only deduplicates cron direct delivery at the execution level using `createCronExecutionId(jobId, runStartedAt)`, so a restart catch-up or timer refire for the same scheduled slot with a different `runStartedAt` can bypass the existing cache. The PR is recent, focused, mergeable per provided context, and it addresses a concrete remaining gap plus a small startup catch-up `deliveryAttempted` propagation issue.

## Best Possible Solution

Keep the PR open and continue normal maintainer review. The best product/code path is to add a process-local scheduled-slot dedup guard alongside the existing execution-level idempotency cache, keyed by planned scheduled time plus delivery target, and to keep the PR's regression test for same-slot replay with different `runStartedAt`. Also include the `deliveryAttempted` propagation fix in startup catch-up so observers see the same outcome shape as the live path.
