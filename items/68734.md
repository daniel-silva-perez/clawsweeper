## Summary

Keep open. Current main still does not expose `availableTools` or `availableSkills` to `before_prompt_build`, does not accept `toolsAllow` or `skillsAllow` from prompt hooks, and only applies static/runtime `params.toolsAllow`. The PR targets a real core plugin API gap, supported by the PR body and svankirk's MCP-heavy deployment comment, but it is not ready to close or merge as-is because review comments identify remaining functional regressions in the proposed diff.

## Best Possible Solution

Keep the PR open as a core plugin API candidate. The best path is to rebase it onto current main, preserve all existing hook prompt mutations and `transformProviderSystemPrompt` behavior when rebuilding a narrowed skills prompt, include client-hosted tools in the hook-visible/narrowed tool surface, document the new hook fields, and add focused tests for hook-derived `toolsAllow`, client-tool preservation, `skillsAllow`, and prompt-transform preservation.

## Risks / Open Questions

- Merging the PR as-is could drop `systemPrompt`, `prependSystemContext`, `appendSystemContext`, or provider-specific system prompt transforms when `skillsAllow` is returned.
- The proposed `availableTools` list still omits client-hosted tool definitions, so a hook-derived `toolsAllow` could unintentionally remove client tools from the active session.
- The PR body says skill narrowing was live-verified, but the submitted tests cover the tools path and helper merge behavior more than the full skill prompt rebuild path.
- The branch was based before current main and should be rebased so the attempt/session prompt code is reviewed against the current structure.
