## Summary

Keep open. Current main still lacks cursor-based `chat.history` pagination and web UI load-older behavior, so the PR is not obsolete. The latest PR head is also not merge-ready: it parses `before` and computes `start`/`end`, but still calls `projectRecentChatDisplayMessages(rawMessages, ...)` on the full raw history, so older-page requests can return the latest page again. Related PR #56383 is open and unmerged, not a shipped or clearly canonical replacement.

## Best Possible Solution

Keep the PR open for maintainer review. Fix the gateway handler so `before` selects the raw `[start, end)` page before projection, or track raw indices through projection/sanitization explicitly; add integration tests for `chat.history` pagination with normal messages, sanitized `NO_REPLY`/commentary drops, and byte-capped leading trims; coordinate with open PR #56383 before choosing which implementation to land.

## Risks / Open Questions

- Functional correctness: latest PR head appears to make `before` affect cursor math but not the raw message selection, so `loadOlderChatHistory` can prepend duplicate latest messages instead of older history.
- Coverage gap: the PR has mocked UI tests and pure helper tests, but no gateway integration test proving `chat.history` returns the page before a cursor, especially with sanitized `NO_REPLY` or commentary messages and byte-budget trimming.
- Scope risk: the PR combines pagination with partial `NO_REPLY` UI handling; current main already has several gateway/live-chat defenses for silent replies, so maintainers may want that portion split or narrowed.
