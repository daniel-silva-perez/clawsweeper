## Summary

Current main already implements the requested behavior: the `openai-completions` transport sends chat-style `messages` payloads to `/chat/completions`, and it includes compatibility handling for stricter OpenAI-compatible backends that require string `messages[].content` values. The issue's premise that OpenClaw still sends legacy `prompt` payloads does not match current main.

## Close Comment

Closing from a Codex review because current `main` already implements the requested behavior. The `openai-completions` transport builds chat-style `{ messages: [...] }` payloads in `src/agents/openai-transport-stream.ts:1672`, and the custom OpenAI-compatible verification path also targets `/chat/completions` with `messages` in `src/commands/onboard-custom-config.ts:318`. OpenClaw also already has compatibility handling for stricter backends that require string `messages[].content` values, covered in `src/agents/openai-transport-stream.test.ts:1502` and documented in `docs/providers/inferrs.md:87`. There is not a separate `prompt` vs `messages` toggle here because the current transport is already on the `messages` shape. If this still reproduces on a current build with MindIE, please open a new bug with the exact provider config, endpoint, and the 422 response body so maintainers can check whether it is a MindIE-specific compatibility gap.
