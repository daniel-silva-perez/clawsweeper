## Summary

Current `main` does not implement a durable gateway-level audit log for all tool calls. What exists today is transient live tool-event broadcasting plus plugin hooks around tool execution; there is no built-in core audit file/config surface matching the request.

## Evidence

- **Live tool events are broadcast-only:** Gateway mirrors tool lifecycle to `session.tool` for operator UIs, but sends it as a best-effort websocket event with `dropIfSlow: true`, which is not a durable audit trail.
  - file: [src/gateway/server-chat.ts:949](https://github.com/openclaw/openclaw/blob/3361593442e295e9bb1f55cbfb38861ea9413e45/src/gateway/server-chat.ts#L949)
  - command: `nl -ba src/gateway/server-chat.ts | sed -n '900,1010p'`
  - sha: [3361593442e2](https://github.com/openclaw/openclaw/commit/3361593442e295e9bb1f55cbfb38861ea9413e45)
- **Only lifecycle state is persisted:** The persistence helper updates stored session lifecycle fields only after resolving a lifecycle phase; it does not persist tool-call events or a tool audit ledger.
  - file: [src/gateway/session-lifecycle-state.ts:146](https://github.com/openclaw/openclaw/blob/3361593442e295e9bb1f55cbfb38861ea9413e45/src/gateway/session-lifecycle-state.ts#L146)

- **Core logging docs describe generic gateway logs, not tool-call audit logs:** Gateway logging is documented as a rolling JSONL file controlled by `logging.file` and `logging.level`; the only tool-specific setting is console summary redaction, explicitly not a file-log audit feature.
  - file: [docs/gateway/logging.md:18](https://github.com/openclaw/openclaw/blob/3361593442e295e9bb1f55cbfb38861ea9413e45/docs/gateway/logging.md#L18)

- **Built-in exports intentionally omit/redact tool payloads:** Gateway diagnostics export keeps sanitized operational logs and explicitly omits or redacts chat text, webhook bodies, and tool outputs, so it is not the requested full audit record.
  - file: [docs/cli/gateway.md:163](https://github.com/openclaw/openclaw/blob/3361593442e295e9bb1f55cbfb38861ea9413e45/docs/cli/gateway.md#L163)

That supports plugin solutions, but it is not the built-in gateway audit logging requested in this issue.

## Risks / Open Questions

- A hook/plugin can likely implement an audit log today, but that is not equivalent to a built-in gateway-level audit facility with first-class config, rotation, and redaction policy.
- The GitHub context shows an open cross-referenced PR for this feature, so there is still active work/value here even though current `main` has not landed it.
