## Summary

Keep this backup-design issue open. Current main already has useful backup primitives: `openclaw backup create`, `.tar.gz` archives, manifest verification, `--only-config`, `--no-include-workspace`, and top-level canonical path dedupe. The central requested work is still not implemented: configurable exclusion rules, fine-grained archive filtering such as plugin dependency pruning, service-aware or quiesced backup behavior for live state, and broader restore/full-mode semantics. Related open items and PRs cover pieces, but no single canonical item fully supersedes this proposal.

## Best Possible Solution

Keep this issue open, but narrow it into a maintainer-readable backup design. The likely path is to keep the shipped create/verify/config-only behavior, add a core exclusion policy surface or CLI/config option, land or supersede the live-file race fixes tracked by #67417 and #72251, decide whether service control should be an explicit quiesce mode, and leave broader restore/import disaster-recovery semantics coordinated with #13616.

## Risks / Open Questions

- The issue is broad; maintainers may want to split it into narrower implementation tasks rather than keep one umbrella open forever.
- Open PR #72251 may resolve one live-backup failure mode after merge, but it would not by itself implement configurable exclusions or service-aware backup behavior.
- Gateway stop/start or quiesce behavior is product-sensitive and should be designed explicitly, likely as an opt-in policy or paired with resilient per-file archiving.
