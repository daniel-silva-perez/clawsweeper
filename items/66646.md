## Summary

Keep open. Current main fixed the broad timeout misclassification for session write-lock errors, but it still does not implement the central requested fail-fast/session-busy behavior: the session write lock is acquired inside the `runWithModelFallback()` attempt path, and unrecognized non-abort errors still continue to remaining fallback candidates until the last candidate.

## Best Possible Solution

Keep this issue open and implement an explicit non-model-failover path for `SessionWriteLockTimeoutError` or nested lock-timeout errors at the fallback wrapper or agent attempt boundary. The likely narrow fix belongs in `src/agents/model-fallback.ts` or the `agentCommand()` attempt wrapper: detect session write-lock timeout before recording a failed model candidate, surface it as a session-busy infrastructure error, and add a regression test where a configured fallback chain receives `SessionWriteLockTimeoutError` and the run callback is called exactly once. Queue/retry/backoff can then be handled by the session lane or caller boundary; #69030 remains a narrower CLI trigger follow-up, not a replacement for this fix.
