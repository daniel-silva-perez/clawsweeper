## Summary

Keep open. Current main partially fixed the misclassification: session write-lock timeouts are typed and no longer resolve to failover reason `timeout`. But the central cascade behavior is still present because `runWithModelFallback()` deliberately continues to remaining candidates for unrecognized non-abort errors, and session-lock acquisition failures from the agent attempt path propagate into that fallback loop. The issue's requested fail-fast/session-busy behavior is therefore not fully implemented on main.

## Best Possible Solution

Keep this issue open and implement an explicit non-model-failover path for `SessionWriteLockTimeoutError` or nested lock-timeout errors in the fallback wrapper. The likely fix belongs in `src/agents/model-fallback.ts` or the agent attempt boundary: detect the session-lock timeout before recording a failed candidate, rethrow/surface it as a session-busy infrastructure error, and add a regression test where a configured fallback chain receives a `SessionWriteLockTimeoutError` and the run function is called exactly once. Queue/retry/backoff semantics can then be added at the session lane or caller boundary; the separate `infer model run --raw` request remains a useful narrower follow-up for that CLI trigger.
