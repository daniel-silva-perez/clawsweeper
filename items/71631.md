## Summary

Keep #71631 open. Current main already improves part of the Web UI chat responsiveness path with optimistic user-message rendering, a started ack from `chat.send`, live stream/reading UI, and history-reload race guards. However the issue is broader than those shipped fixes: full UI-to-gateway-to-provider latency instrumentation is not present, gateway/config screens still use coarse page/form loading and saving states, and directly related narrower items such as #64004 and #71630 remain open.

## Best Possible Solution

Keep the issue open and narrow it into actionable maintainer work: add stage timing from UI click/send through WebSocket request, gateway method, provider/model start, first visible update, and final completion; improve config pages with stale-response guards/cancellation, debounced validation, per-control pending states, and nonblocking secondary data; coordinate the composer status chip with #71630 and the Control UI slowness investigation in #64004.
