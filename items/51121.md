## Summary

Keep PR #51121 open. Current main still has the uncaptured 500ms fallback timer in `IMessageRpcClient.stop()`, so the reported leak is not implemented or obsolete. The PR is focused and relevant, but the existing review comment raises a real lifecycle concern: `this.closed` is created once per client and is not reset on a later `start()`, so the exact patch should be adjusted rather than auto-closed.

## Best Possible Solution

Keep the PR open and ask for an update that fixes the stop-timeout leak while preserving the fallback kill across repeated `start()`/`stop()` cycles. The likely maintainer-quality fix is to make the close promise per child/start, or otherwise race against a non-stale close signal, then clear the stop fallback timer in `finally`. Add focused unit coverage for fast close cleanup and repeated start/stop with a hung child.
