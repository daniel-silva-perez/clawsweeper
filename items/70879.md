## Summary

Current main does not implement Feishu send retry/backoff for rate-limit API errors. Text/card/media sends issue a single SDK request and turn non-zero Feishu API responses into errors; the only nearby Feishu rate-limit handling is for typing indicators, not outbound message delivery.

## Best Possible Solution

Keep #70879 open. The best path is a Feishu-owned bounded retry wrapper around outbound message create/reply and relevant media/card send calls, using the existing retry-runtime helpers or equivalent local policy, with tests for thrown SDK/Axios 429 shapes and non-throwing Feishu response codes such as 11232/2200. The implementation should document which codes are safe to retry and keep duplicate-delivery risk explicit.

## Risks / Open Questions

- A fix must avoid duplicating non-idempotent Feishu messages when the API accepted a send but returned an ambiguous transport/server error; retry predicates should be conservative and preferably honor any Feishu idempotency or retry-after signal available.
