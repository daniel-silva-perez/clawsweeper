## Summary

Current main has not implemented the PR’s central behavior. The TUI still generates a fresh gateway client instance ID per process, default/main TUI session resolution still maps to the ordinary agent main session key, gateway chat routing does not synthesize a TUI affinity OriginatingTo from client.instanceId, and there is no current client-affinity or tui-client-instance module. The PR remains useful, but it needs revision because the existing review context includes a P1 concern about orphan short-reply reattachment overriding explicit missing session keys.

## Best Possible Solution

Keep the PR open or replace it with a revised implementation that preserves TUI main-session continuity through a stable per-terminal TUI instance ID, a documented affinity session-key/origin contract, and regression tests. The revision should address the P1 review by gating orphan short-reply reattachment to default TUI main-session aliases and should align current TUI client naming/mode behavior with gateway routing tests.
