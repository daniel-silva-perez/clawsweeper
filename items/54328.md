## Summary

Current `main` already makes `openclaw cron run` return once the run is enqueued, so this PR's shared timeout increase is obsolete.

## Close Comment

Closing as implemented on `main` after Codex review. Current `main` no longer relies on a long blocking RPC for `openclaw cron run`: the CLI applies a 10-minute default only for that command in `src/cli/cron-cli/register.cron-simple.ts:165`, while the gateway handler now returns immediately from `context.cron.enqueueRun(...)` in `src/gateway/server-methods/cron.ts:357`. The behavior is covered by `src/gateway/server.cron.test.ts:949`, which expects an immediate `{ ok: true, enqueued: true, runId }` response while the isolated work continues in the background, and the docs were updated in `docs/cli/cron.md:41` to match. That makes this shared timeout-bump PR obsolete on current `main`.
