## Summary

Keep open. Current `main` already publishes official multi-arch container manifests, but it does not include the proposed local `scripts/build-multiarch.sh` helper, so this PR still represents distinct, plausible functionality rather than something already implemented on `main`.

## Evidence

- **Official multi-arch publishing already exists on main:** `docker-release` builds separate `linux/amd64` and `linux/arm64` images, then creates combined manifest tags including `main`, version tags, and `latest`. That covers OpenClaw-maintained release images, not a user-run local helper script.
  - file: [.github/workflows/docker-release.yml:323](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/.github/workflows/docker-release.yml#L323)
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Current docs point users to published registry images:** Docker install docs tell users to use `ghcr.io/openclaw/openclaw:latest` and say pre-built images are published with `main`, `latest`, and version tags. The documented path is consuming published images, not building and pushing a custom multi-arch image.
  - file: [docs/install/docker.md:36](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/docs/install/docker.md#L36)
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Proposed helper script is absent from main:** Listing tracked files under `scripts/` on current `HEAD` shows `scripts/docker/setup.sh` but no `scripts/build-multiarch.sh`. The PR therefore is not already implemented on `main` in the form it proposes.
  - command: `git ls-tree -r --name-only HEAD -- scripts | rg 'build-multiarch\\.sh|docker/setup\\.sh'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)

## Risks / Open Questions

- Closing as `implemented_on_main` would be too aggressive because the user-facing convenience this PR adds, a local script for building and pushing a custom multi-arch image, is still missing from current `main`.
- The existing release workflow may reduce urgency, but it is not the same capability as the submitted script, so the PR could still be useful after maintainer review.
