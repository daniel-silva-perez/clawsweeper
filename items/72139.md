## Summary

Keep #72139 open. Current main has partial Control UI auth handling, but the reported stale-token loop is still plausible: `AUTH_TOKEN_MISMATCH` is intentionally excluded from the non-recoverable auth set, reconnects are scheduled for auth errors outside the narrow device-token retry exhaustion case, and a test explicitly expects reconnect after a first token mismatch when no device-token retry occurred. The `chat.history` path also still does synchronous config/store/transcript loading and repeated JSON-byte passes before responding, so the reported event-loop starvation has a concrete code path to investigate rather than being obsolete cleanup.

## Best Possible Solution

Keep the issue open and fix it as a real cross-surface bug. The likely maintainer path is to make stale explicit Control UI auth terminal after a mismatch or failed bounded retry, surface a clear token-expired/re-enter-token state, and prevent scheduled reconnect churn. Separately, move `chat.history` off the event-loop hot path by bounding work before full transcript parsing, avoiding full-file/full-array synchronous JSON byte passes in one tick, and adding a regression check that a large history request does not starve another websocket heartbeat such as Slack Socket Mode.
