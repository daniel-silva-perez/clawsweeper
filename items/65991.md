## Summary

Keep open. Current main still does not accept, persist, document, or propagate a `silent` option on cron `failureAlert`, even though the generic outbound/message path and Telegram sender already support silent delivery. This remains a concrete cron failure-alert integration gap rather than a duplicate, stale report, or ClawHub-only idea.

## Best Possible Solution

Implement a narrow additive cron failure-alert contract change on main: add `silent?: boolean` to per-job `CronFailureAlert` and the intended global `cron.failureAlert` config shape, update TypeBox/Zod schemas, docs, CLI/UI forms, and protocol types, then thread it through `resolveFailureAlert`, `emitFailureAlert`, `sendCronFailureAlert`, `sendGatewayCronFailureAlert`, and `sendFailureNotificationAnnounce` into `deliverOutboundPayloads({ silent })`. Add focused regression coverage proving Telegram cron failure alerts set `disable_notification` when `silent: true` while preserving the default behavior when omitted.
