## Summary

Keep open. Current main has broad proxy work, including gateway/global EnvHttpProxyAgent bootstrap and provider HTTP-helper auto-upgrade, but the reported native Gemini LLM streaming path still uses `buildGuardedModelFetch()` with the guarded fetch default of strict DNS pinning. That means ambient `http_proxy`/`https_proxy` is not automatically used for the Google `google-generative-ai` stream path, and fake-IP DNS can still be blocked before the proxy handles resolution. The issue is recent, non-maintainer-authored, unprotected, and no canonical duplicate was available in the provided context.

## Best Possible Solution

Keep the issue open. The likely fix belongs in the model-provider transport path: add a carefully gated trusted-env-proxy mode for provider LLM streams, analogous to the provider HTTP helper auto-upgrade, while preserving SSRF safeguards for NO_PROXY, ALL_PROXY-only, explicit dispatcher policies, and untrusted targets. Add focused regression coverage for Google `google-generative-ai` with `HTTP_PROXY`/`HTTPS_PROXY` and fake-IP/private DNS behavior, plus a CLI/WebUI-visible error assertion for failed proxy connections.
