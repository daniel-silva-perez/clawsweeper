## Summary

Keep open. Current `main` still intentionally blocks ACP spawns from sandboxed sessions, documents that limitation, and has regression tests for it. I found no shipped bridge/proxy/allowlist mode or sandbox-compatible ACP path, so the feature request is still unmet.

## Evidence

- **Docs still describe the limitation as current behavior:** `docs/tools/acp-agents.md` says ACP sessions run on the host, sandboxed requester sessions are blocked for both `sessions_spawn({ runtime: "acp" })` and `/acp spawn`, and `sandbox: "require"` is unsupported for ACP.
  - file: [docs/tools/acp-agents.md:402](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/docs/tools/acp-agents.md#L402)
  - command: `nl -ba docs/tools/acp-agents.md | sed -n '398,418p'`
  - sha: [f03252aaf9cc](https://github.com/openclaw/openclaw/commit/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022)
- **Runtime code hard-blocks sandboxed ACP spawn:** `resolveAcpSpawnRuntimePolicyError(...)` returns a hard error when the requester is sandboxed, explicitly because `runtime="acp"` runs on the host. It also rejects `sandbox="require"` for ACP.
  - file: [src/agents/acp-spawn.ts:170](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/src/agents/acp-spawn.ts#L170)

  - file: [src/agents/system-prompt.ts:795](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/src/agents/system-prompt.ts#L795)
