## Summary

Keep open. Current main still implements only exact SecretRef dedupe plus per-source/provider exec batching; it does not provide the requested external-vault backend identity model or pass-scoped backend/entity memoization. The docs still include 1Password and Vault examples in the one-exec-provider-per-secret shape this issue is trying to replace or steer away from, and related #17311 remains an open broader design track rather than shipped behavior.

## Best Possible Solution

Keep this issue open as a core SecretRef product/API design item. The best path is to align it with #17311 or make #17311 the explicit canonical tracker, then define a supported shared external-backend provider pattern where one provider represents a backend/account/context, SecretRef ids carry backend paths, one activation pass batches per backend, backend/container/item/payload lookups are memoized in memory for that pass, docs and configure/apply flows guide users to that shape, and regression tests prove repeated refs/entities do not duplicate backend calls.

## Risks / Open Questions

- Closing as implemented would leave only low-level exact-ref/per-provider batching, while the documented 1Password and Vault patterns can still multiply backend lookups.
- Closing as ClawHub would be premature because the requested work includes OpenClaw-owned config, docs, configure/apply guidance, runtime collection, and regression coverage; the supported external-backend pattern is not yet defined or shipped.
- #17311 may become the canonical broader design track, but it is open and not a shipped replacement for this focused batching/memoization issue.
