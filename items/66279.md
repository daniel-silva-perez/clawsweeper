## Summary

Keep open. Current main still matches the reported bug: the TUI path resolves URL/auth and constructs `GatewayClient` without any `tlsFingerprint`, while the shared CLI/status/probe paths already resolve local gateway TLS runtime and pass the fingerprint through. The related ClawSweeper context covers similar local self-signed TLS issues on other surfaces, but none supersedes this TUI-specific report.

## Best Possible Solution

Implement the missing TUI-specific trust handoff: when `openclaw tui` resolves a local configured `wss://127.0.0.1:<port>` gateway with `gateway.tls.enabled`, load the local gateway TLS runtime, include its `fingerprintSha256` in the resolved TUI connection, and pass it to `GatewayClient`. Add focused coverage in `src/tui/gateway-chat.test.ts` for local TLS fingerprint propagation and for the no-fingerprint fallback when TLS runtime cannot be loaded. If possible, reuse or extract the existing shared gateway TLS bootstrap logic from `src/gateway/call.ts` to avoid another drift-prone path.
