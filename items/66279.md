## Summary

Current main still matches the reported TUI-specific bug: `openclaw tui` resolves URL/auth and constructs `GatewayClient` without a `tlsFingerprint`, while the shared CLI/status/probe paths already load and forward the local gateway TLS fingerprint for local `wss://` targets. The related local self-signed TLS reports cover other surfaces and do not supersede this TUI gap.

## Best Possible Solution

Keep this issue open and implement the missing TUI trust handoff. `resolveGatewayConnection` should, for configured local `wss://` targets with `gateway.tls.enabled`, load the local gateway TLS runtime, include `fingerprintSha256` on the resolved connection, and pass it into `GatewayClient`. Add focused tests in `src/tui/gateway-chat.test.ts` for local TLS fingerprint propagation, unavailable-runtime fallback, and override/remote cases so this path stays aligned with the shared gateway call/status behavior.
