## Summary

Keep PR #70799 open. Current main still lacks the proposed outer guards in `markBlueBubblesChatRead` and `sendBlueBubblesTyping`, and the inbound BlueBubbles flow still calls those best-effort Private-API-only endpoints from webhook processing. The PR is a focused bug-fix candidate with useful reproduction/live-validation context, but maintainers should validate the exact cache/null race because the proposed `=== false` guard still preserves null fallthrough.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is to land a narrowly tested BlueBubbles fix that prevents Private-API-disabled typing/read-receipt calls from surfacing fallback warning replies, ideally with a regression covering the production cache/accountId state described in the PR. If the outer guard is insufficient for null state, move the guard to the inbound processing path or make known Private-API-disabled 500s non-fatal for these optional side effects.

## Risks / Open Questions

- The proposed patch may not cover the full reported null-cache race because it intentionally keeps null fallthrough; maintainers should add or request a regression that reproduces the accountId/cache mismatch, or gate these optional calls where `privateApiEnabled` is already computed.
- Changing typing/read behavior affects user-visible BlueBubbles side effects, so the final fix should preserve enabled-Private-API behavior and avoid converting optional side-effect failures into agent-turn failures.
