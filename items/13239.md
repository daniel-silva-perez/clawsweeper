## Summary

Current `main` does not implement the requested public provider compatibility matrix or a documented deprecation policy. OpenClaw has provider directory/docs plus `models list` and `models status`, but those surfaces expose provider catalogs, auth/readiness, and generic compat hints rather than stable/experimental/deprecated/broken support statuses with deadlines or policy windows.

## Evidence

- **Provider docs are a directory, not a compatibility matrix:** The provider landing page is a quick-start plus a flat list of provider docs. It does not define support states, deadlines, or a matrix by provider/model/capability.
  - file: [docs/providers/index.md:9](https://github.com/openclaw/openclaw/blob/3361593442e295e9bb1f55cbfb38861ea9413e45/docs/providers/index.md#L9)

  - sha: [3361593442e2](https://github.com/openclaw/openclaw/commit/3361593442e295e9bb1f55cbfb38861ea9413e45)
- **CLI docs only document auth/catalog views:** `openclaw models status` is documented as auth/default/fallback inspection, and `models list` as a read-only catalog/auth availability view. The documented outputs cover provider usage, auth, catalog rows, and availability, not support-policy statuses or deprecation deadlines.
  - file: [docs/cli/models.md:28](https://github.com/openclaw/openclaw/blob/3361593442e295e9bb1f55cbfb38861ea9413e45/docs/cli/models.md#L28)

## Risks / Open Questions

- There are partial adjacent surfaces already in place (`docs/providers/*`, `openclaw models list`, `openclaw models status`, and model `compat` hints), so future work should define whether the requested matrix extends those surfaces or introduces a new canonical schema to avoid duplicating provider metadata.
