## Summary

Keep #70133 open. Current main still matches the reported Feishu per-chat blocking behavior: normal same-chat messages share a stable Feishu sequential lane, the lane task awaits the full agent reply dispatch, and the queue only advances when the prior task promise settles. I did not find a timeout, abort, stale-tail release, or config/docs surface that would let later same-chat P2P messages proceed after a stuck dispatch. The related #58953 closure covers restart stale-lane recovery, not this plugin-local per-chat dispatch hang; related #43512 and #54409 instead support that Feishu queue edge cases remain unresolved.

## Best Possible Solution

Keep the issue open and fix it in the Feishu plugin. A conservative maintainer path is to add a bounded dispatch or queue-release guard around the per-chat lane, make timeout behavior explicit in logs and config if it is configurable, ensure processing claims and reply dispatcher/typing state settle correctly, and add a regression test where a first same-chat task never resolves but a later message proceeds after the guard without unsafe late-reply interleaving.
