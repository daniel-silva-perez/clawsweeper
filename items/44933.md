## Summary

Current `main` does not implement the requested WhatsApp pre-send registration check, standalone `message(action="check")`, or stronger delivery confirmation. The outbound path still treats Baileys send acceptance as success and returns only a `messageId`, so the feature request remains open work.

## Evidence

- **WhatsApp send path returns success after sendMessage only:** `sendMessageWhatsApp` resolves an active listener, calls `active.sendMessage(...)`, reads `result.messageId`, and returns `{ messageId, toJid }`. There is no pre-send registration/presence validation or post-send delivery wait in this path.
  - file: [extensions/whatsapp/src/send.ts:55](https://github.com/openclaw/openclaw/blob/40be5ad58187fd20b8defe5fc30e649190c131a8/extensions/whatsapp/src/send.ts#L55)
  - sha: [40be5ad58187](https://github.com/openclaw/openclaw/commit/40be5ad58187fd20b8defe5fc30e649190c131a8)
- **Active listener API has no registration-check or delivery-confirmation method:** The exported `ActiveWebListener` contract only exposes `sendMessage`, `sendPoll`, `sendReaction`, and `sendComposingTo`. There is no `onWhatsApp`/presence-check method or ack/delivery confirmation hook.
  - file: [extensions/whatsapp/src/inbound/types.ts:25](https://github.com/openclaw/openclaw/blob/40be5ad58187fd20b8defe5fc30e649190c131a8/extensions/whatsapp/src/inbound/types.ts#L25)
  - sha: [40be5ad58187](https://github.com/openclaw/openclaw/commit/40be5ad58187fd20b8defe5fc30e649190c131a8)
- **Baileys wrapper only extracts key.id from send result:** `createWebSendApi(...).sendMessage` calls `sock.sendMessage(...)` and converts the result to a `messageId` via `result.key.id`. It does not call Baileys `onWhatsApp()` or wait for server receipt/delivery status.
  - file: [extensions/whatsapp/src/inbound/send-api.ts:25](https://github.com/openclaw/openclaw/blob/40be5ad58187fd20b8defe5fc30e649190c131a8/extensions/whatsapp/src/inbound/send-api.ts#L25)
  - sha: [40be5ad58187](https://github.com/openclaw/openclaw/commit/40be5ad58187fd20b8defe5fc30e649190c131a8)
- **WhatsApp message-tool discovery exposes only react/poll extras:** `describeWhatsAppMessageActions` only adds `react` and `poll`; there is no WhatsApp-specific `check` action exposed from the plugin.
  - file: [extensions/whatsapp/src/channel-actions.ts:61](https://github.com/openclaw/openclaw/blob/40be5ad58187fd20b8defe5fc30e649190c131a8/extensions/whatsapp/src/channel-actions.ts#L61)
  - sha: [40be5ad58187](https://github.com/openclaw/openclaw/commit/40be5ad58187fd20b8defe5fc30e649190c131a8)
- **Global message action enum has no `check` action:** The shared message action name list includes `send`, `poll`, `react`, `read`, etc., but no `check`, so the proposed standalone `message(action="check")` is not present on current `main`. A repo search for `onWhatsApp`/presence-check terms also found no implementation in the WhatsApp extension.
  - file: [src/channels/plugins/message-action-names.ts:1](https://github.com/openclaw/openclaw/blob/40be5ad58187fd20b8defe5fc30e649190c131a8/src/channels/plugins/message-action-names.ts#L1)
