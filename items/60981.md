## Summary

Keep open. Current main still has only a binary `tools.fs.workspaceOnly` guard, not the requested PathGuard layer with canonicalized `allowedPaths` / `denyPaths`, structured policy denials, media/apply_patch parity, or spawned-subagent filesystem ceilings. This PR is still a plausible core security implementation candidate, but it is large and security-sensitive and the discussion shows unresolved correctness concerns that need maintainer review or a replacement implementation.

## Best Possible Solution

Keep this PR open for maintainer handling. The best path is to request a rebase and changes addressing the review-raised PathGuard wiring gaps, or replace it with a narrower core implementation under `tools.fs` that adds canonicalized deny-wins `allowedPaths` / `denyPaths`, structured denials, filesystem/media/apply_patch/prompt-image coverage, schema/docs updates, and monotonic spawned-subagent ceilings. Coordinate remaining scope with #7722 and the broader security profile in #8719.

- The PR is size XL and changes package metadata plus lockfile/config/protocol surfaces, so it needs a focused maintainer design pass and fresh tests before merge.
