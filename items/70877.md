## Summary

Keep this PR open. Current main still defaults tool-loop detection off in the detector and production hook path, while docs, tests, schema help, generated schema metadata, and UI fallback behavior still align with default-off behavior. The PR is a plausible focused fix for the reported repeated `web_fetch` DNS-error loop, but its current diff is not ready to close as implemented or duplicate/superseded: it only defaults the production resolver, leaves the lower-level detector default off, adds a direct no-config detector test that would fail against its own implementation, and changes generated schema text without the source schema/help/default metadata needed for regeneration and UI consistency.

## Best Possible Solution

Keep this PR open for maintainer review or consolidation with the overlapping loop-detection PRs. The best implementation is a minimal core change that enables loop detection by default in the actual runtime path, preserves explicit `enabled: false` and global-to-agent inheritance semantics, updates source zod schema/help/docs plus generated config metadata/baseline outputs, and adds focused coverage for no-config default-on, explicit false, global false with partial agent overrides, and UI/schema default consistency. If maintainers choose another canonical PR for the same fix, close this only after that PR is clearly designated or merged.
