## Summary

Keep PR #70877 open. Current main still leaves generic tool-loop detection disabled by default, the resolver passes missing config through without enabling it, tests and docs still lock/document the default-disabled behavior, and the PR discussion contains a real reproduction plus unresolved review feedback that needs author/maintainer handling rather than cleanup closure. The shipped unknown-tool guard is a separate safety net and does not cover repeated registered-tool failures like repeated `web_fetch` DNS errors.

## Best Possible Solution

Keep the PR open but require revisions. The clean path is to choose one authoritative defaulting layer for tool-loop detection, preserve explicit `enabled: false` inheritance across global and per-agent config, update `src/config/zod-schema.agent-runtime.ts`, `src/config/types.tools.ts`, generated schema/baseline docs, and add regression coverage for no-config default-on, explicit disable, and global-disable plus agent-threshold override behavior. If maintainers prefer the current opt-in policy, close or retarget the PR only after saying that explicitly.
