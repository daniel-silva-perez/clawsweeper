## Summary

Keep open. Current main contains a partial remote-CDP hardening change: `withCdpSocket` retries retryable WebSocket handshake/open failures with jittered backoff, and Playwright read paths can evict a stale cached connection once. But the requested auto-reconnect behavior is not fully implemented: the current test suite explicitly preserves a hard failure when the CDP socket closes after a command is in flight, and mutating Playwright operations are not replayed after an ambiguous disconnect. The related PR was closed unmerged, so it is not a canonical landed fix.

## Best Possible Solution

Keep this issue open and scope the fix in the browser plugin. The likely path is to retain the existing handshake retry, then add explicit recovery for stale CDP/Playwright connections after recoverable transport closes: retry safe read/snapshot operations, evict stale cached connections for later actions, avoid replaying ambiguous mutating commands unless the operation can be proven idempotent, and add targeted tests for remote-CDP close-after-open, next-action recovery, and retry exhaustion. The closed related PR #67748 can be used as background, but it should not be treated as a landed fix.

## Risks / Open Questions

- Closing this as implemented would hide the remaining remote-CDP reliability gap for socket closes after the WebSocket has opened or while a CDP command is in flight.
- A safe full fix must distinguish idempotent/read operations from mutating operations so reconnect does not duplicate tab creation, navigation, or other side effects.
