## Summary

Keep this issue open. Current main still routes Feishu/Lark DMs by sender Open ID only, even when inbound DM events include root/thread metadata. Topic-aware session keys, parent peers, thread bootstrap, and MessageThreadId are all gated to group/topic-group sessions, and the Feishu config/docs expose group topic routing but no DM per-topic session-scope option.

## Best Possible Solution

Keep this issue open. Implement the feature inside the bundled Feishu plugin with an explicit opt-in DM topic/session-scope config, preserve the current sender-scoped default, derive direct-message conversation IDs from sender plus canonical topic/root/thread metadata, propagate that conversation ID through routing, configured/runtime bindings, replies, and thread context, then add focused DM topic tests plus Feishu config schema and docs updates.

## Risks / Open Questions

- Feishu/Lark direct-message topic event semantics still need verification before choosing canonical precedence among root_id, thread_id, and message_id fallbacks.
- Changing DM conversation IDs by default would break existing sender-scoped sessions, bindings, dynamic-agent creation, and ACP workflows, so the feature should be opt-in.
- Existing ACP thread binding support for Feishu DMs is a partial workaround for bound conversations, but it does not provide general per-topic DM session isolation for normal inbound routing.
