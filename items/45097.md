## Summary

Close as cannot_reproduce. On current main, media-understanding CLI prompts are interpolated into an argv array and executed through `execFile` with shell execution disabled, so the PR’s built-in Gemini `$(...)`/backtick CWE-78 claim does not reproduce as described.

## Evidence

- **CLI runner passes templated values as argv entries:** `runCliEntry()` builds `templCtx`, substitutes into each arg with `applyTemplate()`, and then calls `runExec(argv[0], argv.slice(1), ...)`. That is argv execution, not shell interpolation.

## Close Comment

Closing after a Codex review because the reported reproducer does not reproduce on current `main`. The media-understanding CLI path templates values into an argv array in `src/media-understanding/runner.entries.ts:733` and executes it through `runExec()` in `src/media-understanding/runner.entries.ts:750`. `runExec()` uses `execFile` and hard-disables shell execution in `src/process/exec.ts:117`, so prompt text like `$(whoami)` or backticks is passed as a literal argument instead of being evaluated by a shell. The built-in Gemini media-understanding entry is also a direct `gemini` invocation with `{{Prompt}}` as its own argv item in `src/media-understanding/runner.ts:418`, not a shell-wrapper command. On Windows, the `.cmd`/`.bat` fallback already rejects real `cmd.exe` metacharacters in `src/process/exec.ts:15`. Based on that current-main evidence, the specific CWE-78 path claimed here is not reproducible. If you have a concrete reproducer against current `main` using an explicit shell-wrapper CLI config, please open a narrower follow-up with the exact config and platform details.
