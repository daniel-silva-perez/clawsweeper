## Summary

Keep open. Current main still handles `doctor.memory.status` by awaiting a fresh live embedding probe before responding, and the memory-core probe still embeds `["ping"]` through the normal batch path with 120s remote / 600s local default batch timeouts. I found caller-side doctor timeouts, but no server-side cache, strict status budget, or lite health method that would satisfy the issue's ask.

## Best Possible Solution

Keep the issue open and fix it in the gateway/memory status boundary: make the default `doctor.memory.status` path use cached or recently observed embedding readiness with a strict short server-side budget, reserve the existing live provider probe for an explicit deep path, and add regression tests proving repeated status calls do not wait on `embedBatchWithRetry` while deep checks still can.
