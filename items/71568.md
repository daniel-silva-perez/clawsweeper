## Summary

Keep open. Current main still implements `doctor.memory.status` as a live embedding-readiness RPC: the handler awaits `manager.probeEmbeddingAvailability()` before responding, and the built-in memory manager still implements that probe by embedding `["ping"]` through the batch embedding path. I found no current cache, `lastProbeAt`/cached-result payload, or short server-side budget that would close the issue’s central ask.

## Best Possible Solution

Keep the issue open and fix the status path in core. The best implementation is a per-agent/provider TTL cache for embedding probe results plus a short default server-side budget for `doctor.memory.status`, with explicit `lastProbeAt`/cached status metadata and a true deep/manual probe path for `openclaw memory status --deep` or an equivalent refresh. Add gateway handler tests proving repeated status calls do not run a live embedding probe within the TTL, timeout tests for a stalled provider, and docs/protocol updates for the cached-vs-deep behavior.
