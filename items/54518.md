## Summary

Current main still does not provide hard per-agent workspace confinement by default. The agent workspace is documented and implemented as the default working directory plus optional guards; `tools.fs.workspaceOnly` remains opt-in, tests assert outside-workspace host file access is allowed when unset/false, and exec/process/plugin/MCP-style surfaces need separate policy or sandbox coverage.

## Best Possible Solution

Keep this issue open for maintainer product and security judgment. A complete solution should define a per-agent confinement profile or onboarding option that enables `tools.fs.workspaceOnly: true`, denies or sandboxes runtime tools such as `exec` and `process`, covers plugin/MCP/media path surfaces, and updates skill guidance to avoid fixed default-workspace paths. If maintainers want hard workspace boundaries by default, it needs an explicit migration plan and regression coverage across file tools, sandbox modes, per-agent overrides, plugin tools, and legacy host-file workflows.

## Risks / Open Questions

- Changing the default workspace boundary would be a compatibility/product-policy change because current docs, config types, implementation, and tests intentionally preserve host file access unless opted into `tools.fs.workspaceOnly` or sandboxing.
- `tools.fs.workspaceOnly` alone is not complete confinement: exec/process, plugin tools, MCP tools, media paths, sandbox mounts, elevated mode, and channel/provider-specific paths need separate policy coverage.
- Third-party or shared skills that hard-code `~/.openclaw/workspace` can still steer agents toward the wrong workspace unless skill guidance and installers favor active-workspace-relative paths.
