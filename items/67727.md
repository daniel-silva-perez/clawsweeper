## Summary

Keep this PR open. Current main still has only uniform/per-channel inbound debounce; `messages.inbound.botDebounceMs` is not in the config types, zod schema, generated config metadata, field help/labels, docs, or debounce resolver. The PR is a plausible implementation path, but it remains incomplete as product behavior until `senderIsBot` is wired through production channel call sites or the PR is explicitly scoped to a preparatory API change.

## Best Possible Solution

Keep this PR open and ask for a complete product path: add the config type/schema/generated metadata/tests, then wire bot-origin detection through the shared debounce API and the relevant channel/plugin handlers that can reliably identify bot senders. If maintainers prefer a staged API-only PR, the PR body and tests should make that explicit and avoid presenting `botDebounceMs` as effective production behavior until the follow-up lands.
