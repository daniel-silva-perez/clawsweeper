## Summary

Keep open. Current main has `openclaw cron show <id>`, but it is implemented by paging `cron.list` and filtering client-side, which is the workaround this issue asks to avoid. The gateway method registry, protocol schemas/validators, method scopes, and cron agent tool still expose `cron.list/status/add/update/remove/run/runs` but no `cron.get`. The linked implementation PR #67677 was closed unmerged, so it did not ship the requested API.

## Best Possible Solution

Keep this issue open and implement the additive read path in core: add `cron.get` to the gateway method list/scopes, protocol schema/types/validators, and `src/gateway/server-methods/cron.ts`, backed by the existing `CronService.getJob` seam with a clean not-found error. Then update `openclaw cron show` to use that direct RPC, optionally add `openclaw cron get <id>` as an alias or command, add the cron agent tool action if agent workflows are in scope, and cover CLI/protocol/server behavior with tests. The closed unmerged PR #67677 is relevant prior art but should be reviewed/rebased rather than treated as shipped.
