## Summary

Keep open. Current main still lacks the requested direct `cron.get` gateway RPC and `openclaw cron get <id>` read path. The existing `cron show` command inspects one job only by paging `cron.list` and filtering locally, which is the workaround this issue asks to avoid. The related implementation PR #67677 was closed unmerged, so it is useful prior art but not shipped behavior.

## Best Possible Solution

Keep this issue open. The best path is to implement the additive read API in core: add `cron.get` to the gateway method list, scopes, protocol schema/types/validators, and `src/gateway/server-methods/cron.ts`, backed by `CronService.getJob(id)` with a clean not-found error. Then update `openclaw cron show` to use the direct RPC, add `openclaw cron get <id>` as the requested CLI path or alias, add a cron agent tool action if agent workflows remain in scope, and cover CLI/protocol/server behavior with focused tests. Review or rebase #67677 as prior art rather than treating it as shipped.
