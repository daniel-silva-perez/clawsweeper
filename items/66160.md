## Summary

Keep this PR open. Current main still accepts `--custom-base-url` but LiteLLM non-interactive onboarding is wired through the generic API-key provider helper, so the flag is not consumed by the LiteLLM setup path. The PR remains a plausible focused fix candidate, but it needs maintainer review, rebase/mergeability work, and regression coverage before landing because it also adds model discovery behavior.

## Best Possible Solution

Keep this PR open as a valid bug-fix candidate. The best maintainer path is to ask for a rebase onto current main, add focused regression coverage for LiteLLM non-interactive `--custom-base-url`, and then decide whether to keep, narrow, or separately review the added `/v1/models` discovery behavior and its security/logging tradeoffs.
