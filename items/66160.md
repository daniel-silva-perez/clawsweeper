## Summary

Keep this PR open. Current main still has the LiteLLM non-interactive onboarding gap: the CLI accepts `--custom-base-url`, but LiteLLM is registered through the generic single-provider API-key helper, whose non-interactive path applies only `applyLitellmConfig(ctx.config)`. That LiteLLM config helper reads an existing LiteLLM baseUrl or falls back to `http://localhost:4000`; it does not consume the onboarding flag. The PR is a focused implementation candidate, though it still deserves maintainer review around the added discovery behavior and a regression test before merge.

## Best Possible Solution

Keep the PR open as a valid bug-fix candidate. The best path is to review the updated branch, require focused regression coverage for LiteLLM non-interactive `--custom-base-url`, and either accept or narrow the added model-discovery behavior after resolving the remaining review concerns.
