## Summary

Current `main` does not add `WORKING.md` to the built-in bootstrap set, but the requested behavior now fits OpenClaw's documented hook/plugin extension path rather than core. This is optional agent-memory workflow customization, and project scope says optional capability should usually ship as plugins.

## Risks / Open Questions

- TypeScript hook authors would still need a cast or JS handler today because `WorkspaceBootstrapFileName` excludes `WORKING.md`; that is extension ergonomics debt, but it does not make this a core-only feature.

## Close Comment

Codex review: closing this as `clawhub` rather than a core cleanup item. Current main now documents `agent:bootstrap` with a mutable `context.bootstrapFiles` array, and OpenClaw supports plugin/workspace hook packs for exactly this kind of customization (`docs/automation/hooks.md:123`, `docs/automation/hooks.md:131`, `docs/automation/hooks.md:144`). The runtime that consumes hook-added bootstrap files only requires a valid `path` and then injects/truncates the provided file, so a community hook/plugin can supply `WORKING.md` without a core bootstrap-file expansion (`src/agents/bootstrap-files.ts:147`, `src/agents/pi-embedded-helpers/bootstrap.ts:286`). That matches project scope: optional capability should usually ship as plugins, not core (`VISION.md:54`). `WORKING.md` is also still not part of the built-in bootstrap contract on main (`src/agents/workspace.ts:136`, `docs/concepts/agent-workspace.md:64`), which makes this a better fit for a ClawHub/community hook or plugin than for expanding the default core bootstrap surface.
