## Summary

Keep PR #63037 open. Current main still has no timeout for WhatsApp Web connection waits, so the central hang it targets is not implemented away. The PR is still useful, but it needs revision for current main: cover both active startup paths and close/abort the Baileys socket when the timeout fires.

## Best Possible Solution

Keep the PR open and revise it against current main: make the connection wait timeout opt-in with default wait-forever semantics for QR login, apply an explicit startup deadline to the WhatsApp monitor/controller retry paths, close the created socket when the timeout fires, and add focused tests for timeout rejection, listener/timer cleanup, QR-login compatibility, and socket cleanup.

## Risks / Open Questions

- The PR diff only passes an explicit timeout through the direct `monitorWebInbox` call; current main also opens WhatsApp sockets through `WhatsAppConnectionController.openConnection`, which needs the same startup-timeout treatment or a shared helper.
