## Summary

Keep this PR open. Current main still has an unbounded WhatsApp connection wait, and the PR remains aimed at a real failure mode, but the current branch is not the best complete fix because it only wires the timeout through the legacy direct inbound path and still leaves the controller-based Gateway monitor startup path unbounded.

## Best Possible Solution

Keep the PR open and ask for a rebase/update: add an opt-in initial WhatsApp connection timeout through waitForWaConnection and WhatsAppConnectionController.openConnection, use it from the Gateway monitor startup path and the legacy direct inbound path, preserve unbounded QR login waits unless the login layer supplies its own deadline, close timed-out sockets, clean up listeners/timers, and add focused tests for timeout rejection, listener cleanup, socket close on startup timeout, and QR-login behavior.

## Risks / Open Questions

- Landing the PR as-is would likely leave the current controller-based WhatsApp Gateway monitor startup path able to hang indefinitely.
- The direct monitor timeout path needs to close the socket on timeout before rethrowing; otherwise repeated reconnect attempts can leave orphaned WhatsApp sockets.
- A global default timeout on waitForWaConnection would regress QR login flows, so the timeout should remain opt-in or be supplied by the caller that owns the deadline.
