## Summary

Keep open. Current main still clears cron maintenance stale `runningAtMs` markers with a fixed 2-hour threshold, and the timeout-aware cleanup proposed by PR #51024 is not present or clearly superseded. Existing timeout policy and normal timeout handling exist, but they do not solve the specific incomplete-finalization recovery case before the 2-hour stale window.

## Best Possible Solution

Keep the PR open and either rebase/review it or implement the equivalent on current main: derive stale `runningAtMs` cleanup in `src/cron/service/jobs.ts` from `resolveCronJobTimeoutMs(job)` with a grace window and conservative fallback for unlimited jobs, then add focused regression coverage for an `agentTurn` job with short explicit `timeoutSeconds` plus explicit coverage for the chosen default-timeout semantics.
