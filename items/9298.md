## Summary

Jami support is a reasonable idea, but current OpenClaw scope routes new messaging-platform integrations to plugins and ClawHub rather than adding them to core/bundled surfaces by default.

## Evidence

## Risks / Open Questions

- If maintainers later decide Jami is a high-demand channel with a strong product or security reason for first-party bundling, this scope call should be revisited.
- The issue is brief, so it does not rule out a narrower core-sdk gap specific to Jami; it only supports closing the broad 'support Jami messenger' request as a core feature request.

## Close Comment

Codex review: this looks better scoped as a ClawHub/community channel plugin than an OpenClaw core item. `VISION.md:54-61` says optional capability should usually ship as plugins and that the bar for adding optional plugins to core is intentionally high. `docs/plugins/community.md:9-22` says community plugins can add new channels and that ClawHub is the canonical discovery surface, and `docs/plugins/sdk-channel-plugins.md:11-35` documents the supported path for connecting OpenClaw to a messaging platform as a channel plugin. I also checked current `main` and there is no first-party Jami integration on top of which this issue is tracking follow-up work. Closing this as `clawhub`. If someone starts a Jami plugin, ClawHub/npm is the intended path; if there is a specific core SDK gap that blocks a Jami plugin, please open a focused issue for that gap.
