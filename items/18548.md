## Summary

Keep open. Current `main` has extension points for custom compaction/context-engine behavior, but the legacy core path targeted by the issue still truncates bootstrap files with fixed head/tail ratios and still finishes built-in staged compaction with a final summary merge pass. That is partial progress, not strong evidence that the requested architecture changes are implemented on `main`.

## Evidence

- **Bootstrap truncation is still positional:** `trimBootstrapContent()` still uses fixed `BOOTSTRAP_HEAD_RATIO`/`BOOTSTRAP_TAIL_RATIO` splits and slices `trimmed.slice(0, headChars)` plus `trimmed.slice(-tailChars)`, so the core bootstrap path still drops the middle of large files rather than selecting content by relevance.
  - file: [src/agents/pi-embedded-helpers/bootstrap.ts:95](https://github.com/openclaw/openclaw/blob/a59d1bd46d9fc0bd2d9194b3f17db2561efc51e3/src/agents/pi-embedded-helpers/bootstrap.ts#L95)
  - command: `nl -ba src/agents/pi-embedded-helpers/bootstrap.ts | sed -n '87,215p'`
  - sha: [a59d1bd46d9f](https://github.com/openclaw/openclaw/commit/a59d1bd46d9fc0bd2d9194b3f17db2561efc51e3)
- **Built-in compaction still ends with a merge-summary pass:** `summarizeInStages()` still splits messages, summarizes each chunk, then converts the partial summaries back into messages and runs `summarizeWithFallback()` again on those summaries. That means the built-in path still relies on a final summary merge step rather than replacing it with a fully incremental non-LLM compactor.
  - file: [src/agents/compaction.ts:444](https://github.com/openclaw/openclaw/blob/a59d1bd46d9fc0bd2d9194b3f17db2561efc51e3/src/agents/compaction.ts#L444)

That partially covers the reporter's plugin-based direction, but it does not prove the default legacy behavior was replaced.
