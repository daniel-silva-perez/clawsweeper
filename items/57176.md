## Summary

The request is coherent and useful, but current main treats this kind of multi-agent observability/orchestration layer as extension scope, not core scope. OpenClaw already exposes task/task-flow/session-lineage primitives that a ClawHub skill or plugin can consume, and the project vision explicitly keeps optional capabilities, new skills, and agent-hierarchy-heavy features out of core by default.

## Risks / Open Questions

- This closes on scope, not because the idea is bad; OpenClaw still does not ship a built-in `openclaw trace` UX for the exact workflow described.
- If maintainers later decide first-party multi-agent observability is strategic, this issue could merit reopening as a product-direction exception to `VISION.md`.

## Close Comment

Thanks for the detailed write-up. After a Codex review of current `main`, this looks better scoped as a ClawHub plugin/skill than a new OpenClaw core feature.

- `VISION.md:54` says optional capability should usually ship as plugins, and `VISION.md:69` says new skills should go to ClawHub first.
- `VISION.md:101` and `VISION.md:106` explicitly keep agent-hierarchy frameworks and heavy orchestration layers out of default core scope.
- `docs/plugins/community.md:9` documents ClawHub/community plugins as the intended extension path, and `docs/plugins/community.md:77` already lists an official trace-export plugin (`@opik/opik-openclaw`).

If you want to pursue this, the ClawHub/plugin route matches the current project scope much better than adding a new built-in `openclaw trace` subsystem.
