## Summary

Keep #51860 open. Current main still models macOS gateway service management as a per-user LaunchAgent, exposes no `gateway install --system` option, and only scans `/Library/LaunchDaemons` as secondary `--deep` discovery rather than treating a system LaunchDaemon as the managed gateway service. The issue discussion also contains a tested LaunchDaemon workaround and a later OpenClaw 2026.4.7 reproduction for stale session-bound `TMPDIR`, which remains consistent with current service environment behavior.

## Best Possible Solution

Keep this issue open and implement the macOS system service path in core. The likely fix belongs in `src/daemon/launchd.ts`, `src/daemon/service.ts`, and the gateway install/status CLI: add an explicit `openclaw gateway install --system` mode, generate a root-owned `/Library/LaunchDaemons/ai.openclaw.gateway.plist` with `UserName` and a durable temp directory, teach status/readRuntime/readCommand to detect and query the system launchd domain, and add doctor/tests/docs for stale LaunchAgent TMPDIR repair and LaunchDaemon status reporting.

## Risks / Open Questions

- A first-party macOS LaunchDaemon path will need careful sudo/root ownership handling, `UserName` selection, and system-domain launchctl status/restart semantics.
- Changing TMPDIR persistence needs regression coverage for the earlier SQLite temp/journal reliability fix while avoiding session-bound `/var/folders/.../T/` values for unattended boots.
