## Summary

Current `main` still reloads chat history after every same-session final chat event because `shouldReloadHistoryForFinalEvent` returns true for any final payload. The PR’s central behavior is not implemented, and the related open reload-race issue suggests this is still part of unresolved Control UI race work rather than a cleanup-close candidate.

## Best Possible Solution

Keep the PR open or fold its intent into the broader Control UI chat reload-race work: skip the final-event history reload when the payload already contains a displayable assistant message, while preserving deferred `session.message` reloads for unrelated transcript updates and adding targeted regression tests in `ui/src/ui/chat-event-reload.test.ts` and `ui/src/ui/app-gateway.node.test.ts`.
