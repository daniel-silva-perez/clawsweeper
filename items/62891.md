## Summary

Keep this issue open. Current main still starts and restarts cron without a bounded dependency on connected channel readiness; startup waits for the channel-start handoff, not for Discord WS READY/connected state, and config hot reload can restart cron before channel restarts complete.

## Best Possible Solution

Implement a bounded cron startup and reload readiness gate at the gateway/cron boundary. It should delay automatic cron timer execution and startup catch-up until configured delivery channels are connected/healthy, a configurable wait expires, or readiness fails with clear diagnostics. Cover full startup, config.patch/SIGUSR1 restart or hot reload, Discord-style async readiness, and existing /readyz/channel-health behavior with targeted gateway and cron tests.
