## Summary

Keep #62891 open. Current main has a related shipped fix in #65365 that delays cron/heartbeat activation until post-attach sidecars finish, but the issue's refined failure is stricter: cron should not fire before configured channel plugins, especially Discord, are actually connected/ready. Current startup still treats channel startup as handed off once the account task is scheduled, while Discord readiness is awaited inside that long-running task. The hot-reload path can also restart cron before channel restarts complete.

## Best Possible Solution

Keep the issue open and implement a bounded channel-readiness gate for scheduled gateway services. Cron/heartbeat startup should wait for configured managed channel accounts to report connected, or for a configurable timeout such as cron.channelReadyWaitMs, before arming due jobs. Apply the same ordering to config hot reload when cron and channels restart together: restart channels first, wait for readiness or timeout, then start cron. Add regression coverage with a channel startAccount task that is handed off but not yet connected, plus a reload case involving both restartCron and restartChannels.

## Risks / Open Questions

- A strict global channel-ready gate could block cron indefinitely when an optional or misconfigured channel never connects; it should be bounded/configurable and scoped to configured managed channels or delivery dependencies.
- Closing this now would lose the remaining distinction between sidecar task handoff and actual Discord/channel readiness.
