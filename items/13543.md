## Summary

Keep open. Current `main` still implements sandboxing as a session/agent-level runtime choice plus sandbox-only tool allow/deny filtering. I found no generic per-tool host-vs-sandbox routing independent of session sandbox mode, so the feature request is not implemented on `main` and does not fit a cleanup close.

## Evidence

- **Sandbox config is session/agent scoped:** `AgentSandboxConfig` only exposes `mode: "off" | "non-main" | "all"` plus backend/workspace/scope settings; there is no per-tool execution-mode field.
  - file: [src/config/types.agents-shared.ts:24](https://github.com/openclaw/openclaw/blob/3361593442e295e9bb1f55cbfb38861ea9413e45/src/config/types.agents-shared.ts#L24)
  - command: `nl -ba src/config/types.agents-shared.ts | sed -n '15,40p'`
  - sha: [3361593442e2](https://github.com/openclaw/openclaw/commit/3361593442e295e9bb1f55cbfb38861ea9413e45)
- **Sandbox runtime resolution is session-based:** `resolveSandboxContext()` first calls `resolveSandboxSession()`, which returns `null` unless the session runtime is already marked sandboxed. That means sandbox activation happens per session key, not per tool call.
  - file: [src/agents/sandbox/context.ts:108](https://github.com/openclaw/openclaw/blob/3361593442e295e9bb1f55cbfb38861ea9413e45/src/agents/sandbox/context.ts#L108)
  - command: `nl -ba src/agents/sandbox/context.ts | sed -n '95,150p'`
  - sha: [3361593442e2](https://github.com/openclaw/openclaw/commit/3361593442e295e9bb1f55cbfb38861ea9413e45)
- **Docs describe sandbox as deciding where tools run, with sandbox tool policy only restricting tools inside sandbox:** The docs say sandbox decides where tools run, with modes `off`/`non-main`/`all`, and separately say `tools.sandbox.tools.*` only applies when sandboxed.
  - file: [docs/gateway/sandbox-vs-tool-policy-vs-elevated.md:8](https://github.com/openclaw/openclaw/blob/3361593442e295e9bb1f55cbfb38861ea9413e45/docs/gateway/sandbox-vs-tool-policy-vs-elevated.md#L8)

  - sha: [3361593442e2](https://github.com/openclaw/openclaw/commit/3361593442e295e9bb1f55cbfb38861ea9413e45)
- **Sandbox tool config only supports allow/deny lists:** `tools.sandbox.tools` supports `allow`, `alsoAllow`, and `deny` only. There is no schema for `always`/`never` per-tool sandbox routing like the issue proposes.
  - file: [src/config/types.tools.ts:642](https://github.com/openclaw/openclaw/blob/3361593442e295e9bb1f55cbfb38861ea9413e45/src/config/types.tools.ts#L642)

  - sha: [3361593442e2](https://github.com/openclaw/openclaw/commit/3361593442e295e9bb1f55cbfb38861ea9413e45)
- **Only exec has an explicit host-vs-sandbox override:** The exec docs expose `host: auto|sandbox|gateway|node`, with `auto` resolving based on whether the session already has a sandbox runtime. That is a tool-specific exception, not a general per-tool sandbox policy system.
  - file: [docs/tools/exec.md:43](https://github.com/openclaw/openclaw/blob/3361593442e295e9bb1f55cbfb38861ea9413e45/docs/tools/exec.md#L43)

  - file: [src/agents/sandbox/constants.ts:13](https://github.com/openclaw/openclaw/blob/3361593442e295e9bb1f55cbfb38861ea9413e45/src/agents/sandbox/constants.ts#L13)
