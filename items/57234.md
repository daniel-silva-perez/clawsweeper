## Summary

Keep open. Current `main` no longer appears to rely on Playwright's artifact dir for the explicit download helpers, but it still hardcodes browser downloads into OpenClaw's own temp downloads root and exposes no documented `browser.downloadsPath` or equivalent config to route CDP-triggered downloads to the user's Downloads folder.

## Evidence

- **No documented downloads-path config:** The browser config reference lists `cdpUrl`, timeouts, profile settings, and related browser options, but no `browser.downloadsPath` or similar setting. A repo search for `downloadsPath`/`browser.downloadsPath` returned no matches.
  - file: [docs/tools/browser.md:100](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/docs/tools/browser.md#L100)
  - command: `rg -n "downloadsPath|browser\.downloadsPath|downloadsPath option" docs src extensions -g '!**/*.test.ts' && nl -ba docs/tools/browser.md | sed -n '100,190p'`
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
- **Downloads are hardcoded to OpenClaw temp storage:** `DEFAULT_DOWNLOAD_DIR` is defined as `<preferred OpenClaw tmp dir>/downloads`, not the user's Downloads folder or a configurable path.
  - file: [extensions/browser/src/browser/paths.ts:25](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/extensions/browser/src/browser/paths.ts#L25)
  - command: `nl -ba extensions/browser/src/browser/paths.ts | sed -n '25,30p'`
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
- **Download routes enforce the isolated temp downloads root:** Both `/wait/download` and `/download` call `ensureOutputRootDir(DEFAULT_DOWNLOAD_DIR)` and resolve requested output paths within that root, so callers cannot redirect downloads to arbitrary locations such as `~/Downloads`.
  - file: [extensions/browser/src/browser/routes/agent.act.download.ts:48](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/extensions/browser/src/browser/routes/agent.act.download.ts#L48)
  - command: `nl -ba extensions/browser/src/browser/routes/agent.act.download.ts | sed -n '48,66p' && nl -ba extensions/browser/src/browser/routes/agent.act.download.ts | sed -n '101,116p'`
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
- **CLI and docs still describe temp-download behavior:** The CLI help says downloads are saved within the OpenClaw temp downloads dir, and the security docs explicitly recommend an isolated downloads directory rather than the user's main Downloads folder.
  - file: [extensions/browser/src/cli/browser-cli-actions-input/register.files-downloads.ts:125](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/extensions/browser/src/cli/browser-cli-actions-input/register.files-downloads.ts#L125)

## Risks / Open Questions

- The original disappearance bug may be partially mitigated for the explicit download helper routes, but current main still does not implement the requested user-Downloads destination or a configurable browser download path.
- Closing as `implemented_on_main` would overstate what shipped, because the current behavior is an isolated OpenClaw temp downloads directory, not the requested `~/Downloads` flow.
