## Summary

Do not close this issue yet. Current main still routes Linux writes through the pinned Python write helper and still normalizes a missing `python3` spawn failure to the generic `path is not a regular file under root` message. There is also an open implementation candidate, #72531, that explicitly uses closing syntax for this issue, so the right maintainer path is to review and land or reject that PR rather than close the issue first.

## Best Possible Solution

Keep this issue open and review #72531 or an equivalent fix. The minimum acceptable product path is to surface a clear missing-`python3` diagnostic with regression coverage for `writeFileWithinRoot`/`copyFileWithinRoot`; a stronger path is to remove the Python runtime dependency or document and doctor-check it wherever Docker/slim images are supported. Close this issue only after the chosen fix lands on main.

## Evidence

- **Current main still wraps missing helper failures generically:** `writeFileWithinRoot` calls `runPinnedWriteHelper` on non-Windows and catches all helper failures through `normalizePinnedWriteError`; that normalizer returns `SafeOpenError("invalid-path", "path is not a regular file under root")` for every non-`SafeOpenError`, preserving the cause but not surfacing `spawn python3 ENOENT` in the message.
  - file: [src/infra/fs-safe.ts:714](https://github.com/openclaw/openclaw/blob/4b9c85776d29b1cc81ecbdacd7feaca6a46c0941/src/infra/fs-safe.ts#L714)
