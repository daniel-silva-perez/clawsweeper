## Summary

Keep open. Current `main` still implements Signal only through `signal-cli` JSON-RPC + SSE, does not expose an `apiMode`/REST path, and still drops all `syncMessage` envelopes instead of allowing the proposed Note to Self handling.

## Evidence

- **Signal client is JSON-RPC/SSE-only:** `signalRpcRequest()` posts only to `/api/v1/rpc`, `signalCheck()` probes only `/api/v1/check`, and `streamSignalEvents()` connects only to `/api/v1/events`.
  - file: [extensions/signal/src/client.ts:71](https://github.com/openclaw/openclaw/blob/754acc4478fe361db1e5303fd60f589d52beea7a/extensions/signal/src/client.ts#L71)

  - sha: [754acc4478fe](https://github.com/openclaw/openclaw/commit/754acc4478fe361db1e5303fd60f589d52beea7a)
- **Monitor loop only uses SSE receiver:** `monitorSignalProvider()` always calls `runSignalSseLoop()` for inbound Signal traffic; there is no alternate REST/WebSocket receive path.
  - file: [extensions/signal/src/monitor.ts:478](https://github.com/openclaw/openclaw/blob/754acc4478fe361db1e5303fd60f589d52beea7a/extensions/signal/src/monitor.ts#L478)
