## Summary

Keep this PR open. Current main still does not request or read Codex CLI `--output-last-message` artifacts, so the central behavior is not implemented or obsolete. The PR is a plausible implementation candidate for the closed related enhancement, but review comments identify correctness and cleanup issues that should be fixed before merge.

## Best Possible Solution

Keep this PR open and ask for a scoped revision: inject `--output-last-message` for the `codex-cli` backend before positional prompt args, read a non-empty artifact as the final reply while preserving stdout for telemetry and fallback, keep cleanup within the existing `finally` flow, mask the artifact path in verbose logs, and add focused tests for artifact precedence, empty-artifact fallback, and Codex JSONL message-content parsing.

## Risks / Open Questions

- The PR changes a command execution path by adding a Codex CLI flag and a temp artifact file. That is consistent with the PR purpose, but the implementation should keep temp creation and cleanup inside the existing guarded cleanup flow.
- The PR's JSONL fallback, as shown in the provided diff and review comments, is not scoped to `codex-cli`; if landed unchanged, future or custom JSONL backends could fold diagnostic `message`, `result`, or `response` fields into assistant replies.
- The PR currently risks fragile argv ordering if `--output-last-message` is appended after the positional prompt for arg-mode backends.
- Verbose CLI argv logging should mask or abbreviate the generated artifact path to avoid leaking host temp directory details.
