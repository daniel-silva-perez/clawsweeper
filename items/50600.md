## Summary

Keep open. Current `main` already guards the empty-reply part of the report, including Telegram-specific `NO_REPLY` handling, but it does not implement the requested configurable busy acknowledgment or a `queue:wait-exceeded` hook. The queue config surface still only exposes mode/debounce/cap/drop, and queue wait exceedance is only logged/diagnosed, not surfaced back to the originating channel.

## Evidence

- **Queued runs already provide immediate typing, not text ack:** Docs say queued inbound messages fire typing indicators immediately on enqueue when the channel supports them; this is the current UX mitigation rather than a configurable busy message.
  - file: [docs/concepts/queue.md:23](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/docs/concepts/queue.md#L23)
  - sha: [ebd7f19a3b20](https://github.com/openclaw/openclaw/commit/ebd7f19a3b20001960474ec9ba9cc424e45f7b23)
- **Queue config has no busy-ack fields:** `QueueConfig` only defines `mode`, `byChannel`, `debounceMs`, `debounceMsByChannel`, `cap`, and `drop`; there is no `busyAckMs` or `busyAckText` setting.
  - file: [src/config/types.messages.ts:13](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/src/config/types.messages.ts#L13)
  - sha: [ebd7f19a3b20](https://github.com/openclaw/openclaw/commit/ebd7f19a3b20001960474ec9ba9cc424e45f7b23)
- **Queue wait-exceeded path only logs:** When a lane wait exceeds the threshold, the command queue invokes an internal `onWait` callback if provided and emits a warning plus dequeue diagnostics; it does not emit a named hook/event like `queue:wait-exceeded` or send a channel acknowledgment.
  - file: [src/process/command-queue.ts:171](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/src/process/command-queue.ts#L171)
  - sha: [ebd7f19a3b20](https://github.com/openclaw/openclaw/commit/ebd7f19a3b20001960474ec9ba9cc424e45f7b23)
- **Empty/silent replies are suppressed before delivery:** Reply normalization returns `null` for empty payloads and for silent-token payloads whose remaining visible content is empty, preventing empty-string sends from the generic reply pipeline.
  - file: [src/auto-reply/reply/normalize-reply.ts:49](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/src/auto-reply/reply/normalize-reply.ts#L49)
  - sha: [ebd7f19a3b20](https://github.com/openclaw/openclaw/commit/ebd7f19a3b20001960474ec9ba9cc424e45f7b23)
- **Telegram path already covers the empty-reply bug:** Telegram delivery tests show exact `NO_REPLY` is rewritten for direct sessions and suppressed for group sessions, and the low-level Telegram sender rejects empty text with `Message must be non-empty for Telegram sends`.
  - file: [extensions/telegram/src/bot/delivery.test.ts:296](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/extensions/telegram/src/bot/delivery.test.ts#L296)
