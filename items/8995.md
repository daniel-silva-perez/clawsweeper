## Summary

Keep open. Current main has adjacent background-task and subagent observability, but it does not implement the issue’s requested `sessions_spawn` verification contract (`verify`/`heartbeat`/`validate`) or new `sessions_poll` / `sessions_verify` APIs. The request is still a plausible core enhancement rather than a clearly obsolete report.

## Evidence

- **`sessions_spawn` still lacks verification/heartbeat/validation inputs:** The current tool schema only accepts fields like `task`, `label`, `runtime`, `agentId`, `model`, `runTimeoutSeconds`, `thread`, `mode`, `cleanup`, `sandbox`, `context`, `streamTo`, `lightContext`, and attachments. There are no `verify`, `heartbeat`, or `validate` options.

They also still describe `sessions_spawn` as always non-blocking and returning only a `runId` and `childSessionKey`.

## Risks / Open Questions

- Closing as `implemented_on_main` would overstate coverage: the repo has monitoring via tasks/subagents/session_status, but not the explicit `sessions_spawn` verification and validation interface requested here.
- Some of the operational pain may already be reduced by the task control plane, so a future maintainer might choose to consolidate this into a narrower follow-up issue rather than implement the proposal verbatim.
