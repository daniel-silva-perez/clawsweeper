## Summary

Keep open. Current `main` still exits the whole process on any non-transient unhandled rejection, and the browser CDP path still uses Playwright `connectOverCDP` without a browser-scoped `registerUnhandledRejectionHandler` or other evidence that Playwright assertion failures are isolated from the gateway process.

## Evidence

- **Process-level unhandled rejection handler is still fatal by default:** `src/infra/unhandled-rejections.ts:349-382` logs generic unhandled rejections and calls `process.exit(1)` unless the error is AbortError, a configured fatal/config error, a transient network/sqlite error, or a custom registered handler returns `true`. A plain Playwright `Error: Assertion error` does not match any listed non-fatal bucket here.
  - file: [src/infra/unhandled-rejections.ts:349](https://github.com/openclaw/openclaw/blob/4630ce3d9e2f089062ccd386b0ae241325961d1a/src/infra/unhandled-rejections.ts#L349)
  - command: `nl -ba src/infra/unhandled-rejections.ts | sed -n '300,430p'`
  - sha: [4630ce3d9e2f](https://github.com/openclaw/openclaw/commit/4630ce3d9e2f089062ccd386b0ae241325961d1a)
- **OpenClaw installs that fatal handler at startup:** `src/index.ts:84-94` installs `installUnhandledRejectionHandler()` in the main entrypoint before running the CLI/server, so an unhandled Playwright rejection still reaches the fatal process-level handler.
  - file: [src/index.ts:84](https://github.com/openclaw/openclaw/blob/4630ce3d9e2f089062ccd386b0ae241325961d1a/src/index.ts#L84)
  - command: `nl -ba src/index.ts | sed -n '70,100p'`
  - sha: [4630ce3d9e2f](https://github.com/openclaw/openclaw/commit/4630ce3d9e2f089062ccd386b0ae241325961d1a)
- **Browser CDP path still goes through Playwright with disconnect-only recovery:** `extensions/browser/src/browser/pw-session.ts:428-465` still connects via `chromium.connectOverCDP(...)`, caches the browser, and only registers a `browser.on("disconnected", ...)` cleanup hook. There is no local try/catch around Playwright internal message dispatch and no browser-specific unhandled-rejection registration in this path.
  - file: [extensions/browser/src/browser/pw-session.ts:428](https://github.com/openclaw/openclaw/blob/4630ce3d9e2f089062ccd386b0ae241325961d1a/extensions/browser/src/browser/pw-session.ts#L428)

- **Repo has scoped unhandled-rejection suppression for other integrations, but not browser:** Search results show `registerUnhandledRejectionHandler(...)` is used in integrations like WhatsApp and Telegram, but no such usage appears under `extensions/browser/src/browser/**`. That means the browser runtime has no equivalent scoped suppression/recovery path for Playwright-originated unhandled rejections.
  - command: `rg -n "registerUnhandledRejectionHandler\(|isUnhandledRejectionHandled\(|installUnhandledRejectionHandler\(" src extensions -g '!**/dist/**' -g '!**/node_modules/**'`
  - sha: [4630ce3d9e2f](https://github.com/openclaw/openclaw/commit/4630ce3d9e2f089062ccd386b0ae241325961d1a)

That is useful hardening, but it is not evidence that an internal Playwright assertion/unhandled rejection is now swallowed instead of terminating the gateway.
