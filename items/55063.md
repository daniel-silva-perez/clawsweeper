## Summary

Keep #55063 open. Current main partially addresses the newer auto-fallback variant by clearing `modelOverrideSource: "auto"` on reset, but it still intentionally preserves explicit or legacy user model overrides across `/new` and `/reset`. That does not satisfy this issue's reported `/model ...` then `/new` repro or its expectation that reset falls back to `agents.defaults.model.primary`.

## Best Possible Solution

Keep the issue open for maintainer judgment on the reset contract. If `/new` and `/reset` should mean a truly model-fresh session, update `resolveResetPreservedSelection` and the chat/gateway reset tests so explicit user `modelOverride`/`providerOverride` are cleared on bare reset while preserving `/new <model>` as an immediate model selection. Also align `/status`/reset messaging and docs with the chosen contract. If preserving explicit pins is the desired behavior, document that explicitly and close this only with maintainer-approved rationale.
