## Summary

Current main already retries ordinary WhatsApp disconnects instead of treating them as fatal relink events. Inbound close handling only stops for logged-out sessions or explicit non-retryable conflict status 440; an ENOTFOUND-style disconnect with no logged-out status falls into the reconnect/backoff path. The latest release tag `v2026.4.22` also contains this behavior and the related reconnect tests.

## Close Comment

Codex review: closing as implemented on current `main`.

Current WhatsApp gateway code already routes connection closes through the reconnect loop unless the session is actually logged out or hits the explicit non-retryable conflict status 440. The close path in `extensions/whatsapp/src/inbound/monitor.ts:683` preserves `lastDisconnect.error`, the decision logic in `extensions/whatsapp/src/connection-controller.ts:425` retries ordinary disconnects with backoff, and the monitor in `extensions/whatsapp/src/auto-reply/monitor.ts:445` emits disconnect/retry notifications instead of immediately forcing relink. That means an `ENOTFOUND`-style DNS failure, which does not map to the logged-out path, now falls into retry behavior rather than permanent exit.

This behavior is also covered by reconnect tests in `extensions/whatsapp/src/auto-reply.web-auto-reply.connection-and-logging.e2e.test.ts:80`, and the latest release `v2026.4.22` already contains the same reconnect controller logic. If you still hit a reproducible DNS-blip relink on a newer build, please open a fresh issue with current version, logs, and the exact `lastDisconnect.error` payload so maintainers can match it against the live failure mode.
