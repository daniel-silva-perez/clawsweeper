## Summary

Keep open. PR #49166 targets a real remaining gap in the `session_status(model=...)` override path. Current `main` has the allowlist-based bare-model provider inference in the shared model-selection resolver used by sessions.patch/Control UI work, but `session_status` still bypasses that resolver and calls `resolveModelRefFromString()` with `defaultProvider: currentProvider`, so the PR's `openai-codex` -> bare `deepseek-chat` failure mode can still happen statically.

## Best Possible Solution

Keep the PR or replace it with a maintainer patch that makes `session_status(model=...)` use the same allowlist-aware provider inference as the shared model-selection path. A clean implementation should likely call the shared allowed-model resolver or apply the same `inferUniqueProviderFromConfiguredModels()` step before `resolveModelRefFromString()`, add the PR's `openai-codex` to `deepseek-chat` regression test in `src/agents/openclaw-tools.session-status.test.ts`, and address the harmless bot nit by using `splitTrailingAuthProfile(raw).model.trim()` without optional chaining.
