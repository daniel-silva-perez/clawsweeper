## Summary

Keep open. Current main still sends Feishu streaming replies as CardKit `card_id` references, but the Feishu read/thread-history path does not request raw card content and only parses inline `elements` / `body.elements`, so a `{type:"card", data:{card_id}}` interactive message still falls back to `[Interactive Card]`. The cross-referenced fix PR #55466 was closed unmerged, and its described `card_msg_content_type=raw_card_content` / `json_card` handling is not present on main.

## Best Possible Solution

Keep the issue open and either revive/supersede #55466 or implement the same focused fix in the Feishu plugin: add `card_msg_content_type: "raw_card_content"` to both single-message reads and thread-list reads, parse `json_card` recursively with summary/body text extraction, preserve legacy `elements` fallbacks, and add regression tests covering the exact CardKit `card_id` streaming-card read path.
