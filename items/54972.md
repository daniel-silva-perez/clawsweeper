## Summary

Current main still has the reported Feishu CardKit read gap: streaming replies are sent as `interactive` messages containing only a `card_id` reference, while message-get/thread-list reads do not request raw CardKit content and the interactive parser has no `json_card` or `card_id` resolution path. The related implementation PR #55466 was closed unmerged, so the issue remains actionable.

## Best Possible Solution

Keep this open and rework or revive #55466 after validating the current Feishu/Lark API contract. The likely fix belongs in the Feishu plugin read path: request raw CardKit content from both `getMessageFeishu` and `listFeishuThreadMessages`, parse `json_card` summary/body text before legacy element/post fallback, and add regression tests for quoted-message and thread-history reads of streaming CardKit cards.
