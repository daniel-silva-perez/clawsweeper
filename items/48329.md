## Summary

Keep open. Current `main` still uses fire-and-forget HTTP server shutdown for Feishu monitors, and this PR's change set is too narrow to cover the abort-driven webhook cleanup path that gateway restarts actually use.

## Evidence

- **Current main still does synchronous helper cleanup:** `stopFeishuMonitorState()` is still synchronous and only calls `server.close()` before deleting state; there is no await/promise-based shutdown in either the targeted or global cleanup branches.
  - file: [extensions/feishu/src/monitor.state.ts:146](https://github.com/openclaw/openclaw/blob/b453293349657dedd069d291fa5b078d910067b1/extensions/feishu/src/monitor.state.ts#L146)
  - command: `nl -ba extensions/feishu/src/monitor.state.ts | sed -n '130,190p'`
  - sha: [b45329334965](https://github.com/openclaw/openclaw/commit/b453293349657dedd069d291fa5b078d910067b1)
- **Gateway stop/restart path flows through abort-driven webhook cleanup:** `channel.ts` passes `ctx.abortSignal` into `monitorFeishuProvider()`, and `monitor.account.ts` routes webhook accounts into `monitorWebhook()`. That means real gateway shutdown uses the local abort cleanup in `monitor.transport.ts`, not `stopFeishuMonitorState()`.
  - file: [extensions/feishu/src/channel.ts:1203](https://github.com/openclaw/openclaw/blob/b453293349657dedd069d291fa5b078d910067b1/extensions/feishu/src/channel.ts#L1203)

- **Abort cleanup still only calls `server.close()`:** In `monitorWebhook()`, the abort `cleanup()` still does plain `server.close()` and immediately deletes `httpServers`, `botOpenIds`, and `botNames`. There is no `closeAllConnections()` or awaited close callback on current `main`.
