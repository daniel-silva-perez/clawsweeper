## Summary

Issue #9753 is a good plugin idea, but current main scope and docs place this kind of tool-governance work in the external plugin ecosystem, not OpenClaw core. Current main already exposes the needed `before_tool_call` hook and approval/block semantics for a community plugin to implement PIC-style intent verification.

## Close Comment

Thanks for the idea. Codex review on current `main` points to this belonging in the community plugin path rather than OpenClaw core: `VISION.md:54-59` says optional capability should usually ship as plugins and be hosted outside the repo, `docs/plugins/building-plugins.md:16-18` says external plugins should be published to ClawHub or npm, and `docs/plugins/community.md:9-16` makes ClawHub the canonical discovery surface for community plugins. Current `main` also already exposes the needed hook surface for this direction: `before_tool_call` is documented for tool/hook plugins in `docs/plugins/building-plugins.md:36-50` and implemented with block/approval semantics in `src/plugins/hook-types.ts:324-336` and `src/agents/pi-tools.before-tool-call.ts:202-256`. Closing this as a ClawHub/plugin-scope idea rather than a core OpenClaw issue.
