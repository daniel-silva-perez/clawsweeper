## Summary

Current main already contains the WhatsApp outbound listener-registry fixes for this failure mode, including call-time lookup through a `globalThis`-backed registry and regression coverage for duplicate bundled module instances and omitted-account/default-account sends.

## Evidence

- **Global singleton registry:** The WhatsApp connection-controller registry is stored behind `Symbol.for("openclaw.whatsapp.connectionControllerRegistry")` on `globalThis`, so duplicated bundled module copies share one registry state instead of separate Maps.
  - file: [extensions/whatsapp/src/connection-controller-registry.ts:11](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/extensions/whatsapp/src/connection-controller-registry.ts#L11)
  - command: `nl -ba extensions/whatsapp/src/connection-controller-registry.ts | sed -n '1,80p'`
- **Outbound send resolves listener at call time:** `requireOutboundActiveWebListener` now resolves the account id, then reads the active listener from `getRegisteredWhatsAppConnectionController(...).getActiveListener()` at send time rather than relying on a stale module-scoped listener map.
  - file: [extensions/whatsapp/src/send.ts:39](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/extensions/whatsapp/src/send.ts#L39)
  - command: `nl -ba extensions/whatsapp/src/send.ts | sed -n '30,55p'`
- **Regression test for split bundle copies:** There is explicit regression coverage proving two separately imported copies of the registry module share the same registered controller, which directly covers the split-chunk/split-module failure mode described in the issue.
  - file: [extensions/whatsapp/src/connection-controller-registry.test.ts:11](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/extensions/whatsapp/src/connection-controller-registry.test.ts#L11)
  - command: `nl -ba extensions/whatsapp/src/connection-controller-registry.test.ts | sed -n '1,40p'`
