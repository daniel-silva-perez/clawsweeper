## Summary

Keep open. Current `main` still drops inbound Slack files when the event payload lacks `url_private_download`/`url_private`, which matches the reported DM failure mode. The fix exists only in an open PR and is not implemented on `main`.

## Evidence

- **Inbound media path still hard-fails without download URLs:** `resolveSlackMedia` only reads `file.url_private_download ?? file.url_private` and immediately returns `null` when neither is present; there is no `files.info` fallback in the inbound path on current `main`. That leaves DM events with partial file metadata unhydrated.
  - file: [extensions/slack/src/monitor/media.ts:242](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/extensions/slack/src/monitor/media.ts#L242)
  - command: `nl -ba extensions/slack/src/monitor/media.ts | sed -n '226,276p'`
  - sha: [45146913007d](https://github.com/openclaw/openclaw/commit/45146913007da723b4f06aba6a66cf4a695974c0)
- **Message preparation does not pass a Slack client into inbound media resolution:** `resolveSlackMessageContent` calls `resolveSlackMedia` and `resolveSlackAttachmentContent` with only `files`, `token`, and `maxBytes`. Without a client parameter, the inbound DM pipeline cannot call `files.info` even though that is the known recovery path.
  - file: [extensions/slack/src/monitor/message-handler/prepare-content.ts:90](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/extensions/slack/src/monitor/message-handler/prepare-content.ts#L90)
  - command: `nl -ba extensions/slack/src/monitor/message-handler/prepare-content.ts | sed -n '70,115p'`
  - sha: [45146913007d](https://github.com/openclaw/openclaw/commit/45146913007da723b4f06aba6a66cf4a695974c0)
- **Current tests encode placeholder-only fallback for missing Slack file URLs:** The Slack prepare test explicitly treats files without `url_private` as a download failure and asserts a placeholder body instead of staged inbound media. That is consistent with the reported bug not being fully fixed.
  - file: [extensions/slack/src/monitor/message-handler/prepare.test.ts:360](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/extensions/slack/src/monitor/message-handler/prepare.test.ts#L360)
  - command: `nl -ba extensions/slack/src/monitor/message-handler/prepare.test.ts | sed -n '356,380p'`
  - sha: [45146913007d](https://github.com/openclaw/openclaw/commit/45146913007da723b4f06aba6a66cf4a695974c0)
- **A `files.info` recovery path already exists elsewhere, but only for explicit download actions:** `downloadSlackFile` fetches fresh metadata via `client.files.info({ file: fileId })` before downloading. That supports the issue's root-cause claim that the inbound message path is missing a fallback already used in another Slack file flow.
  - file: [extensions/slack/src/actions.ts:448](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/extensions/slack/src/actions.ts#L448)

- **The proposed fix is still an open PR, not merged into main:** PR #50200 (`fix(slack): fetch fresh download URL via files.info for DM file attachments`) is open, unmerged, and its diff adds the exact missing `client.files.info` fallback plus tests. Its existence and unmerged state are strong evidence the issue is not yet implemented on `main`.

## Risks / Open Questions

- Slack DM file events that omit private download URLs will still fail to stage inbound media on current `main`.
- Thread-starter/media hydration uses the same inbound resolver pattern, so related Slack thread file context can remain incomplete.
