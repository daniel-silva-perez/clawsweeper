## Summary

Keep open. Current main still appears to route `openclaw gateway start` for an installed service through the restart path, and Linux restart still maps to a systemd restart. The issue is also recent and has a follow-up comment linking #62086 with a similar SIGTERM pattern outside the original `~/.profile` autostart theory, so there is not enough evidence for a conservative close.

## Evidence

- **`gateway start` still delegates to restart for installed services:** `startGatewayService` reads service state and, unless the service is missing, calls `service.restart({ ...args, env: state.env })`; the Linux service registry maps `restart` to `restartSystemdService`.
  - file: [src/daemon/service.ts:96](https://github.com/openclaw/openclaw/blob/d6ef1fcf24eea3a5985d653a3baf400e32aa1b75/src/daemon/service.ts#L96)
  - sha: [d6ef1fcf24ee](https://github.com/openclaw/openclaw/commit/d6ef1fcf24eea3a5985d653a3baf400e32aa1b75)
- **Linux restart path still uses systemd restart:** `restartSystemdService` delegates to `runSystemdServiceAction` with action `restart`, and that helper executes `systemctl ... restart <unit>`. That remains consistent with the reported SIGTERM/restart mechanism rather than proving it fixed.
  - file: [src/daemon/systemd.ts](https://github.com/openclaw/openclaw/blob/d6ef1fcf24eea3a5985d653a3baf400e32aa1b75/src/daemon/systemd.ts)
  - sha: [d6ef1fcf24ee](https://github.com/openclaw/openclaw/commit/d6ef1fcf24eea3a5985d653a3baf400e32aa1b75)
- **Issue discussion contains a broader unresolved report:** The 2026-04-16 comment reports the same exec SIGTERM pattern on Debian 12 with no `~/.profile` or `~/.bashrc` OpenClaw commands and links related issue #62086, so the discussion is still actionable and not limited to the original autostart hypothesis.
