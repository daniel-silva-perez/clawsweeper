## Summary

Keep open. Current main preserves explicit empty plugin-scope semantics, but it does not implement the PR's requested O(1) fast path: `loadOpenClawPlugins({ onlyPluginIds: [] })` still reaches cache lookup, load reentry tracking, registry construction, plugin discovery, and manifest registry loading before returning an empty registry. The PR is recent, contributor-authored, unprotected by maintainer labels, and still contains plausible useful perf work, though it should be rebased/adapted to the current `PluginLoaderCacheState` reentry path and covered by a test that proves discovery/manifest loading are skipped.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is to rebase the fast-path onto current `loadOpenClawPlugins`, place it after the current reentry guard or preserve equivalent `PluginLoaderCacheState.beginLoad`/`finishLoad` behavior, mirror the activating-load state-clear/activation side effects, and add a targeted loader test that proves explicit empty scopes avoid discovery and manifest registry loading while `undefined` still performs a full load.

## Risks / Open Questions

- The PR patch shown in the provided context was written against an older loader shape that referenced an in-flight-load set directly; current main uses `PluginLoaderCacheState.beginLoad`, so a direct merge may need rebase work.
- The PR should add a focused regression test proving `onlyPluginIds: []` skips discovery and manifest loading, not only that the returned registry is empty.
- The PR rationale should be tightened: current CLI command planning does not pass `onlyPluginIds: []` for no command match, but runtime/configured-channel and metadata/provider paths still preserve explicit empty scopes.
