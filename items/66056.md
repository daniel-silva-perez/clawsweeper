## Summary

Keep open. Current main still hardcodes the user-facing failure copy in the runner and sanitizer paths, and the `messages` config contract has no locale/errorLocale field. The existing i18n support is Control UI-scoped, while the referenced pairing locale PR #66055 was closed unmerged and current pairing copy remains English-only.

## Best Possible Solution

Keep this issue open for maintainer product/design work. The likely implementation path is a small core user-facing error catalog or structured error-key seam, plus `messages.locale`/`messages.errorLocale` types, zod schema, generated config docs/help, English fallback behavior, and focused tests around `sanitizeUserFacingText` and `agent-runner-execution`. If maintainers prefer this as plugin work, core should first expose structured error keys and locale context to hooks; relying on a ClawHub plugin to regex current English strings would be brittle and incomplete.
