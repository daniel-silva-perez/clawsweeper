## Summary

Keep this issue open. Current main still hardcodes user-facing runner and sanitizer failure text in English, the `messages` config contract has no locale/errorLocale field, Control UI i18n is browser/UI-scoped, and the referenced pairing-locale PR was closed unmerged. A plugin could regex-rewrite outgoing text via `message_sending`, but current hooks do not expose structured error keys or locale context, so that is not a complete ClawHub-only answer.

## Best Possible Solution

Keep this issue open for maintainer product/design work. The likely path is a small core user-facing error catalog with English fallback, a documented `messages.locale` or `messages.errorLocale` config field wired through types, zod schema, generated config help/docs, and focused tests for `sanitizeUserFacingText` plus runner fallback paths. If maintainers want this delegated to plugins, core should first expose structured error keys and locale context rather than requiring plugins to match English strings.
