## Summary

Current `main` still preserves pre-restart queued work and still replays persisted outbound deliveries on startup without any runtime-generation marker or invalidation check, so the reported stale-reply-after-restart class is not implemented away.

## Evidence

- **Persisted outbound queue has no restart/runtime generation field:** `QueuedDelivery` records payload, routing, timestamps, retry state, session, and scopes, but there is no runtime-generation / restart-epoch field to distinguish pre-restart work from current-runtime work.
  - file: [src/infra/outbound/delivery-queue-storage.ts:36](https://github.com/openclaw/openclaw/blob/b453293349657dedd069d291fa5b078d910067b1/src/infra/outbound/delivery-queue-storage.ts#L36)
  - command: `nl -ba src/infra/outbound/delivery-queue-storage.ts | sed -n '13,60p'`
  - sha: [b45329334965](https://github.com/openclaw/openclaw/commit/b453293349657dedd069d291fa5b078d910067b1)
- **Startup recovery still re-delivers every pending queued entry:** `recoverPendingDeliveries()` loads all pending entries and replays them on startup when backoff allows; it does not compare any generation or restart token before calling `drainQueuedEntry()`.
  - file: [src/infra/outbound/delivery-queue-recovery.ts:344](https://github.com/openclaw/openclaw/blob/b453293349657dedd069d291fa5b078d910067b1/src/infra/outbound/delivery-queue-recovery.ts#L344)
  - command: `nl -ba src/infra/outbound/delivery-queue-recovery.ts | sed -n '340,410p'`
  - sha: [b45329334965](https://github.com/openclaw/openclaw/commit/b453293349657dedd069d291fa5b078d910067b1)
- **Gateway startup always kicks off outbound recovery:** Gateway runtime services call `recoverPendingDeliveries()` after startup, so persisted pre-restart outbound entries are still part of the normal boot path on current `main`.
  - file: [src/gateway/server-runtime-services.ts:55](https://github.com/openclaw/openclaw/blob/b453293349657dedd069d291fa5b078d910067b1/src/gateway/server-runtime-services.ts#L55)
  - command: `nl -ba src/gateway/server-runtime-services.ts | sed -n '55,68p'`
  - sha: [b45329334965](https://github.com/openclaw/openclaw/commit/b453293349657dedd069d291fa5b078d910067b1)
- **In-process restart intentionally preserves queued command-lane work:** `resetAllLanes()` explicitly says queued entries are intentionally preserved across SIGUSR1 restart and then drains those lanes after bumping generation, which is the opposite of invalidating stale queued reply work on restart.
  - file: [src/process/command-queue.ts:327](https://github.com/openclaw/openclaw/blob/b453293349657dedd069d291fa5b078d910067b1/src/process/command-queue.ts#L327)
  - command: `nl -ba src/process/command-queue.ts | sed -n '327,356p'`
  - sha: [b45329334965](https://github.com/openclaw/openclaw/commit/b453293349657dedd069d291fa5b078d910067b1)
- **The restart path actually runs that preserve-and-drain behavior:** The gateway run loop calls `resetAllLanes()` on each in-process restart iteration, and the test suite asserts that queued work runs after `resetAllLanes()` (`task2Ran === true`).
  - file: [src/cli/gateway-cli/run-loop.ts:258](https://github.com/openclaw/openclaw/blob/b453293349657dedd069d291fa5b078d910067b1/src/cli/gateway-cli/run-loop.ts#L258)
  - command: `nl -ba src/cli/gateway-cli/run-loop.ts | sed -n '257,266p'`
  - sha: [b45329334965](https://github.com/openclaw/openclaw/commit/b453293349657dedd069d291fa5b078d910067b1)

## Risks / Open Questions

- I did not run a live restart reproduction because `pnpm` is unavailable in this environment; this decision is based on direct inspection of current code and tests.
- The command-queue evidence is specific to OpenClaw's in-process SIGUSR1 restart path; a full cold process restart may differ, but the persisted outbound-delivery replay path still remains generation-blind.
