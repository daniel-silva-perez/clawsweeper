## Summary

Keep open. Current `main` still leaves plain `filename="..."` Content-Disposition values untouched in Feishu downloads, so the reported mojibake case remains plausible and is easy to reproduce against the current parsing logic.

## Evidence

- **Current parser still returns plain filename unchanged:** `decodeDispositionFileName()` only decodes RFC 5987 `filename*=` values. For plain `filename="..."` headers it returns `plainMatch?.[1]?.trim()` with no Latin-1/UTF-8 recovery.

  - command: `nl -ba extensions/feishu/src/media.ts | sed -n '127,139p'`
  - sha: [4013c658537e](https://github.com/openclaw/openclaw/commit/4013c658537ee2051c44186f197b0710e84c9049)
- **Synthetic repro on current logic still produces mojibake:** Running the current parser logic with a header built from `测试.pdf` encoded as UTF-8 bytes then decoded as Latin-1 returns `decodedByMain: "æµè¯.pdf"` instead of the original filename.
  - command: `node - <<'NODE'
function decodeDispositionFileName(value){
  const utf8Match = value.match(/filename\*=UTF-8''([^;]+)/i);
  if (utf8Match?.[1]) {
    try { return decodeURIComponent(utf8Match[1].trim().replace(/^\"(.*)\"$/, '$1')); }
    catch { return utf8Match[1].trim().replace(/^\"(.*)\"$/, '$1'); }
  }
  const plainMatch = value.match(/filename=\"?([^\";]+)\"?/i);
  return plainMatch?.[1]?.trim();
}
const original = '测试.pdf';
const mojibake = Buffer.from(original, 'utf8').toString('latin1');

console.log(JSON.stringify({original, mojibake, decodedByMain: decodeDispositionFileName(header)}));
