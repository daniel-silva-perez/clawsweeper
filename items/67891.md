## Summary

Current main still matches the reported failure shape: the bundled webhooks plugin entry uses `async register(...)`, awaits config resolution, and only then calls `api.registerHttpRoute(...)`. Because the issue specifically reports that async registration is ignored by the loader, this should not be auto-closed as implemented.

## Evidence

- **Current webhooks entry still has async register:** On current main, `extensions/webhooks/index.ts` defines `async register(api: OpenClawPluginApi)` and awaits `resolveWebhooksPluginConfig(...)` before route registration. That preserves the exact async-registration pattern described in #67891.
  - file: [extensions/webhooks/index.ts:16](https://github.com/openclaw/openclaw/blob/d6ef1fcf24eea3a5985d653a3baf400e32aa1b75/extensions/webhooks/index.ts#L16)
  - command: `web open https://github.com/openclaw/openclaw/blob/main/extensions/webhooks/index.ts`
  - sha: [d6ef1fcf24ee](https://github.com/openclaw/openclaw/commit/d6ef1fcf24eea3a5985d653a3baf400e32aa1b75)
- **Route registration depends on async work completing:** The same entry registers HTTP routes inside the async function after building `targetsByPath`; if the loader ignores the returned Promise, these `api.registerHttpRoute(...)` calls can be skipped, matching the reported 404 behavior.
  - file: [extensions/webhooks/index.ts:38](https://github.com/openclaw/openclaw/blob/d6ef1fcf24eea3a5985d653a3baf400e32aa1b75/extensions/webhooks/index.ts#L38)
