## Summary

Keep open. Current `main` exposes some reusable memory host helpers, but it does not provide the requested plugin-facing file-scan/watch lifecycle API. The active memory runtime contract still only covers search-manager access and backend config, while the full watch/sync/delete pipeline remains private inside `memory-core`.

## Evidence

- **Memory runtime contract is narrow:** The registered memory-plugin runtime only exposes `getMemorySearchManager(...)`, `resolveMemoryBackendConfig(...)`, and optional `closeAllMemorySearchManagers()`. There are no file-scan, file-change, or indexing-event hooks in the current public contract.
  - file: [src/plugins/memory-state.ts:96](https://github.com/openclaw/openclaw/blob/b453293349657dedd069d291fa5b078d910067b1/src/plugins/memory-state.ts#L96)
  - command: `nl -ba src/plugins/memory-state.ts | sed -n '88,118p'`
  - sha: [b45329334965](https://github.com/openclaw/openclaw/commit/b453293349657dedd069d291fa5b078d910067b1)
- **Watcher remains internal to memory-core:** `memory-core` still owns the chokidar watcher setup privately in `ensureWatcher()`, including `extraPaths` handling and add/change/unlink wiring. This is not exposed through the memory-plugin runtime API.
  - file: [extensions/memory-core/src/memory/manager-sync-ops.ts:367](https://github.com/openclaw/openclaw/blob/b453293349657dedd069d291fa5b078d910067b1/extensions/memory-core/src/memory/manager-sync-ops.ts#L367)

- **Full sync pipeline remains internal to memory-core:** `syncMemoryFiles()` in `memory-core` still performs the scan, builds file entries, hashes content, skips unchanged files, and manages stale-path cleanup against the DB. That pipeline is implemented inside the bundled plugin rather than as a plugin-facing lifecycle surface.
  - file: [extensions/memory-core/src/memory/manager-sync-ops.ts:661](https://github.com/openclaw/openclaw/blob/b453293349657dedd069d291fa5b078d910067b1/extensions/memory-core/src/memory/manager-sync-ops.ts#L661)

- **Public SDK shares helpers, not the full pipeline:** The public memory host storage surface exports lower-level helpers such as `listMemoryFiles`, `buildFileEntry`, `chunkMarkdown`, and hashing utilities. That reduces reimplementation, but it is still helper-level access rather than the requested unified scan/watch plugin API.

## Risks / Open Questions

- Some of the underlying building blocks are now public, so maintainers could decide the issue is partially addressed even though the proposed hook-style API is not present.
- This request may overlap with broader memory-plugin architecture work, so the eventual fix might come via a different API shape than the one proposed in the issue.
