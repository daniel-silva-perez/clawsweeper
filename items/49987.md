## Summary

Keep #49987 open. Current main still does not implement a CLI `openclaw gateway restart` restart-sentinel write, while the existing restart-sentinel machinery is present for chat `/restart` and the gateway tool. The PR addresses a real gap, but its current v2 approach of writing the sentinel after `runServiceRestart` returns is likely too late: gateway startup checks for the sentinel during post-attach startup, before the CLI restart call has returned. The existing review discussion also raised the separate daemon-environment/state-dir concern, so this needs maintainer review and a revised implementation rather than cleanup closure.

## Best Possible Solution

Keep the PR open but request changes. The robust implementation should write the CLI restart sentinel after restart preflight/no-op checks prove a restart will proceed, but before the restarted gateway reaches `server-startup-post-attach` sentinel detection. It should use the daemon service environment when resolving the sentinel path, clean up on restart failure/abort, and add focused tests in `src/cli/daemon-cli/lifecycle.test.ts` for successful managed restart, no-op/not-loaded restart, failed restart cleanup, and custom service state-dir handling.

## Risks / Open Questions

- Closing would drop a still-missing CLI restart notification path that is distinct from the already-implemented chat `/restart` and gateway-tool sentinel paths.
- Merging the PR as currently described could write the sentinel after the restarted gateway has already passed its startup sentinel check, leaving a delayed/stale notification for a later start.
- The implementation also needs to resolve the daemon environment/state-dir issue raised in review so managed services using a persisted `OPENCLAW_STATE_DIR` can consume the sentinel.
