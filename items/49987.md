## Summary

Keep open. Current `main` does not implement CLI restart-sentinel notification for `openclaw gateway restart`, and the PR as proposed still appears incomplete because it writes the sentinel with default `process.env` instead of the daemon service environment.

## Evidence

- **Current main lacks the requested behavior:** `runDaemonRestart` still returns `runServiceRestart(...)` directly and does not import or call any restart-sentinel helpers, so CLI restarts do not persist a post-restart notice trigger.
  - file: [src/cli/daemon-cli/lifecycle.ts:186](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/src/cli/daemon-cli/lifecycle.ts#L186)
  - command: `nl -ba src/cli/daemon-cli/lifecycle.ts | sed -n '170,240p'`
  - sha: [45146913007d](https://github.com/openclaw/openclaw/commit/45146913007da723b4f06aba6a66cf4a695974c0)
- **Gateway notices depend on a persisted sentinel file:** Gateway startup only emits the restart notice by consuming the restart sentinel on boot, so without a sentinel written by the CLI restart path there is nothing to notify from.
  - file: [src/gateway/server-restart-sentinel.ts:16](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/src/gateway/server-restart-sentinel.ts#L16)
  - command: `nl -ba src/gateway/server-restart-sentinel.ts | sed -n '1,120p'`
  - sha: [45146913007d](https://github.com/openclaw/openclaw/commit/45146913007da723b4f06aba6a66cf4a695974c0)
- **Sentinel path is environment-dependent:** `writeRestartSentinel` resolves its file path from the passed env and defaults to `process.env`. The PR patch in the provided context calls `writeRestartSentinel(sentinelPayload)` with no daemon env override, so it would write into the CLI state/config context, not necessarily the managed gateway's context.
  - file: [src/infra/restart-sentinel.ts:76](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/src/infra/restart-sentinel.ts#L76)
  - command: `nl -ba src/infra/restart-sentinel.ts | sed -n '1,220p'`
  - sha: [45146913007d](https://github.com/openclaw/openclaw/commit/45146913007da723b4f06aba6a66cf4a695974c0)
- **Daemon-aware code already handles env divergence elsewhere:** Status gathering explicitly merges `service.readCommand(...).environment` into a daemon env before resolving config/state paths, which is the same class of divergence this PR needs to respect for sentinel placement.
  - file: [src/cli/daemon-cli/status.gather.ts:196](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/src/cli/daemon-cli/status.gather.ts#L196)
  - command: `nl -ba src/cli/daemon-cli/status.gather.ts | sed -n '180,220p'`
  - sha: [45146913007d](https://github.com/openclaw/openclaw/commit/45146913007da723b4f06aba6a66cf4a695974c0)
- **Existing restart-sentinel support exists for non-CLI restart flows:** Other restart flows already write restart sentinels before restart orchestration, confirming the feature gap is specific to the CLI daemon restart path rather than already solved globally.
  - file: [src/agents/tools/gateway-tool.ts:400](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/src/agents/tools/gateway-tool.ts#L400)

## Risks / Open Questions

- The PR is not implemented on `main`, so closing as `implemented_on_main` would be incorrect.
- The proposed patch from the PR context appears to miss daemon-environment routing for `writeRestartSentinel`, which can break custom `OPENCLAW_STATE_DIR`/profile installs by writing the sentinel where the restarted gateway will not read it.
- Current `src/cli/daemon-cli/lifecycle.test.ts` has no restart-sentinel coverage, so even after fixing env routing this change still needs regression tests for success and no-op restart cases.
