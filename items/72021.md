## Summary

Keep #72021 open. Current main still implements the behavior reported: promotion ranking and application use a combined signal count for the `minRecallCount` gate, while daily/session ingestion records non-recall signals into that same store. Existing tests and docs also encode daily-only promotion as expected behavior, so this is not already fixed; it needs maintainer product/code judgment rather than cleanup closure.

## Best Possible Solution

Keep the issue open for a focused memory-core fix or maintainer product decision. The likely fix is to separate true recall count from aggregate signal count in rank/apply/explain semantics: either make `minRecallCount` gate only on `recallCount` and retune scoring for recall-only candidates, or explicitly rename/split the threshold into `minSignalCount` versus `minRecallCount`, then update tests and docs to match the chosen contract.
