## Summary

Keep open. Current main still appears to reject the reported `source` metadata on `exec.approvals.set`: the gateway protocol schema allowlist entry does not accept `source`, while the runtime still writes and reads `source: "allow-always"` allowlist metadata. Related PRs/comments are useful evidence, but the issue is not clearly implemented on main.

## Evidence

- **Gateway schema still rejects extra allowlist properties:** `ExecApprovalsAllowlistEntrySchema` includes `id`, `pattern`, `argPattern`, `lastUsedAt`, `lastUsedCommand`, and `lastResolvedPath`, with `additionalProperties: false`; it does not include `source`.
  - file: [src/gateway/protocol/schema/exec-approvals.ts:4](https://github.com/openclaw/openclaw/blob/d6ef1fcf24eea3a5985d653a3baf400e32aa1b75/src/gateway/protocol/schema/exec-approvals.ts#L4)
  - command: `web inspection of GitHub main blob after local read commands failed under sandbox`
  - sha: [d6ef1fcf24ee](https://github.com/openclaw/openclaw/commit/d6ef1fcf24eea3a5985d653a3baf400e32aa1b75)
- **Runtime still uses durable approval source metadata:** Runtime allowlist entries include `source?: "allow-always"`; `addAllowlistEntry`, `addDurableCommandApproval`, and `persistAllowAlwaysPatterns` persist `source: "allow-always"` into allowlist entries.
  - file: [src/infra/exec-approvals.ts:123](https://github.com/openclaw/openclaw/blob/d6ef1fcf24eea3a5985d653a3baf400e32aa1b75/src/infra/exec-approvals.ts#L123)
  - command: `web inspection of GitHub main blob after local read commands failed under sandbox`
  - sha: [d6ef1fcf24ee](https://github.com/openclaw/openclaw/commit/d6ef1fcf24eea3a5985d653a3baf400e32aa1b75)
- **Discussion identifies same mismatch:** Issue comments identify the schema/runtime mismatch and recommend adding `source` to the gateway schema; timeline shows related PR attempts (#60064, #60073, #60099, #60371), but provided context does not show a landed fix, and current main still lacks the schema field.
