## Summary

Keep open. Current `main` does not implement this PR’s timeout behavior or another equivalent fix for a delayed `chat.history` response. `loadChatHistory` still awaits `state.client.request("chat.history")` directly and only retries explicit retryable `UNAVAILABLE` errors; a response that is merely delayed can still leave `chatLoading` true and the chat view stuck on its loading skeleton indefinitely.

## Evidence

- **`loadChatHistory` still has no request timeout:** `loadChatHistory` sets `state.chatLoading = true` and awaits `state.client.request("chat.history", ...)` directly. The only recovery path is retrying caught retryable `UNAVAILABLE` errors for up to 60s; there is no `Promise.race`, abort, or other timeout for a request that never resolves or resolves very late.
  - file: [ui/src/ui/controllers/chat.ts:139](https://github.com/openclaw/openclaw/blob/4630ce3d9e2f089062ccd386b0ae241325961d1a/ui/src/ui/controllers/chat.ts#L139)
  - command: `nl -ba ui/src/ui/controllers/chat.ts | sed -n '139,201p'`
  - sha: [4630ce3d9e2f](https://github.com/openclaw/openclaw/commit/4630ce3d9e2f089062ccd386b0ae241325961d1a)
- **Gateway browser RPCs remain pending until a response frame or disconnect:** `GatewayBrowserClient.request()` only stores the resolver in `this.pending`, sends the frame, and returns the promise. Pending entries are removed in `handleMessage()` only when a matching `res` frame arrives. There is no per-request timeout or cancellation path here.
