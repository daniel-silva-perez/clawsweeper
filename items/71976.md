## Summary

Keep open. Current main still matches the reported behavior: light sleep sorts by recency before recall count, deep promotion gates `minRecallCount` with total signals including daily and grounded counts, and rehydration still uses the narrow `preferredSpan + 3` / `8` search span. Existing tests also explicitly assert that repeated daily-only dreaming signals can clear the default promotion gates, so this needs maintainer/product triage rather than conservative cleanup closure.

## Best Possible Solution

Keep the issue open for maintainer triage. The best path is to add focused regression tests for light-sleep ordering, recall-only versus all-signal promotion gates, scoring/average-score semantics, and wide-range rehydration, then either change the implementation to prioritize true recalls and widen relocation or rename/document the gate if daily and grounded signals are intentionally allowed to satisfy it.
