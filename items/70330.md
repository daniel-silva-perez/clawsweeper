## Summary

Keep #70330 open. Current main still has the core behavior that makes this report actionable: WebChat's New Session path is still a generic `/new` chat send, Control UI connects with admin scope so reset slash commands are authorized, `chat.send` has no explicit session-control/audit flag, and session listing/checkpoint UI is still based on the active session-store row rather than archived reset transcripts. The incident timing also overlaps the documented/default 4:00 AM daily reset behavior, so the best fix should likely distinguish intended daily/idle rotation from accidental replay/reconnect rotation and make every rotation visible and recoverable.

## Best Possible Solution

Keep this open and implement an explicit, auditable WebChat session-rotation model: use a dedicated UI/RPC path for New Session/reset, persist or broadcast a rotation event with trigger/oldSessionId/newSessionId/sessionKey/reason, reject or gate generic chat.send reset control where appropriate, and expose reset archives/checkpoints so users can reopen or restore prior WebChat sessions. Add regression coverage for gateway reconnect around the daily reset boundary and for normal post-reconnect chat preserving or visibly rotating the expected session.
