## Summary

Current main already rebuilds stale `skillsSnapshot` entries for existing sessions when the snapshot version changes, and the skills watcher bumps that version on `SKILL.md` add/change/unlink events. The stale-path described in the issue (`isNewSession || !sessionEntry?.skillsSnapshot`) is no longer the live code.

## Risks / Open Questions

- There is still no dedicated manual `sessions refresh-skills` CLI path in the inspected tree; that appears to be separate feature work, not the stale-snapshot bug itself.
- The auto-refresh path depends on version invalidation from watched skill roots or other explicit bump sources, so edge cases outside those paths may still need separate follow-up.

## Close Comment

Closing from a Codex review: current `main` already implements the automatic refresh path this bug was asking for. The old behavior described in the report and comments (`isNewSession || !sessionEntry?.skillsSnapshot`) is no longer the live code. Existing sessions now rebuild and persist `skillsSnapshot` when the cached version is stale in `src/agents/agent-command.ts:573` and `src/auto-reply/reply/session-updates.ts:145`, and the watcher in `src/agents/skills/refresh.ts:81` bumps the snapshot version on `SKILL.md` add/change/unlink events across the managed skill roots. The docs also now describe mid-session refresh on the next turn in `docs/tools/skills.md:329`. I also confirmed this implementation is present in the shipped `v2026.4.22` tree. The separate idea of a manual refresh command looks like follow-up feature work, but the stale-session bug itself is implemented on main.
