## Summary

Keep PR #50737 open. Current main has generic `authHeader` support, but it does not fully solve this PR’s Anthropic-specific problem: onboarding Anthropic verification still always uses `x-api-key`, and the runtime path still lets the Anthropic SDK construct API-key auth while merely adding `Authorization` as a model header. The PR remains a plausible bug fix, though it should be rebased and reconciled with current main plus the review concerns before merging.

## Best Possible Solution

Keep this PR open as useful work. The best maintainer path is to rebase it onto current main, adapt it to the newer generic `applyAuthHeaderOverride`/provider request configuration code, add or retain focused tests for `api: "anthropic-messages"` plus `authHeader: true`, and ensure onboarding verification and saved custom-provider config stay consistent for Bearer-only Anthropic-compatible gateways such as MiniMax.
