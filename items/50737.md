## Summary

Keep open. Current `main` does not already implement this PR’s Anthropic `authHeader` behavior: custom Anthropic verification still hard-codes `x-api-key`, and the runtime Anthropic transport still derives `x-api-key` from `apiKey` even when `authHeader` has injected `Authorization`, so Bearer-only `anthropic-messages` providers are still not covered.

## Evidence

- **Onboarding probe still hard-codes x-api-key:** `buildAnthropicHeaders()` always sets `x-api-key` and has no `authHeader`/Bearer branch. `buildAnthropicVerificationProbeRequest()` uses it directly for Anthropic endpoint verification.
