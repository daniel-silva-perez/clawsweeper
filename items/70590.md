## Summary

Keep open. Current main has real Control UI i18n infrastructure and v2026.4.24 shipped broad localized UI support, but the reported residual hard-coded Web UI English is still present in user-facing settings/config/cron surfaces. The existing i18n check validates locale bundle drift and key structure; it does not prove all visible UI strings are routed through the translation map.

## Best Possible Solution

Keep this open as Control UI i18n follow-up work. The best path is to audit remaining user-visible Web UI literals, move them into ui/src/i18n/locales/en.ts, regenerate the locale bundles with the existing ui:i18n pipeline, and add either a source scan or documented allowlist so future hard-coded UI copy does not silently bypass translations.
