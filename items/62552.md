## Summary

Keep open. Current `main` still lacks the ACP bridge-key changes described by PR #62552, and the bundled cron maintenance change has a credible unresolved regression risk for active isolated cron jobs.

## Evidence

- **ACP fallback key change is not on main:** `AcpGatewayAgent.newSession()` still falls back to `acp:${sessionId}`, and `resolveSessionKey()` returns that fallback unchanged when no session metadata is provided. That means the PR's `acp-bridge:<uuid>` stabilization is not implemented on current `main`.
  - file: [src/acp/translator.ts:526](https://github.com/openclaw/openclaw/blob/ecac696643c604c6d854341bfe6424bfa31423e9/src/acp/translator.ts#L526)
  - command: `nl -ba src/acp/translator.ts | sed -n '526,535p'; nl -ba src/acp/session-mapper.ts | sed -n '27,85p'`
  - sha: [ecac696643c6](https://github.com/openclaw/openclaw/commit/ecac696643c604c6d854341bfe6424bfa31423e9)
- **Canonical/raw session-key matching is still missing:** `findPendingBySessionKey()` still uses strict string equality, so a pending prompt keyed by a raw ACP fallback session will not match a Gateway-canonicalized key. The current stop-reason harness and tests only exercise `agent:main:main`; there is no coverage for `acp-bridge` or raw/canonical equivalence on `main`.
  - file: [src/acp/translator.ts:1046](https://github.com/openclaw/openclaw/blob/ecac696643c604c6d854341bfe6424bfa31423e9/src/acp/translator.ts#L1046)
  - command: `nl -ba src/acp/translator.ts | sed -n '1046,1055p'; nl -ba src/acp/translator.prompt-harness.test-support.ts | sed -n '15,24p'; nl -ba src/acp/translator.stop-reason.test.ts | sed -n '15,81p'`
  - sha: [ecac696643c6](https://github.com/openclaw/openclaw/commit/ecac696643c604c6d854341bfe6424bfa31423e9)
- **Current main avoids the PR's cron session-store gating:** `hasBackingSession()` still treats cron tasks as live solely from `isCronJobActive(jobId)` and does not consult the session store for cron child sessions. That matches the PR review concern: cron task records are created with `childSessionKey: job.sessionKey`, while isolated cron execution also creates a distinct run session key shaped like `agent:...:cron:<job>:run:<id>`. Gating active cron tasks on the stored child session entry could therefore mark a live job lost for the wrong session key.
  - file: [src/tasks/task-registry.maintenance.ts:124](https://github.com/openclaw/openclaw/blob/ecac696643c604c6d854341bfe6424bfa31423e9/src/tasks/task-registry.maintenance.ts#L124)

  - sha: [ecac696643c6](https://github.com/openclaw/openclaw/commit/ecac696643c604c6d854341bfe6424bfa31423e9)
- **Existing regression coverage does not resolve the cron blocker:** The current issue-60299 regression file only proves that an active cron task stays live when `childSessionKey` is absent. It does not cover the PR's riskier case of an active cron job with a terminal or mismatched child session entry.
