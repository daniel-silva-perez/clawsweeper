## Summary

Keep PR #49981 open. Current main moved this area into the memory-host SDK plus provider plugins, but it still does not thread the configured remote batch timeout into the guarded HTTP calls for upload, submit, poll, terminal-error read, or result download. The PR is stale against old paths and has review feedback to resolve, but the underlying fix is not implemented on main.

## Best Possible Solution

Keep the item open, but ask for a ported fix against current main: add optional timeout/deadline plumbing in packages/memory-host-sdk/src/host/*, pass remaining budget through extensions/openai, extensions/voyage, and extensions/google batch providers, avoid whole-batch resubmission on poll/download aborts, and add focused regression tests for timeout forwarding and deadline behavior.

## Risks / Open Questions

- The PR branch targets old src/memory paths; a direct merge is unlikely to be the right shape after the current memory-host SDK/provider-plugin split.
- Naively adding timeoutMs only to each fetch can still overshoot the documented full batch budget unless one deadline is carried through upload, submit, poll, error-read, retry, and result-download phases.
- Retrying the whole provider batch on an AbortError after remote submission can duplicate remote batch jobs and cost.
