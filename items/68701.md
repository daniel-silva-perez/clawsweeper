## Summary

Keep this PR open. Current main still preserves `to` unconditionally for `channel: "webchat"`, so the proposed problem is not already implemented away; however, the PR head is not yet the best fix because its structured-prefix exception would also preserve foreign structured targets such as `telegram:...`.

## Best Possible Solution

Review and revise the PR rather than close it: make `normalizeDeliveryContext` drop unstructured and foreign-prefixed `to` values when the normalized channel is `webchat`, while preserving only canonical WebChat-owned targets. Add regression coverage for phone-shaped `to`, a foreign structured target such as `telegram:...`, and a valid `webchat:user-...` target, then verify `normalizeSessionDeliveryFields`, `extractDeliveryInfo`, and restart-sentinel fallback behavior.
