## Summary

Keep open. Current `main` still sends TUI `/new` through a local session switch instead of `resetSession(..., "new")`, so the hook-visible `command:new`/session lifecycle path is still bypassed.

## Evidence

- **TUI /new still uses local setSession:** `/new` generates `tui-<uuid>` and calls `setSession(uniqueKey)`, while `/reset` calls `client.resetSession(...)`. That matches the reported regression rather than a fix.
  - file: [src/tui/tui-command-handlers.ts:523](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/src/tui/tui-command-handlers.ts#L523)
  - command: `nl -ba src/tui/tui-command-handlers.ts | sed -n '520,555p'`
  - sha: [45146913007d](https://github.com/openclaw/openclaw/commit/45146913007da723b4f06aba6a66cf4a695974c0)
- **setSession is local-only state/history switching:** `setSession` updates `state.currentSessionKey`, clears local UI state, and reloads history; it does not call the gateway reset path or emit any hook event.
  - file: [src/tui/tui-session-actions.ts:368](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/src/tui/tui-session-actions.ts#L368)
