## Summary

Keep PR #66001 open. Current main still has the no-op `ProcessSupervisor.reconcileOrphans()` behavior that the PR targets, and the linked issue #65983 remains open. The PR is a plausible focused fix, but the existing P1 review is substantiated by current code: supervisor records use `opts.sessionKey?.trim() || sessionId` while the bash process registry is keyed by the generated session slug, so the PR’s proposed `hasSession(record.sessionId)` check would cancel legitimate session-key-owned runs.

## Best Possible Solution

Keep #66001 open and request changes rather than closing it. The best path is a session-key-aware reconciliation design that either checks ownership against both process session slug and agent session key or stores the process-session id separately in the supervisor record, plus regression tests proving sessionKey-owned runs are preserved and truly untracked runs are cancelled. Also narrow the reconciliation cancellation reason type before merge. Leave #65983 open for the broader durable restart/orphan-reaping work unless this PR is expanded to cover it.
