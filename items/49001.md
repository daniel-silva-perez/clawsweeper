## Summary

Keep open. Current main still lacks the durable post-reset startup-window mechanism proposed by #49001, and the streamed queued-message path still steers the raw queued prompt without Discord thread startup recovery. Related #53524 is useful context and broader follow-up work, but it is also open and unmerged, so it is not a shipped implementation and is not strong enough to auto-close this PR as obsolete.

## Best Possible Solution

Keep this PR open for maintainer judgment, or explicitly consolidate it into #53524 if maintainers choose the broader channel-and-thread implementation as canonical. The product fix should add a durable post-reset startup marker, wrap the first queued owner message with startup-recovery instructions, clear the marker only after the recovery path has actually completed, and keep focused regression coverage for Discord thread/channel reset races.
