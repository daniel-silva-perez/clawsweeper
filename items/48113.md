## Summary

Current main already implements the reported fix: authenticated loopback gateway probes now attach device identity so read-scope RPCs are no longer downgraded to shared-token empty scopes, and the same device-bound behavior is present for local shared-auth gateway calls used by status-style RPCs.

## Close Comment

Closing as implemented. Codex review of current `main` found that authenticated loopback probes now attach the local device identity before doing read/detail RPCs, specifically to avoid the shared-auth scope stripping that produced `missing scope: operator.read` (`src/gateway/probe.ts:148`). The probe coverage explicitly checks this path (`src/gateway/probe.test.ts:129` and `src/gateway/probe.test.ts:183`), while the server auth suite still shows the old failure mode only for device-less shared-token auth (`src/gateway/server.auth.default-token.suite.ts:176` and `src/gateway/server.auth.default-token.suite.ts:244`). The local shared-auth gateway call path used by status-style RPCs is also device-bound now (`src/gateway/call.ts:196` and `src/gateway/call.ts:515`). This logic is already present in the shipped `v2026.4.22` release as well.
