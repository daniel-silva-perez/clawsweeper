## Summary

Keep open. Current main has real gateway-enforced tool restrictions and optional call-time hook blocking, but it does not implement the feature as proposed: enforcement is still tool-name/profile based, not manifest-declared capability grants with default-deny high-risk classes, grant/revoke UX, and immutable per-capability audit logging.

## Evidence

- **Gateway tool policy exists, but it is coarse allow/deny by tool name:** The current guidance documents per-agent Gateway enforcement as `tools.allow` / `tools.deny` over concrete tool ids like `read`, `exec`, `message`, and `cron`, not capability-scoped permissions such as `fs.read:<scope>` or `net.fetch:<allowlist>`.

That is narrower than the issue's proposed capability manifest, default-deny grants, and per-capability authorization model.

## Risks / Open Questions

- Closing as implemented would overstate the current state of main: there is no evidence of manifest-declared required capabilities, default-deny grants/revocations, or immutable high-risk capability audit logs.
- Closing as clawhub would be misleading because the request targets core gateway/tool-layer authorization, and current main already treats tool policy as a core enforcement surface.
