## Summary

PR #71185 should remain open. Current main still has the unsafe trajectory support-capture consumer path: `buildSkillsCapture` trusts every `resolvedSkills` entry, calls `redactPathForSupport` on `skill.filePath`/`skill.baseDir`, and sorts with `left.name.localeCompare(right.name)`. `redactPathForSupport` still requires a string and immediately calls `startsWith`, so a persisted or producer-leaked partial skill entry can still crash metadata capture. The PR is recent, contributor-authored, narrowly scoped, and its discussion includes a concrete reproduction plus bot/Codex review comments finding no major issue; it is not protected, not maintainer-authored, and the provided related items are only title-overlap reports about different skill behavior.

## Best Possible Solution

Keep PR #71185 open for maintainer review or merge after normal CI. The best product/code direction is to preserve the symlink-escape rejection behavior while making trajectory support capture tolerant of malformed or legacy `skillsSnapshot.resolvedSkills` data, with the PR’s consumer-side regression tests retained; a follow-up can tighten the producer invariant if maintainers want the snapshot source cleaned up too.
