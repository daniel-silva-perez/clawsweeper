## Summary

Keep open. Current main already has the HTTP `/tools/invoke` surface and the underlying `sessions_spawn` tool, but the reporter narrowed the request to a direct CLI path, and current main still has no `openclaw sessions spawn` or `openclaw tools invoke` command.

## Best Possible Solution

Keep this issue open and implement a core CLI surface backed by the existing Gateway `/tools/invoke`/tool contract. A conservative path is a generic `openclaw tools invoke --tool ... --args ... --json` plus, if maintainers want discoverability, a focused `openclaw sessions spawn` wrapper that maps flags to `sessions_spawn`, uses existing gateway auth/config resolution, preserves all tool-policy gates, and adds CLI tests plus docs under `docs/cli/sessions` or a new tool-invoke CLI page.

## Risks / Open Questions

- A direct spawn CLI would expose a high-risk session orchestration surface; it should preserve the existing Gateway auth, `gateway.tools.allow` default-deny behavior, agent tool policy, max child/depth limits, allowAgents, and sandbox inheritance checks.
- A generic `openclaw tools invoke` command could unintentionally make dangerous tools easier to call; errors and docs should make the default deny list and required opt-in explicit.
