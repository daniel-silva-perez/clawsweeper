## Summary

Keep PR #71651 open. Current main still lacks the PR's reload invalidation behavior: timer ticks force-reload cron stores with `skipRecompute`, `loadCronStore` merges persisted `jobs-state.json` runtime state by job id, and `ensureLoaded` does not compare the newly loaded schedule/enabled inputs against the previous in-memory job before due-job collection. The PR remains a focused fix, and maintainer comments show the initial review concerns were addressed with a semantic schedule comparator, broader regression tests, docs, changelog, and a passing changed gate on the PR branch.

## Best Possible Solution

Keep the PR open for maintainer review/landing, or land an equivalent patch on main: compare reloaded jobs by id after `loadCronStore` merges runtime state, semantically compare the full schedule union plus `enabled`, clear only stale `state.nextRunAtMs` when scheduling inputs actually change, and keep the PR's regression coverage for changed schedules, unchanged/key-reordered schedules, `at`, `every`, and enabled changes plus the docs/changelog updates.
