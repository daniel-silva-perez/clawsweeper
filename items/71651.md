## Summary

Keep PR #71651 open. Current main still reloads cron stores on timer ticks with `forceReload: true` and `skipRecompute: true`, then preserves existing future `state.nextRunAtMs` values without comparing the newly loaded schedule against the previous in-memory job. The PR remains a focused fix for #71607, and the provided discussion shows maintainer follow-up addressed the review concerns with semantic schedule comparison, additional regression tests, docs, and a changelog entry.

## Best Possible Solution

Keep #71651 open and route it through normal maintainer merge/rebase review. The best path is to land the PR's semantic schedule-change invalidation in `src/cron/service/store.ts` with its regression tests and docs, preserving due slots for formatting-only rewrites while clearing stale pending slots when schedule fields or enabled state actually change.
