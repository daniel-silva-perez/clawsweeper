## Summary

Keep this issue open. Current main still allows the reported installer outcome: the documented curl installer can warn that the npm global bin directory is missing from the user's original PATH, then continue to print a successful install message, while post-install verification is opt-in. The existing non-root installer smoke manually adds the npm global bin directory to PATH after the installer runs before verifying the CLI, so it does not cover the user's `openclaw: command not found` path.

## Best Possible Solution

Fix the Unix installer path so default installer success is contingent on `openclaw` being discoverable for the user's next command, or stop printing an unconditional success banner when the original shell PATH is missing the npm global bin directory. Add focused installer coverage that verifies npm global prefix/PATH behavior without a post-test PATH export, and update the public install troubleshooting docs if the intended workflow requires opening a new shell or sourcing shell startup files.
