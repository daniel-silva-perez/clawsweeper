## Summary

Current main shows partial remediation for Telegram reply/react plumbing, but not enough to conservatively close #50872. The inbound Telegram `message_id` is now propagated into session/tool context and reply threading has solid tests, yet the exact reported `messageId:"current"` react flow is still unsupported in code, and I do not have live Telegram repro proof that the native reply-threading inconsistency is gone.

## Evidence

- **`messageId:"current"` is not normalized:** `resolveReactionMessageId` prefers an explicit `messageId` from tool args and only falls back to `toolContext.currentMessageId` when the arg is omitted. A literal string like `"current"` is therefore passed through unchanged rather than resolved.
  - file: [src/channels/plugins/actions/reaction-message-id.ts:11](https://github.com/openclaw/openclaw/blob/81666e586aa6d4305e6472d7075e62f63f701831/src/channels/plugins/actions/reaction-message-id.ts#L11)
  - command: `nl -ba src/channels/plugins/actions/reaction-message-id.ts | sed -n '1,40p'`
  - sha: [81666e586aa6](https://github.com/openclaw/openclaw/commit/81666e586aa6d4305e6472d7075e62f63f701831)
- **Telegram react still rejects non-numeric ids:** The Telegram react runtime requires a finite positive numeric `messageId`; otherwise it returns `missing_message_id`. That means the issue's exact repro shape with `messageId:"current"` would still fail on main, just with a softer error path.
  - file: [extensions/telegram/src/action-runtime.ts:253](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/extensions/telegram/src/action-runtime.ts#L253)
  - command: `nl -ba extensions/telegram/src/action-runtime.ts | sed -n '249,265p'`
  - sha: [ebd7f19a3b20](https://github.com/openclaw/openclaw/commit/ebd7f19a3b20001960474ec9ba9cc424e45f7b23)
- **Current inbound Telegram message ids are now propagated:** Telegram inbound context stores `MessageSid` from `msg.message_id`, and the reply/tool threading context carries that value forward as `currentMessageId`. This appears to fix the omission-based fallback path for reactions and reply threading, but not the explicit `"current"` alias.
  - file: [extensions/telegram/src/bot-message-context.session.ts:342](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/extensions/telegram/src/bot-message-context.session.ts#L342)
  - command: `nl -ba extensions/telegram/src/bot-message-context.session.ts | sed -n '320,350p'`

- **Threading and react fallback are covered by tests:** Tests now cover Telegram react using `toolContext.currentMessageId`, generic reply-threading from `currentMessageId`, and Telegram `replyToMode=all` sending native `reply_to_message_id` on all chunks. That is good evidence of progress, but not enough to prove the production inconsistency report is fully obsolete.

## Risks / Open Questions

- Main likely fixed the 'omit messageId and use current inbound message' path, so the issue may now be narrower than originally reported.
- The native Telegram reply-threading complaint is inherently live-behavior-sensitive; without a real Telegram repro on current main, closing would be guesswork.
- If maintainers want cleanup later, the remaining close blocker is narrow: decide whether `messageId:"current"` should be supported explicitly or whether docs/prompting should prevent models from emitting it.
