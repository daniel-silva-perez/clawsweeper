## Summary

Keep open: current `main` still does not normalize `profiles.*.api_key` in `auth-profiles.json`. The loader only aliases `apiKey` to `key`, runtime resolution only reads `cred.key`/`cred.keyRef`, and load-time validation does not reject `api_key` entries that omit `key`.

## Evidence

- **Loader normalizes `apiKey`, not `api_key`:** `normalizeRawCredentialEntry` maps `mode -> type` and `apiKey -> key`, but there is no matching branch for snake_case `api_key`.
  - file: [src/agents/auth-profiles/persisted.ts:50](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/src/agents/auth-profiles/persisted.ts#L50)
  - command: `nl -ba src/agents/auth-profiles/persisted.ts | sed -n '50,60p'`
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
- **Runtime resolution still only reads `key`/`keyRef`:** `resolveApiKeyForProfile` passes only `cred.key` and `cred.keyRef` into `resolveProfileSecretString`; a stored credential with only `api_key` will resolve to `null`.
  - file: [src/agents/auth-profiles/oauth.ts:291](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/src/agents/auth-profiles/oauth.ts#L291)
  - command: `nl -ba src/agents/auth-profiles/oauth.ts | sed -n '291,306p'`
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
- **Missing key is not rejected at load time:** `parseCredentialEntry` rejects only non-object, invalid type, and missing provider. An `api_key` profile with a provider but no normalized `key` still parses and is not warned about.
  - file: [src/agents/auth-profiles/persisted.ts:63](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/src/agents/auth-profiles/persisted.ts#L63)
  - command: `nl -ba src/agents/auth-profiles/persisted.ts | sed -n '63,85p'`
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
- **Secret resolution returns nothing when both value and ref are absent:** `resolveProfileSecretString` trims `params.value`, optionally resolves `params.valueRef`, and otherwise returns `undefined`; there is no warning for the missing-value case.
  - file: [src/agents/auth-profiles/oauth.ts:209](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/src/agents/auth-profiles/oauth.ts#L209)

- **Docs now describe `key`, but behavior gap remains:** Current docs show API-key profiles as `{ provider, key }` and mention `keyRef` for `api_key` credentials, which documents the expected field but does not address the silent-ignore runtime behavior in the issue.
