## Summary

Keep this PR open. Current main still lacks the proposed post-publish macOS asset verifier and appcast CODEOWNERS protection, while the release/appcast failure mode is real and related to open tracker #48306. ## Best Possible Solution

Keep the PR open for maintainer review. If maintainers want this guard, revise the workflow to distinguish retryable not-ready states from GitHub API failures, recheck before timeout, tighten tag validation, decide whether appcast absence should warn or fail, and align the /appcast.xml CODEOWNERS entry with release-owner policy while linking the work to #48306.

## Risks / Open Questions

- Release workflow and appcast ownership changes affect maintainer-controlled macOS publishing and should be reviewed by release owners, not auto-closed as cleanup.
- The proposed polling loop still treats all gh api failures as zero assets, which can hide auth/rate-limit/outage failures behind a misleading missing-assets timeout.
- The loop sleeps after the final check and then fails without a last asset recheck, creating a possible false failure if private publish completes during the final minute.
- The tag validation is prefix-only; it is low exposure because release creation/workflow dispatch require repository privileges, but tightening the accepted tag format would reduce output/path surprises.
