## Summary

Current main partially handles the direct successful `cron.add` case, but it does not yet cover the Telegram-only leak described here. Telegram outbound direct text, media, and payload paths still forward the resolved text without an issue-specific reminder-note suppressor, and the inspected reminder telemetry still only counts direct `cron` tool calls with `action: "add"`. The prior ClawSweeper comment correctly kept this open; the related closed Telegram NO_REPLY/status leak is a different shipped fix and does not cover this exact reminder note.

## Best Possible Solution

Keep this issue open. The best fix is a narrow, regression-tested change in the Telegram plugin outbound/runtime delivery path to suppress the exact false unscheduled-reminder note from direct text, media, and payload fallback sends, or a broader telemetry fix that reliably marks every real reminder-scheduling path before the guard runs. Tests should cover Telegram direct text, media, interactive/payload fallback, direct `cron.add`, and a helper/exec-created reminder path if that path is supported.
