## Summary

Current main has direct successful cron-add telemetry, and the auto-reply guard uses it to avoid adding the unscheduled-reminder note when `successfulCronAdds > 0`. However, the Telegram outbound paths still do not suppress the specific bogus reminder note described in this issue, and the telemetry currently only recognizes direct `cron` tool calls with `action: "add"`. Because the report explicitly calls out Telegram-only delivery leakage and helper/exec-driven scheduling paths, the issue should stay open until Telegram outbound/runtime handling or broader reminder telemetry is covered with regression tests.

## Best Possible Solution

Keep this issue open. The best fix is a narrow regression-tested change that either removes this exact unscheduled-reminder note from all Telegram outbound text/media/payload fallback delivery paths, or broadens reminder-success telemetry so every real scheduling path that can create a dismissible reminder chain suppresses the guard before delivery. Tests should cover direct Telegram text, media, interactive/payload fallback, and a successful reminder path that previously produced a clean transcript but leaked the note to Telegram.
