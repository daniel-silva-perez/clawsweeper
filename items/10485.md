## Summary

Keep open. Current `main` already strips several markdown constructs before sending TTS text to providers, but the implementation does not fully match this issue's requested behavior: `messages.tts` has no `stripMarkdown` option, and the shared stripper does not handle list markers like `- ` or `1. ` that the report calls out explicitly. That is not strong enough evidence to close as implemented.

## Evidence

- **TTS path strips markdown before synthesis:** `maybeApplyTtsToPayload` normalizes the outbound TTS text with `stripMarkdown(...).trim()` immediately before calling `textToSpeech(...)`. This shows the feature is partially present on current `main`.
  - file: [extensions/speech-core/src/tts.ts:1147](https://github.com/openclaw/openclaw/blob/754acc4478fe361db1e5303fd60f589d52beea7a/extensions/speech-core/src/tts.ts#L1147)
  - command: `nl -ba extensions/speech-core/src/tts.ts | sed -n '1138,1152p'`
  - sha: [754acc4478fe](https://github.com/openclaw/openclaw/commit/754acc4478fe361db1e5303fd60f589d52beea7a)
- **Shared markdown stripper is partial:** `stripMarkdown` removes bold/italic, strikethrough, headers, blockquotes, horizontal rules, and inline code, but there is no handling for unordered or ordered list markers such as `- ` or `1. `. Those were called out in the issue body.
  - file: [src/shared/text/strip-markdown.ts:5](https://github.com/openclaw/openclaw/blob/754acc4478fe361db1e5303fd60f589d52beea7a/src/shared/text/strip-markdown.ts#L5)

## Risks / Open Questions

- Closing as implemented would overstate the current behavior; list-marker speech cleanup still appears uncovered in code and tests.
- The requested configurability is absent, so users cannot opt in or out of markdown stripping through `messages.tts` as proposed.
