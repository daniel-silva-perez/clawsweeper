## Summary

Keep this issue open. Current main still has the synchronous `tools.effective` gateway path implicated by the report, and open PR #72558 is an implementation candidate with closing syntax, not a reason to close the issue before merge. The linked PR is narrow but likely incomplete because a single `setImmediate` yield does not make the subsequent CPU-bound resolver nonblocking.

## Best Possible Solution

Keep this issue open. Review #72558 as the current candidate, but require a real event-loop-safe fix before closing: make effective tool inventory resolution incremental with internal async yield points, cache/debounce the expensive computation, or move CPU-heavy work off the main gateway event loop, then add focused regression coverage for concurrent gateway request responsiveness during `tools.effective`.
