## Summary

Keep open. Current main has Telegram polling-stall recovery and status fallback behavior, but I did not find a shipped or current implementation that addresses the reported whole-Gateway CPU spin, local RPC/status timeout, Telegram reply pipeline stall, or suspected stuck interactive WhatsApp login launched through the Gateway/agent path. The related #63125 is adjacent but older and broader; this issue has newer 2026.4.24 reproduction details and an additional reporter, so it should not be closed as a duplicate yet.

## Best Possible Solution

Keep this issue open and treat it as a current core/runtime bug. The best next path is to use the attached reproduction logs to isolate the hot loop, add Gateway runtime liveness diagnostics for event-loop delay/CPU spin and useful unhandled-rejection context, and add a regression guard so agent-launched interactive channel login commands cannot block or starve the shared Gateway reply pipeline. If the root cause narrows to WhatsApp login, fix it in the WhatsApp plugin with only generic core supervision seams added as needed.
