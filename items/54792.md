## Summary

Keep open. Current main has a clearer runtime/startup error and docs now state that `hooks.enabled=true` requires `hooks.token`, but the schema and `config.patch` validation path still do not encode or reject that dependency before writing the config. The central requested validation/warning behavior remains missing.

## Best Possible Solution

Keep the issue open and implement a shared operational config dependency validator used by `config.set`, `config.apply`, `config.patch`, and startup. For this case, reject `hooks.enabled === true` when `hooks.token` is missing or blank, return an actionable issue at `hooks.token`, add schema/UI hint metadata if the schema lookup contract supports it, and cover the behavior with focused config validation and gateway config.patch tests.

## Risks / Open Questions

- Closing this as implemented would leave the reported `config.patch` failure mode mostly intact: a structurally valid patch can still persist `hooks.enabled: true` without `hooks.token` and only fail later in runtime hook resolution.
- The current docs and runtime error reduce confusion, but they do not generalize dependency metadata for other feature flags as requested.
