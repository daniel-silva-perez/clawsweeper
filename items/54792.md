## Summary

Keep open. Current main has a clear runtime/startup guard and documentation for `hooks.enabled` requiring `hooks.token`, but the requested schema/config-write dependency validation is still missing from the shared validator, Gateway config write RPCs, persistence path, and lookup metadata.

## Best Possible Solution

Keep this issue open. Add a shared cross-field config validation rule that rejects `hooks.enabled=true` unless `hooks.token` is a non-empty string, and route it through `validateConfigObjectRaw` / `validateConfigObjectWithPlugins` so `config.set`, `config.apply`, `config.patch`, `writeConfigFile`, CLI writes, and plugin runtime config writes all reject the invalid config before persistence. Keep `resolveHooksConfig` as the startup/reload backstop, and add focused regression tests for shared validation plus Gateway config write behavior. If `config.schema.lookup` is intended to guide clients, add explicit dependency or warning metadata there too.

## Risks / Open Questions

- Adding a hooks-only check in one Gateway RPC handler would leave CLI, plugin runtime, direct persistence, and other config write surfaces divergent.
- A broad dependency metadata framework may be larger than necessary; the immediate outage path can be addressed with a small shared cross-field validator plus optional lookup hint metadata.
