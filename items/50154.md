## Summary

Keep open. Current `main` does harden LaunchAgent runtime setup on macOS by injecting a minimal PATH, HOME, TMPDIR, proxy vars, and TLS defaults, but I did not find strong evidence that it fixes the reported LaunchAgent-vs-foreground SSH behavior for LAN GitLab pushes. The remaining code and docs still make SSH behavior depend on whatever environment the LaunchAgent already has, and the login-shell fallback does not import SSH-specific variables such as `SSH_AUTH_SOCK`. I could not conservatively classify this as implemented or cannot-reproduce from the current checkout alone.

## Evidence

- **LaunchAgent env setup is generic, not SSH-specific:** Gateway service env on current main includes `HOME`, `TMPDIR`, proxy vars, `NODE_EXTRA_CA_CERTS`, `NODE_USE_SYSTEM_CA`, and a synthesized `PATH`, but there is no explicit handling for `SSH_AUTH_SOCK` or other Git/SSH parity knobs in the LaunchAgent environment builder.
  - file: [src/daemon/service-env.ts:317](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/src/daemon/service-env.ts#L317)
