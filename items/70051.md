## Summary

Keep PR #70051 open. Current main still routes node `exec.started` / `exec.finished` / `exec.denied` events through `enqueueSystemEvent` and `requestHeartbeatNow` after only loading the canonical session key, with no check for `heartbeatIsolatedBaseSessionKey`. The existing session model and heartbeat runner already expose and rely on that marker, so the PR's narrow guard is plausible and not superseded by the related heartbeat fixes.

## Best Possible Solution

Keep the PR open for maintainer review or implementation. The best code path is a narrow `src/gateway/server-node-events.ts` guard that reads `entry` from `loadSessionEntry(sessionKeyRaw)` and suppresses only node `exec.started`, `exec.finished`, and `exec.denied` replay when the resolved session entry has `heartbeatIsolatedBaseSessionKey`, with a regression test proving normal non-heartbeat exec notifications still enqueue and wake while synthetic heartbeat-session node exec events do not.
