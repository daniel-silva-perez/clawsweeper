## Summary

Keep #51337 open. Current main still has the reported hook-session continuation gap: hook agent dispatch always creates an isolated synthetic cron job, while the cron runner treats isolated jobs as forced-new sessions, so persisted CLI session ids are not reused for mapped hook runs. The PR addresses a real open bug (#51208) and its discussion shows the author handled several review findings, but the work is not implemented on main and likely needs rebase/cleanup against the current cron session-key canonicalization.

## Best Possible Solution

Keep the PR or an equivalent focused fix open for #51208. The implementation should let hook mappings opt into non-isolated session continuation, preserve the correct job/session key for delivery routing, keep external request session-target control behind an explicit policy gate, update config schema/docs/baselines, and add regression tests for mapped hooks resuming existing CLI sessions plus main/current/session:* edge cases against current main.
