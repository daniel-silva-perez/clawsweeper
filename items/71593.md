## Summary

Keep #71593 open. Current main has built-in SecretRef primitives and manifest-declared plugin config SecretInput paths, but it still lacks a plugin-defined SecretRef provider contract and does not cover the requested core surfaces such as `skills.entries.*.env`. The linked AWS Secrets Manager PR #43643 was closed unmerged and no equivalent provider or generic plugin provider API exists on main.

## Best Possible Solution

Keep this open as a core/plugin API design issue. Maintainers should either document env/file/exec, especially exec, as the supported extensibility model, or add a versioned plugin SecretRef-provider contract and extend the core schemas, Gateway protocol schema, resolver dispatch, plugin SDK/runtime registration, docs, tests, and remaining requested core SecretInput surfaces such as `skills.entries.*.env`.

## Risks / Open Questions

- Adding plugin-defined SecretRef providers would affect public config types, Zod validation, Gateway protocol schemas, resolver dispatch, plugin SDK/runtime registration, docs, and tests, so it needs maintainer API design rather than conservative cleanup.
- The existing `exec` provider may be the intended extensibility answer for many external secret stores, but that is not equivalent to a plugin-defined provider contract and should be documented or confirmed by maintainers if chosen as the product direction.
