## Summary

Keep this PR open for maintainer review rather than auto-closing. Current main does not contain the added devcontainer, benchmark workflow, or release SBOM workflow, so the PR is not implemented on main or cleanly superseded. It is also actionable in this repository because it edits GitHub workflow/devcontainer configuration. However, it is not merge-ready as-is: the latest PR context still shows unresolved review concerns about privileged hyperfine installation without integrity verification and an SBOM that scans the full workspace/dev dependency set instead of the shipped production artifact.

## Best Possible Solution

Keep the PR open, but request a focused revision or maintainer decision: either split devcontainer, benchmark, and SBOM into separately reviewable PRs, or revise this one to reuse `.github/actions/setup-node-env/action.yml`, call the existing `test:startup:bench`/budget harness where appropriate, pin or avoid the hyperfine package install, and generate the SBOM from the actual shipped production package/artifact rather than the full dev workspace. If maintainers do not want these repo-level workflows, they should close explicitly as an infra/product-policy decision, not as already implemented.

## Risks / Open Questions

- Adding new CI and release workflows is maintainer-policy work; it needs explicit review of schedule, permissions, artifact retention, and whether benchmark/SBOM output is useful enough to maintain.
- The latest PR head still appears to install a downloaded `.deb` with `sudo dpkg -i` without a checksum, which is a supply-chain risk for CI.
- The proposed SBOM scans `path: .` after a full workspace install, so it may publish dev/test dependencies rather than the production release artifact consumers receive.
- The benchmark workflow overlaps existing startup benchmark scripts and docs; accepting it without reusing the maintained harness could create duplicated performance surfaces.
