## Summary

Keep open. Current main still lacks the proposed Telegram send-level `isSilentReplyText` guard, and this PR remains a plausible focused fix for the open linked Telegram issue rather than an auto-close cleanup item. Normal outbound delivery now handles many exact `NO_REPLY` paths, but that does not prove the direct `sendMessageTelegram` entry point is guarded.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is a small Telegram adapter guard using `isSilentReplyText(normalizeOptionalString(text) ?? "")`, with explicit exceptions for media and any structured Telegram reply markup/buttons that should still send, plus targeted send-adapter tests. If maintainainers decide central outbound normalization is the only desired product behavior, close this PR and #38603 manually with that rationale after review.

- The PR diff in the provided context changes `opts` to default to `{}` even though `TelegramSendOpts` requires `cfg`; maintainers should keep `opts` required or adjust the type deliberately before merge.
