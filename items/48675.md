## Summary

Keep open. Current `main` does not implement this pre-touch fix, and the PR as described would bypass current `memoryFlush.enabled` and read/write guards by doing an unconditional append-open before agent startup. It also targets a stale CLI file path that is now just a re-export.

## Evidence

- **Gateway path still lacks any daily-memory precreation on main:** `getReplyFromConfig` sets up the workspace and immediately continues; there is no `ensureMemoryFlushTarget` or equivalent pre-touch in the current gateway entry path.
  - file: [src/auto-reply/reply/get-reply.ts:221](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/src/auto-reply/reply/get-reply.ts#L221)
  - command: `nl -ba src/auto-reply/reply/get-reply.ts | sed -n '221,229p'`
  - sha: [4013c658537e](https://github.com/openclaw/openclaw/commit/4013c658537ee2051c44186f197b0710e84c9049)
- **CLI entrypoint moved; PR touches a stale wrapper path:** `src/commands/agent.ts` is now only a re-export stub, while the real CLI startup path lives in `src/agents/agent-command.ts`, which also just ensures the workspace and proceeds without any memory-file pre-touch.
  - file: [src/commands/agent.ts:1](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/src/commands/agent.ts#L1)

- **Current main intentionally skips memory flush when disabled:** The memory plugin returns `null` when `agents.defaults.compaction.memoryFlush.enabled` is `false`, and the runtime exits early when no flush plan exists. An unconditional startup pre-touch would bypass that current contract.
  - file: [extensions/memory-core/src/flush-plan.ts:105](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/extensions/memory-core/src/flush-plan.ts#L105)

- **Current main guards memory flush behind workspace write access:** Before attempting a memory flush, the runtime checks sandbox/workspace writability and only proceeds when writes are allowed. A pre-touch before startup would move a write attempt ahead of that guard.
  - file: [src/auto-reply/reply/agent-runner-memory.ts:540](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/src/auto-reply/reply/agent-runner-memory.ts#L540)

  - file: [src/infra/fs-safe.ts:540](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/src/infra/fs-safe.ts#L540)

## Risks / Open Questions

- Merging the PR as-is would attempt a workspace write on every gateway/CLI startup even when `agents.defaults.compaction.memoryFlush.enabled` is `false`.
- It would also move a write attempt ahead of the existing `memoryFlushWritable` guard, risking avoidable failures on read-only or sandboxed workspaces.
- The CLI hunk needs rebasing because `src/commands/agent.ts` is now a wrapper and the real implementation lives in `src/agents/agent-command.ts`.
