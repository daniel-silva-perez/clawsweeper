## Summary

Keep open. Current `main` does not implement eligibility-based skill-snapshot invalidation: persisted snapshots still only refresh on snapshot-version bumps or normalized `skillFilter` changes, while actual skill eligibility also depends on runtime bin/env/config checks. Closing this as already fixed would be incorrect.

## Evidence

- **Reply-session snapshots still ignore runtime eligibility changes:** `ensureSkillSnapshot()` refreshes only when `shouldRefreshSnapshotForVersion(...)` or `!matchesSkillFilter(...)` is true. There is no persisted eligibility fingerprint and no comparison against runtime eligibility state.

## Risks / Open Questions

- Closing this PR as already implemented would leave stale skill snapshots possible when runtime env/bin/config eligibility changes without a snapshot-version or `skillFilter` change.
- Even if the PR lands, equivalent invalidation is still absent in other snapshot caches (`src/agents/agent-command.ts`, `src/cron/isolated-agent/skills-snapshot.ts`), so broader follow-up may still be needed.
