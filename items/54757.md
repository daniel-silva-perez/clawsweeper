## Summary

Keep open. Current main has partially mitigated the reported WhatsApp path by suppressing text-only tool lifecycle payloads before `deliverWebReply`, but it has not implemented the requested external-channel suppression policy. Final fallback tool-error warnings can still be delivered to WhatsApp, and the existing `messages.suppressToolErrors` config intentionally does not suppress mutating tool failures like write/edit/message failures, which matches the privacy-sensitive examples in the issue.

## Best Possible Solution

Keep the issue open and implement an explicit external-channel tool-error policy. The likely fix belongs in the shared reply/payload policy or WhatsApp dispatch options: allow WhatsApp and other external-facing channels to suppress all user-visible tool-error warning payloads, including mutating write/edit/message failures, while preserving internal logs and agent context. Add regression tests for a mutating tool failure reaching the final fallback path through WhatsApp delivery, and document the resulting config in the public channel/config docs.
