## Summary

Keep this issue open. Current main still has the reported leak path: WhatsApp suppresses text-only lifecycle tool payloads, but final warning payloads from mutating tool failures can still pass through the buffered dispatcher into deliverWebReply.

## Best Possible Solution

Implement an external-channel-safe tool-error policy for WhatsApp and similar recipient-facing channels. The narrow fix is to pass suppressToolErrorWarnings, or a channel-scoped equivalent, into WhatsApp inbound auto-reply runs, then add regression tests proving mutating read/edit/write/message failure warnings are not delivered as final WhatsApp replies while normal final replies and intentional media/tool artifacts still deliver.
