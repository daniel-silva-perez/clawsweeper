## Summary

Keep this PR open. Current main still omits OpenAI-compatible max token request fields unless `options.maxTokens` is explicitly supplied, while model definitions and the OpenRouter capability path do carry `model.maxTokens`. The proposed transport fix is still relevant, but the PR should be updated before merge because it bundles an unrelated TUI refactor and review comments identify regressions in that refactor plus a small `||` versus `??` cleanup in the max-token fallback.

## Best Possible Solution

Keep the PR open, but ask the contributor to rebase/split it so the OpenAI-compatible max-token fallback is a focused transport change with tests. The best code path is to resolve an effective positive max token value from `options.maxTokens` first and `model.maxTokens` second, then write it to `max_output_tokens`, `max_tokens`, or `max_completion_tokens` according to the existing compatibility logic. The unrelated TUI changes should be dropped from this PR or handled separately after their review regressions are fixed.
