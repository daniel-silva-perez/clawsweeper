## Summary

Keep open. Current main has meaningful partial coverage: docs explain what counts as context, Control UI shows high-pressure context usage and compaction affordances, and cache-TTL pruning has configurable tool-result trimming. But the central request is broader visibility and control over pruning itself, and current pruning still rewrites the in-memory prompt silently without a user-facing preview/log, pruning notification, or pruning metadata hook. Related open reports #8606 and #9409 also confirm narrower missing pieces around pruning events and overflow diagnostics.

## Best Possible Solution

Keep this open, but narrow it into a maintainer-approved pruning observability/control design. The likely implementation belongs in `src/agents/pi-hooks/context-pruning/*`: make the pruner produce safe metadata, emit an internal/plugin-visible pruning lifecycle event, surface a concise Control UI/chat notice when pruning happens, document the existing and new knobs, and coordinate with #8606 for the hook shape and #9409 for overflow diagnostics.
