## Summary

Current main still keeps Slack thread participation in memory only, while the PR addresses a real unshipped restart-survival gap. Related open PRs do not provide a merged canonical replacement yet.

## Best Possible Solution

Keep this PR open while maintainers choose the canonical Slack thread-participation persistence path. The likely implementation is to wire Slack participation tracking to the existing `openclaw/plugin-sdk/persistent-dedupe` helper, or to the accepted persistent keyed-store contract if #70626 becomes the chosen SDK surface, with regression coverage for restart round-trip, TTL expiry, missing/corrupt state, clear semantics, max-entry/LRU behavior, and split module instances.
