## Summary

Keep open. Current main has Anthropic `cacheRetention` support and heartbeat scheduling, but it does not implement the proposed opt-in Cache TTL Warmer: no `cacheWarmer` config surface, no independent scheduler, no synthetic non-persistent minimal request path, and no docs/tests for this behavior. The related heartbeat-cache discussion is adjacent, not a superseding implementation of this dedicated warmer proposal; the CausaMem comment is useful adjacent memory work but not an OpenClaw core implementation.

## Best Possible Solution

Leave #70418 open for maintainer product/API judgment. If accepted, the likely implementation belongs in core agent/provider runtime, with a strict `agents.defaults/list[].cacheWarmer` config, provider capability gating for prompt caching, a scheduler separate from heartbeat delivery, construction of the same stable request prefix as the next real turn, a non-persistent synthetic marker with tiny max tokens, idle-skip logic, no transcript/channel delivery, and focused regression/live tests. If maintainers decide heartbeats should remain the only cache keep-alive, close explicitly after that product decision rather than via automated cleanup.
