## Summary

Keep open. Current main still lacks the requested per-job/per-heartbeat elevated trust model. Cron storage/types, Gateway cron schemas, CLI/UI/agent-tool surfaces, and main-session cron/heartbeat execution still have no `elevated` flag or owner-approved provenance. Existing isolated-cron hardening reduces some ambient context inheritance, but it does not replace `tools.elevated.allowFrom.heartbeat` or `tools.elevated.allowFrom.cron-event` with per-job scoping.

## Best Possible Solution

Keep the issue open and design a core trust-model change: add explicit elevation/provenance metadata to cron job create/update/storage/protocol/CLI/UI/agent-tool surfaces, require a real elevated sender to set or modify it, and make heartbeat elevation depend on explicit owner-approved config or last-modified provenance instead of `heartbeat`/`cron-event` wildcards. Add regression tests for non-elevated users creating/editing cron jobs and heartbeat instructions without inheriting elevated exec.

## Risks / Open Questions

- This is security-sensitive because it changes how elevated exec authority is delegated to scheduled and internal runs.
- The implementation crosses public cron persistence/types, Gateway protocol validation, CLI, Control UI, agent cron tool, heartbeat, docs, and regression tests.
- Existing isolated-cron safeguards may reduce one abuse path, but closing now would leave main-session cron and heartbeat dependent on synthetic-provider allowlists.
