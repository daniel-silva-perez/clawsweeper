## Summary

Keep #62966 open. Current main still has the underlying plugin hook contract gap: reply metadata exists in inbound message context and Telegram populates it, but `deriveInboundMessageHookContext` does not carry `ReplyToId`/`ReplyToBody` into the canonical inbound hook context, and `toPluginInboundClaimEvent` does not expose those values in `metadata`. The PR is focused, additive, and the existing Greptile comment correctly identifies it as following the current optional metadata propagation pattern rather than duplicating shipped behavior.

## Best Possible Solution

Keep the PR or an equivalent maintainer patch open and update `src/hooks/message-hook-mappers.ts` to propagate optional `replyToId` and `replyToBody` through the canonical inbound context into `inbound_claim` metadata. Add a focused regression assertion in `src/hooks/message-hook-mappers.test.ts` using `makeInboundCtx({ ReplyToId, ReplyToBody })` so the hook contract does not regress. This belongs in core hook mapping, not ClawHub, because external plugins cannot recover metadata that core drops before dispatching the hook.
