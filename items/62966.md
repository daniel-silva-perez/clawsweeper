## Summary

Keep this PR open. Current main still drops `ReplyToId` and `ReplyToBody` before the `inbound_claim` plugin event, while the source fields exist upstream and the PR is a narrow additive implementation candidate. The prior ClawSweeper comment reached the same conclusion, and current main still matches that finding.

## Best Possible Solution

Keep this PR open and review it as a small plugin hook API fix. The best path is to add the optional `replyToId` and `replyToBody` fields to the canonical inbound hook context, map them from `FinalizedMsgContext`, expose them in `inbound_claim` metadata, and add a focused test in `src/hooks/message-hook-mappers.test.ts`. The security review should remain simple: verify the diff stays limited to existing untrusted message-context propagation and does not introduce unrelated infrastructure, dependency, or code-execution changes.
