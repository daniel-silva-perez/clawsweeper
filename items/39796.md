## Summary

Keep open. Current main has moved the WhatsApp code under `extensions/whatsapp`, and the old `lastEventAt` gateway health path has been narrowed to provider transport activity, but the WhatsApp extension still has its own inbound-time watchdog that force-closes quiet connections after the timeout window. The PR is stale and needs a rebase/test work, yet the underlying quiet-channel false reconnect problem is still plausible and not implemented on main.

## Best Possible Solution

Keep the item open and either rebase the fix into `extensions/whatsapp` or replace it with a maintainer-owned patch that adds provider-proven WhatsApp transport liveness to the watchdog. The implementation should expose a narrow liveness probe, refresh the watchdog/status only when the probe succeeds, preserve dead-socket detection when it fails, and add regression tests for successful and failed ping behavior.
