## Summary

Keep open. Current `main` has stronger per-session safeguards for Telegram updates, auto-reply runs, and native Codex conversation binding, but I did not find strong evidence that the specific reported path of launching two parallel `codex exec --full-auto` subprocesses from one Telegram-driven session is now rejected, serialized, or otherwise fixed at the exec/process layer.

## Evidence

- **Session runs are serialized at the reply pipeline:** Docs state inbound auto-reply runs are serialized per session key and then queued into a global lane, which addresses concurrent inbound turns but not necessarily concurrent subprocesses started inside one turn.
  - file: [docs/concepts/queue.md:10](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/docs/concepts/queue.md#L10)
  - command: `nl -ba docs/concepts/queue.md | sed -n '1,40p'`
  - sha: [4013c658537e](https://github.com/openclaw/openclaw/commit/4013c658537ee2051c44186f197b0710e84c9049)
- **Active session runs are queued or rejected cleanly:** `agent-runner.ts` enqueues followups when a run is already active, and if a second reply operation still collides it returns `Previous run is still shutting down` instead of corrupting session state.
  - file: [src/auto-reply/reply/agent-runner.ts:1006](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/src/auto-reply/reply/agent-runner.ts#L1006)
  - command: `nl -ba src/auto-reply/reply/agent-runner.ts | sed -n '1000,1100p'`
  - sha: [4013c658537e](https://github.com/openclaw/openclaw/commit/4013c658537ee2051c44186f197b0710e84c9049)
- **Telegram updates are sequenced by chat/topic:** Telegram bot middleware tracks pending updates and then applies `sequentialize(getTelegramSequentialKey)`, which is a real protection against same-chat/topic update races.
  - file: [extensions/telegram/src/bot-core.ts:287](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/extensions/telegram/src/bot-core.ts#L287)
  - command: `nl -ba extensions/telegram/src/bot-core.ts | sed -n '282,420p'`
  - sha: [4013c658537e](https://github.com/openclaw/openclaw/commit/4013c658537ee2051c44186f197b0710e84c9049)
- **Native Codex conversation binding is queued per bound session:** Current native Codex binding wraps inbound turns in `enqueueBoundTurn(data.sessionFile, ...)`, and `enqueueBoundTurn` chains turns through a per-session queue.
  - file: [extensions/codex/src/conversation-binding.ts:121](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/extensions/codex/src/conversation-binding.ts#L121)
  - command: `nl -ba extensions/codex/src/conversation-binding.ts | sed -n '90,140p'; nl -ba extensions/codex/src/conversation-binding.ts | sed -n '330,360p'`
  - sha: [4013c658537e](https://github.com/openclaw/openclaw/commit/4013c658537ee2051c44186f197b0710e84c9049)
- **Exec/background process layer still lacks session-key coordination:** `runExecProcess()` creates a new process session for every exec call and `bash-process-registry.ts` stores running sessions only by generated process-session id; I did not find a per-OpenClaw-session mutex/queue for multiple background exec subprocesses sharing the same `sessionKey`.
  - file: [src/agents/bash-tools.exec-runtime.ts:520](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/src/agents/bash-tools.exec-runtime.ts#L520)

## Risks / Open Questions

- The exact repro was not executed in this read-only review, so the decision relies on code inspection rather than a live Telegram/gateway reproduction.
- Current main clearly hardened inbound/session concurrency, but that does not prove the shell-driven `codex exec --full-auto` subprocess path is fixed.
- Native Codex bind/app-server paths are better guarded now, but the issue reports a generic host-exec orchestration pattern that can still be in use.
