## Summary

Keep open. Current main added a strong WhatsApp fix after the prior review: quiet-but-healthy sessions now use WhatsApp Web transport activity instead of inbound-message silence, with tests and docs. That mitigates the reporter's 30-minute watchdog path, but the broader Windows flapping report still has source-backed remaining gaps: 408/428 retry loops still match current behavior, the libsignal-only Bad MAC follow-up is not clearly covered by the crypto-error classifier, and related listener readiness/socket-teardown issues remain open.

## Best Possible Solution

Keep #70856 open, but narrow it around the remaining WhatsApp lifecycle failures after e672b614. The next maintainer pass should verify Windows scheduled-task behavior on current main, decide whether 408/428 loops are expected or need bounded recovery, broaden Bad MAC/libsignal recovery if appropriate, and coordinate any socket teardown work with #52442 plus listener-readiness delivery behavior with #37428. If live/current-main testing shows e672b614 fully resolves the reporter's scenario, close then with that fix evidence.
