## Summary

Keep PR #67049 open. The underlying bug still exists on current main, and this PR is the active useful implementation direction, but it is incomplete: it only patches the Gateway RPC reset service, does not cover the normal chat `/new`/`/reset` path in `initSessionState()`, lacks the empty-key/failed-reset hardening called out in review, and has no regression test proving queued system events are drained across reset boundaries.

## Best Possible Solution

Keep #67049 open and ask for a focused update: use the normalized keys returned by `clearSessionQueues()` when draining Gateway reset cleanup, add equivalent stale system-event draining for deliberate chat session creation in `src/auto-reply/reply/session.ts`, ensure draining happens only once the reset path succeeds, and add regression tests for both `sessions.reset` and `/new`/`/reset` via `initSessionState()` plus prompt drain behavior.
