## Summary

Keep open. Current main has partial improvements around node pairing semantics, merged node status/listing, canonicalized node exec approval context, and non-blocking remote-node startup work, but this issue is an umbrella over several separate node/connectivity problems and at least one cited gap is still explicitly present on main: the iOS node app remains internal preview only.

## Evidence

- **WS node pairing flow changed, but not to a single resolved surface:** The current docs explicitly say WebSocket nodes pair through device pairing (`openclaw devices list/approve`), while `openclaw nodes pending` is a separate `node.pair.*` store that does not gate the WS handshake. That explains part of the earlier confusion, but it means the issue spans multiple pairing surfaces rather than a single fixed bug.
