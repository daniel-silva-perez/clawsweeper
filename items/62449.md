## Summary

Keep open. Current `main` still builds the file transport around the initially resolved `settings.file` and never re-checks the rolling filename during writes, while subsystem loggers cache their child logger across writes. That matches the reported midnight rollover bug rather than disproving it, and the proposed fix is not already implemented on `main`.

## Evidence

- **Current file transport never switches files after logger construction:** `buildLogger()` creates the log directory, optionally prunes once, snapshots `currentFileBytes`, and then every write path appends to `settings.file`. There is no per-write date/path refresh in current `main`, so a long-lived logger would keep using the original dated filename.
  - file: [src/logging/logger.ts:168](https://github.com/openclaw/openclaw/blob/ecac696643c604c6d854341bfe6424bfa31423e9/src/logging/logger.ts#L168)
  - command: `nl -ba src/logging/logger.ts | sed -n '153,205p'`
  - sha: [ecac696643c6](https://github.com/openclaw/openclaw/commit/ecac696643c604c6d854341bfe6424bfa31423e9)
- **Rolling filename is only computed during settings resolution:** Current `main` computes the default rolling filename in `resolveSettings()` via `defaultRollingPathForToday()`, and `getLogger()` only rebuilds when resolved settings change. Nothing inside the attached file transport re-evaluates the date mid-process.
  - file: [src/logging/logger.ts:127](https://github.com/openclaw/openclaw/blob/ecac696643c604c6d854341bfe6424bfa31423e9/src/logging/logger.ts#L127)
