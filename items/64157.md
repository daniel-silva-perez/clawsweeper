## Summary

Keep #64157 open. Current main preserves an existing external route when WebChat/Control UI touches a session, but the Control UI send path still uses `deliver: false`, the Gateway resolves that as an internal webchat origin, and dispatch explicitly avoids external routing for internal webchat turns unless `ExplicitDeliverRoute` is set. That means the reported Feishu/Web Control UI handoff behavior is not implemented on main; related route-preservation fixes only prevent route metadata from being overwritten.

## Best Possible Solution

Keep the issue open for maintainer/product handling. The likely fix is to define Control UI handoff semantics, then either add a UI/API affordance to continue delivery to the selected session's external route or have `chat.send` safely inherit the persisted Feishu delivery context for this case. Add a regression covering Feishu or a generic deliverable-channel session where a Control UI follow-up during an active run still routes the final reply to the original external conversation without overwriting route metadata.
