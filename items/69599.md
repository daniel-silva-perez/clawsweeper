## Summary

Keep #69599 open. Current main still routes Control UI New Session through `/new` on the active session key, and backend `/new`/`/reset` rotate a new `sessionId` in place for that same canonical key rather than creating a distinct fresh WebChat session key. The direct-created session workaround described in the issue is a separate `sessions.patch`/explicit-key path, not the standard New Session flow.

## Best Possible Solution

Keep #69599 open as the canonical Control UI fresh-session tracker. A complete fix would add an explicit WebChat fresh-session creation/switch path, likely by generating a unique session key through a typed gateway method or a constrained `sessions.patch` flow, then wire `/new`, `/reset`, and the New Session button consistently, update the selector/URL/settings behavior, document the semantics, and add regression coverage. Coordinate with #47561 so reset freshness also remains durable across routing-mode changes.
