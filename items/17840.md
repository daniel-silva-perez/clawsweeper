## Summary

Keep open. Current main still treats channel reactions as queued system events rather than an opt-in wake path, and the QA contract explicitly expects reactions not to trigger a fresh reply.

## Evidence

- **Telegram reactions are enqueue-only:** The Telegram reaction handler resolves a session key and only calls `enqueueSystemEvent(...)` for added reactions; there is no heartbeat wake or turn-trigger path in this flow.
  - file: [extensions/telegram/src/bot-handlers.runtime.ts:978](https://github.com/openclaw/openclaw/blob/a59d1bd46d9fc0bd2d9194b3f17db2561efc51e3/extensions/telegram/src/bot-handlers.runtime.ts#L978)

  - sha: [a59d1bd46d9f](https://github.com/openclaw/openclaw/commit/a59d1bd46d9fc0bd2d9194b3f17db2561efc51e3)
- **Reaction wake primitive exists but is not used here:** Plugin runtime exposes `requestHeartbeatNow` alongside `enqueueSystemEvent`, which shows reactions could opt into a wake path, but current reaction handlers are still using only the queueing half.
  - file: [src/plugins/runtime/runtime-system.ts:17](https://github.com/openclaw/openclaw/blob/a59d1bd46d9fc0bd2d9194b3f17db2561efc51e3/src/plugins/runtime/runtime-system.ts#L17)
