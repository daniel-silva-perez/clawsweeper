## Summary

Keep open. Current main already uses `api: "openai-completions"` for `/v1/chat/completions`, so the endpoint part of the request is covered, but there is still no `openai-chat` API alias/type and generic custom OpenAI-compatible providers still default runtime max-token payloads to `max_completion_tokens`. That means the reported `Unknown parameter: max_completion_tokens` failure can still be real unless users know to set a per-model compat override.

## Best Possible Solution

Keep the existing `openai-completions` transport as the shared chat-completions implementation, but add a maintainer-reviewed UX path for providers that expect classic chat-completions payloads: either document and expose `compat.maxTokensField: "max_tokens"` in custom-provider docs/onboarding, or add an `openai-chat`/chat-compatible alias or preset that maps to `openai-completions` with `maxTokensField: "max_tokens"` and the existing non-native compatibility defaults. Add regression coverage showing a generic custom provider configured through that path sends `max_tokens` and not `max_completion_tokens`. Service-specific Ouyi defaults could live in a provider plugin if they need more than generic chat-completions compatibility.

## Risks / Open Questions

- Closing as implemented would likely be confusing for users whose provider accepts `/v1/chat/completions` but rejects the current generic default `max_completion_tokens` field.
- Adding a fully separate transport may duplicate the existing `openai-completions` chat-completions path; an alias, compat preset, or onboarding/docs improvement may be enough.
