## Summary

Current main still has the reported scheduler shape. Followup drains continue until the followup queue is empty, each followup runs a normal embedded agent turn through the same per-session lane, and the command queue remains single-concurrency FIFO with no user-message priority or preemption. Heartbeat skipping and background context-engine maintenance reduce adjacent latency contributors, but they do not solve fresh inbound messages waiting behind accumulated followup turns.

## Best Possible Solution

Keep this issue open and fix it in the core reply/session scheduler. The likely best path is a bounded or fair followup drain, explicit priority/preemption for fresh user messages over accumulated followup/system-event work on the same session key, phase-aware aggregate timing for queue/preflight/provider/post-turn work, and regression coverage for fresh inbound dispatch behind active followup queues. Related open issues #65504 and #66638 should stay linked as adjacent responsiveness and notification-delivery work, but they do not supersede this specific scheduler fairness bug.
