## Summary

Keep open. Current main still loads remote PDFs through `loadWebMediaRaw` without any `readIdleTimeoutMs`, and `loadWebMediaInternal` does not expose or forward that option to `fetchRemoteMedia`. The lower-level fetch helper can abort stalled body reads when given a timeout, and native Anthropic/Gemini PDF provider calls now have a 120s `AbortSignal`, but the reported large remote-PDF fetch stall remains unfixed. The related fix PR #68747 was closed unmerged, so it is not shipped on main.

## Best Possible Solution

Land a focused core fix on the PDF remote-fetch path: extend `WebMediaOptions` with `readIdleTimeoutMs`, forward it from `loadWebMediaInternal` into `fetchRemoteMedia`, pass a bounded default from `createPdfTool` for remote PDF loads, and add a regression test in `src/agents/tools/pdf-tool.test.ts` plus the existing media/fetch coverage. Track `/new` and `/restart` bypass or session watchdog behavior separately if maintainers want recovery beyond the tool-specific timeout.
