## Summary

Keep open. Current main has session inspection and management surfaces, but the Control UI still does not implement the core missing piece from the issue: creating separate browser chat threads from the UI. The visible "New session" action is wired to `/new`, which resets/rotates the current session state rather than creating a new dashboard session entry, and the frontend has no `sessions.create` call path.

## Evidence

- **Control UI exposes a New session button:** The chat toolbar renders a `New session` button, but it is only a generic callback hook from the UI layer.
  - file: [ui/src/ui/chat/run-controls.ts:24](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/ui/src/ui/chat/run-controls.ts#L24)
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
- **New session button just sends `/new`:** The Control UI wires `onNewSession` to `state.handleSendChat("/new", { restoreDraft: true })` rather than calling `sessions.create` or switching to a newly created session key.
  - file: [ui/src/ui/app-render.ts:2271](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/ui/src/ui/app-render.ts#L2271)
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
- **`/new` rotates the current session rather than creating a separate chat thread in the UI:** When reset is triggered, the runtime generates a new `sessionId` and preserves fields from the existing entry under the same session context; this is session lifecycle/reset behavior, not a Control UI multi-chat creation flow.
  - file: [src/auto-reply/reply/session.ts:488](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/src/auto-reply/reply/session.ts#L488)
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
- **Backend supports session creation, but frontend does not consume it:** The gateway protocol and server expose `sessions.create`, including autogenerated dashboard keys, but `rg -n "sessions\\.create" ui/src/ui` returns no matches, so current Control UI has no frontend path to create a separate dashboard session.
  - file: [src/gateway/server-methods/sessions.ts:797](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/src/gateway/server-methods/sessions.ts#L797)
  - command: `rg -n "sessions\.create" ui/src/ui`
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
- **Existing UI only covers partial request scope:** Current docs and UI support session listing and per-session overrides, and the Sessions view allows label edits and deletion, but that is not the same as ChatGPT-style multi-chat creation/switching from chat.
  - file: [ui/src/ui/views/sessions.ts:518](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/ui/src/ui/views/sessions.ts#L518)
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)

## Risks / Open Questions

- The repo already has partial session-management support (listing, renaming, deleting, overrides), so maintainers should scope follow-up work to the missing create/switch UX rather than treating the whole request as entirely unstarted.
