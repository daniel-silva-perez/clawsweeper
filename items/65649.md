## Summary

Keep #65649 open. Current main still writes timestamped `.clobbered.*` files without a per-path cap in both config I/O implementations, and the existing tests only cover identical suspicious-hash dedupe, not bounded forensic storage when suspicious content keeps changing. The PR addresses a real core config hardening gap and is not superseded by the related in-process dedupe PR.

## Best Possible Solution

Keep #65649 open as the focused core config hardening vehicle. The best fix is a race-aware per-config-path cap for `.clobbered.*` artifacts across both async and sync clobber persistence paths, with audit logging preserved, one cap warning per path, and regression tests covering suspicious-read recovery plus observe/last-known-good paths.
