## Summary

Keep open. The report is recent, concrete, and matches the current task registry restore path: `main` catches and logs restore failures, but there is no source, docs, or test evidence of detecting malformed SQLite files, quarantining/snapshotting the bad DB, rebuilding via recovery, reopening the store, or notifying the operator.

## Best Possible Solution

Keep the issue open and implement a focused task-registry SQLite recovery path in `src/tasks/task-registry.store.sqlite.ts`/`src/tasks/task-registry.ts`: detect SQLite corruption errors during open/load, snapshot or quarantine `runs.sqlite` plus WAL/SHM sidecars, rebuild recoverable rows into a fresh schema using a safe internal recovery path or documented SQLite recovery command, reopen and reload the store, emit an explicit operator/system event, and add regression tests with a malformed or simulated-corruption store failure.
