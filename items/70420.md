## Summary

Keep #70420 open. Current main still matches the reported contract gap: `sessions_list` is backed by persisted session-store rows, the health monitor restart path only stops/starts the Discord channel runtime, and Discord only creates/updates session rows from inbound message processing. The issue comment correctly narrowed this to a docs/product-contract decision, and I did not find a current doc or implementation change that makes session-list emptiness a reliable or explicitly deprecated Discord health signal.

## Best Possible Solution

Keep the issue open and resolve the contract explicitly. The conservative path is a docs/status guidance fix: update the public health/session/heartbeat guidance to say `sessions_list` reflects persisted conversation sessions, not channel socket liveness, and point automated Discord health checks at `openclaw health`, `openclaw status --deep`, channel status/readiness, or the gateway readiness surface. Only materialize synthetic channel-heartbeat sessions if maintainers intentionally want to make `sessions_list` a health signal, with tests covering reconnect plus quiet-workspace behavior.
