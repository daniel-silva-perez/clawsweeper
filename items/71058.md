## Summary

Current main still models Microsoft Teams as a single `channels.msteams` config with one bot credential set and default account. The request is a real remaining bundled Teams plugin feature, not implemented, not superseded by the related Feishu multi-account work, and not a ClawHub-only idea because it concerns an existing bundled channel surface.

## Best Possible Solution

Keep the issue open and implement this in the bundled Microsoft Teams plugin. A conservative path is to preserve the current single-object config, add the repo-standard `channels.msteams.accounts.<id>` plus `defaultAccount` shape, resolve credentials/webhook/JWT validation per account, account-scope stored conversation references and proactive sends, and route named Teams bot identities to agents through existing account-aware `bindings`.

## Risks / Open Questions

- A schema-only change would be incomplete; webhook auth, SDK app/token provider selection, conversation-reference storage, proactive sends, probes, Graph calls, docs, and tests all need account-aware handling.
- The proposed `agentBinding` field should be reconciled with OpenClaw's existing `bindings[].match.accountId` routing model to avoid a parallel routing contract.
