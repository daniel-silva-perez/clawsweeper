## Summary

Keep open. Current main still runs Active Memory as a blocking `before_prompt_build` embedded-agent call, and the current changes only prove timeout/fail-open mitigation, not a fast `bootstrapContextMode: "lightweight"` bootstrap path for blocking plugins. The provided v2026.4.24 follow-up reports the symptom still persisting with a 211s blocked reply and no useful model call.

## Best Possible Solution

Keep this issue open as a focused Active Memory embedded-runner cold-path profiling task. Add stage timings around lane wait, runtime plugin load, hook/model/harness/auth/profile resolution, session lock/session-manager setup, context-engine bootstrap/assembly, tool and bundle runtime materialization, and prompt start; then cache or bypass safe per-run invariants for lightweight blocking runs. Preserve Active Memory's hard timeout/fail-open behavior and update the public Active Memory docs with measured timeout and latency guidance after the root cause is fixed.
