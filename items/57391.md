## Summary

Current `main` addresses this class of failure: OpenClaw-owned config writes now validate the full post-change config before committing, and gateway startup restores the last-known-good config when `openclaw.json` is invalid instead of continuing on the broken file. The changelog records this gateway/config crash-loop recovery in `v2026.4.22`.

## Risks / Open Questions

- If there is no previously promoted last-known-good backup yet, startup still fails loudly and relies on manual repair/`openclaw doctor --fix`; the implemented recovery is strongest for existing installs that had a prior valid config.

## Close Comment

Closing as implemented on current `main`.

- Codex review found that gateway startup now restores an invalid `openclaw.json` from the last-known-good backup before continuing, rather than staying on the broken file (`src/gateway/server-startup-config.ts:72`).
- The startup recovery path is covered by tests that assert the restore happens and that the recovery warning is emitted (`src/gateway/server-startup-config.recovery.test.ts:57`).
- OpenClaw-owned config writes now validate the full post-change config before disk commit and reject destructive/invalid writes into `openclaw.json.rejected.*` instead of replacing the active config (`src/config/io.ts:1643`).
- User-facing docs now describe both protections and the recovery flow (`docs/cli/config.md:357`, `docs/gateway/troubleshooting.md:265`).
- The shipped changelog entry for `v2026.4.22` explicitly calls out last-known-good restore preventing gateway crash loops (`CHANGELOG.md:210`).
