## Summary

Keep open. Current main still leaves the reported plugin-scanning performance concern actionable: discovery and manifest snapshot caches default to 1 second, docs expose manual env tuning but no constrained-host preset, and activation still builds discovery/manifest state before scoped plugin filtering. Existing scanner fixes cover narrower node_modules/dist recursion cases such as #46062, but they do not implement production-safe defaults, adaptive cadence, or a performance mode for low-core/multi-channel deployments.

## Best Possible Solution

Keep this open and narrow it into a measurable plugin discovery performance task: profile gateway startup, reload, plugin inventory, and multi-channel activation on low-core ARM-class hardware; then either lengthen/adapt discovery and manifest cache windows, reuse persisted registry state earlier for activation planning, add reload hysteresis where still missing, or introduce a documented constrained-host performance preset with regression coverage.
