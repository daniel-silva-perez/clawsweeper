## Summary

Keep open. Current main has useful scanner mitigations and cache controls, but it does not yet close the reported low-core/default-plugin-scanning problem: discovery and manifest caches still default to 1 second, default discovery still includes the bundled plugin tree, scoped activation still builds discovery/manifest state before plugin-id filtering, and the docs expose manual env tuning rather than a constrained-host/default performance preset.

## Best Possible Solution

Keep this issue open and narrow it into a measurable plugin discovery performance task. The likely path is to profile gateway startup, reload, plugin inventory, and multi-channel activation on low-core ARM-class hardware, then either lengthen or adapt discovery/manifest cache windows, reuse persisted registry/index data earlier where safe, add reload hysteresis where plugin state changes still churn, or introduce a documented constrained-host performance preset with regression coverage.
