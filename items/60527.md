## Summary

Keep #60527 open. Current main fixes the Telegram private-proxy regression path, but it does not fully implement the PR’s broader requested behavior: explicit public proxies are still validated against the target URL `hostnameAllowlist` unless `allowPrivateProxy` is true.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is to either adapt main to build a proxy-specific SSRF policy for explicit-proxy validation that strips only target-scoped `hostnameAllowlist` while preserving private-network checks, including the public-proxy regression test, or explicitly document and close the broader behavior as intentionally out of scope.
