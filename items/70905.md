## Summary

Keep this issue open. Current main still does not expose a stable agent id on `ChannelOutboundContext`; upstream outbound/session paths can carry `agentId`, but the direct channel adapter context drops it before `sendText`, `sendMedia`, `sendPayload`, formatted sends, and presentation rendering.

## Best Possible Solution

Add optional documented `agentId?: string` to `ChannelOutboundContext`; resolve it in outbound delivery from existing session/mirror context; thread it through `ChannelHandlerParams` and `createChannelOutboundContextBase`; verify it reaches `sendText`, `sendMedia`, `sendPayload`, `sendFormattedText`, `sendFormattedMedia`, and `renderPresentation`; and document stable agent attribution separately from display `identity`.

## Risks / Open Questions

- The exact precedence should be specified and tested when both `session.agentId` and `mirror.agentId` are present and differ.
- Because this changes a public plugin SDK type, implementation should include focused contract tests and plugin docs, not only internal plumbing.
- All `deliverOutboundPayloads` call sites should be audited so autonomous, cron, sessions_send/message-tool, inbound reply, media, and no-agent/system cases behave intentionally.
