## Summary

Current `main` does not implement the requested `auth.profiles.<id>.apiKey` SecretRef shape. The schema for `auth.profiles` still accepts only metadata (`provider`, `mode`, `email`, `displayName`). OpenClaw does support SecretRef-backed provider credentials today, but on the per-agent `auth-profiles.json` surface via `keyRef`/`tokenRef`, not via `openclaw.json` `auth.profiles.*.apiKey`.

## Evidence

- **`auth.profiles` schema still rejects `apiKey`:** `src/config/zod-schema.ts` defines each `auth.profiles` entry as a strict object with only `provider`, `mode`, `email`, and `displayName`. There is no `apiKey` or SecretRef field on this config surface.
  - file: [src/config/zod-schema.ts:435](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/src/config/zod-schema.ts#L435)

## Risks / Open Questions

- The report appears to conflate two different surfaces: `openclaw.json` `auth.profiles` metadata versus per-agent `auth-profiles.json` credentials. That may mean the real cleanup is documentation or UX, but the requested config shape is still not implemented.
- There is already adjacent functionality via `keyRef`, so closing this as implemented would be incorrect even though users may have a workable alternative on a different surface.
