## Summary

Keep open. The issue is concrete and still plausible on current main: the Control UI still relies on Lit decorator side-effect registration for `<openclaw-app>`, and I did not find a targeted Chromium/Vivaldi bootstrap fix or regression test that proves the reported custom-element registration failure is resolved.

## Best Possible Solution

Keep the issue open and add a targeted served-Control-UI Chromium regression path that loads `index.html`, asserts `customElements.get('openclaw-app')`, captures module/CSP/service-worker/resource errors, and then hardens the bootstrap path based on the reproduced failure.
