## Summary

Keep open. Current `main` still silently drops non-image Gateway attachments during parsing, the protocol still accepts generic attachments without rejecting them up front, and the Gateway agent/chat paths still only carry `images` onward.

## Evidence

- **Parser still drops non-image attachments:** `parseMessageWithAttachments()` warns and `continue`s when sniffed MIME is not `image/*`, so the attachment is discarded without throwing an RPC error.
  - file: [src/gateway/chat-attachments.ts:347](https://github.com/openclaw/openclaw/blob/b453293349657dedd069d291fa5b078d910067b1/src/gateway/chat-attachments.ts#L347)
  - command: `nl -ba src/gateway/chat-attachments.ts | sed -n '280,390p'`
  - sha: [b45329334965](https://github.com/openclaw/openclaw/commit/b453293349657dedd069d291fa5b078d910067b1)
- **Current tests encode silent-drop behavior:** Gateway attachment tests explicitly expect non-image payloads to be dropped with only a warning log, which matches the reported bug rather than a fix.
  - file: [src/gateway/chat-attachments.test.ts:84](https://github.com/openclaw/openclaw/blob/b453293349657dedd069d291fa5b078d910067b1/src/gateway/chat-attachments.test.ts#L84)
  - command: `nl -ba src/gateway/chat-attachments.test.ts | sed -n '1,320p'`
  - sha: [b45329334965](https://github.com/openclaw/openclaw/commit/b453293349657dedd069d291fa5b078d910067b1)
- **Protocol still accepts arbitrary attachments:** `chat.send`, `sessions.send`, and `agent` schemas all accept `attachments` as `Type.Array(Type.Unknown())`, so non-image attachments are accepted at the wire contract and not rejected before parsing.
  - file: [src/gateway/protocol/schema/logs-chat.ts:35](https://github.com/openclaw/openclaw/blob/b453293349657dedd069d291fa5b078d910067b1/src/gateway/protocol/schema/logs-chat.ts#L35)
  - command: `nl -ba src/gateway/protocol/schema/logs-chat.ts | sed -n '35,60p'; nl -ba src/gateway/protocol/schema/sessions.ts | sed -n '97,105p'; nl -ba src/gateway/protocol/schema/agent.ts | sed -n '131,145p'`
  - sha: [b45329334965](https://github.com/openclaw/openclaw/commit/b453293349657dedd069d291fa5b078d910067b1)
- **Gateway call sites only surface thrown parse errors and only forward images:** `chat.send` and `agent` call `parseMessageWithAttachments()` and only return an RPC error when it throws; dropped non-image attachments do not throw. The `agent` dispatch path forwards `images`/`imageOrder` only, with no document/file channel.
  - file: [src/gateway/server-methods/agent.ts:424](https://github.com/openclaw/openclaw/blob/b453293349657dedd069d291fa5b078d910067b1/src/gateway/server-methods/agent.ts#L424)

  - sha: [b45329334965](https://github.com/openclaw/openclaw/commit/b453293349657dedd069d291fa5b078d910067b1)

## Risks / Open Questions

- Users can get a successful `chat.send` or `agent` response while PDF/DOCX/XLSX attachments are omitted from model input.
- Closing this now would mark an unresolved Gateway contract/runtime gap as fixed even though `main` still has no non-image attachment handoff path.
