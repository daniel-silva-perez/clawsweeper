## Summary

Keep #50760 open. Current main ships the broader background-task ledger, `openclaw tasks` CLI, and chat `/tasks` board, but it still does not implement the PR's requested Control UI Task Board tab/page. The PR is not merge-ready as-is because review comments identify real correctness issues and the current architecture now has a first-class task registry that the UI should probably consume instead of reconstructing state from sessions and cron APIs, but the remaining Control UI capability is still meaningful and unshipped.

## Best Possible Solution

Keep the PR open as a Control UI implementation candidate, but rework it against current main: add or reuse a Gateway/operator-read task endpoint backed by the current task registry if the UI lacks one, render a Control UI Tasks tab from `TaskRecord`/task-status semantics instead of reconstructing from sessions and cron lists, fix the existing review findings, route strings through the Control UI i18n source, and add focused tests for routing, empty/error states, status mapping, cron/session dedupe, and task-registry-backed data.
