## Summary

Keep open. Current main still uses transient auto-fallback selection: auto fallback overrides are persisted only around a fallback attempt and are intentionally cleared on the next turn so the configured primary is retried. The auth cooldown schema and implementation do not expose a cache-aware sticky fallback window, and the provided related #62992 context is narrower OpenRouter backend pinning rather than this cross-provider sticky fallback request.

## Best Possible Solution

Keep the issue open and implement this as core model-failover behavior, not a ClawHub/plugin close. A good fix would add an explicit cache-aware sticky fallback policy, persist auto fallback selection until the configured/cache-derived window expires, then probe the primary once, while preserving existing user override protections. Tests should cover auto-reply model selection, fallback candidate persistence/rollback, auth cooldown interaction, and docs in model failover/prompt caching.
