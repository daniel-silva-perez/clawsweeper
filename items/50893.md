## Summary

The reported WeChat receive failure is owned by the external `@tencent-weixin/openclaw-weixin` plugin, not by OpenClaw core on current `main`. The repository only contains generic channel/plugin infrastructure plus docs and a separate generic restart-loop regression for WeChat sidecars.

## Evidence

- **WeChat is documented as an external plugin:** `docs/channels/wechat.md` explicitly says the WeChat code does not live in the OpenClaw core repo and that WeChat login, Tencent API calls, media handling, and account monitoring are owned by the external plugin.
  - file: [docs/channels/wechat.md:27](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/docs/channels/wechat.md#L27)
  - sha: [ebd7f19a3b20](https://github.com/openclaw/openclaw/commit/ebd7f19a3b20001960474ec9ba9cc424e45f7b23)
- **Community plugins are third-party-owned:** `docs/plugins/community.md` defines community plugins as third-party packages published on ClawHub or npm, with source hosted in a public repo and its own issue tracker.
  - file: [docs/plugins/community.md:9](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/docs/plugins/community.md#L9)
  - sha: [ebd7f19a3b20](https://github.com/openclaw/openclaw/commit/ebd7f19a3b20001960474ec9ba9cc424e45f7b23)
- **No WeChat runtime implementation exists in this repo:** A repo-wide search for `openclaw-weixin|weixin` under `src`, `extensions`, `packages`, and `test` only found docs plus generic tests (`src/channels/registry.helpers.test.ts` and `src/infra/restart-stale-pids.test.ts`), not a WeChat receive-loop implementation that could be fixed in core.
  - command: `rg -n "openclaw-weixin|weixin" src test packages extensions`
  - sha: [ebd7f19a3b20](https://github.com/openclaw/openclaw/commit/ebd7f19a3b20001960474ec9ba9cc424e45f7b23)
- **The only recent core-side WeChat-related fix is unrelated:** `CHANGELOG.md` shows a generic stale-gateway cleanup fix for WeChat sidecars causing restart loops (`#68451`). That is a core process-management regression, not the one-way send/receive behavior reported here.
  - file: [CHANGELOG.md:543](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/CHANGELOG.md#L543)
