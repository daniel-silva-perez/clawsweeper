## Summary

Current `main` at `45146913007da723b4f06aba6a66cf4a695974c0` does not implement this Nextcloud Talk attachment parsing fix. The reported gap still exists: webhook rich-object payloads are still treated as plain text, there is no inbound `mediaUrls` field on the Nextcloud Talk message type, and the inbound handler still drops empty-text messages before any attachment context could be forwarded.

## Evidence

- **Webhook payload is still passed through as plain text:** `payloadToInboundMessage()` still assigns `text` from `payload.object.content || payload.object.name || ""` and does not parse rich-object JSON or extract attachment links.
  - file: [extensions/nextcloud-talk/src/monitor.ts:209](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/extensions/nextcloud-talk/src/monitor.ts#L209)
  - sha: [45146913007d](https://github.com/openclaw/openclaw/commit/45146913007da723b4f06aba6a66cf4a695974c0)
- **Inbound message type has no attachment URL field:** `NextcloudTalkInboundMessage` on `main` still contains `text` and `mediaType` only; there is no `mediaUrls` or similar attachment-link field to carry parsed shares forward.
  - file: [extensions/nextcloud-talk/src/types.ts:153](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/extensions/nextcloud-talk/src/types.ts#L153)
  - sha: [45146913007d](https://github.com/openclaw/openclaw/commit/45146913007da723b4f06aba6a66cf4a695974c0)
- **Media-only inbound messages are still dropped:** `handleNextcloudTalkInbound()` still returns immediately when the trimmed body is empty, so a share that only produces attachment data would not reach agent routing on current `main`.
  - file: [extensions/nextcloud-talk/src/inbound.ts:70](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/extensions/nextcloud-talk/src/inbound.ts#L70)
  - sha: [45146913007d](https://github.com/openclaw/openclaw/commit/45146913007da723b4f06aba6a66cf4a695974c0)
- **Downstream attachment pipeline expects attachment metadata:** The media-understanding path explicitly collects `MediaPaths`, `MediaUrls`, and `MediaTypes`, so the lack of inbound attachment plumbing in the Nextcloud Talk channel is material rather than cosmetic.
  - file: [docs/nodes/media-understanding.md:26](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/docs/nodes/media-understanding.md#L26)
  - sha: [45146913007d](https://github.com/openclaw/openclaw/commit/45146913007da723b4f06aba6a66cf4a695974c0)
- **Attachment classification falls back to MIME hints or URL/file extension:** Attachment normalization first checks `MediaTypes`; without MIME hints it falls back to file extension, which matters for extensionless share URLs like `/f/<id>`.
  - file: [src/media-understanding/attachments.normalize.ts:32](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/src/media-understanding/attachments.normalize.ts#L32)
  - sha: [45146913007d](https://github.com/openclaw/openclaw/commit/45146913007da723b4f06aba6a66cf4a695974c0)

## Risks / Open Questions

- The proposed PR appears directionally useful, but it likely still needs review for downstream attachment typing on extensionless share URLs, since media understanding relies on `MediaTypes` or detectable file extensions.
