## Summary

Keep this PR open. Current main still lacks Nextcloud Talk rich-object parsing and does not thread inbound media URLs, while the PR targets a real bundled-plugin bug linked to #29152. The remaining review comments are reasons to update the PR or land an equivalent maintainer patch, not reasons to close it as obsolete.

## Best Possible Solution

Keep the PR open and ask for an updated patch, or land an equivalent maintainer-owned fix in the bundled `extensions/nextcloud-talk` plugin. The fix should parse signed Nextcloud rich-object payloads in `monitor.ts`, extract safe file links plus MIME hints, thread `mediaUrls`/`MediaTypes` through `types.ts` and `inbound.ts`, allow media-only inbound messages past the empty-body guard, and add focused regression tests for rich file shares, malformed content fallback, placeholder rendering, URL trust, and MIME classification.

## Risks / Open Questions

- Merging the PR as described may still miss image/audio/video classification for extensionless `/f/<id>` share links unless MIME types are propagated through `MediaTypes`.
- Promoting rich-object links into `MediaUrls` should be constrained to authenticated/trusted Nextcloud webhook payloads and aligned with the existing guarded remote media fetch path.
- The rich-object parser should return human-readable `message` text for valid rich objects even when no placeholder substitution occurs.
