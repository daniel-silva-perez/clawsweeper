## Summary

Keep open. Current main still lacks the systemctl spawn environment recovery described by the linked fix attempt for #63561, and the current Linux systemd path still treats `Failed to connect to bus` as user-systemd unavailable. The related PRs/comments are useful evidence, but they do not match current main behavior closely enough to close this issue as implemented.

## Best Possible Solution

Keep #63561 open. The best maintainer path is to implement a focused Linux systemd fix in `src/daemon/systemd.ts`: preserve existing process env, derive `XDG_RUNTIME_DIR=/run/user/$UID` and `DBUS_SESSION_BUS_ADDRESS=unix:path=/run/user/$UID/bus` only when missing and appropriate, use that env for every `systemctl --user` spawn, and add regression tests covering the `Permission denied`/missing-bus-env case from npm/global wrappers. While there, narrow `isSystemctlMissingDetail()` so unit-level `not found` output is not classified as missing `systemctl`.
