## Summary

Keep open. Current main has partial Feishu tool account routing, but it does not implement the requested cross-channel per-agent account binding. Feishu tools can prefer an explicit tool `accountId` or the active source account id when that id is also a configured Feishu account; they do not resolve a Feishu account bound to the current agent when the run originates from another channel such as DingTalk.

## Best Possible Solution

Keep the issue open and implement a first-class per-agent target-channel account resolution path. A conservative direction is to reuse or extend existing `bindings`/routing helpers so plugin tools can resolve the Feishu account for `ctx.agentId` and target channel `feishu`, then make Feishu tools prefer: explicit tool `accountId`, agent-bound Feishu account, valid active Feishu source account, and finally `channels.feishu.defaultAccount`. Add schema/docs and regression coverage for DingTalk-to-Feishu with non-identical source and target account ids.
