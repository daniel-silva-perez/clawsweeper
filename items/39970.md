## Summary

Current main only partially covers this PR. iMessage text chunking is present again, and shared delivery code preserves audioAsVoice for generic plugin adapters, but the bundled iMessage plugin still has no audioAsVoice-specific delivery path: the flag is not accepted by the iMessage outbound runtime, not passed to sendMessageIMessage, and media-only audio still becomes generic <media:audio> placeholder text.

## Best Possible Solution

Keep this PR open and narrow/rebase it onto extensions/imessage. The best fix is to add an iMessage-owned sendPayload or equivalent outbound path that sends audioAsVoice media without the generic audio placeholder, sends visible caption text separately, preserves chunking for text-only payloads, and adds iMessage-specific regression tests around sendMessageIMessage and deliverOutboundPayloads/monitor delivery.
