## Summary

Keep open. Current main has OpenRouter onboarding support and a provider filter for large model catalogs, but it still appears to fall into the setup wizard after detecting an existing config, which matches the reported forced-wizard behavior.

## Best Possible Solution

Fix the installer or Umbrel setup entrypoint so an existing `$OPENCLAW_CONFIG_PATH` or legacy config exits the onboarding path after doctor/dashboard handling instead of continuing to `openclaw onboard`. Add a focused regression test for that existing-config install path with a TTY/openclaw stub, then verify the packaged Umbrel wizard can filter large model catalogs and select OpenRouter.
