## Summary

Keep open. Current main still models `responseUsage` as per-session state only, and new Discord thread sessions do not have a native default or parent-session inheritance path for that setting. The existing Discord thread inheritance/forking code covers routing/transcript context, not the usage-footer preference, and the provided related items do not supersede this request.

## Best Possible Solution

Keep this issue open. The best fix is a small core/session implementation with tests: either add a validated global session default such as `session.defaults.responseUsage`, or copy parent `responseUsage` when a Discord thread session is created with `ParentSessionKey`. Preserve `/usage off|tokens|full` as a per-session override, cover reset/thread behavior in targeted tests, and update the usage and Discord/config docs.
