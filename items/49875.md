## Summary

Keep PR #49875 open. Current main still does not broadcast `runInboundClaim` from the general dispatch path: ordinary channel messages still proceed from `message_received`/internal hooks directly into `markProcessing()` and fast-abort/agent dispatch, while only plugin-owned bindings call `runInboundClaimForPluginOutcome`. The PR is also not safely obsolete or merge-ready because the existing tests still encode the old no-broadcast behavior and the PR review comments identify real ordering/diagnostic gaps that need follow-up.

## Best Possible Solution

Keep the PR open and ask for a revised implementation: add a broadcast `runInboundClaim` path for non-plugin-bound ordinary messages, but place it so fast-abort and plugin-binding fallback semantics remain intact; call `markIdle` on handled early return; update the existing no-broadcast test into the new expected behavior; and add regression tests for claimed regular messages, unclaimed messages, `/stop`/abort precedence, and plugin-bound fallback behavior.
