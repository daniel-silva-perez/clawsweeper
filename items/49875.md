## Summary

Keep open. Current `main` still intentionally does not broadcast `inbound_claim` for ordinary non-plugin-bound dispatch, and the proposed PR would conflict with that behavior, break existing tests, and move plugin claiming ahead of documented fast-abort handling.

## Evidence

- **Current dispatch path has no general inbound-claim broadcast:** After plugin-bound handling and `message_received` hooks, dispatch goes straight to `markProcessing()` and then `tryFastAbortFromMessage()`. There is no `runInboundClaim()` call in the general path on current `main`.
  - file: [src/auto-reply/reply/dispatch-from-config.ts:578](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/src/auto-reply/reply/dispatch-from-config.ts#L578)
  - command: `nl -ba src/auto-reply/reply/dispatch-from-config.ts | sed -n '470,690p'`
  - sha: [45146913007d](https://github.com/openclaw/openclaw/commit/45146913007da723b4f06aba6a66cf4a695974c0)
- **Current tests explicitly lock in no broadcast for unbound conversations:** `dispatch-from-config.test.ts` still asserts that, without a core-owned plugin binding, `runInboundClaim` is not called and normal reply resolution proceeds. That is the exact behavior this PR changes.
  - file: [src/auto-reply/reply/dispatch-from-config.test.ts:2417](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/src/auto-reply/reply/dispatch-from-config.test.ts#L2417)
  - command: `nl -ba src/auto-reply/reply/dispatch-from-config.test.ts | sed -n '2400,2465p'`
  - sha: [45146913007d](https://github.com/openclaw/openclaw/commit/45146913007da723b4f06aba6a66cf4a695974c0)
- **Current tests also require plugin-bound routing/fallback before any generic broadcast:** There is a dedicated test that plugin-owned bindings route to `runInboundClaimForPluginOutcome` before any generic broadcast, and another that falls back to OpenClaw when the bound plugin has no handler while still asserting `runInboundClaim` was not called.
  - file: [src/auto-reply/reply/dispatch-from-config.test.ts:2565](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/src/auto-reply/reply/dispatch-from-config.test.ts#L2565)

  - sha: [45146913007d](https://github.com/openclaw/openclaw/commit/45146913007da723b4f06aba6a66cf4a695974c0)
- **Fast-abort is a documented user path and currently runs before any generic claim hook:** Docs tell users to send standalone messages like `stop` or `abort` to cancel a running task. In current code, `tryFastAbortFromMessage()` runs immediately after `markProcessing()`. The PR patch inserts generic `runInboundClaim()` before that point, so a blanket claimer could intercept abort messages before the built-in cancel path runs.
