## Summary

Keep open. Current main supports SecretRef/SecretInput on several credential surfaces and documents MCP server env SecretInput support, but the top-level `openclaw.json` `env.vars` path remains string-only in schema, public config types, runtime collection, tests, and docs. The requested exec-backed SecretRef workflow would therefore still need a core config/secrets implementation.

## Best Possible Solution

Keep this issue open and implement it as a core config/secrets change: widen `env.vars` and top-level env sugar to `SecretInput`, add `env.vars.*` to runtime SecretRef resolution, apply resolved values through the existing env filters/no-overwrite rules, and update schema docs plus focused tests for env/file/exec refs and unresolved-ref behavior.

## Risks / Open Questions

- Implementing this changes a global env injection path, so it needs careful ordering: resolve SecretRefs before env application without persisting resolved plaintext into config/service metadata.
- Existing guardrails must be preserved: no overriding already-set process env vars, dangerous env var filtering, unresolved `${VAR}` placeholder avoidance, and service/runtime env behavior.
