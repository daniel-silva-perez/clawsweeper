## Summary

Keep open. Current `main` does not implement the PR’s getUpdates-only abort scoping. The Telegram bot fetch wrapper still applies the shared `fetchAbortSignal` to every API request, and the polling session still aborts that shared signal on restart/stop, so outbound `sendMessage`/`sendChatAction` calls remain plausibly abortable during polling recovery.

## Evidence

- **Shared fetch wrapper still aborts all Telegram API requests:** `createTelegramBotCore` wraps the client fetch with a new `AbortController`, forwards `opts.fetchAbortSignal` into that controller for every request, and documents that shutdown aborts in-flight Telegram API requests immediately. There is no current `getUpdates`-only middleware path here.
  - file: [extensions/telegram/src/bot-core.ts:181](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/extensions/telegram/src/bot-core.ts#L181)
  - command: `nl -ba extensions/telegram/src/bot-core.ts | sed -n '150,260p'`
  - sha: [4013c658537e](https://github.com/openclaw/openclaw/commit/4013c658537ee2051c44186f197b0710e84c9049)
- **Polling restart still aborts the shared bot fetch signal:** `TelegramPollingSession` creates one `fetchAbortController` per polling bot, passes its signal as `fetchAbortSignal` into `createTelegramBot`, and calls `fetchAbortController.abort()` when stopping/restarting the runner. Because the bot fetch wrapper is shared, that restart path still reaches non-polling requests too.
