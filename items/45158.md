## Summary

Current `main` appears to implement exact Feishu DM peer routing correctly, but the report does not include the actual `bindings` config needed to distinguish a real Feishu regression from a broader channel/account-only binding configuration that would legitimately collapse to the first matching agent. That is not strong enough evidence to close conservatively.

## Evidence

- **Feishu inbound routing uses the sender Open ID as the direct-peer route key:** The Feishu bot passes `peer: { kind: "direct", id: peerId }` into `resolveAgentRoute`, then resolves configured/runtime conversation bindings against that same conversation id.
  - file: [extensions/feishu/src/bot.ts:590](https://github.com/openclaw/openclaw/blob/4630ce3d9e2f089062ccd386b0ae241325961d1a/extensions/feishu/src/bot.ts#L590)
  - sha: [4630ce3d9e2f](https://github.com/openclaw/openclaw/commit/4630ce3d9e2f089062ccd386b0ae241325961d1a)
- **Route resolution prefers exact peer bindings before broader account/channel fallbacks:** `resolveAgentRoute` checks `binding.peer` first, then parent-peer and wildcard-peer, and only later falls back to `binding.account` and `binding.channel` tiers.
  - file: [src/routing/resolve-route.ts:733](https://github.com/openclaw/openclaw/blob/4630ce3d9e2f089062ccd386b0ae241325961d1a/src/routing/resolve-route.ts#L733)
  - sha: [4630ce3d9e2f](https://github.com/openclaw/openclaw/commit/4630ce3d9e2f089062ccd386b0ae241325961d1a)
- **Tests cover exact peer precedence over broader matches:** The routing tests explicitly assert that an exact direct-peer binding wins over a wildcard/direct fallback.
  - file: [src/routing/resolve-route.test.ts:975](https://github.com/openclaw/openclaw/blob/4630ce3d9e2f089062ccd386b0ae241325961d1a/src/routing/resolve-route.test.ts#L975)
  - sha: [4630ce3d9e2f](https://github.com/openclaw/openclaw/commit/4630ce3d9e2f089062ccd386b0ae241325961d1a)
- **Current docs require peer-scoped Feishu bindings for multi-agent DM routing:** The Feishu docs show multi-agent routing with `match.peer.kind: "direct"` and `match.peer.id: "ou_xxx"`, which matches the current router semantics.
  - file: [docs/channels/feishu.md:329](https://github.com/openclaw/openclaw/blob/4630ce3d9e2f089062ccd386b0ae241325961d1a/docs/channels/feishu.md#L329)
  - sha: [4630ce3d9e2f](https://github.com/openclaw/openclaw/commit/4630ce3d9e2f089062ccd386b0ae241325961d1a)
- **The report never included the actual config snippet requested by maintainers:** The issue body describes the setup but does not show the concrete `bindings` entries, and the only maintainer follow-up asked for the relevant `openclaw.json` sections. Without that, the claimed regression remains ambiguous.

## Risks / Open Questions

- Closing now could discard a real Feishu regression in released versions such as 2026.3.12, especially if the bug only appears with a specific multi-account or legacy binding shape not shown in the issue.
- Current evidence is code-and-doc inspection on `main`, not a reproduction against the reporter's actual configuration.
- If the reporter used channel/account-only bindings rather than `match.peer`, the observed 'all messages go to one agent' behavior would be expected, but that cannot be proven from the issue as written.
