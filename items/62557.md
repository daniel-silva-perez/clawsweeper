## Summary

Keep #62557 open. Current main still routes `openclaw update` gateway restart detection and restart execution through the managed Linux systemd user-service path, while docs explicitly support system-wide units under `/etc/systemd/system`. A system-wide service therefore remains a supported deployment that the updater can miss, leading to the stale running gateway described in the issue.

## Best Possible Solution

Keep the issue open and fix the updater/gateway lifecycle path for supported Linux system units. The likely product path is to reuse or factor the existing deep systemd unit discovery, detect matching system-scope OpenClaw gateway units during update restart, then either restart them with a privilege-aware command path or print an accurate `sudo systemctl restart <unit>` hint. Add focused tests for update restart generation and not-loaded hints covering user units, system units, custom `OPENCLAW_SYSTEMD_UNIT`, and no-service fallback.
