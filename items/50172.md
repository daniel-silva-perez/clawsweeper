## Summary

Keep #50172 open. Current main still lacks BlueBubbles tapback/reaction-based exec approval resolution, and the open PR contains relevant implementation work, but it is not obsolete or safe to auto-close: maintainer review still asks for a GUID-bound approval-message model and stricter approver identity checks before landing.

## Best Possible Solution

Keep the PR open for maintainer-guided rework. The best path is to implement BlueBubbles as a native approval client: add the appropriate BlueBubbles approval capability/config if product-approved, record the delivered approval prompt message GUID to approval ID, resolve only real matching tapback/reaction events from an authorized approver/requester, use the existing operator-approvals gateway client rather than exporting broad `callGateway`, and cover forged text, unmatched GUID, unauthorized sender, mention-gated group, success, denial, and gateway-failure cases.

## Risks / Open Questions

- Merging the PR as-is could preserve the spoofing class called out in review if approval IDs are still derived from quoted/replied text rather than a tracked bot-sent BlueBubbles message GUID.
- The PR's public `callGateway` SDK export would broaden plugin gateway access; current main already has a narrower operator-approvals client that is a better fit for this surface.
- Closing the PR would discard a focused BlueBubbles implementation attempt while the user-visible feature remains missing on current main.
