## Summary

Keep open. Current main moved memory_search into the bundled memory-core plugin, and the current implementation still awaits manager creation and manager.search without a hard tool-boundary timeout. The linked liveness report #49524 remains relevant; this PR needs to be ported/reworked against the current plugin files rather than closed as obsolete.

## Best Possible Solution

Keep the PR open, but rework it against the current memory-core plugin implementation: add a bounded fail-open wrapper around the full memory_search execution path in extensions/memory-core/src/tools.ts, add timeout-specific unavailable metadata in tools.shared.ts, and add a fake-timer regression test in extensions/memory-core/src/tools.test.ts. The implementation should account for current corpus modes and existing embedding timeout defaults before settling on a hard 30s budget.

## Risks / Open Questions

- The PR patch targets the old src/agents/tools/memory-tool.ts location; current main now owns this behavior under extensions/memory-core/src/tools.ts and extensions/memory-core/src/tools.shared.ts, so the fix needs a careful port.
- A fixed 30s wall-clock timeout may conflict with current memory behavior, including local embedding query defaults up to 5 minutes and QMD/wiki/sessions corpus paths; maintainers should choose the timeout budget intentionally.
