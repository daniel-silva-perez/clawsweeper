## Summary

Keep this PR open. Current main still throws plain parameter-validation errors and wraps tool failures into `status: "error"` tool results without preserving any non-retryable metadata. The existing loop protections are generic and either disabled by default or only cap repeated run-loop attempts; they do not implement the targeted `retryable: false` behavior proposed for validation failures. The related #65235 is adjacent UX work, not a replacement for this fix.

## Best Possible Solution

Review this PR as an active focused fix candidate. The best path is to add a regression test showing a read/write/edit parameter validation failure produces non-retryable tool-result metadata and does not trigger repeated model retries, then use a typed validation error or documented adapter seam if the Pi tool-result contract supports it. Keep #65235 open for the separate user-visible chat-surface cleanup.
