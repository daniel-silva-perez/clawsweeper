## Summary

Current `main` does not implement per-spawn named lanes or per-lane concurrency for `sessions_spawn`. The tool schema exposes no `lane` parameter, subagent spawns are still hard-coded onto the single `subagent` global lane, and config/docs still only describe one `agents.defaults.subagents.maxConcurrent` knob.

## Evidence

- **`sessions_spawn` has no `lane` input:** The tool schema lists `task`, `label`, `runtime`, `agentId`, `model`, `thinking`, `runTimeoutSeconds`, `thread`, `mode`, `cleanup`, `sandbox`, `context`, `streamTo`, `lightContext`, and attachments, but no `lane` field, so callers cannot request named concurrency lanes.
  - file: [src/agents/tools/sessions-spawn-tool.ts:100](https://github.com/openclaw/openclaw/blob/754acc4478fe361db1e5303fd60f589d52beea7a/src/agents/tools/sessions-spawn-tool.ts#L100)

  - sha: [754acc4478fe](https://github.com/openclaw/openclaw/commit/754acc4478fe361db1e5303fd60f589d52beea7a)
- **Subagent spawns are still forced onto one lane:** `spawnSubagentDirect` sends the child run to the gateway with `lane: AGENT_LANE_SUBAGENT`, which is the fixed shared subagent lane rather than a caller-selected lane.
