## Summary

Keep open. Current `main` still uses a two-step Discord `thread-create` flow: it creates the thread first, then posts the initial message in a second API call. If that second call fails, the whole action throws, so the tool has no partial-success return path and no dedupe check for an already-created thread. That matches the reported ghost-thread failure mode rather than showing the issue is already fixed.

## Evidence

- **Thread creation still happens before a separate follow-up message send:** `createThreadDiscord` posts `Routes.threads(...)`, stores the returned thread id, then for non-forum threads does a second `POST /channels/{thread.id}/messages`. A failure on the second call would happen after the thread already exists.
  - file: [extensions/discord/src/send.messages.ts:140](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/extensions/discord/src/send.messages.ts#L140)
  - command: `nl -ba extensions/discord/src/send.messages.ts | sed -n '80,150p'`
  - sha: [4013c658537e](https://github.com/openclaw/openclaw/commit/4013c658537ee2051c44186f197b0710e84c9049)
- **Message tool thread-create returns success only if `createThreadDiscord` completes fully:** The Discord action runtime awaits `createThreadDiscord(...)` and only then returns `jsonResult({ ok: true, thread })`. There is no partial-success payload carrying `thread.id` when the follow-up message send fails.
  - file: [extensions/discord/src/actions/runtime.messaging.ts:468](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/extensions/discord/src/actions/runtime.messaging.ts#L468)

There is no companion test for preserving the created thread id when the second call fails.
