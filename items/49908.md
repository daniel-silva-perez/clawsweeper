## Summary

Current main no longer has a separate failing `openclaw logs` connection path: `openclaw logs` now calls the same `logs.tail` Gateway RPC path as `openclaw gateway call logs.tail`, with test coverage for the loopback pairing/scope-upgrade fallback. The reported two-path disagreement does not match current main or the latest release tree for this surface.

## Close Comment

Closing this as a Codex review for `cannot_reproduce` on current OpenClaw. On current `main`, `openclaw logs` no longer has a separate connection path from `openclaw gateway call logs.tail`: `src/cli/logs-cli.ts:62` calls `callGatewayFromCli("logs.tail", ...)`, that helper uses the standard CLI `callGateway()` path in `src/cli/gateway-rpc.runtime.ts:6`, and `openclaw gateway call` uses the same underlying CLI `callGateway()` path in `src/cli/gateway-cli/call.ts:26`. The specific two-path disagreement described in this issue therefore no longer matches the current code. There is also explicit regression coverage for loopback pairing/scope-upgrade fallback in `src/cli/logs-cli.test.ts:136`, and the CLI docs now describe `openclaw logs` as tailing Gateway logs via RPC in `docs/cli/logs.md:1`. If you still hit this on a current build, please open a fresh issue with the exact version, auth mode/token source, and the full `openclaw logs --json` output so maintainers can compare it against the current RPC path.
