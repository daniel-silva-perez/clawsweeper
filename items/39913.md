## Summary

Keep open. The prior Codex issue comment already identified the missing message-tool prefix path, and current `main` still has the same product gap: normal reply routing resolves and prepends `messages.responsePrefix`, while `message(action=send, ...)` forwards the tool text through outbound send/gateway delivery without resolving or applying the response-prefix cascade. The related per-channel prefix request (#2316) is implemented, but that only proves the cascade exists; it is not wired into the message-tool send path.

## Best Possible Solution

Implement the prefix at the shared message-tool outbound boundary before plugin/core/gateway dispatch: resolve `responsePrefix` with `agentId`, `channel`, and `accountId`, normalize the outbound text through the same reply-prefix helper or an equivalent shared helper, preserve empty/media-only semantics, and add regression tests for `runMessageAction`/gateway send showing configured global and channel/account prefixes are applied. If maintainers want per-send suppression, add that as an explicit documented option instead of leaving the paths inconsistent.
