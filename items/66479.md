## Summary

Current main still matches the reported failure mode: the shared frontmatter parser normalizes line endings and immediately checks for `---` without stripping a leading UTF-8 BOM, while skill discovery reads `SKILL.md` through that parser and drops skills whose parsed `name` or `description` is missing.

## Best Possible Solution

Keep this issue open and fix it in the shared frontmatter parser by stripping only a leading UTF-8 BOM before the `---` delimiter check. Add a focused `parseFrontmatterBlock` regression test for BOM-prefixed content, and ideally a skill-loader test proving a BOM-prefixed `SKILL.md` with valid `name` and `description` is still discovered.

## Evidence
