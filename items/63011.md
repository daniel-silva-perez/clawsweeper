## Summary

Current main still does not provide one canonical outbound message.send path that is simultaneously hook-bearing, deliverable, and auditable. Direct CLI sends can be handled by plugin action code before the core delivery path, Telegram's send action calls its provider runtime directly, hook cancellation in core delivery still produces no delivery result, and gateway send still converts that empty result into `No delivery result`. The related implementation PR #63596 is closed unmerged, while #38340 was a closed Discord-specific attempt rather than a shipped canonical CLI/gateway contract.

## Best Possible Solution

Keep this issue open and implement a core canonical outbound send contract. The likely fix is a shared send wrapper or delivery-layer discriminated result that preserves agent/session/request identity and returns either delivered or blocked with a denial reason, then make direct CLI `message.send`, gateway `send`, and plugin-owned send actions consume the same contract or an equivalent plugin API. Add regression coverage for no-grant blocked, granted delivery result, and revoked-grant blocked across direct and gateway entrypoints; #63596 is useful prior art but is not merged.
