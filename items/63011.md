## Summary

Keep open. Current main still has the semantic split reported in #63011: direct `openclaw message send` can be handled by a channel plugin action before the shared core send path, gateway `send` still turns an empty delivery result into `No delivery result`, and `message_sending` cancellation is represented only by skipping payload delivery rather than a structured blocked result. The related PR #63596 was closed unmerged and its closing comment explicitly says the underlying issue remains; open draft PR #63007 is useful related work but intentionally scopes itself to gateway only.

## Best Possible Solution

Keep #63011 open as the canonical product/contract issue. The best path is a fresh PR on current main that either designates gateway `send` as the canonical auditable outbound path or routes direct `message send` through the same status-aware helper. The implementation should add an explicit delivery blocked status/reason from the outbound producer, pass `agentId` and session/request identity into `message_sending`, map blocked outcomes consistently in gateway and CLI JSON/text, preserve real provider delivery results on success, and cover the seams in `src/infra/outbound/deliver.test.ts`, `src/infra/outbound/message.test.ts`, `src/gateway/server-methods/send.test.ts`, and `src/commands/message-format.test.ts`.

## Risks / Open Questions

- A gateway-only fix may be acceptable if maintainers explicitly choose gateway `send` as the canonical auditable path, but direct CLI/plugin send behavior then needs clear documentation or guardrails so the split is not rediscovered.
- Changing direct plugin send routing can affect channel-specific Telegram features and payload shapes; a fix should preserve provider delivery results and plugin-owned send options while adding a shared blocked/allowed contract.
