## Summary

Keep this issue open. Current main added timeout-triggered compaction and explicit timeout payloads, but the external chat fallback path still intentionally collapses mixed fallback exhaustion to the same generic `/new` message when verbose error details are off. That leaves the reported post-compaction timeout plus unavailable fallback chain unresolved rather than implemented or safely superseded.

## Best Possible Solution

Keep this open and fix the external-run failure formatter so fallback exhaustion after successful compaction preserves structured cause context: compaction was attempted/succeeded, the retried model hit the LLM idle timeout, and fallback candidates were unavailable or exhausted. The likely owner surface is `src/auto-reply/reply/agent-runner-execution.ts`, with regression coverage for a post-compaction timeout followed by mixed timeout/billing fallback exhaustion on an external chat channel.
