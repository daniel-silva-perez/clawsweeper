## Summary

Keep open. Current main has not implemented the central Discord transient-network retry behavior: the Discord retry runner still retries only Carbon RateLimitError, and existing Discord send tests explicitly assert that a non-rate-limit network error is not retried. The related #45161 is closed/unmerged in the provided context and does not supersede this PR; #52396 remains the live user problem. The PR still needs maintainer review because the discussion raises real idempotency/duplicate-message concerns for non-idempotent POST sends.

## Best Possible Solution

Keep the PR open, but require an idempotency-aware Discord retry design before merge: add a general transient retry runner for safe/idempotent operations, keep non-idempotent sends on rate-limit or clearly pre-connect-safe retries only, migrate all message/thread/component/sticker/poll/voice/reply/approval POST paths deliberately, and add regression tests for retried transient failures plus non-retried duplicate-risk send failures.
