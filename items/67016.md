## Summary

Keep open. Current main partially addresses Dreaming visibility by showing enabled/idle state, promoted-today and signal counts, next scheduled sweep/phase times, and a Dream Diary reader, but it still does not expose the requested last run timestamp, run result/partial failure state, processed-trace count per run, or diary updated timestamp/workspace status in the Dreaming UI.

## Best Possible Solution

Keep this issue open and implement a memory-core-owned Dreaming run summary. The likely path is to extend doctor.memory.status, or add a sibling Dreaming status method, sourcing managed cron state/run logs plus Dream Diary metadata per workspace. Then render in the Dreams tab: last sweep time, run status such as ok/error/skipped/partial, processed/promoted/failed counts, diary path and updatedAt/workspace status, and next run. Add focused gateway status, UI controller, and UI view tests.

## Risks / Open Questions

- Dreaming already exposes some counts and next-run information, so the follow-up should add only the missing run-observability fields rather than duplicate the whole Cron tab.
- doctor.memory.status currently merges multiple workspaces; per-workspace diary/run metadata needs a clear aggregation rule so operators can see partial failures without losing which workspace failed.
