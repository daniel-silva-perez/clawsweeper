## Summary

Keep open. Current `main` still mutates long non-whitespace tokens in TUI-rendered assistant text by inserting literal spaces, and there is no code-block-specific handling or regression coverage showing this issue is fixed.

## Evidence

- **Long-token sanitizer still inserts spaces:** `normalizeLongTokenForDisplay()` returns `chunkToken(token, MAX_TOKEN_CHARS).join(" ")` for non-copy-sensitive tokens, so long tokens are still rewritten with literal spaces before rendering.
  - file: [src/tui/tui-formatters.ts:94](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/src/tui/tui-formatters.ts#L94)
  - command: `nl -ba src/tui/tui-formatters.ts | sed -n '1,180p'`
  - sha: [4013c658537e](https://github.com/openclaw/openclaw/commit/4013c658537ee2051c44186f197b0710e84c9049)
- **Hyphenated/file-like tokens are not exempt:** `isCopySensitiveToken()` preserves URLs, paths, slash-containing tokens, underscore file-like tokens, and mixed alnum credential-like tokens, but there is no preserve path for hyphen-separated file-like tokens mentioned in the report.
  - file: [src/tui/tui-formatters.ts:61](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/src/tui/tui-formatters.ts#L61)
