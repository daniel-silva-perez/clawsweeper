## Summary

The PR’s premise does not match current `main`: OpenClaw’s `openai-completions` adapter already uses the Chat Completions transport (`/v1/chat/completions`), so changing self-hosted providers to `openai-responses` would switch them to the Responses API (`/v1/responses`) instead of fixing a legacy `/v1/completions` route.

## Close Comment

Closing after a Codex review of current `main` (`ebd7f19a3b20001960474ec9ba9cc424e45f7b23`). The specific bug described here does not reproduce against the current code path: self-hosted setup still intentionally writes `api: "openai-completions"` in `src/plugins/provider-self-hosted-setup.ts:142`, and that adapter goes through `client.chat.completions.create(...)` in `src/agents/openai-transport-stream.ts:1150`, which is the Chat Completions route, not legacy `/v1/completions`. Changing this to `openai-responses` would instead switch these providers onto `client.responses.create(...)` at `src/agents/openai-transport-stream.ts:732` (`/v1/responses`). The current docs and tests also still codify `openai-completions` for vLLM/self-hosted providers (`docs/providers/vllm.md:9`, `src/plugins/provider-self-hosted-setup.test.ts:110`). Given that evidence, I can’t verify the regression this PR claims on current main, so closing as `cannot_reproduce`. If there is a current self-hosted backend that now requires the Responses API, please open a fresh report with the provider name, exact endpoint surface, and a failing request/log from current main.
