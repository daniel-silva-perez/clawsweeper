## Summary

Keep open. Current main still has the reported core behavior: exec stdout/stderr is accumulated into `session.aggregated` and then returned in model-visible tool-result `content` plus `details.aggregated` without a secret-redaction pass. Existing redaction code is scoped to diagnostics/history/support surfaces, not the exec/process tool result path, and the related items do not supersede this output-redaction issue.

## Best Possible Solution

Keep #71211 open as a core security hardening item. Implement a shared exec-output redaction policy before any stdout/stderr reaches tool-result content or details, apply it consistently across foreground exec, partial updates, background process polling/logging, gateway/node execution results, and follow-up summaries, and add focused regression tests with env-style assignments, bearer/JWT/API-key patterns, and known OpenClaw secret formats. Existing diagnostics/history redaction helpers are useful reference points, but this should be a core exec/process tool fix rather than ClawHub/plugin work.

## Risks / Open Questions

- Changing exec output handling affects agent behavior and debugging ergonomics; maintainers should decide whether to redact heuristically, block known secret-file reads, require explicit raw-output opt-in, or combine those approaches.
- The implementation needs to cover all exec output surfaces, not just foreground completion: partial updates, `details.aggregated`, background `process poll/log`, node/gateway returned stdout/stderr, and approval/system-event summaries.
