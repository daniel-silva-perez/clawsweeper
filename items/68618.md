## Summary

Keep this issue open. Current main still has only the existing `resolveSenderNames` boolean for Feishu sender-name lookup; there is no native sender alias/name mapping config, no resolver support for mapped names, and no docs/tests for the requested override or fallback behavior. The related Feishu sender-name/contact-permission reports reinforce that this is still an active bundled Feishu plugin problem, not cleanup noise.

## Best Possible Solution

Keep the issue open and implement the feature in the bundled Feishu plugin, not core. Add a documented top-level and per-account `senderNameMap` or similarly named typed config field, merge account overrides predictably, apply configured aliases for matching `open_id`/`user_id` before contact lookup so mapped users do not require a profile API call, preserve the current API lookup for unmapped senders when `resolveSenderNames` is enabled, and keep raw ID fallback for everything else. Update `extensions/feishu/src/config-schema.ts`, `extensions/feishu/src/bot-sender-name.ts`, Feishu docs/manifest metadata, and focused tests for override, fallback, per-account behavior, cache behavior, and permission-error paths.

## Risks / Open Questions

- No live Feishu API run was performed in this read-only review; the missing config/schema/resolver support is clear from source, docs, and tests.
- A future implementation should keep authorization ID-based. Display-name aliases should affect presentation and agent context only, not `allowFrom` or group sender allowlist decisions.
- This should be coordinated with the still-open Feishu sender lookup work in #13751 and #46805 so alias fallback, contact-permission reduction, and `user_id`/`open_id` lookup order do not diverge.
