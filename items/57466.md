## Summary

Keep open. Current `main` is explicitly built to persist the browser device identity and paired device token across reconnects, but this issue does not include enough reproduction detail to prove the report is obsolete or already fixed with high confidence.

## Evidence

- **Browser device identity is persisted:** `ui/src/ui/device-identity.ts:18,61-106` stores the browser identity in localStorage under `openclaw-device-identity-v1` and reuses the saved keypair/deviceId on later connects instead of generating a new one when the stored record is valid.
  - file: [ui/src/ui/device-identity.ts:18](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/ui/src/ui/device-identity.ts#L18)
  - command: `nl -ba ui/src/ui/device-identity.ts | sed -n '1,220p'`
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
- **Paired device token is also persisted:** `ui/src/ui/device-auth.ts:10-37` stores the device-auth token store in localStorage under `openclaw.device.auth.v1`, so reconnects are intended to survive browser session restarts.
  - file: [ui/src/ui/device-auth.ts:10](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/ui/src/ui/device-auth.ts#L10)
