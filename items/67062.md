## Summary

Keep this issue open. Current main still routes QQ Bot inbound and fallback media downloads through a hardcoded `QQBOT_MEDIA_SSRF_POLICY`; there is no documented or wired `channels.qqbot.mediaAllowHosts` setting. The specific example host `multimedia.nt.qq.com.cn` appears covered by the current `*.qq.com.cn` default, but the requested user-configurable allowlist capability remains missing.

## Best Possible Solution

Implement this in the bundled QQ Bot plugin: add `mediaAllowHosts` to the QQ Bot config type/schema/docs/generated metadata, normalize it with the existing hostname allowlist helpers, and merge it with `QQBOT_MEDIA_SSRF_POLICY.hostnameAllowlist` before inbound and fallback media downloads. Keep the default Tencent/QQ host list conservative, and add focused tests for default hosts, custom host merge, and rejected non-HTTPS/private-network fetches.
