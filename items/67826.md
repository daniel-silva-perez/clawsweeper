## Summary

Keep this PR open. Current main has superseded part of the proposed Telegram setup `allowFrom` change by switching setup to numeric sender IDs only, but the PR’s stated hardening is not simply implemented on main. The runtime target resolver still uses the narrower `string[]` resolver contract and trims each input directly, so the remaining target-resolution hardening would need maintainer review or a rebase rather than conservative closure.

## Best Possible Solution

Keep the PR open for maintainer/contributor follow-up: rebase it onto current main, drop or rewrite the obsolete setup username-resolution pieces around the numeric-only `allowFrom` contract, and decide whether Telegram target resolution should defensively coerce malformed resolver inputs internally while preserving the public `string[]` resolver contract. Add focused regression coverage if that hardening is retained.
