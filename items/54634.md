## Summary

Keep open. Current main has partial mitigations for config clobbers and parsed service environment carryover, but it still does not fully handle the reported update path where the effective HOME changes from a dedicated OpenClaw user to root. The service lookup and generated unit remain HOME/env-root dependent, and there is no preserve/migrate/warn/block guard for cross-HOME config roots.

## Best Possible Solution

Keep this issue open and fix the update service refresh/install path directly. Before regenerating an installed gateway service, compare the previous effective roots (HOME, OPENCLAW_HOME, OPENCLAW_STATE_DIR, OPENCLAW_CONFIG_PATH) with the update invocation roots, discover the existing service even when HOME differs, and then preserve the prior root when safe or emit an explicit migration/warning/block. Add a regression test for a service installed with HOME=/home/openclaw and refreshed by an update process running with HOME=/root.

## Evidence

- **Issue discussion identifies a real HOME-root split:** The report says the service unit changed from HOME=/home/openclaw to HOME=/root and abandoned the populated config under /home/openclaw/.openclaw; the first comment independently attributes the symptom to root/HOME switching.
