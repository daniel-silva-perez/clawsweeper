## Summary

Keep open. Current `main` still refreshes/reinstalls the gateway service from the invoking process environment, and the Linux systemd unit path plus inline `HOME` still come from `env.HOME`. In a sudo/root or changed-HOME update path, that can still fork the service/config/state away from the original user rather than migrating or pinning to the existing unit.

## Evidence

- **Update refresh reuses current process env:** `openclaw update` refreshes the gateway service by running `gateway install --force` with `resolveServiceRefreshEnv(process.env, invocationCwd)`. That helper only normalizes existing `OPENCLAW_HOME` / `OPENCLAW_STATE_DIR` / `OPENCLAW_CONFIG_PATH` values; it does not discover the already-installed service user's home or state dir before reinstalling.
  - file: [src/cli/update-cli/update-command.ts:256](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/cli/update-cli/update-command.ts#L256)
  - command: `nl -ba src/cli/update-cli/update-command.ts | sed -n '250,280p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Forced reinstall depends on current-user service lookup:** `runDaemonInstall` probes `service.isLoaded({ env: process.env })` and only reads/preserves the existing service environment when that lookup succeeds. If the update is running under a different HOME/user context, the old unit can be missed and the reinstall falls back to the current process environment.
  - file: [src/cli/daemon-cli/install.ts:89](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/cli/daemon-cli/install.ts#L89)
  - command: `nl -ba src/cli/daemon-cli/install.ts | sed -n '80,155p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Gateway service HOME still comes from invoking env:** Generated service environment still sets `HOME` directly from `env.HOME`, with no automatic migration to an existing service home/state. So a root/sudo update can still stamp a different HOME into the refreshed unit.
  - file: [src/daemon/service-env.ts:321](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/daemon/service-env.ts#L321)
  - command: `nl -ba src/daemon/service-env.ts | sed -n '318,332p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Systemd unit file path is keyed off HOME while systemctl may target SUDO_USER:** On Linux, the unit file path is resolved from `resolveHomeDir(env)` (`HOME`/`USERPROFILE`), but `execSystemctlUser` switches systemctl operations to `SUDO_USER` machine scope under sudo. That mismatch means a sudo-driven refresh can target one user's systemd manager while resolving the unit file path from another home directory.
  - file: [src/daemon/systemd.ts:47](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/daemon/systemd.ts#L47)
