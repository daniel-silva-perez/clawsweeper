## Summary

Keep this issue open. Current main still treats ACP task completion as complete when the ACP runtime iterator returns, while the bundled ACPX plugin delegates that lifecycle directly to acpx. The synthetic `provider: "openclaw"`, `model: "acp-runtime"`, zero-usage transcript shape reported here is still present and is not guarded against as insufficient completion evidence.

## Best Possible Solution

Keep this issue open. Maintainers should validate the raw Claude SDK/acpx event order for a failing long run, then fix the completion boundary so OpenClaw does not mark ACP child tasks succeeded until an authoritative terminal ACP result is observed. Add regression coverage around text deltas followed by premature completion/idle and ensure the task registry, lifecycle event, and parent wake only fire after true terminal completion.
