## Summary

Keep #72080 open. Current main still treats ACP task completion as a direct consequence of the ACP runtime turn/lifecycle ending, and the task delivery path then emits the exact `Background task done` notification described by the reporter. I found the synthetic `provider: "openclaw"` / `model: "acp-runtime"` / `stopReason: "stop"` transcript shape, but I did not find a current guard that distinguishes zero-usage synthetic ACP transcript turns from real harness completion or independently verifies that the underlying ACP/acpx harness process has finished. The issue is fresh, detailed, externally authored, unprotected by maintainer labels, and not superseded by the related ClawSweeper reports provided.

## Best Possible Solution

Keep this issue open and fix the ACP child task completion contract. The likely fix belongs around the ACP runtime manager/task lifecycle boundary: do not mark parent-owned ACP tasks succeeded solely from a synthetic zero-usage ACP transcript or premature runtime iterator completion; require a real ACP completion signal, reliable process/session status from acpx, or a conservative in-flight/verification state before emitting `Background task done` to `sessions_yield` parents. Add a regression test where an ACP child emits visible text/placeholder completion while the harness remains active, and assert the parent is not woken until true completion or timeout/cancel.
