## Summary

Current `main` already has configurable tool-loop guardrails, but not the requested explicit `maxTurns` / `maxToolCalls` / `toolCallTimeout` agent config or a forced-final-response behavior. The existing feature is close enough to be relevant, but not strong enough for a conservative `implemented_on_main` closure.

## Evidence

- **Existing config is `tools.loopDetection`, not explicit turn/tool-call caps:** `ToolLoopDetectionConfig` exposes `enabled`, `historySize`, `warningThreshold`, `unknownToolThreshold`, `criticalThreshold`, `globalCircuitBreakerThreshold`, and detector toggles. It does not define `maxTurns`, `maxToolCalls`, or `toolCallTimeout`.
  - file: [src/config/types.tools.ts:165](https://github.com/openclaw/openclaw/blob/754acc4478fe361db1e5303fd60f589d52beea7a/src/config/types.tools.ts#L165)
  - sha: [754acc4478fe](https://github.com/openclaw/openclaw/commit/754acc4478fe361db1e5303fd60f589d52beea7a)
- **Docs describe loop detection as a separate safety feature:** The config reference documents `tools.loopDetection` as disabled-by-default safety checks with per-agent overrides at `agents.list[].tools.loopDetection`; it does not document the requested absolute iteration/time limit knobs.
  - file: [docs/gateway/configuration-reference.md:2302](https://github.com/openclaw/openclaw/blob/754acc4478fe361db1e5303fd60f589d52beea7a/docs/gateway/configuration-reference.md#L2302)
  - sha: [754acc4478fe](https://github.com/openclaw/openclaw/commit/754acc4478fe361db1e5303fd60f589d52beea7a)
- **Runtime behavior blocks repeated no-progress patterns rather than enforcing a total turn budget:** `detectToolCallLoop()` blocks unknown-tool repeats, repeated no-progress outcomes, known polling loops, and ping-pong loops. This is pattern-based loop detection, not a general max-turn/max-tool-call budget.
  - file: [src/agents/tool-loop-detection.ts:428](https://github.com/openclaw/openclaw/blob/754acc4478fe361db1e5303fd60f589d52beea7a/src/agents/tool-loop-detection.ts#L428)
  - sha: [754acc4478fe](https://github.com/openclaw/openclaw/commit/754acc4478fe361db1e5303fd60f589d52beea7a)
- **Blocked loops become tool-call errors, not an explicit 'force final response with last tool result' path:** `runBeforeToolCallHook()` returns `blocked: true` for critical loops, and `wrapToolWithBeforeToolCallHook()` turns that into `throw new Error(outcome.reason)`. I did not find code here implementing the issue's proposed 'stop calling tools and output the last tool result' behavior.
  - file: [src/agents/pi-tools.before-tool-call.ts:146](https://github.com/openclaw/openclaw/blob/754acc4478fe361db1e5303fd60f589d52beea7a/src/agents/pi-tools.before-tool-call.ts#L146)
  - sha: [754acc4478fe](https://github.com/openclaw/openclaw/commit/754acc4478fe361db1e5303fd60f589d52beea7a)
- **Tests prove the shipped guardrail blocks repetitive loops:** E2E tests cover blocking known poll loops at the critical threshold and blocking generic repeated no-progress calls at the global circuit breaker threshold. That confirms overlap with the issue's symptom, but still through loop detection rather than explicit `maxTurns`/`maxToolCalls` config.
  - file: [src/agents/pi-tools.before-tool-call.e2e.test.ts:165](https://github.com/openclaw/openclaw/blob/754acc4478fe361db1e5303fd60f589d52beea7a/src/agents/pi-tools.before-tool-call.e2e.test.ts#L165)
  - sha: [754acc4478fe](https://github.com/openclaw/openclaw/commit/754acc4478fe361db1e5303fd60f589d52beea7a)
- **Search found no requested option names on current main:** `rg -n "maxToolCalls|toolCallTimeout" src docs` returned no matches. `rg -n "maxTurns" src/config src/agents docs/gateway docs/tools` only found unrelated uses such as xAI X-search config and tool docs examples, not an agent iteration limit.
  - command: `rg -n "maxToolCalls|toolCallTimeout" src docs ; rg -n "maxTurns" src/config src/agents docs/gateway docs/tools`
