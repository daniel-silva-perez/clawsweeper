## Summary

Keep open. Current `main` still documents a Hetzner Docker flow that binds the gateway to LAN and then jumps straight to launch/access, but it does not add the required `gateway.controlUi.allowedOrigins` or fallback config step. The runtime still enforces that requirement, so the reported docs gap remains valid on `main` rather than already implemented or obsolete.

## Evidence

- **Hetzner guide still uses LAN bind with no control UI origin setup step:** The guide tells users to set `OPENCLAW_GATEWAY_BIND=lan`, runs `gateway --bind ${OPENCLAW_GATEWAY_BIND} --allow-unconfigured`, and the quick path goes directly from `docker compose up -d` to access instructions. There is no intervening `config set gateway.controlUi.*` step.
  - file: [docs/install/hetzner.md:50](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/docs/install/hetzner.md#L50)

  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Shared Docker VM runtime also jumps straight from build to launch:** The shared VM guide used by Hetzner says `docker compose build` then `docker compose up -d openclaw-gateway`, with no required pre-launch/post-build Control UI origin configuration step.
  - file: [docs/install/docker-vm-runtime.md:76](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/docs/install/docker-vm-runtime.md#L76)

- **Gateway runtime still hard-fails exactly as reported:** Current runtime throws when Control UI is enabled on a non-loopback bind and neither `gateway.controlUi.allowedOrigins` nor `gateway.controlUi.dangerouslyAllowHostHeaderOriginFallback` is configured.
  - file: [src/gateway/server-runtime-config.ts:146](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/gateway/server-runtime-config.ts#L146)

That matches the reporter's claimed recovery path and why a crash-looping gateway makes `openclaw-cli` awkward.
