## Summary

Keep open. Current main partially addresses the older Signal audio bug by preserving `MediaPath`/`MediaType`/`MediaTypes`, so the shared media-understanding pass can classify Signal audio after a message is dispatched. It still does not implement the requested Signal audio preflight or channel-agnostic preflight hook: Signal group mention gating runs against text before attachments are fetched, skipped audio-only group messages remain `<media:audio>`, and Signal has no `transcribeFirstAudio` integration comparable to Telegram, Discord, WhatsApp, or Feishu.

## Best Possible Solution

Keep this issue open and implement the remaining work in the bundled Signal channel using the existing shared `transcribeFirstAudio` helper. The best fix is a small shared channel audio-preflight adapter, then a Signal integration that runs only after access control, makes audio transcripts available to group mention and command gating, sets `Transcript` and `MediaTranscribedIndexes` to avoid double STT, preserves the existing metadata path, and adds Signal tests for DMs, require-mention groups, failed transcription fallback, and no double transcription. Documentation should point users to `tools.media.audio` CLI/provider configuration and the existing `{{MediaPath}}` template variables rather than introducing a separate Signal-only shell-command surface unless maintainers explicitly want that API.
