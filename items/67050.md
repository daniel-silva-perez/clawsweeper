## Summary

Keep open. Current main still has the env-plus-network critical scanner behavior and no test-path exemption, so the linked false positive remains unsolved. This PR is a plausible active fix path, but its current implementation would create a name/path-based bypass for all install-time scanner rules and lacks regression coverage.

## Best Possible Solution

Keep this PR open as the canonical fix path for #66840, but require a safer implementation before merge: keep scanning all runtime-capable files, relax only credible test-file false positives or only the blocking decision for those findings, and add regression tests for both the env-plus-fetch test mock false positive and malicious/runtime-reachable test-named files.

## Risks / Open Questions

- Merging the PR as written would weaken the install-time security scanner: a malicious plugin could place dangerous runtime-reachable code in a `*.test.*`, `*.spec.*`, `*.mock.*`, or test directory path to avoid all built-in scanner rules.
- The PR includes no regression tests for the intended false-positive case or for preserving scanner coverage of runtime-capable test-named files.
- Current users still only have the break-glass `--dangerously-force-unsafe-install` workaround for this false positive, which is documented as an unsafe override rather than a normal install path.
