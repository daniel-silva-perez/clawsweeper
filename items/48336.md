## Summary

Keep open. Current `main` does not implement a general per-IP gateway request limiter, so this PR is not obsolete, and the PR as described still appears to have an unresolved reverse-proxy client-IP bug.

## Evidence

- **No general request limiter in current gateway HTTP server:** `createGatewayHttpServer` on current `main` accepts only the auth `rateLimiter` and has no `requestRateLimiter` option or early 429 flood-protection check before request stages.
  - file: [src/gateway/server-http.ts:851](https://github.com/openclaw/openclaw/blob/b453293349657dedd069d291fa5b078d910067b1/src/gateway/server-http.ts#L851)

  - sha: [b45329334965](https://github.com/openclaw/openclaw/commit/b453293349657dedd069d291fa5b078d910067b1)
- **Runtime state does not wire a request limiter:** `createGatewayRuntimeState` passes only `rateLimiter: params.rateLimiter` into `createGatewayHttpServer`; there is no general per-IP request limiter threaded through runtime setup on `main`.
  - file: [src/gateway/server-runtime-state.ts:184](https://github.com/openclaw/openclaw/blob/b453293349657dedd069d291fa5b078d910067b1/src/gateway/server-runtime-state.ts#L184)

  - file: [src/gateway/auth-rate-limit.ts:1](https://github.com/openclaw/openclaw/blob/b453293349657dedd069d291fa5b078d910067b1/src/gateway/auth-rate-limit.ts#L1)

- **Current codebase resolves real client IP through trusted proxies before rate-limit-style decisions:** Existing hook auth throttling resolves client IP with `resolveRequestClientIp(...)` and only falls back to `req.socket?.remoteAddress`, while `resolveRequestClientIp` itself applies `trustedProxies` and `allowRealIpFallback`. That supports the review concern that the PR's proposed `req.socket?.remoteAddress` check would be wrong behind reverse proxies.
  - file: [src/gateway/server-http.ts:502](https://github.com/openclaw/openclaw/blob/b453293349657dedd069d291fa5b078d910067b1/src/gateway/server-http.ts#L502)

- **PR context shows unresolved integration flaw:** The PR patch and review context show the new limiter checking `req.socket?.remoteAddress` before trusted-proxy client-IP resolution. In reverse-proxy deployments that can exempt all traffic when the proxy is loopback, or collapse all users behind one proxy into a shared bucket. That is a real behavior risk, not a stale or already-fixed PR.
