## Summary

Keep open. Current main still has the reported shape: `openclaw agent` waits for the Gateway final response with a client-side timeout, then falls back to embedded mode on any Gateway error using the same session selectors. The Gateway run can still be active with the session write lock, while the embedded run attempts the same session file lock and can hit the reported `session file locked` error. Adjacent stale-lock recovery from #70004 helps dead/orphaned locks, but does not solve live Gateway-vs-embedded contention; related #70026 and #71605 remain open/kept-open context.

## Best Possible Solution

Fix the CLI/Gateway fallback contract instead of relying on stale-lock cleanup: when a Gateway `agent` request times out, either do not auto-run embedded while the Gateway run may still be alive, explicitly abort/wait for the Gateway run and lock release before retrying, or run embedded under a clearly fresh session with user-visible semantics. Add a regression test in `src/commands/agent-via-gateway.test.ts` for timeout fallback/session isolation and preserve the existing stale-lock recovery behavior.
