## Summary

Keep open. Current main still has a real deletion-after-queue window: cron deletion removes the stored job, but timer workers execute due-job snapshots without re-checking the store before side effects. The PR is not ready to close as implemented or superseded because no equivalent fix is present on main, and the PR discussion already identified correctness risks that should be addressed rather than auto-closing the item.

## Best Possible Solution

Keep the PR open, but require a focused revision: revalidate job existence immediately before executing timer-collected due jobs and queued manual runs, handle store reload/read failures as per-job outcomes inside the worker path, avoid unnecessary per-completion forced I/O, and add regression coverage with `maxConcurrentRuns: 1` where a second due job is deleted before its worker starts and no `enqueueSystemEvent` or isolated delivery occurs.
