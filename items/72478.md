## Summary

Keep open. Current main has substantial bundled Google Meet support, including Chrome/Chrome-node/Twilio joins, realtime bridge status fields, calendar lookup, and post-meeting artifact/transcript export. It does not yet satisfy the central request for a reliable supported headless/listen-first Meet participant: the documented join path still depends on Chrome participation plus BlackHole/SoX or Twilio, and the live coverage is artifact/attendance API proof rather than realtime headless join/audio health proof.

## Best Possible Solution

Keep this issue open and treat it as Google Meet plugin product/bug work. The best path is to split/implement the missing pieces in `extensions/google-meet`: a supported headless or listen-first join mode if feasible, clearer identity/calendar-entry semantics, end-to-end audio/transcription health checks that fail before false success, and live/manual validation that proves admission plus actual Meet audio movement. If Google APIs cannot support true headless bidirectional participation, document that boundary and harden the Chrome-node path instead.
