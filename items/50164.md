## Summary

Keep open. Current `main` still routes Feishu non-image message-resource downloads through `type=file` and does not implement the PR's 502 -> `type=media` retry, so the requested fix is not already on `main` or the latest release.

## Evidence

- **Current Feishu download path has no fallback:** `downloadMessageResourceFeishu` makes a single `client.im.messageResource.get` call with `params: { type }` and immediately reads the response; there is no `try/catch` or retry to `type: "media"`.
  - file: [extensions/feishu/src/media.ts:290](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/extensions/feishu/src/media.ts#L290)
  - command: `nl -ba extensions/feishu/src/media.ts | sed -n '276,301p'`
  - sha: [45146913007d](https://github.com/openclaw/openclaw/commit/45146913007da723b4f06aba6a66cf4a695974c0)
- **Video/media messages still map to file downloads:** `toMessageResourceType` still returns `"file"` for every non-image message type, so Feishu video/media fetches still go through the `type=file` path the PR is trying to harden.
  - file: [extensions/feishu/src/bot-content.ts:307](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/extensions/feishu/src/bot-content.ts#L307)

  - sha: [45146913007d](https://github.com/openclaw/openclaw/commit/45146913007da723b4f06aba6a66cf4a695974c0)
- **Tests do not cover the requested 502 fallback:** Current tests still only assert that non-image resources are fetched with `type=file`; there is no regression test for retrying `type=media` after a 502.
  - file: [extensions/feishu/src/media.test.ts:551](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/extensions/feishu/src/media.test.ts#L551)
