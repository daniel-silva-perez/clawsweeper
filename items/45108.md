## Summary

Keep open. Current `main` adds `python3` to `Dockerfile.sandbox`, but the runtime path that auto-creates the default sandbox image still pulls and tags plain `debian:bookworm-slim`, so the reported missing-`python3` failure is not fully resolved. There is also still a hardcoded `python3` invocation in the remote shell fs bridge.

## Evidence

- **Default sandbox Dockerfile now includes python3:** `Dockerfile.sandbox:7-18` installs `python3` in the built `openclaw-sandbox:bookworm-slim` image.
  - file: [Dockerfile.sandbox:17](https://github.com/openclaw/openclaw/blob/4630ce3d9e2f089062ccd386b0ae241325961d1a/Dockerfile.sandbox#L17)

## Risks / Open Questions

- Fresh installs that explicitly run `scripts/sandbox-setup.sh` may appear fixed, which can hide that automatic sandbox image creation still produces a Python-less image.
- Closing as implemented would leave users exposed when the default image is missing or recreated, especially in shared sandbox setups.
- The remote-shell sandbox path still has a direct `python3` dependency, so the broader missing-runtime class of failure is not fully eliminated.
