## Summary

Current `main` already implements remote session reset paths for crash recovery: a gateway RPC (`sessions.reset`) and chat-level `/new`/`/reset` handling that bypasses normal ACP/model dispatch, rotate the session, and archive the prior transcript.

## Close Comment

Closing from a Codex review: current `main` already implements remote session reset support. There is a gateway `sessions.reset` RPC (`src/gateway/server-methods/sessions.ts:1349`) and the reset path rotates the session, cleans up active runtime state, and archives the prior transcript (`src/gateway/session-reset-service.ts:471`). Chat `/new` and `/reset` are also intercepted before normal ACP/model dispatch for bound sessions (`src/auto-reply/reply/commands-reset.ts:121`, `src/auto-reply/reply/dispatch-acp-command-bypass.test.ts:30`). The behavior is covered by tests that verify transcript archiving on reset/new (`src/auto-reply/reply/session.test.ts:2229`). This surface is also present in release `v2026.4.22`. If there is still a remaining recovery gap on a specific channel or deployment path, please open a new issue with the exact surface and reproduction.
