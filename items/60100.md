## Summary

Keep open. Current main has evidence that the direct embedding factory no longer auto-selects another provider when an explicit provider and fallback="none" are passed, and release notes/related PR evidence indicate Ollama adapter registration was recently restored. However, the issue discussion includes a later Apr 18 reproduction on 2026.4.15 claiming the downstream runtime still bypasses the resolved config, and I could not complete a local read-only verification because shell commands fail in this sandbox before executing. Without inspecting the full current request path from resolved settings through resolveMemoryPrimaryProviderRequest and reproducing a current gateway/memory search path, this is not high-confidence enough to close.

## Evidence

- **Issue has a recent post-fix reproduction claim:** Reporter updated the issue on 2026-04-18 saying 2026.4.15 still walks the auto-select priority list despite explicit provider="ollama" and fallback="none", with observed provider leakage to Voyage after Gemini credentials were removed.
- **Current embedding factory does not auto-select for explicit providers:** On current main, createEmbeddingProvider only walks listAutoSelectAdapters when options.provider === "auto"; otherwise it gets the requested adapter and only tries configured fallback when fallback is not "none" and differs from provider.
  - file: [extensions/memory-core/src/memory/embeddings.ts:767](https://github.com/openclaw/openclaw/blob/d6ef1fcf24eea3a5985d653a3baf400e32aa1b75/extensions/memory-core/src/memory/embeddings.ts#L767)
  - sha: [d6ef1fcf24ee](https://github.com/openclaw/openclaw/commit/d6ef1fcf24eea3a5985d653a3baf400e32aa1b75)
- **Manager delegates provider creation through resolved primary request:** MemoryIndexManager.loadProviderResult calls createEmbeddingProvider with config, agentDir, and the result of resolveMemoryPrimaryProviderRequest({ settings }), so the unresolved question is whether that intermediate request preserves the explicit Ollama settings on the affected path.
  - file: [extensions/memory-core/src/memory/manager.ts:2184](https://github.com/openclaw/openclaw/blob/d6ef1fcf24eea3a5985d653a3baf400e32aa1b75/extensions/memory-core/src/memory/manager.ts#L2184)
  - sha: [d6ef1fcf24ee](https://github.com/openclaw/openclaw/commit/d6ef1fcf24eea3a5985d653a3baf400e32aa1b75)
- **Related merged fix restored explicit Ollama adapter support:** Public release/PR notes for PR #66269 state that memory-core restored the built-in Ollama embedding adapter so explicit memorySearch.provider: "ollama" works again, but that fix was scoped to adapter registration and does not by itself disprove the later downstream-bypass report in this issue.
  - file: [extensions/memory-core/src/memory/provider-adapters.ts:1130](https://github.com/openclaw/openclaw/blob/d6ef1fcf24eea3a5985d653a3baf400e32aa1b75/extensions/memory-core/src/memory/provider-adapters.ts#L1130)
