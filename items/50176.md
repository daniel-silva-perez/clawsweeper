## Summary

Keep open. Current `main` still lacks a guard for assistant messages that already exist in `session.messages` when `subscribeEmbeddedPiSession()` attaches, so the reported historical-replay duplicate path is not implemented away on `main`.

## Evidence

- **No preexisting-session filter in subscribe state:** `subscribeEmbeddedPiSession()` initializes per-run state, but there is no snapshot or key set derived from `params.session.messages` that could suppress replayed historical assistant events during initial subscription.
  - file: [src/agents/pi-embedded-subscribe.ts:77](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/src/agents/pi-embedded-subscribe.ts#L77)
  - command: `nl -ba src/agents/pi-embedded-subscribe.ts | sed -n '68,120p'`
  - sha: [45146913007d](https://github.com/openclaw/openclaw/commit/45146913007da723b4f06aba6a66cf4a695974c0)
- **Existing dedupe is scoped to the current assistant message index only:** `shouldSkipAssistantText()` only suppresses text when `lastAssistantTextMessageIndex === assistantMessageIndex`. A replayed historical assistant turn after a fresh `message_start` gets a new message index and is therefore treated as new output, not historical output.
  - file: [src/agents/pi-embedded-subscribe.ts:235](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/src/agents/pi-embedded-subscribe.ts#L235)
  - command: `nl -ba src/agents/pi-embedded-subscribe.ts | sed -n '224,280p'`
  - sha: [45146913007d](https://github.com/openclaw/openclaw/commit/45146913007da723b4f06aba6a66cf4a695974c0)
- **Message handlers still accept any assistant lifecycle event:** `handleMessageStart()` and `handleMessageEnd()` only reject non-assistant messages and transcript-only OpenClaw assistant mirrors. They do not check whether the assistant message already existed in the session before subscription, so replayed historical assistant events still flow through normal delivery paths.

## Risks / Open Questions

- Current `main` can still re-emit an earlier assistant reply if an embedded session implementation replays historical `message_start`/`message_end` events when a new subscriber attaches.
- The PR's proposed object-identity matching may still miss integrations that replay cloned/deserialized message objects rather than the exact `session.messages` references, so the fix likely needs refinement rather than closure.
