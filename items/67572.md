## Summary

Keep open. This contributor PR is paired with open #48123, and current main only partially overlaps its goal: non-image `chat.send` attachments are saved as `media://inbound/*` refs and transcript metadata, but ordinary dispatch still does not populate the `ctx.MediaPaths`/`MediaTypes`/pre-staged workspace contract this PR is trying to add.

## Best Possible Solution

Keep this PR open as the implementation candidate paired with #48123. The best path is a maintainer-reviewed rebase or smaller split that preserves current #70947 WebChat/file-upload behavior, then either implements the missing ordinary `chat.send` `ctx.MediaPaths`/`MediaTypes` workspace staging contract with size/error/security tests, or explicitly decides that the shipped `media://inbound/*` plus PDF/image tools path supersedes workspace staging and closes both paired items together.

## Risks / Open Questions

- The PR broadens data access by persisting arbitrary non-image user payloads and staging them into the agent workspace; even with existing media-store and sandbox guards, that is security-sensitive enough for maintainer review rather than automated cleanup.
- Current main already has overlapping #70947-era media-ref support, so a rebase must reconcile the shipped `media://inbound/*` transcript/tool path with this PR's proposed `ctx.MediaPaths` workspace-staging path.
- Current main still treats non-image offloads as `imageOrder: 'offloaded'`; mixed image/non-image ordering and staging behavior need focused tests if this PR is revived or split.
