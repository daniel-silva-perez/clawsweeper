## Summary

Current main does support WhatsApp group inbound, but the reported config mixes up the two group gates: `channels.whatsapp.groups` is the group-id allowlist, while `channels.whatsapp.groupPolicy` + `groupAllowFrom` authorize senders. The issue’s sample config put a group JID into `groupAllowFrom`, which current code normalizes as sender E.164 entries and therefore rejects before any `web-inbound` log is emitted. The thread also contains a config-only workaround confirmation from multiple users, which matches current docs and tests.

## Close Comment

Closing from a Codex review.

I could not confirm a current-main product bug here. On `main` (`8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf`), WhatsApp group inbound is gated in two separate ways:
- `channels.whatsapp.groups` controls which group IDs are admitted (`docs/channels/whatsapp.md:181`)
- `channels.whatsapp.groupPolicy` + `groupAllowFrom` authorize senders, not group IDs (`docs/channels/whatsapp.md:185`, `extensions/whatsapp/src/inbound-policy.ts:49`)

The sample config in this issue put the group JID into `groupAllowFrom`, so current code would intentionally reject the sender before the `web-inbound` log is emitted. That also explains the “no inbound log entry at all” symptom: access control runs first and returns early on blocked messages (`extensions/whatsapp/src/inbound/monitor.ts:405`), while the inbound log is only written later for admitted messages (`extensions/whatsapp/src/inbound/monitor.ts:552`). Group mention gating is also on by default unless `requireMention` is disabled (`docs/channels/whatsapp.md:199`).

The issue thread also has a config-only resolution report from 2026-04-16, plus a second user confirming it worked, which matches the current code and docs.

If anyone can still reproduce this on `v2026.4.22` or newer with a minimal current config that uses sender numbers in `groupAllowFrom`, the intended `groups` entries, and the exact `requireMention` setting, please open a fresh issue with that config and verbose logs.
