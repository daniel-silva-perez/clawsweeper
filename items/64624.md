## Summary

Keep open. Current main still enqueues WhatsApp connected/disconnected lifecycle notices immediately into the resolved session system-event queue, and I did not find a threshold, verbosity flag, reconnect coalescing, or surface-aware filter. The prior ClawSweeper keep-open comment remains accurate against current main; related #39406 and #41940 cover different warning/health-reporting surfaces and do not supersede this request.

## Best Possible Solution

Keep this issue open and implement a tested channel lifecycle notification policy. A narrow practical path is a WhatsApp/account-level `statusEvents` and/or `notifyAfterMs` setting with reconnect coalescing for outages that recover before the threshold, while keeping serious stop/auth notices visible. If multiple channels need the behavior, expose an additive plugin SDK helper or generic channel lifecycle notification policy and add regression coverage around WhatsApp monitor queueing plus system-event draining.
