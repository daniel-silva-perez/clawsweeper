## Summary

Keep open. Current main still has the observable gap: `sessions_spawn` resolves the target agent and encodes it in `childSessionKey`, and the task schema/SQLite store can persist `agentId`, but the native subagent registration path never passes the resolved agent id into task creation, so `task_runs.agent_id` can still be null. The reporter's `child_session_key` workaround is real but remains a plugin-side parsing workaround, not the requested core attribution field.

## Best Possible Solution

Keep this open and fix the core attribution path. Add `agentId` to `RegisterSubagentRunParams`, pass `targetAgentId` from `spawnSubagentDirect`, forward it to `createRunningTaskRun`, and add a regression test proving a cross-agent `sessions_spawn` creates a task record with `agentId` populated. Review ACP task creation in the same pass so plugins see consistent task attribution across runtimes.

- ACP-backed `sessions_spawn` task creation also appears to omit `agentId` in `src/agents/acp-spawn.ts:1362` and `src/agents/acp-spawn.ts:1394`; a fix should define consistent `agent_id` semantics across `subagent` and `acp` runtimes.
