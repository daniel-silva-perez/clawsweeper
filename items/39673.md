## Summary

Keep #39673 open. Current main has user-bus classification and machine-scope fallback, but direct `openclaw gateway install` can still fail with the same `systemctl --user unavailable: Failed to connect to bus` class when the user systemd scope remains unreachable. The broad machine-scope fallback was already listed in the 2026.3.1 changelog, while this report is from 2026.3.7, so that fallback alone is not strong implemented-on-main evidence.

## Best Possible Solution

Keep #39673 open. The best maintainer path is to make direct `openclaw gateway install` handle unavailable Linux user-systemd consistently: recover when safe by deriving/repairing the user bus environment or targeting the correct user scope, and when recovery is impossible fail with the same explicit headless/Linux remediation used by status and doctor. Add a regression test for direct gateway install where the real install path, not just the preliminary `is-enabled` probe, sees `systemctl --user unavailable: Failed to connect to bus` / No medium.
