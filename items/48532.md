## Summary

Current `main` already mitigates the issue's original blanket-allow repro by storing hashed exact-command durable `allow-always` approvals when no narrow persisted pattern is available. But the full request here is broader: generalized binary+args matching and pattern inference are still not implemented on `main`, and the documented/manual allowlist path remains path-based overall, with `argPattern` enforcement still Windows-only. That is not strong enough evidence to close conservatively.

## Evidence

- **Exact-command durable trust exists on main:** `hasDurableExecApproval` now checks hashed exact-command entries (`=command:<sha>`), and `addDurableCommandApproval` persists that durable trust for `allow-always` decisions.
  - file: [src/infra/exec-approvals.ts:792](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/src/infra/exec-approvals.ts#L792)
- **Allow-always falls back to exact-command persistence:** Node-host execution persists narrowed allowlist patterns first, but if none are derived it stores a durable exact-command approval instead of broadening trust to a binary path.
  - file: [src/node-host/invoke-system-run.ts:626](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/src/node-host/invoke-system-run.ts#L626)
- **Release notes show the phase-1 style fix shipped:** The changelog for `2026.4.5` says exec approvals now reuse durable exact-command `allow-always` approvals in allowlist mode so identical reruns stop prompting.
  - file: [CHANGELOG.md:1344](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/CHANGELOG.md#L1344)
- **Main docs still describe manual allowlists as path-only:** The exec docs still say manual allowlist enforcement matches resolved binary paths only, which does not match the issue's requested generalized binary+args model.
  - file: [docs/tools/exec.md:178](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/docs/tools/exec.md#L178)
- **Arg-pattern matching is not generally implemented:** `matchAllowlist` explicitly documents that `argPattern` matching is currently Windows-only; on other platforms path matches still win regardless of args.
  - file: [src/infra/exec-command-resolution.ts:338](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/src/infra/exec-command-resolution.ts#L338)
- **Advanced docs still say arguments are the operator's responsibility:** The advanced exec-approvals doc says allowlist argument scope is still path-match only and arguments are otherwise the user's responsibility.
  - file: [docs/tools/exec-approvals-advanced.md:105](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/docs/tools/exec-approvals-advanced.md#L105)
