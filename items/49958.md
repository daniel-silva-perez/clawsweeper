## Summary

Do not close. Current `main` does not contain this fix, and the PR as proposed is not strong close evidence because the helper-only change would broaden authorization for unrelated DM flows and leave the root SDK CommonJS shim out of sync.

## Evidence

- **Main still denies the reported case:** `resolveCommandAuthorizedFromAuthorizers()` still returns only `authorizers.some((entry) => entry.configured && entry.allowed)` on the `useAccessGroups=true` path, and the unit test still asserts that `configured:false` with no configured authorizer must return `false`. That means the reported group-chat behavior is not implemented on `main`.
  - file: [src/channels/command-gating.ts:27](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/src/channels/command-gating.ts#L27)
  - command: `nl -ba src/channels/command-gating.ts | sed -n '1,120p'`
  - sha: [45146913007d](https://github.com/openclaw/openclaw/commit/45146913007da723b4f06aba6a66cf4a695974c0)
- **Current DM contract would regress under the PR logic:** The shared DM/group resolver always builds two authorizers from DM and group allowlists, even for DMs. With both lists empty, both entries are `configured:false`; current tests explicitly require `commandAuthorized=false` for open DMs without explicit allowlists. An unconditional `!anyConfigured => true` in the shared helper would flip that behavior.
  - file: [src/security/dm-policy-shared.ts:269](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/src/security/dm-policy-shared.ts#L269)

- **The DM expectation is locked by tests:** `resolveDmGroupAccessWithCommandGate()` is covered by a test that expects open-mode DMs with empty `allowFrom`/`groupAllowFrom` to stay `commandAuthorized=false`. That directly conflicts with the PR's helper-wide fallback.
  - file: [src/security/dm-policy-shared.test.ts:252](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/src/security/dm-policy-shared.test.ts#L252)

A TS-only helper change would leave root-SDK consumers on different authorization semantics than scoped subpath consumers.

## Risks / Open Questions

- The underlying group-chat bug may still be real on `main`, but this PR is not a conservative cleanup candidate because it appears to fix it at the wrong layer.
- Docs describe `commands.useAccessGroups` as enforcing configured allowlists/policies, so the desired no-allowlist group behavior likely needs a more targeted change plus docs/tests alignment, not a global helper fallback.
