## Summary

Keep open. Current `main` still has an unresolved Linux/systemd install mismatch under `sudo`: the unit file path is derived from `HOME`, but the `systemctl` calls are redirected to `SUDO_USER`'s user manager. That can still yield `Failed to enable unit: Unit file openclaw-gateway.service does not exist.` if the unit was written under `/root/.config/systemd/user` while `systemctl --machine <user>@ --user` looks in the invoking user's unit dir. The code also does not implement the issue's requested non-systemd fallback; it only surfaces hints for headless/session problems.

## Evidence

- **Unit path still comes from HOME, not SUDO_USER:** `resolveSystemdUnitPathForName` builds the systemd user-unit path from `resolveHomeDir(env)`, and `resolveHomeDir` only reads `HOME`/`USERPROFILE`.
  - file: [src/daemon/systemd.ts:48](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/daemon/systemd.ts#L48)
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Home resolution ignores sudo caller:** `resolveHomeDir` has no `SUDO_USER`/sudo-home lookup, so `sudo openclaw gateway install` can still stage the unit into root's home.
  - file: [src/daemon/paths.ts:8](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/daemon/paths.ts#L8)
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **systemctl targets the invoking user under sudo:** `execSystemctlUser` switches to `--machine <SUDO_USER>@ --user` when `SUDO_USER` is set, so activation can target a different user manager than the one the unit file was written for.
  - file: [src/daemon/systemd.ts:333](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/daemon/systemd.ts#L333)
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Install writes then enables that resolved unit path:** `writeSystemdUnit` writes the unit to the HOME-derived path, and `installSystemdService` immediately activates it. If HOME and the sudo target user differ, enable/restart can miss the file.
  - file: [src/daemon/systemd.ts:441](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/daemon/systemd.ts#L441)
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Tests cover sudo restart scope but not sudo install path:** There is regression coverage for restart under `SUDO_USER`, but no corresponding install-path assertion proving the unit is written into the sudo caller's home.
  - file: [src/daemon/systemd.test.ts:844](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/daemon/systemd.test.ts#L844)
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
