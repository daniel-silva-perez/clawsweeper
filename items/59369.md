## Summary

Keep open. The report is recent, author is external, no protected labels are present, and the failure is a plausible unresolved Linux systemd-user/D-Bus environment bug rather than the already-fixed `not-found` stdout case. Related discussion shows prior cleanup addressed missing-unit `not-found`, while bus/session failures such as `Failed to connect to bus` still appear in open reports and docs as environment-specific failures needing clearer handling or fallback.

## Evidence

- **Target issue report:** Issue #59369 reports OpenClaw 2026.4.1 on Ubuntu 24.04.3 in a KVM VM failing `openclaw gateway start` during `systemctl is-enabled` with `Failed to connect to bus: 找不到介质`, and the workaround changes the installed service ExecStart from `openclaw gateway start` to `openclaw gateway`.
- **Related fixed scope differs:** PR #33903 / commit 75738c4 was scoped to `systemctl --user is-enabled` returning exit code 4 with `not-found` on stdout for an uninstalled unit. The target issue is a D-Bus/bus access failure, not the missing-unit `not-found` case.
  - file: [src/daemon/systemd.ts](https://github.com/openclaw/openclaw/blob/75738c4/src/daemon/systemd.ts)
  - sha: [75738c4](https://github.com/openclaw/openclaw/commit/75738c4)
- **Adjacent reports remain open:** Issue #44417 remains open for `systemctl --user` detection failing despite manual `systemctl --user` working, and issue #57188 remains open for headless/VPS recovery failing with `systemctl --user unavailable: Failed to connect to bus: No medium found`. These are close enough to the target failure mode that this item should not be auto-closed without stronger current-main proof.
- **Public troubleshooting still frames this as an operational failure mode:** The public Gateway troubleshooting material still documents `openclaw gateway status`, `openclaw status`, `openclaw logs --follow`, `openclaw doctor`, and `openclaw gateway status --deep` as the path for service issues, while third-party troubleshooting around #11805 describes headless Linux D-Bus/user-systemd setup as a continuing deployment condition rather than an automatic OpenClaw fallback.
  - file: [docs/gateway/troubleshooting.md](https://github.com/openclaw/openclaw/blob/d6ef1fcf24eea3a5985d653a3baf400e32aa1b75/docs/gateway/troubleshooting.md)
