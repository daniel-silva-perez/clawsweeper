## Summary

Current `main` does not contain the PR’s DM-topic bound-session guard. The Telegram route resolver still returns only `{ route, configuredBinding, configuredBindingSessionKey }`, and both inbound-message and native-command paths still unconditionally append a DM thread suffix whenever `dmThreadId` is present. That means a runtime-bound DM topic session key can still be rewritten to a derived `:thread:<chatId>:<topicId>` key after routing, so this PR is not already implemented on `main`. Existing tests cover plain bound DMs, bound group topics, and generic DM-topic thread suffixing, but not the bound DM-topic regression this PR addresses.
