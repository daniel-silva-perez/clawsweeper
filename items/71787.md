## Summary

Keep #71787 open. Current main still only handles the narrower Feishu Schema 2.0 `context.open_chat_id` fallback; it does not normalize nested `operator.user_id.{open_id,user_id,union_id}` identity objects, so the PR remains useful and matches maintainer guidance as the better base for #71670.

## Best Possible Solution

Keep #71787 open and land the nested Feishu Schema 2.0 identity normalization on top of current main: extract `operator.user_id.{open_id,user_id,union_id}` and equivalent event-level nested identity fields, preserve the already-shipped `context.open_chat_id` handling and malformed-payload guards, keep the lifecycle regression test for the nested callback shape, and update the changelog attribution without duplicating the narrower #71670 wording.
