## Summary

Keep #51775 open. Current `main` has improved `chat.abort` signal cancellation, including agent RPC registration, but it does not implement the PR’s specific embedded Pi cleanup, queue clearing, or requester sub-agent cascade in the `chat.abort` handler. The PR also has unresolved P1 scope concerns about run-scoped aborts becoming session-wide cleanup and about preserving owner isolation, so it should not be auto-closed as implemented or blindly merged as-is.

## Best Possible Solution

Keep the PR open, but require a scoped revision rather than the current broad cleanup. The best path is to add `chat.abort` cleanup only when it is safe for the target scope: runId cleanup should avoid clearing shared session queues unless that run is the last active run for the session/lane, and sessionKey cleanup should only run when the requester has aborted all active runs whose queues/sub-agents would be affected. Add regression tests for embedded run abort, queue clearing, sub-agent cascade, concurrent same-session runs, and mixed-owner sessionKey aborts.

## Risks / Open Questions

- A naive version of this PR can turn `chat.abort(runId)` into session-wide cleanup and drop unrelated active or queued work sharing the same session.
- A session-scoped cleanup triggered after only some authorized runs were aborted can let one device/user clear another owner’s active run or queues for the same sessionKey.
- Closing this as implemented would hide a remaining Stop-button parity gap: current `chat.abort` signal cancellation does not explicitly clear queues or stop requester sub-agents.
