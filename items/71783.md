## Summary

Keep #71783 open. Current main still has a real unresolved QMD repair gap: different-name collection conflicts are only rebound when `qmd collection list` exposes usable path and pattern metadata. The issue's `life` vs `life-main` log includes the conflicting name in the add error, but current code does not parse that add-error `Name:` field, so the name-only/add-error-only path can still skip creation and leave searches targeting the missing managed collection.

## Best Possible Solution

Keep the issue open and add a narrow memory-core repair: parse conflicting collection names from QMD add errors like `A collection already exists for this path and pattern` plus `Name: life-main (qmd://life-main/)`, then safely rebind that stale collection to the managed configured name when the attempted add path and pattern match. Add regression coverage for `life` versus `life-main`, including the missing-collection search retry path. If automatic removal is considered too aggressive, add a targeted diagnostic and `openclaw doctor` repair instead.
