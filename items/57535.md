## Summary

Keep open. Current `main` still auto-loads skills from multiple local sources, documents those skills as untrusted, and does not show evidence of the requested mandatory untrusted-source confirmation flow or local trust-whitelist UX.

## Evidence

- **Local skill loading is still a first-class path:** The skills docs say OpenClaw loads skills from `skills.load.extraDirs`, `~/.openclaw/skills`, `~/.agents/skills`, `<workspace>/.agents/skills`, and `<workspace>/skills`, with precedence favoring workspace-local copies.
  - file: [docs/tools/skills.md:9](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/docs/tools/skills.md#L9)
  - command: `nl -ba docs/tools/skills.md | sed -n '1,180p'`
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
- **Loader still merges local sources without a trust prompt:** `loadWorkspaceSkillEntries()` loads bundled, extra-dir, managed, personal-agent, project-agent, and workspace skills, then merges them by precedence. This is evidence that local skills are still consumed directly on current `main`.
  - file: [src/agents/skills/workspace.ts:544](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/src/agents/skills/workspace.ts#L544)
  - command: `nl -ba src/agents/skills/workspace.ts | sed -n '540,585p'`

- **Docs warn about untrusted third-party skills, but only as guidance:** Security notes explicitly say to treat third-party skills as untrusted code and read them before enabling, but the doc does not describe any mandatory confirmation or source-verification step for locally loaded skills.
  - file: [docs/tools/skills.md:131](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/docs/tools/skills.md#L131)

That does not cover the reported gap around existing local skill loading paths.
