## Summary

Keep open. Current main partially mitigates the owner escalation by deriving `senderIsOwner` from owner/non-owner loopback bearer tokens, but it does not implement this PR's per-backend token-bound scope model. Session key, account id, and message channel still come from mutable MCP loopback request headers, and the linked security issue #64993 remains open.

## Best Possible Solution

Keep this PR open for maintainer review, or explicitly supersede it with a narrower maintainer-owned patch for #64993. The best code path is to rebase onto current main, preserve the shipped owner/non-owner mitigation, add a per-backend token registry that binds `sessionKey`, `accountId`, `messageProvider`, and `senderIsOwner`, stop emitting scope headers/env vars, invalidate scoped tool-cache entries on token revocation, and update loopback tests so header spoofing cannot alter any scoped field.
