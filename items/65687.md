## Summary

Current main still appears to leave the reported Discord voice Deepgram follow-up failure actionable. The media capability resolver can still return a partial active media-understanding registry before compat loading, Discord voice still uses the shared batch `tools.media.audio` transcription path, and current tests do not cover the partial-active-registry plus configured-Deepgram case from the report.

## Best Possible Solution

Keep this issue open. Fix `src/plugins/capability-provider-runtime.ts` so `mediaUnderstandingProviders` can collect requested provider ids from `tools.media.audio.models` (and shared media model entries that apply to audio), load only the bundled owner plugins needed for missing configured providers such as Deepgram, and merge those providers into a partial active registry without pulling unrelated bundled providers. Add a regression test where the active media registry contains OpenAI/MiniMax, config requests Deepgram under `tools.media.audio.models`, and the resolved provider set includes Deepgram.
