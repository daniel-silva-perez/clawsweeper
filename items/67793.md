## Summary

Keep open. Current main shows `messages.queue.collect` is active-run followup batching, while idle rapid-message batching should come from `messages.inbound.debounceMs`. Feishu has inbound debounce plumbing and partial regression coverage, but the reporter’s recent Feishu DM repro remains unresolved, and related open #54409 still tracks a same-chat Feishu serialization path that can prevent collect/followup batching.

## Best Possible Solution

Keep this issue open. Add a focused Feishu receive-path regression test around `createFeishuMessageReceiveHandler` for three same-sender p2p text events inside `messages.inbound.debounceMs` while the agent is idle. If it passes, add diagnostics/docs clarifying `messages.queue` versus `messages.inbound`; if it fails, fix the Feishu debounce key/text/effective-window path and coordinate with #54409 for the same-chat serialization behavior.
