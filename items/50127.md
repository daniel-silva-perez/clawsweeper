## Summary

Current main already rewrote the Feishu inbound routing path and route-resolution cache so bindings are resolved per channel/account/peer/parentPeer instead of sharing ambiguous state. That matches the reported multi-group misrouting symptom closely enough to close as implemented on main.

## Evidence

- **Feishu inbound routing passes group-specific peer ids into route resolution:** Current Feishu message handling resolves a per-message `peerId`/`parentPeer` and calls `resolveAgentRoute` with that scoped peer, so different groups/topics do not share the same inbound route input.

## Close Comment

Closing as implemented on current `main` after a Codex review. The Feishu inbound path now resolves a scoped group/topic peer before routing (`extensions/feishu/src/bot.ts:576` and `extensions/feishu/src/bot.ts:590`), and the shared route resolver now keys cached results by `channel + accountId + peer + parentPeer + ...` instead of sharing an ambiguous route entry (`src/routing/resolve-route.ts:564` through `src/routing/resolve-route.ts:650`). That means concurrent messages from different Feishu groups no longer hit the same binding-resolution cache key. There is also regression coverage for Feishu group/topic peer routing in `extensions/feishu/src/bot.test.ts:1920` and cache-isolation coverage in `src/routing/resolve-route.test.ts:1064`. Best proof commit in this checkout is `512d3d2287d509f75a32a2344f1b71b8fbfdb0c1`. I could not tie that to a shipped release tag yet, so this looks fixed on `main` but not necessarily in the latest tagged release.
