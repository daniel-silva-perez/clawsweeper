## Summary

Current `main` already implements the reported Discord behavior: bot-authored messages are accepted under `allowBots: "mentions"` when they carry a real mention, including the previously failing case where the gateway payload text contains `<@...>` but the inbound mention metadata is missing and must be hydrated from Discord REST before mention gating runs.

## Evidence

## Close Comment

Codex review: current `main` already implements this behavior. In `extensions/discord/src/monitor/message-handler.preflight.ts:327`, Discord inbound preflight now detects mention syntax in message text and hydrates the full message via REST when the gateway payload is missing populated mention metadata. Then, at `extensions/discord/src/monitor/message-handler.preflight.ts:1016`, `allowBots: "mentions"` only drops bot-authored traffic when the bot still is not mentioned after that processing. The regression coverage in `extensions/discord/src/monitor/message-handler.preflight.test.ts:578` and `extensions/discord/src/monitor/message-handler.preflight.test.ts:598` covers both the normal bot-mention case and the previously failing missing-metadata case.

This change is present on `main` at `81666e586aa6d4305e6472d7075e62f63f701831`. I could not tie it to a tagged release yet, so I’m closing this as implemented on main rather than as a shipped release fix. If you still see a current-build issue specifically where bot-authored Discord messages disappear entirely after a reconnect/restart, please open a fresh issue with reconnect logs, because that would be a narrower gateway transport bug than the mention-gating problem reported here.
