## Summary

Keep open. Current `main` does not implement the PR's two reported fixes, and the proposed approach still appears incomplete against today's gateway connection/security code.

## Evidence

- **Local gateway target is still hardcoded to loopback on main:** `buildGatewayConnectionDetailsWithResolvers()` still builds the local URL as `127.0.0.1` and labels it `local loopback`, regardless of `gateway.bind` mode.
  - file: [src/gateway/connection-details.ts:39](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/src/gateway/connection-details.ts#L39)
  - command: `nl -ba src/gateway/connection-details.ts | sed -n '20,110p'`
  - sha: [ebd7f19a3b20](https://github.com/openclaw/openclaw/commit/ebd7f19a3b20001960474ec9ba9cc424e45f7b23)
- **Server bind resolution can choose a tailnet address:** `resolveGatewayBindHost()` still resolves `gateway.bind="tailnet"` to the primary tailnet IPv4 when available, so a client that always targets loopback can miss the actual listener.
  - file: [src/gateway/net.ts:298](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/src/gateway/net.ts#L298)

  - file: [src/gateway/call.ts:408](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/src/gateway/call.ts#L408)

  - file: [src/gateway/call.test.ts:577](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/src/gateway/call.test.ts#L577)
