## Summary

Keep this PR open. Current main still has the Telegram model-picker fallback bug the PR describes: when no stored override exists, `resolveTelegramSessionState()` still returns `agents.defaults.model` instead of the routed agent's resolved default model, so the picker can mark the global default as current for an agent with its own `agents.list[].model`.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to land an equivalent Telegram fix that computes the routed agent default once inside `resolveTelegramSessionState()`, uses it both for `resolveStoredModelOverride()` and the final no-override fallback, and keeps the contributor's regression test for an agent default that differs from the global default. Then run the Telegram extension test lane and changed gate before merge.
