## Summary

Keep open. Current main still derives lastMessagePreview from any tail transcript entry whose message role is user or assistant, without checking the transcript entry type, so the PR's diagnostic heartbeat preview case is not implemented on main. The related open PR #66544 targets a different displayName/origin-label fallback path and does not supersede this lastMessagePreview change.

## Best Possible Solution

Keep this PR open for maintainer review. The likely best code path is to update readLastMessagePreviewFromOpenTranscript with focused regression coverage so non-user-facing diagnostic transcript entries cannot become lastMessagePreview, while preserving meaningful heartbeat alert output and coordinating separately with #66544 for displayName leakage.
