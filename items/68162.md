## Summary

Keep this issue open. Current main's intended Control UI reconnect path does not obviously call `/new` or `sessions.reset`, but the issue has recent concrete evidence of `action=new` and transcript `.reset.*` rotation on supported releases, and I did not find a targeted fix or regression test proving reconnect/page refresh preserves the same gateway session id.

## Best Possible Solution

Add a focused regression for Control UI WebSocket reconnect/page refresh: seed an existing webchat session and transcript, connect as `openclaw-control-ui`, force close/reconnect with the same settings, then call `chat.history` and a normal `chat.send`. Assert the same `sessionId` is retained, no `sessions.reset` or command hook with `action=new` fires, and no `.jsonl.reset.*` archive is created. If it fails, fix the frontend session-selection/reconnect path or the gateway reset trigger; if it passes, document the exact distinction from explicit New Session/reset and then reconsider closure.
