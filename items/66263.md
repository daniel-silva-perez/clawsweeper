## Summary

Keep this issue open. Current main still does not meet the focused Codex native-compaction acceptance criteria: the native compaction path does not parse or return a post-compaction token count, and session/status fallback can still promote transcript-derived pre-compaction assistant usage to a fresh current total after the cached store total is cleared. The newer compaction-checkpoint machinery is useful for other compaction flows, but it is not wired into transcript usage fallback and the Codex harness compaction path can return before that checkpoint layer runs.

## Best Possible Solution

Keep this issue open and implement the native Codex compaction accounting boundary end to end. The best fix is to parse and propagate a reliable post-compaction token snapshot if the app-server exposes one; otherwise persist an explicit unknown/stale post-compaction state, add or reuse a compaction checkpoint boundary that transcript fallback actually honors, and add regression coverage for known tokensAfter, missing tokensAfter, and status/session fallback after native Codex compaction. The #64669 cumulative-token projection fix should remain as the separate already-merged narrower guardrail.
