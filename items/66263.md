## Summary

Keep open. Current main contains the related #64669 cumulative-token projection fix, and manual compaction clears cached totals when `tokensAfter` is missing, but the focused Codex native-compaction acceptance criteria are still not met: native compaction does not capture a post-compaction token count, and transcript fallback can still re-promote old assistant usage as fresh after the store total is cleared.

## Best Possible Solution

Keep this issue open and fix the Codex native compaction boundary end to end: parse and propagate `tokensAfter` when the app-server provides it; when unavailable, persist an explicit unknown/stale post-compaction state; add a compaction marker or checkpoint boundary that prevents transcript fallback from using pre-compaction assistant usage as fresh context; and add regression tests for known `tokensAfter`, missing `tokensAfter`, and session/status fallback after native Codex compaction.
