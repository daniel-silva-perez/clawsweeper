## Summary

Keep open. Current `main` still runs the secrets-in-config audit against the resolved config, while `${VAR}` substitution is applied during config read before those values reach the collector. The reported false positive remains valid on `ecac696643c604c6d854341bfe6424bfa31423e9`.

## Evidence

- **Audit still passes only resolved config into the secrets-in-config collector:** `runSecurityAudit()` calls `collectGatewayConfigFindings(cfg, context.sourceConfig, env)` but still calls `collectSecretsInConfigFindings(cfg)` with only the resolved runtime config.
  - file: [src/security/audit.ts:978](https://github.com/openclaw/openclaw/blob/ecac696643c604c6d854341bfe6424bfa31423e9/src/security/audit.ts#L978)
  - command: `nl -ba src/security/audit.ts | sed -n '950,995p'`
  - sha: [ecac696643c6](https://github.com/openclaw/openclaw/commit/ecac696643c604c6d854341bfe6424bfa31423e9)
- **Collector suppresses findings only when the inspected value literally looks like `${VAR}`:** `collectSecretsInConfigFindings()` checks `gateway.auth.password` and `hooks.token` with `looksLikeEnvRef()`, but it only receives `cfg`, so once substitution has happened the suppression path cannot fire.
  - file: [src/security/audit-extra.sync.ts:484](https://github.com/openclaw/openclaw/blob/ecac696643c604c6d854341bfe6424bfa31423e9/src/security/audit-extra.sync.ts#L484)
  - command: `nl -ba src/security/audit-extra.sync.ts | sed -n '484,510p'`
  - sha: [ecac696643c6](https://github.com/openclaw/openclaw/commit/ecac696643c604c6d854341bfe6424bfa31423e9)
- **Config read path resolves `${VAR}` before the audit sees those values:** `resolveConfigForRead()` calls `resolveConfigEnvVars(...)`, and the snapshot builder stores `sourceConfig` from `effectiveConfigRaw` after that substitution step. ## Risks / Open Questions

- Closing this would discard a still-valid bug: the false positive is statically evident on current `main` without needing historical reconstruction.
- A narrow fix that only threads current `sourceConfig` into `collectSecretsInConfigFindings()` is unlikely to be sufficient on current `main`, because `sourceConfig` is already post-substitution on the read path.
