## Summary

Keep open. Current main still wires the host read tool through upstream createReadTool with workspace-only enforcement only when tools.fs.workspaceOnly is explicitly true, while the related security issue remains open. This PR is a plausible implementation candidate, but it is XL, bundles unrelated security-sensitive changes, and still needs maintainer/security review rather than cleanup closure.

## Best Possible Solution

Keep this PR open for maintainer/security review, but ask for a narrower branch. The best path is to split unrelated changes, decide whether read-tool isolation should be default-on or explicitly opt-out, then land focused changes in src/agents/pi-tools.read.ts and src/agents/pi-tools.ts with regression coverage for cross-agent memory reads, absolute paths, parent traversal, symlink escapes, and explicit workspaceOnly=false behavior.
