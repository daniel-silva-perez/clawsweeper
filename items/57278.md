## Summary

Current `main` still appears to have the reported Telegram reply-media bug. The inbound handler skips self-authored top-level messages, but `resolveReplyMediaForMessage` still downloads media from `reply_to_message` without checking whether that replied-to message was authored by the bot.

## Evidence

- **Reply media path lacks self-authored guard:** `resolveReplyMediaForMessage` returns reply media whenever `reply_to_message` has inbound media and a file ID. There is no check against `ctx.me.id` or other bot-authored filtering before `resolveMedia()` is called.
  - file: [extensions/telegram/src/bot-handlers.runtime.ts:489](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/extensions/telegram/src/bot-handlers.runtime.ts#L489)
  - command: `nl -ba extensions/telegram/src/bot-handlers.runtime.ts | sed -n '489,515p'`
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
- **Top-level self-authored filtering exists only on inbound message entry:** The normal `message` handler explicitly returns early when `normalizedMsg.from.id === ctx.me.id`, which matches the issue description that self-authored filtering exists at the top level but not in reply-media resolution.
  - file: [extensions/telegram/src/bot-handlers.runtime.ts:1953](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/extensions/telegram/src/bot-handlers.runtime.ts#L1953)

  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
- **Current tests still assert reply-photo download/injection:** The Telegram tests cover fetching reply media and expect `ReplyToBody` to become `<media:image>` with a download of `reply-photo-1`. I did not find a compensating test that skips reply media when the replied-to message is bot-authored.
