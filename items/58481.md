## Summary

Keep open. Current main has WhatsApp reconnect/watchdog recovery and Baileys application-level ping behavior, but it does not enable TCP keepalive on the underlying WhatsApp WebSocket socket. The report is recent, has concrete WSL2 details plus a same-problem comment, and the related #61788 is still open rather than a shipped canonical fix.

## Best Possible Solution

Keep this issue open and implement the mitigation in the WhatsApp plugin socket setup, likely near `extensions/whatsapp/src/session.ts`. The best path is to enable TCP keepalive on the underlying Node socket after the Baileys/ws connection is established, keep core channel code generic, and add a unit test with a mocked Baileys/WebSocket socket proving `setKeepAlive(true, 15000)` or the chosen interval is applied without breaking proxy-agent behavior.

## Evidence

- **WhatsApp socket setup lacks TCP keepalive:** `createWaSocket` builds the Baileys socket with auth, version, logger, browser metadata, `markOnlineOnConnect: false`, `agent`, and `fetchAgent`; there is no `setKeepAlive`, TCP keepalive helper, or `keepAliveIntervalMs`/raw socket hook in this setup. A targeted `rg` for `setKeepAlive`, `TCP`, and keepalive terms found no WhatsApp TCP keepalive implementation.
  - file: [extensions/whatsapp/src/session.ts:149](https://github.com/openclaw/openclaw/blob/7c0fdae9b95bfcc667af3ad341b63ecc194464e4/extensions/whatsapp/src/session.ts#L149)
  - command: `nl -ba extensions/whatsapp/src/session.ts | sed -n '1,340p'; rg -n "setKeepAlive|keepAlive|keepalive|TCP|socket\\.set|underlying socket" extensions/whatsapp src/channels src/plugin-sdk patches package.json pnpm-lock.yaml -S`
  - sha: [7c0fdae9b95b](https://github.com/openclaw/openclaw/commit/7c0fdae9b95bfcc667af3ad341b63ecc194464e4)
- **Current watchdog is post-drop recovery, not NAT prevention:** The controller starts heartbeat/watchdog timers and listens for WebSocket `frame` activity; when transport or application activity is stale it force-closes with `watchdog-timeout` so the reconnect loop can replace the connection. That detects/reconnects after loss, but does not keep the TCP NAT mapping alive.
  - file: [extensions/whatsapp/src/connection-controller.ts:571](https://github.com/openclaw/openclaw/blob/7c0fdae9b95bfcc667af3ad341b63ecc194464e4/extensions/whatsapp/src/connection-controller.ts#L571)

It does not document an implemented WSL2/TCP keepalive mitigation.
