## Summary

Keep this PR open. Current main still has the Google Chat automatic-threading path the PR is trying to fix: the monitor copies inbound `message.thread?.name` into reply context and sends the default typing placeholder into that thread before the shared `replyToMode` filter can suppress final payload threading. The PR is useful and includes live verification, but it should not be closed or merged as-is because its delivery-time guards would also suppress explicit `[[reply_to_*]]` overrides that the shared reply-threading contract intentionally preserves when `replyToMode` is `off`.

## Best Possible Solution

Keep this PR open and request a small revision: suppress implicit Google Chat inbound thread propagation for `replyToMode: "off"` at the monitor/typing boundary, keep the delivery path honoring already-filtered `payload.replyToId` so explicit reply directives still work, and add a focused Google Chat monitor/reply-delivery regression test for `replyToMode: "off"`.
