## Summary

Keep this PR open. Current main still propagates Google Chat inbound thread IDs into the reply context and typing placeholder, so the reported `replyToMode: "off"` behavior is not fixed on main. The PR is useful and includes live Google Chat verification, but it should be revised before merge because its delivery-time guards would suppress explicit reply directives that the shared reply-threading contract preserves.

## Best Possible Solution

Keep this PR open and request a focused revision: suppress implicit Google Chat inbound thread propagation for `replyToMode: "off"` at the monitor context and typing-placeholder boundary, leave final delivery honoring the already-filtered `payload.replyToId` so explicit reply tags still work, and add a Google Chat monitor/reply-delivery regression test for the off-mode DM path.
