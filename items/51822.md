## Summary

Keep PR #51822 open. Current main still does not reject embedded credentials in cron webhook URLs; the shared normalizer only validates string shape and http(s) protocol, and the cron create/update validation depends on that helper. The PR addresses a real hardening gap, but it should not be auto-closed or merged as-is because review comments identify current adjacent paths that need coordinated handling: raw invalid-delivery logging, legacy notify fallback warnings, doctor migration behavior, and regression tests.

## Best Possible Solution

Keep the PR open but revise it into a coordinated cron hardening change: reject `URL.username` or `URL.password` in `normalizeHttpWebhookUrl`, add focused tests for primary webhook delivery and failure destinations, redact invalid webhook targets in all warnings, make legacy `notify`/`cron.webhook` and `doctor --fix` paths warn or skip safely instead of silently creating invalid jobs, and decide whether empty-authority `http://:@host` should also be rejected.

## Risks / Open Questions

- A normalizer-only change can break existing stored credentialed webhook jobs without a clear migration path or warning on the legacy `notify` fallback path.
- The current PR shape may introduce secret exposure in the invalid per-job webhook warning unless `deliveryTo` is redacted before rejection is shipped.
- Without regression tests for primary delivery, failure destination, legacy fallback, and doctor migration, this hardening is easy to regress or ship with inconsistent behavior.
