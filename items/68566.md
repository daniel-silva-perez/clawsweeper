## Summary

Keep open. Current main has not implemented this auth-profile cooldown change, but the PR is not ready to close as implemented or merge as-is: its raw HTTP 400 bypass conflicts with the current classifier contract that 400 responses can still be billing, auth, rate-limit, model-not-found, or context-overflow failures based on body text. The useful direction is a narrower, reason-scoped or call-site-scoped fix with runner coverage.

## Best Possible Solution

Keep this PR open for revision rather than closing it as implemented or duplicate. The best fix should preserve current 400 body classification, skip profile penalties only for canonical `format` failures or clearly helper-local/call-site-scoped malformed-payload failures, and add regression coverage at the embedded-runner path. If the remaining user problem is helper-local poisoning, follow the #71709 direction by using `authProfileFailurePolicy: "local"` or an equivalent scoped policy instead of changing all HTTP 400 auth-profile behavior. The PR should also address assistant-side call sites, avoid free-form first-match status parsing, and clean up the changelog attribution/placement before maintainer review.

## Risks / Open Questions

- The PR's unconditional raw HTTP 400 bypass can prevent real `billing`, `auth`, or `rate_limit` failures from cooling down or disabling bad profiles, causing repeated retries instead of rotating away from unhealthy credentials.
- The status extraction uses first-match regex semantics on free-form error text; a non-status number before the actual HTTP status can produce the wrong cooldown decision.
- The PR's tests cover `markAuthProfileFailure` directly but not the runner extraction/plumbing path or assistant-side failover paths, so the behavioral fix is not locked at the surface where the bug occurs.
- The changelog entry is inserted without the required contributor attribution and was flagged by review for section placement.
