## Summary

Keep open. Current `main` does not support the issue's strongest security claim that plain-text simulated tool calls bypass the real tool pipeline: OpenClaw builds executable tool calls from structured `tool_calls`/`toolCall` data and wraps real executions with `before_tool_call`. But the underlying report is still live because `main` also does not implement the requested retry/forced-tool enforcement when an OpenAI-compatible proxy/local model answers in plain text instead of choosing a tool call.

## Evidence

- **OpenAI-compatible transport only materializes structured tool calls:** `openai-transport-stream` appends normal assistant text from `choice.delta.content`, and only creates executable `toolCall` blocks when `choice.delta.tool_calls` is present.
  - file: [src/agents/openai-transport-stream.ts:1289](https://github.com/openclaw/openclaw/blob/4630ce3d9e2f089062ccd386b0ae241325961d1a/src/agents/openai-transport-stream.ts#L1289)
- **Assistant tool-call extraction ignores plain text:** `extractToolCallsFromAssistant()` only returns blocks from assistant `content[]` whose typed block is `toolCall`/`toolUse`/`functionCall` and has an `id`; it does not parse free-form assistant text or JSON snippets.
  - file: [src/agents/tool-call-id.ts:63](https://github.com/openclaw/openclaw/blob/4630ce3d9e2f089062ccd386b0ae241325961d1a/src/agents/tool-call-id.ts#L63)
- **Actual tool execution still goes through `before_tool_call`:** `wrapToolWithBeforeToolCallHook()` calls `runBeforeToolCallHook()` before invoking the tool's `execute()` function, so real executions do pass the hook/policy path.
  - file: [src/agents/pi-tools.before-tool-call.ts:385](https://github.com/openclaw/openclaw/blob/4630ce3d9e2f089062ccd386b0ae241325961d1a/src/agents/pi-tools.before-tool-call.ts#L385)
- **Downgraded/plain-text tool-call scaffolding is sanitized from visible output:** Streaming reply assembly strips downgraded tool-call text before emitting user-visible chunks, and there is coverage asserting `[Tool Call: ...]` / `[Historical context: ...]` blocks are removed while surrounding text is preserved.
  - file: [src/agents/pi-embedded-subscribe.ts:616](https://github.com/openclaw/openclaw/blob/4630ce3d9e2f089062ccd386b0ae241325961d1a/src/agents/pi-embedded-subscribe.ts#L616)
- **Sanitizer behavior is covered by tests:** `stripDowngradedToolCallText` tests explicitly remove downgraded tool-call marker blocks instead of treating them as executable calls.
  - file: [src/agents/pi-embedded-utils.test.ts:666](https://github.com/openclaw/openclaw/blob/4630ce3d9e2f089062ccd386b0ae241325961d1a/src/agents/pi-embedded-utils.test.ts#L666)
- **Requested enforcement is still not present as a generic agent-loop feature:** Searches for forced/required `tool_choice` handling in the agent path did not find a generic retry/enforcement layer; the explicit `tool_choice=required` handling I found is in the OpenResponses HTTP surface, not the main embedded agent loop.
  - file: [src/gateway/openresponses-http.ts:290](https://github.com/openclaw/openclaw/blob/4630ce3d9e2f089062ccd386b0ae241325961d1a/src/gateway/openresponses-http.ts#L290)
  - command: `rg -n "tool_choice.*required|required.*tool_choice|force.*tool_choice|retry.*tool_choice" src/agents src/gateway --glob '!**/*.test.ts'`

## Risks / Open Questions

- The issue comments overstate the current-main behavior: I found no evidence that plain-text simulated tool calls are executed as real tool invocations on `main`.
- I did not run a live reproduction against LiteLLM/vLLM because this review was constrained to read-only inspection and the local environment does not expose that stack.
- The remaining open part is product behavior: OpenClaw still appears to accept a plain-text assistant answer when the model declines to emit a tool call, rather than generically retrying with forced tool choice.
