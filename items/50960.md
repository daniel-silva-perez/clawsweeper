## Summary

Keep #50960 open. Current main still blocks sensitive-pattern keys from `skills.entries.*.env` unless the skill metadata declares them as `primaryEnv` or `requires.env`, so the linked #31583 bug is not implemented on main. The PR is a plausible focused fix, but it should be reworked before merge because the provided review comments identify real risks: env overrides are applied to the full loaded skill list before prompt eligibility filtering, and duplicate sensitive env keys still use first-wins process-global state.

## Best Possible Solution

Keep the PR open but require a narrower implementation: allow explicitly configured sensitive keys only for the skills that are actually eligible/selected for the run or snapshot, preserve the dangerous host-env blocklist, define duplicate-key semantics for per-skill env collisions, and add regression tests for custom sensitive env keys, disabled/ineligible skills, and duplicate sensitive keys. The linked issue #31583 should remain the canonical bug until that reworked fix lands.

## Risks / Open Questions

- The PR's core change may broaden sensitive env injection before eligibility filtering, so disabled, allowlist-blocked, or otherwise non-included skills need explicit negative tests before merge.
- `activeSkillEnvEntries` is process-global and first-wins for overlapping keys; allowing arbitrary `*_TOKEN` or `*_PASSWORD` keys makes duplicate-key behavior more security-sensitive and needs a product/code decision.
- The linked canonical bug #31583 is still open in the provided context, so closing this PR now would lose a useful implementation attempt while the user-visible regression remains.
