## Summary

Current main partially covers the request: per-agent `agents.list[].tools.deny` exists and deny rules win during tool filtering. The fine-grained path scoping requested here is still missing: the config surface only exposes `tools.fs.workspaceOnly`, not per-agent allow/deny glob lists, and host write/edit behavior remains intentionally unrestricted when workspace-only mode is unset or false.

## Best Possible Solution

Keep this issue open. The core path should retain the existing per-agent `tools.deny` behavior for OpenClaw tool IDs, then add a backwards-compatible, validated per-agent filesystem path policy with canonical path/glob matching and tests across `write`, `edit`, `read` if included, `apply_patch`, sandbox bridges, and any designed exec/bash write sinks. The command-ban examples should either map to a documented exec approval/allowlist model or become a separate command-policy feature rather than overloading tool names.
