## Summary

Keep open. Current main does not implement this PR’s exact defensive behavior: there is no `arg_value` suffix sanitizer in source/docs/tests, exec still forwards `params.command` unchanged, and read/write/edit validation does not normalize these malformed string values. The PR is also stale against current main and has a P1 review concern about record-wide string mutation corrupting legitimate write/edit XML content, so it needs a current-main rewrite rather than auto-close.

## Best Possible Solution

Keep this open or ask for a refreshed patch against current main. The right fix should be narrow and tested: strip the malformed `</arg_value>` suffix only from fields where provider XML leakage is known to corrupt tool dispatch, especially read path aliases and exec command/path-like controls, while explicitly preserving legitimate write/edit payload text such as `content`, `oldText`, and `newText`. Add regression coverage for #48780 plus a negative test proving valid XML content is not truncated.
