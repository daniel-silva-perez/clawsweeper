## Summary

Keep #71136 open. Current main still gives agent-level `contextTokens` precedence over provider model metadata and discovery, so the requested conservative minimum behavior is not implemented. The existing per-model `contextTokens` runtime cap is documented and tested, but it is still a lower-priority value when an agent/default override is present.

## Best Possible Solution

Keep the issue open and implement a shared positive-integer token-limit resolver that collects agent default/per-agent config, provider model `contextTokens` or `contextWindow`, discovered cache values, and `DEFAULT_CONTEXT_TOKENS`, then returns the minimum available safe limit while preserving documented explicit opt-ins. Wire it through `src/agents/context.ts`, reply model selection, memory flush, session persistence, gateway/status summary surfaces, and add regression tests for `agent=500000` plus `model=131072` resolving to `131072`, agent smaller than model resolving to the agent value, provider-qualified discovery collisions, and missing metadata fallback. Separately audit `params.maxTokens` merging and provider transports so requested output limits are clamped to model `maxTokens` where the provider contract requires it.

## Risks / Open Questions

- Changing precedence can reduce effective context for users whose provider metadata is stale or intentionally conservative; preserve explicit opt-ins such as Anthropic `context1m` and the provider-qualified cache collision safeguards.
- `maxTokens` behavior is transport-specific, so the fix should be contract-tested per provider path instead of applying an unsafe blanket clamp where a provider has different semantics.
