## Summary

Keep open. Current `main` does not implement per-agent session maintenance overrides: agent config/schema has no `maintenance` field, maintenance resolution is still global-only, and both cleanup and normal session-write paths continue to read `session.maintenance` from the top-level session config.

## Evidence

- **Agent config type has no per-agent maintenance field:** `AgentConfig` includes per-agent overrides like `model`, `heartbeat`, `sandbox`, and `params`, but no `maintenance` property.
  - file: [src/config/types.agents.ts:70](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/src/config/types.agents.ts#L70)

- **Agent runtime schema rejects an agent-level maintenance key:** `AgentEntrySchema` enumerates allowed agent fields and does not include `maintenance`, so `agents.list[].maintenance` is not part of the current config contract on `main`.
  - file: [src/config/zod-schema.agent-runtime.ts:812](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/src/config/zod-schema.agent-runtime.ts#L812)

## Risks / Open Questions

- Any future implementation still needs to define shared-store semantics: cleanup target resolution dedupes by `storePath`, so multiple agents can collapse onto one `sessions.json` target (`src/config/sessions/targets.ts:35`).
- Override lookup should normalize agent ids the same way normal agent resolution does (`src/agents/agent-scope-config.ts:92`), or mixed-case/humanized ids can miss their override.
