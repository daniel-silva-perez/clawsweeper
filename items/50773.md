## Summary

Keep open. Current `main` does not implement the requested heartbeat collision avoidance for cron or broader lane activity. Heartbeats skip only when the main queue or the resolved session lane is busy, and the config/docs surface has no `heartbeat.skipWhenBusy`, `cron-in-progress`, or `lanes-busy` support.

## Evidence

- **Heartbeat runner only checks main lane first:** `runHeartbeatOnce` reads `getQueueSize(CommandLane.Main)` and returns `requests-in-flight` when that queue is nonzero; there is no cron-lane check at that decision point.
  - file: [src/infra/heartbeat-runner.ts:759](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/src/infra/heartbeat-runner.ts#L759)
  - command: `nl -ba src/infra/heartbeat-runner.ts | sed -n '730,820p'`
  - sha: [ebd7f19a3b20](https://github.com/openclaw/openclaw/commit/ebd7f19a3b20001960474ec9ba9cc424e45f7b23)
- **Second guard is session lane, not cron lane:** After preflight, the only additional busy guard checks `resolveEmbeddedSessionLane(sessionKey)` and again maps the result to `requests-in-flight`. No `CommandLane.Cron` or all-lanes scan exists here.
  - file: [src/infra/heartbeat-runner.ts:785](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/src/infra/heartbeat-runner.ts#L785)

- **Wake retry logic recognizes only `requests-in-flight`:** The wake layer reschedules retries only when the handler returns skipped with reason `requests-in-flight`; there is no handling for proposed new reasons like `cron-in-progress` or `lanes-busy`.
