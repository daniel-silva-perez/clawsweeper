## Summary

Current `main` has deterministic skill source precedence and per-agent skill allowlists, but it does not implement any per-skill priority configuration, priority-aware selection logic, or `openclaw skills priority ...` CLI. The request is coherent and not already done, so there is not strong evidence to close it.

## Evidence

- **Skills config schema has no priority field:** `SkillsConfig` exposes `allowBundled`, `load`, `install`, `limits`, and `entries`; `SkillConfig` only supports `enabled`, `apiKey`, `env`, and `config`. There is no `priority` or equivalent ordering field.
  - file: [src/config/types.skills.ts:3](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/src/config/types.skills.ts#L3)

- **Documented config surface has no priority config:** The skills config docs describe `skills.allowBundled`, `skills.load.*`, `skills.install.*`, `skills.entries.*`, and `agents.*.skills`; no priority file or priority key is documented.
  - file: [docs/tools/skills-config.md:9](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/docs/tools/skills-config.md#L9)

## Risks / Open Questions

- The issue body's claim that skill choice is 'random' was not directly reproduced from runtime behavior; current code shows deterministic prompt ordering, but model-driven skill choice can still feel inconsistent to users.
- Several example skills named in the report appear to be third-party/community skills, so the exact overlap scenario depends on external skill content not present in this checkout.
- Closing as implemented or cannot_reproduce would be unsafe because the requested config/CLI surface plainly does not exist on `main`.
