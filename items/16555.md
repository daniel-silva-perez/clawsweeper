## Summary

Keep open. Current `main` still uses the file-backed delivery queue, persists entries with `enqueuedAt`, and replays all pending queue files on gateway startup subject only to retry/backoff limits. I found no age-based expiry check or config surface for delivery-queue TTL/expiry.

## Evidence

- **Startup recovery still replays all pending entries:** `recoverPendingDeliveries()` loads every pending queue entry, sorts by `enqueuedAt`, and iterates them with only max-retry and backoff checks. There is no `maxAge`, TTL, or expiry branch in this path.
  - file: [src/infra/outbound/delivery-queue-recovery.ts:344](https://github.com/openclaw/openclaw/blob/a59d1bd46d9fc0bd2d9194b3f17db2561efc51e3/src/infra/outbound/delivery-queue-recovery.ts#L344)

  - file: [src/infra/outbound/delivery-queue-storage.ts:129](https://github.com/openclaw/openclaw/blob/a59d1bd46d9fc0bd2d9194b3f17db2561efc51e3/src/infra/outbound/delivery-queue-storage.ts#L129)

## Risks / Open Questions

- The recently shipped two-phase ACK cleanup reduces one duplicate-after-crash path, so some originally reported duplicate cases may now be narrower than when the issue was filed.
- The requested behavior is still a product/config design choice: config location, default TTL, and whether expiry should skip, delete, or move entries to `failed/` are not settled by current code.
