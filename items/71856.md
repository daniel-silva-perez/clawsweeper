## Summary

Keep open. Current main has the Gateway/TUI session metadata needed for this feature, but the TUI startup connection path still only loads full history and sets connection status; it does not show a concise recent-session startup summary. The PR is not obsolete or outside core, but it needs maintainer review and at least the reconnect-duplication issue from the review discussion addressed before it is mergeable.

## Best Possible Solution

Keep the PR open for revision or maintainer product review. The best path is a narrow TUI implementation using the existing `sessions.list` derived-title/last-message metadata, gated to initial non-local connection only, with tests proving no reconnect duplication and no startup failure when the API call fails.
