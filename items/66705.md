## Summary

Keep open. Current main still does not inject `OPENCLAW_SESSION_KEY` or `OPENCLAW_AGENT_ID` into exec child-process environments. The exec stack has trusted session and agent context available, but spawned command envs are only marked with the existing `OPENCLAW_SHELL=exec` and shared `OPENCLAW_CLI=1` markers, and current docs/tests do not define the requested env contract.

## Best Possible Solution

Keep this issue open for a focused core exec-runtime change. Add a shared helper that injects `OPENCLAW_SESSION_KEY` from trusted session context and `OPENCLAW_AGENT_ID` from trusted agent context after existing env merging/sanitization, thread `agentId` through `runExecProcess` and node exec env construction, document the new runtime markers, and add regression tests for gateway, PTY, sandbox, and node paths without weakening existing host-env filtering.

## Risks / Open Questions

- Implementation needs a clear precedence rule so trusted runtime-injected values cannot be spoofed by user-provided env overrides, and stale inherited values should be cleared when no session is bound.
- Coverage should include host gateway, PTY, sandbox, and node exec paths. Related PR #68859 appears to cover only the session-key part, while this issue also asks for `OPENCLAW_AGENT_ID`.
