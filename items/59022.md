## Summary

Keep open. The report is recent, has concrete reproduction details, and describes a plausible Gateway/node A2UI capability-url failure rather than a clearly obsolete or already-implemented request. Current docs explain why the raw curl returns Unauthorized on a LAN bind, but that does not prove the reported `canvas.a2ui.push` command failure is fixed on main.

## Evidence

- **Issue has actionable repro and environment:** Issue #59022 reports OpenClaw 2026.3.31 on macOS with `gateway.bind: lan`; `canvas a2ui_push --node ...` fails with `UNAVAILABLE: A2UI_HOST_UNAVAILABLE: A2UI host not reachable`, while other canvas commands work.
- **Current docs describe protected A2UI routes and node capability URLs:** The configuration reference says A2UI is served under the Gateway port at `/__openclaw__/a2ui/`; non-loopback binds require Gateway auth, and node WebViews use node-scoped capability URLs tied to the active WS session. This makes the issue's raw Unauthorized curl result expected, but leaves the node command failure plausible.
  - file: [docs/gateway/configuration-reference.md:727](https://github.com/openclaw/openclaw/blob/d6ef1fcf24eea3a5985d653a3baf400e32aa1b75/docs/gateway/configuration-reference.md#L727)
  - sha: [d6ef1fcf24ee](https://github.com/openclaw/openclaw/commit/d6ef1fcf24eea3a5985d653a3baf400e32aa1b75)
- **Android docs still treat A2UI reachability as an active troubleshooting path:** The Android integration README says A2UI checks require the app Screen tab/canvas WebView, canvas host reachability, and may need reconnect if auto-refresh after an A2UI reachability failure does not resolve it. That is not strong enough evidence to close this macOS-node report as fixed.
