## Summary

Keep open. Current main still lacks sender-aware non-owner exec deny behavior, and the broader channel tool policy has not landed. This PR is a plausible security fix candidate, but it should not be closed or merged as-is without maintainer/security review because the PR head can still inherit permissive downstream exec defaults from partial agent exec config and overlaps with the same-author open hardening PR #64719.

## Best Possible Solution

Keep this PR open for maintainer/security review. Revise the non-owner branch so user-injectable inline/session exec fields are ignored, no-override non-owner turns fail closed with security=deny and ask=always, and partial agentExecDefaults cannot inherit permissive downstream defaults. Coordinate the final boundary with #64719, then add regression tests for no overrides, inline directive injection, persisted session overrides, partial agent defaults, and explicit safe operator defaults.

## Risks / Open Questions

- Merging this PR as-is may still allow non-owner auto-reply exec when an operator has a partial agents.list[].tools.exec object that omits security and ask, because downstream defaults can still fill security=full and ask=off.
- This PR overlaps with #64719's broader channel dangerous-tool policy; landing either independently could leave inconsistent non-owner/channel exec semantics.
- The affected surface is remote channel-triggered command execution, so maintainer/security review and focused regression tests are warranted before merge.
