## Summary

Keep #66923 open. Current main has OpenAI-managed forward-compat handling for GPT-5.5/GPT-5.4-family models, but I did not find matching GPT-4.1/nano/mini registry entries, OpenAI dynamic-model handling, or an unmanaged-model startup warning. The older #63301 is the same report re-filed from the wrong account, and the related closed #66652 model_spawn PR does not supersede this registry/validation request.

## Best Possible Solution

Implement the product decision in the OpenAI provider path: either add verified GPT-4.1, GPT-4.1-mini, and GPT-4.1-nano managed catalog/dynamic entries with context, max-token, cost, reasoning, transport, and regression coverage, or add a visible config/startup/runtime warning for model ids that resolve through the generic configured-provider fallback. Cover the resolver path in src/agents/pi-embedded-runner/model.ts and the OpenAI plugin catalog/dynamic path in extensions/openai/openai-provider.ts.

## Risks / Open Questions

- Leaving this open is appropriate because users with an OpenAI provider config can still fall into generic synthesized model metadata for unknown ids, with no explicit unmanaged-model warning.
- Adding GPT-4.1-family support should verify current OpenAI metadata and decide whether these models are intentionally supported managed targets or should instead produce a clear unsupported/unmanaged warning.
