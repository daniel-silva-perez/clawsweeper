## Summary

Current main already bounds the Tailscale hostname probe across `status`, `status --json`, and `status --all` through the shared status bootstrap, and the latest release already contains that implementation. The PR’s original inline timeout changes are obsolete against the refactored status architecture.

## Close Comment

Closing this as a Codex review because the requested behavior is already implemented on current `main` and is already shipped in `v2026.4.22`. Today, `status`, `status --json`, and `status --all` all flow through the shared status overview/bootstrap path (`src/commands/status.scan.ts:19`, `src/commands/status.scan.fast-json.ts:28`, `src/commands/status-all.ts:13`), and that bootstrap runs the Tailscale hostname probe through `runExec(..., { timeoutMs: 1200, maxBuffer: 200_000 })` and falls back to `null` on failure (`src/commands/status.scan.bootstrap-shared.ts:93`). The underlying exec path is also genuinely time-bounded via Node `execFile` timeout handling (`src/process/exec.ts:131`). In other words, the timeout/fallback behavior this PR was trying to add is already present through the newer shared architecture, so this PR is obsolete against current `main`.
