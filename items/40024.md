## Summary

Keep #40024 open. Current main still does not guarantee that a custom local OpenAI-compatible provider with only `baseUrl` configured will send `POST /v1/chat/completions`; fallback model resolution still defaults to `openai-responses`, and the transport path is selected strictly from the resolved `model.api`. There is also still a plausible local-loopback blocker unless the provider request config opts into private-network access.

## Best Possible Solution

Keep the issue open and fix or explicitly decide the product contract for custom local OpenAI-compatible providers. A strong fix would make the embedded runner either infer/use `openai-completions` for custom `/v1` local providers that do not support Responses, or require and validate/document `models.providers.<id>.api` plus `request.allowPrivateNetwork` so users get an immediate configuration error instead of a timeout/no-POST symptom. Add a regression test covering a `local-agent-proxy` provider with `baseUrl: "http://127.0.0.1:3000/v1"` and the expected transport path.
