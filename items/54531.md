## Summary

Keep open. Current main still intentionally avoids routing replies from internal webchat/dashboard turns back to the originating messaging channel unless `ExplicitDeliverRoute` is set, and the requested persisted-origin or force-origin config does not exist for normal interactive turns.

## Evidence

- **Routing policy still blocks internal webchat fallback:** `resolveReplyRoutingDecision` marks `webchat` as an internal turn unless `explicitDeliverRoute === true`, then refuses origin routing when the current surface matches internal webchat.
  - file: [src/auto-reply/reply/routing-policy.ts:16](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/auto-reply/reply/routing-policy.ts#L16)

- **Main dispatch path still uses current-surface comparison:** `dispatchReplyFromConfig` comments that it only routes when the originating channel differs from the current surface, and computes `hasRouteReplyCandidate` with the same `!isInternalWebchatTurn` and `normalizedRouteReplyChannel !== normalizedCurrentSurface` checks.
  - file: [src/auto-reply/reply/dispatch-from-config.ts:316](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/auto-reply/reply/dispatch-from-config.ts#L316)
