## Summary

Keep this open. Current main has narrower safeguards that preserve an established external delivery route when WebChat/dashboard touches a session, but it still does not implement the requested force-origin-channel or immutable persisted-origin behavior. The linked implementation PR #57802 is useful prior art, but it was closed unmerged, and current tests still assert that internal WebChat turns do not route to an external origin unless explicit delivery is set.

## Best Possible Solution

Keep this open for maintainer product judgment. The best path is a narrow reply-routing design at the session/reply boundary: either an explicit force-origin setting or an immutable persisted-origin route captured from the first external user turn, with regression tests for Telegram/Discord/WhatsApp-style inbound sessions while WebChat/dashboard is active and explicit safeguards against stale-route or wrong-channel delivery.

## Risks / Open Questions

- A broad force-origin default could misdeliver replies into stale or unintended external chats, especially for shared/main sessions and dashboard-initiated turns.
- Any fix needs to preserve the existing safeguards that prevent WebChat/control-ui clients from silently inheriting stale external delivery routes.
- The closed PR #57802 would need current-main review and regression tests against the newer route-preservation safeguards before it could be revived or replaced.
