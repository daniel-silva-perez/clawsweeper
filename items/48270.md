## Summary

Current `main` does not fully implement this PR’s fix. The repo still has the same heavy module-scope AJV compilation in the gateway client import path, `probeGateway()` still arms its timeout and starts the socket immediately with no readiness preflight, and the only related change on `main` is a single `setImmediate()` in `callGateway()` that does not cover direct probe paths.

## Evidence

- **Direct probe path still has no readiness preflight:** `probeGateway()` records `startedAt`, arms the probe timeout, and calls `client.start()` directly. There is no `waitForEventLoopReady`-style helper or equivalent bounded preflight in this path.
  - file: [src/gateway/probe.ts:137](https://github.com/openclaw/openclaw/blob/b453293349657dedd069d291fa5b078d910067b1/src/gateway/probe.ts#L137)

## Risks / Open Questions

- Direct `probeGateway()` callers may still report false timeouts on Windows or other slow-import environments because the probe timeout starts before any event-loop-readiness check.
- The current `callGateway()` mitigation is intentionally narrower than this PR and does not prove the deferred-module-evaluation problem is fully addressed across all gateway probe/status paths.
