## Summary

Current `main` still appears vulnerable to the reported Feishu filename mojibake path. The Feishu download metadata parser decodes RFC 5987 `filename*=` values, but for plain `filename=` it returns the header bytes as-is, with no UTF-8/Latin-1 recovery. That filename is then passed straight into inbound media saving, so a garbled Feishu `Content-Disposition` value would still be persisted as a garbled saved basename. I did not find evidence on `main` that #48578/#50435 or an equivalent fix has landed.

## Evidence

- **Feishu header parser still lacks recovery on plain filename=:** `decodeDispositionFileName` decodes `filename*=UTF-8''...`, but the plain `filename=` branch only regex-extracts and trims the value; there is no Latin-1/UTF-8 repair or alternative decoding path.
