## Summary

Current `main` still appears to drop WebChat TTS audio on the UI path. Gateway/chat now emits assistant `content` blocks with `type: "audio"`, but the WebChat normalizer/renderer only knows how to render audio when it arrives as an `attachment.kind === "audio"` item or `MEDIA:` text-derived attachment. Raw audio content blocks are not converted into a renderable attachment, so the reported "success but no audible playback/downloadable audio" issue remains plausible on `main`.

## Evidence

- **Gateway emits audio blocks for WebChat assistant replies:** `buildWebchatAudioContentBlocksFromReplyPayloads` creates `{ type: "audio", source: { type: "base64", ... } }` blocks, and `chat.ts` uses that content for appended/broadcast assistant messages.
  - file: [src/gateway/server-methods/chat-webchat-media.ts:268](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/src/gateway/server-methods/chat-webchat-media.ts#L268)
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
- **Chat send path persists/broadcasts the audio-block content:** `buildAssistantDisplayContentFromReplyPayloads` is called for final payloads, and the resulting `assistantContent` is used in the assistant message content that gets appended/broadcast.
  - file: [src/gateway/server-methods/chat.ts:2582](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/src/gateway/server-methods/chat.ts#L2582)
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
- **Regression test expects transcript updates with `type: "audio"` content:** The current test suite asserts that an assistant transcript update contains `content: [{ type: "text", text: "Audio reply" }, { type: "audio", source: ... }]`.
  - file: [src/gateway/server-methods/chat.directive-tags.test.ts:546](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/src/gateway/server-methods/chat.directive-tags.test.ts#L546)
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
- **WebChat normalizer does not handle raw audio content blocks:** When `content` is an array, the normalizer only special-cases `attachment`, `canvas`, and assistant `text`. Other block types fall through to a generic item with `type`/`text`/`name`/`args`, so an `{ type: "audio", source: ... }` block loses its audio payload and does not become a renderable attachment.
  - file: [ui/src/ui/chat/message-normalizer.ts:277](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/ui/src/ui/chat/message-normalizer.ts#L277)
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
- **WebChat renderer only renders audio attachments, not raw audio blocks:** Assistant audio UI is implemented for normalized `attachment.kind === "audio"` items, while other raw content-block extraction in this file only handles image variants (`image`, `image_url`, `input_image`).
  - file: [ui/src/ui/chat/grouped-render.ts:1149](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/ui/src/ui/chat/grouped-render.ts#L1149)
