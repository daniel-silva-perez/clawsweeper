## Summary

Keep open. Current `main` does not expose a `messages.suppressMediaPlaceholders`-style config, and at least Signal and iMessage still synthesize `<media:...>` text for attachment-only outbound sends, so the feature request is not implemented on `main` or in the latest release snapshot I could inspect.

## Evidence

- **No messages config knob for placeholder suppression:** `MessagesConfig` lists the available `messages.*` settings through `tts`, but there is no `suppressMediaPlaceholders`, `mediaPlaceholders`, or similar outbound-media placeholder setting.
  - file: [src/config/types.messages.ts:86](https://github.com/openclaw/openclaw/blob/a59d1bd46d9fc0bd2d9194b3f17db2561efc51e3/src/config/types.messages.ts#L86)
- **Config metadata also lacks the requested setting:** The schema labels/help enumerate current `messages.*` surfaces such as `responsePrefix`, `queue`, `suppressToolErrors`, `ackReaction`, `statusReactions`, and `tts`, but no placeholder-suppression option is present.
  - file: [src/config/schema.labels.ts:769](https://github.com/openclaw/openclaw/blob/a59d1bd46d9fc0bd2d9194b3f17db2561efc51e3/src/config/schema.labels.ts#L769)
- **Signal still injects placeholder text for media-only sends:** When `mediaUrl` is present and outbound text is empty, Signal explicitly sets `message = kind === "image" ? "<media:image>" : `<media:${kind}>`` to avoid an empty body.
  - file: [extensions/signal/src/send.ts:154](https://github.com/openclaw/openclaw/blob/a59d1bd46d9fc0bd2d9194b3f17db2561efc51e3/extensions/signal/src/send.ts#L154)
- **iMessage still injects placeholder text for media-only sends:** `resolveDeliveredIMessageText()` converts empty-text media sends into `<media:${kind}>`, and `sendMessageIMessage()` applies that helper when `mediaUrl` is supplied.
  - file: [extensions/imessage/src/send.ts:86](https://github.com/openclaw/openclaw/blob/a59d1bd46d9fc0bd2d9194b3f17db2561efc51e3/extensions/imessage/src/send.ts#L86)
- **Latest release snapshot still contains the Signal placeholder logic:** The latest release commit provided in the task (`v2026.4.22`, `00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`) shows the same Signal attachment-only placeholder injection in the shipped file content.
  - command: `git show 00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5:extensions/signal/src/send.ts | sed -n '145,190p'`
  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)

- The local clone is shallow and `gh` could not be used without `GH_TOKEN`, so I could not inspect the referenced PR discussions directly; this decision is based on current source and the provided release commit snapshot.
