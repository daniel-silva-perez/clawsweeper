## Summary

Keep open. Current `main` at `40be5ad58187fd20b8defe5fc30e649190c131a8` still does not recognize or format top-level FastAPI-style `{"detail":"..."}` error payloads, so the PR’s fix is not already implemented on main.

## Evidence

- **Top-level error payload detection still excludes `detail`:** `isErrorPayloadObject` only accepts payloads with `type === "error"`, a request id, or nested `error.{message,type,code}`. A payload shaped like `{"detail":"..."}` is rejected.
  - file: [src/shared/assistant-error-format.ts:27](https://github.com/openclaw/openclaw/blob/40be5ad58187fd20b8defe5fc30e649190c131a8/src/shared/assistant-error-format.ts#L27)
  - command: `nl -ba src/shared/assistant-error-format.ts | sed -n '20,80p'`
  - sha: [40be5ad58187](https://github.com/openclaw/openclaw/commit/40be5ad58187fd20b8defe5fc30e649190c131a8)
- **Parsed message extraction still ignores `payload.detail`:** `parseApiErrorInfo` sets `topMessage` from `payload.message` only. There is no fallback to `payload.detail`, so even if such a payload were parsed later, its message would still be dropped.
  - file: [src/shared/assistant-error-format.ts:152](https://github.com/openclaw/openclaw/blob/40be5ad58187fd20b8defe5fc30e649190c131a8/src/shared/assistant-error-format.ts#L152)
  - command: `nl -ba src/shared/assistant-error-format.ts | sed -n '122,176p'`
  - sha: [40be5ad58187](https://github.com/openclaw/openclaw/commit/40be5ad58187fd20b8defe5fc30e649190c131a8)
- **Formatter still falls back to raw text when parsing fails:** `formatRawAssistantErrorForUi` returns the original trimmed string whenever `parseApiErrorInfo` returns null. For an unrecognized `{"detail":"..."}` payload, that means the raw JSON would still be shown to the user.
  - file: [src/shared/assistant-error-format.ts:206](https://github.com/openclaw/openclaw/blob/40be5ad58187fd20b8defe5fc30e649190c131a8/src/shared/assistant-error-format.ts#L206)
  - command: `nl -ba src/shared/assistant-error-format.ts | sed -n '178,214p'`
  - sha: [40be5ad58187](https://github.com/openclaw/openclaw/commit/40be5ad58187fd20b8defe5fc30e649190c131a8)
- **Agent error path only sanitizes recognized API payloads:** `formatAssistantErrorText` only delegates to `formatRawAssistantErrorForUi` when `isLikelyHttpErrorText(raw)` or `isRawApiErrorPayload(raw)` is true. `isRawApiErrorPayload` depends on `parseApiErrorPayload`, so the current parser gap means FastAPI-style `detail` payloads bypass the structured sanitization path.
  - file: [src/agents/pi-embedded-helpers/errors.ts:1094](https://github.com/openclaw/openclaw/blob/40be5ad58187fd20b8defe5fc30e649190c131a8/src/agents/pi-embedded-helpers/errors.ts#L1094)
