## Summary

Current `main` does not implement transcript-side pruning of consecutive `NO_REPLY` assistant messages before model input. The only related behavior on `main` is UI/display filtering for `chat.history` and existing pruning that targets heartbeat pairs or old `toolResult` content.

## Evidence

- **Context pruning only touches tool results:** `pruneContextMessages()` iterates prunable messages and skips anything whose role is not `toolResult`, so assistant `NO_REPLY` transcript entries are not removed from model context by the current pruning path.
  - file: [src/agents/pi-hooks/context-pruning/pruner.ts:309](https://github.com/openclaw/openclaw/blob/68e2d6f0889e5365ecdc8f01ab5e3bf6065cbeeb/src/agents/pi-hooks/context-pruning/pruner.ts#L309)
  - command: `nl -ba src/agents/pi-hooks/context-pruning/pruner.ts | sed -n '210,360p'`
  - sha: [68e2d6f0889e](https://github.com/openclaw/openclaw/commit/68e2d6f0889e5365ecdc8f01ab5e3bf6065cbeeb)
- **Pre-prompt filtering only removes heartbeat pairs:** Right before prompt assembly, the runner applies `filterHeartbeatPairs(activeSession.messages, ...)`; there is no analogous `NO_REPLY` pruning step in this pre-model path.
  - file: [src/agents/pi-embedded-runner/run/attempt.ts:2099](https://github.com/openclaw/openclaw/blob/68e2d6f0889e5365ecdc8f01ab5e3bf6065cbeeb/src/agents/pi-embedded-runner/run/attempt.ts#L2099)
  - command: `nl -ba src/agents/pi-embedded-runner/run/attempt.ts | sed -n '2088,2145p'`
  - sha: [68e2d6f0889e](https://github.com/openclaw/openclaw/commit/68e2d6f0889e5365ecdc8f01ab5e3bf6065cbeeb)
- **Heartbeat filter is narrowly scoped:** `filterHeartbeatPairs()` only drops a `user` heartbeat prompt followed by an assistant `HEARTBEAT_OK` acknowledgment; it does not target repeated assistant `NO_REPLY` entries.
  - file: [src/auto-reply/heartbeat-filter.ts:71](https://github.com/openclaw/openclaw/blob/68e2d6f0889e5365ecdc8f01ab5e3bf6065cbeeb/src/auto-reply/heartbeat-filter.ts#L71)
  - command: `nl -ba src/auto-reply/heartbeat-filter.ts | sed -n '1,120p'`
  - sha: [68e2d6f0889e](https://github.com/openclaw/openclaw/commit/68e2d6f0889e5365ecdc8f01ab5e3bf6065cbeeb)
- **Current NO_REPLY handling is display-only history sanitization:** Gateway `sanitizeChatHistoryMessages()` and the UI controller both hide assistant messages whose visible text is exactly `NO_REPLY`, but those paths are for `chat.history` presentation rather than transcript pruning before LLM input.
  - file: [src/gateway/server-methods/chat.ts:1196](https://github.com/openclaw/openclaw/blob/68e2d6f0889e5365ecdc8f01ab5e3bf6065cbeeb/src/gateway/server-methods/chat.ts#L1196)
  - command: `nl -ba src/gateway/server-methods/chat.ts | sed -n '1196,1238p'; nl -ba ui/src/ui/controllers/chat.ts | sed -n '35,80p'`
  - sha: [68e2d6f0889e](https://github.com/openclaw/openclaw/commit/68e2d6f0889e5365ecdc8f01ab5e3bf6065cbeeb)
- **No feature/config hook for the proposed setting exists on main:** A repo-wide search found no `pruneConsecutiveNoReply` or similar implementation/config symbol, which matches the missing behavior in the code paths above.
  - command: `rg -n "pruneConsecutiveNoReply|consecutiveNoReply|auto-prune consecutive NO_REPLY" src docs ui packages extensions`
  - sha: [68e2d6f0889e](https://github.com/openclaw/openclaw/commit/68e2d6f0889e5365ecdc8f01ab5e3bf6065cbeeb)

- GitHub CLI was unavailable in this environment without `GH_TOKEN`, so PR #13388 could not be hydrated via `gh`; the decision is based on the checked-out `main` code and local git inspection.
