## Summary

Keep this PR open. Current main still does not make runtime `toolsAllow` authoritative over the final bundled MCP/LSP tool list; it only partially mitigates core-only allowlists by skipping bundle-MCP startup when the allowlist cannot name bundled MCP tools. The PR targets a real active path, but its one-line fix likely needs maintainer adjustment to preserve plugin-key semantics such as `bundle-mcp` and to add regression coverage.

## Best Possible Solution

Keep this PR open or supersede it with a maintainer patch in `src/agents/pi-embedded-runner/run/attempt.ts` that applies runtime `toolsAllow` after bundled MCP/LSP materialization while preserving exact tool-name and plugin-key semantics (`bundle-mcp`, likely `bundle-lsp` if supported). Add focused regression coverage for `toolsAllow: ["exec"]`, one exact MCP tool name, `toolsAllow: ["bundle-mcp"]`, and configured bundled LSP.

## Risks / Open Questions

- Functional risk: the submitted one-line patch likely makes `toolsAllow: ["bundle-mcp"]` filter out all bundle-MCP tools, because `applyEmbeddedAttemptToolsAllow` exact-matches `tool.name` while `bundle-mcp` is a plugin/group key expanded by the policy pipeline.
- Coverage risk: neither current main nor the PR evidence includes focused regression coverage for final combined MCP/LSP filtering, exact bundled tool names, `bundle-mcp` plugin-key allowlists, and bundled LSP tools.
- Security review: no supply-chain or CI/security-sensitive file changes are visible in the PR diff; the patch is limited to runtime tool selection. The security-sensitive correctness requirement is that runtime allowlists become stricter without accidentally widening or breaking plugin-key authorization semantics.
