## Summary

Current main no longer matches the reported Slack socket-mode crash path: OpenClaw disables Slack SDK auto-reconnect, owns Slack reconnect handling with bounded backoff, and gateway channel failures are caught and restarted instead of surfacing as unhandled rejections.

## Risks / Open Questions

## Close Comment

Current `main` already covers the Slack socket-mode crash path described here, so this is being closed as implemented on main.

- Codex review: `extensions/slack/src/monitor/provider-support.ts:134` constructs `SocketModeReceiver` with `autoReconnectEnabled: false`, and `extensions/slack/src/monitor/provider.interop.test.ts:132` locks that behavior in. OpenClaw no longer depends on Slack SDK's zero-attempt reconnect path.
- `extensions/slack/src/monitor/reconnect-policy.ts:4` and `extensions/slack/src/monitor/provider.ts:438` implement OpenClaw-owned bounded exponential reconnects across disconnect, startup-failure, and socket error events instead of treating the first abnormal close as fatal.
- `src/gateway/server-channels.ts:431` catches channel task failures, records `lastError`, and schedules bounded channel restarts rather than letting the rejection crash the gateway process.
- The shipped changelog already records this as `Slack/Socket reconnect reliability` in `CHANGELOG.md:3702`, and the current release `v2026.4.23` contains the same implementation.
