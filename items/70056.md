## Summary

Keep PR #70056 open. Current main still has the stop-timeout bug the PR targets, and the PR is a focused, recent gateway fix with a regression-test update, but the proposed implementation has unresolved review feedback about a secondary race where the timed-out task can later clobber runtime state. This should be revised rather than cleanup-closed.

## Best Possible Solution

Revise PR #70056 rather than close it: keep the intended behavior change of reporting `running: false`, clearing or replacing stale lifecycle ownership safely, and allowing recovery after timeout, but add an identity/lifecycle guard so the orphaned timed-out task cannot later overwrite a newer task's runtime state or trigger an extra restart. The regression test should cover a timed-out task that eventually settles after a restart, not only a never-settling task.
