## Summary

Current `main` already fixes the fallback-context hot-reload problem through a different implementation: the fallback path now resolves the live gateway request context, and that context reads cron state via live getters from `runtimeState` after reload.

## Evidence

## Close Comment

Closing from a Codex review because current `main` already implements this behavior. The fallback path now uses `setFallbackGatewayContextResolver(() => gatewayRequestContext)` in `src/gateway/server.impl.ts:767`, and the request context itself reads `cron` and `cronStorePath` through live getters over `runtimeState.cronState` in `src/gateway/server-request-context.ts:67-73`. Hot reload replaces that runtime cron state in `src/gateway/server.impl.ts:879-885`, so fallback plugin dispatch sees the refreshed cron service and store path without needing the callback proposed here. The current tests cover both pieces in `src/gateway/server-request-context.test.ts:6-79` and `src/gateway/server-plugins.test.ts:1065-1090`. Evidence was verified against current `main` at `45146913007da723b4f06aba6a66cf4a695974c0`.
