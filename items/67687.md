## Summary

Keep open. Current main does not clearly satisfy the reported Feishu config shape. The Feishu config loader can materialize `groupPolicy: "allowlist"`, and the inbound handler checks group admission through `groupAllowFrom` before it ever resolves `groups.<chat_id>.requireMention`; the reporter's config has `groups.oc_xxx.requireMention=false` but no `groupAllowFrom`, which can still drop all group messages before a session is created. `requireMention: false` itself is honored once a group is admitted, so the remaining issue is the Feishu group admission/config contract rather than the mention gate alone.

## Best Possible Solution

Keep this issue open and fix or clarify Feishu group admission. The conservative implementation path is to either treat explicit `channels.feishu.groups.<chat_id>` entries as admitted groups under `groupPolicy: "allowlist"` or add a targeted doctor/config warning that `groups.<chat_id>.requireMention=false` does not admit the group without `groupAllowFrom` or `groupPolicy: "open"`. Add a regression test that goes through plugin-aware config validation with the reporter's config shape and then verifies a Feishu group text event reaches route/session dispatch. Update `docs/channels/feishu.md` if the intended contract stays `groupAllowFrom`-only.
