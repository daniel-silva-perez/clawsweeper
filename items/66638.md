## Summary

Keep this issue open. Current main has meaningful partial work for subagent completion delivery, including direct-first completion dispatch, queue fallback, idempotency, retry/resume bookkeeping, and plugin hooks for spawn/end routing. It still does not implement the central requested contract: framework-owned immediate user-facing spawn ACKs, per-run progress lifecycle notifications, or a durable idempotent announce queue that survives gateway restarts and channel outages independently of the parent session lane.

## Best Possible Solution

Keep this open as the umbrella runtime architecture request. The best path is to extend the existing subagent completion delivery work with runtime-owned lifecycle events for spawned/progress/completed/failed/cancelled, a framework-generated immediate user-facing spawn acknowledgement independent of the parent turn, and durable idempotent announce persistence/retry, likely by integrating with the existing durable delivery queue or by adding a purpose-built run-lifecycle queue. Coordinate the implementation with #22358 and #54488 rather than closing this until those pieces are actually shipped.

## Risks / Open Questions

- Current main has several partial delivery and retry mechanisms, so any implementation should build on them rather than replacing the completion announce path wholesale.
- Channel-direct lifecycle notices need a narrow framework/status-event contract so they do not become a bypass for arbitrary model output or normal channel safety/routing rules.
