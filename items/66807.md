## Summary

Keep this PR open. Current main still lacks peer-aware outbound account selection in runMessageAction, and the PR is a focused fix attempt for a plausible multi-account Slack routing bug. It should not be merged as-is because its raw binding scan can include ACP bindings and can misroute same-peer multi-account setups.

## Best Possible Solution

Keep this PR open and request a revised implementation. The best path is to add a typed outbound account resolver in src/infra/outbound/message-action-runner.ts that reuses or mirrors resolveFirstBoundAccountId/listRouteBindings, preserves explicit --account/defaultAccountId precedence, keeps agent/account scoping, excludes ACP bindings, handles target aliases and peer kind, and adds regression tests for the reported Slack multi-account peer binding plus ACP and same-peer collision cases.

## Risks / Open Questions

- Leaving current main unchanged preserves the reported default-agent fallback failure mode for CLI-originated sends in multi-account Slack configs.
- The PR as written can select ACP bindings because it scans raw cfg.bindings instead of listRouteBindings(cfg).
- The PR as written can misroute when multiple route bindings share the same peer but differ by agent/account, because the new fast path matches only channel plus peer id before the agent-scoped fallback.
- No supply-chain or CI execution-path regression is visible in the provided one-file diff.
