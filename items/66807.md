## Summary

Keep this PR open. Current main still does not perform peer-aware account selection in `runMessageAction`, and the CLI `message` path still supplies the default agent id, so the reported multi-account Slack routing bug remains plausible. The PR is a useful focused fix attempt with a concrete reproduction, but it should not be merged as-is because review comments identify real misrouting risks.

## Best Possible Solution

Keep the PR open and ask for a revised implementation. The best path is to add a typed outbound account resolver in `src/infra/outbound/message-action-runner.ts` that reuses or mirrors `resolveFirstBoundAccountId`/`listRouteBindings`, preserves explicit `--account` precedence, keeps agent/account scoping, excludes ACP bindings, handles target aliases and peer kind, and adds regression tests for the reported Slack-style multi-account peer binding plus the two review-comment edge cases.

## Risks / Open Questions

- The PR as written can select ACP bindings for outbound account routing because it scans raw `cfg.bindings` instead of route bindings only.
- The PR as written can misroute same-peer multi-account setups because the peer-aware fast path ignores the active agent unless the match is unique or explicitly scoped.
- Leaving current main unchanged preserves the reported default-agent fallback failure mode for CLI-originated sends in multi-account Slack configs.
