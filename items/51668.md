## Summary

Keep #51668 open. Current main still does not implement gateway-layer outbound secret sanitization, and the PR remains a plausible security-hardening path, but the review thread shows unresolved correctness gaps that make it unsuitable for auto-close or merge-as-is.

## Best Possible Solution

Keep the PR open for maintainer/product-security judgment, or supersede it with a narrower maintainer-owned implementation that sanitizes at the final pre-send boundary, covers text plus structured outbound payload fields, derives credential handles from the existing sensitive/SecretRef registry instead of a partial manual walk, wires the config schema/types/docs, and adds regression tests for hook rewrites, JSON/quoted values, nested account secrets, and interactive payload content.
