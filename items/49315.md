## Summary

Keep #49315 open. Current main still does not implement `--profile-id` for `openclaw models auth login`, and the PR is a plausible focused implementation attempt rather than obsolete cleanup. The remaining provider-side correctness concern from review is also still grounded in current main for GitHub Copilot-style auth flows, so this should not be auto-closed as implemented or safely superseded.

## Best Possible Solution

Keep the PR open, or keep the canonical feature request #40402 open if maintainers decide this PR should be reworked. The best implementation is to add a typed `profileId` path from `models auth login` through `LoginOptions` into the provider auth context so providers that can honor a caller-selected single profile id do so before persistence; providers with ambiguous multi-profile results or side-effectful fixed-profile flows should reject or explicitly opt out. Update the CLI docs, gateway/auth docs, OpenAI provider docs, and focused tests for successful single-profile remapping plus ambiguous-provider rejection.
