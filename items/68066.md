## Summary

Current main does not implement the reported OpenRouter billed-cost reconciliation. The OpenAI-compatible text path captures streamed usage and computes `usage.cost` from model pricing, while the OpenRouter provider derives that pricing from the model catalog; there is no `/api/v1/generation` lookup or `total_cost` handling. Related usage-cost dashboard items do not supersede this provider-specific billing correctness issue.

## Best Possible Solution

Keep this open and fix it in the bundled OpenRouter provider/transport path: after a verified OpenRouter chat-completion stream finishes and `responseId` is known, query OpenRouter's generation metadata with the resolved OpenRouter credential, then persist the authoritative billed `total_cost` either as `usage.cost.total` or a versioned billed-cost field. Add regression coverage with a mocked tier-priced OpenRouter response and a graceful fallback when the generation lookup fails or is unavailable.

## Risks / Open Questions

- A fix that blocks final stream completion on OpenRouter's generation lookup could add latency or fail when metadata is delayed; it should degrade gracefully and preserve the streamed response.
- Choosing whether to overwrite `usage.cost.total` or add a separate billed field affects downstream dashboards and transcript readers, so the schema contract needs an explicit maintainer decision.
