## Summary

Keep #7338 open. Current main still does not implement an opt-in, domain-scoped OpenClaw agent attestation contract for arbitrary external HTTP requests from web_fetch, skills, or tools. Existing support is limited to basic web_fetch headers, model-provider request attribution/custom headers, and provider-owned web fetch tools; that does not solve the issue's requested static or signed OpenClaw attestation for external services. The existing issue discussion also already points to keeping this open, and the related closed #11137/PR path concerned LLM/provider sender-header propagation, not general external-request attestation.

## Best Possible Solution

Keep #7338 open as the canonical product/API design tracker. A complete solution should decide whether attestation belongs in core or a new plugin/API seam, define opt-in domain-scoped config, specify static versus cryptographic header semantics, wire supported external HTTP paths consistently, document privacy defaults, and add tests proving allowed-domain injection plus non-leakage to other domains.

## Risks / Open Questions
