## Summary

Keep this issue open. The reported mixed-case non-default agent session keys still expose a plausible current-main WebChat event mismatch: the UI preserves the raw `?session=agent:DB_Dude:main` value, while gateway session resolution canonicalizes keys to lowercase before final chat broadcasts. The live chat handlers then compare session keys with exact string equality, so the final event can be ignored and the UI can remain stuck until a refresh reloads history. The related #39743 is broader and closed stale, not a canonical active fix for this specific non-default-agent case.

## Best Possible Solution

Normalize session keys consistently at the Control UI boundary or compare canonicalized keys in chat/session event handlers, then add a regression test using a mixed-case non-default agent deep link such as `agent:DB_Dude:main` with a final event for `agent:db_dude:main`. The fix likely belongs in the shared UI session-key handling plus the chat event path, with gateway behavior kept canonical.
