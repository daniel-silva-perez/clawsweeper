## Summary

Keep #71227 open. Current main has useful session-store maintenance that can prune old entries and cap very high entry counts, but it does not close the reported latency/run-budget bug: session updates still re-read and parse the whole `sessions.json`, still stringify and atomically write the whole store, and `skillsSnapshot` is still persisted inline in each session entry. The related #69451 is open and overlaps the monolithic-store root cause, but it is framed around memory/RSS rather than this issue's Slack/agent.wait run-budget failure, so it is not a clear canonical replacement.

## Best Possible Solution

Keep #71227 open. The best maintainer path is to fix the session persistence layer rather than only relying on retention: deduplicate or content-address `skillsSnapshot`, shard session metadata per session or move it to SQLite, or make large-store writes off the run-critical path. Add regression/performance coverage for fresh below-500-entry stores with repeated snapshots and coordinate any broader storage redesign with #69451.

## Evidence

- **Issue scope:** The provided issue body reports a production OpenClaw 2026.4.15 failure where agents with 4.7 MB to 18.5 MB per-agent `sessions.json` stores time out at `agent.wait` after 60001 ms, with repeated inline `skillsSnapshot` accounting for 71% of one 18.5 MB store. It explicitly distinguishes this latency/run-budget bug from related open issue #69451's memory/RSS angle.
- **Whole-file read and parse remains:** `loadSessionStore` reads `storePath` with `fs.readFileSync` and `JSON.parse`. Load-time maintenance only runs when `Object.keys(store).length > maintenance.maxEntries`, so a fresh 286-entry or 98-entry fat store can still be loaded as one large JSON object.
