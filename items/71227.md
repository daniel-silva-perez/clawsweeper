## Summary

Current main still has the reported size-proportional session metadata path: active updates re-read, parse, normalize, stringify, and atomically rewrite the whole sessions.json file, while skillsSnapshot and systemPromptReport remain inline per session. Existing pruning, entry caps, and rotation are useful mitigations, but they do not implement snapshot deduplication, per-session sharding, SQLite-backed metadata, or off-critical-path writes. Related #69451 tracks the memory/RSS angle and does not supersede this latency/run-budget report.

## Best Possible Solution

Keep this issue open and fix the core session metadata storage path. A conservative first patch would content-address or otherwise deduplicate skillsSnapshot while preserving backward-compatible reads of existing inline entries. The durable fix is per-session metadata sharding or SQLite-backed metadata, with expensive writes moved off the run-critical path where possible. Add regression coverage for a large retained sessions.json with repeated skillsSnapshot data so updating one active session no longer scales with whole-store size; coordinate with #69451 for the broader memory/RSS storage redesign.
