## Summary

Keep PR #48904 open. Current main has moved thinking support behind provider thinking profiles, but it still does not let `models.providers.*.models[*].compat` opt a custom config-defined model into `xhigh`. The PR discussion includes a real custom-provider reproduction and follow-up commits addressing review gaps, so this remains plausible product/code work rather than a cleanup close.

## Best Possible Solution

Keep the PR open and ask for an updated implementation against current main. The best direction is to make config-defined custom model metadata participate in the same thinking-profile source used by `isThinkingLevelSupported`, `listThinkingLevels`, gateway rows, ACP/native commands, TUI/help, reply validation, cron, and `llm-task`, with tests for explicit opt-in, explicit opt-out, binary-provider rejection, and picker/help consistency. Maintainers should decide whether to reuse `compat.supportedReasoningEfforts` or add the narrower `compat.supportsXHighThinking` field.
