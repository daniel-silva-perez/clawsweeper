## Summary

Keep open. Current `main` still surfaces duplicate Slack reaction API errors on the direct `message react` tool path; only the Slack status-reaction lifecycle path suppresses `already_reacted`/`no_reaction` as no-ops.

## Evidence

- **Direct Slack reaction helpers still throw raw API errors:** `reactSlackMessage` and `removeSlackReaction` call `client.reactions.add/remove` without catching `already_reacted` or `no_reaction`, so duplicate add/remove attempts still reject.
  - file: [extensions/slack/src/actions.ts:95](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/extensions/slack/src/actions.ts#L95)
  - command: `nl -ba extensions/slack/src/actions.ts | sed -n '95,120p'`
  - sha: [ebd7f19a3b20](https://github.com/openclaw/openclaw/commit/ebd7f19a3b20001960474ec9ba9cc424e45f7b23)
- **Agent-visible message tool path forwards those rejections:** The Slack action runtime calls those helpers directly for `action === "react"` and does not translate duplicate-reaction failures into success/no-op results.
  - file: [extensions/slack/src/action-runtime.ts:191](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/extensions/slack/src/action-runtime.ts#L191)
  - command: `nl -ba extensions/slack/src/action-runtime.ts | sed -n '191,214p'`
  - sha: [ebd7f19a3b20](https://github.com/openclaw/openclaw/commit/ebd7f19a3b20001960474ec9ba9cc424e45f7b23)
- **Tool failures become user-facing warning messages:** Mutating tool failures are turned into `⚠️ … failed` warnings in the runner payload formatter, matching the issue’s visible Slack warning symptom.
  - file: [src/agents/pi-embedded-runner/run/payloads.ts:350](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/src/agents/pi-embedded-runner/run/payloads.ts#L350)
  - command: `nl -ba src/agents/pi-embedded-runner/run/payloads.ts | sed -n '350,376p'`
  - sha: [ebd7f19a3b20](https://github.com/openclaw/openclaw/commit/ebd7f19a3b20001960474ec9ba9cc424e45f7b23)
- **Only the status-reaction lifecycle path suppresses duplicate reaction errors:** Slack monitor dispatch explicitly swallows `already_reacted` and `no_reaction`, which explains why part of the problem was addressed there but not in the general action helper/runtime path.
  - file: [extensions/slack/src/monitor/message-handler/dispatch.ts:279](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/extensions/slack/src/monitor/message-handler/dispatch.ts#L279)
  - command: `nl -ba extensions/slack/src/monitor/message-handler/dispatch.ts | sed -n '279,300p'`
  - sha: [ebd7f19a3b20](https://github.com/openclaw/openclaw/commit/ebd7f19a3b20001960474ec9ba9cc424e45f7b23)

## Risks / Open Questions

- Users can still get false-negative tool warnings when an agent re-adds an existing Slack reaction through the normal `message react` tool path.
- The issue may look partially fixed in status-reaction flows, which makes it easy to close incorrectly while the general reaction action remains unfixed.
