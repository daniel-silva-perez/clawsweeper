## Summary

Keep this issue open. Current main still returns the rotated device token in the `device.token.rotate` RPC success payload, and the surrounding tests, CLI docs, and Control UI code still treat that plaintext token as the expected contract. The related fix PR in the provided context is closed and unmerged, so it does not supersede or resolve the report.

## Best Possible Solution

Keep the issue open and treat it as a gateway protocol/security design fix. The likely path is to change `src/gateway/server-methods/devices.ts` so normal `device.token.rotate` responses do not expose `entry.token`, add/update regression tests to assert the token is absent or explicitly redacted, and update the CLI, Control UI, and public device-token docs to use a safer bootstrap or out-of-band retrieval flow if plaintext token delivery remains necessary in any narrowly scoped path.
