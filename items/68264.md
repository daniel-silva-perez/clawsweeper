## Summary

Keep open. Current main has a supported Control UI embed path and a node-backed canvas tool, but I do not see enough evidence that the reported canvas/browser chat-visibility regression is fixed. The `canvas` tool still resolves a paired node before `present`/`navigate`, can still surface `node required` when no default node exists, and those actions return `{ ok: true }` rather than an inline webpage preview. The inline chat rendering path is separate `[embed ...]`/structured canvas content, not the same as `canvas(action='present')` or `canvas(action='navigate', url='...')`.

## Best Possible Solution

Keep this issue open and triage the intended product contract. If `canvas present`/`canvas navigate` are expected to give visible chat confirmation, add a snapshot/preview-producing follow-up or a clearer Control UI status path. If they are only node controls, tighten the tool description/error guidance so no-node cases direct users to pair a node, use `canvas snapshot` for visual confirmation, use the `browser` screenshot/snapshot flow for webpages, or use `[embed ...]` for Control UI inline rendering. Add focused regression coverage for the chosen behavior.
