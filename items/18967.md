## Summary

Current `main` does not implement the requested parent-scoped child-to-parent session send. Sub-agents still deny `sessions_send` by default, and the session visibility guard only allows `tree` access to the current session plus sessions spawned by that session, not its parent.

## Evidence

- **Sub-agents still deny `sessions_send` by default:** `SUBAGENT_TOOL_DENY_ALWAYS` still includes `sessions_send` with the comment `subagents communicate through announce chain`, so the requested capability is not available out of the box.
  - file: [src/agents/pi-tools.policy.ts:34](https://github.com/openclaw/openclaw/blob/a59d1bd46d9fc0bd2d9194b3f17db2561efc51e3/src/agents/pi-tools.policy.ts#L34)
  - command: `nl -ba src/agents/pi-tools.policy.ts | sed -n '1,220p'`
  - sha: [a59d1bd46d9f](https://github.com/openclaw/openclaw/commit/a59d1bd46d9fc0bd2d9194b3f17db2561efc51e3)
- **Docs still say leaf sub-agents do not get session tools:** The session tool docs say default leaf sub-agents do not get session tools, and define `tree` visibility as `Current session + spawned sub-agents`, which excludes the spawning parent.
  - file: [docs/concepts/session-tool.md:114](https://github.com/openclaw/openclaw/blob/a59d1bd46d9fc0bd2d9194b3f17db2561efc51e3/docs/concepts/session-tool.md#L114)
  - command: `nl -ba docs/concepts/session-tool.md | sed -n '1,180p'`
  - sha: [a59d1bd46d9f](https://github.com/openclaw/openclaw/commit/a59d1bd46d9fc0bd2d9194b3f17db2561efc51e3)
- **Tree visibility enumerates only descendants:** `listSpawnedSessionKeys()` queries `sessions.list` with `spawnedBy: requesterSessionKey`, so a child session's visible tree is built from its own spawned descendants, not its parent.
  - file: [src/agents/tools/sessions-resolution.ts:50](https://github.com/openclaw/openclaw/blob/a59d1bd46d9fc0bd2d9194b3f17db2561efc51e3/src/agents/tools/sessions-resolution.ts#L50)
  - command: `nl -ba src/agents/tools/sessions-resolution.ts | sed -n '1,220p'`
  - sha: [a59d1bd46d9f](https://github.com/openclaw/openclaw/commit/a59d1bd46d9fc0bd2d9194b3f17db2561efc51e3)
- **Visibility guard still forbids child-to-parent sends under `tree`:** The session access guard rejects non-self targets under `tools.sessions.visibility=tree` unless they are present in `spawnedKeys`, so a sub-agent cannot use `sessions_send` to reach its parent session via the default tree-scoped access model.
  - file: [src/agents/tools/sessions-access.ts:242](https://github.com/openclaw/openclaw/blob/a59d1bd46d9fc0bd2d9194b3f17db2561efc51e3/src/agents/tools/sessions-access.ts#L242)
  - command: `nl -ba src/agents/tools/sessions-access.ts | sed -n '1,280p'`
  - sha: [a59d1bd46d9f](https://github.com/openclaw/openclaw/commit/a59d1bd46d9fc0bd2d9194b3f17db2561efc51e3)
- **There is only a broader config override, not the requested parent-only mechanism:** Current policy code allows `tools.subagents.tools.allow`/`alsoAllow` to re-enable `sessions_send`, but that is a generic override and not a dedicated parent-only send path like the issue requests.
  - file: [src/agents/pi-tools.policy.ts:81](https://github.com/openclaw/openclaw/blob/a59d1bd46d9fc0bd2d9194b3f17db2561efc51e3/src/agents/pi-tools.policy.ts#L81)
  - command: `nl -ba src/agents/pi-tools.policy.ts | sed -n '1,220p'`
  - sha: [a59d1bd46d9f](https://github.com/openclaw/openclaw/commit/a59d1bd46d9fc0bd2d9194b3f17db2561efc51e3)

## Risks / Open Questions

- Current main has a partial workaround: operators can re-enable `sessions_send` for sub-agents and broaden `tools.sessions.visibility`, but that is wider than the requested parent-only safety boundary and does not prove the feature is implemented.
