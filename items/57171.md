## Summary

Close as `clawhub`: current `main` does not implement `postSearchCommand`, and the repo’s documented direction pushes niche memory enrichment into plugin-owned surfaces rather than adding a new core QMD shell hook.

## Close Comment

Codex review: closing this as a better fit for plugin space than OpenClaw core.

- `main` still does not expose `memory.qmd.postSearchCommand` or `memory.qmd.postSearchTimeoutMs`; the current QMD config surface in `src/config/types.memory.ts:13`, `src/config/zod-schema.ts:108`, and `docs/reference/memory-config.md:392` omits both fields.
- The current QMD search path in `extensions/memory-core/src/memory/qmd-manager.ts:1053` goes straight from QMD results into the existing trim/clamp/diversify pipeline. There is no post-search external command hook on `main`.
- Repo scope points this kind of user-specific enrichment toward plugins, not core: `VISION.md:52` says optional capability should usually ship as plugins, and memory is already treated as a plugin slot.
- OpenClaw already exposes plugin seams for adjacent memory/search customization in `docs/plugins/sdk-overview.md:102` and `docs/plugins/sdk-overview.md:200`, plus a documented companion-knowledge-layer pattern in `docs/plugins/memory-wiki.md:10`.
- The exact core implementation proposed here was already tried in PR `#57183`, and that PR is closed, which is consistent with treating this as plugin-owned rather than adding a new core shell-exec surface. This was a Codex review. ([github.com](https://github.com/openclaw/openclaw/pull/57183))
