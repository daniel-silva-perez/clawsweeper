## Summary

Keep this PR open. Current `main` still lacks an `amazon-bedrock-mantle` `config.discovery.enabled` opt-out: the plugin manifest rejects plugin config, the catalog path calls Mantle discovery unconditionally, and the resolver still tries IAM bearer-token generation when no explicit Mantle bearer token is set. The linked bug remains open and the supplied discussion includes an EC2 confirmation of the log spam plus agreement that a discovery gate is the right fix. The PR is not obsolete, but it should be adapted to current live-config patterns and get regression coverage before landing.

## Best Possible Solution

Keep this PR open and use it as the implementation candidate for the open linked bug. Update it against current `main` to add the Mantle manifest schema/uiHints, resolve the current plugin config using the established live-config pattern, gate discovery before IAM token generation, and add regression tests for `discovery.enabled: false`, `true`, and unset/default behavior. After that, land it to resolve #67288.

## Risks / Open Questions

- The PR patch in the supplied context captures `api.pluginConfig` at registration time; current main has moved similar discovery controls toward resolving plugin config from the live runtime snapshot, so the PR likely needs a small rebase/adaptation before merge.
- The PR lacks a regression test; the earlier closed PR #67311 described resolver-level tests for disabled, enabled, and default discovery states that would be useful to carry forward.
- A catalog-only guard fixes the gateway catalog path, but putting the config gate in or alongside `resolveImplicitMantleProvider` would better protect direct callers and make the behavior easier to unit test.
