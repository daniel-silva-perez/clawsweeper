## Summary

Keep open. Current `main` does not implement the requested behavior: project-local `.env` values for `OPENCLAW_CONFIG_PATH` and `OPENCLAW_STATE_DIR` are still explicitly blocked during CLI dotenv loading, and the shipped tests assert that they remain ignored. The latest release snapshot I could inspect (`v2026.4.23`) still contains the same blocklist.

## Evidence

- **Workspace `.env` still blocks config/state overrides on main:** `BLOCKED_WORKSPACE_DOTENV_KEYS` still includes both `OPENCLAW_CONFIG_PATH` and `OPENCLAW_STATE_DIR`, and the broader `OPENCLAW_` prefix is fail-closed for workspace `.env` keys.
  - file: [src/infra/dotenv.ts:45](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/src/infra/dotenv.ts#L45)
  - command: `nl -ba src/infra/dotenv.ts | sed -n '1,140p'`
  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
- **CLI bootstrap uses the blocked workspace dotenv loader before fallback:** `loadCliDotEnv()` reads `process.cwd()/.env` via `loadWorkspaceDotEnvFile(...)`, then loads the global fallback from `resolveStateDir(process.env)`. Because the workspace loader blocks these keys, project-local `.env` cannot steer config/state path resolution during bootstrap.
  - file: [src/cli/dotenv.ts:8](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/src/cli/dotenv.ts#L8)

  - file: [src/infra/dotenv.test.ts:189](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/src/infra/dotenv.test.ts#L189)
