## Summary

Current main still reuses a fresh stored agent session for `openclaw agent --agent <id>` when no `--session-id` is supplied. The code and tests show the existing `sessionId` is intentionally reused until normal session reset freshness expires, and there is still no `--new-session`/`--fresh` flag or resume banner for this CLI path.

## Best Possible Solution

Keep the issue open and add an explicit operator-safe path for one-shot CLI runs: likely a `--new-session`/`--fresh` flag that forces a new `sessionId` for the selected session key in both gateway and local paths, plus a concise non-JSON stderr banner when `openclaw agent --agent <id>` resumes an existing session without an explicit `--session-id`. Cover both `agent-via-gateway` and embedded `agentCommand` session resolution with tests and update `docs/cli/agent.md`.
