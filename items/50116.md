## Summary

Current main still has the Windows-on-POSIX session path bug this PR targets. The PR should stay open, but it needs rework because the proposed basename-only rewrite can lose agent/root context and lacks regression tests.

## Best Possible Solution

Keep the PR open for maintainer review, but request changes before merge: detect Windows drive-letter absolute paths on POSIX using a shared helper or `path.win32`, preserve/validate the `agents/<agentId>/sessions/<file>` structure instead of collapsing to a bare basename, reuse the existing realpath/agent fallback containment behavior, reject or fall back on degenerate unsafe inputs, and add focused Vitest coverage in the session path tests for backslash, forward-slash, trailing-separator, and cross-agent Windows-authored paths.

## Evidence

- **Current resolver still treats Windows drive paths as relative on POSIX:** `resolvePathWithinSessionsDir` uses `path.isAbsolute(trimmed)` before normalizing, and later accepts non-empty relative-looking values that do not start with `..`. On POSIX, a drive-letter path such as `C:\Users\foo\sessions\abc.jsonl` is not absolute, so this code path still resolves it under the current sessions directory as a literal relative name.

## Risks / Open Questions

- The PR's current basename-only rewrite can silently alias a Windows-authored transcript from another agent/root into the current agent's sessions directory.
- Without regression coverage for both `C:\...` and `C:/...` sessionFile values on POSIX, this bug can reappear or be fixed only for one separator form.
- A correct fix needs clear behavior for degenerate Windows paths and for paths that do not preserve the expected `agents/<agentId>/sessions/<file>` structure.
