## Summary

Keep this PR open. Current main still does not lazily initialize ACP for an agent configured with `runtime.type: "acp"` when the TUI/webchat reply-dispatch path sees no persisted ACP metadata; it returns `null` and continues into embedded dispatch. The PR targets a real remaining product gap, but its current branch still needs maintainer review for the policy/side-effect ordering concerns raised in review comments.

## Best Possible Solution

Keep the PR open and ask for a narrower, policy-safe implementation: lazily initialize ACP in `tryDispatchAcpReply` for `runtime.type: "acp"` agents only after existing policy and no-op-turn gates, normalize `runtime.acp` fields consistently, re-resolve the session unconditionally after initialization, and add focused tests for ACP-configured versus non-ACP agents. Treat CLI lazy-init parity as separate scope unless maintainers explicitly want it in this PR.
