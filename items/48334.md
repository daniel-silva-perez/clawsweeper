## Summary

Keep open. Current `main` still uses the non-blocking/read-only context-token lookup in the cron persistence path, and `status` still prefers stored `entry.contextTokens` while disabling async warmup, so the stale `200000` session value described in PR #48334 is not repaired on `main`. I did not find an equivalent merged implementation or regression coverage in the current checkout.

## Evidence
