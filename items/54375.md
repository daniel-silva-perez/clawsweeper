## Summary

Current `main` does not implement this PR. It intentionally keeps DBus bus-unavailable results as an error in `isSystemdServiceEnabled`, while handling that probe non-fatally only in higher-level install/configure flows. Closing as implemented or cannot-reproduce would be unsupported.

## Evidence

- **Current main still throws for bus-unavailable in `isSystemdServiceEnabled`:** `isSystemdServiceEnabled` returns `false` only for missing systemctl or disabled/not-found unit states; all other failures, including DBus bus-unavailable details, still throw.

  - file: [src/daemon/systemd.test.ts:268](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/daemon/systemd.test.ts#L268)

## Risks / Open Questions

- The PR as written would change a transport/infrastructure error into 'disabled', which can skip Linux systemd cleanup paths such as uninstall.
- Current main already solves the likely install/configure symptom at higher layers, but the lower-level semantic change proposed here is still not present, so closing as implemented would be inaccurate.
