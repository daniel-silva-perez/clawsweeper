## Summary

Keep PR #52601 open. Current main still routes FreeBSD write/copy operations through the Python pinned-write helper, the strict non-Windows device/inode identity comparison remains in place, and the PR discussion identified a real crash-safety tradeoff that needs maintainer judgment rather than cleanup closure.

## Best Possible Solution

Keep this PR open for maintainer review. The remaining product decision is whether to accept the narrow FreeBSD legacy fallback with tests documenting the behavior and crash-safety tradeoff, or replace it with a pinned-write identity fix that normalizes FreeBSD `dev_t` values without changing all-or-nothing write semantics.
