## Summary

Keep this issue open. Current main still treats WebChat/Control UI as an internal WebChat surface that defaults to the selected agent's main session, while identityLinks resolution is only exercised when a direct peer id is supplied by a channel route. The UI has manual session switching, but there is no supported WebChat canonical-direct identity mode or docs clarification that identityLinks excludes WebChat/operator surfaces.

## Best Possible Solution

Keep the issue open for maintainer product/API judgment. The likely path is either an explicit WebChat/Control UI session mode that can resolve a configured WebChat identity such as webchat:default into the same canonical direct key as linked Telegram DMs, or a docs/config clarification that identityLinks applies only to external direct-channel identities and WebChat remains an operator/main-session surface unless manually switched.
