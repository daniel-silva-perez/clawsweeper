## Summary

Keep this issue open. Current main still treats WebChat/Control UI as an operator/internal WebChat surface selected by an existing session key, and the direct-message identityLinks path still requires a concrete direct peer id. The provided comments and related reports do not include a maintainer decision, merged fix, or canonical duplicate that resolves the requested WebChat canonical-direct mode or docs clarification.

## Best Possible Solution

Keep the issue open for maintainer product/API judgment. The likely fix is either to add an explicit WebChat/Control UI session mode that can bind a configured WebChat identity such as webchat:default to the same canonical direct key as linked Telegram DMs, or to update the public session, channel-routing, and WebChat docs to state that identityLinks applies only to direct channel peers and WebChat remains an operator/main-session surface unless the user manually selects another session.
