## Summary

Keep open. Current `main` still sends proactive IRC messages through a transient connection whenever `sendMessageIrc` is called without a reply-scoped client, so the reported bug is not implemented on `main`. The PR’s approach is plausible, but it is not a strong close candidate because the codebase still lacks the fix and the proposed process-local registry has unresolved edge cases for standalone CLI sends and join timing.

## Evidence

- **Current main still uses transient IRC sends:** `sendMessageIrc` only reuses `opts.client`; otherwise it opens a fresh IRC connection, sends `PRIVMSG`, and immediately quits. There is no active monitor-client lookup on `main`.
  - file: [extensions/irc/src/send.ts:81](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/extensions/irc/src/send.ts#L81)
  - command: `nl -ba extensions/irc/src/send.ts | sed -n '1,220p'`
  - sha: [f03252aaf9cc](https://github.com/openclaw/openclaw/commit/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022)
- **Monitor client is not shared for outbound reuse:** `monitorIrcProvider` keeps its persistent `client` in local state and returns only `stop()`. On `main` it does not register that client anywhere for proactive outbound sends.
  - file: [extensions/irc/src/monitor.ts:60](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/extensions/irc/src/monitor.ts#L60)
  - command: `nl -ba extensions/irc/src/monitor.ts | sed -n '1,240p'`
  - sha: [f03252aaf9cc](https://github.com/openclaw/openclaw/commit/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022)
- **IRC outbound path is still direct, not gateway-backed:** The IRC plugin defines outbound send handlers but no gateway delivery mode, and core outbound delivery defaults non-gateway channels to direct local delivery. That means `openclaw message send` still executes IRC sends in the caller process rather than through the gateway’s live monitor runtime.
  - file: [src/infra/outbound/message.ts:239](https://github.com/openclaw/openclaw/blob/f03252aaf9cc946dd6a4f6189f30d5c5dcd64022/src/infra/outbound/message.ts#L239)

- **Join timing remains an edge case for the proposed fix:** `connectIrcClient()` marks the client ready on numeric `001`, then sends `JOIN` commands and resolves readiness immediately, before any join confirmation is observed. A monitor-client registry alone would not guarantee the client has actually joined channels yet.
