## Summary

Keep this issue open. Current main intentionally prevents workspace hooks from overriding managed/plugin/bundled hooks with the same name, and the exact warning path in the report still exists. The report also identifies a real diagnostics gap: hooks with empty `metadata.openclaw.events` are skipped by the loader, while the status/check path currently treats enabled hooks as loadable based on config and requirements only, not event registration viability.

## Best Possible Solution

Keep the managed-over-workspace precedence unless maintainers explicitly change the hook trust model, but make the failure actionable: surface same-name collisions and empty `metadata.openclaw.events` in `openclaw hooks list/info/check` and gateway startup diagnostics, mark enabled hooks with no events as not ready, and document that users should rename workspace hooks or remove/update the managed hook pack when a managed hook shadows local workspace code.
