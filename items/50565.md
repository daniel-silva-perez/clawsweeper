## Summary

Keep open. Current `main` does not show a channel-aware fix for tool execution attribution. The latest tool-execution diagnostics added on April 24, 2026 still carry only run/session identifiers and trace metadata, not the originating channel, and the tool-wrapping call sites do not pass channel context.

## Evidence

- **Tool hook context has no channel field:** `HookContext` only carries `agentId`, `sessionKey`, `sessionId`, `runId`, `trace`, and `loopDetection`; there is no channel or subsystem field available to tag tool execution by originating channel.
  - file: [src/agents/pi-tools.before-tool-call.ts:22](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/src/agents/pi-tools.before-tool-call.ts#L22)
  - sha: [ebd7f19a3b20](https://github.com/openclaw/openclaw/commit/ebd7f19a3b20001960474ec9ba9cc424e45f7b23)
- **Tool execution events omit channel attribution:** `wrapToolWithBeforeToolCallHook()` builds `eventBase` from `runId`, `sessionKey`, `sessionId`, `trace`, `toolName`, `toolCallId`, and `paramsSummary`, then emits `tool.execution.started/completed/error` with that payload only.
  - file: [src/agents/pi-tools.before-tool-call.ts:473](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/src/agents/pi-tools.before-tool-call.ts#L473)
  - sha: [ebd7f19a3b20](https://github.com/openclaw/openclaw/commit/ebd7f19a3b20001960474ec9ba9cc424e45f7b23)
- **Call site does not pass channel into tool hooks:** `pi-tools.ts` wraps tools with hook context containing `sessionKey`, `sessionId`, `runId`, and `trace`, but no channel identifier, so downstream tool execution logging/events cannot distinguish Telegram vs Discord vs WhatsApp vs Feishu.
  - file: [src/agents/pi-tools.ts:707](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/src/agents/pi-tools.ts#L707)
  - sha: [ebd7f19a3b20](https://github.com/openclaw/openclaw/commit/ebd7f19a3b20001960474ec9ba9cc424e45f7b23)
- **Diagnostic schema still lacks channel on tool execution events:** `DiagnosticToolExecutionBaseEvent` defines `runId`, `sessionKey`, `sessionId`, `toolName`, `toolCallId`, and `paramsSummary` only. Unlike `model.usage` and other events, it does not include a `channel` field.
  - file: [src/infra/diagnostic-events.ts:163](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/src/infra/diagnostic-events.ts#L163)
  - sha: [ebd7f19a3b20](https://github.com/openclaw/openclaw/commit/ebd7f19a3b20001960474ec9ba9cc424e45f7b23)
- **Current tests assert tool execution events without channel:** The e2e test for emitted tool execution events matches only `runId`, `sessionKey`, `sessionId`, `toolName`, `toolCallId`, `paramsSummary`, and `trace`; there is no expectation for channel attribution.
  - file: [src/agents/pi-tools.before-tool-call.e2e.test.ts:378](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/src/agents/pi-tools.before-tool-call.e2e.test.ts#L378)
  - sha: [ebd7f19a3b20](https://github.com/openclaw/openclaw/commit/ebd7f19a3b20001960474ec9ba9cc424e45f7b23)
- **Latest relevant history adds diagnostics, not channel tagging:** Commit `cead2ea4b10688250068bbbe41cb5da19ffcc0b6` (`feat(diagnostics): emit tool execution events`) is the newest relevant change I found. It adds structured tool execution events on current main, but the shipped schema/context above still lacks channel attribution.
