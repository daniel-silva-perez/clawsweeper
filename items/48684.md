## Summary

Keep open. Current `main` at `4013c658537ee2051c44186f197b0710e84c9049` still routes inbound debounce through `hasControlCommand()` only, while abort-trigger handling for bare phrases lives separately in `isControlCommandMessage()`. The reported mismatch is still present on `main` and in the latest release tag `v2026.4.22`, so there is no strong basis to close this as fixed or obsolete.

## Risks / Open Questions

- Bare abort phrases like `stop` or `wait` can still sit behind inbound debounce instead of preempting immediately.
- If another message arrives in the debounce window, the abort text can be merged with later content and cease to match abort-trigger detection, causing silent failure of the stop intent.
- The current tests cover blank text, media, and slash commands such as `/status`, but do not cover bare abort phrases in `src/channels/inbound-debounce-policy.test.ts`, so this regression remains unguarded.
