## Summary

Keep open. Current main still has the reported auto-reply CLI fallback auth-profile gap: fallback execution can run through google-gemini-cli while provider-scoped auth drops the profile id, and CLI session reuse still invalidates stored/current authProfileId mismatches.

## Best Possible Solution

Keep this issue open. The maintainer fix should update the auto-reply fallback CLI path to use the same provider/runtime-aware auth planning as the newer runtime paths, or otherwise resolve a google-gemini-cli-appropriate auth profile before CLI session reuse, without reintroducing cross-provider auth leakage from #1156. Add regression coverage for google-gemini-cli fallback session reuse and stale cliSessionBindings where authProfileId is missing on one side.
