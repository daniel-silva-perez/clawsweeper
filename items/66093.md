## Summary

Keep open. Current main still matches the reported fallback CLI auth-profile gap: auto-reply fallback clears non-primary provider auth profiles, while CLI session reuse still invalidates when stored and current auth profile ids differ. The issue is recent, concrete, and not superseded by the current command-path runtime auth planning.

## Best Possible Solution

Keep this issue open. The maintainer fix should update the auto-reply fallback CLI path to use provider/runtime-aware auth planning, or otherwise resolve a google-gemini-cli-appropriate auth profile before CLI session reuse, while preserving cross-provider auth isolation from #1156. Add regression coverage for google-gemini-cli fallback session reuse and stale `cliSessionBindings` where `authProfileId` is missing on one side.
