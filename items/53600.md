## Summary

Current main has relevant mechanics and partial diagnostics for several surfaces named in the report, but it still does not contain the requested native-VPS reasoning-latency analysis or instrumentation plan with layer-by-layer latency estimates, quick-win config guidance, code optimization targets, and explicit instrumentation gaps.

## Best Possible Solution

Keep this open and scope it into a bounded native-Linux gateway performance investigation. The best maintainer path is to capture one timing trace covering context/bootstrap assembly, prompt/token estimation, memory query embedding/search, compaction trigger and model-call duration, queue waits, subagent spawn and announce-back, Telegram preview/final delivery, bundled hook execution, auth fallback decisions, and process RSS/heap/GC; then publish a VPS/debugging runbook with quick-win config guidance and file narrower optimization tickets for measured hot spots.
