## Summary

Keep open. Current main has Bonjour hardening for ciao cancellation rejections, stuck probing/announcing watchdog loops, Docker bridge defaults, and one IPv4 interface-churn assertion, but it does not cover the issue's specific `getNetAddress` IP/netmask version assertion from inbound mDNS packet handling. The report is recent, concrete, and follow-up comments say related ciao gateway-exit failures still occur on 2026.4.24 on non-Docker platforms.

## Best Possible Solution

Keep this issue open and fix it in the bundled `@openclaw/bonjour` surface or dependency pin: either upgrade/patch `@homebridge/ciao` so `MDNSServer.handleMessage` treats IP/netmask mismatches as malformed packets, or add an OpenClaw-owned defensive layer that prevents ciao packet-parse assertions from reaching the global uncaught-exception exit path. Add focused coverage for the exact `IP address version must match` signature, and keep the Docker bridge default fix tracked separately via #71879/#71881 because it only covers one deployment class.

## Risks / Open Questions

- Closing this now would conflate the related ciao cancellation/watchdog fixes with a different synchronous assertion path that still appears unhandled on current main.
- The documented workaround, `OPENCLAW_DISABLE_BONJOUR=1` or disabling the `bonjour` plugin, avoids LAN multicast advertising but removes local Bonjour discovery rather than fixing malformed-packet resilience.
