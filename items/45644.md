## Summary

Issue #45644 asks for conditional routing and multi-model orchestration in core, but current main intentionally keeps agent model config static and exposes plugin hooks for request-time model/provider routing. The project vision also says optional capability should usually ship as plugins and that heavy orchestration layers should not land in core, so this belongs in ClawHub/plugin space rather than OpenClaw core.

## Risks / Open Questions

- A future docs-first effort could still standardize a recommended plugin pattern for intent-based routing without moving the feature into core.
- The existing hook seam covers deterministic per-run routing; full parallel multi-model collaboration would still require plugin-specific orchestration design.

## Close Comment

Thanks for the write-up. After a Codex review against current `main` (`f03252aaf9cc946dd6a4f6189f30d5c5dcd64022`), this looks like a better fit for ClawHub/plugin space than OpenClaw core.

Concrete reasons:
- core agent model config is still intentionally limited to `string` or `{ primary, fallbacks }` in `src/config/types.agents-shared.ts:8` and `src/config/zod-schema.agent-model.ts:3`
- OpenClaw already exposes the plugin seam for request-time routing through `before_model_resolve`, including attachment metadata plus `modelOverride` / `providerOverride` in `src/plugins/hook-before-agent-start.types.ts:7`
- the docs explicitly point model/provider override work at that hook in `docs/concepts/agent-loop.md:93` and `docs/plugins/architecture.md:98`
- `VISION.md:54` says optional capability should usually ship as plugins, and `VISION.md:107` explicitly calls out heavy orchestration layers duplicating existing agent/tool infrastructure as out of scope for core

Given that direction, I’m closing this as a ClawHub/plugin request rather than a core OpenClaw feature. If someone wants to pursue this, the supported path is a plugin that uses `before_model_resolve` for deterministic routing and any higher-level orchestration outside core.
