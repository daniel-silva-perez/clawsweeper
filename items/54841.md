## Summary

Keep #54841 open. Current main still ignores the configured Feishu `webhookPath` in the active webhook request handler: it reads and logs the path and includes it in status/rate-limit keys, but never compares `req.url`/pathname before body parsing, signature validation, challenge handling, or `eventDispatcher.invoke`. The linked PR #54904 is a plausible direct fix, but it is still open and not present on main, so the issue should remain open until that or an equivalent fix lands.

## Best Possible Solution

Keep #54841 open and review/land PR #54904 or an equivalent narrow Feishu plugin fix. The fix should normalize the configured `webhookPath`, compare the incoming request pathname before any body parsing/signature validation/dispatch, reject mismatches with 404 or an explicit routing response, preserve query-string acceptance on the configured path, and add a regression test in the Feishu webhook security/transport tests proving wrong-path requests do not reach `eventDispatcher.invoke`.
