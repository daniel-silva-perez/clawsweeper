## Summary

Keep #50522 open. Current main still has the Discord adapter pass `ThreadStarterBody` through unconditionally on thread messages, while `previousTimestamp` is already available in the same function and Slack uses the proposed guard. The core reply path can still prepend a thread-starter context block on follow-up turns when a channel supplies `ThreadStarterBody`, so the PR is still a focused, useful fix for the linked Discord bug rather than obsolete cleanup.

## Best Possible Solution

Refresh and land the narrow Discord adapter guard, ideally with a Discord regression test proving that when `readSessionUpdatedAt` returns an existing timestamp, `ThreadStarterBody` is omitted from the dispatched/recorded context while new thread sessions still receive the starter body.
