## Summary

Keep open. Current main at b565e6e96310c03e81b32caf82894ce89f32fec5 still does not include the `file` alias in the read/write/edit display path resolver, and the read-start diagnostic still ignores both `file` and `filePath`. The latest release tag has no diff from current main for these touched files, so this is not shipped as fixed.

## Best Possible Solution

Keep #54882 open until a narrow fix lands or maintainers explicitly decide that read/write/edit tools should be canonical `path` only. A complete fix should update `resolvePathArg()` for `file`, align `handleToolExecutionStart()` with the same alias set, add focused regression coverage for `{ file: ... }`, and update the generated/shared display config only if that fallback remains part of the display contract.
