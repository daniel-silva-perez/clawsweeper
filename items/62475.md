## Summary

Current `main` has documented heartbeat-based cache warming and the cache-aware timestamp/runtime-context foundations referenced in the issue, but it does not implement the requested dedicated non-turn prompt-cache keep-warm ping path or config surface.

## Evidence

- **Prompt-caching docs point users to heartbeat:** Public prompt-caching docs say heartbeat can keep cache windows warm and reduce repeated cache writes after idle gaps, configured via `agents.defaults.heartbeat.every`. There is no separate prompt-cache keep-warm feature documented there.
  - file: [docs/reference/prompt-caching.md:82](https://github.com/openclaw/openclaw/blob/ecac696643c604c6d854341bfe6424bfa31423e9/docs/reference/prompt-caching.md#L82)
  - command: `nl -ba docs/reference/prompt-caching.md | sed -n '78,96p'`
  - sha: [ecac696643c6](https://github.com/openclaw/openclaw/commit/ecac696643c604c6d854341bfe6424bfa31423e9)
- **Current keep-warm mechanism is still a full heartbeat turn:** Agent config supports `session: "main"`, `target: "none"`, and `isolatedSession: false`, but the same docs explicitly say heartbeats run full agent turns and shorter intervals burn more tokens. That is materially different from the issue's proposed invisible non-turn `max_tokens: 1` ping mechanism.
  - file: [docs/gateway/config-agents.md:507](https://github.com/openclaw/openclaw/blob/ecac696643c604c6d854341bfe6424bfa31423e9/docs/gateway/config-agents.md#L507)
