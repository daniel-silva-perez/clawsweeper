## Summary

Keep #71609 open. Current main has partial Control UI auth hardening, including explicit shared-token precedence and a bounded cached-device-token retry, but the specific stale cached device-token recovery path is still not fully implemented: browser cleanup is narrow, AUTH_TOKEN_MISMATCH can still schedule reconnects when no safe retry was used, a checked-in test preserves that reconnect behavior, and the gateway still emits the rotate/reissue wording called out by the issue.

## Best Possible Solution

Keep the issue open and fix the browser Control UI auth recovery path. The maintainer path should clear stale cached device auth on AUTH_DEVICE_TOKEN_MISMATCH or stale stored-token auth, stop automatic reconnect loops when no safe retry is possible or after the one-shot retry fails, avoid burning the auth rate limiter, keep the stable app-level clientInstanceId behavior, and replace rotate/reissue guidance with stale identity cleanup/reconnect guidance. Add focused browser tests for the scope-upgrade stale-token case and the non-looping recovery path.

## Evidence
