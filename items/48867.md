## Summary

Keep #48867 open. Current main includes the Chrome MCP existing-session command/argument work from #71560, but this report is specifically about Android/Termux subprocess stdio lifecycle: chrome-devtools-mcp sees stdin close and exits via parent-death detection. The browser plugin still launches Chrome MCP through the SDK `StdioClientTransport` path, and I found no current main code, test, changelog entry, or doc proving a Termux stdin-lifetime fix. A maintainer also explicitly left this issue open after #71560 as a separate process-lifecycle bug.

## Best Possible Solution

Keep #48867 open and track a separate process-lifecycle fix for Chrome MCP stdio subprocesses on Android/Termux. The likely implementation path is to move the browser Chrome MCP launch onto an OpenClaw-owned stdio transport or equivalent process manager that keeps stdin open until intentional shutdown, with a focused regression test proving the spawned server does not observe stdin EOF immediately. Treat #71560 as the existing-session command/args fix, not the canonical fix for this issue.
