## Summary

Keep open. Current `main` still hardcodes a 120s timeout for non-local builtin embedding batch requests, while `remote.batch.timeoutMinutes` remains a separate provider-batch wait setting. The PR is not implemented on `main`, and the reported timeout behavior for remote indexing remains plausible.

## Evidence

- **Hardcoded remote batch timeout still exists on main:** `EMBEDDING_BATCH_TIMEOUT_REMOTE_MS` is still `2 * 60_000`, and `resolveEmbeddingTimeout("batch")` still returns that constant for all non-local batch requests.
  - file: [extensions/memory-core/src/memory/manager-embedding-ops.ts:52](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/extensions/memory-core/src/memory/manager-embedding-ops.ts#L52)
  - sha: [45146913007d](https://github.com/openclaw/openclaw/commit/45146913007da723b4f06aba6a66cf4a695974c0)
- **Builtin indexing still uses the hardcoded timeout path:** `embedChunksInBatches()` routes normal indexing through `embedBatchWithRetry()` / `embedBatchInputsWithRetry()`, which call `resolveEmbeddingTimeout("batch")` before timing out with `memory embeddings batch timed out after ...`.
  - file: [extensions/memory-core/src/memory/manager-embedding-ops.ts:91](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/extensions/memory-core/src/memory/manager-embedding-ops.ts#L91)
  - sha: [45146913007d](https://github.com/openclaw/openclaw/commit/45146913007da723b4f06aba6a66cf4a695974c0)
- **Configured batch timeout is only wired to provider batch APIs:** `resolveBatchConfig()` still derives `timeoutMs` from `remote.batch.timeoutMinutes`, and the async provider-batch path passes `this.batch.timeoutMs` only to `batchEmbed(...)`. The fallback path still returns to `embedChunksInBatches()`.
  - file: [extensions/memory-core/src/memory/manager-sync-ops.ts:1054](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/extensions/memory-core/src/memory/manager-sync-ops.ts#L1054)

  - file: [packages/memory-host-sdk/src/host/batch-status.ts:45](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/packages/memory-host-sdk/src/host/batch-status.ts#L45)

## Risks / Open Questions

- Closing this PR as already implemented would be incorrect: current `main` still preserves the 120s hardcoded timeout on the non-batch builtin indexing path described in the report.
- The PR as written also appears to change the meaning of `remote.batch.timeoutMinutes` beyond the currently documented async provider-batch scope, so it should stay open for maintainer judgment rather than cleanup closure.
