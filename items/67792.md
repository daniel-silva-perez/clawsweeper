## Summary

Keep open. Current main still builds Telegram model buttons from the provider-local model id, so an OpenRouter nested id such as `openai/gpt-5.4-mini` is still displayed without the `openrouter/` prefix unless a catalog display name happens to override it. The related #67797 thread is adjacent OpenRouter reasoning behavior, not a duplicate or shipped fix for this Telegram label issue.

## Best Possible Solution

Keep this issue open and fix it in the Telegram bundled plugin. Update `extensions/telegram/src/model-buttons.ts` so the fallback display label for nested provider-local model ids, and possibly all model rows, uses the canonical `provider/model` text while leaving callback data unchanged. Add a regression test in `extensions/telegram/src/model-buttons.test.ts` for `provider: "openrouter"` and `models: ["openai/gpt-5.4-mini"]` expecting the visible button text to include `openrouter/openai/gpt-5.4-mini`.
