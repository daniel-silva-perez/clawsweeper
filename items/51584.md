## Summary

Current main does not implement or supersede the PR's static built-in channel fallback for `sandbox explain`. The command still resolves elevated channel diagnostics through the runtime plugin channel registry only, and that helper explicitly returns null when no active channel registry is loaded. The linked Telegram runtime path has improved since the issue report by using agent-scoped slash session keys and recording inbound metadata, but `sandbox-explain.ts` still does not use the static built-in channel registry or recorded `origin.provider`, so the PR remains plausibly useful rather than obsolete.

## Best Possible Solution

Keep the PR or its canonical bug #51245 open. The best maintainer path is to land a focused sandbox-explain fix that normalizes built-in channels with the static `normalizeChatChannelId` fallback wherever active channel is resolved, adds regression coverage for Telegram slash elevated diagnostics without a loaded plugin registry, and decides whether legacy raw `telegram:slash:<id>` input and `origin.provider` session metadata should be supported in the same patch.

## Risks / Open Questions

- The PR's one-line fallback is likely directionally right for modern `agent:<agent>:telegram:slash:<id>` keys, but may be incomplete for the exact legacy raw `telegram:slash:<id>` command shown in #51245 because current inference only parses `agent:` session keys.
- A more robust fix may also need `resolveActiveChannel` to read `entry.origin.provider` or apply the same static fallback to stored `lastChannel`/`channel` values, not only to session-key inference.
