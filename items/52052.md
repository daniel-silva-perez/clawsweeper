## Summary

Keep open. Current main still captures Telegram sticker metadata, including `Sticker.fileId`, but the agent-visible inbound user context does not project a replayable sticker handle. The PR targets a real remaining gap, but it should address the existing review concern that the surfaced `sticker_file_id` key does not match the Telegram send tool's accepted `fileId`/`stickerId` arguments.

## Best Possible Solution

Keep the PR open and revise it rather than close it. The best path is a narrow prompt-context fix that keeps sticker data out of stable system metadata, exposes the replayable handle only in per-turn untrusted conversation info, and aligns the exposed name with the send tool contract. Either surface a key the model can directly use with the existing tool (`fileId`/`stickerId`) or add `sticker_file_id` as an accepted, documented Telegram send alias with schema and action-runtime tests, plus the PR's focused inbound-meta tests for system exclusion, webchat omission, and Telegram inclusion.
