## Summary

Keep this issue open. Current main has addressed adjacent pieces, including ACP task tracking and hook delivery suppression, but the main requested capability is still missing: channel-less hook-triggered ACP spawns cannot create persistent `mode:"session"` sessions without `thread:true`, the accepted spawn result still does not expose the underlying ACP/backend agent session identity, and `/hooks/agent` still runs through the isolated agent path rather than routing known ACP session keys directly into ACP `runTurn`.

## Best Possible Solution

Keep this issue open and narrow it to a conservative headless ACP persistence contract. The likely path is to add an explicit headless persistent ACP mode, or narrowly relax the `mode:"session"` thread guard when `streamTo:"parent"` and lifecycle cleanup are well-defined; expose the ACP backend/agent session identity or a documented status lookup path; add regression coverage for channel-less hook-triggered ACP `mode:"session"` spawns; and separately decide whether `/hooks/agent` should route known ACP session keys into `acpManager.runTurn` or whether a dedicated `sessions_send`/ACP send tool should own that follow-up path.

## Risks / Open Questions

- Closing this now would leave hook-triggered/headless ACP workflows without a supported persistent `mode:"session"` path.
- A fix should not simply delete the thread guard; headless persistent ACP sessions need explicit cleanup, visibility, delivery, ownership, and timeout semantics.
- ACP runtimes execute outside the OpenClaw sandbox, so exposing backend resume identity or hook-to-ACP routing should preserve existing session visibility and policy checks.
- Any future hook delivery changes should preserve the existing `deliver:false` and message-tool duplicate-suppression behavior.
