## Summary

Keep #71581 open. Current main still resolves exec-event and cron-event heartbeat wakes from the literal forced/session key path, and `resolveHeartbeatSession` only canonicalizes main aliases before returning same-agent non-main session keys. It does not consult `session.dmScope` or the single-owner allowlist helper, so the reported per-channel orphan-session behavior remains plausible and unimplemented. The related #72160 cross-reference is unrelated and closed unmerged; adjacent heartbeat/session items do not supersede this narrower request.

## Best Possible Solution

Keep this open and implement a core heartbeat/session fix. Add a small shared helper that, for same-agent direct-message session keys, maps to `agent:<id>:main` only when `session.dmScope="main"` is effective under the same single-owner allowlist/pinned-owner precondition used by inbound DMs. Apply it consistently at enqueue/preflight/wake resolution for exec-event and main-session cron-event flows, and add regression tests proving the event text is processed in the main session while delivery context remains intact.

## Risks / Open Questions

- A fix needs to keep system-event queue keys and resolved heartbeat run session in sync; redirecting only the wake would make the main-session run miss events still queued under the original per-channel key.
- Delivery routing should keep using the queued event delivery context where present, so exec follow-ups still target the originating channel/topic even if the transcript/run session is remapped to main.
