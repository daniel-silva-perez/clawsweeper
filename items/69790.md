## Summary

Keep open. Current main has session-lock guardrails, but I do not see enough source/test/release evidence to close #69790 as fixed. The reporter's standalone synthetic bot import bypasses the gateway's per-session queue and can still legitimately contend on the file lock, but the issue also reports normal Telegram inbound failures on the same DM session. Current main still routes Telegram inbound through the same embedded-agent/session-write-lock path, and there is no Telegram-specific regression coverage proving that a live gateway-owned lock no longer wedges or times out subsequent inbound handling.

## Best Possible Solution

Keep the issue open and narrow it into two maintainer-trackable outcomes: first, add a regression test or live reproduction for normal Telegram DM inbound when a prior same-session embedded run owns or leaks a session write lock; second, document or provide a supported gateway/plugin API for injecting transcribed Telegram audio as user input instead of relying on internal `dist/extensions/telegram/bot-*.js` imports. If current main cannot reproduce the normal inbound lock failure, close later with explicit reproduction proof and a note that cross-process synthetic bot injection is intentionally outside the supported lock/queue contract.
