## Summary

Keep open. Current `main` still has the raw-error parsing gap described in the issue: provider JSON payloads are only recognized when they look like OpenAI-style `{type,error,request_id}` objects, so a top-level FastAPI/Codex-style `{"detail":...}` payload would bypass sanitization and can still flow into assistant-visible text.

## Evidence

- **Raw API payload detection still excludes top-level `detail`:** `isErrorPayloadObject` accepts `type === "error"`, `request_id`/`requestId`, or a nested `error` object with `message`/`type`/`code`. It does not recognize a top-level `detail` field, so `parseApiErrorPayload()` returns `null` for the reported Codex shape.
  - file: [src/shared/assistant-error-format.ts:27](https://github.com/openclaw/openclaw/blob/40be5ad58187fd20b8defe5fc30e649190c131a8/src/shared/assistant-error-format.ts#L27)
  - command: `nl -ba src/shared/assistant-error-format.ts | sed -n '27,80p'`
  - sha: [40be5ad58187](https://github.com/openclaw/openclaw/commit/40be5ad58187fd20b8defe5fc30e649190c131a8)
- **User-visible assistant text still depends on that parser for error rewriting:** Assistant output is funneled through `sanitizeUserFacingText(..., { errorContext })`, and in error context it only rewrites raw payloads when `isRawApiErrorPayload()` or HTTP-text detection matches. A `{"detail":...}` payload misses both checks and would remain raw text.
