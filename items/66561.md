## Summary

Keep this issue open. Current main has adjacent OpenAI Codex routing/transport fixes and model-call diagnostics, but the embedded-run timeout path still uses one attempt-level deadline that aborts the active session and then normalizes timed-out assistant failover to timeout/408 without carrying whether an upstream SSE response had already started. The closed related PR did not merge, and current tests still preserve AbortError-as-timeout behavior for the relevant shape.

## Best Possible Solution

Keep the issue open and fix it in the embedded Pi runner/failover path. Track provider-call lifecycle separately from the whole-attempt timer, carry whether a request/response/SSE event started into failover observation, and distinguish no-response timeout from post-start local abort, parser/stream handling failure, or upstream explicit error. Add a focused regression test that simulates a streamed Codex response being aborted by the enclosing attempt timeout and asserts the surfaced reason/status/log fields are not the generic no-response timeout shape.
