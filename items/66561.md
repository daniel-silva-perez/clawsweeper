## Summary

Keep this issue open. Current main has the adjacent OpenAI Codex routing/transport fixes, but the specific reported failure class remains plausible: the embedded runner still uses a single whole-attempt wall-clock timeout, aborts the active session when it fires, and downstream failover/logging still normalizes that state to timeout/408 without tracking whether an upstream SSE response had already started. The closed unmerged PR #66599 did not land, and current tests explicitly preserve AbortError-as-timeout behavior rather than proving the requested distinction.

## Best Possible Solution

Keep the issue open and fix it in the embedded Pi runner/failover path: track provider stream lifecycle separately from the whole-attempt timer, record whether request/response/SSE events started, and surface/log post-start local aborts or parser failures distinctly from no-response timeouts. Add a focused regression test that simulates an active streamed Codex response being aborted by the enclosing attempt timeout and asserts the failover reason/status/log fields are not the generic no-response timeout shape.
