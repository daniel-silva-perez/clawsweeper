## Summary

Keep open. Current main partially improves the visible drain UX, but it still rejects new inbound command enqueues during the restart drain window, so the requested auto-retry/replay of unprocessed messages is not implemented. The linked Kafka PR was closed unmerged and is not present on main; it would have provided observability/event logging, not the core FIFO replay behavior requested here.

## Best Possible Solution

Keep this issue open and implement the remaining core gateway behavior: either allow drain-window inbound tasks to enter the existing command lane queue without starting until `resetAllLanes()`, or add a durable FIFO per-channel pending inbound queue for restart windows. Preserve the friendlier restart text as a fallback, and add tests covering multiple messages arriving during drain, ordering, SIGUSR1 reset replay, and full stop/start behavior if durability is included.
