## Summary

Keep open. Current `main` has core-only doctor health contributions and some narrower plugin/channel doctor contract surfaces, but it does not expose a public SDK API for registering arbitrary custom doctor checks or plugin-owned fixes. The existing discussion correctly identifies this as a missing core/plugin API design item, not something already implemented or suitable to close as ClawHub-only.

## Best Possible Solution

Keep this open as a core/plugin-SDK design item. Add a narrow, documented doctor-check extension contract with plugin source attribution, duplicate-id handling, tests, and docs; start with read-only plugin checks, then add fix support through the existing doctor repair prompting modes. The multi-gateway/channel-overlap check can be the proving use case, either bundled if maintainers want it in core or external once the API exists.

## Risks / Open Questions

- A public third-party doctor fix API would run with local config/state access; it needs explicit trust boundaries, source attribution, prompting, and safe behavior for unattended `--repair`/`--yes` modes.
- The motivating multi-gateway/Telegram allowlist drift check may need host-wide profile/config discovery, which is broader than the existing per-channel doctor adapter model and should be designed deliberately.
