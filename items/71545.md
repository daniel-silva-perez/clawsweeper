## Summary

Keep open. Current main supports configured streamable-http MCP transports, but the session runtime still caches one client/transport per server and `callTool` directly forwards to the cached client without detecting lost streamable-http sessions, rebuilding the transport, refreshing tool metadata, or retrying. PR #71545 is a focused, recent runtime fix with regression coverage and a maintainer hardening commit, so it is not obsolete or safely closeable as cleanup.

## Best Possible Solution

Keep #71545 open for normal PR review. The best code direction is to land a tested streamable-http-only recovery path in `createSessionMcpRuntime`: detect specific lost-session errors, share one in-flight reconnect per server, recreate the client with OpenClaw's custom MCP JSON schema validator, refresh tool metadata with `listTools`, retry the original call once, preserve failed reconnect retryability, and leave SSE/stdio behavior unchanged.
