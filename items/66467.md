## Summary

Keep this issue open. Current main still bundles an ACPX/runtime stack whose ACP SDK schema requires `usage_update.used` to be a number, so a `session/update` notification with `used: null` can still fail validation before OpenClaw's best-effort usage projection path sees it. The provided related ACP reports are adjacent but do not appear to be a canonical fix for this narrower schema mismatch.

## Best Possible Solution

Keep this issue open and fix the ACPX/OpenClaw ACP runtime boundary: either update/pin ACPX and the ACP SDK to a version whose `usage_update` schema accepts nullable or absent usage fields, or normalize/drop invalid nullable usage fields before SDK validation. Add a focused regression test that a `session/update` payload with `sessionUpdate: "usage_update"`, `used: null`, and a numeric `size` does not destabilize the turn and projects a best-effort status event.
