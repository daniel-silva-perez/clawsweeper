## Summary

Keep this issue open. Current main still bundles an ACPX/runtime stack where `session/update` is validated by the locked ACP SDK before OpenClaw's best-effort ACP event projection can tolerate missing numeric usage fields. The locked SDK schema still requires `usage_update.used` and `size` to be numbers, so the reported `used: null` payload can still fail validation. The existing ClawSweeper comment is consistent with current main, and I found no paired open PR or canonical duplicate that resolves this narrower schema mismatch.

## Best Possible Solution

Fix the ACPX/OpenClaw ACP runtime boundary and then add focused regression coverage. The likely paths are to update or pin ACPX and the ACP SDK to a schema that accepts nullable or absent usage fields, or to normalize/drop nullable `usage_update` fields before SDK validation rejects the notification. The regression should prove that a `session/update` payload with `sessionUpdate: "usage_update"`, `used: null`, and numeric `size` does not destabilize a Discord-bound ACP turn and projects a best-effort status event.
