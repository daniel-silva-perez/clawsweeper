## Summary

Current main still has the reported unbounded `Promise.all(filePaths.map(... fs.readFile ...))` in `readPageSummaries`, and `compileMemoryWikiVault` calls that helper during wiki compile. The CLI, gateway, and ingest paths still route through `compileMemoryWikiVault`, so the reported `wiki compile` / auto-compile behavior remains plausibly affected. The issue is recent, concrete, and not covered by an existing close reason.

## Best Possible Solution

Keep this issue open. Implement the fix in the `memory-wiki` plugin by bounding `readPageSummaries` page reads with a small default worker limit, optionally expose a compile/read concurrency config if maintainers want it, update memory-wiki docs/config schema if a config key is added, and add a regression test that instruments reads to prove peak concurrency stays within the cap while preserving deterministic page ordering.
