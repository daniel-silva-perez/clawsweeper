## Summary

Keep #71563 open. Current main still routes only abort, status, BTW, and approval updates to separate Telegram sequential lanes; `/new` and `/reset` fall through to the normal chat/topic key even though they are registered session commands. The Telegram bot installs grammY sequentialization using this key, and existing tests cover status bypass plus ordinary same-topic serialization but not session reset bypass. The PR is focused, includes regression coverage, and the review discussion's duplication concern was addressed in the current patch.

## Best Possible Solution

Keep the PR open for maintainer review. The best fix is a narrow Telegram change that routes registered session lifecycle commands (`new`, `reset`) through shared command resolution to `telegram:<chatId>:control`, with regression coverage for bare and bot-username-suffixed commands, and without adding unregistered `/clear` behavior.
