## Summary

Keep #71563 open. Current main still only sends abort/status/approval/btw Telegram updates to special sequential lanes; registered session commands `/new` and `/reset` remain ordinary chat/topic sequential keys, so the PR is not obsolete. The PR discussion also shows a focused fix with author follow-up after the Greptile duplication comment, not a low-signal or superseded PR.

## Best Possible Solution

Keep the PR open and review or adapt it as the Telegram-owned fix: route registered session lifecycle commands `/new` and `/reset` to a Telegram control sequential lane, add regression tests for plain and bot-suffixed commands, and verify the active-run/reset semantics after the command bypasses grammY's per-chat queue.

## Evidence

There is no session-control predicate on current main.
