## Summary

Keep open. Current main still lacks the requested Telegram native exec-approval delivery scoping from `approvals.exec.targets[].accountId`: the explicit forwarding path preserves `accountId`, but the native Telegram handler still gates on turn-source/session account binding and can allow every eligible bot account when no bound account is present. This matches the maintainer comment on #69916, and the related fix work is not yet landed on main.

## Best Possible Solution

Keep #69916 open until a fix lands that scopes Telegram native exec approval handling to the single configured `approvals.exec.targets[].accountId` when that target is unambiguous, while preserving existing turn-source/session-bound routing. Add a multi-account regression around `shouldHandleTelegramExecApprovalRequest` proving only the target account handles the approval and non-target accounts with the same approver `allowFrom` do not.

## Evidence

- **Native Telegram matcher still ignores explicit approval target accountId:** `matchesTelegramRequestAccount` resolves a bound account only through `resolveApprovalRequestChannelAccountId`; if no bound account is found, the final predicate allows handling for any account.
