## Summary

Keep open. Current `main` has run lifecycle and tool-progress plumbing, but it does not implement the issue's requested atomic intentâ†’execution contract. The code still models lifecycle as `start`/`end`/`error`, relies on paired `tool_execution_start`/`tool_execution_end` events to retire active items, and does not synthesize a mandatory commit/abort terminal state for every announced action when a run is interrupted.

## Evidence

- **Lifecycle is still start/end/error, not begin/commit/abort:** `handleAgentStart()` emits lifecycle `phase: "start"`, and `handleAgentEnd()` emits only `phase: "end"` or `phase: "error"`. There is no run-level commit/abort state machine or blocker payload matching the issue proposal.

  - command: `nl -ba src/agents/pi-embedded-subscribe.handlers.lifecycle.ts | sed -n '1,220p'`
  - sha: [a59d1bd46d9f](https://github.com/openclaw/openclaw/commit/a59d1bd46d9fc0bd2d9194b3f17db2561efc51e3)
- **Active tool items only retire on explicit end events:** `emitTrackedItemEvent()` increments active items on `phase === "start"` and decrements them only on `phase === "end"`. Tool starts are emitted immediately, while completion/blocked status is assigned only inside `handleToolExecutionEnd()`. There is no generic interrupted-run cleanup that converts started items into abort states.
