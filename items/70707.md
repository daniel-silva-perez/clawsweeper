## Summary

Keep PR #70707 open. Current main still responds successfully to device.pair.remove, device.token.rotate, and device.token.revoke before scheduling device-client disconnect in a microtask, and there is no current invalidated-client flag, invalidateClientsForDevice API, or per-RPC device-token revalidation guard. The PR is a focused gateway/auth hardening change with useful discussion from the bot review, but it still needs maintainer review rather than cleanup closure.

## Best Possible Solution

Keep the PR open for maintainer review. The right fix belongs in the gateway auth/request-dispatch path: synchronously mark affected device-token clients invalid when remove/rotate/revoke succeeds, reject later RPC dispatch for those clients before business logic, and add regression coverage for remove, rotate, revoke, and pipelined/concurrent request ordering.

## Risks / Open Questions

- This is a security-sensitive gateway/auth path; maintainer review should require a regression test that proves pipelined or buffered RPCs cannot execute after token rotation, revoke, or pair removal.
