## Summary

Keep open. Current main already uses GET for MiniMax remains, so the narrow #63056 method bug is gone, but the PR's host/baseUrl behavior is not implemented: main still hardcodes the usage request to api.minimaxi.com and does not pass the configured MiniMax provider baseUrl into fetchMinimaxUsage. The PR remains potentially useful work, though it still needs maintainer review around host/credential alignment and current MINIMAX_API_HOST routing policy.

## Best Possible Solution

Keep the item open or consolidate it with the active MiniMax usage-host work (#65054/#65059), but do not auto-close it. The best fix is a maintainer-reviewed MiniMax usage host resolver that derives from the same provider/auth source as the usage credential, preserves the /v1/api/openplatform/coding_plan/remains path, supports global and CN baseUrls, avoids untrusted workspace env routing, and adds tests for global, CN, invalid baseUrl, and mixed minimax/minimax-portal configurations.
