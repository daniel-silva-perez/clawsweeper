## Summary

Keep this issue open. Current main still has only the existing timer/debounce-based queue modes for `messages.queue`; there is no operator-selectable readiness-aware `until_ready` mode or option for ordinary chat surfaces. Main does have a narrower internal subagent-announce deferral path with a 15s safety cap, but that is hard-coded announce delivery behavior, not the requested public queue collection mode.

## Best Possible Solution

Keep this issue open as core auto-reply queue work. Add an operator-selectable readiness-aware `messages.queue` mode or option, wire it through config types/schema/generated help, `/queue`, command metadata, docs, and focused tests, then implement a generic drain condition tied to real session/run readiness with a bounded starvation fallback. Existing subagent-announce deferral can inform the design, but it does not replace the public chat queue mode requested here.
