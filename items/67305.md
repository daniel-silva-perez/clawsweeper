## Summary

Keep open. Current main still has no readiness-aware `messages.queue` mode or option: the public types/schema only accept the existing queue modes, the docs describe `collect` as debounce/timer-based, and the followup drain flushes after `debounceMs` quiet time rather than a session-ready signal. Existing system-event/exec completion handling can queue events and wake heartbeats, but it is not an operator-selectable until-ready aggregation mode for ordinary chat surfaces. Related collect/queue reports remain open, so this is not implemented or superseded.

## Best Possible Solution

Keep this issue open as core auto-reply queue work, not ClawHub/plugin work. Add an operator-selectable `messages.queue` mode or option for readiness-aware aggregation, wire it through config types/schema/help, `/queue`, docs, and tests, flush on a real session/run readiness signal, and define a bounded starvation fallback for async completion/status notices.
