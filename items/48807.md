## Summary

Keep open. Current main still does not implement PR #48807's central behavior: the health monitor does not drain active channel runs before restarting, does not enforce the proposed 60s cooldown floor, and does not resolve fresh timing policy per check. The PR discussion is substantive and the author responded to multiple review-found races, so the work remains potentially useful rather than obsolete or closeable.

## Best Possible Solution

Keep #48807 open and either rebase/review the PR branch or reimplement the same product behavior on main: before health-monitor-triggered `stopChannel`, poll current runtime `activeRuns` for a bounded drain window, skip stale inherited busy flags using the same lifecycle/staleness rules as `channel-health-policy`, re-check stop state, account presence, health-monitor enablement, and recovered health after the wait, then record cooldown/hourly restart bookkeeping with a fresh timestamp. Add focused regression tests for active reply drain, drain timeout fallback, stopped/recovered/removed/disabled post-drain cases, and cooldown floor behavior.
