## Summary

Current `main` does not implement `/skill` with no args or `/skill list`; the core parser still rejects both, and the reply path only handles resolved skill invocations. The issue remains a valid enhancement request rather than cleanup-close material.

## Evidence

- **`/skill` without args still returns null on main:** `resolveSkillCommandInvocation()` checks `if (commandName === "skill")`, reads `remainder`, and immediately returns `null` when no remainder is present, so `/skill` does not produce a listing flow.
  - file: [src/auto-reply/skill-commands-base.ts:75](https://github.com/openclaw/openclaw/blob/754acc4478fe361db1e5303fd60f589d52beea7a/src/auto-reply/skill-commands-base.ts#L75)
  - command: `nl -ba src/auto-reply/skill-commands-base.ts | sed -n '55,100p'`
  - sha: [754acc4478fe](https://github.com/openclaw/openclaw/commit/754acc4478fe361db1e5303fd60f589d52beea7a)
- **No special handling for `/skill list`:** The same parser treats the first token after `/skill` as a skill name and only succeeds if `findSkillCommand(...)` matches an actual skill command. There is no list-marker return type or branch for `list`.
