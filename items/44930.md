## Summary

Keep open. Current `main` still has the local persisted-watermark guard, but I did not find a code change that defers Telegram's server-side `getUpdates` offset advancement until in-flight updates finish. That means there is not strong evidence that #44930 is already fixed or obsolete, and the issue is also too recent for `stale_insufficient_info`.

## Evidence

- **Runner still uses standard grammY polling with concurrent sink:** `createTelegramRunnerOptions` configures grammY runner fetch/retry/concurrency, but there is no custom source or batch-ack logic here to tie Telegram-side offset advancement to completion of in-flight updates.
  - file: [extensions/telegram/src/monitor.ts:24](https://github.com/openclaw/openclaw/blob/40be5ad58187fd20b8defe5fc30e649190c131a8/extensions/telegram/src/monitor.ts#L24)
  - command: `nl -ba extensions/telegram/src/monitor.ts | sed -n '24,43p'`
- **Polling session only confirms the persisted offset once, then hands control to grammY runner:** The polling session does a one-shot startup `getUpdates({ offset: lastUpdateId + 1, limit: 1, timeout: 0 })` confirmation and then immediately calls `run(bot, this.opts.runnerOptions)`. I did not find code here that holds back later Telegram-side confirmations while updates remain pending locally.
  - file: [extensions/telegram/src/polling-session.ts:228](https://github.com/openclaw/openclaw/blob/40be5ad58187fd20b8defe5fc30e649190c131a8/extensions/telegram/src/polling-session.ts#L228)
