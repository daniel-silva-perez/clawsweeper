## Summary

Current `main` still creates Telegram native-command slash session keys and records inbound metadata for them using the group/topic context, so the reported `agent:<helper>:telegram:slash:<user>` -> group/topic shell behavior is still plausible. Runtime dispatch now prefers `CommandTargetSessionKey`, which may reduce the stale-routing impact, but there is no strong evidence that the stored slash-session aliasing itself has been fixed on `main`.

## Evidence

- **Telegram native commands still persist slash-session metadata from topic/group context:** `extensions/telegram/src/bot-native-commands.ts:855` builds `commandSessionKey` with `telegram:slash:<userId>`, then `extensions/telegram/src/bot-native-commands.ts:881-917` builds a context with `From` set to the group/topic route, `ChatType: "group"`, `MessageThreadId`, and persists it via `recordInboundSessionMetaSafe` under `ctxPayload.SessionKey` (the slash session).
  - file: [extensions/telegram/src/bot-native-commands.ts:855](https://github.com/openclaw/openclaw/blob/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213/extensions/telegram/src/bot-native-commands.ts#L855)

  - sha: [5dfc1b90e176](https://github.com/openclaw/openclaw/commit/5dfc1b90e176a9e4d13b9e91ad6affec6b92f213)
