## Summary

Current main still has the delivery-queue permanent-error classifier, but it does not classify the reported local-media allowlist rejection or the proposed Telegram "message is too long" string. The related local-media staging issue does not supersede this narrow retry-classification request.

## Best Possible Solution

Keep this issue open. The narrow fix belongs in `src/infra/outbound/delivery-queue-recovery.ts`: add permanent classifier coverage for the exact local-media allowlist rejection, add policy and recovery tests proving the queue entry moves directly to `failed/`, and separately verify whether Telegram `message is too long` can still surface on current Telegram delivery before adding that second pattern. Enqueue-time validation can be a follow-up, but the classifier remains useful for already-queued entries.
