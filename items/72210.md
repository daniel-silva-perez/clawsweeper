## Summary

Keep open. Current main has already fixed the older false deliveryStatus/lastDeliveryError path for `delivery.mode: "none"`, and it now avoids the original multi-channel target-resolution error for a bare no-deliver job. However, the run-record trace still gets a `delivery.resolved` object with `ok: false` and an `error` (`"delivery is disabled"`) for the no-delivery path, because the trace builder always serializes failed resolution metadata. That is the specific diagnostic-noise problem reported here, so this issue is not safely closeable as implemented or duplicate of the earlier status-only fixes.

## Best Possible Solution

Keep the issue open and make a narrow cron trace fix: when `deliveryRequested` is false because `delivery.mode === "none"` and there is no explicit chat target to preserve for the message tool, omit `delivery.resolved` or emit neutral skipped metadata instead of `ok:false/error`. Preserve the existing `deliveryStatus: "not-requested"`, preserve explicit no-deliver target context for agent-initiated messaging, and add a focused regression test that inspects the emitted run-log `delivery` object for `delivery.mode="none"`.
