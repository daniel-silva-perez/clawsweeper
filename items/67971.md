## Summary

Current main still matches this issue. `fetchDiscordApplicationId()` only falls back to the token-derived application ID when the lookup throws; a non-OK HTTP response still returns `undefined`, and Discord startup still converts that into `Failed to resolve Discord application id`. The existing shipped work from PR #29695 was explicitly transport/timeout-only, so it does not cover the transient non-OK response path described here.

## Best Possible Solution

Keep this issue open and implement a narrow Discord plugin fix in `extensions/discord/src/probe.ts`: classify non-OK application-id lookup responses so 401/403 continue to fail fast, while transient statuses use the existing token-derived application ID fallback. Add focused Vitest coverage for 429 or 5xx plus 401/403, and keep the startup caller unchanged so it still fails when neither Discord nor the token can provide an application ID.

## Risks / Open Questions

- A fix needs to preserve the intended 401/403 invalid-credential fast failure while allowing token fallback for transient statuses such as 429 and likely 5xx.
