## Summary

Keep open. Current main still does not implement the requested schema-driven tool security classification properties. OpenClaw has related safety controls through owner-only filtering, allow/deny tool policy, plugin `before_tool_call` approvals, exec approvals, and provider-level `parallel_tool_calls`, but those are not a standard per-tool metadata contract for read-only/destructive/concurrency/interrupt/open-world/classifier behavior.

## Best Possible Solution

Keep this open. The best implementation path is a backwards-compatible tool metadata contract on `AnyAgentTool` and the public plugin SDK, preserved through wrappers/adapters/inventory/conformance, with core tools annotated incrementally and policy/UI/scheduling consumers added behind conservative fallbacks.

## Risks / Open Questions

- This is a core/public SDK contract change; defaults and optionality need to preserve third-party plugin compatibility.
- Wrappers, tool inventories, runtime adapters, hooks, trajectory/export surfaces, and provider bridges need to preserve or intentionally drop the metadata consistently.
- Per-tool concurrency metadata should not weaken existing provider, sandbox, owner-only, allow/deny, or approval controls; current heuristics should remain conservative fallbacks.
- `toAutoClassifierInput` would need redaction and SecretRef/media handling so classifier prompts do not leak sensitive arguments.
