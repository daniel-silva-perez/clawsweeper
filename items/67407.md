## Summary

Keep this issue open. Current main has adjacent safeguards, including config validation before managed service start and a separate fix for the cited lossless-claw context-engine regression, but it still does not expose a first-class `openclaw gateway preflight [--json]` command that runs the real gateway startup/plugin/context-engine validation path and exits before binding a port.

## Best Possible Solution

Keep this issue open and implement a first-class `openclaw gateway preflight [--json]` command by extracting the current gateway startup validation, auth/SecretRef resolution, plugin bootstrap, context-engine resolution, and port availability checks into a shared no-bind routine. The routine should be used by `gateway run` before binding and by the new command without starting listeners or mutating session state; update `docs/cli/gateway.md` and add focused CLI/startup tests for the JSON report and failure exit code.

## Risks / Open Questions

- A future implementation should reuse the real startup/config/plugin/context-engine paths; a separate schema-only validator would recreate the drift problem described in the issue.
- The current startup path has side effects such as auth bootstrap, config seeding, startup maintenance, bundled runtime dependency staging, and session migration hooks, so a proper preflight needs a carefully factored no-bind/no-session-write mode rather than directly invoking full startup.
