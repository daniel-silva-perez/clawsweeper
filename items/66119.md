## Summary

Keep this issue open. The report is recent, detailed, and affects the bundled Signal plugin. Current main still auto-starts `signal-cli daemon` without a preflight for a `registered:false` account file, backup step, or startup guard. I also found no changelog/release evidence or related-item context showing this exact account-deletion regression was fixed or superseded; the provided related items are only loose Signal/update title overlaps and do not track this failure mode.

## Best Possible Solution

Keep the issue open and investigate in the bundled Signal plugin. A good fix would first establish a safe reproduction/root cause around the 2026.4.11 to 2026.4.12 update/restart path, then add Signal-owned protection such as a pre-autostart account-file health check, clear warning/fail-closed behavior for `registered:false`, and a documented backup/recovery path. If the destructive transition is proven to be upstream `signal-cli` behavior, OpenClaw should still document the risk and avoid starting the daemon on obviously invalid local account state when feasible.
