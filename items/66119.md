## Summary

Keep this issue open. The report is recent, detailed, affects the bundled Signal plugin, and current main still starts `signal-cli daemon` without checking the local signal-cli account file for `registered:false`, creating a backup, or documenting/failing closed for this destructive state. The existing Codex review comment on the issue reached the same conclusion, and I found no current code, docs, changelog, history, or related-item evidence that supersedes or fixes this exact regression.

## Best Possible Solution

Keep the issue open and investigate in the bundled Signal plugin. The best fix is to establish the 2026.4.11 to 2026.4.12 update/restart root cause, then add Signal-owned protection before auto-starting the daemon: a non-mutating account-state health check where feasible, fail-closed warning behavior for `registered:false`, regression coverage with mocked account data, and user-facing backup/recovery docs. If the destructive transition is ultimately upstream signal-cli behavior, OpenClaw should still avoid starting the daemon on obviously invalid local account state when it can detect it safely.
