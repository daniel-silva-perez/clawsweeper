## Summary

Current `main` still has the reported long-message gap, and this PR is not safe to close as implemented: it would fix the missing webhook split, but its core fallback would pre-chunk Discord sends that already split internally and would regress forum/media delivery into multiple threads.

## Evidence

- **Core delivery still skips text limits when `chunker` is null:** `deliverOutboundPayloadsCore` only resolves `configuredTextLimit` when `handler.chunker` is truthy, and `sendTextChunks` sends the full text unchanged when `handler.chunker` is null. Discord declares `chunker: null` with `textChunkLimit: 2000`, so current `main` does not generically split long Discord text before delivery.
  - file: [src/infra/outbound/deliver.ts:769](https://github.com/openclaw/openclaw/blob/b453293349657dedd069d291fa5b078d910067b1/src/infra/outbound/deliver.ts#L769)
  - command: `nl -ba src/infra/outbound/deliver.ts | sed -n '740,840p'`
  - sha: [b45329334965](https://github.com/openclaw/openclaw/commit/b453293349657dedd069d291fa5b078d910067b1)
- **Discord webhook delivery still sends one unsplit payload:** `maybeSendDiscordWebhookText` passes `params.text` straight to `sendWebhookMessageDiscord`, and `sendWebhookMessageDiscord` serializes that text into a single webhook `content` field. There is no chunking on the webhook path on current `main`.
  - file: [extensions/discord/src/outbound-adapter.ts:143](https://github.com/openclaw/openclaw/blob/b453293349657dedd069d291fa5b078d910067b1/extensions/discord/src/outbound-adapter.ts#L143)
  - command: `nl -ba extensions/discord/src/outbound-adapter.ts | sed -n '100,210p'`
  - sha: [b45329334965](https://github.com/openclaw/openclaw/commit/b453293349657dedd069d291fa5b078d910067b1)
- **Discord bot sends already chunk internally:** `sendDiscordText` calls `buildDiscordTextChunks` and sends each chunk sequentially. A generic core fallback for all `textChunkLimit` adapters would therefore double-split normal Discord bot sends instead of only fixing the missing webhook path.
  - file: [extensions/discord/src/send.shared.ts:366](https://github.com/openclaw/openclaw/blob/b453293349657dedd069d291fa5b078d910067b1/extensions/discord/src/send.shared.ts#L366)
  - command: `nl -ba extensions/discord/src/send.shared.ts | sed -n '259,520p'`
  - sha: [b45329334965](https://github.com/openclaw/openclaw/commit/b453293349657dedd069d291fa5b078d910067b1)
- **Forum/media sends create a new thread per `sendMessageDiscord` call:** When the target resolves to a forum/media channel, `sendMessageDiscord` creates a thread with `Routes.threads(channelId)` and only posts remaining chunks inside that created thread. If core pre-chunks before calling this sender, each pre-chunk against a forum parent would create a separate thread.
  - file: [extensions/discord/src/send.outbound.ts:169](https://github.com/openclaw/openclaw/blob/b453293349657dedd069d291fa5b078d910067b1/extensions/discord/src/send.outbound.ts#L169)

That existing behavior would be violated by pre-chunking before `sendMessageDiscord`.
