## Summary

Keep open. The related #56715 review found a console-settings re-entrancy guard, but current main still has the specific `config set` warning path described here: `config set` writes through config validation, disabled plugin entries with config still emit `Config warnings`, patched `console.warn` still lazily initializes the file logger, and file logger settings still read config again without the console-settings re-entrancy guard. I did not find source or test evidence proving the reported `openclaw config set ...` recursion is fixed.

## Best Possible Solution

Keep this issue open and add a regression test that exercises `openclaw config set channels.discord.enabled false` with a disabled plugin entry that still has config while console capture is enabled. The likely fix belongs in the logging/config boundary: prevent logger settings resolution from recursively loading config while config validation warnings are being emitted, or skip config-backed logging resolution for config mutation commands.
