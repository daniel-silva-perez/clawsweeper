## Summary

Keep open. Current main still has the Discord inbound worker timeout behavior described by the reporter: the timeout is real and configurable, but the Discord fallback reply remains the generic `Discord inbound worker timed out.` message, and logs only add channel/message IDs rather than the requested account/session/run/queue/stage context. The related ClawSweeper items cover narrower listener-timeout, websocket, and attachment-wedge fixes, not this newer diagnostics/unblockability request against 2026.4.24.

## Best Possible Solution

Keep this open and treat it as Discord plugin diagnostics/runtime hardening. The likely implementation belongs around `extensions/discord/src/monitor/inbound-worker.ts`, `message-handler.process.ts`, and the queue/status helpers: capture account id, queue key/session key, timeout source, and best-known run/stage state when the job is enqueued and as it progresses; include safe context in the timeout log and Discord fallback reply; add focused queue tests for the enriched message/log; consider a small production queue-depth/status API if maintainers want queue depth in the timeout report.
