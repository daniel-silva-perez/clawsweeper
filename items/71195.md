## Summary

Keep #71195 open. Current main has the backend and browser Realtime Talk path, but the issue’s remaining core request is native macOS Talk integration, and maintainer discussion explicitly says to keep it open and narrow the target to macOS consuming the existing `talk.realtime.session` flow.

## Best Possible Solution

Keep the issue open, narrowed to native macOS Talk: add a `GatewayConnection` wrapper for `talk.realtime.session`, implement a Swift realtime audio/WebRTC session using the existing Gateway-minted ephemeral session, wire Talk overlay/runtime cancellation and playback state to it, and keep the current STT -> chat -> TTS flow as fallback. Track the ElevenLabs PCM 403/cache behavior separately because `talk.providers.elevenlabs.outputFormat` already exists and is not the same Realtime Talk feature.
