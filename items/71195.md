## Summary

Keep this issue open. Current main has the shared gateway/browser OpenAI Realtime Talk session path, but native macOS Talk still runs the legacy SFSpeechRecognizer -> chat.send -> chat.history -> talk.speak flow and lacks a client wrapper/session implementation for `talk.realtime.session`. The discussion also has explicit maintainer direction to keep it open and narrow the work to macOS client integration.

## Best Possible Solution

Keep this issue open as the canonical native macOS Talk realtime follow-up. The likely implementation is to add a macOS `GatewayConnection` wrapper for `talk.realtime.session`, build a native realtime audio/WebRTC session modeled on `ui/src/ui/chat/realtime-talk.ts`, wire it into `TalkModeRuntime` and overlay phase/cancellation, and preserve the current SFSpeechRecognizer -> chat.send -> talk.speak flow as fallback. Only add backend PCM streaming RPCs if maintainers decide native macOS must use a provider-agnostic backend bridge rather than the existing WebRTC session flow.
