## Summary

Keep #70928 open. Current main still routes WebChat and remote openclaw-tui turns through `chat.send`; that handler builds a bare internal WebChat dispatcher and calls `dispatchInboundMessage` directly, bypassing the wrappers that install `message_sending` and never reaching `deliverOutboundPayloads`, where `message_sent` is emitted. The issue does overstate `before_message_write` and `agent_end`, which are already wired in the embedded runner, but the central outbound lifecycle gap remains.

## Best Possible Solution

Keep the issue open and fix the `chat.send` / internal WebChat delivery path. Prefer a shared outbound observer that runs `message_sending` before internal reply delivery and emits `message_sent` after success/failure, or route the WebChat/TUI path through an existing hook-aware dispatcher without changing external-channel routing semantics. Add regression tests for Control UI WebChat and remote `openclaw-tui` `chat.send` replies, and update hook docs if any channel coverage remains intentionally limited.
