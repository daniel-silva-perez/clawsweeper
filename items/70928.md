## Summary

Current main still leaves the WebChat/Gateway TUI `chat.send` outbound path outside the hook-aware reply dispatch seams. The Gateway handler constructs a raw reply dispatcher and calls the lower-level dispatch function, so `message_sending` is not bridged and `message_sent` is not emitted for that path. The TUI client uses the same `chat.send` method. The `agent_end` subclaim is less clear because the embedded runner has `agent_end` wiring, but the confirmed `message_sending`/`message_sent` gap is enough to keep this issue open.

## Best Possible Solution

Keep this issue open. Fix the Gateway `chat.send` reply path used by WebChat and `openclaw-tui` by routing it through the shared hook-aware reply dispatcher/observer seam, or by adding equivalent `message_sending` and `message_sent` handling around the current delivery path. Also audit Gateway-injected/synthetic assistant transcript appends for `before_message_write` coverage, and add focused regression tests for WebChat `chat.send` and Gateway-backed TUI replies.
