## Summary

Keep open. Current main still does not strip base64 image block data at write-time persistence; it caps text/details and prunes/sanitizes replay/history views instead. The PR targets a real remaining core persistence gap, though it needs maintainer review and production-path tests before landing.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is a focused core fix in the session persistence guard/wrapper that strips image block data after plugin persistence hooks and before JSONL append, plus production-path tests using SessionManager/guardSessionManager to verify image data is cleared, non-image content is preserved, plugin hook ordering remains stable, and empty persisted image blocks replay as placeholders.
