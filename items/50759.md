## Summary

Current main already avoids the reported Ollama missing-key preflight by synthesizing non-secret local auth for Ollama and by auto-registering local Ollama discovery without a real API key, so the issue no longer matches shipped code.

## Evidence

- **Ollama plugin synthesizes local auth for configured provider entries:** The Ollama provider plugin returns `apiKey: OLLAMA_DEFAULT_API_KEY` with source `models.providers.ollama (synthetic local key)` when `resolveSyntheticAuth` sees meaningful explicit Ollama config and no explicit key is required.
  - file: [extensions/ollama/index.ts:187](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/extensions/ollama/index.ts#L187)

  - file: [extensions/ollama/src/discovery-shared.ts:6](https://github.com/openclaw/openclaw/blob/ebd7f19a3b20001960474ec9ba9cc424e45f7b23/extensions/ollama/src/discovery-shared.ts#L6)

## Risks / Open Questions

- Docs still describe `OLLAMA_API_KEY="ollama-local"` as a manual setup path in `docs/providers/ollama.md`, which may confuse users even though the code now supports synthetic local auth in the relevant paths.
- The checkout is shallow/grafted here, so I could confirm the behavior on current main and the latest release tag but not identify the exact first fixing commit or first shipping release with high confidence.

## Close Comment

Closing from a Codex review because current OpenClaw already implements this well enough. On current `main`, the Ollama plugin now synthesizes a non-secret local auth marker for configured Ollama provider entries (`extensions/ollama/index.ts:187`), the auth resolver accepts provider-owned synthetic local auth before the `No API key found for provider "ollama"` failure path (`src/agents/model-auth.ts:563`), and the Ollama discovery coverage explicitly verifies that local model discovery succeeds without a real `OLLAMA_API_KEY` by using `ollama-local` instead (`extensions/ollama/provider-discovery.test.ts:194`). I also verified the same implementation is present in the current release tag `v2026.4.22` at `00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`. If you still hit the old preflight on a newer build, please open a fresh issue with the exact config and gateway logs so it can be compared against the current code path.
