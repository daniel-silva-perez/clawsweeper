## Summary

Keep open. Current main still has the underlying Telegram-sourced unbound approval fan-out shape: the ambiguity guard only applies to non-Telegram turn sources, while an unbound Telegram request still matches every eligible account. The PR addresses a real remaining bug related to #69916, but it needs contributor/maintainer follow-up because the branch has unresolved P1 review feedback around `mode: "both"` and includes unrelated changes.

## Best Possible Solution

Keep the PR open for revision or replace it with a focused maintainer patch for #69916. The best fix is to make unbound Telegram-sourced native approvals fail closed in multi-account setups unless exactly one eligible Telegram account can handle the request, preserve explicit `turnSourceAccountId` routing, honor `approvals.exec.targets[].accountId` only where it does not break `mode: "both"` session delivery, and add regression tests in `extensions/telegram/src/exec-approvals.test.ts` and `extensions/telegram/src/approval-native.test.ts`. Split unrelated script/runtime changes out of this PR.
