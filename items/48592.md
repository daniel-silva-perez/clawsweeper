## Summary

Keep open. Current `main` has a generic Codex OAuth manual fallback, but it does not implement this PR's `localhost:1455` preflight, port-owner diagnostics, or conflict-specific immediate fallback, so there is not strong evidence that the PR is already superseded.

## Evidence

- **Current local flow is still generic and does not preflight the callback port:** `loginOpenAICodexOAuth` shows the standard local note and starts OAuth without probing `localhost:1455`; there is no occupied-port diagnostic path in this file on current `main`. The PR's proposed `tryListenOnPort`/`describePortOwner` behavior is absent.
  - file: [src/plugins/provider-openai-codex-oauth.ts:140](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/src/plugins/provider-openai-codex-oauth.ts#L140)
  - command: `rg -n "detectOpenAICodexCallbackPortConflict|OPENAI_CODEX_CALLBACK_PORT|tryListenOnPort|describePortOwner" src/plugins/provider-openai-codex-oauth.ts -S`
  - sha: [4013c658537e](https://github.com/openclaw/openclaw/commit/4013c658537ee2051c44186f197b0710e84c9049)
- **Manual fallback after browser start is timeout-based, not immediate:** For local flows where browser auth has started, `onManualCodeInput` waits `15000ms` plus a `1000ms` grace window before prompting. Immediate prompting only happens when `hasBrowserAuthStarted()` is false, which is not the PR's occupied-port case as described.
  - file: [src/plugins/provider-openai-codex-oauth.ts:78](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/src/plugins/provider-openai-codex-oauth.ts#L78)
  - sha: [4013c658537e](https://github.com/openclaw/openclaw/commit/4013c658537ee2051c44186f197b0710e84c9049)
- **Current tests cover timeout and no-start fallback, not occupied-port diagnostics:** The test suite on `main` asserts delayed fallback after browser start and immediate fallback when the local callback flow never starts, but it does not cover `EADDRINUSE`, port-owner reporting, or the conflict-specific note proposed by this PR.
  - file: [src/plugins/provider-openai-codex-oauth.test.ts:201](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/src/plugins/provider-openai-codex-oauth.test.ts#L201)
  - sha: [4013c658537e](https://github.com/openclaw/openclaw/commit/4013c658537ee2051c44186f197b0710e84c9049)
- **Docs still describe the callback-bind case as a distinct flow:** The OAuth docs say that if the callback cannot bind, the user should paste the redirect URL/code. That aligns with the PR's targeted UX improvement, and current code does not show equivalent bind-conflict diagnostics.
  - file: [docs/concepts/oauth.md:116](https://github.com/openclaw/openclaw/blob/4013c658537ee2051c44186f197b0710e84c9049/docs/concepts/oauth.md#L116)
  - sha: [4013c658537e](https://github.com/openclaw/openclaw/commit/4013c658537ee2051c44186f197b0710e84c9049)
