## Summary

Keep #39232 open. Current main still lacks a pre-filter raw inbound plugin hook: the typed hook catalog has no `on_raw_inbound`/`onRawInbound`/`before_channel_filter`, unknown typed hooks are ignored, and the existing WhatsApp `message_received` opt-in fires only after access control and group mention/activation gating. The related PR #39240 was closed unmerged, so it does not supersede the request.

## Best Possible Solution

Keep the issue open for maintainer/API design. A good implementation would add an observe-only, privacy-aware pre-filter inbound contract to the typed plugin/channel SDK, wire bundled channels before access control and mention gating only when explicitly enabled, document the privacy model on public plugin/channel docs, and add contract plus channel-specific ordering tests. If maintainers decide raw ambient observation is not supported, document that policy and point users to narrower post-gating alternatives such as the current opt-in WhatsApp `message_received` hook.

## Risks / Open Questions

- A pre-filter raw inbound observer would intentionally expose message content and identifiers that current access control and mention gating can suppress from plugins, so any implementation needs explicit opt-in, privacy documentation, and security review.
- The product/API shape remains a maintainer decision: a global typed hook, per-channel observe sink, channel-specific opt-in, or a documented policy that ambient raw observation is out of scope.
