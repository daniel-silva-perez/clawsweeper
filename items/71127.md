## Summary

Current main still matches the reported gap: stuck-session diagnostics detect and emit `session.stuck`, but there is no diagnostic-triggered abort/recovery path or abort threshold config. The report is recent, detailed, and not maintainer-authored or protected-label blocked. Related #71455 is a provider/history poisoning case, not a canonical replacement for the generic stuck-session recovery request here.

## Best Possible Solution

Keep the issue open. The best path is to add an explicit, conservative stuck-session recovery design: a documented diagnostics abort threshold or policy, a heartbeat-owned recovery hook that resolves sessionKey/sessionId to the active run, cancels embedded/reply/ACP work where applicable, clears or releases the affected queue, emits a structured `session.aborted`/recovery diagnostic event, and has focused tests for hung dispatch state plus non-abort long-turn behavior.

## Evidence

## Risks / Open Questions

- The issue still lacks a deterministic minimal reproduction, so the root cause of the original hang remains unknown.
- A generic stuck-session auto-abort needs careful policy design to avoid killing legitimate long-running multi-tool turns or approval waits.
- Recovery likely has to coordinate several runtime surfaces: embedded runs, reply-run registry, ACP/session-manager cancellation, session queue clearing, diagnostics events, and persisted session state.
