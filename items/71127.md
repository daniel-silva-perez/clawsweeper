## Summary

Keep open. Current main still detects stuck sessions but does not wire that diagnostic to a recovery action. Existing abort paths are manual or tied to the much longer generic run timeout, and the related #71455 DeepSeek issue is provider-specific rather than a superseding fix for this general self-healing request.

## Best Possible Solution

Keep the issue open and implement a core recovery path that connects stuck-session detection to existing run-abort primitives: add a documented `diagnostics.stuckSessionAbortMs` or equivalent policy, abort the active embedded/chat run for the session, clear/release queued work safely, emit a structured `session.aborted` diagnostic event, and add focused tests for heartbeat-driven abort plus false-positive avoidance for long but active runs.
