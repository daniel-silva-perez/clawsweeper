## Summary

Current `main` already implements the requested competing-process handling for gateway/systemd flow: service-mode startup clears stale gateway listeners before bind, the gateway emits explicit `EADDRINUSE` lock errors with remediation, `--force` exists for takeover semantics, and `gateway status --deep`/restart health now detect stale competing listeners. The changelog says this shipped in `v2026.3.7`.

## Risks / Open Questions

- The current cleanup logic targets stale OpenClaw gateway listeners; if a different non-OpenClaw process owns the port, startup still fails fast with diagnostics instead of force-killing that unrelated process.

## Close Comment

Closing from a Codex review because this is already implemented on current `main` and was shipped in `v2026.3.7` (`CHANGELOG.md:3224`). The current gateway service path clears stale gateway listeners before bind (`src/cli/gateway-cli/run.ts:440`, `src/infra/restart-stale-pids.ts:518`), the bind path now surfaces an explicit competing-instance error instead of failing silently (`src/gateway/server/http-listen.ts:49`), and operators have both a takeover path (`openclaw gateway --force`, `docs/cli/gateway.md:58`) and diagnostics (`openclaw gateway status --deep`, `src/cli/daemon-cli/status.print.ts:220`). There is also regression coverage for the old `EADDRINUSE` restart-loop failure mode (`src/infra/restart-stale-pids.test.ts:690`). If you still see a live problem on a newer build, please open a fresh issue with the exact OpenClaw version, service unit, and the output of `openclaw gateway status --deep`.
