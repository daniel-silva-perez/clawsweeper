## Summary

Keep this issue open. Current main still has hardcoded Control UI tabs/routes and the plugin manifest, bootstrap config, SDK API, and docs do not expose a plugin-owned native Control UI page contribution seam. Existing plugins can serve HTTP routes or standalone pages, but that is the limitation this issue is trying to remove, and VISION.md explicitly keeps missing plugin API design work in this repo rather than closing it to ClawHub.

## Best Possible Solution

Keep this issue open as the canonical design/implementation track for a versioned Control UI plugin-page extension API. A good solution would add manifest or SDK page declarations, gateway bootstrap metadata, a browser Custom Element loader, navigation grouping, auth/CSP handling, docs/tests, and explicit answers for non-Lit Custom Elements and plugin-owned backend traffic.

## Risks / Open Questions

- Loading third-party plugin JavaScript inside the Control UI shell is a trust, CSP, auth, and lifecycle design surface; it needs explicit maintainer/product review rather than an auto-close.
- The discussion has unresolved contract questions for non-Lit Custom Elements and backend-paired plugin routes, so an implementation should start with a narrow spec before code.
