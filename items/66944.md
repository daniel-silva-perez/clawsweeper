## Summary

Keep this issue open. Current main still has hardcoded Control UI navigation and page rendering, and the plugin manifest, bootstrap config, SDK API, and docs expose HTTP routes/services but not a native Control UI page contribution seam. The issue discussion also contains a real plugin workaround from Openclaw-Quest and unresolved API questions for non-Lit Custom Elements and plugin-owned backend traffic, so this needs product/API design rather than cleanup closure.

## Best Possible Solution

Keep this issue open as the canonical design and implementation track for a versioned Control UI plugin-page extension API. A good path would add manifest or SDK page declarations, gateway bootstrap metadata, a browser Custom Element loader, navigation grouping and routing, auth/CSP rules, docs/tests, and explicit decisions for non-Lit Custom Elements and plugin-owned backend traffic.
