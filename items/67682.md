## Summary

Current main does not implement the requested per-agent MCP credential overlay. MCP credentials are still scoped to top-level `mcp.servers` entries, the config merge path has no active-agent parameter, and the stdio launcher receives only the server-level env. The related MCP tool-policy path is now useful for a duplicate-server workaround, but it is not equivalent to injecting different env credentials for the same MCP server id per agent.

## Best Possible Solution

Keep this open as a focused core MCP runtime/config feature. The likely path is a schema-backed per-agent MCP credential overlay or identity mechanism, applied before MCP runtime creation and included in the runtime cache fingerprint, with tests proving two agents can use the same MCP server id with different effective env credentials. Document the duplicate-`mcp.servers` plus per-agent tool-policy workaround until that exists.

## Risks / Open Questions

- A duplicate named-server workaround can solve attribution operationally, but it is not transparent: it changes tool names, requires per-agent policy maintenance, and may still discover/start servers before final tool filtering.
- A real fix needs careful cache and lifecycle design: agent-specific MCP env should affect runtime fingerprints, hot reload invalidation, secret handling, and session/process reuse.
