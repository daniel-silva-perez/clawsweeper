## Summary

Keep open. Current main does not show a compaction code path that flips TTS on, and default TTS remains off, but the issue is recent, has a concrete long-conversation/dashboard reproduction, and the reporter followed up that it happened twice. The remaining uncertainty is a real product/debugging gap: whether a persisted TTS preference/config, session `ttsAuto`, model-emitted TTS directive, or post-compaction prompt/context behavior is causing audio-only replies in the reported Gemini/opencode dashboard flow.

## Best Possible Solution

Keep the issue open and ask for a current-main reproduction bundle: sanitized `/tts status`, `messages.tts` config, `~/.openclaw/settings/tts.json` state, the affected session entry including `ttsAuto` before/after compaction, and a transcript/log slice around the first audio-only reply. If those show auto mode was enabled, improve dashboard/status surfacing and recovery guidance. If auto mode is off but audio is still generated after compaction, fix the TTS gate or post-compaction prompt/directive path and add a regression test for dashboard compaction with TTS disabled.
