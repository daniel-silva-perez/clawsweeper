## Summary

Issue #45042 proposes useful memory and compaction capabilities, but current main already treats this area as plugin territory: memory is an exclusive plugin slot, context assembly/compaction are pluggable via context-engine and compaction-provider APIs, and service-backed memory retrieval is already documented through external plugins rather than core changes.

## Risks / Open Questions

- There is no in-tree Supabase/pgvector plugin on current main, so closing as ClawHub assumes the community or reporter will carry that integration externally.
- Privacy isolation and decay/ranking semantics would still need careful implementation and tests in the plugin; core only provides the seams.

## Close Comment

Closing after a Codex review. The proposal is useful, but on current `main` it fits OpenClaw’s plugin surfaces rather than OpenClaw core. `VISION.md:52` says optional capability should usually ship as plugins and `VISION.md:63` makes memory an exclusive plugin slot. The current extension points already cover the requested shape: context-engine plugins own per-turn assembly/recall and compaction (`docs/concepts/context-engine.md:14`, `docs/concepts/context-engine.md:67`, `docs/concepts/context-engine.md:97`), the plugin SDK exposes memory runtime/corpus/embedding hooks (`docs/plugins/sdk-overview.md:102`, `docs/plugins/sdk-overview.md:195`), and compaction providers are explicitly pluggable (`docs/reference/session-management-compaction.md:278`, `src/plugins/compaction-provider.ts:1`). The bundled `memory_search` remains intentionally local to workspace memory files (`extensions/memory-core/src/tools.ts:187`), while service-backed retrieval is already documented as plugin territory (`docs/concepts/memory-honcho.md:9`, `docs/concepts/memory-honcho.md:104`). This should land as a ClawHub memory/context-engine plugin rather than a core issue.
