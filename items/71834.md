## Summary

Keep #71834 open. The reported 2026.4.22 generated thinking bundle syntax failure appears superseded by 2026.4.24 and current main has moved on, but the remaining Telegram webhook startup regression is still present on current main: after binding the local listener, `setWebhook` failure closes the HTTP server, stops the bot, and rethrows into the generic channel auto-restart path. That still creates the reported window where the gateway is running but the local Telegram webhook listener is absent.

## Best Possible Solution

Keep this issue open and fix the Telegram webhook startup path in `extensions/telegram/src/webhook.ts` and/or `extensions/telegram/src/monitor.ts`. The likely product direction is to keep the local listener alive once it has bound, retry recoverable `setWebhook` failures with bounded backoff using existing Telegram network-error classification, and add a regression test that forces `setWebhook` to fail once while asserting the local health/webhook listener remains reachable or is restored before the retry wait. The generated bundle syntax validation should remain covered separately by release/package verification.
