## Summary

Keep open. Current main still does not implement proxy circuit-breaker/direct-fallback behavior for the shared proxy fetch path, and Discord/Telegram proxy consumers still depend on the existing non-circuit-breaking behavior. The PR is not merge-ready as-is because review comments identified real logic bugs and the Discord files have since moved under extensions/discord, but there is no strong close reason or canonical duplicate that preserves this specific dead-proxy CPU/churn issue.

## Best Possible Solution

Keep the PR open, but treat it as a stale implementation sketch rather than merge-ready code. The best path is to port a corrected circuit breaker to the current shared proxy surface in src/infra/net/proxy-fetch.ts, wire current Discord/Telegram consumers deliberately, fix the review bugs by attributing failures only while the proxy is actually used and allowing only one half-open probe, and add focused regression tests for fallback, cooldown, recovery, and proxy-required semantics.
