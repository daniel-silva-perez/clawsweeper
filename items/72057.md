## Summary

Keep #72057 open. Current main has an explicit `models.providers.*.request.proxy` path, but the reported setup only uses container-level `HTTP_PROXY`/`HTTPS_PROXY`. The Google LLM transport still routes through `buildGuardedModelFetch(model)`, and the guarded fetch path does not prove ambient env proxies are honored for model-provider LLM requests without a configured `request.proxy.mode: "env-proxy"`. This is a recent regression report with concrete provider/version/log context, so it should not be auto-closed.

## Best Possible Solution

Keep the issue open and decide the product policy for model-provider LLM proxies. If ambient container `HTTP_PROXY`/`HTTPS_PROXY` should work for Google LLMs, add a regression test around `google-generative-ai`/`buildGuardedModelFetch` and promote env-proxy dispatching while preserving SSRF pinning and `NO_PROXY`. If explicit opt-in is intentional, add a clear Google/provider troubleshooting doc or doctor hint telling users to set `models.providers.google.request.proxy.mode: "env-proxy"`.
