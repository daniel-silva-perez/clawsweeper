## Summary

The request is for a new agent tool surface (`local_llm`) rather than missing core local-model support. Current main already supports Ollama/LM Studio/vLLM/custom OpenAI-compatible local providers, aliases, fallbacks, and usage surfaces; per project vision, optional new tools/skills should ship as plugins/ClawHub rather than core.

## Close Comment

Closing after a Codex review. Current `main` already supports local-model integration through model providers rather than shell-only workarounds: see `docs/providers/ollama.md:24`, `docs/providers/lmstudio.md:9`, and `docs/gateway/local-models.md:14` for local Ollama, LM Studio/llama.cpp, and generic OpenAI-compatible local backends. Core also already has aliases, ordered model fallbacks, and usage surfaces in `docs/cli/models.md:21` and `docs/concepts/model-failover.md:10`.

What remains here is a request for a new agent tool surface (`local_llm`). Per `VISION.md:54` and `VISION.md:68`, optional capability/new skills should ship as plugins or ClawHub skills first, not as new core skills by default. OpenClaw’s plugin API already exposes `registerTool` for exactly this kind of feature (`docs/tools/plugin.md:334`, `docs/plugins/building-plugins.md:47`). Because of that scope boundary, this is better pursued as a ClawHub plugin/skill layered on top of the existing local-provider support, rather than as OpenClaw core.
