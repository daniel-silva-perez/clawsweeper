## Summary

Keep open. Current main still supports only the narrow helper-level case where host read is unwrapped when tools.fs.workspaceOnly=false; I did not find source, docs, tests, or related-item evidence that the reported direct-session/tool-host mismatch for approved non-workspace roots has been fixed. The issue comment already narrows the problem in the same way: the existing unit test passes, but the direct-chat/session read path still needs tracing and regression coverage.

## Best Possible Solution

Keep #70795 open. The best maintainer path is to trace the actual direct-session read-tool host path, decide whether tools.fs.workspaceOnly=false should expose specific readonly roots such as config/state and installed skills, then implement one canonical filesystem policy shared by read, sandbox, OpenShell, and tool-host layers. Add a regression test that exercises direct-session read against an approved non-workspace root rather than only the direct helper-level read tool.
