## Summary

Keep #54878 open. Current main still matches the reported bug: `sessions_history` resolves the target then calls `createSessionVisibilityGuard`, and that guard rejects a cross-agent ACP child before consulting the spawned-session tree. ACP spawn still creates child keys like `agent:<targetAgentId>:acp:<uuid>` and records `spawnedBy`, so a parent-owned ACP child can be both spawned/listed and then blocked from history under default `tree` visibility. The open PR #55983 is a plausible fix track, but it is not merged into current main, so the issue should not be closed yet.

## Best Possible Solution

Keep the issue open and land a narrow fix like #55983: in `createSessionVisibilityChecker`, allow a `tree` target that is in the requester’s spawned set before applying generic cross-agent/a2a checks, then add guard-level and `sessions_history` regression tests for a parent `agent:main:main` reading a spawned child such as `agent:codex:acp:<uuid>` without broad `tools.agentToAgent.allow`.
