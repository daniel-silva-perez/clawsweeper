## Summary

Keep open. This is a fresh regression report, and current main still has code paths matching the reported loss of Stop control after reconnect/reload: the Control UI clears local active-run state on reconnect, the Stop button is rendered only from that local state, and there is no shipped transport-level WebSocket ping fix on main. The related closed #46376 is not a stronger canonical replacement; #56668 remains relevant to the connection-drop side but does not cover the UI/run-control recovery behavior.

## Best Possible Solution

Keep #70991 open. The best maintainer path is to make active chat runs reconnect/reload-safe for Control UI, expose or rehydrate server-side active-run state so Stop remains available, ensure `/stop`/`chat.abort { sessionKey }` works after reconnect for the same browser/session or an admin, and address the gateway transport keepalive gap tracked by related PR #56668 with regression tests covering long task disconnect, reload, Stop, and `/stop`.

## Evidence

Once reconnect/reload loses `chatRunId`, the button disappears even if the Gateway still has an active run.
