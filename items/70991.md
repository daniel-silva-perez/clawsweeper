## Summary

Keep this issue open. Current main still has the reported Control UI recovery gap: Stop visibility is tied to volatile browser-local `chatRunId`, reconnect clears that state, disconnected abort without a local run id is a no-op, and gateway WebSocket paths still lack protocol-level ping/pong. Session-scoped backend abort and `/stop` handling are useful partial fixes, but they do not make Stop survive reload/reconnect or prove the long-task connection drop is fixed.

## Best Possible Solution

Keep this issue open as the canonical Control UI/Gateway long-task recovery regression. A maintainer-quality fix should expose recoverable active run or abortable session state after reconnect/reload, keep a session-level Stop affordance available while gateway work can be aborted, make `/stop` visibly work after a fresh reload, add WS protocol ping/pong or equivalent proven keepalive, and add regression coverage for active-run disconnect/reconnect/reload.
