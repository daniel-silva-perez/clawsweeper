## Summary

Keep open. Current main still rejects Gemini `web_search` freshness before the Gemini request is sent: the Google plugin schema labels `freshness` as unsupported, the runtime calls the shared unsupported-filter guard, and the actual Gemini REST body only sends `tools: [{ google_search: {} }]`. Public OpenClaw docs also say Gemini supports only `query` plus compatibility `count`. The related #17925 tracker is broader and was kept open for missing ZAI/GLM native search, not for this narrower Gemini freshness behavior.

## Best Possible Solution

Keep this open and implement it in the bundled Google plugin, not core: normalize OpenClaw freshness values in `extensions/google/src/gemini-web-search-provider.runtime.ts`, pass the resulting Gemini `google_search.timeRangeFilter` through the request body, include the time filter in the cache key, update `extensions/google/src/gemini-web-search-provider.ts` and `docs/tools/gemini-search.md`, and add focused Google web-search tests for accepted freshness values and invalid/combined date behavior.

## Risks / Open Questions

- The exact REST payload shape should be verified against the pinned `@google/genai` contract or a live Gemini Developer API call before implementation; Google docs/examples still emphasize the empty `google_search` object while SDK docs expose `timeRangeFilter`.
- Mapping relative values like `day`, `week`, `month`, `year`, `pd`, `pw`, and `pm` to absolute start/end timestamps must be deterministic enough for cache keys and tests.
