## Summary

Keep open. Current main still has the reported behavior: the plugin-bound `inbound_claim` handled path returns `queuedFinal: false`, while Telegram sends the silent-reply fallback when `queuedFinal` is false and nothing has been delivered yet. The linked PR #70288 appears to be the right focused fix, but it is still open and the current main SHA has not incorporated it.

## Best Possible Solution

Keep #70286 open until #70288 or an equivalent fix lands. The minimal code fix is to make the plugin-bound handled path communicate that the message has been claimed so Telegram and other channel handlers do not emit fallback text; the better follow-up is an explicit async-claim state that can also support typing indicators or timeout/error feedback when a plugin claims but never responds.
