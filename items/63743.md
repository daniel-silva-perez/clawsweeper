## Summary

Keep #63743 open. Current main has partial IPv6 handling for loopback only, but the reported Gateway LAN/all-interface listener path still resolves to `0.0.0.0`, `custom` bind is validated as IPv4-only, and docs/tests continue to encode IPv4-only non-loopback behavior. The commenter’s `::` / dual-stack suggestion still describes missing core Gateway behavior rather than a ClawHub/plugin concern.

## Best Possible Solution

Keep the issue open and fix this in core Gateway networking. The likely path is to add an explicit IPv6/dual-stack listener strategy for `lan` and/or `custom` binds, decide whether to prefer one `::` dual-stack socket or separate IPv4+IPv6 sockets per platform, preserve the existing non-loopback auth/origin safeguards, add regression coverage for `ws://[::1]` and non-loopback IPv6 listening, then update the config reference, CLI help, status/probe display, and any bind diagnostics.
