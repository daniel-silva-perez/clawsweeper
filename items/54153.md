## Summary

Keep this issue open. Current main still has the reported Signal installer hardening gap: the bundled Signal plugin downloads the selected GitHub release asset through raw `node:https` redirect recursion and streams it directly to disk without the shared guarded fetch helper, an explicit timeout, or byte-limit enforcement.

## Best Possible Solution

Fix this in the bundled Signal plugin. Replace the private raw `node:https` `downloadToFile` implementation in `extensions/signal/src/install-signal-cli.ts` with `fetchWithSsrFGuard` from `openclaw/plugin-sdk/ssrf-runtime`, wire an explicit install-download timeout and redirect limit, decide whether trusted env-proxy mode is appropriate for this GitHub release flow, enforce both declared and streamed byte caps, sanitize the final temp filename, clean partial downloads, and add focused Signal installer regression tests for timeout wiring, private-network redirect blocking, proxy mode selection, oversized downloads, streamed over-limit abort, and cleanup.

## Risks / Open Questions

- The user-triggered Signal install path can still hang on a stalled download, follow redirects without the shared SSRF guard, and write an unbounded response body to disk.
- The eventual fix should choose a conservative Signal archive size cap and clean partial temp files on content-length rejection, streamed over-limit abort, fetch failure, and extraction failure.
- A related Signal installer temp-filename PR in local ClawSweeper history shows adjacent installer hardening is still unresolved, so the implementation should be careful around both URL and asset-name trust boundaries.
