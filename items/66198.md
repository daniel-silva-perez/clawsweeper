## Summary

Keep open. Current main still sends Signal inbound text to the model through raw `BodyForAgent`/`RawBody`, so the hardening request remains relevant. The submitted PR is not safe to merge as-is because it wraps only the legacy `formatInboundEnvelope(...)`/`Body` strings, while the active reply path prefers `BodyForAgent` and turns that into `BodyStripped` for the prompt.

## Best Possible Solution

Keep this PR open, but require a rework before merge. The fix should protect the actual Signal model-facing path, either by deriving `BodyForAgent` from wrapped untrusted user text or by using `UntrustedContext`/structured context consistently, while keeping command parsing on deliberate raw command fields. Pending group history should wrap user text once or use stable marker ids, keep generated message ids outside the untrusted block, and add focused Signal tests for direct messages, group messages, debounced bodies, quote-only context, and pending group history.

## Risks / Open Questions

- Merging the PR as submitted would create a false sense of Signal prompt-injection protection while the normal model prompt still comes from raw `BodyForAgent`.
- Wrapping pending history inside `formatEntry` would call `wrapExternalContent` on render; that helper generates a fresh marker id per call, which can make identical history byte-unstable and hurt prompt-cache determinism.
- The PR's history path places generated `[id:...]` metadata inside the untrusted content block, making trusted application metadata look like attacker-controlled text.
- A correct fix needs to preserve command parsing through `CommandBody`/`RawBody` while marking only model-facing user content or structured supplemental context as untrusted.
