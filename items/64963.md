## Summary

Keep #64963 open. Current main still lacks the PR's active reply-run guard in `runHeartbeatOnce`: heartbeat skips on the global main lane and resolved session lane queue size, but does not consult `resolveActiveReplyRunSessionId(sessionKey)`. The linked root bug #64810 remains open, and the prior identical PR #64823 was closed unmerged by automation rather than superseded by shipped code.

## Best Possible Solution

Keep the PR open for maintainer review or merge a small equivalent fix in `src/infra/heartbeat-runner.ts`: after heartbeat preflight resolves the target `sessionKey` and before the existing session-lane queue check, skip with `requests-in-flight` when `resolveActiveReplyRunSessionId(sessionKey)` is truthy. Keep the regression test that proves an idle lane with an active reply operation is deferred.

## Evidence

- **Heartbeat runner only gates on queue depth:** `runHeartbeatOnce` checks `CommandLane.Main` queue size before preflight, then checks the resolved session lane queue size after preflight and returns `requests-in-flight` only when those queues are nonzero. There is no active reply-run registry guard between preflight session resolution and the session-lane check.
