## Summary

Current main already contains released Telegram polling fixes that match the reported Windows stall symptoms from 2026.3.23-2, including Windows/IPv6 long-poll hardening, transport rebuild/disposal on stalls, and liveness-based restart behavior.

## Evidence

- **Released fix note for the main polling hardening:** The 2026.4.18 changelog explicitly says Telegram polling now uses bounded transport lifecycle, publishes successful getUpdates liveness, avoids false stall restarts after recoverable getUpdates errors, and forces Telegram dispatchers to HTTP/1.1.
  - file: [CHANGELOG.md:499](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/CHANGELOG.md#L499)
  - command: `nl -ba CHANGELOG.md | sed -n '475,505p'`
- **Current code forces HTTP/1.1 for Telegram polling on Windows/IPv6-prone paths:** The dispatcher creation path documents and implements an HTTP/1.1-only pool because Undici HTTP/2 ALPN could stall Telegram long-polling on Windows/IPv6 networks.
  - file: [extensions/telegram/src/fetch.ts:298](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/extensions/telegram/src/fetch.ts#L298)
  - command: `nl -ba extensions/telegram/src/fetch.ts | sed -n '290,380p'`
- **Current polling loop rebuilds transport after stalls and recoverable polling errors:** The polling session marks the transport dirty on stall/error, logs the same stall signatures mentioned in the report, and restarts the polling cycle.
  - file: [extensions/telegram/src/polling-session.ts:243](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/extensions/telegram/src/polling-session.ts#L243)
  - command: `nl -ba extensions/telegram/src/polling-session.ts | sed -n '220,410p'`
- **Current transport state closes stale dispatcher pools instead of leaking dead keep-alive sockets:** Dirty transport rebuilds now explicitly close the previous Telegram transport before reuse, addressing long-running stale-socket behavior.
  - file: [extensions/telegram/src/polling-transport-state.ts:22](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/extensions/telegram/src/polling-transport-state.ts#L22)
  - command: `nl -ba extensions/telegram/src/polling-transport-state.ts | sed -n '1,140p'`
- **Regression coverage exists for the recovered behavior:** Tests now cover rebuilding the transport after stalled polling cycles and ensuring dispatcher pools are bounded and destroyed on close.
  - file: [extensions/telegram/src/polling-session.test.ts:430](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/extensions/telegram/src/polling-session.test.ts#L430)
  - command: `nl -ba extensions/telegram/src/polling-session.test.ts | sed -n '430,560p'`
- **Docs now route this exact symptom to upgrade plus network/polling diagnostics:** Telegram troubleshooting documents polling stalls/reconnect slowness as an upgrade-first issue and points operators at proxy/DNS/IPv6 causes when stalls persist.
  - file: [docs/channels/troubleshooting.md:51](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/docs/channels/troubleshooting.md#L51)
  - command: `nl -ba docs/channels/troubleshooting.md | sed -n '40,60p'`

## Close Comment

Closing as implemented on current main after Codex review. The report was against 2026.3.23-2, and OpenClaw has since shipped Telegram polling hardening that matches the reported failure mode: HTTP/1.1-only Telegram dispatchers for Windows/IPv6 long-poll stability in `extensions/telegram/src/fetch.ts:298`, transport rebuild/disposal on polling stalls and recoverable errors in `extensions/telegram/src/polling-session.ts:243` and `extensions/telegram/src/polling-transport-state.ts:22`, plus released changelog entries in `CHANGELOG.md:499` and `CHANGELOG.md:500`. There is also a later follow-up in `CHANGELOG.md:222` for rebuilding the polling transport after `getUpdates` 409 conflicts. Based on current main and the shipped `v2026.4.18` release notes, this issue is no longer actionable as an open cleanup item. If the same behavior still reproduces on 2026.4.22 or newer, please open a new issue with fresh logs/diagnostics from the current build so it can be triaged as a new regression.
