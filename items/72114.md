## Summary

Keep open. Current main has meaningful bundled runtime-deps hardening, but I do not see source/test evidence that the reported cold-cache update.run/systemd restart race is fully solved. The remaining risk overlaps the still-open #72084/#72058 postinstall/runtime-deps regression family, and the service restart path still uses a direct systemctl restart without a proven wait/deferral around in-progress bundled runtime-deps staging.

## Best Possible Solution

Keep #72114 open as the focused tracker for the update.run cold-cache bundled runtime-deps startup race. The best fix is to eliminate the startup repair window after package updates, likely by landing a robust postinstall stale-version repair, and add a regression path for browser bundled runtime deps through update/restart on a cold cache. If runtime staging can still happen during service restart, add restart deferral or documented service-manager grace handling around in-progress bundled runtime-deps installs.
