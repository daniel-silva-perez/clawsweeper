## Summary

Current main has not implemented the requested media-server response-header hardening. The PR is still a plausible security hardening candidate, but it needs rebasing/adaptation to the current Node HTTP media handler and deliberate maintainer review of the cache and CSP policy.

## Best Possible Solution

Keep this PR open for maintainer review. Adapt the hardening to the current `createMediaRequestHandler` path, add shared assertions for success and error responses, and make an explicit cache/CSP decision in light of active-content attachment handling and Tlon-style media URL fallback behavior.

## Risks / Open Questions

- The PR targets an older Express-style media handler shape; current main uses `createMediaRequestHandler`, so the patch needs a rebase/adaptation before it can land.
- `Cache-Control: no-store` may make one-shot media URLs more fragile for integrations that fall back to the hosted media URL after upload/fetch failure.
- The PR's `Content-Security-Policy` includes `style-src 'unsafe-inline'`; current main's active-content download handling may make a stricter CSP feasible, so the exact policy should be reviewed rather than merged verbatim.
