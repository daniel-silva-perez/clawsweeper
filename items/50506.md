## Summary

Keep this PR open. Current main has not already implemented the requested media-server response-header hardening: the media handler still only sets `X-Content-Type-Options`, and the gateway default headers do not cover this separate media server. Main does include newer active-content download handling, so the PR should be rebased/adapted and its cache/CSP choices reviewed rather than auto-closed.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is to adapt the header hardening to the current media server, add assertions for success and error responses, and deliberately decide the cache and CSP policy in light of current active-content attachment handling and Tlon-style media URL fallback behavior.

## Risks / Open Questions

- The PR's exact diff targets an older media-server shape and needs rebasing onto the current `createMediaRequestHandler` implementation.
- `Cache-Control: no-store` may make one-shot media URLs more fragile for integrations that fall back to the original hosted URL.
- The CSP value should be reconsidered now that active HTML/SVG/XML content is forced to download as opaque bytes; `style-src 'unsafe-inline'` may no longer be the right tradeoff.
