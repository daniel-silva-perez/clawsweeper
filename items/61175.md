## Summary

Keep this issue open. Current main still has the exact misleading dual-error route: after an npm plugin install failure, non-forced security-scan failures are not treated as terminal plugin failures, the bundled fallback only applies to package-not-found, and the CLI then attempts hook-pack installation and formats the result with `Also not a valid hook pack: ...`. The provided GitHub context also includes a recent maintainer comment explicitly keeping this as the broader canonical tracker after closing #69181 as a duplicate.

## Best Possible Solution

Keep this issue open and fix the CLI install routing. The best implementation is to classify hook-pack fallback more narrowly, for example only after a missing/non-plugin package classification or an explicit hook-pack-looking package, while reporting plugin security-policy failures and valid-plugin validation failures as plugin errors only. Add a regression in `src/cli/plugins-cli.install.test.ts` for a normal npm plugin security-block failure without `--dangerously-force-unsafe-install`, asserting `installHooksFromNpmSpec()` is not called and `Also not a valid hook pack` is absent.
