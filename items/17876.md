## Summary

Keep open. Current `main` has manual model-setting and fallback primitives, plus provider/auth probing, but it does not implement the requested staged model migration workflow (`test`/`promote`/`rollback`).

## Evidence

- **CLI surface has no staged migration commands:** `openclaw models` currently registers `list`, `status`, `set`, `set-image`, `aliases`, `fallbacks`, `image-fallbacks`, `scan`, and auth-order/auth helpers. There is no `models test`, `models migrate`, or `models rollback` command.
  - file: [src/cli/models-cli.ts:12](https://github.com/openclaw/openclaw/blob/a59d1bd46d9fc0bd2d9194b3f17db2561efc51e3/src/cli/models-cli.ts#L12)
  - command: `nl -ba src/cli/models-cli.ts | sed -n '1,240p'`
  - sha: [a59d1bd46d9f](https://github.com/openclaw/openclaw/commit/a59d1bd46d9fc0bd2d9194b3f17db2561efc51e3)
- **Docs still describe direct set/probe/fallback flows only:** The user docs list `openclaw models status`, `list`, `set`, `scan`, plus fallback management and `status --probe`. They do not document any staged migration, validation-before-promotion, or rollback workflow.
  - file: [docs/cli/models.md:19](https://github.com/openclaw/openclaw/blob/a59d1bd46d9fc0bd2d9194b3f17db2561efc51e3/docs/cli/models.md#L19)

  - file: [src/commands/models/set.ts:6](https://github.com/openclaw/openclaw/blob/a59d1bd46d9fc0bd2d9194b3f17db2561efc51e3/src/commands/models/set.ts#L6)

## Risks / Open Questions

- `models status --probe` provides useful auth/provider probing, but it is not evidence that a candidate replacement model can safely run the user's real sub-agent/tool workload.
- The issue is a clear UX feature request, not incoherent or stale-insufficient, and it still fits current OpenClaw scope around setup reliability and ergonomics rather than ClawHub-only functionality.
