## Summary

Keep open. Current main has already shipped the timeout-bump portion of PR #49751 via a shared 10s pre-auth/connect-challenge timeout and docs/tests, but it has not implemented the PR's core timer-ordering change: the server still awaits auth resolution and later setup before calling `clearHandshakeTimer()`. Because the open PR's early-clear approach also has an unresolved review concern about removing the only auth-stall watchdog, this should not be auto-closed as implemented or merged as-is.

## Best Possible Solution

Do not auto-close this PR as implemented. A maintainer should either close it manually as superseded by the shipped 10s shared timeout policy if that is the intended product boundary, or ask for a revised PR that keeps the shipped shared timeout helper, moves the timer boundary only with a dedicated auth-resolution timeout/watchdog, and adds a regression test for slow-but-valid auth versus permanently stalled auth.
