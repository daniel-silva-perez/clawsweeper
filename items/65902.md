## Summary

Keep open. Current main has generic, store-backed session discovery, but I do not see source or tests proving the specific WhatsApp consistency requested in #65902: a direct WhatsApp session that status/cron can use is always exposed through agent `sessions_list` and then targetable by `sessions_send`. The current agent send tool still resolves only by session key or label, and gateway search does not include phone/delivery target fields, which matches part of the reported failure mode.

## Best Possible Solution

Keep #65902 open for a focused maintainer fix or regression test. The likely path is to make the gateway/agent session discovery contract explicit for WhatsApp direct sessions: either ensure the same store row used by status/cron is always visible through `sessions.list` and `sessions_list`, including useful `deliveryContext` and phone search fields, or document and test that `sessions_send` only targets full session keys/labels and add a supported number-to-session resolution path if product wants that workflow.
